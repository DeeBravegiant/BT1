### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Inject Arbitrary Sequencer Address into Block Execution and Hash - (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates the `proposer` identity field in `ProposalInit` (via `handle_proposal` in `manager.rs`) but never validates the `builder` field. The `builder` field is the sequencer/builder address that flows directly into `BlockInfo.sequencer_address`, which is (a) exposed to every contract via the `get_execution_info` syscall, (b) included in `PartialBlockHashComponents.sequencer` and therefore in the `PartialBlockHash` / `ProposalCommitment`, and (c) used for fee collection. A legitimate consensus proposer can set `builder` to any arbitrary address; the validator accepts it without complaint, executes the block under that address, and commits the result.

---

### Finding Description

`ProposalInit` carries two identity fields:

| Field | Description | Validated? |
|---|---|---|
| `proposer` | Consensus proposer address | ✓ checked in `handle_proposal` (manager.rs:860) |
| `builder` | Block builder / sequencer address | ✗ **never checked** | [1](#0-0) 

`is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, timestamp bounds, all four L1 gas prices, and `fee_proposal_fri`. It never reads `init_proposed.builder`. [2](#0-1) 

`ProposalInitValidation` — the struct that carries the validator's locally-trusted reference values — has no `builder` field at all, so there is no expected value to compare against. [3](#0-2) 

On the **proposer** side, `builder` is set from the node's own config (`args.builder_address`): [4](#0-3) 

On the **validator** side, `convert_to_sn_api_block_info(&args.init)` is called with the wire-received `init` (including its unchecked `builder`) and the result is passed directly to `batcher.validate_block`: [5](#0-4) 

`PartialBlockHashComponents` includes `sequencer` (derived from `block_info.sequencer_address`) in the block hash: [6](#0-5) 

`calculate_block_hash` chains `sequencer.0` into the Poseidon hash: [7](#0-6) 

The `sequencer_address` is also exposed to every contract via the `get_execution_info` syscall: [8](#0-7) 

The `ProposalFinMismatch` guard at line 244 does **not** catch this: if the proposer sends `builder = X` and executes with `sequencer_address = X`, it will produce `partial_block_hash_X`. The validator also executes with `sequencer_address = X` (from the unchecked `init.builder`) and produces the same `partial_block_hash_X`. Both sides agree, so the mismatch check passes — even though `X` is not the legitimate builder address. [9](#0-8) 

---

### Impact Explanation

A malicious consensus proposer (a legitimate validator whose turn it is to propose) can set `ProposalInit.builder` to any address — including their own, zero, or a contract they control. Every validator node will:

1. Execute the block with the injected `sequencer_address`, causing every contract that calls `get_execution_info` to observe the wrong sequencer address — **wrong execution state from syscall**.
2. Commit a `PartialBlockHash` / `ProposalCommitment` that encodes the injected address — **wrong block hash**.
3. Direct sequencer fee revenue to the injected address — **incorrect fee/balance effect with economic impact**.

---

### Likelihood Explanation

Any validator that wins a proposer slot can trigger this. No external attacker capability is required beyond being a member of the active validator set. The attack is silent: no error is logged, no metric is incremented, and the `ProposalFinMismatch` guard is bypassed by construction.

---

### Recommendation

Add `builder` to `ProposalInitValidation` and validate it in `is_proposal_init_valid`, mirroring the existing `proposer` check in `handle_proposal`:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid:
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The validator's `ProposalInitValidation` should be populated with the locally-configured `builder_address` (the same value the proposer reads from `static_config.builder_address`). [10](#0-9) 

---

### Proof of Concept

```
1. Attacker is the legitimate proposer for height H, round R.
2. Attacker constructs ProposalInit with:
     builder = <attacker_controlled_address>   // arbitrary, not the real builder_address
     (all other fields valid and within margin)
3. Attacker streams the proposal to all validators.
4. Each validator calls is_proposal_init_valid() — passes (builder is never read).
5. initiate_validation() calls convert_to_sn_api_block_info(&init), which maps
   init.builder → BlockInfo.sequencer_address = <attacker_controlled_address>.
6. Batcher executes the block; PartialBlockHashComponents.sequencer = <attacker_controlled_address>.
7. PartialBlockHash is computed over the injected address; ProposalCommitment matches
   the proposer's Fin (both used the same builder value) → ProposalFinMismatch check passes.
8. Consensus reaches decision; block is committed with:
   - sequencer_address = <attacker_controlled_address> in all get_execution_info calls
   - block hash encoding the injected address
   - sequencer fees directed to <attacker_controlled_address>
```
