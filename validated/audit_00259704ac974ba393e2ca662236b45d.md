### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Enables Repeated Re-Verification and Cross-Caller Signature Sharing — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` function uses a `VerifyForeignTransactionRequest` map key that contains no caller account ID, unlike `sign` which binds each request to its predecessor. Combined with the absence of any "completed verifications" set, the same foreign-chain transaction can be submitted, signed, and responded to an unlimited number of times by any unprivileged caller. Each completed signing round produces a fresh, independently valid MPC signature over the same payload hash, enabling double-spend conditions in bridge contracts that consume these proofs.

---

### Finding Description

**Root cause 1 — caller-agnostic request key.**

`VerifyForeignTransactionRequest` is defined as:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

It contains no caller account ID. The conversion from args to request in `args_into_verify_foreign_tx_request` confirms the predecessor is silently dropped:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [2](#0-1) 

By contrast, `sign` explicitly binds the request to the predecessor:

```rust
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,
    &request.path,
);
``` [3](#0-2) 

The contract's own unit test acknowledges this design explicitly — "a different account would today be blocked from receiving a response by alice's submission" — and confirms that Alice and Bob both receive the identical `VerifyForeignTransactionResponse` from a single MPC signing round: [4](#0-3) 

**Root cause 2 — no completed-verification set.**

After `respond_verify_foreign_tx` drains the pending queue, the entry is simply removed from `pending_verify_foreign_tx_requests`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [5](#0-4) 

No record of the completed verification is retained. The same `VerifyForeignTransactionRequest` can be enqueued again immediately, causing MPC nodes to perform a second independent signing round over the same payload.

**Root cause 3 — signature is not caller-bound.**

`respond_verify_foreign_tx` verifies the signature against the **root public key** with no per-caller tweak:

```rust
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // root key, no tweak
)
``` [6](#0-5) 

`ForeignTxSignPayloadV1` contains only `{request, values}` — no caller identity: [7](#0-6) 

Every signature produced for the same `(tx_id, extractors, domain_id)` is therefore interchangeable and usable by any party.

---

### Impact Explanation

ECDSA is non-deterministic: two independent signing sessions over the same 32-byte payload hash produce two **different** but **equally valid** signatures. A bridge contract that:

1. accepts any valid MPC signature over a foreign-transaction payload hash, and
2. does not maintain its own settled-transaction ledger

can be presented with `signature_1` and `signature_2` (from two separate `verify_foreign_transaction` → `respond_verify_foreign_tx` cycles) and authorize two separate asset releases for the same on-chain foreign event — a direct double-spend. This matches the allowed High impact: *"Cross-chain replay … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

The attack requires no privileges. Any NEAR account can call `verify_foreign_transaction` with an arbitrary `ForeignChainRpcRequest`. After the first cycle completes and the pending entry is removed, the same request can be re-submitted immediately. The attacker controls the timing and repetition count entirely. [8](#0-7) 

---

### Recommendation

1. **Persist a completed-verification set.** After `respond_verify_foreign_tx` succeeds, store the Borsh hash of the `VerifyForeignTransactionRequest` in a `LookupSet`. Reject any new `verify_foreign_transaction` call whose request hash is already present.

2. **Bind the signature to the caller.** Include `predecessor_account_id` in `VerifyForeignTransactionRequest` (mirroring `SignatureRequest`) so that the signed payload is caller-specific and cannot be reused across accounts.

---

### Proof of Concept

```
// Round 1
alice.call("verify_foreign_transaction", { bitcoin_tx_id: X, ... })
// MPC nodes verify and sign → alice receives { payload_hash: H, signature: sig_1 }

// Round 2 — same request, no restriction
alice.call("verify_foreign_transaction", { bitcoin_tx_id: X, ... })
// MPC nodes verify and sign again → alice receives { payload_hash: H, signature: sig_2 }

// sig_1 ≠ sig_2 (ECDSA non-determinism), but both verify against root_pubkey over H.
// A bridge contract that does not track settled hashes accepts both,
// releasing assets twice for the single foreign-chain event X.
```

The pending-request map confirms re-enqueuing is unconditional: [9](#0-8)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
