Audit Report

## Title
Missing Chainlink `minAnswer`/`maxAnswer` Circuit Breaker Validation Allows Inflated Price Consumption During LST Depeg - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and immediately casts the result to `uint256` with zero validation. Chainlink aggregators clamp their return value at a hardcoded `minAnswer` floor when the true market price crashes below it. The protocol will accept the depegged LST at the artificially inflated `minAnswer` price, allowing an attacker to mint excess rsETH and extract value from existing holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` contains no guards of any kind: [1](#0-0) 

The raw `int256 price` from `latestRoundData()` is cast directly to `uint256` and returned. There is no check that `price > 0`, no staleness check on `updatedAt`, no `answeredInRound >= roundId` check, and no check against the aggregator's `minAnswer`/`maxAnswer` circuit breaker bounds.

This price flows directly into rsETH minting: [2](#0-1) 

Which is called from the public `depositAsset()` entry point: [3](#0-2) 

The `LRTOracle._updateRsETHPrice()` has a `pricePercentageLimit` downside protection that can pause the protocol, but this is only triggered when `updateRSETHPrice()` is called externally — it is not invoked during `depositAsset()`. An attacker can deposit before any price update is triggered. [4](#0-3) 

`LRTWithdrawalManager._validatePrices()` provides operator-set price bounds in the withdrawal path, but these are separate from Chainlink's `minAnswer`/`maxAnswer` and do not protect the deposit path. [5](#0-4) 

`LRTOracle.updatePriceOracleForValidated()` performs a one-time sanity check (price between `1e16` and `1e19`) only at oracle registration time, not at runtime during deposits. [6](#0-5) 

## Impact Explanation

**Critical — Direct theft of user funds at rest.**

If a supported LST (stETH, cbETH) depegs severely below the Chainlink aggregator's `minAnswer`:
1. The aggregator freezes its return at `minAnswer` (e.g., 0.95e18) while the true market price is far lower (e.g., 0.05e18).
2. An attacker buys the depegged LST cheaply on the open market.
3. The attacker calls `depositAsset()` with the depegged LST.
4. `getRsETHAmountToMint()` uses the inflated `minAnswer` price, minting far more rsETH than the deposited asset is worth.
5. The attacker redeems or sells the excess rsETH, extracting ETH value from all existing rsETH holders.

This directly dilutes and steals from existing rsETH holders' backing, constituting direct theft of funds at rest — a Critical impact under the allowed scope.

## Likelihood Explanation

LST depeg events have historical precedent (stETH depeg in 2022). The Chainlink `minAnswer` circuit breaker behavior is well-documented and has been exploited in other protocols. The attack requires no special permissions — any unprivileged depositor can execute it by calling the public `depositAsset()` function. The SECURITY.md exclusion for "depegging of an external stablecoin" does not apply here because stETH and cbETH are liquid staking tokens, not stablecoins. The "incorrect data supplied by third-party oracles" exclusion also does not apply because the `minAnswer` clamping is a known, documented Chainlink design behavior — the bug is the protocol's failure to account for it. [7](#0-6) 

## Recommendation

Retrieve `minAnswer` and `maxAnswer` from the underlying Chainlink aggregator and revert if the returned price is at or beyond either bound. Also add staleness and round completeness checks:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    require(price > 0, "Invalid price");
    require(updatedAt != 0, "Incomplete round");
    require(answeredInRound >= roundId, "Stale price");

    int192 minAnswer = AggregatorV2V3Interface(address(priceFeed)).minAnswer();
    int192 maxAnswer = AggregatorV2V3Interface(address(priceFeed)).maxAnswer();
    require(price > minAnswer, "Price at lower circuit breaker bound");
    require(price < maxAnswer, "Price at upper circuit breaker bound");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept

**Foundry fork test plan:**

1. Fork mainnet with a supported LST asset (e.g., stETH) and its Chainlink feed.
2. Deploy a mock Chainlink aggregator that returns `minAnswer` (e.g., `0.95e8` for an 8-decimal feed) regardless of the true price.
3. Register the mock feed via `ChainlinkPriceOracle.updatePriceFeedFor()`.
4. Acquire 1000 stETH at the true crashed market price (e.g., 0.05 ETH each = 50 ETH total cost).
5. Call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
6. Assert that `getRsETHAmountToMint()` returns rsETH valued at ~950 ETH (using `minAnswer` = 0.95e18) rather than ~50 ETH.
7. Redeem or sell the excess rsETH, demonstrating ~900 ETH net extraction from existing holders.

The absence of any guard in `getAssetPrice()` means step 6 will succeed with the inflated price, confirming the vulnerability. [8](#0-7)
