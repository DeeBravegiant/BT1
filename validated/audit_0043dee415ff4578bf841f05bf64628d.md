Audit Report

## Title
Missing Stale Price Check in `ChainlinkPriceOracle.getAssetPrice()` Enables Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` discards all `latestRoundData()` return values except `answer`, performing no staleness validation. When a Chainlink feed freezes at a price above the true current price (e.g., feed froze before a depeg event), any depositor can call `LRTDepositPool.depositAsset()` and receive more rsETH than their collateral is worth, diluting all existing rsETH holders and trending toward protocol insolvency. The claim's PoC has the direction of impact reversed (stale-low causes under-minting, not over-minting), but the critical over-minting impact is valid and reachable via the stale-high scenario.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at L52, all five return values of `latestRoundData()` are available but only `answer` is captured:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No check is performed on `updatedAt` (time-based staleness), `answeredInRound < roundId` (round-based staleness), `updatedAt == 0` (incomplete round), or `price <= 0` (invalid price).

This contrasts directly with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`, which implements three of these four checks: [2](#0-1) 

The stale price flows through `LRTOracle.getAssetPrice()`: [3](#0-2) 

Into `LRTDepositPool.getRsETHAmountToMint()`: [4](#0-3) 

The minting formula is `rsethAmountToMint = (amount * assetPrice) / rsETHPrice`. Critically, `rsETHPrice` is a **stored state variable** updated only when `updateRSETHPrice()` is called explicitly — it is not refreshed on every deposit. This creates a window where `assetPrice` (from the stale feed) and `rsETHPrice` (from a prior update) diverge.

**Correct over-minting scenario (stale-high):**
1. Protocol holds 1000 stETH; rsETH supply = 1000; `rsETHPrice` = 1.0 ETH/rsETH (last updated when stETH/ETH = 1.0).
2. stETH depegs to 0.9 ETH, but the Chainlink feed freezes at 1.0 (stale-high).
3. `rsETHPrice` is not updated (or was last updated before the depeg).
4. Attacker deposits 1000 stETH (true value: 900 ETH).
5. `getAssetPrice(stETH)` returns stale 1.0; `rsethAmountToMint = (1000e18 × 1.0e18) / 1.0e18 = 1000 rsETH`.
6. Fair rsETH for 900 ETH at rsETHPrice 1.0 = 900 rsETH. Attacker receives 100 extra rsETH.

The `minRSETHAmountExpected` slippage guard in `_beforeDeposit` does not protect against this — it only prevents the depositor from receiving *fewer* rsETH than expected, not from receiving *more*: [5](#0-4) 

## Impact Explanation
When a Chainlink feed freezes at a price above the true current price (the canonical depeg scenario), any depositor can mint rsETH at an inflated rate. This directly dilutes the backing of all existing rsETH holders. Repeated exploitation drains the protocol's collateral base relative to outstanding rsETH supply, constituting **protocol insolvency** (Critical). The `_getTotalEthInProtocol()` function used to compute `rsETHPrice` also calls `getAssetPrice()`, meaning a stale-high price inflates the apparent TVL, masking the insolvency until the feed recovers. [6](#0-5) 

## Likelihood Explanation
Chainlink feeds can go stale due to oracle node downtime, network congestion, or L2 sequencer issues. The stale-high scenario (feed freezes before or during a depeg recovery) is historically observed (e.g., stETH depeg events in 2022). `depositAsset()` is a permissionless public function callable by any address at any time the protocol is unpaused. No privileged role is required to trigger the exploit path. [7](#0-6) 

## Recommendation
Apply the same staleness checks already present in `ChainlinkOracleForRSETHPoolCollateral` and add a time-based threshold in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > STALE_PRICE_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`STALE_PRICE_THRESHOLD` should be configured per feed based on its documented heartbeat (e.g., 3600 seconds for 1-hour heartbeat feeds, with a small buffer). Consider also adding a circuit-breaker that pauses deposits when any price feed is detected as stale.

## Proof of Concept
**Foundry fork test outline:**

```solidity
// Fork mainnet at a block where stETH/ETH Chainlink feed is live
// Mock the Chainlink aggregator to return a stale answer:
//   roundId = 100, answeredInRound = 100, updatedAt = block.timestamp - 2 hours
//   answer = 1.0e8 (stale-high; true price has dropped to 0.9e8)

// 1. Record rsETHPrice (stored state, last updated at true price 1.0)
// 2. Warp block.timestamp forward 2 hours (feed now stale)
// 3. Call depositAsset(stETH, 1000e18, 0, "")
// 4. Assert rsETH minted == 1000e18 (uses stale 1.0 price)
// 5. Assert fair rsETH at true price 0.9 == 900e18
// 6. Assert over-mint == 100e18 rsETH extracted from existing holders
```

The test demonstrates that `getAssetPrice()` returns the stale 1.0 price without reverting, and the depositor receives 1000 rsETH for collateral worth only 900 ETH, with no existing guard preventing the over-mint.
