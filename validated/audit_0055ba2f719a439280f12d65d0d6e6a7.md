Audit Report

## Title
USDC-Blacklisted LP Permanently Loses Principal: `removeLiquidity` Hardcodes Transfer to `owner` With No Alternative Recipient - (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`LiquidityLib.removeLiquidity` unconditionally transfers withdrawn tokens to the position `owner` address with no `recipient` parameter. If the pool contains USDC or USDT and the LP's address is subsequently blacklisted, every `removeLiquidity` call reverts at the `safeTransfer` step. Because `MetricOmmPool` enforces `msg.sender == owner` and provides no alternative-recipient or position-transfer mechanism, the LP's principal is permanently locked in the pool with no on-chain recovery path.

## Finding Description

`LiquidityLib.removeLiquidity` computes token amounts owed to the LP and transfers them unconditionally to `owner`:

```solidity
// metric-core/contracts/libraries/LiquidityLib.sol L242-247
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
}
```

`MetricOmmPool.removeLiquidity` enforces that only the position owner can call the function:

```solidity
// metric-core/contracts/MetricOmmPool.sol L206
if (msg.sender != owner) revert NotPositionOwner();
```

The function signature accepts no `recipient` parameter:

```solidity
// metric-core/contracts/MetricOmmPool.sol L199
function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
```

Positions are keyed by `keccak256(abi.encode(owner, salt, bin))` (LiquidityLib.sol L258), permanently binding them to the `owner` address with no transfer mechanism. USDC's `transfer` reverts when the recipient is blacklisted; `safeTransfer` propagates this revert, rolling back all state changes and leaving the LP's shares intact but permanently unreachable.

By contrast, `swap` already accepts a caller-specified `recipient` (MetricOmmPool.sol L219) and routes output tokens there (L254, L268), demonstrating the pattern is known and intentionally omitted from `removeLiquidity`.

## Impact Explanation

An LP who deposited into a USDC- or USDT-paired Metric OMM pool and is subsequently blacklisted by Circle loses 100% of their deposited principal with no on-chain recovery path. The funds remain in the pool indefinitely as unclaimable dead weight. This constitutes direct, permanent loss of user principal — qualifying as a Critical/High impact under the Allowed Impact Gate (direct loss of user principal, broken core withdraw flow).

## Likelihood Explanation

USDC blacklisting is a documented, actively exercised mechanism; Circle has blacklisted hundreds of addresses. Any LP in any USDC- or USDT-paired Metric OMM pool is exposed. The LP need not be malicious — regulatory action, sanctions, or exchange-level compliance events can trigger blacklisting after the LP has already deposited. The condition is realistic, unprivileged from the protocol's perspective, and repeatable across any pool containing USDC or USDT.

## Recommendation

Add an optional `recipient` parameter to `MetricOmmPool.removeLiquidity` and `LiquidityLib.removeLiquidity`. When provided, transfer withdrawn tokens to `recipient` instead of `owner`. The `msg.sender == owner` check continues to enforce that only the position owner can initiate the withdrawal; the owner simply directs proceeds to a non-blacklisted address:

```solidity
function removeLiquidity(
    address owner,
    address recipient,   // <-- new
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external ...
```

Inside `LiquidityLib.removeLiquidity`, replace `safeTransfer(owner, ...)` with `safeTransfer(recipient, ...)`. This mirrors the pattern already used in `swap` where output tokens are sent to a caller-specified `recipient` rather than `msg.sender`.

## Proof of Concept

1. Alice provides liquidity to a USDC/WETH Metric OMM pool via `addLiquidity`. Her position is recorded under `keccak256(abi.encode(alice, salt, bin))`.
2. Circle blacklists Alice's address (e.g., due to a regulatory freeze).
3. Alice calls `removeLiquidity(alice, salt, deltas, "")`. The pool enforces `msg.sender == owner == alice` — passes (L206).
4. `LiquidityLib.removeLiquidity` computes `amount1Removed > 0` (USDC owed) and calls `IERC20(usdc).safeTransfer(alice, amount1Removed)` (L246).
5. USDC's `transfer` reverts because `alice` is blacklisted. The entire transaction reverts.
6. Alice's shares remain in the pool. She cannot specify a different recipient. She cannot transfer her position. Her USDC principal is permanently locked.
