### Title
Unprivileged `TransferToGasKey` Griefs Victim's `DeleteKey` / `DeleteAccount` via `GasKeyBalanceTooHigh` - (File: runtime/runtime/src/access_keys.rs)

### Summary

Any unprivileged account can send a `TransferToGasKeyAction` targeting another account's gas key. Because `delete_gas_key` and `action_delete_account` both hard-revert with `GasKeyBalanceTooHigh` when the gas key balance exceeds `GasKeyInfo::MAX_BALANCE_TO_BURN` (1 NEAR), an attacker can permanently grief a victim's `DeleteKey` or `DeleteAccount` transaction by frontrunning it with a small deposit that pushes the gas key balance over the threshold.

### Finding Description

`TransferToGasKeyAction` is a public protocol action that any account can include in a transaction whose `receiver_id` is the target account. The runtime handler `action_transfer_to_gas_key` performs no ownership check — it only verifies that the named public key exists and is a gas key on the receiver account, then unconditionally increments its balance: [1](#0-0) 

When the account owner later submits `DeleteKey`, the runtime calls `delete_gas_key`, which hard-fails if the balance exceeds the 1 NEAR burn cap: [2](#0-1) 

The same guard exists in `action_delete_account` for the aggregate of all gas key balances: [3](#0-2) 

The threshold is defined as: [4](#0-3) 

The `TransferToGasKeyAction` variant is a first-class protocol action available to any signer: [5](#0-4) 

**Attack flow:**

1. Victim's gas key has balance `B < 1 NEAR`.
2. Victim broadcasts `DeleteKey { public_key }`.
3. Attacker observes the pending transaction and broadcasts `TransferToGasKey { public_key, deposit: (1 NEAR - B + 1 yocto) }` with a higher gas price, targeting the victim's account.
4. Attacker's transaction lands first; gas key balance is now `> 1 NEAR`.
5. Victim's `DeleteKey` executes and returns `GasKeyBalanceTooHigh` — the key is not deleted.

The same sequence applies to `DeleteAccount` by pushing the *sum* of all gas key balances above 1 NEAR.

### Impact Explanation

The victim cannot delete a gas key or their own account as long as the attacker keeps the gas key balance above `MAX_BALANCE_TO_BURN`. The corrupted protocol value is the `ActionResult::result` field, which is set to `Err(GasKeyBalanceTooHigh)` instead of `Ok(())`, causing the on-chain state transition to be a no-op for the deletion. The victim's gas key and/or account remain in state indefinitely against their will.

A victim can attempt to combine `WithdrawFromGasKey` and `DeleteKey` in a single atomic transaction, but the attacker can frontrun the entire batch by depositing enough NEAR that even after the victim's withdrawal the balance remains above 1 NEAR. Because the attacker's deposit is permanently burned on eventual deletion, the attack has a real cost, but 1 NEAR is cheap enough to sustain repeated griefing rounds.

### Likelihood Explanation

The attack requires only a standard signed transaction targeting the victim's account with `TransferToGasKey`. No special privileges, validator access, or contract deployment is needed. The attacker must spend slightly more than 1 NEAR per sustained grief cycle (the deposit is burned), but this is a low barrier for a motivated adversary. The attack is directly analogous to the confirmed DYAD M-06 finding.

### Recommendation

Add an ownership check to `action_transfer_to_gas_key` so that only the account owner (i.e., `predecessor_id == receiver_id`) can fund a gas key on that account. Alternatively, remove the hard `GasKeyBalanceTooHigh` guard and instead allow deletion to burn any balance (with appropriate compute/fee accounting), eliminating the griefable threshold entirely.

### Proof of Concept

```
// Attacker (bob.near) submits this transaction:
Transaction {
    signer_id:   "bob.near",
    receiver_id: "alice.near",   // victim's account
    actions: [
        TransferToGasKey {
            public_key: <alice's gas key public key>,
            deposit:    1_000_000_000_000_000_000_000_001,  // 1 NEAR + 1 yocto
        }
    ]
}

// Alice's pending DeleteKey now fails:
// ActionErrorKind::GasKeyBalanceTooHigh {
//     account_id: "alice.near",
//     public_key: Some(<gas key pk>),
//     balance: <balance now > MAX_BALANCE_TO_BURN>,
// }
```

The runtime path is: `lib.rs` `Action::TransferToGasKey` branch → `action_transfer_to_gas_key` (increments balance, no ownership check) → victim's subsequent `Action::DeleteKey` → `action_delete_key` → `delete_gas_key` → `GasKeyBalanceTooHigh` guard fires. [6](#0-5) [7](#0-6)

pub(crate) fn action_delete_key(
    config: &RuntimeConfig,
    state_update: &mut TrieUpdate,
    account: &mut Account,
    result: &mut ActionResult,

    if let Some(access_key) = access_key {
        if let Some(gas_key_info) = access_key.gas_key_info() {

            public_key: delete_key.public_key.clone().into(),
            account_id: account_id.clone(),
        }
        .into());
    }
    Ok(())
}

pub(crate) fn action_transfer_to_gas_key(
    state_update: &mut TrieUpdate,
    result: &mut ActionResult,
    account_id: &AccountId,
    action: &TransferToGasKeyAction,
) -> Result<(), RuntimeError> {
    let Some(mut access_key) = get_access_key(state_update, account_id, &action.public_key)? else {
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };
    let Some(gas_key_info) = access_key.gas_key_info_mut() else {
        // Key exists but is not a gas key
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };

    gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| {
        RuntimeError::StorageError(StorageError::StorageInconsistentState(
            "gas key balance integer overflow".to_string(),
        ))
    })?;
    set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
    Ok(())
