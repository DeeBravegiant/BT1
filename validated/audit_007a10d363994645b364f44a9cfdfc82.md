Audit Report

## Title
Dual-Oracle Staleness Divergence Enables Yield Arbitrage in RSETHPool Token Deposits — (`contracts/pools/RSETHPool.sol`, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`, `contracts/pools/oracle/InterimRSETHOracle.sol`)

## Summary
`RSETHPool.viewSwapRsETHAmountAndFee` prices token deposits using two independent oracles — a Chainlink wrapper with no `block.timestamp` staleness guard and a manually-updated `InterimRSETHOracle` with no staleness check of any kind. When the Chainlink collateral feed ticks upward while the `InterimRSETHOracle` rsETH/ETH rate is stale-low, any unprivileged caller can deposit collateral via the public `deposit(address,uint256,string)` function and receive more wrsETH than the fair value of their deposit, extracting yield that belongs to existing rsETH holders.

## Finding Description
The swap formula in `RSETHPool.viewSwapRsETHAmountAndFee` (L339–346) is:

```solidity
uint256 rsETHToETHrate = getRate();                                    // InterimRSETHOracle
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // ChainlinkOracleForRSETHPoolCollateral
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

**Oracle 1 — `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L26–37):** Validates only round-completeness (`answeredInRound < roundID`), zero-timestamp (`timestamp == 0`), and non-positive price (`ethPrice <= 0`). There is no `block.timestamp - timestamp > MAX_STALENESS` check. A Chainlink price up to the full heartbeat period old (e.g., 24 h for some feeds) is accepted without revert.

**Oracle 2 — `InterimRSETHOracle.getRate()` (L49–51):** Returns the storage variable `rate` unconditionally. The rate is set exclusively by a `MANAGER_ROLE` holder via `setRate`. There is no `lastUpdated` timestamp, no maximum age check, and no on-chain freshness enforcement of any kind.

**Exploit path:**
1. The `InterimRSETHOracle` rate has not been updated by the manager since the last update cycle (routine maintenance gap, off-hours, or network delay). The stored rate is stale-low relative to the true accrued rsETH/ETH rate.
2. A Chainlink deviation-triggered update moves the collateral token price (e.g., wstETH/ETH) upward.
3. The attacker calls `deposit(token, amount, referralId)` — a public, permissionless function gated only by `whenNotPaused` and `onlySupportedToken`.
4. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` with a numerator that reflects the fresh Chainlink price and a denominator that reflects the stale-low rsETH rate.
5. The attacker receives more wrsETH than the fair-market value of their collateral. The excess wrsETH represents a claim on protocol assets exceeding the deposited value.

**Existing checks are insufficient:** The `answeredInRound < roundID` check in `ChainlinkOracleForRSETHPoolCollateral` only detects an incomplete round, not a stale-but-completed round. `InterimRSETHOracle` has zero freshness enforcement. Neither oracle cross-validates freshness against the other.

## Impact Explanation
The rsETH/ETH rate in `InterimRSETHOracle` is a monotonically increasing value representing accumulated staking yield. When it is stale-low, the denominator of the swap formula is artificially depressed, causing the attacker to receive wrsETH at a discount. The excess wrsETH is a claim on protocol assets that exceeds the deposited collateral's fair value — this is a direct extraction of yield that belongs to existing rsETH holders.

**Impact: High — Theft of unclaimed yield.** The attack is repeatable on every block during the staleness window, and the magnitude scales linearly with deposit size and the degree of oracle divergence.

## Likelihood Explanation
- `InterimRSETHOracle` is explicitly designed as a manually-updated interim oracle; any gap between manager update cycles (routine maintenance, off-hours, network congestion) creates an exploitable window.
- `ChainlinkOracleForRSETHPoolCollateral` has no timestamp staleness guard, so a deviation-triggered Chainlink update that moves the collateral price while the rsETH oracle is stale is sufficient to open the window.
- No privileged role is required by the attacker; `deposit(address,uint256,string)` is public and permissionless beyond `whenNotPaused` and `onlySupportedToken`.
- The attacker only needs to monitor two on-chain storage values (`InterimRSETHOracle.rate` and the Chainlink feed) and submit a transaction when the ratio is favorable — a straightforward on-chain monitoring strategy requiring no special access.

## Recommendation
1. **Add a `block.timestamp` staleness check to `ChainlinkOracleForRSETHPoolCollateral.getRate()`:**
   ```solidity
   uint256 public constant MAX_STALENESS = 3600;
   if (block.timestamp - timestamp > MAX_STALENESS) revert StalePrice();
   ```
2. **Add a `lastUpdated` timestamp and `maxStaleness` guard to `InterimRSETHOracle`:**
   ```solidity
   uint256 public lastUpdated;
   uint256 public maxStaleness;
   function getRate() external view returns (uint256) {
       if (block.timestamp - lastUpdated > maxStaleness) revert StaleRate();
       return rate;
   }
   ```
3. **Long-term:** Replace `InterimRSETHOracle` with a fully on-chain, automatically-updating oracle (e.g., backed by `LRTOracle.rsETHPrice()`) that does not depend on manual manager intervention and can be validated for freshness without off-chain coordination.

## Proof of Concept
Minimal reproducible steps using mock contracts (no fork required):

1. Deploy `MockChainlinkFeed` (wstETH/ETH) returning `P0 = 1.15e18`, `answeredInRound == roundID`, `timestamp = block.timestamp`.
2. Deploy `ChainlinkOracleForRSETHPoolCollateral` wrapping `MockChainlinkFeed`.
3. Deploy `InterimRSETHOracle` with `initRate = 1.05e18`.
4. Deploy `RSETHPool`; set `supportedTokenOracle[wstETH] = ChainlinkOracleForRSETHPoolCollateral`, `rsETHOracle = InterimRSETHOracle`. Fund pool with wrsETH.
5. **Baseline deposit:** `rsETHAmount_fair = 1e18 * 1.15e18 / 1.05e18 ≈ 1.0952e18` rsETH per 1 wstETH.
6. Simulate staleness: do **not** call `InterimRSETHOracle.setRate()` (rate stays at `1.05e18`). Update `MockChainlinkFeed` to `P1 = 1.16e18` (deviation tick). True rsETH/ETH has accrued to `1.06e18` but oracle is not updated.
7. **Exploit deposit:** `rsETHAmount_exploit = 1e18 * 1.16e18 / 1.05e18 ≈ 1.1047e18` rsETH per 1 wstETH.
8. **Fair amount at true rates:** `1e18 * 1.16e18 / 1.06e18 ≈ 1.0943e18` rsETH per 1 wstETH.
9. **Excess:** `≈ 0.0104e18` rsETH per 1 wstETH (~1% yield theft per deposit).
10. Assert `rsETHAmount_exploit > rsETHAmount_fair` ✓ and `excess * true_rsETH_rate > 0` ✓.

The test is reproducible on unmodified contracts with mock Chainlink feeds and the deployed `InterimRSETHOracle`. No admin compromise is required — the attacker only calls the public `deposit(token, amount, referralId)` function.
