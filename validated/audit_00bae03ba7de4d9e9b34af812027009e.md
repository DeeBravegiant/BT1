### Title
Proposal Deposits Permanently Locked When `do_update` Clears All Competing Proposals - (`File: crates/contract/src/update.rs`)

### Summary

When `propose_update` is called, each proposer must attach a NEAR deposit to cover storage staking for their proposal. When any update reaches threshold and `do_update` executes, it clears **all** pending proposals and votes — but never refunds the deposits paid by non-winning proposers. Because `UpdateEntry` does not record the proposer's account ID or deposit amount, there is no mechanism to return these funds. The deposits are permanently absorbed into the contract's balance.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a deposit calculated by `ProposedUpdates::required_deposit(&update)`, which is `storage_byte_cost() × bytes_used(update)`. For a full contract binary (~1.5 MB), this is on the order of tens of NEAR tokens. Only the excess above the required amount is refunded to the proposer at call time; the required portion is retained by the contract.

```rust
// lib.rs:1308-1331
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// ...
let id = self.proposed_updates.propose(update);
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

The `UpdateEntry` struct stores only the update payload and `bytes_used` — **not** the proposer's account ID or the deposit amount:

```rust
// update.rs:132-135
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
```

When `do_update` executes, it clears all entries and votes with no refund logic:

```rust
// update.rs:195-200
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ... no refund to non-winning proposers
```

The freed storage staking returns to the contract's general balance, but the original proposers have no way to reclaim their deposits. There is no function in the contract that returns proposal deposits to proposers after `do_update` runs.

### Impact Explanation

Every participant who proposed a competing update loses their storage deposit permanently. For a contract code proposal, the required deposit is proportional to the WASM binary size (up to ~1.5 MB), which translates to tens of NEAR tokens per proposer. This is a direct, permanent loss of funds for MPC participants who submitted non-winning proposals. The contract's balance is silently enriched by the sum of all non-winning deposits with no accounting trail and no recovery path.

This matches the allowed impact: **Medium — balance/accounting invariant broken in production safety flow without requiring DoS or operator misconfiguration.**

### Likelihood Explanation

The scenario is triggered by normal protocol operation: multiple participants independently propose different updates (e.g., different code versions or config changes), which is an expected governance pattern. When one proposal reaches threshold, `do_update` fires and all competing proposals are wiped. This is not an edge case — the README and tests explicitly show multiple simultaneous proposals as a supported workflow. The loss is automatic and requires no adversarial action.

### Recommendation

Store the proposer's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,      // add
    pub(super) deposit: NearToken,       // add
}
```

In `do_update`, before clearing entries, iterate over all non-winning entries and schedule `Promise::new(entry.proposer).transfer(entry.deposit)` for each. This mirrors the refund pattern already used in `require_deposit` and `submit_participant_info`.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. Contract stores `UpdateEntry { update: ..., bytes_used: 1_000_000 }` — proposer identity and deposit are discarded.
2. Participant B calls `propose_update` with a different 1 MB binary, attaching ~10 NEAR deposit. Same: deposit is retained, identity discarded.
3. Threshold participants vote for B's proposal. `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` (removing A's entry) and `self.vote_by_participant.clear()`. No refund is issued to A.
5. A's ~10 NEAR is permanently absorbed into the contract balance. A has no recourse.

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
