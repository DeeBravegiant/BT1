### Title
Native Script `RequireMOf` Threshold Bypass via Negative Integer Deserialization — (File: `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs`, `eras/shelley/impl/src/Cardano/Ledger/Shelley/Scripts.hs`)

---

### Summary

The CBOR decoders for native scripts (`MultiSig` in Shelley, `Timelock` in Allegra through Conway) accept a negative signed integer for the `m` threshold in `RequireMOf` without any non-negativity validation. The script evaluators (`evalMultiSig`, `evalTimelock`) treat any `m ≤ 0` as an unconditional pass, analogous to the biometric bypass in the external report where `evaluatePolicy` was forced to return `True` regardless of actual authentication. An attacker can craft and publish a CBOR-encoded native script with a negative threshold that always validates, bypassing all signature requirements for any UTxO locked by that script hash.

---

### Finding Description

**Root cause — data type accepts negative `Int`:**

`TimelockRaw` in Allegra (and all subsequent eras) stores the threshold as a signed `Int`:

```haskell
| TimelockMOf !Int !(StrictSeq (Timelock era))
| -- Note that the Int may be negative in which case (TimelockMOf (-2) [..]) is always True
``` [1](#0-0) 

The inline comment is the developers' own acknowledgment that a negative `m` makes the script unconditionally true.

`MultiSigRaw` in Shelley has the same type:

```haskell
| MultiSigMOf !Int !(StrictSeq (MultiSig era))
``` [2](#0-1) 

**Root cause — CBOR decoder performs no non-negativity check:**

The Allegra `Annotator` decoder for `TimelockRaw` tag 3 uses a plain `Ann From` for the `Int` field:

```haskell
decRaw 3 = Ann (SumD TimelockMOf) <*! Ann From <*! D (sequence <$> decCBOR)
``` [3](#0-2) 

The non-`Annotator` decoder in the same file also uses a bare `decCBOR` for `requiredCount`:

```haskell
3 -> do
  requiredCount <- decCBOR
  timelocks <- decCBOR
  pure (3, TimelockMOf requiredCount timelocks)
``` [4](#0-3) 

The Shelley `MultiSigRaw` decoder is identical in structure:

```haskell
3 -> do
  m <- decCBOR
  multiSigs <- sequence <$> decCBOR
  pure (3, MultiSigMOf m <$> multiSigs)
``` [5](#0-4) 

Neither decoder calls `decodeNatural` (which enforces `n ≥ 0`) or any equivalent guard. Compare with the `MaryValue` decoder, which explicitly rejects negative amounts with a bounds check — no such check exists here. [6](#0-5) 

**Root cause — evaluator short-circuits on `n ≤ 0`:**

`evalTimelock`'s inner helper:

```haskell
isValidMOf n SSeq.Empty = n <= 0
isValidMOf n (ts SSeq.:<| tss) =
  n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
``` [7](#0-6) 

`evalMultiSig` is identical:

```haskell
isValidMOf n StrictSeq.Empty = n <= 0
isValidMOf n (msig StrictSeq.:<| msigs) =
  n <= 0 || if go msig then isValidMOf (n - 1) msigs else isValidMOf n msigs
``` [8](#0-7) 

When `m` is any negative integer (e.g., `-1`), the very first call `isValidMOf (-1) xs` evaluates `(-1) ≤ 0 = True` and returns immediately, regardless of the sub-script list or the transaction's witness set.

**UTXOW validation does not catch this:**

`validateFailedNativeScripts` (Shelley) and `validateFailedBabbageScripts` (Babbage/Conway) only check whether `validateNativeScript tx script` returns `True`. A script with negative `m` always returns `True`, so it passes this check unconditionally. [9](#0-8) [10](#0-9) 

---

### Impact Explanation

An attacker who controls the script-authoring step (e.g., a DeFi protocol operator, a multisig wallet coordinator, or a governance guardrail author) can publish a native script whose CBOR encodes `m = -1` while the human-readable representation shows a plausible positive threshold. Any UTxO locked by the resulting script hash can be spent by a transaction carrying **zero** key witnesses. This constitutes direct, unauthorized transfer of ADA or native assets — a loss of funds through an invalid ledger state transition that the ledger accepts as valid because the script evaluates to `True`.

The same mechanism applies to governance guardrail scripts (Conway `constitutionScriptHash`): a guardrail script with negative `m` would unconditionally approve every `ParameterChange` or `TreasuryWithdrawals` proposal, enabling unauthorized governance or treasury actions.

---

### Likelihood Explanation

**Low.** The attacker must control the script that a victim locks funds against, requiring either social engineering (presenting a malicious script as legitimate) or a bug in upstream tooling that generates negative thresholds. Legitimate tooling (e.g., `cardano-cli`, `cardano-api`) generates `m ≥ 0` by construction. However, the protocol itself imposes no barrier: the CBOR wire format accepts negative integers for tag-3 scripts, and the ledger rules accept the resulting script as valid. Any party who can supply a serialized script to a victim — including a DeFi protocol, a multisig coordinator, or a governance proposal author — can exploit this.

---

### Recommendation

Replace the bare `decCBOR` / `Ann From` for the threshold field with a non-negative decoder in both `TimelockRaw` and `MultiSigRaw`:

```haskell
-- Allegra TimelockRaw (Annotator path)
decRaw 3 = do
  m <- Ann (D (decodeNatural >>= \n -> pure (fromIntegral n :: Int)))
  Ann (SumD TimelockMOf) <*! m <*! D (sequence <$> decCBOR)

-- Shelley MultiSigRaw
3 -> do
  m <- decCBOR
  when (m < (0 :: Int)) $ fail "RequireMOf threshold must be non-negative"
  multiSigs <- sequence <$> decCBOR
  pure (3, MultiSigMOf m <$> multiSigs)
```

Alternatively, change the field type from `Int` to `Natural` or `Word` in both `MultiSigMOf` and `TimelockMOf`, which would make the type system enforce non-negativity at construction and deserialization time.

### Proof of Concept

1. Construct a CBOR byte string for `RequireMOf (-1) [RequireSignature <keyHash>]`:
   - CBOR array tag 3 (MOf), integer `-1` (CBOR major type 1: `0x20`), followed by a one-element array containing a `RequireSignature` sub-script.
2. Hash the resulting bytes to obtain `scriptHash`.
3. Send ADA to `Addr (ScriptHashObj scriptHash) ...` — this locks funds under the script.
4. Submit a transaction spending that UTxO with **no** `addrTxWits` entries.
5. The ledger calls `validateNativeScript tx script`, which calls `evalTimelock {} vi script`, which calls `isValidMOf (-1) [RequireSignature kh]`. Since `-1 ≤ 0`, it returns `True` immediately.
6. `validateFailedNativeScripts` finds no failing scripts; the transaction is accepted and the funds are transferred without any signature.
