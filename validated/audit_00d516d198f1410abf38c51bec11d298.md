Audit Report

## Title
Stale `rsETHPrice` in `initiateWithdrawal` and `instantWithdrawal` Causes Systematic Yield Loss for Withdrawing Users - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal()` and `instantWithdrawal()` both call `getExpectedAssetAmount()`, which reads the stored `lrtOracle.rsETHPrice()` without first invoking `updateRSETHPrice()`. Because `rsETHPrice` is a manually-updated state variable that can lag behind true protocol value by hours, users who withdraw during a staleness window receive fewer assets than their rsETH is currently worth. For `initiateWithdrawal`, the `_calculatePayoutAmount` min-cap permanently locks in the stale-lower amount, causing the accrued yield to be silently retained by the protocol rather than delivered to the withdrawing user.

## Finding Description

`LRTOracle.rsETHPrice` is a stored state variable written only inside `_updateRsETHPrice()`: [1](#0-0) [2](#0-1) 

`updateRSETHPrice()` is public and callable by anyone, but nothing in the withdrawal path calls it: [3](#0-2) 

`getExpectedAssetAmount()` reads the stored value directly: [4](#0-3) 

Both withdrawal entry points call this without a prior price refresh: [5](#0-4) [6](#0-5) 

**`initiateWithdrawal` exploit path:**
When rewards have accrued since the last `updateRSETHPrice()` call, the stored price is lower than the true rate. `initiateWithdrawal` computes a stale-lower `expectedAssetAmount` and stores it in the `WithdrawalRequest`. When `unlockQueue` later runs with the refreshed price, `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`: [7](#0-6) 

Because `expectedAssetAmount` (stale-lower) < `currentReturn` (fresh-higher), the user is permanently capped at the stale amount. The accrued yield is never delivered.

**`instantWithdrawal` exploit path (Scenario A):**
The user burns rsETH and immediately receives `assetAmountUnlocked` computed from the stale price — fewer assets than the current rate entitles them to, with no recourse. [8](#0-7) 

**Secondary accounting impact:**
`assetsCommitted[asset]` is under-incremented by the stale-lower amount, causing `getAvailableAssetAmount` to return an inflated figure and allowing additional withdrawals to be queued that would otherwise be blocked: [9](#0-8) [10](#0-9) 

## Impact Explanation

**High — Theft of unclaimed yield.**

rsETH is a yield-bearing token: its value increases as EigenLayer rewards accrue. A user's rsETH represents a proportional claim on the protocol's total assets, including all rewards earned up to the moment of withdrawal. When `initiateWithdrawal` uses a stale-lower price, the user's `expectedAssetAmount` is set below the true current value of their rsETH. The `min()` cap in `_calculatePayoutAmount` then permanently locks in this shortfall — the accrued yield that belongs to the withdrawing user is never paid out and instead remains in the protocol, benefiting remaining rsETH holders at the withdrawing user's expense. This is a concrete, repeatable loss of earned yield for every user who withdraws during a staleness window, matching the **High: Theft of unclaimed yield** impact class.

## Likelihood Explanation

`rsETHPrice` is updated by operators on a periodic schedule (not atomically before every withdrawal). Any user may call `updateRSETHPrice()` themselves, but neither `initiateWithdrawal` nor `instantWithdrawal` enforces this. EigenLayer rewards accrue continuously, so the stored price will routinely lag the true rate between operator updates. The staleness window can span hours. No special attacker capability is required — any ordinary withdrawing user is affected. Likelihood is **Medium to High** given the frequency of normal withdrawal activity relative to price-update cadence.

## Recommendation

Call `updateRSETHPrice()` (or an internal equivalent) at the start of both `initiateWithdrawal` and `instantWithdrawal` before reading `lrtOracle.rsETHPrice()`:

```solidity
// At the top of initiateWithdrawal and instantWithdrawal:
ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
```

This ensures `getExpectedAssetAmount` always uses the latest protocol state. If the price update itself triggers a pause (downside-protection path in `_updateRsETHPrice`), the withdrawal will revert cleanly, which is the correct behavior. [11](#0-10) 

## Proof of Concept

1. EigenLayer rewards accrue; `rsETHPrice` stored in `LRTOracle` is `1.00e18` but the true current rate (as would be computed by `_updateRsETHPrice`) is `1.05e18`. No one has called `updateRSETHPrice()`.
2. User calls `initiateWithdrawal(ETH, 100e18, ...)`.
3. `getExpectedAssetAmount` computes `100e18 * 1.00e18 / 1.00e18 = 100 ETH` instead of the correct `105 ETH`.
4. `assetsCommitted[ETH] += 100 ETH` (under-counted by 5 ETH); `getAvailableAssetAmount` returns 5 ETH more than it should.
5. Operator calls `updateRSETHPrice()` (or it is called by anyone), refreshing `rsETHPrice` to `1.05e18`.
6. Operator calls `unlockQueue`; `_calculatePayoutAmount` computes `currentReturn = 100e18 * 1.05e18 / 1e18 = 105 ETH`, then returns `min(100, 105) = 100 ETH`.
7. User receives 100 ETH despite having burned rsETH worth 105 ETH at the current rate. The 5 ETH yield shortfall is permanently retained by the protocol.

**Foundry fork test outline:**
- Fork mainnet; snapshot `rsETHPrice` from `LRTOracle`.
- Simulate reward accrual by advancing time and calling `updateRSETHPrice()` to record the new higher price; then reset `rsETHPrice` to the old value via a storage slot write (or simply skip the update call).
- Call `initiateWithdrawal` as a normal user.
- Call `updateRSETHPrice()` and then `unlockQueue`.
- Assert `withdrawalRequests[requestId].expectedAssetAmount < (rsETHUnstaked * currentRsETHPrice / assetPrice)`, confirming the yield shortfall.
