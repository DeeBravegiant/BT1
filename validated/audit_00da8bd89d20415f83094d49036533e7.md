### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Reward via Unenforced `farmer_reward_address_override` - (File: `chia/farmer/farmer_api.py`)

---

### Summary

A third-party harvester connected to a farmer can set `farmer_reward_address_override` to any puzzle hash it controls in a `NewProofOfSpace` or `RespondSignatures` message. The farmer's fee-quality check (`notify_farmer_reward_taken_by_harvester_as_fee`) is purely advisory — it only emits log warnings on violation and never rejects the override. As a result, `_process_respond_signatures` unconditionally substitutes the harvester-supplied address for the farmer's configured reward address, causing the farmer's 1/8 XCH block reward to be paid to the attacker.

---

### Finding Description

CHIP-22 defines a voluntary convention where a third-party harvester may take the farmer's block reward as a service fee, gated by a probabilistic "fee quality" check. The farmer is expected to enforce this gate.

**Step 1 — Harvester sends override with no valid fee quality:**

In `NewProofOfSpace`, the harvester sets `farmer_reward_address_override` to an address it controls and either omits `fee_info` entirely or supplies an `applied_fee_threshold` that does not pass the quality check.

**Step 2 — Farmer logs but does not reject:**

`FarmerAPI.new_proof_of_space` calls `notify_farmer_reward_taken_by_harvester_as_fee` when `farmer_reward_address_override is not None`:

```python
# chia/farmer/farmer_api.py  line 128-129

    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
```

Inside that function, both the "threshold not met" and "no fee_info provided" branches only call `self.log.warning(...)` — there is no `return`, no exception, and no flag set to block the override:

```python
# chia/farmer/farmer.py  lines 920-934
else:
    self.log.warning(
        f"Invalid fee threshold for challenge '{challenge_str}': ..."
    )
    self.log.warning(
        "Harvester illegitimately took a fee reward that "
        + "did not belong to it or it incorrectly applied the fee convention."
    )
```

Execution continues normally after the call.

**Step 3 — Override is applied unconditionally:**

In `_process_respond_signatures`, the harvester-supplied address replaces the farmer's configured target with no further check:

```python

farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
```

This address is then embedded in `DeclareProofOfSpace.farmer_reward_address` and forwarded to the full node, which creates the farmer coin paying that puzzle hash.

---

### Impact Explanation

Every block the farmer wins while connected to the malicious harvester will pay the 1/8 XCH farmer block reward to the harvester's address instead of the farmer's. This is a direct, permanent diversion of XCH with no recovery path for the farmer. The impact matches the allowed High scope: **unauthorized payout redirection of XCH via bypass of farmer reward authorization**.

---

### Likelihood Explanation

Any harvester that establishes a valid connection to the farmer can exploit this. The farmer-harvester connection is a standard part of the Chia farming setup; third-party harvesters (e.g., GPU-accelerated plot services) are a common deployment pattern explicitly anticipated by CHIP-22. The attacker needs no keys, no admin access, and no cryptographic break — only a TCP connection to the farmer's harvester port.

---

### Recommendation

`notify_farmer_reward_taken_by_harvester_as_fee` must return a boolean indicating whether the override is legitimate, and `new_proof_of_space` must drop the proof (return early) when the check fails. Alternatively, the farmer should expose a configuration flag (`allow_harvester_fee_override: bool`) that defaults to `false`, and only accept `farmer_reward_address_override` when explicitly opted in by the farmer operator.

---

### Proof of Concept

1. Malicious harvester connects to a farmer.
2. On finding a valid proof, it sends `NewProofOfSpace` with:
   - `farmer_reward_address_override = <attacker_puzzle_hash>`
   - `fee_info = None` (or `ProofOfSpaceFeeInfo(applied_fee_threshold=0xFFFFFFFF)` — always fails)
3. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` → logs warning → **returns normally**.
4. Farmer proceeds to `_process_respond_signatures`, sets `farmer_reward_address = attacker_puzzle_hash`.
5. `DeclareProofOfSpace` is sent to the full node with the attacker's address.
6. Full node creates the farmer coin at `attacker_puzzle_hash`; the farmer receives nothing.

```python
    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
```
