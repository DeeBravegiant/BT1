Audit Report

## Title
No Time-Based Staleness Check on Chainlink Price Feeds Allows Stale Prices to Be Consumed - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` return value, performing no time-based freshness validation. If a Chainlink feed stops updating, the protocol will consume the last known price indefinitely, causing incorrect rsETH minting and redemption amounts for all depositors and withdrawers. A secondary instance in `ChainlinkOracleForRSETHPoolCollateral.getRate()` relies solely on the deprecated `answeredInRound < roundID` check, which is always false on OCR-based feeds.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at L52, `latestRoundData()` is called with a destructuring pattern that drops `updatedAt` entirely:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No check of the form `block.timestamp - updatedAt > MAX_STALENESS` exists anywhere in the function or its callers. This price propagates through `LRTOracle.getAssetPrice()`: [2](#0-1) 

Into `LRTDepositPool.getRsETHAmountToMint()`, which determines how many rsETH tokens a depositor receives: [3](#0-2) 

And into `LRTWithdrawalManager._createUnlockParams()`, which determines withdrawal payout amounts: [4](#0-3) 

The secondary instance in `ChainlinkOracleForRSETHPoolCollateral.getRate()` uses only `answeredInRound < roundID`, which is structurally always false on OCR-based Chainlink feeds (where `answeredInRound == roundID` by design), and performs no `block.timestamp - updatedAt` check: [5](#0-4) 

The `pricePercentageLimit` circuit breaker in `_updateRsETHPrice()` does not protect against stale prices at deposit/withdrawal time â€” it only fires when `updateRSETHPrice()` is called and the computed rsETH price deviates beyond the threshold: [6](#0-5) 

## Impact Explanation
**Primary â€” Low (Contract fails to deliver promised returns):** Any depositor calling `depositAsset()` or `depositETH()` during a stale-feed period receives rsETH computed at the stale price rather than the true market price. If the stale price is inflated, the depositor receives excess rsETH, diluting existing holders. If deflated, the depositor receives fewer rsETH tokens than owed. Withdrawers are symmetrically affected via `_createUnlockParams()`. This directly matches the allowed Low impact: "Contract fails to deliver promised returns."

**Secondary â€” Medium (Temporary freezing of funds):** If the stale price diverges sufficiently from `highestRsethPrice` to exceed `pricePercentageLimit`, the next call to `updateRSETHPrice()` will trigger the auto-pause of `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle`, freezing all deposits and withdrawals until an admin manually unpauses. This matches the allowed Medium impact: "Temporary freezing of funds."

## Likelihood Explanation
Chainlink feeds can go stale during L2 sequencer downtime, extreme network congestion, or oracle node failures. The protocol is deployed on Ethereum mainnet and potentially other chains. No privileged access is required â€” any unprivileged depositor or withdrawer interacting with the protocol during a stale-feed period triggers the vulnerable path. The exposure window is unbounded: the stale price persists until the feed resumes. The `answeredInRound < roundID` check in `ChainlinkOracleForRSETHPoolCollateral` provides no real protection on OCR feeds, making that path equally exposed.

## Recommendation
Add a per-asset configurable staleness threshold in `ChainlinkPriceOracle` and validate `updatedAt`:

```solidity
mapping(address asset => uint256 maxStaleness) public maxStalenessPerAsset;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
    uint256 staleness = maxStalenessPerAsset[asset];
    if (staleness > 0 && block.timestamp - updatedAt > staleness) revert StalePrice();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Replace the `answeredInRound < roundID` check in `ChainlinkOracleForRSETHPoolCollateral.getRate()` with a time-based check using the appropriate heartbeat for the specific feed and chain (e.g., `if (block.timestamp - timestamp > HEARTBEAT) revert StalePrice();`).

## Proof of Concept
1. Chainlink's stETH/ETH feed stops updating (e.g., L2 sequencer downtime). Last reported price: `1.05e18`. True market price drops to `0.98e18`.
2. Attacker/depositor calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `_beforeDeposit` â†’ `getRsETHAmountToMint(stETH, 100e18)` â†’ `lrtOracle.getAssetPrice(stETH)` â†’ `ChainlinkPriceOracle.getAssetPrice(stETH)` â†’ `latestRoundData()` returns stale `price = 1.05e18`, `updatedAt` is never checked.
4. `rsethAmountToMint = (100e18 * 1.05e18) / rsETHPrice` â€” depositor receives rsETH at the inflated stale price.
5. No revert at any step. Existing holders are diluted.
6. Foundry fork test: fork mainnet at a block after a simulated feed freeze (use `vm.mockCall` to return a stale `updatedAt = block.timestamp - 2 days`), call `depositAsset`, assert `rsethAmountToMint` is computed at the stale price and no revert occurs.
