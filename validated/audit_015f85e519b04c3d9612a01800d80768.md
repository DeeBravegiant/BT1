### Title
Cross-Chain `deployToken` Signature Replay — MPC Metadata Signature Accepted on Any Chain — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `aptos/sources/bridge_types.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `deployToken` / `deploy_token` signature digest on every destination chain (EVM, Starknet, Aptos, Solana) encodes only `[PayloadType::Metadata, token, name, symbol, decimals]` — **no chain ID is included**. The `finTransfer` / `fin_transfer` path on every chain correctly binds the signed hash to the destination chain by interleaving `omniBridgeChainId` / `chain_id` / `SOLANA_OMNI_BRIDGE_CHAIN_ID` into the Borsh payload. The `deployToken` path has no equivalent binding. Because all chains share the same NEAR MPC-derived signer address, a single MPC-signed metadata payload is cryptographically valid on every chain simultaneously. An unprivileged attacker who observes a legitimate `deployToken` transaction on chain A can replay the identical signature on chain B, deploying the same bridge token there without any MPC authorization for chain B.

---

---

### Impact Explanation

The bridge is deployed on multiple EVM chains simultaneously (Eth = 0, Sol = 2, Arb = 3, Base = 4, Bnb = 5, Pol = 8, HyperEvm = 9, Abs = 11, Strk = 10, Aptos = 13 per `ChainKind`). [12](#0-11) 

All share the same `nearBridgeDerivedAddress` / `omni_bridge_derived_address`. A single MPC-signed `MetadataPayload` for token T on chain A is byte-identical to the payload that would be signed for chain B, because neither contains a chain discriminant. An attacker who observes the signature on chain A can submit it to chain B's `deployToken` entry point. The signature check passes; the token is deployed on chain B.

Once deployed on chain B, users may call `initTransfer` with the token. Their assets are burned or locked on chain B. If the NEAR side has not registered this token for chain B (because no authorized deployment was intended), the inbound transfer cannot be finalized on NEAR, and the locked/burned assets are permanently unrecoverable. This satisfies the **Critical** irreversible fund-lock impact, and at minimum the **High** "insufficiently-bound proof bypasses execution gate" impact.

---

### Likelihood Explanation

The attack requires zero privilege. The attacker only needs to:
1. Watch any chain for a legitimate `DeployToken` event (public, on-chain).
2. Extract the `signatureData` from the originating transaction's calldata.
3. Submit the same `(signatureData, metadata)` tuple to `deployToken` on any other chain where the token has not yet been deployed.

The `ERR_TOKEN_EXIST` / `E_TOKEN_ALREADY_DEPLOYED` guard only prevents re-deployment on the *same* chain; it provides no cross-chain protection. [13](#0-12) 

Every new token deployment on any chain is an immediately exploitable trigger. The bridge supports many chains and many tokens, so opportunities are frequent.

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` Borsh digest, mirroring the pattern already used in `finTransfer`. On EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent change to `MetadataPayload.to_borsh()` in Starknet (`bridge_types.cairo`), `metadata_to_borsh()` in Aptos (`bridge_types.move`), and `DeployTokenPayload::serialize_for_near()` in Solana (`deploy_token.rs`). The NEAR MPC signing path must also include the target chain ID when constructing the metadata payload hash so that signatures are chain-specific.

---

### Proof of Concept

1. Deploy the bridge on two EVM chains sharing the same `nearBridgeDerivedAddress` (e.g., Ethereum `omniBridgeChainId = 0` and Arbitrum `omniBridgeChainId = 3`).
2. The MPC signs a `MetadataPayload` for token `"near:usdc.near"` intended for Ethereum. A relayer submits `deployToken(sig, payload)` on Ethereum — succeeds, token deployed at address `T_eth`.
3. Attacker copies `sig` and `payload` from the Ethereum transaction calldata.
4. Attacker calls `deployToken(sig, payload)` on Arbitrum. The Borsh encoding is identical (no chain ID), `ECDSA.recover` returns `nearBridgeDerivedAddress`, the `ERR_TOKEN_EXIST` check passes (token not yet on Arbitrum), and a new bridge token `T_arb` is deployed.
5. A user, believing `T_arb` is a legitimate bridge token, calls `initTransfer` on Arbitrum, burning their USDC. The NEAR side has no record of `T_arb` for Arbitrum and cannot finalize the transfer. The user's funds are permanently locked.

```rust
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
```
