Audit Report

## Title
Token Decimal Precision Not Normalized in `viewSwapRsETHAmountAndFee` Causes Drastically Undercalculated rsETH Minted for Non-18-Decimal Deposits - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All five L2 pool contracts compute the rsETH amount to mint for ERC-20 token deposits using `amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18-decimal precision. Because `ChainlinkOracleForRSETHPoolCollateral.getRate()` always returns a 1e18-precision rate, any token with fewer than 18 decimals produces an rsETH output that is `10^(18 - tokenDecimals)` times smaller than correct. A depositor of a 6-decimal token (e.g., USDC) receives approximately 1e12 times less rsETH than owed, constituting a near-total permanent loss of deposited principal.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee`, the token-deposit branch computes:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink feed answer to 1e18 regardless of the feed's native decimals:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [2](#0-1) 

So `tokenToETHRate` is always 1e18-precision, but `amountAfterFee` is in the token's native decimals. For a 6-decimal token, the formula silently treats `1_000e6` (1000 USDC) as if it were `1_000e6` in 1e18 scale, producing `≈380_952` rsETH units instead of `≈3.8e17`. The identical pattern is present in all five pool contracts:

- `RSETHPoolNoWrapper.sol` line 308–311 [3](#0-2) 
- `RSETHPoolV3ExternalBridge.sol` line 449–452 [4](#0-3) 
- `RSETHPoolV3WithNativeChainBridge.sol` line 367–370 [5](#0-4) 
- `RSETHPool.sol` line 343–346 [6](#0-5) 

The `addSupportedToken` function accepts any ERC-20 and oracle pair with no decimal check — the only guard is that `oracle.getRate() != 0`: [7](#0-6) 

No existing check in the deposit or view path compensates for token decimal differences. The `decimals()` function is never called anywhere in the pool contracts. [8](#0-7) 

## Impact Explanation
**Critical — Direct theft of user funds.** A user depositing a non-18-decimal token transfers their full token balance to the pool (which is bridged to L1 and permanently absorbed), but receives rsETH/wrsETH worth `10^(18 - tokenDecimals)` times less than fair value. For a 6-decimal token the shortfall is a factor of 1e12 — effectively 100% loss of deposited principal. The user holds insufficient rsETH to redeem anything meaningful and has no recovery mechanism. This matches the Critical impact class: direct theft of user funds in motion. [1](#0-0) 

## Likelihood Explanation
The currently deployed supported token is wstETH (18 decimals), for which the formula is coincidentally correct. However, `addSupportedToken` is a standard governance function with no decimal guard, and the protocol's architecture explicitly anticipates future token expansion. Any legitimate governance action to add a common asset with fewer than 18 decimals (USDC, USDT, WBTC) immediately activates the bug for every subsequent depositor of that token. The harm is triggered by unprivileged users calling the public `deposit()` function — no attacker capability is required beyond making a normal deposit after a new token is listed. [7](#0-6) 

## Recommendation
Normalize `amountAfterFee` to 18-decimal precision before applying the rate ratio in all five pool contracts:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 amountNormalized = amountAfterFee * 1e18 / (10 ** tokenDecimals);
rsETHAmount = amountNormalized * tokenToETHRate / rsETHToETHrate;
```

Additionally, consider adding a decimal check in `addSupportedToken` to revert or warn if the token has non-18 decimals until the fix is deployed. [2](#0-1) 

## Proof of Concept
1. Admin calls `RSETHPoolV3.addSupportedToken(usdc, chainlinkUSDCOracle)` — USDC has 6 decimals; oracle returns `4e14` (USDC/ETH at $2500/ETH, normalized to 1e18).
2. User calls `RSETHPoolV3.deposit(usdc, 1_000e6, "")` — depositing 1000 USDC (fair rsETH value ≈ `3.8e17`).
3. `viewSwapRsETHAmountAndFee` computes:
   - `amountAfterFee ≈ 1_000e6` (assuming 0 fee for simplicity)
   - `tokenToETHRate = 4e14`
   - `rsETHToETHrate = 1.05e18`
   - `rsETHAmount = 1_000e6 * 4e14 / 1.05e18 ≈ 380_952`
4. User receives `380_952` wrsETH (≈ `3.8e-13` rsETH) instead of `≈ 3.8e17` wrsETH — a loss of ~1e12×.
5. The 1000 USDC is bridged to L1 and absorbed. The user cannot redeem meaningful value.

A Foundry fork test can reproduce this by: (a) deploying or forking a pool with USDC added as a supported token, (b) calling `viewSwapRsETHAmountAndFee(1_000e6, usdc)`, and (c) asserting the returned `rsETHAmount` is ~`3.8e17` — the test will fail, returning `~380_952` instead. [9](#0-8)
