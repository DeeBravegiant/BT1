### Title
Hardcoded `Fee(1)` in `convert_consensus_l1_handler_to_internal_l1_handler` Bypasses `paid_fee_on_l1` Fee Check for All Validator-Side L1 Handler Executions — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`TransactionConverter::convert_consensus_l1_handler_to_internal_l1_handler` hardcodes `Fee(1)` as the `paid_fee_on_l1` argument when constructing the executable `L1HandlerTransaction` from a consensus message. The blockifier's `execute_raw` for L1 handler transactions contains a guard that rejects any transaction whose `paid_fee_on_l1 == Fee(0)`. Because `Fee(1)` is always non-zero, this guard is permanently bypassed on every validator node, regardless of the fee that was actually paid on L1. This is the direct Sequencer analog of the Gondi H-11 bug: a hardcoded sentinel value replaces the real protocol-level value, silently disabling the only in-execution fee enforcement for L1 handler transactions.

---

### Finding Description

**Root cause — hardcoded fee in the consensus-to-executable converter:**

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),   // ← hardcoded; real paid_fee_on_l1 is discarded
    )?)
}
``` [1](#0-0) 

This function is called from `convert_consensus_tx_to_internal_consensus_tx` whenever a `ConsensusTransaction::L1Handler` arrives at a validator node. [2](#0-1) 

**The guard that is bypassed — blockifier `execute_raw`:**

After a successful execution the blockifier checks:

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee { paid_fee, actual_fee: receipt.fee },
    )));
}
``` [3](#0-2) 

Because the converter always supplies `Fee(1)`, the condition `paid_fee == Fee(0)` is **never true** on any validator node. The check is structurally dead code for the entire consensus validation path.

**Contrast with the proposer path:**

The proposer obtains L1 handler transactions from the L1 events provider, which constructs them with the real fee scraped from Ethereum:

```rust
// crates/apollo_l1_events_types/src/lib.rs
L1Event::LogMessageToL2 { tx, fee, block_timestamp, .. } => {
    let tx = L1HandlerTransaction::create(tx, chain_id, fee)?;  // real fee
    ...
}
``` [4](#0-3) 

The proposer therefore executes with the real `paid_fee_on_l1`; the validator always executes with `Fee(1)`. The two sides apply different fee semantics to the same transaction.

**`paid_fee_on_l1` is not part of the transaction hash:**

```rust
// crates/starknet_api/src/executable_transaction.rs
pub fn create(raw_tx: ..., chain_id: &ChainId, paid_fee_on_l1: Fee) -> ... {
    let tx_hash = raw_tx.calculate_transaction_hash(chain_id, &raw_tx.version)?;
    Ok(Self { tx: raw_tx, tx_hash, paid_fee_on_l1 })
}
``` [5](#0-4) 

The hash is computed from `raw_tx` only; `paid_fee_on_l1` is stored separately. This means the discrepancy is invisible to any hash-based consistency check.

---

### Impact Explanation

**Execution divergence between proposer and validator.** A malicious proposer can include an L1 handler transaction whose actual `paid_fee_on_l1` is `Fee(0)`. On the proposer side, `execute_raw` returns `TransactionFeeError::InsufficientFee` and the transaction is treated as failed/reverted. On every validator side, the same transaction is executed with `Fee(1)`, the guard passes, and the transaction is treated as succeeded. The two sides produce different receipts, different state diffs, and therefore different block hashes — breaking consensus or, depending on which side's result is authoritative, corrupting committed state.

**Fee enforcement is permanently disabled on validators.** Even for honest proposals, the validator never verifies that the fee actually paid on L1 meets the minimum. Any future tightening of the fee check (e.g., checking `paid_fee >= actual_fee`) would be silently bypassed on all validator nodes because the converter always supplies `Fee(1)`.

**Affected impact category:** "Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input" and "Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."

### Likelihood Explanation

The `ConsensusTransaction::L1Handler` variant carries only `transaction::L1HandlerTransaction`, which has no `paid_fee_on_l1` field. Every validator node that processes an L1 handler transaction through the consensus path hits this code path unconditionally. No special attacker capability is required beyond being a proposer (or colluding with one) who can include an L1 handler transaction with a zero or insufficient fee. The TODO comment confirms the developers are aware the value is wrong and intend to fix it.

---

### Recommendation

Replace the hardcoded `Fee(1)` with the actual fee. The validator should look up the `paid_fee_on_l1` for the transaction from its own L1 events provider before constructing the executable transaction, mirroring the proposer path:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    // Fetch the real paid_fee_on_l1 from the L1 events provider using tx_hash.
    let paid_fee = self.l1_events_provider.get_paid_fee(tx_hash)?;
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        paid_fee,
    )?)
}
```

If the L1 events provider is not available in this context, the `paid_fee_on_l1` should be propagated through the `ConsensusTransaction::L1Handler` variant itself so that the converter can use it directly.

---

### Proof of Concept

1. Attacker controls a proposer node.
2. Attacker constructs an `L1HandlerTransaction` with `paid_fee_on_l1 = Fee(0)` (e.g., by directly crafting the consensus message, bypassing the L1 events provider).
3. Proposer includes this transaction in a proposal. On the proposer's own execution, `execute_raw` returns `InsufficientFee` — the transaction is reverted.
4. Validator receives the proposal. `convert_consensus_tx_to_internal_consensus_tx` calls `convert_consensus_l1_handler_to_internal_l1_handler`, which sets `paid_fee_on_l1 = Fee(1)`.
5. Validator's `execute_raw` runs: `paid_fee == Fee(1) != Fee(0)` → guard passes → transaction succeeds.
6. Proposer receipt: `execution_status = Reverted`. Validator receipt: `execution_status = Succeeded`.
7. State diffs diverge; block hashes diverge; consensus is broken or the validator commits incorrect state. [1](#0-0) [6](#0-5)
