Missing `fee_proposal_fri` Bounds Validation When `fee_actual` Is `None` Allows Arbitrary Fee Injection During Startup/Transition Window â€” (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

In `is_proposal_init_valid`, the `fee_proposal_fri` bounds check is gated on `fee_actual` being `Some`. When `fee_actual` is `None` â€” which occurs for the first `window_size` (10) blocks of the chain and during the entire V0_14_3 protocol-upgrade transition window â€” the validator accepts any proposer-supplied `fee_proposal_fri` value without restriction. A malicious proposer can inject an extreme value (e.g., `0` or `u128::MAX`) that honest validators will accept, commit to the block header, and feed into the `fee_proposals_window`. This corrupts `fee_actual` for subsequent blocks and, through `calculate_next_l2_gas_price`, produces wrong on-chain gas prices with direct economic impact.

---

### Finding Description

`is_proposal_init_valid` in `validate_proposal.rs` validates the proposer-supplied `fee_proposal_fri` field of `ProposalInit` against a geometric band around `fee_actual`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
``` [1](#0-0) 

The `if let` pattern silently drops the entire check when `fee_actual` is `None`. `fee_actual` is `None` in two production-reachable situations:

1. **Genesis / startup**: `compute_fee_actual` returns `None` whenever `height < window_size` (currently 10), because `height.0.checked_sub(window_size)` underflows. [2](#0-1) 

2. **V0_14_3 protocol-upgrade transition**: `compute_fee_actual` also returns `None` if any block in the sliding window carries `fee_proposal_fri = None` (pre-V0_14_3 blocks). During the 10-block transition window, every height in `[upgrade_height, upgrade_height + window_size)` will have at least one pre-V0_14_3 predecessor in the window, so `fee_actual` stays `None` for the entire transition. [3](#0-2) 

The proposer's own fallback when `fee_actual` is `None` is to freeze at `self.l2_gas_price`:

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
``` [4](#0-3) 

But the validator enforces **no corresponding constraint**. A malicious proposer can deviate from `l2_gas_price` by any amount and the validator will not reject the proposal.

The accepted `fee_proposal_fri` is then:

- Written verbatim into `BlockHeaderWithoutHash.fee_proposal_fri` and committed to state-sync: [5](#0-4) 

- Recorded into `fee_proposals_window` via `record_fee_proposal`, which feeds `compute_fee_actual` for all future heights: [6](#0-5) 

- Hashed into the `ProposalCommitment` via `proposal_commitment_from(partial_block_hash, fee_proposal_fri)`, so honest validators compute the same commitment as the malicious proposer and vote for it: [7](#0-6) 

---

### Impact Explanation

`fee_proposals_window` is the sole input to `compute_fee_actual`, which is the sole input to `calculate_next_l2_gas_price`. Corrupting `fee_proposals_window` entries during the unguarded window directly corrupts the on-chain L2 gas price for every subsequent block.

- Setting `fee_proposal_fri = u128::MAX` for a majority of the 10-block window drives `fee_actual` to `u128::MAX`, making all L2 transactions prohibitively expensive (economic denial-of-service).
- Setting `fee_proposal_fri = 0` drives `fee_actual` toward zero, making gas effectively free and enabling spam/resource exhaustion.

Because `fee_proposal_fri` is bound into the `ProposalCommitment` hash, honest validators independently reproduce the same commitment from the proposer-supplied value and vote for the block â€” the corrupted value is therefore finalized on-chain without any node detecting the manipulation.

This matches the allowed impact: **Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

---

### Likelihood Explanation

- **Trigger is unprivileged**: any node that wins the proposer lottery during the unguarded window can execute the attack. No special key or admin role is required.
- **Window is bounded but real**: the genesis window (blocks 0â€“9) is a one-time opportunity; the V0_14_3 transition window recurs on every protocol upgrade and lasts exactly `window_size` (10) blocks.
- **Median resistance limits single-block impact**: with `window_size = 10`, a single corrupted entry shifts the median only if it is in the top or bottom 5. Controlling â‰¥ 6 of the 10 window blocks is sufficient to fully control `fee_actual`. Controlling even 1â€“2 blocks shifts the median measurably.
- **No existing guard**: the comment explicitly acknowledges the bypass ("bounds are not enforced"), confirming no secondary check exists.

---

### Recommendation

When `fee_actual` is `None`, the validator should enforce that `fee_proposal_fri` equals the locally-computed fallback value (`l2_gas_price`), rather than accepting any value. Concretely, replace the silent `if let` bypass with an explicit check:

```rust
match (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri) {
    (Some(fee_actual), Some(fee_proposal)) => {
        let (lower_bound, upper_bound) = fee_proposal_bounds(fee_actual, margin_ppt);
        if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
            return Err(InvalidProposalInit(...));
        }
    }
    (None, Some(fee_proposal)) => {
        // Fallback: proposer must freeze at l2_gas_price.
        if fee_proposal != proposal_init_validation.l2_gas_price_fri {
            return Err(InvalidProposalInit(..., "fee_proposal must equal l2_gas_price when fee_actual is unavailable"));
        }
    }
    _ => {}
}
```

`ProposalInitValidation` already carries `l2_gas_price_fri`, so no new data needs to be threaded through. [8](#0-7)

### Proof of Concept

**Scenario: V0_14_3 transition window**

1. The chain upgrades to V0_14_3 at block height `H`. Blocks `H-9` through `H-1` carry `fee_proposal_fri = None` (pre-V0_14_3).
2. For blocks `H` through `H+9`, `compute_fee_actual` returns `None` because the window `[H-10, H-1]` (or `[H-9, H]`, etc.) always contains at least one `None` entry.
3. A malicious proposer wins the lottery for block `H` and sends `ProposalInit { fee_proposal_fri: Some(GasPrice(u128::MAX)), ... }`.
4. `is_proposal_init_valid` reaches the `if let (Some(fee_actual), Some(fee_proposal))` check. `fee_actual` is `None`, so the entire block is skipped â€” no error is returned. [1](#0-0) 
5. `initiate_validation` is called; the batcher executes the block normally.
6. The validator computes `proposal_commitment_from(partial_block_hash, Some(u128::MAX))` and matches the proposer's `ProposalFin`. Consensus votes prevote/precommit for the block. [7](#0-6) 
7. `decision_reached` calls `record_fee_proposal(H, Some(GasPrice(u128::MAX)))`, inserting the extreme value into `fee_proposals_window`. [6](#0-5) 
8. The malicious proposer repeats for blocks `H+1` through `H+5` (6 of 10 window slots). After block `H+10`, `compute_fee_actual` returns `Some(u128::MAX)` (median of a window majority-filled with `u128::MAX`).
9. `calculate_next_l2_gas_price` uses this `fee_actual`, producing `next_l2_gas_price â‰ˆ u128::MAX` for all subsequent blocks. All L2 transactions become economically infeasible.
