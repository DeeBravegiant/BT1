### Title
Unvalidated `IndexerHeight` P2P Message Allows a Byzantine Participant to Inflate `maximum_height`, Causing All Pending Signature Requests to Be Permanently Dropped — (File: `crates/node/src/network/indexer_heights.rs`, `crates/node/src/requests/queue.rs`)

---

### Summary

A Byzantine MPC participant (strictly below the signing threshold) can send a single crafted `IndexerHeight` P2P message with `height = u64::MAX - MAX_INDEXER_HEIGHT_DIFF`. Because `IndexerHeightTracker::set_height` is monotonically increasing and never validated against the sender's real chain state, the stored height for that participant is permanently inflated for the lifetime of the honest node process. While the Byzantine participant remains bidirectionally connected, `eligible_leaders_and_maximum_height` uses their inflated height as `maximum_height`, which drives `cutoff_block` to an astronomically large value. Every real pending signature request — whose block height is in the hundreds-of-millions range — satisfies `block_height < cutoff_block` and is immediately and permanently dropped from the in-memory queue with `DropReason::RequestTimedOut`. No signatures can be produced while the attacker is connected, and the attack can be repeated on every reconnection because the stored height never resets.

---

### Finding Description

**Root cause — `set_height` accepts any value without upper-bound validation:** [1](#0-0) 

The function only enforces monotonicity (`if height > current`). There is no check that the claimed height is plausible relative to the honest node's own chain tip. Once a participant's stored height is set to a large value it can never decrease, even across reconnections.

**Ingestion path — `IndexerHeight` packets are accepted from any connected peer without validation:** [2](#0-1) 

The receive loop unconditionally calls `indexer_heights.set_height(message.from, message.message.height)` for every `PeerMessage::IndexerHeight` received over the TLS channel.

**`maximum_height` is the uncapped max of all alive participants' stored heights:** [3](#0-2) 

If the Byzantine participant is bidirectionally connected, `alive_participants` includes them, and their inflated height becomes `maximum_height`.

**Alive-participant check does not overflow with the chosen attack value:** [4](#0-3) 

With `peer_height = u64::MAX - MAX_INDEXER_HEIGHT_DIFF`, the expression `peer_height + MAX_INDEXER_HEIGHT_DIFF = u64::MAX` (no overflow). The condition `my_height <= u64::MAX` is always true, so the Byzantine participant is always considered alive while connected.

**`cutoff_block` is derived directly from `maximum_height`:** [5](#0-4) 

With `maximum_height ≈ u64::MAX`, `cutoff_block ≈ u64::MAX - 199`. Every real request at a normal NEAR block height (≈ 10⁸) satisfies `block_height < cutoff_block`.

**All requests are permanently dropped:** [6](#0-5) 

The `is_older_than(cutoff_block)` check fires before the leader-election check. Dropped requests are removed from `self.requests` and never re-queued from the indexer (requests are only added on first observation of the on-chain event).

---

### Impact Explanation

While the Byzantine participant is connected, every call to `get_requests_to_attempt` drops all pending signature, CKD, and foreign-tx-verification requests from the node's in-memory queue. The on-chain yield-resume promises time out after 200 blocks, causing every in-flight user request to fail. Because the stored height is monotonically locked at the inflated value, the attacker only needs to send the malicious message once per node process lifetime; subsequent reconnections immediately restore the attack state without any additional message. This breaks the request-lifecycle invariant of the MPC network: no threshold signatures can be produced while the attacker is connected.

---

### Likelihood Explanation

Any single registered MPC participant (below the signing threshold) can execute this attack. It requires only a valid mutual-TLS P2P connection (already established as part of normal operation) and sending one Borsh-serialized `Packet::IndexerHeight` message with `height = u64::MAX - MAX_INDEXER_HEIGHT_DIFF`. No cryptographic material, no collusion, and no special privileges are needed beyond being a registered participant. The attack is trivially scriptable and can be sustained indefinitely by maintaining the connection.

---

### Recommendation

1. **Short term:** In `set_height`, reject any claimed height that exceeds the honest node's own current indexer height by more than `MAX_INDEXER_HEIGHT_DIFF`. This mirrors the existing staleness check used in `all_alive_participant_ids` and prevents any participant from claiming a height the honest node cannot independently verify.

2. **Long term:** Derive `maximum_height` from the honest node's own locally-observed chain tip rather than from peer-reported heights. Peer heights should only be used for the staleness/liveness filter, not as the authoritative upper bound for request expiry.

---

### Proof of Concept

1. Byzantine participant `B` establishes a bidirectional TLS connection to honest node `H` (normal startup behavior).
2. `B` sends a single `Packet::IndexerHeight(IndexerHeightMessage { height: u64::MAX - MAX_INDEXER_HEIGHT_DIFF })` over the connection.
3. `H`'s `run_receive_message` calls `indexer_heights.set_height(B_id, u64::MAX - MAX_INDEXER_HEIGHT_DIFF)`.
4. `B` is included in `alive_participants` because `my_height <= (u64::MAX - MAX_INDEXER_HEIGHT_DIFF) + MAX_INDEXER_HEIGHT_DIFF = u64::MAX` is always true.
5. `eligible_leaders_and_maximum_height` returns `maximum_height = u64::MAX - MAX_INDEXER_HEIGHT_DIFF`.
6. `cutoff_block = (u64::MAX - MAX_INDEXER_HEIGHT_DIFF).saturating_sub(200) + 1 ≈ u64::MAX - 200`.
7. For every queued request at real block height `H_req ≈ 10^8`: `H_req < cutoff_block` → `RequestStatus::Drop(RequestTimedOut)` → removed from `self.requests`.
8. All in-flight user signing requests time out on-chain. `B` maintains the connection to sustain the attack; the stored height never resets until `H` restarts.
