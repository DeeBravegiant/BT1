Audit Report

## Title
`PriceVelocityGuardExtension.beforeSwap` updates reference price before the velocity check, allowing cumulative intra-block price drift to multiply the per-block cap — (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`beforeSwap` unconditionally writes `lastMidPriceX64` and `lastUpdateBlock` to storage before performing the velocity check. When multiple swaps occur in the same block, each swap is validated against the previous swap's price (not the block-opening price), and `blockDiff` is 0 for all but the first swap. This allows cumulative intra-block oracle price movement of `N × maxChangePerBlockE18` for `N` swaps, directly violating the documented per-block invariant.

## Finding Description
In `beforeSwap` (lines 53–76), the implementation reads `prevMid` and `prevBlock` from storage, then immediately overwrites both fields before the guard check:

```solidity
uint128 prevMid = s.lastMidPriceX64;   // line 54
uint64 prevBlock = s.lastUpdateBlock;   // line 55

s.lastMidPriceX64 = midPrice;          // line 57 — mutated BEFORE check
s.lastUpdateBlock = uint64(block.number); // line 58 — mutated BEFORE check
``` [1](#0-0) 

The velocity check then computes:
```solidity
uint256 blockDiff = block.number - prevBlock;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
``` [2](#0-1) 

For the first swap in block N (coming from block B_old), `blockDiff = N − B_old ≥ 1`, so `allowedSq` is at least `maxChange²`. After that swap, `lastUpdateBlock = N`. For every subsequent swap in the same block, `prevBlock = N`, so `blockDiff = 0` and `allowedSq = maxChange²`. Each subsequent swap is checked only against the price stored by the immediately preceding swap — not the block-opening price. The cumulative movement across `N` swaps in one block is `N × maxChangePerBlockE18`, while the NatSpec invariant (lines 15–17) guarantees at most `maxChangePerBlockE18` per block. [3](#0-2) 

## Impact Explanation
The velocity guard's per-block price-movement cap is bypassed. LPs execute swaps at oracle prices that can be arbitrarily far from the block-opening price (bounded only by the number of swaps an attacker can fit in one block). This constitutes **bad-price execution**: an unclamped oracle price reaches a pool swap, causing direct LP principal loss. The impact matches the allowed gate: *bad-price execution: stale, inverted, unbounded, or unclamped bid/ask quote reaches a pool swap*.

## Likelihood Explanation
Pyth oracle price updates are permissionless — any on-chain actor can submit a valid signed VAA. On fast L2s (Arbitrum, Base, etc.) where multiple Pyth price updates are available within a single block, an attacker can atomically bundle alternating Pyth updates and `swap()` calls in a single transaction or across multiple transactions in the same block. No privileged role is required; the only precondition is that the pool has `PriceVelocityGuardExtension` enabled with a nonzero `maxChangePerBlockE18`.

## Recommendation
Only advance `lastMidPriceX64` when entering a new block. Within the same block, leave the stored price unchanged so every swap is validated against the block-opening price:

```solidity
if (block.number > prevBlock) {
    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);
}
// If same block: do NOT update lastMidPriceX64; check current price against block-open price

if (prevMid != 0) {
    uint256 blockDiff = block.number - prevBlock;
    // ... existing check unchanged ...
}
```

This ensures all swaps within the same block are bounded against the same reference price, matching the documented invariant.

## Proof of Concept
```
Setup:
  maxChangePerBlockE18 = 0.05e18  (5% per block)
  lastMidPriceX64 = P0 = 1e18    (set in block B_old)

Block N (attacker bundles atomically):

  Tx1: Pyth update → oracle price = P1 = 1.05e18 (+5%)
  Tx2: swap() → beforeSwap:
       prevMid=P0, prevBlock=B_old, blockDiff=N-B_old ≥ 1
       changeE18=5%, actualSq=(0.05e18)², allowedSq≥actualSq → PASS
       lastMidPriceX64=P1, lastUpdateBlock=N

  Tx3: Pyth update → oracle price = P2 = 1.1025e18 (+5% from P1)
  Tx4: swap() → beforeSwap:
       prevMid=P1, prevBlock=N, blockDiff=0
       changeE18=5%, actualSq=(0.05e18)², allowedSq=(0.05e18)²×1 → PASS
       lastMidPriceX64=P2, lastUpdateBlock=N

  Tx5: Pyth update → oracle price = P3 = 1.1576e18 (+5% from P2)
  Tx6: swap() → beforeSwap:
       prevMid=P2, prevBlock=N, blockDiff=0
       changeE18=5% → PASS
       lastMidPriceX64=P3

Result:
  Total intra-block movement: P0→P3 = +15.76%
  Intended per-block cap:     5%
  Guard bypassed by factor:   ~3×
  LPs execute at P3, which is 10.76% above the guard's intended ceiling.
```
