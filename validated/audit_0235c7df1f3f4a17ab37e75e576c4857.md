Audit Report

## Title
Missing Decimal Normalization in Token-to-rsETH Swap Calculation Causes Near-Zero rsETH Minting for Sub-18-Decimal Tokens — (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

## Summary
The `viewSwapRsETHAmountAndFee(uint256 amount, address token)` overload computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18 decimals. Because the oracle always returns a 1e18-precision rate, the formula is structurally correct only for 18-decimal tokens. For any token with fewer than 18 decimals (e.g., USDC at 6), the result is off by a factor of `10^(18 − tokenDecimals)`, causing a depositor to receive a near-zero rsETH mint while their full token balance is transferred to the pool.

## Finding Description
The ETH deposit path in `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256)` correctly accounts for ETH's 18-decimal representation:

```solidity
// RSETHPoolV3.sol L307
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The token deposit overload omits the equivalent normalization:

```solidity
// RSETHPoolV3.sol L334
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`tokenToETHRate` is sourced from `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which explicitly normalizes Chainlink answers to 1e18 precision (`uint256(ethPrice) * 1e18 / 10 ** decimals()`). For USDC (6 decimals), `tokenToETHRate ≈ 3e14` (0.0003 ETH per whole USDC). Depositing 1,000 USDC (`amountAfterFee = 1e9`) yields:

```
rsETHAmount = 1e9 * 3e14 / 1.05e18 ≈ 285,714  (≈ 2.86e-13 rsETH)
```

The correct result requires the missing `1e12` normalization factor:

```
rsETHAmount = 1e9 * 1e12 * 3e14 / 1.05e18 ≈ 2.86e17  (≈ 0.2857 rsETH)
```

`addSupportedToken` performs no decimal check — it only verifies the oracle returns a non-zero rate. The `deposit()` function performs no minimum-output check, so a near-zero `rsETHAmount` does not revert; `wrsETH.mint(msg.sender, rsETHAmount)` executes with the dust value. The identical formula defect is present in `RSETHPoolV3ExternalBridge` (L452) and `RSETHPoolV3WithNativeChainBridge` (L370).

## Impact Explanation
**Critical — Direct theft of user funds.** Once a sub-18-decimal token is added to `supportedTokenOracle`, any unprivileged depositor calling `deposit(token, amount, referralId)` loses virtually their entire deposit: the full token amount is transferred to the pool via `safeTransferFrom`, but the minted rsETH is `10^(18 − tokenDecimals)` times smaller than the correct amount. For USDC (6 decimals), the loss factor is `1e12`. The pool retains the tokens with no corresponding liability, constituting direct theft of depositor funds.

## Likelihood Explanation
**Medium.** The precondition is that a TIMELOCK_ROLE admin adds a sub-18-decimal token via `addSupportedToken`. This is a normal protocol operation — the function accepts arbitrary ERC20 tokens with no decimal restriction. Common DeFi tokens (USDC: 6, USDT: 6, WBTC: 8) are plausible candidates for cross-chain pool collateral. Once such a token is listed, the exploit is triggered by any unprivileged user on every deposit call, is repeatable, and requires no special attacker capability beyond holding the token.

## Recommendation
Normalize `amountAfterFee` to 18 decimals before the rate calculation, mirroring the ETH path:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * (10 ** (18 - tokenDecimals));
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply this fix identically in all three contracts. Alternatively, enforce in `addSupportedToken` that `IERC20Metadata(token).decimals() == 18` and document the restriction explicitly.

## Proof of Concept
**Setup (Foundry fork test):**
- Deploy `RSETHPoolV3` with `feeBps = 0`
- Add mock USDC (6 decimals) with `tokenToETHRate = 3e14` (via `ChainlinkOracleForRSETHPoolCollateral` wrapping a mock feed)
- Set `rsETHToETHrate = 1.05e18`

**Steps:**
1. Mint 1,000 USDC (`1_000_000_000`) to attacker
2. Call `deposit(USDC, 1_000_000_000, "")`
3. Assert `wrsETH.balanceOf(attacker) == 285_714` (actual, broken)
4. Assert expected value `≈ 2.86e17` (correct, with normalization)
5. Assert `IERC20(USDC).balanceOf(pool) == 1_000_000_000` (full deposit retained)

The test demonstrates that the pool retains 1,000 USDC while the depositor receives `~2.86e-13` rsETH instead of `~0.2857` rsETH — a loss of `~$1,000` of value per deposit. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
