Audit Report

## Title
Accounting Underflow in `transferAssetToDepositPool` Zeros `ethValueInWithdrawal` While LSTs Remain in Converter, Enabling Cheap rsETH Minting — (`contracts/LRTConverter.sol`)

## Summary

`transferAssetToDepositPool` debits `ethValueInWithdrawal` at the current oracle price, but credits were recorded at an earlier (lower) price. When the debit for a partial return exceeds the total accumulated credit, the variable is floored to zero even though other LST batches remain physically in the converter. Because converter LST holdings are tracked exclusively through `ethValueInWithdrawal`, the remaining tokens become invisible to the TVL calculation, deflating `rsETHPrice` and allowing an unprivileged attacker to mint excess rsETH at the expense of existing holders.

## Finding Description

**Root cause — `LRTConverter.sol` line 163:**

```solidity
ethValueInWithdrawal = ethValueInWithdrawal > assetValue ? ethValueInWithdrawal - assetValue : 0;
```

Credits are recorded at the oracle price at deposit time (`transferAssetFromDepositPool`, line 140); debits are recorded at the oracle price at return time. If the oracle price rises between the two operations, the ETH-value debit for a partial return can exceed the entire accumulated credit, flooring `ethValueInWithdrawal` to zero while a residual LST batch remains in the converter.

**Why the remaining tokens become invisible:**

`getAssetDistributionData` explicitly sets `assetLyingInConverter = 0` for every LST at line 460, delegating converter accounting entirely to `getETHDistributionData`. `getETHDistributionData` reads only `ILRTConverter(lrtConverter).ethValueInWithdrawal()` at line 499. A zeroed `ethValueInWithdrawal` makes the residual batch completely invisible to `getTotalAssetDeposits` and therefore to `_getTotalEthInProtocol`.

**Exploit flow:**

1. Operator calls `transferAssetFromDepositPool(stETH, A)` and `transferAssetFromDepositPool(stETH, B)` at price P1 → `ethValueInWithdrawal = (A+B)*P1/1e18`.
2. Oracle price rises to P2 such that `A*P2 >= (A+B)*P1`.
3. Operator calls `transferAssetToDepositPool(stETH, A)` (legitimate) → `assetValue = A*P2/1e18 >= ethValueInWithdrawal` → `ethValueInWithdrawal = 0`. Batch B (worth `B*P2/1e18` ETH) is now invisible.
4. Attacker calls public `updateRSETHPrice()` → `newRsETHPrice` is computed on understated TVL → stored `rsETHPrice` is deflated.
5. Attacker calls `depositAsset` or `depositETH` → `getRsETHAmountToMint` divides by the deflated `rsETHPrice` → attacker receives excess rsETH.
6. Operator calls `transferAssetToDepositPool(stETH, B)` (legitimate) → TVL is restored.
7. Attacker calls `updateRSETHPrice()` again → price rises → attacker redeems at profit, extracting value from existing rsETH holders.

**Why existing guards are insufficient:**

The downside-protection auto-pause in `_updateRsETHPrice` (lines 270–281) only triggers when `pricePercentageLimit > 0` and the price drop exceeds `pricePercentageLimit * highestRsethPrice`. If `pricePercentageLimit` is zero, or if batch B is small relative to total TVL (making the price drop sub-threshold), the protocol does not pause and the deflated price is stored and exploitable.

## Impact Explanation

**High — Theft of unclaimed yield.**

The yield accrued by the converter's LST holdings (batch B) is temporarily excluded from TVL. An attacker who mints rsETH during this window acquires a larger share of the protocol than their deposit warrants. When batch B is returned and TVL is restored, the rsETH price rises and the attacker redeems at a profit, extracting value that belongs to pre-existing rsETH holders (their proportional share of the staking yield generated while the LSTs were in the converter).

## Likelihood Explanation

The triggering condition `A * P2 >= (A + B) * P1` is achievable with realistic parameters. For a large first batch (A) and small residual (B), even a 1–2% oracle price appreciation (routine staking yield over weeks) satisfies the condition. The operator performs entirely legitimate operations; no malicious intent or role compromise is required. `updateRSETHPrice()` is permissionless, so the attacker can atomically trigger the price update and deposit in the same block. The attack is repeatable whenever the operator moves assets in multiple batches.

## Recommendation

Track token amounts rather than ETH values in the converter. Replace the single `ethValueInWithdrawal` scalar with a per-asset token balance mapping and compute the ETH value live from the current oracle price:

```solidity
mapping(address => uint256) public assetAmountInConverter;

// On transferAssetFromDepositPool:
assetAmountInConverter[_asset] += _amount;

// On transferAssetToDepositPool:
assetAmountInConverter[_asset] -= _amount; // reverts on underflow

// ethValueInWithdrawal (view):
function ethValueInWithdrawal() external view returns (uint256 total) {
    address[] memory assets = lrtConfig.getSupportedAssetList();
    for (uint256 i; i < assets.length; ++i) {
        if (assetAmountInConverter[assets[i]] > 0) {
            total += (assetAmountInConverter[assets[i]] * lrtOracle.getAssetPrice(assets[i])) / 1e18;
        }
    }
}
```

This ensures the reported ETH value always reflects actual tokens held, regardless of price movements between deposit and return, and the subtraction reverts on underflow rather than silently zeroing.
