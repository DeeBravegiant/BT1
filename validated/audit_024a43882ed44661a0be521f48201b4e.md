Audit Report

## Title
Sequential Unbounded External Calls in `updateRate()` Can Permanently Block Cross-Chain rsETH Rate Broadcasts - (File: `contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary
`MultiChainRateProvider.updateRate()` iterates over an unbounded `rateReceivers` array, executing two external LayerZero calls per entry with no cap on array length. As the protocol adds destination chains via `addRateReceiver()`, gas consumption grows linearly until `updateRate()` exceeds the block gas limit and becomes permanently uncallable, freezing rsETH rate updates to all destination chains.

## Finding Description
`updateRate()` (lines 108–137) is the sole mechanism for pushing the rsETH/ETH exchange rate to all registered cross-chain receivers. The loop at lines 119–134 iterates over every entry in `rateReceivers` and, for each one, executes:

1. `ILayerZeroEndpoint.estimateFees(...)` — an external `staticcall` with memory allocation for `_payload` and `remoteAndLocalAddresses`.
2. `ILayerZeroEndpoint.send{value: estimatedFee}(...)` — a state-changing external call with ETH transfer and cross-chain messaging overhead.

`addRateReceiver()` (lines 72–76) unconditionally pushes to the array with no length cap:

```solidity
rateReceivers.push(RateReceiver({ _chainId: _chainId, _contract: _contract }));
```

There is no guard in `updateRate()` to limit the number of iterations, no partial-update mechanism, and no way to split the work across multiple transactions. Once the cumulative gas for all iterations exceeds the block gas limit, every invocation of `updateRate()` reverts with out-of-gas, and no caller — privileged or otherwise — can recover without a contract upgrade.

## Impact Explanation
`updateRate()` is the only push path for rsETH rates to destination chains. If it becomes permanently uncallable, all `RSETHRateReceiver` contracts on destination chains hold a stale rate indefinitely. This directly matches **Medium — Unbounded gas consumption** from the allowed impact scope. Secondary consequences (mispriced collateral, incorrect swap rates) are real but downstream; the primary in-scope impact is the unbounded gas consumption causing permanent denial of the rate-update function.

## Likelihood Explanation
rsETH is already live on multiple chains (Arbitrum, Optimism, etc.) and the protocol's stated design is continued multi-chain expansion. Each new chain deployment requires exactly one `addRateReceiver()` call — normal, expected protocol operation. LayerZero `send` with cross-chain messaging overhead costs on the order of 50,000–100,000 gas per iteration; at 20–30 receivers this approaches or exceeds Ethereum's 30M block gas limit when accounting for base transaction overhead. No attacker action is required: the threshold is reached through routine protocol growth. `updateRate()` has no access control (`external payable`, no role modifier), so any caller hits the same gas wall once the array is large enough.

## Recommendation
1. **Enforce a maximum cap** on `rateReceivers.length` inside `addRateReceiver()` (e.g., `require(rateReceivers.length < MAX_RECEIVERS)`).
2. **Add a ranged update overload**: `updateRate(uint256 fromIndex, uint256 toIndex)` so callers can broadcast to a subset of receivers per transaction, keeping each call within gas limits.
3. **Pull-based alternative**: emit an on-chain event with the new rate and let each destination chain's relay pull the rate independently, eliminating the push-loop entirely.

## Proof of Concept
1. Owner calls `addRateReceiver()` for each new destination chain as rsETH expands (e.g., 25 chains).
2. Any caller invokes `updateRate{ value: totalFee }()`.
3. The loop executes `estimateFees` + `send` for each of the 25 receivers sequentially.
4. Gas consumed per iteration ≈ 50,000–100,000 gas; at 25 receivers this is 1.25M–2.5M gas for the loop body alone, plus base transaction and storage write overhead.
5. Beyond the threshold, every invocation reverts with out-of-gas.
6. No receiver on any destination chain can receive an updated rsETH rate; `RSETHRateReceiver.rate` on all chains is permanently stale.
