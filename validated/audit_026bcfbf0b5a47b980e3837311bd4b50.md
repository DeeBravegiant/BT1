### Title
Block-Level Reference Script Size Check Undercounts Sub-Transaction Reference Scripts in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

`dijkstraBbodyTransition` calls `Conway.validateBodyRefScriptsSizeTooBig`, which internally uses `totalRefScriptSizeInBlock`. That function calls `txNonDistinctRefScriptsSize` per top-level transaction — a function that only inspects the top-level tx's own `referenceInputsTxBodyL` and `inputsTxBodyL`. It never recurses into sub-transactions (`DijkstraSubTx`). A separate, Dijkstra-aware function `batchNonDistinctRefScriptsSize` exists and does include sub-transactions, but it is only used in the per-tx LEDGER check, not in the block-level BBODY check.

---

### Finding Description

**Block-level check (BBODY rule):**

`dijkstraBbodyTransition` at line 363 calls:
```haskell
Conway.validateBodyRefScriptsSizeTooBig @era pp blockBody (ls ^. utxoL)
``` [1](#0-0) 

This calls `totalRefScriptSizeInBlock`, which iterates over `blockBody ^. txSeqBlockBodyL` and calls `txNonDistinctRefScriptsSize` for each top-level tx: [2](#0-1) 

`txNonDistinctRefScriptsSize` only looks at the top-level tx's own inputs and reference inputs — it does not recurse into sub-transactions: [3](#0-2) 

**Per-tx check (LEDGER rule):**

`dijkstraLedgerTransition` at line 392 calls `validateAllRefScriptSize`, which uses `batchNonDistinctRefScriptsSize`: [4](#0-3) 

`batchNonDistinctRefScriptsSize` correctly sums the top-level tx AND all sub-transactions: [5](#0-4) 

**The gap:** The per-tx limit uses `batchNonDistinctRefScriptsSize` (sub-tx-aware). The per-block limit uses `txNonDistinctRefScriptsSize` (sub-tx-blind). Sub-transaction reference inputs are never counted toward `ppMaxRefScriptSizePerBlock`.

Sub-transactions carry their own `dstbrReferenceInputs` field: [6](#0-5) 

---

### Impact Explanation

An unprivileged block producer can craft a block containing top-level transactions whose sub-transactions each reference large scripts. The block-level check sees only the top-level tx reference inputs (which may be zero), so `totalRefScriptSizeInBlock` returns a value well below `ppMaxRefScriptSizePerBlock`. The actual combined reference script data processed by the node (including all sub-transaction reference inputs) can exceed the intended block limit by an arbitrary multiple.

This is a **Medium** impact: attacker-controlled block content exceeds the intended `ppMaxRefScriptSizePerBlock` validation limit. All honest nodes run the same code and reach the same (incorrect) conclusion, so there is no ledger divergence — the Critical divergence impact does not apply. The impact is resource-limit bypass: nodes process more reference script bytes per block than the protocol parameter is intended to allow.

---

### Likelihood Explanation

Any block producer (no privilege required) can submit a block with top-level transactions containing sub-transactions with large reference inputs. The per-tx check (`validateAllRefScriptSize`) would reject a single tx whose sub-tx reference scripts exceed `ppMaxRefScriptSizePerTx`, but the block-level bypass is achievable by spreading sub-tx reference script usage across multiple top-level transactions, each individually within the per-tx limit, while their combined sub-tx reference script total exceeds `ppMaxRefScriptSizePerBlock`.

---

### Recommendation

Replace the call to `Conway.validateBodyRefScriptsSizeTooBig` in `dijkstraBbodyTransition` with a Dijkstra-aware equivalent that sums `batchNonDistinctRefScriptsSize` (or an equivalent accumulating function) across all top-level transactions in the block, so that sub-transaction reference inputs are included in the block-level total. [7](#0-6) 

---

### Proof of Concept

1. Set `ppMaxRefScriptSizePerBlock = S` and `ppMaxRefScriptSizePerTx = S` (equal, for simplicity).
2. Produce UTxO outputs each carrying a reference script of size `S/2`.
3. Construct two top-level transactions, each with one sub-transaction whose `dstbrReferenceInputs` points to one such output. Each top-level tx has no reference inputs of its own.
4. `totalRefScriptSizeInBlock` returns 0 (no top-level reference inputs) — block passes the BBODY check.
5. `validateAllRefScriptSize` per tx returns `S/2` for each — both pass the per-tx check.
6. Actual reference script bytes processed = `S` (from two sub-tx reference inputs) = exactly the block limit, but with three or more such transactions the limit is exceeded while the check still passes.

totalRefScriptSizeInBlock ::
  (AlonzoEraTx era, BabbageEraTxBody era) => ProtVer -> StrictSeq (Tx TopTx era) -> UTxO era -> Int
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where

-- | Total size of reference scripts across a top-level transaction and all its subtransactions.
batchNonDistinctRefScriptsSize ::
  ( EraTx era
  , DijkstraEraTxBody era
  ) =>
  UTxO era ->
  Tx TopTx era ->
  Int
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
