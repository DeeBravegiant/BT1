### Title
`estimate_minimal_gas_vector` Omits `n_compiled_class_hash_updates` for Declare Transactions, Causing Gateway to Accept Under-Resourced Declare Txs That Revert on Execution - (File: crates/blockifier/src/fee/gas_usage.rs)

### Summary

`estimate_minimal_gas_vector` hard-codes `n_compiled_class_hash_updates: 0` for every `Declare` transaction, even though V2/V3 Declare transactions always write one compiled-class-hash entry to state. The function is the sole source of the minimum-gas floor used by `check_fee_bounds` inside `perform_pre_validation_stage`. Because the floor is too low, the gateway's stateful validator and the blockifier's pre-validation stage both accept Declare transactions whose resource bounds are above the (wrong) floor but below the true minimum. Those transactions pass admission, enter the mempool, are sequenced into a block, and then revert at post-execution with `MaxGasAmountExceeded`, charging the user a revert fee for a transaction that should have been rejected at the gate.

### Finding Description

`estimate_minimal_gas_vector` builds a `StateChangesCount` for each transaction type to estimate the minimum DA gas cost:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,   // ← wrong; V2/V3 Declare always writes 1
    n_modified_contracts: 1,
},
``` [1](#0-0) 

The TODO comment is self-documenting: the correct value is `1`. The test helper `declare_expected_state_changes_count` confirms that V2/V3 Declare transactions produce exactly `n_compiled_class_hash_updates: 1` at execution time:

```rust
} else if version == TransactionVersion::TWO || version == TransactionVersion::THREE {
    StateChangesCount {
        n_storage_updates: 1,
        n_modified_contracts: 1,
        n_compiled_class_hash_updates: 1, // Also set compiled class hash.
        ..StateChangesCount::default()
    }
``` [2](#0-1) 

The DA-gas formula weights each compiled-class-hash update as **2 field elements**:

```rust
let on_chain_data_segment_length = state_changes_count.n_storage_updates * 2
    + state_changes_count.n_class_hash_updates
    + state_changes_count.n_compiled_class_hash_updates * 2   // ← 2 felts per entry
    + state_changes_count.n_modified_contracts * 2;
``` [3](#0-2) 

So the minimum-gas estimate is short by `2 × DATA_GAS_PER_FIELD_ELEMENT` blob-gas units (KZG-DA mode) or `2 × GAS_PER_FIELD_ELEMENT` L1-gas units (calldata-DA mode).

`check_fee_bounds` calls `estimate_minimal_gas_vector` and rejects any transaction whose resource bounds fall below the returned floor:

```rust

    let minimal_gas_amount_vector = estimate_minimal_gas_vector(
        &tx_context.block_context,
        self,
        &tx_context.get_gas_vector_computation_mode(),
    );
``` [4](#0-3) 

`check_fee_bounds` is invoked from `perform_pre_validation_stage`, which is called both by the gateway's `run_validate_entry_point` (via `StatefulValidator::validate`) and by the blockifier during actual execution:

The gateway's stateful validator builds a `BlockContext` and calls `StatefulValidator::validate`, which runs `perform_pre_validation_stage` with `charge_fee = true` for any Declare transaction that has non-zero resource bounds:

```rust

### Impact Explanation

A V2/V3 Declare transaction whose resource bounds satisfy:

```
estimated_floor  <  user_bounds  <  actual_floor
```

passes the gateway's stateful pre-validation check, is admitted to the mempool, is sequenced into a block, and then reverts at post-execution with `MaxGasAmountExceeded` (or `MaxGasPriceTooLow` for the data-gas component). The user is charged a revert fee for a transaction that should have been rejected at the gateway. Block space is consumed by a predictably-failing transaction. Any upstream caller (SDK, wallet, RPC client) that derives resource bounds from the gateway's own minimum-gas estimate will construct a Declare transaction that the gateway accepts but execution rejects — the exact ERC-4626 `maxWithdraw`/`withdraw` failure pattern.

### Likelihood Explanation

The affected path is the default path for every V2/V3 Declare transaction. The gap between the underestimated floor and the true floor is small but fixed and deterministic (`2 × DATA_GAS_PER_FIELD_ELEMENT` blob-gas units in KZG-DA mode). Any caller that sets resource bounds to exactly the gateway-reported minimum will land in the gap. The TODO comment confirms the bug is known and unresolved.

### Recommendation

Change `n_compiled_class_hash_updates` from `0` to `1` for the `Declare` arm in `estimate_minimal_gas_vector`:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1,  // V2/V3 Declare always writes compiled class hash
    n_modified_contracts: 1,
},
``` [1](#0-0) 

Add a regression test that submits a V2/V3 Declare transaction with resource bounds equal to the value returned by `estimate_minimal_gas_vector` and asserts it does not revert.

### Proof of Concept

1. Call `estimate_minimal_gas_vector` for a V3 Declare transaction in KZG-DA mode. The returned `l1_data_gas` component is `N` blob-gas units.
2. Submit a V3 Declare transaction with `l1_data_gas.max_amount = N` (exactly the reported minimum).
3. The gateway's `check_fee_bounds` passes because `N >= N`.
4. The transaction is admitted to the mempool and sequenced.
5. During execution, the actual DA gas is `N + 2 × DATA_GAS_PER_FIELD_ELEMENT` (the missing compiled-class-hash entry).
6. Post-execution `FeeCheckReport::check_all_gas_amounts_within_bounds` fires `MaxGasAmountExceeded { resource: L1DataGas, max_amount: N, actual_amount: N + delta }`. [7](#0-6) 

The transaction is included in the block as a revert, and the user is charged a revert fee — identical in structure to the ERC-4626 `maxWithdraw`→`withdraw` revert: the capacity-reporting function omits an overhead cost, the caller uses the reported value verbatim, and the downstream operation fails because the true cost exceeds the reported maximum.

        max_amount_bounds: &GasVector,
        gas_vector: &GasVector,
    ) -> FeeCheckResult<()> {
        // TODO(Arni): Consider refactoring the returned error. The first failed check will hide
        // future checks.
        for (resource, max_amount, actual_amount) in [
            (L1Gas, max_amount_bounds.l1_gas, gas_vector.l1_gas),
            (L2Gas, max_amount_bounds.l2_gas, gas_vector.l2_gas),
            (L1DataGas, max_amount_bounds.l1_data_gas, gas_vector.l1_data_gas),

            if max_amount < actual_amount {
                return Err(FeeCheckError::MaxGasAmountExceeded {
                    resource,
                    max_amount,
                    actual_amount,
                });
            }
        }

        Ok(())
    }
