### Title
NFT Offer Taker Overpays Royalties Due to Unvalidated `driver_dict` Royalty Percentage in `TradeManager._create_offer_for_ids` - (File: `chia/wallet/trade_manager.py`)

### Summary
When a taker accepts an NFT1 offer via `TradeManager.respond_to_offer`, the royalty percentage used to compute royalty payments is read from the offer's `driver_dict` without being validated against the actual on-chain NFT puzzle. A malicious maker can craft an offer with an arbitrarily inflated `royalty_percentage` in the `driver_dict`, causing the taker to pay significantly more royalties than the NFT's actual on-chain royalty rate. If the maker set their own address as the royalty address (common for NFT creators), they receive the excess, draining the taker's XCH or CAT balance.

### Finding Description

**Root cause — missing driver_dict validation on the receiving side:**

In `_create_offer_for_ids`, the driver_dict is validated against the on-chain puzzle only when the local wallet owns the asset being offered:

```python
# chia/wallet/trade_manager.py lines 560-573
if asset_id is not None and wallet is not None:   # ← guard
    if callable(getattr(wallet, "get_puzzle_info", None)):
        puzzle_driver = await wallet.get_puzzle_info(asset_id)
        if asset_id in driver_dict and driver_dict[asset_id] != puzzle_driver:
            ...
            raise ValueError(...)
        else:
            driver_dict[asset_id] = puzzle_driver
``` [1](#0-0) 

When the taker is **receiving** an NFT they do not yet own, `get_wallet_for_asset_id(asset_id)` returns `None`, so the guard `wallet is not None` is false and the driver_dict is never cross-checked against the on-chain puzzle. The offer's (potentially manipulated) driver_dict is passed directly into `check_for_special_offer_making` → `NFTWallet.make_nft1_offer`.

**How royalty is computed from the unvalidated driver_dict:**

`make_nft1_offer` reads the royalty percentage directly from the driver_dict:

```python
royalty_percentage_raw = transfer_info["transfer_program"]["royalty_percentage"]
royalty_percentage = int(royalty_percentage_raw)   # attacker-controlled value
``` [2](#0-1) 

It then computes the taker's royalty payment:

```python
extra_royalty_amount = compute_royalty_amount(amount, request_side_royalty_split, percentage)
payment_list.append((launcher_id, CreateCoin(address, extra_royalty_amount, [address])))
``` [3](#0-2) 

`compute_royalty_amount` only rejects values strictly ≥ 100% (10000 basis points) or values where the royalty meets or exceeds the offered amount:

```python
def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(...)
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
``` [4](#0-3) 

Values up to 9999 basis points (≈99.99%) pass unchecked.

**Why the on-chain NFT puzzle does not prevent overpayment:**

The royalty percentage and address are curried immutably into the NFT transfer program at mint time:

```python
transfer_program = NFT_TRANSFER_PROGRAM_DEFAULT.curry(singleton_struct, royalty_puzzle_hash, percentage)
``` [5](#0-4) 

The on-chain puzzle enforces a **minimum** royalty (e.g., 5% of the trade price). If the taker's wallet creates a royalty payment coin that is **larger** than the minimum (e.g., 90%), the on-chain assertion is satisfied and the spend succeeds. The excess goes to the royalty address — which is the maker's address if they set it at mint time.

**Attack path:**

1. Malicious maker mints an NFT with `royalty_percentage = 500` (5%) and `royalty_address = maker_address`.
2. Maker crafts an offer with a manipulated `driver_dict` containing `royalty_percentage = 9000` (90%) and the same `royalty_address`.
3. Taker calls `respond_to_offer`. The taker does not own the NFT, so `get_wallet_for_asset_id` returns `None` and driver_dict validation is skipped.
4. `make_nft1_offer` computes royalty = 90% of the trade price and creates a royalty payment coin to `maker_address`.
5. The on-chain NFT puzzle only requires 5%, so the spend bundle is valid.
6. Maker receives 90% of the trade price as "royalty" instead of 5%.

**`respond_to_offer` passes the offer's driver_dict directly:**

```python
result = await self._create_offer_for_ids(
    take_offer_dict,
    inner_action_scope,
    offer.driver_dict,   # ← maker-controlled, unvalidated
    solver,
    fee=fee,
    extra_conditions=extra_conditions,
    taking=True,
)
``` [6](#0-5) 

### Impact Explanation

The taker loses up to ~99.99% of the trade price as inflated royalties. If the maker is the royalty recipient (their address is the royalty address), they receive the excess directly. This is an unauthorized accounting change affecting XCH and CATs in offer settlement, caused by an unprivileged attacker (any NFT creator). The taker's funds are drained without their informed consent, as the wallet software silently accepts the manipulated royalty rate.

### Likelihood Explanation

Any NFT creator can execute this attack by crafting a malicious offer. The taker's CLI does display a royalty summary before acceptance, but:
- Programmatic takers (bots, automated market makers) typically skip manual review.
- Inattentive users may not notice the inflated royalty percentage.
- The offer file itself is opaque binary/bech32 — the royalty percentage is not visible without decoding.

### Recommendation

In `_create_offer_for_ids`, when `taking=True` and the taker is receiving an NFT (`amount > 0`), independently resolve the on-chain NFT puzzle info via a full-node coin lookup (using the launcher ID from the driver_dict) and compare the curried `royalty_percentage` and `royalty_address` against the driver_dict values. If they differ, reject the offer or override the driver_dict with the on-chain values. This mirrors the existing validation already performed for assets the taker is offering (the `wallet is not None` branch).

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
