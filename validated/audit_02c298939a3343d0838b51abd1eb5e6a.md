### Title
`skip_verify=true` + Attacker-Controlled `Q` Enables Arbitrary Scalar Injection in PVE Decrypt - (`src/cbmpc/api/pve_base_pke.cpp`)

---

### Summary

The public API function `coinbase::api::pve::decrypt()` unconditionally passes `skip_verify=true` to `ec_pve_t::decrypt()`, bypassing the `ec_pve_t::verify()` proof check. The only remaining guard inside `restore_from_decrypted()` — the check `x_value * G == Q` — is trivially defeated because the attacker fully controls the `Q` field inside the deserialized ciphertext blob. An attacker who can supply a crafted ciphertext blob can cause `decrypt()` to return an arbitrary attacker-chosen scalar as `out_x`, regardless of what was originally encrypted.

---

### Finding Description

**Entrypoint:** `coinbase::api::pve::decrypt()` — a public C++ API function.

**Step 1 — Deserialization with no structural validation:** [1](#0-0) 

The attacker-supplied `ciphertext` blob is deserialized into a `pve_ciphertext_blob_v1_t`, then into an `ec_pve_t`. All fields — `Q`, `L`, `b`, `x_rows[i]`, `r[i]`, `c[i]` — are fully attacker-controlled after this step.

**Step 2 — Proof verification unconditionally skipped:** [2](#0-1) 

`skip_verify=true` is hardcoded. Inside `ec_pve_t::decrypt()`: [3](#0-2) 

The `verify()` call — which re-derives `b` from the commitment and checks `b' == b` — is entirely skipped.

**Step 3 — `inner_label` is derived from attacker-controlled `Q`:** [4](#0-3) [5](#0-4) 

`inner_label = label || "-" || hex(sha256(Q))`. Since the attacker chooses `Q`, they can compute `inner_label` offline.

**Step 4 — Attacker forges a valid `c[i]` using the public key `ek`:**

The loop in `ec_pve_t::decrypt()` decrypts `c[i]` under `dk` with `inner_label`: [6](#0-5) 

`ek` is a public key passed to `decrypt()` and known to the attacker. The attacker encrypts their chosen `x_value` under `ek` with the computed `inner_label` to produce a valid `c[0]`.

**Step 5 — The only remaining guard is defeated by attacker-controlled `Q`:** [7](#0-6) 

`restore_from_decrypted()` checks `x_value * G == Q`. Since the attacker set `Q = x_value * G` in the blob, this check always passes for their chosen `x_value`.

---

### Impact Explanation

Any caller of the public `coinbase::api::pve::decrypt()` API that accepts a ciphertext from an untrusted source will receive an attacker-chosen scalar as `out_x`. The caller has no way to distinguish this from a legitimately decrypted value. If `out_x` is used as a key share, private scalar, or seed material, the attacker has fully substituted it with a value of their choosing — a complete key injection attack.

---

### Likelihood Explanation

The attack requires only:
1. The ability to supply a crafted `ciphertext` blob to `decrypt()` (standard API input).
2. Knowledge of `ek` (a public key, passed as a parameter) and `label` (also a parameter).
3. The ability to call `base_pke.encrypt(ek, inner_label, x_value, rho)` — a public operation.

No private key material, threshold collusion, or privileged access is required.

---

### Recommendation

Remove the `skip_verify=true` hardcoding in `pve_base_pke.cpp::decrypt()`. The `verify()` call must execute before any decryption attempt. If performance is a concern for HSM paths, `verify()` should still be called (it only uses the public key `ek`), and the result must gate the decryption. [2](#0-1) 

Change to `skip_verify=false` (or remove the parameter entirely from this call site).

---

### Proof of Concept

```
Given: public API decrypt(curve, dk, ek, ciphertext, label, out_x)
       where ek is known (public key), label is known (API parameter)

Attack:
1. Choose x_attack (any scalar in [1, q-1])
2. Set Q_attack = x_attack * G
3. Compute inner_label = label || "-" || hex(sha256(Q_attack))
4. Encrypt x_attack under ek with inner_label → c_attack
5. Build ec_pve_t blob:
     Q       = Q_attack
     L       = label
     b       = 0b...0001  (bit 0 = 1, rest = 0)
     x_rows[0] = 0
     r[0]    = arbitrary (unused when bi=1)
     c[0]    = c_attack
     c[1..kappa-1] = garbage (will fail decryption, loop continues)
6. Serialize into pve_ciphertext_blob_v1_t → crafted_ciphertext

Call: decrypt(curve, dk, ek, crafted_ciphertext, label, out_x)

Result:
  - verify() is skipped (skip_verify=true)
  - c[0] decrypts to x_attack (valid ciphertext under ek)
  - restore_from_decrypted(0, x_attack, curve, x_out):
      x_bi = x_rows[0] = 0
      x_value = x_attack + 0 = x_attack
      x_attack * G == Q_attack  ✓  (check passes)
  - out_x = x_attack  ← attacker-chosen value returned as legitimate decryption
```
