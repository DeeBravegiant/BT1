Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool.sol` uses `totalAssetDeposits > limit` for ETH but `totalAssetDeposits + amount > limit` for ERC20 assets. This asymmetry means the ETH branch only reverts when the cap is already exceeded, never when the incoming deposit would push it over. Any unprivileged ETH depositor can bypass the configured deposit ceiling.

## Finding Description
The root cause is at line 679 of `contracts/LRTDepositPool.sol`:

```solidity
// L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
``` [1](#0-0) 

The ETH branch tests whether the existing total already exceeds the limit, not whether adding `amount` would exceed it. The ERC20 branch correctly includes `amount`.

`_beforeDeposit` (lines 661–663) relies entirely on this function's return value to gate minting:

```solidity
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [2](#0-1) 

The public entry point `depositETH` (lines 76–93) passes `msg.value` as `depositAmount` to `_beforeDeposit`, making this reachable by any external caller with no special privileges: [3](#0-2) 

No other guard exists between `depositETH` and `_mintRsETH` that would catch the over-limit ETH deposit.

## Impact Explanation
The deposit limit is a protocol-enforced invariant. When `totalAssetDeposits` is below but near the cap, any ETH deposit of arbitrary size is accepted without revert, pushing total ETH deposits arbitrarily above the configured ceiling. This constitutes **Low — Contract fails to deliver promised returns** (the deposit limit invariant is broken). It escalates toward Critical (protocol insolvency) only if the limit is a hard EigenLayer capacity guard, which is not concretely proven here; the confirmed impact is the Low category.

## Likelihood Explanation
No special role, front-running, oracle manipulation, or victim mistake is required. The only precondition is that `totalAssetDeposits` for ETH is below the configured limit — the normal operating state of a live protocol. Any unprivileged external caller can invoke `depositETH` with a large `msg.value`. Likelihood is **Medium**.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // fixed
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Protocol accumulates `999.9 ether` in ETH deposits (`totalAssetDeposits = 999.9 ether`).
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999.9 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 500 ETH; total ETH deposits become `1499.9 ether`, 50% above the cap.
6. The equivalent ERC20 call evaluates `999.9 + 500 > 1000` → `true` → `MaximumDepositLimitReached` revert — confirming the asymmetry.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock `lrtConfig` returning `depositLimitByAsset = 1000 ether`. Seed `totalAssetDeposits` to `999.9 ether`. Call `depositETH{value: 500 ether}`. Assert the call succeeds (no revert) and `getTotalAssetDeposits(ETH_TOKEN)` returns `1499.9 ether > 1000 ether`. Then repeat with an ERC20 asset and assert `MaximumDepositLimitReached` is thrown.
