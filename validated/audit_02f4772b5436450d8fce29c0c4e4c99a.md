### Title
Unbounded O(N×M) Iteration in `ratifyTransition` Over All Proposals × DRep Distribution at Epoch Boundary — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The Conway `RATIFY` rule processes every active governance proposal at the epoch boundary by recursively calling `ratifyTransition`, which for each proposal performs a full linear scan of the entire DRep stake distribution via `dRepAcceptedRatio`. This creates O(N\_proposals × N\_dreps) synchronous work inside `finishDRepPulser`. Neither the number of active proposals nor the number of registered DReps is subject to a hard protocol-enforced cap; only economic deposits act as a deterrent. An attacker with sufficient ADA can inflate both dimensions to push epoch-boundary processing well beyond its intended time budget.

---

### Finding Description

**Root cause — `ratifyTransition` (recursive, non-tail-recursive):** [1](#0-0) 

The function pattern-matches on `RatifySignal rsig` (a `StrictSeq` of `GovActionState`), processes the head, then recursively calls `trans @(RATIFY era)` on the tail. In the `else` branch the recursive call is **not** in tail position — work is performed after it — so the call stack grows linearly with the number of proposals.

**Per-proposal full scan of DRep distribution:** [2](#0-1) 

`dRepAcceptedRatio` executes `Map.foldlWithKey' accumStake (0, 0) reDRepDistr` — a complete traversal of the DRep distribution map — for **every** proposal processed by `ratifyTransition`. The same pattern applies to `committeeAcceptedRatio` (iterates all committee members) and `spoAcceptedRatio` (iterates all stake pools).

**The RATIFY rule is not pulsed:**

The DRep distribution *computation* is spread across blocks via the `DRepPulser`: [3](#0-2) 

But `finishDRepPulser` calls `runConwayRatify` — which invokes the full `ratifyTransition` recursion — **synchronously and atomically** at the epoch boundary: [4](#0-3) 

**No hard cap on proposals or DReps:**

`govActionLifetime` is an `EpochInterval` (a `Word32`) with no enforced maximum: [5](#0-4) 

`dRepDeposit` and `govActionDeposit` are the only barriers: [6](#0-5) [7](#0-6) 

These are economic deterrents, not protocol-enforced hard caps.

**Proposals accumulate across epochs:** Each proposal lives for `govActionLifetime` epochs. With `govActionLifetime = 6` (current mainnet), proposals submitted across multiple epochs accumulate in `dpProposals` until they expire or are ratified. The snapshot passed to `ratifyTransition` is the full set of all active proposals: [8](#0-7) 

---

### Impact Explanation

The total synchronous work at the epoch boundary is **O(N\_proposals × N\_dreps)**. If an attacker inflates both dimensions:

- **N\_proposals**: submit proposals across multiple epochs; each survives for `govActionLifetime` epochs. With `govActionLifetime` set to a large value via a prior governance action, the accumulation window grows.
- **N\_dreps**: register many DReps; each requires only `dRepDeposit` ADA.

The epoch-boundary `finishDRepPulser` call — which must complete before the node can continue — takes proportionally longer. If it exceeds the node's processing budget, honest nodes may fall behind the chain tip, causing them to be unable to produce or validate blocks in time. In the worst case this leads to a chain stall or deterministic divergence between nodes with different hardware/timing characteristics.

This matches: **Medium — attacker-controlled proposals and certificates exceed intended validation limits.**

---

### Likelihood Explanation

Moderate-low. The `govActionDeposit` (currently 100,000 ADA on mainnet) and `dRepDeposit` (500 ADA) impose real economic cost. However:

1. A well-funded attacker or a coalition can afford it.
2. `govActionLifetime` can be increased via a prior governance action, multiplying the accumulation window without additional per-proposal cost.
3. The attack is amplified by the quadratic interaction: doubling both proposals and DReps quadruples the epoch-boundary work.

---

### Recommendation

1. **Hard cap on active proposals**: Enforce a protocol-level maximum on the total number of simultaneously active proposals (e.g., via a `maxActiveProposals` protocol parameter), rejecting new proposals when the cap is reached.
2. **Pulse the RATIFY rule**: Apply the same pulsing strategy used for DRep distribution computation to the `ratifyTransition` loop, spreading per-proposal vote-counting across blocks rather than executing it atomically.
3. **Cap `govActionLifetime`**: Enforce an upper bound on `govActionLifetime` in the `canUpdate` / `ppuWellFormed` validation to prevent governance from setting an arbitrarily large accumulation window.
4. **Bound DRep count**: Consider a `maxDReps` protocol parameter, or ensure that `dRepDeposit` is calibrated against the per-DRep cost imposed on every epoch boundary.

---

### Proof of Concept

1. Governance enacts a `ParameterChange` raising `govActionLifetime` to a large value (e.g., 100 epochs) and lowering `govActionDeposit` to a small value.
2. Attacker registers K DReps (cost: K × `dRepDeposit`).
3. Attacker submits P proposals per epoch over multiple epochs (cost: P × `govActionDeposit` per epoch). After L epochs, up to P × L proposals are simultaneously active.
4. At the next epoch boundary, `finishDRepPulser` calls `runConwayRatify` → `ratifyTransition` → for each of the P × L proposals, `dRepAcceptedRatio` scans all K DReps.
5. Total work: O(P × L × K). With P=50, L=100, K=10,000 this is 50,000,000 map traversal steps executed synchronously, stalling all nodes at the epoch boundary. [9](#0-8) [10](#0-9)

dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr

ratifyTransition ::
  forall era.
  ( Embed (EraRule "ENACT" era) (RATIFY era)
  , State (EraRule "ENACT" era) ~ EnactState era
  , Environment (EraRule "ENACT" era) ~ ()
  , Signal (EraRule "ENACT" era) ~ EnactSignal era
  , ConwayEraPParams era
  , ConwayEraAccounts era
  ) =>
  TransitionRule (RATIFY era)
ratifyTransition = do
  TRC
    ( env@RatifyEnv {reCurrentEpoch}
      , st@( RatifyState
               rsEnactState@EnactState
                 { ensCurPParams
                 , ensTreasury
                 , ensPrevGovActionIds
                 }
               _rsEnacted
               _rsExpired
               rsDelayed
             )
      , RatifySignal rsig
      ) <-
    judgmentContext
  case rsig of
    gas@GovActionState {gasId, gasExpiresAfter} SSeq.:<| sigs -> do
      let govAction = gasAction gas
      if prevActionAsExpected gas ensPrevGovActionIds
        && validCommitteeTerm govAction ensCurPParams reCurrentEpoch
        && not rsDelayed
        && withdrawalCanWithdraw govAction ensTreasury
        && acceptedByEveryone env st gas
        then do
          newEnactState <-
            trans @(EraRule "ENACT" era) $
              TRC ((), rsEnactState, EnactSignal gasId govAction)
          let
            st' =
              st
                & rsEnactStateL .~ newEnactState
                & rsDelayedL .~ delayingAction govAction
                & rsEnactedL %~ (Seq.:|> gas)
          trans @(RATIFY era) $ TRC (env, st', RatifySignal sigs)
        else do
          -- This action hasn't been ratified yet. Process the remaining actions.
          st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
          -- Finally, filter out actions that have expired.
          if gasExpiresAfter < reCurrentEpoch
            then pure $ st' & rsExpiredL %~ Set.insert gasId
            else pure st'
    SSeq.Empty -> pure $ st & rsEnactStateL . ensTreasuryL .~ Coin 0
