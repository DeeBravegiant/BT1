Audit Report

## Title
`RSETHPriceFeed.latestRoundData()` Returns ETH/USD Round Metadata Masking rsETH Price Staleness, and `getRoundData()` Returns Incorrect Historical Prices - (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed` implements `AggregatorV3Interface` as the rsETH/USD price feed for external consumption. In `latestRoundData()`, the `updatedAt` and `answeredInRound` fields are sourced from the ETH/USD Chainlink feed while `answer` is computed from the current `LRTOracle.rsETHPrice`, meaning standard heartbeat staleness checks by consumers validate ETH/USD freshness rather than rsETH oracle freshness. In `getRoundData(_roundId)`, a historical ETH/USD price is multiplied by the **current** rsETH/ETH rate, producing a structurally incorrect historical price for every possible `_roundId`. The contract fails to deliver the accurate price data it is deployed to provide.

## Finding Description
In `RSETHPriceFeed.latestRoundData()` (lines 63–70):

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

All five return fields except `answer` originate from the ETH/USD Chainlink feed. `updatedAt` reflects when the ETH/USD feed last updated, not when `LRTOracle.rsETHPrice` was last written. `LRTOracle` stores `rsETHPrice` as a plain `uint256` with no accompanying timestamp — there is no `rsETHPriceUpdatedAt` variable. `updateRSETHPrice()` is public with only a `whenNotPaused` guard and no on-chain enforcement of update frequency. If the keeper is offline, `rsETHPrice` can be arbitrarily stale while `updatedAt` from the ETH/USD feed remains recent, causing any consumer performing `block.timestamp - updatedAt < heartbeat` to incorrectly conclude the rsETH price is fresh.

In `RSETHPriceFeed.getRoundData(_roundId)` (lines 53–61):

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

This fetches the ETH/USD price at a historical round but multiplies it by the **current** `rsETHPrice`. The returned `answer` is unconditionally incorrect for every historical round — it is a synthetic value mixing a past ETH/USD price with a present rsETH/ETH rate, not the actual rsETH/USD price at that round.

Existing checks are insufficient: `LRTOracle` has no timestamp tracking for `rsETHPrice` updates, and `RSETHPriceFeed` performs no validation of the age of `rsETHPrice` before returning it.

## Impact Explanation
The contract fails to deliver the accurate price data it is deployed to provide. `latestRoundData()` returns a `updatedAt` that does not reflect rsETH oracle freshness, and `getRoundData()` always returns a structurally incorrect historical price. This maps to **Low — Contract fails to deliver promised returns**: the oracle contract does not lose value itself, but it does not fulfill its core function of providing accurate rsETH/USD price data with correct freshness metadata to external consumers.

## Likelihood Explanation
The `getRoundData()` historical price corruption is unconditional — it is always incorrect for any `_roundId` regardless of oracle state. The `latestRoundData()` staleness masking requires a keeper gap (keeper offline, network congestion, or low-activity period), which is a realistic operational scenario given that `updateRSETHPrice()` has no on-chain enforcement of update frequency. Any external caller can trigger both functions.

## Recommendation
1. **`latestRoundData()`**: Store a `rsETHPriceUpdatedAt` timestamp in `LRTOracle` that is written alongside `rsETHPrice = newRsETHPrice` in `_updateRsETHPrice()`. In `RSETHPriceFeed.latestRoundData()`, return `min(ethToUsdUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt` so consumers' staleness checks reflect the freshness of the least-recently-updated component.
2. **`getRoundData()`**: Revert with `NotSupported()` since accurate historical rsETH/USD prices cannot be reconstructed without a historical rsETH price index. Alternatively, document clearly that historical round data is not meaningful and remove the function from the interface.
3. Consider storing a monotonically increasing `rsETHRoundId` in `LRTOracle` that increments on each `updateRSETHPrice()` call and returning it as `answeredInRound` in `latestRoundData()`.

## Proof of Concept
**Staleness masking (`latestRoundData`):**
1. At `T=0`, `LRTOracle.updateRSETHPrice()` is called; `rsETHPrice = 1.05e18`. ETH/USD `updatedAt = T`.
2. At `T+25h`, ETH/USD feed updates normally; `updatedAt = T+25h`. Keeper is offline; `rsETHPrice` unchanged.
3. Consumer calls `RSETHPriceFeed.latestRoundData()`.
4. Returned: `updatedAt = T+25h` (ETH/USD), `answer = 1.05e18 * currentEthUsdPrice / 1e18` (25h-stale rsETH rate).
5. Consumer checks `block.timestamp - updatedAt < heartbeat` → passes. Stale rsETH price accepted as fresh.

**Unconditional historical price corruption (`getRoundData`):**
1. At round `R` (historical), ETH/USD = 2000e8, rsETH/ETH = 1.02. True rsETH/USD at R = 2040e8.
2. Today, rsETH/ETH = 1.08.
3. Consumer calls `RSETHPriceFeed.getRoundData(R)`.
4. Returned: `answer = 1.08e18 * 2000e8 / 1e18 = 2160e8` — not the actual 2040e8 at round R.
5. Any TWAP or historical verification using this value is corrupted.
