Audit Report

## Title
Unbounded Withdrawal Queue Enables Temporary Freezing of Legitimate Withdrawals via Minimum-Amount Request Flooding - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`initiateWithdrawal` accepts any non-zero rsETH amount with no per-user cap on pending requests, and `minRsEthAmountToWithdraw` defaults to zero (effectively allowing 1-wei requests). Because `_unlockWithdrawalRequests` processes the global FIFO queue with `break` semantics on both the delay check and the asset-availability check, an attacker who floods the queue before a legitimate user forces the operator to exhaust all attacker entries before the legitimate user's request can ever be unlocked, temporarily freezing the legitimate user's rsETH in the contract.

## Finding Description
`minRsEthAmountToWithdraw` is a mapping that is never initialized in `initialize()`, so it defaults to zero for every asset. The only guard in `initiateWithdrawal` is `rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]` (line 162), which with a zero minimum reduces to rejecting only exact-zero amounts — 1-wei requests pass freely.

`_addUserWithdrawalRequest` (line 744) appends to the global nonce counter and to `userAssociatedNonces` with no cap check (lines 755–757). The `DoubleEndedQueue` only reverts on a uint128 wraparound, so the per-user queue is effectively unlimited.

`_unlockWithdrawalRequests` (lines 790–815) iterates `nextLockedNonce_` upward in strict FIFO order. Two conditions halt the entire loop with `break`:
- Line 795: `block.number < request.withdrawalStartBlock + withdrawalDelayBlocks` — delay not elapsed.
- Line 800: `availableAssetAmount < payoutAmount` — insufficient assets.

Neither condition skips an entry; both stop all further processing. Therefore every attacker-inserted entry at the front of the queue must be individually processed (one storage read + multiple writes per iteration) before `nextLockedNonce` advances past them to reach any later legitimate request.

The `ExceedAmountToWithdraw` check at line 170 limits each individual request to available assets, but with 1-wei rsETH the `expectedAssetAmount` is approximately 1 wei of the underlying, so the attacker can submit up to `totalAssets_in_wei` requests — astronomically many for a protocol holding thousands of ETH.

## Impact Explanation
**Temporary freezing of funds (Medium):** A legitimate user's rsETH is transferred into the contract at line 166 upon `initiateWithdrawal`. It cannot be recovered until `completeWithdrawal` succeeds, which requires `usersFirstWithdrawalRequestNonce < nextLockedNonce[asset]` (line 707). `nextLockedNonce` only advances through `_unlockWithdrawalRequests`, which must sequentially exhaust all preceding attacker entries. With N attacker entries ahead, the legitimate user's funds are frozen until the operator completes ceil(N / batchSize) `unlockQueue` calls. The attacker can reclaim their rsETH after the 8-day delay and resubmit, sustaining the freeze indefinitely.

**Unbounded gas consumption (Medium):** Each `unlockQueue` call pays gas proportional to the number of entries processed. With `minRsEthAmountToWithdraw = 0`, N is bounded only by total protocol assets in wei, making total operator work O(N) with no upper bound enforced by the contract.

## Likelihood Explanation
- `minRsEthAmountToWithdraw` is unset by default and requires an explicit admin call to configure; many deployments will leave it at zero.
- Even when set to a small value (e.g., 0.001 ETH), an attacker holding 1 ETH of rsETH can submit 1,000 requests. The rsETH is returned after 8 days, so the recurring cost is only gas.
- `initiateWithdrawal` is permissionless — any rsETH holder can execute this attack.
- The attack is repeatable: after each 8-day cycle the attacker reclaims their rsETH and re-floods the queue.

## Recommendation
1. **Enforce a non-zero `minRsEthAmountToWithdraw`** at initialization time; require it to be set before an asset is usable for withdrawals, rather than defaulting to zero.
2. **Add a per-user cap** on simultaneously pending withdrawal requests: `require(userAssociatedNonces[asset][msg.sender].length() < MAX_PENDING_PER_USER)` in `_addUserWithdrawalRequest`.
3. Consider a **small non-refundable protocol fee** on `initiateWithdrawal` to raise the economic cost of flooding the queue.

## Proof of Concept
```solidity
// Precondition: minRsEthAmountToWithdraw[asset] == 0 (default)
// Attacker holds N wei of rsETH (or N * minAmount if set to a small value)

for (uint i = 0; i < N; i++) {
    rsETH.approve(address(lrtWithdrawalManager), 1);
    lrtWithdrawalManager.initiateWithdrawal(asset, 1, "");
}

// Legitimate user submits — their request is at global nonce N
rsETH.approve(address(lrtWithdrawalManager), largeAmount);
lrtWithdrawalManager.initiateWithdrawal(asset, largeAmount, ""); // victim at position N

// After 8-day delay, operator must call unlockQueue ceil(N / batchSize) times
// before nextLockedNonce ever reaches the victim's entry.
// Victim's rsETH remains locked in the contract throughout.

// Attacker calls completeWithdrawal after delay, reclaims rsETH, and repeats.
```

The only cost to the attacker is gas per `initiateWithdrawal` call and the 8-day opportunity cost of locking 1 wei per request. With `minRsEthAmountToWithdraw == 0`, this is economically negligible relative to the disruption caused.
