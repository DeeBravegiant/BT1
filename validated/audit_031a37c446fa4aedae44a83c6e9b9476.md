### Title
Missing Caller Identity Binding in `verify_foreign_transaction` Allows Any Party to Acquire the Same MPC Signature and Execute Invalid Bridge Claims - (File: crates/contract/src/lib.rs)

### Summary

`verify_foreign_transaction` discards the caller's identity (`predecessor`) when constructing the `VerifyForeignTransactionRequest` map key. Unlike `sign()`, which cryptographically binds the caller via a derived tweak, the foreign-tx flow produces a caller-agnostic signature. Any unprivileged account can submit an identical request and receive the same MPC-signed attestation, enabling front-running attacks that allow an adversary to claim bridge funds belonging to the legitimate requester.

### Finding Description

**`sign()` correctly binds the caller** by capturing `predecessor` from `check_request_preconditions` and passing it into `SignatureRequest::new`, which calls `derive_tweak(predecessor_id, path)`. The resulting `tweak` is part of the map key, so two different callers with the same payload and path produce *different* pending-request entries and *different* derived signatures. [1](#0-0) 

**`verify_foreign_transaction()` silently drops the caller identity.** The return value of `check_request_preconditions` — which includes the validated `predecessor` — is discarded entirely (no `let (_, predecessor) = …` binding). The request is then built by `args_into_verify_foreign_tx_request`, which copies only `{domain_id, request, payload_version}` with no caller field. [2](#0-1) 

`VerifyForeignTransactionRequest` has no `predecessor`, `tweak`, or any caller-identifying field: [3](#0-2) 

`args_into_verify_foreign_tx_request` confirms the caller is never included: [4](#0-3) 

The MPC nodes sign with the **root key** (no per-caller derivation), so the `VerifyForeignTransactionResponse` — `{payload_hash, signature}` — is identical for every account that submits the same foreign-tx request: [5](#0-4) 

A contract test explicitly documents that two different callers (alice, bob) submitting the same request share one pending-request queue entry and both receive the same response when a single `respond_verify_foreign_tx` is delivered: [6](#0-5) 

The design documentation (`docs/foreign-chain-transactions.md`) describes a planned tweak-derivation step for `verify_foreign_transaction` using a foreign-tx-specific prefix and a `derivation_path` argument, but neither the `VerifyForeignTransactionRequestArgs` struct nor the contract method implements this — the `derivation_path` field is absent from the live struct: [7](#0-6) 

### Impact Explanation

A bridge contract built on `verify_foreign_transaction` receives a `VerifyForeignTransactionResponse` and uses it to release funds on NEAR. Because the MPC signature is not bound to any caller, an adversary who obtains the same response (by submitting an identical request) can present it to the bridge contract and claim funds that belong to the legitimate requester. This constitutes **invalid bridge execution** — the bridge releases assets to an unauthorized party based on a valid but caller-agnostic MPC attestation. The MPC network itself is not compromised; the vulnerability is that the contract produces an undifferentiated attestation that any observer can exploit.

### Likelihood Explanation

The attacker only needs to:
1. Monitor the NEAR mempool (or on-chain receipts) for `verify_foreign_transaction` calls.
2. Submit the same `ForeignChainRpcRequest` before the MPC nodes respond (or independently, since the signature is reusable once obtained).
3. Present the received `VerifyForeignTransactionResponse` to a bridge contract that does not independently verify the claimer's identity against the foreign-chain transaction's recipient.

Step 1 is trivial on a public blockchain. Step 2 is straightforward given NEAR's block times. Step 3 is the realistic deployment scenario for bridge contracts that rely on the MPC attestation as the sole authorization signal. Likelihood is **medium-high**.

### Recommendation

Bind the caller's identity to the `VerifyForeignTransactionRequest` key, mirroring the `sign()` pattern:

1. Add a `derivation_path: String` field to `VerifyForeignTransactionRequestArgs` (as the design doc already specifies).
2. Capture `predecessor` from `check_request_preconditions` in `verify_foreign_transaction`.
3. Derive a caller-specific tweak using a foreign-tx-specific prefix (e.g., `"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`) and include it in `VerifyForeignTransactionRequest`.
4. Have MPC nodes sign with the derived key (not the root key), so the resulting signature is only valid for the specific `(predecessor, derivation_path)` pair.

This ensures that two different callers submitting the same foreign-chain transaction produce different pending-request entries and different signatures, eliminating the free-riding and front-running attack surface.

### Proof of Concept

```
1. Alice calls verify_foreign_transaction({bitcoin_tx_id: X, ...})
   → yield_A is created; pending_verify_foreign_tx_requests[Request{tx_id:X}] = [yield_A]

2. Bob (attacker) monitors mempool, sees Alice's call, submits identical args:
   verify_foreign_transaction({bitcoin_tx_id: X, ...})
   → yield_B is queued; pending_verify_foreign_tx_requests[Request{tx_id:X}] = [yield_A, yield_B]

3. MPC nodes call respond_verify_foreign_tx(Request{tx_id:X}, response_S)
   → resolve_yields_for drains both yield_A and yield_B with the same response_S
   → Alice's NEAR tx receives response_S
   → Bob's NEAR tx receives response_S  (identical signature)

4. Bob immediately calls bridge_contract.claim(response_S)
   → bridge releases Alice's locked NEAR funds to Bob

5. Alice's subsequent claim(response_S) is rejected (funds already claimed)
   → Alice loses her funds despite having initiated the legitimate verification
```

The root cause — `predecessor` discarded at line 526-531 and absent from `VerifyForeignTransactionRequest` — is a necessary step in this attack; without it the signatures would be caller-specific and non-transferable.
