Audit Report

## Title
Merkle Proof Leaf Does Not Bind to Contract Address, Enabling Cross-Instance Replay Claims - (File: contracts/KERNEL/KernelTop100MerkleDistributor.sol)

## Summary
`KernelTop100MerkleDistributor._verifyClaimProof()` computes the merkle leaf as `keccak256(abi.encodePacked(user, amount))` with no contract address or distributor-specific identifier in the preimage. If two instances are deployed sharing the same merkle root, a user's valid proof on instance A is equally valid on instance B, and because each instance maintains an independent `userClaims` mapping, the user can claim their full allocation from every such instance.

## Finding Description
In `_verifyClaimProof()` at line 293, the leaf is constructed as:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
```

No `address(this)`, `block.chainid`, or any distributor-specific salt is included. The merkle root is immutably fixed at initialization (line 216: `merkleRoot = _merkleRoot`) and never updated. Claim state is tracked per-instance in `userClaims[user]` (line 158), which is a fresh mapping on every new deployment.

Because the leaf commits only to `(user, amount)`, a proof `π` satisfying `MerkleProof.verify(π, R, keccak256(abi.encodePacked(alice, 100))) == true` on instance A is identically valid on any instance B initialized with the same root `R`. Instance B's `userClaims[alice].amountClaimed` starts at zero, so `_getUnclaimedVestedAmount` returns the full vested amount again, and the `claim()` call at lines 310–338 succeeds, transferring tokens a second time.

No existing guard prevents this: `_verifyClaimProof` only checks proof validity against the root (line 294), and `_getUnclaimedVestedAmount` only checks the local `userClaims` mapping (line 239). There is no cross-instance state.

## Impact Explanation
An eligible user can drain KERNEL tokens from every deployed instance of `KernelTop100MerkleDistributor` that shares the same merkle root, receiving a multiple of their legitimate allocation. This constitutes direct **theft of unclaimed yield** — matching the allowed High-severity impact class.

## Likelihood Explanation
The contract is an upgradeable, initializable distributor explicitly designed to be deployed as independent instances. Reusing the same merkle root across deployments (e.g., same Top-100 eligible set for a second vesting tranche, or a replacement deployment after reconfiguration) is a natural operational pattern. No privileged access is required; any address present in the merkle tree can execute the replay by calling the public `claim()` function on each instance. The attack is repeatable across every instance sharing root `R`.

**Likelihood: Medium.**

## Recommendation
Bind the leaf to the specific contract instance by including `address(this)` (and optionally `block.chainid`) in the preimage:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(address(this), block.chainid, user, amount));
```

This cryptographically invalidates any proof generated for one distributor instance when submitted to another, even if both share the same root. The off-chain merkle tree generation must be updated to match.

## Proof of Concept
1. Deploy **instance A** (`KernelTop100MerkleDistributor`) with `merkleRoot = R`, funded with 1,000 KERNEL.
2. Deploy **instance B** with the same `merkleRoot = R` for a second vesting tranche, funded with another 1,000 KERNEL.
3. Alice is in the merkle tree with `amount = 100`. She holds proof `π` such that `MerkleProofUpgradeable.verify(π, R, keccak256(abi.encodePacked(alice, 100))) == true`.
4. Alice calls `claim(100, π)` on **instance A** → `_verifyClaimProof` passes (line 293–294); `userClaims[alice].amountClaimed` on A is updated; Alice receives ~100 KERNEL.
5. Alice calls `claim(100, π)` on **instance B** → `_verifyClaimProof` passes again with the identical leaf and root; `userClaims[alice].amountClaimed` on B is 0; `_getUnclaimedVestedAmount` returns the full vested amount; Alice receives another ~100 KERNEL.
6. Alice has claimed ~200 KERNEL instead of the intended ~100, draining instance B's allocation for her entry.

**Foundry test sketch:**
```solidity
function test_crossInstanceReplay() public {
    // Deploy instanceA and instanceB with same merkleRoot R
    // Fund both with 1000e18 KERNEL

    vm.prank(alice);

}
```
