Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a Synthetix-style per-token accumulator to distribute rewards. When `totalKernelStaked` reaches zero mid-period, `rewardPerToken()` freezes the accumulator while time continues to elapse. When a new staker eventually arrives, `updateReward` silently advances `updatedAt` to the current time, permanently discarding the reward tokens that accrued during the zero-staked window. There is no recovery path for these tokens.

## Finding Description
`rewardPerToken()` at L408–414 returns `rewardPerTokenStored` unchanged whenever `totalKernelStaked == 0`, without advancing `updatedAt`. The `updateReward` modifier at L232–242 always sets `updatedAt = lastTimeRewardApplicable()` regardless of whether `totalKernelStaked` is zero. This creates the gap: when the last staker calls `initiateWithdrawal`, the modifier correctly snapshots the accumulator and sets `updatedAt = now` before `totalKernelStaked` is decremented to zero in the function body (L325–326). From that point, every call to `rewardPerToken()` returns the frozen value. When a new staker later calls `stake()`, `updateReward` runs with `totalKernelStaked` still zero (the new stake is added after the modifier), so `rewardPerTokenStored` stays frozen but `updatedAt` is advanced to the current time — silently consuming the entire zero-staked interval. The reward tokens implied by `rewardRate * elapsed_zero_staked_time` are never credited to any account and cannot be recovered, as there is no admin rescue function. The `notifyRewardAmount` guard at L570 (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new period with no stakers; it does not protect against mid-period drain.

## Impact Explanation
Reward tokens emitted during the zero-staked window are permanently locked inside `KernelDepositPool`. No user can claim them (no staker existed to accumulate them), and no admin function can recover them. This constitutes **permanent freezing of unclaimed yield**, matching the allowed Medium impact. The contract's own NatSpec at L18–22 explicitly acknowledges this behavior, confirming it is a known design limitation with no on-chain enforcement.

## Likelihood Explanation
Any unprivileged user can call `initiateWithdrawal` to reduce their stake. If all stakers exit during an active reward period — a realistic scenario during market stress, a coordinated migration, or simply organic user behavior — `totalKernelStaked` reaches zero and the remaining period's rewards are locked. No privileged action, oracle manipulation, or external dependency is required. The trigger is ordinary user withdrawals alone, making this repeatable across every reward period.

## Recommendation
1. **In `rewardPerToken()`**: when `totalKernelStaked == 0`, also update `updatedAt` to `lastTimeRewardApplicable()` so the zero-staked interval is not silently consumed when stakers return. This prevents future accumulator gaps but does not recover already-locked tokens.
2. **Add an admin recovery function**: allow the admin to reclaim undistributed reward tokens (i.e., `rewardRate * (finishAt - block.timestamp)`) when `totalKernelStaked == 0`, analogous to returning unallocated rewards to the source.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000_000e18)` with `duration = 7 days`. `rewardRate ≈ 1653 tokens/sec`, `finishAt = T + 7 days`.
2. Alice is the sole staker: `balanceOf[Alice] = 1000e18`, `totalKernelStaked = 1000e18`.
3. At `T + 3 days`, Alice calls `initiateWithdrawal(1000e18)`. The `updateReward` modifier snapshots `rewardPerTokenStored` and sets `updatedAt = T+3days`. The function body sets `totalKernelStaked = 0`.
4. From `T+3days` onward, `rewardPerToken()` returns the frozen `rewardPerTokenStored`. The ~571,392 tokens for the remaining 4 days are in limbo.
5. After `withdrawalDelay`, Alice calls `claimWithdrawal` and `getReward()`, correctly receiving her 3-day share.
6. At `T+5days`, Bob calls `stake(1 wei)`. `updateReward(Bob)` runs: `totalKernelStaked` is still 0 (Bob's stake added after modifier), so `rewardPerTokenStored` stays frozen, but `updatedAt` is advanced to `T+5days`. The days 3–5 rewards (~285,696 tokens) are permanently discarded.
7. Bob earns only from `T+5days` onward. The tokens for days 3–5 remain locked in the contract with no recovery path.
