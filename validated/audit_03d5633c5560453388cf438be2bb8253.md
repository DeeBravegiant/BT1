Audit Report

## Title
Stale `rsETHPrice` Cache After `updatePriceOracleFor()` Allows Depositors to Receive Excess rsETH - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle` stores `rsETHPrice` as a manually-updated cached exchange rate. When an admin calls `updatePriceOracleFor(asset, newOracle)`, the new oracle immediately affects `getAssetPrice(asset)` (live read), but `rsETHPrice` is not refreshed. Any depositor who calls `depositAsset()` or `depositETH()` during the window between the oracle update and the next `_updateRsETHPrice()` execution receives rsETH computed using the live (higher) asset price divided by the stale (lower) `rsETHPrice`, minting more rsETH than their proportional share and diluting existing holders.

## Finding Description
`LRTOracle.rsETHPrice` is a state variable updated only inside `_updateRsETHPrice()`, which is called by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. The function `updatePriceOracleFor()` (`LRTOracle.sol:113-119`) writes `assetPriceOracle[asset] = priceOracle` and emits an event, but never calls `_updateRsETHPrice()`.

`LRTDepositPool.getRsETHAmountToMint()` (`LRTDepositPool.sol:520`) computes:
```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
`getAssetPrice(asset)` is a live call to the newly-set oracle, while `rsETHPrice()` returns the stale cached value. If the new oracle returns a higher price, the numerator increases immediately while the denominator remains at its old lower value, minting excess rsETH.

The public `updateRSETHPrice()` cannot be used to close the window when the price increase exceeds `pricePercentageLimit`: `_updateRsETHPrice()` at lines 260–265 reverts with `PriceAboveDailyThreshold` for non-managers in that case, leaving the window open until a manager explicitly calls `updateRSETHPriceAsManager()`. Even when the increase is below the threshold, the window spans at least the current block.

Existing guards (`minRSETHAmountExpected`, `nonReentrant`, deposit limits) do not protect against this: the attacker sets `minRSETHAmountExpected = 0` or any value below the inflated mint amount, and the deposit limit check uses asset amounts, not rsETH value.

## Impact Explanation
Existing rsETH holders are diluted. The oracle price increase represents yield accrued by the protocol (e.g., stETH rebasing, LST appreciation). That yield is reflected in the true `rsETHPrice` but not yet in the cached value. A depositor who acts in the window receives rsETH at the stale lower price, claiming a portion of the accrued yield that belongs to existing holders. This is **theft of unclaimed yield (High)** under the allowed impact scope.

Concrete example:
- Protocol TVL: 1050 ETH, rsETH supply: 1000, `rsETHPrice` = 1.05 ETH/rsETH
- Admin updates stETH oracle: new oracle returns 1.05 ETH/stETH (was 1.0). True new `rsETHPrice` ≈ 1.1 ETH/rsETH
- User deposits 100 stETH: minted = `(100 × 1.05) / 1.05 = 100 rsETH`; fair amount = `(100 × 1.05) / 1.1 ≈ 95.45 rsETH`
- Excess: ~4.55 rsETH stolen from existing holders, unrecoverable after minting

## Likelihood Explanation
Oracle updates are routine admin operations (upgrading oracle contracts, fixing price feeds, adding LST oracles). The exploit requires no special privileges: any user monitoring the chain for `AssetPriceOracleUpdate` events can immediately call `depositAsset()` in the same or next block. When the price jump exceeds `pricePercentageLimit`, the window is extended until a manager acts, increasing the exploitable duration. The attack is repeatable on every oracle update that increases an asset's price.

## Recommendation
Call `_updateRsETHPrice()` atomically at the end of `updatePriceOracleFor()`:

```solidity
function updatePriceOracleFor(address asset, address priceOracle) public onlyLRTAdmin {
    if (lrtConfig.isSupportedAsset(asset)) {
        UtilLib.checkNonZeroAddress(priceOracle);
    }
    assetPriceOracle[asset] = priceOracle;
    emit AssetPriceOracleUpdate(asset, priceOracle);
    _updateRsETHPrice(); // atomically sync cache
}
```

This should be called with manager privileges (or the caller must be granted manager role) to avoid reverting on large price moves. Alternatively, add a staleness guard to `depositAsset()` and `depositETH()` requiring `rsETHPrice` to have been updated within the current block.

## Proof of Concept
1. Deploy/fork with: rsETH supply = 1000e18, stETH deposits = 1050e18, `rsETHPrice` = 1.05e18 (set at block N).
2. At block N+1, admin calls `LRTOracle.updatePriceOracleFor(stETH, newOracle)` where `newOracle.getAssetPrice(stETH)` returns `1.05e18` (up from `1.0e18`). `rsETHPrice` remains `1.05e18`; true value should be `~1.1e18`.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")` at block N+1 (same or next block):
   - `getRsETHAmountToMint` returns `(100e18 × 1.05e18) / 1.05e18 = 100e18` rsETH minted.
   - Fair amount: `(100e18 × 1.05e18) / 1.1e18 ≈ 95.45e18` rsETH.
   - Attacker receives ~4.55e18 excess rsETH.
4. Admin calls `LRTOracle.updateRSETHPrice()` (or `updateRSETHPriceAsManager()` if above threshold) — cache syncs, but excess rsETH already minted and cannot be recovered.

Foundry fork test plan: fork mainnet, mock `newOracle` returning a higher price, call `updatePriceOracleFor`, then `depositAsset`, assert minted rsETH > `(depositAmount × newPrice) / trueNewRsETHPrice`, and assert existing holder share decreased.
