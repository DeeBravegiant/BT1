### Title
Harvester Can Unconditionally Redirect Farmer Block Reward to Arbitrary Address — (`File: chia/farmer/farmer_api.py`)

### Summary
A connected harvester (including third-party harvesters such as DrPlotter) can set `farmer_reward_address_override` in a `RespondSignatures` message to any arbitrary puzzle hash. The farmer accepts this value without any cryptographic authorization check and uses it verbatim as the `farmer_reward_puzzle_hash` in the `DeclareProofOfSpace` message sent to the full node, permanently diverting the 1.75 XCH farmer block reward to the attacker's address.

### Finding Description

The CHIP-22 fee mechanism allows a harvester to redirect the farmer reward by setting `farmer_reward_address_override` in the `RespondSignatures` protocol message. In `_process_respond_signatures()`, the farmer unconditionally replaces its own configured reward address with whatever the harvester provides:

```python
# chia/farmer/farmer_api.py lines 916-918
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [1](#0-0) 

This `farmer_reward_address` is then placed directly into the `DeclareProofOfSpace` message broadcast to the full node:

```python
return farmer_protocol.DeclareProofOfSpace(
    ...
    farmer_reward_address,   # ← attacker-controlled
    ...
)
``` [2](#0-1) 

The only guard in the codebase is `notify_farmer_reward_taken_by_harvester_as_fee()`, which is triggered by the *separate* `NewProofOfSpace.farmer_reward_address_override` field and only **logs a warning** — it does not block the override from being applied, and it does not gate the `RespondSignatures` path at all:

```python
# chia/farmer/farmer.py lines 920-928
self.log.warning(
    "Harvester illegitimately took a fee reward that "
    + "did not belong to it or it incorrectly applied the fee convention."
)
``` [3](#0-2) 

The `RespondSignatures` message type carries `farmer_reward_address_override` as an optional field with no associated signature or authorization proof: [4](#0-3) 

### Impact Explanation

When a block is won, the farmer reward coin (1.75 XCH at current schedule) is created with `farmer_reward_puzzle_hash` set to the attacker-supplied address. This is an on-chain, irreversible coin creation. The farmer receives nothing. This constitutes **unauthorized reward diversion** — a High-severity impact under the allowed scope ("payout redirection … affecting XCH").

### Likelihood Explanation

Any harvester connected to the farmer can exploit this. Third-party harvesters (DrPlotter and similar services) are an explicitly supported and common deployment pattern. A malicious third-party harvester operator needs only to:

1. Connect to the victim farmer as a normal harvester peer.
2. Return a valid `RespondSignatures` with `farmer_reward_address_override` set to the attacker's puzzle hash.
3. The farmer signs and broadcasts the block with the attacker's address as the reward destination.

No key compromise, no admin access, and no cryptographic break is required. The attacker only needs a valid harvester connection.

### Recommendation

The farmer must cryptographically verify that the harvester is authorized to redirect the reward before accepting `farmer_reward_address_override`. Options include:

1. **Require a signature over the override address** using the plot's local key (already held by the harvester), binding the override to the specific proof and signage point. The farmer can verify this before accepting the field.
2. **Enforce the fee-quality gate**: if `fee_quality > fee_threshold` (i.e., the harvester does not qualify for the fee under CHIP-22), reject the `RespondSignatures` message entirely rather than logging a warning and proceeding.
3. **Whitelist override addresses** in the farmer's configuration, so only pre-approved harvester reward addresses are accepted.

### Proof of Concept

1. Operate a harvester that connects to a victim farmer.
2. When the harvester wins a block-quality proof, return a `RespondSignatures` message with `farmer_reward_address_override = attacker_puzzle_hash` (any 32-byte value).
3. The farmer's `_process_respond_signatures()` at line 917 replaces `self.farmer.farmer_target` with `attacker_puzzle_hash` and constructs `DeclareProofOfSpace` with the attacker's address.
4. The full node creates the farmer reward coin to `attacker_puzzle_hash`. The farmer receives 0 XCH for the block they won. [5](#0-4) [6](#0-5)

    @metadata.request()
    async def respond_signatures(self, response: harvester_protocol.RespondSignatures) -> None:
        request = self._process_respond_signatures(response)
        if request is None:
            return None

        message: Message | None = None
        if isinstance(request, DeclareProofOfSpace):
            self.farmer.state_changed("proof", {"proof": request, "passed_filter": True})
            message = make_msg(ProtocolMessageTypes.declare_proof_of_space, request)
        if isinstance(request, SignedValues):
            message = make_msg(ProtocolMessageTypes.signed_values, request)
        await self.farmer.server.send_to_all([message], NodeType.FULL_NODE)
