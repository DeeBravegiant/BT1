Audit Report

## Title
Premature `totalKernelStaked` Decrement in `initiateWithdrawal` Inflates Rewards for Remaining Stakers at Withdrawing Users' Expense - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`initiateWithdrawal` immediately decrements both `balanceOf[msg.sender]` and `totalKernelStaked` at call time, even though the KERNEL tokens remain locked in the contract for up to 30 days until `claimWithdrawal` is called. Because `rewardPerToken()` divides by `totalKernelStaked` and `earned()` multiplies by `balanceOf[_account]`, the withdrawing user earns zero rewards during the entire mandatory lock period while remaining stakers receive an inflated reward rate on the same fixed `rewardRate`.

## Finding Description
In `initiateWithdrawal`, the `updateReward(msg.sender)` modifier runs first and correctly snapshots the user's accrued rewards up to that moment into `rewards[msg.sender]`. Immediately after, the function decrements:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L325-326
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

The tokens are not transferred — they remain in the contract until `claimWithdrawal` is called after `withdrawalDelay` (up to `MAX_WITHDRAWAL_DELAY = 30 days`, L35). From this point forward:

- `rewardPerToken()` (L412-413) divides by the now-reduced `totalKernelStaked`, inflating the per-token rate for all remaining stakers.
- `earned(alice)` (L422) computes `balanceOf[alice] * (rewardPerToken() - userRewardPerTokenPaid[alice])`, which is `0 * X = 0` since `balanceOf[alice]` was zeroed. Alice accrues no further rewards.

The withdrawing user's tokens are economically captive — they cannot be restaked, transferred, or used elsewhere — yet they are excluded from both the reward denominator and the user's reward accumulation for the full lock window.

## Impact Explanation
**High — Theft of unclaimed yield.** A user who calls `initiateWithdrawal` unconditionally forfeits all rewards accruing during the mandatory lock period. Those rewards are not burned; they are redistributed to remaining stakers via the inflated `rewardPerToken`. The magnitude is: `rewardRate × withdrawalDelay × (withdrawnAmount / totalKernelStaked_before)`. With a 30-day delay and a meaningful stake, this is a material, concrete loss of yield that falls squarely within the "Theft of unclaimed yield" impact class.

## Likelihood Explanation
**High.** Every user who calls `initiateWithdrawal` is unconditionally affected. No special conditions, no front-running, no admin involvement required. The withdrawal delay is a mandatory protocol parameter enforced on-chain (up to 30 days). This is a normal, expected user action on the critical path of the protocol, making the yield loss window large and the impact certain for every withdrawing user.

## Recommendation
Track tokens in the pending-withdrawal state separately (e.g., `totalPendingWithdrawal`) and include them in the reward denominator:

```solidity
function rewardPerToken() public view returns (uint256) {
    uint256 effectiveStake = totalKernelStaked + totalPendingWithdrawal;
    if (effectiveStake == 0) return rewardPerTokenStored;
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / effectiveStake;
}
```

Maintain a per-user `pendingWithdrawalBalance` mapping and include it in `earned()`. Decrement `totalPendingWithdrawal` only when `claimWithdrawal` actually transfers tokens out. This ensures withdrawing users continue to earn rewards proportional to their locked capital throughout the delay window.

## Proof of Concept
**Setup:** Alice and Bob each stake 1000 KERNEL. `totalKernelStaked = 2000`. Admin calls `notifyRewardAmount` with `rewardRate = 1 token/second` and `duration = 30 days`.

**Step 1:** Alice calls `initiateWithdrawal(1000)`.
- `updateReward(alice)` snapshots Alice's rewards to date (0, since no time has passed).
- `balanceOf[Alice] = 0`, `totalKernelStaked = 1000` (immediately, L325-326).
- Alice's 1000 KERNEL remain locked in the contract.

**Step 2:** For the next 30 days, `rewardPerToken()` computes:
```
rewardRate * elapsed * DECIMAL_PRECISION / totalKernelStaked
= 1 * 2592000 * 1e18 / 1000   // Bob receives ALL rewards
```

**Step 3:** Bob calls `getReward()` and receives the full 2,592,000 reward tokens.

**Step 4:** Alice calls `claimWithdrawal` and receives only her 1000 KERNEL principal — zero rewards for 30 days of locked, inaccessible capital.

**Expected:** Alice should have earned ~50% of rewards (~1,296,000 tokens) during the lock period since her tokens were still locked and unavailable to her. Instead, Bob received 100% of rewards.
