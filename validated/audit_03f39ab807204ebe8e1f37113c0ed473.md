### Title
`get_sortable_validator_online_ratio` Ignores `endorsement_cutoff_threshold`, Causing Incorrect Validator Kickout Exemption Ordering - (File: `chain/epoch-manager/src/validator_stats.rs`)

### Summary

`get_sortable_validator_online_ratio` always passes `None` for `endorsement_cutoff_threshold` when calling `get_validator_online_ratio`, while the reward calculation uses the actual configured threshold. This is the direct analog to the LoopFi bug: a public/wrapper function silently drops a parameter that the internal function correctly applies, causing two code paths to compute different values for the same logical quantity.

### Finding Description

In `chain/epoch-manager/src/validator_stats.rs`, `get_sortable_validator_online_ratio` is defined as:

```rust
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);  // always None
    ...
}
``` [1](#0-0) 

It hardcodes `None` for `endorsement_cutoff_threshold`, discarding the protocol-configured threshold entirely.

This function is called inside `compute_validators_to_reward_and_kickout` to sort validators by online ratio for the kickout-exemption decision:

```rust
let mut sorted_validators = validator_block_chunk_stats
    .iter()
    .map(|(account, stats)| (get_sortable_validator_online_ratio(stats), account))
    .collect_vec();
sorted_validators.sort_by(validator_comparator);
``` [2](#0-1) 

The sorted order feeds directly into `compute_exempted_kickout`, which walks from highest to lowest online ratio and exempts validators until a stake threshold is met: [3](#0-2) 

Meanwhile, the reward calculator uses the **correct** call with the actual threshold:

```rust
let production_ratio =
    get_validator_online_ratio(&stats, online_thresholds.endorsement_cutoff_threshold);
``` [4](#0-3) 

The `endorsement_cutoff_threshold` parameter converts a raw endorsement ratio into a binary 0-or-1 value: if a validator's endorsement ratio is below the threshold it is treated as 0, otherwise as 1. [5](#0-4) 

**Concrete divergence example** (threshold = 50%):

| Validator | Raw endorsement | With cutoff (reward path) | Without cutoff (sort path) |
|---|---|---|---|
| A | 45% | 0 (below cutoff → treated as 0) | 0.45 |
| B | 55% | 1 (above cutoff → treated as 1) | 0.55 |

Validator B has a higher actual uptime under the cutoff rule (1.0 vs 0.0 for A), but the sort path sees them as nearly equal (0.55 vs 0.45). The exemption order is therefore wrong relative to the actual uptime used for rewards and kickout decisions.

### Impact Explanation

The kickout-exemption mechanism is the protocol's safety valve: it prevents more than `validator_max_kickout_stake_perc` of total stake from being kicked out in a single epoch. The exempted set is supposed to contain the highest-uptime validators. When the sort uses a different (uncutoff-adjusted) ratio, the wrong validators may be exempted:

- A validator with high cutoff-adjusted uptime (endorsement just above threshold → treated as 1.0) is sorted lower than its true rank, making it less likely to be exempted and more likely to be kicked out.
- A validator with low cutoff-adjusted uptime (endorsement just below threshold → treated as 0) is sorted higher than its true rank, making it more likely to be exempted and protected from kickout.

The corrupted value is the **validator set** written into `EpochInfo` at the epoch boundary — a consensus-critical state root component. [6](#0-5) 

### Likelihood Explanation

The production epoch config JSON files currently do not include `endorsement_cutoff_threshold`, meaning it defaults to `None` and the two code paths produce identical results today. The bug is latent. However:

1. The parameter is a first-class field of `ValidatorOnlineThresholds` and is exercised by multiple unit tests with `Some(threshold)` values.
2. The code comment in `ValidatorOnlineThresholds` explicitly describes the intended semantics of the cutoff.
3. If the parameter is activated via a protocol upgrade (the natural next step given the existing infrastructure), the bug immediately corrupts kickout-exemption ordering every epoch. [7](#0-6) 

### Recommendation

`get_sortable_validator_online_ratio` should accept and forward the `endorsement_cutoff_threshold` parameter:

```rust
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    ...
}
```

And the call site in `compute_validators_to_reward_and_kickout` must pass the actual configured threshold from `EpochConfig`.

### Proof of Concept

Given `endorsement_cutoff_threshold = Some(50)` and two chunk-validator-only validators:

- **V1**: endorsement ratio = 49/100 → cutoff-adjusted = 0, raw = 0.49
- **V2**: endorsement ratio = 51/100 → cutoff-adjusted = 1, raw = 0.51

Both fail the `chunk_validator_only_kickout_threshold` (say 70%). `validator_max_kickout_stake_perc = 30` means only one can be kicked out. The exemption should protect V2 (higher cutoff-adjusted uptime = 1.0). But `get_sortable_validator_online_ratio` sorts by raw ratio: V2 (0.51) > V1 (0.49), so V2 is exempted — which happens to be correct in this case. However, if V1 had raw ratio 0.60 (but still below cutoff → adjusted = 0) and V2 had raw ratio 0.51 (above cutoff → adjusted = 1.0), the sort would exempt V1 (0.60 > 0.51) while V2 — the validator with the higher actual uptime — gets kicked out. This directly corrupts the validator set stored in `EpochInfo`. [8](#0-7)

/// Computes the overall online (uptime) ratio of the validator for sorting.
/// The reason for this function is that U256 used in the core implementation
/// cannot be used with `Ratio<U256>` for sorting since it does not implement `num_integer::Integer`.
/// Instead of having a full-blown implementation of `U256`` for `num_integer::Integer`
/// we wrap the value in a `BigInt` for now.
/// TODO: Implement `num_integer::Integer` for `U256` and remove this function.
/// cspell:words bigdenom bignumer
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);
    let mut bytes: [u8; size_of::<U256>()] = [0; size_of::<U256>()];
    ratio.numer().to_little_endian(&mut bytes);
    let bignumer = BigUint::from_bytes_le(&bytes);
    ratio.denom().to_little_endian(&mut bytes);
    let bigdenom = BigUint::from_bytes_le(&bytes);
    BigRational::new(bignumer.try_into().unwrap(), bigdenom.try_into().unwrap())
}

fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    let (numer, denom) = if stats.expected == 0 {
        debug_assert_eq!(stats.produced, 0);
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
    (numer, denom)
}

```rust
    fn compute_exempted_kickout(
        epoch_info: &EpochInfo,
        accounts_sorted_by_online_ratio: &Vec<AccountId>,
        total_stake: Balance,
        exempt_perc: u8,
        prev_validator_kickout: &HashMap<AccountId, ValidatorKickoutReason>,
    ) -> HashSet<AccountId> {
        // We want to make sure the total stake of validators that will be kicked out in this epoch doesn't exceed
        // config.validator_max_kickout_stake_ratio of total stake.
        // To achieve that, we sort all validators by their average uptime (average of block and chunk
        // uptime) and add validators to `exempted_validators` one by one, from high uptime to low uptime,
        // until the total excepted stake exceeds the ratio of total stake that we need to keep.
        // Later when we perform the check to kick out validators, we don't kick out validators in
        // exempted_validators.
        let mut exempted_validators = HashSet::new();
        let min_keep_stake = Balance::from_yoctonear(
            (U256::from(total_stake.as_yoctonear()) * U256::from(exempt_perc as u128)
                / U256::from(100u128))
            .as_u128(),
        );
        let mut exempted_stake = Balance::ZERO;
        for account_id in accounts_sorted_by_online_ratio.into_iter().rev() {
            if exempted_stake >= min_keep_stake {
                break;
            }
            if !prev_validator_kickout.contains_key(account_id) {
                let validator_stake = epoch_info
                    .get_validator_by_account(account_id)
                    .map(|v| v.stake())
                    .unwrap_or_default();
                exempted_stake = exempted_stake.checked_add(validator_stake).unwrap();
                exempted_validators.insert(account_id.clone());
            }
        }
        exempted_validators
    }
