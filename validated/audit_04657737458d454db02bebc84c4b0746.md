### Title
Forged Foreign-Chain Verification via Unchecked `payload_hash` in `respond_verify_foreign_tx` - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`, but never checks that `payload_hash` is the correct hash for the pending `request`. A Byzantine attested participant (below threshold) can replay any previously produced root-key signature over an arbitrary `payload_hash` to resolve a completely different pending foreign-tx request, delivering a forged attestation to the caller.

### Finding Description

In `respond` (the regular sign path), the payload hash used for signature verification is taken from the **original request** stored in the contract — it is trusted and cannot be substituted:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(signature_response, payload_hash, &expected_public_key)
``` [1](#0-0) 

In `respond_verify_foreign_tx`, the payload hash is taken from the **response submitted by the node** — it is fully attacker-controlled:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [2](#0-1) 

After this check passes, the contract resolves the pending yield for the **original request key** with the **attacker-supplied response**:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [3](#0-2) 

The contract never verifies that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific `request` being resolved. The correct hash is defined by the sign-payload serialization spec: [4](#0-3) 

The `args_into_verify_foreign_tx_request` conversion also confirms no hash binding is established at submission time: [5](#0-4) 

### Impact Explanation

A Byzantine attested participant who has observed any prior legitimate `verify_foreign_transaction` response `{payload_hash_A, signature_A}` (produced by the MPC network for request A) can:

1. Submit `verify_foreign_transaction(request_B)` for a foreign-chain transaction that was **not** confirmed, or with different extracted values.
2. Call `respond_verify_foreign_tx(request_B, {payload_hash_A, signature_A})`.
3. The contract accepts: `signature_A` is valid over `payload_hash_A` using the root key ✓, and `request_B` has a pending yield ✓.
4. The caller of `request_B` receives `{payload_hash_A, signature_A}` — a valid root-key signature over a hash that does not correspond to their request.

Any downstream NEAR contract that gates bridge operations on the returned `(payload_hash, signature)` pair without independently recomputing the expected hash from the original request data will accept this forged attestation. This enables **invalid bridge execution or double-spend conditions** (e.g., claiming an inbound Omnibridge transfer was finalized when it was not).

**Impact category**: High — Cross-chain replay / forged foreign-chain verification causing invalid bridge execution.

### Likelihood Explanation

- Requires one Byzantine **attested** participant (below threshold). TEE attestation reduces but does not eliminate this risk; a compromised or malicious node operator satisfies this condition.
- The attacker needs only one previously produced legitimate signature from the MPC network — trivially available after any successful `verify_foreign_transaction` call.
- No cryptographic break is required; the attack is purely a contract-logic exploit.
- The `respond_verify_foreign_tx` function is callable by any single attested participant without threshold agreement. [6](#0-5) 

### Recommendation

Recompute the expected `payload_hash` on-chain from the pending request before accepting the response. Since the contract stores the `VerifyForeignTransactionRequest` (which includes the `ForeignChainRpcRequest`), the contract should require the node to also submit the `extracted_values`, recompute `SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))`, and verify that this matches `response.payload_hash` before accepting the signature. Alternatively, mirror the `respond` pattern: derive the expected hash from the stored request rather than trusting the node-supplied hash.

### Proof of Concept

**Setup**: MPC network has processed one legitimate request, producing `(payload_hash_A, sig_A)` signed by the root key.

**Attack**:
```
1. attacker.call(contract, "verify_foreign_transaction", {
       request: { Bitcoin: { tx_id: UNCONFIRMED_TX, ... } },
       domain_id: foreign_tx_domain_id,
       payload_version: V1
   }, deposit=1)
   // → pending yield queued under request_B key

2. attacker.call(contract, "respond_verify_foreign_tx", {
       request: request_B,          // the unconfirmed-tx request
       response: {
           payload_hash: payload_hash_A,   // hash from a DIFFERENT, legitimate tx
           signature: sig_A               // valid root-key signature over payload_hash_A
       }
   })
   // Contract checks: verify_ecdsa(sig_A, payload_hash_A, root_pk) → OK ✓
   // Contract resolves request_B's yield with {payload_hash_A, sig_A}

3. Caller of request_B receives {payload_hash_A, sig_A}
   // Downstream bridge contract sees a valid root-key signature and
   // processes the inbound transfer for the unconfirmed transaction.
```

The contract accepts the call at step 2 because `respond_verify_foreign_tx` only checks signature validity over the node-supplied hash, not hash correctness for the resolved request. [7](#0-6)
