### Title
Caller-Controlled Zero-Confirmation Bitcoin Finality Bypass in `verify_foreign_transaction` - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

### Summary

The `verify_foreign_transaction` contract entry point accepts a caller-supplied `BlockConfirmations` value with no lower-bound enforcement. Setting `confirmations: 0` causes the node-side Bitcoin inspector to unconditionally pass its finality check, allowing the MPC network to issue a threshold signature attesting to an unconfirmed (mempool-only) Bitcoin transaction. A bridge contract that trusts this signature to release NEAR-side funds is exposed to a double-spend: the attacker broadcasts the Bitcoin transaction, obtains the MPC signature, claims NEAR funds, then double-spends the Bitcoin input.

---

### Finding Description

`BlockConfirmations` is defined as a plain `u64` wrapper with no minimum constraint:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema confirms the field accepts `0`:

```json
"BlockConfirmations": {
  "type": "integer",
  "format": "uint64",
  "minimum": 0.0
}
```

The on-chain `verify_foreign_transaction` handler performs no validation on the `confirmations` field before enqueuing the request:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);
    // only checks: domain exists, gas, deposit, chain whitelisted
    // NO check that request.request.confirmations >= 1
    let request = args_into_verify_foreign_tx_request(request);
    self.enqueue_yield_request(...);
}
```

The node-side `BitcoinInspector::extract` enforces finality with:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

When `block_confirmations_threshold = 0`, the condition `0 <= actual_confirmations` is trivially true for any transaction visible to the RPC node, including unconfirmed mempool transactions that report `confirmations: 0`. The inspector proceeds to sign the payload without any finality guarantee.

The signed payload encodes the caller-supplied `confirmations: 0` as part of the request, so the resulting MPC signature attests to "this transaction exists with at least 0 confirmations" — a meaningless security bound identical in spirit to setting `amountAMin = 1` in the original report.

---

### Impact Explanation

**High.** The MPC network issues a valid threshold signature over a `ForeignTxSignPayload` that includes an unconfirmed Bitcoin transaction. Any NEAR bridge contract that consumes this signature to release funds (the primary stated use-case of `verify_foreign_transaction`) will do so before the Bitcoin transaction has any on-chain finality. The attacker can then double-spend the Bitcoin input, leaving the bridge holding a signed attestation for a transaction that no longer exists on the canonical chain. This constitutes a forged foreign-chain verification enabling double-spend conditions, matching the "High" impact tier.

---

### Likelihood Explanation

**Medium.** The attack requires only a 1 yoctoNEAR deposit and a whitelisted Bitcoin chain. Any unprivileged NEAR account can submit the request. The attacker must control a Bitcoin UTXO and have access to a cooperating (or slow-confirming) mempool, which is a realistic condition. The MPC nodes will process the request as long as the transaction is visible to their configured RPC providers.

---

### Recommendation

1. **Enforce a minimum confirmation count on-chain.** In `verify_foreign_transaction`, reject `BitcoinRpcRequest` entries where `confirmations == 0`:

```rust
if let ForeignChainRpcRequest::Bitcoin(ref btc) = request.request {
    if btc.confirmations == BlockConfirmations(0) {
        env::panic_str("Bitcoin confirmations must be >= 1");
    }
}
```

2. **Consider a protocol-level minimum.** Introduce a per-chain minimum confirmation floor in the `ChainEntry` governance structure (voted in by participants), so the on-chain whitelist can enforce chain-specific finality requirements (e.g., 6 for Bitcoin mainnet) rather than relying on callers to supply safe values.

3. **Node-side defense-in-depth.** The `BitcoinInspector` should also enforce a hard floor of `>= 1` regardless of the caller-supplied threshold, treating `0` as an error.

---

### Proof of Concept

1. Attacker broadcasts a Bitcoin transaction `T` spending UTXO `U` to address `A_bridge` (the bridge deposit address).
2. Before `T` is confirmed, attacker calls `verify_foreign_transaction` on the MPC contract with:
   ```json
   {
     "request": {
       "Bitcoin": {
         "tx_id": "<hash of T>",
         "confirmations": 0,
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": 1
   }
   ```
   with `1 yoctoNEAR` deposit.
3. MPC nodes query their Bitcoin RPC. `T` is in the mempool; `getrawtransaction` returns it with `confirmations: 0`. The check `0 <= 0` passes. Nodes sign the payload.
4. Attacker receives the MPC threshold signature attesting to `T`.
5. Attacker submits the signature to the NEAR bridge contract, which releases NEAR-side funds.
6. Attacker broadcasts a conflicting transaction `T'` spending `U` to a different address, double-spending the input. With sufficient miner fee, `T'` confirms instead of `T`.
7. The bridge has released funds for a Bitcoin transaction that never confirmed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
