### Title
Deposited NEAR Permanently Locked in `MpcContract` — No Withdrawal Path for `propose_update` Deposits - (File: `crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract` collects a mandatory NEAR deposit from every caller of `propose_update`. When any update reaches threshold votes and `do_update` executes, it unconditionally clears **all** pending proposals — including those whose proposers paid deposits — without refunding any of those deposits. No withdrawal or cancellation mechanism exists anywhere in the contract. The deposited NEAR is permanently locked.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` is a `#[payable]` method that requires a deposit proportional to the byte size of the proposed update (contract code or config): [1](#0-0) 

The required deposit is computed in `update.rs`: [2](#0-1) [3](#0-2) 

For a contract-code update, `bytes_used` includes the full code length plus a fixed overhead for 128 participant-vote slots: [4](#0-3) 

Only the **excess** above the required deposit is refunded to the proposer; the required portion stays in the contract: [5](#0-4) 

When `vote_update` accumulates enough votes and calls `do_update`, the implementation removes the winning entry and then **clears every other pending entry** with no refund: [6](#0-5) 

Specifically, lines 198–200 (`self.entries.clear(); self.vote_by_participant.clear();`) discard all competing proposals and their associated deposits silently. There is no `cancel_update`, `withdraw_proposal`, or any other mechanism that returns a proposer's deposit. [7](#0-6) 

---

### Impact Explanation

**Medium — balance/accounting invariant broken; permanent loss of deposited NEAR.**

A contract-code update for a 1 MB WASM binary requires roughly `(1_000_000 + 128 × 64) bytes × ~10 yN/byte ≈ 10 NEAR` per proposal. In a realistic scenario where two or more participants independently propose competing upgrades (e.g., a code update vs. a config update, or two different code versions), all non-winning proposers permanently lose their deposits once any single update reaches threshold. The freed storage accrues to the contract's own balance with no path for the original depositors to reclaim it.

This matches the allowed Medium impact: *"Balance … manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

**High.** Competing proposals are a normal operational scenario: participants may propose a config change while another participant simultaneously proposes a code upgrade, or two participants may propose different code versions. The clearing of all entries on `do_update` is unconditional and happens every time any update is executed. No special attacker capability is required — any authorized participant (voter) can trigger the loss simply by proposing an update that loses the vote race. [8](#0-7) 

---

### Recommendation

1. **Track the proposer and deposit amount per `UpdateEntry`** — store `(proposer: AccountId, deposit: NearToken)` alongside each entry.
2. **Refund non-winning proposers inside `do_update`** — before calling `self.entries.clear()`, iterate over remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each.
3. **Add a `cancel_update` method** — allow a proposer to withdraw their own pending proposal (and receive their deposit back) at any time before the update executes.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR. The deposit is accepted; only the excess is refunded.
2. Participant B calls `propose_update` with a config update, attaching a smaller deposit.
3. Threshold participants vote for B's config update via `vote_update`.
4. `vote_update` calls `do_update(&id_B, gas)`.
5. Inside `do_update`: `self.entries.remove(&id_B)` removes B's entry; `self.entries.clear()` silently discards A's entry (and its ~10 NEAR deposit); `self.vote_by_participant.clear()` wipes all votes.
6. A's ~10 NEAR is now permanently locked in the contract. No function exists to recover it. [6](#0-5) [9](#0-8)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
