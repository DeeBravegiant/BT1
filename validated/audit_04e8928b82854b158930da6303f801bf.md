### Title
Excess Attached Deposit Not Refunded in `submit_participant_info` Re-Submission Path — (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is a `#[payable]` function that accepts attached NEAR deposits. When an **existing participant** re-submits their attestation (e.g., for a TEE upgrade), the deposit-handling branch is skipped entirely. Any NEAR tokens attached by the caller in this path are permanently absorbed into the contract with no refund and no recovery mechanism.

---

### Finding Description

`submit_participant_info` computes a boolean flag to decide whether the caller must pay for storage:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... storage cost computed, deposit checked, excess refunded
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
// ← no else branch: if flag is false, attached deposit is silently kept
``` [1](#0-0) 

The flag is `false` when the caller **is** an existing participant **and** the attestation is an update (not a new insertion). In this case the entire `if` block — including the `env::attached_deposit()` read and the refund `Promise` — is never reached. Any NEAR attached to the call is silently retained by the contract. [2](#0-1) 

The contract has no owner-withdrawal or sweep function, so there is no viable way to recover the stuck tokens after the transaction succeeds.

The three scenarios mirror the ASDRouter report exactly:

| Scenario | Deposit consumed | Refund issued |
|---|---|---|
| New participant (first submission) | Storage cost | Excess refunded ✓ |
| Existing participant, re-submission | 0 | **None — stuck** ✗ |
| Call reverts (e.g. invalid attestation) | 0 | Full rollback ✓ |

---

### Impact Explanation

Any NEAR tokens attached during a successful re-submission by an existing participant are permanently frozen inside the MPC contract. There is no withdrawal path. This breaks the implicit accounting invariant that a `#[payable]` function either consumes the deposit for a stated purpose or refunds the excess — an invariant the contract itself upholds in every other payable method (`sign`, `request_app_private_key`, `verify_foreign_transaction`, `propose_update`). [3](#0-2) [4](#0-3) 

This maps to the **Medium** allowed impact: *balance or contract execution-flow manipulation that breaks production safety/accounting invariants*.

---

### Likelihood Explanation

The automated node software (`submit_remote_attestation` in `crates/node/src/tee/remote_attestation.rs`) sends `ChainSendTransactionRequest::SubmitParticipantInfo` without attaching any deposit, so the normal automated re-submission path is unaffected. [5](#0-4) 

However, the operator guide explicitly documents that operators must **manually** call `submit_participant_info` with `--deposit` for first-time submissions, and that the node may fail to submit on its own (requiring the operator to submit on the node's behalf). [6](#0-5) 

An operator who copy-pastes the first-time deposit command for a subsequent re-submission (e.g., during a TEE upgrade cycle) will have their NEAR permanently stuck. The likelihood is low-to-medium but the loss is irreversible.

---

### Recommendation

Add an unconditional refund of any excess attached deposit after the `if attestation_storage_must_be_paid_by_caller` block, mirroring the pattern already used in `propose_update` and `require_deposit`:

```rust
// After the if attestation_storage_must_be_paid_by_caller { ... } block:
if !attestation_storage_must_be_paid_by_caller {
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, restructure the deposit handling so `env::attached_deposit()` is always read and any amount above the computed cost (zero for re-submissions) is always refunded, regardless of which branch is taken.

---

### Proof of Concept

1. Deploy the MPC contract and initialize it with participant `alice.near`.
2. `alice.near` calls `submit_participant_info` (first time) with a valid attestation and sufficient deposit — succeeds, storage cost deducted, excess refunded.
3. `alice.near` calls `submit_participant_info` again (re-submission, e.g. TEE upgrade) and accidentally attaches `1 NEAR`.
4. The call succeeds (`Ok(())`).
5. `attestation_storage_must_be_paid_by_caller` is `false` (alice is a participant, not a new insertion).
6. The `if` block is skipped; no refund promise is scheduled.
7. `alice.near`'s balance is reduced by `1 NEAR`; the contract balance increases by `1 NEAR`.
8. There is no function in the contract to withdraw or recover this `1 NEAR`. [7](#0-6)

    /// (Prospective) Participants can submit their tee participant information through this
    /// endpoint.
    #[payable]
    #[handle_result]
    pub fn submit_participant_info(
        &mut self,
        proposed_participant_attestation: dtos::Attestation,
        tls_public_key: dtos::Ed25519PublicKey,
    ) -> Result<(), Error> {
        let proposed_participant_attestation =
            proposed_participant_attestation.try_into_contract_type()?;

        let account_key = env::signer_account_pk();
        let account_id = Self::assert_caller_is_signer();

        log!(
            "submit_participant_info: signer={}, proposed_participant_attestation={:?}, account_key={:?}",
            account_id,
            proposed_participant_attestation,
            account_key
        );

        // Save the initial storage usage to know how much to charge the proposer for the storage
        // used
        let initial_storage = env::storage_usage();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        // The node always signs submissions with an Ed25519 key
        // (`near_signer_key`), so the signer key here is Ed25519 in practice.
        // Reject non-Ed25519 signer keys rather than silently storing a value
        // we could never match against in `is_caller_an_attested_participant`.
        let account_public_key = dtos::Ed25519PublicKey::try_from(&account_key).map_err(|_| {
            InvalidParameters::InvalidTeeRemoteAttestation {
                reason: "signer account key must be Ed25519".to_string(),
            }
        })?;

        // Add the participant information to the contract state
        let attestation_insertion_result = self
            .tee_state
            .add_participant(
                NodeId {
                    account_id: account_id.clone(),
                    tls_public_key,
                    account_public_key,
                },
                proposed_participant_attestation,
                tee_upgrade_deadline_duration,
            )
            .map_err(|err| {
                let reason = match &err {
                    AttestationSubmissionError::InvalidAttestation(_) => {
                        format!("TeeQuoteStatus is invalid: {err}")
                    }
                    AttestationSubmissionError::TlsKeyOwnedByOtherAccount => err.to_string(),
                };
                InvalidParameters::InvalidTeeRemoteAttestation { reason }
            })?;

        let caller_is_not_participant = self.voter_account().is_err();
        let is_new_attestation = matches!(
            attestation_insertion_result,
            ParticipantInsertion::NewlyInsertedParticipant
        );

        let attestation_storage_must_be_paid_by_caller =
            is_new_attestation || caller_is_not_participant;

        if attestation_storage_must_be_paid_by_caller {
            // `saturating_sub`: if a re-submission shrinks the entry, charge nothing
            // rather than underflow. Intentional asymmetry: we do not refund freed bytes
            // either — the caller already paid for the larger entry, and we'd rather
            // accept that asymmetry than open a refund path for payload-shrinking games.
            let storage_used = env::storage_usage().saturating_sub(initial_storage);
            let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
            let attached = env::attached_deposit();

            if attached < cost {
                return Err(InvalidParameters::InsufficientDeposit {
                    attached: attached.as_yoctonear(),
                    required: cost.as_yoctonear(),
                }
                .into());
            }

            // Refund the difference if the proposer attached more than required
            if let Some(diff) = attached.checked_sub(cost)
                && diff > NearToken::from_yoctonear(0)
            {
                Promise::new(account_id).transfer(diff).detach();
            }
        }

        Ok(())
    }
