### Title
Double `PromiseResume` Receipt Creation via Repeated `promise_yield_resume` for Same `data_id` - (`runtime/runtime/src/ext.rs`)

### Summary

`submit_promise_resume_data` in `runtime/runtime/src/ext.rs` does not guard against a second call for the same `data_id` within a single function-call execution. After the first call sets `PromiseYieldStatus::ResumeInitiated` in the `TrieUpdate` staging layer, the `PromiseYieldReceipt` trie entry is never removed, so the guard condition `has_yield_receipt_in_state || has_yield_status_in_state` remains true on the second call, causing a second `PromiseResume` receipt to be appended to `data_receipts` with no deduplication. An unprivileged user who deploys a contract calling `promise_yield_resume` twice for the same `data_id` can produce duplicate `PromiseResume` receipts in the outgoing set, violating the one-resume-per-yield invariant and producing a divergent receipt set and state root.

---

### Finding Description

The guard in `submit_promise_resume_data` reads two independent trie keys:

```
has_yield_receipt_in_state  →  TrieKey::PromiseYieldReceipt { receiver_id, data_id }
has_yield_status_in_state   →  TrieKey::PromiseYieldStatus  { receiver_id, data_id }
``` [1](#0-0) 

On the **first** call the yield receipt is present in the trie (placed there when the yield was created in a prior block), so `has_yield_receipt_in_state` is `true`. The function then:

1. Calls `create_promise_resume_receipt`, which unconditionally pushes a new `DataReceiptMetadata` onto `data_receipts`.
2. Calls `set_promise_yield_status(..., ResumeInitiated)`, which writes only to `TrieKey::PromiseYieldStatus`. [2](#0-1) 

Critically, **`TrieKey::PromiseYieldReceipt` is never removed** by `set_promise_yield_status`. [3](#0-2) 

On the **second** call for the same `data_id` within the same WASM execution:

- `has_yield_receipt_in_state` is still `true` — the `TrieUpdate` staging layer still contains the yield receipt entry because nothing deleted it.
- `has_yield_status_in_state` is also `true` — `ResumeInitiated` was written by the first call.

The `||` condition passes again, and `create_promise_resume_receipt` pushes a second entry: [4](#0-3) 

There is no deduplication in `create_promise_resume_receipt`; it is a plain `Vec::push`. The `TrieUpdate` staging layer is visible to subsequent reads within the same execution (as the codebase itself notes for the `create_promise_yield_receipt_with_id` duplicate-yield-id guard at line 369–370), confirming that the second call sees the `ResumeInitiated` status but also still sees the unremoved yield receipt. [5](#0-4) 

---

### Impact Explanation

Two `PromiseResume` receipts with the same `data_id` appear in the outgoing receipt set produced by the chunk. This violates the protocol invariant that each yield `data_id` resolves at most once. Downstream effects include:

- The yield callback action receipt may be unblocked and executed twice, enabling double-execution of arbitrary contract logic (balance transfers, storage writes, cross-contract calls).
- The outgoing receipt set hash differs from the expected value, producing a divergent state root / outcome root for the chunk — a consensus-level impact.

---

### Likelihood Explanation

Any unprivileged user can deploy a contract and invoke it via a signed transaction. The contract only needs to call `promise_yield_resume` twice with the same `data_id` in a single function call. No validator, admin, or special privilege is required. The precondition (a live `PromiseYieldReceipt` in the trie for the caller's account) is the normal state after `promise_yield_create` is used, which is the intended use-case for this API.

---

### Recommendation

In `submit_promise_resume_data`, retrieve the current status value (not just its presence) and return `Ok(false)` immediately if it is already `ResumeInitiated`:

```rust
fn submit_promise_resume_data(...) -> Result<bool, VMLogicError> {
    let yield_status = get_promise_yield_status(
        self.trie_update, &self.account_id, data_id
    ).map_err(wrap_storage_error)?;

    // Already resumed — reject duplicate.
    if yield_status == Some(PromiseYieldStatus::ResumeInitiated) {
        return Ok(false);
    }

    let has_yield_receipt_in_state =
        has_promise_yield_receipt(...)?;

    if has_yield_receipt_in_state || yield_status.is_some() {
        self.receipt_manager.create_promise_resume_receipt(data_id, data);
        set_promise_yield_status(..., PromiseYieldStatus::ResumeInitiated);
        return Ok(true);
    }
    Ok(false)
}
```

Alternatively, remove the `PromiseYieldReceipt` trie entry inside `submit_promise_resume_data` after the first successful resume, so that `has_yield_receipt_in_state` returns `false` on any subsequent call.
