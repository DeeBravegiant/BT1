### Title
Missing `caller_deposit` Refund in `nep_141_storage_balance_callback` Error Paths Causes Permanent NEAR Token Loss - (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary

The `WalletContract::nep_141_storage_balance_callback` (and `address_check_callback`) in the ETH-implicit wallet contract returns early in multiple error paths without refunding the `caller_deposit` attached by an external relayer. This is the direct analog of the reported missing `receive()` function: a handler that is supposed to return assets to the caller is simply absent in the error branches, causing the caller's NEAR tokens to be permanently locked inside the wallet contract.

### Finding Description

`rlp_execute` is `#[payable]` and accepts an attached NEAR deposit from any external caller. When an ERC-20 transfer is initiated, `inner_rlp_execute` captures the full attached deposit in a `CallerDeposit` struct and passes it through the promise chain: [1](#0-0) [2](#0-1) 

The `rlp_execute_callback` correctly refunds `caller_deposit` when the downstream cross-contract call fails: [3](#0-2) 

However, `nep_141_storage_balance_callback` has **three** early-return error paths that return `PromiseOrValue::Value(...)` without issuing any refund:

**Path 1 — `storage_balance_of` call fails:** [4](#0-3) 

**Path 2 — response cannot be deserialized:** [5](#0-4) 

**Path 3 — action is not a `FunctionCall` (defensive guard):** [6](#0-5) 

The same omission exists in `address_check_callback` for the registrar-call-failed and deserialization-failed paths: [7](#0-6) [8](#0-7) 

In all these paths, `caller_deposit` is received as a parameter but is silently dropped. The NEAR tokens it represents remain in the wallet contract's balance with no mechanism to recover them.

### Impact Explanation

Any external relayer who attaches a non-zero NEAR deposit to `rlp_execute` for an ERC-20 transfer permanently loses that deposit if `storage_balance_of` on the token contract fails or returns malformed data. The corrupted protocol value is the **account balance** of the relayer: it is decremented by the attached deposit amount and never restored. The wallet contract's balance is correspondingly inflated by the same amount, permanently.

### Likelihood Explanation

The trigger is reachable by an unprivileged external user:

1. Any user can deploy a NEP-141-like contract whose `storage_balance_of` always panics or returns non-JSON bytes.
2. A relayer submits an ERC-20 transfer via `rlp_execute` targeting that contract, attaching NEAR tokens as a fee/deposit.
3. `storage_balance_of` fails → `nep_141_storage_balance_callback` is invoked with `PromiseResult::Failed`.
4. The callback returns without refunding `caller_deposit`.
5. The relayer's NEAR is permanently locked.

The `CallerDeposit` type itself documents the intent: *"This allows us to refund the caller's deposit if the cross-contract call fails."* [9](#0-8) 

The refund is correctly implemented in `rlp_execute_callback` but is absent in the two intermediate callbacks, making this an inconsistency that is easy to miss and easy to trigger.

### Recommendation

In every early-return error branch of `nep_141_storage_balance_callback` and `address_check_callback`, add the same refund logic that `rlp_execute_callback` already uses:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This should be applied to all five early-return paths identified above before the `return PromiseOrValue::Value(...)` statement.

### Proof of Concept

Call flow that triggers the loss:

```
relayer → rlp_execute(target=malicious_nep141, tx_bytes=ERC20_TRANSFER, deposit=1 NEAR)
  └─ inner_rlp_execute
       ├─ caller_deposit = Some(CallerDeposit { account_id: relayer, yocto_near: 1e24 })
       └─ Promise::new(malicious_nep141)
            .function_call("storage_balance_of", ...)   // panics → PromiseResult::Failed
            .then(nep_141_storage_balance_callback(..., caller_deposit=Some(...)))
                 └─ match PromiseResult::Failed =>
                      return PromiseOrValue::Value(error_response)
                      // caller_deposit is DROPPED — 1 NEAR permanently locked
```

The relayer's 1 NEAR is now inside the wallet contract with no recovery path, while `rlp_execute_callback` (the only place that issues the refund) is never reached.
