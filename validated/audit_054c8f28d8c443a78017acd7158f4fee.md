### Title
Unrestricted `EvmFinality::Latest` Accepted by MPC Network Enables Signature Over Reorgable State — (`crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

An unprivileged caller can submit a `VerifyForeignTransactionRequestArgs` with `finality: EvmFinality::Latest`. No guard at any layer rejects this variant. The MPC network will then sign a bridge payload derived from a transaction in a non-finalized ("latest") EVM block. If that block is later reorganized away, the attacker retains a valid threshold signature over a transaction that no longer exists on the canonical chain, enabling a double-spend on the destination bridge.

---

### Finding Description

**Step 1 — Caller-controlled input accepted without restriction.**

`EvmFinality` is a public, Borsh-serializable enum with `Latest` as its first variant (discriminant 0): [1](#0-0) 

There is no contract-level validation that rejects `Latest`. Any unprivileged NEAR account can submit `VerifyForeignTransactionRequestArgs { finality: EvmFinality::Latest, ... }` for any supported EVM chain (Abstract, BNB, Base, Arbitrum, HyperEVM, Polygon).

**Step 2 — Conversion layer passes `Latest` through without error.**

`TryFrom<dtos::EvmFinality>` for `EthereumFinality` explicitly maps `Latest` to `EthereumFinality::Latest` and returns `Ok`: [2](#0-1) 

**Step 3 — `verify_finality_level` uses `FinalityTag::Latest`.**

When `EthereumFinality::Latest` is passed, `verify_finality_level` queries `eth_getBlockByNumber("latest")` and accepts any receipt whose `block_number <= latest_head.number`. The "latest" tag returns the most recently seen block, which has **not** reached `safe` or `finalized` status: [3](#0-2) 

**Step 4 — `verify_block_is_canonical` is a point-in-time check only.**

The canonical check compares the receipt's block hash against the canonical block at that height *at the moment of signing*. It passes if the block is canonical right now, but provides no guarantee after a reorg: [4](#0-3) 

**Step 5 — The signed payload includes the non-finalized block hash.**

`ForeignTxSignPayloadV1` Borsh-serializes the full `request` (including `finality: EvmFinality::Latest`) and the extracted `values` (including the block hash from the non-finalized block). The MPC network signs the SHA-256 of this: [5](#0-4) 

**Step 6 — The destination-side verifier does not re-check source chain state.**

`ForeignChainSignatureVerifier::verify_signature` only verifies the MPC signature against the payload hash. It does not re-query the source chain. Once the signature is issued, it is permanently valid for the payload it covers, regardless of subsequent reorgs: [6](#0-5) 

---

### Impact Explanation

The MPC network issues a threshold signature over a `ForeignTxSignPayloadV1` that commits to a transaction in a non-finalized block. If that block is reorganized away:

- The source-chain transaction ceases to exist on the canonical chain.
- The attacker retains the MPC signature, which remains cryptographically valid.
- The destination bridge contract, which trusts the MPC signature as the attestation of source-chain state, will accept the signature and release funds.
- The attacker has received funds on the destination chain for a source-chain transaction that was never finalized — a double-spend.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

- The attack requires zero privilege: any NEAR account can call `verify_foreign_transaction`.
- Reorgs of 1–2 blocks occur regularly on Polygon, BNB, and Base (all supported chains). An attacker can monitor the mempool, submit a deposit transaction, immediately request MPC signing with `Latest` finality, and then attempt to induce or wait for a natural reorg.
- No threshold collusion, TEE compromise, or operator access is required.

---

### Recommendation

Reject `EvmFinality::Latest` at the node's `execute_foreign_chain_request` dispatch layer in `crates/node/src/providers/verify_foreign_tx/sign.rs`, before the inspector is called. For each EVM chain arm, after converting `request.finality.clone().try_into()?`, add an explicit guard:

```rust
if finality == EthereumFinality::Latest {
    bail!("EvmFinality::Latest is not permitted for bridge signing");
}
```

Alternatively, remove `Latest` from the `EvmFinality` enum entirely if it has no production use case, or restrict it to a per-chain allowlist enforced in the contract's domain configuration. [7](#0-6) 

---

### Proof of Concept

```
1. Deploy a supported EVM chain (e.g., Polygon) with a mock RPC.
2. Configure the mock to return:
   - eth_getTransactionReceipt → receipt at block 100, hash 0xAAA...
   - eth_getBlockByNumber("latest") → block 100 (same height as receipt)
   - eth_getBlockByNumber(100) → block 100, hash 0xAAA... (canonical match)
3. Submit VerifyForeignTransactionRequestArgs {
       request: ForeignChainRpcRequest::Polygon(EvmRpcRequest {
           tx_id: <tx>,
           finality: EvmFinality::Latest,   // discriminant 0
           extractors: vec![EvmExtractor::BlockHash],
       }),
       ...
   }
4. Assert: verify_finality_level returns Ok (latest head >= receipt block).
5. Assert: verify_block_is_canonical returns Ok (hashes match).
6. Assert: MPC network issues a valid threshold signature S over payload P.
7. Simulate reorg: update mock so eth_getBlockByNumber(100) → hash 0xBBB...
8. Assert: a second extract call on the same tx_id now returns NonCanonicalBlock.
9. Demonstrate: signature S from step 6 is still cryptographically valid over P,
   and the destination bridge contract accepts S to release funds — double-spend.
```

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())

        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
