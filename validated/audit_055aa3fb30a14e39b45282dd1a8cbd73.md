### Title
Cross-Request Replay in `respond_verify_foreign_tx` Allows a Single Below-Threshold Malicious Participant to Corrupt Pending Foreign-Chain Verification Requests — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid threshold signature over `response.payload_hash` using the root public key, but **never checks that `response.payload_hash` is the hash derived from the original `request`**. Because all prior responses are publicly visible on-chain, a single malicious attested participant (strictly below the signing threshold) can replay a signature produced for a completely different foreign-chain request to permanently resolve any pending `verify_foreign_transaction` yield with incorrect data, breaking the request-lifecycle safety invariant and potentially locking bridge users' funds.

---

### Finding Description

In `respond_verify_foreign_tx`, the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // taken from the response, not recomputed
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,                                        // attacker-supplied hash
    &secp_pk,                                             // root public key
)
.is_ok()
``` [1](#0-0) 

The contract only asserts that `signature` is a valid ECDSA signature over `payload_hash` under the root key. It does **not** assert that `payload_hash == SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))` for any `extracted_values` consistent with the original `request`.

By contrast, the node-side code correctly computes the hash from the payload before submitting:

```rust
let payload_hash = response.0.0.compute_msg_hash()?;
``` [2](#0-1) 

And the SDK-level verifier used by bridge contracts independently recomputes the expected hash and checks it against `response.payload_hash`:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [3](#0-2) 

The on-chain contract performs no equivalent check. Every `respond_verify_foreign_tx` call and its `{payload_hash, signature}` pair is permanently visible on the NEAR blockchain. A malicious participant can therefore:

1. Observe any previously accepted response `{H_Y, sig_Y}` for a resolved request Y.
2. Wait for a new pending request X (different `tx_id`, different key in `pending_verify_foreign_tx_requests`).
3. Call `respond_verify_foreign_tx(request_X, {payload_hash: H_Y, signature: sig_Y})`.

The contract accepts the call because `sig_Y` is a valid root-key signature over `H_Y`. It then resolves **all** queued yields for `request_X` with the wrong response and removes the entry from the map. [4](#0-3) 

The `VerifyForeignTransactionResponse` type confirms that `payload_hash` is a free field in the response DTO, not recomputed by the contract: [5](#0-4) 

---

### Impact Explanation

Every user who submitted `verify_foreign_transaction(request_X)` receives a response whose `payload_hash` belongs to a completely different foreign-chain transaction. The SDK-level `ForeignChainSignatureVerifier::verify_signature` rejects this response with `IncorrectPayloadSigned`. Because `resolve_yields_for` removes the pending entry from the map, the user **cannot retry**; the yield is permanently consumed with incorrect data. [6](#0-5) 

If the bridge contract had already locked or committed user funds contingent on a successful verification, those funds may be permanently frozen. This satisfies the **Medium** allowed impact: *request-lifecycle manipulation that breaks production safety/accounting invariants*.

---

### Likelihood Explanation

The attack requires exactly **one** malicious attested TEE participant — strictly below the signing threshold. No threshold collusion is needed. The attacker does not need to be the signing leader for any request; they only need to read a previously submitted `respond_verify_foreign_tx` transaction from the public NEAR blockchain to obtain a valid `{H_Y, sig_Y}` pair. The access guard (`assert_caller_is_attested_participant_and_protocol_active`) is satisfied by any single participant, which is within the MPC system's stated Byzantine adversary model. [7](#0-6) 

---

### Recommendation

The contract must independently recompute the expected `payload_hash` from the original `request` and the `extracted_values` included in the response, then assert equality before accepting the response. Concretely:

- Include `extracted_values: Vec<ExtractedValue>` in `VerifyForeignTransactionResponse` (or a separate argument to `respond_verify_foreign_tx`).
- In `respond_verify_foreign_tx`, compute `expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, extracted_values }))` and assert `expected_hash == response.payload_hash` before the signature check.

This mirrors the validation already performed by the SDK-level `ForeignChainSignatureVerifier::verify_signature` and closes the gap between what the contract accepts and what bridge contracts can actually use. [8](#0-7) 

---

### Proof of Concept

```
1. User A submits verify_foreign_transaction(request_Y) for Bitcoin tx_id_Y.
2. MPC nodes sign H_Y = SHA256(borsh(ForeignTxSignPayload{request_Y, [BlockHash_Y]})).
3. Honest leader calls respond_verify_foreign_tx(request_Y, {payload_hash: H_Y, sig: sig_Y}).
   → Contract resolves request_Y. {H_Y, sig_Y} is now permanently on-chain.

4. User B submits verify_foreign_transaction(request_Z) for Bitcoin tx_id_Z (different tx).
   → Pending yield created for request_Z.

5. Malicious attested participant reads {H_Y, sig_Y} from the NEAR blockchain.
6. Malicious participant calls:
     respond_verify_foreign_tx(request_Z, {payload_hash: H_Y, signature: sig_Y})

7. Contract checks:
   a. assert_caller_is_attested_participant_and_protocol_active() → passes (single participant)
   b. verify_ecdsa_signature(sig_Y, H_Y, root_pk) → passes (sig_Y is a valid root-key sig)
   c. No check that H_Y corresponds to request_Z.
   → Contract resolves request_Z with {payload_hash: H_Y, sig: sig_Y}.

8. User B's bridge contract calls ForeignChainSignatureVerifier::verify_signature:
   expected_hash = SHA256(borsh(ForeignTxSignPayload{request_Z, [BlockHash_Z]})) ≠ H_Y
   → Returns IncorrectPayloadSigned error.

9. User B's bridge operation fails. The pending yield for request_Z is permanently consumed.
   User B cannot retry. Funds committed to the bridge operation are locked.
```
