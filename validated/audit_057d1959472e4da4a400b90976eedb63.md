### Title
Farmer Signs Attacker-Controlled Foliage Hash Without Reward Address Verification When `foliage_block_data` Is `None` — (`chia/farmer/farmer_api.py`)

### Summary

A malicious full node can send a `RequestSignedValues` message with `foliage_block_data_hash = hash(FoliageBlockData{farmer_reward_puzzle_hash: attacker_address, ...})` and `foliage_block_data = None`. Because the farmer only verifies the reward address when `foliage_block_data` is present, it blindly signs the attacker-supplied hash. The resulting `SignedValues` contains a valid plot-key signature over a foliage block that redirects the farmer reward to the attacker. The block passes all consensus validation checks and is accepted by the network.

---

### Finding Description

**Entry point — `RequestSignedValues` protocol message**

`RequestSignedValues` has two separate fields for the foliage block data:

```python
# chia/protocols/farmer_protocol.py:80-86
class RequestSignedValues(Streamable):
    quality_string: bytes32
    foliage_block_data_hash: bytes32          # hash to sign
    foliage_transaction_block_hash: bytes32   # hash to sign
    foliage_block_data: FoliageBlockData | None = None          # optional source
    foliage_transaction_block_data: FoliageTransactionBlock | None = None
    rc_block_unfinished: RewardChainBlockUnfinished | None = None
``` [1](#0-0) 

**Farmer's `request_signed_values` — no hash-vs-source verification**

When `foliage_block_data is None`, `message_data` stays `None` and the attacker-supplied `foliage_block_data_hash` is forwarded directly to the harvester for signing:

```python

    ...

**Harvester signs whatever hash it receives**

There is no check that `foliage_block_data_hash` corresponds to a `FoliageBlockData` with `farmer_reward_puzzle_hash == self.farmer.farmer_target`. [4](#0-3) 

**Consensus validation accepts the block**

Block header validation checks that the signature is valid over `foliage_block_data.get_hash()`:

```python
# chia/consensus/block_header_validation.py:731-737
if not AugSchemeMPL.verify(
    header_block.reward_chain_block.proof_of_space.plot_public_key,
    header_block.foliage.foliage_block_data.get_hash(),
    header_block.foliage.foliage_block_data_signature,
):
    return None, ValidationError(Err.INVALID_PLOT_SIGNATURE)
``` [5](#0-4) 

Since the farmer signed `hash(FoliageBlockData{farmer_reward_puzzle_hash: attacker_address})` and the block contains exactly that `FoliageBlockData`, the signature is valid and the block is accepted.

**The full node's own `signed_values` handler does not protect against this**

The honest full node's `signed_values` verifies the returned signature against its own stored candidate block:

```python
# chia/full_node/full_node_api.py:1242-1248
if not AugSchemeMPL.verify(
    candidate.reward_chain_block.proof_of_space.plot_public_key,
    candidate.foliage.foliage_block_data.get_hash(),   # its own candidate
    farmer_request.foliage_block_data_signature,
):
    return None
``` [6](#0-5) 

But the **attacker is the full node**. It stores a candidate block with `farmer_reward_puzzle_hash = attacker_address`, so its own verification passes. It then broadcasts the block directly to the network.

---

### Impact Explanation

A malicious full node operator causes the farmer to sign a block that pays the farmer reward (currently 0.25 XCH per block) to the attacker's address instead of the farmer's address. The block is indistinguishable from a legitimate block to all other network participants. This is an unauthorized XCH reward diversion — Critical impact.

---

### Likelihood Explanation

- The attacker only needs to operate a full node that the victim farmer connects to. Farmers routinely connect to external full nodes.
- The attack requires no leaked keys, no broken cryptography, and no special privileges.
- The attacker must wait for the farmer to win a block (expected frequency depends on plot count), then send one crafted protocol message.
- The `foliage_block_data = None` path is the **default** for non-CHIP-22 full nodes (see `request_signed_values_json` in test data where both fields are `None`). [7](#0-6) 

---

### Recommendation

In `request_signed_values`, when `foliage_block_data` is provided, the farmer already passes it as source data to the harvester. The missing guard is: **when `foliage_block_data` is `None`, the farmer must still verify that `foliage_block_data_hash` matches a `FoliageBlockData` with `farmer_reward_puzzle_hash == self.farmer.farmer_target`**.

The simplest fix is to require the full node to always supply `foliage_block_data` (make it non-optional), and reject `RequestSignedValues` messages where `foliage_block_data is None` or where `foliage_block_data.get_hash() != foliage_block_data_hash` or where `foliage_block_data.farmer_reward_puzzle_hash != self.farmer.farmer_target`.

---

### Proof of Concept

1. Attacker operates a full node and connects to the victim farmer.
2. Attacker sends a valid `NewSignagePoint` to the farmer. The farmer relays it to harvesters.
3. A harvester finds a valid proof; the farmer sends `DeclareProofOfSpace` to all connected full nodes (including the attacker), populating `quality_str_to_identifiers`.
4. Attacker constructs `evil_foliage = FoliageBlockData(farmer_reward_puzzle_hash=attacker_address, unfinished_reward_block_hash=<correct_rc_hash>, ...)`.
5. Attacker sends `RequestSignedValues(quality_string=<valid_qs>, foliage_block_data_hash=evil_foliage.get_hash(), foliage_block_data=None, ...)`.
6. Farmer's `request_signed_values` forwards `evil_foliage.get_hash()` to the harvester with `message_data=None`.
7. Harvester signs it; farmer's `_process_respond_signatures` also signs it; `SignedValues` is returned.
8. Attacker assembles a full block with `evil_foliage` and the farmer's signature, broadcasts it.
9. All nodes accept the block (signature is valid over `evil_foliage.get_hash()`); attacker receives the farmer reward.
