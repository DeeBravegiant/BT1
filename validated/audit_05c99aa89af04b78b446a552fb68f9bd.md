### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards via Unvalidated `farmer_reward_address_override` in `RespondSignatures` - (File: chia/farmer/farmer_api.py)

### Summary
The farmer's `_process_respond_signatures()` method unconditionally accepts `farmer_reward_address_override` from a harvester's `RespondSignatures` message and uses it as the destination for the farmer's block reward in `DeclareProofOfSpace`, with no enforcement of the CHIP-22 fee-quality convention. A malicious harvester can set this field to any arbitrary puzzle hash, permanently redirecting the farmer's XCH block reward to an attacker-controlled address.

### Finding Description

The `NewProofOfSpace` protocol message includes a `farmer_reward_address_override` field (CHIP-22), intended to allow third-party harvesters to take a fee by redirecting the farmer reward to their own address — but only when a cryptographically-derived fee quality threshold is satisfied.

When the farmer receives `NewProofOfSpace` with this field set, it calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

That function checks the fee quality against the harvester-supplied threshold, but **only logs warnings** when the check fails — it never rejects the override or halts processing: [2](#0-1) 

After this non-blocking call, the farmer proceeds to send `RequestSignatures` to the harvester. When the harvester responds with `RespondSignatures`, the farmer's `_process_respond_signatures()` applies the override with **zero validation**: [3](#0-2) 

This overridden address is then embedded directly into `DeclareProofOfSpace` as `farmer_reward_address` and broadcast to the full node: [4](#0-3) 

Critically, the harvester can also set `farmer_reward_address_override` **only in `RespondSignatures`** (not in `NewProofOfSpace`), bypassing even the warning-only fee check entirely. The farmer stores no record of what override (if any) was declared in `NewProofOfSpace`, so there is no cross-message consistency check.

The `RespondSignatures` message structure confirms this field is freely settable by the harvester: [5](#0-4) 

### Impact Explanation

A malicious harvester can redirect every XCH block reward earned by the farmer to an attacker-controlled puzzle hash. The farmer's configured `farmer_target` is silently replaced. This constitutes unauthorized payout redirection of XCH — a High-severity impact under the allowed scope ("payout redirection... singleton-controlled assets").

### Likelihood Explanation

The harvester connects to the farmer using a certificate signed by the farmer's private CA. In the CHIP-22 third-party harvester model, farmers are expected to connect to external harvester services they do not fully control. Any such harvester can exploit this without any additional privilege beyond holding a valid connection certificate. The attack is silent (no on-chain evidence, no user-visible error) and persistent across every block the farmer wins.

### Recommendation

1. In `_process_respond_signatures()`, reject (return `None`) any `RespondSignatures` where `farmer_reward_address_override` is set but the corresponding `NewProofOfSpace` either did not declare an override or failed the fee-quality threshold check.
2. Store the validated `farmer_reward_address_override` (or `None`) from `NewProofOfSpace` in `proofs_of_space` alongside the proof, and compare it against the value in `RespondSignatures` before accepting it.
3. In `notify_farmer_reward_taken_by_harvester_as_fee()`, treat a failed fee-quality check as a hard rejection: do not proceed with block-making when the harvester's claimed threshold is invalid.

### Proof of Concept

1. Malicious harvester connects to farmer (holds a valid private-CA-signed certificate).
2. Harvester finds a valid proof of space for a signage point.
3. Harvester sends `NewProofOfSpace` with `farmer_reward_address_override=None` and `fee_info=None` — no warning is triggered.
4. Farmer sends `RequestSignatures` back to the harvester.
5. Harvester responds with `RespondSignatures` where `farmer_reward_address_override = attacker_puzzle_hash`.
6. Farmer executes lines 917–918 of `farmer_api.py`, setting `farmer_reward_address = attacker_puzzle_hash`.
7. Farmer broadcasts `DeclareProofOfSpace` with `farmer_reward_address = attacker_puzzle_hash` to the full node.
8. Full node creates a block paying the farmer reward to `attacker_puzzle_hash`. The farmer receives nothing. [6](#0-5) [7](#0-6)
