### Title
Gateway `skip_stateful_validations` ignores `max_nonce_for_validation_skip` config, unconditionally skipping `__validate__` for nonce-1 Invoke transactions — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is a free function that hardcodes the skip condition to `nonce == 1 && account_nonce == 0`. It never receives or consults `StatefulTransactionValidatorConfig::max_nonce_for_validation_skip`. The config field is defined, serialized, and documented, but is dead code in the gateway path. An attacker who has any entry in the mempool for their address (via `account_tx_in_pool_or_recent_block`) can submit an Invoke with nonce=1 and a garbage signature and have it admitted to the mempool without `__validate__` being called.

### Finding Description

`skip_stateful_validations` is defined as a free function:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
``` [1](#0-0) 

It has no access to `self.config` and hardcodes the skip condition:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    ...
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...
``` [2](#0-1) 

The config field `max_nonce_for_validation_skip` is defined and serialized in `StatefulTransactionValidatorConfig`: [3](#0-2) 

But it is **never passed to or read by** `skip_stateful_validations`. The field is only actually used in the legacy Python/native_blockifier path (`PyValidator::should_run_stateful_validations`), which has the complete check:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
let skip_validate = deploy_account_not_processed
    && is_post_deploy_nonce
    && nonce_small_enough_to_qualify_for_validation_skip;
``` [4](#0-3) 

The gateway path never applies this bound.

### Impact Explanation

An attacker can:
1. Submit a `DeployAccount` for a custom account contract whose `__validate__` accepts any signature (or any already-declared permissive class), getting it into the mempool.
2. Immediately submit an `Invoke` with `nonce=1` and a garbage/empty signature from the same address.
3. `skip_stateful_validations` sees `nonce == 1`, `account_nonce == 0`, and `account_tx_in_pool_or_recent_block == true`, returns `true`.
4. `run_validate_entry_point` is called with `validate: false`, so `__validate__` is never executed. [5](#0-4) 

The Invoke with an invalid signature is admitted to the mempool. Setting `max_nonce_for_validation_skip = Nonce(Felt::ZERO)` in config has zero effect because the gateway never reads it.

### Likelihood Explanation

The precondition (`account_tx_in_pool_or_recent_block == true`) is easily satisfied by any attacker who can submit a `DeployAccount` — a standard, unprivileged RPC operation. The attack requires no special privileges and is repeatable.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` and apply the same three-part guard used in `PyValidator::should_run_stateful_validations`:
- `deploy_account_not_processed` (account_tx_in_pool_or_recent_block)
- `is_post_deploy_nonce` (nonce >= 1)
- `nonce <= max_nonce_for_validation_skip`

This mirrors the intended logic already implemented in the legacy path.

### Proof of Concept

Configure `max_nonce_for_validation_skip = Nonce(Felt::ZERO)`, mock `account_tx_in_pool_or_recent_block` to return `true`, call `skip_stateful_validations` with an Invoke at nonce=1 and account_nonce=0, and observe it returns `true` — confirming the config field has no effect. The same test with the PyValidator path would return `false` (skip disabled), demonstrating the divergence between the two paths.
