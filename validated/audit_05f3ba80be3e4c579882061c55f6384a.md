### Title
Unvalidated `builder` address in `ProposalInit` allows a proposer to redirect sequencer fees to an arbitrary address — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates most `ProposalInit` fields but never checks `init.builder`. That field is passed verbatim as `sequencer_address` into every block's `BlockInfo`, making it the fee-collection address for all transactions in the block. A malicious proposer can set `builder` to any address, silently redirecting all sequencer fees with no detection or penalty mechanism — a risk-free economic attack that is the direct Sequencer analog of the external report's "no penalties / risk-free economic censorship" class.

---

### Finding Description

`ProposalInit.builder` is documented as "Address of the one who builds/sequences the block."

**Proposer side** — `initiate_build` in `build_proposal.rs` sets it from a static config value:

```rust
builder: args.builder_address,   // line 174
``` [1](#0-0) 

**Validator side** — `is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, all four L1 gas prices, and `fee_proposal_fri`. It never touches `builder`:

```rust
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
{ ... }
// builder is never compared
``` [2](#0-1) 

`ProposalInitValidation` itself carries no `builder` field, so there is no reference value to compare against: [3](#0-2) 

**Execution path** — the unchecked `builder` value flows directly into block execution and state commitment:

```rust
// convert_to_sn_api_block_info
sequencer_address: init.builder,   // line 332
``` [4](#0-3) 

```rust
// update_state_sync_with_new_block
let sequencer = SequencerContractAddress(init.builder);   // line 397
``` [5](#0-4) 

The `sequencer_address` in `BlockInfo` is the Starknet fee-collection address: all transaction fees in the block are credited to it. It is also returned by the `get_sequencer_address()` syscall and committed to the block header via `BlockHeaderWithoutHash.sequencer`.

---

### Impact Explanation

A committee member whose turn it is to propose can set `builder` to any address — their own secondary wallet, a burn address, or a competitor's address — and every validator will accept the proposal without objection. All sequencer fees for that block are then credited to the attacker-chosen address rather than the legitimate sequencer. The wrong `sequencer_address` is also committed to the block header stored in state sync, making the corruption permanent and observable by RPC clients querying block metadata. This matches the **Critical** impact class: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

---

### Likelihood Explanation

Any committee member selected as proposer can execute this attack. Committee membership is determined by stake weight, not by admin privilege. The attack requires no special tooling beyond modifying the `builder_address` field before calling `initiate_build`. There is no on-chain or off-chain penalty mechanism for fee redirection (the external report's acknowledged gap), so the attack is entirely risk-free for the proposer.

---

### Recommendation

1. Add a `builder: ContractAddress` field to `ProposalInitValidation`, populated from the committee's registered builder address for the current proposer.
2. In `is_proposal_init_valid`, assert `init_proposed.builder == proposal_init_validation.builder` and return `InvalidProposalInit` on mismatch.
3. Derive the expected builder address from the staking/committee layer (`StakingManager`) so it is consensus-enforced rather than locally configured.

---

### Proof of Concept

```
1. Committee selects validator V as proposer for height H.
2. V modifies its node: in `initiate_build`, override `builder_address`
   with an attacker-controlled address A.
3. V broadcasts ProposalInit { height: H, builder: A, ... }.
4. Each validator calls is_proposal_init_valid:
     - timestamp ✓, starknet_version ✓, l1_da_mode ✓,
       l2_gas_price_fri ✓, L1 prices ✓, fee_proposal ✓
     - builder: NOT CHECKED → passes
5. Validators call initiate_validation → batcher executes the block
   with sequencer_address = A.
6. Consensus reaches decision; all transaction fees in block H are
   credited to A, not to the legitimate sequencer.
7. BlockHeaderWithoutHash.sequencer = A is committed to state sync
   and returned by RPC block queries.
```
