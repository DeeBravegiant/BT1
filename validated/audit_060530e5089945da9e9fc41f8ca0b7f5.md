### Title
Solana `FinalizeTransferPayload` Missing `message` Field Causes Permanent Fund Lock for Cross-Chain Transfers with Non-Empty Messages — (File: `solana/programs/bridge_token_factory/src/state/message/finalize_transfer.rs`)

---

### Summary

The Solana `FinalizeTransferPayload.serialize_for_near()` omits the `message` field that NEAR's `TransferMessagePayload.encode_hashable()` includes when the message is non-empty. This interface inconsistency causes a permanent signature verification mismatch: NEAR signs a hash that includes the message bytes, but Solana verifies against a hash that never includes them. Any transfer to Solana that carries a non-empty `message` will fail signature verification on Solana and can never be finalized, permanently locking the user's funds on the source chain.

---

### Finding Description

**NEAR side — signing (`near/omni-types/src/lib.rs`):**

`TransferMessagePayload` has a `message: Vec<u8>` field. Its `encode_hashable()` method branches on whether the message is empty:

```rust
pub fn encode_hashable(&self) -> Result<Vec<u8>, String> {
    if self.message.is_empty() {
        borsh::to_vec(&TransferMessagePayloadV1::from(self.clone())).map_err(stringify)
    } else {
        borsh::to_vec(self).map_err(stringify)   // includes message bytes
    }
}
``` [1](#0-0) 

`TransferMessagePayload` (V2) includes `message: Vec<u8>` as its last field, while `TransferMessagePayloadV1` does not: [2](#0-1) 

NEAR's `sign_transfer` constructs the payload with the `message` field populated from the stored transfer message, then calls `encode_hashable()` to produce the bytes that MPC signs: [3](#0-2) 

**Solana side — verification (`solana/programs/bridge_token_factory/src/state/message/finalize_transfer.rs`):**

`FinalizeTransferPayload` has no `message` field at all:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
}
``` [4](#0-3) 

Its `serialize_for_near()` serializes: prefix, destination\_nonce, transfer\_id, token, amount, recipient, fee\_recipient — and **never appends any message bytes**: [5](#0-4) 

`SignedPayload::verify_signature()` hashes the output of `serialize_for_near()` and checks it against the MPC signature: [6](#0-5) 

**The divergence:**

| Condition | NEAR hash input | Solana hash input | Match? |
|---|---|---|---|
| `message.is_empty()` | V1 fields (no message) | same fields (no message) | ✓ |
| `message` non-empty | V2 fields **+ message bytes** | same fields, no message | ✗ |

When `message` is non-empty, NEAR signs `keccak256(borsh(V2_payload_with_message))` but Solana verifies against `keccak256(borsh(payload_without_message))`. The hashes differ; `verify_signature` returns `ErrorCode::SignatureVerificationFailed`; the instruction reverts; the nonce is never marked used; the transfer can never be finalized.

---

### Impact Explanation

**Irreversible fund lock.** When a user initiates a transfer from EVM (or any source chain) to Solana with a non-empty `message`, the source-chain tokens are burned or locked at `initTransfer` time. The NEAR-signed `FinalizeTransferPayload` for Solana will always fail signature verification. There is no fallback path: the nonce is never consumed, no refund mechanism exists on the source chain for a transfer that was successfully initiated, and the funds are permanently unclaimable.

---

### Likelihood Explanation

The EVM `initTransfer` function accepts a public `message` parameter. Any user who passes a non-empty string triggers this path. The `message` field is a documented, first-class feature of the bridge (used for cross-chain messaging/hooks). A user or integrating protocol that uses this feature for a Solana-destined transfer will lose funds with certainty.

---

### Recommendation

Add a `message: Vec<u8>` field to `FinalizeTransferPayload` and append it in `serialize_for_near()` to mirror NEAR's `encode_hashable()` logic:

```rust
pub struct FinalizeTransferPayload {
    pub destination_nonce: u64,
    pub transfer_id: TransferId,
    pub amount: u128,
    pub fee_recipient: Option<String>,
    pub message: Vec<u8>,   // add this
}

// In serialize_for_near:
// 7. message (only if non-empty, matching NEAR's encode_hashable V2 branch)
if !self.message.is_empty() {
    self.message.serialize(&mut writer)?;
}
```

Ensure the NEAR `sign_transfer` path for Solana-destined transfers and the Solana `serialize_for_near` produce byte-identical output for all message values, including empty.

---

### Proof of Concept

1. User calls `OmniBridge.initTransfer(tokenAddress, amount, fee, nativeFee, "solana:recipient", "hello")` on EVM. Tokens are burned/locked; `InitTransfer` event emitted with `message = "hello"`.
2. NEAR relayer picks up the event, stores the transfer with `msg = "hello"`.
3. NEAR relayer calls `sign_transfer`. NEAR constructs `TransferMessagePayload { ..., message: b"hello" }`, calls `encode_hashable()` → serializes V2 (includes `b"hello"`), hashes it, sends to MPC. MPC returns signature `S` over `keccak256(V2_bytes)`.
4. Relayer submits `finalize_transfer` on Solana with `FinalizeTransferPayload { destination_nonce, transfer_id, amount, fee_recipient: None }` + signature `S`.
5. Solana calls `serialize_for_near((mint, recipient))` → produces bytes without `b"hello"` → `keccak256(V1_bytes)` ≠ `keccak256(V2_bytes)` → `secp256k1_recover` returns a different address → `require!(signer.0 == *derived_near_bridge_address)` fails → instruction reverts.
6. Funds remain locked on EVM forever; no refund path exists.

```rust
impl<P: Payload> SignedPayload<P> {
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
