Audit Report

## Title
Unconditional Zero-Value Fee Transfer in `claim()` Permanently Freezes Unclaimed Yield - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `safeTransfer(protocolTreasury, fee)` at line 144 even when `fee == 0`. When `feeInBPS` is configured to zero (a valid value accepted by both `initialize()` and `setFeeInBPS()`) and the distributed token reverts on zero-value transfers, every invocation of `claim()` reverts, permanently preventing all users from claiming their allocated tokens.

## Finding Description
In `contracts/utils/MerkleDistributor/MerkleDistributor.sol`, the `claim()` function computes the fee and unconditionally executes two transfers:

```solidity
// L138-144
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);  // ← always called, even when fee == 0
``` [1](#0-0) 

When `feeInBPS == 0`, `fee` evaluates to `0`, and `safeTransfer(protocolTreasury, 0)` is called unconditionally. The `initialize()` function only rejects `_feeInBPS > MAX_FEE_IN_BPS`, so `_feeInBPS = 0` is explicitly accepted: [2](#0-1) 

The same permissive validation applies to `setFeeInBPS()`: [3](#0-2) 

Certain deployed ERC-20 tokens (e.g., `LEND`, `cUSDCv3`) revert on zero-value transfers. If the `token` configured in this distributor is such a token, every call to `claim()` reverts after the user transfer succeeds but before the function completes, rolling back the entire transaction. No user can ever successfully claim.

The sibling contract `KernelMerkleDistributor` already applies the correct guard: [4](#0-3) 

This guard is absent in `MerkleDistributor`.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** When triggered, no user can claim any tokens from the distributor. Tokens remain locked in the contract with no alternative withdrawal path for claimants. This exactly matches the allowed impact class "Permanent freezing of unclaimed yield."

## Likelihood Explanation
Two conditions must hold simultaneously: (1) `feeInBPS` is set to `0`, which is a normal operational choice explicitly permitted by the contract's validation logic; and (2) the configured `token` reverts on zero-value transfers, a known property of several deployed ERC-20 tokens. Neither condition requires any privileged compromise beyond routine admin configuration. The combination is realistic whenever the protocol deploys a zero-fee distributor for a token with this transfer restriction. The triggering action — calling `claim()` with a valid merkle proof — is performed by an unprivileged external user.

## Recommendation
Guard the fee transfer with a zero-check, mirroring the pattern already used in `KernelMerkleDistributor`:

```solidity
// Before (vulnerable):
IERC20(token).safeTransfer(protocolTreasury, fee);

// After (fixed):
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy `MerkleDistributor` with `feeInBPS = 0` and a `token` that implements `require(amount > 0)` on transfer.
2. Owner calls `setMerkleRoot(root)` with a valid distribution root.
3. Any user calls `claim(index, account, cumulativeAmount, merkleProof)` with a valid proof and non-zero `cumulativeAmount`.
4. At line 138: `fee = (claimableAmount * 0) / 10_000 = 0`.
5. Line 141 succeeds: `safeTransfer(account, claimableAmount)`.
6. Line 144 reverts: `safeTransfer(protocolTreasury, 0)` — the token rejects the zero-value transfer.
7. The entire transaction reverts. Every subsequent claim attempt hits the same revert path. All unclaimed tokens are permanently frozen in the contract.
