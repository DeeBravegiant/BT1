### Title
Forged Foreign-Chain Verification Response via Cross-Request Signature Replay in `respond_verify_foreign_tx` - (File: crates/contract/src/lib.rs)

---

### Summary

`respond_verify_foreign_tx` accepts a caller-supplied `payload_hash` and verifies only that the provided signature is valid over that hash under the root MPC public key. It does **not** verify that `payload_hash` is the correct hash of `ForeignTxSignPayload{request, actual_extracted_values}`. A single attested participant (Byzantine, below threshold) can replay any previously produced valid MPC signature over an arbitrary `payload_hash` to resolve a completely different pending foreign-tx request, delivering a forged verification response to the waiting caller.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract receives two caller-controlled arguments:

- `request: VerifyForeignTransactionRequest` — used as the map key to look up and drain the pending yield queue.
- `response: VerifyForeignTransactionResponse` — contains `payload_hash` (a `Hash256`) and `signature`.

The signature check at lines 718–734 is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← caller-supplied

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← verified against THIS, not against request content
    &secp_pk,
)
.is_ok()
```

The contract confirms only that `signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It never checks that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload{ request: request.request, values: <actual_extracted_values> }))`.

The `ForeignTxSignPayload` structure that nodes are supposed to sign is:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
// msg_hash = SHA-256(borsh(ForeignTxSignPayload))
```

Because the contract never reconstructs or checks this hash, a caller can supply any `payload_hash` for which a valid MPC root-key signature exists — regardless of whether that hash corresponds to the `request` being resolved.

The `request` parameter is used only as a lookup key into `pending_verify_foreign_tx_requests`. Once the signature check passes, `resolve_yields_for` drains all yields queued under that key and delivers `serde_json::to_vec(&response)` — the forged response — to every waiting caller. [1](#0-0)

### Impact Explanation

The `verify_foreign_transaction` / `respond_verify_foreign_tx` flow is the MPC network's mechanism for attesting that a specific foreign-chain transaction occurred with specific observed values (block hash, log data, etc.). Bridge contracts (e.g., Omnibridge inbound flow) rely on this attestation to release funds or trigger state transitions.

By replaying a valid MPC signature over an arbitrary `payload_hash` into a pending request for a different transaction, a Byzantine attested participant can:

1. Make a bridge contract believe that a foreign transaction was verified with fabricated extracted values (e.g., a different block hash, a different log payload, or values from a completely unrelated transaction).
2. Cause the bridge to release funds or execute state transitions based on forged foreign-chain state.
3. Enable double-spend conditions: replay the attestation of a single real transaction to satisfy multiple independent bridge requests.

This matches the allowed impact: **"High. Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."**

---

### Likelihood Explanation

The attacker is a single attested participant — explicitly listed as a valid attacker type ("Byzantine participant strictly below the signing threshold"). The attack requires:

1. **A valid MPC signature over any `payload_hash`** — the attacker participates in the MPC signing protocol and therefore has access to every signature the network has ever produced (they co-produced them). No threshold collusion is needed; the attacker simply reuses an existing signature.
2. **A pending `verify_foreign_transaction` request** — all pending requests are visible on-chain. The attacker can front-run or wait for any request.
3. **Calling `respond_verify_foreign_tx` with the forged arguments** — the attacker is an attested participant and is authorized to call this function.

No key leakage, no TEE attack, no network-level DoS, and no threshold collusion is required.

---

### Recommendation

The contract must bind the `payload_hash` to the `request` content. The simplest on-chain fix is to require the caller to also supply the `extracted_values`, then recompute and assert the hash:

```rust
// In respond_verify_foreign_tx, after the signature check:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: provided_values,   // caller must supply these
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, the contract can store a commitment to the expected `payload_hash` when the request is first queued (once the node-side protocol is extended to commit to the hash before signing), and verify it here. Either approach closes the gap between what the signature covers and what the contract enforces.

### Proof of Concept

**Setup:** MPC network is running. Bitcoin is a supported foreign chain. A legitimate `verify_foreign_transaction` request for Bitcoin tx `A` is pending (`request_A`). The MPC network previously produced a valid root-key signature `sig_X` over `payload_hash_X` (for any prior request — e.g., a Bitcoin tx `B` with a different block hash).

**Attack steps (executed by one attested participant):**

```
1. Observe pending request_A on-chain.
2. Retrieve any previously produced (payload_hash_X, sig_X) — e.g., from the
   respond_verify_foreign_tx call that resolved request_B.
3. Call:
     respond_verify_foreign_tx(
         request  = request_A,
         response = { payload_hash: payload_hash_X, signature: sig_X }
     )
4. Contract executes:
     verify_ecdsa_signature(sig_X, payload_hash_X, root_pk)  →  OK
     resolve_yields_for(pending_verify_foreign_tx_requests, request_A,
                        serialize({payload_hash_X, sig_X}))
5. Every caller waiting on request_A receives {payload_hash_X, sig_X}.
6. A bridge contract that does not independently recompute the expected
   payload_hash from (request_A, actual_values) accepts the forged attestation
   and releases funds.
```

The contract's own test suite confirms that `respond_verify_foreign_tx` succeeds when given a valid signature over an arbitrary `payload_hash` — the test constructs `payload_hash` from fabricated `values` (`[42u8; 32]`) that are never validated against the original request: [4](#0-3)
