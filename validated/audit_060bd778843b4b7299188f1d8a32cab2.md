### Title
Dijkstra Era Subtransaction Reference Script Deserialization Cost Not Charged in Minimum Fee — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee calculation for a top-level transaction does not account for reference scripts referenced by its embedded subtransactions. However, the reference-script size limit check (`validateAllRefScriptSize`) correctly aggregates reference scripts from both the top-level transaction and all subtransactions. This asymmetry allows an unprivileged transaction author to force every validating node to deserialize up to `ppMaxRefScriptSizePerTxG` bytes of reference scripts from subtransaction inputs without paying the `minFeeRefScriptCostPerByte` surcharge for those bytes.

---

### Finding Description

**Background.** The Conway era introduced `minFeeRefScriptCostPerByte` specifically because reference-script deserialization imposes CPU/memory overhead on validators that is not captured by the base per-byte transaction fee. The ADR `docs/adr/2024-08-14_009-refscripts-fee-change.md` documents a real June 2024 attack exploiting this gap in Babbage. Conway fixed it by charging `tierRefScriptFee` on the total size of reference scripts reachable from a transaction's inputs.

**New surface in Dijkstra.** The Dijkstra era adds subtransactions (CDDL field 23 of `transaction_body`). A top-level `DijkstraTx` may embed an `OMap` of `Tx SubTx` values, each of which has its own `inputsTxBodyL` and `referenceInputsTxBodyL`. Validators must deserialize every reference script reachable from those subtransaction inputs.

**Fee calculation — only top-level inputs counted.**

`getMinFeeTxUtxo` for `DijkstraEra` is wired to `getConwayMinFeeTxUtxo`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs, line 141
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
```

`getConwayMinFeeTxUtxo` passes only the top-level transaction's reference-script size to `getMinFeeTx`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs, lines 174-175
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

`txNonDistinctRefScriptsSize` unions only `referenceInputsTxBodyL` and `inputsTxBodyL` of the **top-level** transaction body:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs, lines 183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

Subtransaction inputs are never visited.

**Size-limit check — all inputs counted.**

`validateAllRefScriptSize` in the Dijkstra LEDGER rule uses `batchNonDistinctRefScriptsSize`, which explicitly sums reference scripts from the top-level transaction **and** every subtransaction:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs, lines 321-329
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $
        DijkstraTxRefScriptsSizeTooBig ...
```

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs, lines 271-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

**The gap.** The fee is computed from `txNonDistinctRefScriptsSize` (top-level only); the size cap is enforced on `batchNonDistinctRefScriptsSize` (top-level + subtransactions). An attacker can place all large reference-script inputs inside subtransactions, keeping the top-level reference-script size at zero, while the total batch size approaches `ppMaxRefScriptSizePerTxG`. The `tierRefScriptFee` surcharge for the subtransaction portion is never added to the minimum fee.

---

### Impact Explanation

**Medium — Attacker-controlled transactions modify fees outside design parameters.**

An unprivileged transaction author can craft a Dijkstra top-level transaction whose subtransactions reference UTxO outputs containing large Plutus scripts. Every validating node must deserialize those scripts to execute the subtransactions, but the `minFeeRefScriptCostPerByte` surcharge is not applied to them. The maximum underpayment per transaction is bounded by `ppMaxRefScriptSizePerTxG × minFeeRefScriptCostPerByte`. At current Dijkstra genesis values (`maxRefScriptSizePerTx = 204800` bytes, `minFeeRefScriptCostPerByte = 15` lovelace/byte) this is approximately 3,072,000 lovelace (~3 ADA) per transaction. Repeated at scale this constitutes a sustained below-cost resource drain on all validating nodes, directly analogous to the June 2024 reference-script DDoS documented in the ADR.

---

### Likelihood Explanation

**High.** The attack requires only the ability to submit a valid Dijkstra transaction — no privileged role, no key compromise, no governance majority. The attacker needs only to:
1. Publish UTxO outputs containing large Plutus scripts (one-time cost, paid once).
2. Repeatedly submit top-level transactions whose subtransactions reference those outputs.

The fee savings per transaction are modest but the deserialization overhead imposed on every node is real and unbounded in aggregate.

---

### Recommendation

Replace the `getMinFeeTxUtxo` implementation for `DijkstraEra` with one that passes `batchNonDistinctRefScriptsSize utxo tx` (instead of `txNonDistinctRefScriptsSize utxo tx`) to `getMinFeeTx`, so that the `tierRefScriptFee` surcharge covers reference scripts reachable from all subtransaction inputs as well as the top-level transaction inputs. This mirrors the existing pattern where `validateAllRefScriptSize` already uses `batchNonDistinctRefScriptsSize` for the size-cap check.

---

### Proof of Concept

1. Publish N UTxO outputs, each containing a large Plutus script (total ≤ `ppMaxRefScriptSizePerTxG`).
2. Construct a Dijkstra `TopTx` with an empty `inputsTxBodyL` / `referenceInputsTxBodyL` (zero top-level reference scripts → zero `tierRefScriptFee` surcharge).
3. Embed one `SubTx` whose `referenceInputsTxBodyL` points to all N UTxO outputs from step 1.
4. Submit the transaction. `validateAllRefScriptSize` passes (total ≤ cap). The minimum fee check passes because `txNonDistinctRefScriptsSize` returns 0 for the top-level transaction. Every node deserializes all N scripts from the subtransaction reference inputs, paying the deserialization cost that the fee does not cover.

## Context

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. Unfortunately things did not go exactly as planned, because on [June 25th 2024 an attack like this actually took place](https://cardanospot.io/news/ddos-attack-on-the-cardano-blockchain-mRIKAzZTNnzq5NGd). This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
