Audit Report

## Title
`_afterTimelock` uint32 Truncation Allows Pool Admin to Bypass LP-Protective Timelock Instantly — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

## Summary

`_afterTimelock` computes `uint32(block.timestamp + oracleStopLossConfig[pool_].timelock)`. When `timelock = type(uint32).max`, the `uint256` sum overflows the `uint32` cast and wraps to approximately `block.timestamp − 1`, a timestamp already elapsed. Every subsequent `_requireElapsed` check passes immediately, letting the pool admin propose and execute any parameter change with zero LP reaction time.

## Finding Description

The root cause is in `_afterTimelock`:

```solidity
// L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` (~1.753 × 10⁹ today) + `type(uint32).max` (4,294,967,295) = 6,047,967,295. Truncated to `uint32`: `6,047,967,295 mod 4,294,967,296 ≈ 1,752,999,999 ≈ block.timestamp − 1`.

`_requireElapsed` at L301–303 checks `block.timestamp < executeAfter`; since `executeAfter` is already in the past, the check never reverts.

**Exploit path (initial timelock = 0):**

1. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`. With `timelock = 0`, `_afterTimelock` returns `uint32(block.timestamp)` — no overflow. `executeAfter = block.timestamp`.
2. Admin calls `executeOracleStopLossTimelock(pool)` in the same block. `_requireElapsed(block.timestamp)` passes (`block.timestamp < block.timestamp` is false). Timelock is now `type(uint32).max`.
3. Admin calls `proposeOracleStopLossHighWatermarks(pool, 0, type(uint104).max, type(uint104).max)`. `_afterTimelock` now returns `uint32(block.timestamp + type(uint32).max) ≈ block.timestamp − 1`.
4. Admin calls `executeOracleStopLossHighWatermarks(pool)` immediately. `_requireElapsed(block.timestamp − 1)` passes. Watermarks are set to `type(uint104).max`.
5. Any swap on bin 0 hits `_checkAndUpdateWatermarks`, finds `metric < hwm * floorMultiplier / E6`, and reverts with `OracleStopLossTriggered` — pool is bricked.

`initialize` also stores `timelock` without any upper-bound check (L56–62), so a pool can be deployed with `timelock = type(uint32).max` from day one, skipping steps 1–2 entirely.

Neither `proposeOracleStopLossTimelock` (L78–84) nor `initialize` (L56–62) validate the new timelock value against any cap.

## Impact Explanation

Two concrete harm paths for LPs:

1. **Permanent swap DoS**: Admin sets per-bin watermarks to `type(uint104).max` with zero delay. Every subsequent swap reverts with `OracleStopLossTriggered`. Watermarks can only be lowered through the same now-bypassable timelocked path, making the pool permanently unusable for swaps.

2. **Silent removal of LP protection**: Admin immediately sets `drawdownE6 = 0`. `_afterSwapOracleStopLoss` returns early at L217 (`if (drawdown == 0) return`), silently disabling all stop-loss checks. LPs who deposited expecting stop-loss protection receive none.

Both match the allowed impact gate: "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" and "Admin-boundary break: pool admin exceeds caps, bypasses timelocks."

## Likelihood Explanation

- Pool admin is explicitly semi-trusted; the timelock is the only governance boundary constraining their power over LP-facing parameters.
- Attack requires only two transactions (propose + execute) when starting from `timelock = 0`, or is available from block 1 if `initialize` is called with `timelock = type(uint32).max`.
- No special privileges beyond `onlyPoolAdmin` are required; no off-chain coordination needed.
- The overflow is deterministic and reproducible on any EVM chain where `block.timestamp > 0`.

## Recommendation

Add an upper-bound constant and enforce it in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days;

// In initialize:
if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);

// In proposeOracleStopLossTimelock:
if (newTimelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(newTimelock);
```

Alternatively, guard against overflow in `_afterTimelock` itself:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
    if (result > type(uint32).max) revert OracleStopLossTimelockOverflow();
    return uint32(result);
}
```
