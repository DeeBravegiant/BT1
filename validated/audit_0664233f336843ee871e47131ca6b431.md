Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Deposits Beyond Protocol Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison: the ETH branch checks only `totalAssetDeposits > limit` while the ERC-20 branch correctly checks `totalAssetDeposits + amount > limit`. Any depositor can push the ETH total above the configured cap in a single `depositETH` call, causing excess ETH to sit idle in the pool without generating restaking yield and diluting rsETH returns.

## Finding Description
At [1](#0-0)  the ETH branch on line 679 evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)`, discarding the `amount` parameter entirely. The ERC-20 branch on line 681 correctly includes `amount`. This function is the sole deposit guard, called from `_beforeDeposit` at [2](#0-1) , which is invoked by `depositETH` at [3](#0-2) . Because `amount` is never added before the comparison, a deposit that would push the running total from `limit - ε` to `limit + X` passes the check and rsETH is minted for the full amount. No other guard exists to catch this.

## Impact Explanation
The deposit limit is the protocol's primary mechanism for capping ETH exposure to match EigenLayer restaking capacity. Bypassing it allows ETH to accumulate in the deposit pool beyond what can be restaked, leaving the excess idle and not generating yield. rsETH holders receive diluted returns on that portion. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The condition is reachable by any unprivileged caller invoking `depositETH` with `msg.value > 0` whenever `totalAssetDeposits ≤ limit`. No special role, timing window, or front-running is required. The asymmetry is always present and the gap grows proportionally with deposit size.

## Recommendation
Apply the same inclusive check used for ERC-20 assets to the ETH branch in `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Depositor calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 500 ETH; total ETH becomes `1499 ether`, 49.9% above the cap.
6. Any depositor can repeat until EigenLayer capacity or other external constraints are hit.

Foundry test plan: deploy `LRTDepositPool` with a mock `lrtConfig` returning `depositLimitByAsset = 1000 ether`, seed `totalAssetDeposits` to `999 ether`, call `depositETH{value: 500 ether}`, assert the call succeeds and `getTotalAssetDeposits` returns `1499 ether` — confirming the cap is breached without revert.

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
