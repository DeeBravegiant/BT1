### Title
Farmer Block Reward Unconditionally Diverted by Any Connected Harvester Despite Failed Fee-Quality Check — (`File: chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

### Summary
CHIP-22 introduced a fee convention allowing third-party harvesters to redirect the farmer's block reward (1.75 XCH) to themselves as a fee, gated by a deterministic fee-quality check. The farmer's implementation calls `notify_farmer_reward_taken_by_harvester_as_fee()` only for **logging**, then unconditionally accepts `farmer_reward_address_override` regardless of whether the fee-quality check passes or fails. Any connected harvester can silently divert the farmer's entire block reward to an arbitrary address with no enforcement.

### Finding Description

When a harvester sends `NewProofOfSpace` with `farmer_reward_address_override != None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

Inside that function, the fee-quality check compares `fee_quality` against the harvester-supplied `applied_fee_threshold`. If the check fails, the function **only logs a warning** and returns — it does not reject the override or abort block creation: [2](#0-1) 

After this logging-only call, `_process_respond_signatures()` unconditionally overwrites `farmer_reward_address` with the harvester-supplied value: [3](#0-2) 

The resulting `DeclareProofOfSpace` carries the harvester's address as `farmer_reward_address`, which is embedded in the block's foliage and determines where the 1.75 XCH farmer reward coin is created: [4](#0-3) 

The `fee_info` field is optional (`None` is accepted), and even when present, a failing threshold only produces a `log.warning`. There is no code path that rejects the override or prevents block submission when the fee-quality check fails. [5](#0-4) 

### Impact Explanation

Every block won while a malicious harvester is connected results in the farmer's 1.75 XCH reward being sent to the harvester's chosen address instead of the farmer's configured address. The farmer receives nothing for that block. This is a direct, on-chain payout redirection of XCH with no recourse after the block is confirmed.

### Likelihood Explanation

Any harvester that has been granted a connection to the farmer (including third-party harvesters explicitly supported by CHIP-22) can exploit this. The harvester simply sets `farmer_reward_address_override` to its own puzzle hash and omits or falsifies `fee_info`. No cryptographic material beyond the harvester's existing plot keys is required. The farmer has no mechanism to detect or block this at the protocol level — only after-the-fact log inspection.

### Recommendation

The farmer must enforce the fee-quality check as a hard gate, not a logging advisory. If `farmer_reward_address_override` is set and the fee-quality check fails (either because `fee_info` is absent or `fee_quality > applied_fee_threshold`), the farmer should refuse to proceed with block creation for that proof and log an error. The override should only be accepted when `fee_quality <= applied_fee_threshold` is verified.

### Proof of Concept

1. Malicious harvester connects to a farmer node.
2. Harvester sends `NewProofOfSpace` with a valid proof and `farmer_reward_address_override = attacker_puzzle_hash`, `fee_info = None` (or with `applied_fee_threshold` set to `0xFFFFFFFF`).
3. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()` → logs a warning, returns.
4. Farmer sends `RequestSignatures` to harvester.
5. Harvester responds with `RespondSignatures` also carrying `farmer_reward_address_override = attacker_puzzle_hash`.
6. Farmer sets `farmer_reward_address = attacker_puzzle_hash` unconditionally and emits `DeclareProofOfSpace`.
7. Full node creates the block with the farmer reward coin locked to `attacker_puzzle_hash`.
8. Attacker claims 1.75 XCH; legitimate farmer receives 0 XCH for that block.
