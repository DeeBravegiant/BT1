Audit Report

## Title
Uninitialized `tokenFeeBps` Causes Zero-Fee Token Deposits, Permanently Losing Protocol Fee Revenue — (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps[token]` that is read in `viewSwapRsETHAmountAndFee(uint256, address)` to compute the fee on ERC-20 token deposits. Because `addSupportedToken` never writes to this mapping, it retains Solidity's default value of zero for every newly added token. As a result, every token deposit (e.g., wstETH) pays zero protocol fee, while ETH deposits correctly apply the global `feeBps`. The protocol permanently loses all fee revenue from token deposits unless an admin separately calls `setTokenFeeBps` after each `addSupportedToken` invocation.

## Finding Description
`viewSwapRsETHAmountAndFee(uint256 amount, address token)` at L335–336 reads the fee rate exclusively from the per-token mapping:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];   // defaults to 0
fee = amount * feeBpsForToken / 10_000;         // always 0
``` [1](#0-0) 

`addSupportedToken` at L651–653 registers the oracle and bridge but never touches `tokenFeeBps`:

```solidity
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never set → remains 0
``` [2](#0-1) 

The ETH deposit path at L312 correctly uses the global `feeBps`:

```solidity
fee = amount * feeBps / 10_000;
``` [3](#0-2) 

The token deposit function at L298–300 calls `viewSwapRsETHAmountAndFee(amount, token)` and accumulates the returned fee:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;   // fee is always 0
``` [4](#0-3) 

`setTokenFeeBps` exists as a separate admin call at L583–593 but is never invoked inside `addSupportedToken`, creating a permanent zero-fee state for every token unless the admin explicitly follows up. [5](#0-4) 

The divergence from `RSETHPoolV3.sol` confirms the intent: `RSETHPoolV3` applies the global `feeBps` uniformly to both ETH and token deposits at L324:

```solidity
fee = amount * feeBps / 10_000;
``` [6](#0-5) 

## Impact Explanation
Every token deposit through the permissionless `deposit(address token, uint256 amount, string referralId)` on the live Arbitrum `RSETHPool` pays zero protocol fee instead of the intended `feeBps` rate. The fee revenue that should accrue to the protocol treasury is never collected for any token deposit. This constitutes **High — Theft of unclaimed yield**: yield (fee revenue) the protocol is entitled to collect is permanently foregone on every token deposit.

## Likelihood Explanation
The entry path is fully permissionless — any user can call `deposit(token, amount, referralId)` without any special role or precondition. The zero-fee condition holds for every supported token whose `tokenFeeBps` has not been explicitly set via a separate `setTokenFeeBps` admin call after `addSupportedToken`. This is the default state for all tokens added through the normal `addSupportedToken` flow, making the condition persistent and repeatable across all token deposits.

## Recommendation
Initialize `tokenFeeBps[token]` to the global `feeBps` inside `addSupportedToken`, or replace the per-token lookup in `viewSwapRsETHAmountAndFee(uint256, address)` with the global `feeBps` to match `RSETHPoolV3` behavior:

```solidity
// Option A – initialize on token addition (in addSupportedToken)
tokenFeeBps[token] = feeBps;

// Option B – use global feeBps as fallback in viewSwapRsETHAmountAndFee
uint256 feeBpsForToken = tokenFeeBps[token] == 0 ? feeBps : tokenFeeBps[token];

// Option C – align with RSETHPoolV3 and remove per-token fee entirely
fee = amount * feeBps / 10_000;
```

## Proof of Concept
1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0` (never written).
2. Any user calls `deposit(wstETH, 10 ether, "ref")`.
3. `viewSwapRsETHAmountAndFee(10 ether, wstETH)` computes `feeBpsForToken = tokenFeeBps[wstETH] = 0`, so `fee = 10 ether * 0 / 10_000 = 0`.
4. `feeEarnedInToken[wstETH] += 0` — protocol collects nothing.
5. User receives `rsETHAmount = 10 ether * tokenToETHRate / rsETHToETHrate` (full amount, no deduction), while an ETH depositor of equivalent value pays `feeBps` basis points.
6. **Foundry fork test**: Deploy/fork Arbitrum `RSETHPool`, call `addSupportedToken` with a mock oracle returning non-zero rate, assert `tokenFeeBps[token] == 0`, call `deposit(token, 1e18, "")`, assert `feeEarnedInToken[token] == 0` and emitted `fee == 0` in `SwapOccurred`.
