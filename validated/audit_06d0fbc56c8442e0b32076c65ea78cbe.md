### Title
Hardcoded EVM `CHAIN_ID` in `WalletContract` Enables Signature Replay on a Forked NEAR Chain — (`File: runtime/near-wallet-contract/implementation/wallet-contract/src/internal.rs`)

---

### Summary

The NEAR wallet contract (`WalletContract`) validates incoming RLP-encoded Ethereum transactions by comparing the transaction's `chain_id` field against a compile-time constant (`CHAIN_ID = 397` for mainnet). Because this value is baked in at build time and never read from the live chain state, any signed Ethereum transaction that is valid on NEAR mainnet is equally valid on a hard-forked copy of the chain, enabling full signature replay attacks.

---

### Finding Description

`internal.rs` defines the EVM chain ID as a compile-time constant sourced from the `CHAIN_ID` file:

```rust
// internal.rs line 20
pub const CHAIN_ID: u64 = std::include!("CHAIN_ID");
```

The `CHAIN_ID` file contains the literal value `397` (mainnet) or `398` (testnet), set once at build time:

```
// CHAIN_ID file
{
    // The contents of this file are set by the build file in `near-wallet-contract`.
    // The purpose is to factor out the hard-coded EVM chain ID so that it can be set for
    // each network (mainnet, testnet, etc). The value included in this file is the
    // [one for mainnet](https://chainlist.org/chain/397). The one for testnet is
    // [398](https://chainlist.org/chain/398).
    397
}
```

This constant is the sole chain-binding check in `validate_tx_relayer_data`, which is called for every transaction submitted through the public `rlp_execute` entry point:

```rust
// internal.rs line 328
if tx.chain_id != Some(CHAIN_ID) {
    return Err(Error::Relayer(RelayerError::InvalidChainId));
}
```

Because the check is against a static constant rather than a value derived from the live NEAR chain state (e.g., `env::chain_id()`), both the original chain and any hard fork that inherits the same deployed contract bytecode will accept transactions signed with `chain_id = 397`. The nonce is the only remaining guard, and it is per-contract-instance state that starts at zero on a freshly forked chain (or at whatever value it had at the fork point, which may be lower than on mainnet).

---

### Impact Explanation

An attacker who observes a user's signed Ethereum transaction on NEAR mainnet (e.g., a NEAR token transfer, a function call, an `AddKey` or `DeleteKey` action) can submit the identical RLP bytes to the forked chain via `rlp_execute`. If the contract's nonce on the forked chain is at or below the nonce embedded in the signed transaction, the replay succeeds and the corresponding NEAR action executes with the user's authority. Concretely:

- **Token theft**: A replayed `Transfer` action drains the user's NEAR balance on the forked chain.
- **Key management abuse**: A replayed `AddKey` action grants the attacker a function-call access key; a replayed `DeleteKey` removes the user's own key.
- **Arbitrary function calls**: A replayed `FunctionCall` action executes any contract method the user previously authorized.

The corrupted protocol values are: the wallet contract's `nonce` storage slot, the NEAR account balance, and the account's access-key set — all on the forked chain.

---

### Likelihood Explanation

NEAR hard forks are rare but not hypothetical (protocol upgrades, emergency patches). The attack requires no special privilege: any party who can observe a signed transaction on mainnet (e.g., a relayer, a block explorer user, or anyone who received the signed bytes) can replay it. The only precondition is that the forked chain's contract nonce has not yet advanced past the replayed transaction's nonce, which is trivially true immediately after a fork.

---

### Recommendation

Replace the compile-time constant with a runtime lookup of the NEAR chain ID, then map it to the appropriate EVM chain ID. The NEAR SDK exposes `env::chain_id()` (returns the genesis chain-id string such as `"mainnet"`). The wallet contract should derive or look up the EVM chain ID from this value at transaction-validation time rather than embedding it as a `const`. If gas efficiency is a concern, the EVM chain ID can be cached in contract storage and invalidated whenever `env::chain_id()` changes — mirroring the EIP-712 pattern recommended in the reference report.

```diff
-pub const CHAIN_ID: u64 = std::include!("CHAIN_ID");
+fn current_chain_id() -> u64 {
+    match env::chain_id().as_str() {
+        "mainnet" => 397,
+        "testnet" => 398,
+        _ => panic!("unknown chain id"),
+    }
+}
```

And in `validate_tx_relayer_data`:

```diff
-if tx.chain_id != Some(CHAIN_ID) {
+if tx.chain_id != Some(current_chain_id()) {
```

### Proof of Concept

1. User signs an Ethereum transaction on NEAR mainnet with `chain_id = 397`, `nonce = N`, encoding a `Transfer` action to drain their wallet.
2. NEAR undergoes a hard fork. The forked chain inherits the same `WalletContract` bytecode (with `CHAIN_ID = 397` baked in) and the same contract state (nonce = N or lower).
3. Attacker submits the original signed RLP bytes to the forked chain via `rlp_execute(target, tx_bytes_b64)`.
4. `validate_tx_relayer_data` checks `tx.chain_id != Some(397)` → passes (both chains use 397).
5. Nonce check passes because the forked chain's nonce ≤ N.
6. The `Transfer` action executes, moving the user's NEAR tokens on the forked chain to the attacker-controlled address.
