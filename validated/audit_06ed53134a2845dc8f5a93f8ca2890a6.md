Audit Report

## Title
stETH Negative Rebase Causes `completeWithdrawal()` to Revert for Last Queued Users - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager` transfers a fixed amount of stETH from `LRTUnstakingVault` to itself during `unlockQueue()`, setting each unlocked request's `expectedAssetAmount` to a fixed payout. Because stETH is a rebasing token, a Lido slashing event between `unlockQueue()` and `completeWithdrawal()` reduces the contract's actual stETH balance below the sum of all committed `expectedAssetAmount` values. The last user(s) to call `completeWithdrawal()` receive a revert from `safeTransfer`, while their rsETH has already been burned and cannot be recovered on-chain.

## Finding Description
stETH is registered as a first-class supported asset in `LRTConfig.initialize()` at `contracts/LRTConfig.sol` L54–57.

During `unlockQueue()`, `_unlockWithdrawalRequests()` iterates over pending requests and sets each `request.expectedAssetAmount = payoutAmount`, accumulating `assetAmountToUnlock` as the exact sum of all payout amounts (`contracts/LRTWithdrawalManager.sol` L802–807). Immediately after, `unstakingVault.redeem(asset, assetAmountUnlocked)` transfers exactly that sum to the withdrawal manager (`contracts/LRTWithdrawalManager.sol` L307). At this point the contract holds exactly `assetAmountUnlocked` stETH and the sum of all stored `expectedAssetAmount` values equals `assetAmountUnlocked`.

rsETH is burned at this same step (`contracts/LRTWithdrawalManager.sol` L305), so users have no on-chain recourse.

If a negative stETH rebase occurs before all users call `completeWithdrawal()`, the contract's stETH balance falls below the sum of outstanding `expectedAssetAmount` values. `_processWithdrawalCompletion()` contains a balance-check guard only for the ETH+Aave path (`contracts/LRTWithdrawalManager.sol` L719–732); for ERC20 assets it unconditionally calls `_transferAsset(asset, user, request.expectedAssetAmount)` (`contracts/LRTWithdrawalManager.sol` L734). `_transferAsset` calls `IERC20(asset).safeTransfer(to, amount)` (`contracts/LRTWithdrawalManager.sol` L881), which reverts when the contract balance is insufficient. There is no partial-fill path, no mechanism to reduce `expectedAssetAmount` post-unlock, and no on-chain recovery for the affected user.

## Impact Explanation
**Medium — Temporary freezing of funds.** Affected users have their rsETH permanently burned but cannot receive their stETH. Recovery requires out-of-band operator action (manually topping up the contract with stETH), which is not guaranteed. Until that action occurs, the user's funds are frozen with no on-chain remedy.

## Likelihood Explanation
stETH is a core supported asset. Lido validator slashing is rare but has occurred on mainnet and is a documented, expected behavior of the stETH rebasing mechanism — not an external protocol compromise. The exposure window is the `withdrawalDelayBlocks` period (default ~8 days) between `unlockQueue()` and `completeWithdrawal()`. Any negative rebase of any magnitude during this window, however small, will cause the final user's transfer to fail if the contract holds no excess stETH buffer. No privileged access or attacker action is required; the condition arises passively.

## Recommendation
In `_processWithdrawalCompletion()`, add a balance check for ERC20 assets analogous to the existing ETH+Aave guard: before calling `_transferAsset`, read `IERC20(asset).balanceOf(address(this))` and cap the transfer to `min(request.expectedAssetAmount, contractBalance)`, emitting an event for any shortfall. Alternatively, after `unstakingVault.redeem()` in `unlockQueue()`, record the actual received balance and distribute it proportionally among all unlocked requests rather than using fixed `expectedAssetAmount` values. A simpler mitigation is to always retain a small stETH buffer in the vault and re-read the live balance at completion time.

## Proof of Concept
1. 10 users each call `initiateWithdrawal(stETH, X)`. Each request stores `expectedAssetAmount = A`.
2. Operator calls `unlockQueue(stETH, ...)`. `_unlockWithdrawalRequests` sets each request's `expectedAssetAmount = A` and `assetAmountToUnlock = 10A`. rsETH is burned. `unstakingVault.redeem(stETH, 10A)` transfers exactly `10A` stETH to the withdrawal manager.
3. A Lido slashing event causes a 0.1% negative rebase. The withdrawal manager now holds `9.99A` stETH; all 10 stored `expectedAssetAmount` values remain `A`.
4. Users 1–9 call `completeWithdrawal(stETH)` successfully, each receiving `A` stETH. Contract balance: `9.99A − 9A = 0.99A`.
5. User 10 calls `completeWithdrawal(stETH)`. `_transferAsset` attempts `safeTransfer(user10, A)` but the contract holds only `0.99A`. The call reverts. User 10's rsETH is already burned; their `0.99A` stETH is frozen in the contract with no on-chain recovery path.

**Foundry fork test plan:** Fork mainnet, deploy the withdrawal manager with stETH as the asset, simulate 10 withdrawal requests through `initiateWithdrawal`, call `unlockQueue`, use a `vm.store` write to the stETH shares mapping to simulate a 0.1% negative rebase, have users 1–9 call `completeWithdrawal` successfully, then assert that user 10's `completeWithdrawal` reverts.
