Audit Report

## Title
Unbounded Gas Consumption in `updateRSETHPrice()` via N_assets × N_NDCs Redundant `getQueuedWithdrawals` Calls — (File: `contracts/LRTOracle.sol`, `contracts/NodeDelegator.sol`, `contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is a public function with no access control beyond `whenNotPaused`. Its internal call chain causes `NodeDelegator.getAssetUnstaking()` to be invoked once per supported asset per NDC, each time making a fresh external call to EigenLayer's `getQueuedWithdrawals` and iterating over all queued withdrawals × strategies. The existing `maxUncompletedWithdrawalCount` cap of 80 bounds only the total withdrawal count, not the N_assets × N_NDCs × K_withdrawals × L_strategies product that determines actual gas cost. As the protocol adds assets or NDCs, gas grows proportionally and can approach or exceed the block gas limit, permanently disabling price updates and the auto-pause safety circuit.

## Finding Description

**Call chain:**

`updateRSETHPrice()` (public, `LRTOracle.sol:87`) → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` (`LRTOracle.sol:331`) loops over all N supported assets, calling `ILRTDepositPool.getTotalAssetDeposits(asset)` for each. `getTotalAssetDeposits` calls `getAssetDistributionData(asset)` (`LRTDepositPool.sol:385`), which for each of M NDCs calls `INodeDelegator(ndc).getAssetUnstaking(asset)` (`LRTDepositPool.sol:451`). For ETH, `getETHDistributionData()` (`LRTDepositPool.sol:488`) performs the same per-NDC `getAssetUnstaking` call.

`NodeDelegator.getAssetUnstaking()` (`NodeDelegator.sol:405-427`) unconditionally calls `_getDelegationManager().getQueuedWithdrawals(address(this))` — a full external call returning all queued withdrawals for that NDC — then iterates over every withdrawal × every strategy to filter by asset. This entire traversal is repeated once per asset per NDC.

**Why the existing cap is insufficient:**

`maxUncompletedWithdrawalCount` is hard-capped at 80 (`LRTUnstakingVault.sol:153`). This bounds K_total (total withdrawals across all NDCs). However, the actual gas cost is proportional to:

```
N_assets × M_NDCs × (K_total / M_NDCs) × L_strategies
= N_assets × K_total × L_strategies
```

The M factor cancels, so adding more NDCs does not reduce per-asset gas. Adding more supported assets multiplies the cost linearly. With N=5, K_total=80, L=2: 5 × 80 × 2 = 800 inner strategy iterations plus 5 × M external `getQueuedWithdrawals` calls. The developers' own comment (`LRTUnstakingVault.sol:151`) states "120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price" — this bound counts only K_total, not the N_assets multiplier. The number of supported assets has no hard cap in the codebase.

**Confirmed code references:**

- `updateRSETHPrice()` public, no role guard: `LRTOracle.sol:87-89`
- Asset loop calling `getTotalAssetDeposits` per asset: `LRTOracle.sol:336-348`
- NDC loop calling `getAssetUnstaking` per NDC per asset: `LRTDepositPool.sol:447-455`
- ETH path also calls `getAssetUnstaking` per NDC: `LRTDepositPool.sol:484-493`
- `getAssetUnstaking` external call + nested loops: `NodeDelegator.sol:405-427`
- Incomplete cap with developer acknowledgment: `LRTUnstakingVault.sol:150-158`

## Impact Explanation

If `updateRSETHPrice()` exceeds the block gas limit: (1) `rsETHPrice` becomes permanently stale, causing incorrect rsETH minting and redemption amounts for all users; (2) the auto-pause mechanism in `_updateRsETHPrice()` (`LRTOracle.sol:270-281`) that pauses `LRTDepositPool` and `LRTWithdrawalManager` on excessive price drops can never trigger, removing the protocol's primary downside protection. Both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` share the same internal path and would fail identically.

**Impact: Medium — Unbounded gas consumption leading to permanent freezing of the price update mechanism and disabling of the auto-pause safety circuit.**

## Likelihood Explanation

No attacker action is required. The gas cost grows automatically as the protocol adds supported assets (no hard cap) or processes EigenLayer withdrawals (normal operational activity). The condition is reachable by any public caller of `updateRSETHPrice()` once the protocol reaches sufficient scale. The developers' own comment confirms awareness of the gas sensitivity but the bound they computed understates the true cost by the N_assets factor.

**Likelihood: Medium.**

## Recommendation

1. Cache the result of `getQueuedWithdrawals(ndc)` per NDC and reuse it across all asset iterations in a single price update, eliminating the N_assets × M_NDCs redundant external calls.
2. Introduce an incrementally-maintained `totalAssetUnstaking` mapping updated on `initiateUnstaking` / `completeUnstaking`, eliminating the need to traverse EigenLayer queued withdrawals at price-update time entirely.
3. Enforce a hard cap on the number of supported assets, or restructure `_getTotalEthInProtocol` to avoid the N × K × L traversal.
4. Update the developer comment and cap in `setMaxUncompletedWithdrawalCount` to account for the N_assets multiplier when computing the safe maximum.

## Proof of Concept

**Foundry fork test plan:**

```solidity
// Fork mainnet, configure: 5 supported assets, 10 NDCs, 8 queued withdrawals per NDC (80 total), 2 strategies each
// 1. Deploy/configure LRTConfig with 5 supported assets
// 2. Add 10 NDCs to nodeDelegatorQueue (maxNodeDelegatorLimit = 10)
// 3. Mock EigenLayer DelegationManager.getQueuedWithdrawals to return 8 withdrawals × 2 strategies per NDC
// 4. Call updateRSETHPrice() and measure gas with vm.gas() before/after
// 5. Assert gas consumed > 5_000_000
// 6. Increase to 8 assets (no hard cap prevents this), re-run, assert gas > 8_000_000
// 7. Demonstrate that at realistic mainnet block gas limit (30M), the function fails
// Key assertion: getQueuedWithdrawals is called 50 times (5 assets × 10 NDCs),
// each returning 8 withdrawals × 2 strategies = 800 inner iterations + 50 external calls
```

The call sequence is fully reachable by any unprivileged external caller via `updateRSETHPrice()` with no preconditions beyond the protocol being unpaused and having reached normal operational scale.
