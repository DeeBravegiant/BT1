Audit Report

## Title
Missing `address(0)` Validation for `owner` in `addLiquidity` Permanently Locks LP Tokens - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no zero-address guard. When `owner = address(0)` is supplied, shares are minted under a position key derived from `address(0)`, real tokens are pulled from the caller, but the position can never be burned because `removeLiquidity` enforces `msg.sender == owner` and no EOA can produce `msg.sender == address(0)`. Deposited tokens are permanently locked inside the pool.

## Finding Description
`addLiquidity` at [1](#0-0)  performs two input checks — empty `binIdxs` and length mismatch — but no check that `owner != address(0)`.

`LiquidityLib.addLiquidity` computes the position key via `_positionBinKey(owner, salt, int8(binIdx))` at [2](#0-1)  and writes shares to `positionBinShares[posKey]` at [3](#0-2) . The modify-liquidity callback then pulls real tokens from the caller into the pool.

`removeLiquidity` enforces `msg.sender == owner` at [4](#0-3) . Because no transaction can originate from `address(0)`, shares minted under the zero-address key can never be burned and the underlying tokens can never be recovered.

The periphery guard at [5](#0-4)  only protects callers who route through `MetricOmmPoolLiquidityAdder`. `MetricOmmPool.addLiquidity` is a public external function and any caller can interact with it directly, bypassing the periphery entirely.

## Impact Explanation
Any tokens deposited into a position owned by `address(0)` are permanently locked. The pool's `binTotals` correctly account for them, but no address can ever call `removeLiquidity` to reclaim them. The depositor suffers a total, irrecoverable loss of principal — a direct loss of user funds meeting Critical/High severity under Sherlock thresholds.

## Likelihood Explanation
`addLiquidity` is a public external function requiring no special privilege and no unusual pool state. A caller who interacts with the pool directly and passes `address(0)` as `owner` — whether by mistake, through a buggy integration, or through a malicious wrapper — will permanently lose their deposited tokens. The trigger is trivially reachable and repeatable.

## Recommendation
Add a zero-address guard at the top of `addLiquidity` in `MetricOmmPool.sol`, mirroring the check already present in the periphery:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
+   if (owner == address(0)) revert InvalidOwner();
    if (deltas.binIdxs.length == 0) return (0, 0);
    ...
}
```

## Proof of Concept
```solidity
// Caller interacts with the pool directly, bypassing MetricOmmPoolLiquidityAdder
pool.addLiquidity(
    address(0),   // owner = zero address — no revert
    0,            // salt
    deltas,       // e.g. 10_000 shares in bin 4
    callbackData, // pays tokens from caller's balance
    ""
);

// Shares now exist under keccak256(abi.encode(address(0), 0, 4))
// Tokens are inside the pool and accounted for in binTotals
// removeLiquidity(address(0), ...) requires msg.sender == address(0) — impossible
// => tokens are permanently locked
```
