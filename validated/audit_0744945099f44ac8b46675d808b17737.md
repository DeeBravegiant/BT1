### Title
Unguarded `init` Entrypoint Allows Any Caller to Seize MPC Network Initialization — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract::init` function is callable by **any NEAR account** on an uninitialized contract. Because the standard deployment workflow deploys the contract binary in one transaction and calls `init` in a separate transaction, a racing attacker can front-run the legitimate `init` call, inject attacker-controlled participants, and permanently prevent the legitimate MPC network from initializing.

---

### Finding Description

The `init` function is decorated with `#[init]` but **not** with `#[private]`:

```rust
// crates/contract/src/lib.rs  line 1924-1929
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

In NEAR, `#[init]` only prevents *re-initialization* (panics if state already exists). It does **not** restrict the caller. `#[private]` is the attribute that restricts a call to `predecessor_account_id == current_account_id`. Both sibling init functions correctly carry `#[private]`:

```rust
// init_running — line 1976-1979
#[private]
#[init]
pub fn init_running(...)

// migrate — line 2060-2063
#[private]
#[init(ignore_state)]
pub fn migrate() -> Result<Self, Error> {
``` [2](#0-1) [3](#0-2) 

The documented deployment workflow explicitly separates deployment from initialization into two distinct transactions:

```shell
# Step 1 — deploy without init
near contract deploy mpc-contract.test.near use-file "$MPC_CONTRACT_PATH" \
  without-init-call ...

# Step 2 — init (separate transaction, separate block)
near contract call-function as-transaction mpc-contract.test.near init \
  file-args /tmp/init_args.json ...
``` [4](#0-3) 

The same two-step pattern appears in the TEE deployment script: [5](#0-4) [6](#0-5) 

Between the two transactions the contract state is empty and `init` is callable by anyone.

Compounding the severity: `init` internally calls `TeeState::with_mocked_participant_attestations`, which immediately inserts `VerifiedAttestation::Mock(MockAttestation::Valid)` for every participant supplied by the caller:

```rust
// crates/contract/src/tee/tee_state.rs  line 104-139
pub(crate) fn with_mocked_participant_attestations(participants: &Participants) -> Self {
    ...
    tee_state.stored_attestations.insert(
        tls_public_key,
        NodeAttestation {
            node_id,
            verified_attestation: VerifiedAttestation::Mock(
                attestation::MockAttestation::Valid,
            ),
        },
    );
``` [7](#0-6) 

This means the attacker's injected participants are immediately treated as TEE-attested, bypassing the real attestation submission flow entirely.

---

### Impact Explanation

An attacker who wins the race:

1. Calls `init` with themselves as the sole participant and `threshold = 1`.
2. The contract transitions to `ProtocolContractState::Running` with attacker-controlled `ThresholdParameters`.
3. All attacker accounts receive `MockAttestation::Valid` in `tee_state.stored_attestations`, satisfying `assert_caller_is_attested_participant_and_protocol_active` for every subsequent governance call.
4. The legitimate deployer cannot call `init` again — `#[init]` panics on a non-empty state.
5. The attacker can now vote to add signing domains, trigger DKG (`start_keygen_instance`), and ultimately obtain threshold signing keys for the MPC network — issuing unauthorized chain signatures on behalf of the protocol.

This satisfies **Critical** impact: *Unauthorized threshold signature issuance and bypass of threshold-signature requirements without the required participant authorization.*

---

### Likelihood Explanation

- The deployment scripts universally use `without-init-call` followed by a separate `init` transaction, creating a guaranteed inter-block window.
- Any NEAR account can submit a transaction to an uninitialized contract; no special privilege is required.
- An attacker monitoring the NEAR chain for `DeployContract` actions targeting the known contract account ID can automate the race.
- The window exists on every fresh deployment and every contract upgrade that redeploys the binary before re-initializing.

---

### Recommendation

Add `#[private]` to `init`, consistent with `init_running` and `migrate`:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

With `#[private]`, only a call whose `predecessor_account_id == current_account_id` (i.e., the contract account itself) is accepted, eliminating the open-initialization window. Alternatively, combine deployment and initialization into a single atomic transaction using NEAR's `DeployContract` + function-call batch action.

---

### Proof of Concept

1. Attacker monitors the NEAR chain for a `DeployContract` action on the target MPC contract account (e.g., `v1.signer`).
2. In the next block (before the legitimate `init` is submitted), attacker submits:

```json
near contract call-function as-transaction v1.signer init json-args '{
  "parameters": {
    "participants": {
      "next_id": 1,
      "participants": [[
        "attacker.near",
        {"id": 0},
        {"tls_public_key": "<attacker_ed25519_pubkey>"}
      ]]
    },
    "threshold": {"value": 1}
  }
}' prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
   sign-as attacker.near network-config mainnet sign-with-keychain send
```

3. The contract initializes in `Running` state with `attacker.near` as the sole participant, carrying a `MockAttestation::Valid` entry in `tee_state`.
4. The legitimate `init` call fails: `"State already exists"`.
5. Attacker calls `vote_add_domains` and `start_keygen_instance` from `attacker.near`, completing DKG with their own node, and gains the ability to issue threshold signatures for any foreign-chain payload. [8](#0-7) [9](#0-8)
