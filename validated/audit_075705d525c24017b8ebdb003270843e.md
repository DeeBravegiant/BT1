Audit Report

## Title
Unbounded Gas Loop in `_unlockWithdrawalRequests` Enables Temporary Freezing of Legitimate Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

## Summary
Any rsETH holder can call `initiateWithdrawal()` with the minimum allowed amount (1 wei when `minRsEthAmountToWithdraw[asset]` is unset, defaulting to 0) to flood the global FIFO withdrawal queue. The `_unlockWithdrawalRequests` while-loop iterates O(N) over all queued entries with no per-call iteration cap, allowing an attacker to force the operator's `unlockQueue()` call to exhaust the block gas limit. Legitimate users whose requests sit behind the attacker's entries in the queue cannot complete withdrawals until all attacker entries are processed.

## Finding Description
`initiateWithdrawal()` enforces only `rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]`. Since `minRsEthAmountToWithdraw` is a mapping that defaults to 0 in Solidity, the effective minimum is 1 wei of rsETH. [1](#0-0) 

Each call to `_addUserWithdrawalRequest` writes a new `WithdrawalRequest` struct to storage and increments `nextUnusedNonce[asset]`, growing the global queue without bound. [2](#0-1) 

`_unlockWithdrawalRequests` iterates from `nextLockedNonce[asset]` up to `firstExcludedIndex` (capped at `nextUnusedNonce[asset]`). The only early-exit conditions are: (1) the withdrawal delay has not yet elapsed, and (2) available assets are insufficient to cover the current request's payout. [3](#0-2) 

Because each attacker entry commits only a tiny `expectedAssetAmount` (e.g., 1 wei), condition (2) never triggers early. After the 8-day delay passes, the loop must visit every attacker-inserted entry before reaching legitimate users' requests. Each iteration performs multiple SSTOREs and SLOADs (~60k gas), so ~500 iterations exhaust the 30M block gas limit. With tens of thousands of spam entries, a single `unlockQueue()` call reverts.

`completeWithdrawal()` checks `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]` and reverts with `WithdrawalLocked` if the request has not been unlocked. Since `nextLockedNonce[asset]` cannot advance past the attacker's entries without processing them, legitimate users behind those entries are blocked. [4](#0-3) 

## Impact Explanation
**Temporary freezing of funds (Medium):** Legitimate users whose withdrawal requests have nonces greater than the attacker's entries cannot call `completeWithdrawal()` successfully until the operator processes all preceding attacker entries. This requires many batched `unlockQueue()` calls, during which legitimate funds remain locked in the contract.

**Unbounded gas consumption (Medium):** If the operator passes a large `firstExcludedIndex` (e.g., `type(uint256).max` or `nextUnusedNonce[asset]`), the while-loop runs out of gas and reverts, completely blocking the unlock path until the operator switches to smaller batches.

Both impacts are explicitly listed in the allowed impact scope.

## Likelihood Explanation
- `initiateWithdrawal()` is callable by any rsETH holder with no rate limiting or per-user queue cap.
- `minRsEthAmountToWithdraw[asset]` defaults to 0, making each spam entry cost only gas (~21k + calldata + storage writes, feasible at scale).
- The attacker recovers their rsETH value as the underlying asset after the delay; net cost is gas fees only.
- The attack is repeatable across all supported assets and is not dependent on any privileged action or external condition.
- No existing guard in the contract limits the number of withdrawal requests per user or globally per asset.

## Recommendation
1. **Enforce a non-zero minimum withdrawal amount** for every supported asset via `setMinRsEthAmountToWithdraw`, making spam economically prohibitive.
2. **Add a per-user or global per-asset cap** on pending withdrawal requests to bound queue growth.
3. **Add an explicit iteration cap** inside `_unlockWithdrawalRequests` (e.g., `maxIterations` parameter) so the operator cannot accidentally pass an unbounded range.
4. **Document and enforce safe batch sizes** for `firstExcludedIndex` in `unlockQueue()` operator tooling.

## Proof of Concept
1. Attacker deposits ETH via `LRTDepositPool.depositETH()` to receive rsETH.
2. Attacker calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "")` 50,000 times (valid when `minRsEthAmountToWithdraw[ETH_TOKEN] == 0`). Each call increments `nextUnusedNonce[ETH_TOKEN]` and writes a new storage entry.
3. Legitimate users submit withdrawal requests at nonces 50,000+.
4. After the 8-day (`withdrawalDelayBlocks`) delay, the operator calls `unlockQueue(ETH_TOKEN, type(uint256).max, ...)`.
5. `firstExcludedIndex` is capped at `nextUnusedNonce[ETH_TOKEN]` (50,000+). The while-loop attempts to iterate all 50,000 attacker entries; each iteration costs ~60k gas. At 30M block gas limit, the transaction reverts after ~500 iterations.
6. `nextLockedNonce[ETH_TOKEN]` does not advance past the attacker's entries.
7. Legitimate users call `completeWithdrawal(ETH_TOKEN, "")`. The check `usersFirstWithdrawalRequestNonce >= nextLockedNonce[ETH_TOKEN]` reverts with `WithdrawalLocked` for all of them.
8. The operator must issue ~100 batched `unlockQueue()` calls with `firstExcludedIndex` incremented by ~500 each time to drain the attacker's entries before legitimate users can withdraw.
