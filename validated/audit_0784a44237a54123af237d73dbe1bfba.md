### Title
Caller Deposit Permanently Lost on Early-Return Error Paths in `nep_141_storage_balance_callback` - (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary

The `WalletContract::nep_141_storage_balance_callback` function has multiple early-return error paths that return an `ExecuteResponse` failure value without first refunding the caller's attached deposit tracked in `caller_deposit`. Any external caller who attached a non-zero NEAR deposit to `rlp_execute` for an ERC-20 transfer path permanently loses that deposit when the intermediate `storage_balance_of` call fails or returns unparseable data.

### Finding Description

`rlp_execute` is `#[payable]` and captures the caller's attached deposit into a `CallerDeposit` struct at the start of `inner_rlp_execute`: [1](#0-0) 

`CallerDeposit::new` records the `predecessor_account_id` and the attached yoctoNEAR amount for any external caller with a non-zero deposit: [2](#0-1) 

For ERC-20 transfers, `inner_rlp_execute` chains a `storage_balance_of` view call to `nep_141_storage_balance_callback`, passing `caller_deposit` through: [3](#0-2) 

Inside `nep_141_storage_balance_callback`, two early-return paths exist that return an error `ExecuteResponse` **without issuing any refund to the caller**: [4](#0-3) 

The `caller_deposit` argument is received but silently dropped in both early-return branches. The correct refund path only executes in `rlp_execute_callback` when the downstream promise fails: [5](#0-4) 

The same defect exists in `address_check_callback` when the address registrar call fails: [6](#0-5) 

### Impact Explanation

The corrupted protocol value is the caller's NEAR token balance. When `storage_balance_of` fails or returns malformed JSON, the wallet contract retains the caller's attached deposit permanently — it is neither forwarded to the token contract nor returned. The deposit is effectively burnt from the caller's perspective. This directly corrupts the `account.amount` balance entry for the caller in the NEAR state trie.

### Likelihood Explanation

`rlp_execute` is `#[payable]` and the `CallerDeposit` mechanism was explicitly designed to handle external callers who attach deposits (as demonstrated by `test_caller_refunds`). A caller attaches a deposit when the Ethereum transaction encodes a NEAR function call with a non-zero `yocto_near` value, or when a relayer forwards value on behalf of a user. If the target token contract is temporarily unavailable, panics on `storage_balance_of`, or returns non-JSON data (e.g., a malicious or buggy token contract), the early-return path is triggered and the deposit is lost. The trigger is reachable by any unprivileged external caller through a standard public RPC transaction.

### Recommendation

In both early-return error branches of `nep_141_storage_balance_callback` (and `address_check_callback`), issue the caller deposit refund before returning, mirroring the pattern used in `rlp_execute_callback`:

```rust
PromiseResult::Failed => {
    // Refund caller deposit before returning
    if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
        let refund_promise = env::promise_batch_create(&account_id);
        env::promise_batch_action_transfer(
            refund_promise,
            NearToken::from_yoctonear(yocto_near.into()),
        );
    }
    return PromiseOrValue::Value(ExecuteResponse { ... });
}
```

### Proof of Concept

1. Deploy a token contract whose `storage_balance_of` method panics unconditionally.
2. External caller calls `rlp_execute` on a wallet contract with an attached deposit of, say, 3 NEAR, encoding an ERC-20 transfer to that token contract.
3. `inner_rlp_execute` creates `CallerDeposit { account_id: caller, yocto_near: 3e24 }` and schedules `storage_balance_of → nep_141_storage_balance_callback(caller_deposit)`.
4. `storage_balance_of` panics; `nep_141_storage_balance_callback` receives `PromiseResult::Failed` and hits the early-return at line 204–209 — `caller_deposit` is dropped without a refund transfer.
5. Caller's balance decreases by 3 NEAR permanently; the wallet contract retains those tokens with no mechanism to recover them. [7](#0-6) [5](#0-4)
