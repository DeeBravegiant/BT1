Audit Report

## Title
Missing Caller Authentication in `MerkleDistributor.claim()` Enables Forced Claims at Unfavorable Fee Rates — (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` accepts an arbitrary `account` parameter with no check that `msg.sender == account`, allowing any unprivileged caller to execute a valid claim on behalf of any user. Because `feeInBPS` (up to 10%) is deducted at claim time and is owner-adjustable, an attacker can front-run a fee-reduction transaction by force-claiming for all eligible users at the current high fee rate, permanently diverting the fee-difference to `protocolTreasury` and away from users who would have claimed at the lower rate. The sister contract `KernelMerkleDistributor` explicitly guards against this with an identity check that is absent here.

## Finding Description

`MerkleDistributor.claim()` accepts `account` as a caller-supplied parameter and performs no identity check:

```solidity
// contracts/utils/MerkleDistributor/MerkleDistributor.sol L97-106
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
```

The merkle proof binds `(index, account, cumulativeAmount)` together, so the proof is valid only for the correct `account`. However, anyone who possesses a valid proof (public off-chain data served by the frontend) can supply it and execute the claim on behalf of any victim:

```solidity
// L120-123
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
    revert InvalidMerkleProof();
}
```

After the claim, the fee is irrevocably deducted and sent to `protocolTreasury`:

```solidity
// L138-144
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;
IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);
```

The fee rate is mutable by the owner at any time:

```solidity
// L198-206
function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
    ...
    feeInBPS = _feeInBPS;
}
```

Once a claim is processed, `userClaims[account].lastClaimedIndex` is updated, and `isClaimed` returns `true`, preventing any re-claim at a lower fee rate:

```solidity
// L115-117
if (isClaimed(index, account)) {
    revert AlreadyClaimed();
}
```

By contrast, `KernelMerkleDistributor._processClaim()` enforces the identity check that is missing here:

```solidity
// KernelMerkleDistributor.sol L311-313
if (account != msg.sender) {
    revert Unauthorized();
}
```

The exploit path:
1. `feeInBPS = 1000` (10%). Owner broadcasts `setFeeInBPS(0)` to the mempool.
2. Attacker observes the pending transaction and front-runs it by calling `claim(index, victim, cumulativeAmount, proof)` for every eligible user while the fee is still 10%.
3. Each victim's claim is processed at 10% fee; the fee is irrevocably sent to `protocolTreasury`.
4. Owner's `setFeeInBPS(0)` lands, but all claims are already settled at the old rate.
5. Every victim permanently loses up to 10% of their allocation — yield they would have retained had they claimed after the fee reduction. Re-claiming is blocked by `AlreadyClaimed`.

## Impact Explanation

**High — Theft of unclaimed yield.**

Users' unclaimed token allocations are permanently reduced by up to 10% (`MAX_FEE_IN_BPS = 1000`) without their consent. The fee-difference between the old and new rate represents yield that was unclaimed at the time of the attack and is permanently diverted to `protocolTreasury` rather than to the rightful owner. The user cannot recover this amount because the `isClaimed` guard prevents re-claiming. This directly matches the "Theft of unclaimed yield" impact category.

## Likelihood Explanation

**Medium.**

- `feeInBPS` is non-zero by design (up to 10% max); fee reductions are a normal protocol lifecycle event.
- Mempool front-running of a `setFeeInBPS` call requires no privileged access — any EOA can execute it.
- The attacker needs only the merkle proof data, which is public off-chain data served by the frontend.
- The attack is fully permissionless and requires no capital beyond gas.
- The attack is repeatable for every fee-reduction event.

## Recommendation

Add an identity check at the top of `claim()`, mirroring the pattern already used in `KernelMerkleDistributor._processClaim()`:

```solidity
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
+   if (account != msg.sender) revert Unauthorized();
    // ... rest of function
}
```

Alternatively, remove the `account` parameter entirely and derive it from `msg.sender`, as `KernelTop100MerkleDistributor.claim()` does.

## Proof of Concept

```
Setup:
  - MerkleDistributor deployed with feeInBPS = 1000 (10%)
  - Alice is entitled to 1000 tokens
    (leaf: keccak256(abi.encodePacked(1, alice, 1000)))
  - Owner broadcasts setFeeInBPS(0) to mempool

Attack (front-runs setFeeInBPS(0)):
  attacker.call(
    merkleDistributor.claim(1, alice, 1000, aliceProof)
  )

Result:
  - Alice receives 900 tokens (10% fee = 100 tokens sent to treasury)
  - setFeeInBPS(0) lands — but Alice's claim is already settled
  - isClaimed(1, alice) == true → AlreadyClaimed on any re-claim attempt
  - Alice permanently lost 100 tokens she would have kept at 0% fee
  - Attacker spent only gas; no capital required

Foundry test outline:
  1. Deploy MerkleDistributor with feeInBPS = 1000.
  2. Build a merkle tree with Alice's leaf; call setMerkleRoot.
  3. As attacker (not Alice), call claim(index, alice, 1000, proof).
  4. Assert Alice received 900 tokens and treasury received 100.
  5. Call setFeeInBPS(0) as owner.
  6. Attempt claim(index, alice, 1000, proof) again → expect AlreadyClaimed revert.
  7. Confirm Alice cannot recover the 100-token fee difference.
```
