Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables Yield Theft via Deposit Frontrunning — (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` carries no access control, allowing any external caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at an arbitrary time. An attacker deposits ETH immediately before calling `sendFunds()`, diluting the reward share of all pre-existing rsETH holders and capturing yield they did not earn. The impact is theft of unclaimed yield from long-term rsETH holders.

## Finding Description
`FeeReceiver.sendFunds()` at line 53 has no role modifier:

```solidity
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

When called, the ETH lands in `LRTDepositPool` and is immediately counted in `getETHDistributionData()` via `address(this).balance`: [2](#0-1) 

`_getTotalEthInProtocol()` in `LRTOracle` aggregates this balance when computing the new rsETH price: [3](#0-2) 

The new price is computed as `(totalETHInProtocol - protocolFeeInETH) / rsethSupply`: [4](#0-3) 

`updateRSETHPrice()` is also publicly callable: [5](#0-4) 

**Exploit path:**
1. Attacker observes `FeeReceiver.balance = R` (accumulated MEV rewards).
2. Attacker calls `LRTDepositPool.depositETH()` with amount `X`, receiving rsETH at the current price. TVL and supply both increase proportionally; price is unchanged.
3. Attacker calls `FeeReceiver.sendFunds()`. `R` ETH is added to the deposit pool TVL; rsETH supply is unchanged.
4. Attacker calls `LRTOracle.updateRSETHPrice()`. The new price reflects the inflated TVL.
5. Attacker's rsETH is now worth `X + X·R/(T+X)` ETH, where `T` is the pre-deposit TVL.
6. Attacker initiates withdrawal through `LRTWithdrawalManager` and recovers principal plus stolen yield after the EigenLayer delay.

**Existing guard analysis:** `_updateRsETHPrice()` contains a `pricePercentageLimit` check at lines 252–266 that reverts for non-manager callers if the price increase exceeds the configured threshold: [6](#0-5) 

This guard is **insufficient** for two reasons: (a) it is only active when `pricePercentageLimit > 0` — if the value is 0 (default/unconfigured), the check is entirely bypassed; (b) even when configured, the attacker can execute the attack with reward amounts small enough to stay within the threshold, repeating the attack across multiple reward accumulation cycles. The `_checkAndUpdateDailyFeeMintLimit` check similarly does not prevent the price update itself — it only limits fee minting and is bypassed when `protocolFeeInBPS == 0`. [7](#0-6) 

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders lose a fraction of every MEV/execution-layer reward batch proportional to the attacker's deposit size relative to TVL. In the numerical example: 1000 ETH TVL, 10 ETH reward, attacker deposits 100 ETH → attacker captures 0.909 ETH of the 10 ETH reward that should have accrued entirely to pre-existing holders. MEV bots can automate this continuously across every reward accumulation cycle.

## Likelihood Explanation
**Medium.** The attack requires locking up capital for the EigenLayer withdrawal delay (~7 days), but no privileged access is needed. Both `sendFunds()` and `updateRSETHPrice()` are permissionless. The attack is fully on-chain, automatable by MEV searchers, and profitable whenever accumulated `FeeReceiver` balance is non-trivial. No flash loan is required. The `pricePercentageLimit` guard can be circumvented by sizing the attack to stay within the threshold.

## Recommendation
Restrict `sendFunds()` to an authorized role to prevent attackers from controlling the timing of reward distribution:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
``` [1](#0-0) 

Alternatively, implement a streaming reward model (e.g., Synthetix `rewardRate`) so rewards accrue continuously rather than in lump sums, eliminating the timing advantage entirely.

## Proof of Concept
**Foundry fork test outline:**

```solidity
function testDepositFrontrunSendFunds() public fork {
    // Setup: FeeReceiver has accumulated 10 ETH in MEV rewards
    vm.deal(address(feeReceiver), 10 ether);

    uint256 tvlBefore = lrtOracle.rsETHPrice(); // record price

    // Step 1: Attacker deposits 100 ETH at current price
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 100 ether}(0, "");
    uint256 attackerRsETH = rsETH.balanceOf(attacker);

    // Step 2: Attacker flushes rewards
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // Step 3: Attacker triggers price update
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();

    uint256 priceAfter = lrtOracle.rsETHPrice();
    // Assert: price increased, attacker's rsETH is worth more than deposited
    assertGt(priceAfter, tvlBefore);
    uint256 attackerETHValue = attackerRsETH * priceAfter / 1e18;
    assertGt(attackerETHValue, 100 ether); // attacker profited from existing holders' yield
}
```

The test confirms: (1) `sendFunds()` is callable by any address, (2) the price update succeeds for a non-manager when the reward is within `pricePercentageLimit`, and (3) the attacker's rsETH redeems for more ETH than deposited, at the expense of pre-existing holders.
