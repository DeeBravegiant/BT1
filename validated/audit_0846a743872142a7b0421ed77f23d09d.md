Audit Report

## Title
Stale-Price Sandwich on `updateRSETHPrice()` Enables Theft of Unclaimed Yield — (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is an unrestricted `public` function gated only by `whenNotPaused`. Because `LRTDepositPool.getRsETHAmountToMint()` divides by the stored `rsETHPrice` and `LRTWithdrawalManager.getExpectedAssetAmount()` multiplies by the same stored value, an attacker who deposits before a price update and triggers the update themselves can lock in a withdrawal entitlement at the post-reward price while having paid only the pre-reward price — extracting a pro-rata share of pending yield they never earned.

## Finding Description

**Step 1 — Unrestricted price update**

`updateRSETHPrice()` carries only `whenNotPaused`; any EOA can call it at any time. [1](#0-0) 

**Step 2 — Deposit mints at stale stored price**

`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()`, the last *written* value. If reward ETH has arrived in the pool (via `receive()` or `receiveFromRewardReceiver()`) but the price has not yet been updated, this value is stale and the attacker receives more rsETH than they are entitled to. [2](#0-1) 

**Step 3 — Withdrawal locks in the post-update price**

`initiateWithdrawal` calls `getExpectedAssetAmount`, which reads the *current* stored `rsETHPrice` and stores the result as `expectedAssetAmount`. After the attacker calls `updateRSETHPrice()`, the stored price reflects the reward, so the withdrawal entitlement is computed at the higher post-reward price. [3](#0-2) [4](#0-3) 

**Step 4 — `_calculatePayoutAmount` does not undo the profit**

At `unlockQueue` time, `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`. If the price at unlock time equals or exceeds P₂ (the post-update price at initiation), `currentReturn ≥ expectedAssetAmount` and the attacker receives the full `expectedAssetAmount` computed at P₂. The min() guard only protects against price *decreases* after initiation; it does not reverse the profit already embedded in `expectedAssetAmount`. [5](#0-4) 

**Step 5 — Payout is transferred directly from the stored value**

`_processWithdrawalCompletion` transfers `request.expectedAssetAmount` (as updated by `unlockQueue`) to the user with no further price check. [6](#0-5) 

**`pricePercentageLimit` is a partial mitigation only**

`_updateRsETHPrice` reverts with `PriceAboveDailyThreshold` for non-managers when the price increase exceeds `pricePercentageLimit`. However: (a) `pricePercentageLimit` defaults to 0 (no protection), and (b) even when set, any reward small enough to stay within the threshold is fully exploitable. The attacker can repeat the attack every update cycle. [7](#0-6) 

## Impact Explanation

Let T = TVL before reward, S = rsETH supply, P₁ = T/S (stored stale price), R = reward ETH in pool, D = attacker deposit.

- Attacker mints at P₁: `rsETH_attacker = D·S/T`
- After `updateRSETHPrice()`: P₂ = (T+D+R)·T / (S·(T+D))
- `expectedAssetAmount` = `rsETH_attacker × P₂` = D·(T+D+R)/(T+D) = D + D·R/(T+D)
- **Attacker profit = D·R/(T+D)** — extracted from existing rsETH holders' pending yield

This is **High — Theft of unclaimed yield**. The protocol remains solvent; existing holders each lose `R·D / (S·(T+D))` ETH per rsETH of pending reward.

## Likelihood Explanation

- `updateRSETHPrice()` requires no privilege; any EOA can call it.
- Reward ETH regularly arrives in `LRTDepositPool` via `receive()` / `receiveFromRewardReceiver()` before the off-chain keeper updates the price — this is the normal operational pattern.
- The only required privileged step (`unlockQueue` by an operator) is a routine operational action that occurs regardless of the attacker's presence.
- The 8-day `withdrawalDelayBlocks` delays but does not prevent profit realization, since `expectedAssetAmount` is locked in at initiation.
- No admin compromise, governance capture, or oracle manipulation is required.
- The attack is repeatable every reward cycle when `pricePercentageLimit` is 0 or the reward is within the threshold.

## Recommendation

1. **Restrict `updateRSETHPrice()`** to a trusted keeper role (e.g., `onlyLRTManager` or a dedicated `PRICE_UPDATER_ROLE`). The existing `updateRSETHPriceAsManager()` already provides the privileged path; the public variant should be removed or similarly gated.
2. **Alternatively**, compute rsETH-to-mint using a freshly calculated price (calling `_getTotalEthInProtocol()` inline) rather than the stored `rsETHPrice`, so a stale stored value cannot be exploited at deposit time.
3. Ensure `pricePercentageLimit` is always set to a non-zero value as defence-in-depth.

## Proof of Concept

```solidity
// Foundry fork test (local fork only)
function testSandwichYieldTheft() public {
    // Precondition: T=1000 ETH TVL, S=1000 rsETH, P1=1e18
    // Reward: 10 ETH sent to depositPool before price update
    vm.deal(address(depositPool), address(depositPool).balance + 10 ether);

    uint256 D = 100 ether;
    vm.deal(attacker, D);

    // Step 1: deposit at stale price P1 — attacker gets 100 rsETH
    vm.prank(attacker);
    depositPool.depositETH{value: D}(0, "");
    uint256 rsETHReceived = rsETH.balanceOf(attacker);
    // rsETHReceived = 100e18

    // Step 2: attacker triggers price update — no access control
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();
    // P2 = (1000+100+10)*1e18 / (1000+100) ≈ 1.00909e18

    // Step 3: initiate withdrawal at P2
    vm.prank(attacker);
    rsETH.approve(address(withdrawalManager), rsETHReceived);
    vm.prank(attacker);
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHReceived, "");
    // expectedAssetAmount ≈ 100.909 ETH

    // Step 4: operator calls unlockQueue after delay
    vm.roll(block.number + withdrawalManager.withdrawalDelayBlocks() + 1);
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, 0, type(uint256).max, 0, type(uint256).max);

    // Step 5: attacker completes withdrawal
    vm.prank(attacker);
    withdrawalManager.completeWithdrawal(ETH_TOKEN, "");

    // Attacker deposited 100 ETH, receives ≈100.909 ETH
    // Profit ≈ 100 * 10 / (1000+100) ≈ 0.909 ETH stolen from existing holders
    assertGt(attacker.balance, D);
}
```
