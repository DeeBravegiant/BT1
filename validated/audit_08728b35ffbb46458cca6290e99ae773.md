### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Enables Cross-Request Payload Substitution — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never verifies that `response.payload_hash` is actually derived from the `request` argument supplied in the same call. A single Byzantine attested participant (below the signing threshold) who holds any previously-obtained valid MPC signature can resolve a pending foreign-chain verification request with a `payload_hash` that encodes a completely different transaction, delivering a forged attestation to the waiting bridge contract.

---

### Finding Description

The two-phase foreign-chain verification flow is:

**Phase 1 — `verify_foreign_transaction`** (`crates/contract/src/lib.rs` lines 519–557): a user submits a `VerifyForeignTransactionRequest` (containing `tx_id`, `extractors`, `domain_id`, `payload_version`). The contract stores it in `pending_verify_foreign_tx_requests` keyed by the full request struct and issues a yield.

**Phase 2 — `respond_verify_foreign_tx`** (`crates/contract/src/lib.rs` lines 691–754): an attested MPC node submits `(request, response)` where `response = {payload_hash, signature}`. The contract performs exactly two checks:

```rust
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← taken verbatim from the caller-supplied response
    &secp_pk,
)
.is_ok()
```

and then:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,                          // ← used only as a map key
    serde_json::to_vec(&response).unwrap(),  // ← response with unchecked payload_hash
)
```

The contract **never** verifies that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request: request, values: <any> }))`. The `payload_hash` field is completely unconstrained with respect to the `request` argument — it only needs to carry a valid MPC signature.

The canonical hash that honest nodes compute is:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs lines 34-35
let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
    foreign_tx_payload.compute_msg_hash()?.into();
```

where `foreign_tx_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.clone(), values })`. The `request` field is embedded inside the hash. The contract never re-derives or checks this embedding.

---

### Impact Explanation

A Byzantine attested participant (strictly below the signing threshold) who has previously obtained any valid MPC signature `(payload_hash_B, sig_B)` — e.g., from a prior legitimate `verify_foreign_transaction` call for a different or attacker-controlled transaction B — can call:

```
respond_verify_foreign_tx(
    request  = <pending request for tx_id A>,
    response = { payload_hash: payload_hash_B, signature: sig_B }
)
```

The contract accepts the call (signature over `payload_hash_B` is valid), resolves the yield for tx_id A, and returns `VerifyForeignTransactionResponse { payload_hash: payload_hash_B, signature: sig_B }` to the bridge contract that submitted the request for tx_id A.

The bridge contract now holds a valid MPC-signed attestation whose `payload_hash` encodes transaction B (e.g., a small or fake deposit), not transaction A (the real deposit it was waiting for). If the bridge contract does not independently recompute the expected hash and compare it — a step that is optional and performed only in the off-chain SDK helper `ForeignChainSignatureVerifier::verify_signature` (`crates/near-mpc-sdk/src/foreign_chain.rs` lines 53–63), not enforced on-chain — it will process the wrong attestation, enabling double-spend or invalid minting.

This maps to: **High — forged foreign-chain verification that causes invalid bridge execution**.

---

### Likelihood Explanation

- The attacker needs only to be a single attested MPC participant (no threshold collusion required).
- Obtaining a prior valid signature is trivial: submit any legitimate `verify_foreign_transaction` request for a transaction the attacker controls, wait for the honest MPC network to sign it, and retain the response.
- The attack is executable in a single on-chain transaction once the attacker holds a prior signature.
- Bridge contracts that do not use the SDK's `ForeignChainSignatureVerifier` (or implement equivalent hash-binding checks) are directly vulnerable to fund loss.

---

### Recommendation

In `respond_verify_foreign_tx`, require the caller to also supply the full `ForeignTxSignPayload` (or at minimum the `values` vector), then recompute and assert:

```rust
let expected_hash = payload.compute_msg_hash()?;
assert!(payload.request() == &request.request, "request mismatch");
assert!(expected_hash == response.payload_hash, "payload_hash mismatch");
```

Alternatively, store a commitment to the expected `payload_hash` prefix (the Borsh encoding of the `request` portion) at submission time and verify it in the response phase. This mirrors the fix pattern from the referenced report: the "start" phase must constrain what the "end" phase will accept.

---

### Proof of Concept

1. Attacker controls MPC node `N` (one attested participant, below threshold).
2. Attacker submits `verify_foreign_transaction` for their own transaction B on Bitcoin (small value). Honest nodes sign it; attacker retains `(payload_hash_B, sig_B)`.
3. Bridge contract submits `verify_foreign_transaction` for a large inbound deposit tx_id A. Request is stored in `pending_verify_foreign_tx_requests` keyed by `request_A`.
4. Attacker's node `N` calls:
   ```
   respond_verify_foreign_tx(request = request_A, response = {payload_hash_B, sig_B})
   ```
5. Contract at line 729 verifies `sig_B` over `payload_hash_B` — valid. No check that `payload_hash_B` encodes `request_A`.
6. `resolve_yields_for` resolves the yield for `request_A` with `response = {payload_hash_B, sig_B}`.
7. Bridge contract receives `VerifyForeignTransactionResponse` with `payload_hash_B` (encoding tx_id B, not tx_id A). If it skips the SDK hash-binding check, it mints tokens for the large deposit based on the small-deposit attestation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
