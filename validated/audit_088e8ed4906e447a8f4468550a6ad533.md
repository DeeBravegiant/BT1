### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards to Arbitrary Address via Unenforced `farmer_reward_address_override` — (`File: chia/farmer/farmer_api.py`)

### Summary
The CHIP-22 third-party harvester fee mechanism allows a connected harvester to set `farmer_reward_address_override` in both `NewProofOfSpace` and `RespondSignatures` protocol messages. The farmer's only response to an invalid or absent fee threshold is to emit a log warning — it never rejects the override. As a result, any connected harvester can unconditionally redirect 100% of the farmer's block rewards (the 1/8 XCH farmer coinbase) to an arbitrary puzzle hash, with no enforcement or recourse.

### Finding Description

CHIP-22 introduced a fee convention allowing third-party harvesters to take the farmer's block reward as payment. The harvester signals this by setting `farmer_reward_address_override` (a `bytes32` puzzle hash) in the `NewProofOfSpace` and `RespondSignatures` protocol messages.

**Step 1 — Harvester sends `NewProofOfSpace` with override:**

In `farmer_api.py`, when the farmer receives a `NewProofOfSpace` with a non-`None` `farmer_reward_address_override`, it calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

**Step 2 — "Validation" is logging-only, never blocks:**

`notify_farmer_reward_taken_by_harvester_as_fee()` computes a `fee_quality` from the proof and challenge, compares it to the harvester-supplied `fee_threshold`, and emits a warning if the threshold is violated — but **always returns normally** regardless of the outcome: [2](#0-1) 

There is no `return`, `raise`, or any mechanism to abort the block-making flow.

**Step 3 — Farmer unconditionally uses the override address in `DeclareProofOfSpace`:**

In `_process_respond_signatures()`, when the harvester's `RespondSignatures` contains a non-`None` `farmer_reward_address_override`, the farmer replaces its own configured `farmer_target` with the harvester-supplied address: [3](#0-2) 

This overridden address is then passed directly into `DeclareProofOfSpace` as `farmer_reward_address`: [4](#0-3) 

The `farmer_reward_address_override` field is defined in both protocol message types: [5](#0-4) [6](#0-5) 

### Impact Explanation

The farmer's block reward (1/8 of the block reward, currently ~0.25 XCH per block) is determined by `farmer_reward_puzzle_hash` in the block's foliage. A malicious harvester can set `farmer_reward_address_override` to any arbitrary puzzle hash — including the attacker's own address — for every block it helps win. The farmer has no enforcement mechanism to prevent this; the fee threshold check is advisory only. This constitutes unauthorized XCH reward diversion affecting the farmer's coinbase coins.

### Likelihood Explanation

Any harvester connected to a farmer can exploit this. The CHIP-22 feature was explicitly designed to allow third-party harvesters (i.e., harvesters not controlled by the farmer), making this attack surface reachable by an unprivileged network peer. The attacker needs only to operate a harvester service and have the farmer connect to it — a normal operational scenario for farmers using remote or commercial harvesting services.

### Recommendation

The farmer should enforce the fee threshold check rather than merely logging violations. Specifically, in `_process_respond_signatures()`, before accepting `response.farmer_reward_address_override`, the farmer should:

1. Verify that `fee_info` is present in the corresponding `NewProofOfSpace`.
2. Compute `fee_quality = calculate_harvester_fee_quality(proof, challenge_hash)`.
3. Reject (return `None`) if `fee_quality > fee_info.applied_fee_threshold`, i.e., the fee threshold was not legitimately passed.
4. Optionally, cross-check that the override address in `RespondSignatures` matches the one declared in `NewProofOfSpace`. [7](#0-6) 

### Proof of Concept

1. Attacker operates a harvester service and has a farmer connect to it.
2. When the harvester finds a valid proof of space, it sends `NewProofOfSpace` with `farmer_reward_address_override = attacker_puzzle_hash` and `fee_info = None` (or any `fee_info`).
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`, which logs a warning (`"Harvester illegitimately took reward by failing to provide its fee rate"`) but does not abort.
4. The farmer requests signatures from the harvester.
5. The harvester returns `RespondSignatures` with `farmer_reward_address_override = attacker_puzzle_hash`.
6. `_process_respond_signatures()` sets `farmer_reward_address = attacker_puzzle_hash` with no further check.
7. `DeclareProofOfSpace` is sent to the full node with `farmer_reward_address = attacker_puzzle_hash`.
8. The block is accepted by the network; the farmer's 1/8 coinbase reward is paid to the attacker's address.

def calculate_harvester_fee_quality(proof: bytes, challenge: bytes32) -> uint32:
    """
    This calculates the 'fee quality' given a convention between farmers and third party harvesters.
    See CHIP-22: https://github.com/Chia-Network/chips/pull/88
    """
    return uint32(int.from_bytes(std_hash(proof + challenge)[32 - 4 :], byteorder="big", signed=False))
