### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Enables Queue-Exhaustion Denial of Service Against Legitimate Bridge Users - (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary

`verify_foreign_transaction` uses a request key (`VerifyForeignTransactionRequest`) that contains no caller identity. Any unprivileged account can pre-fill the 128-slot fan-out queue for any specific foreign-chain transaction, permanently blocking a legitimate user from submitting that same request. This is the direct analog of the Connext `TransactionManager.prepare` front-running bug: the request key lacks an initiator/caller binding, so an adversary can occupy all queue slots before the victim's call lands.

### Finding Description

The `sign` and `request_app_private_key` methods both bind the pending-request map key to the caller's identity. `SignatureRequest::new` derives a `tweak` from `(predecessor_id, path)`, making each caller's key unique:

```rust
// crates/contract/src/lib.rs:379-384
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // ŌåÉ caller identity baked in
    &request.path,
);
```

`verify_foreign_transaction` does not do this. The stored key is:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    // ŌåÉ no caller field
}
```

The conversion from args to request discards the predecessor entirely:

```rust
// crates/contract/src/dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

The fan-out queue enforces a hard cap of 128 entries per key:

```rust
// crates/contract/src/pending_requests.rs:37,51-57
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
// ...
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
    );
}
```

The codebase itself acknowledges this property in a test comment:

```rust
// crates/contract/src/lib.rs:3242-3243
// And: caller bob submits the identical request ŌĆö a different account would today
// be blocked from receiving a response by alice's submission.
```

**Attack path:**

1. Attacker observes (via mempool or block scanning) that a user intends to call `verify_foreign_transaction` for a specific `(tx_id, chain, extractors, finality, domain_id, payload_version)`.
2. Attacker submits 128 identical `verify_foreign_transaction` calls with those exact parameters, each attaching the minimum 1 yoctoNEAR deposit.
3. The queue for that `VerifyForeignTransactionRequest` key is now full.
4. The victim's call panics with `PendingRequestQueueFull` and is rejected.
5. When the queue drains (via MPC response or per-yield timeout), the attacker refills it, indefinitely blocking the victim.

### Impact Explanation

A legitimate user who needs a specific foreign-chain transaction verified and signed by the MPC network (e.g., to complete a bridge withdrawal or cross-chain settlement) can be permanently denied service for that specific transaction. The attacker controls the queue for any `(tx_id, chain, domain_id, payload_version)` tuple at a cost of 128 ├Ś 1 yoctoNEAR + gas per blocking round. If the user's downstream protocol depends on receiving the MPC signature over that specific transaction (e.g., to release locked funds), the inability to obtain it constitutes a request-lifecycle invariant break: a valid, well-formed request from a legitimate user is systematically rejected by the contract.

This maps to the **Medium** allowed impact: *"request-lifecycle... manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

### Likelihood Explanation

**Medium.** The attacker needs only to know the foreign-chain transaction parameters (publicly visible on the foreign chain and in the NEAR mempool) and submit 128 cheap transactions before the victim. The minimum deposit is 1 yoctoNEAR per call; the total cost per blocking round is negligible. The attack is repeatable: each time the queue drains (either via MPC response or NEAR yield timeout), the attacker can refill it. No privileged access, key material, or threshold collusion is required.

### Recommendation

Bind the `VerifyForeignTransactionRequest` map key to the caller's identity, mirroring the pattern used by `sign` and `request_app_private_key`. The simplest fix is to add a `caller: AccountId` field to `VerifyForeignTransactionRequest` (analogous to the `initiator` field recommended in the Connext report) and populate it from `env::predecessor_account_id()` inside `verify_foreign_transaction`, before the request is stored. This ensures each caller occupies a distinct queue slot namespace, eliminating the cross-caller queue exhaustion vector while preserving the fan-out feature for the same caller submitting duplicate requests.

### Proof of Concept

The existing unit test at `crates/contract/src/lib.rs:3208-3298` already demonstrates the precondition: alice and bob submitting identical `VerifyForeignTransactionRequestArgs` share a single queue entry. Extending that test to fill the queue to `MAX_PENDING_REQUEST_FAN_OUT` (128) from a single attacker account before the victim submits would produce a `PendingRequestQueueFull` panic on the victim's call, confirming the denial-of-service.

```

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
