### Title
`respond` Participant Whitelist Excludes Old Signers During Resharing, Permanently Freezing Pending Signature Requests — (`crates/contract/src/lib.rs`)

### Summary

The `respond` (and `respond_verify_foreign_tx`) entry points enforce that the caller must be an **attested participant in the currently active set**. During a `Resharing` state, `active_participants()` returns exclusively the **new (proposed) participants**. Old participants who are being removed from the network hold the only valid key shares for pending requests but are rejected by this check, causing those requests to time out permanently — an exact structural analog to the `TaxTokenReceipt::transferFrom` whitelist that omits the `Auction` contract.

---

### Finding Description

`respond` calls `assert_caller_is_attested_participant_and_protocol_active`, which resolves the allowed caller set via `protocol_state.active_participants()`:

```rust
// crates/contract/src/lib.rs:563-573
pub fn respond(...) -> Result<(), Error> {
    let signer = Self::assert_caller_is_signer();
    self.assert_caller_is_attested_participant_and_protocol_active(); // panics if not in active set
    if !self.protocol_state.is_running_or_resharing() { ... }
    ...
}
```

`active_participants()` is defined in `crates/contract/src/state.rs`:

```rust
// crates/contract/src/state.rs:255-269
pub fn active_participants(&self) -> &Participants {
    match self {
        ProtocolContractState::Running(state) => state.parameters.participants(),
        ProtocolContractState::Resharing(state) => {
            state.resharing_key.proposed_parameters().participants() // NEW participants only
        }
        ...
    }
}
```

During `Resharing`, this returns only the **proposed (new) participants**. `assert_caller_is_attested_participant_and_protocol_active` then calls `is_caller_an_attested_participant(participants)`, which immediately returns `CallerNotParticipant` for any old participant not in the new set:

```rust
// crates/contract/src/tee/tee_state.rs:476-478
let info = participants
    .info(&signer_id)
    .ok_or(AttestationCheckError::CallerNotParticipant)?;
```

The `assert_matches!` macro then **panics** the transaction.

**Attack / trigger path:**

1. System is `Running` with participants `[A, B, C]`, threshold 2.
2. User submits `sign(payload)` → pending yield created, deposit paid.
3. Governance votes to reshare to `[B, C, D]` (removing A). Contract enters `Resharing`.
4. Old participants `[A, B, C]` run the off-chain signing protocol using the old key shares. The leader (e.g., A) aggregates the signature.
5. A calls `respond` on-chain → `active_participants()` returns `[B, C, D]` → A is not in the set → transaction panics.
6. B or C could call `respond`, but in the OT-based ECDSA protocol only the leader holds the aggregated signature; other nodes hold only their own shares.
7. New participant D has no key shares yet (they are being generated during resharing).
8. The pending yield times out → `fail_on_timeout` fires → user's transaction fails permanently.

The same flaw applies to `respond_verify_foreign_tx` at line 692, which has an identical guard.

---

### Impact Explanation

Pending `sign` and `verify_foreign_transaction` requests submitted before a resharing that removes participants are permanently unresolvable. The yield-resume callback fires `fail_on_timeout`, the user's transaction fails, and any downstream action (e.g., unlocking funds on a foreign chain via a bridge) cannot proceed. Funds locked behind the undelivered signature are permanently frozen. This matches the **Medium** allowed impact: *"request-lifecycle or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

Any production resharing that removes at least one participant while there are in-flight `sign` requests triggers this. Resharing is a routine governance operation (node rotation, threshold changes, TEE upgrades). The existing integration test `test_signature_requests_in_resharing_are_processed` only covers the **add-only** case (no participant removal), so the removal case is untested and the bug is latent in production.

---

### Recommendation

`assert_caller_is_attested_participant_and_protocol_active` should, during `Resharing`, accept callers who are attested participants in **either** the previous running set **or** the new proposed set. Concretely, replace the single `active_participants()` lookup with a union check analogous to `is_existing_or_prospective_participant` (already defined in `crates/contract/src/state.rs:272-294`) but applied to the TEE attestation check. Alternatively, introduce a dedicated `respond_participants()` helper that returns the old set during resharing (since signing still uses the old key) and the new set during `Running`.

---

### Proof of Concept

**State:** `Running([A,B,C], t=2)` → reshare to `([B,C,D], t=2)` → `Resharing`.

1. User calls `sign(payload)` just before resharing is voted in. Pending yield is created.
2. Resharing is voted in. `active_participants()` now returns `[B,C,D]`.
3. Node A (leader, being removed) computes the signature off-chain and calls `respond(request, sig)`.
4. Contract executes `assert_caller_is_attested_participant_and_protocol_active()`:
   - `active_participants()` → `[B,C,D]`
   - `is_caller_an_attested_participant([B,C,D])` → `Err(CallerNotParticipant)` for A
   - `assert_matches!` → **panic**
5. No other node can submit the response (B/C lack the aggregated signature; D has no key share).
6. Yield times out → `fail_on_timeout` → user's `sign` call fails permanently.

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
