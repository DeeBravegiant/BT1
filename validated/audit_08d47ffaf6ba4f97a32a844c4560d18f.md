Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch comparison, checking only `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. This allows any depositor to push ETH deposits arbitrarily beyond the configured cap in a single transaction, while the equivalent ERC20 path correctly rejects such deposits.

## Finding Description
At `contracts/LRTDepositPool.sol` L676–682, the function has two branches:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
```

The ETH branch evaluates the pre-deposit total only. If `totalAssetDeposits` is below `depositLimitByAsset`, the check returns `false` regardless of how large `amount` is, so `_beforeDeposit` (L661–663) never reverts with `MaximumDepositLimitReached`. The call chain is: `depositETH` (L76–93, public payable, no role required) → `_beforeDeposit` (L648–670) → `_checkIfDepositAmountExceedesCurrentLimit`. No other guard in this path accounts for the incoming ETH amount against the cap.

## Impact Explanation
The deposit cap is the protocol's primary mechanism for bounding ETH TVL. Because the check is evaluated against the pre-deposit total, a single unprivileged call can mint rsETH for an amount that far exceeds the configured cap. No funds are stolen and no funds are frozen, but the protocol accepts and mints rsETH for ETH beyond the limit it was configured to enforce. This maps to **Low: Contract fails to deliver promised returns** — the deposit cap guarantee is not upheld.

## Likelihood Explanation
`depositETH` is fully permissionless and payable. The only precondition is that `totalAssetDeposits` is below (not already above) the limit, which is the normal operating state of the protocol. No special role, timing window, or external dependency is required. The condition is trivially reachable and the exploit is repeatable in any block.

## Recommendation
Include `amount` in the ETH branch to match the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Normal deposits bring `getTotalAssetDeposits(ETH_TOKEN)` to `999.9 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit`: `999.9 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 500 ETH; `getTotalAssetDeposits(ETH_TOKEN)` becomes `1499.9 ether`.
6. The same call with an ERC20 would have reverted: `999.9 + 500 > 1000` → `true` → `MaximumDepositLimitReached`.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock `lrtConfig` returning `depositLimitByAsset = 1000 ether`. Seed `totalAssetDeposits` to `999.9 ether` via a prior deposit. Call `depositETH{value: 500 ether}`. Assert the call succeeds and `getTotalAssetDeposits` exceeds `1000 ether`. Then assert the equivalent ERC20 path reverts. [1](#0-0) [2](#0-1) [3](#0-2)
