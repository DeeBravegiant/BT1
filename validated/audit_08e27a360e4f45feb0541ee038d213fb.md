### Title
Deposits Paid for Non-Winning Update Proposals Are Permanently Locked in Contract When `do_update` Clears All Entries — (File: `crates/contract/src/update.rs`)

---

### Summary

When `vote_update` reaches threshold and calls `do_update`, it clears **all** pending update entries — including proposals from other participants — without refunding the NEAR token deposits those proposers paid. Because `UpdateEntry` does not store the proposer's account ID or the deposit amount, the contract has no way to issue refunds, and those tokens are permanently locked.

---

### Finding Description

`propose_update` requires a deposit proportional to the serialized size of the proposed update: [1](#0-0) 

The exact required amount is computed from `bytes_used`, which includes the code/config payload plus an overhead for 128 participant-vote slots: [2](#0-1) 

Only the **excess** above `required` is refunded to the proposer at call time. The `required` portion stays in the contract. `UpdateEntry` — the struct persisted in storage — records only the update payload and `bytes_used`; it does not record the proposer's account ID or the deposit amount: [3](#0-2) 

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()` on **all remaining entries**: [4](#0-3) 

No refund is issued for the cleared entries. Because the proposer's identity and deposit amount were never stored in `UpdateEntry`, the contract cannot reconstruct who to pay back or how much. There is no `remove_update` or `cancel_update` endpoint that would let a proposer withdraw their proposal before a competing one wins: [5](#0-4) 

The parallel to the Kinetiq bug is exact: in Kinetiq, `hypeBuffer` is decremented when a withdrawal is queued and never restored on cancellation; here, the deposit is deducted from the proposer's account when a proposal is submitted and never returned when a competing proposal wins and clears all entries.

---

### Impact Explanation

NEAR tokens paid as storage deposits for non-winning update proposals are permanently locked in the MPC contract. For a large contract binary (e.g., a 1 MB WASM), `bytes_used` is dominated by the code length, making `required_deposit` potentially hundreds of NEAR tokens. Every governance cycle in which two or more participants propose competing updates and one wins silently destroys the losing proposers' deposits with no recovery path.

This breaks the production accounting invariant: *deposits paid to the contract must either be consumed for their stated purpose or returned to the depositor*.

---

### Likelihood Explanation

The scenario is realistic in any multi-participant governance round: participant A proposes update X, participant B proposes update Y, threshold votes accumulate for Y, `do_update` fires, and participant A's deposit is gone. No adversarial coordination is required — it is a natural consequence of the normal governance flow. The only precondition is that more than one update proposal exists simultaneously, which is expected in a live network.

---

### Recommendation

Store the proposer's account ID and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over the remaining entries and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the refund pattern already used in `propose_update` for excess deposits.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM binary, attaching the required deposit (~50 NEAR).
2. Participant B calls `propose_update` with a different WASM binary, attaching ~50 NEAR.
3. Threshold participants call `vote_update` for B's proposal ID.
4. `vote_update` calls `do_update(&B_id, gas)`.
5. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry is deleted with no refund.
6. Participant A's ~50 NEAR is permanently locked in the contract. There is no `remove_update` endpoint to recover it. [6](#0-5) [7](#0-6)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
