### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect Block Fees to Arbitrary Address — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates many fields of the network-supplied `ProposalInit` (height, L1/L2 gas prices, timestamp, starknet version, version constant commitment, fee proposal) but never validates the `builder` field. That field is passed verbatim as `sequencer_address` into `BlockInfo` for execution. Because the sequencer address is where transaction fees are credited, a malicious proposer can set `builder` to any address — including their own — and redirect all block fees to that address. Validators accept the proposal because no guard on `builder` exists.

---

### Finding Description

`is_proposal_init_valid` in `validate_proposal.rs` enforces the following fields of `ProposalInit`: [1](#0-0) 

```rust
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
{
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
```

It also validates `timestamp`, `starknet_version`, `version_constant_commitment`, all four L1 gas price fields (within margin), and `fee_proposal_fri`. The `ProposalInitValidation` struct that carries the locally-trusted reference values contains no `expected_builder_address` field: [2](#0-1) 

The `builder` field is therefore never compared against any locally-derived expectation.

After `is_proposal_init_valid` passes, `initiate_validation` calls `convert_to_sn_api_block_info(init)`, which maps `init.builder` directly to `sequencer_address` in the `BlockInfo` handed to the batcher: [3](#0-2) 

```rust
sequencer_address: init.builder,
```

On the proposer side, `builder` is always set to the node's own registered address: [4](#0-3) 

```rust
let init = ProposalInit {
    ...
    builder: args.builder_address,   // node's own address
    ...
};
```

No equivalent enforcement exists on the validator side.

The `ProposalFin` commitment check (`built_block != received_fin.proposal_commitment`) does not catch this because both proposer and validator derive the block hash from the same proposer-supplied `builder` value; they agree on the hash regardless of what address was placed there: [5](#0-4) 

---

### Impact Explanation

In Starknet execution, `sequencer_address` is the account that receives transaction fees. By setting `builder` to an attacker-controlled address, a malicious proposer causes every transaction fee in the block to be credited to that address instead of the legitimate sequencer. The committed block header also records the wrong sequencer address, so the on-chain state permanently reflects the fee redirection. This is a direct, quantifiable economic loss matching the "Critical — Incorrect fee … with economic impact" category.

Additionally, any contract that calls `get_sequencer_address()` during that block receives the attacker-supplied address, which can corrupt application-level logic that depends on the sequencer identity.

---

### Likelihood Explanation

Any committee member who wins the proposer slot for a round can trigger this with a single modified `ProposalInit`. No special tooling is required beyond the ability to send a crafted proposal — the normal consensus flow delivers it to all validators, who accept it without checking `builder`.

---

### Recommendation

Add `expected_builder_address: ContractAddress` to `ProposalInitValidation` and enforce it inside `is_proposal_init_valid`:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder_address {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.expected_builder_address,
            init_proposed.builder
        ),
    ));
}
```

The `expected_builder_address` should be populated in `validate_current_round_proposal` from the node's own registered builder address, mirroring how `l2_gas_price_fri` and `l1_da_mode` are sourced from local state. [6](#0-5) 

---

### Proof of Concept

1. Attacker is a committee member and wins the proposer slot for height H, round R.
2. Attacker constructs `ProposalInit` with all legitimate fields **except** `builder`, which is set to `attacker_address`.
3. Attacker streams the proposal to all validators.
4. Each validator calls `validate_proposal` → `is_proposal_init_valid`. All checked fields pass; `builder` is never compared to any reference value.
5. `initiate_validation` calls `convert_to_sn_api_block_info(init)`, producing `BlockInfo { sequencer_address: attacker_address, ... }`.
6. The batcher executes all transactions with `sequencer_address = attacker_address`; every fee is credited to the attacker.
7. Both proposer and validator compute `partial_block_hash` from the same inputs (including `attacker_address` as sequencer). The `ProposalFinMismatch` check passes.
8. Consensus commits the block. The on-chain state records `attacker_address` as the sequencer for block H, and all fees are permanently redirected.
