Audit Report

## Title
ETH Deposit Limit Bypass via Missing `amount` in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the `amount` parameter in its ETH branch, checking only whether total deposits have already exceeded the limit rather than whether the incoming deposit would exceed it. As a result, any ETH deposit succeeds as long as the pool has not already breached its cap, regardless of how large the deposit is. The ETH deposit limit — a core protocol invariant — is completely unenforced.

## Finding Description
`depositETH` passes `msg.value` as `depositAmount` to `_beforeDeposit` (line 87), which calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` (line 661) and reverts on `true`.

Inside `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682):

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC20
```

The ETH branch evaluates `totalAssetDeposits > limit` — a strict greater-than that is `false` when the pool is at exactly the limit and `false` when the pool is below the limit regardless of `amount`. The incoming deposit size is never factored in. The ERC20 path correctly uses `totalAssetDeposits + amount > limit`.

`getAssetCurrentLimit` (lines 402–409) computes remaining capacity uniformly for all assets using `limit - totalAssetDeposits`, so it correctly advertises a non-zero remaining capacity, but the enforcement gate ignores that capacity for ETH entirely.

Exploit path: pool has `totalAssetDeposits = limit - 1 wei` → `getAssetCurrentLimit` returns `1 wei` → attacker calls `depositETH{value: 1000 ether}()` → guard evaluates `(limit - 1) > limit` → `false` → no revert → 1000 ETH accepted and rsETH minted far beyond the cap.

Even when `totalAssetDeposits == limit` exactly, the check returns `false` (equal is not greater-than), so one more deposit still passes.

## Impact Explanation
The ETH deposit cap is a stated protocol invariant. Any depositor can exceed it by an arbitrary amount with a single transaction. No funds are stolen and no yield is diverted, but the contract fails to enforce its own deposit limit for native ETH. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
No special role, no front-running, no oracle dependency. Any unprivileged user calling `depositETH` when the pool is near or at its ETH limit triggers the bypass. The path is entirely public and requires only a standard ETH transfer. The condition is met whenever the pool approaches its configured ETH cap, which is a normal operational state.

## Recommendation
Mirror the ERC20 logic in the ETH branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This makes the ETH check consistent with both the ERC20 path and the semantics of `getAssetCurrentLimit`.

## Proof of Concept

```solidity
function test_ETHLimitBypass() public {
    uint256 limit = 10 ether;
    // Setup: ETH supported asset, depositLimitByAsset = 10 ether,
    // totalAssetDeposits = 10 ether - 1 wei (1 wei below limit)

    uint256 currentLimit = depositPool.getAssetCurrentLimit(LRTConstants.ETH_TOKEN);
    assertEq(currentLimit, 1); // 1 wei remaining per getAssetCurrentLimit

    // Deposit 5 ETH — should revert MaximumDepositLimitReached, but does not
    vm.deal(alice, 5 ether);
    vm.prank(alice);
    depositPool.depositETH{value: 5 ether}(0, "");

    // Total now far exceeds limit; no revert occurred
    uint256 total = depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN);
    assertGt(total, limit);
}
```

The test demonstrates that `_checkIfDepositAmountExceedesCurrentLimit` returns `false` for the 5 ETH deposit (because `totalAssetDeposits > limit` is false), allowing the deposit to proceed and push total ETH deposits well beyond the configured cap.
