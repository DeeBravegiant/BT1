### Title
Single Byzantine Participant Can Permanently Block Removal of Compromised Launcher Image Hash - (File: `crates/contract/src/lib.rs`)

---

### Summary

`vote_remove_launcher_hash` (and `vote_remove_os_measurement`) require **unanimity** — every participant must vote — to remove a whitelisted launcher hash or OS measurement. A single Byzantine participant strictly below the signing threshold can permanently block removal of a compromised launcher, keeping nodes running the vulnerable image whitelisted and eligible to participate in threshold signing indefinitely.

---

### Finding Description

`vote_add_launcher_hash` requires only a **threshold** of votes to whitelist a new launcher image hash: [1](#0-0) 

But `vote_remove_launcher_hash` requires **all** participants to vote: [2](#0-1) 

This asymmetry — threshold to add, unanimity to remove — means a single uncooperative participant (offline, compromised, or malicious) can block removal indefinitely. The same unanimity requirement applies to `vote_remove_os_measurement`: [3](#0-2) 

Furthermore, the last remaining launcher hash in the whitelist **can never be removed under any circumstances**, even with full unanimity: [4](#0-3) 

The `TeeState` enforces this at the storage layer: [5](#0-4) 

For MPC Docker image hashes (`vote_code_hash`), there is **no removal function at all** — only time-based expiry after a newer hash is voted in, and the most recently added hash never expires: [6](#0-5) [7](#0-6) 

The operator guide explicitly acknowledges this gap: *"There is no `vote_remove_code_hash`. Once a successor hash is voted in, the previous hash remains valid for a 7-day grace period and then auto-expires."*

---

### Impact Explanation

The launcher binary is the root of the TEE trust chain: it verifies the MPC node image hash, extends RTMR3, and starts the MPC node container. A compromised launcher can bypass image-hash verification and start a malicious MPC node container. If a vulnerability is discovered in a whitelisted launcher hash and a single participant blocks its removal:

1. Nodes running the compromised launcher continue to pass `submit_participant_info` attestation verification.
2. Those nodes remain in `stored_attestations` as valid TEE participants.
3. They can participate in threshold signing, potentially enabling unauthorized signature issuance or key-share exposure.

**Impact: Medium** — participant-state manipulation that breaks the production safety invariant that only nodes running verified, trusted code may participate in signing. Escalates to **Critical** if the compromised launcher enables key-share extraction or unauthorized signing.

---

### Likelihood Explanation

Only **one** participant needs to be Byzantine — offline, compromised, or deliberately uncooperative — to block removal. This is a realistic condition:

- A participant node could be offline during a security incident response window.
- A participant whose infrastructure is compromised may be unable or unwilling to vote.
- A malicious participant can intentionally refuse to vote to protect a launcher they control.

The attacker is a single Byzantine participant strictly below the signing threshold, which is an explicitly in-scope attacker model.

---

### Recommendation

Change `vote_remove_launcher_hash` and `vote_remove_os_measurement` to require **threshold** votes (not unanimity) for removal, mirroring the add path. The rationale for unanimity — that removal invalidates attestations of running nodes — is a liveness concern, not a security one; threshold-based removal still requires broad consensus while eliminating the single-participant veto. The HOT TEE governance contract design already notes this pattern for `vote_remove_code_hash`: [8](#0-7) 

Concretely, replace:

```rust
let total_participants = threshold_parameters.participants().len() as u64;
if votes >= total_participants {
```

with:

```rust
if votes >= self.threshold()?.value() {
```

in both `vote_remove_launcher_hash` and `vote_remove_os_measurement`. The "last hash cannot be removed" guard can remain as a liveness safeguard.

---

### Proof of Concept

1. Network: 5 participants, signing threshold 3.
2. Launcher hash `A` is voted in via `vote_add_launcher_hash` (3 votes — threshold reached). [9](#0-8) 
3. A critical vulnerability is discovered in launcher `A` (e.g., it can be tricked into starting an arbitrary container).
4. Network votes in new launcher hash `B` (3 votes — threshold reached). Hash `A` is still whitelisted.
5. Network initiates removal of `A` via `vote_remove_launcher_hash`. Requires all 5 votes.
6. Participant 5 is offline (or refuses). Only 4 votes are cast. Removal never triggers. [2](#0-1) 
7. Hash `A` remains in `allowed_launcher_images` indefinitely.
8. Nodes running launcher `A` continue to pass `submit_participant_info` attestation checks and remain in `stored_attestations` as valid TEE participants. [10](#0-9) 
9. Those nodes participate in threshold signing, with the compromised launcher able to bypass image-hash verification for the MPC node container they start.

    #[handle_result]
    pub fn vote_add_launcher_hash(
        &mut self,
        launcher_hash: LauncherImageHash,
    ) -> Result<(), Error> {
        log!(
            "vote_add_launcher_hash: signer={}, launcher_hash={:?}",
            env::signer_account_id(),
            launcher_hash,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let action = LauncherVoteAction::Add(launcher_hash);
        let votes = self.tee_state.vote_launcher(action, &participant);

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        if votes >= self.threshold()?.value() {
            let added = self
                .tee_state
                .add_launcher_image(launcher_hash, tee_upgrade_deadline_duration);
            log!("launcher hash add result: {}", added);
        }

        Ok(())
    }
