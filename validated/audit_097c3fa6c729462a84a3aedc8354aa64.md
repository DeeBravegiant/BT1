### Title
Dijkstra Batch Transaction Pool Re-Registration Causes Incorrect Deposit Double-Charge — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

### Summary

In the Dijkstra era's batch transaction model, the top-level transaction's UTXOW/UTXO rule is explicitly passed the **original** `CertState` (before any sub-transactions ran). When a sub-transaction registers a new stake pool P, and the top-level transaction contains a `RegPoolTxCert` for the same pool P (a legitimate parameter update), the top-level UTXO rule's deposit calculation uses the original `CertState` where P is not yet registered, incorrectly charging a full pool deposit. The ENTITIES/POOL rule, however, runs with the post-sub-transaction `CertState` and correctly treats the top-level cert as an update (no deposit). This mismatch causes either a `ValueNotConservedUTxO` rejection of a legitimate batch, or — if the tx author includes the extra deposit to satisfy value conservation — a permanent loss of funds with `utxosDeposited` permanently overstated.

### Finding Description

**Root cause — original `CertState` passed to top-level UTXOW:**

In `dijkstraLedgerTransition`, sub-transactions are processed first via `SUBLEDGERS`, producing `certStateAfterSubLedgers`. The ENTITIES rule for the top-level tx correctly uses this updated state. However, UTXOW is explicitly passed `lsCertState ledgerState` — the **original** cert state: [1](#0-0) [2](#0-1) 

**Deposit calculation uses original `CertState`:**

`conwayCertsTotalDepositsTxBody` (used by Dijkstra via delegation) checks `psStakePools` of the cert state it receives: [3](#0-2) 

`shelleyTotalDepositsTxCerts` only skips the deposit if `isRegPoolRegistered sppId` returns `True`: [4](#0-3) 

Since the original `CertState` does not contain pool P (it was registered by a sub-transaction), `isRegPoolRegistered` returns `False`, and a full `ppPoolDepositL` is charged.

**`updateUTxOStateNoFees` increases `utxosDeposited` using the same stale `CertState`:** [5](#0-4) 

**ENTITIES runs with the updated `CertState` and treats the cert as an update:** [6](#0-5) 

The POOL rule sees pool P already in `psStakePools` (from the sub-transaction) and does not add a deposit to `psDeposits`.

**The same stale-certState pattern applies inside each sub-ledger:** [7](#0-6) 

### Impact Explanation

Two concrete outcomes for a batch `[sub-tx1: RegPoolTxCert P, top-tx: RegPoolTxCert P]`:

1. **Tx rejected (false negative):** If the tx author does not include the extra deposit `D` in the top-level tx's inputs, `validateValueNotConservedUTxO` fails with `ValueNotConservedUTxO`. A legitimate batch is permanently unsubmittable.

2. **Permanent loss of funds:** If the tx author includes `D` to satisfy value conservation, the tx is accepted. `utxosDeposited` is increased by `D` twice (once by sub-tx1's SUBUTXO, once by the top-level UTXO), but `psDeposits[P]` only records `D` once. The extra `D` is permanently stuck in the deposit pot and can never be refunded, since pool retirement refunds are based on `psDeposits`, not `utxosDeposited`. This violates the deposit-equals-obligation invariant asserted in `shelleyLedgerAssertions`. [8](#0-7) 

### Likelihood Explanation

The scenario requires a batch where a sub-transaction registers a new pool and the top-level transaction updates the same pool's parameters in the same batch. This is a natural and intended use case of the Dijkstra batch model (e.g., register pool and immediately update its metadata or pledge). Any pool operator using this pattern will encounter the bug. The attacker is the tx author themselves; no external party needs to be involved.

### Recommendation

The top-level UTXOW environment should receive `certStateAfterSubLedgers` (the cert state after all sub-transactions have been processed) rather than the original `lsCertState ledgerState`. This mirrors how ENTITIES already uses `certStateAfterSubLedgers`. Alternatively, `certsTotalDepositsTxBody` for the top-level tx should be computed using the post-sub-ledger cert state so that pools registered by sub-transactions are correctly recognized as already-registered when computing the top-level deposit obligation.

### Proof of Concept

```
Initial state: pool P not in psStakePools

Batch:
  sub-tx1: [RegPoolTxCert P]   -- new registration, deposit D required
  top-tx:  [RegPoolTxCert P]   -- intended as parameter update, deposit should be 0

Execution:
  1. SUBLEDGERS processes sub-tx1:
     - SUBENTITIES: adds P to psStakePools, psDeposits[P] = D
     - SUBUTXO (certState = original, P not registered):
         updateUTxOStateNoFees charges D → utxosDeposited += D  ✓

  2. ENTITIES processes top-tx (certState = certStateAfterSubLedgers, P registered):
     - POOL rule: P already in psStakePools → update only, psDeposits unchanged  ✓

  3. UTXOW/UTXO processes top-tx (certState = ORIGINAL, P not registered):
     - validateValueNotConservedUTxO: certsTotalDepositsTxBody charges D
       → tx rejected unless author includes extra D in inputs
     - If author includes D: updateUTxOStateNoFees → utxosDeposited += D again

Final state (if tx accepted):
  utxosDeposited increased by 2D total
  psDeposits[P] = D  (only one deposit recorded)
  Invariant violated: utxosDeposited ≠ sum(psDeposits) + sum(stakeDeposits) + ...
  Extra D permanently unrecoverable.
```

shelleyTotalDepositsTxCerts pp isRegPoolRegistered certs =
  numKeys
    <×> (pp ^. ppKeyDepositL)
    <+> numNewRegPoolCerts
    <×> (pp ^. ppPoolDepositL)

      , utxosGovState = govState
