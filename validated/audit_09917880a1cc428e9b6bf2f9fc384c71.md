### Title
Inconsistent Zero-Payload Handling Between Contract Validation and Node-Level Signing Across Protocols — (File: `crates/contract/src/lib.rs`, `crates/threshold-signatures/src/ecdsa/robust_ecdsa/sign.rs`, `crates/threshold-signatures/src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The contract's `sign` function accepts a zero ECDSA payload (32 zero bytes) for **both** `CaitSith` and `DamgardEtAl` domains. However, the node-level signing for `DamgardEtAl` (`robust_ecdsa::sign`) explicitly rejects `msg_hash == 0`, while the `CaitSith` path (`ot_based_ecdsa::sign`) has no such guard. This creates two distinct inconsistencies that mirror the external report's vulnerability class: one code path (DamgardEtAl) silently accepts a request the nodes will always reject, and the other (CaitSith) produces a valid threshold signature for the zero scalar without restriction.

---

### Finding Description

**Code path 1 — DamgardEtAl (inconsistency: contract accepts, nodes reject)**

The contract's `sign` function validates the ECDSA payload for both `CaitSith` and `DamgardEtAl` by converting the raw bytes to a `k256::Scalar`:

```rust
Protocol::CaitSith | Protocol::DamgardEtAl => {
    let hash = *request.payload.as_ecdsa().expect("Payload is not Ecdsa");
    k256::Scalar::from_repr(hash.into())
        .into_option()
        .expect("Ecdsa payload cannot be converted to Scalar");
}
``` [1](#0-0) 

`k256::Scalar::from_repr([0u8; 32])` succeeds — the zero scalar is a valid field element — so a zero payload passes this check. The contract's own comment states: *"It's important we fail here because the MPC nodes will fail in an identical way."*

However, the node-level `robust_ecdsa::sign` function for `DamgardEtAl` explicitly rejects zero:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
``` [2](#0-1) 

This check fires at initialization, before any presignature is consumed. The result: the contract accepts the request (deposit taken, yield slot allocated), the nodes reject it at signing time, the request times out, and the user receives a timeout error rather than the expected validation error. The contract's stated safety invariant is broken.

**Code path 2 — CaitSith (zero payload is fully signed)**

The `ot_based_ecdsa::sign` function for `CaitSith` has **no zero-hash guard**:

```rust
pub fn sign<T>(
    participants: &[Participant],
    coordinator: Participant,
    threshold: T,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption> + use<T>, InitializationError>
``` [3](#0-2) 

With `msg_hash = 0`, the signature share computation reduces to:

```
s_i = 0 * k_i + r * sigma_i  =  r * sigma_i
``` [4](#0-3) 

The coordinator aggregates shares, normalizes `s`, and the final signature `(r, s)` with `s = r·k·x` passes the internal `sig.verify` check and is returned to the contract. The contract's `respond` path then resolves the yield and delivers the signature to the caller.
