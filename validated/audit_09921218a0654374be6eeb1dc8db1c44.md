Audit Report

## Title
Free Share Minting in Fully-Drained Bins Dilutes Existing LP Claims — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

When a bin's token balances have been reduced to zero by swaps but its `binTotalShares` remains positive, `LiquidityLib.addLiquidity` computes zero token amounts for any new deposit into that bin, skips the payment callback entirely, yet unconditionally commits the new shares and updated total to storage. An attacker can mint an arbitrary number of shares at zero cost, then redeem a proportional fraction of tokens that flow back into the bin when the price reverses, directly stealing from existing LPs.

## Finding Description

In `LiquidityLib.addLiquidity`, when `binTotalSharesVal > 0` the proportional branch executes:

```solidity
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
``` [1](#0-0) 

If both `token0BalanceScaled` and `token1BalanceScaled` are zero (a fully-drained bin), both `ceilDiv(0, binTotalSharesVal)` calls return zero. The subsequent balance updates are guarded by `if (amount0Scaled > 0)` / `if (amount1Scaled > 0)`, but the share accounting writes are unconditional:

```solidity
binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd;
positionBinShares[posKey] = newUserShares;
``` [2](#0-1) 

After the loop, the payment callback is only invoked when the aggregate owed amount is non-zero:

```solidity
if (amount0Added > 0 || amount1Added > 0) {
    IMetricOmmModifyLiquidityCallback(msg.sender)
        .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
    ...
}
``` [3](#0-2) 

Because both amounts are zero, the guard is false, no tokens are transferred, and the attacker's shares are committed to storage for free.

The drained-bin precondition is reachable through normal swap execution. The swap loops check `binState.token0BalanceScaled == 0` to skip empty bins but never touch `_binTotalShares`; shares are exclusively managed by `addLiquidity`/`removeLiquidity`. A bin above the current price that is fully consumed by `!zeroForOne` swaps will have `token0BalanceScaled = 0` while `_binTotalShares[binIdx]` remains at its pre-swap value. [4](#0-3) 

When the price later reverses and `zeroForOne` swaps refill the bin, `removeLiquidity` distributes tokens proportionally to shares:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [5](#0-4) 

The attacker's free shares dilute the denominator, reducing every legitimate LP's withdrawal to a fraction of what they deposited.

The only existing guard relevant to new deposits is the `minimalMintableLiquidity` check, which only enforces a minimum share count and does not prevent zero-cost minting: [6](#0-5) 

## Impact Explanation

This is a direct theft of LP principal. Existing LPs who deposited real tokens into a bin receive a smaller proportional withdrawal than they are owed once the attacker's free shares dilute `binTotalShares`. The loss scales with `attackerShares / (originalShares + attackerShares)` applied to all tokens that subsequently flow into the bin. The attacker can set `sharesToAdd` to any value, making the dilution arbitrarily large relative to the pre-existing share count. This constitutes pool insolvency: the pool's token balances no longer cover all legitimate LP claims.

## Likelihood Explanation

Bins being fully drained is a routine market event in any active pool. No privileged access is required: `addLiquidity` is open to any caller with any `owner` address. The attacker only needs to monitor on-chain `BinSwapped` events, detect when a bin's `token0BalanceScaled` (or `token1BalanceScaled`) reaches zero while `_binTotalShares` remains positive, and call `addLiquidity` before the price reverses. The attack is repeatable across any bin and any pool instance. [7](#0-6) 

## Recommendation

Add a guard in `LiquidityLib.addLiquidity` inside the `else` branch (proportional path) that reverts when `binTotalSharesVal > 0` and both computed amounts are zero:

```solidity
} else {
    amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
    amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
    if (amount0Scaled == 0 && amount1Scaled == 0) revert DrainedBin();
}
```

This enforces the invariant that shares in a non-empty bin must always represent a non-zero token claim, mirroring the behavior of the `binTotalSharesVal == 0` branch which always computes a positive amount.

## Proof of Concept

1. Deploy a pool with bins `[-1, 0, 1]`. Alice calls `addLiquidity` for bin `1` (above current price), depositing token0 and receiving 10,000 shares. `_binTotalShares[1] = 10_000`, `_binStates[1].token0BalanceScaled = X > 0`.
2. A series of `!zeroForOne` swaps fully consume bin `1`: `_binStates[1].token0BalanceScaled` reaches `0`, `_binStates[1].token1BalanceScaled` remains `0`. `_binTotalShares[1]` is unchanged at `10_000`.
3. Bob calls `addLiquidity(owner=Bob, salt=0, deltas={binIdxs:[1], shares:[10_000]}, ...)`.
   - `binTotalSharesVal = 10_000 > 0` → proportional branch.
   - `amount0Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - `amount1Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - `amount0Added = 0`, `amount1Added = 0` → callback skipped, no tokens transferred.
   - `_binTotalShares[1]` updated to `20_000`; `_positionBinShares[key(Bob,0,1)]` set to `10_000`.
4. Price reverses; `zeroForOne` swaps refill bin `1` with 1,000 units of token0 (scaled).
5. Bob calls `removeLiquidity` for his 10,000 shares: receives `1_000 * 10_000 / 20_000 = 500` units of token0 — paid nothing.
6. Alice calls `removeLiquidity` for her 10,000 shares: receives only `500` units instead of `1_000` — loses 500 units of token0 she legitimately deposited. [8](#0-7)
