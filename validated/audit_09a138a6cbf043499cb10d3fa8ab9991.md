### Title
Unconstrained `fee_proposal_fri` during bootstrap window enables arbitrary fee-market manipulation — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

During the first `fee_proposal_window_size` blocks, the `fee_proposal_fri` field in `ProposalInit` is accepted without any bounds check because `fee_actual` is `None`. A malicious proposer can set `fee_proposal_fri` to any arbitrary value (including `u128::MAX`). All validators accept it, it is committed into the `ProposalCommitment` that consensus signs over, and it is recorded in `fee_proposals_window`. Once the window fills, `fee_actual` is computed from these extreme values, permanently skewing `next_l2_gas_price` for all subsequent blocks.

### Finding Description

In `is_proposal_init_valid` the bounds check on `fee_proposal_fri` is gated on `fee_actual` being `Some`:

```rust
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
``` [1](#0-0) 

`fee_actual` is `None` whenever `height < window_size`, because `compute_fee_actual` returns `None` via the `checked_sub` guard:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
``` [2](#0-1) 

During this bootstrap period, any `fee_proposal_fri` value — including `u128::MAX` — passes `is_proposal_init_valid` without rejection.

The corrupted value then propagates through three steps:

**Step 1 — Committed into `ProposalCommitment`:** `proposal_commitment_from` hashes `fee_proposal_fri` into the commitment that consensus votes on, so validators sign over the extreme value:

```rust
ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
``` [3](#0-2) 

**Step 2 — Stored in `fee_proposals_window`:** At `decision_reached`, `record_fee_proposal(height, init.fee_proposal_fri)` inserts the extreme value into the sliding window: [4](#0-3) 

**Step 3 — Propagates to `next_l2_gas_price`:** `compute_fee_actual` computes the median of the window; `calculate_next_l2_gas_price_for_fin` uses `fee_actual` as a floor for `next_l2_gas_price`: [5](#0-4) 

The validator path constructs `ProposalInitValidation.fee_actual` from the same window at validation time: [6](#0-5) 

### Impact Explanation

A malicious proposer during the bootstrap window sets `fee_proposal_fri = u128::MAX`. All validators accept it (no bounds check when `fee_actual = None`). After `window_size` blocks, `fee_actual` becomes the median of these extreme values. `next_l2_gas_price` is then set to an extreme value, causing all users to pay incorrect (inflated or deflated) fees for all subsequent blocks. This is a persistent, chain-wide economic corruption.

Matching allowed impact: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

### Likelihood Explanation

In the BFT consensus model implemented here, the proposer role rotates among validators. Any validator who holds the proposer slot during the bootstrap window (the first `fee_proposal_window_size` blocks) can trigger this without any special privilege beyond normal consensus participation. The bootstrap window is a fixed, predictable period at chain genesis or after a restart.

### Recommendation

Add a bootstrap-specific bound on `fee_proposal_fri` that does not depend on `fee_actual`. For example:

- Require `fee_proposal_fri` to be within `[min_gas_price, some_configured_max]` unconditionally.
- Or use `l2_gas_price` (the current EIP-1559 base fee) as the reference for `fee_proposal_bounds` when `fee_actual` is `None`, instead of skipping the check entirely.
- Or initialize `fee_proposals_window` with `window_size` copies of `l2_gas_price` at genesis so `fee_actual` is always `Some` and the existing bounds check always fires.

### Proof of Concept

1. Chain starts at height 0; `fee_proposals_window` is empty; `compute_fee_actual` returns `None` for all heights `< window_size`.
2. Malicious proposer builds block at height 0 with `fee_proposal_fri = u128::MAX`.
3. `is_proposal_init_valid` reaches the `if let (Some(fee_actual), Some(fee_proposal))` guard — `fee_actual` is `None` — the entire bounds block is skipped; the proposal is accepted.
4. `proposal_commitment_from(partial, Some(u128::MAX))` produces the `ProposalCommitment`; consensus votes on it.
5. `decision_reached` calls `record_fee_proposal(BlockNumber(0), Some(GasPrice(u128::MAX)))`.
6. Repeat for blocks 1 … `window_size - 1`.
7. At height `window_size`, `compute_fee_actual` returns `Some(u128::MAX)` (median of all-`u128::MAX` window).
8. `calculate_next_l2_gas_price_for_fin` uses `fee_actual = u128::MAX` as a floor; `next_l2_gas_price` is set to an extreme value.
9. All transactions in every subsequent block are priced against this corrupted `l2_gas_price`.
