Audit Report

## Title
Unchecked Zero `owner` Address in `MetricOmmPool.addLiquidity` Permanently Locks LP Tokens - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no zero-address guard. If `owner = address(0)` is passed, the pool mints shares to an unreachable position key, pulls real tokens from `msg.sender` via the modify-liquidity callback, and permanently locks those tokens because `removeLiquidity` enforces `msg.sender == owner`, which can never be satisfied for `address(0)`.

## Finding Description
`MetricOmmPool.addLiquidity` (L182–196) accepts `address owner` and passes it directly to `LiquidityLib.addLiquidity` with no zero-address check:

```solidity
function addLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    bytes calldata callbackData, bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    if (deltas.binIdxs.length == 0) return (0, 0);
    if (deltas.binIdxs.length != deltas.shares.length) revert LiquidityDeltaLengthMismatch();
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData); // no zero check
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(..., owner, ...);
    ...
}
```

Inside `LiquidityLib.addLiquidity` (L72), the position key is computed as `keccak256(abi.encode(owner, salt, binIdx))` and shares are credited to that key (L121). Tokens are then pulled from `msg.sender` via `IMetricOmmModifyLiquidityCallback(msg.sender).metricOmmModifyLiquidityCallback(...)` (L147–148). The pool's `binState.token0BalanceScaled`, `binState.token1BalanceScaled`, and `binTotalShares` are all updated (L114, L118, L120).

`removeLiquidity` (L206) enforces `if (msg.sender != owner) revert NotPositionOwner()`. Since `msg.sender` can never equal `address(0)`, the position minted to the zero-address key is permanently frozen and the deposited tokens are irrecoverable.

The `_beforeAddLiquidity` hook in `ExtensionCalling.sol` (L88–99) performs no zero-address check — it only dispatches to configured extensions. No guard exists anywhere in the core pool call path.

The periphery `MetricOmmPoolLiquidityAdder` does guard against this at L65 (`addLiquidityExactShares`) and L101 (`addLiquidityWeighted`) via `_validateOwner(owner)` (L247–249). However, the core pool is a public interface callable by any contract or EOA directly, bypassing the periphery entirely.

## Impact Explanation
Direct, permanent loss of user principal. Tokens deposited under `owner = address(0)` are credited to an unreachable position key. They remain inside the pool's accounting (`binTotals`, per-bin balances) and cannot be extracted via `removeLiquidity`, `collectFees`, or any other mechanism. The locked shares inflate `binTotalShares` permanently, creating a phantom position that can never be burned.

## Likelihood Explanation
Medium-Low. The core pool is a public contract. Any integrator, router, or user calling `addLiquidity` directly (not through `MetricOmmPoolLiquidityAdder`) can trigger this. A single mistaken or malicious call is sufficient to permanently lock the deposited tokens. The periphery guard is not enforced at the core pool level.

## Recommendation
Add a zero-address guard at the top of `MetricOmmPool.addLiquidity`:

```solidity
function addLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    bytes calldata callbackData, bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    require(owner != address(0), "MetricOmmPool: zero owner");
    ...
}
```

Optionally add the same guard in `LiquidityLib.addLiquidity` as defense-in-depth.

## Proof of Concept
```solidity
// Direct caller (not using MetricOmmPoolLiquidityAdder) calls the core pool:
pool.addLiquidity(
    address(0),   // owner = zero address — no revert
    0,            // salt
    deltas,       // valid bin/share deltas
    callbackData, // callback that transfers tokens from msg.sender
    ""
);
// LiquidityLib credits shares to keccak256(abi.encode(address(0), 0, binIdx))
// Tokens are pulled from msg.sender via metricOmmModifyLiquidityCallback
// removeLiquidity(address(0), ...) always reverts NotPositionOwner
// because msg.sender != address(0) is always true
// Tokens are permanently locked in the pool
```
