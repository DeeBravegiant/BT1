### Title
`FanOut` Inspector Accepts Single-Provider Response Without Enforcing On-Chain RPC Quorum, Enabling Forged Foreign-Chain Verification — (File: `crates/foreign-chain-inspector/src/lib.rs`)

---

### Summary

The `FanOut` inspector, which is the production mechanism for multi-source foreign-chain RPC verification, accepts a result from a **single provider** whenever all other providers return transient errors. The on-chain `ChainEntry.quorum` field — voted in by network participants to require a minimum number of agreeing providers — is never read or enforced by the `FanOut` implementation. This reduces the effective per-node verification to a single data source under adversarial transient-error conditions, directly analogous to the single-source-of-truth vulnerability in the external report.

---

### Finding Description

The design specification in `docs/foreign-chain-transactions.md` and `docs/design/allowing-per-node-foreign-chain-rpc-configuration.md` is explicit:

> "A node considers the foreign TX verified iff at least the per-chain RPC quorum of providers agreed. If fewer than the RPC quorum agree, the node errors out that foreign-tx validation."

The on-chain `ChainEntry.quorum` is voted in by threshold participants alongside the provider list, so the network collectively decides how many providers must agree before a node accepts a verification result.

However, the production `FanOut::extract` implementation in `crates/foreign-chain-inspector/src/lib.rs` has **no quorum parameter and no quorum check**:

```rust
if let Some(first_values) = successes.first() {
    let all_successes_agree = successes.iter().all(|(_, v)| v == &first_values.1);
    if !all_successes_agree {
        return Err(ForeignChainInspectionError::InspectorResponseMismatch);
    }
    let (_, first) = successes.into_iter().next().expect("checked non-empty");
    return Ok(first);  // ← accepted with successes.len() == 1
}
``` [1](#0-0) 

Transient errors (`ClientError`, `RpcRequestFailed`, `NotFinalized`, `NotEnoughBlockConfirmations`) are silently tolerated and excluded from the substantive verdict:

```rust
Err(err) if err.is_transient() => {
    first_transient_error.get_or_insert(err);
}
``` [2](#0-1) 

This behavior is explicitly tested and confirmed as intentional:

```rust
// fan_out__should_tolerate_transient_when_only_one_inspector_succeeds
let succeeding = mock_returning(ok(vec![99]));
let make_transient = || mock_returning(err(|| ForeignChainInspectionError::NotFinalized));

``` [3](#0-2) 

The `FanOut` is constructed with only a `NonEmptyVec<Inspector>` — no quorum value is passed in or enforced:

```rust
pub struct FanOut<Inspector> {
    inspectors: NonEmptyVec<Inspector>,
}
``` [4](#0-3) 

The `ForeignChainInspectors` struct wraps every chain's providers in a `FanOut` at startup, and this is the sole verification mechanism used in `execute_foreign_chain_request`: [5](#0-4) [6](#0-5) 

The design document itself acknowledges this gap:

> "This sub-quorum outcome must be terminal — the leader must not re-attempt the request. **Implementation requirement, not current behavior**: the generic queue retries every request, so the foreign-tx path must special-case a sub-quorum result as non-retryable." [7](#0-6) 

---

### Impact Explanation

**Impact: High** — Forged foreign-chain verification leading to invalid bridge execution.

The `verify_foreign_transaction` flow is the trust anchor for the Omnibridge inbound path: the MPC network signs `SHA-256(borsh(ForeignTxSignPayload{request, observed_values}))` and returns the signature on-chain. A forged `observed_values` set (e.g., a fabricated `BlockHash` for a non-existent or reorg'd transaction) causes the MPC network to attest to a false foreign-chain event. Any bridge contract that gates fund release on this attestation will release funds for a transaction that never finalized on the foreign chain — a direct theft. [8](#0-7) [9](#0-8) 

---

### Likelihood Explanation

**Likelihood: Medium.**

The attack requires two simultaneous conditions:

1. **Transient suppression of all but one provider per node.** Shared API providers (Alchemy, QuickNode, Ankr) are subject to per-key rate limits (HTTP 429) and transient 5xx responses. An attacker who can flood a provider's API key with requests — or who operates infrastructure that causes selective timeouts — can cause `RpcRequestFailed` (classified transient) on all providers except one across multiple nodes simultaneously. This does not require full network-level DoS; targeted rate-limit exhaustion against shared API keys is a realistic, low-cost attack.

2. **Control of the surviving provider.** If all nodes share a common fallback provider (e.g., a public Ankr endpoint without authentication), and that provider is compromised or the attacker operates a provider that was voted into the whitelist, the attacker controls the single surviving data source.

The design doc explicitly acknowledges that the current MVP has each node querying only one provider per request:

> "Each MPC node only queries 1 RPC provider per verification request." [10](#0-9) 

Even with `FanOut`, the missing quorum enforcement means the security improvement over the MVP is illusory under adversarial transient-error conditions.

---

### Recommendation

1. **Enforce the on-chain quorum in `FanOut`.** Add a `quorum: usize` field to `FanOut` and reject the result if `successes.len() < quorum`, returning a new `SubQuorumResult` error variant (non-retryable).

2. **Pass `ChainEntry.quorum` to `FanOut` at construction time.** The node-side wiring (PR 3) must read `ChainEntry.quorum` from the on-chain whitelist and supply it to `build_fanout`.

3. **Treat sub-quorum as terminal.** As the design doc requires, a sub-quorum outcome must not be retried by the generic request queue.

4. **Enforce minimum provider count at registration.** Reject `register_available_foreign_chain_config` calls where the node's configured provider count for a chain is below `rpc_quorum(C)`, so nodes that cannot reach quorum are not counted as covering the chain.

---

### Proof of Concept

**Setup**: Network with signing threshold = 3, on-chain `ChainEntry.quorum` = 2 for Bitcoin. Each node has 3 providers: Alchemy, QuickNode, Ankr.

**Attack**:

1. Attacker submits `verify_foreign_transaction` for a Bitcoin transaction ID that does not exist (or exists only on a reorg'd fork).

2. Attacker exhausts the Alchemy and QuickNode API keys used by all nodes (e.g., via rate-limit flooding), causing both to return HTTP 429 → `RpcRequestFailed` → classified as `is_transient() == true`.

3. Attacker controls the Ankr endpoint (e.g., a public unauthenticated endpoint, or one voted into the whitelist). Ankr returns a fabricated `GetRawTransactionVerboseResponse` with a forged `blockhash` and `confirmations = 100`.

4. On each node, `FanOut::extract` collects:
   - Alchemy: `Err(RpcRequestFailed)` → transient, ignored
   - QuickNode: `Err(RpcRequestFailed)` → transient, ignored
   - Ankr: `Ok([BitcoinExtractedValue::BlockHash([0xde; 32])])` → single success

5. `FanOut` reaches line 130: `successes.len() == 1`, `all_successes_agree == true` (trivially), returns `Ok` with the forged block hash. No quorum check is performed.

6. `execute_foreign_chain_request` returns `ForeignTxSignPayload::V1 { request, values: [BlockHash([0xde; 32])] }`.

7. Each node computes `msg_hash = SHA-256(borsh(payload))` and produces a signature share. After threshold shares are collected, `respond_verify_foreign_tx` is called with the forged payload hash and valid threshold signature.

8. The contract verifies the ECDSA signature against the MPC public key — it is valid. The forged attestation is accepted on-chain. Any bridge contract gating fund release on this attestation releases funds for a transaction that never finalized. [11](#0-10) [6](#0-5) [9](#0-8)
