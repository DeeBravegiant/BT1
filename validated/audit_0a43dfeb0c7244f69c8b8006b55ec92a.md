Audit Report

## Title
Stale Cross-Chain rsETH Rate in `CrossChainRateReceiver` Enables Excess wrsETH Minting on Token Deposits — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns a cached rsETH/ETH rate with no staleness enforcement despite storing `lastUpdated`. L2 pool contracts (`RSETHPool`, `RSETHPoolV3`, `RSETHPoolNoWrapper`) use this stale rate alongside a live Chainlink collateral rate in the same deposit pricing formula. Because rsETH is monotonically appreciating, a stale (lower) rsETH rate causes the formula to mint more wrsETH/rsETH than the deposited collateral is worth, extracting accrued yield from existing holders.

## Finding Description

`CrossChainRateReceiver` stores `lastUpdated` on every `lzReceive` call but never enforces it in `getRate()`:

```solidity
// CrossChainRateReceiver.sol L13-16, L97, L103-105
uint256 public rate;
uint256 public lastUpdated;
// ...
lastUpdated = block.timestamp;   // set on receipt
// ...
function getRate() external view returns (uint256) {
    return rate;   // no staleness check
}
```

The rate is only refreshed when `MultiChainRateProvider.updateRate()` is called on L1 and the LayerZero message is delivered. There is no on-chain heartbeat or minimum-freshness requirement.

Each pool's `getRate()` delegates directly to the oracle:

```solidity
// RSETHPoolV3.sol L235-237
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

In `viewSwapRsETHAmountAndFee(amount, token)`, the stale rsETH rate and a live Chainlink collateral rate are used together:

```solidity
// RSETHPoolV3.sol L328-334
uint256 rsETHToETHrate = getRate();                                    // stale cached
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // live Chainlink
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls `latestRoundData()` live on every invocation and checks `answeredInRound < roundID` for Chainlink staleness, but there is no equivalent guard on the rsETH side. The identical vulnerable pattern exists in `RSETHPool.viewSwapRsETHAmountAndFee` (L340-346) and `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee` (L305-311).

**Exploit path:**
1. The L2 `CrossChainRateReceiver` holds a stale rate (e.g., `1.050e18`) while the true L1 rate has grown to `1.054e18`.
2. Attacker calls `deposit(wstETH, 100e18, "")` on any pool.
3. Pool computes: `rsETHAmount = 100e18 * 1.15e18 / 1.050e18 ≈ 109.52 rsETH`.
4. Fair value at current rate: `100e18 * 1.15e18 / 1.054e18 ≈ 109.11 rsETH`.
5. Attacker receives ~0.41 rsETH excess per 100 wstETH.
6. Attacker bridges wrsETH to L1, unwraps, and redeems at the actual rate — extracting yield accrued by existing holders.

The `deposit(token, amount, referralId)` function is fully public and requires no privileged role.

## Impact Explanation

**High — Theft of unclaimed yield.** The pool mints excess wrsETH to the depositor. When the pool bridges the deposited collateral to L1 and deposits into the LRT protocol, it receives rsETH at the actual (higher) rate, but the outstanding wrsETH claims exceed the rsETH backing. The deficit is borne by all existing wrsETH/rsETH holders as dilution of their accrued yield. At 5% APY, a 30-day staleness window creates ~0.4% mispricing; on a $10M deposit this is ~$40K of excess wrsETH minted.

## Likelihood Explanation

**Medium.** The rate becomes stale whenever `updateRate()` is not called on `MultiChainRateProvider` and the LayerZero message is not delivered. This can occur due to LayerZero delivery delays or failures, high gas costs discouraging updates, or operational gaps. No on-chain mechanism forces the rate to be fresh before a deposit is accepted. The longer the staleness window, the larger the extractable profit. The attack is repeatable by any depositor with no special access.

## Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and enforce it in `getRate()`:

```solidity
uint256 public maxStaleness; // e.g., 86400 (24 hours)

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes deposits to revert when the rsETH rate is stale, preventing mispriced minting until the rate is refreshed via LayerZero.

## Proof of Concept

**Foundry fork test outline:**

1. Fork an L2 (e.g., Arbitrum) at a block where `CrossChainRateReceiver.lastUpdated` is >24 hours old.
2. Confirm `CrossChainRateReceiver.rate` is lower than the current L1 `LRTOracle.rsETHPrice()`.
3. Call `RSETHPool.deposit(wstETH, 100e18, "")` as an unprivileged attacker.
4. Record `wrsETHReceived` from the attacker's balance delta.
5. Compute fair value: `100e18 * chainlinkWstETHRate / currentL1rsETHRate`.
6. Assert `wrsETHReceived > fairValue` — the excess is the extracted yield.
7. Bridge wrsETH to L1, unwrap, redeem rsETH at the actual rate, and confirm net profit.
