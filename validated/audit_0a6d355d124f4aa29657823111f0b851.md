Audit Report

## Title
Free Share Minting in Fully-Drained Bins Dilutes Existing LP Claims — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`LiquidityLib.addLiquidity` enters the proportional-share branch whenever `binTotalSharesVal > 0`, but if both `token0BalanceScaled` and `token1BalanceScaled` are zero (a bin fully drained by swaps), both computed amounts are zero and the token-payment callback is never invoked. Shares and bin-state are written to storage unconditionally, allowing an attacker to mint an arbitrary number of shares at zero cost. When tokens later flow back into the bin, the attacker's free shares entitle them to a proportional fraction of those tokens, directly stealing from existing LPs.

## Finding Description
In `LiquidityLib.addLiquidity`, the proportional branch at lines 108–111 computes:

```solidity
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
```

When both `token0BalanceScaled` and `token1BalanceScaled` are zero, `_checkedMul(0, sharesToAdd) = 0`, so `ceilDiv(0, binTotalSharesVal) = 0` for both. The conditional writes at lines 112–119 are skipped (both guards are false), but the unconditional share writes at lines 120–121 always execute:

```solidity
binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd;
positionBinShares[posKey] = newUserShares;
```

After the loop, `totalToken0ToAddScaled` and `totalToken1ToAddScaled` remain zero, so `amount0Added = 0` and `amount1Added = 0`. The callback guard at line 144 (`if (amount0Added > 0 || amount1Added > 0)`) is false, and no tokens are collected from the caller.

Swap functions (`_swapToken1ForToken0SpecifiedInput`, `_swapToken0ForToken1SpecifiedOutput`, etc.) only call `_saveBinState` to update `_binStates`; they never touch `_binTotalShares`. A bin drained to zero balances therefore retains its full pre-drain share count, making the precondition a persistent on-chain state. The `removeLiquidity` path at lines 205–206 computes payouts proportionally to shares held, so the attacker's free shares yield real tokens once the bin is refilled.

No access control exists on `addLiquidity` — any address may call it with any `owner`.

## Impact Explanation
Direct theft of LP principal. After the attacker mints free shares and the bin is refilled by reversed-price swaps, `removeLiquidity` transfers real tokens to the attacker proportional to their inflated share count. Existing LPs receive a smaller fraction than their deposited capital entitles them to. The loss scales linearly with the ratio of free shares minted to pre-existing `binTotalShares`. This constitutes pool insolvency: the pool's token balances no longer cover all legitimate LP claims.

## Likelihood Explanation
Bins being fully drained is a routine market event in any active pool — it occurs whenever the oracle price moves far enough that all liquidity in a bin is consumed by swaps. The attacker only needs to monitor `BinSwapped` events for a bin whose `delta0Scaled` or `delta1Scaled` brings the balance to zero while `_binTotalShares` remains positive, then call `addLiquidity` before the price reverses. No privileged access, special tokens, or unusual conditions are required. The attack is repeatable across any bin and any pool.

## Recommendation
Add a guard in the proportional branch of `LiquidityLib.addLiquidity` that rejects a deposit into a non-empty bin whose balance has been fully drained. Before writing any state, check:

```solidity
} else {
    amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
    amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
    // ADD: require at least one token is owed when joining an existing bin
    if (amount0Scaled == 0 && amount1Scaled == 0) revert DrainedBin(binIdx);
}
```

Alternatively, require `binState.token0BalanceScaled > 0 || binState.token1BalanceScaled > 0` before entering the proportional branch, mirroring the invariant that shares must always represent a non-zero token claim.

## Proof of Concept
1. Deploy a pool with bins `[-1, 0, 1]`. Alice calls `addLiquidity` for bin `1` (above current price), depositing token0 and receiving 10,000 shares. `_binTotalShares[1] = 10_000`, `_binStates[1].token0BalanceScaled = X > 0`.
2. A series of `!zeroForOne` swaps (token1 in, token0 out) fully consume bin `1`. `_swapToken1ForToken0SpecifiedInput` calls `_saveBinState` setting `token0BalanceScaled = 0`. `_binTotalShares[1]` is untouched and remains `10_000`.
3. Bob calls `addLiquidity(owner=Bob, salt=0, deltas={binIdxs:[1], shares:[10_000]}, ...)`.
   - `binTotalSharesVal = 10_000 > 0` → proportional branch taken (line 108).
   - `amount0Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - `amount1Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - Lines 120–121 write `_binTotalShares[1] = 20_000` and `_positionBinShares[key(Bob,0,1)] = 10_000`.
   - `amount0Added = 0`, `amount1Added = 0` → callback at line 144 is skipped. **Bob pays nothing.**
4. Price reverses; `zeroForOne` swaps refill bin `1` with 1,000 units of token0 (scaled). `_binStates[1].token0BalanceScaled = 1_000`.
5. Bob calls `removeLiquidity` for his 10,000 shares:
   - `amount0Scaled = 1_000 * 10_000 / 20_000 = 500` → Bob receives 500 units of token0 he never paid for.
   - Alice removes her 10,000 shares and receives only 500 units instead of 1,000 — a direct loss of 500 units of token0 principal.
