### Title
Unvalidated CKD Response for `AppPublicKey` Variant Allows Any Attested Participant to Deliver Arbitrary Key Material - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function in the MPC contract performs **no cryptographic validation** of the CKD response when the request uses the `CKDAppPublicKey::AppPublicKey` variant. Any single attested participant can call `respond_ckd` with completely arbitrary `big_y` and `big_c` values, and the contract will accept and deliver the corrupted output to the requesting user. The `AppPublicKeyPV` variant is correctly protected by `ckd_output_check`, but the `AppPublicKey` variant — the default, non-publicly-verifiable path used by the reference CLI — has no analogous guard.

### Finding Description

In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the request's `app_public_key` type:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no validation
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKeyPV` arm calls `ckd_output_check`, which verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, binding the response to the network's public key and the user's app identity. [2](#0-1) 

The `AppPublicKey` arm is an empty block — the response is accepted unconditionally. The contract's own unit test demonstrates this explicitly: it passes `[1u8; 48]` and `[2u8; 48]` (structurally invalid BLS12-381 points) as `big_y` and `big_c` and the call succeeds: [3](#0-2) 

The `AppPublicKey` variant is the default path used by the reference CKD CLI tool: [4](#0-3) 

The caller authorization check (`assert_caller_is_attested_participant_and_protocol_active`) only confirms the caller is a registered, TEE-attested participant — it does not require threshold agreement or coordinator status. A single attested node is sufficient to call `respond_ckd`. [5](#0-4) 

### Impact Explanation

**Critical — Confidential key derivation output without required participant authorization.**

A single malicious attested participant can race to call `respond_ckd` for any pending `AppPublicKey` CKD request and supply arbitrary `(big_y, big_c)` values. The contract resolves the yield and delivers the attacker-chosen output to the user. The user's derived secret — computed as `big_c − big_y · private_key` — will be garbage or attacker-controlled. Because the yield is resolved and the pending request is removed from state, the user cannot recover the correct key material from this request. The user's application-level secret is permanently corrupted for that derivation path/request.

This matches the allowed critical impact: *"Unauthorized … confidential key derivation output without the required participant authorization."*

### Likelihood Explanation

**High.** The `AppPublicKey` variant is the default, non-PV path. Any single attested participant — one that has passed TEE attestation and is in the active participant set — can exploit this. No threshold collusion is required. The attacker only needs to observe a pending CKD request on-chain (public state) and submit a `respond_ckd` call before the honest coordinator does. The attack is a simple NEAR transaction with no special cryptographic capability required.

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. For the non-PV case, the verifiable relationship is `e(big_c − H(app_id) · msk, g2) = e(big_y, pk1_as_g2)`, but since `pk1` is a G1 point and the contract only holds the G2 public key, the simplest fix is to require callers to use `AppPublicKeyPV` for all new requests (deprecating the unverifiable `AppPublicKey` variant), or to add an equivalent on-chain check using the G2 public key and the G1 app-key. At minimum, add a comment and a contract-level panic for the `AppPublicKey` arm until a proper check is implemented:

```rust
dtos::CKDAppPublicKey::AppPublicKey(_) => {
    // TODO: no on-chain verifiability; consider requiring AppPublicKeyPV
    // or add equivalent pairing check here.
    env::panic_str("AppPublicKey variant is not supported for respond_ckd; use AppPublicKeyPV");
}
```

### Proof of Concept

1. Alice submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)` and pays the deposit. The request is stored in `pending_ckd_requests`.
2. Mallory, a single attested participant, observes the pending `CKDRequest` on-chain.
3. Mallory calls `respond_ckd(ckd_request, CKDResponse { big_y: [0u8;48], big_c: [0u8;48] })`.
4. The contract's `match` arm for `AppPublicKey` executes the empty block — no validation occurs.
5. `pending_requests::resolve_yields_for` resolves Alice's yield with Mallory's garbage response.
6. Alice's callback receives `big_y = 0`, `big_c = 0`; her derived secret is the identity point — permanently corrupted.

The existing unit test at `crates/contract/src/lib.rs:3403–3441` already proves step 4 and 5 pass with structurally invalid point encodings `[1u8; 48]` / `[2u8; 48]`. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
