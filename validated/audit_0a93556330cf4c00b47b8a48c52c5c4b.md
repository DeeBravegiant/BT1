Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Deposit Cap Bypass — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses `totalAssetDeposits > limit` for ETH but `totalAssetDeposits + amount > limit` for LSTs. When `totalAssetDeposits == limit`, the ETH check returns `false` and the deposit proceeds, pushing total ETH deposits above the configured cap. Any unprivileged user can breach the deposit limit in a single transaction.

## Finding Description
In `contracts/LRTDepositPool.sol` at L676–682, the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit` evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding the incoming `amount`, while the LST branch correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. When `totalAssetDeposits` equals the configured limit exactly, the ETH condition is `limit > limit` → `false`, so `_beforeDeposit` (L661–663) does not revert, `getRsETHAmountToMint` is called, and `_mintRsETH` mints rsETH for the full deposit. The entry point `depositETH` (L76–93) is `external payable` with no privileged access control, making this reachable by any caller. No existing guard compensates for the missing `amount` term.

## Impact Explanation
The deposit limit is the protocol's primary mechanism for bounding ETH exposure to EigenLayer strategies and NodeDelegators. After a successful over-limit deposit, `getTotalAssetDeposits(ETH_TOKEN) > depositLimitByAsset(ETH_TOKEN)`, breaking the invariant `totalAssetDeposits ≤ depositLimit` for ETH while the same invariant is correctly enforced for every LST. No funds are directly stolen, but the protocol fails to enforce its promised deposit cap. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
Any unprivileged user can call `depositETH`. The precondition `totalAssetDeposits == depositLimit` is fully attacker-controllable: monitor on-chain state, deposit exactly `depositLimit − totalAssetDeposits` ETH to reach the boundary, then immediately deposit any additional amount. No privileged role, oracle manipulation, or external dependency is required. The attack is repeatable as long as the limit is not raised.

## Recommendation
Add `amount` to the ETH branch to match the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // fix: add amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Read `D = lrtConfig.depositLimitByAsset(ETH_TOKEN)` and `T = getTotalAssetDeposits(ETH_TOKEN)`.
2. Call `depositETH{value: D − T}(0, "")` — brings `totalAssetDeposits` to exactly `D`.
3. Call `depositETH{value: X}(0, "")` for any `X > 0`.
   - `_checkIfDepositAmountExceedesCurrentLimit`: `D > D` → `false` → no revert.
   - `_mintRsETH` mints rsETH for `X`.
4. Assert `getTotalAssetDeposits(ETH_TOKEN) == D + X > D` — deposit cap broken.

Foundry invariant test: set `depositLimitByAsset(ETH_TOKEN) = D`, fuzz `depositETH` calls, assert `getTotalAssetDeposits(ETH_TOKEN) <= D` after every call. The invariant will be violated at step 3.
