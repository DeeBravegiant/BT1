### Title
Caller-Agnostic Request Key in `verify_foreign_transaction` Enables Queue Saturation, Blocking Legitimate User Resubmission - (File: `crates/contract/src/lib.rs`, `crates/contract/src/dto_mapping.rs`)

### Summary

`verify_foreign_transaction` constructs its pending-request map key (`VerifyForeignTransactionRequest`) without binding it to the caller's account ID. Any unprivileged caller can submit the identical request and occupy slots in the same fan-out queue. An attacker can saturate the queue to its hard cap of 128, so that after a legitimate user's yield times out they cannot resubmit — their bridge operation is blocked until all attacker-controlled entries also time out.

### Finding Description

`sign` and `request_app_private_key` both derive a `tweak` from `predecessor_account_id + path`, making each caller's request key unique:

```rust
// SignatureRequest::new — caller identity is baked into the key
let tweak = crate::kdf::derive_tweak(predecessor_id, path);
```

`verify_foreign_transaction` does not. `args_into_verify_foreign_tx_request` maps only `{domain_id, request, payload_version}` into the stored key:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

The contract itself confirms this is caller-agnostic — the unit test comment reads: *"both yields are queued under the single (caller-agnostic) request key."*

The fan-out queue is hard-capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`:

```rust
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
    );
}
```

**Attack path:**
1. Alice submits `verify_foreign_transaction(bitcoin_tx_X)` — queue depth = 1, Alice's yield is registered.
2. Bob submits the identical request 127 more times — queue depth = 128 (full).
3. Alice's yield times out (~200 NEAR blocks). `pop_oldest_pending_yield` removes only Alice's entry; Alice's transaction fails.
4. Alice attempts to resubmit — `PendingRequestQueueFull` is returned because Bob's 127 entries still occupy the queue.
5. Alice is blocked from resubmitting until all of Bob's entries individually time out.

Bob's cost: 127 × 1 yoctoNEAR deposit + gas. Alice's cost: lost gas, lost deposit, and a multi-hundred-block delay on a time-sensitive bridge operation.

### Impact Explanation

This is a **request-lifecycle manipulation** that breaks the production safety invariant that a legitimate user who paid the required deposit can always resubmit a timed-out request. In a bridge context where `verify_foreign_transaction` is used to attest that a foreign-chain deposit occurred, a sustained queue-saturation attack can cause Alice's claim window to expire, resulting in permanent loss of bridged funds. This maps to the allowed Medium impact: *"request-lifecycle … manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

### Likelihood Explanation

The attack requires only a valid NEAR account and 127 × 1 yoctoNEAR + gas — effectively free. The attacker needs to know Alice's pending request parameters (observable on-chain via `get_pending_verify_foreign_tx_request`). The attack is repeatable: after Bob's entries time out, he can immediately re-saturate the queue. Any bridge service that relies on `verify_foreign_transaction` for inbound settlement is a realistic target.

### Recommendation

Bind the `VerifyForeignTransactionRequest` key to the caller's account ID, mirroring the pattern used by `sign` and `request_app_private_key`. Introduce a `tweak` field derived from `(predecessor_account_id, derivation_path)` using the foreign-tx-specific prefix already documented in `docs/foreign-chain-transactions.md`:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    pub tweak: Tweak,  // derived from predecessor_id + derivation_path
}
```

`args_into_verify_foreign_tx_request` should accept the caller's `predecessor_account_id` and `derivation_path` and compute the tweak before constructing the key, exactly as `CKDRequest::new` and `SignatureRequest::new` do.

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
