Audit Report

## Title
Donation Attack via Open `receive()` Inflates `rsETHPrice` Due to Uninitialized `pricePercentageLimit`, Enabling Theft from Depositors - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle._updateRsETHPrice()` computes the rsETH price using `address(this).balance` and `IERC20.balanceOf(address(this))` of `LRTDepositPool` directly, without distinguishing protocol-deposited funds from arbitrary donations. Because `pricePercentageLimit` is never set in `initialize()` and defaults to `0`, the price-cap guard is permanently disabled, allowing a single ETH donation via the open `receive()` to inflate `rsETHPrice` by an arbitrary multiple. Subsequent depositors who set `minRSETHAmountExpected = 0` receive near-zero rsETH for their full deposit, while the attacker—who initiated a withdrawal at the inflated price—extracts the victim's funds.

## Finding Description

**Root cause 1 — `pricePercentageLimit` uninitialized:**

`LRTOracle.initialize()` never sets `pricePercentageLimit`:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
``` [1](#0-0) 

The guard in `_updateRsETHPrice()` short-circuits to `false` whenever `pricePercentageLimit == 0`:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [2](#0-1) 

This means any price increase of any magnitude passes unchecked until an admin explicitly calls `setPricePercentageLimit`.

**Root cause 2 — raw balance used in TVL accounting:**

`getETHDistributionData()` sets `ethLyingInDepositPool = address(this).balance`: [3](#0-2) 

`getAssetDistributionData()` sets `assetLyingInDepositPool = IERC20(asset).balanceOf(address(this))`: [4](#0-3) 

Both are included in `getTotalAssetDeposits`, which feeds `_getTotalEthInProtocol()`, which feeds `newRsETHPrice`: [5](#0-4) 

**Root cause 3 — open `receive()`:**

`LRTDepositPool` accepts arbitrary ETH with no accounting: [6](#0-5) 

**Exploit flow:**

1. Attacker deposits a small amount of ETH via `depositETH()` → receives `N` wei rsETH at price `1e18`.
2. Attacker sends `D` ETH directly to `LRTDepositPool` via `receive()` (no rsETH minted).
3. Attacker calls the public `updateRSETHPrice()` → `rsETHPrice` jumps to `≈ D * 1e18 / N`. The guard is bypassed because `pricePercentageLimit == 0`.
4. Attacker calls `LRTWithdrawalManager.initiateWithdrawal()`, which calls `getExpectedAssetAmount()`:
   ```solidity
   underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
   ```
   locking in the inflated `expectedAssetAmount` for the attacker's rsETH. [7](#0-6) 
5. Victim calls `depositETH()` with `minRSETHAmountExpected = 0`. `getRsETHAmountToMint` divides by the inflated `rsETHPrice`, rounding down to near-zero rsETH:
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ``` [8](#0-7) 
   `_beforeDeposit` does not revert because `rsethAmountToMint (≈0) >= minRSETHAmountExpected (0)`. [9](#0-8) 
6. After the withdrawal delay, the attacker completes withdrawal and receives the inflated ETH amount (which now includes the victim's deposit), netting a profit proportional to the victim's deposit minus the donation cost.

**Existing checks reviewed and shown insufficient:**

- `pricePercentageLimit` guard: disabled by default (zero value).
- `minRSETHAmountExpected`: user-supplied slippage parameter; defaults to 0 in many integrations and is not enforced by the protocol to be non-zero.
- `_checkIfDepositAmountExceedesCurrentLimit`: only checks against a configured deposit cap, which is unrelated to price manipulation.

## Impact Explanation

**Critical — Direct theft of user funds.**

A victim depositing `V` ETH with `minRSETHAmountExpected = 0` receives near-zero rsETH. The attacker, holding a disproportionate share of rsETH supply, redeems at the inflated price and extracts a large fraction of `V`. The attacker's net profit scales with `V` minus the donation cost `D`. For large `V` (e.g., 1999 ETH) and moderate `D` (e.g., 1000 ETH), the attacker profits ~499 ETH at the victim's expense. This is direct, at-rest fund theft matching the Critical impact class.

## Likelihood Explanation

- `pricePercentageLimit` is `0` by default on every fresh deployment; no admin action is required to leave the guard disabled.
- `updateRSETHPrice()` is a permissionless public function callable by any EOA.
- `LRTDepositPool.receive()` is unconditionally open; no minimum, no access control.
- `minRSETHAmountExpected = 0` is a common default in programmatic integrations, scripts, and frontends that do not compute expected output before submission.
- The attack is repeatable: after each victim deposit, the attacker can re-inflate and re-initiate a new withdrawal.

## Recommendation

1. **Initialize `pricePercentageLimit` in `initialize()`** to a safe non-zero value (e.g., `1e16` for 1%) so the guard is active from deployment.
2. **Decouple TVL accounting from raw balances**: track deposited amounts in a storage variable incremented only through `depositETH`/`depositAsset`, rather than reading `address(this).balance` or `IERC20.balanceOf(address(this))` directly.
3. **Enforce a minimum `minRSETHAmountExpected`** in `_beforeDeposit` (e.g., revert if `rsethAmountToMint == 0`).
4. Consider restricting or removing the open `receive()` fallback, or routing it through an accounting path.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// 1. Deploy/fork with pricePercentageLimit == 0 (default)
// 2. attacker.depositETH{value: 1 wei}(0, "")  → attacker holds 1 wei rsETH
// 3. address(lrtDepositPool).call{value: 1000 ether}("")  → donate, no rsETH minted
// 4. lrtOracle.updateRSETHPrice()  → rsETHPrice ≈ 1000e18 (guard bypassed)
// 5. withdrawalManager.initiateWithdrawal{from: attacker}(ETH, 1 wei, "")
//    → expectedAssetAmount ≈ 1000 ETH locked for attacker
// 6. victim.depositETH{value: 1999 ether}(0, "")
//    → rsethAmountToMint = (1999e18 * 1e18) / 1000e18 = 1 wei rsETH for victim
// 7. lrtOracle.updateRSETHPrice()  → rsETHPrice ≈ 1499.5e18
// 8. (after withdrawalDelayBlocks) operator.unlockQueue(...)
// 9. attacker.completeWithdrawal(ETH, "")  → receives ≈1000 ETH
// Assert: attacker net gain ≈ 0 ETH (1000 donated - 1000 received) ... 
//         but victim deposited 1999 ETH and holds 1 wei rsETH worth ≈1499.5 ETH
//         → victim loss ≈ 499.5 ETH; attacker profit ≈ 499.5 ETH
```

The test should assert `victim_rseth_value < victim_deposit * 0.99` and `attacker_profit > 0` to confirm the theft.
