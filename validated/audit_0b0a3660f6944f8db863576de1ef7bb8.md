### Title
Precommit Vote Digest Omits `round` and `chain_id`, Enabling Cross-Round and Cross-Chain Signature Replay — (`File: crates/apollo_signature_manager/src/signature_manager.rs`)

### Summary

`build_precommit_vote_message_digest` constructs the signed payload for precommit votes using only a domain separator and the `block_hash`. It omits `round` and `chain_id`. Because the `Vote` wire struct carries `round` as a separate, unsigned field, any network observer can take a legitimately-signed precommit for `(height H, round 0, block_hash X)` and rebroadcast it with `round = 1` (or any later round). When the currently-stubbed signature verification (`// TODO(Asmaa): verify the signature`) is wired in, the replayed vote will pass cryptographic verification because the digest is identical across rounds.

### Finding Description

`build_precommit_vote_message_digest` in `crates/apollo_signature_manager/src/signature_manager.rs` builds the signed payload as:

```
PRECOMMIT_VOTE || block_hash_bytes
``` [1](#0-0) 

The `Vote` struct that is broadcast over the network carries `height`, `round`, `voter`, `vote_type`, `proposal_commitment`, and `signature` as separate fields. The signature covers only `proposal_commitment` (the block hash). `round` and `chain_id` are not bound. [2](#0-1) 

The corresponding verification function `verify_precommit_vote_signature` reconstructs the same digest from only `block_hash`, so it will accept any `Vote` whose `round` field has been altered as long as the `block_hash` and signer key match. [3](#0-2) 

`SingleHeightConsensus::handle_vote` currently skips signature verification entirely with a TODO comment, so the broken digest is latent today but will become the active verification path when that TODO is resolved. [4](#0-3) 

### Impact Explanation

**Cross-round replay (primary impact):** In Tendermint, when round 0 fails and round 1 begins, the proposer may repropose the same locked block (same `block_hash`). A network observer who captured a valid precommit from validator V for `(H, round=0, block_hash=X)` can rebroadcast it with `round=1`. Once signature verification is enabled using the current digest, the state machine at round 1 will accept this as a genuine precommit from V, potentially manufacturing a false quorum and causing consensus to decide a block in a round where the real quorum was never reached.

**Cross-chain replay (secondary impact):** Because `chain_id` is absent from the digest, a precommit signature produced on one Starknet deployment is cryptographically valid on any other deployment where the same validator key and block hash appear.

The broken invariant is: *a precommit signature must bind the signer to a specific `(height, round, block_hash, chain_id)` tuple*. Currently it binds only to `block_hash`.

### Likelihood Explanation

- The attack requires only passive observation of broadcast votes (no privileged access, no validator key).
- The Tendermint reproposal path (`update_for_reproposal`) is a normal code path that reuses the same block hash across rounds, making the cross-round scenario reachable in any multi-round consensus execution. [5](#0-4) 

- The vulnerability becomes exploitable the moment the `TODO(Asmaa)` is resolved; the broken digest is already committed to production code.

### Recommendation

Include `round`, `height`, and `chain_id` in the precommit vote message digest:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    height: BlockNumber,
    round: Round,
    chain_id: &ChainId,
) -> MessageDigest {
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash.to_bytes_be());
    message.extend_from_slice(&height.0.to_be_bytes());
    message.extend_from_slice(&round.to_be_bytes());
    message.extend_from_slice(chain_id.as_bytes());
    MessageDigest(blake2s_to_felt(&message))
}
```

Update `sign_precommit_vote` and `verify_precommit_vote_signature` to accept and pass these additional fields. Ensure the same fields are passed at the call site in `communication.rs` and at the future verification site in `single_height_consensus.rs`.

### Proof of Concept

1. Consensus is at height H, round 0. Proposer broadcasts block with `block_hash = X`.
2. Validator V signs and broadcasts `Vote { height: H, round: 0, proposal_commitment: X, vote_type: Precommit, signature: sig_V }`.
3. Round 0 fails to reach quorum (e.g., only V and one other validator precommit).
4. Consensus advances to round 1. The same block X is reproposed (locked value path).
5. Attacker observes V's precommit from step 2 and rebroadcasts it as `Vote { height: H, round: 1, proposal_commitment: X, vote_type: Precommit, signature: sig_V }` — only the `round` field is changed; the signature is unchanged.
6. When `verify_precommit_vote_signature(X, sig_V, V.public_key)` is called, it recomputes `blake2s(PRECOMMIT_VOTE || X_bytes)` — identical to the original digest — and returns `true`.
7. The state machine at round 1 counts V's replayed precommit as genuine, potentially reaching a false quorum and committing a block that the real validator set did not agree to in round 1. [6](#0-5) [1](#0-0)
