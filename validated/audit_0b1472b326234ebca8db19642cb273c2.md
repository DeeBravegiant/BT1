### Title
Incomplete Conditional in `validate_delegate_action_key` Allows Nonce Advancement on Rejected `DelegateAction` - (File: runtime/runtime/src/actions.rs)

### Summary

In `validate_delegate_action_key`, when a `DelegateAction` signed with a `FunctionCallPermission` key includes `deposit > 0`, the `DepositWithFunctionCall` error is set but execution falls through to subsequent checks due to a missing early return. When the receiver_id and method_name also match the permission, the nonce update at the end of the function executes, advancing the access key nonce in the trie even though the action is rejected. This is the direct analog of the reported "incomplete conditional causes assets to be locked" bug class: a missing branch in a public entry path causes incorrect protocol state to be committed.

### Finding Description

In `validate_delegate_action_key` at `runtime/runtime/src/actions.rs` lines 637–699, the `DepositWithFunctionCall` error path lacks an unconditional early return before protocol version 85:

```rust
if function_call.deposit > Balance::ZERO {
    result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
        InvalidAccessKeyError::DepositWithFunctionCall,
    ).into());
    // Missing early return before fix:
    if ProtocolFeature::FixDelegateActionDepositWithFunctionCallError
        .enabled(apply_state.current_protocol_version)
    {
        return Ok(());
    }
}
// Falls through to receiver_id check...
if delegate_action.receiver_id() != &function_call_permission.receiver_id {
    result.result = Err(...ReceiverMismatch...);
    return Ok(());
}
// Falls through to method_name check...
// If both pass, falls through to nonce update:
match nonce_update {
    DelegateNonceUpdate::AccessKey => {
        access_key.nonce = delegate_nonce.nonce();
        set_access_key(state_update, ...);
    }
    DelegateNonceUpdate::GasKey { nonce_index } => {
        set_gas_key_nonce(state_update, ...);
    }
}
```

Two distinct broken outcomes exist before the fix:

1. **Error overwrite**: If `deposit > 0` AND `receiver_id` does not match the permission, `ReceiverMismatch` overwrites `DepositWithFunctionCall`. The action is still rejected, but the wrong error is returned to the user.

2. **Nonce corruption (concrete state impact)**: If `deposit > 0` AND `receiver_id` matches AND `method_name` matches (or `method_names` is empty), execution reaches the nonce update block. The access key nonce (or gas key nonce slot) is advanced in the trie even though `result.result` is `Err(DepositWithFunctionCall)`. Back in `apply_delegate_action`, the check `if result.result.is_err() { return Ok(()); }` prevents the inner receipt from being generated, but the nonce write to state has already occurred. [1](#0-0) [2](#0-1) 

The code comment at line 642 explicitly acknowledges the pre-fix behavior: *"Before this fix, the missing early return allowed execution to fall through to the receiver_id and method_name checks, which could overwrite this error with a different one."* [3](#0-2) 

The fix is gated on `ProtocolFeature::FixDelegateActionDepositWithFunctionCallError`, assigned to protocol version 85. [4](#0-3) [5](#0-4) 

### Impact Explanation

The corrupted protocol value is the **access key nonce** (or gas key nonce slot) stored in the trie. Specifically:

- `set_access_key` writes an updated `access_key.nonce` to the trie for `DelegateNonceUpdate::AccessKey`.
- `set_gas_key_nonce` writes an updated nonce row to the trie for `DelegateNonceUpdate::GasKey { nonce_index }`.

Both writes occur even though the delegate action is invalid and rejected. The result is that the on-chain nonce for the sender's key is advanced without any corresponding action being executed. The user must use a strictly higher nonce for all future valid delegate actions using that key. For gas keys with `FunctionCallPermission` and a small `num_nonces`, repeated triggering of this path consumes nonce slots without executing any action.

The action itself is rejected (no inner receipt is generated), so no token transfer occurs. However, the trie DB entry for the nonce is permanently corrupted relative to what the protocol should have committed.

### Likelihood Explanation

Triggering the nonce-corruption path requires all of the following simultaneously (before protocol version 85):

1. A `DelegateAction` signed with a `FunctionCallPermission` key.
2. The inner `FunctionCall` action has `deposit > 0`.
3. The `receiver_id` in the delegate action matches the permission's `receiver_id`.
4. The `method_name` matches (or `method_names` is empty).

This is a low-likelihood scenario in practice, most likely triggered accidentally by a user or relayer that incorrectly constructs a meta transaction. An adversary cannot use this to harm another user's nonce, since the `DelegateAction` must be signed by the sender's own private key.

### Recommendation

The fix (`ProtocolFeature::FixDelegateActionDepositWithFunctionCallError`) is already present and assigned to protocol version 85. Nodes running protocol version ≥ 85 are not affected. The vulnerable code path is retained only for backward-compatible replay of historical blocks at older protocol versions. No further action is required beyond ensuring all nodes upgrade to protocol version 85.

### Proof of Concept

The repository's own test suite confirms both behaviors: [6](#0-5) 

`test_delegate_deposit_with_function_call_reports_receiver_mismatch_before_fix` (line 1764) demonstrates that before the fix, `ReceiverMismatch` overwrites `DepositWithFunctionCall`. The nonce-corruption path (receiver_id and method_name both match) is the logical extension of the same missing early return: execution reaches `set_access_key`/`set_gas_key_nonce` at lines 685–698 and commits the nonce to the trie before `apply_delegate_action` checks `result.result.is_err()` and discards the inner receipt. pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
    if result.result.is_err() {
        // Validation failed. Need to return Ok() because this is not a runtime error.
        // "result.result" will be return to the User as the action execution result.
        return Ok(());
    }

    // Generate a new receipt from DelegateAction.
    let new_receipt = Receipt::V0(ReceiptV0 {
        predecessor_id: sender_id.clone(),
        receiver_id: delegate_action.receiver_id().clone(),
        receipt_id: CryptoHash::default(),

        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: action_receipt.signer_id().clone(),
            signer_public_key: action_receipt.signer_public_key().clone(),
            gas_price: action_receipt.gas_price(),
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: delegate_action.get_actions(),
        }),
    });

    // Note, Relayer prepaid all fees and all things required by actions: attached deposits and attached gas.
    // If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction.
    // Gas is refunded to the signer, this is Relayer.
    // Some contracts refund the deposit. Usually they refund the deposit to the predecessor and this is sender_id/Sender from DelegateAction.
    // Therefore Relayer should verify DelegateAction before submitting it because it spends the attached deposit.

    let prepaid_send_fees = total_prepaid_send_fees(&apply_state.config, action_receipt.actions())?;
    let required_cost = receipt_required_cost(apply_state, &new_receipt)?;
    // This gas will be burnt by the receiver of the created receipt.
    // Compute costs of that are not relevant at this point, the "used" gas is
    // only reserved for execution later, potentially on a different shard.
    result.gas_used = result.gas_used.checked_add_result(required_cost.gas)?;
    // This gas was prepaid on Relayer shard. Need to burn it because the receipt is going to be sent.
    // gas_used is incremented because otherwise the gas will be refunded. Refund function checks only gas_used.
    result.gas_used = result.gas_used.checked_add_result(prepaid_send_fees.gas)?;
    result.gas_burnt = result.gas_burnt.checked_add_result(prepaid_send_fees.gas)?;
    result.compute_usage = safe_add_compute(result.compute_usage, prepaid_send_fees.compute)?;
    result.new_receipts.push(new_receipt);

    Ok(())
}
