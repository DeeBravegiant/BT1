Audit Report

## Title
No Staleness Check on `CrossChainRateReceiver.getRate()` Allows Depositors to Mint Excess rsETH at Stale Rate — (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver` stores both a `rate` and a `lastUpdated` timestamp, but `getRate()` returns `rate` unconditionally without verifying freshness. Because `updateRate()` is permissionless and requires the caller to pay LayerZero fees, the L2 rate can drift below the true L1 rsETH/ETH rate. Any depositor can exploit this divergence to mint more rsETH than their ETH contribution is worth, then redeem on L1 at the true rate, extracting accrued yield from existing rsETH holders.

## Finding Description
`CrossChainRateReceiver.getRate()` (lines 102–105) returns the stored `rate` with no staleness guard:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
```

`lastUpdated` is written in `lzReceive()` (line 97) on every cross-chain message but is **never read** inside `getRate()`. The rate is pushed from L1 via `RSETHMultiChainRateProvider.updateRate()` or `CrossChainRateProvider.updateRate()`, both of which are `external payable` with no access control — the caller must supply ETH to cover LayerZero messaging fees with no reimbursement. During periods of high gas costs or low protocol activity, no one has an on-chain incentive to refresh the rate.

All L2 pool contracts (`RSETHPoolV3`, etc.) call `IOracle(rsETHOracle).getRate()` to price minting:

```solidity
// RSETHPoolV3.sol:234-237
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

The minting formula in `viewSwapRsETHAmountAndFee` (line 307) divides by this rate:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

A lower (stale) `rsETHToETHrate` produces a larger `rsETHAmount`. The `dailyMintLimit` modifier caps total daily minting but does not prevent the stale-rate exploit — it only bounds the scale per day.

**Exploit path:**
1. L1 `rsETHPrice` rises due to staking yield accrual; no one calls `updateRate()`.
2. L2 `CrossChainRateReceiver.rate` remains at the old lower value.
3. Attacker calls `RSETHPoolV3.deposit{value: N ether}("")`.
4. `viewSwapRsETHAmountAndFee` computes `rsETHAmount` using the stale (lower) rate, minting more rsETH than the deposit is worth at the true rate.
5. Attacker bridges rsETH to L1 and redeems at the true (higher) rate, receiving more ETH than deposited.
6. The excess ETH comes from yield that belonged to existing rsETH holders.

## Impact Explanation
**High — Theft of unclaimed yield.** The stale rate causes the pool to over-mint rsETH relative to the depositor's ETH contribution. When the attacker redeems on L1 at the true rate, they extract ETH representing yield that has accrued to existing rsETH holders but has not yet been reflected in the L2 oracle. The magnitude scales with the rate divergence and deposit size. The `dailyMintLimit` bounds per-day exposure but does not eliminate it.

## Likelihood Explanation
**Medium.** No privileged access is required. The attacker only needs ETH to deposit and the ability to monitor the divergence between the on-chain L1 `rsETHPrice` and the L2 `CrossChainRateReceiver.rate`. The attack is most profitable when LayerZero fees are high (discouraging voluntary updates) and when significant yield has accrued. It is repeatable across multiple days up to the daily mint limit each time.

## Recommendation
Add a configurable `MAX_STALENESS` constant and revert in `getRate()` if the rate is too old:

```solidity
uint256 public constant MAX_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate is stale");
    return rate;
}
```

This converts silent mis-pricing into a hard revert, preventing deposits when the oracle has not been refreshed. Pair this with an on-chain keeper incentive (e.g., a small ETH reward funded from protocol fees) to ensure timely updates and avoid locking deposits unnecessarily.

## Proof of Concept
1. At T₀: L1 `rsETHPrice = 1.05e18`, L2 `CrossChainRateReceiver.rate = 1.05e18`.
2. Staking yield accrues; at T₁: L1 `rsETHPrice = 1.10e18`. No one calls `updateRate()` (LayerZero fees are non-trivial, no reimbursement).
3. L2 `CrossChainRateReceiver.rate` remains `1.05e18`.
4. Attacker calls `RSETHPoolV3.deposit{value: 10 ether}("")`:
   - `viewSwapRsETHAmountAndFee(10e18)` → `rsETHAmount = 10e18 * 1e18 / 1.05e18 ≈ 9.524e18` rsETH minted.
   - Correct amount at true rate: `10e18 * 1e18 / 1.10e18 ≈ 9.091e18` rsETH.
   - Excess: `≈ 0.433e18` rsETH.
5. Attacker bridges rsETH to L1 and redeems at `1.10e18` rate: receives `≈ 10.476 ETH`.
6. Profit: `≈ 0.476 ETH` per 10 ETH deposited, extracted from existing rsETH holders' accrued yield.

**Foundry fork test plan:** Fork the target L2 chain. Deploy or mock `CrossChainRateReceiver` with `rate = 1.05e18` and `lastUpdated` set to a past timestamp. Set L1 oracle mock to `1.10e18` without calling `updateRate()`. Call `RSETHPoolV3.deposit{value: 10 ether}("")` and assert that minted rsETH exceeds `10e18 * 1e18 / 1.10e18`, confirming over-minting at the stale rate.
