### Title
Unvalidated Peer Response in `coins_of_interest_farmed` Allows Malicious Node to Forge Offer Settlement Confirmation — (File: `chia/wallet/trade_manager.py`)

---

### Summary

`TradeManager.coins_of_interest_farmed` updates a trade's status to `CONFIRMED` (or `FAILED`/`CANCELLED`) based on coin-state data returned by a raw, unvalidated `get_coin_state` network call to the connected peer. Because this secondary peer call is made **after** the initial coin-state trigger but **before** the trade-status write, a malicious full-node peer can return fabricated settlement-coin states and force the wallet to record a false trade confirmation, corrupting offer/trade settlement state and the wallet's displayed balance.

---

### Finding Description

**Call chain:**

```
state_update_received (wallet_node.py:1098)
  └─ add_states_from_peer
       └─ add_coin_states / _add_coin_states (wallet_state_manager.py:1817-1818)
            └─ trade_manager.coins_of_interest_farmed (trade_manager.py:141)
```

Inside `_add_coin_states`, when a spent coin belongs to a pending trade, the code calls `coins_of_interest_farmed` while still holding the `db_wrapper.writer()` lock: [1](#0-0) 

Inside `coins_of_interest_farmed`, the function:

1. **Reads** the current trade record from the DB. [2](#0-1) 

2. **Makes an external network call** to the same peer, asking for coin states of the offer's settlement-payment coins — with **no cryptographic validation** (no header-block fetch, no Merkle-inclusion proof): [3](#0-2) 

3. **Updates trade status** based solely on whether the peer's response contains the expected coin IDs: [4](#0-3) 

This is a classic **check-effects-interactions** violation: the effect (trade status write) occurs after an untrusted external interaction, and the interaction result is accepted without proof.

Compare with the validated path used for the *primary* coin-state processing, which fetches header blocks and verifies Merkle addition/removal roots before accepting any state: [5](#0-4) [6](#0-5) 

No equivalent validation is applied to the `get_coin_state` call inside `coins_of_interest_farmed`.

---

### Impact Explanation

**High — Corruption of offer/trade settlement state with direct security impact.**

A malicious peer can:

- Force a `PENDING_ACCEPT` trade to be recorded as `CONFIRMED`, causing the wallet to add confirmed `INCOMING_TRADE` transaction records and display a balance the user never actually received.
- Force a legitimately completed trade to be recorded as `FAILED`, causing `delete_trade_transactions` to erase the correct accounting and the wallet to show the user lost funds.

In both cases the wallet's local DB diverges from on-chain reality. A merchant wallet that relies on `TradeStatus.CONFIRMED` to release goods or services is directly defrauded. The wallet's `confirmed_wallet_balance` and `unconfirmed_wallet_balance` are corrupted for the duration until a re-sync with an honest peer corrects the state. [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

Any unprivileged attacker who can position themselves as the wallet's full-node peer (e.g., by running a malicious node that the victim connects to, or by being a peer on the default peer list) can trigger this. The offer's coin IDs are public once an offer file is shared (e.g., on an offer marketplace), so the attacker can construct the correct fake `CoinState` objects. No key material, admin access, or cryptographic break is required.

---

### Recommendation

1. **Apply the check-effects-interactions pattern**: update the trade status *before* making any external peer call, or avoid making external calls inside the DB writer context for state-changing decisions.

2. **Validate the secondary `get_coin_state` response** the same way the primary coin-state path does — fetch the corresponding header block and verify Merkle inclusion/exclusion proofs before trusting the returned coin states.

3. **Alternatively**, derive the settlement-payment coin existence from the already-validated primary `coin_state` and its children (fetched via `request_children`, which is already used elsewhere), rather than issuing a new unvalidated query.

---

### Proof of Concept

1. Alice creates an offer (XCH ↔ CAT). The offer is posted publicly; the settlement-payment coin IDs are deterministic and known.
2. Alice's wallet connects to a malicious full node (attacker-controlled).
3. The attacker sends a `CoinStateUpdate` claiming one of Alice's offer input coins was spent at some height, with a valid (or minimally plausible) block reference sufficient to pass the primary validation gate.
4. `_add_coin_states` detects the coin is in `trade_removals` and calls `coins_of_interest_farmed`.
5. Inside `coins_of_interest_farmed`, the wallet issues `get_coin_state(our_addition_ids, peer=attacker_peer)`.
6. The attacker returns a list of `CoinState` objects whose `.coin.name()` values exactly match `our_addition_ids` (constructed from the public offer data).
7. The check `set(our_addition_ids) == set(coin_state_names)` passes.
8. `trade_store.set_status(trade_id, TradeStatus.CONFIRMED)` is written to the DB.
9. Confirmed `INCOMING_TRADE` transaction records are added; Alice's wallet displays a balance she never received. [9](#0-8)
