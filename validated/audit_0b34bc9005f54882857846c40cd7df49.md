Audit Report

## Title
Wrong Guard Condition for ETH Deposit Limit Allows Limit to Be Exceeded - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch comparison, checking only whether the current total already exceeds the cap rather than whether the deposit *would* exceed it. Any unprivileged caller can push the ETH deposit total above the configured limit by up to one full deposit in a single transaction.

## Finding Description
The function at [1](#0-0)  contains two branches. The ERC-20 branch correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`, but the ETH branch at line 679 evaluates only `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)`, silently dropping `amount`. This means the guard only triggers when the cap is *already* breached, not when the incoming deposit *would* breach it.

The guard is called unconditionally from `_beforeDeposit` at [2](#0-1) , which is invoked by `depositETH` at [3](#0-2) . When the check passes incorrectly, `_mintRsETH` is called for the full `msg.value`, minting rsETH against ETH that exceeds the governance-configured cap.

No existing check compensates for this omission. The `onlySupportedAsset` modifier only verifies ETH is a supported asset, not that the limit is respected.

## Impact Explanation
The ETH deposit cap is the protocol's primary mechanism for bounding total ETH exposure. Because the cap is bypassable, the protocol accepts and mints rsETH against more ETH than governance has authorised. This matches the allowed impact: **Low â€” Contract fails to deliver promised returns, but doesn't lose value** (deposit limit not enforced for ETH).

## Likelihood Explanation
The entry point `depositETH` is fully permissionless â€” any address can call it with arbitrary `msg.value`. No special role, front-running, or external dependency is required. The vulnerable condition (`totalAssetDeposits < cap`) is the normal operating state of the protocol for most of its lifetime, making this trivially and repeatedly triggerable.

## Recommendation
Remove the ETH special-case entirely and apply the same guard used for ERC-20 tokens:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Assume `depositLimitByAsset[ETH_TOKEN] = 100 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 99 ether`.

1. Attacker calls `depositETH{value: 10 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)`.
3. ETH branch evaluates `99 ether > 100 ether` â†’ `false` â†’ no revert.
4. `_mintRsETH` mints rsETH for the full 10 ETH; total ETH deposits become 109 ETH â€” 9 ETH above the cap.

**Foundry test sketch:**
```solidity
function test_ethDepositExceedsLimit() public {
    // set limit to 100 ETH, deposit 99 ETH first
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 100 ether);
    depositPool.depositETH{value: 99 ether}(0, "");
    // next deposit of 10 ETH should revert but does not
    depositPool.depositETH{value: 10 ether}(0, ""); // passes incorrectly
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 100 ether);
}
```
