Audit Report

## Title
`rsETHPrice` Uninitialized to Zero Causes Division-by-Zero Revert Blocking All Deposits - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTOracle.rsETHPrice` is a `uint256` storage variable that defaults to `0` and is never written during `initialize()`. `LRTDepositPool.getRsETHAmountToMint()` unconditionally divides by `lrtOracle.rsETHPrice()`, so every deposit attempt before the first successful `updateRSETHPrice()` call reverts with a division-by-zero panic. No funds are lost, but the contract cannot accept any deposits during this window.

## Finding Description
`LRTOracle` declares `rsETHPrice` as a plain storage variable with no initial value:

```solidity
// contracts/LRTOracle.sol L28
uint256 public override rsETHPrice;   // Solidity default: 0
```

`initialize()` only sets `lrtConfig` and emits an event — `rsETHPrice` is never touched:

```solidity
// contracts/LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

`rsETHPrice` is only written inside `_updateRsETHPrice()`, reached via the public `updateRSETHPrice()`. Until that call is made, `rsETHPrice == 0`.

Every deposit path calls `_beforeDeposit()` → `getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

There is no zero-check on `lrtOracle.rsETHPrice()` before the division. When `rsETHPrice == 0`, the EVM triggers an unconditional division-by-zero panic (Solidity 0.8.x), reverting the entire transaction. This affects both `depositETH()` (L76-93) and `depositAsset()` (L99-118), which both route through `_beforeDeposit()` (L648-670) → `getRsETHAmountToMint()` (L506-521).

`updateRSETHPrice()` does handle the bootstrap case — when `rsethSupply == 0` it sets `rsETHPrice = 1 ether` (L218-222) — but there is no on-chain enforcement that this call precedes the first deposit attempt.

## Impact Explanation
All user deposits (`depositETH`, `depositAsset`) revert with a division-by-zero panic during the window between contract deployment/initialization and the first successful `updateRSETHPrice()` call. No user funds are lost (reverts are atomic), but the contract's core function — accepting deposits — is completely unavailable during this window. This matches the allowed impact: **Low — contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`updateRSETHPrice()` is public and permissionless (any EOA can call it), and when `rsethSupply == 0` it always succeeds and sets `rsETHPrice = 1 ether`. In practice the vulnerable window is short. However, there is no on-chain guard preventing a deposit from being submitted before the first price update — a race condition exists between deployment and the first `updateRSETHPrice()` call. Any user who attempts a deposit in this window will have their transaction revert. Likelihood: **Low**.

## Recommendation
Initialize `rsETHPrice` (and `highestRsethPrice`) to `1 ether` inside `initialize()`, mirroring the bootstrap logic already present in `_updateRsETHPrice()`:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    rsETHPrice = 1 ether;          // add this
    highestRsethPrice = 1 ether;   // add this
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

Alternatively, add an explicit zero-check in `getRsETHAmountToMint()` that reverts with a descriptive custom error (e.g., `RsETHPriceNotInitialized()`) instead of silently panicking.

## Proof of Concept
1. Deploy `LRTConfig`, `LRTOracle.initialize(lrtConfigAddr)`, `LRTDepositPool.initialize(lrtConfigAddr)`.
2. Confirm `LRTOracle.rsETHPrice()` returns `0` — no `updateRSETHPrice()` has been called.
3. Call `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `getRsETHAmountToMint()` at `LRTDepositPool.sol` L520: `(1e18 * assetPrice) / 0` → EVM division-by-zero panic → revert.
5. Deposit fails. Same result for `depositAsset()`.
6. Call `LRTOracle.updateRSETHPrice()` (public, no role required). Because `rsethSupply == 0`, it sets `rsETHPrice = 1 ether` and returns.
7. Repeat step 3 — deposit now succeeds.

Foundry test sketch:
```solidity
function test_depositRevertsBeforePriceUpdate() public {
    // deploy and initialize contracts, skip updateRSETHPrice()
    vm.expectRevert(); // division-by-zero panic
    depositPool.depositETH{value: 1 ether}(0, "");

    oracle.updateRSETHPrice(); // rsethSupply == 0 → sets price to 1 ether
    depositPool.depositETH{value: 1 ether}(0, ""); // succeeds
}
```
