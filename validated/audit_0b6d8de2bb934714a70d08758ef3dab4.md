Audit Report

## Title
Missing Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Incorrect rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `answer`, performing no staleness, completeness, or validity checks. This stale price flows directly into `LRTDepositPool.getRsETHAmountToMint()`, which uses it to compute rsETH minting amounts. A stale price that is higher than the true current price (e.g., a feed frozen before an LST depeg) allows a depositor to receive more rsETH than the deposited collateral is worth, diluting all existing rsETH holders and trending toward protocol insolvency.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` discards all `latestRoundData()` return values except `price`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is performed on `updatedAt` vs `block.timestamp`, `answeredInRound < roundId`, `updatedAt == 0`, or `price <= 0`.

This is directly contradicted by `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` (lines 30–32), which implements `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0` checks on the same Chainlink interface, confirming the protocol is aware of the requirement.

The stale price propagates as follows:
1. `depositAsset()` → `_beforeDeposit()` (`LRTDepositPool.sol` L665) → `getRsETHAmountToMint()` (L520)
2. `getRsETHAmountToMint`: `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`
3. `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice(asset)` returns the unchecked stale price

**Correct attack direction (note: the submitted PoC has the direction inverted):** The profitable exploit requires a stale price that is *higher* than the true current price — e.g., a Chainlink LST/ETH feed freezes at `1.0005e18` while the asset has since depegged to `0.9995e18`. With `rsETHPrice` reflecting the true backing, the attacker deposits depegged LST and receives rsETH computed at the inflated stale price, extracting more value than deposited.

## Impact Explanation
**Protocol insolvency (Critical):** When a Chainlink feed freezes at a price above the true current value (e.g., an LST depeg event), any depositor can call `depositAsset()` permissionlessly and receive rsETH minted at the stale-high price. Each such deposit reduces the per-rsETH backing, diluting all existing holders. Repeated exploitation trends toward insolvency. The secondary direction (stale-low price) causes under-minting, matching the Low impact "contract fails to deliver promised returns."

## Likelihood Explanation
Chainlink LST/ETH feeds (e.g., stETH/ETH) commonly use 24-hour heartbeat intervals with a 0.5% deviation threshold. During an LST depeg event, the price may move beyond the deviation band but the feed can still lag or stall due to node issues or network congestion. Any unprivileged depositor can trigger `depositAsset()` at any time; no special role or precondition beyond a stale feed is required. This scenario has occurred historically (e.g., stETH depeg in 2022).

## Recommendation
Apply the same staleness checks already present in `ChainlinkOracleForRSETHPoolCollateral` and add a time-based threshold in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
if (block.timestamp - updatedAt > STALE_PRICE_THRESHOLD) revert StalePrice();
```

`STALE_PRICE_THRESHOLD` should be set per feed based on its documented heartbeat (e.g., 3600 seconds for 1-hour feeds, 86400 seconds for 24-hour feeds), with a small buffer.

## Proof of Concept
1. Chainlink stETH/ETH feed freezes at `1.0005e18` (last `updatedAt` is 25+ hours ago on a 24-hour heartbeat feed). True stETH price has moved to `0.9995e18` due to a depeg.
2. `rsETHPrice` (computed from `LRTOracle.getRSETHPrice()`) reflects the true backing and is approximately `1.0000e18`.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint(stETH, 1000e18)` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `1.0005e18` (stale).
5. `rsethAmountToMint = (1000e18 * 1.0005e18) / 1.0000e18 = 1000.5e18` rsETH.
6. True value deposited: `1000 stETH * 0.9995 ETH/stETH = 999.5 ETH`. Attacker receives rsETH worth `1000.5 ETH` at current rsETH price — a `1 ETH` extraction per `1000 stETH` deposited.
7. Attacker repeats until the depeg resolves or the pool is drained, with each iteration further diluting rsETH backing.

**Foundry fork test plan:** Fork mainnet at a block where the stETH/ETH Chainlink feed's `updatedAt` is stale. Deploy a mock `AggregatorV3Interface` returning a frozen high price. Set it as the price feed via `updatePriceFeedFor`. Call `depositAsset` and assert `rsethAmountToMint > (depositAmount * truePriceRatio / rsETHPrice)`.
