Audit Report

## Title
Stale `rsETHPrice` Denominator Against Live `swETH.getRate()` Numerator Enables Excess rsETH Minting — (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`, `contracts/oracles/SwETHPriceOracle.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint()` computes the mint amount by dividing a live, on-chain `swETH.getRate()` value by a stored `rsETHPrice` that is only refreshed on explicit calls to `updateRSETHPrice()`. Any gap between price updates — widened by block stuffing — lets a depositor receive more rsETH than the true backing ratio warrants, diluting all existing holders.

## Finding Description

The mint formula at `LRTDepositPool.sol` line 520 is:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

- **Numerator:** `lrtOracle.getAssetPrice(swETH)` routes through `LRTOracle.getAssetPrice()` → `SwETHPriceOracle.getAssetPrice()` → `ISwETH(swETHAddress).getRate()`, which is fetched live from the swETH contract on every call (`SwETHPriceOracle.sol` line 39).
- **Denominator:** `lrtOracle.rsETHPrice()` is a plain storage variable (`LRTOracle.sol` line 28) written only inside `_updateRsETHPrice()` (`LRTOracle.sol` line 313), which is triggered solely by the public `updateRSETHPrice()` or the manager-gated `updateRSETHPriceAsManager()`. Neither `depositAsset()` nor `depositETH()` calls `updateRSETHPrice()` before computing the mint amount.

Because swETH accrues staking rewards continuously, `swETH.getRate()` rises over time while `rsETHPrice` stays frozen at the value from the last explicit update. An attacker who stuffs blocks to prevent `updateRSETHPrice()` from being included widens this gap. After stuffing, the attacker calls `depositAsset(swETH, largeAmount, 0, "")` and receives:

```
rsethMinted = largeAmount × liveSwETHRate / staleRsETHPrice
            > largeAmount × liveSwETHRate / correctRsETHPrice
```

The `pricePercentageLimit` guard (`LRTOracle.sol` lines 260–265) compounds the issue: if the accumulated price increase exceeds the configured threshold, the public `updateRSETHPrice()` reverts for non-managers, meaning the stale price persists even after stuffing ends until a manager intervenes. Every `depositAsset(swETH)` call during this window continues to mint at the stale-low denominator.

## Impact Explanation

The attacker receives excess rsETH tokens. Because rsETH is a share token backed by a fixed pool of assets, minting more shares than the deposit warrants dilutes every existing holder's proportional claim on the underlying ETH. This maps to **Low — Block stuffing**: the attacker pays block-stuffing costs to achieve a profitable mint at a stale price, extracting value from existing holders proportional to `(correctRsETHPrice − staleRsETHPrice) / correctRsETHPrice × depositAmount`.

## Likelihood Explanation

- The structural precondition (live numerator, stored denominator, no atomic update in `depositAsset`) is always present; no special protocol state is required.
- Block stuffing on Ethereum mainnet is expensive but economically rational for large deposit amounts. swETH accrues ~4–5% APR; even a few hours of stuffed blocks produces a measurable rate delta.
- The `pricePercentageLimit` guard paradoxically extends the exploitable window by preventing public price correction after stuffing ends, requiring only a single large deposit to extract the accumulated delta.
- No privileged access is required; any external account can call `depositAsset()`.

## Recommendation

1. **Atomically refresh the price before minting:** Call `_updateRsETHPrice()` (or an equivalent inline computation) at the start of `depositAsset()` and `depositETH()` so the denominator is always current at the moment of minting.
2. **Alternatively, compute the price inline:** Replace the stored `rsETHPrice` read with a freshly computed value using `_getTotalEthInProtocol() / rsethSupply` at mint time.
3. **If on-demand updates are too gas-expensive:** Add a maximum staleness check — revert if `rsETHPrice` was last updated more than N blocks (or seconds) ago — to bound the exploitable window.

## Proof of Concept

```solidity
// Foundry fork test outline
// 1. Deploy a mock swETH whose getRate() returns baseRate + blockNumber * ratePerBlock.
// 2. Record staleRsETHPrice = lrtOracle.rsETHPrice()  (set at last updateRSETHPrice call).
// 3. vm.roll(block.number + N);  // simulate N blocks without updateRSETHPrice
// 4. liveRate = mockSwETH.getRate();  // now higher than at step 2
// 5. attacker calls depositAsset(swETH, AMOUNT, 0, "")
// 6. rsethMinted = lrtDepositPool.getRsETHAmountToMint(swETH, AMOUNT)
//    = AMOUNT * liveRate / staleRsETHPrice
// 7. correctRsETHPrice = _getTotalEthInProtocol() / rsethSupply  (computed with live rates)
//    correctMint = AMOUNT * liveRate / correctRsETHPrice
// 8. assert(rsethMinted > correctMint);  // attacker received excess rsETH
```

The assertion at step 8 holds for any `N > 0` where `swETH.getRate()` has increased since the last `updateRSETHPrice()` call, confirming the invariant break without requiring any privileged access.
