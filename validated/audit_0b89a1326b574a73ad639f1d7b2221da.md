Audit Report

## Title
Unbounded `merkleProof` Array in `claim()` Enables Unbounded Gas Consumption and Block Stuffing — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary
`MerkleDistributor.claim()` accepts a `bytes32[] calldata merkleProof` with no length bound. An unprivileged attacker can submit transactions with arbitrarily large proof arrays that pass all cheap pre-checks before reaching `MerkleProofUpgradeable.processProof()`, which iterates over every element unconditionally. The transaction reverts with `InvalidMerkleProof` after exhausting the gas budget, allowing an attacker to monopolize block space and temporarily deny legitimate claimants access to their tokens.

## Finding Description
The `claim()` function at lines 97–147 of `MerkleDistributor.sol` accepts an unbounded `bytes32[] calldata merkleProof`. Before reaching proof verification, it performs only three cheap checks: (1) `currentMerkleRoot != bytes32(0)` — one SLOAD; (2) `index != 0 && index <= currentIndex` — one SLOAD, satisfied with `index = 1` after any `setMerkleRoot` call; (3) `!isClaimed(index, account)` — one SLOAD, satisfied with any fresh address since `userClaims[account].lastClaimedIndex` defaults to 0.

After passing these checks, execution reaches line 121:
```solidity
if (!MerkleProofUpgradeable.verify(merkleProof, currentMerkleRoot, node)) {
    revert InvalidMerkleProof();
}
```

`verify()` delegates to `processProof()` in `MerkleProofUpgradeable.sol` at lines 48–54, which iterates over every element with no size guard:
```solidity
for (uint256 i = 0; i < proof.length; i++) {
    computedHash = _hashPair(computedHash, proof[i]);
}
```

Each `_hashPair` call executes a `keccak256` via inline assembly. Calldata costs (~16 gas per non-zero byte × 32 bytes × N elements) plus hashing costs (~30 gas per element) scale linearly with proof length. A proof of ~18,000 elements can approach the 30M block gas limit in a single transaction. The transaction always reverts (the junk proof never matches `currentMerkleRoot`), but the full gas budget is consumed before the revert. No existing guard in `claim()` limits proof length.

The `account` parameter is not restricted to `msg.sender`, so the attacker does not need to own any valid claim — any unclaimed address satisfies the `isClaimed` check.

## Impact Explanation
**Medium. Unbounded gas consumption / Low. Block stuffing.** A small number of oversized-proof transactions per block can consume the entire block gas limit, crowding out all legitimate `claim()` transactions. Claimants cannot retrieve their tokens for as long as the attack is sustained. Funds are not permanently lost, but access is temporarily denied. Both "Medium. Unbounded gas consumption" and "Low. Block stuffing" are explicitly within the allowed impact scope for this program.

## Likelihood Explanation
**Medium.** The attacker requires no privileged access, no valid Merkle leaf knowledge, and no leaked credentials. The only inputs needed are a valid `index` value (publicly readable from `currentIndex`) and any unclaimed address. The economic cost is real (attacker pays gas for each stuffing transaction), but the attack is mechanically trivial and repeatable with a simple script. The attack can be sustained as long as the attacker is willing to pay gas costs.

## Recommendation
Add a maximum proof length check at the top of `claim()`, sized to the maximum expected tree depth (e.g., 32 for a tree of up to 2³² leaves):

```solidity
uint256 public constant MAX_PROOF_LENGTH = 32;

function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
    if (merkleProof.length > MAX_PROOF_LENGTH) revert InvalidMerkleProof();
    // ... existing logic
}
```

This eliminates the unbounded gas path entirely and mirrors the exact-length enforcement pattern already present in `BeaconChainProofs.sol`.

## Proof of Concept
Deploy `MerkleDistributor`, call `setMerkleRoot(bytes32(uint256(1)))` to set `currentIndex = 1`, then submit:

```solidity
bytes32[] memory bigProof = new bytes32[](500);
for (uint256 i = 0; i < 500; i++) { bigProof[i] = bytes32(i + 1); }
uint256 gasBefore = gasleft();
try distributor.claim(1, address(this), 1, bigProof) {} catch {}
uint256 gasUsed = gasBefore - gasleft();
assertGt(gasUsed, 100_000);
```

The transaction passes all three pre-checks (root is set, `index = 1 <= currentIndex = 1`, `address(this)` is unclaimed), then `processProof()` iterates all 500 elements before reverting. Scaling to ~18,000 elements approaches the 30M block gas limit. Multiple such transactions per block exclude all legitimate claimants.
