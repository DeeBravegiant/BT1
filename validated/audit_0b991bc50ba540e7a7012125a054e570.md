Audit Report

## Title
Zero `rsETHAmount` Output Deposit Proceeds Without Revert, Permanently Locking User Funds - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All L2 pool deposit functions guard against `amount == 0` but never check whether the computed `rsETHAmount` is zero after integer-division truncation in `viewSwapRsETHAmountAndFee`. When `amountAfterFee * 1e18 < rsETHToETHrate` (ETH path) or `amountAfterFee * tokenToETHRate < rsETHToETHrate` (token path), Solidity truncates `rsETHAmount` to zero. The contract then accepts the user's ETH or tokens, mints/transfers 0 wrsETH/rsETH to the user, and emits a misleading `SwapOccurred` event — the deposited assets are permanently locked in the pool with no depositor-facing recovery path.

## Finding Description
Every pool deposit function follows the same pattern. In `RSETHPoolV3.sol`, the ETH path:

```solidity
// RSETHPoolV3.sol L256-264
if (amount == 0) revert InvalidAmount();   // only guards zero input
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);      // mints 0 — no-op
emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
```

The rate conversion:

```solidity
// RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // truncates to 0
}
```

Since `rsETHToETHrate` is always slightly above `1e18` (rsETH appreciates over time), a 1-wei ETH deposit yields `1 * 1e18 / rsETHToETHrate = 0`. For the token path (`RSETHPoolV3.sol` L324-334), `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` truncates to zero for any deposit where `amountAfterFee * tokenToETHRate < rsETHToETHrate`. There is no subsequent guard of the form `if (rsETHAmount == 0) revert InvalidAmount()` in any pool.

The identical pattern is confirmed in:
- `RSETHPoolV2.sol` L207-219, L225-234
- `RSETHPoolV3ExternalBridge.sol` L366-384, L401-412, L418-427, L442-453
- `RSETHPoolV3WithNativeChainBridge.sol` L282-301, L307-329, L335-344, L360-371
- `RSETHPoolNoWrapper.sol` L231-244, L250-271, L277-286, L301-311

For `RSETHPoolNoWrapper`, the ETH path calls `rsETH.safeTransfer(msg.sender, 0)` instead of minting; whether this reverts depends on the rsETH token implementation, but the token deposit path (`safeTransferFrom` before the calculation) is unambiguously affected.

## Impact Explanation
**Low — Contract fails to deliver promised returns.**

The deposited ETH or tokens are accepted by the pool and permanently inaccessible to the depositor: there is no depositor-facing withdrawal function to recover a deposit for which 0 shares were issued. `wrsETH.mint(user, 0)` is a no-op; no shares are recorded. The user receives nothing in exchange for their deposit. The impact is Low because the affected amounts are dust (1 wei for ETH; proportionally larger for low-value tokens), and no privileged action is required to trigger it.

## Likelihood Explanation
Any unprivileged depositor can trigger this without special permissions or oracle manipulation. For the ETH path, a 1-wei deposit always satisfies the truncation condition because `rsETHToETHrate > 1e18`. For the token path, the zero-output threshold scales as `rsETHToETHrate / tokenToETHRate`; for a token with `tokenToETHRate = 1e15` (worth 0.001 ETH), any deposit under 1050 token-wei produces `rsETHAmount = 0`. UI rounding errors or automated scripts sending dust amounts can reach this path in normal operation.

## Recommendation
Add a zero-output guard immediately after computing `rsETHAmount` in every pool deposit function, before any state change:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This must be applied to both the ETH and token deposit paths in all five pool contracts.

## Proof of Concept
**Setup:** `RSETHPoolV3` deployed with `feeBps = 0`, `rsETHToETHrate = 1.05e18`.

1. User calls `deposit{value: 1}("")` (1 wei ETH).
2. `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * 0 / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
3. `feeEarnedInETH += 0`; 1 wei is now in the pool.
4. `wrsETH.mint(msg.sender, 0)` — no-op, user receives 0 wrsETH.
5. `SwapOccurred(user, 0, 0, "")` emitted.
6. User's 1 wei is permanently locked; no depositor withdrawal path exists.

**Token path:** With `tokenToETHRate = 1e15` and `rsETHToETHrate = 1.05e18`, any deposit of `amount < 1050` token-wei triggers `rsETHAmount = 0` while `safeTransferFrom` has already pulled the tokens from the user's wallet.

**Foundry fuzz test plan:**
```solidity
function testFuzz_zeroRsETHAmountDeposit(uint256 amount) public {
    vm.assume(amount > 0 && amount * 1e18 < rsETHToETHrate);
    vm.deal(user, amount);
    vm.prank(user);
    pool.deposit{value: amount}("");
    assertEq(wrsETH.balanceOf(user), 0);   // user got nothing
    assertEq(address(pool).balance, amount); // pool kept ETH
}
```
