Audit Report

## Title
ETH Push-Transfer Revert Permanently Freezes User Funds in FIFO Queue - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`_processWithdrawalCompletion` pops the front of a user's FIFO deque, deletes the request, and decrements `unlockedWithdrawalsCount` before calling `_transferAsset`. If the ETH push to a contract recipient reverts, Solidity atomically undoes all state changes, leaving the queue pointer stuck. Every future completion attempt for that user hits the same frozen entry first, permanently blocking all of that user's queued ETH withdrawals. The user's rsETH was already burned during `unlockQueue`, making the loss unrecoverable.

## Finding Description
`_processWithdrawalCompletion` (L699–738) executes in this order:

1. `userAssociatedNonces[asset][user].popFront()` — L705, removes the oldest nonce from the FIFO deque.
2. `delete withdrawalRequests[requestId]` — L712, erases the request record.
3. `unlockedWithdrawalsCount[asset]--` — L717, decrements the unlocked counter.
4. `_transferAsset(asset, user, request.expectedAssetAmount)` — L734, pushes ETH.

`_transferAsset` (L876–883) performs an uncapped low-level call:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract whose `receive()` reverts (or returns `false`), `EthTransferFailed` is thrown. Solidity reverts the entire call frame, undoing steps 1–3. The deque pointer is restored to the same stuck entry. Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` for that user re-presents the identical entry and reverts again. The NatSpec on `completeWithdrawalForUser` (L191) acknowledges ETH gas-grief but incorrectly dismisses it as non-impactful; a hard revert is categorically different from a gas-grief. `sweepRemainingAssets` (L395–414) is gated on `!hasUnlockedWithdrawals(asset)`, which checks `unlockedWithdrawalsCount[asset] > 0`; because the stuck request was already counted as unlocked in `unlockQueue` and the decrement is perpetually reverted, this counter never reaches zero, permanently blocking the sweep path as well.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is burned irreversibly during `unlockQueue`. The corresponding ETH sits in `LRTWithdrawalManager` indefinitely with no admin rescue path: `completeWithdrawal` and `completeWithdrawalForUser` both revert, and `sweepRemainingAssets` is blocked by the inflated `unlockedWithdrawalsCount`. All other ETH withdrawal requests queued by the same user are also frozen because the FIFO queue always presents the stuck entry first.

## Likelihood Explanation
Smart-contract wallets (Gnosis Safe, Argent, ERC-4337 accounts) are standard DeFi users. A wallet loses ETH-receive capability when: all signers lose access, the implementation is upgraded to a version without `receive()`, or `receive()` is intentionally or accidentally made to revert. No privileged role, oracle manipulation, or external protocol compromise is required. The attacker is the victim themselves — no adversarial action is needed; the freeze is triggered by the normal completion flow after the wallet becomes unable to receive ETH.

## Recommendation
Replace the push-payment pattern with a pull-payment pattern: instead of calling `_transferAsset` inside `_processWithdrawalCompletion`, credit the owed amount to a `mapping(address => uint256) claimableETH` and expose a separate `claimETH()` function. Alternatively, wrap the `_transferAsset` call in a `try`/`catch` (or check `sent` without reverting) and, on failure, credit the amount to a claimable balance rather than reverting the entire completion.

## Proof of Concept
1. Deploy `VictimWallet` — a proxy contract with a `receive()` that initially succeeds.
2. From `VictimWallet`, call `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; a withdrawal request is enqueued.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. `_unlockWithdrawalRequests` advances `nextLockedNonce`, sets `request.expectedAssetAmount`, increments `unlockedWithdrawalsCount[ETH]`, and burns the rsETH. ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`.
4. Upgrade `VictimWallet`'s implementation so `receive()` reverts (simulating key loss, wallet upgrade, or intentional bricking).
5. Call `completeWithdrawal(ETH_TOKEN, "")` from `VictimWallet` — reverts with `EthTransferFailed` (L879). State is fully rolled back; deque pointer unchanged.
6. Operator calls `completeWithdrawalForUser(ETH_TOKEN, VictimWallet, "")` — identical revert.
7. Assert: `userAssociatedNonces[ETH][VictimWallet].front()` still returns the original nonce; `unlockedWithdrawalsCount[ETH]` is still > 0; `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist`; `VictimWallet`'s ETH balance is 0 and rsETH balance is 0.
