Audit Report

## Title
Precision loss in `rewardPerToken()` combined with unconditional `updatedAt` advancement permanently destroys unclaimed yield - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`rewardPerToken()` performs integer division that truncates to zero when `rewardRate * timeDelta * 1e18 < totalKernelStaked`. The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` regardless of whether `rewardPerToken()` changed, permanently discarding the elapsed time interval. Because `getReward()` is permissionless and requires no stake, any external caller can repeatedly trigger this modifier within the truncation window, silently destroying the entire reward budget over a full distribution period.

## Finding Description

`rewardPerToken()` computes:

```solidity
// L412-413
return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

When `rewardRate * timeDelta * 1e18 < totalKernelStaked`, Solidity integer division truncates the increment to zero, leaving `rewardPerTokenStored` unchanged.

The `updateReward` modifier at L232-242 unconditionally executes:

```solidity
rewardPerTokenStored = rewardPerToken();   // unchanged (truncated to 0 increment)
updatedAt = lastTimeRewardApplicable();    // advances regardless
```

`updatedAt` advances even when no reward was recorded. The consumed time interval is permanently unrecoverable — `rewardPerToken()` will never retroactively account for it.

`getReward()` at L382-390 is `external` with no stake requirement and no access control beyond `nonReentrant`. An attacker with zero KERNEL staked calls it within the truncation window. The modifier fires, `updatedAt` advances, `rewardPerTokenStored` stays the same. Repeating this every few seconds across the full reward period results in `rewardPerTokenStored` remaining at zero throughout, leaving all reward tokens permanently locked in the contract.

Concrete truncation window with USDT (6 decimals):
- `rewardRate = 100e6 / 604800 ≈ 165`
- `totalKernelStaked = 1_000e18`
- Truncation when: `165 * timeDelta * 1e18 < 1_000e18` → `timeDelta < ~6 seconds`

On L2 chains (Arbitrum, Base, Optimism) with 0.25–2 second block times and negligible gas, a bot can sustain this attack trivially. The attack requires zero capital.

Existing guards are insufficient: `nonReentrant` prevents same-transaction reentrancy but not repeated calls across blocks. There is no minimum-stake check on `getReward()`, no minimum elapsed-time guard before advancing `updatedAt`, and no check that `rewardPerToken()` actually increased before committing the timestamp update.

## Impact Explanation

**Medium. Permanent freezing of unclaimed yield.**

The attacker does not receive the destroyed rewards — they remain locked in the contract with no recovery mechanism. All stakers earn zero rewards for every interval consumed by the attack. Over a full 7-day reward period with calls every 5 seconds, `rewardPerTokenStored` remains at zero and the entire reward budget (e.g., 100 USDT) is permanently frozen in the contract. This matches the allowed impact class "Medium. Permanent freezing of unclaimed yield."

## Likelihood Explanation

Likelihood is Medium-High when `rewardsToken` has low decimals (USDT/USDC, 6 decimals). The contract accepts any ERC-20 as `rewardsToken` at initialization with no decimal floor. The attacker needs no KERNEL stake, no privileged role, and only needs to submit one transaction per truncation window (~6 seconds in the USDT example). L2 deployment makes bot-driven exploitation cheap and reliable. For 18-decimal reward tokens the truncation window collapses to sub-nanosecond and the attack is not practical, so exploitability is conditional on the configured `rewardsToken`.

## Recommendation

1. **Only advance `updatedAt` when `rewardPerToken()` strictly increased:**

```solidity
modifier updateReward(address _account) {
    uint256 newRewardPerToken = rewardPerToken();
    if (newRewardPerToken > rewardPerTokenStored) {
        rewardPerTokenStored = newRewardPerToken;
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

2. **Scale `rewardRate` by `DECIMAL_PRECISION` at storage time** in `notifyRewardAmount` so the numerator is always large enough to avoid truncation for any reasonable `timeDelta`.

3. **Restrict `getReward()` to callers with `balanceOf[msg.sender] > 0`**, eliminating the zero-stake attack vector entirely.
