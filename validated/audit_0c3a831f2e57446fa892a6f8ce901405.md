Audit Report

## Title
Lack of Slippage Control on `MetricOmmPool.removeLiquidity` - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`removeLiquidity` burns LP shares and transfers underlying tokens computed from live bin balances with no minimum-amount-out parameters. The `msg.sender != owner` guard prevents any periphery wrapper from providing slippage protection, so LPs must call the pool directly and are fully exposed to sandwich attacks that can drain token composition from their bins before withdrawal executes.

## Finding Description
The function signature accepts only shares to burn and opaque extension bytes, returning `(amount0Removed, amount1Removed)` with no caller-specified minimums:

```solidity
// metric-core/contracts/MetricOmmPool.sol L199-212
function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
    external
    nonReentrant(PoolActions.REMOVE_LIQUIDITY)
    returns (uint256 amount0Removed, uint256 amount1Removed)
{
    ...
    if (msg.sender != owner) revert NotPositionOwner();
    ...
}
```

The amounts are computed from live bin state in `LiquidityLib.removeLiquidity`:

```solidity
// metric-core/contracts/libraries/LiquidityLib.sol L205-206
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

`token0BalanceScaled` and `token1BalanceScaled` are mutated by every swap crossing the bin. The `msg.sender != owner` check is the critical blocker: a periphery wrapper calling `pool.removeLiquidity(lpAddress, ...)` would have `msg.sender == wrapper != lpAddress`, causing a `NotPositionOwner` revert. This makes it structurally impossible to add a protective wrapper without a core contract change.

By contrast, `addLiquidity` is protected by `MetricOmmPoolLiquidityAdder` with `maxAmountToken0`/`maxAmountToken1` caps, and swaps are protected by `MetricOmmSimpleRouter` with `amountOutMinimum`. No analogous periphery contract exists for `removeLiquidity` — confirmed by the absence of any `removeLiquidity` call in `metric-periphery/`.

## Impact Explanation
An LP submitting `removeLiquidity` has no on-chain mechanism to enforce minimum token receipts. A sandwich attacker can front-run with a large swap draining token0 from the target bin, allow the LP's removal to execute receiving near-zero token0, then back-run to restore price. The LP suffers a direct loss of principal proportional to the manipulating swap size relative to bin depth. This constitutes a medium-severity direct loss of LP assets.

## Likelihood Explanation
Every LP calling `removeLiquidity` directly on the pool is exposed. No privileged access is required — only a standard sandwich setup. The oracle-driven bid/ask spread means bin composition can shift legitimately between blocks, making even non-adversarial slippage a routine risk in volatile markets. The attack is repeatable against any LP withdrawal.

## Recommendation
Add `minAmount0Removed` and `minAmount1Removed` parameters to `removeLiquidity` and revert if computed amounts fall below them:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Removed,
    uint256 minAmount1Removed,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (uint256 amount0Removed, uint256 amount1Removed) {
    // ... existing logic ...
    if (amount0Removed < minAmount0Removed || amount1Removed < minAmount1Removed) {
        revert InsufficientOutput(amount0Removed, amount1Removed, minAmount0Removed, minAmount1Removed);
    }
}
```

Alternatively, relax the `msg.sender != owner` check to allow approved periphery routers to call on behalf of the owner, enabling a protective wrapper analogous to `MetricOmmPoolLiquidityAdder`.

## Proof of Concept
1. LP holds 10,000 shares in bin 0 holding 50 token0 and 50 token1 scaled.
2. LP submits `removeLiquidity` for all 10,000 shares expecting ~50 token0 and ~50 token1.
3. Attacker front-runs with a large `zeroForOne` swap draining all token0 from bin 0; bin 0 now holds 0 token0 and 100 token1 scaled.
4. LP's `removeLiquidity` executes: `amount0Scaled = 0 * 10000 / 10000 = 0`, `amount1Scaled = 100 * 10000 / 10000 = 100`. LP receives 0 token0.
5. Attacker back-runs to restore price.

LP lost ~50 token0 of principal. Reproducible as a Foundry fork test by sandwiching a `removeLiquidity` call with opposing swaps on the same bin.
