Audit Report

## Title
Missing Staleness Check in `CrossChainRateReceiver.getRate()` Enables Excess wrsETH Minting at Stale Rate — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver` tracks `lastUpdated` but never uses it in `getRate()`, which unconditionally returns the cached `rate`. When the LayerZero delivery pipeline stalls, the stored rate becomes stale while the true L1 rsETH/ETH rate appreciates. Any depositor calling `RSETHPoolV3.deposit()` during the stale window receives more `wrsETH` than the current L1 rate justifies, diluting the yield already accrued by existing holders.

## Finding Description

`CrossChainRateReceiver.getRate()` returns `rate` with no freshness guard:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
``` [1](#0-0) 

The contract does record when the rate was last set via `lzReceive()`:

```solidity
rate = _rate;
lastUpdated = block.timestamp;
``` [2](#0-1) 

But `lastUpdated` is never read in `getRate()` — there is no staleness guard anywhere in the contract or in its callers. [3](#0-2) 

`RSETHPoolV3.getRate()` delegates directly to whatever address is stored in `rsETHOracle` (on L2, a concrete `CrossChainRateReceiver` deployment): [4](#0-3) 

The stale rate flows into `viewSwapRsETHAmountAndFee()`:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [5](#0-4) 

And `deposit()` mints without any independent freshness check: [6](#0-5) 

`wrsETH.mint()` in `RsETHTokenWrapper` is gated only by `MINTER_ROLE`, which `RSETHPoolV3` holds — no additional rate validation occurs there: [7](#0-6) 

The only existing guard is `dailyMintLimit`, which caps per-day exposure but does not prevent the attack — it only bounds the magnitude per epoch: [8](#0-7) 

**Exploit path:**
1. LZ pipeline stalls (fee underpayment, congestion, or relayer failure); `lzReceive()` is not called.
2. True L1 rate appreciates from `r_stale` to `r_true > r_stale`.
3. Attacker observes `lastUpdated` is old (public state variable).
4. Attacker calls `RSETHPoolV3.deposit{value: X}()`.
5. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = X * 1e18 / r_stale`, which is larger than `X * 1e18 / r_true`.
6. The excess wrsETH minted represents yield already accrued to existing holders, now captured by the attacker.

## Impact Explanation

rsETH is a monotonically appreciating yield-bearing token. A stale rate `r_stale < r_true` causes the formula `rsETHAmount = ETH * 1e18 / r_stale` to over-mint wrsETH relative to the deposited ETH's true rsETH value. The over-minted wrsETH dilutes the pool's backing ratio, transferring accrued yield from existing holders to the late depositor. This is a direct, quantifiable transfer of unclaimed yield, matching **High — Theft of unclaimed yield**.

## Likelihood Explanation

LayerZero pipeline stalls are a known operational risk: the rate provider must pay relayer/oracle fees on every update, and LZ network congestion or fee underpayment can halt delivery for hours or days. No attacker action is required to cause the stall. The attacker only needs to observe that `lastUpdated` is old (a public state variable) and then call `deposit()`. The attack is permissionless, repeatable up to the daily mint cap, and requires no special privileges or capital beyond the deposit amount.

## Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes `deposit()` to revert during a stale window rather than silently minting at an incorrect rate, protecting existing holders.

## Proof of Concept

```solidity
// Foundry fork test
function test_staleRateMintExcess() public {
    // Initial rate: 1.05e18 (1 rsETH = 1.05 ETH)
    uint256 staleRate = 1.05e18;
    vm.prank(lzEndpoint);
    receiver.lzReceive(srcChainId, abi.encodePacked(rateProvider), 0,
                       abi.encode(staleRate));

    // Warp 7 days — LZ pipeline stalled, true L1 rate is now 1.10e18
    vm.warp(block.timestamp + 7 days);
    // lzReceive is NOT called — rate stays at 1.05e18

    uint256 depositAmount = 1 ether;
    uint256 trueRate = 1.10e18;

    vm.deal(attacker, depositAmount);
    vm.prank(attacker);
    pool.deposit{value: depositAmount}("ref");

    uint256 received = wrsETH.balanceOf(attacker);
    uint256 expectedAtTrueRate  = depositAmount * 1e18 / trueRate;  // ~0.909e18
    uint256 expectedAtStaleRate = depositAmount * 1e18 / staleRate; // ~0.952e18

    assertGt(received, expectedAtTrueRate);   // attacker received more than entitled
    assertEq(received, expectedAtStaleRate);  // minted at stale rate
    assertGt(received - expectedAtTrueRate, 0); // yield stolen > 0
}
```
