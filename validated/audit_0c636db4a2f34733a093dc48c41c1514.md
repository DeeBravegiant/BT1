Audit Report

## Title
Dust Withdrawal Request Spam Causes Unbounded Gas in `_unlockWithdrawalRequests` and Temporary Freezing of Legitimate User Funds - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`minRsEthAmountToWithdraw` defaults to `0` for every asset, allowing any amount ≥ 1 wei rsETH to pass the only guard in `initiateWithdrawal`. With no per-user cap on queued requests, an attacker can flood the FIFO withdrawal queue with thousands of dust entries. Because `nextLockedNonce` advances strictly in order, the operator's `unlockQueue` → `_unlockWithdrawalRequests` loop must exhaust all attacker nonces before any subsequent legitimate request can be unlocked, temporarily freezing legitimate users' rsETH and causing unbounded gas consumption per operator call.

## Finding Description
`initiateWithdrawal` enforces a single amount check:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

`minRsEthAmountToWithdraw` is a plain mapping never initialized in `initialize()`, so its Solidity default is `0`. The condition collapses to `rsETHUnstaked == 0`, accepting any amount ≥ 1 wei. There is no per-address limit on simultaneously pending requests.

Each accepted request is appended to the global FIFO queue:

```solidity
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
```

`_unlockWithdrawalRequests` contains an unbounded while-loop that iterates from `nextLockedNonce` to the caller-supplied `firstExcludedIndex`:

```solidity
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);
    if (availableAssetAmount < payoutAmount) break;
    ...
    unchecked { nextLockedNonce_++; }
}
```

For a dust request of 1 wei rsETH, `_calculatePayoutAmount` returns `(1 * rsETHPrice) / assetPrice` ≈ 1 wei. The `availableAssetAmount < payoutAmount` guard only breaks when available assets are fully exhausted; with any non-zero available balance, every dust entry is processed without triggering the break. Each iteration performs multiple storage reads and writes (~5,000–10,000 gas). With N = 10,000 dust entries, a single `unlockQueue` call covering all entries consumes 50–100 M gas, exceeding the block gas limit.

Even when the operator batches calls with a small `firstExcludedIndex`, `nextLockedNonce` advances strictly in order with no mechanism to skip entries. The legitimate user's request at nonce N cannot be unlocked until all N preceding attacker nonces are processed, regardless of batch size.

The `ExceedAmountToWithdraw` check (`expectedAssetAmount > getAvailableAssetAmount(asset)`) does not prevent spam: each dust request commits only ~1 wei to `assetsCommitted`, so with any meaningful asset pool (e.g., 1,000 ETH = 1e21 wei), the attacker can queue up to ~1e21 dust requests before this guard triggers.

## Impact Explanation
**Medium — Temporary freezing of funds and unbounded gas consumption.**

Legitimate users' rsETH is transferred to the contract at `initiateWithdrawal` (line 166) and cannot be recovered until their nonce is unlocked. Because `nextLockedNonce` is strictly sequential, all preceding attacker nonces must be processed first. The delay is proportional to the number of spam entries and the operator's batch throughput. This matches the allowed impact: *Medium. Temporary freezing of funds* and *Medium. Unbounded gas consumption*.

## Likelihood Explanation
**Medium.** The attacker only needs rsETH (obtainable by depositing into the protocol). `initiateWithdrawal` is a public, permissionless function requiring no privileged role. The default `minRsEthAmountToWithdraw = 0` is the live on-chain state unless the admin has explicitly called `setMinRsEthAmountToWithdraw` for every supported asset. The economic cost per dust request is gas fees only (~0.0015 ETH at 30 gwei); 10,000 requests costs ~15 ETH — non-trivial but feasible for a motivated attacker. The attack is repeatable after each batch of operator unlocks.

## Recommendation
1. **Set a meaningful non-zero `minRsEthAmountToWithdraw` for every supported asset** in `initialize()`, or add a `require(minRsEthAmountToWithdraw[asset] != 0)` guard in `initiateWithdrawal` to prevent the mapping from remaining at its zero default.
2. **Add a per-user cap** on simultaneously pending withdrawal requests per asset (e.g., `require(userAssociatedNonces[asset][msg.sender].length() < MAX_REQUESTS_PER_USER)`).
3. **Add a `maxIterations` parameter** to `_unlockWithdrawalRequests` so the operator can bound gas per call without being forced to process all attacker entries in a single transaction.

## Proof of Concept
1. Attacker acquires N wei of rsETH (e.g., N = 10,000).
2. Attacker calls `initiateWithdrawal(asset, 1, "")` 10,000 times. Each call passes because `minRsEthAmountToWithdraw[asset] == 0`, so the check reduces to `1 == 0` (false). Each call commits ~1 wei to `assetsCommitted[asset]` and appends nonce 0…9,999 to the queue.
3. Legitimate user calls `initiateWithdrawal(asset, largeAmount, "")`. Their request is assigned nonce 10,000.
4. Operator calls `unlockQueue(asset, 10001, ...)`. `_unlockWithdrawalRequests` loops from nonce 0 to 10,000. Each dust entry has `payoutAmount ≈ 1 wei`; `availableAssetAmount < 1` is false while any assets remain, so no early break occurs. 10,000 iterations × ~8,000 gas = ~80 M gas, exceeding the 30 M block gas limit.
5. Even with batched calls (e.g., 500 entries per call), the legitimate user's request at nonce 10,000 cannot be unlocked until all 10,000 attacker nonces are processed across ~20 operator transactions, delaying their withdrawal by many blocks.

**Foundry test sketch:**
```solidity
function testDustSpamFreezesLegitimateWithdrawal() public {
    // Attacker queues 10_000 dust requests of 1 wei rsETH each
    for (uint i = 0; i < 10_000; i++) {
        vm.prank(attacker);
        withdrawalManager.initiateWithdrawal(asset, 1, "");
    }
    // Legitimate user queues a real withdrawal
    vm.prank(user);
    withdrawalManager.initiateWithdrawal(asset, 1 ether, "");
    // Advance blocks past withdrawal delay
    vm.roll(block.number + withdrawalDelayBlocks + 1);
    // Operator attempts to unlock up to nonce 10_001
    uint256 gasBefore = gasleft();
    vm.prank(operator);
    withdrawalManager.unlockQueue(asset, 10_001, ...);
    uint256 gasUsed = gasBefore - gasleft();
    // Assert gasUsed > block gas limit or legitimate request still locked
    assertGt(gasUsed, 30_000_000);
}
```
