Audit Report

## Title
Daily Fee Mint Limit Blocks `rsETHPrice` Updates, Enabling Theft of Unclaimed Yield from Existing rsETH Holders - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle._updateRsETHPrice()` unconditionally calls `_checkAndUpdateDailyFeeMintLimit()` on every invocation. When `maxFeeMintAmountPerDay` is `0` (its uninitialized default, never set in `initialize()` or `reinitialize()`) or the daily budget is exhausted, every call to `updateRSETHPrice()` and `updateRSETHPriceAsManager()` reverts with `DailyFeeMintLimitExceeded`, freezing `rsETHPrice`. A stale, lower-than-actual `rsETHPrice` causes `LRTDepositPool.getRsETHAmountToMint()` to over-mint rsETH to new depositors, diluting and effectively stealing accrued yield from existing rsETH holders.

## Finding Description
**Root cause — `maxFeeMintAmountPerDay` defaults to `0` and is never initialized:**

`maxFeeMintAmountPerDay` is declared as a plain `uint256` storage variable (line 35) and is not assigned in `initialize()` (lines 64–68) or `reinitialize()` (lines 72–79). It remains `0` until the manager explicitly calls `setMaxFeeMintAmountPerDay()`.

**Revert path:**

Inside `_updateRsETHPrice()`, whenever the protocol is not paused, TVL has grown, and `protocolFeeInBPS > 0`, a non-zero `protocolFeeInETH` is computed (lines 244–246). This produces a non-zero `rsethAmountToMintAsProtocolFee` (line 301), which is passed to `_checkAndUpdateDailyFeeMintLimit()` (line 303).

Inside `_checkAndUpdateDailyFeeMintLimit()` (lines 197–210), the check at line 205 is:
```
currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay
```
With `maxFeeMintAmountPerDay == 0` and any `feeAmount > 0`, this reduces to `feeAmount > 0` → always `true` → always `revert DailyFeeMintLimitExceeded`.

**No bypass exists:** Both public entry points `updateRSETHPrice()` (lines 87–89) and `updateRSETHPriceAsManager()` (lines 94–96) route exclusively through `_updateRsETHPrice()`. The `rsETHPrice` write at line 313 is never reached.

**Second trigger — limit exhaustion:** Even after a non-zero limit is set, a single large TVL increase or multiple updates within the same 24-hour window can exhaust `maxFeeMintAmountPerDay`. All subsequent calls revert until the next period.

**Existing checks are insufficient:** The `else` branch at line 310 calls `_checkAndUpdateDailyFeeMintLimit(0)`, which does not revert when `maxFeeMintAmountPerDay == 0` (since `0 > 0` is false). This only applies when `protocolFeeInETH == 0`, i.e., when there is nothing to protect against. The `remainingDailyFeeMintLimit()` view function (line 171) has a `if (maxFeeMintAmountPerDay == 0) return 0` guard, but this guard is absent from the internal write path.

## Impact Explanation
A frozen `rsETHPrice` (stale, lower-than-actual value) directly causes `LRTDepositPool.getRsETHAmountToMint()` (line 520) to compute:
```
rsethAmountToMint = (amount * assetPrice) / rsETHPrice
```
With a stale lower `rsETHPrice` as denominator, new depositors receive more rsETH than their ETH contribution warrants. This dilutes the per-token value of rsETH held by existing holders, transferring their accrued staking yield to new depositors. This is **High — Theft of unclaimed yield** from existing rsETH holders. The window lasts up to 24 hours per occurrence, or indefinitely while `maxFeeMintAmountPerDay` remains `0`.

## Likelihood Explanation
The `maxFeeMintAmountPerDay == 0` default is active from deployment until the manager explicitly calls `setMaxFeeMintAmountPerDay()`. Any deployment where `protocolFeeInBPS > 0` and staking rewards accrue before that call is made will hit this immediately. `updateRSETHPrice()` is a permissionless public function — any external caller (keeper, MEV searcher, ordinary user) can trigger it. The limit-exhaustion variant is repeatable every 24 hours. No special privileges or victim mistakes are required; normal protocol operation (reward accrual + public price update call) is sufficient.

## Recommendation
1. **Decouple the fee-mint limit from the price-write path.** If the fee mint limit is exceeded, skip or cap the fee mint but still commit the updated `rsETHPrice` to storage.
2. **Treat `maxFeeMintAmountPerDay == 0` as "unlimited"** inside `_checkAndUpdateDailyFeeMintLimit`: add `if (maxFeeMintAmountPerDay == 0) return;` at the top of the function, consistent with the existing guard in `remainingDailyFeeMintLimit()` (line 171).
3. **Initialize `maxFeeMintAmountPerDay`** to a safe non-zero value in `initialize()` or `reinitialize()` so the contract is never deployed in a state where fee-bearing price updates are impossible.
