### Title
Unauthenticated Pool-Server-Controlled Redirect Permanently Overwrites `pool_url` in `pooling_share_state.yaml` — (`chia/farmer/farmer.py`, `chia/pools/pool_config.py`)

### Summary
The farmer's `_pool_get_pool_info` method follows HTTP 301/308 redirects from the pool server and, with no domain allowlist or singleton-owner signature check, writes the redirect destination URL directly to `pooling_share_state.yaml`. Any pool server (or attacker who has compromised one) can permanently redirect all farmer traffic — partials, authentication tokens, GET/POST/PUT `/farmer` — to an arbitrary HTTPS endpoint they control.

### Finding Description

`Farmer._pool_get_pool_info` (farmer.py lines 350–393) issues a GET to `{pool_config.pool_url}/pool_info` via aiohttp, which follows redirects automatically. After the request completes, the code checks:

```python
# farmer.py lines 372-379
new_pool_url: str | None = None
response_url_str = f"{resp.url}"
if (
    response_url_str != url
    and len(resp.history) > 0
    and all(r.status in {301, 308} for r in resp.history)
):
    new_pool_url = response_url_str.replace("/pool_info", "")
``` [1](#0-0) 

The only guards are:
- The final URL must differ from the original
- At least one redirect must exist in history
- All redirects must be 301 or 308

There is **no check** that the redirect destination is on the same domain, no allowlist, and no cryptographic authorization from the singleton owner.

Back in `update_pool_state`, if `new_pool_url` is non-None, it is written unconditionally to `pooling_share_state.yaml`:

```python
# farmer.py lines 619-623
if pool_info_result is not None and pool_info_result.new_pool_url is not None:
    with PoolingShareState.acquire(
        root_path=self._root_path, p2_singleton_puzzle_hash=p2_singleton_puzzle_hash
    ) as editable_pool_config:
        editable_pool_config.pool_url = pool_info_result.new_pool_url
``` [2](#0-1) 

`PoolingShareState.acquire` (pool_config.py lines 88–113) writes the mutated object back to YAML on context-manager exit with no validation of the new URL value: [3](#0-2) 

The HTTPS enforcement check at lines 600–603 only validates the **current** `pool_config.pool_url` (the pre-redirect value) and runs **before** the redirect is followed, so it does not protect the redirect destination: [4](#0-3) 

### Impact Explanation

After the redirect URL is persisted:

1. All subsequent `GET /farmer`, `POST /farmer`, `PUT /farmer` calls use `pool_config.pool_url` from the YAML — they now go to the attacker's server.
2. All partial proof submissions (`POST /partial` in `farmer_api.py`) use the same `pool_config.pool_url` — the legitimate pool never receives them, so the farmer earns no pool rewards.
3. The attacker receives every partial proof and every signed authentication token.
4. The change is permanent across farmer restarts (written to disk).

This satisfies **High: Corruption of pool membership state with direct security impact** — the farmer's pool endpoint is durably redirected to an attacker-controlled server without the singleton owner's authorization, causing loss of all pool rewards and exposure of partial proofs.

### Likelihood Explanation

- The attacker only needs to control (or compromise) the pool server the farmer has already joined.
- A valid TLS certificate for the attacker's domain is freely obtainable (Let's Encrypt), satisfying the SSL context check used by aiohttp.
- No special network position is required; the attack is a single HTTP response.
- The farmer polls `update_pool_state` every 60 seconds (`_periodically_update_pool_state_task`), so the redirect takes effect within one polling cycle. [5](#0-4) 

### Recommendation

1. **Domain-lock redirects**: After following a redirect, verify that the final URL's registered domain (eTLD+1) matches the original `pool_config.pool_url`. Reject cross-domain redirects.
2. **Require singleton-owner authorization**: Any `pool_url` change should require a signature from the `owner_public_key` stored in `PoolingShareState`, binding the new URL to the launcher ID.
3. **Validate the new URL before writing**: Apply the same HTTPS enforcement check to `new_pool_url` before calling `PoolingShareState.acquire`.
