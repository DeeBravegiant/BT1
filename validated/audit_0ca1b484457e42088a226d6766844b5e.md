### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Map Key Enables Queue Saturation to Block Legitimate Foreign-Transaction Verification Submissions - (File: crates/contract/src/pending_requests.rs)

### Summary

The `pending_verify_foreign_tx_requests` map uses `VerifyForeignTransactionRequest` — a struct that omits the caller's account ID — as its lookup key. Because the fan-out queue for each key is hard-capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`, an unprivileged attacker who knows a target foreign transaction ID can saturate the queue with 128 cheap calls, causing every subsequent legitimate `verify_foreign_transaction` submission for that same transaction to panic with `PendingRequestQueueFull` and blocking the bridge user from getting their cross-chain funds released.

---

### Finding Description

**Root cause — caller-agnostic key**

`VerifyForeignTransactionRequest` is defined as:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,   // contains tx_id, extractors, finality
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

The caller's `predecessor_account_id` is never incorporated into this key. This is in direct contrast to `SignatureRequest`, whose `tweak` field is derived from `(predecessor, path)`, making every `sign()` key unique per-caller.

**How the key is stored**

`verify_foreign_transaction()` converts the user-supplied args directly to a `VerifyForeignTransactionRequest` (no caller field added) and pushes a yield index into the fan-out map:

```rust
let request = args_into_verify_foreign_tx_request(request);
// ...
move |this, id| this.add_verify_foreign_tx_request(request, id)
``` [2](#0-1) 

`add_verify_foreign_tx_request` calls `push_pending_yield`, which panics once the queue reaches the cap:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
// ...
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
}
``` [3](#0-2) 

**Attack path**

1. Attacker monitors the foreign chain (e.g., Bitcoin) and observes an inbound bridge transaction `tx_id_X` destined for a victim.
2. Attacker submits 128 `verify_foreign_transaction` calls with identical `{ tx_id: tx_id_X, domain_id, payload_version, extractors }`. Each call costs only 1 yoctonear deposit plus gas.
3. The fan-out queue for the key `VerifyForeignTransactionRequest { tx_id_X, … }` is now full.
4. The victim (bridge service or end user) calls `verify_foreign_transaction(tx_id_X)` — the contract panics with `PendingRequestQueueFull`. The victim's NEAR-side promise is never created.
5. The victim cannot submit again until the 128 attacker yields drain — either via MPC response (which also resolves the attacker's slots) or via NEAR's ~200-block yield timeout per slot.

The developers themselves acknowledge the caller-agnostic design in a test comment:

> "a different account would today be blocked from receiving a response by alice's submission" [4](#0-3) 

That comment addresses the fan-out response delivery case; it does not address the queue-saturation blocking case.

---

### Impact Explanation

This breaks the **request-lifecycle invariant** for the `verify_foreign_transaction` flow: any user must be able to submit a verification request for any foreign transaction they have a legitimate interest in. When the queue is saturated, that invariant is violated.

In the Omnibridge use case — the primary stated motivation for this feature — a user who has sent funds on a foreign chain and needs MPC attestation to release them on NEAR is blocked from submitting the verification request. Their funds remain locked on the NEAR side until the attacker's queued slots drain. Because the attacker can repeat the saturation immediately after each drain cycle, the blocking can be sustained indefinitely at low cost.

This maps to the allowed Medium impact: **"request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."**

---

### Likelihood Explanation

- **Attacker entry point**: any unprivileged NEAR account; no special role required.
- **Required knowledge**: the target `tx_id`, which is publicly visible on the foreign chain.
- **Cost**: 128 × (1 yoctonear + ~7 TGas) ≈ negligible NEAR.
- **No threshold collusion needed**: the attack operates entirely at the contract layer before any MPC node is involved.
- **Repeatability**: after each drain cycle the attacker can immediately re-saturate.

---

### Recommendation

Include the caller's `predecessor_account_id` in the map key for `verify_foreign_transaction`, mirroring the `sign()` design where the tweak encodes the caller identity. Concretely, define a wrapper key type:

```rust
pub struct VerifyForeignTransactionRequestWithCaller {
    pub caller: AccountId,
    pub request: VerifyForeignTransactionRequest,
}
```

and use it as the `LookupMap` key. This makes each caller's queue slot independent, so one caller cannot saturate another's slot.

Alternatively, enforce a per-`(caller, VerifyForeignTransactionRequest)` submission limit at the contract level before enqueuing.

### Proof of Concept

```
// Setup: Bitcoin bridge, domain_id = 3, tx_id = [0xAB; 32]
let args = VerifyForeignTransactionRequestArgs {
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: [0xAB; 32].into(),
        confirmations: 1.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
    domain_id: DomainId(3),
    payload_version: ForeignTxPayloadVersion::V1,
};

// Attacker submits 128 identical calls (each costs ~1 yoctonear + gas)
for _ in 0..128 {
    attacker_account.call(contract_id, "verify_foreign_transaction")
        .args_json(json!({ "request": args }))
        .deposit(NearToken::from_yoctonear(1))
        .transact();
}

// Victim now tries to submit the same request
let result = victim_account.call(contract_id, "verify_foreign_transaction")
    .args_json(json!({ "request": args }))
    .deposit(NearToken::from_yoctonear(1))
    .transact();

// result.is_failure() == true
// error: "Pending-request queue is full for this request key (limit: 128)."
// Victim's cross-chain funds are locked until the 128 attacker slots drain.
```

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```
