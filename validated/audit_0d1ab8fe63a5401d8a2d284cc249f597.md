### Title
`respond_ckd` Accepts Unvalidated CKD Output for `AppPublicKey` Variant, Allowing a Single Byzantine Participant to Inject Garbage Key Material - (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_ckd` function in the MPC contract validates the CKD response cryptographically **only** when the request uses the `AppPublicKeyPV` (publicly verifiable) variant. When the request uses the legacy `AppPublicKey` (privately verifiable) variant, **no on-chain validation is performed**. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary garbage `CKDResponse`, and the contract will accept it, resolve all pending yields for that request, and deliver the invalid key material to the user. The user's CKD request is permanently consumed and they receive a useless derived key.

---

### Finding Description

In `respond_ckd` (lines 653–689 of `crates/contract/src/lib.rs`), the response validation is gated on the variant of `app_public_key`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VALIDATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which performs a BLS12-381 pairing check verifying `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`: [2](#0-1) 

For `AppPublicKey`, the match arm is empty — the response bytes are serialized and forwarded directly to `resolve_yields_for` with no cryptographic check: [3](#0-2) 

The `resolve_yields_for` call drains **all** queued yields for the request key, meaning every duplicate submission of the same CKD request is simultaneously resolved with the garbage value.

The `respond_ckd` function only requires the caller to be a single attested participant (`assert_caller_is_attested_participant_and_protocol_active`), not a threshold quorum: [4](#0-3) 

This is structurally identical to the M-33 pattern: the user's input (CKD request + deposit) is consumed, the output (derived key material) is zero/garbage, and the contract does not revert.

---

### Impact Explanation

A user submitting a `request_app_private_key` with the `AppPublicKey` variant receives a `CKDResponse` (`big_y`, `big_c`) that is entirely attacker-controlled. The user's yield is permanently resolved — the request cannot be retried under the same yield ID. Any downstream application that trusts the returned `big_c` as a valid encrypted key share will operate on garbage, silently breaking the confidential key derivation guarantee. This breaks the production safety invariant that a CKD response must be a valid encryption of the master-key-derived secret under the user's public key.

**Impact class**: Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants (the CKD output correctness guarantee) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

Any single attested MPC participant that turns Byzantine can exploit this. The attacker does not need to compromise a threshold number of nodes — one node is sufficient. The `AppPublicKey` variant is the legacy/default path documented in the contract README and used in the majority of existing integrations, making it the most commonly exercised code path. [5](#0-4) 

---

### Recommendation

Add the same cryptographic output validation for the `AppPublicKey` variant that already exists for `AppPublicKeyPV`. Since `AppPublicKey` does not carry a G2 component, a different check is needed — for example, verifying `e(big_c - big_s, g2) = e(big_y, pk2)` using the master public key, or requiring the response to include a proof of correct encryption. At minimum, reject responses where `big_y` or `big_c` are the identity/zero point on G1.

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {
        // Add: validate big_y and big_c are non-identity G1 points
        // and optionally verify a proof of correct encryption
        if response.big_y == IDENTITY_G1 || response.big_c == IDENTITY_G1 {
            env::panic_str("CKD response contains identity point");
        }
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. Alice calls `request_app_private_key` with `AppPublicKey` variant and a valid BLS12-381 G1 point, attaching 1 yoctonear deposit. The request is enqueued. [6](#0-5) 

2. A single Byzantine attested participant calls `respond_ckd` with:
   ```rust
   CKDResponse {
       big_y: Bls12381G1PublicKey([0u8; 48]),  // garbage / identity
       big_c: Bls12381G1PublicKey([0u8; 48]),  // garbage / identity
   }
   ```

3. The `AppPublicKey` match arm performs no validation and falls through: [7](#0-6) 

4. `resolve_yields_for` resolves all pending yields for Alice's request with the garbage bytes. `return_ck_and_clean_state_on_success` returns the garbage `CKDResponse` to Alice: [8](#0-7) 

5. Alice's request is permanently consumed. She receives `big_y = 0`, `big_c = 0` — a useless derived key. The test at line 3424 confirms the contract accepts `[1u8; 48]` / `[2u8; 48]` (arbitrary bytes) without any validation for the `AppPublicKey` variant: [9](#0-8)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

```markdown
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
```
