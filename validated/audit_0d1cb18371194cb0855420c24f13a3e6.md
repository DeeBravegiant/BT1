Audit Report

## Title
ETH Deposit Limit Not Enforced — Incoming Amount Excluded from Cap Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ERC-20 assets it correctly evaluates `totalAssetDeposits + amount > depositLimit`, but for ETH it evaluates only `totalAssetDeposits > depositLimit`, silently discarding the incoming `amount`. Any unprivileged depositor can call `depositETH` with an arbitrarily large `msg.value` and bypass the `depositLimitByAsset` cap entirely, as long as the pre-existing total has not already crossed the limit.

## Finding Description
The root cause is in `_checkIfDepositAmountExceedesCurrentLimit` at line 679:

```solidity
// contracts/LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC-20
}
``` [1](#0-0) 

The parameter `amount` (which equals `msg.value`) is received but never used in the ETH branch. The full call chain is:

- `depositETH` (line 76, `external payable nonReentrant whenNotPaused`) passes `msg.value` to `_beforeDeposit` [2](#0-1) 
- `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` and reverts only if it returns `true` [3](#0-2) 
- For ETH, the check returns `false` whenever `totalAssetDeposits <= depositLimit`, regardless of how large `amount` is — so the revert never fires

No existing guard compensates for this: `nonReentrant` and `whenNotPaused` are unrelated to the cap logic, and `onlySupportedAsset` only verifies ETH is a supported asset.

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's primary risk-management control over how much ETH enters EigenLayer strategies. Bypassing it allows rsETH to be minted for ETH far beyond the intended ceiling. Excess ETH accumulates idle in the deposit pool and cannot be deployed to EigenLayer strategies until governance intervenes. No ETH is stolen, but the contract fails to deliver its promised deposit-cap guarantee. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`depositETH` is `external payable` with no role restriction. Any depositor who observes `totalAssetDeposits < depositLimitByAsset(ETH)` can send a single transaction with an arbitrarily large `msg.value`. No front-running, privileged access, or multi-step setup is required. The condition is trivially observable on-chain and the exploit is repeatable.

## Recommendation
Include the incoming deposit amount in the ETH branch, mirroring the ERC-20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

Additionally, `getAssetCurrentLimit` uses a strict `>` comparison (line 404), which reports a non-zero remaining limit even when the cap is exactly met; changing it to `>=` would make it consistent. [4](#0-3) 

## Proof of Concept
Assume `depositLimitByAsset(ETH) = 100_000 ether` and `totalAssetDeposits(ETH) = 99_999 ether`.

1. Attacker calls `depositETH{value: 500_000 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500_000 ether)` evaluates `99_999 ether > 100_000 ether` → `false` → no revert.
3. `getRsETHAmountToMint` mints rsETH for the full 500,000 ETH.
4. `totalAssetDeposits(ETH)` becomes 599,999 ETH — nearly 6× the intended cap — with no protocol-level rejection.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    // set depositLimitByAsset(ETH) = 100_000 ether, pre-fill 99_999 ether
    vm.deal(attacker, 500_000 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 500_000 ether}(0, "");
    assertGt(lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN), 100_000 ether);
}
```
