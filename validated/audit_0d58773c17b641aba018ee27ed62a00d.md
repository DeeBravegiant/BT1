Audit Report

## Title
Direct ETH Donation Inflates `address(this).balance`, Enabling rsETH Price Manipulation and Deposit Limit Griefing - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool` exposes an unconditional `receive()` at line 58 that accepts ETH from any caller. `getETHDistributionData()` reads `address(this).balance` raw at line 480, so any donated ETH is counted as protocol-owned collateral. This inflated total propagates into `getTotalAssetDeposits()`, which feeds both the ETH deposit-limit gate and `LRTOracle._getTotalEthInProtocol()`, enabling an unprivileged attacker to (a) freeze new ETH deposits and (b) inflate the recorded rsETH price so subsequent depositors are under-minted.

## Finding Description

**Root cause — open `receive()` and raw balance read:**

`LRTDepositPool` line 58 accepts ETH from any sender with no access control: [1](#0-0) 

`getETHDistributionData()` at line 480 assigns the entire native balance as `ethLyingInDepositPool`: [2](#0-1) 

**Deposit-limit freeze path:**

`_checkIfDepositAmountExceedesCurrentLimit()` for ETH evaluates `totalAssetDeposits > limit` — not `totalAssetDeposits + depositAmount > limit`: [3](#0-2) 

Because the check is a strict `>` on the running total (not on total + incoming amount), any donation that pushes `address(this).balance` above the configured cap causes every subsequent `depositETH` call to revert with `MaximumDepositLimitReached` via `_beforeDeposit`: [4](#0-3) 

**rsETH price manipulation path:**

`LRTOracle._getTotalEthInProtocol()` calls `getTotalAssetDeposits(ETH_TOKEN)` on the deposit pool, which in turn calls `getETHDistributionData()` and therefore reads the inflated `address(this).balance`: [5](#0-4) 

The inflated `totalETHInProtocol` is used to compute `newRsETHPrice`: [6](#0-5) 

A `pricePercentageLimit` guard exists at lines 256–266 that reverts `updateRSETHPrice()` if the price increase exceeds the configured threshold (unless the caller holds MANAGER role). However, this guard is bypassed entirely when `pricePercentageLimit == 0`, and for donations sized to stay within the threshold the manipulation succeeds silently. Once the elevated price is recorded, every depositor who mints rsETH afterward receives fewer shares than the ETH they contribute warrants; the shortfall accrues to existing rsETH holders. [7](#0-6) 

## Impact Explanation

**High — Theft of unclaimed yield:** When `updateRSETHPrice()` is called while the balance is inflated, the stored `rsETHPrice` is higher than the true backing ratio. All depositors who mint rsETH after that update receive fewer tokens than the ETH they contribute warrants. The shortfall is a direct transfer of value from new depositors to incumbent rsETH holders. This matches the allowed impact "Theft of unclaimed yield."

**Medium — Temporary freezing of funds:** A donation that pushes `totalAssetDeposits` above `depositLimitByAsset(ETH_TOKEN)` causes every `depositETH` call to revert until an operator moves the excess ETH to a NodeDelegator. This matches "Temporary freezing of funds." Note: the deposit-freeze vector alone would be griefing-only; it is included here because it compounds the price-manipulation impact by preventing corrective deposits.

## Likelihood Explanation

The `receive()` function has no role check, whitelist, or minimum-amount guard. Any EOA can trigger the inflation with a single ETH transfer. The cost to the attacker equals the donated ETH, which is not destroyed but locked in the pool. When the deposit cap is near the current total (a common operational state), a small donation suffices for the freeze. The price manipulation requires only that `updateRSETHPrice()` be called while the inflated balance persists, which is a routine keeper operation. The `pricePercentageLimit` guard limits the magnitude of a single manipulation but does not prevent repeated small donations or attacks when the limit is set to zero.

## Recommendation

Replace the raw `address(this).balance` read in `getETHDistributionData()` with a storage variable (e.g., `ethReceivedFromDeposits`) that is incremented only inside the explicit, access-controlled entry points: `depositETH`, `receiveFromNodeDelegator`, `receiveFromRewardReceiver`, and `receiveFromLRTConverter`. Alternatively, apply a before/after balance delta inside each entry point and credit only the delta. Either approach ensures that externally donated ETH is never counted as protocol-owned collateral. Additionally, the ETH deposit-limit check should be changed from `totalAssetDeposits > limit` to `totalAssetDeposits + depositAmount > limit` to match the ERC-20 asset check at line 681.

## Proof of Concept

1. Record `D = getTotalAssetDeposits(ETH_TOKEN)` and `L = lrtConfig.depositLimitByAsset(ETH_TOKEN)` with `D < L`.
2. Attacker sends `L - D + 1` wei directly to `LRTDepositPool` via `address(lrtDepositPool).call{value: L - D + 1}("")`.
3. `address(this).balance` increases by `L - D + 1`; `getETHDistributionData()` now returns `ethLyingInDepositPool = old_balance + (L - D + 1)`.
4. `getTotalAssetDeposits(ETH_TOKEN)` returns `L + 1`.
5. Any `depositETH(...)` call executes `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, amount)` → `L + 1 > L` → `true` → reverts with `MaximumDepositLimitReached`.
6. Keeper calls `LRTOracle.updateRSETHPrice()`; `_getTotalEthInProtocol()` reads the inflated pool balance, computes an elevated `newRsETHPrice`, and (if within `pricePercentageLimit` or if the limit is zero) records it as `rsETHPrice`.
7. All subsequent depositors receive `depositAmount / inflated_rsETHPrice` rsETH instead of `depositAmount / true_rsETHPrice` rsETH; the difference accrues to existing holders.

**Foundry test outline:**
```solidity
function test_donationInflatesPrice() public {
    uint256 D = lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN);
    uint256 L = lrtConfig.depositLimitByAsset(ETH_TOKEN);
    // Step 1: donate to push total above limit
    vm.deal(attacker, L - D + 1);
    vm.prank(attacker);
    (bool ok,) = address(lrtDepositPool).call{value: L - D + 1}("");
    assertTrue(ok);
    // Step 2: verify deposit is blocked
    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
    lrtDepositPool.depositETH{value: 1 ether}(0, referralId);
    // Step 3: verify price is inflated
    uint256 priceBefore = lrtOracle.rsETHPrice();
    vm.prank(keeper);
    lrtOracle.updateRSETHPrice();
    assertGt(lrtOracle.rsETHPrice(), priceBefore);
}
```
