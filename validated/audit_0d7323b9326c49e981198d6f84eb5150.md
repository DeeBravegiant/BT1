### Title
Stale Foreign-Chain Observation Replay via Missing Temporal Binding in `ForeignTxSignPayload` ŌĆö (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `ForeignTxSignPayloadV1` struct that MPC nodes threshold-sign during `verify_foreign_transaction` contains only `request` and `values` ŌĆö no timestamp, no block number, and no `observed_at` field. Because the signed payload carries no temporal binding, a threshold signature produced at time TŌéü is cryptographically indistinguishable from one produced at TŌéé. A single Byzantine MPC participant (strictly below the signing threshold) can replay a previously produced threshold signature to resolve a new pending `verify_foreign_transaction` request, delivering a stale foreign-chain observation to the bridge contract without the MPC contract detecting the replay.

---

### Finding Description

The design document for the foreign-chain verification feature states that MPC nodes sign a canonical encoding of `(request, observed_values, observed_at)`. However, the actual implementation of `ForeignTxSignPayloadV1` omits `observed_at` entirely:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
    // ŌåÉ no timestamp, no block number, no observed_at
}
```

The 32-byte hash that nodes sign is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
```

The `respond_verify_foreign_tx` entry point on the contract verifies only that:
1. The caller is an attested MPC participant.
2. The ECDSA signature in the response is valid over `response.payload_hash` under the domain's root public key.

```rust
// crates/contract/src/lib.rs  (respond_verify_foreign_tx)
let signature_is_valid = match (&response.signature, public_key) {
    (dtos::SignatureResponse::Secp256k1(sig), PublicKeyExtended::Secp256k1 { near_public_key }) => {
        near_mpc_signature_verifier::verify_ecdsa_signature(sig, &payload_hash, &secp_pk).is_ok()
    }
    ...
};
if !signature_is_valid { return Err(RespondError::InvalidSignature.into()); }
pending_requests::resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request, ...)
```

The contract does **not** check:
- Whether `payload_hash` corresponds to a fresh observation of the foreign chain.
- Whether the `values` embedded in the payload (e.g., `block_hash_B`) are still canonical on the foreign chain.
- Whether the response was produced within any recency window.

Because `VerifyForeignTransactionRequest` is keyed on `{domain_id, payload_version, request}` (i.e., the tx-id and extractor list, not a nonce or block height), any new submission of the same foreign transaction produces an identical pending-request key. A Byzantine participant who observed a prior threshold signature for that key can call `respond_verify_foreign_tx` with the old `{payload_hash, signature}` pair and the contract will accept it.

---

### Impact Explanation

**Impact: High ŌĆö Forged foreign-chain verification / invalid bridge execution.**

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a user deposits on a foreign chain, the bridge submits a verification request, and the MPC network's signed attestation authorises the NEAR-side release of funds.

Attack scenario (Bitcoin example):

1. User deposits on Bitcoin; tx T lands in block B (1 confirmation).
2. Bridge submits `verify_foreign_transaction` for T with `confirmations: 1`.
3. MPC threshold-signs `{request: {tx_id: T, confirmations: 1, extractors: [BlockHash]}, values: [block_hash_B]}` ŌåÆ signature S.
4. Bitcoin reorgs: block B is replaced by BŌĆ▓; tx T is now in BŌĆ▓ (different hash) or dropped.
5. Bridge retries: submits `verify_foreign_transaction` for T again (same request key).
6. Byzantine MPC participant calls `respond_verify_foreign_tx` with the old `{payload_hash: SHA256({T, block_hash_B}), signature: S}`.
7. Contract verifies S is valid over `payload_hash` Ō£ō, resolves the pending yield.
8. Bridge contract receives an MPC-signed attestation that T was in block B ŌĆö a block no longer on the canonical chain.
9. If the bridge contract does not independently verify block canonicality, it releases NEAR-side funds for a deposit that was reorganised away ŌĆö a double-spend.

The `ForeignChainSignatureVerifier::verify_signature` helper in `near-mpc-sdk` requires the bridge contract to supply `expected_extracted_values`. A bridge that does not know the expected block hash in advance (which is the entire reason it uses the MPC verification service) cannot distinguish the stale `block_hash_B` from a fresh one.

---

### Likelihood Explanation

**Likelihood: Low-Medium.**

- A foreign-chain reorg of sufficient depth is uncommon but not rare (Bitcoin 1-conf reorgs occur; EVM chains have had multi-block reorgs).
- The bridge retrying the same request after a reorg is the expected operational behaviour.
- Only a **single** attested MPC participant is needed to replay the old signature ŌĆö no threshold collusion is required. Any participant who observed the prior signing round has the old threshold signature.
- The window of opportunity is bounded by the NEAR yield-resume timeout (`REQUEST_EXPIRATION_BLOCKS = 200` blocks), but that is ample time for a Byzantine participant to act.

---

### Recommendation

Bind the signed payload to a specific point in time or chain state by including an `observed_at` field in `ForeignTxSignPayloadV1`:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
    pub observed_at_block: u64,   // foreign-chain block height at observation time
    pub observed_at_timestamp: u64, // unix seconds
}
```

The MPC node populates these fields from the RPC response at signing time (e.g., `block.height` already fetched during `verify_block_is_canonical`). The bridge contract can then enforce a maximum age on the observation before accepting the attestation, analogous to how `check_collateral_freshness` enforces `MAX_COLLATERAL_AGE` on PCCS collateral in `crates/tee-authority/src/tee_authority.rs`.

Additionally, consider adding a per-request nonce or NEAR block height to the pending-request key so that a resubmitted request for the same foreign tx cannot be resolved by a signature produced for an earlier submission.

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,

    })
