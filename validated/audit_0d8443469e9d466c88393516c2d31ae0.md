### Assessment

The described attack path is **real and reachable**:

1. Attacker crafts a v2 blob with `Q_compressed = Q'` (attacker-chosen valid curve point), keeping `Qis` and `x_share` consistent with each other (`x_share * G == Qi_self` passes).
2. Calls `cbmpc_ecdsa_mp_get_public_key_compressed` ‚Üí returns `Q'` with `SUCCESS`.
3. Application stores `Q'` as the MPC public key for verification.
4. Attacker signs messages with the private key corresponding to `Q'`.
5. Application verifies those signatures against `Q'` and accepts them.

The signing path would reject the same crafted blob (the `SUM(Qis) != Q` check in `sign` would fire), but `get_public_key_compressed` is used independently ‚Äî e.g., to register a public key before any signing occurs, or to verify a signature produced by a different party.

### Title
Unverified `Q_compressed` in `get_public_key_compressed` for AC (v2) key blobs enables public-key substitution ‚Äî (`src/cbmpc/api/ecdsa_mp.cpp`)

### Summary
`get_public_key_compressed` accepts both additive (v1) and AC (v2) key blobs and returns `blob.Q_compressed` after only checking it is a valid curve point. For v2 blobs, no check is performed that `Q` equals the value reconstructable from the stored `Qis`. An attacker who can supply a crafted blob can cause the function to return an arbitrary attacker-chosen public key.

### Finding Description
In `src/cbmpc/api/ecdsa_mp.cpp`, `get_public_key_compressed` (lines 459‚Äì479) parses the blob, validates the curve, decodes `blob.Q_compressed` into a point, and returns it. It performs no integrity check linking `Q` to the `Qis` map. For v1 blobs, `deserialize_key_blob` (used by `sign_additive`) enforces `Q == sum(Qis)` at lines 172‚Äì174, but `get_public_key_compressed` never calls that function. For v2 blobs, `deserialize_ac_key_blob` only enforces the self-share binding (`x_share * G == Qi_self`) and explicitly defers full Q validation to "use sites" ‚Äî but `get_public_key_compressed` is not a use site that performs any such validation.

### Impact Explanation
An attacker who can supply a crafted v2 blob (e.g., by tampering with a persisted key blob, or by being a malicious serialized-input provider) can cause `get_public_key_compressed` to return an arbitrary valid curve point `Q'` as the MPC public key. If the application uses this returned value to register a verification key, the attacker ‚Äî who knows the discrete log of `Q'` ‚Äî can produce signatures that the application accepts as legitimate MPC-protocol output. This is a public-key substitution attack: the returned key does not correspond to the actual secret shares held by the MPC parties.

### Likelihood Explanation
Requires the attacker to supply a crafted key blob to the API. This is realistic for applications that accept key blobs from external storage, user input, or network peers. The crafted blob requires only a valid curve point for `Q'` and a self-consistent `x_share`/`Qi_self` pair; no knowledge of other parties' shares is needed.

### Recommendation
In `get_public_key_compressed`, for v1 blobs, verify `Q == sum(Qis)` before returning. For v2 blobs, either (a) require the caller to supply the access structure and verify Q via `ac.reconstruct_exponent`, or (b) recompute Q from the Qis using the same reconstruction logic used at signing time, and reject if it does not match `blob.Q_compressed`. Alternatively, route `get_public_key_compressed` through the full `deserialize_key_blob` / `deserialize_ac_key_blob` path rather than doing a raw blob parse.

### Proof of Concept
```
1. Run a legitimate dkg_ac to obtain a valid v2 key blob B.
2. Deserialize B, replace Q_compressed with Q' = sk'*G for a known sk'.
   Keep all Qis and x_share unchanged (x_share*G == Qi_self still holds).
3. Re-serialize to produce crafted blob B'.
4. Call cbmpc_ecdsa_mp_get_public_key_compressed(B', &out) ‚Üí returns Q', SUCCESS.
5. Assert out != true_Q (sum reconstructed from Qis).
6. Sign an arbitrary message with sk' using standard ECDSA.
7. Verify the signature against out (Q') ‚Üí verification succeeds.
```
The returned public key `Q'` does not correspond to the MPC key shares, yet the API returns it without error.

error_t get_public_key_compressed(mem_t key_blob, buf_t& pub_key) {
  if (const error_t rv = coinbase::api::detail::validate_mem_arg_max_size(key_blob, "key_blob",
                                                                          coinbase::api::detail::MAX_OPAQUE_BLOB_SIZE))
    return rv;
  key_blob_v1_t blob;
  error_t rv = coinbase::convert(blob, key_blob);
  if (rv) return rv;
  if (blob.version != key_blob_version_v1 && blob.version != ac_key_blob_version_v1)
    return coinbase::error(E_FORMAT, "unsupported key blob version");

  const auto cid = static_cast<curve_id>(blob.curve);
  if (cid == curve_id::ed25519) return coinbase::error(E_FORMAT, "invalid key blob curve");
  const auto curve = to_internal_curve(cid);
  if (!curve.valid()) return coinbase::error(E_FORMAT, "invalid key blob curve");

  coinbase::crypto::ecc_point_t Q(curve);
  if (rv = Q.from_bin(curve, blob.Q_compressed)) return coinbase::error(rv, "invalid key blob");

  pub_key = Q.to_compressed_bin();
  return SUCCESS;
}
