Audit Report

## Title
`queuedWithdrawalsBuffer` Defaults to Zero, Enabling Instant Withdrawals to Drain Assets Committed to Queued Withdrawals - (File: `contracts/LRTUnstakingVault.sol`)

## Summary
`queuedWithdrawalsBuffer[asset]` defaults to `0` in `LRTUnstakingVault`, and `setInstantWithdrawalEnabled` in `LRTWithdrawalManager` does not atomically configure the buffer. When a manager enables instant withdrawal, the entire vault balance is immediately available for instant withdrawal — including assets already committed to pending queued withdrawal requests. Any rsETH holder can drain the vault before the operator separately calls `setQueuedWithdrawalsBuffer`, causing `unlockQueue` to revert and temporarily freezing queued withdrawal users' rsETH locked in the contract.

## Finding Description
`queuedWithdrawalsBuffer` is declared as a plain mapping in `LRTUnstakingVault`:

```solidity
mapping(address asset => uint256 buffer) public queuedWithdrawalsBuffer;
``` [1](#0-0) 

This defaults to `0` for every asset. `getAssetsAvailableForInstantWithdrawal` computes available instant withdrawal capacity as:

```solidity
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
``` [2](#0-1) 

With `reservedBuffer = 0`, `availableAmount = vaultBalance` — the full vault balance is exposed.

`setInstantWithdrawalEnabled` in `LRTWithdrawalManager` does not set the buffer:

```solidity
function setInstantWithdrawalEnabled(address asset, bool enabled)
    external onlySupportedAsset(asset) onlyLRTManager
{
    isInstantWithdrawalEnabled[asset] = enabled;
    emit InstantWithdrawalEnabledUpdated(asset, enabled);
}
``` [3](#0-2) 

The buffer can only be set by a separate operator call to `setQueuedWithdrawalsBuffer` in `LRTUnstakingVault`: [4](#0-3) 

Once instant withdrawal is live, `instantWithdrawal` checks only against `getAssetsAvailableForInstantWithdrawal`:

```solidity
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
unstakingVault.redeem(asset, assetAmountUnlocked);
``` [5](#0-4) 

The `assetsCommitted` mechanism only gates new `initiateWithdrawal` calls via `getAvailableAssetAmount`; it does not restrict instant withdrawals from consuming vault assets already committed to pending queued requests. [6](#0-5) 

After the vault is drained, `unlockQueue` computes `totalAvailableAssets` as `unstakingVault.balanceOf(asset)`: [7](#0-6) 

and immediately reverts:

```solidity
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
``` [8](#0-7) 

Queued withdrawal users have their rsETH locked in the `LRTWithdrawalManager` contract (transferred at `initiateWithdrawal`, line 166, burned only at `unlockQueue`, line 305). There is no `cancelWithdrawal` function, so they cannot recover their rsETH until the vault is replenished and `unlockQueue` can execute. [9](#0-8) [10](#0-9) 

## Impact Explanation
**Medium — Temporary freezing of funds.** Queued withdrawal users have transferred their rsETH to the contract and are waiting for `unlockQueue` to process their requests. If an attacker drains the vault before the operator sets a non-zero buffer, `unlockQueue` reverts and those users cannot complete their withdrawals. Their rsETH remains locked in the contract with no cancellation path. The freeze persists until the protocol replenishes the vault (e.g., by moving assets from a `NodeDelegator`). This matches the allowed impact: *Medium — Temporary freezing of funds*.

## Likelihood Explanation
**Medium.** The manager (`onlyLRTManager`) enabling instant withdrawal and the operator (`onlyLRTOperator`) setting the buffer are distinct roles executing separate transactions. In a realistic feature rollout, the manager may enable instant withdrawal without the operator having pre-configured the buffer. Any rsETH holder can immediately call `instantWithdrawal` to drain the vault during this window. No special privileges or victim mistakes are required beyond the manager performing a legitimate protocol action.

## Recommendation
Require the buffer to be set atomically when enabling instant withdrawal. Modify `setInstantWithdrawalEnabled` to accept a `buffer` parameter and call `setQueuedWithdrawalsBuffer` on the unstaking vault in the same transaction when `enabled = true`:

```solidity
function setInstantWithdrawalEnabled(address asset, bool enabled, uint256 buffer)
    external onlySupportedAsset(asset) onlyLRTManager
{
    isInstantWithdrawalEnabled[asset] = enabled;
    if (enabled) {
        ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT))
            .setQueuedWithdrawalsBuffer(asset, buffer);
    }
    emit InstantWithdrawalEnabledUpdated(asset, enabled);
}
```

This eliminates the window between enabling instant withdrawal and configuring the buffer. Alternatively, enforce a non-zero buffer requirement inside `setInstantWithdrawalEnabled` by reverting if the buffer is not already set.

## Proof of Concept
1. Protocol has 100 stETH in `LRTUnstakingVault`. Users A, B, C have called `initiateWithdrawal`, locking 80 stETH worth of rsETH in `LRTWithdrawalManager` (`assetsCommitted[stETH] = 80e18`).
2. Manager calls `setInstantWithdrawalEnabled(stETH, true)`. `queuedWithdrawalsBuffer[stETH]` remains `0`.
3. Attacker (any rsETH holder) calls `instantWithdrawal(stETH, largeRsETHAmount)`. `getAssetsAvailableForInstantWithdrawal(stETH)` returns `100e18` (full vault balance). Attacker redeems up to 100 stETH from the vault.
4. Operator calls `unlockQueue(stETH, ...)`. `_createUnlockParams` reads `unstakingVault.balanceOf(stETH) = 0`, setting `params.totalAvailableAssets = 0`. The call reverts with `AmountMustBeGreaterThanZero`.
5. Users A, B, C cannot complete their withdrawals. Their rsETH is locked in `LRTWithdrawalManager` with no cancellation path. Funds are frozen until the vault is replenished.

**Foundry test sketch:**
```solidity
function test_instantWithdrawalDrainsQueuedAssets() public {
    // Setup: fund vault, queue withdrawals for users A/B/C
    // Manager enables instant withdrawal (no buffer set)
    withdrawalManager.setInstantWithdrawalEnabled(stETH, true);
    // Attacker drains vault
    vm.prank(attacker);
    withdrawalManager.instantWithdrawal(stETH, attackerRsETH, "");
    // Operator tries to unlock queue — should revert
    vm.expectRevert(AmountMustBeGreaterThanZero.selector);
    withdrawalManager.unlockQueue(stETH, type(uint256).max, ...);
}
```
