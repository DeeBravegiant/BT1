### Title
Unconditional cache write in `request_and_validate_additions` allows malicious peer to bypass Merkle proof validation on reconnect — (`chia/wallet/util/wallet_sync_utils.py`)

### Summary
`request_and_validate_additions` writes to `peer_request_cache._additions_in_block` unconditionally after calling `validate_additions`, even when validation returns `False`. Because the per-peer cache is never evicted on disconnect, a malicious peer can poison the cache with a failed proof, reconnect with the same node identity, and have subsequent coin-state validations for the same `(header_hash, puzzle_hash)` pair return `True` from cache without any Merkle proof check.

### Finding Description
In `chia/wallet/util/wallet_sync_utils.py`:

```python
result: bool = validate_additions(
    additions_res.coins,
    additions_res.proofs,
    additions_root,
)
peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)  # line 223 — unconditional
return result
```

The fix should be `if result: peer_request_cache.add_to_additions_in_block(...)`.

The `PeerRequestCache` is stored in `WalletNode.peer_caches: dict[bytes32, PeerRequestCache]` keyed by `peer.peer_node_id`. No code removes or resets this entry when a peer disconnects. When the same peer reconnects (same node key → same `peer_node_id`), `get_cache_for_peer` returns the existing, poisoned cache.

### Impact Explanation
A malicious peer can make the wallet accept coin states (e.g., fake received XCH or CAT payments) without a valid Merkle inclusion proof. The wallet's coin records and displayed balance are corrupted. This maps to **High: Corruption of coin records / wallet sync state with direct security impact**.

### Likelihood Explanation
Any unprivileged peer can trigger this. The attacker only needs to:
1. Connect and respond to a `RequestAdditions` with a malformed `RespondAdditions` (e.g., `len(coins) != len(proofs)`).
2. Accept the forced disconnect.
3. Reconnect with the same node key and send the target coin state again.

No keys, admin access, or cryptographic breaks are required.

### Recommendation
Guard the cache write:
```python
result: bool = validate_additions(
    additions_res.coins,
    additions_res.proofs,
    additions_root,
)
if result:
    peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)
return result
```

### Proof of Concept
1. Stand up a wallet node in untrusted-peer mode.
2. Connect a custom peer that, upon receiving `RequestAdditions` for block H / puzzle-hash P, replies with `RespondAdditions(coins=[(P, [fake_coin])], proofs=[(P, b"", b""), (P, b"", b"")])` — length mismatch triggers `validate_additions → False`.
3. Observe: `add_to_additions_in_block(H, P, height)` is called; peer is disconnected.
4. Reconnect the same peer (same node key). Send a `CoinStateUpdate` containing a `CoinState` for `fake_coin` at height H.
5. Observe: `request_and_validate_additions` hits the cache and returns `True`; `validate_received_state_from_peer` returns `True`; `fake_coin` is written to the wallet's coin store.
