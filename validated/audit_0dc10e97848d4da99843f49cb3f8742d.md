### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Replay by a Single Byzantine MPC Node - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over the caller-supplied `response.payload_hash` using the root public key. It never checks that `response.payload_hash` is actually the canonical hash of the `request` content (`SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))`). A single Byzantine attested participant can therefore replay a previously produced signature — one that was legitimately generated for a different foreign-chain request — against any currently pending `VerifyForeignTransactionRequest`, causing the contract to deliver a forged `VerifyForeignTransactionResponse` to every caller waiting on that request.

---

### Finding Description

**Root cause — missing binding between `request` and `response.payload_hash`**

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` (lines 691–754) accepts two independent parameters:

- `request: VerifyForeignTransactionRequest` — used only as the map key to look up and drain the pending yield queue.
- `response: VerifyForeignTransactionResponse` — contains `payload_hash` and `signature`, which are verified only against each other. [1](#0-0) 

The signature check at lines 726–734 verifies:

```
ECDSA.verify(response.signature, response.payload_hash, root_public_key) == OK
```

There is no step that computes the expected hash from `request.request` (the `ForeignChainRpcRequest` embedded in the pending-queue key) and asserts it equals `response.payload_hash`. The `VerifyForeignTransactionRequest` struct carries the full chain-specific request: [2](#0-1) 

The canonical hash the MPC nodes are supposed to sign is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request, values })))
``` [3](#0-2) 

Because the contract never reconstructs or checks this hash, the `request` and `response` parameters are completely decoupled at the contract level.

**Attack path — cross-request replay by one Byzantine node**

1. Byzantine attested participant P observes the MPC network complete signing for request **R1** (e.g., `verify_foreign_transaction(bitcoin_tx_123)`). P records the resulting `(payload_hash=H1, signature=S1)`.
2. A new request **R2** (e.g., `verify_foreign_transaction(evil_tx)`) enters the pending queue. `evil_tx` may be unconfirmed or entirely fabricated.
3. P calls `respond_verify_foreign_tx(request=R2, response={payload_hash=H1, signature=S1})`.
4. The contract checks: is `S1` a valid ECDSA signature over `H1` under the root key? **Yes** — it was produced by the honest MPC network for R1.
5. `pending_requests::resolve_yields_for` drains **all** queued yields for R2 and delivers `{payload_hash=H1, signature=S1}` to every caller waiting on R2. [4](#0-3) 

The callers of R2 receive a `VerifyForeignTransactionResponse` whose `payload_hash` is the hash of R1's content, not R2's. The contract has accepted and forwarded a forged verification response.

**SDK check is off-chain and not enforced by the contract**

The `near-mpc-sdk` does provide a `verify_signature` helper that checks `payload_hash` against the locally reconstructed expected hash: [5](#0-4) 

However, this check is entirely off-chain and optional. The on-chain contract — the only authoritative source of truth — never performs it. Any bridge or dApp that trusts the contract's response without independently verifying `payload_hash` is vulnerable.

---

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can cause the MPC contract to deliver a `VerifyForeignTransactionResponse` whose `payload_hash` is bound to a completely different foreign-chain transaction than the one the caller requested. A bridge contract that:

- calls `verify_foreign_transaction(evil_tx)`,
- receives `{payload_hash=H1, signature=S1}`,
- verifies the ECDSA signature is valid (it is — over H1), and
- does not independently check that H1 encodes `evil_tx`

will conclude that `evil_tx` was verified by the MPC network and release funds, enabling a double-spend or invalid bridge execution. This matches the **High** impact category: *forged foreign-chain verification that causes invalid bridge execution or double-spend conditions*.

Additionally, because `resolve_yields_for` drains the entire fan-out queue for R2, the Byzantine node simultaneously prevents any legitimate response from ever reaching the callers of R2.

---

### Likelihood Explanation

The attacker must be a single attested MPC participant — the minimum privileged role in the system and explicitly within the allowed Byzantine attacker model ("Byzantine participant strictly below the signing threshold"). No threshold collusion is required. The attacker only needs to:

1. Participate honestly in at least one prior signing round to obtain a valid `(payload_hash, signature)` pair.
2. Submit a single NEAR transaction calling `respond_verify_foreign_tx` with the recycled pair against any pending R2.

Both conditions are trivially achievable by any participant who has been active for more than one signing epoch.

---

### Recommendation

In `respond_verify_foreign_tx`, after verifying the ECDSA signature, add a binding check that `response.payload_hash` is consistent with `request`. Because the contract does not have the extracted `values` (they are off-chain), the most robust fix is to include the extracted values in `VerifyForeignTransactionResponse` and have the contract recompute and compare the hash:

```rust
// Proposed addition inside respond_verify_foreign_tx, after signature check:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // add field to response
});
let expected_hash = expected_payload
    .compute_msg_hash()
    .map_err(|_| RespondError::InvalidPayloadHash)?;
if expected_hash != response.payload_hash {
    return Err(RespondError::InvalidPayloadHash.into());
}
```

If including extracted values in the response is undesirable (e.g., size constraints), an alternative is to restructure the sign payload so that `payload_hash` is derived solely from `request` (the chain-specific RPC request), allowing the contract to recompute it without the off-chain `values`.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
