Audit Report

## Title
`maxFeeMintAmountPerDay` Uninitialized Causes `updateRSETHPrice()` to Revert and Permanently Freezes Protocol Fee Yield — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle.reinitialize()` sets `feePeriodStartTime` but never assigns `maxFeeMintAmountPerDay`, leaving it at its default value of `0`. The internal function `_checkAndUpdateDailyFeeMintLimit()` unconditionally reverts when `feeAmount > maxFeeMintAmountPerDay`, so any call to `updateRSETHPrice()` that computes a non-zero protocol fee will revert with `DailyFeeMintLimitExceeded`. The treasury permanently receives no fee yield until an admin separately calls `setMaxFeeMintAmountPerDay()`.

## Finding Description

Three state variables govern daily fee minting:

```solidity
// contracts/LRTOracle.sol L32-35
uint256 public currentPeriodMintedFeeAmount;
uint256 public feePeriodStartTime;
uint256 public maxFeeMintAmountPerDay;
```

`reinitialize()` (L72-79) sets `feePeriodStartTime` but leaves `maxFeeMintAmountPerDay` at `0`:

```solidity
function reinitialize(uint256 _feePeriodStartTime) external reinitializer(2) onlyLRTManager {
    ...
    feePeriodStartTime = _feePeriodStartTime;   // maxFeeMintAmountPerDay never touched
}
```

`_checkAndUpdateDailyFeeMintLimit()` (L197-210) has no guard for the zero case:

```solidity
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
```

When `maxFeeMintAmountPerDay == 0` and `feeAmount > 0`, the condition `feeAmount > 0` is always `true` → always reverts.

`_updateRsETHPrice()` (L298-303) calls this unconditionally whenever `protocolFeeInETH > 0`:

```solidity
if (protocolFeeInETH > 0) {
    uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
    _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);  // reverts
    ...
}
```

The asymmetry is confirmed by `remainingDailyFeeMintLimit()` (L170-171), which explicitly handles the zero case that `_checkAndUpdateDailyFeeMintLimit()` does not:

```solidity
function remainingDailyFeeMintLimit() external view returns (uint256) {
    if (maxFeeMintAmountPerDay == 0) return 0;  // view handles it; internal function does not
```

`protocolFeeInETH > 0` is reached whenever `protocolFeeInBPS > 0` (set via `LRTConfig.setProtocolFeeBps()`, L196-199) and `totalETHInProtocol > previousTVL` (L244-247), both of which are the normal production state.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

The treasury address configured in `LRTConstants.PROTOCOL_TREASURY` never receives its rsETH fee share. Every call to the public `updateRSETHPrice()` or manager-only `updateRSETHPriceAsManager()` reverts as long as `maxFeeMintAmountPerDay == 0` and fees are non-zero. The protocol fee yield is permanently inaccessible until an admin intervenes, constituting a permanent freezing of unclaimed yield for the protocol treasury.

## Likelihood Explanation

The conditions are the expected production state: `protocolFeeInBPS > 0` is set by a MANAGER via `LRTConfig.setProtocolFeeBps()`, and TVL growth (rewards accruing) is the normal operating condition. `reinitialize()` sets `feePeriodStartTime` but silently omits `maxFeeMintAmountPerDay`, making it trivially easy to deploy the fee-minting feature in a broken state with no initialization-time error. No attacker action is required — the revert is triggered by any public caller of `updateRSETHPrice()` once both conditions hold.

## Recommendation

Initialize `maxFeeMintAmountPerDay` inside `reinitialize()`:

```solidity
function reinitialize(uint256 _feePeriodStartTime, uint256 _maxFeeMintAmountPerDay)
    external reinitializer(2) onlyLRTManager
{
    ...
    feePeriodStartTime = _feePeriodStartTime;
    maxFeeMintAmountPerDay = _maxFeeMintAmountPerDay;
}
```

Alternatively, align `_checkAndUpdateDailyFeeMintLimit()` with the existing behavior of `remainingDailyFeeMintLimit()` by treating `maxFeeMintAmountPerDay == 0` as "no cap configured":

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (maxFeeMintAmountPerDay == 0) return; // no cap configured, skip check
    ...
}
```

## Proof of Concept

1. Deploy `LRTOracle` and call `initialize(lrtConfigAddr)` — `maxFeeMintAmountPerDay` is `0`.
2. Call `reinitialize(_feePeriodStartTime)` — `feePeriodStartTime` is set, `maxFeeMintAmountPerDay` remains `0`.
3. Call `LRTConfig.setProtocolFeeBps(500)` (5% fee, callable by MANAGER).
4. Allow EigenLayer rewards to accrue so `totalETHInProtocol > previousTVL`.
5. Any address calls `updateRSETHPrice()`.
6. Inside `_updateRsETHPrice()`: `protocolFeeInETH > 0` → `rsethAmountToMintAsProtocolFee > 0` → `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` → `0 + rsethAmountToMintAsProtocolFee > 0` → **reverts with `DailyFeeMintLimitExceeded`**.
7. Treasury receives no fees; price update never completes.

**Foundry test sketch:**
```solidity
function test_updateRSETHPrice_revertsWhenMaxFeeMintNotSet() public {
    // setup: reinitialize sets feePeriodStartTime, not maxFeeMintAmountPerDay
    oracle.reinitialize(block.timestamp - 1 hours);
    lrtConfig.setProtocolFeeBps(500);
    // simulate TVL increase by mocking getTotalAssetDeposits to return higher value
    vm.expectRevert(abi.encodeWithSelector(DailyFeeMintLimitExceeded.selector, ...));
    oracle.updateRSETHPrice();
}
```
