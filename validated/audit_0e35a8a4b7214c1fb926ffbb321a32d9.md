Audit Report

## Title
Unbounded `referralId` String in `deposit()` Functions Causes Unbounded Gas Consumption - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All pool `deposit()` entry points accept a caller-controlled `string memory referralId` parameter with no length bound. The string is copied into EVM memory and emitted as event data with zero validation. Because EVM memory expansion costs grow quadratically and `LOG` opcodes charge 8 gas per byte of event data, a sufficiently large `referralId` causes unbounded gas consumption, enabling block stuffing by any unprivileged depositor on cheap L2 networks.

## Finding Description
Every pool contract exposes public `deposit()` overloads that accept an unbounded `string memory referralId`:

**`RSETHPoolV3.sol` L246–265:**
```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER) {
    ...
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```
**`RSETHPoolV3.sol` L271–293:**
```solidity
function deposit(address token, uint256 amount, string memory referralId) external nonReentrant whenNotPaused ... {
    ...
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId, token);
}
```

The identical pattern is confirmed in `RSETHPoolV3ExternalBridge.sol` (L366–384), `RSETHPoolNoWrapper.sol` (L231–244), `RSETHPool.sol` (L265–278), and `RSETHPoolV2ExternalBridge.sol` (L289–301). In every case, `referralId` flows directly from calldata into an event emission with no `bytes(referralId).length` check anywhere in the call path — not in the function body, not in any modifier.

Root cause: EVM memory expansion gas cost is approximately `n/32 + (n/32)²/512` words, growing quadratically. Additionally, `LOG3`/`LOG4` opcodes charge 8 gas per byte of non-indexed event data. A 500 KB `referralId` incurs millions of gas units from these two sources alone, before any other execution cost. No guard prevents this.

## Impact Explanation
**Medium — Unbounded gas consumption / Low — Block stuffing.**

On every L2 where these pools are deployed (Arbitrum, Optimism, Base), execution gas is orders of magnitude cheaper than on L1. An attacker can repeatedly call `deposit{value: minDeposit}(largeString)` with a multi-megabyte `referralId`, consuming the majority of each block's gas budget and preventing legitimate depositors from having transactions included. The attacker's cost is bounded only by cheap L2 gas fees.

## Likelihood Explanation
Any unprivileged external account holding a minimal amount of ETH or a supported LST can trigger this. No role, whitelist, or special condition is required. The `deposit()` functions are fully public, and the `referralId` parameter is entirely caller-controlled with no on-chain constraint. The attack is trivially repeatable across consecutive blocks.

## Recommendation
Add a maximum length check at the top of every `deposit()` overload before any other logic executes:

```solidity
uint256 constant MAX_REFERRAL_ID_LENGTH = 128;

function deposit(string memory referralId) external payable ... {
    if (bytes(referralId).length > MAX_REFERRAL_ID_LENGTH) revert ReferralIdTooLong();
    ...
}
```

Apply the same guard to the `(address token, uint256 amount, string memory referralId)` overload in all six affected pool contracts. The check must appear before any modifier body executes.

## Proof of Concept

```solidity
contract BlockStuffer {
    IPool pool;
    constructor(address _pool) { pool = IPool(_pool); }

    function stuff() external payable {
        // Build a 500 KB referralId string
        string memory bigId = string(new bytes(500_000));
        // Deposit the minimum viable ETH; gas cost dominated by string
        pool.deposit{value: msg.value}(bigId);
    }
}
```

**Foundry fork test plan:**
1. Fork an L2 (e.g., Optimism or Base) at a recent block.
2. Deploy `BlockStuffer` pointing at the live `RSETHPoolV3` address.
3. Call `stuff{value: 1 wei}()` and record `gasleft()` before and after.
4. Verify gas consumed exceeds several million units from memory expansion and `LOG` data costs alone.
5. Repeat across consecutive blocks to demonstrate block stuffing of the deposit queue.
