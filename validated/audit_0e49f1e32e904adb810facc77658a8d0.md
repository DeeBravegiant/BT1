### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Submitted `request`, Enabling Cross-Request Signature Replay by a Single Byzantine Leader — (`crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function verifies that `response.signature` is a valid threshold ECDSA signature over `response.payload_hash` using the root public key, but it never checks that `response.payload_hash` was actually derived from the `request` argument. A single Byzantine attested participant acting as signing leader can replay a legitimately-produced threshold signature (computed for one pending request) as the response to a completely different pending request, delivering a forged foreign-chain verification result to the second caller.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract performs two independent checks and then resolves the pending yield:

1. Verify `response.signature` over `response.payload_hash` using the root key.
2. Look up `request` in `pending_verify_foreign_tx_requests` and drain its yield queue. [1](#0-0) 

There is no step that ties `response.payload_hash` to `request`. The `payload_hash` is supposed to be `SHA-256(borsh(ForeignTxSignPayload{request: rpc_request, values: extracted_values}))`, where `rpc_request` comes from the pending `request`. The contract never recomputes or validates this binding.

On the node side, `build_signature_request` correctly derives the payload hash from the actual foreign-chain data: [2](#0-1) 

And the leader assembles the respond args from the computed hash: [3](#0-2) 

However, the contract never enforces this binding. A malicious leader can call `respond_verify_foreign_tx` directly on-chain with any `(request, payload_hash, signature)` triple where the signature is valid over `payload_hash` under the root key — regardless of whether `payload_hash` encodes the foreign-chain data for `request`.

The `ForeignTxSignPayload` struct that defines what gets hashed and signed: [4](#0-3) 

The hash encodes `(ForeignChainRpcRequest, Vec<ExtractedValue>)`. The contract never re-derives this hash from the `request` it is resolving.

---

### Impact Explanation

**High — Forged foreign-chain verification enabling invalid bridge execution.**

A malicious leader can deliver a threshold signature that was honestly produced for request R1 (attesting to R1's foreign-chain data) as the verified response for a different pending request R2. The user of R2 receives `(payload_hash_R1, sig_R1)` — a cryptographically valid signature, but one that attests to R1's transaction data, not R2's. Any bridge contract that calls `verify_foreign_transaction` and then acts on the returned `payload_hash` (e.g., to release funds or mint tokens) will be operating on forged attestation data. This directly enables invalid bridge execution and potential double-spend conditions.

---

### Likelihood Explanation

The attack requires a single Byzantine attested participant to act as the signing leader for one request while simultaneously submitting its signature as the response for a different pending request. This is strictly below the signing threshold — no collusion with other nodes is needed. The leader already possesses the assembled threshold signature after the MPC round completes. The only prerequisite is that two distinct `verify_foreign_transaction` requests are pending at the same time, which is a normal production condition for any active bridge.

---

### Recommendation

In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` argument and the `response.payload_hash`, or require the contract to verify that `payload_hash` encodes the correct `ForeignChainRpcRequest`. Concretely, the contract should reject any response where `payload_hash` does not match a hash derived from the `request.request` field. Since the contract cannot re-execute the foreign-chain RPC, the minimal fix is to include the `ForeignChainRpcRequest` (from the pending `request`) as a domain-separation prefix in the signed payload and verify it on-chain — or, equivalently, store the expected `payload_hash` alongside the pending request at submission time (if the hash can be partially pre-committed) and check it on respond.

At minimum, the `payload_hash` should be verified to encode the same `ForeignChainRpcRequest` as the `request` being resolved, preventing cross-request replay.

---

### Proof of Concept

1. Two users submit `verify_foreign_transaction` for distinct foreign-chain transactions: R1 (Bitcoin tx A) and R2 (Bitcoin tx B). Both are pending in `pending_verify_foreign_tx_requests`.
2. The Byzantine leader participates in the MPC signing round for R1. The MPC network produces threshold signature `S` over `H1 = SHA-256(borsh(ForeignTxSignPayload{request: R1_rpc, values: V1}))`.
3. Instead of calling `respond_verify_foreign_tx(R1, H1, S)`, the leader calls `respond_verify_foreign_tx(R2, H1, S)`.
4. The contract checks: is `S` a valid ECDSA signature over `H1` under the root key? **Yes** — `S` was honestly produced by the threshold network.
5. The contract checks: is `R2` in `pending_verify_foreign_tx_requests`? **Yes**.
6. The contract resolves R2's yield with `VerifyForeignTransactionResponse { payload_hash: H1, signature: S }`.
7. The user of R2 receives a response claiming their Bitcoin tx B was verified, but the signature attests to tx A's data (`H1`). Any downstream bridge contract verifying `S` over `H1` will succeed, but will be acting on forged attestation. [5](#0-4) [6](#0-5) [2](#0-1)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
