### Title
Missing ExUnits Limit Enforcement for Sub-Transactions Allows Block-Level Execution Budget Bypass — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

In the Dijkstra era, the `dijkstraSubUtxoTransition` rule omits the `validateExUnitsTooBigUTxO` check that is mandatory in the top-level `utxoTransition`. Simultaneously, the block-level `Alonzo.validateExUnits` call in `dijkstraBbodyTransition` uses `foldMap totExUnits txs`, where `totExUnits` only reads redeemers from the **top-level** transaction's witness set and is never overridden for Dijkstra. Sub-transaction redeemers are invisible to both checks. Because sub-transactions carry no fee field, there is also no economic deterrent against declaring arbitrarily large ExUnits in sub-transaction redeemers. A single crafted top-level transaction containing many sub-transactions with Plutus scripts can therefore cause nodes to perform computation that far exceeds `maxBlockExUnits` while the block-level guard reports zero or minimal ExUnits consumed.

---

### Finding Description

**Root cause 1 — missing per-sub-transaction ExUnits check.**

`dijkstraSubUtxoTransition` in `SubUtxo.hs` validates sub-transactions but never calls `validateExUnitsTooBigUTxO`: [1](#0-0) 

Compare with the top-level rule, which explicitly enforces the limit: [2](#0-1) 

The `DijkstraSubUtxoPredFailure` type has no `ExUnitsTooBigUTxO` constructor, confirming no such check exists anywhere in the sub-transaction path: [3](#0-2) 

The mapping function `dijkstraUtxoToDijkstraSubUtxoPredFailure` explicitly marks `ExUnitsTooBigUTxO` as `error "Impossible"`, confirming the design gap: [4](#0-3) 

**Root cause 2 — block-level check ignores sub-transaction ExUnits.**

`dijkstraBbodyTransition` delegates to the unmodified Alonzo block-level check: [5](#0-4) 

`Alonzo.validateExUnits` computes `foldMap totExUnits txs`: [6](#0-5) 

`totExUnits` reads only the top-level transaction's redeemer map: [7](#0-6) 

No Dijkstra-specific override of `totExUnits` exists in the Dijkstra source tree. Sub-transaction redeemers, stored in each sub-transaction's own witness set, are never folded in.

**Root cause 3 — sub-transactions carry no fee, removing the economic deterrent.**

`DijkstraTxBodyRaw SubTx` has no `fee` field: [8](#0-7) 

In contrast, the top-level `DijkstraTxBodyRaw TopTx` includes `dtbrFee`: [9](#0-8) 

The minimum-fee formula (`alonzoMinFeeTx`) adds `txscriptfee prices allExunits` for the top-level transaction only. Sub-transaction script costs are not priced.

**Sub-transactions do execute Plutus scripts.**

`mkDijkstraStAnnSubTx` calls `resolveNeededPlutusScriptsWithPurpose` and populates `dsastPlutusScriptsWithContext` for each sub-transaction: [10](#0-9) 

The top-level annotation aggregates sub-transaction script contexts: [11](#0-10) 

---

### Impact Explanation

An attacker submits one top-level Dijkstra transaction containing `N` sub-transactions. Each sub-transaction declares redeemers with ExUnits equal to `maxTxExUnits`. The top-level transaction itself carries no scripts (ExUnits = 0). The block-level guard (`Alonzo.validateExUnits`) observes 0 ExUnits for this transaction and accepts the block. Nodes must actually execute `N × maxTxExUnits` worth of Plutus computation — unbounded by any protocol parameter — to validate the block. This directly exceeds the intended `maxBlockExUnits` validation limit, which exists precisely to bound per-block validation time and memory.

**Matched allowed impact**: *Medium — Attacker-controlled transactions exceed intended validation limits.*

---

### Likelihood Explanation

Any unprivileged transaction sender in the Dijkstra era can craft such a transaction. No special key, governance role, or majority is required. The only prerequisite is the ability to submit a valid Dijkstra-era transaction, which is the standard attacker model for the ledger. The `OMap TxId (Tx SubTx era)` field in the transaction body accepts an unbounded number of sub-transactions (no protocol parameter caps the count), amplifying the effect linearly. [12](#0-11) 

---

### Recommendation

1. **Add a per-sub-transaction ExUnits check** in `dijkstraSubUtxoTransition` analogous to the top-level check:
   ```haskell
   runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
   ```
   and add a corresponding `SubExUnitsTooBigUTxO` constructor to `DijkstraSubUtxoPredFailure`.

2. **Fix the block-level aggregation** so that `totExUnits` (or a Dijkstra-specific replacement) folds over sub-transaction redeemers as well as top-level redeemers before comparing against `ppMaxBlockExUnitsL`.

3. **Consider a protocol parameter** capping the number of sub-transactions per top-level transaction to bound the multiplicative amplification factor independently of ExUnits.

---

### Proof of Concept

```
Attacker constructs:
  TopTx {
    inputs = [some_utxo],
    fee    = minFee (no scripts in top-level),
    subTransactions = OMap [
      SubTx_1 { redeemers = [(purpose_1, (data, ExUnits maxMem maxSteps))] },
      SubTx_2 { redeemers = [(purpose_1, (data, ExUnits maxMem maxSteps))] },
      ...
      SubTx_N { redeemers = [(purpose_1, (data, ExUnits maxMem maxSteps))] }
    ]
  }

Block-level check:
  foldMap totExUnits [TopTx] = ExUnits 0 0   -- sub-tx redeemers invisible
  0 ≤ maxBlockExUnits  →  PASSES

Actual node computation:
  N × maxTxExUnits  >>  maxBlockExUnits
```

The block is accepted by the ledger rules while nodes perform `N × maxTxExUnits` of Plutus evaluation, with `N` limited only by the transaction size (`maxTxSize`), not by any ExUnits parameter.
