Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
When all stakers call `initiateWithdrawal()` during an active reward window, `totalKernelStaked` immediately drops to zero, halting reward accrual in `rewardPerToken()`. The reward tokens that should have been distributed during the zero-staker interval are never allocated to any user and are not recycled into future reward periods, leaving them permanently locked in the contract.

## Finding Description
`rewardPerToken()` guards against division-by-zero by returning the stored value unchanged when `totalKernelStaked == 0`: [1](#0-0) 

`initiateWithdrawal()` decrements `totalKernelStaked` immediately — before the unlock delay — so the staked supply drops to zero the moment the last staker initiates withdrawal: [2](#0-1) 

The `updateReward` modifier snapshots `updatedAt = lastTimeRewardApplicable()` at the moment of the last withdrawal. From that point forward, `rewardPerToken()` returns `rewardPerTokenStored` unchanged for the entire zero-staker interval: [3](#0-2) 

When a new staker eventually calls `stake()`, `updateReward` fires before `totalKernelStaked` is incremented (it is still 0 at that instant), so `rewardPerTokenStored` remains unchanged and `updatedAt` is advanced to the current time — permanently skipping the gap.

When `notifyRewardAmount()` is called after `finishAt`, the new rate is computed solely from the freshly deposited amount: [4](#0-3) 

The stranded tokens are not `remaining` (that term covers only future-rate rewards) and are not `receivedAmount`. They sit in the contract balance with no code path to recover or redistribute them. The contract's own NatSpec acknowledges this at L17–23 but relies entirely on an off-chain operational assumption with no on-chain enforcement: [5](#0-4) 

The guard in `notifyRewardAmount()` (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new period with zero stakers; it does not prevent stakers from withdrawing during an already-active period: [6](#0-5) 

## Impact Explanation
Reward tokens emitted during the zero-staker interval are permanently frozen in the contract. They cannot be claimed by any user and are not recycled into future reward periods. This is a concrete instance of **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
`initiateWithdrawal()` is a permissionless, standard user action callable by any staker at any time. No coordination among stakers is required — each independently decides to exit. The withdrawal delay (up to 30 days) delays token transfer but does not prevent `totalKernelStaked` from hitting zero immediately upon initiation. Market conditions, end-of-season behavior, or loss of confidence can independently drive all stakers to exit during an active reward window.

## Recommendation
In `notifyRewardAmount()`, when `block.timestamp >= finishAt`, compute the unallocated surplus (contract reward token balance minus the sum of all pending user rewards) and fold it into `receivedAmount` before calculating the new `rewardRate`. Alternatively, add an admin-only sweep function callable only after `finishAt` that transfers any reward surplus (contract balance minus owed rewards) to a treasury address.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with 1,000 KERNEL staked → `rewardRate = 1_000e18 / duration`.
2. The sole staker calls `initiateWithdrawal(1_000)` → `totalKernelStaked = 0`, `updatedAt` snapped to current block.
3. The full reward window (e.g., 30 days) elapses. `rewardPerToken()` returns `rewardPerTokenStored` unchanged throughout; ~1,000e18 reward tokens accrue to no one.
4. A new staker calls `stake(1)`. `updateReward` fires with `totalKernelStaked` still 0 → `rewardPerTokenStored` unchanged, `updatedAt` advanced to now.
5. Admin calls `notifyRewardAmount(100e18)` after `finishAt` → `rewardRate = 100e18 / duration`. The original ~1,000e18 tokens remain in the contract balance, permanently unclaimable.

Foundry test plan: deploy `KernelDepositPool`, stake 1,000 tokens, call `notifyRewardAmount`, warp past `finishAt` with `totalKernelStaked == 0` throughout, call `notifyRewardAmount` again with a small amount, then assert `rewardsToken.balanceOf(address(pool)) > sum_of_all_claimable_rewards`.
