### Title
`sign_transfer` Permanently Locks User Funds When Normalized Transfer Amount Rounds to Zero - (`near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` accepts and burns/locks user tokens for any amount satisfying `fee < amount`, but `sign_transfer` later rejects the transfer with `InvalidAmountToTransfer` if `normalize_amount(amount_without_fee())` floors to zero due to decimal-scale differences. Because there is no user-callable cancel path, the tokens are permanently unclaimable.

### Finding Description

The NEAR bridge contract stores a transfer message and burns/locks user tokens in `init_transfer_internal` before any decimal normalization is performed. [1](#0-0) 

The only guard at ingress is `fee.fee < amount`: [2](#0-1) 

Decimal normalization ŌĆö which uses floor division ŌĆö is deferred entirely to `sign_transfer`: [3](#0-2) 

The helper itself: [4](#0-3) 

For any token whose `origin_decimals > decimals` (e.g., 24 ŌåÆ 18, a 10ŌüČ scale factor), any `amount_without_fee()` below `10^(origin_decimals ŌłÆ decimals)` normalizes to 0. `sign_transfer` then panics with `ERR_INVALID_AMOUNT_TO_TRANSFER` on every call, and the transfer message stays in `pending_transfers` indefinitely.

The code's own comment acknowledges the dust case but only for the fee path: [5](#0-4) 

There is no user-callable cancel or DAO rescue function for `pending_transfers`. `remove_transfer_message` is only called internally from `sign_transfer_callback` (on success) and `claim_fee_callback`: [6](#0-5) 

### Impact Explanation

Tokens are burned (for bridge-deployed tokens) or locked (for native tokens) in `init_transfer_internal` before the normalization check. Once the transfer message is stored and tokens are gone, no on-chain path exists to recover them. This is an irreversible fund lock matching the "Critical ŌĆö Irreversible fund lock, frozen redemption path, or permanently unclaimable user value" category.

### Likelihood Explanation

The scenario requires a user to send an amount below the normalization threshold for a token with a decimal gap. For a 24ŌåÆ18 decimal token the threshold is 1,000,000 base units (one micro-token). This is a small but realistic amount for dust transfers or automated systems. The protocol provides no warning and accepts the transfer silently, making accidental loss plausible.

### Recommendation

Add a normalization check inside `init_transfer` (or `init_transfer_internal`) before burning/locking tokens:

```rust
// In init_transfer, after computing transfer_message:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(addr) = token_address {
    if let Some(decimals) = self.token_decimals.get(&addr) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

This mirrors the fix suggested in the reference report: validate the problematic component at the entry point rather than deep in the settlement path, so the transaction reverts cleanly and the user retains their tokens.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (scale factor 10ŌüČ).
2. Call `ft_transfer_call` with `amount = 500_000` (below the 10ŌüČ threshold) and `fee = 0`.
3. `init_transfer_internal` burns the 500,000 base units and stores the transfer message.
4. Any trusted relayer calls `sign_transfer` for the resulting `transfer_id`.
5. `normalize_amount(500_000, {24, 18})` = `500_000 / 1_000_000` = 0.
6. `require!(0 > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. Repeat step 4 indefinitely ŌĆö the result is always the same panic.
8. The 500,000 base units are permanently burned with no recovery path.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,

            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }

    /// Uses floor division ŌĆö any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
