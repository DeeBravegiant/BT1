Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables DoS of Both Oracle Price Update Paths via `DailyFeeMintLimitExceeded` - (File: contracts/FeeReceiver.sol / contracts/LRTOracle.sol)

## Summary
`FeeReceiver.sendFunds()` has no access control, allowing any caller to flush arbitrary ETH into `LRTDepositPool`. This inflates the protocol's TVL, causing `_updateRsETHPrice()` to compute a protocol fee that exceeds `maxFeeMintAmountPerDay`. Because `_checkAndUpdateDailyFeeMintLimit()` reverts unconditionally on overflow, and both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` call the same `_updateRsETHPrice()` without a bypass for the daily limit check, both oracle update paths are permanently blocked until admin intervention.

## Finding Description

**Root cause — permissionless ETH injection:**

`FeeReceiver.sendFunds()` carries no role modifier: [1](#0-0) 

It forwards the full ETH balance to `LRTDepositPool.receiveFromRewardReceiver()`, which is also unrestricted: [2](#0-1) 

**Injected ETH is immediately counted in TVL:**

`getETHDistributionData()` reads the raw ETH balance of the deposit pool: [3](#0-2) 

Every wei routed through `sendFunds()` is therefore reflected in `totalETHInProtocol` on the next oracle update.

**Inflated TVL produces a fee that exceeds the daily limit:**

Inside `_updateRsETHPrice()`, when `totalETHInProtocol > previousTVL`, a protocol fee is computed: [4](#0-3) 

The resulting `rsethAmountToMintAsProtocolFee` is passed to `_checkAndUpdateDailyFeeMintLimit()`: [5](#0-4) 

`_checkAndUpdateDailyFeeMintLimit()` reverts unconditionally when the amount exceeds the cap: [6](#0-5) 

**Both price update paths are blocked:**

`updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` (manager-only) both call the same `_updateRsETHPrice()`: [7](#0-6) 

The manager role check at line 263 only gates the `PriceAboveDailyThreshold` revert path, not the `DailyFeeMintLimitExceeded` path. There is no manager bypass for the daily limit check, so `updateRSETHPriceAsManager()` also reverts.

**DoS persists across 24-hour resets:**

`_checkAndUpdateDailyFeeMintLimit()` resets `currentPeriodMintedFeeAmount` to 0 at the start of each new period: [8](#0-7) 

However, the attacker's ETH remains in the deposit pool. `previousTVL` is computed as `rsethSupply.mulWad(rsETHPrice)`, and since `rsETHPrice` never updates (the call always reverts), the inflated `totalETHInProtocol` continues to exceed `previousTVL` every period. The DoS repeats indefinitely without further attacker action.

## Impact Explanation

The rsETH price oracle is the critical pricing primitive for the protocol. With both update paths blocked, the price becomes permanently stale. Protocol operations that depend on a current price — including fee accrual and any downstream logic gated on a fresh oracle — are disrupted. Admin intervention (increasing `maxFeeMintAmountPerDay` or draining the injected ETH via privileged transfer functions) is required to recover. This constitutes **Medium — Temporary freezing of funds**, as the freeze is not self-healing at the 24-hour boundary and requires privileged action to resolve.

## Likelihood Explanation

- The attack requires only ETH; no tokens, approvals, or special roles are needed.
- `sendFunds()` is fully permissionless and callable by any EOA or contract.
- `maxFeeMintAmountPerDay` is a finite value set by the manager; any sufficiently large ETH injection will exceed it.
- The attacker's ETH is not lost — it increases TVL and therefore the value of any rsETH the attacker holds, making the net cost of the attack near zero.
- Likelihood: **High**.

## Recommendation

1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or a dedicated `OPERATOR_ROLE` so only trusted parties can flush rewards into the deposit pool.
2. **Cap rather than revert in `_checkAndUpdateDailyFeeMintLimit()`** — instead of reverting when the fee exceeds the daily limit, mint only up to the remaining limit and carry the excess to the next period. This eliminates the daily limit as a DoS vector.
3. **Provide a manager bypass for the daily limit check** — `updateRSETHPriceAsManager()` should be able to proceed even when the daily limit is exceeded, ensuring the manager can always recover the price oracle.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Preconditions:
//   maxFeeMintAmountPerDay = 1 wei
//   rsETHPrice = 1e18, rsethSupply > 0, protocolFeeInBPS > 0
//   feePeriodStartTime set within last 24 h

function testDoS() public {
    // 1. Seed FeeReceiver with 1 ETH
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    (bool ok,) = address(feeReceiver).call{value: 1 ether}("");
    require(ok);

    // 2. Flush ETH into deposit pool (permissionless)
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // 3. Public price update reverts with DailyFeeMintLimitExceeded
    vm.expectRevert(abi.encodeWithSelector(
        ILRTOracle.DailyFeeMintLimitExceeded.selector
    ));
    lrtOracle.updateRSETHPrice();

    // 4. Manager price update also reverts — no bypass exists
    vm.prank(manager);
    vm.expectRevert(abi.encodeWithSelector(
        ILRTOracle.DailyFeeMintLimitExceeded.selector
    ));
    lrtOracle.updateRSETHPriceAsManager();

    // 5. Advance 24 hours — DoS persists because ETH remains in pool
    vm.warp(block.timestamp + 1 days + 1);
    vm.expectRevert(abi.encodeWithSelector(
        ILRTOracle.DailyFeeMintLimitExceeded.selector
    ));
    lrtOracle.updateRSETHPrice();
}
```
