Audit Report

## Title
Stale Cross-Chain Rate in `AGETHRateReceiver` Allows Excess agETH Minting, Diluting Existing Holders — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/agETH/AGETHPoolV3.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no freshness check despite the contract tracking `lastUpdated` on every `lzReceive` call. `AGETHPoolV3.deposit()` uses this rate to mint agETH, so during a LayerZero message delay the stale-low rate causes the pool to mint more agETH than the deposited ETH warrants, transferring accrued yield from existing holders to the depositor.

## Finding Description

`CrossChainRateReceiver` stores `lastUpdated` at line 97 on every `lzReceive` call but `getRate()` at lines 103–105 returns the raw `rate` with no staleness guard: [1](#0-0) 

`AGETHRateReceiver` is a thin subclass that adds no additional checks: [2](#0-1) 

`AGETHPoolV3.getRate()` blindly proxies to the oracle: [3](#0-2) 

`viewSwapAgETHAmountAndFee` uses the rate directly in the mint calculation: [4](#0-3) 

`deposit(string)` calls this and mints unconditionally with no staleness revert path: [5](#0-4) 

**Exploit flow:**
1. LayerZero message is delayed; on-chain `rate` is stale-low (e.g., `1.05e18`) while the true rate on the source chain has advanced (e.g., `1.10e18`).
2. Attacker calls `deposit{value: 1 ether}("")`.
3. `agETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.9524 agETH` is minted instead of the correct `≈ 0.9091 agETH`.
4. The excess `~0.0433 agETH` is backed by no additional ETH, diluting the ETH-per-agETH ratio for all existing holders.

No existing check prevents this: `nonReentrant` guards against reentrancy only; `isEthDepositEnabled` is a kill-switch, not a staleness guard; and `IOracle.getRate()` has no freshness logic.

## Impact Explanation

The agETH-to-ETH rate is monotonically increasing (accumulated staking yield). A stale-low rate causes the mint formula to over-issue agETH. The unbacked agETH reduces the ETH-per-agETH ratio for all existing holders, effectively transferring their accrued but unclaimed yield to the attacker. This is a concrete instance of **High — Theft of unclaimed yield**. The SECURITY.md exclusion for "Incorrect data supplied by third-party oracles" does not apply: `AGETHRateReceiver` is the protocol's own contract, and the data was correct when received; the defect is the protocol's own failure to enforce freshness.

## Likelihood Explanation

LayerZero message delays are a realistic, non-adversarial operational condition (network congestion, bridge downtime, sequencer issues). The attacker requires no privileged access, no governance capture, and no key compromise. They only need to: (1) compare the on-chain `rate` in `AGETHRateReceiver` against the current rate on the source chain, and (2) call the public `payable` `deposit(string)` function during the staleness window. The attack is repeatable for as long as the rate remains stale.

## Recommendation

Add a staleness check in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours; // configurable by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

Alternatively, enforce the check inside `AGETHPoolV3.viewSwapAgETHAmountAndFee` by reading `lastUpdated` directly from the oracle interface and reverting if the rate is too old. The `MAX_RATE_AGE` constant should be set conservatively relative to the expected LayerZero update frequency.

## Proof of Concept

```solidity
// Foundry fork test (unmodified contracts)
function testStaleRateDilutesHolders() public {
    // 1. Deploy AGETHRateReceiver; simulate stale rate
    //    rate = 1.05e18, lastUpdated = block.timestamp - 2 days
    vm.store(address(receiver), RATE_SLOT, bytes32(uint256(1.05e18)));
    vm.store(address(receiver), LAST_UPDATED_SLOT, bytes32(block.timestamp - 2 days));

    // 2. Record existing holder's agETH balance
    uint256 holderAgETH = agETH.balanceOf(existingHolder);

    // 3. Attacker deposits 1 ETH
    vm.prank(attacker);
    pool.deposit{value: 1 ether}("");

    // 4. Assert attacker received more agETH than the true rate warrants
    uint256 attackerAgETH = agETH.balanceOf(attacker);
    uint256 correctAmount = 1e18 * 1e18 / 1.10e18; // true rate
    assertGt(attackerAgETH, correctAmount); // attacker got excess

    // 5. Assert existing holder's ETH backing per agETH has decreased
    uint256 totalAgETH = agETH.totalSupply();
    uint256 totalETH = address(pool).balance - pool.feeEarnedInETH();
    uint256 ethPerAgETH = totalETH * 1e18 / totalAgETH;
    assertLt(ethPerAgETH, 1.05e18); // dilution confirmed
}
```
