Audit Report

## Title
Unbounded Nested Loop in `NodeDelegator.getAssetUnstaking()` Blocks Public Deposit and Price-Update Paths - (File: contracts/NodeDelegator.sol)

## Summary
`NodeDelegator.getAssetUnstaking()` performs an unbounded nested loop over all EigenLayer-queued withdrawals and their strategies, making external storage calls on each iteration. This view function is invoked on every call to the public `depositETH()`, `depositAsset()`, and `updateRSETHPrice()` functions. As the protocol scales with more NDCs and concurrent unstaking operations — through normal protocol operation — the cumulative gas cost can exceed the block gas limit, permanently reverting all deposits and price updates.

## Finding Description
`getAssetUnstaking()` in `contracts/NodeDelegator.sol` (L405–427) fetches all queued withdrawals from EigenLayer's `DelegationManager` and iterates over them with a nested loop, making external calls (`strategy.underlyingToken()`, `strategy.sharesToUnderlyingView()`, `lrtConfig.beaconChainETHStrategy()`) on each inner iteration:

```solidity
// contracts/NodeDelegator.sol L405-427
function getAssetUnstaking(address asset) external view returns (uint256 amount) {
    (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
        _getDelegationManager().getQueuedWithdrawals(address(this));
    for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
        for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
            // external calls per iteration
        }
    }
}
```

The confirmed call chain to public entry points:

1. `depositETH()` / `depositAsset()` → `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()` → `getTotalAssetDeposits()` → `getAssetDistributionData()` → `getAssetUnstaking()` per NDC. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

2. `updateRSETHPrice()` (public, L87–89) → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getTotalAssetDeposits()` per supported asset → same chain. [5](#0-4) [6](#0-5) 

The effective gas cost scales as:

> `supportedAssets.length × nodeDelegatorQueue.length × queuedWithdrawals.length × strategies.length`

There is no cap on `queuedWithdrawals.length` or `strategies.length` inside `getAssetUnstaking()`. The `maxNodeDelegatorLimit` is admin-configurable upward via `updateMaxNodeDelegatorLimit()`. [7](#0-6) [8](#0-7) 

No existing guard bounds the total iteration count or caps the number of queued withdrawals returned by `getQueuedWithdrawals()`.

## Impact Explanation
When the product of the four dimensions grows large enough to exhaust the block gas limit, every call to `depositETH()`, `depositAsset()`, and `updateRSETHPrice()` reverts with out-of-gas. This constitutes **temporary (potentially permanent) freezing of the deposit path** and a stale rsETH price. The withdrawal path is also broken since `getExpectedAssetAmount()` depends on `rsETHPrice`. Existing deposited funds are not directly stolen, but the protocol becomes inoperable until queued withdrawals are completed to reduce state. This matches the allowed impact: **Medium — Unbounded gas consumption** and **Medium — Temporary freezing of funds**.

## Likelihood Explanation
No malicious actor is required. As the protocol scales organically — admin adds NDCs (normal operation), operators call `initiateUnstaking()` with multiple strategies (normal operation), and the number of supported assets grows — the combined loop depth increases without any adversarial intent. The trigger is any unprivileged user calling the public `depositETH()`, `depositAsset()`, or `updateRSETHPrice()` functions after the state has grown sufficiently. The preconditions are achievable through routine protocol operation, not through an attack.

## Recommendation
1. **Decouple unstaking accounting from the hot path**: Store `assetUnstaking` as a cached state variable updated lazily on `initiateUnstaking()` and `completeUnstaking()` rather than recomputing it inline on every deposit/price-update call.
2. **Cap strategies per withdrawal** in `initiateUnstaking()` to bound the inner loop.
3. **Cap queued withdrawals per NDC** by enforcing `maxUncompletedWithdrawalCount` before enqueuing new withdrawals.
4. If a view recomputation is required, process it off-chain and submit the result via a keeper/oracle pattern rather than computing it inline in state-changing transactions.

## Proof of Concept
1. Admin raises `maxNodeDelegatorLimit` to 10 and deploys 10 NDCs (normal operation).
2. Operators call `initiateUnstaking()` on each NDC with 5 strategies, 50 times each (50 pending withdrawals per NDC, normal operation).
3. Protocol supports 5 LST assets.
4. Any user calls `depositETH()`. The call chain reaches `getAssetUnstaking()` for each (asset × NDC) pair: `5 × 10 × 50 × 5 = 12,500` inner iterations, each involving multiple external storage reads from EigenLayer (`getQueuedWithdrawals`, `underlyingToken`, `sharesToUnderlyingView`). The transaction reverts with out-of-gas.
5. `updateRSETHPrice()` similarly reverts, freezing the rsETH price and breaking the withdrawal path.

A Foundry fork test can reproduce this by deploying 10 NDCs, queuing 50 withdrawals with 5 strategies each on a mainnet fork, then calling `depositETH{value: 1 ether}(0, "")` and asserting it reverts with out-of-gas.

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
