### Title
Missing Output Validation for `AppPublicKey` CKD Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

### Summary

In `respond_ckd`, the contract applies `ckd_output_check` (a BLS12-381 pairing equation check) only when the request uses the `AppPublicKeyPV` variant. When the request uses the `AppPublicKey` variant — the default/legacy mode — the contract performs **zero cryptographic validation** of the submitted `CKDResponse`. A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with arbitrary `big_y` and `big_c` values, permanently resolving all queued yields with forged confidential key material.

### Finding Description

The `respond_ckd` function in `crates/contract/src/lib.rs` handles two variants of `CKDAppPublicKey`:

```rust
// Lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VALIDATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC network's public key and the user's app public key: [2](#0-1) 

For `AppPublicKey`, the arm is an empty block — any `big_y` and `big_c` values pass through unconditionally. The contract then calls `resolve_yields_for`, which drains the entire fan-out queue and delivers the forged bytes to every waiting caller: [3](#0-2) 

The `resolve_yields_for` function removes the pending request entry and resumes all queued yields with the supplied response bytes, making the corruption permanent and irreversible: [4](#0-3) 

The contrast with `respond` (ECDSA/EdDSA) is stark: that function always verifies the signature against the derived public key before resolving any yield: [5](#0-4) 

The existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms the `AppPublicKey` path accepts completely arbitrary byte arrays `[1u8; 48]` / `[2u8; 48]` as `big_y`/`big_c` without complaint: [6](#0-5) 

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) can:

1. Monitor the NEAR chain for any pending `AppPublicKey` CKD request.
2. Call `respond_ckd` with an arbitrary `CKDResponse{big_y: <garbage>, big_c: <garbage>}`.
3. The contract accepts the call (all access-control checks pass — the attacker is a legitimate attested participant), performs no cryptographic validation, and permanently resolves every queued yield with the forged bytes.
4. Every user who submitted that CKD request receives a forged confidential key derivation output they cannot use to decrypt the actual confidential key.
5. The pending request is removed; the user cannot retry.

This is **unauthorized confidential key derivation output without the required participant authorization**: the threshold security guarantee (t-of-n cooperation) is completely bypassed for the `AppPublicKey` code path. A single participant suffices to forge the output.

**Allowed impact match:** Critical — "Unauthorized … confidential key derivation output without the required participant authorization."

### Likelihood Explanation

- `AppPublicKey` is the default/legacy CKD mode, used by all callers that pass a plain G1 point string (the original API format). It is the most common variant in production.
- Any single attested participant can exploit this. The attacker does not need to collude with others, compromise a TEE, or perform any off-chain attack.
- The attacker only needs to observe a pending CKD request on-chain and submit `respond_ckd` before the honest nodes do. Because the contract resolves on the first valid `respond_ckd` call, a racing Byzantine participant wins if it acts first.

### Recommendation

For `AppPublicKeyPV`, the contract can verify the output on-chain because the user supplies both `pk1` (G1) and `pk2` (G2), enabling the pairing check. For `AppPublicKey`, the user supplies only a G1 point, so the contract cannot reconstruct `pk2` and cannot run `ckd_output_check` directly.

Mitigations in order of preference:

1. **Deprecate `AppPublicKey` and require `AppPublicKeyPV` for all new CKD requests.** The `AppPublicKeyPV` variant was introduced precisely to enable on-chain verification; `AppPublicKey` should be phased out.
2. **If `AppPublicKey` must be retained**, require the submitting node to also provide a proof of correct encryption (e.g., a Schnorr proof of knowledge of the discrete log of `big_y` relative to the app public key), or require threshold-many independent `respond_ckd` submissions that agree on the same `(big_y, big_c)` before resolving.
3. At minimum, add a point-validity check (subgroup membership) on `big_y` and `big_c` for the `AppPublicKey` arm to prevent trivially malformed responses.

### Proof of Concept

An attested participant `byzantine.near` observes a pending `AppPublicKey` CKD request on-chain and calls:

```rust
// byzantine.near calls respond_ckd directly on the contract
contract.respond_ckd(
    CKDRequest {
        app_public_key: CKDAppPublicKey::AppPublicKey(victim_g1_pk),
        app_id: victim_app_id,
        domain_id: bls_domain_id,
    },
    CKDResponse {
        big_y: Bls12381G1PublicKey([0xde; 48]),  // arbitrary garbage
        big_c: Bls12381G1PublicKey([0xad; 48]),  // arbitrary garbage
    },
)
```

The contract passes all access-control checks (attested participant, running state, TEE accepted), skips the empty `AppPublicKey` arm, and calls `resolve_yields_for`, delivering the forged `(big_y, big_c)` to the victim. The victim's `request_app_private_key` promise resolves with garbage — the victim cannot decrypt the confidential key, and the request is permanently consumed. The existing unit test at line 3403 already demonstrates this path succeeds with `[1u8; 48]` / `[2u8; 48]` dummy values. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
