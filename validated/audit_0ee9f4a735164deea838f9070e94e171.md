Audit Report

## Title
`PriceVelocityGuardExtension` Uses `block.number` Instead of `block.timestamp` to Measure Price Velocity, Making the Guard Systematically Miscalibrated on L2s - (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`PriceVelocityGuardExtension.beforeSwap` computes the allowed oracle mid-price movement using `block.number` differences and a `maxChangePerBlockE18` parameter. Because L2 chains produce blocks far faster than Ethereum mainnet (Arbitrum ~0.25 s/block vs. ~12 s/block), the same parameter value produces a guard that is up to 48× weaker per real-time second on fast-block L2s. Any swap that exploits this wider-than-intended envelope causes permanent LP loss, as the swap settles before any admin can react.

## Finding Description
In `beforeSwap`, the guard computes:

```solidity
uint256 blockDiff = block.number - prevBlock;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
```

`prevBlock` is stored as `lastUpdateBlock` (a `uint64` block number) and updated with `block.number` on every swap and every admin `setLastMidPrice` call. The parameter `maxChangePerBlockE18` is named, documented, and emitted as a per-block rate. An admin calibrating this value will reason about block intervals — the NatSpec explicitly states "Allowed deviation scales as `maxChangePerBlockE18 * sqrt(1 + blockDifference)`."

On Arbitrum (~0.25 s/block), 48 blocks elapse in the same 12 real seconds as 1 Ethereum block. With `maxChangePerBlockE18 = 1e16` (1% per block):
- **Mainnet** (1 block in 12 s): `allowedSq = (1e16)² × 2 = 2e32` → allows ~1.41% change
- **Arbitrum** (48 blocks in 12 s): `allowedSq = (1e16)² × 49 = 4.9e33` → allows ~7% change

A 6% oracle price move in 12 seconds passes on Arbitrum (`3.6e33 < 4.9e33`) but correctly reverts on mainnet (`3.6e33 > 2e32`). The guard is the sole on-chain rate-of-change check in the swap path; no other mechanism catches this. `PriceProviderL2.sol` confirms L2 deployment is in scope and uses `block.timestamp` for its own staleness logic, making the inconsistency clear.

## Impact Explanation
The velocity guard is the only on-chain mechanism preventing bad-price execution in the swap path. When it is miscalibrated to be orders of magnitude too loose on L2s, swaps execute at oracle prices that have moved far beyond the intended envelope. LPs receive less token input (or pay more token output) than the true market price warrants. The loss is permanent — tokens are settled before any admin transaction can intervene. This directly matches the allowed impact: **bad-price execution** (unbounded bid/ask quote reaches a pool swap) and **direct loss of LP assets**.

## Likelihood Explanation
L2 deployment is confirmed in scope via `PriceProviderL2.sol`. Any pool admin who enables `PriceVelocityGuardExtension` on an L2 and calibrates `maxChangePerBlockE18` using mainnet block-time intuition — or copies a mainnet deployment value — silently deploys a guard that is orders of magnitude weaker than intended. No privileged attacker is required: any public swap call triggers `beforeSwap`, and the miscalibrated guard simply does not revert when it should. The miscalibration is silent and persistent until an admin transaction corrects it.

## Recommendation
Replace `block.number` / `lastUpdateBlock` / `maxChangePerBlockE18` with `block.timestamp` / `lastUpdateTimestamp` / `maxChangePerSecondE18` throughout `PriceVelocityGuardExtension` and `IPriceVelocityGuardExtension`. The velocity formula becomes:

```solidity
uint256 secondsDiff = block.timestamp - prevTimestamp;
uint256 allowedSq = uint256(maxChangePerSecond) * uint256(maxChangePerSecond) * (1 + secondsDiff);
```

`block.timestamp` manipulation by validators is bounded to a few seconds and is non-cumulative, making it far safer than block-number arithmetic for cross-chain rate limiting.

## Proof of Concept
```
// Arbitrum deployment (~0.25 s/block)
// Admin sets maxChangePerBlockE18 = 1e16 (1% per block, mainnet-calibrated)
//
// Block N: swap executes, lastMidPriceX64 = P, lastUpdateBlock = N
// 12 real seconds later: block N+48, oracle price moved 6%
//
// blockDiff = 48
// changeE18 = 6e16
// actualSq  = (6e16)^2 = 3.6e33
// allowedSq = (1e16)^2 * 49 = 4.9e33
// 3.6e33 < 4.9e33 → guard PASSES, swap executes at manipulated price
//
// Same scenario on mainnet (1 block in 12 s):
// allowedSq = (1e16)^2 * 2 = 2e32
// 3.6e33 > 2e32 → guard REVERTS as intended
//
// LPs on Arbitrum lose funds on every swap that exploits this 7× wider envelope.
```

A Foundry fork test on Arbitrum mainnet can reproduce this by deploying the extension, setting `maxChangePerBlockE18 = 1e16`, rolling forward 48 blocks (12 real seconds), pushing a 6% oracle price move, and asserting that `beforeSwap` does not revert.
