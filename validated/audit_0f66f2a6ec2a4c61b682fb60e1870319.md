### Title
Harvester Can Unconditionally Redirect Farmer Block Reward to Arbitrary Address — (`chia/farmer/farmer_api.py`)

### Summary
A connected harvester can set `farmer_reward_address_override` in a `RespondSignatures` message to any arbitrary puzzle hash. The farmer accepts this override without any authorization check and uses it as the `farmer_puzzle_hash` in `DeclareProofOfSpace`, causing the full node to permanently mint the farmer's block reward coin at the attacker-controlled address.

### Finding Description

In `_process_respond_signatures` in `chia/farmer/farmer_api.py`, the farmer unconditionally replaces its configured `farmer_target` with whatever `farmer_reward_address_override` the harvester supplies in `RespondSignatures`: [1](#0-0) 

This value is then placed directly into `DeclareProofOfSpace.farmer_puzzle_hash`: [2](#0-1) 

The full node receives `DeclareProofOfSpace` and uses `request.farmer_puzzle_hash` verbatim to construct the block foliage, with no validation against any expected farmer address: [3](#0-2) 

The farmer reward coin is then minted at that puzzle hash by consensus: [4](#0-3) 

The only guard that exists is `notify_farmer_reward_taken_by_harvester_as_fee`, which is triggered by `NewProofOfSpace.farmer_reward_address_override` (a separate, independent field) and **only logs a warning** — it does not reject the override or prevent block creation: [5](#0-4) 

Critically, the two fields are independent. A harvester can send `NewProofOfSpace` with `farmer_reward_address_override = None` (bypassing the logging entirely) and then send `RespondSignatures` with `farmer_reward_address_override = attacker_address` (actually diverting the reward). The farmer never cross-checks these two fields.

The `RespondSignatures` message type carrying the override: [6](#0-5) 

Block header validation does not validate `farmer_reward_puzzle_hash` against any expected value (only the genesis block is checked for a pre-farm hash): [7](#0-6) 

### Impact Explanation

Every time the farmer wins a block, the farmer reward coin (1/8 of the block reward) is permanently minted at the attacker-controlled address. The farmer receives nothing. This is an irreversible on-chain payout redirection of XCH. This matches the **High** impact category: "Bypass of wallet, daemon, keychain, RPC, pool, or Data Layer authorization that enables unauthorized signing, key use, coin control, **payout redirection**."

### Likelihood Explanation

Under CHIP-22, third-party harvesters are an explicitly supported use case. Any harvester that is connected to the farmer — including a malicious third-party harvester — can trigger this. The attacker only needs a valid proof of space and an active connection to the farmer. No key compromise, no admin access, and no cryptographic break is required.

### Recommendation

1. **Reject the override when fee quality threshold is not met**: In `_process_respond_signatures`, check `calculate_harvester_fee_quality` against the harvester's declared `fee_info.applied_fee_threshold` before accepting `farmer_reward_address_override`. If the threshold is not met, drop the block candidate rather than logging a warning.
2. **Cross-validate the two override fields**: Ensure `RespondSignatures.farmer_reward_address_override` matches `NewProofOfSpace.farmer_reward_address_override` for the same proof. A mismatch should cause the response to be rejected.
3. **Allowlist override addresses**: Allow farmers to configure a set of permitted override addresses (e.g., their own addresses or known harvester fee addresses). Reject any override that is not in the allowlist.

### Proof of Concept

1. Attacker operates a harvester connected to a victim farmer.
2. Harvester finds a valid proof of space and sends `NewProofOfSpace` with `farmer_reward_address_override = None` (no logging triggered).
3. Farmer sends `RequestSignatures` to the harvester.
4. Harvester responds with `RespondSignatures` where `farmer_reward_address_override = attacker_puzzle_hash`.
5. Farmer's `_process_respond_signatures` sets `farmer_reward_address = attacker_puzzle_hash` (line 918) and emits `DeclareProofOfSpace` with `farmer_puzzle_hash = attacker_puzzle_hash`.
6. Full node builds the block foliage with `farmer_ph = request.farmer_puzzle_hash` (line 1069) and the farmer reward coin is minted at `attacker_puzzle_hash`.
7. Farmer's block reward is permanently diverted; the farmer receives 0 XCH for the block they won.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
