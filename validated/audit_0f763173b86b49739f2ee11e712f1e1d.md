Audit Report

## Title
Stale `CrossChainRateReceiver` Rate With No Freshness Validation Enables Over-Minting of wrsETH — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/pools/RSETHPoolV3.sol`)

## Summary

`CrossChainRateReceiver.getRate()` unconditionally returns the stored `rate` field without ever consulting `lastUpdated`, so any LayerZero bridge delay leaves the denominator in `RSETHPoolV3`'s swap formula at a stale low value. Because rsETH is a monotonically appreciating yield-bearing token, a stale rate causes every deposit to mint more wrsETH than the deposited collateral is worth in rsETH terms, breaking the full-backing invariant and rendering the pool insolvent.

## Finding Description

`CrossChainRateReceiver.lzReceive()` writes `lastUpdated = block.timestamp` on every bridge message, but `getRate()` never reads it: [1](#0-0) [2](#0-1) 

There is no heartbeat threshold, no revert on staleness, and no circuit-breaker. `RSETHPoolV3.getRate()` delegates directly to this oracle: [3](#0-2) 

`viewSwapRsETHAmountAndFee(amount, token)` uses the stale rate as the **denominator**: [4](#0-3) 

The collateral oracle (`ChainlinkOracleForRSETHPoolCollateral`) checks `answeredInRound < roundID`, zero timestamp, and non-positive price, but has **no heartbeat/timestamp freshness check**, so it continues returning a live market price while the cross-chain rate is stale: [5](#0-4) 

When the LayerZero bridge is delayed, `rsETHToETHrate` stays at an old low value while `tokenToETHRate` (e.g., wstETH/ETH) reflects the current market. The formula becomes:

```
rsETHAmount = amountAfterFee * currentHighTokenRate / staleLowRsETHRate
            >> amountAfterFee * currentHighTokenRate / currentRsETHRate
```

`deposit` mints this inflated amount directly: [6](#0-5) 

The `dailyMintLimit` guard runs through the same `viewSwapRsETHAmountAndFee` call and therefore uses the same stale rate — it caps total daily volume but does not prevent the per-unit over-minting: [7](#0-6) 

## Impact Explanation

Every deposit made while the receiver rate is stale mints more wrsETH than the deposited collateral is worth in rsETH terms. Because wrsETH is redeemable 1:1 for rsETH, the pool becomes undercollateralized: the aggregate wrsETH supply exceeds the rsETH backing. This constitutes **Critical — Protocol Insolvency**.

## Likelihood Explanation

rsETH is a yield-bearing token whose ETH rate increases monotonically. Any LayerZero bridge delay — network congestion, relayer downtime, gas spike on the source chain — leaves the receiver rate stale at a value below the true current rate. This is a normal operational condition, not an adversarial prerequisite. An attacker monitoring the public `lastUpdated` field on-chain can time deposits to exploit the widest divergence window. The exploit is executable by any unprivileged external caller via the public `deposit(address token, uint256 amount, string memory referralId)` function with no special roles required.

## Recommendation

Add a configurable staleness threshold and revert in `CrossChainRateReceiver.getRate()` if the rate is too old:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours; // configurable

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

Additionally, add a heartbeat check in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
uint256 public constant MAX_ORACLE_AGE = 24 hours;

if (block.timestamp - timestamp > MAX_ORACLE_AGE) revert StalePrice();
```

## Proof of Concept

Deploy a fork test against a chain where `RSETHPoolV3` is live with a `CrossChainRateReceiver` as `rsETHOracle`:

1. Warp time forward past the bridge update interval so `block.timestamp - receiver.lastUpdated() > threshold`.
2. Call `pool.viewSwapRsETHAmountAndFee(1e18, wstETH)` — the returned `rsETHAmount` uses the stale low denominator.
3. Compute the correct amount using the true current rsETH rate (e.g., fetched from the mainnet oracle).
4. Assert `rsETHAmount > correctAmount` — this will pass, confirming over-minting.
5. Execute `pool.deposit(wstETH, 1e18, "poc")` and verify the minted wrsETH balance exceeds the collateral's true rsETH value.
6. The invariant `minted wrsETH ≤ depositValue / currentRsETHRate` will be broken, confirming protocol insolvency.
