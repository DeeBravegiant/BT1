### Title
Delegate-Authorized `init_transfer` Bridges Victim's Tokens With Attacker Credited as Sender — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

The `InitTransfer` account struct imposes no ownership constraint on the `from` token account. Because SPL Token's `transfer_checked` accepts an approved delegate as a valid authority, an attacker who holds a delegate approval on a victim's token account can call `init_transfer`, lock the victim's tokens in the bridge vault, and have the Wormhole message credit the attacker — not the victim — as the sender on NEAR.

---

### Finding Description

The `from` account in `InitTransfer` is constrained only by mint and token-program equality:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

There is no `token::authority = user` constraint, so Anchor never verifies that `from.owner == user`.

In `process()`, the native-token path calls `transfer_checked` with `authority: self.user.to_account_info()`:

```rust
transfer_checked(
    CpiContext::new(
        self.token_program.to_account_info(),
        TransferChecked {
            from: self.from.to_account_info(),
            to: vault.to_account_info(),
            authority: self.user.to_account_info(),   // signer, not necessarily owner
            mint: self.mint.to_account_info(),
        },
    ),
    ...
)?;
``` [2](#0-1) 

SPL Token's `transfer_checked` accepts either the account owner **or** an approved delegate as a valid authority. If the attacker has been approved as a delegate on the victim's token account (even for an unrelated purpose), the CPI succeeds.

After the transfer, the Wormhole message is posted with `self.user.key()` — the attacker's key — as the `sender` field:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),   // attacker's pubkey becomes the credited sender
    self.mint.key(),
))?)?;
``` [3](#0-2) 

The serialized message encodes `params.1` (the attacker's key) as the `sender` field in the cross-chain payload: [4](#0-3) 

The same flaw exists in the bridged-token (burn) path: `burn` also accepts a delegate as authority, so the attacker can burn the victim's wrapped tokens and claim the equivalent on NEAR. [5](#0-4) 

---

### Impact Explanation

- The victim's tokens are irreversibly locked in the vault (or burned).
- The attacker is recorded as the sender in the Wormhole VAA and receives the bridged value on NEAR.
- The victim has no recourse: the bridge has no refund path for a completed `init_transfer`.

This satisfies **Critical — Unauthorized custody escape / unauthorized transfer of another user's tokens**.

---

### Likelihood Explanation

The prerequisite is that the attacker holds a delegate approval on the victim's token account. This is a realistic scenario in the Solana DeFi ecosystem: users routinely approve delegates for DEXes, lending protocols, and other programs. An attacker who controls such a program, or who has obtained a delegate approval through any means, can immediately exploit this path. No privileged role, leaked key, or guardian compromise is required.

---

### Recommendation

Add `token::authority = user` to the `from` account constraint in `InitTransfer`:

```rust
#[account(
    mut,
    token::mint = mint,
    token::authority = user,      // <-- enforce ownership
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This causes Anchor to verify `from.owner == user` at deserialization time, before any CPI is attempted, rejecting delegate-only callers.

---

### Proof of Concept

1. Victim holds token account `V` with 1 000 tokens; attacker `A` has been approved as a delegate on `V` for any amount ≥ 1 000.
2. Attacker calls `init_transfer` with:
   - `from = V` (victim's account)
   - `user = A` (attacker, signs the transaction)
   - `vault` = the bridge vault PDA for the mint
3. Anchor constraint check on `from`: only `token::mint` and `token::token_program` are verified — **passes**.
4. `transfer_checked` CPI: `from = V`, `authority = A` — SPL Token accepts `A` as a valid delegate — **passes**, 1 000 tokens move from `V` to the vault.
5. `post_message` emits a Wormhole VAA with sender = `A`'s pubkey.
6. On NEAR, the VAA is finalized; `A` receives 1 000 bridged tokens. Victim's tokens are gone.
