### Title
Single Attested Participant Can Forge CKD Response for `AppPublicKey` Variant, Bypassing Threshold Requirement - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function in the MPC contract performs cryptographic output verification only for the `AppPublicKeyPV` variant of CKD requests. For the `AppPublicKey` (plain G1 point) variant — the default and most commonly used form — the match arm is an empty no-op, meaning any single attested participant can submit an arbitrary forged `CKDResponse` and the contract will accept it unconditionally. This bypasses the threshold-MPC requirement: instead of requiring t+1 nodes to cooperate to produce the correct confidential key derivation output, a single Byzantine node can race to deliver a forged result.

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function handles both CKD variants:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the BLS pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically proves the response is the correct threshold-computed output. [2](#0-1) 

For `AppPublicKey`, the arm is `{}` — no pairing check, no signature verification, no proof of correctness. The response is immediately serialized and delivered to the user via `resolve_yields_for`, which also removes the request from the pending map so no subsequent honest response can overwrite it. [3](#0-2) 

The `AppPublicKey` variant is the plain single-G1-point form and is the default/legacy variant used by the CLI and documented examples. [4](#0-3) 

The `AppPublicKeyPV` variant is explicitly described as the one that "allows the contract to verify the CKD response on-chain." [5](#0-4) 

**Attack path:**
1. A user submits `request_app_private_key` with `AppPublicKey` variant (a plain BLS G1 point `A`).
2. A single malicious attested participant constructs an arbitrary `CKDResponse { big_c, big_y }` — for example, one where `big_c = r·G1` and `big_y = r·G1` for a random scalar `r` they control.
3. The attacker calls `respond_ckd` with this forged response before honest nodes can respond.
4. The contract passes the empty `AppPublicKey` match arm, calls `resolve_yields_for`, drains the pending queue, and delivers the forged `(big_c, big_y)` to the user.
5. The user decrypts the "confidential key" using their private scalar `a`: `key = big_c - a·big_y`. Since the attacker chose `big_c` and `big_y`, they can compute `key` themselves.
6. The attacker now knows the user's derived private key.

The `resolve_yields_for` call removes the request from `pending_ckd_requests`, so any subsequent honest `respond_ckd` call returns `RequestNotFound`. [6](#0-5) 

### Impact Explanation

**Critical.** A single Byzantine MPC participant (below the signing threshold) can unilaterally forge the output of any `AppPublicKey` CKD request. The user receives a `(big_c, big_y)` pair that the attacker fully controls and knows. When the user unmasks it with their private scalar `a` via `CKDOutput::unmask(a)`, they obtain a value the attacker can independently compute. This gives the attacker the user's derived private key — the exact secret the CKD protocol is designed to protect. Any funds, credentials, or secrets controlled by that derived key are compromised. This is a direct bypass of the threshold-signature/key-derivation requirement: the protocol is supposed to require t+1 nodes to cooperate, but a single node can forge the result for the `AppPublicKey` variant.

### Likelihood Explanation

**Medium-High.** The `AppPublicKey` variant is the default form used in the CLI and documentation. Any user who does not explicitly opt into `AppPublicKeyPV` is vulnerable. The attacker only needs to be one attested participant (a Byzantine node below threshold) and must win a race against honest nodes to call `respond_ckd` first — a realistic condition since the attacker can monitor the chain for new CKD requests and immediately submit a forged response.

### Recommendation

Apply `ckd_output_check` to both variants. For `AppPublicKey`, the contract already has the BLS12-381 domain public key available. The check `e(big_c, g2) = e(big_y, app_pk1) · e(hash_point, public_key)` (using only `pk1`, the G1 component) can be performed without the G2 component of the app key. Alternatively, deprecate the `AppPublicKey` variant and require all callers to use `AppPublicKeyPV`, which provides full on-chain verifiability. At minimum, document clearly that `AppPublicKey` provides no on-chain integrity guarantee and that a single malicious node can forge the result.

### Proof of Concept

```
1. User calls request_app_private_key({
       derivation_path: "mykey",
       app_public_key: "bls12381g1:<A>",   // AppPublicKey variant
       domain_id: 2
   })
   → contract stores CKDRequest in pending_ckd_requests, yields execution

2. Attacker (one attested participant) picks random r, computes:
       big_c = r * G1
       big_y = r * G1
   Calls respond_ckd(request, { big_c, big_y })

3. Contract executes respond_ckd:
   - assert_caller_is_attested_participant_and_protocol_active() → passes
   - match AppPublicKey(_) => {}  ← no check
   - resolve_yields_for → delivers { big_c, big_y } to user, removes request

4. User computes: key = big_c - a * big_y = r*G1 - a*r*G1 = r*(1-a)*G1
   Attacker computes: key = r*(1-a)*G1  (knows r, knows a from the request's app_public_key field)
   → Attacker recovers user's derived private key
```

The `app_public_key` field `A` is stored in the `CKDRequest` on-chain and is visible to all participants, so the attacker knows `a` (the user's private scalar) only if the user's key is weak — but more precisely, the attacker chooses `big_c` and `big_y` such that the resulting `key = big_c - a·big_y` is a value they can compute independently of `a`. For example, setting `big_y = G1` (the generator) and `big_c = k·G1 + a·G1` for a chosen `k` yields `key = k·G1`, which the attacker knows. Since `a` appears in the public request, the attacker can compute `a·G1` from the stored `app_public_key` and construct `big_c` accordingly. [7](#0-6) [8](#0-7) [9](#0-8)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
