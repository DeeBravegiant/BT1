Audit Report

## Title
Stale `rsETHPrice` Denominator vs. Live `getExchangeRate()` Numerator Allows Depositors to Capture Accrued rETH Yield From Existing Holders — (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`, `contracts/oracles/RETHPriceOracle.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint()` computes the mint amount by dividing a live rETH exchange rate by a stale cached `rsETHPrice`. Because rETH accrues staking rewards continuously while `rsETHPrice` is only updated on explicit calls to `updateRSETHPrice()`, any deposit made in the window between a rETH reward tick and the next price update mints more rsETH than the depositor's ETH contribution warrants, diluting the yield already earned by existing holders.

## Finding Description

The mint formula at `LRTDepositPool.sol:520` is:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

**Numerator — always live.** `lrtOracle.getAssetPrice(rETH)` delegates to `RETHPriceOracle.getAssetPrice()`, which calls `IrETH(rETHAddress).getExchangeRate()` directly on the rETH token with no caching. [2](#0-1) 

**Denominator — stale until manually refreshed.** `lrtOracle.rsETHPrice()` is a storage variable written only inside `_updateRsETHPrice()`, which is triggered by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. [3](#0-2) [4](#0-3) [5](#0-4) 

`_beforeDeposit` contains no guard that detects or prevents over-minting. The `minRSETHAmountExpected` slippage parameter only enforces a floor on what the depositor receives; it cannot prevent the depositor from receiving *more* than fair value. [6](#0-5) 

The exploit path requires no special role or oracle manipulation. A normal `depositAsset()` call during the window between a rETH reward accrual and the next `updateRSETHPrice()` call is sufficient.

## Impact Explanation

**High — Theft of unclaimed yield.**

Concrete example: after rETH rate ticks from 1.050 to 1.060 but before `updateRSETHPrice()` is called, an attacker depositing 100 rETH receives `100 × 1.060 / 1.050 = 100.952 rsETH` instead of the fair 100.000. After the price update, original holders' 100 rsETH is worth 105.497 ETH instead of the 106 ETH they earned — a 0.503 ETH transfer of yield to the attacker. Profit scales linearly with deposit size and with the rate delta since the last `updateRSETHPrice()` call. The protocol remains solvent (all rsETH is backed by real ETH), so the correct classification is High — Theft of unclaimed yield, not Critical insolvency.

## Likelihood Explanation

- rETH exchange rate increases continuously; the attack window is always open between `updateRSETHPrice()` calls.
- `updateRSETHPrice()` is public but not called atomically with deposits; typical keeper cadence is hours to days.
- No special role, front-running, or oracle manipulation is required — a normal `depositAsset()` call suffices.
- Profit is proportional to deposit size, so a whale can extract meaningful yield in a single transaction.
- The condition (`rETH rate ticked since last price update`) is the normal operating state of the protocol.

## Recommendation

Enforce that `rsETHPrice` is refreshed atomically before computing the mint amount. The simplest fix is to call `_updateRsETHPrice()` (or compute a fresh price inline) inside `getRsETHAmountToMint()` rather than reading the cached `rsETHPrice` storage variable. Alternatively, compute the mint ratio directly from `totalETHInProtocol / rsethSupply` on-the-fly at deposit time, bypassing the stale cache entirely.

## Proof of Concept

```solidity
// Foundry fork test
// 1. Fork mainnet at block B; call lrtOracle.updateRSETHPrice() → rsETHPrice = RATE_OLD
// 2. vm.mockCall(rETHAddress, getExchangeRate.selector, abi.encode(RATE_NEW))
//    where RATE_NEW > RATE_OLD (simulates a Rocket Pool reward interval)
// 3. attacker.depositAsset(rETH, DEPOSIT_AMOUNT, 0, "")
// 4. uint256 minted = rsETH.balanceOf(attacker)
// 5. assertGt(minted, DEPOSIT_AMOUNT);  // over-mint confirmed
// 6. lrtOracle.updateRSETHPrice();
// 7. uint256 attackerValue = minted * lrtOracle.rsETHPrice() / 1e18;
// 8. assertGt(attackerValue, DEPOSIT_AMOUNT * RATE_NEW / 1e18); // profit from existing holders confirmed
```

The assertion at step 5 passes for any `RATE_NEW > RATE_OLD` and any non-zero `DEPOSIT_AMOUNT`, confirming the vulnerability is unconditionally present whenever the rETH rate has ticked since the last `updateRSETHPrice()` call.
