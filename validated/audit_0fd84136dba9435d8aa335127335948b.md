### Title
Old Participants Excluded from `respond` During Resharing Due to Participant-Set Mismatch — (`crates/contract/src/lib.rs`, `crates/contract/src/state.rs`)

---

### Summary

During resharing, the `respond` / `respond_ckd` / `respond_verify_foreign_tx` entry points gate submission on `active_participants()`, which returns the **new (proposed)** participant set. However, threshold signing during resharing is performed by the **old** participants using the old key shares. Old participants that are being dropped from the new set cannot call `respond`, even though they are the only nodes capable of computing valid signatures at that moment. In a complete or near-complete participant rotation this causes every in-flight sign request to time out, permanently failing user requests without refund of the deposit.

---

### Finding Description

`respond` (and its siblings) enforces two guards in sequence:

```rust
// crates/contract/src/lib.rs:569-573
let signer = Self::assert_caller_is_signer();
self.assert_caller_is_attested_participant_and_protocol_active();
if !self.protocol_state.is_running_or_resharing() { … }
```

`assert_caller_is_attested_participant_and_protocol_active` resolves the participant list via `active_participants()`:

```rust
// crates/contract/src/state.rs:255-269
pub fn active_participants(&self) -> &Participants {
    match self {
        ProtocolContractState::Resharing(state) => {
            state.resharing_key.proposed_parameters().participants()   // ← NEW set
        }
        ProtocolContractState::Running(state) => state.parameters.participants(),
        …
    }
}
```

During resharing the function returns the **proposed (new)** participants. The same codebase uses the **old** participants for governance votes during resharing:

```rust
// crates/contract/src/state.rs:232-235
ProtocolContractState::Resharing(state) => {
    AuthenticatedParticipantId::new(
        state.previous_running_state.parameters.participants(),   // ← OLD set
    )?;
}
```

This is the structural inconsistency. Governance authority is correctly scoped to old participants; signing authority is incorrectly scoped to new participants.

Off-chain, the MPC nodes sign pending requests using the **old** key shares held by the **old** participants. Any old participant that is not included in the new set:

1. Can compute a valid threshold signature (holds old key shares).
2. **Cannot** call `respond` — `is_caller_an_attested_participant(new_participants)` returns `AttestationCheckError::CallerNotParticipant`.

Conversely, new participants that are not in the old set:

1. Can call `respond` (pass the participant check).
2. **Cannot** compute a valid signature (do not hold old key shares; resharing is still in progress).

When the old and new sets are disjoint — a complete participant rotation — no node can both compute and submit a signature. Every in-flight sign request times out via `return_signature_and_clean_state_on_success` → `fail_on_timeout`, permanently failing the user's transaction.

---

### Impact Explanation

Every pending `sign`, `request_app_private_key`, or `verify_foreign_transaction` request that was enqueued before resharing began will time out if the only nodes capable of computing the signature are being dropped. The yield-resume timeout fires `fail_on_timeout`, which panics the callback receipt and returns a `Timeout` error to the original caller. The user's 1-yoctoNEAR deposit is not refunded (the refund path only exists for excess deposit, not for the base deposit). This is a request-lifecycle invariant break: the contract accepted the request and the deposit, but cannot fulfill or refund it.

---

### Likelihood Explanation

Resharing with participant rotation is a routine governance operation (node upgrades, operator changes, security rotations). Any resharing that drops at least one participant triggers the partial form of this issue. A complete rotation — while constrained by the resharing cryptographic requirement that enough old participants participate — is architecturally permitted by the contract and is a realistic operational scenario (e.g., migrating to an entirely new operator set). The issue requires no adversarial action; it is triggered by normal governance.

---

### Recommendation

`respond` (and its siblings) should authenticate the caller against **both** the old and new participant sets during resharing, mirroring the split already present in `is_existing_or_prospective_participant`:

```rust
// crates/contract/src/state/resharing.rs:198-206
pub fn is_participant_or_prospective_participant(&self, account_id: &AccountId) -> bool {
    self.previous_running_state.is_participant_given_account_id(account_id)
        || self.resharing_key.proposed_parameters().participants()
               .is_participant_given_account_id(account_id)
}
```

A dedicated `active_signing_participants()` helper that returns the **old** set during resharing (since signing uses the old key) should be used in `assert_caller_is_attested_participant_and_protocol_active` when the call is from a `respond*` method, or the check should be relaxed to accept either old or new participants during resharing.

---

### Proof of Concept

1. Contract is `Running` with participants `[A, B, C]`, threshold 2.
2. User calls `sign(…)` — a `SignatureRequest` is enqueued with a yield data-id.
3. Participants vote `vote_new_parameters` to reshare to `[D, E, F]` (complete rotation). Contract enters `Resharing`.
4. Off-chain: nodes A, B, C compute the threshold signature using the old key shares.
5. Node A calls `respond(request, signature)`.
6. `assert_caller_is_attested_participant_and_protocol_active()` calls `active_participants()` → returns `[D, E, F]`.
7. `is_caller_an_attested_participant([D,E,F])` → `CallerNotParticipant` for A → **panic**.
8. Nodes D, E, F do not hold old key shares; they cannot produce a valid signature to submit.
9. After ~200 blocks the yield times out; `return_signature_and_clean_state_on_success` fires with `Err(PromiseError::Failed)`, calls `fail_on_timeout`, and the user's transaction fails permanently.

                    vec![],
                    NearToken::from_near(0),
                    fail_on_timeout_gas,
                );
                near_sdk::PromiseOrValue::Promise(promise.as_return())
            }
        }
    }
