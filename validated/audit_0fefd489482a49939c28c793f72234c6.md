### Title
Single Byzantine Participant Can Forge CKD Response for `AppPublicKey` Requests — (`crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, when a CKD request uses the `CKDAppPublicKey::AppPublicKey` variant, the contract performs **no cryptographic output verification**. For the `AppPublicKeyPV` variant, a pairing-based check enforces that the response is mathematically consistent with the MPC network's public key. For `AppPublicKey`, the check is silently skipped and any response from a single attested participant is accepted and delivered to all waiting users. This is the direct analog of the original report: when verification "cannot be done" (no verification key pair is available), the system falls back to a permissive default (accept anything) instead of enforcing a stricter control.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function contains:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` (defined in `crates/contract/src/primitives/ckd.rs`) enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC network's master key. For `AppPublicKey`, the arm is an empty block — the contract accepts whatever `(big_y, big_c)` the responding participant supplies.

After the match, `pending_requests::resolve_yields_for` is called unconditionally, which drains **all** queued yield slots for that request key and delivers the unverified response to every waiting caller.

The `AppPublicKey` variant is the standard production variant (used in the e2e test `ckd_response__passes_cryptographic_verification` and in the SDK). The `AppPublicKeyPV` variant was added later specifically to enable on-chain verification, but the older path was left unguarded.

**Structural analog to the original report:**

| Original (Carapace) | NEAR MPC |
|---|---|
| `_totalCapital < _minRequiredCapital` → risk factor cannot be calculated | `AppPublicKey` variant → output cannot be verified on-chain |
| Falls back to minimum premium (permissive) | Falls back to empty match arm (permissive) |
| Protection should be expensive/blocked | CKD output should require threshold agreement |
| Single user exploits cheap protection | Single Byzantine participant forges CKD output |

### Impact Explanation

A single malicious attested participant (Byzantine participant strictly below the signing threshold) can:

1. Observe a pending `AppPublicKey`-type CKD request in the contract.
2. Call `respond_ckd` with a crafted `CKDResponse` — arbitrary `big_y = g1 * y'` and `big_c = g1 * c'` — where the attacker knows `y'` and `c'`.
3. The contract accepts the response without any verification and resolves all queued yields, delivering the forged output to every user who submitted the same request.
4. The user derives a key from the forged `(big_y, big_c)`. Because the attacker chose these values, the attacker can compute the corresponding derived private key and steal any assets the user subsequently controls with it.

This is **unauthorized confidential key derivation output without the required participant authorization**: the threshold guarantee (requiring agreement among ≥ threshold participants) is entirely bypassed for `AppPublicKey` requests at the contract level.

---

### Likelihood Explanation

- The `AppPublicKey` variant is the default, production-facing CKD path.
- Only **one** attested participant needs to be malicious — well below the signing threshold.
- The attack is deterministic and requires no special timing or network conditions: the attacker simply races to call `respond_ckd` before the honest coordinator does.
- The contract's `respond_ckd` is a public endpoint callable by any attested participant.

---

### Recommendation

**Option A (preferred):** Require `AppPublicKeyPV` for all new CKD requests and deprecate `AppPublicKey`. The `AppPublicKeyPV` path already has a sound on-chain pairing check.

**Option B:** For `AppPublicKey` requests, require threshold-many participants to submit identical responses before resolving the yield (a commit-reveal or vote-counting scheme), so a single Byzantine participant cannot unilaterally determine the output.

**Option C (minimal):** At minimum, document that `AppPublicKey` requests carry no on-chain output integrity guarantee and advise callers to migrate to `AppPublicKeyPV`.

### Proof of Concept

1. Alice calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)` and a derivation path. The request is queued in `pending_ckd_requests`.

2. Mallory, a single attested participant, observes the pending request on-chain.

3. Mallory calls `respond_ckd(ckd_request, CKDResponse { big_y: g1*y_fake, big_c: g1*c_fake })` where she chose `y_fake` and `c_fake` herself.

4. The contract executes:
   ```rust
   match &request.app_public_key {
       dtos::CKDAppPublicKey::AppPublicKey(_) => {}  // no check
       ...
   }
   pending_requests::resolve_yields_for(..., serde_json::to_vec(&response).unwrap())
   ```
   The forged response is accepted and all of Alice's queued yields are resolved with Mallory's values.

5. Alice receives `(big_y_fake, big_c_fake)` and derives a key she believes is backed by the MPC network. Mallory, knowing `y_fake` and `c_fake`, can compute the corresponding private key and drain any funds Alice deposits to the derived address.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
