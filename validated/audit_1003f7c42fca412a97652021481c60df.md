### Title
Fee-on-Transfer Token Accounting Divergence Creates Unbacked Wrapped Supply - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

In `OmniBridge.sol`'s `initTransfer`, the contract locks ERC-20 tokens via `safeTransferFrom(msg.sender, address(this), amount)` and then emits `InitTransfer` with the caller-supplied `amount`. For fee-on-transfer tokens, the contract receives fewer tokens than `amount`, but the NEAR settlement layer reads the emitted `amount` and mints/releases that full value on the destination chain — creating unbacked wrapped supply.

### Finding Description

In `initTransfer`, the native-token (non-bridge, non-custom-minter) path is:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-controlled
);
```

followed immediately by:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,         // same caller-controlled value, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) [2](#0-1) 

No balance snapshot is taken before or after the `safeTransferFrom`. The emitted `amount` is the input parameter, not `balanceAfter - balanceBefore`.

The NEAR bridge processes this event and credits the recipient with `amount` tokens on the destination chain. For a fee-on-transfer token, the vault holds only `amount - fee`, but the destination chain has minted `amount` wrapped tokens — the difference is unbacked.

The same pattern exists in the Starknet bridge:

```cairo
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
// ...
Event::InitTransfer(InitTransfer { ..., amount, ... })
``` [3](#0-2) [4](#0-3) 

### Impact Explanation

Each bridge of a fee-on-transfer token inflates the wrapped supply relative to the locked collateral. When users on the destination chain bridge back, the EVM vault is short by the cumulative fee amount. The last redeemers cannot withdraw — their wrapped tokens are permanently unclaimable. This matches **Critical: unauthorized creation of unbacked wrapped bridge assets** and **Critical: irreversible fund lock / permanently unclaimable user value**. [5](#0-4) 

### Likelihood Explanation

Likelihood is **Low-to-Medium**. Fee-on-transfer tokens (e.g., PAXG, STA, early USDT on some chains) are real and deployed on mainnet. The attack requires only that such a token be registered with the bridge (either by the protocol or by a permissionless registration path). No privileged access is needed — `initTransfer` is a public entry point callable by any EOA. [6](#0-5) 

### Recommendation

Replace the fixed-`amount` accounting with a balance-delta pattern for the native-token lock path:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived in the emitted event and downstream accounting
```

Apply the same fix to the Starknet `init_transfer`. Additionally, document which token types are unsupported (rebasing, max-transfer-sentinel tokens such as cUSDCv3) and add an allowlist or explicit revert for known non-standard tokens. [7](#0-6) 

### Proof of Concept

1. Deploy or use an existing fee-on-transfer ERC-20 token `T` with a 1% transfer fee, registered with the EVM OmniBridge.
2. Call `OmniBridge.initTransfer(T, 1_000_000, 0, 0, nearRecipient, "")`.
3. `safeTransferFrom` moves 1,000,000 from the caller; the contract receives 990,000 (1% fee taken).
4. `InitTransfer` event is emitted with `amount = 1_000_000`.
5. NEAR prover validates the event; NEAR bridge mints 1,000,000 wrapped-T to `nearRecipient`.
6. `nearRecipient` bridges back 1,000,000 wrapped-T to EVM.
7. EVM vault holds only 990,000 T; the `finTransfer` `safeTransfer` of 1,000,000 reverts or drains other depositors' funds.
8. Repeat to widen the shortfall; eventually the vault is drained and legitimate redeemers are permanently locked out. [8](#0-7)
