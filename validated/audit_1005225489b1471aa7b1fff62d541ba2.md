### Title
Excess NEAR Deposit Silently Absorbed on Attestation Re-submission — (`crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is a `#[payable]` function that conditionally handles deposit refunds. When an existing participant re-submits their attestation (the `UpdatedExistingParticipant` branch), the entire deposit-handling block is skipped. Any NEAR attached to that call is permanently absorbed by the contract with no refund, violating the documented invariant that "any excess deposit is automatically refunded."

---

### Finding Description

`submit_participant_info` computes a boolean gate before deciding whether to check or refund the attached deposit:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check deposit sufficiency, refund excess ...
}
``` [1](#0-0) 

The gate is `false` — and the block is entirely skipped — when **both** conditions hold simultaneously:

- `is_new_attestation == false`: the TLS key already exists in `stored_attestations`, so `add_participant` returns `UpdatedExistingParticipant`.
- `caller_is_not_participant == false`: the caller is a current voting participant. [2](#0-1) 

In this branch, `env::attached_deposit()` is never read, never checked, and never refunded. Because the function is `#[payable]`, the NEAR runtime accepts any deposit amount and credits it to the contract account. The caller's NEAR is permanently lost.

The analog to M-09 is exact: in M-09 the swap callback can send more tokens than required and the pool silently keeps the surplus; here, a re-submitting participant can attach more NEAR than required (zero, in this case) and the contract silently keeps the surplus.

---

### Impact Explanation

**Medium.** This breaks the production accounting invariant explicitly documented in the contract README:

> "Both `sign` and `request_app_private_key` require a deposit of at least 1 yoctonear. Any excess deposit is automatically refunded."

The same refund guarantee is implemented for every other payable entry point (`sign`, `request_app_private_key`, `verify_foreign_transaction`, `propose_update`, `submit_participant_info` first-time path) via `require_deposit` or an explicit `checked_sub` + `Promise::transfer`. The re-submission path is the sole exception. [3](#0-2) 

The lost funds are NEAR tokens belonging to the participant or operator, not MPC-controlled signing key material. The impact is therefore **Medium** (balance/accounting invariant broken) rather than Critical.

---

### Likelihood Explanation

The automated `periodic_attestation_submission` task in `mpc-node` attaches 0 NEAR for re-submissions, so the bug is not triggered in the normal automated path. However, the operator guide explicitly documents a manual fallback:

> "If MPC Node fails to submit remote attestation: Operator → Contract: Submits the remote attestation on behalf of the node."

An operator following this fallback for a re-submission (e.g., after a TEE upgrade) is likely to attach NEAR — either because they previously had to for the first submission, or because the function is `#[payable]` with no inline documentation warning that deposits are not refunded on re-submission. The 1-hour periodic re-submission cadence also means the surface is exercised frequently.

---

### Recommendation

Unconditionally refund any excess deposit at the end of `submit_participant_info`, regardless of whether the call is a first-time insertion or an update. The simplest fix mirrors the pattern already used in `require_deposit`:

```rust
// After the conditional storage-cost block, always refund any surplus.
let attached = env::attached_deposit();
let cost = if attestation_storage_must_be_paid_by_caller {
    env::storage_byte_cost()
        .saturating_mul(env::storage_usage().saturating_sub(initial_storage) as u128)
} else {
    NearToken::from_yoctonear(0)
};
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
```

Alternatively, add a top-level unconditional refund after the existing `if` block, reading `env::attached_deposit()` once and refunding anything above the computed cost (or above zero for the update path).

---

### Proof of Concept

1. Participant `alice.near` has already submitted a valid attestation (entry exists in `stored_attestations`).
2. `alice.near` calls `submit_participant_info` again (e.g., to refresh an expiring quote) and attaches `1 NEAR` thinking it may be needed.
3. `add_participant` returns `UpdatedExistingParticipant` → `is_new_attestation = false`.
4. `voter_account()` succeeds for `alice.near` → `caller_is_not_participant = false`.
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if` block is skipped entirely; `env::attached_deposit()` is never read.
7. The call succeeds, the attestation is updated, and `1 NEAR` is permanently credited to the contract with no refund issued. [4](#0-3) [5](#0-4)
