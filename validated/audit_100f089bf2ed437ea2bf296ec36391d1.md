Audit Report

## Title
Single Depositor Can Exhaust Global Daily Mint Limit, DoS-ing All Subsequent Deposits for Up to 24 Hours - (File: contracts/RSETH.sol)

## Summary
`RSETH.sol` enforces a single shared `currentPeriodMintedAmount` counter against `maxMintAmountPerDay` with no per-user sub-limit. Any unprivileged depositor can call `LRTDepositPool.depositETH()` or `depositAsset()` with an amount sized to consume the entire remaining daily quota, causing every subsequent `RSETH.mint()` call to revert with `DailyMintLimitExceeded` for up to 24 hours. The attack is repeatable at the start of each new period at the cost of only opportunity capital.

## Finding Description
`RSETH.checkDailyMintLimit` ( [1](#0-0) ) maintains a single global counter `currentPeriodMintedAmount` ( [2](#0-1) ) with no per-address tracking. `RSETH.mint()` is gated by this modifier ( [3](#0-2) ) and requires `MINTER_ROLE`, which is held by `LRTDepositPool` — not by end users directly.

The deposit entry points `depositETH` and `depositAsset` are fully permissionless (any external caller, no role required) ( [4](#0-3) ). They call `_mintRsETH`, which calls `IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint)` ( [5](#0-4) ). The `MINTER_ROLE` check is satisfied by `LRTDepositPool` itself, not the end user, so any depositor can trigger `RSETH.mint()` indirectly.

Exploit path:
1. Attacker calls `LRTDepositPool.depositETH{value: V}()` where `V` is sized so that `getRsETHAmountToMint` returns an amount equal to `maxMintAmountPerDay - currentPeriodMintedAmount`.
2. `checkDailyMintLimit` passes (equality is allowed: `>` not `>=`), and `currentPeriodMintedAmount` is set to `maxMintAmountPerDay`. ( [6](#0-5) )
3. Every subsequent `RSETH.mint()` call reverts with `DailyMintLimitExceeded` until `block.timestamp >= periodStartTime + 1 days`. ( [7](#0-6) )

No existing check in `_beforeDeposit` prevents a single large deposit from consuming the full daily quota — it only checks the per-asset total deposit limit, not the rsETH daily mint limit. ( [8](#0-7) )

## Impact Explanation
**Medium — Temporary freezing of funds.** All honest depositors are completely blocked from depositing ETH or LSTs into the protocol for up to 24 hours per attack cycle. Both `depositETH` and `depositAsset` revert because they ultimately call `RSETH.mint()`, which is gated by the exhausted daily limit. This is a temporary but complete denial of the deposit service for all users, matching the allowed impact "Medium. Temporary freezing of funds."

## Likelihood Explanation
**Medium.** The attacker must supply capital equal to the remaining daily quota denominated in ETH/LST value. They receive rsETH proportional to their deposit and do not permanently lose funds; the net cost is only the opportunity cost of capital locked in the EigenLayer withdrawal queue (7+ days). No special role or privileged access is required — only a standard `depositETH` call. A well-capitalised actor (competing protocol, large holder, or griefing party) can execute this repeatedly at the start of each 24-hour window.

## Recommendation
1. **Per-user sub-limits**: Track how much each address has minted within the current period and cap individual contributions to a fraction of `maxMintAmountPerDay`.
2. **Per-transaction cap**: Enforce a maximum rsETH amount mintable per single transaction (e.g., a configurable percentage of `maxMintAmountPerDay`).
3. **Minimum deposit cooldown**: Introduce a per-address cooldown between large deposits to prevent rapid quota exhaustion.

## Proof of Concept
```
Precondition:
  maxMintAmountPerDay = D (e.g., 1000 ETH worth of rsETH, set by LRT manager)
  currentPeriodMintedAmount = 0 (fresh period)

Step 1: Attacker calls LRTDepositPool.depositETH{value: V}(0, "")
        where V is the ETH amount such that getRsETHAmountToMint(ETH, V) == D.

Step 2: _beforeDeposit passes (deposit limit not exceeded, rsethAmountToMint >= 0).
        _mintRsETH calls RSETH.mint(attacker, D).
        checkDailyMintLimit(D): 0 + D == maxMintAmountPerDay → condition (> not >=) passes.
        currentPeriodMintedAmount is now D.

Step 3: Honest user calls LRTDepositPool.depositETH{value: 1 ether}(0, "").
        RSETH.mint() runs checkDailyMintLimit(rsethForUser):
        D + rsethForUser > maxMintAmountPerDay → REVERT DailyMintLimitExceeded.

Step 4: All deposits revert until block.timestamp >= periodStartTime + 1 days.
        Attacker repeats at the start of the next period.

Foundry test sketch:
  1. Fork mainnet or deploy mock contracts with maxMintAmountPerDay = 1000e18.
  2. vm.prank(attacker); depositPool.depositETH{value: V}(0, "");
  3. vm.prank(victim); vm.expectRevert(DailyMintLimitExceeded.selector);
     depositPool.depositETH{value: 1 ether}(0, "");
  4. Assert revert is triggered.
```

```text
    modifier checkDailyMintLimit(uint256 amount) {
        // Check if we need to reset the period if it has been more than 24 hours
        if (block.timestamp >= periodStartTime + 1 days) {
            currentPeriodMintedAmount = 0;
            periodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
            revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
        }

        currentPeriodMintedAmount += amount;
        _;
    }
```
