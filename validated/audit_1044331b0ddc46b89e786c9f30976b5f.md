Audit Report

## Title
Zero-output deposit silently drains user tokens with no rsETH minted ‚ÄĒ (`contracts/pools/RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPool.sol`)

## Summary
All four pool `deposit(token, amount, referralId)` variants compute `rsETHAmount` via integer division without a zero-output guard. When a user deposits a small amount of a low-decimal token, the division truncates to zero: the pool pulls the user's tokens via `safeTransferFrom`, records zero fee, and mints or transfers zero rsETH to the user ‚ÄĒ all without reverting. The deposited tokens are absorbed into the pool's general balance and swept to L1 on the next bridge call, permanently lost to the user.

## Finding Description
Every pool variant shares the same vulnerable pattern. In `RSETHPoolV3.sol`:

```solidity
// L282-293
if (amount == 0) revert InvalidAmount();
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // rsETHAmount can be 0
```

The computation at L324‚Äď334:
```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;  // truncates to 0
```

When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, integer division yields `rsETHAmount = 0`. The only existing guard (`amount == 0`) does not catch this case.

The `limitDailyMint` modifier also fails to protect: it computes `rsETHAmount` and checks `dailyMintAmount + rsETHAmount > dailyMintLimit`. With `rsETHAmount == 0`, this check adds 0 and never reverts.

Downstream calls with zero amount also succeed silently:
- `wrsETH.mint(msg.sender, 0)` ‚ÄĒ OpenZeppelin `_mint(to, 0)` is a no-op.
- `rsETH.safeTransfer(msg.sender, 0)` (`RSETHPoolNoWrapper`) ‚ÄĒ ERC-20 transfer of 0 succeeds per EIP-20.
- `IERC20(address(wrsETH)).safeTransfer(msg.sender, 0)` (`RSETHPool`) ‚ÄĒ same.

The identical pattern is confirmed in all four files:
- `RSETHPoolV3.sol` L282‚Äď293, L324‚Äď334
- `RSETHPoolV3ExternalBridge.sol` L401‚Äď411, L442‚Äď452
- `RSETHPoolNoWrapper.sol` L260‚Äď270, L301‚Äď311
- `RSETHPool.sol` L294‚Äď305

## Impact Explanation
A user who deposits a small amount of a supported low-decimal token permanently loses those tokens: they are transferred into the pool, never attributed to the user, and bridged to L1 on the next `bridgeTokens` / `moveAssetsForBridging` call. This constitutes a **direct, permanent loss of user funds** ‚ÄĒ matching the allowed impact: **Critical: Direct theft of any user funds, whether at-rest or in-motion**.

## Likelihood Explanation
The precondition is that at least one low-decimal token (e.g., USDC with 6 decimals) is added as a supported token via `addSupportedToken`. The pool architecture explicitly supports arbitrary ERC-20 tokens with oracles; this is a realistic operational state. Once such a token is supported, any unprivileged user depositing below the truncation threshold triggers the loss. For USDC with `tokenToETHRate ‚Čą 3.33e14` and `rsETHToETHrate ‚Čą 1.05e18`, the threshold is ~3154 USDC units (~$0.003). For a 2-decimal token (e.g., GUSD), the threshold rises to ~$0.30. The likelihood is **Low** in normal usage but the loss is real and unrecoverable per occurrence.

## Recommendation
Add a zero-output guard immediately after computing `rsETHAmount` in every `deposit(token, amount, referralId)` path across all four pool variants:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

This check must be placed **before** `safeTransferFrom` to avoid pulling tokens that will yield nothing, or at minimum before the mint/transfer call.

## Proof of Concept
Deploy `RSETHPoolV3` with a 6-decimal token (USDC) whose oracle reports `tokenToETHRate = 3.33e14`, `rsETHToETHrate = 1.05e18`, `feeBps = 0`:

```
amount = 3000  // 3000 USDC units ‚Čą $0.003
rsETHAmount = 3000 * 3.33e14 / 1.05e18
            = 9.99e17 / 1.05e18
            = 0  (integer truncation)
```

Call sequence:
1. `USDC.approve(pool, 3000)` ‚ÄĒ user approves pool.
2. `pool.deposit(USDC, 3000, "")` ‚ÄĒ executes without revert.
3. Pool state: `USDC.balanceOf(pool) += 3000`, `wrsETH.balanceOf(user)` unchanged.
4. Next `bridgeTokens(USDC, ...)` call sweeps the 3000 USDC units to L1; user has no claim.

Foundry fuzz test plan: fuzz `amount` in `[1, threshold)` for a mock 6-decimal token with the above oracle rates; assert that after `deposit`, `wrsETH.balanceOf(user) == 0` while `token.balanceOf(pool) == amount`. Confirm no revert occurs.
