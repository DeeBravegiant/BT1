### Title
`accept_requests` Flag and Stored Attestations Not Invalidated After TEE Whitelist Changes — (`File: crates/contract/src/lib.rs`)

---

### Summary

`vote_remove_launcher_hash()` and `vote_remove_os_measurement()` modify the TEE whitelist but do **not** update the `accept_requests` flag or invalidate stored attestations. The `is_caller_an_attested_participant()` helper does not re-verify stored attestations against the current whitelist. As a result, nodes running software whose launcher/OS hash has been unanimously revoked can continue to call `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` during the window between the whitelist change and the next explicit `verify_tee()` call — the exact same cache-inconsistency pattern as the NestedFactory `operatorCache` bug.

---

### Finding Description

`MpcContract` stores two pieces of derived TEE state:

1. **`accept_requests: bool`** — set to `true`/`false` by `verify_tee()`. All three `respond*` endpoints and `check_request_preconditions()` gate on this flag.
2. **`tee_state.stored_attestations`** — populated when a node calls `submit_participant_info()`. `is_caller_an_attested_participant()` only checks that a stored entry exists and that the account/key fields match; it does **not** re-verify the attestation against the current launcher-hash or OS-measurement whitelist.

The whitelist is mutated by two governance functions:

```rust
// crates/contract/src/lib.rs:1470-1495
pub fn vote_remove_launcher_hash(&mut self, launcher_hash: LauncherImageHash) -> Result<(), Error> {
    ...
    if votes >= total_participants {
        let removed = self.tee_state.remove_launcher_image(&launcher_hash);
        // ← accept_requests is NOT touched here
    }
    Ok(())
}
```

```rust
// crates/contract/src/lib.rs:1527-1552
pub fn vote_remove_os_measurement(&mut self, measurement: ContractExpectedMeasurements) -> Result<(), Error> {
    ...
    if votes >= total_participants {
        let removed = self.tee_state.remove_measurement(&measurement);
        // ← accept_requests is NOT touched here
    }
    Ok(())
}
```

Neither function calls `verify_tee()` or sets `accept_requests = false`. The only way to re-sync the flag is an explicit call to `verify_tee()`, which is not triggered automatically.

Meanwhile, `is_caller_an_attested_participant()` only checks the stored entry:

```rust
// crates/contract/src/tee/tee_state.rs:469-498
pub(crate) fn is_caller_an_attested_participant(&self, participants: &Participants) -> Result<(), AttestationCheckError> {
    ...
    let attestation = self.stored_attestations.get(&info.tls_public_key)
        .ok_or(AttestationCheckError::AttestationNotFound)?;
    // ← no re-verification against current launcher/OS whitelist
    Ok(())
}
```

`clean_invalid_attestations()` would prune stale entries, but it is only spawned as a detached promise after resharing completes — not after a whitelist removal.

The three `respond*` endpoints all rely on both guards:

```rust
// respond() — lib.rs:573-581
self.assert_caller_is_attested_participant_and_protocol_active(); // stale stored entry passes
if !self.accept_requests { return Err(...) }                      // stale true passes

// respond_ckd() — lib.rs:658-666
if !self.accept_requests { return Err(...) }                      // stale true passes
self.assert_caller_is_attested_participant_and_protocol_active(); // stale stored entry passes

// respond_verify_foreign_tx() — lib.rs:705-713
self.assert_caller_is_attested_participant_and_protocol_active(); // stale stored entry passes
if !self.accept_requests { return Err(...) }                      // stale true passes
```

---

### Impact Explanation

After a unanimous `vote_remove_launcher_hash()` or `vote_remove_os_measurement()` governance action — taken precisely because the software is considered unsafe — nodes running the revoked software can still:

- Call `respond()` to fulfill pending ECDSA/EdDSA signature requests.
- Call `respond_ckd()` to fulfill confidential key-derivation requests.
- Call `respond_verify_foreign_tx()` to fulfill foreign-chain verification requests.

The contract accepts these responses because `accept_requests` is still `true` and the stored attestation still passes the identity check. The TEE security boundary — the primary mechanism preventing compromised nodes from participating in signing — is bypassed for the entire window between the whitelist change and the next `verify_tee()` call.

This maps to the allowed impact: **Medium — contract execution-flow manipulation that breaks production safety/accounting invariants** (the invariant being "only nodes running approved software may submit signing responses").

---

### Likelihood Explanation

- Launcher-hash removal requires **all** participants to vote unanimously, so it is a deliberate, coordinated governance action taken when software is known to be unsafe.
- After the vote completes, there is no automatic mechanism to call `verify_tee()`. Nodes call it periodically, but the interval is not bounded by the contract.
- A node operator running the revoked software has a direct incentive to submit responses before being kicked out (e.g., to complete in-flight signing sessions that benefit the attacker).
- The attacker-controlled entry path is fully on-chain: call `respond()` / `respond_ckd()` / `respond_verify_foreign_tx()` directly.

---

### Recommendation

1. **Call `verify_tee()` (or set `accept_requests = false`) atomically inside `vote_remove_launcher_hash()` and `vote_remove_os_measurement()`** when the removal threshold is reached, analogous to how `rebuildCache()` should be called inside `removeOperator()`.
2. **Spawn `clean_invalid_attestations()` as a detached promise** after every successful whitelist removal, not only after resharing.
3. **Add a "whitelist-dirty" flag** (analogous to the NestedFactory recommendation to track removed operators in a separate array) so that `is_caller_an_attested_participant()` — or a new `is_tee_state_synced()` view — can signal that re-verification is required before the next signing response is accepted.

---

### Proof of Concept

```
1. Contract is Running; all N participants have valid attestations;
   accept_requests = true.

2. All N participants call vote_remove_launcher_hash(hash_X).
   → hash_X is removed from the whitelist.
   → accept_requests remains true.
   → stored_attestations for nodes running hash_X are NOT pruned.

3. A node running hash_X (now revoked) calls respond(request, sig).
   → assert_caller_is_attested_participant_and_protocol_active() passes
     (stored attestation exists, account/key fields match).
   → accept_requests check passes (still true).
   → Signature is accepted and the pending yield is resolved.

4. verify_tee() is eventually called.
   → reverify_and_cleanup_participants() finds hash_X nodes invalid.
   → Contract transitions to Resharing to kick them out.
   → But the signing response in step 3 has already been committed.

        Ok(())

    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
                }

                // here, we set it to true, because at this point, we have at least `threshold`
                // number of participants with an accepted Tee status.
                self.accept_requests = true;

                // do we want to adjust the threshold?
                //let n_participants_new = new_participants.len();
                //let new_threshold = (3 * n_participants_new + 4) / 5; // minimum 60%
                //let new_threshold = new_threshold.max(2); // but also minimum 2
                let new_threshold = usize::try_from(current_params.threshold().value())
                    .expect("threshold value fits in usize");

                let threshold_parameters = ThresholdParameters::new(
                    participants_with_valid_attestation,
                    Threshold::new(new_threshold as u64),
                )
                .expect("Require valid threshold parameters"); // this should never happen.
                current_params.validate_incoming_proposal(&threshold_parameters)?;
                // This resharing only changes the participant set, so the
                // per-domain reconstruction-threshold updates map is empty.
                let proposed_parameters =
                    ProposedThresholdParameters::new(threshold_parameters, BTreeMap::new());
                let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
                if let Some(resharing) = res {
                    self.protocol_state = ProtocolContractState::Resharing(resharing);
                }

                Ok(true)
            }
        }
    }

```rust
    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
