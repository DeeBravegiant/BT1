Audit Report

## Title
Missing Minimum Output Guard in `instantWithdrawal` Exposes Users to Unprotected Slippage - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager::instantWithdrawal` burns rsETH irreversibly before computing the final asset output, yet accepts no `minAssetAmountExpected` parameter and performs no post-fee floor check on `userAmount`. Because both oracle prices are read at execution time, any price movement between a user's off-chain preview and on-chain execution silently reduces the asset received with no on-chain recourse. The deposit path (`LRTDepositPool::depositAsset` / `depositETH`) already enforces `minRSETHAmountExpected`, making the withdrawal path inconsistently protected.

## Finding Description
The execution order in `instantWithdrawal` (L212–253) is:

1. `assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked)` — oracle read at execution time (L228).
2. `IRSETH(...).burnFrom(msg.sender, rsETHUnstaked)` — rsETH destroyed, irreversible (L229).
3. Fee deducted: `fee = assetAmountUnlocked * instantWithdrawalFee / 10_000` (L237).
4. `userAmount = assetAmountUnlocked - fee` (L238) — no floor check.
5. `_transferAsset(asset, msg.sender, userAmount)` (L250).

`getExpectedAssetAmount` (L580–594) computes:
```
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)
```
Both prices are live oracle reads. Any block-level movement in `rsETHPrice` or `getAssetPrice(asset)` between the user's preview call and the transaction landing changes `assetAmountUnlocked` — and therefore `userAmount` — with no guard to abort. The function signature `(address asset, uint256 rsETHUnstaked, string calldata referralId)` has no `minAssetAmountExpected` slot. By contrast, `LRTDepositPool::_beforeDeposit` (L667–669) explicitly checks `rsethAmountToMint < minRSETHAmountExpected` and reverts with `MinimumAmountToReceiveNotMet`. No equivalent protection exists on the withdrawal path.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews via `getExpectedAssetAmount` and then calls `instantWithdrawal` may receive materially fewer assets than expected. The rsETH is already burned at step 2; there is no on-chain mechanism to enforce a minimum acceptable output. The contract fails to deliver the return the user was shown, matching the allowed Low impact class exactly.

## Likelihood Explanation
**Medium.** No adversarial action is required. Oracle prices for rsETH and LSTs update on every oracle refresh cycle. Any block-level price movement between preview and execution silently reduces `userAmount`. The condition is triggered by ordinary market behavior and affects every `instantWithdrawal` call that is not executed atomically with its preview. The manager fee-change vector (acknowledged in the NatSpec at L210–211) is a separate, privileged-action concern and is not relied upon here.

## Recommendation
Add a `minAssetAmountExpected` parameter and enforce it after fee deduction, mirroring the deposit-side pattern:

```solidity
function instantWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    uint256 minAssetAmountExpected,   // add
    string calldata referralId
) external ... {
    ...
    uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
    uint256 userAmount = assetAmountUnlocked - fee;
    if (userAmount < minAssetAmountExpected) revert SlippageExceeded(); // add
    _transferAsset(asset, msg.sender, userAmount);
}
```

## Proof of Concept
1. rsETH/ETH oracle = 1.05e18; stETH/ETH oracle = 1.00e18.
2. User calls `getExpectedAssetAmount(stETH, 1e18)` → 1.05e18 stETH previewed.
3. User submits `instantWithdrawal(stETH, 1e18, referralId)`.
4. Before the tx lands, oracle updates: rsETH/ETH drops to 1.00e18.
5. `assetAmountUnlocked = 1e18 * 1.00e18 / 1.00e18 = 1.00e18` (L228, L593).
6. rsETH burned at L229 — irreversible.
7. `fee = 1.00e18 * 500 / 10_000 = 0.05e18` (5% fee, L237).
8. `userAmount = 0.95e18` (L238) — 9.5% less than previewed, no revert.

Foundry fork test plan: deploy against a mainnet fork, mock the oracle to return 1.05e18 at preview time and 1.00e18 at execution time, call `instantWithdrawal`, assert `userAmount < 1.05e18 * 0.95` and that rsETH supply decreased by `rsETHUnstaked`.
