Audit Report

## Title
Excess ETH Permanently Locked in `MultiChainRateProvider` Due to No Refund Mechanism - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`MultiChainRateProvider.updateRate()` is a `payable` function with no access control that forwards only the on-chain `estimatedFee` to each LayerZero `send()` call. Any ETH included in `msg.value` beyond the sum of all per-chain estimated fees is permanently trapped in the contract, as the contract declares no `receive()` fallback, no `fallback()` function, and no owner-callable ETH sweep. There is no recovery path for the stranded ETH.

## Finding Description
`updateRate()` at line 108 is `external payable nonReentrant` with no access restriction. Inside the loop (lines 119–134), it calls `estimateFees()` per destination chain and then calls `send{ value: estimatedFee }(...)`, forwarding only that exact amount to LayerZero. The refund address passed to LayerZero is `payable(msg.sender)` (line 128), so LayerZero refunds any unused portion of the `estimatedFee` slice back to the caller — but only for the ETH it actually received. The delta `msg.value − Σ estimatedFee` is never forwarded to LayerZero and never returned to the caller; it remains in the contract's own balance. The contract has no `receive()`, no `fallback()`, and no function that transfers out native ETH, so this balance is irrecoverable. [1](#0-0) [2](#0-1) 

## Impact Explanation
Any ETH sent to `updateRate()` in excess of the exact sum of on-chain estimated fees is permanently frozen in the contract with no recovery path. This constitutes **permanent freezing of funds** — a Critical-tier impact in the allowed scope. The contract provides `estimateTotalFee()` as a view helper, but callers routinely add a safety buffer to cross-chain calls to guard against fee fluctuations between the estimation block and the execution block; every such call results in a permanent, unrecoverable ETH loss. [3](#0-2) 

## Likelihood Explanation
`updateRate()` has no access control and is expected to be called regularly by any external account to push rate updates cross-chain. Callers who query `estimateTotalFee()` off-chain and add even a 1 wei buffer will permanently lose that buffer on every call. The function is designed for repeated, routine invocation, making cumulative losses realistic over the protocol's lifetime.

## Recommendation
Refund any unused ETH to `msg.sender` at the end of `updateRate()`:

```solidity
function updateRate() external payable nonReentrant {
    // ... existing loop ...
    uint256 remaining = address(this).balance;
    if (remaining > 0) {
        (bool ok,) = payable(msg.sender).call{value: remaining}("");
        require(ok, "ETH refund failed");
    }
}
```

Alternatively, compute the total required fee before the loop and `require(msg.value == totalFee)` to reject over-payments entirely.

## Proof of Concept
1. Deploy `MultiChainRateProvider` with two registered rate receivers on different chains.
2. Call `estimateTotalFee()` off-chain; receive `X` wei.
3. Call `updateRate{value: X + 1000}()` (1000 wei safety buffer).
4. The loop forwards exactly `estimatedFee_1 + estimatedFee_2 = X` to LayerZero across two `send()` calls.
5. The remaining 1000 wei stays in the contract's ETH balance.
6. No function in the contract can move that balance. The ETH is permanently locked.

Foundry test sketch:
```solidity
function test_excessEthLocked() public {
    uint256 totalFee = provider.estimateTotalFee();
    uint256 excess = 1000 wei;
    provider.updateRate{value: totalFee + excess}();
    assertEq(address(provider).balance, excess); // excess is stuck
}
```
