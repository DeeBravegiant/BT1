### Title
Queued Future-Round Proposals Bypass `block_timestamp_window_seconds` Check, Allowing a Malicious Proposer to Commit a Block with an Out-of-Window Timestamp — (`crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

### Summary

When `validate_proposal` receives a `ProposalInit` for a **future round**, it stores the proposal in `queued_proposals` without performing any timestamp validation. When the proposal is later dequeued in `set_height_and_round`, `is_proposal_init_valid` evaluates the timestamp against `clock.unix_now()` at **dequeue time**, not at **receipt time**. Because clock time has advanced, a timestamp that would have been rejected at receipt time passes at dequeue time. The effective upper bound on the block timestamp becomes `window + queuing_delay` seconds in the future rather than just `window` seconds, breaking the invariant that `block_timestamp_window_seconds` is meant to enforce. The committed block timestamp flows directly into `PartialBlockHashComponents`, the block hash, and the `get_block_timestamp` syscall context seen by every transaction in the block.

### Finding Description

**Receipt path (current round — timestamp validated at receipt time T0):**

```rust

        block_timestamp_window_seconds: self.config.static_config.block_timestamp_window_seconds,

```

`is_proposal_init_valid` is called immediately, checking `init.timestamp <= T0 + window`.

**Queue path (future round — NO timestamp validation at receipt time T0):**

```rust
// sequencer_consensus_context.rs:873-878
std::cmp::Ordering::Greater => {
    trace!("Queueing proposal for future round.");
    self.queued_proposals
        .insert(init.round, ((init, timeout, content_receiver), fin_sender));
    fin_receiver
}
```

No validation occurs. The raw `init` (including its proposer-supplied `timestamp`) is stored verbatim.

**Dequeue path (in `set_height_and_round`, at time T1 > T0):**

```rust
// sequencer_consensus_context.rs:1178-1207
let proposal_init_validation = ProposalInitValidation {
    block_timestamp_window_seconds: self.config.static_config.block_timestamp_window_seconds,
    ...
};
self.validate_current_round_proposal(init, proposal_init_validation, ...).await;
```

`is_proposal_init_valid` now runs at T1, checking `init.timestamp <= T1 + window`. Since T1 = T0 + queuing_delay, the effective ceiling is `T0 + queuing_delay + window`.

**The timestamp check itself:**

```rust
// validate_proposal.rs:260,273-284
let now: u64 = clock.unix_now();   // T1 at dequeue time
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
    return Err(...);
}
```

A proposer who knows they will be the proposer for round R (deterministic in Tendermint) can send the proposal early with `timestamp = T0 + window + delta`. At T0 this would fail (`T0 + window + delta > T0 + window`). At T1 = T0 + delta it passes (`T0 + window + delta <= T0 + delta + window`).

**L1 gas price lookup uses the future timestamp:**

```rust
// validate_proposal.rs:322-328

    l1_gas_price_provider,
```

If the oracle has no data for that future timestamp, it falls back to previous-block prices, so the committed L1 gas prices may be stale.

### Impact Explanation

1. **Wrong block timestamp committed to chain state.** `PartialBlockHashComponents::timestamp` is set from `init.timestamp` and hashed into the `PartialBlockHash` / block hash. Every downstream consumer (provers, L1 anchoring, RPC `starknet_getBlock`) sees a timestamp that is further in the future than the protocol's `block_timestamp_window_seconds` bound allows.

2. **Wrong execution context for all transactions in the block.** The `get_block_timestamp` syscall returns the future timestamp. Smart contracts that use block timestamp for time-based logic (timelocks, expiry checks, oracle freshness guards) will observe an incorrect value, potentially allowing premature or delayed execution of time-sensitive operations.

3. **Incorrect L1 gas prices committed.** `get_l1_prices_in_fri_and_wei` is called at the future timestamp. If the oracle lacks data for that point in time, it silently falls back to previous-block prices. The committed `l1_gas_price` / `l1_data_gas_price` in the block header and block hash may therefore be stale, causing incorrect fee accounting for all transactions in the block.

### Likelihood Explanation

In Tendermint-style consensus the proposer for each round is deterministic (derived from the validator set and round number). Any validator can therefore know in advance that they will be the proposer for round R and send the proposal early. The attack requires no special privilege beyond being a validator. The queuing mechanism is a normal part of the protocol (proposals for future rounds are explicitly queued), so the window is always open when consensus is running multiple rounds.

### Recommendation

Validate the timestamp at **receipt time**, not only at dequeue time. The simplest fix is to call `is_proposal_init_valid` (or at minimum the timestamp bounds check) inside the `Ordering::Greater` branch before inserting into `queued_proposals`, using `clock.unix_now()` at that moment. Alternatively, record the receipt timestamp alongside the queued proposal and enforce `init.timestamp <= receipt_time + window` at dequeue time instead of `init.timestamp <= dequeue_time + window`.

### Proof of Concept

Let `block_timestamp_window_seconds = 30`, round duration ≈ 10 s, and the attacker be the deterministic proposer for round 3.

1. **T0 = 0 s** — Consensus is at round 0. Attacker sends `ProposalInit { round: 3, timestamp: 91 }` (= T0 + window + 61 = 0 + 30 + 61). At T0 the check would be `91 > 0 + 30` → **rejected**. But the `Ordering::Greater` branch queues it without checking.

2. **T1 = 61 s** — Consensus advances to round 3 (`set_height_and_round`). The proposal is dequeued. `is_proposal_init_valid` runs: `now = 61`, check `91 > 61 + 30 = 91` → **false** (equal, not greater) → **passes**.

3. The block is committed with `block_timestamp = 91`, which is 61 seconds in the future relative to when the proposal was received, far exceeding the intended 30-second window.

4. Every transaction in the block sees `get_block_timestamp() == 91` instead of the correct ≈ 61. L1 gas prices are fetched at timestamp 91; if the oracle has no data that far ahead, stale previous-block prices are committed.
