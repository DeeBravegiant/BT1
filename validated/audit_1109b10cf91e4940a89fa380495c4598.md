Audit Report

## Title
Missing Slippage Protection in L2 Pool `deposit()` Functions Allows Users to Receive Fewer wrsETH/rsETH Than Expected - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The L2 pool `deposit()` functions in `RSETHPoolV3`, `RSETHPoolNoWrapper`, and `RSETHPoolV3WithNativeChainBridge` compute the output token amount at execution time using the live oracle rate, with no caller-specified minimum output parameter. Because `CrossChainRateProvider.updateRate()` is permissionlessly callable and the rsETH/ETH rate monotonically increases with staking rewards, any deposit transaction can execute at a worse rate than the user previewed via `viewSwapRsETHAmountAndFee()`. The L1 counterpart `LRTDepositPool.depositETH()` already enforces a `minRSETHAmountExpected` guard; the L2 pools do not.

## Finding Description
`RSETHPoolV3.deposit()` calls `viewSwapRsETHAmountAndFee()` at execution time, which reads the live rate via `getRate()`:

```solidity
// RSETHPoolV3.sol L246-265
function deposit(string memory referralId) external payable nonReentrant whenNotPaused ... {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    wrsETH.mint(msg.sender, rsETHAmount);
}

// RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (...) {
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
```

The same pattern is present in `RSETHPoolNoWrapper.deposit()` (L231-244) and `RSETHPoolV3WithNativeChainBridge.deposit()` (L282-301). None accept a `minRSETHAmountExpected` parameter.

The rate source, `CrossChainRateProvider.updateRate()`, is callable by any external account with no access control:

```solidity
// CrossChainRateProvider.sol L85
function updateRate() external payable nonReentrant { ... }
```

This pushes a fresh rate to `CrossChainRateReceiver`, which stores it and returns it via `getRate()`. Because the rsETH/ETH rate only increases (staking rewards), any rate update between a user's preview and their deposit execution results in fewer output tokens.

By contrast, `LRTDepositPool.depositETH()` on L1 enforces:

```solidity
// LRTDepositPool.sol L76-93
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId) external payable ...
// LRTDepositPool.sol L665-669
if (rsethAmountToMint < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
```

No equivalent guard exists in any L2 pool. The user has no on-chain mechanism to reject execution at a worse-than-previewed rate.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.** A user who previews the swap and submits a deposit receives fewer wrsETH/rsETH tokens than the preview indicated. The ETH value is approximately preserved (each token is worth more ETH), but the contract fails to deliver the token quantity the user observed and acted upon. No ETH is stolen; the shortfall is structural and non-recoverable per transaction.

## Likelihood Explanation
The rsETH/ETH rate increases continuously as EigenLayer staking rewards accrue, meaning any deposit pending in the mempool is exposed to a rate change without any attacker involvement. Additionally, `CrossChainRateProvider.updateRate()` is permissionlessly callable by any external account, allowing anyone to push a fresh (higher) rate to the L2 oracle at any time, including in the same block as a user's deposit. This is a routine operational condition, not a rare edge case.

## Recommendation
Add a `minRSETHAmountExpected` parameter to all L2 pool `deposit()` functions, mirroring the protection in `LRTDepositPool.depositETH()`. Revert if the computed output falls below the caller-specified minimum:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused ...
{
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

Apply the same fix to the token-deposit overload and to `RSETHPoolNoWrapper`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolV3ExternalBridge`.

## Proof of Concept
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3`. Oracle rate is `1.05e18`. Preview shows `~0.952 wrsETH`.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before inclusion, any account calls `CrossChainRateProvider.updateRate()` on L1, pushing rate `1.06e18` to `CrossChainRateReceiver` via LayerZero.
4. User's `deposit()` executes. `getRate()` returns `1.06e18`. User receives `~0.943 wrsETH` — ~1% fewer tokens than previewed.
5. No on-chain check rejects this outcome.

**Foundry fork test plan**: Fork an L2 where `RSETHPoolV3` is deployed. Call `viewSwapRsETHAmountAndFee(1 ether)` to record `expectedAmount`. Mock `CrossChainRateReceiver.rate` to a higher value (simulating a `lzReceive` update). Call `deposit{value: 1 ether}("ref")`. Assert `wrsETH.balanceOf(user) < expectedAmount` and that no revert occurred.
