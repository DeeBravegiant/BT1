Audit Report

## Title
Unbounded `referralId` String Enables Unbounded Gas Consumption and Block Stuffing — (`contracts/pools/RSETHPoolV2NBA.sol`)

## Summary
The `deposit` function accepts an unbounded `string memory referralId` parameter with no length validation. Because the string is copied into EVM memory during ABI decoding and re-encoded into log data during event emission, an attacker can craft a single deposit call with a multi-hundred-kilobyte `referralId` to consume gas approaching the block gas limit, enabling block stuffing or sustained network griefing at the attacker's own expense.

## Finding Description
The `deposit` function at line 106 accepts `string memory referralId` with no length guard:

```solidity
function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
``` [1](#0-0) 

The only precondition checked is `amount == 0`: [2](#0-1) 

The string is then passed directly to `emit SwapOccurred`, which carries an unindexed `string referralId` field, causing the full string to be ABI-encoded into log data: [3](#0-2) [4](#0-3) 

This results in two memory allocations for the string: once during ABI decoding (`string memory`) and once during event log encoding. EVM memory expansion cost is quadratic in the number of 32-byte words (`words² / 512 + 3 × words`), so gas consumption grows super-linearly with string length. At approximately 1.5 MB of non-zero calldata, the combined calldata cost (~25 M gas) plus memory expansion cost (~4.9 M gas) reaches the Ethereum block gas limit of ~30 M gas. No existing modifier (`nonReentrant`, `whenNotPaused`) constrains the string length.

## Impact Explanation
This matches two explicitly allowed impacts: **Medium — Unbounded gas consumption** and **Low — Block stuffing**. A single transaction with a sufficiently large `referralId` can consume the entire block gas limit, crowding out all other transactions in that block. The attacker bears the full gas cost, but the impact on the network and other users is concrete and reproducible.

## Likelihood Explanation
The attack path is fully permissionless. Any externally owned account can call `deposit` with `msg.value > 0` and an arbitrarily large `referralId`. No role check, whitelist, or access control applies. Sustained block stuffing is economically costly but feasible during time-sensitive MEV windows or targeted griefing campaigns. A single such transaction is trivially constructable with any Ethereum client or Foundry test.

## Recommendation
Add a maximum length guard at the top of `deposit`:

```solidity
uint256 private constant MAX_REFERRAL_ID_BYTES = 128;

function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
    require(bytes(referralId).length <= MAX_REFERRAL_ID_BYTES, "referralId too long");
    ...
}
```

Changing the parameter to `calldata` avoids the initial memory copy but does not eliminate the memory allocation during event emission, so the length check is necessary regardless of parameter location.

## Proof of Concept
Deploy `RSETHPoolV2NBA` on a local Foundry fork. Run the following test:

```solidity
function testUnboundedReferralIdGas() public {
    uint256[] memory lengths = new uint256[](4);
    lengths[0] = 1_000;
    lengths[1] = 10_000;
    lengths[2] = 100_000;
    lengths[3] = 500_000;

    for (uint256 i = 0; i < lengths.length; i++) {
        bytes memory b = new bytes(lengths[i]);
        for (uint256 j = 0; j < b.length; j++) b[j] = 0x41;

        uint256 gasBefore = gasleft();
        pool.deposit{value: 1 ether}(string(b));
        uint256 gasUsed = gasBefore - gasleft();

        emit log_named_uint("length", lengths[i]);
        emit log_named_uint("gasUsed", gasUsed);
        assertLt(gasUsed, 30_000_000, "exceeded block gas limit");
    }
}
```

The assertion passes for small lengths and fails (or reverts OOG) as `referralId` approaches hundreds of kilobytes, confirming super-linear gas growth. The gas measurements will show the quadratic memory expansion term dominating at large string sizes.
