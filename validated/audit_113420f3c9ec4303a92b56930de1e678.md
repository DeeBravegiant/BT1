### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards to Arbitrary Address via Unenforced `farmer_reward_address_override` - (File: chia/farmer/farmer_api.py)

### Summary
A connected harvester can set `farmer_reward_address_override` to any arbitrary puzzle hash in both `NewProofOfSpace` and `RespondSignatures` protocol messages. The farmer's only defense — a fee-quality check — is purely advisory (logs a warning only) and never blocks the override. The farmer unconditionally uses the attacker-supplied address in `DeclareProofOfSpace`, permanently diverting the 0.25 XCH farmer block reward to the attacker.

### Finding Description

CHIP-22 introduced a voluntary fee convention allowing third-party harvesters to take a portion of the farmer reward by overriding `farmer_reward_address_override`. The farmer is supposed to validate this via a "fee quality" check. However, the check only logs — it never rejects the override.

**Step 1 — Harvester sends `NewProofOfSpace` with override set:**

In `chia/farmer/farmer_api.py`, when a proof is good enough to make a block, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` if the override is present:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [1](#0-0) 

**Step 2 — Fee quality check only logs, never blocks:**

`notify_farmer_reward_taken_by_harvester_as_fee` in `chia/farmer/farmer.py` computes the fee quality and emits a warning if the threshold is violated, but returns `None` and does not raise or signal rejection:

```python
else:
    self.log.warning(
        "Harvester illegitimately took a fee reward that "
        + "did not belong to it or it incorrectly applied the fee convention."
    )
``` [2](#0-1) 

**Step 3 — Farmer unconditionally uses the override in `_process_respond_signatures`:**

When the harvester returns `RespondSignatures` with `farmer_reward_address_override` set, the farmer replaces its own configured `farmer_target` with the attacker-supplied address, with no fee quality check at all:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [3](#0-2) 

**Step 4 — Attacker address is broadcast in `DeclareProofOfSpace`:**

The attacker-controlled `farmer_reward_address` is placed directly into the `DeclareProofOfSpace` message sent to the full node, which then creates the coinbase reward coin to that address: [4](#0-3) 

The `RespondSignatures` message structure confirms `farmer_reward_address_override` is an optional field the harvester controls: [5](#0-4) 

### Impact Explanation

Every time the farmer wins a block, the 0.25 XCH farmer reward is permanently sent to the attacker's address instead of the farmer's configured `xch_target_address`. The farmer has no on-chain recourse — the coinbase coin is created directly to the attacker's puzzle hash. This is a direct, irreversible theft of XCH from the farmer on every won block.

This matches the allowed Critical/High impact: **reward diversion affecting XCH** and **bypass of farmer authorization that enables payout redirection**.

### Likelihood Explanation

Any harvester connected to the farmer — including third-party harvesters explicitly supported by CHIP-22 — can exploit this. The attacker does not need any keys, admin access, or cryptographic break. They only need to be a connected harvester peer, which is the normal operating mode for third-party harvesting services. The farmer has no mechanism to detect or prevent the diversion beyond a log message that most operators will never see.

### Recommendation

The fee quality check must be **enforced**, not merely logged. In `_process_respond_signatures`, before accepting `farmer_reward_address_override`, the farmer should:

1. Verify that the fee quality computed from the proof meets or exceeds the harvester's declared `applied_fee_threshold`.
2. If the check fails, reject the override and use `self.farmer.farmer_target` instead (or drop the block entirely).
3. Require `fee_info` to be present whenever `farmer_reward_address_override` is set; reject the override if `fee_info` is absent.

The check in `notify_farmer_reward_taken_by_harvester_as_fee` should be moved into `_process_respond_signatures` and made blocking (return `None` / drop the proof) rather than advisory.

### Proof of Concept

1. Operate a harvester that connects to a victim farmer.
2. When the harvester finds a valid proof of space that wins a block, send `NewProofOfSpace` with `farmer_reward_address_override` set to the attacker's puzzle hash and `fee_info` set to `applied_fee_threshold=0xFFFFFFFF` (or omit `fee_info` entirely).
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, which logs a warning but does not block.
4. The farmer sends `RequestSignatures` to the harvester.
5. The harvester responds with `RespondSignatures` also containing `farmer_reward_address_override` set to the attacker's address.
6. The farmer's `_process_respond_signatures` sets `farmer_reward_address = response.farmer_reward_address_override` with no validation.
7. The farmer broadcasts `DeclareProofOfSpace` with the attacker's address as `farmer_reward_address`.
8. The full node creates the 0.25 XCH farmer coinbase reward to the attacker's address.

The farmer's log will contain a warning, but the XCH is already irreversibly sent to the attacker.
