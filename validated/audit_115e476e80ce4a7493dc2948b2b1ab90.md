### Title
Empty Extractor List Accepted by `verify_foreign_transaction`, Enabling Signature Over Zero Verified Foreign-Chain Data - (File: `crates/contract/src/lib.rs`)

### Summary

The `verify_foreign_transaction` contract method accepts requests with an empty `extractors: vec![]` list. No minimum-extractor guard exists anywhere in the contract or node pipeline. As a result, an unprivileged caller can obtain a valid threshold MPC signature that commits to a foreign transaction ID but to **no actual on-chain data** (no block hash, no program ID, no data hash, no log). Any bridge or downstream NEAR contract that relies on the MPC signature as proof of specific foreign-chain state — without independently re-verifying the extracted values — will accept this vacuous attestation as genuine verification.

---

### Finding Description

The vulnerability class from the reference report is **silent-failure / zero-return from a fallible external call that bypasses a safety check**, where the "slippage guard" (`minReturn > 0`) is the analog of the extractor-count guard (`extractors.len() > 0`).

**Root cause — contract layer (`crates/contract/src/lib.rs`)**

`verify_foreign_transaction` performs three checks before enqueuing the yield:

1. Gas/deposit preconditions (`check_request_preconditions`)
2. Chain is in the supported set
3. Domain purpose is `ForeignTx`

There is no check that `request.request.extractors()` is non-empty. [1](#0-0) 

**Root cause — node layer (`crates/node/src/providers/verify_foreign_tx/sign.rs`)**

`execute_foreign_chain_request` maps the caller-supplied extractor list directly to the inspector call. When the list is empty, the inspector returns `values = vec![]` without error, and the function constructs a `ForeignTxSignPayload::V1 { request, values: vec![] }` that is then signed by the threshold protocol. [2](#0-1) 

**Root cause — respond layer (`crates/contract/src/lib.rs`)**

`respond_verify_foreign_tx` verifies only that the ECDSA signature is valid against `response.payload_hash`. It does **not** verify that `payload_hash` encodes a non-empty `values` list, nor that the number of extracted values matches the number of requested extractors. [3](#0-2) 

**Confirmed by tests**

Inspector-level tests explicitly confirm that passing `Vec::new()` as the extractor list succeeds and returns an empty extracted-value list for Bitcoin, EVM chains, and Starknet. [4](#0-3) [5](#0-4) [6](#0-5) 

The SDK builder tests also confirm that building a request without calling any `with_extractor_*` method produces `extractors: vec![]`, and this is treated as a valid request. [7](#0-6) 

---

### Impact Explanation

The signed payload is `SHA-256(borsh(ForeignTxSignPayload { request, values: [] }))`. [8](#0-7) 

This signature is cryptographically valid and passes `respond_verify_foreign_tx`'s signature check. It is returned to the caller as a `VerifyForeignTransactionResponse`. The response commits to the transaction ID and chain, but to **zero** extracted properties — no block hash, no program ID, no data hash, no log.

The primary use case documented for this feature is the Omnibridge inbound flow, where a NEAR bridge contract must confirm that a specific foreign-chain transaction (a) finalized, (b) called the correct program, and (c) carried specific payload data. With `extractors: []`, properties (b) and (c) are never verified and never committed to in the signature. An attacker can:

1. Submit **any** finalized transaction on the foreign chain (including a no-op transfer to themselves).
2. Call `verify_foreign_transaction` with `extractors: []`.
3. Receive a valid threshold MPC signature over `(tx_id, [])`.
4. Present this to a bridge contract that does not independently re-verify the extracted values.
5. The bridge contract, seeing a valid MPC signature over the correct transaction ID, releases funds or mints tokens.

This maps to the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

- The attack requires only a valid NEAR account and a deposit of 1 yoctoNEAR — no privileged access, no threshold collusion.
- The SDK's `ForeignChainRequestBuilder` defaults to `extractors: vec![]` when no extractor is added, making it easy for a developer to accidentally submit an empty-extractor request.
- Bridge contracts that use the MPC signature as a "seal of approval" without calling `ForeignChainSignatureVerifier::verify_signature` with the expected extracted values are directly exploitable.
- The `ForeignChainSignatureVerifier` in the SDK is provided as a helper but its use is not enforced by the protocol. [9](#0-8) 

---

### Recommendation

1. **Contract-level guard**: In `verify_foreign_transaction`, reject requests where `request.request.extractors().is_empty()` with a clear error (`InvalidParameters::EmptyExtractorList`).
2. **Node-level guard**: In `execute_foreign_chain_request`, `bail!` if the resolved extractor list is empty before calling the inspector.
3. **Respond-level guard**: In `respond_verify_foreign_tx`, verify that `response.payload_hash` decodes to a payload whose `values` list length equals the number of extractors in the pending request (requires storing extractor count in the pending-request map entry).
4. **SDK enforcement**: Make `ForeignChainRequestBuilder::build()` return an error or panic if no extractor has been added.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)

        .with_foreign_tx_domain()
        .build()
        .await;
    let foreign_tx_key = setup.foreign_tx_key();
    register_foreign_chain_configuration(chain, &setup.contract, &setup.mpc_signer_accounts).await;

    let user = setup.worker.dev_create_account().await.unwrap();
    let domain_id = dtos::DomainId(foreign_tx_key.domain_id().0);

    let request_args = dtos::VerifyForeignTransactionRequestArgs {
        domain_id,
        payload_version: ForeignTxPayloadVersion::V1,
        request: rpc_request.clone(),
    };

    let status = user
        .call(
            setup.contract.id(),
            method_names::VERIFY_FOREIGN_TRANSACTION,
        )
        .args_json(json!({ "request": request_args }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact_async()
        .await
        .unwrap();

    let verify_request = VerifyForeignTransactionRequest {
        domain_id,
        payload_version: ForeignTxPayloadVersion::V1,
        request: rpc_request,
    };

    await_pending_foreign_tx_request_observed_on_contract(&setup.contract, &verify_request).await;

    let (payload, response) = sign_foreign_tx_response(
        &verify_request.request,
        extracted_values,
        foreign_tx_key.as_secp256k1(),
    );

    let respond_result = setup.mpc_signer_accounts[0]
        .call(setup.contract.id(), method_names::RESPOND_VERIFY_FOREIGN_TX)
        .args_json(json!({
            "request": verify_request,
            "response": response,
        }))
        .max_gas()
        .transact()
        .await
        .unwrap()
        .into_result();

    assert!(
        respond_result.is_ok(),
        "respond_verify_foreign_tx should succeed for {chain:?}: {respond_result:?}",
    );

    let execution = status.await.unwrap().into_result().unwrap();
    let returned: VerifyForeignTransactionResponse = execution.json().unwrap();
    assert_eq!(returned.payload_hash, payload.compute_msg_hash().unwrap());
}
