### Title
Unbounded Extractor List in `verify_foreign_transaction` Passes Unvalidated Request Parameters to Foreign-Chain RPC, Enabling Presignature Exhaustion and Request-Lifecycle Manipulation — (`crates/near-mpc-contract-interface/src/types/foreign_chain.rs`, `crates/contract/src/lib.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

The `verify_foreign_transaction` contract endpoint accepts a `ForeignChainRpcRequest` whose `extractors` field is an unbounded `Vec<T>` with no length validation at the contract or node level. The design document explicitly states "The request includes a bounded number of extractors," but this invariant is never enforced in code. An unprivileged caller can submit a request with an arbitrarily large extractor list. Each MPC node will attempt to process every extractor within a 5-second timeout; if the timeout fires, the presignature already taken from the pool is permanently consumed and the pending on-chain request is never resolved. This breaks the request-lifecycle and presignature-accounting invariants of the MPC network.

---

### Finding Description

The design contract for `verify_foreign_transaction` states three bounding invariants:

> "Each extractor returns exactly one typed value. The request includes a **bounded number of extractors**. Extracted values have strict size limits."

In the actual implementation, every chain-specific request struct uses a plain, unbounded `Vec`:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs
pub struct EvmRpcRequest {
    pub tx_id: EvmTxId,
    pub extractors: Vec<EvmExtractor>,   // no length bound
    pub finality: EvmFinality,
}

pub struct SolanaRpcRequest {
    pub tx_id: SolanaTxId,
    pub finality: SolanaFinality,
    pub extractors: Vec<SolanaExtractor>, // no length bound
}

pub struct BitcoinRpcRequest {
    pub tx_id: BitcoinTxId,
    pub confirmations: BlockConfirmations,
    pub extractors: Vec<BitcoinExtractor>, // no length bound
}
```

The contract's `verify_foreign_transaction` function validates only the domain purpose, the supported-chain list, gas, and deposit. It performs **no validation of the extractor count**:

```rust
// crates/contract/src/lib.rs
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(
        request.domain_id,
        DomainPurpose::ForeignTx,
        Gas::from_tgas(...),
        MINIMUM_SIGN_REQUEST_DEPOSIT,
    );
    let requested_chain = request.request.chain();
    let supported_chains = self.get_supported_foreign_chains();
    if !supported_chains.contains(&requested_chain) { env::panic_str(...); }
    // No extractor count check — request is enqueued as-is
    let request = args_into_verify_foreign_tx_request(request);
    self.enqueue_yield_request(...);
}
```

The MPC node's `execute_foreign_chain_request` iterates over every extractor in the caller-supplied list and passes them all to the inspector, also without any count guard:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs
let extractors: Vec<BitcoinExtractor> = request
    .extractors
    .iter()
    .cloned()
    .map(TryInto::try_into)
    .collect::<Result<_, _>>()?;
let extracted_values = inspector
    .extract(transaction_id, block_confirmations, extractors)
    .timeout(FOREIGN_CHAIN_INSPECTION_TIMEOUT)   // 5 seconds
    .await
    .context("timed out during execution of foreign chain request")??;
```

Critically, the presignature is **taken from the pool before** `execute_foreign_chain_request` is called:

```rust
// make_verify_foreign_tx_leader
let (presignature_id, presignature) = domain_data.presignature_store.take_owned().await;
// ...
let response_payload = self
    .execute_foreign_chain_request(&foreign_tx_request.request, ...)
    .await?;   // if this errors/times out, presignature is already gone
```

If `execute_foreign_chain_request` times out or errors, the presignature is permanently consumed and cannot be returned to the pool.

---

### Impact Explanation

**Impact: Medium** — Request-lifecycle and presignature-accounting invariant violation.

1. **Presignature exhaustion**: An attacker submits many `verify_foreign_transaction` requests with large extractor lists (e.g., thousands of `EvmExtractor::Log { log_index: N }` entries). Each request causes every MPC node to attempt processing all extractors within 5 seconds. If the timeout fires, the presignature taken from the pool is permanently consumed. Presignatures are expensive to generate (they require multi-round OT-based MPC computation). Exhausting the pool delays or blocks all legitimate signing operations until new presignatures are generated.

2. **Pending-request queue pollution**: Each submitted request creates a pending yield on-chain that is never resolved (no `respond_verify_foreign_tx` call arrives). These zombie entries occupy the `pending_verify_foreign_tx_requests` map until the on-chain timeout fires, consuming contract storage and blocking duplicate-request deduplication for legitimate callers submitting the same `(chain, tx_id)`.

3. **Node divergence on partial timeout**: If some nodes complete processing (with a small extractor list) while others time out (with a large one), the leader/follower signing protocol diverges — the leader has a payload to sign but followers do not, causing the threshold signature to fail and the presignature to be wasted on all participating nodes.

This matches the allowed Medium impact: *"request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

**Likelihood: Medium.**

- Any unprivileged NEAR account can call `verify_foreign_transaction` with a 1 yoctonear deposit (essentially free).
- The chain only needs to be in the supported-chain list (e.g., Bitcoin, Abstract, BNB, Base, Arbitrum, Polygon, Starknet, Aptos — all registered in production).
- No special knowledge or privileged access is required.
- The attack is repeatable and can be automated.

---

### Recommendation

1. **Enforce an extractor count limit at the contract level** in `verify_foreign_transaction`, rejecting requests whose extractor list exceeds a small constant (e.g., 8 or 16):

```rust
const MAX_EXTRACTORS: usize = 16;
// In verify_foreign_transaction:
let extractor_count = match &request.request {
    ForeignChainRpcRequest::Bitcoin(r) => r.extractors.len(),
    ForeignChainRpcRequest::Abstract(r) | ForeignChainRpcRequest::Ethereum(r) | ... => r.extractors.len(),
    // ...
};
if extractor_count > MAX_EXTRACTORS {
    env::panic_str("too many extractors");
}
```

2. **Replace `Vec<T>` with `BoundedVec<T, 0, MAX_EXTRACTORS>`** in all `*RpcRequest` structs (the codebase already has `near_mpc_bounded_collections::BoundedVec` available and used elsewhere).

3. **Return the presignature to the pool on `execute_foreign_chain_request` failure** (or restructure the leader flow to validate the request before consuming the presignature).

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
