### Title
Missing `DirectDeposits` in Preservation-of-Value Check Allows Unbounded ADA Creation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

In the Dijkstra era, a new transaction body field `DirectDeposits` allows a transaction sender to credit arbitrary amounts of ADA directly to registered staking accounts. The ENTITIES/SUBENTITIES rules apply these credits to account balances unconditionally, but the amounts are never included in the `produced` side of the preservation-of-value check (`consumed = produced`). An unprivileged transaction sender can therefore include `directDeposits` of any magnitude in a transaction, pass the conservation check, and have those amounts credited to accounts — creating ADA out of thin air.

---

### Finding Description

**Vulnerability type:** Funds/accounting bug — missing term in preservation-of-value equation.

The Dijkstra era introduces `DirectDeposits`, a map from `AccountAddress` to `Coin` embedded in both top-level and sub-transaction bodies. [1](#0-0) 

When a transaction is processed, the ENTITIES rule applies these deposits to account balances: [2](#0-1) 

The same applies in the SUBENTITIES rule for sub-transactions: [3](#0-2) 

`applyDirectDeposits` unconditionally adds each deposit amount to the matching account balance: [4](#0-3) 

The preservation-of-value check in the UTXO rule is: [5](#0-4)

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

Consider a transaction with:
- Inputs: 100 ADA
- Outputs: 95 ADA
- Fee: 5 ADA
- `directDeposits`: 10 ADA (to a registered account)

The conservation check evaluates:
- `consumed` = 100 ADA (inputs)
- `produced` = 95 + 5 = 100 ADA (outputs + fee; `directDeposits` absent)
- Check passes: 100 = 100

Post-transaction ADA distribution:
- 95 ADA in UTxO outputs
- 5 ADA in fee pot
- 10 ADA in account balance (from `directDeposits`)
- **Total: 110 ADA — 10 ADA created from nothing**

The credited account balance can subsequently be withdrawn via the normal withdrawal mechanism, extracting the fabricated ADA into the UTxO. This applies equally to sub-transactions via the SUBENTITIES rule.

---

### Likelihood Explanation

Any transaction sender in the Dijkstra era can include a `directDeposits` field in their transaction body. No privileged role, governance majority, or key compromise is required. The only prerequisite is that the target account address is registered, which the attacker can arrange by registering their own staking credential. The attack is deterministic and repeatable.

---

### Recommendation

Include the sum of all `directDeposits` amounts in the `produced` value. For top-level transactions, `dijkstraProducedValue` should add:

```haskell
<> inject (fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
```

The same correction must be applied in `dijkstraSubTxProducedValue` for sub-transactions. This ensures that every lovelace credited to an account via `directDeposits` is deducted from the transaction's inputs, preserving the invariant `consumed = produced`.

---

### Proof of Concept

1. Register a staking credential `C` and obtain its `AccountAddress` `A`.
2. Construct a Dijkstra-era transaction:
   - Inputs: one UTxO output of value `V`
   - Outputs: one output of value `V − fee`
   - Fee: `fee` (minimum valid fee)
   - `directDeposits`: `{A ↦ D}` for any `D > 0`
3. Submit the transaction. The UTXO rule evaluates:
   - `consumed = V`
   - `produced = (V − fee) + fee = V` ✓ (check passes)
4. The ENTITIES rule applies `applyDirectDeposits`, crediting `D` lovelace to account `C`.
5. Submit a withdrawal transaction draining account `C` for `D` lovelace into a UTxO output.
6. Net result: `D` lovelace has been created and extracted into the UTxO with no corresponding deduction from any prior balance. [11](#0-10) [12](#0-11) [13](#0-12)
