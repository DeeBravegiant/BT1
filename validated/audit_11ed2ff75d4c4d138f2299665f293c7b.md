### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Submitted Request — Cross-Request Signature Replay by a Single Byzantine Participant - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that the MPC signature in the response is valid for `response.payload_hash`, but it never checks that `response.payload_hash` was actually derived from the `request` that is being resolved. A single Byzantine attested participant (strictly below the signing threshold) can reuse any previously observed, legitimately-produced MPC signature to resolve a completely different pending `verify_foreign_transaction` request, delivering a forged `VerifyForeignTransactionResponse` to the waiting caller.

### Finding Description

The `verify_foreign_transaction` flow is a two-step process:

**Step 1 — Request submission.** A user calls `verify_foreign_transaction(request)`. The contract stores the request in `pending_verify_foreign_tx_requests` and parks the caller's transaction via a yield promise.

**Step 2 — Response delivery.** An attested MPC participant calls `respond_verify_foreign_tx(request, response)`. The contract:
1. Verifies the caller is an attested participant.
2. Verifies `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key.
3. Looks up `request` in `pending_verify_foreign_tx_requests` and resumes all parked yields with `response`.

The critical gap is in step 2: the contract **only** checks that the signature is valid for `response.payload_hash`. It does **not** verify that `response.payload_hash` encodes the same `request.request` that was submitted in step 1.

The intended `payload_hash` is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, extracted_values }))
```

Because `extracted_values` are determined off-chain by the MPC nodes, the contract cannot recompute the hash. However, the `request` portion is fully known on-chain and is never checked for consistency with `response.payload_hash`.

The relevant code in `respond_verify_foreign_tx`:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← taken directly from the response, never bound to `request`
    &secp_pk,
)
.is_ok()
```

After this check passes, the contract resolves the yield for `request` with the full `response` (including the unverified `payload_hash`):

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),  // ← forged payload_hash delivered verbatim
)
```

### Impact Explanation

A NEAR contract that calls `verify_foreign_transaction` to gate a bridge action (e.g., Omnibridge inbound transfer) receives a `VerifyForeignTransactionResponse` containing `payload_hash` and `signature`. The contract is supposed to verify that `payload_hash` encodes the expected `(request, extracted_values)`. However, the MPC contract itself delivers the response without enforcing this binding, so any caller contract that trusts the MPC contract's output without re-verifying the `payload_hash` will accept a forged attestation.

Concretely: a single Byzantine attested participant can make the MPC contract attest that Bitcoin transaction T₁ was finalized with block-hash B₁, when in fact the user asked about transaction T₂. A bridge contract that does not re-derive and compare the `payload_hash` will release funds based on a fabricated foreign-chain observation, enabling theft.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution**.

### Likelihood Explanation

The attacker needs only:
1. **To be a single attested MPC participant** — Byzantine, strictly below the signing threshold. No collusion with other participants is required.
2. **A previously observed valid MPC signature** — any legitimately processed `verify_foreign_transaction` response published on-chain provides `(payload_hash_old, sig_old)`. The attacker does not need to produce a new signature.

Both conditions are easily satisfied in a live network. The attack is a straightforward on-chain transaction replay requiring no cryptographic capability beyond reading the blockchain.

### Recommendation

The contract must verify that `response.payload_hash` encodes the same `request.request` that is being resolved. Because `extracted_values` are unknown on-chain, the contract cannot recompute the full hash. Two mitigations are possible:

1. **Prefix binding**: Require that `payload_hash` is the hash of a structure whose first field is a canonical encoding of `request.request`. The contract can then verify the prefix of the pre-image by requiring the submitter to also provide the `extracted_values`, or by storing a commitment to `request.request` at submission time and checking it against the hash prefix.

2. **Commitment at submission time**: When `verify_foreign_transaction` is called, store `SHA-256(borsh(request.request))` alongside the yield index. In `respond_verify_foreign_tx`, require the submitter to provide `extracted_values` in the response, recompute `expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1{request.request, extracted_values}))`, and assert `expected_hash == response.payload_hash` before accepting the signature.

### Proof of Concept

**Setup**: Attacker is a single attested MPC participant. Two `verify_foreign_transaction` requests exist or have existed:
- R_old: Bitcoin tx `[0x01; 32]`, extractor `BlockHash` — already resolved on-chain with `(payload_hash_old, sig_old)`.
- R_new: Bitcoin tx `[0x02; 32]`, extractor `BlockHash` — currently pending in `pending_verify_foreign_tx_requests`.

**Attack**:

```rust
// Attacker reads (payload_hash_old, sig_old) from a previous on-chain respond_verify_foreign_tx call.
// Attacker calls:
contract.respond_verify_foreign_tx(
    request  = R_new,                          // matches the pending request key
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_old,        // hash of R_old's payload, NOT R_new's
        signature:    sig_old,                 // valid MPC signature for payload_hash_old
    },
);
```

**Contract execution**:
1. `assert_caller_is_attested_participant_and_protocol_active()` — passes (attacker is attested).
2. `verify_ecdsa_signature(sig_old, payload_hash_old, root_pk)` — passes (sig_old is a valid signature for payload_hash_old).
3. `resolve_yields_for(&mut pending_verify_foreign_tx_requests, &R_new, serialize(response))` — R_new is in the map, so the yield is resumed with the forged response.

**Result**: The user who submitted R_new receives `{payload_hash_old, sig_old}`. `payload_hash_old` encodes Bitcoin tx `[0x01; 32]`, not `[0x02; 32]`.
