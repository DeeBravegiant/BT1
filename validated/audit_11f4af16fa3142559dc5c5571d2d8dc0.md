### Title
Missing subgroup check on `root_Q`/`root_K` in `blob_to_keyset` allows torsion public key to survive refresh — (`src/cbmpc/api/hd_keyset_eddsa_2p.cpp`)

### Summary
`blob_to_keyset` deserializes `root_Q` and `root_K` using `from_bin`, which only validates that the bytes decode to a point on the Ed25519 curve. It never calls `curve.check()`, which is the function that enforces prime-order subgroup membership. A caller-controlled keyset blob encoding a torsion `root_Q` (e.g., the order-2 point `(0, p-1)`) is accepted without error, and the torsion point is then copied verbatim into the refreshed keyset by `key_share_eddsa_hdmpc_2p_t::refresh()`, which also performs no re-validation. The output `new_keyset_blob` encodes the torsion `root_Q` as if it were a valid public key.

### Finding Description
The deserialization path `refresh()` → `deserialize_keyset_blob()` → `blob_to_keyset()` contains a missing subgroup guard:

```cpp
// blob_to_keyset — hd_keyset_eddsa_2p.cpp lines 78-80
error_t rv = keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed);
if (rv) return rv;
return keyset.root.K.from_bin(keyset.curve, blob.root_K_compressed);
// ← NO curve.check(keyset.root.Q) or curve.check(keyset.root.K)
```

`ecurve_ed_t::from_bin` only decodes the compressed encoding and verifies the point satisfies the curve equation; it does not check subgroup membership. The Ed25519 curve has cofactor 8, so there are 7 non-trivial torsion points (orders 2, 4, 8) that are on the curve but not in the prime-order subgroup. All of them pass `from_bin` and fail `curve.check()`.

`key_share_eddsa_hdmpc_2p_t::refresh()` then copies the tainted point directly:

```cpp
// hd_keyset_eddsa_2p.cpp lines 61-62
new_key.root.Q = key.root.Q;   // torsion point copied, no check
new_key.root.K = key.root.K;
```

The refreshed blob is then serialized and returned to the caller with the torsion `root_Q` intact.

Every other Ed25519 public-key deserializer in the same codebase calls `curve.check()` after `from_bin` (e.g., `eddsa_mp.cpp` line 137, `eddsa2pc.cpp` line 204/216). The `hd_keyset_eddsa_2p` path is the only one that omits this guard.

### Impact Explanation
An API caller who controls the `keyset_blob` argument to `refresh()` can supply a blob whose `root_Q_compressed` encodes a torsion point. The call succeeds, and the returned `new_keyset_blob` encodes the same torsion point as the root public key. Subsequent calls to `derive_eddsa_2p_keys()` or `extract_root_public_key_compressed()` on the refreshed blob will operate on a torsion `root_Q`, producing derived keys whose public keys are `T + delta*G` (torsion-shifted) rather than the correct `Q + delta*G`. This is a public-API-reachable validation bypass in the refresh path that creates accepted invalid cryptographic output, matching the "High" impact criterion.

### Likelihood Explanation
The attacker only needs to craft a 32-byte compressed point encoding a known torsion point (the order-2 point encoding is documented in the codebase's own tests) and embed it in an otherwise valid keyset blob. No protocol interaction with the other party is required; the entire attack is local to the `refresh()` call. The preconditions (valid `role`, `x_share`/`k_share` in range) are trivially satisfiable.

### Recommendation
Add `curve.check()` calls immediately after each `from_bin` in `blob_to_keyset`, mirroring the pattern used in every other Ed25519 deserializer in the codebase:

```cpp
error_t rv = keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed);
if (rv) return rv;
if (rv = keyset.curve.check(keyset.root.Q)) return coinbase::error(E_FORMAT, "invalid root_Q");
rv = keyset.root.K.from_bin(keyset.curve, blob.root_K_compressed);
if (rv) return rv;
if (rv = keyset.curve.check(keyset.root.K)) return coinbase::error(E_FORMAT, "invalid root_K");
return SUCCESS;
```

The same fix should be applied to the analogous `blob_to_keyset` in `hd_keyset_ecdsa_2p.cpp`. [7](#0-6) 

### Proof of Concept
1. Take any valid keyset blob produced by `dkg()`.
2. Deserialize it, replace `root_Q_compressed` with the 32-byte encoding of the Ed25519 order-2 torsion point (`0xec 0xff…0xff 0x7f`), re-serialize.
3. Call `refresh()` with the crafted blob. Observe `SUCCESS` return.
4. Call `extract_root_public_key_compressed()` on the returned `new_keyset_blob`.
5. Decode the returned bytes as an Ed25519 point and call `is_in_subgroup()` — it returns `false`, confirming the torsion point survived the refresh cycle.
