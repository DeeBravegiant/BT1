### Title
`Map.withoutKeys` Unconditionally Deletes Shared VRF Key Hash Ignoring Reference Count, Allowing Post-v11 Duplicate VRF Registration — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

In `poolReapTransition`, dangling VRF key hashes (old VRF keys of re-registering pools) are removed from `psVRFKeyHashesL` via `Map.withoutKeys`, which deletes the map entry entirely regardless of its reference count. When a second pool shares the same VRF key (registered pre-v11 when no uniqueness check existed), the count in `psVRFKeyHashes` is ≥ 2, but `Map.withoutKeys` still removes the entry completely. After the epoch boundary, the VRF key is absent from `psVRFKeyHashes`, so the post-v11 `Map.notMember` guard passes and a new pool can register with that VRF key — violating the uniqueness invariant introduced in Conway.

---

### Finding Description

**Root cause — `PoolReap.hs` lines 224–227:**

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )
```

In Haskell composition `(f . g) x = f (g x)`, so the application order is:

1. `Map.withoutKeys danglingVRFKeyHashes` — deletes the key entirely, ignoring count
2. `removeVRFKeyHashOccurrences retiredVRFKeyHashes` — correctly decrements counts

`danglingVRFKeyHashes` is computed as the set of *current* VRF hashes for pools that have a pending re-registration with a *different* VRF:

```haskell
danglingVRFKeyHashes =
  Set.fromList $ Map.elems $
    Map.merge Map.dropMissing Map.dropMissing
      ( Map.zipWithMaybeMatched $ \_ sps sppF ->
          if sps ^. spsVrfL /= sppF ^. sppVrfL then Just (sps ^. spsVrfL) else Nothing
      )
      (ps0 ^. psStakePoolsL)
      (ps0 ^. psFutureStakePoolParamsL)
```

This correctly identifies the old VRF hash, but the subsequent `Map.withoutKeys` call does not respect the reference count — it removes the entry even when count = 2.

**Contrast with the correct path for retired pools:** `removeVRFKeyHashOccurrences` (line 234–237) uses `Map.update (mapNonZero (\n -> n - 1))`, which only removes the key when the count reaches zero. The dangling path has no equivalent decrement — it hard-deletes.

**How `psVRFKeyHashes` gets count ≥ 2:** At the Conway hard fork, `populateVRFKeyHashes` (HardFork.hs lines 107–125) iterates all pools and uses `Map.insertWith combine` with an incrementing combiner, so if two pools share VRF-X, the count is set to 2.

**Attack steps (all via pool registration certificates, no privileged access):**

1. Before v11: Attacker registers Pool P with VRF-X. Pool Q (innocent or attacker-controlled) is also registered with VRF-X. No uniqueness check exists pre-v11.
2. Conway hard fork: `populateVRFKeyHashes` sets `psVRFKeyHashes[VRF-X] = 2`.
3. After v11: Attacker re-registers P with VRF-Y. `updateFutureVRFKeyHash` inserts VRF-Y with count 1 and does not touch VRF-X. `psVRFKeyHashes[VRF-X]` remains 2.
4. Epoch boundary: `danglingVRFKeyHashes = {VRF-X}`. `Map.withoutKeys` deletes VRF-X entirely. `psVRFKeyHashes[VRF-X]` is now absent, even though Q still uses VRF-X.
5. Attacker registers new Pool R with VRF-X. The guard `Map.notMember sppVrf psVRFKeyHashes` passes (VRF-X is absent). R is registered with VRF-X.
6. Q and R now both hold VRF-X post-v11, violating the uniqueness invariant.

---

### Impact Explanation

The VRF key uniqueness invariant introduced in Conway is bypassed. Pools Q and R share VRF-X, meaning they are elected for identical slots (VRF output is identical for any given slot nonce). The attacker (R) competes directly with Q for block production on every slot Q is elected for, enabling reward diversion outside design parameters. The corrupted `psVRFKeyHashes` state persists across epochs and is not self-correcting.

**Impact category: Medium** — attacker-controlled pool registration certificates exceed the intended VRF uniqueness validation limit and enable reward manipulation outside design parameters.

---

### Likelihood Explanation

- Requires two pools sharing VRF-X before v11. Pre-v11 VRF keys are public on-chain; an attacker can copy any existing pool's VRF key when registering their own pool before the hard fork.
- Requires one re-registration with a new VRF key after v11 — a normal, permitted operation.
- Requires waiting one epoch boundary — no special timing or racing needed.
- No governance majority, leaked keys, or privileged access required.
- Fully reproducible in a local test environment.

---

### Recommendation

Replace `Map.withoutKeys danglingVRFKeyHashes` with a reference-count-aware decrement, consistent with how `removeVRFKeyHashOccurrences` handles retired pools:

```haskell
-- Instead of:
. (`Map.withoutKeys` danglingVRFKeyHashes)

-- Use:
. removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
```

This ensures that if VRF-X has count 2 (shared by two pools), removing one pool's reference decrements to 1 rather than deleting the entry entirely.

### Proof of Concept

Construct a state with:
- `psStakePools = { P → spsVrf=VRF-X, Q → spsVrf=VRF-X }`
- `psFutureStakePoolParams = { P → sppVrf=VRF-Y }`
- `psVRFKeyHashes = { VRF-X → 2, VRF-Y → 1 }`

Run `poolReapTransition` for the epoch boundary. Assert:

1. After transition: `Map.lookup VRF-X psVRFKeyHashes == Just 1` (Q still holds VRF-X)
2. **Actual result:** `Map.lookup VRF-X psVRFKeyHashes == Nothing` (deleted by `Map.withoutKeys`)
3. Consequence: a subsequent `RegPool` certificate for new Pool R with VRF-X passes the `Map.notMember` guard and succeeds, producing `psVRFKeyHashes = { VRF-X → 1, VRF-Y → 1 }` with both Q and R holding VRF-X.

```haskell
populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
```
