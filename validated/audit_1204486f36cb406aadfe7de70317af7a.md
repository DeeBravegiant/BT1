### Title
Deposit Permanently Burnt When `promise_set_refund_to` Targets a Non-Existent Account - (`runtime/near-vm-runner/src/logic/logic.rs`)

### Summary

The `promise_set_refund_to` host function accepts any syntactically valid `AccountId` as the refund destination without verifying the account exists on-chain. If the associated promise fails, the runtime generates a deposit refund receipt to that account. Because refund receipts cannot create accounts and a failed system receipt causes the deposit to be **permanently burnt**, any deposit attached to the failing promise is irrecoverably lost.

### Finding Description

`promise_set_refund_to` in `logic.rs` only validates the account ID format via `read_and_parse_account_id`, then unconditionally stores it as the `refund_to` field: [1](#0-0) 

No check is made that the account exists at call time or will exist when the refund is processed. The stored value is later retrieved by `balance_refund_receiver()`: [2](#0-1) 

When the promise fails, `refund_unspent_gas_and_deposits` generates a deposit refund receipt addressed to that account: [3](#0-2) 

The refund receipt is a system receipt (`predecessor_id == "system"`). The protocol explicitly states that if a refund receipt fails, the amount is **burnt**: [4](#0-3) 

This is enforced in `apply_action_receipt`: [5](#0-4) 

Refund receipts cannot create accounts — this is an explicit design constraint: [6](#0-5) 

So a transfer refund to a non-existent named account fails with `AccountDoesNotExist`, and the deposit is burnt.

### Impact Explanation

The corrupted protocol value is the **user's account balance**: a deposit attached to a failing cross-contract call is permanently burnt instead of being returned. This is a direct, irreversible loss of user funds. The broken invariant is that deposit refunds are supposed to ensure funds are never lost due to failed calls — the exact guarantee the `refund_to` mechanism was designed to provide.

### Likelihood Explanation

Any deployed contract can call `promise_set_refund_to` with an arbitrary account ID. Realistic triggers include:

1. A contract redirects refunds to a treasury/DAO account that is later deleted.
2. A contract is ported from another chain and sets `refund_to` to an account that does not exist on NEAR.
3. A contract developer makes a typo in the account ID string.

In all cases, every user who subsequently triggers a failing call with a deposit loses their funds permanently. The host function is publicly callable by any deployed contract, making this reachable by any unprivileged user who interacts with such a contract.

### Recommendation

`promise_set_refund_to` should validate that the provided `refund_to` account exists on-chain at the time of the call, similar to how `promise_batch_create` validates the target account. Alternatively, the runtime should fall back to `predecessor_id` if the `refund_to` account does not exist when the refund receipt is processed, rather than burning the funds.

### Proof of Concept

1. Deploy a contract that implements:
   ```rust
   // In a function call handler:
   let promise = env::promise_batch_create("target.near");
   env::promise_batch_action_function_call(promise, "method", b"", deposit, gas);
   env::promise_set_refund_to(promise, "nonexistent-account.near"); // no existence check
   ```
2. Call the contract with a non-zero deposit attached.
3. Ensure `target.near` does not have the called method (so the promise fails).
4. Observe: the deposit refund receipt is generated for `nonexistent-account.near`.
5. The refund receipt fails with `AccountDoesNotExist` (since refunds cannot create accounts per `check_transfer_to_nonexisting_account`).
6. Since `predecessor_id == "system"`, the failure path at `runtime/runtime/src/lib.rs:916-921` burns the deposit.
7. The caller's deposit is permanently lost. [7](#0-6) [8](#0-7) [9](#0-8)

        &mut self,
        promise_idx: u64,
        account_id_len: u64,

        if self.context.is_view() {
            return Err(HostError::ProhibitedInView {
                method_name: "promise_set_refund_to".to_string(),
            }
            .into());
        }
        let refund_to = self.read_and_parse_account_id(account_id_ptr, account_id_len)?;
        let promise = self
            .promises
            .get(promise_idx as usize)
            .ok_or(HostError::InvalidPromiseIndex { promise_idx })?;

        let receipt_idx = match &promise {
            Promise::Receipt(receipt_idx) => Ok(*receipt_idx),
            Promise::NotReceipt(_) => Err(HostError::CannotSetRefundToOnJointPromise),
        }?;

        self.ext.set_refund_to(receipt_idx, refund_to);
        Ok(())
