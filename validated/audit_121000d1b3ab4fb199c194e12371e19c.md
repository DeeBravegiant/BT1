### Title
Silent Coin-State Drop for R-CAT Coins When Victim Already Holds a Non-Empty CATWallet for the Same Asset — (`chia/wallet/wallet_state_manager.py`)

### Summary

`WalletStateManager.handle_cat` contains a `return None` at line 1323 that fires when `RCATWallet.convert_to_revocable` refuses to convert a CATWallet whose `lineage_store` is non-empty. This early return bypasses the DEFAULT_CATS / `automatically_add_unknown_cats` fallback at line 1325, causing the incoming R-CAT coin state to be silently discarded. Any unprivileged on-chain actor who sends an R-CAT coin (same tail hash) to a victim who already holds confirmed CAT coins for that asset can permanently hide those R-CAT coins from the victim's wallet.

### Finding Description

**Exact call sequence (all line numbers verified against the repo):**

1. `_add_coin_states` → `determine_coin_type` → `handle_cat` [1](#0-0) 

2. Inside `handle_cat`, the hint resolves to a valid derivation record (line 1248). The plain-CAT puzzle hash does not match the R-CAT coin's puzzle hash (line 1258), so the code uncurries the puzzle reveal and calls `match_revocation_layer` — which succeeds — setting `wallet_type = RCATWallet` (lines 1263–1265). [2](#0-1) 

3. The block at line 1297 iterates over all `WalletType.CAT` wallets. It finds the victim's existing `CATWallet` for the same `asset_id` (line 1303) and calls `RCATWallet.convert_to_revocable` (lines 1313–1318). [3](#0-2) 

4. `convert_to_revocable` checks `lineage_store.is_empty()` at line 165. Because the victim already has at least one confirmed CAT coin, the store is non-empty → the function logs an error and returns `False`. [4](#0-3) 

5. Back in `handle_cat`, the `else` branch at line 1322–1323 executes `return None` **immediately**, without breaking out of the for-loop or falling through to the DEFAULT_CATS / `automatically_add_unknown_cats` check at line 1325. [5](#0-4) 

6. `determine_coin_type` returns `(None, cat_data)`. Back in `_add_coin_states`, `wallet_identifier is None` at line 1852 → the code logs "No wallet for coin state" and `continue`s — the coin state is permanently dropped. [6](#0-5) 

**Root cause:** The `return None` at line 1323 is inside the `for wallet_info in ...` loop. It should be a `break` (or `continue` on the outer loop) so that execution falls through to line 1325, where a fresh `RCATWallet` would be created for the incoming R-CAT coins. The analogous `CRCATWallet.convert_to_cr` path (lines 1304–1312) always returns a `WalletIdentifier` on success and never returns `None` on failure, so it does not share this defect.

**Secondary observation (not the primary finding):** Line 1321 returns `WalletType(WalletType.CRCAT)` instead of `WalletType.RCAT` on a successful conversion, which would mis-classify the converted wallet. This is a separate bug. [7](#0-6) 

### Impact Explanation

The victim's R-CAT coins for the affected asset are never written to the coin store, never surfaced in the wallet UI, and cannot be spent via the Chia wallet. Because the coin state is dropped at the sync layer (not merely deferred), no retry mechanism rescues it. The coins remain on-chain but are permanently invisible to the wallet software. This constitutes **permanent corruption of wallet sync state** and a **long-lived inability to process valid sync updates**, both of which are listed High impacts.

### Likelihood Explanation

Preconditions are realistic:
- **Victim side:** Any user who has received and confirmed at least one CAT coin for a DEFAULT_CATS token (Spacebucks, Marmot, wUSDC, etc.) satisfies the non-empty `lineage_store` requirement. This is a common state.
- **Attacker side:** The attacker needs to possess R-CAT coins with the same tail hash and know the victim's hint (inner puzzle hash), which is broadcast in prior transactions. Minting or acquiring R-CAT coins for a given tail is a valid on-chain action requiring no special privilege.

No leaked keys, admin access, or broken cryptography is required.

### Recommendation

In `handle_cat`, replace `return None` at line 1323 with a `break` so that the loop exits and execution continues to the DEFAULT_CATS / `automatically_add_unknown_cats` fallback, which will create a new `RCATWallet` for the incoming R-CAT coins:

```python
# chia/wallet/wallet_state_manager.py  ~line 1319
if success:
    self.state_changed("converted cat wallet to revocable", wallet_info.id)
    return WalletIdentifier(wallet_info.id, WalletType(WalletType.RCAT))  # also fix type
else:
    break  # fall through to create a new RCATWallet below
```

Additionally, fix the `WalletType.CRCAT` typo on line 1321 to `WalletType.RCAT`.

### Proof of Concept

```
1. Create wallet A with a CATWallet for a DEFAULT_CATS asset_id (e.g. Spacebucks).
2. Confirm at least one CAT coin → lineage_store is non-empty.
3. From wallet B, construct and broadcast an R-CAT spend bundle:
   - inner puzzle = revocation_layer(hidden_ph, wallet_A_inner_ph)
   - tail hash = Spacebucks asset_id
   - hint = wallet_A_inner_ph  (so handle_cat finds a derivation record)
4. After the block is confirmed, assert that wallet A's coin store does NOT
   contain the R-CAT coin → demonstrates permanent invisibility.
5. Confirm the bug: add a breakpoint at wallet_state_manager.py:1323 and
   observe `return None` fires while line 1325 is never reached.
```
