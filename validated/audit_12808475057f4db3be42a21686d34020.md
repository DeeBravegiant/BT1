### Title
Unprotected `init` Function Allows Anyone to Initialize the MPC Contract with Attacker-Controlled Participant Set — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract::init` function carries no caller access control. Any account can call it before the legitimate deployer does. If an attacker races the deployer's initialization transaction, they can supply a `ThresholdParameters` value that names only themselves as the sole participant with a threshold of 1, seizing full control of the MPC signing network.

---

### Finding Description

`MpcContract::init` is the primary initialization entry point for the on-chain MPC coordinator. It accepts an arbitrary `ThresholdParameters` value (participant set + governance threshold) and writes it directly into `ProtocolContractState::Running`. The NEAR SDK `#[init]` attribute guarantees the function executes only once — but it does **not** restrict who may call it. There is no `#[private]` guard, no `predecessor_account_id` check, and no comparison against the contract's own account ID or any pre-authorized deployer address.

```rust
// crates/contract/src/lib.rs  lines 1924-1973
#[handle_result]
#[init]                          // ← one-shot, but open to any caller
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
    // only logs the signer; no authorization check
    log!("init: signer={}, ...", env::signer_account_id(), ...);
    parameters.validate()?;      // validates structure, not caller identity
    Ok(Self {
        protocol_state: ProtocolContractState::Running(
            RunningContractState::new(
                DomainRegistry::default(),
                Keyset::new(EpochId::new(0), Vec::new()),
                parameters,          // ← attacker-supplied
                AddDomainsVotes::default(),
            )
        ),
        ...
    })
}
```

By contrast, the sibling `init_running` function — which also initializes the contract — carries `#[private]`, restricting it to self-calls only. The absence of the same guard on `init` is the root cause. [1](#0-0) 

The deployment workflow in the e2e test harness shows that contract deployment and `init` are issued as **separate transactions**: [2](#0-1) 

This two-step pattern creates a window between the `DeployContract` action and the `init` call during which any observer can race in with their own `init` transaction.

---

### Impact Explanation

An attacker who wins the race supplies a `ThresholdParameters` containing only their own account as the sole participant and a governance threshold of 1. After that:

- Every subsequent governance vote (`vote_new_parameters`, `vote_add_domains`, `vote_update`, etc.) requires only the attacker's signature.
- The attacker can immediately call `vote_add_domains` to register signing domains, then drive DKG to completion, producing MPC key shares under their sole control.
- With threshold 1 and a single participant, the attacker can unilaterally call `respond` to resolve any pending signature request with a signature of their choosing, or drive resharing to lock out the legitimate operators permanently.
- All funds and cross-chain assets whose custody depends on the MPC network's threshold signatures are at risk of theft or permanent freezing.

This matches the **Critical** impact class: unauthorized threshold-signature issuance and permanent freezing of funds controlled by the MPC network. [3](#0-2) 

---

### Likelihood Explanation

NEAR does not expose a public gas-price mempool in the same way Ethereum does, so classical miner-extractable front-running is not directly applicable. However:

1. NEAR block explorers and RPC nodes expose all transactions in near-real time. An attacker monitoring for `DeployContract` actions on the target account can submit their `init` call within the same or the next block.
2. The two-step deploy-then-init pattern is used in the production deployment workflow (confirmed by the e2e cluster setup code), so the window reliably exists.
3. The attacker needs no special privilege — any NEAR account with enough NEAR for gas can execute the attack.
4. The attack is a one-shot, irreversible action: once `init` succeeds, the `#[init]` guard prevents any further initialization, and the legitimate deployer's call will fail.

Likelihood: **Medium** (requires active monitoring of the target account, but no privileged access).

---

### Recommendation

**Short term:** Add an explicit caller check at the top of `init` that restricts execution to the contract's own account (i.e., the deployer):

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    // Only the contract account itself (the deployer) may initialize.
    assert_eq!(
        env::predecessor_account_id(),
        env::current_account_id(),
        "init: caller must be the contract account"
    );
    // ... rest of function unchanged
}
```

Alternatively, add `#[private]` (which enforces `predecessor == current_account_id` in NEAR SDK) and issue the `init` call as a self-call batched in the same transaction as `DeployContract`.

**Long term:** Audit all `#[init]` entry points for missing caller restrictions. Document the required deployment procedure (deploy + init in a single batched transaction) so operators cannot accidentally create the vulnerable two-step window.

---

### Proof of Concept

1. Legitimate deployer submits `DeployContract` action to account `mpc.near` (transaction A, block N).
2. Attacker observes the deployment via RPC polling or a block explorer.
3. Attacker submits a `FunctionCall` to `mpc.near::init` with:
   ```json
   {
     "parameters": {
       "participants": { "participants": [["attacker.near", 0, { "tls_public_key": "<attacker_key>" }]] },
       "threshold": 1
     }
   }
   ```
   in block N or N+1 (transaction B).
4. If transaction B is included before the deployer's own `init` call (transaction C), the contract is initialized with the attacker as the sole participant.
5. Transaction C fails with "contract already initialized."
6. The attacker now controls the MPC network: they can drive DKG, produce threshold signatures unilaterally, and sign arbitrary foreign-chain transactions. [4](#0-3) [5](#0-4)

async fn init_contract(
    blockchain: &NearBlockchain,
    contract: &DeployedContract,
    ports: &E2ePortAllocator,

) -> anyhow::Result<()> {
    let InitContractArgs {
        near_keys,
        p2p_keys,
        threshold,
        participant_indices,
        init_format,
    } = args;

    let participants = build_participants(&participant_indices, &p2p_keys, ports);
    let params = ThresholdParameters {
        threshold: Threshold(threshold as u64),
        participants,
    };

    tracing::info!(
        threshold,
        num_participants = participant_indices.len(),
        ?init_format,
        "initializing contract"
    );
    let init_config = json!({ "key_event_timeout_blocks": KEY_EVENT_TIMEOUT_BLOCKS });
    let parameters_json = init_format.init_parameters_json(&params)?;
    let outcome = contract
        .call(
            method_names::INIT,
            json!({ "parameters": parameters_json, "init_config": init_config }),
        )
        .await?;

        outcome.is_success(),
        "init failed: {:?}",
