Audit Report

## Title
`_calculatePayoutAmount` Applies One-Sided `min()` at Unlock Time, Delivering Less Than the Amount Committed at Initiation — (`contracts/LRTWithdrawalManager.sol`)

## Summary

When a user calls `initiateWithdrawal`, their rsETH is immediately locked and `expectedAssetAmount` is computed from the current oracle price and stored in the `WithdrawalRequest`. When an operator later calls `unlockQueue`, `_calculatePayoutAmount` recomputes the return using the unlock-time oracle price and applies `min(expectedAtInitiation, currentReturn)`. If the rsETH price has fallen, the user receives strictly less than the stored `expectedAssetAmount` while their full rsETH is burned. The protocol retains the difference in the unstaking vault.

## Finding Description

**Initiation — price-locked commitment stored:**

`initiateWithdrawal` transfers the user's rsETH to the contract and stores `expectedAssetAmount` computed from the live oracle price at that moment:

```solidity
// L166
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
// L168-175
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
assetsCommitted[asset] += expectedAssetAmount;
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
```

`getExpectedAssetAmount` uses the live oracle price at call time:

```solidity
// L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

The user cannot cancel; their rsETH is held by the contract from this point.

**Unlock — asymmetric `min()` applied:**

When `unlockQueue` is called (operator-only, but no user recourse), `_unlockWithdrawalRequests` calls `_calculatePayoutAmount` with fresh oracle prices:

```solidity
// L833-834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

- Price rose → payout capped at `expectedAssetAmount`; protocol keeps the upside.
- Price fell → payout is `currentReturn < expectedAssetAmount`; user bears the full downside.

The overwritten `request.expectedAssetAmount` (now the lower value) is what gets transferred:

```solidity
// L802-804
assetsCommitted[asset] -= request.expectedAssetAmount;  // decremented by original higher value
request.expectedAssetAmount = payoutAmount;              // overwritten to lower value
```

```solidity
// L734
_transferAsset(asset, user, request.expectedAssetAmount);
```

The full `rsETHUnstaked` is burned (L305, L805) regardless of whether the user received the originally committed amount. The gap `(expectedAtInitiation − payoutAmount)` remains in the unstaking vault.

**Why existing checks are insufficient:**

`_validatePrices` in `unlockQueue` only enforces operator-supplied price bounds — it does not guarantee the user receives `expectedAssetAmount`. The operator can legitimately call `unlockQueue` at any price within those bounds, and the user has no mechanism to cancel or delay their request.

## Impact Explanation

This matches the allowed Low impact: **"Contract fails to deliver promised returns, but doesn't lose value."** The `expectedAssetAmount` stored in `WithdrawalRequest` and visible via `getUserWithdrawalRequest` is the figure the user was shown at initiation. The contract commits `assetsCommitted[asset] += expectedAssetAmount` at initiation, signaling a reservation of that amount. Yet the user can receive strictly less if the rsETH oracle price declines before `unlockQueue` is called. The protocol does not lose value — the shortfall stays in the vault — but the user is made worse off relative to the stored commitment.

## Likelihood Explanation

Any user who calls `initiateWithdrawal` is exposed without any privileged access. The default withdrawal delay is `8 days / 12 seconds` blocks (L94), providing an 8-day window for rsETH price movement. rsETH is a yield-bearing LRT whose price is derived from underlying LST valuations and EigenLayer strategy returns; slashing events, LST depegs, or routine oracle updates can cause price decreases within this window. No attacker action is required — the scenario triggers passively whenever the oracle price falls between initiation and the operator's `unlockQueue` call.

## Recommendation

Two options:

1. **Lock prices at initiation**: Store `rsETHPrice` and `assetPrice` in `WithdrawalRequest` at initiation time. In `_calculatePayoutAmount`, use the stored prices instead of the unlock-time prices. This guarantees the user receives exactly `expectedAssetAmount` regardless of subsequent price movement (subject to available assets).

2. **Return excess rsETH on price drop**: If `currentReturn < expectedAssetAmount` at unlock time, compute the proportional rsETH difference and return it to the user rather than burning the full `rsETHUnstaked`. This makes the user whole in rsETH terms.

## Proof of Concept

```solidity
// Foundry fork test outline:
// 1. Fork mainnet at block N
// 2. User calls initiateWithdrawal(asset, rsETHUnstaked)
//    → expectedAssetAmount = rsETHUnstaked * P1 / assetPrice stored in request
//    → rsETH transferred to contract
// 3. Mock oracle to return P2 < P1 for rsETHPrice (simulate slashing/depeg)
// 4. Advance block.number by withdrawalDelayBlocks
// 5. Operator calls unlockQueue(asset, ..., minimumRsEthPrice=P2, maximumRsEthPrice=P2+ε, ...)
//    → _calculatePayoutAmount returns rsETHUnstaked * P2 / assetPrice < expectedAssetAmount
//    → request.expectedAssetAmount overwritten to lower value
//    → full rsETHUnstaked burned
// 6. User calls completeWithdrawal(asset)
//    → receives rsETHUnstaked * P2 / assetPrice
// 7. Assert: received < expectedAssetAmount stored at step 2
//    → shortfall = rsETHUnstaked * (P1 - P2) / assetPrice remains in vault
```

The invariant `payout >= expectedAssetAmount_at_initiation` is broken whenever `rsETHPrice` at unlock time is less than `rsETHPrice` at initiation time.
