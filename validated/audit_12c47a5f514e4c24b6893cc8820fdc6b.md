Audit Report

## Title
TOCTOU Race Condition in `bridgeKernelToBSC` Permanently Freezes User Funds — (`contracts/KERNEL/KernelVaultETH.sol`)

## Summary
`bridgeKernelToBSC` unconditionally sets `lastBridgedDepositId = counter - 1` using the on-chain `counter` at execution time, not the value the operator observed when computing `amount`. A user deposit that lands between the operator's off-chain read and the transaction's on-chain execution is silently marked as bridged while its KERNEL tokens remain permanently locked in the vault with no recovery path.

## Finding Description
The operator's off-chain workflow reads `counter = N`, computes `amount` as the sum of `userDeposits[lastBridgedDepositId+1 .. N-1]`, and submits `bridgeKernelToBSC(amount, ...)`. If any user calls `depositKernel` before the operator's transaction is mined, `counter` becomes `N+1` and `userDeposits[N]` is recorded on-chain.

When the operator's transaction executes:

- Line 262: `lastBridgedDepositId = counter - 1 = N` — the watermark advances to include deposit `N`.
- The `amount` bridged covers only deposits `0..N-1`; deposit `N`'s tokens remain in the vault.
- The only guard (lines 238–240) checks `kernel.balanceOf(address(this)) >= amount`, which passes because the vault holds deposit `N`'s tokens in addition to the bridged amount.

On the next operator cycle, `lastBridgedDepositId = N` and `counter = N+1`, so the off-chain system computes zero new deposits to bridge. There is no `withdraw`, `rescue`, or re-bridge function anywhere in the contract, so deposit `N`'s KERNEL tokens are permanently locked.

## Impact Explanation
The stranded tokens are the user's deposited KERNEL principal, not merely unclaimed yield. The contract has no withdrawal or rescue path for users, and no admin function to re-bridge a deposit whose ID already falls behind the watermark. This constitutes **permanent freezing of funds** (Critical).

## Likelihood Explanation
No attacker coordination is required. Any ordinary user deposit mined in the window between the operator's off-chain counter read and the operator's transaction execution triggers the bug. On Ethereum mainnet with ~12-second block times, this window routinely spans multiple blocks. The operator acts in good faith throughout; the condition is a structural property of the design, not an edge case.

## Recommendation
Add an `expectedLastDepositId` parameter and assert it matches `counter - 1` at execution time, causing the transaction to revert (rather than silently mismatch) if a new deposit lands before execution:

```solidity
function bridgeKernelToBSC(
    uint256 amount,
    uint256 minAmount,
    uint256 nativeFee,
    address refundAddress,
    uint256 expectedLastDepositId
) external payable nonReentrant onlyRole(OPERATOR_ROLE) {
    require(expectedLastDepositId == counter - 1, "counter mismatch");
    lastBridgedDepositId = expectedLastDepositId;
    // ... rest unchanged
}
```

This allows the operator to retry with the correct `amount` after any intervening deposit.

## Proof of Concept

```solidity
// 1. Alice deposits 100 KERNEL → depositId=0, counter=1
vault.depositKernel(100e18);  // Alice

// 2. Operator reads counter=1, computes amount=100e18, submits tx

// 3. Bob deposits 50 KERNEL → depositId=1, counter=2  (mined before operator tx)
vault.depositKernel(50e18);   // Bob

// 4. Operator's tx executes:
//    lastBridgedDepositId = counter - 1 = 1   (line 262)
//    bridges only 100e18 (Alice's amount)
//    Bob's 50e18 remains in vault
vault.bridgeKernelToBSC{value: fee}(100e18, 99e18, fee, refund);

// 5. Invariant broken:
assert(lastBridgedDepositId == 1);             // watermark claims Bob was bridged
assert(kernel.balanceOf(address(vault)) >= 50e18); // Bob's tokens still in vault

// 6. Next operator cycle: lastBridgedDepositId=1, counter=2 → 0 new deposits
//    Bob's 50e18 KERNEL is permanently locked; no rescue path exists
