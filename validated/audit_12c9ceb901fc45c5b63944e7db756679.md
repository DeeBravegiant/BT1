### Title
Stale Votes from Removed Participants Counted Toward TEE Governance Threshold in `CodeHashesVotes::count_votes` and `LauncherHashVotes::count_votes` — (`crates/contract/src/tee/proposal.rs`)

---

### Summary

`CodeHashesVotes::count_votes()` and `LauncherHashVotes::count_votes()` tally **all** stored votes for a proposal without filtering to only current participants. Because the post-resharing cleanup (`clean_tee_status`) is dispatched as an asynchronous promise, there is a window where stale votes from removed participants remain in the map and are counted toward the governance threshold. A single current participant can combine with those stale votes to whitelist a malicious Docker image hash or launcher hash, bypassing the required threshold of current-participant approval.

---

### Finding Description

`CodeHashesVotes::count_votes()` iterates over every entry in `proposal_by_account` and counts those matching the target hash, with no participant-membership filter:

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
``` [1](#0-0) 

`LauncherHashVotes::count_votes()` has the identical defect:

```rust
fn count_votes(&self, action: &LauncherVoteAction) -> u64 {
    u64::try_from(
        self.vote_by_account
            .values()
            .filter(|a| *a == action)
            .count(),
    )
    .expect("participant count should not overflow u64")
}
``` [2](#0-1) 

Both structs expose a `get_remaining_votes()` method that **does** filter by current participants, but this is only used during cleanup — never during the threshold check itself. [3](#0-2) 

The callers `vote_code_hash` and `vote_add_launcher_hash` authenticate the **new** voter against the current participant set, but then compare the unfiltered `count_votes()` result against the threshold:

```rust
let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
let votes = self.tee_state.vote(code_hash, &participant);
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
``` [4](#0-3) 

```rust
let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
let action = LauncherVoteAction::Add(launcher_hash);
let votes = self.tee_state.vote_launcher(action, &participant);
if votes >= self.threshold()?.value() {
    ...
}
``` [5](#0-4) 

The stale-vote cleanup is performed by `clean_tee_status`, which is documented as a **private endpoint callable only by the contract itself via a promise** — i.e., it is dispatched asynchronously after resharing completes, not atomically within the same transaction. [6](#0-5) 

The test `test_clean_non_participant_votes_removes_stale_votes` confirms that stale votes from removed participants persist until the cleanup promise executes: [7](#0-6) 

---

### Impact Explanation

Between resharing completion and the execution of the `clean_tee_status` promise, any current participant can call `vote_code_hash` or `vote_add_launcher_hash`. The returned vote count includes stale votes from removed participants. If those stale votes plus the new vote meet or exceed the threshold, a malicious Docker image hash or launcher hash is immediately whitelisted.

A whitelisted malicious image hash means nodes running that unauthorized code pass `is_caller_an_attested_participant` checks and are accepted as valid TEE participants: [8](#0-7) 

Such a node can then call `respond`, `respond_ckd`, and other participant-gated endpoints, participate in threshold signing, and potentially access or influence MPC key material — constituting unauthorized access to MPC key shares and signing capability.

---

### Likelihood Explanation

**Concrete attack scenario (N=5, T=3):**

1. Participants P1 and P2 vote for `malicious_hash` before a resharing event.
2. Resharing removes P1 and P2; new participant set is {P3, P4, P5}, threshold remains 3.
3. `clean_tee_status` is scheduled as a promise but has not yet executed.
4. P3 (a colluding new participant, or any participant tricked into voting) calls `vote_code_hash(malicious_hash)`.
5. `count_votes()` returns: P1 (stale) + P2 (stale) + P3 = 3 ≥ threshold.
6. `malicious_hash` is whitelisted with only **1 current participant** voting for it.

The window exists in every resharing. A Byzantine participant below the signing threshold who anticipates removal can pre-vote for a malicious hash, then coordinate with a single colluding new participant to trigger the threshold crossing during the cleanup gap. No privileged access or key leakage is required.

---

### Recommendation

Filter votes by current participants inside `count_votes()` (or at the call site in `vote_code_hash` / `vote_add_launcher_hash`) rather than relying on the asynchronous cleanup. Concretely, pass the current `Participants` set into `count_votes()` and apply the same membership filter already present in `get_remaining_votes()`:

```rust
fn count_votes(&self, proposal: &NodeImageHash, participants: &Participants) -> u64 {
    self.proposal_by_account
        .iter()
        .filter(|(p_id, prop)| {
            prop == proposal
                && participants.is_participant_given_participant_id(&p_id.get())
        })
        .count() as u64
}
```

Apply the same fix to `LauncherHashVotes::count_votes()`. This makes the threshold check consistent with the current participant set at the time of the vote, eliminating the stale-vote window regardless of when `clean_tee_status` executes.

---

### Proof of Concept

```
State: N=5, T=3. Participants = {P1, P2, P3, P4, P5}.

Step 1: P1 calls vote_code_hash(malicious_hash).
        proposal_by_account = {P1 → malicious_hash}
        count_votes() = 1 < 3. Not whitelisted.

Step 2: P2 calls vote_code_hash(malicious_hash).
        proposal_by_account = {P1 → malicious_hash, P2 → malicious_hash}
        count_votes() = 2 < 3. Not whitelisted.

Step 3: Resharing completes. New participants = {P3, P4, P5}.
        clean_tee_status promise is scheduled (not yet executed).
        proposal_by_account still = {P1 → malicious_hash, P2 → malicious_hash}

Step 4: P3 calls vote_code_hash(malicious_hash).
        AuthenticatedParticipantId::new({P3,P4,P5}) succeeds for P3.
        proposal_by_account = {P1 → malicious_hash, P2 → malicious_hash, P3 → malicious_hash}
        count_votes() = 3 >= threshold(3). ✓ WHITELISTED.

Result: malicious_hash is now an allowed Docker image hash with only 1 current
        participant (P3) having voted for it. Nodes running malicious_hash pass
        TEE attestation checks and gain full participant privileges.
```

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
