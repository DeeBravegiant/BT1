Audit Report

## Title
`RSETHPriceFeed` Returns ETH/USD Staleness Metadata for rsETH/USD Price and Corrupts Historical Round Data - (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed` implements `AggregatorV3Interface` as the rsETH/USD price feed for external consumption. In `latestRoundData()`, the `updatedAt` and `answeredInRound` fields are sourced from the ETH/USD Chainlink feed while `answer` is computed from `LRTOracle.rsETHPrice`, which has no on-chain update timestamp — meaning standard heartbeat staleness checks validate ETH/USD freshness, not rsETH oracle freshness. In `getRoundData(_roundId)`, a historical ETH/USD price is multiplied by the **current** `rsETHPrice`, producing a value that is never the actual rsETH/USD price at that historical round.

## Finding Description
In `latestRoundData()` (lines 63–70 of `contracts/oracles/RSETHPriceFeed.sol`):

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

All five return fields are populated from `ETH_TO_USD.latestRoundData()`, then only `answer` is overwritten. The `updatedAt` timestamp reflects when the ETH/USD Chainlink feed last updated, not when `LRTOracle.rsETHPrice` was last written. In `LRTOracle`, `rsETHPrice` is a bare storage variable (line 28) updated only inside `_updateRsETHPrice()` (line 313) — no companion timestamp is stored. `updateRSETHPrice()` (line 87) is public but not automatically triggered; it depends on off-chain keepers or manual calls. If the keeper is offline, `rsETHPrice` can be arbitrarily stale while `updatedAt` from the ETH/USD feed remains recent, causing any consumer performing `block.timestamp - updatedAt < heartbeat` to incorrectly conclude the rsETH price is fresh.

In `getRoundData(_roundId)` (lines 53–61):

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

This fetches the ETH/USD price at a historical round but multiplies it by the **current** `rsETHPrice`. The returned `answer` is therefore never the actual rsETH/USD price at that historical round — it is a synthetic value mixing a past ETH/USD price with a present rsETH/ETH rate. This is unconditionally incorrect for every historical round query.

## Impact Explanation
The contract is deployed as an `AggregatorV3Interface`-compliant rsETH/USD price feed and is explicitly designed for external protocol consumption. It fails to deliver its core promise on two axes: (1) `latestRoundData()` returns freshness metadata that does not reflect rsETH oracle staleness, and (2) `getRoundData()` returns a price that is structurally incorrect for any historical round. This maps to **Low — Contract fails to deliver promised returns, but doesn't lose value** within the LRT-rsETH protocol itself.

## Likelihood Explanation
The `getRoundData` historical price corruption is unconditional — it is always wrong for any `_roundId` input, regardless of oracle state. The `latestRoundData` staleness masking requires a keeper gap (off-chain keeper fails to call `updateRSETHPrice()` for an extended period), which is a realistic operational condition. Both issues are reachable by any unprivileged external caller via public view functions.

## Recommendation
1. **`latestRoundData()`**: Store a `rsETHPriceUpdatedAt` timestamp in `LRTOracle` that is written alongside `rsETHPrice = newRsETHPrice` (line 313). In `RSETHPriceFeed.latestRoundData()`, return `min(ethToUsdUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt` so consumers' staleness checks reflect the least-recently-updated component.
2. **`getRoundData()`**: Revert with a `NotSupported()` error, since accurate historical rsETH/USD prices cannot be reconstructed without a historical rsETH/ETH rate index. Alternatively, remove the function from the interface if historical round queries are not a supported use case.
3. Consider storing a monotonically increasing `rsETHRoundId` in `LRTOracle` that increments on each `updateRSETHPrice()` call and returning it as `answeredInRound` in `latestRoundData()`.

## Proof of Concept
**`getRoundData` — always incorrect (no preconditions required):**
1. Call `RSETHPriceFeed.getRoundData(R)` for any historical round `R`.
2. The returned `answer = int256(RS_ETH_ORACLE.rsETHPrice()) * ethUsdAtRoundR / 1e18` uses the **current** `rsETHPrice`, not the rsETH/ETH rate at round `R`.
3. The value is structurally wrong for every historical round.

**`latestRoundData` — staleness masking (requires keeper gap):**
1. At `T=0`, `LRTOracle.updateRSETHPrice()` is called; `rsETHPrice = 1.05e18`.
2. At `T+25h`, the ETH/USD Chainlink feed updates normally; `updatedAt = T+25h`. `rsETHPrice` has not been updated (keeper offline).
3. A consumer calls `RSETHPriceFeed.latestRoundData()`.
4. Returned: `updatedAt = T+25h` (ETH/USD), `answer` computed from a 25-hour-old `rsETHPrice`.
5. Consumer checks `block.timestamp - updatedAt ≈ 0 < heartbeat` → staleness check passes despite stale rsETH rate.

**Foundry test plan:**
- Deploy `LRTOracle` and `RSETHPriceFeed` against a mainnet fork.
- Record `rsETHPrice` at block `B`.
- `vm.warp(block.timestamp + 26 hours)` without calling `updateRSETHPrice()`.
- Call `latestRoundData()`; assert `updatedAt` is recent (from ETH/USD) while `answer` still encodes the stale `rsETHPrice` from block `B`.
- Call `getRoundData(historicalRoundId)`; assert returned `answer` differs from the true rsETH/USD price at that historical round.
