### Title
Bootstrap-Phase DRep Reverse-Delegation Stale Entry Allows Attacker to Silently Erase Delegators' Vote Weight - (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs`, `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version < 10), `processDelegationInternal` intentionally skips removing a stake credential from the old DRep's reverse-delegation set (`drepDelegs`) when the credential re-delegates to a new DRep. When the old DRep later unregisters, `ConwayUnRegDRep` unconditionally clears the forward delegation (`dRepDelegationAccountStateL`) of every credential still listed in its stale `drepDelegs`, including credentials that have already moved to a different DRep. This silently erases those credentials' vote delegation, removing their stake from the active DRep vote-weight distribution and potentially altering governance ratification outcomes.

---

### Finding Description

**Root cause — `processDelegationInternal`, `Deleg.hs` lines 347–377**

When `preserveIncorrectDelegation` is `True` (i.e., `pvMajor pv < natVersion @10`, the bootstrap phase), the `delegVote` branch only inserts `stakeCred` into the new DRep's `drepDelegs` set; it never removes `stakeCred` from the old DRep's `drepDelegs` set:

```haskell
| isNothing mAccountState || preserveIncorrectDelegation ->
    certVStateL . vsDRepsL
      %~ Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
``` [1](#0-0) 

The correct post-bootstrap path calls `unDelegReDelegDRep`, which removes `stakeCred` from the old DRep's set before inserting it into the new one:

```haskell
| Just accountState <- mAccountState ->
    certVStateL %~ unDelegReDelegDRep stakeCred accountState (Just dRep)
``` [2](#0-1) 

`unDelegReDelegDRep` in `VState.hs` explicitly deletes the credential from the old DRep's set:

```haskell
vsDRepsL %~ addNewDelegation . Map.adjust (drepDelegsL %~ Set.delete stakeCred) dRepCred
``` [3](#0-2) 

**Propagation — `ConwayUnRegDRep`, `GovCert.hs` lines 234–254**

When the old DRep unregisters, `clearDRepDelegations` unconditionally sets `dRepDelegationAccountStateL` to `Nothing` for every credential in `drepDelegs dRepState`, with no check of whether the credential's current forward delegation still points to this DRep:

```haskell
clearDRepDelegations delegs accountsMap =
  foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
``` [4](#0-3) 

Because `drepDelegs` still contains the re-delegated credential (stale entry from the bootstrap bug), this wipes out the credential's forward delegation to the new DRep.

**Effect on vote-weight computation — `computeDRepDistr`, `DRepPulser.hs` lines 228–241**

The DRep vote-weight distribution is built by iterating over `AccountState` entries and reading `dRepDelegationAccountStateL` (the forward delegation):

```haskell
addToDRepDistr accountState stakeAndDeposits distr = fromMaybe distr $ do
  dRep <- accountState ^. dRepDelegationAccountStateL
  ...
``` [5](#0-4) 

Once the forward delegation is cleared to `Nothing`, the credential's stake is excluded from every DRep's distribution. The codebase itself acknowledges the downstream uncertainty in a TODO comment at line 237–239:

```haskell
-- TODO: Potential optimization. Avoid this membership check, since delegation is
-- guaranteed to exist. I believe it would also be safe for PV9, but we need to verify
-- that it is in fact true due to #4772
``` [6](#0-5) 

The existing test in `DelegSpec.hs` confirms the forward delegation is cleared in bootstrap after the old DRep unregisters:

```haskell
expectNothingExpr (lookupDRepDelegation cred accounts)
``` [7](#0-6) 

---

### Impact Explanation

The `dRepAcceptedRatio` function computes the ratification ratio as `yes / (yes + no + not_voted + noconfidence)`, where the denominator is the total stake delegated to registered, non-expired DReps: [8](#0-7) 

By causing targeted stake credentials' forward delegations to be cleared, an attacker reduces the effective vote weight of a legitimate DRep and shrinks the denominator. This can flip a governance action from failing to passing. Affected action types include treasury withdrawals, protocol parameter changes, hard fork initiations, committee updates, and constitution changes — all of which are enacted on-chain without further recourse. This matches **Critical: Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.**

---

### Likelihood Explanation

The attack is fully executable by an unprivileged transaction sender during the bootstrap phase (Conway protocol version 9.x):

1. Attacker submits `ConwayRegDRep` to register as a DRep (deposit is returned on unregistration — net cost is zero).
2. Attacker's own stake credential, or any delegator who was attracted to the attacker's DRep, submits `ConwayDelegCert` with `DelegVote (DRepCredential attackerDRep)`.
3. That credential later re-delegates to a legitimate DRep via another `ConwayDelegCert` with `DelegVote (DRepCredential legitimateDRep)`. The stale entry in `attackerDRep.drepDelegs` is not removed.
4. Attacker submits `ConwayUnRegDRep`, recovering the deposit. `clearDRepDelegations` fires and sets the re-delegated credential's forward delegation to `Nothing`.
5. At the next epoch boundary, `computeDRepDistr` excludes that credential's stake from `legitimateDRep`'s distribution.
6. A governance action that `legitimateDRep` is blocking may now reach the ratification threshold.

The attacker can repeat this with multiple stake credentials to amplify the effect. The only prerequisite is that the bootstrap phase is active (protocol version < 10).

---

### Recommendation

1. **Immediate mitigation**: In `clearDRepDelegations` (`GovCert.hs`), before setting `dRepDelegationAccountStateL` to `Nothing`, verify that the credential's current forward delegation still points to the unregistering DRep. Only clear it if it matches.

2. **Correct fix (already applied post-bootstrap)**: Ensure `processDelegationInternal` always calls `unDelegReDelegDRep` when re-delegating an existing account, removing the stale entry from the old DRep's `drepDelegs` at delegation time rather than relying on cleanup at unregistration time. The `preserveIncorrectDelegation` flag should be removed or set to `False` as soon as the bootstrap phase ends.

---

### Proof of Concept

```
Epoch N (bootstrap phase, pvMajor = 9):

1. Attacker registers DRep_A.
   State: DRep_A.drepDelegs = {}

2. StakeCred_X delegates to DRep_A.
   State: DRep_A.drepDelegs = {StakeCred_X}
          StakeCred_X.dRepDelegation = DRep_A

3. StakeCred_X re-delegates to DRep_B (legitimate DRep voting No on a proposal).
   processDelegationInternal: preserveIncorrectDelegation=True
     → DRep_B.drepDelegs += StakeCred_X   (correct)
     → DRep_A.drepDelegs unchanged         (BUG: stale entry retained)
   State: DRep_A.drepDelegs = {StakeCred_X}  ← stale
          DRep_B.drepDelegs = {StakeCred_X}
          StakeCred_X.dRepDelegation = DRep_B

4. Attacker unregisters DRep_A (recovers deposit).
   ConwayUnRegDRep → clearDRepDelegations(DRep_A.drepDelegs={StakeCred_X})
     → StakeCred_X.dRepDelegation := Nothing  ← forward delegation wiped
   State: StakeCred_X.dRepDelegation = Nothing
          DRep_B.drepDelegs = {StakeCred_X}   ← reverse entry orphaned

Epoch N+1 boundary:
   computeDRepDistr iterates accounts:
     StakeCred_X: dRepDelegationAccountStateL = Nothing → stake NOT added to any DRep
   DRep_B's vote weight is reduced by StakeCred_X's stake.
   Governance proposal that DRep_B was blocking may now pass.
```

```haskell
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
```
