### Title
Attached NEAR Deposit Permanently Stuck When `deploy_token` Proof Verification Fails — (`near/omni-bridge/src/lib.rs`)

### Summary

`deploy_token` is a public `#[payable]` function that accepts a NEAR deposit for token-deployment storage costs. When the upstream `verify_proof` cross-contract call fails, the callback panics and discards the deposit inside the contract with no refund path. The sibling function `bind_token` demonstrates the correct pattern with an explicit `bind_token_refund` step, confirming the omission in `deploy_token` is unintentional.

### Finding Description

`deploy_token` schedules two promises: `verify_proof` followed by `deploy_token_callback` with `NO_DEPOSIT` attached: [1](#0-0) 

```rust
pub fn deploy_token(&mut self, #[serializer(borsh)] args: DeployTokenArgs) -> Promise {
    self.verify_proof(args.chain_kind, args.prover_args).then(
        Self::ext(env::current_account_id())
            .with_attached_deposit(NO_DEPOSIT)          // ← deposit NOT forwarded
            .with_static_gas(DEPLOY_TOKEN_CALLBACK_GAS)
            .deploy_token_callback(near_sdk::env::attached_deposit()),
    )
}
```

The original `env::attached_deposit()` is transferred to the contract at call time. Because `NO_DEPOSIT` is attached to the callback, the deposit sits in the contract's balance. When `verify_proof` returns an error, `deploy_token_callback` immediately panics: [2](#0-1) 

```rust
pub fn deploy_token_callback(
    &mut self,
    attached_deposit: NearToken,
    #[callback_result] call_result: Result<ProverResult, PromiseError>,
) -> Promise {
    let Ok(ProverResult::LogMetadata(metadata)) = call_result else {
        env::panic_str(BridgeError::InvalidProofMessage.to_string().as_str()); // ← panics, deposit lost
    };
    ...
}
```

In NEAR's execution model, a callback panic reverts only that callback's state changes; it does **not** automatically return the deposit that was transferred to the contract during the original call. The deposit is permanently locked in the contract.

By contrast, `bind_token` — which follows the identical `verify_proof → callback` pattern — appends a third promise that explicitly refunds the excess deposit to the caller: [3](#0-2) 

```rust
pub fn bind_token(&mut self, #[serializer(borsh)] args: BindTokenArgs) -> Promise {
    self.verify_proof(args.chain_kind, args.prover_args)
        .then(
            Self::ext(env::current_account_id())
                .with_attached_deposit(NO_DEPOSIT)
                .with_static_gas(BIND_TOKEN_CALLBACK_GAS)
                .bind_token_callback(near_sdk::env::attached_deposit()),
        )
        .then(
            Self::ext(env::current_account_id())
                .with_attached_deposit(env::attached_deposit())   // ← deposit forwarded for refund
                .with_static_gas(BIND_TOKEN_REFUND_GAS)
                .bind_token_refund(near_sdk::env::predecessor_account_id()),
        )
}
``` [4](#0-3) 

`deploy_token` is missing this third promise entirely.

### Impact Explanation

Any caller of `deploy_token` who attaches a NEAR deposit (required for on-chain storage of the new token account) permanently loses that deposit if the prover returns an error — whether due to an invalid proof, a prover contract bug, gas exhaustion, or any other failure. There is no recovery function in the contract for stuck NEAR balances. This constitutes an irreversible, permanently unclaimable loss of user value in the bridge token-deployment flow.

### Likelihood Explanation

`deploy_token` is a public, unguarded (no role restriction) `#[payable]` function callable by any account when the contract is not paused. Proof verification failures are a normal operational occurrence (e.g., submitting a proof before the prover has indexed the block, network timeouts, or prover upgrades). Any such failure silently destroys the caller's deposit. The developers already identified this risk for `bind_token` and added the refund step there, making the omission in `deploy_token` a realistic, reachable loss path.

### Recommendation

Mirror the `bind_token` pattern: append a `bind_token_refund`-style promise to `deploy_token` that forwards the original deposit and returns any unused amount to the caller on both success and failure:

```rust
pub fn deploy_token(&mut self, #[serializer(borsh)] args: DeployTokenArgs) -> Promise {
    self.verify_proof(args.chain_kind, args.prover_args)
        .then(
            Self::ext(env::current_account_id())
                .with_attached_deposit(NO_DEPOSIT)
                .with_static_gas(DEPLOY_TOKEN_CALLBACK_GAS)
                .deploy_token_callback(near_sdk::env::attached_deposit()),
        )
        .then(
            Self::ext(env::current_account_id())
                .with_attached_deposit(env::attached_deposit())   // ← add this
                .with_static_gas(BIND_TOKEN_REFUND_GAS)
                .bind_token_refund(near_sdk::env::predecessor_account_id()),
        )
}
```

`bind_token_refund` already exists and handles both the success case (refunds the remainder after storage cost) and the failure case (refunds the full deposit): [4](#0-3) 

### Proof of Concept

1. User calls `deploy_token` attaching 1 NEAR for storage.
2. The prover contract returns `Err` (e.g., proof not yet indexed).
3. `deploy_token_callback` hits the `else` branch and calls `env::panic_str`.
4. NEAR runtime reverts the callback's state changes; the 1 NEAR deposit remains in the bridge contract's balance.
5. No function exists to return it to the user. The deposit is permanently lost.
