Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary

The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. If all stakers exit during an active reward window, the time elapsed with no stakers is silently consumed: `rewardPerTokenStored` does not increase (correct), but `updatedAt` advances, so when the next staker arrives the reward formula starts from the post-gap timestamp. Reward tokens accrued during the zero-staker interval are permanently locked in the contract with no recovery path.

## Finding Description

The `updateReward` modifier at L232–242 unconditionally sets `updatedAt = lastTimeRewardApplicable()`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // always advances
    ...
}
```

`rewardPerToken()` at L408–414 correctly short-circuits when `totalKernelStaked == 0`, returning `rewardPerTokenStored` unchanged. However, because `updatedAt` is still advanced unconditionally in the modifier, the elapsed time during the zero-staker gap is consumed.

The critical interaction occurs when a new staker calls `stake()`. The `updateReward` modifier executes **before** `totalKernelStaked` is incremented (L281–289). At that moment `totalKernelStaked` is still 0, so `rewardPerToken()` returns the unchanged `rewardPerTokenStored`, and then `updatedAt` is set to `lastTimeRewardApplicable()` (the current time). Only after the modifier completes does `totalKernelStaked += _amount` execute. The entire gap interval is thus skipped from reward accounting.

`initiateWithdrawal` at L320–338 is permissionless and decrements `totalKernelStaked` after the modifier runs, meaning any staker can reduce the total to zero. The `notifyRewardAmount` guard at L566–570 (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new reward period with zero stakers; it does not prevent all stakers from exiting mid-period.

The contract's own NatSpec at L18–22 explicitly acknowledges this behavior and relies on an off-chain operational guarantee rather than on-chain enforcement.

## Impact Explanation

Reward tokens transferred into the contract via `notifyRewardAmount` and accrued during a zero-staker interval are permanently unclaimable. There is no admin recovery function, no sweep mechanism, and no way to re-allocate the locked tokens. This constitutes **permanent freezing of unclaimed yield** — a Medium severity impact per the allowed scope.

## Likelihood Explanation

`initiateWithdrawal` is callable by any staker at any time without restriction. A full exit of all stakers during an active reward window is achievable through organic behavior (e.g., a market event, a better yield opportunity, or a coordinated exit). The contract's own documentation acknowledges this as a known operational risk, confirming the scenario is realistic and not purely theoretical.

## Recommendation

In the `updateReward` modifier, only advance `updatedAt` when `totalKernelStaked > 0`, mirroring the guard already present in `rewardPerToken()`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures time elapsed with no stakers does not consume the reward budget, preserving those rewards for future stakers.

## Proof of Concept

1. Admin calls `notifyRewardAmount(1_000e18)` with one staker present (`totalKernelStaked > 0`). `finishAt = block.timestamp + duration`, `rewardRate = 1_000e18 / duration`.
2. The sole staker calls `initiateWithdrawal(totalStake)`. `updateReward` fires: `rewardPerToken()` accumulates correctly up to this point, `updatedAt` is set to `block.timestamp` (T0). Then `totalKernelStaked` becomes 0.
3. 15 days pass (half the reward period). No on-chain interaction occurs; `updatedAt` remains at T0.
4. A new staker calls `stake(amount)`. `updateReward` fires: `totalKernelStaked == 0` at modifier execution time → `rewardPerToken()` returns `rewardPerTokenStored` unchanged. Then `updatedAt = lastTimeRewardApplicable()` = T0 + 15 days. Only after the modifier does `totalKernelStaked += amount`.
5. The new staker earns rewards only for the remaining ~15 days. The ~500e18 tokens corresponding to the 15-day gap are permanently locked in the contract.

**Foundry test plan**: Deploy `KernelDepositPool`, call `notifyRewardAmount`, have the single staker call `initiateWithdrawal` for full balance, `warp` forward by `duration / 2`, have a new staker call `stake`, `warp` to `finishAt`, call `getReward`, and assert that `rewardsToken.balanceOf(contract)` ≈ 500e18 (unreclaimable) while the new staker received only ≈ 500e18 instead of the full 1_000e18.
