Audit Report

## Title
Permanent Freezing of ETH Withdrawal Funds for Non-ETH-Receiving Contract Users - (File: contracts/LRTWithdrawalManager.sol)

## Summary
When a contract address that cannot receive ETH initiates an ETH withdrawal, the two-phase withdrawal lifecycle creates an irrecoverable state: rsETH is irreversibly burned during `unlockQueue` (Phase 2), but the subsequent `completeWithdrawal` call permanently reverts because the ETH transfer to the non-receiving contract always fails. No cancellation, redirection, or admin recovery path exists, permanently freezing the corresponding ETH in `LRTWithdrawalManager`.

## Finding Description
The withdrawal lifecycle proceeds as follows:

**Phase 1 — `initiateWithdrawal`:** The user's rsETH is transferred into the contract and a withdrawal request is recorded, with `msg.sender` stored as the beneficiary via `userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_)`. [1](#0-0) 

**Phase 2 — `unlockQueue` (operator-only):** rsETH held by the contract is burned and the corresponding ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. This burn is irreversible. [2](#0-1) 

**Phase 3 — `completeWithdrawal` / `completeWithdrawalForUser`:** Both functions call `_processWithdrawalCompletion(asset, user, ...)`, which ultimately calls `_transferAsset(asset, user, request.expectedAssetAmount)`. [3](#0-2) 

`_transferAsset` for ETH performs a low-level call: [4](#0-3) 

If `payable(to).call{ value: amount }("")` fails (because `to` is a contract without a `receive()` function), the entire `_processWithdrawalCompletion` reverts, rolling back the `popFront()`, `delete withdrawalRequests[requestId]`, and `unlockedWithdrawalsCount[asset]--`. The withdrawal record is restored, but the rsETH burned in Phase 2 is gone.

Both `completeWithdrawal` (sends to `msg.sender`) and `completeWithdrawalForUser` (sends to the original `user` address) route ETH to the same non-receiving address — operator intervention cannot unblock the withdrawal. The protocol's own comment at L191 acknowledges ETH transfer issues for `completeWithdrawalForUser` but provides no recovery mechanism. [5](#0-4) 

`sweepRemainingAssets` is explicitly blocked while any unlocked withdrawal exists: [6](#0-5) 

There is no `cancelWithdrawal` function, no alternative-recipient mechanism, and no admin override for stuck ETH. The ETH is permanently frozen.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is burned (irreversible) during `unlockQueue`. The corresponding ETH is allocated to the user's withdrawal in `LRTWithdrawalManager` but can never be claimed. There is no cancellation, redirection, or admin recovery path. The ETH is permanently frozen in the contract, and `sweepRemainingAssets` cannot be used to recover it while the unlocked withdrawal record persists.

## Likelihood Explanation
**Low-Medium.** Smart contracts interacting with DeFi protocols are common (smart wallets, multisigs, protocol-owned treasuries, aggregators, yield strategies). Many such contracts do not implement a `receive()` function or intentionally revert on unexpected ETH receipt. Any such contract that calls `initiateWithdrawal` for ETH will have its funds permanently frozen once the operator runs `unlockQueue`. The scenario requires no attacker — it is triggered by normal user behavior from a contract account.

## Recommendation
1. Add a `cancelWithdrawal` function allowing users to reclaim their rsETH before `unlockQueue` is called (i.e., while `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]`).
2. After `unlockQueue`, allow users or operators to specify an alternative recipient address for ETH withdrawals, decoupling the beneficiary from the original `msg.sender`.
3. Alternatively, if the ETH transfer fails in `_transferAsset`, wrap the ETH as WETH and send WETH to the user instead, ensuring the transfer always succeeds.

## Proof of Concept
1. Deploy `VictimContract` (no `receive()` function). From `VictimContract`, call `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; withdrawal request is recorded with `VictimContract` as beneficiary.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH held by the manager is burned; ETH is moved from `LRTUnstakingVault` to `LRTWithdrawalManager`. `unlockedWithdrawalsCount[ETH_TOKEN]` increments to 1.
3. `VictimContract` calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion`, `payable(VictimContract).call{ value: amount }("")` returns `(false, ...)`. The function reverts with `EthTransferFailed()`. All state changes in `_processWithdrawalCompletion` are rolled back.
4. Operator calls `completeWithdrawalForUser(ETH_TOKEN, VictimContract, "")`. Same result — ETH is sent to `VictimContract`, which still cannot receive it. Reverts again.
5. rsETH is burned (gone). ETH is stuck in `LRTWithdrawalManager`. `hasUnlockedWithdrawals(ETH_TOKEN)` returns `true`, blocking `sweepRemainingAssets`. Funds are permanently frozen.

**Foundry test sketch:**
```solidity
contract NoReceive {}

function test_permanentFreeze() public {
    NoReceive victim = new NoReceive();
    // Fund victim with rsETH, approve manager
    vm.startPrank(address(victim));
    rsETH.approve(address(manager), amount);
    manager.initiateWithdrawal(ETH_TOKEN, amount, "");
    vm.stopPrank();

    // Operator unlocks queue — rsETH burned, ETH moved to manager
    vm.prank(operator);
    manager.unlockQueue(ETH_TOKEN, type(uint256).max, minRsETH, minAsset, maxRsETH, maxAsset);

    // completeWithdrawal always reverts
    vm.prank(address(victim));
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);
    manager.completeWithdrawal(ETH_TOKEN, "");

    // sweepRemainingAssets blocked
    vm.prank(manager_role);
    vm.expectRevert(LRTWithdrawalManager.PendingWithdrawalsExist.selector);
    manager.sweepRemainingAssets(ETH_TOKEN);

    // ETH permanently frozen
    assertGt(address(manager).balance, 0);
}
```
