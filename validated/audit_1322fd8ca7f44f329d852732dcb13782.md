### Title
Unauthenticated P2P Peer Can Inject Arbitrary Transaction Hashes Into Node Storage via Unverified `n_transactions` and Missing Hash Validation — (`crates/apollo_p2p_sync/src/client/transaction.rs`, `crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

The p2p sync client accepts `SignedBlockHeader` messages from unauthenticated peers with no cryptographic signature verification and no validation of `transaction_commitment`. The stored `n_transactions` field from that header is then used as the exact loop bound for collecting peer-supplied `FullTransaction` messages, whose `transaction_hash` fields are explicitly acknowledged as unvalidated. A malicious peer can therefore anchor a node to an entirely fabricated transaction set for any synced block.

---

### Finding Description

**Step 1 — Header acceptance with no cryptographic checks.**

`HeaderStreamBuilder::parse_data_for_block` performs exactly two checks before accepting and storing a `SignedBlockHeader`: [1](#0-0) 

- `block_number` matches the expected value.
- `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (== 1).

There is no verification of:
- The block hash itself.
- The cryptographic validity of the signature against any known validator key.
- The `transaction_commitment` field.
- The `n_transactions` field against any external source of truth.

The header — including its peer-supplied `n_transactions` — is written directly to storage. [2](#0-1) 

**Step 2 — Transaction count read from the corrupted header.**

`TransactionStreamFactory::parse_data_for_block` reads `target_transaction_len` directly from the stored header: [3](#0-2) 

This value is entirely peer-controlled.

**Step 3 — Transaction hashes accepted without validation.**

The loop collects exactly `target_transaction_len` `FullTransaction` messages from the peer and pushes each peer-supplied `transaction_hash` into `block_body.transaction_hashes` with an explicit developer TODO acknowledging the missing validation: [4](#0-3) 

The `transaction_commitment` stored in the header is never used to verify the received hashes. No cross-check against the canonical chain occurs anywhere in this path.

**Step 4 — Corrupted body written to storage.**

The fabricated `BlockBody` (with peer-chosen hashes, transactions, and outputs) is committed unconditionally: [5](#0-4) 

---

### Impact Explanation

After the attack, the node's storage contains a `BlockBody` for the targeted block height whose `transaction_hashes` are entirely attacker-controlled. Every RPC call that reads transaction data for that block — `starknet_getBlockWithTxs`, `starknet_getTransactionByHash`, `starknet_getTransactionReceipt`, etc. — will return fabricated data. The `body_marker` is advanced past the corrupted block, so the node considers the block fully synced. This maps to **High** impact under the allowed scope: *RPC execution, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

---

### Likelihood Explanation

Any node running p2p sync is reachable by any peer on the network. No special privilege is required. The attacker only needs to:
1. Be selected as a sync peer (standard p2p behavior).
2. Send a `SignedBlockHeader` with one signature (any bytes) and the desired `n_transactions`.
3. Send exactly `n_transactions` `FullTransaction` messages with chosen `transaction_hash` fields.

The peer is never reported as bad because it satisfies all checked invariants (correct block number, correct signature count, correct transaction count).

---

### Recommendation

1. **Verify the block hash and signature cryptographically** in `HeaderStreamBuilder::parse_data_for_block` against a known validator key or a trusted finality source before storing the header.
2. **Validate transaction hashes against `transaction_commitment`** in `TransactionStreamFactory::parse_data_for_block` — resolve the TODO at line 88 of `transaction.rs`.
3. **Validate `n_transactions`** against `concatenated_counts` in the header commitments (a parallel TODO already exists in `convert_sync_block_to_block_data` at line 141 of `header.rs`). [6](#0-5) 

---

### Proof of Concept

A deterministic protocol-state test would:

1. Instantiate a `P2pSyncClient` with a mock network peer.
2. Have the peer respond to the header query with a `SignedBlockHeader` for block 0 containing `n_transactions = 3`, one arbitrary signature, and a fabricated `transaction_commitment`.
3. Have the peer respond to the transaction query with 3 `FullTransaction` messages each carrying an attacker-chosen `transaction_hash` (e.g., `H(1)`, `H(2)`, `H(3)`).
4. After sync completes, call `storage_reader.begin_ro_txn()?.get_block_transaction_hashes(BlockNumber(0))` and assert it returns `[H(1), H(2), H(3)]` — hashes that do not correspond to any real transaction on the canonical chain.

The test would pass, confirming the injected hashes are durably stored and served by the node.
