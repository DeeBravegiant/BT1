### Title
Unconditional Cache Population in `request_and_validate_additions` Bypasses Merkle Proof Validation on Reconnect - (File: chia/wallet/util/wallet_sync_utils.py)

### Summary

`request_and_validate_additions` in `chia/wallet/util/wallet_sync_utils.py` unconditionally populates the `PeerRequestCache` additions cache regardless of whether Merkle proof validation succeeded or failed. A malicious peer can poison this cache by sending a crafted `RespondAdditions` with an invalid proof, causing the cache entry to be marked as validated. On a subsequent connection (if the cache persists across peer connections), the wallet bypasses all proof validation for the same `(header_hash, puzzle_hash)` pair and accepts fraudulent coin states.

### Finding Description

In `request_and_validate_additions`, the call to `peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)` at line 223 is unconditional — it executes regardless of whether `validate_additions` returned `True` or `False`:

```python
result: bool = validate_additions(
    additions_res.coins,
    additions_res.proofs,
    additions_root,
)
peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)  # always runs
return result
```

On any subsequent call with the same `(header_hash, puzzle_hash)`, the early-exit path at line 205 fires unconditionally:

```python
if peer_request_cache.in_additions_in_block(header_hash, puzzle_hash):
    return True  # no proof validation performed
```

This is structurally analogous to the EigenLayer bug: a guard that is supposed to enforce proof validity can be bypassed because the "validated" marker is set independently of the validation outcome.

### Impact Explanation

A malicious full node peer can:
1. Send a `CoinState` claiming a fake coin was created at block H with puzzle hash PH.
2. Respond to the wallet's `RequestAdditions` with an invalid Merkle proof (e.g., empty bytes or a forged proof that fails `confirm_included_already_hashed`).
3. `validate_additions` returns `False`; the peer is closed — but `add_to_additions_in_block(header_hash, PH, H)` has already been called.
4. The attacker reconnects (or the wallet connects to another peer controlled by the attacker).
5. The same `CoinState` is sent again. `in_additions_in_block(header_hash, PH)` returns `True` → `request_and_validate_additions` returns `True` without any proof check.
6. `validate_received_state_from_peer` accepts the fraudulent coin state as valid.

The wallet's coin store is updated to reflect a coin that was never actually added to the blockchain, corrupting wallet sync state and potentially deceiving automated payment processors into releasing goods or services.

### Likelihood Explanation

The attack requires an unprivileged attacker to operate a malicious full node that the victim wallet connects to (standard Chia peer-to-peer network). No keys, admin access, or cryptographic breaks are needed. The attacker only needs to send two crafted protocol messages across two connections. The exploitability depends on whether `PeerRequestCache` persists across peer connections within a wallet session — if it does (which the shared-cache design suggests), the attack is directly reachable.

### Recommendation

Move `add_to_additions_in_block` inside a conditional that only caches on successful validation:

```python
result: bool = validate_additions(
    additions_res.coins,
    additions_res.proofs,
    additions_root,
)
if result:
    peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)
return result
```

### Proof of Concept

The root cause is visible at: [1](#0-0) 

`validate_additions` is called and its result stored, but `add_to_additions_in_block` is called unconditionally on line 223 before returning `result`. [2](#0-1) 

The early-return cache hit at line 205-206 then returns `True` on any subsequent call for the same `(header_hash, puzzle_hash)`, bypassing the entire proof validation path. [3](#0-2) 

The caller `validate_received_state_from_peer` treats a `True` result from `request_and_validate_additions` as proof that the coin was legitimately included in the block, and proceeds to accept the coin state. [4](#0-3)
