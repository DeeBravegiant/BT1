Audit Report

## Title
Unbacked `mint()` Tokens Allow Drainage of Depositor Collateral via `withdraw()` — (`contracts/agETH/AGETHTokenWrapper.sol`)

## Summary

`AGETHTokenWrapper` issues fungible wrapper tokens via two paths: `deposit()` (backed 1:1 by altAgETH held in the contract) and `mint()` (no collateral transferred, intended for bridge/L2 use). Because `_withdraw()` burns any wrapper token and transfers altAgETH from the contract balance without distinguishing token origin or enforcing a backing invariant, bridge-minted tokens can be redeemed against altAgETH deposited by regular users. This constitutes direct theft of depositor funds at rest.

## Finding Description

`mint()` is gated by `MINTER_ROLE` and is the documented mechanism for a bridge to represent bridged supply on L2: [1](#0-0) 

It calls `_mint()` directly with no altAgETH transferred into the contract. `_withdraw()` burns wrapper tokens from `msg.sender` and transfers altAgETH from the contract's balance with no check on how the burned tokens were originally created and no invariant enforcing `balanceOf(contract) >= totalSupply()`: [2](#0-1) 

The backing mechanism `depositBridgerAssets()` is a separate, role-gated call that is explicitly described as a "Legacy function" and is not atomically coupled to `mint()`: [3](#0-2) 

`maxAmountToDepositBridgerAsset()` confirms the design acknowledges that minted supply can exceed backing (`agETHSupply - balanceOfAssetInWrapper`), but nothing prevents withdrawal during this undercollateralized window: [4](#0-3) 

Because wrapper tokens from both paths are fungible ERC-20 tokens, any holder of bridge-minted tokens can call `withdraw()` and receive altAgETH that was deposited by a regular user. The bridge minting tokens is the **intended, documented behavior** — not a malicious deviation — so the vulnerability is structural in the protocol design, not dependent on a rogue privileged actor.

## Impact Explanation

**Critical — Direct theft of user funds at rest.**

Concrete scenario:
1. User A calls `deposit(altAgETH, 100e18)` → contract holds 100e18 altAgETH, User A holds 100e18 wrapper tokens.
2. Bridge (`MINTER_ROLE`) calls `mint(UserB, 100e18)` → contract still holds 100e18 altAgETH, total supply is now 200e18.
3. User B calls `withdraw(altAgETH, 100e18)` → burns 100e18 wrapper tokens, receives 100e18 altAgETH.
4. Contract holds 0 altAgETH. User A's 100e18 wrapper tokens are worthless — their deposited collateral is permanently gone.

This matches the allowed impact: **Critical — Direct theft of any user funds at rest.**

## Likelihood Explanation

**High.** Granting `MINTER_ROLE` to a bridge contract is the explicitly documented and intended deployment pattern per the contract's NatSpec. No admin compromise or malicious behavior is required — the bridge legitimately mints tokens as part of its normal operation, and any recipient can immediately call `withdraw()`. The window between `mint()` and `depositBridgerAssets()` is structurally always present. The `depositBridgerAssets()` function being labeled "Legacy" further suggests it may not be reliably called in current deployments.

## Recommendation

1. **Enforce a backing invariant in `_withdraw()`**: Before transferring altAgETH, assert `ERC20Upgradeable(_asset).balanceOf(address(this)) - _amount >= totalSupply() - _amount` (i.e., the contract remains fully backed after withdrawal).
2. **Atomic backing**: Require `depositBridgerAssets()` to be called atomically with or before `mint()`, or combine them into a single privileged function that transfers altAgETH and mints in one transaction.
3. **Separate accounting**: Track deposit-backed supply vs. bridge-minted supply. Only allow `withdraw()` to redeem against deposit-backed tokens; bridge-minted tokens should only be redeemable by burning back through the bridge.

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Foundry test — deploy AGETHTokenWrapper, grant MINTER_ROLE to bridge/attacker

function test_unbacked_mint_drains_depositor() public {
    // 1. User A deposits 100e18 altAgETH
    vm.prank(userA);
    altAgETH.approve(address(wrapper), 100e18);
    vm.prank(userA);
    wrapper.deposit(address(altAgETH), 100e18);

    assertEq(altAgETH.balanceOf(address(wrapper)), 100e18);
    assertEq(wrapper.totalSupply(), 100e18);

    // 2. Bridge (MINTER_ROLE) mints 100e18 wrapper tokens to attacker — no altAgETH deposited
    vm.prank(bridge); // bridge has MINTER_ROLE
    wrapper.mint(attacker, 100e18);

    assertEq(wrapper.totalSupply(), 200e18);
    assertEq(altAgETH.balanceOf(address(wrapper)), 100e18); // still only 100e18 backing

    // 3. Attacker withdraws using unbacked wrapper tokens
    vm.prank(attacker);
    wrapper.withdraw(address(altAgETH), 100e18);

    // Attacker stole User A's deposited altAgETH
    assertEq(altAgETH.balanceOf(attacker), 100e18);
    assertEq(altAgETH.balanceOf(address(wrapper)), 0);
    // User A's 100e18 wrapper tokens are now completely unbacked
}
```

    function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
        if (!allowedTokens[_asset]) return 0;

        // get totalSupply of wrapped agETH minted
        uint256 agETHSupply = totalSupply();
        // balance of _asset with the contract
        uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));

        if (balanceOfAssetInWrapper > agETHSupply) return 0;

        return agETHSupply - balanceOfAssetInWrapper;
    }
