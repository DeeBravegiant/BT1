Looking at the `respond_ckd` function in `crates/contract/src/lib.rs`, I can identify a direct analog to the external report's vulnerability class: **a missing validation check that allows a restricted/unauthorized action to proceed**.

---

### Title
Single Attested Participant Can Deliver Arbitrary CKD Output for `AppPublicKey` Requests, Bypassing Threshold Consensus — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function applies a cryptographic output check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of CKD requests. For the `AppPublicKey` variant, **no verification of the response is performed**. A single malicious attested participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary `CKDResponse`, delivering a fake derived key to the requesting user without any threshold consensus.

---

### Finding Description

The `respond_ckd` function at lines 653–689 of `crates/contract/src/lib.rs` handles two CKD request variants asymmetrically: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which cryptographically validates the response against the user's G1+G2 public key pair before resolving the yield. For `AppPublicKey`, the match arm is an empty block — the response is accepted unconditionally and immediately delivered to the user via `resolve_yields_for`. [2](#0-1) 

By contrast, the analogous `respond` function for threshold signatures always verifies the signature cryptographically before resolving: [3](#0-2) 

The authorization guard (`assert_caller_is_attested_participant_and_protocol_active`) only confirms the caller is an attested participant — it does **not** enforce that the submitted response reflects threshold consensus among t-of-n participants. [4](#0-3) 

The `assert_caller_is_attested_participant_and_protocol_active` function checks participant membership and TEE attestation, but a single participant satisfies this check: [5](#0-4) 

---

### Impact Explanation

A single malicious attested participant (1-of-n, strictly below the signing threshold) can:

1. Observe a pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Construct an arbitrary `CKDResponse` — specifically, a `big_c` value that encrypts a private key the attacker already knows, rather than the legitimately derived MPC key.
3. Call `respond_ckd` with this forged response before any honest node submits the real one. Because `resolve_yields_for` resolves the yield on first call and removes the entry, the honest response will subsequently fail with `RequestNotFound`.
4. The user's yield callback fires with the attacker-chosen key material. The user, believing they received a legitimate MPC-derived key, uses it to control assets on foreign chains.
5. The attacker, knowing the private key behind `big_c`, can spend those assets.

This is **confidential key derivation output delivered without the required threshold participant authorization**, matching the Critical impact tier: *"Unauthorized … confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

The attacker must be an attested participant — they must be voted into the participant set and run a TEE node that passes attestation. This is a non-trivial precondition. However, once a participant is malicious (compromised TEE operator, insider threat, or a participant whose TEE is bypassed), the exploit requires only a single on-chain transaction with no additional privileges. The window of opportunity is every `AppPublicKey` CKD request that arrives while the attacker is an active participant. Likelihood is **Medium** given the participant precondition, but the exploit itself is trivially executable once that precondition is met.

---

### Recommendation

Apply the same pattern used for `AppPublicKeyPV`: require a cryptographically verifiable proof in the `AppPublicKey` response, or redesign the `AppPublicKey` flow to require threshold-many identical responses before resolving the yield (on-chain quorum). At minimum, document that `AppPublicKey` CKD requests carry a weaker security guarantee than `AppPublicKeyPV` and consider deprecating the unverifiable variant.

---

### Proof of Concept

```
1. Contract is Running with participants [A, B, C], threshold = 2.
2. User calls request_app_private_key({ app_public_key: AppPublicKey(user_pk), ... })
   → pending_ckd_requests[req_key] = [yield_id]
3. Malicious participant A constructs:
     fake_big_c = Encrypt(attacker_private_key, user_pk)
     fake_response = CKDResponse { big_y: attacker_public_key, big_c: fake_big_c }
4. A calls respond_ckd(request, fake_response).
   → assert_caller_is_attested_participant_and_protocol_active() passes (A is attested).
   → AppPublicKey arm: empty, no check.
   → resolve_yields_for delivers fake_response to user's yield callback.
   → pending_ckd_requests[req_key] is removed.
5. Honest nodes B and C later call respond_ckd with the real response
   → RequestNotFound error; real response is silently dropped.
6. User receives attacker_public_key as their "MPC-derived" key.
7. User deposits assets to attacker_public_key on a foreign chain.
8. Attacker signs with attacker_private_key and steals the assets.
``` [1](#0-0)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
