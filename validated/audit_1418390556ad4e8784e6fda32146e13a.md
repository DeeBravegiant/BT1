**Key code paths:**

1. `verify_against_receipt_root` (`core/primitives/src/sharding.rs:981-986`): hashes only `(to_shard_id, receipts)` — `from_shard_id` is never part of the hash.

2. `set_state_header` (`chain/chain/src/state_sync/adapter.rs:475-503`): the Pigeonhole check only enforces uniqueness of `from_shard_id` values; it does NOT verify that each `RootProof(root, block_proof)` corresponds to a distinct shard position in the block's outgoing receipts Merkle tree.

3. `collect_receipts_from_response` (`chain/chain/src/chain.rs:4141-4147`): flat-maps all receipts from all proofs with no deduplication.

4. `set_state_finalize` (`chain/chain/src/chain_update.rs:487,540`): applies the collected receipts directly to state.

**The concrete attack:**

For a block with 2 included chunks (shards A and B), a malicious peer constructs:
- Proof 1: `from_shard_id=A`, `receipts=R_B`, `proof=proof_B`, `root_proof=(root_B, block_proof_B)`
- Proof 2: `from_shard_id=B`, `receipts=R_B`, `proof=proof_B`, `root_proof=(root_B, block_proof_B)`

All four checks pass:
- 4c: count == `chunks_included` (2 == 2) ✓
- 4d: `from_shard_id` values {A, B} are unique ✓
- 4e: `verify_path(root_B, proof_B, hash(R_B, T))` passes for both ✓
- 4f: `verify_path(block_outgoing_root, block_proof_B, root_B)` passes for both ✓

Result: `collect_receipts_from_response` returns `R_B ++ R_B` instead of `R_A ++ R_B`. `apply_chunk` applies the wrong receipt set → state root diverges from canonical chain.

**However**, the critical question under the scope rules is whether this is a malicious-peer-only scenario. The state sync response is peer-supplied state data, which is explicitly listed as a valid attack vector. The impact is not mere rejection/resource waste — the node *accepts* the forged data and produces a divergent state root.

### Title
Broken Pigeonhole Completeness in State Sync: Unbound `from_shard_id` Allows Malicious Peer to Cause Receipt Duplication and State Divergence — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` relies on a Pigeonhole argument to prove that all incoming receipts from all source shards are present. The argument is broken: `from_shard_id` is never cryptographically bound to the Merkle proof, and the `RootProof` entries are not required to be distinct. A malicious peer can supply N proofs with distinct `from_shard_id` values that all reference the same shard's outgoing receipts root, causing the syncing node to apply one shard's receipts multiple times while omitting another shard's receipts entirely.

### Finding Description

`ReceiptProof::verify_against_receipt_root` hashes only `(to_shard_id, receipts)`: [1](#0-0) 

`from_shard_id` is never part of any hash. In `set_state_header`, the only check involving `from_shard_id` is a uniqueness (Pigeonhole) check: [2](#0-1) 

The `RootProof(root, block_proof)` values at steps 4e/4f are peer-supplied and are never checked for distinctness. Two proofs can share the same `root` and `block_proof` (both pointing to shard B's outgoing receipts root) while carrying different `from_shard_id` labels (A and B). Both pass 4e and 4f independently.

`collect_receipts_from_response` performs no deduplication: [3](#0-2) 

`set_state_finalize` applies the collected receipts directly to state: [4](#0-3) 

### Impact Explanation

The syncing node applies `R_B ++ R_B` instead of `R_A ++ R_B`. The resulting state root differs from the canonical chain. The node's `incoming_receipts` DB entries are also corrupted with wrong `from_shard_id` metadata. The node cannot participate in the network after sync without re-syncing.

### Likelihood Explanation

Any node in the NEAR network can act as a peer and serve state sync responses. The forged response requires only public block data (outgoing receipts roots and Merkle paths are all derivable from public chain data). No validator or privileged role is required.

### Recommendation

Bind `from_shard_id` cryptographically to the Merkle proof. The `RootProof` should be selected by the verifier using `from_shard_id` to look up the correct shard index in the block's chunk list (as the honest server does at line 202–206 of `adapter.rs`), rather than accepting a peer-supplied `root_proof` at position `j`. Alternatively, verify that each `block_proof` proves a distinct leaf position in the block's outgoing receipts Merkle tree. [5](#0-4) 

### Proof of Concept

```
1. Obtain a valid block B with 2 included chunks (shards A, B).
2. Compute root_B, proof_B (Merkle proof for R_B in root_B), block_proof_B.
3. Construct ShardStateSyncResponseHeader with:
     incoming_receipts_proofs[0] = ReceiptProofResponse(B, [
       ReceiptProof(R_B, ShardProof { from_shard_id: A, to_shard_id: T, proof: proof_B }),
       ReceiptProof(R_B, ShardProof { from_shard_id: B, to_shard_id: T, proof: proof_B }),
     ])
     root_proofs[0] = [RootProof(root_B, block_proof_B), RootProof(root_B, block_proof_B)]
4. Call set_state_header(...) → returns Ok(()).
5. Call set_state_finalize(...) → applies R_B twice, omits R_A.
6. Assert resulting state_root != canonical state_root for shard T.
```
