Audit Report

## Title
Missing Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Severe rsETH Under-Minting for Non-18 Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256,address)` multiplies `amountAfterFee` (in the token's native decimals) directly against `tokenToETHRate` (1e18-scaled) without first normalizing to 18 decimals. For any supported token with fewer than 18 decimals, users receive a fraction of the rsETH they are owed proportional to `10^(18 - decimals)`, and the deposited tokens are stranded in the pool with no user-accessible redemption path.

## Finding Description
The token deposit path computes rsETH output at [1](#0-0) :

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

Both rates are 1e18-scaled. `amountAfterFee` is in the token's native decimals. For an 18-decimal token the units cancel correctly. For a 6-decimal token (e.g., USDC), `amountAfterFee` is `1e6` instead of `1e18`, so the result is `10^12×` smaller than correct.

Compare with the ETH path at [2](#0-1) , which explicitly multiplies by `1e18` to normalize ETH's 18-decimal amount against the rate — the token path has no equivalent normalization.

The `addSupportedToken` entry point at [3](#0-2)  contains no decimal guard, so any sub-18-decimal token can be listed. Once listed, every call to `deposit(token, amount, referralId)` at [4](#0-3)  silently mints a negligible rsETH amount.

The only reverse path, `swapAssetToPremintedRsETH`, is restricted to `OPERATOR_ROLE` at [5](#0-4) , so the depositor has no self-service way to recover stranded value. The companion view `viewSwapAssetToPremintedRsETH` at [6](#0-5)  has the symmetric bug, meaning even the operator-executed reverse swap would return only the tiny rsETH amount's worth of tokens, not the full deposit.

## Impact Explanation
Any user depositing a supported sub-18-decimal token receives a negligible rsETH amount (e.g., 317 wei instead of ~3.17e14 wei for 1 USDC). The deposited tokens are held by the pool but the user holds insufficient rsETH to reclaim them through any user-accessible path. This constitutes **temporary freezing of funds** (Medium), as operator intervention is theoretically possible but the user cannot act unilaterally.

## Likelihood Explanation
`addSupportedToken` is gated by `TIMELOCK_ROLE`, requiring a governance decision to list a sub-18-decimal token. This is not governance capture — it is a legitimate operational action the contract is explicitly designed to support (multi-asset pool). No technical barrier prevents listing USDC, USDT, or WBTC. Once listed, the impact is immediate and automatic for every depositor of that token; no attacker action is required beyond making a normal deposit.

## Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate formula:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the symmetric inverse normalization in `viewSwapAssetToPremintedRsETH`. Alternatively, enforce `decimals() == 18` inside `addSupportedToken` to prevent listing sub-18-decimal tokens entirely.

## Proof of Concept
1. Deploy a mock ERC-20 with 6 decimals and a mock oracle returning `3.33e14` (≈ USDC/ETH rate in 1e18 precision).
2. Call `addSupportedToken(mockUSDC, mockOracle)` as `TIMELOCK_ROLE`.
3. Approve and call `deposit(mockUSDC, 1e6, "")` (1 USDC) as a normal user.
4. Assert `wrsETH.balanceOf(user) == 317` (actual) vs. expected `~3.17e14`.
5. Confirm the pool holds `1e6` USDC units and the user has no self-service redemption path — `swapAssetToPremintedRsETH` reverts for non-`OPERATOR_ROLE` callers, and the 317 wei rsETH held by the user is insufficient to recover the deposit even via operator intervention using the bugged `viewSwapAssetToPremintedRsETH`.
