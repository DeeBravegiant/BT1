### Title
Decimal Normalization Floor Division Permanently Locks User Funds When Fee Causes `amount_without_fee` to Round to Zero in `sign_transfer` - (File: `near/omni-bridge/src/lib.rs`)

### Summary

In `sign_transfer`, the net transfer amount is computed as `normalize_amount(amount_without_fee(), decimals)` using floor division. When a token has a large decimal difference between NEAR and the destination chain (e.g., 24 vs 18 = divisor of 10^6), a user-set non-zero fee can cause `amount_without_fee()` to fall below the divisor, making `normalize_amount` return 0. The function then panics with `InvalidAmountToTransfer`. Because the fee can only be increased (never decreased) via `update_transfer_fee`, and no cancel/refund path exists for outbound pending transfers, the user's locked tokens become permanently irrecoverable.

### Finding Description

**Step 1 — User initiates transfer with a small fee.**

A user calls `ft_transfer_call` → `init_transfer`, which stores a `TransferMessage` in `pending_transfers` and locks the tokens in the contract. The only validation at this stage is `fee.fee < amount`: [1](#0-0) 

**Step 2 — `sign_transfer` computes normalized amount and panics.**

When a relayer calls `sign_transfer`, it computes:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(amount_to_transfer > 0, BridgeError::InvalidAmountToTransfer.as_ref());
``` [2](#0-1) 

`normalize_amount` uses floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

For a token with `origin_decimals = 24`, `decimals = 18` (common for NEAR-native tokens bridged to EVM), the divisor is `10^6`. If `amount = 10^6` and `fee = 1`, then `amount_without_fee() = 999_999`, and `normalize_amount(999_999) = 0`. The `require!` panics, reverting the transaction. The `TransferMessage` remains in `pending_transfers` unchanged.

**Step 3 — Fee cannot be reduced; no cancel path exists.**

`update_transfer_fee` enforces that the fee can only be **increased**, never decreased:

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [4](#0-3) 

Increasing the fee makes `amount_without_fee()` even smaller, worsening the situation. There is no `cancel_transfer` or refund function for outbound pending transfers. The only removal paths for a pending transfer are:
- `sign_transfer_callback` — only removes the message when `fee.is_zero()` [5](#0-4) 
- `claim_fee_callback` — requires a proof that the transfer was already finalized on the destination chain, which is impossible if `sign_transfer` always panics

The user's tokens are permanently locked.

### Impact Explanation

**Critical — Irreversible fund lock.** A user who initiates an outbound transfer with a non-zero fee where `(amount - fee) < 10^(origin_decimals - destination_decimals)` will have their tokens permanently locked in the bridge contract. There is no recovery path: the relayer cannot sign the transfer, the user cannot reduce the fee, and no cancel/refund function exists for outbound pending transfers.

### Likelihood Explanation

**Medium-High.** NEAR tokens commonly have 24 decimals while EVM tokens have 18, giving a divisor of `10^6`. Any user who transfers a small amount (e.g., `amount = 10^6` base units) with any non-zero fee triggers this. The `init_transfer` validation only checks `fee < amount`, not that `normalize_amount(amount - fee) > 0`, so the invalid state is accepted at deposit time and only discovered irrecoverably at signing time. A malicious actor could also grief a victim by front-running with a fee update (increasing the fee on a victim's pending transfer) to push `amount_without_fee()` below the normalization threshold.

### Recommendation

1. **At `init_transfer` time**, validate that `normalize_amount(amount_without_fee(), decimals) > 0` before accepting the deposit. This requires looking up the destination token's decimals at deposit time.
2. **Alternatively**, add a `cancel_transfer` function that allows the original sender to reclaim their tokens from a stuck pending outbound transfer, subject to appropriate access controls.
3. **For `update_transfer_fee`**, add a check that the new fee does not cause `normalize_amount(amount - new_fee) == 0`.

### Proof of Concept

```
Token: NEAR-native, origin_decimals=24, destination_decimals=18 (divisor = 10^6)

1. User calls ft_transfer_call with:
   - amount = 1_000_000 (10^6 base units)
   - fee    = 1

2. init_transfer accepts: fee(1) < amount(1_000_000) ✓
   TransferMessage stored in pending_transfers.

3. Relayer calls sign_transfer:
   amount_without_fee() = 1_000_000 - 1 = 999_999
   normalize_amount(999_999, {18, 24}) = 999_999 / 1_000_000 = 0
   require!(0 > 0) → PANIC: ERR_INVALID_AMOUNT_TO_TRANSFER

4. User tries update_transfer_fee to fee=0:
   require!(0 >= 1) → PANIC: ERR_INVALID_FEE  (fee can only increase)

5. No cancel_transfer exists.
   Tokens are permanently locked.
```
