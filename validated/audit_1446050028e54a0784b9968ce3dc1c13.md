I need to examine the oracle implementations to understand what `getRate()` actually returns — this is critical to validating whether the decimal normalization claim is correct.

The oracle implementation is the critical piece. `ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink answer to 1e18 precision, where the answer represents the price of **1 whole token** in ETH. For USDC (6 decimals), `tokenToETHRate = 5e14` means "1 whole USDC = 5e14 wei ETH." But `amountAfterFee` is in native 6-decimal units, not whole tokens. The formula `amountAfterFee * tokenToETHRate / rsETHToETHrate` is missing the `/ 10^tokenDecimals` and `* 1e18` scaling. Audit Report

## Title
Token Decimal Normalization Missing in L2 Pool Token Swap Calculation Causes Near-Zero rsETH Minting for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolNoWrapper.sol)

## Summary
`viewSwapRsETHAmountAndFee(uint256 amount, address token)` in both `RSETHPoolV3` and `RSETHPoolNoWrapper` computes rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate`, where `tokenToETHRate` is the oracle's 1e18-normalized price of **1 whole token** in ETH, but `amountAfterFee` is in the token's native decimals. For any token with fewer than 18 decimals (e.g., USDC=6, WBTC=8), the result is understated by `10^(18 - tokenDecimals)`, causing depositors to lose their full token deposit while receiving a negligible rsETH balance.

## Finding Description

**Root cause — oracle semantics vs. amount units mismatch:**

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink answer to 1e18 precision:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [1](#0-0) 

This returns "ETH wei per 1 whole token." For USDC (6 decimals), a Chainlink feed returning `50000` (8-decimal feed, 0.0005 ETH) yields `tokenToETHRate = 5e14`.

The token deposit path in both pool contracts:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) [3](#0-2) 

`amountAfterFee` is in native token units (e.g., `1000e6` for 1000 USDC). The formula treats it as if it were already in 18-decimal form, but it is not.

**Contrast with the ETH path**, which correctly scales:
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [4](#0-3) [5](#0-4) 

ETH `amountAfterFee` is in wei (18 decimals), so `* 1e18` correctly produces an 18-decimal rsETH amount. The token path omits the equivalent `* 1e18 / 10^tokenDecimals` normalization.

**Verification that 18-decimal tokens are unaffected:** `WETHOracle.getRate()` returns `1e18`, and WETH `amountAfterFee` is in wei (18 decimals). The formula `1e18_amount * 1e18 / rsETHToETHrate` produces the correct result — confirming the bug is specific to sub-18-decimal tokens. [6](#0-5) 

**Concrete arithmetic for USDC (6 decimals):**

- Deposit: 1,000 USDC → `amountAfterFee = 1_000e6`
- `tokenToETHRate = 5e14`, `rsETHToETHrate = 1.05e18`
- Actual: `1_000e6 * 5e14 / 1.05e18 ≈ 476_190` wei rsETH (≈ $0.000000000001)
- Correct: `(1_000e6 / 1e6) * 5e14 * 1e18 / 1.05e18 ≈ 4.76e17` wei rsETH (≈ $0.476)
- Error factor: `10^12`

The deposit function transfers tokens from the user before minting: [7](#0-6) [8](#0-7) 

The tokens are taken, the negligible rsETH is minted, and the deposited tokens accumulate in the pool to be bridged to L1 — permanently lost to the depositor.

**Symmetric error in reverse swap:** `viewSwapAssetToPremintedRsETH` in `RSETHPoolV3` has the inverse miscalculation:
```solidity
tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate;
``` [9](#0-8) 

For USDC this would return `~9.996e20` units instead of `1000e6`, causing the operator to over-transfer tokens from the pool.

## Impact Explanation

**Critical — Direct theft of any user funds.**

Any user depositing a non-18-decimal token (USDC, WBTC, etc.) through `deposit(address token, uint256 amount, string referralId)` on either pool contract loses their entire deposit. The full token amount is transferred from the user, but the minted rsETH is economically indistinguishable from zero. The deposited tokens are subsequently bridged to L1 and are permanently unrecoverable by the depositor. This matches the allowed impact: *"Direct theft of any user funds, whether at-rest or in-motion."*

## Likelihood Explanation

**High.** The `deposit(address token, ...)` function is publicly callable with no access restriction beyond `onlySupportedToken`. The protocol explicitly provides `addSupportedToken` to extend support to ERC-20 tokens, and the `ChainlinkOracleForRSETHPoolCollateral` wrapper is purpose-built for non-ETH collateral. USDC and WBTC are the most common non-18-decimal assets on L2s. No attacker coordination, front-running, or special conditions are required — every deposit of a non-18-decimal token triggers the loss automatically and repeatably. [10](#0-9) [11](#0-10) 

## Recommendation

Normalize `amountAfterFee` to 18 decimals before applying the oracle rate ratio. In `viewSwapRsETHAmountAndFee(uint256 amount, address token)` in both contracts:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

// Inside viewSwapRsETHAmountAndFee:
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * (10 ** (18 - tokenDecimals));
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the inverse fix to `viewSwapAssetToPremintedRsETH` in `RSETHPoolV3`:

```solidity
tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate / (10 ** (18 - tokenDecimals));
```
