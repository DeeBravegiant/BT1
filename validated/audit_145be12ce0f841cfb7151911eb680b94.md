Audit Report

## Title
`_unlockWithdrawalRequests` Loop `break` on Insufficient Liquidity Permanently Blocks Later Requests - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager::_unlockWithdrawalRequests` uses a `break` at line 800 when `availableAssetAmount < payoutAmount` for a queued request. Because `nextLockedNonce_` is only incremented on successful processing, the state variable `nextLockedNonce[asset]` is written back to the blocked nonce on every call, causing all subsequent requests — regardless of their individual payout sizes — to be permanently skipped until the blocking request can be fully covered.

## Finding Description
The loop at lines 790–814 iterates withdrawal requests in FIFO order. When `availableAssetAmount < payoutAmount` for request at nonce N (line 800), the loop `break`s without incrementing `nextLockedNonce_`. At line 815, `nextLockedNonce[asset]` is written back as N. Every future invocation of `_unlockWithdrawalRequests` restarts at nonce N and hits the same `break`, making nonces N+1, N+2, … unreachable. Requests at those nonces may have `payoutAmount` values well within the available liquidity, but they are never evaluated. The `_calculatePayoutAmount` function (lines 833–834) computes `min(expectedAssetAmount, rsETHUnstaked * rsETHPrice / assetPrice)`, so different users naturally produce different payout sizes, making it routine for a large request to precede smaller ones.

## Impact Explanation
Users whose withdrawal requests are queued behind a single large request cannot withdraw their assets until the protocol accumulates enough liquidity to cover the large request first. This is a concrete **temporary freezing of funds** (Medium) for an unbounded set of users, matching the allowed impact scope.

## Likelihood Explanation
Any rsETH holder can queue a withdrawal via the public withdrawal initiation path. A user who queues a large withdrawal early (low nonce) will block all later, smaller requests whenever available liquidity is below their `payoutAmount`. No privilege is required. The condition arises naturally during high withdrawal demand or after a large EigenLayer unstaking cycle, making it realistically achievable without any coordination.

## Recommendation
Replace the `break` at line 800 with logic that advances `nextLockedNonce_` and skips the underfunded request rather than halting the entire loop. The simplest structural fix is to always increment the nonce and use `continue` for the insufficient-liquidity case, or maintain a separate pending set for skipped requests to allow revisiting them when liquidity recovers, depending on the protocol's fairness requirements.

## Proof of Concept
**Preconditions:**
- `withdrawalDelayBlocks` has elapsed for all requests.
- `availableAssetAmount = 10 ETH`
- Nonce 5: `payoutAmount = 50 ETH`
- Nonce 6: `payoutAmount = 1 ETH`
- `nextLockedNonce[ETH] = 5`, `firstExcludedIndex = 7`

**Execution:**
1. Loop starts: `nextLockedNonce_ = 5`. Delay check passes.
2. `payoutAmount = 50 ETH`. `10 < 50` → `break` (line 800). `nextLockedNonce_` remains 5.
3. Line 815 writes `nextLockedNonce[ETH] = 5`.
4. Nonce 6 (1 ETH, fully coverable) is never reached.
5. Every subsequent call to `_unlockWithdrawalRequests` repeats steps 1–3. The user at nonce 6 cannot withdraw until 50 ETH of liquidity is available. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
