Audit Report

## Title
Missing Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `answeredInRound`, and `roundId`, accepting any stale price without validation. The same repository already applies staleness and round-completeness guards in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming developer awareness of the pattern. A stale inflated price for any supported LST asset allows an unprivileged attacker to receive excess rsETH on deposit and redeem it for real assets, directly stealing from existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at [1](#0-0)  captures only `price` and discards all validity fields:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` at [2](#0-1)  explicitly guards against stale and incomplete rounds:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

`LRTOracle.getAssetPrice()` at [3](#0-2)  delegates directly to the unprotected oracle with no additional validation. The stale price then propagates into three critical paths:

1. **Deposit / rsETH minting** — `LRTDepositPool.getRsETHAmountToMint()` at [4](#0-3)  computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`. An inflated stale asset price directly inflates the rsETH minted per unit deposited.

2. **rsETH price update** — `LRTOracle._getTotalEthInProtocol()` at [5](#0-4)  sums `getAssetPrice(asset) * totalAssetDeposits` for every supported asset. A stale inflated price inflates the computed TVL and thus the rsETH price, while a stale depressed price triggers the downside-protection pause, freezing all user funds.

3. **Withdrawal unlock** — `LRTWithdrawalManager._createUnlockParams()` at [6](#0-5)  reads `lrtOracle.getAssetPrice(asset)` to determine payout amounts.

The `_validatePrices()` guard at [7](#0-6)  only checks caller-supplied min/max bounds, not oracle freshness, and does not mitigate this issue. The `pricePercentageLimit` downside-protection in `_updateRsETHPrice()` only triggers on price *decrease* and only when `updateRSETHPrice()` is called; it does not block deposits using a stale inflated price against a stored `rsETHPrice`.

## Impact Explanation
**Critical — Direct theft of user funds / protocol insolvency.**

If a Chainlink feed for any supported LST (stETH, ETHx, rETH, swETH) stops updating and the last reported price is above the true market price, an attacker deposits that LST via the public `depositAsset()` function and receives rsETH minted at the inflated rate. The excess rsETH represents a claim on ETH never deposited. The attacker immediately initiates withdrawal for a different asset at its correct price, extracting real ETH value from the pool. Every existing rsETH holder is left with a proportionally smaller claim on protocol assets, constituting direct theft and potential protocol insolvency.

## Likelihood Explanation
Chainlink feeds can stop updating due to network congestion, sequencer downtime, or feed deprecation — all documented, non-hypothetical failure modes. Any single feed among the supported LST assets going stale is sufficient. No special permissions are required: `depositAsset()` is callable by any address. The attacker needs only to observe that a feed has stopped updating (publicly visible on-chain) and submit a deposit transaction before the feed resumes or an admin intervenes.

## Recommendation
Add staleness and round-completeness checks inside `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```diff
+uint256 public constant MAX_PRICE_AGE = 3600; // configure per-feed heartbeat

 function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
     AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

-    (, int256 price,,,) = priceFeed.latestRoundData();
+    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
+    require(answeredInRound >= roundId, "Stale price");
+    require(updatedAt != 0, "Incomplete round");
+    require(block.timestamp - updatedAt <= MAX_PRICE_AGE, "Price too old");
+    require(price > 0, "Invalid price");

     return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
 }
```

`MAX_PRICE_AGE` should be configurable per feed (e.g., 3600s for 1-hour heartbeat feeds, 86400s for 24-hour feeds) to avoid false positives on slower-updating feeds.

## Proof of Concept
**Foundry fork test outline:**

1. Fork mainnet at a recent block.
2. Deploy or reference the live `ChainlinkPriceOracle` and `LRTDepositPool`.
3. Using `vm.mockCall`, override `latestRoundData()` for the stETH/ETH feed to return a stale `updatedAt` (e.g., `block.timestamp - 7200`) with an inflated price of `1.05e18`.
4. Call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")` as an unprivileged attacker address.
5. Assert that `rsethAmountToMint` is ~5% greater than it would be at the true price of `1.00e18`.
6. Initiate a withdrawal for ETHx at its correct price and assert the attacker receives more ETH value than deposited.
7. Assert that existing rsETH holders' redemption value has decreased proportionally.

The root cause — `(, int256 price,,,) = priceFeed.latestRoundData()` with no validity checks at [8](#0-7)  — is entirely within the protocol's own code and requires no external compromise beyond a Chainlink feed ceasing to update.
