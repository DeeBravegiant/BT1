Let me look at the exact ordering and condition in `add_ready_declares` and `validate_no_delayed_declare_front_run` more carefully.

The execution path is concrete and traceable. Here is the full analysis:

---

### Title
Delayed-Declare Front-Run Protection Bypassed at Delay Expiry via `add_ready_declares` / `validate_no_delayed_declare_front_run` Ordering — (`crates/apollo_mempool/src/mempool.rs`)

### Summary

`add_tx` calls `add_ready_declares` **before** running `validate_no_delayed_declare_front_run`. When the declare delay has elapsed, `add_ready_declares` moves the declare from `delayed_declares` into `tx_pool`. The subsequent front-run guard only inspects `delayed_declares`, so it sees nothing and passes. Fee-escalation logic then finds the declare in `tx_pool` and evicts it, defeating the protection entirely.

### Finding Description

The execution order inside `add_tx` is:

```
add_ready_declares()          ← moves D: delayed_declares → tx_pool if delay elapsed
  └─ add_tx_validations()
       └─ validate_fee_escalation()
            └─ validate_no_delayed_declare_front_run()   ← checks delayed_declares only
            └─ tx_pool.get_by_address_and_nonce()        ← finds D, proceeds to evict
``` [1](#0-0) 

`add_ready_declares` pops the declare from `delayed_declares` and calls `add_tx_inner` (inserting into `tx_pool`) whenever `now - declare_delay >= submission_time`: [2](#0-1) 

`validate_no_delayed_declare_front_run` only queries `delayed_declares`: [3](#0-2) 

Once D has been moved to `tx_pool` by `add_ready_declares`, this guard returns `Ok(())`. `validate_fee_escalation` then finds D in `tx_pool` via `get_by_address_and_nonce` and, if the incoming invoke passes `should_replace_tx`, calls `remove_replaced_tx` to permanently evict D: [4](#0-3) [5](#0-4) 

### Impact Explanation

A valid declare transaction D is permanently dropped from the mempool for a protocol-invalid reason: the front-run protection that was supposed to shield it is silently bypassed. The invoke I is admitted in its place. This falls under **High — mempool admission drops a valid transaction before sequencing**.

The attacker controls the account at `address`. They can:
1. Submit declare D → enters `delayed_declares`
2. Compute expiry: `submission_time + declare_delay` (both values are known/public)
3. Submit invoke I with fee above the escalation threshold at exactly that moment
4. `add_ready_declares` moves D to `tx_pool`; the guard misses it; D is evicted

### Likelihood Explanation

The timing is fully deterministic. The attacker knows their own submission time and the public `declare_delay` config. No privileged access is required — only the ability to submit transactions from the same account. The existing test `no_delay_declare_front_run` only covers the case where the clock has **not** advanced past the delay; it does not cover the boundary case where `add_ready_declares` fires during the same `add_tx` call. [6](#0-5) 

### Recommendation

Move `add_ready_declares` to **after** `add_tx_validations`, or — better — make `validate_no_delayed_declare_front_run` check both `delayed_declares` **and** any declare that was just promoted to `tx_pool` in the same call. The simplest fix is to swap the order so validation runs against the pre-promotion state:

```rust
pub fn add_tx(&mut self, args: AddTransactionArgs) -> MempoolResult<()> {
    let mut account_nonce_updates = self.remove_expired_txs();
    // Validate BEFORE promoting delayed declares.
    let tx_reference = TransactionReference::new(&args.tx);
    self.add_tx_validations(tx_reference, &args.tx, args.account_state.nonce)?;

    if !self.is_fifo() {
        self.add_ready_declares();
    }
    ...
}
```

#[test]
fn delayed_declare_front_run_at_expiry() {
    let fake_clock = Arc::new(FakeClock::default());
    let declare_delay = Duration::from_secs(5);
    let mut mempool = Mempool::new(
        MempoolConfig {
            static_config: MempoolStaticConfig {
                declare_delay,
                enable_fee_escalation: true,
                fee_escalation_percentage: 0, // any higher fee wins
                ..Default::default()
            },
            ..Default::default()
        },
        fake_clock.clone(),
    );

    // Step 1: submit declare D at (address=0x0, nonce=0).
    let declare = declare_add_tx_input(declare_tx_args!(
        tx_hash: tx_hash!(1),
        sender_address: contract_address!("0x0"),
        nonce: nonce!(0),
        resource_bounds: valid_resource_bounds_for_testing(),
    ));
    add_tx(&mut mempool, &declare);
    assert!(mempool.mempool_snapshot().unwrap().delayed_declares.contains(&tx_hash!(1)));

    // Step 2: advance clock to exactly the expiry boundary.
    fake_clock.advance(declare_delay);

    // Step 3: submit invoke I at (address=0x0, nonce=0) with higher fee.
    // add_ready_declares fires first, moving D to tx_pool.
    // validate_no_delayed_declare_front_run sees empty delayed_declares → passes.
    // Fee escalation evicts D and admits I.
    let invoke = add_tx_input!(
        tx_hash: 2, address: "0x0", tx_nonce: 0, account_nonce: 0, tip: 999
    );
    add_tx(&mut mempool, &invoke); // should be rejected but is accepted

    // D must still be in the mempool; I must be rejected.
    let snapshot = mempool.mempool_snapshot().unwrap();
    assert!(
        snapshot.transactions.contains(&tx_hash!(1)),
        "Declare D was evicted — front-run protection bypassed"
    );
    assert!(!snapshot.transactions.contains(&tx_hash!(2)));
}

    fn add_ready_declares(&mut self) {
        let now = self.clock.now();
        while let Some((submission_time, _args)) = self.delayed_declares.front() {
            if now - self.config.static_config.declare_delay < *submission_time {
                break;
            }
            let (_submission_time, args) =
                self.delayed_declares.pop_front().expect("Delay declare should exist.");
            self.add_tx_inner(args);
        }
        self.update_state_metrics();

    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }
