### Title
User-Controlled Finality Threshold Enables MPC Signing of Unfinalized Bitcoin Transactions - (File: crates/foreign-chain-inspector/src/bitcoin/inspector.rs)

### Summary
The `BitcoinInspector` uses the caller-supplied `confirmations` field from `BitcoinRpcRequest` as the sole finality threshold, with no protocol-level minimum enforced anywhere in the contract or inspector. An unprivileged caller can set `confirmations: 1` (or `confirmations: 0`) and obtain a valid MPC threshold signature attesting to a Bitcoin transaction that has not reached economic finality, enabling double-spend attacks in bridge scenarios.

### Finding Description
In `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, the `extract` function receives `block_confirmations_threshold: BlockConfirmations` directly from the user's `BitcoinRpcRequest.confirmations` field and applies it as the only finality gate:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;

if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations {
        expected: block_confirmations_threshold,
        got: transaction_block_confirmation,
    });
}
```

The upstream entry point `verify_foreign_transaction` in `crates/contract/src/lib.rs` performs no minimum-confirmation check before enqueuing the request â€” it only validates the domain, gas, deposit, and chain whitelist membership:

```rust

        request.domain_id,
        DomainPurpose::ForeignTx,

        MINIMUM_SIGN_REQUEST_DEPOSIT,
    );
    let requested_chain = request.request.chain();
    let supported_chains = self.get_supported_foreign_chains();
    if !supported_chains.contains(&requested_chain) { ... }
    // request is enqueued with no confirmation-count validation
}
```

The signed payload `ForeignTxSignPayloadV1` embeds the full `ForeignChainRpcRequest` â€” including the user-supplied `confirmations` value â€” so the MPC threshold signature attests to "this transaction had at least N confirmations" where N is entirely attacker-controlled. No per-chain minimum is stored in `ChainEntry`, `ForeignChainsMetadata`, or anywhere else in the contract state.

This is the direct analog of the external report: just as `_replenishPaymentBalance` uses a cost hardcoded for USDT without adjusting for other tokens, the MPC's finality gate uses a user-supplied threshold without enforcing chain-specific security minimums. Bitcoin requires 6 confirmations for economic finality; the protocol enforces zero.

### Impact Explanation
In the primary production use case â€” bridge inbound flows (foreign chain â†’ NEAR) â€” this enables a double-spend:

1. Attacker broadcasts a Bitcoin transaction sending funds to a bridge deposit address.
2. Attacker immediately calls `verify_foreign_transaction` with `confirmations: 1`.
3. MPC nodes observe â‰Ą 1 confirmation and each produce a signing share; the threshold signature is returned on-chain.
4. Attacker submits the MPC signature to the NEAR bridge contract to claim wrapped BTC.
5. Attacker mines a competing Bitcoin chain tip that excludes the original transaction (1-block reorg), recovering the BTC.
6. Attacker retains both the NEAR-side wrapped BTC and the original BTC.

The MPC threshold signature is the trust anchor for the bridge â€” downstream contracts have no independent way to verify Bitcoin finality and rely entirely on the MPC attestation. A signature over a 1-confirmation transaction is indistinguishable in form from one over a 6-confirmation transaction; the confirmation count is embedded in the payload but not separately authenticated.

### Likelihood Explanation
Medium. Bitcoin 1-block reorgs occur in practice (several per year on mainnet). The attacker controls the confirmation threshold, so they can minimize the required reorg depth to 1 block. The attack is profitable whenever the bridge value exceeds the cost of executing a 1-block reorg (which is non-trivial but has been demonstrated). Any bridge contract that trusts the MPC signature without separately enforcing a minimum confirmation count from the payload is directly exploitable.

### Recommendation
Enforce a per-chain minimum confirmation count at the contract level before enqueuing the request. The `ChainEntry` governance structure already supports per-chain configuration voted in by threshold participants; a `min_confirmations: u64` field should be added to `ChainEntry` and checked in `verify_foreign_transaction` against `request.request.confirmations()` before the request is accepted. This mirrors the external report's recommendation: verify that the security parameter (confirmation count / token cost) meets chain-specific requirements rather than accepting whatever the caller supplies.

1. The contract accepts the request (chain is whitelisted, deposit and gas are sufficient).

3. The check `1 <= actual_confirmations` passes as soon as the transaction has one confirmation.
4. All threshold nodes produce signing shares; the on-chain callback delivers a valid MPC signature.
5. Attacker submits the signature to the bridge contract to claim funds.
6. Attacker executes a 1-block Bitcoin reorg, recovering the deposited BTC.
