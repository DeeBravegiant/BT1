### Title
Missing CKD Response Verification for `AppPublicKey` Variant Allows Single Malicious Participant to Deliver Forged Confidential Key - (File: `crates/contract/src/lib.rs`)

### Summary

In `respond_ckd`, the cryptographic output check (`ckd_output_check`) is only applied for the `AppPublicKeyPV` variant of a CKD request. For the `AppPublicKey` variant, the response is accepted and delivered to the user with **no verification whatsoever**. Because `respond_ckd` requires only a single attested participant to call it (not a threshold), a single malicious participant can submit an arbitrary forged `CKDResponse` for any `AppPublicKey`-variant request and have it accepted and delivered to the user.

This is the direct analog to the reported Solana/DYAD bug: just as `updateXP` was called instead of `beforeKeroseneWithdrawn` (skipping the slashing hook for the liquidated note), here `ckd_output_check` is skipped for the `AppPublicKey` branch (skipping the integrity hook for the CKD response), while the correct check is applied for the `AppPublicKeyPV` branch.

---

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 654–689), after confirming the caller is an attested participant and the domain is BLS12-381, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies that the encrypted output is correctly derived from the MPC root key and the user's app public key pair. For `AppPublicKey`, the arm is a no-op — the response bytes are serialized and delivered to the user unconditionally.

Contrast this with `respond` (signature), where the signature is always verified against the expected derived public key before `resolve_yields_for` is called: [2](#0-1) 

The `respond_ckd` function requires only **one** attested participant to call it — there is no threshold aggregation check at the contract level: [3](#0-2) 

---

### Impact Explanation

A single malicious attested participant (strictly below the signing threshold) can:

1. Observe a pending `AppPublicKey`-variant CKD request on-chain (the request is public, including the user's `AppPublicKey`).
2. Construct an arbitrary `CKDResponse` — for example, one that encrypts an attacker-chosen secret under the user's `AppPublicKey`.
3. Call `respond_ckd` with this forged response before the honest nodes do.
4. The contract performs no verification and calls `resolve_yields_for`, delivering the forged response to the user.

The user receives a confidential key that the attacker chose and knows. Any assets or secrets the user subsequently protects with this key are compromised. This constitutes unauthorized delivery of forged confidential key material — matching the **Critical/High** impact class of "unauthorized access to MPC key derivation output" or at minimum **Medium** impact of "request-lifecycle manipulation that breaks production safety invariants."

---

### Likelihood Explanation

**Medium.** The attacker must be a single attested participant in the MPC network (i.e., hold a valid TEE attestation accepted by the contract). This is a realistic threat model for a Byzantine participant strictly below the signing threshold. The `AppPublicKey` variant is explicitly supported in the public API (`request_app_private_key`), so the attack surface is reachable by any user who submits a CKD request using the non-PV key type.

---

### Recommendation

Apply `ckd_output_check` (or an equivalent integrity check) for the `AppPublicKey` variant as well, or — if the `AppPublicKey` variant is intentionally unverifiable by design — require a threshold of participants to independently submit matching responses before resolving the yield, mirroring the trust model used for threshold signatures.

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: AppPublicKey(user_g1_point)` and `domain_id` pointing to a BLS12-381 CKD domain.
2. The contract enqueues the request and creates a yield promise.
3. Attacker (a single malicious attested participant) observes the pending `CKDRequest` on-chain.
4. Attacker constructs a `CKDResponse` encrypting an attacker-chosen secret under `user_g1_point`.
5. Attacker calls `respond_ckd(request, forged_response)`.
6. Contract executes: `assert_caller_is_attested_participant_and_protocol_active()` passes (attacker is attested), the `AppPublicKey` match arm is a no-op, and `resolve_yields_for` delivers `forged_response` to the user.
7. User decrypts the response and receives the attacker-chosen key — which the attacker also knows. [4](#0-3)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
