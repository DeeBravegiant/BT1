### Title
Stale Votes from Removed Participants Counted in TEE Hash Whitelisting Threshold Check — (File: `crates/contract/src/tee/proposal.rs`, `crates/contract/src/lib.rs`)

---

### Summary

`vote_code_hash`, `vote_add_launcher_hash`, and `vote_add_os_measurement` count **all** stored votes — including stale votes from participants removed during resharing — when checking whether the threshold is met. The post-resharing cleanup (`clean_tee_status`) is dispatched as a **detached, fire-and-forget promise** that can silently fail. If it does, stale votes persist indefinitely and are counted toward the threshold in subsequent calls, allowing a malicious TEE image hash to be whitelisted with fewer valid current-participant votes than the threshold requires.

---

### Finding Description

After resharing completes in `vote_reshared`, the contract transitions to the new `Running` state and then spawns several detached cleanup promises:

```rust
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        vec![],
        NearToken::from_yoctonear(0),
        Gas::from_tgas(self.config.clean_tee_status_tera_gas),
    )
    .detach();
``` [1](#0-0) 

The `.detach()` call means the outcome of `clean_tee_status` is never checked. If it fails (e.g., out-of-gas), stale votes from removed participants remain in `CodeHashesVotes::proposal_by_account` indefinitely. The code comment on the analogous `remove_non_participant_update_votes` promise explicitly acknowledges this failure mode:

```rust
// Note: MpcContract::vote_update uses filtering to ensure correctness even if this cleanup fails.
``` [2](#0-1) 

`vote_update` was hardened to filter non-participant votes at counting time. `vote_code_hash` was not:

```rust
pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
    ...
    let votes = self.tee_state.vote(code_hash, &participant);
    if votes >= self.threshold()?.value() {
        self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
    }
    Ok(())
}
``` [3](#0-2) 

The underlying `CodeHashesVotes::count_votes` counts every entry in `proposal_by_account` with no participant-membership filter:

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
``` [4](#0-3) 

Contrast this with `vote_update`, which explicitly re-filters against the current participant set at threshold-check time:

```rust
let valid_votes_count = running_state
    .parameters
    .participants()
    .participants()
    .iter()
    .filter(|(account_id, _, _)| {
        self.proposed_updates
            .vote_by_participant
            .get(account_id)
            .is_some_and(|voted_id| *voted_id == id)
    })
    .count();
``` [5](#0-4) 

And `TeeVerifierVotes::vote`, which also filters at count time:

```rust
voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
``` [6](#0-5) 

The same unfiltered counting flaw applies to `vote_add_launcher_hash` and `vote_add_os_measurement`, which use analogous vote/count helpers. [7](#0-6) 

---

### Impact Explanation

A malicious TEE Docker image hash, launcher image hash, or OS measurement set can be whitelisted with fewer valid current-participant votes than the governance threshold requires. Once whitelisted, a node running the malicious image passes `assert_caller_is_attested_participant_and_protocol_active()` and is accepted as a valid attested participant. This breaks the production safety invariant that threshold-many current participants must approve any TEE image before it can be used in the network — a contract execution-flow manipulation that corrupts participant-state accounting invariants.

---

### Likelihood Explanation

Two realistic paths exist:

1. **Cleanup failure (more likely):** `clean_tee_status` is a detached promise with a fixed gas budget (`clean_tee_status_tera_gas`). If it runs out of gas or panics, stale votes persist indefinitely. Any subsequent `vote_code_hash` call by a single current participant can then cross the threshold using the accumulated stale votes from removed participants.

2. **Race window (less likely but possible):** On NEAR, detached promises execute as separate receipts in the same or next block. A `vote_code_hash` transaction submitted in the same block as `vote_reshared` but scheduled before the `clean_tee_status` receipt executes would observe the stale votes.

The existing regression test `test_clean_non_participant_votes_removes_stale_votes` confirms the stale-vote accumulation is a real, tested concern — but the fix (filtering at count time) was only applied to `vote_update` and `vote_tee_verifier_change`, not to the TEE hash voting functions. [8](#0-7) 

---

### Recommendation

Apply the same at-count-time filtering used in `vote_update` to `CodeHashesVotes::count_votes` (and the equivalent helpers for launcher and OS-measurement votes). Pass the current `Participants` set into the counting function and skip entries whose `AuthenticatedParticipantId` is no longer a current participant:

```rust
fn count_votes_for_current_participants(
    &self,
    proposal: &NodeImageHash,
    participants: &Participants,
) -> u64 {
    self.proposal_by_account
        .iter()
        .filter(|(participant_id, prop)| {
            *prop == proposal
                && participants.is_participant_given_participant_id(&participant_id.get())
        })
        .count() as u64
}
```

This makes the threshold check correct regardless of whether `clean_tee_status` succeeds, matching the defensive pattern already applied to `vote_update`.

---

### Proof of Concept

**Setup:** 5 participants P1–P5, threshold = 3. P1 and P2 are adversarial.

1. **Before resharing:** P1 and P2 call `vote_code_hash(malicious_hash)`. `count_votes` = 2 < 3. Hash not whitelisted.
2. **Resharing:** New participant set = {P3, P4, P5}, threshold = 3. P1 and P2 are removed. `vote_reshared` transitions state and schedules `clean_tee_status` as a detached promise.
3. **Cleanup fails:** `clean_tee_status` runs out of gas (or is never triggered). P1 and P2's votes remain in `proposal_by_account`.
4. **Exploit:** P3 calls `vote_code_hash(malicious_hash)`. `count_votes` returns 3 (P1 stale + P2 stale + P3 current). `3 >= threshold(3)` → `whitelist_tee_proposal` is called. The malicious hash is now whitelisted with only **1 valid current-participant vote**.
5. **Result:** A node running the malicious image passes `assert_caller_is_attested_participant_and_protocol_active()` and is accepted as a valid attested participant in the MPC network.

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
