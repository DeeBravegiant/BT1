### Title
Missing G2 Subgroup Check in `alt_bn128_pairing_check` Allows False Pairing Verification - (`runtime/near-vm-runner/src/logic/alt_bn128.rs`)

### Summary

The `decode_g2` function in `alt_bn128.rs` only verifies that a supplied G2 point lies on the twist curve (via `bn::AffineG2::new`), but does **not** verify that the point is in the prime-order subgroup of the twist. The nearcore documentation explicitly promises this check. Any contract using `alt_bn128_pairing_check` for Groth16 verification is therefore exposed to a forged-proof attack using a non-subgroup G2 point.

### Finding Description

`decode_g2` calls `bn::AffineG2::new(x, y)`, which in the substrate-bn crate only validates that `(x, y)` satisfies the twist curve equation `y² = x³ + 3/(i+9)` over Fq2. It does **not** multiply the candidate point by the prime order `r` to confirm it is in the G2 prime-order subgroup. [1](#0-0) 

The documented API contract for `alt_bn128_pairing_check` explicitly states:

> "If point coordinates are not on curve, **point is not in the subgroup**, scalar is not in the field or data are wrong serialized … the function returns `AltBn128InvalidInput`." [2](#0-1) 

The same promise appears in the wasmtime runner: [3](#0-2) 

Yet `pairing_check` calls `decode_g2` and proceeds directly to `bn::pairing_batch` with no subsequent subgroup check: [4](#0-3) 

Contrast this with the BLS12-381 pairing implementation, which explicitly calls `blst_p1_affine_in_g1` and `blst_p2_affine_in_g2` before computing the pairing: [5](#0-4) 

### Impact Explanation

For BN128, the twist E'(Fq2) has order `r * h2` where `h2 > 1`. Points in the cofactor part of E'(Fq2) (i.e., on the curve but outside G2) can be passed to `pairing_check`. Because the Ate pairing is not well-defined outside the prime-order subgroup, an attacker can craft G2 inputs such that the pairing equation evaluates to `Gt::one()` for a statement that is false. Any on-chain Groth16 verifier contract (bridge, light client, validator-set updater) that relies on `alt_bn128_pairing_check` and trusts the documented subgroup-rejection guarantee is vulnerable to accepting a forged proof.

### Likelihood Explanation

The attack path is fully attacker-controlled: submit a transaction calling a contract that invokes `alt_bn128_pairing_check` with a crafted non-subgroup G2 point. No privileged access is required. The mathematical construction of such points for BN128 is well-documented in the literature (EIP-197 rationale, "Invalid Curve Attacks on (Hyper)elliptic Curve Cryptosystems").

### Recommendation

In `decode_g2`, after constructing the `bn::G2` point, perform an explicit subgroup check by verifying `point * r == G2::zero()` (scalar multiplication by the group order). Alternatively, use a library that performs this check inside `AffineG2::new`. This is exactly what EIP-197 mandates and what the BLS12-381 path already does via `blst_p2_affine_in_g2`.

### Proof of Concept

1. Find a point `Q ∈ E'(Fq2)` with `Q ∉ G2` (i.e., `Q * r ≠ 0`). Such points exist because `h2 > 1`.
2. Deploy a minimal Groth16 verifier contract that calls `alt_bn128_pairing_check`.
3. Submit a transaction with a "proof" where the B element is `Q` instead of a valid G2 point.
4. Observe that `decode_g2` accepts `Q` (it is on the curve), `pairing_batch` is called, and the check returns `true` for a false statement.
5. The documented invariant — that non-subgroup points are rejected with `AltBn128InvalidInput` — is violated. [1](#0-0) [4](#0-3)

pub(crate) fn pairing_check(
    elements: &[[u8; PAIRING_CHECK_ELEMENT_SIZE]],
) -> Result<bool, InvalidInput> {
    let elements: Vec<(bn::G1, bn::G2)> = elements
        .iter()
        .map(|chunk| {
            let (g1, g2) = stdx::split_array(chunk);
            let g1 = decode_g1(g1)?;
            let g2 = decode_g2(g2)?;
            Ok((g1, g2))
        })
        .collect::<Result<Vec<_>, InvalidInput>>()?;

    let res = bn::pairing_batch(&elements) == bn::Gt::one();

    Ok(res)
}

fn decode_g2(raw: &[u8; 2 * POINT_SIZE]) -> Result<bn::G2, InvalidInput> {
    let (x, y) = stdx::split_array(raw);
    let x = decode_fq2(x)?;
    let y = decode_fq2(y)?;
    if x.is_zero() && y.is_zero() {
        Ok(bn::G2::zero())
    } else {
        bn::AffineG2::new(x, y)
            .map_err(|_err| InvalidInput::new("invalid g2", raw))
            .map(bn::G2::from)
    }
}
