### Title
Stale `target_shard_index` After Shard Split Produces Silently-Wrong `outcome_root_proof` in `GetExecutionOutcome` — (`chain/client/src/view_client_actor.rs`)

---

### Summary

`GetExecutionOutcome` computes `target_shard_index` once from the **outcome block's** shard layout, then uses that same index to select a merkle path from `outcome_roots`, which is built from the **next block's** chunk headers. When a shard split occurs between those two blocks, the next block has a different (larger) shard layout, so the stale index silently selects the wrong shard's outcome root. The returned `outcome_root_proof` is cryptographically invalid, causing every light-client proof for transactions executed just before the split to fail verification.

---

### Finding Description

The handler for `GetExecutionOutcome` in `view_client_actor.rs` follows this sequence:

**Step 1 — index computed from the outcome block's epoch (old layout):** [1](#0-0) 

`epoch_id` is the epoch of the block that contains the outcome. `shard_layout` is that epoch's layout (N shards). `target_shard_index` is the position of `target_shard_id` in that N-shard layout.

**Step 2 — `get_next_block_hash_with_new_chunk` transparently updates `target_shard_id` to a child shard:** [2](#0-1) 

When the next block crosses an epoch boundary with a shard split, `shard_ids` is replaced with the child shards of the original shard, and the function returns one of those child shard IDs. The caller receives an updated `target_shard_id` (a child shard in the new N+1-shard layout).

**Step 3 — `outcome_roots` is built from the next block (new layout, N+1 entries), but the stale index is used:** [3](#0-2) 

`outcome_roots` has N+1 entries (one per chunk in the post-split block). The bounds check at line 1174 passes silently because `target_shard_index` (< N) is always less than N+1. Line 1182 then reads `outcome_roots[target_shard_index]` using the **old** index, which points to a different shard's outcome root than the one that actually contains the transaction's outcome.

**Concrete example (1→2 shard split):**

| Variable | Value |
|---|---|
| Old layout (epoch E) | 1 shard: shard 0 |
| `target_shard_index` | 0 |
| New layout (epoch E+1) | 2 shards: shard 0 (left child), shard 1 (right child) |
| `get_next_block_hash_with_new_chunk` returns | `(B', shard_1)` |
| `outcome_roots` | `[root_shard_0, root_shard_1]` |
| `outcome_roots[0]` accessed | `root_shard_0` ← **wrong shard** |
| Correct index needed | `shard_layout_new.get_shard_index(shard_1)` = 1 |

The fix is to recompute `target_shard_index` from the new shard layout using the updated `target_shard_id` returned by `get_next_block_hash_with_new_chunk`, rather than reusing the index computed before the call.

---

### Impact Explanation

Every call to `EXPERIMENTAL_light_client_proof` (or `light_client_proof`) for a transaction or receipt whose outcome block is the last block before a shard split will receive a structurally valid but cryptographically wrong `outcome_root_proof`. The merkle path will be for the wrong shard's outcome root. Light-client verification:

```python
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
```

will produce a value that does not match `block_header_lite.inner_lite.outcome_root`, causing the proof to be rejected. From the light client's perspective, a fully confirmed, valid transaction appears unprovable — it is effectively rejected.

The stated impact of `InconsistentState` error is **not** what actually occurs. The bounds check at line 1174 passes silently (old index < N < N+1), so no error is raised. The bug is a **silent wrong-proof** rather than an explicit error, which is arguably worse because callers receive no indication that the proof is invalid.

---

### Likelihood Explanation

The trigger is a normal protocol event (shard split / resharding) combined with a normal public RPC call. No special privileges are required. Any user who submitted a transaction in the epoch immediately before a shard split and then queries `light_client_proof` will hit this path. With dynamic resharding enabled, shard splits are expected to occur periodically in production.

---

### Recommendation

After `get_next_block_hash_with_new_chunk` returns the updated `target_shard_id`, recompute `target_shard_index` from the **next block's** shard layout before indexing into `outcome_roots`. Concretely, after line 1163 (`if let Some((h, target_shard_id)) = res`), fetch the epoch of block `h`, get its shard layout, and call `shard_layout.get_shard_index(target_shard_id)` to obtain the correct index for the post-split layout.

---

### Proof of Concept

Write a test-loop test that:
1. Configures a network with dynamic resharding enabled and a short epoch length.
2. Submits a transaction and waits for it to execute in the last block of epoch E (just before the split).
3. Advances the chain into epoch E+1 (post-split, N+1 shards).
4. Calls `view_client.handle(GetExecutionOutcome { id })` for that transaction.
5. Asserts that `verify_path(block_outcome_root, &outcome_root_proof, &chunk_outcome_root)` returns `true`.

Without the fix, step 5 will fail because `outcome_root_proof` is the merkle path for the wrong shard's outcome root.
