Audit Report

## Title
Missing Staleness Check in `CrossChainRateReceiver.getRate()` Enables Mispriced wrsETH Minting — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the last stored `rate` with no validation against `lastUpdated`. All three L2 deposit pools (`RSETHPoolV3`, `RSETHPoolV2`, `RSETHPoolNoWrapper`) consume this value directly to compute how many wrsETH/rsETH tokens to mint or transfer. When the LayerZero relay is delayed or stalled, the stored rate diverges from the true L1 rsETH price, causing every deposit to be priced at the wrong rate. In the direction where the stale rate is lower than the true rate, depositors receive excess tokens minted from yield already accrued by existing holders — constituting theft of unclaimed yield.

## Finding Description

`CrossChainRateReceiver` stores two state variables updated only inside `lzReceive`:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L95-97
rate = _rate;
lastUpdated = block.timestamp;
```

The public read function exposes the raw stored value with no age validation:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

All three pools delegate pricing through the same `IOracle` interface:

- `RSETHPoolV3.getRate()` → `IOracle(rsETHOracle).getRate()` (L235-237)
- `RSETHPoolV2.getRate()` → `IOracle(rsETHOracle).getRate()` (L201-203)
- `RSETHPoolNoWrapper.getRate()` → `IOracle(rsETHOracle).getRate()` (L220-222)

The minting formula in all three pools divides by this rate:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
// RSETHPoolV3 L307, RSETHPoolV2 L233, RSETHPoolNoWrapper L285
```

There is no guard anywhere in the deposit path that checks `block.timestamp - lastUpdated < maxStaleness`. By contrast, the protocol's own `ChainlinkOracleForRSETHPoolCollateral` contract does implement staleness protection (`answeredInRound < roundID`, `timestamp == 0`), confirming the protocol is aware of and applies this pattern for Chainlink feeds but omitted it for the cross-chain receiver.

The `updateRate()` function on the provider side is permissionless — any caller willing to pay the LayerZero fee can trigger it — but it is not automated on-chain. Any period of bridge congestion, fee spike, or operator inattention leaves the L2 rate stale.

**Exploit path (Scenario A — stale rate < true rate, highest severity):**
1. LayerZero relay stalls; `lastUpdated` is hours/days old.
2. True rsETH price on L1 has increased (e.g., from 1e18 to 1.05e18) due to accrued yield.
3. Attacker (or any depositor) calls `pool.deposit{value: 1 ether}("ref")`.
4. Pool calls `getRate()` → returns stale 1e18.
5. `rsETHAmount = 1e18 * 1e18 / 1e18 = 1e18 wrsETH` is minted.
6. At the true rate, the depositor should receive only `1e18 * 1e18 / 1.05e18 ≈ 0.952e18 wrsETH`.
7. The excess `~0.048e18 wrsETH` is minted from yield already accrued by existing holders — permanently diluting their unclaimed yield.

## Impact Explanation

**High: Theft of unclaimed yield.**

When the stale rate is lower than the true rate (Scenario A), new depositors receive more wrsETH than their ETH entitles them to. The excess tokens represent yield that existing wrsETH holders had already accrued but not yet claimed. This yield is permanently diluted and unrecoverable by the original holders. The magnitude scales with the size of the deposit and the degree of rate drift; during a multi-day relay outage with normal rsETH yield accrual, the discrepancy is material and repeatable by any depositor.

A secondary impact (Medium: Permanent freezing of unclaimed yield) applies in Scenario B (stale rate > true rate), where depositors receive fewer tokens than entitled and the shortfall is permanently unclaimable.

## Likelihood Explanation

The `updateRate()` function on `CrossChainRateProvider`/`MultiChainRateProvider` is permissionless but requires the caller to pay a LayerZero messaging fee. It is not automated on-chain. Any period of bridge congestion, fee spike, or operator inattention produces a stale rate. rsETH accrues yield continuously, so even a 24-hour gap produces measurable mispricing. The exploit requires no special privileges — any depositor triggers it simply by calling `deposit()` during a stale period. The condition is realistic and repeatable.

## Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the stored rate is too old:

```solidity
uint256 public maxStaleness; // e.g., set to 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes all three pool deposit functions to revert rather than execute at a stale price, giving operators time to refresh the rate before funds are mispriced. `maxStaleness` should be owner-configurable and initialized to a safe default (e.g., 24 hours).

## Proof of Concept

```solidity
// Foundry fork test (Scenario A: stale rate < true rate → theft of unclaimed yield)
function test_staleRateCausesYieldTheft() external {
    // Simulate: lzReceive was called 3 days ago with rate = 1.00e18
    // True current rsETH price has since risen to 1.05e18 due to yield accrual
    vm.store(address(rateReceiver), bytes32(uint256(0)), bytes32(uint256(1.00e18))); // rate slot
    vm.store(address(rateReceiver), bytes32(uint256(1)), bytes32(block.timestamp - 3 days)); // lastUpdated slot

    uint256 depositAmount = 1 ether;
    vm.deal(user, depositAmount);
    vm.prank(user);
    pool.deposit{value: depositAmount}("ref");

    uint256 wrsETHReceived = wrsETH.balanceOf(user);
    // At stale 1.00e18: user gets 1.000e18 wrsETH
    // At true 1.05e18: user should get ~0.952e18 wrsETH
    // Excess ~0.048e18 wrsETH is minted from existing holders' accrued yield
    assertGt(wrsETHReceived, 0.952e18, "User received more wrsETH than entitled at true rate");
}
```
