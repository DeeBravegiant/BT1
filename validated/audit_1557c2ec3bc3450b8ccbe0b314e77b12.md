Looking at the full data flow from the SQMR sync path through to storage and RPC, I need to verify whether any validation guards exist between the peer-supplied `transaction_hash` and the stored value.

### Title
Unauthenticated Sync Peer Can Inject Arbitrary Transaction Hashes into Node Storage via Missing Validation in `TransactionStreamFactory::parse_data_for_block` — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

The p2p sync client accepts `transaction_hash` values from untrusted peers verbatim and writes them to persistent storage without any validation against the block header's `transaction_commitment` or by recomputing the hash from the transaction body. A malicious sync peer can inject arbitrary `TransactionHash` values for any synced block, permanently corrupting the node's storage and causing RPC endpoints to serve authoritative-looking wrong transaction hashes.

---

### Finding Description

In `TransactionStreamFactory::parse_data_for_block`, the `transaction_hash` field of each peer-supplied `FullTransaction` is pushed directly into `block_body.transaction_hashes` with an explicit TODO acknowledging the missing guard: [1](#0-0) 

The resulting `BlockBody` is then written to persistent storage unconditionally: [2](#0-1) 

`append_body` → `write_transactions` stores the peer-supplied hash in both the `transaction_hash_to_idx` table and `TransactionMetadata.tx_hash`: [3](#0-2) 

Two independent validations that could catch this are both absent in the p2p sync path:

1. **No block-header signature verification**: `HeaderStreamBuilder::parse_data_for_block` only checks block number ordering and that exactly one signature is present. `verify_block_signature` (which would authenticate the `transaction_commitment` in the header) is never called anywhere in `crates/apollo_p2p_sync/`: [4](#0-3) 

2. **No transaction hash recomputation**: `validate_transaction_hash` exists in `starknet_api` but is never called in the p2p sync path. The grep for both `validate_transaction_hash` and `verify_block_signature` in `crates/apollo_p2p_sync/**` returns zero matches.

The `transaction_commitment` stored in the header (itself from an untrusted peer) is never used to verify the transaction hashes received from that same peer.

---

### Impact Explanation

The injected hashes are served directly by RPC:

- `get_block_transaction_hashes` reads from `TransactionMetadata.tx_hash` and returns the injected values: [5](#0-4) 

- `get_block_w_full_transactions` zips stored transactions with stored hashes and returns them as authoritative: [6](#0-5) 

- `get_block_tx_hashes_by_number` returns the injected hashes directly: [7](#0-6) 

Consequences for block N:
- `starknet_getBlockWithTxHashes` returns injected hashes instead of real ones.
- `starknet_getTransactionByHash(real_hash)` returns not-found.
- `starknet_getTransactionByHash(injected_hash)` returns the wrong transaction.
- Event lookups keyed by transaction hash are misattributed.

This matches the allowed impact: **High — RPC returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

Any node running the p2p sync client (`P2pSyncClient::run`) that connects to a malicious peer is vulnerable. No special privileges are required — the attacker only needs to respond to a standard SQMR `/starknet/transactions/1` query with crafted `FullTransaction` messages. The attack is persistent (storage is not self-healing) and silent (no error is logged or returned).

---

### Recommendation

1. **Recompute and verify the transaction hash** from the transaction body before accepting it. Call `get_transaction_hash(transaction, chain_id, &TransactionOptions::default())` and reject the peer (return `BadPeerError`) if it does not match the supplied `transaction_hash`.

2. **Verify the block header signature** against a known sequencer public key in `HeaderStreamBuilder::parse_data_for_block` before storing the header, so the `transaction_commitment` it carries is trustworthy.

3. **Optionally**, after collecting all transaction hashes for a block, recompute `calculate_transaction_commitment` and compare it against the stored header's `transaction_commitment` as a defense-in-depth check.

---

### Proof of Concept

Drive `BlockDataStreamBuilder::create_stream` with a mock `ClientResponsesManager` that returns a `FullTransaction` where `transaction_hash` is a crafted value unrelated to `hash(transaction)`. After the stream yields and `write_to_storage` is called:

```rust
// Crafted hash != hash(transaction)
let injected_hash = TransactionHash(felt!("0xdeadbeef"));
let real_tx = invoke_tx(InvokeTxArgs::default());
let full_tx = FullTransaction {
    transaction: Transaction::Invoke(real_tx),
    transaction_output: TransactionOutput::Invoke(Default::default()),
    transaction_hash: injected_hash,  // attacker-controlled
};
// ... drive parse_data_for_block with mock returning this full_tx ...

// Assert storage was corrupted:
assert_eq!(
    storage_reader.begin_ro_txn().unwrap()
        .get_block_transaction_hashes(BlockNumber(0)).unwrap().unwrap()[0],
    injected_hash  // injected value is stored verbatim
);
```

The assertion passes because `parse_data_for_block` pushes `transaction_hash` without any check. [8](#0-7)

    fn parse_data_for_block<'a>(
        transactions_response_manager: &'a mut ClientResponsesManager<DataOrFin<FullTransaction>>,
        block_number: BlockNumber,
        storage_reader: &'a StorageReader,
    ) -> BoxFuture<'a, Result<Option<Self::Output>, ParseDataError>> {
        async move {
            let mut block_body = BlockBody::default();
            let mut current_transaction_len = 0;
            let target_transaction_len = storage_reader
                .begin_ro_txn()?
                .get_block_header(block_number)?
                .expect("A header with number lower than the header marker is missing")
                .n_transactions;
            while current_transaction_len < target_transaction_len {
                let maybe_transaction = transactions_response_manager.next().await.ok_or(
                    ParseDataError::BadPeer(BadPeerError::SessionEndedWithoutFin {
                        type_description: Self::TYPE_DESCRIPTION,
                    }),
                )?;
                let Some(FullTransaction { transaction, transaction_output, transaction_hash }) =
                    maybe_transaction?.0
                else {
                    if current_transaction_len == 0 {
                        return Ok(None);
                    } else {
                        return Err(ParseDataError::BadPeer(BadPeerError::NotEnoughTransactions {
                            expected: target_transaction_len,
                            actual: current_transaction_len,
                            block_number: block_number.0,
                        }));
                    }
                };
                block_body.transactions.push(transaction);
                block_body.transaction_outputs.push(transaction_output);
                // TODO(eitan): Validate transaction hash from untrusted sources
                block_body.transaction_hashes.push(transaction_hash);
                current_transaction_len += 1;
            }
            Ok(Some((block_body, block_number)))
        }
        .boxed()
    }

pub fn get_block_tx_hashes_by_number<Mode: TransactionKind>(
    txn: &StorageTxn<'_, Mode>,
    block_number: BlockNumber,
) -> Result<Vec<TransactionHash>, ErrorObjectOwned> {
    let transaction_hashes = txn
        .get_block_transaction_hashes(block_number)
        .map_err(internal_server_error)?
        .ok_or_else(|| ErrorObjectOwned::from(BLOCK_NOT_FOUND))?;

    Ok(transaction_hashes)
}
