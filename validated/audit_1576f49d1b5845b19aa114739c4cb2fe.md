Let me examine the relevant code to evaluate this claim.

The code is clear enough to render a definitive verdict. Here is the full trace:

**Step 1 – Can a nonce=2 invoke reach the mempool for a non-existent account?**

`validate_nonce` allows any invoke nonce in `[account_nonce, account_nonce + max_allowed_nonce_gap]` (default gap = 200). [1](#0-0) 

For a nonce=2 invoke, `skip_stateful_validations` returns `false` (only triggers at nonce=1), so `run_validate_entry_point` IS called with `skip_validate=false`. [2](#0-1) 

If address A has no deployed contract (account_nonce=0, counterfactual), `__validate__` on the nonce=2 tx fails — the contract does not exist. The nonce=2 tx is **rejected**. The attacker cannot pre-seed the mempool for a non-existent account.

**Step 2 – What if address A is an existing account (nonce=0, contract deployed)?**

Then the nonce=2 invoke passes `__validate__` (contract exists, attacker signs it correctly) and enters the mempool. Now a *third party* (or the same attacker) submits a nonce=1 invoke for address A with an **invalid/forged signature**.

`skip_stateful_validations` fires:
- tx is Invoke ✓
- tx.nonce() == 1 ✓
- account_nonce == 0 ✓
- `account_tx_in_pool_or_recent_block(A)` → `tx_pool.contains_account(A)` → **true** (nonce=2 tx is there) [3](#0-2) 

Result: `skip_validate = true`.

**Step 3 – What does `skip_validate=true` actually skip?**

`run_validate_entry_point` sets `execution_flags.validate = !skip_validate = false`. [4](#0-3) 

In `StatefulValidator::perform_validations`, when `!tx.execution_flags.validate`, the function returns `Ok(())` immediately — `__validate__` is never called. [5](#0-4) 

The invalid-signature nonce=1 tx is admitted to the mempool.

**Step 4 – Is the comment's assumption correct?**

The code comment claims:
> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that **passed validations**." [6](#0-5) 

The assumption is that a future-nonce tx in the pool implies the account is valid. But it does **not** imply the *current* nonce=1 tx has a valid signature. The nonce=2 tx was signed by whoever submitted it; the nonce=1 tx can be signed by anyone (or not at all).

**Step 5 – What happens at execution time?**

The batcher executes the nonce=1 tx and calls `__validate__`. The invalid signature causes `__validate__` to fail. The tx is not committed to a block. However, the **admission bypass is real**: the gateway accepted a transaction whose signature was never verified.

**Verdict:**

The vulnerability is **real** but the impact is bounded:

- An unprivileged third party can submit a nonce=1 invoke for any address A that already has a future-nonce tx in the mempool, with an arbitrary/invalid signature, and have it admitted without `__validate__` being called.
- The batcher will reject it at execution time (no state corruption, no fund theft).
- The practical harm is **mempool griefing**: the attacker can flood the mempool with unverified transactions for any active account, consuming mempool capacity and batcher CPU.
- This satisfies the scope criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

### Title
Gateway `skip_stateful_validations` admits invoke transactions with unverified signatures by triggering on any pooled tx, not only `deploy_account` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
`skip_stateful_validations` skips the `__validate__` entry-point call for any nonce=1 invoke whose sender address appears in the mempool, regardless of whether the pooled transaction is a `deploy_account`. An unprivileged attacker can pre-seed the mempool with a legitimately-signed future-nonce invoke for an existing account, then submit a nonce=1 invoke for the same address with an invalid/forged signature. The gateway admits the second transaction without ever calling `__validate__`.

### Finding Description
`skip_stateful_validations` is intended to support the UX pattern of sending `deploy_account + invoke(nonce=1)` simultaneously, where the account does not yet exist on-chain and `__validate__` would fail. The guard is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        ...
}
``` [7](#0-6) 

`account_tx_in_pool_or_recent_block` returns `true` if **any** transaction from that address is in the pool or a recent block — it does not filter for `deploy_account` type:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

When `skip_validate=true`, `run_validate_entry_point` sets `execution_flags.validate = false`, and `StatefulValidator::perform_validations` returns `Ok(())` before calling `__validate__`: [4](#0-3) [5](#0-4) 

`validate_by_mempool` (called before `skip_stateful_validations`) only checks nonce ordering and duplicates — it does not verify signatures: [8](#0-7) 

### Impact Explanation
The gateway admits a nonce=1 invoke transaction whose signature was never verified by the account contract. This satisfies the "High" admission impact criterion. The batcher will reject the transaction at execution time (no state corruption), but the mempool accepts it, consuming capacity and batcher resources. An attacker can repeat this for any active account visible in the mempool.

### Likelihood Explanation
Moderate. The attacker needs an existing account (nonce=0, contract deployed) with a future-nonce tx already in the mempool. Active accounts with pending future-nonce transactions are observable on-chain/via RPC. The attacker does not need any privileged access.

### Recommendation
In `skip_stateful_validations`, replace the generic `account_tx_in_pool_or_recent_block` check with a dedicated query that verifies a `deploy_account` transaction specifically exists for the sender address. Alternatively, add a transaction-type filter inside `account_tx_in_pool_or_recent_block` so it only returns `true` when a `deploy_account` (or a committed block containing one) is present for the address.

### Proof of Concept
1. Address A exists on-chain with nonce=0 (contract deployed, attacker holds the key).
2. Attacker submits `invoke(sender=A, nonce=2, valid_signature)` → passes `__validate__` → enters mempool.
3. Any party submits `invoke(sender=A, nonce=1, signature=[0xff,0xff,...])` (garbage signature).
4. Gateway: `skip_stateful_validations` fires (nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)=true`).
5. `run_validate_entry_point` is called with `skip_validate=true` → `execution_flags.validate=false` → `__validate__` not called.
6. The garbage-signature nonce=1 tx is forwarded to the mempool and admitted.
7. Batcher picks it up, calls `__validate__`, it fails — tx is discarded without state change, but mempool slot was consumed.
