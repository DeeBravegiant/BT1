### Title
Unsigned `farmer_reward_address_override` in `RespondSignatures` Enables Malicious Harvester to Redirect Farmer Block Rewards - (File: `chia/farmer/farmer_api.py`)

### Summary
The `farmer_reward_address_override` field in the `RespondSignatures` harvester-to-farmer protocol message is not covered by any BLS signature. A malicious harvester can inject an arbitrary puzzle hash into this field, causing the farmer to redirect the block reward (1.75 XCH) to an attacker-controlled address. The CHIP-22 fee-quality threshold check is bypassed entirely because it is only evaluated on `NewProofOfSpace`, not on `RespondSignatures`, and even then it is only a log warning with no enforcement.

### Finding Description

**Unsigned field in `RespondSignatures`**

`RespondSignatures` carries `farmer_reward_address_override: bytes32 | None` alongside the actual BLS-signed message tuples: [1](#0-0) 

The BLS signatures in `message_signatures` cover only the SP hashes (`challenge_chain_sp`, `reward_chain_sp`). `farmer_reward_address_override` is a plain, unauthenticated field â€” no key signs it, no hash of it is included in any signed payload.

**Unconditional acceptance in `_process_respond_signatures`**

In `FarmerAPI._process_respond_signatures`, the farmer blindly substitutes the harvester-supplied address for its own configured `farmer_target`: [2](#0-1) 

This `farmer_reward_address` is then forwarded verbatim as `farmer_puzzle_hash` in `DeclareProofOfSpace`: [3](#0-2) 

The full node uses `request.farmer_puzzle_hash` directly to determine where the block reward is paid: [4](#0-3) 

**Fee-quality check is bypassed and unenforced**

CHIP-22 introduced `farmer_reward_address_override` as a harvester-fee mechanism. The only validation is in `new_proof_of_space`, triggered when `NewProofOfSpace.farmer_reward_address_override is not None`: [5](#0-4) 

That call only logs a warning â€” it never blocks the flow: [6](#0-5) 

Critically, `_process_respond_signatures` performs **no** fee-quality check on `response.farmer_reward_address_override`. A harvester can therefore:

1. Send `NewProofOfSpace` **without** `farmer_reward_address_override` â€” no fee-quality check is triggered at all.
2. In the subsequent `RespondSignatures`, set `farmer_reward_address_override` to an attacker-controlled address.
3. The farmer accepts it unconditionally and forwards it to the full node.

The `RespondSignatures` struct definition confirms the field is entirely outside the signed payload: [7](#0-6) 

The signed messages are only the SP hashes: [8](#0-7) 

### Impact Explanation
Every block won by the farmer pays 1.75 XCH to `farmer_puzzle_hash`. A malicious harvester that controls any connected harvester process can redirect 100 % of the farmer's block rewards to an attacker-controlled address for every block won while the harvester is connected. This is unauthorized payout redirection of XCH with no on-chain or cryptographic guard.

### Likelihood Explanation
CHIP-22 explicitly targets third-party harvester operators (e.g., GPU-farm services, remote plot managers). Any such operator is an unprivileged network peer who can exploit this without leaked keys, host compromise, or cryptographic breaks. The farmer's harvester port is the only prerequisite.

### Recommendation
Include `farmer_reward_address_override` in the signed payload. For example, the harvester should sign `hash(challenge_chain_sp || reward_chain_sp || farmer_reward_address_override)` with the plot key, and the farmer must verify this before accepting the override. Alternatively, enforce the CHIP-22 fee-quality threshold as a hard rejection (not a log warning) in `_process_respond_signatures` and cross-check that `RespondSignatures.farmer_reward_address_override` matches what was declared in the corresponding `NewProofOfSpace`.

### Proof of Concept

1. Operate a harvester connected to a victim farmer.
2. When the farmer sends `RequestSignatures` for a winning proof, respond with valid BLS signatures over the SP hashes (normal harvester behavior).
3. In the `RespondSignatures` response, set `farmer_reward_address_override` to an attacker-controlled puzzle hash â€” **without** having set it in the earlier `NewProofOfSpace` message (so no fee-quality log is even emitted).
4. `_process_respond_signatures` (line 917â€“918) substitutes the attacker's address for `farmer_reward_address`.
5. `DeclareProofOfSpace` is sent to the full node with `farmer_puzzle_hash = attacker_address`.
6. `create_foliage` embeds this address in `FoliageBlockData.farmer_reward_puzzle_hash`; the block reward coin is created payable to the attacker. [9](#0-8)
