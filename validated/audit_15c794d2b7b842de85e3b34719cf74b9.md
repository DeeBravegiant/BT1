### Title
Attacker Can Grief a Relayer by Front-Running a `SignedDelegateAction` (Meta-Transaction) to Consume the User's Nonce — (`runtime/runtime/src/actions.rs`)

### Summary

NEAR's meta-transaction system (NEP-366) allows a user to sign a `DelegateAction` off-chain and hand it to a relayer, who wraps it in their own transaction and pays gas. Because the `SignedDelegateAction` is a self-contained, reusable signed payload with no binding to the outer transaction's signer, an attacker who observes the relayer's transaction in the mempool can extract the `SignedDelegateAction` and submit it inside their own transaction first. This consumes the user's access-key nonce, causing the relayer's original transaction to fail with `DelegateActionInvalidNonce`. The relayer loses gas with no compensation, and the meta-transaction service is disrupted.

### Finding Description

A `SignedDelegateAction` contains:
- `sender_id` (the user)
- `receiver_id` (the target contract)
- `actions` (the inner actions)
- `nonce` (the user's access-key nonce)
- `max_block_height` (expiry)
- `public_key` (the user's signing key)
- `signature` (the user's cryptographic signature over all of the above) [1](#0-0) 

The outer transaction that the relayer submits has the form:

```
Transaction {
    signer_id:   relayer,
    receiver_id: user  (= delegate_action.sender_id),
    actions:     [Action::Delegate(signed_delegate_action)],
    nonce:       relayer's nonce,
    signature:   relayer's signature,
}
```

The `apply_delegate_action` function in `runtime/runtime/src/actions.rs` verifies the inner `SignedDelegateAction`'s cryptographic signature and then calls `validate_delegate_action_key`, which reads the user's current access-key nonce, checks that `delegate_nonce > current_nonce`, and **updates the nonce** on success: [2](#0-1) [3](#0-2) [4](#0-3) 

There is **no field in `SignedDelegateAction` that binds it to a specific outer-transaction signer**. Any account can create a valid outer transaction with `receiver_id = delegate_action.sender_id` and include the same `SignedDelegateAction`. The runtime accepts it as long as the inner signature and nonce are valid.

**Attack steps:**

1. User signs a `DelegateAction` with nonce `N` and sends it off-chain to a relayer.
2. Relayer broadcasts: `Transaction { signer: relayer, receiver: user, actions: [Delegate(SDA)] }`.
3. Attacker observes this transaction in the public mempool, extracts `SDA`.
4. Attacker broadcasts their own: `Transaction { signer: attacker, receiver: user, actions: [Delegate(SDA)] }`.
5. If the attacker's transaction is processed first, the user's access-key nonce advances to `N`.
6. When the relayer's transaction is processed, `validate_delegate_action_key` finds `delegate_nonce (N) <= current_nonce (N)` and returns `DelegateActionInvalidNonce`. [3](#0-2) 

The relayer's transaction is charged gas and fails. The relayer's NEAR balance is debited with no useful work done.

### Impact Explanation

- **Corrupted protocol value:** The user's `access_key.nonce` is advanced by the attacker's transaction, making the relayer's transaction fail with `DelegateActionInvalidNonce`. The relayer's NEAR balance is reduced (gas burnt) for a failed receipt.
- **Service disruption:** A targeted attacker can monitor any relayer's submitted transactions and systematically front-run every meta-transaction, rendering the relayer's service unusable and draining its NEAR balance.
- **No financial gain required:** The attacker pays only their own gas for the front-running transaction; the user's actions still execute (via the attacker's transaction), but the relayer is griefed.

### Likelihood Explanation

NEAR does not have a gas-price auction, so ordering is not as deterministic as Ethereum. However:
- The NEAR mempool is public and transactions are gossiped to all nodes before inclusion.
- An attacker running a full node can observe pending transactions and submit a competing transaction to the same chunk producer with minimal latency.
- The attack requires no special privileges — only the ability to submit a standard signed transaction via public RPC.
- Relayers are high-value targets because they pay gas for many users; systematic griefing is economically motivated even without direct profit.

### Recommendation

Bind the `SignedDelegateAction` to the relayer's identity by including the relayer's account ID (or public key) as a field inside `DelegateAction` before signing. The runtime should then verify that the outer transaction's `signer_id` matches the bound relayer field. This makes the signed payload non-transferable to a different outer signer, eliminating the front-running vector.

Alternatively, follow the EVM pattern: if `DelegateActionInvalidNonce` is returned but the user's nonce is already `>= delegate_nonce` (meaning the action was already executed), treat the relayer's transaction as a no-op success rather than a failure, so the relayer is not penalized for a race it did not cause.

### Proof of Concept

```
// Relayer submits:
let relayer_tx = SignedTransaction::from_actions(
    relayer_nonce,
    relayer_id.clone(),
    user_id.clone(),           // receiver_id = sender_id of DelegateAction
    &relayer_signer,
    vec![Action::Delegate(Box::new(signed_delegate_action.clone()))],
    block_hash,
);

// Attacker extracts signed_delegate_action from the mempool and submits first:
let attacker_tx = SignedTransaction::from_actions(
    attacker_nonce,
    attacker_id.clone(),
    user_id.clone(),           // same receiver_id — passes sender_id check
    &attacker_signer,
    vec![Action::Delegate(Box::new(signed_delegate_action.clone()))],
    block_hash,
);
// attacker_tx is processed first → user nonce advances to N
// relayer_tx is processed second → DelegateActionInvalidNonce (nonce N <= N)
```

The `apply_delegate_action` check at line 439 only verifies `delegate_action.sender_id == account_id` (the receipt's receiver), which the attacker satisfies by setting their transaction's `receiver_id` to the user's account. [5](#0-4) [6](#0-5)

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
    if result.result.is_err() {
        // Validation failed. Need to return Ok() because this is not a runtime error.
        // "result.result" will be return to the User as the action execution result.
        return Ok(());
    }
