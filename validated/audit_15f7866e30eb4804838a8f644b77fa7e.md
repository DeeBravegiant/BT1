### Title
`OmniBridgeWormhole.initializeWormhole` Always Reverts Due to Double `initializer` Modifier Conflict — (`evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridgeWormhole.initializeWormhole` is marked `initializer` and internally calls `OmniBridge.initialize`, which is also marked `initializer`. Under OpenZeppelin Upgradeable v5 (the version in use), a nested `initializer`-to-`initializer` call always reverts with `InvalidInitialization()`. This makes `initializeWormhole` permanently uncallable, leaving the proxy uninitialized and open to a front-running takeover by an unprivileged attacker.

---

### Finding Description

`OmniBridgeWormhole` inherits `OmniBridge` and provides `initializeWormhole` as its intended entry-point initializer:

```solidity
// OmniBridgeWormhole.sol lines 32-46
function initializeWormhole(
    address tokenImplementationAddress,
    address nearBridgeDerivedAddress,
    uint8 omniBridgeChainId,
    address wormholeAddress,
    uint8 consistencyLevel
) external initializer {          // <-- sets _initialized=1, _initializing=true
    initialize(                   // <-- calls OmniBridge.initialize, also marked `initializer`
        tokenImplementationAddress,
        nearBridgeDerivedAddress,
        omniBridgeChainId
    );
    _wormhole = IWormhole(wormholeAddress);
    _consistencyLevel = consistencyLevel;
}
```

`OmniBridge.initialize` is declared `public initializer`:

```solidity
// OmniBridge.sol lines 72-86
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {   // <-- also `initializer`, not `onlyInitializing`
    ...
}
```

The OZ v5 `initializer` modifier (package `@openzeppelin/contracts-upgradeable ^5.4.0`) executes this logic:

```
bool initialSetup = initialized == 0 && isTopLevelCall;
bool construction = initialized == 1 && address(this).code.length == 0;
if (!initialSetup && !construction) revert InvalidInitialization();
```

When `initializeWormhole` runs, it sets `_initialized = 1` and `_initializing = true` before entering its body. When the body then calls `initialize`, the nested invocation sees `isTopLevelCall = false` (because `_initializing` is already `true`) and `initialized = 1`, so both `initialSetup` and `construction` are `false` — the modifier unconditionally reverts.

The correct pattern for a function meant to be called from within another `initializer` is `onlyInitializing`, not `initializer`. Every other base-contract init helper in the codebase (`__UUPSUpgradeable_init`, `__AccessControl_init`, `__Pausable_init_unchained`) correctly uses `onlyInitializing`.

---

### Impact Explanation

Because `initializeWormhole` always reverts, the proxy's `_initialized` slot is never advanced (the revert rolls back all state). The proxy remains permanently uninitialized (`_initialized == 0`). Any caller — including an unprivileged attacker — can then call the `public` `OmniBridge.initialize` directly on the proxy, supplying attacker-controlled parameters:

- `nearBridgeDerivedAddress` set to an attacker-owned key
- `DEFAULT_ADMIN_ROLE` granted to the attacker (`_grantRole(DEFAULT_ADMIN_ROLE, _msgSender())`)

With admin control and a forged `nearBridgeDerivedAddress`, the attacker can produce valid ECDSA signatures accepted by `finTransfer`, minting or releasing bridge tokens to arbitrary recipients. This satisfies **Critical — unauthorized release/withdrawal of bridge assets through verification failure**.

Even without the front-running path, if the protocol works around the revert by calling `initialize` directly and omitting a subsequent `setWormholeAddress` call, `_wormhole` remains `address(0)`. Every bridge operation that reaches an extension function (`deployTokenExtension`, `logMetadataExtension`, `finTransferExtension`, `initTransferExtension`) will call `_wormhole.publishMessage(...)` on the zero address and revert, permanently freezing the redemption path.

---

### Likelihood Explanation

`initializeWormhole` fails on every invocation without exception — this is not a race condition or edge case. The protocol will observe the revert on first deployment and must retry. During the window between the failed `initializeWormhole` transaction and any retry, the proxy is uninitialized. An attacker monitoring the mempool can submit `initialize(attacker_params)` with higher gas and take ownership before the protocol recovers. The attack requires no special privilege, no leaked key, and no chain-level assumption beyond standard mempool visibility.

---

### Recommendation

Change `OmniBridge.initialize` from `initializer` to `onlyInitializing` so it can be safely called from within `OmniBridgeWormhole.initializeWormhole`:

```solidity
// OmniBridge.sol
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public onlyInitializing {   // was: initializer
    ...
}
```

`OmniBridgeWormhole.initializeWormhole` retains the sole `initializer` modifier and becomes the true top-level entry point. This mirrors the standard OZ pattern used by every `__X_init` helper in the codebase.

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` implementation (constructor calls `_disableInitializers()` on the implementation — proxy storage is unaffected).
2. Deploy an ERC-1967 proxy pointing to the implementation.
3. Call `proxy.initializeWormhole(impl, nearAddr, chainId, wormholeAddr, level)` → **reverts with `InvalidInitialization()`** because `OmniBridge.initialize` is also `initializer`.
4. Proxy `_initialized` slot remains `0`.
5. Attacker calls `proxy.initialize(impl, attackerKey, chainId)` → succeeds; attacker holds `DEFAULT_ADMIN_ROLE` and `nearBridgeDerivedAddress = attackerKey`.
6. Attacker signs a `TransferMessagePayload` with `attackerKey` and calls `proxy.finTransfer(sig, payload)` → signature check passes, bridge mints/releases tokens to attacker.
