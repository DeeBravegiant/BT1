Audit Report

## Title
`purgeStateGuardRole` Leaves `pendingStateGuard` Intact, Enabling Unauthorized `stateGuard` Seizure After Abdication — (File: smart-contracts-poc/contracts/oracles/providers/OracleBase.sol)

## Summary
`purgeStateGuardRole` deletes `stateGuard[feedId]` but never clears `pendingStateGuard[feedId]`. A pending nominee set before the abdication can immediately call `acceptStateGuardRole` and install themselves as the new `stateGuard`, gaining exclusive control over `setPriceGuard` for that feed. Once installed, neither ADMIN nor any other party can remove them because `checkRole` locks out ADMIN when `stateGuard != address(0)`.

## Finding Description
The two-step guard-transfer flow in `OracleBase.sol` is:

- `setStateGuardRole(feedId, newGuard)` → `pendingStateGuard[feedId] = newGuard` (L99-103)
- `acceptStateGuardRole(feedId)` → `stateGuard[feedId] = msg.sender; delete pendingStateGuard` (L111-118)

The abdication path at L120-124 only executes `delete stateGuard[feedId]` and emits an event. `pendingStateGuard[feedId]` is not touched.

`acceptStateGuardRole` (L111-118) has no guard on `stateGuard`; it only checks `pendingStateGuard[feedId] == msg.sender`. After `purgeStateGuardRole` runs, the pending nominee can call `acceptStateGuardRole` and write themselves into `stateGuard[feedId]`.

After seizure, `checkRole` (L65-68) requires `stateGuard[feedId] == msg.sender` for all `setPriceGuard`, `setStateGuardRole`, `purgePendingStateGuardRole`, and `purgeStateGuardRole` calls. ADMIN has no override path while `stateGuard != address(0)`, so the attacker's control is permanent unless they voluntarily abdicate.

`purgePendingStateGuardRole` (L105-109) exists and is also gated by `checkRole`, confirming the protocol intended pending state to be clearable — but `purgeStateGuardRole` never invokes it.

## Impact Explanation
The attacker, once installed as `stateGuard`, can call `setPriceGuard(feedId, 1, type(uint128).max)`, effectively disabling the price band for that feed. With no price guard, any oracle price — including stale or manipulated values — passes the guard check and flows into `MetricOmmPool` swap execution. This satisfies the allowed impact gate: **bad-price execution** (unbounded bid/ask quote reaches a pool swap) and **admin-boundary break** (oracle role check bypassed by an unprivileged path). ADMIN cannot restore the price guard or remove the attacker without the attacker's cooperation.

## Likelihood Explanation
The trigger requires the current guard to: (1) nominate a pending guard via `setStateGuardRole`, and (2) later call `purgeStateGuardRole` to abdicate. This is a realistic operational sequence — a guard proposes a successor, then reconsiders and decides to fully step down. The attacker (the pending nominee) can front-run any ADMIN `purgePendingStateGuardRole` call in the same block as the abdication, making the window exploitable even with a fast ADMIN response. The exploit is repeatable across any feed where this sequence occurs.

## Recommendation
Clear `pendingStateGuard` inside `purgeStateGuardRole`:

```solidity
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];
+   delete pendingStateGuard[feedId];   // prevent stale nominee from claiming
    emit StateGuardDeleted(feedId);
}
```

## Proof of Concept
```
1. ADMIN calls setStateGuardRole(feedId, guard1)
2. guard1 calls acceptStateGuardRole(feedId)
   → stateGuard[feedId] = guard1

3. guard1 calls setStateGuardRole(feedId, attacker)
   → pendingStateGuard[feedId] = attacker

4. guard1 decides to abdicate:
   guard1 calls purgeStateGuardRole(feedId)
   → stateGuard[feedId] = address(0)
   → pendingStateGuard[feedId] = attacker  ← NOT cleared

5. attacker calls acceptStateGuardRole(feedId)
   → pendingStateGuard[feedId] == attacker ✓
   → stateGuard[feedId] = attacker         ← unauthorized seizure

6. attacker calls setPriceGuard(feedId, 1, type(uint128).max)
   → price guard disabled; any oracle price accepted
   → pool swaps execute at unbounded prices

7. ADMIN cannot call purgeStateGuardRole or setPriceGuard:
   checkRole requires stateGuard[feedId] == msg.sender,
   but stateGuard[feedId] == attacker, not ADMIN
   → attacker control is permanent
```
