### Title
Unvalidated `builder` Field in `ProposalInit` Allows Proposer to Inject Arbitrary `sequencer_address` into Block Hash and Execution Context — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates many fields of a received `ProposalInit` (height, L1/L2 gas prices, `l1_da_mode`, `starknet_version`, `version_constant_commitment`, `fee_proposal_fri`) but never checks the `builder` field. That field is unconditionally forwarded as `sequencer_address` into `BlockInfo` by `convert_to_sn_api_block_info`, which is then used both in the block-hash Poseidon chain and as the value returned to contracts by the `get_sequencer_address` syscall. A malicious-but-legitimately-selected proposer can therefore inject an arbitrary `sequencer_address` that every validator will silently accept, producing a wrong block hash and wrong syscall results for all transactions in that block.

---

### Finding Description

**Unvalidated field in `ProposalInit`**

`ProposalInit` carries a `builder` field described as "Address of the one who builds/sequences the block." [1](#0-0) 

`is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, all four L1 gas-price fields, and `fee_proposal_fri`. It never checks `builder`: [2](#0-1) 

`ProposalInitValidation` — the struct that carries the validator's reference values — has no `builder` field at all, so there is no reference value to compare against: [3](#0-2) 

**`builder` becomes `sequencer_address` in `BlockInfo`**

`convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address`: [4](#0-3) 

This `BlockInfo` is passed to the batcher for both the proposer path (`propose_block`) and the validator path (`validate_block`): [5](#0-4) 

**`sequencer_address` enters the block hash**

The Starknet OS block-hash function hashes `block_info.sequencer_address` as one of its inputs: [6](#0-5) 

**`sequencer_address` is returned by the `get_sequencer_address` syscall**

The OS hint that populates the Cairo `BlockInfo` struct for execution reads `sequencer_address` from the same `block_info`: [7](#0-6) 

**Proposer sets `builder` from its own config; validators never cross-check it**

On the proposer side, `builder` is set from `args.builder_address` (a local config value): [8](#0-7) 

On the validator side, `is_proposal_init_valid` is called with a `ProposalInitValidation` that contains no `builder` reference, so the field passes through unchecked: [9](#0-8) 

---

### Impact Explanation

A proposer that wins a consensus round can set `init.builder` to any `ContractAddress` value. Every validator will:

1. Accept the `ProposalInit` (no check on `builder`).
2. Build a `BlockInfo` with `sequencer_address = attacker_chosen_address`.
3. Execute all transactions in that block with the wrong `sequencer_address` visible to contracts via `get_sequencer_address`.
4. Commit a block whose Poseidon hash includes the wrong `sequencer_address`, producing a permanently wrong block hash on-chain.

Concrete consequences:
- **Wrong block hash** committed to storage and propagated to L1 — the block hash is a Poseidon over `sequencer_address` among other fields.
- **Wrong `get_sequencer_address` syscall result** for every transaction in the block — any contract that branches on the sequencer address (e.g., fee-token contracts, access-control logic) will behave incorrectly.
- **Fee misdirection** — protocol-level fee collection that routes to `sequencer_address` will credit the wrong account.

This matches the allowed impact: *"Wrong state, receipt, event … or revert result from blockifier/syscall/execution logic for accepted input"* and *"RPC execution … returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

The trigger requires the attacker to be the elected proposer for a round, which is a role assigned by the Tendermint consensus layer to legitimate validator nodes. In the current single-sequencer deployment this is always the same node, but as the network moves toward decentralized consensus any validator node that wins a round can exploit this. No external user interaction is needed; the malicious node simply sets `builder` to an arbitrary value before broadcasting `ProposalInit`.

---

### Recommendation

Add `builder` (the expected sequencer/builder address) to `ProposalInitValidation` and check it inside `is_proposal_init_valid`, analogous to how `l1_da_mode` and `l2_gas_price_fri` are checked:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid:
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The reference value for `builder` should be derived from the node's own configured `builder_address` (the same source used by the proposer), stored in `SequencerConsensusContext` and forwarded into `ProposalInitValidation` at both call sites in `sequencer_consensus_context.rs`. [10](#0-9) [11](#0-10)

### Proof of Concept

1. A validator node is selected as proposer for round R at height H.
2. It constructs `ProposalInit` with `builder = ContractAddress(0xdead)` instead of its real address.
3. It broadcasts the `ProposalInit` to all peers.
4. Each peer calls `validate_proposal` → `is_proposal_init_valid`. The function checks `height`, `l1_da_mode`, `l2_gas_price_fri`, gas prices, `starknet_version`, `version_constant_commitment`, and `fee_proposal_fri` — all of which are correct. `builder` is never checked. The proposal is accepted.
5. Each validator calls `convert_to_sn_api_block_info(init)` which sets `sequencer_address = 0xdead` in `BlockInfo`.
6. The blockifier executes all transactions with `sequencer_address = 0xdead`; any `get_sequencer_address` syscall returns `0xdead`.
7. `calculate_block_hash` hashes `0xdead` as `block_info.sequencer_address`, producing a block hash that differs from what an honest proposer would have produced.
8. Consensus reaches agreement on this corrupted block; the wrong block hash is stored and eventually anchored to L1.
