Audit Report

## Title
Stale `rsETHPrice` at `initiateWithdrawal` permanently caps user payout below fair value — (`contracts/LRTWithdrawalManager.sol`)

## Summary

`initiateWithdrawal` computes `expectedAssetAmount` using the cached `lrtOracle.rsETHPrice()` without triggering a price refresh. Because `_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`, a stale (lower-than-true) price at initiation time permanently caps the user's payout at the stale amount even after the oracle is later updated. The rsETH is burned at `unlockQueue` time and the user receives fewer underlying assets than their rsETH is currently worth.

## Finding Description

`getExpectedAssetAmount` reads the stored `rsETHPrice` directly without refreshing it:

```solidity
// LRTWithdrawalManager.sol:593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [1](#0-0) 

This value is stored as `request.expectedAssetAmount` and added to `assetsCommitted[asset]`: [2](#0-1) 

When the operator later calls `unlockQueue`, `_calculatePayoutAmount` computes:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
``` [3](#0-2) 

If `rsETHPrice` was stale (lower) at initiation: `expectedAssetAmount` = `rsETHUnstaked × stalePrice / assetPrice` (lower), `currentReturn` = `rsETHUnstaked × truePrice / assetPrice` (higher), and `min(lower, higher)` = `lower`. The request's `expectedAssetAmount` is overwritten with this capped value at L804: [4](#0-3) 

And transferred at `completeWithdrawal`: [5](#0-4) 

`updateRSETHPrice()` is public but never called automatically within the withdrawal initiation path: [6](#0-5) 

There is no staleness guard anywhere in `initiateWithdrawal`. The loss is irreversible once the request is queued because `expectedAssetAmount` is the binding cap used at unlock time.

## Impact Explanation

A user who calls `initiateWithdrawal` while `rsETHPrice` has not been refreshed (e.g., after yield accrual with no keeper update) will have their payout permanently capped at the stale lower amount. Their rsETH is burned and they receive fewer underlying assets than the rsETH is currently worth. No protocol funds are lost, but the user fails to receive the returns their rsETH has accrued. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation

Any unprivileged user calling `initiateWithdrawal` during a window where the off-chain keeper has not called `updateRSETHPrice()` is silently harmed. The user has no on-chain indication that the price is stale. While `updateRSETHPrice()` is permissionless and any user *can* call it first, the contract provides no enforcement or revert path to protect users who do not know to do so. The condition (stale oracle) is a normal operational state, not an edge case.

## Recommendation

Call `updateRSETHPrice()` (or an equivalent internal `_updateRsETHPrice()`) at the start of `initiateWithdrawal` before computing `expectedAssetAmount`, or add a staleness check that reverts if `rsETHPrice` has not been updated within an acceptable window (e.g., 24 hours). This ensures the committed amount always reflects the current fair value of rsETH.

## Proof of Concept

```solidity
function test_stalePrice_locksLowerPayout() public {
    uint256 stalePrice = 1.00e18;
    uint256 truePrice  = 1.05e18; // 5% yield accrued, not yet reflected
    uint256 assetPrice = 1.00e18;

    mockOracle.setRsETHPrice(stalePrice); // oracle NOT updated

    uint256 rsETHAmount = 1e18;
    vm.prank(user);
    withdrawalManager.initiateWithdrawal(asset, rsETHAmount, "");
    // expectedAssetAmount stored = 1.00e18

    mockOracle.setRsETHPrice(truePrice); // oracle updated after initiation

    withdrawalManager.unlockQueue(asset, type(uint256).max, 0, 0, type(uint256).max, type(uint256).max);
    // _calculatePayoutAmount: currentReturn = 1.05e18, min(1.00e18, 1.05e18) = 1.00e18

    vm.prank(user);
    withdrawalManager.completeWithdrawal(asset, "");

    assertEq(asset.balanceOf(user), 1.00e18); // received stale amount, not fair 1.05e18
}
```
