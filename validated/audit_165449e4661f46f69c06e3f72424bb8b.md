Audit Report

## Title
`LRTOracle` lacks `MINTER_ROLE` and `PAUSER_ROLE`, breaking fee minting and auto-pause - (File: contracts/LRTOracle.sol)

## Summary

`LRTOracle._updateRsETHPrice()` calls `RSETH.mint()` and `LRTDepositPool.pause()` / `LRTWithdrawalManager.pause()`, but both target functions enforce role checks against `msg.sender` (i.e., `LRTOracle`). `LRTOracle.initialize()` grants no roles to itself, and no deployment mechanism ensures these roles are present. When `protocolFeeInETH > 0`, the fee-mint call reverts and the price oracle stalls, permanently withholding protocol fees. When the rsETH price drops beyond `pricePercentageLimit`, the auto-pause call reverts, leaving the protocol unpaused and operating on a stale price.

## Finding Description

**Root cause:** `LRTOracle._updateRsETHPrice()` makes two privileged cross-contract calls without holding the required roles.

**Path 1 – Fee minting (lines 299–308, `LRTOracle.sol`):**

```solidity
if (rsethAmountToMintAsProtocolFee > 0) {
    address treasury = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
    IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee);
```

`RSETH.mint()` is guarded by `onlyRole(LRTConstants.MINTER_ROLE)` (line 234, `RSETH.sol`). The `onlyRole` modifier in `LRTConfigRoleChecker` resolves to:

```solidity
if (!IAccessControl(address(lrtConfig)).hasRole(role, msg.sender)) { revert ... }
```

When `LRTOracle` calls `RSETH.mint()`, `msg.sender` is `LRTOracle`. If `LRTOracle` does not hold `MINTER_ROLE` in `LRTConfig`, the call reverts and the entire `updateRSETHPrice()` transaction reverts — the price is never updated and fees are never minted.

**Path 2 – Auto-pause on price drop (lines 277–282, `LRTOracle.sol`):**

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

`LRTDepositPool.pause()` (line 349) and `LRTWithdrawalManager.pause()` (line 347) are both guarded by `onlyRole(LRTConstants.PAUSER_ROLE)`. If `LRTOracle` lacks `PAUSER_ROLE`, the first `pause()` call reverts, the entire transaction reverts, the protocol is never paused, and the price is never updated.

**Why existing checks are insufficient:** `LRTOracle.initialize()` only sets `lrtConfig` — it grants no roles. There is no `reinitializer`, deployment script hook, or comment mandating that `MINTER_ROLE` and `PAUSER_ROLE` be granted to `LRTOracle`. The natural deployment default omits these grants.

## Impact Explanation

**Theft of unclaimed yield (High):** Whenever `protocolFeeInETH > 0` (i.e., TVL grew and `protocolFeeInBPS > 0`), `updateRSETHPrice()` reverts at the `RSETH.mint()` call. Protocol fees are never minted to the treasury. The price oracle stalls on a stale value. This constitutes permanent withholding of protocol fee yield — matching the "Theft of unclaimed yield" impact class.

**Temporary freezing of funds (Medium):** When the rsETH price drops beyond `pricePercentageLimit`, the auto-pause branch reverts instead of pausing. The protocol remains unpaused and the price is not updated. Users continue depositing and withdrawing against a stale (inflated) price, exposing them to loss — matching the "Temporary freezing of funds" impact class.

## Likelihood Explanation

`MINTER_ROLE` is conventionally granted to `LRTDepositPool` (the deposit contract). `PAUSER_ROLE` is conventionally granted to human operator addresses. Neither role is naturally associated with `LRTOracle`. The `initialize()` function provides no self-grant. Any deployment that follows the natural pattern — granting `MINTER_ROLE` to `LRTDepositPool` and `PAUSER_ROLE` to operators — will silently omit these grants for `LRTOracle`. The public `updateRSETHPrice()` function can be called by any external actor, making the failure immediately observable and repeatable.

## Recommendation

Grant `MINTER_ROLE` and `PAUSER_ROLE` to `LRTOracle` in `LRTConfig` during deployment. To make this robust and self-documenting, add explicit role grants inside `LRTOracle.initialize()` (if `LRTConfig` is already deployed at that point) or enforce them in the deployment script. Alternatively, refactor `RSETH.mint()`, `LRTDepositPool.pause()`, and `LRTWithdrawalManager.pause()` to accept `LRTOracle` as an authorized caller via a dedicated modifier (e.g., `onlyLRTOracle`), analogous to the `onlyLRTNodeDelegator` pattern in `LRTUnstakingVault`.

## Proof of Concept

1. Deploy `LRTConfig`, `RSETH`, `LRTDepositPool`, `LRTWithdrawalManager`, `LRTOracle` without granting `MINTER_ROLE` or `PAUSER_ROLE` to `LRTOracle` (natural default).
2. Set `protocolFeeInBPS > 0` in `LRTConfig`; allow TVL to grow so `totalETHInProtocol > previousTVL`, making `protocolFeeInETH > 0`.
3. Call `LRTOracle.updateRSETHPrice()`.
4. Transaction reverts at `IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee)` with `CallerNotLRTConfigAllowedRole` — `LRTOracle` lacks `MINTER_ROLE`. Price is not updated; fees are not minted.
5. Separately, set `pricePercentageLimit` to a non-zero value and simulate a price drop beyond the threshold (e.g., reduce TVL so `newRsETHPrice < highestRsethPrice - threshold`).
6. Call `LRTOracle.updateRSETHPrice()`.
7. Transaction reverts at `lrtDepositPool.pause()` with `CallerNotLRTConfigAllowedRole` — `LRTOracle` lacks `PAUSER_ROLE`. Protocol is not paused; price is not updated; users continue operating on stale price.
