### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Deriving It From the Pending Request — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never checks that `response.payload_hash` is actually derived from the content of the pending `request`. A Byzantine attested participant below the signing threshold can replay any previously observed valid `(payload_hash, signature)` pair against a new pending request, causing the contract to resolve that request with a forged foreign-chain verification result.

---

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` accepts two caller-supplied arguments: a `VerifyForeignTransactionRequest` (used as the map key to locate the pending yield) and a `VerifyForeignTransactionResponse` (containing `payload_hash` and `signature`).

The only cryptographic check performed is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It does **not** verify that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` — i.e., that the hash is actually derived from the pending request's content and the extracted foreign-chain values.

The correct hash structure is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does enforce this check — it recomputes the expected hash from `(request, expected_extracted_values)` and compares it to `response.payload_hash`:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [3](#0-2) 

But this check lives in the user-side SDK, not in the MPC contract itself. The MPC contract never enforces it on-chain.

---

### Impact Explanation

A Byzantine attested participant can:

1. Monitor the NEAR blockchain for any previously accepted `respond_verify_foreign_tx` transaction. The `payload_hash` and `signature` are public on-chain data.
2. When a new request `R2` is pending in `pending_verify_foreign_tx_requests`, call `respond_verify_foreign_tx(request=R2, response={payload_hash=H_old, signature=sig_old})` where `H_old` and `sig_old` are from a completely different prior foreign-chain verification.
3. The contract accepts the call because `sig_old` is a valid ECDSA signature over `H_old` under the root public key.
4. The contract resolves the yield for `R2` with `serde_json::to_vec(&response)` — delivering `{payload_hash: H_old, signature: sig_old}` to the user who submitted `R2`. [4](#0-3) 

The user's contract receives a `VerifyForeignTransactionResponse` that appears cryptographically valid (the signature checks out against the root key) but whose `payload_hash` corresponds to a completely different foreign-chain transaction and set of extracted values. Any user contract that does not independently re-derive and compare the expected hash — or that trusts the `payload_hash` field as the contract's attestation of what was verified — will accept forged foreign-chain state as genuine. This enables invalid bridge execution: a bridge contract could release funds or execute state transitions based on fabricated foreign-chain evidence.

---

### Likelihood Explanation

The attacker must be an attested MPC participant (requires a valid TEE attestation). However, the attack requires only a **single** Byzantine participant — no threshold cooperation is needed. The required `(payload_hash, signature)` material is fully public on-chain from any prior accepted `respond_verify_foreign_tx` call. The attacker does not need to forge a new signature; they only need to replay an existing one against a new pending request. This is a realistic threat from any compromised or malicious node that has passed TEE attestation.

---

### Recommendation

The contract should derive the expected `payload_hash` from the `request` and the extracted values supplied in the response, and reject any response where the hash does not match. Since the contract does not currently receive the extracted values, the response structure should be extended to include them:

```rust
pub struct VerifyForeignTransactionResponse {
    pub extracted_values: Vec<ExtractedValue>, // NEW: include extracted values
    pub signature: SignatureResponse,
}
```

The contract then computes:
```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),
}).compute_msg_hash()?;
```

And verifies the signature against `expected_hash` rather than a caller-supplied `payload_hash`. This mirrors the check already present in `ForeignChainSignatureVerifier::verify_signature` in the SDK, bringing it on-chain where it cannot be bypassed.

---

### Proof of Concept

1. User submits `verify_foreign_transaction(R1)` for Bitcoin tx `0xAA...`. MPC nodes verify it, sign `H1 = SHA-256(borsh({request: R1, values: [BlockHash(0xBB...)]}))`, and call `respond_verify_foreign_tx(request=R1, response={payload_hash: H1, sig: sig1})`. This is accepted and `H1, sig1` are now public on-chain.

2. A second user submits `verify_foreign_transaction(R2)` for a different Bitcoin tx `0xCC...`. The expected hash for an honest response would be `H2 = SHA-256(borsh({request: R2, values: [BlockHash(0xDD...)]}))`.

3. A Byzantine attested participant calls `respond_verify_foreign_tx(request=R2, response={payload_hash: H1, signature: sig1})`.

4. The contract at lines 726–734 verifies `verify_ecdsa_signature(sig1, H1, root_pk)` — this passes because `sig1` is a valid signature over `H1`. [5](#0-4) 

5. `resolve_yields_for` resolves the pending yield for `R2` with `{payload_hash: H1, signature: sig1}`. [4](#0-3) 

6. The second user's contract receives `{payload_hash: H1, signature: sig1}` — a response that is cryptographically valid but corresponds to a completely different foreign-chain transaction. If the user contract does not re-derive the expected hash from `R2`'s content and compare it to `H1`, it will accept this as a genuine verification of `R2`.
