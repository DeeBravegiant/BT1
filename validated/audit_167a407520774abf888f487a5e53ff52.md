Audit Report

## Title
Missing Staleness Check on Chainlink `latestRoundData()` Allows Stale Prices to Corrupt rsETH Exchange Rate - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards the `updatedAt` timestamp, accepting arbitrarily stale prices without validation. Because `LRTOracle.updateRSETHPrice()` is a permissionless `public` function, any caller can trigger an rsETH price update using a stale Chainlink price, causing the global `rsETHPrice` to be set incorrectly and directly harming depositors or existing rsETH holders.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `latestRoundData()` is called and only `price` is captured; `updatedAt` is silently dropped:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

This stale price flows into `LRTOracle._getTotalEthInProtocol()` (L339), which sums the ETH-denominated value of all supported LSTs using the stale exchange rate. The result directly sets `newRsETHPrice` (L250) and is written to the global `rsETHPrice` state variable (L313).

The `pricePercentageLimit` guard is insufficient: it only compares `newRsETHPrice` against `highestRsethPrice`. A stale price that falls between the actual current price and the historical peak passes both the upside check (no revert) and the downside check (no pause), silently corrupting the exchange rate. The permissionless `updateRSETHPrice()` (L87) means any unprivileged caller can deliberately trigger this at any time, including during a known stale window.

**Exploit path (stale price lower than actual — depositor dilution of existing holders):**
1. A Chainlink LST/ETH feed goes stale; the actual LST price has risen but `updatedAt` is hours old.
2. Attacker calls `LRTOracle.updateRSETHPrice()`.
3. `_getTotalEthInProtocol()` uses the stale (deflated) price → `totalETHInProtocol` is understated → `newRsETHPrice` is set below true value.
4. Attacker immediately deposits assets and mints rsETH at the deflated rate, receiving more rsETH than their deposit is worth.
5. When the feed recovers and price is updated upward, the attacker's excess rsETH is now worth more, diluting all existing holders.

**Exploit path (stale price higher than actual — depositor harm):**
1. LST price has dropped but feed is stale at the old higher value.
2. Attacker calls `updateRSETHPrice()` → `rsETHPrice` is inflated.
3. New depositors receive fewer rsETH than deserved for their deposit.
4. When feed recovers, `rsETHPrice` corrects downward, and the attacker (holding rsETH from before the stale update) profits at the expense of new depositors.

## Impact Explanation
**Critical — Direct theft of any user funds.** In the deflated-price scenario, an attacker mints excess rsETH at below-true-value pricing and extracts that value from existing holders when the price corrects. In the inflated-price scenario, new depositors are systematically shortchanged. Both paths represent direct, concrete fund loss to protocol participants, not merely a theoretical accounting error. The `pricePercentageLimit` guard does not prevent moderate staleness from passing silently.

## Likelihood Explanation
Chainlink ETH-denominated LST feeds (stETH/ETH, ETHx/ETH) have heartbeats of up to 24 hours, meaning a feed can be legally stale for up to 24 hours before Chainlink's own circuit breaker fires. `updateRSETHPrice()` is callable by any address with no access control. An attacker monitoring mempool or Chainlink feed timestamps can deterministically identify stale windows and call the function at will. No special privileges, governance capture, or external protocol compromise is required — only a public function call.

## Recommendation
Add a configurable `maxStaleness` parameter per asset in `ChainlinkPriceOracle` and validate `updatedAt` in `getAssetPrice()`:

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
if (block.timestamp - updatedAt > maxStaleness[asset]) revert StalePriceFeed();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Set `maxStaleness` per asset to the Chainlink feed's documented heartbeat plus a small buffer (e.g., heartbeat + 1 hour). Also consider adding a `answeredInRound >= roundId` check to guard against incomplete rounds.

## Proof of Concept
**Foundry fork test outline:**
1. Fork mainnet at a block where a supported LST Chainlink feed is near its heartbeat boundary.
2. `vm.warp(block.timestamp + heartbeat + 1)` to simulate a stale feed without a new round.
3. Record `rsETHPrice` before the call.
4. Call `lrtOracle.updateRSETHPrice()` as an unprivileged address (no role required).
5. Assert `rsETHPrice` has been set using the stale price (verify against a mock aggregator returning an `updatedAt` that is `heartbeat + 1` seconds old).
6. Deposit assets as the attacker at the incorrect rate; verify rsETH minted exceeds the fair share.
7. Advance time, allow feed to update, call `updateRSETHPrice()` again, and assert existing holder NAV has decreased — confirming value extraction.

The minimal reproducible call sequence requires no privileged roles: only `ChainlinkPriceOracle.getAssetPrice()` (L49–55) and `LRTOracle.updateRSETHPrice()` (L87–89) called by `address(this)`.
