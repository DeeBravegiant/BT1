### Title
Missing Value Conservation Check in Dijkstra Sub-Transaction Validation Allows Arbitrary ADA/Asset Creation - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era introduces sub-transactions (`SubTx`) embedded within a top-level transaction. The top-level `UTXO` rule enforces value conservation (`consumed == produced`) only for the top-level transaction body. The `SUBUTXO` rule that validates each sub-transaction body contains no corresponding `validateValueNotConservedUTxO` check. An unprivileged transaction sender can craft a sub-transaction whose outputs exceed its inputs, creating ADA or native assets out of thin air, while the top-level value conservation check passes unaffected.

---

### Finding Description

In `dijkstraUtxoTransition`, value conservation is enforced for the top-level transaction body against `originalUtxo`:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
```

`txBody` here is the **top-level** transaction body only. Sub-transactions are processed separately via the `UTXOS → SUBUTXOW → SUBUTXO` chain.

In `dijkstraSubUtxoTransition`, the full list of validation checks is:

- `validateOutsideValidityIntervalUTxO`
- `validateOutsideForecast`
- `validateOutputTooBigUTxO`
- `validateInputSetEmptyUTxO`
- `validateBadInputsUTxO` (inputs exist in `originalUtxo`)
- `validateBadInputsUTxO` (inputs exist in current UTxO state, preventing double-spend)
- `validateOutputBootAddrAttrsTooBig`
- `validateOutputTooSmallUTxO`
- `validateWrongNetwork`, `validateWrongNetworkWithdrawal`, `validateWrongNetworkInDirectDeposit`, `validateWrongNetworkInTxBody`

**There is no `validateValueNotConservedUTxO` call.** After these checks, `Shelley.updateUTxOStateNoFees` is called unconditionally (when `isValid = True`), which removes the sub-transaction's inputs from the UTxO and adds its outputs — with no check that the values balance.

The function `dijkstraSubTxProducedValue` exists in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` and computes the produced value for a sub-transaction, but it is never invoked in any validation rule.

The top-level value conservation check cannot compensate for this omission: it uses `originalUtxo` and the top-level `txBody` only. If a sub-transaction spends 1 ADA from `originalUtxo` and creates 1,000,000 ADA in outputs, the top-level check sees neither the sub-transaction's inputs nor its outputs, and passes normally.

---

### Impact Explanation

**Critical — Direct creation of ADA or native assets through an invalid ledger state transition.**

An attacker submits a Dijkstra-era transaction containing a sub-transaction whose outputs exceed its inputs by an arbitrary amount. The sub-transaction passes all validation checks (inputs exist, outputs are well-formed, etc.) and `updateUTxOStateNoFees` adds the inflated outputs to the UTxO. The top-level transaction's value conservation check is satisfied independently. The excess value persists in the UTxO as spendable funds, constituting direct, unbounded creation of ADA or native assets.

---

### Likelihood Explanation

Any unprivileged transaction sender can construct and submit such a transaction. No special role, key, or governance threshold is required. The Dijkstra era is the current development target, so this affects the next production era. The attack requires only knowledge of the sub-transaction format and the ability to submit a valid-looking transaction.

---

### Recommendation

Add a value conservation check inside `dijkstraSubUtxoTransition`, analogous to the check in the top-level rule. The existing `dijkstraSubTxProducedValue` helper in `Cardano.Ledger.Dijkstra.UTxO` already computes the produced side for sub-transactions; a corresponding consumed-value computation and equality assertion should be added:

```haskell
-- In dijkstraSubUtxoTransition, after input validation:
runTest $ validateSubTxValueNotConserved pp originalUtxo certState txBody
```

where `validateSubTxValueNotConserved` mirrors `Shelley.validateValueNotConservedUTxO` but uses the sub-transaction body and accounts for the sub-transaction's mint field, withdrawals, deposits, and treasury donation.

---

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction `txTop` with:
   - A single spend input `txIn_top` (e.g., 10 ADA) from `originalUtxo`
   - A single output `txOut_top` (10 ADA minus fee)
   - One sub-transaction `txSub` with:
     - Spend input `txIn_sub` (1 ADA) from `originalUtxo`
     - Output `txOut_sub` (1,000,000 ADA) — value exceeds input by 999,999 ADA

2. Submit `txTop`. The ledger processes:
   - `dijkstraUtxoTransition`: checks `consumed(txTop) == produced(txTop)` using `originalUtxo` and `txTop.body` — passes (10 ADA in, 10 ADA out minus fee).
   - `dijkstraSubUtxoTransition` for `txSub`: checks inputs exist, outputs are well-formed — passes. Calls `updateUTxOStateNoFees`, adding `txOut_sub` (1,000,000 ADA) to the UTxO.

3. After the block is applied, `txOut_sub` containing 1,000,000 ADA is a valid UTxO entry, spendable by the attacker. 999,999 ADA were created from nothing. [1](#0-0) [2](#0-1) [3](#0-2)

dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG

  let txBody = tx ^. bodyTxL

  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  let allSizedOutputs = txBody ^. allSizedOutputsTxBodyF
  let allOutputs = fmap sizedValue allSizedOutputs
  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs

  runTest $ Shelley.validateInputSetEmptyUTxO txBody

  let inputs = txBody ^. inputsTxBodyL
  let refInputs = txBody ^. referenceInputsTxBodyL
  runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs

  runTestOnSignal $ Shelley.validateOutputBootAddrAttrsTooBig allOutputs

  runTestOnSignal $ Babbage.validateOutputTooSmallUTxO pp allSizedOutputs

  netId <- liftSTS $ asks networkId
  runTestOnSignal $ Shelley.validateWrongNetwork netId allOutputs
  runTestOnSignal $ Shelley.validateWrongNetworkWithdrawal netId txBody
  runTestOnSignal $ validateWrongNetworkInDirectDeposit netId txBody
  runTestOnSignal $ Alonzo.validateWrongNetworkInTxBody netId txBody

  if isValid
    then do
      newState <-
        Shelley.updateUTxOStateNoFees
          pp
          utxoState
          txBody
          certState
          (utxosGovState utxoState)
          (tellEvent . TotalDeposits (hashAnnotated txBody))
          (\a b -> tellEvent $ TxUTxODiff a b)
      pure $ newState & utxosDonationL <>~ txBody ^. treasuryDonationTxBodyL
