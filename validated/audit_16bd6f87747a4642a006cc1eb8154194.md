### Title
Harvester Can Unconditionally Redirect Farmer Block Rewards to Arbitrary Address Without Enforcement - (`chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

### Summary

A connected harvester can set `farmer_reward_address_override` to any arbitrary puzzle hash in a `NewProofOfSpace` or `RespondSignatures` message. The farmer's `FarmerAPI` accepts this override unconditionally and uses it as the `farmer_reward_puzzle_hash` in the block foliage, permanently redirecting the farmer's XCH block reward. The CHIP-22 fee quality convention is only logged as a warning — it is never enforced. No farmer consent or threshold check blocks the diversion.

### Finding Description

CHIP-22 introduced `farmer_reward_address_override` in `NewProofOfSpace` and `RespondSignatures` to allow third-party harvesters to take a fee by redirecting the farmer reward. The intended convention is that a harvester may only redirect the reward when the proof's "fee quality" meets a threshold it declares in `fee_info.applied_fee_threshold`.

In `FarmerAPI.new_proof_of_space()`, when `farmer_reward_address_override` is not `None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [1](#0-0) 

That function only logs a warning when the convention is violated — it never returns an error, raises an exception, or prevents the override from being applied:

```python
else:
    self.log.warning(
        "Harvester illegitimately took reward by failing to provide its fee rate ..."
    )
``` [2](#0-1) 

Execution then continues into `_process_respond_signatures()`, where the override is unconditionally accepted:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
``` [3](#0-2) 

This `farmer_reward_address` is then placed into `DeclareProofOfSpace`, which the full node uses directly as `farmer_ph` when constructing the block foliage:

```python
farmer_ph = request.farmer_puzzle_hash
``` [4](#0-3) 

The full node then creates the farmer reward coin at that puzzle hash via `create_farmer_coin()`: [5](#0-4) 

Block header validation only checks the pool target for contract plots (rule 20c) and the pool signature for pool-key plots (rule 20b). There is no consensus-level check that `farmer_reward_puzzle_hash` matches the farmer's configured address: [6](#0-5) 

The `NewProofOfSpace` protocol message explicitly carries `farmer_reward_address_override: bytes32 | None`, which any connected harvester can set to any value: [7](#0-6) 

Similarly, `RespondSignatures` carries the same field: [8](#0-7) 

### Impact Explanation

A malicious harvester connected to a farmer can redirect every farmer block reward (currently ~0.25 XCH per block) to an attacker-controlled address. The farmer's configured `farmer_target` is silently bypassed. The farmer receives only a log warning, which may go unnoticed. This is a direct, permanent, on-chain XCH payout redirection with no recovery path once the block is confirmed.

This matches the allowed High impact: "Bypass of wallet, daemon, keychain, RPC, pool, or Data Layer authorization that enables unauthorized signing, key use, coin control, **payout redirection**."

### Likelihood Explanation

Any harvester connected to the farmer — including third-party harvesters explicitly supported by CHIP-22 — can trigger this. No keys, admin access, or cryptographic break is required. The attacker only needs a network connection to the farmer's harvester port. In third-party harvester deployments (e.g., compressed plot services), this is a realistic and reachable attack surface.

### Recommendation

1. **Enforce the fee quality convention**: In `FarmerAPI.new_proof_of_space()` and `_process_respond_signatures()`, reject (return `None`) any `farmer_reward_address_override` where `fee_quality > applied_fee_threshold` or where `fee_info` is absent.
2. **Whitelist override addresses**: Allow farmers to configure a set of permitted harvester fee addresses; reject overrides to unlisted addresses.
3. **Cap the override**: Enforce that the override address is the harvester's registered fee address, not an arbitrary puzzle hash.
4. **Audit logging**: At minimum, emit a structured event (not just a log line) that wallet/monitoring tooling can act on when a reward is redirected.

### Proof of Concept

1. Attacker operates a third-party harvester service (e.g., a compressed-plot provider).
2. Farmer connects their node to the attacker's harvester.
3. When the harvester finds a winning proof, it sends `NewProofOfSpace` with `farmer_reward_address_override = attacker_puzzle_hash` and either omits `fee_info` or sets `applied_fee_threshold = 0xFFFFFFFF`.
4. `FarmerAPI.new_proof_of_space()` calls `notify_farmer_reward_taken_by_harvester_as_fee()`, which logs a warning and returns — execution continues.
5. `_process_respond_signatures()` sets `farmer_reward_address = attacker_puzzle_hash`.
6. `DeclareProofOfSpace` is sent to the full node with `farmer_puzzle_hash = attacker_puzzle_hash`.
7. The full node constructs the block with `create_farmer_coin(height, attacker_puzzle_hash, reward, genesis_challenge)`.
8. The block is accepted by consensus (no rule checks `farmer_reward_puzzle_hash` against the farmer's config).
9. The farmer reward coin is created at `attacker_puzzle_hash`. The farmer receives nothing.
