## Detailed Code Trace

The question identifies a real gap. Here is the exact evidence:

**`blob_to_keyset` in `src/cbmpc/api/hd_keyset_ecdsa_2p.cpp` (lines 98–100):** [1](#0-0) 


No `icurve.check()` is called on either `root_Q` or `root_K`.

**Compare with the analogous `blob_to_key` in `src/cbmpc/api/ecdsa2pc.cpp` (lines 81–82):** [2](#0-1) 

```cpp

```

The `ecdsa2pc` blob deserializer explicitly calls `curve.check()` after `from_bin`. The HD keyset deserializer does not.

## What `from_bin` Does vs. What `check` Does

**`ecurve_ossl_t::from_bin` (P256 backend) explicitly accepts the infinity encoding:** [3](#0-2) 

If `bin[0] == 0` and all bytes are zero (33-byte all-zero for P256), it sets `bin.size = 1` and calls `EC_POINT_oct2point`, which OpenSSL interprets as the point at infinity. Returns `SUCCESS`.

**`ecurve_t::check` explicitly rejects infinity:** [4](#0-3) 

```cpp
if (point.is_infinity()) return crypto::error("EC-point is infinity");
```

**`ecurve_secp256k1_t::from_bin` (secp256k1 backend):** [5](#0-4) 

`secp256k1_eckey_pubkey_parse` only accepts 33-byte (0x02/0x03) or 65-byte (0x04) encodings. An all-zero 33-byte infinity encoding is rejected. **The attack does not work for secp256k1.**

## Verdict

### Title
Missing `icurve.check()` on `root_Q`/`root_K` in `blob_to_keyset` allows P256 infinity-point injection — (`src/cbmpc/api/hd_keyset_ecdsa_2p.cpp`)

### Summary
`blob_to_keyset` calls `from_bin()` on `root_Q_compressed` and `root_K_compressed` but never calls `icurve.check()` on the resulting points. For P256, `ecurve_ossl_t::from_bin` explicitly accepts the all-zero infinity encoding and returns `SUCCESS`. The missing guard allows a crafted keyset blob with an infinity root point to pass deserialization, propagate through `refresh`, and cause `derive_keys` to produce derived public keys that diverge from the honest party's.

### Finding Description
In `blob_to_keyset` (`src/cbmpc/api/hd_keyset_ecdsa_2p.cpp:98–100`), after calling `from_bin(icurve, blob.root_Q_compressed)` and `from_bin(icurve, blob.root_K_compressed)`, there is no call to `icurve.check()`. The analogous `blob_to_key` in `src/cbmpc/api/ecdsa2pc.cpp:82` does call `curve.check(key.Q)` and would reject the infinity point. For P256, `ecurve_ossl_t::from_bin` accepts a 33-byte all-zero buffer as the point at infinity and returns `SUCCESS`. The `ecurve_t::check` function would reject this (`"EC-point is infinity"`), but it is never called. The infinity point propagates through `refresh` (which copies `key.root.Q` to `new_key.root.Q` without validation) and into `derive_keys`, where `Q_derived = Q + delta_G` evaluates to `delta_G` instead of the correct `Q_honest + delta_G`.

### Impact Explanation
An attacker who controls a keyset blob (e.g., their own party's persisted blob, or a blob injected via storage compromise) can supply a P256 keyset blob with `root_Q_compressed` = 33 zero bytes. The API accepts this blob and returns `SUCCESS` from `refresh`. The resulting new keyset blob also contains infinity as `root_Q`. When `derive_keys` is subsequently called, the derived public key is `delta_G` rather than the correct value, causing the two parties to hold derived ECDSA-2PC key blobs with different public keys. `extract_root_public_key_compressed` returns the all-zero encoding of infinity without error. This constitutes unsafe state acceptance and invalid HD-derived child keys.

### Likelihood Explanation
The attack requires the attacker to control a keyset blob that is fed to the API. This is the normal API usage pattern (each party stores and provides their own blob). The crafted blob is trivially constructable: a valid serialized `keyset_blob_v1_t` with `curve = p256`, `root_Q_compressed` = 33 zero bytes, and otherwise valid fields. The secp256k1 curve is not affected because `secp256k1_eckey_pubkey_parse` rejects the infinity encoding.

### Recommendation
Add `icurve.check()` calls after both `from_bin` calls in `blob_to_keyset`, mirroring the pattern in `blob_to_key`:

```cpp
error_t rv = keyset.root.Q.from_bin(icurve, blob.root_Q_compressed);
if (rv) return rv;
if (icurve.check(keyset.root.Q)) return coinbase::error(E_FORMAT, "invalid keyset blob root_Q");

rv = keyset.root.K.from_bin(icurve, blob.root_K_compressed);
if (rv) return rv;
if (icurve.check(keyset.root.K)) return coinbase::error(E_FORMAT, "invalid keyset blob root_K");
```

Also add the missing `paillier.has_private_key()` vs. role check and `paillier.verify_cipher(c_key)` check to match the defensive validation in `blob_to_key`.

### Proof of Concept
Construct a `keyset_blob_v1_t` with:
- `version = 1`, `role = 0` (p1), `curve = 2` (p256)
- `root_Q_compressed` = `{0x00, 0x00, ..., 0x00}` (33 zero bytes — P256 infinity encoding)
- `root_K_compressed` = any valid P256 compressed point
- `x_share`, `k_share`, `paillier`, `c_key` = valid values satisfying the existing checks

Serialize this blob and pass it to `hd_keyset_ecdsa_2p::refresh`. Assert that the call returns `SUCCESS` (it will, because `from_bin` accepts the infinity encoding and no `check` is called). Then call `extract_root_public_key_compressed` on the output blob and assert the returned bytes are all zero — confirming the infinity point was accepted and propagated.
