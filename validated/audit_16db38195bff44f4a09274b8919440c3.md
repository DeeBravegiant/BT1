### Title
`respond_ckd` Accepts Unverified CKD Output for `AppPublicKey` Variant, Allowing Any Single Attested Participant to Forge Confidential Key Derivation Output - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` in the MPC contract performs a full BLS12-381 pairing check (`ckd_output_check`) on responses for the `AppPublicKeyPV` (publicly-verifiable) variant, but performs **no cryptographic verification whatsoever** on responses for the `AppPublicKey` (privately-verifiable, legacy) variant. Any single attested participant can call `respond_ckd` with an arbitrary forged `(big_c, big_y)` pair for any pending `AppPublicKey` CKD request, and the contract will accept and deliver it to the user. This bypasses the threshold requirement for confidential key derivation.

### Finding Description

In `respond_ckd`, the match on `request.app_public_key` has an empty arm for `AppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the MPC master key and the app's public key. [2](#0-1) 

For the `AppPublicKey` variant, no such check exists. The contract simply serializes whatever `CKDResponse` the caller submitted and resolves the pending yield with it. [3](#0-2) 

The existing unit test for the `AppPublicKey` variant confirms this: it submits `big_y: [1u8; 48], big_c: [2u8; 48]` — clearly invalid BLS12-381 points — and the contract accepts them without error. [4](#0-3) 

The `AppPublicKey` variant is the legacy default and is actively used in production e2e tests. [5](#0-4) 

### Impact Explanation

**Critical. Unauthorized confidential key derivation output without the required participant authorization.**

A single malicious attested participant can:
1. Monitor the NEAR chain for pending `AppPublicKey` CKD requests.
2. Call `respond_ckd` with an attacker-chosen `(big_c, big_y)` pair before the honest leader node does.
3. The contract accepts the forged response and delivers it to the user.

The user computes `s = big_c − a · big_y` (where `a` is their private scalar). With a forged response, `s` does not equal `msk · H(app_id)` — the user receives a key that is not derived from the MPC master secret. The attacker can also craft `(big_c, big_y)` such that `s` equals any target value (e.g., zero, or a value the attacker knows), fully controlling the derived key.

This breaks the threshold guarantee: the CKD protocol requires threshold-many nodes to contribute shares, but a single participant can unilaterally determine the output for the `AppPublicKey` variant.

### Likelihood Explanation

**Medium.** The attacker must be an attested participant — a legitimate MPC node operator who has passed TEE attestation. However, the TEE model explicitly considers the operator untrusted (the TEE hardware is trusted, not the operator). A compromised or malicious operator running a modified binary inside a TEE that passes attestation (e.g., during a grace period or with a valid but modified image) can exploit this. The `AppPublicKey` variant is the legacy default path and is actively used. The attack requires only monitoring the chain and submitting a transaction before the honest leader, which is straightforward given NEAR's public mempool visibility.

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` variant. Since `AppPublicKey` provides only `pk1` (a G1 point, not a G2 point), the current `ckd_output_check` signature requires `AppPublicKeyPV`. The fix requires either:

1. **Require `AppPublicKeyPV` for all new requests** and deprecate `AppPublicKey`. The `AppPublicKeyPV` variant already has full on-chain verification.
2. **Add an alternative on-chain check for `AppPublicKey`**: since `pk1 = a·G1` is known, verify `e(big_c − big_y·pk1, g2) = e(H(pk, app_id), mpc_pk)` using the available BLS12-381 host functions — this is equivalent to checking `big_c = msk·H(app_id) + y·pk1` without knowing `a`.

Option 1 is simpler and eliminates the unverifiable variant entirely.

### Proof of Concept

1. User submits `request_app_private_key` with `AppPublicKey(pk1)` for `derivation_path = "mykey"`.
2. Attacker (attested participant) observes the pending request on-chain.
3. Attacker calls `respond_ckd(request, CKDResponse { big_y: [0u8;48], big_c: [0u8;48] })`.
4. The contract's `respond_ckd` hits the empty `AppPublicKey(_) => {}` arm, skips all verification, and resolves the yield with the forged response.
5. The user receives `(big_y=0, big_c=0)` and computes `s = 0 − a·0 = 0` — a zero key — instead of the correct MPC-derived key.

The existing test at `crates/contract/src/lib.rs:3403` already demonstrates this path succeeds with arbitrary byte values `[1u8;48]` and `[2u8;48]` for `big_y` and `big_c` respectively, with no rejection. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
