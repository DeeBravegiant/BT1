Audit Report

## Title
Unconditional Zero-Value Fee Transfer in `claim()` Freezes All User Claims When `feeInBPS == 0` - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `safeTransfer(protocolTreasury, fee)` at line 144 even when `fee == 0`. When `feeInBPS` is configured to zero (a valid and explicitly permitted value) and the distributed token reverts on zero-value transfers, every invocation of `claim()` reverts, permanently freezing all unclaimed yield for all users.

## Finding Description
In `contracts/utils/MerkleDistributor/MerkleDistributor.sol`, the `claim()` function computes the fee and unconditionally executes two `safeTransfer` calls:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);  // always called, even when fee == 0
``` [1](#0-0) 

When `feeInBPS == 0`, `fee` evaluates to `0`, and `safeTransfer(protocolTreasury, 0)` is called unconditionally. Tokens that revert on zero-value transfers (e.g., `LEND`, `cUSDCv3`) will cause every `claim()` call to revert.

`feeInBPS` can legitimately be zero in two ways:
1. **Initialization**: `initialize()` only rejects `_feeInBPS > MAX_FEE_IN_BPS` (1000), so `_feeInBPS = 0` is accepted. [2](#0-1) 
2. **Post-deployment**: `setFeeInBPS(0)` passes the same validation. [3](#0-2) 

The sibling contract `KernelMerkleDistributor._processClaim()` already applies the correct guard, confirming this is a known pattern that was omitted in `MerkleDistributor`: [4](#0-3) 

## Impact Explanation
When triggered, no user can successfully call `claim()`. All tokens allocated in the distributor remain permanently locked with no alternative withdrawal path for users. This constitutes **permanent freezing of unclaimed yield** — a Medium impact in the allowed scope.

## Likelihood Explanation
Two conditions must hold simultaneously: (1) `feeInBPS` is set to `0`, which is a normal operational choice for zero-fee distributions and is explicitly permitted by the contract's validation logic; (2) the configured `token` reverts on zero-value transfers, a known property of several deployed ERC-20 tokens. Neither condition requires any privileged compromise beyond routine admin configuration. The combination is realistic whenever the protocol deploys a zero-fee distributor for a token with this transfer restriction.

## Recommendation
Guard the fee transfer with a zero-check, mirroring the pattern already used in `KernelMerkleDistributor`:

```solidity
// Before (vulnerable):
IERC20(token).safeTransfer(protocolTreasury, fee);

// After (fixed):
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
``` [5](#0-4) 

## Proof of Concept
1. Deploy `MerkleDistributor` with `feeInBPS = 0` and a `token` that reverts on zero-value transfers (e.g., a mock ERC-20 implementing `require(amount > 0, "zero transfer")`).
2. Owner calls `setMerkleRoot(root)` with a valid distribution root.
3. Any user calls `claim(index, account, cumulativeAmount, merkleProof)` with a valid proof and non-zero `cumulativeAmount`.
4. Line 138: `fee = (claimableAmount * 0) / 10_000 = 0`.
5. Line 141: `safeTransfer(account, claimableAmount)` succeeds.
6. Line 144: `safeTransfer(protocolTreasury, 0)` reverts — the token rejects the zero-value transfer.
7. The entire transaction reverts. Every subsequent attempt by any user hits the same revert path, permanently blocking all claims.

**Foundry test sketch:**
```solidity
function testClaimRevertsOnZeroFee() public {
    // Deploy token that reverts on zero-value transfer
    RevertOnZeroToken tok = new RevertOnZeroToken();
    distributor.initialize(address(tok), treasury, 0); // feeInBPS = 0
    // ... set merkle root, fund distributor ...
    vm.expectRevert();
    distributor.claim(1, user, 100e18, proof);
}
```
