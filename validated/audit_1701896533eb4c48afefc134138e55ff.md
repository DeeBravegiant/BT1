Audit Report

## Title
Excess ETH Permanently Locked in `MultiChainRateProvider` Due to `payable` `updateRate()` Not Forwarding Full `msg.value` - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`MultiChainRateProvider.updateRate()` is `external payable` and accepts arbitrary ETH, but inside its loop it forwards only the on-chain `estimatedFee` per receiver to LayerZero, not `msg.value`. Any ETH in excess of the sum of per-receiver estimated fees is permanently locked in the contract, which has no withdrawal function and no `receive()` fallback.

## Finding Description
In `MultiChainRateProvider.updateRate()` (L108–137), for each entry in `rateReceivers`, the contract calls `ILayerZeroEndpoint.estimateFees()` to obtain `estimatedFee`, then calls `ILayerZeroEndpoint.send{ value: estimatedFee }(...)` — forwarding only `estimatedFee`, not `msg.value`, to LayerZero. The refund address passed to LayerZero is `payable(msg.sender)`, so LayerZero refunds any excess above `estimatedFee` back to the caller — but the contract itself only ever forwards `estimatedFee` per receiver. Therefore, if `msg.value > Σ estimatedFee_i`, the difference `msg.value - Σ estimatedFee_i` remains in the `MultiChainRateProvider` contract.

The abstract contract defines no `receive()` fallback and no withdrawal function. No concrete implementation adds one. By contrast, `CrossChainRateProvider.updateRate()` (L96) correctly passes `send{ value: msg.value }(...)`, forwarding the full caller-supplied ETH to LayerZero and relying on LayerZero's own refund mechanism to return any excess to `msg.sender`. `MultiChainRateProvider` does not follow this pattern.

## Impact Explanation
Any ETH sent to `updateRate()` in excess of the sum of on-chain estimated LayerZero fees is permanently locked in the contract with no recovery path. This matches the **Critical — Permanent freezing of funds** impact class. Because `updateRate()` is intended to be called repeatedly (rate updates), the cumulative locked amount grows with every call that includes any buffer.

## Likelihood Explanation
`updateRate()` has no access control and is callable by any external address. LayerZero fees fluctuate between blocks, so callers routinely add a safety buffer (10–20%) to avoid reverts — a standard and expected practice for LZ integrations. Even a caller who queries `estimateTotalFee()` in the same block may see a different fee on-chain due to gas price changes. Every such call with any buffer results in permanently locked ETH.

## Recommendation
Replace the per-receiver `estimatedFee` forwarding pattern with proportional distribution of `msg.value` across receivers (mirroring `CrossChainRateProvider`), or add an explicit refund of `msg.value - Σ estimatedFee_i` to `msg.sender` after the loop, or add an owner-only ETH rescue function. The simplest fix consistent with the sibling contract's design is to refund any remaining balance to `msg.sender` after the loop:

```solidity
// After the loop:
uint256 remaining = address(this).balance;
if (remaining > 0) {
    (bool ok,) = payable(msg.sender).call{ value: remaining }("");
    require(ok, "refund failed");
}
```

## Proof of Concept
1. Deploy `MultiChainRateProvider` with two rate receivers configured.
2. Call `estimateTotalFee()` — returns 0.01 ETH (0.005 ETH per receiver).
3. Call `updateRate{ value: 0.012 ETH }()` (20% buffer, standard practice).
4. Inside the loop: `estimatedFee` for receiver 0 = 0.005 ETH → forwarded to LZ; `estimatedFee` for receiver 1 = 0.005 ETH → forwarded to LZ.
5. Total forwarded to LZ = 0.010 ETH. Remaining in contract = 0.002 ETH.
6. No function exists to withdraw the 0.002 ETH — it is permanently locked.
7. After N calls with buffers, the contract accumulates N × buffer ETH, all irrecoverable.

Foundry fork test plan: deploy a mock `ILayerZeroEndpoint` that returns a fixed `estimatedFee` and accepts `send` calls; call `updateRate{ value: estimatedFee * 1.2 }()`; assert `address(multiChainRateProvider).balance == estimatedFee * 0.2 * numReceivers` after the call and that no function can drain it.
