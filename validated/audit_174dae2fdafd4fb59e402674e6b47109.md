### Title
Tokens Permanently Locked When Transfer Amount Normalizes to Zero — (`near/omni-bridge/src/lib.rs`)

### Summary
`init_transfer` accepts and irreversibly burns/locks user tokens without validating that `normalize_amount(amount_without_fee)` produces a non-zero result on the destination chain. The zero-amount guard exists only in `sign_transfer`, which is called after tokens are already gone. With no cancel path, the tokens are permanently unrecoverable.

### Finding Description

`init_transfer` (called via `ft_on_transfer`) validates only that `fee < amount`: [1](#0-0) 

It then immediately burns or locks the full token amount in `init_transfer_internal`: [2](#0-1) 

The normalized-amount guard lives exclusively in `sign_transfer`, which runs only after tokens are already committed: [3](#0-2) 

`normalize_amount` uses floor division: [4](#0-3) 

The function's own doc comment acknowledges the permanent-loss case: *"When fee = 0, dust stays locked/burned."* — but this applies to the **entire transfer amount**, not just sub-unit dust, when the amount is smaller than the decimal-scaling factor. [5](#0-4) 

### Impact Explanation

**Critical — Irreversible fund lock.**

Concrete scenario:
- Token registered with `origin_decimals = 24` (NEAR yoctoNEAR precision), `decimals = 6` (EVM side). Divisor = 10^18.
- User calls `ft_transfer_call` with `amount = 500_000_000_000_000_000` (0.5 NEAR-equivalent units) and `fee = 0`.
- `init_transfer` accepts: `0 < 500_000_000_000_000_000` ✓. Tokens are burned/locked.
- Every subsequent `sign_transfer` call computes `500_000_000_000_000_000 / 10^18 = 0` and panics with `InvalidAmountToTransfer`.
- No cancel or refund function exists; the transfer record stays in `pending_transfers` forever and the tokens are unrecoverable. [6](#0-5) 

### Likelihood Explanation

**Medium.** Any token pair where `origin_decimals − decimals ≥ 1` (e.g., NEAR 24-decimal tokens bridging to a 6-decimal EVM token, diff = 18) creates a "dead zone" for amounts below `10^(origin_decimals − decimals)`. A user sending a small but non-trivial amount (e.g., a few hundred thousand yoctoNEAR) hits this silently. The protocol gives no warning at deposit time.

### Recommendation

Add a pre-validation in `init_transfer` (before tokens are burned/locked) that fetches the destination-chain decimals and asserts `normalize_amount(amount_without_fee, decimals) > 0`. This mirrors the fix applied in the referenced DEX audit: reject the operation at entry rather than after funds are committed.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 6` on the destination chain.
2. Call `ft_transfer_call` on the bridge with `amount = 999_999_999_999_999_999` (< 10^18) and `msg` encoding `InitTransferMsg { fee: 0, ... }`.
3. `init_transfer` accepts; `init_transfer_internal` burns/locks the tokens and returns `U128(0)` to `ft_transfer_call` (tokens kept by bridge).
4. Call `sign_transfer` for the resulting `TransferId`.
5. `normalize_amount(999_999_999_999_999_999, {24, 6}) = 0` → `require!(0 > 0)` panics.
6. Repeat step 4 indefinitely — always panics. Tokens are permanently locked with no recovery path. [3](#0-2) [7](#0-6)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
