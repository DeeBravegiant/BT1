### Title
Missing Protocol Feature Gate for `Action::DeterministicStateInit` in `validate_action_with_mode` — (File: runtime/runtime/src/action_validation.rs)

### Summary
`validate_action_with_mode` enforces protocol feature gates for several new action types (`DelegateV2`, `TransferToGasKey`, `WithdrawFromGasKey`) via `require_protocol_feature()`, but the `Action::DeterministicStateInit` arm (and `Action::DeployGlobalContract` / `Action::UseGlobalContract`) is missing this guard entirely. An unprivileged user can submit a transaction containing `DeterministicStateInit` on a node running a pre-feature protocol version, bypass the admission gate, and have the action admitted to the mempool and included in a chunk — the direct analog of the missing `whenNotPaused()` modifier.

### Finding Description

In `validate_action_with_mode`, every action type introduced behind a protocol feature is expected to call `require_protocol_feature()` before performing its specific validation. The pattern is consistent for the other gated actions:

- `Action::DelegateV2` → `require_protocol_feature(ProtocolFeature::DelegateV2, "DelegateV2", current_protocol_version)?;`
- `Action::TransferToGasKey` → `validate_transfer_to_gas_key_action(current_protocol_version)` → `require_protocol_feature(ProtocolFeature::GasKeys, ...)`
- `Action::WithdrawFromGasKey` → `validate_withdraw_from_gas_key_action(current_protocol_version)` → `require_protocol_feature(ProtocolFeature::GasKeys, ...)` [1](#0-0) 

However, the `DeterministicStateInit` arm calls only the content-level validator with no protocol feature gate:

```rust
Action::DeterministicStateInit(a) => {
    validate_deterministic_state_init(limit_config, a, receiver)
}
``` [2](#0-1) 

`validate_deterministic_state_init` itself only checks the derived account ID, key/value length limits, and the global contract identifier — it never calls `require_protocol_feature`. [3](#0-2) 

The same omission applies to `Action::DeployGlobalContract` and `Action::UseGlobalContract`: [4](#0-3) 

`validate_deploy_global_contract_action` only checks contract size; `validate_use_global_contract_action` only validates the identifier — neither calls `require_protocol_feature`. [5](#0-4) 

`validate_actions` (and `validate_actions_with_mode`) is the single entry point for both **RPC/mempool transaction admission** and **receipt validation** during chunk processing. The `current_protocol_version` is threaded through to every arm precisely so that feature gates can be enforced at both layers. [6](#0-5) 

The CHANGELOG for the most recent release explicitly notes: *"Fix deterministic account ID creation to allow creation by any incoming transfer (unless it's a refund) and correctly check whether the feature is enabled."* This confirms the feature-enabled check was identified as missing/incorrect and was patched in the execution path, but the validation path (`validate_action_with_mode`) still lacks the guard. [7](#0-6) 

### Impact Explanation

During the protocol upgrade window — after nodes deploy code containing `DeterministicStateInit` but before the network-wide protocol version activates the feature — an unprivileged user can:

1. Submit a `SignedTransaction` containing `Action::DeterministicStateInit` to any RPC node.
2. The transaction passes `validate_actions` (no feature gate), is admitted to the mempool, and forwarded to chunk producers.
3. The receipt created from the transaction also passes `validate_actions_with_mode` (same missing gate), so it is not rejected at the receipt-validation layer.
4. If the execution path also lacks the gate (as suggested by the code in `action_deterministic_state_init`), the action executes and writes deterministic account state before the feature is protocol-active, producing a **state root that diverges** from nodes that correctly reject the action.

The corrupted value is the **state root** of the affected shard: accounts with `0s`-prefixed IDs and associated contract storage are written into the trie on pre-feature nodes, while fully-upgraded nodes that enforce the gate produce a different state root — a consensus-breaking divergence. [8](#0-7) 

### Likelihood Explanation

The attack window is the protocol upgrade transition period. NEAR protocol upgrades are announced in advance and the window between code deployment and protocol version activation spans multiple epochs. Any user with a funded account and access to an RPC endpoint can craft and submit the transaction with no special privileges. The `DeterministicStateInit` action type is fully documented and its Borsh encoding is public, making crafting such a transaction straightforward.

### Recommendation

Add `require_protocol_feature` calls for each affected action arm in `validate_action_with_mode`, mirroring the pattern used for `DelegateV2`, `TransferToGasKey`, and `WithdrawFromGasKey`:

```rust
Action::DeterministicStateInit(a) => {
    require_protocol_feature(
        ProtocolFeature::DeterministicStateInit,
        "DeterministicStateInit",
        current_protocol_version,
    )?;
    validate_deterministic_state_init(limit_config, a, receiver)
}
Action::DeployGlobalContract(a) => {
    require_protocol_feature(
        ProtocolFeature::GlobalContracts,
        "GlobalContracts",
        current_protocol_version,
    )?;
    validate_deploy_global_contract_action(limit_config, a)
}
Action::UseGlobalContract(a) => {
    require_protocol_feature(
        ProtocolFeature::GlobalContracts,
        "GlobalContracts",
        current_protocol_version,
    )?;
    validate_use_global_contract_action(a)
}
```

1. Identify the protocol version at which `DeterministicStateInit` activates (from `core/primitives-core/src/version.rs`).
2. During the upgrade window (nodes running code that parses the action but protocol version not yet activated), submit via JSON-RPC:
   ```json
   {
     "actions": [{
       "DeterministicStateInit": {
         "state_init": { "V1": { "code": {"AccountId": "ft.near"}, "data": {} } },
         "deposit": "0"
       }
     }],
     "receiver_id": "0s<derived_keccak_hash>"
   }
   ```
3. The transaction passes `validate_actions` (no `require_protocol_feature` call for this arm), is admitted to the mempool, forwarded to a chunk producer, and the resulting receipt also passes `validate_actions_with_mode`.
4. The action executes via `action_deterministic_state_init`, writing account state to the trie on a pre-feature protocol version, producing a state root that diverges from nodes enforcing the gate. [9](#0-8) [10](#0-9)

pub(crate) fn validate_actions_with_mode(
    limit_config: &LimitConfig,
    actions: &[Action],
    receiver: &AccountId,
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ActionsValidationError> {
    if actions.len() as u64 > limit_config.max_actions_per_receipt {
        return Err(ActionsValidationError::TotalNumberOfActionsExceeded {
            total_number_of_actions: actions.len() as u64,
            limit: limit_config.max_actions_per_receipt,
        });
    }

    // Centralized post-quantum gate. Mirrors the tx-admission gate in
    // `check_valid_for_config`, and is load-bearing for actions emitted by
    // contracts via host functions: those actions create new receipts that
    // never go through tx admission, so on a pre-feature protocol they must
    // be rejected here. The exhaustive match in
    // `Action::post_quantum_signatures_required` (including the recursive
    // walk into `Delegate`) forces every future action variant to make an
    // explicit decision at compile time.
    if !ProtocolFeature::PostQuantumSignatures.enabled(current_protocol_version)
        && actions.iter().any(Action::post_quantum_signatures_required)
    {
        return Err(ActionsValidationError::UnsupportedProtocolFeature {
            protocol_feature: "PostQuantumSignatures".to_owned(),
            version: current_protocol_version,
        });
    }

    if mode == ValidateReceiptMode::NewReceipt {
        validate_number_of_deploy_actions(actions, limit_config.max_deploy_actions_per_receipt)?;
    }

    let mut found_delegate_action = false;
    let mut iter = actions.iter().peekable();
    while let Some(action) = iter.next() {
        if let Action::DeleteAccount(_) = action {
            if iter.peek().is_some() {
                return Err(ActionsValidationError::DeleteActionMustBeFinal);
            }
        } else {
            if let Action::Delegate(_) | Action::DelegateV2(_) = action {
                if found_delegate_action {
                    return Err(ActionsValidationError::DelegateActionMustBeOnlyOne);
                }
                found_delegate_action = true;
            }
        }
        validate_action_with_mode(limit_config, action, receiver, current_protocol_version, mode)?;
    }

    let total_prepaid_gas =
        total_prepaid_gas(actions).map_err(|_| ActionsValidationError::IntegerOverflow)?;
    if total_prepaid_gas > limit_config.max_total_prepaid_gas {
        return Err(ActionsValidationError::TotalPrepaidGasExceeded {
            total_prepaid_gas,
            limit: limit_config.max_total_prepaid_gas,
        });
    }

    Ok(())
}

/// Validates a single given action.
/// The `mode` only affects nested validation of `Action::Delegate` payloads
fn validate_action_with_mode(
    limit_config: &LimitConfig,
    action: &Action,
    receiver: &AccountId,
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ActionsValidationError> {
    match action {
        Action::CreateAccount(_) => Ok(()),
        Action::DeployContract(a) => validate_deploy_contract_action(limit_config, a),
        Action::DeployGlobalContract(a) => validate_deploy_global_contract_action(limit_config, a),
        Action::UseGlobalContract(a) => validate_use_global_contract_action(a),
        Action::FunctionCall(a) => validate_function_call_action(limit_config, a),
        Action::Transfer(_) => Ok(()),
        Action::Stake(a) => validate_stake_action(a),
        Action::AddKey(a) => validate_add_key_action(limit_config, a, current_protocol_version),
        Action::DeleteKey(_) => Ok(()),
        Action::DeleteAccount(a) => validate_delete_action(a),
        Action::Delegate(a) => validate_delegate_action(
            limit_config,
            (&a.delegate_action).into(),
            receiver,
            current_protocol_version,
            mode,
        ),
        Action::DelegateV2(a) => {
            require_protocol_feature(
                ProtocolFeature::DelegateV2,
                "DelegateV2",
                current_protocol_version,
            )?;
            validate_delegate_action(
                limit_config,
                (&a.delegate_action).into(),
                receiver,
                current_protocol_version,
                mode,
            )
        }
        Action::DeterministicStateInit(a) => {
            validate_deterministic_state_init(limit_config, a, receiver)
        }
        Action::TransferToGasKey(_) => {
            validate_transfer_to_gas_key_action(current_protocol_version)
        }
        Action::WithdrawFromGasKey(_) => {
            validate_withdraw_from_gas_key_action(current_protocol_version)
        }
    }
}

fn validate_deterministic_state_init(
    limit_config: &LimitConfig,
    action: &DeterministicStateInitAction,
    receiver_id: &AccountId,
) -> Result<(), ActionsValidationError> {
    validate_global_contract_identifier(action.state_init.code())?;

    let derived_id = derive_near_deterministic_account_id(&action.state_init);

    if derived_id != *receiver_id {
        return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver {
            derived_id,
            receiver_id: receiver_id.clone(),
        });
    }

    // State init entries must not violate limits of individual state keys and values.
    for (key, value) in action.state_init.data() {
        if key.len() as u64 > limit_config.max_length_storage_key {
            return Err(ActionsValidationError::DeterministicStateInitKeyLengthExceeded {
                length: key.len() as u64,
                limit: limit_config.max_length_storage_key,
            }
            .into());
        }

        if value.len() as u64 > limit_config.max_length_storage_value {
            return Err(ActionsValidationError::DeterministicStateInitValueLengthExceeded {
                length: value.len() as u64,
                limit: limit_config.max_length_storage_value,
            }
            .into());
        }
    }

    Ok(())
}

pub(crate) fn action_deterministic_state_init(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    maybe_account: &mut Option<Account>,
    account_id: &AccountId,
    receipt: &Receipt,
    action: &DeterministicStateInitAction,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    // See https://github.com/near/NEPs/blob/master/neps/nep-0616.md#account-state
    // for the detailed description around deterministic account state.
    let storage_usage_config = &apply_state.config.fees.storage_usage_config;
    let account = match maybe_account {
        Some(account) => account,
        None => {
            // cspell:ignore nonexist
            // `nonexist` -> `uninit` account state transition
            // Create with zero balance now and check later how much of the
            // provided deposit is needed.
            let new_account = create_deterministic_account(Balance::ZERO, storage_usage_config);
            maybe_account.insert(new_account)
        }
    };
    if account.contract().is_none() {
        // `uninit` -> `active` account state transition
        deploy_deterministic_account(
            state_update,
            account,
            account_id,
            &action.state_init,
            result,
            storage_usage_config,
        )?;
    }
