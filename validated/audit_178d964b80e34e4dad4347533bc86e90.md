### Title
Missing Lower-Bound Timestamp Check in `is_proposal_init_valid` Allows Malicious Proposer to Commit Blocks with Stale Timestamps and Wrong L1 Gas Prices — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` enforces only an **upper** bound on `ProposalInit.timestamp` (not more than `block_timestamp_window_seconds` seconds in the future) but has **no lower bound** (not more than `block_timestamp_window_seconds` seconds in the past). A malicious proposer can freely set `timestamp` to any value in the range `[last_block_timestamp, now]`, which can be many seconds stale. Because the validator then uses that same proposer-supplied timestamp to query its own L1 gas price provider, both sides agree on the (historically stale) gas prices, the proposal passes validation, and a block is committed with a wrong `block_timestamp` and wrong L1 gas prices embedded in `BlockInfo`.

---

### Finding Description

In `is_proposal_init_valid` the timestamp window is enforced asymmetrically:

```rust
// lower bound: only prevents going backwards vs. previous block
if init_proposed.timestamp < last_block_timestamp { reject }

// upper bound: prevents timestamps more than window_seconds in the future
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds { reject }
``` [1](#0-0) 

There is no corresponding check:
```rust
if init_proposed.timestamp < now.saturating_sub(block_timestamp_window_seconds) { reject }
```

The production configuration sets `block_timestamp_window_seconds = 1`, described as "Maximum allowed deviation (seconds) of a proposed block's timestamp from the current time." [2](#0-1) [3](#0-2) 

The intent is a symmetric ±1 s window, but the implementation only enforces the future half. A proposer can therefore set `timestamp = last_block_timestamp` (which may be many seconds in the past) and the check passes.

After the timestamp check, the validator calls `get_l1_prices_in_fri_and_wei` using the **proposer-supplied** timestamp:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,   // ← proposer-controlled stale value
    ...
)
.await;
``` [4](#0-3) 

Because the proposer also computed its gas prices at that same stale timestamp, both sides produce identical values; the `within_margin` check (10 %) passes trivially, and the proposal is accepted. [5](#0-4) 

The stale timestamp is then forwarded verbatim into `convert_to_sn_api_block_info`, which sets `block_timestamp: BlockTimestamp(init.timestamp)` in the `BlockInfo` passed to the execution engine:

```rust
Ok(starknet_api::block::BlockInfo {
    block_timestamp: BlockTimestamp(init.timestamp),
    ...
    gas_prices: GasPrices { ... },   // derived from stale timestamp
})
``` [6](#0-5) 

The L1 gas price provider's `get_price_info` has a guard against timestamps **too far in the future** (`max_time_gap_seconds = 900 s`) but no guard against timestamps **too far in the past**; it simply returns the mean over the N blocks whose timestamps precede the requested value: [7](#0-6) 

---

### Impact Explanation

Two concrete corrupted values reach the committed chain:

1. **Wrong `block_timestamp`** — Cairo contracts that call `get_block_timestamp()` see a value up to `now − last_block_timestamp` seconds in the past. With ~10 s block times this is a ~10 s deviation; during multi-round consensus failures the gap can be larger. Time-sensitive DeFi contracts (options expiry, loan liquidation windows, TWAP oracles) can be manipulated.

2. **Wrong L1 gas prices** — The mean Ethereum base-fee and blob-fee embedded in `BlockInfo.gas_prices` are computed over the N L1 blocks preceding the stale timestamp, not the current ones. If L1 gas prices were lower in the past, the proposer can deliberately understate L1 costs, causing every transaction in the block to be undercharged for L1 resource consumption. This is a direct "Incorrect L1 gas price effect with economic impact."

---

### Likelihood Explanation

The attacker must be selected as the proposer for a consensus round, which in a decentralized validator set is a normal, non-privileged role reachable by any validator with sufficient stake. No external user input is required; the malicious proposer simply sets `ProposalInit.timestamp = last_block_timestamp` when constructing the proposal. The manipulation is invisible to other validators because they independently reproduce the same stale gas prices using the same timestamp.

---

### Recommendation

Add a symmetric lower-bound check immediately after the existing upper-bound check in `is_proposal_init_valid`:

```rust
// Existing upper bound (future)
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
    return Err(...);
}
// Missing lower bound (past) — add this:
if init_proposed.timestamp < now.saturating_sub(
    proposal_init_validation.block_timestamp_window_seconds
) {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "Timestamp is too far in the past: now={}, block_timestamp_window_seconds={}, \
             proposed={}",
            now,
            proposal_init_validation.block_timestamp_window_seconds,
            init_proposed.timestamp
        ),
    ));
}
``` [8](#0-7) 

If the 1-second window is too tight for normal network jitter, a separate `block_timestamp_past_window_seconds` configuration parameter can be introduced. The same fix should be applied to the identical pattern in `try_sync`. [9](#0-8) 

---

### Proof of Concept

1. Validator V is selected as proposer for height H, round R. The previous block was committed with `last_block_timestamp = T_prev` (e.g., 30 s ago, `now = T_prev + 30`).
2. V constructs `ProposalInit` with `timestamp = T_prev` (the minimum allowed value).
3. V queries `get_l1_prices_in_fri_and_wei(provider, T_prev, ...)` → receives mean gas prices over L1 blocks before `T_prev − 60 s` (lag margin). Suppose these are 50 gwei vs. current 100 gwei.
4. V broadcasts the proposal with `l1_gas_price_wei = 50 gwei`.
5. Each validator calls `is_proposal_init_valid`:
   - `T_prev >= T_prev` → lower-bound check passes ✓
   - `T_prev <= (T_prev + 30) + 1` → upper-bound check passes ✓
   - Validator queries `get_l1_prices_in_fri_and_wei(provider, T_prev, ...)` → same 50 gwei ✓
   - `within_margin(50, 50, 10%)` → passes ✓
6. All validators accept the proposal. Block H is committed with `block_timestamp = T_prev` and `l1_gas_price_wei = 50 gwei` instead of the correct `now` and `100 gwei`.
7. Every transaction in block H is charged for L1 gas at 50 gwei; the protocol absorbs the 50 gwei/unit shortfall on all L1 data posted for that block.
