Audit Report

## Title
Stale Cross-Chain Rate Enables Over-Minting of wrsETH/agETH, Causing Protocol Insolvency — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.getRate()` returns the last stored `rate` with no staleness check against `lastUpdated`. All three downstream pools (`RSETHPoolV2`, `RSETHPoolV3`, `AGETHPoolV3`) call `getRate()` unconditionally when computing mint amounts. If the LayerZero bridge stops delivering updates while the L1 rsETH/agETH price appreciates, all pools continue minting wrapped tokens at the outdated lower rate, issuing more shares than the deposited ETH can back at the true L1 rate — constituting protocol insolvency.

## Finding Description
**Root cause — `CrossChainRateReceiver.getRate()` (lines 97, 103–105):**

`lzReceive` records `lastUpdated = block.timestamp` on every update, but `getRate()` ignores it entirely:

```solidity
// line 97
lastUpdated = block.timestamp;   // stored, never read again

// lines 103-105
function getRate() external view returns (uint256) {
    return rate;                 // no staleness guard
}
```

**Downstream consumers — all three pools call `getRate()` unconditionally:**

- `RSETHPoolV3.viewSwapRsETHAmountAndFee` (ETH path, line 304): `uint256 rsETHToETHrate = getRate();` → `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;`
- `RSETHPoolV3.viewSwapRsETHAmountAndFee` (token path, line 328): same pattern with `tokenToETHRate` numerator
- `RSETHPoolV2.viewSwapRsETHAmountAndFee` (line 230): identical ETH path
- `AGETHPoolV3.viewSwapAgETHAmountAndFee` (line 165): `uint256 agETHToETHrate = getRate();` → `agETHAmount = amountAfterFee * 1e18 / agETHToETHrate;`

None of these functions check `lastUpdated` before using the rate. The `limitDailyMint` modifier in `RSETHPoolV2`/`RSETHPoolV3` calls `viewSwapRsETHAmountAndFee` to compute the mint amount, so the stale rate propagates into the daily limit accounting as well — the limit is enforced in stale-rate units, not true-rate units, allowing the full daily quota to be consumed at the inflated mint ratio each day throughout the staleness window.

**Exploit path:**
1. LayerZero bridge experiences an outage (documented real-world event; no attacker action required).
2. `rate` in `CrossChainRateReceiver` freezes at the last delivered value (e.g., `1.01e18`).
3. True L1 rsETH rate appreciates to, e.g., `1.05e18` over 30 days.
4. Any user calls `RSETHPoolV3.deposit{value: 1 ether}(...)` — a fully public, permissionless function.
5. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = 1e18 * 1e18 / 1.01e18 ≈ 0.9901e18`.
6. `wrsETH.mint(msg.sender, 0.9901e18)` is executed.
7. The ETH is later bridged to L1 and converted to rsETH at the true rate: `1e18 / 1.05e18 ≈ 0.9524 rsETH`.
8. Outstanding wrsETH (`0.9901`) exceeds backing rsETH (`0.9524`) — the protocol is undercollateralized by ~3.9% per deposited ETH.
9. This compounds across every deposit made during the entire staleness window, across all three pools.

**Existing guards are insufficient:**
- `dailyMintLimit` caps per-day exposure in stale-rate units but resets every day, allowing the insolvency to accumulate day-by-day across the full staleness window.
- `paused` flag exists but requires a privileged `PAUSER_ROLE` holder to act; no automatic circuit-breaker triggers on staleness.
- `AGETHPoolV3` has no `dailyMintLimit` at all, providing zero rate-of-accumulation protection.

## Impact Explanation
**Critical — Protocol insolvency.** The ETH collected by the pools is bridged to L1 and converted to rsETH at the true (higher) L1 rate. The wrsETH/agETH minted on L2 at the stale (lower) rate exceeds the rsETH backing. Holders of the over-minted wrsETH/agETH cannot all redeem at par; the last redeemers receive less than 1:1 rsETH backing. The shortfall is permanent and grows with the magnitude and duration of the staleness, directly matching the "Protocol insolvency" critical impact class.

## Likelihood Explanation
No attacker is required. LayerZero bridge liveness failures are documented, real-world events. The protocol has no on-chain circuit-breaker that pauses deposits when `lastUpdated` is too old. Any bridge interruption lasting more than a few hours while rsETH/agETH accrues yield is sufficient. Ordinary users depositing during the outage are the unwitting source of the insolvency. The condition is repeatable across all three pools simultaneously and accumulates every day the bridge remains silent.

## Recommendation
Add a configurable `maxStaleness` threshold and revert in `getRate()` if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g., 86400 (24 hours)

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes all three pools' `viewSwapRsETHAmountAndFee` / `viewSwapAgETHAmountAndFee` calls — and therefore all `deposit()` calls — to revert automatically when the rate is stale, without requiring any privileged action. `maxStaleness` should be set conservatively (e.g., 24 hours) and be updatable by the owner.

## Proof of Concept

```solidity
// Foundry fork test
function testStaleRateCausesOverMint() public {
    // 1. Simulate one LZ update at rate = 1.01e18
    vm.prank(layerZeroEndpoint);
    receiver.lzReceive(
        srcChainId,
        abi.encodePacked(rateProvider),
        0,
        abi.encode(uint256(1.01e18))
    );

    // 2. Warp 30 days — bridge goes silent, true L1 rate is now ~1.05e18
    vm.warp(block.timestamp + 30 days);

    // 3. User deposits 1 ETH into RSETHPoolV3 (permissionless public call)
    uint256 deposit = 1 ether;
    (uint256 wrsETHMinted,) = pool.viewSwapRsETHAmountAndFee(deposit);

    // 4. wrsETHMinted uses stale 1.01e18 → ~0.9901e18
    //    true backing at 1.05e18 → ~0.9524e18
    uint256 trueRsETHBacking = deposit * 1e18 / 1.05e18;
    assertGt(wrsETHMinted, trueRsETHBacking);
    // wrsETHMinted ≈ 0.9901e18 > trueRsETHBacking ≈ 0.9524e18
    // Protocol is undercollateralized by ~3.9% per deposited ETH
    // Repeat across dailyMintLimit * 30 days for total insolvency magnitude
}
```
