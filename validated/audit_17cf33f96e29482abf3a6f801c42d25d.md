Audit Report

## Title
Permissionless `updateRSETHPrice()` Enables Yield Sandwich Attack Against rsETH Holders - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.updateRSETHPrice()` carries no access control, allowing any caller to trigger a price update at will. An attacker can deposit ETH at the stale (pre-reward) price to receive inflated rsETH, immediately call `updateRSETHPrice()` to capture the reward-driven price increase, and then withdraw at the higher price. This transfers a portion of accrued yield from existing rsETH holders to the attacker.

## Finding Description

`updateRSETHPrice()` is unconditionally public:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_updateRsETHPrice()` computes `previousTVL = rsethSupply.mulWad(rsETHPrice)` using the **old** price and the **current** supply. When EigenLayer rewards or LST rebases have accrued, `totalETHInProtocol > previousTVL`, so `newRsETHPrice` increases. Crucially, a deposit made just before this call is priced at the old rate:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Because the deposit is included in `totalETHInProtocol` but priced at the stale rate, the attacker receives more rsETH than the post-update price would grant. The reward gap (`totalETHInProtocol - previousTVL`) is unchanged by the deposit, so the full reward increment is still applied when `updateRSETHPrice()` is called, but it is now spread over a larger rsETH supply that includes the attacker's inflated position.

**Concrete math (simplified, no fee):**
- S = 100 rsETH, old price = 1.0 ETH, accrued rewards = 10 ETH.
- Without attack: new price = 110/100 = 1.1; existing holders gain 10 ETH.
- Attacker deposits 10 ETH at old price → gets 10 rsETH; supply = 110.
- After `updateRSETHPrice()`: new price = 120/110 ≈ 1.0909.
- Attacker's 10 rsETH = 10.909 ETH → **profit ≈ 0.909 ETH**.
- Existing holders gain only 9.09 ETH instead of 10 ETH → **loss ≈ 0.91 ETH**.

**Attack Path A — instant withdrawal (when `isInstantWithdrawalEnabled[asset]` is true):**
1. Deposit X ETH → receive `X / rsETHPrice_old` rsETH.
2. Call `updateRSETHPrice()` → price rises to `rsETHPrice_new`.
3. Call `instantWithdrawal()` → `getExpectedAssetAmount` uses `rsETHPrice_new`; attacker receives `rsETH * rsETHPrice_new > X`.

```solidity
// contracts/LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

**Attack Path B — queued withdrawal (always available):**
1. Deposit at `rsETHPrice_old`.
2. Call `updateRSETHPrice()` → price rises.
3. Call `initiateWithdrawal()` → `expectedAssetAmount` locked at `rsETHPrice_new`.
4. After `withdrawalDelayBlocks`, call `completeWithdrawal()` → receives `request.expectedAssetAmount`.

```solidity
// contracts/LRTWithdrawalManager.sol L168
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
// ...
_addUserWithdrawalRequest(asset, rsETHUnstaked, expectedAssetAmount);
```

```solidity
// contracts/LRTWithdrawalManager.sol L734
_transferAsset(asset, user, request.expectedAssetAmount);
```

**Why the `pricePercentageLimit` guard is insufficient:**
The guard at lines 252–266 only reverts for non-managers when `newRsETHPrice > highestRsethPrice` **and** the increase exceeds `pricePercentageLimit`. It does not block the attack when:
- `pricePercentageLimit == 0` (unset).
- The reward accrual is within the configured threshold (e.g., daily rewards well below 1%).
- The price has previously decreased below `highestRsethPrice` (guard does not apply).

In all these cases, the permissionless call succeeds and the sandwich is profitable.

## Impact Explanation

Every reward cycle, an attacker can capture a fraction of the accrued yield that belongs pro-rata to all existing rsETH holders. The stolen amount scales linearly with deposit size and reward magnitude. With instant withdrawal enabled the attack is atomic and risk-free within a single block. With queued withdrawal it requires an 8-day hold but is still profitable under normal price conditions. This constitutes **theft of unclaimed yield** — a High-severity impact under the allowed scope.

## Likelihood Explanation

`updateRSETHPrice()` is permissionless and must be called regularly to keep the price current. Rewards accrue continuously from EigenLayer strategies and LST rebases. Any on-chain observer can detect a pending reward increment by simulating `_getTotalEthInProtocol()` off-chain and comparing to `rsethSupply * rsETHPrice`. No privileged access, leaked keys, or external protocol compromise is required. The attack is repeatable every reward cycle and is executable by any EOA or contract.

## Recommendation

1. **Restrict `updateRSETHPrice()` to authorized callers** (e.g., `onlyLRTManager` or a dedicated keeper role), removing the permissionless entry point. The existing `updateRSETHPriceAsManager()` already demonstrates the intended privileged path.
2. **Alternatively, stream rewards over time** (e.g., a `rewardPerSecond` model) so no single call causes a step-change in `rsETHPrice` that can be sandwiched.
3. **Add a deposit lock-up or withdrawal cooldown** relative to the last `updateRSETHPrice()` call so that a deposit made in the same block as a price update cannot immediately be used for withdrawal.

**Setup (Foundry fork test):**
```solidity
// 1. Fork mainnet; record rsETHPrice_old = lrtOracle.rsETHPrice()
// 2. Simulate reward accrual: warp time / mock EigenLayer strategy balance increase
//    so that _getTotalEthInProtocol() > rsethSupply * rsETHPrice_old

// Attack sequence (single tx or same block):
// Step 1: Deposit
lrtDepositPool.depositETH{value: 1000 ether}(0, "");
uint256 rsethReceived = rsETH.balanceOf(attacker); // > 1000e18 / rsETHPrice_new

// Step 2: Trigger price update
lrtOracle.updateRSETHPrice();
uint256 rsETHPrice_new = lrtOracle.rsETHPrice(); // > rsETHPrice_old

// Step 3a (instant withdrawal enabled):
lrtWithdrawalManager.instantWithdrawal(ETH_TOKEN, rsethReceived, "");

// Step 3b (queued withdrawal):
lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsethReceived, "");
// warp withdrawalDelayBlocks; operator calls unlockQueue(...)
lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");
uint256 ethReceived = attacker.balance; // > 1000 ether → profit confirmed

// Assert: ethReceived > 1000 ether
// Assert: existing_holder_rseth_value_after < existing_holder_rseth_value_before + full_reward
```

The invariant `∀ depositor: value_out ≤ value_in + pro_rata_share_of_rewards` is violated whenever a deposit precedes a permissionless price update in the same block.
