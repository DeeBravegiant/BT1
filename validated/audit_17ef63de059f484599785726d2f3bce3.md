Audit Report

## Title
First Depositor Price Inflation via Donation Causes Permanent Loss of Subsequent Depositor Funds - (File: contracts/LRTOracle.sol / contracts/LRTDepositPool.sol)

## Summary

`LRTOracle` stores `rsETHPrice` as a cached value updatable by anyone via the public `updateRSETHPrice()`. Because TVL accounting uses raw `balanceOf`/`address(this).balance` (inclusive of unsolicited donations), and because `pricePercentageLimit` defaults to `0` (disabling the price-increase guard), a first depositor can donate assets to inflate the stored price to an arbitrarily large value. Any subsequent depositor who passes `minRSETHAmountExpected = 0` will have `getRsETHAmountToMint` round down to 0, pass the unchecked `_beforeDeposit` guard, and permanently lose their deposited assets while receiving nothing.

## Finding Description

**Root cause 1 — `pricePercentageLimit` defaults to `0`, disabling the price-increase guard.**

`pricePercentageLimit` is declared as a plain `uint256` storage variable with no initialization in `initialize()`, so it starts at `0`. [1](#0-0) 

The guard short-circuits on the `pricePercentageLimit > 0` condition, meaning any arbitrarily large price increase is accepted by an unprivileged caller: [2](#0-1) 

**Root cause 2 — TVL includes raw balances, so donations inflate it.**

For ERC-20 assets, `assetLyingInDepositPool` is `IERC20(asset).balanceOf(address(this))`: [3](#0-2) 

For ETH, `ethLyingInDepositPool` is `address(this).balance`, and the contract has an open `receive()`: [4](#0-3) [5](#0-4) 

**Root cause 3 — `updateRSETHPrice()` is publicly callable with no access control.** [6](#0-5) 

**Root cause 4 — `_beforeDeposit` does not enforce `rsethAmountToMint > 0`.**

The only guard is `rsethAmountToMint < minRSETHAmountExpected`. When a caller passes `minRSETHAmountExpected = 0`, the condition `0 < 0` is `false`, so execution continues and `_mintRsETH(0)` is called — minting nothing for the victim: [7](#0-6) [8](#0-7) 

**Why existing checks fail:**

- The `pricePercentageLimit` guard at LRTOracle.sol L256–257 is a no-op when the variable is `0` (its default).
- `_beforeDeposit` only checks `rsethAmountToMint < minRSETHAmountExpected`; it never checks `rsethAmountToMint == 0` unconditionally.
- There is no minimum-deposit-output enforcement anywhere in the deposit path.

**Note on fee-minting interaction:** If `protocolFeeInBPS > 0` and `maxFeeMintAmountPerDay == 0` (both defaults), the `_checkAndUpdateDailyFeeMintLimit` call inside `_updateRsETHPrice` will revert when a TVL increase triggers a non-zero fee amount. The attack therefore executes cleanly when `protocolFeeInBPS == 0`, or when `maxFeeMintAmountPerDay` has been set to a non-zero value — both realistic deployment configurations. The missing `rsethAmountToMint > 0` check in `_beforeDeposit` is independently exploitable whenever the price is inflated by any means. [9](#0-8) 

## Impact Explanation

A victim who deposits assets and receives 0 rsETH has permanently lost those assets: they hold no rsETH to redeem, and there is no refund path in the contract. This constitutes **direct theft / permanent freezing of deposited funds** — Critical severity under the allowed impact scope.

## Likelihood Explanation

- `updateRSETHPrice()` requires no privilege; any EOA can call it.
- `pricePercentageLimit == 0` is the storage default; it is never set in `initialize()`, so every fresh deployment is vulnerable until an admin explicitly calls `setPricePercentageLimit`.
- The `receive()` fallback and `balanceOf`-based accounting make the donation step trivially executable.
- Passing `minRSETHAmountExpected = 0` is a common pattern for users who do not want to set slippage protection, and is the natural default for many integrations and front-ends.
- The attack is repeatable: after draining Bob's deposit, the attacker can repeat the cycle against further depositors.

## Recommendation

1. **Enforce `rsethAmountToMint > 0` unconditionally** in `_beforeDeposit` (LRTDepositPool.sol):
   ```solidity
   if (rsethAmountToMint == 0) revert ZeroRsETHMinted();
   ```
2. **Set a non-zero default for `pricePercentageLimit`** in `initialize()` (e.g., 10% = `1e17`), or require it to be configured before the first deposit is accepted.
3. **Pre-mint a meaningful amount of rsETH** to the protocol treasury at initialization to make the first-depositor share-inflation attack economically infeasible.
4. **Restrict `updateRSETHPrice()`** to authorized callers (keeper/manager role), or at minimum add a per-block or per-epoch rate-limit.

## Proof of Concept

Preconditions: `pricePercentageLimit == 0` (default), `protocolFeeInBPS == 0` (or `maxFeeMintAmountPerDay` set to a large value), ETH is a supported asset.

1. **Alice** calls `depositETH{value: 1 wei}(0, "")` → receives 1 wei rsETH. `rsETHPrice` remains `1 ether` (supply was 0 before this deposit, so price was initialized to `1 ether`).
2. Alice sends 10 000 ETH directly to `LRTDepositPool` (accepted by `receive()`). Pool balance is now `10_000e18 + 1`.
3. Alice calls `LRTOracle.updateRSETHPrice()`. Inside `_updateRsETHPrice`:
   - `rsethSupply = 1`
   - `totalETHInProtocol ≈ 10_001e18`
   - `newRsETHPrice = 10_001e18 / 1 ≈ 10_001e18`
   - `pricePercentageLimit == 0` → `isPriceIncreaseOffLimit = false` → price committed as `≈10_001e18`.
4. **Bob** calls `depositETH{value: 10_000e18}(0, "")`:
   - `getRsETHAmountToMint` → `(10_000e18 × 1e18) / 10_001e18 = 0` (integer division truncates).
   - `_beforeDeposit`: `0 < 0` is `false` → no revert.
   - `_mintRsETH(0)` → Bob receives 0 rsETH; 10 000 ETH is permanently lost.
5. Alice calls `updateRSETHPrice()` again; her 1 wei rsETH now backs ≈20 001 ETH of TVL. She redeems via the withdrawal path for the full pool, profiting ≈10 000 ETH net.

**Foundry test sketch:**
```solidity
function test_firstDepositorInflation() public {
    // Alice deposits 1 wei
    vm.prank(alice);
    depositPool.depositETH{value: 1}(0, "");
    assertEq(rsETH.balanceOf(alice), 1);

    // Alice donates 10_000 ETH
    vm.deal(address(depositPool), 10_000 ether + 1);

    // Alice inflates price
    vm.prank(alice);
    lrtOracle.updateRSETHPrice();
    assertGt(lrtOracle.rsETHPrice(), 10_000 ether);

    // Bob deposits 10_000 ETH with no slippage guard
    vm.deal(bob, 10_000 ether);
    vm.prank(bob);
    depositPool.depositETH{value: 10_000 ether}(0, "");

    // Bob receives 0 rsETH — funds permanently lost
    assertEq(rsETH.balanceOf(bob), 0);
}
```
