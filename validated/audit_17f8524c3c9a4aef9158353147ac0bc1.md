### Title
Forged CKD Response Accepted Without Cryptographic Verification for `AppPublicKey` Variant — (`crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, the contract performs a cryptographic output check (`ckd_output_check`) only when the request uses the `AppPublicKeyPV` (publicly verifiable) variant. When the request uses the `AppPublicKey` (privately verifiable / legacy) variant, the match arm is empty and no output verification is performed. Any single attested participant can therefore call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept and deliver it to the user.

---

### Finding Description

The `respond_ckd` function in `crates/contract/src/lib.rs` contains the following conditional output check:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `ckd_output_check` function verifies the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk, app_id), pk)`, which proves the response was correctly computed from the MPC network's master secret key. [2](#0-1) 

For `AppPublicKeyPV` requests, a tampered `big_c` is correctly rejected: [3](#0-2) 

But for `AppPublicKey` requests — the legacy and still-supported variant documented in the contract README and the `ckd-example-cli` — the empty match arm means any `CKDResponse` with arbitrary `big_c` and `big_y` values passes through unchecked. [4](#0-3) 

The `respond_ckd` caller must be an attested participant (`assert_caller_is_attested_participant_and_protocol_active`), but no threshold of participants is required to agree on the response — a single participant submitting first wins the yield race. [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe a pending `AppPublicKey` CKD request on-chain.
2. Call `respond_ckd` with an attacker-chosen `big_c` and `big_y` before the honest coordinator does.
3. The contract accepts the forged response (no output check for `AppPublicKey`), resolves the yield, and delivers it to the user.

The user computes `sig = big_c − a · big_y`. Because `big_c` and `big_y` are attacker-controlled, `sig` is not `msk · H(pk, app_id)`. The user derives a wrong, attacker-controlled key instead of their legitimate deterministic secret. This constitutes **unauthorized confidential key derivation output** — the user's derived secret is silently replaced without any threshold of honest participants authorizing it.

This matches the allowed impact: *Critical — Unauthorized confidential key derivation output without the required participant authorization.*

---

### Likelihood Explanation

The `AppPublicKey` variant is the legacy format, is still fully supported, and is the default in the `ckd-example-cli`. Any single attested participant (one compromised or malicious node) can exploit this. No collusion above threshold is required. The attacker only needs to submit the forged `respond_ckd` transaction before the honest coordinator does, which is a straightforward race condition on a public blockchain.

---

### Recommendation

Apply the same `ckd_output_check` to `AppPublicKey` requests. However, `ckd_output_check` requires `app_pk2` (a G2 point), which is absent in the `AppPublicKey` variant. The fix requires one of:

1. **Require `AppPublicKeyPV` for all new requests** and deprecate `AppPublicKey`, so the on-chain check is always possible.
2. **Add a separate output check for `AppPublicKey`** that verifies `e(big_c − big_y · A1, G2) = e(H(pk, app_id), pk)` using the G1 app public key and the network public key — this is verifiable without the G2 component.
3. **Require threshold-many matching `respond_ckd` calls** before resolving the yield, analogous to how `vote_pk` / `vote_reshared` require threshold agreement.

The minimal safe mitigation is option 1 (reject `AppPublicKey` in `respond_ckd` if no output check can be performed), paired with a migration path for existing users.

---

### Proof of Concept

```
1. Alice submits request_app_private_key with AppPublicKey variant (legacy G1 key).
   → Contract stores pending CKD request, yields.

2. Mallory (a single attested participant) observes the pending request on-chain.

3. Mallory calls respond_ckd(request, CKDResponse { big_c: G1::generator(), big_y: G1::generator() }).
   → respond_ckd hits the AppPublicKey arm → empty match → no ckd_output_check.
   → resolve_yields_for delivers the forged response to Alice's yield.

4. Alice receives (big_c=G1, big_y=G1) and computes sig = G1 − a·G1 = (1−a)·G1.
   → sig ≠ msk·H(pk, app_id). Alice derives a wrong key.
   → Mallory, knowing a=1 was forced, can predict Alice's derived key.
```

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

```markdown
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
