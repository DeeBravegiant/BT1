Audit Report

## Title
Multi-Day Price Update Delay Causes `DailyFeeMintLimitExceeded` Revert, Blocking Price Updates and Enabling Dilution of rsETH Holders - (File: contracts/LRTOracle.sol)

## Summary
When `updateRSETHPrice()` is not called for multiple consecutive days, the accumulated multi-day reward generates a protocol fee that exceeds `maxFeeMintAmountPerDay`. This causes `_checkAndUpdateDailyFeeMintLimit` to revert with `DailyFeeMintLimitExceeded`, blocking all price updates — including the manager path — until an admin manually raises the cap. During this window, new depositors mint rsETH at the stale (lower) price, permanently diluting existing holders' proportional claim on accrued rewards.

## Finding Description
In `LRTOracle._updateRsETHPrice()` (L244–246), the protocol fee is computed on the **total accumulated reward since the last update** with no time-weighting:

```solidity
uint256 rewardAmount = totalETHInProtocol - previousTVL;
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
```

The resulting rsETH fee amount is passed to `_checkAndUpdateDailyFeeMintLimit` (L301–303). Inside that function (L197–210), the period counter resets if a day has passed, but the entire N-day accumulated fee is then checked against a **single day's** cap:

```solidity
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
```

Both `updateRSETHPrice()` (L87–89) and `updateRSETHPriceAsManager()` (L94–96) call `_updateRsETHPrice()` with no bypass, so neither path can succeed while the accumulated fee exceeds the cap. The only recovery is for the manager to first call `setMaxFeeMintAmountPerDay()` (L132–135) to raise the cap, then retry. Until that happens, `rsETHPrice` (L313) remains stale.

New depositors compute their rsETH mint amount via `getRsETHAmountToMint` (L519–520 of `LRTDepositPool.sol`):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Because `rsETHPrice` is lower than the true current value, depositors receive more rsETH than they are entitled to, permanently diluting existing holders' share of the accrued rewards.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders are entitled to rewards that have accrued since the last price update. When new depositors mint at the stale (lower) price, they capture a portion of those accrued rewards. Once the price is eventually corrected upward, the over-minted rsETH held by new depositors is worth more than they paid, and existing holders' proportional claim on the TVL is permanently reduced. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation
Low. The condition requires the offchain bot's `updateRSETHPrice()` transactions to fail for multiple consecutive days (e.g., due to sustained high gas fees), and `maxFeeMintAmountPerDay` must be calibrated close to a single day's expected fee. Both conditions are realistic but must coincide. A 2–3 day delay is sufficient to push the accumulated fee over the cap. No attacker action is required to create the delay; any depositor who deposits during the stale window passively captures the dilution.

## Recommendation
1. **Time-weight the fee**: Divide the accumulated fee by the number of elapsed days and apply the daily limit per day, or split the update into multiple sub-steps covering each elapsed day.
2. **Size `maxFeeMintAmountPerDay` conservatively**: Set it to a multiple of the expected daily fee (e.g., 7×) to absorb short outages without reverting.
3. **Add a manager bypass for the daily cap**: Allow `updateRSETHPriceAsManager()` to skip the `DailyFeeMintLimitExceeded` revert (while still recording the mint), so the price can always be corrected even if the cap is exceeded.
