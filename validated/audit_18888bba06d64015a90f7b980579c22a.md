### Title
Proposer Deposits for Non-Winning Update Proposals Are Permanently Locked in the Contract - (`File: crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

When `do_update` executes a winning contract/config proposal, it bulk-clears **all** pending proposals and their associated storage without refunding the NEAR deposits paid by the other proposers. Because `UpdateEntry` does not record the proposer's account ID, there is no mechanism to return those deposits. The funds are permanently transferred to the contract's general balance with no recovery path.

---

### Finding Description

`propose_update` is a `#[payable]` function that requires each proposer to attach a deposit covering the estimated storage cost of their proposal: [1](#0-0) 

The deposit is computed from `ProposedUpdates::required_deposit`, which estimates bytes used by the update payload plus a fixed overhead for up to 128 participant votes: [2](#0-1) 

For a full contract binary (~1.5 MB), this deposit is approximately 17 NEAR (`CURRENT_CONTRACT_DEPLOY_DEPOSIT = NearToken::from_millinear(17000)`). [3](#0-2) 

When `vote_update` reaches threshold, it calls `do_update`, which:
1. Removes the winning entry.
2. **Bulk-clears all remaining entries** (`self.entries.clear()`).
3. **Bulk-clears all votes** (`self.vote_by_participant.clear()`).
4. Issues **no refunds** to any of the other proposers. [4](#0-3) 

The `UpdateEntry` struct stores only the update payload and `bytes_used`; it does **not** record the proposer's `AccountId`: [5](#0-4) 

Because the proposer identity is never stored, there is no way to issue refunds when entries are cleared. The freed storage returns to the contract's general balance, but the original depositors have no mechanism to reclaim their NEAR.

---

### Impact Explanation

Every participant who called `propose_update` for a proposal that was not selected loses their entire deposit permanently. The contract has no `withdraw`, `cancel_proposal`, or `recover_deposit` function. The funds become part of the contract's unreachable balance — an exact analog of the SafEth.sol residual ETH issue, where funds deposited for a specific purpose become permanently unutilized because the accounting mechanism does not track who deposited what.

For a realistic scenario with two competing proposals (e.g., two participants each proposing a different contract upgrade), the non-winning proposer loses ~17 NEAR (~$50+ at current prices) with no recourse. This breaks the production safety invariant that storage-staking deposits must be returned when the associated storage is freed.

**Allowed impact matched:** Medium — balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

The contract explicitly supports multiple concurrent proposals (the `entries` map is an `IterableMap<UpdateId, UpdateEntry>` and `id` is a monotonically incrementing counter). The README states that "all pending update proposals and votes are cleared" on execution, which is the documented behavior — but the deposit loss is an undocumented and unintended consequence. In any governance scenario where participants disagree on which upgrade to apply, multiple proposals will coexist, making this loss routine rather than exceptional. [6](#0-5) 

---

### Recommendation

Store the proposer's `AccountId` in `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add this
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue a `Promise::new(entry.proposer).transfer(required_deposit(entry.bytes_used))` for each one. Alternatively, add a `cancel_proposal` endpoint that allows a proposer to withdraw their own proposal and receive a refund, and only clear entries on execution without refunding (relying on proposers to cancel their own proposals before a competing one executes).

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~11 NEAR deposit.
2. Participant B calls `propose_update` with a different 1 MB contract binary, attaching ~11 NEAR deposit.
3. Threshold participants vote for A's proposal; `vote_update` reaches threshold and calls `do_update`.
4. `do_update` removes A's entry, then calls `self.entries.clear()` (removing B's entry) and `self.vote_by_participant.clear()`.
5. No refund is issued to B. B's ~11 NEAR is permanently absorbed into the contract's balance.
6. B has no function to call to recover the deposit. The `proposed_updates()` view now returns an empty map, confirming the entry is gone. [7](#0-6) [8](#0-7)

    /// Propose update to either code or config, but not both of them at the same time.
    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
