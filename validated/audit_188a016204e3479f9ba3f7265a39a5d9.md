### Title
`available_foreign_chains` Cache Computed But Never Consulted in `verify_foreign_transaction` — (File: `crates/contract/src/lib.rs`)

---

### Summary

`register_foreign_chains_config` correctly computes and persists the `available_foreign_chains` cache using a threshold-based rule (≥ signing-threshold participants must cover a chain). However, `verify_foreign_transaction` ignores this cache entirely and instead gates on `get_supported_foreign_chains()`, which uses the old strict-intersection rule (every active participant must support the chain). The threshold-based cache is dead state: computed, stored, and never applied. A single Byzantine participant strictly below the signing threshold can therefore block all `verify_foreign_transaction` requests for any chain by registering an empty coverage list.

---

### Finding Description

**State that is saved but never applied:**

`register_foreign_chains_config` saves per-node coverage data and immediately recomputes the threshold-based available-chain set: [1](#0-0) 

`recompute_available_foreign_chains` counts, per chain, how many active participants cover it and stores the result in `self.foreign_chains.available_foreign_chains`: [2](#0-1) 

`get_available_foreign_chains` exposes this threshold-based cache: [3](#0-2) 

**The cache is never consulted when accepting requests:**

`verify_foreign_transaction` calls `get_supported_foreign_chains()` — the old intersection rule — not `get_available_foreign_chains()`: [4](#0-3) 

`get_supported_foreign_chains` requires **every** active participant to have registered the chain (`is_superset` check): [5](#0-4) 

The design document explicitly states that `verify_foreign_transaction(C)` must be rejected unless `C` is **available** (threshold-based), and that the intersection rule is to be deprecated precisely because a single non-registering node drops every chain: [6](#0-5) [7](#0-6) 

The `available_foreign_chains` cache is therefore computed and stored but never used in the actual request-acceptance gate — a direct structural analog to the boost delegation bug.

---

### Impact Explanation

**Impact: Medium** — request-lifecycle manipulation that breaks a production safety/accounting invariant without requiring network-level DoS or operator misconfiguration.

A single Byzantine participant strictly below the signing threshold can call `register_foreign_chain_support` with an empty set. Because `get_supported_foreign_chains()` requires unanimous coverage, this immediately causes the function to return an empty set, and every subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported` — regardless of how many honest participants correctly cover the chain. The threshold-based `available_foreign_chains` cache, which would correctly allow the request, is never consulted.

The broken invariant: the contract's own design mandates that a chain is serviceable when ≥ signing-threshold participants cover it. The actual gate enforces a strictly stronger (and incorrect) condition — unanimity — making the threshold-based cache permanently inert.

---

### Likelihood Explanation

Any current participant can call `register_foreign_chain_support` at any time; no threshold collusion is required. A single Byzantine participant (one of n, where n > threshold) suffices. The call is cheap, on-chain, and immediately effective. The attack is persistent until the malicious participant re-registers a non-empty list or is removed via resharing.

---

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with a check against `get_available_foreign_chains()` (the threshold-based cache). The cache is already correctly maintained by `register_foreign_chains_config` and `recompute_available_foreign_chains`; the only missing step is wiring it into the request gate:

```rust
// In verify_foreign_transaction:
let requested_chain = request.request.chain();
let available_chains = self.foreign_chains.get().available_foreign_chains.clone();
if !available_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported {
            requested: requested_chain,
        }
        .to_string(),
    );
}
```

This matches the stated design intent and makes the `available_foreign_chains` cache actually effective.

---

### Proof of Concept

1. Deploy the contract with N participants (N > threshold T).
2. All participants call `register_foreign_chains_config` covering Bitcoin → `available_foreign_chains` now contains Bitcoin; `get_available_foreign_chains()` returns Bitcoin.
3. One Byzantine participant (participant B, below threshold) calls `register_foreign_chain_support` with an empty set.
4. `get_supported_foreign_chains()` now returns empty (B's empty registration breaks unanimity).
5. Any user calls `verify_foreign_transaction` for Bitcoin → panics with `ForeignChainNotSupported`.
6. `get_available_foreign_chains()` still returns Bitcoin (T honest participants cover it), but this is never checked.

The `available_foreign_chains` cache correctly reflects that Bitcoin is serviceable; the request gate ignores it and rejects the request anyway. [8](#0-7) [9](#0-8)

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }

Today, `get_supported_foreign_chains()` returns the **strict intersection** of every
active participant's registered chains, and `verify_foreign_transaction` rejects any
