Audit Report

## Title
`stakedButUnverifiedNativeETH` Overstates TVL When a Validator Is Slashed Before Credential Verification — (File: `contracts/NodeDelegator.sol`)

## Summary
`stakedButUnverifiedNativeETH` is incremented by a fixed 32 ETH per validator on `stake32Eth()` and decremented by the same fixed amount on `verifyWithdrawalCredentials()`, with no mechanism to reflect a beacon-chain balance decrease that occurs in between. If a validator is slashed during this window, `getEffectivePodShares()` overstates the recoverable ETH, inflating the rsETH price. Any rsETH holder who redeems at the inflated price extracts more ETH than their proportional share, with the shortfall socialised to remaining holders when the credential proof is eventually submitted.

## Finding Description
**Increment (fixed 32 ETH):**
`stake32Eth()` unconditionally adds 32 ETH to the counter: [1](#0-0) 

**Decrement (also fixed 32 ETH, ignores actual balance):**
`verifyWithdrawalCredentials()` subtracts `validatorFields.length * 32 ether` — it does not read the validator's actual effective balance from the proof fields: [2](#0-1) 

**TVL aggregation:**
`getEffectivePodShares()` adds the raw counter to the EigenLayer-tracked `withdrawableShare`. Because the slashed validator is unverified, EigenLayer has no record of it and `withdrawableShare` is unaffected: [3](#0-2) 

`getETHDistributionData()` sums `getEffectivePodShares()` across all NDCs: [4](#0-3) 

`_getTotalEthInProtocol()` in `LRTOracle` calls `getTotalAssetDeposits(ETH_TOKEN)` → `getAssetDistributionData` → `getETHDistributionData()`, feeding the inflated figure directly into `_updateRsETHPrice()`: [5](#0-4) 

**Why existing checks are insufficient:**
`beaconChainSlashingFactor` is a pod-level factor updated only during checkpoint completion, which only processes **ACTIVE** (credential-verified) validators. An unverified validator is invisible to EigenLayer; its slashing never touches `beaconChainSlashingFactor` or `withdrawableShare`. There is no on-chain path by which a pre-verification slash reduces `stakedButUnverifiedNativeETH`.

**Exploit flow:**
1. Operator calls `stake32Eth()` → `stakedButUnverifiedNativeETH += 32 ether`.
2. Validator is slashed on the beacon chain (e.g., balance drops to 16 ETH). No EigenLayer state changes.
3. `getEffectivePodShares()` still returns 32 ETH for this validator; rsETH price is inflated by 16 ETH worth.
4. Attacker (any rsETH holder) observes the public beacon-chain slash event and calls the withdrawal/redemption flow at the inflated price, extracting more ETH than their proportional share.
5. Operator eventually calls `verifyWithdrawalCredentials()` → `stakedButUnverifiedNativeETH -= 32 ether`, EigenLayer registers 16 ETH. Price drops; remaining holders absorb the loss.

## Impact Explanation
**High — Theft of unclaimed yield.** The attacker, requiring no privileged access, redeems rsETH at a price inflated by the phantom 16 ETH. The extracted surplus is a direct, quantifiable transfer from remaining rsETH holders to the early redeemer. The loss is concrete and proportional to the slash magnitude and the attacker's rsETH position.

## Likelihood Explanation
Beacon-chain slashing is rare but historically documented, particularly for large operators running many validators. The unverified window is operator-controlled and can span days to weeks. The slashing event is immediately visible on the public beacon chain, giving any observer the opportunity to front-run `verifyWithdrawalCredentials()`. No privileged access is required; the attacker only needs to hold rsETH and initiate a standard withdrawal. Likelihood is **Low**, but the attack is permissionless and repeatable for each slashing event.

## Recommendation
In `verifyWithdrawalCredentials()`, read the validator's actual effective balance from the beacon-chain proof fields (available in `validatorFields` via the Beacon Chain SSZ proof) rather than subtracting a fixed 32 ETH. The difference between 32 ETH and the proven effective balance should be written off immediately — e.g., emit a loss event and reduce `stakedButUnverifiedNativeETH` by the actual proven balance rather than the fixed 32 ETH. Alternatively, maintain a separate `pendingSlashAdjustment` mapping updated by a trusted off-chain keeper and applied to `stakedButUnverifiedNativeETH` before it is included in TVL.
