Audit Report

## Title
Uninitialized `tokenFeeBps` in `addSupportedToken()` Allows Feeless Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps` that defaults to `0` for any newly added token. `addSupportedToken()` never initializes this mapping, and `setTokenFeeBps()` is a separate, independent admin call. Any deposit made between these two transactions pays zero protocol fee, permanently losing that fee revenue.

## Finding Description
The `tokenFeeBps` mapping is declared at line 88 and defaults to `0` for any key not explicitly set:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```

`addSupportedToken()` (lines 637–656) sets `supportedTokenOracle[token]`, `tokenBridge[token]`, and pushes to `supportedTokenList`, but never assigns `tokenFeeBps[token]`. The `onlySupportedToken` modifier (lines 100–103) only checks `supportedTokenOracle[token] != address(0)`, so the token is immediately depositable after `addSupportedToken()` returns.

`deposit(address token, uint256 amount, string referralId)` (lines 284–305) calls `viewSwapRsETHAmountAndFee(amount, token)` (lines 326–347), which reads `tokenFeeBps[token]` at line 335:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token]; // 0 for new token
fee = amount * feeBpsForToken / 10_000;      // fee = 0
uint256 amountAfterFee = amount - fee;        // full amount passes through
```

The depositor receives rsETH computed on the full `amount` with no fee deducted. `feeEarnedInToken[token] += 0` records nothing. The only remedy is a subsequent call to `setTokenFeeBps()` (lines 583–594), which requires `DEFAULT_ADMIN_ROLE` and is entirely separate from the `TIMELOCK_ROLE`-gated `addSupportedToken()`. There is no atomicity guarantee and no guard that blocks deposits while `tokenFeeBps[token] == 0`.

## Impact Explanation
Every deposit made in the window between `addSupportedToken()` and `setTokenFeeBps()` pays zero protocol fee. The depositor receives the full oracle-rate rsETH value with no fee withheld; the protocol permanently loses the fee revenue that should have been collected on those deposits. This is a concrete, irreversible loss of protocol fee revenue — **High: Theft of unclaimed yield**.

## Likelihood Explanation
The window is created by normal protocol operation every time a new ERC-20 token is added. Because `addSupportedToken()` uses `TIMELOCK_ROLE` (typically a timelock contract), the pending action is publicly visible on-chain before execution, giving any observer advance notice. Once the token is added, any unprivileged user can call `deposit(token, amount, referralId)` immediately. No special privileges, flash loans, or oracle manipulation are required. The attacker need only watch for the `AddSupportedToken` event and submit a deposit before `setTokenFeeBps()` is confirmed.

## Recommendation
Pass `_feeBps` as a parameter to `addSupportedToken()` and set it atomically in the same transaction:

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    // ... existing checks ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    tokenFeeBps[token] = _feeBps;   // atomic initialization
    emit AddSupportedToken(token, oracle, bridge);
}
```

This eliminates the race window entirely.

## Proof of Concept
1. Admin calls `addSupportedToken(wstETH, oracle, bridge)`. `tokenFeeBps[wstETH]` is `0`. Token is immediately depositable (`supportedTokenOracle[wstETH] != address(0)`).
2. Before admin calls `setTokenFeeBps(wstETH, 50)`, attacker calls `RSETHPool.deposit(wstETH, 1_000_000e18, "")`.
3. Inside `viewSwapRsETHAmountAndFee(1_000_000e18, wstETH)`:
   - `feeBpsForToken = tokenFeeBps[wstETH] = 0`
   - `fee = 1_000_000e18 * 0 / 10_000 = 0`
   - `amountAfterFee = 1_000_000e18`
   - `rsETHAmount = 1_000_000e18 * tokenToETHRate / rsETHToETHrate` (full value, no fee)
4. Attacker receives rsETH for the full 1,000,000 wstETH. At 50 bps, the protocol loses 5,000 wstETH worth of fee revenue on this single deposit.
5. Admin's subsequent `setTokenFeeBps(wstETH, 50)` is too late — the feeless swap is irreversible.

**Foundry test plan**: Deploy `RSETHPool` with mock oracle and wrsETH. Call `addSupportedToken`. Without calling `setTokenFeeBps`, call `deposit(token, 1e18, "")`. Assert `feeEarnedInToken[token] == 0` and that rsETH received equals the full oracle-rate conversion with no fee deducted.
