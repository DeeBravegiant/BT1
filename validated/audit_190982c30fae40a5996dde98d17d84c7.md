### Title
Direct Deposits Not Included in UTxO Balance Equation — ADA Creation from Thin Air - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxBody.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

`dijkstraTotalDepositsTxBody` omits direct deposits from the produced-value calculation. The ENTITIES rule unconditionally credits direct-deposit amounts to account balances. Because neither the consumed nor the produced side of `validateValueNotConservedUTxO` accounts for direct deposits, a transaction can pass the balance check while causing account balances to increase by an arbitrary amount, creating ADA from thin air.

---

### Finding Description

**`dijkstraTotalDepositsTxBody`** is the Dijkstra implementation of `getTotalDepositsTxBody`:

```haskell
-- TxBody.hs:1003-1007
dijkstraTotalDepositsTxBody pp isPoolRegisted txBody =
  getTotalDepositsTxCerts pp isPoolRegisted (txBody ^. certsTxBodyL)
    <+> conwayProposalsDeposits pp txBody
```

It sums only certificate deposits and proposal deposits. There is no term for `directDepositsTxBodyL`.

The **produced** value for a Dijkstra top-level transaction is computed via the chain:

```
produced  →  getProducedDijkstraValue  →  dijkstraProducedValue
          →  conwayProducedValue  →  getProducedMaryValue
          →  shelleyProducedValue  →  outputs + fee + getTotalDepositsTxBody
```

`shelleyProducedValue` (Shelley/UTxO.hs:147–150) is:
```haskell
shelleyProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <+> Val.inject (txBody ^. feeTxBodyL <+> getTotalDepositsTxBody pp isRegPoolId txBody)
```

Since `getTotalDepositsTxBody` = `dijkstraTotalDepositsTxBody` for DijkstraEra, and that function has no direct-deposit term, **direct deposits never appear in the produced value**.

The **consumed** value (`conwayConsumed` → `getConsumedDijkstraValue` → `getConsumedMaryValue`, Mary/UTxO.hs:77–86) is:
```haskell
consumedValue =
  sumUTxO (txInsFilter utxo (txBody ^. inputsTxBodyL))
    <> inject (refunds <> withdrawals)
```

**Direct deposits also never appear in the consumed value.**

The balance check in `dijkstraUtxoTransition` (Utxo.hs:380–381):
```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
```

passes as long as `UTxO inputs + refunds + withdrawals = outputs + fee + cert_deposits + proposal_deposits + treasury_donation`.

After the balance check passes, the LEDGER rule calls ENTITIES (`dijkstraEntitiesTransition`, Entities.hs:191–216):
```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

`applyDirectDeposits` (State/Account.hs:290–298) unconditionally adds each direct-deposit amount to the matching account balance:
```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
```

The only guard is `directDepositsMissingAccounts`, which only checks that target accounts are registered — it does not verify that the amounts are funded by the transaction's UTxO inputs.

---

### Impact Explanation

An unprivileged transaction author can include a non-empty `directDepositsTxBodyL` field with total value V. The UTxO balance check passes because V is absent from both sides of the equation. ENTITIES then adds V to account balances. This is V lovelace created from thin air per transaction, with no corresponding debit from UTxO, treasury, or reserves. The global ADA invariant (UTxO + accounts + treasury + reserves = constant) is violated by exactly V per transaction.

---

### Likelihood Explanation

The attack requires only:
1. One or more registered accounts (the attacker can register their own).
2. A valid Dijkstra transaction with balanced UTxO inputs/outputs/fee.
3. A non-empty `directDepositsTxBodyL` field targeting those accounts.

No governance majority, no privileged key, no special role is needed. Any node operator or wallet user submitting a Dijkstra-era transaction can exploit this.

---

### Recommendation

Add the sum of all direct-deposit amounts to the **produced** side of the balance equation. The simplest fix is to extend `dijkstraTotalDepositsTxBody` to include direct deposits:

```haskell
dijkstraTotalDepositsTxBody pp isPoolRegisted txBody =
  getTotalDepositsTxCerts pp isPoolRegisted (txBody ^. certsTxBodyL)
    <+> conwayProposalsDeposits pp txBody
    <+> foldMap id (unDirectDeposits (txBody ^. directDepositsTxBodyL))
```

This ensures that the UTxO inputs must cover the direct-deposit amounts, preserving the global ADA invariant.

### Proof of Concept

```
Pre-state:
  UTxO = { txIn₀ ↦ (addr_attacker, 10 ADA) }
  Accounts = { cred_A ↦ balance=0 }   -- registered account

Transaction:
  inputs  = { txIn₀ }                  -- 10 ADA
  outputs = { addr_attacker ↦ 9.8 ADA }
  fee     = 0.2 ADA
  certs   = []
  directDeposits = { AccountAddress Mainnet cred_A ↦ 5 ADA }

Balance check (UTXO):
  consumed = 10 ADA
  produced = 9.8 + 0.2 = 10 ADA  ✓  (direct deposits not counted)

ENTITIES:
  applyDirectDeposits adds 5 ADA to cred_A's balance

Post-state:
  UTxO = { (txId,0) ↦ (addr_attacker, 9.8 ADA) }
  Accounts = { cred_A ↦ balance=5 ADA }

Global ADA delta = +5 ADA  ← created from thin air
```
