### Title
Cross-Chain Replay of `DelegateAction` (Meta Transactions) Due to Missing Chain ID in Signing Hash - (File: `core/primitives/src/action/delegate.rs`)

### Summary
`DelegateAction` (NEP-366 meta transactions) signing hashes do not bind to any chain identifier. A `SignedDelegateAction` created for NEAR mainnet can be replayed verbatim on NEAR testnet (or any other NEAR network) by any party who possesses the signed payload, directly draining the victim's balance on the target chain.

### Finding Description
NEAR's meta-transaction mechanism (`DelegateAction`) allows a user to sign an action off-chain and hand it to a relayer, who wraps it in a regular NEAR transaction and pays the gas. The user's signature covers the NEP-461 hash of the `DelegateAction` struct.

The `DelegateAction` struct contains:

```
sender_id, receiver_id, actions, nonce, max_block_height, public_key
``` [1](#0-0) 

The signing hash is computed as `SHA256(discriminant || borsh(DelegateAction))`: [2](#0-1) 

**There is no chain ID, network ID, or genesis hash in this signed payload.** The outer NEAR transaction that wraps the `DelegateAction` does contain a `block_hash` that binds it to a specific chain, but the inner `SignedDelegateAction` itself carries no such binding.

When the runtime processes the delegate action, it verifies the signature, checks the nonce, and checks `max_block_height` — none of which are chain-specific: [3](#0-2) 

The `DelegateActionV2` variant (gas key support) has the same structure and the same absence of chain binding: [4](#0-3) 

By contrast, the Wallet Contract (ETH-implicit accounts) correctly validates `chain_id` in Ethereum transactions: [5](#0-4) 

Native NEAR transactions are protected by the `block_hash` field, which must reference a block that exists in the receiving chain's store: [6](#0-5) 

But `DelegateAction` has no equivalent protection.

### Impact Explanation
An attacker who obtains a `SignedDelegateAction` (e.g., a malicious or compromised relayer, or anyone who intercepts the off-chain payload) can wrap it in a fresh outer NEAR transaction and submit it to a different NEAR network. If the victim's account exists on that network with the same public key registered and a nonce lower than the `DelegateAction`'s nonce, the inner actions execute — directly debiting the victim's balance on that chain without their consent.

The corrupted protocol value is the victim's **account balance** on the target chain (a state root entry), and the **access key nonce** is advanced, consuming the victim's nonce slot.

### Likelihood Explanation
- NEAR developers routinely use the same account names and key pairs on both mainnet and testnet.
- `SignedDelegateAction` payloads are explicitly designed to be shared off-chain with relayers, making interception or malicious relayer behavior a realistic threat model.
- The `max_block_height` TTL does not prevent cross-chain replay because block heights on mainnet and testnet are comparable in magnitude.
- No special privileges are required: any party who receives the signed payload can act as the replaying relayer.

### Recommendation
Include the chain's genesis block hash or a canonical chain ID string in the data committed to by the `DelegateAction` signature. The simplest approach is to add a `chain_id: String` field to `DelegateAction` (and `DelegateActionV2`) and include it in the NEP-461 signing hash, mirroring how the Wallet Contract hard-codes `CHAIN_ID` into its Ethereum transaction validation. [7](#0-6) 

The genesis `chain_id` string (`"mainnet"`, `"testnet"`, etc.) is already available at runtime and would serve as an unambiguous cross-chain discriminator.

### Proof of Concept
1. Alice has account `alice.near` on both mainnet and testnet, with the same ED25519 key pair registered on both.
2. Alice signs a `DelegateAction` for mainnet: transfer 100 NEAR from `alice.near` to `bob.near`, nonce=5, max_block_height=200000000.
3. Alice hands the `SignedDelegateAction` to a relayer on mainnet. It executes; Alice's mainnet balance decreases.
4. The relayer (or an attacker who intercepted the payload) constructs a new outer NEAR transaction referencing a recent testnet block hash, wraps the **identical** `SignedDelegateAction` inside it, and submits it to testnet.
5. The testnet runtime verifies the signature (valid — same key, same hash, no chain binding), checks the nonce (valid if testnet nonce < 5), checks `max_block_height` (valid if testnet height < 200000000), and executes the transfer — draining 100 NEAR from Alice's testnet account without her authorization. pub struct DelegateAction {

    pub sender_id: AccountId,

    pub receiver_id: AccountId,

    ///
    /// With the meta transactions MVP defined in NEP-366, nested

    pub actions: Vec<NonDelegateAction>,
    /// Nonce to ensure that the same delegate action is not sent twice by a
    /// relayer and should match for given account's `public_key`.
    /// After this action is processed it will increment.
    pub nonce: Nonce,
    /// The maximal height of the block in the blockchain below which the given DelegateAction is valid.
    pub max_block_height: BlockHeight,
    /// Public key used to sign this delegated action.
    pub public_key: PublicKey,
}

pub fn check_transaction_validity_period(
    chain_store: &ChainStoreAdapter,
    prev_block_header: &BlockHeader,
    base_block_hash: &CryptoHash,
    transaction_validity_period: BlockHeightDelta,
) -> Result<(), InvalidTxError> {
    let base_header =
        chain_store.get_block_header(base_block_hash).map_err(|_| InvalidTxError::Expired)?;

    metrics::CHAIN_VALIDITY_PERIOD_CHECK_DELAY
        .observe(prev_block_header.height().saturating_sub(base_header.height()) as f64);

    // First check the distance between blocks
    if prev_block_header.height() > base_header.height() + transaction_validity_period {
        return Err(InvalidTxError::Expired);
    }

    // Then check if there is a path between the blocks (`base` is an ancestor of `prev`)
    validity_period_validate_is_ancestor(&base_header, prev_block_header, chain_store)
}
