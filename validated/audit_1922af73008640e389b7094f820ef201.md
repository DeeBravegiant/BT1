Audit Report

## Title
Permissionless `getReward()` Advances `updatedAt` on Every Call, Causing `rewardPerToken()` Increment to Round to Zero and Permanently Freeze Staker Rewards — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` uses a Synthetix-style reward accumulator where `rewardPerToken()` computes an incremental reward using integer division. The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` on every invocation, including calls to the fully permissionless `getReward()`. When `rewardRate * timeDelta * 1e18 < totalKernelStaked`, the per-token increment rounds to zero in Solidity integer arithmetic, and those rewards are permanently lost — the clock advances but `rewardPerTokenStored` does not. An unprivileged caller with zero staked balance can exploit this by calling `getReward()` every block on Arbitrum (~1 s `timeDelta`), zeroing out the entire reward budget over a distribution period.

## Finding Description

**Root cause:** `rewardPerToken()` (L408–414) computes:

```solidity
rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

When `timeDelta = 1` and `rewardRate * 1e18 < totalKernelStaked`, the numerator is less than the denominator and Solidity truncates to zero. The `updateReward` modifier (L232–242) then writes this unchanged value back to `rewardPerTokenStored` and advances `updatedAt = lastTimeRewardApplicable()` (L234) unconditionally — there is no guard requiring a non-zero increment before the clock moves forward.

`getReward()` (L382) carries `updateReward(msg.sender)` and is `external nonReentrant` with no staked-balance requirement. Any address can call it. The `nonReentrant` guard does not prevent repeated calls across blocks.

**Exploit path:**

1. Attacker deploys or uses any EOA with zero `balanceOf`.
2. Attacker calls `getReward()` once per Arbitrum block (~every 1 s of `block.timestamp`).
3. Each call: `updateReward` fires → `rewardPerToken()` returns `rewardPerTokenStored + 0` → `rewardPerTokenStored` unchanged → `updatedAt` advances by 1 s → that second's accrued rewards are permanently discarded.
4. Stakers' `earned()` never increases; reward tokens remain locked in the contract with no recovery path.

**Why existing checks are insufficient:**

- `nonReentrant` only prevents within-transaction reentrancy; it does not rate-limit cross-block calls.
- `if (rewardAmount > 0)` (L385) only gates the token transfer; `updateReward` runs unconditionally before this check.
- There is no minimum `timeDelta` or non-zero-increment guard before `updatedAt` is advanced.

## Impact Explanation

Stakers permanently lose their entitled reward tokens. The reward tokens remain locked in `KernelDepositPool` with no admin recovery function. This matches **"Permanent freezing of unclaimed yield"** (Medium) at minimum; given that the entire reward budget can be consumed over a full distribution period, it also satisfies **"Theft of unclaimed yield"** (High) in scale, as stakers are deprived of 100% of their expected yield.

## Likelihood Explanation

`getReward()` is fully permissionless and costs negligible gas on Arbitrum (estimated $0.001–$0.01 per call). The rounding condition is met whenever `rewardRate < totalKernelStaked / 1e18`, which is easily satisfied at realistic staking scales (e.g., 1 M KERNEL staked with a 2,000-token annual reward budget). Even without a deliberate attacker, organic user activity — multiple stakers calling `getReward()` in consecutive blocks — triggers the same loss. The condition becomes easier to satisfy as TVL grows, making this a systemic risk that worsens over time.

## Recommendation

1. **Only advance `updatedAt` when the increment is non-zero:** Before writing `updatedAt = lastTimeRewardApplicable()`, check that `rewardPerToken() > rewardPerTokenStored`; otherwise skip the clock advance.
2. **Accumulate remainder ("dust") into a buffer:** Track the fractional reward that rounds off each interval and add it to the next interval's numerator.
3. **Store `rewardRate` with extra precision:** Multiply `rewardRate` by `1e18` at storage time and divide at read time, shrinking the rounding window by 18 orders of magnitude.
4. **Minimum accumulation interval:** Require `block.timestamp - updatedAt >= MIN_INTERVAL` before checkpointing, preventing per-block spam.

## Proof of Concept

```
// Foundry fork/unit test outline
// Setup
totalKernelStaked = 1_000_000e18;   // 1 M KERNEL
duration          = 30 days;         // 2_592_000 s
rewardAmount      = 2_000e18;
// After notifyRewardAmount:
// rewardRate = 2_000e18 / 2_592_000 = 771_604

// Rounding check (static):
// 771_604 * 1 * 1e18 = 7.716e23 < 1e24 (totalKernelStaked) → increment = 0 ✓

// Attack loop (Arbitrum fork, warp 1 s per iteration):
address attacker; // balanceOf[attacker] == 0
for (uint i = 0; i < 2_592_000; i++) {
    vm.warp(block.timestamp + 1);
    pool.getReward(); // called as attacker
    // Assert: rewardPerTokenStored == 0 after each call
    assertEq(pool.rewardPerTokenStored(), 0);
}
// Assert: all stakers earned 0 despite full reward period elapsing
assertEq(pool.earned(staker), 0);
// Assert: reward tokens still locked in contract
assertEq(rewardsToken.balanceOf(address(pool)), 2_000e18);
```
