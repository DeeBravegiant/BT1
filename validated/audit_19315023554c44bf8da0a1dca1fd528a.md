Audit Report

## Title
ETH Deposit Limit Not Enforced Against Deposit Amount - (File: `contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its limit check, testing only whether the current total already exceeds the cap rather than whether the new deposit would push it over. Any unprivileged depositor can therefore bypass the admin-configured `depositLimitByAsset` ceiling for ETH in a single `depositETH` call, rendering the ETH deposit cap meaningless.

## Finding Description
The root cause is an asymmetric comparison in `_checkIfDepositAmountExceedesCurrentLimit` (L676–682):

```solidity
// contracts/LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The `amount` parameter (which carries `msg.value`) is silently dropped for the ETH branch. The call chain is:

```
depositETH(minRSETHAmountExpected, referralId)          [L76-93]
  └─ _beforeDeposit(ETH_TOKEN, msg.value, ...)           [L87]
       └─ _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)  [L661]
            └─ returns (totalAssetDeposits > limit)      // msg.value never tested
```

`_beforeDeposit` (L648–670) reverts with `MaximumDepositLimitReached` only when the function returns `true`. Because `amount` is excluded, the function returns `false` for any deposit made while `totalAssetDeposits < depositLimit`, regardless of how large `msg.value` is. The ERC20 path through `depositAsset` correctly includes `depositAmount` and would revert under the same scenario.

Existing guards (`nonReentrant`, `whenNotPaused`, `onlySupportedAsset`) are unrelated to the limit check and provide no mitigation.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The `depositLimitByAsset` ceiling is a risk-management control set by the MANAGER role to bound the protocol's ETH exposure in EigenLayer. Because the check ignores the incoming deposit amount, the cap is entirely unenforceable for ETH: a single depositor can push total ETH holdings to an arbitrary multiple of the configured limit in one transaction. No user funds are directly stolen or frozen, but the protocol fails to deliver the deposit-cap guarantee it is designed to enforce.

## Likelihood Explanation
**High.** The bypass condition (`totalAssetDeposits < depositLimit`) is the normal operating state of the protocol. No special timing, front-running, privileged access, or external dependency is required. Any unprivileged external caller invoking `depositETH` with a sufficiently large `msg.value` triggers the bypass unconditionally and repeatably.

## Recommendation
Include the deposit amount in the ETH branch to match the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin calls `LRTConfig.updateAssetDepositLimit(ETH_TOKEN, 100 ether)`.
2. Protocol reaches normal operating state: `getTotalAssetDeposits(ETH_TOKEN)` = 99 ether.
3. Attacker calls `LRTDepositPool.depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `99 ether > 100 ether` → `false` → no revert.
5. 10 000 ETH is accepted; `totalAssetDeposits` becomes 10 099 ether — ~100× the intended cap.
6. The equivalent ERC20 call would evaluate `99 + 10_000 > 100` → `true` → revert with `MaximumDepositLimitReached`.

**Foundry test plan:**
```solidity
function test_ethDepositBypassesLimit() public {
    vm.prank(manager);
    lrtConfig.updateAssetDepositLimit(LRTConstants.ETH_TOKEN, 100 ether);

    // Seed 99 ETH into the pool to reach near-limit state
    vm.deal(alice, 99 ether);
    vm.prank(alice);
    lrtDepositPool.depositETH{value: 99 ether}(0, "");

    // Attacker deposits 10_000 ETH — should revert but does not
    vm.deal(attacker, 10_000 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 10_000 ether}(0, ""); // passes, no revert

    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), 100 ether);
}
```
