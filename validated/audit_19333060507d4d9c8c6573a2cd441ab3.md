### Title
Single Byzantine Participant Can Forge CKD Response for Legacy `AppPublicKey` Requests — (File: `crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, the contract performs cryptographic output verification **only** for the `AppPublicKeyPV` (publicly verifiable) variant of CKD requests. For the legacy `AppPublicKey` variant, the verification block is an explicit no-op. Any single attested participant can therefore submit an arbitrary forged `CKDResponse` for a pending legacy CKD request, and the contract will accept and deliver it to the requesting user without any on-chain cryptographic check. This breaks the threshold-authorization guarantee for all legacy CKD requests.

---

### Finding Description

In `respond_ckd` at `crates/contract/src/lib.rs` lines 675–682, the response-validity check is gated on the request variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS pairing relationship between the response `(big_y, big_c)`, the app's public key pair `(pk1, pk2)`, and the master BLS public key, ensuring the output is the genuine threshold-computed result. [1](#0-0) 

For `AppPublicKey` (the legacy "privately verifiable" format), the arm is empty — the contract unconditionally proceeds to `resolve_yields_for`, delivering whatever `(big_y, big_c)` the caller supplied. [2](#0-1) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be a single attested participant — not a threshold majority. [3](#0-2) 

**Attacker-controlled entry path:**

1. A user submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)`. The request is stored in `pending_ckd_requests`. [4](#0-3) 
2. A single malicious attested participant (Byzantine, below threshold) calls `respond_ckd` with a forged `CKDResponse { big_y: <arbitrary>, big_c: <arbitrary> }`.
3. The contract passes all guards (signer check, state check, TEE check, participant check) and reaches the match block. The `AppPublicKey` arm is a no-op.
4. `resolve_yields_for` delivers the forged response to the user's pending yield. [2](#0-1) 

The analog to the external report is exact: in Cosmos, fee validation is skipped in non-CheckTx mode; here, CKD output validation is skipped for the `AppPublicKey` variant. In both cases, a single malicious actor exploits the missing check to inject a forged result.

---

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without the required participant authorization.**

The MPC network's security guarantee for CKD is that the output is the result of a threshold computation: no single participant can unilaterally determine the derived key. For `AppPublicKey` requests, this guarantee is completely absent. A single Byzantine participant can race honest nodes and deliver a forged `(big_y, big_c)` pair. The user's application receives key material that does not correspond to the actual MPC-derived secret. Any data encrypted or authenticated with the forged key is either unrecoverable (if the attacker discards the forged key) or silently compromised (if the attacker retains it). Because the `AppPublicKey` format is "privately verifiable" — requiring the app's own secret key to check — the user has no on-chain mechanism to detect the forgery.

---

### Likelihood Explanation

**High.** The attack requires only a single attested participant, not a threshold majority. The attacker must observe a pending `AppPublicKey` CKD request (visible on-chain) and submit a `respond_ckd` call before honest nodes do. Since all participants monitor the same chain state and the contract accepts the first valid `respond_ckd` that resolves the yield, a racing malicious participant has a realistic opportunity to win. No special privileges, leaked keys, or network-level capabilities are required beyond holding a valid TEE attestation.

---

### Recommendation

1. **Deprecate and reject `AppPublicKey` requests** at the contract level, requiring all new CKD requests to use `AppPublicKeyPV`. This is the cleanest fix: the publicly verifiable format already exists and provides the necessary on-chain check.
2. If `AppPublicKey` must remain supported, introduce a **threshold-consensus mechanism** for CKD responses: require at least `t` participants to submit identical `(big_y, big_c)` values before the response is delivered, mirroring the threshold guarantee of the signing flow.
3. At minimum, add prominent documentation and a contract-level warning that `AppPublicKey` CKD requests carry no on-chain integrity guarantee and are vulnerable to single-participant forgery.

---

### Proof of Concept

```rust
// 1. Legitimate user submits a legacy CKD request
contract.request_app_private_key(CKDRequestArgs {
    derivation_path: "my/path".to_string(),
    app_public_key: CKDAppPublicKey::AppPublicKey(legitimate_pk),
    domain_id: DomainId::default(),
});
// Request is now in pending_ckd_requests.

// 2. Single malicious attested participant forges a response
// (sets environment to a valid attested participant)
with_active_participant_and_attested_context(&contract);

let forged_response = CKDResponse {
    big_y: dtos::Bls12381G1PublicKey([0xde; 48]),  // arbitrary garbage
    big_c: dtos::Bls12381G1PublicKey([0xad; 48]),  // arbitrary garbage
};

// 3. Contract accepts without any cryptographic check
contract
    .respond_ckd(ckd_request, forged_response)
    .expect("accepted — no output check for AppPublicKey variant");

// 4. User receives forged key material via the yield-resume callback.
//    The real threshold-computed output is never delivered.
```

The existing test at `crates/contract/src/lib.rs:4452` (`test_respond_ckd_fails_for_attested_non_participant`) confirms that a valid attested participant **can** call `respond_ckd` successfully — it only tests that a non-participant is rejected, not that the response content is verified. [5](#0-4)
