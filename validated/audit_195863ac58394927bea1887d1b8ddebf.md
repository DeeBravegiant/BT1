### Title
Proposal Inserted into `valid_proposals` Before `ProposalFinMismatch` Commitment Check Leaves Stale Entry That Panics the Node on Subsequent Access — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

In `validate_proposal`, the batcher-executed proposal is unconditionally written into the shared `valid_proposals` cache **before** the final `ProposalFin` commitment equality check. When that check fails (`built_block != received_fin.proposal_commitment`), the function returns `Err(ProposalFinMismatch)` but the stale entry for `(height, round)` remains permanently in the map. Any subsequent attempt to insert a proposal for the same `(height, round)` — or to call `decision_reached` with a commitment that does not match the stale entry — triggers an unconditional `assert!`/`panic!` inside `BuiltProposals`, crashing the node.

### Finding Description

In `validate_proposal` the ordering is:

```rust
// Line 240-241: state mutation happens first

valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

// Line 244-247: validity check happens second
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}
``` [1](#0-0) 

`insert_proposal` stores the entry keyed by `(init.height, init.round)` and asserts no prior entry exists for that key:

```rust
assert!(
    previous.is_none(),
    "Overwriting existing proposal for height {height} round {round}; at most one \
     proposal per (height, round) is allowed"
);
``` [2](#0-1) 

When `ProposalFinMismatch` fires, `validate_and_send` propagates the error, `fin_sender` is dropped (consensus receives `Canceled`), and the spawned task exits — but the stale entry at `(height, round)` with the **batcher's commitment** is never removed. [3](#0-2) 

Two concrete panic paths follow from the stale entry:

**Path 1 — duplicate `insert_proposal` panic.** If a Byzantine proposer sends two proposals for the same round (or if a queued proposal for the same round is later dequeued and re-validated after the first attempt left a stale entry), `insert_proposal` hits `assert!(previous.is_none())` and panics.

**Path 2 — `get_proposal` commitment-mismatch panic.** `decision_reached` calls `get_proposal(&height, &round, &commitment)`, which asserts `stored_commitment == commitment`:

```rust
assert_eq!(
    stored_commitment, commitment,
    "Proposal commitment mismatch for height {height} round {round}: \
     stored={stored_commitment}, requested={commitment}"
);
``` [4](#0-3) 

If a Byzantine proposer sends a `ProposalFin` with commitment `C_net` to this validator (causing `ProposalFinMismatch`, stale entry stored with `C_batcher`) while sending the correct `C_batcher` to 2/3+ peers, consensus reaches a decision on `C_batcher`. `decision_reached(H, R, C_batcher)` is called on this node, finds the stale entry, and succeeds. However, if the proposer instead sends `C_net` to 2/3+ peers and `C_batcher` to this node, `decision_reached(H, R, C_net)` is called, the stale entry has `C_batcher ≠ C_net`, and the node panics.

The comment acknowledges the intentional ordering:

```rust
// Update valid_proposals before sending fin to avoid a race condition
// with `repropose` being called before `valid_proposals` is updated.
``` [5](#0-4) 

The race-condition concern is real, but the fix chosen leaves no cleanup path when the subsequent check fails.

The same pattern exists in `build_proposal` but is benign there because `build_proposal` has no post-insert validity check that can fail. [6](#0-5) 

### Impact Explanation

A Byzantine proposer (one of up to `f` faulty validators in a BFT network) can crash any honest validator node by sending a `ProposalFin` whose `proposal_commitment` field does not match the commitment the validator's own batcher computed. The crash occurs either at the next `insert_proposal` call for the same `(height, round)` or at `decision_reached` when the consensus-decided commitment differs from the stale entry. A crashed node cannot participate in consensus, commit blocks, or serve RPC requests until it restarts and re-syncs — a targeted liveness attack against individual sequencer nodes.

### Likelihood Explanation

Triggering requires a Byzantine proposer to deliberately send a mismatched `ProposalFin.proposal_commitment`. This is within the standard BFT threat model (up to `f < n/3` Byzantine nodes). No special privilege beyond being selected as a round proposer is needed. The `ProposalFin` is a network message with no cryptographic binding to the actual block content at the time of this check (the `TODO(matan): Switch to signature validation` comment confirms this), so forging a mismatched commitment is trivial.

### Recommendation
