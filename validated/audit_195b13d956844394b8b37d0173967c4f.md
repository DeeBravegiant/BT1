Audit Report

## Title
Stale Cross-Chain Rate Consumed Without Freshness Validation Enables Excess rsETH Minting on L2 Pools - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary

`CrossChainRateReceiver` stores the rsETH/ETH rate received over LayerZero and exposes it via `getRate()` with no staleness check. Although `lastUpdated` is recorded on every `lzReceive()` call, it is never consulted when the rate is read. All three L2 deposit pools (`RSETHPool`, `RSETHPoolV3`, `RSETHPoolNoWrapper`) consume this rate directly to compute how many rsETH tokens to issue per deposited ETH. If the rate update pipeline stalls, depositors receive more rsETH than the current protocol backing warrants, diluting all existing rsETH holders' accrued yield.

## Finding Description

`CrossChainRateReceiver.lzReceive()` records the received rate and a timestamp:

```solidity
rate = _rate;
lastUpdated = block.timestamp;
```

`getRate()` returns `rate` unconditionally — `lastUpdated` is stored but never read:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
``` [1](#0-0) 

Each pool's `getRate()` delegates directly to this oracle:

```solidity
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
``` [2](#0-1) [3](#0-2) [4](#0-3) 

All three pools' `viewSwapRsETHAmountAndFee` functions use this rate as the denominator in the minting formula:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [5](#0-4) [6](#0-5) [7](#0-6) 

Because rsETH is yield-bearing, its ETH price monotonically increases over time. A stale (old, lower) rate makes the denominator smaller than the true current rate, so depositors receive more rsETH than the protocol's actual backing justifies. The `deposit()` functions in all three pools are public and callable by any address with no access restriction. [8](#0-7) [9](#0-8) [10](#0-9) 

There is no circuit-breaker, no maximum acceptable age for `lastUpdated`, and no revert or fallback when the rate is stale anywhere in the call chain.

## Impact Explanation

**High — Theft of unclaimed yield.**

Every deposit made while the rate is stale mints excess rsETH. The excess represents a claim on protocol assets that was not earned. This dilutes the share value of all existing rsETH holders, effectively transferring their accrued staking yield to the depositor. The magnitude scales with: (a) how stale the rate is, (b) the deposit volume during the stale window, and (c) the rate of rsETH price appreciation. This maps directly to the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation

**Medium.** LayerZero message delivery is not guaranteed to be continuous. Network congestion, relayer downtime, or a gap in the rate-push cadence from `RSETHRateProvider` / `RSETHMultiChainRateProvider` can all cause the stored rate to lag behind the true L1 price. The condition is reachable by any depositor who simply calls `deposit()` during a stale window — no special privilege required, no victim mistake needed.

## Recommendation

Add a staleness guard inside `getRate()` in `CrossChainRateReceiver`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

Long term, implement a validated rate-consumption framework analogous to Chainlink's `latestRoundData` pattern: always check both the returned value (non-zero, within plausible bounds) and the timestamp before using any externally sourced rate in a minting or pricing calculation.

## Proof of Concept

1. At time T, `CrossChainRateReceiver.rate` = 1.05e18 (rsETH/ETH). `lastUpdated` = T.
2. LayerZero relayer goes offline; no further `lzReceive` calls occur.
3. At time T + 7 days, the true rsETH price on L1 is 1.06e18 (staking rewards accrued).
4. A depositor calls `RSETHPoolV3.deposit{value: 100 ether}("")`.
5. `viewSwapRsETHAmountAndFee(100e18)` calls `getRate()` → returns 1.05e18 (stale).
6. `rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.238 rsETH`.
7. Correct amount at current rate: `100e18 * 1e18 / 1.06e18 ≈ 94.340 rsETH`.
8. Depositor receives ~0.898 excess rsETH (~0.95% over-issuance), extracted from existing holders' accrued yield.
9. `getRate()` never reverted; `lastUpdated` was never checked.

**Foundry fork test plan:** Fork the target L2, deploy a mock `CrossChainRateReceiver` with `rate = 1.05e18` and `lastUpdated = block.timestamp - 7 days`, set it as `rsETHOracle` on `RSETHPoolV3`, call `deposit{value: 100 ether}("")`, and assert that `wrsETH.balanceOf(depositor)` exceeds `100e18 * 1e18 / 1.06e18` (the correct current-rate amount), confirming the over-issuance without any revert.
