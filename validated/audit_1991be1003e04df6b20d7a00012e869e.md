### Title
Unsigned Cross-Binding Between `ChunkEndorsementV2` Inner and Metadata Allows Cache Poisoning to Suppress Chunk Endorsements - (File: `core/primitives/src/stateless_validation/chunk_endorsement.rs`)

### Summary

`ChunkEndorsementV2` carries two independently signed payloads — `inner` (signed over `chunk_hash`) and `metadata` (signed over `account_id`, `shard_id`, `epoch_id`, `height_created`) — with no cryptographic binding between them. An unprivileged network peer who has observed two valid endorsements from the same validator can splice the `inner`+`signature` from one endorsement with the `metadata`+`metadata_signature` from another, producing a forged `ChunkEndorsementV2` that passes `verify()`. Sending this forged message to a block-producing node before the real endorsement arrives poisons the endorsement cache for that validator/chunk slot, causing the real endorsement to be silently dropped and the validator's stake to go uncounted.

### Finding Description

`ChunkEndorsementV2` is defined with four fields:

```
inner              (ChunkEndorsementInnerV1 = {chunk_hash, differentiator})
signature          (sig over borsh(inner))
metadata           (ChunkEndorsementMetadata = {account_id, shard_id, epoch_id, height_created})
metadata_signature (sig over borsh(metadata))
``` [1](#0-0) 

The `verify()` method checks both signatures independently:

```rust
fn verify(&self, public_key: &PublicKey) -> bool {
    let inner = borsh::to_vec(&self.inner).unwrap();
    let metadata = borsh::to_vec(&self.metadata).unwrap();
    self.signature.verify(&inner, public_key)
        && self.metadata_signature.verify(&metadata, public_key)
}
``` [2](#0-1) 

There is no hash or commitment that binds `inner` to `metadata`. The `inner` payload contains only `chunk_hash` and a static differentiator string; it does not include `shard_id`, `epoch_id`, or `height_created`. [3](#0-2) 

An attacker who observes two valid endorsements from the same validator V — one for chunk C1 (chunk_hash CH1) and one for chunk C2 (shard S2, epoch E2, height H2, chunk_hash CH2) — can construct a forged endorsement:

- `inner` = inner from C1 (chunk_hash = CH1)
- `signature` = signature from C1 (valid over inner of C1)
- `metadata` = metadata from C2 (shard S2, epoch E2, height H2, account_id V)
- `metadata_signature` = metadata_signature from C2 (valid over metadata of C2)

Both `verify()` checks pass because each signature is valid over its own payload.

### Impact Explanation

`validate_chunk_endorsement` calls `endorsement.verify()` and, if it passes, stores `(chunk_hash, signature)` in the endorsement cache keyed by `chunk_production_key()` (derived from metadata): [4](#0-3) 

The deduplication guard at the top of `process_chunk_endorsement` drops any subsequent endorsement from the same validator for the same key: [5](#0-4) 

The forged endorsement is cached under key (S2, E2, H2) with value `(CH1, sig_for_CH1)`. When the real endorsement for C2 arrives, the deduplication check finds the existing entry and silently drops it. When `collect_chunk_endorsements` filters by `chunk_hash == CH2`, the cached entry (which has CH1) is excluded: [6](#0-5) 

Validator V's stake is not counted toward C2's endorsement threshold. If enough validators are targeted this way, the chunk fails to accumulate sufficient endorsed stake and may not be included in the block.

### Likelihood Explanation

Chunk endorsements are broadcast over the p2p network. Any peer can observe them. The attacker needs two endorsements from the same validator (for any two different chunks), which is routinely available since validators endorse chunks across many shards and heights. The forged message must arrive at the block producer before the real endorsement; a well-positioned peer on the network path can achieve this. No validator, operator, or privileged access is required.

### Recommendation

Bind `inner` and `metadata` into a single signed payload so that neither component can be substituted independently. Replace the two independent signatures with a single signature over a combined hash:

```rust
fn sign_combined(inner: &ChunkEndorsementInnerV1, metadata: &ChunkEndorsementMetadata, signer: &ValidatorSigner) -> Signature {
    let combined = borsh::to_vec(&(inner, metadata)).unwrap();
    signer.sign_bytes(&combined)
}

fn verify_combined(&self, public_key: &PublicKey) -> bool {
    let combined = borsh::to_vec(&(&self.inner, &self.metadata)).unwrap();
    self.signature.verify(&combined, public_key)
}
```

This ensures that a signature over `inner` for chunk C1 cannot be paired with `metadata` for chunk C2, closing the splicing attack.

### Proof of Concept

1. Attacker peer observes `ChunkEndorsementV2` from validator V for chunk C1: extracts `(inner_C1, sig_C1)`.
2. Attacker peer observes `ChunkEndorsementV2` from validator V for chunk C2 (target slot): extracts `(metadata_C2, meta_sig_C2)`.
3. Attacker constructs forged endorsement `F = {inner: inner_C1, signature: sig_C1, metadata: metadata_C2, metadata_signature: meta_sig_C2}`.
4. `F.verify(V.public_key())` returns `true` because both independent signatures are valid.
5. Attacker sends `F` to the block-producing node before the real C2 endorsement arrives.
6. `validate_chunk_endorsement` accepts `F`; it is cached under key `(S2, E2, H2)` for account V with chunk_hash CH1.
7. Real C2 endorsement arrives; deduplication drops it (`cache already contains V at key (S2, E2, H2)`).
8. `collect_chunk_endorsements` for C2 filters out the cached entry (CH1 ≠ CH2); V's stake is not counted.
9. Repeated across enough validators, C2 fails the endorsement threshold and is not included in the block.
