Audit Report

## Title
Exact-Output Swaps Silently Deliver Less Token Than Requested — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

When an exact-output swap (`amountSpecified < 0`) is requested but the pool holds less liquidity than the requested output, both `_swapToken1ForToken0SpecifiedOutput` and `_swapToken0ForToken1SpecifiedOutput` silently cap the output to whatever is available and proceed without reverting. The `swap` interface exposes no `minAmountOut` guard, so callers receive a partial fill with no on-chain signal.

## Finding Description

In `_swapToken1ForToken0SpecifiedOutput`, the requested output is unconditionally clamped before bin iteration:

```solidity
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
if (amountOutScaled > totalAvailableToken0Scaled) {
    amountOutScaled = totalAvailableToken0Scaled;   // silent cap, no revert
}
``` [1](#0-0) 

The identical pattern exists in `_swapToken0ForToken1SpecifiedOutput`: [2](#0-1) 

The capped value is returned and propagated by `_executeSwap` as `amount0DeltaScaled` (or `amount1DeltaScaled`): [3](#0-2) 

The `swap` function then transfers the reduced amount to `recipient` and invokes the callback with the reduced deltas: [4](#0-3) 

The only existing guard after the callback is `IncorrectDelta`, which checks that the pool received sufficient *input* token — it does not verify that the *output* matched the caller's specification: [5](#0-4) 

The `swap` signature accepts only `priceLimitX64`, which guards against price movement but not against insufficient pool depth: [6](#0-5) 

A grep across all scoped contracts confirms no `minAmountOut`, `InsufficientOutput`, or partial-fill guard exists anywhere.

## Impact Explanation

A router, aggregator, or protocol that calls `swap` with `amountSpecified = -X` to obtain exactly X tokens will silently receive fewer than X tokens. Downstream logic that assumes exact-output semantics — flash loan repayment, minimum-fill enforcement for a user order, or collateral acquisition — will operate on an incorrect state with no revert and no on-chain indication of the shortfall. This constitutes broken core swap functionality causing potential loss of funds for callers relying on exact-output guarantees, meeting the Sherlock Medium threshold.

## Likelihood Explanation

Any pool with thin liquidity relative to a requested exact-output amount is vulnerable. No privileged access is required — any unprivileged caller of `swap` with `amountSpecified < 0` is affected. An attacker can also deliberately drain a pool's token balance in a prior transaction or in the same block to force a victim's exact-output swap to partially fill, making the condition actively exploitable.

## Recommendation

Add a `minAmountOut` parameter to `swap`. After `_executeSwap` returns, verify the actual output meets the caller's minimum:

```solidity
function swap(
    address recipient,
    bool zeroForOne,
    int128 amountSpecified,
    uint128 priceLimitX64,
    uint128 minAmountOut,   // NEW
    bytes calldata callbackData,
    bytes calldata extensionData
) external ...
```

After `_executeSwap`:
```solidity
if (amountSpecified < 0) {
    uint256 actualOut = zeroForOne ? uint256(-amount1Delta) : uint256(-amount0Delta);
    if (actualOut < minAmountOut) revert InsufficientOutput();
}
```

Alternatively, revert inside `_swapToken*ForToken*SpecifiedOutput` when available balance is less than the requested output, rather than silently capping it.

## Proof of Concept

```solidity
// Pool holds only 0.5e18 token0 (scaledToken0 = 0.5e18 * TOKEN_0_SCALE_MULTIPLIER)
// Caller requests exact output of 1e18 token0:
pool.swap(
    recipient,
    false,      // zeroForOne = false → buy token0 with token1
    -1e18,      // exact output: wants 1e18 token0
    0,          // no price limit
    "",         // no callbackData
    ""          // no extensionData
);
// Result:
//   _swapToken1ForToken0SpecifiedOutput caps amountOutScaled to 0.5e18 * scale (line 873-874)
//   recipient receives only 0.5e18 token0 (line 268)
//   callback invoked with amount0Delta = -0.5e18, not -1e18 (line 272)
//   IncorrectDelta check passes because token1 input was correctly received (line 275)
//   No revert, no event indicating partial fill
//   Caller's downstream logic expecting 1e18 token0 operates on wrong state
```
