Audit Report

## Title
`claimAndStake` Permanently Broken Due to Missing `STAKE_FOR_ROLE` Grant on `KernelMerkleDistributor` — (File: `contracts/KERNEL/KernelMerkleDistributor.sol`)

## Summary

`KernelMerkleDistributor.claimAndStake()` and `KernelTop100MerkleDistributor.claimAndStake()` both call `kernelDepositPool.stakeFor(account, amountToStake)`. `KernelDepositPool.stakeFor()` enforces `onlyRole(STAKE_FOR_ROLE)` against `msg.sender`. Neither distributor contract is granted `STAKE_FOR_ROLE` during its `initialize()`, so every call to `claimAndStake` reverts unconditionally. The atomic claim-and-stake user flow is permanently non-functional post-deployment.

## Finding Description

`KernelDepositPool.stakeFor()` is gated by `onlyRole(STAKE_FOR_ROLE)`:

```solidity
// KernelDepositPool.sol L296-L303
function stakeFor(address _account, uint256 _amount)
    external
    nonReentrant
    onlyRole(STAKE_FOR_ROLE)   // msg.sender must hold this role
    updateReward(_account)
```

`KernelMerkleDistributor.claimAndStake()` calls this directly:

```solidity
// KernelMerkleDistributor.sol L280-L284
uint256 amountToStake = _processClaim(index, account, cumulativeAmount, merkleProof);
IKernelDepositPool(kernelDepositPool).stakeFor(account, amountToStake); // always reverts
emit ClaimedAndStaked(index, account, amountToStake);
```

`KernelMerkleDistributor.initialize()` only sets a token approval — no `grantRole` call is made:

```solidity
// KernelMerkleDistributor.sol L224-L226
// Approve the KernelDepositPool contract to spend an unlimited amount of KERNEL tokens
kernel.forceApprove(_kernelDepositPool, type(uint256).max);
// No grantRole(STAKE_FOR_ROLE, address(this)) anywhere
```

`KernelDepositPool.initialize()` only grants `DEFAULT_ADMIN_ROLE` to `_admin` and does not pre-grant `STAKE_FOR_ROLE` to any distributor:

```solidity
// KernelDepositPool.sol L259-L271
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    _setupRole(DEFAULT_ADMIN_ROLE, _admin);
    // STAKE_FOR_ROLE is never granted here
```

The identical defect exists in `KernelTop100MerkleDistributor.claimAndStake()` at line 371, which calls `kernelDepositPool.stakeFor(user, amountToStake)`, and in its `initialize()` at line 220, which also only sets a token approval without granting `STAKE_FOR_ROLE`.

At the point of the `stakeFor` sub-call, `msg.sender` is the distributor contract address. Since neither distributor is ever granted `STAKE_FOR_ROLE`, the `onlyRole` check always fails.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The `claimAndStake` feature — the advertised atomic flow for claiming KERNEL rewards and staking them in a single transaction — is permanently non-functional. Every invocation reverts at the `stakeFor` call. Because the revert rolls back the entire transaction (including the `_processClaim` state update in `KernelMerkleDistributor`), no funds are frozen or lost. Users are forced to use the separate `claim` path and manually stake, defeating the purpose of the combined function. This matches the allowed impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation

**High.** The revert is deterministic and unconditional. Every user who calls `claimAndStake` on either `KernelMerkleDistributor` or `KernelTop100MerkleDistributor` with a valid merkle proof will receive an `AccessControl: account … is missing role …` revert. There is no code path that bypasses the `onlyRole(STAKE_FOR_ROLE)` check. The only remediation requires an out-of-band admin `grantRole` call that is not documented or enforced by the initialization code.

## Recommendation

In `KernelMerkleDistributor.initialize()` and `KernelTop100MerkleDistributor.initialize()`, after setting `kernelDepositPool`, the deployer must call `IAccessControl(_kernelDepositPool).grantRole(STAKE_FOR_ROLE, address(this))` — or alternatively, the `KernelDepositPool` admin must grant `STAKE_FOR_ROLE` to each distributor contract as a required, enforced deployment step. The token approval alone (line 226 / line 220) is insufficient; the role grant is also required for `stakeFor` to succeed.

## Proof of Concept

1. Deploy `KernelDepositPool` and `KernelMerkleDistributor` using their standard `initialize` calls.
2. Confirm `KernelMerkleDistributor` does **not** hold `STAKE_FOR_ROLE`:
   ```solidity
   kernelDepositPool.hasRole(STAKE_FOR_ROLE, address(kernelMerkleDistributor)); // returns false
   ```
3. A user with a valid merkle proof calls `kernelMerkleDistributor.claimAndStake(index, user, amount, proof)`.
4. `_processClaim` succeeds (proof valid, `account == msg.sender`, not already claimed).
5. Execution reaches `IKernelDepositPool(kernelDepositPool).stakeFor(account, amountToStake)` (line 282).
6. `KernelDepositPool.stakeFor` checks `onlyRole(STAKE_FOR_ROLE)` against `msg.sender == address(kernelMerkleDistributor)` → reverts with `AccessControl: account 0x… is missing role 0x…`.
7. The entire transaction reverts. The user's claim state is unchanged. `claimAndStake` is permanently broken for all users.
