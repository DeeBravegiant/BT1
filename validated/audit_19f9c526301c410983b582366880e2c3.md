### Title
Unprotected `init()` Initializer Allows Any Caller to Set Arbitrary Initial Participant Set — (`crates/contract/src/lib.rs`)

---

### Summary

The `init()` function in `MpcContract` carries no access-control guard. Any NEAR account can call it before the legitimate deployer does, supplying an attacker-controlled `ThresholdParameters` that installs the attacker as the sole initial participant. This is a direct structural analog to M-11: a frontrunnable initializer with no caller restriction.

---

### Finding Description

`init()` is declared `pub fn` with only the `#[init]` and `#[handle_result]` attributes. The `#[init]` attribute in `near-sdk` only enforces that the contract has not yet been initialized (state storage is empty); it does **not** restrict which account may call the function.

```rust
// Contract developer helper API
#[near]
impl MpcContract {
    #[handle_result]
    #[init]                          // ← no #[private], no predecessor check
    pub fn init(
        parameters: dtos::ThresholdParameters,
        init_config: Option<dtos::InitConfig>,
    ) -> Result<Self, Error> {
``` [1](#0-0) 

By contrast, the sibling initializer `init_running` is correctly guarded:

```rust
    #[private]          // ← only callable by the contract account itself
    #[init]
    #[handle_result]
    pub fn init_running(
``` [2](#0-1) 

The deployment workflow calls `init` in a **separate** transaction from the contract deployment:

```bash
near contract call-function as-transaction mpc-contract.test.near init \
  file-args ${init_args} ... sign-as mpc-contract.test.near ...
``` [3](#0-2) 

This creates a race window between the `DeployContract` action and the `init` call. Any account that submits a `FunctionCall(init, …)` receipt before the legitimate one wins the initialization race.

When `init` succeeds it immediately calls `TeeState::with_mocked_participant_attestations` for every participant in the supplied `ThresholdParameters`, granting each a `VerifiedAttestation::Mock(Valid)` entry in `stored_attestations`:

```rust
tee_state.stored_attestations.insert(
    tls_public_key,
    NodeAttestation {
        node_id,
        verified_attestation: VerifiedAttestation::Mock(
            attestation::MockAttestation::Valid,
        ),
    },
);
``` [4](#0-3) 

This means the attacker-supplied participants are immediately treated as attested and can vote on governance actions (`vote_add_domains`, `vote_new_parameters`, etc.) without submitting any real TEE attestation.

---

### Impact Explanation

An attacker who wins the initialization race can:

1. Supply a `ThresholdParameters` containing only their own account with `threshold = 1`.
2. Receive a mocked-valid attestation for their account automatically.
3. Call `vote_add_domains` to start key generation for any domain (Secp256k1, Ed25519, BLS12-381).
4. Run a single MPC node; because they are the only participant and the threshold is 1, they complete key generation unilaterally.
5. Issue threshold signatures on arbitrary payloads — satisfying the "Unauthorized transaction execution / threshold signature issuance" critical impact criterion.

At minimum (if the legitimate deployer notices and re-deploys quickly), the impact is participant-state corruption that breaks the production safety invariant that only vetted operators control the initial participant set — a Medium impact under the allowed scope.

---

### Likelihood Explanation

- The NEAR mempool is public; any observer can detect a `DeployContract` action targeting the MPC contract account.
- The deployment scripts issue `init` as a separate CLI invocation, leaving a non-zero block window.
- Submitting a competing `init` call requires only a funded NEAR account and knowledge of the `ThresholdParameters` schema — no privileged access, no key material.
- The attacker must also run an MPC node to exploit beyond the initialization step, raising the bar slightly, but the on-chain state corruption (wrong participant set) is achieved by the `init` call alone.

---

### Recommendation

Add a predecessor-account check to `init()` identical to the one already present on `init_running`:

```rust
#[private]   // enforces env::predecessor_account_id() == env::current_account_id()
#[init]
#[handle_result]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

Alternatively, assert the check explicitly:

```rust
assert_eq!(
    env::predecessor_account_id(),
    env::current_account_id(),
    "init must be called by the contract account itself"
);
```

This ensures only the contract account (whose full-access key the deployer controls) can initialize the participant set, eliminating the frontrunning window.

### Proof of Concept

1. Observe a `DeployContract` action targeting `v1.signer` on NEAR mainnet/testnet.
2. Immediately submit:
   ```json
   {
     "parameters": {
       "threshold": 1,
       "participants": {
         "next_id": 1,
         "participants": [
           ["attacker.near", 0, {"tls_public_key": "<attacker_ed25519_pk>", "url": "https://attacker.example"}]
         ]
       }
     }
   }
   ```
   as a `FunctionCall` to `init` on the contract account, signed by `attacker.near`.
3. If the receipt lands before the legitimate `init`, the contract initializes with `attacker.near` as the sole participant.
4. `TeeState::with_mocked_participant_attestations` immediately stores a `MockAttestation::Valid` for `attacker.near`.
5. `attacker.near` can now call `vote_add_domains` and, running a single MPC node, complete key generation with threshold 1 — issuing signatures without any legitimate participant authorization. [5](#0-4) [6](#0-5)

impl TeeState {
    /// Creates a [`TeeState`] with an initial set of participants that will receive a valid mocked attestation.
    pub(crate) fn with_mocked_participant_attestations(participants: &Participants) -> Self {
        let mut tee_state = Self::default();

        for (account_id, _, participant_info) in participants.participants() {
            let tls_public_key = participant_info.tls_public_key.clone();
            // TODO(#1087): replace account_public_key with a real account public
            // key passed in by the caller. `Participants` does not currently
            // carry the operator's account public key, so a mocked entry
            // cannot record the real one and we use the TLS key as a unique
            // per-participant placeholder. The mock keeps the
            // participant from being kicked out of an empty `TeeState` until
            // a real `submit_participant_info` call replaces it (keyed by
            // TLS), but any caller-facing check that compares
            // `signer_account_pk` against the stored key will fail until
            // then. #1087 tracks threading real attestations through
            // initialization so this sentinel can go away.
            let node_id = NodeId {
                account_id: account_id.clone(),
                tls_public_key: tls_public_key.clone(),
                // Use tls_public_key as account_public_key instead of hardcoded
                // Ed25519PublicKey::from([0u8; 32]) so that same account public
                // key isn't associated with different tls keys.
                // This is not a fix for above issue: #1087, which should be
                // addressed outside this PR.
                account_public_key: tls_public_key.clone(),
            };

            tee_state.stored_attestations.insert(
                tls_public_key,
                NodeAttestation {
                    node_id,
                    verified_attestation: VerifiedAttestation::Mock(
                        attestation::MockAttestation::Valid,
                    ),
                },
            );
        }

        tee_state
    }
