### Title
Stale TEE Attestation State Read in `respond`/`vote_*` Without Re-Verification — (`File: crates/contract/src/tee/tee_state.rs`, `crates/contract/src/lib.rs`)

---

### Summary

`is_caller_an_attested_participant`, called by every node-facing mutating method (`respond`, `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, `vote_reshared`, `start_reshare_instance`, `vote_abort_key_event_instance`), checks only that a stored attestation *exists* and that the stored keys *match* the caller. It never re-verifies whether the attestation is still *valid* (not expired, docker-image hash still allowed, measurement still accepted). The `accept_requests` flag — the only other TEE gate on `respond*` — is a cached boolean set exclusively by `verify_tee()`, which is never called automatically before any of those methods. Both reads are therefore stale, directly mirroring the Fuji Protocol pattern where `deposit` and `paybackLiq` consumed indexes without first calling `updateF1155Balances`.

---

### Finding Description

`MpcContract::assert_caller_is_attested_participant_and_protocol_active` delegates to `TeeState::is_caller_an_attested_participant`:

```rust
// crates/contract/src/tee/tee_state.rs  lines 469-498
pub(crate) fn is_caller_an_attested_participant(
    &self,
    participants: &Participants,
) -> Result<(), AttestationCheckError> {
    let info = participants.info(&signer_id)
        .ok_or(AttestationCheckError::CallerNotParticipant)?;
    let attestation = self.stored_attestations.get(&info.tls_public_key)
        .ok_or(AttestationCheckError::AttestationNotFound)?;
    // checks account_id and account_public_key match — nothing else
    Ok(())
}
```

The function confirms the attestation *record* exists and the stored keys match the caller. It never calls `reverify_participants`, which is the only path that re-runs expiry and allowed-image checks:

```rust
// crates/contract/src/tee/tee_state.rs  lines 206-232

        time_stamp_seconds,
        &allowed_mpc_docker_image_hashes,
        &allowed_launcher_compose_hashes,
        &allowed_measurements,
    ) { ... }
}
```

`reverify_participants` is only invoked inside `verify_tee()` and `vote_new_parameters()`. Neither is called automatically before `respond`, `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, `vote_reshared`, or `start_reshare_instance`.

The second stale read is `accept_requests`. Every `respond*` method gates on it:

```rust
// crates/contract/src/lib.rs  lines 579-581
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
```

But `accept_requests` is only written by `verify_tee()`. Between two `verify_tee()` calls — which are externally triggered, not automatic — the flag is frozen at its last value even if attestations have since expired or the allowed docker-image list has changed.

---

### Impact Explanation

A node whose TEE attestation has expired (certificate lifetime elapsed, docker-image hash removed from the allow-list, or measurement revoked) retains the ability to call `respond`, `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, and `vote_reshared` as long as:

1. `verify_tee()` has not been called since the attestation became invalid (so `accept_requests` is still `true`), and  
2. The stale attestation record is still present in `stored_attestations` (it is never auto-evicted).

Both conditions are trivially satisfied in any window between periodic `verify_tee()` invocations. The node can therefore submit threshold-signing responses and governance votes while no longer provably running inside a valid TEE, breaking the core safety invariant that every signing participant is hardware-attested. This constitutes participant-state and contract execution-flow manipulation that breaks production safety invariants.

**Allowed impact match:** Medium — participant-state manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

TEE attestation certificates have finite lifetimes (typically 30–90 days for Intel TDX PCK certificates). The `verify_tee()` call is not automatic; it must be triggered by a participant. Any gap between attestation expiry and the next `verify_tee()` call is an open window. An adversarial or simply stale node can exploit this window without any special privilege — it only needs its existing NEAR account key and the expired-but-still-stored attestation record.

---

### Recommendation

**Short term:** Inside `is_caller_an_attested_participant`, after confirming the record exists and the keys match, call `reverify_participants` on the retrieved `NodeId` and return `Err` if the result is `TeeQuoteStatus::Invalid`. This ensures every node-facing mutating call re-validates attestation freshness at the point of use, exactly as `updateF1155Balances` should be called before every index-reading operation in the Fuji analogy.

**Long term:** Replace the cached `accept_requests` boolean with an inline re-verification call so that the TEE gate cannot be bypassed by a stale flag. Alternatively, enforce a maximum interval between `verify_tee()` calls at the contract level (e.g., reject `respond` if `verify_tee()` has not been called within the last N blocks), making the freshness requirement structurally impossible to forget.

---

### Proof of Concept

1. Network is Running; all nodes have valid attestations; `verify_tee()` was last called at block B.
2. At block B+1, the allowed docker-image hash list is updated (old hash removed via governance vote). Node X's attestation now references the removed hash.
3. `verify_tee()` is not called again (no participant triggers it).
4. `accept_requests` remains `true` (stale).
5. Node X calls `respond(request, signature)`. The call reaches `assert_caller_is_attested_participant_and_protocol_active` → `is_caller_an_attested_participant`: the record exists, keys match → `Ok(())`. Then `accept_requests` is checked: still `true` → passes. The signature is accepted and the yield is resolved.
6. Node X has successfully submitted a threshold signature response while running software whose image hash is no longer trusted by the network, with no on-chain rejection.

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
