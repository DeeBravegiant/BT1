### Title
Identity-Point `AppPublicKeyPV` Bypasses ElGamal Blinding, Leaking `msk·H(app_id)` On-Chain — (`crates/contract/src/primitives/ckd.rs`, `crates/threshold-signatures/src/confidential_key_derivation/protocol_pv.rs`)

### Summary

An unprivileged caller can submit a `request_app_private_key` CKD request with `pk1 = G1_identity` and `pk2 = G2_identity`. Every validation gate — the contract-side `app_public_key_check`, the node-side `app_public_key_check`, the node-side `aggregated_output_check`, and the contract-side `ckd_output_check` — passes for the identity pair. Because the ElGamal blinding term `y · pk1` collapses to the group identity, the coordinator's aggregated output `big_c` equals the unencrypted derived key `msk · H(app_id)`, which is then written on-chain in the `respond_ckd` transaction and is readable by any observer.

### Finding Description

**Step 1 — Contract-side `app_public_key_check` passes for identity inputs**

`app_public_key_check` verifies `e(pk1, −g2) · e(g1, pk2) = 1`.
With `pk1 = G1_identity` and `pk2 = G2_identity`, both pairings evaluate to 1 (pairing with the identity is always 1), so the product is 1 and the check passes. This is explicitly documented by the test `app_public_key_check__should_accept_identity_key_pair`. [1](#0-0) [2](#0-1) 

**Step 2 — Node-side `app_public_key_check` passes for identity inputs**

The node-side check in `protocol_pv.rs` first calls `check_valid_point_g1(pk1)` and `check_valid_point_g2(pk2)`. The identity element is on the curve and is torsion-free (it is the neutral element of the prime-order subgroup), so both return `true`. The subsequent pairing check is identical to Step 1 and also passes. [3](#0-2) [4](#0-3) 

**Step 3 — ElGamal blinding collapses to zero**

In `compute_signature_share`, each node computes:
```
big_c = big_s + app_pk.pk1 * y
```
With `pk1 = G1_identity`, `app_pk.pk1 * y = 0`, so `big_c = big_s = x_i · H(app_id)`. After Lagrange interpolation the coordinator aggregates `big_c = msk · H(app_id)` — the unencrypted derived key. [5](#0-4) 

**Step 4 — `aggregated_output_check` passes for the unencrypted output**

The coordinator checks `e(big_c, −g2) · e(big_y, pk2) · e(H, pk) = 1`. With `pk2 = G2_identity`, the middle term is 1. The equation reduces to `e(big_c, g2) = e(H, pk)`. Since `big_c = msk·H`, this is `e(msk·H, g2) = e(H, msk·g2) = e(H, pk)`, which is always true. The check passes and the protocol completes without error. [6](#0-5) 

**Step 5 — Contract-side `ckd_output_check` passes and `big_c` is stored on-chain**

The contract applies the identical pairing equation. For the same algebraic reason it passes, and the response `(big_y, big_c)` — where `big_c = msk · H(app_id)` — is accepted and emitted on-chain. [7](#0-6) 

### Impact Explanation

`msk · H(app_id)` is the app's derived private key — the exact secret the CKD protocol is designed to deliver only to the legitimate app running inside a TEE. By submitting identity points, any unprivileged caller strips the ElGamal encryption layer entirely. The derived key is written in plaintext into the `respond_ckd` on-chain transaction, where it is permanently visible to every observer. The attacker can then use this key to impersonate the target app for any operation that key authorizes (e.g., BLS signing under the derived key), without ever running a TEE or holding the app's private scalar.

This matches the Critical impact category: **unauthorized access to secret material that materially enables forgery or secret recovery**.

### Likelihood Explanation

The attack requires only a standard NEAR transaction calling `request_app_private_key` with a well-formed identity-point payload. No threshold collusion, TEE access, or privileged role is needed. The identity-point encoding is valid and accepted by the NEAR host functions. The test `app_public_key_check__should_accept_identity_key_pair` confirms the behavior is reproducible in the existing test environment. [1](#0-0) 

### Recommendation

Add an explicit non-identity check in both `app_public_key_check` functions (contract and node) before the pairing check:

- In `crates/contract/src/primitives/ckd.rs`: after decompressing `pk1` and `pk2`, verify neither is the identity encoding.
- In `crates/threshold-signatures/src/confidential_key_derivation/protocol_pv.rs` (`app_public_key_check`): add `if app_pk.pk1.is_identity().into() || app_pk.pk2.is_identity().into() { return false; }` before the pairing check.

The pairing equation `e(a·G1, g2) = e(g1, a·G2)` is satisfied by `a = 0` (identity) as a degenerate case; the fix is to explicitly exclude it.

### Proof of Concept

1. Encode `G1_identity` in compressed BLS12-381 G1 format (48 bytes, flag byte `0xC0`, rest zero).
2. Encode `G2_identity` in compressed BLS12-381 G2 format (96 bytes, flag byte `0xC0`, rest zero).
3. Submit:
   ```json
   {
     "request": {
       "derivation_path": "any",
       "app_public_key": {
         "AppPublicKeyPV": {
           "pk1": "bls12381g1:<base58 of identity>",
           "pk2": "bls12381g2:<base58 of identity>"
         }
       },
       "domain_id": <bls12381 CKD domain>
     }
   }
   ```
4. Observe the `respond_ckd` transaction. Read `big_c` from the emitted response.
5. Verify: `e(big_c, g2) == e(H(app_id, network_pk), network_pk_g2)`. This equality holds iff `big_c = msk · H(app_id)`, confirming the derived key is exposed in plaintext. [8](#0-7) [9](#0-8)
