### Title
Block-Level Reference Script Size Limit Bypass via Subtransactions in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the block-level reference script size check (`validateBodyRefScriptsSizeTooBig`) uses `totalRefScriptSizeInBlock`, which calls `txNonDistinctRefScriptsSize` per top-level transaction. This function only examines the top-level transaction's own inputs. However, Dijkstra introduces subtransactions (embedded via `subTransactionsTxBodyL`), each of which can independently reference large Plutus scripts. The block-level check does not account for reference scripts consumed by subtransactions, while the per-transaction check correctly uses `batchNonDistinctRefScriptsSize` (which includes subtransactions). An unprivileged transaction sender can exploit this asymmetry to force nodes to deserialize more reference script bytes per block than `ppMaxRefScriptSizePerBlock` is intended to allow.

---

### Finding Description

**Root cause — block-level check omits subtransaction reference scripts:**

`dijkstraBbodyTransition` delegates the block-level reference script size check to the Conway-era function:

```haskell
Conway.validateBodyRefScriptsSizeTooBig @era pp blockBody (ls ^. utxoL)
``` [1](#0-0) 

`validateBodyRefScriptsSizeTooBig` calls `totalRefScriptSizeInBlock`, which iterates over the block's top-level transactions and calls `txNonDistinctRefScriptsSize` for each:

```haskell
accum (!accUtxo, !accSum) tx =
  ...
  in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
``` [2](#0-1) 

`txNonDistinctRefScriptsSize` only inspects the top-level transaction's own `inputsTxBodyL` and `referenceInputsTxBodyL`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

Subtransactions embedded in the top-level tx body are entirely ignored.

**Contrast with the per-transaction check:**

`validateAllRefScriptSize` in the Dijkstra LEDGER rule correctly uses `batchNonDistinctRefScriptsSize`, which sums reference scripts from both the top-level tx and all its subtransactions:

```haskell
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
``` [4](#0-3) 

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [5](#0-4) 

The per-tx check is correct; the block-level check is not.

**Protocol parameter values (Dijkstra):**

`ppMaxRefScriptSizePerTx` = 204,800 bytes (200 KiB); `ppMaxRefScriptSizePerBlock` = 1,048,576 bytes (1 MiB). [6](#0-5) 

---

### Impact Explanation

An attacker crafts a top-level transaction whose own inputs carry zero or minimal reference scripts, but whose subtransactions each reference up to `maxRefScriptSizePerTx` bytes of Plutus scripts. The per-tx check (`validateAllRefScriptSize`) enforces the combined limit correctly and the transaction is accepted. However, the block-level check (`validateBodyRefScriptsSizeTooBig`) sees only the top-level tx's contribution (near zero), so it never fires regardless of how many such transactions are packed into a block.

The actual reference script deserialization burden per block becomes bounded by `maxRefScriptSizePerTx × (block capacity / min tx size)` rather than `maxRefScriptSizePerBlock`. With default parameters this can substantially exceed the 1 MiB block limit, allowing an attacker to force nodes to deserialize significantly more script bytes per block than the protocol parameter was designed to permit.

This matches the allowed impact: **Medium — attacker-controlled transactions exceed intended validation limits.**

---

### Likelihood Explanation

The Dijkstra era is the next production era. Any unprivileged user who can submit a valid transaction can exploit this: no special keys, governance role, or stake threshold is required. The attacker only needs to construct a top-level transaction with subtransactions that reference large Plutus scripts stored in UTxO outputs. The per-tx limit still bounds each individual transaction, but the block-level guard is rendered ineffective for subtransaction reference scripts.

---

### Recommendation

Replace the call to `Conway.validateBodyRefScriptsSizeTooBig` in `dijkstraBbodyTransition` with a Dijkstra-aware variant that uses `batchNonDistinctRefScriptsSize` (or an equivalent block-level accumulation) when computing the total reference script size per block. Specifically, `totalRefScriptSizeInBlock` should be extended to sum subtransaction reference scripts for each top-level transaction, mirroring the logic already present in `batchNonDistinctRefScriptsSize`.

---

### Proof of Concept

1. Deploy several large Plutus scripts as UTxO reference outputs (each near `maxRefScriptSizePerTx / N` bytes, where N is the number of subtransactions per top-level tx).
2. Construct a top-level Dijkstra transaction whose own inputs reference no scripts, but which embeds N subtransactions each referencing one of the large scripts. Ensure the combined size stays at or just below `maxRefScriptSizePerTx` (200 KiB) so `validateAllRefScriptSize` passes.
3. Pack multiple such transactions into a single block. Each passes the per-tx check; the block-level check sees ~0 bytes of reference scripts from the top-level tx inputs and does not fire.
4. The block is accepted by honest nodes even though the total reference script deserialization work far exceeds `maxRefScriptSizePerBlock` (1 MiB), violating the intended resource bound and enabling a sustained resource-exhaustion attack against block-validating nodes.
