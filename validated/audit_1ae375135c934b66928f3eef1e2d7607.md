Audit Report

## Title
Stranded ETH from Payable Non-WETH Calls Is Capturable by Any Subsequent WETH Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
Every user-facing entry point in `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` is marked `payable` with no guard rejecting `msg.value > 0` when the input token is not WETH. The `pay()` helper in `PeripheryPayments` unconditionally consumes `address(this).balance` for any WETH payment, without verifying that the ETH was intentionally deposited by the current caller. ETH accidentally sent alongside a non-WETH call is permanently stranded and fully capturable by the next caller who performs a WETH-input swap or liquidity add.

## Finding Description
`PeripheryPayments.pay()` contains the following branch for WETH payments:

```solidity
// PeripheryPayments.sol lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

When `address(this).balance >= value`, the contract wraps its own accumulated ETH and forwards it to the pool. The `payer` field is completely bypassed — no `transferFrom` is issued against the actual caller. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only applies to plain ETH transfers with no calldata; it does not prevent ETH from entering the contract via `msg.value` in any of the `payable` function calls.

All four swap entry points in `MetricOmmSimpleRouter` are `payable` with no `msg.value == 0` enforcement when `tokenIn != WETH`:
- `exactInputSingle` (line 67)
- `exactInput` (line 92)
- `exactOutputSingle` (line 130)
- `exactOutput` (line 154)

All four liquidity entry points in `MetricOmmPoolLiquidityAdder` are similarly `payable` with no enforcement:
- `addLiquidityExactShares(pool, owner, ...)` (line 56)
- `addLiquidityExactShares(pool, salt, ...)` (line 71)
- `addLiquidityWeighted(pool, owner, ...)` (line 88)
- `addLiquidityWeighted(pool, salt, ...)` (line 123)

Exploit path:
1. Victim calls `exactInputSingle{value: 1 ether}(tokenIn = USDC, ...)`. The swap executes normally via `IERC20(USDC).safeTransferFrom`. The 1 ETH is silently retained by the router with no revert and no automatic refund.
2. Attacker calls `exactInputSingle{value: 0}(tokenIn = WETH, amountIn = 1e18, ...)`. The pool triggers `metricOmmSwapCallback`, which calls `_justPayCallback`, which calls `pay(WETH, attacker, pool, 1e18)`.
3. `address(this).balance == 1e18 >= 1e18`, so the contract wraps the victim's ETH into WETH and transfers it to the pool. The attacker receives the full swap output at zero cost.

`refundETH()` exists but is a separate, optional call that does not protect against front-running between the victim's transaction and the attacker's transaction.

## Impact Explanation
Direct loss of user ETH principal. Any ETH accidentally sent with a non-WETH call is permanently capturable by the next WETH-input caller. The attacker receives the full swap output (or liquidity position) funded entirely by the victim's ETH, at zero cost to themselves. This meets the Sherlock threshold for direct loss of user principal.

## Likelihood Explanation
Low. The attack requires a user error: sending ETH alongside a non-WETH call. However, this is a realistic mistake because all entry points are `payable` and the WETH-via-ETH pattern is explicitly supported and documented. Once stranded ETH appears on-chain, front-running it is trivial and requires no special privileges — any unprivileged caller can execute the theft in the same block.

## Recommendation
1. **Primary fix**: Add `if (msg.value > 0 && tokenIn != WETH) revert ETHNotAccepted();` at the top of each swap and liquidity function that does not use native ETH.
2. **Alternative**: Remove `payable` from functions that have no WETH leg. For functions that may or may not use ETH, enforce `msg.value == 0` when the relevant token is not WETH.
3. **Defense-in-depth**: In `pay()`, assert that `address(this).balance` used for WETH wrapping equals `msg.value` of the current top-level call, or track intentionally deposited ETH separately.

## Proof of Concept
```
Setup:
  - Pool A: token0 = USDC, token1 = DAI  (no WETH leg)
  - Pool B: token0 = WETH, token1 = DAI

Step 1 — Victim:
  victim calls exactInputSingle{value: 1 ether}(
      pool = Pool_A, tokenIn = USDC, tokenOut = DAI, amountIn = 1000e6, ...
  )
  → swap executes via USDC.transferFrom; 1 ETH remains in MetricOmmSimpleRouter

Step 2 — Attacker (same or next block):
  attacker calls exactInputSingle{value: 0}(
      pool = Pool_B, tokenIn = WETH, tokenOut = DAI, amountIn = 1e18, ...
  )
  → pool calls metricOmmSwapCallback → _justPayCallback
  → pay(WETH, attacker, pool, 1e18)
  → address(this).balance == 1e18 >= 1e18
  → contract wraps victim's 1 ETH → 1 WETH → transfers to pool
  → attacker receives DAI output; attacker spent 0 ETH and 0 WETH

Result: Victim lost 1 ETH. Attacker gained DAI equivalent of 1 WETH at zero cost.
```
