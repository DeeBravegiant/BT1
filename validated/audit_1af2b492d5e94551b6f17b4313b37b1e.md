### Title
Shard-Layout Version Mismatch in `spice_shard_congestion_info` Silently Bypasses Congestion Control for Child Shards at Resharding Epoch Boundary — (`chain/chain/src/spice/chunk_application.rs`)

---

### Summary

During the first blocks of a new epoch following a dynamic resharding event, `spice_shard_congestion_info` is called in `rpc_handler.rs` with the **current (new) epoch's** `ShardLayout` but the **last certified block header**, which belongs to the **old epoch**. Because the child shard did not exist in the old epoch, both the `ChunkExtra` lookup (keyed by new-layout `ShardUId`) and the `get_execution_result_from_store` fallback (keyed by `(old_block_hash, child_shard_id)`) return nothing. The function silently returns `None`. `validate_tx` receives `receiver_congestion_info = None` and skips the congestion gate entirely. The chunk producer's `congestion_control_accepts_transaction` also returns `true` for the same reason. Transactions to a fully-congested child shard are admitted at the RPC level, selected by the chunk producer, and executed.

---

### Finding Description

**Step 1 — RPC handler builds mismatched inputs.**

In `rpc_handler.rs` lines 175–198:

```rust
let epoch_id = self.epoch_manager.get_epoch_id_from_prev_block(&head.last_block_hash)?;
let shard_layout = self.epoch_manager.get_shard_layout(&epoch_id)?;          // NEW epoch layout
let receiver_shard = shard_layout.account_id_to_shard_id(receiver_id);       // child shard id
let spice_certified_header = Some(get_last_certified_block_header(...)?);     // OLD epoch block
spice_shard_congestion_info(&chain_store, &shard_layout, certified_header, receiver_shard)
```

`shard_layout` is the **new epoch's** layout (post-resharding). `certified_header` is the last certified block, which — because Spice gates epoch transitions on certification — is necessarily from the **old epoch** when we are in the first blocks of the new epoch. [1](#0-0) 

**Step 2 — `spice_shard_congestion_info` constructs a `ShardUId` with the wrong layout version.**

```rust
let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, shard_layout); // new layout version
if let Ok(chunk_extra) = chunk_store.get_chunk_extra(block_header.hash(), &shard_uid) { ... }
else {
    get_execution_result_from_store(chain_store, block_header.hash(), shard_id)?  // also None
        .chunk_extra.congestion_info()
};
```

`block_header.hash()` is the certified block from the old epoch. The child shard did not exist in the old epoch, so:
- `get_chunk_extra(old_block_hash, new_shard_uid)` → `Err` (no entry; the old epoch stored ChunkExtras under the old-layout `ShardUId` for the parent shard, not the child).
- `get_execution_result_from_store(old_block_hash, child_shard_id)` → `None` (no execution result for a shard that did not exist).

The function returns `None`. [2](#0-1) 

**Step 3 — `validate_tx` skips the congestion gate.**

```rust
if let Some(congestion_info) = receiver_congestion_info {   // None → entire block skipped
    ...
    if let ShardAcceptsTransactions::No(reason) = congestion_control.shard_accepts_transactions() {
        return Err((err, signed_tx));
    }
}
validate_transaction(runtime_config, signed_tx, current_protocol_version)  // always reached
```

With `receiver_congestion_info = None`, the `if let Some(...)` arm is never entered. The transaction passes regardless of the child shard's actual congestion level. [3](#0-2) 

**Step 4 — Chunk producer also bypasses congestion for child shards.**

`spice_block_congestion_info` (called by the chunk producer) uses `block_header.epoch_id()` to derive the shard layout:

```rust
let shard_layout = epoch_manager.get_shard_layout(block_header.epoch_id())?; // OLD epoch
for shard_id in shard_layout.shard_ids() { ... }  // iterates parent shards only
```

Child shard IDs are absent from the resulting `BlockCongestionInfo`. In `congestion_control_accepts_transaction`:

```rust
let congestion_info = prev_block.congestion_info.get(&receiving_shard); // None for child shard
let Some(congestion_info) = congestion_info else { return Ok(true); };  // always true
``` [4](#0-3) [5](#0-4) 

**Mitigating boundary case:** The resharding manager *does* save a `ChunkExtra` for child shards at the resharding block's hash under `new_shard_uid`. If the certified block happens to be exactly the resharding block (last block of old epoch), the `get_chunk_extra` call succeeds and congestion is correctly enforced. The bypass only occurs when the certified block is *before* the resharding block — which is the common case under any execution delay, as confirmed by the existing test `test_spice_certified_results_across_resharding`. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

An unprivileged user submitting transactions via the public RPC to an account whose receiver maps to a newly created child shard, during the window between the epoch boundary and the advancement of the last certified block past the resharding block, will have those transactions:

1. Admitted by `validate_tx` (congestion gate skipped).
2. Selected by the chunk producer (child shard absent from `BlockCongestionInfo`).
3. Executed during chunk application (transactions already in the chunk are applied; the per-transaction congestion rejection in `congestion_control_accepts_transaction` is only called during chunk *production*, not chunk *application*).

This is a complete bypass of the receiver-shard congestion control for child shards at the resharding boundary. The child shard can receive and execute more transactions than the protocol intends, worsening its congestion state and delaying legitimate traffic.

---

### Likelihood Explanation

- Requires dynamic resharding to be active and an epoch transition to occur — a scheduled, deterministic event.
- Requires any non-zero execution delay (Spice's normal operating mode; the epoch gating mechanism guarantees the certified block lags the head by design).
- No special privileges needed: any account with a receiver in the child shard can trigger this via a standard signed transaction over the public RPC.
- The window lasts until the certified block advances past the resharding block, which under normal operation is a matter of seconds to minutes.

---

### Recommendation

In `rpc_handler.rs`, derive `shard_layout` for the congestion lookup from the **certified header's epoch**, not the current head's epoch:

```rust
let certified_shard_layout = epoch_manager.get_shard_layout(certified_header.epoch_id())?;
let receiver_shard_for_congestion =
    certified_shard_layout.account_id_to_shard_id(receiver_id);
// If receiver maps to a child shard not present in the certified epoch,
// resolve to the parent shard using get_parent_shard_id.
```

Alternatively, add an explicit cross-epoch resolution in `spice_shard_congestion_info`: when the `shard_id` does not exist in the certified block's epoch layout, resolve it to its parent shard and look up the parent's `ChunkExtra` instead of returning `None`.

---

### Proof of Concept

1. Configure a test-loop with dynamic resharding enabled and `execution_delay = epoch_length - 1` (so the certified block is always before the resharding block when the new epoch starts).
2. At the first block of the resharded epoch, assert `get_last_certified_block_header` returns a block from the old epoch whose height is less than the resharding block's height.
3. Artificially set the parent shard's `ChunkExtra` congestion info to full congestion (delayed receipts gas at maximum).
4. Submit a transaction whose receiver maps to a child shard.
5. Assert `ProcessTxResponse` is not `InvalidTx(ShardCongested {...})` — the transaction is admitted.
6. Assert the transaction appears in the next chunk and is executed.

The existing test infrastructure in `test-loop-tests/src/tests/spice/resharding.rs` already confirms that the certified block is in the old epoch during the first blocks of the resharded epoch, making step 2 trivially satisfiable. [8](#0-7)

pub fn spice_block_congestion_info(
    chain_store: &ChainStoreAdapter,
    epoch_manager: &dyn EpochManagerAdapter,
    block_header: &BlockHeader,
) -> Result<BlockCongestionInfo, Error> {
    let shard_layout = epoch_manager.get_shard_layout(block_header.epoch_id())?;
    let mut result = BTreeMap::new();
    for shard_id in shard_layout.shard_ids() {
        if let Some(extended) =
            spice_shard_congestion_info(chain_store, &shard_layout, block_header, shard_id)
        {
            result.insert(shard_id, extended);
        }
    }
    Ok(BlockCongestionInfo::new(result))
}

pub fn spice_shard_congestion_info(
    chain_store: &ChainStoreAdapter,
    shard_layout: &ShardLayout,
    block_header: &BlockHeader,
    shard_id: ShardId,
) -> Option<ExtendedCongestionInfo> {
    let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, shard_layout);
    let chunk_store = chain_store.chunk_store();
    let congestion_info =
        if let Ok(chunk_extra) = chunk_store.get_chunk_extra(block_header.hash(), &shard_uid) {
            chunk_extra.congestion_info()
        } else {
            get_execution_result_from_store(chain_store, block_header.hash(), shard_id)?
                .chunk_extra
                .congestion_info()
        };
    Some(ExtendedCongestionInfo::new(congestion_info, 0))
}
