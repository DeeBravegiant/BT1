### Title
Stale Committee Hot Credential Vote Counted for New Cold Credential Authorization - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

### Summary

In the Conway era, a committee member's cold credential can authorize any hot credential via `ConwayAuthCommitteeHotKey`. There is no check preventing multiple cold credentials from authorizing the same hot credential, and there is no cleanup of `gasCommitteeVotes` when a hot credential is re-authorized by a new cold credential. As a result, a stale YES vote cast by a hot credential under one cold credential's authorization is counted again when a different cold credential later authorizes the same hot credential — enabling vote amplification or unauthorized vote attribution.

### Finding Description

**Authorization model.** A committee member's cold credential authorizes a hot credential via `ConwayAuthCommitteeHotKey`. This writes `coldCred → CommitteeHotCredential hotCred` into `csCommitteeCreds` inside `CommitteeState`. [1](#0-0) 

**Vote storage.** Votes are stored in `GovActionState.gasCommitteeVotes` keyed by hot credential (`Map (Credential HotCommitteeRole) Vote`), not by cold credential. [2](#0-1) 

**Ratification counting.** `committeeAcceptedRatio` iterates over cold credentials in the enacted committee, looks up each cold credential's authorized hot credential in `csCommitteeCreds`, then looks up that hot credential's vote in `gasCommitteeVotes`. [3](#0-2) 

**Missing uniqueness check.** `checkAndOverwriteCommitteeMemberState` only checks whether the cold credential has previously resigned. It does **not** check whether the hot credential being authorized is already in use by another cold credential. [4](#0-3) 

**No vote cleanup on re-authorization.** The `cleanupProposalVotes` logic in the GOV rule only removes DRep votes when DReps unregister. There is no equivalent cleanup of `gasCommitteeVotes` when a hot credential is re-authorized by a new cold credential. [5](#0-4) 

**Attack scenario (vote amplification):**
1. Committee has `coldA` and `coldB`, threshold = 2/3 (2 of 3 members needed).
2. `coldA` submits `AuthCommitteeHotKeyTxCert coldA hotA`.
3. `hotA` submits a YES vote on governance action G → `gasCommitteeVotes[hotA] = VoteYes`.
4. `coldB` submits `AuthCommitteeHotKeyTxCert coldB hotA` (same hot credential, no ledger check prevents this).
5. Now `csCommitteeCreds[coldA] = CommitteeHotCredential hotA` and `csCommitteeCreds[coldB] = CommitteeHotCredential hotA`.
6. During ratification, `committeeAcceptedRatio` counts `coldA`'s vote as YES (via `hotA`) **and** `coldB`'s vote as YES (via the same stale `hotA → VoteYes`).
7. G is ratified with 2 YES votes even though only 1 actual vote was cast.

**Attack scenario (stale vote after removal):**
1. `coldA` authorizes `hotA`; `hotA` votes YES on G.
2. `coldA` is removed from the committee via `UpdateCommittee`; epoch boundary clears `coldA`'s entry from `CommitteeState`.
3. `coldB` (a new committee member) authorizes `hotA`.
4. `coldB`'s vote on G is now counted as YES from the stale `hotA → VoteYes` in `gasCommitteeVotes`, without `coldB` ever voting. [6](#0-5) 

### Impact Explanation

A malicious committee member can amplify their voting power or obtain YES votes on governance actions without actually voting, by authorizing a hot credential that has already voted YES. This can cause governance actions — including `NewConstitution`, `HardForkInitiation`, `ParameterChange`, and `TreasuryWithdrawals` — to be ratified without the required threshold of genuine committee votes. This matches the allowed impact: **Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted (Critical)**.

### Likelihood Explanation

The attacker must be an elected committee member. Committee members are elected through governance, but a malicious actor who obtains a committee seat can exploit this with a single transaction (`AuthCommitteeHotKeyTxCert`). The attack requires no special tooling: the attacker only needs to observe which hot credentials have voted YES on pending governance actions (all votes are public on-chain) and then authorize one of those hot credentials. Likelihood is **Medium**.

### Recommendation

1. **Enforce hot credential uniqueness:** In `checkAndOverwriteCommitteeMemberState` (or in the GOV rule's voter validation), reject `AuthCommitteeHotKey` if the hot credential is already authorized by any other cold credential in `csCommitteeCreds`.

2. **Clean up stale votes on re-authorization:** When a cold credential authorizes a new hot credential (overwriting a previous one), remove the old hot credential's votes from all pending `gasCommitteeVotes` maps — analogous to how `cleanupProposalVotes` removes DRep votes on unregistration. [7](#0-6) 

3. **Deduplicate in ratification:** In `committeeAcceptedRatio`, track which hot credentials have already been counted and skip duplicates.

### Proof of Concept

```
Epoch 0:
  - Committee: {coldA (expiry=100), coldB (expiry=100), coldC (expiry=100)}, threshold=2/3
  - coldA submits: AuthCommitteeHotKeyTxCert coldA hotA
  - hotA submits: VotingProcedures { CommitteeVoter hotA → VoteYes } on govAction G

Epoch 1:
  - coldB submits: AuthCommitteeHotKeyTxCert coldB hotA   ← no ledger rejection
  - csCommitteeCreds = { coldA → CommitteeHotCredential hotA,
                         coldB → CommitteeHotCredential hotA,
                         coldC → MemberNotAuthorized }
  - gasCommitteeVotes G = { hotA → VoteYes }

Ratification check for G:
  - coldA: lookup hotA → VoteYes  → (yes=1, tot=1)
  - coldB: lookup hotA → VoteYes  → (yes=2, tot=2)   ← stale vote counted again
  - coldC: not in csCommitteeCreds → (yes=2, tot=3)
  - ratio = 2/3 >= threshold 2/3  → G is RATIFIED

Only 1 actual vote was cast; 2 were counted.

committeeAcceptedRatio ::
  forall era.
  Map (Credential ColdCommitteeRole) EpochNo ->
  Map (Credential HotCommitteeRole) Vote ->
  CommitteeState era ->
  EpochNo ->
  Rational
committeeAcceptedRatio members votes committeeState currentEpoch =
  yesVotes %? totalExcludingAbstain
  where
    accumVotes ::
      (Integer, Integer) ->
      Credential ColdCommitteeRole ->
      EpochNo ->
      (Integer, Integer)
    accumVotes (!yes, !tot) member expiry
      | currentEpoch > expiry = (yes, tot) -- member is expired, vote "abstain" (don't count it)
      | otherwise =
          case Map.lookup member (csCommitteeCreds committeeState) of
            Nothing -> (yes, tot) -- member is not registered, vote "abstain"
            Just (CommitteeMemberResigned _) -> (yes, tot) -- member has resigned, vote "abstain"
            Just (CommitteeHotCredential hotKey) ->
              case Map.lookup hotKey votes of
                Nothing -> (yes, tot + 1) -- member hasn't voted, vote "no"
                Just Abstain -> (yes, tot) -- member voted "abstain"
                Just VoteNo -> (yes, tot + 1) -- member voted "no"
                Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
    (yesVotes, totalExcludingAbstain) = Map.foldlWithKey' accumVotes (0, 0) members
