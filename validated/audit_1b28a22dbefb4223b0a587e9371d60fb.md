### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Reward to Arbitrary Address - (`chia/farmer/farmer_api.py`)

### Summary
A connected harvester can set `farmer_reward_address_override` in `NewProofOfSpace` or `RespondSignatures` to any arbitrary puzzle hash. The farmer accepts this override without any cryptographic authorization or allowlist check, causing the 1.75 XCH farmer block reward to be permanently redirected to the attacker's address. The only guard — `notify_farmer_reward_taken_by_harvester_as_fee()` — is advisory logging only and never blocks the override.

### Finding Description

The `NewProofOfSpace` protocol message includes an optional `farmer_reward_address_override` field: [1](#0-0) 

When a harvester sends a `NewProofOfSpace` with this field set, `FarmerAPI.new_proof_of_space()` calls `notify_farmer_reward_taken_by_harvester_as_fee()` for logging, then proceeds unconditionally: [2](#0-1) 

`notify_farmer_reward_taken_by_harvester_as_fee()` performs a fee-quality check against `fee_info.applied_fee_threshold`, but **only emits log warnings** — it never returns an error, raises an exception, or prevents the override from being used: [3](#0-2) 

Later, in `_process_respond_signatures()`, the farmer unconditionally replaces its own configured `farmer_target` with the harvester-supplied override: [4](#0-3) 

This `farmer_reward_address` is then embedded in `DeclareProofOfSpace` sent to the full node, which constructs the foliage with the attacker's puzzle hash as `farmer_reward_puzzle_hash`. The farmer subsequently signs this foliage (via the `request_signed_values` → `RequestSignatures` round-trip) without ever verifying that the reward address matches its own configured target.

The `RespondSignatures` message also carries `farmer_reward_address_override`, giving the harvester a second opportunity to inject the override even if it was not set in the initial `NewProofOfSpace`: [5](#0-4) 

### Impact Explanation

Every block won by the farmer while a malicious harvester is connected results in the 1.75 XCH farmer reward being sent to the attacker's address instead of the farmer's configured `xch_target_address`. The farmer's own key signs the foliage containing the attacker's address, making the diversion indistinguishable from a legitimate block at the consensus layer. Rewards are permanently lost to the farmer with no on-chain recourse.

This matches the allowed Critical/High impact: **unauthorized payout redirection affecting XCH** and **bypass of farmer authorization enabling coin control**.

### Likelihood Explanation

Any harvester that is network-connected to the farmer can exploit this. In typical deployments, harvesters connect over a local or semi-trusted network, but the protocol itself imposes no cryptographic constraint preventing a harvester from setting `farmer_reward_address_override` to any value. A compromised or malicious third-party harvester (e.g., a DrPlotter-style remote harvester service) can silently steal every block reward without the farmer detecting it until they notice missing balance. The fee quality check is purely advisory and provides no enforcement.

### Recommendation

1. **Enforce an allowlist**: The farmer should maintain a configurable set of permitted override addresses. Any `farmer_reward_address_override` not in the allowlist should cause the proof to be dropped, not just logged.

2. **Require farmer authorization**: Override addresses should require a farmer-signed authorization (e.g., a signature over `(override_address, challenge_hash)` using the farmer's key), so the harvester cannot unilaterally redirect rewards.

3. **Validate before signing foliage**: In `request_signed_values`, before forwarding the `RequestSignatures` to the harvester, the farmer should assert that `foliage_block_data.farmer_reward_puzzle_hash` matches its own `farmer_target` (or an explicitly authorized override), and refuse to sign if it does not.

### Proof of Concept

1. Attacker operates or compromises a harvester connected to the victim farmer.
2. When the harvester finds a valid proof of space, it sends `NewProofOfSpace` with `farmer_reward_address_override = attacker_puzzle_hash` and `fee_info = None`.
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`, which logs a warning (`"Harvester illegitimately took reward by failing to provide its fee rate"`) but does not abort.
4. The farmer proceeds to send `RequestSignatures` to the harvester, receives `RespondSignatures` (also with `farmer_reward_address_override = attacker_puzzle_hash`), and constructs `DeclareProofOfSpace` with `farmer_reward_address = attacker_puzzle_hash`.
5. The full node builds a foliage block with `farmer_reward_puzzle_hash = attacker_puzzle_hash` and requests the farmer's signature.
6. The farmer signs the foliage without checking the reward address and returns `SignedValues`.
7. The block is accepted by consensus; the 1.75 XCH farmer reward is paid to the attacker. The farmer receives nothing. [6](#0-5) [7](#0-6)

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
