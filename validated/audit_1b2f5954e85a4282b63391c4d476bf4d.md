### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Binding It to the Request Content — (`crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is valid over `response.payload_hash` using the domain's root public key, but never checks that `response.payload_hash` was actually derived from the `request` being resolved. A single Byzantine attested participant can recycle a valid MPC-produced signature from any previously completed foreign-tx request and use it to forge a verification response for a completely different pending request, delivering a `payload_hash` that does not correspond to the transaction the user asked to verify.

### Finding Description

In `respond_verify_foreign_tx` (lines 691–754 of `crates/contract/src/lib.rs`), the contract performs three steps:

1. Looks up the root public key for `request.domain_id`.
2. Verifies `response.signature` over `response.payload_hash` using that root key.
3. Resolves all pending yields keyed on `request` with the raw `response` bytes.

```rust
// crates/contract/src/lib.rs  lines 718–733
let signature_is_valid = match (&response.signature, public_key) {
    (
        dtos::SignatureResponse::Secp256k1(signature_response),
        PublicKeyExtended::Secp256k1 { near_public_key },
    ) => {
        let secp_pk = ...;
        let payload_hash: [u8; 32] = response.payload_hash.0;   // ← attacker-supplied
        near_mpc_signature_verifier::verify_ecdsa_signature(
            signature_response,
            &payload_hash,                                        // ← never checked against request
            &secp_pk,
        )
        .is_ok()
    }
    ...
};
```

The missing check is: `response.payload_hash` is never asserted to equal `SHA-256(borsh(ForeignTxSignPayload { request, values }))`. The `payload_hash` is entirely caller-controlled.

This is the direct analog of the YoloV2 type-confusion bug: in YoloV2 the token *address* is whitelisted but the *type* is not enforced, so a caller can specify ERC721 for an ERC20 address and bypass the transfer. Here, the *signature* is verified against the root key but the *hash* is not bound to the request, so a caller can supply a hash from a different request and bypass the content-binding check.

By contrast, the regular `respond` function (lines 600–608) verifies the signature against `request.payload` — the payload that was committed on-chain when the user called `sign()` — so the signature is structurally bound to the request:

```rust
// crates/contract/src/lib.rs  lines 600–608
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← taken from the request, not from the response
    &expected_public_key,
)
```

The `ForeignTxSignPayloadV1` struct does include the `request` field so the hash is *intended* to commit to the request:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs  lines 1499–1509
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
```

But the contract never enforces this binding at response time.

### Impact Explanation

A single Byzantine attested participant can:

1. Read a valid `(payload_hash_A, sig_A)` pair from the blockchain — it is publicly visible in the receipt of any previously completed `respond_verify_foreign_tx` call for request A.
2. While a different request B is pending in `pending_verify_foreign_tx_requests`, call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })` before any honest node responds.
3. The contract accepts because `sig_A` is a valid ECDSA signature over `payload_hash_A` under the root key, and `request_B` exists in the pending map.
4. All yields queued under `request_B` are drained and every waiting caller receives `{ payload_hash: payload_hash_A, signature: sig_A }` — a response that attests to the wrong transaction.

Any downstream smart contract that only checks "is the signature valid over `payload_hash`?" (step 1 of the two-step verification described in the docs) will accept the forged attestation, enabling invalid bridge execution or double-spend conditions. The documentation explicitly notes that callers must reconstruct the expected hash locally and compare it, but this off-chain obligation is not enforced on-chain.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution**.

### Likelihood Explanation

The attack requires only a single Byzantine attested participant — below the signing threshold. The attacker does not need to forge a cryptographic signature; it reuses a legitimately produced MPC signature from a prior request. The only operational requirement is to submit the forged `respond_verify_foreign_tx` call before any honest node submits the correct one.
