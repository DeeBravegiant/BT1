### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Submitted `request`, Enabling Cross-Request Signature Replay — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the supplied `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key, but it **never checks that `response.payload_hash` was derived from the `request` argument** that is being resolved. A single Byzantine attested participant (strictly below the signing threshold) can replay any previously emitted, publicly visible `(payload_hash, signature)` pair — produced by the honest MPC network for a *different* foreign-chain transaction — to close out a pending `verify_foreign_transaction` slot with a forged attestation.

---

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` (lines 691–753) performs the following checks:

1. Caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain root key.
3. All queued yields for `request` are drained with `response` as the data. [1](#0-0) 

What is **absent** is any binding between `response.payload_hash` and the `request` parameter. The intended invariant is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: <observed> }))
``` [2](#0-1) 

Because `ForeignTxSignPayloadV1` embeds the inner `ForeignChainRpcRequest` (including `tx_id`, `extractors`, finality), a `payload_hash` produced for Bitcoin `tx_id = X` is cryptographically distinct from one produced for `tx_id = Y`. The contract never recomputes or compares the hash; it only checks the signature over whatever hash the caller supplies.

The SDK-level verifier (`ForeignChainSignatureVerifier::verify_signature`) does perform this binding check client-side: [3](#0-2) 

But this check lives in the *calling* contract's code, not in the MPC contract itself. The MPC contract is the trust anchor; pushing the binding check to callers creates a gap that a Byzantine participant can exploit.

---

### Impact Explanation

**High — forged foreign-chain verification enabling invalid bridge execution.**

A Byzantine attested participant executes the following steps:

1. **Obtain a valid attestation for a different transaction.** Any previously completed `verify_foreign_transaction` call returns a `VerifyForeignTransactionResponse { payload_hash_A, signature_A }` on-chain. This is public.

2. **Submit or wait for a target pending request.** A victim (or the attacker themselves) submits `verify_foreign_transaction(request_B)` where `request_B.request.tx_id = Y ≠ X`. The contract queues a yield for `request_B`.

3. **Call `respond_verify_foreign_tx(request_B, { payload_hash_A, signature_A })`.** The contract:
   - Finds `request_B` in `pending_verify_foreign_tx_requests` ✓
   - Verifies `signature_A` over `payload_hash_A` under the root key ✓ (signature is genuinely valid)
   - Drains all yields for `request_B` with `{ payload_hash_A, signature_A }` ✓

4. **Victim receives a forged attestation.** The victim's NEAR contract gets back a `VerifyForeignTransactionResponse` whose `payload_hash` encodes `tx_id = X`, not `tx_id = Y`. Any bridge logic that does not independently recompute and compare the expected `payload_hash` will treat this as a valid attestation of `tx_id = Y`, enabling double-spend or unauthorized fund release. [4](#0-3) 

---

### Likelihood Explanation

- Requires **one** Byzantine attested participant — strictly below the signing threshold.
- The replayed `(payload_hash, signature)` is **public on-chain** after any prior successful `verify_foreign_transaction` call for the same domain.
- No key material needs to be leaked; no threshold collusion is required.
- The attacker can self-submit `request_B` to control the target, or race against a victim's pending request.
- Calling contracts that rely solely on the MPC contract as a trust anchor (without re-implementing the SDK's `payload_hash` binding check) are directly vulnerable.

---

### Recommendation

The contract must bind `response.payload_hash` to the `request` being resolved. Two concrete approaches:

**Option A — Require the responder to supply extracted values and verify on-chain:**

```rust
pub fn respond_verify_foreign_tx(
    &mut self,
    request: VerifyForeignTransactionRequest,
    extracted_values: Vec<ExtractedValue>,   // NEW
    response: VerifyForeignTransactionResponse,
) -> Result<(), Error> {
    // ... existing auth checks ...

    // Recompute the expected payload hash from the request + extracted values
    let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
        request: request.request.clone(),
        values: extracted_values,
    });
    let expected_hash = expected_payload.compute_msg_hash()
        .map_err(|_| RespondError::InvalidPayloadHash)?;

    if expected_hash != response.payload_hash {
        return Err(RespondError::PayloadHashMismatch.into());
    }

    // ... existing signature verification and yield resolution ...
}
```

**Option B — Include `domain_id` and a per-request nonce inside `ForeignTxSignPayloadV1`** so that the signed hash is unique per request and cannot be replayed across requests even within the same domain. [5](#0-4)

### Proof of Concept

**Setup:**
- MPC network running with domain `D` (Secp256k1, ForeignTx purpose).
- Attacker is one attested participant (e.g., node index 0 out of 5, threshold = 3).

**Step 1 — Harvest a valid attestation:**
```
Alice calls verify_foreign_transaction({ domain_id: D, request: Bitcoin { tx_id: X, ... } })
MPC network responds → VerifyForeignTransactionResponse { payload_hash_X, sig_X }
payload_hash_X = SHA-256(borsh(ForeignTxSignPayload::V1 { request: Bitcoin{tx_id:X,...}, values: [BlockHash(h1)] }))
```
`payload_hash_X` and `sig_X` are now public on-chain.

**Step 2 — Create a target pending request:**
```
Bob calls verify_foreign_transaction({ domain_id: D, request: Bitcoin { tx_id: Y, ... } })
Contract queues a yield for request_B.
```

**Step 3 — Byzantine participant forges the response:**
```
Attacker (node 0) calls:
  respond_verify_foreign_tx(
    request  = { domain_id: D, request: Bitcoin { tx_id: Y, ... } },   // request_B
    response = { payload_hash: payload_hash_X, signature: sig_X }       // from tx_id=X
  )
```

**Contract execution:**
- `request_B` is in `pending_verify_foreign_tx_requests` → ✓
- `verify_ecdsa(sig_X, payload_hash_X, root_key_D)` → ✓ (genuinely valid)
- `resolve_yields_for(request_B, serialize({ payload_hash_X, sig_X }))` → Bob's yield is resumed

**Result:**
Bob's contract receives `{ payload_hash_X, sig_X }`. If Bob's contract does not recompute `SHA-256(borsh(ForeignTxSignPayload::V1{Bitcoin{tx_id:Y,...}, expected_values}))` and compare it to `payload_hash_X`, it will accept this as a valid attestation that Bitcoin `tx_id = Y` was confirmed — when in fact only `tx_id = X` was ever verified by the MPC network. [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
