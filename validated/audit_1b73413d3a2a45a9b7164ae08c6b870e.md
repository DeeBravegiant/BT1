Audit Report

## Title
Missing Chainlink `latestRoundData` Return Value Validation Allows Stale/Zero Prices to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness, completeness, or sign/zero checks. A stale-high Chainlink answer causes any depositor to receive excess rsETH, diluting existing rsETH holders. The sibling contract `ChainlinkOracleForRSETHPoolCollateral` already implements all three required guards, confirming the protocol is aware of the pattern and that its absence here is a code defect.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at lines 52–54:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`) are available but only `answer` is captured. The following checks are absent:

- **Staleness**: No `answeredInRound >= roundId` check. Chainlink returns the last known answer even after the oracle stops updating.
- **Incomplete round**: No `updatedAt != 0` check. An incomplete round returns `updatedAt == 0`.
- **Zero/negative price**: No `price > 0` guard before the `uint256(price)` cast.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 30–32 correctly implements all three:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

`ChainlinkPriceOracle` is the oracle registered for LST assets (stETH, cbETH, etc.) in `LRTOracle`. Its output flows into two critical paths:

1. **Direct deposit minting** (`LRTDepositPool.getRsETHAmountToMint`, line 520):
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
2. **rsETH price update** (`LRTOracle._getTotalEthInProtocol`, line 339):
   ```solidity
   uint256 assetER = getAssetPrice(asset);
   ```
   which feeds directly into `rsETHPrice` stored on-chain.

**Stale-high price exploit path:**
1. A Chainlink LST/ETH feed goes stale (network congestion, sequencer downtime, feed deprecation). The last stored answer is above the current true price.
2. Any user calls `LRTDepositPool.depositAsset(stETH, amount, 0)`.
3. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)`.
4. `latestRoundData()` returns the stale-high price. No revert occurs.
5. `rsethAmountToMint = (amount * staleHighPrice) / rsETHPrice` — the depositor receives excess rsETH relative to the true value deposited.
6. Existing rsETH holders are diluted; their proportional claim on protocol TVL is reduced.

**Zero price path (requires `minRSETHAmountExpected = 0`):**
If `price == 0`, `rsethAmountToMint = 0`. With `minRSETHAmountExpected = 0`, `_beforeDeposit` passes the check at line 667, the asset transfer executes at line 114, and `_mintRsETH(0)` mints nothing — depositor funds are locked until oracle recovery.

**Negative price**: In Solidity 0.8, `uint256(negativeInt256) * 1e18` overflows and reverts, so this sub-case does not produce silent corruption.

**`updateRSETHPrice` corruption**: `updateRSETHPrice()` is public (line 87). A stale price corrupts `totalETHInProtocol` and thus `rsETHPrice` stored on-chain, affecting all subsequent deposits and withdrawals. The price percentage limit guard (lines 252–267) only triggers if `pricePercentageLimit > 0` and the deviation exceeds the threshold — it does not prevent moderate stale-price corruption.

## Impact Explanation
**High — Theft of unclaimed yield.**

A stale-high oracle price causes depositors to mint excess rsETH beyond the true value of their deposit. This excess comes at the direct expense of existing rsETH holders, whose proportional share of protocol TVL is diluted. This is a concrete, repeatable transfer of value from existing holders to new depositors, matching the "Theft of unclaimed yield" impact class. The zero-price path additionally matches "Temporary freezing of funds" (Medium) when `minRSETHAmountExpected = 0`.

## Likelihood Explanation
Chainlink feeds go stale during L2 sequencer downtime, network congestion, or feed deprecation — all documented, real-world events. The deposit entry point (`depositAsset`, `depositETH`) is fully permissionless. Any user can trigger the stale-price minting path at any time during a staleness window without any special privileges or victim cooperation. The `updateRSETHPrice()` function is also public and callable by anyone.

## Recommendation
Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Optionally add a heartbeat check: `if (block.timestamp - updatedAt > HEARTBEAT_INTERVAL) revert StalePrice();`

## Proof of Concept
**Foundry fork test outline (stale-high price → yield theft):**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed is live.
2. Deploy or reference the existing `ChainlinkPriceOracle` with the stETH feed registered.
3. Record `rsETHPrice` and existing holder's rsETH balance.
4. Use `vm.mockCall` to make `latestRoundData()` return a stale-high answer (e.g., `answeredInRound < roundId`, `updatedAt` = old timestamp, `answer` = 1.05e18 when true price is 1.00e18).
5. Call `LRTDepositPool.depositAsset(stETH, 10 ether, 0)` from an unprivileged address.
6. Assert `rsethAmountToMint > (10e18 * truePrice) / rsETHPrice` — depositor received excess rsETH.
7. Call `updateRSETHPrice()` with the stale price still active; assert `rsETHPrice` is inflated.
