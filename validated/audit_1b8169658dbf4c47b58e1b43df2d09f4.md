Audit Report

## Title
Sandwich Attack via Public `updateRSETHPrice()` Enables Yield Theft from rsETH Holders - (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is `public` with no access control, allowing any caller to trigger a price update atomically. Combined with the public `LRTDepositPool.depositETH()` and `LRTWithdrawalManager.instantWithdrawal()`, an attacker can deposit at a stale lower price, trigger the price update, and immediately withdraw at the new higher price — stealing accrued yield from existing rsETH holders. The `pricePercentageLimit` guard is bypassed when unset (default value is 0) and only partially mitigates the attack when set.

## Finding Description

**Root cause:** `updateRSETHPrice()` is callable by any unprivileged address:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_updateRsETHPrice()` computes the new price as `(totalETHInProtocol - protocolFeeInETH) / rsethSupply`, where `totalETHInProtocol` includes all ETH in the deposit pool, NDCs, EigenLayer, and unstaking vault — including any rewards that have accrued since the last update.

**Deposit minting uses the stored (stale) price:**
```solidity
// contracts/LRTDepositPool.sol L519-521
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Instant withdrawal redeems at the current stored price:**
```solidity
// contracts/LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

**Exploit sequence (single transaction):**
1. Attacker deposits `D` ETH via `depositETH()` at stale `rsETHPrice` → receives more rsETH than their proportional share
2. Attacker calls `updateRSETHPrice()` → stored price jumps to reflect accrued rewards; the attacker's rsETH is now worth more ETH
3. Attacker calls `instantWithdrawal()` → redeems rsETH at the new higher price, receiving more ETH than deposited

**The `pricePercentageLimit` guard is insufficient:**
```solidity
// contracts/LRTOracle.sol L256-257
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```
The guard is disabled when `pricePercentageLimit == 0`, which is the default (no explicit initialization). Even when set, it only caps the per-call price increase, not the attack itself — it merely limits the per-transaction profit to the configured percentage of TVL.

## Impact Explanation

**High — Theft of unclaimed yield.**

Accrued EigenLayer rewards that have not yet been reflected in `rsETHPrice` represent yield owed to existing rsETH holders. An attacker who deposits just before the price update receives more rsETH than their proportional share of the pool. After the update, that rsETH redeems for more ETH than was deposited, with the surplus coming directly from the pending rewards. The attack is executable with a flash loan (zero net capital required), is repeatable every time rewards accrue, and causes a direct, quantifiable loss to existing holders.

Numerical example:
- Protocol: 100 ETH, 100 rsETH, stored price = 1.0
- 10 ETH of EigenLayer rewards accrue; price not yet updated
- Attacker deposits 100 ETH at stale price → receives 100 rsETH
- `updateRSETHPrice()`: new price = 210 ETH / 200 rsETH = 1.05
- Attacker's 100 rsETH → 105 ETH via `instantWithdrawal`
- **Attacker profit: 5 ETH** (50% of the 10 ETH reward pool stolen)

## Likelihood Explanation

**Medium.** Required conditions:
1. `isInstantWithdrawalEnabled[asset]` is `true` — a manager-controlled flag expected to be enabled in normal operation
2. `LRTUnstakingVault` holds sufficient liquid assets for the instant withdrawal
3. Rewards have accrued since the last `updateRSETHPrice()` call — routine between oracle updates
4. `pricePercentageLimit == 0` (default) OR the accrued reward delta is within the configured limit

All four conditions are expected to hold during normal protocol operation. The attack requires no mempool monitoring since all three calls can be made atomically by the attacker. Even if `pricePercentageLimit` is set, the attack is still viable for smaller reward accumulations within the configured threshold.

## Recommendation

Restrict `updateRSETHPrice()` to authorized callers, consistent with the existing `updateRSETHPriceAsManager()` pattern:

```solidity
// contracts/LRTOracle.sol L94-96
function updateRSETHPriceAsManager() external onlyLRTManager {
    _updateRsETHPrice();
}
```

Remove the public `updateRSETHPrice()` function entirely, or add `onlyLRTOperator` / `onlyLRTManager` access control to it. If public price updates are desired for keeper/bot use, implement a TWAP or time-weighted price mechanism so that a single block's deposit cannot capture the full pending reward delta atomically.

## Proof of Concept

```solidity
// Foundry fork test outline
function testSandwichAttack() external {
    // Setup: protocol has 100 ETH, 100 rsETH, price = 1.0
    // Simulate 10 ETH of EigenLayer rewards accruing (increase pod shares)
    // without calling updateRSETHPrice()

    vm.startPrank(attacker);

    // Step 1: Deposit at stale price
    uint256 ethDeposited = 100 ether;
    depositPool.depositETH{value: ethDeposited}(0, "");
    uint256 rsETHReceived = rsETH.balanceOf(attacker);
    // rsETHReceived == 100e18 (stale price 1.0), true price would give ~90.9e18

    // Step 2: Trigger price update — public, no access control
    oracle.updateRSETHPrice();
    // rsETHPrice is now ~1.05

    // Step 3: Instant withdrawal at new higher price
    rsETH.approve(address(withdrawalManager), rsETHReceived);
    withdrawalManager.instantWithdrawal(ETH_TOKEN, rsETHReceived, "");

    uint256 ethReceived = attacker.balance - initialBalance;
    assertGt(ethReceived, ethDeposited); // attacker profits
    vm.stopPrank();
}
```

The minimal reproducible path is: `LRTOracle.updateRSETHPrice()` (public, L87) → `LRTDepositPool.depositETH()` (public, L76) → `LRTWithdrawalManager.instantWithdrawal()` (public when `isInstantWithdrawalEnabled`, L212). All three are reachable by an unprivileged caller in a single transaction.
