### Title
User-Controlled `confirmations` Field in `verify_foreign_transaction` Bypasses Bitcoin Finality Guarantee — (`File: crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint accepts a fully user-controlled `confirmations` field inside `BitcoinRpcRequest`. The MPC contract enforces no minimum confirmation threshold, and the node-side `BitcoinInspector` uses the caller-supplied value verbatim as the finality gate. An unprivileged caller can set `confirmations: 1` (or `0`) to obtain a threshold-signed MPC attestation for a Bitcoin transaction that has not reached economic finality, enabling double-spend attacks against any bridge or NEAR contract that consumes the attestation without independently re-validating the confirmation count embedded in the signed payload.

---

### Finding Description

**Root cause — no minimum confirmation floor in the contract or inspector**

`verify_foreign_transaction` in `crates/contract/src/lib.rs` accepts the raw `VerifyForeignTransactionRequestArgs` from any caller and enqueues it without inspecting the `confirmations` field: [1](#0-0) 

The `BitcoinRpcRequest.confirmations` field is typed as `BlockConfirmations` (a plain `u64` newtype with `minimum: 0` in the ABI): [2](#0-1) 

On the node side, `BitcoinInspector::extract` uses the caller-supplied threshold directly: [3](#0-2) 

The comparison `block_confirmations_threshold <= transaction_block_confirmation` means that with `threshold = 1`, a transaction with exactly 1 confirmation passes. With `threshold = 0`, the check `0 <= N` is trivially true for any `N ≥ 0`, bypassing the finality check entirely.

**What gets signed**

The MPC nodes sign a canonical encoding of `(request, observed_values)` where `request` includes the caller-supplied `confirmations` value. The signed payload therefore attests to "this tx had at least N confirmations" where N is attacker-chosen. The design intent stated in the documentation is to attest that a foreign transaction *finalized successfully*: [4](#0-3) 

A `confirmations: 1` attestation violates this intent — Bitcoin requires ~6 confirmations for economic finality.

**Analog to Oracle Drift**

The external report describes a gap between the oracle deviation threshold and the redemption fee: when the oracle price can drift by up to 2% without updating, arbitrageurs exploit the stale price. The analog here is the gap between the user-supplied confirmation threshold (can be 1 or 0) and the actual finality requirement (≥ 6 for Bitcoin). Just as the oracle deviation threshold is the "tolerance window" that creates the exploitable gap, the user-controlled `confirmations` field is the tolerance window that creates a finality bypass.

---

### Impact Explanation

**Impact class: High — forged foreign-chain verification / light-client-style verification bypass causing invalid bridge execution or double-spend conditions.**

Attack scenario (Omnibridge inbound flow):

1. Attacker broadcasts a Bitcoin transaction sending funds to a bridge deposit address.
2. After 1 confirmation, attacker calls `verify_foreign_transaction` with `confirmations: 1`.
3. MPC nodes observe the transaction has ≥ 1 confirmation and produce a threshold signature over `(request, observed_values)`.
4. Attacker submits the MPC attestation to the NEAR bridge contract to claim bridged tokens.
5. Attacker simultaneously double-spends the Bitcoin transaction (e.g., via a higher-fee replacement or by mining a competing block), causing a chain reorganization that removes the original transaction.
6. The bridge has released tokens for a Bitcoin transaction that no longer exists on the canonical chain.

The MPC signature is cryptographically valid — the network signed exactly what was requested — but the attestation is semantically invalid because the transaction was not final. Any bridge contract that trusts the MPC attestation without re-checking the `confirmations` value embedded in the signed payload is directly exploitable.

---

### Likelihood Explanation

- **Attacker entry point**: `verify_foreign_transaction` is a public payable function requiring only 1 yoctonear deposit. No privileged access is needed.
- **Confirmation threshold**: Setting `confirmations: 1` is a valid, accepted input. The ABI explicitly allows `minimum: 0`.
- **Bitcoin reorg feasibility**: 1-confirmation reorgs occur naturally; deliberate double-spend attacks at 1 confirmation are well-documented and economically viable for high-value bridge transactions.
- **Bridge contract exposure**: The primary documented use case (Omnibridge inbound) involves a NEAR contract consuming the MPC attestation. If that contract does not re-validate the `confirmations` field in the signed payload, it is directly vulnerable.

---

### Recommendation

1. **Enforce a minimum confirmation threshold on-chain**: Add a per-chain minimum confirmation floor to the `ForeignChainsMetadata` / chain configuration voted in by participants. Reject `verify_foreign_transaction` calls whose `confirmations` value falls below the chain-specific minimum (e.g., 6 for Bitcoin mainnet).

2. **Alternatively, fix the confirmation threshold per chain**: Remove the `confirmations` field from the user-supplied request and instead use a governance-voted, per-chain confirmation threshold stored in contract state. This eliminates the user-controlled finality bypass entirely.

3. **Document the invariant for bridge consumers**: Until a fix is deployed, bridge contracts consuming `verify_foreign_transaction` responses MUST validate the `confirmations` field in the signed payload against their own minimum finality requirement.

        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
