Audit Report

## Title
Stale Cross-Chain Rate Used for rsETH Minting Without Staleness Check — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.getRate()` returns a frozen snapshot of the rsETH/ETH price last pushed from L1 via LayerZero, with no check against `lastUpdated`. All five L2 pool contracts divide deposited ETH by this potentially stale rate to compute rsETH mint amounts. Because rsETH accrues staking rewards continuously on L1, any lag between L1 price updates and L2 rate pushes allows a depositor to receive more rsETH than their ETH is currently worth, diluting existing holders' unclaimed yield.

## Finding Description
`CrossChainRateReceiver` stores the rate and timestamp set by the last LayerZero message:

```solidity
uint256 public rate;
uint256 public lastUpdated;
```

`lzReceive` updates both fields when a message arrives. `getRate()` returns the stored value unconditionally:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
```

No staleness check exists anywhere between `lastUpdated` and the returned value.

Every L2 pool (`RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`) calls `IOracle(rsETHOracle).getRate()` and uses the result as the denominator in the mint calculation:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The rate is only refreshed when someone calls `updateRate()` on `CrossChainRateProvider` or `MultiChainRateProvider` and pays the LayerZero fee. There is no on-chain enforcement of any maximum update interval. On L1, `rsETHPrice` grows continuously as `_updateRsETHPrice()` is called and EigenLayer staking rewards accrue. The L2 rate is a frozen snapshot of a past L1 price. The `lastUpdated` field is recorded but never read in any deposit path.

## Impact Explanation
When the L2 rate lags behind the true L1 price, the denominator in `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` is smaller than it should be, so the depositor receives more rsETH per ETH than the current backing ratio justifies. rsETH is a share token whose value equals total protocol ETH divided by total supply; every extra rsETH minted at a stale (lower) rate dilutes the value held by all existing rsETH holders. The yield that accrued since the last rate update is effectively transferred to the new depositor.

**Impact: High — Theft of unclaimed yield.**

## Likelihood Explanation
rsETH accrues staking rewards continuously; the L1 price rises every time `updateRSETHPrice()` is called. Cross-chain rate updates require a manual call and a LayerZero fee payment, so any gap between L1 price updates and L2 rate pushes is a normal operating condition, not an edge case. An attacker needs only to monitor the L1 `rsETHPrice` and the L2 `CrossChainRateReceiver.rate`, wait for a meaningful divergence (which grows with every passing hour), and call `deposit()` on any L2 pool. No special role, no front-running, no oracle manipulation is required — the entry path is the standard public `deposit()` function available to any user.

## Recommendation
Add a staleness guard inside `CrossChainRateReceiver.getRate()` that reverts if `block.timestamp - lastUpdated` exceeds a configurable maximum age (e.g., 24 hours):

```solidity
uint256 public maxRateAge; // e.g. 86400

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxRateAge, "Rate stale");
    return rate;
}
```

Alternatively, pause deposits on all L2 pools when the rate is detected as stale.

## Proof of Concept
1. At time T₀, L1 `rsETHPrice = 1.05e18` and L2 `CrossChainRateReceiver.rate = 1.05e18` (in sync).
2. Staking rewards accrue. At T₁ (48 hours later), L1 `rsETHPrice = 1.06e18` after `updateRSETHPrice()` is called, but no one has called `updateRate()` on the L1 provider, so the L2 rate remains `1.05e18`.
3. Attacker calls `deposit{value: 1 ether}("")` on `RSETHPoolNoWrapper`:
   - `viewSwapRsETHAmountAndFee(1e18)` calls `getRate()` → returns `1.05e18` (stale)
   - `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.952381e18 rsETH`
   - Correct amount at current rate: `1e18 * 1e18 / 1.06e18 ≈ 0.943396e18 rsETH`
   - Attacker receives `≈ 0.008985e18` excess rsETH
4. The excess rsETH represents yield that had accrued to existing holders but is now diluted away.
5. The `lastUpdated` field on `CrossChainRateReceiver` records the time of the last push but is never checked anywhere in the deposit path across all five pool contracts.

**Foundry fork test plan:** Fork the target L2, deploy or point to the live `CrossChainRateReceiver`, warp `block.timestamp` forward by 48 hours without calling `updateRate()`, call `deposit{value: 1 ether}("")` on the pool, and assert that the minted rsETH amount exceeds `1e18 * 1e18 / currentL1Rate`, confirming the overmint.
