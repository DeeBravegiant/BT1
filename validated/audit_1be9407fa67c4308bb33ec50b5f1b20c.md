Audit Report

## Title
Oracle-Price vs. Actual-ETH Mismatch Leaves Permanent Residual in `ethValueInWithdrawal`, Inflating `rsETHPrice` and Freezing Unclaimed Yield — (`contracts/LRTConverter.sol`)

## Summary
`transferAssetFromDepositPool` records the ETH value of stETH at the oracle price at transfer time, while `_sendEthToDepositPool` (called from `claimStEth`) decrements `ethValueInWithdrawal` by the actual ETH received from Lido. When actual ETH is less than the oracle-booked value — due to slashing or oracle drift over the multi-day withdrawal window — a non-zero residual is permanently stranded in `ethValueInWithdrawal`. This phantom value propagates through `getETHDistributionData` → `getTotalAssetDeposits` → `_getTotalEthInProtocol` → `rsETHPrice`, inflating the price and causing every subsequent depositor to receive fewer rsETH than they are entitled to, permanently freezing the over-counted yield in the accounting layer.

## Finding Description
**Root cause:** `ethValueInWithdrawal` is incremented by `amount × P_oracle / 1e18` at transfer time (L140), but decremented by the actual ETH balance `address(this).balance` at claim time (L255–258). The decrement logic uses a partial-subtract branch: if `ethValueInWithdrawal > _amount`, it subtracts `_amount` and leaves the difference; it only zeroes out when actual ETH ≥ booked value. There is no per-request tracking, no admin reset, and no reconciliation path once all stETH has been unstaked.

**Exploit flow:**
1. Operator calls `transferAssetFromDepositPool(stETH, 10_000e18)` when oracle price is `0.9995e18` → `ethValueInWithdrawal += 9995e18`.
2. Operator calls `unstakeStEth(10_000e18)` — stETH leaves the converter.
3. Lido finalizes the withdrawal at `0.9985 ETH/stETH` (slashing or drift) → actual ETH = `9985e18`.
4. Operator calls `claimStEth(requestId, hint)` → `_sendEthToDepositPool(9985e18)` fires the `if`-branch: `ethValueInWithdrawal = 9995e18 − 9985e18 = 10e18`. Residual is permanent.
5. `getETHDistributionData` returns `ethLyingInConverter = 10e18` (phantom ETH).
6. `_getTotalEthInProtocol` includes this phantom value; `rsETHPrice` is inflated by `10e18 / rsethSupply`.
7. Any depositor calling `depositETH` or `depositAsset` receives `amount × assetPrice / inflatedRsETHPrice` rsETH — fewer than fair value.

**Why existing checks fail:** The `else` branch in `_sendEthToDepositPool` only zeroes `ethValueInWithdrawal` when actual ETH ≥ booked value, which is the opposite of the problematic case. The `pricePercentageLimit` upward-price guard in `_updateRsETHPrice` would only trigger for large single-event residuals; routine small-drift residuals accumulate silently below the threshold. `transferAssetToDepositPool` can only decrement `ethValueInWithdrawal` while stETH is physically present in the converter — after `unstakeStEth` this path is closed.

## Impact Explanation
The residual is phantom ETH: the real ETH was correctly sent to the deposit pool (reflected in `address(depositPool).balance`), but `ethValueInWithdrawal` double-counts it in `totalETHInProtocol`. Every depositor after the event receives proportionally fewer rsETH than they should. The over-counted yield can never be claimed by anyone — it is frozen in the accounting layer with no recovery path. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
Lido withdrawal requests are finalized over days to weeks. Oracle drift of even 0.01% on a 10,000 stETH unstake leaves a 1 ETH residual. Slashing events, while rare, produce larger residuals. Both conditions are part of normal protocol operations — no attacker action is required; the operator simply performs their expected duties. The vulnerability is triggered on any normal unstake cycle where the oracle price at transfer time exceeds the actual ETH/stETH ratio at claim time, which is a routine occurrence.

## Recommendation
Replace the partial-decrement logic with per-request bookkeeping:
1. In `unstakeStEth`, record the booked ETH value for each Lido request ID in a `mapping(uint256 requestId => uint256 bookedEthValue)`.
2. In `claimStEth`, subtract exactly the booked value for that request ID from `ethValueInWithdrawal` (not the actual ETH received), and account for any slashing loss separately as a realized loss event.
3. Alternatively, add an admin-callable `syncEthValueInWithdrawal()` that recomputes `ethValueInWithdrawal` from the set of still-pending Lido request IDs, allowing the operator to correct any residual after all requests are finalized.
