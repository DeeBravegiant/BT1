Audit Report

## Title
Missing Chainlink `latestRoundData()` Staleness and Validity Checks Enable Corrupted rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `AggregatorV3Interface.latestRoundData()` but discards all return values except `price`, with no staleness, completeness, or sign validation. This function sits directly in the rsETH minting path: `LRTDepositPool.depositAsset()` → `getRsETHAmountToMint()` → `LRTOracle.getAssetPrice()` → `ChainlinkPriceOracle.getAssetPrice()`. A stale or zero price returned by the Chainlink feed will cause incorrect rsETH minting amounts, leading to protocol insolvency or depositor fund loss. The same codebase already implements the correct three-check pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

## Finding Description

**Vulnerable code** in `contracts/oracles/ChainlinkPriceOracle.sol` L52–54:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`) are available but only `price` is consumed. The three mandatory safety checks are absent:

| Check | Purpose | Present? |
|---|---|---|
| `answeredInRound >= roundID` | Detect stale round | No |
| `updatedAt != 0` | Detect incomplete round | No |
| `price > 0` | Detect zero/negative price | No |

**Correctly validated counterpart** in `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L27–33:

```solidity
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

**Exploit path:**

1. Chainlink's LST/ETH feed (e.g., stETH/ETH) enters a stale round — `answeredInRound < roundID` — returning a stale price.
2. An unprivileged user calls `LRTDepositPool.depositAsset(stETH, amount, minRsETH)`.
3. `_beforeDeposit()` calls `getRsETHAmountToMint(asset, depositAmount)`.
4. `getRsETHAmountToMint()` computes: `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` — where `getAssetPrice(asset)` calls `ChainlinkPriceOracle.getAssetPrice()` live.
5. The stale price is returned without revert, corrupting the numerator of the minting formula.
6. The depositor receives an incorrect rsETH amount; no existing guard in the deposit path catches this.

The `pricePercentageLimit` circuit-breaker in `LRTOracle._updateRsETHPrice()` is irrelevant here — it only fires when `updateRSETHPrice()` is called separately, not during the deposit flow. The `minRSETHAmountExpected` slippage parameter protects depositors from receiving *less* than expected, not from receiving *more* (the insolvency scenario).

**Secondary instance** in `contracts/oracles/RSETHPriceFeed.sol` L68–70: `ETH_TO_USD.latestRoundData()` is forwarded with no validation before multiplying by `rsETHPrice`, propagating a stale ETH/USD price to any downstream consumer of this feed.

## Impact Explanation

- **Stale price (deflated):** `getAssetPrice` returns a value lower than true market price → depositor receives more rsETH than the deposited asset is worth → rsETH supply becomes unbacked → **protocol insolvency**.
- **Stale price (inflated):** depositor receives fewer rsETH tokens than owed → **theft of depositor value**.
- **Zero price:** `uint256(0)` makes `rsethAmountToMint = 0`, minting nothing while the asset transfer still occurs → **permanent loss of depositor funds**.
- **Negative price:** `uint256(negative_int256)` wraps to a huge number → wildly inflated minting → **protocol insolvency**.

All scenarios map directly to allowed critical impacts: direct theft of user funds or protocol insolvency.

## Likelihood Explanation

Chainlink feeds can return stale data during network congestion preventing heartbeat updates, Chainlink node failures, or feed deprecation. These are known, historically observed conditions requiring no special attacker privileges. Any depositor calling `LRTDepositPool.depositAsset()` during such a window triggers the vulnerable path. No front-running, governance access, or economic attack is required.

## Recommendation

Apply the same three-check pattern already used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundID, int256 price,, uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(answeredInRound >= roundID, "Chainlink Price Stale");
require(price > 0, "Chainlink Malfunction");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
```

Apply equivalent validation in `RSETHPriceFeed.latestRoundData()` before multiplying `answer` by `rsETHPrice`.
