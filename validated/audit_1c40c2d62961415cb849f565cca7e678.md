Audit Report

## Title
Uninitialized `rsETHPrice` Causes Division-by-Zero Revert in `getRsETHAmountToMint()` - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTOracle.rsETHPrice` defaults to `0` after deployment because `initialize()` never sets it. Until `updateRSETHPrice()` is called, every invocation of `getRsETHAmountToMint()` divides by zero and reverts, blocking all user deposits. No user funds are lost or frozen, but the protocol cannot deliver its core function of accepting deposits.

## Finding Description
`LRTOracle.rsETHPrice` is a plain `uint256` storage variable that initializes to `0` by default. The `initialize()` function sets only `lrtConfig` and emits an event — it never writes `rsETHPrice`:

```solidity
// LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

`rsETHPrice` is only written inside `_updateRsETHPrice()`, which is invoked by the permissionless `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. When `rsethSupply == 0`, `_updateRsETHPrice()` correctly sets `rsETHPrice = 1 ether`, but this only happens after an explicit call.

Every deposit flows through `_beforeDeposit()` → `getRsETHAmountToMint()`:

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

When `rsETHPrice == 0`, Solidity 0.8.x triggers a division-by-zero panic (revert). There is no zero-check guard on `rsETHPrice` before this division. Both `depositETH()` and `depositAsset()` are therefore completely blocked until `updateRSETHPrice()` is called.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

No user funds are frozen or lost: since `rsETHPrice == 0` only before any deposits succeed, no deposited assets are at risk. The protocol simply cannot accept new deposits during this window. This maps to the allowed Low impact class: *Contract fails to deliver promised returns, but doesn't lose value*. The claim's Medium classification (Temporary freezing of funds) is not supported because no funds are actually held in the contract during the vulnerable window.

## Likelihood Explanation
The window exists between contract deployment/upgrade and the first successful call to `updateRSETHPrice()`. Because `updateRSETHPrice()` is permissionless, any actor can close the window immediately. However, nothing in the contract enforces that it is called before deposits are enabled. A deployment script omission or a failed transaction is sufficient to trigger the condition. Any user attempting to deposit during this window will encounter the revert.

## Recommendation
Set a safe initial value for `rsETHPrice` inside `initialize()`:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

Alternatively, add a zero-check guard in `getRsETHAmountToMint()`:

```solidity
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert RsETHPriceNotInitialized();
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / currentRsETHPrice;
```

## Proof of Concept
1. Deploy `LRTOracle` and call `initialize()`. Confirm `rsETHPrice == 0`.
2. Deploy `LRTDepositPool` pointing to the oracle. Do **not** call `updateRSETHPrice()`.
3. Any user calls `depositETH{value: 1 ether}(0, "")`.
4. Execution: `depositETH` → `_beforeDeposit` → `getRsETHAmountToMint` → `(1e18 * assetPrice) / 0` → panic revert.
5. All deposits revert until `updateRSETHPrice()` is called by any actor.

Foundry test sketch:
