### Title
Dijkstra Batch Transaction Minimum Fee Validation Ignores Sub-Transaction Reference Script Sizes — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The `EraUTxO DijkstraEra` instance delegates minimum fee computation to `getConwayMinFeeTxUtxo`, which only measures reference-script bytes in the **top-level** transaction. Sub-transactions embedded in a Dijkstra batch can each carry their own reference inputs pointing to UTxO entries that contain large Plutus scripts. Those scripts are deserialized during validation but their size is never added to the minimum-fee threshold. The function `batchNonDistinctRefScriptsSize` was introduced specifically to aggregate reference-script sizes across the entire batch, yet it is never wired into the fee-validation path.

---

### Finding Description

**Root cause — wrong size function used in `getMinFeeTxUtxo`**

`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` line 141 sets:

```haskell
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
``` [1](#0-0) 

`getConwayMinFeeTxUtxo` is defined in `eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs`:

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [2](#0-1) 

`txNonDistinctRefScriptsSize` only inspects the inputs and reference inputs of the **top-level** `TxBody`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

It never traverses `subTransactionsTxBodyL`.

**The correct function already exists but is unused**

`batchNonDistinctRefScriptsSize` was added to the Dijkstra module precisely to aggregate reference-script sizes across the whole batch:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

This function is exported but never called from the fee-validation path.

**Fee validation in the UTXO rule**

`dijkstraUtxoTransition` enforces the minimum fee with:

```haskell
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
``` [5](#0-4) 

`validateFeeTooSmallUTxO` calls `getMinFeeTxUtxo pp tx utxo`, which for `DijkstraEra` resolves to `getConwayMinFeeTxUtxo` → `txNonDistinctRefScriptsSize` (top-level only). Sub-transaction reference scripts are therefore invisible to this check.

**Sub-transactions do carry reference inputs**

The `DijkstraSubTxBodyRaw` record includes `dstbrReferenceInputs`:

```haskell
DijkstraSubTxBodyRaw ::
  { dstbrSpendInputs    :: !(Set TxIn)
  , dstbrReferenceInputs :: !(Set TxIn)
  ...
``` [6](#0-5) 

And the `SUBUTXO` rule validates those inputs against `originalUtxo`:

```haskell
runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
``` [7](#0-6) 

So sub-transaction reference inputs are resolved from the real UTxO and their scripts are deserialized during validation — but their byte-size is never added to the minimum fee.

---

### Impact Explanation

The reference-script fee (`minFeeRefScriptCostPerByte` × tiered size) was introduced in Conway specifically to price the deserialization cost of large Plutus scripts and prevent DDoS attacks. By placing large reference scripts exclusively in sub-transactions, an attacker can force nodes to deserialize arbitrarily large scripts while paying only the base size-proportional fee for the top-level transaction body. This allows attacker-controlled transactions to modify the effective fee paid outside the design parameters of the protocol — matching the **Medium** impact class: *attacker-controlled transactions modify fees outside design parameters*.

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a Dijkstra batch transaction. No special role, key, or governance majority is required. The Dijkstra era is the only era with sub-transactions, so this is a new, era-specific attack surface with no prior mitigation. The exploit is straightforward: place a UTxO entry with a large reference script on-chain, then reference it from a sub-transaction's `referenceInputs` field.

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance with a Dijkstra-specific implementation that uses `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`:

```haskell
-- proposed replacement in eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getDijkstraMinFeeTxUtxo ::
  ( EraTx era
  , DijkstraEraTxBody era
  , BabbageEraTxBody era
  ) =>
  PParams era -> Tx TopTx era -> UTxO era -> Coin
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx

instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo
```

This ensures the tiered reference-script fee covers every script deserialized during batch validation, not just those referenced by the top-level transaction.

---

### Proof of Concept

1. Deploy a UTxO entry `u` containing a large Plutus script (e.g., 50 KiB) as a reference script.
2. Construct a Dijkstra sub-transaction `subTx` with `referenceInputs = {u}` and any valid spend input.
3. Construct a top-level Dijkstra transaction `tx` with `subTransactions = [subTx]` and **no** reference inputs of its own.
4. Set `tx.fee` to the minimum fee computed by `getConwayMinFeeTxUtxo pp tx utxo` (which ignores `u`'s script size).
5. Submit `tx`. The `validateFeeTooSmallUTxO` check passes because it only measures the top-level transaction's reference scripts (zero bytes). The node nonetheless deserializes the 50 KiB script from `u` when processing `subTx` in `SUBUTXO`, incurring the full deserialization cost with no corresponding fee premium.
