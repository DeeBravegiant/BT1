Audit Report

## Title
Stale `rsETHPrice` Inflates `assetsCommitted` After Beacon-Chain Checkpoint Finalization, Temporarily Blocking New Withdrawal Initiations â€” (`contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTOracle.rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is explicitly called. Between a beacon-chain balance decrease and the finalization of the corresponding EigenLayer checkpoint, `getWithdrawableShares()` still returns the pre-decrease value because `beaconChainSlashingFactor` has not yet been reduced. If `updateRSETHPrice()` is called in this window, it stores an inflated price. A user who calls `initiateWithdrawal()` during this window commits an inflated `expectedAssetAmount` into `assetsCommitted`. After the checkpoint finalizes, `getTotalAssetDeposits()` (computed live) drops while `assetsCommitted` remains at the inflated value, causing `getAvailableAssetAmount()` to return zero and blocking all subsequent `initiateWithdrawal()` calls for up to 8 days.

## Finding Description

**Stored price vs. live total-assets mismatch**

`rsETHPrice` is a stored `uint256` in `LRTOracle`: [1](#0-0) 

It is only updated by explicit calls to `updateRSETHPrice()` (public, permissionless) or `updateRSETHPriceAsManager()`: [2](#0-1) 

`_updateRsETHPrice()` calls `_getTotalEthInProtocol()`, which calls `getTotalAssetDeposits()` on the deposit pool: [3](#0-2) 

`getTotalAssetDeposits()` calls `INodeDelegator.getAssetBalance()` for each NDC: [4](#0-3) 

`getAssetBalance()` in `NodeDelegatorHelper` calls `getWithdrawableShares()` on EigenLayer's `DelegationManager`: [5](#0-4) [6](#0-5) 

EigenLayer's `getWithdrawableShares()` computes `depositShares * depositScalingFactor * beaconChainSlashingFactor`. The `beaconChainSlashingFactor` is only updated when a checkpoint is finalized (via `verifyCheckpointProofs`). Before finalization, the factor retains its pre-decrease value, so `getWithdrawableShares()` returns an inflated share count relative to the true post-checkpoint balance. This is confirmed by the `BeaconChainSlashingFactorDecreased` event in `IEigenPodManager` and the `SlashingLib.calcWithdrawable()` formula: [7](#0-6) [8](#0-7) 

**`initiateWithdrawal()` uses the stored (potentially stale) price**

`getExpectedAssetAmount()` reads the stored `rsETHPrice`: [9](#0-8) 

`initiateWithdrawal()` adds this inflated amount to `assetsCommitted`: [10](#0-9) 

**`getAvailableAssetAmount()` uses a live computation**

After the checkpoint finalizes, `getTotalAssetDeposits()` drops to `T_low`, but `assetsCommitted` remains at `E_high`: [11](#0-10) 

The result is `max(0, T_low âˆ’ E_high) = 0`, so the guard in `initiateWithdrawal()` reverts with `ExceedAmountToWithdraw` for all subsequent callers.

**`unlockQueue()` cannot immediately clear `assetsCommitted`**

`_unlockWithdrawalRequests()` breaks out of the loop if the withdrawal delay has not passed: [12](#0-11) 

The default delay is 8 days: [13](#0-12) 

`assetsCommitted` is only reduced when `unlockQueue()` processes the request after the delay: [14](#0-13) 

**`pricePercentageLimit` does not fully mitigate this**

The downside-protection pause only triggers when `updateRSETHPrice()` is called *post-checkpoint* and the drop exceeds `pricePercentageLimit`. For drops within the limit (e.g., a 3% inactivity penalty with a 5% limit), the protocol does not pause, and the inflated `assetsCommitted` persists: [15](#0-14) 

## Impact Explanation

All calls to `initiateWithdrawal()` for the affected asset revert with `ExceedAmountToWithdraw` for up to 8 days (the withdrawal delay). Users holding rsETH cannot queue new withdrawals during this window. Their rsETH is not lost, but the standard withdrawal path is temporarily unavailable. This matches **Medium â€” Temporary freezing of funds**.

## Likelihood Explanation

Beacon-chain checkpoints are a routine EigenLayer operation. Any validator balance decrease (inactivity leak, partial slash) creates this window. `updateRSETHPrice()` is a public, permissionless function callable by anyone at any time, including during the pre-checkpoint window. No special privileges or front-running are required; a regular user calling `initiateWithdrawal()` at the wrong moment is sufficient to trigger the freeze. The condition `T_low < E_high` is reachable whenever the committed withdrawal amount is a meaningful fraction of total assets and the slash is non-trivial.

## Recommendation

1. In `initiateWithdrawal()`, compute `expectedAssetAmount` using a freshly computed price (call `_getTotalEthInProtocol()` inline or require a recent `updateRSETHPrice()` block timestamp) rather than the stored `rsETHPrice`.
2. Alternatively, add a staleness guard: revert if `rsETHPrice` was last updated more than N blocks ago.
3. Consider adding a `maximumRsEthPrice` bound to `initiateWithdrawal()` (analogous to the bounds already present in `unlockQueue()`) so users can protect themselves against stale prices.

## Proof of Concept

```
1. Fork mainnet at a block where an NDC has active beacon-chain validators.
2. Simulate a validator balance decrease (e.g., via vm.store on the EigenPod's
   restakedBalanceGwei or by advancing the beacon chain state).
3. Call EigenPod.startCheckpoint() â€” checkpoint is pending, not finalized.
4. Call LRTOracle.updateRSETHPrice() â€” stores P_high (inflated, because
   beaconChainSlashingFactor has not yet been reduced).
5. Call LRTWithdrawalManager.initiateWithdrawal(asset, rsETHAmount, "") â€”
   assert assetsCommitted[asset] == E_high (inflated).
6. Call EigenPod.verifyCheckpointProofs(...) â€” checkpoint finalizes,
   beaconChainSlashingFactor decreases.
7. Assert lrtDepositPool.getTotalAssetDeposits(asset) < assetsCommitted[asset].
8. Call initiateWithdrawal() again â€” assert it reverts with ExceedAmountToWithdraw.
9. Assert block.number < request.withdrawalStartBlock + withdrawalDelayBlocks
   (unlockQueue cannot yet clear assetsCommitted).
```
