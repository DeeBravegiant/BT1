### Title
`valid_proposals` Retains Commitment-Mismatched Proposals After `ProposalFinMismatch`, Enabling Wrong Block Commitment via Reproposal — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

In `validate_proposal`, the proposal is unconditionally inserted into `valid_proposals` **before** the `ProposalFin` commitment check. When the check fails (`ProposalFinMismatch`), the function returns an error but the proposal — whose content was already truncated to the proposer-supplied `executed_transaction_count` — remains permanently in `valid_proposals`. A malicious proposer can exploit this to plant a truncated (fewer-transaction) proposal in every validator's `valid_proposals` store, which a later reproposer will broadcast as a legitimate block, causing the network to commit a block with fewer transactions than the proposer actually executed.

### Impact Explanation

**Attack path:**

1. Malicious proposer (round R) streams N transactions to all validators, but sets `executed_transaction_count = K` where K < N in `ProposalFin`.
2. Each validator's `handle_proposal_part` truncates its local `content` to K transactions and calls `finish_proposal(K)` on the batcher.
3. The batcher executes K transactions and returns commitment C_K.
4. `validate_proposal` inserts `(init, K-tx content, C_K)` into `valid_proposals`.
5. The commitment check fires: proposer's claimed commitment C_N ≠ C_K → `ProposalFinMismatch` returned. The K-tx entry is **never removed** from `valid_proposals`.
6. Round R fails. Round R+1 selects a different proposer V.
7. V has a "valid value" in `valid_proposals`: K transactions with commitment C_K.
8. V calls `send_reproposal`, broadcasting K transactions + C_K.
9. All other validators receive K transactions, execute them, compute C_K, compare → match. Reproposal accepted.
10. Network commits a block with K transactions instead of N.

The N−K excluded transactions must be resubmitted. By choosing K to exclude specific transactions (e.g., a high-value transfer or a contract deployment), the attacker achieves **selective transaction censorship** at the block level. The committed block hash, state root, and all receipts/events are derived from the wrong (truncated) transaction set.

This matches the "Wrong state, receipt, event … or revert result" and "High: RPC execution … returns an authoritative-looking wrong value" impact categories.

### Likelihood Explanation

Any node that is legitimately selected as the round proposer can trigger this with a single malformed `ProposalFin`. No special privilege beyond normal proposer selection is required. The attack succeeds whenever a subsequent validator is selected as reproposer and has the poisoned entry in `valid_proposals`, which is guaranteed if the original proposal reached the `finish_proposal` stage on that validator.

### Recommendation

Remove the proposal from `valid_proposals` if the commitment check fails:

```rust
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    valid_proposals.remove_proposal(&args.proposal_id); // ← add this
    return Err(ValidateProposalError::ProposalFinMismatch);
}
```

Alternatively, restructure the code to insert into `valid_proposals` only after the commitment check passes, and handle the reproposal race condition through a different mechanism (e.g., a pending-proposal slot that is promoted to `valid_proposals` only on success).

### Proof of Concept

```
Round R:
  Proposer sends txs [T1..T10] to all validators.
  ProposalFin.executed_transaction_count = 5  (lie: only 5 of 10)

Each validator:
  handle_proposal_part: content truncated to [T1..T5]
  finish_proposal(5) → batcher returns commitment C_5
  valid_proposals.insert(init, [T1..T5], C_5)   ← stored
  built_block (C_5) ≠ fin.proposal_commitment (C_10)
  → ProposalFinMismatch returned, [T1..T5] entry NOT removed

Round R+1:
  Validator V selected as proposer.
  V calls send_reproposal with stored [T1..T5], C_5.
  Other validators: receive [T1..T5], execute, compute C_5, compare → match.
  Block committed: only T1..T5 included.
  T6..T10 silently dropped from this block.
```
