### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Creates Unbacked Wrapped Supply — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` records and emits the caller-supplied `amount` parameter rather than the actual tokens received by the contract. For fee-on-transfer ERC-20 tokens, the contract receives fewer tokens than `amount`, but the `InitTransfer` event — which drives cross-chain settlement and minting on destination chains — is emitted with the full `amount`. This creates unbacked wrapped token supply on destination chains, breaking the 1:1 backing guarantee.

---

### Finding Description

In `OmniBridge.initTransfer()`, when a non-bridge, non-custom-minter ERC-20 token is deposited, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-supplied; actual received may be less
);
``` [1](#0-0) 

Immediately after, the event is emitted using the original `amount` parameter — not a post-transfer balance check:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← inflated; not actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

The `InitTransfer` event is the authoritative cross-chain message. The NEAR bridge (and other destination chains) read this event to determine how many tokens to mint or release. Because the emitted `amount` exceeds the actual locked amount, the destination chain mints more wrapped tokens than are backed by locked collateral.

The same pattern exists in the Starknet implementation:

```cairo
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
// ...
// InitTransfer event emitted with original `amount`
``` [3](#0-2) 

On the redemption side, `finTransfer` releases exactly `payload.amount` tokens:

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [4](#0-3) 

Since the contract locked less than `payload.amount` per deposit, the vault will eventually be drained before all wrapped token holders can redeem, permanently locking the last redeemers' value.

---

### Impact Explanation

**Critical — Irreversible fund lock / unbacked supply.**

For every `initTransfer` call with a fee-on-transfer token:
- Contract receives `amount - fee_deducted` tokens
- Destination chain mints `amount` wrapped tokens
- Backing deficit accumulates with each transfer
- When the vault is exhausted, later `finTransfer` calls revert, permanently locking wrapped token holders' value with no redemption path

This breaks the core bridge invariant: locked collateral ≥ outstanding wrapped supply.

---

### Likelihood Explanation

**Medium.** Fee-on-transfer tokens (e.g., USDT on some chains, deflationary tokens) are widely deployed. The `initTransfer` function is fully public with no token allowlist. Any user can call it with a fee-on-transfer token. The deficit is small per transaction but compounds over time, making eventual fund lock certain rather than probabilistic.

---

### Recommendation

Replace the caller-supplied `amount` with a pre/post balance check to determine the actual received amount:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// Use actualReceived in the event and downstream accounting
```

Apply the same fix to the Starknet `init_transfer` in `starknet/src/omni_bridge.cairo`. [5](#0-4) 

---

### Proof of Concept

1. Deploy or use any fee-on-transfer ERC-20 token (e.g., 1% fee per transfer).
2. Approve `OmniBridge` for 1000 tokens.
3. Call `initTransfer(tokenAddress, 1000, 0, 0, "recipient.near", "")`.
4. Contract receives 990 tokens (1% fee deducted by the token).
5. `InitTransfer` event is emitted with `amount = 1000`.
6. NEAR bridge reads the event and mints 1000 wrapped tokens to `recipient.near`.
7. Repeat 100 times: vault holds 99,000 tokens, but 100,000 wrapped tokens exist.
8. The 100th redeemer calls `finTransfer` on EVM — the `safeTransfer` of 1000 tokens reverts because the vault is short. Their wrapped tokens are permanently unclaimable.

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }
