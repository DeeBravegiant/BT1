### Title
Production Consensus Quorum Reduced to 1/2 via `assume_no_malicious_validators: true`, Breaking Byzantine Fault Tolerance — (File: `crates/apollo_consensus_manager/src/consensus_manager.rs`)

---

### Summary

The production deployment configuration explicitly sets `assume_no_malicious_validators: true`, which causes `ConsensusManager::create_run_consensus_args` to select `QuorumType::Honest` (1/2 threshold) instead of `QuorumType::Byzantine` (2/3 threshold) for all consensus decisions. A validator controlling more than 50% of total stake weight can unilaterally reach prevote and precommit quorum, forcing a block decision without any other validator's agreement.

---

### Finding Description

`ConsensusManagerConfig` exposes a boolean field `assume_no_malicious_validators`. [1](#0-0) 

When this flag is `true`, `create_run_consensus_args` selects `QuorumType::Honest`: [2](#0-1) 

`QuorumType::Honest` maps to `HONEST_QUORUM`, which is a 1/2 threshold: [3](#0-2) 

This quorum is passed directly into `StateMachine::new` and governs both prevote and precommit quorum checks for every height: [4](#0-3) 

The standard Tendermint Byzantine quorum is 2/3: [5](#0-4) 

The production deployment app config for both the primary and replacer consensus manager nodes has this flag hardcoded to `true`: [6](#0-5) [7](#0-6) 

The same production config also sets `require_virtual_proposer_vote: false`: [8](#0-7) 

This removes the secondary guard that would otherwise require the virtual proposer to have voted in favor before a decision is reached. With both flags active, the only requirement for a block decision is that validators whose combined weight exceeds 50% of total stake cast matching prevotes and precommits.

---

### Impact Explanation

With `HONEST_QUORUM` (1/2) active, a single validator entity controlling more than 50% of total staking weight can:

1. Cast prevotes for a self-built proposal — immediately satisfying prevote quorum alone.
2. Cast precommits for the same proposal — immediately satisfying precommit quorum alone.
3. Force `Decision` to be emitted by the state machine with no other validator's participation.

The decided `ProposalCommitment` is then passed to `decision_reached`, which drives the batcher to commit the block, persist state, and anchor to L1. The committed block may contain any transactions the dominant validator chose to include or exclude, producing wrong state roots, wrong receipts, wrong events, and wrong L1 messages — all of which are accepted as authoritative by the rest of the system.

This maps directly to: **Critical. Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

---

### Likelihood Explanation

The likelihood is low in absolute terms because it requires a single entity to accumulate more than 50% of total staking weight. However, the precondition is already satisfied at the configuration level: the production deployment ships with `assume_no_malicious_validators: true`, meaning the weakened threshold is active on every node that loads this config. No runtime exploit or code change is needed — the invariant is already broken by the deployed configuration. Any future validator set composition that concentrates >50% weight in one entity immediately becomes exploitable with zero additional steps.

---

### Recommendation

1. **Set `assume_no_malicious_validators: false` in all production deployment configs.** The standard Tendermint 2/3 Byzantine quorum (`BYZANTINE_QUORUM`) must be the default for any production network.

2. **Enable `require_virtual_proposer_vote: true` in production.** This provides an additional layer requiring the designated proposer to have explicitly voted in favor before a decision is finalized.

3. **Add a startup assertion or config validator** that rejects `assume_no_malicious_validators: true` unless an explicit `--allow-honest-quorum` override flag is passed, preventing accidental production deployment with the weakened threshold.

4. **Treat `assume_no_malicious_validators` as a test-only flag.** Rename it or gate it behind a `#[cfg(test)]` or a separate `testing_overrides` config section to make its danger surface explicit.

---

### Proof of Concept

Given a committee of two validators with weights `[("A", 6), ("B", 4)]` (total weight 10), and `assume_no_malicious_validators: true` (quorum = 1/2, i.e., weight > 5):

1. Validator A (weight 6) builds and broadcasts a proposal for height H.
2. Validator A casts a prevote for its own proposal. Weight 6 > 5 → **prevote quorum met immediately** by A alone.
3. Validator A casts a precommit for its own proposal. Weight 6 > 5 → **precommit quorum met immediately** by A alone.
4. `StateMachine` emits `Decision` for A's proposal without any vote from B.
5. The block is committed with whatever transactions A chose to include.

Under `BYZANTINE_QUORUM` (2/3, weight > 6.67), A's weight of 6 does not meet quorum, and B's participation is mandatory — preserving the safety guarantee. [9](#0-8)
