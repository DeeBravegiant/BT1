### Title
Unguarded `summaries[sub_epoch_n]` Index Access in Weight Proof Validation Allows Peer-Triggered Sync Crash — (`File: chia/full_node/weight_proof.py`)

### Summary

`_validate_sub_epoch_segments()` indexes into the `summaries` list using `sub_epoch_n` values taken directly from attacker-controlled `sub_epoch_segments` data, without first verifying that `sub_epoch_n < len(summaries)`. A malicious peer can craft a `RespondProofOfWeight` message whose `sub_epoch_segments` field contains a segment with `sub_epoch_n` equal to or greater than the number of entries in `sub_epochs`, causing an unhandled `IndexError` that aborts the syncing node's weight-proof validation and prevents it from completing a sync.

### Finding Description

During sync, a Chia full node calls `request_validate_wp()`, which requests a `WeightProof` from a randomly chosen peer and passes it to `validate_weight_proof()`. [1](#0-0) 

Inside `validate_weight_proof()`, `_validate_sub_epoch_summaries()` processes `weight_proof.sub_epochs` and produces a `summaries` list of length N (one entry per sub-epoch summary in the proof). [2](#0-1) 

These `summaries` are then serialised and passed to `validate_weight_proof_inner()`, which calls `_validate_sub_epoch_segments()`. [3](#0-2) 

Inside `_validate_sub_epoch_segments()`, the code iterates over `segments_by_sub_epoch` — a dict keyed by `sub_epoch_n` values taken from `weight_proof.sub_epoch_segments` (the second, independently-supplied part of the proof). For each `sub_epoch_n`, it directly indexes into `summaries` at two points with no bounds check:

```python
prev_ses = summaries[sub_epoch_n - 1]          # line 983
...
if not summaries[sub_epoch_n].reward_chain_hash == rc_sub_slot_hash:  # line 985
``` [4](#0-3) 

The same unchecked index access occurs inside `__get_rc_sub_slot()`, called at line 979:

```python
ses = summaries[uint32(segment.sub_epoch_n - 1)]   # line 1452
``` [5](#0-4) 

`summaries` has exactly N entries (indices `0` to `N-1`), derived from `weight_proof.sub_epochs`. The `sub_epoch_n` values in `weight_proof.sub_epoch_segments` are a separate, independently-serialised field. There is no validation that every `sub_epoch_n` appearing in `sub_epoch_segments` is less than `len(summaries)`. If an attacker supplies a segment with `sub_epoch_n >= N`, Python raises an `IndexError`.

The `_get_curr_diff_ssi()` helper called at line 974 uses a safe slice (`summaries[0:idx]`), so it does not trigger the crash, but the two direct index accesses at lines 983 and 985 are unguarded. [6](#0-5) 

### Impact Explanation

The `IndexError` propagates out of `validate_weight_proof_inner` and is caught in `request_validate_wp`:

```python
except Exception as e:
    await weight_proof_peer.close(CONSENSUS_ERROR_BAN_SECONDS)
    raise ValueError(f"Weight proof validation threw an error {e}")
``` [7](#0-6) 

The `ValueError` propagates to `_sync`, which catches it and logs an error, aborting the sync attempt. [8](#0-7) 

If the attacker controls all peers the victim is connected to (e.g., an eclipse attack, or a node with only one peer), every weight-proof request returns a crafted proof, every validation attempt raises `IndexError`, and the victim node is permanently unable to sync to the chain tip. This matches the High impact category: long-lived inability for honest nodes to process sync updates under normal network assumptions.

The wallet node is equally affected: `WalletWeightProofHandler.validate_weight_proof()` calls the same `validate_weight_proof_inner()` path. [9](#0-8) 

### Likelihood Explanation

Any unprivileged peer that can respond to a `RequestProofOfWeight` message can trigger this. No keys, admin access, or cryptographic break are required. The attacker only needs to:

1. Run a Chia full node.
2. Wait for a syncing victim to connect and request a weight proof.
3. Respond with a `RespondProofOfWeight` whose `sub_epoch_segments` list contains one segment with `sub_epoch_n` set to any value `>= len(sub_epochs)`.

The victim node selects its weight-proof peer randomly from peers that advertise the target peak, so an attacker with even a single connection to the victim can attempt this. [10](#0-9) 

### Recommendation

Before the loop in `_validate_sub_epoch_segments()`, add an explicit bounds check:

```python
for sub_epoch_n, segments in segments_by_sub_epoch.items():
    if sub_epoch_n >= len(summaries):
        log.error(f"sub_epoch_n {sub_epoch_n} out of range (summaries len={len(summaries)})")
        return None
    ...
```

Apply the same guard inside `__get_rc_sub_slot()` before `summaries[uint32(segment.sub_epoch_n - 1)]`. [6](#0-5) 

### Proof of Concept

1. Obtain a valid `WeightProof` `wp` for a chain with N sub-epoch summaries (so `len(wp.sub_epochs) == N`).
2. Craft a replacement `sub_epoch_segments` list by taking any existing segment and setting its `sub_epoch_n` field to `N` (or any value `>= N`).
3. Serve this crafted proof in response to a `RequestProofOfWeight` from a victim node.
4. The victim calls `_validate_sub_epoch_segments()`, reaches line 985 (`summaries[N]`), and raises `IndexError`.
5. The victim's sync attempt fails with `"Error with syncing: <class 'ValueError'>Weight proof validation threw an error ..."`. [11](#0-10)
