### Title
Governance Proposal Deposit Permanently Lost When Return Address Is Unregistered During Bootstrap Phase — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version 9), the `GOV` rule skips the check that a proposal's return address must correspond to a registered staking credential. An unprivileged user can submit a governance proposal with an unregistered return address, pay the full `ppGovActionDepositL` deposit, and permanently lose that deposit to the treasury when the proposal expires — with no mechanism for recovery.

---

### Finding Description

In `conwayGovTransition`, the return-address registration check is guarded by `unless (hardforkConwayBootstrapPhase ...)`:

```haskell
unless (hardforkConwayBootstrapPhase $ pp ^. ppProtocolVersionL) $ do
  let refundAddress = proposal ^. pProcReturnAddrL
  isAccountRegistered (refundAddress ^. accountAddressCredentialL) (certDState ^. accountsL)
    ?! (injectFailure . ProposalReturnAccountDoesNotExist) refundAddress
``` [1](#0-0) 

`hardforkConwayBootstrapPhase` is `True` exactly when `pvMajor pv == natVersion @9`: [2](#0-1) 

During bootstrap the check is entirely skipped, yet the deposit check (lines 522–530) is **not** inside the `unless` block and always runs — the deposit is always collected: [3](#0-2) 

When the proposal is later removed (expired or enacted), `returnProposalDeposits` in the `EPOCH` rule attempts to credit the deposit to the return-address credential. If that credential is not registered at that point, the deposit is placed in `unclaimed` and transferred to the treasury:

```haskell
processProposal gas (!accounts, !unclaimed)
  | (Just _accountState, newAccounts) <- updateLookupAccountState addRefund cred accounts =
      (newAccounts, unclaimed)
  | otherwise = (accounts, Map.insert (gasId gas) (gasDeposit gas) unclaimed)
``` [4](#0-3) 

The `unclaimed` map is then moved to the treasury: [5](#0-4) 

Bootstrap-allowed actions include `InfoAction`, `HardForkInitiation`, and security-group `ParameterChange`. Any of these can be submitted with an unregistered return address during bootstrap. The test suite explicitly confirms the check is only enforced post-bootstrap: [6](#0-5) 

The `depositMovesToTreasuryWhenStakingAddressUnregisters` test confirms the deposit-to-treasury path is real and exercised: [7](#0-6) 

---

### Impact Explanation

**High — Permanent freezing of deposits where recovery requires a hard fork.**

An unprivileged proposer who submits a governance proposal during bootstrap with an unregistered return address permanently loses their `ppGovActionDepositL` ADA. The deposit is absorbed by the treasury and cannot be reclaimed by the proposer through any on-chain mechanism. The proposer's ADA is effectively destroyed from their perspective. This matches the allowed impact: *"Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork."*

---

### Likelihood Explanation

**Medium.** The bootstrap phase is protocol version 9 (the initial Conway era). Any chain or testnet that starts at protocol version 9 is affected. The entry path requires only a standard governance proposal transaction — no privileged access, no key compromise, no majority. A user who mistakenly (or maliciously, to grief another user via a crafted transaction) supplies an unregistered return address will lose their deposit. The deposit amount is set by `ppGovActionDepositL` (default 100,000,000 lovelace = 100 ADA on mainnet), making the loss economically significant.

---

### Recommendation

The return-address registration check should be enforced unconditionally, including during bootstrap. The `unless (hardforkConwayBootstrapPhase ...)` guard should be removed from the return-address check, or alternatively the deposit should not be accepted when the return address is unregistered, regardless of the current protocol phase. [8](#0-7) 

---

### Proof of Concept

**Entry path** (unprivileged transaction sender during bootstrap, `pvMajor == 9`):

1. Generate a fresh key hash `kh` and derive its account address `addr` — do **not** register it as a staking credential.
2. Submit a governance proposal transaction:
   - `pProcGovAction = InfoAction` (bootstrap-allowed)
   - `pProcReturnAddr = addr` (unregistered)
   - `pProcDeposit = ppGovActionDepositL` (required deposit, e.g. 100 ADA)
3. The `GOV` rule accepts the proposal: `checkBootstrapProposal` passes (InfoAction is a bootstrap action), the return-address check is skipped (`unless hardforkConwayBootstrapPhase`), and the deposit check passes.
4. Advance `ppGovActionLifetimeL + 1` epochs without ratifying the proposal.
5. At the epoch boundary, `returnProposalDeposits` finds `kh` is not in the accounts map → deposit is added to `unclaimed` → transferred to treasury.
6. The proposer's 100 ADA is permanently in the treasury with no on-chain recovery path.

The analog to the external report is exact: the external contract allowed staking after `endTime` (a missing time-bound check at deposit time → funds stuck forever); here the ledger allows a proposal with an unregistered return address during bootstrap (a missing phase-bound check at proposal submission time → deposit permanently lost).
