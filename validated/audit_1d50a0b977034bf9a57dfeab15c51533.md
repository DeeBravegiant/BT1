### Title
Gateway Admission Uses Stale `l2_gas_price` Instead of EIP-1559-Computed `next_l2_gas_price` for Resource Bounds Validation — (File: `crates/apollo_gateway/src/gateway_fixed_block_state_reader.rs`)

---

### Summary

`StatefulTransactionValidator.validate_resource_bounds()` gates admission on the **current block's** `l2_gas_price` read from the committed block header, while the sequencer will actually execute the next block at the EIP-1559-adjusted `next_l2_gas_price` stored in the same header. The two values diverge every time a block is not exactly at the gas target. The gateway therefore admits V3 transactions whose `max_price_per_unit` is too low for the next block (when the previous block was full), or spuriously rejects transactions that would be valid (when the previous block was under-target). A developer TODO in the code explicitly acknowledges the wrong field is being read.

---

### Finding Description

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client()` constructs a `BlockInfo` from the latest committed `BlockHeaderWithoutHash`. For the STRK L2 gas price it reads:

```rust
// crates/apollo_gateway/src/gateway_fixed_block_state_reader.rs  lines 46-49
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [1](#0-0) 

`block_header.l2_gas_price.price_in_fri` is the L2 gas price **of the block that was just committed**. The same `BlockHeaderWithoutHash` also carries `next_l2_gas_price`, which is the EIP-1559-adjusted price that the sequencer will use for the **next** block:

```rust
// crates/apollo_storage/src/header.rs  lines 88-89
pub l2_gas_consumed: GasAmount,
pub next_l2_gas_price: GasPrice,
``` [2](#0-1) 

The `BlockInfo` returned by `get_block_info()` is consumed in two places inside `StatefulTransactionValidator`:

1. **`validate_resource_bounds`** — uses `block_info.gas_prices.strk_gas_prices.l2_gas_price` as `previous_block_l2_gas_price` and rejects any V3 transaction whose `max_price_per_unit` is below `previous_block_l2_gas_price × min_gas_price_percentage / 100`. [3](#0-2) 

2. **`run_validate_entry_point`** — builds a `BlockContext` from the same `block_info` and runs the account's `__validate__` entry point under it. [4](#0-3) 

The developer comment at the call site explicitly flags the wrong field:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [5](#0-4) 

Meanwhile, the consensus orchestrator correctly updates `self.l2_gas_price` to the EIP-1559 result inside `finalize_decision` (called from `decision_reached`) before the next `build_proposal`:

```rust
// crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs  line 517
self.update_l2_gas_price(height, l2_gas_used);
``` [6](#0-5) 

The gateway never reads this updated value; it re-derives the price from the committed block header each time a new `StatefulTransactionValidator` is instantiated, and it reads the wrong field.

---

### Impact Explanation

**Case 1 — full block (`gas_used > gas_target`):**  
`next_l2_gas_price > l2_gas_price`. The gateway threshold is anchored to the lower `l2_gas_price`. A V3 transaction with `max_price_per_unit` in the range `[l2_gas_price × pct/100, next_l2_gas_price × pct/100)` passes gateway admission and enters the mempool, but the batcher will reject it at execution time because the block's actual L2 gas price exceeds the transaction's stated maximum. This is **admission of an invalid transaction before sequencing** (High impact per scope).

**Case 2 — under-target block (`gas_used < gas_target`):**  
`next_l2_gas_price < l2_gas_price`. The gateway threshold is anchored to the higher `l2_gas_price`. A V3 transaction with `max_price_per_unit` in the range `[next_l2_gas_price × pct/100, l2_gas_price × pct/100)` is rejected by the gateway even though it would succeed in the next block. This is **rejection of a valid transaction before sequencing** (High impact per scope).

---

### Likelihood Explanation

The EIP-1559 formula (`calculate_next_base_gas_price`) adjusts the price by at most `price / gas_price_max_change_denominator` per block, so the gap between `l2_gas_price` and `next_l2_gas_price` is bounded. However:

- The discrepancy is present on **every block** that is not exactly at the gas target, which is the common case under real load.
- Any unprivileged user submitting a V3 transaction triggers the path.
- The check is enabled when `config.validate_resource_bounds = true`.
- The `min_gas_price_percentage` multiplier does not close the gap; it scales both the stale and the correct threshold equally. [7](#0-6) 

---

### Recommendation

In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client()`, replace the stale field with the EIP-1559-computed value already present in the header:

```rust
// Before (stale):
l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,

// After (correct):
l2_gas_price: block_header.next_l2_gas_price.try_into()?,
``` [1](#0-0) 

The `next_l2_gas_price` field is already populated in `BlockHeaderWithoutHash` (confirmed by the `try_sync` test that sets it) and is the value the sequencer will actually enforce in the next block.

---

### Proof of Concept

1. Mine a block at maximum capacity so `gas_used ≫ gas_target`. The EIP-1559 formula raises `next_l2_gas_price` by up to `l2_gas_price / gas_price_max_change_denominator` above `l2_gas_price`.
2. Submit a V3 `INVOKE` transaction with `l2_gas.max_price_per_unit = l2_gas_price` (the stale value). The gateway computes `threshold = l2_gas_price × min_gas_price_percentage / 100 ≤ l2_gas_price`, so the transaction passes `validate_resource_bounds` and is admitted to the mempool.
3. The batcher pulls the transaction for the next block, which runs at `next_l2_gas_price > l2_gas_price`. The blockifier's fee enforcement rejects the transaction because `max_price_per_unit < next_l2_gas_price`.
4. The transaction was admitted by the gateway but is invalid for the block it will be sequenced into — a direct analog to the Fuji Protocol's stale-index read: the gateway read `l2_gas_price` without first "updating" to `next_l2_gas_price`.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
