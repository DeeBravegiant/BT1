### Title
Missing Output Validation for `AppPublicKey` CKD Variant Allows Single Byzantine Node to Forge Confidential Key Derivation Response - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` contract method performs cryptographic output validation only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. For the `AppPublicKey` variant, the match arm is empty — no pairing check or any other verification is applied to the submitted `CKDResponse`. A single attested participant (below the signing threshold) can call `respond_ckd` with an arbitrary forged `CKDResponse` and the contract will accept it, resolve all pending yields, and deliver the forged output to every waiting caller.

---

### Finding Description

In `respond_ckd`, after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← empty: no validation
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, binding the response to the network's master public key and the user's app public key. [2](#0-1) 

For `AppPublicKey`, the arm is empty. The contract performs zero cryptographic verification of `big_c` or `big_y`. After this match, `resolve_yields_for` is called unconditionally, delivering the unverified response to all waiting callers. [3](#0-2) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active`, which checks that the caller is a single attested participant — it does not enforce any threshold agreement among participants. [4](#0-3) 

By contrast, the `respond` method for threshold signatures verifies the submitted signature against the cryptographically derived public key before resolving any yields, so a single node cannot forge a signature response. [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe a pending `CKDRequest` with `AppPublicKey` variant on-chain.
2. Call `respond_ckd` with an arbitrary `CKDResponse` (any `big_c`, `big_y` of their choosing).
3. The contract accepts the response without any cryptographic check and resolves all queued yields, delivering the forged output to every caller who submitted that request.

The attacker controls the derived key material returned to the user. They can choose `big_c` and `big_y` such that they know the resulting derived key, or such that the user's derived key is unusable. This constitutes unauthorized confidential key derivation output delivered without the required threshold-participant authorization — a direct bypass of the MPC threshold guarantee for the CKD flow.

**Impact class:** Critical — unauthorized confidential key derivation output without required participant authorization; bypass of threshold-signature requirements for the CKD protocol.

---

### Likelihood Explanation

- The `AppPublicKey` variant is a first-class production code path exposed in the public `request_app_private_key` API.
- Any single attested participant can exploit this; no collusion above threshold is required.
- The attacker only needs to be an active participant with a valid TEE attestation — a realistic adversary model for a Byzantine node operator.
- The exploit requires only a standard on-chain transaction call to `respond_ckd`.

**Likelihood:** High.

---

### Recommendation

Apply the same cryptographic output validation to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. If the `AppPublicKey` variant is intentionally designed to be unverifiable on-chain (because it provides only a G1 point without a paired G2 component), then either:

1. **Remove the `AppPublicKey` variant** from the CKD API and require all callers to use `AppPublicKeyPV`, which supports the pairing-based `ckd_output_check`; or
2. **Enforce a threshold-vote mechanism** in `respond_ckd` analogous to how `respond` requires a cryptographically valid signature: collect responses from at least `threshold` participants and only resolve yields when they agree on the same output.

The current asymmetry — where `AppPublicKeyPV` is protected by `ckd_output_check` but `AppPublicKey` is not — is the root cause and must be closed. [6](#0-5) 

---

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: AppPublicKey(some_g1_point)`, attaching the minimum deposit. The contract enqueues a `CKDRequest` and yields. [7](#0-6) 

2. Attacker (a single attested participant) observes the pending request on-chain and calls:

```rust
respond_ckd(
    request,  // the observed CKDRequest
    CKDResponse {
        big_c: attacker_chosen_g1_point,
        big_y: attacker_chosen_g1_point,
    }
)
```

3. The contract executes `assert_caller_is_attested_participant_and_protocol_active` — passes, because the attacker is a valid participant.

4. The `match` on `app_public_key` hits the `AppPublicKey(_) => {}` arm — no validation occurs. [1](#0-0) 

5. `resolve_yields_for` is called with the forged `CKDResponse`, resuming all pending yields and delivering the attacker-controlled output to the user. [8](#0-7) 

The user receives a `CKDResponse` chosen entirely by the attacker, with no threshold agreement and no cryptographic binding to the network's master key — the threshold MPC guarantee is completely bypassed for this code path.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
