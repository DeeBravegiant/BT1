Audit Report

## Title
Missing Minimum-Output Slippage Protection on `removeLiquidity` with No Periphery Workaround — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.removeLiquidity` accepts only shares to burn per bin with no `minAmount0Out`/`minAmount1Out` guards. Token payouts are computed from live `token0BalanceScaled`/`token1BalanceScaled` at execution time, which change with every swap. Critically, the `msg.sender != owner` restriction prevents any periphery wrapper from adding slippage protection on the LP's behalf, leaving the LP with no on-chain revert path if bin composition shifts adversely before their transaction is mined.

## Finding Description
`LiquidityDelta` carries only `binIdxs` and `shares`; there are no minimum-output fields:

```solidity
struct LiquidityDelta {
  int256[] binIdxs;
  uint256[] shares;
}
```

`LiquidityLib.removeLiquidity` computes payouts directly from live bin state:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

`binState.token0BalanceScaled` and `binState.token1BalanceScaled` are updated by every swap that crosses the bin, so the payout ratio is live and unpredictable at submission time.

The pool enforces `msg.sender == owner` before executing the removal:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
```

This makes it impossible for any periphery contract to call `removeLiquidity` on behalf of the LP. The add-liquidity path has `MetricOmmPoolLiquidityAdder` with `maxAmountToken0`/`maxAmountToken1` caps enforced in the callback, but no equivalent `MetricOmmPoolLiquidityRemover` exists and cannot be built without removing the `msg.sender == owner` restriction. The `afterRemoveLiquidity` extension hook receives the final amounts but extensions are pool-admin-configured, not LP-configurable, and cannot serve as per-LP slippage guards.

The full call path is: `removeLiquidity` → `LiquidityLib.removeLiquidity` → reads live `binState.token0BalanceScaled`/`token1BalanceScaled` → `safeTransfer(owner, amount0Removed)` / `safeTransfer(owner, amount1Removed)` with no floor check anywhere in the path.

## Impact Explanation
An LP burns shares irreversibly and receives whatever the bins hold at execution time. If a swap (or MEV sandwich) drains token0 from the bin and replaces it with token1 between submission and mining, the LP receives a token mix worth materially less than expected. The shares are already burned; the loss is permanent and unrecoverable. This is a direct loss of LP principal with no existing on-chain guard, satisfying the Medium impact threshold.

## Likelihood Explanation
Any swap crossing the LP's bins between transaction submission and mining changes the payout. In volatile markets or during MEV sandwich attacks this window is routinely exploited. The LP has no contract-level mechanism to detect or prevent it. Likelihood is Medium: requires concurrent swap activity, which is normal for an active pool.

## Recommendation
1. **Core**: Add `uint256 minAmount0Out` and `uint256 minAmount1Out` parameters to `removeLiquidity`; revert with a `MinOutputNotMet` error if `amount0Removed < minAmount0Out || amount1Removed < minAmount1Out`.
2. **Periphery**: Create a `MetricOmmPoolLiquidityRemover` that calls `removeLiquidity` and checks the returned amounts against caller-supplied minimums, mirroring how `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` enforces `maxAmountToken0`/`maxAmountToken1`. This also requires relaxing or redesigning the `msg.sender == owner` restriction to allow trusted periphery contracts to act on behalf of owners.

## Proof of Concept
1. LP holds shares in bin 0, which currently holds 1 000 token0 and 0 token1.
2. LP submits `removeLiquidity` for 50% of bin 0 shares, expecting ≈500 token0.
3. Before the LP's tx is mined, a swap sells 1 000 token1 into the pool, buying all 1 000 token0 from bin 0. Bin 0 now holds 0 token0 and ~1 000 token1.
4. LP's tx executes: `amount0Scaled = (0 * sharesToRemove) / binTotalSharesVal = 0`; `amount1Scaled = (token1BalanceScaled * sharesToRemove) / binTotalSharesVal ≈ 500 token1`.
5. LP receives ≈500 token1 instead of ≈500 token0. If token1 is worth half of token0, the LP has lost ~$250 with no revert.
6. No parameter in `removeLiquidity` could have prevented this; the `msg.sender == owner` restriction also prevents any periphery wrapper from adding a post-call minimum check.

**Foundry test sketch:**
```solidity
function test_removeLiquidity_noSlippageProtection() public {
    // 1. LP adds liquidity to bin above current price (token0 only)
    _addLiquidity(LP, 1, 1, 1_000e18, 0);
    // 2. Swap drains all token0 from bin
    _swap(ATTACKER, recipient, true, int128(1_000e18), MAX_PRICE);
    // 3. LP removes liquidity — receives token1 instead of token0, no revert
    (uint256 out0, uint256 out1) = _removeLiquidity(LP, 1, 1, shares, 0);
    assertEq(out0, 0);   // LP expected token0, got none
    assertGt(out1, 0);   // received token1 at potentially unfavorable rate
}
