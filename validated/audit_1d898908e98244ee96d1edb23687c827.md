### Title
Sub-Transaction Value Conservation Not Enforced for Native Token Minting — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era introduces nested transactions (`TopTx`/`SubTx`). Sub-transactions support a `mint` field (`dstbrMint :: !MultiAsset`) identical to top-level transactions, and minting policy scripts are validated for sub-transactions via `SUBUTXOW`. However, the `SUBUTXO` rule never calls `validateValueNotConservedUTxO` for sub-transactions, while the top-level `UTXO` rule does so explicitly. The top-level value conservation check also only inspects the `TopTx` body, not any sub-transaction bodies. An attacker who controls a minting policy script can craft a sub-transaction whose outputs contain more native-asset value than `inputs + mint` specifies, creating native assets from nothing.

---

### Finding Description

The `DijkstraSubTxBodyRaw` data type includes a `mint` field for sub-transactions: [1](#0-0) 

The top-level `dijkstraUtxoTransition` explicitly validates value conservation before updating the UTxO state: [2](#0-1) 

That check uses only the `TopTx` body (`txBody`), not any sub-transaction bodies: [3](#0-2) 

The sub-transaction rule `dijkstraSubUtxoTransition` performs no equivalent check. It validates structural properties (validity interval, input set, output size, network IDs) and then directly calls `Shelley.updateUTxOStateNoFees` to apply the UTxO delta: [4](#0-3) 

There is no `validateValueNotConservedUTxO` call anywhere in `dijkstraSubUtxoTransition`. The `DijkstraSubUtxoPredFailure` sum type has no constructor corresponding to `ValueNotConservedUTxO`, and the mapping function `dijkstraUtxoToDijkstraSubUtxoPredFailure` explicitly marks it as impossible: [5](#0-4) 

The `SUBUTXOW` rule does validate minting policy scripts (via `Babbage.validateFailedBabbageScripts` and `Babbage.babbageMissingScripts` using `scriptsNeeded` which includes `getMintingScriptsNeeded`): [6](#0-5) 

Script validation ensures the minting policy authorises the `mint` field value. However, it does not enforce that the outputs of the sub-transaction are consistent with `inputs + mint`. Without `validateValueNotConservedUTxO`, a sub-transaction may specify outputs whose total native-asset value exceeds `inputs + mint`, creating tokens from nothing, or specify outputs whose value is less than `inputs + mint`, silently destroying tokens.

The pattern in the codebase confirms that `updateUTxOStateNoFees` does not itself check value conservation — the top-level rule calls `validateValueNotConservedUTxO` as a separate, explicit step before calling the update function, making the check non-redundant and therefore not present inside the update function.

---

### Impact Explanation

**Critical — Direct creation or destruction of native assets through an invalid ledger state transition.**

An attacker who controls any minting policy (e.g., a native timelock script or a Plutus script they authored) can:

1. Place a sub-transaction with a valid `mint` field of, say, 100 units of token `T`.
2. Specify outputs totalling 1,000,000 units of `T`.
3. Because `SUBUTXO` never calls `validateValueNotConservedUTxO`, the ledger accepts the sub-transaction and the UTxO is updated with 1,000,000 units of `T` — 999,900 units created from nothing.

This constitutes an invalid ledger state transition that inflates the supply of any native asset the attacker's minting policy governs.

---

### Likelihood Explanation

Any transaction author who can include a sub-transaction with a minting policy they control (a low barrier — native scripts require only a key signature) can trigger this. The Dijkstra era is explicitly designed to allow sub-transactions to mint tokens (`dstbrMint` field exists and is serialised). The attack requires no privileged role, no governance majority, and no external dependency.

---

### Recommendation

Add an explicit value conservation check inside `dijkstraSubUtxoTransition`, analogous to the check in `dijkstraUtxoTransition`:

```haskell
-- In dijkstraSubUtxoTransition, before calling updateUTxOStateNoFees:
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
```

A corresponding `SubValueNotConservedUTxO` constructor should be added to `DijkstraSubUtxoPredFailure`, and the `dijkstraUtxoToDijkstraSubUtxoPredFailure` mapping updated accordingly (removing the `error "Impossible"` branch for `ValueNotConservedUTxO`). [7](#0-6) 

---

### Proof of Concept

1. Construct a `TopTx` containing one `SubTx`.
2. The `SubTx` body has:
   - `dstbrSpendInputs`: one UTxO entry worth 2 ADA.
   - `dstbrMint`: `{policyId → {assetName → 1}}` (mint exactly 1 token).
   - `dstbrOutputs`: one output containing 2 ADA **and 1,000,000 tokens** under `policyId`.
   - A valid witness for `policyId` (e.g., a native `RequireSignature` script signed by the attacker's key).
3. Submit the `TopTx`.
4. `SUBUTXOW` validates the minting script — it passes.
5. `SUBUTXO` calls `updateUTxOStateNoFees` — the UTxO is updated with the output containing 1,000,000 tokens.
6. No `validateValueNotConservedUTxO` is called for the sub-transaction; the 999,999 extra tokens are accepted into the ledger state. [8](#0-7) [9](#0-8)

dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG

  let txBody = tx ^. bodyTxL

  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo
  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  let allSizedOutputs = txBody ^. allSizedOutputsTxBodyF
  let allOutputs = fmap sizedValue allSizedOutputs
  runTest $ Alonzo.validateOutputTooBigUTxO pp allOutputs

  runTest $ Shelley.validateInputSetEmptyUTxO txBody

  let inputs = txBody ^. inputsTxBodyL
  let refInputs = txBody ^. referenceInputsTxBodyL
  runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs

  runTestOnSignal $ Shelley.validateOutputBootAddrAttrsTooBig allOutputs

  runTestOnSignal $ Babbage.validateOutputTooSmallUTxO pp allSizedOutputs

  netId <- liftSTS $ asks networkId
  runTestOnSignal $ Shelley.validateWrongNetwork netId allOutputs
  runTestOnSignal $ Shelley.validateWrongNetworkWithdrawal netId txBody
  runTestOnSignal $ validateWrongNetworkInDirectDeposit netId txBody
  runTestOnSignal $ Alonzo.validateWrongNetworkInTxBody netId txBody

  if isValid
    then do
      newState <-
        Shelley.updateUTxOStateNoFees
          pp
          utxoState
          txBody
          certState
          (utxosGovState utxoState)
          (tellEvent . TotalDeposits (hashAnnotated txBody))
          (\a b -> tellEvent $ TxUTxODiff a b)
      pure $ newState & utxosDonationL <>~ txBody ^. treasuryDonationTxBodyL
    else
      pure utxoState
