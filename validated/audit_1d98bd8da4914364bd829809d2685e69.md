Audit Report

## Title
Withdrawal Queue Permanently Stalls After EigenLayer Slashing Due to `assetsCommitted` Desync - (`contracts/LRTWithdrawalManager.sol`)

## Summary

`initiateWithdrawal` commits `expectedAssetAmount` into `assetsCommitted[asset]` based on `getTotalAssetDeposits`, which includes EigenLayer-held assets explicitly subject to slashing. When EigenLayer slashing reduces the actual amount received into the vault, `_unlockWithdrawalRequests` breaks on the first under-funded request and never decrements `assetsCommitted`, stalling the entire FIFO queue and blocking all subsequent user withdrawals as well as new `initiateWithdrawal` calls.

## Finding Description

**Step 1 — Commitment inflated by EigenLayer assets**

`initiateWithdrawal` computes `expectedAssetAmount` and stores it:

```solidity
// LRTWithdrawalManager.sol:168-173
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` calls `getTotalAssetDeposits`, which aggregates:

```solidity
// LRTDepositPool.sol:394-396
uint256 effectiveAssetWithEigenLayer = assetStakedInEigenLayer + assetUnstakingFromEigenLayer;
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer + ...);
```

These EigenLayer-held assets are explicitly subject to slashing per the integrated interface:

```solidity
// IDelegationManager.sol:314-316
// Withdrawals are still subject to slashing during the delay period so the amount withdrawn
// on completion may actually be less than what was queued if slashing has occurred in that period.
```

**Step 2 — Vault receives less than committed**

`NodeDelegator.completeUnstaking` transfers only the actual post-slash balance to the vault:

```solidity
// NodeDelegator.sol:392-394
assets[i].safeTransfer(
    address(_getUnstakingVault()), assets[i].balanceOf(address(this)) - balancesBefore[i]
);
```

**Step 3 — Queue stalls; `assetsCommitted` never decremented**

`_createUnlockParams` uses the actual vault balance:

```solidity
// LRTWithdrawalManager.sol:849
totalAvailableAssets: unstakingVault.balanceOf(asset)  // post-slash, less than committed
```

`_unlockWithdrawalRequests` breaks on the first request it cannot cover:

```solidity
// LRTWithdrawalManager.sol:800
if (availableAssetAmount < payoutAmount) break;
```

The decrement `assetsCommitted[asset] -= request.expectedAssetAmount` at L802 is only reached after the break check passes. When the break fires, `assetsCommitted` is never reduced, `nextLockedNonce[asset]` is not advanced, and the queue is permanently stalled until the vault is manually topped up. Because `assetsCommitted` remains inflated, `getAvailableAssetAmount` returns an artificially low value, also blocking new `initiateWithdrawal` calls.

## Impact Explanation

**Temporary freezing of funds (Medium).** Users who called `initiateWithdrawal` already had their rsETH transferred to the contract (`safeTransferFrom` at L166). They cannot retrieve their assets until the protocol manually replenishes the vault. All requests behind the stalled nonce are also blocked. New users cannot initiate withdrawals because `getAvailableAssetAmount` remains deflated by the uncommitted `assetsCommitted` balance. The freeze persists until an operator intervention tops up the vault.

## Likelihood Explanation

EigenLayer v2 slashing is an intended, documented protocol feature already integrated into this codebase (`SlashingLib` is imported in `LRTUnstakingVault` and `NodeDelegatorHelper`; the `IDelegationManager` interface explicitly documents slashing during the withdrawal delay). No attacker action is required — any operator slash affecting a node delegator's queued withdrawal during the EigenLayer delay period triggers this condition automatically. The condition is reachable through normal user interactions (`initiateWithdrawal` is a public function) combined with a normal EigenLayer protocol event.

## Recommendation

In `_unlockWithdrawalRequests`, replace the `break` on insufficient balance with logic that handles the shortfall without stalling the queue:

1. **Decouple `assetsCommitted` from EigenLayer assets** — base `getAvailableAssetAmount` only on assets already in the vault (`unstakingVault.balanceOf(asset)`), not on EigenLayer-held assets still subject to slashing.
2. **Cap payout to available balance** — when `availableAssetAmount < payoutAmount`, disburse only what the vault holds for that request rather than breaking, and decrement `assetsCommitted` accordingly.
3. **Skip and continue** — mark under-funded requests with a flag and continue processing subsequent requests that can be fulfilled, preventing a single shortfall from blocking the entire queue.

## Proof of Concept

1. Protocol has 1000 stETH in EigenLayer via `NodeDelegator`. Oracle prices rsETH 1:1 with stETH.
2. User calls `initiateWithdrawal(stETH, 100_rsETH)`. `getAvailableAssetAmount` returns 1000 (EigenLayer assets counted). `assetsCommitted[stETH] = 100`. User's rsETH is transferred to the contract.
3. Operator calls `initiateUnstaking` to queue 100 stETH withdrawal from EigenLayer.
4. EigenLayer operator is slashed 20% during the delay period. `completeUnstaking` receives only 80 stETH into the vault.
5. Operator calls `unlockQueue(stETH, ...)`. `_createUnlockParams` returns `totalAvailableAssets = 80`.
6. `_unlockWithdrawalRequests` computes `payoutAmount ≈ 100`, finds `80 < 100`, and **breaks** — `nextLockedNonce[stETH]` is not advanced, `assetsCommitted[stETH]` remains 100.
7. User's withdrawal and all subsequent withdrawals are stalled. `getAvailableAssetAmount` = `totalAssets - 100`, blocking new `initiateWithdrawal` calls.
8. Funds remain frozen until the protocol manually sends ≥20 stETH to the vault.
