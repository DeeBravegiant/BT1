Audit Report

## Title
Missing Upper Bound on `priceProviderTimelock` in `createPool` Permanently Locks Pool Into Broken Price Provider - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`MetricOmmPoolFactory.createPool` stores `params.priceProviderTimelock` verbatim with no upper-bound validation beyond the exact sentinel `type(uint256).max`. A pool creator who supplies a near-maximal value (e.g., `type(uint256).max - 1`) produces a pool that is not flagged immutable yet whose `proposePoolPriceProvider` call overflows and reverts permanently. If the underlying price provider later stalls or is deprecated, every swap reverts with `PriceProviderFailed` and there is no on-chain recovery path.

## Finding Description
`_validatePoolParameters` validates tokens, admin, fees, and initial amounts but performs no check on `params.priceProviderTimelock`:

```solidity
// MetricOmmPoolFactory.sol L548-563
function _validatePoolParameters(PoolParameters calldata params) internal view {
    if (params.token0 == address(0) || ...) revert InvalidTokenConfig();
    if (params.admin == address(0)) revert InvalidAdmin();
    _validatePriceProvider(...);
    if (params.adminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    if (spreadProtocolFeeE6 > maxProtocolSpreadFeeE6) revert ProtocolFeeTooHigh();
    // ... fee checks ...
    if (params.minimalMintableLiquidity == 0) revert InvalidMinimalMintableLiquidity();
    // NO check on params.priceProviderTimelock
}
```

The only special handling is the exact sentinel at L164:
```solidity
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
```

Any other value is stored verbatim at L213:
```solidity
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

When the pool admin later calls `proposePoolPriceProvider`, L487 performs unchecked addition:
```solidity
uint256 executeAfter = block.timestamp + timelock;
```

With `timelock = type(uint256).max - 1`, this overflows in Solidity 0.8.x and reverts permanently. Even without overflow (e.g., `timelock = type(uint256).max - block.timestamp`), `executeAfter` becomes `type(uint256).max`, and `executePoolPriceProviderUpdate` at L499 always reverts:
```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
```

Once the price provider stalls, `_getBidAndAskPriceX64` (L804-813) catches the revert and re-throws `PriceProviderFailed`, making every swap permanently revert. The pool is not flagged `immutablePriceProvider`, so neither the factory nor off-chain tooling treats it as immutable — yet the update path is permanently bricked.

## Impact Explanation
Swaps are permanently broken with no recovery path once the price provider stalls. The pool is in a false-immutable state: not flagged immutable, yet the update mechanism is irrecoverably bricked. LPs can still call `removeLiquidity` to recover principal, so direct fund loss is avoided, but the pool's primary function — swap execution — is permanently destroyed. This matches the allowed impact: **broken core pool functionality causing unusable swap flows**.

## Likelihood Explanation
Requires the pool creator to supply a near-maximal `priceProviderTimelock` (e.g., `type(uint256).max - 1` instead of `type(uint256).max`, or appending extra digits to a large duration). The pool creator is a public, unprivileged actor. The risk is compounded because there is no on-chain signal distinguishing a legitimately large timelock from a misconfigured one, and the pool operates normally until the price provider stalls.

## Recommendation
Add an upper-bound check inside `_validatePoolParameters`, excluding the immutable sentinel, mirroring the existing fee-cap pattern (`HARD_MAX_SPREAD_FEE_E6`, `HARD_MAX_NOTIONAL_FEE_E8`):

```solidity
uint256 MAX_TIMELOCK = 365 days;
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock > MAX_TIMELOCK
) {
    revert PriceProviderTimelockTooLarge();
}
```

## Proof of Concept
1. Call `createPool` with `params.priceProviderTimelock = type(uint256).max - 1`.
2. `_validatePoolParameters` passes — no check on `priceProviderTimelock`.
3. `immutablePriceProvider = (type(uint256).max - 1 == type(uint256).max)` → `false`. Pool deployed as mutable.
4. `priceProviderTimelock[pool] = type(uint256).max - 1` stored.
5. LPs add liquidity; pool operates normally while price provider is live.
6. Price provider is deprecated or stalls.
7. Every `swap` call hits `_getBidAndAskPriceX64` → `PriceProviderFailed`. Swaps dead.
8. Pool admin calls `proposePoolPriceProvider` with a valid replacement:
   - `uint256 executeAfter = block.timestamp + (type(uint256).max - 1)` → **overflow revert** in Solidity 0.8.x.
9. No recovery path exists. Pool swap functionality is permanently bricked.
