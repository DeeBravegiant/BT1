Audit Report

## Title
Excess native ETH stranded in router is claimable by any subsequent caller - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`MetricOmmSimpleRouter` swap functions are `payable` and accept native ETH to be wrapped as WETH for input payment. The `pay()` helper in `PeripheryPayments` wraps only the exact amount the pool requests via callback, leaving any excess ETH in the router. Because `pay()` draws from `address(this).balance` unconditionally when the token is WETH, a subsequent caller can craft a swap with `msg.value = 0` and have their input settled entirely from the prior caller's stranded ETH, stealing it.

## Finding Description

**Root cause — `PeripheryPayments.pay()` (lines 69–88):**

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

When `nativeBalance >= value`, the function wraps exactly `value` ETH from the contract's balance and sends it to the pool. It does **not** check that the ETH came from the current caller, and it does **not** refund the remainder.

**Stranding path:**

1. Victim calls `exactInputSingle{value: X}(params)` where `params.tokenIn = WETH` and `params.amountIn = X`.
2. The pool's swap hits a `priceLimitX64` before consuming all of `X`, so the callback delta is `Y < X`.
3. `_justPayCallback` calls `pay(WETH, victim, pool, Y)`. The router wraps `Y` ETH and sends it; `X − Y` ETH remains in the router.
4. Alternatively, the victim simply sends `msg.value > params.amountIn`; the callback pays exactly `params.amountIn` and the surplus is stranded.

**Theft path:**

5. Attacker calls `exactInputSingle{value: 0}(params2)` where `params2.tokenIn = WETH`, `params2.amountIn = X − Y`.
6. In the callback, `pay(WETH, attacker, pool, X − Y)` is called. `address(this).balance = X − Y ≥ value`, so the branch wraps the victim's stranded ETH and sends it to the pool.
7. Attacker receives the swap output without paying anything.

**Why existing guards fail:**

- `refundETH()` exists but is never called automatically; it requires the user to include it explicitly in a `multicall`. A single-function call to `exactInputSingle` provides no refund.
- The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only restricts the `receive` fallback; it does not prevent ETH from entering via `payable` swap functions.
- `IncorrectDelta` in the pool only checks that the pool received at least the owed amount; it does not prevent the router from over-receiving ETH.
- There is no `msg.value == amountIn` assertion anywhere in the router entry points.

## Impact Explanation
Direct loss of user principal: any ETH sent to the router in excess of the exact swap settlement amount is permanently claimable by the next caller. This is a swap conservation failure from the user's perspective — the user pays more than the pool receives on their behalf, and the difference is stolen. Severity: **High**.

## Likelihood Explanation
Partial fills occur whenever `priceLimitX64` is hit before the full `amountIn` is consumed, which is a normal operating condition. Users may also routinely send a round ETH value slightly above the required amount. The attacker requires no special privilege, no setup, and can monitor the mempool for transactions that leave ETH in the router, then front-run or follow-run with a zero-value WETH swap. The attack is repeatable and permissionless.

## Recommendation
After each swap settles, refund any remaining native ETH balance to `msg.sender` inside the swap entry points (or enforce `msg.value == amountIn` when `tokenIn == WETH`). Concretely, add at the end of `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`:

```solidity
if (address(this).balance > 0) {
    _transferETH(msg.sender, address(this).balance);
}
```

Alternatively, assert `msg.value == params.amountIn` (or `<= amountIn` for exact-output) when `params.tokenIn == WETH` and revert otherwise, preventing excess ETH from entering the contract at all.

## Proof of Concept

```solidity
// 1. Victim sends 2 ETH but pool only consumes 1 ETH (price limit hit)
router.exactInputSingle{value: 2 ether}(ExactInputSingleParams({
    pool: pool,
    tokenIn: WETH,
    recipient: victim,
    amountIn: 2 ether,
    amountOutMinimum: 0,
    priceLimitX64: tightLimit,   // causes partial fill; pool takes 1 ETH
    zeroForOne: true,
    deadline: block.timestamp,
    extensionData: ""
}));
// router.balance == 1 ether (stranded)

// 2. Attacker sends 0 ETH, claims victim's stranded ETH as swap input
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    pool: pool,
    tokenIn: WETH,
    recipient: attacker,
    amountIn: 1 ether,
    amountOutMinimum: 0,
    priceLimitX64: openLimit,
    zeroForOne: true,
    deadline: block.timestamp,
    extensionData: ""
}));
// attacker receives swap output; paid 0 ETH; victim's 1 ETH consumed
```
