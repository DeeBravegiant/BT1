Audit Report

## Title
Missing Staleness Validation on Chainlink `latestRoundData()` Enables Stale-Price Deposit Exploitation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` and discards every staleness-related return value (`updatedAt`, `roundId`, `answeredInRound`), returning the raw price unconditionally. Because this price feeds directly into `LRTDepositPool.getRsETHAmountToMint`, any depositor transacting while a Chainlink feed is stale receives an rsETH amount computed from an outdated price, diluting existing rsETH holders' proportional share of protocol TVL.

## Finding Description

`contracts/oracles/ChainlinkPriceOracle.sol` line 52 captures only the `price` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

No check of the form `block.timestamp - updatedAt <= heartbeat` exists, nor any `require(price > 0)`, nor any `answeredInRound >= roundId` guard. The raw price is returned unconditionally at line 54.

`LRTOracle.getAssetPrice` (line 157) delegates directly to this oracle:

```solidity
return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
```

This price is consumed in two critical paths:

1. **Deposit minting** — `LRTDepositPool.getRsETHAmountToMint` (line 520):
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
   An inflated stale asset price inflates `rsethAmountToMint` for the depositor.

2. **rsETH price update** — `LRTOracle._getTotalEthInProtocol` (lines 336–343) multiplies each asset balance by `getAssetPrice(asset)` to compute total protocol ETH, which then sets `rsETHPrice`.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` is not a mitigant here: it only fires when `updateRSETHPrice()` is explicitly called, and it compares the computed new price against `highestRsethPrice` — it does not prevent individual deposits from using a stale per-asset price. A depositor can call `depositAsset` at any time without triggering a price update.

`ChainlinkOracleForRSETHPoolCollateral` does include two checks (`answeredInRound < roundID` and `timestamp == 0`), but this contract is not the oracle backing `LRTOracle.getAssetPrice`; it is a separate pool collateral wrapper. Its checks are also insufficient (the `answeredInRound` pattern is deprecated and unreliable; `timestamp == 0` is not a heartbeat check). The primary attack surface — `ChainlinkPriceOracle` — has no staleness validation at all.

## Impact Explanation

**High — Theft of unclaimed yield.**

If a Chainlink feed goes stale with an inflated last-reported price, every depositor during the stale window receives more rsETH than their deposit is worth. This over-minting dilutes all existing rsETH holders' proportional claim on protocol TVL, constituting theft of their unclaimed yield. The magnitude scales with the staleness duration, the price deviation, and the deposit volume during the window.

## Likelihood Explanation

Chainlink feeds have documented heartbeat windows (e.g., 1 hour for ETH/USD, up to 24 hours for some LST feeds). During low-volatility periods, feeds may not update for the full heartbeat duration without any anomaly. Network congestion or sequencer downtime can extend this further. No special privileges, front-running, or oracle manipulation is required — any depositor transacting during a naturally occurring stale window triggers the issue. The condition is repeatable across any supported LST asset that uses `ChainlinkPriceOracle`.

## Recommendation

Add a per-asset heartbeat mapping and staleness check in `ChainlinkPriceOracle.getAssetPrice`:

```solidity
mapping(address asset => uint256 heartbeat) public assetHeartbeat;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    require(price > 0, "Invalid price");
    require(answeredInRound >= roundId, "Stale round");
    require(block.timestamp - updatedAt <= assetHeartbeat[asset], "Stale price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Require `assetHeartbeat[asset]` to be set (non-zero) alongside `updatePriceFeedFor`. Replace the `timestamp == 0` check in `ChainlinkOracleForRSETHPoolCollateral` with a proper heartbeat check as well.

## Proof of Concept

1. Deploy a fork of mainnet. Chainlink's stETH/ETH feed last updated 2+ hours ago at price `1.05e18` (true current price: `1.00e18`). No `updateRSETHPrice()` call has been made since.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice` returns `1.05e18` (stale).
4. `rsethAmountToMint = (100e18 * 1.05e18) / rsETHPrice` → attacker receives ~5% more rsETH than fair value.
5. Existing rsETH holders' proportional share of TVL is diluted by the over-minted rsETH.
6. No admin action, no special role, no oracle manipulation required — any depositor during the stale window benefits.

**Foundry fork test outline:**
```solidity
