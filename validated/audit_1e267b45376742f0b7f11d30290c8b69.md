Audit Report

## Title
`LiquidityLib.addLiquidity` unchecked `binTotalShares` overflow corrupts bin share accounting, permanently breaking `removeLiquidity` and enabling future-LP fund theft — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
The entire body of `LiquidityLib.addLiquidity` runs inside a single `unchecked {}` block. When a bin's token balances are both zero (a normal post-swap state), `_checkedMul(0, sharesToAdd)` returns 0 without reverting for any value of `sharesToAdd`, allowing an attacker to supply `sharesToAdd = type(uint256).max - binTotalSharesVal + 1` at zero token cost. The unchecked addition `binTotalSharesVal + sharesToAdd` wraps to zero, permanently breaking `removeLiquidity` for all existing LPs in that bin via division-by-zero, and enabling a subsequent attacker withdrawal that drains any future LP who re-enters the bin.

## Finding Description
`addLiquidity` wraps its entire loop in `unchecked` (line 51). Inside the loop, when `binTotalSharesVal > 0`, token amounts are computed via `_checkedMul(binState.token0BalanceScaled, sharesToAdd)` and `_checkedMul(binState.token1BalanceScaled, sharesToAdd)` (lines 109–110). `_checkedMul` is `internal pure` with body `return a * b;` (lines 261–263) — it retains Solidity 0.8 overflow protection, but `0 * b == 0` never overflows regardless of `b`. After a swap fully drains a bin, both `token0BalanceScaled` and `token1BalanceScaled` become 0 while `binTotalShares[binIdx]` remains the original LP's share count (the swap loop modifies bin balances but never touches `binTotalShares`). An attacker targeting such a bin passes `sharesToAdd = type(uint256).max - S + 1` where `S = binTotalShares[binIdx]`. Both `_checkedMul` calls return 0, so `amount0Scaled = amount1Scaled = 0` — the attacker pays nothing. The unchecked write `binTotalShares[binIdx] = S + (type(uint256).max - S + 1)` wraps to 0. The attacker's `positionBinShares` is set to the huge value `type(uint256).max - S + 1`. Subsequently, any call to `removeLiquidity` for that bin reaches `/ binTotalSharesVal` (lines 205–206) with `binTotalSharesVal == 0`, causing a division-by-zero panic that permanently reverts for every LP in the bin. If a new LP later re-enters the bin, `binTotalSharesVal == 0` triggers the initial-rate branch (line 85), the new LP pays real tokens, and the attacker's stale huge `positionBinShares` allows withdrawing more than the new LP deposited; the resulting `binState.token0BalanceScaled -= uint104(amount0Scaled)` underflow (inside `unchecked`, line 210) further corrupts `binTotals`.

## Impact Explanation
- **Broken core functionality:** `removeLiquidity` permanently reverts (division-by-zero panic) for all positions in any affected bin, locking LP principal.
- **Direct fund loss:** Future LPs who add liquidity to a corrupted bin have their deposit drained by the attacker's stale `positionBinShares`, with the pool's `binTotals` corrupted by the resulting unchecked underflow. Both impacts meet Sherlock Critical/High thresholds for loss of user principal and broken core pool functionality.

## Likelihood Explanation
Bins being fully consumed by swaps is a routine, expected event in any active pool. The attack costs the attacker zero tokens. The only required input — `binTotalShares[binIdx]` — is publicly readable via `PoolStateLibrary`/`EXTSLOAD`. No privileged role is required; `addLiquidity` is a public entry point callable by any address. The attack is repeatable across any bin that reaches the zero-balance state.

## Recommendation
Remove the blanket `unchecked {}` wrapper from `addLiquidity`, or add explicit upper-bound overflow checks before the share writes:

```solidity
uint256 newBinTotal = binTotalSharesVal + sharesToAdd;
if (newBinTotal < binTotalSharesVal) revert SharesOverflow();
binTotalShares[binIdx] = newBinTotal;

uint256 newUserSharesChecked = userShares + sharesToAdd;
if (newUserSharesChecked < userShares) revert SharesOverflow();
positionBinShares[posKey] = newUserSharesChecked;
```

Apply the same guard to `totalToken0ToAddScaled` and `totalToken1ToAddScaled` accumulations. Alternatively, move all share-accounting writes outside the `unchecked` block so Solidity's default overflow protection applies.

## Proof of Concept
1. Deploy pool; LP adds liquidity to bin `B` (below current price, token1 only). `binTotalShares[B] = S`, `token1BalanceScaled = T > 0`.
2. Execute swap(s) that fully drain bin `B`: `token1BalanceScaled → 0`. Verify `binTotalShares[B] == S` still.
3. Attacker calls `addLiquidity` for bin `B` with `sharesToAdd = type(uint256).max - S + 1`. Both `_checkedMul(0, sharesToAdd)` calls return 0; callback receives `amount0Added = amount1Added = 0`; attacker pays nothing.
4. Assert `binTotalShares[B] == 0` and `positionBinShares[attackerKey] == type(uint256).max - S + 1`.
5. LP calls `removeLiquidity` for bin `B` with their original `S` shares → transaction reverts with division-by-zero panic (Foundry: `vm.expectRevert()`).
6. New LP adds liquidity to bin `B` (triggers initial-rate branch, pays real tokens, `binTotalShares[B] = newShares`). Attacker calls `removeLiquidity` with shares `> newShares` → drains new LP's deposit; `binState.token0BalanceScaled` underflows silently inside `unchecked`.
