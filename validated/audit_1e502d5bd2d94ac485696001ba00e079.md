Audit Report

## Title
ETH Withdrawal Permanently Frozen When Recipient Contract Cannot Receive ETH - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager._transferAsset` dispatches native ETH via a low-level `.call{value:}` and reverts on failure. Because rsETH is burned and ETH is pulled from the unstaking vault in a prior `unlockQueue` transaction, a smart-contract withdrawer whose `receive()` is absent or reverting will permanently trap the ETH inside `LRTWithdrawalManager` with no admin rescue path available.

## Finding Description

**Root cause — `_transferAsset` reverts on failed ETH push:** [1](#0-0) 

**State mutation ordering in `_processWithdrawalCompletion`:**

Lines 705, 712, and 717 mutate state (pop nonce, delete request, decrement `unlockedWithdrawalsCount`) before the `_transferAsset` call at line 734. [2](#0-1) [3](#0-2) 

If `_transferAsset` reverts, the entire transaction reverts and all state is restored — the withdrawal request remains in the queue indefinitely.

**rsETH is burned in a prior, separate transaction (`unlockQueue`):** [4](#0-3) 

After `unlockQueue` executes, the user's rsETH is permanently burned and the ETH equivalent sits inside `LRTWithdrawalManager`. No subsequent revert in `completeWithdrawal` can un-burn the rsETH.

**`completeWithdrawalForUser` does not help** — it calls the same `_processWithdrawalCompletion` with the same `user` address as the ETH destination, so it reverts identically: [5](#0-4) 

The dev comment at line 191 ("Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH") explicitly acknowledges ETH issues but incorrectly dismisses them as non-impactful.

**`sweepRemainingAssets` is permanently blocked** while any unlocked withdrawal exists: [6](#0-5) [7](#0-6) 

Because `unlockedWithdrawalsCount[asset]` is never decremented (every `completeWithdrawal` attempt reverts), `hasUnlockedWithdrawals` returns `true` forever, blocking the sweep for the entire ETH asset queue. No other admin function exists to forcibly redirect or cancel a pending ETH withdrawal.

## Impact Explanation

**Critical — Permanent freezing of funds.**

The user's rsETH is irreversibly burned in `unlockQueue`. The ETH equivalent is locked inside `LRTWithdrawalManager` with no reachable code path to deliver it to an alternative address, cancel the request, or return it to the protocol. The funds are permanently inaccessible to both the user and the protocol.

## Likelihood Explanation

**Medium.** Smart-contract callers are common in DeFi: Gnosis Safe multisigs, DAO vaults, yield aggregators, and wrapper contracts routinely hold rsETH and initiate withdrawals. Many such contracts intentionally lack a `receive()` function or have one that reverts to prevent accidental ETH acceptance. A single such caller permanently bricks their own withdrawal and, via the `sweepRemainingAssets` guard, also blocks treasury sweeps for the entire ETH asset queue.

## Recommendation

1. **Pull-over-push pattern**: If the ETH transfer fails, do not revert. Instead, credit the amount to a `claimable[user]` mapping and expose a separate `claimETH()` function so the user can pull funds to any address they control.
2. **Configurable recipient**: Allow the user to specify an alternative ETH recipient at `completeWithdrawal` time, or store a user-configurable `ethRecipient` at `initiateWithdrawal` time.
3. **Decouple rsETH burn from delivery**: Do not burn rsETH in `unlockQueue` until ETH has been successfully delivered, or implement a refund path that re-mints rsETH if delivery permanently fails.

## Proof of Concept

1. Smart-contract wallet `VaultA` (no `receive()`) calls `initiateWithdrawal(ETH_TOKEN, 1e18, "ref")`. rsETH is transferred to `LRTWithdrawalManager`. [8](#0-7) 

2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned; ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. [4](#0-3) 

3. `VaultA` calls `completeWithdrawal(ETH_TOKEN, "ref")`. `_transferAsset` executes `payable(VaultA).call{value: amount}("")` → `sent == false` → `revert EthTransferFailed`. Transaction reverts; state fully restored. [9](#0-8) 

4. Operator calls `completeWithdrawalForUser(ETH_TOKEN, VaultA, "ref")` — same ETH destination, same revert. [10](#0-9) 

5. `unlockedWithdrawalsCount[ETH_TOKEN] > 0` forever → `sweepRemainingAssets` reverts for all callers. [6](#0-5) 

`VaultA`'s rsETH is gone; the ETH is permanently locked in `LRTWithdrawalManager` with no recovery path.

**Foundry test plan**: Deploy a mock contract with no `receive()`. Call `initiateWithdrawal`, simulate `unlockQueue` (burn rsETH, transfer ETH to manager), then call `completeWithdrawal` from the mock contract and assert it reverts with `EthTransferFailed`. Assert `unlockedWithdrawalsCount[ETH_TOKEN] == 1` after the failed attempt. Assert `sweepRemainingAssets` reverts with `PendingWithdrawalsExist`.
