### Title
Unrestricted `clean_invalid_attestations` Allows Any Caller to Prune Participant Attestations During TEE Upgrade Windows - (File: `crates/contract/src/lib.rs`)

---

### Summary

`MpcContract::clean_invalid_attestations` is explicitly documented as "Callable by anyone" and carries no access-control check. During a TEE upgrade window — when participants' attestations have expired under the old image hash but before they have submitted new ones — any unprivileged account can call this function to remove those attestations from `stored_attestations`. Once removed, the affected participants fail `is_caller_an_attested_participant`, blocking them from all threshold-gated operations (signing responses, key-event votes, governance). A subsequent call to `verify_tee` by any participant then treats the missing attestations as invalid TEE status, which can set `accept_requests = false` or trigger an unwanted resharing with a reduced participant set.

---

### Finding Description

Every other state-mutating method in the contract is gated by one of three guards:

- `voter_or_panic()` — requires `signer == predecessor` AND membership in the current participant set
- `assert_caller_is_signer()` — requires `signer == predecessor`
- `assert_caller_is_attested_participant_and_protocol_active()` — requires attested participant membership

`clean_invalid_attestations` has **none of these**:

```rust
/// Prunes up to `max_scan` stored attestations that fail re-verification (expired or
/// referencing stale whitelists). Returns the number of entries removed. Callable by
/// anyone while the protocol is in `Running`.
#[handle_result]
pub fn clean_invalid_attestations(&mut self, max_scan: u32) -> Result<u32, Error> {
    // Running-only: keygen / resharing may reference attestations that have not yet
    // been activated, so cleanup is off-limits during those phases.
    if !matches!(self.protocol_state, ProtocolContractState::Running(_)) {
        return Err(InvalidState::ProtocolStateNotRunning.into());
    }
    let tee_upgrade_deadline_duration =
        Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);
    Ok(self
        .tee_state
        .clean_invalid_attestations(tee_upgrade_deadline_duration, max_scan as usize))
}

If an entry is removed from `stored_attestations`, the participant fails this check with `AttestationCheckError::AttestationNotFound`, blocking `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`, `respond`, `respond_ckd`, and `respond_verify_foreign_tx`.

Additionally, `verify_tee` (callable by any participant) calls `reverify_and_cleanup_participants`, which also reads `stored_attestations`. With entries missing, it treats those participants as having no valid TEE status, which can set `self.accept_requests = false` or trigger an automatic resharing with a reduced participant set: [4](#0-3) 

---

### Impact Explanation

**Medium.** An attacker can manipulate participant-state and contract execution-flow in a way that breaks production safety invariants:

1. Affected participants are locked out of all threshold-gated operations until they resubmit attestations via `submit_participant_info`.
2. If the number of locked-out participants drops the effective attested set below the signing threshold, `accept_requests` is set to `false`, halting all user-facing signing and CKD requests.
3. An automatic resharing can be triggered with a reduced participant set, disrupting the key-management lifecycle.

This matches the allowed Medium impact: *"participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

**Medium.** The attack window opens during every TEE upgrade cycle. When a new docker image hash is voted in and the `tee_upgrade_deadline_duration` for the old hash elapses, nodes still running the old image have attestations that fail `reverify_participants`. An attacker monitoring the chain can detect this window and call `clean_invalid_attestations(max_scan: u32::MAX)` before the affected nodes submit new attestations. TEE upgrades are a routine operational event, so this window recurs predictably.

---

### Recommendation

Apply the same access-control pattern used by `remove_non_participant_update_votes`: allow only self-calls (spawned by the contract itself after resharing) or direct calls from a current participant. Reject all other callers:

```rust
pub fn clean_invalid_attestations(&mut self, max_scan: u32) -> Result<u32, Error> {
    // Restrict to self-calls or current participants
    let caller = env::predecessor_account_id();
    let is_self_call = caller == env::current_account_id();
    if !is_self_call {
        let participants = match &self.protocol_state {
            ProtocolContractState::Running(state) => state.parameters.participants(),
            _ => return Err(InvalidState::ProtocolStateNotRunning.into()),
        };
        if !participants.is_participant_given_account_id(&caller) {
            return Err(InvalidState::NotParticipant { account_id: caller }.into());
        }
    }
    // ... existing logic
}

### Proof of Concept

1. Monitor the NEAR chain for a TEE upgrade event: participants vote in a new docker image hash via `vote_code_hash`, and the `tee_upgrade_deadline_duration` for the old hash elapses.
2. Identify participants still running the old image whose attestations now fail `reverify_participants` (their `TeeQuoteStatus` is no longer `Valid`).
3. From any unprivileged NEAR account, call:
   ```
   clean_invalid_attestations(max_scan: 4294967295)
   ```
4. The contract removes all failing attestations from `stored_attestations` in a single transaction.
5. Affected participants now fail `assert_caller_is_attested_participant_and_protocol_active` — they cannot call `respond`, `vote_pk`, `vote_reshared`, or any other threshold-gated method.
6. Call `verify_tee` from any remaining participant. The contract sees the missing attestations, treats those participants as having invalid TEE status, and either sets `accept_requests = false` (halting all signing) or triggers an automatic resharing with a reduced participant set — both without any threshold of participants authorizing this outcome.     pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
                }

                // here, we set it to true, because at this point, we have at least `threshold`
                // number of participants with an accepted Tee status.
                self.accept_requests = true;

                // do we want to adjust the threshold?
                //let n_participants_new = new_participants.len();
                //let new_threshold = (3 * n_participants_new + 4) / 5; // minimum 60%
                //let new_threshold = new_threshold.max(2); // but also minimum 2
                let new_threshold = usize::try_from(current_params.threshold().value())
                    .expect("threshold value fits in usize");

                let threshold_parameters = ThresholdParameters::new(
                    participants_with_valid_attestation,
                    Threshold::new(new_threshold as u64),
                )
                .expect("Require valid threshold parameters"); // this should never happen.
                current_params.validate_incoming_proposal(&threshold_parameters)?;
                // This resharing only changes the participant set, so the
                // per-domain reconstruction-threshold updates map is empty.
                let proposed_parameters =
                    ProposedThresholdParameters::new(threshold_parameters, BTreeMap::new());
                let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
                if let Some(resharing) = res {
                    self.protocol_state = ProtocolContractState::Resharing(resharing);
                }

                Ok(true)
            }
        }
    }

    pub fn clean_invalid_attestations(
        &mut self,
        tee_upgrade_deadline_duration: Duration,
        max_scan: usize,
    ) -> u32 {
        let has_invalid_attestation = |node_id: &NodeId| {
            !matches!(
                self.reverify_participants(node_id, tee_upgrade_deadline_duration),
                TeeQuoteStatus::Valid
            )
        };

        // Materialize candidates before any mutation to avoid iterator invalidation.
        let invalid_tls_keys: Vec<Ed25519PublicKey> = self
            .stored_attestations
            .iter()
            .take(max_scan)
            .filter(|(_, node_attestation)| has_invalid_attestation(&node_attestation.node_id))
            .map(|(tls_pk, _)| tls_pk.clone())
            .collect();

        let removed = u32::try_from(invalid_tls_keys.len())
            .expect("u32 should always be convertible from usize on wasm32");

        for tls_pk in invalid_tls_keys {
            self.stored_attestations.remove(&tls_pk);
        }
        removed
