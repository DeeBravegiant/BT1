### Title
Sub-Transaction Plutus ExUnits Bypass Per-Transaction and Per-Block Execution Limits — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may embed an ordered set of sub-transactions (`sub_transactions`, field 23 of the transaction body). Each sub-transaction carries its own witness set, including Plutus redeemers with declared `ExUnits` budgets. The `maxTxExUnits` check and the `maxBlockExUnits` check both operate exclusively on the top-level transaction's redeemers via `totExUnits`. Sub-transaction redeemers are never counted. An unprivileged transaction submitter can therefore pack arbitrarily many sub-transactions, each declaring up to `maxTxExUnits` worth of Plutus execution work, into a single top-level transaction whose declared top-level ExUnits are zero — bypassing both protocol-parameter limits entirely.

---

### Finding Description

**Root cause — `totExUnits` ignores sub-transaction redeemers**

`totExUnits` is defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It reads only the top-level transaction's redeemer map. Sub-transaction redeemers live inside `tx ^. bodyTxL . subTransactionsTxBodyL`, each sub-transaction having its own `witsTxL . rdmrsTxWitsL`. There is no Dijkstra-era override of `totExUnits` that folds over sub-transactions (confirmed: `grep totExUnits eras/dijkstra/**/*.hs` returns no matches).

**Path 1 — per-transaction limit not applied to sub-transactions**

`dijkstraUtxoTransition` (the top-level UTXO rule) calls:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

`validateExUnitsTooBigUTxO` calls `totExUnits tx`, which returns only the top-level redeemers. Sub-transaction redeemers are invisible to this check.

The `SUBUTXO` rule (`dijkstraSubUtxoTransition`) never calls `validateExUnitsTooBigUTxO`. The `DijkstraSubUtxoPredFailure` type has no `ExUnitsTooBigUTxO` constructor, and the mapping function `dijkstraUtxoToDijkstraSubUtxoPredFailure` explicitly marks that case as `error "Impossible: ExUnitsTooBigUTxO for SUBUTXO"`.

**Path 2 — per-block limit not applied to sub-transactions**

`dijkstraBbodyTransition` calls:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
```

`Alonzo.validateExUnits` is:

```haskell
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
  in pointWiseExUnits (<=) txTotal ppMax ?! ...
```

`foldMap totExUnits txs` sums `totExUnits` over every top-level transaction in the block. Because `totExUnits` ignores sub-transaction redeemers, the block-level sum is also blind to all sub-transaction Plutus work.

---

### Impact Explanation

**Impact: Medium** — Attacker-controlled transactions exceed intended validation limits.

`maxTxExUnits` and `maxBlockExUnits` are the two protocol parameters that bound the real-world CPU and memory cost of validating a block. Their purpose is to guarantee that block validation completes within the slot time. By embedding N sub-transactions each declaring `maxTxExUnits` worth of Plutus work into a single top-level transaction with zero top-level redeemers, an attacker causes the actual script execution budget consumed during block validation to be `N × maxTxExUnits`, with no upper bound enforced by the ledger rules. This can force honest nodes to spend far more time validating a block than the protocol parameters were designed to allow, degrading liveness.

---

### Likelihood Explanation

The Dijkstra era is the first era to support nested transactions. Any transaction submitter (no special privilege required) can construct a top-level transaction with an arbitrary number of sub-transactions, each carrying Plutus redeemers. The only cost to the attacker is the transaction fee, which is computed from `totExUnits tx` (top-level only) and therefore does not scale with sub-transaction ExUnits. The attacker pays fees proportional to zero declared top-level ExUnits while forcing validators to execute an unbounded amount of Plutus work.

---

### Recommendation

1. **Extend `totExUnits` for Dijkstra** to recursively sum ExUnits across all sub-transactions:
   ```haskell
   dijkstraTotExUnits tx =
     totExUnits tx
       <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
   ```
   and use this in both `validateExUnitsTooBigUTxO` (top-level UTXO rule) and `validateExUnits` (BBODY rule).

2. **Add `validateExUnitsTooBigUTxO` to `dijkstraSubUtxoTransition`** so each sub-transaction is individually bounded by `maxTxExUnits`, mirroring the check applied to top-level transactions.

3. **Update fee calculation** (`alonzoMinFeeTx` / `minfee`) to include sub-transaction ExUnits so that the fee paid reflects the actual script execution cost of the entire batch.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction with zero top-level redeemers and `K` sub-transactions, each containing a Plutus script redeemer declaring `ExUnits { exUnitsMem = maxMem, exUnitsSteps = maxSteps }` (equal to `maxTxExUnits`).
2. Submit the transaction. The UTXO rule checks `totExUnits tx = ExUnits 0 0 ≤ maxTxExUnits` — passes.
3. The BBODY rule checks `foldMap totExUnits [tx] = ExUnits 0 0 ≤ maxBlockExUnits` — passes.
4. The `SUBUTXOW` rule executes each sub-transaction's Plutus scripts with the declared budget. Total execution work = `K × maxTxExUnits`, unbounded by any ledger check.
5. With `K` large enough, block validation time exceeds the slot duration, degrading node liveness.

-- | Validate that total execution units (all transactions) do not exceed block limit.
-- ∑(tx ∈ txs)(totExunits tx) ≤ maxBlockExUnits pp
validateExUnits ::
  forall era.
  ( AlonzoEraTx era
  , InjectRuleFailure "BBODY" AlonzoBbodyPredFailure era
  ) =>
  StrictSeq.StrictSeq (Tx TopTx era) ->
  -- | Max block exunits protocol parameter.
  ExUnits ->
  Rule (EraRule "BBODY" era) 'Transition ()
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax
        ?! injectFailure
          ( TooManyExUnits $
              Mismatch
                { mismatchSupplied = txTotal
                , mismatchExpected = ppMax
                }
          )
