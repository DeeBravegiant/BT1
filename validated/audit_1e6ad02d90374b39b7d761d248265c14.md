### Title
Missing `validateAllRefScriptSize` Check in `SUBLEDGER` Rule Allows Sub-Transactions to Bypass Reference Script Size Limit — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedger.hs`)

---

### Summary

The Dijkstra era's `SUBLEDGER` transition rule omits the `validateAllRefScriptSize` check that is present in the top-level `LEDGER` rule. This means sub-transactions embedded in a Dijkstra top-level transaction are never validated against the `ppMaxRefScriptSizePerTxG` protocol parameter, allowing an unprivileged sender to include sub-transactions with arbitrarily large reference scripts and exceed the intended per-transaction resource limit.

---

### Finding Description

The Dijkstra era introduces nested ("sub") transactions. The top-level `LEDGER` rule (`dijkstraLedgerTransition`) enforces a reference script size limit via `validateAllRefScriptSize`:

```haskell
runTest $ Conway.validateTreasuryValue txBody (chainAccountState ^. casTreasuryL)
runTest $ validateAllRefScriptSize pp originalUtxo tx   -- ← present
``` [1](#0-0) 

The `SUBLEDGER` rule (`dijkstraSubLedgersTransition`), which processes each sub-transaction, calls `validateTreasuryValue` but **never calls `validateAllRefScriptSize`** (or any equivalent):

```haskell
runTest $ Conway.validateTreasuryValue txBody (chainAccountState ^. casTreasuryL)
-- validateAllRefScriptSize is absent here
``` [2](#0-1) 

`validateAllRefScriptSize` is defined as:

```haskell
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $ ...
``` [3](#0-2) 

It takes a `Tx TopTx era` (top-level transaction type), not a `Tx SubTx era`, so it structurally cannot cover sub-transactions even if called from the top-level rule. Sub-transactions are `StAnnTx SubTx era` and are dispatched exclusively through `SUBLEDGERS → SUBLEDGER`. [4](#0-3) 

The `SUBLEDGERS` rule simply folds over sub-transactions and delegates each to `SUBLEDGER` without any size pre-check: [5](#0-4) 

---

### Impact Explanation

An attacker can submit a single top-level Dijkstra transaction whose sub-transactions collectively reference scripts far exceeding `ppMaxRefScriptSizePerTxG`. Because the `SUBLEDGER` rule never enforces this limit, every honest block-producing node must execute script validation against the oversized reference scripts. This allows an attacker-controlled transaction to exceed the intended per-transaction validation resource limit, matching the allowed Medium impact: *"Attacker-controlled transactions… exceed intended validation limits."*

---

### Likelihood Explanation

**Medium.** The Dijkstra era is present in the production codebase and is the next planned era. Any unprivileged transaction sender can craft a top-level transaction with sub-transactions containing large reference scripts. No privileged access, key compromise, or governance majority is required. The only prerequisite is that the Dijkstra era is active on the network.

---

### Recommendation

Add a reference script size validation step inside `dijkstraSubLedgersTransition` (or inside `dijkstraSubLedgersTransition` per sub-transaction) analogous to the check in the top-level rule. A dedicated `validateSubRefScriptSize` function operating on `Tx SubTx era` should be introduced and called from `SUBLEDGER` alongside `validateTreasuryValue`:

```haskell
runTest $ Conway.validateTreasuryValue txBody (chainAccountState ^. casTreasuryL)
runTest $ validateSubRefScriptSize pp originalUtxo tx   -- add this
```

The protocol parameter `ppMaxRefScriptSizePerTxG` should apply uniformly to both top-level and sub-transactions.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction `T` whose `subTransactions` field contains `N` sub-transactions `s₁ … sₙ`.
2. Each `sᵢ` references UTxO outputs whose inline scripts sum to just under `ppMaxRefScriptSizePerTxG` bytes.
3. Submit `T`. The top-level `LEDGER` rule calls `validateAllRefScriptSize` only on `T` itself (which has no reference scripts of its own), so it passes.
4. `SUBLEDGERS` folds over `[s₁ … sₙ]`, calling `SUBLEDGER` for each. `SUBLEDGER` calls `SUBUTXOW` for script execution but never calls `validateAllRefScriptSize`.
5. All `N × ppMaxRefScriptSizePerTxG` bytes of reference scripts are loaded and executed, far exceeding the intended per-transaction limit. [6](#0-5) [3](#0-2)

    judgmentContext

dijkstraSubLedgersTransition ::
  forall era.
  ( EraRule "SUBLEDGERS" era ~ SUBLEDGERS era
  , EraRule "SUBLEDGER" era ~ SUBLEDGER era
  , Embed (EraRule "SUBLEDGER" era) (SUBLEDGERS era)
  ) =>
  TransitionRule (EraRule "SUBLEDGERS" era)
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
