Audit Report

## Title
Missing Zero-Value Check on Computed rsETH Mint Amount Allows Silent Loss of Deposited Assets - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._beforeDeposit` computes `rsethAmountToMint` via integer division and guards it only with a caller-supplied `minRSETHAmountExpected` comparison. When `minRSETHAmountExpected = 0` (which the protocol never rejects, as `minAmountToDeposit` defaults to 0 and is never set in `initialize`), a deposit whose computed mint amount truncates to zero passes all checks, calls `_mintRsETH(0)`, and leaves the depositor's ETH or LST held by the pool with no rsETH issued. The same structural gap exists across all L2 RSETHPool variants, which have no slippage parameter at all.

## Finding Description
`getRsETHAmountToMint` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

When `amount * assetPrice < rsETHPrice` (e.g., a 1-wei ETH deposit when `rsETHPrice ≈ 1.05e18`), Solidity integer division truncates the result to 0.

`_beforeDeposit` then applies:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

With `minRSETHAmountExpected = 0`, the condition evaluates to `0 < 0 = false` and does not revert. There is no `require(rsethAmountToMint > 0)` guard. Execution returns to `depositETH`, which calls `_mintRsETH(0)`, minting nothing while the user's ETH has already been received by the contract.

`minAmountToDeposit` is declared as a storage variable but never assigned in `initialize`, so it defaults to 0, permitting 1-wei deposits.

For all RSETHPool variants (`RSETHPool`, `RSETHPoolV2`, `RSETHPoolV3`, etc.), `viewSwapRsETHAmountAndFee` computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

and the `deposit` function unconditionally transfers or mints `rsETHAmount` with no zero check and no slippage parameter. A `safeTransfer(msg.sender, 0)` or `mint(msg.sender, 0)` succeeds silently under standard ERC-20 semantics.

## Impact Explanation
A depositor loses their deposited ETH or LST while receiving 0 rsETH/wrsETH. The protocol retains the deposited value without issuing any claim token. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**: the user's asset is consumed, the protocol's accounting is unaffected, and no rsETH is ever issued for that deposit.

## Likelihood Explanation
Any unprivileged external caller can trigger this. `minAmountToDeposit = 0` means 1-wei deposits are accepted. On mainnet, `rsETHPrice` has already exceeded 1e18 due to staking rewards, so a 1-wei ETH deposit produces `rsethAmountToMint = 0`. A user who omits or zeroes the slippage parameter (a common default in integrations and scripts) is silently harmed. On all RSETHPool L2 contracts the risk is structural — there is no slippage parameter to set, so any deposit small enough to truncate is affected unconditionally.

## Recommendation
Add an explicit non-zero guard in `_beforeDeposit` immediately after computing the mint amount:

```solidity
if (rsethAmountToMint == 0) revert ZeroRsETHMintAmount();
```

Apply the same guard in every RSETHPool `deposit` function after computing `rsETHAmount`:

```solidity
if (rsETHAmount == 0) revert ZeroRsETHAmount();
```

Additionally, consider enforcing `minAmountToDeposit > 0` during initialization, or at minimum documenting that passing `minRSETHAmountExpected = 0` disables slippage protection entirely.

## Proof of Concept
**LRTDepositPool path:**
1. `rsETHPrice` on mainnet is ~1.05e18 (staking rewards accrued).
2. User calls `LRTDepositPool.depositETH{value: 1}(0, "")`.
3. `_beforeDeposit`: `depositAmount = 1` passes `depositAmount == 0 || depositAmount < minAmountToDeposit` (both 0).
4. `getRsETHAmountToMint`: `(1 * 1e18) / 1.05e18 = 0` (integer truncation).
5. Slippage check: `0 < 0 = false` → no revert.
6. `_mintRsETH(0)` → 0 rsETH minted; user's 1 wei ETH is held by the pool.

**RSETHPool path:**
1. User calls `RSETHPool.deposit{value: N}("")` where `N` is small enough that `(N - fee) * 1e18 / rsETHToETHrate = 0`.
2. `viewSwapRsETHAmountAndFee` returns `rsETHAmount = 0`.
3. `IERC20(address(wrsETH)).safeTransfer(msg.sender, 0)` succeeds silently.
4. User's ETH is consumed; 0 wrsETH received.

**Foundry fuzz test sketch:**
```solidity
function testFuzz_zeroMintOnSmallDeposit(uint96 amount) public {
    vm.assume(amount > 0 && amount < rsETHPrice / assetPrice);
    uint256 balBefore = rsETH.balanceOf(user);
    vm.prank(user);
    depositPool.depositETH{value: amount}(0, "");
    assertEq(rsETH.balanceOf(user), balBefore); // receives nothing
}

        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
