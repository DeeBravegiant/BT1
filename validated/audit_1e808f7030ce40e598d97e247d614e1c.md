Audit Report

## Title
Unchecked Zero `owner` Address in `addLiquidity` Permanently Locks LP Tokens - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no zero-address guard. If `owner = address(0)` is supplied, the pool mints shares to an unreachable position key, pulls real tokens from `msg.sender` via the modify-liquidity callback, and makes those tokens permanently irrecoverable because `removeLiquidity` enforces `msg.sender == owner`, which can never be satisfied for `address(0)`.

## Finding Description
`MetricOmmPool.addLiquidity` (L182–196) accepts `address owner` and passes it directly to `LiquidityLib.addLiquidity` with no zero-address check:

```solidity
function addLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    bytes calldata callbackData, bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    ...
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
        _liquidityContext(), owner, salt, deltas, callbackData, ...
    );
}
```

Inside `LiquidityLib.addLiquidity`, the position key is computed as `keccak256(abi.encode(address(0), salt, binIdx))` and shares are credited to that key (L72, L121). Tokens are then pulled from `msg.sender` via the callback (L147–148). `removeLiquidity` enforces `msg.sender == owner` (L206) before burning shares — a condition that can never be satisfied for `address(0)`, permanently freezing the deposited tokens inside `binTotals` and per-bin balances.

The periphery contract `MetricOmmPoolLiquidityAdder` guards against this via `_validateOwner` (L247–249), but the core pool is a public interface. The `addLiquidityWeighted(pool, owner, ...)` overload (L88–116) does call `_validateOwner(owner)` at L101, but any direct caller of the core pool bypasses this entirely.

## Impact Explanation
Direct, permanent loss of user principal. Tokens deposited under `owner = address(0)` are credited to an unreachable position key. They remain inside the pool's accounting (`binTotals.scaledToken0/1`, `binState.token0/1BalanceScaled`) and cannot be extracted via `removeLiquidity`, `collectFees`, or any other mechanism. The pool's LP accounting becomes insolvent relative to the locked shares, diluting other LPs' proportional claims.

## Likelihood Explanation
Medium-Low. The core pool is a public contract; any integrator, router, or user calling it directly (not through `MetricOmmPoolLiquidityAdder`) can trigger this. A single mistaken or malicious call is sufficient. Future periphery extensions or third-party routers that omit the `_validateOwner` call would silently lock funds.

## Recommendation
Add a zero-address guard at the top of `MetricOmmPool.addLiquidity`:

```solidity
function addLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    bytes calldata callbackData, bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    require(owner != address(0), "MetricOmmPool: zero owner");
    ...
}
```

Optionally add the same guard at the top of `LiquidityLib.addLiquidity` as a defense-in-depth measure.

## Proof of Concept
```solidity
// Direct call to core pool with owner = address(0)
pool.addLiquidity(
    address(0),   // owner = zero address
    0,            // salt
    deltas,       // valid bin/share deltas
    callbackData, // callback that pays tokens
    ""
);
// Tokens are pulled from msg.sender via metricOmmModifyLiquidityCallback.
// Shares are minted to keccak256(abi.encode(address(0), 0, binIdx)).
// removeLiquidity(address(0), ...) always reverts NotPositionOwner
// because msg.sender != address(0) is always true.
// Tokens are permanently locked in binTotals and binState balances.
```
