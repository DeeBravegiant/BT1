### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Allows Forged Foreign-Chain Verification Attestations — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash` using the MPC root public key, but never verifies that `payload_hash` was actually derived from the `request` argument. A single compromised attested participant (strictly below the signing threshold) can reuse a valid MPC-network signature obtained from one legitimate foreign-tx request to forge a passing response for a completely different pending request.

---

### Finding Description

`VerifyForeignTransactionResponse` carries only two fields:

```rust
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,   // caller-supplied, unchecked
    pub signature: SignatureResponse,
}
``` [1](#0-0) 

The contract's verification logic in `respond_verify_foreign_tx` is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [2](#0-1) 

The contract only checks: **"is `signature` a valid ECDSA signature over `payload_hash` under the root key?"** It never checks: **"was `payload_hash` computed as `SHA-256(borsh(ForeignTxSignPayload{ request, extracted_values }))`?"**

By contrast, the node-side code that produces the hash is:

```rust
let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
    foreign_tx_payload.compute_msg_hash()?.into();
// ...
tweak: Tweak::new([0u8; 32]),   // zero tweak → root key, no derivation
``` [3](#0-2) 

The design document confirms the intended binding:

> `msg_hash = SHA-256(borsh(ForeignTxSignPayload))` … Only the hash is included in the response to stay within NEAR's promise data limits. [4](#0-3) 

Because `extracted_values` are absent from `VerifyForeignTransactionResponse` and the contract never recomputes the expected hash from `request`, the binding is entirely absent on-chain.

---

### Impact Explanation

**High — Forged foreign-chain verification / invalid bridge execution.**

A single compromised attested participant executes the following steps without any threshold-level collusion:

1. Submit `verify_foreign_transaction(request_A)` — a legitimate request for a real foreign transaction.
2. Observe the MPC network's threshold-signed output `(hash_A, sig_A)` where `hash_A = SHA-256(borsh(ForeignTxSignPayload{ request_A, values_A }))`.
3. Separately submit `verify_foreign_transaction(request_B)` — a fraudulent request (e.g., a non-existent or already-spent transaction).
4. Call `respond_verify_foreign_tx(request_B, { payload_hash: hash_A, signature: sig_A })`.
5. The contract passes the signature check (valid `sig_A` over `hash_A` under root key) and calls `resolve_yields_for`, delivering `(hash_A, sig_A)` to every caller waiting on `request_B`. [5](#0-4) 

The bridge contract (e.g., Omnibridge) receives a `VerifyForeignTransactionResponse` that carries a genuine MPC-network signature but one that attests to `request_A`, not `request_B`. Because the response struct contains no `extracted_values`, the bridge contract cannot reconstruct the expected hash and distinguish the forgery. This enables double-spend or theft of bridged funds.

---

### Likelihood Explanation

**Medium.** The attacker must be a single attested participant — a legitimate MPC node operator whose TEE attestation is accepted by the contract. This is a realistic threat model: a single node below the signing threshold is compromised or acts maliciously. No threshold collusion, no key-share leakage, and no network-level attack is required. The attacker only needs to observe a completed signing session (which every participating node does) and submit one fraudulent `respond_verify_foreign_tx` transaction before any honest node does.

---

### Recommendation

Include `extracted_values` in `VerifyForeignTransactionResponse` and have `respond_verify_foreign_tx` recompute the expected hash on-chain:

```rust
// In respond_verify_foreign_tx, after retrieving the public key:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),   // new field
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
// then verify signature over response.payload_hash as today
```

This mirrors how `respond` binds the signature to the on-chain `request.payload` and `request.tweak` rather than trusting a caller-supplied hash. [6](#0-5)

### Proof of Concept

```
// Setup: contract in Running state, ForeignTx domain, Bitcoin supported.

// Step 1 – attacker (attested participant P) submits a real request.
P → contract.verify_foreign_transaction({ tx_id: REAL_TX, extractors: [BlockHash] })
// MPC network signs hash_A = SHA256(borsh(ForeignTxSignPayload{ REAL_TX, [block_hash_A] }))
// P observes (hash_A, sig_A) from the signing session.

// Step 2 – attacker submits a fraudulent request for a fake/spent tx.
Attacker → contract.verify_foreign_transaction({ tx_id: FAKE_TX, extractors: [BlockHash] })
// request_B is now pending in pending_verify_foreign_tx_requests.

// Step 3 – attacker forges the response.
P → contract.respond_verify_foreign_tx(
    request  = { tx_id: FAKE_TX, ... },   // resolves request_B
    response = { payload_hash: hash_A, signature: sig_A }
)
// Contract checks: verify_ecdsa_signature(sig_A, hash_A, root_pk) → OK ✓
// No check that hash_A == SHA256(borsh({ FAKE_TX, ... }))
// resolve_yields_for delivers (hash_A, sig_A) to bridge contract waiting on FAKE_TX.

// Bridge contract receives a valid MPC signature but for the wrong transaction.
// It cannot detect the mismatch because extracted_values are absent from the response.
``` [7](#0-6) [1](#0-0)
