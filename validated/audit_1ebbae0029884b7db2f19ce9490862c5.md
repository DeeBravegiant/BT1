### Title
Single Malicious Participant Can Deliver Arbitrary CKD Output for `AppPublicKey` Requests, Bypassing Threshold Requirement - (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs cryptographic output verification only for the `AppPublicKeyPV` variant of a CKD request. For the `AppPublicKey` (legacy) variant, no on-chain verification is performed. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary, self-chosen CKD response, unilaterally resolving a pending CKD request and delivering a key the attacker controls to the user.

---

### Finding Description

The `respond_ckd` function in `crates/contract/src/lib.rs` handles two variants of `CKDAppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the response against the BLS12-381 master public key on-chain, ensuring the output is consistent with the threshold-derived key. For `AppPublicKey`, the branch is a no-op — any `CKDResponse` value is accepted unconditionally.

By contrast, `respond` (for ECDSA/EdDSA signatures) always verifies the response cryptographically against the derived public key before accepting it:

```rust
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [2](#0-1) 

This asymmetry means the threshold guarantee — that `t`-of-`n` participants must collaboratively produce the CKD output — is enforced for `AppPublicKeyPV` but entirely absent for `AppPublicKey`.

The caller must be an attested participant:

```rust
self.assert_caller_is_attested_participant_and_protocol_active();
``` [3](#0-2) 

But a single Byzantine participant who has passed TEE attestation satisfies this check. After passing it, they can call `respond_ckd` with a `CKDResponse` containing key material they generated themselves (e.g., `big_y` and `big_c` corresponding to a key they know). `resolve_yields_for` then drains the pending queue and delivers the attacker-chosen key to the user:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [4](#0-3) 

The off-chain threshold protocol is bypassed entirely — the attacker never participates in the MPC rounds.

---

### Impact Explanation

A single malicious participant delivers a CKD output they fully control to the user. Because the attacker chose `big_y` and `big_c`, they know the corresponding plaintext key. Any secret encrypted to the delivered "app private key" is immediately readable by the attacker. This constitutes **unauthorized confidential key derivation output without the required participant authorization** — the threshold requirement (`t`-of-`n`) is bypassed by a single participant acting alone.

---

### Likelihood Explanation

Any single attested participant can execute this attack against any pending `AppPublicKey` CKD request. The `AppPublicKey` variant is the legacy default path (documented first in the README and contract README). Users who do not explicitly opt into `AppPublicKeyPV` are vulnerable. The attacker only needs to be faster than the honest MPC network in calling `respond_ckd` — a race that is straightforward to win by monitoring the NEAR chain for new CKD requests and immediately submitting a crafted response.

---

### Recommendation

Apply the same on-chain output verification to `AppPublicKey` responses as is applied to `AppPublicKeyPV`. For `AppPublicKey`, the BLS12-381 master public key is available on-chain (`self.public_key_extended(request.domain_id)`), so it is possible to verify that the encrypted output in `CKDResponse` is consistent with the threshold-derived key, even without the user's private key. If such verification is cryptographically infeasible for the legacy format, the `AppPublicKey` variant should be deprecated and removed, requiring all callers to use `AppPublicKeyPV`.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey` variant; a pending CKD request is stored in `pending_ckd_requests`.
2. Attacker (a single attested participant) observes the pending request on-chain.
3. Attacker generates arbitrary BLS12-381 key material `(big_y, big_c)` for a key they control.
4. Attacker calls `respond_ckd(request, CKDResponse { big_y, big_c })`.
5. `respond_ckd` passes all checks (attested participant, running state, `accept_requests`), enters the `AppPublicKey` branch with no verification, and calls `resolve_yields_for`, delivering the attacker-chosen response to the user.
6. The user's NEAR yield-resume callback fires with the attacker-controlled key material. The attacker knows the corresponding plaintext private key.

The threshold protocol was never run; no other participant was involved.
