Audit Report

## Title
Stale `rsETHPrice` Allows Over-Minting of rsETH, Diluting Existing Holders' Accrued Yield — (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`, `contracts/oracles/RETHPriceOracle.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint` divides a live rETH/ETH exchange rate by a stored, potentially stale `rsETHPrice`. Because `updateRSETHPrice()` is never called atomically before a deposit, any depositor can mint rsETH at a rate that over-represents the current rETH yield. The resulting over-minted rsETH dilutes the accrued-but-uncaptured yield that belongs to existing rsETH holders.

## Finding Description

`getRsETHAmountToMint` at `LRTDepositPool.sol:520` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

The numerator calls `LRTOracle.getAssetPrice(rETH)`, which delegates to `RETHPriceOracle.getAssetPrice`, returning the **live** on-chain value:

```solidity
return IrETH(rETHAddress).getExchangeRate();
``` [2](#0-1) 

The denominator reads `lrtOracle.rsETHPrice()`, a **stored state variable** only updated when `_updateRsETHPrice()` is explicitly invoked:

```solidity
rsETHPrice = newRsETHPrice;
``` [3](#0-2) 

`updateRSETHPrice()` is permissionless but is **never called** inside the deposit path:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [4](#0-3) 

`_beforeDeposit` is declared `private view`, making it structurally incapable of calling any state-mutating function, and it calls `getRsETHAmountToMint` directly without any price-refresh step: [5](#0-4) 

`_updateRsETHPrice` computes `newRsETHPrice = totalETHInProtocol / rsethSupply`, where `totalETHInProtocol` uses the live rETH exchange rate. When rETH accrues yield, `totalETHInProtocol` rises, so the fair `rsETHPrice` rises too — but the stored value lags until someone calls `updateRSETHPrice()`. [6](#0-5) 

**Exploit path:**
1. rETH accrues staking yield: live `getExchangeRate()` rises from R₀ to R₁ > R₀.
2. Stored `rsETHPrice` is still P₀ (computed when rETH was at R₀); fair updated price would be P₁ > P₀.
3. Attacker calls `depositAsset(rETH, X)` — no `updateRSETHPrice()` required.
4. Attacker receives `X * R₁ / P₀` rsETH.
5. Fair amount (post-update) would be `X * R₁ / P₁`; since P₀ < P₁, attacker receives excess rsETH.
6. When `updateRSETHPrice()` is eventually called, the new price is computed over a larger rsETH supply (inflated by the attacker's over-mint), permanently reducing the per-rsETH ETH value for all pre-existing holders.

## Impact Explanation

The yield accrued by rETH between price updates belongs to existing rsETH holders — it should increase `rsETHPrice`. By depositing before the price update, the attacker captures a portion of that yield as extra rsETH backed by less ETH than it claims. The shortfall is borne by existing holders whose effective `rsETHPrice` is diluted. This is a direct, quantifiable **theft of unclaimed yield** (High severity per scope).

## Likelihood Explanation

- rETH accrues yield continuously (~4% APY ≈ ~0.011% per day); any gap between `updateRSETHPrice()` calls creates an exploitable window.
- No special role, front-running, or oracle manipulation is required — any depositor can trigger this permissionlessly.
- The attack is repeatable: deposit, wait for the next yield accrual window, repeat.
- Profit scales with deposit size and the yield accrued since the last price update.

## Recommendation

Enforce a price update atomically before computing the mint amount. In `depositAsset` and `depositETH`, call `updateRSETHPrice()` before `_beforeDeposit` (since `_beforeDeposit` is `private view` and cannot do so itself):

```solidity
// At the start of depositAsset and depositETH, before _beforeDeposit:
ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
```

This ensures `rsETHPrice` always reflects the current live asset rates before any mint calculation, eliminating the staleness window entirely.

## Proof of Concept

```solidity
// Foundry fork test (mainnet fork, any block where rETH has accrued yield
// since the last rsETHPrice update)
function testStealUnclaimedYield() external {
    // 1. Record current state
    uint256 rsETHPriceBefore = lrtOracle.rsETHPrice();
    uint256 rETHRate = rETH.getExchangeRate(); // live; > rsETHPriceBefore if yield accrued

    // Precondition: yield has accrued since last update
    require(rETHRate > rsETHPriceBefore, "no yield window");

    // 2. Attacker deposits WITHOUT calling updateRSETHPrice first
    uint256 depositAmount = 10 ether;
    deal(address(rETH), attacker, depositAmount);
    vm.startPrank(attacker);
    rETH.approve(address(lrtDepositPool), depositAmount);
    lrtDepositPool.depositAsset(address(rETH), depositAmount, 0, "");
    vm.stopPrank();

    uint256 rsETHMinted = rsETH.balanceOf(attacker);

    // 3. Now update the price (as any user would eventually do)
    lrtOracle.updateRSETHPrice();
    uint256 rsETHPriceAfter = lrtOracle.rsETHPrice();

    // 4. Fair amount would have been: depositAmount * rETHRate / rsETHPriceAfter
    uint256 fairRsETHAmount = (depositAmount * rETHRate) / rsETHPriceAfter;

    // 5. Attacker received more than fair share — yield stolen from existing holders
    assertGt(rsETHMinted, fairRsETHAmount, "Attacker stole unclaimed yield");
}
```

The assertion passes whenever `rETHRate > rsETHPriceBefore`, i.e., any time rETH has accrued yield since the last `updateRSETHPrice()` call.
