Audit Report

## Title
L2 Pool `deposit()` Functions Lack User-Configurable Minimum Output Parameter - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPool.sol)

## Summary
All four L2 pool `deposit()` functions accept no minimum output parameter, meaning the output amount is determined solely by the oracle rate at execution time with no floor. L1 `LRTDepositPool.depositETH()` provides `minRSETHAmountExpected` and reverts via `MinimumAmountToReceiveNotMet()` if the minted amount falls short. Any L2 depositor who previews the rate off-chain and submits a transaction may receive materially fewer wrsETH/rsETH than expected if the oracle rate updates before inclusion, with no on-chain mechanism to revert.

## Finding Description
`LRTDepositPool.depositETH()` passes `minRSETHAmountExpected` into `_beforeDeposit`, which enforces:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [1](#0-0) 

Every L2 pool `deposit()` function omits this guard entirely. In `RSETHPoolV3ExternalBridge`:

```solidity
function deposit(string memory referralId)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
``` [2](#0-1) 

The output is computed as:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

The same pattern is present in `RSETHPoolV3.deposit()` [4](#0-3) , `RSETHPoolV2ExternalBridge.deposit()` [5](#0-4) , and `RSETHPool.deposit()` [6](#0-5) .

The only existing check is `amount == 0`; there is no floor on the output amount. The minimum output is implicitly hardcoded to zero.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.**

A depositor who previews the exchange rate and submits a transaction may receive fewer wrsETH/rsETH than the amount they observed, with no on-chain mechanism to revert. The deposited ETH is not lost, but the user receives fewer liquid restaking tokens than the rate they observed at submission time. This directly matches the allowed impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
**Low.** The `rsETHOracle` rate is protocol-controlled and changes gradually as EigenLayer rewards accrue and underlying LST prices shift — not on every block like an AMM. However, the rate does change across blocks, and any depositor whose transaction is delayed in the mempool or reordered can receive a worse rate than previewed. No attacker is required; normal oracle updates are sufficient to trigger the condition. Every L2 deposit is subject to this risk with no opt-out.

## Recommendation
Add a `minRsETHAmount` parameter to every L2 pool `deposit()` function, mirroring the L1 pattern:

```diff
- function deposit(string memory referralId)
-     external payable nonReentrant whenNotPaused
-     limitDailyMint(msg.value, ETH_IDENTIFIER)
+ function deposit(uint256 minRsETHAmount, string memory referralId)
+     external payable nonReentrant whenNotPaused
+     limitDailyMint(msg.value, ETH_IDENTIFIER)
  {
      uint256 amount = msg.value;
      if (amount == 0) revert InvalidAmount();
      (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
+     if (rsETHAmount < minRsETHAmount) revert InsufficientOutputAmount();
      feeEarnedInETH += fee;
      wrsETH.mint(msg.sender, rsETHAmount);
      emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
  }
```

Apply the same change to the token-deposit overloads in all four pool contracts.

## Proof of Concept
1. The current rsETH/ETH oracle rate is `R`. A user calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain and observes they will receive `X` wrsETH.
2. The user submits `deposit{value: 1 ether}("ref")` to `RSETHPoolV3ExternalBridge`.
3. Before the transaction is included, the protocol oracle is updated to rate `R' > R` (rsETH is now worth more ETH, so fewer wrsETH are minted per ETH deposited).
4. The transaction executes. `viewSwapRsETHAmountAndFee` returns `X' < X`. Because there is no minimum-output check, `wrsETH.mint(msg.sender, X')` succeeds silently.
5. The user receives `X'` wrsETH instead of `X`, with no revert and no recourse.
6. On L1, the same scenario would revert with `MinimumAmountToReceiveNotMet` if the user had set `minRSETHAmountExpected = X`.

**Foundry fork test plan:**
- Fork the relevant L2 at a block where the oracle rate is known.
- Call `viewSwapRsETHAmountAndFee(1 ether)` to record expected output `X`.
- Use `vm.mockCall` or advance state to simulate an oracle rate increase.
- Call `deposit{value: 1 ether}("ref")` and assert the minted balance is `X' < X` with no revert — confirming the absence of any floor check.
