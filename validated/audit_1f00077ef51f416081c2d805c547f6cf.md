### Title
`vote_reshared` Does Not Reset `accept_requests` After Successful Resharing — (File: `crates/contract/src/lib.rs`)

---

### Summary

When `verify_tee` sets `accept_requests = false` due to a broken threshold relation, and participants subsequently trigger a resharing via `vote_new_parameters` with a fully valid proposed participant set, the `vote_reshared` function completes the resharing and transitions the contract to `Running` state — but **never resets `accept_requests` to `true`**. The contract remains frozen: every `sign`, `respond`, `respond_ckd`, and `respond_verify_foreign_tx` call fails with `TeeValidationFailed`, even though the resharing succeeded and all new participants carry valid TEE attestations.

This is the direct analog of the Bond M-1 finding: a state-transition function (`vote_reshared` ↔ `pushOwnership`) does not re-validate or restore the authorization flag (`accept_requests` ↔ `callbackAuthorized`) that the usage-time check depends on, leaving the system permanently broken until an out-of-band recovery call is made.

---

### Finding Description

**Authorization check at usage time** — every user-facing and node-facing request method calls `check_request_preconditions`, which panics when `accept_requests == false`: [1](#0-0) 

The same guard appears in `respond`, `respond_ckd`, and `respond_verify_foreign_tx`: [2](#0-1) [3](#0-2) [4](#0-3) 

**Flag set to `false` by `verify_tee`** — when the surviving valid-TEE participant count would violate the governance-threshold relation, `verify_tee` sets `accept_requests = false` and returns without triggering resharing: [5](#0-4) 

**State transition that never restores the flag** — `vote_reshared` transitions the contract from `Resharing` to `Running` with the new participant set, but contains no assignment to `accept_requests`: [6](#0-5) 

The only code paths that set `accept_requests = true` are both inside `verify_tee`: [7](#0-6) [8](#0-7) 

**`vote_new_parameters` validates TEE at proposal time only** — when participants call `vote_new_parameters` to start a recovery resharing, the contract validates that every proposed participant has a valid TEE attestation at that moment: [9](#0-8) 

This validation is not repeated when `vote_reshared` finalises the transition. After resharing completes, `accept_requests` retains whatever value it held before resharing began.

---

### Impact Explanation

**Impact: Medium** — request-lifecycle and contract execution-flow manipulation that breaks a production safety invariant without relying on network-level DoS or operator misconfiguration.

After a successful resharing (all new participants have valid TEE attestations, the epoch ID is bumped, the keyset is updated), the contract is structurally `Running` but functionally frozen: every `sign`, `respond`, `respond_ckd`, and `respond_verify_foreign_tx` call reverts with `TeeValidationFailed`. Pending yield-resume requests time out and are lost. The freeze persists until a participant calls `verify_tee` again and it returns `TeeValidationResult::Full` — an event that is not automatically scheduled after resharing and may be delayed by up to the periodic `verify_tee` interval (documented as 7 days).

---

### Likelihood Explanation

The trigger sequence is a realistic operational event, not an adversarial one:

1. A TEE image upgrade causes threshold-or-above participants' attestations to expire simultaneously (or a docker-image-hash vote removes the old hash from the whitelist).
2. `verify_tee` is called → `TeeValidationResult::Partial`, threshold relation broken → `accept_requests = false`.
3. Participants re-submit valid attestations for the new image and call `vote_new_parameters` with the corrected set.
4. Resharing completes via `vote_reshared`.
5. `accept_requests` remains `false`; the contract is frozen.

No collusion is required. The scenario arises from the normal TEE-upgrade lifecycle described in the design documentation.

---

### Recommendation

In `vote_reshared`, after the resharing concludes and the contract transitions to `Running`, re-validate TEE attestations for the new participant set and restore `accept_requests`:

```rust
if let Some(new_state) = self.protocol_state.vote_reshared(key_event_id)? {
    self.protocol_state = new_state;
    self.recompute_available_foreign_chains();

    // Restore accept_requests: resharing succeeded with a fully attested set.
    let tee_upgrade_deadline_duration =
        Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);
    if let ProtocolContractState::Running(ref running_state) = self.protocol_state {
        let result = self.tee_state.reverify_and_cleanup_participants(
            running_state.parameters.participants(),
            tee_upgrade_deadline_duration,
        );
        self.accept_requests = matches!(result, TeeValidationResult::Full);
    }

    // ... existing cleanup promises ...
}
```

Alternatively, add a post-resharing call to `verify_tee` as a spawned promise (analogous to the existing `clean_tee_status` promise), so that `accept_requests` is updated atomically with the state transition.

### Proof of Concept

```
Setup: 4 participants, GovernanceThreshold = 3, one CaitSith domain.

Step 1 — Expire 2 participants' attestations (e.g., remove their docker image hash
         from the whitelist via threshold vote).

Step 2 — Call verify_tee().
         → TeeValidationResult::Partial { 2 valid participants }
         → validate_governance_against_reconstruction(2, threshold=3, ...) fails
         → accept_requests = false
         → no resharing triggered

Step 3 — All 4 participants re-submit valid attestations for the new image.

Step 4 — Threshold participants call vote_new_parameters(epoch+1, proposal={all 4, t=3}).
         → TEE check passes (all 4 now valid)
         → Resharing state entered

Step 5 — Resharing completes: start_reshare_instance + vote_reshared × 4.
         → protocol_state = Running (new epoch, new keyset)
         → accept_requests unchanged → still false

Step 6 — Any account calls sign(...).
         → check_request_preconditions panics: "TEE validation"
         → request rejected despite fully healthy Running state
```
