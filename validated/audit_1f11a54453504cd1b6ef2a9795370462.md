Audit Report

## Title
Missing `expectedAssetAmount > 0` Validation Allows rsETH to Be Burned for Zero Assets - (File: contracts/LRTWithdrawalManager.sol)

## Summary
Both `initiateWithdrawal` and `instantWithdrawal` validate that `rsETHUnstaked` is non-zero but never validate that the computed output amount is greater than zero. When `minRsEthAmountToWithdraw[asset]` is at its default value of `0`, a user submitting a `rsETHUnstaked` value small enough that integer division truncates to zero will have their rsETH permanently burned while receiving zero underlying assets.

## Finding Description
`minRsEthAmountToWithdraw` is a mapping that defaults to `0` for every asset: [1](#0-0) 

The input guard in both functions reduces to `rsETHUnstaked == 0` when the minimum is unset: [2](#0-1) 

The output amount is computed via integer division: [3](#0-2) 

When `rsETHUnstaked * rsETHPrice < assetPrice` (e.g., `rsETHUnstaked = 1 wei`, `rsETHPrice = 1.05e18`, `assetPrice = 1.06e18`), `underlyingToReceive` truncates to `0`.

In `instantWithdrawal`, rsETH is burned **before** any check on `assetAmountUnlocked`: [4](#0-3) 

The availability check `assetAmountUnlocked > available` evaluates `0 > X`, which is always false, so execution continues. `redeem(asset, 0)` and `_transferAsset(asset, msg.sender, 0)` are called, delivering nothing to the user while their rsETH has already been destroyed.

In `initiateWithdrawal`, rsETH is transferred to the contract and the request is stored with `expectedAssetAmount = 0`: [5](#0-4) 

During `_unlockWithdrawalRequests`, `_calculatePayoutAmount` returns `min(0, currentReturn) = 0`: [6](#0-5) 

The request is processed: `rsETHAmountToBurn += request.rsETHUnstaked` while `assetAmountToUnlock += 0`. The rsETH is burned at line 305 and `completeWithdrawal` transfers 0 assets to the user. [7](#0-6) 

## Impact Explanation
A user calling either withdrawal function with a `rsETHUnstaked` value that produces `expectedAssetAmount = 0` permanently loses their rsETH while receiving zero underlying assets. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value** (the protocol's TVL is unaffected; only the individual user's rsETH is destroyed). In realistic price conditions, the maximum loss per call is bounded to a few wei of rsETH (the threshold where truncation occurs), making the per-transaction loss trivial but real.

## Likelihood Explanation
The condition requires `minRsEthAmountToWithdraw[asset] == 0` (the default for every asset unless explicitly configured by admin) and `rsETHUnstaked` small enough that `rsETHUnstaked * rsETHPrice < assetPrice`. With rsETH prices typically close to but slightly below LST prices (e.g., `rsETHPrice = 1.05e18`, `assetPrice = 1.06e18`), only 1 wei of rsETH satisfies the truncation condition. Any unprivileged user with rsETH balance can trigger this without any special access.

## Recommendation
Add an explicit output validation immediately after computing the expected amount in both `initiateWithdrawal` and `instantWithdrawal`:

```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount == 0) revert InvalidAmountToWithdraw();
```

Additionally, `setMinRsEthAmountToWithdraw` should enforce a non-zero minimum for all supported assets at the time they are added, rather than relying on a post-deployment admin call.

## Proof of Concept
1. Admin adds stETH as a supported asset with instant withdrawal enabled; `minRsEthAmountToWithdraw[stETH]` remains `0` (default).
2. Assume `rsETHPrice = 1.05e18`, `stETHPrice = 1.06e18` (realistic: rsETH slightly trails stETH).
3. User calls `instantWithdrawal(stETH, 1, "")` (1 wei rsETH).
4. `assetAmountUnlocked = 1 * 1.05e18 / 1.06e18 = 0` (integer truncation).
5. `burnFrom(user, 1)` executes — 1 wei rsETH is permanently destroyed.
6. `0 > getAssetsAvailableForInstantWithdrawal(stETH)` is false — check passes.
7. `redeem(stETH, 0)` and `_transferAsset(stETH, user, 0)` execute — user receives nothing.

Foundry fuzz test plan: fuzz `rsETHUnstaked` over `[1, assetPrice/rsETHPrice)` with mocked oracle prices and assert that `getExpectedAssetAmount` returns 0, then confirm `instantWithdrawal` completes without reverting while burning the input rsETH and transferring 0 assets.
