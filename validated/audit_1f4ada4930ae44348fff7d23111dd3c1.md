Audit Report

## Title
Missing Zero-Address Validation on `swap()` `recipient` Allows Permanent Token Burns - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.swap()` accepts a caller-supplied `recipient` address and unconditionally transfers the output token leg to it before the callback settles the input. No guard prevents `recipient == address(0)`. On USDC/USDT (explicitly in scope), `safeTransfer` to `address(0)` succeeds and permanently burns the output, while the callback still pulls the full input from the caller.

## Finding Description
`swap()` at line 225 checks only `amountSpecified != 0` and proceeds directly to transfer output tokens to `recipient`:

```solidity
// line 225
require(amountSpecified != 0, InvalidAmount());
// ... no recipient != address(0) check ...

// zeroForOne path (line 251-254)
if (amount1Delta < 0) {
    transferToken1(recipient, uint256(-amount1Delta));
}
// !zeroForOne path (line 265-268)
if (amount0Delta < 0) {
    transferToken0(recipient, uint256(-amount0Delta));
}
```

`transferToken0`/`transferToken1` (lines 565–571) are direct wrappers around `IERC20.safeTransfer`, which adds no zero-address guard of its own. OpenZeppelin's `SafeERC20.safeTransfer` only checks the return value; it does not validate the destination address.

USDC (FiatToken) and USDT do not revert on `transfer` to `address(0)` — they credit the zero address, permanently burning the tokens. This is a well-known property of both tokens, which are explicitly in scope.

`MetricOmmSimpleRouter` passes `recipient` straight through from user-supplied params in all four entry points (`exactInputSingle` line 74, `exactInput` line 106, `exactOutputSingle` line 137, `exactOutput` line 167) with no zero-address guard, making the router a realistic integration path for this failure.

The factory does validate `adminFeeDestination` and `admin` against `address(0)` at pool-creation time (lines 548–554 of `MetricOmmPoolFactory.sol`), but this pattern was not applied to the per-call `recipient` parameter.

## Impact Explanation
If `recipient == address(0)`:
1. Pool calls `transferToken0(address(0), amount)` or `transferToken1(address(0), amount)`.
2. On USDC/USDT, `safeTransfer` to `address(0)` succeeds; tokens are credited to the zero address and permanently burned.
3. The swap callback then pulls the full input amount from the caller.
4. Net result: caller pays full input, receives zero output — complete, irreversible loss of the swap output principal.

The loss is unbounded (up to the pool's entire liquid reserve for that token leg) and irreversible. This is a direct loss of user principal, meeting the Critical/High threshold under the allowed impact gate.

## Likelihood Explanation
- Any contract integrating the pool that leaves a `recipient` variable uninitialized (Solidity default: `address(0)`) triggers this silently.
- `MetricOmmSimpleRouter` forwards `params.recipient` from user-supplied calldata without validation across all four swap entry points; a user or upstream contract that passes `address(0)` will trigger the burn.
- The condition requires no special privilege — any unprivileged caller of `swap()` or the router can trigger it, intentionally or accidentally.
- The scenario is repeatable and requires no setup beyond a normal swap call.

## Recommendation
Add a zero-address guard at the top of `swap()`, mirroring the factory pattern:

```solidity
require(recipient != address(0), ZeroRecipient());
require(amountSpecified != 0, InvalidAmount());
```

The same guard should be applied to `addLiquidity()` for the `owner` parameter, since LP shares minted to `address(0)` are also permanently lost.

## Proof of Concept
```solidity
// Misconfigured integrator or uninitialized recipient variable
pool.swap(
    address(0),    // recipient — zero address, no revert
    true,          // zeroForOne
    int128(1000),  // exact input of 1000 token0
    0,             // no price limit
    callbackData,
    ""
);
// Pool calls transferToken1(address(0), outputAmount)
// USDC/USDT safeTransfer to address(0) succeeds → tokens burned
// Callback pulls 1000 token0 from msg.sender → paid in full
// Net: caller loses 1000 token0 AND receives 0 token1
```

A Foundry fork test against a USDC/USDT pool can confirm `transfer(address(0), amount)` returns `true` and emits a `Transfer` event to `address(0)` without reverting, while the pool's `balance0Before + amount0Delta > balance0()` check passes (input was received), completing the swap with zero output delivered.
