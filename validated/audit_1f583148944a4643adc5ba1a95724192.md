Audit Report

## Title
Unprivileged Caller Can Trigger Protocol-Wide Pause via `updateRSETHPrice()` During Price Drops, Temporarily Freezing User Withdrawals — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is a public, permissionless function protected only by `whenNotPaused`. When the computed rsETH price has dropped more than `pricePercentageLimit` below `highestRsethPrice`, any external account can call it to atomically pause `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` itself. All three user-facing withdrawal paths in `LRTWithdrawalManager` are gated by `whenNotPaused`, so the freeze persists until an admin manually unpauses each contract.

## Finding Description

`updateRSETHPrice()` carries no role check — only `whenNotPaused`:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Inside `_updateRsETHPrice()`, when `newRsETHPrice < highestRsethPrice` and the drop exceeds the configured threshold, the function unconditionally pauses both downstream contracts and itself:

```solidity
// contracts/LRTOracle.sol L269-282
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

The condition is:

```solidity
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
```

For the oracle to successfully call `withdrawalManager.pause()` and `lrtDepositPool.pause()`, the oracle contract must hold `PAUSER_ROLE` on those contracts — a configuration the downside-protection code is explicitly designed to require (otherwise the branch is dead code). Once paused, all three user-facing withdrawal functions in `LRTWithdrawalManager` revert:

- `initiateWithdrawal` — `whenNotPaused` at L158
- `completeWithdrawal` — `whenNotPaused` at L183
- `instantWithdrawal` — `whenNotPaused` at L219

Recovery requires `onlyLRTAdmin` to call `unpause()` on each contract individually (L352 of `LRTWithdrawalManager.sol`, L143 of `LRTOracle.sol`). There is no time-lock or automatic recovery path.

The contrast with the upside-protection branch is telling: when the price *increases* beyond the threshold, a non-manager caller receives `PriceAboveDailyThreshold` (L263-265) — the call reverts rather than pausing. The downside branch has no equivalent gate; it pauses unconditionally regardless of who called.

## Impact Explanation

**Medium — Temporary freezing of funds.** Users with pending withdrawal requests cannot call `completeWithdrawal()` to retrieve their assets. New withdrawals via `initiateWithdrawal()` and `instantWithdrawal()` are also blocked. The freeze is not permanent (admin can unpause), but it directly matches the allowed impact class "Temporary freezing of funds" and is triggered without any capital or special role.

## Likelihood Explanation

**Medium.** The trigger condition — rsETH price dropping beyond `pricePercentageLimit` relative to `highestRsethPrice` — is realistic during EigenLayer slashing events, LST depeg events, or large coordinated withdrawals that reduce TVL. Once the on-chain price condition is met, any EOA can call `updateRSETHPrice()` in a single transaction with zero capital. A griefing actor can monitor oracle prices off-chain and call the function the moment the threshold is breached, racing legitimate callers. The attack is repeatable: after admin unpauses, if the price condition still holds, the same caller can immediately re-trigger the pause.

## Recommendation

Add an access-control modifier to `updateRSETHPrice()` (e.g., `onlyLRTOperator` or a dedicated keeper role), matching the pattern already used by `updateRSETHPriceAsManager()`. Alternatively, decouple the auto-pause trigger from the public price-update path: emit an event or set a flag when the threshold is breached, and require a privileged actor to explicitly call a separate `triggerDownsidePause()` function. The privileged `updateRSETHPriceAsManager()` path already exists and can serve as the sole entry point for price updates that may trigger a pause.

## Proof of Concept

1. Admin sets `pricePercentageLimit = 1e16` (1%) via `setPricePercentageLimit`.
2. `highestRsethPrice` is recorded at `1.05e18` after a period of rewards.
3. An EigenLayer slashing event reduces total ETH in protocol; the new computed price is `1.034e18` — a drop of ~1.52%, exceeding the 1% limit.
4. Any EOA calls `LRTOracle.updateRSETHPrice()`.
5. Inside `_updateRsETHPrice()`: `diff = 1.05e18 - 1.034e18 = 0.016e18`; `pricePercentageLimit.mulWad(highestRsethPrice) = 0.01 * 1.05e18 = 0.0105e18`; `isPriceDecreaseOffLimit = true`.
6. `lrtDepositPool.pause()` executes → `LRTDepositPool` paused.
7. `withdrawalManager.pause()` executes → `LRTWithdrawalManager` paused.
8. `_pause()` executes → `LRTOracle` paused.
9. All users calling `completeWithdrawal()`, `initiateWithdrawal()`, or `instantWithdrawal()` receive `Pausable: paused` reverts.
10. Protocol remains frozen until admin calls `unpause()` on each of the three contracts.

**Foundry fork test outline:** Deploy against a mainnet fork, set `pricePercentageLimit`, manipulate the Chainlink mock to return a price that causes `_getTotalEthInProtocol()` to fall below the threshold, call `updateRSETHPrice()` from an unprivileged address, and assert that `LRTWithdrawalManager.paused() == true` and that `completeWithdrawal()` reverts.
