Audit Report

## Title
`LRTDepositPool.getAssetCurrentLimit()` Ignores rsETH Daily Mint Cap, Returning Misleading Deposit Headroom - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getAssetCurrentLimit()` only checks the per-asset `depositLimitByAsset` cap from `LRTConfig` and never consults the independent `maxMintAmountPerDay` daily mint cap enforced by the `checkDailyMintLimit` modifier in `RSETH.mint()`. When the daily rsETH quota is exhausted, the function returns a large positive headroom value while every actual deposit call will revert with `DailyMintLimitExceeded`. The contract fails to deliver the accurate deposit capacity it promises to callers.

## Finding Description
`getAssetCurrentLimit()` at `contracts/LRTDepositPool.sol` L402–409 computes headroom solely as `depositLimitByAsset(asset) - totalAssetDeposits`:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    return lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;
}
```

The actual deposit execution path (`depositAsset()` / `depositETH()` → `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()` → `_mintRsETH()` → `RSETH.mint()`) passes through the `checkDailyMintLimit` modifier in `RSETH.sol` L42–56, which enforces a completely separate cap:

```solidity
if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
    revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
}
```

`RSETH` exposes `remainingDailyMintLimit()` at L265–272 for exactly this purpose, but `getAssetCurrentLimit()` never calls it. When the daily quota is fully consumed, `getAssetCurrentLimit()` continues to report the full per-asset headroom while every deposit reverts at the `RSETH.mint()` layer.

## Impact Explanation
The function is the canonical on-chain view for deposit capacity. When the daily cap is exhausted, it returns a materially incorrect positive value. Any smart-contract router, aggregator, or off-chain keeper that relies on it will submit transactions that revert, failing to execute intended deposits. No funds are permanently lost because the revert is clean, but the contract fails to deliver the accurate deposit headroom it promises. This maps to **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
`maxMintAmountPerDay` is an active operational control. During high-demand periods or after a single large institutional deposit, the daily quota can be fully consumed well before the 24-hour window resets. Any integrator polling `getAssetCurrentLimit()` during that window receives incorrect data. The condition is routine, predictable, and repeatable every day the quota is hit, making likelihood **Medium**.

## Recommendation
`getAssetCurrentLimit()` should incorporate the remaining rsETH daily mint headroom. The `remainingDailyMintLimit()` view already exists on `RSETH`. The fix should retrieve that value, convert it to asset units via the existing price oracle, and return the minimum of the per-asset headroom and the rsETH daily headroom:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    uint256 assetHeadroom = lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;

    address rsethToken = lrtConfig.rsETH();
    uint256 rsethHeadroom = IRSETH(rsethToken).remainingDailyMintLimit();
    uint256 rsethHeadroomInAsset = convertRsETHToAsset(asset, rsethHeadroom);

    return assetHeadroom < rsethHeadroomInAsset ? assetHeadroom : rsethHeadroomInAsset;
}
```

## Proof of Concept
1. Set `depositLimitByAsset[stETH] = 10_000e18`, `getTotalAssetDeposits(stETH) = 1_000e18`, and `maxMintAmountPerDay` fully consumed (`currentPeriodMintedAmount == maxMintAmountPerDay`).
2. Call `getAssetCurrentLimit(stETH)` → returns `9_000e18`.
3. Call `RSETH.remainingDailyMintLimit()` → returns `0`.
4. Submit `depositAsset(stETH, 100e18, 0, referral)`.
5. Execution reaches `RSETH.mint()`, `checkDailyMintLimit` fires, transaction reverts with `DailyMintLimitExceeded`.
6. `getAssetCurrentLimit()` reported `9_000e18` of available capacity; zero was actually available.

A Foundry fork test can reproduce this by: (a) deploying or forking the contracts, (b) minting rsETH up to `maxMintAmountPerDay` in a single call within the same period, (c) asserting `getAssetCurrentLimit(stETH) > 0`, and (d) asserting that `depositAsset(stETH, 1e18, 0, address(0))` reverts with `DailyMintLimitExceeded`.
