### Title
Dual-Role Credential Registration Allows Same Key to Vote as Both DRep and Committee Hot Key — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

The `GOVCERT` transition rule in Conway/Dijkstra performs no cross-role uniqueness check when registering a DRep credential or when authorizing a committee hot key. The same underlying key hash can therefore be simultaneously present in `vsDRepsL` (as a `Credential DRepRole`) and in `vsCommitteeStateL` (as a `Credential HotCommitteeRole`). Because the `GOV` rule validates each voter role against a separate map, a single key can cast a legally-accepted vote as both a `DRepVoter` and a `CommitteeVoter` on the same governance action, inflating both vote tallies and potentially satisfying both ratification thresholds with fewer distinct principals than the protocol intends.

---

### Finding Description

**Root cause — `ConwayRegDRep` (GOVCERT rule)**

When a `ConwayRegDRep` certificate is processed, the only guard is:

```haskell
Map.notMember cred (certState ^. certVStateL . vsDRepsL)
  ?! (injectFailure . ConwayDRepAlreadyRegistered) cred
```

There is no check that `cred` (as a raw key hash) is not already authorized as a committee hot key in `vsCommitteeStateL`. [1](#0-0) 

**Root cause — `ConwayAuthCommitteeHotKey` (GOVCERT rule)**

When a committee hot key is authorized, `checkAndOverwriteCommitteeMemberState` only verifies that the *cold* credential has not previously resigned and is a known member. There is no check that the supplied `hotCred` is not already registered as a DRep in `vsDRepsL`.

```haskell
ConwayAuthCommitteeHotKey coldCred hotCred ->
  checkAndOverwriteCommitteeMemberState coldCred $ CommitteeHotCredential hotCred
``` [2](#0-1) 

**How the dual role is exploited in the GOV rule**

Voter validation in `conwayGovTransition` resolves each voter against a *separate* map:

```haskell
internVoter = \case
  CommitteeVoter hotCred -> CommitteeVoter <$> internSet hotCred knownCommitteeMembers
  DRepVoter cred         -> DRepVoter      <$> internMap cred    knownDReps
  StakePoolVoter poolId  -> StakePoolVoter <$> internMap poolId  knownStakePools
``` [3](#0-2) 

Because `Credential HotCommitteeRole` and `Credential DRepRole` are distinct Haskell types (phantom role parameter), `CommitteeVoter (KeyHashObj kh)` and `DRepVoter (KeyHashObj kh)` are distinct `Voter` constructors and therefore distinct keys in `VotingProcedures`. [4](#0-3) 

A single transaction may therefore include both entries, and both pass `failOnNonEmpty unknownVoters` independently. [5](#0-4) 

The same flaw is inherited by the Dijkstra era, which reuses `conwayGovCertTransition` via `InjectRuleFailure`. [6](#0-5) 

---

### Impact Explanation

A committee member who also registers the same key hash as a DRep can:

1. Accumulate delegated stake to that DRep credential.
2. Submit a `VotingProcedures` map containing both `CommitteeVoter (KeyHashObj kh)` and `DRepVoter (KeyHashObj kh)` for the same `GovActionId`.
3. Have both votes accepted, contributing simultaneously to the committee acceptance ratio and the DRep acceptance ratio.

If the attacker controls enough delegated stake and holds a committee seat, they can single-handedly satisfy both the committee threshold and the DRep threshold for actions such as `TreasuryWithdrawals`, `ParameterChange`, `NewConstitution`, or `HardForkInitiation`. This constitutes an **unauthorized governance action being enacted** — matching the Critical impact tier ("Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted").

---

### Likelihood Explanation

The attacker must:
1. Hold (or obtain) a committee cold key that is either a current member or listed in a pending `UpdateCommittee` proposal — achievable by any legitimate committee member.
2. Register the corresponding hot key hash as a DRep and attract delegators — achievable by any unprivileged transaction sender.

No privileged access, key leakage, or consensus majority is required. The entry path is entirely through normal certificate and voting transactions submitted by an unprivileged user who happens to be (or becomes) a committee member.

---

### Recommendation

In `conwayGovCertTransition`, add cross-role guards:

1. **In `ConwayRegDRep`**: reject if the credential's key hash is already authorized as a committee hot key in `vsCommitteeStateL`.
2. **In `ConwayAuthCommitteeHotKey`**: reject if `hotCred`'s key hash is already present in `vsDRepsL`.

Analogous guards should be added in the Dijkstra era's `GOVCERT` rule. [7](#0-6)

### Proof of Concept

```
-- Setup
kh  ← freshKeyHash                          -- attacker's key

-- Step 1: register kh as a DRep
submit RegDRepTxCert (KeyHashObj kh) deposit SNothing
-- Accepted: vsDRepsL now contains KeyHashObj kh :: Credential DRepRole

-- Step 2: elect a committee cold key (attacker controls coldKH)
submit UpdateCommittee ... [(KeyHashObj coldKH, expiry)] ...
-- Enacted after ratification

-- Step 3: authorize kh as the hot key for coldKH
submit AuthCommitteeHotKeyTxCert (KeyHashObj coldKH) (KeyHashObj kh)
-- Accepted: vsCommitteeStateL now maps coldKH → CommitteeHotCredential (KeyHashObj kh)
-- No check that (KeyHashObj kh) is already in vsDRepsL

-- Step 4: attract delegators to the DRep
-- (delegators submit DelegVote (DRepCredential (KeyHashObj kh)) certs)

-- Step 5: vote on a governance action gaId
submit VotingProcedures {
  CommitteeVoter (KeyHashObj kh) → { gaId → VoteYes },   -- accepted as committee vote
  DRepVoter     (KeyHashObj kh) → { gaId → VoteYes }    -- accepted as DRep vote
}
-- Both votes pass internVoter independently.
-- The same key contributes to both committeeAcceptedRatio and dRepAcceptedRatio,
-- potentially satisfying both thresholds and enacting the governance action.
```
