Audit Report

## Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Additional Fees â€” (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` (each a `uint16`, max 65 535 â‰ˆ 6.55 %) directly to the pool with no check against `maxAdminSpreadFeeE6`. Because these per-bin fees are additive on top of the oracle-derived base spread fee and are exclusively settable by the pool admin, the factory owner's governance cap on admin-controlled fee extraction is silently bypassed for every swap that crosses an affected bin.

## Finding Description
`setPoolAdminFees` correctly enforces the cap:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

The parallel path `setPoolBinAdditionalFees` performs no such check:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

In the pool's swap execution the per-bin fee is added directly on top of `baseFeeX64` (the oracle-derived spread):

```solidity
// MetricOmmPool.sol L910, L999
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```

A pool admin can therefore call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` and impose an additional 6.55 % fee on every swap through that bin, regardless of what `maxAdminSpreadFeeE6` is set to by the factory owner. The only bound is `uint16.max`; there is no factory-level cap check anywhere in the call path.

## Impact Explanation
Every trader swapping through an affected bin pays a higher effective fee than the factory owner's governance cap permits. The excess fee accrues to LPs in that bin; if the pool admin is also an LP (a common and expected configuration), they directly extract value beyond the cap. This is a concrete admin-boundary break: the factory owner's `maxAdminSpreadFeeE6` mechanism fails to bound all admin-controlled fee extraction paths, violating the protocol's stated governance model.

## Likelihood Explanation
The pool admin is a single EOA or multisig that can call `setPoolBinAdditionalFees` at any time with no timelock. The attack requires no special market conditions, no external oracle manipulation, and no cooperation from other parties. It is repeatable on every block and affects every swap through the configured bin. Any pool whose admin is adversarial or compromised is immediately exploitable.

## Recommendation
Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (uint256(addFeeBuyE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (uint256(addFeeSellE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, introduce a dedicated `maxAdminBinFeeE6` state variable so the factory owner can set a separate ceiling for per-bin fees, and enforce it here.
