### Title
Harvester Can Unconditionally Redirect Farmer Block Reward via Unenforced `farmer_reward_address_override` - (File: chia/farmer/farmer_api.py)

### Summary
The CHIP-22 third-party harvester protocol allows a harvester to set `farmer_reward_address_override` in `NewProofOfSpace` and `RespondSignatures` messages. The farmer's fee-quality check in `notify_farmer_reward_taken_by_harvester_as_fee()` only **logs a warning** when the threshold is violated — it never rejects or suppresses the override. As a result, any connected harvester can unconditionally redirect the farmer's XCH block reward to an arbitrary address.

### Finding Description

`NewProofOfSpace` carries an optional `farmer_reward_address_override` field: [1](#0-0) 

When the farmer receives this message in `new_proof_of_space()`, it calls `notify_farmer_reward_taken_by_harvester_as_fee()` if the override is set: [2](#0-1) 

That function checks the fee quality against the harvester-supplied threshold and emits a log warning on violation — but **does not return an error, does not suppress the override, and does not abort processing**: [3](#0-2) 

Execution continues normally. Later, in `_process_respond_signatures()`, the farmer unconditionally substitutes the harvester-supplied address for its own `farmer_target`: [4](#0-3) 

This overridden address is then placed directly into `DeclareProofOfSpace` as the block reward destination: [5](#0-4) 

There is no cryptographic binding between the harvester's identity and the override address, and no enforcement path that rejects an illegitimate override.

### Impact Explanation

A malicious third-party harvester (as introduced by CHIP-22) can set `farmer_reward_address_override` to any puzzle hash it controls. Every block the farmer wins while connected to that harvester will pay the 1.75 XCH farmer reward to the attacker's address instead of the farmer's configured `farmer_target`. This is a direct, permanent loss of XCH with no on-chain recourse. It matches the allowed impact: **unauthorized payout redirection affecting XCH**.

### Likelihood Explanation

CHIP-22 explicitly anticipates third-party harvesters that are not fully trusted. A farmer who delegates plot-farming to an external harvester service is exposed to this attack by design. The attacker only needs a valid harvester connection — no key material, no admin access, and no cryptographic break is required. The log warning is easily missed in production.

### Recommendation

In `new_proof_of_space()`, after calling `notify_farmer_reward_taken_by_harvester_as_fee()`, the farmer should **reject the proof** (return early without proceeding to `RequestSignatures`) when the fee quality check fails — i.e., when `fee_quality > fee_threshold` or when `fee_info` is absent. The current code path at: [2](#0-1) 

should be extended to abort processing if `notify_farmer_reward_taken_by_harvester_as_fee` determines the override is illegitimate. Additionally, `_process_respond_signatures()` should independently verify that any `farmer_reward_address_override` in `RespondSignatures` was pre-authorized during the `NewProofOfSpace` phase before applying it.

### Proof of Concept

1. Attacker operates a harvester service and convinces a farmer to connect to it (standard CHIP-22 third-party harvester setup).
2. When the harvester finds a valid proof of space, it sends `NewProofOfSpace` with `farmer_reward_address_override` set to the attacker's puzzle hash and `fee_info` set to `applied_fee_threshold=0xFFFFFFFF` (maximum, always passes) or omitted entirely.
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`:
   - If `fee_info` is absent: logs `"Harvester illegitimately took reward by failing to provide its fee rate"` — no abort.
   - If `fee_info.applied_fee_threshold` is set too high: logs `"Invalid fee threshold"` — no abort.
4. The farmer proceeds to `RequestSignatures`, receives `RespondSignatures` (also carrying `farmer_reward_address_override`), and at line 917–918 substitutes the attacker's address for `self.farmer.farmer_target`.
5. `DeclareProofOfSpace` is sent to the full node with `farmer_reward_address` = attacker's puzzle hash.
6. The block is accepted; the 1.75 XCH farmer reward is paid to the attacker. [6](#0-5)
