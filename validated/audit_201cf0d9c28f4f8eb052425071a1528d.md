Audit Report

## Title
`completeWithdrawal` and `unlockQueue` Blocked by Pause Freezes In-Flight User rsETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` immediately transfers the user's rsETH into the contract, but both `completeWithdrawal` and `unlockQueue` are gated by `whenNotPaused`. If the contract is paused after a user has deposited rsETH but before their withdrawal is completed, their rsETH is frozen with no cancel or recovery path available until an admin manually unpauses.

## Finding Description
The withdrawal lifecycle is a two-step process. In step one, `initiateWithdrawal` (itself also `whenNotPaused`) pulls rsETH from the user into the contract:

```solidity
// LRTWithdrawalManager.sol L166
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
```

In step two, the operator calls `unlockQueue` (L279: `whenNotPaused`) to process the queue, and the user calls `completeWithdrawal` (L183: `whenNotPaused`) to receive their LST/ETH. Both are blocked when the contract is paused.

There is no `cancelWithdrawal` function anywhere in the contract. The only exit paths for in-flight rsETH are `completeWithdrawal` (L183) and `completeWithdrawalForUser` (L199) — both blocked by `whenNotPaused`. The `pause()` function is callable by the lower-threshold `PAUSER_ROLE` (L347), while `unpause()` requires `onlyLRTAdmin` (L352). A user who called `initiateWithdrawal` while the contract was live has no recourse until the admin manually unpauses.

## Impact Explanation
**Medium. Temporary freezing of funds.** Any user who has called `initiateWithdrawal` and had their rsETH transferred into `LRTWithdrawalManager` cannot complete the withdrawal or recover their rsETH for the entire duration of the pause. The freeze is bounded only by admin responsiveness; if the pause is extended (e.g., during a prolonged security investigation), users are locked out of their funds for an indeterminate period.

## Likelihood Explanation
**Low.** The `PAUSER_ROLE` is a distinct, lower-threshold role designed for reactive use in security incidents. A realistic scenario: a suspected exploit triggers an emergency pause; the `PAUSER_ROLE` holder pauses the contract without accounting for users who have already surrendered their rsETH and now have no recourse. No malicious intent or collusion is required — the harm is a side effect of a legitimate security action. The scenario is repeatable any time the contract is paused while withdrawal requests are in-flight.

## Recommendation
Remove `whenNotPaused` from `completeWithdrawal` and `completeWithdrawalForUser`. Users who have already transferred rsETH into the contract should always be able to claim assets they are owed. Alternatively, add a `cancelWithdrawal` function that returns the user's rsETH when the contract is paused, mirroring the fix applied to the reference `CollateralJoin` vulnerability. Keeping `whenNotPaused` on `initiateWithdrawal` and `unlockQueue` is sufficient to halt new activity during an emergency without stranding already-committed user funds.

## Proof of Concept
1. User calls `initiateWithdrawal(stETH, 1e18, "")` while contract is unpaused → 1 rsETH transferred from user to `LRTWithdrawalManager` (L166).
2. `PAUSER_ROLE` calls `LRTWithdrawalManager.pause()` (L347–349).
3. Operator attempts `unlockQueue(stETH, ...)` → reverts: `"Pausable: paused"` (L279). Request remains locked.
4. User calls `completeWithdrawal(stETH, "")` → reverts: `"Pausable: paused"` (L183).
5. User's 1 rsETH remains in `LRTWithdrawalManager` with no recovery path until admin calls `unpause()` (L352).

**Foundry test plan:** Deploy `LRTWithdrawalManager` on a local fork, grant `PAUSER_ROLE` to a test address, call `initiateWithdrawal` as a user, call `pause()`, then assert that both `completeWithdrawal` and `unlockQueue` revert with `"Pausable: paused"`, and that the user's rsETH balance remains in the contract.
