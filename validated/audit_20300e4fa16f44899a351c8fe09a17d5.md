### Title
Farmer Does Not Verify `foliage_block_data_hash` Matches `foliage_block_data` in `RequestSignedValues`, Enabling Reward Diversion by a Malicious Full Node — (`File: chia/farmer/farmer_api.py`)

---

### Summary

The `FarmerAPI.request_signed_values` handler accepts a `RequestSignedValues` message that carries both a pre-computed hash (`foliage_block_data_hash`) and the raw data it supposedly hashes (`foliage_block_data`). The farmer forwards the hash to the harvester as the message to sign, and forwards the raw data as CHIP-22 source data for inspection — but **never verifies that `foliage_block_data.get_hash() == foliage_block_data_hash`**. A malicious full node can exploit this by sending a mismatched pair: legitimate-looking source data (with the farmer's real reward address) paired with a hash that is actually the hash of attacker-controlled data (with the attacker's reward address). The CHIP-22-aware harvester inspects the source data, sees the legitimate address, and signs the hash — which is actually the hash of the attacker's block. The farmer also co-signs the same hash. The resulting aggregate signature is valid for a block that pays rewards to the attacker.

---

### Finding Description

`RequestSignedValues` is defined in `chia/protocols/farmer_protocol.py` and carries two correlated fields:

```python
class RequestSignedValues(Streamable):
    quality_string: bytes32
    foliage_block_data_hash: bytes32          # hash to sign
    foliage_transaction_block_hash: bytes32   # hash to sign
    foliage_block_data: FoliageBlockData | None = None          # raw data (CHIP-22)
    foliage_transaction_block_data: FoliageTransactionBlock | None = None
    rc_block_unfinished: RewardChainBlockUnfinished | None = None
``` [1](#0-0) 

In `FarmerAPI.request_signed_values`, the farmer passes the hash as the signing target and the raw data as CHIP-22 source data, with **no cross-validation**:

```python
request = harvester_protocol.RequestSignatures(
    plot_identifier,
    challenge_hash,
    sp_hash,
    [full_node_request.foliage_block_data_hash, full_node_request.foliage_transaction_block_hash],
    message_data=message_data,   # contains foliage_block_data verbatim
    ...
)
``` [2](#0-1) 

The missing guard is the check `full_node_request.foliage_block_data.get_hash() == full_node_request.foliage_block_data_hash`. This check is performed only in a test interceptor, not in production code: [3](#0-2) 

After the harvester signs, the farmer's `_process_respond_signatures` also co-signs the same hash without any data verification: [4](#0-3) 

The full node's `signed_values` handler verifies the returned signature against its own stored candidate block — which the malicious full node itself constructed with the attacker's reward address: [5](#0-4) 

---

### Impact Explanation

A malicious full node can redirect all farming rewards from a connected farmer to an attacker-controlled puzzle hash. The attack undermines the CHIP-22 source data inspection mechanism entirely: the harvester inspects source data that appears legitimate but signs a hash corresponding to a different block. This is a **High** impact: unauthorized payout redirection of XCH farming rewards via bypass of farmer signing authorization. [6](#0-5) 

---

### Likelihood Explanation

Any operator who connects their farmer to an external or third-party full node is exposed. This is a common configuration. The attacker only needs to run a full node that the farmer connects to — no key compromise, no cryptographic break, no admin access required. The malicious full node simply crafts a `RequestSignedValues` with a mismatched hash/data pair. [7](#0-6) 

---

### Recommendation

In `FarmerAPI.request_signed_values`, before forwarding to the harvester, add an explicit check:

```python
if full_node_request.foliage_block_data is not None:
    if full_node_request.foliage_block_data.get_hash() != full_node_request.foliage_block_data_hash:
        self.farmer.log.error("foliage_block_data hash mismatch in RequestSignedValues — rejecting")
        return None
if full_node_request.foliage_transaction_block_data is not None:
    if full_node_request.foliage_transaction_block_data.get_hash() != full_node_request.foliage_transaction_block_hash:
        self.farmer.log.error("foliage_transaction_block_data hash mismatch — rejecting")
        return None
```

This mirrors the validation already performed in the test interceptor and closes the gap between the hash used for signing and the data presented for inspection. [8](#0-7) 

---

### Proof of Concept

1. Attacker runs a malicious full node and waits for a farmer to connect.
2. When the farmer submits a valid `DeclareProofOfSpace`, the malicious full node constructs two `FoliageBlockData` objects:
   - `real_data`: with `farmer_reward_puzzle_hash = attacker_ph`
   - `fake_data`: with `farmer_reward_puzzle_hash = farmer_ph` (the farmer's legitimate address)
3. The malicious full node sends `RequestSignedValues` with:
   - `foliage_block_data_hash = real_data.get_hash()` (hash of attacker's block)
   - `foliage_block_data = fake_data` (legitimate-looking source data)
4. The farmer's `request_signed_values` passes both to the harvester without checking `fake_data.get_hash() == real_data.get_hash()`.
5. A CHIP-22-aware harvester inspects `fake_data`, sees the farmer's legitimate reward address, and signs `real_data.get_hash()`.
6. The farmer co-signs `real_data.get_hash()`.
7. The malicious full node uses the aggregate signature to produce a valid block paying rewards to `attacker_ph`.
8. The farmer's reward is permanently diverted for every block won while connected to the malicious node. [8](#0-7) [7](#0-6)

    @metadata.request()
    async def request_signed_values(self, full_node_request: farmer_protocol.RequestSignedValues) -> Message | None:
        if full_node_request.quality_string not in self.farmer.quality_str_to_identifiers:
            self.farmer.log.error(f"Do not have quality string {full_node_request.quality_string}")
            return None

        (plot_identifier, challenge_hash, sp_hash, node_id) = self.farmer.quality_str_to_identifiers[
            full_node_request.quality_string
        ]

        message_data: list[SignatureRequestSourceData | None] | None = None

        if full_node_request.foliage_block_data is not None:
            message_data = [
                SignatureRequestSourceData(
                    uint8(SigningDataKind.FOLIAGE_BLOCK_DATA), bytes(full_node_request.foliage_block_data)
                ),
                (
                    None
                    if full_node_request.foliage_transaction_block_data is None
                    else SignatureRequestSourceData(
                        uint8(SigningDataKind.FOLIAGE_TRANSACTION_BLOCK),
                        bytes(full_node_request.foliage_transaction_block_data),
                    )
                ),
            ]

        request = harvester_protocol.RequestSignatures(
            plot_identifier,
            challenge_hash,
            sp_hash,
            [full_node_request.foliage_block_data_hash, full_node_request.foliage_transaction_block_hash],
            message_data=message_data,
            rc_block_unfinished=full_node_request.rc_block_unfinished,
        )

        response = await self.farmer.server.call_api_of_specific(HarvesterAPI.request_signatures, request, node_id)
        if response is None or not isinstance(response, harvester_protocol.RespondSignatures):
            self.farmer.log.error(f"Invalid response from harvester {node_id} for request_signatures: {response}")
            return None

        # Use the same processing as for unsolicited respond signature requests
        signed_values = self._process_respond_signatures(response)
        if signed_values is None:
            return None
        assert isinstance(signed_values, SignedValues)

        return make_msg(ProtocolMessageTypes.signed_values, signed_values)
