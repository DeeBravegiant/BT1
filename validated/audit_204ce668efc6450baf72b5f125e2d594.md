### Title
Pool Operator Can Indefinitely Reset `LEAVING_POOL` Timelock via Permissionless Absorb Spend - (File: chia/pools/pool_puzzles.py)

### Summary

The pool singleton's absorb-reward spend path is permissionless (requires no signature) and is valid while the singleton is in `LEAVING_POOL` (waiting room) state. Each absorb spend destroys the current singleton coin and recreates it at the current block height. Because the pool waiting room puzzle enforces exit via `ASSERT_HEIGHT_RELATIVE`, the countdown is measured from the **new** coin's creation height — not the original `LEAVING_POOL` coin's height. A pool operator can therefore repeatedly submit absorb spends just before the user's exit window opens, resetting the `relative_lock_height` countdown each time and trapping the user in the pool indefinitely.

### Finding Description

When a user calls `self_pool()` or `join_pool()` to leave a pool, the singleton transitions to `LEAVING_POOL` state. The pool waiting room puzzle (`POOL_WAITINGROOM_INNERPUZ`) enforces the exit delay with `ASSERT_HEIGHT_RELATIVE`, which is relative to the **current singleton coin's creation height**. This is confirmed by the `ASSERT_HEIGHT_RELATIVE_FAILED` error raised in the lifecycle test when an exit is attempted too early. [1](#0-0) 

The `create_absorb_spend` function in `chia/pools/pool_puzzles.py` constructs a spend bundle that spends the current singleton coin and recreates it as a new coin (new coin ID, new creation height). Critically, this spend requires **no signature** — it is submitted with `G2Element()` (empty aggregated signature): [2](#0-1) 

The lifecycle test explicitly demonstrates that an absorb spend is accepted while the singleton is in the waiting room state: [3](#0-2) 

After the absorb, the singleton coin is recreated at the absorb block's height. The `ASSERT_HEIGHT_RELATIVE` countdown now runs from this new height, not from the original `LEAVING_POOL` block. The wallet's own `new_peak` logic also recalculates the exit height from `tip_height` (the height of the last singleton spend), which is updated by the absorb: [4](#0-3) 

The `join_pool` and `self_pool` functions similarly use `history[-1][0]` (the last singleton spend height) to gate the exit: [5](#0-4) [6](#0-5) 

### Impact Explanation

A pool operator can submit a permissionless absorb spend at block `H + relative_lock_height - 1` (one block before the user's exit window opens). This recreates the singleton at height `H + relative_lock_height - 1`, resetting the countdown to `H + 2·relative_lock_height - 1`. The operator can repeat this every time a new farming reward arrives (which happens continuously), permanently preventing the user from leaving the pool. The user's farming rewards continue flowing to the pool's `target_puzzle_hash` indefinitely. This constitutes unauthorized singleton state control and payout redirection — a High-severity impact under the allowed scope.

### Likelihood Explanation

The absorb spend path is permissionless by design (the pool must be able to absorb rewards without the farmer's cooperation). Any pool operator running standard pool software has the capability and economic incentive to exploit this to retain farmers. The attack requires no special access, no leaked keys, and no cryptographic break — only the ability to submit valid on-chain transactions, which any pool operator already does routinely.

### Recommendation

Replace `ASSERT_HEIGHT_RELATIVE` in the pool waiting room puzzle with `ASSERT_HEIGHT_ABSOLUTE`, anchoring the exit height to the block at which `LEAVING_POOL` was entered (stored in the singleton's state or derived from the spend history). This ensures that absorb spends — which recreate the singleton coin — cannot advance the reference point for the timelock. Alternatively, the absorb path in the waiting room puzzle should be modified to carry forward the original `LEAVING_POOL` creation height as an absolute exit height, immune to coin recreation.

### Proof of Concept

1. User calls `self_pool()` at block **H** with `relative_lock_height = N`. Singleton enters `LEAVING_POOL` state; the new singleton coin is created at height H.
2. On-chain, exit requires `current_block >= H + N` (enforced by `ASSERT_HEIGHT_RELATIVE`).
3. At block **H + N − 1**, the pool operator constructs a valid absorb spend using `create_absorb_spend` against any pending farming reward coin at `p2_singleton_puzzle_hash`. No signature is needed.
4. The absorb spend is broadcast and confirmed. The singleton coin is destroyed and recreated at height **H + N − 1**.
5. The `ASSERT_HEIGHT_RELATIVE` countdown now requires `current_block >= (H + N − 1) + N = H + 2N − 1`.
6. The wallet's `new_peak` recalculates `leave_height = tip_height + relative_lock_height = (H + N − 1) + N`, so it will not submit the exit transaction until block `H + 2N + 1` (with the +2 buffer).
7. The pool operator repeats step 3–6 each time a new farming reward arrives, indefinitely resetting the countdown and trapping the user in the pool. [7](#0-6) [8](#0-7)

def create_absorb_spend(
    last_coin_spend: CoinSpend,
    current_state: PoolState,
    launcher_coin: Coin,
    height: uint32,
    genesis_challenge: bytes32,
    delay_time: uint64,
    delay_ph: bytes32,
) -> list[CoinSpend]:
    inner_puzzle: Program = pool_state_to_inner_puzzle(
        current_state, launcher_coin.name(), genesis_challenge, delay_time, delay_ph
    )
    reward_amount: uint64 = calculate_pool_reward(height)
    if is_pool_member_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol: Program = Program.to([reward_amount, height])
    elif is_pool_waitingroom_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, destination_puzhash, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol = Program.to([0, reward_amount, height])
    else:
        raise ValueError
    # full sol = (parent_info, my_amount, inner_solution)
    coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(last_coin_spend)
    assert coin is not None

    if coin.parent_coin_info == launcher_coin.name():
        parent_info: Program = Program.to([launcher_coin.parent_coin_info, launcher_coin.amount])
    else:
        p = Program.from_bytes(bytes(last_coin_spend.puzzle_reveal))
        last_coin_spend_inner_puzzle: Program | None = get_inner_puzzle_from_puzzle(p)
        assert last_coin_spend_inner_puzzle is not None
        parent_info = Program.to(
            [
                last_coin_spend.coin.parent_coin_info,
                last_coin_spend_inner_puzzle.get_tree_hash(),
                last_coin_spend.coin.amount,
            ]
        )
    full_solution: SerializedProgram = SerializedProgram.to([parent_info, last_coin_spend.coin.amount, inner_sol])
    full_puzzle: SerializedProgram = create_full_puzzle(inner_puzzle, launcher_coin.name()).to_serialized()
    assert coin.puzzle_hash == full_puzzle.get_tree_hash()

    reward_parent: bytes32 = pool_parent_id(height, genesis_challenge)
    p2_singleton_puzzle = create_p2_singleton_puzzle(
        SINGLETON_MOD_HASH, launcher_coin.name(), delay_time, delay_ph
    ).to_serialized()
    reward_coin: Coin = Coin(reward_parent, p2_singleton_puzzle.get_tree_hash(), reward_amount)
    p2_singleton_solution = SerializedProgram.to([inner_puzzle.get_tree_hash(), reward_coin.name()])
    assert p2_singleton_puzzle.get_tree_hash() == reward_coin.puzzle_hash
    assert full_puzzle.get_tree_hash() == coin.puzzle_hash
    assert get_inner_puzzle_from_puzzle(Program.from_bytes(bytes(full_puzzle))) is not None

    coin_spends = [
        CoinSpend(coin, full_puzzle, full_solution),
        CoinSpend(reward_coin, p2_singleton_puzzle, p2_singleton_solution),
    ]
    return coin_spends
