Audit Report

## Title
Daily Fee Mint Hard Revert Permanently Blocks Price Updates and Freezes Unclaimed Yield - (File: contracts/LRTOracle.sol)

## Summary
`_checkAndUpdateDailyFeeMintLimit` unconditionally reverts with `DailyFeeMintLimitExceeded` once the accumulated fee for the current 24-hour period reaches `maxFeeMintAmountPerDay`. Because this check is embedded inside `_updateRsETHPrice()`, every subsequent call to both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` reverts for the remainder of the day, freezing the rsETH price and permanently forfeiting all protocol fee yield that would have accrued during the blocked window.

## Finding Description
`_checkAndUpdateDailyFeeMintLimit` at line 205–206 performs a hard revert when `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`:

```solidity
// contracts/LRTOracle.sol L205-206
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
}
```

This function is called unconditionally from `_updateRsETHPrice()` at line 303 (and also at line 310 with `feeAmount = 0` when there is no fee, though the zero-fee path cannot trigger the revert). The `rsETHPrice` write at line 313 and the `highestRsethPrice` write at line 295 both occur **after** the revert point, so neither is updated when the revert fires.

Both public entry points call `_updateRsETHPrice()` with no additional logic:
- `updateRSETHPrice()` (L87–89): `public whenNotPaused`
- `updateRSETHPriceAsManager()` (L94–96): `external onlyLRTManager`

Neither provides a bypass around `_checkAndUpdateDailyFeeMintLimit`. Once the cap is hit, both revert identically until `block.timestamp >= feePeriodStartTime + 1 days` triggers the reset at line 199–202.

The protocol fee rsETH that would have been minted to the treasury for the remainder of the blocked day is permanently lost — the period resets to zero, so there is no mechanism to retroactively mint the skipped fees.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

All protocol fee yield (rsETH minted to treasury) that would have accrued from the moment the cap is hit until the 24-hour period resets is permanently forfeited. The treasury cannot recover this yield after the reset because `currentPeriodMintedFeeAmount` is zeroed and the prior period's unclaimed fees are discarded. Additionally, `rsETHPrice` and `highestRsethPrice` remain stale for the blocked window, affecting any protocol component that reads `rsETHPrice` (deposits, withdrawals, cross-chain rate pushes).

## Likelihood Explanation
`maxFeeMintAmountPerDay` is set by the manager via `setMaxFeeMintAmountPerDay`. If the cap is set conservatively relative to TVL, or if TVL grows rapidly in a single day (large deposits, staking reward accrual), cumulative fee mints across multiple `updateRSETHPrice()` calls within the same period can exhaust the cap. This is a normal operational scenario requiring no attacker — any caller of the public `updateRSETHPrice()` function can trigger the final cap-exceeding call. The likelihood increases monotonically with TVL growth.

## Recommendation
Replace the hard revert with a cap-and-continue pattern. Return the capped fee amount from `_checkAndUpdateDailyFeeMintLimit` and use it for the actual mint, so the price update always proceeds:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal returns (uint256 cappedFee) {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }
    uint256 remaining = maxFeeMintAmountPerDay > currentPeriodMintedFeeAmount
        ? maxFeeMintAmountPerDay - currentPeriodMintedFeeAmount
        : 0;
    cappedFee = feeAmount > remaining ? remaining : feeAmount;
    currentPeriodMintedFeeAmount += cappedFee;
}
```

Then in `_updateRsETHPrice()`, use the returned `cappedFee` for the mint instead of the original `rsethAmountToMintAsProtocolFee`, ensuring the price update always completes regardless of fee cap state.

## Proof of Concept
1. Manager sets `maxFeeMintAmountPerDay = X` rsETH.
2. During a high-yield day, `updateRSETHPrice()` is called repeatedly. After several calls, `currentPeriodMintedFeeAmount` reaches `X`.
3. TVL continues to grow. The next call computes `rsethAmountToMintAsProtocolFee > 0` at line 301.
4. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` evaluates line 205: `X + rsethAmountToMintAsProtocolFee > X` → reverts with `DailyFeeMintLimitExceeded`.
5. The revert unwinds `_updateRsETHPrice()`: `rsETHPrice` (line 313) and `highestRsethPrice` (line 295) are **not** written.
6. Every subsequent call to `updateRSETHPrice()` or `updateRSETHPriceAsManager()` reverts identically until `block.timestamp >= feePeriodStartTime + 1 days`.
7. At period reset, `currentPeriodMintedFeeAmount` is zeroed (line 200) — all treasury fee yield from the blocked window is permanently lost.

**Foundry test plan:** Deploy `LRTOracle` on a fork, set `maxFeeMintAmountPerDay` to a small value, simulate TVL growth to exhaust the cap, then assert that `updateRSETHPrice()` reverts and that `rsETHPrice` remains stale. Advance time by less than 24 hours and confirm the revert persists. Advance past 24 hours and confirm the reset allows one successful call, with no retroactive fee minting for the skipped period.
