### Title
Wrong In-Memory Cache Namespace Used in `add_interested_puzzle_hashes` Corrupts Wallet Coin Interest Tracking - (File: chia/wallet/wallet_state_manager.py)

### Summary

`WalletStateManager.add_interested_puzzle_hashes` writes puzzle-hash → wallet-id mappings into `self.interested_coin_cache` (the coin-ID namespace) instead of `self.interested_ph_cache` (the puzzle-hash namespace). This is the direct Chia analog of the external report's wrong-namespace bug: a function that handles one data type silently writes into the storage namespace reserved for a different data type.

### Finding Description

`WalletStateManager` declares two distinct in-memory caches:

```python
interested_ph_cache: dict[bytes32, list[int]] = {}   # puzzle_hash → wallet_ids
interested_coin_cache: dict[bytes32, list[int]] = {}  # coin_id    → wallet_ids
```

`add_interested_coin_ids` correctly writes into `interested_coin_cache`. However, `add_interested_puzzle_hashes` also writes into `interested_coin_cache` instead of `interested_ph_cache`:

```python
async def add_interested_puzzle_hashes(self, puzzle_hashes: list[bytes32], wallet_ids: list[int]) -> None:
    for puzzle_hash in puzzle_hashes:
        if puzzle_hash in self.interested_coin_cache:          # ← wrong cache
            wallet_ids_to_add = list({w for w in wallet_ids if w not in self.interested_coin_cache[puzzle_hash]})
            self.interested_coin_cache[puzzle_hash].extend(wallet_ids_to_add)  # ← wrong cache
        else:
            self.interested_coin_cache[puzzle_hash] = list(set(wallet_ids))    # ← wrong cache
``` [1](#0-0) 

The two caches are defined as separate namespaces: [2](#0-1) 

The correct parallel function `add_interested_coin_ids` uses `interested_coin_cache` for coin IDs: [3](#0-2) 

### Impact Explanation

Two concrete consequences:

**1. `interested_ph_cache` is never populated.** Any hot-path code that reads `interested_ph_cache` to resolve a puzzle hash to a wallet ID will always get a cache miss. Puzzle-hash interest registered via `add_interested_puzzle_hashes` (called for pool wallet p2-singleton puzzle hashes, DL wallet puzzle hashes, CAT puzzle hashes, etc.) is invisible to those readers, causing coins arriving at those puzzle hashes to be silently dropped from wallet tracking — corrupting wallet sync state and coin records.

**2. `interested_coin_cache` is polluted with puzzle-hash keys.** `_add_coin_states` queries `interested_coin_cache` by coin name to decide whether a REMOTE wallet should claim a coin:

```python
cache = self.get_interested_coin_cache()
if coin_name in cache:
    ...
    wallet_identifier = WalletIdentifier(remote_wallet.id(), WalletType.REMOTE)
``` [4](#0-3) 

Puzzle hashes injected into this cache can collide with coin IDs (both are `bytes32`), causing a coin whose ID matches a stored puzzle hash to be misattributed to a REMOTE wallet, corrupting its lineage and ownership record.

### Likelihood Explanation

`add_interested_puzzle_hashes` is called on every pool wallet creation, DL wallet creation, and CAT wallet setup — all normal, unprivileged user operations. The DB is correctly updated (line 2746), so the bug is purely in the in-memory cache layer and is present from the moment any sub-wallet registers a puzzle hash interest. No special attacker capability is required; sending a coin to a registered puzzle hash is sufficient to trigger the missed-tracking path.

### Recommendation

Replace all three references to `self.interested_coin_cache` inside `add_interested_puzzle_hashes` with `self.interested_ph_cache`, mirroring the correct pattern used in `add_interested_coin_ids`.

### Proof of Concept

1. Create a pool wallet. `PoolWallet.create` calls `wallet_state_manager.add_interested_puzzle_hashes([p2_puzzle_hash], [pool_wallet.wallet_id])`.
2. Due to the bug, `p2_puzzle_hash` is written into `interested_coin_cache`, not `interested_ph_cache`.
3. A block reward coin arrives at `p2_puzzle_hash`. `_add_coin_states` calls `get_wallet_identifier_for_puzzle_hash`, which queries the DB (correctly finds the wallet). However, any code path that relies on `interested_ph_cache` for fast resolution will miss the entry.
4. Simultaneously, if any coin whose `coin_id == p2_puzzle_hash` (adversarially crafted or by collision) arrives, `_add_coin_states` finds it in `interested_coin_cache` and misattributes it to the REMOTE wallet, corrupting the coin record. [5](#0-4) [6](#0-5)
