### Title
`respond_verify_foreign_tx` does not bind `payload_hash` to the stored request, enabling cross-request response replay by a single Byzantine participant - (`crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` contract method verifies that the submitted signature is valid for the submitted `payload_hash`, but never checks that `payload_hash` was actually derived from the stored `VerifyForeignTransactionRequest`. A single malicious attested participant (below the signing threshold) can replay a legitimately produced `(payload_hash, signature)` pair from any prior response to resolve an entirely different pending request, delivering a forged foreign-chain attestation to the waiting caller.

### Finding Description

The `respond_verify_foreign_tx` function accepts two arguments: a `VerifyForeignTransactionRequest` (used as the lookup key into `pending_verify_foreign_tx_requests`) and a `VerifyForeignTransactionResponse` (containing `payload_hash` and `signature`). [1](#0-0) 

The contract verifies only that `signature` is valid for `response.payload_hash` under the root public key. It does **not** verify that `response.payload_hash` was derived from the `request` argument.

The correct `payload_hash` for a given request is defined by the node-side computation:

```
payload = ForeignTxSignPayload::V1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }
payload_hash = SHA-256(borsh(payload))
``` [2](#0-1) 

The node assembles `payload_hash` from the actual RPC-observed data and passes it to the contract via `VerifyForeignTransactionRespondArgs::from_signature`: [3](#0-2) 

The contract receives this `payload_hash` opaquely and only checks the ECDSA signature over it: [4](#0-3) 

Because `payload_hash` is caller-supplied and not recomputed from the stored request, any previously observed valid `(payload_hash_A, signature_A)` pair — both of which are permanently visible on-chain after any prior `respond_verify_foreign_tx` call — can be submitted alongside a **different** pending `request_B`. The contract will:

1. Find `request_B` in `pending_verify_foreign_tx_requests` (it exists).
2. Verify `signature_A` over `payload_hash_A` against the root key — valid.
3. Call `resolve_yields_for(&request_B, serialize(response_A))`, delivering the wrong attestation to every caller waiting on `request_B`. [5](#0-4) 

Contrast this with the regular `respond` method, where the payload is taken directly from the stored `SignatureRequest` and never from the response: [6](#0-5) 

The `ForeignTxSignPayload` design explicitly encodes the `ForeignChainRpcRequest` inside the signed payload so that the signature attests to a specific chain query and its observed values: [7](#0-6) 

But the contract never enforces this binding.

### Impact Explanation

A caller whose `verify_foreign_transaction(request_B)` is pending receives a `VerifyForeignTransactionResponse` whose `payload_hash` was derived from a completely different `request_A`. The signature is cryptographically valid, so any downstream contract that trusts the MPC attestation without independently recomputing the expected `payload_hash` from its own request will accept a forged verification. In bridge or omnibridge scenarios this directly enables invalid fund releases: the MPC network is made to appear to have attested that a foreign transaction matching `request_B` finalized, when in fact it attested to `request_A`.

This matches the allowed impact: **High — forged foreign-chain verification causing invalid bridge execution**.

### Likelihood Explanation

The attacker must be a single attested MPC participant (below the signing threshold). All inputs needed for the attack — `payload_hash_A` and `signature_A` — are permanently on-chain after any prior legitimate `respond_verify_foreign_tx` call. No key material, threshold collusion, or TEE bypass is required. The attacker only needs to submit one NEAR transaction calling `respond_verify_foreign_tx` with a mismatched `(request_B, response_A)` pair while `request_B` is pending.

### Recommendation

The contract must recompute `payload_hash` from the stored request rather than accepting it from the response. Two concrete options:

1. **Include `extracted_values` in the response**: Extend `VerifyForeignTransactionResponse` to carry `Vec<ExtractedValue>`. The contract then reconstructs `ForeignTxSignPayload::V1 { request: stored_request, values }`, computes `SHA-256(borsh(payload))`, and verifies the signature against that locally computed hash — never against the caller-supplied `payload_hash`.

2. **Include the full `ForeignTxSignPayload` in the response**: The contract verifies `payload.request == stored_request`, then recomputes and checks the hash.

Either approach closes the gap by making the signature verification depend on the stored request, mirroring how `respond` derives the payload from `request.payload` rather than from the response.

### Proof of Concept

```
1. User A calls verify_foreign_transaction(request_A) → pending in contract.
2. MPC network legitimately processes request_A:
   - Nodes compute payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1{request_A, values_A}))
   - Threshold signs payload_hash_A → signature_A
   - Leader calls respond_verify_foreign_tx(request_A, {payload_hash_A, signature_A}) → on-chain.
3. User B calls verify_foreign_transaction(request_B) → pending in contract.
4. Malicious attested participant reads payload_hash_A and signature_A from chain history.
5. Malicious participant calls respond_verify_foreign_tx(request_B, {payload_hash_A, signature_A}).
6. Contract checks: verify_ecdsa_signature(signature_A, payload_hash_A, root_key) → VALID.
7. Contract calls resolve_yields_for(&request_B, serialize({payload_hash_A, signature_A})).
8. User B's waiting promise resumes with a response whose payload_hash attests to request_A,
   not request_B — a forged foreign-chain verification.
```

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
}

    fn from_signature(

        payload_hash: dtos::Hash256,
        signature: Signature,
        public_key: VerifyingKey,

        let dto_signature = dtos::K256Signature {
            big_r: dtos::K256AffinePoint::from(signature.big_r),
            s: dtos::K256Scalar::from(signature.s),
            recovery_id,
        };
        Ok(contract_args::VerifyForeignTransactionRespondArgs::new(
            dtos::VerifyForeignTransactionRequest {
                request: request.request,
                domain_id: request.domain_id,
                payload_version: request.payload_version,
            },
            dtos::VerifyForeignTransactionResponse {
                payload_hash,
                signature: dtos::SignatureResponse::Secp256k1(dto_signature),
