### Title
Missing shard_id cross-check in `set_state_header` allows a malicious peer to store a wrong-shard chunk as the state sync header for shard B — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and validates it, but never asserts that the embedded chunk's `shard_id()` equals the `shard_id` parameter. Combined with a position-blind `verify_path` check on the chunk Merkle proof, a malicious peer can supply chunk A (belonging to shard A) while claiming it is the header for shard B. All five validation steps pass, the wrong header is committed to `DBCol::StateHeaders` under key `StateHeaderKey(B, sync_hash)`, and subsequent `set_state_finalize` applies shard A's state root and transactions under shard B's `ShardUId`, permanently corrupting the syncing node's state for shard B.

### Finding Description

**Step 1 — `validate_chunk_proofs` passes:**
`validate_chunk_proofs` checks the chunk's internal self-consistency (header hash, body hash, tx_root, outgoing receipts root) but never compares `chunk.shard_id()` to the caller-supplied `shard_id` parameter. [8](#0-7) 

**Step 2 — `verify_path` for the chunk Merkle proof passes without position check:**
The call at line 394 checks that `compute_root_from_path(chunk_proof, hash(ChunkHashHeight(chunk_A))) == chunk_headers_root`. For a 2-shard block `[chunk_A, chunk_B]`, the root is `combine_hash(h_A, h_B)`. The proof for chunk A at index 0 is `[{h_B, Right}]`, which satisfies `combine_hash(h_A, h_B) == root` regardless of which shard the caller claims. `verify_path` has no index argument; `verify_path_with_index` (which does check position via `verify_path_matches_index`) is never called here. [3](#0-2) [9](#0-8) 

**Step 3 — Receipt proofs (steps 4e/4f) pass with real blockchain data:**
`receipts_hash = hash(ReceiptList(shard_id=B, receipts_A_to_B))`. Shard A's `prev_outgoing_receipts_root` is a Merkle root over `hash(ReceiptList(to_shard, receipts_A_to_shard))` for every destination shard, so `receipts_A_to_B` is a valid leaf under shard A's root. The attacker sets `root = chunk_A.prev_outgoing_receipts_root()` and supplies the corresponding `block_proof` (shard A's position in `prev_chunk_outgoing_receipts_root`), both of which are publicly readable from the chain. `to_shard_id` in `ShardProof` is explicitly discarded (`to_shard_id: _`), so no cross-check occurs there either. [10](#0-9) 

**Step 4 — Wrong header committed:** [11](#0-10) 

**Step 5 — `set_state_finalize` applies shard A's state root under shard B's `ShardUId`:**
`chunk_header.prev_state_root()` is shard A's state root. `shard_uid` is derived from the `shard_id` parameter (B). The attacker also provides shard A's state parts (which validate against shard A's state root), so `set_state_part` passes too. The node ends up with shard A's trie data stored as shard B's state. [12](#0-11) 

### Impact Explanation

The syncing node permanently stores shard A's state under shard B's `ShardUId`. When the node subsequently tries to apply new chunks for shard B, the state root it holds does not match the expected state root in the canonical block headers, causing all chunk application to fail. If the node is a validator, it cannot produce or endorse valid chunks for shard B. The node is effectively ejected from participation in shard B consensus until it re-syncs — and re-sync will again be vulnerable to the same attack from the same peer.

### Likelihood Explanation

During state sync, a node downloads `ShardStateSyncResponseHeader` from peers over the network. Any peer can serve a crafted header. All required proof material (chunk data, Merkle paths, receipt proofs) is publicly available from the canonical chain. No validator key, stake, or privileged access is required. The attack is locally reproducible with a unit test as described in the question.

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the position-blind `verify_path` call for the chunk Merkle proof with `verify_path_with_index`, passing the shard's index in the epoch's shard layout as the expected position, to prevent any future bypass via a different chunk that happens to hash into the same root path.

### Proof of Concept

```
1. Obtain sync_prev_block with shards [A=0, B=1].
2. Fetch chunk_A (shard_id=0) and its Merkle proof at index 0 in chunk_headers_root.
3. Fetch receipts_A_to_B and their Merkle proof within chunk_A.prev_outgoing_receipts_root.
4. Fetch chunk_A.prev_outgoing_receipts_root's Merkle proof within prev_chunk_outgoing_receipts_root.

7. Observe: returns Ok(()), header stored under StateHeaderKey(B, sync_hash).
```

```rust
pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
```
