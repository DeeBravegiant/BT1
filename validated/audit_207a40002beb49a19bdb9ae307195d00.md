### Title
Non-atomic block commit silently swallows L1 provider failure, leaving L1 handler transaction state inconsistent — (File: `crates/apollo_batcher/src/batcher.rs`)

### Summary

`commit_proposal_and_block` performs two sequential, non-atomic state mutations: (1) committing the block to persistent storage, and (2) notifying the L1 events provider. If step 1 succeeds and step 2 fails, a misleading comment reads *"Return error if the commit to the L1 provider failed"* but the code only logs the error and increments a metric, then returns `Ok(())`. The L1 events provider is left at the old height with consumed L1 handler transactions still in staged state, causing valid L1 handler transactions to be rejected in subsequent blocks.

### Finding Description

`commit_proposal_and_block` in `crates/apollo_batcher/src/batcher.rs` performs three sequential, non-atomic operations:

**Step 1 — Storage commit (propagates error):** [1](#0-0) 

**Step 2 — L1 events provider notification (error silently swallowed):** [2](#0-1) 

The comment at line 1123 reads `// Return error if the commit to the L1 provider failed.` but the implementation only logs and increments `BATCHER_L1_EVENTS_PROVIDER_ERRORS`, then falls through to return `Ok(())`. Compare this with the mempool case, which explicitly documents its intent as `// Recoverable error`: [3](#0-2) 

The L1 events provider's `commit_block` advances its internal `current_height` and marks consumed L1 handler transactions as committed: [4](#0-3) 

If the `commit_block` RPC call fails (e.g., transient network error between batcher and L1 events provider service), the provider's `current_height` stays at H and consumed transactions remain in `Staged` state. When the next block H+1 starts, `start_block(H+1)` calls `check_height_with_error(H+1)`: [5](#0-4) 

This returns `UnexpectedHeight`, causing `start_block` to fail. Every subsequent `get_txs(H+1)` and `validate(tx_hash, H+1)` call also fails the same height check: [6](#0-5) [7](#0-6) 

The catching-up mechanism is only triggered by a subsequent `commit_block` call with a mismatched height, not by `start_block`: [8](#0-7) 

So during the entire build and validation of block H+1, the L1 events provider is in an inconsistent state.

### Impact Explanation

**Broken invariant:** After `decision_reached` returns `Ok`, the L1 events provider must be at height H+1 with all consumed L1 handler transactions marked committed. This invariant is violated when the `commit_block` RPC fails and the error is swallowed.

**Concrete wrong value:** The L1 events provider's `current_height` remains at H instead of H+1. Consumed L1 handler transactions remain in `Staged` state instead of `Committed`.

**Admission impact:** For block H+1, `validate(tx_hash, H+1)` returns `UnexpectedHeight` for every L1 handler transaction, causing the validator's `ValidateTransactionProvider` to treat them as `L1EventsProviderError` and return `L1HandlerTransactionValidationFailed`. Valid L1 handler transactions (L1-to-L2 messages) are rejected before sequencing. This matches: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."* [9](#0-8) 

### Likelihood Explanation

**Low.** Requires a transient network/RPC failure between the batcher process and the L1 events provider process after the storage write succeeds. This is a realistic failure mode in distributed deployments (e.g., Kubernetes pod restart, network partition, gRPC timeout). The window is narrow but the error-swallowing is unconditional — any `L1EventsProviderClientError` variant triggers it.
