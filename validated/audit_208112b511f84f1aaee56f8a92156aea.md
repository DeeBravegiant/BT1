Audit Report

## Title
Uninitialized `rate` in `CrossChainRateReceiver` causes division-by-zero revert in pool deposit paths — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.rate` is a `uint256` that Solidity zero-initializes at deployment. `getRate()` returns it unconditionally with no zero-guard. Every pool variant divides by this value in `viewSwapRsETHAmountAndFee`, so any deposit call made before the first successful `lzReceive` reverts with a Solidity 0.8 `Panic(0x12)` division-by-zero. No funds are lost because the revert rolls back the transaction, but the pool cannot serve users during this bootstrapping window.

## Finding Description
`CrossChainRateReceiver` declares `uint256 public rate` which defaults to `0`. [1](#0-0) 

`getRate()` returns it with no zero-check or staleness check: [2](#0-1) 

The only write path is `lzReceive`, which requires the LayerZero endpoint to call with the correct `srcChainId` and `rateProvider`. Until that first message arrives, `rate` stays `0`. [3](#0-2) 

Every pool delegates its own `getRate()` to the oracle: [4](#0-3) 

Both swap paths use `rsETHToETHrate` as a bare divisor:
- ETH path: `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` [5](#0-4) 
- Token path: `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` [6](#0-5) 

The same unguarded division exists in `RSETHPoolV3.viewSwapRsETHAmountAndFee`: [7](#0-6) 

**Existing guards are insufficient.** ETH deposits in `RSETHPool` are gated by `isEthDepositEnabled` (defaults to `false`), which provides partial protection for that path. However, token deposits have no equivalent gate — they only require `onlySupportedToken(token)`. Critically, `addSupportedToken` validates the *token* oracle's rate is non-zero, but never validates the rsETH oracle: [8](#0-7) 

`setRSETHOracle` similarly only checks for a non-zero address, not a non-zero rate: [9](#0-8) 

This means an admin can add a supported token (token oracle check passes) while the rsETH oracle is still uninitialized, opening the token deposit path to division-by-zero for any user.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

All calls to `deposit()` (ETH or token variant) on any pool using `RSETHRateReceiver` as its oracle revert with `Panic(0x12)` while `rate == 0`. The pool cannot deliver its promised swap service during this window. No funds are permanently lost because the revert rolls back the entire transaction, including any `safeTransferFrom` that already executed within the same call.

## Likelihood Explanation
The bootstrapping window exists in every deployment — it is bounded only by the time between contract deployment and the first successful `lzReceive`. The window is extended by LayerZero message delays, dropped messages, or misconfigured constructor parameters (`layerZeroEndpoint`, `srcChainId`, `rateProvider`). The contract has been deployed across 15+ chains, each with its own bootstrapping window. A normal user calling `deposit(token, amount, ref)` after an admin has added a supported token but before the first oracle update will deterministically trigger the revert with no special privileges required.

## Recommendation
1. Add a zero-rate guard in `CrossChainRateReceiver.getRate()`:
   ```solidity
   function getRate() external view returns (uint256) {
       require(rate != 0, "Rate not initialized");
       return rate;
   }
   ```
2. Optionally add a staleness check using `lastUpdated`:
   ```solidity
   require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate stale");
   ```
3. Apply the same non-zero rate validation to `setRSETHOracle` that `addSupportedToken` already applies to token oracles (line 648 of `RSETHPool.sol`).
4. Consider allowing the owner to seed an initial rate at construction time so the pool is functional from block 0.

## Proof of Concept
```solidity
function test_zeroRateCausesRevert() public {
    // Deploy receiver with a mock endpoint that never delivers
    RSETHRateReceiver receiver = new RSETHRateReceiver(
        1,                      // srcChainId
        address(0xdead),        // rateProvider (never sends)
        address(mockLzEndpoint)
    );

    // Confirm rate is zero
    assertEq(receiver.getRate(), 0);

    // Wire pool to this receiver and add a supported token
    // (addSupportedToken checks token oracle rate != 0, not rsETH oracle)
    RSETHPool pool = deployPool(address(receiver));
    pool.addSupportedToken(address(wstETH), address(wstETHOracle), address(bridge));

    // Token deposit reverts with Panic(0x12) — division by zero in viewSwapRsETHAmountAndFee
    vm.expectRevert(); // Panic: division by zero
    pool.deposit(address(wstETH), 1e18, "ref");
}
```

The revert is deterministic and reproducible on unmodified production code with no `lzReceive` call ever having succeeded.
