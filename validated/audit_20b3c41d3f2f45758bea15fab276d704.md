### Title
Unbound Response-to-Request Binding in `respond_verify_foreign_tx` Allows Single Byzantine Participant to Forge Foreign-Chain Verification Results - (File: crates/contract/src/lib.rs)

### Summary
`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over the supplied `payload_hash`, but never checks that `payload_hash` is actually derived from the `request` parameter. A single Byzantine attested participant can replay any previously obtained valid MPC signature — produced for a different foreign-chain request — against any currently pending `verify_foreign_transaction` request, causing the contract to deliver a fabricated verification result to all waiting callers.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` accepts two caller-supplied arguments: a `request: VerifyForeignTransactionRequest` (used as the lookup key into `pending_verify_foreign_tx_requests`) and a `response: VerifyForeignTransactionResponse` (containing `payload_hash` and `signature`). [1](#0-0) 

The only cryptographic check performed is:

```
verify_ecdsa_signature(response.signature, response.payload_hash, root_public_key)
```

This confirms the signature is valid over `payload_hash` under the MPC root key. It does **not** verify that `payload_hash == SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <any> }))`. The `request` parameter and the `response.payload_hash` are completely decoupled. [2](#0-1) 

After the signature check passes, `resolve_yields_for` drains all queued yields for `request` and delivers the raw `response` bytes to every waiting caller: [3](#0-2) 

The `ForeignTxSignPayloadV1` struct that defines what the MPC network is supposed to sign binds `request` to `values`: [4](#0-3) 

But this binding is only enforced off-chain (by honest MPC nodes). The contract never reconstructs or checks the hash.

### Impact Explanation

A single Byzantine attested participant can:

1. Observe that request B (for `tx_id_B`) is pending in `pending_verify_foreign_tx_requests`.
2. Retrieve a previously obtained valid MPC signature `S_A` over `H_A = SHA-256(borsh(ForeignTxSignPayload{tx_id_A, values_A}))` from any prior completed `verify_foreign_transaction` for a different transaction.
3. Call `respond_verify_foreign_tx(request_B, {payload_hash: H_A, signature: S_A})`.
4. The contract accepts the call (signature is valid over `H_A`, request_B is pending) and delivers `{H_A, S_A}` to all callers waiting on request_B.

Callers (e.g., bridge contracts) receive a `payload_hash` that does not correspond to `tx_id_B`. Any downstream logic that trusts this hash to confirm a foreign-chain event (e.g., an inbound bridge transfer) is deceived. This enables invalid bridge execution or double-spend conditions — matching the **High** allowed impact: *forged foreign-chain verification that causes invalid bridge execution*.

### Likelihood Explanation

- Requires only **one** Byzantine attested participant, well below the signing threshold.
- The attacker needs a previously obtained valid MPC signature, which is available from any past successful `verify_foreign_transaction` response (these are public on-chain).
- No threshold collusion, no TEE attack, no network-level DoS required.
- The attack is executable in a single on-chain transaction.

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, recompute the expected `payload_hash` from the `request` parameter and the extracted values embedded in the response, and assert equality. Concretely, the response should include the full `ForeignTxSignPayload` (not just its hash), and the contract should:

1. Compute `expected_hash = SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: response.values }))`.
2. Assert `expected_hash == response.payload_hash` before accepting the response.

Alternatively, include the `request` key inside the signed payload at the contract level so that a signature produced for request A is cryptographically unusable for request B.

### Proof of Concept

```
// Setup: request A was previously completed; its response is on-chain.
// H_A = SHA-256(borsh(ForeignTxSignPayload{tx_id_A, [BlockHash(0xAA..AA)]}))
// S_A = valid MPC ECDSA signature over H_A (root key, no tweak)

// Step 1: User submits request B for tx_id_B
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    domain_id: domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: tx_id_B,  // different from tx_id_A
        confirmations: 2,
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
});
// request_B is now pending in pending_verify_foreign_tx_requests

// Step 2: Byzantine attested participant replays S_A against request_B
contract.respond_verify_foreign_tx(
    request_B,                                    // pending request for tx_id_B
    VerifyForeignTransactionResponse {
        payload_hash: H_A,                        // hash from tx_id_A's verification
        signature: S_A,                           // valid MPC signature over H_A
    }
);
// Contract accepts: signature is valid over H_A, request_B is pending.
// All callers waiting on request_B receive {H_A, S_A}.
// They believe tx_id_B was verified, but H_A encodes tx_id_A's extracted values.
```
