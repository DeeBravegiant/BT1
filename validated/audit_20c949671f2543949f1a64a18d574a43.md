### Title
Unauthenticated P2P Peer Can Corrupt Block Transaction Storage via Unvalidated `n_transactions` and Transaction Hashes — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

A malicious p2p peer can supply a crafted `SignedBlockHeader` with an arbitrary `n_transactions` value, then send exactly that many `FullTransaction` messages with arbitrary bodies and hashes. The sync client accepts and stores them verbatim, corrupting `BlockBody::transactions`, `BlockBody::transaction_hashes`, and `BlockBody::transaction_outputs` for the targeted block.

### Finding Description

**Step 1 — Peer-controlled `n_transactions` written to storage.**

In `header.rs`, the `SignedBlockHeader` received from the network is stored as-is. The only checks performed are that `block_number` matches the expected value and that `signatures.len() == 1`. No block hash is verified against L1 or any trusted source, and no signature is verified against a known sequencer public key. [1](#0-0) 

The `n_transactions` field is decoded directly from the peer-supplied protobuf field `transactions.n_leaves`: [2](#0-1) 

**Step 2 — `target_transaction_len` is read from that peer-written header.**

In `TransactionStreamFactory::parse_data_for_block`, the loop bound is taken from the stored header's `n_transactions`: [3](#0-2) 

**Step 3 — Transaction hashes are accepted without validation.**

The loop collects exactly `target_transaction_len` `FullTransaction` messages and pushes their contents — including the peer-supplied `transaction_hash` — directly into the `BlockBody`. An explicit TODO acknowledges this is unvalidated: [4](#0-3) 

There is no check that the received `transaction_hash` values are consistent with the transaction bodies, nor that the set of hashes matches the `transaction_commitment` Merkle root stored in the header (which is itself peer-supplied).

**Step 4 — The corrupted `BlockBody` is committed to storage.** [5](#0-4) 

### Impact Explanation

Any node running the p2p sync client that connects to a malicious peer will store an attacker-chosen `BlockBody` for the targeted block. Subsequent RPC calls (`starknet_getBlockWithTxs`, `starknet_getTransactionByHash`, etc.) will return the attacker's crafted transactions as authoritative data. This is a **High** impact: RPC returns an authoritative-looking wrong value for sync/storage data.

### Likelihood Explanation

The attack requires only that the victim node connects to a malicious peer — a realistic scenario in a permissionless p2p network. No cryptographic material, validator keys, or operator privileges are needed. The malicious peer simply needs to respond to the victim's block header and transaction queries with crafted data.

### Recommendation

1. **Verify the block signature** against the known sequencer public key (the `verify_block_signature` function in `starknet_api/src/block.rs` exists but is not called in the p2p sync path).
2. **Verify transaction hashes** against the transaction body content before storing (resolve the TODO at `transaction.rs:88`).
3. **Verify the `transaction_commitment`** Merkle root stored in the header against the set of received transaction hashes, so a peer cannot supply a mismatched count or wrong hashes.
4. **Verify `block_hash`** against a trusted source (L1 or consensus) before accepting any block data from a p2p peer.

### Proof of Concept

```
1. Attacker runs a malicious p2p peer.
2. Victim node queries for block N's header.
3. Attacker responds with SignedBlockHeader { block_number: N, n_transactions: 2, block_hash: <arbitrary>, transaction_commitment: <arbitrary>, signatures: [<any>] }.
4. Victim stores this header (no validation).
5. Victim queries for block N's transactions.
6. Attacker responds with 2 FullTransaction { transaction: <crafted>, transaction_output: <crafted>, transaction_hash: <crafted> } messages, followed by Fin.
7. Victim stores BlockBody { transactions: [crafted_tx_0, crafted_tx_1], transaction_hashes: [crafted_hash_0, crafted_hash_1], transaction_outputs: [crafted_output_0, crafted_output_1] }.
8. starknet_getBlockWithTxs(N) returns the 2 crafted transactions.
```
