Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ERC20 assets it correctly evaluates `totalAssetDeposits + amount > depositLimit`, but for ETH it evaluates only `totalAssetDeposits > depositLimit`, silently dropping the incoming `amount`. Any depositor can push total ETH deposits arbitrarily beyond the configured cap as long as the existing total has not already crossed it.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the function reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch never adds `amount` to `totalAssetDeposits`, so the guard returns `false` (i.e., "limit not exceeded") for any deposit size as long as the running total is still at or below the cap. The function is called from `_beforeDeposit` (lines 648–670), which reverts on `true`; since the ETH branch never returns `true` for a deposit that would cross the cap, the revert never fires. `_beforeDeposit` is invoked by the public `depositETH` entry point (lines 76–93) with `msg.value` as the deposit amount, making the full path reachable by any external caller with no special role.

## Impact Explanation
The deposit cap is a protocol-level risk-management parameter. Bypassing it allows rsETH to be minted against ETH collateral beyond the intended ceiling. At minimum this constitutes a failure to deliver the promised deposit-cap guarantee (Low: contract fails to deliver promised returns). If the cap is operationally sized to ensure the protocol can back all outstanding rsETH, exceeding it can lead to protocol insolvency (Critical). The ERC20 path enforces the same cap correctly, making the ETH-specific bypass a concrete, demonstrable inconsistency rather than a theoretical concern.

## Likelihood Explanation
The exploit path is fully public: `depositETH` is `external payable` with no role restriction. The attacker only needs to observe `getTotalAssetDeposits(ETH_TOKEN)` approaching the configured limit (readable on-chain) and submit a single large ETH deposit. No victim interaction, governance capture, or privileged access is required. The condition is trivially detectable and the attack is repeatable on every deposit cycle.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999e18 > 1000e18` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 500 ETH; total ETH deposits become 1499 ETH, 49.9% above the cap.
6. An equivalent ERC20 deposit correctly reverts: `999e18 + 500e18 > 1000e18` → `true` → `MaximumDepositLimitReached`.

Foundry fuzz test plan: fuzz `amount` over `[1, type(uint128).max]` with `totalAssetDeposits` fixed just below `depositLimit`; assert that `depositETH{value: amount}` reverts with `MaximumDepositLimitReached` whenever `totalAssetDeposits + amount > depositLimit`. The current code will fail this assertion for all `amount` values where `totalAssetDeposits ≤ depositLimit`.
