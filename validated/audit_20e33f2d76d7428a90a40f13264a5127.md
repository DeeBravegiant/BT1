Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice` Accepts Stale Chainlink Prices With No Staleness Validation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-related return values (`updatedAt`, `answeredInRound`), performing no time-based or round-based freshness check. A stale price flows directly into `LRTDepositPool.getRsETHAmountToMint()`, allowing any depositor to receive rsETH computed against an inflated stale price, diluting the ETH-per-rsETH ratio for all existing holders and constituting theft of unclaimed yield.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at [1](#0-0)  destructures the `latestRoundData()` return tuple as `(, int256 price,,,)`, silently discarding `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`. No check of the form `block.timestamp - updatedAt <= heartbeat` or `answeredInRound >= roundId` is present.

This raw, unchecked price is consumed via two paths:

1. **Deposit minting** тАФ `LRTDepositPool.getRsETHAmountToMint()` computes [2](#0-1)  `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`, using the stale price as the numerator multiplier.

2. **rsETH price update** тАФ `LRTOracle._getTotalEthInProtocol()` sums `assetER * totalAssetAmt` for every supported asset [3](#0-2) , so a stale high price inflates the computed TVL and thus the stored `rsETHPrice`.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` checks `answeredInRound < roundID` and `timestamp == 0` [4](#0-3) , demonstrating the project is aware of staleness concerns but failed to apply equivalent guards to the core deposit oracle.

No other staleness guard exists anywhere in the deposit flow. The `pricePercentageLimit` mechanism in `_updateRsETHPrice()` only triggers on price *updates*, not on individual deposits, and does not protect against minting at a stale price between oracle updates.

## Impact Explanation
**High тАФ Theft of unclaimed yield.**

When a Chainlink LST/ETH feed goes stale at a price higher than the current market price, any depositor can call `LRTDepositPool.depositAsset()` and receive rsETH computed against the inflated stale price. The excess rsETH minted is backed by fewer real assets than implied, permanently reducing the ETH-per-rsETH exchange rate for all existing holders. Their accrued yield, reflected in the rsETH/ETH ratio, is irreversibly diluted. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation
**Medium.** Chainlink feeds update on deviation thresholds (typically 0.5%) or heartbeat intervals (24 hours for many LST/ETH feeds). During network congestion, oracle downtime, or slow-moving asset prices, feeds can remain stale for hours while the underlying asset's market price diverges. This is a documented real-world event. No special permissions are required тАФ any external depositor can trigger the path by calling `depositAsset()` or `depositETH()`.

## Recommendation
Add a per-feed configurable heartbeat mapping and validate freshness in `getAssetPrice()`:

```solidity
mapping(address asset => uint256 heartbeat) public assetHeartbeat;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale round");
    require(block.timestamp - updatedAt <= assetHeartbeat[asset], "Stale price");
    require(price > 0, "Invalid price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Set `assetHeartbeat` per feed to slightly above the Chainlink-documented heartbeat (e.g., 25 hours for a 24-hour feed). Also add a `setAssetHeartbeat()` admin function restricted to `onlyLRTManager`.

## Proof of Concept

1. Assume `stETH/ETH` Chainlink feed has a 24-hour heartbeat and last updated at price `1.05e18` (1.05 ETH per stETH).
2. A slashing event causes the actual stETH market price to drop to `0.98e18`, but the feed has not yet updated (deviation threshold not crossed, heartbeat not elapsed).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `_beforeDeposit` тЖТ `getRsETHAmountToMint` тЖТ `lrtOracle.getAssetPrice(stETH)` тЖТ `ChainlinkPriceOracle.getAssetPrice(stETH)` returns stale `1.05e18`.
5. `rsethAmountToMint = (1000e18 * 1.05e18) / rsETHPrice` тАФ attacker receives rsETH valued at 1050 ETH worth of assets while depositing only 980 ETH worth.
6. The 70 ETH surplus is extracted from existing rsETH holders' accrued yield, permanently reducing the rsETH/ETH exchange rate.

**Foundry fork test plan:**
- Fork mainnet at a block where a target LST/ETH feed is near its heartbeat boundary.
- Warp `block.timestamp` forward past the heartbeat without triggering a feed update (mock the aggregator to return a fixed stale answer).
- Call `depositAsset` as an unprivileged address and assert that `rsethAmountToMint` exceeds the fair value computed with the correct current price.
- Assert that `rsETHPrice` after a subsequent `updateRSETHPrice()` call is lower than before the deposit, confirming dilution of existing holders.
