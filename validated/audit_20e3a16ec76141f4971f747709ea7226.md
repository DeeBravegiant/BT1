Audit Report

## Title
`_updateRsETHPrice` Zero-Supply Branch Silently Orphans In-Flight EigenLayer ETH, Enabling First-Depositor Theft of Unclaimed Yield ŌĆö (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle._updateRsETHPrice()` contains an early-return branch that fires when `rsethSupply == 0`, unconditionally resetting both `rsETHPrice` and `highestRsethPrice` to `1 ether` without ever calling `_getTotalEthInProtocol()`. Any ETH still in-flight through EigenLayer's withdrawal queue at that moment ŌĆö counted as protocol TVL under normal operation ŌĆö is silently orphaned. The first depositor after the reset mints rsETH at the artificially deflated `1 ether` price and captures that ETH when it eventually settles, constituting theft of unclaimed yield from prior rsETH holders.

## Finding Description

`_updateRsETHPrice()` in `LRTOracle.sol` handles the zero-supply case as follows:

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;                        // ŌåÉ never reads _getTotalEthInProtocol()
}
``` [1](#0-0) 

The function returns before reaching:

```solidity
uint256 totalETHInProtocol = _getTotalEthInProtocol();
``` [2](#0-1) 

`_getTotalEthInProtocol()` calls `ILRTDepositPool.getTotalAssetDeposits(asset)` for every supported asset: [3](#0-2) 

For ETH, `getTotalAssetDeposits` delegates to `getETHDistributionData()`, which explicitly includes ETH mid-unstake from EigenLayer:

```solidity
ethStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getEffectivePodShares();
ethUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i])
    .getAssetUnstaking(LRTConstants.ETH_TOKEN);
``` [4](#0-3) 

This ETH is real protocol value ŌĆö it is counted in every normal price update ŌĆö but the zero-supply branch skips the entire accounting path and resets the price anchor to `1 ether`.

The zero-supply condition is reached via the normal withdrawal lifecycle. `unlockQueue()` in `LRTWithdrawalManager` burns rsETH:

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
``` [5](#0-4) 

When the last batch of rsETH is burned, `rsethSupply` drops to zero. EigenLayer's mandatory withdrawal delay (~7 days) means ETH queued for unstaking is routinely still in-flight at this moment. The public `updateRSETHPrice()` function can be called by anyone while the protocol is unpaused: [6](#0-5) 

A secondary consequence is that `highestRsethPrice` is also reset to `1 ether`, permanently lowering the circuit-breaker reference below the true historical peak, weakening downside protection for the next cycle: [7](#0-6) 

## Impact Explanation

**Theft of unclaimed yield (High).** After all rsETH is burned and the price is reset to `1 ether`, ETH still in EigenLayer's withdrawal queue is not reflected in the price. The first depositor mints rsETH at `1 ether` per token. When the in-flight ETH settles and `updateRSETHPrice()` is next called, `_getTotalEthInProtocol()` returns the orphaned ETH plus the new deposit, so `rsETHPrice` jumps above `1 ether`. The first depositor immediately withdraws at the higher price, capturing yield that accrued to the previous rsETH holders. The magnitude equals all ETH still in EigenLayer pods or unstaking queues at the moment of the reset.

## Likelihood Explanation

The precondition ŌĆö `rsethSupply == 0` with non-zero in-flight EigenLayer ETH ŌĆö arises naturally at the end of any full withdrawal cycle. EigenLayer's ~7-day withdrawal delay makes it structurally likely that some ETH is in-flight when the last rsETH is burned. `updateRSETHPrice()` is callable by any unprivileged address while unpaused, so an attacker only needs to monitor for the zero-supply condition, call `updateRSETHPrice()` to lock in the reset, and then be the first depositor. No privileged access, no oracle manipulation, and no victim mistakes are required.

## Recommendation

Before returning early in the zero-supply branch, check whether `_getTotalEthInProtocol()` is non-zero. If it is, either:

1. **Revert** ŌĆö prevent the price reset until all orphaned ETH has been accounted for (e.g., EigenLayer withdrawals completed and ETH swept to the deposit pool), or
2. **Preserve the last known price** ŌĆö keep `rsETHPrice` and `highestRsethPrice` at their last non-zero values rather than resetting to `1 ether`.

Do **not** reset `highestRsethPrice` to `1 ether` unconditionally; preserve the historical peak so the circuit-breaker reference is not weakened on protocol restart.

## Proof of Concept

1. Protocol runs normally; `rsETHPrice = 1.1 ether`, `highestRsethPrice = 1.1 ether`. NodeDelegator has 0.05 ETH queued for unstaking from EigenLayer (`getAssetUnstaking(ETH_TOKEN) = 0.05 ether`).
2. All users call `initiateWithdrawal(ETH, ...)`. rsETH is transferred to `LRTWithdrawalManager`.
3. Operator calls `unlockQueue(ETH, ...)`. `burnFrom` burns all rsETH; `rsethSupply = 0`. The 0.05 ETH is still in EigenLayer's withdrawal queue.
4. Attacker (or anyone) calls `updateRSETHPrice()`. Zero-supply branch fires: `rsETHPrice = 1 ether`, `highestRsethPrice = 1 ether`. The 0.05 ETH is never read.
5. Attacker calls `LRTDepositPool.depositETH{value: 1 ether}(0, "")`. `getRsETHAmountToMint` computes `1e18 * 1e18 / 1e18 = 1e18` rsETH minted at `rsETHPrice = 1 ether`. [8](#0-7) 
6. EigenLayer withdrawal completes; 0.05 ETH lands in the protocol. `getTotalAssetDeposits(ETH)` now returns `1.05 ether` (1 ETH deposit + 0.05 ETH settled).
7. Anyone calls `updateRSETHPrice()`. `newRsETHPrice = 1.05e18 / 1e18 = 1.05 ether`.
8. Attacker calls `initiateWithdrawal` ŌåÆ `unlockQueue` ŌåÆ `completeWithdrawal`, receiving `1.05 ETH` for their `1 rsETH`. Net gain: `0.05 ETH` of yield that belonged to the previous depositors.
