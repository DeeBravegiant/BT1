### Title
Deferred Deposit Check After State Mutation in `submit_participant_info` Allows Free Participant Registration - (File: `crates/contract/src/lib.rs`)

### Summary
`submit_participant_info` mutates TEE participant state via `add_participant` **before** verifying the caller attached sufficient deposit to cover storage costs. Because `#[handle_result]` in NEAR SDK does **not** revert state on `Err` return (only `env::panic_str` reverts), the state write persists even when the deposit check subsequently fails. Any caller with a valid TEE attestation can register participant info into the contract's TEE state without paying for the storage consumed.

### Finding Description
In `submit_participant_info` the execution order is:

1. Capture baseline storage: `let initial_storage = env::storage_usage();`
2. **Mutate state**: `self.tee_state.add_participant(NodeId { … }, proposed_participant_attestation, …)` — this writes the attestation into `self.tee_state.stored_attestations`.
3. Compute actual storage delta and required cost.
4. If `attached < cost` → `return Err(InvalidParameters::InsufficientDeposit { … }.into());` [1](#0-0) 

Step 4 uses `return Err(…)` inside a `#[handle_result]` function. In NEAR SDK, `#[handle_result]` serialises the `Err` value and returns it as the transaction's result — the transaction is **not** reverted. All state mutations from step 2 are committed to storage. [2](#0-1) 

The deposit check that is supposed to gate the write: [3](#0-2) 

runs **after** the write has already been durably committed, making it ineffective as a guard.

### Impact Explanation
An attacker (or a legitimate node operator acting adversarially) who possesses a valid TEE attestation can:

- Call `submit_participant_info` with `attached_deposit = 0`.
- `add_participant` succeeds and writes the `NodeId` + attestation into `tee_state.stored_attestations`.
- The deposit check fires, returns `Err(InsufficientDeposit)`, but the state write is already committed.
- The participant entry now exists in the TEE state at zero cost to the caller.

Repeated invocations (e.g., with rotating TLS keys or re-submissions that expand the stored entry) allow unbounded free storage consumption, breaking the storage-accounting invariant. More critically, the TEE state controls which nodes are considered attested participants for governance votes (`vote_new_parameters` reads `tee_state` to validate the proposed participant set). Polluting this state without payment authorization constitutes **participant-state manipulation that breaks production safety/accounting invariants** — matching the Medium impact tier. [4](#0-3) 

### Likelihood Explanation
The barrier is possessing a valid TEE attestation. Any legitimate node operator already satisfies this condition. A Byzantine participant below the signing threshold (the explicit attacker model in scope) who is a current or former node operator can exploit this without any collusion. The call is a single, direct NEAR transaction with no prerequisite beyond a valid attestation.

### Recommendation
Move the deposit check **before** calling `add_participant`, so no state is written if the deposit is insufficient. Alternatively, replace `return Err(…)` with `env::panic_str(…)` at the deposit-check site to guarantee state reversion. The pattern used correctly elsewhere in the contract (`require_deposit` in `check_request_preconditions`) calls `env::panic_str` precisely to ensure atomicity. [5](#0-4) 

### Proof of Concept

```
1. Attacker holds a valid TEE attestation (e.g., is a current node operator).
2. Attacker calls:
     submit_participant_info(
         proposed_participant_attestation = <valid_attestation>,
         tls_public_key = <attacker_tls_key>
     )
   with attached_deposit = 0 yoctoNEAR.
3. Inside the function:
   a. initial_storage = env::storage_usage()          // e.g. 1000 bytes
   b. add_participant(...) succeeds → tee_state grows  // e.g. 1200 bytes
   c. storage_used = 200, cost = 200 * storage_byte_cost > 0
   d. attached (0) < cost → return Err(InsufficientDeposit)
4. NEAR runtime commits the transaction (no panic); tee_state now contains
   the attacker's NodeId + attestation at zero cost.
5. Repeat with different tls_public_key values to grow contract storage
   indefinitely without payment, or to inject participant entries that
   influence subsequent governance vote validation.
```

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}
