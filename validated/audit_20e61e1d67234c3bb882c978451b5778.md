Audit Report

## Title
Cross-Token Arbitrage Drains Higher-Value Reserves via Unrestricted Mixed-Token Deposit/Withdraw — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary

`RsETHTokenWrapper` supports multiple allowed tokens and mints/burns wrsETH at a strict 1:1 ratio for any allowed token. Because `_deposit` and `_withdraw` perform no cross-token reserve accounting and impose no constraint that withdrawal must use the same token as deposit, any unprivileged caller can deposit a cheaper allowed token to receive wrsETH and immediately withdraw a more expensive allowed token at the same nominal amount, extracting value from the wrapper and leaving it undercollateralized with respect to the higher-value asset.

## Finding Description

The contract explicitly supports multiple allowed tokens. `reinitialize` (line 47) adds a second token via an admin upgrade, and `addAllowedToken` (line 174) allows further additions via `TIMELOCK_ROLE`. Once two tokens are registered, the exploit path is fully open to any unprivileged user.

`_deposit` (lines 134–141) only checks `allowedTokens[_asset]`, then mints `_amount` wrsETH 1:1 regardless of which token is deposited:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);
    emit Deposit(_asset, msg.sender, _to, _amount);
}
```

`_withdraw` (lines 120–128) only checks `allowedTokens[_asset]`, then burns `_amount` wrsETH and transfers `_amount` of the *requested* token — with no constraint that it must be the same token that was deposited and no check that the wrapper holds sufficient balance of that specific token relative to its per-token liabilities:

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    emit Withdraw(_asset, msg.sender, _to, _amount);
}
```

There is no per-token reserve accounting, no price oracle, and no restriction preventing a caller from depositing token-A and withdrawing token-B. The entire multi-token pool is treated as a single fungible bucket at 1:1 with wrsETH. The existing `TokenNotAllowed` guard is entirely insufficient — it only verifies the token is registered, not that the wrapper has per-token solvency.

## Impact Explanation

**Critical — Direct theft of user funds and protocol insolvency.**

When two allowed tokens trade at different market prices (e.g., altRsETH-A at 0.98 ETH and altRsETH-B at 1.00 ETH), an attacker extracts `N × (price_B − price_A)` per iteration. The wrapper's token-B reserves are fully drained while token-A accumulates. Remaining wrsETH holders who deposited token-B cannot redeem — their funds are directly stolen. The wrapper becomes insolvent with respect to the higher-value asset. This matches the allowed Critical impact: "Direct theft of any user funds" and "Protocol insolvency."

## Likelihood Explanation

**High.** The multi-token design is intentional — `reinitialize` was added specifically to introduce a second token, and `addAllowedToken` exists for further additions. L2 deployments routinely have multiple bridge-issued versions of the same L1 token (native bridge vs. third-party bridge), and these routinely trade at a spread due to differing bridge risk and liquidity. No special privilege is required to execute the exploit once two tokens are registered; `deposit` and `withdraw` are fully public and callable by any EOA or contract. The attack is atomic, repeatable, and requires no flash loan or complex setup.

## Recommendation

1. **Per-token reserve accounting**: Track how much of each allowed token backs the outstanding wrsETH supply. On `_withdraw`, enforce that the amount redeemed against token-X cannot exceed the wrapper's balance of token-X that was deposited (i.e., maintain a `mapping(address => uint256) tokenReserves` incremented on deposit and decremented on withdrawal, reverting if the withdrawal would exceed the per-token reserve).

2. **Alternatively, enforce single-token redemption per user**: Record which token each user deposited (e.g., `mapping(address => mapping(address => uint256)) userDeposits`) and only allow withdrawal of the same token in the same amount.

3. **Price-parity guard**: If multi-token fungibility is intentional by design, integrate a price oracle and reject deposits/withdrawals when the price spread between any two allowed tokens exceeds a configured threshold, preventing profitable arbitrage.

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Preconditions:
//   tokenA.price = 0.98 ETH  (cheaper, e.g. third-party bridge version)
//   tokenB.price = 1.00 ETH  (more expensive, e.g. native bridge version)
//   Both are in allowedTokens (set by admin via reinitialize/addAllowedToken).
//   Wrapper holds 1000 tokenB deposited by legitimate users.

function testCrossTokenArbitrage() public {
    uint256 amount = 1000e18;

    // Attacker acquires 1000 tokenA at market (~980 ETH equivalent)
    tokenA.mint(attacker, amount);

    vm.startPrank(attacker);

    // Step 1: deposit cheap tokenA → mint 1000 wrsETH at 1:1
    tokenA.approve(address(wrapper), amount);
    wrapper.deposit(address(tokenA), amount);
    // wrapper: 1000 tokenA + 1000 tokenB, wrsETH supply = 2000

    // Step 2: withdraw expensive tokenB → burn 1000 wrsETH, receive 1000 tokenB
    wrapper.withdraw(address(tokenB), amount);
    // wrapper: 1000 tokenA + 0 tokenB, wrsETH supply = 1000
    // All remaining wrsETH backed only by tokenA (worth ~980 ETH)
    // Original tokenB depositors cannot redeem → funds stolen

    vm.stopPrank();

    assertEq(tokenB.balanceOf(address(wrapper)), 0);
    assertGt(wrapper.totalSupply(), tokenB.balanceOf(address(wrapper)));
}
```

The root cause is confirmed at lines 120–128 (`_withdraw`) and 134–141 (`_deposit`) of `contracts/L2/RsETHTokenWrapper.sol`: `_withdraw` has no guard preventing redemption of a different token than was deposited, and neither function performs per-token reserve tracking.
