### Title
Verifier Account Code Hash Discarded After Threshold Vote — Metamorphic Verifier Substitution Enables TEE Attestation Bypass - (File: `crates/contract/src/tee/verifier_votes.rs`)

---

### Summary

`vote_tee_verifier_change` requires each voter to commit to a `(candidate_account_id, expected_code_hash)` pair, but when the vote crosses threshold, **only the `AccountId` is persisted** — the `expected_code_hash` is silently discarded. The contract has no on-chain mechanism to re-verify the code at `tee_verifier_account_id` at call time, nor can it confirm the account is locked (no full-access keys). A malicious verifier deployer who retains a full-access key can upgrade the verifier code after the vote passes, substituting a contract that returns `VerificationResult::Verified` for any input, bypassing TEE attestation entirely.

---

### Finding Description

In `crates/contract/src/tee/verifier_votes.rs`, `TeeVerifierVotes::vote` uses the full `VerifierChangeProposal { candidate_account_id, expected_code_hash }` only to compute a `ProposalHash` bucket. Once a bucket crosses threshold, the function returns `Some(proposal.candidate_account_id)` — the code hash is dropped:

```rust
// verifier_votes.rs line 82-87
if count >= protocol_threshold {
    self.pending.clear();
    Ok(Some(proposal.candidate_account_id))   // ← code hash discarded here
} else {
    Ok(None)
}
```

In `crates/contract/src/lib.rs`, `vote_tee_verifier_change` stores only the returned `AccountId`:

```rust
// lib.rs line 1642-1647
if let Some(new_verifier) =
    self.tee_verifier_votes
        .vote(proposal, participant, threshold_parameters)?
{
    self.tee_verifier_account_id = Some(new_verifier);  // ← only AccountId stored
}
```

The contract state field is:

```rust
// lib.rs line 171
tee_verifier_account_id: Option<AccountId>,
// "Not yet used to dispatch verification."
// TODO(#3639): once participants have voted a verifier in, make this
// non-optional via a migration that requires it be set.
```

The design document (`docs/design/attestation-verifier-contract.md`) explicitly acknowledges the limitation:

> *"mpc-contract cannot verify either claim itself — a NEAR contract has no way to read another account's code hash or access-key list — so the audit is the voter's responsibility, not the contract's."*

The intended mitigation is that voters manually audit the candidate account off-chain before voting. However, this audit is a **point-in-time snapshot** — it verifies the code at vote time, not at call time. If the verifier account retains a full-access key (or uses an upgradeable proxy pattern), the code can be silently replaced after the vote passes and after voters have completed their audit.

The attack is structurally identical to the metamorphic contract pattern in the external report: the "implementation" (verifier WASM) can change after the "proxy" (`tee_verifier_account_id`) is pointed at it, and the contract that holds the pointer has no way to detect the change.

---

### Impact Explanation

Once TODO `#3639` is resolved and `submit_participant_info` dispatches cross-contract calls to `tee_verifier_account_id` (as described in the design doc's `Promise::new(self.tee_verifier_account_id.clone()).function_call("verify_quote"...)`), a malicious verifier returning `VerificationResult::Verified` for any input would:

1. Allow any node — including non-genuine TEE nodes running arbitrary code — to pass `submit_participant_info` and join the MPC participant set.
2. Enable those rogue nodes to participate in threshold signing rounds, constituting **unauthorized transaction execution** and potential **unauthorized access to MPC key shares**.
3. Bypass the entire TEE attestation chain of trust that the security model depends on.

The voting mechanism (`vote_tee_verifier_change`) is already live in the deployed contract. A malicious `tee_verifier_account_id` can be voted in now; it becomes exploitable the moment the TODO is resolved.

---

### Likelihood Explanation

The attack requires:
1. A verifier deployer who retains a full-access key on the candidate account (or deploys an upgradeable contract) — trivially achievable by any actor who controls the deployment.
2. Threshold participants to vote for the candidate — they audit the code at vote time and see the correct WASM, so they act in good faith. This is **not** threshold collusion; the participants are deceived, not complicit.
3. The TODO `#3639` to be resolved — already planned and in active development.

The social-engineering surface is low: the MPC team publishes a candidate `account_id` and `expected_code_hash` alongside the on-chain vote. Voters are encouraged to trust the published hash rather than rebuild from source. A malicious deployer who controls the candidate account can present the correct hash at audit time and swap the code afterward.

---

### Recommendation

1. **Persist the code hash alongside the account ID.** Store `(AccountId, TeeVerifierCodeHash)` in `tee_verifier_account_id` rather than `AccountId` alone. This creates an on-chain record of the audited hash.

2. **Emit an on-chain event at vote-crossing time** that includes both the `candidate_account_id` and the `expected_code_hash`. Off-chain monitors can then alert if the code at the verifier account ever diverges from the emitted hash.

3. **Prefer NEP-591 global contracts for the verifier.** The design doc notes that globals expose `view_account(account_id).contract` as a `CodeHash` directly. While `mpc-contract` still cannot read this on-chain, it narrows the audit surface and makes the hash more easily verifiable by operators.

4. **Add a mandatory CLI pre-vote check** (`attestation-cli audit-verifier <account-id>`) that verifies both the code hash and the absence of full-access keys before any operator casts a vote, and make this a documented, enforced step in the governance runbook.

---

### Proof of Concept

**Setup**: Attacker controls `evil-verifier.near`.

**Step 1 — Deploy legitimate code.** Attacker deploys the correct `tee-verifier` WASM to `evil-verifier.near` but retains a full-access key.

**Step 2 — Solicit votes.** Attacker (or a compromised governance participant) proposes `vote_tee_verifier_change(evil-verifier.near, H_correct)`. Threshold participants audit `evil-verifier.near`, confirm the code hash matches `H_correct`, and vote yes.

**Step 3 — Vote crosses threshold.** `TeeVerifierVotes::vote` returns `Some("evil-verifier.near")`. `tee_verifier_account_id` is set to `evil-verifier.near`. The `H_correct` hash is discarded from contract state.

**Step 4 — Swap the code.** Attacker uses the retained full-access key to redeploy `evil-verifier.near` with malicious WASM that implements:
```rust
pub fn verify_quote(&self, ...) -> VerificationResult {
    VerificationResult::Verified(forge_report())
}
```

**Step 5 — Exploit.** Once TODO `#3639` is resolved, any node calls `submit_participant_info(Attestation::Dstack(fake_dstack), fake_tls_pk)`. `mpc-contract` dispatches `Promise::new("evil-verifier.near").function_call("verify_quote", ...)`. The malicious verifier returns `Verified`. `resolve_verification` runs post-DCAP checks against the allowlist — but the forged `VerifiedReport` can be crafted to pass those checks (image hash, RTMR values, etc. are attacker-controlled in the fake report). The rogue node is admitted to the participant set and participates in threshold signing. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```rust
    pub fn vote(
        &mut self,
        proposal: VerifierChangeProposal,
        participant: AuthenticatedParticipantId,
        threshold_parameters: &ThresholdParameters,
    ) -> Result<Option<AccountId>, Error> {
        let protocol_threshold = threshold_parameters.threshold().value();
        let participants = threshold_parameters.participants();
        let proposal_hash: ProposalHash = proposal.clone().into();

        let count_usize = {
            let voter_set = self.pending.vote(participant, proposal_hash);
            voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
        };
        let count = u64::try_from(count_usize).map_err(|e| ConversionError::DataConversion {
            reason: format!("vote count {count_usize} does not fit in u64: {e}"),
        })?;

        if count >= protocol_threshold {
            self.pending.clear();
            Ok(Some(proposal.candidate_account_id))
        } else {
            Ok(None)
        }
    }
```
