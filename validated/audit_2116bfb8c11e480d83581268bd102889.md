Audit Report

## Title
Missing Staleness Check on Chainlink `latestRoundData()` Enables Phantom Yield Fee Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` silently discards the `updatedAt` timestamp from `latestRoundData()`, accepting any price regardless of age. When a Chainlink feed is stale at a price higher than the last stored rsETH price, any unprivileged caller can invoke the public `updateRSETHPrice()` to trigger fee minting to the treasury based on phantom yield, diluting all rsETH holders with unbacked rsETH.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at line 52 destructures `latestRoundData()` as `(, int256 price,,,)`, discarding `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`. No staleness guard of the form `block.timestamp - updatedAt <= MAX_STALENESS` exists.

This price feeds into `LRTOracle._getTotalEthInProtocol()` (lines 331–349), which multiplies each supported asset's balance by its oracle price. `_getTotalEthInProtocol()` is called inside `_updateRsETHPrice()` (line 231), which is triggered by the public, permissionless `updateRSETHPrice()` (line 87).

Inside `_updateRsETHPrice()`, `previousTVL` is computed as `rsethSupply.mulWad(rsETHPrice)` (line 234) — the last stored price. If `totalETHInProtocol > previousTVL` (line 244), the delta is treated as yield and a protocol fee is minted to the treasury as rsETH (lines 299–308).

**Exploit path:**
1. Oracle last updated to P2 > P1 (where P1 was the price when `rsETHPrice` was last set). The true current price has since dropped to P_true < P2 (e.g., slashing event, LST depeg), but the oracle has not heartbeat-updated and remains stale at P2.
2. Attacker calls `LRTOracle.updateRSETHPrice()` — no role required.
3. `_getTotalEthInProtocol()` returns `deposits * P2`, which exceeds `previousTVL = supply * P1`.
4. `rewardAmount = deposits * P2 - supply * P1`; `protocolFeeInETH` is computed from this delta.
5. `rsethAmountToMintAsProtocolFee` is minted to the treasury (line 306), backed by no real ETH for the phantom portion `(P2 - P_true) * deposits`.
6. `rsETHPrice` is updated to the inflated value, affecting subsequent deposits and withdrawals.

**Existing checks are insufficient:**
- The `pricePercentageLimit` check (lines 252–266) only reverts if the price increase exceeds the configured threshold relative to `highestRsethPrice`; it does not prevent minting when the stale price is within the threshold.
- The `maxFeeMintAmountPerDay` cap (line 205) bounds per-day minting but does not prevent the phantom fee from being minted up to that cap.
- `ChainlinkOracleForRSETHPoolCollateral.sol` (lines 30–31) checks `answeredInRound < roundID` and `timestamp == 0` but still lacks a `block.timestamp - timestamp > threshold` guard; `ChainlinkPriceOracle` has none of these checks at all.

## Impact Explanation
**High — Theft of unclaimed yield.** The phantom fee minted to the treasury is backed by no real ETH, diluting all existing rsETH holders. The yield that should accrue to rsETH holders is instead partially captured as a protocol fee computed against a stale inflated price. This is a direct, concrete loss of value to rsETH holders proportional to `(P_stale - P_true) * totalDeposits * protocolFeeInBPS / 10_000`.

## Likelihood Explanation
Chainlink ETH-correlated LST feeds (rETH/ETH, stETH/ETH) operate on 24-hour heartbeats and deviation thresholds. During low-volatility periods, feeds routinely go many hours without updating. A price drop after the last oracle update (slashing, temporary depeg) combined with a stale feed creates the phantom yield condition. The attack requires no capital, no privileged role, and no flashloan — only a public call to `updateRSETHPrice()`. The condition is realistic and repeatable.

## Recommendation
Add a staleness guard in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(answeredInRound >= roundId, "Stale round");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
require(price > 0, "Non-positive price");
```

`MAX_STALENESS` should be set per feed based on its documented heartbeat (e.g., 25 hours for a 24-hour heartbeat feed). Apply the same pattern to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which already checks `answeredInRound < roundID` and `timestamp == 0` but still lacks the `block.timestamp - timestamp > threshold` guard.

## Proof of Concept
1. Deploy a mock Chainlink aggregator returning `updatedAt = block.timestamp - 25 hours` and `answer = 1.05e18` (stale high price).
2. Set `rsETHPrice` to a value corresponding to `1.04e18` per rETH (simulating the last stored price).
3. Set `maxFeeMintAmountPerDay` to a non-zero value and `pricePercentageLimit` to 0 (or a value that permits the 1% increase).
4. Call `LRTOracle.updateRSETHPrice()` from an unprivileged EOA.
5. Observe that `totalETHInProtocol > previousTVL`, `protocolFeeInETH > 0`, and rsETH is minted to the treasury.
6. Verify that the minted rsETH is not backed by any real ETH deposit (phantom yield).

Foundry fork test: fork mainnet, manipulate a Chainlink feed's `latestRoundData` return via `vm.mockCall` to return a stale timestamp and inflated price, then call `updateRSETHPrice()` and assert treasury rsETH balance increased while total ETH deposits did not change.
