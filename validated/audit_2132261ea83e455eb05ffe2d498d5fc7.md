### Title
Cross-Chain Replay of `deploy_token` MPC Signature Due to Missing Chain ID Binding — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `aptos/sources/bridge_types.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

### Summary

The `MetadataPayload` Borsh-encoded hash used to verify the NEAR MPC signature in `deploy_token` / `deployToken` does not include any chain identifier. A single MPC signature obtained from a legitimate `deploy_token` call on one destination chain (e.g., EVM) is byte-identical to the hash verified on every other destination chain (Starknet, Aptos, Solana), enabling an unprivileged attacker to replay it and deploy the same bridge token on any chain without NEAR authorization for that chain.

### Finding Description

`fin_transfer` correctly binds its signed payload to the destination chain by interleaving `chain_id` (as the OmniAddress tag) twice in the Borsh encoding — once before `token_address` and once before `recipient`. This prevents cross-chain replay of transfer messages.

`deploy_token` / `deployToken` uses a structurally identical signing pattern but **omits the chain ID entirely**. The signed Borsh blob is:

```
PayloadType::Metadata (0x01) || token || name || symbol || decimals
```

This is identical across all four destination chains for the same token metadata.

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on chain A (e.g., EVM) can extract the raw MPC signature and immediately submit it to `deploy_token` on chain B (e.g., Starknet, Aptos, or Solana). The signature verifies successfully because the hash is identical. The bridge token is deployed on chain B and the `nearToXToken` mapping is populated — without any NEAR-side authorization for chain B.

Consequences:
1. **Unauthorized bridge token deployment**: A bridge token is created on a chain the NEAR protocol has not yet authorized, bypassing the execution gate that the MPC signature is supposed to enforce.
2. **Permanent blocking of legitimate deployment**: Once deployed, the `ERR_TOKEN_EXIST` guard prevents the legitimate relayer from ever deploying the canonical token on that chain. The mapping is permanently poisoned.
3. **Potential unauthorized minting path**: If the NEAR side subsequently signs `fin_transfer` messages for that chain (because the token appears deployed), tokens can be minted against the attacker-triggered deployment. The `fin_transfer` signature is chain-bound, but the deployment itself is not — so the attacker controls the timing of when a chain becomes "live" for a token.

This matches: **High — Acceptance of insufficiently-bound signatures that bypass execution gates**, and **Critical — Unauthorized creation of wrapped bridge assets through deployment failure**.

### Likelihood Explanation

Exploitation requires only:
1. Monitoring any public chain for a `deployToken` / `deploy_token` transaction (trivially observable on-chain).
2. Extracting the signature bytes from calldata.
3. Submitting the same calldata to `deploy_token` on any other chain.

No privileged access, leaked keys, or colluding parties are required. The attacker is a standard unprivileged external caller. The attack is executable the moment the first `deploy_token` for any token appears on any chain.

### Recommendation

Include the destination chain ID in the `MetadataPayload` Borsh encoding, mirroring the pattern already used in `TransferMessagePayload`. Concretely, add `chain_id` as a field (or as the OmniAddress tag prefix) in `metadata_to_borsh` / `MetadataPayloadImpl::to_borsh` / `deployToken`'s `borshEncoded` / `DeployTokenPayload::serialize_for_near` on all four chains, and update the NEAR MPC signing logic to include the target chain ID when producing the signature.

### Proof of Concept

```
1. NEAR MPC signs MetadataPayload{token="usdc.near", name="USD Coin", symbol="USDC", decimals=6}
   → hash H = keccak256(0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06)
   → signature S

2. Legitimate relayer calls EVM deployToken(S, {token:"usdc.near",...})
   → EVM verifies keccak256(same encoding) == H, recovers nearBridgeDerivedAddress ✓
   → USDC bridge token deployed on EVM

3. Attacker observes S in EVM calldata.

4. Attacker calls Starknet deploy_token(S, {token:"usdc.near",...})
   → Starknet verifies keccak256(0x01 || borsh("usdc.near") || ...) == H (identical)
   → recovers nearBridgeDerivedAddress ✓
   → USDC bridge token deployed on Starknet without NEAR authorization for Starknet

5. Attacker repeats for Aptos and Solana using the same S.

6. All four chains now have a USDC bridge token deployed from a single MPC signature
   intended for exactly one chain. The nearToXToken mappings are set on all chains.
   Legitimate deployment attempts on any of these chains will now revert with ERR_TOKEN_EXIST.
```
