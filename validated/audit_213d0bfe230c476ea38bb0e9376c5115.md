Audit Report

## Title
Reward Tokens Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. Because `rewardPerToken()` short-circuits and returns `rewardPerTokenStored` unchanged during a zero-stake window, the elapsed time is silently discarded and the rewards that accrued during that window are permanently unclaimable. The contract's own NatSpec acknowledges this behavior but relies solely on an off-chain operational promise rather than an on-chain invariant.

## Finding Description
The `updateReward` modifier performs two independent assignments on every call:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L232-242
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // (A) — no-op when totalKernelStaked == 0
    updatedAt = lastTimeRewardApplicable();    // (B) — always advances
    ...
}
```

Line (A) calls `rewardPerToken()`, which short-circuits when `totalKernelStaked == 0`:

```solidity
// L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // no accumulation
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

Line (B) always advances `updatedAt` regardless. The two assignments are decoupled: `rewardPerTokenStored` stays flat while `updatedAt` leaps forward. When the next user action occurs after the zero-stake gap, the `(lastTimeRewardApplicable() - updatedAt)` term in `rewardPerToken()` starts from the new `updatedAt`, permanently erasing the rewards that accrued during the empty window.

The zero-stake condition is reachable via the permissionless `initiateWithdrawal()`:

```solidity
// L320-326
function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
    ...
    balanceOf[msg.sender] -= _amount;
    totalKernelStaked -= _amount;
    ...
}
```

If the last staker withdraws, `totalKernelStaked` drops to zero and the freeze condition is triggered. The `notifyRewardAmount` guard (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new reward period with zero stake; it does not prevent stake from draining to zero after a period has started. There is no admin rescue, sweep, or re-notify mechanism to recover frozen rewards.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens transferred into the contract via `notifyRewardAmount` and accrued during any zero-stake window are permanently unclaimable. The `rewardsToken` balance of the contract will exceed the sum of all claimable `rewards[user]` values by exactly the amount frozen during the empty period, with no recovery path.

## Likelihood Explanation
The trigger is a normal, permissionless user action (`initiateWithdrawal`). It requires only that the last remaining staker withdraws while a reward period is active — a realistic scenario during low-participation periods, market stress, or protocol migration. No attacker capability beyond being a staker is required. The condition is repeatable across multiple reward periods.

## Recommendation
Decouple the `updatedAt` advancement from the zero-stake short-circuit. Only advance `updatedAt` when `totalKernelStaked > 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This preserves the unaccumulated time so that when stakers return, the rewards for the empty window are distributed to them rather than frozen. Alternatively, add an admin-callable `recoverUnallocatedRewards()` function that sweeps any surplus `rewardsToken` balance exceeding the sum of all pending rewards.

## Proof of Concept
**Setup:** `duration = 7 days`, `rewardRate = 1000 / 7 days ≈ 1.653e-3 tokens/sec`.

1. **T0** — Admin calls `notifyRewardAmount(1000)`. `finishAt = T0 + 7 days`, `updatedAt = T0`.
2. **T0** — Alice stakes 100 KERNEL. `totalKernelStaked = 100`.
3. **T0 + 1 day** — Alice calls `initiateWithdrawal(100)`.
   - `updateReward(Alice)` fires: `rewardPerTokenStored` accumulates 1 day of rewards; `updatedAt = T0 + 1 day`. Alice's earned rewards are saved.
   - `totalKernelStaked = 0`.
4. **T0 + 2 days** — Bob calls `stake(100)`.
   - `updateReward(Bob)` fires: `rewardPerToken()` returns `rewardPerTokenStored` unchanged (zero-stake branch). **`updatedAt` is advanced to `T0 + 2 days`**, silently discarding 1 day of rewards (~143 tokens).
   - `totalKernelStaked = 100`.
5. **T0 + 3 days** — Bob calls `getReward()`.
   - Bob earns rewards for only 1 day (`T0+2` → `T0+3`), not 2 days.
   - ~143 tokens are permanently frozen in the contract.
