Audit Report

## Title
Global Shared Daily Mint Counter Exhaustible by Single Depositor, Blocking All Deposits for Up to 24 Hours - (File: contracts/RSETH.sol)

## Summary
`RSETH.sol` enforces a single protocol-wide `currentPeriodMintedAmount` counter in the `checkDailyMintLimit` modifier. Any depositor whose single transaction mints rsETH equal to or exceeding the remaining daily budget will exhaust the counter, causing every subsequent `depositETH` or `depositAsset` call by any user to revert with `DailyMintLimitExceeded` for up to 24 hours. No per-user sub-limit or single-transaction cap exists to prevent this.

## Finding Description
`RSETH.checkDailyMintLimit` is a global accumulator modifier:

```solidity
// contracts/RSETH.sol L42-56
modifier checkDailyMintLimit(uint256 amount) {
    if (block.timestamp >= periodStartTime + 1 days) {
        currentPeriodMintedAmount = 0;
        periodStartTime = getCurrentPeriodStartTime();
    }
    if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
        revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
    }
    currentPeriodMintedAmount += amount;
    _;
}
```

`currentPeriodMintedAmount` is a single storage slot shared by every caller. It is incremented unconditionally before the function body executes. The `mint` function applies this modifier:

```solidity
// contracts/RSETH.sol L229-240
function mint(address to, uint256 amount)
    external
    onlyRole(LRTConstants.MINTER_ROLE)
    whenNotPaused
    checkDailyMintLimit(amount)
{ ... }
```

Both public deposit entry points in `LRTDepositPool` call `_mintRsETH`, which calls `IRSETH.mint`:

```solidity
// contracts/LRTDepositPool.sol L686-690
function _mintRsETH(uint256 rsethAmountToMint) private {
    address rsethToken = lrtConfig.rsETH();
    IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint);
}
```

`depositETH` (L76-93) and `depositAsset` (L99-118) both route through `_mintRsETH`. There is no per-depositor sub-limit, no maximum single-transaction cap, and no mechanism to reserve budget for other users. Once `currentPeriodMintedAmount` reaches `maxMintAmountPerDay`, the check `currentPeriodMintedAmount + amount > maxMintAmountPerDay` is true for any nonzero `amount`, and every subsequent deposit reverts until the 24-hour window resets.

The same structural pattern is present in `contracts/pools/RSETHPoolV3.sol` (L96-125) with its own `dailyMintAmount`/`dailyMintLimit` pair, though the L1 path through `LRTDepositPool` is the primary affected surface.

## Impact Explanation
All users are temporarily unable to deposit ETH or LSTs into the L1 protocol for up to 24 hours. Because `LRTDepositPool.depositETH` and `depositAsset` are the sole entry points for obtaining rsETH on L1, the entire deposit functionality is frozen for the duration of the window. This maps to **Medium — Temporary freezing of funds**: users cannot move their assets into the protocol and are locked out of the deposit path for up to one full day.

## Likelihood Explanation
`maxMintAmountPerDay` is a finite value set by the LRT manager. Any depositor whose intended deposit maps to an rsETH amount that fills the remaining daily budget triggers the condition. Large institutional depositors or protocols routinely move hundreds or thousands of ETH in a single transaction. A deliberate attacker can also monitor the mempool for the daily reset and immediately front-run it with a deposit that exhausts the new period's full budget, repeating this every 24 hours at the cost of only the gas for the deposit (the deposited ETH/LST is retained as rsETH). No privileged access, leaked keys, or external protocol compromise is required.

## Recommendation
Introduce a per-transaction maximum deposit cap so no single call can consume the entire daily budget, or replace the global accumulator with a per-depositor sub-limit. Alternatively, remove the rolling time-window cap from `RSETH.mint` entirely and rely solely on the cumulative `depositLimitByAsset` ceiling in `LRTConfig`, which is not susceptible to single-transaction exhaustion.

## Proof of Concept
1. LRT manager sets `maxMintAmountPerDay = X` (e.g., 1000 rsETH).
2. Attacker calls `LRTDepositPool.depositETH{value: V}(0, "")` where `getRsETHAmountToMint(ETH, V) >= X`.
3. `_mintRsETH(X)` → `RSETH.mint(attacker, X)` → `checkDailyMintLimit(X)`: `currentPeriodMintedAmount` becomes `X`.
4. Any subsequent call by any user to `depositETH` or `depositAsset` within the same 24-hour window evaluates `currentPeriodMintedAmount + amount > maxMintAmountPerDay` as true for any nonzero `amount` and reverts with `DailyMintLimitExceeded`.
5. All deposits are blocked for up to 24 hours. The attacker retains their rsETH and can repeat the attack at the next period reset.

**Foundry test plan:** Deploy `LRTDepositPool` and `RSETH` on a local fork; set `maxMintAmountPerDay` to a small value; call `depositETH` with a value that mints the full limit; assert the next `depositETH` call from a different address reverts with `DailyMintLimitExceeded`; advance time by less than 24 hours and assert the revert persists; advance past 24 hours and assert deposits succeed again.
