### Title
Block-Level Reference Script Size Check Undercounts Sub-Transaction Ref Scripts in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

### Summary

`dijkstraBbodyTransition` delegates the block-level ref-script-size guard to `Conway.validateBodyRefScriptsSizeTooBig`, which internally calls `totalRefScriptSizeInBlock` → `txNonDistinctRefScriptsSize` (top-level inputs only). Dijkstra introduces sub-transactions whose reference inputs are invisible to this path. The Dijkstra-specific `batchNonDistinctRefScriptsSize` — which correctly sums top-level + sub-tx ref scripts — is used only for the per-tx check (`validateAllRefScriptSize`), never for the block-level check. A block producer can therefore submit a block whose true combined ref-script size exceeds `ppMaxRefScriptSizePerBlock` while the block-level guard passes.

**The gap:** `ppMaxRefScriptSizePerTx` (per-tx, uses `batchNonDistinctRefScriptsSize`) is enforced correctly. `ppMaxRefScriptSizePerBlock` (block-level, uses `txNonDistinctRefScriptsSize`) ignores sub-tx ref scripts entirely.

### Impact Explanation

A block producer can construct N top-level transactions, each carrying sub-transactions that reference large scripts. Each tx individually satisfies `ppMaxRefScriptSizePerTx` (because `batchNonDistinctRefScriptsSize` is checked). The block-level guard sees only the top-level ref-script contribution from each tx, so the sum it computes is strictly less than the true total. The block is accepted even though the actual ref-script data processed by all nodes exceeds `ppMaxRefScriptSizePerBlock`.

This falls under **Medium** impact: attacker-controlled block content exceeds intended validation limits (`ppMaxRefScriptSizePerBlock`). All honest nodes run the same code and reach the same (incorrect) conclusion, so there is no ledger divergence. The "permanently freezing funds" framing in the question is **not supported** — the ref-script size limit is a resource cap, not a fund-custody invariant.

### Likelihood Explanation

Any block producer (stake pool operator) can exploit this without any privileged access. The Dijkstra era must be active. The attacker needs only to craft transactions with sub-transactions referencing large scripts, which is a normal, unprivileged operation.

### Recommendation

Replace the call to `Conway.validateBodyRefScriptsSizeTooBig` in `dijkstraBbodyTransition` with a Dijkstra-specific block-level check that uses `batchNonDistinctRefScriptsSize` when summing ref-script sizes across all top-level transactions (and their sub-transactions) in the block. The existing `batchNonDistinctRefScriptsSize` function already provides the correct semantics; it just needs to be wired into the block-level validation path.

### Proof of Concept

1. Set `ppMaxRefScriptSizePerBlock = 1 MiB`, `ppMaxRefScriptSizePerTx = 200 KiB`.
2. Construct a top-level tx with 50 KiB of top-level ref scripts and three sub-transactions each referencing 50 KiB of ref scripts → `batchNonDistinctRefScriptsSize` = 200 KiB (passes per-tx check). `txNonDistinctRefScriptsSize` = 50 KiB (what the block check sees).
3. Pack 6 such transactions into one block. True total = 6 × 200 KiB = 1.2 MiB > 1 MiB limit. Block-check total = 6 × 50 KiB = 300 KiB → passes.
4. Assert: `sum (map batchNonDistinctRefScriptsSize txs) > ppMaxRefScriptSizePerBlock` while `totalRefScriptSizeInBlock` returns a value ≤ `ppMaxRefScriptSizePerBlock`. The block is accepted.

totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
