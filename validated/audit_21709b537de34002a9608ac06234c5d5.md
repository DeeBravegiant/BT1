Audit Report

## Title
`LRTDepositPool.getAssetCurrentLimit()` Returns Inflated Deposit Capacity When Daily rsETH Mint Quota Is Exhausted - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getAssetCurrentLimit()` computes remaining deposit capacity solely against the per-asset cap in `LRTConfig.depositLimitByAsset`, ignoring the independent daily rsETH mint cap enforced by `RSETH.checkDailyMintLimit`. When the daily quota is exhausted, the function returns a positive value while any deposit attempt will revert at `RSETH.mint()`, making the view function's output misleading and unreliable as a deposit-sizing signal.

## Finding Description
`getAssetCurrentLimit()` (L402–409) reads only `lrtConfig.depositLimitByAsset(asset)`:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) return 0;
    return lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;
}
```

The deposit execution path is:
1. `depositAsset()` / `depositETH()` → `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()` — checks only `lrtConfig.depositLimitByAsset` (L676–682).
2. `_mintRsETH()` (L686–690) → `IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint)`.
3. `RSETH.mint()` (L229–240) applies `checkDailyMintLimit(amount)` (L42–56), which reverts with `DailyMintLimitExceeded` if `currentPeriodMintedAmount + amount > maxMintAmountPerDay`.

`getAssetCurrentLimit()` never reads `RSETH.remainingDailyMintLimit()` (L265–272), which is the authoritative view of the second, independent gate. The two limits are completely decoupled in the view layer while being sequentially enforced in the execution layer.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.**

When `maxMintAmountPerDay` is set and the daily quota is exhausted, `getAssetCurrentLimit()` returns a positive value while every deposit attempt reverts. For ETH deposits the native value is returned on revert; for LST deposits `safeTransferFrom` has already executed but the entire transaction reverts atomically, so no tokens are permanently lost. The concrete harm is failed transactions and a broken protocol invariant: the canonical capacity-query function gives an answer that is inconsistent with what the protocol will actually accept.

## Likelihood Explanation
**Medium.** `maxMintAmountPerDay` has a dedicated setter (`setMaxMintAmountPerDay`), a dedicated view (`remainingDailyMintLimit()`), and a dedicated reset-timestamp getter (`getNextDailyLimitResetTimestamp()`), all indicating it is an active production control. Any depositor who queries `getAssetCurrentLimit()` when the per-asset cap still has headroom but the daily rsETH mint quota is exhausted will receive an inflated value and experience a revert. No special attacker capability is required; this is a normal operating condition at the end of a high-volume day.

## Recommendation
`getAssetCurrentLimit()` should factor in the remaining rsETH daily mint capacity. Convert `RSETH.remainingDailyMintLimit()` back to asset units using the current exchange rate and return the minimum of the two limits:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) return 0;
    uint256 assetCapRemaining = lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;

    address rsethToken = lrtConfig.rsETH();
    uint256 rsethDailyRemaining = IRSETH(rsethToken).remainingDailyMintLimit();
    if (rsethDailyRemaining == 0) return 0;

    address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
    ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);
    uint256 assetFromDailyLimit = (rsethDailyRemaining * lrtOracle.rsETHPrice()) / lrtOracle.getAssetPrice(asset);

    return assetCapRemaining < assetFromDailyLimit ? assetCapRemaining : assetFromDailyLimit;
}
```

## Proof of Concept
1. Admin calls `RSETH.setMaxMintAmountPerDay(1000 ether)`.
2. During the day, 999 ether of rsETH is minted; `RSETH.remainingDailyMintLimit()` returns `1 ether`.
3. The per-asset deposit cap for stETH still has `500 ether` of headroom.
4. User calls `LRTDepositPool.getAssetCurrentLimit(stETH)` → returns `500 ether`.
5. User calls `depositAsset(stETH, 2 ether, ...)` (within asset cap, but `rsethAmountToMint ≈ 2 ether` exceeds the `1 ether` daily remainder).
6. `_beforeDeposit` passes — `_checkIfDepositAmountExceedesCurrentLimit` sees no asset-cap violation.
7. `_mintRsETH` calls `RSETH.mint(user, rsethAmountToMint)`.
8. `checkDailyMintLimit` reverts: `DailyMintLimitExceeded(currentPeriodMintedAmount + rsethAmountToMint, maxMintAmountPerDay)`.
9. Transaction reverts despite `getAssetCurrentLimit()` having indicated 500 ether of capacity.

A Foundry fork test can reproduce this by: (a) deploying/forking the contracts, (b) calling `setMaxMintAmountPerDay(1000 ether)`, (c) minting 999 ether of rsETH via a prior deposit, (d) asserting `getAssetCurrentLimit(stETH) > 0`, then (e) asserting `depositAsset(stETH, 2 ether, 0)` reverts with `DailyMintLimitExceeded`.
