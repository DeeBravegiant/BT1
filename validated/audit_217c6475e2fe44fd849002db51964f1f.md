Audit Report

## Title
Stale `rsETHPrice` Allows Depositor to Capture Unclaimed Yield Before Price Update — (`contracts/LRTOracle.sol::updateRSETHPrice` / `contracts/FeeReceiver.sol::sendFunds` / `contracts/LRTDepositPool.sol::depositETH`)

## Summary

`LRTOracle.rsETHPrice` is a stored state variable updated only when `_updateRsETHPrice()` is explicitly called. Because both `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` carry no access-control restrictions, any EOA can atomically flush accumulated MEV/EL rewards into the deposit pool, deposit ETH at the pre-reward (stale) price, and then trigger the price update — capturing a proportional share of yield that belonged to existing rsETH holders.

## Finding Description

**Stored price, never refreshed on deposit.**
`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()`, which reads the stored state variable `rsETHPrice` set at line 28 of `LRTOracle.sol`. [1](#0-0)  The value is only written inside `_updateRsETHPrice()` at line 313. [2](#0-1)  `_beforeDeposit` never calls `_updateRsETHPrice()` before computing the mint amount. [3](#0-2) 

**`FeeReceiver.sendFunds()` is fully permissionless.**
The function has no role modifier; any caller can push the entire ETH balance of `FeeReceiver` into the deposit pool. [4](#0-3) 

**`updateRSETHPrice()` is fully permissionless.**
Only the `whenNotPaused` modifier is present; no role check. [5](#0-4) 

**Reward ETH immediately inflates TVL.**
`getETHDistributionData()` reads `address(this).balance` live, so ETH pushed by `sendFunds()` is immediately counted in TVL before `rsETHPrice` is updated. [6](#0-5) 

**`pricePercentageLimit` is not a reliable guard.**
`_updateRsETHPrice()` does check whether the new price exceeds `highestRsethPrice` by more than `pricePercentageLimit`, reverting for non-managers if so. [7](#0-6)  However: (a) `pricePercentageLimit` is initialized to `0` and the check is skipped entirely when it is zero; (b) even when set, the attacker can size the deposit to keep the price increase within the limit; (c) the attacker does not need to call `updateRSETHPrice()` themselves — they only need to deposit before the next legitimate update.

**Exploit path:**
1. Observe (or cause) ETH rewards sitting in `FeeReceiver`.
2. Call `FeeReceiver.sendFunds()` — rewards land in `LRTDepositPool`, TVL rises, `rsETHPrice` remains stale.
3. Call `LRTDepositPool.depositETH{value: X}(0, "")` — minted rsETH is computed at the pre-reward price, over-minting relative to the true TVL.
4. Call `LRTOracle.updateRSETHPrice()` (or wait for the protocol to do so) — price rises to reflect the new TVL, now shared with the attacker's freshly minted rsETH.

**Existing checks that fail to prevent this:**
- `nonReentrant` / `whenNotPaused` on `depositETH` do not enforce price freshness.
- The ETH deposit-limit check (`totalAssetDeposits > limit`, omitting the incoming amount) does not block the deposit. [8](#0-7) 
- `minRSETHAmountExpected` is attacker-controlled and set to `0` in the PoC.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders accumulate yield as ETH rewards raise TVL without increasing rsETH supply. That yield is "unclaimed" until `rsETHPrice` is updated. An attacker who deposits at the stale price and then triggers (or waits for) the update dilutes existing holders' yield proportionally to their deposit size. No principal is lost; only yield is diverted. This matches the allowed impact class "High. Theft of unclaimed yield."

Numerical example (no protocol fee for clarity):

| State | TVL | rsETH supply | rsETHPrice |
|---|---|---|---|
| Before rewards | 100 ETH | 100 | 1.000 |
| After `sendFunds()` (5 ETH) | 105 ETH | 100 | 1.000 (stale) |
| Attacker deposits 100 ETH | 205 ETH | 200 | 1.000 (stale) |
| After `updateRSETHPrice()` | 205 ETH | 200 | 1.025 |

Attacker: 100 rsETH × 1.025 = **102.5 ETH** (+2.5 ETH). Original holders: 100 rsETH × 1.025 = 102.5 ETH instead of the 105 ETH owed (**−2.5 ETH stolen yield**).

## Likelihood Explanation

- No privileged role required.
- No front-running required; all three steps are callable by the same EOA in consecutive transactions within the same block.
- MEV/EL rewards accumulate continuously; the attack window is always open.
- Profit scales linearly with deposit size and accumulated reward amount.
- The `pricePercentageLimit` guard is off by default and bypassable even when set.

## Recommendation

1. **Refresh the price inside `_beforeDeposit`**: call `_updateRsETHPrice()` (or compute the price on-the-fly from current TVL) before `getRsETHAmountToMint`, so every deposit uses a price that reflects the current TVL.
2. **Alternatively, restrict `FeeReceiver.sendFunds()`** to a privileged role (e.g., `MANAGER`) so reward flushing cannot be weaponized by an arbitrary caller.
3. **Defense-in-depth**: fix the ETH deposit-limit check to include the incoming amount (`totalAssetDeposits + amount > depositLimit`), matching the ERC-20 path. [9](#0-8) 

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "../contracts/LRTDepositPool.sol";
import "../contracts/LRTOracle.sol";
import "../contracts/FeeReceiver.sol";
import "../contracts/RSETH.sol";

contract YieldDiversionTest is Test {
    LRTDepositPool depositPool;
    LRTOracle oracle;
    FeeReceiver feeReceiver;
    RSETH rseth;
    address existingHolder = address(0xBEEF);
    address attacker     = address(0xBAD);

    // setUp: deploy contracts, mint 100 rsETH to existingHolder at price 1e18,
    //        seed FeeReceiver with 5 ETH of simulated MEV rewards.

    function test_stalePrice_yieldTheft() public {
        // 1. Rewards sit in FeeReceiver (no price update yet)
        vm.deal(address(feeReceiver), 5 ether);

        uint256 holderRsETH = rseth.balanceOf(existingHolder); // 100e18

        // 2. Attacker flushes rewards — TVL rises, rsETHPrice stays stale
        vm.prank(attacker);
        feeReceiver.sendFunds();

        // 3. Attacker deposits at stale price (1e18), receives 100 rsETH
        vm.deal(attacker, 100 ether);
        vm.prank(attacker);
        depositPool.depositETH{value: 100 ether}(0, "");

        // 4. Price update — now reflects 205 ETH / 200 rsETH = 1.025e18
        vm.prank(attacker);
        oracle.updateRSETHPrice();

        uint256 price = oracle.rsETHPrice();

        // Attacker profited: 100 rsETH × 1.025 = 102.5 ETH
        uint256 attackerValue = rseth.balanceOf(attacker) * price / 1e18;
        assertGt(attackerValue, 100 ether, "attacker captured yield");

        // Existing holders lost: 100 rsETH × 1.025 = 102.5 ETH, not 105 ETH
        uint256 holderValue = holderRsETH * price / 1e18;
        assertLt(holderValue, 105 ether, "existing holders lost unclaimed yield");
    }
}
```
