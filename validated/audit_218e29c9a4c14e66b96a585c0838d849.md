### Title
`respond_ckd` Accepts Unconstrained CKD Output for `AppPublicKey` Requests, Enabling a Single Byzantine Participant to Forge Derived Keys — (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_ckd` function in the MPC contract performs on-chain output verification only when the request uses the `AppPublicKeyPV` variant. For `AppPublicKey` requests, the contract performs **no verification whatsoever** of the submitted `CKDResponse`, allowing a single malicious attested participant (the leader node) to submit an arbitrary response that the contract accepts unconditionally. This is a direct structural analog to the `is_buffer_start` flag not being enforced: `AppPublicKeyPV` activates the constraint, but `AppPublicKey` silently bypasses it.

### Finding Description

In `respond_ckd`, the contract branches on the `app_public_key` type:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the response against the app's public key and the domain's root key. For `AppPublicKey`, the arm is an empty block — any `CKDResponse` with arbitrary `big_y` and `big_c` values is accepted and forwarded to the user via `resolve_yields_for`. [2](#0-1) 

`respond_ckd` is callable by any single attested participant (enforced by `assert_caller_is_attested_participant_and_protocol_active`). [3](#0-2) 

The contract does not verify that the submitted response was produced by threshold-many nodes running the CKD protocol. The leader is the sole caller of `respond_ckd`, and for `AppPublicKey` requests, it faces zero on-chain constraint on what it submits.

**Exploit path:**
1. A malicious attested participant waits to become the leader for a `request_app_private_key` call that uses `AppPublicKey`.
2. It calls `respond_ckd` with a strategically chosen `CKDResponse` — e.g., `big_y = g^x` for a scalar `x` it knows — without running the CKD protocol at all.
3. The contract's `match` arm for `AppPublicKey` is an empty block; no check fires.
4. `resolve_yields_for` delivers the forged response to the waiting user yield. [4](#0-3) 

### Impact Explanation

The user receives a forged CKD output. If the malicious leader chose `big_y = g^x` for a known scalar `x`, the user's application derives a private key that the attacker already knows. This constitutes **unauthorized confidential key derivation output** — the attacker gains control of the user's derived key without the required threshold-participant authorization. Any funds or assets controlled by that derived key are at risk of theft. This maps directly to the Critical impact category: "confidential key derivation output without the required participant authorization."
