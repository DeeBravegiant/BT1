Audit Report

## Title
Insufficient Precision in `rewardRate` Causes Permanent Yield Loss for Stakers - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool.sol` computes `rewardRate` via unscaled integer division in `notifyRewardAmount()`, permanently discarding the truncation remainder. More critically, `rewardPerToken()` rounds to zero for short time deltas when `totalKernelStaked` is large relative to `rewardRate`, and the `updateReward` modifier advances `updatedAt` unconditionally — permanently consuming time windows without recording any reward accrual.

## Finding Description

**Root cause 1 — `notifyRewardAmount()` line 580:**

```solidity
rewardRate = receivedAmount / duration;
```

`rewardRate` is stored as raw token-units-per-second with no scaling multiplier. The remainder `receivedAmount % duration` is silently discarded. There is no `recoverERC20` or equivalent admin function in the contract, so these tokens are permanently locked.

**Root cause 2 — `rewardPerToken()` lines 412–413:**

```solidity
return rewardPerTokenStored
    + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

When `rewardRate` is small (e.g., USDC with 6 decimals) and `totalKernelStaked` is large (e.g., 1e24), the numerator `rewardRate * Δt * 1e18` is less than `totalKernelStaked` for small Δt, causing the entire expression to truncate to **zero**.

**Critical interaction — `updateReward` modifier lines 232–241:**

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // stays unchanged (zero delta)
    updatedAt = lastTimeRewardApplicable();    // ADVANCES regardless
    ...
}
```

`updatedAt` advances even when `rewardPerToken()` returns the same value. The rewards accrued during that interval are permanently unrecoverable — the time window is consumed but no `rewardPerTokenStored` increase is recorded. This modifier fires on every call to `stake()` (line 281), `initiateWithdrawal()` (line 320), and `getReward()` (line 382) by any unprivileged user.

## Impact Explanation

**Permanent freezing of unclaimed yield (Medium).**

Every call to any state-changing function triggers `updateReward`, which advances `updatedAt` without increasing `rewardPerTokenStored` whenever the per-second accrual rounds to zero. The reward tokens transferred into the contract by the admin via `notifyRewardAmount()` remain locked with no recovery path. The fraction lost scales with interaction frequency and the ratio `totalKernelStaked / (rewardRate × 1e18)`.

## Likelihood Explanation

- `KernelDepositPool` accepts an arbitrary `rewardsToken` set at `initialize()` with no decimal restriction (line 270). A 6-decimal token (USDC, USDT) is a realistic and common choice.
- `totalKernelStaked` grows naturally as users stake; 1 million KERNEL tokens (1e24 units) is a realistic TVL.
- With the PoC parameters, `rewardPerToken()` rounds to zero for any Δt shorter than ~605 seconds (~10 minutes). In an active pool, user interactions more frequent than this threshold permanently discard yield.
- No admin action can prevent this once the reward period is live; the only mitigation would be to prevent all user interactions, which defeats the protocol's purpose.

## Recommendation

Scale `rewardRate` by a precision multiplier (e.g., `1e27`) in `notifyRewardAmount()`:

```solidity
uint256 internal constant RATE_PRECISION = 1e27;

// in notifyRewardAmount():
rewardRate = receivedAmount * RATE_PRECISION / duration;

// in rewardPerToken():
return rewardPerTokenStored
    + (rewardRate * (lastTimeRewardApplicable() - updatedAt))
    / totalKernelStaked;

// in earned():
return (balanceOf[_account]
    * (rewardPerToken() - userRewardPerTokenPaid[_account])
    / RATE_PRECISION)
    + rewards[_account];
```

This eliminates both the `notifyRewardAmount` truncation remainder and the `rewardPerToken` rounding-to-zero problem.

## Proof of Concept

**Setup:**
- `rewardsToken` = USDC (6 decimals)
- `receivedAmount` = 1,000 USDC = `1_000_000_000` (1e9)
- `duration` = 7 days = 604,800 seconds
- `totalKernelStaked` = 1,000,000 KERNEL = `1e24` (18 decimals)

**Step 1 — `rewardRate` truncation (line 580):**
```
rewardRate = 1_000_000_000 / 604_800 = 1653
Distributed = 1653 × 604_800 = 999_734_400
Permanently stuck = 265_600 ≈ 0.27 USDC
```

**Step 2 — `rewardPerToken()` rounding to zero (lines 412–413):**
```
For Δt = 1 second:
  numerator   = 1653 * 1 * 1e18 = 1.653e21
  denominator = 1e24
  result      = 0  (truncated)

Minimum Δt for non-zero result:
  1653 * Δt * 1e18 >= 1e24
  Δt >= 1e6 / 1653 ≈ 605 seconds
```

**Step 3 — `updateReward` consumes the time window (lines 232–241):**

Any user calls `stake()`, `initiateWithdrawal()`, or `getReward()` within a 605-second window. `updatedAt` advances to `block.timestamp`, but `rewardPerTokenStored` is unchanged. The rewards for that interval are permanently lost.

**Foundry test plan:**
1. Deploy `KernelDepositPool` with USDC as `rewardsToken`.
2. Stake 1e24 KERNEL tokens.
3. Call `notifyRewardAmount(1e9)` with `duration = 604800`.
4. Warp forward 300 seconds; call `stake(1)` (triggers `updateReward`).
5. Warp forward another 300 seconds; call `getReward()`.
