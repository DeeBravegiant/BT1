Now I have enough context to analyze the full vulnerability. Let me look at one more critical piece — how the `trade_prices_list` filtering interacts with the NFT puzzle enforcement.

### Title
Integer Division Truncation in NFT Royalty Calculation Enables Complete Royalty Bypass in Offer Settlement - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
`compute_royalty_amount` uses integer division that silently truncates to zero when the offered amount is small relative to the royalty percentage. The wallet then skips the royalty payment entirely and passes an empty `trade_prices_list` to the NFT ownership transfer puzzle. Because the puzzle accepts an empty `trade_prices_list` (confirmed by the lifecycle test), the NFT transfers with no royalty paid to the creator.

### Finding Description
`compute_royalty_amount` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

When `abs(offered_amount) * percentage < MAX_ROYALTY_BASIS_POINTS`, integer division floors the result to zero. The project's own test suite explicitly asserts this behavior:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

Downstream in `make_nft1_offer`, three separate guards all gate on the royalty being non-zero:

1. **Announcement assertion skipped** — `AssertPuzzleAnnouncement` entries are only added `if p.amount > 0`, so no cross-spend announcement enforces the royalty payment. [3](#0-2) 

2. **Royalty coin not funded** — the extra output to `OFFER_MOD_HASH` is only created when `payment_sum > 0`. [4](#0-3) 

3. **Royalty spend skipped entirely** — the royalty `CoinSpend` is skipped when the total payment is zero. [5](#0-4) 

For the seller-side path (offering an NFT, requesting fungible assets), trade prices are filtered out with the same truncation check:

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [6](#0-5) 

This passes an empty `trade_prices_list` to the NFT ownership transfer puzzle. The lifecycle test confirms the puzzle accepts `[-10, [], [], []]` (empty trade prices, empty DID) without error:

```python
empty_spend = make_spend(
    new_ownership_coin,
    new_ownership_puzzle,
    Program.to([[[51, ACS_PH, 1], [-10, [], [], []]]]),
)
result = await sim_client.push_tx(empty_bundle)
assert result == (MempoolInclusionStatus.SUCCESS, None)
``` [7](#0-6) 

The same truncation exists in the `royalty_calculation` static method used by the `nft_calculate_royalties` RPC, meaning third-party clients relying on this API also receive a zero royalty figure:

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
``` [8](#0-7) 

### Impact Explanation
An unprivileged buyer can craft a valid offer bundle where the fungible amount offered is below the truncation threshold. The resulting spend bundle is accepted on-chain with no royalty coin created. The NFT creator's on-chain royalty entitlement — encoded in the curried `royalty_percentage` of the NFT ownership transfer puzzle — is silently bypassed. This is an unauthorized accounting change affecting XCH/CAT settlement for NFT trades.

Truncation thresholds (single NFT, `royalty_split=1`):
- 1% royalty (`percentage=100`): any offer below **100 mojos** pays zero royalty
- 0.01% royalty (`percentage=1`): any offer below **10,000 mojos** pays zero royalty

### Likelihood Explanation
The attack requires the NFT owner to accept a low-value offer. However, the royalty bypass is invisible to the seller — they see only the offered price, not whether the royalty coin was included. For NFTs priced in the sub-10,000 mojo range (common for test or low-value assets), or for NFTs with very small royalty percentages, the threshold is easily reached with economically plausible offer amounts.

### Recommendation
In `compute_royalty_amount`, after computing the integer-divided amount, add a guard that raises (or rounds up to 1) when the percentage is non-zero but the computed royalty is zero:

```python
if percentage > 0 and amount == 0:
    raise ValueError("Royalty amount truncated to zero; offered amount is too small for this royalty percentage")
```

Apply the same guard in `royalty_calculation`. This prevents offers that would silently bypass royalties from being constructed or reported via the RPC.

### Proof of Concept
1. Mint an NFT with `royalty_percentage = 100` (1%).
2. As the buyer, call `make_nft1_offer` with `offer_dict = {None: -50, nft_asset_id: 1}` (offering 50 mojos for the NFT).
3. `compute_royalty_amount(-50, 1, 100)` → `50 // 1 * 100 // 10000 = 0`.
4. `payment_sum = 0` → no royalty output is created, no announcement is asserted, royalty spend is skipped.
5. The NFT owner accepts the offer; the spend bundle is submitted and confirmed on-chain.
6. The NFT creator's royalty address receives 0 mojos despite the 1% royalty being encoded in the puzzle. [1](#0-0) [9](#0-8)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
