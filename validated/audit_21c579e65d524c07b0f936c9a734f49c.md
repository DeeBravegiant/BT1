### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Enables Queue-Saturation DoS on Foreign-Chain Verification - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary

The `VerifyForeignTransactionRequest` map key does not include the caller's account ID, unlike `SignatureRequest` and `CKDRequest`. Combined with the hard cap of 128 concurrent yields per key (`MAX_PENDING_REQUEST_FAN_OUT`), an unprivileged attacker can saturate the queue for any specific foreign-chain transaction by submitting 128 cheap duplicate calls, causing every subsequent legitimate caller trying to verify the same transaction to receive a `PendingRequestQueueFull` panic for approximately 200 NEAR blocks.

### Finding Description

`VerifyForeignTransactionRequest` is defined as:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

It contains no `predecessor_account_id` field. By contrast, `SignatureRequest` is constructed with `&predecessor` and `CKDRequest` derives its `app_id` from `derive_app_id(predecessor_id, derivation_path)`, making both of those keys caller-specific. [2](#0-1) [3](#0-2) 

The fan-out queue is capped at 128 entries per key:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
// ...
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
}
``` [4](#0-3) 

The existing test explicitly documents that different callers share the same queue entry:

```
// And: caller bob submits the identical request — a different account would today
// be blocked from receiving a response by alice's submission.
// Then: both yields are queued under the single (caller-agnostic) request key.
``` [5](#0-4) 

**Attack path:**

1. Attacker identifies a foreign-chain transaction (e.g., a specific Bitcoin `tx_id`) that a victim intends to verify via `verify_foreign_transaction`.
2. Attacker submits 128 calls to `verify_foreign_transaction` with the identical `(domain_id, payload_version, request)` tuple, paying only 128 yoctonear total (the minimum deposit per call).
3. `push_pending_yield` fills the `pending_verify_foreign_tx_requests` queue to capacity for that key.
4. Every subsequent call from any account with the same parameters hits the `PendingRequestQueueFull` panic and reverts.
5. The attacker's 128 yields time out after ~200 NEAR blocks (the yield-resume window), at which point the queue drains via `pop_oldest_pending_yield` in `return_verify_foreign_tx_and_clean_state_on_success`. [6](#0-5) [7](#0-6) 

### Impact Explanation

Any user attempting to verify a specific foreign-chain transaction is blocked for ~200 NEAR blocks (~200 seconds at 1 block/second). In a bridge context, this delays or permanently prevents the victim from completing a cross-chain operation tied to that transaction. Because `verify_foreign_transaction` is the gateway for foreign-chain transaction verification and the MPC nodes will only sign after a successful verification, blocking this call breaks the request lifecycle for that transaction. This maps to the allowed Medium impact: *"request-lifecycle manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

The attack requires no privileged access, no threshold collusion, and no leaked keys. The cost is 128 yoctonear plus gas — economically negligible. The attacker only needs to observe the mempool or know the target transaction ID in advance (trivially available for public blockchains like Bitcoin). The 200-block window is long enough to be operationally disruptive for time-sensitive bridge operations.

### Recommendation

Include `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring the approach used by `SignatureRequest` and `CKDRequest`. This makes each queue slot caller-specific, so an attacker can only saturate their own queue entries, not those of other users.

Alternatively, enforce a per-account submission rate limit or require a meaningful deposit that scales with the number of concurrent pending requests for the same key.

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
