Audit Report

## Title
Publicly Callable `updateRSETHPrice()` Enables Sandwich Attack to Extract Yield from rsETH Holders - (File: contracts/LRTOracle.sol)

## Summary

`LRTOracle.updateRSETHPrice()` is unrestricted (`public`, no role check) and updates `rsETHPrice`, the single variable used to price both deposits in `LRTDepositPool` and instant redemptions in `LRTWithdrawalManager`. An unprivileged attacker can atomically deposit at the stale low price, trigger the price update themselves, and immediately redeem at the new higher price — extracting accrued yield that belongs to all existing rsETH holders. The `pricePercentageLimit` guard does not prevent this because normal reward accrual produces price increases well within any reasonable threshold.

## Finding Description

`LRTOracle.updateRSETHPrice()` carries no access control:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

`_updateRsETHPrice()` computes `newRsETHPrice = totalETHInProtocol / rsethSupply` and writes it to storage:

```solidity
rsETHPrice = newRsETHPrice;
``` [2](#0-1) 

The only guard for non-manager callers is the `pricePercentageLimit` check, which only fires when the price increase exceeds the configured threshold relative to `highestRsethPrice`:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
if (isPriceIncreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        revert PriceAboveDailyThreshold();
    }
}
``` [3](#0-2) 

Normal staking reward accrual (e.g., ~0.003% per day for ETH staking) produces price increases far below any operationally reasonable `pricePercentageLimit`, so the guard does not trigger and the public caller succeeds. If `pricePercentageLimit == 0`, there is no guard at all.

`LRTDepositPool.getRsETHAmountToMint()` reads the live `rsETHPrice` at deposit time:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

`LRTWithdrawalManager.getExpectedAssetAmount()` also reads the live `rsETHPrice` at redemption time:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [5](#0-4) 

`instantWithdrawal()` is a same-block redemption path with no delay — it burns rsETH and transfers assets immediately:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
``` [6](#0-5) 

Because all three operations read `rsETHPrice` at execution time with no cooldown, snapshot isolation, or block delay between deposit and instant withdrawal, the attacker can compose them atomically in a single block or multicall.

**Why existing checks fail:**
- `pricePercentageLimit`: Only blocks large single-step price jumps. Normal reward accrual is always within the limit. The attack is profitable for any non-zero price gap, not just large ones.
- `instantWithdrawalFee`: Reduces but does not eliminate profit; the fee is manager-controlled and may be zero or low.
- `nonReentrant` on deposit and withdrawal: Prevents reentrancy within a single call, but the three-step sequence uses separate top-level calls, so reentrancy guards are irrelevant.
- `minRsEthAmountToWithdraw`: A minimum size floor, not a timing or price-staleness guard.

## Impact Explanation

**High — Theft of unclaimed yield.**

When staking rewards accrue, `totalETHInProtocol` grows above `rsethSupply × rsETHPrice`. Until `updateRSETHPrice()` is called, `rsETHPrice` is stale. An attacker who deposits at the stale price receives more rsETH than the fair post-reward price would grant. After triggering the price update themselves, they redeem at the inflated price, capturing yield that should have been distributed pro-rata to all existing rsETH holders. This is a direct, concrete transfer of value from honest holders to the attacker, matching the allowed impact "Theft of unclaimed yield."

## Likelihood Explanation

**Medium.** Prerequisites are all routinely satisfied during normal protocol operation:
- `isInstantWithdrawalEnabled[asset] == true` for at least one asset (a normal operational state when instant withdrawals are live).
- Sufficient assets in `LRTUnstakingVault` for the instant redemption.
- A non-zero gap between the last `updateRSETHPrice()` call and the current TVL (rewards accrue continuously).

The attacker needs no special role — only capital and the ability to submit three transactions in sequence or a single multicall. The attack is repeatable every reward accrual cycle (daily or more frequently).

## Recommendation

1. **Restrict `updateRSETHPrice()` to a privileged role** (e.g., `OPERATOR_ROLE` or `MANAGER`), removing the ability for an unprivileged caller to self-trigger the price update. The existing `updateRSETHPriceAsManager()` already demonstrates the intended privileged path.
2. **Alternatively, enforce a per-address cooldown** between deposit and instant withdrawal (e.g., require that `instantWithdrawal` cannot be called in the same block as a deposit from the same address).
3. **Snapshot the rsETH price at deposit initiation** and use that snapshot for the corresponding withdrawal, rather than reading the live price at redemption time.

## Proof of Concept

**Setup:**
- `rsETHPrice` = 1.00e18 (stale; rewards have accrued, true price = 1.01e18)
- Protocol TVL: 10,000 ETH backing 10,000 rsETH (at stale price)
- `isInstantWithdrawalEnabled[ETH_TOKEN]` = `true`
- `LRTUnstakingVault` holds ≥ 101 ETH available for instant withdrawal
- `pricePercentageLimit` = 1e16 (1%) — the 1% price jump is within limit for this scenario

**Block N (single multicall or sequential transactions):**

1. Attacker calls `LRTDepositPool.depositETH{value: 100e18}(0, "")`.
   - `rsethAmountToMint = 100e18 * 1e18 / 1.00e18 = 100e18` rsETH minted to attacker.
   - Protocol now holds 10,100 ETH backing 10,100 rsETH.

2. Attacker calls `LRTOracle.updateRSETHPrice()`.
   - `totalETHInProtocol` = 10,100 ETH (includes attacker's deposit + accrued rewards).
   - `newRsETHPrice = 10,100e18 / 10,100e18 ≈ 1.0099e18` (diluted but still above 1.00e18).
   - Price increase ≈ 0.99% — within a 1% `pricePercentageLimit`, so no revert.
   - `rsETHPrice` updated to ~1.0099e18.

3. Attacker calls `LRTWithdrawalManager.instantWithdrawal(ETH_TOKEN, 100e18, "")`.
   - `assetAmountUnlocked = 100e18 * 1.0099e18 / 1e18 ≈ 100.99 ETH`.
   - Attacker receives ~100.99 ETH (minus any `instantWithdrawalFee`), having deposited 100 ETH.

**Result:** Attacker extracts ~0.99 ETH of yield that belonged to all existing rsETH holders, with no privileged access. The attack is repeatable every reward cycle. For larger deposit amounts relative to protocol TVL, the dilution effect decreases and profit approaches `depositAmount × (P_new − P_old) / P_old`.

**Foundry test plan:** Fork mainnet, set `isInstantWithdrawalEnabled[ETH_TOKEN] = true`, simulate reward accrual by directly crediting ETH to the deposit pool (or advancing time to accrue EigenLayer rewards), then execute the three-step sequence from an unprivileged address and assert the attacker's ETH balance after step 3 exceeds the balance before step 1.
