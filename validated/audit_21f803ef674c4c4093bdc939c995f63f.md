Audit Report

## Title
`KernelDepositPool::rewardPerToken()` permanently locks rewards when `totalKernelStaked` drops to zero mid-period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
When the last staker calls `initiateWithdrawal`, `totalKernelStaked` drops to zero while `rewardRate` and `finishAt` remain active. The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `rewardPerToken()` skips accumulation due to the zero-supply guard. Any rewards that accrued during the zero-staked window are permanently unrecoverable, as there is no admin sweep or recovery function.

## Finding Description
`rewardPerToken()` (L408–414) returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`. The `updateReward` modifier (L232–242) unconditionally sets `updatedAt = lastTimeRewardApplicable()` regardless of whether accumulation occurred. When `initiateWithdrawal` (L320–337) reduces `totalKernelStaked` to zero, the modifier fires correctly for the exiting staker, then advances `updatedAt`. Every subsequent `updateReward` call during the zero-staked window silently consumes elapsed time without accumulating `rewardPerTokenStored`. When a new staker eventually arrives, `updatedAt` has already advanced past the gap, and the rewards for that entire window are irrecoverably lost in the contract. The `notifyRewardAmount` guard (`if (totalKernelStaked == 0) revert NoStakedTokens()`, L570) only prevents starting a new period with zero stakers; it does not prevent mid-period drain. There is no admin rescue or sweep function anywhere in the contract.

The contract's own NatSpec (L18–22) acknowledges this behavior and states the mitigation is purely operational ("ensuring there are always some tokens staked … for the entire duration of the reward period"), but this invariant is not enforced on-chain. Any single unprivileged user can break it.

## Impact Explanation
Rewards tokens (`rewardsToken`) that accrued during the zero-staked window are permanently locked in `KernelDepositPool` with no recovery path. This constitutes **Medium — Permanent freezing of unclaimed yield**, matching the allowed impact scope. The locked amount scales with `rewardRate × zero-staked duration`, which can be up to `rewardRate × 30 days` given the maximum `withdrawalDelay`.

## Likelihood Explanation
No privileged access or coordination is required. Any ordinary staker who is the last remaining depositor can call `initiateWithdrawal` during an active reward period, immediately reducing `totalKernelStaked` to zero. The `withdrawalDelay` (up to 30 days per `MAX_WITHDRAWAL_DELAY`) guarantees a window during which rewards are lost. The action is a standard user workflow, is non-reversible, and can be repeated across reward periods.

## Recommendation
Only advance `updatedAt` when `totalKernelStaked > 0`, so the reward clock pauses during zero-staked windows and resumes correctly when staking resumes:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures rewards for the zero-staked period are distributed to returning stakers rather than silently discarded.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1000e18)` with one staker present. `rewardRate = 1000e18 / duration`, `finishAt = block.timestamp + duration`.
2. The sole staker calls `initiateWithdrawal(totalKernelStaked)`. `updateReward` fires: `rewardPerTokenStored` is correctly updated for the staker's earned share; `updatedAt = block.timestamp`. `totalKernelStaked → 0`.
3. `withdrawalDelay` seconds elapse (e.g., 7 days). `rewardRate` is still active; `rewardPerToken()` returns `rewardPerTokenStored` unchanged on every call.
4. A new staker calls `stake(1)`. `updateReward` fires: `rewardPerToken()` returns `rewardPerTokenStored` (zero-supply branch, no change), but `updatedAt` advances to `block.timestamp`. The 7-day reward tranche (`rewardRate × 7 days`) is permanently lost.
5. Original staker calls `claimWithdrawal` — tokens returned, but the 7-day rewards remain locked in the contract with no recovery path.

Foundry test outline:
```solidity
function test_rewardsFrozenOnZeroStake() public {
    // setup: mint tokens, stake, notifyRewardAmount
    vm.warp(block.timestamp + 1 days);
    pool.initiateWithdrawal(totalStaked); // totalKernelStaked → 0
    vm.warp(block.timestamp + 7 days);   // 7-day gap, rewards lost
    pool.stake(1);                        // updatedAt advances, gap consumed
    vm.warp(block.timestamp + duration);
    // assert: rewardsToken.balanceOf(pool) > 0 (locked forever)
    // assert: no user can claim the 7-day tranche
}
```
