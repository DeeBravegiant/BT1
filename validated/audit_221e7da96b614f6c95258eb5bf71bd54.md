### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund Without Cross-Sub-Transaction Validation - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`)

---

### Summary

In the Dijkstra era, `dijkstraTotalRefundsTxCerts` computes deposit refunds for a sub-transaction's certificate list by blindly summing the `deposit` field from every `UnRegDepositTxCert` / `UnRegDRepTxCert` certificate, without verifying that the referenced credential is actually registered in the current ledger state. Because each sub-transaction's balance is checked independently using this function, two sub-transactions in the same batch can each declare `UnRegDepositTxCert stakingCred D` for the same credential and each have their individual balance equation satisfied, allowing the same deposit `D` to be claimed twice (or N times). The developers have already identified this gap and left a disabled test with the comment `"TODO: predicate failure not yet implemented"`.

---

### Finding Description

**Root cause — `dijkstraTotalRefundsTxCerts` performs no existence check:** [1](#0-0) 

```haskell
-- | Unlike previous eras, we no longer need to lookup refunds from the ledger
-- state, since all of the certificates specify the actual refund and ledger
-- rules will validate that they are accurate.
dijkstraTotalRefundsTxCerts :: (Foldable f, ConwayEraTxCert era) => f (TxCert era) -> Coin
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert  _ deposit -> deposit
  _ -> zero
```

The comment asserts "ledger rules will validate that they are accurate," but no such cross-sub-transaction guard exists. The function is a pure fold over the certificate list; it never consults the `CertState` to confirm the credential is registered.

**How the function is wired into sub-transaction balance accounting:**

Each sub-transaction is processed by `dijkstraSubUtxoTransition`, which calls `Shelley.updateUTxOStateNoFees`. That helper computes:

```
depositChange = totalDeposits <-> totalRefunds
utxosDeposited' = utxosDeposited <> depositChange
```

where `totalRefunds = certsTotalRefundsTxBody pp certState txBody` ultimately resolves to `dijkstraTotalRefundsTxCerts` for the Dijkstra era. [2](#0-1) 

Because each sub-transaction's UTxO state is updated independently, and `dijkstraTotalRefundsTxCerts` does not check whether the credential is still registered after a prior sub-transaction already deregistered it, the deposit pot is decremented once per sub-transaction that carries the certificate — regardless of whether the credential still exists.

**Contrast with Conway/Shelley, which do check registration:**

In all prior eras, `shelleyTotalRefundsTxCerts` and `conwayDRepRefundsTxCerts` accept a `lookupDeposit` function that returns `Nothing` for unregistered credentials, so an unregistered credential contributes zero to the refund total: [3](#0-2) 

The Dijkstra era deliberately removed this lookup (the comment says it is no longer needed), but the compensating cross-sub-transaction guard was never implemented.

**Developers' own acknowledgement — disabled test:** [4](#0-3) 

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = ... & subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

The test is disabled (`xit`) precisely because the predicate failure that should reject this transaction has not been implemented. The `error "TODO: predicate failure not yet implemented"` placeholder confirms the transaction currently succeeds when it should be rejected.

---

### Impact Explanation

An attacker who has registered a staking credential with deposit `D` can construct a Dijkstra-era top-level transaction containing `N` sub-transactions, each carrying `UnRegDepositTxCert stakingCred D`. Each sub-transaction's individual balance equation is satisfied because `dijkstraTotalRefundsTxCerts` counts `D` as a refund for each one. The deposit pot (`utxosDeposited`) is decremented by `D` for every sub-transaction, while the attacker's UTxO outputs collectively receive `N × D` ADA. Only `D` ADA was ever deposited. The net effect is creation of `(N-1) × D` ADA from nothing, directly violating the preservation-of-value invariant.

This matches the **Critical** impact category: *Direct creation of ADA through an invalid ledger state transition.*

---

### Likelihood Explanation

- The Dijkstra era is the newest era in this repository and introduces sub-transactions as a novel primitive.
- The attack requires only an unprivileged transaction sender: register a credential (paying deposit `D`), then submit a single crafted transaction.
- No special privileges, governance majority, or external oracle manipulation is required.
- The attack is fully deterministic and atomic within one transaction.

---

### Recommendation

1. **Implement the missing cross-sub-transaction guard.** Before processing sub-transactions, collect all `UnRegDepositTxCert` / `UnRegDRepTxCert` credentials across the entire batch and verify that no credential appears more than once. Emit a new predicate failure (e.g., `DuplicateUnRegDepositInBatch`) if a duplicate is found. This is the direct fix the disabled test is waiting for.

2. **Restore registration-state lookup in `dijkstraTotalRefundsTxCerts`.** The function should accept a `lookupDeposit` function (as the Conway/Shelley versions do) so that a credential that is no longer registered at the time of the sub-transaction contributes zero to the refund total, preventing the deposit pot from being decremented for a non-existent registration.

3. **Enable and complete the test** at `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53` once the predicate failure is implemented.

### Proof of Concept

```
1. Alice registers stakingCred, paying keyDeposit D.
   → CertState: stakingCred ↦ AccountState{deposit=D}
   → utxosDeposited increases by D

2. Alice constructs a Dijkstra top-level transaction with two sub-transactions:
     subTx1: certs = [UnRegDepositTxCert stakingCred D]
             inputs = {utxo1}  outputs = {addr: value1 + D}
     subTx2: certs = [UnRegDepositTxCert stakingCred D]
             inputs = {utxo2}  outputs = {addr: value2 + D}

3. SUBLEDGERS processes subTx1:
   - dijkstraTotalRefundsTxCerts [UnRegDepositTxCert stakingCred D] = D
   - updateUTxOStateNoFees: utxosDeposited -= D  ✓ (balance satisfied)
   - (If DELEG is applied: stakingCred deregistered)

4. SUBLEDGERS processes subTx2:
   - dijkstraTotalRefundsTxCerts [UnRegDepositTxCert stakingCred D] = D
     (no registration check — returns D regardless)
   - updateUTxOStateNoFees: utxosDeposited -= D again  ✓ (balance satisfied)
   - No predicate failure is raised (the guard is not implemented)

5. Result: Alice's outputs contain value1 + D + value2 + D.
   She spent value1 + value2 from UTxO inputs and only ever deposited D.
   Net gain: D ADA created from nothing.
   utxosDeposited has been decremented by 2D for a single deposit of D.
```
