Audit Report

## Title
Permissionless `updateRSETHPrice()` with Stale Price in Deposit Minting Enables Selective Yield Capture - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is callable by any unprivileged user, and `LRTDepositPool.getRsETHAmountToMint()` uses the stored `rsETHPrice` rather than a freshly computed value. Because the new price is fully simulatable from public on-chain state before execution, an attacker can always choose whether to deposit before or after the price update to maximize their rsETH allocation, capturing yield that belongs to existing rsETH holders.

## Finding Description
`updateRSETHPrice()` is declared `public whenNotPaused`, making it callable by any EOA or contract.

`_updateRsETHPrice()` computes the new price deterministically from three fully public inputs: `_getTotalEthInProtocol()` (itself composed of `getTotalAssetDeposits()` per asset and `getAssetPrice()` per asset — all public view functions), `IRSETH.totalSupply()`, and `lrtConfig.protocolFeeInBPS()`. No private or privileged state is involved.

`getRsETHAmountToMint()` computes `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`, where `rsETHPrice` is the stored value last written by a prior `updateRSETHPrice()` call. `_beforeDeposit()` calls this function with no forced price refresh.

**Exploit path (price-rise case — most common):**
1. Rewards accrue; `totalETHInProtocol` grows above `previousTVL`. The stored `rsETHPrice` is stale (lower than the true current value).
2. Attacker reads `rsETHPrice`, simulates `newRsETHPrice` using the same arithmetic as `_updateRsETHPrice()` from public state.
3. `simulatedNewPrice > storedPrice` → attacker calls `depositETH` (or `depositAsset`) at the stale lower price, receiving more rsETH per ETH than the protocol intends.
4. Attacker then calls `updateRSETHPrice()` to lock in the new price.

**Exploit path (price-fall case):**
1. An asset oracle price drops; `newRsETHPrice < storedPrice`.
2. Attacker calls `updateRSETHPrice()` first to write the lower price, then deposits at the lower price, again receiving more rsETH per ETH than existing holders paid.

**Why existing guards fail:**
- `minRSETHAmountExpected` in `depositETH`/`depositAsset` is a caller-supplied slippage floor; it does not prevent the attacker from receiving *more* than fair value.
- The `pricePercentageLimit` guard at lines 256–265 actively worsens the situation: when accumulated rewards push the price increase above the threshold, non-managers are *blocked* from calling `updateRSETHPrice()`, causing the stale price to persist longer and widening the exploitation window.
- There is no atomic price refresh inside `_beforeDeposit()`.

## Impact Explanation
Every rsETH minted to the attacker at a stale lower price is backed by the same ETH pool as existing holders' rsETH. The attacker's excess rsETH dilutes the per-token ETH backing of all existing holders, directly reducing the yield they would otherwise have received when the price is next updated. This is **theft of unclaimed yield** (High severity). The magnitude scales linearly with deposit size and price staleness: a 0.1% daily reward rate on a 1,000 ETH deposit yields approximately 1 ETH of excess rsETH per cycle, repeatable every reward period.

## Likelihood Explanation
The attack requires no privileged access, no special infrastructure, and no external dependencies. All inputs needed to simulate the price update are readable in a single multicall of public view functions. The attack is repeatable every time rewards accrue before `rsETHPrice` is refreshed. The `pricePercentageLimit` guard can extend the staleness window to multiple days. Likelihood is **High**.

## Recommendation
Force an atomic price update inside `_beforeDeposit()` before computing `rsethAmountToMint`, or replace the `lrtOracle.rsETHPrice()` read in `getRsETHAmountToMint()` with a freshly computed value (e.g., expose a `computeRSETHPrice()` view function that mirrors `_updateRsETHPrice()` arithmetic without writing state). Alternatively, restrict `updateRSETHPrice()` to an authorized keeper and enforce a maximum staleness window so the exploitable gap is bounded.

## Proof of Concept
**Minimal Foundry fork test outline:**

```solidity
// Fork mainnet at block B where rewards have accrued since last updateRSETHPrice()
// 1. Record storedPrice = oracle.rsETHPrice()
// 2. Simulate newPrice using _getTotalEthInProtocol() inputs + fee formula
// 3. Assert simulatedNewPrice > storedPrice (rewards accrued)
// 4. Attacker deposits D ETH → receives rsethA = D * 1e18 / storedPrice
// 5. Call oracle.updateRSETHPrice()
// 6. Compute fair amount: rsethFair = D * 1e18 / newPrice
// 7. Assert rsethA > rsethFair  (attacker received excess rsETH)
// 8. Compute existing holder's rsETH ETH-backing before and after:
//    backingBefore = totalETH_before / totalSupply_before
//    backingAfter  = totalETH_after  / totalSupply_after
// 9. Assert backingAfter < backingBefore  (existing holders diluted)
```

The test demonstrates that the attacker captures yield proportional to `(newPrice - storedPrice) / newPrice * depositAmount`, which is directly subtracted from existing holders' accrued yield.

```text
    function _updateRsETHPrice() internal {
        address rsETHTokenAddress = lrtConfig.rsETH();
        uint256 rsethSupply = IRSETH(rsETHTokenAddress).totalSupply();

        if (rsethSupply == 0) {
            rsETHPrice = 1 ether;
            highestRsethPrice = 1 ether;
            return;
        }

        if (highestRsethPrice == 0) {
            highestRsethPrice = rsETHPrice;
        }

        uint256 previousPrice = rsETHPrice;

        // get total ETH in the protocol (normalized to 1e18)
        uint256 totalETHInProtocol = _getTotalEthInProtocol();

        // calculate previousTVL using rsethSupply multiplied by rsETHPrice
        uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);

        IPausable lrtDepositPool = IPausable(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IPausable withdrawalManager = IPausable(lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER));

        // determine if the protocol is active (not paused)
        bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;

        // only take fee if TVL increased and protocol is not paused
        uint256 protocolFeeInETH = 0;
        if (!protocolPaused && totalETHInProtocol > previousTVL) {
            uint256 rewardAmount = totalETHInProtocol - previousTVL;
            protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
        }

        // compute new rsETH price based on total ETH minus fee
        uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```
