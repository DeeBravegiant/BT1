Unprivileged `directDeposits` Can Front-Run and Block Stake Key Deregistration and Legacy-Mode Withdrawals - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

### Summary

In the Dijkstra era, any unprivileged transaction author can include a `directDeposits` field in their transaction body to increase the reward balance of any registered stake account. The `ConwayUnRegCert` deregistration rule requires the reward balance to be exactly zero before proceeding. An attacker can front-run a victim's deregistration transaction by submitting a `directDeposits` transaction that adds even 1 lovelace to the victim's account, causing the deregistration to fail with `StakeKeyHasNonZeroAccountBalanceDELEG`. The same mechanism blocks exact-amount withdrawals in legacy mode (Plutus V1/V2 scripts present), where withdrawals must drain the full balance.

---

### Finding Description

**Root cause — `directDeposits` allows any user to increase any registered account's balance:**

In `dijkstraEntitiesTransition`, after certificates are processed, `applyDirectDeposits` is called unconditionally on the `directDeposits` field of the transaction body: [1](#0-0) 

`applyDirectDeposits` adds the specified coin amounts to the balance of any registered account, with no restriction on who the transaction author is: [2](#0-1) 

The only validation is that the target account must be registered (`directDepositsMissingAccounts`): [3](#0-2) 

**Precondition check — deregistration requires zero balance:**

The `ConwayUnRegCert` handler in the DELEG rule enforces that the reward balance must be zero before deregistration is allowed: [4](#0-3) 

The same check exists in the Shelley DELEG rule (used for pre-Conway certs): [5](#0-4) 

**Precondition check — legacy-mode withdrawals must drain the full balance:**

In `validateWithdrawals` (Dijkstra ENTITIES), when `legacyMode` is true (Plutus V1/V2 scripts present), `withdrawalsThatDoNotDrainAccounts` requires the withdrawal amount to equal the exact balance: [6](#0-5) 

`withdrawalsThatDoNotDrainAccounts` enforces strict equality: [7](#0-6) 

**Attack flow:**

1. Alice has a registered stake credential with zero reward balance and submits a `ConwayUnRegCert` transaction to reclaim her deposit.
2. Attacker monitors the mempool, sees Alice's pending deregistration, and submits a transaction containing `directDeposits` crediting 1 lovelace to Alice's account.
3. Attacker's transaction is included in the block before Alice's.
4. Alice's deregistration fails: `StakeKeyHasNonZeroAccountBalanceDELEG (Coin 1)`.

For the withdrawal variant: Alice constructs a Plutus V2 transaction withdrawing exactly X lovelace. Attacker front-runs with a `directDeposit` of 1 lovelace. Alice's withdrawal fails with `IncompleteWithdrawals` because the balance is now X+1 but the withdrawal specifies X.

The `directDeposits` field is defined in `DijkstraTxBodyRaw` and is available to any transaction author: [8](#0-7) 

The same field exists in sub-transactions: [9](#0-8)

### Impact Explanation

**Impact: Medium** — Attacker-controlled transactions modify rewards outside design parameters, blocking deposit refunds and withdrawals.

- An attacker can indefinitely prevent a victim from deregistering their stake credential, locking the victim's key deposit (2 ADA minimum) for as long as the attacker continues front-running. The victim can recover only by first withdrawing the accumulated balance and then deregistering, but the attacker can repeat the attack each time.
- In legacy mode (Plutus V1/V2 scripts), the attacker can prevent exact-amount withdrawals from succeeding, blocking scripts that depend on draining the full reward balance.
- The attacker's net cost per front-run is only the transaction fee (the deposited lovelace goes to the victim's account), making sustained griefing economically viable.
- Recovery does not require a hard fork; the victim can eventually withdraw and deregister if the attacker stops. However, the attack can be sustained indefinitely at low cost.

This matches: *"Medium. Attacker-controlled transactions… modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."*

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attack requires only a standard Dijkstra transaction with a `directDeposits` field targeting the victim's registered stake credential — no special privileges, keys, or scripts are needed.
- The attacker must monitor the mempool for pending deregistration or legacy-mode withdrawal transactions, which is straightforward with a full node.
- The cost per attack is one transaction fee (≈0.17 ADA), making repeated front-running economically viable against any victim whose deposit or withdrawal is worth more than the fee.
- The attack is most effective against automated scripts or DApps that submit deregistration or exact-amount withdrawal transactions without retry logic.

---

### Recommendation

1. **Restrict `directDeposits` to self-deposits only**: Require that each `AccountAddress` in `directDeposits` corresponds to a stake credential controlled by the transaction author (i.e., a witness for that credential must be present in the transaction). This mirrors the witness requirement for withdrawals.

2. **Alternatively, drain the balance before the zero-balance check in deregistration**: Similar to the fix in the referenced report (calling `settleDebt` before the balance check), the DELEG rule could automatically drain the account balance into the UTxO outputs before enforcing the zero-balance precondition, removing the griefing vector entirely.

3. **For legacy-mode withdrawals**: Consider whether the exact-drain requirement is still necessary in the Dijkstra era given that `directDeposits` can externally modify balances between transaction construction and inclusion.

```haskell
directDepositsMissingAccounts ::
  EraAccounts era =>
  DirectDeposits ->
  Accounts era ->
  Maybe DirectDeposits
directDepositsMissingAccounts (DirectDeposits dds) accounts
  | Map.foldrWithKey' checkRegistered True dds = Nothing
  | otherwise = Just $ DirectDeposits $ Map.foldrWithKey' collectMissing Map.empty dds
  where
    isRegistered (AccountAddress _ (AccountId credential)) =
      isAccountRegistered credential accounts
    checkRegistered addr _ acc = acc && isRegistered addr
    collectMissing addr amount acc
      | isRegistered addr = acc
      | otherwise = Map.insert addr amount acc
```
