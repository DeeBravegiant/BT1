### Title
Receipt Size Limit Bypassed via Post-Validation Mutation and Data Receipt Creation - (`runtime/runtime/src/lib.rs`)

### Summary

The `max_receipt_size` protocol limit is validated on a receipt at creation time using `ValidateReceiptMode::NewReceipt`, but the runtime subsequently mutates the receipt by appending `output_data_receivers` (for `promise_return`) or creates `DataReceipt`s from return values that are validated only against `max_length_returned_data` (not total receipt size). In both cases, the final forwarded receipt can exceed `max_receipt_size` without triggering a rejection, violating the hard protocol invariant.

### Finding Description

The `validate_receipt` function in `runtime/runtime/src/verifier.rs` enforces `max_receipt_size` only when `mode == ValidateReceiptMode::NewReceipt`: [1](#0-0) 

However, in `apply_action_receipt` in `runtime/runtime/src/lib.rs`, after new receipts are created and validated, the runtime appends `output_data_receivers` from the parent action receipt to a child receipt selected by `promise_return`: [2](#0-1) 

This mutation happens **after** the child receipt passed its `NewReceipt` size check. If the child receipt was crafted to be exactly at `max_receipt_size`, appending even one `DataReceiver` entry pushes it over the limit with no re-validation.

In the second variant, when a contract returns a value (not a receipt index), the runtime creates a `DataReceipt` wrapping the return value: [3](#0-2) 

This `DataReceipt` is validated only by `validate_data_receipt`, which checks `max_length_returned_data` (4 MB) but **not** the total borsh-serialized receipt size: [4](#0-3) 

A return value of exactly `max_length_returned_data` bytes produces a `DataReceipt` whose total borsh size exceeds `max_receipt_size`. The codebase explicitly acknowledges this as a known bug: [5](#0-4) 

The congestion control layer has a workaround that clamps the reported size to `max_receipt_size` for bandwidth accounting, but the oversized receipt is still forwarded and stored: [6](#0-5) 

### Impact Explanation

Oversized receipts are forwarded across shards and included in `ChunkStateWitness`. The `max_receipt_size` limit (4 MB) is a hard protocol limit designed to keep `ChunkStateWitness` under 17 MiB. Receipts exceeding this limit inflate the witness beyond its intended bound. When incoming receipts are processed, they are validated with `ValidateReceiptMode::ExistingReceipt`, which skips the size check, so the oversized receipt is accepted and executed. The concrete corrupted protocol value is the `max_receipt_size` invariant on forwarded receipts stored in `DBCol::OutgoingReceipts` and included in `ChunkStateWitness`, potentially causing chunk validators to diverge from the chunk producer's result. [7](#0-6) 

### Likelihood Explanation

Any unprivileged user can deploy a contract and trigger this via a signed transaction. Two concrete paths exist: (1) craft a receipt at exactly `max_receipt_size` and use `promise_return` so the runtime appends `output_data_receivers` post-validation; (2) return a value of `max_length_returned_data` bytes from a function call that has a `.then()` callback, causing the runtime to create an oversized `DataReceipt`. Both are demonstrated in the existing test suite: [8](#0-7) [9](#0-8) 

### Recommendation

After all post-processing mutations to new receipts (appending `output_data_receivers`, creating `DataReceipt`s from return values), re-validate the final receipt size against `max_receipt_size` before forwarding. Alternatively, enforce the size limit on `DataReceipt`s in `validate_data_receipt` by accounting for the borsh envelope overhead, so that `max_length_returned_data` is set to `max_receipt_size` minus the minimum `DataReceipt` envelope size.

### Proof of Concept

1. Deploy a contract with a method `return_large_value` that returns exactly `max_receipt_size` (4,194,304) bytes.
2. Call it with a `.then(mark_test_completed)` callback: `A[return_large_value()] -then-> B[mark_test_completed()]`.
3. The runtime creates a `DataReceipt` with 4,194,304 bytes of data. `validate_data_receipt` passes (data ≤ `max_length_returned_data`). The total borsh size of the `DataReceipt` exceeds `max_receipt_size`.
4. The oversized receipt is forwarded. `assert_oversized_receipt_occurred` confirms it appears in the chain's incoming receipts above the limit. [10](#0-9)

// A function call will generate a new receipt. Size of this receipt will be equal to
// `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and
// the receipt's size will go above max_receipt_size. The receipt should be rejected, but currently
// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
