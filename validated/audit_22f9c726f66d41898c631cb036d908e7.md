### Title
`OmniBridge::initTransfer` accepts ERC721 tokens via identical `transferFrom` selector, causing permanent NFT lock — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge::initTransfer` performs no token-type validation before pulling tokens from the caller. Because ERC721's `transferFrom(address,address,uint256)` shares the exact same 4-byte selector as ERC20's `transferFrom`, `SafeERC20.safeTransferFrom` silently succeeds for ERC721 tokens. The prerequisite step `logMetadata` reverts for ERC721 (it calls `IERC20Metadata.decimals()`, which ERC721 does not implement), so the token is never registered on NEAR. The resulting `InitTransfer` event cannot be finalized by the NEAR bridge, and the NFT is permanently locked in the `OmniBridge` contract with no recovery path.

---

### Finding Description

**Step 1 — `logMetadata` silently blocks ERC721 registration but does not block `initTransfer`.**

`logMetadata` calls `IERC20Metadata(tokenAddress).decimals()`: [1](#0-0) 

ERC721 tokens do not implement `decimals()`, so this call reverts. The token is therefore never registered on NEAR. However, `initTransfer` has no dependency on prior `logMetadata` execution — it does not check `ethToNearToken`, `isBridgeToken`, or any registration state for the "plain ERC20" branch.

**Step 2 — `initTransfer` accepts any token address without type validation.**

For a token that is neither a custom minter nor a bridge token, `initTransfer` executes: [2](#0-1) 

`SafeERC20.safeTransferFrom` internally calls `_callOptionalReturn`, which accepts an empty return buffer. ERC721's `transferFrom(address from, address to, uint256 tokenId)` has the identical 4-byte selector to ERC20's `transferFrom(address from, address to, uint256 amount)` and returns nothing — so `SafeERC20` treats the call as successful. The NFT is transferred into the bridge.

**Step 3 — The emitted event cannot be finalized.**

`initTransfer` emits `InitTransfer`: [3](#0-2) 

The NEAR relayer picks up this event and calls `fin_transfer_callback`, which looks up `token_decimals` for the token address: [4](#0-3) 

Because `logMetadata` was never successfully executed for this ERC721, `token_decimals` has no entry, and the callback panics with `BridgeError::TokenDecimalsNotFound`. The transfer is permanently unfinalizeable.

**Step 4 — No recovery mechanism exists.**

There is no admin function, escape hatch, or refund path in `OmniBridge.sol` to recover tokens that were locked via a failed `initTransfer`. The NFT is permanently custodied by the contract.

---

### Impact Explanation

**Critical — Irreversible fund lock.** Any ERC721 token transferred through `initTransfer` is permanently locked in the `OmniBridge` contract. The NEAR side cannot finalize the transfer (missing decimals entry), and there is no on-chain recovery path. This matches the "Irreversible fund lock, frozen redemption path, or permanently unclaimable user value in bridge flows" impact category.

---

### Likelihood Explanation

**Low–Medium.** A user must call `initTransfer` directly with an ERC721 address, bypassing the `logMetadata` step (which would revert). This is a realistic mistake: ERC721 and ERC20 share `name()`, `symbol()`, and `transferFrom()` signatures, making them superficially indistinguishable to a caller who does not check `decimals()`. Tooling or scripts that iterate over token addresses without type-checking could trigger this path. The absence of any guard in `initTransfer` means the protocol offers no protection.

---

### Recommendation

Add an ERC20 type guard in `initTransfer` before the `safeTransferFrom` call. The most direct approach mirrors the Linea fix: call `decimals()` on the token and revert if it fails (ERC20 tokens must implement `decimals()`; ERC721 tokens do not):

```solidity
// Before safeTransferFrom in the else branch:
(bool success, ) = tokenAddress.staticcall(
    abi.encodeWithSignature("decimals()")
);
require(success, "OmniBridge: token must implement decimals()");
```

Alternatively, use ERC-165 to check for `IERC721` (`0x80ac58cd`) and revert if detected, though not all ERC721 implementations support ERC-165.

---

### Proof of Concept

1. Deploy a standard ERC721 contract (e.g., OpenZeppelin `ERC721`).
2. Mint token ID `5` to `user`.
3. `user` calls `mockERC721.approve(omniBridgeAddress, 5)`.
4. `user` calls `omniBridge.initTransfer(mockERC721Address, 5, 0, 0, "user.near", "")`.
   - `customMinters[mockERC721Address]` is zero → falls to the `else` branch.
   - `isBridgeToken[mockERC721Address]` is false → falls to `safeTransferFrom`.
   - ERC721's `transferFrom(user, bridge, 5)` executes; NFT moves to bridge.
   - `InitTransfer` event is emitted with `amount = 5`.
5. NEAR relayer calls `fin_transfer_callback`; it panics at `token_decimals.get(&token).near_expect(BridgeError::TokenDecimalsNotFound)`.
6. NFT ID 5 is permanently owned by `OmniBridge` with no recovery path.
