Audit Report

## Title
Pool Admin Can Set Uncapped Per-Bin Fees Instantly, Enabling Front-Running of Swaps — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`setPoolBinAdditionalFees()` allows the pool admin to set `addFeeBuyE6` / `addFeeSellE6` on any bin to any `uint16` value (up to 65535, ≈ 6.55% in E6 units) with no cap and no timelock. This contrasts with `setPoolAdminFees()`, which enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps. A pool admin can front-run a pending swap by spiking the per-bin fee, forcing the swapper to pay significantly more input tokens than expected, with no on-chain protection available to the swapper.

## Finding Description

**Asymmetric cap enforcement between admin fee setters**

`setPoolAdminFees()` enforces hard caps before updating global fees: [1](#0-0) 

`setPoolBinAdditionalFees()` performs no equivalent cap check — it passes the caller-supplied values directly to the pool: [2](#0-1) 

`setBinAdditionalFees()` in `MetricOmmPool.sol` also performs no cap validation, only a bin-index bounds check: [3](#0-2) 

**Fee is read live from storage at swap time**

During `swap()`, the per-bin additional fee is loaded from `_binStates` at the moment each bin is processed and combined with `baseFeeX64`: [4](#0-3) 

The combined fee is then applied to the gross input amount: [5](#0-4) 

A fee increase committed in a front-running transaction is therefore fully reflected in the victim's swap execution.

**`priceLimitX64` does not protect against fee manipulation**

`priceLimitX64` is compared against the interpolated bin-position price to cap output quantity, not to bound the fee multiplier: [6](#0-5) 

A swapper can set a tight price limit and still be charged an arbitrarily higher fee on the same output quantity.

## Impact Explanation

This is an admin-boundary break: the pool admin role is semi-trusted and is expected to operate within caps (as demonstrated by the cap enforcement in `setPoolAdminFees()`), but `setPoolBinAdditionalFees()` has no cap, allowing the pool admin to exceed the intended boundary. For an exact-output swap, the swapper's required input is directly multiplied by `(1 + addFeeBuyE6/1e6 + baseFeeX64/ONE_X64)`. Raising `addFeeBuyE6` from 0 to 65535 immediately before a swap causes the swapper to pay up to ~6.55% more input tokens than observed at quote time. The excess accrues to LPs in the bin; if the pool admin is also an LP, they capture it directly. This constitutes a direct, quantifiable loss of user principal with no on-chain recourse — a High severity impact under Sherlock thresholds.

## Likelihood Explanation

The attack requires: (1) a pool with a mutable pool admin (not an immutable oracle or factory owner), (2) the admin monitoring the public mempool (trivial on any public EVM chain), and (3) a single `setPoolBinAdditionalFees` call to spike the fee, followed by a reset after the victim's transaction confirms. No timelock, no cap, and no fee-dimension slippage guard exist to prevent this. The pool admin role is a per-pool operator that can be set by the factory owner at pool creation, making this a realistic threat surface.

## Recommendation

1. **Add a cap** inside `setPoolBinAdditionalFees()` (e.g., bounded by `maxAdminSpreadFeeE6` or a dedicated `maxBinAdditionalFeeE6` constant), mirroring the cap already enforced in `setPoolAdminFees()`.
2. **Add a timelock** for per-bin fee increases to eliminate mempool front-running.
3. Alternatively, expose a `maxTotalFeeE6` parameter on `swap()` so callers can bound the combined fee they accept.

## Proof of Concept

```
Setup:
  - Pool deployed with addFeeBuyE6 = 0 on bin 0 (active bin).
  - Alice quotes a swap: 1000 token1 in → 990 token0 out (fee ≈ 1%).

Attack:
  1. Alice submits swap(recipient, false, -990e18, 0, ...) to the mempool.
  2. Bob (pool admin) sees Alice's tx.
  3. Bob front-runs with:
       factory.setPoolBinAdditionalFees(pool, 0, 65535, 0)
     This sets addFeeBuyE6 = 65535 (≈ 6.55% extra fee on bin 0).
  4. Alice's swap executes. The fee applied is now baseFeeX64 + 6.55%.
     Alice pays ≈ 1066 token1 instead of 1000 — a ~6.6% overcharge.
  5. Bob resets: factory.setPoolBinAdditionalFees(pool, 0, 0, 0).

Result:
  Alice loses ~66 token1 (6.6% of principal) with no on-chain protection.
  The excess accrues to LPs in bin 0; if Bob is also an LP, he captures it directly.

Foundry test outline:
  1. Deploy factory + pool with addFeeBuyE6 = 0.
  2. Record Alice's quoted amountIn for a fixed amountOut.
  3. Call setPoolBinAdditionalFees(pool, 0, 65535, 0) as pool admin.
  4. Execute Alice's swap.
  5. Assert actual amountIn > quoted amountIn by ~6.55%.
```
