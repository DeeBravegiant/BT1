Audit Report

## Title
`RSETHPriceFeed.latestRoundData()` Returns ETH/USD Timestamp for Composite rsETH/USD Price, Masking rsETH Component Staleness - (File: `contracts/oracles/RSETHPriceFeed.sol`)

## Summary
`RSETHPriceFeed.latestRoundData()` computes a composite rsETH/USD price from two independent sources — the ETH/USD Chainlink feed and `LRTOracle.rsETHPrice()` — but returns only the ETH/USD feed's `updatedAt` timestamp. Because `LRTOracle` exposes no update timestamp and `RSETHPriceFeed` never tracks one, external Chainlink-compatible consumers cannot detect when the rsETH/ETH component is stale. When `LRTOracle` auto-pauses due to a price drop, `rsETHPrice` freezes while the ETH/USD feed continues updating, causing `latestRoundData()` to return a fresh-looking `updatedAt` paired with a stale composite price.

## Finding Description
`RSETHPriceFeed.latestRoundData()` at lines 63–70 of `contracts/oracles/RSETHPriceFeed.sol`:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

`updatedAt` is taken entirely from `ETH_TO_USD.latestRoundData()`. The `IRSETHOracle` interface (line 22–24) exposes only `rsETHPrice()` with no timestamp, and `ILRTOracle` likewise exposes no `lastUpdatedAt`. `LRTOracle.rsETHPrice` is a plain storage variable (line 28 of `contracts/LRTOracle.sol`) with no associated timestamp tracking.

The auto-pause path in `LRTOracle._updateRsETHPrice()` (lines 270–281) triggers without any privileged action: when `newRsETHPrice < highestRsethPrice` by more than `pricePercentageLimit`, the function calls `_pause()` and returns early, freezing `rsETHPrice` at its last value. After this, `updateRSETHPrice()` (line 87) reverts due to `whenNotPaused`, so `rsETHPrice` cannot be updated until an admin manually unpauses. During this window, `ETH_TO_USD` continues to update normally, so every call to `RSETHPriceFeed.latestRoundData()` returns a current `updatedAt` from the live ETH/USD feed alongside a frozen rsETH/ETH rate.

Existing checks are insufficient: `ChainlinkOracleForRSETHPoolCollateral.getRate()` checks `answeredInRound < roundID` and `timestamp == 0`, but neither check catches a stale `rsETHPrice` because the timestamp returned is always fresh from the ETH/USD feed.

## Impact Explanation
External lending protocols consuming `RSETHPriceFeed` as a Chainlink `AggregatorV3Interface` oracle perform heartbeat/staleness checks against the returned `updatedAt`. Because `updatedAt` reflects only the ETH/USD feed, these checks pass even when the rsETH/ETH component is hours or days stale. If `rsETHPrice` is frozen above the true market rate, borrowers can extract excess value against inflated rsETH collateral — direct theft of funds from the lending protocol. If frozen below the true rate, rsETH holders are liquidated at below-market valuations — direct theft of user funds. This constitutes **Critical: Direct theft of any user funds**.

## Likelihood Explanation
No attacker action is required. The auto-pause is triggered automatically by normal market events (price decline beyond `pricePercentageLimit`, e.g., during a slashing event or market downturn). The condition is a documented, expected protocol state. Any external protocol using `RSETHPriceFeed` during a pause window is affected. The window persists until an admin calls `unpause()`, which may take hours or days.

## Recommendation
`latestRoundData()` should return the minimum of the two component timestamps and revert if the rsETH/ETH component exceeds a configurable maximum age. This requires `LRTOracle` to record and expose a `lastUpdatedAt` timestamp on each successful `_updateRsETHPrice()` call, and `IRSETHOracle` to expose it. The composite feed should then use:

```solidity
updatedAt = rsETHUpdatedAt < updatedAt ? rsETHUpdatedAt : updatedAt;
require(block.timestamp - rsETHUpdatedAt <= MAX_RSETH_STALENESS, "rsETH price stale");
```

## Proof of Concept
1. Deploy `LRTOracle` with `pricePercentageLimit > 0` and call `updateRSETHPrice()` at time `T0`, recording `rsETHPrice = P0`.
2. Simulate a price drop: manipulate underlying asset prices so `newRsETHPrice < highestRsethPrice * (1 - pricePercentageLimit)`. Call `updateRSETHPrice()` — `_updateRsETHPrice()` auto-pauses and returns early; `rsETHPrice` remains `P0`.
3. Advance time by several hours (beyond any reasonable heartbeat, e.g., 24h).
4. Call `RSETHPriceFeed.latestRoundData()`. Observe: `updatedAt` equals the current ETH/USD round timestamp (fresh), `answer = P0 * currentEthPrice / 1e18` (stale rsETH/ETH component).
5. A lending protocol's staleness check (`block.timestamp - updatedAt < heartbeat`) passes. The protocol prices rsETH using the stale composite, enabling over-borrowing or incorrect liquidation.
