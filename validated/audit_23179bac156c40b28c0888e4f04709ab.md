### Title
Deposit Refund from Failed Meta-Transaction Inner Receipt Sent to Sender (Alice) Instead of Relayer Who Paid — (`runtime/runtime/src/actions.rs`)

### Summary
In meta transactions (NEP-366), when a relayer submits a `DelegateAction` containing inner actions with an attached deposit, the relayer pays the deposit. However, if the inner receipt fails on the receiver's shard, the deposit refund is sent to `Alice` (the `DelegateAction` sender), not to the relayer who actually funded it. An unprivileged user can exploit this to drain the relayer's deposit by deliberately crafting meta transactions that will fail.

### Finding Description
In `apply_delegate_action` (`runtime/runtime/src/actions.rs`), when the inner receipt is constructed from the `DelegateAction`, its `predecessor_id` is set to `sender_id` (Alice):

```rust
let new_receipt = Receipt::V0(ReceiptV0 {
    predecessor_id: sender_id.clone(),   // Alice, not the relayer
    receiver_id: delegate_action.receiver_id().clone(),
    ...
    receipt: ReceiptEnum::Action(ActionReceipt {
        signer_id: action_receipt.signer_id().clone(), // relayer
        ...
        actions: delegate_action.get_actions(),
    }),
});
``` [1](#0-0) 

When this inner receipt fails on Bob's shard, `refund_unspent_gas_and_deposits` issues a deposit refund via:

```rust
result.new_receipts.push(Receipt::new_balance_refund(
    receipt.balance_refund_receiver(),
    deposit_refund,
));
``` [2](#0-1) 

`balance_refund_receiver()` returns `refund_to.as_ref().unwrap_or_else(|| self.predecessor_id())`: [3](#0-2) 

Since the inner receipt uses `ActionReceipt` (V1, no `refund_to` field), `refund_to` is `None`, so the refund goes to `predecessor_id` = Alice. The relayer, who paid the deposit when the transaction was converted to a receipt, receives nothing.

The protocol documentation explicitly acknowledges this behavior:

> "At this point, the predecessor is Alice and therefore she receives the token balance refunded, not the relayer. This is something relayer implementations must be aware of since there is a financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard." [4](#0-3) 

The `ActionReceiptV2` struct already has a `refund_to` field that could redirect the refund to the relayer, but `apply_delegate_action` constructs a V1 `ActionReceipt` without it: [5](#0-4) 

### Impact Explanation
The corrupted protocol value is the **balance** of both the relayer (loses the deposit) and Alice (gains the deposit). Alice can steal an arbitrary amount of the relayer's NEAR tokens per meta transaction by attaching a large deposit to an inner function call that is guaranteed to fail (e.g., calling a non-existent method on a contract, or targeting an account with no contract). The deposit is deducted from the relayer at transaction processing time and refunded to Alice when the inner receipt fails.

### Likelihood Explanation
High. Any user of any public meta-transaction relayer can exploit this. The only prerequisite is the ability to submit a `SignedDelegateAction` to a relayer — which is the entire purpose of the relayer service. The attack requires no special privileges, no validator access, and no malformed data. It is a valid, well-formed transaction that the protocol processes correctly according to its current logic.

### Recommendation
In `apply_delegate_action`, construct the inner receipt using `ActionReceiptV2` with `refund_to: Some(relayer_account_id)` (where the relayer is `receipt.predecessor_id()` at the time the delegate action is processed). This ensures that if the inner receipt fails, the deposit refund is directed to the relayer who paid it, not to Alice.

Alternatively, use `ReceiptManager::set_refund_to` (already implemented at `runtime/runtime/src/receipt_manager.rs` line 698) to set the refund beneficiary to the relayer when generating the inner receipt. [6](#0-5) 

### Proof of Concept
1. Alice creates a `DelegateAction` targeting Bob's shard with a `FunctionCall` action attaching `N` NEAR deposit to a method that does not exist on Bob's contract.
2. Alice signs the `DelegateAction` and sends it to a relayer.
3. The relayer wraps it in a transaction (relayer is `signer_id`), paying `N` NEAR deposit.
4. On Alice's shard, `apply_delegate_action` creates an inner receipt with `predecessor_id = Alice`, `signer_id = relayer`.
5. On Bob's shard, the function call fails (method not found). `refund_unspent_gas_and_deposits` issues a balance refund to `predecessor_id` = Alice.
6. Alice receives `N` NEAR. The relayer loses `N` NEAR. Gas refund goes to the relayer (correct), but the deposit does not.

Repeat for each meta transaction to drain the relayer's balance.

pub struct ActionReceiptV2 {
    /// A signer of the original transaction
    pub signer_id: AccountId,
    /// The receiver of any balance refunds form this receipt if it is different from receiver_id.
    pub refund_to: Option<AccountId>,
    /// An access key which was used to sign the original transaction
    pub signer_public_key: PublicKey,
    /// A gas_price which has been used to buy gas in the original transaction
    pub gas_price: Balance,
    /// If present, where to route the output data
    pub output_data_receivers: Vec<DataReceiver>,
    /// A list of the input data dependencies for this Receipt to process.
    /// If all `input_data_ids` for this receipt are delivered to the account
    /// that means we have all the `ReceivedData` input which will be than converted to a
    /// `PromiseResult::Successful(value)` or `PromiseResult::Failed`
    /// depending on `ReceivedData` is `Some(_)` or `None`
    pub input_data_ids: Vec<CryptoHash>,
    /// A list of actions to process when all input_data_ids are filled
    pub actions: Vec<Action>,
}
