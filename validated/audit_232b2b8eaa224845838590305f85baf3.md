Audit Report

## Title
ETH Deposit Limit Not Enforced Due to Missing `+ amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` omits the incoming `amount` from the ETH branch comparison, while the LST branch correctly includes it. This allows any depositor to push total ETH deposits above the configured `depositLimitByAsset` cap, bypassing the protocol's intended deposit ceiling.

## Finding Description
At [1](#0-0)  the ETH branch on line 679 evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding `amount`, while the LST branch on line 681 correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`.

When `totalAssetDeposits == depositLimit`, the ETH check returns `false` (not exceeded), so the guard at [2](#0-1)  does not revert, and the deposit proceeds. The ETH branch only reverts when `totalAssetDeposits` already exceeds the limit before the deposit, meaning the limit can be exceeded by exactly one deposit at a time, repeatedly.

## Impact Explanation
**Low.** The `depositLimitByAsset` cap is a protocol-level risk management parameter. Any depositor can bypass it when total ETH deposits are at or near the limit, minting rsETH beyond the intended ceiling and staking excess ETH into EigenLayer beyond the intended exposure. No direct fund theft or loss of value occurs, but the protocol fails to deliver its promised deposit constraint. This matches the allowed impact: *Contract fails to deliver promised returns, but doesn't lose value.*

## Likelihood Explanation
**Medium.** No special privileges are required. Any unprivileged depositor monitoring the pool can trigger this whenever `totalAssetDeposits >= depositLimit`. The condition is observable on-chain and trivially reachable via a normal `depositETH` call.

## Recommendation
Add `+ amount` to the ETH branch to match the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether` via `LRTConfig.updateAssetDepositLimit`.
2. Total ETH deposits reach exactly `100 ether` via prior deposits.
3. Attacker calls `depositETH{value: 1 ether}(0)` on `LRTDepositPool`.
4. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)`.
5. `totalAssetDeposits = 100 ether`; check evaluates `100 ether > 100 ether` → `false` → no revert.
6. `_mintRsETH` mints rsETH for the attacker; total ETH deposits become `101 ether`, exceeding the cap.
7. Any subsequent depositor can repeat steps 3–6 indefinitely, each time pushing deposits further above the cap.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock `LRTConfig` returning `depositLimitByAsset = 100 ether` and `getTotalAssetDeposits = 100 ether`. Call `depositETH{value: 1 ether}` and assert it does not revert (demonstrating the bypass). Then apply the fix and assert the same call reverts with `MaximumDepositLimitReached`.

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
