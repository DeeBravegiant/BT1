Audit Report

## Title
Missing Zero-rsETH-Minted Validation in `_beforeDeposit` Allows Silent Loss of Deposited Assets - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._beforeDeposit` computes `rsethAmountToMint` via integer division in `getRsETHAmountToMint` but never validates that the result is non-zero. When a caller passes `minRSETHAmountExpected = 0` and the deposit amount is small enough that the division truncates to zero, the deposit succeeds, the user's assets are transferred into the protocol, and `_mintRsETH(0)` is called — minting nothing to the depositor. The user's funds are permanently absorbed into the protocol TVL with no recovery path.

## Finding Description
`_beforeDeposit` (L648–670) performs the following checks in order:

1. `if (depositAmount == 0 || depositAmount < minAmountToDeposit)` — `minAmountToDeposit` is a storage variable initialized to `0` (never set in `initialize`), so this only blocks a literal zero-amount deposit.
2. `_checkIfDepositAmountExceedesCurrentLimit` — unrelated to share calculation.
3. `rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount)` — computes `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` via integer division; truncates to `0` when `amount * assetPrice < rsETHPrice`.
4. `if (rsethAmountToMint < minRSETHAmountExpected)` — when the caller passes `minRSETHAmountExpected = 0`, the condition `0 < 0` is always false; no revert fires.

`_beforeDeposit` returns `0`, and `_mintRsETH(0)` is called. Inside `RSETH.mint`, the `checkDailyMintLimit(0)` modifier passes (adding 0 to `currentPeriodMintedAmount` never exceeds any limit), and OpenZeppelin's `_mint(to, 0)` is a no-op — no tokens are issued. For `depositETH`, the ETH is already in the contract (sent as `msg.value`). For `depositAsset`, `safeTransferFrom` executes before `_mintRsETH`, so the ERC-20 is already transferred. In both cases the user holds no rsETH and has no withdrawal mechanism.

## Impact Explanation
The depositor's assets are permanently absorbed into the protocol's TVL. Because the user receives 0 rsETH, they have no shares to redeem and no on-chain path to reclaim their funds. The assets silently inflate TVL, benefiting all existing rsETH holders at the depositor's expense. This constitutes **permanent freezing of funds** (Critical) from the depositor's perspective, as no admin or user action can return the deposited amount to the original sender.

## Likelihood Explanation
- `minAmountToDeposit` defaults to `0` — no admin configuration is required to open the vulnerable path.
- `minRSETHAmountExpected` is fully caller-controlled; passing `0` is a common pattern for users who do not set slippage protection.
- The truncation condition (`rsETHPrice > assetPrice`) is reached as soon as any yield accrues — a normal, expected protocol state.
- No privileged access, front-running, oracle compromise, or governance action is required. Any unprivileged external caller can trigger this via `depositETH` or `depositAsset`.

## Recommendation
Add an explicit non-zero check on `rsethAmountToMint` inside `_beforeDeposit`, before the slippage check:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint == 0) {
    revert ZeroRsETHMinted();
}

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

Additionally, consider setting `minAmountToDeposit` to a non-zero value during initialization to prevent dust deposits that are economically guaranteed to round down.

## Proof of Concept
1. Protocol has accrued yield; `rsETHPrice = 1.05e18`, `getAssetPrice(ETH) = 1e18`.
2. Caller executes: `lrtDepositPool.depositETH{value: 1}(0, "")`.
3. Inside `_beforeDeposit`:
   - `depositAmount = 1`, passes `depositAmount == 0` check (since `minAmountToDeposit = 0`).
   - `rsethAmountToMint = (1 * 1e18) / 1.05e18 = 0` (integer truncation).
   - `0 < 0` → false → no revert.
4. `_mintRsETH(0)` → `RSETH.mint(msg.sender, 0)` → `checkDailyMintLimit(0)` passes → `_mint(msg.sender, 0)` is a no-op.
5. The 1 wei ETH is now part of the protocol TVL. The caller holds 0 rsETH and has no withdrawal path.

Foundry fuzz test plan: fuzz `depositAmount` in range `[1, rsETHPrice/1e18 - 1]` with `minRSETHAmountExpected = 0`; assert that after the call `rsETH.balanceOf(caller) == 0` while `address(lrtDepositPool).balance` increased — confirming funds are absorbed with no shares issued.
