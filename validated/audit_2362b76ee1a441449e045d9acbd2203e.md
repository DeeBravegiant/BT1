Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero Mid-Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` enforces `totalKernelStaked > 0` only at the moment `notifyRewardAmount` is called, but provides no code-level guard against all stakers withdrawing during an active reward period. When `totalKernelStaked` reaches zero mid-period, `rewardPerToken()` freezes while `updateReward` continues advancing `updatedAt`, permanently consuming the elapsed time without distributing the corresponding rewards. Those reward tokens have no recovery path.

## Finding Description
Three code elements interact to produce the loss:

**1. `rewardPerToken()` short-circuits on zero stake** [1](#0-0) 

When `totalKernelStaked == 0`, the function returns `rewardPerTokenStored` unchanged — no new rewards are accrued regardless of elapsed time.

**2. `updateReward` always advances `updatedAt`** [2](#0-1) 

`updatedAt = lastTimeRewardApplicable()` executes unconditionally on every state-changing call, even when `totalKernelStaked == 0`. The time window is consumed silently.

**3. `initiateWithdrawal` decrements `totalKernelStaked` immediately** [3](#0-2) 

The withdrawal delay only defers the token transfer; the accounting reduction is instant.

**Exploit flow:**
- At time T, `notifyRewardAmount` is called with `totalKernelStaked > 0` — passes the guard. [4](#0-3) 
- At T+3600, the last staker calls `initiateWithdrawal`. `updateReward` checkpoints `updatedAt = T+3600`, then `totalKernelStaked` drops to 0.
- From T+3600 to T+7200, no one is staked. No interaction occurs.
- At T+7200, a new staker calls `stake`. `updateReward` fires: `rewardPerToken()` sees `totalKernelStaked == 0` → returns `rewardPerTokenStored` unchanged, then sets `updatedAt = T+7200`. The 3600-second gap is consumed.
- The new staker's `userRewardPerTokenPaid` is set to the unchanged `rewardPerTokenStored`. Rewards for the gap (`rewardRate × 3600`) are never distributed and remain locked in the contract with no sweep or rescue function.

The NatSpec explicitly acknowledges this behavior but relies on an off-chain operational assumption ("ensuring there are always some tokens staked for the entire duration of the reward period") rather than any code-level enforcement. [5](#0-4) 

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens corresponding to any zero-staked window are irrecoverable: there is no admin sweep, no rescue function, and no mechanism to roll the lost window into the next period. The loss equals `rewardRate × (duration of zero-staked window)`, which can be substantial for long gaps or high reward rates.

## Likelihood Explanation
Any staker can call `initiateWithdrawal` at any time without restriction. If the last remaining staker exits (organically or deliberately), `totalKernelStaked` drops to zero immediately. A single large staker holding the majority of stake can trigger this unilaterally. No privileged access, coordination, or external dependency is required. The condition is repeatable across reward periods.

## Recommendation
Enforce the invariant in code rather than relying on operational discipline. Options:
1. **Prevent zero-stake during active periods**: In `initiateWithdrawal`, add a check `if (totalKernelStaked - _amount == 0 && block.timestamp < finishAt) revert(...)`.
2. **Pause `updatedAt` advancement when unstaked**: In `updateReward`, only advance `updatedAt` when `totalKernelStaked > 0`, so the zero-staked window is not consumed.
3. **Admin recovery function**: Allow the admin to reclaim unallocated rewards after `finishAt`, or track the lost window and roll it into the next `notifyRewardAmount` call.

## Proof of Concept
Foundry test sequence:

```solidity
// 1. Admin sets duration = 86400
pool.setRewardsDuration(86400);

// 2. Alice stakes 1000e18 KERNEL
vm.prank(alice);
pool.stake(1000e18); // totalKernelStaked = 1000e18

// 3. Admin notifies 86400e18 rewards → rewardRate = 1e18/s
pool.notifyRewardAmount(86400e18);

// 4. Warp 1 hour; Alice initiates full withdrawal
vm.warp(block.timestamp + 3600);
vm.prank(alice);
pool.initiateWithdrawal(1000e18); // totalKernelStaked = 0, updatedAt = T+3600

// 5. Warp another hour (zero-staked gap)
vm.warp(block.timestamp + 3600); // T+7200

// 6. Bob stakes
vm.prank(bob);
pool.stake(1000e18);
// updateReward: rewardPerToken() returns rewardPerTokenStored (unchanged), updatedAt = T+7200

// 7. Warp to end; Bob claims
vm.warp(block.timestamp + 79200); // T+86400
vm.prank(bob);
pool.getReward();

// Assert: 3600e18 tokens are stuck in contract
// Alice earned: 3600e18 (T to T+3600)
// Bob earned:   79200e18 (T+7200 to T+86400)
// Stuck:        3600e18 (T+3600 to T+7200) — no recipient, no recovery
assertEq(rewardsToken.balanceOf(address(pool)), 3600e18);
```
