### Title
Missing Caller-Binding Tweak in `verify_foreign_transaction` Enables Cross-Bridge Attestation Replay - (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` flow produces MPC attestation signatures using a hardcoded zero tweak and no caller/predecessor binding in the signed payload. The design specification explicitly requires a caller-specific `derivation_path` and a distinct `FOREIGN_TX_TWEAK_DERIVATION_PREFIX` to isolate each bridge contract's attestation key. The implementation omits both entirely. As a result, any attestation produced for one bridge contract can be replayed to any other bridge contract that trusts the same MPC network root key, enabling double-spend or invalid bridge execution.

---

### Finding Description

The design document at `docs/foreign-chain-transactions.md` specifies that `VerifyForeignTransactionRequestArgs` must include a `derivation_path: String` field, and that the contract must internally derive a caller-specific tweak using a dedicated prefix:

```rust
// From docs/foreign-chain-transactions.md (design spec)
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub struct VerifyForeignTransactionRequestArgs {
    pub request: ForeignChainRpcRequest,
    pub derivation_path: String, // Key derivation path
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}

pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub tweak: Tweak,           // derived from (predecessor_id, derivation_path)
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The actual production implementation diverges completely. `VerifyForeignTransactionRequestArgs` has no `derivation_path` field and `VerifyForeignTransactionRequest` has no `tweak` field:

```rust
// Actual: crates/near-mpc-contract-interface/src/types/foreign_chain.rs
pub struct VerifyForeignTransactionRequestArgs {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    // NO derivation_path
}

pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    // NO tweak
}
```

The conversion function `args_into_verify_foreign_tx_request` in `crates/contract/src/dto_mapping.rs` performs a trivial field copy with no tweak derivation and no capture of `env::predecessor_account_id()`:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
        // predecessor_id never captured, tweak never derived
    }
}
```

The node-side signing function `build_signature_request` in `crates/node/src/providers/verify_foreign_tx/sign.rs` hardcodes the tweak to all-zero bytes:

```rust
Ok(SignatureRequest {
    id: request.id,
    receipt_id: request.receipt_id,
    payload: Payload::Ecdsa(payload_bytes),
    tweak: Tweak::new([0u8; 32]),   // hardcoded zero tweak = root key
    entropy: request.entropy,
    timestamp_nanosec: request.timestamp_nanosec,
    domain: request.domain_id,
})
```

The contract's `respond_verify_foreign_tx` verifies the signature against the **root public key** (no tweak applied), confirming the zero-tweak path is the only one that can succeed:

```rust
// crates/contract/src/lib.rs
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // root public key, no tweak derivation
)
.is_ok()
```

The signed payload `ForeignTxSignPayload::V1` contains only `(request, extracted_values)` — the caller's identity (`predecessor_id`) is never included. The test comment at line 3694 of `crates/contract/src/lib.rs` even acknowledges this explicitly: `"simulate signature with the root key (no tweak for foreign tx)"`.

---

### Impact Explanation

**Cross-bridge attestation replay (High):** Because the attestation signature is produced over `(ForeignChainRpcRequest, ExtractedValues)` with a zero tweak (root key) and no caller binding, the same signature is valid for any bridge contract that trusts the MPC network's root public key for the `ForeignTx` domain. An attacker who receives a valid attestation for foreign transaction T on bridge A can present the identical `VerifyForeignTransactionResponse` to bridge B. If bridge B independently verifies the signature against the MPC root public key (which is the only key that can verify it), it will accept the attestation as proof that T was verified for its own context, enabling double-spend or unauthorized fund release.

The design's explicit goal — "key material used for validated foreign transactions is **always** distinct from general-purpose `sign()` keys, even if the same account and derivation path are reused" — is partially achieved (domain separation exists), but the per-caller isolation within the `ForeignTx` domain is entirely absent.

---

### Likelihood Explanation

The `verify_foreign_transaction` feature is production-ready and actively tested across Bitcoin, Ethereum, Solana, Starknet, and other chains. Any bridge contract that integrates with it and relies on the MPC attestation to release funds is immediately exposed. The attacker path requires only a valid deposit (1 yoctoNEAR) and a real foreign-chain transaction — no privileged access, no threshold collusion, no TEE bypass. The zero-tweak behavior is structurally baked in and cannot be worked around by callers.

---

### Recommendation

1. **Short term:** Add `derivation_path: String` to `VerifyForeignTransactionRequestArgs`. In `verify_foreign_transaction`, capture `env::predecessor_account_id()` and derive the tweak using a dedicated prefix (e.g., `"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`). Store the derived tweak in `VerifyForeignTransactionRequest`. In `build_signature_request`, use the stored tweak instead of `Tweak::new([0u8; 32])`. In `respond_verify_foreign_tx`, verify the signature against the tweak-derived public key, not the root key.

2. **Long term:** Include the caller's `predecessor_id` in the signed payload (`ForeignTxSignPayloadV1`) so that downstream verifiers can confirm the attestation was produced for their specific bridge contract, not replayed from another context.

---

### Proof of Concept

1. Bridge contract A calls `verify_foreign_transaction` for Bitcoin tx `T`, paying 1 yoctoNEAR. MPC nodes verify `T`, sign `hash(Bitcoin(T), BlockHash([...]))` with zero tweak (root key), and return `VerifyForeignTransactionResponse { payload_hash, signature }`.

2. Attacker observes the response on-chain (it is public NEAR state).

3. Bridge contract B (a different bridge, same MPC network) also accepts `VerifyForeignTransactionResponse` as proof of foreign-chain finality. Attacker submits the identical `response` to bridge B, claiming it proves `T` was verified for B's context.

4. Bridge B calls `near_mpc_signature_verifier::verify_ecdsa_signature(signature, payload_hash, root_public_key)`. This succeeds — the signature is valid under the root key with zero tweak, which is the only key the contract ever uses for `ForeignTx` domain responses.

5. Bridge B releases funds, constituting a double-spend or unauthorized bridge execution.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
