Audit Report

## Title
Absence of Default Minimum Withdrawal Amount Enables Withdrawal Queue Spam, Causing Unbounded Gas in `_unlockWithdrawalRequests` - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`minRsEthAmountToWithdraw[asset]` defaults to `0` for all assets because `initialize()` never sets it, and `setMinRsEthAmountToWithdraw` is admin-only and not called at deployment. The guard in `initiateWithdrawal` therefore only rejects zero-amount calls, allowing any unprivileged rsETH holder to flood the global nonce queue with dust entries. Because `_unlockWithdrawalRequests` processes entries strictly in FIFO order, legitimate users' requests queued behind spam entries cannot be unlocked until all preceding spam entries are processed, temporarily freezing their ability to complete withdrawals.

## Finding Description

**Root cause — missing minimum in `initialize`:**

`initialize()` sets only `withdrawalDelayBlocks` and `lrtConfig`; `minRsEthAmountToWithdraw` is never initialised:

```solidity
// contracts/LRTWithdrawalManager.sol L90-98
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    __Pausable_init();
    __ReentrancyGuard_init();
    withdrawalDelayBlocks = 8 days / 12 seconds;
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

The mapping therefore holds `0` for every asset. The guard in `initiateWithdrawal` (L162-164) becomes:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < 0 /* always false for uint256 */) {
    revert InvalidAmountToWithdraw();
}
```

Any call with `rsETHUnstaked ≥ 1` passes.

**Exploit path:**

1. Attacker approves the withdrawal manager and calls `initiateWithdrawal(asset, 1, "")` N times. Each call:
   - Passes the guard (1 ≥ 1, 1 ≥ 0).
   - Computes `expectedAssetAmount = 1 * rsETHPrice / assetPrice` (≥ 0 wei of asset).
   - Pushes a new entry at `nextUnusedNonce[asset]` and increments it (L756-757).
2. Legitimate users queue their requests at nonces N, N+1, …
3. After the 8-day delay, the operator calls `unlockQueue`. `_unlockWithdrawalRequests` (L790-814) iterates from `nextLockedNonce` to `firstExcludedIndex` in strict order. It cannot skip spam entries; it must process all N of them before reaching any legitimate request.
4. Each iteration performs multiple storage reads and writes (`withdrawalRequests[requestId]`, `assetsCommitted[asset]`, `unlockedWithdrawalsCount[asset]`). With N large enough, the loop exceeds the block gas limit for any practical `firstExcludedIndex` window, or forces the operator to issue many batched transactions, stalling the unlock pipeline and temporarily freezing legitimate users' queued funds.

**Why existing checks are insufficient:**

- The `onlySupportedAsset` and `onlySupportedStrategy` modifiers do not limit call frequency.
- The `ExceedAmountToWithdraw` check (L170) only rejects requests that exceed currently available assets; dust requests with `expectedAssetAmount ≈ 0` never trigger it and do not reduce `getAvailableAssetAmount` meaningfully.
- The operator-controlled `firstExcludedIndex` parameter does not help: legitimate entries sit at higher nonces and cannot be unlocked out of order; the operator must drain all spam entries first.
- `setMinRsEthAmountToWithdraw` (L330-332) exists but is admin-only and is never called during deployment, leaving the default at `0`.

## Impact Explanation

**Medium — Temporary freezing of funds.**

Legitimate users whose withdrawal requests are queued behind spam entries cannot have those requests unlocked (and therefore cannot call `completeWithdrawal`) until the operator has processed every preceding spam entry. With a sufficiently large spam queue this delay can span many blocks or days, temporarily freezing users' rsETH and their expected asset payouts. The attacker recovers their rsETH value (burned at near-zero cost) and pays only L1 gas per spam call.

## Likelihood Explanation

**Medium.** No privileged access is required; `initiateWithdrawal` is a public, permissionless function. The attacker needs only to hold rsETH (liquid on mainnet) and pay L1 gas per call. The rsETH cost per entry is 1 wei (negligible). The attack is repeatable and economically viable whenever L1 gas is cheap relative to the disruption caused. The 8-day withdrawal delay is a cost (capital lockup) but not a deterrent for a patient or well-capitalised attacker.

## Recommendation

1. **Set a non-zero minimum in `initialize`** for every supported asset:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    ...
    minRsEthAmountToWithdraw[LRTConstants.ETH_TOKEN] = 1e15; // 0.001 rsETH
}
```

2. **Enforce a non-zero lower bound in the setter** to prevent an admin from accidentally re-opening the vector:

```solidity
function setMinRsEthAmountToWithdraw(address asset, uint256 min) external onlyLRTAdmin {
    if (min == 0) revert MinimumMustBeNonZero();
    minRsEthAmountToWithdraw[asset] = min;
    emit MinAmountToWithdrawUpdated(asset, min);
}
```

## Proof of Concept

```solidity
// Foundry fork test (mainnet fork, post-deployment state)
function testSpamWithdrawalQueue() public {
    address asset = LRTConstants.ETH_TOKEN;
    uint256 N = 5000;

    // Attacker holds N wei of rsETH
    deal(address(rsETH), attacker, N);
    vm.startPrank(attacker);
    rsETH.approve(address(withdrawalManager), type(uint256).max);

    // Spam N dust withdrawal requests
    for (uint256 i = 0; i < N; i++) {
        withdrawalManager.initiateWithdrawal(asset, 1, "spam");
    }
    vm.stopPrank();

    // Legitimate user queues a real withdrawal
    deal(address(rsETH), user, 1 ether);
    vm.prank(user);
    rsETH.approve(address(withdrawalManager), type(uint256).max);
    vm.prank(user);
    withdrawalManager.initiateWithdrawal(asset, 1 ether, "legit");

    // Advance past withdrawal delay
    vm.roll(block.number + withdrawalManager.withdrawalDelayBlocks() + 1);

    // Operator attempts to unlock all entries in one call — measures gas
    uint256 gasBefore = gasleft();
    vm.prank(operator);
    withdrawalManager.unlockQueue(asset, N + 1, minAssetPrice, minRsEthPrice, maxAssetPrice, maxRsEthPrice);
    uint256 gasUsed = gasBefore - gasleft();

    // Assert gas consumed scales with N (unbounded), and legitimate user's
    // request at nonce N cannot be unlocked until all N spam entries are processed.
    assertGt(gasUsed, 10_000_000); // exceeds practical per-tx gas budget for large N
}
```
