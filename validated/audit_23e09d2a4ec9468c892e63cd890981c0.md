### Title
Missing ERC-20 `transferFrom` Selector in Wallet Contract Ethereum Emulation Causes Silent Receipt Misclassification - (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/eth_emulation.rs`)

---

### Summary

The `try_emulation` function in the wallet contract's Ethereum emulation module handles only three ERC-20 selectors (`balanceOf`, `transfer`, `totalSupply`) but omits `transferFrom` (selector `0x23b872dd`). When a user submits an Ethereum-formatted `transferFrom` transaction through the wallet contract, the missing selector causes the call to fall through to an `UnknownFunctionSelector` error, which is then silently reinterpreted as a base NEAR token transfer for eth-implicit targets. The intended NEP-141 token transfer is never executed, and the receipt outcome is corrupted relative to the user's signed intent.

---

### Finding Description

The `try_emulation` function in `eth_emulation.rs` dispatches on the first 4 bytes of `tx.data` to map ERC-20 calls to NEP-141 actions: [1](#0-0) 

Three selectors are handled: [2](#0-1) 

The standard ERC-20 `transferFrom(address from, address to, uint256 value)` selector `0x23b872dd` is absent. When this selector arrives, `try_emulation` returns `Err(Error::User(UserError::UnknownFunctionSelector))`.

Back in `parse_rlp_tx_to_action`, that error is caught in the combined `InvalidAbiEncodedData | UnknownFunctionSelector` arm: [3](#0-2) 

For a `TargetKind::EthImplicit` target (another eth-implicit wallet contract), the transaction is **silently converted** to a base NEAR token transfer with `yocto_near: 0` and `address_check: None`. For a `TargetKind::OtherNearAccount` target (a named NEP-141 contract such as `usdc.near`), the error propagates and the transaction fails outright.

The `EthEmulationKind` and `ParsableEthEmulationKind` enums confirm there is no `ERC20TransferFrom` variant anywhere in the type system: [4](#0-3) 

---

### Impact Explanation

Two concrete corrupted outcomes depending on target type:

1. **Named NEP-141 contract target** (`usdc.near`, etc.): The `transferFrom` call returns a user-facing error. The receipt outcome records a failure where the user expected a successful token transfer. The NEP-141 token balance is not updated.

2. **Eth-implicit target** (another wallet contract address): The `transferFrom` call is silently rewritten into a 0-yoctoNEAR NEAR base token transfer. The receipt outcome records success for a NEAR transfer, not a token transfer. The user's NEP-141 token balance is unchanged, the intended recipient receives no tokens, and the relayer is still paid `tx_fee` from the user's NEAR balance. The signed Ethereum transaction's intent is not honored.

The corrupted protocol value is the **receipt outcome** (wrong action type, wrong token balance change) and the **NEAR balance** (fee deducted for a no-op in case 2).

---

### Likelihood Explanation

`transferFrom` is one of the two most-used ERC-20 functions (alongside `transfer`). It is the standard mechanism used by DEX routers, lending protocols, and any contract that pulls tokens from a user's allowance. Any Ethereum-native user interacting with the NEAR wallet contract who attempts to use `transferFrom`-based DeFi flows will trigger this path. The entry point is fully unprivileged: any holder of an eth-implicit account can craft and sign such a transaction and submit it via any relayer.

---

### Recommendation

Add `ERC20_TRANSFER_FROM_SELECTOR` (`0x23b872dd`) to `try_emulation` in `eth_emulation.rs`, decoding `(address from, address to, uint256 value)` and mapping it to the NEP-141 `ft_transfer_call` method (or a dedicated `ft_transfer_from` if the token contract supports it). A corresponding `ERC20TransferFrom` variant should be added to `ParsableEthEmulationKind` and `EthEmulationKind` in `types.rs`, and the dispatch in `lib.rs` should handle the new variant analogously to `ERC20Transfer`. [5](#0-4) 

---

### Proof of Concept

1. User holds NEP-141 tokens on a named contract `token.near` and has granted an allowance.
2. User signs an Ethereum transaction with:
   - `to` = `account_id_to_address("token.near")`
   - `data` = `0x23b872dd` + ABI-encoded `(from_address, to_address, amount)`
   - `value` = 0
3. Relayer submits via `rlp_execute("token.near", signed_tx)`.
4. `parse_tx_data` falls through all known selectors to `eth_emulation::try_emulation`.
5. `try_emulation` hits the `_ => Err(Error::User(UserError::UnknownFunctionSelector))` arm.
6. `parse_rlp_tx_to_action` catches the error; since `target_kind` is `OtherNearAccount`, it returns the error to the caller.
7. The wallet contract records a failed receipt. No `ft_transfer_call` is ever issued. Token balances are unchanged.
8. For an eth-implicit target variant: the error arm instead produces a silent NEAR transfer receipt, consuming the user's fee with no token movement. [6](#0-5) [7](#0-6)

pub fn try_emulation(
    target: &AccountId,
    tx: &NormalizedEthTransaction,
    fee: NearToken,
    context: &ExecutionContext,
) -> Result<(Action, ParsableEthEmulationKind), Error> {
    if tx.data.len() < 4 {
        return Err(Error::User(UserError::InvalidAbiEncodedData));
    }

    let suffix = context.current_account_suffix();
    match &tx.data[0..4] {
        ERC20_BALANCE_OF_SELECTOR => {
            let (address,): (Address,) =
                ethabi_utils::abi_decode(&ERC20_BALANCE_OF_SIGNATURE, &tx.data[4..])?;
            // The account ID is assumed to have the same suffix as the current account because
            // (1) in production this is correct as all eth-implicit accounts are top-level and
            // (2) in testing environments where the addresses are sub-accounts, they are still
            // assumed to all be deployed to the same namespace so that they will all have the
            // same suffix.
            let args = format!(r#"{{"account_id": "0x{}{}"}}"#, hex::encode(address), suffix);
            Ok((
                Action::FunctionCall {
                    receiver_id: target.to_string(),
                    method_name: "ft_balance_of".into(),
                    args: args.into_bytes(),
                    gas: FIVE_TERA_GAS,
                    yocto_near: 0,
                },
                ParsableEthEmulationKind::ERC20Balance,
            ))
        }
        ERC20_TRANSFER_SELECTOR => {
            // We intentionally map to `u128` instead of `U256` because the NEP-141 standard
            // is to use u128.
            let (to, value): (Address, u128) =
                ethabi_utils::abi_decode(&ERC20_TRANSFER_SIGNATURE, &tx.data[4..])?;
            let receiver_id: AccountId = format!("0x{}{}", hex::encode(to), suffix)
                .parse()
                .unwrap_or_else(|_| env::panic_str("eth-implicit accounts are valid account ids"));

            // Include any data after the main args as a memo in the transfer.
            // The main data takes 68 bytes because there is a 4-byte selector followed
            // by two arguments which are each allocated 32 bytes according to the
            // Solidity ABI standard.
            let memo = if tx.data.len() > 68 {
                Some(format!(r#""0x{}""#, hex::encode(&tx.data[68..])))
            } else {
                None
            };
            let args = format!(
                r#"{{"receiver_id": "{}", "amount": "{}", "memo": {}}}"#,
                receiver_id.as_str(),
                value,
                memo.as_deref().unwrap_or("null"),
            );
            Ok((
                Action::FunctionCall {
                    receiver_id: target.to_string(),
                    method_name: "ft_transfer".into(),
                    args: args.into_bytes(),
                    gas: 2 * FIVE_TERA_GAS,
                    yocto_near: 1,
                },
                ParsableEthEmulationKind::ERC20Transfer { receiver_id, fee },
            ))
        }
        ERC20_TOTAL_SUPPLY_SELECTOR => Ok((
            Action::FunctionCall {
                receiver_id: target.to_string(),
                method_name: "ft_total_supply".into(),
                args: Vec::new(),
                gas: FIVE_TERA_GAS,
                yocto_near: 0,
            },
            ParsableEthEmulationKind::ERC20TotalSupply,
        )),
        _ => Err(Error::User(UserError::UnknownFunctionSelector)),
    }
}

/// Describes a kind of transaction that is directly parsable
/// from an Ethereum-formatted transaction's calldata. Notably
/// `EthEmulationKind::EOABaseTokenTransfer` is missing because
/// on Ethereum base token transfers are inferred from the target
/// of the transaction, not its data.
#[must_use]
pub enum ParsableTransactionKind {
    /// Near native actions with an explicit receiver
    /// (i.e. `FunctionCall` and `Transfer`).
    NearNativeAction,
    /// Near native actions where the receiver should be equal
    /// to the current account (i.e. `AddKey` and `DeleteKey`).
    SelfNearNativeAction,
    /// Emulated Ethereum standards
    EthEmulation(ParsableEthEmulationKind),
}

/// See docs for `ParsableTransactionKind`.
#[must_use]
pub enum ParsableEthEmulationKind {
    ERC20Balance,
    ERC20Transfer { receiver_id: AccountId, fee: NearToken },
