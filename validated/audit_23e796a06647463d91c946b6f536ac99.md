### Title
Deposits for Non-Winning `propose_update` Proposals Are Permanently Locked When `do_update` Clears All Entries - (File: `crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires a NEAR deposit proportional to the byte-size of the proposed update (contract code or config). When `vote_update` reaches threshold and triggers `do_update`, **all** pending proposals are cleared in bulk. The deposits paid by proposers of the non-winning proposals are never refunded and have no recovery path — they are permanently locked in the contract.

---

### Finding Description

`propose_update` collects a mandatory deposit from each proposer:

```rust
// crates/contract/src/lib.rs:1308-1331
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
if attached < required {
    return Err(InvalidParameters::InsufficientDeposit { ... }.into());
}
let id = self.proposed_updates.propose(update);
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

The deposit is sized by `ProposedUpdates::required_deposit`, which is a function of `bytes_used` — for a full contract binary this can be multiple NEAR. [1](#0-0) 

The deposit amount is **not stored** in `UpdateEntry`; only `bytes_used` is kept:

```rust
// crates/contract/src/update.rs:132-135
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
``` [2](#0-1) 

When threshold votes are reached, `do_update` removes the winning entry and then bulk-clears **all remaining entries** with no refund logic:

```rust
// crates/contract/src/update.rs:195-201
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
``` [3](#0-2) 

There is no `cancel_update` function, no stored deposit amount per entry, and no sweep-refund path anywhere in the contract. The NEAR paid for every non-winning proposal is absorbed into the contract's balance with no recovery mechanism.

The parallel to the external report is exact: just as `cvxRewards.getReward()` silently pulls extra reward tokens into `ConvexDepositToken` with no transfer-out path, `do_update` silently absorbs the storage deposits of all cleared proposals with no refund path.

---

### Impact Explanation

Every time a contract or config upgrade is executed, the deposits of all competing proposals are permanently frozen inside the MPC contract. For a full contract binary (hundreds of KB), `required_deposit` can be several NEAR per proposal. With `N` competing proposals, `(N-1) × required_deposit` NEAR is irrecoverably locked. There is no admin withdrawal function, no `cancel_update`, and no post-execution sweep. This breaks the production accounting invariant that storage deposits are returned when the associated storage is freed.

---

### Likelihood Explanation

The `test_propose_update_contract_many` sandbox test and the `only_one_vote_from_participant` test both demonstrate that multiple simultaneous proposals are a normal, expected usage pattern. [4](#0-3) [5](#0-4) 

Every production upgrade cycle where more than one participant independently proposes an update triggers this loss. Given that the devnet tooling explicitly supports multi-proposal workflows, this is a routine operational condition, not an edge case.

---

### Recommendation

Store the deposit amount alongside `bytes_used` in `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,
    pub(super) deposit_paid: NearToken,
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over the remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit_paid)` for each. Alternatively, expose a `cancel_update` method that lets a proposer withdraw their own proposal and receive a refund, analogous to the `transferTokens` function recommended in the external report.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, paying ~5 NEAR deposit. Proposal ID = 0.
2. Participant B calls `propose_update` with a different 500 KB binary, paying ~5 NEAR deposit. Proposal ID = 1.
3. Threshold participants vote for proposal 1 via `vote_update(1)`.
4. `do_update` executes: removes entry 1, then calls `self.entries.clear()` — entry 0 (and its 5 NEAR deposit) is silently discarded.
5. Participant A's 5 NEAR is permanently locked in the contract. No refund is issued; no recovery path exists. [6](#0-5) [7](#0-6)

async fn test_propose_update_contract_many() {
    let SandboxTestSetup {
        contract,
        mpc_signer_accounts,
        ..
    } = SandboxTestSetup::builder()
        .with_protocols(ALL_PROTOCOLS)
        .build()
        .await;
    dbg!(contract.id());

    const PROPOSAL_COUNT: usize = 2;
    let mut proposals = Vec::with_capacity(PROPOSAL_COUNT);
    // Try to propose multiple updates to check if they are being proposed correctly
    // and that we can have many at once living in the contract state.
    for i in 0..PROPOSAL_COUNT {
        let execution = mpc_signer_accounts[i % mpc_signer_accounts.len()]
            .call(contract.id(), method_names::PROPOSE_UPDATE)
            .args_borsh(current_contract_proposal())
            .max_gas()
            .deposit(CURRENT_CONTRACT_DEPLOY_DEPOSIT)
            .transact()
            .await
            .unwrap();

        assert!(
            execution.is_success(),
            "failed to propose update [i={i}]; {execution:#?}"
        );
        let proposal_id = execution.json().expect("unable to convert into UpdateId");
        proposals.push(proposal_id);
    }

    // Vote for the last proposal
    vote_update_till_completion(&contract, &mpc_signer_accounts, proposals.last().unwrap()).await;

    // Ensure all proposals are removed after update
    for proposal in proposals {
        let voter = mpc_signer_accounts.first().unwrap();
        let execution = voter
            .call(contract.id(), method_names::VOTE_UPDATE)
            .args_json(serde_json::json!({
                "id": proposal,
            }))
            .gas(GAS_FOR_VOTE_UPDATE)
            .transact()
            .await
            .unwrap();
        dbg!(&execution);

        assert!(execution.is_failure());
    }

    // Let's check that we can call into the state and see all the proposals.
    let state: ProtocolContractState = get_state(&contract).await;
    dbg!(state);
}
