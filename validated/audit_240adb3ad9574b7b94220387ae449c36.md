Audit Report

## Title
Stale rsETH Rate Broadcast With Refreshed Timestamp When LRTOracle Is Paused — (`contracts/cross-chain/RSETHRateProvider.sol`)

## Summary
`RSETHRateProvider.updateRate()` has no access control and no oracle-pause check. When `LRTOracle` is paused — either manually by `PAUSER_ROLE` or automatically by the price-drop circuit-breaker — `rsETHPrice` is frozen at its pre-pause value. Any unprivileged caller can invoke `updateRate()` to broadcast that frozen value cross-chain via LayerZero, causing `CrossChainRateReceiver` to overwrite its `rate` with the stale value **and** reset `lastUpdated` to the current block timestamp, making the stale rate appear fresh to downstream consumers.

## Finding Description
**Root cause:** `RSETHRateProvider.getLatestRate()` reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()`, which is a plain `uint256` public storage variable on `LRTOracle` — not a function that reverts when paused. [1](#0-0) 

`LRTOracle.rsETHPrice` is a raw storage slot: [2](#0-1) 

`updateRSETHPrice()` is gated by `whenNotPaused`, so once paused the slot cannot be refreshed: [3](#0-2) 

The auto-pause path in `_updateRsETHPrice()` pauses and returns **before** writing `newRsETHPrice`, so `rsETHPrice` retains the last pre-anomaly value: [4](#0-3) 

`CrossChainRateProvider.updateRate()` has no caller restriction and no pause check: [5](#0-4) 

`CrossChainRateReceiver.lzReceive()` blindly overwrites `rate` and resets `lastUpdated` to `block.timestamp`: [6](#0-5) 

The identical pattern exists in `RSETHMultiChainRateProvider.getLatestRate()`: [7](#0-6) 

**Exploit flow:**
1. `LRTOracle` is paused (manual or auto circuit-breaker); `rsETHPrice` is frozen at pre-pause value `X`.
2. Any EOA calls `RSETHRateProvider.updateRate{value: fee}()`.
3. `getLatestRate()` returns `X` (no revert, no pause check).
4. LayerZero delivers the payload to `CrossChainRateReceiver.lzReceive()`.
5. Receiver sets `rate = X` and `lastUpdated = block.timestamp`.
6. Downstream protocols calling `getRate()` receive `X` with a fresh timestamp, unaware the source oracle is in a degraded/anomaly state.

The critical aggravating factor is the **timestamp reset**: without the attacker's action, the receiver's `lastUpdated` would remain at the time of the last legitimate update, signaling staleness to any freshness-aware consumer. After the attacker's action, `lastUpdated` is current, masking the staleness entirely.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** The rate provider's stated purpose is to relay the *current* rsETH/ETH exchange rate. During an oracle pause it instead relays a frozen, potentially anomalous rate with a falsely fresh timestamp. Downstream pools and integrations that price rsETH via the receiver's `getRate()` will misprice it for the entire pause duration. No direct fund theft occurs, but the contract fails to deliver its core promise.

## Likelihood Explanation
- The oracle pause is a normal, expected protocol event with two independent trigger paths (manual `PAUSER_ROLE` call; automatic price-drop circuit-breaker in `_updateRsETHPrice()`).
- `updateRate()` requires no role — only ETH for LayerZero fees, which is a trivial cost.
- The longer the pause, the greater the divergence between the broadcast rate and the true rate.
- The attack is repeatable: the attacker can keep refreshing `lastUpdated` throughout the pause.

## Recommendation
Add a pause-awareness check in `RSETHRateProvider.getLatestRate()` (and identically in `RSETHMultiChainRateProvider.getLatestRate()`) that reverts if the source oracle is paused. The `ILRTOracle` interface should expose `paused()`:

```solidity
// In ILRTOracle:
function paused() external view returns (bool);

// In RSETHRateProvider and RSETHMultiChainRateProvider:
function getLatestRate() public view override returns (uint256) {
    require(!ILRTOracle(rsETHPriceOracle).paused(), "Oracle paused");
    return ILRTOracle(rsETHPriceOracle).rsETHPrice();
}
```

`LRTOracle` already exposes `paused` as a public state variable, so no contract changes are needed there — only the interface and the two rate providers need updating.

```solidity
// Foundry fork/unit test outline:
// 1. Deploy LRTOracle (or fork mainnet), deploy RSETHRateProvider pointing to it,
//    deploy mock LayerZero endpoint that records payloads.

// 3. PAUSER_ROLE calls LRTOracle.pause().
// 4. vm.warp(block.timestamp + 7 days).
//    → True rate would have drifted; rsETHPrice is still X.
// 5. anyEOA calls RSETHRateProvider.updateRate{value: fee}().
//    → getLatestRate() returns X without reverting.
//    → Mock endpoint records payload = abi.encode(X).
// 6. Simulate lzReceive on mock receiver:
//    → receiver.rate == X  (frozen pre-pause value)
//    → receiver.lastUpdated == block.timestamp  (appears fresh)
// 7. Assert: broadcast rate == X == pre-pause rsETHPrice
//            receiver.lastUpdated == block.timestamp (staleness masked)
//    → stale rate with fresh timestamp successfully propagated during oracle pause.
```
