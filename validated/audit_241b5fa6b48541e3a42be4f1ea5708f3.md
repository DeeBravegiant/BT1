### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but it never verifies that `response.payload_hash` was actually derived from the `request` argument. A single Byzantine attested participant (the signing leader) can replay a threshold signature obtained for one foreign-chain request to resolve a completely different pending request, delivering a forged attestation to the caller.

---

### Finding Description

In `respond_verify_foreign_tx`, the on-chain signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← caller-supplied, not derived from `request`
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract only checks that the signature is valid over the caller-supplied `payload_hash`. It does **not** recompute the expected hash from the `request` and compare. The `request` argument is used solely as a lookup key to drain the pending-yield queue:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The correct `payload_hash` for a request is `SHA-256(borsh(ForeignTxSignPayload::V1({ request, values })))`:

```rust
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
``` [3](#0-2) 

Because `values` (the extracted foreign-chain observations) are never stored on-chain, the contract cannot recompute the hash itself. This creates a gap: the `payload_hash` in the response is entirely decoupled from the `request` at the contract level.

**Contrast with `respond`** (regular signatures): there, the signature is verified against `request.payload` — a value that was committed on-chain when the request was submitted — so the payload is always bound to the request:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,   // ← taken from the on-chain request, not caller-supplied
    &expected_public_key,
)
``` [4](#0-3) 

`respond_verify_foreign_tx` has no equivalent binding.

---

### Impact Explanation

A Byzantine leader node can resolve pending request A with a `payload_hash` that was legitimately signed (via the threshold protocol) for a completely different request B. The caller of `verify_foreign_transaction(tx_A)` receives a `VerifyForeignTransactionResponse` whose `payload_hash` and `signature` attest to transaction B, not transaction A. [5](#0-4) 

Any downstream bridge contract that trusts the returned `(payload_hash, signature)` pair to authorize a cross-chain action (e.g., minting tokens on NEAR after verifying a deposit on Ethereum) will be operating on a forged attestation. This enables invalid bridge execution or double-spend conditions: the attacker can cause the bridge to accept a fraudulent inbound transfer by substituting the attestation of a legitimate transaction for one that was never verified.

**Impact category:** High — forged foreign-chain verification causing invalid bridge execution.

---

### Likelihood Explanation

The attacker must be a single attested MPC participant (below the signing threshold). No threshold collusion is required. The attack path is:

1. Participate honestly in the threshold signing protocol for request B, obtaining the final signature `sig(hash(B))` as the designated leader.
2. Instead of submitting the response for request B, save `(hash(B), sig(hash(B)))`.
3. When request A is pending, call `respond_verify_foreign_tx(request=A, response={payload_hash=hash(B), signature=sig(hash(B))})`.
4. The contract accepts the call (valid signature, attested caller, request A exists in the pending map) and delivers the forged response to the caller of request A. [6](#0-5) 

The leader node already holds the final threshold signature before submitting it on-chain, so no additional capability is needed beyond being the elected leader for one signing round.

---

### Recommendation

The contract must bind `response.payload_hash` to the `request`. Since the `values` (extracted observations) are not stored on-chain, the binding cannot be a full hash recomputation. Two complementary mitigations:

1. **Include the request hash in the signed payload.** Modify `ForeignTxSignPayloadV1` to include a commitment to the `VerifyForeignTransactionRequest` (e.g., its Borsh hash). The contract can then recompute `SHA-256(borsh(request))` and verify it is embedded in the signed payload, ensuring the signature is request-specific.

2. **Store a per-request expected-hash commitment on-chain.** When `verify_foreign_transaction` is called, store `SHA-256(borsh(request))` alongside the yield index. In `respond_verify_foreign_tx`, verify that `response.payload_hash` starts with (or is derived from) this commitment before accepting the response.

Either approach closes the gap by making the `payload_hash` verifiably bound to the originating request at the contract level, analogous to how `respond` binds the signature to `request.payload`. [7](#0-6) 

---

### Proof of Concept

**Setup:** Two foreign-chain verification requests are submitted:
- Request A: `verify_foreign_transaction(Bitcoin, tx_id=[0xAA;32], ...)` — a fraudulent or low-value transaction
- Request B: `verify_foreign_transaction(Bitcoin, tx_id=[0xBB;32], ...)` — a legitimate high-value bridge deposit

**Steps:**

1. The Byzantine leader node is elected for request B. It runs the threshold signing protocol with the other nodes, who each independently compute `hash(B) = SHA-256(borsh(ForeignTxSignPayload::V1({request: tx_B, values: observed_B})))` and sign it. The leader obtains the final threshold signature `sig(hash(B))`.

2. The leader does **not** submit the response for request B. Instead, it calls:
   ```
   respond_verify_foreign_tx(
       request = VerifyForeignTransactionRequest { tx_id=[0xAA;32], ... },  // request A
       response = { payload_hash: hash(B), signature: sig(hash(B)) }
   )
   ```

3. The contract at line 729 verifies `sig(hash(B))` over `hash(B)` using the root public key → **valid** (it is a genuine threshold signature).

4. The contract at line 749 resolves request A's yield queue with `serde_json::to_vec(&response)`, delivering `{payload_hash: hash(B), signature: sig(hash(B))}` to the caller of request A.

5. The caller of request A (e.g., a bridge contract) receives a `VerifyForeignTransactionResponse` that appears to be a valid MPC attestation. It reconstructs the expected hash for tx_A and finds it does not match `payload_hash` — **but only if the caller performs this check**. If the bridge contract trusts the returned signature without recomputing the hash from the original request, it will accept the forged attestation and authorize an invalid cross-chain action.

6. The leader then submits the correct response for request B (or lets it time out), and separately withdraws the bridge funds unlocked by the forged attestation for request A. [8](#0-7)
