### Title
`respond_verify_foreign_tx` Accepts Replayed Signatures From Unrelated Requests — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the MPC signature in the response is valid over `response.payload_hash`, but never verifies that `response.payload_hash` is actually the hash of `ForeignTxSignPayload::V1({request, extracted_values})` for the submitted `request`. A single attested participant can replay any previously-observed legitimate `(payload_hash, signature)` pair against a different pending request, causing the contract to resolve that request's yield with a forged attestation.

### Finding Description

The external report's vulnerability class is: **accounting state updated without the corresponding action being taken** — `pendingTokenWithdrawals` is incremented but the token transfer that should back it is never executed. The NEAR MPC analog is: **request state resolved (yield resumed) without verifying that the response actually corresponds to the request**.

In `respond_verify_foreign_tx` the contract performs two independent steps:

1. Verify `response.signature` is a valid MPC signature over `response.payload_hash`.
2. Drain all yields queued under `request` from `pending_verify_foreign_tx_requests`. [1](#0-0) 

The critical missing step is: **verify that `response.payload_hash` is the hash of `ForeignTxSignPayload::V1({request, values})`**. The `payload_hash` is a free caller-supplied field in the response; the contract never checks that it encodes the same `request` that is being resolved.

Compare this to the regular `respond` function, where the signature is verified over `request.payload` — a field that is part of the request itself, so the binding is structural: [2](#0-1) 

For foreign-tx verification the signed payload is `SHA-256(borsh(ForeignTxSignPayload::V1({request, values})))`: [3](#0-2) 

The SDK-level verifier (`ForeignChainSignatureVerifier::verify_signature`) does check this binding on the caller side: [4](#0-3) 

But the on-chain contract does not, so the contract itself delivers an unverified response to the caller.

### Impact Explanation

A single attested participant can:

1. Observe any past legitimate `respond_verify_foreign_tx` call on-chain, obtaining a valid `(hash_A, sig_A)` pair for Request A (e.g., Bitcoin tx X, block-hash `0xAA`).
2. Wait for (or themselves submit) a new `verify_foreign_transaction` Request B (e.g., Bitcoin tx Y).
3. Call `respond_verify_foreign_tx(request = B, response = {payload_hash: hash_A, signature: sig_A})`.
4. The contract checks `verify_ecdsa_signature(sig_A, hash_A, mpc_pk)` — valid, because `sig_A` is a genuine MPC signature.
5. `resolve_yields_for` drains all yields queued under Request B and resumes them with the forged response. [5](#0-4) 

The caller of Request B receives `{payload_hash: hash_A, signature: sig_A}`. Any bridge contract that checks only signature validity (not the `payload_hash` binding) will accept this as proof that Bitcoin tx Y was verified, when in fact the MPC network only ever attested to Bitcoin tx X. This enables invalid bridge execution and double-spend conditions: the same `(hash_A, sig_A)` can be replayed against every pending foreign-tx request whose chain and domain match.

### Likelihood Explanation

The attacker needs only to be a single attested participant — strictly below the signing threshold. No threshold collusion is required. Legitimate `(payload_hash, signature)` pairs are published on-chain by every honest `respond_verify_foreign_tx` call and are trivially observable. The attacker submits their own `verify_foreign_transaction` request (costing 1 yoctoNEAR) to create a pending target, then immediately replays the harvested response. The attack is fully permissionless within the participant set and requires no special tooling.

### Recommendation

The contract must verify that `response.payload_hash` encodes the same `request` that is being resolved. Because the hash also covers the extracted `values` (which the contract does not know), the cleanest fix is to extend `VerifyForeignTransactionResponse` to carry the extracted values alongside the hash:

```rust
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,
    pub values: Vec<ExtractedValue>,   // add this
    pub signature: SignatureResponse,
}
```

Then in `respond_verify_foreign_tx`, before accepting the signature, recompute and assert:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::InvalidSignature.into());
}
```

This mirrors the check already present in `ForeignChainSignatureVerifier::verify_signature` and closes the replay path at the contract layer.

### Proof of Concept

```
// Setup: two Bitcoin tx verification requests are pending.
// Request A: tx_id = [0xAA; 32], pending in contract.
// Request B: tx_id = [0xBB; 32], pending in contract.

// Step 1: Honest nodes process Request A and one participant calls:
contract.respond_verify_foreign_tx(
    request = { domain_id: 0, request: Bitcoin { tx_id: [0xAA;32], ... } },
    response = { payload_hash: hash_A, signature: sig_A }
);
// hash_A = SHA-256(borsh(ForeignTxSignPayload::V1({ Bitcoin{0xAA}, [BlockHash(0x11)] })))
// sig_A  = valid MPC ECDSA signature over hash_A

// Step 2: Malicious participant replays the same response against Request B:
contract.respond_verify_foreign_tx(
    request = { domain_id: 0, request: Bitcoin { tx_id: [0xBB;32], ... } },
    response = { payload_hash: hash_A, signature: sig_A }   // ← replayed
);
// Contract checks: verify_ecdsa_signature(sig_A, hash_A, mpc_pk) → OK
// Contract resolves yields for Request B with { payload_hash: hash_A, sig_A }

// Result: caller of Request B receives a response whose payload_hash
// encodes Bitcoin tx 0xAA, not 0xBB. Any bridge that trusts the
// contract's response without re-checking the payload_hash binding
// will incorrectly treat 0xBB as verified.

        Ok(Hash256(hasher.finalize().into()))

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
