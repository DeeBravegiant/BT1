Audit Report

## Title
ETH Deposit Limit Bypass via Missing Amount in Limit Check — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits `amount` from the ETH branch of its cap check, evaluating whether the limit has *already* been exceeded rather than whether the incoming deposit *would* exceed it. Any unprivileged depositor can bypass the ETH deposit cap entirely in a single transaction, depositing an arbitrarily large amount as long as the pre-call running total has not yet crossed the limit.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the function `_checkIfDepositAmountExceedesCurrentLimit` contains an asymmetric check:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount absent
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount present
}
``` [1](#0-0) 

The ETH branch asks "has the cap already been breached?" rather than "would this deposit breach the cap?" The `amount` parameter (`msg.value`) is never factored in.

The public entry point `depositETH` (lines 76–93) passes `msg.value` as `depositAmount` to `_beforeDeposit` (lines 648–670), which calls this check at line 661. Because the check ignores `amount` for ETH, the guard returns `false` (not exceeded) for any deposit size as long as `totalAssetDeposits ≤ limit` before the call. [2](#0-1) [3](#0-2) 

The ERC20 path (`depositAsset`, lines 99–118) correctly includes `depositAmount` in the comparison and is unaffected. [4](#0-3) 

## Impact Explanation
The deposit limit is a risk-management parameter set by the admin to cap protocol exposure to EigenLayer strategy capacity. Because the ETH branch of the guard is effectively a no-op until the cap is already exceeded, a single depositor can push the protocol's ETH holdings arbitrarily above the intended ceiling in one transaction. Excess ETH that cannot be deployed to EigenLayer strategies sits idle in the deposit pool earning no yield, diluting returns for all rsETH holders.

**Impact: Low — Contract fails to deliver promised returns, but does not lose value.**

## Likelihood Explanation
`depositETH` is `external payable` with no role restriction. No flash loan, multi-step setup, or privileged access is required. The condition is trivially met whenever the protocol is below its cap, which is the normal operating state. Any depositor with sufficient ETH can trigger this in a single transaction.

## Recommendation
Include `amount` in the ETH branch, consistent with the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [5](#0-4) 

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `0` (fresh protocol).
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `0 > 100 ether` → `false` → guard does not revert.
5. `_mintRsETH` mints rsETH proportional to 10,000 ETH; protocol now holds 100× the intended cap.
6. All subsequent ETH deposits are blocked (total now exceeds limit), but the excess is already accepted and rsETH minted.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(LRTConstants.ETH_TOKEN, 100 ether);

    address attacker = makeAddr("attacker");
    vm.deal(attacker, 10_000 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 10_000 ether}(0, "");

    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), 100 ether);
}
```
