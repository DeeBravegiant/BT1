### Title
Cache Poisoning via Unconditional `add_to_additions_in_block` After Failed Additions Validation — (`chia/wallet/util/wallet_sync_utils.py`)

### Summary

In `request_and_validate_additions`, `peer_request_cache.add_to_additions_in_block` is called **unconditionally** at line 223 regardless of whether `validate_additions` returned `True` or `False`. Because the cache check at line 205 returns `True` whenever the entry exists, a subsequent call for the same `(header_hash, puzzle_hash)` pair bypasses the Merkle proof entirely. Combined with the fact that `peer_caches` is keyed by `peer_node_id` and is **never evicted on disconnect**, an attacker who reconnects with the same node ID exploits the poisoned cache to have a fabricated coin state accepted.

---

### Finding Description

**Root cause — unconditional cache write:** [1](#0-0) 

```python
result: bool = validate_additions(
    additions_res.coins,
    additions_res.proofs,
    additions_root,
)
peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)  # line 223 — unconditional
return result
```

`add_to_additions_in_block` is called even when `result` is `False`. The cache stores only presence, not the result: [2](#0-1) 

So on the next call for the same key, line 205–206 fires: [3](#0-2) 

…and returns `True` unconditionally, skipping all Merkle proof work.

**Cache persists across reconnections:** [4](#0-3) 

`peer_caches` is a plain dict on `WalletNode`, keyed by `peer_node_id`. There is no eviction on peer disconnect. An attacker who reconnects with the same key (same `peer_node_id`) receives the same poisoned `PeerRequestCache`.

**Peer closure does not protect the second call:**

When `validate_additions` returns `False`, the peer is closed: [5](#0-4) 

But the guard at the top of `validate_received_state_from_peer` checks `peer.closed` on the **connection object**, not on the cache: [6](#0-5) 

A freshly reconnected peer object is not closed, so this guard is bypassed. The `can_use_peer_request_cache` shortcut checks `_states_validated` (which was never populated for the fabricated state), so it also does not fire. Execution reaches `request_and_validate_additions`, which hits the poisoned cache and returns `True`.

**Subsequent validation does not re-check coin inclusion:**

After the additions check returns `True` via cache, `validate_block_inclusion` is called: [7](#0-6) 

This validates only that the **block** is in the longest chain (weight proof + signatures). It does **not** re-verify that the specific coin/puzzle hash is in the block's additions Merkle tree. If the attacker is on the real chain and references a real block, this check passes.

---

### Impact Explanation

A wallet in untrusted sync mode accepts a fabricated `CoinState` as valid — crediting a coin that was never created on-chain. This satisfies the Critical impact criterion: *unauthorized creation of XCH/asset coin records with direct accounting impact*.

---

### Likelihood Explanation

The attacker needs only to:
1. Run a full node on the real chain (no special privilege).
2. Connect to the victim wallet as an untrusted peer.
3. Send a `CoinStateUpdate` with a fabricated coin at a real block's puzzle hash.
4. Respond to `RequestAdditions` with deliberately invalid proof data (trivially crafted).
5. Reconnect with the same node key and repeat the coin state claim.

No leaked keys, no cryptographic breaks, no admin access required.

---

### Recommendation

Condition the cache write on a successful validation result:

```python
result: bool = validate_additions(
    additions_res.coins,
    additions_res.proofs,
    additions_root,
)
if result:
    peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)
return result
```

Additionally, consider evicting `peer_caches[peer_node_id]` when a peer is closed with a ban code, to prevent stale poisoned entries from surviving reconnections.
