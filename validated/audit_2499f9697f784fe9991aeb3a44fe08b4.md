Audit Report

## Title
ETH Transfer to Contract Recipient Without `receive` Permanently Freezes User Funds After rsETH Burn - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._transferAsset` delivers ETH via `payable(to).call{value: amount}("")`. If the recipient is a smart contract without a `receive` or `fallback` function, the call returns `false`, `EthTransferFailed` is thrown, and the entire `completeWithdrawal` transaction reverts. Because rsETH is burned in a prior, separate `unlockQueue` transaction, the user's rsETH is permanently destroyed while the corresponding ETH allocation is permanently frozen inside `LRTWithdrawalManager` with no recovery path.

## Finding Description
The withdrawal lifecycle spans two separate transactions:

**Transaction 1 — `initiateWithdrawal` (L150-178):** The user's rsETH is pulled into `LRTWithdrawalManager` via `safeTransferFrom` and a `WithdrawalRequest` is recorded.

**Transaction 2 — `unlockQueue` (L280-320):** The operator calls this function, which burns all queued rsETH via `IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned)` at L305 and redeems the corresponding ETH from `LRTUnstakingVault` into `LRTWithdrawalManager`. After this transaction, the rsETH no longer exists.

**Transaction 3 — `completeWithdrawal` / `completeWithdrawalForUser` (L183-204):** Both call `_processWithdrawalCompletion` (L699-738), which:
1. Pops the nonce from `userAssociatedNonces` (L705)
2. Deletes `withdrawalRequests[requestId]` (L712)
3. Decrements `unlockedWithdrawalsCount[asset]` (L717)
4. Calls `_transferAsset(asset, user, request.expectedAssetAmount)` (L734)

`_transferAsset` (L876-883) executes:
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract without `receive`/`fallback`, `sent` is `false`, `EthTransferFailed` is thrown, and the **entire transaction reverts** — undoing the `popFront`, the `delete`, and the counter decrement. The withdrawal request is restored to the queue, but every subsequent attempt to complete it will fail identically. The ETH is permanently locked.

There is no admin escape hatch: `sweepRemainingAssets` (L395-414) checks `hasUnlockedWithdrawals(asset)` (L629-631), which returns `true` while `unlockedWithdrawalsCount[asset] > 0`. Since the failing `completeWithdrawal` always reverts before decrementing that counter, `sweepRemainingAssets` is permanently blocked for the affected asset.

The developer comment at L191 — *"Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH"* — acknowledges ETH-specific risks on `completeWithdrawalForUser` but does not address the permanent freeze scenario.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is irreversibly burned in `unlockQueue` (a committed on-chain state change in a separate transaction). The ETH allocated to that user sits in `LRTWithdrawalManager` indefinitely. No admin function can redirect the locked withdrawal to an alternative address, and `sweepRemainingAssets` is gated behind `hasUnlockedWithdrawals` returning `false`, which it never will for the stuck request.

## Likelihood Explanation
**Medium.** Smart-contract wallets (Gnosis Safe without ETH module), protocol treasuries, EigenLayer operator contracts, and DeFi vaults routinely hold rsETH and interact with LRT withdrawal flows. Any such contract that calls `initiateWithdrawal` without implementing `receive() external payable` will trigger this freeze. The precondition — a contract caller without a `receive` function — is a common, realistic pattern and requires no privileged access or attacker cooperation.

## Recommendation
Replace the push-ETH pattern with a pull pattern: record the owed ETH amount per user in a mapping and expose a separate `claimETH()` function that the user calls to withdraw. Alternatively, wrap ETH into WETH before transferring (`IWETH(WETH_ADDRESS).deposit{value: amount}(); IERC20(WETH_ADDRESS).safeTransfer(to, amount);`), which always succeeds regardless of the recipient's fallback implementation. The `WETH_ADDRESS` constant is already present in the contract at L66.

## Proof of Concept
1. Deploy `VaultUser` — a contract with no `receive` function that holds rsETH and can call `initiateWithdrawal`.
2. `VaultUser` approves `LRTWithdrawalManager` and calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; a `WithdrawalRequest` is stored.
3. After `withdrawalDelayBlocks`, the operator calls `unlockQueue(ETH_TOKEN, ...)`. At L305, `burnFrom(address(this), rsETHBurned)` permanently destroys the rsETH. ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager`.
4. `VaultUser` calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion`, `_transferAsset` executes `payable(VaultUser).call{value: amount}("")`. Because `VaultUser` has no `receive`, the call returns `false`. `EthTransferFailed` is thrown; the transaction reverts entirely.
5. Step 4 can be repeated indefinitely — it always reverts. `VaultUser`'s rsETH is permanently burned and the ETH is permanently locked in `LRTWithdrawalManager`. `sweepRemainingAssets` also reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN]` remains > 0.

**Foundry test sketch:**
```solidity
contract NoReceive {
    function doInitiate(address wm, address rsETH, uint256 amt) external {
        IERC20(rsETH).approve(wm, amt);
        ILRTWithdrawalManager(wm).initiateWithdrawal(ETH_TOKEN, amt, "");
    }
    function doComplete(address wm) external {
        ILRTWithdrawalManager(wm).completeWithdrawal(ETH_TOKEN, ""); // always reverts
    }
}
// After unlockQueue is called by operator, doComplete() reverts with EthTransferFailed every time.
```
