### Title
Missing Cryptographic Output Verification in `respond_ckd` for `AppPublicKey` Variant Allows Single Node to Forge CKD Response — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in `MpcContract` performs **no cryptographic verification** of the submitted `CKDResponse` when the request was made with the `CKDAppPublicKey::AppPublicKey` variant. A single attested MPC participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary `big_c` / `big_y` pair, the contract accepts it unconditionally, resolves the yield, and the user receives a fully attacker-controlled confidential key derivation output. This bypasses the threshold requirement that is the core security property of the MPC network.

---

### Finding Description

The `respond` and `respond_ckd` (PV path) functions both verify the cryptographic correctness of the submitted response before resolving the yield:

- `respond` verifies the ECDSA/EdDSA signature against the derived public key via `near_mpc_signature_verifier::verify_ecdsa_signature` / `verify_eddsa_signature`. [1](#0-0) 
- `respond_ckd` with `AppPublicKeyPV` calls `ckd_output_check`, which verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`. [2](#0-1) 

However, when the `AppPublicKey` variant is used, the match arm is a no-op:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [2](#0-1) 

After this match, `resolve_yields_for` is called unconditionally, which removes the request from the pending map and resumes every queued yield with the (unverified) response bytes. [3](#0-2) 

The `resolve_yields_for` helper drains the entire queue in one call and removes the map entry, so once a single node submits a response the request is gone and no honest node can respond. [4](#0-3) 

The existing unit test for this path confirms the absence of verification: it submits `big_y: [1u8; 48]` and `big_c: [2u8; 48]` — byte arrays that are not valid BLS12-381 G1 points — and the test expects success. [5](#0-4) 

The `ckd_output_check` function that *is* used for the PV path demonstrates that the contract has all the information needed to verify the response: `app_id`, `big_c`, `big_y`, `app_pk2`, and the network public key. The `AppPublicKey` variant simply omits `pk2`, which is the only reason the pairing check cannot be applied directly. [6](#0-5) 

---

### Impact Explanation

The CKD protocol's security guarantee is that no single node below the threshold can learn or control the derived key `s = msk · H(app_id)`. By submitting an arbitrary `(big_y, big_c)` pair, a single malicious node can force the user to derive any key of the attacker's choosing:

- Set `big_y = G1_identity` (the identity point). The user computes `s = big_c − app_sk · identity = big_c`. The attacker fully controls `s`.
- Alternatively, set `big_y` and `big_c` to any values; the user derives a key that has no relationship to the honest MPC computation.

This is **unauthorized confidential key derivation output without the required participant authorization** — a single node below the signing threshold produces a result that should require threshold-many honest nodes to compute. The user's application (running in a TEE) will derive and use a key that the attacker chose, breaking the confidentiality and integrity guarantees of the CKD protocol.

---

### Likelihood Explanation

- The `AppPublicKey` variant is the primary CKD variant (the `AppPublicKeyPV` variant is the publicly-verifiable extension). Any `request_app_private_key` call using `AppPublicKey` is vulnerable.
- The attacker only needs to be a single attested MPC participant — a threshold-1 Byzantine node suffices.
- The attack is a simple race: the malicious node monitors the NEAR chain for pending CKD requests and calls `respond_ckd` with a forged response before honest nodes do. Because `resolve_yields_for` removes the request on first response, honest nodes cannot overwrite the forged result.
- No special knowledge of the user's `app_sk` is required to corrupt the output; arbitrary `big_c` / `big_y` values are sufficient.

---

### Recommendation

For the `AppPublicKey` variant, the contract cannot apply the full pairing check (because `pk2` is absent). Two mitigations are possible:

1. **Require `AppPublicKeyPV` for all on-chain CKD requests.** The `AppPublicKey` variant provides no on-chain verifiability and should be deprecated in favour of `AppPublicKeyPV`, which admits a complete pairing check.
2. **Require threshold-many matching responses before resolving the yield.** Collect `respond_ckd` calls and only resolve when at least `threshold` nodes have submitted the same `(big_c, big_y)` pair, analogous to how `vote_pk` collects threshold votes before accepting a public key.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)`. A `CKDRequest` is stored in `pending_ckd_requests`.
2. Malicious attested node (participant index `i`, `i < threshold`) calls:
   ```
   respond_ckd(ckd_request, CKDResponse { big_y: G1_identity, big_c: attacker_chosen_point })
   ```
3. The contract executes the `AppPublicKey(_) => {}` branch — no check — and calls `resolve_yields_for`, which resumes the yield with the forged bytes and removes the request from the map.
4. Honest nodes attempt `respond_ckd` and receive `RequestNotFound`.
5. The user's callback receives `CKDResponse { big_y: G1_identity, big_c: attacker_chosen_point }` and derives `s = attacker_chosen_point − app_sk · identity = attacker_chosen_point`.

The existing test at line 3403 already demonstrates step 3 with invalid point encodings accepted without error. [5](#0-4)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
