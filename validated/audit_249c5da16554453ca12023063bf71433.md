### Title
Direct Deposits Not Included in Value Conservation Check Allows Arbitrary ADA Creation - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary

In the Dijkstra era, the `directDeposits` field of a transaction body allows arbitrary coin amounts to be credited to registered account balances via `applyDirectDeposits`. However, these amounts are absent from both the `getConsumedDijkstraValue` and `getProducedDijkstraValue` calculations. The preservation-of-value check (`validateValueNotConservedUTxO`) therefore never accounts for them. Any transaction author can include arbitrarily large direct-deposit amounts, pass the balance check, and have those amounts added to account balances — effectively minting ADA out of thin air.

### Finding Description

The Dijkstra era introduces `directDeposits` (`directDepositsTxBodyL`) as a new transaction-body field. During the `ENTITIES` rule transition, after certificates are processed, the rule reads the field and calls `applyDirectDeposits`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [1](#0-0) 

The only guard is `directDepositsMissingAccounts`, which rejects deposits targeting unregistered accounts. There is no check that the deposited amounts are covered by UTxO inputs.

`applyDirectDeposits` unconditionally adds the attacker-supplied coin to each target account balance:

```haskell
-- libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [2](#0-1) 

Meanwhile, the Dijkstra produced-value calculation is:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody          -- outputs + fee + cert deposits + treasury donation
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
``` [3](#0-2) 

`conwayProducedValue` sums outputs, fee, certificate deposits, and treasury donation — direct deposits are absent.

The consumed-value calculation is equally silent about direct deposits:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels txBody
    (\topTxBody -> txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody)
    txBodyConsumedValue
  where
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
``` [4](#0-3) 

`getConsumedMaryValue` sums UTxO inputs, minted multi-assets, refunds, and withdrawals — again, no direct deposits. [5](#0-4) 

The preservation-of-value check in `dijkstraUtxoTransition` therefore never sees the direct-deposit amounts:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [6](#0-5)

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

Because the direct-deposit amounts are invisible to the preservation-of-value check, a transaction author can credit any amount to any registered account without providing the corresponding ADA in UTxO inputs. The total lovelace in the system (UTxO + account balances + deposits + treasury + reserves) increases by the sum of all direct-deposit amounts in every such transaction, violating the fundamental invariant of the Cardano ledger.

### Likelihood Explanation

**High.** The attack requires only:
1. A valid transaction with at least one registered target account address.
2. Sufficient UTxO inputs to cover outputs + fee (a normal, small amount).
3. A `directDeposits` map with arbitrarily large coin values.

No privileged access, governance majority, or leaked key is needed. Any unprivileged transaction sender can exploit this in the Dijkstra era.

### Recommendation

Include the sum of all direct-deposit amounts on the **produced** side of the value-conservation equation in `dijkstraProducedValue` (and the analogous sub-transaction function `dijkstraSubTxProducedValue`):

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> inject (foldMap id . unDirectDeposits $ txBody ^. directDepositsTxBodyL)  -- ADD THIS
    <> foldMap'
         (getProducedValue pp isRegPoolId . view bodyTxL)
         (txBody ^. subTransactionsTxBodyL)
```

This ensures that the ADA credited to account balances via direct deposits must be supplied by the transaction's UTxO inputs, preserving the total-ADA invariant.

### Proof of Concept

1. Attacker controls a UTxO output worth 2 ADA and a registered account address `victimAddr`.
2. Attacker submits a Dijkstra-era transaction:
   - `inputs  = { utxo_2ada }`
   - `outputs = { attacker_change: 1 ADA }`
   - `fee     = 1 ADA`
   - `directDeposits = { victimAddr: 1_000_000_000_000 ADA }`
3. `validateValueNotConservedUTxO`: consumed = 2 ADA, produced = 1 + 1 = 2 ADA → **passes**.
4. `directDepositsMissingAccounts`: `victimAddr` is registered → **passes**.
5. `applyDirectDeposits` credits 1 000 000 000 000 ADA to `victimAddr`'s account balance.
6. Total lovelace in the ledger has increased by 1 000 000 000 000 ADA — ADA created from nothing.

The attacker can repeat this with any registered account and any amount, bounded only by the `Coin` type's maximum value.

getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  consumedValue <> MaryValue mempty mintedMultiAsset
  where
