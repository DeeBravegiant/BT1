Audit Report

## Title
Missing Slippage Protection on `deposit()` Functions in L2 RSETHPool Contracts - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV2.sol`)

## Summary
All L2 pool `deposit()` functions compute the minted `wrsETH`/`rsETH` amount using a live oracle rate fetched at execution time but accept no `minRSETHAmountOut` parameter. When the oracle rate is updated via LayerZero between a user's transaction submission and its on-chain execution, the user receives fewer tokens than the view function quoted at submission time with no on-chain recourse. The L1 `LRTDepositPool` has an equivalent `minRSETHAmountExpected` guard that the L2 contracts lack.

## Finding Description
Every L2 pool variant's `deposit()` function calls `viewSwapRsETHAmountAndFee()`, which reads the live oracle rate via `getRate()` at execution time:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) 

The rate is updated asynchronously by `CrossChainRateReceiver.lzReceive()` whenever a LayerZero message arrives from L1: [2](#0-1) 

None of the five affected `deposit()` overloads accept or enforce a minimum output: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

By contrast, the L1 `LRTDepositPool.depositETH()` accepts `minRSETHAmountExpected` and enforces it in `_beforeDeposit()`: [9](#0-8) [10](#0-9) 

No existing check in any L2 pool prevents execution when the computed output falls below the user's expectation.

## Impact Explanation
The contract fails to deliver the token quantity the user observed via `viewSwapRsETHAmountAndFee` at submission time. Because `wrsETH` is pegged 1:1 to `rsETH` and the rate increase means each `rsETH` is worth more ETH, the user's deposited ETH value is preserved — there is no fund loss. The impact is therefore: **Low — Contract fails to deliver promised returns, but doesn't lose value**, which is an explicitly allowed impact in scope.

## Likelihood Explanation
The rsETH/ETH rate increases monotonically as EigenLayer staking rewards accrue. Rate updates are pushed periodically from L1 to L2 via LayerZero. Any user whose deposit transaction is pending in the mempool at the moment a rate update lands will receive fewer tokens than the UI quoted. No attacker action is required — this is a structural property of the protocol's asynchronous rate propagation model and will recur with every rate update cycle.

## Recommendation
Add a `minRSETHAmountOut` parameter to all `deposit()` overloads across all five affected contracts and revert if the computed amount falls below it:

```diff
- function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
+ function deposit(string memory referralId, uint256 minRSETHAmountOut) external payable nonReentrant whenNotPaused {
      ...
      (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
+     if (rsETHAmount < minRSETHAmountOut) revert InsufficientOutputAmount();
      ...
  }
```

Apply the same pattern to the token `deposit(address token, uint256 amount, string memory referralId)` overloads in `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `RSETHPoolNoWrapper.sol`. The UI should pre-compute the expected output via `viewSwapRsETHAmountAndFee` and pass a user-configurable slippage tolerance as `minRSETHAmountOut`.

## Proof of Concept
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3`. Oracle rate is `1.05e18`. Expected output: `~0.952 wrsETH`.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the tx is mined, a LayerZero message updates the oracle rate to `1.06e18` via `CrossChainRateReceiver.lzReceive()`.
4. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.06e18 ≈ 0.943 wrsETH`.
5. User receives `~0.943 wrsETH` instead of `~0.952 wrsETH` (~0.9% fewer tokens) with no revert and no on-chain protection.

**Foundry fork test plan:**
```solidity
function testDepositSlippage() public {
    uint256 rateBefore = pool.getRate(); // e.g. 1.05e18
    // Simulate LayerZero rate update landing before user tx
    vm.prank(layerZeroEndpoint);
    crossChainRateReceiver.lzReceive(..., abi.encode(1.06e18));
    // User deposits 1 ETH expecting ~0.952 wrsETH
    uint256 balBefore = wrsETH.balanceOf(user);
    vm.prank(user);
    pool.deposit{value: 1 ether}("ref");
    uint256 received = wrsETH.balanceOf(user) - balBefore;
    // received ≈ 0.943e18, less than the 0.952e18 quoted at submission time
    assertLt(received, 0.952e18);
}
```
