Audit Report

## Title
Missing Chainlink Oracle Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Stale/Invalid Price Acceptance - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `roundId`, `updatedAt`, and `answeredInRound`, performing no staleness, completeness, or sign checks on the returned price. A stale inflated asset price allows any depositor to receive more rsETH than their deposit is worth, diluting existing rsETH holders. The same codebase already implements all three checks in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the protocol is aware of the requirement.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` (contracts/oracles/ChainlinkPriceOracle.sol, L49–55) fetches price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Three validations are absent:

1. **No staleness check** — `answeredInRound >= roundId` is never verified. A round whose answer was carried over from a prior round (`answeredInRound < roundId`) is silently accepted.
2. **No incomplete-round check** — `updatedAt != 0` is never verified. An in-progress round returns `updatedAt == 0` and is silently accepted.
3. **No sign check** — `price > 0` is never verified. In Solidity 0.8, `uint256(negative_int256)` does not revert; it wraps to a value near `type(uint256).max`.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` (contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol, L30–32) performs all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The exploit path for the stale-price case:
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) enters a stale state — `answeredInRound < roundId` — with the last reported price above the true current value.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")` (public, no role required).
3. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns the stale inflated price with no revert.
4. `rsethAmountToMint = (amount * stalePriceInflated) / lrtOracle.rsETHPrice()` — attacker receives more rsETH than their deposit is worth.
5. Attacker redeems the excess rsETH, extracting value from existing holders.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (LRTOracle.sol, L252–266) does not protect against this: a modestly stale price (e.g., 5% above true value) produces a rsETH price increase well within normal daily bounds and does not trigger `PriceAboveDailyThreshold`. The sanity check in `updatePriceOracleForValidated()` (LRTOracle.sol, L103–105) runs only at oracle registration time, not at each price fetch.

## Impact Explanation
**High — Theft of unclaimed yield.** When a depositor mints rsETH using a stale inflated asset price, they receive more rsETH than the deposited assets are worth. This dilutes the rsETH/ETH exchange rate for all existing holders, effectively transferring their accrued yield to the attacker. The entry point (`depositAsset`) is public and requires no special role or privilege.

## Likelihood Explanation
Chainlink feeds have documented heartbeat windows (24 h on mainnet, 1 h on some L2s). During network congestion, sequencer downtime (L2), or a Chainlink node outage, a feed can go stale within its heartbeat window without triggering a deviation update. This is a well-known, historically observed condition. The exploit is permissionless and repeatable for as long as the feed remains stale. The protocol's own use of the three checks in `ChainlinkOracleForRSETHPoolCollateral` confirms awareness of the risk.

## Recommendation
Apply the same three-check pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "Stale price");
    require(updatedAt != 0, "Incomplete round");
    require(price > 0, "Invalid price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, add a `block.timestamp - updatedAt <= MAX_STALENESS` check with a configurable heartbeat threshold per asset feed.

## Proof of Concept
**Foundry fork test outline:**

```solidity
// Fork mainnet at a block where stETH/ETH feed is fresh.
// Warp forward past the feed's heartbeat (e.g., 25 hours).
// At this point answeredInRound < roundId on the live feed.
// Record rsETHPrice before attack.
uint256 priceBefore = lrtOracle.rsETHPrice();

// Attacker deposits stETH.
vm.startPrank(attacker);
stETH.approve(address(depositPool), 1000e18);
depositPool.depositAsset(address(stETH), 1000e18, 0, "");
uint256 rsethReceived = rsETH.balanceOf(attacker);

// Compute fair rsETH amount at true price.
uint256 fairAmount = (1000e18 * truePrice) / priceBefore;

// Assert attacker received more than fair amount.
assertGt(rsethReceived, fairAmount);
// Assert rsETHPrice decreased after price update, confirming dilution.
lrtOracle.updateRSETHPrice();
assertLt(lrtOracle.rsETHPrice(), priceBefore);
```

The test demonstrates that `depositAsset` succeeds with a stale price, the attacker receives excess rsETH, and the subsequent `rsETHPrice` update reflects the dilution of existing holders.
