### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Byzantine Participant - (File: crates/contract/src/lib.rs)

### Summary
`respond_verify_foreign_tx` verifies only that the node-supplied `response.payload_hash` carries a valid root-key signature. It never checks that `payload_hash` is the canonical hash of `(request, observed_values)` for the specific pending request being resolved. A single malicious attested participant — strictly below the signing threshold — can replay a legitimately produced signature from any prior completed request onto any other pending request, causing the contract to deliver a forged foreign-chain verification attestation to the caller.

### Finding Description
In `respond_verify_foreign_tx`, the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

`payload_hash` is taken verbatim from the node-supplied `response` without any derivation from the stored `request`. The contract only checks that the signature is valid over whatever hash the node chose to submit. It never computes or compares the expected hash `SHA-256(borsh(ForeignTxSignPayload{request: R, values: observed_values}))`.

The correct hash is defined in `ForeignTxSignPayload::compute_msg_hash`:

```rust
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
```

where `ForeignTxSignPayloadV1` binds both the original `request` and the `values` extracted from the foreign chain. The contract stores the `request` in `pending_verify_foreign_tx_requests` but never uses it to reconstruct or validate the expected `payload_hash` at response time.

After the signature check passes, the contract resolves the pending request with the full, unchecked `response`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

The `request` key is used only to look up which yield to resume — not to validate the content of `response`. [1](#0-0) [2](#0-1) 

### Impact Explanation
A single Byzantine attested participant (below the signing threshold) can:

1. Observe a legitimately produced signature `S1` over `H1 = SHA-256(borsh(ForeignTxSignPayload{request: R1, values: V1}))` from any prior completed request `R1`. This signature is published on-chain when the honest leader calls `respond_verify_foreign_tx(R1, {payload_hash: H1, signature: S1})`.
2. Wait for any new pending request `R2` (for a different foreign-chain transaction) to appear in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(R2, {payload_hash: H1, signature: S1})`.
4. The contract verifies `S1` is valid over `H1` using the root public key — it is — and resolves `R2` with the forged response `(H1, S1)`.
5. The caller of `R2` receives a `VerifyForeignTransactionResponse` whose `payload_hash` is `H1` (the hash of `R1`'s data), not the hash of `R2`'s data.

Bridge contracts or NEAR smart contracts consuming this response without independently re-deriving the expected hash from `R2`'s transaction data would accept a forged attestation. This enables invalid bridge execution or double-spend conditions: an attacker can make the MPC network appear to have verified a transaction it never inspected.

The design documentation acknowledges that callers are expected to verify `payload_hash` locally, but the contract itself provides no enforcement, making the security guarantee entirely dependent on every downstream consumer implementing this check correctly. [3](#0-2) 

### Likelihood Explanation
The attack requires only a single malicious attested participant — no threshold collusion. The replayed signature `S1` is publicly observable on-chain after any legitimate `respond_verify_foreign_tx` call. The attacker does not need to be the designated leader for `R2`; `respond_verify_foreign_tx` only checks `assert_caller_is_attested_participant_and_protocol_active()`, not that the caller is the leader for the specific request. Any attested participant can call it for any pending request at any time. [4](#0-3) 

### Recommendation
In `respond_verify_foreign_tx`, after verifying the signature, the contract must also verify that `response.payload_hash` equals the hash the MPC network was supposed to sign for this specific request. Two concrete approaches:

1. **Include `ForeignTxSignPayload` in the response**: Change `VerifyForeignTransactionResponse` to carry the full `ForeignTxSignPayload` (not just the hash). The contract then calls `compute_msg_hash()` on the submitted payload, checks it equals `response.payload_hash`, and also checks that `payload.request == request.request`. This binds the hash to the stored request.

2. **Store the expected hash at request time**: This is not directly possible because `values` are unknown at submission time. However, the contract could at minimum verify that the `ForeignChainRpcRequest` embedded in the submitted payload matches the stored request, requiring nodes to submit the full payload.

The analogous fix in the Curve 3Pool report was to replace the incorrectly calculated parameter with the correct one (`_amount`). Here, the fix is to replace the unchecked `response.payload_hash` with a value derived from the stored `request`.

### Proof of Concept
```
1. Alice submits verify_foreign_transaction(R1) for Bitcoin tx T1.
   → contract stores R1 in pending_verify_foreign_tx_requests.

2. Honest MPC nodes process R1:
   - Inspect Bitcoin, observe block hash B1.
   - Compute H1 = SHA-256(borsh(ForeignTxSignPayloadV1{request: R1, values: [BlockHash(B1)]}))
   - Produce threshold signature S1 over H1.
   - Leader calls respond_verify_foreign_tx(R1, {payload_hash: H1, signature: S1}).
   → R1 resolved; Alice receives (H1, S1). S1 is now public on-chain.

3. Bob submits verify_foreign_transaction(R2) for Bitcoin tx T2 (different tx_id).
   → contract stores R2 in pending_verify_foreign_tx_requests.

4. Malicious attested participant M (single node, below threshold) calls:
   respond_verify_foreign_tx(R2, {payload_hash: H1, signature: S1})
   
   Contract checks:
   - Is M an attested participant? Yes. ✓
   - Is S1 a valid signature over H1 under the root key? Yes. ✓
   - Is H1 == SHA-256(borsh(ForeignTxSignPayloadV1{request: R2, values: ?}))? NOT CHECKED.
   → R2 resolved; Bob receives (H1, S1).

5. Bob's bridge contract receives (H1, S1).
   - Verifies S1 over H1 → valid.
   - Does NOT re-derive expected hash from T2's data.
   → Bridge accepts forged attestation for T2, enabling invalid bridge execution.
``` [5](#0-4) [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
