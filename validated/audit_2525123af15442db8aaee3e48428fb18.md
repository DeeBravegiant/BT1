### Title
Stale Votes from Removed Participants Bypass Unanimity Requirement in `vote_remove_launcher_hash` / `vote_remove_os_measurement` ŌĆö (`File: crates/contract/src/lib.rs`)

### Summary

`vote_remove_launcher_hash` and `vote_remove_os_measurement` compare a raw, unfiltered vote count against the current participant count to enforce unanimity. Because `LauncherHashVotes::count_votes()` and the analogous measurement vote counter tally **all** entries in the map ŌĆö including stale votes from participants removed during resharing ŌĆö the unanimity condition `votes >= total_participants` can be satisfied with fewer than all current participants voting whenever the post-resharing cleanup promise (`clean_tee_status`) has not yet executed or has failed.

### Finding Description

`vote_remove_launcher_hash` reads the vote tally and the participant count as follows:

```rust
let votes = self.tee_state.vote_launcher(action, &participant);
let total_participants = threshold_parameters.participants().len() as u64;
if votes >= total_participants {
    let removed = self.tee_state.remove_launcher_image(&launcher_hash);
``` [1](#0-0) 

`vote_launcher` delegates to `LauncherHashVotes::vote`, which calls `count_votes`:

```rust
fn count_votes(&self, action: &LauncherVoteAction) -> u64 {
    u64::try_from(
        self.vote_by_account.values().filter(|a| *a == action).count(),
    )
    .expect("participant count should not overflow u64")
}
``` [2](#0-1) 

`count_votes` iterates **all** entries in `vote_by_account` without filtering by the current participant set. This is in direct contrast to `ThresholdParametersVotes::n_votes`, which explicitly cross-checks each voter against the live `Participants` set:

```rust
.filter(|&(acc, prop)| {
    participants.participants().iter().any(|(acc_id, _, _)| acc.get() == acc_id)
    && prop == proposal
})
``` [3](#0-2) 

And `TeeVerifierVotes::vote`, which also filters live:

```rust
voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
``` [4](#0-3) 

The stale-vote cleanup is performed by `clean_tee_status`, a `#[private]` endpoint spawned as a promise after resharing:

```rust
pub fn clean_tee_status(&mut self) -> Result<(), Error> {
    ...
    self.tee_state.clean_non_participant_votes(participants);
    Ok(())
}
``` [5](#0-4) 

Because this cleanup is asynchronous and promise-based, there is a window ŌĆö and a failure path ŌĆö during which stale votes from removed participants remain in `vote_by_account`. The codebase's own test documents the exact risk:

> "Stale CodeHashesVotes entries from removed participants must not count toward quorum after resharing." [6](#0-5) 

The same unfiltered counting flaw exists in `vote_remove_os_measurement`:

```rust
let votes = self.tee_state.vote_measurement(action, &participant);
let total_participants = threshold_parameters.participants().len() as u64;
if votes >= total_participants {
    let removed = self.tee_state.remove_measurement(&measurement);
``` [7](#0-6) 

### Impact Explanation

Removing a launcher image hash invalidates TEE attestations for every node running that launcher, forcing those nodes offline. The unanimity requirement exists precisely because this is a destructive, network-wide action. If stale votes from removed participants are counted, a strict minority of current participants (potentially as few as one, if enough stale votes exist) can satisfy `votes >= total_participants` and trigger removal. This breaks the unanimity safety invariant and constitutes unauthorized participant-state manipulation that can disrupt the MPC network's signing capability.

This maps to: **Medium ŌĆö participant-state manipulation that breaks production safety/accounting invariants.**

### Likelihood Explanation

The preconditions are:
1. A resharing occurs that removes one or more participants who had previously voted `Remove` for a launcher hash or OS measurement.
2. The `clean_tee_status` promise fails (e.g., gas exhaustion on a large vote map, a panic in the cleanup path) or is not yet executed.
3. A subset of current participants (fewer than all) then cast the same `Remove` vote.

NEAR promise failures are a known operational reality. The cleanup is not atomic with the resharing state transition, and no on-chain guard prevents the removal check from running before cleanup completes. The `LauncherHashVotes::get_remaining_votes` helper ŌĆö which does filter by current participants ŌĆö exists but is never called in the hot path. [8](#0-7) 

### Recommendation

Replace the raw `count_votes` call in both `vote_remove_launcher_hash` and `vote_remove_os_measurement` with a participant-filtered count, analogous to `ThresholdParametersVotes::n_votes`. Concretely, use the existing `LauncherHashVotes::get_remaining_votes` helper to obtain a filtered tally before comparing against `total_participants`, or add a `count_votes_for_participants` method that accepts the current `Participants` set and filters inline ŌĆö matching the pattern already used by `TeeVerifierVotes::vote` and `ThresholdParametersVotes::n_votes`.

### Proof of Concept

1. Deploy contract with N=5 participants, threshold=3.
2. All 5 participants vote `Remove(hash_H)` via `vote_remove_launcher_hash`. Unanimity not yet reached because the 5th vote triggers removal ŌĆö but suppose only 4 vote before resharing.
3. Resharing removes 3 of the 4 voters; new participant set = {P4, P5_new, P6_new} (N=3).
4. `clean_tee_status` promise fails (simulate by exhausting gas or injecting a panic in test).
5. Stale votes from P1, P2, P3 remain in `vote_by_account` (3 entries for `Remove(hash_H)`).
6. P4 (the one remaining original voter) calls `vote_remove_launcher_hash(hash_H)`.
7. `count_votes` returns 4 (3 stale + 1 fresh). `total_participants` = 3. `4 >= 3` ŌåÆ `remove_launcher_image` executes.
8. `hash_H` is removed with only 1 of 3 current participants voting ŌĆö unanimity bypassed.
