Audit Report

## Title
Unvalidated Arbitrary Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain Approved User Tokens via Callback — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address in all four `addLiquidity*` entry points without verifying it against any factory registry. The constructor takes only a `weth` parameter and stores no `FACTORY` reference, leaving the callback settlement path exploitable: a malicious pool contract can re-enter `metricOmmModifyLiquidityCallback` and cause the adder to pull any ERC-20 tokens the victim has approved — up to the caller-provided max caps — by returning attacker-controlled token addresses from `getImmutables()`.

## Finding Description

The constructor at line 37 accepts only `weth`, with no factory parameter and no `FACTORY` immutable anywhere in the contract: [1](#0-0) 

The contract's own NatSpec at lines 19–21 explicitly documents the missing guard: [2](#0-1) 

The internal `_addLiquidity` flow stores the attacker-supplied pool as the expected callback caller in transient storage, then calls into it: [3](#0-2) 

Inside `metricOmmModifyLiquidityCallback`, the only guards are:
- `kind == KIND_PAY` — attacker controls calldata
- `msg.sender == expectedPool` — malicious pool IS the stored expected pool
- `amount0Delta <= max0 && amount1Delta <= max1` — attacker sets amounts ≤ caps [4](#0-3) 

All three checks pass. The callback then fetches token addresses directly from `msg.sender` (the malicious pool): [5](#0-4) 

The `pay()` helper executes `safeTransferFrom(payer, recipient, value)`, pulling attacker-chosen tokens from the victim's wallet to the malicious pool: [6](#0-5) 

By contrast, the sibling `MetricOmmSwapRouterBase` stores a `FACTORY` immutable and calls `_requireFactoryPool(pool)` before storing any pool in transient context, and additionally validates `FACTORY.isPool(caller)` inside the callback: [7](#0-6) [8](#0-7) 

## Impact Explanation

Any user who has approved `MetricOmmPoolLiquidityAdder` to spend their tokens and is tricked into calling any `addLiquidity*` function with a malicious pool address loses up to `maxAmountToken0` of any ERC-20 and `maxAmountToken1` of any other ERC-20. The malicious pool controls which tokens are pulled by returning arbitrary addresses from `getImmutables()`. This is a direct, unrecoverable loss of user principal — a Critical/High impact under Sherlock thresholds.

## Likelihood Explanation

`MetricOmmPoolLiquidityAdder` is a shared periphery contract requiring pre-approval. A phishing front-end, compromised UI, or social-engineering attack substituting a malicious pool address is a realistic trigger. The attack requires no privileged role, no flash loan, no special token behavior, and no technical sophistication from the victim — pool addresses are opaque 20-byte values most users do not verify on-chain. The exploit is repeatable against any approving user.

## Recommendation

Add a factory reference mirroring `MetricOmmSwapRouterBase` and validate every caller-supplied pool address before storing it in the transient pay context:

```solidity
IMetricOmmPoolFactory internal immutable FACTORY;

constructor(address weth, address factory) PeripheryPayments(weth) {
    FACTORY = IMetricOmmPoolFactory(factory);
}

function _requireFactoryPool(address pool) internal view {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
}
```

Call `_requireFactoryPool(pool)` at the top of `_addLiquidity` and at the top of both `addLiquidityWeighted` overloads before the probe call. Additionally, inside `metricOmmModifyLiquidityCallback`, verify `FACTORY.isPool(msg.sender)` after the `msg.sender == expectedPool` check, mirroring `_requireExpectedCallbackCaller` in `MetricOmmSwapRouterBase`.

## Proof of Concept

1. Victim approves `MetricOmmPoolLiquidityAdder` for USDC and WETH (`type(uint256).max`).
2. Attacker deploys `MaliciousPool(USDC, WETH, adder)` — `getImmutables()` returns `token0=USDC, token1=WETH`; `addLiquidity()` immediately calls back `metricOmmModifyLiquidityCallback(1_000e6, 1_000e18, abi.encode(uint8(1)))`.
3. Victim is tricked into calling:
   ```solidity
   adder.addLiquidityExactShares(
       address(maliciousPool), victim, 0, deltas,
       1_000e6,   // maxAmountToken0
       1_000e18,  // maxAmountToken1
       ""
   );
   ```
4. `_addLiquidity` stores `maliciousPool` as `expectedPool`, calls `maliciousPool.addLiquidity()`.
5. Malicious pool re-enters callback with `KIND_PAY` and amounts ≤ caps — all guards pass.
6. Callback fetches `token0=USDC, token1=WETH` from malicious pool's `getImmutables()`.
7. `pay(USDC, victim, maliciousPool, 1_000e6)` and `pay(WETH, victim, maliciousPool, 1_000e18)` execute via `safeTransferFrom`.
8. Victim loses 1,000 USDC + 1,000 WETH; malicious pool receives them.
