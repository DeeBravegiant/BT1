### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Binding It to the Pending Request, Enabling Cross-Request Response Injection — (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_verify_foreign_tx` function verifies that the ECDSA signature in the response is valid over `response.payload_hash`, but never verifies that `response.payload_hash` was actually derived from the pending `request`. Because `payload_hash` is entirely caller-supplied and unchecked against the request key, a single malicious attested participant can replay a valid response produced for one foreign-chain request to resolve a completely different pending request, delivering a forged verification to the original caller.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs two checks before resolving the yield:

1. The ECDSA signature is valid over `response.payload_hash` (using the root MPC public key).
2. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What is **absent** is any check that `response.payload_hash` was computed from the specific pending `request`. By design, the signed payload is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request, values })))
``` [2](#0-1) 

The `request` field is embedded inside the preimage, so every valid `payload_hash` is cryptographically bound to a specific `request`. However, the contract never reconstructs or checks this binding — it only verifies the signature over the hash value the caller provides.

Contrast this with `respond` for regular signatures, where the payload is taken directly from `request.payload` and is therefore structurally impossible to substitute: [3](#0-2) 

The SDK-level verifier (`ForeignChainSignatureVerifier::verify_signature`) does enforce the binding by recomputing `expected_payload_hash` from the request and the expected extracted values and comparing it to `response.payload_hash`: [4](#0-3) 

But this check is entirely client-side. The on-chain contract imposes no such constraint, so any caller that trusts the contract to deliver a correct response — without re-running the SDK verification — is exposed.

---

### Impact Explanation

A malicious attested participant (strictly below the signing threshold) can:

1. Observe any previously submitted, on-chain `respond_verify_foreign_tx` call that produced a valid `(payload_hash_B, signature_B)` for `request_B` (responses are public on-chain).
2. Call `respond_verify_foreign_tx(request_A, { payload_hash: payload_hash_B, signature: signature_B })` while `request_A` is pending.
3. The contract accepts the call: `signature_B` is valid over `payload_hash_B`, and `request_A` exists in the pending map.
4. `request_A` is resolved and removed from the queue; the caller of `request_A` receives `{ payload_hash: payload_hash_B, signature: signature_B }`.

`payload_hash_B` encodes the extracted values from `request_B` (e.g., a different Bitcoin tx-id, a different block hash, or a different confirmation count). A bridge contract that does not re-run `ForeignChainSignatureVerifier::verify_signature` will accept this as a valid attestation of `request_A`'s foreign-chain state, enabling **forged foreign-chain verification** and potentially **invalid bridge execution or double-spend conditions** — matching the High impact tier.

Additionally, even when the caller does validate, the pending slot for `request_A` is permanently consumed, forcing a resubmission and deposit.

---

### Likelihood Explanation

Medium-High. The attacker only needs to be an attested participant — a role that any node operator can hold. No threshold collusion is required. Valid `(payload_hash, signature)` pairs for prior requests are permanently visible on-chain, so the attacker does not need to be the signing leader; they can replay any historical response. The only constraint is that a matching pending request must exist at the time of the call, which is trivially satisfiable in a live bridge environment with concurrent requests.

---

### Recommendation

The contract should enforce that `response.payload_hash` is bound to the pending `request`. Two concrete approaches:

1. **Include extracted values in the response and recompute on-chain.** Have the responding node supply the `Vec<ExtractedValue>` alongside the hash. The contract recomputes `SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request, values })))` and asserts it equals `response.payload_hash` before accepting the response. This mirrors how `respond` derives the payload from the request itself.

2. **Bind the request hash into the verification.** Compute `expected_request_hash = SHA-256(borsh(request))` on-chain and verify that the first 32 bytes (or a tagged prefix) of the borsh-encoded preimage match, using a deterministic layout guarantee.

Until one of these is implemented, the contract documentation and the SDK must make it explicit that callers **must** call `ForeignChainSignatureVerifier::verify_signature` and must not treat a successful `respond_verify_foreign_tx` call alone as proof of correct foreign-chain state.

### Proof of Concept

```
Setup:
  - request_A = VerifyForeignTransactionRequest { tx_id: 0xAA..., extractors: [BlockHash], ... }
  - request_B = VerifyForeignTransactionRequest { tx_id: 0xBB..., extractors: [BlockHash], ... }

Step 1: Alice submits verify_foreign_transaction(request_A) with 1 yoctoNEAR deposit.
        Contract stores request_A in pending_verify_foreign_tx_requests.

Step 2: Separately, the MPC network legitimately processes request_B and an honest node
        calls respond_verify_foreign_tx(request_B, { payload_hash: H_B, signature: S_B }).
        H_B = SHA-256(borsh(ForeignTxSignPayload::V1({ request: request_B, values: [BlockHash(0xCC...)] })))
        S_B = valid ECDSA signature over H_B under the MPC root key.
        This call is visible on-chain.

Step 3: Mallory (attested participant) calls:
        respond_verify_foreign_tx(
            request_A,                                    // pending request
            { payload_hash: H_B, signature: S_B }        // response from request_B
        )

Step 4: Contract checks:
        - assert_caller_is_attested_participant_and_protocol_active() → PASS (Mallory is attested)
        - verify_ecdsa_signature(S_B, H_B, root_pk) → PASS (S_B is a valid MPC signature)
        - pending_verify_foreign_tx_requests.get(request_A) → PASS (Alice's request exists)
        → resolve_yields_for resolves Alice's yield with { payload_hash: H_B, signature: S_B }

Step 5: Alice's bridge contract receives { payload_hash: H_B, signature: S_B }.
        H_B encodes BlockHash(0xCC...) for tx 0xBB, NOT Alice's tx 0xAA.
        If Alice's contract skips ForeignChainSignatureVerifier::verify_signature,
        it accepts this as proof that tx 0xAA was confirmed with block hash 0xCC,
        and releases bridge funds for a transaction that was never verified.
``` [5](#0-4) [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
