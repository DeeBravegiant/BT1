Audit Report

## Title
`CrossChainRateReceiver.getRate()` Returns Stale Rate Enabling wrsETH Overminting at Depositors' Expense - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver` stores both `rate` and `lastUpdated` but `getRate()` returns `rate` unconditionally without consulting `lastUpdated`. All L2 pool contracts use this oracle to compute wrsETH minting amounts. When the stored rate lags the true rsETH/ETH rate, any depositor receives more wrsETH than their ETH is worth at the current rate, diluting accrued yield belonging to existing holders.

## Finding Description
`CrossChainRateReceiver` maintains two state variables: `rate` (the rsETH/ETH exchange rate) and `lastUpdated` (the timestamp of the last LayerZero update). [1](#0-0) 

`lzReceive()` updates both on every cross-chain message delivery: [2](#0-1) 

However, `getRate()` returns `rate` with no staleness check against `lastUpdated`: [3](#0-2) 

`RSETHRateReceiver` is the concrete L2 deployment and is set as `rsETHOracle` in pool contracts: [4](#0-3) 

`RSETHPoolV3.getRate()` delegates directly to this oracle: [5](#0-4) 

The minting formula in `viewSwapRsETHAmountAndFee` divides by this rate: [6](#0-5) 

The public `deposit()` function mints wrsETH to any caller using this formula: [7](#0-6) 

Because rsETH/ETH is monotonically increasing (staking rewards accrue), a stale `rate` is always lower than the true current rate. A lower denominator in `amountAfterFee * 1e18 / rsETHToETHrate` produces a larger `rsETHAmount`. The excess wrsETH minted claims a larger pro-rata share of the pool's rsETH than the deposited ETH justifies, extracting yield from prior holders.

## Impact Explanation
**High â€” Theft of unclaimed yield.**

Each wrsETH is redeemable for a pro-rata share of the rsETH held by the wrapper. When new wrsETH is minted at a stale (lower) rate, the new tokens claim a disproportionate share of the pool. The difference between the overminted amount and the fair amount represents accrued staking yield that rightfully belongs to existing holders. The magnitude scales with both the staleness duration and deposit size; neither is bounded by the protocol.

## Likelihood Explanation
**Medium.**

The rate is updated only when an off-chain keeper calls `updateRate()` on the provider and the LayerZero message is delivered on L2. There is no on-chain enforcement of a minimum update frequency. During periods of high L1 gas costs, keeper inactivity, or LayerZero congestion, the stored rate can lag by hours or days. A sophisticated actor can monitor the divergence between the stale L2 rate and the live mainnet `rsETHPrice` and time deposits to maximize the overmint. No special privilege is required â€” only the public `deposit()` function.

## Recommendation
Add a configurable `maxRateAge` and revert in `getRate()` if the stored rate is too old:

```solidity
uint256 public maxRateAge = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxRateAge, "Rate is stale");
    return rate;
}
```

This ensures deposits revert rather than proceed at a stale rate, preventing overminting while giving the operator a configurable tolerance window.

## Proof of Concept
1. On mainnet, `rsETHPrice` is updated to `1.10e18` (reflecting recent staking rewards).
2. The keeper has not called `updateRate()`; the L2 `RSETHRateReceiver.rate` still holds `1.05e18` from 48 hours ago.
3. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("")` on L2.
4. `viewSwapRsETHAmountAndFee(100 ether)` computes `rsETHAmount = 100e18 * 1e18 / 1.05e18 â‰ˆ 95.24 wrsETH`.
5. At the true rate the deposit is worth `100e18 / 1.10e18 â‰ˆ 90.91 wrsETH`.
6. Attacker receives `â‰ˆ 4.33 wrsETH` in excess â€” yield extracted from existing holders â€” with no special privilege required.

**Foundry fork test plan:** Fork the target L2, deploy `RSETHRateReceiver` with a stale rate, set it as `rsETHOracle` in `RSETHPoolV3`, call `deposit{value: 100 ether}("")`, assert `wrsETH.balanceOf(attacker) > 100e18 * 1e18 / trueRate`, and assert the difference equals the yield diluted from prior holders.
