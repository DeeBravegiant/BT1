### Title
Missing Cross-Check Between `blob.batch_count` and Inner `ec_pve_ac_t::Q.size()` in `combine_ac` Enables Silent Truncation of Recovered Key Shares - (`src/cbmpc/api/pve_batch_ac.cpp`)

---

### Summary

`combine_ac` parses the outer `pve_ac_ciphertext_blob_v1_t` and validates `blob.batch_count`, but never asserts that the deserialized inner `ec_pve_ac_t`'s `Q.size()` equals `blob.batch_count`. Because `aggregate_to_restore_row` is called with `skip_verify=true`, the ZK proof is bypassed entirely, and `batch_size` is derived solely from the inner `Q.size()`. An attacker who supplies a crafted ciphertext with `batch_count=N` but an inner blob encoding only `M < N` Q-points causes `combine_ac` to return `out_xs` with M elements while the caller expects N, producing silent truncation of recovered key shares.

---

### Finding Description

**Outer blob structure and its validation:**

`pve_ac_ciphertext_blob_v1_t` holds two independent fields: `batch_count` (a `uint32_t`) and `ct` (the serialized `ec_pve_ac_t`). [1](#0-0) 

`parse_ac_ciphertext` validates only that `batch_count > 0` and `<= MAX_BATCH_COUNT`. It does not inspect `ct` at all. [2](#0-1) 

**Deserialization of the inner object:**

`ec_pve_ac_t::convert` deserializes `Q` as a plain `std::vector<ecc_point_t>` whose length is determined entirely by the serialized bytes — there is no external size constraint applied. [3](#0-2) 

**The missing guard in `combine_ac`:**

After calling `parse_ac_ciphertext` (which validates `blob.batch_count`) and `coinbase::convert(pve_ct, blob.ct)` (which populates `pve_ct.Q` from the inner bytes), there is **no check** that `pve_ct.get_Q().size() == blob.batch_count`. The code proceeds directly to `aggregate_to_restore_row` with `skip_verify=true`. [4](#0-3) 

**`aggregate_to_restore_row` derives `batch_size` from the inner `Q.size()`:**

With `skip_verify=true` the ZK proof is skipped entirely. `batch_size` is set to `int(Q.size())` — the inner blob's count, not the outer `batch_count`. The output vector `x` is resized to this inner count and returned. [5](#0-4) 

The only remaining check is the per-element EC consistency test `x[j] * G != Q[j]`, which the attacker satisfies trivially by choosing Q points for which they know the discrete logs. [6](#0-5) 

**`combine_ac` propagates the truncated output without any size assertion:**

`out_xs` is set to `xs_bn.size()` elements — M, not N — and `SUCCESS` is returned. [7](#0-6) 

**Why `verify_ac` does not save callers who skip it:**

`verify_ac` does check `Qs_compressed.size() != blob.batch_count` and internally `pve_ct.verify` checks `Q.size() != this->Q.size()`, which would catch the mismatch. However, the API header explicitly documents that `combine_ac` intentionally skips verification and that callers may omit `verify_ac`. There is no mandatory guard inside `combine_ac` itself. [8](#0-7) 

---

### Impact Explanation

A caller that uses `get_ac_batch_count` to learn N=5, then calls `combine_ac` and receives `out_xs` with M=3 elements, faces two concrete failure modes:

1. **Silent truncation of key shares.** If the caller distributes `out_xs[i]` as key shares to N parties, the last N−M parties receive no share. The caller has no indication of failure because `combine_ac` returned `SUCCESS`.

2. **Out-of-bounds vector access (undefined behavior).** If the caller indexes `out_xs` using the `batch_count`-derived bound (e.g., `for (int i = 0; i < batch_count; i++) use(out_xs[i])`), accessing `out_xs[M]` through `out_xs[N-1]` is undefined behavior in C++, which on common implementations causes a heap read past the vector's allocation.

The attacker fully controls which M scalars are returned: they craft the inner `ec_pve_ac_t` with chosen Q points and matching encrypted values; with `skip_verify=true` the ZK proof is never checked, so the only constraint is the per-element `x[j]*G == Q[j]` test, which the attacker satisfies by construction.

---

### Likelihood Explanation

The attack requires only the ability to supply a crafted ciphertext blob to `combine_ac` — a standard API input. No threshold collusion, no host compromise, and no side-channel is needed. Any caller that omits the optional `verify_ac` step (which the API explicitly permits) is exposed. The crafted blob is straightforward to construct: serialize a valid `ec_pve_ac_t` with M < N Q-points and matching rows, then wrap it in an outer blob with `batch_count = N`.

---

### Recommendation

Add an explicit cross-check in `combine_ac` immediately after deserializing the inner object:

```cpp
if (pve_ct.get_Q().size() != static_cast<size_t>(blob.batch_count))
    return coinbase::error(E_FORMAT, "batch_count mismatch between outer blob and inner ciphertext");
```

The same guard should be added in `partial_decrypt_ac_attempt` and `get_public_keys_compressed_ac` for defense-in-depth, since those functions also deserialize the inner blob without cross-checking `blob.batch_count`. [9](#0-8) [10](#0-9)

### Proof of Concept

```
1. Encrypt a legitimate batch of 3 scalars {x0, x1, x2} using encrypt_ac with batch_count=3.
   This produces a valid inner ec_pve_ac_t blob with Q.size()=3.

2. Craft a new outer pve_ac_ciphertext_blob_v1_t:
     blob.version     = 1
     blob.batch_count = 5          // outer claims 5 scalars
     blob.ct          = <the valid inner ec_pve_ac_t from step 1>  // inner has Q.size()=3

3. Serialize this crafted blob to bytes (crafted_ct).

4. Call partial_decrypt_ac_attempt for each leaf to obtain quorum shares
   (these decrypt correctly because the inner blob is cryptographically valid).

5. Call combine_ac(curve, ac, crafted_ct, attempt_index, label, quorum_shares, out_xs).
   -> Returns SUCCESS.
   -> out_xs.size() == 3, not 5.

6. Call get_ac_batch_count(crafted_ct, count).
   -> count == 5.

7. Assert: out_xs.size() != count  // truncation confirmed, SUCCESS returned
   OR: access out_xs[3] -> undefined behavior (out-of-bounds on a size-3 vector).
```

static error_t parse_ac_ciphertext(mem_t ciphertext, pve_ac_ciphertext_blob_v1_t& out_blob) {
