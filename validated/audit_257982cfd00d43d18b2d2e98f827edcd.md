### Title
`deployToken` Metadata Signature Lacks Chain-ID Binding, Enabling Cross-Chain Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `aptos/sources/bridge_types.move`)

---

### Summary

The NEAR MPC signature verified in `deployToken` / `deploy_token` is computed over a Borsh-encoded `MetadataPayload` that contains no destination chain identifier. Because the same `nearBridgeDerivedAddress` is used on every EVM chain, Starknet, and Aptos, a single metadata signature obtained for one chain is cryptographically valid on every other chain where the bridge is deployed. An unprivileged attacker can replay the signature to deploy a wrapped token on any chain without NEAR's per-chain authorization.

---

### Finding Description

`finTransfer` / `fin_transfer` correctly binds its signed hash to the destination chain by interleaving `omniBridgeChainId` twice in the Borsh encoding: [1](#0-0) 

```solidity
bytes1(omniBridgeChainId),          // before token_address
...
bytes1(omniBridgeChainId),          // before recipient
```

`deployToken` performs no equivalent binding. Its entire signed message is: [2](#0-1) 

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

No `omniBridgeChainId` field appears anywhere in the hash. The identical omission exists in Starknet: [3](#0-2) 

```cairo
fn to_borsh(self: @MetadataPayload) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append(@borsh::encode_byte_array(self.token));
    borsh_bytes.append(@borsh::encode_byte_array(self.name));
    borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
    borsh_bytes.append_byte(*self.decimals);
    borsh_bytes
}
```

Called without a chain argument at `starknet/src/omni_bridge.cairo` line 205: [4](#0-3) 

```cairo
_verify_borsh_signature(ref self, @payload.to_borsh(), signature);
```

Whereas `fin_transfer` passes the chain ID: [5](#0-4) 

```cairo
_verify_borsh_signature(
    ref self, @payload.to_borsh(self.omni_bridge_chain_id.read()), signature,
);
```

The same pattern is present in Aptos — `metadata_to_borsh` encodes no chain ID, while `transfer_message_to_borsh` takes and embeds `chain_id`: [6](#0-5) 

The `nearBridgeDerivedAddress` / `omni_bridge_derived_address` is a single NEAR MPC-derived key shared across all deployed chains: [7](#0-6) 

This means the hash produced for a `MetadataPayload` on Ethereum (`omniBridgeChainId = 0`) is byte-for-byte identical to the hash on Arbitrum (`omniBridgeChainId = 3`), Base (`omniBridgeChainId = 4`), BNB (`omniBridgeChainId = 5`), Polygon (`omniBridgeChainId = 8`), etc. [8](#0-7) 

---

### Impact Explanation

**High — cross-domain signature replay that bypasses the deployment execution gate.**

1. **Unauthorized token deployment on any chain.** A single NEAR-signed metadata payload authorizes `deployToken` on every chain simultaneously. An attacker who observes a valid signature on chain A can call `deployToken` on chains B, C, … without NEAR ever authorizing those deployments.

2. **Permanent blocking of legitimate deployment on EVM chains.** On EVM, the wrapped-token proxy address is determined by the OmniBridge contract's nonce at the moment of deployment (`new ERC1967Proxy(...)`). If the attacker deploys first on chain B, the proxy lands at a nonce-dependent address that differs from the address NEAR would have chosen. The guard `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` then permanently prevents NEAR from deploying the canonical token on that chain. [9](#0-8) 

3. **Unbacked supply risk.** NEAR's off-chain relayer observes `DeployToken` events to learn the canonical token address per chain. If it accepts the attacker-deployed token as canonical, it will sign `finTransfer` messages minting into that token, creating supply backed only by the attacker's replay rather than a legitimate NEAR-side deposit.

---

### Likelihood Explanation

**Medium.** The attack requires only:
- Monitoring any chain for a valid `deployToken` transaction (or the NEAR MPC signing event).
- Replaying the same `(signatureData, metadata)` calldata on any other chain where the bridge is deployed.

No privileged access, leaked keys, or chain-level attack is needed. The bridge is deployed on at least six mainnet EVM chains simultaneously, making cross-chain replay straightforward.

---

### Recommendation

Include `omniBridgeChainId` in the `MetadataPayload` Borsh encoding used for signature verification, mirroring the pattern already used in `finTransfer`. On the EVM side:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent change to `MetadataPayloadImpl::to_borsh` in `starknet/src/bridge_types.cairo` (accepting `chain_id: u8`) and `metadata_to_borsh` in `aptos/sources/bridge_types.move`. Update the NEAR MPC signing logic to include the destination chain ID when producing metadata signatures.

---

### Proof of Concept

1. NEAR signs a `MetadataPayload` for `wrap.near` in response to a `logMetadata` call on Ethereum (`omniBridgeChainId = 0`). The signed hash is `keccak256(0x01 || borsh("wrap.near") || borsh("Wrapped NEAR") || borsh("wNEAR") || 0x18)`.

2. Attacker captures `(signatureData, metadata)` from the Ethereum mempool or transaction history.

3. Attacker calls `OmniBridge.deployToken(signatureData, metadata)` on Arbitrum (`omniBridgeChainId = 3`). The contract computes the identical hash (chain ID is absent), `ECDSA.recover` returns `nearBridgeDerivedAddress`, and the check passes. [10](#0-9) 

4. A `BridgeToken` proxy for `wrap.near` is deployed on Arbitrum at an attacker-controlled nonce-dependent address and registered in `nearToEthToken["wrap.near"]`.

5. When NEAR later attempts to legitimately deploy `wrap.near` on Arbitrum, `require(!isBridgeToken[nearToEthToken["wrap.near"]], "ERR_TOKEN_EXIST")` reverts, permanently blocking the canonical deployment. [9](#0-8)
