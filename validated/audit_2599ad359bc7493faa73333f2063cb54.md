### Title
Missing CKD Output Validation for `AppPublicKey` Variant Allows Byzantine Participant to Forge Confidential Key Derivation Response - (File: crates/contract/src/lib.rs)

### Summary

The `respond_ckd` contract method performs cryptographic validation of the CKD output **only** for the `AppPublicKeyPV` (publicly verifiable) variant. For the `AppPublicKey` (legacy) variant, the contract accepts any `big_y` and `big_c` values from any attested participant without any pairing-based or algebraic check. A single Byzantine attested participant below the signing threshold can race to submit a forged `CKDResponse`, either delivering a wrong secret to the user or — if the Byzantine node is the coordinator — exposing the plaintext BLS signature `msk·H(app_id)` on-chain by setting `big_y = 0`.

---

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`), the match on `request.app_public_key` has an empty arm for `AppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no validation whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` enforces the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk‖app_id), msk_pk)`, which cryptographically binds the response to the MPC public key and the user's ephemeral key pair. [2](#0-1) 

No equivalent check exists for the `AppPublicKey` variant. The contract simply calls `resolve_yields_for` with whatever bytes the caller supplied. [3](#0-2) 

The `AppPublicKey` variant is still documented and supported as the "legacy" path. [4](#0-3) 

---

### Impact Explanation

**Attack A — Forged secret (any Byzantine attested participant):**  
Any single attested participant can call `respond_ckd` for a pending `AppPublicKey` CKD request and supply arbitrary `big_y` / `big_c` bytes. The contract resolves the yield and returns the forged blob to the user. The user's TEE application decrypts `s = big_c − a·big_y` and obtains a wrong, attacker-controlled secret. If the application does not independently verify the BLS signature (which the contract does not enforce), it proceeds with a compromised key.

**Attack B — Plaintext secret exposure on-chain (Byzantine coordinator):**  
The CKD coordinator aggregates partial BLS signatures from all participants and therefore learns `msk·H(pk‖app_id)` — the user's actual secret — before encrypting it. If the coordinator is Byzantine, it can submit `big_y = 0` (the G1 identity) and `big_c = msk·H(pk‖app_id)`. The user still decrypts the correct secret (`s = big_c − a·0 = big_c`), but the raw BLS signature is now stored in plaintext in the on-chain response, visible to every blockchain observer. This breaks the fundamental confidentiality guarantee of CKD: that the secret is never revealed to any party other than the requesting application. [5](#0-4) 

The CKD design explicitly states that the secret must be "only known by the app" and that "no single node in the MPC network should be capable of computing s." [6](#0-5) 

---

### Likelihood Explanation

- The `AppPublicKey` variant is the legacy default path and is actively used.
- Any single attested MPC participant (well below the signing threshold) can call `respond_ckd`; the contract does not restrict the call to the designated coordinator.
- The coordinator role is rotated per-request, so a Byzantine node will eventually be coordinator.
- The attack requires no special tooling: the attacker simply calls `respond_ckd` with crafted JSON arguments.
- The race window (between the honest coordinator computing the response and submitting it) is realistic on a live network.

---

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. Because the `AppPublicKey` variant does not supply a G2 component, a weaker but still binding check is possible: verify that `e(big_c − big_y·a, G2) = e(H(pk‖app_id), msk_pk)` using the stored MPC public key. Alternatively, deprecate the `AppPublicKey` variant and require all new requests to use `AppPublicKeyPV`, which already has on-chain verification.

**Short term:** Add a pairing check for `AppPublicKey` responses analogous to `ckd_output_check`, using the stored BLS public key and the hash point derived from `app_id`.

**Long term:** Validate all cryptographically sensitive MPC outputs before resolving yields, regardless of which protocol variant is used.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey` variant and ephemeral key `A = a·G1`.
2. Byzantine attested participant calls `respond_ckd` before the honest coordinator, supplying:
   - `big_y = 0` (48-byte compressed identity point in G1)
   - `big_c = msk·H(pk‖app_id)` (the raw BLS signature, which the coordinator already computed)
3. Contract passes the empty `AppPublicKey` arm with no validation and calls `resolve_yields_for`.
4. The yield resolves; the on-chain receipt contains `big_c = msk·H(pk‖app_id)` in plaintext.
5. Any observer reads `big_c` from the receipt and obtains the user's confidential key without knowing `a`.
6. The user's TEE application, decrypting `s = big_c − a·0 = big_c`, receives the correct value but the secret has already been leaked. [7](#0-6) [2](#0-1)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
