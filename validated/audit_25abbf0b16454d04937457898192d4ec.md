### Title
Attached NEAR Deposit Silently Lost on `rlp_execute` Early Return When `has_in_flight_tx` Is True - (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary

The `rlp_execute` entry point of the Wallet Contract is marked `#[payable]` and accepts an attached NEAR deposit from any external caller. When `has_in_flight_tx` is `true`, the function returns early with an error `ExecuteResponse` value — without panicking and without refunding the caller's deposit. Because the NEAR runtime only auto-refunds deposits on receipt failure (panic), a non-zero deposit attached to this rejected call is permanently credited to the wallet contract and never returned to the caller.

### Finding Description

`rlp_execute` is the sole public entry point of the Wallet Contract deployed to every ETH-implicit account on NEAR. It is decorated `#[payable]`, meaning callers may attach NEAR tokens to fund cross-contract calls (e.g., a `FunctionCall` action that requires a deposit). The `CallerDeposit` mechanism was specifically designed to track and refund such deposits when the downstream cross-contract call fails.

However, there is an early-return guard at the very top of `rlp_execute`:

```rust
#[payable]
pub fn rlp_execute(
    &mut self,
    target: AccountId,
    tx_bytes_b64: String,
) -> PromiseOrValue<ExecuteResponse> {
    if self.has_in_flight_tx {
        return PromiseOrValue::Value(ExecuteResponse {   // ← early return
            success: false,
            success_value: None,
            error: Some(
                "Error: transaction already in progress, please try again later.".into(),
            ),
        });
    }
    // CallerDeposit tracking only happens below, inside inner_rlp_execute
    ...
```

When this branch is taken, execution never reaches `inner_rlp_execute` (line 108), so `CallerDeposit::new` (line 345) is never called, and no refund promise is ever created. The function returns a successful `PromiseOrValue::Value(...)` — it does not panic — so the NEAR runtime does not trigger its automatic deposit-refund mechanism. The attached deposit is silently absorbed into the wallet contract's balance.

The same silent-loss applies to the `Err(e) => PromiseOrValue::Value(e.into())` arm at line 126, which handles user and account-ID errors returned from `inner_rlp_execute` after the `CallerDeposit` has already been constructed but before any refund promise is scheduled.

### Impact Explanation

Any external caller who attaches a non-zero NEAR deposit to `rlp_execute` while `has_in_flight_tx == true` permanently loses that deposit. The deposit is credited to the ETH-implicit wallet account and is irrecoverable by the caller. The corrupted protocol value is the caller's NEAR account balance (a concrete `u128` balance entry in the NEAR state trie). The wallet contract's balance is correspondingly inflated by the same amount.

The `test_caller_refunds` test confirms that attaching a deposit to `rlp_execute` is an intended and documented use case (external callers fund cross-contract calls this way). The `has_in_flight_tx` condition is a normal operational state, not an exceptional one, making this reachable in ordinary usage.

### Likelihood Explanation

The `has_in_flight_tx` flag is `true` for the entire duration of any in-flight transaction — from the moment `rlp_execute` schedules a promise until the final callback resolves. During this window, any caller who retries with a deposit (e.g., a relayer retrying after a timeout, or a user submitting a second call) will lose their deposit. The wallet contract is deployed globally to all ETH-implicit accounts, so the attack surface is broad. No special privileges are required; any NEAR account can call `rlp_execute` with an attached deposit.

### Recommendation

Before the `has_in_flight_tx` early-return, check whether `env::attached_deposit()` is non-zero and, if so, either:
1. Panic (which causes the NEAR runtime to automatically refund the deposit), or
2. Explicitly schedule a transfer promise back to `env::predecessor_account_id()` before returning the error `Value`.

The same fix should be applied to the `Err(e) => PromiseOrValue::Value(e.into())` arm at line 126 when a `CallerDeposit` was already constructed.

### Proof of Concept

1. Deploy the Wallet Contract to an ETH-implicit account (standard production setup).
2. Submit a valid `rlp_execute` call that schedules a cross-contract promise, setting `has_in_flight_tx = true`.
3. Before the promise resolves, submit a second `rlp_execute` call from any external account with `attached_deposit = 5 NEAR`.
4. The second call returns `ExecuteResponse { success: false, error: "transaction already in progress" }`.
5. Observe that the caller's balance decreased by 5 NEAR and the wallet contract's balance increased by 5 NEAR, with no refund receipt generated.

/// A data type to keep track of the deposit given by an external caller.
/// This allows us to refund the caller's deposit if the cross-contract call fails.
#[derive(Debug, PartialEq, Eq, Clone, serde::Serialize, serde::Deserialize)]
pub struct CallerDeposit {
    pub account_id: AccountId,
    pub yocto_near: NonZeroU128,
}

impl CallerDeposit {
    pub fn new(context: &ExecutionContext) -> Option<Self> {
        // Only track for external (non-self) callers
        if context.current_account_id == context.predecessor_account_id {
            return None;
        }

        NonZeroU128::new(context.attached_deposit.as_yoctonear()).map(|yocto_near| Self {
            account_id: context.predecessor_account_id.clone(),
            yocto_near,
        })
    }
}
