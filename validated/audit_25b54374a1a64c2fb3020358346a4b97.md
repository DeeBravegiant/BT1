Audit Report

## Title
Missing Staleness Validation in `getAssetPrice()` Enables Over-Minting of rsETH at Stale Prices - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all staleness metadata (`updatedAt`, `answeredInRound`). If a supported LST's Chainlink feed goes stale while the asset's market price has dropped, any depositor can mint rsETH calculated against the inflated stale price, diluting all existing rsETH holders. The same protocol already applies the missing checks in its L2 oracle wrapper, confirming this is an unintentional omission.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at line 52 destructures only the `price` field from `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol:52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`roundId`, `startedAt`, `updatedAt`, and `answeredInRound` are all discarded. No check is made that `updatedAt != 0`, `answeredInRound >= roundId`, or that `block.timestamp - updatedAt` is within an acceptable heartbeat window.

This contrasts directly with `ChainlinkOracleForRSETHPoolCollateral.getRate()` at lines 30–32, which performs all three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The stale price propagates through the following confirmed call chain:

1. `LRTOracle.getAssetPrice(asset)` → `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)` (line 157)
2. `LRTDepositPool.getRsETHAmountToMint(asset, amount)` → `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` (line 520)
3. `LRTDepositPool.depositAsset(...)` → `_beforeDeposit` → `getRsETHAmountToMint` → `_mintRsETH` (lines 111–115)

`lrtOracle.rsETHPrice()` is a stored value updated separately; only the asset price numerator uses a live Chainlink call with no staleness guard. No existing check in `_beforeDeposit` (lines 648–670) or `getRsETHAmountToMint` validates oracle freshness.

## Impact Explanation
When a supported LST (stETH, rETH, sfrxETH) depegs while its Chainlink feed is stale at the pre-depeg price, a depositor receives rsETH computed against the inflated stale price. The over-minted rsETH represents a direct dilution of all existing rsETH holders' pro-rata claims on the underlying ETH pool — this is theft of unclaimed yield from existing holders. At sufficient scale (e.g., a 10% depeg on a large deposit), the protocol's backing ratio falls below 1:1, constituting insolvency.

**Impact: High — Theft of unclaimed yield from existing rsETH holders.**

## Likelihood Explanation
Chainlink feeds go stale during L1 network congestion preventing keeper transactions, feed deprecation/migration, or heartbeat misses during low-volatility periods. The protocol supports multiple LST assets, each with its own feed, multiplying the attack surface. An attacker can monitor on-chain `updatedAt` timestamps permissionlessly and time a large `depositAsset` call precisely when a feed is stale. No privileged access is required; the path is fully open via the public `depositAsset` function.

**Likelihood: Medium.**

## Recommendation
Add staleness and round-completeness checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optional: if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept
**Setup:** stETH/ETH Chainlink feed last updated 4 hours ago at `1.0e18`. stETH has since depegged to `0.9e18` on-chain. Current stored `rsETHPrice = 1.05e18`.

**Attack sequence:**
1. Attacker observes `updatedAt` is stale on the stETH Chainlink feed.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` computes: `1000e18 * 1.0e18 / 1.05e18 = 952.38e18 rsETH` (using stale price).
4. Actual ETH value deposited: `1000 * 0.9 = 900 ETH`.
5. Fair rsETH for 900 ETH at 1.05 price: `900 / 1.05 = 857.14 rsETH`.
6. Attacker receives `952.38 rsETH` instead of `857.14 rsETH` — **~95 rsETH over-minted (~$190k at $2000/ETH)** — extracted from existing holders.

**Foundry fork test plan:**
- Fork mainnet at a block where a supported LST feed's `updatedAt` is beyond the heartbeat window.
- Deploy/use the existing `ChainlinkPriceOracle` pointing to the stale feed.
- Call `depositAsset` with a large LST amount.
- Assert that `rsethAmountToMint` exceeds the fair value computed using the current spot price.
- Confirm the test passes without revert, demonstrating the missing guard.
