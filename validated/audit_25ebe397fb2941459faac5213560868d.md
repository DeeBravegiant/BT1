### Title
Per-Block Reference Script Size Limit Not Enforced for Dijkstra Subtransactions — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the per-block reference script size limit (`ppMaxRefScriptSizePerBlock`) is not enforced for reference scripts contained in subtransactions. The `totalRefScriptSizeInBlock` function used in the BBODY rule only calls `txNonDistinctRefScriptsSize` on each top-level transaction, which does not traverse subtransactions. Meanwhile, the per-transaction check in the Dijkstra LEDGER rule correctly uses `batchNonDistinctRefScriptsSize`, which sums reference script sizes across the top-level transaction and all its subtransactions. This asymmetry allows an attacker to craft a block whose actual total reference script processing far exceeds `ppMaxRefScriptSizePerBlock`, bypassing the intended per-block resource cap.

---

### Finding Description

**Root cause — `totalRefScriptSizeInBlock` does not traverse subtransactions:**

`totalRefScriptSizeInBlock` iterates over the top-level transactions in a block and calls `txNonDistinctRefScriptsSize` on each one: [1](#0-0) 

`txNonDistinctRefScriptsSize` is a Conway-era function that only examines the inputs and reference inputs of a single transaction body. It has no knowledge of Dijkstra subtransactions. The Dijkstra-era `batchNonDistinctRefScriptsSize` — which correctly sums ref script sizes across the top-level tx and all its subtransactions — is never called from the BBODY rule: [2](#0-1) 

**Per-transaction check is correct; per-block check is not:**

The Dijkstra LEDGER rule's `validateAllRefScriptSize` correctly uses `batchNonDistinctRefScriptsSize`, bounding the entire batch (top-level + all subtransactions) to `ppMaxRefScriptSizePerTx`: [3](#0-2) 

But `validateBodyRefScriptsSizeTooBig` in the BBODY rule — which enforces the per-block cap — calls `totalRefScriptSizeInBlock`, which only sees the top-level tx's ref scripts: [4](#0-3) 

**Unbounded subtransaction iteration in SUBLEDGERS:**

`dijkstraSubLedgersTransition` uses `foldM` to process every subtransaction through a full `SUBLEDGER` rule transition (UTxO validation, certificate processing, governance) with no explicit count cap beyond the implicit transaction-size limit: [5](#0-4) 

Each subtransaction can carry its own reference inputs, certificates, and governance actions, all of which are processed in full.

---

### Impact Explanation

The per-block reference script size limit was introduced specifically to cap the total validation work a block can impose on honest nodes (see ADR 009). By placing all reference script bytes in subtransactions rather than the top-level transaction body, an attacker can submit a block where:

- Each top-level transaction has **zero** ref scripts in its own body → contributes **0** to `totalRefScriptSizeInBlock`.
- Each top-level transaction has subtransactions with up to `ppMaxRefScriptSizePerTx` bytes of ref scripts → passes the per-tx check.
- The block-level check passes (`0 ≤ ppMaxRefScriptSizePerBlock`).
- Actual ref script processing per block = `N × ppMaxRefScriptSizePerTx`, which can be a multiple of `ppMaxRefScriptSizePerBlock`.

With current Conway/Dijkstra parameters (`ppMaxRefScriptSizePerTx = 200 KiB`, `ppMaxRefScriptSizePerBlock = 1 MiB`), a block with just 6 such transactions would impose 1.2 MiB of ref script work while the block-level check reports 0. This exceeds the intended per-block validation limit, matching the **Medium** impact category: *attacker-controlled blocks exceed intended validation limits*.

---

### Likelihood Explanation

Any block producer can exploit this. A valid Dijkstra transaction with subtransactions carrying large reference inputs is indistinguishable from a legitimate transaction at the mempool level. No privileged key or governance majority is required. The attacker only needs to be a normal transaction submitter whose transaction is included in a block, or a block producer themselves. The Dijkstra era is present in the production codebase and its BBODY rule inherits the Conway `validateBodyRefScriptsSizeTooBig` without a Dijkstra-specific override.

---

### Recommendation

Replace the call to `txNonDistinctRefScriptsSize` inside `totalRefScriptSizeInBlock` (or in a Dijkstra-era override of `validateBodyRefScriptsSizeTooBig`) with `batchNonDistinctRefScriptsSize` when processing Dijkstra-era top-level transactions, so that subtransaction reference scripts are counted toward the per-block cap. Alternatively, introduce a Dijkstra-specific `totalRefScriptSizeInBlock` that dispatches to `batchNonDistinctRefScriptsSize` for `TopTx`-level transactions.

---

### Proof of Concept

1. Set `ppMaxRefScriptSizePerTx = 200 KiB`, `ppMaxRefScriptSizePerBlock = 1 MiB`.
2. Produce a reference script UTxO of size ~200 KiB.
3. Craft a Dijkstra top-level transaction with **no** reference inputs in its own body, but with 1 subtransaction whose `referenceInputs` field points to the 200 KiB script UTxO.
4. `batchNonDistinctRefScriptsSize` = 0 (top-level) + 200 KiB (subtx) = 200 KiB ≤ `ppMaxRefScriptSizePerTx` → per-tx check **passes**.
5. `totalRefScriptSizeInBlock` for this tx = `txNonDistinctRefScriptsSize` of the top-level body = **0 KiB**.
6. Include 10 such transactions in one block: block-level check sees 0 KiB ≤ 1 MiB → **passes**.
7. Actual ref script bytes processed by honest nodes during block validation: 10 × 200 KiB = **2 MiB**, double the intended per-block cap.

```haskell
totalRefScriptSizeInBlock ::
  (AlonzoEraTx era, BabbageEraTxBody era) => ProtVer -> StrictSeq (Tx TopTx era) -> UTxO era -> Int
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      let updatedUtxo = accUtxo `Map.union` unUTxO toAdd
          toAdd
            | IsValid True <- tx ^. isValidTxL = txouts $ tx ^. bodyTxL
            | otherwise = collOuts $ tx ^. bodyTxL
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
```

```haskell
batchNonDistinctRefScriptsSize ::
  ( EraTx era

batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
```
