Audit Report

## Title
Permanent Freezing of Undistributed Rewards When `totalKernelStaked` Drops to Zero — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
When the last staker calls `initiateWithdrawal()` during an active reward period, `totalKernelStaked` drops to zero and the Synthetix-style `rewardPerToken()` accumulator short-circuits, halting all reward distribution. Rewards scheduled for the zero-staking interval are permanently locked in the contract with no on-chain recovery path, as the admin cannot restart distribution (`notifyRewardAmount` reverts on zero stake) and no rescue function exists.

## Finding Description
The `rewardPerToken()` function short-circuits when `totalKernelStaked == 0`, returning the stored accumulator unchanged: [1](#0-0) 

The `updateReward` modifier snapshots `updatedAt = lastTimeRewardApplicable()` on every interaction: [2](#0-1) 

When a new staker deposits after `totalKernelStaked` has been zero for some interval, `rewardPerTokenStored` is still unchanged (short-circuit path), but `updatedAt` advances to the current time. The rewards that accrued during the zero-staking window are permanently skipped — they can never be credited to any user.

`initiateWithdrawal()` has no guard preventing the last staker from exiting mid-period: [3](#0-2) 

The admin cannot restart distribution because `notifyRewardAmount` hard-reverts on zero stake: [4](#0-3) 

The contract's own NatSpec acknowledges this risk and explicitly relies on an off-chain operational assumption rather than any code-level enforcement: [5](#0-4) 

There is no admin rescue function anywhere in the contract to recover stranded reward tokens.

## Impact Explanation
Reward tokens transferred into the contract for a distribution period become permanently unclaimable by any user and unrecoverable by the admin. This is a concrete, on-chain permanent freezing of unclaimed yield, matching **Medium — Permanent freezing of unclaimed yield** in the allowed impact scope.

## Likelihood Explanation
The trigger is a standard, unprivileged user action (`initiateWithdrawal`). It requires only that the last remaining staker exits while a reward period is active — a realistic scenario during low-participation periods, market downturns, or long reward windows. The protocol's only mitigation is an off-chain operational promise (NatSpec comment), which is not enforceable on-chain and provides no protection against normal user behavior.

## Recommendation
1. **Track skipped rewards**: When `totalKernelStaked` drops to zero mid-period, accumulate the unallocated reward amount in a separate variable and allow the admin to roll it into the next `notifyRewardAmount` call.
2. **Admin rescue function**: Add an admin-only function to recover rewards that were never distributed (e.g., `rewardsToken.balanceOf(address(this)) - totalClaimableRewards`).
3. **Minimum stake invariant**: Enforce a protocol-held minimum stake (e.g., 1 wei) so `totalKernelStaked` can never reach zero during an active reward period.

## Proof of Concept
1. User A stakes 100 KERNEL at `t = 0`.
2. Admin calls `notifyRewardAmount(1000)` for a 100-day period → `rewardRate = 10/day`, `finishAt = t+100`.
3. At `t = 50`, User A calls `initiateWithdrawal(100)`:
   - `updateReward` snapshots `rewards[A] = 500`, `updatedAt = t+50`.
   - `totalKernelStaked` → 0.
4. From `t = 50` onward, `rewardPerToken()` returns `rewardPerTokenStored` unchanged (short-circuit). 500 remaining reward tokens are never distributed.
5. At `t = 75`, a new staker deposits 1 wei:
   - `updateReward` runs: `rewardPerToken()` still short-circuits (zero stake at call time), `rewardPerTokenStored` unchanged, `updatedAt` advances to `t+75`.
   - Rewards for `t=50` to `t=75` (250 tokens) are permanently skipped.
6. At `t = 100`, the new staker calls `getReward()` and receives only ~250 tokens (for `t=75` to `t=100`).
7. The 250 tokens for `t=50` to `t=75` remain locked in the contract with no function able to recover them.

**Foundry test plan**: Deploy `KernelDepositPool`, stake 100e18 KERNEL, call `notifyRewardAmount`, warp to midpoint, call `initiateWithdrawal` for full balance, warp to end, assert `rewardsToken.balanceOf(address(pool)) > 0` while no user can claim the remainder and `notifyRewardAmount` reverts.

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
