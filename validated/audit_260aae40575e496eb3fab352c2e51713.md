Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Minting Calculation Without Forcing an Update - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, a stored state variable in `LRTOracle` that is only updated on explicit external calls. Because neither `depositETH()` nor `depositAsset()` triggers a price refresh before computing the rsETH mint amount, any depositor who acts while the stored price is below the true current rate receives more rsETH than the fair rate entitles them to, extracting accumulated yield from existing holders.

## Finding Description
`LRTOracle` stores the rsETH/ETH exchange rate in the state variable `rsETHPrice` (`LRTOracle.sol:28`). This value is only written by `_updateRsETHPrice()`, which is invoked either through the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()` (`LRTOracle.sol:87-96`). No on-chain mechanism forces a refresh before a deposit is processed.

`LRTDepositPool.getRsETHAmountToMint()` computes the mint amount as:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
(`LRTDepositPool.sol:520`)

Both `depositETH()` (`LRTDepositPool.sol:87`) and `depositAsset()` (`LRTDepositPool.sol:111`) delegate to `_beforeDeposit()`, which is declared `private view` (`LRTDepositPool.sol:648,654`) — meaning it structurally cannot call any state-changing function such as `updateRSETHPrice()`. There is therefore no path by which the deposit flow can refresh `rsETHPrice` before using it.

As EigenLayer staking rewards accrue, the true ETH backing per rsETH token rises continuously, but `rsETHPrice` remains at its last-written value. A depositor who acts during this staleness window uses a denominator that is artificially low, receiving more rsETH than the current fair rate warrants.

The `pricePercentageLimit` guard (`LRTOracle.sol:256-265`) compounds the issue: if accumulated rewards push the true price increase above the configured threshold, `updateRSETHPrice()` reverts for any non-manager caller. This can extend the staleness window from hours to multiple days, proportionally increasing the extractable amount.

## Impact Explanation
The excess rsETH minted during the staleness window represents accumulated staking yield that belongs to existing rsETH holders. Once `updateRSETHPrice()` is eventually called and the price rises to reflect the true TVL, the attacker's inflated rsETH balance is redeemable for more ETH than was deposited. This is a direct, quantifiable transfer of unclaimed yield from existing holders to the attacker, matching the allowed impact **High — Theft of unclaimed yield**.

## Likelihood Explanation
Staking rewards accrue every block. Every block that passes without a price update creates a staleness window. The attack requires no special permissions: any external account can call `depositETH()` or `depositAsset()`. The attacker does not need to front-run any specific transaction — they simply need to deposit before the next keeper-driven `updateRSETHPrice()` call. Keeper-driven price updates are off-chain and not enforced on-chain, making gaps routine and predictable. When `pricePercentageLimit` is active, the window can span multiple days without manager intervention. Likelihood is **Medium**, yielding an overall severity of **High**.

## Recommendation
Refactor `_beforeDeposit()` from `private view` to `private` (removing the `view` modifier) and call `_updateRsETHPrice()` (or `updateRSETHPrice()`) at its start before invoking `getRsETHAmountToMint()`. Alternatively, add a staleness check that reverts if `rsETHPrice` was not updated within the current block, or expose an internal price-refresh path that bypasses `pricePercentageLimit` specifically for the deposit flow so that the price can always be refreshed atomically with the mint calculation.

## Proof of Concept
**Setup (fork test against mainnet or local state):**
1. Deploy or fork with `rsETHPrice = 1.00e18` (last updated 12 hours ago).
2. Simulate 12 hours of EigenLayer staking rewards so the true rate is `1.005e18` per rsETH (verifiable via `_getTotalEthInProtocol()` returning a higher value than `rsethSupply * rsETHPrice`).

**Attack sequence:**
1. Attacker calls `depositETH{value: 100 ether}(0, "")`.
2. `_beforeDeposit` → `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.00e18 = 100e18` rsETH.
3. Fair amount at true rate: `100e18 * 1e18 / 1.005e18 ≈ 99.502e18` rsETH.
4. Attacker receives `≈ 0.498e18` rsETH excess.
5. Keeper calls `updateRSETHPrice()`; `rsETHPrice` rises to `1.005e18`.
6. Attacker's 100 rsETH is now redeemable for `100 * 1.005 = 100.5 ETH` — a `0.5 ETH` profit extracted from existing holders.

**Foundry invariant test sketch:**
```solidity
// Invariant: for any deposit, rsethMinted * rsETHPrice_after_update <= depositValue
// Fuzz over depositAmount and staleness duration; assert no excess rsETH is minted
```
The invariant will break whenever `rsETHPrice` at deposit time is lower than `rsETHPrice` after the subsequent `updateRSETHPrice()` call, which occurs in every realistic staleness scenario.
