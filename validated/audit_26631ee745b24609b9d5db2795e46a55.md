Audit Report

## Title
Missing Slippage Protection in `deposit()` Functions Across Pool Contracts - (`File: contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`, `contracts/pools/RSETHPoolV2.sol`, `contracts/pools/RSETHPoolV2ExternalBridge.sol`)

## Summary
All `deposit()` overloads across the RSETH pool family compute the output rsETH/wrsETH amount from a live oracle rate at execution time but accept no `minRsETHAmount` parameter, giving depositors no on-chain mechanism to enforce a minimum output. A legitimate, non-adversarial oracle rate update between transaction submission and mining silently delivers fewer tokens than the user previewed, with no revert and no recourse. The protocol already implements the correct guard in `LRTDepositPool` but omits it from every pool contract.

## Finding Description
Every pool variant's `deposit()` function follows the same pattern:

1. Accept ETH or ERC-20 from the caller.
2. Call `viewSwapRsETHAmountAndFee()`, which reads the live oracle rate via `IOracle(rsETHOracle).getRate()`.
3. Transfer or mint the computed `rsETHAmount` to the caller with no floor check.

Confirmed instances (code verified against repository):

- `RSETHPoolV3ExternalBridge.sol` lines 366–384 (ETH) and 390–412 (token): `wrsETH.mint(msg.sender, rsETHAmount)` with no guard.
- `RSETHPoolV3.sol` lines 246–265 (ETH) and 271–293 (token): same pattern.
- `RSETHPool.sol` lines 265–278 (ETH) and 284–305 (token): `IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount)` with no guard.
- `RSETHPoolNoWrapper.sol` lines 231–244 (ETH) and 250–271 (token): `rsETH.safeTransfer(msg.sender, rsETHAmount)` with no guard.
- `RSETHPoolV3WithNativeChainBridge.sol` lines 282–301 (ETH) and 307–329 (token): `wrsETH.mint(msg.sender, rsETHAmount)` with no guard.
- `RSETHPoolV2.sol` lines 207–219: `wrsETH.mint(msg.sender, rsETHAmount)` with no guard.
- `RSETHPoolV2ExternalBridge.sol` lines 289–301: `wrsETH.mint(msg.sender, rsETHAmount)` with no guard.

The rate computation in `viewSwapRsETHAmountAndFee` is `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. When `rsETHToETHrate` increases (oracle update reporting staking rewards), the output decreases. There is no check preventing the minted/transferred amount from being arbitrarily lower than what the user previewed.

`LRTDepositPool.sol` at lines 667–669 demonstrates the correct pattern already used by the same protocol:
```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```
No equivalent guard exists in any pool contract.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A depositor who previews the exchange rate off-chain and submits a `deposit()` transaction can receive materially fewer rsETH/wrsETH tokens than expected if the oracle rate is updated before the transaction is mined. The depositor's ETH or LST is fully consumed; the shortfall in rsETH is unrecoverable. The protocol itself does not lose value — the impact is borne entirely by the depositor. This maps exactly to the allowed Low impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
The rsETH oracle rate increases monotonically under normal staking conditions and jumps discretely when staking rewards are reported. No adversarial action is required: any depositor whose transaction is pending across a routine oracle update is affected. This is a repeatable, non-adversarial scenario that applies to every depositor who relies on an off-chain quote (the only mechanism available, since there is no on-chain slippage parameter). The scenario is triggered by any unprivileged external user calling `deposit()`.

## Recommendation
Add a `minRsETHAmount` parameter to all `deposit()` overloads in every pool contract and revert if the computed output falls below it, mirroring the pattern in `LRTDepositPool`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert MinimumAmountToReceiveNotMet();

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same change to every token `deposit()` overload across all pool contracts.

## Proof of Concept
1. Alice calls `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee(1 ether)` off-chain and observes she will receive `X` wrsETH at the current oracle rate `R`.
2. Alice submits `deposit{value: 1 ether}("ref")`.
3. Before Alice's transaction is mined, the rsETH oracle is updated (routine staking reward reporting), increasing `getRate()` from `R` to `R + delta`.
4. Alice's transaction executes: `viewSwapRsETHAmountAndFee` computes `rsETHAmount = 1e18 * 1e18 / (R + delta)`, which is less than `X`.
5. Alice receives fewer wrsETH than previewed. Her 1 ETH is fully consumed. No revert occurs. No recourse exists.

**Foundry fork test outline:**
```solidity
function testDepositSlippage() public {
    // Fork mainnet at block N (before oracle update)
    uint256 preview = pool.viewSwapRsETHAmountAndFee(1 ether);

    // Simulate oracle update (roll to block N+1 where oracle reports rewards)
    vm.prank(oracleUpdater);
    oracle.update(...); // increases getRate()

    // Alice deposits
    vm.prank(alice);
    pool.deposit{value: 1 ether}("ref");

    uint256 received = wrsETH.balanceOf(alice);
    assertLt(received, preview); // Alice received less than previewed, no revert
}
```
