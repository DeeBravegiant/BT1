Audit Report

## Title
Excess ETH Permanently Locked in `updateRate` - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`MultiChainRateProvider.updateRate()` is `payable` and accepts ETH from any caller to fund LayerZero cross-chain messaging fees. For each configured receiver, it forwards exactly `estimatedFee` to the LayerZero endpoint rather than `msg.value`, leaving any surplus ETH permanently trapped in the contract. The contract contains no ETH withdrawal or sweep function, making the excess irrecoverable.

## Finding Description
In `MultiChainRateProvider.updateRate()` (L108–137), the loop calls `estimateFees()` per receiver and sends only that exact amount to LayerZero:

```solidity
(uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
    .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
    dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
);
``` [1](#0-0) 

Although `payable(msg.sender)` is passed as the LayerZero refund address, LayerZero can only refund what it receives — exactly `estimatedFee`. The delta `msg.value − Σ(estimatedFee_i)` is never forwarded anywhere and remains in `MultiChainRateProvider`. The contract defines no `receive()`, `fallback()`, or owner-callable ETH sweep function, so the balance is permanently unrecoverable. [2](#0-1) 

This contrasts directly with `CrossChainRateProvider.updateRate()`, which correctly passes the full `msg.value` to LayerZero, enabling LayerZero's native refund mechanism to return any surplus:

```solidity
ILayerZeroEndpoint(layerZeroEndpoint).send{ value: msg.value }(
    dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
);
``` [3](#0-2) 

The `estimateTotalFee()` helper exists but does not prevent the issue: it is a separate view call, fees can shift between estimation and execution, and callers routinely add a buffer to avoid reverts. [4](#0-3) 

## Impact Explanation
Any ETH sent above the exact sum of per-receiver LayerZero fees is permanently frozen inside `MultiChainRateProvider`. Because the contract has no ETH recovery path, every caller who overpays — a common and expected pattern when buffering against fee fluctuations — suffers an irreversible loss. This constitutes **permanent freezing of funds**, matching the Critical impact class in the allowed scope.

## Likelihood Explanation
`updateRate()` has no access control and is callable by any external account. [5](#0-4) 
Callers must estimate the total fee off-chain and send at least that amount; any rounding, stale estimate, or deliberate buffer results in locked ETH. The receiver list is dynamic (`addRateReceiver`/`removeRateReceiver`), making exact off-chain estimation error-prone. [6](#0-5) 

## Recommendation
After the loop, refund any remaining contract balance to `msg.sender`:

```solidity
function updateRate() external payable nonReentrant {
    // ... existing loop ...

    uint256 remaining = address(this).balance;
    if (remaining > 0) {
        (bool success,) = payable(msg.sender).call{ value: remaining }("");
        require(success, "ETH refund failed");
    }

    emit RateUpdated(rate);
}
```

Alternatively, mirror `CrossChainRateProvider` by splitting `msg.value` proportionally across receivers and forwarding the full allocated amount to each LayerZero `send` call so LayerZero's native refund mechanism handles the surplus.

## Proof of Concept
1. Deploy `MultiChainRateProvider` with 3 configured rate receivers.
2. Call `estimateTotalFee()` off-chain; suppose it returns `0.024 ether`.
3. Call `updateRate{ value: 0.030 ether }()` (caller adds a 25% buffer).
4. The loop sends `0.008 ether` × 3 = `0.024 ether` to LayerZero.
5. `0.006 ether` remains in the contract after the call.
6. No function exists to withdraw it; `address(MultiChainRateProvider).balance == 0.006 ether` permanently.
7. Foundry fork test: assert `address(provider).balance == 0` after `updateRate` — the assertion fails, confirming the bug.
