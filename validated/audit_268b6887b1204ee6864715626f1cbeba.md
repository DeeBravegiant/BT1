Audit Report

## Title
Stale `rsETHPrice` Enables Theft of Accumulated MEV/Execution-Layer Rewards from `FeeReceiver` - (File: contracts/LRTDepositPool.sol)

## Summary
`getETHDistributionData()` in `LRTDepositPool` intentionally excludes the `FeeReceiver` balance from protocol TVL accounting. Because `rsETHPrice` is a stored value updated only on explicit calls, and both `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` are permissionless, an attacker can deposit at a stale (artificially low) price, then trigger the reward transfer and price update, locking in a withdrawal request at the inflated price. After the mandatory delay, the attacker redeems more ETH than deposited, stealing a portion of accumulated rewards that belonged to pre-existing rsETH holders.

## Finding Description

`getETHDistributionData()` explicitly excludes the `FeeReceiver` balance from ETH accounting, as documented in its own NatSpec:

```solidity
/// @dev rewards are not accounted here
/// it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool
``` [1](#0-0) 

`getTotalAssetDeposits()` aggregates only from `getAssetDistributionData()` / `getETHDistributionData()`, so the `FeeReceiver` balance is absent from the total used to compute `rsETHPrice`. [2](#0-1) 

`getRsETHAmountToMint()` divides by the stored `rsETHPrice`, which was last computed without the `FeeReceiver` balance: [3](#0-2) 

`FeeReceiver.sendFunds()` has no access control — any caller can push accumulated rewards into the deposit pool: [4](#0-3) 

`LRTOracle.updateRSETHPrice()` is a public, permissionless function (gated only by `whenNotPaused`): [5](#0-4) 

`initiateWithdrawal()` locks in `expectedAssetAmount` at the **current** `rsETHPrice` at the time of the call: [6](#0-5) 

`_calculatePayoutAmount()` in `unlockQueue()` pays `min(expectedAssetAmount, currentReturn)` — so the amount locked in at `initiateWithdrawal()` time is honored if the price has not fallen: [7](#0-6) 

**Exploit path:**

Let TVL = `T`, rsETH supply = `S`, stored price `P = T/S`, and `FeeReceiver` holds `R` ETH (not yet flushed).

1. Attacker calls `depositETH{value: D}()`. Minted rsETH = `D * 1e18 / P = D*S/T`. Because `P` excludes `R`, the attacker receives more rsETH than their true proportional share of `T + R`.
2. Attacker calls `FeeReceiver.sendFunds()` — `R` ETH moves into `LRTDepositPool`.
3. Attacker calls `LRTOracle.updateRSETHPrice()` — new price `P' = (T + D + R) / (S + D*S/T)`. Since `R > 0`, `P' > P`.
4. Attacker calls `initiateWithdrawal()` — `expectedAssetAmount` is locked at `(D*S/T) * P' / 1e18 = D*(T+D+R)/(T+D) = D + D*R/(T+D) > D`.
5. After the 8-day withdrawal delay (`withdrawalDelayBlocks = 8 days / 12 seconds`), the operator calls `unlockQueue()` as part of normal operations. `_calculatePayoutAmount()` returns `min(expectedAssetAmount, currentReturn)`. Since `R` is now permanently in the protocol, the price remains at `P'`, so `currentReturn = expectedAssetAmount` and the full inflated amount is paid.
6. Attacker calls `completeWithdrawal()` and receives `D + D*R/(T+D)` ETH — a profit of `D*R/(T+D)` at the expense of pre-existing holders. [8](#0-7) 

The `pricePercentageLimit` guard in `_updateRsETHPrice()` is a partial mitigation: if the price jump from `P` to `P'` exceeds the configured limit, a non-manager caller receives `PriceAboveDailyThreshold`. However, this does not fully prevent the attack because: (a) `pricePercentageLimit` may be zero (unset), (b) for small `R` relative to `T`, the increase is within any reasonable limit, and (c) the attacker can wait for the manager to call `updateRSETHPriceAsManager()` as part of normal operations. [9](#0-8) 

## Impact Explanation

**High — Theft of unclaimed yield.** Pre-existing rsETH holders accumulate MEV and execution-layer rewards in `FeeReceiver`. An attacker who deposits before those rewards are flushed receives a disproportionate share of rsETH. After triggering the flush and price update, the attacker's rsETH redeems for more ETH than was deposited, with the surplus drawn directly from the accumulated rewards that belonged to existing holders. The stolen fraction is `D*R/(T+D)`, which scales with both the deposit size `D` (amplifiable via flash loans for the deposit, repaid after withdrawal) and the accumulated reward balance `R`.

## Likelihood Explanation

**High.** No privileged role is required for any attacker-controlled step. The `FeeReceiver` balance is publicly visible on-chain. The attack requires only standard contract calls (`depositETH`, `sendFunds`, `updateRSETHPrice`, `initiateWithdrawal`, `completeWithdrawal`) available to any EOA. The 8-day withdrawal delay is a friction cost, not a security barrier, since `expectedAssetAmount` is locked in at `initiateWithdrawal()` time. The attack is repeatable every time rewards accumulate in `FeeReceiver`.

## Recommendation

Include the `FeeReceiver` (and any other reward-holding contract) balance in `getETHDistributionData()` so that `getTotalAssetDeposits()` and therefore `rsETHPrice` always reflect the true protocol TVL. This eliminates the price discrepancy window entirely. Alternatively, add access control to `FeeReceiver.sendFunds()` so that only a privileged role can flush rewards, preventing an attacker from controlling the timing of the price jump relative to their deposit.
