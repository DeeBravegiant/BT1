Audit Report

## Title
Daily Mint Limit Exhaustion Enables Block Stuffing DoS on ETH Deposits — (File: contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

## Summary
The `limitDailyMint` modifier enforces a global daily minting cap with no per-address sub-limit, allowing a single caller to exhaust the entire `dailyMintLimit` in one `deposit()` call. On L2 chains where this contract is deployed, an attacker can follow up with block stuffing to prevent `block.timestamp` from crossing the day boundary, extending the DoS beyond the natural 86400-second reset window and blocking all ETH deposits for the duration of the attack.

## Finding Description
`getCurrentDay()` at line 390–392 computes the current epoch as `(block.timestamp - startTimestamp) / 1 days`. The `limitDailyMint` modifier (lines 108–137) resets `dailyMintAmount` only when `currentDay > lastMintDay`; otherwise it enforces a hard cap: if `dailyMintAmount + rsETHAmount > dailyMintLimit`, the call reverts with `DailyMintLimitExceeded`. There is no per-address accounting anywhere in the modifier or in `deposit()`.

The `deposit(string)` function (lines 282–301) is fully permissionless — it requires only `whenNotPaused` and `isEthDepositEnabled`. A single caller can send ETH equivalent to the full `dailyMintLimit` in one transaction, setting `dailyMintAmount == dailyMintLimit`. Every subsequent `deposit()` call from any address then reverts for the rest of the day.

Because the contract is deployed on L2 chains (evidenced by `l1VaultETHForL2Chain` and the native bridge architecture), block stuffing is economically feasible. After exhausting the daily limit, the attacker submits high-gas dummy transactions to fill every subsequent block, keeping `block.timestamp` from advancing past the day boundary. `getCurrentDay()` never exceeds `lastMintDay`, so the limit never resets, and the DoS extends indefinitely beyond the natural 86400-second window. The attacker recovers their ETH as wrsETH (no capital loss beyond gas), making the attack cheap to sustain.

Existing guards are insufficient: `nonReentrant` prevents reentrancy but not a single large deposit; `whenNotPaused` requires privileged action to invoke; `isEthDepositEnabled` is a protocol-level toggle, not a per-user guard.

## Impact Explanation
**Low — Block stuffing.** All ETH `deposit()` calls revert with `DailyMintLimitExceeded` for the duration of the attack. The ETH deposit functionality is temporarily frozen for all users except the attacker. No funds already in the contract are at risk, but new deposits are blocked for as long as the attacker continues stuffing blocks.

## Likelihood Explanation
The attack is permissionless — no special role is required. Preconditions are: (1) `isEthDepositEnabled == true` (normal operating state), (2) `dailyMintLimit` is finite (always true in production), and (3) the attacker holds ETH sufficient to match the daily quota (recovered as wrsETH, so no net capital loss). On L2 chains with low block gas limits and cheap gas (e.g., Optimism, Base, Scroll), block stuffing is a documented and repeatable attack vector. The attack is repeatable each day if the attacker is willing to continue.

## Recommendation
1. Add a per-address daily deposit cap inside `limitDailyMint` (e.g., `mapping(address => uint256) public userDailyMintAmount`) to prevent a single actor from consuming the global quota.
2. Enforce a maximum single-deposit size relative to `dailyMintLimit` (e.g., no single deposit may exceed 10% of the daily limit).
3. Consider a sliding-window rate limiter instead of a fixed epoch boundary to mitigate the cliff-reset griefing pattern.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";

interface IPool {
    function deposit(string memory referralId) external payable;
    function dailyMintLimit() external view returns (uint256);
    function dailyMintAmount() external view returns (uint256);
    error DailyMintLimitExceeded();
}

contract BlockStuffingPoC is Test {
    IPool pool; // fork-initialized to deployed RSETHPoolV3WithNativeChainBridge

    address attacker = address(0xA77);
    address victim   = address(0xB0B);

    function testBlockStuffingDoS() public {
        // Step 1: attacker exhausts the full daily limit in one tx
        uint256 quota = pool.dailyMintLimit();
        vm.deal(attacker, quota);
        vm.prank(attacker);
        pool.deposit{value: quota}("ref");
        assertEq(pool.dailyMintAmount(), quota);

        // Step 2: simulate block stuffing — advance block number but
        //         hold block.timestamp constant (attacker fills every block)
        uint256 frozenTs = block.timestamp;
        vm.roll(block.number + 5_000);
        vm.warp(frozenTs); // timestamp pinned: day never resets

        // Step 3: victim's deposit reverts for the entire stuffed window
        vm.deal(victim, 1 ether);
        vm.prank(victim);
        vm.expectRevert(IPool.DailyMintLimitExceeded.selector);
        pool.deposit{value: 1 ether}("ref");
    }
}
```
The `vm.roll()` + `vm.warp()` combination models block stuffing: blocks are produced (block number advances) but `block.timestamp` does not cross the day boundary, so `getCurrentDay()` never exceeds `lastMintDay` and the limit never resets. The test demonstrates that after the attacker exhausts `dailyMintLimit`, all subsequent `deposit()` calls revert.
