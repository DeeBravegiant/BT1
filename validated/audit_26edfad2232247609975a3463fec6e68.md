### Title
Missing Zero-Value Validation for `maxRefScriptSizePerBlock` and `maxRefScriptSizePerTx` in Dijkstra Era `ppuWellFormed` â€” (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/PParams.hs`)

---

### Summary

The `ppuWellFormed` function for `DijkstraEra` does not validate the two new Dijkstra-specific protocol parameters `maxRefScriptSizePerBlock` and `maxRefScriptSizePerTx` against zero. This allows a governance `ParameterChange` proposal setting either parameter to `0` to pass the `MalformedProposal` check and be accepted into the on-chain proposals queue. If such a proposal is ratified, it would cause every block or transaction containing reference scripts to fail ledger validation, effectively freezing all DApp activity that relies on reference scripts.

---

### Finding Description

**Vulnerability class:** Insufficient input validation / missing bounds check on protocol parameters (direct analog of the external report's "extreme term values" class).

**Root cause â€” `ppuWellFormed` for `DijkstraEra`:**

The `ConwayEraPParams` type class exposes `ppuWellFormed`, which is called by the `GOV` rule's `actionWellFormed` predicate to reject `ParameterChange` proposals that contain semantically invalid parameter values before they enter the proposals queue. [1](#0-0) 

The `DijkstraEra` instance of `ppuWellFormed` (which completely overrides the `ConwayEra` instance) checks only the parameters inherited from Conway: [2](#0-1) 

The two new Dijkstra parameters â€” `dppMaxRefScriptSizePerBlock` and `dppMaxRefScriptSizePerTx` â€” are declared as plain `Word32` fields: [3](#0-2) 

Neither `ppuMaxRefScriptSizePerBlockL` nor `ppuMaxRefScriptSizePerTxL` appears in the `ppuWellFormed` list, so a proposal setting either to `0` passes the well-formedness gate without error.

**Downstream enforcement â€” `validateBodyRefScriptsSizeTooBig`:**

When `maxRefScriptSizePerBlock = 0` is enacted, the block-level check becomes `totalSize <= 0`. Any block containing even one transaction that references a script output will have `totalSize > 0` and will be rejected: [4](#0-3) 

**Downstream enforcement â€” `validateRefScriptSize`:**

When `maxRefScriptSizePerTx = 0` is enacted, the per-transaction check becomes `totalRefScriptSize <= 0`, rejecting every transaction that uses a reference script: [5](#0-4) 

**Contrast with analogous parameters:** Every other size-limit parameter that can break the ledger if set to zero is explicitly guarded. For example, `ppuMaxBBSizeL`, `ppuMaxTxSizeL`, `ppuMaxBHSizeL`, and `ppuMaxValSizeL` all have `/= 0` checks in both `ConwayEra` and `DijkstraEra`: [6](#0-5) 

The new Dijkstra parameters were simply not added to this list when the era was introduced.

---

### Impact Explanation

**Impact: Medium** â€” Attacker-controlled governance proposals modify transaction validation limits outside design parameters.

If a `ParameterChange` proposal setting `maxRefScriptSizePerBlock = 0` or `maxRefScriptSizePerTx = 0` is ratified:

- `validateBodyRefScriptsSizeTooBig` will reject every block that contains any transaction referencing a script output, because `totalSize > 0 = maxSize`.
- `validateRefScriptSize` will reject every individual transaction that uses a reference script.
- All DApps relying on reference scripts (the primary Babbage/Conway/Dijkstra DApp pattern) would be permanently frozen until a hard fork restores the parameters.

This maps to the allowed impact: **"High. Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork"** (if enacted) and **"Medium. Attacker-controlled proposalsâ€¦ modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters"** (at the proposal-submission stage, which is the attacker-controlled step).

---

### Likelihood Explanation

Any actor who pays the `govActionDeposit` can submit a `ParameterChange` governance proposal. The `ppuWellFormed` check is the only ledger-level gate preventing a zero-valued proposal from entering the queue. Because that gate is absent for these two parameters, the submission step requires no special privilege â€” only the deposit. Ratification requires governance majority, but the submission of a well-formed-looking malformed proposal is fully attacker-controlled and reachable by an unprivileged transaction sender.

---

### Recommendation

Add non-zero guards for `ppuMaxRefScriptSizePerBlockL` and `ppuMaxRefScriptSizePerTxL` inside the `ppuWellFormed` implementation for `DijkstraEra`, mirroring the existing pattern for `ppuMaxBBSizeL`, `ppuMaxTxSizeL`, etc.:

```haskell
instance ConwayEraPParams DijkstraEra where
  ppuWellFormed _pv ppu =
    and
      [ -- existing checks ...
        isValid (/= 0) ppuMaxBBSizeL
      , isValid (/= 0) ppuMaxTxSizeL
      -- ... add:
      , isValid (/= 0) ppuMaxRefScriptSizePerBlockL
      , isValid (/= 0) ppuMaxRefScriptSizePerTxL
      ]
``` [2](#0-1)

### Proof of Concept

1. Construct a `ParameterChange` governance action with a `PParamsUpdate` that sets only `maxRefScriptSizePerBlock = SJust 0`.
2. Submit it via the `GOV` rule. `actionWellFormed` calls `ppuWellFormed`, which iterates the list at lines 541â€“556 of `DijkstraEra`'s instance. None of those checks cover `ppuMaxRefScriptSizePerBlockL`, so `ppuWellFormed` returns `True`.
3. The proposal is accepted into the `Proposals` state without a `MalformedProposal` failure.
4. After ratification and enactment, `validateBodyRefScriptsSizeTooBig` computes `maxSize = fromIntegral (0 :: Word32) = 0`. Any block with `totalSize > 0` (i.e., any block containing a transaction that references a script output) fails with `BodyRefScriptsSizeTooBig`. [4](#0-3) [7](#0-6)
