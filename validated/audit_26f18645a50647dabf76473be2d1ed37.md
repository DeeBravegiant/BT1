### Title
User-Controlled Finality Parameters Allow MPC Signing of Reorg-Susceptible Foreign-Chain Transactions - (File: `crates/contract/src/lib.rs`, `crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` entry point accepts a fully user-controlled finality parameter (`EvmFinality::Latest` for EVM chains, `BlockConfirmations(0)` for Bitcoin) with no minimum finality floor enforced anywhere in the contract or node code. This is the direct analog of the reported Paraspace vulnerability: just as the fallback oracle blindly used Uniswap V2 spot reserves (manipulable in one transaction), the MPC network will blindly sign a foreign-chain attestation against the current chain tip (manipulable via a reorg), producing a valid threshold signature over a transaction that may never finalize.

---

### Finding Description

`verify_foreign_transaction` is a public, payable contract method. Its only guards are a domain-purpose check, a gas attachment check, a minimum deposit, and a chain-whitelist check. No minimum finality level is enforced:

```rust
// crates/contract/src/lib.rs ~L519-L557
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(
        request.domain_id,
        DomainPurpose::ForeignTx,
        Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
        MINIMUM_SIGN_REQUEST_DEPOSIT,
    );
    let requested_chain = request.request.chain();
    let supported_chains = self.get_supported_foreign_chains();
    if !supported_chains.contains(&requested_chain) { env::panic_str(...); }
    // ← no finality floor check; request is enqueued as-is
    self.enqueue_yield_request(...);
}
```

The `EvmFinality` enum explicitly exposes `Latest` as a valid variant:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs ~L768-L772
pub enum EvmFinality {
    Latest,   // ← spot chain head; reorg-susceptible
    Safe,
    Finalized,
}
```

Each MPC node's EVM inspector maps `Latest` directly to `FinalityTag::Latest` and accepts the block if `head.number >= receipt_block_number` — i.e., the transaction is in any block at or before the current tip:

```rust
// crates/foreign-chain-inspector/src/evm/inspector.rs ~L107-L124
let finality_tag = match finality {
    EthereumFinality::Finalized => FinalityTag::Finalized,
    EthereumFinality::Safe      => FinalityTag::Safe,
    EthereumFinality::Latest    => FinalityTag::Latest,   // ← spot head
};
// ...
if head.number < receipt_block_number {
    return Err(ForeignChainInspectionError::NotFinalized);
}
Ok(())
```

For Bitcoin, `BlockConfirmations` is a plain `u64` with no lower bound. The inspector's only guard is:

```rust
// crates/foreign-chain-inspector/src/bitcoin/inspector.rs ~L51-L58
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

With `BlockConfirmations(0)` the inequality `0 <= N` is always true for any confirmed transaction, bypassing the confirmation guard entirely.

The signed payload is `SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))`. The `request` field carries the caller-supplied finality level, but the MPC network does not refuse to sign it — it attests to whatever finality the caller requested.

---

### Impact Explanation

The MPC network's stated purpose is to let NEAR contracts react to external chain events **without a trusted relayer**. When the network signs a payload for a `Latest`-finality EVM transaction or a 0-confirmation Bitcoin transaction, it issues a valid threshold signature attesting that the transaction occurred. If the underlying transaction is subsequently reorged out, the signature remains cryptographically valid. A downstream bridge contract (e.g., Omnibridge inbound flow) that does not independently re-validate the finality field in the signed payload will accept the forged attestation and release funds for a transaction that never finalized — a double-spend.

This maps directly to the allowed High impact: **forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

---

### Likelihood Explanation

- **EVM `Latest`**: Any EVM chain where the attacker can cause or wait for a natural 1-block reorg (common on chains with sub-second block times or during network congestion) is sufficient. The attacker does not need to control a majority of validators — a single-block reorg is enough if the MPC signing round completes before the reorg is detected.
- **Bitcoin `BlockConfirmations(0)`**: A transaction with 0 required confirmations passes the guard for any confirmed transaction. A 1-confirmation Bitcoin transaction is well-known to be susceptible to double-spend attacks with modest hash power.
- The attack requires no privileged access, no TEE compromise, and no collusion among MPC participants — only the ability to submit a `verify_foreign_transaction` call and influence the foreign chain's tip.

---

### Recommendation

1. **Enforce a minimum finality floor in the contract**: Reject `EvmFinality::Latest` (and optionally `EvmFinality::Safe`) at the `verify_foreign_transaction` entry point. Require at least `Finalized` for EVM chains.
2. **Enforce a minimum confirmation count for Bitcoin**: Reject `BlockConfirmations(N)` where `N` is below a protocol-defined minimum (e.g., 6).
3. **Alternatively, encode the minimum in the on-chain foreign-chain configuration** so governance can set per-chain floors without a contract upgrade.
4. **Document the finality guarantee** so downstream bridge contracts know they must re-check the finality field in the signed payload if they accept `Latest` or low-confirmation requests.

---

### Proof of Concept

1. Attacker deposits 1 ETH into a bridge contract on an EVM chain (e.g., Abstract). The deposit transaction lands in block N (the current `latest` head).
2. Attacker immediately calls `verify_foreign_transaction` on the NEAR MPC contract with:
   ```json
   { "request": { "Abstract": { "tx_id": "<deposit_tx>", "finality": "Latest", "extractors": ["BlockHash"] } }, "domain_id": <ForeignTx domain>, "payload_version": 1 }
   ```
3. MPC nodes query their RPC providers. Block N is the `latest` head; `head.number >= N` passes. Nodes extract the block hash and collectively sign `SHA-256(borsh({ request, [BlockHash(N)] }))`.
4. Before the NEAR yield resolves, the attacker (or a colluding miner/validator) causes a 1-block reorg that replaces block N with block N′ that does not contain the deposit transaction.
5. The NEAR yield resolves with the valid MPC signature. The attacker submits this signature to the Omnibridge NEAR contract, which releases 1 ETH worth of wrapped tokens.
6. The deposit transaction no longer exists on the canonical EVM chain. The attacker has received NEAR-side tokens for a deposit that was never finalized — a double-spend. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    /// Checks that the receipt's block has reached the requested finality level — i.e. that the
    /// head of the chain at `finality` is at or past `receipt_block_number`.
    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }
