### Title
Missing Vote Signature Verification in `handle_vote` Allows Any Peer to Forge Validator Votes and Drive Unauthorized Consensus Decisions — (File: `crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

`SingleHeightConsensus::handle_vote` accepts any `Vote` message whose `voter` field names a committee member, without verifying the cryptographic signature. Any network peer can forge precommit votes from arbitrary validators, accumulate a quorum of forged votes for a proposal the node has already validated, and cause the node to emit `DecisionReached` — committing a block that the real validator set never agreed on.

---

### Finding Description

`handle_vote` in `crates/apollo_consensus/src/single_height_consensus.rs` performs two checks before accepting a vote:

1. `vote.height == self.state_machine.height()` — height guard
2. `self.committee.members().iter().any(|s| s.address == vote.voter)` — committee membership check

There is no cryptographic verification that the message was actually signed by the private key of `vote.voter`. The code carries an explicit acknowledgement of this gap:

```rust
// TODO(Asmaa): verify the signature   ← line 242
```

On the signing side, `make_self_vote` in `state_machine.rs` emits votes with an empty placeholder:

```rust
// TODO(Asmaa): sign the vote
signature: RawSignature::default(),   ← line 254-255
```

The library function `verify_precommit_vote_signature` exists in `crates/apollo_signature_manager/src/signature_manager.rs` and is tested in isolation, but it is **never called** anywhere in the consensus vote-handling path. The codebase itself documents the consequence in `manager.rs`:

```
// Since vote signatures are not yet verified, a peer can forge votes with
// arbitrary voter addresses; without this cap that would grow `future_votes`
// without bound and exhaust memory.   ← lines 327-329
```

The cap mentioned there only bounds the future-height cache; it does not prevent forged votes from being processed for the **current** height.

---

### Impact Explanation

**Broken invariant**: BFT consensus requires that a `DecisionReached` event is only emitted when ≥ 2/3 (by weight) of the real validator set has cryptographically committed to a proposal. Without signature verification, this invariant is broken.

**Concrete attack path**:

1. The legitimate proposer for round `R` broadcasts a valid `ProposalInit` + content. The target node validates it through the batcher; the state machine records the commitment in `self.proposals[R]` and the context stores it in `valid_proposals`.

2. A malicious network peer (not a validator) constructs `Vote` messages with:
   - `voter` = any committee member address (e.g., the virtual proposer + two other validators)
   - `proposal_commitment` = the commitment just validated in step 1
   - `vote_type` = `Precommit`
   - `signature` = any bytes (never checked)

3. The peer sends these forged votes to the target node. `handle_vote` passes both guards (height matches, `voter` is in the committee) and forwards them to the state machine.

4. `upon_decision` in `state_machine.rs` fires once:
   - `self.proposals.get(&R)` returns the validated commitment ✓
   - `value_has_enough_votes` reaches quorum with the forged votes ✓
   - `virtual_proposer_in_favor` is satisfied because the virtual proposer's vote was also forged ✓

5. `SMRequest::DecisionReached` is returned → `context.decision_reached()` is called → `batcher.decision_reached()` commits the block to storage.

The node has now committed a block without the real validators ever casting their votes. It will advance to height `H+1` while the rest of the network is still at `H`, causing a permanent fork for this node.

**Impact category**: High — transaction/signature/hash logic binds the wrong signer (the `voter` field is accepted as authoritative without proof of key ownership), and the resulting wrong consensus decision causes the node to commit state that the network has not agreed on.

---

### Likelihood Explanation

- The attack requires only a single network peer with the ability to send `Vote` protobuf messages to the target node — no validator key, no stake, no privileged access.
- The only prerequisite is that the legitimate proposer has already sent a valid proposal that the target node has validated (a normal condition during every block).
- The codebase explicitly acknowledges the gap and notes it is deferred until "the network expands beyond the current trusted set" (`manager.rs` line 333). As the network grows beyond a closed trusted set, this becomes immediately exploitable.

---

### Recommendation

1. **Verify signatures in `handle_vote`**: Call `verify_precommit_vote_signature` (already implemented in `crates/apollo_signature_manager/src/signature_manager.rs`) against the public key of `vote.voter` before accepting the vote. Reject votes with invalid or missing signatures.

2. **Sign votes in `make_self_vote`**: Replace `signature: RawSignature::default()` with an actual ECDSA signature over the vote digest using the node's private key (the `SignatureManager::sign_precommit_vote` path already exists).

3. **Apply the same fix to future-height cached votes** in `cache_future_vote` (`manager.rs`): verify signatures before caching, not just cap the cache size.
