Audit Report

## Title
First Depositor rsETH Price Inflation via ETH Donation to LRTDepositPool - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

## Summary
An attacker who is the first depositor can mint 1 wei of rsETH, then donate ETH directly to `LRTDepositPool` via its open `receive()` function. Because `getETHDistributionData()` counts `address(this).balance` as protocol ETH and `_updateRsETHPrice()` computes price as `totalETHInProtocol / rsethSupply`, the attacker inflates the rsETH price arbitrarily. Subsequent depositors receive 0 rsETH due to integer division truncation and permanently lose their deposited ETH, which the attacker recovers upon withdrawal.

## Finding Description

**Root cause — three cooperating weaknesses:**

1. `_updateRsETHPrice()` initializes `rsETHPrice = 1 ether` when supply is zero, allowing a 1-wei deposit to mint exactly 1 wei rsETH. [1](#0-0) 

2. `getETHDistributionData()` uses `address(this).balance` as the ETH balance of the deposit pool, so any ETH sent via the open `receive()` fallback is immediately counted as protocol TVL. [2](#0-1) [3](#0-2) 

3. The price-increase guard in `_updateRsETHPrice()` is gated on `pricePercentageLimit > 0`, but `initialize()` never sets `pricePercentageLimit`, so it defaults to `0` and the guard is permanently disabled until an admin explicitly calls `setPricePercentageLimit`. [4](#0-3) [5](#0-4) 

**Exploit flow:**

- **Step 1:** Attacker calls `depositETH{value: 1}(0, "")`. With `rsETHPrice = 1e18`, `getRsETHAmountToMint` yields `(1 * 1e18) / 1e18 = 1` wei rsETH. Attacker now holds 100% of rsETH supply (1 wei). [6](#0-5) 

- **Step 2:** Attacker sends D = 1 ETH directly to `LRTDepositPool` via `receive()`. Pool balance becomes `1 + 1e18` wei.

- **Step 3:** Attacker calls `updateRSETHPrice()` (public, no access control). `_getTotalEthInProtocol()` reads `getTotalAssetDeposits(ETH)` → `getETHDistributionData()` → `address(this).balance = 1 + 1e18`. With `rsethSupply = 1`, `newRsETHPrice = (1 + 1e18) * 1e18 / 1 ≈ 1e36`. Since `pricePercentageLimit == 0`, `isPriceIncreaseOffLimit = false` and no revert occurs. [7](#0-6) [8](#0-7) 

- **Step 4:** Victim calls `depositETH{value: 1 ether}(0, "")`. `getRsETHAmountToMint` computes `(1e18 * 1e18) / 1e36 = 1`, which rounds to 0 for any victim deposit smaller than the inflated price. With `minRSETHAmountExpected = 0`, the check `rsethAmountToMint < minRSETHAmountExpected` → `0 < 0` is false, so the deposit succeeds and `_mintRsETH(0)` mints nothing to the victim. [9](#0-8) [10](#0-9) 

- **Step 5:** Attacker still holds 100% of rsETH supply. After `updateRSETHPrice()` is called again, `rsETHPrice` reflects `(1 + D + V) * 1e18`. The attacker redeems 1 wei rsETH for the full pool balance, recovering their donation plus the victim's ETH.

## Impact Explanation

**Critical — Direct theft of any user funds.** A victim depositing V ETH with `minRSETHAmountExpected = 0` (the default in most integrations) receives 0 rsETH and permanently loses their entire deposit. The attacker recovers both the donation and the victim's deposit upon withdrawal. This matches the allowed Critical impact class: "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation

**High.** All preconditions are permissionless and realistic at protocol launch:
1. Being the first depositor — trivially achievable by front-running or acting at deployment.
2. Sending ETH to `receive()` — no access control.
3. Calling `updateRSETHPrice()` — public function, no role required.
4. `pricePercentageLimit == 0` — the **default state** since `initialize` does not set it. [11](#0-10) 

No governance capture, oracle compromise, or privileged access is required. The attack is executable atomically or across a few transactions.

## Recommendation

1. **Seed the pool at initialization** with a non-trivial ETH deposit minted to a dead/burn address (e.g., `1e15` wei) to make price manipulation economically infeasible.
2. **Set `pricePercentageLimit` in `initialize`** to a non-zero value (e.g., `5e16` for 5%) so the price-increase guard is active from the first block. [5](#0-4) 
3. **Reject unaccounted ETH donations** by tracking expected ETH balance separately and excluding surplus from `totalETHInProtocol`, or by removing the open `receive()` fallback and only accepting ETH through named entry points (`receiveFromRewardReceiver`, `receiveFromNodeDelegator`, etc.). [12](#0-11) 
4. **Enforce a minimum rsETH output at the protocol level** — revert in `_beforeDeposit` if `rsethAmountToMint == 0`, regardless of the caller-supplied `minRSETHAmountExpected`. [9](#0-8) 

## Proof of Concept

```solidity
// Precondition: rsETHPrice = 1e18 (supply == 0, updateRSETHPrice called once)

// Step 1: Attacker deposits 1 wei ETH → receives 1 wei rsETH
lrtDepositPool.depositETH{value: 1}(0, "");
// rsETH supply = 1 wei, pool ETH balance = 1 wei

// Step 2: Attacker donates 1 ETH directly to pool
(bool ok,) = address(lrtDepositPool).call{value: 1 ether}("");
require(ok);
// pool ETH balance = 1 + 1e18 wei

// Step 3: Attacker updates price (public, pricePercentageLimit == 0 → no revert)
lrtOracle.updateRSETHPrice();
// newRsETHPrice = (1 + 1e18) * 1e18 / 1 ≈ 1e36

// Step 4: Victim deposits 1 ETH with minRSETHAmountExpected = 0
lrtDepositPool.depositETH{value: 1 ether}(0, "");  // victim tx
// rsethAmountToMint = (1e18 * 1e18) / 1e36 = 1 → rounds to 0
// Victim receives 0 rsETH, loses 1 ETH permanently

// Step 5: Attacker updates price again to reflect new pool balance
lrtOracle.updateRSETHPrice();
// totalETHInProtocol ≈ 2e18 + 1, rsethSupply = 1
// rsETHPrice ≈ 2e36

// Step 6: Attacker redeems 1 wei rsETH via withdrawal manager
// Withdrawal value = 1 * 2e36 / 1e18 = 2e18 wei ≈ 2 ETH
// Attacker recovers: 1 wei deposit + 1 ETH donation + 1 ETH victim deposit
// Net profit ≈ 1 ETH (victim's full deposit)
```
