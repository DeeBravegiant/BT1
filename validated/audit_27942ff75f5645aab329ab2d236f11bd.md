Audit Report

## Title
Missing Chainlink Staleness Validation Enables Over-Minting of rsETH - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-related return values, accepting any price regardless of age. A stale, inflated asset price flows directly into `LRTDepositPool.getRsETHAmountToMint()`, allowing an attacker to mint excess rsETH and extract ETH value from existing rsETH holders. The sibling contract `ChainlinkOracleForRSETHPoolCollateral` implements the same staleness checks the team omitted here, confirming this is an inconsistency in the codebase rather than an intentional design choice.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at lines 49–55, `getAssetPrice()` captures only the `answer` field from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The `updatedAt` (time-based staleness) and `answeredInRound` (round-completeness) return values are silently discarded. No `block.timestamp - updatedAt > threshold` check and no `answeredInRound < roundId` check exist anywhere in the call chain.

This price propagates through two critical paths:

**Path 1 — Over-minting on deposit:**
`LRTOracle.getAssetPrice()` (line 157) delegates directly to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`. `LRTDepositPool.getRsETHAmountToMint()` (lines 519–520) computes:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
A stale-high `assetPrice` inflates the numerator, minting excess rsETH.

**Path 2 — TVL inflation on price update:**
`LRTOracle._getTotalEthInProtocol()` (lines 339–343) sums `assetER * totalAssetAmt` for every supported asset using the same `getAssetPrice()`. A stale-high price overstates TVL, inflating `rsETHPrice` via `_updateRsETHPrice()`, causing subsequent depositors to receive fewer rsETH than deserved.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (lines 252–266) only triggers if the new price exceeds `highestRsethPrice` by the configured threshold. A stale price that is merely elevated (not dramatically so) bypasses this guard entirely.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 30–32 performs:
```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```
The team applied these checks in one oracle wrapper but not the other.

## Impact Explanation
**Critical — Direct theft of user funds.**

When a Chainlink feed for a supported LST goes stale at an inflated price, an attacker deposits that LST via the permissionless `depositAsset()` function and receives more rsETH than the deposit is worth. Redeeming that excess rsETH extracts ETH from the pool at the expense of all existing rsETH holders. The `depositAsset()` entry point has no staleness guard, no keeper circuit-breaker, and no admin intervention required. This constitutes direct theft of funds held by existing rsETH holders.

## Likelihood Explanation
Chainlink feeds go stale in documented real-world conditions: L2 sequencer downtime, network congestion preventing keeper transactions, or a slow price drift that never crosses the deviation threshold. The protocol supports multiple LST assets, each with its own feed, multiplying the attack surface. The attacker needs only to monitor feed `updatedAt` timestamps on-chain and call `depositAsset()` when a feed is stale — no privileged access, no flash loan, no governance capture required. The attack is repeatable until the feed resumes or the protocol is manually paused.

## Recommendation
Add staleness validation in `ChainlinkPriceOracle.getAssetPrice()`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > priceStalenessThreshold) revert StalePrice();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`priceStalenessThreshold` should be a per-asset or global configurable state variable (e.g., 3600 seconds for hourly heartbeat feeds) settable by the LRT admin.

## Proof of Concept
1. Deploy a fork of mainnet. Advance `block.timestamp` by 2 hours without triggering any Chainlink keeper update for the `stETH/ETH` feed, leaving `updatedAt` 2 hours in the past with price `1.05e18` (stETH at a premium). Real price has since dropped to `0.95e18`.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `getRsETHAmountToMint` computes `(1000e18 × 1.05e18) / rsETHPrice`. With `rsETHPrice = 1e18`, attacker receives `1050` rsETH instead of the correct `950` rsETH — a `~10.5%` over-mint.
4. Attacker redeems `1050` rsETH, extracting `~100 ETH` worth of value from the pool.
5. No revert occurs anywhere in the call chain: `ChainlinkPriceOracle.getAssetPrice()` never inspects `updatedAt`, `LRTOracle.getAssetPrice()` adds no checks, and `_beforeDeposit` only validates `minRSETHAmountExpected` (which the attacker sets to 0).

Foundry fork test outline:
```solidity
function testStaleOracleOvermint() public {
    // 1. Fork mainnet, warp 2 hours forward without Chainlink update
    vm.warp(block.timestamp + 2 hours);
    // 2. Attacker deposits stETH
    vm.prank(attacker);
    depositPool.depositAsset(stETH, 1000e18, 0, "");
    // 3. Assert attacker received > 1000e18 rsETH (should be ~950e18 at fair price)
    assertGt(rsETH.balanceOf(attacker), 1000e18);
}
```
