### Title
User-Controlled `confirmations: 0` Bypasses Bitcoin Finality Check in `verify_foreign_transaction` — (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `BitcoinRpcRequest.confirmations` field is fully user-controlled and flows without any minimum-value enforcement into the `BitcoinInspector::extract()` finality check. Setting `confirmations: 0` makes the check `0 <= actual_confirmations` trivially true for any `u64`, allowing the MPC network to sign a "verified" attestation for a Bitcoin transaction that has zero confirmations (mempool-only). A bridge consuming this signature can be tricked into releasing NEAR-side funds for a Bitcoin transaction that is never mined or is double-spent.

---

### Finding Description

`BitcoinRpcRequest` is defined with a `confirmations: BlockConfirmations` field that is a plain `u64` wrapper with no enforced minimum: [1](#0-0) 

The JSON schema confirms `minimum: 0.0`, meaning `confirmations: 0` is a valid on-wire value: [2](#0-1) 

The contract's `verify_foreign_transaction` entry point performs no validation of the `confirmations` value — it stores the request as-is and enqueues it for MPC nodes: [3](#0-2) 

Each MPC node then calls `BitcoinInspector::extract()` using the user-supplied value directly as `block_confirmations_threshold`: [4](#0-3) 

Inside the inspector, the finality gate is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [5](#0-4) 

When `block_confirmations_threshold` is `BlockConfirmations(0)`, the comparison becomes `0 <= actual_confirmations`, which is always `true` for any `u64` value returned by the RPC — including 0 (mempool transaction). The check is structurally bypassed.

**Analogy to the external report:** In the fusion-swap bug, a user inflates `estimated_dst_amount` so the condition `computed_amount > estimated` is never met, bypassing the surplus fee. Here, a user deflates `confirmations` to `0` so the condition `threshold <= actual` is always met, bypassing the finality requirement. Both are user-controlled threshold parameters that gate a security-critical check.

---

### Impact Explanation

The MPC network produces a valid threshold signature over a `ForeignTxSignPayload` that encodes the request (including `confirmations: 0`) and the extracted `BlockHash`. A bridge contract (e.g., Omnibridge inbound flow) that trusts this signature as proof of Bitcoin finality will release NEAR-side assets for a transaction that:

- Has not been mined (0 confirmations = mempool),
- Can be double-spent by the sender before it is ever included in a block, or
- May be replaced-by-fee (RBF) and never confirm.

This constitutes **forged foreign-chain verification** enabling **double-spend conditions** on the bridge — matching the High impact tier: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

The attack requires only a single unprivileged `verify_foreign_transaction` call with `confirmations: 0`. No threshold collusion, no privileged access, and no TEE bypass is needed. The attacker broadcasts a Bitcoin transaction, immediately calls `verify_foreign_transaction` before any block confirms it, and receives a valid MPC signature. The attack is cheap, repeatable, and requires no special knowledge beyond the public contract ABI.

---

### Recommendation

Enforce a protocol-level minimum on `confirmations` at two points:

1. **On-chain (contract):** Reject `BitcoinRpcRequest` with `confirmations == 0` inside `verify_foreign_transaction`, returning an `InvalidParameters` error before the request is enqueued.

2. **Node-side (inspector):** Add a guard in `BitcoinInspector::extract()` that returns `Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { … })` when `block_confirmations_threshold == 0`, as a defense-in-depth measure.

The minimum acceptable value should be defined as a named constant (e.g., `MIN_BITCOIN_CONFIRMATIONS: u64 = 1`) and ideally made configurable via the on-chain foreign-chain whitelist entry so the network can raise it without a contract upgrade.

---

### Proof of Concept

```json
// Call verify_foreign_transaction with confirmations: 0
{
  "request": {
    "request": {
      "Bitcoin": {
        "tx_id": "<mempool_tx_id_hex>",
        "confirmations": 0,
        "extractors": ["BlockHash"]
      }
    },
    "domain_id": <foreign_tx_domain_id>,
    "payload_version": 1
  }
}
```

1. Attacker broadcasts a Bitcoin transaction (e.g., a double-spend attempt) — it enters the mempool with 0 confirmations.
2. Attacker immediately calls `verify_foreign_transaction` with `confirmations: 0` and attaches 1 yoctoNEAR deposit.
3. The contract enqueues the request without validating the `confirmations` field. [6](#0-5) 
4. Each MPC node calls `BitcoinInspector::extract(tx_id, BlockConfirmations(0), …)`. The RPC returns the transaction (it is in the mempool/index), `rpc_response.confirmations` may be 0 or 1. The check `0 <= actual` passes unconditionally. [5](#0-4) 
5. Nodes sign the payload and call `respond_verify_foreign_tx`. The contract resolves the yield and returns a valid `VerifyForeignTransactionResponse` with a threshold signature.
6. Attacker presents this signature to the bridge contract to claim NEAR-side funds, then double-spends the Bitcoin transaction before it confirms.
