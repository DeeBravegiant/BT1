### Title
Stale-Vote-Satisfies-Unanimity in `vote_remove_os_measurement` Allows Below-Threshold Byzantine Participants to Remove Valid OS Measurements and Freeze Signing — (`crates/contract/src/tee/measurements.rs`, `crates/contract/src/lib.rs`)

---

### Summary

`MeasurementVotes::count_votes` counts all entries in `vote_by_account` without filtering for current participants. After a resharing that removes a participant who had already voted `Remove(M)`, their stale vote persists in storage. Because the post-resharing cleanup (`clean_non_participant_votes`) is dispatched as a **detached** NEAR promise, there is a receipt-processing window in which the stale vote is still present. Two Byzantine participants (below the signing threshold) can cast their own `Remove(M)` votes in this window, causing the inflated count to satisfy the unanimity check against the new (smaller) `total_participants`. This removes M from `allowed_measurements`, causing all nodes whose attestations were verified against M to fail `reverify_and_cleanup_participants`, which freezes signing.

---

### Finding Description

**Root cause — `count_votes` does not filter for current participants** [1](#0-0) 

`count_votes` iterates over every entry in `vote_by_account` and counts those matching the action. There is no predicate that restricts the count to accounts that are still in the current participant set.

**Unanimity check uses the post-resharing participant count** [2](#0-1) 

`total_participants` is read from `threshold_parameters.participants().len()` at call time — i.e., the new, smaller set (N-1). A stale vote from the removed participant inflates the raw count returned by `count_votes`, making it possible to satisfy `votes >= total_participants` with fewer live votes than intended.

**Cleanup is asynchronous — a window exists**

After `vote_reshared` completes and the contract transitions to Running with N-1 participants, it spawns `CLEAN_TEE_STATUS` as a detached promise: [3](#0-2) 

That promise calls `clean_non_participant_votes`: [4](#0-3) 

Because the promise is detached, other transactions — including `vote_remove_os_measurement` calls from P2 and P3 — can be processed before the cleanup receipt executes. The stale vote from the removed participant is still present in `measurement_votes` during this window.

**Downstream impact — `reverify_and_cleanup_participants`**

Once M is removed from `allowed_measurements`, `reverify_participants` re-verifies each participant's stored attestation against the updated allowlist: [5](#0-4) 

Any participant whose attestation was bound to M now fails. `verify_tee` then checks whether the surviving set still satisfies the threshold relation; if not, it sets `accept_requests = false`: [6](#0-5) 

---

### Impact Explanation

Signing and CKD operations are frozen (`accept_requests = false`) until operators manually re-add M via `vote_add_os_measurement` (requiring a threshold of the remaining participants to cooperate). The `allowed_measurements` state is corrupted without true unanimity of the current participant set. This breaks the participant-state and contract execution-flow safety invariants.

---

### Likelihood Explanation

The attack requires:
1. A participant (P1) who voted `Remove(M)` before being removed from the set — this is a normal operational step during an OS upgrade cycle.
2. Two Byzantine participants (P2, P3) out of the N-1 remaining — below the signing threshold T.
3. Both Byzantine participants submit `vote_remove_os_measurement` in the receipt-processing window between resharing completion and the `CLEAN_TEE_STATUS` cleanup receipt.

The window is narrow (one NEAR block, ~1 s) but non-zero and deterministically exploitable by a Byzantine participant who monitors the chain and submits transactions immediately after observing the resharing completion receipt. The precondition (P1 having voted `Remove(M)` before removal) arises naturally during any OS upgrade workflow.

---

### Recommendation

Apply the same participant-filtering logic used by `get_remaining_votes` directly inside `count_votes`, or — more robustly — filter stale votes eagerly at the point of counting in `vote_remove_os_measurement` before comparing against `total_participants`. Concretely, `count_votes` should accept the current `Participants` set and only count entries whose key is still a current participant:

```rust
fn count_votes(&self, action: &MeasurementVoteAction, participants: &Participants) -> u64 {
    self.vote_by_account
        .iter()
        .filter(|(id, a)| {
            *a == action
                && participants.is_participant_given_participant_id(&id.get())
        })
        .count() as u64
}
```

This mirrors the pattern already used by `get_remaining_votes` and eliminates the stale-vote window entirely, regardless of whether the async cleanup has executed.

---

### Proof of Concept

```
1. Deploy contract with N=4 participants {P1,P2,P3,P4}, T=3, measurement M in allowed_measurements.
2. As P1: call vote_remove_os_measurement(M).
   → measurement_votes = {P1: Remove(M)}
3. Reshare to N=3 {P2,P3,P4} (P1 removed). vote_reshared completes.
   → CLEAN_TEE_STATUS receipt is queued but not yet processed.
4. As P2: call vote_remove_os_measurement(M).
   → count_votes(Remove(M)) = 2 (P1 stale + P2) < total_participants=3. No removal.
5. As P3: call vote_remove_os_measurement(M).
   → count_votes(Remove(M)) = 3 (P1 stale + P2 + P3) >= total_participants=3.
   → remove_measurement(M) executes. M is gone from allowed_measurements.
6. CLEAN_TEE_STATUS executes — removes P1's stale vote, but M is already removed.
7. Call verify_tee().
   → reverify_and_cleanup_participants checks P4's attestation against allowed_measurements (no M).
   → P4 fails re-verification. participants_with_valid_attestation = {P2, P3}.
   → validate_governance_against_reconstruction fails (2 < T=3).
   → accept_requests = false. All signing and CKD frozen.
```
