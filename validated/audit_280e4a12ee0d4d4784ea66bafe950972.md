Audit Report

## Title
Missing Staleness and Price Validity Checks in `ChainlinkPriceOracle.getAssetPrice()` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `roundId`, and `answeredInRound`, performing no staleness check and no `price > 0` validation before casting `int256` to `uint256`. A stale or zero price propagates directly into the rsETH exchange rate computation via the publicly callable `LRTOracle.updateRSETHPrice()`, enabling an unprivileged attacker to set an incorrect `rsETHPrice` and profit at the expense of existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` (L49–55) fetches the Chainlink price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values except `price` are discarded. There is no `require(updatedAt >= block.timestamp - heartbeat)` staleness check and no `require(price > 0)` guard.

The stale price flows through the following confirmed call chain:
1. `ChainlinkPriceOracle.getAssetPrice()` returns a stale price.
2. `LRTOracle.getAssetPrice()` (L156–158) returns it unchanged.
3. `LRTOracle._getTotalEthInProtocol()` (L339, L343) multiplies it by `totalAssetAmt`.
4. `LRTOracle._updateRsETHPrice()` (L250) computes `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)`.
5. `rsETHPrice` is stored (L313) and used for all subsequent deposits and withdrawals.

`LRTOracle.updateRSETHPrice()` (L87–89) is `public whenNotPaused` with no access control, so any external caller can trigger this at will.

The existing `pricePercentageLimit` guard does not substitute for a staleness check: the upside guard (L256–265) only reverts for non-managers, and the downside guard (L273–281) pauses rather than reverts — neither prevents a stale price from being committed within the allowed percentage band.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L30–32) already checks `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0`, but this contract is used only for the pool collateral path, not for the core rsETH pricing path.

## Impact Explanation
**Stale deflated price → Theft of unclaimed yield (High):** If a Chainlink feed's last reported price is lower than the true current price (e.g., the asset appreciated after the last update, or the feed is lagging), `totalETHInProtocol` is understated, `rsETHPrice` is set below its true value, and an attacker who deposits at this moment receives more rsETH tokens than the true value of their deposit. This dilutes existing rsETH holders' share of the underlying assets — a direct theft of unclaimed yield from existing holders.

**Stale inflated price → Loss for depositors (Low/Medium):** If the last reported price is higher than the true current price, `rsETHPrice` is overstated and new depositors receive fewer rsETH tokens than they should.

**Zero price → Temporary freezing of funds (Medium):** A zero price for any supported asset causes `_getTotalEthInProtocol()` to undercount ETH, potentially triggering `isPriceDecreaseOffLimit` and pausing deposits and withdrawals.

The primary in-scope impact is **High — Theft of unclaimed yield** via the stale deflated price scenario.

## Likelihood Explanation
Chainlink feeds update on a heartbeat interval (e.g., 1 hour for ETH/USD on mainnet) or a deviation threshold. During low volatility, feeds may not update for the full heartbeat window, leaving a window where the price is technically stale but within the deviation band. More critically, if a feed is deprecated, a sequencer goes offline (on L2 deployments), or network congestion delays updates, the last price can persist indefinitely. Since `updateRSETHPrice()` is publicly callable with no access control, an attacker can monitor the Chainlink feed's last update timestamp and deliberately call `updateRSETHPrice()` during a stale window to set an incorrect `rsETHPrice`, then immediately deposit to profit from the mispricing. This is repeatable and requires no special privileges.

## Recommendation
1. Capture `updatedAt` and `answeredInRound` from `latestRoundData()` and validate them:
   ```solidity
   (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
       priceFeed.latestRoundData();
   require(answeredInRound >= roundId, "stale round");
   require(updatedAt != 0, "incomplete round");
   require(price > 0, "invalid price");
   require(updatedAt >= block.timestamp - maxStaleness[asset], "stale price");
   ```
2. Store a per-feed `maxStaleness` parameter (or a protocol-wide constant matching the shortest Chainlink heartbeat among all registered feeds).
3. Align `ChainlinkPriceOracle` with the stricter checks already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

## Proof of Concept
1. A supported asset's Chainlink feed (e.g., stETH/ETH) has not updated for longer than its heartbeat due to low volatility or sequencer lag. The last reported price is lower than the true current price.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (no access control, `public whenNotPaused`).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale (deflated) price with no staleness check.
4. `newRsETHPrice` is computed below its true value and stored as `rsETHPrice`.
5. Attacker immediately calls `LRTDepositPool.depositAsset()` and receives more rsETH tokens than the true value of their deposit, at the expense of existing rsETH holders.

**Foundry fork test plan:**
- Fork mainnet at a block where a supported asset's Chainlink feed has not updated recently.
- Call `LRTOracle.updateRSETHPrice()` as an unprivileged address.
- Assert that `rsETHPrice` differs from the price computed using the current true market price.
- Deposit assets and assert that the attacker's rsETH balance exceeds the fair-value equivalent.
