### Title
Proposer Deposits Permanently Locked When `do_update` Clears Competing Proposals - (File: `crates/contract/src/update.rs`)

### Summary

When `vote_update` reaches threshold and calls `do_update`, it unconditionally clears **all** pending update proposals from storage. However, the deposits paid by the proposers of those cleared proposals are never refunded. Because `UpdateEntry` does not record the proposer's account ID or the deposit amount, there is no recovery path. The deposited NEAR tokens are permanently locked in the MPC contract.

### Finding Description

`propose_update` is `#[payable]` and requires a deposit proportional to the size of the proposed update (contract binary or config), calculated as `env::storage_byte_cost() * bytes_used(update)`. [1](#0-0) 

The deposit is accepted and only the excess is refunded to the proposer. The required portion stays in the contract as storage staking. [2](#0-1) 

When `vote_update` reaches threshold, it calls `do_update`, which removes the winning entry and then **clears all remaining entries and votes without issuing any refunds**: [3](#0-2) 

The `UpdateEntry` struct stores only `update` and `bytes_used` — it does not record the proposer's `AccountId` or the deposit amount paid: [4](#0-3) 

There is no `remove_update` or `cancel_update` function that would allow a proposer to withdraw their proposal and recover their deposit before execution. The only deposit-related path in `propose_update` is the excess refund at proposal time. [5](#0-4) 

### Impact Explanation

Any participant who proposes an update that is not the one ultimately executed loses their entire storage deposit permanently. For a large contract binary (e.g., ~500 KB), the required deposit is on the order of several NEAR tokens. The contract README explicitly documents that "all pending update proposals and votes are cleared" on execution, but does not mention that deposits are not refunded. This breaks the accounting invariant that storage deposits are returned when storage is freed.

This maps to the allowed Medium impact: **"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."**

### Likelihood Explanation

The scenario is realistic and requires no privileged access. Any current participant can call `propose_update`. In a network with multiple participants, it is normal for different participants to propose competing updates (e.g., different contract versions or config changes). When one update reaches threshold, all competing proposals are silently cleared and their proposers' deposits are lost. The README itself acknowledges multiple concurrent proposals are possible. [6](#0-5) 

### Recommendation

1. Add a `proposer: AccountId` and `deposit: NearToken` field to `UpdateEntry` at proposal time.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each one.
3. Alternatively, add a `remove_update(id: UpdateId)` function that allows a proposer to cancel their own proposal and recover their deposit.

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, paying ~5 NEAR deposit. Receives `UpdateId(0)`.
2. Participant B calls `propose_update` with a different 500 KB contract binary, paying ~5 NEAR deposit. Receives `UpdateId(1)`.
3. Threshold participants call `vote_update(UpdateId(0))` until threshold is reached.
4. `do_update` executes: removes `UpdateId(0)`, then calls `self.entries.clear()` which removes `UpdateId(1)` without refunding Participant B's ~5 NEAR.
5. Participant B's deposit is permanently locked in the MPC contract with no recovery mechanism. [7](#0-6)

    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
