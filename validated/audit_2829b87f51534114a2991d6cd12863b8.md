Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables Atomic Yield Front-Running via Stale `rsETHPrice` — (`contracts/FeeReceiver.sol` / `contracts/LRTDepositPool.sol`)

## Summary

`FeeReceiver.sendFunds()` carries no access control and can be called by any external account, immediately pushing accumulated MEV/EL rewards into `LRTDepositPool` and raising TVL. Because `depositETH()` / `depositAsset()` mint rsETH using the **stored** `rsETHPrice` — which is never refreshed during the deposit flow — an attacker can atomically flush rewards and deposit in the same transaction, minting rsETH at the pre-reward price and capturing yield that belongs to existing rsETH holders.

## Finding Description

`FeeReceiver.sendFunds()` is declared `external` with no role modifier: [1](#0-0) 

This call increases `address(depositPool).balance`, which is immediately reflected in `getETHDistributionData()`: [2](#0-1) 

and therefore in `_getTotalEthInProtocol()` inside `LRTOracle`. However, `depositETH()` / `depositAsset()` compute rsETH to mint via `getRsETHAmountToMint()`, which reads the **last stored** `rsETHPrice`: [3](#0-2) 

`rsETHPrice` is only updated when `updateRSETHPrice()` is explicitly called — it is never invoked inside the deposit path: [4](#0-3) 

The full exploit path in a single transaction:
1. Call `FeeReceiver.sendFunds()` — moves accumulated rewards into the deposit pool, raising live TVL while `rsETHPrice` remains stale.
2. Call `LRTDepositPool.depositETH()` — mints rsETH at the pre-reward (lower) price.
3. When `updateRSETHPrice()` is eventually called, `previousTVL = rsethSupply × rsETHPrice` now includes the attacker's rsETH in `rsethSupply`, so the full reward `R` is still detected as a gain, but it is diluted across the enlarged supply. The attacker's rsETH is immediately worth more than they paid; the surplus is extracted from existing holders' accrued yield.

The `nonReentrant` guard on `depositETH` does not prevent this because `sendFunds()` is a separate, independent call. The `minRSETHAmountExpected` slippage parameter provides no protection since the attacker sets it to `0`. [5](#0-4) 

## Impact Explanation

**High — Theft of unclaimed yield.**

Concrete example:
- TVL = 1,000 ETH, supply = 1,000 rsETH, stored price = 1 ETH/rsETH.
- 10 ETH of MEV rewards sit in `FeeReceiver`.
- Attacker calls `sendFunds()` then `depositETH(100 ETH)` atomically.
  - Attacker receives `100 / 1 = 100 rsETH` at the stale price.
- `updateRSETHPrice()` is called later:
  - `previousTVL = 1,100 × 1 = 1,100 ETH`
  - `rewardAmount = 1,110 − 1,100 = 10 ETH`
  - `protocolFeeInETH = 1 ETH` (10% example)
  - `newRsETHPrice = 1,109 / 1,100 ≈ 1.00818 ETH/rsETH`
- Attacker's 100 rsETH is now worth ≈ 100.82 ETH — a profit of ≈ 0.82 ETH at zero risk.
- Pre-existing holders receive ≈ 8.18 ETH of yield instead of the 9 ETH they were entitled to.

Profit scales linearly with deposit size and accumulated reward size. The attack is risk-free and repeatable.

## Likelihood Explanation

Any unprivileged external account can call `sendFunds()` — no role, no whitelist, no time-lock is required. The attack requires only a proxy contract and ETH for the deposit, both standard on-chain operations. MEV rewards accumulate continuously between `updateRSETHPrice()` calls, so the opportunity recurs on every reward accumulation cycle. The attack is fully atomic and MEV-bot compatible.

## Recommendation

Apply one or more of the following mitigations:

1. **Restrict `sendFunds()`** to a trusted role (e.g., `LRTConstants.MANAGER`) so only the protocol can decide when to flush rewards into the deposit pool.
2. **Refresh the price before minting**: call `_updateRsETHPrice()` (or `updateRSETHPrice()`) at the start of `depositETH()` / `depositAsset()` so deposits always use the current TVL-inclusive price.
3. **Same-block deposit guard**: record the block number of the last `sendFunds()` call and revert deposits in the same block, preventing atomic exploitation.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IFeeReceiver { function sendFunds() external; }
interface IDepositPool { function depositETH(uint256 minRSETH, string calldata ref) external payable; }

contract YieldFrontRunner {
    IFeeReceiver public feeReceiver;
    IDepositPool public depositPool;

    constructor(address _feeReceiver, address _depositPool) {
        feeReceiver = IFeeReceiver(_feeReceiver);
        depositPool = IDepositPool(_depositPool);
    }

    function attack() external payable {
        // Step 1: flush accumulated MEV rewards into deposit pool → TVL rises, rsETHPrice stays stale
        feeReceiver.sendFunds();
        // Step 2: deposit at the stale (pre-reward) price → receive more rsETH than fair value
        depositPool.depositETH{value: msg.value}(0, "");
    }

    receive() external payable {}
}
```

**Foundry fork test plan:**
1. Fork mainnet at a block where `FeeReceiver` holds accumulated ETH.
2. Deploy `YieldFrontRunner` and fund it with ETH.
3. Record attacker's rsETH balance after `attack()`.
4. Call `LRTOracle.updateRSETHPrice()`.
5. Assert attacker's rsETH value (balance × new price) exceeds deposited ETH, and that pre-existing holder yield is reduced by the corresponding amount.
