### Title
Harvester Can Redirect Farmer Block Reward to Zero Puzzle Hash via Unvalidated `farmer_reward_address_override` - (`chia/farmer/farmer_api.py`)

### Summary

The CHIP-22 third-party harvester fee mechanism allows a connected harvester to supply a `farmer_reward_address_override` in `RespondSignatures`. The farmer accepts any non-`None` `bytes32` value — including `bytes32.zeros` — without validation, and forwards it verbatim as `farmer_puzzle_hash` in `DeclareProofOfSpace` to the full node. The full node also performs no zero-check. The result is that a malicious harvester can silently redirect the farmer's block reward (1/8 of the block reward) to the zero puzzle hash, permanently burning it.

### Finding Description

`RespondSignatures` carries an optional `farmer_reward_address_override: bytes32 | None` field introduced by CHIP-22 to let third-party harvesters claim a fee by redirecting the farmer reward to themselves. [1](#0-0) 

In `FarmerAPI._process_respond_signatures`, the farmer unconditionally replaces its own `farmer_target` with whatever the harvester supplied, as long as it is not `None`: [2](#0-1) 

There is no guard of the form `if response.farmer_reward_address_override == bytes32.zeros: return None`. The resulting `farmer_reward_address` is placed directly into `DeclareProofOfSpace.farmer_puzzle_hash`: [3](#0-2) 

The full node's `declare_proof_of_space` handler reads `request.farmer_puzzle_hash` without any zero-check and passes it as `farmer_ph` to `create_unfinished_block`: [4](#0-3) 

Block header validation (`validate_unfinished_header_block`) only enforces the genesis pre-farm hash for block 0 and verifies the pool signature for non-genesis blocks; it does not reject a zero `farmer_reward_puzzle_hash` for ordinary blocks: [5](#0-4) 

The `notify_farmer_reward_taken_by_harvester_as_fee` call that precedes the signature request only logs a warning; it does not abort the block-making flow: [6](#0-5) 

### Impact Explanation

When a harvester sets `farmer_reward_address_override = bytes32.zeros` in `RespondSignatures`, the farmer reward coin for that block is created with puzzle hash `0x0000…0000`. No private key is known for this puzzle hash; the coin is permanently unspendable (burned). The farmer loses 1/8 of the block reward for every block farmed through the malicious harvester. This constitutes unauthorized reward diversion affecting XCH.

### Likelihood Explanation

Any harvester that has an established connection to the farmer can trigger this. CHIP-22 explicitly supports third-party harvesters, so the farmer may be connected to harvesters it does not fully control. The harvester only needs to return a crafted `RespondSignatures` message with `farmer_reward_address_override = bytes32.zeros` for a valid proof of space. No cryptographic break or key compromise is required.

### Recommendation

In `FarmerAPI._process_respond_signatures`, add a zero-address guard before accepting the override:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    if response.farmer_reward_address_override == bytes32.zeros:
        self.farmer.log.error(
            "Harvester supplied zero puzzle hash as farmer_reward_address_override; ignoring."
        )
        return None
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

Optionally, apply the same guard when processing `NewProofOfSpace.farmer_reward_address_override` in `new_proof_of_space` before the signature request is sent.

### Proof of Concept

1. A third-party harvester connects to the farmer (standard CHIP-22 setup).
2. The harvester finds a valid proof of space and sends `NewProofOfSpace` with `farmer_reward_address_override = bytes32(b"\x00" * 32)`.
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` (logs only) and sends `RequestSignatures` to the harvester.
4. The harvester returns `RespondSignatures` with `farmer_reward_address_override = bytes32(b"\x00" * 32)`.
5. `_process_respond_signatures` sets `farmer_reward_address = bytes32.zeros` and returns a `DeclareProofOfSpace` with `farmer_puzzle_hash = bytes32.zeros`.
6. The full node creates an unfinished block with `farmer_ph = bytes32.zeros` and sends `RequestSignedValues` back to the farmer.
7. After the timelord finishes the block, the farmer reward coin is created at puzzle hash `0x0000…0000` — permanently unspendable. [2](#0-1) [1](#0-0) [4](#0-3)
