Audit Report

## Title
Stale Cross-Chain rsETH/ETH Rate in `CrossChainRateReceiver` Causes Inflated wrsETH Minting in `RSETHPoolV3` Token Deposits — (File: `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the last LZ-delivered `rate` with no staleness check, while `lastUpdated` is stored but never consulted. `RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)` divides a fresh Chainlink collateral rate by this potentially stale rsETH/ETH rate, yielding an inflated wrsETH mint amount. Any depositor of a supported non-ETH token during a staleness window receives more wrsETH than their collateral backs at the true current rate, degrading the pool's backing ratio.

## Finding Description

`CrossChainRateReceiver.getRate()` unconditionally returns the stored `rate`: [1](#0-0) 

`lastUpdated` is written in `lzReceive` but is never read in `getRate()`: [2](#0-1) 

`RSETHPoolV3.getRate()` delegates directly to this oracle: [3](#0-2) 

`viewSwapRsETHAmountAndFee(amount, token)` pairs this stale rsETH rate with a fresh Chainlink collateral rate: [4](#0-3) 

Because rsETH is yield-bearing and its ETH value monotonically increases, the stored `rate` is almost always below the true current rate between LZ updates. A stale (lower) denominator inflates `rsETHAmount`. The deposit function immediately mints this inflated amount: [5](#0-4) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` enforces freshness via `answeredInRound < roundID` and `timestamp == 0` guards: [6](#0-5) 

This asymmetry — fresh numerator, stale denominator — is the root cause. Note: the `limitDailyMint` modifier does compute `rsETHAmount` via `viewSwapRsETHAmountAndFee` (not the raw token amount as the submission claims), but since it uses the same stale rate, the inflated amount passes the daily limit check and does not prevent over-minting: [7](#0-6) 

## Impact Explanation

**Low — contract fails to deliver promised returns, but doesn't lose value.**

When `rsETHToETHrate_stale < rsETHToETHrate_true`, the pool mints `rsETHAmount_minted > rsETHAmount_correct`. The wrsETH supply is inflated relative to the pool's actual collateral, degrading the backing ratio for all wrsETH holders. The magnitude per deposit is small (rsETH yield accrual is ~a few percent per year, and staleness windows are hours), but the condition is persistent and affects every non-ETH token deposit during any staleness window.

## Likelihood Explanation

LZ rate updates are event-driven and not continuous; gaps of hours are normal operational behavior. rsETH accrues yield every block, so the stored rate is almost always slightly below the true rate. Any unprivileged depositor calling `deposit(token, amount, referralId)` with any supported non-ETH token (e.g., wstETH) during a staleness window triggers the path with no special preconditions. The condition is effectively always present to some degree and requires no attacker capability beyond making a normal deposit.

## Recommendation

Add a staleness threshold check in `CrossChainRateReceiver.getRate()` that reverts if `block.timestamp - lastUpdated` exceeds an acceptable heartbeat, mirroring the pattern in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate too stale");
    return rate;
}
```

`MAX_RATE_AGE` should be set to match the expected LZ update frequency plus a safety buffer, and should be configurable by the owner.

## Proof of Concept

1. Fork the target L2 at a block where `CrossChainRateReceiver.lastUpdated` is 12+ hours old.
2. Advance `block.timestamp` by 12 hours without triggering a new LZ message (so `rate` remains at the old, lower value).
3. Confirm `ChainlinkOracleForRSETHPoolCollateral.getRate()` returns a fresh wstETH/ETH price.
4. Call `RSETHPoolV3.deposit(wstETH, 1e18, "")` as a normal depositor.
5. Record `rsETHAmount` minted (from the `SwapOccurred` event or return value of `viewSwapRsETHAmountAndFee`).
6. Compute the correct amount: `1e18 * wstETHRate / trueRsETHRate` using the true current rsETH/ETH rate from L1 (e.g., via KelpDAO's `LRTOracle.rsETHPrice()`).
7. Assert `rsETHAmount_minted > rsETHAmount_correct` — the pool over-minted wrsETH relative to the true backing.
