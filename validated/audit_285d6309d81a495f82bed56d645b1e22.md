Looking at the exact code path, I need to verify: (1) how the solution is fetched and validated, (2) what `singleton_removed` does with it, and (3) whether the root extracted from the memo is cross-checked against anything on-chain.

### Title
Peer-Supplied Unvalidated Solution Allows Forged DL Singleton Root Injection — (`chia/data_layer/data_layer_wallet.py`, `chia/wallet/util/wallet_sync_utils.py`)

---

### Summary

`singleton_removed` extracts the Data Layer store root from the memo field of a CREATE_COIN condition produced by running the spent coin's puzzle against a **peer-supplied, unvalidated solution**. Because `fetch_coin_spend` only validates the puzzle hash — not the solution — a malicious peer can return the correct puzzle reveal (which passes the hash check) paired with a crafted solution that causes the CLVM to emit a CREATE_COIN with an attacker-chosen root in `condition[3][1]`. The wallet records this forged root as the confirmed DL store root with no further cross-check.

---

### Finding Description

**Step 1 — Entry point in `_add_coin_states`**

When a spent DATA_LAYER coin state arrives from a peer, the wallet fetches the spend and immediately passes it to `singleton_removed`: [1](#0-0) 

**Step 2 — `fetch_coin_spend` validates puzzle hash but not solution**

`fetch_coin_spend` checks that the returned puzzle's tree-hash matches the coin's on-chain `puzzle_hash`, and that the coin name matches. It performs **no validation of the solution**: [2](#0-1) 

The solution field is accepted verbatim from the peer.

**Step 3 — `singleton_removed` runs the puzzle with the peer-supplied solution and trusts the memo**

```python

for condition in conditions:
    if condition[0] == ConditionOpcode.CREATE_COIN and int.from_bytes(condition[2], "big") % 2 == 1:

        ...
        root = bytes32(condition[3][1])          # ← attacker-controlled

``` [3](#0-2) 

The extracted `root` is then written directly into `dl_store` as the confirmed root for the next singleton generation: [4](#0-3) 

There is **no check** that `root` is consistent with `full_puzzle_hash` (i.e., that `create_host_fullpuz(inner_puzzle_hash, root, launcher_id).get_tree_hash() == full_puzzle_hash`), and no check that the solution matches what was actually included in the block.

**Step 4 — Why a crafted solution succeeds at the CLVM level**

The DL singleton puzzle is `SINGLETON_TOP_LAYER_MOD` wrapping `NFT_STATE_LAYER_MOD` wrapping a standard `p2_delegated_puzzle_or_hidden_puzzle`. The standard inner puzzle requires an `AGG_SIG_ME` over the delegated puzzle hash, but `run_with_cost` in the wallet **does not validate BLS signatures** — it only executes CLVM and returns conditions. An attacker can supply:

- A valid lineage proof (publicly known from the blockchain)
- A crafted delegated puzzle that outputs `(CREATE_COIN <any_puz_hash> <odd_amount> (<launcher_id> <forged_root> <any_inner_puz_hash>))`

The CLVM execution completes without error, the AGG_SIG_ME condition appears in the output but is never checked by the wallet, and `root = bytes32(condition[3][1])` is set to the attacker's chosen value. [5](#0-4) 

---

### Impact Explanation

- **Corrupted DL store root**: The wallet's `dl_store` records an attacker-chosen root as the confirmed state of the singleton. Any subsequent `create_update_state_spend` call will use this wrong root as the current state, producing transactions that are invalid on-chain and permanently breaking the wallet's ability to update the store.
- **Broken DL offer verification**: `finish_graftroot_solutions` and `verify_offer` use the stored root to validate proofs of inclusion. A forged root causes valid proofs to be rejected and can be used to manipulate offer settlement logic.
- **Forged child coin tracking**: `full_puzzle_hash` (also from the memo) is equally unvalidated, so the attacker can make the wallet subscribe to and track a completely fabricated child coin ID.

This meets the **High** bar: "Corruption of… Data Layer root/store state with direct security impact."

---

### Likelihood Explanation

The attacker must be a peer the wallet connects to (any full node the wallet syncs from). The targeted DL singleton must be tracked by the wallet and must be spent on-chain (by anyone). Both conditions are routinely satisfied in normal operation. No key material, admin access, or cryptographic break is required.

---

### Recommendation

After `run_with_cost` produces conditions, cross-validate the extracted `root` and `inner_puzzle_hash` against the child coin's actual puzzle hash:

```python
expected_full_puz_hash = create_host_fullpuz(
    inner_puzzle_hash, root, singleton_record.launcher_id
).get_tree_hash_precalc(inner_puzzle_hash)
if full_puzzle_hash != expected_full_puz_hash:
    self.log.warning("Memo root/inner_puz_hash inconsistent with child puzzle hash; ignoring spend")
    return
```

Additionally, validate the child coin's existence on-chain (confirm `new_singleton.name()` is a known addition at `height`) before writing to `dl_store`. The solution itself should ideally be validated against the block's generator, or at minimum the derived child coin should be cross-checked against the peer-reported children.

        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )

def match_dl_singleton(puzzle: Program | SerializedProgram) -> tuple[bool, Iterator[Program]]:
    """
    Given a puzzle test if it's a CAT and, if it is, return the curried arguments
    """
    mod, singleton_curried_args = uncurry(puzzle)
    if mod == SINGLETON_TOP_LAYER_MOD:
        mod, dl_curried_args = uncurry(singleton_curried_args.at("rf"))
        if mod == NFT_STATE_LAYER_MOD and dl_curried_args.at("rrf") == ACS_MU_PH:
            launcher_id = singleton_curried_args.at("frf")
            root = dl_curried_args.at("rff")
            innerpuz = dl_curried_args.at("rrrf")
            return True, iter((innerpuz, root, launcher_id))

    return False, iter(())
