### Title
`DelegateAction` Signed Payload Lacks Network Binding, Enabling Cross-Network Replay - (File: `core/primitives/src/action/delegate.rs`)

### Summary

The `SignedDelegateAction` (NEP-366 meta-transaction) signature covers a payload that contains no chain ID or any other network-specific identifier. A malicious relayer who receives a user's signed `DelegateAction` on one NEAR network (e.g., mainnet) can replay it verbatim on another NEAR network (e.g., testnet) where the same key exists and the nonce is valid, executing unauthorized actions and draining the user's balance on the target network.

### Finding Description

`SignedDelegateAction::verify()` computes the signed hash as:

```rust
pub fn verify(&self) -> bool {
    let delegate_action = &self.delegate_action;
    let hash = delegate_action.get_nep461_hash();
    let public_key = &delegate_action.public_key;
    self.signature.verify(hash.as_ref(), public_key)
}
``` [1](#0-0) 

`get_nep461_hash()` serializes a `SignableMessage` containing only a NEP-366 discriminant and the `DelegateAction` body:

```rust
let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
let bytes = borsh::to_vec(&signable).expect("failed to serialize");
hash(&bytes)
``` [2](#0-1) 

The `DelegateAction` struct contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — but **no chain ID and no network-specific identifier**: [3](#0-2) 

The `SignableMessage` discriminant is a fixed NEP number (`MIN_ON_CHAIN_DISCRIMINANT + 366`), identical on every NEAR network: [4](#0-3) 

By contrast, NEAR native transactions include a `block_hash` field that is network-specific (different genesis blocks produce different hashes), providing network binding. `DelegateAction` has only `max_block_height`, which is a bare integer with no network binding — block heights on mainnet and testnet are not synchronized and can overlap.

The `DelegateActionV2` variant (`VersionedDelegateActionPayload::V2`) has the same structural omission: [5](#0-4) 

Its hash is computed identically, with only a different NEP discriminant (`NEP_611_GAS_KEYS`): [6](#0-5) 

Compare this to the wallet contract's Ethereum-style transaction path, which **does** enforce chain ID:

```rust
if tx.chain_id != Some(CHAIN_ID) {
    return Err(Error::Relayer(RelayerError::InvalidChainId));
}
``` [7](#0-6) 

The wallet contract's `CHAIN_ID` is baked in at compile time per network (397 mainnet, 398 testnet, 399 localnet): [8](#0-7) 

No equivalent protection exists for `DelegateAction`.

### Impact Explanation

A user signs a `DelegateAction` on NEAR mainnet (e.g., `Transfer { deposit: 10 NEAR }` to `bob.near`) and hands it to a relayer. The relayer submits the same signed bytes on NEAR testnet. If the user holds the same key on testnet (common when the same seed phrase is used) and the access-key nonce on testnet is one less than the `DelegateAction`'s nonce, the runtime accepts the action and debits the user's testnet balance. The corrupted protocol value is the user's **account balance** on the target network, modified without the user's intent for that network.

### Likelihood Explanation

- Users routinely derive the same key pair on mainnet and testnet from the same seed phrase.
- The `max_block_height` bound does not prevent cross-network replay because block heights on different NEAR networks are independent integers.
- A relayer is an unprivileged actor: no on-chain role or stake is required to call `broadcast_tx_async` on any NEAR RPC endpoint. Any party who receives a signed `DelegateAction` (e.g., a relayer service the user contacted) can submit it on a different network without any additional privilege.
- The nonce constraint is the only practical barrier, and it is easily satisfied when the user's key is freshly added on the target network.

### Recommendation

Include the network's chain ID (or a network-specific genesis hash) in the `DelegateAction` and `DelegateActionV2` signed payload, analogous to how the wallet contract enforces `CHAIN_ID` for Ethereum-style transactions. This requires a new `DelegateAction` version (e.g., `DelegateActionV3`) with a `chain_id: String` field included before hashing, gated behind a protocol feature flag so old signatures remain valid until the cutover.

### Proof of Concept

1. User on NEAR mainnet creates and signs:
   ```
   DelegateAction {
     sender_id: "alice.near",
     receiver_id: "bob.near",
     actions: [Transfer { deposit: 5_000_000_000_000_000_000_000_000 }],
     nonce: 1,
     max_block_height: 200_000_000,  // far future, valid on both networks
     public_key: <alice's key>,
   }
   ```
2. User hands `SignedDelegateAction` to a relayer service.
3. Malicious relayer wraps the identical `SignedDelegateAction` in a NEAR testnet transaction (the outer transaction's `block_hash` is testnet-specific, but the inner delegate signature is not) and submits it via testnet RPC.
4. The runtime calls `SignedDelegateAction::verify()`, which recomputes `get_nep461_hash()` — the hash is identical on testnet because no network identifier is in the payload.
5. Signature verification passes; the runtime checks alice's testnet access-key nonce (0 < 1 ✓) and `max_block_height` (testnet height < 200_000_000 ✓).
6. `Transfer` executes, draining alice's testnet balance to bob without alice's authorization for the testnet network.

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}

```rust
impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
}
