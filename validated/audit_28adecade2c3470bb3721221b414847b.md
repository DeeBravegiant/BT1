Audit Report

## Title
Fee Baseline Uses `rsETHPrice` Instead of `highestRsethPrice`, Charging Fees on Slashing Recovery — (File: `contracts/LRTOracle.sol`)

## Summary
In `_updateRsETHPrice()`, `previousTVL` is computed using `rsETHPrice` (the last stored price) rather than `highestRsethPrice` (the all-time high). After a slashing event that is small enough to not trigger the downside-protection pause, `rsETHPrice` is updated to the post-slashing lower value while `highestRsethPrice` remains at the pre-slashing peak. Any subsequent price recovery is then treated as new yield and subjected to protocol fees, even though the price has not yet returned to its previous peak. rsETH holders who already absorbed the slashing loss are further penalized by having their recovery gains partially extracted as protocol fees.

## Finding Description
`_updateRsETHPrice()` computes the fee baseline at line 234:

```solidity
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);
```

The fee is charged whenever `totalETHInProtocol > previousTVL` (line 244). `highestRsethPrice` is tracked and updated at lines 293–296 but is never consulted for the fee decision.

**Exploit path:**

1. `rsETHPrice = highestRsethPrice = 1.1e18` (normal state).
2. A slashing event reduces `totalETHInProtocol`. `updateRSETHPrice()` is called (public, no access control). The computed `newRsETHPrice = 1.0e18`.
3. The downside-protection block (lines 270–281) checks `diff > pricePercentageLimit.mulWad(highestRsethPrice)`. If the drop is within `pricePercentageLimit` (or `pricePercentageLimit == 0`), no pause is triggered and execution continues.
4. `rsETHPrice` is updated to `1.0e18` (line 313); `highestRsethPrice` stays at `1.1e18`.
5. Yield accrues; `totalETHInProtocol` rises to `rsethSupply × 1.05e18`.
6. Any caller invokes `updateRSETHPrice()`:
   - `previousTVL = rsethSupply × 1.0e18`
   - `rewardAmount = rsethSupply × 0.05e18` — treated as new yield
   - `protocolFeeInETH = rsethSupply × 0.05e18 × feeBPS / 10_000`
   - Fee rsETH is minted to treasury (line 306).

**Why existing checks fail:** The downside-protection logic (lines 270–291) correctly uses `highestRsethPrice` to decide whether to pause, but this guard only fires for drops exceeding `pricePercentageLimit`. For smaller slashing events, `rsETHPrice` is written to the post-slashing value and the fee baseline is permanently lowered. The `highestRsethPrice >= rsETHPrice` invariant holds throughout (since `highestRsethPrice` is only ever increased), confirming the two variables diverge after any sub-threshold slashing event.

**Contrast with large-slashing behavior:** When the drop exceeds `pricePercentageLimit`, the function returns early at line 281 without updating `rsETHPrice`. After unpausing, `previousTVL` uses the pre-slashing `rsETHPrice`, so no fee is charged on recovery — the correct behavior. The small-slashing path lacks this protection, creating an inconsistency.

## Impact Explanation
rsETH holders who suffered a slashing loss have their recovery gains partially extracted as protocol fees. With a 10% fee rate: price drops 1.1 → 1.0 (slashing absorbed by holders), then recovers to 1.05. `rewardAmount = rsethSupply × 0.05e18`; `protocolFeeInETH = rsethSupply × 0.005e18` minted to treasury. Holders, who already lost 0.1 ETH/rsETH, now lose an additional fee on their 0.05 ETH/rsETH recovery and can never fully recover their slashing loss through normal yield accrual. This is a concrete, quantifiable extraction of value that belongs to rsETH holders — **High: Theft of unclaimed yield**.

## Likelihood Explanation
EigenLayer slashing is a realistic and anticipated event for a restaking protocol; the protocol already tracks `highestRsethPrice` specifically to handle price-drop scenarios. Any slashing event small enough to fall within `pricePercentageLimit` (or when `pricePercentageLimit == 0`, any slashing at all) followed by partial recovery triggers this without any special conditions. `updateRSETHPrice()` is public and callable by any address when the oracle is unpaused, requiring no privileged access. **Likelihood: Medium.**

## Recommendation
Use `highestRsethPrice` as the fee baseline so that fees are only charged on genuine new yield above the previous peak:

```solidity
// Before (buggy):
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);

// After (fixed):
// highestRsethPrice >= rsETHPrice is invariant, so this simplifies directly:
uint256 previousTVL = rsethSupply.mulWad(highestRsethPrice);
```

This aligns the fee logic with the downside-protection logic, which already treats `highestRsethPrice` as the authoritative reference price.

## Proof of Concept
Foundry fork/unit test sequence:

1. Deploy with `rsETHPrice = highestRsethPrice = 1.1e18`, `protocolFeeInBPS = 1000` (10%), `pricePercentageLimit = 0.05e18` (5%).
2. Simulate a 9% slashing event (within the 5% limit — use `pricePercentageLimit = 0.10e18` to keep it sub-threshold): manipulate mock asset oracle so `_getTotalEthInProtocol()` returns `rsethSupply × 1.0e18`.
3. Call `updateRSETHPrice()` as an unprivileged address. Assert: `rsETHPrice == 1.0e18`, `highestRsethPrice == 1.1e18`, no pause triggered.
4. Advance mock oracle to `rsethSupply × 1.05e18`.
5. Call `updateRSETHPrice()` again. Assert: `FeeMinted` event emitted with non-zero amount; treasury rsETH balance > 0.
6. Repeat with `previousTVL = rsethSupply.mulWad(highestRsethPrice)` (patched): assert `FeeMinted` is NOT emitted (since `1.05e18 < 1.1e18` baseline), confirming zero fee on recovery below the previous peak.
