### Title
`respond_ckd` Silently Skips Cryptographic Output Verification for `AppPublicKey` Variant, Allowing a Single Byzantine Participant to Deliver an Attacker-Controlled Derived Key - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs a pairing-based cryptographic output check only for the `AppPublicKeyPV` variant of CKD requests. For the legacy `AppPublicKey` variant, the match arm is an empty no-op (`{}`), meaning any single attested participant can call `respond_ckd` with a fully fabricated `CKDResponse` and the contract will accept it unconditionally, bypassing the threshold requirement and delivering an attacker-controlled derived key to the user.

---

### Finding Description

In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`: [2](#0-1) 

This pairing check cryptographically proves the response was computed using the actual BLS master secret key `msk`, which requires threshold participation to produce. A single node cannot forge a response that satisfies this equation without knowing `msk`.

For `AppPublicKey`, the contract has no `pk2` (G2 point) to run the pairing check, so the arm is empty. The `CKDResponse` fields `big_c` and `big_y` are accepted as-is and immediately delivered to the user via `resolve_yields_for`: [3](#0-2) 

The `AppPublicKey` variant is the legacy default format, documented as "privately verifiable": [4](#0-3) 

It is the format used in the CLI example and the majority of tests: [5](#0-4) 

---

### Impact Explanation

The CKD protocol is designed so that deriving `hash_point * msk` requires threshold-many participants to cooperate. For `AppPublicKeyPV`, the contract enforces this cryptographically on-chain. For `AppPublicKey`, it does not.

A single Byzantine attested participant can:

1. Observe a pending `AppPublicKey` CKD request in the contract.
2. Call `respond_ckd` with a fabricated response: `big_y = G1_identity` (the identity point, which is a valid G1 point accepted by `bls12381_p1_decompress`) and `big_c = target_point` (any G1 point whose discrete log the attacker knows).
3. The contract's empty match arm accepts the response without any check.
4. The user computes their derived secret as `big_c − big_y * a = target_point − 0 = target_point`.
5. The attacker knows `target_point` and therefore the user's derived private key.

The user's derived key is now fully known to the attacker. Any data encrypted with that key can be decrypted; any signature scheme using that key can be forged. This is a bypass of the threshold-signature requirement for CKD: instead of requiring t+1 participants, a single participant suffices to corrupt the output.

---

### Likelihood Explanation

The `AppPublicKey` variant is the legacy default and is actively used in production flows (CLI, sandbox tests, e2e tests). Any single attested MPC participant who is Byzantine (below the threshold) can race the legitimate leader's `respond_ckd` call. Since `resolve_yields_for` drains the queue on the first successful call, the attacker only needs to submit their fabricated transaction before the honest leader. This is a realistic threat in the MPC model, where individual participants below the threshold are explicitly considered potentially adversarial.

---

### Recommendation

1. **Deprecate `AppPublicKey`** and require all new CKD requests to use `AppPublicKeyPV`, which enables on-chain cryptographic verification of the response.
2. If `AppPublicKey` must be retained for backward compatibility, add prominent documentation that it provides **no threshold enforcement at the contract level** and that users must verify the response off-chain before trusting the derived key.
3. Consider whether the contract should reject `AppPublicKey` requests entirely when `accept_requests` is gated on TEE validation, since the TEE is the only protection for this variant.

---

### Proof of Concept

```
1. User calls request_app_private_key({
       derivation_path: "mykey",
       app_public_key: "bls12381g1:<pk1>",   // AppPublicKey variant
       domain_id: 2
   })
   → Contract stores CKDRequest in pending_ckd_requests.

2. Attacker (single attested participant, below threshold) calls respond_ckd({
       request: <matching CKDRequest>,
       response: {
           big_y: bls12381g1:<G1_identity>,   // identity point, passes decompression
           big_c: bls12381g1:<target_point>   // attacker-chosen point
       }
   })
   → Contract hits: match AppPublicKey(_) => {}   // no check
   → resolve_yields_for delivers fabricated response to user.

3. User computes derived secret:
       secret = big_c - big_y * a
              = target_point - identity * a
              = target_point
   → Attacker knows target_point → user's derived key is compromised.
```

The fabricated response passes all contract checks because `assert_caller_is_attested_participant_and_protocol_active()` only verifies the caller's identity, not the mathematical validity of the response, and the `AppPublicKey` match arm performs no cryptographic check. [6](#0-5) [7](#0-6)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
