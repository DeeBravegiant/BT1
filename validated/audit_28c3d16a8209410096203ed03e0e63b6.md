### Title
Script-Hash Staking Credentials Bypass DRep Delegation Check for Withdrawals - (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

### Summary

`validateWithdrawalsDelegated` uses `credKeyHash` to extract staking credentials from withdrawal addresses. `credKeyHash` returns `Nothing` for `ScriptHashObj` credentials, so the list comprehension silently drops all script-hash staking credentials from the DRep delegation check. Any withdrawal whose reward account is controlled by a script hash bypasses the Conway-era requirement that staking credentials must be delegated to a DRep before rewards can be withdrawn.

### Finding Description

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`, the function `validateWithdrawalsDelegated` is responsible for enforcing the Conway governance rule that a staking credential must be delegated to a DRep before its rewards can be withdrawn:

```haskell
validateWithdrawalsDelegated accounts tx =
  let wdrls = unWithdrawals $ tx ^. bodyTxL . withdrawalsTxBodyL
      wdrlsKeyHashes =
        [ kh | (ra, _) <- Map.toList wdrls
             , Just kh <- [credKeyHash $ ra ^. accountAddressCredentialL]
        ]
      isNotDRepDelegated keyHash = isNothing $ do
        accountState <- lookupAccountState (KeyHashObj keyHash) accounts
        accountState ^. dRepDelegationAccountStateL
      nonExistentDelegations = filter isNotDRepDelegated wdrlsKeyHashes
   in failureOnNonEmpty nonExistentDelegations ConwayWdrlNotDelegatedToDRep
```

`credKeyHash` is defined as:

```haskell
credKeyHash :: Credential r -> Maybe (KeyHash r)
credKeyHash = \case
  KeyHashObj hk -> Just hk
  ScriptHashObj _ -> Nothing
```

The list comprehension `[ kh | ..., Just kh <- [credKeyHash ...] ]` silently discards every withdrawal whose reward-account credential is a `ScriptHashObj`. Those credentials are never passed to `isNotDRepDelegated`, so the DRep delegation check is never executed for them. The same delegated call path exists in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs` at line 197, which calls `Conway.validateWithdrawalsDelegated` directly.

The `Credential` type supports both `KeyHashObj` and `ScriptHashObj` as valid staking credentials, and the CDDL specification confirms `credential = [0, addr_keyhash // 1, script_hash]`. Script-hash staking credentials are a first-class, fully supported credential type throughout the ledger.

The analog to the external report is exact: the external report uses `address` (too restrictive) instead of `string` as the mapping key type, causing non-EVM addresses to bypass the whitelist. Here, `credKeyHash` (too restrictive, only extracts `KeyHashObj`) is used instead of operating on the full `Credential Staking`, causing `ScriptHashObj` credentials to bypass the DRep delegation enforcement.

### Impact Explanation

A transaction sender who controls a script-hash staking credential can withdraw accumulated rewards without that credential ever being delegated to a DRep. This violates the Conway governance design parameter that ties reward withdrawal to governance participation. The withdrawal modifies the reward/withdrawal accounting outside the intended design parameters.

**Allowed impact matched**: *Medium — Attacker-controlled transactions… modify… withdrawals outside design parameters.*

### Likelihood Explanation

Script-hash staking credentials are a standard, protocol-supported credential type. Any user can register a staking credential backed by a native script or Plutus script. No privileged access, leaked key, or consensus majority is required. The bypass is automatic: submitting a withdrawal transaction with a script-hash reward account simply skips the DRep delegation check with no error.

### Recommendation

Replace the `credKeyHash`-based extraction with a check over the full `Credential Staking` type, so that both `KeyHashObj` and `ScriptHashObj` credentials are subject to the DRep delegation requirement:

```haskell
validateWithdrawalsDelegated accounts tx =
  let wdrls = unWithdrawals $ tx ^. bodyTxL . withdrawalsTxBodyL
      wdrlsCreds =
        [ cred | (ra, _) <- Map.toList wdrls
               , let cred = ra ^. accountAddressCredentialL
        ]
      isNotDRepDelegated cred = isNothing $ do
        accountState <- lookupAccountState cred accounts
        accountState ^. dRepDelegationAccountStateL
      nonExistentDelegations = filter isNotDRepDelegated wdrlsCreds
   in failureOnNonEmpty nonExistentDelegations ConwayWdrlNotDelegatedToDRep
```

The `ConwayWdrlNotDelegatedToDRep` error constructor would also need to be widened from `NonEmpty (KeyHash Staking)` to `NonEmpty (Credential Staking)` to carry the full credential in the failure report.

### Proof of Concept

1. Register a staking credential using a script hash: `RegTxCert (ScriptHashObj myScriptHash)`.
2. Accumulate rewards to the corresponding reward account.
3. Do **not** submit a DRep delegation certificate for `ScriptHashObj myScriptHash`.
4. Submit a transaction with a withdrawal from that reward account.
5. `validateWithdrawalsDelegated` builds `wdrlsKeyHashes` via `credKeyHash`; `credKeyHash (ScriptHashObj myScriptHash)` returns `Nothing`, so `myScriptHash` is excluded from the list.
6. `nonExistentDelegations` is empty; `failureOnNonEmpty` does not fire.
7. The withdrawal succeeds despite the credential having no DRep delegation, bypassing the Conway governance requirement. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
