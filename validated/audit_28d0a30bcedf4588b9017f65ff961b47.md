Audit Report

## Title
Stale Cross-Chain Rate in `CrossChainRateReceiver.getRate()` Causes Over-Minting of agETH — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/agETH/AGETHPoolV3.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the last stored `rate` with no check against `lastUpdated`. `AGETHPoolV3.deposit()` calls this function unconditionally via `viewSwapAgETHAmountAndFee()`. When the LayerZero relayer is delayed or `updateRate()` is not called on the provider side, the stale (lower) rate causes the pool to mint more agETH per ETH than the current L1 backing justifies, diluting existing agETH holders.

## Finding Description

`CrossChainRateReceiver` records `lastUpdated` on every `lzReceive` call: [1](#0-0) 

However, `getRate()` never consults `lastUpdated` — it unconditionally returns the stored `rate`: [2](#0-1) 

`AGETHPoolV3.getRate()` delegates directly to this oracle: [3](#0-2) 

`deposit()` calls `viewSwapAgETHAmountAndFee()`, which computes the mint amount using the potentially stale rate: [4](#0-3) 

The formula `agETHAmount = amountAfterFee * 1e18 / agETHToETHrate` means a stale lower rate (e.g., 1.01e18 instead of current 1.05e18) produces a larger `agETHAmount` (≈0.9901e18 vs. the correct ≈0.9524e18). The minted agETH is not backed by sufficient ETH at the current L1 rate, breaking the collateral invariant. There is no `maxStaleness` guard anywhere in the receiver or pool.

## Impact Explanation

Every depositor who calls `deposit()` while the receiver holds a stale rate receives more agETH than the current agETH/ETH exchange rate backs. The ETH is still held by the pool, but the excess agETH supply dilutes existing holders' claims. This matches **Low — contract fails to deliver promised returns, but doesn't lose value**: the ETH collateral is intact, but the agETH/ETH backing ratio is degraded for existing holders.

## Likelihood Explanation

LayerZero relayer delays are a documented operational reality (network congestion, gas spikes, relayer downtime). The `updateRate()` call on the provider side must be triggered externally; if no one calls it, the receiver rate can be arbitrarily old. No special privilege is required — any depositor passively benefits by depositing during a stale window. The `AGETHRateReceiver` constructor confirms this is the live oracle path used by the pool: [5](#0-4) 

## Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(lastUpdated != 0 && block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes `deposit()` to revert rather than mint at a stale rate, protecting the protocol's collateral invariant.

## Proof of Concept

```solidity
// Foundry fork test
// 1. Deploy AGETHRateReceiver with a mock LZ endpoint.
// 2. Call lzReceive with rate = 1.01e18 (agETH slightly above peg).
// 3. vm.warp(block.timestamp + 7 days) — no further lzReceive.
// 4. On L1, agETH rate has appreciated to 1.05e18 (currentL1Rate).
// 5. Call AGETHPoolV3.deposit{value: 1 ether}().
// 6. Assert:
//    agETHMinted == 1e18 * 1e18 / 1.01e18  ≈ 0.9901e18   (stale, over-minted)
//    correctAmount == 1e18 * 1e18 / 1.05e18 ≈ 0.9524e18
//    agETHMinted > correctAmount  // invariant broken — no privileged action required
```
