Audit Report

## Title
Unbounded `rateReceivers` Array Causes Unbounded Gas Consumption in `updateRate()` — (`contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary

`MultiChainRateProvider.updateRate()` is a permissionless `external payable` function that iterates over the entire `rateReceivers` array, executing two external LayerZero calls per entry (`estimateFees` + `send`). Because `addRateReceiver` enforces no upper-bound cap on the array length, gas consumption grows linearly and without bound as the protocol deploys to additional chains. Once the cumulative gas cost of a single `updateRate()` call exceeds the Ethereum block gas limit (~30 M gas), the function becomes permanently uncallable by anyone, bricking cross-chain rate delivery.

## Finding Description

`updateRate()` carries no access control — it is callable by any address: [1](#0-0) 

Inside the function, a `for` loop iterates over every element of `rateReceivers` with no bound: [2](#0-1) 

Each iteration performs two external calls to the LayerZero endpoint — `estimateFees` and `send` — both of which are gas-heavy cross-chain operations consuming on the order of 100 k–200 k gas each: [3](#0-2) 

`addRateReceiver` pushes to the array with no length check: [4](#0-3) 

The only existing guard on `updateRate()` is `nonReentrant`, which does not limit gas consumption. There is no pagination, no batch size cap, and no maximum receiver count. Gas cost is strictly O(n) in the number of registered receivers, with no ceiling enforced anywhere in the contract.

## Impact Explanation

**Medium — Unbounded gas consumption.** Gas consumption in `updateRate()` grows linearly and without bound with the number of registered receivers. At approximately 150–200 receivers (each `send` costing ~150–200 k gas), a single call exceeds Ethereum's ~30 M block gas limit. Once this threshold is crossed, `updateRate()` is permanently uncallable by any account — including the owner — without removing receivers (breaking service to those chains) or upgrading the contract. This matches the allowed impact: **Medium. Unbounded gas consumption.**

## Likelihood Explanation

The precondition is legitimate protocol operation: the owner calls `addRateReceiver` for each new chain deployment. No key compromise or governance attack is required. The protocol is already live on 10+ chains and actively expanding. `updateRate()` is permissionless, so any caller can expose the DoS once the threshold is crossed. The owner cannot undo the DoS without removing receivers or upgrading the contract.

## Recommendation

1. **Enforce a maximum array length** in `addRateReceiver`:
   ```solidity
   uint256 public constant MAX_RATE_RECEIVERS = 20;
   require(rateReceivers.length < MAX_RATE_RECEIVERS, "Too many receivers");
   ```
2. **Or restrict `updateRate()` to `onlyOwner`** so the caller controls gas supply and update timing.
3. **Or implement paginated delivery**: accept `startIndex` and `endIndex` parameters so a single call processes only a bounded subset of receivers.

## Proof of Concept

Deploy `RSETHMultiChainRateProvider` on a mainnet fork. Register N receivers via `addRateReceiver`. Call `updateRate{value: 10 ether}()` and measure `gasleft()` delta for N = 1, 50, 100, 200. Gas usage will grow linearly; at N ≈ 150–200 the delta will exceed 30 M, causing the transaction to revert with out-of-gas on any standard Ethereum block. The `removeRateReceiver` function confirms the owner can remove entries, but doing so breaks service to those chains — confirming the irreversible nature of the DoS without a contract upgrade.
