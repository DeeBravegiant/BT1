### Title
Rosetta RPC Emits Forged NEP-141 FT Transfer Events from Failed Receipts Without Status Filtering - (File: chain/rosetta-rpc/src/adapters/nep141.rs)

### Summary
The Rosetta RPC's `collect_nep141_events()` function processes `EVENT_JSON:` logs from **all** execution outcomes regardless of whether the receipt succeeded or failed. Because the NEAR runtime always preserves logs in the `ExecutionOutcome` even when a receipt aborts, any unprivileged user can deploy a contract that emits fake NEP-141 `ft_transfer` events and then panics, causing the Rosetta `/block` endpoint to report fabricated `FungibleTokenEvent` balance-change records for transfers that never occurred.

### Finding Description

**Step 1 – Logs survive receipt failure in the runtime.**

In `runtime/runtime/src/function_call.rs`, logs are unconditionally appended to the `ActionResult` regardless of whether the Wasm execution succeeded:

```rust
result.logs.extend(outcome.logs);   // line 149 – always, before the `if execution_succeeded` gate
``` [1](#0-0) 

In `runtime/runtime/src/lib.rs`, `ActionReceiptResult::merge()` also appends logs unconditionally, and `set_error()` (called on failure) explicitly keeps them:

```rust
self.logs.append(&mut next_result.logs);   // line 458 – before the Ok/Err branch
// set_error() clears new_receipts, validator_proposals, tokens_burnt – but NOT logs
``` [2](#0-1) 

The final `ExecutionOutcomeWithId` is built with `logs: result.logs` regardless of whether `status` is `Failure` or `Success`: [3](#0-2) 

This is confirmed by the existing test `test_deploy_and_call_local_receipts`, which asserts `o2.logs == ["hello"]` while `o2.status` is `Failure`: [4](#0-3) 

**Step 2 – Rosetta RPC processes logs from all outcomes without status filtering.**

`collect_nep141_events()` iterates over every `ExecutionOutcomeWithIdView` and calls `extract_events()` on each one with no check on `outcome.outcome.status`: [5](#0-4) 

`extract_events()` simply scans `execution_outcome.outcome.logs` for `EVENT_JSON:` prefixed entries: [6](#0-5) 

The call site in `transactions.rs` passes the full, unfiltered `execution_outcomes` vector: [7](#0-6) 

**Step 3 – The resulting `FungibleTokenEvent` is emitted with `status: "FAILURE"`.**

`build_event()` does set the `status` field via `get_status()`, which returns `"FAILURE"` for failed receipts: [8](#0-7) 

However, `collect_nep141_events()` still **returns** these events to the caller; it does not discard them. The `status` field is a nearcore-specific extension not present in the standard Rosetta specification, so conforming Rosetta clients have no obligation to check it.

### Impact Explanation

An unprivileged user deploys a contract that:
1. Emits one or more `EVENT_JSON:{"standard":"nep141","version":"1.0.0","event":"ft_transfer","data":[{"old_owner_id":"victim.near","new_owner_id":"attacker.near","amount":"1000000000000000000000000"}]}` log lines.
2. Immediately panics (e.g., `env::panic_str("abort")`).

The receipt fails (`ExecutionStatus::Failure`), so no actual token transfer occurs and no on-chain state changes. However, the Rosetta `/block` response for that block will contain a `FungibleTokenEvent` record asserting a large FT transfer from `victim.near` to `attacker.near`. Off-chain systems — exchanges, wallets, indexers — that consume the Rosetta API to track FT balances will see a fabricated balance change. The corrupted value is the **Rosetta RPC result** (`FungibleTokenEvent` records returned by `collect_nep141_events()`).

### Likelihood Explanation

Any account holder can deploy a NEAR smart contract and submit a transaction calling it. No special privileges are required. The attack is cheap (only gas costs) and repeatable. The Rosetta RPC is a production endpoint used by Coinbase's Mesh/Rosetta infrastructure and third-party integrators.

### Recommendation

Filter out failed outcomes before extracting events in `collect_nep141_events()`:

```rust
pub(crate) fn collect_nep141_events(
    receipt_execution_outcomes: &Vec<ExecutionOutcomeWithIdView>,
    ...
) -> crate::errors::Result<Vec<FungibleTokenEvent>> {
    let mut res = Vec::new();
    for outcome in receipt_execution_outcomes {
        // Skip failed receipts – their logs do not represent real state changes.
        if matches!(outcome.outcome.status, ExecutionStatusView::Failure(_)) {
            continue;
        }
        let events = extract_events(outcome);
        ...
    }
    Ok(res)
}
```

Alternatively, filter at the call site in `ExecutionToReceipts::for_block()` before passing outcomes to `collect_nep141_events()`.

### Proof of Concept

1. Deploy the following contract to `attacker.near`:
   ```rust
   #[near_bindgen]
   impl Contract {
       pub fn forge_transfer(&self) {
           env::log_str(r#"EVENT_JSON:{"standard":"nep141","version":"1.0.0","event":"ft_transfer","data":[{"old_owner_id":"victim.near","new_owner_id":"attacker.near","amount":"1000000000000000000000000"}]}"#);
           env::panic_str("abort");
       }
   }
   ```
2. Call `forge_transfer()` on `usdt.tether-token.near`'s shard (or any tracked FT contract address configured in the Rosetta node's `currencies`).
3. Query the Rosetta `/block` endpoint for the block containing that transaction.
4. Observe that the response includes a `FungibleTokenEvent` with `cause: "TRANSFER"`, `affected_account_id: "victim.near"`, `delta_amount: -1000000...`, and `status: "FAILURE"` — a fabricated balance-change record for a transfer that never occurred on-chain.

The root cause path is: `forge_transfer()` → `VMOutcome::abort()` preserves logs → `function_call.rs:149` extends `result.logs` → `lib.rs:458` appends to `ActionReceiptResult.logs` → `lib.rs:1132` emits `ExecutionOutcome{status:Failure, logs:[...]}` → `nep141.rs:collect_nep141_events()` processes without status check → Rosetta `/block` returns forged `FungibleTokenEvent`.

    }

pub(crate) fn extract_events(
    execution_outcome: &ExecutionOutcomeWithIdView,
) -> Vec<crate::models::Nep141Event> {
    let prefix = "EVENT_JSON:";
    execution_outcome
        .outcome
        .logs
        .iter()
        .filter_map(|untrimmed_log| {
            let log = untrimmed_log.trim();
            if !log.starts_with(prefix) {
                return None;
            }

            match serde_json::from_str::<'_, crate::models::Nep141Event>(log[prefix.len()..].trim())
            {
                Ok(result) => Some(result),
                Err(_err) => None,
            }
        })
        .collect()
}
