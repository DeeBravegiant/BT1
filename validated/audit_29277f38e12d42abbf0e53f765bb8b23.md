Audit Report

## Title
Fee Truncates to Zero for Small Claim Amounts Due to Integer Division - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

## Summary
In `KernelMerkleDistributor._processClaim()`, the protocol fee is computed as `(claimableAmount * feeInBPS) / FEE_DENOMINATOR`. Solidity integer division truncates toward zero, so any `claimableAmount` where `claimableAmount * feeInBPS < 10_000` produces `fee = 0`. The `if (fee > 0)` guard then skips the treasury transfer entirely, and the claimer receives the full amount with no fee deduction.

## Finding Description
The fee calculation at [1](#0-0)  performs integer division that truncates toward zero. With `FEE_DENOMINATOR = 10_000` [2](#0-1)  and `MAX_FEE_IN_BPS = 1000` [3](#0-2) , any `claimableAmount < FEE_DENOMINATOR / feeInBPS` produces `fee = 0`. The `claimableAmount` is the incremental amount since the user's last claim: `cumulativeAmount - userClaims[account].cumulativeAmount` [4](#0-3) . When `fee == 0`, the guard at line 341 skips the treasury transfer, and `amountToSend = claimableAmount - 0 = claimableAmount` is returned in full to the claimer. The exploit path is: (1) a merkle root is published with a dust-sized incremental reward for a user; (2) the user calls `claim()` or `claimAndStake()` with a valid merkle proof; (3) `_processClaim` computes `fee = 0`; (4) the treasury receives nothing while the user receives the full amount. No privileged access is required — any user whose incremental reward falls below the truncation threshold triggers this silently. The same pattern exists in `MerkleDistributor.sol` and `KernelTop100MerkleDistributor.sol`. [5](#0-4) 

## Impact Explanation
The protocol fails to collect its configured fee revenue for dust-sized claims. No user funds are lost; the claimer simply receives slightly more than intended. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.** [6](#0-5) 

## Likelihood Explanation
KERNEL is an 18-decimal ERC-20 token. The truncation threshold ranges from 10 wei (at 10% fee) to 10,000 wei (at 0.01% fee) — all sub-dust values that would not arise from normal off-chain reward accounting. The user cannot forge a smaller `claimableAmount` than what the merkle tree encodes, but they can claim at a root where their incremental reward happens to be dust-sized. This is a low-likelihood condition over a long-running distribution with many small-allocation users. [7](#0-6) 

## Recommendation
1. Round up the fee using `Math.mulDiv(claimableAmount, feeInBPS, FEE_DENOMINATOR, Math.Rounding.Up)` so any non-zero fee always rounds to at least 1 wei.
2. Alternatively, enforce a minimum claim amount: require `claimableAmount >= FEE_DENOMINATOR / feeInBPS` so dust claims are rejected or deferred.
3. Apply the same fix to `contracts/utils/MerkleDistributor/MerkleDistributor.sol` and `contracts/KERNEL/KernelTop100MerkleDistributor.sol` which share the identical pattern. [8](#0-7) 

## Proof of Concept
With `feeInBPS = 100` (1%) and an incremental reward of `99 wei`:
```
claimableAmount = 99
fee = (99 * 100) / 10_000 = 9_900 / 10_000 = 0  ← truncated
amountToSend = 99 - 0 = 99
```
`if (fee > 0)` is false → `protocolTreasury` receives 0, user claims 99 wei with no deduction. A Foundry fuzz test parameterizing `claimableAmount` over `[1, FEE_DENOMINATOR/feeInBPS - 1]` and asserting `fee > 0` would fail for all values in that range, confirming the truncation. [5](#0-4)
