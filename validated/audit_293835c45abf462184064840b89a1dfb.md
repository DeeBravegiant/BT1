### Title
Missing Minimum Block Confirmation Enforcement Enables Double-Spend via Zero-Confirmation Foreign Transaction Signing - (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary

The `verify_foreign_transaction` entry point in the MPC contract accepts a user-supplied `BlockConfirmations(0)` value for Bitcoin requests without enforcing any minimum. MPC nodes honor the caller-specified threshold verbatim, so a request with zero confirmations always passes the node-side check (`0 <= actual_confirmations`). This allows an unprivileged caller to obtain a valid threshold signature attesting to an unconfirmed Bitcoin transaction, enabling a bridge double-spend.

### Finding Description

`BlockConfirmations` is a plain `u64` newtype with no lower-bound constraint: [1](#0-0) 

The contract's `verify_foreign_transaction` method performs no validation on the `confirmations` field of a `BitcoinRpcRequest` before enqueuing the request: [2](#0-1) 

On the node side, `BitcoinInspector::extract` enforces the threshold using a simple `<=` comparison:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [3](#0-2) 

When `block_confirmations_threshold` is `0`, this condition is trivially satisfied for any transaction — including one with zero confirmations (i.e., still in the mempool). The nodes then sign a `ForeignTxSignPayload` that embeds the original request (including `confirmations: 0`) and the extracted values: [4](#0-3) 

The contract's `respond_verify_foreign_tx` only verifies that the ECDSA signature is valid over the supplied `payload_hash`; it does not re-derive or validate the confirmation count embedded in the signed payload: [5](#0-4) 

### Impact Explanation

An attacker operating a bridge can:

1. Broadcast a Bitcoin transaction sending funds to the bridge deposit address.
2. Immediately call `verify_foreign_transaction` with `confirmations: BlockConfirmations(0)` before any block confirmation occurs.
3. MPC nodes query the Bitcoin RPC, find the transaction in the mempool (0 confirmations ≥ 0 threshold), extract the block hash (or other values), and produce a valid threshold signature.
4. The attacker submits the signature to the NEAR bridge contract to claim the bridged funds.
5. The attacker then double-spends or RBF-replaces the original Bitcoin transaction, recovering the Bitcoin while keeping the NEAR-side funds.

This constitutes a **forged foreign-chain verification bypass causing a double-spend condition**, matching the High impact tier.

### Likelihood Explanation

The attack path is fully reachable by any unprivileged NEAR account with a 1 yoctoNEAR deposit. No threshold collusion, privileged access, or TEE compromise is required. The nodes behave honestly — they faithfully execute the caller-specified confirmation threshold of zero. The only prerequisite is that the attacker controls a Bitcoin transaction in the mempool, which is trivially achievable.

### Recommendation

Enforce a protocol-level minimum confirmation count in `verify_foreign_transaction` before enqueuing the request. For Bitcoin, a reasonable minimum is 1 (to require at least one block confirmation) or a governance-configurable value stored in the foreign chain configuration. The check should be added in the contract alongside the existing chain-support check:

```rust
// Example: enforce minimum confirmations for Bitcoin
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    if btc_req.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
        env::panic_str("Bitcoin confirmations below protocol minimum");
    }
}
```

Alternatively, store a per-chain minimum confirmation count in the on-chain foreign chain configuration and validate against it at request time.

### Proof of Concept

1. Deploy the MPC contract with Bitcoin as a supported foreign chain.
2. Broadcast a Bitcoin transaction (e.g., a bridge deposit) — do not wait for confirmation.
3. Call `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "domain_id": <foreign_tx_domain_id>,
       "payload_version": 1,
       "request": {
         "Bitcoin": {
           "tx_id": "<mempool_tx_id>",
           "confirmations": 0,
           "extractors": ["BlockHash"]
         }
       }
     }
   }
   ```
4. MPC nodes call `getrawtransaction` on the Bitcoin RPC. The transaction is found in the mempool with 0 confirmations. The check `0 <= 0` passes.
5. Nodes sign the payload and call `respond_verify_foreign_tx`. The contract accepts the response and resolves the yield with a valid signature.
6. Use the signature to claim bridge funds on NEAR, then RBF-replace the original Bitcoin transaction to recover the Bitcoin.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
