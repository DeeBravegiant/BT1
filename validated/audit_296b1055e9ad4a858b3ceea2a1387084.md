Audit Report

## Title
Stale Cross-Chain Rate Enables agETH Over-Issuance via Block Stuffing — (`contracts/agETH/AGETHPoolV3.sol`)

## Summary

`AGETHPoolV3.deposit()` mints agETH using a rate fetched from `agETHOracle` (backed by `AGETHRateReceiver` / `CrossChainRateReceiver`) with no staleness enforcement. `CrossChainRateReceiver` stores a `lastUpdated` timestamp on each LayerZero delivery but `getRate()` never checks it. An attacker who uses block stuffing to suppress `MultiChainRateProvider.updateRate()` calls can hold the receiver's stored rate artificially low and receive more agETH than the deposited ETH backs at the live rate.

## Finding Description

**Rate delivery path:**

`MultiChainRateProvider.updateRate()` is permissionless and payable; it reads the live rate and dispatches it via LayerZero to `AGETHRateReceiver` on the destination chain. [1](#0-0) 

`CrossChainRateReceiver.lzReceive()` stores the received rate and sets `lastUpdated = block.timestamp`. [2](#0-1) 

However, `CrossChainRateReceiver.getRate()` returns `rate` unconditionally — `lastUpdated` is stored but never checked against any maximum age. [3](#0-2) 

`AGETHPoolV3.getRate()` blindly forwards whatever the oracle returns. [4](#0-3) 

`viewSwapAgETHAmountAndFee()` uses that rate directly to compute the mint amount with no freshness guard. [5](#0-4) 

`deposit()` calls `viewSwapAgETHAmountAndFee()` and mints the result unconditionally. [6](#0-5) 

**Exploit flow:**

1. agETH is yield-bearing; its ETH rate drifts monotonically upward (e.g., `1.00e18` → `1.05e18`).
2. Attacker stuffs blocks on the source chain to prevent any `updateRate()` call from being mined, freezing `AGETHRateReceiver.rate` at stale-low value `S < L` (live rate).
3. Attacker calls `AGETHPoolV3.deposit{value: V}()`.
4. Minted: `agETHAmount = (V - fee) * 1e18 / S` — inflated vs. correct `(V - fee) * 1e18 / L`.
5. Over-issuance: `Δ = (V - fee) * 1e18 * (L - S) / (S * L)`.
6. Attacker unstuffs; rate normalizes. Attacker holds agETH redeemable for more ETH than deposited.

No existing check prevents this: `CrossChainRateReceiver` has `lastUpdated` but it is never consumed by any caller in the mint path. [7](#0-6) 

## Impact Explanation

The protocol mints agETH in excess of the ETH deposited at the live rate. The pool's ETH backing per outstanding agETH unit decreases with each exploited deposit. This matches two allowed impacts: **Low. Block stuffing** (the required attack mechanism is explicitly in scope) and **Low. Contract fails to deliver promised returns, but doesn't lose value** (the pool's 1:1-at-live-rate mint accounting is violated).

## Likelihood Explanation

`updateRate()` is permissionless, so block stuffing is the only required mechanism — no key compromise or admin collusion needed. On L2 environments with low block gas costs or a centralized sequencer, suppressing a specific transaction for several blocks is materially cheaper than on Ethereum mainnet. The agETH rate drifts upward continuously with staking yield, so any staleness window creates an exploitable gap. The attacker only needs ETH for gas and block stuffing costs.

## Recommendation

`CrossChainRateReceiver.getRate()` should enforce a maximum rate age using the already-stored `lastUpdated` field:

```solidity
uint256 public constant MAX_RATE_AGE = 1 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Stale rate");
    return rate;
}
```

Alternatively, enforce the staleness check in `AGETHPoolV3.getRate()` by reading `lastUpdated` directly from the receiver. Either location breaks the exploit path. The fix should be applied at the receiver level so all consumers benefit automatically.

## Proof of Concept

```solidity
// Fork test — deploy AGETHPoolV3 pointing to AGETHRateReceiver
// Simulate block stuffing by simply not calling updateRate() after rate increases

uint256 liveRate  = 1.05e18;  // current agETH/ETH rate on source chain
uint256 staleRate = 1.00e18;  // rate frozen in AGETHRateReceiver (no updateRate called)
uint256 deposit   = 10 ether;
uint256 feeBps    = 10;       // 0.1%

uint256 fee       = deposit * feeBps / 10_000;
uint256 afterFee  = deposit - fee;

uint256 agETH_stale = afterFee * 1e18 / staleRate;  // ~9.99 agETH
uint256 agETH_live  = afterFee * 1e18 / liveRate;   // ~9.514 agETH
uint256 overIssue   = agETH_stale - agETH_live;      // ~0.476 agETH (~0.5 ETH at live rate)

// AGETHPoolV3.deposit() succeeds with no revert — no staleness check exists
// CrossChainRateReceiver.getRate() returns staleRate unconditionally
assert(overIssue > 0); // always true when staleRate < liveRate
```

A Foundry fork test can confirm this by: (1) deploying or forking the contracts, (2) advancing time without calling `updateRate()`, (3) calling `deposit()`, and (4) asserting the minted agETH exceeds `afterFee * 1e18 / liveRate`. The `deposit()` call will not revert because no staleness guard exists anywhere in the call chain. [5](#0-4) [3](#0-2)
