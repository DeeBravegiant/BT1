### Title
Uninitialized `_systemAddress` in `HlBridgeToken` After Proxy Upgrade Burns All Bridged Assets on the 3-arg Mint Path — (File: `evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`HlBridgeToken` introduces a critical new state variable `_systemAddress` that is set only inside its `initialize()` function, which is guarded by the `initializer` modifier. When an existing `BridgeToken` proxy is upgraded to `HlBridgeToken` via `OmniBridge.upgradeToken()`, the upgrade call passes empty bytes, so `initialize()` is never invoked. Because the proxy's initializer version counter is already `≥ 1`, calling `initialize()` afterward would revert. No `reinitializer()` path exists. The result is that `_systemAddress` permanently remains `address(0)`. Every subsequent cross-chain settlement that carries a non-empty `message` field calls the 3-arg `mint`, which immediately burns the just-minted tokens via `_update(account, address(0), value)`, permanently destroying the user's bridged assets.

---

### Finding Description

`OmniBridge.upgradeToken()` upgrades a deployed `BridgeToken` proxy to a new implementation:

```solidity
function upgradeToken(address tokenAddress, address implementation)
    external onlyRole(DEFAULT_ADMIN_ROLE)
{
    require(isBridgeToken[tokenAddress], "ERR_NOT_BRIDGE_TOKEN");
    BridgeToken proxy = BridgeToken(tokenAddress);
    proxy.upgradeToAndCall(implementation, bytes(""));   // ← empty calldata
}
``` [1](#0-0) 

The empty second argument means no initialization call is made on the new implementation after the upgrade. `HlBridgeToken` declares `_systemAddress` as a plain storage variable and sets it only inside `initialize()`:

```solidity
address internal _systemAddress;
...
function initialize(..., address systemAddress_, address hyperCoreDeployer_)
    external initializer          // ← version-1 guard; already consumed
{
    ...
    _systemAddress = systemAddress_;
    ...
}
``` [2](#0-1) 

Because the proxy was already initialized as `BridgeToken` (version 1), the `initializer` modifier will revert any subsequent call to `initialize()`. There is no `reinitializer(2)` function anywhere in the contract or the codebase. After the upgrade `_systemAddress == address(0)`.

The 3-arg `mint` override in `HlBridgeToken` is:

```solidity
function mint(address account, uint256 value, bytes memory)
    external override onlyOwner
{
    _mint(account, value);
    _update(account, _systemAddress, value);  // _systemAddress == address(0) → burn
}
``` [3](#0-2) 

`_update(from, address(0), value)` is the standard ERC-20 burn path: it reduces `account`'s balance by `value` and decrements total supply. The net effect is that the user receives zero tokens.

`OmniBridge.finTransfer()` routes to the 3-arg `mint` whenever `payload.message.length != 0`:

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount);
    } else {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient, payload.amount, payload.message
        );
    }
}
``` [4](#0-3) 

---

### Impact Explanation

Any cross-chain transfer whose `message` field is non-empty and whose destination token is an upgraded `HlBridgeToken` proxy will have its bridged amount minted and immediately burned in the same transaction. The user's locked or burned source-chain assets are gone; the EVM-side tokens are destroyed. The loss is irreversible: there is no recovery path because the nonce is marked used (`completedTransfers[payload.destinationNonce] = true`) before the mint, so the settlement cannot be replayed.

This satisfies: **Critical — Irreversible fund lock / permanently unclaimable user value in bridge token flows.**

---

### Likelihood Explanation

`HlBridgeToken` is a production contract in the repository, and `upgradeToken()` is the designated upgrade path for bridge tokens. Any admin-initiated upgrade of an existing `BridgeToken` proxy to `HlBridgeToken` silently leaves `_systemAddress` uninitialized. After that, every user who sends a cross-chain transfer with a non-empty `message` (e.g., a HyperCore routing instruction) loses their funds. The triggering action — sending a cross-chain transfer with a message — is a normal, documented protocol feature available to any unprivileged user.

---

### Recommendation

Replace the `initializer` modifier in `HlBridgeToken.initialize()` with `reinitializer(2)` and add a dedicated `initializeV2(address systemAddress_, address hyperCoreDeployer_)` function that uses `reinitializer(2)`. Update `OmniBridge.upgradeToken()` (or add a separate `upgradeTokenAndInit()` helper) to pass the ABI-encoded `initializeV2` call as the second argument to `upgradeToAndCall`, ensuring `_systemAddress` and `HYPER_CORE_DEPLOYER_SLOT` are always set atomically with the upgrade. This mirrors the fix described in the [OpenZeppelin Initializable documentation](https://github.com/OpenZeppelin/openzeppelin-contracts-upgradeable/blob/v4.9.6/contracts/proxy/utils/Initializable.sol#L8-L57).

---

### Proof of Concept

1. `BridgeToken` proxy `P` is deployed and initialized (version counter = 1, `_systemAddress` slot = `0x0`).
2. Admin calls `OmniBridge.upgradeToken(P, address(hlBridgeTokenImpl))`.
   - `upgradeToAndCall(hlBridgeTokenImpl, bytes(""))` executes; no init call is made.
   - `P._systemAddress` remains `address(0)`.
3. A user on NEAR initiates a transfer of 1000 tokens to EVM with `message = abi.encode(...)` (non-empty).
4. The MPC signs the payload; a relayer calls `OmniBridge.finTransfer(sig, payload)`.
5. Signature check passes; `completedTransfers[nonce]` is set to `true`.
6. Because `payload.message.length != 0`, `IBridgeToken(P).mint(user, 1000, payload.message)` is called.
7. Inside `HlBridgeToken.mint`: `_mint(user, 1000)` credits the user, then `_update(user, address(0), 1000)` burns all 1000 tokens.
8. User receives 0 tokens. The nonce is consumed. Funds are permanently lost with no recourse. [1](#0-0) [5](#0-4) [3](#0-2) [6](#0-5)
