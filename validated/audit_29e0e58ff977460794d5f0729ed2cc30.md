Audit Report

## Title
Attacker Can Capture All Remaining Staking Rewards by Staking 1 Wei When `totalKernelStaked` Drops to Zero - (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

When `totalKernelStaked` reaches zero during an active reward period, the `rewardPerToken()` function freezes at `rewardPerTokenStored`. Any unprivileged attacker who calls `stake(1)` immediately after this condition is met will have `updatedAt` reset to the current timestamp via the `updateReward` modifier, causing the entire remaining reward stream (from stake time to `finishAt`) to be divided by a `totalKernelStaked` of 1. The attacker earns `rewardRate × (finishAt − stakeTime)`, capturing all remaining rewards funded by the protocol for legitimate stakers.

## Finding Description

The root cause is the interaction between `rewardPerToken()`, the `updateReward` modifier, and the ordering of state changes in `stake()`.

**`rewardPerToken()`** (L408–414): When `totalKernelStaked == 0`, returns the frozen `rewardPerTokenStored` without advancing the accumulator.

**`updateReward` modifier** (L232–242): On every state-changing call, sets `updatedAt = lastTimeRewardApplicable()`. This timestamp reset is unconditional — it fires even when `totalKernelStaked == 0`.

**`stake()`** (L281–289): The `updateReward(msg.sender)` modifier executes *before* `balanceOf[msg.sender]` and `totalKernelStaked` are incremented.

Attack flow:

1. Admin calls `notifyRewardAmount` while stakers are present. `rewardRate`, `finishAt`, and `updatedAt` are set.
2. All legitimate stakers call `initiateWithdrawal()`. On the last withdrawal, `updateReward` stamps `updatedAt = block.timestamp`, then `totalKernelStaked` is decremented to 0.
3. Attacker calls `stake(1)`. Inside `updateReward`:
   - `rewardPerTokenStored = rewardPerToken()` → still frozen (totalKernelStaked == 0 in the view call)
   - `updatedAt = lastTimeRewardApplicable()` → **reset to NOW**
   - `rewards[attacker] = earned(attacker)` → 0 (balance still 0 at this point)
   - `userRewardPerTokenPaid[attacker] = rewardPerTokenStored` (the frozen value)
4. After the modifier body executes: `balanceOf[attacker] = 1`, `totalKernelStaked = 1`.
5. From this point, `rewardPerToken()` advances at `rewardRate × Δt × DECIMAL_PRECISION / 1`.
6. At `finishAt`, attacker calls `getReward()` and receives `rewardRate × (finishAt − stakeTime)` — the entire remaining reward budget.

The only existing guard is in `notifyRewardAmount` (L570): `if (totalKernelStaked == 0) revert NoStakedTokens()`. This prevents *starting* a reward period with zero stakers but does nothing to prevent stakers from leaving *after* the period has begun. The protocol's own NatSpec (L18–22) explicitly acknowledges this attack vector, relying solely on an off-chain operational guarantee with no on-chain enforcement.

## Impact Explanation

**High — Theft of unclaimed yield.** The attacker redirects the entire remaining reward budget — funded by the admin/protocol for the benefit of legitimate stakers — to themselves by staking a negligible 1 wei. Legitimate stakers who re-stake after the attacker find `rewardPerTokenStored` already at its maximum, leaving them with negligible earnings for the remainder of the period.

## Likelihood Explanation

**Medium.** The precondition (`totalKernelStaked == 0` during an active reward window) can occur naturally if all stakers call `initiateWithdrawal()`, which is a normal, permissionless user action. A sophisticated attacker monitoring the chain can detect this condition in a single block and front-run any re-staking attempt. The protocol's NatSpec acknowledges the risk and relies entirely on an off-chain guarantee, providing no on-chain resistance to this scenario.

## Recommendation

Track the timestamp at which `totalKernelStaked` last became zero (e.g., `zeroStakeTimestamp`). In `updateReward`, when computing `rewardPerTokenStored`, skip the time interval during which no tokens were staked — i.e., use `max(updatedAt, zeroStakeTimestamp)` as the lower bound for the time delta. Rewards that accrued during the zero-stake window should either be sent to the treasury or rolled into the next reward period via a separate accounting variable, rather than being capturable by the next staker. Additionally, consider adding a minimum stake threshold or a proportionality check in `stake()` to prevent a single 1-wei deposit from capturing a disproportionate share of accumulated rewards.

## Proof of Concept

```solidity
// Foundry test outline
function test_stealRewardsWithOneWei() public {
    // 1. Setup: admin notifies reward amount while legitimate staker is present
    vm.prank(admin);
    kernelDepositPool.notifyRewardAmount(1_000_000e18);
    // rewardRate = 1_000_000e18 / duration, finishAt = block.timestamp + duration

    // 2. Legitimate staker withdraws — totalKernelStaked drops to 0
    vm.prank(legitimateStaker);
    kernelDepositPool.initiateWithdrawal(stakedAmount);
    assertEq(kernelDepositPool.totalKernelStaked(), 0);

    // 3. Attacker stakes 1 wei — updatedAt reset to NOW, userRewardPerTokenPaid set to frozen value
    vm.prank(attacker);
    kernelToken.approve(address(kernelDepositPool), 1);
    kernelDepositPool.stake(1);
    assertEq(kernelDepositPool.totalKernelStaked(), 1);

    // 4. Warp to finishAt
    vm.warp(kernelDepositPool.finishAt());

    // 5. Attacker claims rewards ≈ rewardRate * (finishAt - stakeTime)
    uint256 attackerRewardsBefore = rewardsToken.balanceOf(attacker);
    vm.prank(attacker);
    kernelDepositPool.getReward();
    uint256 attackerRewardsAfter = rewardsToken.balanceOf(attacker);

    // Attacker receives nearly all remaining rewards
    assertApproxEqRel(attackerRewardsAfter - attackerRewardsBefore, 1_000_000e18, 0.01e18);
}
```
