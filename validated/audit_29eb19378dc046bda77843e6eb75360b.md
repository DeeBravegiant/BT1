Audit Report

## Title
Caller-Controlled Asset Selection in `_withdraw()` Enables Cross-Token Theft When Multiple Tokens Are Allowed — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary
`RsETHTokenWrapper` supports multiple allowed tokens via `reinitialize` and `addAllowedToken`, but its `_withdraw` function allows the caller to freely choose which allowed token to receive, with no linkage to the token originally deposited. A user who deposits a depegged bridge variant of rsETH can withdraw the more valuable canonical variant 1:1, draining the wrapper's premium token reserves. The identical structural flaw exists in `AGETHTokenWrapper`, but that contract explicitly blocks adding a second token (no `addAllowedToken` function, comment at line 153 confirms single-token intent), so the multi-token precondition is not currently reachable there.

## Finding Description
`RsETHTokenWrapper` maintains an `allowedTokens` mapping and provides two paths to populate it with multiple tokens:

- `reinitialize(address _altRsETH)` (admin-only, `reinitializer(2)`) explicitly adds a second allowed token. [1](#0-0) 
- `addAllowedToken(address _asset)` (TIMELOCK_ROLE) can add further tokens at any time. [2](#0-1) 

`_deposit` mints wrsETH 1:1 for any allowed token with no record of which token was deposited: [3](#0-2) 

`_withdraw` burns wrsETH and transfers whichever allowed token the caller specifies — the only guard is the allowlist check: [4](#0-3) 

There is no per-user, per-token deposit accounting anywhere in the contract. The wrapper's total supply is a single undifferentiated pool of wrsETH regardless of which underlying token was deposited. Once two tokens with different market prices are simultaneously in `allowedTokens`, any holder of wrsETH can redeem it for the more valuable token, regardless of what they deposited.

The public entry points `withdraw` and `withdrawTo` are fully permissionless: [5](#0-4) 

## Impact Explanation
**Critical — direct theft of user funds.**

When two allowed tokens trade at different prices (e.g., a depegged bridge variant vs. the canonical rsETH), an attacker can acquire the cheaper token, deposit it for wrsETH at 1:1, then withdraw the more expensive token at 1:1. The entire balance of the premium token held by the wrapper is at risk. This is a direct, permissionless fund-theft path with no admin collusion required after the second token is added.

## Likelihood Explanation
**High.** The precondition — two tokens in `allowedTokens` — is the explicitly intended design (the `reinitialize` function exists solely for this purpose). Adding the second token is a normal admin operation, not an attack. Once it is done, any external user can exploit any price discrepancy between the two variants. Bridge variants of the same underlying asset routinely depeg temporarily due to liquidity imbalances, bridge delays, or market stress. The exploit is atomic, requires no victim interaction, and is repeatable until the premium token reserve is exhausted.

## Recommendation
Either enforce a single canonical token at any given time (remove multi-token support or require the old token to be removed before a new one is added), or introduce per-user per-token deposit accounting and validate on withdrawal that the caller is redeeming the same token they deposited:

```solidity
mapping(address user => mapping(address asset => uint256 balance)) public depositedBalance;

// in _deposit:
depositedBalance[_to][_asset] += _amount;

// in _withdraw:
if (depositedBalance[msg.sender][_asset] < _amount) revert InsufficientDepositedBalance();
depositedBalance[msg.sender][_asset] -= _amount;
```

Alternatively, if the wrapper is intended to treat all allowed tokens as fungible (i.e., they are guaranteed 1:1 by an on-chain mechanism), document and enforce that invariant at the contract level and ensure no token can be added unless it is provably at parity.

## Proof of Concept
**Setup:** Admin calls `reinitialize(rsETH_B)` after `initialize` already added `rsETH_A`. Both are now in `allowedTokens`. `rsETH_A` trades at 0.90 ETH, `rsETH_B` at 1.00 ETH. The wrapper holds 1000 `rsETH_B` from prior bridger deposits.

1. Attacker acquires 1000 `rsETH_A` for ~900 ETH equivalent.
2. Attacker calls `deposit(rsETH_A, 1000e18)` → `_deposit` transfers 1000 `rsETH_A` in, mints 1000 wrsETH. [3](#0-2) 
3. Attacker calls `withdraw(rsETH_B, 1000e18)` → `_withdraw` checks `allowedTokens[rsETH_B]` (passes), burns 1000 wrsETH, transfers 1000 `rsETH_B` to attacker. [4](#0-3) 
4. Net result: attacker spent ~900 ETH, received ~1000 ETH worth of `rsETH_B`. The wrapper's `rsETH_B` reserve is drained.

**Foundry test sketch:**
```solidity
function test_crossTokenTheft() public {
    // admin adds second token
    vm.prank(admin);
    wrapper.reinitialize(address(rsETH_B));

    // attacker deposits cheap token
    vm.startPrank(attacker);
    rsETH_A.approve(address(wrapper), 1000e18);
    wrapper.deposit(address(rsETH_A), 1000e18);

    // attacker withdraws premium token
    wrapper.withdraw(address(rsETH_B), 1000e18);
    vm.stopPrank();

    assertEq(rsETH_B.balanceOf(attacker), 1000e18); // attacker drained rsETH_B
    assertEq(rsETH_B.balanceOf(address(wrapper)), 0);
}
```
