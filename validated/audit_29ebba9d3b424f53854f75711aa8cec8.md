### Title
Storage Deposit for Update Proposals Permanently Locked in MPC Contract With No Refund Path - (File: `crates/contract/src/update.rs`)

---

### Summary

`propose_update` requires a NEAR token deposit proportional to the size of the proposed update (potentially megabytes of contract code). The `UpdateEntry` struct never records the proposer's account ID, so when `do_update` clears all entries upon execution, the storage is freed but the deposited NEAR is permanently locked in the contract with no refund path.

---

### Finding Description

In `propose_update` (`crates/contract/src/lib.rs`), the contract collects a mandatory deposit from the proposer:

```rust
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// ...
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(proposer).transfer(diff).detach();
}
```

Only the *excess* above `required` is refunded. The `required` amount — computed as `storage_byte_cost * bytes_used` — stays in the contract. [1](#0-0) 

The `UpdateEntry` struct that is stored records `bytes_used` but **not** the proposer's account ID:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
``` [2](#0-1) 

When `do_update` is called after threshold votes are reached, it clears **all** entries and votes — not just the winning proposal:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    // ...
}
``` [3](#0-2) 

Because the proposer's identity is never stored in `UpdateEntry`, there is no way to issue refunds when entries are cleared. A search of the entire contract confirms there is no `cancel_update`, `withdraw_deposit`, or equivalent function. [4](#0-3) 

The deposit calculation for a 1 MB contract binary:

```rust
fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
``` [5](#0-4) 

At NEAR's storage cost of ~10 yoctoNEAR/byte, a 1 MB contract requires ≈10 NEAR per proposal. Multiple competing proposals accumulate proportionally.

---

### Impact Explanation

Every participant who calls `propose_update` with a contract binary permanently loses their storage deposit when any update is executed (or when the contract is upgraded via the winning proposal). The contract's balance increases by the freed storage, but the original depositors receive nothing back. This breaks the balance accounting invariant: funds paid into the contract for a specific purpose (storage reservation) are irrecoverably stranded once that storage is freed. This matches:

> **Medium. Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

This occurs in normal protocol operation every time a contract upgrade is executed. If two or more participants propose competing upgrades (a realistic scenario during governance disagreements), all non-winning proposers permanently lose their deposits. The loss is proportional to the contract binary size, which for a production WASM binary is routinely 500 KB–2 MB.

---

### Recommendation

1. Add a `proposer: AccountId` field to `UpdateEntry` so the depositor's identity is preserved.
2. In `do_update`, before clearing non-winning entries, iterate over them and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each.
3. Alternatively, add a `cancel_update(id: UpdateId)` method that allows the original proposer to withdraw their entry and receive a refund, callable before the update is executed.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching 10 NEAR (the required deposit). The deposit stays in the contract; only excess is refunded.
2. Participant B calls `propose_update` with a different 1 MB binary, also attaching 10 NEAR.
3. Threshold participants vote for A's update via `vote_update`.
4. `do_update` executes: it removes A's entry, then calls `self.entries.clear()` — deleting B's entry. The storage for both entries is freed, increasing the contract's balance by ~20 NEAR.
5. Participant B's 10 NEAR is permanently locked in the MPC contract. There is no function to recover it.

impl ProposedUpdates {
    pub fn required_deposit(update: &Update) -> NearToken {
        required_deposit(bytes_used(update))
    }

    /// Propose an update given the new contract code and/or config.
    pub fn propose(&mut self, update: Update) -> UpdateId {
        let bytes_used = bytes_used(&update);

        let id = self.id.generate();
        self.entries.insert(id, UpdateEntry { update, bytes_used });

        id
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
