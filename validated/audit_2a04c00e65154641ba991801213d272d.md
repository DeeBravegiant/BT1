Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Withdrawal Initiations Post-Deployment - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal` guards with `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and any `uint256` array length is always `>= 0`, every call to `initiateWithdrawal` unconditionally reverts with `WithdrawalLimitReached` until an admin separately calls `setMaxNumberOfWithdrawalsPerUser`. Any KERNEL tokens staked before that admin action are temporarily frozen with no user-accessible recovery path.

## Finding Description
`initialize()` sets `kernelToken`, `rewardsToken`, and the admin role, but leaves `maxNumberOfWithdrawalsPerUser` at its default value of `0`: [1](#0-0) 

The withdrawal guard in `initiateWithdrawal` is: [2](#0-1) 

When `maxNumberOfWithdrawalsPerUser == 0`, the condition `userWithdrawalIds[msg.sender].length >= 0` evaluates to `true` for every caller (including fresh users with an empty array, since `0 >= 0`). This makes `initiateWithdrawal` unconditionally revert.

The setter `setMaxNumberOfWithdrawalsPerUser` explicitly rejects `0` as an invalid argument, confirming `0` is not a valid operational value — yet `initialize()` leaves the variable at exactly that value: [3](#0-2) 

`stake()` is ungated and callable by any user immediately after deployment: [4](#0-3) 

There is no on-chain enforcement requiring the admin to call `setMaxNumberOfWithdrawalsPerUser` before or atomically with deployment.

## Impact Explanation
Any user who calls `stake()` before the admin calls `setMaxNumberOfWithdrawalsPerUser` has their KERNEL tokens temporarily frozen: `initiateWithdrawal` always reverts, so tokens cannot be queued for return. This matches **Medium — Temporary freezing of funds**. The freeze is not permanent because the admin can eventually call the setter, but the duration is unbounded and not enforced on-chain.

## Likelihood Explanation
The deployment sequence is not enforced on-chain. `stake()` is open to any user in the same block as deployment. If the admin does not call `setMaxNumberOfWithdrawalsPerUser` atomically in the deployment transaction, a window exists — potentially indefinitely — during which all withdrawal initiations revert. The missing initialization is not surfaced by any revert or event, making it easy to overlook. Likelihood is **Medium**.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
```

This mirrors the pattern used for `maxNodeDelegatorLimit` in `LRTDepositPool.initialize()`: [5](#0-4) 

## Proof of Concept
1. Admin deploys `KernelDepositPool` and calls `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` is `0`.
2. User calls `stake(1e18)`. Succeeds — `balanceOf[user] = 1e18`, `totalKernelStaked = 1e18`.
3. User calls `initiateWithdrawal(1e18)`:
   - `_amount == 0` → false
   - `balanceOf[msg.sender] < _amount` → false
   - `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` → `0 >= 0` → **true** → reverts `WithdrawalLimitReached`
4. User's `1e18` KERNEL is locked with no recovery path until admin calls `setMaxNumberOfWithdrawalsPerUser(n)` where `n > 0`.

**Foundry test plan:**
```solidity
function test_withdrawalBlockedOnFreshDeploy() public {
    // deploy + initialize without calling setMaxNumberOfWithdrawalsPerUser
    kernelPool.initialize(admin, address(kernelToken), address(rewardToken));
    kernelToken.mint(user, 1e18);
    vm.prank(user);
    kernelToken.approve(address(kernelPool), 1e18);
    vm.prank(user);
    kernelPool.stake(1e18);
    vm.prank(user);
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    kernelPool.initiateWithdrawal(1e18);
}
```
