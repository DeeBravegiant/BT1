### Title
Single-Participant Unconditional Key-Event Abort Enables Permanent Blocking of Key Generation and Resharing - (File: `crates/contract/src/state/key_event.rs`)

---

### Summary

`vote_abort_key_event_instance` requires only **one** attested participant vote to immediately destroy the active key-event instance. Because there is no threshold requirement on abort (unlike the threshold requirement on success), a single Byzantine participant — strictly below the signing threshold — can call this function repeatedly to abort every new instance the leader starts, permanently blocking key generation (`Initializing` state) or key resharing (`Resharing` state).

---

### Finding Description

The public contract method `vote_abort_key_event_instance` is gated only by `assert_caller_is_attested_participant_and_protocol_active()`, which verifies that the signer is an attested member of the currently active participant set. [1](#0-0) 

Once that check passes, the call is forwarded to `ProtocolContractState::vote_abort_key_event_instance`, which dispatches to either `InitializingContractState::vote_abort` or `ResharingContractState::vote_abort`. [2](#0-1) 

Both delegate to `KeyEvent::vote_abort`, which — after verifying the participant identity and the `key_event_id` match — unconditionally sets `self.instance = None` on the **first** valid vote, with no threshold check whatsoever: [3](#0-2) 

This is structurally asymmetric with the success path (`vote_pk` / `vote_reshared`), which requires **all** participants to agree before the instance advances: [4](#0-3) 

After an abort, the leader must call `start_keygen_instance` or `start_reshare_instance` with the next `attempt_id` to begin a new instance. The Byzantine participant can immediately call `vote_abort_key_event_instance` again with the new `key_event_id`, destroying the new instance as well. The `VoteAlreadySubmitted` guard only prevents double-voting within the **same** instance; it does not prevent the same participant from aborting successive instances. [5](#0-4) 

The `active_participants()` function confirms that during `Initializing` the abort-eligible set is the proposed keygen participants, and during `Resharing` it is the proposed new participants: [6](#0-5) 

---

### Impact Explanation

**Medium — contract execution-flow manipulation that permanently breaks the key-generation and resharing lifecycle invariants.**

- **Initializing state**: A single Byzantine participant in the proposed set can abort every keygen instance indefinitely. The MPC network can never complete key generation, so it can never transition to `Running` and can never issue any threshold signatures. Any funds or cross-chain operations that depend on the MPC network becoming operational are permanently blocked.
- **Resharing state**: A single Byzantine participant in the proposed new set can abort every resharing instance indefinitely. The network is stuck in `Resharing`, unable to update its participant set or threshold. The old running state's signing capability is preserved, but the network cannot rotate keys, add/remove participants, or respond to a compromised node.

In both cases the attacker needs no collusion, no leaked key, and no network-level capability — only a valid TEE attestation and membership in the proposed participant set.

---

### Likelihood Explanation

**Medium.** Any single participant in the proposed set who turns Byzantine (e.g., a compromised TEE node, a financially motivated saboteur, or a participant who disagrees with a governance decision) can execute this attack. The cost is one NEAR transaction per abort, which is negligible. The attack is fully on-chain, deterministic, and requires no special timing. The only prerequisite — a valid TEE attestation — is already satisfied by any legitimate participant.

---

### Recommendation

Require a **threshold** number of abort votes before destroying the active instance, mirroring the threshold requirement on the success path. Specifically:

1. Replace the immediate `self.instance = None` in `KeyEvent::vote_abort` with an abort-vote accumulator (analogous to `cancel_votes` in `InitializingContractState`).
2. Only set `self.instance = None` once the abort-vote count reaches the reconstruction threshold (or the governance threshold).
3. Alternatively, restrict `vote_abort_key_event_instance` to the leader only, since the leader is already the sole entity authorized to start a new instance.

---

### Proof of Concept

**Setup**: MPC network with `n = 5` participants, signing threshold `t = 3`. Network is in `Resharing` state. Participant `P1` (one of the 5 proposed new participants) is Byzantine.

**Attack loop**:

1. Leader calls `start_reshare_instance(key_event_id_attempt_0)` → instance is active.
2. `P1` calls `vote_abort_key_event_instance(key_event_id_attempt_0)`.
   - `assert_caller_is_attested_participant_and_protocol_active()` passes (P1 is attested).
   - `KeyEvent::vote_abort` verifies P1 is a participant, checks `key_event_id` matches, then executes `self.instance = None`.
   - Instance is destroyed. `next_attempt_id` is now `1`.
3. Leader calls `start_reshare_instance(key_event_id_attempt_1)` → new instance is active.
4. `P1` calls `vote_abort_key_event_instance(key_event_id_attempt_1)` → instance destroyed again.
5. Steps 3–4 repeat indefinitely.

The `VoteAlreadySubmitted` guard at line 154 of `key_event.rs` does not fire because each abort targets a **different** instance (different `attempt_id`). The resharing never completes. The network is permanently stuck in `Resharing` state. [3](#0-2) [7](#0-6) [8](#0-7) [9](#0-8)

    pub fn vote_abort(&mut self, key_event_id: KeyEventId) -> Result<(), Error> {
        let candidate = self.verify_vote(&key_event_id)?;
        if self
            .instance
            .as_ref()
            .unwrap()
            .completed
            .contains(&candidate)
        {
            return Err(VoteError::VoteAlreadySubmitted.into());
        }
        self.instance = None;
        Ok(())
    }
