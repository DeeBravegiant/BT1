Audit Report

## Title
Instant Withdrawal Consumes Assets Committed to Queued Withdrawal Users Due to Unsynchronized Accounting - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`instantWithdrawal` in `LRTWithdrawalManager` draws assets from `LRTUnstakingVault` using only the `queuedWithdrawalsBuffer` guard, which defaults to zero and is never synchronized with `assetsCommitted`. When `queuedWithdrawalsBuffer` is zero and instant withdrawals are enabled, any rsETH holder can drain vault assets already committed to queued withdrawal users. This causes subsequent `unlockQueue` calls to revert, freezing the rsETH that queued users transferred into the contract.

## Finding Description

Two independent accounting systems govern the same pool of vault assets:

**System 1 — `assetsCommitted` in `LRTWithdrawalManager`**

`initiateWithdrawal` increments `assetsCommitted[asset]` by the expected payout amount and transfers the user's rsETH into the contract: [1](#0-0) 

`getAvailableAssetAmount` subtracts `assetsCommitted` from total protocol deposits to prevent over-commitment of new queued withdrawals: [2](#0-1) 

**System 2 — `queuedWithdrawalsBuffer` in `LRTUnstakingVault`**

`instantWithdrawal` checks `getAssetsAvailableForInstantWithdrawal`, which subtracts only the manually-set `queuedWithdrawalsBuffer` from the vault balance: [3](#0-2) 

`queuedWithdrawalsBuffer` defaults to zero and is only updated by an operator calling `setQueuedWithdrawalsBuffer`: [4](#0-3) 

**The gap:** `instantWithdrawal` never reads or decrements `assetsCommitted`. With `queuedWithdrawalsBuffer == 0`, the entire vault balance is available for instant withdrawal, including assets already committed to queued users: [5](#0-4) 

After the vault is drained, `unlockQueue` reads `unstakingVault.balanceOf(asset)` as `totalAvailableAssets`: [6](#0-5) 

With the vault empty, `params.totalAvailableAssets == 0` causes an immediate revert before `assetsCommitted` is ever decremented (which only happens inside `_unlockWithdrawalRequests` at line 802): [7](#0-6) 

The queued users' rsETH remains locked in `LRTWithdrawalManager` with no path to completion until the vault is externally replenished.

## Impact Explanation

Queued withdrawal users who called `initiateWithdrawal` have their rsETH locked in `LRTWithdrawalManager` (transferred at line 166) and their expected asset amount committed, but the vault assets backing those commitments have been consumed by instant withdrawal users. `unlockQueue` reverts, preventing the withdrawal queue from advancing. This constitutes **temporary freezing of funds** (Medium severity). Recovery requires the protocol to complete additional EigenLayer unstaking cycles to replenish the vault, which is not guaranteed to happen promptly.

## Likelihood Explanation

All three required conditions are simultaneously true in the default operational configuration:

1. `isInstantWithdrawalEnabled[asset] == true` — a normal operational state when the feature is live.
2. `queuedWithdrawalsBuffer[asset] == 0` — the **default** value; operators must explicitly set a non-zero value to provide any protection.
3. Assets present in `LRTUnstakingVault` while `assetsCommitted[asset] > 0` — a routine state during the withdrawal lifecycle after EigenLayer unstaking completes.

Any unprivileged rsETH holder can trigger this by calling the public `instantWithdrawal` function. No privileged access, oracle manipulation, or governance capture is required.

## Recommendation

`getAssetsAvailableForInstantWithdrawal` (or the check inside `instantWithdrawal`) should subtract `assetsCommitted[asset]` from the vault balance instead of relying on the manually-set `queuedWithdrawalsBuffer`. The cleanest fix is to have `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal` accept or read `assetsCommitted` from `LRTWithdrawalManager`, or to move the guard entirely into `instantWithdrawal` where it can directly compare against `assetsCommitted[asset]`. Alternatively, `instantWithdrawal` should decrement `assetsCommitted` when it consumes committed assets, but this requires cross-contract coordination and is more error-prone.

## Proof of Concept

```
Setup:
  LRTUnstakingVault holds 100 stETH
  queuedWithdrawalsBuffer[stETH] = 0 (default)
  isInstantWithdrawalEnabled[stETH] = true

Step 1 — UserA calls initiateWithdrawal(stETH, rsETH_A):
  → rsETH_A transferred from UserA to LRTWithdrawalManager (line 166)
  → assetsCommitted[stETH] += 100 stETH (line 173)
  → getAvailableAssetAmount(stETH) = totalAssets - 100 = 0 (no new queued withdrawals possible)

Step 2 — UserB (any rsETH holder) calls instantWithdrawal(stETH, rsETH_B):
  → getAssetsAvailableForInstantWithdrawal(stETH) = 100 - 0 = 100 stETH (buffer is 0)
  → check passes (line 231)
  → unstakingVault.redeem(stETH, 100) drains vault (line 235)
  → assetsCommitted[stETH] remains 100 (never touched)

Step 3 — LRTUnstakingVault now holds 0 stETH

Step 4 — Operator calls unlockQueue(stETH, ...):
  → _createUnlockParams: totalAvailableAssets = unstakingVault.balanceOf(stETH) = 0 (line 849)
  → revert AmountMustBeGreaterThanZero() (line 297)

Result:
  UserA's rsETH is permanently locked in LRTWithdrawalManager.
  assetsCommitted[stETH] = 100 but vault = 0.
  unlockQueue cannot advance until vault is externally replenished.
  UserA's funds are frozen.
```

Foundry test plan: deploy `LRTWithdrawalManager` and `LRTUnstakingVault` on a local fork; seed the vault with 100 stETH; set `isInstantWithdrawalEnabled[stETH] = true` with `queuedWithdrawalsBuffer[stETH] = 0`; have UserA call `initiateWithdrawal`; have UserB call `instantWithdrawal` draining the vault; assert that `unlockQueue` reverts with `AmountMustBeGreaterThanZero` and that UserA's rsETH balance in the contract is non-zero.
