Audit Report

## Title
Missing Staleness and Zero-Value Validation in Chainlink Price Feed â€” (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, performing no staleness check on `updatedAt`, no incomplete-round check on `answeredInRound`, and no zero-value guard on the returned `int256 price`. A stale or zero Chainlink answer propagates unchecked into the publicly callable `LRTOracle.updateRSETHPrice()`, which can produce an artificially depressed rsETH price that triggers the protocol's automatic downside-protection pause, temporarily freezing all user deposits and withdrawals.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` reads the Chainlink feed as:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values are available but only `price` is used. The contract does not check `updatedAt` against any staleness threshold, does not validate `price > 0`, and does not check `answeredInRound >= roundId`.

By contrast, the sibling contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository correctly performs all three checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unchecked price flows through `LRTOracle.getAssetPrice()` â†’ `_getTotalEthInProtocol()` â†’ `_updateRsETHPrice()`: [3](#0-2) 

`updateRSETHPrice()` is public with no access control beyond `whenNotPaused`: [4](#0-3) 

When `newRsETHPrice` drops below `highestRsethPrice` by more than `pricePercentageLimit`, the downside-protection logic automatically pauses `lrtDepositPool`, `withdrawalManager`, and `LRTOracle` itself: [5](#0-4) 

Existing guards are insufficient: `updatePriceOracleForValidated()` performs a sanity check at oracle registration time (`price > 1e16 && price < 1e19`), but this is a one-time check at setup and does not protect against a feed going stale after registration. [6](#0-5) 

## Impact Explanation
**Medium â€” Temporary freezing of funds.**

When a registered Chainlink feed goes stale (returning a price below the true market price) or returns zero (circuit-breaker scenario), any unprivileged caller can invoke `updateRSETHPrice()`. The stale/zero price causes `_getTotalEthInProtocol()` to undercount TVL, producing an artificially depressed `newRsETHPrice`. If the computed drop exceeds `pricePercentageLimit * highestRsethPrice`, the auto-pause fires, blocking all user deposits and withdrawals until an admin manually calls `unpause()`. This matches the allowed impact: **Medium â€” Temporary freezing of funds**. [7](#0-6) 

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 24 hours for some ETH-denominated pairs). During low-volatility periods, feeds may not update for the full heartbeat window. On L2 networks, sequencer downtime can cause feeds to go stale. These are realistic, non-adversarial conditions requiring no attacker privilege. Once a feed is stale, any external account can call the public `updateRSETHPrice()` to trigger the incorrect price update and resulting pause. The only prerequisite is that `pricePercentageLimit` is set to a non-zero value, which is the expected production configuration for downside protection to function at all. [8](#0-7) 

## Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

1. Capture all five return values from `latestRoundData()`.
2. Revert if `answeredInRound < roundId` (incomplete round).
3. Revert if `updatedAt == 0` (round not completed).
4. Revert if `block.timestamp - updatedAt > heartbeat` (stale price), using a per-asset configurable heartbeat stored alongside `assetPriceFeed`.
5. Revert if `price <= 0` (zero or negative answer).

This mirrors the fix already present in `ChainlinkOracleForRSETHPoolCollateral` and eliminates the inconsistency between the two oracle contracts. [9](#0-8) 

## Proof of Concept
1. A Chainlink feed registered in `ChainlinkPriceOracle` (e.g., stETH/ETH) goes stale â€” its `updatedAt` timestamp is older than the feed's heartbeat, and the returned price is, say, 5% below the true market price.
2. `pricePercentageLimit` is set to `3e16` (3%) â€” a reasonable production value.
3. An unprivileged attacker calls `LRTOracle.updateRSETHPrice()`.
4. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale (depressed) price with no revert.
5. `newRsETHPrice` is computed as artificially low (5% below `highestRsethPrice`).
6. `diff > pricePercentageLimit.mulWad(highestRsethPrice)` evaluates to true (5% > 3%).
7. `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()` are called.
8. All user deposits and withdrawals are blocked until an admin calls `unpause()`.
