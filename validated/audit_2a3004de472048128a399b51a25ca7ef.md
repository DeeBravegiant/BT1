Audit Report

## Title
Pool Admin Can Set Uncapped Per-Bin Fees Instantly, Enabling Front-Running of Swaps â€” (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`setPoolBinAdditionalFees()` allows the pool admin to set `addFeeBuyE6`/`addFeeSellE6` to any `uint16` value (up to 65535, â‰ˆ6.55% in E6 units) with no cap and no timelock. This is an admin-boundary break: the global admin fee path enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps, but the per-bin fee path has no equivalent guard. A pool admin can front-run a pending swap by spiking the bin fee, forcing the swapper to pay materially more input tokens than expected, with no on-chain protection available to the swapper.

## Finding Description

**Cap asymmetry between global and per-bin admin fees**

`setPoolAdminFees()` enforces caps before writing: [1](#0-0) 

`setPoolBinAdditionalFees()` performs no cap check whatsoever â€” it passes the caller-supplied values directly to the pool: [2](#0-1) 

`setBinAdditionalFees()` on the pool validates only the bin index, not the fee magnitude: [3](#0-2) 

**Fee is read live from storage during swap execution**

At swap time, `binState.addFeeBuyE6` is loaded from `_binStates` and combined with `baseFeeX64` to form the effective fee multiplier: [4](#0-3) 

The combined fee is then applied multiplicatively to the gross input: [5](#0-4) 

A fee spike committed in a front-running transaction is therefore fully reflected in the victim's swap cost.

**`priceLimitX64` provides no protection**

`priceLimitX64` is compared against the interpolated bin-position price to cap how far into the bin the swap travels: [6](#0-5) 

It has no relationship to the fee multiplier. A swapper can set a tight price limit and still be charged an arbitrarily higher fee on the same output quantity.

**No timelock on per-bin fee changes**

The interface confirms that `proposePoolPriceProvider` has a timelock, but `setPoolBinAdditionalFees` is a direct, single-transaction mutator with no scheduling step: [7](#0-6) 

## Impact Explanation

This is a confirmed admin-boundary break: the pool admin can exceed the effective fee cap that the factory owner intended to enforce via `maxAdminSpreadFeeE6`, because the per-bin fee path bypasses that cap entirely. The direct consequence is loss of swapper principal â€” for an exact-output swap, the swapper pays up to ~6.55% more input tokens than they observed when constructing the transaction. The excess accrues to LPs in the affected bin; if the pool admin is also an LP, they capture it directly. This meets the High/Critical threshold for direct loss of user principal with no on-chain recourse.

## Likelihood Explanation

The attack requires: (1) a pool with a mutable admin, (2) the admin monitoring the mempool (trivial on any public EVM chain), and (3) a single `setPoolBinAdditionalFees` call to spike the fee before the victim's transaction, followed by a reset. No special privileges beyond the pool admin role are needed. The attack is repeatable on every swap and leaves no on-chain trace beyond the fee event.

## Recommendation

1. **Add a cap** inside `setPoolBinAdditionalFees()` (e.g., bounded by a dedicated `maxBinAdditionalFeeE6` constant or reusing `maxAdminSpreadFeeE6`), mirroring the cap already enforced for global admin fees in `setPoolAdminFees()`.
2. **Add a timelock** for per-bin fee increases, consistent with the timelock already applied to price provider rotation.
3. Alternatively, expose a `maxTotalFeeE6` parameter on `swap()` so callers can bound the combined fee they accept, providing an on-chain slippage guard on the fee dimension.
