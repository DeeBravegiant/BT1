### Title
Harvester-Controlled `farmer_reward_address_override` Accepted Without Enforcement Allows Unconditional Farmer Reward Diversion - (File: chia/farmer/farmer_api.py)

### Summary
The Chia farmer protocol (CHIP-22) allows a connected harvester to supply a `farmer_reward_address_override` field in both `NewProofOfSpace` and `RespondSignatures` messages. The farmer is supposed to enforce a fee-quality gate before honoring this override, but the gate only emits log warnings and never blocks the override from being applied. Any connected harvester can therefore redirect 100% of the farmer's block reward (XCH) to an arbitrary puzzle hash of its choosing.

### Finding Description

The `NewProofOfSpace` harvester protocol message carries an optional `farmer_reward_address_override` field: [1](#0-0) 

Similarly, `RespondSignatures` carries the same field: [2](#0-1) 

When `FarmerAPI.new_proof_of_space` receives a `NewProofOfSpace` with a non-`None` override, it calls `notify_farmer_reward_taken_by_harvester_as_fee`: [3](#0-2) 

That function checks the fee quality against the harvester-supplied threshold and, if the check fails, emits only log warnings — it **never raises an exception, returns a sentinel, or otherwise prevents the override from being used**: [4](#0-3) 

Execution then continues into `_process_respond_signatures`, where the harvester-supplied address from `RespondSignatures.farmer_reward_address_override` is accepted unconditionally and placed directly into `DeclareProofOfSpace.farmer_reward_address`: [5](#0-4) 

That `DeclareProofOfSpace` message is broadcast to all connected full nodes: [6](#0-5) 

The full node then uses `farmer_reward_address` as the puzzle hash for the farmer coinbase reward in the block it farms. There is no on-chain or off-chain check that the address belongs to the farmer.

### Impact Explanation

A malicious or compromised harvester can set `farmer_reward_address_override` to any `bytes32` puzzle hash — including one it controls — in every `NewProofOfSpace` or `RespondSignatures` message it sends. Because the fee-quality gate is advisory (log-only), the farmer will unconditionally redirect its block reward (currently ~0.25 XCH per block) to the attacker's address for every block won while the malicious harvester is connected. This constitutes unauthorized reward diversion of XCH from the farmer to the attacker.

### Likelihood Explanation

Any party that can connect a harvester to the farmer — including third-party harvester services explicitly supported by CHIP-22 — can trigger this. No key material, admin access, or cryptographic break is required. The attacker only needs a valid proof of space (which the farmer itself verifies) and the ability to set the override field in the protocol message.

### Recommendation

The fee-quality check in `notify_farmer_reward_taken_by_harvester_as_fee` must be made **blocking**: if the harvester-supplied `applied_fee_threshold` does not satisfy the fee-quality convention, the farmer should discard the override (fall back to `self.farmer.farmer_target`) rather than merely logging a warning. Concretely, `notify_farmer_reward_taken_by_harvester_as_fee` should return a boolean indicating whether the override is legitimate, and `new_proof_of_space` / `_process_respond_signatures` should only apply the override when that boolean is `True`.

### Proof of Concept

1. Operator a third-party harvester (CHIP-22 explicitly supports this).
2. When the harvester finds a valid proof of space, send `NewProofOfSpace` with:
   - `farmer_reward_address_override` = attacker-controlled puzzle hash
   - `fee_info.applied_fee_threshold` = `0` (or omit `fee_info` entirely)
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, which logs a warning but returns `None` and does not abort.
4. The farmer sends `RequestSignatures` to the harvester; the harvester replies with `RespondSignatures` also carrying `farmer_reward_address_override` = attacker address.
5. `_process_respond_signatures` sets `farmer_reward_address = response.farmer_reward_address_override` (attacker address) with no further check.
6. `DeclareProofOfSpace` is broadcast with the attacker's puzzle hash as `farmer_reward_puzzle_hash`.
7. The full node includes the block; the farmer coinbase reward is paid to the attacker's address.

    async def respond_signatures(self, response: harvester_protocol.RespondSignatures) -> None:
        request = self._process_respond_signatures(response)

            return None

        message: Message | None = None
        if isinstance(request, DeclareProofOfSpace):
