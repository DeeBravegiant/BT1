Audit Report

## Title
rsETH Share-Price Inflation via Open `receive()` + Unguarded Public `updateRSETHPrice()` Enables First-Depositor Fund Theft - (File: contracts/LRTDepositPool.sol, contracts/LRTOracle.sol)

## Summary
An unprivileged attacker can perform a classic share-price inflation attack against `LRTDepositPool`. By making a 1-wei seed deposit, donating ETH directly via the unrestricted `receive()` fallback, and calling the public `updateRSETHPrice()`, the attacker inflates `rsETHPrice` to an arbitrarily large value. Any subsequent depositor who passes `minRSETHAmountExpected = 0` receives 0 rsETH for their full ETH deposit while the attacker—holding the entire rsETH supply—captures the victim's funds. This constitutes direct theft of user funds at Critical severity.

## Finding Description

**Root cause — four individually weak properties combine into a critical exploit:**

**1. Open `receive()` fallback**
`LRTDepositPool` accepts arbitrary ETH from any sender with no restriction: [1](#0-0) 

`getETHDistributionData()` counts `address(this).balance` directly, so any donated ETH is immediately reflected in `totalETHInProtocol`: [2](#0-1) 

**2. Public, unguarded `updateRSETHPrice()`**
No access control restricts who may trigger a price update: [3](#0-2) 

The price-increase guard is gated on `pricePercentageLimit > 0`. Because `pricePercentageLimit` is a `uint256` state variable never set during `initialize()`, it defaults to 0, making `isPriceIncreaseOffLimit` permanently `false` until an admin explicitly sets it: [4](#0-3) 

**3. Zero-rsETH mint passes the slippage check**
`_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`. When both are 0, the condition `0 < 0` is false and the deposit succeeds with 0 rsETH minted: [5](#0-4) 

**4. Zero `minAmountToDeposit` allows dust seed deposits**
`minAmountToDeposit` is also never set during `initialize()`, defaulting to 0. The guard `depositAmount < minAmountToDeposit` evaluates `1 < 0 = false`, so a 1-wei deposit is accepted: [6](#0-5) 

**Exploit flow:**

- Step 1: Attacker calls `updateRSETHPrice()`. Since `rsethSupply == 0`, `rsETHPrice` is hard-coded to `1 ether` and `highestRsethPrice = 1 ether`: [7](#0-6) 

- Step 2: Attacker calls `depositETH{value: 1}(0, "")`. `getRsETHAmountToMint` computes `1 * 1e18 / 1e18 = 1`, minting 1 wei rsETH: [8](#0-7) 

- Step 3: Attacker sends D wei ETH directly to `LRTDepositPool` via `receive()`. `address(this).balance` becomes `1 + D`.

- Step 4: Attacker calls `updateRSETHPrice()`. `totalETHInProtocol = 1 + D`, `rsethSupply = 1`. With `protocolFeeInBPS == 0` (default in `LRTConfig`, never set in `initialize()`), `protocolFeeInETH = 0`. New price: `newRsETHPrice = (1 + D) * 1e18 / 1 ≈ D * 1e18`. Since `pricePercentageLimit == 0`, no revert occurs: [9](#0-8) 

- Step 5: Victim calls `depositETH{value: Y}(0, "")`. `rsethAmountToMint = Y * 1e18 / (D * 1e18) = Y / D`. For `Y ≤ D`, integer division rounds to 0. The check `0 < 0` is false, so the deposit succeeds. Victim's Y ETH is transferred to the pool; 0 rsETH is minted.

**Why existing checks are insufficient:**
- `_addNewSupportedAsset` enforces `depositLimit != 0`, so the deposit-limit check never blocks a functional deployment: [10](#0-9) 
- `pricePercentageLimit` and `minAmountToDeposit` have no non-zero enforcement at initialization.
- The slippage check is purely caller-supplied and does not enforce a protocol-level minimum of 1.

## Impact Explanation

The victim's ETH is permanently captured in the pool. The attacker holds 1 wei rsETH representing 100% of the rsETH supply, which now backs `1 + D + Y` ETH worth of assets. This is **direct theft of user funds** — matching the Critical impact class. The attacker's net profit is `Y - D` (profitable whenever the victim's deposit exceeds the donation), and the attack is repeatable against every subsequent depositor who passes `minRSETHAmountExpected = 0`.

## Likelihood Explanation

All preconditions hold at default deployment state:
- `receive()` is unconditionally open.
- `updateRSETHPrice()` requires no role.
- `pricePercentageLimit` defaults to 0 (never set in `initialize()`).
- `minAmountToDeposit` defaults to 0 (never set in `initialize()`).
- `protocolFeeInBPS` defaults to 0 (never set in `LRTConfig.initialize()`), so `_checkAndUpdateDailyFeeMintLimit(0)` succeeds even with `maxFeeMintAmountPerDay == 0`.
- Users and integrating contracts commonly pass `minRSETHAmountExpected = 0`.
- The full attack sequence is atomic and front-runnable: an attacker can observe a pending `depositETH` transaction in the mempool and execute steps 1–4 before it lands.

## Recommendation

1. **Enforce a protocol-level minimum rsETH output**: revert in `_beforeDeposit` if `rsethAmountToMint == 0`, regardless of `minRSETHAmountExpected`.
2. **Restrict `receive()`**: only accept ETH from known trusted senders (`NodeDelegator`, `LRTConverter`, `RewardReceiver`); revert on arbitrary donations.
3. **Set `pricePercentageLimit` to a non-zero value during initialization** (e.g., 10% = `1e17`) so the price-increase guard is active from deployment.
4. **Set `minAmountToDeposit` to a meaningful floor** (e.g., 100 gwei) during `initialize()` to prevent dust seed deposits.

## Proof of Concept

```
Preconditions (all default): pricePercentageLimit == 0, minAmountToDeposit == 0,
                              protocolFeeInBPS == 0, maxFeeMintAmountPerDay == 0

1. Attacker calls LRTOracle.updateRSETHPrice()
   → rsethSupply == 0 → rsETHPrice = 1e18, highestRsethPrice = 1e18

2. Attacker calls LRTDepositPool.depositETH{value: 1}(0, "")
   → rsethAmountToMint = 1 * 1e18 / 1e18 = 1
   → rsethSupply = 1 wei, address(LRTDepositPool).balance = 1 wei

3. Attacker sends 1 ETH to LRTDepositPool via receive()
   → address(LRTDepositPool).balance = 1 + 1e18 wei

4. Attacker calls LRTOracle.updateRSETHPrice()
   → totalETHInProtocol = 1 + 1e18
   → newRsETHPrice = (1 + 1e18) * 1e18 / 1 ≈ 1e36
   → pricePercentageLimit == 0 → no revert
   → rsETHPrice = ~1e36

5. Victim calls LRTDepositPool.depositETH{value: 0.999 ether}(0, "")
   → rsethAmountToMint = 0.999e18 * 1e18 / 1e36 = 0 (integer truncation)
   → 0 < 0 is false → no revert
   → Victim's 0.999 ETH transferred to pool, 0 rsETH minted

Result: Attacker holds 1 wei rsETH = 100% of supply backing ~2 ETH.
        Net attacker profit ≈ 0.999 ETH − 1 ETH donation = −0.001 ETH on this victim.
        Attack becomes profitable when victim deposit ≥ donation (e.g., donate 1 ETH, victim deposits ≥ 1 ETH).

Foundry test sketch:
  vm.prank(attacker); oracle.updateRSETHPrice();
  vm.prank(attacker); pool.depositETH{value: 1}(0, "");
  vm.deal(attacker, 1 ether + 1);
  vm.prank(attacker); (bool ok,) = address(pool).call{value: 1 ether}("");
  vm.prank(attacker); oracle.updateRSETHPrice();
  vm.prank(victim);   pool.depositETH{value: 0.999 ether}(0, "");
  assertEq(rsETH.balanceOf(victim), 0);
  assertEq(rsETH.balanceOf(attacker), 1);
```
