Audit Report

## Title
Stale Cached `rsETHPrice` Allows Depositors to Mint Excess rsETH, Stealing Yield from Existing Holders - (File: contracts/LRTDepositPool.sol)

## Summary
`getRsETHAmountToMint` divides a live asset price by a cached `rsETHPrice` storage variable that is only updated when `updateRSETHPrice()` is explicitly called. Because rebasing LSTs such as stETH continuously increase their `balanceOf` between oracle updates, the stale (too-low) `rsETHPrice` causes new depositors to receive more rsETH than their contribution warrants, permanently diluting existing holders and transferring their unclaimed rebase yield to the attacker.

## Finding Description
`getRsETHAmountToMint` (LRTDepositPool.sol L506–521) computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` (LRTOracle.sol L156–158) delegates to a live `IPriceFetcher` oracle on every call. `lrtOracle.rsETHPrice()` (LRTOracle.sol L28) reads a plain storage variable that is only written inside `_updateRsETHPrice()`, which is only reachable via the public `updateRSETHPrice()` (LRTOracle.sol L87–89) or the manager-gated `updateRSETHPriceAsManager()`. A grep of `LRTDepositPool.sol` confirms zero calls to `updateRSETHPrice` anywhere in the deposit contract; neither `depositETH` (L76–93) nor `depositAsset` (L99–118) refreshes the price before invoking `_beforeDeposit` → `getRsETHAmountToMint`.

`_updateRsETHPrice` (LRTOracle.sol L214–316) computes the new price from `_getTotalEthInProtocol()` (L331–349), which in turn calls `ILRTDepositPool.getTotalAssetDeposits(asset)` (LRTDepositPool.sol L385–397). For ERC-20 assets, `getAssetDistributionData` reads `IERC20(asset).balanceOf(address(this))` (L444) — a live, rebasing balance. After a stETH rebase the actual TVL rises immediately, but `rsETHPrice` in storage still reflects the pre-rebase value. The minting formula therefore uses a denominator that is too small, minting excess rsETH.

No existing guard prevents this: `minRSETHAmountExpected` is a slippage floor set by the depositor themselves and cannot protect existing holders; the `pricePercentageLimit` check (LRTOracle.sol L252–267) only fires when `updateRSETHPrice()` is eventually called, not at deposit time.

## Impact Explanation
**High — Theft of unclaimed yield.** stETH rebase yield accrues to the protocol's TVL and belongs to existing rsETH holders. Because `rsETHPrice` is stale, a depositor during the rebase window receives rsETH priced at the pre-rebase rate. When `updateRSETHPrice()` is subsequently called, the new price is computed over a larger supply (including the excess minted tokens), permanently reducing the per-token ETH value for all prior holders. The excess rsETH held by the attacker represents a direct, irreversible transfer of the rebase yield from existing holders to the attacker.

## Likelihood Explanation
stETH rebases approximately every 24 hours. `updateRSETHPrice()` is a separate public call with no atomicity guarantee relative to deposits. Any unprivileged user can call `depositAsset` or `depositETH` at any time; no front-running, governance access, or special role is required. The profitable window (post-rebase, pre-price-update) is a predictable, recurring condition that repeats daily. The attacker need only monitor the stETH rebase event and deposit before the keeper/bot calls `updateRSETHPrice()`.

## Recommendation
Call `updateRSETHPrice()` (or an equivalent internal `_updateRsETHPrice()`) at the beginning of `depositETH` and `depositAsset`, before `getRsETHAmountToMint` is invoked, so that the price used for minting always reflects the current TVL including any rebase that has occurred. Alternatively, compute `rsETHPrice` on-the-fly inside `getRsETHAmountToMint` by calling `_getTotalEthInProtocol()` directly rather than reading the cached storage variable.

## Proof of Concept
**Minimal call sequence (no fork required for logic verification):**

1. Deploy protocol; 100 stETH deposited, 100 rsETH minted. `rsETHPrice = 1e18`.
2. stETH rebases: protocol's `IERC20(stETH).balanceOf(depositPool)` increases to reflect 101 ETH of value. Actual fair price = `101e18 / 100 = 1.01e18`. `rsETHPrice` in storage remains `1e18`.
3. Attacker calls `depositAsset(stETH, 10e18, 0, "")`.
4. `getRsETHAmountToMint` computes `10e18 * 1e18 / 1e18 = 10e18` rsETH minted.
5. Correct amount: `10e18 * 1e18 / 1.01e18 ≈ 9.9009e18` rsETH.
6. Attacker receives ≈ 0.099 excess rsETH at no cost.
7. Keeper calls `updateRSETHPrice()`: new TVL = 111 ETH, new supply = 110 rsETH, new price = `111/110 ≈ 1.009e18`.
8. Original 100 rsETH holders' position is now worth `100 × 1.009e18 ≈ 100.9 ETH` instead of the `101 ETH` they were entitled to — approximately 0.1 ETH of rebase yield permanently transferred to the attacker.

**Foundry fork test plan:** Fork mainnet with stETH as a supported asset. Record `rsETHPrice` before rebase. Advance time by 1 day (triggering stETH rebase via `vm.warp` + `ILido.submit`). Call `depositAsset` without calling `updateRSETHPrice` first. Assert that rsETH minted exceeds `depositAmount * assetPrice / actualPrice`. Then call `updateRSETHPrice` and assert that original holders' ETH-equivalent value is less than their pre-rebase entitlement.
