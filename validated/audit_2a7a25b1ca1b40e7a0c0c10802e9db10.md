### Title
Stale Votes from Removed Participants Count Toward TEE Code-Hash Threshold Without Participant-Role Check — (`crates/contract/src/tee/proposal.rs`)

---

### Summary

`CodeHashesVotes::count_votes` and `LauncherHashVotes::count_votes` tally **all** stored votes without filtering to the current participant set. After a resharing event, votes cast by participants who were removed from the network remain in the map until a detached cleanup promise (`CLEAN_TEE_STATUS`) executes. During the window between resharing completion and cleanup, a single new participant's vote can be combined with stale votes from removed participants to prematurely satisfy the threshold, whitelisting a TEE code hash that was not actually approved by a threshold of current participants.

---

### Finding Description

In `vote_code_hash` (and `vote_add_launcher_hash`), the contract authenticates the caller as a current participant, records their vote, and then calls `self.tee_state.vote(code_hash, &participant)` which internally calls `CodeHashesVotes::count_votes`:

```rust
// crates/contract/src/tee/proposal.rs
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
```

This function counts **every** entry in `proposal_by_account` that matches the proposal hash — including entries from participants who were removed during a prior resharing. The caller-side check in `vote_code_hash` only authenticates the *current* voter; it does not filter the accumulated tally to current participants:

```rust
// crates/contract/src/lib.rs
let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
let votes = self.tee_state.vote(code_hash, &participant);
// ...
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
}
```

Contrast this with `ThresholdParametersVotes::n_votes`, which correctly accepts a `participants` argument and filters:

```rust
// crates/contract/src/primitives/threshold_votes.rs
pub fn n_votes(&self, proposal: &ProposedThresholdParameters, participants: &Participants) -> u64 {
    self.proposal_by_account.iter()
        .filter(|&(acc, prop)| {
            participants.participants().iter().any(|(acc_id, _, _)| acc.get() == acc_id)
                && prop == proposal
        })
        .count() as u64
}
```

The stale votes persist because the cleanup is a **detached, asynchronous promise** spawned in `vote_reshared`:

```rust
// crates/contract/src/lib.rs
Promise::new(env::current_account_id())
    .function_call(method_names::CLEAN_TEE_STATUS.to_string(), ...)
    .detach();
```

Between the resharing completion and the execution of `CLEAN_TEE_STATUS`, other transactions — including `vote_code_hash` — can be processed. The `get_remaining_votes` helper confirms that stale votes are expected to linger:

```rust
// crates/contract/src/tee/proposal.rs
pub fn get_remaining_votes(&self, participants: &Participants) -> Self {
    let remaining = self.proposal_by_account.iter()
        .filter(|(participant_id, _)| {
            participants.is_participant_given_participant_id(&participant_id.get())
        })
        ...
}
```

---

### Impact Explanation

A TEE code hash (or launcher image hash) can be added to the whitelist with fewer than the required threshold of **current** participant votes. Whitelisting a TEE code hash controls which MPC node binaries are permitted to submit valid attestations and participate in the network. If a malicious or compromised code hash is whitelisted, nodes running that binary can pass attestation checks, join the participant set, and participate in threshold signing or DKG rounds — enabling unauthorized signature issuance or key-share exposure.

This maps to: **Medium — participant-state manipulation that breaks production safety/accounting invariants.**

---

### Likelihood Explanation

The attack window is the gap between `vote_reshared` completing and the `CLEAN_TEE_STATUS` detached promise executing (at minimum one NEAR receipt processing cycle, typically within the same block). An attacker who was a participant in epoch N and voted for a target code hash before resharing can coordinate with a single new participant in epoch N+1 to cast the second vote during this window. The coordination requirement and short window make this low-to-medium likelihood, but it is reachable by an unprivileged former participant without any threshold collusion.

---

### Recommendation

Pass the current `Participants` set into `count_votes` and filter to only current participants, mirroring the pattern already used in `ThresholdParametersVotes::n_votes`:

```rust
fn count_votes(&self, proposal: &NodeImageHash, participants: &Participants) -> u64 {
    self.proposal_by_account
        .iter()
        .filter(|(participant_id, prop)| {
            prop == &proposal
                && participants.is_participant_given_participant_id(&participant_id.get())
        })
        .count() as u64
}
```

Apply the same fix to `LauncherHashVotes::count_votes` and any analogous counting function in `MeasurementVotes`. This eliminates the dependency on the asynchronous cleanup promise for correctness of the threshold check.

---

### Proof of Concept

1. **Epoch N** — participants A, B, C; threshold = 2.
2. A calls `vote_code_hash(X)` → `count_votes` returns 1; threshold not reached.
3. Resharing is triggered; A is removed. New epoch N+1 — participants D, E, F; threshold = 2.
4. `vote_reshared` completes and spawns `CLEAN_TEE_STATUS` as a detached promise (not yet executed).
5. D calls `vote_code_hash(X)` in the same or next block, before cleanup runs.
6. `CodeHashesVotes::count_votes` iterates `proposal_by_account`, finds A's stale entry (for X) **and** D's new entry (for X) → returns 2.
7. `2 >= threshold(2)` → `whitelist_tee_proposal(X)` executes; X is whitelisted.
8. Only 1 current participant (D) actually voted; A's vote was from a removed participant who never held power in epoch N+1. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
