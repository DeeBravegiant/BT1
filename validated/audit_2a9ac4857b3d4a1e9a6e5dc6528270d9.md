### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Reward via Unenforced `farmer_reward_address_override` — (`File: chia/farmer/farmer_api.py`)

### Summary

A connected harvester can set `farmer_reward_address_override` to any arbitrary puzzle hash in either `NewProofOfSpace` or `RespondSignatures`, and the farmer will unconditionally use that address as the `farmer_puzzle_hash` in the farmed block. The CHIP-22 fee-quality check is advisory-only (log-only), so a malicious harvester can permanently redirect 100% of the farmer's XCH block reward to itself with no enforcement or consent mechanism.

### Finding Description

CHIP-22 introduced `farmer_reward_address_override` to allow third-party harvesters to take a negotiated fee from the farmer's block reward. The protocol defines a "fee quality" convention: the harvester is only entitled to redirect the reward if a deterministic quality value derived from the proof and challenge falls below a pre-agreed threshold.

**Step 1 — `NewProofOfSpace` handler (farmer_api.py line 128–129):**

When a harvester sends `NewProofOfSpace` with `farmer_reward_address_override != None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`. That function computes the fee quality and compares it to the harvester-supplied `fee_info.applied_fee_threshold`. If the threshold is violated, it emits a `log.warning`. It does **not** reject the proof, does not return early, and does not prevent the block from being built. [1](#0-0) [2](#0-1) 

**Step 2 — `_process_respond_signatures` (farmer_api.py lines 916–919):**

After the farmer requests signatures from the harvester, the harvester's `RespondSignatures` message also carries `farmer_reward_address_override`. The farmer unconditionally replaces its own configured `farmer_target` with whatever address the harvester supplies — no fee-quality check, no threshold comparison, no allowlist check:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
``` [3](#0-2) 

**Step 3 — Block construction (full_node_api.py line 1069):**

The full node uses `request.farmer_puzzle_hash` (which is the `farmer_reward_address` from the `DeclareProofOfSpace` message) directly as the farmer reward destination. No further validation of this address occurs at the full-node level: [4](#0-3) 

The `RespondSignatures` protocol message definition confirms `farmer_reward_address_override` is an unconstrained optional field: [5](#0-4) 

### Impact Explanation

A malicious harvester that has established a connection to a farmer (the normal operating mode for third-party harvesters under CHIP-22, but also applicable to any compromised local harvester) can:

1. Find a valid proof of space.
2. Send `NewProofOfSpace` with `farmer_reward_address_override` set to the attacker's puzzle hash.
3. Respond to `RequestSignatures` with `RespondSignatures` carrying the same override.
4. The farmer builds and submits a block where the farmer reward coin (currently 0.25 XCH) is created at the attacker's puzzle hash.
5. The farmer receives nothing; the attacker receives the full block reward.

This is a direct, permanent, on-chain diversion of XCH farming rewards — matching the "Critical: Unauthorized … reward diversion … affecting XCH" and "High: Bypass of … authorization that enables … payout redirection" impact categories.

### Likelihood Explanation

Any harvester connected to the farmer can exploit this. In the CHIP-22 third-party harvester model, farmers are explicitly expected to connect to external harvesters they do not fully control. The attacker needs no keys, no admin access, and no cryptographic break — only a valid network connection to the farmer's harvester port and the ability to find a proof of space (which is the harvester's normal job).

### Recommendation

The farmer must enforce the CHIP-22 fee convention rather than merely logging violations. Specifically:

1. In `new_proof_of_space`, if `farmer_reward_address_override` is set and `fee_info` is absent or `fee_quality > applied_fee_threshold`, **return early and discard the proof** rather than logging a warning and continuing.
2. In `_process_respond_signatures`, if `response.farmer_reward_address_override` is set but the corresponding `NewProofOfSpace` did not carry a valid, threshold-passing `fee_info`, **reject the response**.
3. Add a farmer-side configurable allowlist of harvester node IDs that are permitted to use `farmer_reward_address_override` at all, defaulting to empty (opt-in).

### Proof of Concept

1. Attacker runs a harvester process and connects it to the victim farmer.
2. When the harvester finds a proof of space, it sends:
   ```python
   NewProofOfSpace(
       ...,
       farmer_reward_address_override=attacker_puzzle_hash,
       fee_info=ProofOfSpaceFeeInfo(applied_fee_threshold=uint32(0xFFFFFFFF))  # or None
   )
   ```
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, which logs a warning (if `fee_info` is present and threshold is violated) or a different warning (if `fee_info` is `None`), then **continues**.
4. The farmer sends `RequestSignatures` to the harvester.
5. The harvester responds:
   ```python
   RespondSignatures(
       ...,
       farmer_reward_address_override=attacker_puzzle_hash
   )
   ```
6. `_process_respond_signatures` sets `farmer_reward_address = attacker_puzzle_hash` unconditionally.
7. `DeclareProofOfSpace` is sent to the full node with `farmer_puzzle_hash = attacker_puzzle_hash`.
8. The full node creates the block; the farmer reward coin is locked to `attacker_puzzle_hash`.
9. The attacker spends the coin at their leisure. The farmer receives 0 XCH for the block they farmed. [1](#0-0) [3](#0-2) [6](#0-5) [7](#0-6) [5](#0-4) [4](#0-3)

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
