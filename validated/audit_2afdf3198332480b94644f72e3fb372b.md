Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ERC20 assets it evaluates `totalAssetDeposits + amount > limit`, but for native ETH it evaluates only `totalAssetDeposits > limit`, omitting the incoming `amount`. This allows any depositor to bypass the configured ETH deposit cap in a single transaction, rendering the protocol's sole on-chain ETH exposure limit ineffective.

## Finding Description
The root cause is confirmed at [1](#0-0)  — the ETH branch on line 679 returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding `amount`, while the ERC20 branch on line 681 correctly includes `amount`. This function is called from `_beforeDeposit` at [2](#0-1) , which gates the public `depositETH` entry point at [3](#0-2) . Because the ETH check uses strict greater-than without the incoming amount, the guard returns `false` (no revert) whenever `totalAssetDeposits <= limit`, regardless of how large the deposit is. No other guard exists to enforce the cap. The deposit limit is the sole on-chain mechanism, stored in `depositLimitByAsset` mapping in `LRTConfig`. [4](#0-3) 

## Impact Explanation
The ETH deposit cap configured by the protocol is rendered entirely ineffective. An unprivileged depositor can mint rsETH against an arbitrarily large ETH deposit in a single call, pushing total ETH exposure in EigenLayer far beyond the intended limit. No user funds are directly stolen, but the protocol fails to deliver its promised invariant (the deposit cap). This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The entry point `depositETH()` is a public payable function with no access control beyond `nonReentrant`, `whenNotPaused`, and `onlySupportedAsset`. [5](#0-4)  The bypass condition — `getTotalAssetDeposits(ETH_TOKEN) <= depositLimitByAsset(ETH_TOKEN)` — is true for the entire operational lifetime of the protocol until the cap is already breached. No flash loan, privileged role, multi-step sequence, or special precondition is required. **Likelihood: High.**

## Recommendation
Unify the check to include `amount` for both ETH and ERC20, mirroring the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. Legitimate users deposit until `getTotalAssetDeposits(ETH_TOKEN) == 99 ether`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit`: `99 ether > 100 ether` → `false` → no revert. [6](#0-5) 
5. `_mintRsETH` mints rsETH for 10,000 ETH; total ETH in protocol becomes 10,099 ETH — 100× the intended cap. [7](#0-6) 
6. **Foundry test plan:** Deploy `LRTDepositPool` on a local fork, configure `depositLimitByAsset[ETH_TOKEN] = 100 ether`, deposit 99 ether from a legitimate address, then call `depositETH{value: 10_000 ether}` from an attacker address and assert the call succeeds and `getTotalAssetDeposits(ETH_TOKEN)` returns `10_099 ether`, confirming the cap was bypassed.
