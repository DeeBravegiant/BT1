### Title
Unconditional Relayer Fee Transfer Drains User NEAR Balance Even on Failed ERC-20 (`ft_transfer`) Execution - (File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs)

### Summary
In the `near-wallet-contract`, the relayer fee for emulated ERC-20 transfers is dispatched as an unconditional NEAR transfer to the relayer **before** the `ft_transfer` promise is created and before its result is known. If the subsequent `ft_transfer` call fails for any reason (insufficient FT balance, unregistered receiver, contract panic, etc.), the user's NEAR balance is permanently reduced by the fee amount with no refund, while the FT transfer itself never completes.

### Finding Description
In `inner_rlp_execute` (`lib.rs` lines 374–384), when the transaction kind is `EthEmulationKind::ERC20Transfer`, the code unconditionally creates a promise to transfer the fee to the relayer (`context.predecessor_account_id`) before the actual `ft_transfer` action promise is dispatched:

```rust
if let TransactionKind::EthEmulation(EthEmulationKind::ERC20Transfer { fee, .. }) =
    &transaction_kind
{
    if !fee.is_zero() && context.predecessor_account_id != context.current_account_id {
        let refund_promise = env::promise_batch_create(&context.predecessor_account_id);
        env::promise_batch_action_transfer(refund_promise, *fee);  // unconditional
    }
}
```

This fee promise is independent of the `ft_transfer` promise that follows. In NEAR's execution model, independent promise batches execute regardless of each other's outcomes. The `rlp_execute_callback` (lines 296–316) correctly detects a failed `ft_transfer` and returns `success: false`, and even refunds the `caller_deposit` if present — but it does **not** and **cannot** reverse the already-dispatched fee transfer to the relayer.

The fee amount is derived from the user's signed Ethereum transaction fields (`gas_price * gas_limit * MAX_YOCTO_NEAR`), computed in `internal.rs` lines 59–64:

```rust
let tx_fee = {
    let wei_amount = tx.max_fee_per_gas.saturating_mul(tx.gas_limit).min(VALUE_MAX).low_u128();
    NearToken::from_yoctonear(wei_amount.saturating_mul(MAX_YOCTO_NEAR as u128))
};
```

A user who sets a meaningful fee (e.g., to incentivize fast relayer inclusion) and whose `ft_transfer` fails for any reason loses that NEAR permanently.

### Impact Explanation
The corrupted protocol value is the **user's NEAR account balance**: it is reduced by the fee amount even when the FT transfer produces no effect. The `ft_transfer` failure leaves the user's NEP-141 token balance unchanged, but the NEAR fee is irrecoverably sent to the relayer. This is a direct balance discrepancy: the user pays for a service (FT transfer) that was never delivered.

### Likelihood Explanation
Any scenario where `ft_transfer` can fail triggers this loss:
- User's FT balance is insufficient at execution time (race condition with another transfer)
- The NEP-141 contract panics or returns an error
- The receiver is not registered and `storage_deposit` in the chained path fails
- The token contract has non-standard behavior

A relayer who observes that a user's FT balance is insufficient can still submit the signed transaction, collect the fee, and leave the user with a failed transfer. Since anyone can operate a relayer, this is reachable by an unprivileged external actor.

### Recommendation
The relayer fee transfer should be conditioned on the success of the `ft_transfer` promise. One approach is to move the fee payment into `rlp_execute_callback` and only dispatch it when `env::promise_result(0)` is `PromiseResult::Successful`. The fee amount would need to be passed as a callback argument (similar to how `caller_deposit` is already passed). This ensures the relayer is only compensated when the user's intended action actually completes.

### Proof of Concept
1. User signs an ERC-20 transfer Ethereum transaction with `gas_price > 0` and `gas_limit > 0`, producing a non-zero `tx_fee`.
2. User's FT balance on the NEP-141 contract is 0 (or becomes 0 before execution).
3. Relayer submits the transaction via `rlp_execute`.
4. `inner_rlp_execute` dispatches `promise_batch_action_transfer(relayer, fee)` unconditionally (lib.rs line 383).
5. The `ft_transfer` promise is dispatched and fails with insufficient balance.
6. `rlp_execute_callback` returns `ExecuteResponse { success: false, ... }`.
7. Result: relayer's NEAR balance increased by `fee`; user's NEAR balance decreased by `fee`; user's FT balance unchanged. The user paid for a transfer that never happened. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
