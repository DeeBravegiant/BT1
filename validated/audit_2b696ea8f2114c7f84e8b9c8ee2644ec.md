Audit Report

## Title
Pool Deposit Functions Allow Zero rsETH Minting When Deposit Amount Rounds to Zero - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All pool `deposit` functions guard only against `amount == 0` but never validate that the computed `rsETHAmount > 0`. Because `viewSwapRsETHAmountAndFee` performs integer division, any deposit where `amountAfterFee * 1e18 < rsETHToETHrate` truncates `rsETHAmount` to zero. The transaction succeeds, the deposited ETH or tokens are retained by the pool, and the caller receives zero wrsETH with no on-chain recourse.

## Finding Description
In `RSETHPoolV3.deposit(string)`, the only input guard is:

```solidity
if (amount == 0) revert InvalidAmount();
``` [1](#0-0) 

`viewSwapRsETHAmountAndFee` then computes:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [2](#0-1) 

When `amountAfterFee * 1e18 < rsETHToETHrate`, Solidity integer division truncates `rsETHAmount` to `0`. With `feeBps = 0` and a realistic post-accrual rate of `rsETHToETHrate = 1.1e18`, depositing `1 wei` yields `rsETHAmount = 1 * 1e18 / 1.1e18 = 0`. The function then proceeds unconditionally to:

```solidity
wrsETH.mint(msg.sender, rsETHAmount);  // mints 0 tokens
``` [3](#0-2) 

For the ETH path, the deposited ETH sits in the contract balance and is swept to L1 by `BRIDGER_ROLE` via `getETHBalanceMinusFees()`. For the token path, `safeTransferFrom` executes before the mint, so tokens are transferred to the pool before `rsETHAmount = 0` is discovered — the tokens are permanently retained. [4](#0-3) 

The identical pattern is present across all pool variants: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) 

## Impact Explanation
**Low — Contract fails to deliver promised returns.** A depositor sending a non-zero but sub-threshold amount has their ETH or tokens accepted and permanently retained by the pool while receiving zero wrsETH. The user has no claim on the deposited funds and cannot recover them. The ETH is swept to L1 by `BRIDGER_ROLE`; deposited tokens are similarly inaccessible to the user. The contract accepts the deposit but does not deliver the promised output token.

## Likelihood Explanation
Low. The threshold is extremely small — for ETH with `feeBps = 0` and `rsETHToETHrate ≈ 1e18`, any deposit under `rsETHToETHrate / 1e18 = ~1 wei` triggers the issue. A normal user would not intentionally deposit 1 wei. However, the path is reachable by: (1) a contract integration that does not validate the output amount before calling `deposit`; (2) an automated script depositing dust amounts; (3) a user on a token path where the token has a lower oracle rate, raising the threshold to a non-trivial amount. No privileged access is required — any external caller can trigger this.

## Recommendation
Add a post-computation output guard in each `deposit` function immediately after calling `viewSwapRsETHAmountAndFee`:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This must be applied to both the ETH and token deposit paths in all pool variants: `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, `RSETHPoolNoWrapper`, `RSETHPool`, and `AGETHPoolV3`.

## Proof of Concept
1. Deploy `RSETHPoolV3` with `feeBps = 0` and an oracle returning `rsETHToETHrate = 1.1e18`.
2. Call `deposit("")` with `msg.value = 1 wei`.
3. `viewSwapRsETHAmountAndFee(1)` returns `(rsETHAmount=0, fee=0)` — integer division truncates.
4. `feeEarnedInETH += 0` — no fee recorded.
5. `wrsETH.mint(msg.sender, 0)` executes — caller receives 0 wrsETH.
6. The 1 wei remains in `address(this).balance`, counted as bridgeable ETH via `getETHBalanceMinusFees()`, and will be swept to L1 by `BRIDGER_ROLE`.
7. Caller has permanently lost 1 wei with no on-chain recourse.

For the token path: repeat with a supported ERC-20 token where `amount * tokenToETHRate / rsETHToETHrate = 0`; `safeTransferFrom` executes at step 2 before the zero-output is discovered, making the token loss immediate upon transfer. [10](#0-9) [11](#0-10)
