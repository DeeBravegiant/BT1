## Race Window

The `RpcHandlerActor` is explicitly multithreaded: [7](#0-6) 

The sequence:
1. Epoch boundary block B is accepted; `Client` calls `reshard()` on the pool (migrating all old-UID entries to new UIDs).
2. A concurrent RPC thread already read `head = chain_store.head()` pointing to block B-1 (still in old epoch).
3. That thread computes `shard_uid` using the old shard layout, acquires the pool mutex, and calls `insert_transaction(old_uid, tx)`.
4. The transaction is stored under the old `ShardUId` key.
5. The chunk producer for the new epoch calls `get_pool_iterator(new_uid)` → `None` for the old key → transaction never drained.

## Assessment

### Title
Race at Resharding Epoch Boundary Causes Transactions to Be Admitted as ValidTx but Never Included — (`chain/client/src/client.rs`, `chain/client/src/rpc_handler.rs`)

### Summary
At every resharding epoch boundary, there is a concrete race window where the multithreaded `RpcHandlerActor` can insert a transaction under the old-epoch `ShardUId` after `ShardedTransactionPool::reshard()` has already migrated all entries to new-epoch keys. The transaction is confirmed as `ValidTx` but is permanently orphaned in the pool under a key that the chunk producer never drains.

### Finding Description
`ShardedTransactionPool` is a `HashMap<ShardUId, TransactionPool>`. `ShardUId` encodes both the shard layout version and the shard ID. When the shard layout changes at an epoch boundary, `Client::on_block_accepted_with_optional_chunk_produce` calls `guarded_pool.reshard(&old, &new)`, which drains all old-UID pools and re-inserts under new UIDs. However, the `RpcHandlerActor` runs on separate threads and reads `head` independently. A thread that read `head` before the epoch boundary block became the canonical head will compute `shard_uid` from the old layout and insert into the pool after `reshard()` has completed, creating a new old-UID entry that `reshard()` will never see again. The code acknowledges this at `chain/client/src/client.rs:1890` with a TODO comment.

### Impact Explanation
Any transaction submitted near a resharding epoch boundary that lands in this race window is:
- Accepted and returned as `ProcessTxResponse::ValidTx` to the submitter.
- Stored in the pool under the old `ShardUId`.
- Never returned by `get_pool_iterator(new_shard_uid)` in `prepare_transactions`.
- Never included in any chunk until the transaction's validity period expires.

### Likelihood Explanation
Resharding epoch boundaries are rare but deterministic events. The race window is the interval between `reshard()` completing and the `RpcHandlerActor` threads observing the new canonical head. Under load (many concurrent RPC threads), this window is non-negligible. An unprivileged user submitting transactions continuously near a known resharding boundary will reliably trigger this for some fraction of their submissions.

### Recommendation
After calling `reshard()`, the node must prevent the `RpcHandlerActor` from inserting under old-layout shard UIDs. Options include: (a) storing the current canonical shard layout in a shared atomic reference that `process_tx_internal` reads instead of re-deriving from `head`; (b) after `reshard()`, re-checking any transactions inserted since the epoch boundary and migrating them; or (c) adding a post-insert validation step that rejects insertions whose `shard_uid.version()` does not match the current layout version.

### Proof of Concept
Insert a transaction with `ShardUId::new(old_version, shard_id)` into a `ShardedTransactionPool`, call `reshard(&old_layout, &new_layout)`, then insert another transaction with `ShardUId::new(old_version, shard_id)` (simulating the race), and verify that `get_pool_iterator(ShardUId::new(new_version, shard_id))` does not return the second transaction. The `test_transaction_pool_resharding` test in `chain/chunks/src/client.rs` already validates the happy path but does not cover post-reshard insertions. [8](#0-7) [4](#0-3)

    pub fn reshard(&mut self, old_shard_layout: &ShardLayout, new_shard_layout: &ShardLayout) {
        tracing::debug!(
            target: "resharding",
            old_shard_layout_version = old_shard_layout.version(),
            new_shard_layout_version = new_shard_layout.version(),
            "resharding the transaction pool"
        );
        debug_assert!(old_shard_layout != new_shard_layout);

        let mut validated_txs = vec![];

        for old_shard_uid in old_shard_layout.shard_uids() {
            if let Some(mut iter) = self.get_pool_iterator(old_shard_uid) {
                while let Some(group) = iter.next() {
                    while let Some(validated_tx) = group.next() {
                        validated_txs.push(validated_tx);
                    }
                }
            }
        }

        for validated_tx in validated_txs {
            let signer_id = validated_tx.signer_id();
            let new_shard_uid = new_shard_layout.account_id_to_shard_uid(signer_id);
            self.insert_transaction(new_shard_uid, validated_tx);
        }
    }
