Based on my research, I found a directly analogous vulnerability in the Dijkstra era's sub-transaction processing.

### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund — (`eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

### Summary

The Dijkstra era introduces nested (sub-)transactions. When a top-level transaction contains multiple sub-transactions, the cert state (specifically the accounts/deposit tracking) is not updated between sub-transaction iterations. This is the direct analog of the BerabotRouter bug: just as `amountIn` was not updated between loop iterations, the intermediate cert state is not propagated from one sub-transaction to the next. As a result, two sub-transactions in the same batch can each claim the deposit refund for the same staking credential, effectively creating ADA from nothing.

### Finding Description

The Dijkstra era `dijkstraUtxoTransition` processes a top-level transaction and its sub-transactions. The environment passed to the rule contains `certState` — the original cert state before any sub-transactions are applied: [1](#0-0) 

The value-conservation check for the top-level transaction body uses this original `certState`: [2](#0-1) 

Sub-transactions are then processed through the `UTXOS` rule: [3](#0-2) 

Because the cert state is not updated between sub-transaction iterations (unlike the `validateBatchWithdrawals` function which correctly aggregates all withdrawals), two sub-transactions can each independently present an `UnRegDepositTxCert` for the same staking credential and each receive the deposit refund. The predicate failure that should prevent this is explicitly acknowledged as unimplemented by the developers: [4](#0-3) 

The test is marked `xit` (disabled/pending) with `error "TODO: predicate failure not yet implemented"` as the expected failure — confirming the check does not exist in the current codebase.

Compare this to `validateBatchWithdrawals`, which correctly aggregates withdrawals across the entire batch before checking against the original balance: [5](#0-4) 

No equivalent cross-sub-transaction deduplication exists for deposit refunds.

### Impact Explanation

An attacker can craft a top-level transaction containing N sub-transactions that each include `UnRegDepositTxCert` for the same staking credential. Each sub-transaction independently passes its value-conservation check using the original cert state (where the credential is still registered and the deposit is still present). The deposit pot (`utxosDeposited`) is debited N times for a single deposit, and the attacker receives N × `keyDeposit` ADA in outputs while only one deposit was ever made. This is a direct creation of ADA through an invalid ledger state transition.

**Impact: Critical** — Direct creation of ADA through an invalid ledger state transition.

### Likelihood Explanation

The Dijkstra era is production-targeted code in the repository. The vulnerability requires only the ability to submit a transaction (unprivileged), and the attack is straightforward to construct. The developers have already identified the missing check and left a `TODO`, confirming the gap is real and reachable.

### Recommendation

Implement cross-sub-transaction deduplication for deposit refunds, analogous to how `validateBatchWithdrawals` aggregates withdrawals across the batch. Specifically, before processing sub-transactions, collect all `UnRegDepositTxCert` (and `UnRegDRepTxCert`) certificates across all sub-transactions and verify that no staking credential appears in more than one sub-transaction's deregistration certificates. Alternatively, thread the updated cert state through each sub-transaction iteration so that the second sub-transaction sees the credential as already unregistered.

### Proof of Concept

1. Register staking credential `C` with deposit `D`.
2. Construct `subTx1` with `UnRegDepositTxCert C D` and a UTxO input worth `V1`.
3. Construct `subTx2` with `UnRegDepositTxCert C D` and a UTxO input worth `V2`.
4. Construct top-level `tx` containing `[subTx1, subTx2]`.
5. Submit `tx`. Both sub-transactions pass their individual value-conservation checks against the original cert state (where `C` is still registered). The deposit pot is debited `2D` but only `D` was deposited. The attacker receives `D` ADA of free value.

The disabled test at `CertSpec.hs:53–75` directly encodes this scenario and confirms no predicate failure currently prevents it. [4](#0-3) [6](#0-5)

validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL

      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr

                    else Nothing
          )

  TRC (DijkstraUtxoEnv slot pp certState originalUtxo, utxos, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG
  -- this is the original Accounts, before any transactions were applied
  let accounts = certState ^. certDStateL . accountsL

  let txBody = tx ^. bodyTxL

  {- inInterval (SlotOf Γ) (ValidIntervalOf txTop) -}
  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  ei <- liftSTS $ asks epochInfo

  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  {- SpendInputs ≠ ∅ -}
  runTestOnSignal $ Shelley.validateInputSetEmptyUTxO txBody

  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxos) inputs
