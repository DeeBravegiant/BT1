Audit Report

## Title
Unconditional Zero-Value Fee Transfer in `claim()` Permanently Freezes All User Claims - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` at line 144 regardless of whether `fee` is zero. When `feeInBPS` is set to `0` (a valid configuration) and the distributed token reverts on zero-value transfers, every invocation of `claim()` reverts, permanently freezing all unclaimed tokens for all users with no alternative withdrawal path.

## Finding Description
In `contracts/utils/MerkleDistributor/MerkleDistributor.sol`, the `claim()` function computes the fee and unconditionally executes two `safeTransfer` calls:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;   // line 138
uint256 amountToSend = claimableAmount - fee;           // line 139

IERC20(token).safeTransfer(account, amountToSend);     // line 141
IERC20(token).safeTransfer(protocolTreasury, fee);     // line 144 — always called
``` [1](#0-0) 

When `feeInBPS == 0`, `fee` evaluates to `0`, and `safeTransfer(protocolTreasury, 0)` is called unconditionally. ERC-20 tokens that enforce `require(amount > 0)` will revert on this call, causing the entire transaction to revert.

`feeInBPS` can legitimately be zero in two ways:
1. **Initialization** — `initialize()` only rejects `_feeInBPS > MAX_FEE_IN_BPS` (1000), so `_feeInBPS = 0` is accepted. [2](#0-1) 
2. **Post-deployment** — `setFeeInBPS(0)` passes the same validation. [3](#0-2) 

The sibling contract `KernelMerkleDistributor` already guards against this exact scenario with a zero-check before the fee transfer, confirming the pattern is known within the codebase: [4](#0-3) 

The fix was applied to `KernelMerkleDistributor` but was not applied to `MerkleDistributor`.

## Impact Explanation
When triggered, no user can claim any tokens from the distributor. The tokens remain locked in the contract with no alternative withdrawal path for users. This constitutes **permanent freezing of unclaimed yield** — a Medium-severity impact within the allowed scope.

## Likelihood Explanation
Two conditions must hold simultaneously:
1. `feeInBPS` is set to `0` — this is a normal operational choice (zero-fee distribution), explicitly permitted by the contract's validation logic. No privileged compromise is required; it is a routine admin configuration.
2. The configured `token` reverts on zero-value transfers — a known property of several deployed ERC-20 tokens (e.g., `LEND`, `cUSDCv3`).

Neither condition is adversarial. The combination is realistic whenever the protocol deploys a zero-fee distributor for a token with this transfer restriction. Any user calling `claim()` with a valid proof will trigger the revert.

## Recommendation
Guard the fee transfer with a zero-check, mirroring the pattern already used in `KernelMerkleDistributor`:

```solidity
// Before (vulnerable):
IERC20(token).safeTransfer(protocolTreasury, fee);

// After (fixed):
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
``` [4](#0-3) 

## Proof of Concept
1. Deploy a mock ERC-20 token that reverts on zero-value transfers (`require(amount > 0, "zero transfer")`).
2. Deploy `MerkleDistributor` via proxy with `feeInBPS = 0`, the mock token, and a valid `protocolTreasury`.
3. Owner calls `setMerkleRoot(root)` with a valid distribution root; `currentIndex` becomes `1`.
4. Fund the distributor with the mock token.
5. Any user calls `claim(1, account, cumulativeAmount, merkleProof)` with a valid proof and non-zero `cumulativeAmount`.
6. Execution reaches line 138: `fee = (claimableAmount * 0) / 10_000 = 0`.
7. Line 141 succeeds: `safeTransfer(account, claimableAmount)`.
8. Line 144 reverts: `safeTransfer(protocolTreasury, 0)` — the token rejects the zero-value transfer.
9. The entire transaction reverts. Every subsequent `claim()` call hits the same revert path. All tokens are permanently frozen in the contract.
