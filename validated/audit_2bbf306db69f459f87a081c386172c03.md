### Title
Unchecked `sub_epoch_n` Index Into `summaries` List Allows Malicious Peer to Crash Weight Proof Validation — (`File: chia/full_node/weight_proof.py`)

### Summary
In `_validate_sub_epoch_segments`, the loop variable `sub_epoch_n` is derived entirely from attacker-controlled weight proof bytes, while `summaries` is a fixed-length list built from the validated `sub_epochs` field. There is no bounds check before `summaries[sub_epoch_n]` is accessed, so a crafted weight proof with a segment whose `sub_epoch_n` ≥ `len(summaries)` raises an unhandled `IndexError`, crashing or aborting weight proof validation for any syncing node or wallet that receives it.

### Finding Description
`_validate_sub_epoch_segments` deserializes two independent structures from the incoming `WeightProof`:

1. `summaries` — built from `weight_proof.sub_epochs` via `_validate_sub_epoch_summaries` / `summaries_from_bytes`. Its length is determined by the number of sub-epoch summary entries the prover included.
2. `sub_epoch_segments` — deserialized from `weight_proof_bytes` (the `SubEpochSegments` blob). Each `SubEpochChallengeSegment` carries a `sub_epoch_n` field that is fully attacker-controlled.

The loop at line 969 iterates over `segments_by_sub_epoch.items()`, where the key is the `sub_epoch_n` read from the segment. At lines 974, 983, and 985 the code indexes into `summaries` using that key without any prior bounds check:

```python
for sub_epoch_n, segments in segments_by_sub_epoch.items():
    ...
    curr_difficulty, curr_ssi = _get_curr_diff_ssi(constants, sub_epoch_n, summaries)  # line 974
    ...
    prev_ses = summaries[sub_epoch_n - 1]   # line 983
    ...
    if not summaries[sub_epoch_n].reward_chain_hash == rc_sub_slot_hash:  # line 985
```

If an attacker sets `sub_epoch_n` to any value ≥ `len(summaries)`, Python raises `IndexError`. This exception is not caught anywhere in `_validate_sub_epoch_segments`, `validate_weight_proof_inner`, or the `WalletWeightProofHandler.validate_weight_proof` call chain, so it propagates as an unhandled exception, aborting the sync attempt and potentially crashing the process.

### Impact Explanation
Any unprivileged peer can send a crafted `WeightProof` to a syncing full node or wallet. The syncing node calls `validate_weight_proof` → `validate_weight_proof_inner` → `_validate_sub_epoch_segments`. The `IndexError` propagates through the async call chain. At minimum this permanently prevents the victim from syncing from that peer; if the exception is not caught at a higher level (no try/except wraps the weight proof validation in the sync path), it can crash the node or wallet process entirely, causing a long-lived inability to sync.

This maps to: **High — Permanent or long-lived inability for honest nodes or wallets to process sync updates under normal network assumptions.**

### Likelihood Explanation
- The attacker needs only a network connection to a syncing peer; no keys, no stake, no privileged access.
- Crafting a `WeightProof` with an out-of-range `sub_epoch_n` in a `SubEpochChallengeSegment` requires only knowledge of the serialization format (publicly documented Streamable types).
- The check at line 970 (`len(segments) > max_segments`) only limits the number of segments per sub-epoch, not the value of `sub_epoch_n` itself.
- No existing guard prevents `sub_epoch_n` from exceeding `len(summaries) - 1`.

### Recommendation
Add an explicit bounds check at the top of the `for` loop body in `_validate_sub_epoch_segments`:

```python
for sub_epoch_n, segments in segments_by_sub_epoch.items():
    if sub_epoch_n >= len(summaries):
        log.error(f"sub_epoch_n {sub_epoch_n} out of range (summaries len={len(summaries)})")
        return None
    ...
```

Apply the same guard before `summaries[sub_epoch_n - 1]` (line 983) to handle `sub_epoch_n == 0` edge cases as well.

### Proof of Concept
1. Obtain a valid `WeightProof` from the network.
2. Deserialize `sub_epoch_segments`; take any `SubEpochChallengeSegment` and replace its `sub_epoch_n` field with a value equal to `len(weight_proof.sub_epochs)` (i.e., one past the end of the summaries list).
3. Re-serialize and send the modified `WeightProof` to a syncing node via the `RespondProofOfWeight` protocol message.
4. The node calls `_validate_sub_epoch_segments`; at line 985, `summaries[sub_epoch_n]` raises `IndexError` because `sub_epoch_n == len(summaries)`.
5. The unhandled exception aborts weight proof validation; the node fails to complete sync.
