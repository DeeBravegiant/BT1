Audit Report

## Title
Rewards permanently frozen when `totalKernelStaked` drops to zero mid-period via `initiateWithdrawal` - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` inherits the classic Synthetix staking rewards flaw: when `totalKernelStaked` reaches zero during an active reward window, `rewardPerToken()` returns a stale value while the `updateReward` modifier unconditionally advances `updatedAt`. Every reward token accrued during the zero-supply interval (`rewardRate × Δt`) becomes permanently unclaimable with no recovery mechanism. The contract's own NatSpec explicitly acknowledges this outcome.

## Finding Description
`rewardPerToken()` short-circuits and returns the stored (stale) `rewardPerTokenStored` whenever `totalKernelStaked == 0`: [1](#0-0) 

The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` on every invocation, regardless of whether `totalKernelStaked` is zero: [2](#0-1) 

The only on-chain guard is in `notifyRewardAmount`, which reverts if `totalKernelStaked == 0` at reward-notification time: [3](#0-2) 

This guard is insufficient. It only prevents starting a reward period with zero stakers. It does not prevent `totalKernelStaked` from dropping to zero during an already-active period. `initiateWithdrawal` decrements `totalKernelStaked` immediately with no floor check: [4](#0-3) 

Once the last staker calls `initiateWithdrawal`, every subsequent `updateReward` invocation (triggered by `stake`, `stakeFor`, `initiateWithdrawal`, or `getReward`) advances `updatedAt` without increasing `rewardPerTokenStored`. The reward tokens for the entire zero-supply interval are permanently unclaimable. The contract's own NatSpec explicitly acknowledges this outcome and states the mitigation is purely operational: [5](#0-4) 

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** For every second `totalKernelStaked == 0` during an active reward window, exactly `rewardRate` reward tokens become permanently irrecoverable. There is no admin rescue function, no sweep mechanism, and no redistribution path. This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation
No privileged access, oracle manipulation, or front-running is required. Any user who is the last staker can trigger this by calling `initiateWithdrawal` for their full balance. The `withdrawalDelay` mechanism does not prevent it — `totalKernelStaked` is decremented at `initiateWithdrawal` time, not at `claimWithdrawal` time. The scenario is a standard user action with no coordination required if only one staker remains.

## Recommendation
Do not advance `updatedAt` in the `updateReward` modifier when `totalKernelStaked == 0`. This preserves the time gap so rewards are correctly distributed once stakers return:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 100 days` while Alice is the sole staker. `rewardRate = 1_000e18 / (100 days)`, `finishAt = now + 100 days`, `updatedAt = now`.
2. Alice calls `initiateWithdrawal(aliceBalance)`. `totalKernelStaked` becomes 0. `updateReward` fires: `rewardPerTokenStored` stays stale (supply is 0), `updatedAt` advances to `now`.
3. 50 days pass with no stakers. `totalKernelStaked == 0` the entire time.
4. Bob calls `stake(1e18)`. `updateReward` fires: `rewardPerToken()` returns stale `rewardPerTokenStored` (supply was 0), `updatedAt` advances to `now + 50 days`.
5. The 50 days × `rewardRate` = 500e18 reward tokens are permanently unclaimable. Bob only earns rewards from day 50 onward. The 500e18 tokens remain locked in the contract with no recovery path.

**Foundry test plan:** Deploy `KernelDepositPool`, stake as Alice, call `notifyRewardAmount`, call `initiateWithdrawal` as Alice (draining `totalKernelStaked` to 0), `warp` 50 days, stake as Bob, `warp` to `finishAt`, call `getReward` as Bob, assert Bob's claimed rewards equal only ~500e18 and that the contract's `rewardsToken` balance retains the other ~500e18 with no claimant.
