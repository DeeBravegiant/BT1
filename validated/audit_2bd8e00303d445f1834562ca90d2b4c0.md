Let me look at the `PendingData` struct to confirm whether `starknet_version` is available but ignored.

### Title
Pending-block `estimate_fee` always uses `StarknetVersion::LATEST` versioned constants, ignoring the pending block's actual `starknet_version` — (`crates/apollo_rpc_execution/src/lib.rs`)

---

### Summary

`create_block_context` in `crates/apollo_rpc_execution/src/lib.rs` resolves the `StarknetVersion` for fee estimation by querying storage for the pending block number. Because the pending block has not been committed yet, `get_starknet_version` always returns `None` for it, and the code falls back unconditionally to `StarknetVersion::LATEST`. The actual `starknet_version` string carried inside `PendingData.block` is never consulted. Any public caller of `estimate_fee` or `simulate_transactions` with `BlockId::Tag(Tag::Pending)` therefore receives fee estimates computed against the wrong versioned constants.

---

### Finding Description

**Step 1 — pending block number is always beyond the header marker.**

In `create_block_context`, when `maybe_pending_data` is `Some`, `block_number` is set to `block_context_number.unchecked_next()`: [1](#0-0) 

This is the next block number after the latest stored block — it has not been written to storage yet.

**Step 2 — `get_starknet_version` returns `None` for any block at or beyond the header marker.** [2](#0-1) 

**Step 3 — the fallback is unconditionally `StarknetVersion::LATEST`.** [3](#0-2) 

**Step 4 — `VersionedConstants::get` is called with the wrong version.** [4](#0-3) 

**Step 5 — the pending block's actual `starknet_version` is available but never used.**

`PendingBlockOrDeprecated` exposes a `starknet_version()` accessor: [5](#0-4) 

Both `DeprecatedPendingBlock` and `PendingBlock` carry the field: [6](#0-5) [7](#0-6) 

`create_block_context` extracts every other field from `pending_data` (timestamp, gas prices, sequencer, DA mode) but never reads `starknet_version` from it.

**Step 6 — the constants differ materially across versions.**

An existing test in the codebase confirms the divergence: [8](#0-7) 

`invoke_tx_max_n_steps` is 3 000 000 for v0.13.0 and 10 000 000 for v0.13.2. Validate step limits, sierra gas limits, and syscall gas cost tables also differ across the 14 registered versions. [9](#0-8) 

---

### Impact Explanation

Any unprivileged user who calls `estimate_fee` or `simulate_transactions` with `Tag::Pending` receives a fee estimate computed with `StarknetVersion::LATEST` versioned constants regardless of what version the pending block actually declares. When the pending block's version is older than LATEST (e.g., during a protocol upgrade window), the returned `gas_consumed` and `overall_fee` are wrong — they reflect step limits and gas cost tables that the sequencer will not actually apply when it executes the transaction. This is a public RPC endpoint returning an authoritative-looking wrong value.

Impact category: **High** — "RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."

---

### Likelihood Explanation

The mismatch is latent and silent during steady-state operation (when the live sequencer version equals LATEST). It becomes active during every protocol upgrade window — the period between when the sequencer starts producing blocks at a new version and when the node binary is updated to make that version its LATEST. Given that Starknet has had 14 versioned constant sets across its history, this window recurs with every release.

---

### Recommendation

In `create_block_context`, when `maybe_pending_data` is `Some`, parse the version string from `pending_data.block.starknet_version()` and use it (with a fallback to LATEST for unrecognised future versions, mirroring the pattern already used in `starknet_version_or_latest`) instead of querying storage:

```rust
let starknet_version = match maybe_pending_data {
    Some(pending_data) => {
        StarknetVersion::try_from(pending_data.block.starknet_version())
            .unwrap_or(StarknetVersion::LATEST)
    }
    None => storage_reader
        .begin_ro_txn()?
        .get_starknet_version(block_number)?
        .unwrap_or(StarknetVersion::LATEST),
};
```

### Proof of Concept

1. Store one block with `starknet_version = StarknetVersion::V0_13_0` in the test storage.
2. Populate `PendingData` with `starknet_version = "0.13.0"` and a parent hash matching that block.
3. Call `estimate_fee` (or `execute_transactions`) with `Tag::Pending` for a DeclareV2 transaction whose Sierra compilation requires more than 3 000 000 steps but fewer than 10 000 000.
4. Under the bug, the call succeeds and returns a non-zero fee (LATEST constants allow 10 M steps).
5. Under the correct behaviour, the call should fail or return a reverted result (v0.13.0 constants cap at 3 M steps).

The existing test at `crates/apollo_rpc_execution/src/execution_test.rs:839` already demonstrates that `VersionedConstants::get(&V0_13_0).invoke_tx_max_n_steps == 3_000_000` while LATEST is `10_000_000`, confirming the constants diverge enough to change execution outcomes.

fn test_get_versioned_constants() {
