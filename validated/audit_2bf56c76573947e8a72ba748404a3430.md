### Title
Missing RSA Key Size Validation in `parse_ek_blob` Allows Weak-Key PVE Encryption to Succeed - (`src/cbmpc/api/pve_internal.h`, `src/cbmpc/api/pve_base_pke.cpp`)

### Summary

`parse_ek_blob` only validates the blob version field. It does not validate that the deserialized `rsa_pub_key_t` inside a `key_type=rsa_oaep_2048` blob is actually 2048 bits. An attacker who supplies a crafted EK blob containing a 1024-bit RSA public key causes the entire `cbmpc_pve_encrypt` path to succeed, producing a structurally valid PVE ciphertext encrypted under a 1024-bit key.

### Finding Description

**Step 1 — Entry point: `parse_ek_blob`** [1](#0-0) 

`parse_ek_blob` deserializes the blob and checks only `version == 1`. There is no post-deserialization check that `blob.rsa_ek.size() == RSA_KEY_LENGTH / 8`.

**Step 2 — `rsa_pub_key_t::convert` accepts any modulus size** [2](#0-1) 

The deserializer reconstructs the key from raw `n` and `e` bignums with no bit-length check. A 1024-bit modulus is accepted identically to a 2048-bit one.

**Step 3 — `unified_key_blob_base_pke_t::encrypt` dispatches without size guard** [3](#0-2) 

After `parse_ek_blob` returns `SUCCESS`, the code dispatches directly to `pve_base_pke_rsa().encrypt(pve_keyref(blob.rsa_ek), ...)` with no key-size assertion.

**Step 4 — `kem_policy_rsa_oaep_t::encapsulate` has no size check** [4](#0-3) 

**Step 5 — `encrypt_oaep_with_seed` uses `size() * 8` dynamically** [5](#0-4) 

`size()` returns the actual modulus byte length of the key in hand. For a 1024-bit key, `size() * 8 = 1024`, so OAEP padding is computed for a 128-byte block and `encrypt_raw` produces a 128-byte RSA ciphertext. The call returns `SUCCESS`.

**The only size guard is in a different, unrelated entry point** [6](#0-5) 

`base_pke_rsa_ek_from_modulus` enforces 256-byte modulus, but an attacker who directly serializes a crafted `base_pke_ek_blob_v1_t` never passes through this function.

**`RSA_KEY_LENGTH` is defined but never enforced at parse time** [7](#0-6) 

### Impact Explanation

The `key_type=rsa_oaep_2048` tag is a semantic contract that the library fails to enforce. An attacker who controls the EK blob (e.g., a malicious serialized-input provider or a Byzantine participant supplying the encryption key) can substitute a 1024-bit RSA key. The library accepts the blob, encrypts the PVE scalar under the weak key, and returns `SUCCESS`. The resulting ciphertext is structurally indistinguishable from a legitimate one but is encrypted under a key that can be factored with modern hardware (GNFS), allowing recovery of the protected scalar `x`. This is a **High** impact finding: a public API reachable validation bypass in PVE creates accepted invalid cryptographic output.

### Likelihood Explanation

The attack requires only the ability to supply a crafted EK blob to `cbmpc_pve_encrypt` (or any API that calls `unified_key_blob_base_pke_t::encrypt`). No threshold collusion, no memory access, and no privileged position is required. The serialization format is documented and the blob structure is straightforward to craft.

### Recommendation

Add a key-size check immediately after deserialization in `parse_ek_blob` (or inside `base_pke_ek_blob_v1_t::convert` for the `rsa_oaep_2048` branch):

```cpp
// In parse_ek_blob, after convert succeeds:
if (static_cast<base_pke_key_type_v1>(out.key_type) == base_pke_key_type_v1::rsa_oaep_2048) {
  if (out.rsa_ek.size() != coinbase::crypto::RSA_KEY_LENGTH / 8)
    return coinbase::error(E_BADARG, "RSA key must be exactly 2048 bits");
}
```

Alternatively, enforce the check inside `rsa_pub_key_t::convert` when the converter is in read mode, or add a dedicated `validate()` method called from all EK blob parsers.

  uint8_t parts = 0;
  bn_t e, n;

  if (converter.is_write()) {
    data_t data = get();

    if (data.e) {
      parts |= part_e;
      e = bn_t(data.e);
    }
    if (data.n) {
      parts |= part_n;
      n = bn_t(data.n);
    }

    BN_free(data.n);
    BN_free(data.e);
  }

  converter.convert(parts);

  if (converter.is_error()) return;
  if (parts & part_e) converter.convert(e);
  if (parts & part_n) converter.convert(n);

  if (!converter.is_write() && !converter.is_error()) {
    create();
    switch (parts) {
      case 0:
        break;
      case part_e | part_n:
        set(n, e);
        break;
      default:
        converter.set_error();
        free();
        return;
    }
  }
