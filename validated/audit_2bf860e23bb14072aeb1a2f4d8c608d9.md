Audit Report

## Title
Sequential `send()` Loop Without `try/catch` Allows Any Caller to Stall Cross-Chain Rate Propagation — (File: `contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary
`MultiChainRateProvider.updateRate()` iterates over all registered `rateReceivers` and calls `ILayerZeroEndpoint.send()` for each with no error isolation. If any `send()` reverts — most trivially via fee exhaustion when `msg.value` is less than the cumulative `estimatedFee` — the entire transaction reverts. Because `updateRate()` is permissionless, any caller can trigger this condition, leaving destination-chain receivers with a stale (or zero) rate. The concrete allowed impact is **Low — Contract fails to deliver promised returns**: stale rates cause `AGETHPoolV3.deposit()` to mint incorrect agETH amounts; a zero rate (never-set) causes division-by-zero, reverting deposits entirely.

## Finding Description
In `MultiChainRateProvider.updateRate()` (lines 108–137), `rate` and `lastUpdated` are written at lines 111 and 113 before the loop. The loop then calls `ILayerZeroEndpoint.send{ value: estimatedFee }(...)` at line 127 with no `try/catch`. If `send()` reverts on any iteration, the EVM rolls back the entire transaction including those storage writes.

Fee exhaustion is the simplest trigger: the contract deducts `estimatedFee` from `msg.value` on each iteration. A caller who provides less than `estimateTotalFee()` will cause a mid-loop revert. Because `updateRate()` has no access control (`external payable`), any address can call it with an under-funded value. The `estimateTotalFee()` helper exists but is not enforced inside `updateRate()`.

When `updateRate()` is persistently blocked, `CrossChainRateReceiver.rate` on destination chains stagnates at its last-stored value (or zero if never set). `AGETHPoolV3.viewSwapAgETHAmountAndFee()` at line 168 computes `agETHAmount = amountAfterFee * 1e18 / agETHToETHrate` where `agETHToETHrate = getRate()` → `IOracle(agETHOracle).getRate()` → `CrossChainRateReceiver.getRate()`. A stale rate produces mispriced agETH; a zero rate produces a division-by-zero revert on every `deposit()` call.

The owner can call `removeRateReceiver(uint256 _index)` to unblock the loop, but until that action occurs, no rate update reaches any chain.

## Impact Explanation
**Low — Contract fails to deliver promised returns.** Stale rates cause `AGETHPoolV3.deposit()` to mint agETH at an incorrect exchange rate, meaning users do not receive the promised value. A zero rate (possible before the first successful `updateRate()`) causes `deposit()` to revert with division-by-zero, blocking new deposits. In neither case are already-deposited funds frozen (ETH is returned on revert; previously minted agETH is unaffected), so the impact does not rise to Medium — Temporary Freezing of Funds. The claim's Medium framing is overstated.

## Likelihood Explanation
`updateRate()` is permissionless (`external payable`, no role check). Fee exhaustion requires only that a caller supplies less ETH than `estimateTotalFee()` returns — a trivially reachable condition. With 10+ registered receiver chains, the cumulative fee is non-trivial and easy to under-estimate. No key compromise, governance action, or special role is required. The condition is repeatable on every block until an admin removes the problematic receiver.

## Recommendation
1. Pre-validate `msg.value` against `estimateTotalFee()` at the top of `updateRate()` and revert early with a descriptive error before any storage writes or sends.
2. Wrap each `ILayerZeroEndpoint.send()` call in a `try/catch`; emit a per-receiver failure event on catch rather than reverting the whole transaction.
3. Consider committing `rate` and `lastUpdated` unconditionally (or after at least one successful send) so the on-chain rate always reflects the latest value even when some sends fail.

## Proof of Concept
```solidity
// Caller provides less than total required fees
uint256 totalFee = provider.estimateTotalFee(); // e.g. 0.02 ether for 2 receivers
provider.updateRate{value: totalFee - 1}();     // reverts mid-loop on second send

// On destination chain after the revert:
assertEq(receiver.rate(), staleValue);          // rate not updated
// AGETHPoolV3.deposit() mints agETH at stale rate → wrong amount returned to user
```
A Foundry fork test can confirm this by deploying `AGETHMultiChainRateProvider` with two receivers, calling `updateRate` with `msg.value = estimateFees(receiver1) + estimateFees(receiver2) - 1`, and asserting that `rate` is unchanged and that a subsequent `AGETHPoolV3.deposit()` call uses the stale oracle value.
