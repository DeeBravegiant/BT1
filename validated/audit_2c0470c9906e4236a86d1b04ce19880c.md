Audit Report

## Title
Missing Access Control on `sendETHToL1ViaBridge` Enables Linea Bridge Period-Limit Saturation, Freezing Pool Yield — (`contracts/bridges/LineaMessenger.sol`)

## Summary

`LineaMessenger.sendETHToL1ViaBridge` is declared `external payable nonReentrant` with no role guard, allowing any caller to invoke it with arbitrary parameters. An attacker can use this to saturate the Linea native bridge's per-period ETH cap at near-zero net cost (principal is recovered on L1), causing every subsequent call to `RSETHPool.bridgeAssetsViaNativeBridge` to revert for the entire period. Repeating this each period permanently blocks the native bridge path for pool yield.

## Finding Description

`LineaMessenger` inherits `Recoverable` (which extends `AccessControl`) but `sendETHToL1ViaBridge` carries no `onlyRole` modifier: [1](#0-0) 

The function accepts caller-supplied `l2bridge` and `target` with no allowlist, and forwards `msg.value` directly to `ILineaMessageService.sendMessage`. The Linea bridge enforces a per-period cap queryable via `limitInWei()` and `currentPeriodAmountInWei()`: [2](#0-1) 

The pool's authorized bridge path calls through this same function: [3](#0-2) 

**Exploit flow:**
1. Attacker reads `limitInWei()` and `currentPeriodAmountInWei()` to compute remaining capacity `R`.
2. Attacker calls `LineaMessenger.sendETHToL1ViaBridge{value: R}(lineaBridge, attacker_l1, R)`. The call succeeds; attacker's ETH is bridged to their own L1 address. Net cost: only `minimumFeeInWei`.
3. Period limit is now saturated.
4. `BRIDGER_ROLE` calls `bridgeAssetsViaNativeBridge()` → `sendETHToL1ViaBridge` → `sendMessage` → **reverts** (period limit exceeded).
5. Attacker repeats at the start of each new period. Cost per period ≈ `minimumFeeInWei`.

The `Recoverable` base contract provides `AccessControl` infrastructure but `sendETHToL1ViaBridge` never uses it: [4](#0-3) 

## Impact Explanation

Pool ETH accumulated from swap fees cannot be bridged to L1 via the native bridge for as long as the attacker repeats the saturation each period. The `BRIDGER_ROLE` has no role-restricted fallback within `bridgeAssetsViaNativeBridge`; the alternative `bridgeAssets` (LayerZero/Stargate) path exists in the contract but may not be configured for the Linea deployment, and in any case the native bridge path is permanently blocked. This matches **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation

- No privileged access required; any EOA with enough ETH to cover the remaining period capacity can execute the attack.
- The attacker recovers their principal on L1 each period, making the net cost only `minimumFeeInWei` per period (a few wei/gwei).
- No front-running dependency, no oracle manipulation, no governance capture.
- The attack is trivially repeatable every period indefinitely.
- Likelihood: **High**.

## Recommendation

Add a role guard to `sendETHToL1ViaBridge` so only the pool or a designated caller can invoke it:

```solidity
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value)
    external payable nonReentrant onlyRole(BRIDGER_ROLE) {
```

Alternatively, restrict `l2bridge` and `target` to protocol-controlled allowlists set at construction/admin time, preventing arbitrary callers from consuming bridge capacity.

## Proof of Concept

```solidity
// Fork Linea mainnet; deploy LineaMessenger with admin address

// 1. Read remaining capacity
uint256 limit     = ILineaMessageService(lineaBridge).limitInWei();
uint256 used      = ILineaMessageService(lineaBridge).currentPeriodAmountInWei();
uint256 remaining = limit - used;

// 2. Attacker saturates the period limit (ETH recovered on L1)
vm.prank(attacker);
lineaMessenger.sendETHToL1ViaBridge{value: remaining}(
    lineaBridge, attacker, remaining
);

// 3. Pool BRIDGER_ROLE attempts to bridge — must revert
vm.prank(bridger);
vm.expectRevert(); // Linea bridge: period limit exceeded
rsethPool.bridgeAssetsViaNativeBridge();

// 4. Repeat steps 1-3 at the start of each new period
```
