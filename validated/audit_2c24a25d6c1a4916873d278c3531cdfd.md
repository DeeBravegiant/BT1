### Title
Double Deposit Refund via Multiple Sub-Transactions Claiming the Same Credential Deregistration — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, `getConsumedDijkstraValue` computes the total consumed value for a top-level transaction by summing the top-level body's consumed value with each sub-transaction's consumed value. Critically, the deposit-lookup functions (`lookupStakingDeposit`, `lookupDRepDeposit`) passed into this function are derived from the **pre-transaction** ledger state and are applied uniformly to **all** sub-transactions, regardless of state changes made by earlier sub-transactions in the same batch. When two sub-transactions both contain `UnRegDepositTxCert` for the same staking credential that was registered before the transaction, the consumed-value calculation counts the deposit refund **twice** — once per sub-transaction — because the pre-transaction lookup returns `Just D` for both. If the sequential `SUBLEDGERS` rule does not prevent the second deregistration from succeeding (as explicitly acknowledged by a pending `xit` test with `error "TODO: predicate failure not yet implemented"`), an attacker can cause the ledger to accept a transaction whose consumed side is overstated by `D` lovelace, effectively creating ADA from nothing.

---

### Finding Description

**Root cause — `getConsumedDijkstraValue` uses a fixed pre-transaction deposit snapshot for all sub-transactions:** [1](#0-0) 

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
```

The same `lookupStakingDeposit` closure (capturing the pre-transaction `CertState`) is applied to every sub-transaction body via `foldMap'`. There is no mechanism to thread the updated state through successive sub-transaction lookups.

**Attack scenario:**

1. Attacker registers staking credential `C` in a prior transaction, paying deposit `D`.
2. Attacker submits a Dijkstra top-level transaction containing two sub-transactions:
   - `subTx1`: `UnRegDepositTxCert C D`
   - `subTx2`: `UnRegDepositTxCert C D`
3. `getConsumedDijkstraValue` evaluates both sub-transactions against the pre-transaction state where `C` is registered:
   - `lookupStakingDeposit C = Just D` → refund `D` counted for `subTx1`
   - `lookupStakingDeposit C = Just D` → refund `D` counted again for `subTx2`
   - Total consumed overstated by `D`.
4. If `SUBLEDGERS` processes `subTx1` (deregistering `C`, returning `D` to UTxO) and then allows `subTx2` to also succeed (the missing predicate failure), the produced side only reflects one deposit change, but the consumed side claims two refunds.
5. Value conservation check: `consumed = inputs + 2D`, `produced = outputs + fees`. The attacker can balance the equation by constructing outputs that absorb the extra `D`, effectively extracting `D` lovelace that was never legitimately released from the deposit pot.

**Developer acknowledgement of the missing guard:** [2](#0-1) 

The test is marked `xit` (skipped/pending) with `error "TODO: predicate failure not yet implemented"`, explicitly signalling that no predicate failure currently prevents two sub-transactions from claiming the same refund. If the `SUBDELEG` rule already rejected the second deregistration, the test would either pass with a different error or be removed; the `TODO` comment indicates the check does not yet exist.

**Structural analogy to the Mellow oracle bug:**

The Mellow vulnerability arises because `priceX96()` forces a cross-oracle path for every token, using a single fixed base-token oracle regardless of which underlying token is being priced. When a second underlying token is added, the same fixed oracle path produces incorrect prices for it.

Here, `getConsumedDijkstraValue` forces a single fixed deposit-lookup path (pre-transaction state) for every sub-transaction, regardless of which sub-transaction is being evaluated. When a second sub-transaction targets the same credential that an earlier sub-transaction already modified, the same fixed lookup path produces an incorrect (doubled) refund amount.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

If the double-deregistration succeeds (no predicate failure blocking `subTx2`), the consumed value is overstated by `D` lovelace. The attacker constructs outputs that absorb this phantom `D`, passing the value-conservation check while extracting lovelace that was never legitimately released. This constitutes direct creation of ADA outside the protocol's monetary rules, violating the preservation-of-value invariant that is the foundational correctness property of the Cardano ledger.

---

### Likelihood Explanation

**Medium.** The attack requires:
- A registered staking credential (trivially obtained by any user).
- A Dijkstra-era transaction with two sub-transactions targeting the same credential — a fully attacker-controlled, serialized ledger input.
- No privileged access, no governance majority, no key compromise.

The only uncertainty is whether `SUBLEDGERS`/`SUBDELEG` independently rejects the second deregistration. The `xit` test with `error "TODO: predicate failure not yet implemented"` is strong evidence that no such rejection is currently implemented.

---

### Recommendation

1. **Fix `getConsumedDijkstraValue`**: Thread the updated `CertState` through sub-transaction consumed-value calculations so that each sub-transaction's deposit lookup reflects the state after all preceding sub-transactions have been applied.

2. **Implement the missing predicate failure**: Add a check in `SUBDELEG` (or `SUBLEDGERS`) that detects when a credential targeted by a deregistration certificate in sub-transaction `N` was already deregistered by an earlier sub-transaction in the same batch, and rejects the transaction with an explicit `DuplicateSubTxRefund` (or similar) predicate failure.

3. **Enable and complete the `xit` test**: The test at `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53` should be converted from `xit` to `it` once the predicate failure is implemented.
