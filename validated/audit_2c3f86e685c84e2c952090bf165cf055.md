Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks All Withdrawal Initiations Until Admin Intervention - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at its default value of `0`. The guard in `initiateWithdrawal()` evaluates `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` as `0 >= 0 → true` for every caller, causing every withdrawal attempt to revert with `WithdrawalLimitReached()`. Any user who stakes KERNEL tokens before an admin manually calls `setMaxNumberOfWithdrawalsPerUser()` has their funds frozen in the contract with no self-service recovery path.

## Finding Description
The contract declares two withdrawal-cap identifiers:

- `MAX_WITHDRAWALS_PER_USER = 100` (line 38) — a constant that is never referenced in the withdrawal guard.
- `maxNumberOfWithdrawalsPerUser` (line 108) — a mutable state variable used in the guard but never initialized.

`initialize()` (lines 259–271) sets `kernelToken`, `rewardsToken`, and access-control roles, but makes no assignment to `maxNumberOfWithdrawalsPerUser`, so it remains `0` after deployment.

`initiateWithdrawal()` (line 323) performs:
```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```
For any fresh caller, `userWithdrawalIds[msg.sender].length == 0` and `maxNumberOfWithdrawalsPerUser == 0`, so `0 >= 0` is always `true` and the function always reverts. The check fires before any balance deduction, so no partial withdrawal is possible. `stake()` and `stakeFor()` have no dependency on this variable and succeed normally, creating a one-way door.

The only recovery path is an admin call to `setMaxNumberOfWithdrawalsPerUser()` (lines 610–620), which enforces `_maxNumberOfWithdrawalsPerUser > 0`, confirming that `0` is an invalid operational value that should never have been the default.

## Impact Explanation
Staked KERNEL tokens are temporarily frozen. No user can initiate any withdrawal — regardless of their staked balance — until an admin separately configures `maxNumberOfWithdrawalsPerUser`. This matches **Medium — Temporary freezing of funds**.

## Likelihood Explanation
Every deployment or upgrade that calls `initialize()` without a subsequent admin call to `setMaxNumberOfWithdrawalsPerUser()` leaves the contract in the broken state. Because `stake()` succeeds normally, users receive no on-chain signal that withdrawals are blocked until they attempt one. The constant `MAX_WITHDRAWALS_PER_USER = 100` exists in the same file and is the natural default value, making the omission easy to overlook during deployment. Likelihood is **Medium**.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to `MAX_WITHDRAWALS_PER_USER` inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    UtilLib.checkNonZeroAddress(_admin);
    UtilLib.checkNonZeroAddress(_kernelToken);
    UtilLib.checkNonZeroAddress(_rewardToken);

    __AccessControl_init();
    __ReentrancyGuard_init();

    _setupRole(DEFAULT_ADMIN_ROLE, _admin);

    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // add this line
}
```

## Proof of Concept
1. Admin deploys `KernelDepositPool` proxy and calls `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` remains `0`.
2. User approves the contract and calls `stake(100e18)`. Succeeds — 100 KERNEL tokens are transferred in, `balanceOf[user] = 100e18`.
3. User calls `initiateWithdrawal(100e18)`. Execution reaches line 323:
   ```
   userWithdrawalIds[user].length  →  0
   maxNumberOfWithdrawalsPerUser   →  0
   0 >= 0                          →  true  →  revert WithdrawalLimitReached()
   ```
4. User's 100 KERNEL tokens are stuck. `claimWithdrawal()` is also unreachable because no withdrawal record was ever created.
5. Recovery requires the admin to call `setMaxNumberOfWithdrawalsPerUser(n)` with `n > 0`, after which step 3 succeeds.

**Foundry test sketch:**
```solidity
function test_withdrawalBlockedOnFreshDeploy() public {
    // deploy proxy, call initialize
    kernelPool.initialize(admin, address(kernelToken), address(rewardToken));
    // user stakes
    vm.startPrank(user);
    kernelToken.approve(address(kernelPool), 100e18);
    kernelPool.stake(100e18);
    // withdrawal reverts
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    kernelPool.initiateWithdrawal(100e18);
    vm.stopPrank();
}
```
