Looking at the vulnerability class — a permissionless operation with an exact-match constraint that an attacker can exploit to cause a legitimate transaction to fail — I need to find an analog in Cardano Ledger. Let me examine the `currentTreasuryValue` / `validateTreasuryValue` pattern and treasury donations more closely.

### Title
Treasury Donation Griefing Forces Repeated Failure of `currentTreasuryValue`-Bearing Governance Proposals — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs`)

---

### Summary

`validateTreasuryValue` enforces an **exact equality** between the `currentTreasuryValue` field in a transaction body and the actual treasury amount. Treasury donations are **permissionless** — any unprivileged sender can include a `treasuryDonation` in their transaction. Because donations accumulate intra-epoch and are applied to `casTreasuryL` at every epoch boundary, an attacker can donate in epoch N to invalidate any governance-proposal transaction that was built against the epoch-N treasury value and submitted in epoch N+1. The attacker can repeat this every epoch, forcing the proposer to perpetually rebuild their transaction and never successfully land a proposal that relies on `txInfoCurrentTreasuryAmount` being present in the Plutus script context.

---

### Finding Description

**Root cause — exact-match check against a permissionlessly mutable value:**

`validateTreasuryValue` in the Conway LEDGER rule checks:

```haskell
SJust submittedTreasuryValue ->
  failureUnless (submittedTreasuryValue == actualTreasuryValue) $
    ConwayTreasuryValueMismatch ...
``` [1](#0-0) 

`actualTreasuryValue` is `chainAccountState ^. casTreasuryL`, which is updated at every epoch boundary by flushing the accumulated `utxosDonationL` into the treasury. [2](#0-1) 

The donation accumulator is written by `updateTreasuryDonation`, which is called for every valid transaction that carries a non-zero `treasuryDonation`:

```haskell
IsValid True -> utxos & utxosDonationL <>~ tx ^. bodyTxL . treasuryDonationTxBodyL
``` [3](#0-2) 

`treasuryDonationTxBodyL` carries no authorization requirement — any transaction sender can set it to any positive `Coin`. [4](#0-3) 

**Why `currentTreasuryValue` is mandatory for certain proposals:**

`txInfoCurrentTreasuryAmount` in the Plutus V3 script context is derived directly from `currentTreasuryValueTxBodyL`:

```haskell
PV3.txInfoCurrentTreasuryAmount =
  strictMaybe Nothing (Just . transCoinToLovelace) $ txBody ^. currentTreasuryValueTxBodyL
``` [5](#0-4) 

The production guardrails script `ensureTreasuryReserve` **fails** when `txInfoCurrentTreasuryAmount` is `Nothing`:

```haskell
case PV3.txInfoCurrentTreasuryAmount txInfo of
  Just treasury -> treasury P.- totalWithdrawal P.>= 100_000_000
  _ -> False
``` [6](#0-5) 

Any governance proposal whose constitution references this (or any analogous) guardrails script **must** include `currentTreasuryValue` in the transaction body. That field is then subject to the exact-match check.

**Attack sequence (epoch-level):**

1. In epoch N the treasury is T. Proposer builds a `TreasuryWithdrawals` governance proposal with `currentTreasuryValue = T` (required by the guardrails script).
2. Attacker submits any transaction with `treasuryDonation = 1` (minimum positive coin) in epoch N. This costs only the transaction fee.
3. At the N→N+1 epoch boundary, `casTreasuryL` becomes T+1.
4. In epoch N+1 the proposer's transaction fails: `ConwayTreasuryValueMismatch { mismatchSupplied = T, mismatchExpected = T+1 }`.
5. Proposer rebuilds with `currentTreasuryValue = T+1`.
6. Attacker donates again in epoch N+1.
7. At the N+1→N+2 boundary, `casTreasuryL` becomes T+2.
8. In epoch N+2 the proposer's transaction fails again.
9. Repeat indefinitely.

The proposer can always succeed if they submit within the **same epoch** they observe the treasury value. The attacker's leverage is to donate in epoch N so that the value the proposer observed at the start of epoch N is stale by epoch N+1. Because the proposer must re-observe and rebuild every epoch, and the attacker can donate at negligible cost, the attacker can sustain the grief indefinitely.

---

### Impact Explanation

Any governance proposal that must include `currentTreasuryValue` (because its guardrails script reads `txInfoCurrentTreasuryAmount` and rejects `Nothing`) can be persistently blocked from landing. The proposer is forced to rebuild and resubmit every epoch, paying fees each time, while the attacker pays only a single small donation per epoch. If the governance action has a finite lifetime (`ppGovActionLifetime`), the attacker can exhaust it before the proposal is ever accepted, causing the deposit to be forfeited to the treasury. This constitutes attacker-controlled modification of treasury donations and governance proposal outcomes outside design parameters.

**Allowed impact matched:** *Medium — Attacker-controlled transactions modify treasury donations or withdrawals outside design parameters.*

---

### Likelihood Explanation

- Treasury donations require no special key, credential, or governance role — any ADA holder can submit one.
- The attacker's cost per epoch is one transaction fee (a few ADA).
- The attack is detectable only by the victim observing repeated `ConwayTreasuryValueMismatch` failures.
- Governance action lifetimes are bounded (default 6 epochs on mainnet), so a sustained attack over that window can cause deposit forfeiture.
- Likelihood: **Medium** — requires mempool monitoring and per-epoch action, but is cheap and requires no privileged access.

---

### Recommendation

1. **Relax the exact-match to a lower-bound check.** Change `validateTreasuryValue` to accept `submittedTreasuryValue <= actualTreasuryValue`. This preserves the guardrails script's ability to reason about a minimum treasury floor while tolerating donations that increase the treasury.
2. **Alternatively, expose the actual treasury value to the script context without requiring the field in the tx body.** Pass `casTreasuryL` directly into the Plutus context regardless of whether `currentTreasuryValueTxBodyL` is set, removing the need for the proposer to commit to an exact value.
3. **Document the griefing risk** in the guardrails script authoring guide so constitution authors are aware that scripts depending on `txInfoCurrentTreasuryAmount` being `Just` create a mandatory-field dependency that is exploitable.

---

### Proof of Concept

```
Epoch N:
  treasury = T (e.g. 1_000_000_000 lovelace)

  Proposer builds tx_P:
    currentTreasuryValue = SJust T
    proposalProcedures   = [TreasuryWithdrawals ... guardrailsScript]
    -- guardrailsScript requires txInfoCurrentTreasuryAmount = Just T

  Attacker builds tx_A (same epoch, any slot):
    treasuryDonation = 1   -- 1 lovelace, no authorization needed
    -- tx_A is submitted and accepted; utxosDonationL += 1

Epoch boundary N → N+1:
  casTreasuryL := T + 1

Epoch N+1:
  Proposer submits tx_P (built in epoch N):
    validateTreasuryValue:
      submittedTreasuryValue = T
      actualTreasuryValue    = T + 1
      --> FAIL: ConwayTreasuryValueMismatch { supplied = T, expected = T+1 }

  Proposer rebuilds tx_P' with currentTreasuryValue = SJust (T+1)

  Attacker submits tx_A' with treasuryDonation = 1

Epoch boundary N+1 → N+2:
  casTreasuryL := T + 2

Epoch N+2:
  tx_P' fails: ConwayTreasuryValueMismatch { supplied = T+1, expected = T+2 }

... (repeats until ppGovActionLifetime epochs elapse and the deposit is forfeited)
```

The attacker entry path is entirely through the permissionless `treasuryDonationTxBodyL` field. The vulnerable ledger step is `validateTreasuryValue` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ledger.hs` (lines 442–454), which enforces `RelEQ` rather than `RelLTEQ` on a value that any unprivileged sender can increment. validateTreasuryValue ::
  ConwayEraTxBody era => TxBody l era -> Coin -> Test (ConwayLedgerPredFailure era)
validateTreasuryValue txBody actualTreasuryValue =
