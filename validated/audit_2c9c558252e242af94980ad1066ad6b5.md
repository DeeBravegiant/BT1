### Title
`ValidateTransactionProvider` Does Not Enforce `max_l1_handler_txs_per_block` Limit, Allowing a Malicious Proposer to Exceed the L1 Handler Transaction Cap Per Block - (`crates/apollo_batcher/src/transaction_provider.rs`)

---

### Summary

The `ProposeTransactionProvider` enforces `max_l1_handler_txs_per_block` when building a block proposal, but `ValidateTransactionProvider` has no equivalent check. A malicious or misconfigured proposer can stream more than `max_l1_handler_txs_per_block` L1 handler transactions in a proposal, and the validator will execute all of them without enforcing the limit. This breaks the invariant that at most `max_l1_handler_txs_per_block` (200 in production) L1 handler transactions are included per block, causing wrong L1 message state, receipts, and events to be committed.

---

### Finding Description

**Proposer side (enforces the limit):**

`ProposeTransactionProvider` tracks `n_l1handler_txs_so_far` and caps L1 handler transactions at `max_l1_handler_txs_per_block`:

```rust
let n_l1handler_txs_to_get =
    min(self.max_l1_handler_txs_per_block - self.n_l1handler_txs_so_far, n_txs);
let mut l1handler_txs = self.get_l1_handler_txs(n_l1handler_txs_to_get).await?;
self.n_l1handler_txs_so_far += l1handler_txs.len();
// ...
let reached_max_l1handler_txs_in_block =
    self.n_l1handler_txs_so_far == self.max_l1_handler_txs_per_block;
if no_more_l1handler_in_provider || reached_max_l1handler_txs_in_block {
    self.phase = TxProviderPhase::Mempool;
}
``` [1](#0-0) 

**Validator side (no limit):**

`ValidateTransactionProvider` receives transactions from the proposer via `tx_receiver` and only validates each L1 handler transaction against the L1 events provider. There is no `max_l1_handler_txs_per_block` parameter, no counter, and no cap:

```rust
pub struct ValidateTransactionProvider {
    tx_receiver: tokio::sync::mpsc::Receiver<InternalConsensusTransaction>,
    final_n_executed_txs_receiver: tokio::sync::oneshot::Receiver<usize>,
    l1_events_provider_client: SharedL1EventsProviderClient,
    height: BlockNumber,
}
``` [2](#0-1) 

The `get_txs` implementation only validates each L1 handler transaction's state (not cancelled, not consumed, etc.) but never counts how many L1 handler transactions have been received: [3](#0-2) 

**Confirmed at construction:** The batcher creates `ValidateTransactionProvider` without passing `max_l1_handler_txs_per_block`:

```rust
let tx_provider = ValidateTransactionProvider::new(
    input_tx_receiver,
    final_n_executed_txs_receiver,
    self.l1_events_provider_client.clone(),
    validate_block_input.block_info.block_number,
);
``` [4](#0-3) 

**No guard in the consensus validation path:** `handle_proposal_part` in `validate_proposal.rs` only checks that `executed_transaction_count` does not exceed the number of received transactions — it does not check the number of L1 handler transactions: [5](#0-4) 

**The bouncer does not close this gap:** The bouncer enforces a total `n_txs` limit (500 in production) and gas/resource limits, but has no concept of a per-type L1 handler transaction cap: [6](#0-5) 

The production `max_l1_handler_txs_per_block_proposal` is 200, while the bouncer's `n_txs` is 500: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

A malicious proposer can include up to 500 L1 handler transactions (the bouncer's `n_txs` limit) in a single block proposal, while the intended maximum is 200. The validator will execute all of them, committing a block with up to 2.5× the intended number of L1 messages. This results in:

- **Wrong L1 message state**: More L1-to-L2 messages are consumed than the protocol intends per block.
- **Wrong receipts and events**: Extra L1 handler transaction receipts and events are committed to the block.
- **Wrong block commitment**: The `PartialBlockHash` and `ProposalCommitment` are computed over the inflated set of transactions, producing a commitment that encodes a protocol-violating block.

This matches the "Critical. Wrong state, receipt, event, L1 message" impact category.

---

### Likelihood Explanation

Triggering this requires a malicious consensus proposer — a node that is a valid consensus participant but acts adversarially. In a Byzantine fault tolerant system with `f` faulty nodes out of `3f+1`, a single malicious proposer can produce such a block in any round where it is selected. The attack requires no special privileges beyond being a consensus participant and no user interaction.

---

### Recommendation

Pass `max_l1_handler_txs_per_block` to `ValidateTransactionProvider` and add a counter in `get_txs` that rejects the proposal (via `TransactionProviderError`) when the number of received L1 handler transactions exceeds the limit:

```rust
pub struct ValidateTransactionProvider {
    tx_receiver: tokio::sync::mpsc::Receiver<InternalConsensusTransaction>,
    final_n_executed_txs_receiver: tokio::sync::oneshot::Receiver<usize>,
    l1_events_provider_client: SharedL1EventsProviderClient,
    height: BlockNumber,
+   max_l1_handler_txs_per_block: usize,
+   n_l1handler_txs_so_far: usize,
}
```

In `get_txs`, after receiving the buffer, count L1 handler transactions and return an error if the running total exceeds `max_l1_handler_txs_per_block`.

---

### Proof of Concept

1. A malicious proposer node is configured with `max_l1_handler_txs_per_block_proposal = 500` (or the limit is removed from `ProposeTransactionProvider`).
2. The L1 events provider has ≥ 201 pending L1 handler transactions past their cooldown.
3. The proposer builds a block proposal streaming 201 L1 handler transactions followed by a `ProposalFin` with `executed_transaction_count = 201`.
4. The validator's `ValidateTransactionProvider::get_txs` receives all 201 L1 handler transactions, validates each against the L1 events provider (all pass), and returns them to the block builder.
5. The block builder executes all 201 transactions; the bouncer does not trigger (201 < 500 `n_txs`).
6. `finish_proposal` is called with `final_n_executed_txs = 201`; the batcher closes the block and computes a `PartialBlockHash` over 201 L1 handler transactions.
7. The validator's computed commitment matches the proposer's `ProposalFin.proposal_commitment` (both computed over the same 201 transactions), so `validate_proposal` returns `Ok`.
8. Consensus reaches decision; the block with 201 L1 handler transactions (exceeding the 200 limit) is committed to storage.

impl TransactionProvider for ValidateTransactionProvider {
    async fn get_txs(&mut self, n_txs: usize) -> TransactionProviderResult<NextTxs> {
        assert!(n_txs > 0, "The number of transactions requested must be greater than zero.");

        if self.tx_receiver.is_empty() {
            // Return immediately to avoid blocking the caller.
            return Ok(vec![]);
        }

        let mut buffer = Vec::with_capacity(n_txs);
        self.tx_receiver.recv_many(&mut buffer, n_txs).await;

        for tx in &buffer {
            if let InternalConsensusTransaction::L1Handler(tx) = tx {
                let l1_validation_status = self
                    .l1_events_provider_client
                    .validate(tx.tx_hash, self.height)
                    .await
                    .inspect_err(|err| {
                        warn!(
                            "L1 provider error while validating L1 handler transaction: {:?}",
                            err
                        );
                        BATCHER_L1_EVENTS_PROVIDER_ERRORS.increment(1);
                    })
                    .unwrap_or(L1ValidationStatus::Invalid(
                        L1InvalidValidationStatus::L1EventsProviderError,
                    ));
                if let L1ValidationStatus::Invalid(validation_status) = l1_validation_status {
                    return Err(TransactionProviderError::L1HandlerTransactionValidationFailed {
                        tx_hash: tx.tx_hash,
                        validation_status,
                    });
                }
                continue;
            }
        }
        Ok(buffer)
    }
