Audit Report

## Title
`assetsCommitted` Decremented by Original Amount While Only `payoutAmount` Is Redeemed, Inflating `getAvailableAssetAmount` and Enabling Over-Commitment of Withdrawal Requests - (File: contracts/LRTWithdrawalManager.sol)

## Summary
In `_unlockWithdrawalRequests`, `assetsCommitted[asset]` is decremented by `request.expectedAssetAmount` (the original committed amount) while the unstaking vault is only drained by `payoutAmount = min(expectedAssetAmount, currentReturn)`. When the rsETH/asset exchange rate moves unfavourably between initiation and unlock, `payoutAmount < expectedAssetAmount`, creating phantom liquidity in `getAvailableAssetAmount` equal to the difference. Subsequent callers of `initiateWithdrawal` can commit against this phantom liquidity, locking their rsETH in the contract against assets that do not exist in the vault.

## Finding Description
**Root cause — `_unlockWithdrawalRequests` (L798–815):**

```solidity
uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);
// ...
assetsCommitted[asset] -= request.expectedAssetAmount;  // ← original (larger) amount
request.expectedAssetAmount = payoutAmount;             // ← updated to actual (smaller) amount
```

`_calculatePayoutAmount` (L833–834) returns `min(request.expectedAssetAmount, currentReturn)` where `currentReturn = (rsETHUnstaked * rsETHPrice) / assetPrice`. When the exchange rate has moved unfavourably, `currentReturn < request.expectedAssetAmount`, so `payoutAmount < request.expectedAssetAmount`.

After `_unlockWithdrawalRequests` returns, `unlockQueue` redeems exactly `assetAmountUnlocked` (the sum of `payoutAmount` values) from the vault (L307). The vault balance therefore decreases by `payoutAmount`, but `assetsCommitted` decreases by `expectedAssetAmount`.

**Accounting impact on `getAvailableAssetAmount` (L599–603):**
```
availableAssetAmount = getTotalAssetDeposits(asset) - assetsCommitted[asset]
```
`getTotalAssetDeposits` includes `assetLyingUnstakingVault`. After unlock:
- `assetLyingUnstakingVault` decreases by `payoutAmount`
- `assetsCommitted` decreases by `expectedAssetAmount`

Net change = `+(expectedAssetAmount − payoutAmount)` — phantom liquidity injected into the available amount.

**Over-commitment path — `initiateWithdrawal` (L168–173):**
```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```
Any unprivileged user can call `initiateWithdrawal` and pass the guard using the inflated `getAvailableAssetAmount`. Their rsETH is transferred to the contract, but the corresponding assets do not exist in the vault. When `unlockQueue` later runs, `unstakingVault.balanceOf(asset)` is insufficient to cover the phantom request, so the loop exits early (L800: `if (availableAssetAmount < payoutAmount) break`) and the request remains permanently locked until an operator manually replenishes the vault.

## Impact Explanation
**Temporary freezing of funds (Medium).** Users whose withdrawal requests are backed by phantom liquidity have their rsETH transferred into `LRTWithdrawalManager` and cannot retrieve it until an operator manually tops up the unstaking vault. The magnitude of the phantom amount equals the cumulative price-drift discount across all previously unlocked requests, which can be material during volatile market conditions.

## Likelihood Explanation
The condition `payoutAmount < expectedAssetAmount` is triggered whenever the rsETH/asset exchange rate moves unfavourably between a user's `initiateWithdrawal` call and the operator's `unlockQueue` call. The withdrawal delay is configurable and can span many blocks; rsETH/LST prices fluctuate continuously. No privileged access or special setup is required — ordinary market price movement is sufficient. This condition will occur in normal protocol operation.

## Recommendation
Decrement `assetsCommitted` by `payoutAmount` rather than `request.expectedAssetAmount` so the accounting reflects the actual vault outflow:

```solidity
// contracts/LRTWithdrawalManager.sol, _unlockWithdrawalRequests
assetsCommitted[asset] -= payoutAmount;   // use actual payout, not original committed amount
request.expectedAssetAmount = payoutAmount;
```

This ensures `getAvailableAssetAmount` accurately reflects only assets that are genuinely available in the vault.

## Proof of Concept
**Setup:** 100 ETH in `LRTUnstakingVault`. rsETH price = 1.0 ETH/rsETH. ETH asset price = 1.0.

1. **User A** calls `initiateWithdrawal(ETH, 10 rsETH)`.
   - `expectedAssetAmount = 10 ETH`; `assetsCommitted = 10`
   - `getAvailableAssetAmount = 100 − 10 = 90`

2. rsETH price drops to 0.8 ETH/rsETH.

3. Operator calls `unlockQueue(ETH, ...)`.
   - `payoutAmount = min(10, 8) = 8 ETH`
   - `assetsCommitted -= 10` → `assetsCommitted = 0`
   - Vault redeems 8 ETH → vault balance = 92 ETH
   - `getAvailableAssetAmount = 92 − 0 = 92` ← **phantom +2 ETH**

4. **User B** calls `initiateWithdrawal(ETH, 92 rsETH)` (at current price 0.8, this maps to 73.6 ETH expected, but adjust amounts so `expectedAssetAmount = 92 ETH` at whatever price).
   - Guard passes (92 ≤ 92); `assetsCommitted = 92`; User B's rsETH locked in contract.

5. Operator calls `unlockQueue(ETH, ...)`.
   - Vault only has 92 ETH but User B's request requires assets that include the phantom 2 ETH.
   - If any further price drift occurs, `payoutAmount > vault balance`, loop breaks at L800, User B's request remains locked.

6. User B's rsETH remains trapped in `LRTWithdrawalManager` until an operator manually replenishes the vault.

**Foundry test plan:** Deploy `LRTWithdrawalManager` with a mock vault and oracle. Set rsETH price = 1e18, asset price = 1e18, vault balance = 100e18. User A initiates withdrawal of 10 rsETH. Drop rsETH price to 0.8e18. Call `unlockQueue`. Assert `getAvailableAssetAmount == 92e18` (phantom +2e18). User B initiates withdrawal of 92 rsETH worth. Drop price again. Call `unlockQueue`. Assert User B's request is not unlocked and their rsETH balance in the contract is non-zero.
