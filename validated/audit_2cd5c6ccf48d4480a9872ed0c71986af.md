### Title
Blacklisted Recipient Permanently Locks Bridged Funds in `finTransfer` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`)

### Summary
When a non-bridge ERC20 token with a blacklist (e.g., USDC, USDT) is used, a `finTransfer` call whose `recipient` is blacklisted will always revert. Because the source-chain leg is already irreversible (tokens locked or burned), and no alternative delivery or rescue path exists in the bridge, the bridged value is permanently unclaimable.

### Finding Description

**EVM — `OmniBridge.finTransfer`**

The nonce is consumed at line 287 before any transfer:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
```

For non-bridge ERC20 tokens the final branch is:

```solidity
IERC20(payload.tokenAddress).safeTransfer(           // line 351-354
    payload.recipient,
    payload.amount
);
```

`SafeERC20.safeTransfer` reverts on failure, which also reverts the nonce write, so the nonce is not permanently consumed. However, the signed payload fixes `recipient` — the MPC signature binds the exact recipient address. A relayer can retry indefinitely, but every attempt reverts because the recipient remains blacklisted. The source-chain leg (tokens locked in the NEAR bridge or burned) is already final and irreversible. There is no admin rescue function, no alternative-recipient override, and no pull-based withdrawal fallback.

The same applies to the native-ETH path:

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");  // line 319
if (!success) revert FailedToSendEther();                               // line 322
```

A contract recipient that always reverts permanently blocks ETH delivery.

**Starknet — `OmniBridge.fin_transfer`**

The nonce bitmap is written before the transfer:

```cairo
_set_transfer_finalised(ref self, payload.destination_nonce);   // line 250
```

Then for non-bridge tokens:

```cairo
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());         // line 260-261
assert(success, 'ERR_TRANSFER_FAILED');                          // line 262
```

Same outcome: the assert panic reverts the whole transaction including the nonce write, but the signed payload permanently fixes the recipient, so the transfer can never succeed if the recipient is blacklisted.

### Impact Explanation

**Critical — Irreversible fund lock.**

Once a user initiates a cross-chain transfer on the source chain, the source-side tokens are locked or burned. The destination `finTransfer` payload is MPC-signed with a fixed `recipient`. If that recipient is blacklisted by the token contract (a realistic event for USDC/USDT under sanctions enforcement), every `finTransfer` attempt reverts. No privileged rescue path exists in either contract. The bridged value is permanently unclaimable by anyone.

### Likelihood Explanation

USDC and USDT — both supported as non-bridge tokens — maintain on-chain blacklists. Blacklisting can occur after a user initiates a transfer (e.g., due to regulatory action between initiation and finalization). The user need not make any mistake; the external blacklisting event is sufficient to trigger the lock. The entry path (`finTransfer` / `fin_transfer`) is fully public and unprivileged.

### Recommendation

1. **Pull-based withdrawal fallback**: Instead of pushing tokens directly to `recipient` in `finTransfer`, credit an internal balance mapping. Add a separate `withdraw()` function that lets the recipient (or a designated fallback address) pull their funds.
2. **Fallback recipient**: Allow the MPC to sign an alternative-recipient override when the primary delivery has failed N times, enabling re-routing without re-locking source funds.
3. **Try/catch delivery**: Wrap the transfer in a try/catch (Solidity) or safe-dispatcher (Starknet) and emit a `DeliveryFailed` event, preserving the nonce as consumed but crediting an internal claimable balance for the amount.

### Proof of Concept

1. User calls `initTransfer` on NEAR, locking 10,000 USDC. The NEAR bridge records the lock; the cross-chain message is signed by the MPC with `recipient = 0xVictim`.
2. USDC issuer blacklists `0xVictim` (e.g., OFAC sanction) before the relayer submits `finTransfer` on EVM.
3. Relayer calls `OmniBridge.finTransfer(sig, payload)` where `payload.tokenAddress = USDC`, `payload.recipient = 0xVictim`.
4. Execution reaches line 351: `IERC20(USDC).safeTransfer(0xVictim, 10000e6)` — USDC reverts because `0xVictim` is blacklisted.
5. Entire transaction reverts. Nonce is not consumed. Relayer retries — same result every time.
6. The 10,000 USDC held in the EVM bridge contract is permanently inaccessible. The user's NEAR-side lock is also permanent because no `finTransfer` proof can ever be produced. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
