Looking at the code carefully to determine if the state diff commitment is verified against the stored header before committing peer-supplied state diff data.

### Title
Missing State Diff Commitment Verification in P2P Sync Allows Forged Nonce Injection — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`parse_data_for_block` in `StateDiffStreamBuilder` validates only the **length** of the assembled state diff against the stored header's `state_diff_length`, but never computes or compares the **Poseidon hash** of the assembled diff against the stored `state_diff_commitment`. A malicious p2p peer can therefore send a `ContractDiff` with an arbitrary forged nonce (preserving the correct total length), which is committed verbatim to the nonces storage table and subsequently returned by `get_nonce_at`, corrupting the nonce baseline used by the gateway's stateful transaction validator.

---

### Finding Description

In `parse_data_for_block`, the only integrity check performed on the assembled `ThinStateDiff` is a length comparison: [1](#0-0) 

The function reads `state_diff_length` from the stored header and verifies that the sum of `StateDiffChunk::len()` values equals it. It never reads `state_diff_commitment` from the header and never calls `calculate_state_diff_hash` on the assembled result. [2](#0-1) 

`calculate_state_diff_hash` explicitly includes nonces in the Poseidon commitment: [3](#0-2) 

A `ContractDiff` with `nonce=Some(x)` contributes exactly `1` to `StateDiffChunk::len()`: [4](#0-3) 

So a peer can substitute any nonce value while keeping the length identical. The forged nonce is inserted unconditionally by `unite_state_diffs`: [5](#0-4) 

`write_to_storage` then commits it directly: [6](#0-5) 

which calls `append_state_diff` → `write_nonces`, persisting the forged nonce to the `nonces` LMDB table: [7](#0-6) 

---

### Impact Explanation

The gateway's stateful validator calls `get_nonce_from_state` → `gateway_fixed_block_state_reader.get_nonce` to obtain `account_nonce`, then passes it to `validate_nonce`: [8](#0-7) 

`validate_nonce` enforces `account_nonce <= incoming_tx_nonce <= account_nonce + max_gap` for invoke, and exact equality for declare: [9](#0-8) 

If the forged nonce is **higher** than the canonical nonce, all legitimate transactions with the real nonce are rejected as `NonceTooOld`. If the forged nonce is **lower**, transactions that should be rejected (already-used nonces) may be accepted. This is a **High** admission-path impact: the gateway rejects valid transactions or accepts invalid ones before sequencing.

---

### Likelihood Explanation

Any unauthenticated p2p peer that responds to `StateDiff` protocol queries can trigger this. The p2p sync client sends queries to network peers and accepts responses from any peer that replies. No operator privilege is required — a single malicious node on the network suffices. The attack is silent (no error is returned; the bad peer is not reported) and persistent (the forged nonce is durably committed to storage).

---

### Recommendation

After assembling the full `ThinStateDiff` in `parse_data_for_block`, compute `calculate_state_diff_hash(&result)` and compare it to `header.state_diff_commitment`. If they differ, return `ParseDataError::BadPeer`. This mirrors the existing `state_diff_length` check and closes the integrity gap.

---

### Proof of Concept

1. Node A syncs headers from a legitimate source; block N's header is stored with `state_diff_commitment = H_canonical` and `state_diff_length = L`.
2. Malicious peer M intercepts the `StateDiff` query for block N.
3. M sends a `ContractDiff` for address A with `nonce = forged_nonce` (instead of `canonical_nonce`), keeping all other fields identical so total length = L.
4. `parse_data_for_block` accepts it (length matches, no duplicate nonces).
5. `write_to_storage` commits `forged_nonce` for address A.
6. User submits a valid invoke transaction for account A with `nonce = canonical_nonce`.
7. Gateway calls `get_nonce_at` → returns `forged_nonce` (e.g., `canonical_nonce + 1`).
8. `validate_nonce` rejects the transaction: `canonical_nonce < forged_nonce`, error `InvalidTransactionNonce`.

```rust
    fn validate_nonce(
        &self,
        executable_tx: &ExecutableTransaction,
        account_nonce: Nonce,
    ) -> StatefulTransactionValidatorResult<()> {
        let incoming_tx_nonce = executable_tx.nonce();

        let create_error = |message: String| {
            debug!("{message}");
            StarknetError {
                code: StarknetErrorCode::KnownErrorCode(
                    KnownStarknetErrorCode::InvalidTransactionNonce,
                ),
                message,
            }
        };

        match executable_tx {
            // Declare transactions must have the same nonce as the account nonce.
            ExecutableTransaction::Declare(_) if self.config.reject_future_declare_txs => {
                if incoming_tx_nonce != account_nonce {
                    return Err(create_error(format!(
                        "Invalid transaction nonce. Expected: nonce = {account_nonce}, got: \
                         {incoming_tx_nonce}."
                    )));
                }
            }
            // Deploy account transactions must have nonce 0.
            ExecutableTransaction::DeployAccount(_) => {
                if account_nonce != Nonce(Felt::ZERO) {
                    return Err(create_error(format!(
                        "Invalid deploy account transaction. Account is already deployed \
                         (nonce={account_nonce})."
                    )));
                }
                if incoming_tx_nonce != Nonce(Felt::ZERO) {
                    return Err(create_error(format!(
                        "Invalid transaction nonce. Expected: nonce = 0, got: {incoming_tx_nonce}."
                    )));
                }
            }
            // Other transactions must be within the allowed nonce range.
            _ => {
                let max_allowed_nonce =
                    Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
                if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
                    return Err(create_error(format!(
                        "Invalid transaction nonce. Expected: {account_nonce} <= nonce <= \
                         {max_allowed_nonce}, got: {incoming_tx_nonce}."
                    )));
                }
            }
        }

        Ok(())
```
