Audit Report

## Title
Missing Chainlink Heartbeat Staleness Check Enables Stale-Price Deposits — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs three validity checks on Chainlink round data but omits a `block.timestamp - updatedAt > heartbeat` guard. A feed that has not been updated beyond its heartbeat window (e.g., 24 h for stETH/ETH) but still satisfies `answeredInRound >= roundID` and `timestamp != 0` will pass all guards and return a stale price. Any unprivileged caller can then invoke `deposit(token, amount, referralId)` on `RSETHPoolV3` or `RSETHPoolV3WithNativeChainBridge` and receive excess wrsETH, constituting theft of unclaimed yield from existing holders.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` at lines 30–32 performs exactly three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0)            revert IncompleteRound();
if (ethPrice <= 0)             revert InvalidPrice();
```

None of these verify `block.timestamp - updatedAt <= heartbeat`. A feed last updated 25+ hours ago satisfies all three conditions and returns the stale price without reverting.

This oracle is consumed by `RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)`:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

The same pattern exists identically in `RSETHPoolV3WithNativeChainBridge.viewSwapRsETHAmountAndFee`. If `tokenToETHRate` is stale and inflated relative to the true current price (possible when the price has drifted downward within the deviation threshold since the last update), `rsETHAmount` is over-computed and `wrsETH.mint(msg.sender, rsETHAmount)` mints excess tokens. The `limitDailyMint` modifier calls the same `viewSwapRsETHAmountAndFee` with the same stale oracle, so it only caps the total inflated amount — it does not prevent the per-unit over-mint.

## Impact Explanation
The excess wrsETH minted represents a claim on more rsETH than the deposited collateral is worth at the true current price. When the attacker later unwraps wrsETH → rsETH, they extract rsETH backed by yield accrued by existing holders, diluting the rsETH/ETH rate for all wrsETH/rsETH holders. This constitutes **theft of unclaimed yield** (High), bounded in magnitude by the feed's deviation threshold (e.g., up to 0.5% per deposit for stETH/ETH), but repeatable across multiple deposits and multiple stale windows.

## Likelihood Explanation
Chainlink feeds approach their heartbeat boundary during periods of low price volatility when the deviation threshold is not triggered. The stETH/ETH feed (24 h heartbeat, 0.5% deviation) has historically approached this boundary during stable market periods. No privileged access is required: any EOA can call `deposit(token, amount, referralId)` on a non-paused pool with a supported token. The condition is passively exploitable — the attacker only needs to monitor the feed's `updatedAt` timestamp and act when it is near or past the heartbeat.

## Recommendation
Add a `heartbeat` parameter to `ChainlinkOracleForRSETHPoolCollateral` and enforce it in `getRate()`:

```solidity
uint256 public immutable heartbeat;

constructor(address _oracle, uint256 _heartbeat) {
    oracle    = _oracle;
    heartbeat = _heartbeat;
}

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 updatedAt, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID)               revert StalePrice();
    if (updatedAt == 0)                          revert IncompleteRound();
    if (block.timestamp - updatedAt > heartbeat) revert StalePrice();   // ← add this
    if (ethPrice <= 0)                           revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

Set `heartbeat` to the feed's documented heartbeat (e.g., 86 400 s for stETH/ETH) plus a small buffer (e.g., 3 600 s) to avoid false reverts from minor Chainlink node delays.

## Proof of Concept
Deploy `ChainlinkOracleForRSETHPoolCollateral` against a mock aggregator returning `updatedAt = block.timestamp - 25 hours` with `answeredInRound == roundID` and `answer = 0.9999e18`. Call `getRate()` — it returns the stale price without reverting, confirming the missing guard. In a fork test, assign this oracle as `supportedTokenOracle[stETH]` in `RSETHPoolV3`, call `deposit(stETH, largeAmount, "")`, and compare the minted wrsETH against the fair-value baseline computed with the current on-chain price. The difference equals `largeAmount * (stalePrice - truePrice) / rsETHToETHrate`, which is extractable as excess rsETH upon unwrapping.
