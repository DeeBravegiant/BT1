### Title
Missing `ft_on_transfer` / `nft_on_transfer` Receiver Callbacks in ETH-Implicit Account Wallet Contract - (File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs)

### Summary
The `WalletContract` automatically deployed to every ETH-implicit account does not implement the `ft_on_transfer` callback required by NEP-141's `ft_transfer_call`, nor the `nft_on_transfer` callback required by NEP-171's `nft_transfer_call`. Any unprivileged user who calls `ft_transfer_call` or `nft_transfer_call` targeting an ETH-implicit account will receive a `MethodNotFound` execution failure, permanently preventing ETH-implicit accounts from receiving tokens via the safe-transfer pattern.

### Finding Description
The `WalletContract` struct exposes only `get_nonce`, `rlp_execute`, and three `#[private]` internal callbacks (`address_check_callback`, `nep_141_storage_balance_callback`, `rlp_execute_callback`). [1](#0-0) 

The `eth_emulation.rs` module explicitly states it only supports ERC-20 (NEP-141 `ft_transfer`, `ft_balance_of`, `ft_total_supply`) and maps those to outgoing calls. There is no inbound receiver interface. [2](#0-1) 

The NEP-141 standard's `ft_transfer_call` flow requires the token contract to call `ft_on_transfer(sender_id, amount, msg)` on the receiver after crediting the tokens. If the receiver does not export that method, the NEAR runtime returns `MethodNotFound`, the token contract's `ft_resolve_transfer` refunds the sender, and the ETH-implicit account's balance never increases.

The NEP-171 NFT standard's `nft_transfer_call` has the identical requirement: the NFT contract calls `nft_on_transfer` on the receiver. The Wallet Contract exports no such method.

Because every ETH-implicit account is assigned this global contract at creation time and cannot deploy a different contract, the limitation is permanent and protocol-wide. [3](#0-2) 

### Impact Explanation
**Corrupted protocol value**: The receipt outcome for any `ft_on_transfer` or `nft_on_transfer` action receipt targeting an ETH-implicit account is `Failure(FunctionCallError(MethodResolveError(MethodNotFound)))` instead of `Success`. Consequently, `ft_resolve_transfer` / `nft_resolve_transfer` on the token contract refunds the sender, and the ETH-implicit account's fungible-token or NFT balance is never updated.

ETH-implicit accounts are the NEAR equivalent of EIP-7702 smart wallets: they are the primary on-ramp for Ethereum-native users. Inability to receive tokens via `ft_transfer_call` or `nft_transfer_call` blocks participation in any NEAR DeFi protocol or NFT marketplace that uses the safe-transfer pattern, which is the dominant pattern on NEAR.

### Likelihood Explanation
**High**. The trigger requires no special privilege: any account can call `ft_transfer_call` on any NEP-141 token contract with an ETH-implicit account as the receiver. The failure is deterministic and reproducible on every attempt. The affected population is every ETH-implicit account on mainnet and testnet.

### Recommendation
Add `ft_on_transfer` and `nft_on_transfer` method implementations to `WalletContract` in `lib.rs`. The simplest correct implementation returns `"0"` (accept all tokens) for `ft_on_transfer` and `false` (keep the NFT) for `nft_on_transfer`, matching the behavior expected by NEP-141 and NEP-171 token contracts when the receiver is a passive holder. Alternatively, expose these callbacks through the `rlp_execute` dispatch path so the account owner can configure acceptance logic.

### Proof of Concept
1. Create an ETH-implicit account (send any amount to a `0x…` address); the runtime automatically assigns the Wallet Contract.
2. Deploy any standard NEP-141 token contract and mint tokens to a normal account.
3. Call `ft_transfer_call` on the token contract with `receiver_id` set to the ETH-implicit account and any `msg`.
4. Observe: the token contract calls `ft_on_transfer` on the ETH-implicit account → NEAR runtime returns `MethodNotFound` → `ft_resolve_transfer` refunds the sender → ETH-implicit account token balance remains zero.

#[near_bindgen]
impl WalletContract {
    /// Return the nonce value currently stored in the contract.
    /// Following the Ethereum protocol, only transactions with nonce equal
    /// to the current value will be accepted.
    /// Additionally, the Ethereum protocol requires the nonce of an account increment
    /// by 1 each time a transaction with the correct nonce and a valid signature
    /// is submitted (even if that transaction eventually fails). In this way, each
    /// nonce value can only be used once (hence the name "nonce") and thus transaction
    /// replay is prevented.
    pub fn get_nonce(&self) -> U64 {
        U64(self.nonce)
    }

    /// This is the main entry point into this contract. It accepts an RLP-encoded
    /// Ethereum transaction signed by the private key associated with the address
    /// for the account where this contract is deployed. RLP is a binary format,
    /// so the argument is actually passed as a base64-encoded string.
    /// The Ethereum transaction represents a Near action the owner of the address
    /// wants to perform. This method decodes that action from the Ethereum transaction
    /// and crates a promise to perform that action.
    /// Actions on Near are sent to a particular account ID where they are supposed to
    /// be executed (for example, a `FunctionCall` action is sent to the contract
    /// which will execute the method). In the Ethereum transaction only the address
    /// of the target can be specified because it does not have a notion of named accounts
    /// like Near has. The `target` field of this method gives the actual account ID
    /// that the action will be sent to. The `target` must itself be an eth-implicit
    /// account and match the `to` address of the Ethereum transaction; or `target`
    /// must hash to the address given in the `to` field of the Ethereum transaction.
    /// The output of this function is an `ExecuteResponse` which gives the output
    /// of the Near action or an error message if there was a problem during the execution.
    #[payable]
    pub fn rlp_execute(
        &mut self,
        target: AccountId,
        tx_bytes_b64: String,
    ) -> PromiseOrValue<ExecuteResponse> {
        // To ensure user actions are executed in the desired order,
        // having multiple transactions in flight at the same time is
        // not allowed.
        if self.has_in_flight_tx {
            return PromiseOrValue::Value(ExecuteResponse {
                success: false,
                success_value: None,
                error: Some(
                    "Error: transaction already in progress, please try again later.".into(),
                ),
            });
        }
        let current_account_id = env::current_account_id();
        let predecessor_account_id = env::predecessor_account_id();
        let result = inner_rlp_execute(
            current_account_id.clone(),
            predecessor_account_id,
            target,
            tx_bytes_b64,
            &mut self.nonce,
        );

        match result {
            Ok(promise) => {
                self.has_in_flight_tx = true;
                PromiseOrValue::Promise(promise)
            }
            Err(Error::Relayer(_)) if env::signer_account_id() == current_account_id => {
                let promise = create_ban_relayer_promise(current_account_id);
                self.has_in_flight_tx = true;
                PromiseOrValue::Promise(promise)
            }
            Err(e) => PromiseOrValue::Value(e.into()),
        }
    }
