Audit Report

## Title
Dust rsETH Input Causes Zero-Asset Return with Permanent rsETH Burn in `instantWithdrawal` and `initiateWithdrawal` — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`getExpectedAssetAmount` performs integer division that truncates to zero for dust rsETH inputs when `rsETHPrice < assetPrice`. Neither `instantWithdrawal` nor `initiateWithdrawal` guards against a zero output before burning the user's rsETH. The result is that rsETH is permanently destroyed and the user receives zero underlying assets, with no recovery path.

## Finding Description

`getExpectedAssetAmount` at L593 computes:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

When `amount` is small enough that `amount * rsETHPrice < assetPrice`, Solidity integer division yields `underlyingToReceive = 0`. For example, with `rsETHPrice = 1.00e18` and `assetPrice = 1.05e18`, any `amount < 2` truncates to zero.

**`instantWithdrawal` path (L224–252):**

The only input guard is:
```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```
`minRsEthAmountToWithdraw[asset]` is a mapping that defaults to `0` (L35), so `rsETHUnstaked = 1` passes. After computing `assetAmountUnlocked = 0` at L228, the contract immediately burns the user's rsETH at L229 with no zero-output check:
```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
```
`unstakingVault.redeem(asset, 0)` is a no-op, `fee = 0`, `userAmount = 0`, and `_transferAsset` sends nothing.

**`initiateWithdrawal` + `unlockQueue` path (L162–175, L790–815):**

`initiateWithdrawal` transfers rsETH to the contract and stores a request with `expectedAssetAmount = 0` (no zero-output guard at L168–175). During `unlockQueue`, `_calculatePayoutAmount` at L833–834 returns `min(0, currentReturn) = 0`. The loop at L800 checks `availableAssetAmount < payoutAmount` — since `payoutAmount = 0`, this is always false and the request is processed. `rsETHAmountToBurn` accumulates the full `rsETHUnstaked` (L805), which is burned at L305. `assetAmountToUnlock` remains 0. On `completeWithdrawal`, `_transferAsset` sends 0 to the user (L734).

Existing checks are insufficient: the `rsETHUnstaked != 0` guard does not prevent a zero-output scenario, and there is no post-computation check on `assetAmountUnlocked` or `expectedAssetAmount` in either function.

## Impact Explanation

**Critical — Permanent loss of user funds.**

The user's rsETH is irreversibly burned (`burnFrom`) and they receive zero underlying assets. The `AssetWithdrawalFinalized` event emits `amountReceived = 0`, providing no on-chain error signal. There is no recovery mechanism: the burn is final and the contract holds no record of the lost value. This constitutes direct, permanent destruction of user funds, matching the allowed impact: *Critical. Direct theft of any user funds, whether at-rest or in-motion.*

## Likelihood Explanation

**Low-Medium.** Three conditions are required:

1. `minRsEthAmountToWithdraw[asset] == 0` — this is the **default state** of the mapping; no admin action is needed.
2. `rsETHPrice < assetPrice` at call time — achievable when a supported LST (e.g., stETH) trades at a premium to rsETH's current exchange rate, or during early protocol life.
3. A dust rsETH input — possible via a UI bug, a script, or a user probing the minimum.

Condition 1 is always satisfied unless the admin proactively sets a minimum per asset. Conditions 2 and 3 are realistic but not constant. The scenario is triggerable by any unprivileged user holding rsETH.

## Recommendation

Add a zero-output guard immediately after computing the expected asset amount in both `instantWithdrawal` and `initiateWithdrawal`:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
if (assetAmountUnlocked == 0) revert InvalidAmountToWithdraw();
```

Additionally, enforce a non-zero `minRsEthAmountToWithdraw` for every supported asset at configuration time rather than relying on the default-zero mapping value.

## Proof of Concept

**Setup:** `rsETHPrice = 1.00e18`, `stETH assetPrice = 1.05e18`. `minRsEthAmountToWithdraw[stETH] = 0` (default). Instant withdrawal enabled for stETH.

1. User holds `1` wei of rsETH.
2. User calls `instantWithdrawal(stETH, 1, "")`.
3. Guard passes: `1 != 0` and `1 >= 0`.
4. `getExpectedAssetAmount(stETH, 1)` → `1 * 1e18 / 1.05e18 = 0`.
5. `burnFrom(msg.sender, 1)` — 1 wei rsETH permanently destroyed.
6. `redeem(stETH, 0)` — no-op.
7. `fee = 0`, `userAmount = 0`.
8. `_transferAsset(stETH, msg.sender, 0)` — 0 stETH transferred.
9. Transaction succeeds. User has lost 1 wei rsETH and received nothing.

**Foundry fuzz test plan:** Fuzz `rsETHUnstaked` over `[1, 1e15]`, mock `rsETHPrice = 1e18` and `assetPrice = 1.05e18`, assert that `assetAmountUnlocked > 0` before any burn occurs. The test will fail for all inputs where `rsETHUnstaked * 1e18 / 1.05e18 == 0`, confirming the truncation path. The same test applies to `initiateWithdrawal` by asserting `expectedAssetAmount > 0` before the withdrawal request is stored.
