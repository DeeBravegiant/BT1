### Title
Missing Output Verification in `respond_ckd` for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Forged Confidential Key - (File: crates/contract/src/lib.rs)

### Summary

The `respond_ckd` function in `MpcContract` performs cryptographic output verification only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`, but performs **no verification whatsoever** for the `AppPublicKey` variant. A single attested participant (a Byzantine MPC node below the signing threshold) can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` CKD request, and the contract will accept and deliver the forged confidential key to the user without any on-chain check.

This is a direct analog to the `BasePool.mint()` missing `onlyRouter` modifier: just as `BasePool.mint()` lacked the modifier that would have forced the router's input validation to run, `respond_ckd` for `AppPublicKey` lacks the output-validity check that `AppPublicKeyPV` enforces, allowing a single participant to bypass the threshold-computation guarantee entirely.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function handles two variants of `CKDAppPublicKey`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check` to verify the response cryptographically against the app public key before resolving the yield. For `AppPublicKey`, the match arm is an empty block — the response is accepted unconditionally.

After this match, `resolve_yields_for` is called immediately, resolving the pending yield and delivering the (potentially forged) `CKDResponse` to the original caller:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active()`, which confirms the caller is an attested participant — but does **not** require threshold agreement on the response value. [3](#0-2) 

Compare this to `respond` (threshold signatures), where the contract verifies the signature cryptographically against the derived public key before resolving: [4](#0-3) 

For signatures, a forged response is rejected on-chain regardless of who submits it. For `AppPublicKey` CKD, there is no equivalent on-chain guard.

---

### Impact Explanation

**Critical.** A single Byzantine attested participant (below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Construct an arbitrary `CKDResponse` with any `big_y` and `big_c` values.
3. Call `respond_ckd(request, forged_response)` directly.
4. The contract accepts the forged response and delivers it to the user via the yield-resume mechanism.

The user receives a wrong confidential key. Depending on how the key is used:
- Data encrypted under the forged key becomes permanently inaccessible (loss of funds/data).
- If the attacker controls the forged key material, they can decrypt the user's data (confidential key disclosure).

This constitutes **unauthorized confidential key derivation output without the required participant authorization** — a critical impact under the allowed scope.

---

### Likelihood Explanation

**High.** The attack requires only:
- Being a single attested MPC participant (not a threshold majority).
- Observing a pending `AppPublicKey` CKD request (publicly visible in contract state via `get_pending_ckd_request`).
- Calling `respond_ckd` before the honest leader does.

The MPC system is explicitly designed to tolerate Byzantine participants below the threshold. The `AppPublicKey` CKD path provides no threshold guarantee — it collapses to a single-node trust assumption, directly contradicting the system's security model.

---

### Recommendation

1. **Add on-chain output verification for `AppPublicKey`**: If the user's app public key is known, derive the expected output commitment from the master public key and verify the response against it, analogous to `ckd_output_check` for `AppPublicKeyPV`.

2. **Require threshold agreement before resolving**: If on-chain verification is not feasible for `AppPublicKey`, require a threshold of attested participants to submit matching responses before resolving the yield (similar to how `vote_pk` accumulates votes before transitioning state).

3. **Document the trust assumption explicitly**: If the current behavior is intentional, document that `AppPublicKey` CKD requests rely entirely on the honesty of whichever single node responds first, and consider deprecating this variant in favor of `AppPublicKeyPV`.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(app_pk)`. A pending CKD request is stored.
2. Attacker (a single attested participant) reads the pending request from contract state.
3. Attacker constructs a forged `CKDResponse { big_y: [0u8; 48].into(), big_c: [0u8; 48].into() }`.
4. Attacker calls `respond_ckd(request, forged_response)`.
5. The contract passes the empty `AppPublicKey` match arm with no check, then calls `resolve_yields_for`, delivering the forged response to the user.
6. The user's yield resolves with the attacker-controlled key material.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
