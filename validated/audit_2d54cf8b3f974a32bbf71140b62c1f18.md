Audit Report

## Title
ETH Branch of `_checkIfDepositAmountExceedesCurrentLimit` Omits Incoming `amount`, Bypassing `depositLimitByAsset` Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool` evaluates `totalAssetDeposits > limit` for native ETH deposits, never adding the incoming `amount` to the comparison. The ERC-20 branch correctly evaluates `totalAssetDeposits + amount > limit`. Any unprivileged depositor can therefore push total ETH deposits arbitrarily above the configured `depositLimitByAsset` cap in a single `depositETH` call, violating the protocol's stated deposit ceiling.

## Finding Description
The vulnerable function at `contracts/LRTDepositPool.sol` L676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount never added
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC-20
}
```

The ETH branch returns `false` (i.e., "not exceeded") for any deposit as long as the running total has not already crossed the limit, regardless of how large the incoming `amount` is. `_beforeDeposit` (L648–670) calls this function and reverts with `MaximumDepositLimitReached` only when it returns `true`. Because `amount` is never included in the ETH branch, the revert never fires for a deposit that would push the total over the limit. `depositETH` (L76–93) is a public, payable function that calls `_beforeDeposit` with `msg.value` as the deposit amount, making this reachable by any caller with no special privileges.

## Impact Explanation
The `depositLimitByAsset[ETH_TOKEN]` cap is the protocol's safety ceiling on total ETH exposure. Because the check omits `amount`, a single depositor can send an arbitrarily large ETH value and the cap is never enforced. The protocol mints rsETH for the full amount and forwards ETH to EigenLayer beyond the intended ceiling. No user funds are stolen or frozen; the protocol simply fails to enforce the deposit limit it advertises. This matches **Low – contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The entry point is the public, payable `depositETH` function — no role, whitelist, or special condition is required. Any depositor who observes that `totalAssetDeposits` is below the limit can trigger this in a single transaction with an arbitrarily large ETH value. The condition is trivially observable on-chain and repeatable. Likelihood is **High**.

## Recommendation
Add `amount` to the ETH branch, mirroring the ERC-20 branch:

```diff
 if (asset == LRTConstants.ETH_TOKEN) {
-    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
 }
```

## Proof of Concept
Assume:
- `depositLimitByAsset[ETH_TOKEN] = 1000 ether`
- `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`

A depositor calls `depositETH{value: 500 ether}(0, "")`:

1. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)`.
2. `totalAssetDeposits = 999 ether`.
3. ETH branch evaluates `999 ether > 1000 ether` → `false`.
4. No revert; 500 ETH is accepted and rsETH is minted.
5. Total ETH in protocol becomes `1499 ether`, 49.9% above the configured cap.

The same call with any ERC-20 LST correctly evaluates `999 + 500 > 1000` → `true` and reverts with `MaximumDepositLimitReached`.
