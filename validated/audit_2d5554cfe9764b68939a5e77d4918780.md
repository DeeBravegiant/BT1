Audit Report

## Title
Missing `+ amount` in ETH Deposit Limit Check Allows Limit Bypass — (`contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric limit check: the ERC-20 branch correctly adds the incoming `amount` to `totalAssetDeposits` before comparing against the cap, but the ETH branch omits `amount` entirely. As a result, any depositor can push the protocol's ETH holdings above the configured deposit limit in a single `depositETH` call, because the guard only fires when the limit is already breached before the deposit lands, never for the deposit that causes the breach.

## Finding Description
At [1](#0-0)  the function branches on asset type:

- **ETH branch (line 679):** `return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));`
- **ERC-20 branch (line 681):** `return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));`

The ETH branch is missing `+ amount`. `_beforeDeposit` at [2](#0-1)  reverts with `MaximumDepositLimitReached` only when this function returns `true`. Because the ETH branch never returns `true` for a deposit that would push the total over the limit (only for one arriving after the limit is already breached), the revert is never triggered for the boundary-crossing deposit itself. `depositETH` at [3](#0-2)  passes `msg.value` as `depositAmount` to `_beforeDeposit`, which is the value that should be included in the check but is not for ETH.

## Impact Explanation
The deposit limit is the protocol's primary mechanism for capping ETH exposure. Bypassing it allows the protocol to hold more ETH than the admin configured, violating the promised cap. No funds are stolen or frozen; the contract simply fails to enforce its stated deposit ceiling. This maps to the allowed impact: **Low — contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The exploit requires only a standard `depositETH` call with `msg.value > 0` from any unprivileged address. No special role, flash loan, or oracle manipulation is needed. Any depositor who monitors `getTotalAssetDeposits(ETH_TOKEN)` and submits a deposit that would cross the limit will trigger the bug. The entry point is fully permissionless and repeatable.

## Recommendation
Add `+ amount` to the ETH branch to match the ERC-20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This unifies both branches and correctly blocks any deposit that would cause the total to exceed the configured limit.

## Proof of Concept
Assume `depositLimitByAsset(ETH_TOKEN) = 1000 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.

1. Attacker calls `depositETH{value: 10 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)`.
3. ETH branch evaluates `999 ether > 1000 ether` → `false`. No revert.
4. `_mintRsETH` executes; attacker receives rsETH for 10 ETH.
5. Protocol now holds 1009 ETH — 9 ETH above the configured limit.

Under identical conditions, the ERC-20 branch would evaluate `999 ether + 10 ether > 1000 ether` → `true`, reverting with `MaximumDepositLimitReached`.

**Foundry test plan:**
```solidity
function test_eth_limit_bypass() public {
    // Set ETH deposit limit to 1000 ether
    lrtConfig.setDepositLimitByAsset(LRTConstants.ETH_TOKEN, 1000 ether);
    // Simulate existing deposits of 999 ether (mock getTotalAssetDeposits)
    // Call depositETH with 10 ether
    vm.deal(attacker, 10 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 10 ether}(0, "");
    // Assert total deposits now exceed 1000 ether — limit was not enforced
    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), 1000 ether);
}
```
