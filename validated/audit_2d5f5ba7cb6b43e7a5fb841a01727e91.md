### Title
Missing CKD Output Correctness Check for `AppPublicKey` Variant in `respond_ckd` - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC smart contract verifies the cryptographic correctness of a CKD response only when the request uses the `AppPublicKeyPV` (publicly-verifiable) variant. When the request uses the legacy `AppPublicKey` (privately-verifiable) variant, the response `(big_y, big_c)` is accepted and delivered to the user with **no on-chain correctness check whatsoever**. A single malicious attested MPC participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse`, causing the user to receive a cryptographically invalid confidential key.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function handles two variants of CKD requests:

```rust
// crates/contract/src/lib.rs lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which performs a BLS12-381 pairing check verifying that `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` — i.e., that the response is cryptographically bound to the MPC master key and the user's app identity. [2](#0-1) 

For `AppPublicKey`, the match arm is **empty** (`{}`). The response is immediately serialized and delivered to the user via `pending_requests::resolve_yields_for` with no cryptographic validation. [3](#0-2) 

This asymmetry is confirmed by the existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists`, which passes `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (raw garbage bytes) and the contract accepts them without complaint: [4](#0-3) 

Compare this to `respond` (ECDSA/EdDSA), which always verifies the signature before accepting it: [5](#0-4) 

The `AppPublicKey` variant is the **default and most commonly used** form of CKD request (it is the non-`--publicly-verifiable` path in the example CLI and the form used in the basic sandbox test). [6](#0-5) 

---

### Impact Explanation

A single malicious attested MPC participant (below the signing threshold) can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request. The contract will accept the forged response and deliver it to the requesting user or TEE application.

Consequences:
1. **Confidential key corruption**: The user's TEE application receives `(big_y, big_c)` that do not satisfy `big_c = hash_point * msk + app_pk * y`. When the app computes `confidential_key = big_c - app_sk * big_y`, it obtains a value that is not `hash_point * msk` — the intended deterministic secret. The app's key material is silently wrong.
2. **Attacker-controlled key substitution**: If the attacker submits a valid G1 point as `big_y = G * 0` (identity) and an arbitrary `big_c = G * k`, the user computes `G * k - app_sk * identity = G * k`. The attacker chose `k`, so they know the user's derived "confidential key". This enables impersonation of the TEE application or decryption of data the app encrypted under the substituted key.
3. **Silent failure**: Because the `AppPublicKey` variant is "privately verifiable," the user is expected to verify the output themselves. However, the contract's acceptance of the forged response provides a false signal of legitimacy, and many callers may not perform off-chain verification.

This breaks the production safety invariant that CKD outputs are cryptographically bound to the MPC network's master key and the user's app identity — the core security property of the CKD feature.

**Impact class**: Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without requiring threshold collusion or network-level DoS.

---

### Likelihood Explanation

- The attacker needs only to be a single attested MPC participant — strictly below the signing threshold. This is a realistic threat model for a Byzantine participant.
- The `AppPublicKey` variant is the default/legacy form of CKD request and is the path taken by the example CLI and basic tests.
- The attack requires only one malicious `respond_ckd` call, which is a standard node API call. No special privileges beyond being an attested participant are needed.
- The victim user has no on-chain mechanism to detect the forgery; they must perform off-chain verification, which is not enforced.

---

### Recommendation

Apply the same correctness-check discipline to the `AppPublicKey` variant as is already applied to `AppPublicKeyPV`. Specifically, for `AppPublicKey` requests, verify that the response satisfies the privately-verifiable CKD equation before accepting it. Since `AppPublicKey` only provides a G1 point (no G2 component), a different check is needed — for example, verifying `e(big_c, g2) = e(big_y, g2)^y · e(hash_point, public_key)` using the known `app_pk` as a G1 point. Alternatively, require all new CKD requests to use `AppPublicKeyPV` (which is already publicly verifiable on-chain) and deprecate the unverifiable `AppPublicKey` variant.

---

### Proof of Concept

1. User submits a CKD request using `AppPublicKey` variant:
   ```json
   {"request": {"derivation_path": "mykey", "app_public_key": "bls12381g1:<valid_g1_point>", "domain_id": 2}}
   ```
2. A malicious attested MPC participant calls `respond_ckd` with a forged response:
   ```json
   {"request": {...}, "response": {"big_y": "bls12381g1:<attacker_chosen_point>", "big_c": "bls12381g1:<attacker_chosen_point>"}}
   ```
3. The contract executes lines 675–688 of `crates/contract/src/lib.rs`: the `AppPublicKey` arm is empty, no check is performed, and `resolve_yields_for` delivers the forged `(big_y, big_c)` to the user.
4. The user's TEE application computes `confidential_key = big_c - app_sk * big_y`, obtaining a value that is not `hash_point * msk`. The application silently uses a wrong key.

This is directly confirmed by the existing unit test at line 3424, which passes `[1u8; 48]` / `[2u8; 48]` (invalid G1 encodings) as the response and the contract accepts them without error. [7](#0-6)
