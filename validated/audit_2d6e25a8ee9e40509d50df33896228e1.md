### Title
Cross-Chain Metadata Signature Replay Enables Unauthorized Token Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`)

---

### Summary

The `deployToken` / `deploy_token` functions on EVM and Starknet verify a metadata signature that contains **no chain-ID binding**, while the `finTransfer` / `fin_transfer` functions on the same chains explicitly include `omniBridgeChainId` in the signed payload. Because all EVM deployments and Starknet share the same MPC-derived key (`nearBridgeDerivedAddress` / `omni_bridge_derived_address`), a metadata signature produced for one chain is cryptographically valid on every other chain. Any observer can replay it to deploy a token on an unintended chain, permanently blocking the official deployment there.

---

### Finding Description

**`finTransfer` (chain-bound — correct)**

In `OmniBridge.sol`, the Borsh-encoded payload passed to `keccak256` before ECDSA recovery includes `omniBridgeChainId` twice:

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [1](#0-0) 

In `omni_bridge.cairo`, `fin_transfer` passes the stored chain ID into serialisation:

```cairo
_verify_borsh_signature(
    ref self, @payload.to_borsh(self.omni_bridge_chain_id.read()), signature,
);
``` [2](#0-1) 

**`deployToken` (not chain-bound — vulnerable)**

In `OmniBridge.sol`, the Borsh-encoded payload for `deployToken` is:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

No `omniBridgeChainId` byte is present. The same omission exists on Starknet:

```cairo
_verify_borsh_signature(ref self, @payload.to_borsh(), signature);
``` [4](#0-3) 

The NEAR-side `MetadataPayload` struct confirms no chain field exists in the signed object:

```rust
pub struct MetadataPayload {
    pub prefix: PayloadType,
    pub token: String,
    pub name: String,
    pub symbol: String,
    pub decimals: u8,
}
``` [5](#0-4) 

The signature is emitted publicly as a NEAR event in `sign_log_metadata_callback`: [6](#0-5) 

All EVM chains (Eth=0, Arb=3, Base=4, BNB=5, Pol=8, HyperEvm=9, Abs=11) and Starknet (10) share the same `nearBridgeDerivedAddress` derived from the single NEAR MPC key. The `ChainKind` u8 discriminants confirm the distinct IDs: [7](#0-6) 

---

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on chain A (e.g., Ethereum) extracts the `signatureData` from calldata and submits it to `deployToken` on chain B (e.g., Arbitrum). The signature passes `ECDSA.recover` / `verify_eth_signature` because the hash is identical — no chain discriminant was mixed in. The token is then registered in chain B's bridge mappings (`nearToEthToken`, `isBridgeToken`). Consequences:

1. **Permanent deployment block**: The guard `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` now fires for every future official deployment attempt on chain B. The admin must fall back to `addCustomToken` (privileged path) to recover. [8](#0-7) 

2. **Unauthorized token registration**: The attacker-triggered deployment registers the token in the bridge's canonical mapping. If the NEAR bridge backend automatically begins issuing `finTransfer` signatures for any token it observes registered on a chain, this opens a path to minting on chain B without the token having been officially supported there — producing unbacked supply.

3. **Cross-domain signature acceptance**: A signature scoped to chain A is accepted as authoritative on chain B, violating the "sufficiently-bound" property required of bridge signatures.

---

### Likelihood Explanation

- The metadata signature is emitted as a public NEAR event immediately after `log_metadata` is called; no privileged access is needed to observe it.
- Replaying it requires only a standard `deployToken` call on the target chain — a fully unprivileged, public entrypoint.
- The bridge supports at least seven EVM chains plus Starknet, all sharing the same MPC key, so every metadata signature is replayable on six or more additional chains.
- The attack is front-runnable: an attacker can monitor the NEAR mempool/events and replay before the official relayer does.

---

### Recommendation

Include `omniBridgeChainId` in the Borsh-encoded metadata payload before hashing, mirroring the pattern already used in `finTransfer`:

**EVM (`OmniBridge.sol`)**:
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Starknet (`omni_bridge.cairo`)**:
```cairo
- _verify_borsh_signature(ref self, @payload.to_borsh(), signature);
+ _verify_borsh_signature(ref self, @payload.to_borsh(self.omni_bridge_chain_id.read()), signature);
```

**NEAR (`near/omni-types/src/lib.rs`)**:
Add a `chain_id: u8` field to `MetadataPayload` and populate it in `log_metadata_callback` before signing, so the MPC key signs a chain-specific payload.

---

### Proof of Concept

1. Call `log_metadata(tokenAddress)` on the NEAR bridge for token `usdc.near`. The MPC key signs `keccak256(borsh(Metadata | "usdc.near" | "USD Coin" | "USDC" | 6))`. The `SignTransferEvent` is emitted on NEAR with `signatureData = S`.

2. The official relayer calls `deployToken(S, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Ethereum (`omniBridgeChainId = 0`). Token deployed at `0xAAA`.

3. Attacker reads `S` from the NEAR event log. Calls `deployToken(S, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Arbitrum (`omniBridgeChainId = 3`). The hash is identical — `ECDSA.recover(hash, S) == nearBridgeDerivedAddress` — so the check passes. A new token `0xBBB` is deployed on Arbitrum and registered as the canonical USDC bridge token.

4. Any subsequent official `deployToken` call for `usdc.near` on Arbitrum reverts with `"ERR_TOKEN_EXIST"`. The bridge is permanently locked into the attacker-triggered deployment on Arbitrum.
