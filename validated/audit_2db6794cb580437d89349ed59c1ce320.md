### Title
Frankenstein ChunkEndorsementV2 bypasses cache deduplication guard, silently evicting legitimate endorsements and blocking chunk inclusion — (`core/primitives/src/stateless_validation/chunk_endorsement.rs`)

### Summary

`ChunkEndorsementV2::verify()` verifies `signature` against `inner` and `metadata_signature` against `metadata` as two independent checks, but never cross-checks that `inner.chunk_hash` belongs to the chunk identified by `metadata.(shard_id, epoch_id, height_created)`. A network peer who observes two legitimate endorsements from the same validator for two different shards (or heights) can splice them into a "Frankenstein" endorsement that passes all validation, poisons the `ChunkEndorsementTracker` cache under the target shard's key with the wrong `chunk_hash`, and causes the "already received" guard to silently drop the legitimate endorsement when it arrives. `collect_chunk_endorsements` then filters out the poisoned entry by `chunk_hash`, so the validator's stake is never counted, preventing the chunk from reaching the endorsement threshold.

### Finding Description

**Struct layout** (`core/primitives/src/stateless_validation/chunk_endorsement.rs` lines 100–118):

```
ChunkEndorsementV2 {
    inner:              ChunkEndorsementInnerV1 { chunk_hash, signature_differentiator }
    signature:          Signature  // signs `inner`
    metadata:           ChunkEndorsementMetadata { account_id, shard_id, epoch_id, height_created }
    metadata_signature: Signature  // signs `metadata`
}
```

`verify()` checks:
- `signature` is a valid signature of `inner` under the validator's key ✓
- `metadata_signature` is a valid signature of `metadata` under the validator's key ✓

No check that `inner.chunk_hash` is the hash of the chunk at `(metadata.shard_id, metadata.epoch_id, metadata.height_created)`.

**Attack construction:**

Given two legitimate endorsements from validator V observed on the network:
- **Endorsement A**: shard 0, height H₁, `chunk_hash_A`
- **Endorsement B**: shard 1, height H₂, `chunk_hash_B`

Construct Frankenstein F:
```
F.inner              = A.inner              // chunk_hash_A
F.signature          = A.signature          // valid sig of inner_A
F.metadata           = B.metadata           // shard_id=1, epoch_id=E, height=H₂
F.metadata_signature = B.metadata_signature // valid sig of metadata_B
```

**Verification passes** (`verify()` lines 112–117): both independent checks succeed because each signature is valid for its own component.

**`validate_chunk_relevant_as_validator`** uses `chunk_production_key()` derived from `metadata` → checks validator V is assigned to shard 1 at height H₂ → TRUE (since B was legitimate).

**Cache insert** (`process_chunk_endorsement` lines 59–63): stores `(chunk_hash_A, sig_A)` under key `(shard_id=1, epoch_id=E, height=H₂)` for account V.

**Legitimate endorsement B arrives**: the "already received" guard at line 50 finds V already in the cache for key `(shard_id=1, epoch_id=E, height=H₂)` → returns `Ok(())` without updating.

**`collect_chunk_endorsements`** for shard 1's chunk (lines 107–111): filters `chunk_hash == chunk_header.chunk_hash()` → `chunk_hash_A ≠ chunk_hash_B` → V's entry is excluded → V's stake not counted.

### Impact Explanation

If the attacker can deliver Frankenstein endorsements for a sufficient number of validators assigned to a target shard before their legitimate endorsements arrive, `collect_chunk_endorsements` will return `is_endorsed = false` for that shard's chunk. The block producer cannot include the chunk, stalling chain progress for that shard. This is a targeted liveness attack against chunk inclusion in stateless validation.

### Likelihood Explanation

The attacker is any network peer who can observe `ChunkEndorsementMessage` traffic (these are broadcast peer messages). The only requirement is observing two endorsements from the same validator for any two different shards or heights within the same epoch. In a multi-shard network, validators are routinely assigned to multiple shards per epoch, making cross-shard endorsement pairs readily observable. The attack requires no validator key material, no privileged access, and no special timing beyond delivering the Frankenstein before the legitimate endorsement.

### Recommendation

Add a cross-check inside `ChunkEndorsementV2::verify()` (or in `validate_chunk_endorsement_signature`) that asserts `inner.chunk_hash` is consistent with the chunk identified by `metadata`. Since the chunk hash is not directly derivable from `(shard_id, epoch_id, height_created)` alone at verification time, the simplest fix is to bind `inner` and `metadata` together cryptographically: sign a single combined structure (e.g., `(chunk_hash, shard_id, epoch_id, height_created)`) rather than two independent structures. Alternatively, `process_chunk_endorsement` should reject any endorsement where the cached `chunk_hash` differs from the incoming one, rather than treating it as a duplicate.
