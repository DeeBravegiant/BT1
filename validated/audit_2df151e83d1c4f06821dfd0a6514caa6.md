### Title
Unverified `payload_hash` Binding in `respond_verify_foreign_tx` Allows Cross-Request Signature Substitution - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract verifies that a submitted signature is cryptographically valid over `response.payload_hash`, but never verifies that `response.payload_hash` is actually derived from the foreign-chain transaction described in `request`. A single malicious attested participant can reuse a legitimately produced signature from a prior signing session to resolve any pending `verify_foreign_transaction` request with an arbitrary, mismatched `payload_hash`.

---

### Finding Description

In `respond_verify_foreign_tx` (lines 692–753 of `crates/contract/src/lib.rs`), the contract performs the following checks:

1. The caller is an attested participant.
2. The submitted `response.signature` is a valid ECDSA signature over `response.payload_hash` against the root MPC public key.
3. The `request` key matches a pending yield in `pending_verify_foreign_tx_requests`.

What it does **not** check is whether `response.payload_hash` is the correct hash derived from the foreign-chain transaction encoded in `request.request`. The `payload_hash` lives exclusively in the `VerifyForeignTransactionResponse` and is never cross-referenced against the `VerifyForeignTransactionRequest` fields. [1](#0-0) 

The `VerifyForeignTransactionRequest` stored as the pending-yield key contains `domain_id`, `request` (the foreign-chain tx), and `payload_version` — but no `payload_hash`. [2](#0-1) 

The response is resolved directly into the yield without any binding check: [3](#0-2) 

---

### Impact Explanation

A single malicious attested participant can:

1. Observe a legitimately completed signing session for `request_A`, obtaining `(payload_hash_A, sig_A)` — a valid signature produced by threshold cooperation.
2. While `request_B` (a different foreign-chain transaction) is pending, call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
3. The contract accepts the call: `sig_A` is valid over `payload_hash_A` against the root key, and `request_B` is a live pending yield.
4. `request_B`'s yield is resolved with `payload_hash_A` and `sig_A` — a signature over the **wrong** transaction.

The caller of `request_B` receives a valid MPC signature attesting to transaction A's hash in the context of their bridge operation. This constitutes **forged foreign-chain verification**: the MPC network's attestation is decoupled from the actual transaction it was supposed to verify, enabling double-spend or replay conditions on any bridge that trusts the returned `(payload_hash, signature)` pair.

This matches: **High — forged foreign-chain verification / light-client-style verification bypass causing invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

- The attack requires only **one** malicious attested participant — strictly below the signing threshold.
- No threshold cooperation is needed for the exploit step; the attacker reuses a signature already produced by honest nodes.
- The `respond_verify_foreign_tx` entry point is callable by any single attested participant with no additional access control beyond attestation.
- Any bridge contract consuming the `(payload_hash, signature)` output without independently re-deriving the expected hash from the transaction ID is immediately vulnerable. [4](#0-3) 

---

### Recommendation

Before accepting the response, the contract must recompute the expected `payload_hash` from the fields of `request` (chain ID, transaction ID, `payload_version`) using the same deterministic derivation the nodes use, and assert:

```rust
assert_eq!(response.payload_hash, expected_payload_hash_from_request);
```

This binding check should be performed on-chain in `respond_verify_foreign_tx`, analogous to how `respond` derives and verifies the expected public key from `request.tweak` before accepting a signature. [5](#0-4) 

---

### Proof of Concept

```
// Step 1: Honest flow — request_A is submitted and signed
verify_foreign_transaction(request_A)  // pending yield created for request_A
// Nodes collectively sign payload_hash_A → sig_A is produced off-chain

// Step 2: request_B is submitted by a different user
verify_foreign_transaction(request_B)  // pending yield created for request_B

// Step 3: Malicious attested participant submits mismatched response
respond_verify_foreign_tx(
    request  = request_B,           // valid pending key → yield lookup succeeds
    response = {
        payload_hash: payload_hash_A,  // hash of transaction A, not B
        signature:    sig_A,           // valid sig over payload_hash_A → passes verify_ecdsa_signature
    }
)

// Result: request_B's yield resolves with (payload_hash_A, sig_A)
// Caller of request_B receives a valid MPC attestation for the WRONG transaction
```

The contract's only guard — `verify_ecdsa_signature(sig_A, payload_hash_A, root_pk)` — passes because the signature is cryptographically correct; the missing check is that `payload_hash_A ≠ hash(request_B.transaction)`. [6](#0-5)
