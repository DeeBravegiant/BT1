### Title
Stale `fast_transfers` Entries for Non-NEAR Destination Chains Permanently Lock Relayer Storage Deposits — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a fast transfer is finalized to a **NEAR** destination, the `fast_transfers` map entry is fully removed via `remove_fast_transfer`, which refunds the relayer's storage deposit. When a fast transfer is finalized to a **non-NEAR** destination, the entry is only marked as finalised via `mark_fast_transfer_as_finalised` and is **never removed**. The storage deposit charged to the relayer at fast-transfer registration time is permanently locked and unclaimable.

---

### Finding Description

The `fast_transfers` map (`LookupMap<FastTransferId, FastTransferStatusStorage>`) tracks relayer-executed fast transfers. Two distinct finalization paths exist:

**Path A — NEAR destination** (`process_fin_transfer_to_near`, line 1899):
```rust
self.remove_fast_transfer(&fast_transfer.id());
```
`remove_fast_transfer` (lines 2284–2300) removes the entry from the map **and** refunds the storage cost to `storage_owner` (the relayer).

**Path B — Non-NEAR destination** (`process_fin_transfer_to_other_chain`, line 2045):
```rust
self.mark_fast_transfer_as_finalised(&fast_transfer.id());
```
`mark_fast_transfer_as_finalised` (lines 2275–2282) only sets `finalised = true` and re-inserts the entry. It does **not** remove the entry and does **not** refund the storage deposit.

The storage deposit is charged when `add_fast_transfer` is called during the relayer's fast-transfer execution for non-NEAR chains. Since `remove_fast_transfer` is never called for non-NEAR destinations, the storage deposit is permanently deducted from the relayer's `accounts_balances` entry and can never be reclaimed — even via `storage_unregister`.

---

### Impact Explanation

Every relayer-executed fast transfer to a non-NEAR chain (EVM, Solana, Starknet, BTC, etc.) leaves a permanently stale entry in `fast_transfers`. The NEAR storage deposit charged to the relayer at registration time is irreversibly locked. Over time, as relayers process many fast transfers to non-NEAR chains, the cumulative locked storage deposits grow unboundedly and are permanently unclaimable. This matches the allowed impact: **permanently unclaimable user value in fast-transfer flows**.

---

### Likelihood Explanation

The fast-transfer path to non-NEAR chains is a normal, intended protocol operation. Any relayer executing fast transfers to EVM/Solana/Starknet/BTC destinations triggers this bug on every single finalization. No special conditions or attacker privileges are required — any unprivileged relayer participating in the protocol is affected.

---

### Recommendation

In `process_fin_transfer_to_other_chain`, replace the call to `mark_fast_transfer_as_finalised` with `remove_fast_transfer`, mirroring the cleanup performed in `process_fin_transfer_to_near`. If the finalised state must be preserved for replay protection, the replay guard is already provided by `finalised_transfers` (which is populated by `add_fin_transfer` at line 1990 in both paths), so the `fast_transfers` entry is redundant after finalization.

---

### Proof of Concept

1. Relayer calls `fast_fin_transfer` with a non-NEAR recipient → `fast_fin_transfer_to_other_chain` is invoked → `add_fast_transfer` charges storage from the relayer's `accounts_balances`.
2. Later, `process_fin_transfer_to_other_chain` is called with the matching proof → `get_fast_transfer_status` returns `Some` → `mark_fast_transfer_as_finalised` is called → tokens are sent to the relayer.
3. The `fast_transfers` entry remains in the map with `finalised = true`. The storage deposit is never refunded.
4. The relayer calls `storage_unregister` — it can only reclaim `available` balance. The portion consumed by the stale entry is permanently gone.

    fn remove_fast_transfer(&mut self, fast_transfer_id: &FastTransferId) {
        let storage_usage = env::storage_usage();
        let fast_transfer = self
            .fast_transfers
            .remove(fast_transfer_id)
            .map(storage::FastTransferStatusStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&fast_transfer.storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances
                .insert(&fast_transfer.storage_owner, &storage);
        }
    }
