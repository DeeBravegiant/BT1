### Title
Unprivileged Peer Can Trigger `AssertionError` in `get_minter_did` / `puzzle_solution_received`, Permanently Dropping DID-NFT Coin States from Wallet Sync — (`chia/wallet/wallet_state_manager.py`, `chia/wallet/nft_wallet/nft_wallet.py`)

---

### Summary

An unprivileged peer can cause `AssertionError` inside `puzzle_solution_received` or `get_minter_did` by returning an empty coin-state list (or a list with `spent_height=None`) in response to `get_coin_state` queries made during NFT sync. Because `AssertionError` is not a `PeerRequestException`, the outer exception handler in `_add_coin_states` silently drops the coin state from the retry store instead of scheduling a retry. The NFT is never added to the wallet DB for as long as the wallet is connected to the malicious peer.

**Why an untrusted peer can trigger these:** `get_coin_state` for untrusted peers validates each returned coin state individually, but does **not** require the peer to return any coin states at all. If the peer returns an empty `coin_states` list, `valid_list` is empty and the function returns `[]` — no exception, no ban. [4](#0-3) 

**Why the failure is non-retriable:** The outer handler in `_add_coin_states` only schedules a retry for `PeerRequestException` or `aiosqlite.Error`. `AssertionError` falls into the `else` branch, which calls `retry_store.remove_state` and continues — the coin state is permanently discarded for this sync cycle. [5](#0-4) 

---

### Impact Explanation

Any unprivileged peer the wallet connects to can prevent DID-enabled NFTs from being added to the wallet DB. The NFT remains invisible and unspendable in the wallet for as long as the wallet is connected to the malicious peer. Because the state is removed from the retry store (not added), the wallet will not automatically recover — it requires a manual resync against a trusted peer. This is a long-lived inability to process valid sync updates, matching the **High** impact category.

---

### Likelihood Explanation

- Wallets routinely connect to untrusted peers (no trusted peer configured is the default for many users).
- The attack requires no special privileges: any peer can respond to `register_for_coin_updates` with an empty list.
- The attacker does not need to forge any on-chain data; they only need to omit coin states from their responses.
- The attack is repeatable: every time the wallet re-subscribes and the malicious peer is still connected, the same failure recurs.

---

### Recommendation

1. Replace the bare `assert` statements in `puzzle_solution_received` and `get_minter_did` with explicit checks that raise `PeerRequestException` on unexpected peer responses. This ensures the retry store schedules a retry against a different peer.
2. Alternatively, wrap the `get_minter_did` call in a `try/except PeerRequestException` and propagate it upward so the outer handler in `_add_coin_states` treats it as a retriable peer error.
