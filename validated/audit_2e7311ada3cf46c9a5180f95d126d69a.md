### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Creates Unbacked Cross-Chain Supply — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records and emits the caller-supplied `amount` as the canonical bridged value without measuring the actual tokens received. For fee-on-transfer ERC-20s (e.g., USDT in fee mode, STA), the vault receives `amount − fee_on_transfer`, but the `InitTransfer` event broadcasts the full `amount`. The NEAR bridge processes that event and credits the full `amount` to the recipient, creating unbacked wrapped supply. When those tokens are bridged back, the EVM vault cannot cover the release, permanently draining reserves shared by all depositors.

---

### Finding Description

In `initTransfer`, when `tokenAddress` is neither a bridge token nor a custom-minter token, the contract executes a plain `safeTransferFrom`:

```solidity
// OmniBridge.sol lines 407–412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← caller-supplied; actual receipt may be less
    );
}
```

Immediately after, the original `amount` is emitted without any before/after balance check:

```solidity
// OmniBridge.sol lines 427–436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // ← inflated for fee-on-transfer tokens
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) [2](#0-1) 

The NEAR bridge (`near/omni-bridge/src/lib.rs`) consumes the `InitTransfer` event via `fin_transfer_callback`, which reads `init_transfer.amount` directly from the prover result and uses it to mint or release tokens to the recipient on NEAR:

```rust
// lib.rs lines 726–736
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [3](#0-2) 

Because the event value is the inflated `amount`, the NEAR side credits more tokens than the EVM vault actually holds.

---

### Impact Explanation

Each deposit of a fee-on-transfer token creates a deficit: the vault holds `amount − fee_on_transfer` but the NEAR side mints `amount`. When the recipient bridges back, `finTransfer` on EVM attempts:

```solidity
// OmniBridge.sol lines 351–354
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount      // ← the inflated amount
);
``` [4](#0-3) 

The vault is short by `fee_on_transfer` per round-trip. Repeated deposits accumulate the deficit until the vault is exhausted, at which point all remaining depositors' redemptions revert — a permanent fund lock for honest users. This is a backing-guarantee break: wrapped supply on NEAR exceeds locked collateral on EVM.

---

### Likelihood Explanation

The bridge has no token whitelist that excludes fee-on-transfer tokens. Any ERC-20 that is not in `isBridgeToken` and has no `customMinters` entry is accepted by the `else` branch. USDT (with fee mode enabled), STA, and similar tokens are real, deployed assets. A single user depositing such a token triggers the discrepancy; no privileged access or colluding party is required. [5](#0-4) 

---

### Recommendation

Replace the direct `safeTransferFrom` with a before/after balance check and use the measured delta as the canonical amount:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(received > 0, "zero received");
amount = uint128(received);   // use actual received amount downstream
```

Alternatively, maintain an explicit allowlist of supported tokens and reject any token whose `transfer` behaviour is non-standard.

---

### Proof of Concept

1. USDT is configured with a 1% transfer fee and is not in `isBridgeToken` / `customMinters`.
2. Attacker calls `initTransfer(USDT, 1000e6, 0, 0, "near:attacker.near", "")`.
3. `safeTransferFrom` moves 1000 USDT from attacker; vault receives 990 USDT (1% fee deducted by token).
4. `InitTransfer` event is emitted with `amount = 1000e6`.
5. NEAR relayer calls `fin_transfer` with the proof; NEAR bridge credits attacker with 1000 USDT-equivalent wrapped tokens.
6. Attacker calls NEAR bridge to bridge 1000 tokens back to EVM.
7. NEAR bridge burns 1000 tokens and signs a `TransferMessagePayload` for 1000 USDT.
8. Relayer calls `finTransfer` on EVM; vault attempts `safeTransfer(attacker, 1000e6)` but only holds 990 USDT — reverts, or if other deposits cover it, drains 10 USDT from the shared pool.
9. Repeated across many deposits, the vault is progressively drained until honest users' withdrawals fail permanently.
