Audit Report

## Title
Unbounded Gas Consumption in `LRTOracle._getTotalEthInProtocol()` via Nested Iteration Over `supportedAssets` × `nodeDelegatorQueue` — (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle._getTotalEthInProtocol()` performs an O(n × m) nested external-call loop: for each supported asset it calls `LRTDepositPool.getTotalAssetDeposits()`, which itself iterates over every entry in `nodeDelegatorQueue` issuing three external calls per NDC per asset. Because `updateRSETHPrice()` is unrestricted and both arrays grow through normal governance operations with no on-chain cap on their product, the transaction gas cost is unbounded and will eventually exceed the block gas limit, permanently freezing the rsETH price.

## Finding Description
`updateRSETHPrice()` is declared `public whenNotPaused` with no caller restriction, making it callable by any EOA or contract. [1](#0-0) 

It delegates to `_updateRsETHPrice()`, which calls `_getTotalEthInProtocol()`. That function fetches the full `supportedAssetList` and, for each asset, issues one oracle `staticcall` and one cross-contract call to `LRTDepositPool.getTotalAssetDeposits()`. [2](#0-1) 

`getTotalAssetDeposits` delegates to `getAssetDistributionData`, which loops over every entry in `nodeDelegatorQueue` and issues **three** external calls per NDC per asset: `IERC20.balanceOf`, `INodeDelegator.getAssetBalance`, and `INodeDelegator.getAssetUnstaking`. [3](#0-2) 

The ETH path (`getETHDistributionData`) similarly loops over all NDCs with three external calls each. [4](#0-3) 

`maxNodeDelegatorLimit` is freely updatable by admin with no on-chain upper bound, and the check only prevents shrinking below the current queue length. [5](#0-4) 

New supported assets are added via `TIME_LOCK_ROLE` with no cap on the list length. There is no on-chain constraint on the product `supportedAssets.length × nodeDelegatorQueue.length`, so total external calls per `updateRSETHPrice()` invocation = `n_assets × n_NDCs × 3 + n_assets` with no ceiling.

## Impact Explanation
If `updateRSETHPrice()` reverts out-of-gas, `rsETHPrice` is permanently frozen at its last stored value. Every subsequent mint uses the stale price: [6](#0-5) 

A stale price causes depositors to receive incorrect rsETH amounts (over- or under-minting), breaking share accounting for all users. The price-update mechanism cannot be restored without a contract upgrade. This matches the allowed impact **Medium — Unbounded gas consumption**, with secondary share mis-accounting.

## Likelihood Explanation
The protocol is explicitly designed to support multiple LSTs and scale its EigenLayer delegation infrastructure. Adding assets via `TIME_LOCK_ROLE` and raising `maxNodeDelegatorLimit` are routine governance operations, not attacks. At 10 assets × 20 NDCs, the inner loop alone issues 600 external calls; each `getAssetBalance` call into EigenLayer strategy contracts realistically costs 20,000–50,000 gas, putting the loop at 12–30 M gas before any overhead — approaching or exceeding the 30 M block gas limit. Any unprivileged caller can trigger the revert once the protocol reaches this scale.

## Recommendation
1. Enforce an on-chain cap on `maxNodeDelegatorLimit` that accounts for the maximum supported asset count, preventing the product from exceeding a safe gas budget.
2. Refactor `_getTotalEthInProtocol()` to maintain an incrementally updated TVL accumulator (updated on deposit/withdrawal/rebalance events) rather than recomputing from scratch on every price update.
3. Alternatively, introduce a paginated TVL aggregation pattern where per-asset totals are stored lazily and `updateRSETHPrice()` reads cached values rather than iterating live.

## Proof of Concept
1. Governance adds 10 supported LST assets via `LRTConfig.addNewSupportedAsset` (TIME_LOCK_ROLE).
2. Admin calls `LRTDepositPool.updateMaxNodeDelegatorLimit(20)` then `addNodeDelegatorContractToQueue` with 20 NDC addresses.
3. Any unprivileged EOA calls `LRTOracle.updateRSETHPrice()`.
4. `_getTotalEthInProtocol()` executes 10 × 20 × 3 = 600 external calls into NDC and EigenLayer strategy contracts, plus 10 oracle calls.
5. At a realistic 20,000–50,000 gas per NDC call, the loop consumes 12–30 M gas; with warm-storage overhead and the rest of `_updateRsETHPrice()` logic, the transaction reverts out-of-gas.
6. `rsETHPrice` is frozen; all subsequent `getRsETHAmountToMint` calls divide by the stale price, mis-accounting rsETH minted for every depositor.

**Foundry fork test plan:** Deploy against a mainnet fork, impersonate the timelock/admin to add 10 assets and 20 NDCs, then call `updateRSETHPrice()` with `vm.expectRevert()` and assert the transaction gas usage exceeds the block gas limit as counts are incremented in a fuzz loop over `(n_assets, n_NDCs)`.

```text
    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
```
