Audit Report

## Title
Missing Minimum Output Slippage Guard in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The L2 pool `deposit()` functions in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`, and `RSETHPool` accept no `minRsETHAmount` parameter, while the mainnet `LRTDepositPool.depositETH()` enforces one. The L2 oracle rate is updated periodically via cross-chain LayerZero messages, not every block. If the oracle rate is pushed between a user's off-chain quote via `viewSwapRsETHAmountAndFee()` and their `deposit()` execution, the user receives fewer wrsETH/rsETH than the protocol's own view function quoted, with no on-chain recourse.

## Finding Description
`RSETHPoolV3.deposit(string)` and `RSETHPoolV3.deposit(address,uint256,string)` compute the minted amount entirely from the live oracle rate at execution time with no floor check:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
// ...
wrsETH.mint(msg.sender, rsETHAmount);
```

The oracle backing `getRate()` is a `CrossChainRateReceiver` whose `rate` state variable is updated only when a LayerZero message arrives from the mainnet provider. Between oracle pushes, the stored rate is stale. When a new push arrives and `rate` increases (rsETH has appreciated), any pending `deposit()` transaction that was quoted at the old rate will execute at the new, higher rate, yielding fewer wrsETH than quoted.

The mainnet `LRTDepositPool.depositETH()` already guards against this with `minRSETHAmountExpected`, passed into `_beforeDeposit()`. No equivalent guard exists in any of the four L2 pool variants.

Existing checks (`nonReentrant`, `whenNotPaused`, `limitDailyMint`, `InvalidAmount`) are entirely orthogonal to this issue and provide no protection.

## Impact Explanation
**Low.** The user receives fewer wrsETH/rsETH tokens than the protocol's own `viewSwapRsETHAmountAndFee()` quoted. The deposited ETH value is not stolen — the fewer tokens received are each worth proportionally more ETH — but the contract fails to deliver the promised token quantity. This matches the allowed impact: *"Contract fails to deliver promised returns, but doesn't lose value."* Downstream failures are possible (e.g., a user depositing to meet a specific collateral threshold receives less than required).

## Likelihood Explanation
**Low.** No attacker is required. The condition arises naturally whenever the L2 oracle is updated between a user's off-chain quote and their transaction's execution. The rsETH/ETH rate appreciates slowly (staking rewards), so the shortfall per transaction is small but non-zero and grows with deposit size and time elapsed since the last oracle push. The oracle update is a permissionless LayerZero message delivery, so the timing is not under the user's control.

## Recommendation
Add a `minRsETHAmount` parameter to all four `deposit()` overloads across `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolNoWrapper`, and `RSETHPool`, mirroring the pattern in `LRTDepositPool.depositETH()`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    // ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    require(rsETHAmount >= minRsETHAmount, "Slippage: insufficient output");
    // ...
}
```

## Proof of Concept
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain at oracle rate `R`; UI shows expected output `X = (1e18 - fee) * 1e18 / R`.
2. Before the user's transaction is mined, the LayerZero relayer delivers a new rate `R' > R` to `CrossChainRateReceiver.lzReceive()`, updating `rate` on-chain.
3. User's `deposit{value: 1 ether}(referralId)` executes; `rsETHAmount = (1e18 - fee) * 1e18 / R'` which is strictly less than `X`.
4. `wrsETH.mint(msg.sender, rsETHAmount)` succeeds with no revert; user receives fewer tokens than quoted with no recourse.

Foundry fork test plan: fork the target L2, deploy a mock `CrossChainRateReceiver` with rate `R`, call `viewSwapRsETHAmountAndFee` to record `X`, then call `lzReceive` to push `R' = R * 1001 / 1000`, then call `deposit{value: 1 ether}`, and assert `wrsETH.balanceOf(user) < X`.
