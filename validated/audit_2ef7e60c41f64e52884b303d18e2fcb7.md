### Title
Attached NEAR Deposit Not Refunded When Existing Participant Updates Attestation - (`crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is a `#[payable]` method that accepts NEAR deposits. Its deposit-handling and refund logic is gated behind a condition that evaluates to `false` when an existing governance participant updates (rather than first-inserts) their attestation. In that branch, any NEAR attached to the call is silently retained by the contract with no refund path.

---

### Finding Description

In `submit_participant_info`, after `add_participant` succeeds, the code computes:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit check and refund logic ...
}
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` is `false` — i.e., the caller is an existing governance participant (`voter_account().is_ok()`) AND the insertion result is `UpdatedExistingParticipant` — the entire deposit-handling block is skipped. There is no `else` branch and no unconditional refund. Any NEAR attached to the call is permanently absorbed by the contract.

`ParticipantInsertion::UpdatedExistingParticipant` is returned by `add_participant` whenever the TLS key already exists in `stored_attestations` and the submitting account matches the stored owner:

```rust
Ok(match insertion {
    Some(_previous_attestation) => ParticipantInsertion::UpdatedExistingParticipant,
    None => ParticipantInsertion::NewlyInsertedParticipant,
})
``` [2](#0-1) 

This is the normal path for periodic re-attestation by an active participant. The function is declared `#[payable]`, so the NEAR runtime will accept any deposit the caller attaches without reverting. [3](#0-2) 

---

### Impact Explanation

Any NEAR tokens attached to a `submit_participant_info` call by an existing participant performing a routine attestation update are permanently locked in the contract. There is no recovery path: the contract has no withdrawal mechanism for accidentally deposited funds, and the refund promise is never scheduled for this branch. This breaks the production accounting invariant that excess deposits must be returned to the caller — the same invariant explicitly enforced in `require_deposit`, `propose_update`, and the new-participant branch of `submit_participant_info` itself. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The MPC node's `periodic_attestation_submission` task re-submits attestations on a 1-hour cadence and on attestation-removal events. The function is `#[payable]`, so any caller — including the node itself or a script — that attaches even 1 yoctoNEAR to cover potential storage costs will silently lose those funds. The update path (`UpdatedExistingParticipant`) is the steady-state path for every active participant after their first submission, making this the most frequently exercised branch in production.

---

### Recommendation

Add an unconditional refund of any excess deposit outside the `if attestation_storage_must_be_paid_by_caller` block, or add an `else` branch that refunds the full attached deposit when no storage payment is required:

```rust
} else {
    // No storage cost for this update; refund any attached deposit.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, restructure the deposit handling so the refund is always computed and issued regardless of which branch is taken, mirroring the pattern already used in `require_deposit`. [6](#0-5) 

---

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time → `NewlyInsertedParticipant` → deposit-handling block runs, excess refunded correctly.
2. `alice.near` calls `submit_participant_info` again (routine re-attestation) with `attached_deposit = 1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR) → `UpdatedExistingParticipant` → `attestation_storage_must_be_paid_by_caller = false` → deposit-handling block is skipped entirely → 1 NEAR is permanently retained by the contract with no refund.

The `UpdatedExistingParticipant` variant is confirmed reachable by the existing test: [7](#0-6)
