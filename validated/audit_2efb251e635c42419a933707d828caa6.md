Audit Report

## Title
Unvalidated Chainlink Price Data in `ChainlinkPriceOracle.getAssetPrice()` Enables Stale Price to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `priceFeed.latestRoundData()` and discards all validation fields, using only the raw `price` value without checking staleness, round completeness, or sign. This unvalidated price flows directly into `LRTDepositPool.getRsETHAmountToMint()`, where a stale inflated asset price against a correctly-set stored `rsETHPrice` causes depositors to receive excess rsETH, diluting the yield of all existing rsETH holders. The same codebase already implements the correct validation pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at lines 52–54 silently discards all `latestRoundData()` return values except `price`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol:52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No checks are performed for `price <= 0`, `updatedAt == 0` (incomplete round), or `answeredInRound < roundId` (stale round). In contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` at lines 30–32 correctly validates all three:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated price propagates through two paths:

**Path 1 — Direct minting:** `LRTDepositPool.getRsETHAmountToMint()` at line 520 uses `lrtOracle.getAssetPrice(asset)` live in the numerator:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
`lrtOracle.rsETHPrice()` is a stored state variable set at a prior `updateRSETHPrice()` call. If the asset price feed goes stale (returning an inflated last-known price) between two `rsETHPrice` updates, the numerator is inflated while the denominator remains at the previously-correct stored value, causing excess rsETH to be minted.

**Path 2 — Public price update:** `LRTOracle.updateRSETHPrice()` at line 87 is callable by any unprivileged user. It calls `_getTotalEthInProtocol()` at line 339, which iterates all supported assets and calls `getAssetPrice(asset)` for each, accumulating `totalETHInProtocol`. A stale inflated price overstates TVL, causing `rsETHPrice` to be set above its true value, which then affects all subsequent minting calculations.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` only catches deviations exceeding the configured threshold from `highestRsethPrice`; a moderately stale price (e.g., a few percent above true value) passes through undetected. The `updatePriceOracleForValidated()` sanity check only runs at oracle registration time, not at runtime.

## Impact Explanation
**Theft of unclaimed yield (High):** When a Chainlink LST/ETH feed goes stale with an inflated last-known price, any depositor calling `depositAsset()` receives more rsETH than their deposit warrants (numerator inflated, denominator at prior correct value). The excess rsETH dilutes the proportional claim of all existing rsETH holders on the protocol's ETH, constituting theft of unclaimed yield. This matches the allowed High impact class.

**Permanent freezing of funds (Critical, secondary):** If a feed returns `price = 0`, `getAssetPrice` returns 0, `getRsETHAmountToMint` returns 0, and a depositor whose `minRSETHAmountExpected` is 0 has their LST transferred in while receiving 0 rsETH — a permanent loss of deposited assets. This matches the Critical permanent freezing impact class.

## Likelihood Explanation
Chainlink feed staleness is a documented, recurring risk during L2 sequencer downtime, extreme network congestion, or node operator issues. The public `updateRSETHPrice()` entry point requires no privileged access. The exploit window exists whenever the feed's last reported price diverges from the true price and `rsETHPrice` has not been updated in the same block. The zero-price scenario is rare (uninitialized feed or circuit-breaker edge case) but the stale-price scenario has realistic likelihood. Likelihood is Low-to-Medium for staleness; the High impact warrants a High severity rating overall.

## Recommendation
Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > HEARTBEAT_INTERVAL) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept
**Stale price — theft of yield:**
1. At block N, `updateRSETHPrice()` is called; `rsETHPrice` is set to the correct value (e.g., 1.05e18) based on accurate stETH/ETH price of 0.9998e18.
2. At block N+100, the stETH/ETH Chainlink feed goes stale (sequencer downtime). True price has dropped to 0.9990e18, but the feed still returns 0.9998e18.
3. Attacker (or any user) calls `LRTOracle.updateRSETHPrice()` (public, no access control). `_getTotalEthInProtocol()` uses the stale 0.9998e18 price, overstating TVL. `rsETHPrice` is updated to an inflated value.
4. A depositor calls `LRTDepositPool.depositAsset(stETH, 1e18, 0, "")`. `getRsETHAmountToMint` computes `(1e18 * stalePriceHigh) / rsETHPrice`. If only the per-asset price is stale but `rsETHPrice` reflects a prior correct state, the depositor receives excess rsETH.
5. Existing rsETH holders' proportional claim on protocol ETH is diluted.

**Zero price — permanent fund loss:**
1. stETH/ETH feed returns `answer = 0`.
2. `getAssetPrice(stETH)` returns 0.
3. Depositor calls `depositAsset(stETH, 1e18, 0, "")` with `minRSETHAmountExpected = 0`.
4. `_beforeDeposit` computes `rsethAmountToMint = 0`; check `0 >= 0` passes.
5. `safeTransferFrom` transfers 1 stETH to the pool; `_mintRsETH(0)` mints nothing.
6. Depositor permanently loses 1 stETH with no rsETH received.

**Foundry fork test outline:**
```solidity
function testStalePrice_ExcessMint() public fork {
    // 1. Record current rsETHPrice
    uint256 correctRsETHPrice = lrtOracle.rsETHPrice();

    assertGt(rsethMinted, fairRseth);
}
