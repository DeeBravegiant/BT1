Audit Report

## Title
rsETH Price Inflation via Unrestricted ETH Donation Enables First-Depositor Fund Theft ŌĆö (File: `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
An attacker can donate ETH directly to `LRTDepositPool` via its unrestricted `receive()` function, inflating the TVL counted by `getETHDistributionData()`. Because `pricePercentageLimit` defaults to `0` and `updateRSETHPrice()` is public, the attacker can then push `rsETHPrice` to an arbitrarily large value. A subsequent depositor who passes `minRSETHAmountExpected = 0` will have their ETH accepted while receiving zero rsETH due to integer division rounding, permanently losing their deposit to the attacker.

## Finding Description

**Root cause 1 ŌĆö Unrestricted ETH donation inflates TVL**

`LRTDepositPool` exposes a bare `receive()` with no access control: [1](#0-0) 

`getETHDistributionData()` reads `address(this).balance` directly, so any donated ETH is immediately counted as protocol TVL: [2](#0-1) 

**Root cause 2 ŌĆö Public price update with no cap when `pricePercentageLimit == 0`**

`updateRSETHPrice()` is callable by any unprivileged address: [3](#0-2) 

`initialize()` never sets `pricePercentageLimit`, so it defaults to `0`: [4](#0-3) 

The guard is short-circuited when the limit is zero ŌĆö `pricePercentageLimit > 0` is `false`, so `isPriceIncreaseOffLimit` is always `false` and no revert occurs regardless of the price jump magnitude: [5](#0-4) 

`protocolFeeInBPS` also defaults to `0` in `LRTConfig` (never set in `initialize()`), so `protocolFeeInETH = 0`, and `_checkAndUpdateDailyFeeMintLimit(0)` does not revert ŌĆö the price update completes cleanly: [6](#0-5) 

**Root cause 3 ŌĆö No zero-mint guard in `_beforeDeposit`**

`getRsETHAmountToMint` uses integer division: [7](#0-6) 

`_beforeDeposit` only checks `rsethAmountToMint < minRSETHAmountExpected`; it never asserts `rsethAmountToMint > 0`: [8](#0-7) 

When `minRSETHAmountExpected == 0`, the condition `0 < 0` is `false`, so the deposit proceeds and `_mintRsETH(0)` is called ŌĆö the victim's ETH is absorbed into the pool while they receive nothing.

**Full exploit path:**

1. Attacker deposits 1 wei ETH ŌåÆ receives 1 wei rsETH; `rsETHPrice` = `1e18`.
2. Attacker sends X ETH directly to `LRTDepositPool` via `receive()` (no rsETH minted).
3. Attacker calls `lrtOracle.updateRSETHPrice()`. `totalETHInProtocol Ōēł X`, `rsethSupply = 1 wei`, so `newRsETHPrice = X * 1e18 / 1 = X * 1e18`. No revert because `pricePercentageLimit == 0`.
4. Victim calls `depositETH{value: Y}(0, "")` where `Y < X`. `rsethAmountToMint = Y * 1e18 / (X * 1e18) = Y / X ŌåÆ 0`. Check `0 < 0` is false ŌåÆ no revert. Victim's Y ETH is accepted; 0 rsETH minted.
5. Attacker redeems their 1 wei rsETH (the only rsETH in existence) and recovers the entire pool: `1 wei + X + Y`. Net profit: Y ETH (victim's full deposit). The donated X ETH is fully recovered.

## Impact Explanation

**Critical ŌĆö Direct theft of user funds.**

The victim's entire deposit is permanently transferred to the attacker. The attacker recovers their donated ETH in full and nets the victim's deposit. This matches the allowed impact class: *Direct theft of any user funds, whether at-rest or in-motion*. [9](#0-8) 

## Likelihood Explanation

**Medium.** Two conditions must hold simultaneously:

1. `pricePercentageLimit == 0` ŌĆö this is the **on-chain default** because `initialize()` never sets it. It remains `0` until an admin explicitly calls `setPricePercentageLimit()`.
2. Victim passes `minRSETHAmountExpected = 0` ŌĆö common for users interacting directly with the contract, scripted integrations, or frontends that omit slippage protection.

No privileged access is required. The attack is a classic frontrun: the attacker observes a pending `depositETH` transaction in the mempool, prepends the donation + price update, and the victim's transaction executes at the inflated price. The attack is repeatable against every new depositor until `pricePercentageLimit` is set by an admin.

## Recommendation

1. **Enforce a non-zero mint amount**: add `if (rsethAmountToMint == 0) revert ZeroRsETHMinted();` inside `_beforeDeposit()` immediately after computing `rsethAmountToMint`. [10](#0-9) 

2. **Set `pricePercentageLimit` during initialization** to a safe value (e.g., 1% = `1e16`) so that a single `updateRSETHPrice()` call cannot inflate the price by an arbitrary factor. [4](#0-3) 

3. **Restrict the `receive()` function** to only accept ETH from known, trusted callers (NodeDelegators, reward receivers, etc.) rather than from arbitrary addresses. [11](#0-10) 

## Proof of Concept

```solidity
// Preconditions: pricePercentageLimit == 0 (default), protocolFeeInBPS == 0 (default),
//                minAmountToDeposit == 0 (default), ETH deposit limit set to >= 20 ETH

// 1. Attacker deposits 1 wei ETH ŌåÆ receives 1 wei rsETH at price 1e18
lrtDepositPool.depositETH{value: 1}(0, "");

// 2. Attacker donates 10 ETH directly to the deposit pool (no rsETH minted)
(bool ok,) = address(lrtDepositPool).call{value: 10 ether}("");
require(ok);

// 3. Attacker triggers price update ŌĆö new price = (10e18 + 1) * 1e18 / 1 Ōēł 10e36
//    pricePercentageLimit == 0 ŌåÆ isPriceIncreaseOffLimit is false ŌåÆ no revert
lrtOracle.updateRSETHPrice();

// 4. Victim deposits 5 ETH with no slippage guard
//    rsethAmountToMint = 5e18 * 1e18 / 10e36 = 0 (integer division)
//    _beforeDeposit: 0 < 0 is false ŌåÆ no revert ŌåÆ victim gets 0 rsETH
lrtDepositPool.depositETH{value: 5 ether}(0, ""); // called by victim

// 5. Attacker redeems 1 wei rsETH through the withdrawal manager
//    Receives Ōēł 10 ETH (donation) + 5 ETH (victim) + 1 wei (own deposit)
//    Net profit: 5 ETH (victim's full deposit)
```

**Foundry test plan**: Deploy `LRTConfig`, `LRTOracle`, `LRTDepositPool`, and `RSETH` with default initialization (no `setPricePercentageLimit` call). Execute the five steps above with two distinct accounts. Assert that after step 4, `RSETH.balanceOf(victim) == 0` and `address(lrtDepositPool).balance == 15 ether + 1 wei`. Assert that after step 5, the attacker's recovered ETH equals `15 ether + 1 wei`.
