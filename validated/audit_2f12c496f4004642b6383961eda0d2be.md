Audit Report

## Title
Immutable Reverting Extension Permanently Freezes `removeLiquidity` With No Admin Escape Hatch — (`metric-core/contracts/ExtensionCalling.sol`, `metric-core/contracts/libraries/CallExtension.sol`)

## Summary

Extensions are stored in immutable slots at pool construction time with zero post-deployment removal or replacement path. `CallExtension.callExtension` propagates every extension revert verbatim to the caller with no isolation. Because `removeLiquidity` unconditionally calls `_beforeRemoveLiquidity` / `_afterRemoveLiquidity` and carries no `whenNotPaused` guard, a single reverting extension registered on the remove-liquidity hook permanently locks all LP positions in the pool with no admin recovery path.

## Finding Description

**Extension immutability.** `EXTENSION_1` through `EXTENSION_7` are declared `internal immutable` in `ExtensionCalling` and written once in the constructor. A full-repository search finds zero occurrences of `setExtension`, `updateExtension`, `removeExtension`, or `replaceExtension`. The only post-deployment admin levers are fee changes, pause/unpause, and a timelocked price-provider rotation — none touch extensions.

**Revert propagation without isolation.** `_callExtensionsInOrder` iterates the packed order word and calls each extension through `CallExtension.callExtension` with no try/catch:

```solidity
// ExtensionCalling.sol L75-86
CallExtension.callExtension(extension, data);   // no isolation
```

`callExtension` re-raises the extension revert verbatim:

```solidity
// CallExtension.sol L9-16
(bool success, bytes memory result) = extension.call(data);
if (!success) {
    if (result.length > 0) {
        assembly ("memory-safe") { revert(add(result, 32), mload(result)) }
    }
    revert ExtensionCallFailed();
}
```

**`removeLiquidity` carries no pause guard.** The function signature is:

```solidity
// MetricOmmPool.sol L199-202
function removeLiquidity(...) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
```

Only `swap` carries `whenNotPaused`. The interface NatSpec at `IMetricOmmPoolActions.sol` line 8 explicitly documents this as intentional ("unwind while swaps are off"), but this design means the pause mechanism provides zero relief when an extension on the remove-liquidity hook begins reverting.

**Realistic non-malicious trigger paths:**
1. An extension contains a latent arithmetic bug triggered by a specific bin index, share count, or zero-balance bin state reached through normal LP operations.
2. An extension reads from an external price feed or registry that becomes unavailable or starts reverting.
3. An extension is an upgradeable proxy; its owner (a party distinct from the pool admin) upgrades the implementation.

In all three cases the pool admin has no power to fix the situation: no `removeExtension`, no bypass flag, and `pausePool` does not unblock `removeLiquidity`.

## Impact Explanation

If any extension registered for `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` begins reverting, every call to `removeLiquidity` reverts. All LP positions in the pool are permanently unwithdrawable — users cannot recover token0 or token1 principal. This is a direct, permanent loss of user principal with no on-chain recovery path, satisfying "broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows."

## Likelihood Explanation

Extensions are external contracts with no enforced immutability requirement. Any extension that is an upgradeable proxy, reads from a third-party feed, or contains a latent state-dependent bug can trigger this condition without any privileged action by the pool admin or factory owner. The condition is permanent once triggered and affects all LPs in the pool simultaneously.

## Recommendation

1. **Wrap extension calls in try/catch for the remove-liquidity path.** Change `_callExtensionsInOrder` (or a dedicated variant used by `_beforeRemoveLiquidity` / `_afterRemoveLiquidity`) to catch extension reverts, emit an `ExtensionCallFailed(extensionIndex)` event, and continue. This ensures LP funds can always be recovered even if an extension breaks.
2. **Alternatively, add a factory-level force-remove path.** Introduce a timelocked factory function that allows the pool admin to zero out a specific extension slot and remove it from all order words.
3. **At minimum, ensure `removeLiquidity` is never blocked by extensions.** Either remove extension hooks from the withdrawal path entirely, or wrap only the withdrawal-path extension calls in try/catch.

## Proof of Concept

```solidity
// 1. Pool is deployed with an extension registered for BEFORE_REMOVE_LIQUIDITY_ORDER.
//    Extension initially behaves correctly; LPs add liquidity normally.

// 2. Extension hits a latent bug triggered by a specific bin state (e.g., zero-share bin),
//    OR its external oracle dependency reverts, OR its owner upgrades it to always revert:
contract BrokenExtension is IMetricOmmExtensions {
    function beforeRemoveLiquidity(...) external pure returns (bytes4) {
        revert("bricked");
    }

//    _beforeRemoveLiquidity -> _callExtensionsInOrder -> CallExtension.callExtension

  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);
      if (extension == address(0)) revert PanicEmptyExtension();
      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
  }
```
