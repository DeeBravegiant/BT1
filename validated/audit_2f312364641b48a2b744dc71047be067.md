### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Causes Undercollateralization and Frozen Redemption — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records the caller-supplied `amount` in the cross-chain settlement event without verifying the actual token balance received by the contract. For fee-on-transfer ERC20 tokens, the bridge custodies `amount − transfer_fee` while the NEAR side mints the full `amount` of wrapped tokens, permanently breaking the 1:1 backing guarantee and eventually freezing redemptions for later redeemers.

---

### Finding Description

In `OmniBridge.initTransfer`, the branch that handles ordinary (non-bridge, non-custom-minter) ERC20 tokens performs:

```solidity
// OmniBridge.sol lines 406–411
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← caller-controlled, not verified post-transfer
    );
}
``` [1](#0-0) 

Immediately after, the unverified `amount` is forwarded to `initTransferExtension` and emitted in `InitTransfer`:

```solidity
// OmniBridge.sol lines 415–436
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message, extensionValue
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
``` [2](#0-1) 

The NEAR side processes this event and mints `amount` wrapped tokens. There is no balance-before / balance-after check to confirm the bridge actually received `amount`. For any fee-on-transfer token the bridge holds `amount − fee_taken` while the cross-chain record claims `amount`.

On the redemption path (`finTransfer`), the bridge attempts:

```solidity
// OmniBridge.sol lines 351–354
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount   // ← the original over-stated amount
);
``` [3](#0-2) 

Once cumulative shortfalls exhaust the bridge's token balance, `safeTransfer` reverts and the remaining redeemers' funds are permanently locked.

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token silently inflates the cross-chain supply relative to the EVM custody. The bridge becomes progressively undercollateralized. The last cohort of users to call `finTransfer` for that token will receive a revert, making their wrapped-token holdings permanently unclaimable. This matches:

- **High**: balance-accounting divergence that breaks backing guarantees.
- **Critical**: irreversible fund lock / permanently unclaimable user value in bridge flows.

---

### Likelihood Explanation

`initTransfer` is a public, permissionless function that accepts any ERC20 token address with no whitelist enforcement in the non-bridge-token branch. Fee-on-transfer tokens are a well-established, deployed token class (e.g., tokens with reflection mechanics or protocol fees). An attacker can deliberately bridge such a token, sell the over-minted NEAR-side wrapped tokens to other users, and exit before the shortfall is discovered. No privileged access, leaked keys, or external dependency compromise is required.

---

### Recommendation

Record the actual balance delta rather than trusting the caller-supplied `amount`:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// Use `received` (cast to uint128 with overflow check) in place of `amount`
// for all downstream accounting, event emission, and cross-chain messaging.
```

This ensures the cross-chain settlement amount exactly matches the tokens held in custody, preserving the 1:1 backing guarantee regardless of token transfer mechanics.

---

### Proof of Concept

1. Deploy or identify a fee-on-transfer ERC20 token `T` with a 1% transfer fee.
2. Call `OmniBridge.initTransfer(T, 1_000e18, 0, 0, nearRecipient, "")`.
   - Bridge receives `990e18` of `T` (1% fee taken).
   - `InitTransfer` event records `amount = 1_000e18`.
3. NEAR processes the event and mints `1_000e18` wrapped-`T` to `nearRecipient`.
4. Attacker sells the `1_000e18` wrapped-`T` on NEAR to victim users.
5. Victims call `finTransfer` to redeem `1_000e18` wrapped-`T` back to EVM.
6. Bridge holds only `990e18`; after the first `990e18` is paid out, subsequent `safeTransfer` calls revert.
7. Remaining victims' wrapped-`T` holdings are permanently unclaimable — their funds are locked.
