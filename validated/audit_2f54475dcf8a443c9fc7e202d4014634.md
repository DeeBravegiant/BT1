### Title
Unvalidated Peer-Supplied `transaction_hash` in P2P Sync Corrupts Transaction Hash Index and RPC Lookups — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `FullTransaction` messages from unauthenticated p2p peers and pushes the peer-supplied `transaction_hash` directly into `block_body.transaction_hashes` without any validation. The resulting `BlockBody` is committed to storage verbatim, permanently poisoning the `transaction_hash_to_idx` index and the per-transaction metadata. Every downstream RPC that resolves a transaction by hash — `starknet_getTransactionByHash`, `starknet_getTransactionReceipt`, `starknet_getTransactionStatus`, `starknet_getBlockWithTxs` — reads from this corrupted index and returns authoritative-looking wrong values.

---

### Finding Description

In `crates/apollo_p2p_sync/src/client/transaction.rs`, `parse_data_for_block` iterates until `current_transaction_len < target_transaction_len`, destructures each `FullTransaction` from the peer, and unconditionally appends the peer-controlled `transaction_hash` field:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);   // line 89
``` [1](#0-0) 

`write_to_storage` then calls `append_body`, which calls `write_transactions`. Inside `write_transactions`, the peer-supplied hash is used as the key in the `transaction_hash_to_idx` table and embedded in `TransactionMetadata`:

```rust
transaction_hash_to_idx_table.insert(txn, tx_hash, &transaction_index)?;   // line 622
transaction_metadata_table.append(
    txn, &transaction_index,
    &TransactionMetadata { tx_location, tx_output_location, tx_hash: *tx_hash },  // line 626
)?;
``` [2](#0-1) 

There is no cross-check against the `transaction_commitment` already stored in the block header (which was synced independently via the header protocol and contains the Merkle commitment over the correct hashes).

---

### Impact Explanation

Every RPC that resolves a transaction by hash reads from the poisoned index:

- `starknet_getTransactionByHash` calls `get_transaction_idx_by_hash` on the injected hash — it finds the entry and returns the transaction body paired with the **fake** hash. The real hash returns `TRANSACTION_HASH_NOT_FOUND`. [3](#0-2) 

- `starknet_getTransactionReceipt` and `starknet_getTransactionStatus` follow the same `get_transaction_idx_by_hash` path. [4](#0-3) 

- `starknet_getBlockWithTxs` zips stored transactions with stored hashes — it returns the injected fake hash for every transaction in the block. [5](#0-4) 

The corruption is permanent: once `append_body` commits, the fake hash is the canonical key in the LMDB `transaction_hash_to_idx` table. No subsequent re-sync of the same block number is possible without a full revert.

**Partial inaccuracy in the original claim:** The `transaction_commitment` field in the block header is written by the *header* sync protocol, not derived from the body transaction hashes at body-write time. A peer that controls only the transaction stream cannot alter the already-stored header commitment. The actual corrupted values are the `transaction_hash_to_idx` index entries and the `tx_hash` fields in `TransactionMetadata`.

---

### Likelihood Explanation

The attack requires only that the victim node connects to a single malicious peer during p2p sync. The peer needs no credentials, keys, or validator role — it only needs to respond to a `TransactionQuery` with `FullTransaction` messages carrying crafted `transaction_hash` fields. The `BadPeerError` path is only triggered for count mismatches, not for hash content mismatches, so the malicious peer is not disconnected. [6](#0-5) 

---

### Recommendation

After receiving each `FullTransaction`, recompute the hash from the transaction body and compare it to the peer-supplied value. Reject the peer (return `BadPeerError`) if they differ:

```rust
let computed_hash = transaction.calculate_hash(chain_id);
if computed_hash != transaction_hash {
    return Err(ParseDataError::BadPeer(BadPeerError::InvalidTransactionHash { ... }));
}
block_body.transaction_hashes.push(computed_hash);
```

Alternatively, ignore the peer-supplied hash entirely and always use the locally computed value.

---

### Proof of Concept

1. Stand up a node that is syncing via p2p.
2. Intercept or implement a peer that responds to `TransactionQuery` for block N with `FullTransaction { transaction: T, transaction_output: O, transaction_hash: FAKE_HASH }` where `FAKE_HASH ≠ hash(T)`.
3. After the node commits block N, call `starknet_getTransactionByHash(hash(T))` → `TRANSACTION_HASH_NOT_FOUND`.
4. Call `starknet_getTransactionByHash(FAKE_HASH)` → returns transaction `T` with hash `FAKE_HASH` (wrong pairing).
5. Call `starknet_getBlockWithTxs(N)` → all transaction hashes in the response are the injected fake values.
