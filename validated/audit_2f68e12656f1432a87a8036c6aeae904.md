### Title
Caller-Agnostic Request Key in `verify_foreign_transaction` Enables Queue Saturation DoS Against Legitimate Bridge Users - (File: `crates/contract/src/lib.rs`)

### Summary

The `verify_foreign_transaction` public method uses a **caller-agnostic** request key when inserting into the pending-request fan-out queue. Any unprivileged account can saturate the 128-slot queue for a specific foreign transaction, causing all subsequent legitimate `verify_foreign_transaction` calls for that same transaction to fail with `PendingRequestQueueFull` — permanently blocking bridge verification for that transaction until all attacker-controlled yields time out.

### Finding Description

The `sign()` method derives a per-caller key by hashing `(predecessor_id, path)` into a `Tweak`, so each user's pending queue is isolated: [1](#0-0) 

```rust
impl SignatureRequest {
    pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
        let tweak = crate::kdf::derive_tweak(predecessor_id, path);
        SignatureRequest { domain_id: domain, tweak, payload }
    }
}
```

An attacker cannot fill another user's `sign` queue because the tweak differs per caller.

`verify_foreign_transaction`, however, converts the user-supplied args into a `VerifyForeignTransactionRequest` that contains **no caller identity**: [2](#0-1) 

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

The resulting struct — `(domain_id, payload_version, request)` — is the map key used in `pending_verify_foreign_tx_requests`. The contract's own test explicitly documents this property: [3](#0-2) 

> "both yields are queued under the single (caller-agnostic) request key"

The fan-out queue is hard-capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`: [4](#0-3) 

Once the queue reaches 128 entries, `push_pending_yield` panics with `PendingRequestQueueFull`, rejecting all further submissions for that request key.

The attack path:
1. Attacker observes a legitimate `verify_foreign_transaction` call in the mempool (or simply knows the target `tx_id`).
2. Attacker submits 128 identical `verify_foreign_transaction` calls (same `domain_id`, `payload_version`, `request`) from any account(s), each with the minimum 1 yoctonear deposit.
3. The queue for that specific foreign transaction is now full.
4. The legitimate user's call (and any subsequent legitimate call for the same transaction) is rejected with `PendingRequestQueueFull`.
5. The attacker's 128 yields eventually time out, each calling `pop_oldest_pending_yield` to drain the queue — but the attacker can immediately re-saturate it. [5](#0-4) 

### Impact Explanation

This breaks the **request lifecycle** for `verify_foreign_transaction`. In the Omnibridge inbound flow, a user deposits funds on a foreign chain (Bitcoin, Ethereum, etc.) and must call `verify_foreign_transaction` to obtain the MPC signature needed to claim those funds on NEAR. An attacker can block verification of any specific foreign transaction indefinitely by continuously re-saturating the 128-slot queue. The user's bridge deposit is frozen until the attacker stops — there is no on-chain mechanism to evict attacker-controlled yields before their natural timeout.

This maps to the allowed Medium impact: **request-lifecycle manipulation that breaks production safety/accounting invariants** — specifically, the invariant that a valid foreign transaction can always be submitted for verification by its rightful claimant.

### Likelihood Explanation

- **Attacker cost**: 128 × 1 yoctonear ≈ zero. No privileged access required.
- **Attacker knowledge**: Only the target `(domain_id, payload_version, ForeignChainRpcRequest)` is needed, which is fully observable on-chain once the victim submits their first call.
- **Sustainability**: The attacker can re-saturate the queue after each timeout cycle with negligible cost.
- **Contrast with `sign()`**: The same attack is impossible against `sign()` because the key includes the caller's tweak; an attacker cannot forge another user's tweak.

### Recommendation

Include the `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, analogous to how `sign()` derives a per-caller tweak. This ensures each caller's queue slot is isolated and an attacker cannot saturate another user's queue entry.

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    pub predecessor_id: AccountId,  // add caller identity
}
```

Alternatively, derive a tweak from `(predecessor_id, tx_id)` as is done for `sign()`, so the map key is caller-scoped.

### Proof of Concept

```
// Attacker script (pseudocode):
let target_request = VerifyForeignTransactionRequestArgs {
    domain_id: <victim's domain_id>,
    payload_version: V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: <victim's tx_id>,
        confirmations: 1,
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};

// Submit 128 calls from attacker account(s), each with 1 yoctonear deposit
for _ in 0..128 {
    contract.verify_foreign_transaction(target_request.clone());
}

// Now victim calls:
contract.verify_foreign_transaction(target_request);
// => panics: "Pending-request queue is full for this request key (limit: 128)"
```

The victim's bridge deposit on Bitcoin is now unclaimable on NEAR until all 128 attacker yields time out, at which point the attacker can immediately repeat the attack. [6](#0-5) [7](#0-6)

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```
