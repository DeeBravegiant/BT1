### Title
Permissionless `verify_foreign_transaction` with Caller-Controlled Finality Parameters Enables Forged Foreign-Chain Verification and Double-Spend - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` function is callable by any account with a 1 yoctoNEAR deposit. The caller fully controls the finality/confirmation parameters (`confirmations` for Bitcoin, `finality` for EVM chains) with no minimum threshold enforced by the contract. A malicious caller can set `confirmations: 1` or `EvmFinality::Latest` to obtain a valid MPC-signed attestation for a weakly-confirmed transaction that later gets reorganized, enabling a double-spend on the NEAR side.

---

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` accepts a `VerifyForeignTransactionRequestArgs` from any caller with a 1 yoctoNEAR deposit. The contract performs no validation on the finality/confirmation parameters embedded in the `ForeignChainRpcRequest`:

```rust
// crates/contract/src/lib.rs:519-557
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(
        request.domain_id,
        DomainPurpose::ForeignTx,
        Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
        MINIMUM_SIGN_REQUEST_DEPOSIT,
    );
    // Only checks: domain exists, chain is supported, gas attached
    // No minimum confirmations or finality level enforced
    ...
}
```

The `BitcoinRpcRequest` struct exposes `confirmations: BlockConfirmations` with ABI-declared `minimum: 0.0`, meaning a caller can submit `confirmations: 1` (or even `0`). For EVM chains, the caller can specify `EvmFinality::Latest` — the weakest finality level.

The node-side inspector in `crates/foreign-chain-inspector/src/bitcoin/inspector.rs` uses the caller-supplied value directly as the threshold:

```rust
// crates/foreign-chain-inspector/src/bitcoin/inspector.rs:50-58
let transaction_block_confirmation = rpc_response.confirmations.into();
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

And in `crates/node/src/providers/verify_foreign_tx/sign.rs`, the caller-specified value is passed directly to the inspector:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs:137-149
let block_confirmations = request.confirmations.0.into();
let extracted_values = inspector
    .extract(transaction_id, block_confirmations, extractors)
    ...
```

The signed payload (`ForeignTxSignPayloadV1`) includes the full `request` struct (including `confirmations`/`finality`), but the contract never validates these fields against any minimum before accepting the request and triggering MPC signing. The analog to the Balancer `reinvestReward` vulnerability is exact: just as any caller could specify a DEX and slippage that benefited themselves, here any caller specifies the finality threshold that governs whether the MPC network will attest to a transaction — with no protocol-level floor.

---

### Impact Explanation

A malicious caller can:

1. Submit a Bitcoin transaction to a bridge that uses `verify_foreign_transaction` for inbound attestation.
2. Once the transaction has 1 confirmation, call `verify_foreign_transaction` with `confirmations: 1`.
3. MPC nodes verify the threshold is met and produce a valid threshold signature over `ForeignTxSignPayloadV1{request, extracted_values}`.
4. The attacker presents this signature to the NEAR bridge contract to claim the inbound transfer.
5. The Bitcoin transaction is subsequently reorganized out of the canonical chain (1-confirmation reorgs are not rare on Bitcoin).
6. The attacker has received NEAR-side funds while the Bitcoin-side transaction no longer exists — a double-spend.

For EVM chains, the same attack applies with `EvmFinality::Latest`, which the `EvmInspector::verify_finality_level` function accepts by querying the `latest` tag head, a block that can be reorganized.

This matches the allowed High impact: **forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

---

### Likelihood Explanation

- `verify_foreign_transaction` is fully permissionless; the only barrier is 1 yoctoNEAR.
- Bitcoin 1-confirmation reorgs occur naturally (estimated 0.5–2% of blocks historically); an attacker with mining resources can induce them deliberately.
- EVM `Latest`-finality reorgs are common on chains with fast block times (Polygon, BNB, Arbitrum).
- The localnet example args file (`docs/localnet/args/verify_foreign_tx_bitcoin.json`) itself uses `"confirmations": 1`, normalizing low-confirmation requests.
- No on-chain governance vote or operator action is required; any NEAR account can exploit this immediately.

---

### Recommendation

Enforce a minimum confirmation count and minimum finality level in the contract before enqueuing a `verify_foreign_transaction` request. Concretely:

- For `BitcoinRpcRequest`: reject requests where `confirmations < MIN_BITCOIN_CONFIRMATIONS` (e.g., 6), configurable via governance.
- For `EvmRpcRequest`: reject requests where `finality == EvmFinality::Latest`; require at least `Safe` or `Finalized`.
- For `SolanaRpcRequest`: reject `SolanaFinality::Processed`; require at least `Confirmed` or `Finalized`.

These minimums should be stored in `ConfigV1` and updatable via `update_config` (owner-only), so the network can adjust them without a contract upgrade.

---

### Proof of Concept

**Entry path** (Bitcoin double-spend):

1. Attacker controls a NEAR account `attacker.near` and a Bitcoin wallet.
2. Attacker sends 1 BTC to the bridge's Bitcoin deposit address.
3. After 1 Bitcoin block confirmation, attacker calls:
   ```json
   {
     "request": {
       "Bitcoin": {
         "tx_id": "<attacker_tx_id>",
         "confirmations": 1,
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": 3,
     "payload_version": 1
   }
   ```
   with 1 yoctoNEAR attached to `verify_foreign_transaction`.
4. MPC nodes query Bitcoin RPC, observe `confirmations >= 1`, pass `verify_block_is_canonical`, and produce a threshold ECDSA signature over `SHA-256(borsh(ForeignTxSignPayloadV1{request, [BlockHash(...)]}))`.
5. Attacker receives the `VerifyForeignTransactionResponse` containing the MPC signature.
6. Attacker submits the signature to the NEAR bridge contract, which credits 1 wBTC to `attacker.near`.
7. Attacker (or a colluding miner) reorganizes the Bitcoin chain, removing the 1-confirmation block.
8. The Bitcoin deposit never finalized; the attacker holds 1 wBTC on NEAR with no corresponding Bitcoin locked — double-spend complete.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
