Audit Report

## Title
Fee Truncation to Zero via Integer Division Allows Fee-Free Claims — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary
In `MerkleDistributor.claim`, the protocol fee is computed as `(claimableAmount * feeInBPS) / 10_000` using integer division. For any `claimableAmount` below `floor(10_000 / feeInBPS)`, the fee truncates to zero and the full amount is transferred to the user while the treasury receives nothing. Unlike `KernelMerkleDistributor`, which wraps the treasury transfer in `if (fee > 0)`, `MerkleDistributor` has no such guard and unconditionally calls `safeTransfer(protocolTreasury, 0)`.

## Finding Description
At `MerkleDistributor.sol` line 138–144:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;   // truncates
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);     // called even when fee == 0
``` [1](#0-0) 

The truncation threshold is `floor(10_000 / feeInBPS)`. With `MAX_FEE_IN_BPS = 1000`, the worst-case threshold is 10 wei for 18-decimal tokens, but grows to 10,000 wei for `feeInBPS = 1`. [2](#0-1) 

The `isClaimed` check only prevents re-claiming the same index:

```solidity
return userClaims[account].lastClaimedIndex >= index;
``` [3](#0-2) 

This means a user can claim at index N, then claim again at index N+1 when the owner calls `setMerkleRoot` (which increments `currentIndex`), each time with a small incremental `claimableAmount = cumulativeAmount_new - cumulativeAmount_prev`. If each increment is below the truncation threshold, every individual claim pays zero fee. Claiming in bulk would pay the correct fee; claiming greedily at each root update avoids it entirely. [4](#0-3) 

`KernelMerkleDistributor` explicitly guards against this with `if (fee > 0)` before the treasury transfer, confirming the zero-fee case is a known concern that was addressed in one contract but not the other: [5](#0-4) 

## Impact Explanation
Every claim where `claimableAmount < floor(10_000 / feeInBPS)` transfers the full amount to the user and zero to the treasury. Over many such claims the protocol collects no fees on that portion of the distribution. This is a direct, quantifiable loss of fee yield that should have accrued to the protocol treasury — matching **High: Theft of unclaimed yield**.

## Likelihood Explanation
The condition is entirely user-controlled: any claimant can call `claim` at every `setMerkleRoot` update rather than waiting to accumulate. No special privileges, no Merkle forgery, and no victim mistakes are required. The only precondition is that the per-period Merkle increment for the user is below the truncation threshold, which is common for small-balance holders or tokens with low decimal counts.

## Recommendation
Add a `if (fee > 0)` guard before the treasury transfer (consistent with `KernelMerkleDistributor`), and optionally enforce a minimum fee or use ceiling division:

```solidity
// Option A: ceiling division
uint256 fee = feeInBPS > 0
    ? (claimableAmount * feeInBPS + 9_999) / 10_000
    : 0;

// Option B: minimum of 1 when fee would otherwise truncate to zero
if (feeInBPS > 0 && claimableAmount > 0 && fee == 0) {
    fee = 1;
}

// Always guard the transfer
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
```solidity
// Foundry fuzz test — run with `forge test --fuzz-runs 10000`
function testFeeNeverZero(uint256 feeInBPS, uint256 claimableAmount) public pure {
    feeInBPS = bound(feeInBPS, 1, 1000);
    uint256 threshold = 10_000 / feeInBPS;
    // Only test amounts strictly below the truncation threshold
    if (threshold <= 1) return;
    claimableAmount = bound(claimableAmount, 1, threshold - 1);

    uint256 fee = (claimableAmount * feeInBPS) / 10_000;
    // FAILS: fee == 0 for all inputs in this range
    assert(fee > 0);
}
```

Concrete example: `feeInBPS = 1000` (10%), `claimableAmount = 9` → `fee = (9 * 1000) / 10_000 = 0`. User receives 9 tokens; treasury receives 0. Repeating this across every root update index accumulates unbounded fee-free yield extraction.
