### Title
`DelegateAction` Signature Hash Omits Chain Identifier, Enabling Cross-Chain Replay of Meta Transactions - (File: `core/primitives/src/action/delegate.rs`)

### Summary
The `DelegateAction::get_nep461_hash()` and `VersionedDelegateActionPayload::get_nep461_hash()` functions compute the signed payload hash without including any chain-specific identifier. A malicious relayer can capture a `SignedDelegateAction` produced on one NEAR network and replay it on another network where the same account and key exist with a compatible nonce, causing unauthorized execution of the user's actions on the wrong chain.

### Finding Description
`DelegateAction` (NEP-366 meta transactions) allows a user to sign an inner action that a third-party relayer submits on their behalf. The user's signature covers the hash produced by `get_nep461_hash()`:

```rust
// core/primitives/src/action/delegate.rs
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
```

`SignableMessage` prepends only a NEP-366 discriminant (`1 << 30 | 366`) to the serialized `DelegateAction` body. The `DelegateAction` body contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — but **no chain identifier** (no genesis hash, no chain_id, no network name).

The outer `SignedTransaction` that wraps the `DelegateAction` does include a `block_hash` field that prevents replay of the outer transaction across chains. However, the inner `DelegateAction` signature is independent of the outer transaction. Any relayer can extract the `SignedDelegateAction` and embed it in a freshly constructed outer transaction on a different NEAR network with a valid block hash for that network.

The same pattern applies to `VersionedDelegateActionPayload::get_nep461_hash()` (used by `DelegateActionV2` / `VersionedSignedDelegateAction`):

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateActionV2);
    let bytes = borsh::to_vec(&signable).expect("failed to serialize");
    hash(&bytes)
}
```

### Impact Explanation
If a user has the same account ID and the same access key registered on two NEAR networks (e.g., mainnet and testnet — a common configuration), and the nonce in the `DelegateAction` is greater than the current access key nonce on the target network, a malicious relayer can replay the signed action on the unintended network. The corrupted protocol values are:

- **Account balance**: a transfer action drains the user's funds on the target chain without consent.
- **Access keys**: an `AddKey`/`DeleteKey` action modifies the user's key set on the target chain.
- **Contract state**: a `FunctionCall` action executes arbitrary contract logic on the target chain.

The `max_block_height` field limits the validity window by block height, but block heights are not chain-specific — both networks advance independently and can have overlapping height ranges.

### Likelihood Explanation
- NEAR accounts frequently share the same account ID and key pair across mainnet and testnet.
- Nonce alignment is plausible: if a user has performed similar numbers of transactions on both networks, the nonce in a freshly signed `DelegateAction` may be valid on both.
- The attacker is any party acting as a relayer — an unprivileged role open to anyone. No validator or node-operator privilege is required.
- The attack requires only capturing a `SignedDelegateAction` (which the user hands to the relayer) and submitting it to a different network's RPC.

### Recommendation
Include a chain-specific identifier in the signed payload of `DelegateAction` and `DelegateActionV2`. The natural candidate is the genesis block hash (already available as `CryptoHash` in the protocol), which uniquely identifies each NEAR network. This should be added to `PartialEncodedStateWitnessInner`-style structs as a `chain_id` or `genesis_hash` field, and the NEP-461 `SignableMessage` scheme should be extended to mandate its inclusion for on-chain delegate action types.

### Proof of Concept

1. Alice signs a `DelegateAction` on mainnet to transfer 100 NEAR to Bob. She hands the `SignedDelegateAction` to a relayer.
2. The relayer, instead of submitting on mainnet, constructs a new outer `SignedTransaction` on testnet:
   - `block_hash` = a valid recent testnet block hash
   - `actions` = `[Action::Delegate(Box::new(alice_signed_delegate_action))]`
   - The outer transaction is signed by the relayer's testnet key.
3. The relayer submits this to a testnet RPC node via `broadcast_tx_commit`.
4. The testnet runtime calls `apply_delegate_action`, which calls `signed_delegate_action.verify()`:
   ```rust
   // core/primitives/src/action/delegate.rs:84-90
   pub fn verify(&self) -> bool {
       let delegate_action = &self.delegate_action;
       let hash = delegate_action.get_nep461_hash(); // no chain_id in hash
       let public_key = &delegate_action.public_key;
       self.signature.verify(hash.as_ref(), public_key)
   }
   ```
5. Because `get_nep461_hash()` produces the same digest on both networks (no chain identifier), the signature verifies successfully on testnet.
6. The nonce check in `validate_delegate_action_key` passes if Alice's testnet access key nonce is below the `DelegateAction`'s nonce.
7. The transfer executes on testnet, draining Alice's testnet balance without her consent.

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
