### Title
Presignature Irrevocably Consumed Before Foreign Chain Verification Enables Attacker to Deplete MPC Presignature Pool — (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

In `make_verify_foreign_tx_leader`, a presignature is permanently removed from the store **before** the foreign chain transaction is verified. Any unprivileged NEAR account can submit `verify_foreign_transaction` requests referencing non-existent or unconfirmed foreign-chain transaction IDs. Each such request causes the leader node to irrevocably waste one presignature on a request that will always fail, depleting the ForeignTx domain's presignature pool and stalling all legitimate bridge operations.

---

### Finding Description

`make_verify_foreign_tx_leader` in `crates/node/src/providers/verify_foreign_tx/sign.rs` executes in this order:

```
1. presignature_store.take_owned().await          // presignature consumed — IRREVERSIBLE
2. new_channel_for_task(...)                      // P2P channel opened
3. execute_foreign_chain_request(...)             // foreign chain RPC query — CAN FAIL
4. make_signature_leader_given_parameters(...)    // signing
``` [1](#0-0) 

Step 1 calls `presignature_store.take_owned().await`, which removes the presignature from the store and deletes it from RocksDB. There is no mechanism to return it on failure. If step 3 (`execute_foreign_chain_request`) returns an error — because the transaction ID does not exist on the foreign chain, has not reached the required confirmation depth, or the RPC call times out — the function propagates the error but the presignature is already gone. [2](#0-1) 

The on-chain `verify_foreign_transaction` entry point performs only these checks before enqueuing the request:

- Domain exists and has `ForeignTx` purpose
- Prepaid gas ≥ minimum
- Attached deposit ≥ 1 yoctoNEAR
- Requested chain is in the supported set [3](#0-2) 

It does **not** validate whether the referenced transaction actually exists or is finalized on the foreign chain. That validation is deferred entirely to the MPC nodes, which perform it only after consuming the presignature. [4](#0-3) 

Presignatures are expensive precomputed cryptographic assets. Each one requires consuming two triples (themselves the product of multi-party OT-based computation across all threshold participants), followed by a full multi-party presignature protocol. The generation loop runs continuously to maintain a configured buffer (`desired_presignatures_to_buffer`), but the buffer is finite and bounded. [5](#0-4) 

---

### Impact Explanation

An attacker submits a stream of `verify_foreign_transaction` requests with syntactically valid but non-existent foreign-chain transaction IDs (e.g., a 32-byte all-zero Bitcoin `tx_id`). Each request:

1. Passes all on-chain precondition checks (costs 1 yoctoNEAR + gas).
2. Is picked up by the leader node, which calls `presignature_store.take_owned()` — consuming one presignature permanently.
3. Fails at `execute_foreign_chain_request` when the RPC returns "transaction not found."
4. Returns an error; the presignature is not restored.

Once the ForeignTx domain's presignature pool is exhausted, `take_owned()` blocks indefinitely. All subsequent legitimate `verify_foreign_transaction` requests (e.g., Omnibridge inbound flows) stall at this point and eventually time out on the NEAR side. Users waiting for bridge funds to arrive on NEAR experience indefinite delays or permanent failure of their bridge transactions. This breaks the request-lifecycle safety invariant: an accepted request that passed all on-chain checks cannot be fulfilled. [6](#0-5) 

---

### Likelihood Explanation

The attack requires no special privilege. Any NEAR account with 1 yoctoNEAR and enough gas can call `verify_foreign_transaction`. The attacker only needs to know a supported foreign chain (publicly readable from `get_supported_foreign_chains()`) and submit requests with arbitrary fake transaction IDs. The cost per wasted presignature is negligible (1 yoctoNEAR + ~7 Tgas), while the cost to regenerate each presignature is a full multi-party cryptographic protocol across all threshold participants. The asymmetry makes sustained depletion practical. [7](#0-6) 

---

### Recommendation

Move `execute_foreign_chain_request` **before** `presignature_store.take_owned()`. The foreign chain RPC query is independent of the presignature and can be performed first. Only if the foreign chain verification succeeds should the presignature be consumed and the signing protocol initiated. This mirrors the recommendation from the reference report: revert early for checks that can fail before consuming expensive resources.

Concretely, in `make_verify_foreign_tx_leader`:

```rust
// 1. Verify foreign chain FIRST (no presignature consumed yet)
let response_payload = self
    .execute_foreign_chain_request(
        &foreign_tx_request.request,
        foreign_tx_request.payload_version,
    )
    .await?;  // returns early on failure — no presignature wasted

// 2. Only now consume the presignature
let (presignature_id, presignature) = domain_data.presignature_store.take_owned().await;
``` [2](#0-1)

### Proof of Concept

1. Attacker reads `get_supported_foreign_chains()` — learns Bitcoin is supported.
2. Attacker calls `verify_foreign_transaction` with:
   - `domain_id`: the ForeignTx domain ID
   - `request`: `BitcoinRpcRequest { tx_id: [0x00; 32], confirmations: 1, extractors: [BlockHash] }`
   - Deposit: 1 yoctoNEAR
3. Contract accepts the request (all preconditions pass; `[0x00; 32]` is a valid-length tx_id).
4. Leader node's `make_verify_foreign_tx_leader` runs:
   - Line 63: `presignature_store.take_owned().await` → presignature consumed and deleted from RocksDB.
   - Lines 73–78: `execute_foreign_chain_request` → Bitcoin RPC returns "No such mempool or blockchain transaction."
   - Function returns `Err(...)`. Presignature is permanently lost.
5. Attacker repeats until `take_owned()` blocks (pool exhausted).
6. All subsequent legitimate `verify_foreign_transaction` requests stall indefinitely at step 4, line 63. [1](#0-0) [8](#0-7)

## Signature consumption

Source: `crates/node/src/providers/ecdsa/sign.rs`

Unlike triples and presignatures, signatures are not pre-generated.
When a signature request arrives:

1. **Leader** calls `presignature_store.take_owned()` for the relevant
   domain, consuming one presignature.
2. Leader opens a network channel with the presignature's borrowers
   and broadcasts the presignature ID along with the signature request.
3. **Followers** call `presignature_store.take_unowned(id)` to retrieve
   their share, then run the protocol.
4. The leader does **not** wait for all followers to confirm success
   (`leader_waits_for_success` returns `false`).

```

**Explanation:** The asset count gauges are only updated inside the

`take_owned()` blocks indefinitely (waiting for a presignature that will

### Deposit requirement

Both `sign` and `request_app_private_key` require a **deposit of at least 1 yoctonear**. Any excess deposit is automatically refunded.

The deposit exists to prevent abuse by malicious frontends. On NEAR, a dApp frontend can hold a function-call access key that lets it submit transactions on behalf of a user without prompting for approval each time. By default, however, function-call access keys **cannot attach a deposit**. Requiring a deposit therefore guarantees that the call was authorised by the user's full-access key (or a function-call key with an explicit deposit allowance), preventing a compromised or malicious frontend from silently submitting signature requests without the user's knowledge.
```
