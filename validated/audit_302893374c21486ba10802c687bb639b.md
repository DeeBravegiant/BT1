Audit Report

## Title
Unbounded Nested Loop in `NodeDelegator.getAssetUnstaking()` on Public Deposit and Price-Update Paths Can Cause DoS - (File: contracts/NodeDelegator.sol)

## Summary
`NodeDelegator.getAssetUnstaking()` iterates over all EigenLayer-queued withdrawals and their strategies in a nested loop. This function is invoked on every call to `depositETH()`, `depositAsset()`, and the public `updateRSETHPrice()`, with the outer call in `LRTDepositPool` looping over every NDC. As the number of NDCs, queued withdrawals, and strategies per withdrawal grows through normal protocol operations, the cumulative gas cost can exceed the block gas limit, permanently reverting all deposits and price updates.

## Finding Description
`NodeDelegator.getAssetUnstaking()` at L405–427 fetches all queued withdrawals from EigenLayer's `DelegationManager` and iterates over them with a nested loop — one dimension for withdrawals, one for strategies per withdrawal. Neither loop has an on-chain cap independent of the live EigenLayer state.

This function is called inside `LRTDepositPool.getAssetDistributionData()` at L446–456, which itself loops over every address in `nodeDelegatorQueue`. `getAssetDistributionData()` feeds into `getTotalAssetDeposits()`, which is called by `_checkIfDepositAmountExceedesCurrentLimit()` inside `_beforeDeposit()` at L648–670, which is invoked unconditionally by both `depositETH()` (L87) and `depositAsset()` (L111).

Separately, `LRTOracle._getTotalEthInProtocol()` at L331–349 calls `getTotalAssetDeposits()` for **every supported asset** in a loop, and is itself called by the public `updateRSETHPrice()` at L87–89.

The effective iteration count scales as:

```
supportedAssets.length × nodeDelegatorQueue.length × queuedWithdrawals.length × strategies.length
```

All four dimensions grow through normal protocol operations: admins add NDCs (up to `maxNodeDelegatorLimit`), operators call `initiateUnstaking()` with multiple strategies, and the protocol supports multiple LST assets. No existing guard bounds the product of these dimensions against the block gas limit.

## Impact Explanation
When the product of the four loop dimensions is large enough to exhaust the block gas limit, every call to `depositETH()`, `depositAsset()`, and `updateRSETHPrice()` reverts with out-of-gas. This constitutes **unbounded gas consumption** (Medium) leading to **temporary freezing of funds** (Medium): users cannot deposit, and the rsETH price becomes stale, which also breaks the withdrawal path since `getExpectedAssetAmount()` depends on `rsETHPrice`. The state can only be resolved by completing queued withdrawals, which requires operator action and time.

## Likelihood Explanation
No malicious actor is required. The DoS state arises organically as the protocol scales: admins legitimately raise `maxNodeDelegatorLimit` and deploy NDCs; operators legitimately call `initiateUnstaking()` with multiple strategies across multiple NDCs concurrently. With 10 NDCs, 5 supported assets, 50 pending withdrawals per NDC, and 5 strategies per withdrawal, the inner iteration count reaches 12,500 — each involving external storage reads from EigenLayer — which is sufficient to exceed the block gas limit. The public `updateRSETHPrice()` can be called by anyone, making the DoS trivially triggerable once the state is reached.

## Recommendation
1. **Decouple unstaking accounting from the hot path**: Store `assetUnstaking` as a cached state variable updated lazily on `initiateUnstaking()` and `completeUnstaking()` rather than recomputing it via live EigenLayer queries on every deposit and price update.
2. **Cap strategies per withdrawal** in `initiateUnstaking()` to bound the inner loop.
3. **Cap `maxUncompletedWithdrawalCount`** at a value that, combined with `maxNodeDelegatorLimit` and `supportedAssets.length`, provably stays within safe gas bounds.
4. Consider separating the price-update path from the full distribution-data computation, or processing assets in batches across separate transactions.

## Proof of Concept
1. Admin raises `maxNodeDelegatorLimit` to 10 and deploys 10 NDCs.
2. Protocol supports 5 LST assets.
3. Operator calls `initiateUnstaking()` on each NDC with 5 strategies, repeated up to `maxUncompletedWithdrawalCount` times (e.g., 50 pending withdrawals per NDC).
4. Any user calls `depositETH()`. The call chain: `depositETH()` → `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()` → `getTotalAssetDeposits()` → `getAssetDistributionData()` → loop over 10 NDCs, each calling `getAssetUnstaking()` → nested loop over 50 withdrawals × 5 strategies = 250 EigenLayer storage reads per NDC × 10 NDCs = 2,500 reads for one asset. Multiplied by 5 assets in `_getTotalEthInProtocol()` path: 12,500 iterations with external storage reads. Transaction reverts with out-of-gas.
5. `updateRSETHPrice()` (public, no access control) similarly reverts, freezing the rsETH price.

**Foundry fork test plan**: Fork mainnet or a testnet with EigenLayer deployed. Deploy 10 NDCs, call `initiateUnstaking()` with 5 strategies on each NDC 50 times, then call `depositETH{value: 1 ether}()` and assert it reverts with out-of-gas. **File:** contracts/NodeDelegator.sol (L405-427)
```text
    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
```
