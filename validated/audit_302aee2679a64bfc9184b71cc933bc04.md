Audit Report

## Title
Stale Cross-Chain Oracle Rate Enables Structural Over-Minting of wrsETH, Causing Undercollateralization of the L2 Wrapper - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3.sol)

## Summary
The L2 pool contracts (`RSETHPoolV3` and `RSETHPoolV3ExternalBridge`) mint wrsETH using a cross-chain oracle rate sourced from `CrossChainRateReceiver`, which stores the last LayerZero-relayed L1 rsETH price with no on-chain staleness enforcement. Because the L1 rsETH price increases monotonically with staking rewards, any gap between LayerZero deliveries causes the L2 rate to lag the true L1 rate, resulting in more wrsETH being minted per ETH deposited than the ETH will produce in rsETH when bridged and converted on L1. This creates a structural, accumulating collateral deficit in the wrsETH wrapper that dilutes the yield of all existing wrsETH holders.

## Finding Description

**Step 1 — Minting uses the unchecked oracle rate.**

`RSETHPoolV3.viewSwapRsETHAmountAndFee()` and `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee()` both compute:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

where `rsETHToETHrate = getRate()` → `IOracle(rsETHOracle).getRate()`. Neither function performs any staleness check before using this rate to mint wrsETH. [1](#0-0) [2](#0-1) 

**Step 2 — The oracle simply returns the last received value.**

`CrossChainRateReceiver` (the concrete oracle behind `rsETHOracle`) stores `rate` and `lastUpdated` when a LayerZero message arrives, but `getRate()` returns `rate` unconditionally — `lastUpdated` is never consulted:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
``` [3](#0-2) 

Between LayerZero deliveries the stored rate is frozen at the last received value. `updateRate()` in `CrossChainRateProvider` is callable by anyone but requires paying LayerZero fees and is not enforced on-chain before minting. [4](#0-3) 

**Step 3 — The L1 rsETH price increases monotonically.**

`LRTOracle._updateRsETHPrice()` computes `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)`. As EigenLayer staking rewards accumulate, `totalETHInProtocol` grows while `rsethSupply` is constant between updates, so `rsETHPrice` only ever increases. [5](#0-4) 

**Step 4 — L1 vault converts ETH at the current (higher) L1 rate.**

`L1VaultV2.depositETHForL1VaultETH()` calls `lrtDepositPool.getRsETHAmountToMint()`, which uses the live `lrtOracle.rsETHPrice()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [6](#0-5) [7](#0-6) 

**Step 5 — The mismatch.**

| Quantity | Formula | Example (1 week staleness) |
|---|---|---|
| wrsETH minted on L2 | `100 ETH / 1.00e18` | **100.000 wrsETH** |
| rsETH received from L1 | `100 ETH / 1.05e18` | **95.238 rsETH** |
| Structural deficit | | **4.762 rsETH per 100 ETH** |

Every deposit during a staleness window creates a deficit. There is no autonomous mechanism to detect or correct this imbalance.

**Existing guards are insufficient.**

The `dailyMintLimit` modifier limits total wrsETH minted per day but does not check oracle freshness. The `whenNotPaused` modifier is a manual admin action. Neither addresses the rate-staleness root cause. [8](#0-7) 

## Impact Explanation

**High — Theft of unclaimed yield.**

The excess wrsETH minted on L2 represents staking yield that belongs to existing rsETH/wrsETH holders. Each stale-rate deposit dilutes the backing ratio of the wrsETH wrapper. When users later unwrap wrsETH to rsETH, the wrapper is short on rsETH — either some redemptions fail (temporary freeze) or all holders receive less than 1:1 (yield theft). The deficit is proportional to `(rsETHPrice_L1 - rsETHPrice_L2) / rsETHPrice_L2` multiplied by total ETH deposited during the staleness window. This is a concrete, accumulating loss to existing holders, not a hypothetical one.

## Likelihood Explanation

**Medium.**

The rsETH price grows at roughly the staking APR (~4–5% per year). A 24-hour oracle staleness produces ~0.012% over-minting per deposit; a 7-day staleness produces ~0.087%. LayerZero message delivery is not guaranteed to be instantaneous — network congestion, gas spikes, or infrequent `updateRate()` calls all cause staleness. No on-chain staleness guard exists in any pool contract or oracle interface, so this condition is reachable by any depositor without any privileged action. The entry path is fully unprivileged: any user calling `deposit()` on either pool during a staleness window triggers the over-minting.

## Recommendation

1. Add a `MAX_STALENESS` constant (e.g., 24 hours) and revert in `CrossChainRateReceiver.getRate()` if `block.timestamp - lastUpdated > MAX_STALENESS`.
2. Alternatively, add a staleness check directly in `viewSwapRsETHAmountAndFee()` in both pool contracts before using the rate.
3. Implement a circuit breaker that pauses deposits when the oracle rate has not been refreshed within the staleness window.

## Proof of Concept

```
Assumptions:
  L1 rsETHPrice (actual)  = 1.05e18  (after 1 week of staking rewards)
  L2 oracle rate (stale)  = 1.00e18  (last updated 1 week ago, lastUpdated not checked)
  User deposits 100 ETH, feeBps = 0

On L2 (RSETHPoolV3.deposit):
  rsETHAmount = 100e18 * 1e18 / 1.00e18 = 100.000 wrsETH  ← minted to user

BRIDGER bridges 100 ETH to L1 (L1VaultV2.depositETHForL1VaultETH):
  rsETHAmountToMint = 100e18 * 1e18 / 1.05e18 ≈ 95.238 rsETH  ← bridged back to wrapper

Deficit per cycle: 100.000 - 95.238 = 4.762 rsETH
At scale (10,000 ETH/week deposited during staleness): ~476 rsETH deficit per week

Foundry fork test plan:
1. Fork mainnet; deploy RSETHPoolV3 pointing to a mock CrossChainRateReceiver
   initialized with rate = 1.00e18 and lastUpdated = block.timestamp - 8 days.
2. Call deposit() with 100 ETH; assert wrsETH minted = 100e18.
3. Simulate L1VaultV2.depositETHForL1VaultETH() with live LRTOracle returning 1.05e18;
   assert rsETH minted ≈ 95.238e18.
4. Assert wrsETH.totalSupply() > rsETH backing in wrapper → undercollateralization confirmed.
```
