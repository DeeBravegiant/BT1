### Title
Malicious Harvester Unconditionally Redirects Farmer Block Rewards via Unenforced `farmer_reward_address_override` in `RespondSignatures` — (File: `chia/farmer/farmer_api.py`)

### Summary

The CHIP-22 third-party harvester protocol allows a harvester to redirect the farmer's block reward by setting `farmer_reward_address_override` in either `NewProofOfSpace` or `RespondSignatures`. The only guard — a fee-quality check in `notify_farmer_reward_taken_by_harvester_as_fee()` — is purely advisory (logs a warning, never blocks). Worse, the check is only triggered by `NewProofOfSpace.farmer_reward_address_override`, not by `RespondSignatures.farmer_reward_address_override`, which is the field the farmer actually uses when building `DeclareProofOfSpace`. A malicious harvester can therefore redirect 100% of the farmer's XCH block rewards to an arbitrary address with no cryptographic or protocol-level enforcement.

### Finding Description

**Entry point — `NewProofOfSpace` path (advisory-only check):**

In `FarmerAPI.new_proof_of_space()`, when `farmer_reward_address_override is not None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
```

Inside that function, the fee-quality check only emits log warnings on failure — it never raises, returns early, or prevents the override from being used:

```python
if fee_quality <= fee_threshold:
    self.log.info(...)
else:
    self.log.warning("Harvester illegitimately took a fee reward ...")
```

**Bypass path — `RespondSignatures` path (no check at all):**

In `_process_respond_signatures()`, the farmer unconditionally replaces its own `farmer_target` with whatever the harvester supplies in `RespondSignatures.farmer_reward_address_override`:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

There is no fee-quality check here. The attacker-controlled address is then passed directly into `DeclareProofOfSpace` and broadcast to the full node:

```python
return farmer_protocol.DeclareProofOfSpace(
    ...
    farmer_reward_address,   # ← attacker-controlled
    ...
)
```

**Complete bypass path:**

A malicious harvester:
1. Sends `NewProofOfSpace` with `farmer_reward_address_override=None` — the advisory fee-quality check is never triggered.
2. Receives `RequestSignatures` from the farmer.
3. Returns `RespondSignatures` with `farmer_reward_address_override=<attacker_puzzle_hash>`.
4. The farmer builds and broadcasts `DeclareProofOfSpace` with the attacker's puzzle hash as the reward address — no check, no log, no rejection.

Even if the harvester sets the override in `NewProofOfSpace`, the check is still only advisory: the warning is logged but the override is applied regardless.

### Impact Explanation

Every block won by the farmer while connected to a malicious harvester will pay the 1.75 XCH farmer reward (or 0.25 XCH post-halvening) to the attacker's address instead of the farmer's configured address. The farmer loses 100% of their block rewards for the duration of the attack. This is an unauthorized payout redirection of XCH, matching the High impact category.

### Likelihood Explanation

Any harvester that has completed the TLS handshake with a farmer can send `RespondSignatures`. Third-party harvesters (the explicit target of CHIP-22) are untrusted by design. A malicious harvester operator needs only to modify the `farmer_reward_address_override` field in their `RespondSignatures` response — a trivial protocol-level change requiring no cryptographic capability.

### Recommendation

The fee-quality check must be enforced, not merely logged. Specifically:

1. In `_process_respond_signatures()`, apply the same fee-quality check to `RespondSignatures.farmer_reward_address_override` before accepting it.
2. Reject (return `None`) rather than warn when the fee quality does not meet the declared threshold.
3. Optionally, cross-validate that `RespondSignatures.farmer_reward_address_override` matches the value declared in the corresponding `NewProofOfSpace` message, so the harvester cannot introduce a new override at the signature stage.

### Proof of Concept

A malicious harvester intercepts the `request_signatures` call and injects an arbitrary override in the response:

```python
# Malicious harvester: intercept request_signatures
async def malicious_request_signatures(request):
    # Sign normally
    result_msg = await HarvesterAPI.request_signatures(self, request)
    # Inject attacker-controlled reward address with NO fee quality declared
    response = RespondSignatures.from_bytes(result_msg.data)
    response = dataclasses.replace(
        response,
        farmer_reward_address_override=ATTACKER_PUZZLE_HASH,  # arbitrary address
        # fee_info is NOT set in NewProofOfSpace, so no advisory check fires
    )
    return make_msg(ProtocolMessageTypes.respond_signatures, response)
```

The farmer's `_process_respond_signatures()` at lines 916–919 accepts `ATTACKER_PUZZLE_HASH` unconditionally and broadcasts `DeclareProofOfSpace` with it as `farmer_reward_address`. The full node mints the block reward coin to `ATTACKER_PUZZLE_HASH`.
