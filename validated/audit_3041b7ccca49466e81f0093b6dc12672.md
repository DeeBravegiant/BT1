Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`KernelDepositPool` uses a Synthetix-style continuous reward model where `updateReward` unconditionally advances `updatedAt` to `lastTimeRewardApplicable()`, even when `totalKernelStaked` is zero. If all stakers exit during an active reward window, the rewards emitted during the zero-staked interval are silently discarded and permanently locked in the contract, with no on-chain mechanism to recover or redistribute them.

## Finding Description

The root cause is the unconditional `updatedAt` advancement in the `updateReward` modifier:

```solidity
// L232-241
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable(); // always advances, even when totalKernelStaked == 0
    ...
}
```

`rewardPerToken()` short-circuits when `totalKernelStaked == 0`, returning the stale `rewardPerTokenStored`:

```solidity
// L408-413
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    ...
}
```

`initiateWithdrawal()` immediately decrements `totalKernelStaked` before the withdrawal delay:

```solidity
// L325-326
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

**Exploit flow:**
1. Active reward period is running (`rewardRate > 0`, `finishAt` in the future). Alice is the sole staker.
2. Alice calls `initiateWithdrawal(all)`. `updateReward` correctly snapshots her earned rewards and sets `updatedAt = now`. `totalKernelStaked` drops to 0.
3. Time passes. `rewardRate` continues emitting rewards, but `totalKernelStaked == 0` so `rewardPerToken()` freezes.
4. Bob calls `stake(amount)`. `updateReward` fires **before** `totalKernelStaked` is incremented: `rewardPerToken()` returns the stale stored value; `updatedAt` is advanced to the present, skipping the entire idle interval. Then `totalKernelStaked += amount`.
5. All rewards emitted during the zero-staked interval are permanently unclaimable.

The only existing guard is in `notifyRewardAmount()` (L570: `if (totalKernelStaked == 0) revert NoStakedTokens()`), which prevents starting a new period with zero stakers but does nothing to prevent all stakers from exiting after a period has begun. The contract's own NatSpec (L18-22) explicitly acknowledges this behavior and relies solely on an operational promise ("ensuring there are always some tokens staked"), with no on-chain enforcement.

## Impact Explanation

Rewards emitted during the zero-staked interval are permanently locked in `KernelDepositPool`. No staker — present or future — can ever claim them. The reward token balance in the contract becomes irrecoverably inflated relative to what can actually be distributed. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield.**

## Likelihood Explanation

Any unprivileged staker can trigger this by calling `initiateWithdrawal()`. No special role, governance action, or coordination is required. The last remaining staker may exit for entirely legitimate reasons (market conditions, better yield elsewhere) or deliberately. The `withdrawalDelay` does not prevent the accounting loss — `totalKernelStaked` is reduced immediately at `initiateWithdrawal()` time, not at `claimWithdrawal()` time. The scenario is realistic in a low-participation pool or at the tail end of a reward period.

## Recommendation

Do not advance `updatedAt` when `totalKernelStaked` is zero. This preserves the unallocated reward interval so it is correctly distributed once staking resumes:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This is the standard fix applied in audited Synthetix forks and ensures rewards are never silently discarded.

## Proof of Concept

1. Deploy `KernelDepositPool`. Admin calls `setRewardsDuration(100 days)`.
2. Alice calls `stake(1000e18)`. Admin calls `notifyRewardAmount(1000e18)`. `rewardRate = 10e18/day`.
3. Warp 10 days. Alice calls `initiateWithdrawal(1000e18)`. `updateReward` snapshots 100e18 earned; `updatedAt = T+10d`; `totalKernelStaked = 0`.
4. Warp 50 more days (T+60d). No stakers. `rewardRate * 50d = 500e18` emitted but unallocated.
5. Bob calls `stake(1000e18)`. `updateReward` fires: `totalKernelStaked == 0` → `rewardPerToken()` returns stale value; `updatedAt` advances to T+60d. Then `totalKernelStaked = 1000e18`.
6. Assert: `earned(Bob) == 0` at T+60d. Bob only earns from T+60d onward. The 500e18 for days 10–60 is permanently locked.
7. Confirm: `rewardsToken.balanceOf(address(pool))` exceeds total claimable rewards by 500e18.
