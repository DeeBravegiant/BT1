### Title
Single Byzantine Participant Can Forge CKD Output for `AppPublicKey` Requests Due to Missing Response Verification - (File: crates/contract/src/lib.rs)

### Summary

The `respond_ckd` function in the MPC contract enforces cryptographic verification of the CKD response only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`, while the `AppPublicKey` variant accepts any arbitrary response without verification. A single attested-but-Byzantine MPC participant (strictly below the signing threshold) can call `respond_ckd` with a forged `CKDResponse` for any pending `AppPublicKey` request and have it delivered to the user, bypassing the threshold requirement entirely.

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 675–682), the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, the contract runs `ckd_output_check`, a BLS12-381 pairing check that verifies `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`. This ensures the response is a valid output of the threshold CKD protocol.

For `AppPublicKey`, the arm is a no-op. Any `CKDResponse` — with completely arbitrary `big_y` and `big_c` — passes through and is immediately delivered to all waiting callers via `resolve_yields_for`.

The `AppPublicKey` variant is the standard, non-publicly-verifiable path. It is the default path used in the e2e test `ckd_response__passes_cryptographic_verification` and in the sandbox test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists`, where the test explicitly supplies `[1u8; 48]` and `[2u8; 48]` as `big_y`/`big_c` — demonstrating that the contract accepts any byte values without complaint.

The only gate before the match is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be a current participant with a valid TEE attestation. It does **not** require threshold agreement among participants. A single attested participant can call `respond_ckd` and resolve all pending yields for a given request.

This is the direct analog of the MEME404 bug: two code paths that are supposed to enforce the same invariant (cryptographic correctness of the protocol output) do so inconsistently — one path enforces it, the other silently skips it.

### Impact Explanation

**Critical. Bypass of threshold-signature requirements / unauthorized confidential key derivation output without the required participant authorization.**

For the `AppPublicKey` variant, the user provides `pk1 = g1 · app_sk` and expects to receive `(big_y, big_c)` such that `big_c − app_sk · big_y = msk · H(app_id)` — the MPC-derived confidential key.

A malicious node can instead submit:
- `big_y = g1 · y'` for an attacker-chosen scalar `y'`
- `big_c = msk_fake · H(app_id) + pk1 · y'` for an attacker-known scalar `msk_fake`

The user then unmasks: `big_c − app_sk · big_y = msk_fake · H(app_id)`, a key derived from `msk_fake`. The attacker knows `msk_fake` and therefore knows the user's derived confidential key. Any assets the user places under that key (e.g., cross-chain signing keys, encrypted secrets) are fully compromised. The threshold property — requiring t-of-n participants to cooperate — is completely bypassed for this request type.

### Likelihood Explanation

Any single attested MPC participant can exploit this. Attestation is a TEE check, not a threshold check. The attacker does not need to collude with other participants, does not need to know the real MPC master secret, and does not need any privileged access beyond being an active participant. The `AppPublicKey` variant is the primary (non-PV) CKD path and is the one most likely to be used by production applications.

### Recommendation

Apply the same cryptographic output verification to `AppPublicKey` requests as is applied to `AppPublicKeyPV` requests. Since `AppPublicKey` provides only `pk1` (a G1 point) and not `pk2` (the G2 component needed for the pairing check), the protocol should either:

1. Require callers to always use `AppPublicKeyPV` so the contract can verify the output on-chain, or
2. Require threshold-many participants to submit matching responses before resolving the yield (a commit-and-reveal or quorum-based approach), so a single Byzantine node cannot unilaterally deliver a forged result.

The root fix is to ensure that the invariant "the CKD response is cryptographically bound to the MPC network's key and the user's request" is enforced on **both** code paths, not just one.

### Proof of Concept

```
1. Alice calls request_app_private_key with AppPublicKey(pk1) where pk1 = g1 * app_sk.
   The contract stores the pending CKD request and yields.

2. Mallory is an attested MPC participant (below the signing threshold).
   Mallory picks arbitrary scalars y' and msk_fake.
   Mallory constructs:
     forged_big_y = g1 * y'
     forged_big_c = msk_fake * H(app_id) + pk1 * y'

3. Mallory calls respond_ckd(ckd_request, CKDResponse { big_y: forged_big_y, big_c: forged_big_c }).
   The contract reaches the AppPublicKey arm → no-op → resolve_yields_for delivers the forged response.

4. Alice's callback fires with (forged_big_y, forged_big_c).
   Alice unmasks: forged_big_c - app_sk * forged_big_y
                = msk_fake * H(app_id) + pk1 * y' - app_sk * g1 * y'
                = msk_fake * H(app_id)   (since pk1 = g1 * app_sk)
   Alice's derived key is msk_fake * H(app_id), which Mallory knows.

5. Any assets Alice places under this key are under Mallory's control.
   The real threshold CKD protocol was never run; no other participant was involved.
```

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
