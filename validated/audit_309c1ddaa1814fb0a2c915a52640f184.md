### Title
`starknet_traceBlockTransactions` Pending Path Misclassifies Intra-Block Class Replacements as Fresh Deployments â€” (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

An unprivileged RPC caller invoking `starknet_traceBlockTransactions(Tag::Pending)` receives an authoritative-looking but wrong `state_diff` for any transaction that changes the class hash of a contract that was first deployed by an earlier transaction in the same pending block. The contract address is reported in `deployed_contracts` instead of `replaced_classes`.

---

### Finding Description

**Step 1 â€” Empty pending state diff (intentional, but creates the precondition)**

In `trace_block_transactions`, when `block_id` is `Tag::Pending`, `ExecutionPendingData` is constructed with all state diff fields set to `Default::default()`: [1](#0-0) 

The comment at line 1324 confirms this is intentional: the execution engine re-executes all pending transactions sequentially from the start of the pending block.

**Step 2 â€” Sequential execution correctly accumulates intra-block state**

In `execute_transactions`, each transaction runs against a `transactional_state` wrapping a shared `cached_state`. After each transaction, `transactional_state.commit()` propagates the changes back into `cached_state`: [2](#0-1) 

This means tx[1] correctly sees the contract deployed by tx[0] during execution. The `induced_state_diff` for tx[1] is computed only from `transactional_state` (the per-tx delta), so if tx[1] calls `replace_class` on a contract deployed by tx[0], the address appears in `address_to_class_hash` and thus in `deployed_contracts` of the returned `ThinStateDiff`: [3](#0-2) 

**Step 3 â€” `convert_thin_state_diff` checks only the accepted block state**

After execution, `convert_thin_state_diff` is called for each transaction's `induced_state_diff` to split `deployed_contracts` into fresh deployments vs. replaced classes: [4](#0-3) 

For `Tag::Pending`, `prev_block_number = Some(block_number)` (the latest accepted block). `is_deployed` then queries only the accepted block state: [5](#0-4) 

`maybe_get_class_hash_at` is called with a `BlockId::HashOrNumber` pointing to the accepted block â€” it has no knowledge of intra-block state changes from earlier pending transactions: [6](#0-5) 

**The bug:** If contract A was deployed by tx[0] in the pending block and tx[1] calls `replace_class` on it, `is_deployed(block_number, A)` returns `false` (A is not in the accepted state). Therefore `convert_thin_state_diff` leaves A in `deployed_contracts` instead of moving it to `replaced_classes`. The returned `TransactionTraceWithHash` for tx[1] contains a wrong state diff.

---

### Impact Explanation

`starknet_traceBlockTransactions` returns an authoritative-looking wrong value: a contract address that was replaced within the pending block is reported as a fresh deployment. This falls squarely under: *"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

Clients relying on this trace (block explorers, indexers, tooling) will misinterpret the state transition â€” treating a class replacement as a new deployment â€” which can cause incorrect state reconstruction, incorrect event attribution, and incorrect contract history.

---

### Likelihood Explanation

The path is reachable by any unprivileged RPC caller. The scenario (deploy in tx[i], replace_class in tx[j > i] within the same pending block) is a normal contract upgrade pattern. No special privileges are required.

---

### Recommendation

`convert_thin_state_diff` must be made aware of intra-block state when processing pending transactions. Concretely, for the pending path, `is_deployed` should check not only the accepted block state but also the accumulated state diff from all preceding transactions in the same pending block. One approach: pass the cumulative `deployed_contracts` set from all prior transactions in the block as an additional input to `convert_thin_state_diff`, and treat any address present there as already deployed.

---

### Proof of Concept

```
1. Pending block contains:
   tx[0]: DeployAccount or Deploy â†’ deploys contract at address A with class_hash X
   tx[1]: Invoke on A â†’ calls replace_class(Y) on A

2. Call starknet_traceBlockTransactions(Tag::Pending)

3. For tx[1]:
   - induced_state_diff contains: deployed_contracts = {A: Y}
   - convert_thin_state_diff calls is_deployed(latest_accepted_block, A)
   - A is not in accepted state â†’ returns false
   - A stays in deployed_contracts

4. Actual correct result: A should be in replaced_classes = [{A, Y}]
   Returned wrong result: A is in deployed_contracts = [{A, Y}]
```

    async fn convert_thin_state_diff(
        &self,
        mut thin_state_diff: StarknetApiThinStateDiff,
        // TODO(AlonH): Remove the `block_id` parameter once we don't have pending blocks.
        block_id: BlockId,
        block_number: BlockNumber,
    ) -> RpcResult<ThinStateDiff> {
        let prev_block_number = match block_id {
            BlockId::Tag(Tag::Pending) => Some(block_number),
            _ => block_number.prev(),
        };
        let mut replaced_classes = vec![];
        for (&address, &class_hash) in thin_state_diff.deployed_contracts.iter() {
            // Check if the class was replaced.
            if let Some(prev_block_number) = prev_block_number {
                if self.is_deployed(prev_block_number, address).await? {
                    replaced_classes.push((address, class_hash));
                }
            }
        }
        replaced_classes.iter().for_each(|(address, _)| {
            thin_state_diff.deployed_contracts.swap_remove(address);
        });
        Ok(ThinStateDiff::from(thin_state_diff, replaced_classes))
    }

pub fn induced_state_diff(
    transactional_state: &mut CachedState<MutRefState<'_, CachedState<ExecutionStateReader>>>,
    deprecated_declared_class_hash: Option<ClassHash>,
) -> ExecutionResult<ThinStateDiff> {
    let blockifier_state_diff =
        CommitmentStateDiff::from(transactional_state.to_state_diff()?.state_maps);

    Ok(ThinStateDiff {
        deployed_contracts: blockifier_state_diff.address_to_class_hash,
        storage_diffs: blockifier_state_diff.storage_updates,
        class_hash_to_compiled_class_hash: blockifier_state_diff.class_hash_to_compiled_class_hash,
        deprecated_declared_classes: deprecated_declared_class_hash
            .map_or_else(Vec::new, |class_hash| vec![class_hash]),
        nonces: blockifier_state_diff.address_to_nonce,
    })
}
