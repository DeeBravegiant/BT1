Audit Report

## Title
Newly Added Tokens Default to Zero Fee in `RSETHPool`, Allowing Fee-Free Deposits — (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps[token]` that defaults to `0` for any key. The `addSupportedToken` function registers a token's oracle and bridge but never initializes `tokenFeeBps[token]`, so any deposit of that token immediately after listing incurs zero protocol fee. The fee revenue that should accrue to the protocol via `feeEarnedInToken[token]` is permanently lost for all deposits made before the admin separately calls `setTokenFeeBps`.

## Finding Description
Two distinct fee paths exist in the contract:

**ETH deposit** (`deposit(string)`, L265–278) calls `viewSwapRsETHAmountAndFee(uint256)` (L311–312), which charges `fee = amount * feeBps / 10_000`. [1](#0-0) 

**Token deposit** (`deposit(address,uint256,string)`, L284–305) calls `viewSwapRsETHAmountAndFee(uint256,address)` (L335–336), which charges `fee = amount * tokenFeeBps[token] / 10_000`. [2](#0-1) 

`tokenFeeBps` is a plain mapping declared at L88; its default value for any unset key is `0`. [3](#0-2) 

`addSupportedToken` (L637–656) sets `supportedTokenOracle[token]`, `tokenBridge[token]`, and pushes to `supportedTokenList`, but never writes to `tokenFeeBps[token]`. [4](#0-3) 

The only way to set a non-zero token fee is via the separate `setTokenFeeBps` call (L583–594), which is gated on `DEFAULT_ADMIN_ROLE` and is entirely decoupled from token listing. [5](#0-4) 

Because `fee = amount * 0 / 10_000 = 0`, the `feeEarnedInToken[token]` accumulator at L300 is never incremented, and the depositor receives the full token-equivalent rsETH amount with no fee deducted. [6](#0-5) 

No existing guard prevents a deposit immediately after `addSupportedToken`; the `onlySupportedToken` modifier at L100–103 only checks that `supportedTokenOracle[token] != address(0)`, which is satisfied the moment the token is added. [7](#0-6) 

## Impact Explanation
Every token deposit made while `tokenFeeBps[token] == 0` pays zero protocol fee. The fee revenue that should accrue to the protocol and be distributed to fee recipients via `withdrawFees` is permanently lost for those deposits. This constitutes **theft of unclaimed yield** (High severity per the allowed scope). The magnitude scales directly with deposit volume during the uninitialized window.

## Likelihood Explanation
The condition is triggered automatically the moment any token is added via `addSupportedToken`. No special attacker setup is required — any ordinary depositor calling `deposit(token, amount, referralId)` benefits from the zero fee. The window persists until the admin separately calls `setTokenFeeBps`, which is not enforced or atomically coupled to token addition. Since `addSupportedToken` requires `TIMELOCK_ROLE` and `setTokenFeeBps` requires `DEFAULT_ADMIN_ROLE`, these are distinct governance operations that may be batched separately, creating a realistic gap on every new token listing.

## Recommendation
Initialize `tokenFeeBps[token]` to `feeBps` (or a caller-supplied value) inside `addSupportedToken`, so that a newly listed token inherits the global fee rate by default:

```solidity
function addSupportedToken(address token, address oracle, address bridge) external onlyRole(TIMELOCK_ROLE) {
    // ... existing checks ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    tokenFeeBps[token] = feeBps;   // ← initialize to global fee
    emit AddSupportedToken(token, oracle, bridge);
}
```

Alternatively, fall back to `feeBps` inside `viewSwapRsETHAmountAndFee(uint256,address)` when `tokenFeeBps[token]` is zero.

## Proof of Concept
1. Admin calls `addSupportedToken(tokenA, oracleA, bridgeA)`. `tokenFeeBps[tokenA]` remains `0` (never set).
2. Any user immediately calls `deposit(tokenA, 1_000e18, "")`.
3. `viewSwapRsETHAmountAndFee(1_000e18, tokenA)` computes `feeBpsForToken = tokenFeeBps[tokenA] = 0`, so `fee = 1_000e18 * 0 / 10_000 = 0`.
4. `feeEarnedInToken[tokenA] += 0` — accumulator stays at zero.
5. User receives the full rsETH equivalent; protocol collects no fee.
6. Compare: an ETH deposit of equivalent value charges `fee = amount * feeBps / 10_000 > 0`.
7. Repeat for every deposit until admin calls `setTokenFeeBps(tokenA, feeBps)` in a separate transaction.

**Foundry test sketch:**
```solidity
function test_zeroFeeOnNewToken() public {
    vm.prank(timelockAdmin);
    pool.addSupportedToken(address(tokenA), address(oracleA), address(bridgeA));

    // tokenFeeBps[tokenA] is 0 — no setTokenFeeBps called
    uint256 amount = 1_000e18;
    tokenA.mint(user, amount);
    vm.startPrank(user);
    tokenA.approve(address(pool), amount);
    pool.deposit(address(tokenA), amount, "");
    vm.stopPrank();

    assertEq(pool.feeEarnedInToken(address(tokenA)), 0); // fee never collected
    // ETH deposit of same value would yield feeEarnedInETH > 0
}
```
