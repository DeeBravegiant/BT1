Audit Report

## Title
Decoupled `queuedWithdrawalsBuffer` Allows Instant Withdrawals to Drain Vault Assets Reserved for Queued Withdrawers, Temporarily Freezing Their Funds — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`instantWithdrawal` in `LRTWithdrawalManager` redeems assets directly from `LRTUnstakingVault` using only `queuedWithdrawalsBuffer` as a guard. That buffer is a manually-set mapping that defaults to zero and is never automatically synchronized with `assetsCommitted[asset]`. Once a manager legitimately enables instant withdrawal for an asset, any rsETH holder can drain the vault balance that was already committed to queued withdrawers, causing `unlockQueue` to revert and temporarily freezing those users' rsETH inside `LRTWithdrawalManager`.

## Finding Description

**Standard queue accounting (`LRTWithdrawalManager`):**
When `initiateWithdrawal` is called, `assetsCommitted[asset]` is incremented to reserve the expected payout:

```solidity
// LRTWithdrawalManager.sol L170-173
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`assetsCommitted` is decremented only inside `_unlockWithdrawalRequests` (L802), which is called exclusively from `unlockQueue`.

**Instant withdrawal path (`LRTWithdrawalManager`):**
`instantWithdrawal` (L212-253) burns rsETH and calls `unstakingVault.redeem` directly. Its only availability check delegates to `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal`:

```solidity
// LRTWithdrawalManager.sol L231-233
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
```

**Buffer guard (`LRTUnstakingVault`):**
`getAssetsAvailableForInstantWithdrawal` subtracts `queuedWithdrawalsBuffer[asset]` from the vault balance:

```solidity
// LRTUnstakingVault.sol L235-237
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
```

`queuedWithdrawalsBuffer` is a `mapping(address => uint256)` (L43) whose default value is zero. It can only be updated by an explicit operator call to `setQueuedWithdrawalsBuffer` (L199-209). There is no code path in `initiateWithdrawal`, `instantWithdrawal`, or anywhere else that automatically mirrors `assetsCommitted[asset]` into `queuedWithdrawalsBuffer[asset]`. The two accounting systems are completely decoupled.

**`unlockQueue` failure:**
`unlockQueue` reads `unstakingVault.balanceOf(asset)` as `totalAvailableAssets` (L849) and immediately reverts if it is zero (L297):

```solidity
// LRTWithdrawalManager.sol L297
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
```

Even if the vault retains a small residual balance, `unlockQueue` calls `unstakingVault.redeem(asset, assetAmountUnlocked)` (L307), which will revert if the vault cannot cover the full unlock amount.

## Impact Explanation

**Medium — Temporary freezing of funds.**

Queued withdrawers transfer their rsETH to `LRTWithdrawalManager` at `initiateWithdrawal` time (L166). After an attacker drains the vault via `instantWithdrawal`, the operator cannot call `unlockQueue` successfully. The queued users' rsETH remains locked inside `LRTWithdrawalManager` with no way to progress to `completeWithdrawal` until the vault is replenished through a new EigenLayer exit cycle (~7 days). This is a concrete, time-bounded freeze of user funds, matching the "Temporary freezing of funds" impact class.

## Likelihood Explanation

**Medium.** The single precondition is that a manager calls `setInstantWithdrawalEnabled(asset, true)` — a legitimate operational action to activate a protocol feature, not an attack. Once enabled, the exploit requires no special role: any rsETH holder can call `instantWithdrawal` repeatedly. The vault balance is publicly observable on-chain, so the attacker can size their position precisely. The buffer defaults to zero, so no additional misconfiguration is needed. The attack is repeatable every time the vault is replenished until the operator manually sets a non-zero buffer.

## Recommendation

1. **Automatic synchronization:** In `initiateWithdrawal`, after incrementing `assetsCommitted[asset]`, call `unstakingVault.setQueuedWithdrawalsBuffer(asset, assetsCommitted[asset])` so the vault always reserves at least the committed amount.
2. **Explicit guard in `instantWithdrawal`:** Before burning rsETH, verify that `unstakingVault.balanceOf(asset) - assetAmountUnlocked >= assetsCommitted[asset]`.
3. **Initialization on enable:** In `setInstantWithdrawalEnabled`, when enabling for an asset, set `queuedWithdrawalsBuffer[asset]` to the current `assetsCommitted[asset]` value so the buffer is never zero when the feature goes live.

## Proof of Concept

**Setup:**
- `LRTUnstakingVault` holds 100 ETH (vault balance). EigenLayer holds another 100 ETH.
- `queuedWithdrawalsBuffer[ETH] = 0` (default, never set).

**Steps:**

1. User A calls `initiateWithdrawal(ETH, rsETH_for_100_ETH)`.
   - `assetsCommitted[ETH]` becomes 100 ETH.
   - `getAvailableAssetAmount` returns `200 − 100 = 100 ETH` → request accepted.
   - User A's rsETH is transferred to `LRTWithdrawalManager`.

2. Manager calls `setInstantWithdrawalEnabled(ETH, true)`.
   - `queuedWithdrawalsBuffer[ETH]` is still 0.

3. Attacker calls `instantWithdrawal(ETH, rsETH_for_100_ETH)`.
   - `getAssetsAvailableForInstantWithdrawal(ETH)` returns `100 − 0 = 100 ETH`.
   - Check passes. `unstakingVault.redeem(ETH, 100 ETH)` executes. Vault balance → 0.

4. Operator calls `unlockQueue(ETH, ...)`.
   - `_createUnlockParams` reads `unstakingVault.balanceOf(ETH) = 0`.
   - `params.totalAvailableAssets == 0` → reverts with `AmountMustBeGreaterThanZero`.

5. User A cannot call `completeWithdrawal`. Their rsETH is locked in `LRTWithdrawalManager` until the vault is replenished via a new EigenLayer exit (~7 days).

**Foundry test sketch:**
```solidity
function test_instantWithdrawalDrainsQueuedFunds() public {
    // 1. Seed vault with 100 ETH
    vm.deal(address(unstakingVault), 100 ether);
    // 2. User A initiates withdrawal
    vm.prank(userA);
    withdrawalManager.initiateWithdrawal(ETH, rsETHFor100ETH, "");
    assertEq(withdrawalManager.assetsCommitted(ETH), 100 ether);
    // 3. Manager enables instant withdrawal (buffer stays 0)
    vm.prank(manager);
    withdrawalManager.setInstantWithdrawalEnabled(ETH, true);
    // 4. Attacker drains vault
    vm.prank(attacker);
    withdrawalManager.instantWithdrawal(ETH, rsETHFor100ETH, "");
    assertEq(address(unstakingVault).balance, 0);
    // 5. unlockQueue reverts
    vm.prank(operator);
    vm.expectRevert(LRTWithdrawalManager.AmountMustBeGreaterThanZero.selector);
    withdrawalManager.unlockQueue(ETH, type(uint256).max, ...);
}
```
