Audit Report

## Title
Unbounded Withdrawal Request Queue Enables Temporary Freezing of Legitimate Users' Withdrawals — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`initiateWithdrawal` imposes no per-user cap on the number of pending withdrawal requests and `minRsEthAmountToWithdraw` defaults to zero, making the effective minimum 1 wei of rsETH. Because `_unlockWithdrawalRequests` advances `nextLockedNonce[asset]` strictly in FIFO order with no skip mechanism, an attacker who front-loads the global queue with dust requests forces all subsequent users' withdrawals to wait until every preceding entry is processed, temporarily freezing their funds.

## Finding Description
`initiateWithdrawal` (L150–178) accepts any `rsETHUnstaked >= minRsEthAmountToWithdraw[asset]`. Because `minRsEthAmountToWithdraw` is a plain mapping never set in `initialize()` (L90–98), it defaults to `0`, making the only effective guard `rsETHUnstaked == 0`. Each call transfers rsETH from the caller, computes `expectedAssetAmount` (~1 wei of the underlying asset for 1 wei of rsETH), checks it against `getAvailableAssetAmount` (L599–603), increments `assetsCommitted[asset]`, and calls `_addUserWithdrawalRequest` (L744–760), which pushes a new entry into the global nonce counter:

```solidity
// L756-757
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
```

The operator's `_unlockWithdrawalRequests` (L770–816) iterates from `nextLockedNonce[asset]` to `firstExcludedIndex` one entry at a time with no ability to skip or cancel entries:

```solidity
// L790-815
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
```

Exploit path:
1. Attacker holds rsETH and calls `initiateWithdrawal(asset, 1, "")` repeatedly across many transactions, each committing ~1 wei of `assetsCommitted[asset]` and occupying one global nonce.
2. A legitimate user calls `initiateWithdrawal` and is assigned nonce N (after all attacker entries).
3. The operator must call `unlockQueue` enough times to advance `nextLockedNonce[asset]` through all N attacker entries before the legitimate user's request can be unlocked.
4. The legitimate user's rsETH remains locked in the contract and `completeWithdrawal` cannot be called until their nonce is reached.

Existing guards are insufficient: the `ExceedAmountToWithdraw` check (L170) only prevents a single request from exceeding available assets; it does not limit the cumulative number of requests. The `nonReentrant` modifier prevents batching within one call but does not prevent repeated calls across transactions.

## Impact Explanation
**Medium — Temporary freezing of funds.** Legitimate users' rsETH is transferred into the contract at `initiateWithdrawal` time (L166) and cannot be recovered until `completeWithdrawal` is called, which requires the operator to have advanced `nextLockedNonce[asset]` past the user's nonce. With a large number of attacker-inserted dust entries ahead in the queue, the user's funds are inaccessible for an attacker-controlled duration. This matches the allowed impact "Medium. Temporary freezing of funds."

## Likelihood Explanation
**Medium.** The attack is permissionless and requires only rsETH holdings and gas. `minRsEthAmountToWithdraw` defaults to zero, so the minimum cost per request is 1 wei of rsETH plus transaction gas. The attacker recovers their rsETH (as the underlying asset) when their requests are eventually processed, making the net cost gas-only. A moderately funded attacker can create thousands of dust entries at realistic gas prices. No privileged role, oracle compromise, or governance action is required.

## Recommendation
1. **Set a meaningful `minRsEthAmountToWithdraw` at initialization** and prevent it from being set to zero via the admin setter.
2. **Cap pending requests per user per asset**: add `require(userAssociatedNonces[asset][msg.sender].length() < MAX_REQUESTS_PER_USER)` in `_addUserWithdrawalRequest`.
3. **Allow the operator to cancel or skip stale/dust requests** so the queue cannot be permanently blocked by malicious entries.

## Proof of Concept
1. Deploy `LRTWithdrawalManager`; `minRsEthAmountToWithdraw[ETH]` is `0` by default (L35, no initialization in L90–98).
2. Attacker acquires rsETH and calls `initiateWithdrawal(ETH, 1, "")` N times across N transactions, each advancing `nextUnusedNonce[ETH]` by 1 and committing ~1 wei to `assetsCommitted[ETH]`.
3. Legitimate user calls `initiateWithdrawal(ETH, 1 ether, "")` — succeeds, assigned nonce N.
4. After `withdrawalDelayBlocks` pass, operator calls `unlockQueue(ETH, batchSize, ...)` repeatedly; each call advances `nextLockedNonce[ETH]` by at most `batchSize` entries, all of which are attacker dust.
5. Legitimate user's request at nonce N cannot be unlocked until all N preceding attacker entries are processed; `completeWithdrawal` reverts until then, confirming temporary freezing of the legitimate user's rsETH.
