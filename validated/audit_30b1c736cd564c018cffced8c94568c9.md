### Title
Reference Script Fee Calculation Omits Subtransaction Scripts in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee calculation for a top-level transaction uses `getConwayMinFeeTxUtxo`, which only counts reference scripts from the **top-level transaction's inputs**. However, the per-transaction reference-script size limit check uses `batchNonDistinctRefScriptsSize`, which correctly sums reference scripts from **both the top-level transaction and all its subtransactions**. This asymmetry means an attacker can embed large reference scripts exclusively in subtransactions, paying zero reference-script fee surcharge while still forcing validators to deserialize up to `ppMaxRefScriptSizePerTx` bytes of scripts.

---

### Finding Description

The Dijkstra era introduces nested sub-transactions (`subTransactionsTxBodyL`). Two separate functions measure the total reference-script footprint of a batch:

**Fee calculation path** — `getMinFeeTxUtxo` for `DijkstraEra` delegates to `getConwayMinFeeTxUtxo`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs:141
getMinFeeTxUtxo = getConwayMinFeeTxUtxo
```

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs:174-175
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

`txNonDistinctRefScriptsSize` only unions the top-level transaction's `referenceInputsTxBodyL` and `inputsTxBodyL`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs:183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

Subtransactions' reference inputs are **never consulted**.

**Size-limit check path** — `validateAllRefScriptSize` in the Dijkstra LEDGER rule uses `batchNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs:321-329
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $ ...
```

`batchNonDistinctRefScriptsSize` correctly aggregates both levels:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs:271-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

The design intent is clear: the size limit enforces a cap on the **total** deserialization burden (top + sub). The fee surcharge, however, only reflects the top-level portion. This is the same structural error as the external report: a quantity that should scale with N items (all reference scripts across all transaction levels) instead scales with a strict subset (top-level only).

---

### Impact Explanation

**Medium — fees modified outside design parameters.**

An attacker submits a Dijkstra top-level transaction with zero or minimal reference inputs at the top level, while embedding subtransactions that each reference large Plutus scripts. The `tierRefScriptFee` surcharge (the exponential pricing introduced to deter DDoS) is never applied to the subtransaction scripts. The attacker pays only the base `a * txSize + b` fee, while validators must deserialize up to `ppMaxRefScriptSizePerTx` bytes of scripts across the batch. This directly undermines the fee-as-deterrent mechanism that was introduced after the June 2024 DDoS attack and documented in ADR 9.

---

### Likelihood Explanation

Dijkstra subtransactions are a first-class feature; any DApp that uses reference scripts inside subtransactions triggers this path. A deliberate attacker can maximize the gap by placing all reference inputs exclusively in subtransactions. No special privilege, key, or governance majority is required — a normal transaction sender suffices.

---

### Recommendation

Replace `getConwayMinFeeTxUtxo` in the `DijkstraEra` `EraUTxO` instance with a Dijkstra-specific implementation that passes `batchNonDistinctRefScriptsSize utxo tx` (instead of `txNonDistinctRefScriptsSize utxo tx`) as the `refScriptsSize` argument to `getMinFeeTx`. This makes the fee surcharge consistent with the size-limit check and closes the gap.

---

### Proof of Concept

1. Deploy a large Plutus V3 script (e.g., 50 KiB) as a reference script UTxO on Dijkstra.
2. Construct a Dijkstra top-level transaction with:
   - One ordinary spending input (no reference inputs at the top level).
   - One subtransaction whose `referenceInputsTxBodyL` points to the 50 KiB script UTxO.
3. Compute `getMinFeeTxUtxo pp tx utxo` — it returns `alonzoMinFeeTx pp tx` with zero ref-script surcharge, because `txNonDistinctRefScriptsSize` finds no ref scripts at the top level.
4. Compute `batchNonDistinctRefScriptsSize utxo tx` — it returns 50 KiB (from the subtransaction).
5. The transaction is accepted by `validateAllRefScriptSize` (50 KiB ≤ `ppMaxRefScriptSizePerTx`) and by `validateFeeTooSmallUTxO` (fee ≥ `getMinFeeTxUtxo`), but the validator must deserialize 50 KiB of scripts while collecting only the base fee — no `tierRefScriptFee` surcharge is charged.
