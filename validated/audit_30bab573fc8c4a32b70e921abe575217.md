Audit Report

## Title
ETH Withdrawal Completion Temporarily Blocked When Aave External Call Reverts in `_processWithdrawalCompletion` - (File: contracts/LRTWithdrawalManager.sol)

## Summary
When `isAaveIntegrationEnabled` is `true` and the contract's idle ETH balance is insufficient, `_processWithdrawalCompletion` calls `_withdrawFromAave`, which makes an unchecked external call to `aaveWETHGateway.withdrawETH`. If Aave is paused or otherwise unavailable, this call reverts, causing the entire `completeWithdrawal` transaction to revert. Because all EVM state changes are rolled back on revert, the user's request record is preserved and they can retry — but they are temporarily frozen out of completing their withdrawal for as long as Aave remains unavailable. Critically, the admin escape hatches (`setAaveIntegrationEnabled(false)` and `emergencyWithdrawFromAave`) both route through the same `_withdrawFromAave` call and are equally blocked, preventing administrative remediation.

## Finding Description
In `_processWithdrawalCompletion` (lines 699–738), the Aave withdrawal call at line 724 is made with no error handling:

```solidity
// line 724
_withdrawFromAave(amountNeeded);
```

`_withdrawFromAave` at line 917 makes a bare external call:

```solidity
aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
```

If this reverts, the entire transaction reverts (rolling back `popFront()` at line 705 and `delete withdrawalRequests[requestId]` at line 712 — the user's request is **not** permanently lost). However, the user cannot complete their withdrawal until Aave recovers.

The admin remediation paths are equally blocked:
- `setAaveIntegrationEnabled(false)` calls `_withdrawFromAave(aaveBalance)` at line 495 before setting the flag, so it also reverts while Aave is paused.
- `emergencyWithdrawFromAave()` at line 560 calls `_withdrawFromAave(amount)` directly.

This contrasts sharply with the deposit path at lines 311–316, where the developers explicitly wrapped the Aave deposit in `try/catch`, acknowledging that Aave calls can fail:

```solidity
try this.depositToAaveExternal(assetAmountUnlocked) { }
catch (bytes memory reason) {
    emit AaveDepositFailed(assetAmountUnlocked, reason);
    // Silently fail if Aave deposit fails
}
```

The same defensive pattern was not applied to the withdrawal path, creating an asymmetry where deposits degrade gracefully but withdrawals do not.

## Impact Explanation
**Medium — Temporary freezing of funds.** All users with unlocked ETH withdrawal requests are unable to call `completeWithdrawal` while Aave is unavailable. Because all three admin remediation paths (`setAaveIntegrationEnabled(false)`, `emergencyWithdrawFromAave`, and direct `_withdrawFromAave`) share the same broken external call, the freeze cannot be administratively resolved until Aave itself recovers. The duration of the freeze is bounded by Aave's recovery time but is not under the protocol's control.

## Likelihood Explanation
**Medium.** The condition requires: (1) Aave integration enabled (explicit admin opt-in), (2) the contract's idle ETH balance below a user's withdrawal amount (the normal operating state when ETH is deployed to Aave), and (3) Aave being temporarily paused. Aave v3 pools can be paused by Aave governance or the Aave guardian in response to market stress or risk events; the Aave WETH pool on Ethereum mainnet has been paused historically. The developers' own use of `try/catch` on the deposit path confirms they considered Aave unavailability a realistic scenario.

## Recommendation
Apply the same `try/catch` defensive pattern used in `unlockQueue` to the Aave withdrawal call inside `_processWithdrawalCompletion`. The safest approach is to move the `delete withdrawalRequests[requestId]` and `popFront()` operations to **after** the ETH transfer to the user is confirmed successful, so that if the Aave call fails, the request remains intact and the function can revert cleanly without any state mutation. Alternatively, wrap `_withdrawFromAave` in a try/catch and revert with a clear error before any state is mutated if Aave is unavailable. Additionally, `setAaveIntegrationEnabled(false)` should be refactored to not depend on `_withdrawFromAave` succeeding — for example, by skipping the withdrawal if Aave is unreachable and leaving the aWETH in place until Aave recovers.

## Proof of Concept
1. Admin calls `setAaveIntegrationEnabled(true)`.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. Alice's request is unlocked; ETH is deposited to Aave via `_depositToAave`. Contract idle ETH balance is now 0.
3. Aave governance pauses the WETH pool (a documented Aave v3 guardian capability).
4. Alice calls `completeWithdrawal(ETH_TOKEN, "")`.
5. Inside `_processWithdrawalCompletion`: `isAaveIntegrationEnabled` is `true`, `address(this).balance == 0 < request.expectedAssetAmount`, so `_withdrawFromAave(amountNeeded)` is called at line 724.
6. `aaveWETHGateway.withdrawETH(...)` at line 917 reverts because the pool is paused.
7. The entire transaction reverts. Alice's request is preserved (EVM rolls back all state), but she cannot complete her withdrawal.
8. Admin attempts `setAaveIntegrationEnabled(false)` — this also reverts at line 495 because `_withdrawFromAave` is called before the flag is cleared.
9. Admin attempts `emergencyWithdrawFromAave(amount)` — this also reverts at line 560.
10. Alice and all other users with unlocked ETH withdrawals are frozen until Aave's pool is unpaused.
