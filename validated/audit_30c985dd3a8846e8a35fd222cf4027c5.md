Audit Report

## Title
Stale `rsETHPrice` Allows Withdrawers to Shift Slashing Losses to Remaining rsETH Holders - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTOracle` stores `rsETHPrice` as a state variable updated only when `updateRSETHPrice()` is explicitly called, which is not invoked atomically with any withdrawal path. After an EigenLayer slashing event reduces actual TVL, any rsETH holder can call `instantWithdrawal` during the stale-price window and receive more underlying assets than their proportional share, transferring the unabsorbed slashing loss entirely to remaining holders.

## Finding Description
`rsETHPrice` is a plain storage variable in `LRTOracle` (line 28). `updateRSETHPrice()` is a public function (lines 87–89) called by off-chain bots on a periodic schedule, not on every block and not inside any withdrawal path.

`getExpectedAssetAmount` in `LRTWithdrawalManager` (lines 580–594) reads the stored price directly:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

`instantWithdrawal` (lines 212–253) calls this function to determine `assetAmountUnlocked`, immediately burns the caller's rsETH, and redeems the computed amount from `LRTUnstakingVault` — all in a single transaction with no price-freshness check.

The only guard against a falling price is the `pricePercentageLimit` circuit-breaker in `_updateRsETHPrice` (lines 269–282): if the computed new price drops more than `pricePercentageLimit` below `highestRsethPrice`, the protocol is paused. However, this guard fires only when `updateRSETHPrice()` is actually called. Between a slashing event and the next oracle update call, the stored `rsETHPrice` remains at its pre-slash value and the circuit-breaker is never evaluated. For slashing events whose magnitude falls below `pricePercentageLimit`, the protocol would not even pause when the oracle is eventually updated, meaning the stale price is silently used for all withdrawals in the interim.

`initiateWithdrawal` (lines 150–178) also calls `getExpectedAssetAmount` at request time and locks in `expectedAssetAmount` into `assetsCommitted`, so the same staleness window affects queued withdrawals as well.

## Impact Explanation
Users who withdraw during the stale-price window burn rsETH at the inflated stored price and receive more underlying assets than their fair share. When `updateRSETHPrice()` is eventually called, the lower post-slash price is reflected and all remaining rsETH holders bear the full slashing loss, including the portion that should have been borne by the exiting users. This is a concrete, quantifiable transfer of value from remaining holders to early exiters.

**Impact: High — Theft of unclaimed yield (proportional slashing loss shifted from exiting users to remaining holders).**

## Likelihood Explanation
EigenLayer slashing events are an expected, documented risk for restaking protocols. Oracle update bots operate on a periodic schedule, not per-block, making the staleness window non-trivial and predictable. Any rsETH holder can monitor EigenLayer on-chain events (slashing is publicly visible), identify the window before the oracle bot fires, and call the unprivileged `instantWithdrawal` function. No privileged access, flash loan, or external protocol compromise is required. The attack is repeatable across every slashing event whose magnitude is below `pricePercentageLimit`.

## Recommendation
Before computing the payout in `instantWithdrawal` (and at request-time in `initiateWithdrawal`), call `updateRSETHPrice()` to ensure the stored price reflects current TVL. Alternatively, compute the payout inline from a live TVL snapshot rather than reading the cached `rsETHPrice`, or revert if `block.timestamp` exceeds a configurable staleness threshold since the last oracle update.

## Proof of Concept
1. Protocol TVL = 1000 ETH, rsETH supply = 1000, `rsETHPrice` = 1.0 ETH/rsETH.
2. EigenLayer slashing event reduces actual TVL to 990 ETH on-chain. `rsETHPrice` is still 1.0 (oracle bot has not yet called `updateRSETHPrice()`).
3. Alice holds 100 rsETH. She calls `instantWithdrawal(ETH, 100e18, "")`.
4. `getExpectedAssetAmount` returns `100 * 1.0 / 1.0 = 100 ETH` (stale price).
5. Alice's 100 rsETH is burned; vault pays out 100 ETH.
6. Remaining TVL = 890 ETH, remaining rsETH supply = 900.
7. Oracle bot calls `updateRSETHPrice()`: new price = 890/900 ≈ 0.9889 ETH/rsETH.
8. Alice avoided ~1 ETH of her proportional slashing loss. The remaining 900 holders bear the full 10 ETH loss instead of 9 ETH.

**Foundry fork test plan:** Fork mainnet, set `rsETHPrice` to a known value, simulate a slashing event by reducing the EigenLayer delegated balance (or mock `_getTotalEthInProtocol` to return a lower value), call `instantWithdrawal` before calling `updateRSETHPrice()`, then call `updateRSETHPrice()` and assert that the post-withdrawal price is lower than `(TVL_post_slash - withdrawn_assets) / remaining_supply`, confirming the loss was disproportionately absorbed by remaining holders.
