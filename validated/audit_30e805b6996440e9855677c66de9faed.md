Audit Report

## Title
Missing Staleness Checks in `ChainlinkPriceOracle.getAssetPrice()` Enables Over-Minting of rsETH at Expense of Existing Holders - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all staleness-related return values (`roundId`, `startedAt`, `updatedAt`, `answeredInRound`), with no `price > 0` guard either. In a multi-asset protocol, when only one asset's Chainlink feed is stale at an inflated price, the minting formula's numerator (`getAssetPrice(asset)`) is inflated by the full staleness delta while the denominator (`rsETHPrice`, a weighted average across all assets) is inflated by only a fraction of that delta. An unprivileged depositor can exploit this divergence to mint more rsETH than the real ETH value of their deposit, extracting value from all existing rsETH holders upon redemption.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at `contracts/oracles/ChainlinkPriceOracle.sol` lines 49–55:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData()` — `(roundId, answer, startedAt, updatedAt, answeredInRound)` — are received, but only `answer` is used. There is no check for:
- `block.timestamp - updatedAt <= heartbeat` (time-based freshness)
- `answeredInRound >= roundId` (round-completeness guard)
- `price > 0` (negative/zero price guard)

The sibling contract `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 26–37 performs all three of these checks, confirming the team is aware of the requirement but omitted it from the primary deposit-path oracle.

The deposit minting formula in `contracts/LRTDepositPool.sol` line 520:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` delegates directly to `ChainlinkPriceOracle.getAssetPrice()` (`contracts/LRTOracle.sol` line 157). `rsETHPrice` is a stored value computed by `_updateRsETHPrice()` → `_getTotalEthInProtocol()`, which also calls `getAssetPrice()` for every supported asset and sums their ETH values (`contracts/LRTOracle.sol` lines 336–343).

**Why the attack works in a multi-asset protocol:**

`rsETHPrice` is a TVL-weighted average across all supported assets. If only asset A's feed is stale at price `P_stale` (real price `P_real < P_stale`), and asset A represents fraction `f` of total TVL:

- `getAssetPrice(A)` is inflated by factor `P_stale / P_real`
- `rsETHPrice` is inflated by only `1 + f * (P_stale/P_real - 1)` (weighted average)

Since `f < 1`, the numerator is inflated more than the denominator, and the attacker receives more rsETH than the real ETH value of their deposit.

**Concrete example:**
- Protocol holds 500 stETH + 500 rETH; rsETH supply = 1000
- stETH Chainlink feed is stale: reports 1.05 ETH, real price = 0.90 ETH; rETH feed is fresh at 1.00 ETH
- `rsETHPrice` (computed with stale stETH): `(500×1.05 + 500×1.00) / 1000 = 1.025 ETH`
- Attacker deposits 1000 stETH (real value: 900 ETH): `rsethToMint = 1000 × 1.05 / 1.025 ≈ 1024.4 rsETH`
- With accurate prices: `rsethToMint = 1000 × 0.90 / 0.95 ≈ 947.4 rsETH`
- After Chainlink updates to 0.90 ETH and `rsETHPrice` is recalculated: `(1500×0.90 + 500×1.00) / 2024.4 ≈ 0.914 ETH`
- Attacker redeems 1024.4 rsETH → receives `1024.4 × 0.914 ≈ 936 ETH`
- **Attacker profit: ~36 ETH; original holders lose ~36 ETH in real value**

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (`contracts/LRTOracle.sol` lines 252–266) only gates the rsETH price *update* transaction; it does not protect individual `getAssetPrice()` calls made during deposits.

## Impact Explanation

**Critical — Direct theft of user funds.** Existing rsETH holders are diluted: the attacker deposits an asset at an inflated stale price, receives excess rsETH, and upon redemption (after the feed corrects and `rsETHPrice` is recalculated downward) extracts more ETH than they contributed. The shortfall is borne by all other rsETH holders proportionally. This constitutes direct, at-rest theft of user funds.

## Likelihood Explanation

Chainlink feeds update only on price-deviation threshold or heartbeat expiry (typically 1 hour on mainnet, up to 24 hours for some feeds). During rapid market moves (flash crashes), network congestion, or Chainlink node issues, the on-chain price can lag the real market price for the full heartbeat window — a well-documented, realistic scenario. `depositAsset()` is callable by any unprivileged user at any time with no special preconditions beyond the contract not being paused. The attack is repeatable across any supported LST asset whose feed becomes stale.

## Recommendation

Add staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > feedHeartbeat[asset]) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Add `mapping(address asset => uint256 heartbeat) public feedHeartbeat` and require it to be set alongside `assetPriceFeed` in `updatePriceFeedFor`.

## Proof of Concept

**Fork test plan (Foundry):**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed last updated at `T - 2h` (heartbeat = 1h).
2. Warp `block.timestamp` forward by 2 hours without advancing the feed (simulating a stale feed).
3. Record `rsETHPrice` (call `lrtOracle.updateRSETHPrice()` at the warped timestamp to lock in the stale-price-based rsETH price).
4. As an unprivileged attacker, call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
5. Assert that `rsethAmountToMint` exceeds `1000e18 * realStETHPrice / rsETHPrice` (i.e., attacker received excess rsETH).
6. Advance the feed to the real current price; call `lrtOracle.updateRSETHPrice()`.
7. Redeem the attacker's rsETH and assert the ETH received exceeds the real ETH value of the original deposit.
