### Title
Excess Deposit Silently Locked in Contract on Participant Re-Attestation — (`File: crates/contract/src/lib.rs`)

### Summary

`submit_participant_info` is marked `#[payable]` and accepts any attached deposit, but the deposit-check-and-refund block is guarded by a condition that is `false` for the most common production call path: an existing participant re-submitting their attestation. Any deposit attached in that case is silently absorbed by the contract with no refund.

### Finding Description

`submit_participant_info` computes a boolean gate before deciding whether to validate or refund the caller's deposit:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check attached >= cost, refund excess
}
``` [1](#0-0) 

When the caller **is** an existing participant (`caller_is_not_participant = false`) **and** the attestation already exists (`is_new_attestation = false`), the entire block is skipped. The function returns `Ok(())` without ever reading `env::attached_deposit()` or scheduling a refund. Because the function is `#[payable]`, NEAR does not automatically return the deposit; it stays in the contract's balance permanently.

The analogous pattern in the reference report is the `>=` vs `==` check in `buyOption()`: both allow callers to attach more value than required, and neither returns the excess.

The production re-submission path is exercised on every hourly `periodic_attestation_submission` cycle. The unit-test helper `submit_attestation` already attaches `NearToken::from_near(1)` (1 NEAR) to every call, demonstrating that callers do attach deposits to this function. [2](#0-1) 

### Impact Explanation

Any deposit attached by an existing participant during a re-submission is permanently locked inside the contract. There is no recovery path: the contract has no admin withdrawal method, and the deposit is not associated with any pending request that could time out and trigger a refund. This breaks the accounting invariant that deposits are only retained to cover storage costs and are otherwise returned to the caller.

**Impact class**: Medium — balance/accounting invariant violation that causes direct, irreversible fund loss for the caller without requiring any external attacker.

### Likelihood Explanation

The MPC node re-submits attestations on a 1-hour cadence and on attestation-removal events. A misconfigured node, a frontend wrapper that always attaches a safety deposit, or a future code change that adds a deposit to re-submissions would silently lose funds on every cycle. The unit-test helper already demonstrates the pattern of attaching 1 NEAR unconditionally.

### Recommendation

Add an unconditional refund of any non-zero attached deposit when `attestation_storage_must_be_paid_by_caller` is `false`, mirroring the pattern already used in `require_deposit` and `propose_update`:

```rust
} else {
    // Existing participant re-submission: no storage cost.
    // Refund any deposit the caller accidentally attached.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, assert that no deposit is attached in this branch, making accidental overpayment an explicit error rather than a silent loss.

### Proof of Concept

1. Contract is in `Running` state with participant `alice.near` already registered and attested.
2. `alice.near` calls `submit_participant_info` with a valid updated attestation and attaches `1 NEAR`.
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` (not `NewlyInsertedParticipant`).
4. `is_new_attestation = false`, `caller_is_not_participant = false` → `attestation_storage_must_be_paid_by_caller = false`.
5. The `if` block is skipped; `env::attached_deposit()` is never read; no refund promise is scheduled.
6. The call returns `Ok(())`; `alice.near`'s balance is reduced by 1 NEAR with no corresponding storage allocation or refund. [3](#0-2)
