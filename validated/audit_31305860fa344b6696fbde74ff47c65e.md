### Title
Single Participant Can Indefinitely Reset Key Event Instance via `vote_abort_key_event_instance` — (File: `crates/contract/src/state/key_event.rs`)

---

### Summary

The `vote_abort` function in `KeyEvent` immediately nullifies the active key generation or resharing instance when called by **any single attested participant**, with no threshold requirement. This is the structural analog of the original bug: just as a third party committing a payment channel reset the withdrawal lockdown period to zero, a single Byzantine participant below the signing threshold can repeatedly reset the key event instance to `None`, permanently preventing key generation (`Initializing → Running`) or resharing (`Resharing → Running`) from ever completing.

---

### Finding Description

`KeyEvent::vote_abort` in `crates/contract/src/state/key_event.rs` unconditionally destroys the active `KeyEventInstance` on a single participant's call:

```rust
pub fn vote_abort(&mut self, key_event_id: KeyEventId) -> Result<(), Error> {
    let candidate = self.verify_vote(&key_event_id)?;
    if self.instance.as_ref().unwrap().completed.contains(&candidate) {
        return Err(VoteError::VoteAlreadySubmitted.into());
    }
    self.instance = None;   // ← entire attempt wiped by one vote
    Ok(())
}
``` [1](#0-0) 

The only guard is that the aborting participant must not have already cast a `vote_success` for this attempt. There is no threshold check — one participant suffices. The on-chain entry point `vote_abort_key_event_instance` requires only that the caller is an attested participant: [2](#0-1) 

After the abort, the leader must call `start_keygen_instance` or `start_reshare_instance` again to open a new attempt with a fresh `KeyEventInstance`: [3](#0-2) 

The same participant can immediately abort the new instance. Because `next_attempt_id` is monotonically incremented and the aborter simply supplies the new `key_event_id` each time, there is no rate-limit, cooldown, or escalating cost that would deter repetition.

The test suite explicitly confirms that a single participant's abort is immediate and unconditional: [4](#0-3) 

The same pattern holds for the `Initializing` state: [5](#0-4) 

---

### Impact Explanation

A single Byzantine participant, strictly below the signing threshold, can:

1. **Permanently block key generation** — the `Initializing → Running` transition never completes, so new domains (new signing keys) can never be activated.
2. **Permanently block resharing** — the `Resharing → Running` transition never completes. This means the network cannot update its participant set, change the threshold, or remove a compromised node. Critically, a compromised participant can use this to **prevent their own removal** from the network, which is a concrete security invariant violation beyond pure griefing.

The `Resharing` state is reached via `vote_new_parameters`, which itself requires a threshold of votes: [6](#0-5) 

So the governance process to remove a bad actor requires threshold cooperation to *start*, but a single bad actor can *prevent it from finishing* indefinitely.

---

### Likelihood Explanation

Any single attested participant can trigger this. In a production t-of-n network (e.g., 6-of-9), eight participants are individually below threshold. A participant whose removal is being voted on has a direct economic incentive to abort every resharing attempt. The attack requires only a valid TEE attestation and knowledge of the current `key_event_id`, both of which are publicly observable on-chain via the contract state.

---

### Recommendation

Require a threshold number of `vote_abort` votes before nullifying the instance, mirroring the threshold requirement used for all other governance state transitions. Alternatively, track per-participant abort votes and only clear the instance once `t` participants have voted to abort, discarding the instance only when consensus on failure is reached.

---

### Proof of Concept

1. Governance threshold votes trigger `vote_new_parameters`; contract enters `Resharing` state.
2. Leader (lowest participant ID) calls `start_reshare_instance(key_event_id)` — a new `KeyEventInstance` is created.
3. Byzantine participant (any single attested node) calls `vote_abort_key_event_instance(key_event_id)`.
4. `KeyEvent::vote_abort` executes: `self.instance = None`. The entire attempt is wiped.
5. Leader must call `start_reshare_instance` again with `next_attempt_id` incremented.
6. Byzantine participant reads the new `key_event_id` from chain state and repeats step 3.
7. Steps 5–6 repeat indefinitely; `Resharing → Running` never completes; the compromised participant is never removed. [1](#0-0) [2](#0-1) [7](#0-6)
