### Title
Validator Inserts Invalid Proposal into `valid_proposals` Cache Before Commitment Verification — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

In `validate_proposal`, the validator unconditionally inserts the proposal into the `valid_proposals` cache **before** checking whether the locally-computed `built_block` commitment matches the network-supplied `received_fin.proposal_commitment`. If the commitments differ, the function returns `ProposalFinMismatch`, but the poisoned entry is already resident in the cache. This is the direct Sequencer analog of the TON multisig bug: a state-transition is recorded as complete before the authoritative confirmation check is performed, and the recorded state cannot be undone.

### Finding Description

In `validate_proposal` (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, lines 238–247), the sequence is:

```
// 1. Insert into valid_proposals (BEFORE the commitment check)
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

// 2. Check commitment match (AFTER insertion)
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}
``` [1](#0-0) 

The comment at line 238 explicitly acknowledges the ordering is intentional ("to avoid a race condition with `repropose`"), but it does not account for the case where the commitment check subsequently fails. When `built_block != received_fin.proposal_commitment`, the function returns an error, but the entry keyed by `(height, round)` is already permanently stored in `valid_proposals`.

`insert_proposal` asserts that no prior entry exists for the same `(height, round)`:

```rust
assert!(
    previous.is_none(),
    "Overwriting existing proposal for height {height} round {round}; ..."
);
``` [2](#0-1) 

This means the slot is permanently occupied by the mismatched proposal. Any subsequent attempt to insert a valid proposal for the same `(height, round)` — e.g., after a retry or a re-validation — will **panic**.

The `decision_reached` path reads from `valid_proposals` using `get_proposal`, which asserts the stored commitment matches the consensus-decided commitment:

```rust
assert_eq!(
    stored_commitment, commitment,
    "Proposal commitment mismatch for height {height} round {round}: ..."
);
``` [3](#0-2) 

If consensus later decides on the same `(height, round)` with the correct commitment, `decision_reached` will panic because the stored commitment is the mismatched one from the poisoned insertion.

### Impact Explanation

**Impact: High — RPC/consensus admission accepts or permanently blocks a valid proposal.**

The corrupted `valid_proposals` entry for `(height, round)` causes one of two outcomes:

1. **Consensus deadlock / node crash**: If the BFT layer reaches a decision for the same `(height, round)` with the correct commitment, `decision_reached` panics at the `assert_eq!` in `get_proposal`, crashing the node.
2. **Permanent slot poisoning**: If a retry of `validate_proposal` for the same `(height, round)` is attempted, `insert_proposal` panics at the `assert!(previous.is_none(), ...)` guard, again crashing the node.

Either path leaves the node unable to finalize the block at that height, breaking liveness. The wrong `ProposalCommitment` (derived from a mismatched `partial_block_hash`) is the corrupted value.

### Likelihood Explanation

The `ProposalFinMismatch` path is reachable whenever a proposer sends a `ProposalFin` whose `proposal_commitment` does not match the locally re-executed block. This can occur:
- Due to a bug in the proposer's commitment computation.
- Due to a network-level message substitution or corruption.
- Due to a legitimate divergence in execution results (e.g., non-determinism, version skew).

The code path is exercised by the existing test `proposal_fin_mismatch` in `validate_proposal_test.rs`, which confirms the mismatch is reachable and returns an error — but the test does not check whether the poisoned entry was inserted into `valid_proposals`. [4](#0-3) 

### Recommendation

Move `valid_proposals.insert_proposal(...)` to **after** the commitment check succeeds:

```rust
// Check commitment match FIRST
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

// Only insert if the commitment is valid
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);
```

The race condition with `repropose` cited in the comment must be re-evaluated: `repropose` is only called after consensus has already voted on a valid commitment, so it should only be called after `validate_proposal` has returned `Ok`. If the race is real, it should be addressed by a separate synchronization mechanism rather than by inserting an unverified entry.

### Proof of Concept

1. A validator node receives a proposal for `(height=N, round=R)`.
2. The batcher executes the transactions and returns `built_block = commitment_A`.
3. The proposer's `ProposalFin` carries `proposal_commitment = commitment_B` (where `commitment_B != commitment_A`).
4. `validate_proposal` calls `valid_proposals.insert_proposal(...)` — the entry `(N, R) -> commitment_A` is stored.
5. `validate_proposal` detects the mismatch and returns `Err(ProposalFinMismatch)`.
6. Consensus retries or another validator sends a corrected `ProposalFin` with `commitment_A`.
7. `validate_proposal` is called again for `(N, R)`.
8. `insert_proposal` hits `assert!(previous.is_none(), ...)` and **panics**, crashing the node.

Alternatively, if consensus reaches a decision for `(N, R)` with `commitment_A`:

6. `decision_reached(N, R, commitment_A)` calls `get_proposal(N, R, commitment_A)`.
7. The stored entry has `stored_commitment = commitment_A` — this path actually succeeds in this scenario.

However, if the mismatch is `built_block = commitment_A` stored, and consensus decides `commitment_B` (the network value), then `get_proposal` panics at the `assert_eq!`.

The most direct crash path is the retry scenario (step 8 above), which is reachable without any privileged access — only a proposer that sends a `ProposalFin` with a mismatched commitment is required, which any Byzantine or buggy proposer can do. [5](#0-4) [6](#0-5)

    pub(crate) fn insert_proposal(
        &mut self,
        init: ProposalInit,
        transactions: Vec<Vec<InternalConsensusTransaction>>,
        proposal_id: &ProposalId,
        finished_info: FinishedProposalInfo,
    ) {
        let proposal_commitment = proposal_commitment_from(
            finished_info.proposal_commitment.partial_block_hash,
            init.fee_proposal_fri,
        );

        let height = init.height;
        let round = init.round;
        let by_round = self.data.entry(height).or_default();
        let previous = by_round.insert(
            round,
            (proposal_commitment, (init, transactions, *proposal_id, finished_info)),
        );
        assert!(
            previous.is_none(),
            "Overwriting existing proposal for height {height} round {round}; at most one \
             proposal per (height, round) is allowed"
        );
    }

#[tokio::test]
async fn proposal_fin_mismatch() {
    let (mut proposal_args, mut content_sender) = create_proposal_validate_arguments();
    let n_executed = 0;
    // Setup batcher to validate the block.
    proposal_args.deps.batcher.expect_validate_block().returning(|_| Ok(()));
    // Batcher returns a different block hash than the one received in Fin.
    let built_block = PartialBlockHash(Felt::ONE);
    proposal_args
        .deps
        .batcher
        .expect_finish_proposal()
        .withf(move |input: &FinishProposalInput| {
            input.proposal_id == proposal_args.proposal_id
                && input.final_n_executed_txs == n_executed
        })
        .returning(move |_| {
            Ok(FinishProposalStatus::Finished(FinishedProposalInfo {
                artifact: FinishedProposalInfoWithoutParent {
                    proposal_commitment: ProposalCommitment { partial_block_hash: built_block },
                    final_n_executed_txs: n_executed,
                    block_header_commitments: BlockHeaderCommitments::default(),
                    l2_gas_used: GasAmount::default(),
                },
                parent_proposal_commitment: None,
            }))
        });
    let received_fin = ConsensusProposalCommitment::default();
    content_sender
        .send(ProposalPart::Fin(ProposalFin {
            proposal_commitment: received_fin,
            executed_transaction_count: n_executed.try_into().unwrap(),
            fin_payload: None,
        }))
        .await
        .unwrap();

    let res = validate_proposal(proposal_args.into()).await;
    assert!(matches!(res, Err(ValidateProposalError::ProposalFinMismatch)));
