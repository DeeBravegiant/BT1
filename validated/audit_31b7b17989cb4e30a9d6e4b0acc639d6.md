Audit Report

## Title
Unrestricted `updateRSETHPrice()` Allows Any Caller to Atomically Pause Three Contracts — (File: `contracts/LRTOracle.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is a public function with no access control. When the computed rsETH price falls below `highestRsethPrice` by more than `pricePercentageLimit`, it atomically pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle` itself. Restoring the protocol requires the admin to issue three separate `onlyLRTAdmin` unpause transactions. Any unprivileged EOA can trigger this freeze whenever the price condition is satisfied.

## Finding Description

`updateRSETHPrice()` carries no access-control modifier:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Inside `_updateRsETHPrice()`, the downside-protection branch executes unconditionally for any caller:

```solidity
// contracts/LRTOracle.sol L270-282
if (newRsETHPrice < highestRsethPrice) {
    uint256 diff = highestRsethPrice - newRsETHPrice;
    bool isPriceDecreaseOffLimit =
        pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);

    if (isPriceDecreaseOffLimit) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;
    }
```

The oracle contract itself must hold `PAUSER_ROLE` on `LRTDepositPool` and `LRTWithdrawalManager` for this call chain to succeed — which is the intended deployment configuration given the feature's design. The internal `_pause()` call at line 280 bypasses the `onlyRole(LRTConstants.PAUSER_ROLE)` guard on the public `pause()` entry point (L138), so the oracle pauses itself without any role check on the caller.

`highestRsethPrice` is the all-time-high price stored on-chain and never decreases. Any transient dip in underlying LST oracle rates that causes `newRsETHPrice` to fall more than `pricePercentageLimit` below `highestRsethPrice` satisfies the condition. The attacker's only cost is gas.

Unpausing requires three separate `onlyLRTAdmin` transactions:
- `LRTOracle.unpause()` — `onlyLRTAdmin` (L143)
- `LRTDepositPool.unpause()` — `onlyLRTAdmin` (L354)
- `LRTWithdrawalManager.unpause()` — `onlyLRTAdmin`

While the oracle is paused, `updateRSETHPrice()` reverts with `ContractPaused`, so the price cannot be refreshed until the admin acts.

## Impact Explanation

Both deposits (`LRTDepositPool.depositAsset`, `depositETH` — guarded by `whenNotPaused`) and withdrawals (`LRTWithdrawalManager` — guarded by `whenNotPaused`) are frozen simultaneously. Users cannot enter or exit the protocol until the admin manually issues three separate unpause calls. This is a **temporary freezing of funds**, matching the allowed Medium impact scope.

## Likelihood Explanation

`updateRSETHPrice()` is callable by any EOA or contract with no preconditions beyond the price threshold being crossed. LST oracle rates (stETH/ETH, cbETH/ETH, etc.) fluctuate intra-day. A transient dip crossing the configured `pricePercentageLimit` — or a deliberate call timed to a momentarily unfavorable block — is sufficient. `pricePercentageLimit` is a non-zero admin-configured value (its purpose is downside protection, so it is expected to be set in production). No capital, no special role, and no prior state is required. The condition is repeatable: after the admin unpauses, the same attack can be executed again at the next transient dip.

## Recommendation

1. **Restrict the public entry point.** Gate `updateRSETHPrice()` to `onlyLRTManager` (or a keeper role), matching the existing `updateRSETHPriceAsManager()` pattern. The manager variant already exists precisely to handle edge cases; the unrestricted public variant adds no user-facing value that justifies the risk.
2. **Separate the pause trigger from the price-update path.** Emit an event and let an off-chain keeper or a dedicated `pauseIfPriceOffLimit()` function (still restricted) handle the pause decision, rather than embedding it inside the price-update hot path.
3. **Provide a single `unpauseAll()` admin function** so recovery is a single atomic transaction rather than three sequential ones.

## Proof of Concept

1. Admin sets `pricePercentageLimit = 1e16` (1%) and `highestRsethPrice` is recorded at `1.05 ether`.
2. A transient stETH oracle dip causes `_getTotalEthInProtocol()` to return a value pricing rsETH at `1.034 ether` (~1.5% drop from peak).
3. Attacker calls `LRTOracle.updateRSETHPrice()` (no role required).
4. `diff = 1.05e18 - 1.034e18 = 0.016e18`; `pricePercentageLimit.mulWad(1.05e18) = 0.0105e18`; `0.016e18 > 0.0105e18` → `isPriceDecreaseOffLimit = true`.
5. `LRTDepositPool.pause()`, `LRTWithdrawalManager.pause()`, and `LRTOracle._pause()` are all called in the same transaction.
6. All user deposits and withdrawals revert. The oracle cannot be updated. Admin must issue three separate `unpause()` calls to restore the protocol.

**Foundry fork test outline:**
```solidity
function test_anyCallerCanPauseAll() public {
    // fork mainnet, set pricePercentageLimit to 1e16
    // manipulate LST oracle to return a price 1.5% below highestRsethPrice
    vm.prank(attacker); // unprivileged EOA
    lrtOracle.updateRSETHPrice();
    assertTrue(lrtDepositPool.paused());
    assertTrue(lrtWithdrawalManager.paused());
    assertTrue(lrtOracle.paused());
    // verify deposits revert
    vm.expectRevert();
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
}
