### Title
`DelegateAction` Signing Payload Omits Chain ID, Enabling Cross-Chain Signature Replay - (File: `core/primitives/src/action/delegate.rs`)

### Summary
The `DelegateAction` (NEP-366 meta transaction) signing hash is computed without any chain-specific identifier. The fixed `SignableMessage` discriminant (`(1<<30) + 366`) is identical on every NEAR-based chain. An unprivileged attacker who observes a valid `SignedDelegateAction` on one chain (e.g., mainnet) can replay it verbatim on any other NEAR chain (e.g., testnet) where the same account and key exist with a valid nonce, causing the inner actions to execute without the user's authorization on the target chain.

### Finding Description
`DelegateAction::get_nep461_hash()` constructs the signed payload as:

```
SHA256( borsh(MessageDiscriminant) || borsh(DelegateAction) )
``` [1](#0-0) 

The `MessageDiscriminant` is a compile-time constant:

```
discriminant = MIN_ON_CHAIN_DISCRIMINANT + NEP_366_META_TRANSACTIONS
             = (1 << 30) + 366
             = 1_073_742_190   (fixed, chain-agnostic)
``` [2](#0-1) 

The `DelegateAction` struct itself contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — but **no `chain_id`**: [3](#0-2) 

Because neither the discriminant nor the payload encodes a chain identifier, the same `(DelegateAction, Signature)` pair is cryptographically valid on every NEAR chain that shares the same protocol.

By contrast, regular `SignedTransaction` includes `block_hash` in its signed body, which is chain-specific and prevents cross-chain replay: [4](#0-3) 

The `SignedDelegateAction::verify()` path that the runtime calls during execution only checks the NEP-461 hash against the embedded public key — it has no awareness of which chain the signature was produced for: [5](#0-4) 

### Impact Explanation
An attacker who observes a `SignedDelegateAction` broadcast on chain A can submit it to chain B via the public JSON-RPC `broadcast_tx_async` / `broadcast_tx_commit` endpoint. If the same account exists on chain B with the same access key and a nonce that satisfies the delegate action's nonce field, the runtime on chain B will:

1. Accept the signature as valid (the hash is identical on both chains).
2. Advance the access-key nonce on chain B.
3. Execute all inner actions (token transfers, `AddKey`, `DeleteKey`, `FunctionCall`, etc.) with `sender_id` as the predecessor.

Concrete corrupted protocol values: the access-key nonce stored in the trie on chain B, and any balance / key-set / contract-state changes caused by the inner actions.

### Likelihood Explanation
- NEAR accounts with identical names and key pairs routinely exist on both mainnet and testnet (developers, dApps, wallets).
- `SignedDelegateAction` objects are transmitted off-chain to relayers and are visible in submitted transactions, making interception trivial.
- `max_block_height` is a block height, not a hash; block heights on mainnet and testnet are in the same numeric range, so an action signed for mainnet block height H is also valid on testnet if the testnet chain has not yet reached H.
- The nonce constraint is the only remaining barrier, and it is satisfied whenever the target chain's access-key nonce is lower than the delegate action's nonce — a common situation when an account is more active on one chain than the other.

### Recommendation
Include the chain ID in the `DelegateAction` signing payload. The simplest approach is to add a `chain_id: String` field to `DelegateAction` (and `DelegateActionV2`) and incorporate it into `get_nep461_hash()`. Alternatively, the `SignableMessage` discriminant scheme could be extended to encode the chain ID as part of the prefix, analogous to how EIP-712 domain separators bind signatures to a specific `chainId`.

### Proof of Concept
1. Alice signs a `DelegateAction` on mainnet to transfer 100 NEAR to Bob:
   ```
   DelegateAction {
     sender_id: "alice.near",
     receiver_id: "bob.near",
     actions: [Transfer { deposit: 100 NEAR }],
     nonce: 5,
     max_block_height: 200_000_000,
     public_key: alice_key,
   }
   ```
2. The relayer submits the outer transaction on mainnet; the signed delegate action is now public.
3. The attacker extracts the `SignedDelegateAction` bytes from the mainnet transaction.
4. The attacker submits an outer transaction on testnet (where `alice.near` exists with the same key and nonce < 5) wrapping the identical `SignedDelegateAction`.
5. `SignedDelegateAction::verify()` on testnet computes `SHA256((1<<30)+366 || borsh(DelegateAction))` — identical to the mainnet hash — and returns `true`.
6. The runtime on testnet executes the transfer, draining 100 NEAR from Alice's testnet account without her authorization.

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
