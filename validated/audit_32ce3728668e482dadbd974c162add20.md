Audit Report

## Title
Missing Slippage Protection on `deposit()` Functions in L2 RSETHPool Contracts - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV2.sol`)

## Summary
All L2 liquidity pool `deposit()` functions compute the minted `wrsETH`/`rsETH` amount using a live oracle rate fetched at execution time, but accept no `minRSETHAmountOut` parameter. Because the oracle rate is updated asynchronously via LayerZero messages (or by a MANAGER role via `InterimRSETHOracle`), a user's pending transaction may execute at a rate that has increased since submission, delivering fewer tokens than the user observed. The L1 `LRTDepositPool` enforces a `minRSETHAmountExpected` check for the same operation; the L2 pools have no equivalent protection.

## Finding Description
In every L2 pool variant, the minted amount is computed as:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

where `rsETHToETHrate = getRate()` reads from the oracle at execution time. The affected functions are:

- `RSETHPoolV3.deposit(string)` (L246–265) and `RSETHPoolV3.deposit(address,uint256,string)` (L271–293)
- `RSETHPoolV3ExternalBridge.deposit(string)` (L366–384)
- `RSETHPoolV3WithNativeChainBridge.deposit(string)` (L282–301)
- `RSETHPoolNoWrapper.deposit(string)` (L231–244)
- `RSETHPoolV2.deposit(string)` (L207–219)

None accept or enforce a minimum output amount. The oracle rate is updated via `CrossChainRateReceiver.lzReceive()` (L82–100), which sets `rate = _rate` upon receiving a LayerZero message, or via `InterimRSETHOracle.setRate()` (L36–44), callable by any MANAGER role at any time. Both update paths are asynchronous relative to user transactions.

By contrast, `LRTDepositPool.depositETH()` (L76–93) accepts `minRSETHAmountExpected` and enforces it in `_beforeDeposit()` (L665–669):

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

The L2 pools have no equivalent guard. The only existing check is `if (amount == 0) revert InvalidAmount()`, which does not protect against rate-change slippage.

**Exploit path:**
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` — oracle rate is `1.05e18`, expected output `~0.952 wrsETH`.
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the tx is mined, a LayerZero message updates the oracle rate to `1.06e18`.
4. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.06e18 ≈ 0.943 wrsETH`.
5. User receives `~0.943 wrsETH` (~0.9% fewer tokens) with no on-chain revert.

No attacker action is required; this is a structural consequence of asynchronous rate propagation.

## Impact Explanation
The user's deposited ETH value is preserved — they receive fewer tokens, but each token is worth proportionally more ETH at the higher rate. However, the contract fails to deliver the token quantity the user was quoted at submission time, with no on-chain recourse. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The rsETH/ETH rate increases monotonically as EigenLayer staking rewards accrue. Rate updates are pushed periodically from L1 to L2 via LayerZero as a normal protocol operation. Any user whose deposit transaction is pending in the mempool at the moment a rate update lands will receive fewer tokens than the UI quoted. No attacker is required; this is a recurring, structural condition affecting all depositors.

## Recommendation
Add a `minRSETHAmountOut` parameter to all `deposit()` overloads in every L2 pool contract and revert if the computed amount falls below it:

```diff
- function deposit(string memory referralId) external payable nonReentrant whenNotPaused {
+ function deposit(string memory referralId, uint256 minRSETHAmountOut) external payable nonReentrant whenNotPaused {
      ...
      (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
+     if (rsETHAmount < minRSETHAmountOut) revert InsufficientOutputAmount();
      ...
  }
```

Apply the same pattern to all token `deposit(address,uint256,string)` overloads. The UI should pre-compute expected output via `viewSwapRsETHAmountAndFee` and pass a user-configurable slippage tolerance as `minRSETHAmountOut`.

## Proof of Concept
**Foundry fork test outline:**

```solidity
function test_depositSlippage() public {
    // 1. Fork L2 with RSETHPoolV3 deployed
    // 2. Record oracle rate R = pool.getRate()
    // 3. Compute expected = viewSwapRsETHAmountAndFee(1 ether).rsETHAmount
    // 4. Simulate LayerZero rate update: set rate to R * 1.01e18 / 1e18
    // 5. Call pool.deposit{value: 1 ether}("ref") as user
    // 6. Assert wrsETH.balanceOf(user) < expected
    // 7. Assert no revert occurred
}
```

The test will confirm that the user receives fewer tokens than the pre-submission quote with no on-chain protection, reproducing the described impact without any privileged or attacker action.
