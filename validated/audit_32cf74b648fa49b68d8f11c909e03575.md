### Title
Attacker-Controlled `directDeposits` Can Permanently Prevent Stake Key Deregistration via Non-Zero Balance Check - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

In the Dijkstra era, any unprivileged transaction sender can include a `directDepositsTxBodyL` field in a transaction body to credit ADA directly to any registered stake credential's account balance. The stake key deregistration rule (`ConwayUnRegCert` / `UnRegTxCert`) strictly requires the account's reward balance to be zero before allowing deregistration. An attacker can send 1 lovelace via a direct deposit to a victim's account, making the balance non-zero and blocking deregistration until the victim withdraws the balance â€” a process the attacker can continuously front-run.

---

### Finding Description

The Dijkstra era introduces `directDepositsTxBodyL`, a new transaction body field that allows any transaction sender to credit ADA to registered stake credential accounts. In `dijkstraEntitiesTransition`, after certificates are processed, direct deposits are validated and applied unconditionally to any registered account:

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

`applyDirectDeposits` adds the specified coin to `balanceAccountStateL` of the target account with no access control â€” any registered account can be targeted by any sender.

The Conway DELEG rule (inherited by Dijkstra) enforces a strict zero-balance precondition for stake key deregistration (`ConwayUnRegCert`):

```haskell
checkStakeKeyHasZeroRewardBalance = do
    accountState <- mAccountState
    let balanceCompact = accountState ^. balanceAccountStateL
    guard (balanceCompact /= mempty)
    Just $ fromCompact balanceCompact
failOnJust
    checkStakeKeyHasZeroRewardBalance
    (injectFailure . StakeKeyHasNonZeroAccountBalanceDELEG)
```

An attacker submits a single Dijkstra transaction with `directDeposits = { victimAccountAddress â†’ 1 lovelace }`. This credits 1 lovelace to the victim's account. The victim's subsequent deregistration transaction fails with `StakeKeyHasNonZeroAccountBalanceDELEG`. The attacker can repeat this each time the victim attempts to withdraw and deregister, continuously blocking the operation.

The Shelley-era design specification explicitly states that reward accounts "can not receive funds via transactions" â€” the direct deposit feature breaks this invariant and was not accounted for in the deregistration guard.

---

### Impact Explanation

The victim's stake key deposit (set by `ppKeyDepositL`, currently 2 ADA on mainnet) is locked for as long as the attacker continues to front-run the victim's withdrawal+deregistration transactions. The attacker can maintain this DoS at the cost of 1 lovelace + transaction fees per round. The victim's deposit refund is modified outside design parameters: the deregistration path was designed assuming only the epoch-boundary reward mechanism can make a balance non-zero, but direct deposits break this assumption.

This matches the **Medium** allowed impact: "Attacker-controlled transactionsâ€¦ modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."

---

### Likelihood Explanation

- Requires only a single Dijkstra-era transaction with a `directDeposits` entry targeting any registered account address.
- No privileged access, governance majority, or leaked keys are required.
- The attacker's cost per round is 1 lovelace + transaction fees (~0.2 ADA).
- The victim's account address is publicly observable on-chain.
- The attack is reachable from the first Dijkstra block onward.

---

### Recommendation

The `checkStakeKeyHasZeroRewardBalance` guard in the DELEG rule should be removed or replaced. The original rationale for requiring a zero balance before deregistration was to prevent loss of rewards (the user must explicitly withdraw before deregistering). With direct deposits, this check can be weaponized. The correct fix is to allow deregistration regardless of balance, crediting any remaining balance to the transaction outputs (similar to how the deposit refund is handled), or to allow a combined withdraw-and-deregister operation atomically without the zero-balance precondition.

---

### Proof of Concept

1. Victim registers stake credential `cred` (balance = 0, deposit = 2 ADA).
2. Attacker constructs a Dijkstra transaction:
   - Inputs: any UTxO owned by attacker
   - `directDeposits = { AccountAddress Mainnet (AccountId cred) â†’ Coin 1 }`
   - Outputs: change back to attacker minus 1 lovelace + fees
3. Attacker submits the transaction. `applyDirectDeposits` sets victim's `balanceAccountStateL` to `CompactCoin 1`.
4. Victim submits `ConwayUnRegCert cred SNothing`. The DELEG rule evaluates `checkStakeKeyHasZeroRewardBalance`: `balanceCompact = CompactCoin 1 â‰  mempty`, so `failOnJust` fires â†’ transaction rejected with `StakeKeyHasNonZeroAccountBalanceDELEG (Coin 1)`.
5. Victim attempts to withdraw 1 lovelace and deregister in the same transaction. Attacker front-runs with another direct deposit of 1 lovelace, making the balance 2 lovelace. Victim's withdrawal of 1 lovelace leaves balance = 1 lovelace, deregistration still fails.
6. Attacker repeats step 5 indefinitely, locking the victim's 2 ADA deposit.
