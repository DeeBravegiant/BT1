Audit Report

## Title
Missing ETH Deposit Amount in Limit Check Allows Deposit Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of the deposit-limit comparison, while correctly including it for ERC-20 assets. Any unprivileged depositor can push the protocol's total ETH holdings above the configured cap in a single `depositETH` call, violating the deposit-limit guarantee.

## Finding Description
The root cause is an asymmetric range check in `_checkIfDepositAmountExceedesCurrentLimit` (L676–682):

```solidity
// contracts/LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // `amount` absent
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

For ERC-20 assets the prospective `amount` is added to the running total before comparing against the cap. For native ETH, `amount` is silently dropped; only the pre-deposit total is tested. When `totalAssetDeposits == depositLimit` the function returns `false` (not exceeded) and the deposit proceeds.

The exploit path is fully permissionless:

- `depositETH` (L76–93) is `external payable` with no role restriction. [1](#0-0) 
- It calls `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected)`. [2](#0-1) 
- `_beforeDeposit` (L648–670) calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` and reverts only if it returns `true`. [3](#0-2) 
- Because `amount` is excluded from the ETH branch, the guard never fires for any deposit when `totalAssetDeposits <= depositLimit`. [4](#0-3) 

No existing check compensates for the missing `amount` in the ETH branch.

## Impact Explanation
The deposit limit is the protocol's primary on-chain risk-management gate for ETH exposure. Bypassing it allows total ETH under management to grow beyond the intended cap, causing the protocol to accept and mint rsETH for deposits it was designed to reject. No direct fund theft occurs from the bypass alone. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`depositETH` is permissionless and payable. Any depositor can read `getTotalAssetDeposits(ETH_TOKEN)` and `depositLimitByAsset(ETH_TOKEN)` on-chain and call `depositETH` with an arbitrarily large `msg.value` whenever `totalAssetDeposits <= depositLimit`. No special role, flash loan, or oracle manipulation is required. The condition is trivially observable and the attack is repeatable.

## Recommendation
Include the incoming `amount` in the ETH branch, matching the ERC-20 logic:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Deploy/fork with `depositLimitByAsset(ETH_TOKEN) = 1000 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 1000 ether` (exactly at cap).
2. Attacker calls `depositETH{value: 500 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false` → no revert.
4. `_mintRsETH` mints rsETH for the attacker; protocol now holds 1500 ETH against a 1000 ETH cap.
5. Foundry invariant test: assert `getTotalAssetDeposits(ETH_TOKEN) <= depositLimitByAsset(ETH_TOKEN)` after any sequence of `depositETH` calls — the invariant will break immediately.
