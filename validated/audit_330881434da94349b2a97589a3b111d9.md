### Title
Missing On-Chain Response Verification for `AppPublicKey` Variant in `respond_ckd` Enables Byzantine Participant to Deliver Fabricated CKD Output — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function performs cryptographic verification of the `CKDResponse` only for the `AppPublicKeyPV` variant. For the `AppPublicKey` variant, the contract performs **no verification whatsoever** of the response content. Any single attested participant (Byzantine, below threshold) can race the honest leader and call `respond_ckd` with an entirely fabricated `CKDResponse`, which the contract accepts and delivers to the user as the legitimate CKD output.

---

### Finding Description

In `respond_ckd` at lines 675–682 of `crates/contract/src/lib.rs`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` (in `crates/contract/src/primitives/ckd.rs`, lines 80–101) verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`. This is a binding cryptographic proof that the response is the correct MPC output. For `AppPublicKey`, the arm is an empty no-op — the contract accepts any `big_y` and `big_c` unconditionally.

After the match, `resolve_yields_for` is called unconditionally:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

`resolve_yields_for` removes the request from the map and resumes all queued yields with the supplied bytes in a single pass. The first caller wins; subsequent callers receive `RequestNotFound`. This means a Byzantine participant who calls `respond_ckd` before the honest leader does will have their fabricated response delivered to every waiting user.

The root cause is the same class as the Jet v2 "Position Collateral Zeroing" finding: a mistaken assumption that the MPC protocol's internal correctness is sufficient without an on-chain binding check. For `AppPublicKeyPV`, the user supplies a G2 component (`pk2`) that enables the pairing check. For `AppPublicKey`, only a G1 component is provided, and no equivalent on-chain binding exists — leaving the response entirely unverified.

The contrast with `respond` (lines 563–651) makes the gap explicit: `respond` verifies the ECDSA/EdDSA signature against the derived public key before accepting any response. A Byzantine node cannot forge a valid threshold signature without t+1 collusion. `respond_ckd` with `AppPublicKey` has no analogous gate.

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Query `get_pending_ckd_request` to obtain the `CKDRequest` key for any in-flight `AppPublicKey` request (the map is publicly readable).
2. Construct an arbitrary `CKDResponse`, e.g. `big_y = G1 * attacker_secret` — a key the attacker controls.
3. Call `respond_ckd` with the fabricated response before the honest leader does.
4. The contract accepts it, removes the pending entry, and resumes all queued yields with the attacker's bytes.

The user receives a derived key that the attacker controls. Any funds or data protected by that key are accessible to the attacker. This is unauthorized confidential key derivation output delivered without the required threshold of participant authorization.

---

### Likelihood Explanation

Medium-High. The attack requires only:

- Being a single attested participant (a legitimate but below-threshold role; the system explicitly tolerates Byzantine participants below threshold).
- Reading the publicly queryable pending request map.
- Submitting a transaction before the honest leader.

No threshold collusion, no key leakage, and no privileged operator access are required. A Byzantine node that is itself the elected leader for a request has no race to win at all.

---

### Recommendation

Add a binding on-chain verification for the `AppPublicKey` variant analogous to `ckd_output_check`. If the G1-only key does not admit a pairing-based proof, require callers to also supply a G2 component (as `AppPublicKeyPV` does), or deprecate `AppPublicKey` in favor of `AppPublicKeyPV`. At minimum, document that `AppPublicKey` responses are unverified and gate the variant behind an explicit governance flag until a verification path exists.

---

### Proof of Concept

```
1. Alice calls request_app_private_key({ app_public_key: AppPublicKey(g1_pk), ... })
   → CKDRequest stored in pending_ckd_requests

2. Byzantine attested participant B queries get_pending_ckd_request(ckd_request)
   → confirms the request is pending

3. B calls respond_ckd(ckd_request, CKDResponse { big_y: B_controlled_key, big_c: arbitrary })
   → match arm for AppPublicKey is a no-op; no verification runs
   → resolve_yields_for removes the entry and resumes Alice's yield with B's bytes

4. Alice's transaction resolves with big_y = B_controlled_key
   → Alice uses this key believing it is her MPC-derived key
   → B can sign/decrypt anything protected by that key
```

This is the direct analog of the Jet v2 `update_position_balance_handler` vulnerability: just as that handler accepted any `TokenAccount` sharing the right mint without verifying it was the registered account — zeroing the position — `respond_ckd` accepts any `CKDResponse` for an `AppPublicKey` request without verifying it is the correct MPC output, delivering an attacker-controlled key to the user. pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
