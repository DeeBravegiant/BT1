Looking at the actual implementation of `attach_private_scalar` in `src/cbmpc/api/schnorr2pc.cpp` and comparing it to the `schnorr_mp` counterpart, I can trace the exact validation gap.

**Key observation in `schnorr_2p::attach_private_scalar`:**

The function only checks `x * G == Qi_self` (the caller-supplied `public_share_compressed`), but never checks that `Qi_self` matches any Qi stored in the public blob itself. [1](#0-0) 

The `key_blob_v1_t` for schnorr_2p stores only `Q_compressed` (the global public key) and `x_share` — it has no per-party Qi field at all: [2](#0-1) 

So there is nothing in the blob to bind `public_share_compressed` to session A. The function accepts any valid `(x_B, Qi_B)` pair where `x_B * G == Qi_B`, regardless of which session's public blob is provided.

**Contrast with `schnorr_mp::attach_private_scalar`**, which explicitly enforces session binding: [3](#0-2) 

The MP version looks up `Qi_self` from the blob's own `Qis_compressed` map and rejects any mismatch. The 2P version has no such map and no such check.

**What happens at `sign` with the cross-session blob:**

The sign protocol uses `key.Q` (Q_A from session A) for the BIP340 challenge hash and `key.x_share` (x_B from session B) for the partial signature: [4](#0-3) [5](#0-4) 

P2 (honest, using session A's blob) contributes `s2 = e * x_A_2 + k2`. P1 (using cross-session blob) computes `s = e * x_B_1 + k1 + s2`. The combined scalar is `e*(x_B_1 + x_A_2)`, but Q_A = `(x_A_1 + x_A_2)*G`. The final BIP340 verify fails: [6](#0-5) 

The interactive protocol runs to completion before the failure is detected.

### Title
Missing Qi-to-blob session binding in `schnorr_2p::attach_private_scalar` allows cross-session key blob acceptance — (`src/cbmpc/api/schnorr2pc.cpp`)

### Summary
`schnorr_2p::attach_private_scalar` validates only that `x * G == public_share_compressed` (the caller-supplied Qi), but never verifies that the supplied Qi belongs to the session encoded in `public_key_blob`. An attacker who controls two DKG sessions can mix session A's global public key blob with session B's scalar and Qi, obtaining a SUCCESS return and a structurally valid but cryptographically invalid key blob.

### Finding Description
The `key_blob_v1_t` for schnorr_2p contains only `{version, role, curve, Q_compressed, x_share}` — no per-party Qi field. When `attach_private_scalar` is called, it:
1. Parses `public_key_blob` to extract `Q_A` (session A's global public key).
2. Parses the caller-supplied `public_share_compressed` as `Qi_self` (session B's share point).
3. Checks `x_B * G == Qi_B` — passes, since both come from session B.
4. Writes `x_B` into the blob alongside `Q_A` and returns SUCCESS.

There is no check that `Qi_B` is consistent with `Q_A`. The `schnorr_mp` counterpart performs this check explicitly by looking up the stored Qi from the blob's own map and rejecting any mismatch. The 2P version has no equivalent guard.

### Impact Explanation
The resulting blob is accepted as valid by `attach_private_scalar` and passes `deserialize_key_blob` (x_B is in range, Q_A is a valid curve point). It can be passed to `sign`, which runs the full interactive Schnorr-2PC protocol before failing at the final BIP340 verify. The accepted blob represents invalid cryptographic state: it claims to be a key share for Q_A but carries a scalar that is unrelated to Q_A. Any application layer that trusts `attach_private_scalar`'s SUCCESS return and stores or forwards the resulting blob will hold an invalid key blob that silently fails at signing time.

### Likelihood Explanation
Exploiting this requires access to two independent DKG sessions' outputs (both public blobs and one session's scalar). This is a realistic scenario in backup/restore workflows where a party manages multiple keys and uses PVE or similar schemes to back up scalars separately from public blobs.

### Recommendation
Add a session-binding check analogous to `schnorr_mp::attach_private_scalar`. Since the 2P blob does not store a per-party Qi, the fix requires either:
- Storing `Qi_self_compressed` in `key_blob_v1_t` (set during DKG/detach, checked during attach), or
- Verifying that `x * G` is consistent with `Q_compressed` in the blob (i.e., checking that `x * G` is a valid additive share of `Q` given the role — though this requires knowing the other party's Qi, which is not stored).

The simplest fix matching the MP pattern is to persist `Qi_self` in the public blob during `detach_private_scalar` and enforce `public_share_compressed == stored_Qi_self` in `attach_private_scalar`.

### Proof of Concept
```
// Session A: DKG → key_blob_A1 (P1), key_blob_A2 (P2)
// Session B: DKG → key_blob_B1 (P1), key_blob_B2 (P2)

// Detach P1's scalar from session B
detach_private_scalar(key_blob_B1, &pub_B1, &x_B1);

// Get Qi for P1 from session B
get_public_share_compressed(key_blob_B1, &Qi_B1);

// Detach public blob from session A
detach_private_scalar(key_blob_A1, &pub_A1, &x_A1_discarded);

// Cross-session attach: session A's public blob + session B's scalar/Qi
error_t rv = attach_private_scalar(pub_A1, x_B1, Qi_B1, &cross_blob);
assert(rv == SUCCESS);  // ← passes; no session binding check

// sign with cross_blob (P1) and key_blob_A2 (P2)
// → interactive protocol runs to completion
// → final BIP340 verify fails: e*(x_B1 + x_A2) ≠ e*x_A where x_A = x_A1 + x_A2
assert(sign(cross_blob, key_blob_A2, msg, &sig) != SUCCESS);
```
