Audit Report

## Title
Missing Deadline Check in `MetricOmmPoolLiquidityAdder` Allows Stale Liquidity Transactions to Execute at Unfavorable Pool States — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
All four public entry points in `MetricOmmPoolLiquidityAdder` (`addLiquidityExactShares` × 2, `addLiquidityWeighted` × 2) accept no `deadline` parameter and perform no timestamp check, unlike every swap entry point in `MetricOmmSimpleRouter` which calls `_checkDeadline` before touching the pool. A pending liquidity transaction can therefore execute arbitrarily late against a pool cursor position the LP never intended, and MEV bots can exploit this by sandwiching the liquidity addition within the LP's stated cursor-bounds window.

## Finding Description
`MetricOmmSwapRouterBase._checkDeadline` (L91–94) is called at the top of all four swap entry points in `MetricOmmSimpleRouter` (L68, L93, L131, L155). None of the four entry points in `MetricOmmPoolLiquidityAdder` contain any equivalent guard.

For `addLiquidityWeighted`, `_validateBinAndBinPosition` (L263–286) checks that the live cursor falls within `[minimalCurBin/minimalPosition, maximalCurBin/maximalPosition]` at the moment of the call. This is not equivalent to a deadline: the cursor can drift significantly *within* the stated bounds while the transaction is pending, causing the probe to return a composition the LP did not intend. A MEV bot can observe the pending transaction, swap the pool cursor to a position still inside the LP's bounds but maximally unfavorable for the LP's intended composition, allow the liquidity transaction to execute, then reverse the swap — a classic sandwich on a liquidity operation.

For `addLiquidityExactShares` (L56–81), there is no cursor-bounds check at all. The LP specifies exact per-bin shares; the token amounts owed are computed from the live pool state at execution time. A delayed transaction executes against whatever cursor position the pool has reached with no revert path.

The `maxAmountToken0` / `maxAmountToken1` caps (enforced in the callback at L165–166) prevent the LP from paying *more* than their stated maximums, but they do not prevent the LP from receiving a worse share-to-token ratio or from having their position concentrated at an unintended price range, increasing impermanent-loss exposure relative to what a timely execution would have produced.

## Impact Explanation
An LP submitting through `MetricOmmPoolLiquidityAdder` can have their transaction executed at a pool state materially different from the one they observed at submission time. The resulting position may be concentrated at a price range the LP did not intend, reducing the value of their LP claim relative to a timely execution. MEV bots can amplify this by sandwiching the liquidity addition within the cursor-bounds window. This constitutes a medium-severity direct degradation of LP assets (worse position quality / increased impermanent-loss exposure) reachable by any unprivileged LP using the standard periphery.

## Likelihood Explanation
Any liquidity transaction that spends time in the mempool — due to low gas, network congestion, or deliberate MEV delay — is affected. No special permissions or unusual conditions are required. The `addLiquidityWeighted` path is particularly exposed because its cursor-bounds window is typically wide enough to allow meaningful cursor drift. The `addLiquidityExactShares` path is exposed with no cursor guard whatsoever.

## Recommendation
Add a `uint256 deadline` parameter to all four public entry points in `MetricOmmPoolLiquidityAdder` and revert at the top of each function if `block.timestamp > deadline`, mirroring the pattern already established in `MetricOmmSwapRouterBase._checkDeadline`:

```diff
 function addLiquidityExactShares(
     address pool,
     address owner,
     uint80 salt,
     LiquidityDelta calldata deltas,
     uint256 maxAmountToken0,
     uint256 maxAmountToken1,
+    uint256 deadline,
     bytes calldata extensionData
 ) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
+    if (block.timestamp > deadline) revert IMetricOmmSimpleRouter.TransactionExpired(deadline, block.timestamp);
     _validateOwner(owner);
```

Apply the same change to the `msg.sender`-owner overload and both `addLiquidityWeighted` overloads. Update `IMetricOmmPoolLiquidityAdder` accordingly.

## Proof of Concept
1. LP calls `addLiquidityWeighted` with cursor bounds `[bin 0, bin 1]`, `maxAmountToken0 = X`, `maxAmountToken1 = Y`, targeting the current cursor at bin 0 / position 30%.
2. Transaction sits in the mempool (low gas price or deliberate MEV delay).
3. A MEV bot swaps the pool, moving the cursor to bin 1 / position 70% — still within the LP's stated bounds, so `_validateBinAndBinPosition` does not revert.
4. The LP's transaction executes. The probe returns a composition heavily weighted toward token1. Shares are scaled accordingly.
5. The LP's position is now concentrated at a price range they did not intend, with a token split they did not choose.
6. The MEV bot reverses its swap, profiting from the spread while the LP is left with a suboptimal position.

The same scenario applies to `addLiquidityExactShares` without even the cursor-bounds guard: the LP's exact shares are minted against whatever pool state exists at execution time, with no revert path if the cursor has moved far from the LP's intended target.
