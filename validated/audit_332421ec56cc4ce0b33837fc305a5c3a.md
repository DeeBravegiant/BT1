Audit Report

## Title
ETH Deposit Limit Bypass Due to Asymmetric Check in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies inconsistent comparison logic between ETH and ERC20 assets: the ETH branch omits the prospective `amount` from the limit check, using only a strict `>` against the current total. When `totalAssetDeposits == depositLimitByAsset(ETH_TOKEN)`, the check returns `false`, `_beforeDeposit` does not revert, and `depositETH` mints rsETH while pushing the running total above the configured cap.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
``` [1](#0-0) 

For ERC20 assets, `totalAssetDeposits + amount` is compared against the cap, correctly rejecting any deposit that would push the total over the limit. For ETH, `amount` is never used; the function only checks whether the total is **already strictly greater than** the limit. At the boundary condition `totalAssetDeposits == depositLimitByAsset(ETH_TOKEN)`, the expression `totalAssetDeposits > limit` evaluates to `false`, so `_beforeDeposit` does not revert: [2](#0-1) 

`depositETH` is a public payable function with no access control beyond `whenNotPaused` and `onlySupportedAsset`, so any unprivileged caller can trigger this path: [3](#0-2) 

`getTotalAssetDeposits` aggregates across the deposit pool, NDCs, EigenLayer strategies, the converter, and the unstaking vault, so the total is a protocol-wide figure: [4](#0-3) 

The deposit limit is stored in `LRTConfig.depositLimitByAsset` and is the protocol's primary per-asset supply cap: [5](#0-4) 

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The ETH deposit cap is the protocol's primary supply-control invariant for ETH. When `totalAssetDeposits` is exactly at the limit, any depositor can call `depositETH` with an arbitrary `msg.value`, receive freshly minted rsETH, and push the running total above the cap. This violates the invariant `getTotalAssetDeposits(ETH_TOKEN) ≤ depositLimitByAsset(ETH_TOKEN)` after every deposit and causes slightly more rsETH to be minted than the cap intends, diluting existing holders. No funds are directly stolen; the depositor provides real ETH.

## Likelihood Explanation
**Low-Medium.** The condition `totalAssetDeposits == depositLimitByAsset(ETH_TOKEN)` must hold at the moment of the call. This exact equality is transient but operationally reachable — particularly when an operator sets the limit to match the current total to cap further inflows, which is a common operational pattern. Once that condition holds, any unprivileged depositor can trigger it with a single `depositETH` call requiring no special privileges, flash loans, or oracle manipulation.

## Recommendation
Apply the same prospective check to ETH that is already applied to ERC20 tokens, eliminating the branch entirely:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This makes both paths consistent and ensures the cap is never exceeded regardless of the current total.

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 100 ether`.
2. Through normal deposits, `getTotalAssetDeposits(ETH_TOKEN)` reaches exactly `100 ether`.
3. Alice calls `depositETH(0, "ref")` with `msg.value = 1 ether`.
4. `_beforeDeposit(ETH_TOKEN, 1 ether, 0)` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)`.
5. Inside the function: `totalAssetDeposits = 100 ether`, `depositLimit = 100 ether`.
6. ETH branch: `100 ether > 100 ether` → `false` → limit not considered exceeded.
7. `_beforeDeposit` does not revert; `_mintRsETH` mints rsETH for Alice.
8. `getTotalAssetDeposits(ETH_TOKEN)` is now `101 ether`, exceeding the cap by `1 ether`.

**Foundry invariant test plan:**
```solidity
function invariant_ethDepositCapNeverExceeded() public {
    uint256 total = depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN);
    uint256 cap   = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN);
    assertLe(total, cap, "ETH deposit cap exceeded");
}
```
Run with a handler that calls `depositETH` with varying `msg.value` after setting the cap to the current total; the invariant will break on the current code and hold after the fix.
