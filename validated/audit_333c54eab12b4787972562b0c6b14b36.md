### Title
Dev TEE Measurements Included in Production `allowed_measurements` via Migration Seeding — (`crates/mpc-attestation/src/attestation.rs`, `crates/contract/src/tee/measurements.rs`)

### Summary
`default_measurements()` unconditionally compiles both production and dev TCB measurement sets into the binary. The contract's migration seeds the on-chain `AllowedMeasurements` from this function. Unless operators explicitly voted to remove the dev set post-migration, a node running dev firmware (different MRTD/RTMR0-2) passes `verify_any_measurements` and joins the production participant set.

### Finding Description
`default_measurements()` returns a static array of two `ExpectedMeasurements` entries — one from `assets/tcb_info.json` (production) and one from `assets/tcb_info_dev.json` (dev) — with no compile-time feature flag separating them. The contract's migration seeds `TeeState::allowed_measurements` from this function. The live verification path (`add_participant` → `get_accepted_measurements` → `to_attestation_measurements`) reads the on-chain `allowed_measurements` directly. If the dev entry was seeded and not subsequently removed by unanimous participant vote, any node whose MRTD/RTMR0-2 match the dev `tcb_info_dev.json` values passes attestation.

The MPC Docker image hash and launcher compose hash checks (`verify_dstack_mpc_hashes`) are orthogonal — they verify the application container, not the OS/firmware. An attacker running dev firmware with the production MPC Docker image passes all three checks.

### Impact Explanation
A node running unaudited dev firmware joins the production participant set, receives a valid `AcceptedAttestation`, and participates in threshold signing and key derivation. This is a participant/attestation authorization bypass. Impact: **High** — unauthorized participant in the signing set.

### Likelihood Explanation
Likelihood depends on whether the migration was executed and whether operators voted to remove dev measurements. The TODO comment (`#1433`) explicitly acknowledges this has not been done. The `AllowedMeasurements` comment states it is "seeded from `default_measurements()` on migration," making it probable the dev set is present in the live on-chain state.

### Recommendation
1. Gate `assets/tcb_info_dev.json` behind a non-default Cargo feature (e.g., `dev-measurements`) so production builds cannot compile it in.
2. Immediately vote to remove the dev `ExpectedMeasurements` entry from the on-chain `allowed_measurements` on the production contract.
3. Resolve TODO `#1433`.

### Proof of Concept
Call `get_allowed_measurements()` on the production contract. If the returned list contains two entries, one matching `assets/tcb_info_dev.json` values, the dev set is live. Then submit a `DstackAttestation` whose MRTD/RTMR0-2 match `tcb_info_dev.json` and whose MPC image hash matches any currently whitelisted production image hash — `add_participant` will return `Ok(AcceptedAttestation)`.

    fn verify_any_measurements(
        &self,
        report_data: &TDReport10,
        tcb_info: &TcbInfo,
        accepted_measurements: &[ExpectedMeasurements],
    ) -> Result<ExpectedMeasurements, VerificationError> {
        for expected in accepted_measurements {
            if self
                .verify_static_rtmrs(report_data, tcb_info, expected)
                .is_ok()
                && self
                    .verify_key_provider_digest(tcb_info, &expected.key_provider_event_digest)
                    .is_ok()
            {
                return Ok(*expected); // found a valid match
            }
        }

        Err(VerificationError::WrongHash {
            name: "expected_measurements",
            expected: "one of the embedded TCB info sets (prod or dev)".into(),
            found: "none matched".into(),
        })
    }
