Audit Report

## Title
Incorrect ETH Deposit Limit Check Omits Current Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison: the ETH branch checks `totalAssetDeposits > depositLimit` (omitting the incoming `amount`), while the ERC20 branch correctly checks `totalAssetDeposits + amount > depositLimit`. When `totalAssetDeposits == depositLimit`, the ETH check evaluates to `false` and the deposit proceeds, pushing ETH TVL past the configured cap and minting rsETH beyond the intended ceiling.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` ( [1](#0-0) ), the ETH branch at line 679 returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding `amount`. When `totalAssetDeposits` equals the deposit limit exactly, this evaluates to `false`, so `_beforeDeposit` ( [2](#0-1) ) does not revert with `MaximumDepositLimitReached`. The ERC20 branch at line 681 correctly includes `+ amount`, so the same boundary condition correctly reverts for ERC20 assets. The public entry point `depositETH` ( [3](#0-2) ) is payable, unrestricted, and routes through `_beforeDeposit` as its sole guard. No other check prevents the over-limit mint.

## Impact Explanation
The deposit limit is the primary mechanism bounding protocol ETH exposure. Bypassing it causes the contract to mint rsETH beyond the admin-configured ceiling, violating the invariant `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`. This maps to the allowed Low impact: **Contract fails to deliver promised returns** (deposit cap enforcement), with no direct theft of funds.

## Likelihood Explanation
The precondition `totalAssetDeposits == depositLimit` is reachable in normal operation as the cap is approached. Any depositor monitoring on-chain state can time a single `depositETH` call at that moment. No special role, flash loan, or multi-step setup is required. After the first bypass, `totalAssetDeposits > depositLimit` becomes `true` and blocks further deposits — but the cap has already been exceeded by one deposit amount.

## Recommendation
Remove the ETH-specific branch and apply the `+ amount` inclusion uniformly:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This unifies the check for both ETH and ERC20 and aligns with `getAssetCurrentLimit`, which already returns `0` when `totalAssetDeposits == depositLimit` ( [4](#0-3) ).

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Protocol accumulates `getTotalAssetDeposits(ETH) = 1000 ether` through normal deposits.
3. Attacker calls `depositETH{value: 10 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)` evaluates `1000 ether > 1000 ether` → `false` → no revert ( [5](#0-4) ).
5. `_mintRsETH` mints rsETH for the attacker; `getTotalAssetDeposits(ETH)` becomes `1010 ether`, exceeding the cap.
