Audit Report

## Title
`LRTConfig.addNewSupportedAsset` Creates Exploitable Intermediate State Before Oracle Is Set, Freezing rsETH Price Updates and Enabling Yield Dilution - (File: contracts/LRTConfig.sol / contracts/LRTOracle.sol)

## Summary
`LRTConfig.addNewSupportedAsset()` immediately appends a new asset to `supportedAssetList` without requiring a corresponding price oracle to be set in `LRTOracle`. Between this call and the subsequent `updatePriceOracleFor()` call, `LRTOracle.updateRSETHPrice()` reverts for all callers because `_getTotalEthInProtocol()` calls `getAssetPrice()` for every asset in `supportedAssetList`, and `getAssetPrice()` reverts via the `onlySupportedOracle` modifier when `assetPriceOracle[asset] == address(0)`. During this window, depositors of existing assets receive rsETH calculated against the stale stored `rsETHPrice`, diluting existing holders' unclaimed yield.

## Finding Description

**Root cause — no atomicity between asset registration and oracle assignment:**

`LRTConfig._addNewSupportedAsset()` (L106–118) sets `isSupportedAsset[asset] = true` and pushes the asset into `supportedAssetList` in a single transaction gated by `TIME_LOCK_ROLE`. No oracle is required at this point; `assetPriceOracle[newAsset]` in `LRTOracle` remains `address(0)`.

`LRTOracle.updatePriceOracleFor()` (L113–119) is a separate call gated by `DEFAULT_ADMIN_ROLE` — a structurally distinct role and actor from `TIME_LOCK_ROLE`.

**Broken invariant — `_getTotalEthInProtocol()` assumes all listed assets have oracles:**

`LRTOracle._getTotalEthInProtocol()` (L331–349) iterates over every entry returned by `lrtConfig.getSupportedAssetList()` and calls `getAssetPrice(asset)` (L339). `getAssetPrice()` (L156) carries the `onlySupportedOracle` modifier (L40–45):

```solidity
modifier onlySupportedOracle(address asset) {
    if (assetPriceOracle[asset] == address(0)) {
        revert AssetOracleNotSupported();
    }
    _;
}
```

Because `assetPriceOracle[newAsset] == address(0)`, the revert propagates through `_getTotalEthInProtocol()` → `_updateRsETHPrice()` (L231) → the public `updateRSETHPrice()` (L87–89). The function is callable by anyone but reverts for everyone until `updatePriceOracleFor(newAsset, oracle)` is executed.

**Consequence — stale `rsETHPrice` used for deposit minting:**

`LRTDepositPool.getRsETHAmountToMint()` (L506–521) computes:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
It reads the stored `rsETHPrice` state variable, which is only updated inside `_updateRsETHPrice()` (L313). While `updateRSETHPrice()` reverts, `rsETHPrice` remains at its last written value. If EigenLayer staking rewards have accrued since the last successful price update, `rsETHPrice` is understated. Any depositor of an existing supported asset (stETH, ETHx) during this window receives more rsETH than their fair share, diluting existing holders' accrued yield.

**Existing checks are insufficient:**

- `depositAsset()` has no freshness check on `rsETHPrice`.
- `updateRSETHPrice()` has no guard that skips assets with missing oracles; it reverts entirely.
- There is no `whenOracleSet` guard or similar mechanism to prevent deposits when the price is known to be stale.

## Impact Explanation

The concrete allowed impact is **High — Theft of unclaimed yield**. Existing rsETH holders' unclaimed yield (the appreciation of rsETH price from accrued EigenLayer rewards) is transferred to new depositors who receive excess rsETH minted at the stale, understated price. The excess rsETH permanently dilutes the per-token claim on protocol assets for all prior holders. Protocol fee minting is also blocked for the duration of the window, though fees on accumulated rewards are recaptured in the next successful `updateRSETHPrice()` call (making that component a temporary rather than permanent loss).

## Likelihood Explanation

Likelihood is **Low**. The window requires: (1) `TIME_LOCK_ROLE` executes `addNewSupportedAsset` without `DEFAULT_ADMIN_ROLE` atomically setting the oracle in the same block; (2) EigenLayer rewards have accrued since the last `rsETHPrice` update; (3) a depositor acts during the window. The structural role separation makes condition (1) a realistic operational pattern — the timelock may execute automatically while the admin must react manually. The `AddedNewSupportedAsset` event is observable on-chain, allowing a watcher to identify and exploit the window immediately.

## Recommendation

1. **Atomic onboarding:** Extend `addNewSupportedAsset` (or add a new function) to accept the oracle address as a parameter and call `LRTOracle.updatePriceOracleFor` in the same transaction, ensuring the asset is never in `supportedAssetList` without a valid oracle.
2. **Defensive iteration:** In `_getTotalEthInProtocol()`, skip assets whose `assetPriceOracle` is `address(0)` (or revert with a named error) rather than allowing the entire price update to revert silently.
3. **Deposit guard:** Add a staleness check in `_beforeDeposit` or `getRsETHAmountToMint` that reverts if `rsETHPrice` has not been updated within an acceptable window, preventing deposits against a known-stale price.

## Proof of Concept

```
Block N:   TIME_LOCK_ROLE calls LRTConfig.addNewSupportedAsset(newAsset, limit)
           → newAsset pushed to supportedAssetList (LRTConfig.sol L115)
           → assetPriceOracle[newAsset] == address(0) in LRTOracle

Block N+1: Anyone calls LRTOracle.updateRSETHPrice()
           → _updateRsETHPrice() (L214)
           → _getTotalEthInProtocol() (L231)
           → getAssetPrice(newAsset) (L339)
           → onlySupportedOracle: assetPriceOracle[newAsset] == address(0) → REVERT
           ← rsETHPrice remains stale (last written value)

Block N+1: Depositor calls LRTDepositPool.depositAsset(stETH, amount, minRSETH, "")
           → getRsETHAmountToMint uses stale rsETHPrice (L520)
           → depositor receives excess rsETH, diluting existing holders' yield

Block N+k: DEFAULT_ADMIN_ROLE calls LRTOracle.updatePriceOracleFor(newAsset, oracle)
           → assetPriceOracle[newAsset] set; updateRSETHPrice() works again
           → yield diluted during window is unrecoverable for prior holders
```

**Foundry fork test outline:**
1. Fork mainnet; deploy/configure protocol.
2. Call `addNewSupportedAsset(newAsset, limit)` as `TIME_LOCK_ROLE`; do NOT call `updatePriceOracleFor`.
3. Advance one block; assert `updateRSETHPrice()` reverts with `AssetOracleNotSupported`.
4. Record `rsETHPrice` (stale). Simulate EigenLayer reward accrual (increase strategy share value).
5. Deposit stETH as an unprivileged user; record rsETH minted.
6. Call `updatePriceOracleFor(newAsset, oracle)` as admin; call `updateRSETHPrice()`.
7. Assert the depositor's rsETH balance exceeds the fair-value amount computed at the post-update price, quantifying the dilution of existing holders.
