### Title
`ProposalInit.builder` is never validated during proposal validation, allowing a malicious proposer to embed an arbitrary `sequencer_address` in the committed block — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`ProposalInit` carries two distinct address fields: `proposer` (the consensus identity, validated against the committee) and `builder` (the address embedded as `sequencer_address` in the block, never validated). A legitimate but malicious proposer can set `builder` to any arbitrary address. All validator nodes will accept the proposal, execute the block with the attacker-controlled `sequencer_address`, and commit a block whose hash and fee-recipient address are both wrong.

---

### Finding Description

`ProposalInit` contains two address fields with different roles:

```rust
pub struct ProposalInit {
    pub proposer: ContractAddress,  // consensus identity — validated
    pub builder: ContractAddress,   // sequencer_address in block — never validated
    ...
}
``` [1](#0-0) 

The consensus layer validates `proposer` against the committee in `handle_proposal`:

```rust
if init.proposer != proposer_id {
    warn!("Invalid proposer: ...");
    return VecDeque::new();
}
``` [2](#0-1) 

However, `is_proposal_init_valid` — the full proposal-init validation function — checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, L1 gas prices, and `fee_proposal_fri`, but **never checks `builder`**: [3](#0-2) 

The `ProposalInitValidation` struct used to carry the local node's expected values also has no `builder` field: [4](#0-3) 

After passing validation, `initiate_validation` calls `convert_to_sn_api_block_info(init)`, which maps `init.builder` directly to `sequencer_address` in `BlockInfo`:

```rust
Ok(starknet_api::block::BlockInfo {
    ...
    sequencer_address: init.builder,
    ...
})
``` [5](#0-4) 

This `BlockInfo` is passed to the batcher for block execution. The `sequencer_address` is then included in `PartialBlockHashComponents` and the final block hash:

```rust
sequencer: SequencerContractAddress(block_info.sequencer_address),
``` [6](#0-5) 

```rust
.chain(&partial_block_hash_components.sequencer.0)
``` [7](#0-6) 

The `ProposalFin` commitment check at the end of `validate_proposal` does not save the invariant: both the proposer and all validators derive the commitment from the same `init.builder` value, so the check passes regardless of what `builder` contains: [8](#0-7) 

The TODO comment in the proposer's build path confirms this is a known gap:

```rust
// TODO(Asmaa): Get it from committee once we have it.
builder_address: self.config.static_config.builder_address,
``` [9](#0-8) 

---

### Impact Explanation

A malicious but legitimately-selected proposer can set `init.builder` to any address (e.g., an attacker-controlled wallet). Because no validator checks this field:

1. **Wrong `sequencer_address` committed to chain state**: The `sequencer_address` in the block header is the address that receives transaction fees in Starknet execution. Setting it to an attacker address redirects all fee revenue from that block.
2. **Wrong block hash**: `sequencer_address` is a direct input to `calculate_block_hash`. The committed block hash will differ from what an honest sequencer would have produced, corrupting the chain's authoritative hash for that height — affecting retrospective block hash verification, L1 anchoring, and any downstream proof.
3. **Wrong RPC/state view**: Any `starknet_getBlockWithTxHashes` or equivalent call will return the attacker-controlled `sequencer_address` as the authoritative sequencer for that block.

This matches the allowed impact: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input* (Critical) and *Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact* (Critical).

---

### Likelihood Explanation

The attacker must be a legitimately-selected proposer for the target height/round. This is a realistic scenario in a permissioned or semi-permissioned validator set where one validator is compromised or acts maliciously. The attack requires no special tooling — only modifying the `builder_address` field in the node's static config or patching the `initiate_build` function.

---

### Recommendation

Add a `builder` field to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`. Until the committee provides the canonical builder address, the validator should at minimum check that `init.builder` matches the locally-configured `builder_address`:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!("builder mismatch: expected={:?}, got={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder),
    ));
}
```

The `expected_builder` should be sourced from the committee (once available) or from the local node's own `builder_address` config as an interim measure. The TODO at `sequencer_consensus_context.rs:806` should be resolved before production deployment.

---

### Proof of Concept

1. A legitimate validator is selected as proposer for height H.
2. The validator modifies its node so that `builder_address` in static config is set to `attacker_wallet`.
3. The validator broadcasts `ProposalInit { proposer: legitimate_id, builder: attacker_wallet, ... }`.
4. All peer validators call `is_proposal_init_valid` — no check on `builder` — and proceed to `initiate_validation`.
5. `convert_to_sn_api_block_info` maps `attacker_wallet` → `sequencer_address` in `BlockInfo`.
6. The batcher executes all transactions with `sequencer_address = attacker_wallet`; all fees are credited to `attacker_wallet`.
7. The block hash is computed with `attacker_wallet` as the sequencer component.
8. The proposer sends `ProposalFin` with the commitment derived from `attacker_wallet`; validators compute the same commitment (same `init.builder`) → `built_block == received_fin.proposal_commitment` passes.
9. Consensus reaches decision; block with wrong `sequencer_address` and wrong block hash is committed to chain state.

#[instrument(level = "warn", skip_all, fields(?proposal_init_validation, ?init_proposed))]
async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.starknet_version != proposal_init_validation.starknet_version {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "starknet_version mismatch: expected={:?}, proposed={:?}",
                proposal_init_validation.starknet_version, init_proposed.starknet_version
            ),
        ));
    }
    // `version_constant_commitment` is proposer-supplied (network-derived). It is not yet a real
    // commitment (see `expected_version_constant_commitment`): the only valid value is the
    // sentinel, so reject anything else. Enforcing the same value the proposer emits keeps the two
    // sides in lockstep, so a real value cannot ship on one side without the other.
    let expected_commitment = expected_version_constant_commitment();
    if init_proposed.version_constant_commitment != expected_commitment {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "version_constant_commitment mismatch: expected={expected_commitment:?}, \
                 proposed={:?}",
                init_proposed.version_constant_commitment
            ),
        ));
    }
    if !(init_proposed.height == proposal_init_validation.height
        && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
        && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
    {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            "ProposalInit validation failed".to_string(),
        ));
