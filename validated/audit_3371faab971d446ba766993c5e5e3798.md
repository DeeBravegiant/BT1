Audit Report

## Title
No Staleness Check on Chainlink `latestRoundData()` Allows Stale Price Consumption - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` return value, applying no time-based freshness check. A stale Chainlink feed price flows directly into rsETH mint calculations (`LRTDepositPool.getRsETHAmountToMint`) and withdrawal calculations (`LRTWithdrawalManager.getExpectedAssetAmount`), enabling an unprivileged user to extract value from existing rsETH holders by depositing against an inflated stale price or withdrawing against a deflated stale price.

## Finding Description
In `ChainlinkPriceOracle.getAssetPrice()`, the five-tuple from `latestRoundData()` is destructured as `(, int256 price,,,)`, silently dropping `updatedAt`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

There is no `require(block.timestamp - updatedAt <= maxStaleness)` guard anywhere in the oracle path. The live (potentially stale) price is returned directly.

This price propagates to `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

And to `LRTWithdrawalManager.getExpectedAssetAmount()`:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [3](#0-2) 

Note that `lrtOracle.rsETHPrice()` is a stored state variable updated by `updateRSETHPrice()`, while `lrtOracle.getAssetPrice(asset)` fetches live from Chainlink on every call. [4](#0-3)  This asymmetry means a divergence between the stored rsETH price and a stale live asset price is exploitable.

`_getTotalEthInProtocol()` also iterates all supported assets and calls `getAssetPrice()` for each, meaning a stale feed also corrupts the TVL used to compute the next `rsETHPrice` update: [5](#0-4) 

The `unlockQueue` operator path does have `_validatePrices` bounds checks, but `initiateWithdrawal` and `depositAsset` — both callable by any unprivileged user — do not. [6](#0-5) 

## Impact Explanation
**High — Theft of unclaimed yield.**

Deposit path: if a feed is stale at a price above market (e.g., stETH/ETH stale at 1.001 when market is 0.998), `getRsETHAmountToMint` over-mints rsETH for the attacker, diluting all existing rsETH holders' share of protocol TVL. This constitutes theft of yield accrued by honest holders.

Withdrawal path: if a feed is stale at a price below market (e.g., stETH/ETH stale at 0.995 when market is 1.001), `getExpectedAssetAmount` returns more underlying tokens than the rsETH burned is worth, directly draining protocol reserves held for other withdrawers.

Both directions are reachable by any unprivileged external caller through public deposit and withdrawal functions.

## Likelihood Explanation
Chainlink LST/ETH feeds (stETH/ETH, cbETH/ETH, rETH/ETH) have heartbeats of 1–24 hours. Network congestion, Chainlink node outages, or L2 sequencer downtime are historically documented causes of missed heartbeats. No special privilege is required — any depositor or withdrawer can exploit the window between a feed going stale and an admin noticing and pausing. The condition is repeatable across all supported assets independently.

## Recommendation
Add a per-asset configurable `maxStaleness` mapping in `ChainlinkPriceOracle` and enforce it in `getAssetPrice()`:

```solidity
mapping(address asset => uint256 maxStaleness) public assetMaxStaleness;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
    require(block.timestamp - updatedAt <= assetMaxStaleness[asset], "Stale price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Set each asset's `maxStaleness` to slightly above its feed's documented heartbeat (e.g., `3600 + buffer` for 1-hour feeds, `86400 + buffer` for 24-hour feeds). Also validate that `price > 0` to guard against a zeroed-out feed.

## Proof of Concept
1. stETH/ETH Chainlink feed (1-hour heartbeat) has not updated for 3 hours due to network congestion. Last reported price: `1.001e18`. Current market price: `0.998e18`.
2. `rsETHPrice` was last updated when stETH was at `1.000e18` (stored value, e.g., `1.000e18`).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `_beforeDeposit` → `getRsETHAmountToMint(stETH, 1000e18)` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.001e18` with no revert.
5. `rsethAmountToMint = (1000e18 * 1.001e18) / 1.000e18 = 1001e18` — attacker receives 1 extra rsETH (~0.1% over fair value) at the expense of existing holders.
6. Attacker calls `LRTWithdrawalManager.initiateWithdrawal(stETH, 1001e18, "")` to redeem, extracting the excess value.

**Foundry fork test plan:** Fork mainnet, mock the stETH/ETH Chainlink aggregator to return a stale `updatedAt` (e.g., `block.timestamp - 4 hours`) with an inflated answer. Call `depositAsset` as an unprivileged address and assert that `rsethAmountToMint` exceeds the fair-value amount computed using the current on-chain market price. Assert no revert occurs.
