Audit Report

## Title
ETH Deposit Limit Check Ignores Deposit Amount, Allowing Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check for ETH deposits: the ETH branch evaluates only `totalAssetDeposits > limit`, omitting the incoming `amount`, while the ERC-20 branch correctly evaluates `totalAssetDeposits + amount > limit`. Any unprivileged depositor can therefore push the ETH total arbitrarily beyond the configured cap in a single transaction, as long as the running total has not yet crossed the limit at the time of the call.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding `amount`:

```solidity
// contracts/LRTDepositPool.sol L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount NOT added
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC-20
```

This function is the sole deposit-limit guard invoked from `_beforeDeposit` (L661–663), which is called by the public, permissionless `depositETH` entry point (L87). Because the ETH branch never adds `amount` to `totalAssetDeposits`, the guard returns `false` (no revert) for any deposit size as long as the current total is still below the limit. The full call chain is: `depositETH` → `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`. No other limit check exists in this path.

## Impact Explanation
The deposit limit is a protocol-level safety mechanism bounding total ETH exposure. The asymmetric check means the ETH cap is unenforceable: a single depositor can push the total from just below the limit to an arbitrarily large multiple of it in one transaction. No direct fund theft occurs, but the protocol fails to enforce its promised deposit constraint for ETH. This matches the allowed Low impact: **Contract fails to deliver promised returns / enforce promised constraints, but does not directly lose value.**

## Likelihood Explanation
`depositETH` is fully permissionless — no role, whitelist, or special condition is required. The only prerequisite is that `getTotalAssetDeposits(ETH_TOKEN)` has not yet crossed the limit, which is the normal operating condition for any protocol that has not yet filled its cap. The bug is trivially reachable by any depositor at any time before the cap is naturally hit, and is repeatable across any number of deposits.

## Recommendation
Add the deposit amount to the ETH branch, consistent with the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Assume `depositLimitByAsset(ETH_TOKEN) = 1_000 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.
2. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
3. Inside `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)`:
   - `totalAssetDeposits = 999 ether`
   - ETH branch evaluates `999 > 1000` → `false`
   - Returns `false` → no revert in `_beforeDeposit`
4. `_mintRsETH` executes; 10 000 ETH is accepted, bringing the total to 10 999 ETH — 11× the intended cap.
5. An identical deposit in any supported ERC-20 would correctly revert with `MaximumDepositLimitReached`.
