### Title
`respond_verify_foreign_tx` Does Not Verify `response.payload_hash` Is Derived from `request.request` ‚Äî (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract performs two independent checks ‚Äî it verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the domain's root public key, and that `request` matches a pending entry in `pending_verify_foreign_tx_requests` ‚Äî but it never verifies that `response.payload_hash` is actually the canonical hash of a `ForeignTxSignPayload` derived from `request.request`. This is the direct analog of H-09: individual fields are validated in isolation, but their required relationship is never enforced. A single malicious attested MPC participant can replay a legitimately-produced signature from a prior response for foreign-chain request Y as the response to a completely different pending request X, causing the bridge contract to receive a `payload_hash` that attests to the wrong foreign-chain data.

---

### Finding Description

In `crates/contract/src/lib.rs`, `respond_verify_foreign_tx` (lines 691‚Äì754) performs the following checks:

1. The caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`).
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's **root** public key (no tweak/derivation).
3. `request` exists as a key in `pending_verify_foreign_tx_requests`.

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,          // root key ‚Äî no derivation
)
.is_ok()
```

What is **never** checked is whether `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: <anything> })))`. The `payload_hash` field is a free parameter: any 32-byte value for which a valid root-key signature exists will pass.

The signing key used for foreign-tx responses is the **root** key with a zero tweak, as set in `build_signature_request`:

```rust
Ok(SignatureRequest {
    payload: Payload::Ecdsa(payload_bytes),   // = payload_hash
    tweak: Tweak::new([0u8; 32]),             // zero tweak ‚Üí root key
    ...
})
```

This means every `respond_verify_foreign_tx` response ever accepted on-chain carries a reusable root-key signature over its `payload_hash`. Any attested participant who observes a prior accepted response for request Y can extract `(payload_hash_Y, sig_Y)` and submit them verbatim as the response to a different pending request X for the same domain, because:

- `sig_Y` is valid over `payload_hash_Y` under the domain root key ‚úì
- Request X is pending ‚úì
- No check that `payload_hash_Y` encodes `request_X.request` ‚úó

`pending_requests::resolve_yields_for` then resolves request X's yield with `payload_hash_Y` and `sig_Y`, delivering the wrong attestation to the bridge contract.
