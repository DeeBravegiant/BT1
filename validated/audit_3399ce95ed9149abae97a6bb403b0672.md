### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Byzantine Participant - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_verify_foreign_tx` verifies the submitted signature against `response.payload_hash`, a value supplied entirely by the responding node. The contract never checks that `response.payload_hash` is the hash the MPC network was supposed to compute for the original pending request. A Byzantine participant that has obtained a valid threshold signature for one foreign-tx request can replay that `(payload_hash, signature)` pair to resolve a *different* pending request, delivering a forged verification result to every caller waiting on that request.

### Finding Description
In `respond_verify_foreign_tx` the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← comes from the response, not the request
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,                                            // root public key
)
.is_ok()
``` [1](#0-0) 

The contract only asserts: *"this signature is valid over this hash under the root key."* It does **not** assert: *"this hash is the one that should have been signed for the request identified by the `request` argument."*

Compare with `respond` for regular sign requests, where the payload is taken from the on-chain request, not from the response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(signature_response, payload_hash, &expected_public_key)
``` [2](#0-1) 

The `VerifyForeignTransactionRequest` stored on-chain contains the original `ForeignChainRpcRequest` (tx-id, extractors, etc.) and `payload_version`. The expected `payload_hash` is deterministically derivable from those fields plus the extracted values the MPC nodes observe on-chain. The contract never performs that derivation; it trusts whatever hash the responding node supplies. [3](#0-2) 

The SDK-level `ForeignChainSignatureVerifier::verify_signature` does perform this check client-side, but that is a caller-side helper, not an on-chain invariant enforced by the contract. [4](#0-3) 

### Impact Explanation
A Byzantine participant that has legitimately obtained a threshold signature `sig(H2)` for request B (foreign tx T2, payload hash H2) can call `respond_verify_foreign_tx` with:
- `request` = the pending key for request A (foreign tx T1)
- `response.payload_hash` = H2
- `response.signature` = `sig(H2)`

The contract accepts this because `sig(H2)` is valid under the root key for H2. It then drains the entire fan-out queue for request A, delivering `{payload_hash: H2, signature: sig(H2)}` to every caller waiting on T1. Those callers receive a cryptographically valid signature that attests to the *wrong* extracted values (those of T2, not T1). Any downstream bridge logic that trusts the contract's verification and uses the returned signature to authorise a cross-chain action will execute based on falsified foreign-chain data, enabling invalid bridge execution or double-spend conditions.

### Likelihood Explanation
The attacker must be an attested MPC participant (realistic: a single compromised or malicious node). It needs two concurrent pending `verify_foreign_transaction` requests and must have participated in the threshold signing session for one of them to obtain the final signature. Both conditions are routine in a live bridge environment with multiple concurrent users. No threshold collusion is required; a single Byzantine node below the signing threshold suffices.

### Recommendation
Derive the expected `payload_hash` on-chain from the stored request fields before accepting the response, or at minimum verify that `response.payload_hash` matches the hash the contract can independently compute from the `request` argument. Concretely, `respond_verify_foreign_tx` should compute:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),  // nodes must also submit extracted values
}).compute_msg_hash()?;
assert_eq!(expected_hash, response.payload_hash, "payload hash mismatch");
```

Alternatively, bind the `payload_hash` to the request at enqueue time (once the MPC nodes agree on extracted values) so the contract can compare against a stored commitment rather than a node-supplied value.

### Proof of Concept
1. Alice submits `verify_foreign_transaction` for Bitcoin tx T1 → pending request A is stored under key `K_A`.
2. Bob submits `verify_foreign_transaction` for Bitcoin tx T2 → pending request B is stored under key `K_B`.
3. The MPC network runs the threshold signing protocol for request B and produces `sig(H2)` where `H2 = SHA256(borsh(ForeignTxSignPayloadV1{request: T2, values: extracted_T2}))`.
4. The Byzantine node (a legitimate participant) now calls:
   ```
   respond_verify_foreign_tx(
       request = K_A,                          // T1's request key
       response = { payload_hash: H2,          // T2's hash
                    signature: sig(H2) }        // valid signature for T2
   )
   ```
5. The contract checks `verify_ecdsa_signature(sig(H2), H2, root_pk)` → passes.
6. `resolve_yields_for(&mut pending_verify_foreign_tx_requests, &K_A, response)` drains all yields for request A.
7. Alice's callback fires with `{payload_hash: H2, signature: sig(H2)}` — a valid signature over T2's extracted values, delivered as the verification result for T1.
8. Any bridge contract consuming this response and trusting the MPC network's attestation will act on falsified data. [5](#0-4) [6](#0-5)
