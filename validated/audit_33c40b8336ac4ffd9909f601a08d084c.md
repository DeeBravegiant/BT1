### Title
TEE Attestation Configuration Updates Do Not Re-Validate Active Participants — (File: `crates/contract/src/tee/tee_state.rs`)

---

### Summary

When the TEE attestation configuration is updated — specifically when a launcher image hash is removed via `vote_remove_launcher_hash` or an OS measurement set is removed via `vote_remove_os_measurement` — existing participants whose stored attestations are now invalid under the new configuration are **not immediately re-validated**. The `is_caller_an_attested_participant` function checks only for attestation *existence* and key-identity matching, never calling `reverify_participants`. These participants can continue to invoke critical contract methods (`respond`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`) until `verify_tee` is explicitly called by someone.

---

### Finding Description

The TEE state tracks three independently-updatable configuration sets:

- `allowed_docker_image_hashes` — updated by `vote_code_hash` / `whitelist_tee_proposal`
- `allowed_launcher_images` — updated by `vote_add_launcher_hash` / `vote_remove_launcher_hash`
- `allowed_measurements` — updated by `vote_add_os_measurement` / `vote_remove_os_measurement` [1](#0-0) 

When a launcher hash or OS measurement is removed (both require unanimity), the `stored_attestations` map is **not touched**. Every previously-accepted attestation that was tied to the now-removed hash/measurement remains in the map. [2](#0-1) [3](#0-2) 

The gate used on every sensitive node-facing method is `is_caller_an_attested_participant`, which only checks:

1. Is the signer in the participant set?
2. Does a stored attestation exist for the signer's TLS key?
3. Do the `account_id` and `account_public_key` fields match? [4](#0-3) 

It **never** calls `reverify_participants`, which is the function that actually re-checks the attestation against the current `allowed_docker_image_hashes`, `allowed_launcher_images`, and `allowed_measurements`. [5](#0-4) 

The only paths that trigger re-validation of existing participants are:

- `verify_tee` — must be called explicitly by a participant
- `reverify_and_cleanup_participants` — called inside `vote_new_parameters` and `verify_tee`
- `clean_invalid_attestations` — spawned as a detached promise after resharing completes, bounded to `max_scan = 100` entries [6](#0-5) [7](#0-6) 

None of these are triggered automatically when `remove_launcher_image` or `remove_measurement` executes. [2](#0-1) [3](#0-2) 

The `accept_requests` flag — which gates both `sign` and `respond` — is also **not** set to `false` when the configuration changes; it is only updated inside `verify_tee`. [8](#0-7) 

---

### Impact Explanation

After a launcher hash or OS measurement is removed (e.g., because a security vulnerability was found in that software version), nodes running the now-disallowed software retain their stored attestations. They can continue to:

- Call `respond` to submit threshold-signature contributions
- Call `vote_pk` / `vote_reshared` to advance key-generation or resharing state
- Call `start_keygen_instance` / `start_reshare_instance` as leader [9](#0-8) [10](#0-9) [11](#0-10) 

This means the TEE security guarantee — that only nodes running approved, attested software may participate in MPC operations — is **silently bypassed** for the window between the configuration change and the next explicit `verify_tee` call. If the removed measurement or launcher hash was revoked because the corresponding software is vulnerable (e.g., a key-extraction side-channel in the OS), the compromised nodes can still contribute signing shares during this window.

This maps to the allowed impact: **participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions** (High), or at minimum **contract execution-flow manipulation that breaks production safety/accounting invariants** (Medium).

---

### Likelihood Explanation

- `vote_remove_launcher_hash` and `vote_remove_os_measurement` both require **unanimity** (all participants), so the configuration change is a deliberate, coordinated governance action.
- After the vote succeeds, there is no automatic follow-up: `verify_tee` must be called manually by a participant.
- In a real incident (e.g., a disclosed TDX vulnerability), operators may remove the old measurement promptly but delay calling `verify_tee` due to operational complexity or gas constraints.
- The window can last for an arbitrary number of blocks, during which compromised nodes remain authorized signers.
- An adversary who controls a node running the revoked software (below the signing threshold) can exploit this window to continue contributing signing shares.

---

### Recommendation

When `remove_launcher_image` or `remove_measurement` executes (i.e., when the unanimity threshold is crossed in `vote_remove_launcher_hash` or `vote_remove_os_measurement`), immediately spawn a self-call to `verify_tee` — analogous to how `vote_reshared` spawns `CLEAN_TEE_STATUS` and `CLEAN_INVALID_ATTESTATIONS`:

```rust
// Inside vote_remove_launcher_hash / vote_remove_os_measurement,
// after the removal is applied:
Promise::new(env::current_account_id())
    .function_call(
        method_names::VERIFY_TEE.to_string(),
        vec![],
        NearToken::from_yoctonear(0),
        Gas::from_tgas(self.config.verify_tee_tera_gas),
    )
    .detach();
``` [12](#0-11) 

Alternatively, `is_caller_an_attested_participant` should be extended to call `reverify_participants` so that every sensitive entry point re-checks the caller's attestation against the current allowed sets, not just its existence. [4](#0-3) 

---

### Proof of Concept

1. The network is running with launcher hash `L1` and OS measurement `M1` in the allowed sets.
2. A security vulnerability is disclosed in the OS corresponding to `M1`.
3. All participants call `vote_remove_os_measurement(M1)`. Unanimity is reached; `remove_measurement(&M1)` executes and removes `M1` from `allowed_measurements`. [13](#0-12) [3](#0-2) 

4. Node `X` is running the vulnerable OS (measurement `M1`). Its `NodeAttestation` entry remains in `stored_attestations` because no code path removes it at this point. [14](#0-13) 

5. Node `X` calls `respond(request, signature)`. The contract calls `assert_caller_is_attested_participant_and_protocol_active`, which internally calls `is_caller_an_attested_participant`. This function finds `X`'s entry in `stored_attestations`, confirms the key match, and returns `Ok(())` — **without calling `reverify_participants`**. [4](#0-3) 

6. The `respond` call succeeds. Node `X`, running software whose OS measurement has been revoked, has contributed a signing share to a threshold signature — bypassing the TEE attestation requirement. [9](#0-8) 

7. This continues until an operator explicitly calls `verify_tee`, which triggers `reverify_and_cleanup_participants` and eventually sets `accept_requests = false` or initiates a resharing to exclude `X`. [15](#0-14)

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }

    pub fn verify_tee(&mut self) -> Result<bool, Error> {
        log!("verify_tee: signer={}", env::signer_account_id());
        // Caller must be a participant (node or operator).
        self.voter_or_panic();
        let ProtocolContractState::Running(running_state) = &mut self.protocol_state else {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        };
        let current_params = running_state.parameters.clone();

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        match self.tee_state.reverify_and_cleanup_participants(
            current_params.participants(),
            tee_upgrade_deadline_duration,
        ) {
            TeeValidationResult::Full => {
                self.accept_requests = true;
                log!("All participants have an accepted Tee status");
                Ok(true)
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let remaining = participants_with_valid_attestation.len();
                // Defense in depth: the surviving participant set must keep the full
                // threshold relation intact — the GovernanceThreshold must still sit
                // within its bounds for the smaller set (in particular it must not
                // exceed the remaining participant count or the upper cap) and must
                // remain at least every domain's ReconstructionThreshold (the kickout
                // keeps the existing per-domain thresholds). Otherwise we refuse and
                // wait for manual intervention.
                let max_reconstruction_threshold =
                    max_reconstruction_threshold(running_state.domains.domains());
                if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
                    u64::try_from(remaining).expect("participant count fits in u64"),
                    current_params.threshold(),
                    max_reconstruction_threshold,
                ) {
                    log!(
                        "Kicking out participants with an invalid TEE status would break the threshold relation ({:?}); {} participants remain with a valid TEE status. This requires manual intervention. We will not accept new signature requests as a safety precaution.",
                        err,
                        remaining,
                    );
                    self.accept_requests = false;
                    return Ok(false);
                }

                // here, we set it to true, because at this point, we have at least `threshold`
                // number of participants with an accepted Tee status.
                self.accept_requests = true;

                // do we want to adjust the threshold?
                //let n_participants_new = new_participants.len();
                //let new_threshold = (3 * n_participants_new + 4) / 5; // minimum 60%
                //let new_threshold = new_threshold.max(2); // but also minimum 2
                let new_threshold = usize::try_from(current_params.threshold().value())
                    .expect("threshold value fits in usize");

                let threshold_parameters = ThresholdParameters::new(
                    participants_with_valid_attestation,
                    Threshold::new(new_threshold as u64),
                )
                .expect("Require valid threshold parameters"); // this should never happen.
                current_params.validate_incoming_proposal(&threshold_parameters)?;
                // This resharing only changes the participant set, so the
                // per-domain reconstruction-threshold updates map is empty.
                let proposed_parameters =
                    ProposedThresholdParameters::new(threshold_parameters, BTreeMap::new());
                let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
                if let Some(resharing) = res {
                    self.protocol_state = ProtocolContractState::Resharing(resharing);
