### Title
Stale Participant Votes in `CodeHashesVotes`/`LauncherHashVotes` Persist After Resharing, Breaking Governance Threshold Invariant â€” (File: `crates/contract/src/tee/proposal.rs`)

---

### Summary

When participants are removed via resharing, their votes in `CodeHashesVotes` and `LauncherHashVotes` (stored in `TeeState`) are not cleaned up. The `count_votes()` method counts all stored votes without filtering by the current participant set, allowing stale votes from removed participants to contribute to reaching the governance threshold for MPC docker image hash and launcher hash approvals.

---

### Finding Description

`TeeState` contains two vote-tracking structures that persist across resharing epochs:

```rust
pub struct TeeState {
    pub(crate) votes: CodeHashesVotes,
    pub(crate) launcher_votes: LauncherHashVotes,
    ...
}
``` [1](#0-0) 

Both structures use a `BTreeMap<AuthenticatedParticipantId, ...>` to store votes. The `count_votes()` method in `CodeHashesVotes` counts **all** stored votes without filtering by current participants:

```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64
}
``` [2](#0-1) 

The same pattern applies to `LauncherHashVotes.count_votes()`: [3](#0-2) 

Both structures expose a `get_remaining_votes()` method that correctly filters to current participants, but this method is **not called** when resharing completes and the participant set changes. `TeeState` is a top-level field of `MpcContract` and is never reset during resharing: [4](#0-3) 

By contrast, the codebase already applies the correct pattern in two analogous places:

1. `add_domains_votes.get_remaining_votes()` **is** called in `RunningContractState::new()` to prune stale domain-add votes after resharing: [5](#0-4) 

2. `SupportedForeignChainsByNode` entries for removed participants are pruned by `clean_foreign_chain_data` after resharing: [6](#0-5) 

`CodeHashesVotes` and `LauncherHashVotes` have no equivalent post-resharing cleanup, making them the only governance vote stores that accumulate stale entries indefinitely.

---

### Impact Explanation

A removed participant's vote for a docker image hash or launcher hash persists in `proposal_by_account` / `vote_by_account` after resharing. When `count_votes()` is called during a subsequent `vote()` call by a current participant, the stale entry is counted toward the threshold. This means a governance decision â€” whitelisting a new MPC docker image or launcher image â€” can be finalized with fewer current participants than the governance threshold requires. Whitelisting an unauthorized docker image breaks the TEE security model by allowing nodes running unvetted (potentially malicious) software to be treated as valid attestation participants, which is a prerequisite for their involvement in threshold signing and key-share operations.

**Severity: Medium** â€” breaks the production safety invariant that threshold-many *current* participants must agree for governance decisions, without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

**High.** This condition is triggered every time resharing removes a participant who had previously cast a vote in `CodeHashesVotes` or `LauncherHashVotes`. The stale vote persists indefinitely (it is never evicted) and is counted in every subsequent `count_votes()` call for the same proposal hash. No special attacker capability is required beyond having been a legitimate participant at some prior epoch.

---

### Recommendation

After resharing completes and a new `RunningContractState` is instantiated, call `get_remaining_votes()` on both `TeeState.votes` and `TeeState.launcher_votes`, passing the new participant set, and store the filtered result back. This mirrors the existing pattern used for `add_domains_votes`:

```rust
// analogous to what RunningContractState::new() already does for add_domains_votes
self.tee_state.votes =
    self.tee_state.votes.get_remaining_votes(new_parameters.participants());
self.tee_state.launcher_votes =
    self.tee_state.launcher_votes.get_remaining_votes(new_parameters.participants());
``` [7](#0-6) [8](#0-7) 

---

### Proof of Concept

1. Network has 5 participants; governance threshold = 3.
2. Participant A (ID `X`) calls `vote_new_docker_image` for malicious image hash `H`. `TeeState.votes.proposal_by_account` now contains `{AuthenticatedParticipantId(X): H}`.
3. Resharing occurs; Participant A is removed from the participant set. `TeeState.votes` is **not** cleaned up â€” the entry for `X` remains.
4. Two current participants (B and C) call `vote_new_docker_image` for `H`.
5. Inside `CodeHashesVotes::vote()`, `count_votes(&H)` iterates all values in `proposal_by_account` and returns **3** (A's stale vote + B + C).
6. The caller in `lib.rs` sees count == threshold (3) and approves `H` â€” with only **2** current participants having agreed.
7. Malicious image `H` is now whitelisted, allowing nodes running it to submit valid-looking attestations. [9](#0-8)

            .iter()
            .filter(|(participant_id, _)| {
                participants.is_participant_given_participant_id(&participant_id.get())
            })
            .map(|(participant_id, vote)| (participant_id.clone(), *vote))
