Audit Report

## Title
Missing Zero-Amount Guard Before rsETH Burn in `instantWithdrawal` Allows Loss of User Funds - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.instantWithdrawal` burns the caller's rsETH unconditionally before verifying that `assetAmountUnlocked > 0`. When integer division in `getExpectedAssetAmount` truncates to zero — possible when an asset's oracle price exceeds `rsETHPrice` — the user's rsETH is permanently destroyed and they receive no underlying assets in return.

## Finding Description
The burn at L229 executes immediately after computing `assetAmountUnlocked`, with no guard on its value:

```solidity
// L228
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
// L229 — burn is unconditional
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
```

`getExpectedAssetAmount` computes (L593):
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

This truncates to zero whenever `rsETHUnstaked * rsETHPrice < assetPrice`. The only pre-burn guard (L224) is:
```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

`minRsEthAmountToWithdraw` is a plain mapping (L35) that defaults to `0` for any new asset, making the only enforced lower bound `rsETHUnstaked != 0`. A user passing `rsETHUnstaked = 1` when `assetPrice > rsETHPrice` will have their rsETH burned and receive `userAmount = 0`.

## Impact Explanation
The user's rsETH principal is permanently destroyed with zero asset received. This maps to **Low — Contract fails to deliver promised returns**. The loss per call is bounded by the integer-truncation threshold (at most a few wei of rsETH), but the mechanism is structurally broken: the contract burns value before confirming it can deliver the promised return.

## Likelihood Explanation
Under the current asset set (ETH, stETH, ETHx — all priced below rsETH), the truncation-to-zero condition is not reachable. It becomes reachable if: (1) a new LST or yield-bearing token with oracle price > `rsETHPrice` is added and enabled for instant withdrawal, AND (2) `minRsEthAmountToWithdraw[asset]` is left at its default of `0`. Both conditions are plausible during normal protocol expansion. An unprivileged user triggers the loss with a single `instantWithdrawal(asset, 1, "")` call.

## Recommendation
Add a zero-amount check on `assetAmountUnlocked` before executing the burn:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
if (assetAmountUnlocked == 0) revert InvalidAmountToWithdraw();
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
```

Additionally, enforce a non-zero `minRsEthAmountToWithdraw` for every asset before enabling instant withdrawal via `setInstantWithdrawalEnabled`, so that truncation to zero is structurally impossible regardless of price ratios.

## Proof of Concept
1. Admin adds a new LST with oracle price `2e18`. `rsETHPrice = 1.05e18`.
2. Admin calls `setInstantWithdrawalEnabled(asset, true)` but does not call `setMinRsEthAmountToWithdraw(asset, N)` — mapping stays at `0`.
3. User calls `instantWithdrawal(asset, 1, "")`.
4. Check at L224 passes: `1 != 0` and `1 >= 0`.
5. `assetAmountUnlocked = 1 * 1.05e18 / 2e18 = 0` (integer truncation).
6. `burnFrom(user, 1)` executes — 1 wei rsETH destroyed.
7. `unstakingVault.redeem(asset, 0)` — no-op.
8. `fee = 0`, `userAmount = 0`.
9. `_transferAsset(asset, user, 0)` — user receives nothing.
10. Net: user loses 1 wei rsETH, receives zero assets.

**Foundry test sketch:**
```solidity
function test_instantWithdrawal_zeroReturn() public {
    // Setup: asset oracle price = 2e18, rsETHPrice = 1.05e18
    // minRsEthAmountToWithdraw[asset] = 0 (default)
    // isInstantWithdrawalEnabled[asset] = true
    uint256 balanceBefore = rsETH.balanceOf(user);
    vm.prank(user);
    withdrawalManager.instantWithdrawal(asset, 1, "");
    assertEq(rsETH.balanceOf(user), balanceBefore - 1); // rsETH burned
    assertEq(IERC20(asset).balanceOf(user), 0);          // nothing received
}
```
