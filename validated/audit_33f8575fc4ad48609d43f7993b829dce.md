### Title
Unauthenticated P2P Peer Can Corrupt Stored State Diff via Forged `state_diff_length` in `SignedBlockHeader` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`, `header.rs`)

---

### Summary

The P2P sync client stores `SignedBlockHeader` messages from peers without verifying the block signature. The `state_diff_length` field, which controls how many state diff chunks are consumed and stored, is fully peer-controlled. A malicious peer can set `state_diff_length=K` (any small value), send K crafted `StateDiffChunk` entries, and cause the node to permanently store a truncated or fabricated `ThinStateDiff` for block N. No cryptographic check prevents this.

---

### Finding Description

**Step 1 — `state_diff_length` is peer-supplied and unverified.**

The `state_diff_length` is decoded directly from the peer's `StateDiffCommitment.state_diff_length` protobuf field: [1](#0-0) 

This field is part of the `StateDiffCommitment` proto message: [2](#0-1) 

**Step 2 — The header is stored without signature verification.**

`parse_data_for_block` in `header.rs` only checks that the block number matches and that exactly one signature is present. It does **not** call `verify_block_signature` or verify the signature cryptographically: [3](#0-2) 

`verify_block_signature` exists in `starknet_api/src/block.rs` but is never called anywhere in `crates/apollo_p2p_sync/`: [4](#0-3) 

The unverified header (including the peer-controlled `state_diff_length`) is written directly to storage: [5](#0-4) 

**Step 3 — State diff assembly uses the stored, peer-controlled `target_state_diff_len`.**

`parse_data_for_block` in `state_diff.rs` reads `target_state_diff_len` from the already-stored header: [6](#0-5) 

The loop collects chunks until `current_state_diff_len >= target_state_diff_len` and then exits: [7](#0-6) 

**Step 4 — No `state_diff_commitment` verification against actual content.**

After assembling the `ThinStateDiff`, there is no check that the assembled diff hashes to the `state_diff_commitment` stored in the header. The diff is written directly: [8](#0-7) 

**Step 5 — `BadPeerError` retry does not fix the corrupted header.**

When a `BadPeerError` occurs during state diff parsing, the peer is reported and the query is retried: [9](#0-8) 

However, the header is already stored with the forged `state_diff_length`. Any retry — even against an honest peer — will still read `target_state_diff_len=K` from storage and accept only K chunks, permanently storing the truncated diff.

---

### Impact Explanation

A syncing node that accepts a forged header stores a `ThinStateDiff` for block N that is missing entries for `storage_diffs`, `nonces`, and `deployed_contracts`. All subsequent RPC queries for state at block N (storage reads, nonce lookups, contract class lookups) return wrong values. The node's state root diverges from the canonical chain silently, with no error surfaced to operators or users.

---

### Likelihood Explanation

Any node participating in the P2P network can act as a peer. No special privileges are required. The attacker only needs to respond to a header query before an honest peer does. The attack is deterministic and requires no brute force.

---

### Recommendation

1. **Verify the block signature** in `HeaderStreamBuilder::parse_data_for_block` using `verify_block_signature` against the known sequencer public key before storing the header.
2. **Verify the state diff commitment** after assembling the `ThinStateDiff`: compute `calculate_state_diff_hash(&result)` and compare it to the `state_diff_commitment` stored in the header. Reject with `BadPeerError` on mismatch.
3. Until (1) is implemented, treat `state_diff_length` as advisory only and always verify the assembled diff against the commitment hash.

---

### Proof of Concept

```
1. Attacker peer responds to a HeaderQuery for block N.
2. Sends SignedBlockHeader {
       block_header: BlockHeader {
           block_number: N,
           state_diff_commitment: StateDiffCommitment {
               state_diff_length: 1,   // forged — real diff has M > 1 entries
               root: <any value>,
           },
           ...
       },
       signatures: [<any single signature>],
   }
3. Header is stored (no signature check). state_diff_length=1 is now in storage.
4. Attacker peer responds to the StateDiffQuery for block N.
5. Sends one StateDiffChunk::ContractDiff {
       contract_address: <target>,
       storage_diffs: { <slot>: <crafted_value> },
       ...
   }
6. Loop exits: current_state_diff_len(1) == target_state_diff_len(1). Check passes.
7. append_state_diff(N, ThinStateDiff { storage_diffs: {<target>: {<slot>: <crafted_value>}} })
   is committed to storage.
8. get_state_diff(N) now returns only that one entry.
   All other storage slots, nonces, and deployed contracts for block N are missing.
```
