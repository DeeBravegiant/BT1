Audit Report

## Title
Chainlink Price Feed Staleness Not Validated in `ChainlinkPriceOracle.getAssetPrice()`, Enabling Stale-Price rsETH Over-Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` discards all `latestRoundData()` return values except `answer`, performing no staleness, round-completeness, or sign checks. Because `LRTOracle.updateRSETHPrice()` is permissionless, an attacker can trigger a price update at the moment a feed is stale, skewing `rsETHPrice`, then immediately deposit the non-stale asset to receive more rsETH than the deposited value warrants, diluting existing holders' yield.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The full return signature exposes `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`, none of which are checked: [2](#0-1) 

The sibling contract `ChainlinkOracleForRSETHPoolCollateral` in the same repo performs all three checks, proving the team is aware of the pattern: [3](#0-2) 

The stale price propagates through `_getTotalEthInProtocol()`: [4](#0-3) 

Into `_updateRsETHPrice()`, which sets the on-chain `rsETHPrice`: [5](#0-4) 

`updateRSETHPrice()` is public with no access control: [6](#0-5) 

The deposit minting ratio in `LRTDepositPool.getRsETHAmountToMint()` is:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [7](#0-6) 

**Exploit mechanics (multi-asset case):** When the protocol holds multiple LSTs (e.g., stETH and cbETH) and only the stETH feed goes stale *low*:

- `rsETHPrice_stale` = `(stETH_amt × P_stale + cbETH_amt × P_cbETH) / rsethSupply` — reduced.
- An attacker deposits **cbETH** (whose price is fresh): `rsethAmountToMint = (A × P_cbETH) / rsETHPrice_stale` — the denominator is depressed while the numerator is unaffected, so the attacker receives more rsETH than the deposited cbETH value warrants.
- When the stETH feed updates to the true price, `rsETHPrice` rises, and the attacker's excess rsETH is worth more than deposited — a direct transfer of yield from existing holders.

The `pricePercentageLimit` guard only triggers on deviations exceeding the configured threshold; small but economically meaningful staleness (e.g., 0.3–0.5% within a 24-hour heartbeat window) passes through silently. [8](#0-7) 

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders accumulate yield as the protocol TVL grows relative to rsETH supply. When an attacker mints rsETH at a depressed `rsETHPrice` caused by a stale feed, they capture a disproportionate share of that accrued yield. Upon price correction, the attacker's rsETH is worth more than deposited, at the direct expense of pre-existing holders whose per-share value is diluted. This matches the allowed impact "Theft of unclaimed yield."

## Likelihood Explanation

**Medium.** Chainlink LST/ETH feeds carry heartbeats of 1–24 hours and deviation thresholds of 0.5–1%. During network congestion or Chainlink node issues, feeds routinely lag within their heartbeat without triggering any circuit breaker. The entry point (`updateRSETHPrice()`) is permissionless, so any depositor can trigger the price update at the moment most favorable to them after observing a stale feed on-chain. The attack is repeatable whenever a feed lags.

## Recommendation

Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too stale");

return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`MAX_STALENESS` should be stored per feed (e.g., in a parallel `assetPriceFeedStaleness` mapping) and set based on each feed's documented heartbeat plus a small buffer. This mirrors the pattern already implemented in `ChainlinkOracleForRSETHPoolCollateral`.

## Proof of Concept

**Setup:** Protocol holds 500 stETH and 500 cbETH; rsethSupply = 1000; true prices both 1.0 ETH; `pricePercentageLimit` = 1% (1e16).

1. stETH/ETH Chainlink feed has not updated for 6 hours (within its 24-hour heartbeat); reported price = 0.997 ETH (0.3% below true).
2. Attacker observes the stale feed on-chain.
3. Attacker calls `LRTOracle.updateRSETHPrice()` (permissionless).
   - `_getTotalEthInProtocol()` = 500×0.997 + 500×1.0 = 998.5 ETH.
   - `rsETHPrice_stale` = 998.5 / 1000 = 0.9985.
   - Price drop = 0.15% < 1% limit → no pause triggered.
4. Attacker calls `LRTDepositPool.depositAsset(cbETH, 1000e18, ...)`.
   - `rsethAmountToMint` = (1000 × 1.0) / 0.9985 ≈ **1001.5 rsETH**.
   - Fair amount at true prices = (1000 × 1.0) / 1.0 = 1000 rsETH.
   - Attacker receives ~1.5 excess rsETH.
5. stETH feed updates to true price 1.0; next `updateRSETHPrice()` call restores `rsETHPrice` to ~1.0.
6. Attacker's 1001.5 rsETH is now worth ~1001.5 ETH; deposited value was 1000 ETH cbETH. Net gain ≈ 1.5 ETH extracted from existing holders' yield.

**Foundry fork test plan:** Fork mainnet, mock `latestRoundData()` on the stETH/ETH aggregator to return a timestamp `block.timestamp - 6 hours` with a price 0.3% below spot. Call `updateRSETHPrice()`, then `depositAsset(cbETH, 1000e18, 0, "")`. Assert `rsethAmountToMint > 1000e18`. Advance time, restore mock to true price, call `updateRSETHPrice()` again, assert `rsETHPrice` returns to ≥ pre-attack level while attacker's rsETH balance exceeds fair-value equivalent.
