Audit Report

## Title
ETH Withdrawal Permanently Frozen When Recipient Contract Reverts on Native ETH Receipt - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._transferAsset` sends ETH via a low-level `.call{value:}` and hard-reverts with `EthTransferFailed` if the recipient contract rejects ETH. Because rsETH is already burned and ETH already moved into `LRTWithdrawalManager` by a prior `unlockQueue` call, a recipient that cannot accept ETH causes the ETH to be permanently unrecoverable: every completion attempt reverts, `unlockedWithdrawalsCount` stays non-zero, and `sweepRemainingAssets` is permanently blocked.

## Finding Description
The withdrawal lifecycle proceeds in three steps:

**Step 1 — `initiateWithdrawal`**: rsETH is pulled from the user into `LRTWithdrawalManager`. No restriction on `msg.sender` — any contract address may call this. [1](#0-0) 

**Step 2 — `unlockQueue` (operator-only)**: rsETH held by the contract is burned and the corresponding ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager`. After this point, the rsETH is irreversibly gone and the ETH sits in the contract. [2](#0-1) 

**Step 3 — `completeWithdrawal` / `completeWithdrawalForUser`**: Both call `_processWithdrawalCompletion`, which performs state mutations (`popFront`, `delete withdrawalRequests[requestId]`, `unlockedWithdrawalsCount[asset]--`) and then calls `_transferAsset`. [3](#0-2) 

`_transferAsset` for ETH hard-reverts if the call fails: [4](#0-3) 

Because `_transferAsset` is called after the state mutations but within the same transaction, a revert rolls back all state changes — including `unlockedWithdrawalsCount[asset]--`. The count remains non-zero indefinitely.

`sweepRemainingAssets` is gated by `hasUnlockedWithdrawals`: [5](#0-4) [6](#0-5) 

With `unlockedWithdrawalsCount[ETH_TOKEN] > 0`, `sweepRemainingAssets` is permanently blocked. The operator-facing `completeWithdrawalForUser` suffers the identical revert path; its own NatSpec acknowledges the ETH gap but dismisses it as non-impactful for gas grief — it does not address the permanent freeze scenario: [7](#0-6) 

No other on-chain path exists to move the frozen ETH without a contract upgrade.

## Impact Explanation
**Critical — Permanent freezing of funds.** After `unlockQueue` executes, the user's rsETH is irreversibly burned and the ETH equivalent is locked in `LRTWithdrawalManager` with no callable on-chain recovery path. This matches the allowed Critical impact class "Permanent freezing of funds" exactly.

## Likelihood Explanation
**Low-Medium.** Any contract address with no `receive()`/`fallback()` (e.g., a plain multisig, DAO treasury, proxy without an ETH handler) or one with an explicitly reverting `receive()` will trigger this freeze upon initiating an ETH withdrawal. `initiateWithdrawal` places no restriction on `msg.sender`, so any such contract can reach this state without any privileged action. DeFi protocols, DAOs, and institutional custodians routinely interact with restaking protocols from contract addresses, making this a realistic scenario.

## Recommendation
Replace the hard-revert pattern in `_transferAsset` with a WETH-fallback or pull-payment pattern:

```solidity
function _transferAsset(address asset, address to, uint256 amount) internal {
    if (asset == LRTConstants.ETH_TOKEN) {
        (bool sent,) = payable(to).call{ value: amount }("");
        if (!sent) {
            // Fallback: wrap as WETH and send ERC-20 instead
            IWETH(WETH_ADDRESS).deposit{ value: amount }();
            IERC20(WETH_ADDRESS).safeTransfer(to, amount);
        }
    } else {
        IERC20(asset).safeTransfer(to, amount);
    }
}
```

Alternatively, adopt a pull-payment pattern: on failed ETH transfer, record the amount in a `pendingEthClaims[user]` mapping and expose a separate `claimEth()` function, so a stuck transfer never blocks `unlockedWithdrawalsCount` or the `sweepRemainingAssets` path.

## Proof of Concept
1. Deploy `MaliciousReceiver` — a contract with `receive() external payable { revert(); }`.
2. `MaliciousReceiver` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned; ETH moves from `LRTUnstakingVault` into `LRTWithdrawalManager`. `unlockedWithdrawalsCount[ETH_TOKEN]` becomes 1.
4. `MaliciousReceiver` calls `completeWithdrawal(ETH_TOKEN, "")`. `_transferAsset` calls `payable(MaliciousReceiver).call{value: amount}("")`. `receive()` reverts → `sent == false` → `revert EthTransferFailed()`. All state changes roll back; `unlockedWithdrawalsCount[ETH_TOKEN]` remains 1.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, MaliciousReceiver, "")`. Same revert.
6. Manager calls `sweepRemainingAssets(ETH_TOKEN)`. Reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] == 1`.
7. ETH is permanently frozen in `LRTWithdrawalManager`. User's rsETH is already burned. No on-chain recovery path exists.
