Audit Report

## Title
`getRsETHAmountToMint` Returns Non-Zero rsETH Amount for Removed Assets Due to Missing Support Check - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getRsETHAmountToMint` computes a mint amount purely from oracle data without verifying that the queried asset is currently supported. When an asset is removed via `LRTConfig.removeSupportedAsset`, its oracle entry in `LRTOracle` is not cleared, so the view function continues to return a plausible non-zero value for the removed asset while any actual deposit attempt reverts. This creates a concrete inconsistency between the view function's implied promise and the protocol's actual behavior.

## Finding Description
`getRsETHAmountToMint` at [1](#0-0)  performs no support check; it calls `lrtOracle.getAssetPrice(asset)` directly. The `onlySupportedOracle` modifier in `LRTOracle` only checks `assetPriceOracle[asset] != address(0)` [2](#0-1) , not whether the asset is in the supported list.

`LRTConfig.removeSupportedAsset` deletes `isSupportedAsset[asset]` and `assetStrategy[asset]` [3](#0-2)  but never touches `LRTOracle.assetPriceOracle[asset]`. After removal, `assetPriceOracle[removedAsset]` remains non-zero, so `getAssetPrice` succeeds and `getRsETHAmountToMint` returns a non-zero value.

Meanwhile, `depositAsset` is guarded by `onlySupportedERC20Token(asset)` [4](#0-3) , which reverts for any asset not in the supported list. The view function and the state-changing function are therefore inconsistent after a removal event.

`updatePriceOracleFor` further confirms that oracle entries can exist independently of support status: it only enforces a non-zero address check when the asset happens to be supported at the time of the call [5](#0-4) , leaving stale entries in place after removal.

## Impact Explanation
Any caller invoking `getRsETHAmountToMint` with a removed asset receives a non-zero, plausible rsETH amount, falsely implying the deposit is viable. The subsequent deposit reverts. No funds are lost or frozen, but the contract fails to deliver the return implied by its own public view function. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The precondition is a routine admin operation (`removeSupportedAsset`) that is part of the normal protocol lifecycle, not an attacker-controlled action. Once that state exists, any unprivileged external caller can trigger the inconsistency by calling the public `getRsETHAmountToMint`. No special permissions, timing, or economic conditions are required beyond the prior removal event. **Likelihood: Low.**

## Recommendation
Add an `isSupportedAsset` guard to `getRsETHAmountToMint`, consistent with the deposit functions:

```solidity
function getRsETHAmountToMint(address asset, uint256 amount)
    public
    view
    override
    onlySupportedAsset(asset)   // add this modifier
    returns (uint256 rsethAmountToMint)
{
    ...
}
```

Alternatively, update `LRTConfig.removeSupportedAsset` to also call `lrtOracle.updatePriceOracleFor(asset, address(0))` so the oracle entry is cleared atomically with the support removal.

## Proof of Concept
1. Admin calls `LRTConfig.addNewSupportedAsset(tokenX, limit)` and `LRTOracle.updatePriceOracleFor(tokenX, oracleAddr)`.
2. Admin calls `LRTConfig.removeSupportedAsset(tokenX, idx)`. `isSupportedAsset[tokenX]` is deleted; `LRTOracle.assetPriceOracle[tokenX]` remains set.
3. Any user calls `LRTDepositPool.getRsETHAmountToMint(tokenX, 1 ether)`. The call succeeds and returns a non-zero value (e.g., `0.95 ether`) because `getAssetPrice(tokenX)` resolves via the stale oracle entry.
4. The same user calls `LRTDepositPool.depositAsset(tokenX, 1 ether, ...)`. The call reverts with `AssetNotSupported` due to `onlySupportedERC20Token`.
5. The view function promised a valid exchange; the deposit reverted — the contract failed to deliver its promised return.
