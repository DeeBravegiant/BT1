Audit Report

## Title
Permanent Freezing of Unclaimed Yield When `totalKernelStaked` Drops to Zero Mid-Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool` distributes reward tokens at a fixed `rewardRate` over a `duration`. If `totalKernelStaked` reaches zero during an active reward period — which any staker can cause by calling `initiateWithdrawal` — the `rewardPerToken()` accumulator freezes and all rewards that accrue during the zero-staking window are permanently locked in the contract with no recovery path. The contract's own NatSpec acknowledges this behavior but provides no on-chain enforcement to prevent it.

## Finding Description

`rewardPerToken()` contains a zero-guard that freezes accumulation when no tokens are staked:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // accumulation frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

During any window where `totalKernelStaked == 0`, the quantity `rewardRate × elapsed` accrues but is credited to no account and cannot be recovered.

`initiateWithdrawal` reduces `totalKernelStaked` immediately with no floor:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L325-326
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

This is callable by any staker at any time with no restriction beyond their own balance. In a single-staker scenario, one call to `initiateWithdrawal` for the full balance sets `totalKernelStaked = 0`.

The only guard against this scenario is in `notifyRewardAmount`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L570
if (totalKernelStaked == 0) revert NoStakedTokens();
```

This prevents *starting* a new period with zero staked but does nothing to prevent `totalKernelStaked` from reaching zero *during* an active period.

There is no `recoverERC20`, `sweep`, or any other admin function in the contract that can retrieve stranded reward tokens. The contract's NatSpec at L14-23 explicitly acknowledges this limitation and states the mitigation is purely operational ("ensuring there are always some tokens staked"), with no on-chain enforcement.

A secondary, always-present locking mechanism is integer truncation in `notifyRewardAmount`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L580, L583
rewardRate = receivedAmount / duration;
rewardRate = (receivedAmount + remaining) / duration;
```

The remainder `(receivedAmount [+ remaining]) % duration` is transferred in but can never be distributed, permanently locking up to `duration − 1` wei per call.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** Reward tokens deposited by the admin for distribution to stakers become permanently unclaimable once `totalKernelStaked` hits zero mid-period. The contract holds the tokens but has no mechanism to distribute or recover them. This directly matches the allowed impact: *"Medium. Permanent freezing of unclaimed yield."* The truncation dust is a secondary, lower-magnitude instance of the same impact class.

## Likelihood Explanation

Any staker can call `initiateWithdrawal` at any time without restriction. A single-staker scenario (common in early protocol life or low-TVL periods) means one user exiting mid-period is sufficient to trigger the full impact. In multi-staker scenarios, coordinated or panic exits can drain `totalKernelStaked` to zero. No on-chain mechanism prevents this. The NatSpec's own acknowledgment of the issue confirms the developers are aware it is a realistic scenario.

## Recommendation

1. **Add a recovery function** (e.g., `recoverUnclaimedRewards`) callable by admin after `block.timestamp >= finishAt`, to sweep any reward token surplus (contract balance minus staked KERNEL) back to the treasury.
2. **Alternatively**, track `undistributedRewards` explicitly during zero-staking windows and roll them into the next period's `receivedAmount` in `notifyRewardAmount`, similar to how `remaining` is already rolled over.
3. For truncation dust, accumulate `receivedAmount % duration` in a state variable and add it to the next `notifyRewardAmount` call's effective amount.

## Proof of Concept

```
1. Admin calls notifyRewardAmount(1_000e18) with duration = 30 days
   → rewardRate = 1_000e18 / 2_592_000 ≈ 385_802_469_135_802 tokens/sec
   → dust = 1_000e18 - rewardRate * 2_592_000 locked immediately (truncation)

2. Alice is the sole staker: balanceOf[alice] = 100e18, totalKernelStaked = 100e18

3. After 15 days, Alice calls initiateWithdrawal(100e18)
   → updateReward(alice) snapshots her earned ~500e18 rewards correctly
   → totalKernelStaked = 0

4. For the remaining 15 days:
   → rewardPerToken() returns frozen rewardPerTokenStored
   → ~500e18 reward tokens accrue at rewardRate but are credited to no account

5. Alice calls getReward() → receives only ~500e18 (15 days of rewards)
6. ~500e18 reward tokens remain in the contract forever, unclaimable by anyone

Foundry test plan:
- Deploy KernelDepositPool with mock KERNEL and reward tokens
- Admin calls notifyRewardAmount(1_000e18), duration = 30 days
- Alice stakes 100e18
- vm.warp(block.timestamp + 15 days)
- Alice calls initiateWithdrawal(100e18)
- vm.warp(block.timestamp + 15 days) // advance past finishAt
- Alice calls getReward()
- Assert rewardsToken.balanceOf(alice) ≈ 500e18
- Assert rewardsToken.balanceOf(address(pool)) ≈ 500e18 (stranded, unrecoverable)
```
