### Title
Permanent Token Deployment Deadlock via Lamport-Griefing the Mint PDA — (`solana/programs/bridge_token_factory/src/instructions/user/deploy_token.rs`)

---

### Summary

An unprivileged attacker can permanently prevent any NEAR token from ever being deployed on Solana by transferring a trivial amount of SOL to the deterministic mint PDA address before `deploy_token` executes. Anchor's `init` constraint rejects any account with non-zero lamports, and no recovery instruction exists in the program.

---

### Finding Description

The `DeployToken` account struct uses Anchor's `init` constraint on the `mint` account: [1](#0-0) 

The PDA is derived deterministically from public inputs:

```
seeds = [b"wrapped_mint", token.to_hashed_bytes()]
``` [2](#0-1) 

The `token` string is the NEAR token identifier (e.g. `usdt.tether-token.near`), which is broadcast publicly on NEAR via `log_metadata` events before any Solana deployment occurs.

Anchor's `init` constraint enforces:

```rust
if account.lamports() != 0 {
    return Err(ErrorCode::AccountAlreadyInitialized.into());
}
```

In Solana, **any party can transfer SOL to any address** — including PDA addresses — without the recipient's signature. Transferring even 1 lamport to the mint PDA causes Anchor's `init` to fail unconditionally with `AccountAlreadyInitialized`.

**Note on the question's proposed mechanism:** A direct `system_program::create_account` call to a PDA address is not possible by an external party (PDAs cannot sign externally). The actual attack surface is the simpler SOL transfer, which achieves the identical outcome.

**No recovery path exists.** The full instruction set contains no instruction to close, drain, or re-initialize a poisoned mint PDA: [3](#0-2) [4](#0-3) 

`update_metadata` only operates on already-successfully-deployed tokens and cannot help here. [5](#0-4) 

---

### Impact Explanation

- Every NEAR token has exactly one deterministic mint PDA on Solana. Once poisoned, `deploy_token` will always revert.
- All bridge transfers for that token (both `finalize_transfer` and `init_transfer`) depend on the mint existing. With the mint undeployable, those transfers are permanently locked.
- The NEAR-side bridge state (`deployed_tokens`, `locked_tokens`) may record the token as pending or failed, with no on-chain path to retry on Solana.

This satisfies: **Critical — Irreversible fund lock / permanently unclaimable user or protocol value.**

---

### Likelihood Explanation

- Cost to attacker: ~1 lamport + one transaction fee (~5000 lamports total). Economically trivial.
- Information required: the NEAR token string, which is emitted publicly in `log_metadata` events on NEAR before any Solana deployment.
- Timing window: the attacker has the entire period between the NEAR `log_metadata` event and the Solana `deploy_token` transaction — typically minutes to hours.
- No privileged access, no key material, no colluding parties required.

---

### Recommendation

Replace the `init` constraint with `init_if_needed` **only if** a separate idempotency guard (e.g. a program-owned flag account) is used to prevent double-initialization. Alternatively:

1. Add an admin instruction that can close a zero-data, system-program-owned account at the mint PDA (draining its lamports back to a treasury), allowing a clean retry.
2. Or, in `deploy_token`, detect the case where the PDA is owned by the system program with lamports but no data, absorb those lamports into the rent payment, and proceed with initialization — this requires a custom CPI path rather than Anchor's `init` macro.

---

### Proof of Concept

```rust
// 1. Compute the mint PDA off-chain (attacker side)
let token_str = "usdt.tether-token.near";
let seed_bytes = to_hashed_bytes(token_str); // mirrors StringExt::to_hashed_bytes
let (mint_pda, _bump) = Pubkey::find_program_address(
    &[b"wrapped_mint", &seed_bytes],
    &bridge_token_factory::ID,
);

// 2. Transfer 1 lamport to the PDA (no recipient signature needed)
let transfer_ix = system_instruction::transfer(
    &attacker_pubkey,
    &mint_pda,
    1, // lamports
);
// Send and confirm this transaction.

// 3. Now call deploy_token with a valid signed payload for the same token.
// Result: Anchor returns AccountAlreadyInitialized.
// The token is permanently undeployable. No recovery instruction exists.
```
