Audit Report

## Title
ETH Withdrawal Permanently Frozen When Recipient Contract Reverts on ETH Receive - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._processWithdrawalCompletion()` pushes ETH to a user-controlled address via a low-level `call`. If the recipient contract cannot accept ETH, the `completeWithdrawal()` transaction always reverts. Because rsETH is burned in the prior, committed `unlockQueue()` transaction, the user's rsETH is permanently destroyed while the corresponding ETH is permanently frozen inside `LRTWithdrawalManager` with no admin rescue path.

## Finding Description
The withdrawal lifecycle spans three separate transactions. In step 2, `unlockQueue()` irreversibly burns rsETH and pulls ETH into the contract: [1](#0-0) 

In step 3, `_processWithdrawalCompletion()` decrements `unlockedWithdrawalsCount` and then calls `_transferAsset`: [2](#0-1) 

`_transferAsset` uses a bare low-level `call` for ETH: [3](#0-2) 

If `to` is a contract without a `receive()`/`fallback()` function, `sent` is `false` and `EthTransferFailed` is thrown. The entire `completeWithdrawal()` transaction reverts — restoring `unlockedWithdrawalsCount` — but the rsETH burned in the prior `unlockQueue()` transaction is not restored. The ETH sits in `LRTWithdrawalManager` permanently.

The operator-assisted path `completeWithdrawalForUser()` calls the same `_processWithdrawalCompletion()` and is equally blocked: [4](#0-3) 

The only potential escape valve, `sweepRemainingAssets()`, is gated by `hasUnlockedWithdrawals()`: [5](#0-4) 

Because the stuck withdrawal keeps `unlockedWithdrawalsCount[asset] > 0` (the decrement at line 717 is always reverted along with the failing transfer), `sweepRemainingAssets()` is permanently blocked: [6](#0-5) 

There is no admin function to force-redirect ETH to an alternative address.

## Impact Explanation
**Critical — Permanent freezing of funds.** After `unlockQueue()` commits, the user's rsETH is irreversibly burned and the equivalent ETH is held in `LRTWithdrawalManager`. If the user's address cannot accept ETH, both assets are permanently lost: rsETH is gone and ETH is unrecoverable. This matches the allowed impact class "Permanent freezing of funds."

## Likelihood Explanation
**Medium.** Smart-contract wallets (multisigs, DAO treasuries, protocol vaults) that interact with LRT-rsETH are common and many do not implement a `receive()` function. A user operating through such a contract who requests an ETH withdrawal will trigger this freeze automatically — no malicious intent is required. The operator's `unlockQueue()` call, which is a routine operational step, is the irreversible trigger. The NatDoc on `completeWithdrawalForUser` even acknowledges ETH is not expected to be used via that path, confirming no alternative rescue exists. [7](#0-6) 

## Recommendation
Replace the push-payment pattern with a pull-payment pattern: instead of pushing ETH to the user in `_processWithdrawalCompletion()`, record the owed amount in a `claimableETH[user]` mapping and let the user (or any address they designate) pull it in a separate `claimETH()` function. Alternatively, allow the user to specify a separate `recipient` address at withdrawal-completion time so they can redirect ETH to an EOA even if the initiating contract cannot receive it.

## Proof of Concept
1. Deploy `MaliciousVault` — a contract with no `receive()` function.
2. From `MaliciousVault`, call `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred into the withdrawal manager.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned at line 305; ETH is transferred from `LRTUnstakingVault` into `LRTWithdrawalManager`. **This transaction commits.**
4. Anyone calls `completeWithdrawal(ETH_TOKEN, "")` on behalf of `MaliciousVault`. `_transferAsset` executes `payable(MaliciousVault).call{value: amount}("")`; the call returns `false`; `EthTransferFailed` is thrown. Transaction reverts, restoring `unlockedWithdrawalsCount` to its pre-call value.
5. Step 4 can be repeated indefinitely — it always reverts.
6. `sweepRemainingAssets(ETH_TOKEN)` reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.
7. rsETH is permanently burned; ETH is permanently frozen in `LRTWithdrawalManager`.
