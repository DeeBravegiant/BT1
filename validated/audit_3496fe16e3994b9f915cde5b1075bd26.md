Audit Report

## Title
Unconditional Zero-Amount Fee Transfer in `MerkleDistributor.claim()` Permanently Freezes Unclaimed Yield When `feeInBPS` Is Zero - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` even when `fee == 0`. When `feeInBPS` is set to zero — a valid and reachable configuration — and the configured token reverts on zero-amount transfers, every `claim()` invocation reverts, permanently freezing all unclaimed yield in the distributor. The sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` already apply the correct `if (fee > 0)` guard; `MerkleDistributor` is the only distributor in the codebase that omits it.

## Finding Description
In `contracts/utils/MerkleDistributor/MerkleDistributor.sol`, the `claim` function at lines 138–144 computes a fee and unconditionally transfers it:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);   // ← no zero-guard
```

`feeInBPS` is owner-settable via `setFeeInBPS`, which enforces only an upper bound (`> MAX_FEE_IN_BPS`), making `feeInBPS = 0` a valid and reachable state. When `feeInBPS == 0`, `fee` evaluates to `0`, and `safeTransfer(protocolTreasury, 0)` is executed. Any ERC-20 token that reverts on zero-amount transfers will cause the entire `claim` transaction to revert. Because the user's claim state is updated at lines 134–135 before the transfers, the revert rolls back the state update, but the user can never successfully claim — every subsequent attempt hits the same revert path.

The two sibling contracts in the same repository already apply the correct fix:
- `KernelMerkleDistributor` lines 341–343: `if (fee > 0) { kernel.safeTransfer(protocolTreasury, fee); }`
- `KernelTop100MerkleDistributor` lines 332–334: `if (fee > 0) { kernel.safeTransfer(protocolTreasury, fee); }`

`MerkleDistributor` is the only distributor that omits this guard, creating an inconsistency with the established pattern in the codebase.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

When `feeInBPS == 0` and the configured `token` reverts on zero-amount transfers, every call to `claim()` reverts. No user can withdraw their allocated tokens. The tokens remain locked in the distributor contract indefinitely. There is no recovery path available to users; only the owner can pause/unpause or change the token, but neither action releases already-allocated claims to affected users. This matches the allowed impact: **Medium. Permanent freezing of unclaimed yield.**

## Likelihood Explanation
`feeInBPS = 0` is an explicitly reachable configuration requiring no attacker action — it is a natural operational choice (e.g., a promotional distribution with no protocol cut). The `token` field is fully configurable by the owner and can be set to any ERC-20. Tokens that revert on zero-amount transfers exist in production (custom protocol tokens with explicit zero-transfer guards, certain wrapped tokens, etc.). The combination of both conditions is realistic and requires no malicious intent — it arises from normal operational configuration. Once both conditions are present, any user calling the public `claim()` function triggers the revert, and the freeze is permanent without owner intervention to change `feeInBPS` or the token.

## Recommendation
Add a zero-amount guard before the fee transfer, consistent with the pattern already used in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy a mock ERC-20 token that reverts on zero-amount transfers: `if (amount == 0) revert ZeroTransferAmount()`.
2. Deploy `MerkleDistributor` with this token and `feeInBPS = 0` (or call `setFeeInBPS(0)` after deployment).
3. Set a valid Merkle root containing a user allocation.
4. User calls `claim(index, account, cumulativeAmount, merkleProof)` with a valid proof.
5. Execution reaches line 138: `fee = (claimableAmount * 0) / 10_000 = 0`.
6. `IERC20(token).safeTransfer(account, claimableAmount)` succeeds.
7. `IERC20(token).safeTransfer(protocolTreasury, 0)` reverts with `ZeroTransferAmount`.
8. The entire transaction reverts. The user's claim state is not updated.
9. Every subsequent `claim()` call by any user hits the same revert. All yield is permanently frozen.

**Foundry test plan:** Write a `MerkleDistributorTest` fork/unit test that deploys a `ZeroTransferRevertToken`, initializes `MerkleDistributor` with `feeInBPS = 0`, constructs a valid Merkle tree with one leaf, and asserts that `claim()` reverts with the token's zero-transfer error. Then apply the `if (fee > 0)` guard and assert the same call succeeds.
