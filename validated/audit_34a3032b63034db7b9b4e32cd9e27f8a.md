Audit Report

## Title
Zero-wrsETH Mint on Dust Deposits Due to Missing `rsETHAmount == 0` Guard — (`contracts/pools/RSETHPoolV2NBA.sol`)

## Summary
`RSETHPoolV2NBA.deposit()` accepts ETH and mints wrsETH without verifying that the computed `rsETHAmount` is non-zero. Integer division in `viewSwapRsETHAmountAndFee` truncates `rsETHAmount` to zero for sufficiently small deposits when the oracle rate exceeds `1e18`, which is the normal post-genesis operating state. The depositor's ETH is accepted, locked in the contract, and irrecoverable by the depositor, while they receive zero wrsETH.

## Finding Description
In `viewSwapRsETHAmountAndFee` (L132), the output is computed as:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

This is plain Solidity integer division. When `amountAfterFee * 1e18 < rsETHToETHrate`, the result is `0`. `deposit()` (L106–118) only guards the ETH input (`if (amount == 0) revert InvalidAmount()`) and then calls `wrsETH.mint(msg.sender, rsETHAmount)` unconditionally. OpenZeppelin's ERC-20 `_mint` accepts a zero amount without reverting, so the transaction succeeds silently.

`InterimRSETHOracle._setRate()` enforces `rate >= 1e18`. Since rsETH is yield-bearing, the rate starts at `1e18` and increases monotonically — any rate strictly above `1e18` (the normal operating state) enables the truncation for deposits of `amountAfterFee < rsETHToETHrate / 1e18` wei. At a rate of `1.05e18`, a 1-wei deposit produces `rsETHAmount = 0`. The deposited ETH accumulates in the contract balance. Neither `moveAssetsForBridging()` (sends to bridger) nor `withdrawFees()` (sends fees to admin receiver) provides any path for the depositor to recover their ETH.

## Impact Explanation
A depositor sending a dust ETH amount (1 wei up to `floor(rsETHToETHrate / 1e18) - 1` wei) receives 0 wrsETH while their ETH is permanently inaccessible to them. The invariant "every accepted non-zero ETH deposit produces a non-zero wrsETH mint" is broken. This maps to **Medium — Temporary freezing of funds** (the ETH is not destroyed but is irrecoverable by the depositor without admin intervention; no on-chain mechanism exists to return it to the original sender).

## Likelihood Explanation
No privileged access, oracle manipulation, or unusual configuration is required. Any external user can call `deposit{value: 1}()` at any time after the oracle rate exceeds `1e18`, which is the expected normal state of the protocol. The condition is fully permissionless and becomes easier to trigger as the rate grows over time. The per-transaction loss is bounded to dust amounts (at realistic rates, at most a few wei), but the path is unconditionally reachable.

## Recommendation
Add a zero-output guard immediately after computing `rsETHAmount` in `deposit()`:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This mirrors the existing `amount == 0` guard and ensures the invariant holds for all accepted deposits.

## Proof of Concept
```solidity

// 2. Deploy mock wrsETH that records mint calls
// 3. Deploy RSETHPoolV2NBA pointing to oracle and mock wrsETH
// 4. Call pool.deposit{value: 1}("")
// 5. Assert mockWrsETH.lastMintAmount == 0  // PASSES — user got nothing
// 6. Assert address(pool).balance == 1      // PASSES — ETH locked
// 7. Assert tx did not revert               // PASSES — no guard triggered
```
The `deposit()` call does not revert, `rsETHAmount = 1 * 1e18 / 1.05e18 = 0`, and the depositor's 1 wei is locked with no recovery path.
