### Title
Reentrancy in `initTransfer1155` via Malicious ERC1155 `safeTransferFrom` Callback Emits Multiple `InitTransfer` Events Without Proportional Token Locks - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.initTransfer1155` calls `IERC1155(tokenAddress).safeTransferFrom(...)` on a user-supplied, attacker-controlled ERC1155 contract before the `InitTransfer` event is emitted. A malicious ERC1155 can reenter `initTransfer1155` during that external call, causing multiple `InitTransfer` events with distinct nonces to be emitted in a single transaction while only one (or zero) actual token lock occurs. The NEAR side processes each event independently, crediting the attacker with more wrapped tokens than were ever locked on EVM.

### Finding Description

In `initTransfer1155`, the execution order is:

1. `currentOriginNonce += 1` — nonce incremented (line 448)
2. `IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "")` — **external call to attacker-controlled contract** (line 458–464)
3. `initTransferExtension(...)` (line 468–478)
4. `emit BridgeTypes.InitTransfer(...)` — event emitted (line 480–489) [1](#0-0) 

The `tokenAddress` parameter is fully attacker-controlled — any address can be passed. A malicious ERC1155 contract's `safeTransferFrom` implementation can call back into `bridge.initTransfer1155(...)` before returning. Each reentrant call increments `currentOriginNonce` again and eventually emits a new `InitTransfer` event with a fresh nonce. The malicious ERC1155 can choose not to transfer any tokens in the reentrant invocations (or in any invocation), yet the events are still emitted and recorded on-chain.

The bridge's `onERC1155Received` guard does not block this path: [2](#0-1) 

That hook is `external view` — it cannot modify state and is not the reentrancy vector. The reentrancy occurs inside the malicious ERC1155's `safeTransferFrom` body, which calls `bridge.initTransfer1155` directly before returning.

By contrast, `finTransfer` correctly marks the nonce used **before** any external call, making it reentrancy-safe: [3](#0-2) 

`initTransfer1155` has no analogous guard.

### Impact Explanation

The NEAR bridge contract processes each `InitTransfer` event by its `(origin_chain, origin_nonce)` pair. Because each reentrant call produces a distinct nonce, the NEAR side treats each event as an independent, valid transfer and mints wrapped tokens for each one. The attacker receives N× the wrapped tokens on NEAR while locking at most 1× (or 0×) tokens on EVM. This directly breaks the 1:1 backing guarantee of the bridge — unbacked wrapped supply is created.

This maps to the allowed impact: **Unauthorized creation of wrapped bridge assets** and **asset-accounting divergence that breaks backing guarantees**.

### Likelihood Explanation

The attack is fully permissionless:
1. Attacker deploys a malicious ERC1155 contract.
2. Attacker calls `logMetadata1155` to register it (permissionless).
3. Attacker waits for the NEAR side to deploy the corresponding wrapped token via `deployToken`.
4. Attacker calls `initTransfer1155` with the malicious ERC1155 — reentrancy fires, multiple `InitTransfer` events are emitted.
5. Attacker submits each event to the NEAR bridge to claim wrapped tokens.

No privileged role, leaked key, or external dependency compromise is required.

### Recommendation

Apply the checks-effects-interactions pattern by emitting the `InitTransfer` event **before** the external `safeTransferFrom` call, or add a `nonReentrant` modifier (OpenZeppelin `ReentrancyGuardUpgradeable`) to `initTransfer1155`. The same modifier should be applied to `initTransfer` for consistency, since the `customMinters[tokenAddress].burn(...)` path also makes an external call to an admin-registered but potentially upgradeable contract.

```solidity
// Option A: nonReentrant modifier
function initTransfer1155(...) external payable nonReentrant whenNotPaused(PAUSED_INIT_TRANSFER) {
    ...
}

// Option B: emit event before external call (CEI)
emit BridgeTypes.InitTransfer(...);   // emit first
IERC1155(tokenAddress).safeTransferFrom(...);  // then external call
```

### Proof of Concept

```solidity
contract MaliciousERC1155 is IERC1155 {
    OmniBridge bridge;
    uint256 reentrancyCount;

    function safeTransferFrom(address, address, uint256 tokenId, uint256 amount, bytes calldata) external {
        if (reentrancyCount < 3) {
            reentrancyCount++;
            // Reenter without actually transferring tokens
            bridge.initTransfer1155(address(this), tokenId, amount, 0, 0, "attacker.near", "");
        }
        // Return without transferring any tokens — balanceOf(bridge) stays 0
    }
    // ... other IERC1155 stubs
}
```

Attack sequence:
- `currentOriginNonce` starts at N.
- Outer call: nonce → N+1, `safeTransferFrom` fires → reenter.
- Reentrant call 1: nonce → N+2, `safeTransferFrom` fires → reenter.
- Reentrant call 2: nonce → N+3, `safeTransferFrom` fires → reenter.
- Reentrant call 3: nonce → N+4, `safeTransferFrom` returns (no tokens transferred).
- `InitTransfer(nonce=N+4)` emitted.
- Unwind: `InitTransfer(nonce=N+3)`, `InitTransfer(nonce=N+2)`, `InitTransfer(nonce=N+1)` emitted.

Four `InitTransfer` events on-chain, zero tokens locked. NEAR side mints 4× `amount` wrapped tokens to the attacker.
