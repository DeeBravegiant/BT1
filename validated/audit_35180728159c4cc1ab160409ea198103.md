Audit Report

## Title
Rewards Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a rate-per-token reward model where `rewardPerToken()` returns the stored value unchanged when `totalKernelStaked == 0`, but the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()`. Any time elapsed while no tokens are staked is permanently skipped in reward accounting, locking those reward tokens in the contract with no recovery mechanism.

## Finding Description
The root cause is a two-part interaction in the `updateReward` modifier and `rewardPerToken()`:

`rewardPerToken()` at L408–411 returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0` — no new rewards are accumulated for the elapsed time. However, the `updateReward` modifier at L233–234 unconditionally sets `updatedAt = lastTimeRewardApplicable()` regardless of whether `totalKernelStaked` is zero.

When `initiateWithdrawal` is called and reduces `totalKernelStaked` to zero (L325–326), the modifier runs: `rewardPerTokenStored` stays frozen, but `updatedAt` advances to the current timestamp. All time elapsed during the zero-staking window is consumed without accumulating any `rewardPerTokenStored` delta. When the next staker calls `stake()`, the reward formula `rewardRate * (lastTimeRewardApplicable() - updatedAt)` begins from the post-gap `updatedAt`, permanently skipping the zero-staking interval. The reward tokens for that window remain locked in the contract.

`notifyRewardAmount` at L569–570 only guards against starting a reward period with zero stakers; it provides no protection against stakers withdrawing after the period begins. There is no admin sweep or rescue function anywhere in the contract. The contract's own NatSpec at L17–22 explicitly acknowledges this behavior and relies entirely on an off-chain operational assumption.

## Impact Explanation
Reward tokens sent via `notifyRewardAmount` that correspond to any time window where `totalKernelStaked == 0` are permanently locked in the contract. There is no on-chain mechanism to recover or redistribute them. This constitutes **permanent freezing of unclaimed yield**, a Medium-severity impact within the allowed scope.

## Likelihood Explanation
Any staker can call `initiateWithdrawal` at any time with no restriction tied to active reward periods. A single large staker holding 100% of `totalKernelStaked` can reduce it to zero in one transaction. No attacker coordination is required; it can occur through entirely normal user behavior (e.g., a user legitimately exiting their position). The `withdrawalDelay` only delays token receipt — `totalKernelStaked` is decremented immediately at `initiateWithdrawal` L326, not at `claimWithdrawal`.

## Recommendation
In `rewardPerToken()`, do not advance `updatedAt` when `totalKernelStaked == 0`. Instead, keep `updatedAt` frozen so the reward debt is preserved and can be distributed once stakers return. Concretely, change the `updateReward` modifier to only update `updatedAt` when `totalKernelStaked > 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```

This mirrors the standard Synthetix fix and ensures rewards for zero-staking windows are preserved for future stakers. Alternatively, add a guard in `initiateWithdrawal` that reverts if the withdrawal would bring `totalKernelStaked` to zero during an active reward period (`block.timestamp < finishAt`).

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with Alice having staked 100 KERNEL. `rewardRate` is set, `finishAt = block.timestamp + duration`, `updatedAt = block.timestamp`.
2. Alice calls `initiateWithdrawal(100)`. `updateReward(Alice)` runs: since `totalKernelStaked` is still 100 at the time `rewardPerToken()` is called (modifier runs before state change), `rewardPerTokenStored` accumulates Alice's share correctly. Then `updatedAt = block.timestamp` (T1). After the modifier, `totalKernelStaked` becomes 0.
3. Time advances by half the reward duration (T1 → T2). No stakers exist. `rewardRate * (T2 - T1)` worth of rewards accrue to nobody.
4. Bob calls `stake(100)`. `updateReward(Bob)` runs: `rewardPerToken()` sees `totalKernelStaked == 0` (Bob's stake hasn't been added yet), returns `rewardPerTokenStored` unchanged. Then `updatedAt = lastTimeRewardApplicable() = T2`. Bob's stake is added. From this point, rewards only accumulate from T2 onward. The `rewardRate * (T2 - T1)` rewards are permanently unclaimable, locked in the contract.
