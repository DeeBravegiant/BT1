### Title
Stub `toPlutusScriptPurpose` and Discarded Sub-Transaction Context in PlutusV4 Guard Evaluation Cause Script Validation Bypass and Node Crash - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxInfo.hs`)

---

### Summary

The `EraPlutusTxInfo 'PlutusV4 DijkstraEra` instance in the Dijkstra era contains two incomplete implementations that are direct analogs of the "defined but not implemented" vulnerability class:

1. `toPlutusScriptPurpose _ = error "stub: PlutusV4 not yet implemented"` — a stub that throws a Haskell runtime `error` when any PlutusV4 script is evaluated, because it is unconditionally called by `toPlutusV4Args` (the `toPlutusArgs` implementation for PlutusV4).

2. In `toPlutusTxInfo` for PlutusV4, sub-transaction infos (`_subTxInfosForGuards`) are computed but explicitly discarded with `-- TODO: Include _subTxInfosForGuards`, so the guard script receives `topTxInfo` with no sub-transaction data and cannot enforce any constraints on sub-transactions.

---

### Finding Description

**Root cause 1 — stub `toPlutusScriptPurpose`:** [1](#0-0) 

```haskell
instance EraPlutusTxInfo 'PlutusV4 DijkstraEra where
  toPlutusTxCert _ _ = pure . transTxCert

  toPlutusScriptPurpose _ = error "stub: PlutusV4 not yet implemented"
```

This method is called unconditionally by `toPlutusV4Args`, which is the `toPlutusArgs` implementation for PlutusV4: [2](#0-1) 

```haskell
  toPlutusArgs = toPlutusV4Args
  ...
toPlutusV4Args proxy pv txInfo plutusPurpose maybeSpendingData redeemerData = do
  scriptPurpose <- toPlutusScriptPurpose proxy pv plutusPurpose   -- ← always called
  ...
```

`toPlutusArgs` is invoked for every script execution to assemble the `ScriptContext`. Any transaction that causes a PlutusV4 script to be evaluated will reach this call
