### Title
Missing `curve.check()` Guards in `blob_to_keyset` Allow Infinity Root Points, Producing Derived ECDSA-2PC Key Blobs with Attacker-Influenced Public Keys - (File: `src/cbmpc/api/hd_keyset_ecdsa_2p.cpp`)

---

### Summary

`blob_to_keyset` in `src/cbmpc/api/hd_keyset_ecdsa_2p.cpp` calls `ecc_point_t::from_bin` for `root.Q` and `root.K` but omits the `curve.check()` call that the analogous `blob_to_key` in `src/cbmpc/api/ecdsa2pc.cpp` performs. For p256 (OSSL backend), `from_bin` explicitly accepts the point at infinity and returns `SUCCESS`. Without `curve.check()`, a crafted keyset blob with `root_Q_compressed` encoding the p256 infinity point passes `deserialize_keyset_blob` without error. `derive_keys` then proceeds with `root.Q = ∞`, computing `Q_derived = ∞ + delta·G = delta·G`, and emitting derived ECDSA-2PC key blobs whose public key `Q` is `(delta + non_hard_delta)·G` — a value the attacker can influence — rather than the correct `Q_correct + (delta + non_hard_delta)·G`.

### Finding Description

**The missing guard:**

`blob_to_keyset` at lines 98–100:

```cpp
error_t rv = keyset.root.Q.from_bin(icurve, blob.root_Q_compressed);
if (rv) return rv;
return keyset.root.K.from_bin(icurve, blob.root_K_compressed);
``` [1](#0-0) 

No `curve.check()` is called for either point. Compare with `blob_to_key` in `ecdsa2pc.cpp`:

```cpp
if (const error_t rv = key.Q.from_bin(curve, blob.Q_compressed)) return rv;
if (curve.check(key.Q)) return coinbase::error(E_FORMAT, "invalid key blob");
``` [2](#0-1) 

**What `curve.check()` enforces that `from_bin` does not:**

`ecurve_t::check` rejects infinity and verifies subgroup membership:

```cpp
if (!point.is_in_subgroup()) return crypto::error("EC-point is not on curve");
if (!thread_local_store_allow_ecc_infinity)
  if (point.is_infinity()) return crypto::error("EC-point is infinity");
``` [3](#0-2) 

**Why `from_bin` accepts infinity for p256:**

`ecurve_ossl_t::from_bin` explicitly handles the all-zero encoding as infinity and returns `SUCCESS`:

```cpp
if (bin.size > 0 && bin[0] == 0)  // infinity
{
  if (bin.size != 1 + size() && bin.size != 1 + size() * 2) return coinbase::error(E_FORMAT);
  for (int i = 0; i < bin.size; i++)
    if (bin[i]) return coinbase::error(E_CRYPTO);
  bin.size = 1;
}
if (0 >= EC_POINT_oct2point(...)) { return openssl_error(...); }
return SUCCESS;
``` [4](#0-3) 

For p256, `size() = 32`, so a 33-byte all-zero `root_Q_compressed` is accepted as infinity. The secp256k1 backend (`secp256k1_eckey_pubkey_parse`) rejects infinity at the `from_bin` level, so this gap is **p256-specific**.

**Why the converter framework does not save this:**

`ecc_point_t::convert_fixed_curve` does call `curve.check()` after `from_bin`, but only when deserializing an `ecc_point_t` field via the converter. `keyset_blob_v1_t` stores `root_Q_compressed` and `root_K_compressed` as raw `buf_t` fields, not as `ecc_point_t`:

```cpp
buf_t root_Q_compressed;
buf_t root_K_compressed;
``` [5](#0-4) 

The converter serializes them as opaque byte buffers, bypassing `convert_fixed_curve` entirely. The `curve.check()` in `convert_fixed_curve` is never reached. [6](#0-5) 

**Effect in `derive_keys`:**

With `root.Q = ∞`:

```cpp
ecc_point_t Q = key.root.Q;          // Q = ∞
...
ecc_point_t Q_derived = Q + delta_G; // Q_derived = ∞ + delta·G = delta·G  (wrong)
...
derived_keys[i].Q = Q_derived + non_hard_delta_G;  // = (delta + non_hard_delta)·G
``` [7](#0-6) 

The correct value is `Q_correct + (delta + non_hard_delta)·G`. The attacker controls `k_share` in the blob (subject only to a range check, not a consistency check against `root.K`), which influences `delta` via the VRF computation `Z = Z1 + Z2`, `delta = hash(Z) % q`. The attacker can therefore steer the derived public key.

**Effect of `root.K` at infinity:**

`get_other_K_share()` computes `K - get_K_share()`. With `K = ∞`, `other_K_share = -(k_share·G)`, which is wrong. This causes P1's verification of P2's ZK-DH proof to fail (the proof is verified against the wrong base point), aborting the protocol. This is a harmless reject, not an accepted-bad-output path. [8](#0-7)

### Impact Explanation

An attacker who controls a p256 HD-ECDSA-2P keyset blob (e.g., a Byzantine participant who stores and re-submits their own blob) can craft `root_Q_compressed` as 33 zero bytes. `deserialize_keyset_blob` accepts the blob without error. `derive_ecdsa_2p_keys` then emits derived ECDSA-2PC key blobs whose `Q` field is `(delta + non_hard_delta)·G` — an attacker-influenced value — rather than the correct joint public key. The honest party's derived blobs carry the correct `Q`. The two parties' derived blobs are inconsistent: they disagree on the public key for every derived path. Any subsequent signing operation using the attacker's derived blob will either fail or produce a signature that verifies against the wrong public key, enabling public-key substitution / confusion at the derived-key layer.

This falls under **High** (attacker-controlled blob accepted with wrong key material) / **Medium** (honest-party divergence, invalid cryptographic output with security impact).

---

### Likelihood Explanation

The attacker needs only to supply a crafted keyset blob to the `derive_ecdsa_2p_keys` API. No threshold collusion, no network interception, and no privileged access are required. The crafted encoding (33 zero bytes for p256 infinity) is trivial to construct. The only constraint is that the blob must pass version and curve checks (both trivially satisfiable) and the Paillier/scalar range checks (which do not touch `root_Q_compressed`).

---

### Recommendation

Add `curve.check()` calls immediately after each `from_bin` in `blob_to_keyset`, mirroring the pattern in `blob_to_key`:

```cpp
error_t rv = keyset.root.Q.from_bin(icurve, blob.root_Q_compressed);
if (rv) return rv;
if (icurve.check(keyset.root.Q)) return coinbase::error(E_FORMAT, "invalid keyset blob");

rv = keyset.root.K.from_bin(icurve, blob.root_K_compressed);
if (rv) return rv;
if (icurve.check(keyset.root.K)) return coinbase::error(E_FORMAT, "invalid keyset blob");
``` [9](#0-8) 

Additionally, consider storing `root_Q_compressed` and `root_K_compressed` as `ecc_point_t` fields in `keyset_blob_v1_t` so that `convert_fixed_curve` (which already calls `curve.check()`) provides automatic validation during deserialization.

### Proof of Concept

1. Perform a legitimate p256 HD-ECDSA-2P DKG to obtain a valid keyset blob.
2. Deserialize the blob, locate the `root_Q_compressed` field (a length-prefixed 33-byte buffer), and replace it with 33 zero bytes (`\x00` × 33).
3. Re-serialize the blob and pass it to `hd_keyset_ecdsa_2p::derive_ecdsa_2p_keys`.
4. Assert the call returns `SUCCESS` (it does — no `curve.check()` is performed).
5. Inspect the returned derived key blobs: their `Q_compressed` field encodes `(delta + non_hard_delta)·G`, not `Q_correct + (delta + non_hard_delta)·G`.
6. Compare with the honest party's derived key blobs: the two `Q` values differ, confirming divergence and attacker-influenced public keys.
