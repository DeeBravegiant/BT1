### Title
Sub-Transaction Execution Units Bypass `maxTxExUnits` and `maxBlockExUnits` Limits in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may embed an unbounded number of sub-transactions (`dtbrSubTransactions`), each carrying its own Plutus witness set with redeemer-declared `ExUnits` budgets. The `SUBUTXO` transition rule (`dijkstraSubUtxoTransition`) performs no `validateExUnitsTooBigUTxO` check, and the `totExUnits` function used by both the per-transaction UTXO check and the per-block BBODY check reads only the top-level transaction's witness set redeemers. Sub-transaction `ExUnits` are therefore never counted against `maxTxExUnits` or `maxBlockExUnits`, allowing an attacker to submit a single on-chain transaction whose aggregate script execution cost exceeds both protocol-parameter limits without paying the corresponding fees.

---

### Finding Description

**Root cause — `totExUnits` ignores sub-transaction redeemers:**

`totExUnits` is defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It reads only `tx ^. witsTxL` — the top-level transaction's witness set. Sub-transactions (`DijkstraSubTx`) carry their own independent witness sets (`dstWits`) containing their own redeemers and `ExUnits` declarations, which are never traversed by this function. [1](#0-0) 

**Per-transaction limit check — only top-level ExUnits are tested:**

In `dijkstraUtxoTransition`:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

`validateExUnitsTooBigUTxO` calls `totExUnits tx`, which, as shown above, sums only the top-level redeemers. Sub-transaction redeemers are invisible to this check. [2](#0-1) [3](#0-2) 

**SUBUTXO rule — no ExUnits check at all:**

`dijkstraSubUtxoTransition` (the rule that validates each sub-transaction) performs validity-interval, input, output-size, network-id, and output-value checks, but contains **no** `validateExUnitsTooBigUTxO` call. The `DijkstraSubUtxoPredFailure` type has no `ExUnitsTooBigUTxO` constructor, and the mapping function explicitly marks it as `error "Impossible"`:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [4](#0-3) [5](#0-4) 

**Per-block limit check — also ignores sub-transaction ExUnits:**

The Dijkstra BBODY rule delegates to `Alonzo.validateExUnits`:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
```

`Alonzo.validateExUnits` computes `foldMap totExUnits txs` over the sequence of top-level transactions. Because `totExUnits` ignores sub-transaction witness sets, the block-level `maxBlockExUnits` guard is equally blind to sub-transaction script costs. [6](#0-5) [7](#0-6) 

**Sub-transactions carry real Plutus witness sets:**

The CDDL specification confirms that each `sub_transaction` contains a full `transaction_witness_set` (including redeemers with `ex_units`), and `DijkstraStAnnSubTx` stores `dsastPlutusScriptsWithContext`, confirming that sub-transaction Plutus scripts are collected and executed as part of batch validation. [8](#0-7) [9](#0-8) 

**Fee calculation also ignores sub-transaction ExUnits:**

`getConwayMinFeeTx` (reused by Dijkstra) calls `alonzoMinFeeTx`, which computes `txscriptfee (pp ^. ppPricesL) allExunits` where `allExunits = totExUnits tx`. Sub-transaction script fees are therefore not charged to the submitter. [10](#0-9) 

---

### Impact Explanation

An attacker submits a single top-level transaction embedding N sub-transactions, each declaring `ExUnits` just below `maxTxExUnits`. The aggregate script execution cost for the batch is `N × maxTxExUnits`, yet:

1. The per-transaction check passes because only the top-level transaction's (possibly zero) ExUnits are tested.
2. The per-block check passes because `foldMap totExUnits` over the block's top-level transactions also ignores sub-transaction ExUnits.
3. No additional fee is charged for the sub-transaction script execution.

This allows attacker-controlled transactions to exceed the intended per-transaction and per-block validation resource limits (`maxTxExUnits`, `maxBlockExUnits`), causing block validation to consume unbounded CPU and memory. This maps to the **Medium** allowed impact: *"Attacker-controlled transactions exceed intended validation limits."*

---

### Likelihood Explanation

The Dijkstra era introduces sub-transactions as a first-class feature. Any unprivileged transaction sender can embed sub-transactions with arbitrary Plutus witness sets. The only cost is the top-level transaction fee (which does not include sub-transaction ExUnits). The attack requires no special access, no governance majority, and no key compromise. Likelihood is **Medium-High** once the Dijkstra era is active on mainnet.

---

### Recommendation

1. **Extend `totExUnits` (or introduce a batch-aware variant)** to recursively sum ExUnits across all sub-transactions in a batch:

   ```haskell
   totBatchExUnits :: (EraTx era, AlonzoEraTxWits era, DijkstraEraTxBody era)
                   => Tx TopTx era -> ExUnits
   totBatchExUnits tx =
     totExUnits tx
       <> foldMap totExUnits (OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL)
   ```

2. **Apply the per-transaction limit to each sub-transaction individually** in `dijkstraSubUtxoTransition`, analogous to how `validateExUnitsTooBigUTxO` is applied to top-level transactions.

3. **Replace `totExUnits` with `totBatchExUnits`** in both `dijkstraUtxoTransition` (per-transaction check) and `Alonzo.validateExUnits` (per-block check) for the Dijkstra era.

4. **Include sub-transaction ExUnits in the minimum fee calculation** so that script execution costs are fully charged to the submitter.

---

### Proof of Concept

1. Construct a Dijkstra top-level transaction `txTop` with an empty redeemer set (zero ExUnits at the top level).
2. Embed `K` sub-transactions `[subTx_1 … subTx_K]` in `txTop ^. bodyTxL . subTransactionsTxBodyL`, each with a Plutus script redeemer declaring `ExUnits { exUnitsMem = M, exUnitsSteps = S }` where `M × K >> maxBlockExUnits.exUnitsMem`.
3. Submit `txTop` to the mempool. The `MEMPOOL` rule delegates to `LEDGER → UTXO`, which calls `validateExUnitsTooBigUTxO pp txTop`. Since `totExUnits txTop = 0` (no top-level redeemers), the check passes.
4. A block producer includes `txTop`. The `BBODY` rule calls `Alonzo.validateExUnits [txTop] maxBlockExUnits`. Again `totExUnits txTop = 0`, so the block-level check passes.
5. Each sub-transaction is processed by `dijkstraSubUtxoTransition`, which performs no ExUnits check.
6. The aggregate Plutus execution cost for the block is `K × (M steps + S mem)`, far exceeding `maxBlockExUnits`, causing block validation to consume unbounded resources. [11](#0-10) [4](#0-3) [12](#0-11)

validateExUnitsTooBigUTxO pp tx =
  failureUnless (pointWiseExUnits (<=) totalExUnits maxTxExUnits) $
    ExUnitsTooBigUTxO Mismatch {mismatchSupplied = totalExUnits, mismatchExpected = maxTxExUnits}
  where
    maxTxExUnits = pp ^. ppMaxTxExUnitsL
    -- This sums up the ExUnits for all embedded Plutus Scripts anywhere in the transaction:
    totalExUnits = totExUnits tx
