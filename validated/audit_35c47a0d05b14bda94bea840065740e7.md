Audit Report

## Title
LP Funds Permanently Locked When Position Owner Is USDC/USDT-Blacklisted in `removeLiquidity` — (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
`LiquidityLib.removeLiquidity` unconditionally pushes owed tokens to `owner` via `safeTransfer`. If `owner` is blacklisted by USDC or USDT after depositing, every `removeLiquidity` call reverts, and the LP's shares — along with both tokens — are permanently locked in the pool with no on-chain recovery path.

## Finding Description
After computing scaled amounts across all bins, `removeLiquidity` transfers directly to `owner`:

```solidity
// LiquidityLib.sol L242-247
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
}
```

USDC's blacklist causes `transfer` to revert for the blacklisted receiver. This rolls back the entire transaction, including all bin-state mutations computed in the loop above (`binState.token0BalanceScaled`, `binState.token1BalanceScaled`, `binTotalShares`, `positionBinShares`).

`MetricOmmPool.removeLiquidity` enforces `msg.sender == owner` at line 206, so the LP cannot delegate withdrawal to a clean address. Positions are keyed by `keccak256(abi.encode(owner, salt, bin))` with no transfer or reassignment mechanism anywhere in the codebase. The extension hooks (`_beforeRemoveLiquidity`, `_afterRemoveLiquidity`) accept no `recipient` parameter and cannot redirect the token push. There is no pull-based claim function. All recovery paths are closed.

A secondary effect: if token0 is USDC and the LP is blacklisted, the token0 transfer fails first, which also prevents the token1 (e.g., WETH) transfer — the LP loses access to both tokens even if token1 is not USDC.

## Impact Explanation
The LP's shares remain recorded in `_positionBinShares` and `_binTotalShares` but can never be redeemed. The underlying token balances are locked inside the pool's bins indefinitely. This is a direct, permanent loss of user principal with no on-chain recovery path, meeting the "direct loss of user principal" and "unusable withdraw/liquidity flows" criteria.

## Likelihood Explanation
USDC blacklisting is a documented real-world event (OFAC-sanctioned addresses). An LP can be blacklisted at any time after depositing. The pool has no mechanism to detect this condition in advance or to route tokens to an alternative address after the fact. Likelihood is low but impact is total loss of principal, placing this at **Medium** severity under Sherlock's framework.

## Recommendation
Replace the push pattern with a pull pattern: record owed amounts in a claimable mapping and let the LP (or any address they authorize) call a separate `claim` function that transfers to an arbitrary `recipient`. Alternatively, add an explicit `recipient` address parameter to `removeLiquidity` distinct from `owner`, so a blacklisted LP can direct funds to a clean address.

## Proof of Concept
1. Alice provides liquidity to a USDC/WETH pool, creating position `(alice, salt, bin)`.
2. Alice's address is later added to USDC's blacklist.
3. Alice calls `removeLiquidity`; the pool computes `amount0Removed` (USDC) and `amount1Removed` (WETH).
4. `IERC20(ctx.token0).safeTransfer(alice, amount0Removed)` reverts — USDC rejects transfers to blacklisted addresses.
5. The entire transaction reverts; Alice's shares remain in `_positionBinShares` unchanged.
6. Alice cannot call `removeLiquidity` from any other address due to the `msg.sender != owner` guard at `MetricOmmPool.sol` L206.
7. Alice's USDC **and** WETH are permanently locked in the pool.
