Audit Report

## Title
Nested Unbounded Loop in `NodeDelegator.getAssetUnstaking()` Propagates to Public `updateRSETHPrice()` and User-Facing `initiateWithdrawal()` — (File: `contracts/NodeDelegator.sol`)

## Summary

`NodeDelegator.getAssetUnstaking()` issues an external call to EigenLayer's `DelegationManager.getQueuedWithdrawals()` and iterates over all returned withdrawals with a nested loop over strategies. This function is invoked for every NDC inside `LRTDepositPool.getAssetDistributionData()`, which is itself called for every supported asset inside `LRTOracle._getTotalEthInProtocol()` (reached via the public `updateRSETHPrice()`) and for the target asset inside `LRTWithdrawalManager.getAvailableAssetAmount()` (reached via the user-facing `initiateWithdrawal()`). As the number of supported assets, NDCs, and concurrent EigenLayer queued withdrawals grows through normal protocol operation, the cumulative gas cost of these call chains can exhaust the block gas limit, causing both functions to revert.

## Finding Description

**Root cause — `NodeDelegator.getAssetUnstaking()` (L405–427):**

```solidity
function getAssetUnstaking(address asset) external view returns (uint256 amount) {
    (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
        _getDelegationManager().getQueuedWithdrawals(address(this));   // external call per invocation

    for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
        for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
            ...
        }
    }
}
```

The external call `getQueuedWithdrawals` loads all queued withdrawal structs from EigenLayer storage for the NDC. The nested loop then iterates over every strategy in every withdrawal. There is no pagination or early-exit mechanism.

**Propagation — `LRTDepositPool.getAssetDistributionData()` (L446–456):**

```solidity
for (uint256 i; i < ndcsCount;) {
    assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset);
    ...
}
```

`getAssetUnstaking` is called once per NDC per asset. With `maxNodeDelegatorLimit` defaulting to 10 (admin-adjustable upward), this multiplies the cost by the NDC count.

**Path 1 — Public `updateRSETHPrice()` via `_getTotalEthInProtocol()` (L331–349):**

```solidity
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    ...
}
```

`getTotalAssetDeposits` → `getAssetDistributionData` → `getAssetUnstaking` is executed for **every supported asset**. `_addNewSupportedAsset` (L106–118) has no cap on `supportedAssetList`, so the asset dimension is unbounded.

**Path 2 — User-facing `initiateWithdrawal()` via `getAvailableAssetAmount()` (L599–603):**

```solidity
uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
```

Every user withdrawal initiation triggers the full NDC × queued-withdrawal × strategy loop for the target asset.

**Total gas complexity:** O(N\_assets × N\_NDCs × N\_queued\_withdrawals × N\_strategies\_per\_withdrawal), where each inner iteration includes at least one SLOAD and potentially an external `sharesToUnderlyingView` call. The `getQueuedWithdrawals` external call itself is issued N\_assets × N\_NDCs times in the `updateRSETHPrice` path.

**Existing guards are insufficient:** `maxUncompletedWithdrawalCount` bounds queued withdrawals per NDC at the protocol level, but does not prevent the multiplicative blowup across assets and NDCs. The `maxNodeDelegatorLimit` is admin-adjustable upward. Neither guard prevents the combined product from exceeding the block gas limit.

## Impact Explanation

- **Medium — Unbounded gas consumption:** The gas cost of `updateRSETHPrice()` and `initiateWithdrawal()` grows multiplicatively with protocol scale. At realistic operational parameters (5 assets, 10 NDCs, 20 queued withdrawals each with 2 strategies), the `updateRSETHPrice` path issues 100 external calls to `getQueuedWithdrawals` plus 2000 inner loop iterations, each with storage reads and potential external calls. This can approach or exceed the 30M block gas limit.
- **Medium — Temporary freezing of funds:** If `initiateWithdrawal()` consistently OOGs, users cannot queue new withdrawals until the number of queued EigenLayer withdrawals decreases (i.e., until operators call `completeUnstaking`). This is a temporary but operationally significant freeze.

## Likelihood Explanation

Both trigger conditions are reachable by any unprivileged caller: `updateRSETHPrice()` is public with no access control, and `initiateWithdrawal()` is the standard user withdrawal entry point. The growth conditions (more supported assets, more NDCs, more concurrent EigenLayer unstaking operations) arise from normal protocol operation — operators legitimately call `initiateUnstaking` repeatedly as part of restaking management, accumulating queued withdrawals. This is not a theoretical edge case but an expected operational state as the protocol scales.

## Recommendation

1. **Track unstaking amounts in protocol storage:** Update a per-NDC per-asset `unstakingAmount` mapping on `initiateUnstaking` and `completeUnstaking` events, and read from storage in `getAssetUnstaking()` instead of querying EigenLayer on every call. This eliminates the external call and nested loop entirely from the read path.
2. **Cap `supportedAssetList`:** Introduce an explicit maximum in `_addNewSupportedAsset` to bound the asset dimension in `_getTotalEthInProtocol`.
3. **Decouple price update from full distribution computation:** `updateRSETHPrice()` does not need the full `getAssetDistributionData` breakdown; a separate, cheaper accounting path for total ETH value would avoid the per-NDC external calls.

## Proof of Concept

**Setup:**
- 5 supported assets in `lrtConfig.supportedAssetList`
- 10 NDCs in `nodeDelegatorQueue`
- Each NDC has 20 queued EigenLayer withdrawals, each with 2 strategies

**Trigger `updateRSETHPrice()`:**
1. Call `LRTOracle.updateRSETHPrice()` (public, no access control).
2. Internally: `_getTotalEthInProtocol()` loops over 5 assets → for each asset, `getTotalAssetDeposits(asset)` → `getAssetDistributionData(asset)` → loops over 10 NDCs → `getAssetUnstaking(asset)` → `getQueuedWithdrawals(ndcAddr)` (external call, loads 20 withdrawal structs) + 20 × 2 = 40 inner iterations.
3. Total: 5 × 10 = 50 external `getQueuedWithdrawals` calls + 5 × 10 × 40 = 2000 inner loop iterations, each with SLOADs and potential `sharesToUnderlyingView` external calls.

**Trigger `initiateWithdrawal(asset, rsETHAmount, referralId)`:**
1. Any user calls `initiateWithdrawal` for any supported asset.
2. Internally: `getAvailableAssetAmount(asset)` → `getTotalAssetDeposits(asset)` → `getAssetDistributionData(asset)` → 10 NDC iterations → 10 external `getQueuedWithdrawals` calls + 10 × 40 = 400 inner iterations.
