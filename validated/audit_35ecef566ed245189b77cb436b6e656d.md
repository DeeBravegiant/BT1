Audit Report

## Title
`isWithdrawalClaimable` Returns `true` for Already-Claimed Withdrawals - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`isWithdrawalClaimable` evaluates only the time condition (`block.timestamp >= withdrawal.unlockTime`) and never checks `withdrawal.claimed`. After a successful `claimWithdrawal` call sets `claimed = true`, the stale mapping entry causes `isWithdrawalClaimable` to permanently return `true` for that ID, contradicting the separate `isWithdrawalClaimed` view and misleading any caller that relies on it to gate a subsequent `claimWithdrawal` call.

## Finding Description
`claimWithdrawal` (L344–379) enforces two independent conditions before transferring tokens: the unlock time must have passed (L355–357) and the withdrawal must not have been previously claimed (L359–361). On success it sets `withdrawal.claimed = true` (L363) and removes the ID from `userWithdrawalIds`, but the `withdrawals` mapping entry is never deleted.

`isWithdrawalClaimable` (L531–533) reads only the time field:

```solidity
function isWithdrawalClaimable(uint256 _withdrawalId) external view returns (bool) {
    return block.timestamp >= withdrawals[_withdrawalId].unlockTime;
}
```

Because `claimed` is never consulted, the function returns `true` for any withdrawal whose unlock time has passed, regardless of whether it was already claimed. The parallel view `isWithdrawalClaimed` (L540–542) correctly reads `withdrawal.claimed`, creating a direct contradiction: one view says "claimable", the other says "already claimed."

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

No funds are at risk; `claimWithdrawal` correctly prevents double-claims. The broken promise is informational: the view function `isWithdrawalClaimable` is documented as returning "whether the withdrawal is ready to be claimed," but it returns `true` for IDs that are permanently unclaimable. Any wallet UI, aggregator, or smart-contract integration that calls `isWithdrawalClaimable` before calling `claimWithdrawal` will receive a misleading signal and encounter an unexpected `WithdrawalAlreadyClaimed` revert.

## Likelihood Explanation
The inconsistency is triggered by a completely normal, unprivileged user action: querying the status of a withdrawal after having already claimed it. No special role, timing attack, or external dependency is required. Every user who claims a withdrawal and later re-checks its status will encounter the discrepancy, making this a near-certain occurrence in production.

## Recommendation
Add the `claimed` check inside `isWithdrawalClaimable` to mirror the two-condition guard already present in `claimWithdrawal`:

```solidity
function isWithdrawalClaimable(uint256 _withdrawalId) external view returns (bool) {
    Withdrawal storage withdrawal = withdrawals[_withdrawalId];
    return !withdrawal.claimed && block.timestamp >= withdrawal.unlockTime;
}
```

## Proof of Concept

1. Alice calls `initiateWithdrawal(100e18)`, receiving `withdrawalId = 1`.
2. Time advances past `withdrawal.unlockTime`.
3. Alice calls `claimWithdrawal(1)`. Succeeds: `withdrawals[1].claimed` is set to `true`, `1` is removed from `userWithdrawalIds[Alice]`, and 100 KERNEL is transferred.
4. Alice (or any observer) calls `isWithdrawalClaimable(1)`. Returns `true` because only `block.timestamp >= withdrawals[1].unlockTime` is evaluated — still satisfied.
5. Alice calls `claimWithdrawal(1)` again. Reverts with `WithdrawalAlreadyClaimed`.

The view function promised the withdrawal was claimable; the state machine disagreed.
