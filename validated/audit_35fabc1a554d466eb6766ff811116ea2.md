### Title
Unhandled `RuntimeError::ReceiptValidationError` Causes Node-Level `panic!` During Chunk Application — (`File: chain/chain/src/runtime/mod.rs`)

### Summary

When `Runtime::apply()` returns `RuntimeError::ReceiptValidationError`, the nearcore runtime adapter unconditionally calls `panic!("{}", e)` instead of handling the error gracefully. Under `AccountIdValidityRulesVersion::V0` or `V1` (protocol versions < 83), an unprivileged user can deploy a contract that calls `promise_batch_create` with an invalid account ID string. Because `read_and_parse_account_id` uses `AccountId::new_unvalidated()` under those versions, the receipt is stored in chain state with an invalid `receiver_id`. When that receipt is later applied — even by a node running current code — `validate_receipt` returns `ReceiptValidationError::InvalidReceiverId`, which propagates as `RuntimeError::ReceiptValidationError` and hits the `panic!`, crashing the node and halting block production.

### Finding Description

**Root cause — the unconditional `panic!`:**

In `chain/chain/src/runtime/mod.rs`, the `process_state_update` call site maps `RuntimeError` variants to chain errors. Two variants are explicitly marked with `// TODO(#2152): process gracefully` and both call `panic!`:

```rust
// TODO(#2152): process gracefully
RuntimeError::UnexpectedIntegerOverflow(reason) => {
    panic!("RuntimeError::UnexpectedIntegerOverflow {reason}")
}
RuntimeError::StorageError(e) => Error::StorageError(e),
// TODO(#2152): process gracefully
RuntimeError::ReceiptValidationError(e) => panic!("{}", e),
``` [1](#0-0) 

**How an invalid account ID enters a receipt:**

`VMLogic::read_and_parse_account_id` (and its wasmtime counterpart) branches on `account_id_validity_rules_version`. Under `V0` or `V1`, it calls the deprecated `AccountId::new_unvalidated()`, accepting any arbitrary string as an account ID without validation:

```rust
match self.config.limit_config.account_id_validity_rules_version {
    AccountIdValidityRulesVersion::V0
    | AccountIdValidityRulesVersion::V1 => {
        #[allow(deprecated)]
        Ok(AccountId::new_unvalidated(account_id_str))
    }
    AccountIdValidityRulesVersion::V2 => account_id_str
        .parse()
        .map_err(|_| VMLogicError::HostError(HostError::InvalidAccountId)),
}
``` [2](#0-1) 

`promise_batch_create` calls `read_and_parse_account_id` and then passes the result directly to `ext.create_action_receipt`, storing the receipt with the unvalidated `receiver_id` in chain state: [3](#0-2) 

**How the stored receipt triggers the panic:**

`validate_receipt` (called during `Runtime::apply()`) unconditionally runs `AccountId::validate()` on `predecessor_id` and `receiver_id` regardless of `ValidateReceiptMode`. The code comment explicitly acknowledges this is a backwards-compatibility check for receipts created via the unvalidated path:

```rust
// We retain these checks here as to maintain backwards compatibility
// with AccountId validation since we illegally parse an AccountId
// in near-vm-logic/logic.rs#fn(VMLogic::read_and_parse_account_id)
AccountId::validate(receipt.predecessor_id().as_ref()).map_err(|_| {
    ReceiptValidationError::InvalidPredecessorId { ... }
})?;
AccountId::validate(receipt.receiver_id().as_ref()).map_err(|_| {
    ReceiptValidationError::InvalidReceiverId { ... }
})?;
``` [4](#0-3) 

When this returns `Err(ReceiptValidationError::InvalidReceiverId {...})`, the runtime propagates it as `RuntimeError::ReceiptValidationError(e)`, which hits the `panic!` in the runtime adapter.

**Protocol version gate:**

`AccountIdValidityRulesVersion::V2` was activated at protocol version 83: [5](#0-4) 

V1 was activated at protocol version 57: [6](#0-5) 

Any receipt created by a contract under protocol version < 83 with an invalid account ID passed to `promise_batch_create` or `promise_create` remains in chain state and will trigger the panic when applied by any node, including nodes running current code.

### Impact Explanation

The `panic!` at `chain/chain/src/runtime/mod.rs:367` terminates the node process during chunk application. This halts block production and finality advancement for the affected node. If the malicious receipt is included in a chunk that all validators must apply, the panic propagates across all validators processing that chunk, causing a network-wide halt. The corrupted protocol value is **finality/head**: no node that hits the panic can advance its chain head past the block containing the malicious receipt.

### Likelihood Explanation

Under protocol versions < 83 (V0/V1), any user who deployed a contract and called `promise_batch_create` with an invalid account ID string (e.g., `"not valid"`, `"a"`, `"UPPER"`) created a receipt with an invalid `receiver_id` in chain state. Such receipts, if present in the state of a chain that has been running since before protocol version 83, will trigger the panic when applied. The `panic!` path is explicitly marked as unhandled (`TODO(#2152): process gracefully`), confirming it is a known gap.

### Recommendation

Replace the `panic!` with a graceful error return:

```rust
RuntimeError::ReceiptValidationError(e) => {
    tracing::error!(?e, "receipt validation error during apply");
    Err(Error::Other(format!("ReceiptValidationError: {e}")))
}
```

This mirrors how `StorageError` is handled. Additionally, the `TODO(#2152)` for `UnexpectedIntegerOverflow` should be resolved similarly.

### Proof of Concept

1. On a network running protocol version < 83 (V0 or V1 `account_id_validity_rules_version`), deploy a contract with a function that calls:
   ```rust
   promise_batch_create(b"not a valid account id!!".len() as u64,
                        b"not a valid account id!!".as_ptr() as u64)
   ```
2. Call the function. Under V0/V1, `read_and_parse_account_id` accepts the string via `new_unvalidated` and `create_action_receipt` stores a receipt with `receiver_id = "not a valid account id!!"` in chain state.
3. When any node (including one running current code) applies the chunk containing that receipt, `validate_receipt` calls `AccountId::validate("not a valid account id!!")`, which fails, returning `ReceiptValidationError::InvalidReceiverId`.
4. `Runtime::apply()` returns `RuntimeError::ReceiptValidationError(e)`.
5. The match arm at `chain/chain/src/runtime/mod.rs:367` executes `panic!("{}", e)`, crashing the node. [7](#0-6) [8](#0-7)

/// Validates a given receipt. Checks validity of the Action or Data receipt.
pub(crate) fn validate_receipt(
    limit_config: &LimitConfig,
    receipt: &Receipt,
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ReceiptValidationError> {
    if mode == ValidateReceiptMode::NewReceipt {
        let receipt_size: u64 =
            borsh::object_length(receipt).unwrap().try_into().expect("Can't convert usize to u64");
        if receipt_size > limit_config.max_receipt_size {
            return Err(ReceiptValidationError::ReceiptSizeExceeded {
                size: receipt_size,
                limit: limit_config.max_receipt_size,
            });
        }
    }

    // We retain these checks here as to maintain backwards compatibility
    // with AccountId validation since we illegally parse an AccountId
    // in near-vm-logic/logic.rs#fn(VMLogic::read_and_parse_account_id)
    AccountId::validate(receipt.predecessor_id().as_ref()).map_err(|_| {
        ReceiptValidationError::InvalidPredecessorId {
            account_id: receipt.predecessor_id().to_string(),
        }
    })?;
    AccountId::validate(receipt.receiver_id().as_ref()).map_err(|_| {
        ReceiptValidationError::InvalidReceiverId { account_id: receipt.receiver_id().to_string() }
    })?;

    match receipt.versioned_receipt() {
        VersionedReceiptEnum::Action(action_receipt)
        | VersionedReceiptEnum::PromiseYield(action_receipt) => validate_action_receipt(
            limit_config,
            action_receipt,
            receipt.receiver_id(),
            current_protocol_version,
            mode,
        ),
        VersionedReceiptEnum::Data(data_receipt)
        | VersionedReceiptEnum::PromiseResume(data_receipt) => {
            validate_data_receipt(limit_config, &data_receipt)
        }
        VersionedReceiptEnum::GlobalContractDistribution(_) => Ok(()), // Distribution receipt can't be issued without a valid contract
    }
}
