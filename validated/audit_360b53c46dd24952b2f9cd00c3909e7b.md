### Title
Mempool Eviction Policy Ignores Fee Priority, Enabling Admission DoS via Gap-Free Transaction Flooding — (File: `crates/apollo_mempool/src/mempool.rs`)

---

### Summary

The mempool's capacity-overflow eviction mechanism exclusively targets "gap accounts" (accounts whose lowest pooled nonce exceeds the on-chain account nonce). It never evicts low-fee, gap-free transactions to make room for higher-fee ones. An unprivileged attacker who fills the mempool with minimum-fee, gap-free transactions from many accounts renders the entire pool non-evictable, causing every subsequent admission attempt from a new account to return `MempoolError::MempoolFull` regardless of the incoming transaction's fee.

---

### Finding Description

**Root cause — `get_evictable_account` / `try_make_space`**

`get_evictable_account` draws exclusively from `accounts_with_gap`:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 981-988
pub fn get_evictable_account(&self) -> Option<ContractAddress> {
    let len = self.accounts_with_gap.len();
    if len == 0 {
        return None;          // ← no gap accounts → nothing to evict
    }
    let random_index = rng().random_range(0..len);
    self.accounts_with_gap.get_index(random_index).copied()
}
```

`try_make_space` loops over this function and returns `false` the moment the set is empty:

```rust

        break;   // ← exits immediately if accounts_with_gap is empty
    };
    ...
}

```

`handle_capacity_overflow` then rejects the incoming transaction:

```rust
// lines 1031-1052
if !creating_gap && self.try_make_space(required_space) {
    return Ok(());
}
Err(MempoolError::MempoolFull)
```

**Attack path**

A transaction submitted with `tx_nonce == account_nonce` (the normal, gap-free case) sets `creating_gap = false`, so `try_make_space` is called. If `accounts_with_gap` is empty, `try_make_space` returns `false` and the transaction is rejected — regardless of its fee.

An attacker who pre-fills the mempool with minimum-fee, gap-free transactions from many distinct accounts ensures `accounts_with_gap` stays empty. Every subsequent admission attempt from a new account is rejected with `MempoolFull`, even if the incoming transaction carries a much higher fee.

**Secondary issue — irreversible eviction on rejection**

The test suite explicitly documents a second related flaw:

```rust
// crates/apollo_mempool/src/fee_mempool_test.rs  lines 1936-1938
// Transactions tx1 and tx2 are evicted regardless of whether large_tx is accepted or rejected.
// We do not revert the eviction attempt even if adding large_tx ultimately fails.
assert!(!mempool.tx_pool.contains_account(contract_address!("0x1")));
```

An attacker can submit a transaction large enough to trigger eviction of gap-account transactions but too large to fit after eviction. The gap-account transactions are permanently removed; the attacker's transaction is rejected. This drains the pool of gap-account entries without any replacement.

---

### Impact Explanation

**High — Mempool/gateway admission rejects valid transactions before sequencing.**

When the mempool is full of non-evictable, gap-free transactions, every new-account transaction is rejected with `MempoolFull` irrespective of its fee. This breaks the invariant that higher-fee transactions should be admitted over lower-fee ones at capacity, and directly prevents legitimate user transactions from reaching the sequencer.

---

### Likelihood Explanation

**Low.** Filling the default 1 GB mempool requires many funded accounts and continuous transaction submission (transactions expire via `transaction_ttl`). The economic cost is proportional to the mempool size. However, the structural flaw is unconditional: no code path evicts a gap-free transaction based on fee comparison, so the invariant is broken by design whenever the attacker can sustain the fill rate.

---

### Recommendation

1. **Fee-based eviction**: when `accounts_with_gap` is empty and the mempool is full, evict the lowest-fee transaction from the pool (regardless of gap status) to make room for a higher-fee incoming transaction.
2. **Rollback evictions on rejection**: if `try_make_space` evicts transactions but the incoming transaction is ultimately rejected, restore the evicted transactions to the pool rather than permanently discarding them.

---

### Proof of Concept

1. Create *N* accounts, each funded with enough STRK to pay the minimum fee.
2. From each account, submit one transaction with `tx_nonce = account_nonce` (gap-free) and the minimum acceptable fee.
3. Repeat until `size_in_bytes() >= capacity_in_bytes` (default 1 GB).
4. Submit a high-fee transaction from a fresh account (`tx_nonce = account_nonce`).
5. Observe `MempoolError::MempoolFull` returned despite the high fee, because `accounts_with_gap` is empty and `try_make_space` returns `false`.

```rust
    pub fn try_make_space(&mut self, required_space: u64) -> bool {
        let mut total_space_freed = 0;
        let mut evicted_txs = Vec::new();

        while total_space_freed < required_space {
            let Some(address) = self.get_evictable_account() else {
                break;
            };

            let txs: Vec<_> = self.tx_pool.account_txs_sorted_by_nonce(address).copied().collect();
            for tx_ref in txs.iter().rev() {
                let tx = self
                    .tx_pool
                    .remove(tx_ref.tx_hash)
                    .expect("Transaction must exist in the pool.");
                total_space_freed += tx.total_bytes();
                evicted_txs.push(*tx_ref);
                metric_count_evicted_txs(1);
                self.decrement_stuck_txs_if_gap_account(address, 1);
                if total_space_freed >= required_space {
                    break;
                }
            }

            // Clean up if account is now empty.
            if !self.tx_pool.contains_account(address) {
                self.accounts_with_gap.swap_remove(&address);
            }
        }

        // Keep the queue consistent with the pool: the evicted txs were removed from the pool, so
        // drop their (now-orphaned) queue references too. In fee mode gap accounts are never
        // queued, so this is a no-op there; in Echonet/FIFO mode they are, and skipping this leaves
        // a dangling reference that panics the next `get_txs`.
        self.tx_queue.remove_txs(&evicted_txs);

        total_space_freed >= required_space
    }
```

```rust
    fn handle_capacity_overflow(
        &mut self,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
        freed_bytes: u64,
    ) -> Result<(), MempoolError> {
        let address = tx.contract_address();

        let account_has_gap = self.accounts_with_gap.contains(&address);
        let account_has_txs = self.tx_pool.contains_account(address);
        let closing_gap = tx.nonce() == account_nonce;
        let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;

        // Only the net growth must be evicted: an accompanying replacement removal frees
        // `freed_bytes` (0 when there is no replacement).
        let required_space = tx.total_bytes().saturating_sub(freed_bytes);
        if !creating_gap && self.try_make_space(required_space) {
            return Ok(());
        }

        Err(MempoolError::MempoolFull)
    }
```
