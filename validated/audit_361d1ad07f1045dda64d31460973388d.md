### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund in Dijkstra Era — (`eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` / `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`)

---

### Summary

The Dijkstra era introduces nested (sub-)transactions: a single top-level `Tx TopTx` may embed an ordered set of `Tx SubTx` objects in its body. Each sub-transaction is processed sequentially by the `SUBLEDGERS` rule. The predicate that prevents two sub-transactions within the same top-level transaction from each claiming a deposit refund for the **same staking credential** is explicitly acknowledged as **not yet implemented**. An attacker can therefore embed two sub-transactions that both carry `UnRegDepositTxCert` for the same credential, causing the deposit pot to be debited twice while the credential is only deregistered once, directly destroying ADA from the deposit pot.

---

### Finding Description

The Dijkstra era `TxBody TopTx` carries a field `dtbrSubTransactions :: OMap TxId (Tx SubTx era)` — an ordered map of sub-transactions that are processed before the top-level transaction body. [1](#0-0) 

Each sub-transaction body (`DijkstraSubTxBodyRaw`) may contain its own `dstbrCerts` field, including deregistration certificates. [2](#0-1) 

The top-level ledger transition (`dijkstraLedgerTransition`) processes all sub-transactions first via the `SUBLEDGERS` rule, threading the resulting `LedgerState` forward: [3](#0-2) 

Within `SUBLEDGERS`, each sub-transaction is processed by `SUBLEDGER`, which calls `updateUTxOStateNoFees`. That function computes the deposit refund for the sub-transaction body using `certsTotalRefundsTxBody pp certState txBody`: [4](#0-3) 

`certsTotalRefundsTxBody` resolves the refund amount by calling `lookupDepositDState` on the current `CertState`. The critical gap is that **no predicate checks whether the same credential's deposit has already been claimed by a prior sub-transaction in the same batch**. The developers have explicitly acknowledged this gap with a disabled (`xit`) test and a `TODO` comment: [5](#0-4) 

The test constructs two sub-transactions (`subTx1`, `subTx2`) each containing `UnRegDepositTxCert stakingCred keyDeposit` for the **same** credential, bundles them into one top-level transaction, and expects a predicate failure — but the expected failure is `error "TODO: predicate failure not yet implemented"`, meaning the check does not exist in the production code.

The analog to the Nibiru vulnerability is direct:

| Nibiru | Cardano Dijkstra |
|---|---|
| Multiple `MsgEthereumTx` in one SDK tx | Multiple `Tx SubTx` in one `Tx TopTx` |
| Contract-creation msg resets nonce; call msg does not | First sub-tx deregisters credential; second sub-tx also claims refund for same credential |
| Nonce not incremented for call msgs → replay | Deposit not removed from pot after first sub-tx → double-refund |
| Attacker replays last N messages | Attacker claims deposit refund N times |

---

### Impact Explanation

**Critical — Direct loss of ADA through an invalid ledger state transition.**

An attacker who has a registered staking credential (deposit `D`) can submit a single top-level Dijkstra transaction containing `k` sub-transactions each carrying `UnRegDepositTxCert stakingCred D`. If the `SUBLEDGERS` rule does not enforce that a credential's deposit can only be refunded once per batch, the deposit pot (`utxosDeposited`) is decremented by `k × D` while only one deregistration actually occurs. The excess `(k-1) × D` lovelace is effectively created out of thin air in the outputs of the sub-transactions, violating ADA conservation. [6](#0-5) 

---

### Likelihood Explanation

The Dijkstra era is the newest era in this repository and sub-transactions are a new feature. The test is already written and explicitly marks the missing guard as a `TODO`. Any user who can submit a Dijkstra-era transaction (an unprivileged transaction sender) can trigger this path. No special keys, governance majority, or privileged access is required — only the ability to construct a valid top-level transaction with multiple sub-transactions. [7](#0-6) 

---

### Recommendation

In the `SUBLEDGERS` (or `SUBLEDGER`) rule, maintain a set of credentials whose deposits have already been claimed within the current sub-transaction batch. Before processing each sub-transaction's `UnRegDepositTxCert`, verify that the credential is not already in that set. If it is, raise a predicate failure (the `TODO` in `CertSpec.hs` is the correct place to wire this up). Alternatively, ensure the `CertState` is fully committed after each sub-transaction so that the second `UnRegDepositTxCert` for the same credential fails the existing `lookupDepositDState` check (returning `Nothing` because the credential is already deregistered). [8](#0-7) 

---

### Proof of Concept

1. Register staking credential `C`, paying deposit `D`. The deposit pot increases by `D`.
2. Construct sub-transaction `subTx1` with `UnRegDepositTxCert C D`.
3. Construct sub-transaction `subTx2` with `UnRegDepositTxCert C D` (same credential, same deposit amount).
4. Construct top-level transaction `tx` with `subTransactionsTxBodyL = OMap.fromFoldable [subTx1, subTx2]`.
5. Submit `tx`. Because the predicate failure is not implemented, both sub-transactions pass validation.
6. The deposit pot is decremented by `2 × D`; the attacker's outputs receive `2 × D` in refunds while only `D` was ever deposited.
7. Net effect: `D` lovelace is created from nothing, violating ADA conservation.

The developers' own test skeleton confirms this scenario is reachable and unguarded: [5](#0-4)
