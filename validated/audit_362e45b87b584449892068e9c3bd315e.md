### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect Sequencer Fees and Corrupt Block Hash Sequencer Address — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates many `ProposalInit` fields but never checks `builder`. The `builder` field is passed verbatim as `sequencer_address` into `BlockInfo`, which the batcher uses both for fee collection during execution and for computing `PartialBlockHashComponents`. A malicious proposer can set `builder` to an arbitrary address; validators accept the proposal, fees flow to the attacker's address, and the committed block hash encodes the wrong sequencer.

---

### Finding Description

`is_proposal_init_valid` in `validate_proposal.rs` enforces `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices (FRI + WEI), `starknet_version`, `version_constant_commitment`, `timestamp`, and `fee_proposal_fri`. The `builder` field is absent from every check. [1](#0-0) 

`builder` is then forwarded without any guard into `convert_to_sn_api_block_info`, which maps it directly to `sequencer_address`: [2](#0-1) 

That `BlockInfo` is handed to the batcher via `ValidateBlockInput`: [3](#0-2) 

Inside the batcher, `PartialBlockHashComponents::new` captures `sequencer_address` as the `sequencer` field that enters the Poseidon block-hash chain: [4](#0-3) 

The block hash formula chains `sequencer_address` at a fixed position: [5](#0-4) 

The final `proposal_commitment` is `Poseidon(partial_block_hash, fee_proposal_fri)`: [6](#0-5) 

Because the validator re-derives `batcher_block_commitment` from the batcher's output (which already used the attacker-supplied `builder`), the commitment check at line 244 passes: [7](#0-6) 

The proposer also computed `fin.proposal_commitment` using the same `builder`, so both sides agree. Consensus finalises the block, and `decision_reached` commits it: [8](#0-7) 

---

### Impact Explanation

**Fee theft (Critical – economic impact).** Every transaction in the block pays its fee to `sequencer_address`. With `builder` pointing to the attacker's address, all sequencer revenue for that block is permanently redirected. This matches "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."

**Wrong committed state (Critical).** The block hash encodes the attacker's address as `sequencer`. Once committed, the on-chain block hash is irrevocably wrong. This matches "Wrong state … from blockifier/syscall/execution logic for accepted input."

---

### Likelihood Explanation

Any validator that wins a proposer slot can exploit this. No special network position, privileged access, or off-path capability is required — the attacker simply sets `builder` to their own address in the `ProposalInit` they broadcast. The `proposer` field is validated (must match the committee-elected proposer), but `builder` is not, so the attack is available to any honest-but-greedy or malicious elected proposer.

---

### Recommendation

Add a check inside `is_proposal_init_valid` that `init_proposed.builder` equals the locally configured `builder_address` (the same value the node would use when it is the proposer). This mirrors the existing pattern for `l2_gas_price_fri` and `l1_da_mode`:

```rust
if init_proposed.builder != expected_builder_address {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            expected_builder_address, init_proposed.builder
        ),
    ));
}
```

`expected_builder_address` should be sourced from the same configuration field that `ProposalBuildArguments::builder_address` reads, ensuring proposer and validator use the same ground truth.

---

### Proof of Concept

1. Attacker is elected proposer for height H, round R.
2. Attacker constructs `ProposalInit` with all legitimate fields **except** `builder = attacker_address`.
3. Attacker broadcasts the proposal stream (`Init` → `Transactions` → `Fin`).
4. Each validator calls `is_proposal_init_valid` — `builder` is never checked. [9](#0-8) 
5. Each validator calls `initiate_validation` → `convert_to_sn_api_block_info` → `BlockInfo { sequencer_address: attacker_address, … }` is sent to the batcher. [10](#0-9) 
6. The batcher executes all transactions; every fee transfer credits `attacker_address`.
7. `PartialBlockHashComponents { sequencer: attacker_address, … }` is hashed into `partial_block_hash`.
8. `batcher_block_commitment == fin.proposal_commitment` (both derived from the same `attacker_address`) → commitment check passes.
9. Consensus reaches quorum; `decision_reached` commits the block.
10. The on-chain block header records `sequencer = attacker_address`; all fees for height H are permanently owned by the attacker.

    async fn decision_reached(
        &mut self,
        height: BlockNumber,
        round: Round,
        commitment: ProposalCommitment,
        wait_for_last_commitment: bool,
    ) -> Result<(), ConsensusError> {
        info!("Finished consensus for height: {height}. Agreed on block: {:#066x}", commitment.0);

        self.interrupt_active_proposal().await;
        let (init, transactions, proposal_id, finished_info) = {
            let mut proposals = self.valid_proposals.lock().unwrap();
            let (init, transactions, proposal_id, finished_info) =
                proposals.get_proposal(&height, &round, &commitment).clone();
            proposals.remove_proposals_below_or_at_height(&height);
            (init, transactions, proposal_id, finished_info)
        };

        let decision_reached_response =
            self.deps.batcher.decision_reached(DecisionReachedInput { proposal_id }).await?;

        // CRITICAL: The block is now committed. This function must not fail beyond this point
        // unless the state is fully reverted, otherwise the node will be left in an
        // inconsistent state.

        self.finalize_decision(
            height,
            &init,
            commitment,
            transactions,
            decision_reached_response,
            finished_info.block_header_commitments.clone(),
            finished_info.l2_gas_used,
            wait_for_last_commitment,
        )
        .await;
