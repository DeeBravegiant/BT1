### Title
Silent Error Discard in `unpack_multi_sets_tupled_msgs` Allows Byzantine Peer to Inject Stale/Partial Inbox State in ECDSA-MPC Signing - (`include-internal/cbmpc/internal/protocol/mpc_job.h`)

---

### Summary

`unpack_multi_sets_tupled_msgs` returns `void` and its internal fold expression silently discards every `error_t` returned by `unpack_msg_ctx`. The calling `group_message` overload therefore always returns `SUCCESS` after the multi-set round, even when inner-message deserialization failed for one or more parties. A Byzantine peer can craft a wire blob whose outer bundle (`split_packed_msgs`) deserializes cleanly but whose inner payload causes `unpack_msg_ctx` to fail, leaving the honest party's message inbox in a stale or partially-overwritten state that the protocol layer treats as a successfully received message.

---

### Finding Description

**Confirmed defect — call chain:**

`group_message(msg_tuples...)` (line 302) calls `unpack_multi_sets_tupled_msgs` which is declared `void`:

```cpp
// line 302
unpack_multi_sets_tupled_msgs(packed_msgs_tuple, std::tie(msg_tuples...));
return SUCCESS;   // always reached
``` [1](#0-0) 

Inside `unpack_multi_sets_msgs_helper`, every `unpack_msg_ctx` call is made inside a C++17 fold expression whose comma operator discards the `error_t` return value:

```cpp
// line 149
(unpack_msg_ctx(std::get<Is>(t1), std::get<Is>(t2)), ...);
``` [2](#0-1) 

`unpack_msg_ctx` itself returns the result of `unpack_msgs`, which propagates errors from `unpack_msgs_for_party`: [3](#0-2) 

**Two-layer deserialization creates the exploitable gap:**

`split_packed_msgs` (outer layer) deserializes the combined wire blob into per-group `vector<buf_t>` using `coinbase::deser`. This succeeds as long as the outer framing is valid. A Byzantine party can produce a blob where the outer framing is well-formed but one inner `buf_t` payload is malformed. [4](#0-3) 

`unpack_msgs_for_party` (inner layer) processes message fields sequentially. Once the converter enters error state it stops, but any fields already written before the error retain attacker-supplied values:

```cpp
([&](auto& arg) {
    if (!converter.is_error()) arg.unpack(converter, party_index);
}(msgs), ...);
if (rv = converter.get_rv()) return rv;   // error returned but caller discards it
``` [5](#0-4) 

**Confirmed reachable path in ECDSA-MPC signing:**

The multi-tuple `group_message` overload is called from `plain_broadcast_and_pairwise_message` in `ecdsa_mp.h`, which bundles pairwise OT messages with broadcast messages in a single round:

```cpp
if (rv = job.group_message(
        std::tie(receivers, senders, pairwise_msg),
        std::tie(all_parties, all_parties, to_all_msg)))
  return rv;
``` [6](#0-5) 

Because `group_message` returns `SUCCESS` regardless of inner deserialization outcome, the `rv` check on line 118 does not catch the failure.

---

### Impact Explanation

A Byzantine participant in an ECDSA-MPC signing round can send a crafted bundle that:

1. Passes `split_packed_msgs` (valid outer framing).
2. Causes `unpack_msg_ctx` to fail for the pairwise OT slot or the broadcast slot.
3. Leaves the honest party's `nonuniform_msg_t` or `uniform_msg_t` inbox in a stale or partially-written state.
4. Causes `group_message` to return `SUCCESS`, so the honest party proceeds with the corrupted inbox.

At minimum this produces protocol divergence — the honest party computes with wrong OT material, producing an invalid or attacker-influenced signature. If the inner message has multiple fields and the attacker controls the leading fields (valid) while making the trailing field malformed, attacker-chosen scalars or curve points can be injected into the OT inbox before the error fires. The specific impact on key material depends on how the OT multiplication values are consumed downstream in the signing protocol.

The question's specific claim that this reaches **DKG/refresh key output** is **not confirmed** by the code: the multi-tuple `group_message` overload is only called from `plain_broadcast_and_pairwise_message`, which is in the signing path, not in `dkg`/`refresh`. The single-set `group_message(to_set, from_set, msgs...)` overload used by DKG/refresh does check errors correctly. [7](#0-6) 

---

### Likelihood Explanation

The multi-tuple `group_message` path is reachable from the public ECDSA-MPC signing API whenever `plain_broadcast_and_pairwise_message` is invoked with non-empty `to_all_msgs`. Any Byzantine participant below threshold can trigger it by returning a crafted `buf_t` from the transport layer. No privileged access is required.

---

### Recommendation

1. Change `unpack_multi_sets_msgs_helper` and `unpack_multi_sets_tupled_msgs` to return `error_t` and propagate the first non-`SUCCESS` result from `unpack_msg_ctx`.
2. Change the call site in `group_message` to check and propagate that error before returning `SUCCESS`.
3. Add a static assertion or `[[nodiscard]]` annotation on `unpack_msg_ctx` to prevent future silent discards.

---

### Proof of Concept

```
Simulation (3 parties, party 1 is Byzantine):

1. Honest parties 0 and 2 enter plain_broadcast_and_pairwise_message.
2. group_message(msg_tuples...) is called.
3. Party 1 returns a crafted buf_t from receive_all:
   - Outer bundle: valid coinbase::ser of two buf_t blobs (passes split_packed_msgs).
   - Inner blob for pairwise_msg slot: first N-1 fields are attacker-chosen scalars,
     last field is truncated/malformed (triggers converter error in unpack_msgs_for_party).
4. split_packed_msgs succeeds (outer framing valid).
5. unpack_msg_ctx for pairwise_msg slot fails; error discarded by fold expression.
6. group_message returns SUCCESS.
7. Honest party 0's pairwise_msg inbox[1] contains attacker-chosen leading fields.
8. Signing protocol proceeds with corrupted OT material.
```
