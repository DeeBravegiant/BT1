### Title
Malicious Harvester Unconditionally Redirects Farmer Block Rewards via Unenforced `farmer_reward_address_override` in `RespondSignatures` - (File: chia/farmer/farmer_api.py)

### Summary

A harvester connected to a farmer via the private SSL channel can set `farmer_reward_address_override` in its `RespondSignatures` message to any arbitrary puzzle hash. The farmer unconditionally substitutes this address for its own configured `farmer_target` when constructing `DeclareProofOfSpace`, causing the 1/8 XCH farmer block reward to be paid to the attacker's address. The CHIP-22 fee-quality check that is supposed to gate this override is (a) only advisory — it logs a warning but never blocks the flow — and (b) not called at all for the `RespondSignatures` path, only for the earlier `NewProofOfSpace` path.

### Finding Description

**Root cause — two separate `farmer_reward_address_override` fields, one completely unchecked:**

`RespondSignatures` carries its own `farmer_reward_address_override` field:

```
chia/protocols/harvester_protocol.py  line 139
farmer_reward_address_override: bytes32 | None
```

In `FarmerAPI._process_respond_signatures()`, the farmer blindly accepts whatever the harvester puts there:

```python
# chia/farmer/farmer_api.py  lines 916-919
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override   # no check
    include_source_signature_data = True
```

That address is then placed directly into `DeclareProofOfSpace.farmer_reward_address` (line 929), which the full node uses as the puzzle hash for the farmer's coinbase reward coin.

**The fee-quality check is advisory and on a different code path:**

CHIP-22 intended `notify_farmer_reward_taken_by_harvester_as_fee()` to validate the override. That function is called only when `NewProofOfSpace.farmer_reward_address_override is not None` (farmer_api.py line 128-129). Even then, it only emits log warnings on failure — it never returns early or rejects the proof:

```python
# chia/farmer/farmer.py  lines 920-928
if fee_quality <= fee_threshold:
    self.log.info(...)          # passes: just a log
else:
    self.log.warning(...)       # fails: still just a log, flow continues
    self.log.warning("Harvester illegitimately took a fee reward ...")
```

**Exploit path bypassing even the advisory check:**

A harvester can send `NewProofOfSpace` with `farmer_reward_address_override = None` (no fee-quality check triggered at all), wait for the farmer to issue `RequestSignatures`, then reply with `RespondSignatures` containing `farmer_reward_address_override = <attacker_address>`. The farmer reaches `_process_respond_signatures`, finds the override non-None, and substitutes the attacker's address with zero validation.

### Impact Explanation

Every block the farmer wins while the malicious harvester is connected has its 1/8 XCH farmer coinbase reward permanently redirected to the attacker's address. The farmer signs the foliage block data (which commits to `farmer_reward_puzzle_hash`) using their own private key, so the block is valid on-chain and the misdirected coin is unrecoverable. This is direct, irreversible theft of XCH from the farmer. Impact: **High — unauthorized payout redirection of XCH block rewards**.

### Likelihood Explanation

Any harvester that has been granted a private-CA SSL certificate by the farmer operator can exploit this. In the CHIP-22 third-party harvester model (DrPlotter and similar), farmers routinely grant SSL access to external harvester operators. The harvester is a legitimate but untrusted participant; the fee-quality check was the intended authorization boundary, but it is not enforced. No additional privileges, leaked keys, or cryptographic breaks are required.

### Recommendation

1. **Enforce the fee-quality check, do not merely log.** In `new_proof_of_space`, if `notify_farmer_reward_taken_by_harvester_as_fee` determines the threshold is invalid, return `None` and drop the proof rather than continuing.
2. **Apply the same fee-quality check to `RespondSignatures.farmer_reward_address_override`.** In `_process_respond_signatures`, before accepting the override, verify that the proof's fee quality satisfies the declared threshold. If not, fall back to `self.farmer.farmer_target`.
3. **Consider whether `RespondSignatures` should carry `farmer_reward_address_override` at all**, or whether the override should only be accepted from the earlier `NewProofOfSpace` message where the proof bytes are available for the quality calculation.

### Proof of Concept

1. Operator runs a farmer with plots and grants a third-party harvester SSL access (normal CHIP-22 setup).
2. Malicious harvester sends a valid `NewProofOfSpace` with `farmer_reward_address_override = None` and a winning proof.
3. Farmer validates the proof, stores it, and sends `RequestSignatures` back to the harvester.
4. Harvester replies with `RespondSignatures` containing valid signatures **and** `farmer_reward_address_override = <attacker_puzzle_hash>`.
5. `_process_respond_signatures` executes lines 917-918: `farmer_reward_address = response.farmer_reward_address_override`.
6. `DeclareProofOfSpace` is sent to the full node with `farmer_reward_address = <attacker_puzzle_hash>`.
7. Full node creates a block; the farmer coinbase coin (1/8 XCH) is created at the attacker's address. The farmer receives nothing.
