### Title
Stale MPC Response Replay Enables Forged Foreign-Chain Verification — (File: `crates/contract/src/lib.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

The `ForeignTxSignPayloadV1` that MPC nodes sign contains no timestamp, block height, or nonce. A single Byzantine participant (below the signing threshold) who has observed any prior on-chain `respond_verify_foreign_tx` call for a given `(domain_id, ForeignChainRpcRequest, payload_version)` tuple can replay that old, valid MPC signature against a new pending request for the same tuple. The contract's `respond_verify_foreign_tx` only checks that the signature is valid over the caller-supplied `response.payload_hash`; it never verifies that `payload_hash` encodes the *current* foreign-chain state. A bridge contract (e.g., Omnibridge) that trusts the MPC attestation will act on stale extracted values — potentially releasing funds for a transaction that has since been reorganized out of the canonical chain.

---

### Finding Description

**Analog to the sandwich attack:** In VectorBonding, the attacker front-runs a victim's deposit with a price-manipulating swap, then back-runs to profit, because the contract accepted any amount of tokens with no slippage guard. The structural analog here is that the MPC contract accepts any valid MPC signature over any `payload_hash` for a matching pending request, with no guard that the hash encodes *current* foreign-chain state. A participant can "front-run" the honest MPC nodes' response with a stale one.

**Root cause 1 — no temporal binding in the signed payload.**

`build_signature_request` in `crates/node/src/providers/verify_foreign_tx/sign.rs` constructs the ECDSA payload as:

```rust
let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
    foreign_tx_payload.compute_msg_hash()?.into();
```

where `foreign_tx_payload` is a `ForeignTxSignPayloadV1`:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [1](#0-0) 

The struct contains only the chain request and the extracted values. There is no block height, no signing timestamp, no nonce, and no receipt-ID binding. The `entropy` and `timestamp_nanosec` fields present in the node-side `SignatureRequest` are used for the MPC protocol internals but are **not** included in the signed payload.

**Root cause 2 — `respond_verify_foreign_tx` verifies only the signature, not the payload's correspondence to the request.**

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [2](#0-1) 

The contract verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key. It does **not** verify that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayloadV1 { request: request.request, values: <current values> }))`. Any previously obtained valid MPC signature over any `(request, old_values)` pair will pass this check.

**Root cause 3 — caller-agnostic, temporally unbounded request key.**

`args_into_verify_foreign_tx_request` strips the caller identity and produces a key of `(domain_id, ForeignChainRpcRequest, payload_version)`:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [3](#0-2) 

The same key is reused across all submissions of the same foreign-chain query, regardless of when they are submitted. A response obtained for submission #1 is structurally valid for submission #2.

---

### Impact Explanation

**Impact: High — forged foreign-chain verification causing invalid bridge execution / double-spend.**

A bridge contract (e.g., Omnibridge inbound flow) calls `verify_foreign_transaction` to learn whether a foreign-chain deposit transaction is finalized and to obtain its block hash. It trusts the MPC-signed `payload_hash` as an attestation of current foreign-chain state. If a participant replays a stale response:

- The bridge receives `block_hash_old` (from a reorganized block) instead of `block_hash_current`.
- The bridge may release NEAR-side funds for a deposit transaction that no longer exists on the canonical foreign chain.
- This constitutes a double-spend: the depositor's ETH was reorganized away, but the NEAR-side release still occurs.

This matches the allowed impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

**Likelihood: Medium.**

Prerequisites:
1. **Attacker is a participant** — `respond_verify_foreign_tx` is gated by `assert_caller_is_attested_participant_and_protocol_active`. A single participant below the signing threshold suffices; no collusion is required.
2. **Attacker holds a prior valid response** — All `respond_verify_foreign_tx` calls are public on-chain transactions. Any participant can observe and save a prior response for any `(domain_id, ForeignChainRpcRequest, payload_version)` tuple.
3. **Same request is submitted again** — Bridge contracts retry on timeout (the contract itself fires `fail_on_timeout` on yield expiry, prompting the bridge to resubmit). A reorganization also motivates resubmission.
4. **Extracted values differ between submissions** — Required for the replay to cause harm. This occurs when `EvmFinality::Latest` or `EvmFinality::Safe` is used, or for Bitcoin with low `confirmations`.

For production bridge deployments using non-finalized finality settings, all four conditions are realistic.

---

### Recommendation

1. **Bind the signed payload to a temporal anchor.** Include the NEAR block height (or the receipt ID of the `verify_foreign_transaction` call) in `ForeignTxSignPayloadV1`. This makes every signature unique to a specific submission and prevents replay across submissions.

2. **Verify payload correspondence in `respond_verify_foreign_tx`.** The contract should recompute `expected_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request: request.request, values: response_values }))` and assert `expected_hash == response.payload_hash`. This requires the responder to also supply the raw `values`, but closes the gap entirely.

3. **Enforce minimum finality for bridge use cases.** Document and enforce that bridge integrations must use `EvmFinality::Finalized` (or Bitcoin `confirmations >= 6`) to reduce the reorganization window that makes stale responses harmful.

---

### Proof of Concept

```
Time T1:
  Bridge calls verify_foreign_transaction(Bitcoin tx X, confirmations=1)
  → MPC nodes observe tx X in block B1 (hash H1, 1 confirmation)
  → MPC signs payload_hash_1 = SHA-256(borsh({request: X, values: [H1]}))
  → respond_verify_foreign_tx(request=X, {payload_hash_1, sig_1}) submitted on-chain
  → Bridge receives H1, records "tx X confirmed in block H1"

  [Blockchain reorganization: tx X moves to block B2, hash H2]

Time T2:
  Bridge retries: calls verify_foreign_transaction(Bitcoin tx X, confirmations=1)
  → New yield queued under the same key (domain_id, X, V1)
  → Honest MPC nodes begin querying RPC for current state (would return H2)

  Attacker (participant P, below signing threshold):
  → Observes the new pending request on-chain
  → Calls respond_verify_foreign_tx(request=X, {payload_hash_1, sig_1})
     (replaying the T1 response, which is still a valid MPC signature)

  Contract checks:
  → verify_ecdsa_signature(sig_1, payload_hash_1, root_pk) → OK  ✓
  → resolve_yields_for(X, serialize({payload_hash_1, sig_1}))

  Bridge receives payload_hash_1 (encoding H1, the reorganized block hash)
  → Bridge concludes "tx X is in block H1" and releases NEAR-side funds
  → tx X no longer exists on the canonical Bitcoin chain → double-spend
``` [4](#0-3) [1](#0-0) [3](#0-2)

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
}

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
