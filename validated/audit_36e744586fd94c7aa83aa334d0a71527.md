### Title
Block-Level Reference Script Size Limit Bypassed via Dijkstra Subtransaction Reference Scripts — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

### Summary

In the Dijkstra era, the block-level reference script size check (`validateBodyRefScriptsSizeTooBig`) only counts reference scripts from top-level transaction inputs. It does not count reference scripts referenced by subtransactions embedded within a top-level transaction. The per-transaction check correctly uses `batchNonDistinctRefScriptsSize` (which sums top-level + subtransaction reference scripts), but the block-level check reuses the Conway-era `totalRefScriptSizeInBlock` which calls `txNonDistinctRefScriptsSize` — a function that is blind to subtransactions. An unprivileged transaction author can craft a block of Dijkstra top-level transactions whose subtransactions collectively reference far more script bytes than `maxRefScriptSizePerBlock` allows, while the block-level guard reports zero or near-zero bytes.

### Finding Description

The Dijkstra BBODY transition rule (`dijkstraBbodyTransition`) delegates the block-level reference script size check to the Conway helper:

```haskell
Conway.validateBodyRefScriptsSizeTooBig @era pp blockBody (ls ^. utxoL)
``` [1](#0-0) 

`validateBodyRefScriptsSizeTooBig` calls `totalRefScriptSizeInBlock`, which iterates over the top-level transactions in the block and calls `txNonDistinctRefScriptsSize` on each:

```haskell
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      ...
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
``` [2](#0-1) 

`txNonDistinctRefScriptsSize` only inspects the top-level transaction's `inputsTxBodyL` and `referenceInputsTxBodyL`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [3](#0-2) 

It never traverses `subTransactionsTxBodyL`. By contrast, the per-transaction check in the Dijkstra LEDGER rule uses `batchNonDistinctRefScriptsSize`, which explicitly adds subtransaction reference scripts:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
``` [4](#0-3) 

```haskell
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $ ...
``` [5](#0-4) 

The result is a split enforcement model:

| Check | Function used | Counts subtx ref scripts? |
|---|---|---|
| Per-tx (LEDGER) | `batchNonDistinctRefScriptsSize` | Yes |
| Per-block (BBODY) | `txNonDistinctRefScriptsSize` | **No** |

### Impact Explanation

The `maxRefScriptSizePerBlock` parameter (default 1 MiB) was introduced specifically to cap the total script-deserialization CPU/memory cost per block, following the June 2024 mainnet DDoS attack documented in ADR-009. [6](#0-5) 

Because the block-level guard ignores subtransaction reference scripts, an attacker can submit a block where each top-level transaction carries zero top-level reference inputs but embeds subtransactions that collectively reference up to `maxRefScriptSizePerTx` (204,800 bytes) of scripts. With multiple such top-level transactions in a block, the actual deserialization work can be a multiple of `maxRefScriptSizePerBlock`, while the guard reports zero bytes and passes. This matches the **Medium** allowed impact: attacker-controlled transactions exceed intended validation limits.

### Likelihood Explanation

The attack requires no privileged access. Any transaction author can:
1. Pre-fund UTxO outputs containing large Plutus reference scripts (a normal, fee-paying operation).
2. Craft Dijkstra top-level transactions with subtransactions that reference those outputs.
3. Keep each top-level transaction's own reference inputs empty so the block-level check sees 0 bytes per transaction.
4. Ensure the per-tx total (top + subs) stays ≤ 204,800 bytes to pass `validateAllRefScriptSize`.
5. Pack many such transactions into a single block.

The Dijkstra era is protocol version 12, currently the latest era in this repository. [7](#0-6) 

### Recommendation

Replace the call to `Conway.validateBodyRefScriptsSizeTooBig` in `dijkstraBbodyTransition` with a Dijkstra-aware variant that uses `batchNonDistinctRefScriptsSize` (or an equivalent block-level accumulator) so that subtransaction reference scripts are included in the per-block total. Specifically, `totalRefScriptSizeInBlock` should be extended or overridden for the Dijkstra era to iterate over subtransactions of each top-level transaction when computing the block total.

### Proof of Concept

1. Deploy N UTxO outputs each containing a large Plutus script (e.g., 20,000 bytes each, well within individual script limits).
2. For each of K top-level Dijkstra transactions (K chosen so K × 20,000 >> 1,048,576):
   - Set `referenceInputsTxBodyL = []` and `inputsTxBodyL` to a single coin-only input (no reference scripts at the top level).
   - Embed subtransactions whose combined `referenceInputsTxBodyL` references the large scripts, keeping the per-tx total ≤ 204,800 bytes.
3. Submit all K transactions in a single block.
4. `validateBodyRefScriptsSizeTooBig` computes `totalRefScriptSizeInBlock` = 0 (no top-level reference scripts) and passes.
5. `validateAllRefScriptSize` passes for each individual transaction (each ≤ 204,800 bytes).
6. The block is accepted, but the node must deserialize K × (subtransaction reference script bytes) >> 1,048,576 bytes of scripts, exceeding the intended block-level cap. [8](#0-7) [9](#0-8) [10](#0-9)

```markdown
It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
