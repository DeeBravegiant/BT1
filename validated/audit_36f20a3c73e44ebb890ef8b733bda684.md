Looking at the code carefully to trace the full validation path for p2p state diff sync.

### Title
Missing `state_diff_commitment` Hash Verification in P2P State Diff Sync Allows Unauthenticated Peer to Persist Corrupted State Diff — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`StateDiffStreamBuilder::parse_data_for_block` assembles a `ThinStateDiff` from p2p `StateDiffChunk` messages and writes it to storage, but **never calls `calculate_state_diff_hash`** and **never reads or compares against the `state_diff_commitment` field** stored in the block header. The only validations performed are a length count equality check and a duplicate-deprecated-class check. An unauthenticated p2p peer can send chunks whose total `len()` matches `header.state_diff_length` but whose storage values, nonces, or `class_hash_to_compiled_class_hash` entries are entirely fabricated, causing the node to permanently store a wrong `ThinStateDiff` at the correct block number.

---

### Finding Description

In `crates/apollo_p2p_sync/src/client/state_diff.rs`, `parse_data_for_block` reads only `state_diff_length` from the stored header:

```rust
let target_state_diff_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_length   // ← only field read
    .ok_or(...)?;
``` [1](#0-0) 

It then accumulates chunks until the count matches, checks for duplicate deprecated classes, and returns `Ok(Some((result, block_number)))` — with no hash computation:

```rust
if current_state_diff_len != target_state_diff_len { ... }   // line 99
validate_deprecated_declared_classes_non_conflicting(&result)?; // line 106
Ok(Some((result, block_number)))
``` [2](#0-1) 

The `state_diff_commitment` field (`Option<StateDiffCommitment>`) is present in `BlockHeader` and is stored when the header is written: [3](#0-2) 

`calculate_state_diff_hash` exists and is used in the batcher/committer path: [4](#0-3) 

A grep for `calculate_state_diff_hash` across all of `crates/apollo_p2p_sync/` returns **zero matches** — it is never called in the p2p sync client. A grep for `state_diff_commitment` in the same tree returns matches only in `test.rs` and `header.rs`, never in `state_diff.rs`.

The assembled (potentially corrupted) diff is then written unconditionally:

```rust
storage_writer.begin_rw_txn()?.append_state_diff(self.1, self.0)?.commit()?;
``` [5](#0-4) 

The header sync path (`HeaderStreamBuilder::parse_data_for_block`) also performs no cryptographic verification — it only checks block number ordering and that exactly one signature is present, with no ECDSA verification of the block hash or the `state_diff_commitment` value: [6](#0-5) 

---

### Impact Explanation

A malicious p2p peer can:

1. Send a `SignedBlockHeader` for block N with `state_diff_length = L` (matching the real block) and any `state_diff_commitment` value (since it is not cryptographically verified in the header path either).
2. Send `StateDiffChunk` messages whose combined `len()` equals L but whose `storage_diffs`, `nonces`, or `class_hash_to_compiled_class_hash` entries are fabricated.
3. The node accepts and persists the corrupted `ThinStateDiff` at block N.

Downstream effects:
- `starknet_getStorageAt` returns attacker-controlled storage values.
- `starknet_getStateUpdate` returns a wrong state diff.
- Contract nonces and class hashes are wrong, corrupting subsequent execution context for any node relying on this synced state.

This matches the **Critical** impact category: *Wrong state, class hash, or storage value persisted from a sync path.*

---

### Likelihood Explanation

Any node running in p2p sync mode (not central/feeder sync) is exposed. The attacker only needs to be a peer the node connects to — no authentication, no stake, no validator key required. The attack is deterministic and requires no brute force: craft chunks summing to the correct length with any desired values.

---

### Recommendation

After assembling the full `ThinStateDiff` and before returning `Ok(Some(...))`, read `state_diff_commitment` from the stored header and verify:

```rust
let header = storage_reader.begin_ro_txn()?.get_block_header(block_number)?.expect("...");
if let Some(expected_commitment) = header.state_diff_commitment {
    let actual_commitment = calculate_state_diff_hash(&result);
    if actual_commitment != expected_commitment {
        return Err(ParseDataError::BadPeer(BadPeerError::StateDiffHashMismatch { ... }));
    }
}
```

This mirrors the pattern already used in `apollo_committer`: [7](#0-6) 

---

### Proof of Concept

1. Start a node in p2p sync mode.
2. Act as a peer; respond to the header query for block 0 with a `SignedBlockHeader` where `state_diff_length = 1` and `state_diff_commitment` = Poseidon hash of a correct single-entry diff (e.g., nonce update for address `0x1` → `1`).
3. Respond to the state diff query with a single `StateDiffChunk::ContractDiff` for address `0x1` with `nonce = 999` (wrong value, but `len() == 1`).
4. Assert: `storage_reader.get_state_diff(BlockNumber(0))` returns a diff with nonce `999`, not `1`.
5. Assert: `calculate_state_diff_hash(&stored_diff) != header.state_diff_commitment` — the stored diff does not match the commitment in the header.

    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);

    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))
}
