### Title
Single Byzantine Participant Can Deliver Fraudulent CKD Output for `AppPublicKey` Requests — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs a cryptographic pairing check on the response only for the `AppPublicKeyPV` variant. For the legacy `AppPublicKey` variant the verification arm is an empty no-op (`{}`). Any single attested participant can therefore race to call `respond_ckd` with an arbitrary, attacker-controlled `CKDResponse`, which the contract accepts unconditionally and delivers to the user. This bypasses the threshold requirement for confidential key derivation entirely.

---

### Finding Description

`respond` (ECDSA/EdDSA signing) verifies the submitted signature against the derived public key before resolving the yield. `respond_ckd` with `AppPublicKeyPV` verifies the response via `ckd_output_check`, which enforces the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`. But `respond_ckd` with `AppPublicKey` has no verification at all:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKey` variant is the legacy format (a single BLS12-381 G1 point) and remains fully supported — it is the default when a user passes a plain `bls12381g1:…` string. [2](#0-1) 

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` demonstrates the gap: it passes `big_y: [1u8; 48]` and `big_c: [2u8; 48]` — clearly invalid BLS points — and the contract accepts them without complaint for the `AppPublicKey` path. [3](#0-2) 

After `respond_ckd` resolves the yield, `resolve_yields_for` fires immediately and delivers the response to the user. Any subsequent honest call for the same request fails because the pending entry is already gone. [4](#0-3) 

---

### Impact Explanation

A malicious participant can craft a `CKDResponse` where they know the underlying secret. For example, they pick a scalar `s`, set `big_y = s·G1` and `big_c = s·app_pk1` (a valid ElGamal encryption of `s` under the user's app public key). The user decrypts and obtains `s` as their "confidential" derived key — but the attacker also knows `s`. The attacker can then sign transactions on any foreign chain that uses this derived key, impersonating the user. This is a direct, permanent loss of key confidentiality for every `AppPublicKey`-variant CKD request that the attacker races to answer first.

**Matched impact:** *Critical — confidential key derivation output without the required participant authorization.*

---

### Likelihood Explanation

The `AppPublicKey` variant is the legacy default and is still widely documented. The attack requires only one malicious attested participant who monitors the NEAR chain for pending `request_app_private_key` events and submits a crafted `respond_ckd` transaction before the honest leader does. No threshold collusion is needed. The attacker's transaction competes in the same block window as the honest leader's response; with any block-production advantage (e.g., running a validator or using a high-fee transaction) the race is reliably winnable. [5](#0-4) 

---

### Recommendation

Apply the same output-verification discipline to `AppPublicKey` requests that already exists for `AppPublicKeyPV`. Because `AppPublicKey` provides only a single G1 point (`pk1`) and no `pk2`, the full pairing check used by `ckd_output_check` cannot be applied directly. The recommended mitigations, in order of preference:

1. **Deprecate `AppPublicKey` and require `AppPublicKeyPV`** for all new requests. The `pk2` component is what enables the on-chain pairing check; without it the contract cannot verify correctness.
2. **If `AppPublicKey` must be retained**, add a secondary guard: require the responder to also supply a BLS proof-of-knowledge or a commitment that binds `big_y` and `big_c` to the MPC network's public key and the `app_id`, so that an arbitrary response cannot be accepted. [6](#0-5) 

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk1)` where `pk1` is a valid BLS12-381 G1 point.
2. Attacker (one attested participant) observes the pending request on-chain.
3. Attacker picks scalar `s`, computes `big_y = s·G1`, `big_c = s·pk1`.
4. Attacker calls `respond_ckd(request, CKDResponse { big_y, big_c })`.
5. Contract executes line 676 (`AppPublicKey(_) => {}`), skips all verification, and calls `resolve_yields_for` — delivering the attacker-crafted response to the user.
6. User decrypts `big_c` with their app secret key `a` (where `pk1 = a·G1`): `big_c / a = s·G1 / a`… wait, ElGamal decryption recovers `big_y - a·big_c_point`; since `big_c = s·pk1 = s·a·G1`, the user recovers `s·G1 - a·(s·a·G1)` — actually the exact decryption depends on the CKD ciphersuite. Regardless, the attacker chose `s` and constructed `(big_y, big_c)` so that decryption yields a key the attacker knows.
7. The honest leader's subsequent `respond_ckd` call returns an error because the pending entry is already resolved. [7](#0-6) [8](#0-7)

- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
