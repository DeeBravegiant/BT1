### Title
Excess Attached Deposit Permanently Lost in `submit_participant_info` for Existing-Participant Re-submissions — (File: `crates/contract/src/lib.rs`)

---

### Summary
`submit_participant_info` is marked `#[payable]` and contains a deposit-check-and-refund block, but that block is guarded by a condition that evaluates to `false` for existing participants re-submitting their attestation. Any NEAR tokens attached to such a call are silently absorbed by the contract with no refund path executed.

---

### Finding Description
The function `submit_participant_info` computes whether the caller must pay for storage:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
``` [1](#0-0) 

The entire deposit validation and refund logic is nested inside:

```rust
if attestation_storage_must_be_paid_by_caller {
    // ... check attached >= cost, refund excess ...
}
``` [2](#0-1) 

When an existing participant (already in the participant set) re-submits their attestation — a routine operation during TEE upgrades or key rotations — `is_new_attestation` is `false` and `caller_is_not_participant` is `false`, so `attestation_storage_must_be_paid_by_caller` is `false`. The `if` block is never entered. Because the function is `#[payable]`, NEAR does not automatically reject or refund a non-zero deposit; the attached tokens are simply credited to the contract balance and permanently lost to the caller. [3](#0-2) 

By contrast, the three user-facing request methods (`sign`, `request_app_private_key`, `verify_foreign_transaction`) all call `require_deposit`, which unconditionally refunds any excess:

```rust
Some(diff) => {
    if diff > NearToken::from_yoctonear(0) {
        log!("refund excess deposit {diff} to {predecessor}");
        Promise::new(predecessor.clone()).transfer(diff).detach();
    }
}
``` [4](#0-3) 

`submit_participant_info` lacks an equivalent unconditional refund for the re-submission path.

---

### Impact Explanation
**Medium.** Any NEAR tokens attached to a re-submission call by an existing participant are permanently transferred to the contract with no recovery mechanism. This breaks the production accounting invariant that excess deposits must be returned to callers. The contract balance grows by the mistakenly attached amount, and the node operator loses those funds with no on-chain recourse.

---

### Likelihood Explanation
**Low.** The scenario requires node software to mistakenly attach a non-zero deposit when calling `submit_participant_info` for a re-submission (e.g., a misconfigured automation script, a copy-paste error in deployment tooling, or a software bug in the node's NEAR transaction builder). This is the same class of user/software mistake described in the reference report. Re-submissions are a normal operational event (TEE upgrades, attestation refreshes), so the exposure window is recurring.

---

### Recommendation
Add an unconditional refund for any excess deposit outside the `attestation_storage_must_be_paid_by_caller` guard, or refactor to always check and refund:

```diff
+    // Always refund any deposit not consumed by storage costs.
+    let attached = env::attached_deposit();
     if attestation_storage_must_be_paid_by_caller {
         let storage_used = env::storage_usage().saturating_sub(initial_storage);
         let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
-        let attached = env::attached_deposit();
         if attached < cost {
             return Err(...);
         }
-        if let Some(diff) = attached.checked_sub(cost)
-            && diff > NearToken::from_yoctonear(0)
-        {
-            Promise::new(account_id).transfer(diff).detach();
-        }
-    }
+        let refund = attached.checked_sub(cost).unwrap_or(NearToken::from_yoctonear(0));
+        if refund > NearToken::from_yoctonear(0) {
+            Promise::new(account_id.clone()).transfer(refund).detach();
+        }
+    } else if attached > NearToken::from_yoctonear(0) {
+        Promise::new(account_id).transfer(attached).detach();
+    }
```

### Proof of Concept

1. Node operator `alice.near` is already a registered participant (`voter_account()` succeeds → `caller_is_not_participant = false`).
2. Alice's node software re-submits a refreshed TEE attestation via `submit_participant_info`, and due to a configuration bug attaches `1 NEAR` as deposit.
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` → `is_new_attestation = false`.
4. `attestation_storage_must_be_paid_by_caller = false || false = false`.
5. The `if attestation_storage_must_be_paid_by_caller` block is skipped entirely.
6. The call succeeds; `1 NEAR` is credited to the MPC contract balance. Alice has no on-chain mechanism to recover it. [5](#0-4)
