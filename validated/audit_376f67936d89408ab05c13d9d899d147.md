Audit Report

## Title
Reverting-receive contract permanently freezes `unlockedWithdrawalsCount[ETH]`, blocking `sweepRemainingAssets` — (`contracts/LRTWithdrawalManager.sol`)

## Summary
In `_processWithdrawalCompletion`, `unlockedWithdrawalsCount[asset]--` at line 717 executes before `_transferAsset` at line 734. When the recipient is a contract with a reverting `receive()`, the ETH low-level call fails and reverts the entire transaction, rolling back the counter decrement. Because the counter can never reach zero, every subsequent call to `sweepRemainingAssets(ETH)` permanently reverts with `PendingWithdrawalsExist()`, freezing any surplus ETH (rounding residuals, direct sends, Aave interest) in the contract forever.

## Finding Description
**State ordering flaw in `_processWithdrawalCompletion`:**

`unlockedWithdrawalsCount[asset]--` is decremented at line 717, before `_transferAsset` is called at line 734. [1](#0-0) 

`_transferAsset` for ETH uses a low-level call and reverts the whole transaction on failure: [2](#0-1) 

Because Solidity reverts all state changes on a revert, the decrement at line 717 is rolled back. The counter stays at its pre-call value.

**Counter is incremented during `_unlockWithdrawalRequests`:** [3](#0-2) 

**`sweepRemainingAssets` is permanently gated on the counter:** [4](#0-3) 

`hasUnlockedWithdrawals` simply checks `unlockedWithdrawalsCount[asset] > 0`: [5](#0-4) 

**No escape hatch exists.** A grep across all contracts for `cancelWithdrawal`, `forceCancel`, `resetCounter`, `initialize2`, and `initialize3` returns no results. `completeWithdrawalForUser` (the operator-callable variant) also routes through `_processWithdrawalCompletion` and fails identically. The `UnlockedWithdrawalsInitializer.initialize2` is a one-shot migration function already consumed on mainnet and cannot be re-invoked. [6](#0-5) 

## Impact Explanation
`sweepRemainingAssets(ETH)` is permanently blocked for ETH. Any ETH balance that accumulates in the contract beyond committed withdrawals — including rounding residuals from Aave principal withdrawals, Aave interest accrual, or direct ETH sends via `receive()` — cannot be recovered to the treasury. This constitutes **permanent freezing of unclaimed yield** (Medium impact per scope). The comment at line 394 acknowledges ETH can accumulate: "ETH should not accumulate requiring sweeping" — confirming the function is intended as a recovery path for exactly this surplus. [7](#0-6) 

## Likelihood Explanation
The attacker requires only: (1) a trivial contract with a reverting `receive()`, and (2) a small amount of rsETH (obtainable by depositing any supported asset). No privileged role is needed. The operator's `unlockQueue` call is a routine operational action that will occur regardless of attacker intent. The attack is cheap, deterministic, requires no front-running, and is executable by any unprivileged external user.

## Recommendation
**Option A (preferred):** Adopt a pull-payment pattern. Store the owed ETH in a per-user claimable balance mapping instead of pushing ETH in `_processWithdrawalCompletion`. Decrement `unlockedWithdrawalsCount` and delete the request atomically, then let the user pull their ETH separately via a dedicated `claimETH()` function.

**Option B:** Move `unlockedWithdrawalsCount[asset]--` to *after* the successful `_transferAsset` call, and wrap the transfer in a try/catch that marks the request as "permanently failed" (e.g., a `failedWithdrawals` mapping) rather than reverting the whole transaction. This allows the counter to be decremented even when the transfer fails.

**Option C:** Add an admin function that can forcibly cancel a stuck unlocked withdrawal request (redirecting ETH to a recovery address or burning the rsETH claim) and correctly decrements `unlockedWithdrawalsCount`.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

contract RevertingReceiver {
    receive() external payable { revert("no ETH"); }

    function attack(address withdrawalManager, address rsETH, uint256 rsETHAmount) external {
        IERC20(rsETH).approve(withdrawalManager, rsETHAmount);
        ILRTWithdrawalManager(withdrawalManager).initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
    }
}

// Step 1: Deploy RevertingReceiver, fund with rsETH, call attack()
// Step 2: Operator calls unlockQueue(ETH_TOKEN, ...) 
//         → unlockedWithdrawalsCount[ETH] = 1
// Step 3: Call completeWithdrawal(ETH_TOKEN, "") or completeWithdrawalForUser(ETH_TOKEN, receiver, "")
//         → _transferAsset reverts (EthTransferFailed)
//         → decrement at L717 is rolled back
//         → unlockedWithdrawalsCount[ETH] remains 1
// Step 4: Assert:
//         hasUnlockedWithdrawals(ETH_TOKEN) == true  (forever)
//         sweepRemainingAssets(ETH_TOKEN) reverts PendingWithdrawalsExist()  (forever)
```

A Foundry fork test can confirm this by: deploying `RevertingReceiver`, initiating a withdrawal, calling `unlockQueue`, attempting `completeWithdrawalForUser`, and asserting `hasUnlockedWithdrawals(ETH_TOKEN) == true` and that `sweepRemainingAssets` reverts after ETH accumulates in the contract.
