Audit Report

## Title
Incorrect 1:1 Price Set When `rsethSupply == 0` With Protocol ETH Remaining Enables Yield Theft - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle._updateRsETHPrice()` unconditionally sets `rsETHPrice = 1 ether` and `highestRsethPrice = 1 ether` whenever `rsethSupply == 0`, without checking whether ETH value already exists in the protocol. An attacker can exploit this by depositing at the artificially low 1:1 price after accumulated staking rewards have entered the deposit pool, then withdrawing at the correct higher price after the next price update, stealing the accumulated yield.

## Finding Description

In `contracts/LRTOracle.sol` at lines 218–222, `_updateRsETHPrice()` returns early with a hardcoded 1:1 price whenever supply is zero:

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

This branch never calls `_getTotalEthInProtocol()`, so it cannot detect whether ETH value already exists in the protocol. The deposit pool's `receiveFromRewardReceiver()`, `receiveFromNodeDelegator()`, and `receiveFromLRTConverter()` all accept ETH unconditionally with no supply check, meaning staking rewards can accumulate in the deposit pool even when `rsethSupply == 0`.

The deposit minting formula in `LRTDepositPool.getRsETHAmountToMint()` (line 520) reads the stored `rsETHPrice` directly:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

When `rsETHPrice == 1 ether` (set by the zero-supply branch), a depositor receives rsETH at 1:1 even though the protocol holds pre-existing ETH value `Y`. After the deposit, the next call to `updateRSETHPrice()` correctly computes `rsETHPrice = (X + Y - fee) / X > 1 ether`. The attacker then calls `initiateWithdrawal`, which uses the live `rsETHPrice` via `getExpectedAssetAmount()` (line 593 of `LRTWithdrawalManager.sol`):

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

This returns `X + Y - fee` ETH for `X` rsETH, stealing `Y - fee` ETH of accumulated yield.

The `highestRsethPrice` reset to `1 ether` also disables the downside-protection circuit breaker relative to the true historical high, and allows the upside-protection threshold to be measured from `1 ether` rather than the last real price, further weakening the price-deviation guards.

**Existing guards are insufficient:**
- The `pricePercentageLimit` check (lines 256–266) only triggers if `pricePercentageLimit > 0` and the price jump exceeds the limit. The attacker can size `X` large relative to `Y` to keep the price increase within the limit, or exploit the case where `pricePercentageLimit == 0`.
- The `getAvailableAssetAmount` check in `initiateWithdrawal` (line 170) passes because `totalAssets = X + Y` and `assetsCommitted = 0`.
- `unlockQueue` is operator-restricted, but operators are expected to process the queue routinely; the attacker simply waits.

## Impact Explanation

**High — Theft of unclaimed yield.** The attacker deposits `X` ETH at the artificially low 1:1 price while the protocol holds `Y` ETH of accumulated staking rewards, then withdraws `X + Y - fee` ETH after the price corrects, extracting `Y - fee` ETH of yield that belongs to the protocol or future depositors. This is a direct, concrete theft of unclaimed yield matching the allowed High impact class.

## Likelihood Explanation

**Low-to-Medium.** The precondition requires `rsethSupply == 0` while `totalETHInProtocol > 0`. This can occur when all users complete withdrawals but staking rewards continue to arrive in the deposit pool between the last withdrawal and the next deposit. `updateRSETHPrice()` is public and callable by anyone, so no privileged access is required to trigger the mispricing. The scenario is unusual in normal operation but plausible during protocol migrations, low-activity periods, or emergency wind-downs.

## Recommendation

Before returning early, check whether `totalETHInProtocol > 0`. If assets exist with zero supply, the protocol is in an anomalous state and should not silently set the price to `1 ether`:

```solidity
if (rsethSupply == 0) {
    uint256 totalETHInProtocol = _getTotalEthInProtocol();
    if (totalETHInProtocol > 0) {
        revert ProtocolHasAssetsButNoSupply();
    }
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

This prevents the first depositor from receiving rsETH at 1:1 when the protocol already holds ETH value, and prevents `highestRsethPrice` from being incorrectly reset.

## Proof of Concept

```solidity
// Preconditions:
// 1. All rsETH has been burned (rsethSupply == 0)
// 2. Y ETH of staking rewards has accumulated in the deposit pool
//    via receiveFromRewardReceiver() or similar

// Step 1: Anyone (including attacker) calls updateRSETHPrice()
//         rsethSupply == 0 → rsETHPrice = 1 ether (stored on-chain)
lrtOracle.updateRSETHPrice();
assertEq(lrtOracle.rsETHPrice(), 1 ether);

// Step 2: Attacker deposits X ETH
//         getRsETHAmountToMint = (X * 1e18) / 1e18 = X rsETH (1:1)
uint256 X = 100 ether; // size X >> Y to stay within pricePercentageLimit
lrtDepositPool.depositETH{value: X}(X, "");
assertEq(rsETH.balanceOf(attacker), X);

// Step 3: updateRSETHPrice() is called again (by attacker or anyone)
//         rsethSupply = X, totalETHInProtocol = X + Y
//         previousTVL = X * 1e18 = X
//         fee = (Y * protocolFeeInBPS) / 10_000
//         newRsETHPrice = (X + Y - fee) / X > 1 ether
lrtOracle.updateRSETHPrice();
// rsETHPrice > 1 ether

// Step 4: Attacker initiates withdrawal of X rsETH
//         expectedAssetAmount = X * rsETHPrice / 1e18 = X + Y - fee
lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, X, "");

// Step 5: Operator calls unlockQueue (normal operation)
// Step 6: After withdrawalDelayBlocks, attacker calls completeWithdrawal
//         Receives X + Y - fee ETH, stealing Y - fee ETH of accumulated yield
lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");
assertGt(attacker.balance, X); // attacker profited
```
