### Title
Caller Deposit Not Refunded on Intermediate Callback Failure in Wallet Contract — (`runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

---

### Summary

The `near-wallet-contract` (the eth-implicit account wallet deployed on NEAR) tracks an external caller's attached NEAR deposit in `CallerDeposit` so it can be refunded if the final cross-contract call fails. However, two intermediate callbacks — `nep_141_storage_balance_callback` and `address_check_callback` — contain multiple early-return error paths that return without refunding `caller_deposit`. The caller's attached NEAR is permanently absorbed into the wallet contract's balance.

---

### Finding Description

When an external caller (relayer) calls `rlp_execute` with an attached NEAR deposit, `inner_rlp_execute` captures the full deposit in a `CallerDeposit` struct: [1](#0-0) 

The design intent is that `rlp_execute_callback` refunds this deposit if the final promise fails: [2](#0-1) 

However, for ERC-20 transfer transactions, the flow goes through `nep_141_storage_balance_callback` first. This callback has **four** early-return paths that never reach `rlp_execute_callback` and never refund `caller_deposit`:

1. `storage_balance_of` call fails (line 204–209): [3](#0-2) 

2. `storage_balance_of` returns unparseable data (line 213–219): [4](#0-3) 

3. Action is not a `FunctionCall` (line 246–252): [5](#0-4) 

Similarly, for EOA base-token transfers to eth-implicit accounts, `address_check_callback` has early-return paths that drop `caller_deposit`:

4. Registrar lookup fails (line 142–147): [6](#0-5) 

5. Registrar returns unparseable data (line 151–157): [7](#0-6) 

6. Target resolves to a named account (line 168–172): [8](#0-7) 

In all six cases, `PromiseOrValue::Value(...)` is returned immediately. The `caller_deposit` is silently dropped. The NEAR that was attached to `rlp_execute` remains in the wallet contract's account balance, permanently transferred to the wallet owner.

---

### Impact Explanation

The corrupted protocol value is the **NEAR token balance** of the external caller (relayer). The caller's `attached_deposit` — which can be any non-zero amount — is irrecoverably transferred to the wallet contract's balance instead of being returned. The wallet owner's NEAR balance increases by the full `caller_deposit` amount without any legitimate basis.

---

### Likelihood Explanation

A malicious wallet owner can reliably trigger this:

1. Deploy a malicious NEP-141 token contract whose `storage_balance_of` method always panics.
2. Sign an Ethereum ERC-20 transfer transaction targeting this malicious contract.
3. Induce a relayer to submit the transaction via `rlp_execute` with an attached NEAR deposit (relayers routinely attach NEAR to cover storage costs).
4. `storage_balance_of` fails → `nep_141_storage_balance_callback` returns early at line 204–209 → `caller_deposit` is dropped → relayer's NEAR is absorbed.

The attack requires no special privileges: any eth-implicit account holder can deploy a contract and sign Ethereum transactions. The relayer's deposit amount is bounded only by what the relayer attaches, which for storage-deposit flows is typically `NEP_141_STORAGE_DEPOSIT_AMOUNT` (1.25 milliNEAR), but for base-token transfers can be arbitrarily large.

---

### Recommendation

Each early-return error path in `nep_141_storage_balance_callback` and `address_check_callback` must emit a refund transfer before returning, mirroring the logic already present in `rlp_execute_callback`:

```rust
// Before every early `return PromiseOrValue::Value(...)` in both callbacks:
if let Some(CallerDeposit { account_id, yocto_near }) = &caller_deposit {
    let refund = env::promise_batch_create(account_id);
    env::promise_batch_action_transfer(
        refund,
        NearToken::from_yoctonear(yocto_near.get()),
    );
}
```

Alternatively, extract a shared helper that always performs the refund before returning an error response, so future early-return paths cannot accidentally omit it.

---

### Proof of Concept

```
1. Attacker deploys malicious_token.near with storage_balance_of() { panic!() }
2. Attacker (wallet owner) signs Ethereum ERC-20 transfer tx:
     to = address_of(malicious_token.near)
     data = ERC20_TRANSFER_SELECTOR ++ abi_encode(victim_address, amount)
3. Relayer calls:
     rlp_execute(target="malicious_token.near", tx_bytes_b64=...) 
     with attached_deposit = 5 NEAR
4. inner_rlp_execute captures CallerDeposit { account_id: relayer, yocto_near: 5e24 }
5. Promise chain: storage_balance_of(malicious_token.near) → FAILS
6. nep_141_storage_balance_callback fires at line 204:
     returns PromiseOrValue::Value(error) — caller_deposit is dropped
7. Relayer's 5 NEAR is now in the wallet contract's balance.
8. Wallet owner profits; relayer has no recourse.
```
