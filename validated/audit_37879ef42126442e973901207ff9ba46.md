Audit Report

## Title
Pool `swap` silently delivers partial output in exact-output mode when liquidity is insufficient, without reverting — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

`MetricOmmPool.swap` with a negative `amountSpecified` (exact-output mode) silently caps the output at available pool liquidity and transfers fewer tokens than requested to the recipient without reverting. The `IncorrectDelta` guard only verifies the input leg was received; it does not verify the output equals the requested amount. Any direct pool integrator relying on exact-output semantics receives a partial fill with no on-chain signal.

## Finding Description

**Step 1 — Silent cap without revert.**

`_swapToken0ForToken1SpecifiedOutput` silently reduces `amountOutScaled` to whatever token1 the pool holds: [1](#0-0) 

The same pattern exists in `_swapToken1ForToken0SpecifiedOutput`: [2](#0-1) 

**Step 2 — Capped value propagates back.**

`_executeSwap` assigns `amount1DeltaScaled = -int256(amountOutScaled)` using the already-capped return value, not the original requested amount: [3](#0-2) 

**Step 3 — Pool transfers the capped amount and calls callback.**

`swap` transfers the capped `amount1Delta` to the recipient, then calls the callback with `amount0Delta` sized only for the capped output: [4](#0-3) 

**Step 4 — `IncorrectDelta` only checks the input leg.**

The guard verifies `balance0` increased by `amount0Delta` (the input). It does not check that `amount1Delta` equals the originally requested `-N`. A partial fill passes this check silently: [5](#0-4) 

**Step 5 — Mitigation exists only in the periphery router, not the core pool.**

`MetricOmmSimpleRouter.exactOutputSingle` adds a post-swap check, but this protection is absent from the core pool: [6](#0-5) 

## Impact Explanation

This is broken core pool functionality. The pool's `swap` function with `amountSpecified < 0` is the canonical exact-output interface; it does not enforce the exact-output guarantee. Any direct caller — aggregators, custom routers, on-chain protocols — that calls `MetricOmmPool.swap` directly with `amountSpecified = -N` where `N` exceeds available liquidity will silently receive fewer tokens than requested. Downstream obligations that require exactly `N` tokens (flash loan repayment, limit order fulfillment, multi-hop settlement) will fail at a later step, causing cascading reverts and potential loss of gas or locked funds in multi-step protocols. The broken invariant is in the core pool, a public contract designed for direct integration.

## Likelihood Explanation

Any unprivileged caller integrating directly with `MetricOmmPool.swap` in exact-output mode is affected whenever pool liquidity is insufficient to fill the requested output — a normal market condition (thin liquidity, large order). The pool is a public contract; direct integration by aggregators and custom routers is the expected use case. No special privileges or malicious setup are required.

## Recommendation

Add an exact-output enforcement check inside `MetricOmmPool.swap` after `_executeSwap` returns, for the exact-output case (`amountSpecified < 0`):

```solidity
if (amountSpecified < 0) {
    int256 requestedOut = int256(uint256(-amountSpecified));
    int256 actualOut = zeroForOne ? -amount1Delta : -amount0Delta;
    if (actualOut != requestedOut) revert InsufficientLiquidityForExactOutput();
}
```

Alternatively, `_swapToken0ForToken1SpecifiedOutput` and `_swapToken1ForToken0SpecifiedOutput` should revert instead of silently capping when `amountOutScaled > totalAvailableXScaled`.

## Proof of Concept

```solidity
// Foundry integration test
function test_exactOutput_partialFill_noRevert() public {
    // Pool holds 100 token1 (scaled)
    // Trader requests 200 token1 out (exact-output)
    uint128 requested = 200;
    uint256 recipientBefore = token1.balanceOf(recipient);

    // swap(zeroForOne=true, amountSpecified=-200, ...) — should revert or deliver 200
    (int128 amount0Delta, int128 amount1Delta) = pool.swap(
        recipient, true, -int128(requested), 0, callbackData, ""
    );

    uint256 received = token1.balanceOf(recipient) - recipientBefore;

    // These assertions PASS, demonstrating the broken invariant:
    assertLt(received, requested);           // received < requested, no revert
    assertGt(amount1Delta, -int128(requested)); // amount1Delta not equal to -requested
}
```
