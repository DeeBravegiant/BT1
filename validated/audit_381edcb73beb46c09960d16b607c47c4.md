Audit Report

## Title
Late Depositors Steal Yield from Existing rsETH Holders via Stale `rsETHPrice` - (`contracts/LRTOracle.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint()` divides by the stored `rsETHPrice` state variable, which is only updated on explicit calls to `updateRSETHPrice()`. When rewards accrue between updates, a depositor can mint rsETH at the stale (lower) price, then trigger a price update that folds their deposit into `previousTVL`, diluting the yield owed to pre-existing holders.

## Finding Description

**Root cause — stale price used at mint time:**

`getRsETHAmountToMint` at `LRTDepositPool.sol:520` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` reads the stored state variable declared at `LRTOracle.sol:28`. The deposit flow (`depositETH` → `_beforeDeposit` → `getRsETHAmountToMint`) never calls `updateRSETHPrice()` before minting, so any rewards that accrued since the last update are invisible to the mint calculation.

**Root cause — `previousTVL` absorbs the attacker's deposit:**

Inside `_updateRsETHPrice()` at `LRTOracle.sol:234`:

```solidity
uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);
```

`rsethSupply` is the **current** total supply (already including the attacker's newly minted rsETH), while `rsETHPrice` is the **old** stale price. This means the attacker's shares are silently counted as if they were present during the reward accrual period. The `rewardAmount` computed at line 245 (`totalETHInProtocol - previousTVL`) is therefore identical to what it would have been without the deposit, but the new price is divided over a larger supply, diluting existing holders.

**Exploit path:**
1. Protocol has 100 rsETH, 100 ETH, `rsETHPrice = 1.0`.
2. 10 ETH of staking/MEV rewards flow in via `FeeReceiver.sendFunds()` → `receiveFromRewardReceiver()`. `totalETHInProtocol = 110`, `rsETHPrice` still `1.0`.
3. Attacker calls `depositETH(10 ETH)`: mints `10 * 1e18 / 1e18 = 10 rsETH`. Supply = 110, TVL = 120.
4. Attacker calls `updateRSETHPrice()` (public, `LRTOracle.sol:87`): `previousTVL = 110 * 1.0 = 110`, `rewardAmount = 10`, `newRsETHPrice = 120/110 ≈ 1.0909`.
5. Attacker's 10 rsETH → 10.909 ETH (profit ≈ 0.909 ETH from rewards they did not earn).
6. Original 100 rsETH → 109.09 ETH instead of 110 ETH (loss ≈ 0.91 ETH).

**Existing guards are insufficient:**
- `pricePercentageLimit` (lines 252–266) only gates whether a non-manager can call `updateRSETHPrice()` when the price jump is large; it does not prevent the deposit at the stale price, and the attacker's position is already locked in before the price update.
- `minRSETHAmountExpected` is slippage protection for the depositor, not a guard against this attack.

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders lose a portion of staking/MEV rewards that accrued before the attacker's deposit. The loss is proportional to the reward amount and the attacker's deposit size relative to total supply. The attack is repeatable every time rewards accrue and the price is stale, which is a continuous condition between keeper updates.

## Likelihood Explanation

`updateRSETHPrice()` is `public` and callable by any EOA. An attacker needs only to: (a) monitor `_getTotalEthInProtocol()` vs. `rsethSupply * rsETHPrice` off-chain to detect a stale-price window, (b) call `depositETH()` or `depositAsset()`, and (c) call `updateRSETHPrice()`. No privileged access, no victim mistake, and no external dependency is required. The condition is present continuously between keeper updates and after every MEV/reward event.

## Recommendation

**Option 1 (preferred):** Call `_updateRsETHPrice()` at the start of `depositETH()` and `depositAsset()` before `_beforeDeposit()` computes `getRsETHAmountToMint()`. This ensures every depositor pays the current fair price inclusive of all accrued rewards.

**Option 2:** Snapshot `rsethSupply` at the time of the last price update and use that snapshot (not the current supply) when computing `previousTVL` in `_updateRsETHPrice()`. This prevents new deposits made at the stale price from being absorbed into the baseline.

## Proof of Concept

```
Foundry fork/unit test outline:

1. Deploy protocol; alice deposits 100 ETH → 100 rsETH; rsETHPrice = 1e18.
2. Simulate 10 ETH reward: deal(address(depositPool), address(depositPool).balance + 10 ether)
   OR call feeReceiver.sendFunds() with 10 ETH.
   Assert: _getTotalEthInProtocol() == 110e18, rsETHPrice still == 1e18.
3. Attacker calls depositPool.depositETH{value: 10 ether}(0, ""):
   Assert: rsETH.balanceOf(attacker) == 10e18 (minted at stale price 1.0).
   Assert: rsETH.totalSupply() == 110e18.
4. Attacker calls lrtOracle.updateRSETHPrice():
   Assert: lrtOracle.rsETHPrice() ≈ 120e18/110 ≈ 1.0909e18.
5. Compute attacker value: 10e18 * rsETHPrice / 1e18 ≈ 10.909e18 > 10e18 (profit).
6. Compute alice value: 100e18 * rsETHPrice / 1e18 ≈ 109.09e18 < 110e18 (loss).
7. Assert alice_loss + attacker_profit ≈ 0 (conservation), confirming yield transfer.
```
