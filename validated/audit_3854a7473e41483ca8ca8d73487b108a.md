### Title
Stale-pending fallback in `read_pending_data` always reports `l1_da_mode = CALLDATA`, silently lying to RPC callers on Blob-mode networks — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
When the in-memory pending block is stale (its `parent_block_hash` does not match the latest stored header), `read_pending_data` synthesizes a `DeprecatedPendingBlock` fallback. `DeprecatedPendingBlock` has no `l1_da_mode` field, and `PendingBlockOrDeprecated::l1_da_mode()` hard-codes `L1DataAvailabilityMode::Calldata` for every `Deprecated` variant. The latest header's real `l1_da_mode` is available in scope but is never copied. Any unprivileged caller of `starknet_getBlockWithTxHashes` (or `starknet_getBlockWithTxs`) with `Tag::Pending` during this window receives `"l1_da_mode": "CALLDATA"` even when the network operates in Blob mode.

### Finding Description

**Step 1 — stale-pending branch synthesizes `DeprecatedPendingBlock`** [1](#0-0) 

When `parent_block_hash` mismatches, the fallback is constructed at line 1574. `latest_header.block_header_without_hash.l1_da_mode` is in scope but is not included in the struct literal. The struct ends with `..Default::default()`, which fills every omitted field with its type default.

**Step 2 — `DeprecatedPendingBlock` has no `l1_da_mode` field** [2](#0-1) 

There is no `l1_da_mode` field. The struct cannot carry the correct value regardless of what the caller passes.

**Step 3 — `l1_da_mode()` on `Deprecated` always returns `Calldata`** [3](#0-2) 

The comment says "In older versions, all blocks were using calldata." This is correct for real deprecated blocks from the network, but the fallback synthesized here is not a real deprecated block — it is a stand-in for a modern Blob-mode block.

**Step 4 — `Calldata` is the `#[default]` for `L1DataAvailabilityMode`** [4](#0-3) 

**Step 5 — `get_block` reads `l1_da_mode` directly from the synthesized block** [5](#0-4) 

`l1_da_mode: block.l1_da_mode()` at line 1626 propagates the wrong value into `PendingBlockHeader`, which is serialized into the JSON RPC response.

### Impact Explanation
Any unprivileged RPC caller querying `starknet_getBlockWithTxHashes` or `starknet_getBlockWithTxs` with `Tag::Pending` during the stale window receives `"l1_da_mode": "CALLDATA"` on a Blob-mode network. Clients use this field to decide whether to price fees against `l1_gas_price` (Calldata) or `l1_data_gas_price` (Blob). A wrong value causes clients to construct transactions with incorrect resource bounds for the pending block context, and causes monitoring/tooling to misreport the network's DA mode. This fits: **High — RPC pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation
The stale window is a normal operational condition: it occurs on every block finalization, lasting until the pending-data poller refreshes. On a busy network this window is short but guaranteed to occur repeatedly. No special privileges or attack are needed — a passive RPC poll during the window is sufficient.

### Recommendation
In the stale-pending fallback inside `read_pending_data`, switch from `PendingBlockOrDeprecated::Deprecated` to `PendingBlockOrDeprecated::Current` (i.e., `PendingBlock`) so that `l1_da_mode` can be set explicitly from `latest_header.block_header_without_hash.l1_da_mode`. Alternatively, if `DeprecatedPendingBlock` must be kept, add an `l1_da_mode` field to it and populate it from the latest header in the fallback path.

### Proof of Concept
1. Store a block header with `l1_da_mode = Blob` in the storage.
2. Set the in-memory `pending_data.block.parent_block_hash` to a random hash (simulating a stale pending block).
3. Call `starknet_getBlockWithTxHashes` with `Tag::Pending`.
4. Assert the returned `l1_da_mode` equals `"BLOB"` — it will instead be `"CALLDATA"`, demonstrating the bug.
