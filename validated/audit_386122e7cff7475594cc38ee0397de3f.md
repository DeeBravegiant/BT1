Audit Report

## Title
Exact-Output Swap Silently Delivers Less Token0 Than Requested When Pool Liquidity Is Insufficient — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`_swapToken1ForToken0SpecifiedOutput` silently clamps the caller-supplied exact-output amount to `binTotals.scaledToken0` instead of reverting when pool liquidity is insufficient. The capped value propagates back through `_executeSwap` and the `swap` function transfers only the reduced amount to `recipient`, succeeding without error. A caller who specifies an exact output of token0 receives less than requested with no on-chain indication of the shortfall.

## Finding Description
In `_swapToken1ForToken0SpecifiedOutput`, the very first operation inside `unchecked` reads `binTotals.scaledToken0` and overwrites `amountOutScaled` if it exceeds the total available balance: [1](#0-0) 

This capped value is returned and assigned directly to `amount0DeltaScaled` in `_executeSwap`: [2](#0-1) 

The outer `swap` function then transfers only the capped amount to `recipient`: [3](#0-2) 

There is no post-swap assertion that `|amount0Delta| == |amountSpecified|` for exact-output swaps. The function returns `(amount0Delta, amount1Delta)` where `|amount0Delta| < |amountSpecified|` and the transaction succeeds silently. [4](#0-3) 

## Impact Explanation
The exact-output guarantee is broken: a caller who issues `swap(recipient, false, -N, ...)` expecting to receive exactly `N` (scaled) token0 silently receives less. Any downstream logic relying on the exact amount — multi-hop routers, limit-order settlement, liquidation bots — operates on an incorrect balance. The caller's token1 input is consumed for a swap that delivers less token0 than contracted. Because the pool does not revert, callers who interact with the pool directly or whose router checks `amountOut >= minOut` against the returned delta (rather than the requested amount) have no protection. This constitutes a direct loss of swap output relative to the user's specified intent, reachable by any unprivileged caller under normal thin-liquidity market conditions. **Severity: Medium.**

## Likelihood Explanation
Any unprivileged caller can trigger this by calling `swap` with `zeroForOne = false` and `amountSpecified < 0` where `|amountSpecified| * TOKEN_0_SCALE_MULTIPLIER > binTotals.scaledToken0`. This is a routine market condition (thin liquidity, large order, or a partially drained pool). No privileged access, malicious setup, or non-standard token behavior is required. The condition is repeatable and requires no coordination.

## Recommendation
Replace the silent cap with an explicit revert:

```solidity
uint256 totalAvailableToken0Scaled = binTotals.scaledToken0;
require(amountOutScaled <= totalAvailableToken0Scaled, InsufficientLiquidity());
```

Also audit `_swapToken0ForToken1SpecifiedOutput` for the symmetric pattern (silent cap on token1 exact-output). [5](#0-4) 

## Proof of Concept
1. Deploy a pool with `binTotals.scaledToken0 = 1000`.
2. Call `pool.swap(recipient, false, -5000, type(uint128).max, callbackData, "")` — requesting 5000 token0 out (exact output).
3. `_swapToken1ForToken0SpecifiedOutput` silently sets `amountOutScaled = 1000` at line 874.
4. `_executeSwap` sets `amount0DeltaScaled = -1000`.
5. `swap` transfers 1000 token0 to `recipient` (not 5000) and returns `amount0Delta = -1000` — the transaction succeeds with no revert.
6. The caller paid token1 for 5000 token0 worth of output but received only 1000, a 4000-unit shortfall with no on-chain error.
