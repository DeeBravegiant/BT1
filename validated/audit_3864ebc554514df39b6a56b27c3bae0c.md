### Title
Unbounded `drepDelegs` Set Enables Attacker-Controlled DoS on DRep Unregistration via `ConwayUnRegDRep` Certificate Processing - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`)

---

### Summary

In the Conway era, `DRepState` stores a `Set (Credential Staking)` called `drepDelegs` representing every stake credential that has delegated its vote to a given DRep. There is no protocol-enforced upper bound on this set. When a DRep submits a `ConwayUnRegDRep` certificate to unregister and recover its deposit, the `GOVCERT` ledger rule unconditionally iterates over every element of `drepDelegs` to clear each delegator's pointer. An unprivileged attacker can register an arbitrarily large number of stake credentials and delegate them all to a victim DRep, making the DRep's unregistration transaction computationally unbounded relative to the transaction's own size, effectively freezing the DRep's deposit permanently.

---

### Finding Description

**Root cause — `DRepState.drepDelegs` has no size limit.**

`DRepState` is defined in `libs/cardano-ledger-core/src/Cardano/Ledger/DRep.hs`:

```haskell
data DRepState = DRepState
  { drepExpiry  :: !EpochNo
  , drepAnchor  :: !(StrictMaybe Anchor)
  , drepDeposit :: !(CompactForm Coin)
  , drepDelegs  :: !(Set (Credential Staking))   -- ← no bound
  }
``` [1](#0-0) 

Every time a stake credential delegates its vote to a credential DRep, `unDelegReDelegDRep` in `VState.hs` inserts the credential into this set with no size check:

```haskell
Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
``` [2](#0-1) 

**Vulnerable code path — `ConwayUnRegDRep` iterates over the entire set.**

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`, the `GOVCERT` transition rule for `ConwayUnRegDRep` defines:

```haskell
clearDRepDelegations delegs accountsMap =
  foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
...
certState'
  & certDStateL . accountsL . accountsMapL
    %~ clearDRepDelegations (drepDelegs dRepState)
``` [3](#0-2) 

`clearDRepDelegations` performs one `Map.adjust` (O(log A), where A is the total number of registered accounts) for every element of `drepDelegs`. The total work is **O(D × log A)** where D is the number of delegators — a quantity entirely under attacker control and completely decoupled from the size of the unregistration transaction itself, which is a small constant.

**No consent from the DRep is required for delegation.** Any stake credential holder can delegate to any DRep at any time by submitting a `ConwayDelegCert` or `ConwayRegDelegCert` certificate. The DRep has no mechanism to refuse or cap incoming delegations. [4](#0-3) 

---

### Impact Explanation

**High — Permanent freezing of the DRep's deposit.**

A DRep's registration deposit (`ppDRepDeposit`) is only refunded upon successful processing of a `ConwayUnRegDRep` certificate. If `clearDRepDelegations` must iterate over a sufficiently large `drepDelegs` set, the ledger state transition for that single certificate becomes so computationally expensive that:

1. No block producer will include the transaction in a block within the 1-second slot budget, because processing it would cause them to miss the slot.
2. The DRep's deposit is therefore permanently locked — recovery requires a hard fork to either cap `drepDelegs` or restructure the unregistration logic.

This matches the allowed impact: **"High. Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork."**

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender. The only barrier is economic: each delegating stake credential requires a `ppKeyDeposit` (currently 2 ADA on mainnet). However:

- The attacker does not lose these deposits permanently; they can unregister the stake credentials after the attack.
- The attacker only needs to sustain the delegation long enough to prevent the victim DRep from unregistering.
- The attack is asymmetric: the attacker pays O(D) deposits temporarily, while the victim's deposit is frozen permanently.
- There is no protocol mechanism for the DRep to remove delegators or cap the set.

---

### Recommendation

1. **Enforce a maximum delegator count per DRep.** Introduce a protocol parameter (e.g., `ppMaxDRepDelegators`) and reject any `ConwayDelegCert` or `ConwayRegDelegCert` that would cause a DRep's `drepDelegs` set to exceed this limit, analogous to the `MAX_DELEGATES` check in the referenced report.

2. **Decouple unregistration cost from delegator count.** Instead of eagerly clearing all delegation pointers in `clearDRepDelegations` at unregistration time, mark the DRep as unregistered and lazily clear individual delegator pointers when those stake credentials next interact with the ledger (e.g., on re-delegation or unregistration).

3. **Provide an opt-in/opt-out mechanism.** Allow DReps to refuse new delegations or set a cap on their `drepDelegs` set, preventing unsolicited inflation of the set by third parties.

### Proof of Concept

**Setup:**
- Attacker registers N stake credentials (cost: N × `ppKeyDeposit`, recoverable).
- Attacker delegates each to victim DRep via `ConwayDelegCert` / `ConwayRegDelegCert`.
- Victim DRep's `drepDelegs` set now has N entries.

**Trigger:**
- Victim DRep submits a `ConwayUnRegDRep` certificate in a transaction.
- The `GOVCERT` rule calls `clearDRepDelegations (drepDelegs dRepState)`.
- `foldr` performs N calls to `Map.adjust` on the accounts map (size A).
- Total work: O(N × log A).

**Result:**
- For large N (e.g., N = 500,000 with A = 10,000,000), the computation requires ~11.5 million map operations.
- This far exceeds what can be completed within a 1-second Cardano slot budget.
- No block producer includes the transaction; the DRep's deposit is permanently frozen.
- The attacker can then unregister their own stake credentials and recover their deposits, making the attack nearly cost-free in the long run.

- `drepDelegs` field (no bound): [5](#0-4) 
- `clearDRepDelegations` (unbounded iteration): [6](#0-5)
