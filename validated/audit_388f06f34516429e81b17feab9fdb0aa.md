Audit Report

## Title
Missing Staleness Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Oracle Arbitrage on Deposits and Withdrawals - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-related return values (`updatedAt`, `answeredInRound`, `roundId`), accepting any last-reported price regardless of age. This stale price propagates directly into `LRTDepositPool.getRsETHAmountToMint()` and `LRTWithdrawalManager.getExpectedAssetAmount()`, allowing an unprivileged attacker to receive excess rsETH on deposit or excess LST on withdrawal at the expense of other protocol participants. The team's own `ChainlinkOracleForRSETHPoolCollateral` contract, used for the L2 pool, already implements the missing checks, confirming awareness of the requirement.

## Finding Description

**Root cause â€” `contracts/oracles/ChainlinkPriceOracle.sol` L49â€“55:**

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

All five return values of `latestRoundData()` are available but only `answer` is consumed. The `updatedAt` timestamp (heartbeat timeout detection) and the `answeredInRound >= roundId` check (incomplete round detection) are silently discarded.

**Contrast with `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L30â€“32:**

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The L2 pool oracle applies all three guards. The main deposit/withdrawal oracle applies none.

**Propagation path â€” deposit (`contracts/LRTDepositPool.sol` L520):**

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` delegates to `ChainlinkPriceOracle.getAssetPrice()` (`contracts/LRTOracle.sol` L156â€“158). `lrtOracle.rsETHPrice()` is a stored value updated separately by `updateRSETHPrice()`. If the Chainlink feed is stale at a price higher than true market, the numerator is inflated and the attacker receives excess rsETH.

**Propagation path â€” withdrawal (`contracts/LRTWithdrawalManager.sol` L593):**

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

If the Chainlink feed is stale at a price lower than true market, the denominator is deflated and the attacker's `expectedAssetAmount` is inflated, locking in an over-disbursement at claim time.

No other guard in either path checks oracle freshness. The `pricePercentageLimit` circuit-breaker in `LRTOracle._updateRsETHPrice()` only fires when `updateRSETHPrice()` is called, not during view-only deposit/withdrawal price reads.

## Impact Explanation

**Deposit (stale HIGH price):** Attacker deposits LST and receives rsETH computed against an inflated asset price. The excess rsETH represents a proportional dilution of all existing rsETH holders' claim on protocol TVL â€” direct theft of user funds at rest. This matches the Critical impact class.

**Withdrawal (stale LOW price):** Attacker calls `initiateWithdrawal()` while the feed reports a depressed price, locking in an `expectedAssetAmount` larger than fair value. The protocol disburses more LST than the burned rsETH is worth, draining protocol assets â€” protocol insolvency. This also matches the Critical impact class.

Both paths are reachable by any unprivileged external caller holding the relevant LST or rsETH.

## Likelihood Explanation

Chainlink LST/ETH feeds on mainnet have heartbeat intervals (typically 1 hour) and deviation thresholds. Stale windows occur during network congestion, oracle node failures, or periods of low price volatility where the deviation threshold is never crossed. A sophisticated attacker monitoring oracle latency can identify a stale window and execute the deposit or withdrawal within it. No privileged access, governance capture, or external protocol compromise is required. Likelihood is Medium â€” the condition is not constant but is a known, recurring property of Chainlink feeds.

## Recommendation

Apply the same staleness checks already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > HEARTBEAT_TIMEOUT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

A per-feed configurable `heartbeatTimeout` mapping should also be added to enforce maximum acceptable data age.

## Proof of Concept

1. Assume the stETH/ETH Chainlink feed last updated 2 hours ago at `1.05e18` (stale). True market price has since dropped to `1.00e18`.
2. `lrtOracle.rsETHPrice()` is a stored value, e.g., `1.02e18`, set at the last `updateRSETHPrice()` call.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `_beforeDeposit` â†’ `getRsETHAmountToMint(stETH, 100e18)` â†’ `lrtOracle.getAssetPrice(stETH)` â†’ `ChainlinkPriceOracle.getAssetPrice(stETH)` â†’ returns stale `1.05e18`.
5. `rsethAmountToMint = (100e18 * 1.05e18) / 1.02e18 â‰ˆ 102.94e18`.
6. Fair value: `(100e18 * 1.00e18) / 1.02e18 â‰ˆ 98.04e18`. Attacker receives ~4.9 rsETH excess.
7. No revert occurs anywhere; `ChainlinkPriceOracle` performs zero staleness validation.
