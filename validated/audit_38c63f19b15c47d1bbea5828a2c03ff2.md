Audit Report

## Title
ETH Deposit Limit Guard Omits the Deposit Amount, Allowing Over-Deposit — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool` uses an asymmetric check: for ERC-20 assets it correctly evaluates `totalAssetDeposits + amount > limit`, but for ETH it evaluates only `totalAssetDeposits > limit`, silently discarding the incoming `amount`. This means the ETH deposit cap is never enforced prospectively — it only detects a limit that is already exceeded, not one that a deposit would exceed.

## Finding Description
The root cause is confirmed at [1](#0-0)  — the ETH branch on line 679 returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` without adding `amount`, while the ERC-20 branch on line 681 correctly adds `amount`.

This check is the sole pre-deposit guard, called unconditionally from `_beforeDeposit` at [2](#0-1) 

`depositETH` passes `msg.value` as `depositAmount` to `_beforeDeposit` at [3](#0-2) , but that value is dropped inside the ETH branch, so the revert on line 662 is never triggered by a deposit that would push ETH over the cap — only by one made after the cap is already breached.

The deposit limit is stored in [4](#0-3)  and is the primary risk-management knob for ETH exposure.

## Impact Explanation
The ETH deposit cap is the protocol's primary mechanism for bounding EigenLayer exposure. With the guard broken, any depositor can push `getTotalAssetDeposits(ETH)` to an arbitrary multiple of the configured limit in a single transaction. The protocol mints rsETH for every wei deposited beyond the cap, diluting the backing ratio for all existing rsETH holders. This matches **Low — Contract fails to deliver promised returns**, as the protocol fails to enforce its own stated risk bound without direct fund loss to depositors.

## Likelihood Explanation
`depositETH` is a public payable function with no role restriction. [5](#0-4)  The only prerequisite is that `totalAssetDeposits < depositLimitByAsset` (the normal operating state). No special conditions, front-running, or privileged access are required. Likelihood is **High**.

## Recommendation
Apply the same `+ amount` pattern used for ERC-20 assets to the ETH branch, eliminating the special case entirely:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
```
Setup:
  depositLimitByAsset[ETH_TOKEN] = 100 ether
  getTotalAssetDeposits(ETH_TOKEN) = 99 ether  // one wei below cap

Attack:
  attacker calls depositETH{value: 50 ether}()

_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50 ether):
  totalAssetDeposits = 99 ether
  ETH branch: return (99 ether > 100 ether) → false  ← 50 ether never used
  → MaximumDepositLimitReached() NOT triggered

Result:
  50 ether deposited, rsETH minted
  getTotalAssetDeposits(ETH_TOKEN) = 149 ether (49 ether over cap)

ERC-20 under identical conditions:
  return (99 ether + 50 ether > 100 ether) → true → revert ✓

Foundry test plan:
  1. Deploy LRTDepositPool with depositLimitByAsset[ETH] = 100 ether
  2. Seed pool to 99 ether via prior deposits
  3. Call depositETH{value: 50 ether}() — assert it does NOT revert
  4. Assert getTotalAssetDeposits(ETH) == 149 ether (exceeds cap by 49 ether)
  5. Repeat with an ERC-20 asset to confirm the revert fires correctly
```
