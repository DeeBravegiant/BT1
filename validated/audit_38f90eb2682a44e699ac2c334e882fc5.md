### Title
NFT Royalty Bypass via Integer Division Truncation in Offer Construction — (File: `chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
The `make_nft1_offer` function computes per-NFT trade prices and royalty amounts using integer division. An unprivileged seller can include many cheap NFTs in a single offer to dilute the per-NFT trade price to zero, causing the wallet to produce an empty `trade_prices_list` for the NFT ownership-layer spend. Because the on-chain `NFT_TRANSFER_PROGRAM_DEFAULT` puzzle only enforces royalties that appear in that list, the target NFT is transferred with zero royalty paid to the creator.

### Finding Description

**Root cause — `compute_royalty_amount`:** [1](#0-0) 

```python
def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
```

`royalty_split` is the count of royalty-enabled NFTs on the same side of the offer. When this count is large, `abs(offered_amount) // royalty_split` truncates to zero before the percentage is applied, yielding a zero royalty.

**Root cause — `trade_prices` per-NFT dilution:** [2](#0-1) 

```python
trade_prices.append((uint64(amount // offer_side_royalty_split), settlement_ph))
```

`offer_side_royalty_split` is the count of royalty-enabled NFTs being *offered*. If the seller bundles N NFTs and requests fewer mojos than N, `amount // offer_side_royalty_split` truncates to 0.

**Filter that silently drops zero trade prices:** [3](#0-2) 

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
```

A zero trade price is silently dropped. The NFT spend is submitted with an empty `trade_prices_list` inside the `CHANGE_OWNER` (`-10`) condition.

**On-chain consequence:**

The `NFT_TRANSFER_PROGRAM_DEFAULT` (`NFT_OWNERSHIP_TRANSFER_PROGRAM_ONE_WAY_CLAIM_WITH_ROYALTIES`) computes required royalty payments from the trade prices embedded in the spend. With an empty list, it requires no royalty payment. The spend is accepted by consensus.

The existing test explicitly acknowledges that small amounts truncate to zero: [4](#0-3) 

### Impact Explanation

An NFT creator who set a royalty percentage receives zero payment despite a completed sale. This is an unauthorized accounting change affecting NFTs — royalty coins that should be created are never created, and the on-chain puzzle accepts the spend because the `trade_prices_list` it receives is empty. The impact falls under the Critical scope ("accounting change affecting NFTs").

### Likelihood Explanation

The attacker (seller) needs to:
1. Mint N cheap NFTs (zero-royalty, minimal cost).
2. Bundle them with the target NFT in a single offer, requesting fewer mojos than N.
3. Find a buyer willing to accept the bundle (the buyer gets many NFTs cheaply, so this is attractive).

Minting cheap NFTs is permissionless and low-cost. No privileged access, leaked keys, or cryptographic breaks are required.

### Recommendation

1. **Enforce a minimum trade price per NFT.** Before filtering, assert that `amount // offer_side_royalty_split >= 1` (or raise an error if the diluted price would be zero).
2. **Enforce a minimum royalty amount.** In `compute_royalty_amount`, if `percentage > 0` and `offered_amount > 0` but the computed royalty is 0, raise a `ValueError` or return a minimum of 1 mojo.
3. **Cap `offer_side_royalty_split` / `request_side_royalty_split`** to a reasonable maximum to prevent deliberate dilution.

### Proof of Concept

Seller owns NFT-A (10 % royalty, creator expects royalties) and mints 9 999 zero-royalty NFTs (NFT-B₁ … NFT-B₉₉₉₉).

Seller calls `make_nft1_offer` with:
```
offer_dict = {
    NFT_A:    -1,   # offered
    NFT_B_1:  -1,   # offered (cheap)
    ...
    NFT_B_9999: -1, # offered (cheap)
    None:     +9999 # requesting 9 999 mojos XCH
}
```

Inside `make_nft1_offer`:
- `offer_side_royalty_split = 10 000`
- `trade_prices = [(9999 // 10000, OFFER_MOD_HASH)] = [(0, OFFER_MOD_HASH)]`
- Filter: `0 * 1000 // 10000 == 0` → trade price dropped → `trade_prices_list = []`
- Royalty coin: `sum(p.amount for _, p in payments) == 0` → royalty spend skipped [5](#0-4) 

The resulting spend bundle transfers NFT-A to the buyer with an empty `trade_prices_list`. The `NFT_TRANSFER_PROGRAM_DEFAULT` puzzle enforces no royalty. The spend is accepted on-chain. The NFT creator receives 0 mojos.

```python
def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
