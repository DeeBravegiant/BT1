### Title
Missing Lower-Bound Wall-Clock Check on `ProposalInit.timestamp` Allows Proposer to Commit Stale Block Timestamps — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` enforces only a **future** bound on the proposed block timestamp (`timestamp ≤ now + window`) but never enforces a **past** bound (`timestamp ≥ now − window`). The lower bound is anchored only to the previous block's timestamp, not to the current wall clock. A malicious proposer can therefore set `init.timestamp` to any value ≥ `last_block_timestamp` — potentially many seconds or minutes in the past — and every honest validator will accept the proposal, committing a stale timestamp to the canonical block header.

### Finding Description

`is_proposal_init_valid` in `validate_proposal.rs` performs two timestamp checks:

```rust
// lower bound: only vs. previous block, NOT vs. now
if init_proposed.timestamp < last_block_timestamp { ... }

// upper bound: vs. now + window
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds { ... }
``` [1](#0-0) 

The field `block_timestamp_window_seconds` is documented as *"Maximum allowed deviation (seconds) of a proposed block's timestamp from the current time"* and is configured to **1 second** in both the default code and the production deployment config. [2](#0-1) [3](#0-2) 

The description implies a symmetric window `[now − 1 s, now + 1 s]`, but only the upper half is enforced. The lower half is bounded only by `last_block_timestamp`, which in a normally running chain is the timestamp of the previous block (~10 s ago), and after a restart or sync gap can be arbitrarily older.

The same asymmetric pattern appears in the sync path (`try_sync`):

```rust
&& timestamp.0 >= last_block_timestamp          // no now-relative lower bound
&& timestamp.0 <= now + self.config.static_config.block_timestamp_window_seconds
``` [4](#0-3) 

### Impact Explanation

The stale timestamp is passed directly to `convert_to_sn_api_block_info(init)` and forwarded to the batcher as `block_info.block_timestamp`. [5](#0-4) 

Once the batcher executes transactions against this `BlockInfo`, every `get_block_timestamp` syscall inside that block returns the attacker-controlled stale value. Smart contracts that use `get_block_timestamp` for time-locks, auctions, vesting schedules, or expiry checks will observe a wrong timestamp, producing wrong execution results that are committed to the canonical state.

Additionally, `get_l1_prices_in_fri_and_wei` is called with `init_proposed.timestamp` to look up historical L1 gas prices: [6](#0-5) 

A stale timestamp causes the validator to look up older L1 oracle data. The `within_margin` check bounds the deviation, but the margin is a percentage of the reference price, so a large historical price divergence can still cause the committed gas prices to differ from the current market price.

**Impact class**: Critical — wrong `get_block_timestamp` syscall result from blockifier/syscall execution logic for accepted input; wrong gas price committed to block header.

### Likelihood Explanation

Requires a validator that is selected as proposer for a given round. In a permissionless validator set any staker can eventually be a proposer. The attack requires no special tooling: the proposer simply sets `init.timestamp = last_block_timestamp` when constructing `ProposalInit` in `initiate_build`. [7](#0-6) 

### Recommendation

Add a symmetric lower-bound check in `is_proposal_init_valid`, mirroring the existing upper-bound check:

```rust
// Existing upper bound (future):
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds { ... }

// Missing lower bound (past) — add this:
if now > proposal_init_validation.block_timestamp_window_seconds
    && init_proposed.timestamp < now - proposal_init_validation.block_timestamp_window_seconds
{
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "Timestamp is too far in the past: now={}, block_timestamp_window_seconds={}, \
             proposed={}",
            now,
            proposal_init_validation.block_timestamp_window_seconds,
            init_proposed.timestamp
        ),
    ));
}
```

Apply the same fix to the `try_sync` timestamp validation at line 1067–1070 of `sequencer_consensus_context.rs`.

### Proof of Concept

1. Validator A is selected as proposer for height H, round 0.
2. The previous block was committed with `last_block_timestamp = T − 30` (30 seconds ago, e.g. after a brief pause).
3. Validator A constructs `ProposalInit` with `timestamp = T − 30` (equal to `last_block_timestamp`).
4. Honest validators call `is_proposal_init_valid`:
   - `T − 30 >= last_block_timestamp (T − 30)` → **passes** (equal)
   - `T − 30 <= now (T) + 1` → **passes** (past timestamp is well below `now + 1`)
   - No lower-bound check against `now − 1` exists → **no rejection**
5. `initiate_validation` forwards `block_info.block_timestamp = T − 30` to the batcher.
6. Every transaction in the block executes with `get_block_timestamp()` returning `T − 30`.
7. A time-lock contract that checks `get_block_timestamp() >= unlock_time` where `unlock_time = T − 15` would incorrectly report the lock as **not yet expired**, even though 30 seconds have passed since the previous block.
8. The block is finalized and the wrong execution results are committed to the canonical state. [8](#0-7) [9](#0-8)

#[instrument(level = "warn", skip_all, fields(?proposal_init_validation, ?init_proposed))]
async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }

async fn initiate_validation(
    batcher: Arc<dyn BatcherClient>,
    state_sync_client: SharedStateSyncClient,

    timeout_plus_margin: Duration,
    clock: &dyn Clock,
    compare_retrospective_block_hash: bool,

    let chrono_timeout = chrono::Duration::from_std(timeout_plus_margin)
        .expect("Can't convert timeout to chrono::Duration");

    let input = ValidateBlockInput {
        proposal_id,
        deadline: clock.now() + chrono_timeout,
        retrospective_block_hash: retrospective_block_hash(

            init,

        block_info: convert_to_sn_api_block_info(init)?,

        ValidateProposalError::Batcher(
            format!("Failed to initiate validate proposal {input:?}."),
            err,
        )
    })?;
    Ok(())
}
