### Title
Per-Block Reference Script Size Limit Not Enforced for Sub-Transactions in Dijkstra Era — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`, `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the per-transaction ref script size check (`validateAllRefScriptSize`) correctly accounts for sub-transaction ref scripts via `batchNonDistinctRefScriptsSize`, but the per-block check (`validateBodyRefScriptsSizeTooBig` / `totalRefScriptSizeInBlock`) only inspects top-level transaction inputs and is completely blind to sub-transaction ref scripts. An unprivileged block producer can craft batch transactions whose sub-transactions collectively reference far more ref script bytes than `ppMaxRefScriptSizePerBlock` allows, and the block-level guard will not fire.

---

### Finding Description

**Per-tx check** — `validateAllRefScriptSize` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`:

```haskell
runTest $ validateAllRefScriptSize pp originalUtxo tx   -- line 392
```

calls `batchNonDistinctRefScriptsSize`:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
```

This correctly sums ref script bytes from the top-level tx **and every sub-tx**, all against the same `utxo`. The result is compared against `ppMaxRefScriptSizePerTx`. [1](#0-0) [2](#0-1) 

**Per-block check** — `dijkstraBbodyTransition` delegates to the unmodified Conway function:

```haskell
Conway.validateBodyRefScriptsSizeTooBig @era pp blockBody (ls ^. utxoL)  -- line 363
``` [3](#0-2) 

`totalRefScriptSizeInBlock` iterates over `StrictSeq (Tx TopTx era)` and for each top-level tx calls `txNonDistinctRefScriptsSize`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [4](#0-3) [5](#0-4) 

`txNonDistinctRefScriptsSize` only reads `referenceInputsTxBodyL` and `inputsTxBodyL` of the **top-level** tx. It has no knowledge of `subTransactionsTxBodyL`. Sub-transaction inputs — and therefore sub-transaction ref scripts — are **never counted** in the block-level total.

---

### Impact Explanation

The correct impact is **Medium** — not the Critical ledger-divergence impact claimed in the question. All honest nodes evaluate the same (broken) rule identically, so there is no divergence. The actual impact is:

> Attacker-controlled transactions exceed intended validation limits.

The per-block ref script size limit (`ppMaxRefScriptSizePerBlock`, currently 1 MiB) was introduced specifically to cap the script-deserialization work imposed on validators per block. Because `totalRefScriptSizeInBlock` ignores sub-tx ref scripts, a block producer can include a block whose actual ref-script deserialization burden is a multiple of the intended cap, enabling a resource-exhaustion / DDoS attack on all validating nodes.

---

### Likelihood Explanation

- Requires only an unprivileged block producer (stake pool operator).
- Requires pre-existing UTxOs with large ref scripts (trivially created by the attacker in prior transactions).
- The attack is deterministic and locally testable.
- No governance, key leakage, or third-party compromise is needed.

---

### Recommendation

`totalRefScriptSizeInBlock` (or a Dijkstra-specific override) must be extended to also sum `txNonDistinctRefScriptsSize` for every sub-transaction of each top-level batch transaction, mirroring the logic already present in `batchNonDistinctRefScriptsSize`. The simplest fix is to introduce a `dijkstraTotalRefScriptSizeInBlock` that replaces the inner `txNonDistinctRefScriptsSize` call with `batchNonDistinctRefScriptsSize` for Dijkstra-era blocks, and wire it into `dijkstraBbodyTransition` instead of the Conway function.

---

### Proof of Concept

1. Submit a transaction creating N UTxOs each holding a ~100 KB Plutus ref script.
2. Craft a Dijkstra batch transaction with:
   - Top-level tx body: no ref inputs (contributes 0 to `totalRefScriptSizeInBlock`).
   - K sub-transactions each referencing a distinct 100 KB ref-script UTxO.
   - `batchNonDistinctRefScriptsSize` = K × 100 KB ≤ 200 KB (per-tx limit) → per-tx check passes.
3. Include M such batch transactions in one block, where M × K × 100 KB >> 1 MiB.
4. `totalRefScriptSizeInBlock` returns 0 for every batch tx → block-level check passes.
5. The block is accepted by all honest nodes despite imposing M × K × 100 KB of ref-script deserialization work, far exceeding `ppMaxRefScriptSizePerBlock`.

totalRefScriptSizeInBlock ::
  (AlonzoEraTx era, BabbageEraTxBody era) => ProtVer -> StrictSeq (Tx TopTx era) -> UTxO era -> Int
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      let updatedUtxo = accUtxo `Map.union` unUTxO toAdd
          toAdd
            | IsValid True <- tx ^. isValidTxL = txouts $ tx ^. bodyTxL
            | otherwise = collOuts $ tx ^. bodyTxL
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
