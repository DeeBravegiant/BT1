### Title
Missing CKD Output Verification for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge and Deliver Arbitrary CKD Responses - (File: `crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, the `ckd_output_check` cryptographic pairing verification is only executed for the `AppPublicKeyPV` variant. When a user submits a CKD request using the legacy `AppPublicKey` variant, the contract performs **no cryptographic verification** of the response before resolving all pending yields. A single attested participant (Byzantine, below threshold) can call `respond_ckd` with an arbitrary forged `CKDResponse`, which is accepted unconditionally and delivered to all waiting callers, permanently draining the pending queue and preventing the legitimate threshold-computed response from ever being delivered.

---

### Finding Description

The `respond_ckd` function in `crates/contract/src/lib.rs` contains a conditional output check:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which can only be satisfied if the MPC nodes collectively computed the correct CKD output using the threshold key material. [2](#0-1) 

For the `AppPublicKey` (legacy) variant, the arm is an empty no-op (`{}`). The function then unconditionally calls `resolve_yields_for`, which drains the entire fan-out queue for that request key and delivers the unverified response to every waiting caller. [3](#0-2) 

The structural parallel to the original report is exact:
- **Original**: `exitData.exit = false` → ownership check skipped; `unlockData.unlock = true` → `unlock()` proceeds on attacker-controlled `tokenId` without any ownership check.
- **NEAR MPC**: `AppPublicKey` variant → `ckd_output_check` skipped; `resolve_yields_for` proceeds with attacker-controlled `CKDResponse` without any cryptographic check.

The existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` (line 3404) confirms this: it uses `AppPublicKey` with completely arbitrary byte arrays `[1u8; 48]` and `[2u8; 48]` as `big_y` and `big_c`, and the call succeeds with no verification. [4](#0-3) 

The `AppPublicKey` variant is the documented legacy/default format and is actively used in production integrations. [5](#0-4) 

---

### Impact Explanation

**Medium — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants.**

A single Byzantine participant (below threshold) can:

1. Monitor the contract for pending CKD requests using the `AppPublicKey` variant.
2. Race-call `respond_ckd` with an arbitrary `CKDResponse` (e.g., random G1 points) before the legitimate threshold-computed response is submitted.
3. `resolve_yields_for` drains the entire fan-out queue, delivering the forged garbage to all waiting callers and removing the request from `pending_ckd_requests`.
4. The legitimate response submitted later by the honest nodes hits `RequestNotFound` and is discarded.

The user receives a `CKDResponse` whose `big_c` and `big_y` do not satisfy the CKD encryption relation. Decrypting `big_c` with the user's secret key yields garbage — the user cannot recover their app private key. Because the pending queue is fully drained, re-delivery of the correct response is impossible without the user re-submitting a new request (with a fresh ephemeral key pair), which the attacker can block again indefinitely.

This breaks the core safety invariant that CKD responses must be the product of threshold-authorized computation: a single participant below threshold can unilaterally resolve any `AppPublicKey` CKD request.

---

### Likelihood Explanation

**High.** The `AppPublicKey` variant is the legacy default and is widely used. Any single attested participant — the minimum possible adversary in the MPC threat model — can exploit this. No collusion, no leaked key material, and no privileged access beyond being an active participant is required. The attacker only needs to observe the NEAR chain for pending CKD requests (public information) and submit a `respond_ckd` call before the honest nodes do.

---

### Recommendation

Require all CKD requests to use the `AppPublicKeyPV` variant, which carries the G2 component needed for the on-chain pairing check. Deprecate and eventually reject `AppPublicKey` submissions at the `request_app_private_key` entry point. If backward compatibility must be preserved, the contract should refuse to call `resolve_yields_for` for `AppPublicKey` requests until a threshold-consistent mechanism (e.g., a multi-node vote or an off-chain aggregated proof) is in place to authenticate the response.

At minimum, add a guard that panics for the `AppPublicKey` arm rather than silently accepting any response:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {
        // No public verifiability: cannot authenticate response on-chain.
        // Reject until a safe mechanism is available.
        env::panic_str("AppPublicKey variant is not supported in respond_ckd: use AppPublicKeyPV");
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. User Alice submits `request_app_private_key` with `AppPublicKey` variant (legacy G1 point). The request is stored in `pending_ckd_requests`.

2. Malicious participant Eve (a single attested node, below threshold) observes the pending request on-chain.

3. Eve calls `respond_ckd` with the correct `CKDRequest` key but a forged `CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }`.

4. The `AppPublicKey` arm executes `{}` — no check. `resolve_yields_for` drains Alice's yield queue and delivers the forged response. [1](#0-0) 

5. The honest MPC nodes finish their threshold computation and call `respond_ckd` with the correct response. `resolve_yields_for` returns `Err(RequestNotFound)` — the queue is already gone. [6](#0-5) 

6. Alice's yield-resume callback fires with garbage `big_c`/`big_y`. Decryption produces garbage; Alice cannot recover her app private key.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
