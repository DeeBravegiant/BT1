Audit Report

## Title
Missing Slippage Guard in L2 Pool `deposit()` Functions Allows Silent Under-Delivery of rsETH - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

## Summary
Every L2 pool `deposit()` function computes the rsETH output at execution time from a live oracle rate with no minimum-output guard. If the oracle rate is updated between a user's transaction submission and its inclusion in a block, the user silently receives fewer wrsETH/rsETH tokens than the contract's own `viewSwapRsETHAmountAndFee()` preview function indicated. The L1 `LRTDepositPool` already enforces this protection via `minRSETHAmountExpected`; the L2 pools do not.

## Finding Description
All L2 pool `deposit()` functions read the live oracle rate at execution time:

```solidity
// RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [1](#0-0) 

The ETH deposit path in `RSETHPoolV3` calls this function and mints immediately with no floor check:

```solidity
// RSETHPoolV3.sol L258-262
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);
``` [2](#0-1) 

The same unchecked pattern appears in every pool variant:

- `RSETHPoolV3.deposit(address,uint256,string)` — token overload [3](#0-2) 
- `RSETHPool.deposit(string)` and `RSETHPool.deposit(address,uint256,string)` [4](#0-3) 
- `RSETHPoolNoWrapper.deposit(address,uint256,string)` [5](#0-4) 

By contrast, `LRTDepositPool._beforeDeposit()` on L1 enforces a hard revert:

```solidity
// LRTDepositPool.sol L667-669
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [6](#0-5) 

The L2 oracle (`rsETHOracle`) is a cross-chain rate pushed from L1 via `MultiChainRateProvider`. When the rate is pushed (a routine, recurring event), any pending `deposit()` transaction that executes after the push will compute a lower rsETH output than the user observed in `viewSwapRsETHAmountAndFee()`. There is no on-chain guard to revert the transaction.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The deposited ETH/LST is not stolen; it remains in the pool backing the minted rsETH at the updated rate. However, the user receives fewer tokens than the contract's own public view function promised at the time of transaction construction. The shortfall is proportional to the rate increase since submission. For a routine weekly reward accrual (~0.077% rate increase), a 10 ETH deposit yields approximately 0.0077 ETH worth of rsETH less than expected — a silent, unrecoverable under-delivery with no on-chain recourse.

## Likelihood Explanation
No attacker is required. The oracle rate is updated periodically as EigenLayer staking rewards are processed on L1 and bridged to L2 — a routine, recurring, permissionless event. Any user whose `deposit()` transaction is pending in the mempool at the moment of an oracle update is affected. L2 chains (Arbitrum, Optimism, Base) have public mempools and predictable block times, making the timing window well-defined and repeatable. The condition is triggered by normal protocol operation, not adversarial action.

## Recommendation
Add a `minRsETHAmountExpected` parameter to every L2 pool `deposit()` function and revert if the computed output falls below it, mirroring the protection in `LRTDepositPool._beforeDeposit()`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

Apply the same pattern to all token `deposit()` overloads across `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPool`, and `RSETHPoolNoWrapper`.

## Proof of Concept
1. Alice calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolV3` and observes she will receive `X` wrsETH at oracle rate `R`.
2. Alice submits `deposit{value: 1 ether}("ref")`.
3. Before Alice's transaction is included, the L2 oracle is updated: rate increases from `R` to `R'` (routine reward accrual bridged from L1).
4. Alice's transaction executes. `viewSwapRsETHAmountAndFee` now uses `R'`:
   - `rsETHAmount = (1 ether - fee) * 1e18 / R'`
   - Since `R' > R`, `rsETHAmount < X`.
5. Alice receives fewer wrsETH than `viewSwapRsETHAmountAndFee` promised. No revert occurs.

**Foundry fork test plan:**
- Fork an L2 (e.g., Arbitrum) at a block just before a known oracle rate update.
- Call `viewSwapRsETHAmountAndFee(1 ether)` and record `expectedAmount`.
- Roll forward one block (post oracle update) and call `deposit{value: 1 ether}("ref")`.
- Assert `wrsETH.balanceOf(alice) < expectedAmount` — the test passes, confirming the under-delivery with no revert.
