Audit Report

## Title
Missing Chainlink `latestRoundData()` Staleness and Validity Checks Allow Corrupted rsETH Price Commit - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `roundId`, `updatedAt`, and `answeredInRound`, accepting stale or zero prices without reversion. The corrupted asset exchange rate propagates through `LRTOracle._getTotalEthInProtocol()` into `_updateRsETHPrice()`, permanently committing a wrong `rsETHPrice` that governs all subsequent deposits. The same codebase already implements the correct three-check pattern in `ChainlinkOracleForRSETHPoolCollateral`, confirming the omission is inconsistent and unintentional.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink answer as:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All three validity fields are discarded. The sister contract in the same repository performs the correct checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated price flows directly into `LRTOracle._getTotalEthInProtocol()`:

```solidity
// contracts/LRTOracle.sol L339-343
uint256 assetER = getAssetPrice(asset);
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```

`totalETHInProtocol` then drives the committed price:

```solidity
// contracts/LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

`updateRSETHPrice()` is `public whenNotPaused` with no role restriction, so any external caller can trigger the price commit at any time.

**Existing mitigations are insufficient:**

The `pricePercentageLimit` guard at L252-266 and the downside-pause at L270-283 both contain the condition `pricePercentageLimit > 0`, meaning they are entirely disabled when `pricePercentageLimit == 0` (the default after initialization). Even when set, they only catch deviations exceeding the configured threshold — a stale price within the threshold (e.g., a 0.5% stale deviation against a 1% limit) passes through silently and is permanently committed. The Solidity 0.8 arithmetic overflow does prevent the negative-price-cast scenario (since `uint256(negative) * 1e18` overflows), but staleness and zero-price scenarios are not caught.

## Impact Explanation
When a Chainlink feed returns a stale price lower than the true current price, `newRsETHPrice` is computed below its correct value. Any depositor calling `LRTDepositPool.depositAsset()` immediately after receives more rsETH than their deposit warrants (minting is proportional to `1 / rsETHPrice`). This dilutes the ETH backing of every existing rsETH holder — a direct transfer of value from existing holders to the new depositor. This constitutes **theft of unclaimed yield** from existing rsETH holders, matching the High impact class. Repeated exploitation across multiple stale windows compounds the loss.

## Likelihood Explanation
Chainlink feeds go stale during L2 sequencer outages, network congestion, and feed deprecations — all documented, on-chain-observable events. The `answeredInRound < roundId` staleness indicator is explicitly documented by Chainlink as the canonical staleness check. `updateRSETHPrice()` is callable by any unprivileged address with no cooldown, so an attacker can monitor the mempool for a stale-feed condition and immediately call the function. No special privileges, victim mistakes, or governance capture are required.

## Recommendation
Apply the same three-check pattern already used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(answeredInRound >= roundId, "Stale price");
require(updatedAt != 0, "Incomplete round");
require(price > 0, "Invalid price");
```

Additionally, apply equivalent staleness validation in `RSETHPriceFeed.latestRoundData()` and `RSETHPriceFeed.getRoundData()`, which forward the raw ETH/USD Chainlink answer without any validity check before multiplying by `rsETHPrice`.

## Proof of Concept

**Fork test outline (Foundry):**

1. Fork mainnet (or target L2) at a block where the stETH/ETH Chainlink feed is live.
2. Warp time forward past the feed's heartbeat interval without advancing the round (simulating a sequencer outage or feed staleness) so that `answeredInRound < roundId` holds.
3. Call `LRTOracle.updateRSETHPrice()` as an unprivileged EOA — confirm it does not revert.
4. Record `rsETHPrice` after the call; compare against the value that would result from a fresh, valid feed answer.
5. As a second EOA, call `LRTDepositPool.depositAsset(stETH, amount)` and record the rsETH minted.
6. Restore the feed to a valid round and call `updateRSETHPrice()` again; record the corrected `rsETHPrice`.
7. Assert that the rsETH minted in step 5 exceeds what would have been minted at the correct price — quantifying the value extracted from existing holders.
