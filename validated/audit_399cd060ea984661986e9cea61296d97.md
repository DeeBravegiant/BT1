Audit Report

## Title
stETH Transfer Rounding Causes rsETH Over-Minting on Every Deposit - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.depositAsset` computes the rsETH mint amount from the caller-supplied `depositAmount` before executing the `safeTransferFrom`, so the mint is based on a value that is 1–2 wei larger than the stETH the contract actually receives due to Lido's share-based rounding. Because `getTotalAssetDeposits` reads the real on-chain `balanceOf`, the oracle's next `rsETHPrice` update will reflect the actual (slightly lower) stETH balance, permanently diluting all rsETH holders by the shortfall introduced on every stETH deposit.

## Finding Description
In `LRTDepositPool.depositAsset` (lines 111–115), `_beforeDeposit` is called with the user-supplied `depositAmount`, which internally calls `getRsETHAmountToMint(asset, depositAmount)` computing `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`. The `safeTransferFrom` executes afterward; stETH's share-based accounting rounds down on every transfer, delivering `depositAmount − 1` or `depositAmount − 2` wei. The contract never measures the balance delta, so `rsethAmountToMint` is always computed from a value 1–2 wei larger than what was actually received.

`getTotalAssetDeposits` → `getAssetDistributionData` reads `IERC20(asset).balanceOf(address(this))` (the real balance). When `LRTOracle.updateRSETHPrice` runs, `_getTotalEthInProtocol` calls `getTotalAssetDeposits` and therefore sees the actual (slightly lower) stETH balance. The resulting `rsETHPrice` is fractionally lower than it would be if the full `depositAmount` had been received, permanently diluting all rsETH holders by the per-deposit shortfall. No existing check in `_beforeDeposit` or `depositAsset` compares pre/post balances; the `minRSETHAmountExpected` guard only protects the depositor's slippage, not the protocol's accounting integrity.

stETH is a first-class supported asset: `LRTConfig.initialize` calls `_addNewSupportedAsset(stETH, 100_000 ether)` at deployment, and `LRTConstants.ST_ETH_TOKEN` is referenced throughout the codebase.

## Impact Explanation
**Low – Contract fails to deliver promised returns.** Each stETH deposit introduces a 1–2 wei shortfall between stETH held and rsETH minted. The oracle self-corrects by pricing rsETH from actual holdings, but this correction dilutes all existing rsETH holders rather than reversing the over-mint. Across thousands of deposits the cumulative shortfall grows monotonically; the last stETH redeemers receive marginally less stETH than their rsETH entitles them to. No single user loses a meaningful amount, but the protocol's stETH backing is permanently and irreversibly eroded with every deposit.

## Likelihood Explanation
**Medium.** stETH is enabled at deployment and any unprivileged external caller can invoke `depositAsset(stETH, ...)`. The rounding occurs unconditionally on every stETH `transferFrom` call; no special conditions, attacker capability, or victim mistake is required. The effect is deterministic and repeatable.

## Recommendation
Measure the actual stETH received by comparing balances before and after the transfer, and use the real received amount for rsETH minting:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;

uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
_mintRsETH(rsethAmountToMint);
```

This requires moving the `_beforeDeposit` call (and its deposit-limit check) to use `actualReceived`. The balance-delta pattern is already used correctly in `KernelDepositPool.notifyRewardAmount` (lines 573–577).

## Proof of Concept
1. Alice calls `depositAsset(stETH, 1_000_000_000, 0, "")`.
2. `_beforeDeposit` computes `rsethAmountToMint` using `1_000_000_000` as the stETH amount (line 665, feeding into `getRsETHAmountToMint` at line 520).
3. `safeTransferFrom` executes (line 114); due to stETH share rounding the contract receives `999_999_999` wei.
4. `_mintRsETH` mints rsETH corresponding to `1_000_000_000` wei (line 115).
5. The next `LRTOracle.updateRSETHPrice` call reads `balanceOf(depositPool)` = `999_999_999`, computes a slightly lower `rsETHPrice`, diluting all existing holders.
6. A Foundry fork test can confirm this by asserting `IERC20(stETH).balanceOf(depositPool) * rsETHPrice < rsETH.totalSupply() * 1e18` after a single deposit.
