### Title
Meta-Transaction Deposit Refund Misdirection: Relayer Loses Attached Deposit to Attacker-Controlled Sender - (`runtime/runtime/src/lib.rs`)

### Summary

An unprivileged user (Alice) can craft a `DelegateAction` (meta-transaction) that includes inner actions with a large attached deposit targeting a call that will predictably fail at execution. The relayer pays the deposit upfront. When the inner receipt fails on the receiver's shard, the protocol refunds the deposit to the receipt's `predecessor_id` — which is Alice, not the relayer who paid. The relayer permanently loses the deposit amount while Alice gains it.

### Finding Description

The NEAR meta-transaction protocol (NEP-366) allows a relayer to submit a `DelegateAction` signed by Alice. The relayer pays all gas and any attached token deposits for the inner actions. When the inner receipt is created from the delegate action, it is assigned:

- `predecessor_id = Alice` (the `sender_id` of the `DelegateAction`)
- `signer_id = relayer`

The deposit refund logic in `refund_unspent_gas_and_deposits` sends the deposit refund to `receipt.balance_refund_receiver()`: [1](#0-0) 

`balance_refund_receiver()` resolves to `predecessor_id()` (Alice) unless an explicit `refund_to` override is set: [2](#0-1) 

Gas refunds, by contrast, go to `signer_id` (the relayer): [3](#0-2) 

The protocol documentation explicitly acknowledges this as a financial attack vector: [4](#0-3) 

The deposit refund is constructed via `Receipt::new_balance_refund`, which sets `receiver_id = predecessor_id` (Alice): [5](#0-4) 

### Impact Explanation

The corrupted protocol value is the **balance** of the relayer account and Alice's account. The relayer's NEAR balance is permanently reduced by the deposit amount; Alice's balance is permanently increased by the same amount. This is a direct, irreversible token transfer from relayer to attacker with no legitimate basis. The deposit amount is bounded only by the relayer's balance, making the attack scalable.

### Likelihood Explanation

Any unprivileged user can:
1. Create a `DelegateAction` with a large `FunctionCall` deposit targeting a non-existent method on any contract (guaranteed to fail)
2. Present it to a relayer (which is the relayer's intended purpose — accepting signed delegate actions)
3. The relayer submits it, paying the deposit
4. The inner receipt fails on the receiver's shard
5. Alice receives the deposit refund

The attack requires no special privileges. It is repeatable, requires only a valid NEAR account and access key, and can be executed through the public RPC. The only friction is convincing a relayer to accept the meta-transaction, but relayers are designed to accept signed delegate actions from users.

### Recommendation

The deposit refund for inner receipts spawned by a `DelegateAction` should be directed to the `signer_id` (the relayer) rather than the `predecessor_id` (Alice). This can be achieved by setting the `refund_to` field on the inner `ActionReceiptV2` to the relayer's account ID when the receipt is created from a delegate action. The `ActionReceiptV2.refund_to` field already exists for exactly this purpose: [6](#0-5) 

Alternatively, relayer implementations must validate that any attached deposit in a `DelegateAction` targets a call that is guaranteed to succeed, or must not accept delegate actions with attached deposits at all.

### Proof of Concept

1. Alice creates a `DelegateAction`:
   - `sender_id = alice`
   - `receiver_id = some_contract`
   - `actions = [FunctionCall { method_name: "nonexistent", deposit: 1_000_000 NEAR, gas: 300 TGas }]`
   - Signs with her access key

2. Alice presents the `SignedDelegateAction` to a relayer.

3. Relayer wraps it in a transaction (`signer = relayer`) and submits. The relayer's account is charged `1_000_000 NEAR` deposit + gas costs.

4. On Alice's shard, the delegate action is validated and the inner receipt is created with `predecessor_id = alice`, `signer_id = relayer`.

5. On `some_contract`'s shard, the function call fails (`MethodNotFound`).

6. `refund_unspent_gas_and_deposits` fires:
   - `deposit_refund = 1_000_000 NEAR` → sent to `receipt.balance_refund_receiver()` = `alice`
   - Gas refund → sent to `signer_id` = `relayer`

7. Alice's balance increases by `1_000_000 NEAR`. Relayer's balance decreases by `1_000_000 NEAR` (net of gas refund).

The attack is documented in the codebase itself: [7](#0-6)

```markdown
## Balance refunds in meta transactions

Unlike gas refunds, the protocol sends balance refunds to the predecessor
(a.k.a. sender) of the receipt. This makes sense, as we deposit the attached
balance to the receiver, who has to explicitly reattach a new balance to new
receipts they might spawn.

In the world of meta transactions, this assumption is also challenged. If an
inner action requires an attached balance (for example a transfer action) then
this balance is taken from the relayer.

The relayer can see what the cost will be before submitting the meta transaction
and agrees to pay for it, so nothing wrong so far. But what if the transaction
fails execution on Bob's shard? At this point, the predecessor is `Alice` and
therefore she receives the token balance refunded, not the relayer. This is
something relayer implementations must be aware of since there is a financial
incentive for Alice to submit meta transactions that have high balances attached
but will fail on Bob's shard.
```
