### Title
`accept_requests` Flag Not Reset on Resharing Cancellation, Breaking TEE-Gated Request-Lifecycle Invariant — (File: `crates/contract/src/lib.rs`)

---

### Summary

When resharing is cancelled via `vote_cancel_resharing`, the contract correctly reverts `protocol_state` to the previous `RunningContractState`, but it does **not** reset the top-level `accept_requests` flag. This flag was last written by `verify_tee` at the moment resharing was initiated. After cancellation the flag retains that stale value, leaving the contract in an inconsistent state: it is in `Running` mode but either silently refuses all new signature requests (if the flag was `false`) or silently accepts them from a participant set that still contains nodes whose TEE attestations were the reason resharing was triggered in the first place (if the flag was `true`).

---

### Finding Description

`MpcContract` holds two independent pieces of state that together govern whether signature requests are accepted:

```
protocol_state: ProtocolContractState   // Running / Resharing / Initializing
accept_requests: bool                   // TEE-derived gate on new sign() calls
``` [1](#0-0) 

`verify_tee` is the only writer of `accept_requests` during normal operation. When it detects a partial TEE failure but the surviving participant count still satisfies the threshold relation, it sets `accept_requests = true` and immediately transitions the contract into `Resharing` to evict the invalid nodes: [2](#0-1) 

`vote_cancel_resharing` reverts `protocol_state` to the snapshot stored in `ResharingContractState::previous_running_state`, but it touches **only** `self.protocol_state`: [3](#0-2) 

The `accept_requests` field is never written inside `vote_cancel_resharing` or inside the inner `ResharingContractState::vote_cancel_resharing` helper: [4](#0-3) 

After cancellation the contract is back in `Running` with the original participant set — which still contains the nodes whose expired/invalid attestations caused `verify_tee` to initiate resharing — yet `accept_requests` is `true`, so the contract continues to accept new `sign()` calls as if the TEE state were clean.

The symmetric failure also exists: if `verify_tee` set `accept_requests = false` (threshold relation would break) and participants subsequently voted for resharing via `vote_new_parameters` (which never touches `accept_requests`), a cancellation leaves the contract in `Running` with `accept_requests = false`, permanently blocking all new signature requests until `verify_tee` is called again. [5](#0-4) 

---

### Impact Explanation

**Primary path (accept_requests = true after cancellation):** The contract accepts new `sign()` / `request_app_private_key` / `request_verify_foreign_tx` calls from a participant set that still includes nodes whose TEE attestations have expired or been revoked. The TEE-gated safety invariant — "do not issue threshold signatures when the attested participant set is invalid" — is silently violated. Every signing round that completes while the stale flag persists involves at least one node that the contract itself judged unfit to participate.

**Secondary path (accept_requests = false after cancellation):** The contract is in `Running` state with a fully valid participant set but refuses all new signature requests, breaking the request lifecycle. Funds or cross-chain operations that depend on timely MPC signatures are effectively frozen until an operator notices and calls `verify_tee` again.

Both paths match the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The trigger sequence is entirely within normal governance operations:

1. A participant (any single node) calls `verify_tee` — unprivileged, callable by anyone with a valid attestation.
2. `verify_tee` finds ≥1 expired attestation but enough survivors to keep the threshold relation intact, sets `accept_requests = true`, and transitions to `Resharing`.
3. Threshold participants from the previous running state call `vote_cancel_resharing` — a legitimate governance action (e.g., the proposed new set was wrong, or the evicted node renewed its attestation before resharing completed).
4. The contract reverts to `Running` with `accept_requests = true` and the original (partially invalid) participant set.

Step 3 requires threshold votes, but those votes are a designed governance capability, not adversarial collusion. The bug manifests as a side-effect of correct governance behaviour. In a live network where attestations expire on a rolling basis and resharing is a routine operation, this sequence is plausible without any malicious actor.

---

### Recommendation

In `vote_cancel_resharing` (lib.rs), after updating `self.protocol_state`, reset `accept_requests` to `true`. The previous running state is being restored, which means the contract is back in a state where it was previously accepting requests; the TEE gate should be re-evaluated by the next `verify_tee` call rather than left at a stale value:

```rust
if let Some(new_state) = self.protocol_state.vote_cancel_resharing()? {
    self.protocol_state = new_state;
    self.accept_requests = true; // reset: verify_tee will re-evaluate TEE status
}
```

Alternatively, store the value of `accept_requests` at the time resharing begins (inside `ResharingContractState::previous_running_state` or as a separate field) and restore it on cancellation, mirroring how `protocol_state` itself is restored.

    pub fn vote_cancel_resharing(&mut self) -> Result<Option<RunningContractState>, Error> {
        let previous_running_participants = self.previous_running_state.parameters.participants();
        let authenticated_candidate = AuthenticatedAccountId::new(previous_running_participants)?;
        self.cancellation_requests.insert(authenticated_candidate);

        let cancellation_votes_count = self.cancellation_requests.len() as u64;
        let previous_running_threshold = self.previous_running_state.parameters.threshold();

        let threshold_cancellation_votes_reached: bool =
            cancellation_votes_count >= previous_running_threshold.value();

        let running_state = if threshold_cancellation_votes_reached {
            let mut previous_running_state = self.previous_running_state.clone();
            let prospective_epoch_id = self.prospective_epoch_id();
            previous_running_state.previously_cancelled_resharing_epoch_id =
                Some(prospective_epoch_id);

            Some(previous_running_state)
        } else {
            None
        };

        Ok(running_state)
    }
