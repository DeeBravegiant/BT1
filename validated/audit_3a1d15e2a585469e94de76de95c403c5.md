Audit Report

## Title
Stale `rsETHPrice` vs. Fresh `getAssetPrice` Mismatch in Deposit Minting Allows Over-Minting of rsETH at Existing Holders' Expense — (`contracts/LRTDepositPool.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint` divides a live, per-transaction asset price by a stored, lazily-updated `rsETHPrice` state variable. Because `rsETHPrice` is only written inside `_updateRsETHPrice()` — which is never called atomically with a deposit — any period of rising LST prices creates a persistent window where depositors receive more rsETH than their deposit is worth at the true current exchange rate, directly diluting the proportional claim of all existing rsETH holders.

## Finding Description

**Root cause — asymmetric price freshness in `getRsETHAmountToMint`:**

`LRTDepositPool.getRsETHAmountToMint` (line 520) computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.getAssetPrice(asset)` is a pure view call that delegates to an external `IPriceFetcher` and returns the current market price at the moment of the transaction: [2](#0-1) 

`lrtOracle.rsETHPrice()` reads the public state variable `rsETHPrice`, which is only written at the end of `_updateRsETHPrice()`: [3](#0-2) [4](#0-3) 

`_updateRsETHPrice()` is only reachable via `updateRSETHPrice()` (public, permissionless) or `updateRSETHPriceAsManager()` (manager-only). Neither is called inside `depositAsset()` or `depositETH()`: [5](#0-4) [6](#0-5) [7](#0-6) 

**The mismatch:**

The true rsETH price is `totalETHInProtocol / rsethSupply`, where `totalETHInProtocol` is itself computed using `getAssetPrice()` for every supported asset (see `_getTotalEthInProtocol()`). When any supported LST's price rises after the last oracle update, the stored `rsETHPrice` is lower than the true current price, while the fresh `getAssetPrice(asset)` in the numerator reflects the higher current price. The minting formula therefore becomes:

```
rsethAmountToMint = (amount × freshHigherAssetPrice) / staleOldLowerRsETHPrice
                  > (amount × freshHigherAssetPrice) / trueCurrentRsETHPrice
```

The depositor receives more rsETH than their deposit is worth at the true current exchange rate.

**Existing guards are insufficient:**

The `minRSETHAmountExpected` slippage check in `_beforeDeposit` only protects the depositor from receiving too little; it provides no protection to existing holders against dilution: [8](#0-7) 

The `pricePercentageLimit` guard in `_updateRsETHPrice` limits the magnitude of a single oracle update but does not prevent the stale-price window from existing between updates.

## Impact Explanation

**High — Theft of unclaimed yield.**

Every rsETH minted in excess of the true exchange rate dilutes the proportional claim of all existing holders. The value extracted is the accrued appreciation of the underlying LSTs (stETH, ETHx, etc.) since the last oracle update — i.e., yield that has already accrued to existing holders but has not yet been reflected in the stored `rsETHPrice`. In a rising-price environment — the normal condition for yield-bearing liquid staking tokens — this mismatch is continuously present between oracle updates. A large deposit timed to coincide with a period of stale oracle data extracts real ETH-denominated value from all current rsETH holders.

## Likelihood Explanation

**Medium-High.** No special privileges, flash loans, or governance access are required. Any ordinary depositor calling `depositAsset()` or `depositETH()` benefits from the mismatch whenever asset prices have risen since the last oracle update. LST prices (stETH, ETHx) accrue value continuously with each Ethereum epoch, so the stale-price window is always open to some degree. The attacker only needs to observe that `rsETHPrice` is stale relative to current LST prices — both values are publicly readable on-chain — and then call `depositAsset()` before anyone calls `updateRSETHPrice()`.

## Recommendation

Call `lrtOracle.updateRSETHPrice()` at the beginning of `depositAsset()` and `depositETH()` (before `_beforeDeposit` is invoked), so that both the asset price and the rsETH price are computed from the same on-chain state. Alternatively, replace the stored `rsETHPrice` read in `getRsETHAmountToMint` with a live, view-only computation: `_getTotalEthInProtocol() / rsethSupply`, ensuring both sides of the division use prices from the same block.

## Proof of Concept

**Setup:**
- Protocol holds 100 stETH (each worth 1.00 ETH) and 100 ETHx (each worth 1.00 ETH).
- Total TVL = 200 ETH, rsETH supply = 200, stored `rsETHPrice` = 1.00 ETH.

**Price movement (no oracle update yet):**
- stETH appreciates to 1.05 ETH per token (normal LST yield accrual over several days).
- True rsETH price = (100 × 1.05 + 100 × 1.00) / 200 = 205 / 200 = **1.025 ETH**.
- Stored `rsETHPrice` = **1.00 ETH** (stale).

**Attacker deposits 10 stETH without calling `updateRSETHPrice()` first:**

```
rsethAmountToMint = (10 × 1.05e18) / 1.00e18 = 10.5 rsETH   ← actual minted
true entitlement  = (10 × 1.05e18) / 1.025e18 ≈ 10.244 rsETH
over-minted       ≈ 0.256 rsETH  (≈ 2.5% excess)
```

The 0.256 rsETH excess represents ~0.262 ETH of value extracted from existing holders. At scale (e.g., a 1,000 stETH deposit), the over-minting is ~25.6 rsETH ≈ 26.2 ETH of dilution imposed on all current rsETH holders.

**Foundry fork test plan:**
1. Fork mainnet at a block where `rsETHPrice` is known to be stale (i.e., LST prices have risen since last `updateRSETHPrice` call).
2. Record `lrtOracle.rsETHPrice()` and compute the true price via `_getTotalEthInProtocol() / rsethSupply`.
3. Call `depositAsset(stETH, 1000e18, 0, "")` as an unprivileged address.
4. Assert that the minted rsETH amount exceeds `1000e18 * getAssetPrice(stETH) / trueRsETHPrice`.
5. Assert that existing holders' redemption value per rsETH has decreased after the deposit.
