### Title
`ValidateTransactionProvider` Does Not Enforce `max_l1_handler_txs_per_block` Cap, Allowing a Malicious Proposer to Exceed the L1 Handler Transaction Limit — (`File: crates/apollo_batcher/src/transaction_provider.rs`)

---

### Summary

`ProposeTransactionProvider::get_txs()` enforces `max_l1_handler_txs_per_block` during block building. `ValidateTransactionProvider::get_txs()` — the symmetric path used by every validator node — performs no such count check. A malicious proposer can therefore stream more than `max_l1_handler_txs_per_block` L1 handler transactions into a proposal, and every honest validator will accept and execute them all, committing a block that violates the protocol's intended L1 handler rate limit.

---

### Finding Description

`ProposeTransactionProvider` carries two fields that together enforce the cap:

```rust
pub max_l1_handler_txs_per_block: usize,
n_l1handler_txs_so_far: usize,
```

Inside `get_txs()` the proposer computes:

```rust
let n_l1handler_txs_to_get =
    min(self.max_l1_handler_txs_per_block - self.n_l1handler_txs_so_far, n_txs);
```

and switches to the mempool phase once `n_l1handler_txs_so_far == max_l1_handler_txs_per_block`. [1](#0-0) 

`ValidateTransactionProvider` has neither field nor counter:

```rust
pub struct ValidateTransactionProvider {
    tx_receiver: tokio::sync::mpsc::Receiver<InternalConsensusTransaction>,
    final_n_executed_txs_receiver: tokio::sync::oneshot::Receiver<usize>,
    l1_events_provider_client: SharedL1EventsProviderClient,
    height: BlockNumber,
}
```

Its `get_txs()` iterates over every received transaction and, for each `L1Handler` variant, calls `l1_events_provider_client.validate()` to check cancellation/consumption state — but never counts how many L1 handler transactions have been accepted in the current block. [2](#0-1) 

The production limit is 200 L1 handler transactions per block proposal (`max_l1_handler_txs_per_block_proposal = 200` in the deployment config), while the bouncer's total `n_txs` cap is 500. [3](#0-2) 

---

### Impact Explanation

A malicious proposer can include up to the bouncer's `n_txs` limit (500) in L1 handler transactions instead of the intended 200. Every validator's `ValidateTransactionProvider` will accept all of them because the only per-transaction check is the L1 state check (`validate(tx_hash, height)`), not a count check. The block is then committed with more L1 handler transactions than the protocol intends, violating the rate-limiting invariant for L1→L2 message consumption and potentially draining the L1 message queue at 2.5× the intended rate per block.

---

### Likelihood Explanation

Exploitation requires the attacker to be the elected proposer for a given round — a legitimate validator node. Once elected, the attack is trivially reproducible every time that node proposes: it simply streams more than `max_l1_handler_txs_per_block` L1 handler transactions in the `ProposalPart::Transactions` batches. No special tooling is needed beyond a modified batcher that skips the `ProposeTransactionProvider` cap.

---

### Recommendation

Add a counter and cap to `ValidateTransactionProvider` mirroring the proposer-side logic:

```rust
pub struct ValidateTransactionProvider {
    tx_receiver: ...,
    final_n_executed_txs_receiver: ...,
    l1_events_provider_client: SharedL1EventsProviderClient,
    height: BlockNumber,
    max_l1_handler_txs_per_block: usize,   // add
    n_l1handler_txs_so_far: usize,         // add
}
```

Inside `get_txs()`, after the existing `L1ValidationStatus::Invalid` check, increment the counter and return an error (or treat the block as invalid) if `n_l1handler_txs_so_far` would exceed `max_l1_handler_txs_per_block`. The `max_l1_handler_txs_per_block` value must be passed in from `BatcherStaticConfig::max_l1_handler_txs_per_block_proposal` at construction time, the same source used by `ProposeTransactionProvider`. [4](#0-3) 

---

### Proof of Concept

1. Attacker controls a validator node that is elected proposer for height H.
2. Attacker's batcher builds a block with `max_l1_handler_txs_per_block + K` (K > 0) L1 handler transactions, all individually valid (pending, not cancelled, not consumed on L1), and total bouncer weight within limits.
3. The proposal is streamed to validators as `ProposalPart::Transactions` batches.
4. Each honest validator's `ValidateTransactionProvider::get_txs()` calls `l1_events_provider_client.validate(tx_hash, height)` for each L1 handler tx — all return `ValidationStatus::Validated` — and returns the full batch without counting. [5](#0-4) 
5. The block builder on each validator executes all `max_l1_handler_txs_per_block + K` L1 handler transactions.
6. `finish_proposal` is called with `final_n_executed_txs` covering all of them; `BlockExecutionArtifacts` is produced and the block is committed with more L1 handler transactions than the protocol limit. [6](#0-5)

impl TransactionProvider for ValidateTransactionProvider {
    async fn get_txs(&mut self, n_txs: usize) -> TransactionProviderResult<NextTxs> {
        assert!(n_txs > 0, "The number of transactions requested must be greater than zero.");

        if self.tx_receiver.is_empty() {
            // Return immediately to avoid blocking the caller.
            return Ok(vec![]);
        }

        let mut buffer = Vec::with_capacity(n_txs);
        self.tx_receiver.recv_many(&mut buffer, n_txs).await;

        for tx in &buffer {
            if let InternalConsensusTransaction::L1Handler(tx) = tx {
                let l1_validation_status = self
                    .l1_events_provider_client
                    .validate(tx.tx_hash, self.height)
                    .await
                    .inspect_err(|err| {
                        warn!(
                            "L1 provider error while validating L1 handler transaction: {:?}",
                            err
                        );
                        BATCHER_L1_EVENTS_PROVIDER_ERRORS.increment(1);
                    })
                    .unwrap_or(L1ValidationStatus::Invalid(
                        L1InvalidValidationStatus::L1EventsProviderError,
                    ));
                if let L1ValidationStatus::Invalid(validation_status) = l1_validation_status {
                    return Err(TransactionProviderError::L1HandlerTransactionValidationFailed {
                        tx_hash: tx.tx_hash,
                        validation_status,
                    });
                }
                continue;
            }
        }
        Ok(buffer)
    }
