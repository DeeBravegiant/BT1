### Title
Direct Deposits Excluded from Value Conservation Check Enables ADA Creation from Nothing тАФ (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in transaction bodies that adds funds directly to registered account balances. However, neither the consumed-value nor the produced-value calculation used by `validateValueNotConservedUTxO` accounts for `directDeposits`. An unprivileged transaction sender can include arbitrarily large direct deposits in a Dijkstra transaction without providing corresponding UTxO inputs to fund them, causing the ledger to accept an invalid state transition that creates ADA from nothing.

---

### Finding Description

The Dijkstra UTXO transition rule enforces value conservation at line 381:

```haskell
{- consumed pp utxoтВА txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [1](#0-0) 

`validateValueNotConservedUTxO` calls `consumed` and `produced` from the `EraUTxO` type class. For `DijkstraEra`, these resolve to `conwayConsumed` (via `consumed = conwayConsumed`) and `getProducedDijkstraValue`: [2](#0-1) 

**Consumed side** тАФ `getConsumedDijkstraValue` aggregates values by calling `getConsumedMaryValue` for the top-level body and each sub-transaction body: [3](#0-2) 

`getConsumedMaryValue` is a pre-Dijkstra function that covers UTxO inputs, withdrawals, and deposit refunds. It has no knowledge of `directDeposits`.

**Produced side** тАФ `dijkstraProducedValue` calls `conwayProducedValue` for the top-level body and `dijkstraSubTxProducedValue` for each sub-transaction: [4](#0-3) 

`dijkstraSubTxProducedValue` explicitly enumerates what it includes тАФ outputs, deposits, treasury donation, and burned multi-assets тАФ with no mention of `directDeposits`: [5](#0-4) 

**Application of direct deposits** тАФ In the ENTITIES rule, direct deposits are applied to account balances after withdrawals and certificate processing, with the only guard being a check that target accounts exist: [6](#0-5) 

There is no check that the direct deposit amounts are funded by the transaction's UTxO inputs.

**Accounting identity broken.** For a transaction with inputs `X`, outputs `Y`, fee `F`, deposits `D`, and direct deposits `DD`:

| Side | Formula | Value |
|------|---------|-------|
| consumed | `X + W + R` | `X` (assuming W=R=0) |
| produced | `Y + F + D` | `X` (if `X = Y + F + D`) |
| check | consumed == produced | тЬУ passes |
| actual accounts delta | `+DD` | **value created** |

The ledger accepts the transaction while account balances increase by `DD` with no corresponding UTxO deduction.

---

### Impact Explanation

**Critical тАФ Direct creation of ADA through an invalid ledger state transition.**

An attacker can craft a Dijkstra transaction that passes all validation checks while causing the ledger to credit an arbitrary amount of ADA to one or more registered accounts. Because the `directDeposits` map accepts any `Coin` value per `AccountAddress` with no upper bound, the attacker can mint unbounded ADA in a single transaction. This directly violates the preservation-of-value invariant that is the foundational security property of the Cardano ledger. [7](#0-6) 

---

### Likelihood Explanation

The Dijkstra era is currently experimental and not deployed on mainnet. However, the vulnerability is present in the production codebase and is trivially exploitable by any transaction sender тАФ no privileged access, governance majority, or key compromise is required. The entry path is a single well-formed Dijkstra transaction. Once the era is activated, exploitation would be immediate and repeatable.

---

### Recommendation

Add the total direct-deposit amount to the **produced** side of the value conservation equation. Concretely:

1. In `dijkstraSubTxProducedValue` (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`), add `inject (sumDirectDeposits (txBody ^. directDepositsTxBodyL))` to the produced value.
2. In `dijkstraProducedValue` (top-level tx), ensure `conwayProducedValue` is extended or wrapped to include the top-level `directDeposits`.
3. Add a corresponding property-based test asserting that preservation of value holds

  forall era l.
  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody

    txBodyConsumedValue
