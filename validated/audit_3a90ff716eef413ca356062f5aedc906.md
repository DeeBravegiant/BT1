Audit Report

## Title
ETH Push-Transfer to Reverting Recipient Permanently Freezes Funds After rsETH Burn - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.unlockQueue` irreversibly burns rsETH and moves ETH into the withdrawal manager in a single committed transaction. A subsequent `completeWithdrawal` call pushes ETH to the user via a low-level `call`; if the recipient contract reverts on receipt, the push fails and the entire completion transaction reverts. Because the burn is already committed and no on-chain recovery path exists, the ETH is permanently stranded in the contract.

## Finding Description
The withdrawal lifecycle is split across two independent, non-atomic transactions.

**Step 1 — `unlockQueue` (operator-only, lines 305–307):**
rsETH held by the contract is burned and the corresponding ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. Both state changes are committed and irreversible once the transaction is mined. [1](#0-0) 

**Step 2 — `completeWithdrawal` / `completeWithdrawalForUser` (lines 183–204):**
`_processWithdrawalCompletion` decrements `unlockedWithdrawalsCount`, deletes the request record, then calls `_transferAsset`, which pushes ETH via a low-level `call`: [2](#0-1) 

If `to` is a contract whose `receive()` reverts (or is absent), `sent` is `false` and `EthTransferFailed` is thrown. The entire Step 2 transaction reverts — including the `popFront`, the `delete`, and the `unlockedWithdrawalsCount--` — leaving the request intact in the queue. Step 1 cannot be undone.

**No recovery path exists:**

- `sweepRemainingAssets` is permanently blocked because `hasUnlockedWithdrawals(asset)` remains `true` (the stuck request keeps `unlockedWithdrawalsCount[asset] > 0`): [3](#0-2) 

- `completeWithdrawalForUser` (operator path, line 202) passes the same `user` address to `_processWithdrawalCompletion`, so it hits the identical revert: [4](#0-3) 

- There is no `cancelWithdrawal`, no alternative-recipient parameter, and no admin escape hatch for stranded ETH. The developer comment on `completeWithdrawalForUser` ("potential gas grief scenarios are non-impactful for ETH") underestimates the severity — this is not gas grief but a permanent fund freeze. [5](#0-4) 

## Impact Explanation
**Critical — Permanent freezing of funds.** After `unlockQueue` burns the user's rsETH, the ETH allocated to that user is held in `LRTWithdrawalManager`. If the user's address cannot accept ETH, neither the user nor any operator can ever complete the withdrawal. The ETH is locked in the contract indefinitely with no on-chain recovery mechanism. The user loses their entire principal.

## Likelihood Explanation
**Medium.** Any rsETH holder can trigger this path — no privileged access is required. Smart-contract wallets, multisigs, and DeFi protocol treasuries that interact with LRT-rsETH may lack a `receive()` function or may have one that conditionally reverts (e.g., access-controlled, paused, or upgraded away). A malicious actor can also deliberately deploy such a contract, initiate a withdrawal, wait for the operator to burn their rsETH, and then use the permanently stuck ETH as leverage. The attacker bears no cost beyond gas and the rsETH they chose to withdraw.

## Recommendation
Replace the push-payment pattern with a pull-payment (claimable balance) pattern for ETH withdrawals:

1. In `_processWithdrawalCompletion`, instead of calling `_transferAsset` directly, credit `claimableETH[user] += request.expectedAssetAmount`.
2. Add a separate `claimETH(address recipient)` function that lets the user pull their balance to any address they specify.

This decouples the accounting step (which must succeed atomically) from the ETH delivery step (which is user-controlled), eliminating the ability of a reverting `receive()` to freeze funds.

## Proof of Concept
1. Attacker deploys `MaliciousRecipient` — a contract with `receive() external payable { revert(); }`.
2. `MaliciousRecipient` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`; a `WithdrawalRequest` is recorded.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. At line 305, rsETH is burned. At line 307, ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager`. The request is now "unlocked".
4. `MaliciousRecipient` calls `completeWithdrawal(ETH_TOKEN, "")`. `_processWithdrawalCompletion` reaches line 878: `payable(MaliciousRecipient).call{value: amount}("")` → `receive()` reverts → `sent = false` → `revert EthTransferFailed()`. Entire tx reverts; request remains in queue.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, MaliciousRecipient, "")`. Same revert.
6. Operator attempts `sweepRemainingAssets(ETH_TOKEN)`. Reverts: `hasUnlockedWithdrawals(ETH_TOKEN)` is `true`.
7. **Result:** rsETH is permanently burned; ETH is permanently locked in `LRTWithdrawalManager`. No on-chain path exists to recover it.
