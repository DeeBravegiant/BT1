### Title
`convert_thin_state_diff` misclassifies intra-simulation `replace_class` as new deployment in `starknet_simulateTransactions(block_id=Pending)` — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
`is_deployed` only queries committed storage. When a contract is deployed by an earlier transaction within the same `simulateTransactions` batch, a later transaction's `replace_class` on that contract is incorrectly reported as a new deployment in the `induced_state_diff` of the simulation response.

### Finding Description
`convert_thin_state_diff` iterates over `thin_state_diff.deployed_contracts` and calls `is_deployed(prev_block_number, address)` to decide whether each entry is a new deployment or a class replacement. `is_deployed` always constructs a `BlockId::HashOrNumber(...)` and queries only committed storage at block N. It has no visibility into the accumulated simulation state built up by earlier transactions in the same batch. Therefore, if Tx1 deploys contract A and Tx2 calls `replace_class(A, new_class)`, the check for Tx2 returns `false` (A is absent from committed storage), and A is placed in `deployed_contracts` instead of `replaced_classes` in Tx2's `induced_state_diff`. [7](#0-6) 

### Impact Explanation
The `SimulatedTransaction.transaction_trace.induced_state_diff` returned to the caller contains a wrong split: `deployed_contracts` is non-empty and `replaced_classes` is empty for Tx2, when the correct answer is the reverse. Clients using this output to reason about transaction effects (e.g., tooling, block explorers, wallets) receive an incorrect authoritative-looking value from the node's simulation RPC.

### Likelihood Explanation
Any public RPC caller can trigger this with a two-transaction `simulateTransactions(Pending, [Deploy(A), Invoke(replace_class(A, ...))])` call. No special privileges are required.

### Recommendation
`convert_thin_state_diff` must account for contracts deployed by earlier transactions in the same simulation batch. One approach: pass the accumulated `deployed_contracts` addresses from all prior simulation outputs as an additional "pending deployed" set, and treat any address present in that set as a replacement rather than a new deployment when classifying subsequent transactions' state diffs.

### Proof of Concept
```
1. Call starknet_simulateTransactions(
     block_id = "pending",
     transactions = [
       DeployAccount(address=A, class=X),          // Tx1
       Invoke(sender=A, calldata=replace_class(Y))  // Tx2: A calls replace_class
     ]
   )
2. Inspect result[1].transaction_trace.induced_state_diff:
   - Actual:   deployed_contracts=[{A, Y}], replaced_classes=[]
   - Expected: deployed_contracts=[],       replaced_classes=[{A, Y}]
3. The invariant is violated: a contract deployed earlier in the same
   pending simulation is not recognized as already-deployed when
   classifying the class replacement in the induced state diff.
```
