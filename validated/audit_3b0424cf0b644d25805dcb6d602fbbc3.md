### Title
Legacy CKD `AppPublicKey` Variant Accepts Arbitrary `CKDResponse` Without On-Chain Integrity Check — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd()` function in the MPC contract performs a cryptographic pairing check (`ckd_output_check`) on the `CKDResponse` only when the request used the `AppPublicKeyPV` (publicly verifiable) variant. For the legacy `AppPublicKey` variant, **no integrity check is performed**. Any single attested participant can call `respond_ckd` with arbitrary `big_y` / `big_c` values, and the contract will accept and deliver the tampered output to the user. This is the direct analog of the Hinkal `encryptedOutputs` vulnerability: user-visible encrypted output data is not integrity-checked at the protocol boundary.

---

### Finding Description

In `respond_ckd`, the contract branches on the `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(C, G2) = e(Y, app_pk2) · e(H(pk, app_id), pk)`, ensuring the output is a valid encryption of the correct BLS signature under the MPC master key. [2](#0-1) 

For `AppPublicKey` (legacy), the arm is empty — the contract unconditionally proceeds to `resolve_yields_for`, delivering whatever `(big_y, big_c)` the caller supplied. [3](#0-2) 

The CKD protocol itself (`protocol.rs`) also performs no verification of the aggregated output in the non-PV path — the coordinator simply sums shares and returns the result with no pairing check: [4](#0-3) 

Contrast this with the PV coordinator (`protocol_pv.rs`), which calls `aggregated_output_check` before returning: [5](#0-4) 

The test suite confirms the asymmetry: `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` passes arbitrary byte arrays `[1u8; 48]` / `[2u8; 48]` as `big_y` / `big_c` for the legacy variant and the contract accepts them without complaint. [6](#0-5) 

---

### Impact Explanation

A malicious attested participant (a single Byzantine node, strictly below the signing threshold) can:

1. Observe a pending legacy-variant CKD request in the contract state.
2. Race to call `respond_ckd(request, CKDResponse { big_y: arbitrary, big_c: arbitrary })` before the honest coordinator does.
3. The contract accepts the call, resolves the yield, and returns the tampered `(big_y, big_c)` to the requesting user/contract.

The user computes `sig = big_c − a · big_y`, which is a random G1 point unrelated to the MPC master secret. If the user (or a TEE application) does not perform the client-side BLS verification step (`e(H(pk, app_id), pk) = e(sig, G2)`), they derive a wrong wrapping key. The primary use case documented for CKD is TEE key backup: the Archive Signer encrypts its HOT wallet private keys with the CKD-derived wrapping key. [7](#0-6) 

If the TEE app encrypts its private keys with a wrong wrapping key and stores the backup, the backup is permanently unrecoverable (the correct CKD output is deterministic and will differ from the tampered one on any future re-derivation). This constitutes **permanent freezing/loss of funds** controlled by the TEE application. Even if the user detects the tampered output via client-side verification, their request slot is consumed and the deposit is lost, requiring a re-submission.

---

### Likelihood Explanation

- **Single attested participant** is sufficient — no threshold collusion required.
- `respond_ckd` is a public call-method gated only on `assert_caller_is_attested_participant_and_protocol_active()`. Any one of the `n` attested participants can submit the tampered response.
- The legacy `AppPublicKey` variant is the default and most widely used path (the `AppPublicKeyPV` variant is newer and opt-in).
- The attacker wins a simple race: submit `respond_ckd` with arbitrary values before the honest coordinator's transaction lands.

---

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` variant in `respond_ckd`. Since the legacy variant provides only a G1 point (`pk1`) and no G2 counterpart (`pk2`), the existing `ckd_output_check` cannot be used directly. Options:

1. **Deprecate the legacy variant** and require all new requests to use `AppPublicKeyPV`, which already has full on-chain verification.
2. **Add a separate integrity check** for the legacy variant: verify that `e(big_c − big_y · a, G2) = e(H(pk, app_id), pk)` — but this requires the app's private scalar `a` to be known on-chain, which defeats confidentiality. Therefore option 1 is strongly preferred.
3. As an interim measure, **document clearly** that callers of the legacy variant MUST perform client-side BLS verification before trusting or using the derived key, and add an explicit warning in the contract ABI docs.

---

### Proof of Concept

```
1. Alice submits request_app_private_key with AppPublicKey variant (legacy).
   → Contract stores pending CKD request keyed on (app_id, domain_id, ...).

2. Malicious attested participant Bob calls:
   respond_ckd(
     request = <Alice's CKDRequest>,
     response = CKDResponse {
       big_y: Bls12381G1PublicKey([0xde; 48]),  // arbitrary garbage
       big_c: Bls12381G1PublicKey([0xad; 48]),  // arbitrary garbage
     }
   )

3. Contract executes respond_ckd:
   - match AppPublicKey(_) => {}   ← no check, falls through
   - resolve_yields_for(...) delivers the tampered response to Alice's yield.

4. Alice's contract callback receives CKDResponse { big_y: 0xde..., big_c: 0xad... }.
   Alice computes: sig = big_c − a · big_y  →  random G1 point.
   Alice derives wrapping_key = HKDF(sig)   →  wrong key.
   Alice encrypts HOT private keys with wrong wrapping_key and stores backup.

5. On recovery, Alice re-derives CKD correctly (honest nodes), gets a different key,
   and cannot decrypt the backup → permanent loss of the backed-up private keys.
```

The existing test at `crates/contract/src/lib.rs:3403` already demonstrates that the contract accepts `[1u8; 48]` / `[2u8; 48]` (invalid BLS points) as a valid legacy CKD response with no error. [8](#0-7)

### Recovery via Confidential Key Derivation (CKD)

Encrypted disk persistence protects against restarts, but not against total loss of the CVM instance — for example, if the TEE image is updated (changing RTMR measurements and invalidating the disk encryption key) or all CVM instances are destroyed. Without an additional recovery mechanism, this would be unrecoverable after the HOT MPC network is retired, since keyshares can no longer be re-imported.

To address this, we use [Confidential Key Derivation][ckd] (CKD) via the NEAR MPC network to create a hardware-independent encrypted backup:

1. **Derive a wrapping key.** After key reconstruction, the Archive Signer calls the NEAR MPC network's CKD endpoint with its `app_id`. CKD returns a deterministic, confidential key that only this application can obtain.
2. **Encrypt and store a backup.** The reconstructed HOT private keys are encrypted with the CKD-derived wrapping key. The resulting ciphertext is stored in external storage.
3. **Recover from any new instance.** A new instance re-derives the same CKD key — since CKD is deterministic on `app_id`, not on hardware measurements — and decrypts the backup.

This decouples recovery from any specific CVM instance or disk encryption key, requiring only that the NEAR MPC network remains operational.
