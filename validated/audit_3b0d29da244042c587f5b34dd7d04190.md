Audit Report

## Title
Missing `updateRSETHPrice()` Before Deposit Allows Minting Excess rsETH at Stale Price — (`contracts/LRTDepositPool.sol`)

## Summary
`LRTOracle` stores `rsETHPrice` as a state variable that is only updated when `updateRSETHPrice()` is explicitly called. Neither `depositETH()` nor `depositAsset()` in `LRTDepositPool` calls `updateRSETHPrice()` before computing the rsETH mint amount, so any deposit made after rewards have accrued (but before the price is refreshed) uses a stale, lower price. The depositor receives excess rsETH, diluting existing holders of their earned yield.

## Finding Description
`LRTOracle` maintains `rsETHPrice` as a plain storage variable (`LRTOracle.sol:28`). It is only updated inside `_updateRsETHPrice()`, which is invoked by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()` (`LRTOracle.sol:87-96`). There is no automatic update on deposit.

`LRTDepositPool.depositETH()` (`LRTDepositPool.sol:76-93`) and `depositAsset()` (`LRTDepositPool.sol:99-118`) both call `_beforeDeposit()`, which calls `getRsETHAmountToMint()` (`LRTDepositPool.sol:506-521`). The mint formula is:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` returns the stored state variable — not a freshly computed value.

`FeeReceiver.sendFunds()` (`FeeReceiver.sol:53-58`) is permissionless and transfers accumulated MEV/execution-layer rewards directly to `LRTDepositPool` via `receiveFromRewardReceiver()`. After this call, `getETHDistributionData()` immediately reflects the higher `address(this).balance`, so `totalETHInProtocol` increases. However, `rsETHPrice` remains at its last written value. Any deposit in this window uses the stale, lower price.

Additionally, `_updateRsETHPrice()` contains a `pricePercentageLimit` guard (`LRTOracle.sol:252-266`) that reverts for non-manager callers if the price increase exceeds the configured threshold. If rewards are large, this extends the stale-price window until a manager intervenes, making the attack window larger.

Existing guards (`minRSETHAmountExpected`, deposit limits) do not protect against this: the slippage parameter is set by the depositor themselves and provides no protection to existing holders.

## Impact Explanation
**High — Theft of unclaimed yield.**

When `rsETHPrice` is stale (lower than actual), a depositor of `D` ETH receives `D / P_stale` rsETH instead of the correct `D / P_actual`. Since `P_stale < P_actual`, the depositor receives excess rsETH representing a claim on TVL earned by existing holders as staking/MEV yield. Existing holders' rsETH is worth less ETH than it would have been had the price been updated first. The loss scales with the magnitude of accrued rewards and the deposit size.

## Likelihood Explanation
`FeeReceiver.sendFunds()` is permissionless — any account can call it. `updateRSETHPrice()` has no on-chain enforcement that it be called before deposits. Rewards accrue continuously via staking and MEV. The attack requires no special privilege: any depositor passively benefits from a stale price, and a sophisticated actor can deliberately call `sendFunds()` and immediately deposit in the same block to maximize the dilution. The window is routine and repeatable.

## Recommendation
Call `updateRSETHPrice()` at the start of `depositETH()` and `depositAsset()`, before computing `rsethAmountToMint`:

```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice(); // add
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    emit ETHDeposit(msg.sender, msg.value, rsethAmountToMint, referralId);
}
```

Apply the same fix to `depositAsset()`. This ensures the price reflects all accrued rewards before any new rsETH is minted.

## Proof of Concept
1. Protocol state: 1000 ETH TVL, 1000 rsETH supply, `rsETHPrice = 1.0`.
2. Attacker calls `FeeReceiver.sendFunds()` (permissionless), transferring 100 ETH of MEV rewards to `LRTDepositPool`. TVL is now 1100 ETH. `rsETHPrice` remains `1.0` (stale).
3. Attacker calls `depositETH{value: 100 ETH}(0, "")`. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1e18 = 100e18` rsETH. Attacker receives 100 rsETH. TVL = 1200 ETH, supply = 1100 rsETH.
4. Anyone calls `updateRSETHPrice()`. `previousTVL = 1100 * 1.0 = 1100`. `rewardAmount = 1200 - 1100 = 100`. Protocol fee (10%) = 10 ETH. `newRsETHPrice = (1200 - 10) / 1100 ≈ 1.0818`. Fee rsETH minted to treasury ≈ 9.24 rsETH.
5. Attacker's 100 rsETH is worth `100 × 1.0818 ≈ 108.18 ETH` — deposited 100 ETH and immediately captured ~8.18 ETH of yield belonging to original holders.
6. Original 1000 rsETH holders: their rsETH is worth `1000 × 1.0818 = 1081.8 ETH`. Had the price been updated first, they would have received `1000 × 1.09 = 1090 ETH` — a loss of ~8.2 ETH of unclaimed yield.

**Foundry test plan:** Fork mainnet, set up protocol state with known TVL and rsETH supply, call `FeeReceiver.sendFunds()` with a reward amount, then call `depositETH()` without calling `updateRSETHPrice()` first. Assert that the depositor's rsETH balance multiplied by the post-update price exceeds their deposit amount, and that original holders' rsETH is worth less ETH than it would be if `updateRSETHPrice()` had been called before the deposit.
