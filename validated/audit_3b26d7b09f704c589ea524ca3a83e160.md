Audit Report

## Title
Merkle Leaf Hash Missing Chain ID and Contract Address Enables Cross-Chain Proof Replay - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

## Summary
`KernelMerkleDistributor`, `MerkleDistributor`, and `KernelTop100MerkleDistributor` all construct Merkle leaf nodes without binding them to `block.chainid` or `address(this)`. Because the claimed-status tracking is stored in a per-contract-instance mapping, a valid proof accepted on one chain deployment is equally valid on any other deployment sharing the same Merkle root, allowing a recipient to claim their full allocation on every chain where the root is live.

## Finding Description
In `KernelMerkleDistributor._processClaim()` (line 320), the leaf is:

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
``` [1](#0-0) 

The identical pattern appears in `MerkleDistributor.claim()` (line 120):

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
``` [2](#0-1) 

And in `KernelTop100MerkleDistributor._verifyClaimProof()` (line 293):

```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
``` [3](#0-2) 

The replay-protection check in both `KernelMerkleDistributor` and `MerkleDistributor` is:

```solidity
return userClaims[account].lastClaimedIndex >= index;
``` [4](#0-3) 

This mapping is local to each contract instance. [5](#0-4) 

A claim recorded on Chain A sets `userClaims[account].lastClaimedIndex` only in that deployment's storage. The same `(index, account, cumulativeAmount, merkleProof)` tuple is cryptographically valid against the same root on Chain B, where `userClaims[account].lastClaimedIndex` is still `0`. The `isClaimed` check therefore returns `false` and the claim succeeds again.

For `KernelTop100MerkleDistributor`, the analogous guard is `userClaims[user].amountClaimed`, which is also per-instance storage. [6](#0-5) 

## Impact Explanation
Any recipient of a KERNEL distribution can claim their full allocation on every chain where the same Merkle root is deployed, draining tokens that belong to other claimants. This is direct theft of unclaimed yield.

**Impact: High — Theft of unclaimed yield.**

## Likelihood Explanation
Kelp DAO operates across Ethereum mainnet, Arbitrum, Optimism, Base, and other L2s. A single-snapshot, single-root, multi-chain distribution is a natural operational pattern: generate one Merkle tree from a cross-chain snapshot and deploy the same root on each chain so users can claim on their preferred network. The leaf encoding provides no on-chain signal that this is unsafe, and nothing in the contracts prevents the owner from calling `setMerkleRoot` with the same value on multiple deployments. No private key compromise or governance capture is required — only the routine act of setting the same root on two deployments.

**Likelihood: Medium.**

## Recommendation
Bind every leaf to the specific deployment by including `block.chainid` and `address(this)` in the hash:

```solidity
// KernelMerkleDistributor & MerkleDistributor
bytes32 node = keccak256(
    abi.encodePacked(block.chainid, address(this), index, account, cumulativeAmount)
);

// KernelTop100MerkleDistributor
bytes32 leaf = keccak256(
    abi.encodePacked(block.chainid, address(this), user, amount)
);
```

Apply the fix to all three contracts. Off-chain leaf generation must be updated to match. This ensures a proof generated for one chain and one contract address is cryptographically invalid on any other deployment.

## Proof of Concept

1. Kelp DAO deploys `KernelMerkleDistributor` on Ethereum mainnet (`0xAAA...`) and Arbitrum (`0xBBB...`), calling `setMerkleRoot(root)` with the **same** `root` on both.
2. Alice holds a valid proof: `(index=5, account=Alice, cumulativeAmount=1000e18, proof=[...])`.
3. Alice calls `claim(5, Alice, 1000e18, proof)` on mainnet → succeeds; `userClaims[Alice].lastClaimedIndex = 5` on mainnet only. [7](#0-6) 
4. Alice calls `claim(5, Alice, 1000e18, proof)` on Arbitrum → `userClaims[Alice].lastClaimedIndex` is `0` on Arbitrum; `isClaimed` returns `false`; the proof verifies against the same root; Alice receives another `1000e18` KERNEL. [8](#0-7) 
5. Alice repeats on every additional chain where the root is deployed.

**Foundry fork test plan:** Fork mainnet and Arbitrum in the same test. Deploy `KernelMerkleDistributor` on both forks with identical constructor args and the same `setMerkleRoot` call. Generate a leaf and proof off-chain. Call `claim` on the mainnet fork, assert success. Switch to the Arbitrum fork, call `claim` with the identical arguments, assert it also succeeds and transfers tokens — demonstrating the double-claim.
