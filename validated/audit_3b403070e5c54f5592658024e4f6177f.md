Audit Report

## Title
Dual-Oracle Staleness Divergence Enables Excess rsETH Extraction from Pool Inventory — (`contracts/pools/RSETHPool.sol`)

## Summary

`RSETHPool.viewSwapRsETHAmountAndFee(uint256,address)` prices token deposits using two independent oracles with no synchronization: a Chainlink-backed collateral oracle that updates automatically and an `InterimRSETHOracle` that is updated manually by a `MANAGER_ROLE` holder. Any unprivileged depositor can monitor the natural divergence between these two oracles and deposit collateral during the lag window to receive more `wrsETH` from the pool's inventory than the fair value of the deposited collateral, constituting theft of unclaimed yield from the pool's reserves.

## Finding Description

**Pricing path** in `RSETHPool.viewSwapRsETHAmountAndFee(uint256 amount, address token)`:

```solidity
uint256 rsETHToETHrate  = getRate();                                       // InterimRSETHOracle
uint256 tokenToETHRate  = IOracle(supportedTokenOracle[token]).getRate();  // ChainlinkOracleForRSETHPoolCollateral
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

**Oracle 1 — `ChainlinkOracleForRSETHPoolCollateral.getRate()`** reads `latestRoundData()` and applies only three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0)            revert IncompleteRound();
if (ethPrice <= 0)             revert InvalidPrice();
``` [2](#0-1) 

There is no `block.timestamp - timestamp > maxStaleness` heartbeat check. A Chainlink answer that is hours old (up to the feed's full heartbeat period) passes all three guards and is treated as current. More importantly, Chainlink updates automatically on price deviation, meaning the collateral price can reflect a fresh market move while the rsETH oracle has not yet been updated.

**Oracle 2 — `InterimRSETHOracle.getRate()`** returns a storage variable with zero staleness protection:

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
``` [3](#0-2) 

The rate is updated only when a `MANAGER_ROLE` holder calls `setRate()`. Between updates the stored value is arbitrarily stale. [4](#0-3) 

**No minimum-output guard** exists in `deposit(address,uint256,string)`: [5](#0-4) 

**Exploit flow:**
1. Attacker monitors `InterimRSETHOracle.getRate()` and `ChainlinkOracleForRSETHPoolCollateral.getRate()` off-chain (both are public `view` functions).
2. Chainlink updates the collateral price upward (e.g., wstETH/ETH: 1.10 → 1.15) due to a deviation trigger.
3. `InterimRSETHOracle.rate` still holds the prior value (e.g., 1.08) because the manager has not yet called `setRate()`.
4. Attacker calls `deposit(wstETH, amount, "")`. The pool computes `rsETHAmount = amount * 1.15e18 / 1.08e18 ≈ 1.0648 rsETH per wstETH` instead of the fair `1.15 / 1.10 ≈ 1.0455`.
5. The pool transfers the excess `wrsETH` from its own inventory to the attacker.
6. Attacker repeats until the manager updates the rate.

The divergence window is **structurally guaranteed**: Chainlink updates automatically on price deviation; `InterimRSETHOracle` is updated manually and will always lag. No privileged role, no front-running of other users, and no oracle compromise is required.

## Impact Explanation

The pool holds `wrsETH` as inventory representing accrued protocol yield not yet distributed. Each favorable deposit extracts more `wrsETH` from this inventory than the economic value of the deposited collateral. The surplus is drawn directly from the pool's reserves. This is **theft of unclaimed yield** (High severity). The attack is repeatable across multiple transactions within the same divergence window, and the window recurs naturally after every collateral price move that precedes a manager rate update.

## Likelihood Explanation

- `deposit(address,uint256,string)` is public, gated only by `whenNotPaused` and `onlySupportedToken` — no privileged role required.
- Both oracle values are readable via `view` calls; monitoring is trivial.
- The divergence window is not an edge case — it is the normal operating state between every manual `setRate()` call.
- The attacker acts against the pool's own inventory, not against other users, so no victim cooperation or front-running is needed.
- The attack is repeatable and scalable with deposit size.

## Recommendation

1. **Add a heartbeat/age check to `ChainlinkOracleForRSETHPoolCollateral`**: store a `maxStaleness` parameter and revert if `block.timestamp - updatedAt > maxStaleness`. [2](#0-1) 

2. **Add a staleness timestamp to `InterimRSETHOracle`**: record `block.timestamp` on each `setRate()` call and revert in `getRate()` if the stored rate is older than a configured maximum age. [4](#0-3) 

3. **Add a `minRsETHAmount` parameter to `deposit(address,uint256,string)`** and/or a circuit-breaker that reverts if the cross-oracle ratio deviates beyond a configured band. [5](#0-4) 

## Proof of Concept

**Foundry fork test plan (Arbitrum mainnet fork):**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";

interface IPool {
    function deposit(address token, uint256 amount, string memory ref) external;
    function viewSwapRsETHAmountAndFee(uint256 amount, address token)
        external view returns (uint256 rsETH, uint256 fee);
}
interface IInterimOracle { function getRate() external view returns (uint256); }
interface IChainlinkWrapper { function getRate() external view returns (uint256); }

contract DualOracleArbitrageTest is Test {
    IPool          pool        = IPool(RSETH_POOL_ARBITRUM);
    IInterimOracle rsETHOracle = IInterimOracle(INTERIM_ORACLE_ARBITRUM);
    IChainlinkWrapper collOracle = IChainlinkWrapper(WSTETH_ORACLE_ARBITRUM);
    address        wstETH      = WSTETH_ARBITRUM;

    function testOracleDivergenceExcess() public {
        // Fork at a block where Chainlink has updated but InterimOracle has not
        // Step 1: Read both oracle values
        uint256 rsETHRate  = rsETHOracle.getRate();   // e.g. 1.08e18 (stale)
        uint256 collRate   = collOracle.getRate();    // e.g. 1.15e18 (fresh)

        // Step 2: Compute fair vs actual output for 1 wstETH
        uint256 fairRsETHRate = 1.10e18; // true rsETH/ETH at time of Chainlink update
        uint256 fairOutput    = 1e18 * collRate / fairRsETHRate;
        (uint256 actualOutput,) = pool.viewSwapRsETHAmountAndFee(1e18, wstETH);

        // Step 3: Assert attacker receives more than fair value
        assertGt(actualOutput, fairOutput, "No excess: oracles in sync");

        // Step 4: Execute deposit and verify wrsETH balance increase
        deal(wstETH, address(this), 1e18);
        IERC20(wstETH).approve(address(pool), 1e18);
        uint256 balBefore = IERC20(WRSETH_ARBITRUM).balanceOf(address(this));
        pool.deposit(wstETH, 1e18, "arb");
        uint256 balAfter  = IERC20(WRSETH_ARBITRUM).balanceOf(address(this));
        assertEq(balAfter - balBefore, actualOutput);
        assertGt(actualOutput, fairOutput);
    }
}
```

The test passes whenever `InterimRSETHOracle.rate` lags behind the true rsETH/ETH price while `ChainlinkOracleForRSETHPoolCollateral` reflects a fresh collateral price — a condition that occurs naturally between every manual manager update.
