Audit Report

## Title
Missing L2 Sequencer Uptime Check Enables Stale-Price Over-Minting of rsETH - (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

## Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` without verifying that the L2 sequencer is live. When the sequencer is offline, Chainlink L2 feeds freeze at the last recorded price. Because rsETH is a yield-bearing token whose ETH-denominated price only increases, the frozen (stale) price is lower than the true current price, allowing any depositor to mint more rsETH than their ETH deposit is worth at the true rate, diluting existing rsETH holders' unclaimed yield.

## Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs three checks after calling `latestRoundData()`: [1](#0-0) 

- `answeredInRound < roundID` — detects incomplete rounds, but during a sequencer outage Chainlink L2 feeds do not advance rounds; `answeredInRound == roundID` remains true throughout the outage.
- `timestamp == 0` — detects a zero timestamp, but the frozen timestamp from before the outage is non-zero.
- `ethPrice <= 0` — detects a zero/negative price, but the frozen price is a valid positive value.

None of these checks detect a sequencer outage. There is no sequencer uptime feed consulted and no staleness threshold (`block.timestamp - updatedAt > MAX_STALENESS`).

This oracle is consumed by `RSETHPoolV2.getRate()` and `RSETHPoolV3.getRate()`: [2](#0-1) [3](#0-2) 

Both pools use the returned rate to compute the rsETH mint amount: [4](#0-3) [5](#0-4) 

A lower (stale) `rsETHToETHrate` in the denominator produces a higher `rsETHAmount`, minting excess tokens to the depositor.

`ChainlinkPriceOracle.getAssetPrice()` has an even weaker posture — it discards all five return values except `price` and performs no staleness or round-completeness check at all: [6](#0-5) 

## Impact Explanation

**High — Theft of unclaimed yield.**

rsETH is a yield-bearing token; its ETH-denominated price (`rsETHToETHrate`) monotonically increases as staking rewards accrue. When the sequencer resumes after an outage, the Chainlink feed has not yet updated. During this window, every depositor receives excess rsETH tokens. Those excess tokens represent a proportional claim on the pool's underlying ETH, diluting the yield that rightfully belongs to pre-existing rsETH holders. The attacker's gain is the existing holders' loss of accrued-but-unclaimed yield.

## Likelihood Explanation

Arbitrum, Optimism, and Base have each experienced documented sequencer outages. The protocol is deployed on at least eight L2 networks, multiplying exposure. No privileged access is required — any EOA can call `deposit()` during the post-outage window. The window lasts from sequencer resumption until the Chainlink feed catches up to the true price. The daily mint limit caps per-day magnitude but does not prevent the attack.

## Recommendation

Integrate the [Chainlink L2 Sequencer Uptime Feed](https://docs.chain.link/data-feeds/l2-sequencer-feeds) pattern in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

function getRate() public view returns (uint256) {
    (, int256 seqAnswer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (seqAnswer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();

    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();
    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (block.timestamp - timestamp > MAX_STALENESS) revert StalePrice();
    if (ethPrice <= 0) revert InvalidPrice();
    // ...
}
```

Also add a staleness threshold check and round-completeness checks to `ChainlinkPriceOracle.getAssetPrice()`.

## Proof of Concept

1. Protocol is deployed on Arbitrum. `RSETHPoolV2.rsETHOracle` points to `ChainlinkOracleForRSETHPoolCollateral`.
2. True rsETH/ETH rate is `1.05e18`. Sequencer goes offline; last Chainlink price recorded is `1.04e18`.
3. Sequencer resumes. Chainlink feed still shows `1.04e18`. `answeredInRound == roundID`, `timestamp != 0` — all existing checks pass.
4. Attacker calls `RSETHPoolV2.deposit{value: 100 ether}("")`.
5. `viewSwapRsETHAmountAndFee(100e18)` → `getRate()` returns `1.04e18`.
6. `rsETHAmount = 100e18 * 1e18 / 1.04e18 ≈ 96.15 rsETH`; correct amount at true rate: `100e18 / 1.05e18 ≈ 95.24 rsETH`.
7. Attacker receives ~0.91 excess rsETH per 100 ETH, at the expense of existing holders.

**Foundry fork test plan:** Fork Arbitrum mainnet at a block just after a historical sequencer outage. Deploy or point to the live `ChainlinkOracleForRSETHPoolCollateral`. Warp `block.timestamp` forward without advancing the Chainlink round (simulating the post-outage stale window). Call `deposit()` and assert that `rsETHAmount` exceeds `amountAfterFee * 1e18 / trueRate`, confirming over-minting. [1](#0-0) [7](#0-6)
