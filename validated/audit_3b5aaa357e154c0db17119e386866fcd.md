Audit Report

## Title
ETH Withdrawal Permanently Frozen for Contracts Unable to Receive ETH тАФ (`contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTWithdrawalManager.unlockQueue` burns the user's rsETH and redeems ETH into the contract in one finalized transaction. A subsequent `completeWithdrawal` call pushes ETH to the user via a low-level `.call{value}("")`. If the user is a smart contract that cannot receive ETH, every `completeWithdrawal` attempt reverts with `EthTransferFailed()`, and because the rsETH burn is irreversible, the user permanently loses their rsETH with no on-chain path to recover the corresponding ETH.

## Finding Description

The withdrawal lifecycle spans two separate, independent transactions:

**Transaction 1 тАФ `unlockQueue` (operator-only):**
At line 305, `IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned)` permanently burns the rsETH held by the contract. At line 307, `unstakingVault.redeem(asset, assetAmountUnlocked)` transfers the corresponding ETH into `LRTWithdrawalManager`. Both state changes are finalized and cannot be undone by any subsequent revert.

**Transaction 2 тАФ `completeWithdrawal` тЖТ `_processWithdrawalCompletion` тЖТ `_transferAsset`:**
At line 734, `_transferAsset(asset, user, request.expectedAssetAmount)` is called. For ETH, this executes:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```
(lines 878тАУ879)

If `to` is a contract with no `receive()` function or one whose fallback reverts, `sent == false` and the entire transaction reverts. The revert restores the withdrawal request record (the `delete` at line 712, the `popFront` at line 705, and the `unlockedWithdrawalsCount--` at line 717 are all unwound). However, the rsETH burn from Transaction 1 is permanent.

There is no `completeWithdrawalTo(address recipient)` or equivalent function. `completeWithdrawalForUser` (lines 192тАУ204) is operator-only and also hardcodes `user` as the destination by calling `_processWithdrawalCompletion(asset, user, referralId)`, which still routes to `_transferAsset(asset, user, ...)` at line 734. No alternative delivery path exists.

The comment at line 191 тАФ *"Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH"* тАФ acknowledges awareness of ETH transfer edge cases for `completeWithdrawalForUser` but incorrectly dismisses them; a complete ETH transfer failure (not merely gas grief) is not addressed anywhere.

## Impact Explanation

After `unlockQueue` finalizes, the ETH owed to the user sits in `LRTWithdrawalManager`. If the user's address cannot receive ETH, every `completeWithdrawal` call reverts indefinitely. The user's rsETH is already burned and unrecoverable. The ETH is permanently locked in the contract with no on-chain recovery path. This is **Permanent freezing of funds тАФ Critical**.

## Likelihood Explanation

Smart contract wallets (Gnosis Safe multisigs, account-abstraction wallets, protocol treasuries, yield aggregators) are common participants in DeFi withdrawal flows. A multisig with non-trivial receive logic, or any contract with no `receive()` function, will fail every ETH push. The condition is triggered by the normal withdrawal flow тАФ `initiateWithdrawal` is a public function callable by any address including contracts. No attacker action is required; the affected user simply cannot complete their own withdrawal. Any such user whose request is unlocked by the operator is permanently affected.

**Likelihood: Medium** тАФ requires the withdrawing address to be a contract that cannot receive ETH, which is realistic for multisigs and protocol-owned addresses.

## Recommendation

Add a `completeWithdrawalTo(address asset, address payable recipient, string calldata referralId)` function that allows `msg.sender` to redirect their ETH to an arbitrary `recipient`. Only the original requester can choose the redirect target, preserving fund attribution:

```solidity
function completeWithdrawalTo(
    address asset,
    address payable recipient,
    string calldata referralId
) external nonReentrant whenNotPaused {
    _processWithdrawalCompletionTo(asset, msg.sender, recipient, referralId);
}
```

Where `_processWithdrawalCompletionTo` is a variant of `_processWithdrawalCompletion` that calls `_transferAsset(asset, recipient, request.expectedAssetAmount)` instead of `_transferAsset(asset, user, request.expectedAssetAmount)`. The existing `completeWithdrawal` can delegate to this with `recipient = payable(msg.sender)` for backwards compatibility.

## Proof of Concept

1. Deploy a contract `NoReceive` with no `receive()` or `fallback()` function. `NoReceive` holds rsETH and calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager` (line 166).

2. After `withdrawalDelayBlocks`, the operator calls `unlockQueue(ETH_TOKEN, ...)`. At line 305, rsETH is burned from the contract. At line 307, ETH is redeemed into `LRTWithdrawalManager`. Both are finalized.

3. `NoReceive` calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion`, `_transferAsset(ETH_TOKEN, address(NoReceive), amount)` executes `payable(NoReceive).call{value: amount}("")`. The call fails (`sent == false`). `EthTransferFailed()` is thrown. The transaction reverts.

4. The withdrawal request record is restored. `NoReceive` retries тАФ every attempt reverts. The rsETH burn from step 2 is permanent. ETH is permanently locked in `LRTWithdrawalManager`.

**Foundry test sketch:**
```solidity
contract NoReceive {
    // no receive() or fallback()
    function doWithdraw(LRTWithdrawalManager mgr, address asset) external {

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
