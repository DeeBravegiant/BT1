Audit Report

## Title
Rewards Permanently Stuck When `totalKernelStaked` Drops to Zero Mid-Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` prevents starting a reward period with zero stakers via a guard in `notifyRewardAmount`, but any staker can call `initiateWithdrawal` at any time to reduce `totalKernelStaked` to zero during an active period. When this happens, `rewardPerToken()` freezes while `updatedAt` advances on the next interaction, permanently losing all rewards emitted during the zero-staked window with no recovery mechanism.

## Finding Description

The contract's NatSpec explicitly acknowledges this risk and states it is mitigated by off-chain operational controls (ensuring tokens are always staked), but provides no on-chain enforcement. [1](#0-0) 

The `notifyRewardAmount` function guards against starting a period with zero stakers: [2](#0-1) 

However, `initiateWithdrawal` immediately decrements `totalKernelStaked` with no check against an active reward period: [3](#0-2) 

When `totalKernelStaked == 0`, `rewardPerToken()` short-circuits and returns the stored value unchanged: [4](#0-3) 

Meanwhile, the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()`: [5](#0-4) 

The critical interaction: when the next staker calls `stake()`, the `updateReward` modifier executes **before** `totalKernelStaked` is incremented. At that moment `totalKernelStaked` is still 0, so `rewardPerToken()` returns the frozen stored value while `updatedAt` advances to the current time. The entire zero-staked window is consumed without distributing any rewards, and those tokens remain permanently locked — there is no admin sweep or rescue function anywhere in the contract.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** Reward tokens corresponding to any time window where `totalKernelStaked == 0` are irrecoverable. The loss equals `rewardRate × (duration of zero-staked window)`. There is no function in the contract to recover these tokens.

## Likelihood Explanation

Any staker can call `initiateWithdrawal` at any time for their full balance. The last remaining staker (or a dominant staker holding the majority of stake) can reduce `totalKernelStaked` to zero during an active reward period. The withdrawal delay only delays token transfer — `totalKernelStaked` is decremented immediately at `initiateWithdrawal` call time. This is reachable by any unprivileged external user through a standard public function call.

## Recommendation

Add a check in `initiateWithdrawal` that prevents `totalKernelStaked` from reaching zero while a reward period is active (`block.timestamp < finishAt`). Alternatively, implement an admin recovery function to reclaim unallocated rewards after a period ends, or track the zero-staked window duration and redistribute those rewards in the next period.

## Proof of Concept

1. Admin calls `setRewardsDuration(86400)`.
2. Alice calls `stake(1000e18)` → `totalKernelStaked = 1000e18`.
3. Admin calls `notifyRewardAmount(86400e18)` — passes the `totalKernelStaked > 0` check. `rewardRate = 1e18/s`, `finishAt = T+86400`, `updatedAt = T`.
4. At `T+3600`, Alice calls `initiateWithdrawal(1000e18)`:
   - `updateReward(Alice)` runs: `rewardPerTokenStored` updated, `updatedAt = T+3600`, Alice's rewards checkpointed.
   - `totalKernelStaked = 0`.
5. No one stakes from `T+3600` to `T+7200`.
6. At `T+7200`, Bob calls `stake(1000e18)`:
   - `updateReward(Bob)` runs first (before `totalKernelStaked` is incremented): `totalKernelStaked == 0` → `rewardPerToken()` returns `rewardPerTokenStored` unchanged; `updatedAt` advances to `T+7200`.
   - `totalKernelStaked = 1000e18`.
7. **3600 × 1e18 = 3600e18 reward tokens** emitted between `T+3600` and `T+7200` are permanently stuck. Bob earns only from `T+7200` onward; Alice already claimed up to `T+3600`.
