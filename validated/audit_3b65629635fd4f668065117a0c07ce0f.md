Audit Report

## Title
Stale `out0Scaled` after fallback `totalIn1Scaled` assignment causes trader to overpay in exact-input token1→token0 swaps — (`metric-core/contracts/libraries/SwapMath.sol`)

## Summary
In `buyToken0InBinSpecifiedIn`, when two successive approximation steps both underestimate the required token1 input for a given bin position, the fallback at line 621 forces `totalIn1Scaled = state.amountSpecifiedRemainingScaled` without recomputing `out0Scaled`. Settlement then charges the trader the full remaining token1 input while releasing only the token0 computed for the underestimated position, breaking the swap conservation invariant and causing direct trader principal loss. The identical structural bug exists symmetrically in `buyToken1InBinSpecifiedIn` at lines 758–760.

## Finding Description
The function uses a two-step iterative refinement to find the bin position that exactly consumes the trader's specified token1 input:

1. **Analytical quadratic** (`computeAnalyticalTargetPosForBuyToken0`, lines 581–596) gives an initial `targetPos`, `out0Scaled`, and `totalIn1Scaled`.
2. **Proportional scale-up** (lines 598–618): if `totalIn1Scaled < remaining && targetPos < maxFinalBinPos`, scales `delta` proportionally and recomputes `out0Scaled`/`totalIn1Scaled` for the new `scaledTarget`. If `scaledTarget > maxFinalBinPos`, clamps to `maxFinalBinPos` — after which the Step 4 condition `targetPos < maxFinalBinPos` is false and the fallback cannot fire.
3. **Fallback** (lines 620–622): if after Step 2 `totalIn1Scaled` is still below `remaining` and `targetPos < maxFinalBinPos`, the code sets `totalIn1Scaled = remaining` but leaves `out0Scaled` at the value computed for `scaledTarget`. [1](#0-0) 

After the fallback, `totalIn1Scaled == remaining`, so the scale-down branch (lines 624–634) — which correctly rescales `out0Scaled` proportionally — is skipped entirely. [2](#0-1) 

Settlement at lines 639–644 then charges `remaining` token1 and credits the stale, smaller `out0Scaled`: [3](#0-2) 

The condition for the fallback to fire: `scaledTarget <= maxFinalBinPos` (the proportional scale-up did not saturate the price limit) AND the recomputed `totalIn1Scaled` for `scaledTarget` is still below `remaining`. This occurs when the price curve is sufficiently concave within the bin (large `deltaPriceX64 / lowerPriceX64` ratio), causing the linear proportional correction to underestimate the true quadratic relationship.

The symmetric bug in `buyToken1InBinSpecifiedIn` sets `totalIn0Scaled = remaining` without updating `out1Scaled`: [4](#0-3) 

## Impact Explanation
The trader pays `remaining` token1 but receives `out0(scaledTarget)` token0 instead of `out0(true_target)`. The shortfall `out0(true_target) - out0(scaledTarget)` remains in the bin and accrues to LPs. This is a direct, uncompensated loss of trader principal on every swap that triggers the fallback — a swap conservation failure. The magnitude scales with bin spread and token0 balance; for wide bins it can reach several percent of the swap output. This meets the Sherlock High/Critical threshold for direct loss of user principal.

## Likelihood Explanation
Any exact-input token1→token0 swap through a bin with a sufficiently large price spread (`upperPriceX64 / lowerPriceX64` ratio) and an input amount that causes both the analytical and proportional refinement to underestimate will trigger this path. No privileged access, malicious setup, or non-standard token behavior is required — the path is reachable through the normal public `swap` entrypoint by any unprivileged trader.

## Recommendation
Replace the fallback at lines 620–622 with a correct resolution: set `targetPos = maxFinalBinPos` and recompute `out0Scaled`, `finalPriceX64`, `avgPriceX64`, `in1WithoutFeeScaled`, and `totalIn1Scaled` for `maxFinalBinPos`. Since `totalIn1Scaled` for `maxFinalBinPos` exceeds `remaining` (guaranteed by the outer `else` branch entered at line 579), the scale-down branch (lines 624–634) will then fire and correctly proportionally reduce both `targetPos` and `out0Scaled`. This mirrors the existing correct pattern for the over-estimate case and requires no new logic. Apply the same fix symmetrically to `buyToken1InBinSpecifiedIn` at lines 758–760.
