### Title
`respond_ckd` Skips On-Chain Cryptographic Verification for `AppPublicKey` Variant, Allowing a Single Byzantine Participant to Forge CKD Responses - (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in `MpcContract` performs a BLS12-381 pairing check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of CKD requests. For the `AppPublicKey` (legacy/default) variant, the match arm is empty — no cryptographic verification is performed before the response is accepted and the yield is resolved. Any single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept it unconditionally.

---

### Finding Description

In `crates/contract/src/lib.rs`, `respond_ckd` handles two variants of CKD requests:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the network's BLS12-381 master public key and the user's ephemeral key pair. [2](#0-1) 

For `AppPublicKey`, the arm is empty. The contract proceeds directly to `resolve_yields_for`, which drains the pending queue and delivers the response to the user — with no check that `(big_y, big_c)` is consistent with the network's master secret or the user's ephemeral public key. [3](#0-2) 

The `AppPublicKey` variant is the legacy/default format documented as the primary usage path. [4](#0-3) 

The access guard `assert_caller_is_attested_participant_and_protocol_active` requires only that the caller is a single attested participant — there is no threshold requirement on the `respond_ckd` call itself. [5](#0-4) 

This is structurally identical to the `recoverERC20` pattern: one asset type (`AppPublicKeyPV`) is protected by a guard, while the other (`AppPublicKey`) is not, creating a backdoor for the unprotected type.

The contrast with `respond` (for ECDSA/EdDSA signatures) is instructive: `respond` always verifies the signature cryptographically against the derived public key before resolving the yield. [6](#0-5) 

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` confirms the contract accepts completely arbitrary byte values `[1u8; 48]` and `[2u8; 48]` as `big_y`/`big_c` for `AppPublicKey` requests without any rejection. [7](#0-6) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` request. Because `resolve_yields_for` removes the pending entry and resumes all queued yields on the first successful call, the honest coordinator's later response will find `RequestNotFound` and be silently discarded. [8](#0-7) 

The attacker can craft `(big_y, big_c)` such that the user decrypts an attacker-chosen value: given user ephemeral key `a` with `app_pk = a·G1`, the attacker sets `big_y' = y'·G1` and `big_c' = target + app_pk·y'` for any target point. The user computes `big_c' − a·big_y' = target`, receiving the attacker's chosen key material instead of the legitimate MPC-derived key.

This constitutes **unauthorized confidential key derivation output without the required participant authorization** — the threshold is bypassed entirely for the `AppPublicKey` variant.

---

### Likelihood Explanation

- `AppPublicKey` is the legacy/default format and is the most commonly used variant in practice.
- Any single attested participant (one compromised TEE node, strictly below threshold) can execute this attack.
- The attack requires no collusion, no key leakage, and no network-level capability — only a valid TEE attestation and knowledge of a pending `AppPublicKey` request (which is observable on-chain).
- The attacker races to call `respond_ckd` before the honest coordinator; since the forged call is a simple contract invocation with no expensive computation, the race is easily won.

---

### Recommendation

Apply the same on-chain verification discipline to `AppPublicKey` that is applied to `AppPublicKeyPV`. Since `AppPublicKey` provides only a G1 point, the pairing-based `ckd_output_check` cannot be used directly. The recommended mitigations are:

1. **Deprecate `AppPublicKey` entirely** and require all new requests to use `AppPublicKeyPV`, which supports on-chain verification. Existing pending `AppPublicKey` requests should be allowed to time out.

2. **If `AppPublicKey` must be retained**, add a threshold-enforced response mechanism (e.g., require `t` independent `respond_ckd` calls with matching responses before resolving the yield), analogous to how threshold voting is enforced for governance operations.

The fix for `AppPublicKeyPV` is the correct pattern:

```rust
dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
    if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
        env::panic_str("CKD output check failed");
    }
}
``` [9](#0-8)

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(some_g1_point)` and attaches 1 yoctoNEAR deposit. The request is stored in `pending_ckd_requests`.

2. Byzantine attested participant `evil.near` observes the pending request on-chain and immediately calls:
   ```
   respond_ckd(
     request = <the pending CKDRequest>,
     response = CKDResponse { big_y: [attacker_chosen_G1], big_c: [attacker_chosen_G1] }
   )
   ```

3. `respond_ckd` passes `assert_caller_is_attested_participant_and_protocol_active` (evil.near is a valid attested participant), enters the `AppPublicKey(_) => {}` arm with no verification, and calls `resolve_yields_for` — which resumes the user's yield with the forged response and removes the pending entry.

4. The honest coordinator's subsequent `respond_ckd` call returns `RequestNotFound` and is discarded.

5. The user's callback receives `CKDResponse { big_y: [attacker_chosen_G1], big_c: [attacker_chosen_G1] }` and decrypts an attacker-controlled key instead of the legitimate MPC-derived key. [10](#0-9) [11](#0-10)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {

            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
