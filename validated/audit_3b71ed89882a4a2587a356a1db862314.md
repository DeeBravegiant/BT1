### Title
Wrong Receiver Checked in `validate_delegate_action` Allows `DeterministicStateInit` Meta-Transaction to Bypass Initial Tx Validation — (`runtime/runtime/src/action_validation.rs`)

---

### Summary

Before protocol version 85 (`FixDelegatedDeterministicStateInit`), `validate_delegate_action` validates the inner actions of a `DelegateAction` (meta-transaction) against the **outer transaction's `receiver_id`** — which equals `delegate_action.sender_id`, the relayer's target — instead of `delegate_action.receiver_id`, the actual target of the inner actions. This is the direct nearcore analog of the UnionToken bug: the wrong principal is checked for the permission/validity constraint.

---

### Finding Description

In `runtime/runtime/src/action_validation.rs`, `validate_delegate_action` selects which account ID to use when validating inner actions:

```rust
let inner_receiver =
    if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
        // This is the correct receiver id to use for the check.
        delegate_action.receiver_id()
    } else {
        // This is a bug fixed with `FixDelegatedDeterministicStateInit` that
        // validated against the wrong id.
        receiver   // ← outer tx receiver_id, NOT delegate_action.receiver_id()
    };
validate_actions_with_mode(limit_config, &actions, inner_receiver, ...)?;
``` [1](#0-0) 

The `receiver` parameter passed in is the outer transaction's `receiver_id` — the account on whose shard the `DelegateAction` executes, which equals `delegate_action.sender_id`. The actual target of the inner actions is `delegate_action.receiver_id()`, a different account.

For `DeterministicStateInit`, the inner validation calls `validate_deterministic_state_init`, which checks that `derive(state_init) == receiver_id`:

```rust
Action::DeterministicStateInit(a) => {
    validate_deterministic_state_init(limit_config, a, receiver)
}
``` [2](#0-1) 

```rust
fn validate_deterministic_state_init(..., receiver_id: &AccountId) -> Result<...> {
    let derived_id = derive_near_deterministic_account_id(&action.state_init);
    if derived_id != *receiver_id {
        return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver { ... });
    }
    ...
}
``` [3](#0-2) 

Before the fix, the check is performed against the outer `receiver` (= `sender_id` of the delegate action), not against `delegate_action.receiver_id()`. An attacker can therefore craft a meta-transaction where:

- Outer tx `receiver_id` = `det_account_b` = `derive(state_init_b)` ✓ (passes the wrong check)
- `delegate_action.receiver_id` = `det_account_a` (a different deterministic account)
- Inner action = `DeterministicStateInit { state_init: state_init_b }`

The initial tx validation passes because `derive(state_init_b) == det_account_b` (the outer receiver). The tx is admitted to the mempool. When the receipt is unpacked and re-validated, `validate_receipt` checks `state_init_b` against `det_account_a` (the correct inner receiver), which fails with `InvalidDeterministicStateInitReceiver`.

The exploit scenario and both pre-fix and post-fix behaviors are confirmed by the test suite: [4](#0-3) 

The fix is gated behind `ProtocolFeature::FixDelegatedDeterministicStateInit`, assigned to protocol version 85: [5](#0-4) 

---

### Impact Explanation

The corrupted protocol value is the **tx admission validity decision**: a transaction that should be rejected at the RPC/mempool layer is instead admitted and processed. The receipt-level re-validation (`validate_receipt`) catches the mismatch and returns a failure outcome, so no state root or account balance is corrupted. The practical consequence is that a relayer's gas is consumed for a transaction that was always going to fail, and the mempool admits a structurally invalid meta-transaction. The code comment acknowledges: *"The bug cannot be abused, if someone crafts a state init that passes validation here, it will fail when it is checked as incoming receipt."*

---

### Likelihood Explanation

Any unprivileged user who controls a deployed deterministic account (with a full-access key added to it) can craft this transaction. No validator or node-admin privilege is required. The attacker entry path is a standard signed transaction submitted via public RPC.

---

### Recommendation

The fix is already present in the codebase: when `ProtocolFeature::FixDelegatedDeterministicStateInit` is enabled (protocol version 85), `validate_delegate_action` correctly uses `delegate_action.receiver_id()` as the `inner_receiver` for validating inner actions. Ensure all nodes upgrade to protocol version ≥ 85 to activate the fix.

---

### Proof of Concept

The exploit is fully documented and tested in the repository: [6](#0-5) 

Steps:
1. Deploy `det_account_b = derive(state_init_b)` and add a full-access key to it.
2. Craft a `DelegateAction` with `sender_id = det_account_b`, `receiver_id = det_account_a`, inner action = `DeterministicStateInit { state_init: state_init_b }`.
3. Wrap it in an outer tx with `receiver_id = det_account_b`.
4. Submit via public RPC on a node running protocol version < 85.
5. The outer tx passes initial validation (wrong receiver checked). The receipt fails with `InvalidDeterministicStateInitReceiver` at execution time.

fn validate_deterministic_state_init(
    limit_config: &LimitConfig,
    action: &DeterministicStateInitAction,
    receiver_id: &AccountId,
) -> Result<(), ActionsValidationError> {
    validate_global_contract_identifier(action.state_init.code())?;

    let derived_id = derive_near_deterministic_account_id(&action.state_init);

    if derived_id != *receiver_id {
        return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver {
            derived_id,
            receiver_id: receiver_id.clone(),
        });
    }

fn try_meta_tx_deterministic_receiver_exploit(
    protocol_version: ProtocolVersion,
) -> Result<FinalExecutionOutcomeView, InvalidTxError> {
    let mut env = TestEnv::setup_with_version(Balance::from_near(100), protocol_version);
    env.deploy_global_contract(GlobalContractDeployMode::AccountId);

    let (_state_init_a, det_account_a) = env.new_deterministic_account_with_data(small());
    let (state_init_b, det_account_b) = env.new_deterministic_account_with_data(big());
    assert_ne!(det_account_a, det_account_b);

    // Deploy det_account_b and add a full-access key so it can act as meta_tx_sender.
    let user_signer = create_user_test_signer(&env.user_account());
    let storage_balance = env.balance_for_storage(state_init_b.clone());
    let deploy_tx = SignedTransaction::deterministic_state_init(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        env.get_tx_block_hash(),
        state_init_b.clone(),
        storage_balance,
    );
    env.run_tx(deploy_tx);

    let meta_tx_sender_signer = create_user_test_signer(&det_account_b);
    let pk_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&meta_tx_sender_signer.public_key()).unwrap(),
    );
    let add_key_args = serde_json::json!([
        { "batch_create": { "account_id": det_account_b.as_str() }, "id": 0 },
        {
            "action_add_key_with_full_access": {
                "promise_index": 0,
                "public_key": pk_base64,
                "nonce": 0
            },
            "id": 0,
            "return": true
        }
    ]);
    let add_key_tx = SignedTransaction::call(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        Balance::from_near(2),
        "call_promise".to_owned(),
        serde_json::to_vec(&add_key_args).unwrap(),
        Gas::from_teragas(300),
        env.get_tx_block_hash(),
    );
    env.run_tx(add_key_tx);

    // Craft the exploit: outer_tx.receiver = det_account_b = derive(state_init_b).
    // Old check: det_account_b == derive(state_init_b) passes.
    // The delegate action targets det_account_a, which is the wrong account.
    // In no protocol version can this ever be allowed to be executed successfully.
    let relayer = env.independent_account();
    let relayer_signer = create_user_test_signer(&relayer);
    let inner_action = Action::DeterministicStateInit(Box::new(DeterministicStateInitAction {
        state_init: state_init_b,
        deposit: Balance::ZERO,
    }));
    let delegate_nonce = env.next_nonce_for(&det_account_b);
    let delegate_action = DelegateAction {
        sender_id: det_account_b.clone(),
        receiver_id: det_account_a,
        actions: vec![NonDelegateAction::try_from(inner_action).unwrap()],
        nonce: delegate_nonce,
        max_block_height: 1_000_000,
        public_key: meta_tx_sender_signer.public_key(),
    };
    let signed_delegate_action =
        SignedDelegateAction::sign(&meta_tx_sender_signer, delegate_action);
    let tx = SignedTransaction::from_actions(
        env.next_nonce(),
        relayer,
        det_account_b,
        &relayer_signer,
        vec![Action::Delegate(Box::new(signed_delegate_action))],
        env.get_tx_block_hash(),
    );
    env.try_execute_tx(tx)
}
