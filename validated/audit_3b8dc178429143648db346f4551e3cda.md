### Title
VAA Replay via `remove_fin_transfer` in `ft_transfer_call` Refund Path Enables Double Settlement — (`near/omni-bridge/src/lib.rs`)

### Summary
When a `fin_transfer` VAA carries a non-empty `msg` field and the NEAR recipient contract's `ft_on_transfer` returns a full refund, `fin_transfer_send_tokens_callback` calls `remove_fin_transfer`, erasing the `TransferId` from `finalised_transfers`. Because `fin_transfer_callback` has no independent VAA-level replay guard, the identical VAA can be re-submitted and will pass `add_fin_transfer`'s uniqueness check a second time, producing a second settlement.

### Finding Description

The replay protection for Wormhole-path `fin_transfer` is implemented exclusively through `finalised_transfers`:

```
add_fin_transfer  →  finalised_transfers.insert(transfer_id)   // panics if already present
``` [3](#0-2) 

The refund path in `fin_transfer_send_tokens_callback` unconditionally removes that entry:

```rust
if Self::is_refund_required(is_ft_transfer_call) {
    self.burn_tokens_if_needed(...);
    self.revert_lock_actions(&lock_actions);
    self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
    ...
}
``` [7](#0-6) 

`is_refund_required` returns `true` when `ft_transfer_call` returns `0` (i.e., the recipient's `ft_on_transfer` returned the full amount): [9](#0-8) 

After removal, `finalised_transfers` no longer contains the `TransferId`, so re-submitting the same VAA passes `add_fin_transfer`'s `require!(self.finalised_transfers.insert(transfer_id), ...)` check without error.

There is no secondary VAA hash / sequence-number store that would independently block re-submission. [1](#0-0) 

### Impact Explanation

**For deployed (bridge-minted) tokens:** `send_tokens` calls `mint(recipient, amount, Some(msg))`. On the first submission the tokens are minted, the recipient rejects them, the bridge burns the refunded tokens, and removes the `TransferId`. On the second submission the tokens are minted again and this time the recipient accepts them. Net result: one VAA produces two mint events → unbacked supply.

**For native (locked) tokens:** `send_tokens` calls `ft_transfer_call`. On the first submission the bridge transfers tokens from its custody, the recipient rejects them, the tokens are returned to the bridge, and the `TransferId` is removed. On the second submission the bridge transfers the same tokens again. Net result: one VAA drains the bridge's locked balance twice.

### Likelihood Explanation

The attacker must:
1. Initiate a cross-chain transfer on the source chain (e.g., Ethereum) with a non-empty `msg` field and set the NEAR recipient to a contract they control. This is a normal, unprivileged user action.
2. Obtain the Wormhole-signed VAA (standard relayer infrastructure).
3. Deploy a NEAR contract whose `ft_on_transfer` returns the full amount on the first call and `0` on the second call (trivial to implement).
4. Submit the VAA twice.

No privileged role, no key compromise, and no guardian collusion is required. The attacker is the original sender of the cross-chain transfer.

### Recommendation

Do not use `finalised_transfers` as the sole replay guard for VAA-based transfers. One of the following fixes should be applied:

1. **Never call `remove_fin_transfer` for VAA-sourced transfers.** The `TransferId` must remain in `finalised_transfers` permanently regardless of whether the downstream `ft_transfer_call` succeeded or failed. The refund/burn logic can still execute; only the `remove_fin_transfer` call must be removed from the VAA path.

2. **Maintain a separate VAA hash set.** Record the Wormhole VAA hash (or `(emitter_chain, emitter_address, sequence)` tuple) at the start of `fin_transfer_callback`, before any token operations, and never remove it.

### Proof of Concept

```
1. Attacker deploys `MaliciousRecipient` on NEAR:
     - call_count = 0
     - ft_on_transfer: if call_count == 0 { call_count++; return amount } else { return 0 }

2. Attacker sends a cross-chain transfer on Ethereum:
     - recipient = MaliciousRecipient (NEAR address)
     - msg = "trigger_ft_transfer_call"

3. Wormhole guardians sign the VAA (standard flow).

4. Attacker calls fin_transfer(VAA):
     - fin_transfer_callback → process_fin_transfer_to_near
     - add_fin_transfer: finalised_transfers.insert(transfer_id) → OK
     - send_tokens → ft_transfer_call(MaliciousRecipient, amount, msg)
     - MaliciousRecipient.ft_on_transfer returns `amount` (full refund)
     - ft_transfer_call returns 0
     - fin_transfer_send_tokens_callback: is_refund_required = true
     - burn_tokens_if_needed (tokens burned)
     - remove_fin_transfer: finalised_transfers.remove(transfer_id)

5. Attacker calls fin_transfer(same VAA):
     - fin_transfer_callback → process_fin_transfer_to_near
     - add_fin_transfer: finalised_transfers.insert(transfer_id) → OK (was removed)
     - send_tokens → ft_transfer_call(MaliciousRecipient, amount, msg)
     - MaliciousRecipient.ft_on_transfer returns 0 (accepts tokens)
     - Attacker receives tokens

Result: one VAA → two settlements → double credit / unbacked supply.
```

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
