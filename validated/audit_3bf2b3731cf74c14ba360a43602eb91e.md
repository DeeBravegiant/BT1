### Title
Propose-Update Deposits Permanently Locked in Contract When Proposals Are Cleared - (File: crates/contract/src/update.rs)

### Summary

When `propose_update` is called, participants pay a storage-staking deposit that is never refunded — not when their proposal wins, not when it is displaced by a competing proposal, and not when `do_update` bulk-clears all pending entries. Every governance cycle permanently locks NEAR in the contract, breaking the accounting invariant that storage deposits should be returned when the storage they cover is freed.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires the caller to attach a deposit equal to `ProposedUpdates::required_deposit(&update)`, which is computed as `env::storage_byte_cost() × bytes_used(update)`. [1](#0-0) 

The `bytes_used` calculation includes the full contract binary size plus a fixed overhead for 128 participant-vote slots: [2](#0-1) 

The exact `required` amount is retained by the contract; only the excess above `required` is refunded to the proposer: [3](#0-2) 

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry **and bulk-clears every other pending entry and every vote**: [4](#0-3) 

No refund is issued to any proposer — neither the winner nor the losers. The storage freed by `entries.clear()` reduces the contract's storage-staking obligation, so the NEAR that was deposited to cover that storage is now surplus balance permanently held by the contract with no withdrawal path.

### Impact Explanation

For a typical contract binary (~1 MB), `required_deposit` is approximately 10 NEAR (NEAR storage costs ~10 NEAR/MB). In a governance cycle where multiple participants each propose a competing update, every proposer permanently loses their full deposit. The contract's balance grows monotonically with each governance cycle. There is no `cancel_proposal`, `withdraw_deposit`, or any other mechanism to reclaim these funds. This breaks the production safety/accounting invariant that storage-staking deposits must be returned when the storage they cover is released.

**Impact: Medium** — balance/accounting invariant broken; funds permanently locked in the contract without any loss path for the broader network, but with direct, irreversible loss to participants in every governance cycle.

### Likelihood Explanation

This triggers on every successful `vote_update` execution. The README explicitly states that all pending proposals are cleared on every contract upgrade. In a network with multiple active participants who each propose competing updates (a normal governance pattern), every participant except the one whose proposal happens to be executed last loses their deposit. The condition is met in routine protocol operation, not only under adversarial conditions.

**Likelihood: Medium** — occurs in normal governance operation whenever more than one proposal exists at execution time.

### Recommendation

Track the proposer's `AccountId` alongside each `UpdateEntry`. In `do_update`, before calling `entries.clear()`, iterate over all remaining entries and issue a `Promise::new(proposer).transfer(deposit)` refund for each one. Similarly, refund the winning proposer's deposit after the update is applied (e.g., in a post-update callback). This mirrors the pattern already used in `submit_participant_info`, which correctly refunds excess deposits. [5](#0-4) 

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR.
2. Participant B calls `propose_update` with a different 1 MB binary, attaching ~10 NEAR.
3. Threshold participants vote for B's proposal; `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` (removing A's entry) and `self.vote_by_participant.clear()`.
5. No refund is issued to A or B.
6. The contract's balance retains ~20 NEAR that was deposited to cover storage now freed. Neither A nor B can recover their deposits. [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
