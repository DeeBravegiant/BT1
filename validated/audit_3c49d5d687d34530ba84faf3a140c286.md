### Title
Frankenstein ChunkEndorsementV2 Poisons Endorsement Cache, Silently Drops Legitimate Endorsements — (`core/primitives/src/stateless_validation/chunk_endorsement.rs`, `chain/client/src/stateless_validation/chunk_endorsement.rs`)

---

### Summary

`ChunkEndorsementV2::verify()` independently validates `signature` over `inner` and `metadata_signature` over `metadata`, but never cross-checks that `inner.chunk_hash` corresponds to the chunk identified by `metadata.(shard_id, epoch_id, height_created)`. A network peer who observes two legitimate endorsements from the same validator (for different shards at the same height) can splice them into a structurally valid "Frankenstein" endorsement that passes all validation, poisons the `ChunkEndorsementTracker` cache under the wrong `chunk_hash`, and causes the "already received" guard to silently drop the real endorsement when it arrives — permanently preventing that validator's stake from being counted for the targeted shard.

---

### Finding Description

**Structure of `ChunkEndorsementV2`** [1](#0-0) 

The struct has four independent fields: `inner` (containing `chunk_hash`), `signature` (over `inner`), `metadata` (containing `shard_id`, `epoch_id`, `height_created`), and `metadata_signature` (over `metadata`). The `verify()` method checks each signature against its own payload independently: [2](#0-1) 

There is no assertion that `inner.chunk_hash` is the hash of the chunk identified by `metadata.(shard_id, epoch_id, height_created)`.

**Constructing the Frankenstein endorsement**

Given two legitimate endorsements from the same validator `V` at the same height `H` in epoch `E`:
- **Endorsement A**: `inner={chunk_hash_A}`, `sig_A` (valid over `inner_A`), `metadata={shard_id=0, E, H}`, `meta_sig_A`
- **Endorsement B**: `inner={chunk_hash_B}`, `sig_B` (valid over `inner_B`), `metadata={shard_id=1, E, H}`, `meta_sig_B`

The attacker constructs **Frankenstein F**:
- `inner = inner_A` (chunk_hash_A), `signature = sig_A` → valid over `inner_A` ✓
- `metadata = metadata_B` (shard_id=1, E, H), `metadata_signature = meta_sig_B` → valid over `metadata_B` ✓

`ChunkEndorsementV2::verify()` returns `true` for F because both signatures are individually valid.

**Validation pipeline does not catch this**

`validate_chunk_endorsement` derives the routing key from `metadata` (shard_id=1, E, H), checks that validator V is assigned to shard 1 at height H, then calls `validate_chunk_endorsement_signature` which calls `endorsement.verify(public_key)` — the same broken check above. [3](#0-2) 

**Cache poisoning and silent drop**

`process_chunk_endorsement` inserts the Frankenstein into the cache under key `(shard_id=1, E, H)` with value `(chunk_hash_A, sig_A)`: [4](#0-3) 

When the legitimate endorsement B arrives for shard 1 (chunk_hash_B), the "already received" guard fires: [5](#0-4) 

Endorsement B is silently dropped. Later, `collect_chunk_endorsements` filters cached entries by `chunk_hash == chunk_header.chunk_hash()`: [6](#0-5) 

Since `chunk_hash_A ≠ chunk_hash_B`, the poisoned entry is excluded. Validator V's stake is not counted for shard 1.

---

### Impact Explanation

If enough validators are targeted (enough to push the shard below the endorsement stake threshold), `collect_chunk_endorsements` returns `is_endorsed=false` for shard 1's chunk, and the block producer cannot include that chunk. This blocks chunk inclusion for the targeted shard, a direct consensus-layer liveness impact. The comment at line 100–104 of `chunk_endorsement.rs` explicitly notes that `chunk_hash` matching is the *only* remaining check at collection time, making the poisoned cache entry the sole arbiter: [7](#0-6) 

---

### Likelihood Explanation

The attacker must be a network peer that can observe chunk endorsement messages in transit (they are gossiped through the peer-to-peer network as `ChunkEndorsementMessage`). The attacker needs two endorsements from the same validator for two different shards at the same height — feasible in any multi-shard epoch where validators are assigned to multiple shards. The Frankenstein must arrive before the legitimate endorsement B, which is achievable by injecting it immediately upon observing endorsement A (before B is even produced or routed). No validator or admin privileges are required.

---

### Recommendation

Add a cross-check inside `ChunkEndorsementV2::verify()` (or in `validate_chunk_endorsement_signature`) that confirms `inner.chunk_hash` is consistent with `metadata.(shard_id, epoch_id, height_created)`. Since the chunk hash is not directly derivable from the metadata alone (it requires the actual chunk header), the simpler fix is to bind `inner.chunk_hash` into the `metadata` signature — i.e., include `chunk_hash` as a field in `ChunkEndorsementMetadata` and sign it, so that swapping `inner` from a different endorsement invalidates `metadata_signature`. Alternatively, sign a single combined payload covering both `chunk_hash` and `(shard_id, epoch_id, height_created)` to eliminate the split-signature design entirely.

---

### Proof of Concept

```rust
// Pseudocode for a Rust unit test
let signer = create_validator_signer("validator_v");
let header_shard0 = make_chunk_header(shard_id=0, height=H, epoch=E);
let header_shard1 = make_chunk_header(shard_id=1, height=H, epoch=E);

let endorsement_a = ChunkEndorsement::new(E, &header_shard0, &signer); // chunk_hash_A, shard 0
let endorsement_b = ChunkEndorsement::new(E, &header_shard1, &signer); // chunk_hash_B, shard 1

// Splice: take inner+signature from A, metadata+metadata_signature from B
let frankenstein = ChunkEndorsement::V2(ChunkEndorsementV2 {
    inner: endorsement_a.inner.clone(),           // chunk_hash_A
    signature: endorsement_a.signature.clone(),   // valid over inner_A
    metadata: endorsement_b.metadata.clone(),     // shard_id=1, E, H
    metadata_signature: endorsement_b.metadata_signature.clone(), // valid over metadata_B
});

// Both signatures verify independently → true
assert!(frankenstein.verify(&signer.public_key()));

// Process Frankenstein first → poisons cache for (shard_id=1, E, H) with chunk_hash_A
tracker.process_chunk_endorsement(&frankenstein).unwrap();

// Process legitimate endorsement B → silently dropped by "already received" guard
tracker.process_chunk_endorsement(&endorsement_b).unwrap();

// collect_chunk_endorsements for shard 1 (chunk_hash_B) → filters out chunk_hash_A → not endorsed
let state = tracker.collect_chunk_endorsements(&header_shard1).unwrap();
assert!(!state.is_endorsed); // validator V's stake is lost for shard 1
```

```rust
#[derive(Debug, Clone, PartialEq, Eq, BorshSerialize, BorshDeserialize, ProtocolSchema)]
pub struct ChunkEndorsementV2 {
    // This is the part of the chunk endorsement that signed and included in the block header
    inner: ChunkEndorsementInnerV1,
    // This is the signature of the inner field, to be included in the block header
    signature: Signature,
    // This consists of the metadata for chunk endorsement used in validation
    metadata: ChunkEndorsementMetadata,
    // Metadata signature is used to validate that the metadata is produced by the expected validator
    metadata_signature: Signature,
}

impl ChunkEndorsementV2 {
    fn verify(&self, public_key: &PublicKey) -> bool {
        let inner = borsh::to_vec(&self.inner).unwrap();
        let metadata = borsh::to_vec(&self.metadata).unwrap();
        self.signature.verify(&inner, public_key)
            && self.metadata_signature.verify(&metadata, public_key)
    }
}
```
