Audit Report

## Title
Stale `rsETHPrice` Used in Withdrawal Amount Calculation Without Prior Oracle Update - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` and `instantWithdrawal` compute the user's asset entitlement via `getExpectedAssetAmount`, which reads `lrtOracle.rsETHPrice()` — a cached storage variable in `LRTOracle` that is only updated when `updateRSETHPrice()` is explicitly called. Neither withdrawal function triggers this update before computing the payout, so any accrued yield since the last oracle update is not reflected in the amount delivered to the withdrawing user.

## Finding Description
`LRTOracle.rsETHPrice` is a plain storage variable declared at line 28 of `contracts/LRTOracle.sol`. It is written only inside `_updateRsETHPrice()`, which is invoked either by the public `updateRSETHPrice()` (line 87–89) or by the manager-only `updateRSETHPriceAsManager()` (line 94–96). Neither of these is called atomically within the withdrawal path.

In `initiateWithdrawal` (line 168, `contracts/LRTWithdrawalManager.sol`):
```solidity
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
```

In `instantWithdrawal` (line 228, `contracts/LRTWithdrawalManager.sol`):
```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
```

`getExpectedAssetAmount` (lines 580–594) computes:
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

Because `rsETHPrice` reflects the state at the last `updateRSETHPrice()` call and EigenLayer staking rewards / LST appreciation accrue continuously, the stored price is systematically lower than the true current rate between oracle updates. The `expectedAssetAmount` locked in at initiation time is what the user ultimately receives upon `completeWithdrawal`, with no subsequent recalculation. No existing guard in either function checks price freshness or forces an update.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

When `rsETHPrice` is stale (below the true current rate), users who call `initiateWithdrawal` or `instantWithdrawal` receive fewer underlying assets than the fair current value of their rsETH. The shortfall (accrued yield since the last oracle update) remains in the protocol, benefiting remaining rsETH holders rather than the withdrawing user. The protocol itself does not lose funds, but the withdrawing user is systematically under-compensated relative to the true rsETH/ETH exchange rate at the time of their request.

## Likelihood Explanation
**Medium.** `updateRSETHPrice()` is a public function that must be called externally; it is not invoked atomically inside the withdrawal path. The protocol relies on off-chain keepers or manual calls to keep `rsETHPrice` fresh. Any gap between the last oracle update and a user's withdrawal call results in the stale-price condition. Given that staking rewards accrue continuously and oracle updates are periodic (not per-block), this condition is the norm rather than the exception. Any unprivileged user can trigger `initiateWithdrawal` or `instantWithdrawal` at any time without calling `updateRSETHPrice()` first.

## Recommendation
Call `updateRSETHPrice()` (or its internal equivalent `_updateRsETHPrice()`) at the beginning of `initiateWithdrawal` and `instantWithdrawal`, before `getExpectedAssetAmount` is invoked:

```solidity
function initiateWithdrawal(address asset, uint256 rsETHUnstaked, string calldata referralId) external ... {
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    ...
    uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
    ...
}
```

Apply the same fix to `instantWithdrawal`. This ensures the exchange rate used to compute the user's asset entitlement reflects all accrued yield up to the current block.

## Proof of Concept
1. At block N, `updateRSETHPrice()` is called. `rsETHPrice` is stored as `1.05e18`.
2. Staking rewards accrue over the next 24 hours. The true rsETH/ETH rate rises to `1.06e18`, but `updateRSETHPrice()` has not been called again.
3. A user holds 100 rsETH and calls `initiateWithdrawal(stETH, 100e18, "")`.
4. `getExpectedAssetAmount` computes: `100e18 * 1.05e18 / stETHPrice`. The user's `expectedAssetAmount` is locked in using the stale `1.05e18` rate.
5. Had `updateRSETHPrice()` been called first, the rate would be `1.06e18` and the user would receive ~0.95% more stETH.
6. The user completes the withdrawal and receives the under-valued amount.

**Foundry fork test plan:** Fork mainnet, advance time by 24 hours without calling `updateRSETHPrice()`, record `rsETHPrice`, call `updateRSETHPrice()` and record the new price, then demonstrate that `getExpectedAssetAmount` returns a lower value using the pre-update price than the post-update price for the same `rsETHUnstaked` input.
