### Title
Untrusted Peer Can Corrupt Wallet Coin Records via Unvalidated Reorged Coin States — (`File: chia/wallet/wallet_node.py`)

### Summary
`add_states_from_peer` in `chia/wallet/wallet_node.py` unconditionally skips Merkle-proof validation for any `CoinState` whose `created_height` is `None` (classified as a "reorged" state), even when the peer is untrusted. An unprivileged attacker operating a malicious full-node peer can send crafted `CoinStateUpdate` messages containing fake reorg notifications for coins that are actually confirmed on-chain, causing the wallet to corrupt its local coin records and sync state without any cryptographic verification.

### Finding Description

In `add_states_from_peer`, incoming coin states are split into two buckets:

```python
for coin_state in items:
    if coin_state.created_height is None:
        reorged_coin_states.append(coin_state)
    else:
        updated_coin_states.append(coin_state)
``` [1](#0-0) 

The `updated_coin_states` bucket goes through `validate_and_add`, which calls `_collect_valid_states` → `validate_received_state_from_peer`. That function performs full Merkle-proof validation: it requests the header block, verifies the additions/removals root, and checks block inclusion against the weight proof.

The `reorged_coin_states` bucket is applied **directly** to `add_coin_states` with no such validation:

```python
# Reorged coin states don't require any validation in untrusted mode, so we can just always apply them upfront
# instead of adding them to the race cache in untrusted mode.
for batch in to_batches(reorged_coin_states, chunk_size):
    self.log.info(f"Process reorged states: ({len(batch.entries)} / {len(reorged_coin_states)})")
    if not await self.wallet_state_manager.add_coin_states(batch.entries, peer, fork_height):
        self.log.debug("Processing reorged states failed")
        return False
``` [2](#0-1) 

The validation function itself confirms that when `confirmed_height is None` and the coin exists locally, it would enter `reorg_mode` and validate the block — but this path is never reached for reorged states from untrusted peers:

```python
if confirmed_height is None:
    if current is None:
        # Coin does not exist in local DB, so no need to do anything
        return False
    # This coin got reorged
    reorg_mode = True
    confirmed_height = current.confirmed_block_height
``` [3](#0-2) 

The entry point for an attacker is `state_update_received`, which calls `add_states_from_peer` directly for every `CoinStateUpdate` message received from any peer, trusted or not:

```python
async def state_update_received(self, request: CoinStateUpdate, peer: WSChiaConnection) -> None:
    ...
    async with self.wallet_state_manager.lock:
        await self.add_states_from_peer(
            request.items,
            peer,
            request.fork_height,
            request.height,
        )
``` [4](#0-3) 

### Impact Explanation

An attacker who sends `CoinState(victim_coin, spent_height=None, created_height=None)` for any coin the victim's wallet has confirmed locally causes `_add_coin_states` to process it as a reorg of that coin — rolling back the coin record, corrupting the wallet's balance view, and potentially preventing the wallet from constructing valid spend bundles for coins it actually owns on-chain. If the attacker continuously sends such messages, the wallet sync state remains corrupted for as long as the attacker is connected, constituting a long-lived inability to use confirmed coins.

This maps to the allowed High impact: **Corruption of wallet sync state with direct security impact** and **long-lived inability for wallets to process valid spend bundles under normal network assumptions**.

### Likelihood Explanation

Any peer that connects to the wallet's full-node port (or to which the wallet connects) can send `CoinStateUpdate` protocol messages. No keys, admin access, or cryptographic break is required. The wallet actively connects to peers during sync, making this reachable by any attacker who can position themselves as a peer.

### Recommendation

Apply `validate_received_state_from_peer` to reorged coin states from untrusted peers before passing them to `add_coin_states`. The rationale in the comment ("reorged coin states don't require validation") is incorrect: while there is nothing to prove about a coin's *absence*, the wallet must verify that the coin was actually confirmed at the height it believes before accepting a claim that it was reorged out. Specifically, the wallet should verify the coin's original creation block inclusion before processing any reorg notification for it from an untrusted peer.

### Proof of Concept

1. Attacker runs a malicious full-node process that speaks the Chia peer protocol.
2. Victim's wallet connects to the attacker's node (or attacker connects to victim's open port).
3. Attacker observes (or guesses) a coin ID that the victim's wallet has confirmed locally — e.g., by watching the mempool or the victim's puzzle hashes.
4. Attacker sends a `CoinStateUpdate` message containing `CoinState(victim_coin, spent_height=None, created_height=None)`.
5. `state_update_received` → `add_states_from_peer` classifies this as a reorged state (line 1036) and calls `add_coin_states` directly, bypassing `validate_received_state_from_peer`.
6. `_add_coin_states` processes the coin as reorged, rolling back the wallet's local record for that coin.
7. The victim's wallet now shows an incorrect balance and cannot spend the affected coin until a full resync. [5](#0-4) [4](#0-3) [6](#0-5)

    async def validate_received_state_from_peer(
        self,
        coin_state: CoinState,
        peer: WSChiaConnection,
        peer_request_cache: PeerRequestCache,
        fork_height: uint32 | None,
    ) -> bool:
        """
        Returns True if the coin_state is valid and included in the blockchain proved by the weight proof.
        """
        if peer.closed:
            return False
        # Only use the cache if we are talking about states before the fork point. If we are evaluating something
        # in a reorg, we cannot use the cache, since we don't know if it's actually in the new chain after the reorg.
        if can_use_peer_request_cache(coin_state, peer_request_cache, fork_height):
            return True

        spent_height: uint32 | None = None if coin_state.spent_height is None else uint32(coin_state.spent_height)
        confirmed_height: uint32 | None = (
            None if coin_state.created_height is None else uint32(coin_state.created_height)
        )
        current = await self.wallet_state_manager.coin_store.get_coin_record(coin_state.coin.name())
        # if remote state is same as current local state we skip validation

        # CoinRecord unspent = height 0, coin state = None. We adjust for comparison below
        current_spent_height = None
        if current is not None and current.spent_block_height != 0:
            current_spent_height = current.spent_block_height

        # Same as current state, nothing to do
        if (
            current is not None
            and current_spent_height == spent_height
            and current.confirmed_block_height == confirmed_height
        ):
            peer_request_cache.add_to_states_validated(coin_state)
            return True

        reorg_mode = False

        # If coin was removed from the blockchain
        if confirmed_height is None:
            if current is None:
                # Coin does not exist in local DB, so no need to do anything
                return False
            # This coin got reorged
            reorg_mode = True
            confirmed_height = current.confirmed_block_height
