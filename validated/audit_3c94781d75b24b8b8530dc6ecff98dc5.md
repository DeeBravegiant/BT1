### Title
Missing Cryptographic Output Verification for `AppPublicKey` (Legacy) CKD Variant Allows Single Byzantine Participant to Forge Derived Key Output ŌĆö (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs a BLS12-381 pairing-based output check (`ckd_output_check`) only for the `AppPublicKeyPV` branch. The `AppPublicKey` (legacy) branch is an empty match arm with no cryptographic binding. A single attested MPC participant can call `respond_ckd` with arbitrary `big_y`/`big_c` bytes for any pending `AppPublicKey` request, and the contract will accept and deliver the forged output to the victim caller ŌĆö without threshold agreement and without any cryptographic proof that the response is bound to the master secret.

---

### Finding Description

In `respond_ckd`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ŌåÉ no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) ┬Ę e(hash_point, public_key)`, which cryptographically binds the response to both the master secret key and the app's public key. [2](#0-1) 

This check is **structurally impossible** for `AppPublicKey` because that variant only carries a G1 point (`pk1`), not the G2 counterpart (`pk2`) required for the pairing. The `ckd_output_check` signature requires `&dtos::CKDAppPublicKeyPV` (which has both `pk1` and `pk2`). [3](#0-2) 

The result is that for any `AppPublicKey` request, the contract imposes **zero cryptographic constraint** on the response bytes. The only gate is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a single attested participant ŌĆö not a threshold. [4](#0-3) 

`resolve_yields_for` then immediately delivers whatever bytes were passed to all waiting yield-resume promises: [5](#0-4) 

The existing unit test at line 3424ŌĆō3427 already demonstrates this: it passes `[1u8; 48]` / `[2u8; 48]` (invalid BLS points) as the response for an `AppPublicKey` request, and the test passes without error. [6](#0-5) 

---

### Impact Explanation

A single Byzantine attested participant can race honest nodes to call `respond_ckd` with a fabricated `(big_y, big_c)` for any pending `AppPublicKey` CKD request. The victim receives a forged ElGamal ciphertext. When they decrypt it with their private key, they obtain a random G1 point instead of the correct derived key. Any funds or protocol state controlled by the derived key are then inaccessible or misdirected. This is **unauthorized confidential key derivation output without the required threshold participant authorization**.

---

### Likelihood Explanation

- The attacker must be an attested MPC participant (requires passing TEE attestation), which is a meaningful barrier.
- However, only **one** such participant needs to be Byzantine ŌĆö well below the signing threshold.
- The attack is a simple race: submit `respond_ckd` with arbitrary bytes before honest nodes respond. On a congested network or with a fast attacker, this is feasible.
- The `AppPublicKey` variant is still live and accepted by the contract (it is described as "legacy" but not gated or deprecated).
- The victim cannot detect the forgery because the `AppPublicKey` scheme is "privately verifiable" ŌĆö only the holder of the app private key can check the output, and the forged output decrypts to a plausible-looking (but wrong) G1 point.

---

### Recommendation

Since the `AppPublicKey` variant structurally cannot support the same pairing-based output check (it lacks `pk2`), the options are:

1. **Deprecate and remove `AppPublicKey`**: Force all callers to use `AppPublicKeyPV`, which supports full on-chain output verification. Gate `request_app_private_key` to reject `AppPublicKey` submissions.
2. **Add threshold enforcement at the contract level for `AppPublicKey` responses**: Require a quorum of attested participants to submit matching `(big_y, big_c)` bytes before resolving the yield. This is architecturally heavier but preserves backward compatibility.
3. **Document the weaker trust model explicitly**: If the variant is intentionally retained with weaker guarantees, document that `AppPublicKey` relies entirely on off-chain threshold honesty and TEE integrity, and that the contract provides no on-chain cryptographic binding for this variant.

Option 1 is the cleanest fix given that `AppPublicKeyPV` already exists and provides the stronger guarantee.

---

### Proof of Concept

The existing unit test at `crates/contract/src/lib.rs:3403` already constitutes a proof of concept. It submits `CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] }` ŌĆö bytes that are not valid BLS12-381 G1 points and have no cryptographic relationship to the master secret ŌĆö for an `AppPublicKey` request, and `respond_ckd` returns `Ok(())`. [7](#0-6) 

A sandbox integration test confirming the forge path end-to-end would:
1. Submit `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(arbitrary_g1)`.
2. As an attested participant, call `respond_ckd(request, CKDResponse { big_y: random_bytes, big_c: random_bytes })`.
3. Assert the yield resolves successfully with the forged bytes.
4. Confirm `ckd_output_check` is never reached (it is only reachable via the `AppPublicKeyPV` branch).

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
