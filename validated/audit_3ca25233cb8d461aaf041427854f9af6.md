Audit Report

## Title
`RSETH._transfer` Override Missing `whenNotPaused` Guard Allows Token Transfers While Contract Is Paused — (File: contracts/RSETH.sol)

## Summary
`RSETH` inherits `PausableUpgradeable` and correctly gates `mint` and `burnFrom` with `whenNotPaused`, but its `_transfer` override applies only `_enforceNotBlocked` with no pause check. As a result, any rsETH holder can call `transfer` or `transferFrom` freely while the contract is paused, meaning the pause mechanism fails to deliver a complete halt of token movements. This maps to the **Low** impact: *contract fails to deliver promised returns, but doesn't lose value*.

## Finding Description
`RSETH` inherits `ERC20Upgradeable` and `PausableUpgradeable` directly — not `ERC20PausableUpgradeable` — and overrides `_transfer` as follows:

```solidity
// contracts/RSETH.sol L287-291
function _transfer(address from, address to, uint256 amount) internal override {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);   // no pause check
}
```

`mint` (L235) and `burnFrom` (L245) both carry `whenNotPaused`, establishing a clear protocol intent that the pause should halt token operations. However, `ERC20Upgradeable`'s `transfer` and `transferFrom` both route through `_transfer`, which has no `_requireNotPaused()` call. `ERC20PausableUpgradeable` closes this gap via `_beforeTokenTransfer` (L37-41 of the OZ extension), but `RSETH` does not inherit it and does not replicate the check.

Exploit path:
1. `PAUSER_ROLE` calls `RSETH.pause()` in response to a security incident.
2. `mint(...)` and `burnFrom(...)` now revert with `Pausable: paused`.
3. Any rsETH holder calls `RSETH.transfer(recipient, balance)` — **succeeds** because `_transfer` never checks `paused()`.
4. If the admin subsequently calls `blockUserTransfers([attacker])`, the attacker's original address holds zero rsETH; the block is ineffective.

## Impact Explanation
The pause mechanism is the protocol's emergency stop. Its inconsistent application — halting `mint`/`burnFrom` but not `transfer`/`transferFrom` — means the contract does not deliver the complete token-movement halt that the presence of `pause()` and `whenNotPaused` on other functions implies. This is a **Low** impact: *contract fails to deliver promised returns, but doesn't lose value*. No direct fund theft or permanent freeze is demonstrated; the harm is that the pause guarantee is weaker than expected, enabling token movements (including pre-emptive transfers to evade a subsequent `blockUserTransfers`) during an active pause.

## Likelihood Explanation
The condition is triggered any time `PAUSER_ROLE` exercises the pause — the exact scenario the role exists for. Once paused, any rsETH holder (no special access required) can call `transfer` or `transferFrom`. No front-running, complex setup, or victim mistake is needed; a single public call suffices.

## Recommendation
Add `_requireNotPaused()` to the `_transfer` override:

```solidity
function _transfer(address from, address to, uint256 amount) internal override {
    _requireNotPaused();       // add this line
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

Alternatively, replace `ERC20Upgradeable` with `ERC20PausableUpgradeable` in the inheritance chain, which enforces the pause check in `_beforeTokenTransfer` automatically.
