### Title
Direct Deposits Omitted from Value Conservation Check Enables Unbounded ADA Creation — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in the transaction body that credits ADA directly to registered staking accounts. However, the value conservation check (`consumed = produced`) does not include direct deposits on the `produced` side. Because the ENTITIES rule unconditionally applies direct deposits to account balances, a transaction can increase account balances by an arbitrary amount without any corresponding reduction in the UTxO, creating ADA from nothing.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — invalid state transition that violates preservation of value.

The Dijkstra era adds `directDeposits :: DirectDeposits` to the transaction body. During transaction processing, the `ENTITIES` rule applies these deposits to staking account balances: [1](#0-0) 

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

The only guard is that target accounts must be registered. No check verifies that the ADA for the deposits was deducted from the UTxO.

The value conservation check in the UTXO rule is: [2](#0-1) 

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
```

For `DijkstraEra`, the `EraUTxO` instance sets: [3](#0-2) 

```haskell
instance EraUTxO DijkstraEra where
  consumed = conwayConsumed        -- UTxO inputs + withdrawals + refunds
  getConsumedValue = getConsumedDijkstraValue
  getProducedValue = getProducedDijkstraValue
```

`dijkstraProducedValue` is: [4](#0-3) 

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody   -- outputs + fees + stake/pool/drep/proposal deposits
    <> foldMap'
      (getProducedValue pp isRegPoolId . view bodyTxL)
      (txBody ^. subTransactionsTxBodyL)
```

Neither `conwayConsumed` nor `dijkstraProducedValue` mentions `directDeposits`. Direct deposits are therefore invisible to the `consumed = produced` check.

**Accounting proof of ADA creation:**

Let `D` = total direct deposits in a transaction. The full-system ADA change is:

```
ΔUTXO   = outputs − inputs = −(fees + deposits − withdrawals − refunds)   [from consumed=produced]
ΔFees   = +fees
ΔDeposit pot = +deposits − refunds
ΔAccounts = −withdrawals + D

Total Δ = ΔUTXO + ΔFees + ΔDeposit pot + ΔAccounts
        = −(fees + deposits − withdrawals − refunds)
          + fees + (deposits − refunds) + (−withdrawals + D)
        = D
```

Every lovelace in `directDeposits` is net-new ADA injected into the system.

The same flaw applies to sub-transactions, whose direct deposits are processed in `dijkstraSubEntitiesTransition`: [5](#0-4) 

and whose produced value is also computed without direct deposits.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An unprivileged transaction sender can register a staking credential and then submit a Dijkstra-era transaction that includes an arbitrarily large `directDeposits` map targeting their own account. The `consumed = produced` check passes (direct deposits are absent from both sides), and the ENTITIES rule credits the full deposit amount to the attacker's account balance. The attacker can then withdraw those funds. There is no upper bound on the amount that can be created per transaction beyond the transaction size limit.

---

### Likelihood Explanation

**Medium.** The Dijkstra era is experimental and not yet deployed on mainnet. However, the entry path requires only a standard, unprivileged transaction — no special keys, governance majority, or external dependency. Any node operator or user who can submit a Dijkstra-era transaction can trigger this. Once the era is live, exploitation is straightforward for anyone who reads the ledger rules.

---

### Recommendation

Include the sum of `directDeposits` in the `produced` value calculation. Concretely, `dijkstraProducedValue` (and its sub-transaction counterpart) should add the total direct-deposit amount alongside fees and other deposits:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> inject (foldMap id . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
    <> foldMap'
         (getProducedValue pp isRegPoolId . view bodyTxL)
         (txBody ^. subTransactionsTxBodyL)
```

This ensures that the ADA credited to accounts via direct deposits must be explicitly funded by the transaction's UTxO inputs, preserving the global ADA supply invariant.

### Proof of Concept

1. Register staking credential `cred` and obtain `accountAddress`.
2. Construct a Dijkstra-era transaction:
   - `inputs`: one UTxO entry worth `Y` lovelace
   - `outputs`: one UTxO entry worth `Y − minFee` lovelace (pays only the minimum fee)
   - `fee`: `minFee`
   - `directDeposits`: `{ accountAddress → X }` for any `X > 0`
3. Submit the transaction. The `consumed = produced` check evaluates:
   - `consumed = Y`
   - `produced = (Y − minFee) + minFee = Y` ✓
4. The ENTITIES rule calls `applyDirectDeposits`, crediting `X` lovelace to `cred`'s account.
5. Withdraw the `X` lovelace. The attacker has created `X` lovelace from nothing.

Repeat with arbitrarily large `X` (bounded only by `maxTxSize`) to drain the system or inflate the supply.
