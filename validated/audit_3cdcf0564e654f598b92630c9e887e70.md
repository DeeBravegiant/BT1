### Title
Permanently Locked User Funds via Dust-Amount Transfer That Passes `init_transfer` But Always Reverts in `sign_transfer` - (File: near/omni-bridge/src/lib.rs)

### Summary

`init_transfer` accepts any token amount where `fee < amount` without verifying that the decimal-normalized net amount is non-zero. When a user bridges a small amount to a chain with fewer decimals, `normalize_amount` truncates the net amount to zero. Every subsequent `sign_transfer` call then reverts with `InvalidAmountToTransfer`. Because no cancel or refund path exists, the user's tokens are permanently locked in the bridge.

### Finding Description

`init_transfer` in `near/omni-bridge/src/lib.rs` validates only that `fee < amount`: [1](#0-0) 

It does not check whether `normalize_amount(amount_without_fee, decimals) > 0`. The normalization step — which converts the NEAR-side token amount to the destination chain's decimal precision — uses floor division: [2](#0-1) 

For a token with 24 decimals on NEAR bridging to an EVM chain where it has 6 decimals, `diff_decimals = 18` and any `amount_without_fee < 10^18` normalizes to zero.

The `sign_transfer` function — the only path to complete the transfer — performs this normalization and then hard-reverts if the result is zero: [3](#0-2) 

Because `sign_transfer` is `#[trusted_relayer]`-gated, no unprivileged party can call it, and no trusted relayer can succeed either. The transfer entry remains in `pending_transfers` indefinitely. There is no `cancel_transfer` or user-initiated refund function anywhere in the contract. The only removal paths are `sign_transfer_callback` (requires a successful MPC signing, which is blocked) and `claim_fee_callback` (requires the transfer to already be finalized on the destination chain, which is also blocked). [4](#0-3) 

### Impact Explanation

**Critical — Irreversible fund lock.** A user who initiates a small-amount transfer has their tokens locked in the bridge with no recovery path. The protocol accepts the tokens via `ft_transfer_call` → `init_transfer`, stores the transfer in `pending_transfers`, but can never complete or cancel it. The locked tokens are permanently unclaimable by the user.

### Likelihood Explanation

**Medium.** The condition is triggered whenever:
1. A token is registered with a large decimal gap between NEAR and the destination chain (e.g., 24 on NEAR, 6 on EVM — an 18-order-of-magnitude divisor).
2. A user transfers a net amount (after fee) below the divisor threshold.

Many real ERC-20 tokens use 6 or 8 decimals while NEAR NEP-141 tokens commonly use 18 or 24. Any user transferring less than 1 full destination-chain unit (e.g., less than `10^18` base units for a 24→6 decimal pair) triggers the lock. This is reachable by any unprivileged user through the public `ft_transfer_call` entry point.

### Recommendation

Add a normalization check inside `init_transfer` before locking tokens:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(addr) = token_address {
    if let Some(decimals) = self.token_decimals.get(&addr) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

Additionally, implement a user-callable `cancel_transfer` that refunds locked tokens for transfers that have not yet been signed, guarded by a timeout or explicit user invocation.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 6` (18-order gap).
2. User calls `ft_transfer_call` with `amount = 10^17` (below the `10^18` normalization threshold) and `fee = 0`.
3. `init_transfer` passes: `fee (0) < amount (10^17)` ✓. Tokens are locked. Transfer stored in `pending_transfers`.
4. Trusted relayer calls `sign_transfer` for this transfer ID.
5. `normalize_amount(10^17, {decimals:6, origin_decimals:24})` = `10^17 / 10^18` = `0`.
6. `require!(amount_to_transfer > 0, ...)` reverts with `InvalidAmountToTransfer`.
7. Transfer remains in `pending_transfers`. No cancel function exists. User's `10^17` base units are permanently locked. [5](#0-4) [3](#0-2) [6](#0-5)

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
