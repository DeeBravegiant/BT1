Audit Report

## Title
Stale `rsETHPrice` Read in Deposit Minting Allows Theft of Unclaimed Yield — (File: `contracts/LRTDepositPool.sol`)

## Summary
`depositETH` and `depositAsset` compute the rsETH mint amount using `lrtOracle.rsETHPrice()`, a cached state variable in `LRTOracle` that is only written when `_updateRsETHPrice()` is explicitly invoked. Neither deposit function calls `updateRSETHPrice()` before minting. When staking rewards have accrued since the last price update, the stored price is lower than the true TVL-derived rate, causing depositors to receive more rsETH than they are entitled to, diluting existing holders' accrued yield.

## Finding Description
`getRsETHAmountToMint` in `LRTDepositPool.sol` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` is a plain storage variable written only inside `_updateRsETHPrice()`: [2](#0-1) [3](#0-2) 

`depositETH` and `depositAsset` both call `_beforeDeposit` → `getRsETHAmountToMint` without ever invoking `updateRSETHPrice()` first: [4](#0-3) [5](#0-4) 

`_beforeDeposit` is declared `private view`, so it structurally cannot trigger a state update; the non-view deposit functions are the only place where `updateRSETHPrice()` could be called before minting, and they do not do so: [6](#0-5) 

`updateRSETHPrice()` is public but contains a guard that reverts for non-managers when the price increase exceeds `pricePercentageLimit`: [7](#0-6) 

This guard can prevent anyone except a manager from refreshing the price, extending the staleness window beyond normal keeper cycles. During any staleness window, a depositor calling `depositETH{value: A}()` receives `A / P_stale` rsETH instead of the correct `A / P_true` (where `P_stale < P_true`). The excess rsETH represents a claim on yield that accrued to existing holders between the last price update and the deposit.

## Impact Explanation
**High — Theft of unclaimed yield.** The yield that accrued between the last `updateRSETHPrice()` call and the deposit belongs to existing rsETH holders. By depositing while the price is stale, an attacker mints excess rsETH backed by that accrued yield. After the price is updated to reflect the true TVL, the attacker's excess rsETH redeems for more underlying assets than were deposited, extracting value from existing holders. The magnitude scales linearly with deposit size and the duration of price staleness, and the attack is repeatable every keeper cycle.

## Likelihood Explanation
`updateRSETHPrice()` is public but relies entirely on off-chain keepers for regular invocation. Any gap between keeper calls (routine in practice) creates an exploitable window. The `pricePercentageLimit` guard can additionally cause `updateRSETHPrice()` to revert for non-managers when the price increase is large, potentially extending the staleness window significantly: [8](#0-7) 

No special privileges are required; any external account can call `depositETH` or `depositAsset`. The attacker only needs to observe that the on-chain `rsETHPrice` lags behind the true TVL-derived rate and deposit before the next keeper update.

## Recommendation
At the start of `depositETH` and `depositAsset`, call `updateRSETHPrice()` (or an internal equivalent that bypasses the non-manager revert) before computing the mint amount. This ensures `rsETHPrice` reflects the current TVL before any rsETH is issued. Alternatively, compute the mint amount on-the-fly from the live TVL rather than from the cached `rsETHPrice` storage variable.

## Proof of Concept
1. At block T₀, a keeper calls `updateRSETHPrice()`; `rsETHPrice` is set to P₀ = 1.01e18.
2. Staking rewards accrue. At block T₁ (e.g., 24 hours later), the true TVL-derived rate is P₁ = 1.012e18, but `rsETHPrice` still stores P₀.
3. If `pricePercentageLimit` is set and `(P₁ - P₀)/P₀ > limit`, a non-manager call to `updateRSETHPrice()` reverts with `PriceAboveDailyThreshold`, extending the staleness window.
4. Attacker calls `depositETH{value: 1000 ether}(minRSETH, "")`.
5. `getRsETHAmountToMint` returns `1000e18 * 1e18 / P₀ ≈ 990.099e18` rsETH instead of the correct `1000e18 * 1e18 / P₁ ≈ 988.142e18` rsETH — an excess of ≈ 1.957 rsETH.
6. Manager (or keeper) calls `updateRSETHPrice()`; `rsETHPrice` is updated to reflect the true TVL divided by the now-inflated rsETH supply, permanently diluting existing holders.
7. Attacker initiates withdrawal; their excess rsETH redeems for more ETH than deposited, extracting yield from existing holders.
8. **Foundry fork test**: Fork mainnet, snapshot `rsETHPrice` and TVL, advance time by 24 hours without calling `updateRSETHPrice()`, call `depositETH` with a large value, assert that `rsethAmountToMint > amount * 1e18 / trueTVLPrice`, then call `updateRSETHPrice()` and assert that existing holders' rsETH/ETH redemption rate decreased relative to the pre-deposit snapshot.
