### Title
Detached `swap_migrated_token` Promise Causes Irreversible Token Loss on Mint Failure — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `ft_on_transfer`, the `SwapMigratedToken` branch immediately returns `U128(0)` — consuming all user tokens from the NEP-141 perspective — while detaching the swap promise with no callback. If the `mint` call on the new token contract fails for any reason, the old tokens are burned from the bridge's balance but new tokens are never credited to the user. There is no admin recovery path.

---

### Finding Description

`ft_on_transfer` handles `BridgeOnTransferMsg::SwapMigratedToken` as follows:

```rust
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();
    PromiseOrPromiseIndexOrValue::Value(U128(0))
}
``` [1](#0-0) 

The function immediately returns `U128(0)`, signaling to the NEP-141 token contract that all transferred tokens are consumed (no refund). The swap promise is `.detach()`-ed — its result is never observed.

`swap_migrated_token` chains two cross-contract calls:

```rust
fn swap_migrated_token(&mut self, sender_id: AccountId, old_token: AccountId, amount: U128) -> Promise {
    let new_token = self.migrated_tokens.get(&old_token)
        .near_expect(BridgeError::TokenNotMigrated);
    let burn = ext_token::ext(old_token).burn(amount);
    let mint = ext_token::ext(new_token).mint(sender_id, amount, None);
    burn.and(mint)
}
``` [2](#0-1) 

`burn.and(mint)` schedules both calls in parallel. If `mint` fails (e.g., the recipient has no storage registration on the new token contract, the new token is paused, or the bridge lacks minting authority), the old tokens are already burned from the bridge's balance but new tokens are never minted. Since the promise is detached, the failure is silently discarded.

`migrate_deployed_token` only registers the bridge's own storage on the new token — it does not register storage for existing token holders: [3](#0-2) 

So any user who has not independently registered storage on the new token contract will trigger a `mint` failure. There is no callback, no retry, and no admin recovery path — the old tokens no longer exist.

---

### Impact Explanation

**Irreversible fund lock — permanently unclaimable user value in bridge token migration flows.**

Any user holding old (migrated) tokens who calls `ft_transfer_call` with `msg: "SwapMigratedToken"` is exposed to permanent token loss if the new token's `mint` call fails. The bridge has no mechanism to detect the failure or compensate the user. The old tokens are destroyed; the new tokens were never created. No admin operation can recover the funds because the old token supply is gone and the new token was never instructed to mint.

---

### Likelihood Explanation

**Medium.** The failure requires the new token contract's `mint` to fail. Realistic triggers include:

1. **Storage not registered** — NEP-141 requires recipients to have storage registered before receiving tokens. `migrate_deployed_token` only registers the bridge's own storage on the new token, not users'. Any user who has not separately called `storage_deposit` on the new token will cause `mint` to fail.
2. **New token contract paused** — if the new token is paused at migration time, `mint` fails silently.
3. **Bridge not authorized as minter** — if the new token contract's `mint` function requires a specific role and the bridge was not granted it, all swaps fail.

The `SwapMigratedToken` message is a public protocol input accessible to any user holding old tokens via `ft_transfer_call`.

---

### Recommendation

Replace the detached promise with a callback that handles failure:

1. Add a `swap_migrated_token_callback` that checks whether `mint` succeeded.
2. On failure, re-mint the old tokens back to the user (or store the amount in a recovery mapping for admin-assisted refund).
3. Alternatively, use a sequential approach: first verify the recipient has storage on the new token (via `storage_balance_of`), then proceed with burn+mint only if the check passes — otherwise refund immediately.

---

### Proof of Concept

1. DAO calls `migrate_deployed_token(origin_chain, old_token, new_token)` — sets up migration mapping.
2. User holds `amount` of `old_token` but has **not** registered storage on `new_token`.
3. User calls `old_token.ft_transfer_call(bridge, amount, '"SwapMigratedToken"')`.
4. Bridge's `ft_on_transfer` is invoked; it calls `swap_migrated_token` detached and returns `U128(0)`.
5. NEP-141 considers all tokens consumed; bridge now holds `amount` of `old_token`.
6. Bridge calls `burn(old_token, amount)` — succeeds; old tokens destroyed from bridge's balance.
7. Bridge calls `mint(new_token, user, amount)` — **fails** (user not registered on `new_token`).
8. Failure is silently discarded (detached promise).
9. User has 0 old tokens, 0 new tokens. No recovery path exists.
