### Title
Unchecked Multiplication Overflow in `denormalize_amount` Causes Permanent Fund Lock During Transfer Finalization - (File: `near/omni-bridge/src/lib.rs`)

### Summary
`denormalize_amount` performs a bare `u128` multiplication with no overflow guard. For tokens whose `origin_decimals` exceeds the bridge's normalized `decimals`, a sufficiently large (but valid) transfer amount causes the multiplication to overflow, panicking the NEAR runtime and permanently locking the user's already-committed source-chain funds.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}
``` [1](#0-0) 

`Decimals` stores two `u8` fields:

```rust
pub struct Decimals {
    pub decimals: u8,
    pub origin_decimals: u8,
}
``` [2](#0-1) 

When `origin_decimals > decimals` (e.g., a token with 24 origin decimals normalized to 18 on NEAR, giving `diff_decimals = 6`), the multiplier is `10^6`. Any `amount` larger than `u128::MAX / 10^6` (≈ 3.4 × 10^32) causes the multiplication to overflow.

This function is called unconditionally in `fin_transfer_callback` on every inbound cross-chain transfer:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
...
fee: Self::denormalize_fee(&init_transfer.fee, decimals),
``` [3](#0-2) 

It is also called in `fast_fin_transfer`:

```rust
let denormalized_amount =
    Self::denormalize_amount(fast_fin_transfer_msg.amount.0, decimals);
let denormalized_fee = Self::denormalize_fee(&fast_fin_transfer_msg.fee, decimals);
``` [4](#0-3) 

And in `claim_fee_callback`:

```rust
let denormalized_amount = Self::denormalize_amount(
    fin_transfer.amount.0,
    self.token_decimals
        .get(&token_address)
        .near_expect(BridgeError::TokenDecimalsNotFound),
);
``` [5](#0-4) 

NEAR contracts compile with `overflow-checks = true` by default, so the overflow produces a runtime panic rather than silent wrapping.

### Impact Explanation

The user has already locked funds in the source-chain bridge contract before `fin_transfer` is called. When `fin_transfer_callback` panics due to overflow, the NEAR-side finalization is permanently blocked. There is no recovery path: the source-chain lock cannot be reversed by the user, and the NEAR callback will always panic for that transfer. This constitutes an **irreversible fund lock** of the user's bridged assets.

Additionally, `denormalize_fee` delegates to `denormalize_amount`, so an overflowing fee value in the same transfer also triggers the same panic path.

### Likelihood Explanation

Any token registered with `origin_decimals > decimals` (a common configuration for high-precision tokens) is affected. The threshold for overflow depends on `diff_decimals`:

| diff_decimals | Overflow threshold (amount) |
|---|---|
| 6 | > ~3.4 × 10^32 |
| 12 | > ~3.4 × 10^26 |
| 18 | > ~3.4 × 10^20 |

A user transferring a large but individually valid `u128` amount on the source chain (e.g., a whale moving a large position in a high-decimal token) will trigger this. No privileged access is required — the attacker only needs to initiate a transfer on the source chain with a sufficiently large amount.

### Recommendation

Replace the bare multiplication with a checked variant and propagate the error:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount
        .checked_mul(10_u128.pow(diff_decimals))
        .unwrap_or_else(|| env::panic_str("denormalize_amount: overflow"))
}
```

Alternatively, validate at token registration time that `origin_decimals - decimals` is bounded such that `u128::MAX / 10^diff_decimals` is a meaningful upper limit, and enforce that limit on incoming transfer amounts before calling `denormalize_amount`.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (`diff_decimals = 6`).
2. On the source chain, initiate a transfer of `amount = 2^127` (fits in `u128`, ≈ 1.7 × 10^38).
3. Relayer submits the proof via `fin_transfer`.
4. `fin_transfer_callback` calls `denormalize_amount(2^127, Decimals { decimals: 18, origin_decimals: 24 })`.
5. Computes `2^127 * 10^6`, which exceeds `u128::MAX` (≈ 3.4 × 10^38).
6. NEAR runtime panics with an arithmetic overflow, the callback fails, and the user's source-chain funds are permanently locked.
