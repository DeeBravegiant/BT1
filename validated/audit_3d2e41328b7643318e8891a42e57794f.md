Audit Report

## Title
Undistributed Rewards Permanently Locked When All Stakers Exit During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool` distributes `rewardsToken` over a fixed period set by `notifyRewardAmount`. If `totalKernelStaked` reaches zero mid-period, `rewardPerToken()` freezes accumulation and the remaining reward tokens have no exit path. The contract itself acknowledges this behavior in its NatSpec but relies solely on an operational assumption — not an on-chain enforcement — that stakers will remain throughout the period.

## Finding Description
`rewardPerToken()` short-circuits to `rewardPerTokenStored` whenever `totalKernelStaked == 0`:

```solidity
// L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    ...
}
```

`notifyRewardAmount` (L566-592) transfers reward tokens in and sets `rewardRate` and `finishAt`. It guards against starting a period with zero stakers (`if (totalKernelStaked == 0) revert NoStakedTokens()`), but provides no guard against stakers exiting after the period begins. Any staker may call `initiateWithdrawal` (L320-338) permissionlessly, decrementing `totalKernelStaked`. Once it reaches zero:

- `rewardPerToken()` stops advancing; no new rewards accrue to anyone.
- `getReward()` (L382-390) pays `rewards[msg.sender]`, which is zero for any address not currently staked.
- `notifyRewardAmount` cannot be called again until `finishAt` passes (`RewardDurationNotFinished`), and even then it only rolls the stranded balance into a new `rewardRate` — it does not recover the tokens to the admin.
- No `recoverERC20`, `sweep`, or emergency-withdrawal function exists anywhere in the contract.

The contract's own NatSpec (L18-22) explicitly acknowledges this: *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract."* The stated mitigation is purely operational ("ensuring there are always some tokens staked… for the entire duration of the reward period"), which is not enforced on-chain.

## Impact Explanation
Reward tokens transferred into `KernelDepositPool` for a distribution period are permanently frozen whenever the staked supply reaches zero before `finishAt`. This matches **Medium – Permanent freezing of unclaimed yield**. The tokens cannot be recovered by the admin, redistributed to future stakers, or claimed by any address.

## Likelihood Explanation
Any staker can call `initiateWithdrawal` permissionlessly at any time. A single large staker or a coordinated set exiting during an active reward window is a realistic, low-effort path. The `withdrawalDelay` (capped at 30 days, L35) only postpones the moment `totalKernelStaked` hits zero; it does not prevent it. The scenario requires no privileged access, no oracle manipulation, and no external protocol interaction.

## Recommendation
Add an admin-only recovery function gated on `block.timestamp > finishAt` that transfers any `rewardsToken` balance in excess of the sum of all outstanding `rewards[user]` balances back to the admin (or a designated treasury). Alternatively, track the timestamp at which `totalKernelStaked` first hit zero during a period and allow the admin to reclaim `rewardRate * (finishAt - zeroSupplyTimestamp)` after `finishAt`.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 7 days`; `rewardRate = 1_000e18 / 7 days ≈ 1653e12/s`.
2. Alice is the sole staker (`totalKernelStaked = 100e18`).
3. After 1 day Alice calls `initiateWithdrawal(100e18)`; `totalKernelStaked = 0` (pending withdrawal delay).
4. After `withdrawalDelay` Alice calls `claimWithdrawal`; `totalKernelStaked` remains 0.
5. For the remaining ~6 days `rewardPerToken()` returns the frozen `rewardPerTokenStored`; ≈857e18 reward tokens accumulate in the contract with no claimant.
6. After `finishAt` the admin calls `notifyRewardAmount` again — this succeeds (stakers must be present), but the stranded ≈857e18 tokens are silently rolled into the new `rewardRate` rather than recovered, and if no stakers are present the call reverts with `NoStakedTokens`, leaving the tokens permanently locked.
