Audit Report

## Title
Stale `rsETHPrice` Allows Depositors to Capture Accrued Yield Before Oracle Update - (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is a public, permissionless function that applies the full accrued reward in a single lump-sum jump to `rsETHPrice`. Because `LRTDepositPool.getRsETHAmountToMint()` prices deposits against the stored (stale) `rsETHPrice`, an attacker can deposit at the stale lower price, call `updateRSETHPrice()` themselves, and sell the over-minted rsETH on secondary markets at the new higher price — extracting a share of the accrued yield that belongs to existing holders. No front-running is required; the attacker controls the entire sequence.

## Finding Description

`rsETHPrice` is a stored state variable updated only on explicit calls to `updateRSETHPrice()`: [1](#0-0) [2](#0-1) 

The function carries no access control — any address may call it. Between calls, EigenLayer staking rewards and LST appreciation cause the true backing per rsETH to grow, but `rsETHPrice` remains stale. When `updateRSETHPrice()` is finally called, the new price is computed from live `totalETHInProtocol` and written atomically: [3](#0-2) [4](#0-3) 

Deposits use the stored price directly: [5](#0-4) 

The `pricePercentageLimit` guard only fires when `newRsETHPrice > highestRsethPrice`: [6](#0-5) 

This means the guard is entirely absent during price recovery (when the current price is below the all-time high). Even when active, if `pricePercentageLimit == 0` the condition `pricePercentageLimit > 0` short-circuits to `false`, leaving no cap at all. Even a non-zero limit only bounds the per-update jump — it does not prevent the deposit-then-update sequence within the allowed window.

**Exploit sequence (no front-running required):**

1. Attacker observes that `rsETHPrice` is stale (rewards have accrued but not been reflected).
2. Attacker calls `depositETH()` with a large `D`, receiving `D·S/T` rsETH at the stale price `T/S`.
3. Attacker calls `updateRSETHPrice()` themselves in the same or next transaction.
4. New price ≈ `(T' + D)·T / (S·(T + D))` where `T'` is the true backing.
5. Attacker sells rsETH on a DEX (Curve, Uniswap) at the new price.
6. **Attacker profit = `D·(T'−T−protocolFee) / (T+D)`**, positive for any sub-100% protocol fee whenever rewards have accrued.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders earn staking rewards continuously, but those rewards are only reflected in `rsETHPrice` at discrete update points. An attacker who deposits immediately before an update and sells immediately after captures a proportional share of the accrued-but-unreflected yield. The profit scales linearly with deposit size and with the magnitude of the pending reward. For `D ≫ T`, the attacker captures nearly the entire accrued reward `T'−T`, directly diluting the yield of all existing holders. This matches the allowed impact: **Theft of unclaimed yield (High)**.

## Likelihood Explanation

The attack requires no special permissions — only capital. The attacker can monitor on-chain state (time since last `updateRSETHPrice()`, EigenLayer reward accrual via `_getTotalEthInProtocol()`) to estimate the pending price jump, then execute the deposit and price-update in consecutive transactions. No mempool front-running is needed; the attacker calls `updateRSETHPrice()` themselves after depositing. The attack is repeatable every reward cycle. The only costs are deposit capital and gas.

## Recommendation

1. **Adopt a gradual release model**: instead of applying the full accrued reward in one call, stream the price increase over a fixed window (e.g., 24 hours), analogous to Synthetix `StakingRewards`. This eliminates the exploitable lump-sum surge.
2. **Alternatively, add a deposit cooldown**: deposits made within a configurable window before a price update do not benefit from that update (commit-reveal or time-weighted entry price).
3. **Ensure `pricePercentageLimit` is always non-zero** and apply the cap symmetrically against `rsETHPrice` (not only `highestRsethPrice`) so that price-recovery jumps are also bounded.

## Proof of Concept

**Setup:** `rsETHPrice = 1.05e18` (stale), true backing per rsETH = `1.06e18` (rewards accrued). `pricePercentageLimit = 0`. Total ETH `T = 100_000e18`, rsETH supply `S = 100_000e18 / 1.05e18 ≈ 95_238 rsETH`.

**Step 1 — Attacker deposits 10,000 ETH:**
```
rsethAmountToMint = (10_000e18 * 1e18) / 1.05e18 ≈ 9_523.8 rsETH
```
Attacker receives 9,523.8 rsETH. True fair value at 1.06e18 would have been ≈ 9,433.9 rsETH — attacker received ~89.9 rsETH excess.

**Step 2 — Attacker calls `updateRSETHPrice()`:**
- `totalETHInProtocol` = `106_000e18 + 10_000e18 = 116_000e18` (true backing + deposit)
- `rsethSupply` = `95_238 + 9_523.8 = 104_761.8`
- `newRsETHPrice` = `116_000e18 / 104_761.8 ≈ 1.10727e18`

Wait — let me redo with cleaner numbers. `T = 100_000e18`, `T' = 106_000e18` (6% rewards), `S = 100_000e18 / 1.05e18 ≈ 95_238.1 rsETH`, `D = 10_000e18`.

- Attacker mints: `10_000e18 * 1e18 / 1.05e18 ≈ 9_523.81 rsETH`
- After update: `newRsETHPrice = (116_000e18) / (104_761.9) ≈ 1.10727e18`
- Attacker rsETH value: `9_523.81 * 1.10727e18 / 1e18 ≈ 10_545.4 ETH`
- **Profit ≈ 545 ETH** extracted from existing holders' accrued yield.
