Audit Report

## Title
Stale Cross-Chain Rate Used for wrsETH/rsETH Minting With No Staleness Guard - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally, with no check against `lastUpdated`. All four L2 pool contracts resolve their minting rate through this function. Because rsETH/ETH is monotonically increasing, any stale snapshot is always lower than the true rate, causing depositors to receive excess wrsETH or rsETH proportional to the staleness gap, constituting theft of unclaimed yield from existing holders.

## Finding Description
`CrossChainRateReceiver` stores the rsETH/ETH exchange rate in `rate` and records the update time in `lastUpdated` on every `lzReceive()` call:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L93-97
rate = _rate;
lastUpdated = block.timestamp;
```

However, `getRate()` returns `rate` unconditionally:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L102-105
function getRate() external view returns (uint256) {
    return rate;
}
```

`lastUpdated` is written but never read inside `getRate()`. There is no maximum-age threshold, no revert on staleness, and no fallback.

All four L2 pool contracts resolve the minting rate through `IOracle(rsETHOracle).getRate()`, which delegates to this function:

- `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee()` → `getRate()` → `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`
- `RSETHPoolV3WithNativeChainBridge.viewSwapRsETHAmountAndFee()` → same pattern
- `RSETHPoolV2ExternalBridge.viewSwapRsETHAmountAndFee()` → same pattern
- `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee()` → same pattern (transfers pre-held rsETH from pool balance)

The rate update mechanism (`RSETHMultiChainRateProvider.updateRate()`) is permissionless but requires the caller to supply native ETH to cover LayerZero messaging fees. During high gas periods or keeper inactivity, the on-chain `rate` can silently lag the true rsETH/ETH exchange rate for an unbounded period.

Since rsETH accrues staking rewards continuously, the true rate is always ≥ the last stored rate. Dividing by a stale (lower) rate yields a larger `rsETHAmount`, so every depositor during a stale window receives excess tokens at the protocol's expense.

The contrast with `ChainlinkOracleForRSETHPoolCollateral.getRate()` is instructive — that oracle enforces `answeredInRound < roundID` and `timestamp == 0` guards, but `CrossChainRateReceiver` has no equivalent protection.

## Impact Explanation
**High — Theft of unclaimed yield.**

For wrsETH-minting pools (`RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2ExternalBridge`): excess wrsETH is minted to the depositor. When redeemed at the true (higher) rate, the depositor extracts more ETH than they deposited. The surplus comes from diluting existing wrsETH holders' share of the underlying rsETH reserve.

For `RSETHPoolNoWrapper`: the pool directly transfers rsETH from its own balance at the stale rate, draining the pool's pre-held rsETH reserve.

In both cases, the excess represents yield that accrued to existing holders but is instead captured by the depositor. Accumulated over many deposits during a prolonged stale window, this constitutes material theft of unclaimed yield.

## Likelihood Explanation
**Medium.** The `updateRate()` call on `RSETHMultiChainRateProvider` is permissionless but fee-bearing; the caller must supply native ETH for LayerZero messaging. During high gas prices, network congestion, or keeper inactivity, updates can be delayed for hours or longer. The existence of `lastUpdated` in the contract confirms the design anticipates gaps between updates, yet no on-chain guard enforces a maximum gap. Any depositor can read `lastUpdated` on-chain, compute the staleness, and time a deposit to exploit the window. No special privileges are required — only a standard `deposit()` call.

## Recommendation
Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 24 hours, set by owner

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This mirrors the staleness-guard pattern already present in `ChainlinkOracleForRSETHPoolCollateral`. The `maxStaleness` value should be set conservatively (e.g., 24 hours) and adjustable by the owner to accommodate varying keeper cadences.

## Proof of Concept
1. At time T, `CrossChainRateReceiver.rate = 1.05e18` and `lastUpdated = T`.
2. Staking rewards accrue; the true rate rises to `1.06e18` by T+24h, but no keeper calls `updateRate()` (e.g., due to high gas costs).
3. At T+24h, Alice calls `deposit{value: 1 ether}()` on `RSETHPoolV3ExternalBridge`.
4. `viewSwapRsETHAmountAndFee(1e18)` calls `getRate()` → returns stale `1.05e18`.
5. `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.9524 wrsETH`; at the true rate `1.06e18` the correct amount is `≈ 0.9434 wrsETH`.
6. Alice receives ~0.009 excess wrsETH (~0.95% of deposit) at existing holders' expense.
7. Alice redeems wrsETH at the true rate, extracting more ETH than she deposited.

**Foundry fork test plan:**
- Fork an L2 where the pool is deployed.
- Warp `block.timestamp` forward by 24+ hours without triggering `lzReceive()`.
- Call `deposit{value: 1 ether}()` and record `rsETHAmount` received.
- Compare against `1e18 * 1e18 / trueRate` to quantify the excess.
