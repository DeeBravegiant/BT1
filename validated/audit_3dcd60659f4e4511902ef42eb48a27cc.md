### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Produces Unbacked Wrapped Supply — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge::initTransfer` records the caller-supplied `amount` in the cross-chain message/event without verifying the actual tokens received. For fee-on-transfer ERC-20 tokens the bridge custodies `amount − fee` but the NEAR side mints/releases the full `amount`, permanently breaking the 1:1 backing guarantee.

### Finding Description
The external report's root cause class is **asset/accounting divergence**: an operation assumes a nominal amount was transferred but the actual amount differs, and the recorded value uses the nominal figure. The identical pattern exists in `OmniBridge::initTransfer`.

For the non-bridge, non-custom-minter ERC-20 path the contract pulls tokens from the caller:

```solidity
// OmniBridge.sol lines 407-411
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← nominal, not verified against actual receipt
);
``` [1](#0-0) 

Immediately after, the same caller-supplied `amount` is forwarded to `initTransferExtension` and emitted in the `InitTransfer` event without any balance-before/after check:

```solidity
// OmniBridge.sol lines 415-436
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // ← nominal amount, not actual received
    ...
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // ← same nominal amount
    ...
);
``` [2](#0-1) 

`OmniBridgeWormhole::initTransferExtension` encodes this same `amount` verbatim into the Wormhole VAA payload that the NEAR bridge consumes:

```solidity
// OmniBridgeWormhole.sol lines 136-137
Borsh.encodeUint128(amount),   // ← nominal, not actual
``` [3](#0-2) 

The NEAR bridge processes this message and mints or releases `amount` tokens to the recipient. There is no mechanism on the NEAR side to detect that the EVM bridge only received `amount − fee`.

### Impact Explanation
Every `initTransfer` call with a fee-on-transfer token creates a deficit: the EVM bridge holds `amount − fee` but the NEAR-side supply grows by `amount`. The deficit is permanent and cumulative. When users later bridge back (NEAR → EVM), `finTransfer` attempts `safeTransfer(recipient, amount)` from the bridge's custody:

```solidity
// OmniBridge.sol lines 351-354
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [4](#0-3) 

Once the cumulative deficit exceeds the bridge's token balance, `safeTransfer` reverts, permanently freezing redemptions for later users — matching the **Critical/High** impact categories: backing guarantee broken and irreversible fund lock for users who cannot redeem.

### Likelihood Explanation
Fee-on-transfer tokens are a well-known ERC-20 variant (e.g., tokens with auto-liquidity or reflection mechanics). The `else` branch at line 406 is the general-purpose lock path for any ERC-20 that is neither a bridge token nor has a custom minter. Any such token registered by the admin (via `deployToken` or `addCustomToken`) and having a transfer fee is sufficient. The attacker is any unprivileged user who calls `initTransfer` with such a token — no special role or key is required.

### Recommendation
Measure the actual received amount using a balance-before/balance-after pattern and use that value in the event and cross-chain message:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived (cast to uint128) instead of amount in the event and extension call
```

Alternatively, explicitly disallow fee-on-transfer tokens by requiring `actualReceived == amount` and reverting otherwise.

### Proof of Concept
1. A fee-on-transfer ERC-20 token `FOT` (2% fee on every transfer) is registered in the bridge.
2. Alice calls `initTransfer(FOT, 1000, 0, nativeFee, "alice.near", "")`.
3. `safeTransferFrom` pulls 1000 FOT from Alice; the token contract deducts 2% → bridge receives **980 FOT**.
4. The event and Wormhole VAA encode `amount = 1000`.
5. NEAR bridge mints **1000 wrapped-FOT** to Alice on NEAR.
6. Deficit = 20 FOT per transfer, growing with each call.
7. After 50 such transfers the bridge holds 49,000 FOT but 50,000 wrapped-FOT exist on NEAR.
8. The 50th user to bridge back calls `finTransfer` for 1000 FOT; `safeTransfer` reverts because the bridge balance is exhausted — funds permanently locked.
