Audit Report

## Title
Unvalidated Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain Victim's Approved Tokens via Callback - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary
`MetricOmmPoolLiquidityAdder._addLiquidity` stores a caller-supplied `pool` address into transient pay context without any factory registry check, then calls `addLiquidity` on it. A malicious contract passed as `pool` can immediately re-enter `metricOmmModifyLiquidityCallback`, trivially pass the only caller-identity guard (`msg.sender == expectedPool`), supply attacker-controlled token addresses via `getImmutables()`, and cause the adder to transfer up to `maxAmountToken0`/`maxAmountToken1` of any token from the victim to the attacker.

## Finding Description
`_addLiquidity` at line 193 stores the unvalidated `pool` directly into transient context and calls `addLiquidity` on it:

```solidity
_setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
try IMetricOmmPoolActions(pool)
  .addLiquidity(positionOwner, salt, deltas, abi.encode(KIND_PAY), extensionData) ...
``` [1](#0-0) 

The callback's only caller-identity guard is:

```solidity
if (msg.sender != expectedPool) revert InvalidCallbackCaller(msg.sender, expectedPool);
``` [2](#0-1) 

Because `expectedPool` was set to the malicious contract, `msg.sender == expectedPool` is trivially satisfied. The callback then calls `IMetricOmmPool(msg.sender).getImmutables()` — fully attacker-controlled — to obtain token addresses, and executes:

```solidity
pay(token0, payer, msg.sender, amount0Delta);
pay(token1, payer, msg.sender, amount1Delta);
``` [3](#0-2) 

The amount cap check `amount0Delta > max0 || amount1Delta > max1` also passes when the attacker requests exactly the cap values the victim supplied. The contract's own NatSpec at lines 19–21 explicitly acknowledges this gap: [4](#0-3) 

By contrast, `MetricOmmSwapRouterBase._setNextCallbackContext` calls `_requireFactoryPool(pool)` before storing any pool in transient context, and `_requireExpectedCallbackCaller` additionally checks `FACTORY.isPool(caller)` in the callback: [5](#0-4) [6](#0-5) 

`MetricOmmPoolLiquidityAdder` has no equivalent check anywhere in its call path, and it holds no `FACTORY` immutable.

## Impact Explanation
Any user who has granted a token approval to `MetricOmmPoolLiquidityAdder` can have up to their full approved balance drained in a single transaction. The attacker controls both the token addresses (via `getImmutables()`) and the amounts (up to the victim-supplied caps). This is a direct, complete loss of user principal — a Critical/High impact under Sherlock thresholds.

## Likelihood Explanation
The trigger requires only that a victim call `addLiquidityExactShares` or `addLiquidityWeighted` with an attacker-supplied pool address. A malicious frontend, compromised UI, or social-engineering attack can present a fake pool address as legitimate. Users who have pre-approved the adder (a common gas-efficiency pattern) are permanently at risk until they revoke the approval. No privileged access is required by the attacker.

## Recommendation
Add a factory registry check in `_addLiquidity` before storing the pool in transient context, mirroring `MetricOmmSwapRouterBase`:

```solidity
function _addLiquidity(address pool, ...) internal ... {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

This requires adding a `FACTORY` immutable to `MetricOmmPoolLiquidityAdder`, exactly as `MetricOmmSwapRouterBase` does at line 20. [7](#0-6) 

## Proof of Concept
1. Attacker deploys `MaliciousPool` whose `addLiquidity` immediately calls back `metricOmmModifyLiquidityCallback(VICTIM_MAX0, 0, abi.encode(KIND_PAY))` and whose `getImmutables()` returns an attacker-chosen token address.
2. Victim (who has approved the adder for that token) calls `addLiquidityExactShares(address(maliciousPool), victim, 0, deltas, VICTIM_MAX0, 0, "")`.
3. `_addLiquidity` stores `maliciousPool` as `expectedPool` and calls `maliciousPool.addLiquidity(...)`.
4. `maliciousPool` re-enters `metricOmmModifyLiquidityCallback`; `msg.sender == expectedPool` ✓; `VICTIM_MAX0 <= max0` ✓.
5. Callback fetches attacker-controlled `token0` from `maliciousPool.getImmutables()` and executes `pay(token, victim, maliciousPool, VICTIM_MAX0)`.
6. Result: `VICTIM_MAX0` of the attacker-chosen token is transferred from the victim to `maliciousPool`. [8](#0-7)

      pay(token1, payer, msg.sender, amount1Delta);

  }
