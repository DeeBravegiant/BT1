Audit Report

## Title
Stale `rsETHPrice` Mixed With Live Asset Price in `getExpectedAssetAmount` Enables Theft of Depositor Funds — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager::getExpectedAssetAmount` computes withdrawal amounts by dividing the stored `lrtOracle.rsETHPrice()` by the live `lrtOracle.getAssetPrice(asset)`. Because `rsETHPrice` is a state variable updated only on explicit calls to `updateRSETHPrice()` or `updateRSETHPriceAsManager()`, a window exists between an LST price drop and the next price update during which the ratio is inflated. An unprivileged caller can exploit this window via `instantWithdrawal` to receive more underlying assets than their rsETH is worth, directly extracting principal from other depositors.

## Finding Description
`getExpectedAssetAmount` at line 593 of `LRTWithdrawalManager.sol` computes:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

`lrtOracle.rsETHPrice()` is a stored `uint256` state variable in `LRTOracle` (line 28). It is updated only when `_updateRsETHPrice()` is invoked — either via the public `updateRSETHPrice()` (line 87) or the manager-gated `updateRSETHPriceAsManager()` (line 94). `lrtOracle.getAssetPrice(asset)` (line 156–157) always reads the live Chainlink feed at call time.

The `instantWithdrawal` function (lines 212–253) calls `getExpectedAssetAmount` at line 228, immediately burns the caller's rsETH at line 229, and redeems the computed asset amount from the unstaking vault at line 235 — all in a single transaction with no subsequent price recalculation or guard on the output:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
...
unstakingVault.redeem(asset, assetAmountUnlocked);
```

When an LST depegs, `getAssetPrice(asset)` immediately reflects the lower price while `rsETHPrice` remains at the pre-depeg value. The ratio `rsETHPrice / getAssetPrice(asset)` is therefore larger than the true exchange rate, and `underlyingToReceive` is overstated. The only on-chain guard that could prevent this — the downside-protection pause in `_updateRsETHPrice()` (lines 270–281) — only fires when `updateRSETHPrice()` is actually called; if no one calls it before the attacker acts, the protocol remains unpaused with a stale price.

For the `initiateWithdrawal` path, `_calculatePayoutAmount` (lines 833–834) takes the minimum of the stored `expectedAssetAmount` and a recalculated value using `rsETHPrice` from `_createUnlockParams` (line 847), which also reads the same stored `lrtOracle.rsETHPrice()`. If `rsETHPrice` is still stale when `unlockQueue` is called, both values are inflated and the minimum provides no protection.

## Impact Explanation
**Critical — Direct theft of depositor principal.** The attacker surrenders rsETH whose current fair value (at the depegged rate) is less than the assets redeemed. The surplus comes directly from the backing of other depositors' rsETH. This is not unclaimed yield; it is principal held at-rest in the unstaking vault. The PoC demonstrates a concrete, quantified extraction of depositor funds in a single transaction.

## Likelihood Explanation
The preconditions are: (1) `isInstantWithdrawalEnabled[asset]` is `true` — a normal operational state set by the manager for supported LSTs; (2) a non-zero time window exists between an LST Chainlink price update and the next `updateRSETHPrice()` call. Because `updateRSETHPrice()` is not called atomically with every Chainlink heartbeat, this window is structurally guaranteed on every price move. An attacker monitoring mempool or Chainlink feed updates can act within the same block as the feed update, before any keeper or user calls `updateRSETHPrice()`. The attack is repeatable as long as the price remains stale and vault liquidity exists.

## Recommendation
Replace the stored `lrtOracle.rsETHPrice()` in `getExpectedAssetAmount` with a live computation of the rsETH/ETH rate using the same inputs as `_updateRsETHPrice` — specifically `totalETHInProtocol / rsETHSupply` — so that both the numerator and denominator of the withdrawal ratio are evaluated at the same point in time. Alternatively, require that `updateRSETHPrice()` be called atomically within the same transaction before any withdrawal amount is computed, or add a staleness check that reverts if `rsETHPrice` was last updated more than N blocks ago.

## Proof of Concept
1. Protocol state: 100 stETH in unstaking vault, 100 rsETH supply. `rsETHPrice = 1.0e18`, Chainlink stETH feed = `1.0e18`.
2. Chainlink stETH feed updates to `0.9e18`. `rsETHPrice` is **not** yet updated (no one has called `updateRSETHPrice()`).
3. Attacker holds 10 rsETH (fair value at depegged rate: ~9 ETH).
4. Attacker calls `instantWithdrawal(stETH, 10e18, "")`.
5. `getExpectedAssetAmount(stETH, 10e18)` returns `10e18 × 1.0e18 / 0.9e18 = 11.111e18`.
6. 10 rsETH is burned; 11.111 stETH is redeemed from the vault and transferred to the attacker.
7. Attacker receives 11.111 stETH (worth ~10 ETH at current prices) for rsETH worth only ~9 ETH — extracting ~1.111 stETH (~1 ETH) of principal from remaining depositors.

**Foundry fork test plan:** Fork mainnet, set `rsETHPrice` to `1e18` via storage slot, mock the stETH Chainlink feed to return `0.9e18`, call `instantWithdrawal` from an attacker address holding rsETH, and assert that the attacker's stETH balance increase exceeds the ETH value of the rsETH burned at the current asset price.
