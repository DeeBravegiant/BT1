### Title
Missing CKD Response Validation for `AppPublicKey` Variant Allows Single Byzantine Participant to Deliver Fabricated Key — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` validates the CKD output only for the `AppPublicKeyPV` (publicly verifiable) variant. For the legacy `AppPublicKey` (privately verifiable) variant, **no validation is performed on the response**. A single Byzantine participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary fabricated `CKDResponse` for any pending request that used the `AppPublicKey` format, and the contract will accept and deliver it to the user. This is the direct analog of the Parallel Finance `repay_borrow(amount=0)` issue: a missing input-validation guard allows a no-op or adversarial value to corrupt critical state without the required multi-party authorization.

---

### Finding Description

In `respond_ckd` the contract retrieves the BLS12-381 public key and then branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the encrypted output is consistent with the MPC network's BLS public key and the user's ephemeral key pair. For `AppPublicKey`, the arm is a bare `{}` — the response bytes are serialized and delivered to the caller with zero verification.

By contrast, both `respond` and `respond_verify_foreign_tx` unconditionally verify their cryptographic outputs before resolving yields:

```rust
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [2](#0-1) 

The `AppPublicKey` format is still documented and accepted as a supported input path: [3](#0-2) 

The contract requires only a single attested participant to call `respond_ckd` — there is no on-chain threshold check on the responder side: [4](#0-3)

### Impact Explanation

**Critical — unauthorized confidential key derivation output without the required participant authorization.**

A single Byzantine participant (one node, below the reconstruction threshold `t`) can:

1. Monitor the contract for any pending `CKDRequest` whose `app_public_key` is the `AppPublicKey` variant.
2. Construct an arbitrary `CKDResponse` — e.g., one that encrypts a key the attacker already knows.
3. Call `respond_ckd(request, fabricated_response)`. The contract accepts it, removes the pending entry, and delivers the fabricated bytes to the waiting user via `promise_yield_resume`.

The user's application receives a "derived private key" that is entirely attacker-controlled. Because the `AppPublicKey` format is privately verifiable only (the user cannot check the output against the MPC public key without the app secret), the user has no way to detect the substitution. The attacker now knows the user's derived key and can use it to decrypt data or forge signatures on the user's behalf.

---

### Likelihood Explanation

**Medium.** The attacker must be an attested MPC participant — a non-trivial prerequisite. However, the `AppPublicKey` (legacy) format is still accepted in production, and any single participant who turns Byzantine (or whose node is compromised) can exploit this without colluding with any other participant. The attack requires only watching the NEAR chain for pending CKD requests and submitting one transaction.

---

### Recommendation

Apply the same response-validation discipline used for `AppPublicKeyPV` to the `AppPublicKey` variant. Because the `AppPublicKey` format is privately verifiable (only the app holding the secret can check), on-chain validation is not possible in the same way. The correct mitigations are:

- **Short term:** Reject `AppPublicKey` requests in `respond_ckd` unless the caller can supply a zero-knowledge proof or the response is validated off-chain before submission. At minimum, add a comment and a `todo!`/`unimplemented!` guard so the missing check is not silently skipped.
- **Long term:** Migrate all callers to `AppPublicKeyPV`, which supports on-chain verification via `ckd_output_check`, and deprecate/remove the unverifiable `AppPublicKey` path from `respond_ckd`.

### Proof of Concept

```
1. Alice submits:
   request_app_private_key({
     derivation_path: "my-app",
     app_public_key: { "AppPublicKey": "bls12381g1:<alice_pk>" },
     domain_id: <ckd_domain>
   })
   with deposit=1 yoctoNEAR

2. The request is stored in pending_ckd_requests.

3. Mallory (a single attested participant) constructs:
   fabricated_response = CKDResponse { encrypted_key: <key_mallory_knows> }

4. Mallory calls:
   respond_ckd(request=<alice's request>, response=fabricated_response)

5. The contract executes:
   match &request.app_public_key {
       AppPublicKey(_) => {}   // no check — falls through
       ...
   }
   resolve_yields_for(...)     // delivers fabricated_response to Alice

6. Alice's promise resolves with Mallory's fabricated key.
   Mallory knows the plaintext of Alice's "confidential" derived key.
```

The `AppPublicKey` branch in `respond_ckd` is the exact structural analog of `repay_borrow(amount=0)`: a missing guard on an input variant allows an adversarial value to pass through and corrupt the output delivered to the end user, without the threshold of honest participants required by the protocol's security model.
