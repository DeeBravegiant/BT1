Audit Report

## Title
LP Principal Permanently Locked When USDC/USDT Blacklists Position Owner in `removeLiquidity` — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`LiquidityLib.removeLiquidity` unconditionally calls `safeTransfer` to `owner` for both pool tokens with no alternative recipient path. If `owner` is blacklisted by USDC or USDT, every withdrawal attempt reverts atomically, preserving shares in storage but making them permanently unredeemable. The `msg.sender == owner` guard in `MetricOmmPool.removeLiquidity` eliminates any workaround via a non-blacklisted caller, and no position-transfer or admin-rescue mechanism exists anywhere in the contract.

## Finding Description
In `LiquidityLib.removeLiquidity` (lines 242–247), after all bin accounting is updated and shares are decremented, the function unconditionally transfers tokens to `owner`:

```solidity
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
}
```

If either token is USDC or USDT and `owner` is on the issuer's blacklist, `safeTransfer` reverts. Because Solidity reverts are atomic, the share-burn and bin-balance updates at lines 210–214 also roll back — the LP's shares are restored in storage but can never be redeemed.

The entry point in `MetricOmmPool.removeLiquidity` (line 206) enforces:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
```

This eliminates any workaround via a proxy or helper contract. A grep across all `metric-core/contracts/**/*.sol` finds zero occurrences of `transferPosition`, `positionTransfer`, or `rescue`, confirming there is no position-transfer mechanism or admin recovery path. The extension hooks (`_beforeRemoveLiquidity`, `_afterRemoveLiquidity` in `ExtensionCalling.sol`) are called outside `LiquidityLib` and cannot redirect the token transfer that occurs inside the library call.

## Impact Explanation
An LP whose address is USDC/USDT-blacklisted loses their entire deposited principal permanently. The pool's bin accounting correctly records the shares, but no withdrawal can ever succeed. This is a direct, irrecoverable loss of user principal with no admin mitigation path, satisfying the "Critical/High direct loss of user principal" impact gate. USDC/USDT blacklisting is explicitly carved out of the non-standard ERC20 exclusion in the contest rules.

## Likelihood Explanation
Circle and Tether have blacklisted hundreds of addresses for OFAC compliance and exchange enforcement actions. The protocol explicitly deploys USDC pools on Base, Optimism, and Linea (per `smart-contracts-poc/script/js/config/base/feeds/default.json`). Any LP using an EOA that is later sanctioned — or any LP whose address is blacklisted after deposit — triggers this condition without any privileged or malicious setup. No attacker capability is required; the condition arises from normal regulatory action on a real-world token.

## Recommendation
Add an optional `recipient` parameter to `removeLiquidity` (separate from `owner`) so a blacklisted owner can redirect token output to a non-blacklisted address they control. Alternatively, implement a two-step withdrawal pattern that separates share burning from token claiming, allowing the owner to specify a recipient at claim time. The `msg.sender == owner` authorization check should remain on the share-burning step; only the token destination needs to be flexible.

## Proof of Concept
1. Alice (`0xAlice`) calls `addLiquidity` on a USDC/WETH pool; her position is keyed `(0xAlice, salt)` in `_positionBinShares`.
2. Circle blacklists `0xAlice` (OFAC compliance action).
3. Alice calls `MetricOmmPool.removeLiquidity(0xAlice, salt, deltas, "")`.
4. `msg.sender == owner` check passes (Alice is calling).
5. `LiquidityLib.removeLiquidity` burns shares and updates bin balances in storage.
6. `IERC20(USDC).safeTransfer(0xAlice, amount0Removed)` reverts — USDC blacklist check fails.
7. The entire transaction reverts; all storage changes roll back; shares are restored.
8. Alice cannot call from any other address (`msg.sender != owner` guard at `MetricOmmPool.sol:206`).
9. Alice's USDC principal is permanently locked in the pool with no recovery path.
