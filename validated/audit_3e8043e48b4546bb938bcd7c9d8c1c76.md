### Title
Fee-on-Transfer Token Accounting Discrepancy Drains Bridge Vault Reserves - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.initTransfer()` records the caller-supplied `amount` as the canonical bridged value and emits it in the `InitTransfer` event, but uses a plain `safeTransferFrom` that does not measure the actual tokens received. For any ERC-20 with a fee-on-transfer mechanism the vault receives `amount − fee_on_transfer` while the bridge commits to releasing the full `amount` on the destination chain, permanently undercollateralizing the vault by the fee amount on every such transfer.

---

### Finding Description

In `OmniBridge.initTransfer()`, the branch that handles plain ERC-20 tokens (i.e., tokens that are neither bridge-minted nor custom-minter-managed) executes:

```solidity
// OmniBridge.sol lines 406-411
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← requested amount, not measured received amount
);
```

Immediately after, the function passes the same caller-supplied `amount` to `initTransferExtension` and emits it in the `InitTransfer` event:

```solidity
// OmniBridge.sol lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← unverified, may exceed actual vault credit
    ...
);
```

The NEAR bridge observes this event and creates a `TransferMessage` for the full `amount`. When the transfer is later finalized back to EVM via `finTransfer()`, the bridge releases the full `payload.amount` from its vault:

```solidity
// OmniBridge.sol lines 351-354
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount   // ← full amount, but vault only holds amount − fee
);
```

The vault is therefore short by exactly `fee_on_transfer` per round-trip. There is no pre/post balance check, no `balanceOf` snapshot, and no mechanism to reconcile the discrepancy.

The same structural issue exists on the Solana side: `init_transfer.rs` uses `transfer_checked` with `payload.amount` directly into the vault without measuring the post-transfer vault balance, so Token-2022 tokens with the `TransferFeeConfig` extension produce the same undercollateralization.

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer ERC-20 causes the bridge vault to hold strictly fewer tokens than the amount committed on the destination chain. Repeated transfers accumulate this deficit. Eventually the vault cannot satisfy legitimate `finTransfer` redemptions, causing:

- **Irreversible fund lock / frozen redemption path**: later users who bridged standard (non-fee) tokens cannot withdraw because the vault balance has been silently depleted by fee-token transfers.
- **Backing guarantee violation**: the wrapped supply on the destination chain exceeds the locked collateral on the source chain, breaking the 1:1 peg.

This matches the allowed impact: *"Asset-identity, token-mapping, decimals, fee-routing, refund, or balance-accounting divergence that breaks backing guarantees or sends value to the wrong party."*

---

### Likelihood Explanation

The entry point is the fully public `initTransfer()` function — no role, whitelist check, or privileged access is required. Any user can call it with any ERC-20 address. Fee-on-transfer tokens exist in production (e.g., tokens with explicit transfer-tax logic, or USDT if its dormant fee switch is ever enabled). The attacker does not need to exploit a race condition or chain-level assumption; a single well-formed call is sufficient to begin draining the vault.

---

### Recommendation

1. **Measure actual received amount**: snapshot `balanceOf(address(this))` before and after `safeTransferFrom`, use the delta as the canonical `amount` for the event and downstream accounting.

```solidity
uint256 before = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualAmount = IERC20(tokenAddress).balanceOf(address(this)) - before;
// use actualAmount in event and extension call
```

2. **Apply the same fix on Solana**: read the vault token account balance before and after `transfer_checked` and use the delta as the payload amount posted to Wormhole.

3. **Enforce a token allowlist** at the contract level (not just off-chain policy) to reject fee-on-transfer tokens before they can be locked.

---

### Proof of Concept

1. Deploy or use any ERC-20 with a 1% fee-on-transfer.
2. Call `OmniBridge.initTransfer(feeToken, 1000e18, 0, 0, "near:recipient", "")`.
3. The vault receives `990e18` tokens; the `InitTransfer` event records `amount = 1000e18`.
4. NEAR bridge signs a `TransferMessagePayload` for `1000e18`.
5. User calls `finTransfer` on EVM with the signed payload; the bridge attempts `safeTransfer(recipient, 1000e18)` but only holds `990e18`, reverting — or, if other users' deposits cover the gap, the vault is silently drained by `10e18`.
6. Repeat until the vault is insolvent and legitimate withdrawals revert. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
