### Title
`near_withdraw_callback` Panic on Failure Silently Commits Finalization, Permanently Locking User Funds Without Delivery â€” (File: near/omni-bridge/src/lib.rs)

---

### Summary

When the wNEAR `near_withdraw` call fails during a `fin_transfer` to a NEAR recipient, `near_withdraw_callback` panics. The downstream callback `fin_transfer_send_tokens_callback` is still invoked but unconditionally takes the **success path** because `is_ft_transfer_call = false` for the wNEAR path. The transfer is permanently marked as finalised and `locked_tokens` are permanently decremented, yet the recipient never receives their NEAR tokens. The same logic flaw affects the plain `ft_transfer` path for non-deployed tokens with an empty `msg`.

---

### Finding Description

**Step 1 â€” State committed before external call.**

In `process_fin_transfer_to_near`, two irreversible state mutations happen synchronously before any external call:

- `add_fin_transfer` inserts the `TransferId` into `finalised_transfers` (replay guard â€” permanent).
- `unlock_tokens_if_needed` decrements `locked_tokens` for the origin chain. [1](#0-0) 

**Step 2 â€” wNEAR path in `send_tokens`.**

When `token == self.wnear_account_id && msg.is_empty()`, `send_tokens` calls `near_withdraw` with only `WNEAR_WITHDRAW_GAS = 5 TGas`, then chains `near_withdraw_callback`: [2](#0-1) 

**Step 3 â€” `near_withdraw_callback` panics on failure.**

If `near_withdraw` fails for any reason (insufficient wNEAR balance in the bridge, gas exhaustion, wNEAR contract revert), `near_withdraw_callback` calls `env::panic_str`: [3](#0-2) 

In NEAR Protocol, a panicking callback does not abort the outer promise chain â€” the next `.then()` callback is still scheduled and invoked with a failed promise result.

**Step 4 â€” `fin_transfer_send_tokens_callback` ignores the failure.**

`fin_transfer_send_tokens_callback` is chained after `send_tokens()`. It delegates failure detection entirely to `is_refund_required(is_ft_transfer_call)`: [4](#0-3) 

`is_refund_required` only reads the promise result when `is_ft_transfer_call = true`: [5](#0-4) 

For the wNEAR path, `msg` is always empty, so `is_ft_transfer_call = !msg.is_empty() = false`. `is_refund_required(false)` returns `false` unconditionally â€” the success branch executes, emitting `FinTransferEvent` and sending the fee, while the recipient receives nothing.

The same flaw applies to the plain `ft_transfer` branch for non-deployed tokens with empty `msg`: [6](#0-5) 

**Step 5 â€” No recovery path.**

`remove_fin_transfer` and `revert_lock_actions` are only called inside the `is_refund_required == true` branch: [7](#0-6) 

Because that branch is never reached for the wNEAR/ft_transfer paths on failure, the `finalised_transfers` entry and the decremented `locked_tokens` are never restored. The transfer ID is consumed; no relayer can re-submit it.

---

### Impact Explanation

**Critical â€” Irreversible fund lock / permanently unclaimable user value.**

After a failed `near_withdraw`:
- The `TransferId` is permanently in `finalised_transfers`; any re-submission reverts with `ERR_TRANSFER_ALREADY_FINALISED`.
- `locked_tokens` for the origin chain are permanently decremented, breaking the backing invariant.
- The recipient's NEAR tokens are undeliverable with no on-chain recovery path (only DAO manual intervention via `set_locked_tokens` could partially restore accounting, but cannot deliver the lost NEAR).
- The bridge emits `FinTransferEvent` as if the transfer succeeded, misleading off-chain indexers.

---

### Likelihood Explanation

**Low-Medium.** The wNEAR contract is external to the bridge. Failure scenarios include:

1. The bridge's wNEAR balance falls below the transfer amount due to any prior accounting divergence (e.g., a separate bug, a failed `ft_transfer` that left tokens in limbo).
2. `WNEAR_WITHDRAW_GAS = 5 TGas` is insufficient if the wNEAR contract's execution path changes after an upgrade.
3. The wNEAR contract itself reverts for any internal reason.

No malicious actor is required; a legitimate user's transfer combined with any of the above conditions triggers permanent fund loss.

---

### Recommendation

In `fin_transfer_send_tokens_callback`, check the promise result unconditionally â€” not only for `ft_transfer_call` paths. A minimal fix:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    // Always check promise result; for ft_transfer_call, refund on zero-return.
    // For all other paths, refund on promise failure.
    match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
        Err(_) => true,  // promise failed â†’ always refund
        Ok(value) if is_ft_transfer_call => {
            serde_json::from_slice::<U128>(&value).map_or(false, |a| a.0 == 0)
        }
        Ok(_) => false,
    }
}
```

Additionally, `near_withdraw_callback` should not panic â€” it should return a value that `fin_transfer_send_tokens_callback` can inspect, or the callback chain should be restructured so that a failed `near_withdraw` is surfaced as a failed promise result to `fin_transfer_send_tokens_callback`.

---

### Proof of Concept

1. User bridges NEAR from EVM â†’ NEAR. The bridge holds wNEAR representing the locked NEAR.
2. Relayer calls `fin_transfer` with a valid proof for a wNEAR transfer to a NEAR recipient with empty `msg`.
3. `process_fin_transfer_to_near` runs synchronously:
   - `add_fin_transfer` â†’ `finalised_transfers.insert(transfer_id)` âœ“ committed.
   - `unlock_tokens_if_needed` â†’ `locked_tokens` decremented âœ“ committed.
4. `send_tokens` dispatches `near_withdraw(amount)` with 5 TGas.
5. `near_withdraw` fails (e.g., bridge wNEAR balance = amount âˆ’ 1 due to prior dust loss).
6. `near_withdraw_callback` is invoked with `Err(_)` â†’ `env::panic_str("ERR_NEAR_WITHDRAW_FAILED")`.
7. NEAR runtime schedules `fin_transfer_send_tokens_callback` with a failed promise result.
8. `fin_transfer_send_tokens_callback` executes: `is_ft_transfer_call = false` â†’ `is_refund_required(false) = false` â†’ success branch â†’ emits `FinTransferEvent`.
9. **Result**: transfer permanently finalised, `locked_tokens` permanently reduced, recipient receives 0 NEAR, no retry possible.
