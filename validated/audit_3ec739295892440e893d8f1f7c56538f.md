Audit Report

## Title
Missing Chainlink Price Feed Staleness and Validity Checks in `getAssetPrice()` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `roundId`, `updatedAt`, and `answeredInRound`, performing no staleness, round-completeness, or sign checks on the returned price. A stale or zero price propagates directly into rsETH minting calculations in `LRTDepositPool`, causing depositors to receive incorrect rsETH amounts. The sibling oracle `ChainlinkOracleForRSETHPoolCollateral` in the same repository already implements the correct checks, confirming the omission is unintentional.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `latestRoundData()` is called but only `price` is captured:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

No checks are performed on:
- `answeredInRound < roundId` (incomplete/stale round)
- `updatedAt == 0` (round not finalized)
- `price <= 0` (invalid or negative answer)
- `block.timestamp - updatedAt > heartbeat` (time-based staleness)

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 27–32 explicitly validates all three conditions before returning a price.

The stale price flows through `LRTOracle.getAssetPrice()` (line 157), which is a pure passthrough to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, and is consumed in `LRTDepositPool.getRsETHAmountToMint()` (line 520):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`rsETHPrice` is a stored value updated separately via `updateRSETHPrice()`, so the denominator reflects a previously-computed correct price while the numerator uses the live (potentially stale) asset price. Any divergence between the stale feed and the true market price directly mis-scales the rsETH minted.

## Impact Explanation
**Stale price higher than market (e.g., feed last reported 1.02 ETH, market is 1.00 ETH):** The numerator is inflated, so the depositor receives more rsETH than the fair value of their deposit. This dilutes all existing rsETH holders and, at scale, drives the protocol toward insolvency — a Critical allowed impact.

**Stale price lower than market (e.g., feed last reported 0.98 ETH, market is 1.00 ETH):** The numerator is deflated, so the depositor receives fewer rsETH tokens than owed — the contract fails to deliver promised returns, a Low allowed impact.

Both directions are reachable without any privileged action. The higher-severity path (stale price above market) maps to **Critical: Protocol insolvency** under the allowed impact scope.

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for ETH/stETH). During L1 congestion, Chainlink node downtime, or sequencer issues on L2, feeds can lag beyond their heartbeat. No attacker action is required — any depositor transacting during a stale window is affected. The condition is passive, repeatable, and requires no special capability from the caller. The missing check is a straightforward code omission already corrected in the sibling oracle in the same repository.

## Recommendation
Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_PRICE_AGE) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_PRICE_AGE` should be set per-feed to match the Chainlink heartbeat (e.g., 3600 seconds for a 1-hour feed).

## Proof of Concept
**Scenario: stale price above market → excess rsETH minted, diluting existing holders**

1. Chainlink's stETH/ETH feed last updated 2 hours ago at 1.02 ETH; market has since corrected to 1.00 ETH.
2. Any user calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `1.02e18` (stale, no revert).
4. `rsethAmountToMint = (1000e18 * 1.02e18) / rsETHPrice` — user receives ~2% more rsETH than the fair value of their deposit.
5. Existing rsETH holders are diluted; repeated at scale this drives the protocol toward insolvency.
6. No revert occurs; the stale price is accepted silently.
