### Title
`ForeignTxSignPayload::compute_msg_hash` Omits Request-Binding Context, Enabling Cross-Request Replay of Foreign-Chain Attestations — (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash` using the domain's root public key, but **never verifies that `response.payload_hash` corresponds to the pending `request`**. Because `ForeignTxSignPayload::compute_msg_hash` produces a hash that contains only `(ForeignChainRpcRequest, Vec<ExtractedValue>)` — with no MPC contract account ID, no NEAR chain ID, and no domain ID — a single Byzantine MPC participant can replay a previously observed valid `(payload_hash, signature)` pair as the response to any new pending `verify_foreign_transaction` request. The contract accepts it, resolves the yield, and returns a forged attestation to the caller.

---

### Finding Description

**Root cause — missing binding in `compute_msg_hash`:**

`ForeignTxSignPayload::compute_msg_hash` computes:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
```

where `ForeignTxSignPayloadV1` contains only:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

No MPC contract account ID, no NEAR chain ID, and no domain ID are included in the hash. [1](#0-0) 

**Root cause — `respond_verify_foreign_tx` does not bind the signature to the pending request:**

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

The contract verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key. It does **not** recompute the expected hash from the pending `request` and compare it to `response.payload_hash`. [2](#0-1) 

After this check passes, the contract resolves all pending yields for `request` with the unverified `response`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [3](#0-2) 

**The node-side `build_signature_request` confirms the hash is computed without any contract-binding context:**

```rust
fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    ...
    Ok(SignatureRequest {
        ...
        tweak: Tweak::new([0u8; 32]),  // zero tweak — root key, no derivation
        ...
    })
}
``` [4](#0-3) 

The zero tweak confirms the root key signs the raw `compute_msg_hash()` output with no additional binding.

---

### Impact Explanation

A single Byzantine MPC participant (below the signing threshold) who has observed any previously valid `(payload_hash_A, sig_A)` pair — produced during normal operation — can call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })` for any new pending request B. The contract accepts the call because `sig_A` is a valid signature over `payload_hash_A` under the root key. All callers waiting on request B receive a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to foreign transaction A, not B.

A bridge contract that calls `verify_foreign_transaction` and does not independently verify `response.payload_hash` against its expected transaction will accept this forged attestation as proof that transaction B was verified on the foreign chain. This enables **invalid bridge execution** (e.g., releasing funds on NEAR for a foreign deposit that never occurred) or a **double-spend** (replaying the attestation for transaction A across multiple bridge requests).

Even callers that do verify the payload hash (e.g., via the SDK's `ForeignChainSignatureVerifier`) are harmed: the yield is consumed and cannot be retried, causing a permanent DoS on that request. [5](#0-4) 

---

### Likelihood Explanation

The attack requires only a **single Byzantine MPC participant** — well below the signing threshold. Such a participant:

1. Participates in normal signing rounds and observes valid `(payload_hash, signature)` pairs.
2. Calls `respond_verify_foreign_tx` with a replayed pair targeting any pending request.

The `assert_caller_is_attested_participant_and_protocol_active()` check is the only gate, and a Byzantine participant satisfies it by definition. No threshold collusion, no key leakage, and no network-level attack is required. The attack is executable in a single transaction.

---

### Recommendation

The contract must bind the signature to the specific pending request. Since the contract does not have the extracted `values` at request time, the recommended fix is to include the `ForeignChainRpcRequest` (which the contract does have) and the MPC contract account ID in the signed payload:

```rust
// In ForeignTxSignPayload::compute_msg_hash, include contract_id and domain_id:
msg_hash = SHA-256(borsh((contract_account_id, domain_id, ForeignTxSignPayload)))
```

Alternatively, require the responding node to submit the full `ForeignTxSignPayload` (including extracted values) in the response, allowing the contract to recompute and compare the hash against the pending request's `ForeignChainRpcRequest`.

### Proof of Concept

```
1. Alice submits verify_foreign_transaction for Bitcoin tx 0xAAAA (request_A).
   Contract stores pending yield keyed by request_A.

2. MPC network processes request_A honestly:
   payload_A = ForeignTxSignPayload::V1 { request: Bitcoin(0xAAAA), values: [BlockHash(0x1234)] }
   payload_hash_A = SHA-256(borsh(payload_A))
   sig_A = ECDSA_sign(root_key, payload_hash_A)
   respond_verify_foreign_tx(request_A, { payload_hash_A, sig_A }) → accepted, yield resolved.

3. Bob submits verify_foreign_transaction for Bitcoin tx 0xBBBB (request_B).
   Contract stores pending yield keyed by request_B.

4. Byzantine MPC node (single participant, below threshold) calls:
   respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })

5. Contract check (lib.rs:726-734):
   verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK  ✓
   (No check that payload_hash_A corresponds to request_B)

6. Contract resolves yield for request_B with { payload_hash_A, sig_A }.

7. Bob's bridge contract receives VerifyForeignTransactionResponse claiming
   Bitcoin tx 0xAAAA was verified — not 0xBBBB.
   If Bob's contract does not verify payload_hash, it releases funds for a
   transaction that was never verified.
```

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
