Audit Report

## Title
Single `MAX_REF_STALENESS` Applied to Both Legs of Synthetic Ratio Pairs Allows Stale Quote-Feed Price to Reach Pool Swaps — (File: `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary

`AnchoredPriceProvider` uses a single immutable `MAX_REF_STALENESS` for both the base and quote feed legs in `_readLeg()`. The `AnchoredProviderFactory` validates this threshold only against the base feed's class envelope; the quote feed's heartbeat is never checked. When the quote feed has a shorter heartbeat than the base feed, the staleness guard is too lenient for the quote leg, allowing a stale price to pass and reach pool swaps, producing a mispriced synthetic ratio.

## Finding Description

`_readLeg()` applies `MAX_REF_STALENESS` uniformly to every feed it reads: [1](#0-0) 

`_getBidAndAskPrice()` calls `_readLeg` for both `baseFeedId` and `quoteFeedId` with no per-leg staleness distinction: [2](#0-1) 

`MAX_REF_STALENESS` is a single immutable set at construction with no quote-leg counterpart: [3](#0-2) 

In `createAnchoredProvider()`, the staleness parameter is validated exclusively against the base feed's class envelope. The quote feed's class is never looked up and its envelope's `stalenessMin`/`stalenessMax` are never consulted: [4](#0-3) 

The factory's own NatSpec confirms this is the current design: *"The envelope is keyed on `baseFeedId` (the provider's class); the ref feed only contributes its uncertainty and is validated for existence at provider construction."* [5](#0-4) 

The only other per-leg guards — `priceGuard` min/max and the `spreadBps >= ORACLE_BPS` off-hours marker — do not substitute for a staleness check: the price guard range may be wide enough to pass a 12-hour-stale price, and the off-hours marker is a separate Chainlink-specific signal unrelated to keeper latency. [6](#0-5) 

## Impact Explanation

For a synthetic pair such as BTC/ETH (= BTC/USD ÷ ETH/USD), if `MAX_REF_STALENESS` is calibrated to the BTC/USD 24-hour heartbeat (86400 s) and the ETH/USD feed goes stale for 12 hours, `_isStale(now-43200, now, 86400)` returns `false`. The stale ETH price passes, `mid = BTC_USD_fresh / ETH_USD_stale` is computed with a potentially significantly wrong denominator, and the resulting bid/ask is anchored to that wrong mid. Pool swaps execute at the mispriced rate, allowing traders to extract value from LPs. This is a direct bad-price execution impact reaching a live pool swap.

## Likelihood Explanation

Synthetic ratio mode (`quoteFeedId != 0`) is an explicitly supported and documented feature. `createAnchoredProvider` is permissionless — any caller may deploy a provider with `MAX_REF_STALENESS` calibrated to the base feed while the quote feed has a shorter heartbeat, and the factory enforces no envelope check to prevent this. Feed staleness events (keeper outages, network congestion) are realistic operational scenarios requiring no attacker action to trigger; the attacker only needs to observe the stale condition and trade against the mispriced pool.

## Recommendation

Introduce a separate `maxQuoteRefStaleness` constructor parameter and immutable in `AnchoredPriceProvider`. In `AnchoredProviderFactory.createAnchoredProvider()`, look up the quote feed's class envelope independently and validate `maxQuoteRefStaleness` against it. Pass the appropriate threshold to `_readLeg` for each leg:

```solidity
// AnchoredPriceProvider: add immutable
uint256 public immutable MAX_QUOTE_REF_STALENESS;

// _getBidAndAskPrice():
(uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId, MAX_REF_STALENESS);
if (_quote != bytes32(0)) {
    (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote, MAX_QUOTE_REF_STALENESS);
    ...
}

// AnchoredProviderFactory.createAnchoredProvider():
if (_quote != bytes32(0)) {
    bytes32 quoteClassId = feedClass[quoteFeedId];
    if (quoteClassId == bytes32(0)) quoteClassId = DEFAULT_CLASS;
    Envelope storage quoteEnv = envelopes[quoteClassId];
    if (!quoteEnv.exists) revert EnvelopeNotFound(quoteClassId);
    if (maxQuoteRefStaleness < quoteEnv.stalenessMin || maxQuoteRefStaleness > quoteEnv.stalenessMax)
        revert ParamsOutOfEnvelope();
}
```

## Proof of Concept

1. Admin sets envelope for class `BTC` with `stalenessMax = 86400` (24 h); ETH class has `stalenessMax = 3600` (1 h).
2. Creator calls `createAnchoredProvider(oracle, BTC_USD_FEED, ETH_USD_FEED, ..., maxRefStaleness=86400, ...)`.
   - Factory looks up `feedClass[BTC_USD_FEED]` → `BTC` class; checks `86400 <= 86400` → passes.
   - No lookup or check is performed for `ETH_USD_FEED`.
3. ETH/USD oracle is not updated for 12 hours (keeper outage).
4. Trader calls `pool.swap(...)`. Pool calls `getBidAndAskPrice()`.
5. `_readLeg(ETH_USD_FEED)` runs `_isStale(now-43200, now, 86400)` → `43200 < 86400` → **not stale**, returns stale ETH price.
6. `mid = BTC_USD_fresh / ETH_USD_stale` — ratio is wrong by ETH price drift over 12 hours.
7. Pool quotes bid/ask anchored to the wrong mid; trader swaps at the mispriced rate, extracting value from LPs.
