Audit Report

## Title
Missing `address(0)` Validation on `addLiquidity` `owner` Parameter Permanently Locks LP Tokens - (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address and delegates to `LiquidityLib.addLiquidity` with no zero-address guard. If `owner = address(0)` is supplied, shares are minted to the `keccak256(abi.encode(address(0), salt, bin))` position key, tokens are pulled from `msg.sender` via callback and deposited into the pool, and the position becomes permanently irrecoverable because `removeLiquidity` enforces `msg.sender == owner` — a condition that can never be satisfied when `owner` is `address(0)`.

## Finding Description
`MetricOmmPool.addLiquidity` passes the caller-supplied `owner` directly to `LiquidityLib.addLiquidity` without any validation: [1](#0-0) 

Inside `LiquidityLib.addLiquidity`, the position key is computed as `keccak256(abi.encode(owner, salt, bin))` and shares are written unconditionally: [2](#0-1) [3](#0-2) 

Tokens are then pulled from `msg.sender` via the modify-liquidity callback: [4](#0-3) 

`removeLiquidity` enforces `msg.sender == owner` before allowing any withdrawal: [5](#0-4) 

Because no EOA or contract can produce a transaction where `msg.sender == address(0)` in the EVM, the position at `address(0)` can never be unwound. The deposited tokens inflate `binTotals.scaledToken0/scaledToken1` and per-bin `token(0|1)BalanceScaled` permanently, with no recovery path. Neither `LiquidityLib.addLiquidity` nor `MetricOmmPool.addLiquidity` contains any `require(owner != address(0))` guard. [6](#0-5) 

## Impact Explanation
Any tokens deposited under `owner = address(0)` are irrecoverably locked inside the pool. The pool's `binTotals` and per-bin balances reflect the deposit, so the tokens inflate LP accounting without being claimable by anyone. This constitutes a direct, permanent loss of user principal matching the "direct loss of user principal or owed LP assets" impact gate.

## Likelihood Explanation
The protocol explicitly supports an operator pattern where `msg.sender` (e.g., a router such as `MetricOmmPoolLiquidityAdder`) may differ from `owner`. A router that fails to validate its `owner` argument, or a user who passes a zero address by mistake, triggers the loss with a single transaction. No privileged access is required; any caller of `addLiquidity` can reach this path.

## Recommendation
Add a zero-address check at the top of `LiquidityLib.addLiquidity` (or equivalently in `MetricOmmPool.addLiquidity` before delegation):

```solidity
require(owner != address(0), "owner cannot be zero address");
```

## Proof of Concept
1. Attacker (or mistaken user/router) calls `pool.addLiquidity(address(0), salt, deltas, callbackData, extensionData)`.
2. `LiquidityLib.addLiquidity` computes `posKey = keccak256(abi.encode(address(0), salt, binIdx))` and mints shares there; `binTotalShares[binIdx]` and `positionBinShares[posKey]` are incremented.
3. The modify-liquidity callback fires on `msg.sender`; tokens are transferred into the pool.
4. `binTotals.scaledToken0 / scaledToken1` increase; `binState.token(0|1)BalanceScaled` increases.
5. Caller attempts `pool.removeLiquidity(address(0), salt, deltas, "")` → reverts `NotPositionOwner` because `msg.sender != address(0)`.
6. Tokens are permanently locked; no recovery path exists.
