### Title
`respond_verify_foreign_tx` Accepts Replayed Signatures from Prior Requests, Enabling Forged Foreign-Chain Verification — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is cryptographically valid over `response.payload_hash` under the root public key, but **never checks that `response.payload_hash` is actually derived from the `request` argument**. Any `(payload_hash, signature)` pair that was legitimately produced for a prior foreign-transaction request is a valid signature under the root key and can be replayed by a single malicious attested participant against any other pending request. The contract accepts the replay, resolves the victim's yield with the wrong response, and the caller receives a forged attestation.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs two independent checks and then resolves the yield:

```
// 1. Verify signature over payload_hash under the ROOT public key (no tweak)
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // taken directly from response, never re-derived
    &secp_pk,               // root key, no derivation
)

// 2. Resolve the pending yield for `request` with the full response
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

The `payload_hash` that nodes are supposed to sign is `SHA-256(borsh(ForeignTxSignPayload { request, values }))` — it commits to both the `ForeignChainRpcRequest` and the extracted on-chain values. The contract cannot recompute this hash because `values` are not stored on-chain; they are only known to nodes after querying the foreign chain. The contract therefore has no way to enforce the binding between `response.payload_hash` and `request`.

This is structurally identical to the external report's root cause: in both cases a function that is supposed to authenticate a piece of context (caller identity / response payload) reads that context from caller-supplied data without verifying it against the authoritative source.

The node-side `build_signature_request` hardcodes `tweak: Tweak::new([0u8; 32])`, so every foreign-tx signature is produced under the **root key with no derivation**. Any previously published `VerifyForeignTransactionResponse` on the NEAR blockchain is therefore a reusable credential: its `(payload_hash, signature)` pair will pass the contract's only check regardless of which `request` it is paired with.

---

### Impact Explanation

A single malicious attested participant (Byzantine node strictly below the signing threshold) can:

1. Scan the NEAR blockchain for any prior `verify_foreign_transaction` response `R_X = { payload_hash_X, sig_X }` produced for foreign transaction X.
2. Wait for a new pending request `R_Y` (for a different transaction Y) to appear in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(request_Y, R_X)` before honest nodes respond.
4. The contract verifies `sig_X` over `payload_hash_X` under the root key — this passes — and resolves the yield for `request_Y` with `R_X`.
5. The caller of `verify_foreign_transaction(Y)` receives `payload_hash_X` and `sig_X`.

Consequences:
- **Forged foreign-chain verification**: the caller receives a valid MPC signature that attests to transaction X's data, not Y's. Any downstream bridge or NEAR contract that does not independently re-derive and check `payload_hash` will process X's extracted values as if they belonged to Y, enabling invalid bridge execution or double-spend conditions.
- **Irrecoverable denial of service**: the yield for `request_Y` is consumed and cannot be retried; the caller is permanently stuck with the wrong response.

---

### Likelihood Explanation

- Requires only **one** malicious TEE-attested participant — well below the signing threshold.
- The replay material (`payload_hash`, `signature`) is publicly available in NEAR transaction history; no key material or threshold cooperation is needed.
- The attacker must submit before honest nodes respond, which is realistic: a single node acting immediately after seeing the request on-chain can race honest nodes that must first query the foreign-chain RPC.
- The attack is repeatable across every pending `verify_foreign_transaction` request.

---

### Recommendation

Bind `response.payload_hash` to `request` inside the contract. Because `values` are not stored on-chain, the simplest fix is to include a commitment to `request` inside the signed payload and verify it in `respond_verify_foreign_tx`:

```rust
// In respond_verify_foreign_tx, after verifying the signature:
let expected_request_hash = sha256(borsh(request.request));
let actual_request_hash   = sha256(borsh(response.request_commitment)); // new field
assert_eq!(expected_request_hash, actual_request_hash,
    "payload_hash does not commit to the submitted request");
```

Alternatively, restructure `ForeignTxSignPayload` so that `compute_msg_hash` always includes a hash of the `ForeignChainRpcRequest`, and have the contract re-derive and compare that portion of the hash from the stored `request` field.

### Proof of Concept

```
// Setup: contract is Running, ForeignTx domain exists, Bitcoin is supported.

// Step 1 — Alice submits a legitimate request for Bitcoin tx Y.
alice.call(contract, "verify_foreign_transaction", {
    request: { request: BitcoinRpcRequest { tx_id: Y, ... }, domain_id, payload_version }
})
// → pending_verify_foreign_tx_requests[request_Y] = [yield_alice]

// Step 2 — Eve (malicious attested participant) fetches a prior response
//           for Bitcoin tx X from NEAR transaction history:
//   response_X = { payload_hash_X, signature_X }
//   where signature_X = ECDSA_sign(root_key, payload_hash_X)

// Step 3 — Eve calls respond_verify_foreign_tx with mismatched (request_Y, response_X).
eve.call(contract, "respond_verify_foreign_tx", {
    request: request_Y,   // pending request for tx Y
    response: response_X  // valid signature, but for tx X
})

// Step 4 — Contract checks:
//   verify_ecdsa_signature(sig_X, payload_hash_X, root_pk) → OK  ✓
//   (no check that payload_hash_X derives from request_Y)
//   resolve_yields_for(request_Y, serialize(response_X))   → yield_alice resumed

// Step 5 — Alice's promise resolves with response_X:
//   { payload_hash: payload_hash_X, signature: sig_X }
//   payload_hash_X = SHA-256(borsh(ForeignTxSignPayload { request: request_X, values: values_X }))
//   ≠ SHA-256(borsh(ForeignTxSignPayload { request: request_Y, values: ... }))
//
// Alice's bridge contract receives a valid MPC signature attesting to tx X's data,
// not tx Y's. If the bridge does not re-derive and verify payload_hash, it processes
// X's extracted values as Y's — forged foreign-chain verification.
```

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
}
