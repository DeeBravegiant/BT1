### Title
Tokens Permanently Locked When `normalize_amount` Truncates Outbound Transfer Amount to Zero — (`near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates an outbound transfer (NEAR → EVM/Solana/Starknet) with an amount that floor-divides to zero after decimal normalization, their tokens are irreversibly burned or locked on NEAR. The zero-amount guard exists only in `sign_transfer`, which executes **after** token commitment, leaving the transfer permanently uncompletable with no refund path.

### Finding Description

**Bug class**: Asset/accounting — zero-amount transfer succeeds at commitment but fails at signing, creating an unrecoverable state.

The `normalize_amount` function performs integer floor division: [1](#0-0) 

For any token where `origin_decimals > decimals` (e.g., a NEAR-native token with 24 decimals bridged to EVM with 18 decimals), any amount strictly less than `10^(origin_decimals − decimals)` normalizes to 0.

The outbound transfer flow is:

1. **Token commitment** — `init_transfer_internal` burns (bridge tokens) or locks (native tokens) the user's tokens unconditionally, with no check that `normalize_amount(amount_without_fee) > 0`: [2](#0-1) 

2. **Zero-amount guard** — the guard only fires in `sign_transfer`, called by a relayer **after** tokens are already committed: [3](#0-2) 

Once `sign_transfer` reverts with `InvalidAmountToTransfer`, the transfer can never be signed or completed. There is no `cancel_transfer` or outbound-refund function in the contract — the only refund path (`fin_transfer_send_tokens_callback`) applies exclusively to the inbound direction (other chain → NEAR): [4](#0-3) 

The `normalize_amount` call in `sign_transfer` operates on `amount_without_fee()`, which is `amount - fee.fee`: [5](#0-4) 

### Impact Explanation

**Irreversible fund lock.** Any user who initiates an outbound transfer with a sub-threshold amount loses their tokens permanently:

- **Bridge tokens** are burned via `ext_token::burn` — supply is reduced, tokens are gone.
- **Native tokens** are locked in the bridge contract — they can never be unlocked because `unlock_tokens_if_needed` is only called from `process_fin_transfer_to_near` (inbound path) or from the refund branch of `fin_transfer_send_tokens_callback`, neither of which is reachable for a stuck outbound transfer.

The nonce is consumed and the `TransferMessage` is stored, but the transfer is permanently frozen.

### Likelihood Explanation

**Medium.** The condition requires a token registered with `origin_decimals > decimals`. This is the normal case for NEAR-native tokens (24 decimals) bridged to EVM (18 decimals), giving a normalization divisor of `10^6`. Any user sending fewer than 1,000,000 yocto-units of such a token triggers the bug. The amount threshold is small but non-trivial; a user testing with a dust amount or making a UI rounding error is a realistic path. No privileged access is required — the entry point is the public `ft_transfer_call` NEP-141 interface.

### Recommendation

Add a normalization check **before** token commitment. In `init_transfer_internal` (or at the `ft_on_transfer` entry point), compute `normalize_amount(amount_without_fee)` using the registered `token_decimals` for the destination chain and revert if the result is zero. This mirrors the guard already present in `sign_transfer` and ensures tokens are never committed for a transfer that can never be signed.

### Proof of Concept

1. Token is registered with `origin_decimals = 24`, `decimals = 18` (NEAR → EVM, 6-decimal gap, divisor = `10^6`).
2. User calls `ft_transfer_call` with `amount = 500_000` (below the `10^6` threshold), `fee = 0`.
3. `init_transfer_internal` burns 500,000 yocto-units of the bridge token (or locks them for a native token).
4. `TransferMessage { amount: U128(500_000), fee: Fee { fee: U128(0), ... }, ... }` is stored.
5. Relayer calls `sign_transfer`.
6. `normalize_amount(500_000, { decimals: 18, origin_decimals: 24 })` = `500_000 / 1_000_000` = **0** (floor division).
7. `require!(0 > 0, BridgeError::InvalidAmountToTransfer)` — transaction reverts.
8. No cancel or refund function exists for outbound transfers. The 500,000 yocto-units are permanently lost.

    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
