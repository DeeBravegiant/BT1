### Title
`accept_requests` Guard in `respond`/`respond_ckd`/`respond_verify_foreign_tx` Blocks Delivery of Cryptographically-Valid Signatures for Already-Accepted Requests - (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond`, `respond_ckd`, and `respond_verify_foreign_tx` contract methods gate signature delivery on the **current** value of `accept_requests` — a TEE-validation flag that can be flipped to `false` by any single participant calling `verify_tee()`. A request that was legitimately accepted while `accept_requests = true` can never be fulfilled if the flag changes before the MPC nodes submit their response, even though the signature is cryptographically verified on-chain. All queued yields time out and fail, permanently breaking the request lifecycle for those callers.

---

### Finding Description

The `MpcContract` stores a boolean field `accept_requests` that is set to `false` by `verify_tee()` when fewer than `threshold` participants hold a currently-valid TEE attestation. [1](#0-0) 

The three node-facing response methods each hard-reject when this flag is `false`:

```
// respond (line 579)
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}

// respond_ckd (line 662)
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}

// respond_verify_foreign_tx (line 711)
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
``` [2](#0-1) [3](#0-2) [4](#0-3) 

This check is applied **after** the request was already accepted (when `accept_requests` was `true`) and **before** the on-chain cryptographic signature verification that follows. The contract already independently verifies the signature's mathematical correctness against the stored public key and the original payload before calling `pending_requests::resolve_yields_for`. [5](#0-4) 

The `verify_tee()` function is callable by **any single participant** (voter), not a threshold: [6](#0-5) 

When attestations expire for enough participants, one participant calling `verify_tee()` sets `accept_requests = false` and all in-flight `respond` calls fail. The pending yields then sit until the NEAR runtime's ~200-block yield-timeout fires, returning an error to the original callers.

The structural parallel to the Symmio `isSolventAfterClosePosition` bug is exact:

| Symmio | NEAR MPC |
|---|---|
| Checks solvency at **market price** (current state) AND **close price** (final state) | Checks `accept_requests` (current TEE state) AND signature validity (final state) |
| Reverts even when final state is valid | Reverts even when signature is cryptographically valid |
| Unfair to partyA who could have been made solvent | Unfair to users whose requests were legitimately accepted |

---

### Impact Explanation

**Medium — request-lifecycle invariant break.**

All pending signature, CKD, and foreign-transaction-verification requests that were accepted while `accept_requests = true` are permanently blocked from being fulfilled if `accept_requests` transitions to `false` before the MPC nodes submit their responses. The callers receive a timeout error instead of the signature they are entitled to. Cross-chain operations depending on those signatures fail. Users must resubmit after TEE is restored, with no guarantee the window will not repeat.

This matches: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

TEE attestation expiry is a **normal operational event** (attestations have a configurable TTL). Any single participant can call `verify_tee()` at any time. The window between a user's `sign` call and the MPC nodes' `respond` call spans multiple NEAR blocks (the nodes must run the threshold protocol, which takes real time). The overlap between an attestation-expiry event and an in-flight signing round is a realistic, non-adversarial scenario. A Byzantine participant below the signing threshold can also deliberately time the `verify_tee()` call to maximally disrupt pending requests.

---

### Recommendation

Remove the `accept_requests` guard from `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. The guard is appropriate for **new** requests (preventing fresh work from being accepted under a degraded TEE state) but is incorrect for **responses** to already-accepted requests. The responding node's TEE validity is already enforced by `assert_caller_is_attested_participant_and_protocol_active()`, and the signature's correctness is independently verified on-chain. Only the final state — a valid signature for a legitimately pending request — should matter. [7](#0-6) 

---

### Proof of Concept

1. Contract is `Running`, `accept_requests = true`. Three participants, threshold = 2. All attestations valid.
2. User calls `sign(request)`. The contract accepts it, parks a yield, and stores the request in `pending_signature_requests`.
3. MPC nodes run the threshold ECDSA protocol and produce a valid signature. Node A prepares a `respond(request, signature)` transaction.
4. Before Node A's transaction lands, Participant B calls `verify_tee()`. One participant's attestation has just expired (TTL elapsed). Remaining valid = 2 = threshold, so the `validate_governance_against_reconstruction` check fails (or the threshold relation breaks), and `verify_tee()` sets `self.accept_requests = false`. [8](#0-7) 
5. Node A's `respond` transaction executes. The guard at line 579 fires: `return Err(TeeError::TeeValidationFailed.into())`. The call reverts.
6. No other node can deliver the response either — all hit the same guard.
7. After ~200 blocks the NEAR runtime fires the yield-timeout callback. The user's `sign` call resolves with an error. The user's cross-chain transaction fails.
8. The cryptographically valid signature computed by the MPC nodes is discarded. The user must resubmit from scratch. [9](#0-8) [10](#0-9)
