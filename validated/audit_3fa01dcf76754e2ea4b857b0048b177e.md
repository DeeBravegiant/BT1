Audit Report

## Title
Stale Cross-Chain Oracle Rate Enables Structural Over-Minting of wrsETH, Causing Undercollateralization of the L2 Wrapper - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3.sol)

## Summary
Both L2 pool contracts mint wrsETH using a cross-chain oracle rate that is frozen at the last LayerZero-delivered value and carries no staleness check. Because the L1 rsETH price increases monotonically with staking rewards, any deposit made while the L2 oracle lags the L1 price mints more wrsETH than the deposited ETH will produce in rsETH when bridged and converted on L1. The resulting deficit accumulates in the wrsETH wrapper and dilutes the yield owed to existing holders.

## Finding Description
**Root cause — `CrossChainRateReceiver.getRate()` returns a frozen value with no staleness guard.**

`CrossChainRateReceiver` stores `lastUpdated` on every `lzReceive` call but never consults it in `getRate()`, which simply returns the stored `rate`:

```solidity
// CrossChainRateReceiver.sol L95-105
rate = _rate;
lastUpdated = block.timestamp;
...
function getRate() external view returns (uint256) {
    return rate;
}
```

Both pool contracts delegate to this function without any additional check:

- `RSETHPoolV3.getRate()` → `IOracle(rsETHOracle).getRate()` (line 235–237)
- `RSETHPoolV3ExternalBridge.getRate()` → `IOracle(rsETHOracle).getRate()` (line 355–357)

**Exploit path:**

1. The L2 oracle rate is last updated at `T=0` with value `1.00e18`.
2. Staking rewards accumulate on L1; `LRTOracle._updateRsETHPrice()` raises `rsETHPrice` to `1.05e18` (line 250 of `LRTOracle.sol`).
3. No new LayerZero message arrives; the L2 oracle remains at `1.00e18`.
4. A user calls `RSETHPoolV3.deposit{value: 100 ETH}()`. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = 100e18 * 1e18 / 1.00e18 = 100 wrsETH` and mints it (lines 299–308 of `RSETHPoolV3.sol`).
5. The BRIDGER later calls `L1VaultV2.depositETHForL1VaultETH()`, which calls `lrtDepositPool.getRsETHAmountToMint(ETH_IDENTIFIER, 100e18)`. This uses the current L1 price: `100e18 * 1e18 / 1.05e18 ≈ 95.24 rsETH` (line 520 of `LRTDepositPool.sol`).
6. The wrapper now holds 95.24 rsETH but has 100 wrsETH outstanding — a 4.76 rsETH deficit per cycle.

**Existing guards are insufficient.** The `dailyMintLimit` modifier only caps total wrsETH minted per day; it does not detect or prevent rate-staleness-induced over-minting. The `whenNotPaused` modifier is irrelevant to this path. No slippage parameter or minimum-rsETH-out check exists in either `deposit()` function.

## Impact Explanation
**High — Theft of unclaimed yield.**

The wrsETH wrapper is designed to be 1:1 backed by rsETH. Every wrsETH minted in excess of the rsETH that will be received permanently dilutes the backing ratio. When existing holders later unwrap wrsETH → rsETH, the wrapper is short on rsETH, meaning all holders receive proportionally less than 1:1. The stolen value is exactly the staking yield that should have accrued to existing wrsETH holders between oracle updates. The deficit is proportional to `(rsETHPrice_L1 − rsETHPrice_L2) / rsETHPrice_L2` multiplied by total ETH deposited during the staleness window — a concrete, measurable loss of yield belonging to prior depositors.

## Likelihood Explanation
**Medium.** The rsETH price grows at roughly the staking APR (~4–5%/year). A 24-hour staleness window produces ~0.012% over-minting per deposit; a 7-day window produces ~0.087%. LayerZero message delivery is not instantaneous and is subject to network congestion, gas spikes, or infrequent rate-provider calls. No on-chain staleness guard exists in any pool contract or oracle interface. The condition is reachable by any depositor without any privileged action, and the deficit accumulates with every deposit made during a staleness window.

## Recommendation
1. Add a `MAX_STALENESS` constant (e.g., 24 hours) and revert in `CrossChainRateReceiver.getRate()` if `block.timestamp - lastUpdated > MAX_STALENESS`.
2. Alternatively, add the staleness check directly in `viewSwapRsETHAmountAndFee()` in both pool contracts before consuming the rate.
3. As a defense-in-depth measure, implement a circuit breaker in both `deposit()` functions that pauses minting when the oracle has not been refreshed within the staleness window.
