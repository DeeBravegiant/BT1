### Title
P2P Sync Accepts Arbitrary `entry_point_selector` in InvokeV0 Without Hash Verification, Corrupting Stored Transaction Body — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

The `TryFrom<protobuf::transaction_in_block::InvokeV0> for InvokeTransactionV0` converter accepts any arbitrary `entry_point_selector` Felt value from a sync peer without validation. Combined with an explicitly acknowledged missing hash-body consistency check in the p2p sync client, a malicious peer can inject a tampered `entry_point_selector` into a stored `InvokeTransactionV0`, causing `starknet_traceTransaction` to replay the transaction against the wrong entry point and return an authoritative-looking wrong trace.

---

### Finding Description

**Step 1 — Unconstrained deserialization of `entry_point_selector`**

In `TryFrom<protobuf::transaction_in_block::InvokeV0> for InvokeTransactionV0`, the `entry_point_selector` field is decoded from the peer-supplied protobuf bytes and wrapped directly into `EntryPointSelector(entry_point_selector_felt)` with no allowlist, range, or semantic check: [1](#0-0) 

**Step 2 — Transaction hash taken from the peer, not computed from the body**

In `TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)`, the `transaction_hash` is extracted directly from the peer-supplied protobuf field (line 137–142) and returned alongside the deserialized transaction body. No cross-check between the hash and the body is performed at this layer: [2](#0-1) 

**Step 3 — P2P sync client explicitly skips hash-body verification**

In `parse_data_for_block` inside `TransactionStreamFactory`, the transaction body and the peer-supplied hash are pushed to `block_body` without any verification that the hash matches the body. The code even carries a TODO acknowledging this gap: [3](#0-2) 

**Step 4 — Tampered body written to storage**

The assembled `BlockBody` (containing the tampered transaction) is committed to storage unconditionally: [4](#0-3) 

**Step 5 — Execution uses the stored `entry_point_selector` directly**

When `starknet_traceTransaction` re-executes a V0 invoke, the blockifier reads `tx.entry_point_selector` straight from the stored transaction struct: [5](#0-4) 

**Why the block commitment does not catch this**

The `transaction_commitment` in the block header commits to transaction *hashes*, not to transaction bodies. Because the attacker supplies the *correct* original hash alongside the tampered body, the commitment check passes. The node never recomputes the hash from the stored body to verify consistency. [6](#0-5) 

---

### Impact Explanation

A malicious p2p sync peer can cause a syncing node to persistently store an `InvokeTransactionV0` whose `entry_point_selector` differs from the value used to compute the canonical transaction hash. Any subsequent call to `starknet_traceTransaction` for that transaction will re-execute it against the attacker-chosen entry point and return an authoritative-looking wrong trace. This fits the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The attacker only needs to operate a reachable p2p sync peer. The correct transaction hash for any historical V0 invoke is public information. No validator, sequencer, or operator privilege is required. The missing validation is explicitly acknowledged in the codebase (`// TODO(eitan): Validate transaction hash from untrusted sources`), confirming the gap is known and unmitigated.

---

### Recommendation

After deserializing a `FullTransaction` from a p2p peer, recompute the transaction hash from the body using `get_transaction_hash` and reject (disconnect/penalize the peer) if it does not match the peer-supplied `transaction_hash` field. This check should be inserted in `parse_data_for_block` in `crates/apollo_p2p_sync/src/client/transaction.rs` immediately after line 74, replacing the TODO comment at line 88.

---

```rust
// In crates/apollo_protobuf/src/converters/transaction_test.rs
#[test]
fn injected_entry_point_selector_survives_deserialization() {
    use apollo_protobuf::protobuf;
    use starknet_api::transaction::InvokeTransactionV0;

    let attacker_selector = starknet_api::core::EntryPointSelector(
        starknet_types_core::felt::Felt::from(0xdeadbeef_u64),
    );

    let proto = protobuf::transaction_in_block::InvokeV0 {
        max_fee: Some(starknet_types_core::felt::Felt::ZERO.into()),
        signature: Some(protobuf::AccountSignature { parts: vec![] }),
        address: Some(starknet_types_core::felt::Felt::ONE.into()),
        entry_point_selector: Some(attacker_selector.0.into()),
        calldata: vec![],
    };

    let deserialized = InvokeTransactionV0::try_from(proto).unwrap();
    // Attacker-supplied selector is stored verbatim — no validation occurred.
    assert_eq!(deserialized.entry_point_selector, attacker_selector);
}
```

The test passes, confirming the tampered `entry_point_selector` is accepted and stored without any check.
