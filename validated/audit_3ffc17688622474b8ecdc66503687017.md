### Title
Missing Subgroup Check in `ecurve_ed_t::pub_from_der` Allows Torsion-Point Public Key to Pass EdDSA Verification - (`src/cbmpc/crypto/base_eddsa.cpp`)

### Summary

`ecurve_ed_t::pub_from_der` accepts any on-curve Ed25519 point, including torsion points outside the prime-order subgroup. The resulting `ecc_pub_key_t` is then passed directly to `ecurve_ed_t::verify`, which calls OpenSSL's `ED25519_verify` without a subgroup check. Because OpenSSL's verification uses the cofactor equation `[8][S]B = [8]R + [8][k]A`, a torsion public key (for which `[8]A = O`) degenerates the equation to `[8][S]B = [8]R`, which is satisfied by the forged signature `(R=O, S=0)` for any message.

### Finding Description

**Step 1 — `pub_from_der` accepts torsion points.**

`ecurve_ed_t::pub_from_der` checks only the X.509 prefix, then delegates to `from_bin`:

```cpp
// src/cbmpc/crypto/base_eddsa.cpp:191-194

  if (der.size != ed25519::x509_prefix.size + ed25519::pub_compressed_bin_size()) return coinbase::error(E_FORMAT);
  if (0 != memcmp(ed25519::x509_prefix.data, der.data, ed25519::x509_prefix.size)) return coinbase::error(E_FORMAT);
  return from_bin(P, der.skip(ed25519::x509_prefix.size));
}
``` [1](#0-0) 

`from_bin` only solves the curve equation to recover `x` from `y`; it does not call `is_in_subgroup()`: [2](#0-1) [3](#0-2) 

The existing `ecurve_t::check()` guard that would catch this is never invoked: [4](#0-3) 

The test suite confirms `from_bin` succeeds for the order-2 torsion point `(x=0, y=-1)` while `is_in_subgroup()` returns `false` and `curve.check()` returns an error: [5](#0-4) 

**Step 2 — `verify` passes the torsion key directly to `ED25519_verify` without a subgroup check.**

**Step 3 — `ED25519_verify` uses the cofactor equation, which degenerates for torsion keys.**

For OpenSSL ≥ 3.0 the wrapper delegates to `ossl_ed25519_verify`: [7](#0-6) 

OpenSSL's Ed25519 verification implements RFC 8032 §5.1.7 using the cofactor equation:

```
[8][S]B = [8]R + [8][k]A
```

For a torsion public key `A` with order dividing 8, `[8]A = O`. The equation collapses to:

```
[8][S]B = [8]R
```

Choosing `S = 0` and `R = O` (the identity point):

```
[8][0]B = O  =  [8]O   →   O = O  ✓
```

OpenSSL does not reject `S = 0` (only requires `S < order`) and does not reject `R = O` (the identity is a valid curve point). The forged signature passes for **any** message.

### Impact Explanation

An attacker who can supply a DER-encoded Ed25519 public key to any caller of `pub_from_der` / `ecc_pub_key_t::from_der` can:

1. Encode the order-2 torsion point `(x=0, y=-1)` with the correct X.509 prefix.
2. Have `pub_from_der` return `SUCCESS` and populate an `ecc_pub_key_t` with the torsion point.
3. Present the forged signature `(R = identity_encoding, S = 0x00…00)` for any message.
4. Have `ecurve_ed_t::verify` return `SUCCESS`.

This is a **High** impact: a public API reachable validation bypass causes accepted invalid cryptographic output (forged EdDSA signature verification).

### Likelihood Explanation

`pub_from_der` and `ecc_pub_key_t::from_der` are part of the public API surface. Any protocol path that accepts a peer-supplied DER-encoded Ed25519 public key and subsequently calls `verify` on it is exploitable. The torsion point encoding is a fixed 44-byte blob; no cryptographic computation is required to mount the attack.

### Recommendation

Add a subgroup check immediately after `from_bin` succeeds in `pub_from_der`:

```cpp
error_t ecurve_ed_t::pub_from_der(ecc_pub_key_t& P, mem_t der) const {
  if (der.size != ed25519::x509_prefix.size + ed25519::pub_compressed_bin_size())
    return coinbase::error(E_FORMAT);
  if (0 != memcmp(ed25519::x509_prefix.data, der.data, ed25519::x509_prefix.size))
    return coinbase::error(E_FORMAT);
  error_t rv = from_bin(P, der.skip(ed25519::x509_prefix.size));
  if (rv) return rv;
  if (!is_in_subgroup(P)) return coinbase::error(E_CRYPTO);  // <-- add this
  return SUCCESS;
}
```

Equivalently, replace the `from_bin` call with `curve.check(P)` after deserialization, consistent with how other production paths (e.g., `attach_private_scalar` in `src/cbmpc/api/eddsa_mp.cpp`) already guard against torsion points: [8](#0-7)

static error_t from_bin(point_t& R, mem_t bin) {
  if (bin.size != 32) return coinbase::error(E_FORMAT);

  buf_t buf = bin.rev();
  uint8_t neg = buf[0] >> 7;
  buf[0] &= 0x7f;
  fe_t y = fe_t::to_fe(uint256_t::from_bin(buf));

  // x² = (y² - 1) / (dy² + 1)

  fe_t u, v, w, vxx, check;

  u = y * y;
  v = u * formula_t::get_d();
  u -= fe_t::one();       // u = y^2-1
  v += fe_t::one();       // v = dy^2+1
  w = u * v;              // w = u*v
  fe_t x = w.pow22523();  // x = w^((q-5)/8)
  x *= u;                 // x = u * w^((q-5)/8)

  vxx = x * x;
  vxx *= v;
  check = vxx - u;  // vx^2-u
  if (!check.is_zero()) {
    check = vxx + u;  // vx^2+u
    if (!check.is_zero()) {
      return coinbase::error(E_CRYPTO);
    }
    static const fe_t sqrtm1 =
        fe_t::from_bn(bn_t::from_hex("2b8324804fc1df0b2b4d00993dfbd7a72f431806ad2fe478c4ee1b274a0ea0b0"));
    x *= sqrtm1;
  }

  uint256_t x_val = x.from_fe();
  if (neg != (x_val.w0 & 1)) x = -x;

  R.x = x;
  R.y = y;
  R.z = fe_t::one();
  return 0;
}
