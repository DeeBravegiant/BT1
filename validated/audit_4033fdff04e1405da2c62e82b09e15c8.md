### Title
NFT Royalty Payment Silently Bypassed via Integer Truncation in `compute_royalty_amount` - (File: `chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
`compute_royalty_amount` uses integer floor division that silently truncates to 0 for small offered amounts or large royalty splits. When the computed royalty is 0, the entire royalty enforcement path is skipped: no royalty coin is created, no announcement assertion is added to the spend bundle, and the NFT ownership-layer puzzle receives an empty `trade_prices_list`, meaning it does not enforce any royalty payment on-chain. An unprivileged attacker can craft an offer that satisfies all these conditions and acquire a royalty-enabled NFT without paying the creator's royalty.

### Finding Description

`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py` computes the royalty due using two sequential integer floor divisions:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

When `abs(offered_amount) // royalty_split * percentage < 10000`, the result is 0. The existing test suite explicitly documents this as accepted behavior:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

In `make_nft1_offer`, when `compute_royalty_amount` returns 0, the `CreateCoin` for the royalty recipient has `amount=0`. Three downstream guards then all silently skip royalty enforcement:

**Guard 1** — Announcement assertions are only added for payments where `p.amount > 0`, so a zero-royalty payment generates no `AssertPuzzleAnnouncement` in the spend bundle: [3](#0-2) 

**Guard 2** — The royalty coin output is only created if `payment_sum > 0`, so no royalty coin is sent to the OFFER_MOD: [4](#0-3) 

**Guard 3** — The royalty spend is skipped entirely when the total payment sum is 0: [5](#0-4) 

Additionally, the `trade_prices_list` passed to the NFT ownership-layer puzzle is filtered to exclude entries where the per-NFT royalty would be zero: [6](#0-5) 

`trade_prices` itself is built by dividing the total fungible amount by `offer_side_royalty_split`: [7](#0-6) 

When the per-NFT price after division is small enough that `price[0] * royalty_percentage // 10000 == 0`, the entry is excluded. The NFT ownership-layer puzzle then receives an empty `trade_prices_list` and does not enforce any royalty payment on-chain.

### Impact Explanation

The royalty recipient (NFT creator) is a third party who has no ability to reject or modify offers accepted by the NFT holder. When an attacker crafts an offer that causes `compute_royalty_amount` to return 0, the NFT transfers on-chain with no royalty coin created and no puzzle-level enforcement of royalty payment. The royalty recipient receives nothing despite the NFT being sold. This is an unauthorized accounting change affecting XCH/CAT payments in NFT offer settlement — the royalty recipient's entitled payment is silently diverted to zero.

### Likelihood Explanation

The condition is reachable by any unprivileged user who can create an offer. Two concrete trigger paths exist:

1. **Low royalty percentage + any amount**: An NFT with `royalty_percentage = 1` (0.01%) and an offer of 9999 mojos yields `9999 * 1 // 10000 = 0`.
2. **Large royalty split + moderate amount**: An offer requesting 100 NFTs (setting `request_side_royalty_split = 100`) with a total fungible offer of 9999 mojos and `royalty_percentage = 100` (1%) yields `(9999 // 100) * 100 // 10000 = 99 * 100 // 10000 = 0`. The royalty on 9999 mojos at 1% should be ~99 mojos, but truncation produces 0.

The NFT holder must accept the offer, but the royalty recipient has no veto. Marketplaces or automated takers that accept low-value offers are particularly exposed.

### Recommendation

1. **Reject zero-royalty results when a non-zero royalty is expected**: In `compute_royalty_amount`, if `percentage > 0` and the computed `royalty` is 0, raise a `ValueError` (analogous to the `royalty >= abs(offered_amount)` guard already present) rather than silently returning 0.
2. **Alternatively**, enforce a minimum royalty of 1 mojo when `percentage > 0`, rounding up instead of truncating: `amount = ceil(abs(offered_amount) / royalty_split * percentage / MAX_ROYALTY_BASIS_POINTS)`.
3. **Wallet-level guard**: Before submitting an offer, verify that the computed royalty for each NFT with `royalty_percentage > 0` is non-zero, and reject offer creation if it would be.

### Proof of Concept

**Scenario**: NFT with `royalty_percentage = 100` (1%). Attacker requests 100 such NFTs in a single offer, offering 9999 mojos total.

```
request_side_royalty_split = 100   # 100 NFTs requested
offered_amount = -9999             # 9999 mojos offered
percentage = 100                   # 1% royalty

# compute_royalty_amount:
amount = abs(-9999) // 100 * 100 // 10000
       = 99 * 100 // 10000
       = 9900 // 10000
       = 0   ← royalty silently becomes 0
```

Result: `royalty_payments[asset]` contains `CreateCoin(royalty_address, 0, ...)`. All three downstream guards skip royalty enforcement. The NFT ownership-layer puzzle receives `trade_prices_list=[]`. All 100 NFTs transfer on-chain; the royalty recipient receives 0 mojos instead of the ~99 mojos owed. [8](#0-7) [9](#0-8)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
