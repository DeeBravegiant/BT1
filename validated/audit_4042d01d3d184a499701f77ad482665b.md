Audit Report

## Title
Unset Minimum Withdrawal Amount Allows Queue Flooding to Delay Legitimate Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`minRsEthAmountToWithdraw[asset]` defaults to `0` for every asset because `initialize` never sets it, and `setMinRsEthAmountToWithdraw` is admin-only. When the mapping value is `0`, the guard in `initiateWithdrawal` reduces to a zero-amount check only, allowing any attacker to enqueue thousands of dust withdrawal requests. Because `_unlockWithdrawalRequests` advances a global sequential cursor (`nextLockedNonce`) that cannot skip entries, legitimate users whose requests are queued after the spam cannot complete their withdrawals until the operator has processed every preceding spam entry.

## Finding Description
`initialize` sets only `withdrawalDelayBlocks` and `lrtConfig`; it never initializes `minRsEthAmountToWithdraw` for any asset. [1](#0-0) 

The guard in `initiateWithdrawal` is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [2](#0-1) 

When `minRsEthAmountToWithdraw[asset] == 0`, the second condition `rsETHUnstaked < 0` is always `false` for `uint256`, so the guard accepts any amount ≥ 1 wei. Each accepted call transfers rsETH to the contract and appends a new entry to the global nonce sequence. [3](#0-2) 

`_unlockWithdrawalRequests` starts from `nextLockedNonce[asset]` and increments it one-by-one with no ability to skip or cancel entries: [4](#0-3) 

`completeWithdrawal` enforces that a user's nonce must be strictly less than `nextLockedNonce[asset]` before the withdrawal can be finalized: [5](#0-4) 

Therefore, any legitimate withdrawal queued after spam entries is blocked until the operator has advanced the cursor past every spam entry via repeated `unlockQueue` calls.

## Impact Explanation
Alice's rsETH is transferred to the contract at `initiateWithdrawal` time and cannot be recovered until `nextLockedNonce` advances past her nonce. The operator must pay gas to process every spam entry first. This constitutes **Temporary freezing of funds (Medium)** and **Unbounded gas consumption (Medium)**, both of which are explicitly listed in the allowed impact scope.

## Likelihood Explanation
The attack requires only rsETH tokens (obtainable by depositing ETH/LSTs into the protocol) and repeated public calls to `initiateWithdrawal`. No privileged access is needed. The default state of every new asset deployment has `minRsEthAmountToWithdraw == 0`, so the window is open unless an admin proactively calls `setMinRsEthAmountToWithdraw`. On L2 deployments where gas is cheap, large-scale queue flooding is economically feasible at minimal cost per spam entry.

## Recommendation
1. In `initialize`, set a non-zero default for each supported asset (e.g., `minRsEthAmountToWithdraw[asset] = 1e15`) so the guard is meaningful from deployment.
2. Alternatively, enforce a protocol-wide floor inside `initiateWithdrawal` that does not depend on the admin having called `setMinRsEthAmountToWithdraw`.
3. Consider adding a skip/cancel mechanism that allows operators to remove dust entries from the queue without advancing the cursor through them sequentially.

## Proof of Concept
1. Confirm `minRsEthAmountToWithdraw[ETH] == 0` on a fresh deployment (no admin call to `setMinRsEthAmountToWithdraw`).
2. Attacker acquires 10,000 wei of rsETH and calls `initiateWithdrawal(ETH, 1, "")` 10,000 times. Each call passes the guard because `1 != 0` and `1 < 0` is false for `uint256`. Nonces 0–9,999 are assigned to the attacker.
3. Alice calls `initiateWithdrawal(ETH, 10 ether, "")`. Her request is assigned nonce 10,000.
4. After the withdrawal delay, the operator calls `unlockQueue`. `_unlockWithdrawalRequests` starts at `nextLockedNonce[ETH] == 0` and must iterate through all 10,000 spam entries before reaching Alice's nonce.
5. Until the operator has paid gas to process all 10,000 spam entries across multiple `unlockQueue` calls, Alice's `completeWithdrawal` reverts with `WithdrawalLocked` because `10000 >= nextLockedNonce[ETH]`.

**Foundry test sketch:**
```solidity
function testQueueFlood() public {
    // attacker floods queue with 10_000 x 1-wei entries
    for (uint i = 0; i < 10_000; i++) {
        vm.prank(attacker);
        withdrawalManager.initiateWithdrawal(ETH, 1, "");
    }
    // Alice queues a legitimate withdrawal
    vm.prank(alice);
    withdrawalManager.initiateWithdrawal(ETH, 10 ether, "");
    // advance blocks past delay
    vm.roll(block.number + withdrawalDelayBlocks + 1);
    // operator unlocks only first batch — Alice's nonce not yet reached
    withdrawalManager.unlockQueue(ETH, 5000, ...);
    // Alice's completeWithdrawal reverts
    vm.expectRevert(WithdrawalLocked.selector);
    vm.prank(alice);
    withdrawalManager.completeWithdrawal(ETH, "");
}
```
