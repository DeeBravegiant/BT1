### Title
Silent `Word64` Underflow in `subtractCompactCoin` via `applyWithdrawals` Bypasses Account Balance Integrity - (File: `libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs`)

---

### Summary

`subtractCompactCoin` performs `Word64` subtraction without underflow protection. `applyWithdrawals`, which calls it directly, explicitly documents that it performs no validation. This function is introduced in the Dijkstra era as a replacement for the safe `drainAccounts` (which zeros balances unconditionally). If a withdrawal amount exceeds the account balance at the point `applyWithdrawals` is applied, the `Word64` wraps silently to a near-`maxBound` value, inflating the account balance from nothing.

---

### Finding Description

**Root cause — `subtractCompactCoin` (no underflow guard):**

`libs/cardano-ledger-core/src/Cardano/Ledger/Coin.hs`, lines 208–218:

```haskell
-- | /Warning/ - Be careful about integer overflow.
-- Amount to be subtracted must be smaller or equal to the value that is being adjusted.
-- This function does not protect against this.
subtractCompactCoin ::
  CompactForm Coin ->   -- Amount to be subtracted
  CompactForm Coin ->   -- Value from which it will be subtracted
  CompactForm Coin
subtractCompactCoin (CompactCoin amount) (CompactCoin source) =
  CompactCoin $ subtract amount source
```

`CompactForm Coin` is `newtype CompactCoin { unCompactCoin :: Word64 }`. Haskell's `Word64` arithmetic is modular: if `amount > source`, the result wraps to `maxBound Word64 - (amount - source - 1)`, silently producing a near-`18446744073709551615` lovelace balance. [1](#0-0) 

**Propagation — `applyWithdrawals` (no pre-check):**

`libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs`, lines 273–285:

```haskell
-- /Note/ - No checks on the accounts or amount being withdrawn are made in this function.
-- Verify that it's safe to call on the supplied arguments before calling it.
applyWithdrawals ::
  EraAccounts era =>
  Withdrawals ->
  Accounts era ->
  Accounts era
applyWithdrawals (Withdrawals wdrls) =
  updateAccountBalances
    (\amount account -> subtractCompactCoin amount (account ^. balanceAccountStateL))
    wdrls
``` [2](#0-1) 

The internal helper `updateAccountBalances` calls `compactCoinOrError` on the withdrawal `Coin` (which would `error` only if the coin is negative or overflows `Word64`, not if it exceeds the balance), then passes it directly to `subtractCompactCoin`. [3](#0-2) 

**Era divergence — Dijkstra uses `applyWithdrawals`; all prior eras use `drainAccounts`:**

Prior eras (Shelley, Alonzo, Conway) call `drainAccounts`, which unconditionally zeros the balance regardless of the withdrawal amount:

```haskell
drainAccounts (Withdrawals wdrls) = updateAccountBalances (\_ _ -> mempty) wdrls
``` [4](#0-3) 

`drainAccounts` is called in `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Ledger.hs`, `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Ledger.hs`, and `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`. `applyWithdrawals` is called exclusively in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs` and `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs`.

**Validation gap — the two guard functions are defined but not found in any production rule:**

`Account.hs` exports two functions designed to pre-validate withdrawals before applying them:

- `withdrawalsThatDoNotDrainAccounts` — checks `amount == balance`
- `withdrawalsThatExceedAccountBalance` — checks `amount <= balance` [5](#0-4) 

A search across the entire repository finds zero uses of either function in any production ledger rule. The Dijkstra rules that call `applyWithdrawals` do not appear to invoke either guard, leaving the "verify that it's safe" precondition unmet at the call site.

---

### Impact Explanation

If `applyWithdrawals` is reached with a withdrawal amount that exceeds the account balance — whether due to a missing validation in the Dijkstra UTXO/WDRL rule, a TOCTOU discrepancy between validation and application, or a future rule extension — `subtractCompactCoin` wraps the `Word64` balance to approximately `2^64 - 1` lovelace (~18.4 × 10^18 ADA). This constitutes **direct creation of ADA from nothing**, an invalid ledger state transition. The inflated balance persists in the `Accounts` map and can be withdrawn in subsequent transactions, permanently destroying the ADA supply invariant. Recovery would require a hard fork.

**Allowed impact matched:** *Critical — Direct creation of ADA through an invalid ledger state transition.*

---

### Likelihood Explanation

The Dijkstra era is the only era that uses `applyWithdrawals`. The two exported guard functions (`withdrawalsThatDoNotDrainAccounts`, `withdrawalsThatExceedAccountBalance`) are not called in any production rule. The function's own documentation explicitly states it performs no checks and requires the caller to pre-validate. An unprivileged transaction author who can submit a withdrawal transaction in the Dijkstra era is the attacker-controlled entry point. If the Dijkstra WDRL/UTXO predicate failure path has any gap — including a partial-withdrawal design change that the new era introduces — the underflow is reachable without any privileged access.

**Note:** The Dijkstra rules files (`Entities.hs`, `SubEntities.hs`) could not be fully read in this analysis. It is possible those rules contain inline validation that prevents the underflow. However, the absence of the exported guard functions from all production code, combined with the explicit "no checks" documentation on `applyWithdrawals` and the silent-failure nature of `subtractCompactCoin`, constitutes a confirmed structural risk that matches the vulnerability class.

---

### Recommendation

1. **Replace `subtractCompactCoin` with a safe variant** that returns `Maybe (CompactForm Coin)` or `Either` and propagates the failure as a `PredicateFailure` rather than silently wrapping.
2. **Revert `applyWithdrawals` to `drainAccounts` semantics** (zero the balance) in the Dijkstra era, matching all prior eras. Subtraction is only needed if partial withdrawals are intentionally introduced; if so, add an explicit guard.
3. **Call `withdrawalsThatExceedAccountBalance` (or `withdrawalsThatDoNotDrainAccounts`) in the Dijkstra UTXO/WDRL rule** before `applyWithdrawals`, and emit a `PredicateFailure` if the check fails, so the node rejects the transaction rather than silently corrupting state.

### Proof of Concept

```
Given:
  account balance = CompactCoin 100   (100 lovelace)
  withdrawal amount = CompactCoin 200 (200 lovelace)

subtractCompactCoin (CompactCoin 200) (CompactCoin 100)
  = CompactCoin (subtract 200 100)   -- Haskell: subtract x y = y - x
  = CompactCoin (100 - 200)          -- Word64 modular arithmetic
  = CompactCoin 18446744073709551516 -- ≈ 18.4 × 10^18 ADA created from nothing
```

An attacker submits a Dijkstra-era transaction with a withdrawal of `balance + 1` lovelace. If the Dijkstra WDRL predicate does not reject it, `applyWithdrawals` is called, `subtractCompactCoin` wraps, and the account now holds `~18.4 × 10^18` lovelace. The attacker drains this in a subsequent transaction, destroying the total ADA supply invariant permanently. [6](#0-5) [7](#0-6)

```haskell
updateAccountBalances ::
  EraAccounts era =>
  -- | Balance update: given a coin amount and the account, returns the new balance.
  (CompactForm Coin -> AccountState era -> CompactForm Coin) ->
  Map AccountAddress Coin ->
  Accounts era ->
  Accounts era
updateAccountBalances updateBalance balanceMap =
  accountsMapL %~ \accountsMap ->
    Map.foldrWithKey'
      ( \(AccountAddress _ (AccountId credential)) amount ->
          Map.adjust
            ( \account ->
                account & balanceAccountStateL .~ updateBalance (compactCoinOrError amount) account
            )
            credential
      )
      accountsMap
      balanceMap
{-# INLINE updateAccountBalances #-}
