Audit Report

## Title
Zero-rsETHAmount Deposit Silently Consumes Depositor ETH — (`contracts/pools/RSETHPoolV2NBA.sol`)

## Summary

`RSETHPoolV2NBA.deposit` accepts any non-zero ETH value but does not guard against `rsETHAmount` rounding to zero via integer division in `viewSwapRsETHAmountAndFee`. When a deposit is below the rate threshold, the transaction succeeds, zero wrsETH is minted, and the depositor's ETH is permanently lost: it accumulates in the pool's balance and is swept irrecoverably to L1 on the next `moveAssetsForBridging` call.

## Finding Description

`deposit` enforces only `amount == 0`: [1](#0-0) 

`viewSwapRsETHAmountAndFee` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` using truncating integer division: [2](#0-1) 

Because rsETH continuously accrues yield, `rsETHToETHrate > 1e18` always holds. For any deposit where `amountAfterFee * 1e18 < rsETHToETHrate` (e.g., 1 wei with a typical rate of ~1.05e18), `rsETHAmount` truncates to `0`. No guard exists between the computation and the mint call.

`RsETHTokenWrapper.mint` delegates directly to OpenZeppelin's `ERC20Upgradeable._mint`: [3](#0-2) 

OpenZeppelin's `_mint` does **not** revert on `_amount == 0`; it emits `Transfer(address(0), to, 0)` and returns normally. The depositor receives nothing.

The deposited ETH lands in `address(this).balance`. Since `fee` also rounds to 0 for sub-threshold amounts, `feeEarnedInETH` is not incremented. On the next `moveAssetsForBridging` call, the full `address(this).balance - feeEarnedInETH` is swept to the bridger: [4](#0-3) 

The depositor has no claim to the swept ETH and no recovery mechanism.

## Impact Explanation

A depositor permanently loses their ETH: zero wrsETH is issued, and the ETH is swept to L1 with no attribution. This is **Critical — Permanent freezing of funds**. The "temporary" framing in the submission undersells the impact; once `moveAssetsForBridging` executes, the loss is irreversible and the depositor has no on-chain recourse.

## Likelihood Explanation

`deposit` is a public payable function requiring no special role. Any user sending a sub-threshold amount (including 1 wei, dust amounts from automated scripts, or users testing the contract) triggers the bug. With a typical rate of ~1.05e18, the minimum safe `amountAfterFee` is 2 wei. No front-running or external dependency is required; the condition is deterministic given the current oracle rate.

## Recommendation

Add a post-computation zero-amount guard in `deposit`:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, enforce a minimum deposit at the entry point: `require(msg.value >= minDeposit)` where `minDeposit` is updated alongside the oracle rate.

## Proof of Concept

```
Preconditions:
  feeBps = 100 (1%)
  rsETHToETHrate = 1.05e18

Attacker calls: deposit{value: 1}("")

Step 1: amount = 1 wei → passes `amount == 0` check
Step 2: viewSwapRsETHAmountAndFee(1)
        fee = 1 * 100 / 10_000 = 0
        amountAfterFee = 1 - 0 = 1
        rsETHAmount = 1 * 1e18 / 1.05e18 = 0  ← truncated
Step 3: feeEarnedInETH += 0  (unchanged)
Step 4: wrsETH.mint(depositor, 0)  ← no-op, no revert
Step 5: 1 wei sits in address(this).balance; depositor holds 0 wrsETH
Step 6: moveAssetsForBridging() sweeps 1 wei to L1 bridger
        depositor has no recourse

Foundry fuzz test:
  for all amount in [1, rsETHToETHrate / 1e18]:
    assert: either rsETHAmount > 0 OR tx reverts
  → currently NEITHER holds for amount = 1 wei
```

```text
    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
    }
```
