### Title
Stale `drepExpiry` in DRep Ratification Check Ignores Accumulated `numDormantEpochs` — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

In the Conway era, `DRepState.drepExpiry` is a lazily-updated field: it is only written when a governance proposal is submitted (via `updateDormantDRepExpiries`) or when a DRep explicitly votes/updates. Between those events, accumulated dormant epochs are tracked separately in `VState.vsNumDormantEpochs`. The **actual** DRep expiry is `drepExpiry + numDormantEpochs`. However, the `dRepAcceptedRatio` function used during ratification checks only the raw `drepExpiry` field, ignoring `numDormantEpochs`. This mirrors the `totalCoverTokens` bug exactly: a value is only updated when a specific action occurs, not when the underlying time-based condition changes.

---

### Finding Description

`DRepState` stores a raw `drepExpiry` epoch number: [1](#0-0) 

`VState` separately tracks `vsNumDormantEpochs` — the count of consecutive epochs with no active proposals: [2](#0-1) 

The **actual** expiry is `drepExpiry + numDormantEpochs`. This is correctly computed in `isDRepExpired`: [3](#0-2) 

`drepExpiry` is only updated lazily — when a proposal is submitted, `updateDormantDRepExpiries` adds `numDormantEpochs` to all DRep expiries and resets the counter to zero: [4](#0-3) [5](#0-4) 

`numDormantEpochs` is incremented at each epoch boundary when no proposals are active: [6](#0-5) 

The DRep pulser is initialized at the epoch boundary (`setFreshDRepPulsingState`, line 372 of `Epoch.hs`) with a snapshot of `vsDReps` containing the raw `drepExpiry` values at that moment. During the subsequent epoch, `updateDormantDRepExpiries` may update the live ledger state's `drepExpiry` values, but the pulser's `reDRepState` snapshot is frozen.

At ratification time, `dRepAcceptedRatio` checks expiry using only the raw field: [7](#0-6) 

Line 267 is the defect:
```haskell
| reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired
```

The correct check should be:
```haskell
| reCurrentEpoch > binOpEpochNo (+) numDormantEpochs (drepExpiry drepState) -> (yes, tot)
```

The `numDormantEpochs` value is not threaded into `RatifyEnv` at all, so the ratification rule has no way to apply the dormant-epoch adjustment.

---

### Impact Explanation

A DRep with `drepExpiry = E` and accumulated `numDormantEpochs = K` is **not** actually expired as long as `E + K >= currentEpoch`. However, `dRepAcceptedRatio` treats it as expired whenever `currentEpoch > E`. If such a DRep voted **No** on a governance action, its stake is removed from both the numerator and denominator. Removing a No-voter from the denominator **increases** the Yes ratio, potentially pushing a proposal over the ratification threshold that would otherwise fail. This can cause an unauthorized protocol-parameter change, treasury withdrawal, hard-fork initiation, committee update, or constitution change to be enacted.

This matches: **Critical — Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.**

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender who can submit a governance proposal:

1. Allow `numDormantEpochs` to accumulate naturally (no proposals for K epochs).
2. Identify No-voting DReps whose raw `drepExpiry` satisfies `currentEpoch - K ≤ drepExpiry < currentEpoch` (i.e., they appear expired by raw check but are actually active).
3. Submit the attacker's proposal at the precise epoch where those DReps' raw expiry falls below `currentEpoch` but their actual expiry (`drepExpiry + K`) does not.
4. At the next epoch boundary, the pulser's frozen `reDRepState` excludes those DReps from the denominator, inflating the Yes ratio.

The attacker controls proposal timing and deposit; no privileged access is required. The scenario requires careful timing but is deterministic and fully on-chain.

---

### Recommendation

Pass `vsNumDormantEpochs` into `RatifyEnv` and replace the expiry check in `dRepAcceptedRatio` with the actual expiry:

```haskell
| reCurrentEpoch > binOpEpochNo (+) reNumDormantEpochs (drepExpiry drepState) -> (yes, tot)
```

Alternatively, pre-adjust the `reDRepState` snapshot when initializing the pulser by applying `updateDormantDRepExpiry` to the `vsDReps` map before freezing it, so the snapshot already reflects the actual expiries. This mirrors the fix in the external report: making the epoch-info update callable independently of the action that previously triggered it.

---

### Proof of Concept

```
Epoch 0: DRep D registers. drepExpiry = 3. numDormantEpochs = 0.
Epoch 1: No proposals. numDormantEpochs = 1.
Epoch 2: No proposals. numDormantEpochs = 2.
Epoch 3: No proposals. numDormantEpochs = 3.
  → Pulser initialized with reDRepState{drepExpiry=3}, reCurrentEpoch will be 4.
  → Actual expiry of D = 3 + 3 = 6. D is NOT expired.

Epoch 3 (same epoch, after pulser init): Attacker submits proposal P.
  → updateDormantDRepExpiries fires: D.drepExpiry updated to 3+3=6 in live state.
  → numDormantEpochs reset to 0 in live state.
  → D votes No on P.

Epoch 4 boundary: Pulser extracts ratification result.
  → reDRepState still has drepExpiry=3 (frozen snapshot).
  → reCurrentEpoch = 4.
  → Check: 4 > 3 → D is "expired" → D's No vote is excluded.
  → Yes ratio increases. P is ratified despite D's No vote.

```haskell
updateDormantDRepExpiry ::
  -- | Current Epoch
  EpochNo ->
  VState era ->
  VState era
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
  where
    numDormantEpochs = vState ^. vsNumDormantEpochsL
    updateExpiry =
      drepExpiryL
        %~ \currentExpiry ->
          let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
           in if actualExpiry < currentEpoch
                then currentExpiry
                else actualExpiry

```haskell
dRepAcceptedRatio ::
  forall era.
  RatifyEnv era ->
  Map (Credential DRepRole) Vote ->
  GovAction era ->
  Rational
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
