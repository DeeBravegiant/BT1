### Title
Sub-Transaction Execution Units Bypass `maxTxExUnits` and `maxBlockExUnits` Limits in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`)

---

### Summary

In the Dijkstra era, the `totExUnits` function only sums redeemers from the top-level transaction's own witness set. Sub-transactions embedded in a Dijkstra batch each carry their own independent witness sets with their own redeemers. Because `validateExUnitsTooBigUTxO` (per-transaction limit) and `validateExUnits` (per-block limit) both delegate to `totExUnits`, sub-transaction execution units are never counted against either `maxTxExUnits` or `maxBlockExUnits`. Additionally, the minimum-fee calculation (`alonzoMinFeeTx`) uses the same `totExUnits`, so sub-transaction script fees are excluded from the required fee, allowing script execution to occur without paying for it.

---

### Finding Description

**`totExUnits` ignores sub-transaction redeemers**

`totExUnits` is defined in `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It reads only `tx ^. witsTxL` — the top-level transaction's witness set. A Dijkstra top-level transaction (`DijkstraTx TopTx era`) embeds sub-transactions in `dtbrSubTransactions`, each of which is a full `DijkstraSubTx` with its own `dstWits :: TxWits era` containing independent redeemers. [1](#0-0) [2](#0-1) 

**Per-transaction limit check misses sub-transaction units**

In `dijkstraUtxoTransition`, the check is:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

`validateExUnitsTooBigUTxO` calls `totExUnits tx`, which only covers the top-level transaction's redeemers. Sub-transaction redeemers are invisible to this check. [3](#0-2) [4](#0-3) 

**Per-block limit check also misses sub-transaction units**

In `dijkstraBbodyTransition`:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
```

`validateExUnits` is `foldMap totExUnits txs`, which again only sums top-level redeemers across all transactions in the block. [5](#0-4) [6](#0-5) 

**Sub-transaction execution unit check is explicitly absent**

The `dijkstraUtxoToDijkstraSubUtxoPredFailure` mapping in `SubUtxo.hs` marks `ExUnitsTooBigUTxO` as `error "Impossible"`, confirming no execution unit check is ever applied to sub-transactions:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [7](#0-6) 

**Minimum fee calculation also excludes sub-transaction script fees**

`alonzoMinFeeTx` uses `totExUnits tx` to compute the script fee component:

```haskell
alonzoMinFeeTx pp tx =
  (tx ^. sizeTxF <×> ...) <+> ... <+> txscriptfee (pp ^. ppPricesL) allExunits
  where allExunits = totExUnits tx
```

Sub-transaction script execution costs are therefore not included in the required minimum fee. [8](#0-7) 

---

### Impact Explanation

An attacker can craft a Dijkstra-era top-level transaction whose own redeemers declare execution units just below `maxTxExUnits`, then embed N sub-transactions each declaring execution units approaching `maxTxExUnits`. The combined execution budget of the batch is `(N+1) × maxTxExUnits`, but the ledger accepts it because only the top-level transaction's units are checked. The same bypass applies at the block level against `maxBlockExUnits`. Additionally, the attacker pays only the script fee for the top-level transaction's execution units; sub-transaction script execution is obtained without paying the corresponding fee, modifying the effective fee outside design parameters.

This matches the **Medium** allowed impact: *attacker-controlled transactions exceed intended validation limits or modify fees outside design parameters.*

---

### Likelihood Explanation

Any Dijkstra-era transaction author (unprivileged) can include sub-transactions with arbitrarily large declared execution unit budgets. No special privilege, key, or governance threshold is required. The Dijkstra era is the production target for this feature, and the sub-transaction mechanism is the primary new capability it introduces. The bypass is structural and deterministic.

---

### Recommendation

1. Extend `totExUnits` (or introduce a Dijkstra-specific variant) to recursively sum execution units across all sub-transactions in `dtbrSubTransactions`:

```haskell
totExUnitsDijkstra tx =
  totExUnits tx
    <> foldMap totExUnits (OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL)
```

2. Use this extended function in `validateExUnitsTooBigUTxO` when called from `dijkstraUtxoTransition`, and in `validateExUnits` when called from `dijkstraBbodyTransition`.

3. Use the same extended function in the minimum fee calculation for Dijkstra-era transactions so that sub-transaction script fees are included in the required fee.

4. Remove the `error "Impossible: ExUnitsTooBigUTxO for SUBUTXO"` branch or replace it with a proper per-sub-transaction limit check if individual sub-transaction limits are also desired.

---

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction `txTop` with a single redeemer declaring `ExUnits { exUnitsMem = maxMem - 1, exUnitsSteps = maxSteps - 1 }` (just below `maxTxExUnits`).
2. Embed K sub-transactions `[subTx_1 .. subTx_K]` in `txTop ^. bodyTxL . subTransactionsTxBodyL`, each with a redeemer declaring `ExUnits { exUnitsMem = maxMem, exUnitsSteps = maxSteps }`.
3. Submit `txTop`. The ledger evaluates `totExUnits txTop` = only the top-level redeemer, which is below `maxTxExUnits`. The check passes.
4. The actual script execution budget consumed by the batch is `(K+1) × maxTxExUnits`, far exceeding the protocol limit.
5. The minimum fee paid covers only the top-level script fee; sub-transaction script fees are zero.

The `validateExUnitsTooBigUTxO` call at line 415 of `Utxo.hs` passes because `totExUnits txTop` only reads `dtWits` of the top-level `DijkstraTx`, never traversing `dtbrSubTransactions`. [3](#0-2) [9](#0-8) [1](#0-0)

-- | Validate that total execution units (all transactions) do not exceed block limit.
-- ∑(tx ∈ txs)(totExunits tx) ≤ maxBlockExUnits pp
validateExUnits ::
  forall era.
  ( AlonzoEraTx era
  , InjectRuleFailure "BBODY" AlonzoBbodyPredFailure era
  ) =>
  StrictSeq.StrictSeq (Tx TopTx era) ->
  -- | Max block exunits protocol parameter.
  ExUnits ->
  Rule (EraRule "BBODY" era) 'Transition ()
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax
        ?! injectFailure
          ( TooManyExUnits $
              Mismatch
                { mismatchSupplied = txTotal
                , mismatchExpected = ppMax
                }
          )
