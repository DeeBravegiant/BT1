### Title
Unbounded Subtransaction Iteration in `SUBLEDGERS` Rule Bypasses Per-Transaction Execution Unit Limit — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`)

---

### Summary

The Dijkstra era introduces batch transactions with subtransactions. The `SUBLEDGERS` rule iterates over all subtransactions via an uncapped `foldM` loop, and no protocol parameter limits the count of subtransactions per top-level transaction. Each subtransaction runs through the full `SUBLEDGER` rule (including `SUBENTITIES`, `SUBGOV`, `SUBUTXOW`). Critically, the per-transaction execution unit check (`validateExUnitsTooBigUTxO`) is applied independently to the top-level transaction and to each subtransaction, but never to the aggregate total. An attacker can therefore craft a single top-level transaction containing N subtransactions, each carrying up to `ppMaxTxExUnits` execution units, causing the effective per-transaction execution unit budget to be multiplied by N — exceeding the intended `ppMaxTxExUnits` limit without any ledger-level rejection at the transaction boundary.

---

### Finding Description

**Root cause 1 — Uncapped `foldM` over subtransactions in `SUBLEDGERS`:**

`dijkstraSubLedgersTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs` (lines 128–135) iterates over every subtransaction with no count guard:

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
``` [1](#0-0) 

The only bound on the number of subtransactions is the serialized transaction size (`ppMaxTxSize`). No protocol parameter such as `ppMaxSubTransactions` exists in `DijkstraPParams`: [2](#0-1) 

The CDDL schema confirms no upper bound on the count:

```
sub_transactions = nonempty_oset<sub_transaction>
``` [3](#0-2) 

**Root cause 2 — `validateExUnitsTooBigUTxO` is absent from the `SUBUTXO` rule:**

In `dijkstraUtxoTransition` (the top-level UTXO rule), the execution unit check is applied to the top-level transaction:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```
