Audit Report

## Title
Free Share Minting in Fully-Drained Bins Dilutes Existing LP Claims — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
In `LiquidityLib.addLiquidity`, when a bin has been fully drained by swaps (`token0BalanceScaled == 0` and `token1BalanceScaled == 0`) but still carries a positive `binTotalShares`, the proportional share-price calculation yields zero for both token amounts. Because the token-payment callback is only invoked when the aggregate owed amount is non-zero, an attacker can mint an arbitrary number of shares in such a bin without transferring any tokens. When the price reverses and swaps refill the bin, the attacker's free shares entitle them to a proportional fraction of the incoming tokens, directly stealing from existing LPs.

## Finding Description
In `LiquidityLib.addLiquidity`, the `else` branch (non-empty bin) computes per-share token amounts proportionally:

```solidity
// LiquidityLib.sol L108-111
} else {
    amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
    amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
}
``` [1](#0-0) 

When both balances are zero, `ceilDiv(0, binTotalSharesVal) = 0` for both tokens. The bin-state and share accounting are then updated unconditionally:

```solidity
// LiquidityLib.sol L120-121
binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd;
positionBinShares[posKey] = newUserShares;
``` [2](#0-1) 

The token-payment callback is guarded by a non-zero aggregate check:

```solidity
// LiquidityLib.sol L144-155
if (amount0Added > 0 || amount1Added > 0) {
    ...
    IMetricOmmModifyLiquidityCallback(msg.sender)
        .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
    ...
}
``` [3](#0-2) 

Since `amount0Added = 0` and `amount1Added = 0`, the callback is skipped entirely and no tokens are transferred, yet the attacker's shares are committed to storage.

The precondition — a bin with zero balances but positive total shares — is a normal on-chain state. Swap logic in `MetricOmmPool._swapToken1ForToken0SpecifiedInput` skips bins where `token0BalanceScaled == 0` without touching `_binTotalShares`:

```solidity
// MetricOmmPool.sol L978
if (binState.token0BalanceScaled == 0 || curPosInBinCache >= type(uint104).max) {
``` [4](#0-3) 

`_binTotalShares` is never decremented during swaps, so after a bin is fully drained, `binTotalShares[binIdx] > 0` while both balances are zero — a persistent, detectable state.

When the price reverses and swaps refill the bin, `removeLiquidity` computes the attacker's withdrawal proportionally:

```solidity
// LiquidityLib.sol L205-206
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [5](#0-4) 

The attacker receives tokens they never paid for, and existing LPs receive less than their proportional entitlement.

No existing guard prevents this. The only share-count check (`minimalMintableLiquidity`) validates a minimum threshold, not a zero-cost condition. [6](#0-5) 

## Impact Explanation
This is a direct theft of LP principal. Existing LPs who deposited real capital into a bin receive a smaller fraction of refilled tokens than they are owed, proportional to the attacker's free share count relative to the pre-existing `binTotalShares`. The loss is unbounded — the attacker can mint as many free shares as desired (subject only to `minimalMintableLiquidity`), and the stolen fraction approaches 100% as the free share count grows large. This constitutes pool insolvency: LP claims exceed the tokens the pool can cover for legitimate depositors.

## Likelihood Explanation
Bins being fully drained is a routine market event in any active pool. The attacker requires no privileged access — `addLiquidity` is open to any caller with any `owner` address. [7](#0-6) 
The attacker only needs to monitor `BinSwapped` events, detect when a bin's balance reaches zero while `binTotalShares` remains positive, and call `addLiquidity` before the price reverses. This is a simple, repeatable, permissionless attack.

## Recommendation
Add a guard in the `else` branch of `LiquidityLib.addLiquidity` that rejects a deposit into a non-empty bin whose balances have been fully drained. Specifically, after computing `amount0Scaled` and `amount1Scaled`, if `binTotalSharesVal > 0` and both values are zero, revert with a new `DrainedBin` error. Alternatively, require `amount0Scaled > 0 || amount1Scaled > 0` whenever `binTotalSharesVal > 0` before writing any state, enforcing the invariant that shares must always represent a non-zero token claim.

## Proof of Concept
1. Pool deployed with bins `[-1, 0, 1]`. Alice calls `addLiquidity` for bin `1` (above current price), depositing token0 and receiving 10,000 shares. `_binTotalShares[1] = 10_000`, `_binStates[1].token0BalanceScaled > 0`.
2. A series of `!zeroForOne` swaps (token1 in, token0 out) fully consume bin `1`: `_binStates[1].token0BalanceScaled = 0`, `_binStates[1].token1BalanceScaled = 0`. `_binTotalShares[1]` remains `10_000` (swaps never modify share accounting).
3. Bob calls `addLiquidity(owner=Bob, salt=0, deltas={binIdxs:[1], shares:[10_000]}, callbackData="", extensionData="")`.
   - `binTotalSharesVal = 10_000 > 0` → proportional branch taken.
   - `amount0Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - `amount1Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - `amount0Added = 0`, `amount1Added = 0` → callback skipped, no tokens transferred.
   - `_binTotalShares[1]` updated to `20_000`; `_positionBinShares[key(Bob,0,1)]` set to `10_000`. **No tokens paid.**
4. Price reverses; `zeroForOne` swaps refill bin `1` with 1,000 units of token0 (scaled). `_binStates[1].token0BalanceScaled = 1_000`.
5. Bob calls `removeLiquidity` for his 10,000 shares:
   - `amount0Scaled = 1_000 * 10_000 / 20_000 = 500` → Bob receives 500 units of token0 he never paid for.
   - Alice removes her 10,000 shares and receives only 500 units instead of the 1,000 she is owed.
   - Alice loses 500 units of token0 she legitimately deposited.
