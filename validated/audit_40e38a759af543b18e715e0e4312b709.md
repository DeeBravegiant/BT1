### Title
Accepted `EvmFinality::Latest` for BNB Smart Chain Enables Signature Issuance for Non-Finalized Transactions — (`crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The MPC system accepts `EvmFinality::Latest` for BNB Smart Chain requests at every layer — contract, node, and inspector — with no minimum finality enforcement. Because BNB's "latest" block is not finalized and can be reorged within the ~3-second block time, MPC nodes can issue a threshold signature over a `ForeignTxSignPayload` for a BNB transaction that is subsequently removed from the canonical chain. The signature remains cryptographically valid after the reorg, enabling a bridge double-spend.

---

### Finding Description

**Layer 1 — Contract (`crates/contract/src/lib.rs`)**

`verify_foreign_transaction` performs only two checks: request preconditions (gas/deposit) and whether the chain is in the supported set. There is no per-chain minimum finality enforcement. [1](#0-0) 

**Layer 2 — Node (`crates/node/src/providers/verify_foreign_tx/sign.rs`)**

The BNB branch converts the caller-supplied `EvmFinality` directly to `EthereumFinality` and passes it to the inspector without any BNB-specific floor: [2](#0-1) 

**Layer 3 — Conversion (`crates/foreign-chain-inspector/src/contract_interface_conversions.rs`)**

`TryFrom<dtos::EvmFinality> for EthereumFinality` maps `Latest → EthereumFinality::Latest` without restriction: [3](#0-2) 

**Layer 4 — Inspector (`crates/foreign-chain-inspector/src/evm/inspector.rs`)**

`verify_finality_level` maps `EthereumFinality::Latest` to `FinalityTag::Latest` and queries `eth_getBlockByNumber("latest")`. The check passes as long as the latest block number ≥ the receipt's block number — which is trivially true for any recently included transaction: [4](#0-3) 

`verify_block_is_canonical` checks the block hash at signing time only. It does **not** guarantee the block will remain canonical: [5](#0-4) 

**The signed payload**

`ForeignTxSignPayload::V1` encodes the full `request` (including `finality: EvmFinality::Latest`) and the extracted values. The signature is over `SHA-256(borsh(payload))`. Once issued, it is permanently valid regardless of subsequent chain state: [6](#0-5) 

---

### Impact Explanation

An attacker submits `verify_foreign_transaction` with `ForeignChainRpcRequest::Bnb(EvmRpcRequest { finality: EvmFinality::Latest, ... })`. MPC nodes sign the payload while the transaction is in the latest (non-finalized) block. If that block is subsequently reorged — returning the attacker's BNB to them — the already-issued MPC signature remains valid. A bridge contract that does not independently enforce finality in the payload will accept the signature and release NEAR-side funds, completing a double-spend.

Impact category: **High — forged foreign-chain verification enabling double-spend conditions.**

---

### Likelihood Explanation

BNB Smart Chain (Parlia/PoSA, 21 validators, ~3 s blocks) does experience occasional 1–2 block natural reorgs. The attacker does not need to cause a reorg — they only need to:
1. Submit a bridge deposit on BNB.
2. Immediately call `verify_foreign_transaction` with `EvmFinality::Latest`.
3. Obtain the MPC signature before the reorg is detected.
4. Redeem on NEAR while the BNB reorg returns their deposit.

The race window is narrow and natural reorgs are infrequent, making this low-probability but non-zero and requiring no privileged access.

---

### Recommendation

Enforce a minimum finality level for BNB at the node layer (or contract layer). The simplest fix is to reject any `EvmFinality::Latest` (and optionally `Safe`) for `ForeignChainRpcRequest::Bnb` in the node's BNB branch of `sign.rs`, returning an error before calling `inspector.extract`. Alternatively, add a per-chain minimum-finality policy to the on-chain foreign-chain configuration so the contract itself can reject under-finalized requests.

---

2. Confirm the conversion accepts it without error: [8](#0-7) 

3. Confirm `verify_finality_level` uses `FinalityTag::Latest` (non-finalized head) when `Latest` is passed: [9](#0-8) 

4. Confirm no BNB-specific guard exists anywhere between the contract entry point and the inspector call: [10](#0-9) 

Submit `ForeignChainRpcRequest::Bnb(EvmRpcRequest { tx_id: <valid_bnb_tx>, finality: EvmFinality::Latest, extractors: [BlockHash] })` to `verify_foreign_transaction`. The contract accepts it, the node converts `Latest` without restriction, `verify_finality_level` passes trivially, and a threshold signature is issued over a non-finalized BNB transaction.

impl TryFrom<dtos::EvmFinality> for EthereumFinality {

            dtos::EvmFinality::Latest => Ok(EthereumFinality::Latest),

    }

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
