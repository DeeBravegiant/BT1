### Title
NFT Royalty Misdistribution in Multi-NFT Offer Settlement via Equal-Split Assumption - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
When multiple royalty-enabled NFTs are traded in a single offer, Chia's wallet computes each NFT's royalty by dividing the total fungible payment equally among all NFTs by count, regardless of their individual values. When NFTs carry different royalty percentages, this equal-split assumption causes systematic misdistribution of on-chain royalty coin payments: high-royalty NFT recipients receive less than owed, and low-royalty NFT recipients receive more.

### Finding Description

Two functions in `chia/wallet/nft_wallet/nft_wallet.py` share the same flawed assumption.

**`NFTWallet.royalty_calculation()`** (the `nft_calculate_royalties` RPC handler):

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
```

The total fungible amount is divided by `len(royalty_assets_dict)` — the raw count of NFTs — before applying each NFT's individual `percentage`. Every NFT is treated as if it costs an equal share of the total price. [1](#0-0) 

**`NFTWallet.make_nft1_offer()`** (the actual spend-bundle construction path):

```python
offer_side_royalty_split += 1   # incremented once per offered NFT
...
trade_prices.append((uint64(amount // offer_side_royalty_split), settlement_ph))
...
extra_royalty_amount = compute_royalty_amount(amount, request_side_royalty_split, percentage)
```

`request_side_royalty_split` and `offer_side_royalty_split` are plain counts of NFTs. `compute_royalty_amount` then does:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [2](#0-1) [3](#0-2) 

The resulting `CreateCoin` conditions are embedded directly in the spend bundle and committed on-chain. [4](#0-3) 

### Impact Explanation

Consider a single offer: buy NFT-A (2% royalty) and NFT-B (10% royalty) for 100 XCH total.

| | Equal-split (current) | Correct (value-weighted) |
|---|---|---|
| NFT-A price assumed | 50 XCH | 10 XCH (actual) |
| NFT-A royalty paid | 1 XCH | 0.2 XCH |
| NFT-B price assumed | 50 XCH | 90 XCH (actual) |
| NFT-B royalty paid | 5 XCH | 9 XCH |

NFT-B's royalty recipient is underpaid by 4 XCH on-chain. The misdistribution is baked into the `CreateCoin` conditions inside the spend bundle, so it is final once the block is confirmed. This constitutes a direct, irreversible loss of royalty yield for the higher-royalty NFT recipient and an unearned gain for the lower-royalty recipient — an accounting change affecting XCH/CAT settlement in offer transactions.

### Likelihood Explanation

Any unprivileged user can construct a multi-NFT offer. The scenario requires only that two or more royalty-enabled NFTs with different royalty percentages appear on the same side of a single offer — a common pattern in NFT marketplace aggregation. No special privileges, leaked keys, or cryptographic breaks are needed. The `make_nft1_offer` path is reachable via the standard `create_offer_for_ids` RPC. [5](#0-4) 

### Recommendation

Replace the count-based split with a value-weighted split. When constructing `royalty_payments`, compute each NFT's share of the total fungible amount proportionally to its individual trade price (if specified in the offer), or require that multi-NFT offers specify per-NFT prices. At minimum, `compute_royalty_amount` should accept a per-NFT price rather than `total_amount // nft_count`.

### Proof of Concept

1. Alice creates an offer: `{NFT_A: +1, NFT_B: +1, XCH: -100_000_000_000_000}` where NFT_A carries 2% royalty and NFT_B carries 10% royalty.
2. Inside `make_nft1_offer`, `request_side_royalty_split = 2`.
3. `compute_royalty_amount(-100_000_000_000_000, 2, 200)` → `50_000_000_000_000 * 200 // 10000` = **1_000_000_000_000 mojos** for NFT_A.
4. `compute_royalty_amount(-100_000_000_000_000, 2, 1000)` → `50_000_000_000_000 * 1000 // 10000` = **5_000_000_000_000 mojos** for NFT_B.
5. If NFT_B was actually worth 90 XCH, its correct royalty is 9 XCH. The on-chain `CreateCoin` pays only 5 XCH — a 4 XCH shortfall committed irreversibly to the blockchain. [3](#0-2) [6](#0-5)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```

    @staticmethod
    def royalty_calculation(
        royalty_assets_dict: dict[Any, tuple[Any, uint16]],
        fungible_asset_dict: dict[Any, uint64],
    ) -> dict[Any, list[dict[str, Any]]]:
        summary_dict: dict[Any, list[dict[str, Any]]] = {}
        for id, royalty_info in royalty_assets_dict.items():
            address, percentage = royalty_info
            summary_dict[id] = []
            for name, amount in fungible_asset_dict.items():
                summary_dict[id].append(
                    {
                        "asset": name,
                        "address": address,
                        "amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
                    }
                )

        return summary_dict
