Audit Report

## Title
Missing Staleness Check in `ChainlinkPriceOracle.getAssetPrice()` Enables Stale Price to Trigger Protocol Pause or Distort rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `roundId`, `updatedAt`, and `answeredInRound`, performing no heartbeat or round-completeness validation. Because `LRTOracle.updateRSETHPrice()` is a public, permissionless function, any caller can push a stale Chainlink price into `_updateRsETHPrice()`, either inflating/deflating the rsETH exchange rate or triggering the protocol's automatic pause mechanism and temporarily freezing all deposits and withdrawals.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` (L49–55) fetches the price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`roundId`, `updatedAt`, and `answeredInRound` are all silently discarded. No `block.timestamp - updatedAt <= heartbeat` check, no `answeredInRound >= roundId` check, and no `price > 0` guard exist. [1](#0-0) 

The stale price propagates through the following call chain:

1. `LRTOracle.updateRSETHPrice()` (public, `whenNotPaused`) → `_updateRsETHPrice()` [2](#0-1) 

2. `_updateRsETHPrice()` calls `_getTotalEthInProtocol()`, which iterates over supported assets and calls `getAssetPrice(asset)` for each. [3](#0-2) 

3. `getAssetPrice()` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)`, which resolves to `ChainlinkPriceOracle.getAssetPrice()`. [4](#0-3) 

4. The resulting `newRsETHPrice` is compared against `highestRsethPrice`. If the stale price causes `newRsETHPrice` to fall below `highestRsethPrice` by more than `pricePercentageLimit`, the protocol auto-pauses `lrtDepositPool`, `withdrawalManager`, and `LRTOracle` itself: [5](#0-4) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` — used for pool collateral — does check `answeredInRound < roundID` and `timestamp == 0`, demonstrating the project is aware of these validation requirements but omitted them from the main asset pricing path. [6](#0-5) 

## Impact Explanation
**Medium — Temporary freezing of funds (scenario A):** If a Chainlink feed goes stale at a price sufficiently below `highestRsethPrice` (gap exceeds `pricePercentageLimit`, which must be non-zero for the guard to activate), calling `updateRSETHPrice()` triggers the auto-pause, freezing all deposits and withdrawals for all users until an admin manually unpauses. This is a concrete, externally triggerable freeze of user funds.

**Low — Contract fails to deliver promised returns (scenario B):** If the stale price is higher than the true price, `_getTotalEthInProtocol()` overstates TVL, `newRsETHPrice` is inflated, and new depositors receive fewer rsETH tokens than the true exchange rate warrants. Existing holders are silently enriched at depositors' expense.

The highest applicable impact is **Medium — Temporary freezing of funds**.

## Likelihood Explanation
`updateRSETHPrice()` is public and callable by any unprivileged address with no preconditions beyond the contract not already being paused. [2](#0-1) 

Chainlink feeds on Ethereum mainnet have heartbeats of 1–24 hours depending on the asset. A feed that has not updated within its heartbeat window is a realistic condition during L1 network congestion or sequencer downtime on L2s. No special access, front-running, or victim mistake is required; the attacker simply waits for a feed to go stale and calls the public function.

## Recommendation
Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale price: round not complete");
require(block.timestamp - updatedAt <= maxHeartbeat[asset], "Price too old");
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Store a per-asset `maxHeartbeat` mapping alongside `assetPriceFeed`, set at feed registration time (e.g., 3600 s for 1-hour feeds, 90 000 s for 24-hour feeds). This mirrors the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`, extended with the missing time-based check. [7](#0-6) 

## Proof of Concept

**Scenario A — Temporary freeze (Medium):**

1. `pricePercentageLimit` is set to `5e16` (5%).
2. `highestRsethPrice` is `1.05 ether` (recorded from a prior update).
3. The `stETH/ETH` Chainlink feed last updated 25 hours ago at `1.05 ETH`; the true current price is `0.98 ETH` (feed is stale-high relative to truth, but the stale value itself is below `highestRsethPrice` only if the feed stalled at a lower value — alternatively, consider a feed that stalled at `0.99 ETH` while `highestRsethPrice` is `1.05 ETH`, a ~5.7% gap).
4. Any address calls `LRTOracle.updateRSETHPrice()`.
5. `ChainlinkPriceOracle.getAssetPrice(stETH)` returns the stale `0.99e18`.
6. `_getTotalEthInProtocol()` uses the stale price; `newRsETHPrice` computes to ~`0.99 ether`.
7. `diff = 1.05e18 - 0.99e18 = 0.06e18 > 0.05 * 1.05e18 = 0.0525e18` → `isPriceDecreaseOffLimit = true`.
8. `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `LRTOracle._pause()` are called.
9. All user deposits and withdrawals are frozen until an admin calls `unpause()`.

**Foundry fork test outline:**

```solidity
function testStaleChainlinkTriggersPause() public {
    // fork mainnet, warp past feed heartbeat
    vm.warp(block.timestamp + 25 hours);
    // call public updateRSETHPrice
    lrtOracle.updateRSETHPrice();
    // assert deposit pool and withdrawal manager are paused
    assertTrue(lrtDepositPool.paused());
    assertTrue(withdrawalManager.paused());
}
```
