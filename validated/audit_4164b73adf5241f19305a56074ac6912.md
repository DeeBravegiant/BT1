Audit Report

## Title
Forced assignment in `buyToken0InBinSpecifiedIn` consumes all remaining input without updating `out0Scaled`, shortchanging the trader — (`metric-core/contracts/libraries/SwapMath.sol`)

## Summary
After the single iterative refinement in `buyToken0InBinSpecifiedIn`, a fallback forced assignment at lines 620–622 sets `totalIn1Scaled = state.amountSpecifiedRemainingScaled` without updating `out0Scaled`. The subsequent scale-down guard at line 624 only fires when `totalIn1Scaled > remaining`; after the forced assignment they are equal, so the guard is skipped entirely. The pool consumes all remaining token1 input but credits the trader with token0 output computed for a strictly smaller bin position. The identical structural bug exists in `buyToken1InBinSpecifiedIn` at lines 758–760.

## Finding Description
`buyToken0InBinSpecifiedIn` uses a three-stage approach to find the correct bin position for an exact-input token1→token0 swap:

**Stage 1 — Full-bin check** (lines 566–578): Compute cost for `maxFinalBinPos`. If affordable, set `targetPos = maxFinalBinPos` and skip the else branch. Otherwise enter the else branch, confirming the full bin costs more than `remaining`. [1](#0-0) 

**Stage 2 — Analytical + linear refinement** (lines 581–618): Compute `targetPos` via closed-form quadratic, then recompute `out0Scaled` and `totalIn1Scaled`. If still under, apply one linear proportional scale-up of `delta` and recompute again. [2](#0-1) 

**Stage 3 — Forced assignment** (lines 620–622): If `totalIn1Scaled` is still below `remaining` and `targetPos < maxFinalBinPos`, set `totalIn1Scaled = remaining` — `out0Scaled` is not touched. [3](#0-2) 

**Scale-down guard** (lines 624–634): Rescales `out0Scaled` proportionally only when `totalIn1Scaled > remaining`. After the forced assignment `totalIn1Scaled == remaining`, so the condition is false and the block is skipped — `out0Scaled` is never corrected. [4](#0-3) 

**Final accounting** (lines 643–644): Deducts all of `remaining` from `amountSpecifiedRemainingScaled` (full input consumed) and adds the under-computed `out0Scaled` to `amountCalculatedScaled` (reduced output credited). [5](#0-4) 

The root cause is that `totalIn1Scaled(targetPos)` is quadratic in `targetPos` (product of a linear `out0Scaled` and a linear `avgPrice`), while the refinement applies a linear proportional scale to `delta`. When the bin has a non-trivial price spread, the linear refinement systematically underestimates the required input, leaving `totalIn1Scaled < remaining` after Stage 2. The forced assignment then fires, setting `totalIn1Scaled = remaining` while `out0Scaled` retains the value for the under-scaled position.

The symmetric bug in `buyToken1InBinSpecifiedIn` is confirmed at lines 758–760, where `totalIn0Scaled = state.amountSpecifiedRemainingScaled` is forced without updating `out1Scaled`, and the scale-down guard at line 762 is similarly bypassed. [6](#0-5) 

## Impact Explanation
The trader pays the full `amountSpecifiedRemainingScaled` of token1 but receives token0 computed for a strictly smaller bin position. The shortfall accrues to the pool's token1 balance (line 641: `binState.token1BalanceScaled += totalIn1Scaled - protocolFeeAmountScaled`), benefiting LPs at the trader's direct expense. This is a swap conservation failure: the pool receives more input than the output it delivers is worth at the bin price. For a bin with a 10% price spread and a swap consuming ~50% of the bin, the second-order underestimate is ≈ 2.5% of the swap amount — above Sherlock Medium thresholds. [7](#0-6) 

## Likelihood Explanation
Any exact-input token1→token0 swap that (1) does not exhaust the bin to `maxFinalBinPos` (partial bin fill) and (2) operates in a bin with a non-trivial price spread (causing the linear refinement to underestimate) will trigger the forced assignment. No privileged access, malicious setup, or non-standard token behavior is required. The condition is reachable through the public `swap` entrypoint on any pool with standard bin configuration, via `_swapToken1ForToken0SpecifiedInput` → `SwapMath.buyToken0InBinSpecifiedIn`. [8](#0-7) 

## Recommendation
Replace the forced assignment with a proper correction that updates `out0Scaled` consistently:

**Option A** — Consume all remaining input and compute correct output proportionally:
```solidity
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    out0Scaled = (out0Scaled * state.amountSpecifiedRemainingScaled) / totalIn1Scaled;
    if (out0Scaled > binState.token0BalanceScaled) out0Scaled = binState.token0BalanceScaled;
    totalIn1Scaled = state.amountSpecifiedRemainingScaled;
}
```

**Option B** — Remove the forced assignment entirely and let the swap loop continue to the next bin, preserving the exact input-output relationship without approximation.

Apply the same fix symmetrically to `buyToken1InBinSpecifiedIn` at lines 758–760.

```solidity
pragma solidity ^0.8.35;

import "forge-std/Test.sol";
import {SwapMath, ONE_X64} from "src/libraries/SwapMath.sol";
import {BinState} from "src/types/PoolStorage.sol";
import {Math} from "@openzeppelin/contracts/utils/math/Math.sol";

contract ForcedAssignmentPoC is Test {
    function testFuzz_buyToken0_conservationInvariant(
        uint104 token0Balance,
        uint104 currBinPos,
        uint128 lowerPrice,
        uint128 upperPrice,
        uint128 amountIn,
        uint24 buyFeeE6
    ) public {
        token0Balance = uint104(bound(token0Balance, 1, type(uint104).max));
        lowerPrice    = uint128(bound(lowerPrice, 1, type(uint64).max));
        upperPrice    = uint128(bound(upperPrice, lowerPrice + 1, type(uint128).max / 2));
        currBinPos    = uint104(bound(currBinPos, 0, type(uint104).max - 1));
        amountIn      = uint128(bound(amountIn, 1, type(uint128).max));
        buyFeeE6      = uint24(bound(buyFeeE6, 0, 1e6));

        BinState memory binState;
        binState.token0BalanceScaled = token0Balance;

        SwapMath.SwapState memory state;
        state.amountSpecifiedRemainingScaled = amountIn;

        uint256 buyFeeX64 = (uint256(buyFeeE6) * ONE_X64) / 1e6;

        (, uint256 out0Scaled,,,) = SwapMath.buyToken0InBinSpecifiedIn(
            binState, currBinPos, state, buyFeeX64,
            lowerPrice, upperPrice, type(uint128).max, 0
        );

        // If all input was consumed, out0Scaled must be >= fair minimum output
        uint256 consumed = amountIn - state.amountSpecifiedRemainingScaled;
        if (consumed == amountIn && out0Scaled > 0) {
            uint256 minExpectedOut = Math.mulDiv(amountIn, ONE_X64, upperPrice);
            assertGe(out0Scaled, minExpectedOut,
                "Conservation failure: trader paid full input but received less than minimum expected output");
        }
    }
}
```

Run with: `forge test --match-test testFuzz_buyToken0_conservationInvariant --fuzz-runs 100000`

The fuzzer will find bin configurations (large `upperPrice/lowerPrice` ratio, mid-range `currBinPos`, partial fill) where the forced assignment fires and `out0Scaled` is below the fair minimum, demonstrating the conservation failure.
