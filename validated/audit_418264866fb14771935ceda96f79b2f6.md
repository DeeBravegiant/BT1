Audit Report

## Title
Withdrawal Queue FIFO `break` Temporarily Freezes Subsequent Users' Funds Behind Unfulfillable Head Request - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`_unlockWithdrawalRequests` in `LRTWithdrawalManager` processes the withdrawal queue in strict FIFO order and unconditionally `break`s when the vault's liquid balance is insufficient to cover the head-of-queue request. Any withdrawal requests queued behind a large unfulfillable request are stalled until that blocking request is serviced, even if sufficient liquidity exists to cover all subsequent smaller requests. Users' rsETH is already transferred into the contract at `initiateWithdrawal` time and cannot be reclaimed.

## Finding Description
`_unlockWithdrawalRequests` iterates from `nextLockedNonce[asset]` and exits the loop entirely on the first request it cannot cover:

```solidity
// contracts/LRTWithdrawalManager.sol L790-815
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    if (availableAssetAmount < payoutAmount) break; // Exit if not enough assets
    ...
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
``` [1](#0-0) 

The `break` at line 800 halts the loop and persists `nextLockedNonce[asset]` at the blocking request's position. Because `_processWithdrawalCompletion` enforces:

```solidity
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
``` [2](#0-1) 

no request behind the blocking one can ever be completed until `nextLockedNonce` advances past it. There is no skip, partial-fill, or out-of-order processing mechanism.

The `availableAssetAmount` passed to `_unlockWithdrawalRequests` is sourced from `unstakingVault.balanceOf(asset)` — the vault's raw liquid balance — not the full `totalAssetDeposits`:

```solidity
// contracts/LRTWithdrawalManager.sol L846-850
return UnlockParams({
    ...
    totalAvailableAssets: unstakingVault.balanceOf(asset)
});
``` [3](#0-2) 

Assets still restaked in EigenLayer are illiquid and cannot satisfy the payout, making the blocking condition realistic whenever the vault has not yet received a completed EigenLayer unstaking for the large request.

At `initiateWithdrawal` time, rsETH is immediately transferred into the contract and cannot be reclaimed:

```solidity
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
``` [4](#0-3) 

## Impact Explanation
**Medium — Temporary freezing of funds.** Users whose withdrawal requests are queued behind a large unfulfillable request have their rsETH locked in the contract with no recourse. The freeze persists until the blocking request is serviced. Given EigenLayer's ~7-day unstaking delay plus the protocol's own `withdrawalDelayBlocks` (up to 16 days per `setWithdrawalDelayBlocks`), the freeze can realistically last weeks. This matches the allowed impact: *Medium — Temporary freezing of funds*.

## Likelihood Explanation
**Medium.** No privileged access is required. Any rsETH holder can call `initiateWithdrawal` to place a large request at the head of the queue before other users submit smaller ones. The vault holding less liquid asset than the large request requires is the normal operating state, since assets are predominantly deployed to EigenLayer and must be explicitly unstaked first. The two required conditions (large request at head, insufficient vault liquidity) arise naturally in normal protocol operation.

## Recommendation
Replace the `break` at line 800 with `continue` so the queue processor skips requests it cannot currently fulfill and services smaller subsequent requests:

```solidity
if (availableAssetAmount < payoutAmount) continue; // Skip, try next request
```

Note that the `withdrawalDelayBlocks` check at line 795 also uses `break`; since requests are ordered by submission time, a `break` there is correct (all subsequent requests will also fail the delay check). Only the liquidity check at line 800 should become a `continue`.

## Proof of Concept
1. Protocol has 100 ETH liquid in `LRTUnstakingVault`; 900 ETH is still restaked in EigenLayer.
2. Alice calls `initiateWithdrawal(ETH, rsETH_for_500_ETH)`. Her request gets nonce 0. `assetsCommitted[ETH] += 500`.
3. Bob calls `initiateWithdrawal(ETH, rsETH_for_1_ETH)`. His request gets nonce 1. `assetsCommitted[ETH] += 1`.
4. Operator calls `unlockQueue(ETH, 2, ...)`. Inside `_unlockWithdrawalRequests`:
   - `availableAssetAmount = 100` (vault liquid balance via `unstakingVault.balanceOf`).
   - Nonce 0: `payoutAmount = 500`. `100 < 500` → **`break`**. Loop exits immediately.
   - Nonce 1 (Bob's 1 ETH request) is never reached.
5. `nextLockedNonce[ETH]` remains at 0. Bob calls `completeWithdrawal` → reverts with `WithdrawalLocked` because his nonce (1) `>= nextLockedNonce[ETH]` (0).
6. Bob's rsETH remains locked in the contract until Alice's 500 ETH request is fully serviced — potentially weeks later.

**Foundry test plan:** Deploy `LRTWithdrawalManager` on a local fork. Have Alice initiate a 500 ETH withdrawal and Bob initiate a 1 ETH withdrawal. Fund the vault with only 100 ETH. Call `unlockQueue`. Assert `nextLockedNonce[ETH] == 0` and that Bob's `completeWithdrawal` reverts with `WithdrawalLocked`.
