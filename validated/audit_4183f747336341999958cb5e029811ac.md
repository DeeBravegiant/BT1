Audit Report

## Title
Uninitialized `withdrawalDelay` and `maxNumberOfWithdrawalsPerUser` in `initialize()` Cause Temporary Fund Freeze and Withdrawal Delay Bypass - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns `withdrawalDelay` or `maxNumberOfWithdrawalsPerUser`, leaving both at Solidity's default of `0`. Because `initiateWithdrawal()` guards with `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, a value of `0` causes every withdrawal attempt to revert unconditionally, temporarily freezing all staked KERNEL tokens. Separately, `withdrawalDelay = 0` sets `unlockTime = block.timestamp`, allowing same-block claims and defeating the protocol's time-lock guarantee.

## Finding Description
`initialize()` (L259–271) sets only `kernelToken`, `rewardsToken`, and `DEFAULT_ADMIN_ROLE`; neither `withdrawalDelay` (L96) nor `maxNumberOfWithdrawalsPerUser` (L108) is assigned.

**Effect 1 — Unconditional revert on `initiateWithdrawal()`:**
At L323:
```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```
With `maxNumberOfWithdrawalsPerUser = 0`, `length >= 0` is always `true` for any `uint256`, so every call reverts with `WithdrawalLimitReached()`. All staked balances are inaccessible until admin calls `setMaxNumberOfWithdrawalsPerUser()`.

**Effect 2 — Zero-delay withdrawal window:**
At L330:
```solidity
uint256 unlockTime = block.timestamp + withdrawalDelay; // = block.timestamp when delay is 0
```
The `claimWithdrawal()` check at L355 (`block.timestamp < withdrawal.unlockTime`) is immediately false, so withdrawals are claimable in the same block they are initiated.

Both setters correctly reject `0` (L599, L614), but this only prevents the admin from re-setting to `0`; it does not prevent the contract from starting in that state.

## Impact Explanation
**Primary — Medium. Temporary freezing of funds:** Every user who has staked KERNEL via `stake()` or `stakeFor()` is completely unable to initiate a withdrawal from the moment of deployment until the admin separately calls `setMaxNumberOfWithdrawalsPerUser()`. There is no on-chain enforcement requiring this post-deployment call, so the freeze is the default state of any freshly deployed proxy.

**Secondary — Low. Contract fails to deliver promised returns:** With `withdrawalDelay = 0`, the time-lock protection is absent; users can stake and claim in the same block, defeating the protocol's stated withdrawal delay guarantee without any loss of principal.

## Likelihood Explanation
Likelihood is high for the freeze scenario. The `initialize()` function is the sole initialization path for an upgradeable proxy. If the admin does not separately call `setMaxNumberOfWithdrawalsPerUser()` after deployment — a step that is neither enforced on-chain nor documented in the initializer — the contract is immediately broken for all withdrawers. Any normal user attempting to withdraw after staking will trigger the revert. No attacker capability is required; the broken state is the default.

## Recommendation
1. Initialize both variables to safe defaults inside `initialize()`:
   ```solidity
   withdrawalDelay = 7 days;
   maxNumberOfWithdrawalsPerUser = 10;
   ```
2. Add a minimum threshold check in `setWithdrawalDelay()` (e.g., `>= 1 hours`) analogous to the `MAX_WITHDRAWAL_DELAY` upper bound already present at L600.
3. Emit initialization events so off-chain monitoring can confirm correct deployment state.

## Proof of Concept
1. Deploy `KernelDepositPool` proxy and call `initialize(admin, kernelToken, rewardToken)`.
2. Admin does **not** call `setMaxNumberOfWithdrawalsPerUser()` (no on-chain enforcement requires it).
3. User calls `stake(1e18)` — succeeds; KERNEL tokens are transferred in.
4. User calls `initiateWithdrawal(1e18)`:
   - `userWithdrawalIds[user].length` = `0`
   - `maxNumberOfWithdrawalsPerUser` = `0`
   - `0 >= 0` → `true` → reverts `WithdrawalLimitReached()`
5. User's KERNEL tokens are frozen in the contract with no recourse until admin intervention.

**Foundry test sketch:**
```solidity
function test_withdrawalFrozenOnDeploy() public {
    // Deploy proxy, call initialize without setMaxNumberOfWithdrawalsPerUser
    pool.initialize(admin, address(kernelToken), address(rewardToken));
    kernelToken.approve(address(pool), 1e18);
    pool.stake(1e18);
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(1e18);
}
```
