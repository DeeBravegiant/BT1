Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` + Public `LRTOracle.updateRSETHPrice()` Enables Sandwich Attack to Steal Yield from Existing rsETH Holders - (File: `contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`)

## Summary

`FeeReceiver.sendFunds()` carries no access control and `LRTOracle.updateRSETHPrice()` is callable by any unprivileged address. An attacker can atomically deposit ETH at the stale (pre-reward) rsETH price, flush pending MEV/execution-layer rewards into the deposit pool, and trigger a price update — capturing a disproportionate share of rewards that belong to pre-existing holders. The `pricePercentageLimit` guard is a partial mitigation but is bypassable when set to zero or when reward accumulations are small enough to stay within the configured threshold.

## Finding Description

**Root cause — two permissionless state-changing functions create a sandwichable reward window.**

`FeeReceiver.sendFunds()` has no role modifier: [1](#0-0) 

`LRTOracle.updateRSETHPrice()` is `public` with only a `whenNotPaused` guard: [2](#0-1) 

**Price calculation uses current supply against post-flush TVL.**

`_updateRsETHPrice()` computes `previousTVL` as `rsethSupply × rsETHPrice` (using the *current* supply, which already includes the attacker's freshly minted rsETH) and then computes the new price against `totalETHInProtocol` which now includes the flushed rewards: [3](#0-2) 

This means the attacker's rsETH participates in the reward distribution even though those rewards were already pending before the deposit.

**Exploit flow:**

1. Observe `FeeReceiver` balance on-chain (e.g., 10 ETH pending rewards, current price = 1.0 ETH/rsETH).
2. Call `LRTDepositPool.depositETH{value: 10 ether}()` — mints 10 rsETH at the stale price. Pool: 110 ETH, supply: 110 rsETH.
3. Call `FeeReceiver.sendFunds()` — 10 ETH rewards transferred to pool. Pool: 120 ETH, supply: 110 rsETH.
4. Call `LRTOracle.updateRSETHPrice()` — new price ≈ 1.0818 ETH/rsETH (at 10% protocol fee). Protocol fee rsETH is minted to treasury.
5. Call `LRTWithdrawalManager.initiateWithdrawal()` — `expectedAssetAmount` is locked at the inflated price (≈ 10.818 ETH for 10 rsETH).
6. After 8-day delay and operator `unlockQueue()` call, `completeWithdrawal()` pays out 10.818 ETH.

**Why existing guards are insufficient.**

The `pricePercentageLimit` guard reverts the public `updateRSETHPrice()` call if the price increase exceeds the configured threshold: [4](#0-3) 

However: (a) `pricePercentageLimit` defaults to 0, disabling the guard entirely; (b) even when set, the attacker can target smaller reward accumulations that stay within the threshold; (c) the guard only blocks the public path — the underlying accounting flaw (stale price window) remains.

The `_calculatePayoutAmount` min() in `unlockQueue` only protects against price *decreases* after initiation, not against the initial inflation: [5](#0-4) 

No deposit lockup exists — `depositETH` mints rsETH immediately: [6](#0-5) 

## Impact Explanation

**High — Theft of unclaimed yield.**

Using the example above (ignoring fee for simplicity): the attacker deposits 10 ETH, captures ~0.909 ETH of the 10 ETH reward that belonged to existing holders, and existing 100 rsETH holders receive ~108.18 ETH instead of 109 ETH. The stolen fraction scales linearly with the ratio of attacker deposit to total TVL and with the size of the pending reward balance. For large MEV accumulations or low-TVL periods, the stolen yield is material. This directly matches the "High. Theft of unclaimed yield" impact class.

## Likelihood Explanation

**High.** Both trigger functions are permissionless and callable by any EOA or contract. The `FeeReceiver` balance is publicly visible on-chain, enabling a bot to monitor it and execute the sandwich atomically. No flash loan, oracle manipulation, or privileged access is required. The 8-day withdrawal delay (`withdrawalDelayBlocks = 8 days / 12 seconds`) does not prevent the attack — the profit is locked in at `initiateWithdrawal` time and the attacker simply waits: [7](#0-6) 

The attack is repeatable every time rewards accumulate in `FeeReceiver`.

## Recommendation

1. **Add access control to `FeeReceiver.sendFunds()`** — restrict to `MANAGER` or `OPERATOR` role so reward flushing cannot be triggered permissionlessly.
2. **Atomically flush and update price** — have the authorised caller flush rewards and call `updateRSETHPrice()` in the same transaction, eliminating the window between the two operations.
3. **Ensure `pricePercentageLimit` is always set to a non-zero value** as a defence-in-depth measure, but note this alone is insufficient since it can be bypassed with small reward accumulations.
4. **Alternatively, implement a deposit lockup** — prevent newly minted rsETH from being used in `initiateWithdrawal` for at least one price-update cycle.

## Proof of Concept

```
// State:
//   LRTDepositPool: 100 ETH, rsETH supply: 100, rsETHPrice: 1.0 ETH (stale)
//   FeeReceiver: 10 ETH pending rewards

// Tx 1: Attacker deposits 10 ETH at stale price
LRTDepositPool.depositETH{value: 10 ether}(0, "")
// rsethAmountToMint = 10e18 * 1e18 / 1e18 = 10e18
// Pool: 110 ETH, supply: 110 rsETH

// Tx 2: Attacker flushes rewards
FeeReceiver.sendFunds()
// Pool: 120 ETH, supply: 110 rsETH

// Tx 3: Attacker triggers price update
LRTOracle.updateRSETHPrice()
// previousTVL = 110 * 1.0 = 110 ETH
// totalETHInProtocol = 120 ETH
// rewardAmount = 10 ETH, protocolFeeInETH = 1 ETH (10% fee)
// newRsETHPrice = (120 - 1) / 110 ≈ 1.0818 ETH/rsETH

// Tx 4: Attacker initiates withdrawal
LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 10e18, "")
// expectedAssetAmount = 10 * 1.0818 / 1.0 = 10.818 ETH (locked in)

// After 8 days + operator unlockQueue():
LRTWithdrawalManager.completeWithdrawal(ETH_TOKEN, "")
// Attacker receives 10.818 ETH
// Net profit: 0.818 ETH stolen from existing 100 rsETH holders
// Existing holders: 108.18 ETH instead of 109 ETH
```

Foundry fork test plan: fork mainnet, impersonate an EOA, execute the four-transaction sequence above against live contract addresses, assert attacker ETH balance after `completeWithdrawal` exceeds initial deposit by the expected reward fraction.
