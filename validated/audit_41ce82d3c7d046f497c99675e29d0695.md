### Title
Attached NEAR deposit silently consumed when existing participant re-submits attestation - (File: crates/contract/src/lib.rs)

### Summary
`submit_participant_info` is `#[payable]` but only enters its deposit-refund block when `attestation_storage_must_be_paid_by_caller` is `true`. When an already-registered participant re-submits their attestation (e.g., after a TEE upgrade), that flag is `false` and any NEAR tokens attached to the call are permanently frozen in the chain-signature contract with no check and no refund.

### Finding Description

`submit_participant_info` computes a boolean guard:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... deposit check and refund ...
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}

Ok(())
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` is `false` — which happens whenever the caller is already a registered participant **and** the submission is an update rather than a first-time insertion — the entire `if` block is skipped. `env::attached_deposit()` is never read, never checked, and never refunded. The NEAR tokens are absorbed into the contract's balance with no recovery path.

Compare this with the `require_deposit` helper used by `sign()` and `request_app_private_key()`, which always refunds excess:

```rust
Some(diff) => {
    if diff > NearToken::from_yoctonear(0) {
        log!("refund excess deposit {diff} to {predecessor}");
        Promise::new(predecessor.clone()).transfer(diff).detach();
    }
}
``` [2](#0-1) 

`submit_participant_info` has no equivalent unconditional refund path.

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are permanently frozen inside the chain-signature contract. The contract exposes no withdrawal or sweep mechanism for such stranded balances. This is a direct, permanent loss of funds held by the chain-signature contract, matching the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

Re-submission is a normal, expected operation: participants must refresh their attestation after TEE image upgrades or TLS key rotations. A participant who is uncertain about the required deposit amount — or who copies the pattern from their initial registration call — will attach NEAR tokens. Because the function is `#[payable]` with no guard that rejects a non-zero deposit in the re-submission path, the mistake is silent and unrecoverable. Likelihood is **low-medium**: the operation is routine but the deposit attachment is an error of omission rather than a forced condition.

### Recommendation

Add an explicit zero-deposit assertion in the branch where storage costs need not be paid by the caller, mirroring the fix suggested in the reference report:

```rust
if attestation_storage_must_be_paid_by_caller {
    // existing deposit check + refund logic
} else {
    // Existing participant re-submission: no storage cost, so no deposit should be attached.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

This ensures that any accidentally attached deposit is always returned to the caller regardless of which branch is taken.

### Proof of Concept

1. Node `alice.near` calls `submit_participant_info` for the first time, attaching the required storage deposit. `is_new_attestation = true`, so the deposit block runs and any excess is refunded. Alice is now a registered participant.

2. A TEE image upgrade occurs. Alice must re-submit her attestation. She calls `submit_participant_info` again, attaching 1 NEAR "just in case" (same pattern as step 1).

3. Inside the function:
   - `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` → `is_new_attestation = false`
   - `self.voter_account()` succeeds (Alice is a participant) → `caller_is_not_participant = false`
   - `attestation_storage_must_be_paid_by_caller = false || false = false`
   - The `if` block is skipped entirely.
   - `Ok(())` is returned.

4. Alice's 1 NEAR is now permanently held by the contract. No error is emitted, no refund is issued, and no contract method exists to recover it. [3](#0-2)
