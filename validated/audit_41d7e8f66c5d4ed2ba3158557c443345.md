Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` checks router address instead of actual swapper, enabling allowlist bypass — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is intended to gate `pool.swap` by individual swapper identity per pool. However, `MetricOmmPool.swap` passes `msg.sender` (the direct caller) as `sender` to `_beforeSwap`, which is the router address when users route through `MetricOmmSimpleRouter`. The extension then checks `allowedSwapper[pool][router]` instead of `allowedSwapper[pool][actual_user]`, completely nullifying per-user access control for router-mediated swaps.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol L230-231
_beforeSwap(
  msg.sender,  // router address when called via MetricOmmSimpleRouter
  ...
```

`ExtensionCalling._beforeSwap` forwards this verbatim to all configured extensions:

```solidity
// metric-core/contracts/ExtensionCalling.sol L160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then evaluates:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is the router — so the check is `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actual_user]`.

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router `msg.sender` inside the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol L72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ...);
```

**Exploit path (allowlist bypass):**
1. Pool admin deploys pool with `SwapAllowlistExtension` to restrict swaps to KYC'd users.
2. Admin calls `setAllowedToSwap(pool, router, true)` so legitimate users can use the periphery.
3. Unpermissioned attacker calls `router.exactInputSingle({pool: pool, recipient: attacker, ...})`.
4. Router calls `pool.swap(attacker, ...)` — `msg.sender` inside pool is the router.
5. `beforeSwap` receives `sender = router`; `allowedSwapper[pool][router] == true` → check passes.
6. Attacker receives output tokens despite never being individually allowlisted.

**Denial-of-service path:** If the admin does NOT allowlist the router (only individual users), every allowlisted user who routes through the router is blocked with `NotAllowedToSwap`, forcing direct pool interaction and losing slippage protection, deadline enforcement, and multi-hop routing.

There is no mechanism in the current interface for the router to pass the real originating user identity — `extensionData` is passed through from `params.extensionData` (user-controlled), so it cannot be trusted as an identity proof without a trusted-router registry.

## Impact Explanation
**Scenario A (Critical):** Unpermissioned users bypass a pool's KYC/allowlist access control entirely by routing through `MetricOmmSimpleRouter`. The allowlist invariant — that only individually approved addresses may swap — is nullified for all router-mediated swaps. This is an admin-boundary break triggered by an unprivileged trader with no special preconditions beyond the router being allowlisted.

**Scenario B (Medium):** If the router is not allowlisted, all individually allowlisted users are denied access to the primary swap interface, breaking core swap functionality for legitimate users.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public-facing swap interface. Any pool deploying `SwapAllowlistExtension` for access control faces this structural failure the moment they allowlist the router. The bug requires no special attacker capability — any EOA can call `exactInputSingle`. The pool admin has no configuration option to fix this without modifying the extension or pool interface.

## Recommendation
The `beforeSwap` hook must identify the economically relevant actor, not the direct pool caller. Two viable approaches:

1. **Trusted router + `extensionData` encoding:** The router encodes `msg.sender` (the real user) into `extensionData` before calling `pool.swap`. The extension decodes this only when `msg.sender` (the pool's caller) is a known trusted router, verified against a factory-maintained router registry.

2. **Pool interface extension:** Add a `payer`/`originator` field to `pool.swap` distinct from `msg.sender`, populated by the router with the real user address, and forwarded to extensions.

As a minimal mitigation, document that `SwapAllowlistExtension` gates the direct pool caller (router address), not individual users — but this fundamentally changes the security model the extension advertises and does not fix the bypass.

## Proof of Concept
```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin: setAllowedToSwap(pool, router, true)
   (router allowlisted so legitimate users can use periphery)
3. Attacker (not individually allowlisted) calls:
   router.exactInputSingle({pool: pool, recipient: attacker, amountIn: X, ...})
4. Router executes: pool.swap(attacker, zeroForOne, amountIn, ...)
   → msg.sender inside pool.swap = router
5. _beforeSwap(sender=router, ...) → SwapAllowlistExtension.beforeSwap(sender=router, ...)
   → allowedSwapper[pool][router] == true → passes
6. Attacker receives output tokens. Allowlist bypassed.

Foundry test: assert swap succeeds for address not in allowedSwapper[pool][attacker]
when router is in allowedSwapper[pool][router].
```
