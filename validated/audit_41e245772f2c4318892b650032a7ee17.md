Audit Report

## Title
`instantWithdrawal()` pays out at stale pre-slash `rsETHPrice`, enabling extraction of excess assets before oracle update — (File: contracts/LRTWithdrawalManager.sol)

## Summary
`instantWithdrawal()` computes the asset payout using `lrtOracle.rsETHPrice()`, a stored state variable that is only updated when `updateRSETHPrice()` is explicitly called. After an EigenLayer slashing event reduces the protocol's underlying assets, a window exists before `updateRSETHPrice()` is invoked during which any rsETH holder can call `instantWithdrawal()` and receive assets at the inflated pre-slash rate. The resulting shortfall is absorbed by remaining rsETH holders when the oracle is eventually updated.

## Finding Description
`instantWithdrawal()` at line 228 of `contracts/LRTWithdrawalManager.sol` calls `getExpectedAssetAmount()`:

```solidity
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
```

`getExpectedAssetAmount()` (lines 580–594) reads `rsETHPrice` directly from the oracle's stored state:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

`rsETHPrice` in `LRTOracle` (line 28) is a plain state variable written only inside `_updateRsETHPrice()` (line 313), which is only reachable via explicit calls to `updateRSETHPrice()` or `updateRSETHPriceAsManager()`. There is no forced price refresh inside `instantWithdrawal()`.

The auto-pause downside-protection (lines 270–281 of `LRTOracle.sol`) only fires when `_updateRsETHPrice()` is actually executed. Until that call happens, the protocol is not paused and `instantWithdrawal()` remains open at the stale price.

By contrast, the queued withdrawal path uses `_calculatePayoutAmount()` (lines 833–834), which takes the **minimum** of the amount locked at request time and the amount recomputed at unlock time:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

`instantWithdrawal()` has no equivalent minimum-of-locked-vs-current guard. It pays whatever `rsETHPrice` is stored at call time, with no delay and no freshness check.

The `isInstantWithdrawalEnabled[asset]` gate (lines 78–81) is an operational configuration toggle, not a permanent barrier, and does not mitigate the stale-price issue when enabled.

## Impact Explanation
An rsETH holder who observes an EigenLayer slashing event on-chain before `updateRSETHPrice()` is called can call `instantWithdrawal()` and receive assets computed at the pre-slash `rsETHPrice`. The excess assets come from the `LRTUnstakingVault`. When `updateRSETHPrice()` is eventually called, the price drops to reflect the slash, and remaining rsETH holders bear the full slashing loss that the attacker escaped. This constitutes a direct transfer of value from remaining rsETH holders to the attacker.

**Impact: High — Theft of unclaimed yield / value from remaining rsETH holders.**

## Likelihood Explanation
- EigenLayer slashing is an explicitly anticipated event; `SlashingLib` is imported and used in `LRTUnstakingVault` (line 21).
- `updateRSETHPrice()` is not called atomically with slashing; an observable delay always exists between the on-chain slashing and the oracle update.
- `rsETHPrice` is a public state variable; any on-chain observer can detect staleness by comparing it against a fresh `_getTotalEthInProtocol()` computation.
- No special privileges are required beyond holding rsETH.
- The attack is repeatable across multiple slashing events and by multiple independent actors in the same window.
- The only operational precondition is `isInstantWithdrawalEnabled[asset] == true`, which is a normal operational state for the feature to be usable at all.

## Recommendation
1. In `instantWithdrawal()`, call `updateRSETHPrice()` (or an equivalent fresh price computation) before computing `assetAmountUnlocked`, so the payout always reflects the current state of the protocol.
2. Alternatively, apply the same `_calculatePayoutAmount()` minimum-of-locked-vs-current logic used in the queued withdrawal path to `instantWithdrawal()`, capping the payout at the lower of the price at call time and the price at some prior checkpoint.
3. Consider adding at minimum a single-block delay to `instantWithdrawal()` to prevent same-block front-running of oracle updates.

## Proof of Concept
1. An EigenLayer slashing event is finalized on-chain; the NodeDelegator's share balance decreases, reducing the true ETH value backing rsETH.
2. `rsETHPrice` in `LRTOracle` remains at the pre-slash value because `updateRSETHPrice()` has not yet been called.
3. Attacker (rsETH holder) calls `instantWithdrawal(asset, rsETHAmount, "")` before any call to `updateRSETHPrice()`.
4. `getExpectedAssetAmount()` computes `rsETHAmount * rsETHPrice_stale / assetPrice`, returning a larger asset amount than the post-slash fair value.
5. The check `assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)` passes because the vault still holds pre-slash assets.
6. Attacker receives the inflated asset amount via `unstakingVault.redeem()`.
7. `updateRSETHPrice()` is eventually called; `rsETHPrice` drops to reflect the slash.
8. Remaining rsETH holders now hold rsETH worth less than before the slash, having absorbed the full slashing loss the attacker escaped.

**Foundry fork test outline:**
- Fork mainnet/testnet at a block before a known slashing event (or simulate one by directly reducing a NodeDelegator's EigenLayer share balance in a local fork).
- Assert `lrtOracle.rsETHPrice()` is stale (higher than `_getTotalEthInProtocol() / rsethSupply`).
- Call `instantWithdrawal()` as an unprivileged rsETH holder.
- Assert the received asset amount exceeds `rsETHAmount * freshPrice / assetPrice`.
- Call `updateRSETHPrice()` and assert `rsETHPrice` drops.
- Assert remaining holders' rsETH is now worth less by the exact amount the attacker over-received.
