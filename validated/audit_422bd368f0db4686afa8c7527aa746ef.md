### Title
NEAR→Solana Transfers With Non-Empty `DestHexMsg` Produce Permanently Unverifiable Signatures, Locking User Funds - (File: `near/omni-types/src/lib.rs`, `solana/programs/bridge_token_factory/src/state/message/finalize_transfer.rs`)

---

### Summary

When a user initiates a NEAR→Solana transfer and includes a `DestHexMsg` in the `msg` field, NEAR's `sign_transfer` produces an MPC signature over a Borsh payload that includes the `message` bytes (V2 encoding). Solana's `FinalizeTransferPayload::serialize_for_near` never includes a `message` field, so it always reconstructs a V1-style hash. The two hashes never match, signature verification always fails, and the user's tokens are permanently locked in the NEAR bridge with no recovery path.

---

### Finding Description

**Step 1 — NEAR signs a V2 payload when `message` is non-empty.**

`sign_transfer` in `near/omni-bridge/src/lib.rs` extracts the destination message:

```rust
let message = DestinationChainMsg::from_json(&transfer_message.msg)
    .and_then(|s| s.destination_msg())
    .unwrap_or_default();
``` [1](#0-0) 

`destination_msg()` returns `Some(Vec<u8>)` for the `DestHexMsg` variant:

```rust
pub fn destination_msg(&self) -> Option<Vec<u8>> {
    if let Self::DestHexMsg(msg) = self { Some(msg.clone()) } else { None }
}
``` [2](#0-1) 

The resulting `message` bytes are placed into `TransferMessagePayload.message` and then `encode_hashable()` is called:

```rust
pub fn encode_hashable(&self) -> Result<Vec<u8>, String> {
    if self.message.is_empty() {
        borsh::to_vec(&TransferMessagePayloadV1::from(self.clone())).map_err(stringify)
    } else {
        borsh::to_vec(self).map_err(stringify)   // ← V2: includes `message` field
    }
}
``` [3](#0-2) 

`TransferMessagePayload` (V2) carries the extra `message: Vec<u8>` field that `TransferMessagePayloadV1` does not: [4](#0-3) 

The MPC therefore signs `keccak256(borsh(V2_payload_with_message))`.

---

**Step 2 — Solana's verification payload never includes `message`.**

Solana's `FinalizeTransferPayload::serialize_for_near` reconstructs the hash for ECDSA verification:

```rust
fn serialize_for_near(&self, params: Self::AdditionalParams) -> Result<Vec<u8>> {
    // prefix, destination_nonce, transfer_id, token, amount, recipient, fee_recipient
    // ← NO message field written at any point
}
``` [5](#0-4) 

`FinalizeTransferPayload` itself has no `message` member: [6](#0-5) 

Solana therefore always computes `keccak256(borsh(V1_payload_without_message))`.

---

**Step 3 — Signature verification always fails; funds are permanently locked.**

In `finalize_transfer`:

```rust
data.verify_signature(
    (ctx.accounts.mint.key(), ctx.accounts.recipient.key()),
    &ctx.accounts.common.config.derived_near_bridge_address,
)?;
``` [7](#0-6) 

The recovered address will never equal `derived_near_bridge_address` because the hash Solana reconstructs (V1, no message) differs from the hash NEAR signed (V2, with message). Every submission reverts. The tokens remain locked in the NEAR bridge with no on-chain cancellation or refund mechanism.

---

**Contrast with EVM/Starknet/Aptos — they handle `message` correctly.**

EVM's `finTransfer` conditionally appends the message bytes, matching NEAR's V1/V2 branching:

```solidity
bytes(payload.message).length == 0
    ? bytes("")
    : Borsh.encodeBytes(payload.message)
``` [8](#0-7) 

Starknet and Aptos do the same — append nothing when `message` is `None`, append encoded bytes when `Some`. Solana is the only chain that has no `message` field at all in its verification path.

---

### Impact Explanation

**Critical — Irreversible fund lock.** A user who initiates a NEAR→Solana transfer with `msg = '{"DestHexMsg":"<hex>"}'` will have their tokens locked in the NEAR bridge permanently. NEAR signs a V2 payload; Solana can only verify V1 payloads. Every `finalize_transfer` call reverts. There is no on-chain cancel or refund path visible in the codebase.

---

### Likelihood Explanation

The `msg` field in `InitTransferMsg` is a public, user-controlled input with no destination-chain validation:

```rust
msg: init_transfer_msg.msg.map(String::from).unwrap_or_default(),
``` [9](#0-8) 

Any user who passes `{"DestHexMsg":"..."}` for a Solana-destined transfer triggers the lock. The protocol accepts the transfer without error, giving the user no indication that their funds will be unrecoverable.

---

### Recommendation

Two complementary fixes:

1. **Solana-side**: Add a `message` field to `FinalizeTransferPayload` and include it in `serialize_for_near`, matching the NEAR V2 encoding when non-empty.

2. **NEAR-side (defensive)**: In `sign_transfer`, validate that `DestHexMsg` is only set for chains whose destination contracts support the `message` field (EVM, Starknet, Aptos). Reject or strip the message for Solana transfers before signing.

---

### Proof of Concept

1. User calls `ft_transfer_call` on a NEAR token with:
   ```json
   {
     "recipient": "sol:<solana_address>",
     "fee": "0",
     "native_token_fee": "0",
     "msg": "{\"DestHexMsg\":\"deadbeef\"}"
   }
   ```
2. NEAR bridge stores the transfer; tokens are locked.
3. Relayer calls `sign_transfer`. NEAR extracts `message = [0xde, 0xad, 0xbe, 0xef]` (non-empty), builds a V2 `TransferMessagePayload`, and the MPC signs `keccak256(borsh(V2))`.
4. Relayer submits `finalize_transfer` to Solana with `FinalizeTransferPayload` (no message field). Solana computes `keccak256(borsh(V1))` — a different hash. `verify_signature` returns `InvalidSignature`.
5. Every subsequent `finalize_transfer` attempt fails identically. Tokens are permanently locked in the NEAR bridge.
