### Title
Caller-Supplied `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` accepts a caller-supplied `response.payload_hash` and only checks that the submitted ECDSA signature is valid over that hash against the **root public key**. It never verifies that `payload_hash` encodes the `ForeignChainRpcRequest` belonging to the pending request being resolved. A single attested participant can replay a root-key signature obtained from any prior signing session — for a completely different `ForeignChainRpcRequest` — to resolve an unrelated pending foreign-tx verification request with a fabricated `payload_hash`, delivering a forged verification result to the waiting bridge caller.

---

### Finding Description

`verify_foreign_transaction` stores a `VerifyForeignTransactionRequest` (containing `ForeignChainRpcRequest`, `domain_id`, `payload_version`) as the pending-request key and parks the caller's yield. [1](#0-0) 

When `respond_verify_foreign_tx` is later called by an attested participant, the contract:

1. Verifies the ECDSA signature over `response.payload_hash` against the **root public key** (no tweak, no request binding).
2. Looks up the pending request by the caller-supplied `request` key.
3. Serialises the entire `response` (including the unverified `payload_hash`) and delivers it to the waiting yield. [2](#0-1) 

The contract performs **no check** that `response.payload_hash` is `SHA-256(borsh(ForeignTxSignPayload { request: <the pending ForeignChainRpcRequest>, values: … }))`. The `payload_hash` field is defined as a plain `Hash256` in the response DTO with no binding to the stored request. [3](#0-2) 

Contrast this with `respond` for ordinary signatures, which derives the expected public key from `request.tweak` (itself derived from the predecessor and path at submission time), cryptographically binding the signature to the specific request: [4](#0-3) 

For foreign-tx responses the root key is used with no tweak, so any root-key signature over **any** 32-byte value passes the check. The `payload_hash` is the only field that could bind the response to the correct foreign-chain observation, and it is never validated on-chain.

The off-chain node constructs `payload_hash` by computing `ForeignTxSignPayload::compute_msg_hash()` over the observed extracted values, then passes it as a parameter to `respond_verify_foreign_tx`: [5](#0-4) [6](#0-5) 

Because the contract never recomputes or constrains `payload_hash`, the two-step split (off-chain signing → on-chain respond) is exploitable by a single participant in exactly the same way the BWGToken report describes: the privileged caller supplies the "computed" value rather than the contract deriving it.

---

### Impact Explanation

A single attested participant (strictly below threshold) executes the following:

1. **Collect a prior root-key signature.** Participate honestly in signing session for request A (`tx_id = 0xAAAA`). The MPC network produces `(payload_hash_A, sig_A)` where `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload { request: RpcRequest_A, values: [BlockHash(0x1111)] }))`.

2. **Target a different pending request.** A bridge service has submitted `verify_foreign_transaction` for request B (`tx_id = 0xBBBB`), which is now pending in `pending_verify_foreign_tx_requests`.

3. **Submit the replayed response.** Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`. The contract finds `request_B` in the pending map, verifies `sig_A` over `payload_hash_A` against the root key — which succeeds — and resolves the yield with `payload_hash_A`.

4. **Bridge service receives forged result.** The bridge caller for request B receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: sig_A }`. `payload_hash_A` encodes `RpcRequest_A` and its extracted values, not `RpcRequest_B`. Any downstream logic that trusts the contract's response without independently recomputing the expected hash from the foreign chain accepts a forged foreign-chain verification.

The `ForeignChainSignatureVerifier` in the SDK does check `expected_payload_hash == response.payload_hash`, but it requires the caller to supply `expected_extracted_values` independently: [7](#0-6) 

Bridge services that do not independently re-query the foreign chain — the common case, since the entire purpose of `verify_foreign_transaction` is to outsource that query — cannot detect the substitution. The contract is the intended trust anchor; its failure to bind `payload_hash` to the pending request is the root cause.

Impact category: **High — forged foreign-chain verification causing invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

- The attacker is a **single attested MPC participant**, strictly below the signing threshold. No collusion is required.
- The attacker needs only a root-key signature over some `payload_hash` — obtained by participating in any prior legitimate signing session for any foreign-tx request.
- The attack is a direct function call to a public contract method; no network-level access or TEE bypass is required.
- The window is bounded by the yield timeout (~200 blocks), but the attacker controls the timing of the `respond_verify_foreign_tx` call and can race honest nodes.

---

### Recommendation

The contract must bind `response.payload_hash` to the pending request on-chain. Two complementary approaches:

1. **Include extracted values in the response and recompute on-chain.** Extend `VerifyForeignTransactionResponse` with the `Vec<ExtractedValue>` the nodes observed. In `respond_verify_foreign_tx`, recompute `expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request.clone(), values: response.values.clone() }).compute_msg_hash()` and assert `expected_hash == response.payload_hash`. This binds the hash to the pending `ForeignChainRpcRequest` and prevents cross-request replay. The contract still cannot verify the correctness of the extracted values themselves (that requires querying the foreign chain), but it eliminates the ability to substitute a hash from a different request.

2. **Derive a request-specific signing key (tweak).** Mirror the `respond` pattern: derive a tweak from the `ForeignChainRpcRequest` (e.g., its Borsh hash) and verify the signature against the derived key rather than the root key. This cryptographically prevents any signature produced for one request from being accepted for another.

### Proof of Concept

```
Setup:
  - MPC network running with participants [P1 (attacker), P2, P3], threshold = 2
  - ForeignTx domain configured for Bitcoin

Step 1 — Honest signing for request A:
  Bridge_A calls verify_foreign_transaction({ tx_id: 0xAAAA, confirmations: 6, extractors: [BlockHash] })
  MPC nodes query Bitcoin RPC → block_hash = 0x1111
  Threshold signing produces:
    payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1 { request: RpcRequest_A, values: [BlockHash(0x1111)] }))
    sig_A = ECDSA_sign(root_key, payload_hash_A)
  P1 (attacker) records (payload_hash_A, sig_A) but does NOT submit respond_verify_foreign_tx.
  P2 submits respond_verify_foreign_tx(request_A, { payload_hash_A, sig_A }) → resolves Bridge_A normally.

Step 2 — Target request B:
  Bridge_B calls verify_foreign_transaction({ tx_id: 0xBBBB, confirmations: 6, extractors: [BlockHash] })
  request_B is now pending in pending_verify_foreign_tx_requests.

Step 3 — Attacker submits replayed response:
  P1 calls respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })

  Contract checks:
    assert_caller_is_attested_participant_and_protocol_active() → PASS (P1 is attested)
    verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → PASS (valid signature)
    pending_verify_foreign_tx_requests.get(request_B) → FOUND
    resolve_yields_for(request_B, serialize({ payload_hash_A, sig_A })) → resolves yield

Step 4 — Bridge_B receives forged result:
  Bridge_B gets VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: sig_A }
  payload_hash_A encodes tx_id=0xAAAA and block_hash=0x1111, not tx_id=0xBBBB.
  Bridge_B, trusting the contract as the verification oracle, accepts the response as proof
  that tx_id=0xBBBB was confirmed — a forged foreign-chain verification.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
