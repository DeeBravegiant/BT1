Audit Report

## Title
Stale `rsETHPrice` Read Without Prior Update in Deposit Minting Leads to Excess rsETH Issued — (File: `contracts/LRTDepositPool.sol`)

## Summary
`depositETH` and `depositAsset` compute the rsETH mint amount via `getRsETHAmountToMint`, which divides by `lrtOracle.rsETHPrice()` — a cached storage variable in `LRTOracle` that is only written when `updateRSETHPrice()` is explicitly called. Neither deposit function triggers a price update before reading it. When staking rewards have accrued since the last update, the stored price is lower than the true TVL-derived rate, causing depositors to receive more rsETH than they are entitled to, diluting existing holders' accrued yield.

## Finding Description
`_beforeDeposit` (called by both `depositETH` and `depositAsset`) delegates directly to `getRsETHAmountToMint` with no price refresh:

```solidity
// LRTDepositPool.sol L665
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
```

`getRsETHAmountToMint` computes:

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` is a plain storage variable in `LRTOracle` (line 28), written only inside `_updateRsETHPrice()` (line 313), which is only reachable via the public `updateRSETHPrice()` (line 87) or the manager-only `updateRSETHPriceAsManager()` (line 94). Neither `depositETH` nor `depositAsset` calls either function.

As staking rewards accrue between keeper calls, `totalETHInProtocol` grows while `rsETHPrice` stays at its last-written (lower) value. A depositor who deposits during this window receives:

```
rsethAmountToMint = amount × assetPrice / stalePrice   (stalePrice < truePrice)
```

which is strictly greater than the correct `amount × assetPrice / truePrice`. The excess rsETH represents a claim on yield that belongs to existing holders.

The `pricePercentageLimit` guard (lines 260–265 of `LRTOracle.sol`) compounds the issue: when the price increase since `highestRsethPrice` exceeds the configured threshold, non-manager callers of `updateRSETHPrice()` receive a `PriceAboveDailyThreshold` revert, potentially extending the staleness window beyond a normal keeper cycle.

Existing guards are insufficient: `minRSETHAmountExpected` is a depositor-supplied floor (protecting the depositor, not existing holders), and the deposit-limit check (`_checkIfDepositAmountExceedesCurrentLimit`) is unrelated to price freshness.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders' accrued staking rewards are diluted by the excess rsETH minted to the depositor. After `updateRSETHPrice()` is eventually called (reflecting the true higher price), the attacker's excess rsETH redeems for more underlying assets than were deposited, extracting value from existing holders. The magnitude scales linearly with deposit size and the duration of price staleness. This matches the allowed impact: *High. Theft of unclaimed yield.*

## Likelihood Explanation
`updateRSETHPrice()` is a public function with no on-chain enforcement of call frequency. The protocol relies entirely on off-chain keepers. Any gap between keeper calls (routine in practice — daily or longer) creates an exploitable window. An attacker can permissionlessly monitor the chain, compute the true TVL-derived price off-chain, identify when `rsETHPrice` lags, and deposit a large amount immediately before the next keeper update. No privileged access, leaked keys, or victim mistakes are required. The attack is repeatable every keeper cycle.

## Recommendation
Call `_updateRsETHPrice()` (or an equivalent internal function) at the start of `depositETH` and `depositAsset` before computing the mint amount, ensuring `rsETHPrice` reflects the current TVL before any rsETH is issued. Alternatively, compute the mint amount on-the-fly from the live TVL rather than from the cached storage variable.

## Proof of Concept
1. At block T₀, keeper calls `updateRSETHPrice()`; `rsETHPrice` is set to P₀ = 1.01 ETH/rsETH.
2. Staking rewards accrue. At block T₁ (e.g., 24 hours later), the true TVL-derived rate is P₁ = 1.012 ETH/rsETH, but `rsETHPrice` still stores P₀.
3. Attacker calls `depositETH{value: 1000 ETH}(minRSETH, "")`.
4. `getRsETHAmountToMint` returns `1000e18 * 1e18 / P₀ ≈ 990.099 rsETH` instead of the correct `1000e18 * 1e18 / P₁ ≈ 988.142 rsETH`.
5. Attacker receives ≈ 1.957 excess rsETH at no cost.
6. Keeper calls `updateRSETHPrice()`, setting `rsETHPrice = P₁` (now reflecting the larger rsETH supply, so the post-deposit price is slightly below P₁ — diluting existing holders).
7. Attacker initiates withdrawal; their excess rsETH redeems for more ETH than was deposited, extracting value from existing holders.
8. The attack is repeatable every keeper cycle and scales linearly with deposit size.

**Foundry fork test plan:**
- Fork mainnet at a block where `rsETHPrice` is known.
- Warp forward to simulate reward accrual without calling `updateRSETHPrice()`.
- Call `depositETH` with a large value; record `rsethAmountToMint`.
- Call `updateRSETHPrice()`; record new `rsETHPrice`.
- Assert that `rsethAmountToMint * newRsETHPrice > depositAmount * 1e18`, proving the attacker received excess value at the expense of existing holders.
