### Title
NFT Royalty Payment Bypass via Integer Truncation in `compute_royalty_amount` — (File: `chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py` applies two sequential integer floor-division operations when computing the royalty owed on an NFT offer. For small offered amounts (or high `royalty_split` values), the result truncates to zero. Because the wallet uses this computed value to construct the actual on-chain spend bundle — including the `trade_prices_list` passed to the NFT singleton puzzle — an unprivileged offer-maker can craft offers where the royalty payment is silently omitted from the settlement, depriving the NFT creator of their entitled royalty.

### Finding Description

`compute_royalty_amount` computes:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

Two floor-division operations are chained. The first divides by `royalty_split` (the number of NFTs on the same side of the trade), and the second divides by `MAX_ROYALTY_BASIS_POINTS` (10 000). Both truncate toward zero. The test suite explicitly documents and accepts this behavior:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

`compute_royalty_amount` is called inside `make_nft1_offer` to populate `royalty_payments`, which drives two downstream effects:

1. **Royalty coin omission**: if `payment_sum == 0`, the royalty coin spend is skipped entirely. [3](#0-2) 

2. **`trade_prices_list` filter**: the NFT singleton's `trade_prices_list` (which the CLVM puzzle uses to enforce royalties) is filtered to exclude entries where the per-NFT royalty also truncates to zero. [4](#0-3) 

The `royalty_calculation` static method used for display/RPC has the identical truncation:

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
``` [5](#0-4) 

### Impact Explanation

An unprivileged offer-maker can craft an NFT offer where the fungible amount is small enough that `compute_royalty_amount` returns 0. The resulting spend bundle contains no royalty coin and no `trade_prices_list` entry. The NFT singleton puzzle accepts the spend because the wallet-supplied solution carries no royalty obligation. The NFT creator (royalty recipient) receives 0 mojos instead of their entitled royalty. This is a systematic corruption of offer/trade settlement state: the royalty enforcement mechanism is bypassed entirely for any offer below the threshold `10000 // percentage` mojos per NFT per royalty split.

Concrete threshold examples:
- 1% royalty (100 bp): any offer below **100 mojos** pays 0 royalties
- 0.1% royalty (10 bp): any offer below **1 000 mojos** pays 0 royalties
- 0.01% royalty (1 bp): any offer below **10 000 mojos** pays 0 royalties

With `royalty_split > 1` (multiple NFTs on the same side), the first division amplifies the truncation further.

### Likelihood Explanation

Any unprivileged user can create an NFT offer. The offer-maker controls the offered amount. The NFT owner must accept the offer, but the royalty recipient is typically a third party (the original creator) who has no veto. The NFT owner may accept a low-value offer without considering the royalty recipient's interests. The truncation is deterministic and requires no special conditions beyond choosing a small enough offer amount.

### Recommendation

Replace the two-step floor division with a single division that rounds up (ceiling division) to ensure royalties are never under-collected:

```python
# Instead of:
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
# Use ceiling division:
numerator = abs(offered_amount) * percentage
amount = (numerator + royalty_split * MAX_ROYALTY_BASIS_POINTS - 1) // (royalty_split * MAX_ROYALTY_BASIS_POINTS)
```

Apply the same fix to `NFTWallet.royalty_calculation`. Additionally, enforce a minimum royalty of 1 mojo whenever `percentage > 0` and `offered_amount != 0`, so that no non-zero royalty can silently become zero.

### Proof of Concept

```
NFT royalty_percentage = 100 (1%)
Offer: maker offers 99 mojos of XCH for the NFT
royalty_split = 1

compute_royalty_amount(offered_amount=-99, royalty_split=1, percentage=100):
  amount = 99 // 1 * 100 // 10000
         = 99 * 100 // 10000
         = 9900 // 10000
         = 0   ← royalty truncated to zero

Result: royalty_payments is empty, no royalty coin is created,
        trade_prices_list is empty, NFT puzzle enforces no royalty.
        NFT creator receives 0 mojos in royalties.
        Correct royalty would be 0.99 mojos → should round up to 1 mojo.

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
