Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Withdrawals Post-Deployment - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`KernelDepositPool.initialize()` does not set `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal()` reverts when `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and `0 >= 0` is always true, every withdrawal attempt reverts from the moment of deployment until an admin explicitly calls `setMaxNumberOfWithdrawalsPerUser`. Any user who stakes KERNEL tokens before that admin call has their funds temporarily frozen with no self-service remedy.

## Finding Description

`initialize()` sets only `kernelToken`, `rewardsToken`, and the admin role: [1](#0-0) 

`maxNumberOfWithdrawalsPerUser` is never assigned, so it remains `0`. The guard in `initiateWithdrawal()`: [2](#0-1) 

evaluates `0 >= 0 → true` for every caller, unconditionally reverting with `WithdrawalLimitReached`. There is no alternative unstake path. The only remedy is an admin call to `setMaxNumberOfWithdrawalsPerUser`, which itself rejects `0` as invalid: [3](#0-2) 

so the broken state cannot be re-entered once fixed, but it is the guaranteed initial state on every fresh deployment.

## Impact Explanation

Any user who calls `stake()` before the admin sets `maxNumberOfWithdrawalsPerUser` has their KERNEL tokens locked in the contract with no self-service exit. This is a concrete **temporary freezing of funds** (Medium severity per the allowed scope). The freeze window spans from deployment until admin action and is especially harmful during the early high-activity period immediately after launch.

## Likelihood Explanation

This condition is present on every fresh deployment of `KernelDepositPool`. The `initialize()` function is the sole initialization path and structurally omits the parameter. No on-chain guard prevents staking while withdrawals are blocked. Any user who stakes before post-deployment admin configuration is affected, requiring no attacker — the broken state is the default.

## Recommendation

Set a safe non-zero default for `maxNumberOfWithdrawalsPerUser` inside `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = 10; // or any reasonable value <= MAX_WITHDRAWALS_PER_USER
```

Alternatively, accept it as an `initialize()` parameter and validate it is non-zero before storing.

## Proof of Concept

1. Admin deploys and calls `KernelDepositPool.initialize(admin, kernelToken, rewardToken)`.
2. `maxNumberOfWithdrawalsPerUser` is `0` (Solidity default). [1](#0-0) 
3. Alice calls `stake(100e18)` — succeeds; `balanceOf[Alice] = 100e18`. [4](#0-3) 
4. Alice calls `initiateWithdrawal(100e18)`:
   - `userWithdrawalIds[Alice].length` is `0`
   - Check: `0 >= 0` → `true` → reverts with `WithdrawalLimitReached`. [2](#0-1) 
5. Alice's `100e18` KERNEL tokens remain locked; no other withdrawal function exists.
6. Alice must wait for admin to call `setMaxNumberOfWithdrawalsPerUser(N)` with `N > 0`. [5](#0-4) 

**Foundry test sketch:**
```solidity
function test_withdrawalBlockedOnFreshDeploy() public {
    // deploy + initialize (no setMaxNumberOfWithdrawalsPerUser call)
    pool.initialize(admin, address(kernelToken), address(rewardToken));
    kernelToken.mint(alice, 100e18);
    vm.prank(alice);
    kernelToken.approve(address(pool), 100e18);
    vm.prank(alice);
    pool.stake(100e18);
    vm.prank(alice);
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(100e18);
}
```

        external
        onlyRole(DEFAULT_ADMIN_ROLE)

            revert InvalidMaxNumberOfWithdrawalsPerUser();
