### Title
Identity-Point `AppPublicKeyPV` Bypasses CKD Encryption, Leaking Raw BLS Signature On-Chain — (`crates/threshold-signatures/src/confidential_key_derivation/protocol_pv.rs`, `crates/contract/src/primitives/ckd.rs`)

---

### Summary

An unprivileged attacker can submit an `AppPublicKeyPV` request with both `pk1 = G1::identity()` and `pk2 = G2::identity()`. Both the contract-side and node-side `app_public_key_check` functions accept this pair because the identity point is on the curve and torsion-free, satisfying the pairing equation trivially. With `pk1 = identity`, the per-node blinding term `y * pk1` collapses to zero, so the aggregated `big_c` equals the raw BLS signature `msk * H(pk, app_id)` — unencrypted — which is then written on-chain and readable by anyone.

---

### Finding Description

**Node-side `app_public_key_check` does not reject identity points.** [1](#0-0) 

`check_valid_point_g1` and `check_valid_point_g2` only test `is_on_curve() & is_torsion_free()`. The identity (point at infinity) satisfies both conditions in every prime-order subgroup, so both return `true` for identity inputs. [2](#0-1) 

With `pk1 = pk2 = identity`, the pairing check becomes `e(identity, -G2) * e(G1, identity) = 1 * 1 = 1`, which passes.

**Contract-side `app_public_key_check` also accepts identity pairs.** [3](#0-2) 

The contract delegates entirely to `env::bls12381_pairing_check`. The test at line 483 explicitly documents this as "pre-existing behavior": [4](#0-3) 

**Blinding term collapses when `pk1 = identity`.** [5](#0-4) 

Each node computes `big_c = (x_i * H(pk, app_id)) + y * identity = x_i * H(pk, app_id)`. After Lagrange interpolation across threshold participants, the coordinator aggregates this to `big_c = msk * H(pk, app_id)` — the raw BLS signature of `app_id` under the MPC master key, with no blinding.

**`aggregated_output_check` also passes with `pk2 = identity`.** [6](#0-5) 

The check verifies `e(big_c, -G2) * e(big_y, pk2) * e(hash_point, pk) = 1`. With `pk2 = identity`, the middle term is `e(big_y, identity) = 1`, so the equation reduces to `e(msk*H, -G2) * e(H, msk*G2) = 1`, which holds. The check passes.

**Contract `ckd_output_check` passes for the same reason.** [7](#0-6) 

The contract resolves the yield and writes `(big_y, big_c)` on-chain, where `big_c = msk * H(pk, app_id)`.

---

### Impact Explanation

The CKD protocol's confidentiality guarantee is that `big_c = msk * H(pk, app_id) + a * big_y` is an ElGamal-style encryption: only the holder of the app secret key `a` can subtract `a * big_y` to recover `msk * H(pk, app_id)`. By supplying identity points, the attacker eliminates the blinding entirely. The on-chain response directly contains `msk * H(pk, app_id)` for any `app_id` the attacker chooses. This is unauthorized access to MPC secret material (the BLS signature under the master key), enabling forgery of CKD outputs for arbitrary `app_id`s without holding the corresponding app private key.

This matches: **Critical — Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery.**

---

### Likelihood Explanation

The attack requires only a single unprivileged call to `request_app_private_key` with crafted identity-point inputs. No collusion, no key leakage, no TEE compromise. The contract test at line 483 explicitly confirms the identity pair is accepted today. The exploit is fully local-testable on a devnet with a running CKD domain.

---

### Recommendation

Add an explicit identity-point rejection in both validation paths:

1. **Node-side** (`protocol_pv.rs`, `app_public_key_check`): after the `check_valid_point_g1`/`check_valid_point_g2` guards, add `if app_pk.pk1.is_identity().into() || app_pk.pk2.is_identity().into() { return false; }`.

2. **Contract-side** (`ckd.rs`, `app_public_key_check`): before calling `bls12381_pairing_check`, decompress and explicitly reject the identity encoding (all-zero compressed point with the infinity flag set).

3. Update the test `app_public_key_check__should_accept_identity_key_pair` to assert `!accepted` after the fix.

---

### Proof of Concept

```
1. Deploy a local MPC network with a CKD domain (msk known from test setup).
2. Call request_app_private_key with:
     AppPublicKeyPV {
       pk1: G1Projective::identity().to_compressed(),   // 0x40 00...00 (48 bytes)
       pk2: G2Projective::identity().to_compressed(),   // 0xc0 00...00 (96 bytes)
     }
3. Wait for the MPC nodes to complete the protocol and the contract to resolve the yield.
4. Read big_c from the on-chain CKDResponse.
5. Verify: e(big_c, G2) == e(H(pk, app_id), msk*G2)
   i.e., big_c is exactly msk * H(pk, app_id).
```

The test `app_public_key_check__should_accept_identity_key_pair` already confirms step 2 passes. The math in `compute_signature_share` confirms steps 3–5. [8](#0-7)

fn aggregated_output_check(
    output: &CKDOutput,
    app_pk: &PublicVerificationKey,
    public_key: &VerifyingKey,
    hash_point: &ElementG1,
) -> bool {
    if !check_valid_point_g1(output.big_c.into()) || !check_valid_point_g1(output.big_y.into()) {
        return false;
    }
    multi_miller_loop(&[
        (output.big_c, -ElementG2::generator()),
        (output.big_y, app_pk.pk2),
        (*hash_point, public_key.to_element()),
    ])
}
