### Title
Cross-Shard Hybrid `ChunkEndorsement` Bypasses Semantic Validation, Suppressing Legitimate Validator Endorsements — (`chain/client/src/stateless_validation/chunk_endorsement.rs`, `chain/client/src/stateless_validation/validate.rs`)

---

### Summary

`ChunkEndorsementV2` has two independently-signed fields: `inner` (covering only `chunk_hash`) and `metadata` (covering `account_id`, `shard_id`, `epoch_id`, `height_created`). Because `validate_chunk_endorsement` never checks that `inner.chunk_hash` belongs to the shard named in `metadata.shard_id`, an unprivileged peer can splice the `inner` from a validator's shard-A endorsement with the `metadata` from the same validator's shard-B endorsement. The resulting hybrid passes every validation gate, is stored in the endorsement cache under shard B's `ChunkProductionKey`, and permanently blocks the legitimate shard-B endorsement from that validator via the "already received" early-exit guard.

---

### Finding Description

**Struct layout** — `ChunkEndorsementV2` carries two independent signatures: [1](#0-0) 

- `signature` covers only `ChunkEndorsementInnerV1 { chunk_hash, signature_differentiator }`.
- `metadata_signature` covers only `ChunkEndorsementMetadata { account_id, shard_id, epoch_id, height_created }`.

Neither signature commits to the other field. There is no binding between `chunk_hash` and `shard_id`.

**Validation path** — `validate_chunk_endorsement` does two things: [2](#0-1) 

1. `validate_chunk_relevant_as_validator` → `ensure_chunk_validator`: checks that `account_id` is in the validator assignments for `metadata.shard_id`. It uses only the `ChunkProductionKey` derived from `metadata`. [3](#0-2) 

2. `validate_chunk_endorsement_signature`: verifies both signatures against the validator's public key. [4](#0-3) 

Neither step checks that `inner.chunk_hash` is a hash of a chunk on `metadata.shard_id`.

**Attack construction** — Suppose validator V is assigned to both shard A and shard B for the same `(epoch_id, height_created)`. V legitimately broadcasts:

- Endorsement-A: `{inner_A={chunk_hash_A}, sig_A, metadata_A={shard_id_A,...}, meta_sig_A}`
- Endorsement-B: `{inner_B={chunk_hash_B}, sig_B, metadata_B={shard_id_B,...}, meta_sig_B}`

A peer attacker splices them into a hybrid:

```
{inner_A={chunk_hash_A}, sig_A,   ← from shard A's endorsement
 metadata_B={shard_id_B,...}, meta_sig_B}  ← from shard B's endorsement
```

Validation of the hybrid:
- `ensure_chunk_validator(shard_id_B, V)` → V is in shard B's assignments → **PASS**
- `sig_A.verify(inner_A, V.pubkey)` → valid signature → **PASS**
- `meta_sig_B.verify(metadata_B, V.pubkey)` → valid signature → **PASS**
- Result: `ChunkRelevance::Relevant`

**Cache poisoning** — The hybrid is stored: [5](#0-4) 

Cache entry for key `{shard_id_B, epoch_id, height}` now maps `V.account_id → (chunk_hash_A, sig_A)`.

**Legitimate endorsement blocked** — When the real shard-B endorsement from V arrives: [6](#0-5) 

`entry.contains_key(account_id)` is true → the legitimate endorsement is silently dropped.

**Endorsement not counted** — During `collect_chunk_endorsements`, the cached entry is filtered out because `chunk_hash_A ≠ chunk_hash_B`: [7](#0-6) 

V's stake contributes zero to shard B's endorsement threshold.

---

### Impact Explanation

An attacker who is a peer of the target node and has observed V's endorsements for two shards can suppress V's endorsement for shard B entirely. If the attacker targets enough validators assigned to shard B, the chunk for shard B may fail to reach the endorsement threshold, preventing it from being included as a new chunk in the block. This is a targeted liveness attack on specific shards: chunk production stalls or degrades for the targeted shard without any validator key compromise.

---

### Likelihood Explanation

- Validators assigned to multiple shards is a normal operating condition in NEAR's sharded design.
- Chunk endorsements are broadcast to peers; an attacker peer receives them in the clear.
- The hybrid can be constructed entirely from observed, legitimately-signed network messages — no key material is needed.
- The attack is per-validator, per-shard, and can be repeated across many validators simultaneously.
- The only timing constraint is that the hybrid must arrive at the target node before the legitimate shard-B endorsement.

---

### Recommendation

Bind `inner.chunk_hash` to `metadata.shard_id` cryptographically. The simplest fix is to include `shard_id` (and ideally `epoch_id` and `height_created`) inside `ChunkEndorsementInnerV1` so that the `inner` signature commits to the shard. Alternatively, add an explicit check in `validate_chunk_endorsement` that the `chunk_hash` stored in `inner` is consistent with the shard identified in `metadata` (e.g., by looking up the chunk header and verifying its `shard_id` matches `metadata.shard_id`). The "already received" deduplication guard should also be moved to after validation, or the cache should store the full endorsement and re-validate on update.

---

### Proof of Concept

```rust
// Pseudocode unit test
let validator = create_validator_assigned_to_shards([shard_a, shard_b]);

// Legitimate endorsements
let endorsement_a = ChunkEndorsement::new(epoch_id, &chunk_header_a, &validator.signer);
let endorsement_b = ChunkEndorsement::new(epoch_id, &chunk_header_b, &validator.signer);

// Splice: inner from shard A, metadata from shard B
let hybrid = ChunkEndorsementV2 {
    inner: endorsement_a.inner.clone(),       // chunk_hash_A
    signature: endorsement_a.signature.clone(), // sig over chunk_hash_A
    metadata: endorsement_b.metadata.clone(), // shard_id_B
    metadata_signature: endorsement_b.metadata_signature.clone(),
};

// Assert hybrid passes validate_chunk_endorsement (it should NOT, but currently does)
assert!(validate_chunk_endorsement(&epoch_manager, &hybrid, &store).is_ok());

// Assert legitimate shard-B endorsement is now blocked
tracker.process_chunk_endorsement(&ChunkEndorsement::V2(hybrid)).unwrap();
tracker.process_chunk_endorsement(&endorsement_b).unwrap();

// Assert shard B endorsement count for validator is zero
let state = tracker.collect_chunk_endorsements(&chunk_header_b).unwrap();
assert!(!state.is_endorsed); // validator's stake not counted
```

pub struct ChunkEndorsementV2 {
    // This is the part of the chunk endorsement that signed and included in the block header
    inner: ChunkEndorsementInnerV1,
    // This is the signature of the inner field, to be included in the block header
    signature: Signature,
    // This consists of the metadata for chunk endorsement used in validation
    metadata: ChunkEndorsementMetadata,
    // Metadata signature is used to validate that the metadata is produced by the expected validator
    metadata_signature: Signature,
}

impl ChunkEndorsementV2 {
    fn verify(&self, public_key: &PublicKey) -> bool {
        let inner = borsh::to_vec(&self.inner).unwrap();
        let metadata = borsh::to_vec(&self.metadata).unwrap();
        self.signature.verify(&inner, public_key)
            && self.metadata_signature.verify(&metadata, public_key)
    }
}

        endorsement.account_id(),

    Ok(ChunkRelevance::Relevant)
}

fn ensure_chunk_validator(
    epoch_manager: &dyn EpochManagerAdapter,
    chunk: &ChunkProductionKey,
    account_id: &AccountId,
) -> Result<(), Error> {
    let chunk_validator_assignments = epoch_manager.get_chunk_validator_assignments(
        &chunk.epoch_id,
        chunk.shard_id,
        chunk.height_created,
    )?;
    if !chunk_validator_assignments.contains(account_id) {
        return Err(Error::NotAChunkValidator);
    }
    Ok(())
}

fn validate_chunk_endorsement_signature(
    epoch_manager: &dyn EpochManagerAdapter,
    endorsement: &ChunkEndorsement,
) -> Result<(), Error> {
    let validator = epoch_manager.get_validator_by_account_id(
        &endorsement.chunk_production_key().epoch_id,
        &endorsement.account_id(),
    )?;
    if !endorsement.verify(validator.public_key()) {
        return Err(Error::InvalidChunkEndorsement);
    }
    Ok(())
