Audit Report

## Title
Unguarded Oracle Call in Loop Permanently Blocks `updateRSETHPrice()` and Bypasses Downside-Protection Auto-Pause — (File: `contracts/LRTOracle.sol`)

## Summary

`LRTOracle._getTotalEthInProtocol()` iterates over every supported asset and calls an external price oracle for each one with no `try/catch`. If any single oracle reverts — a realistic condition for Chainlink aggregators during sequencer outages, circuit-breaker events, or rapid price moves — the entire `updateRSETHPrice()` call reverts. Because the downside-protection auto-pause is computed inside `_updateRsETHPrice()` *after* `_getTotalEthInProtocol()` returns, a stuck oracle also permanently disables the protocol's only automated safety mechanism against slashing-induced price drops, leaving `rsETHPrice` frozen at a stale (inflated) value while deposits and withdrawals continue at that incorrect rate.

## Finding Description

**Root cause:** `_getTotalEthInProtocol()` (lines 336–348) calls `getAssetPrice(asset)` for every supported asset with no error handling:

```solidity
uint256 assetER = getAssetPrice(asset);   // line 339 — no try/catch
```

`getAssetPrice` (line 157) delegates unconditionally to the registered `IPriceFetcher`:

```solidity
return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
```

`ChainlinkPriceOracle.getAssetPrice()` (lines 49–54) calls `latestRoundData()` with no staleness check and no try/catch:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Chainlink aggregators revert (not just return stale data) during L2 sequencer outages, aggregator pauses, and certain circuit-breaker conditions. Any such revert propagates up the call stack: `getAssetPrice` → `_getTotalEthInProtocol` → `_updateRsETHPrice` → `updateRSETHPrice` / `updateRSETHPriceAsManager`.

**Why existing checks fail:** The downside-protection block (lines 270–281) sits *after* the call to `_getTotalEthInProtocol()` at line 231. If `_getTotalEthInProtocol()` reverts, execution never reaches the pause logic:

```solidity
uint256 totalETHInProtocol = _getTotalEthInProtocol();  // line 231 — reverts here
// ... lines 270-281 never reached
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

**Exploit path:**
1. Protocol supports N assets, each with a Chainlink oracle.
2. Asset X undergoes a rapid price drop; its Chainlink aggregator triggers a circuit-breaker and reverts on `latestRoundData()`.
3. Any caller invokes `updateRSETHPrice()` (public, no access control).
4. `_getTotalEthInProtocol()` reaches asset X's oracle call and reverts.
5. `rsETHPrice` remains at its last (pre-crash, inflated) value.
6. The downside-protection pause never fires; `LRTDepositPool` and `LRTWithdrawalManager` remain unpaused.
7. Withdrawers redeem rsETH at the stale inflated price, receiving more underlying ETH than they are entitled to, extracting value from existing rsETH holders.
8. The oracle recovers hours later; by then the damage is done.

## Impact Explanation

**Severity: High**

Two concrete harms:

1. **Direct theft of existing rsETH holders' funds** — `rsETHPrice` is frozen at a stale (inflated) value. Withdrawers redeem rsETH at this inflated rate and receive more underlying ETH than the protocol's actual NAV supports. This is direct, at-rest fund extraction from existing holders, matching the Critical allowed impact "Direct theft of any user funds." The likelihood of the specific correlated scenario (oracle reverts *because* the asset is crashing) moderates the overall severity to High.

2. **Bypass of the only automated safety mechanism** — The auto-pause that should halt the protocol during a slashing event or rapid NAV decline cannot execute. The protocol continues accepting deposits and processing withdrawals at an incorrect price, compounding losses for the duration of the oracle outage.

## Likelihood Explanation

**Likelihood: Medium**

- `updateRSETHPrice()` is public and callable by any unprivileged address.
- Chainlink aggregators on L2s (Arbitrum, Optimism) revert during sequencer outages — a documented, recurring operational condition.
- The protocol supports multiple LST assets (stETH, ETHx, sfrxETH, etc.), each with its own oracle. The probability that *at least one* oracle reverts during a market-stress event is non-trivial and positively correlated with the exact conditions (rapid price drops) that make the downside-protection pause most critical.
- No attacker capability is required; the failure is triggered by normal market conditions.

## Recommendation

Wrap each oracle call in a `try/catch` inside `_getTotalEthInProtocol()`. On revert, either skip the asset (treating its contribution as zero, which is conservative and safe) or propagate a dedicated error that still allows the downside-protection path to execute separately:

```solidity
try IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset) returns (uint256 price) {
    totalETHInProtocol += totalAssetAmt.mulWad(price);
} catch {
    emit OracleFetchFailed(asset);
    // skip asset or revert with a flag
}
```

Alternatively, decouple the auto-pause logic from the price-aggregation loop so that a single oracle failure cannot suppress the safety mechanism.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// Fork mainnet/Arbitrum at a block where ETHx Chainlink feed is live
// Mock the ETHx Chainlink aggregator to revert on latestRoundData()
vm.mockCallRevert(ethxPriceFeed, abi.encodeWithSelector(AggregatorV3Interface.latestRoundData.selector), "circuit breaker");

// Record rsETHPrice before
uint256 priceBefore = lrtOracle.rsETHPrice();

// Attempt price update — should revert

lrtOracle.updateRSETHPrice();

// rsETHPrice unchanged (stale)
assertEq(lrtOracle.rsETHPrice(), priceBefore);

// Downside-protection pause never fired
assertFalse(lrtDepositPool.paused());
assertFalse(withdrawalManager.paused());

// Withdrawer redeems at stale inflated price — receives excess ETH
// (demonstrate by computing expected vs actual withdrawal amount)
```
