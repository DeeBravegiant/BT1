Audit Report

## Title
Stale Pre-Pause `rsETHPrice` Propagated to L2 Receiver After L1 Oracle Auto-Pause Causes Under-Minting of rsETH - (File: contracts/cross-chain/RSETHRateReceiver.sol)

## Summary

When `LRTOracle._updateRsETHPrice()` detects a price drop beyond `pricePercentageLimit`, it pauses the oracle and returns early without updating `rsETHPrice`, leaving it frozen at the pre-drop (inflated) value. Because `CrossChainRateProvider.updateRate()` is permissionless and reads `rsETHPrice` directly with no oracle-pause guard, any caller can immediately push this stale inflated rate to L2. `RSETHRateReceiver.getRate()` returns the cached rate unconditionally, so L2 pool depositors receive fewer rsETH tokens than the true current rate entitles them to.

## Finding Description

**L1 — Auto-pause freezes `rsETHPrice` at pre-drop value**

In `LRTOracle._updateRsETHPrice()`, when the computed `newRsETHPrice` drops more than `pricePercentageLimit` below `highestRsethPrice`, the function calls `_pause()` and `return`s before the `rsETHPrice = newRsETHPrice` assignment at line 313: [1](#0-0) [2](#0-1) 

`rsETHPrice` (line 28) remains at its last written value — higher than the true current value — and is always publicly readable regardless of pause state.

**L1 — Rate provider reads `rsETHPrice` with no oracle-pause guard**

`RSETHRateProvider.getLatestRate()` reads `rsETHPrice` directly: [3](#0-2) 

`RSETHMultiChainRateProvider.getLatestRate()` does the same: [4](#0-3) 

Neither checks `ILRTOracle(rsETHPriceOracle).paused()` before returning the price.

**L1 — `updateRate()` is permissionless**

`CrossChainRateProvider.updateRate()` has no access control and no pause guard: [5](#0-4) 

Any external caller can invoke it after the oracle is paused, pushing the stale inflated `rsETHPrice` to L2 via LayerZero.

**L2 — `RSETHRateReceiver` / `CrossChainRateReceiver` has no pause mechanism and no staleness check**

`CrossChainRateReceiver.getRate()` returns the cached `rate` unconditionally: [6](#0-5) 

`lastUpdated` is stored but never used in any guard. `RSETHRateReceiver` adds nothing on top: [7](#0-6) 

**L2 — Pool mints rsETH using the stale rate**

`RSETHPoolV2.deposit()` and `RSETHPoolV3.deposit()` call `viewSwapRsETHAmountAndFee()`, which divides by the stale inflated rate: [8](#0-7) [9](#0-8) 

With an inflated `rsETHToETHrate`, `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` yields fewer rsETH tokens than the true rate would produce. The L2 pool's own `paused` flag is independent of the L1 oracle state and is not automatically set.

## Impact Explanation

Users depositing ETH on L2 during the window between L1 oracle auto-pause and a manual L2 pool pause receive fewer rsETH tokens than the true current rate entitles them to. The protocol retains the full ETH deposit, but users are under-compensated in rsETH. This matches the allowed Low impact: **Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation

The auto-pause trigger in `_updateRsETHPrice()` is a normal operational event activated whenever the price moves beyond `pricePercentageLimit`. The window between L1 oracle pause and a manual L2 pool pause is non-zero and can span multiple blocks. Because `updateRate()` is permissionless, any external actor can actively re-push the stale rate to L2 at any time after the oracle is paused, extending or reopening the vulnerable window. No special privileges are required.

## Recommendation

1. In `RSETHRateProvider.getLatestRate()` and `RSETHMultiChainRateProvider.getLatestRate()`, check `ILRTOracle(rsETHPriceOracle).paused()` and revert if the oracle is paused, blocking stale rate propagation at the source.
2. Add a `whenNotPaused` guard to `CrossChainRateProvider.updateRate()` (or its overriding subclasses) so the rate cannot be pushed to L2 when the L1 oracle is paused.
3. Add a staleness check in `CrossChainRateReceiver.getRate()` using the stored `lastUpdated` timestamp, reverting if the rate is older than a configurable threshold.
4. Add a `pause()`/`unpause()` mechanism to `CrossChainRateReceiver` (and therefore `RSETHRateReceiver`) with a `whenNotPaused` guard on `lzReceive()` and `getRate()`, enabling operators to halt L2 rate serving independently.

## Proof of Concept

```solidity
// Foundry fork test outline
function test_stalePriceAfterAutoPause() public {
    // 1. Trigger auto-pause on L1 by simulating a price drop beyond pricePercentageLimit
    //    (e.g., manipulate underlying asset prices so newRsETHPrice < highestRsethPrice * (1 - limit))
    vm.prank(address(0)); // public caller
    lrtOracle.updateRSETHPrice(); // triggers _pause() + early return, rsETHPrice NOT updated

    assertTrue(lrtOracle.paused());
    uint256 stalePriceOnL1 = lrtOracle.rsETHPrice(); // still returns pre-pause inflated value

    // 2. Anyone pushes the stale rate to L2 — no access control on updateRate()
    uint256 lzFee = rsETHRateProvider.estimateFees();
    vm.deal(attacker, lzFee);
    vm.prank(attacker);
    rsETHRateProvider.updateRate{value: lzFee}(); // reads rsETHPrice directly, no pause check

    // 3. Simulate LayerZero delivery to L2 receiver
    // lzReceive() stores the stale rate
    assertEq(rsETHRateReceiver.getRate(), stalePriceOnL1);

    // 4. L2 pool deposit uses the stale inflated rate
    uint256 trueRate = /* computed off-chain or via separate oracle */;
    uint256 expectedAtTrueRate = 1 ether * 1e18 / trueRate;

    vm.deal(user, 1 ether);
    vm.prank(user);
    rsETHPoolV2.deposit{value: 1 ether}("ref");

    uint256 received = wrsETH.balanceOf(user);
    // User receives fewer rsETH than the true rate would give
    assertLt(received, expectedAtTrueRate);
}
```
