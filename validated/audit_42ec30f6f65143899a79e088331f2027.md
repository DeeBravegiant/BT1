Audit Report

## Title
`updateRate()` Dispatches LayerZero Message to `address(0)` When `rateReceiver` Is Uninitialized — (`contracts/cross-chain/CrossChainRateProvider.sol`)

## Summary
`CrossChainRateProvider.updateRate()` has no access control and no zero-address guard on `rateReceiver`. Because `RSETHRateProvider`'s constructor never initialises `rateReceiver`, it defaults to `address(0)`. Any caller can invoke `updateRate()` before the owner calls `updateRateReceiver()`, causing a LayerZero message to be permanently dispatched to `address(0)` on the destination chain while the provider's local `rate` and `lastUpdated` are written as if the update succeeded.

## Finding Description
`rateReceiver` is declared as a plain storage variable in `CrossChainRateProvider` and is never assigned in `RSETHRateProvider`'s constructor, leaving it at `address(0)` until the owner explicitly calls `updateRateReceiver()`. [1](#0-0) [2](#0-1) 

`updateRate()` carries only a `nonReentrant` modifier — no `onlyOwner` or similar guard — so any EOA or contract can call it at any time: [3](#0-2) 

Inside `updateRate()`, `rateReceiver` is used directly with no zero-address check to build the remote destination blob: [4](#0-3) 

That blob is passed straight to `ILayerZeroEndpoint.send()`: [5](#0-4) 

LayerZero accepts the send on the source chain; the relayer attempts delivery to `address(0)` on the destination chain, which has no `lzReceive` implementation, so the message is permanently undeliverable. Meanwhile, `rate` and `lastUpdated` are written locally, creating a false impression of a successful update: [6](#0-5) 

## Impact Explanation
The `RSETHRateReceiver` on the destination chain never receives the rate update; its stored rate remains at zero (or stale). Any protocol or AMM that reads the receiver's rate will misprice rsETH for cross-chain users. No funds are directly drained from the contracts, placing this squarely in the **Low — Contract fails to deliver promised returns, but doesn't lose value** scope.

## Likelihood Explanation
The vulnerable window opens at deployment and closes only when the owner calls `updateRateReceiver()`. During this window, any unprivileged EOA or bot can call `updateRate()` with a small ETH value to cover LZ fees. No special privileges, flash loans, or victim interaction are required. Front-running the owner's setup transaction is one path, but even an accidental or automated call during the window produces the same outcome. The call is repeatable for as long as `rateReceiver` remains `address(0)`.

## Recommendation
Add a zero-address guard at the top of `updateRate()`:

```solidity
function updateRate() external payable nonReentrant {
    require(rateReceiver != address(0), "CrossChainRateProvider: receiver not set");
    // ...
}
```

Alternatively, require `rateReceiver` to be supplied and validated (non-zero) in `RSETHRateProvider`'s constructor, eliminating the uninitialised window entirely.

## Proof of Concept

```solidity
// 1. Deploy RSETHRateProvider — rateReceiver == address(0) by default
RSETHRateProvider provider = new RSETHRateProvider(
    oracle, dstChainId, lzEndpoint
);
// Owner has NOT yet called provider.updateRateReceiver()

// 2. Any unprivileged caller invokes updateRate with enough ETH for LZ fees
provider.updateRate{value: 0.01 ether}();
// LZ send() is called with remoteAndLocalAddresses = abi.encodePacked(address(0), address(provider))
// Message is dispatched to address(0) on destination chain — permanently undeliverable
// provider.rate() and provider.lastUpdated() are written, falsely indicating success

// 3. Verify receiver state is unchanged
assertEq(RSETHRateReceiver(receiver).rate(), 0);
assertEq(RSETHRateReceiver(receiver).lastUpdated(), 0);
```

A Foundry fork test against mainnet (or a mock LZ endpoint that records the destination address) can confirm that `remoteAndLocalAddresses[:20]` equals `address(0)` when `rateReceiver` has not been set.
