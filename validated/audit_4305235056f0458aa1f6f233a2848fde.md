### Title
Expired DRep Credential Passes Voter Validation, Enabling Expiry Self-Extension and Unauthorized Governance Participation — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

---

### Summary

The `conwayGovTransition` function in the Conway `GOV` rule validates DRep voters by checking only whether their credential exists in the registered-DRep map (`knownDReps = vsDReps certVState`). It does **not** check whether the DRep is expired. This is the direct analog of the reported NFT vulnerability: the code checks the credential (the "key") but omits the active/non-expired status check (the "verified" flag). An expired DRep can submit a vote that passes all GOV-rule predicates, which then causes `updateVotingDRepExpiries` to reset the DRep's expiry epoch, effectively resurrecting the expired DRep and allowing its votes to count in subsequent ratification.

---

### Finding Description

**Root cause — incomplete voter validation in `conwayGovTransition`:**

`knownDReps` is built from the full `vsDReps` map, which contains every registered DRep regardless of expiry:

```haskell
knownDReps = vsDReps certVState   -- includes expired DReps
```

The `internVoter` helper classifies a `DRepVoter` as "known" if and only if its credential appears in `knownDReps`:

```haskell
internVoter = \case
  ...
  DRepVoter cred -> DRepVoter <$> internMap cred knownDReps
  ...
```

Voters not found here are collected into `unknownVoters` and rejected with `VotersDoNotExist`. Because expired DReps remain in `vsDReps`, they pass this gate.

The second predicate, `checkVotersAreValid`, dispatches on voter type:

```haskell
checkVotersAreValid currentEpoch committeeState votes =
  checkDisallowedVotes votes DisallowedVoters $ \gas ->
    \case
      CommitteeVoter {} -> isCommitteeVotingAllowed currentEpoch committeeState (gasAction gas)
      DRepVoter {}      -> isDRepVotingAllowed (gasAction gas)   -- ← no expiry check
      StakePoolVoter {} -> isStakePoolVotingAllowed (gasAction gas)
```

`isDRepVotingAllowed` receives only the `GovAction` type; it has no access to the DRep's `drepExpiry` field or the current epoch. By contrast, `isCommitteeVotingAllowed` receives `currentEpoch` and `committeeState`, enabling it to enforce term expiry for committee members. No equivalent expiry guard exists for DReps.

**Consequence — expiry self-extension via `updateVotingDRepExpiries`:**

After the GOV rule accepts the vote and stores it in `gasDRepVotes`, the LEDGER rule calls `updateVotingDRepExpiries`, which iterates over every `DRepVoter` in the transaction's voting procedures and unconditionally resets the expiry:

```haskell
updateVotingDRepExpiries tx currentEpoch drepActivity certState =
  let numDormantEpochs = certState ^. certVStateL . vsNumDormantEpochsL
      updateVSDReps vsDReps =
        Map.foldlWithKey'
          ( \dreps voter _ -> case voter of
              DRepVoter cred ->
                Map.adjust
                  (drepExpiryL .~ computeDRepExpiry drepActivity currentEpoch numDormantEpochs)
                  cred
                  dreps
              _ -> dreps
          )
          vsDReps
          (unVotingProcedures $ tx ^. bodyTxL . votingProceduresTxBodyL)
   in certState & certVStateL . vsDRepsL %~ updateVSDReps
```

There is no guard of the form `if currentEpoch > drepExpiry drepState then skip`. An expired DRep that votes has its `drepExpiry` overwritten with `computeDRepExpiry drepActivity currentEpoch numDormantEpochs`, a future epoch, making it active again.

**Consequence — vote counted in ratification:**

`dRepAcceptedRatio` excludes DReps whose stored `drepExpiry` is in the past at ratification time:

```haskell
Just drepState
  | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- expired, skip
  | otherwise -> ...count vote...
```

Because the expiry was already reset by `updateVotingDRepExpiries` within the same epoch, the DRep is no longer expired when the epoch-boundary ratification pulse runs. Its `VoteYes` (or `VoteNo`) is therefore counted in the DRep acceptance ratio for every governance action it voted on.

---

### Impact Explanation

An attacker who controls a DRep credential whose `drepExpiry` has passed can:

1. Construct a transaction containing a `DRepVoter` vote on any live governance action.
2. The GOV rule accepts the vote (credential is in `vsDReps`; `isDRepVotingAllowed` passes).
3. `updateVotingDRepExpiries` resets the DRep's expiry to a future epoch.
4. At epoch-boundary ratification, the DRep is active; its vote is included in `dRepAcceptedRatio`.

If the expired DRep holds significant delegated stake, this can tip the DRep acceptance ratio for `ParameterChange`, `TreasuryWithdrawals`, `HardForkInitiation`, `NewConstitution`, or `UpdateCommittee` proposals, causing an unauthorized governance action to be enacted. This maps to the **Critical** impact class: *unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted*.

Even without tipping a vote, the expiry self-extension bypasses the DRep activity requirement, allowing a DRep that should have been pruned from the active set to persist indefinitely by voting once per activity window — matching the **Medium** impact class: *attacker-controlled votes exceed intended validation limits*.

---

### Likelihood Explanation

- Any DRep whose `drepExpiry` epoch has been reached is eligible to exploit this.
- DReps expire naturally after `ppDRepActivity` epochs of inactivity; this is a normal operational state.
- No privileged access, key compromise, or majority collusion is required — only possession of the DRep's signing key.
- The attack is a single ordinary transaction (a voting procedure); it is indistinguishable from a legitimate vote at the transaction level.
- Likelihood is **Medium-High**: expired DReps are a routine ledger state, and the exploit requires only a standard signed transaction.

---

### Recommendation

In `conwayGovTransition`, filter `knownDReps` to exclude expired credentials before building the voter intern set, mirroring how committee member expiry is enforced:

```haskell
-- Only non-expired DReps are "known" voters
knownDReps =
  Map.filter (\ds -> currentEpoch <= drepExpiry ds) (vsDReps certVState)
```

Alternatively, add an explicit predicate in `checkVotersAreValid` for `DRepVoter`:

```haskell
DRepVoter cred ->
  isDRepVotingAllowed (gasAction gas)
    && maybe False (\ds -> currentEpoch <= drepExpiry ds)
                   (Map.lookup cred (vsDReps certVState))
```

Additionally, `updateVotingDRepExpiries` should guard against resetting the expiry of a DRep that was already expired at the time of the vote, to prevent the self-resurrection side-effect.

### Proof of Concept

1. Register a DRep with credential `cred` and let `ppDRepActivity` epochs pass without voting → `drepExpiry cred < currentEpoch` (DRep is expired).
2. Submit a governance proposal (e.g., `InfoAction`).
3. Construct a transaction with `VotingProcedures { DRepVoter cred → VoteYes }` for the proposal.
4. Submit the transaction. The GOV rule accepts it because `cred ∈ vsDReps certVState` and `isDRepVotingAllowed InfoAction = True`.
5. Observe that `updateVotingDRepExpiries` resets `drepExpiry cred` to `currentEpoch + drepActivity - numDormantEpochs`.
6. At epoch boundary, `dRepAcceptedRatio` includes the DRep's stake in the yes-vote tally, despite the DRep having been expired when the vote was cast. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```haskell
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
```
