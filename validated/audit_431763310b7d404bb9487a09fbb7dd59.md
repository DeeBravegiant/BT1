Audit Report

## Title
Missing Zero-Amount Guard on `rsETHAmount` Allows ETH Deposits to Succeed While Minting 0 wrsETH — (`contracts/pools/RSETHPoolV2NBA.sol`)

## Summary
`RSETHPoolV2NBA.deposit()` accepts ETH and calls `wrsETH.mint(msg.sender, rsETHAmount)` without verifying `rsETHAmount > 0`. Because `viewSwapRsETHAmountAndFee` uses integer division, any deposit small enough that `amountAfterFee * 1e18 < rsETHToETHrate` silently rounds `rsETHAmount` to zero. The ETH is accepted, no wrsETH is minted, and the depositor has no on-chain path to recover their funds.

## Finding Description
In `deposit()` (L106–118), the only guard is `if (amount == 0) revert InvalidAmount()` on the raw ETH value. After that, `viewSwapRsETHAmountAndFee` computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // L132
```

With `rsETHToETHrate > 1e18` (always true for yield-bearing rsETH), a deposit of 1 wei with `feeBps = 0` yields `1 * 1e18 / 1.05e18 = 0` due to integer floor division. `wrsETH.mint(msg.sender, 0)` at L115 calls `_mint(_to, 0)` in `RsETHTokenWrapper.sol` (L190–192), which is a no-op in OpenZeppelin and does not revert. The deposit completes: the pool's ETH balance increases by 1 wei, `feeEarnedInETH` is unchanged, and the user's wrsETH balance is unchanged. The only ETH-exit path for non-admin users is none — `moveAssetsForBridging()` (L151–159) is gated to `BRIDGER_ROLE` and sends funds to `msg.sender` (the bridger), not back to the depositor.

## Impact Explanation
Any user who deposits a dust amount (e.g., 1 wei) permanently loses those funds from their perspective: the contract accepts the ETH, emits a `SwapOccurred` event with `rsETHAmount = 0`, and the user receives nothing. The ETH is later swept by the bridger via `moveAssetsForBridging()`. This constitutes **Medium: Temporary freezing of funds** — the ETH is not destroyed but is inaccessible to the depositor with no on-chain recovery mechanism.

## Likelihood Explanation
No special role or oracle manipulation is required. Any unprivileged user can call `deposit{value: 1}()`. The condition triggers under a normal, unmanipulated oracle rate (any rate `> 1e18`, which is always true for rsETH). Accidental dust deposits from UI rounding, scripted bots, or fuzz testing are realistic. The `feeBps > 0` case raises the threshold slightly but does not eliminate the issue.

## Recommendation
Add a post-computation zero-amount guard in `deposit()`:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This ensures every accepted ETH deposit results in a non-zero wrsETH mint.

## Proof of Concept
```solidity
// Preconditions:
//   rsETHOracle.getRate() returns 1.05e18 (normal post-yield rate)
//   feeBps = 0
//   Pool is unpaused, wrsETH MINTER_ROLE granted to pool

// Step 1: user calls deposit with 1 wei
pool.deposit{value: 1}("ref");

// Step 2: viewSwapRsETHAmountAndFee(1):
//   fee = 1 * 0 / 10_000 = 0
//   amountAfterFee = 1
//   rsETHAmount = 1 * 1e18 / 1.05e18 = 0  (integer division)

// Step 3: feeEarnedInETH += 0  (unchanged)
// Step 4: wrsETH.mint(user, 0)  → _mint(user, 0) → no-op, no revert

// Assertions:
assert(wrsETH.balanceOf(user) == 0);
assert(address(pool).balance == 1);
// user has no function to call to recover the 1 wei
// bridger later sweeps it via moveAssetsForBridging()
```

A Foundry fuzz test parameterizing `amount` over `[1, rsETHToETHrate/1e18]` and asserting `wrsETH.balanceOf(depositor) > 0` after each deposit would reproduce this across the full vulnerable range.
