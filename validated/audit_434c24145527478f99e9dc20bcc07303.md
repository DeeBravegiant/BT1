Audit Report

## Title
Uninitialized `tokenFeeBps` Mapping Allows Fee-Free Token Deposits ‚Äî (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains two separate fee paths: ETH deposits use the global `feeBps` variable, while ERC20 token deposits use `tokenFeeBps[token]`, a mapping that defaults to `0` and is never written by `addSupportedToken` or any `reinitialize` path. Any depositor can call `deposit(token, amount, referralId)` and receive the full rsETH equivalent with zero fee deducted, permanently bypassing protocol fee collection for every token whose `tokenFeeBps` has not been explicitly set via a separate admin call.

## Finding Description
The ETH deposit path at L311‚Äì312 computes fees using the global `feeBps`:

```solidity
fee = amount * feeBps / 10_000;
```

The token deposit path at L335‚Äì336 instead reads from the per-token mapping:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```

`tokenFeeBps` is declared at L88 as `mapping(address token => uint256 feeBps) public tokenFeeBps;`. Solidity initializes all mapping values to `0`. The only function that writes to this mapping is `setTokenFeeBps` (L583‚Äì594), which requires `DEFAULT_ADMIN_ROLE` and must be called separately after token onboarding.

`addSupportedToken` (L637‚Äì656) sets `supportedTokenOracle[token]` and `tokenBridge[token]` but never touches `tokenFeeBps[token]`. The `reinitialize` function at L163‚Äì185 similarly sets `tokenBridge[_token]` for wstETH without setting `tokenFeeBps[_token]`. As a result, every token added through either path has `tokenFeeBps[token] == 0` until an admin issues a separate `setTokenFeeBps` call.

The `deposit(address token, uint256 amount, string memory referralId)` function at L284‚Äì305 calls `viewSwapRsETHAmountAndFee(amount, token)`, which returns `fee = 0` when `tokenFeeBps[token] == 0`. The line `feeEarnedInToken[token] += fee` therefore adds `0`, and the caller receives the full rsETH equivalent with no fee withheld. No guard in the deposit flow checks whether `tokenFeeBps[token]` has been initialized.

## Impact Explanation
Protocol fee revenue on token deposits is the yield that should accrue to the treasury. When `tokenFeeBps[token] == 0`, every token deposit transfers the full rsETH value to the depositor ‚Äî the portion that should have been retained as a fee is instead given away. `feeEarnedInToken[token]` remains `0`, so `withdrawFees` and `bridgeTokens` collect nothing. This is a continuous, compounding loss of protocol yield for every token deposit made while the fee is unset.

**Impact: High ‚Äî Theft of unclaimed yield.** The stolen value per deposit equals `feeBps / 10_000 √ó amount √ó tokenToETHRate / rsETHToETHrate` in rsETH terms, and is immediately reachable on the live Arbitrum deployment where wstETH already has `tokenFeeBps == 0`.

## Likelihood Explanation
The precondition ‚Äî `tokenFeeBps[token] == 0` ‚Äî is the default state for every token and requires no attacker action to establish. wstETH was onboarded via `reinitialize` (L163‚Äì185) without a `setTokenFeeBps` call, so the condition is already live. Any depositor who reads the public `tokenFeeBps` mapping can confirm the zero value and immediately exploit it by calling `deposit(wstETH, amount, "")`. The exploit requires no special role, no flash loan, no oracle manipulation, and no victim cooperation. It is repeatable on every subsequent deposit until an admin corrects the fee.

**Likelihood: High.**

## Recommendation
Initialize `tokenFeeBps[token]` to the current global `feeBps` inside `addSupportedToken` and all `reinitialize` paths that onboard tokens:

```solidity
tokenFeeBps[token] = feeBps;
```

Alternatively, add a fallback in `viewSwapRsETHAmountAndFee(uint256, address)` to mirror the ETH path when the per-token fee is unset:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token] != 0 ? tokenFeeBps[token] : feeBps;
fee = amount * feeBpsForToken / 10_000;
```

The first approach is preferred because it makes the intended fee explicit and auditable at onboarding time.

## Proof of Concept
1. Read `tokenFeeBps[wstETH]` on the deployed Arbitrum `RSETHPool` ‚Äî it returns `0` because wstETH was added via `reinitialize` (L163‚Äì185) with no subsequent `setTokenFeeBps` call.
2. Call `deposit(wstETH, 10 ether, "")`.
3. Internally, `viewSwapRsETHAmountAndFee(10 ether, wstETH)` executes `fee = 10 ether * 0 / 10_000 = 0`.
4. `feeEarnedInToken[wstETH] += 0` ‚Äî treasury accrues nothing.
5. Caller receives `10 ether * tokenToETHRate / rsETHToETHrate` rsETH ‚Äî the full amount with no fee deducted.
6. Repeat for any future token added via `addSupportedToken` before `setTokenFeeBps` is called.

Foundry fork test plan: fork Arbitrum mainnet, impersonate a wstETH holder, call `deposit(wstETH, amount, "")`, assert `feeEarnedInToken[wstETH] == 0` and that the caller received more rsETH than they would have if `feeBps` had been applied.
