### Title
Unauthorized Spend of Data Layer Mirror Coins via Unguarded `P2_PARENT` Inner Puzzle — (File: `chia/wallet/db_wallet/db_wallet_puzzles.py`)

### Summary
The Data Layer mirror puzzle is constructed as `P2_PARENT.curry(Program.to(1))`, where the curried inner puzzle `1` returns the solution directly as conditions (i.e., "anyone can spend"). The only authorization check is a wallet-layer Python guard in `DataLayerWallet.delete_mirror()` that is never enforced on-chain. Any unprivileged attacker who observes a mirror coin on-chain can reconstruct its parent coin information (publicly available) and submit a valid spend bundle that redirects the mirror coin's XCH to an arbitrary address, while simultaneously removing the mirror from the Data Layer.

### Finding Description

`create_mirror_puzzle()` in `chia/wallet/db_wallet/db_wallet_puzzles.py` constructs the mirror puzzle as:

```python
def create_mirror_puzzle() -> Program:
    return P2_PARENT.curry(Program.to(1))
``` [1](#0-0) 

`P2_PARENT` is a "pay-to-parent" puzzle that verifies the spending coin's parent matches the provided parent info (via `ASSERT_MY_PARENT_ID`), then runs the curried inner puzzle against the provided inner solution. The curried inner puzzle is `Program.to(1)`, which in CLVM simply returns the solution as conditions — no signature or ownership check is performed.

The solution structure used in `DataLayerWallet.delete_mirror()` confirms this:

```python
mirror_spend = make_spend(
    mirror_coin,
    create_mirror_puzzle(),
    Program.to([
        parent_coin.parent_coin_info,
        parent_inner_puzzle,
        parent_coin.amount,
        inner_sol,
    ]),
)
mirror_bundle = WalletSpendBundle([mirror_spend], G2Element())
``` [2](#0-1) 

The bundle is submitted with `G2Element()` (empty aggregate signature), confirming the mirror spend itself requires no cryptographic authorization. The `inner_sol` is fully attacker-controlled.

The wallet-layer ownership check exists only in Python:

```python
inner_puzzle_derivation = await self.wallet_state_manager.puzzle_store.get_derivation_record_for_puzzle_hash(parent_coin.puzzle_hash)
if inner_puzzle_derivation is None:
    raise ValueError(f"DL Wallet does not have permission to delete mirror with ID {mirror_id}")
``` [3](#0-2) 

This guard is bypassed entirely when an attacker constructs the spend bundle directly, outside the wallet software.

Mirror coins are created by sending XCH to `MIRROR_PUZZLE_HASH` with memos encoding the `launcher_id` and URLs:

```python
await self.standard_wallet.generate_signed_transaction(
    amounts=[amount],
    puzzle_hashes=[create_mirror_puzzle().get_tree_hash()],
    ...
    memos=[[launcher_id, *(url for url in urls)]],
    ...
)
``` [4](#0-3) 

All information needed to spend a mirror coin (parent coin's `parent_coin_info`, puzzle reveal, and amount) is permanently recorded on-chain and publicly readable.

### Impact Explanation

An unprivileged attacker can:

1. Enumerate all unspent coins with `puzzle_hash == MIRROR_PUZZLE_HASH` from any full node.
2. For each target mirror coin, fetch its parent coin's `parent_coin_info`, puzzle reveal, and amount from the blockchain.
3. Construct a spend bundle with `inner_sol = Program.to([[CREATE_COIN, attacker_puzhash, mirror_coin.amount]])` and no aggregate signature.
4. Submit the bundle to the mempool.

Upon confirmation:
- The XCH locked in the mirror coin is transferred to the attacker — an **unauthorized spend of XCH in a Data Layer-linked coin** (Critical).
- The wallet's `singleton_removed` handler calls `dl_store.delete_mirror(parent_name)`, removing the mirror record from the victim's Data Layer database and from on-chain tracking, corrupting Data Layer mirror/store state. [5](#0-4) 

### Likelihood Explanation

Mirror coins are publicly visible on-chain. Their puzzle hash is a fixed constant (`MIRROR_PUZZLE_HASH`). All inputs required to construct the spend (parent coin info, puzzle reveal, amount) are permanently recorded on-chain. No leaked keys, admin access, or cryptographic break is required. The attack is fully scriptable.

### Recommendation

The mirror puzzle must enforce ownership on-chain. Options:

1. **Curry the owner's public key** into the mirror puzzle and require an `AGG_SIG_ME` condition in the inner solution, so only the key-holder can spend.
2. **Replace `Program.to(1)` with a standard inner puzzle** (e.g., `puzzle_for_pk(owner_pubkey)`) so the spend requires a valid BLS signature from the creator.
3. **Use a singleton-controlled spend path** that requires the Data Layer singleton to co-sign mirror deletion, binding mirror lifetime to singleton ownership.
