Audit Report

## Title
Stale `rsETHPrice` Returned with Fresh ETH/USD `updatedAt` in `RSETHPriceFeed.latestRoundData()` During Oracle Pause - (`contracts/oracles/RSETHPriceFeed.sol`)

## Summary
When `LRTOracle._updateRsETHPrice()` detects a price drop beyond `pricePercentageLimit`, it pauses and returns early at line 281 without writing `rsETHPrice`, leaving it frozen at its pre-drop (inflated) value. `RSETHPriceFeed.latestRoundData()` then returns this stale `rsETHPrice` multiplied by the ETH/USD answer, but sources `updatedAt` exclusively from the Chainlink ETH/USD feed — which continues to update normally — making the composite rsETH/USD answer appear fresh to any downstream staleness check.

## Finding Description
**Step 1 — `_updateRsETHPrice()` freezes `rsETHPrice` on pause.**
At `contracts/LRTOracle.sol` lines 277–282, when `isPriceDecreaseOffLimit` is true the function calls `_pause()` and `return`s immediately. The assignment `rsETHPrice = newRsETHPrice` at line 313 is never reached, so `rsETHPrice` retains its pre-drop value.

**Step 2 — `updateRSETHPrice()` is permanently blocked for unprivileged callers.**
`updateRSETHPrice()` at line 87 carries `whenNotPaused`, which reverts with `ContractPaused` once the oracle is paused. Only `updateRSETHPriceAsManager()` (line 94, `onlyLRTManager`) can invoke `_updateRsETHPrice()` while paused. No on-chain time bound forces the pause to lift.

**Step 3 — `RSETHPriceFeed.latestRoundData()` conflates two independent timestamps.**
At `contracts/oracles/RSETHPriceFeed.sol` lines 68–69:
```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```
`updatedAt` is taken verbatim from the Chainlink ETH/USD feed and reflects only when that feed was last updated — not when `rsETHPrice` was last written. A consumer checking `block.timestamp - updatedAt < maxStaleness` will pass the check while receiving an inflated rsETH/USD answer.

**Existing guards are insufficient.** There is no check in `latestRoundData()` for whether `RS_ETH_ORACLE` is paused, no separate `rsETHPriceUpdatedAt` timestamp, and no revert path when the rsETH component is known stale.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**
`RSETHPriceFeed` is a Chainlink-compatible price feed whose sole purpose is to deliver an accurate, fresh rsETH/USD price. During the pause window it delivers an inflated answer (pre-drop `rsETHPrice`) paired with a fresh-looking `updatedAt`, failing to fulfill its core contract. Any protocol or integration that relies on this feed's staleness indicator for correctness will silently consume incorrect data for the entire duration of the pause, which has no on-chain expiry.

## Likelihood Explanation
- `pricePercentageLimit` is a live, admin-settable parameter; a non-zero value is the expected production configuration.
- A price drop beyond the limit (LST slashing, correlated asset repricing) is a realistic market event.
- `updateRSETHPrice()` is a public, permissionless function — any address (keeper, MEV bot, or the attacker themselves) can trigger the pause once the price condition is met.
- No privileged access, front-running, or exotic precondition is required beyond the price drop itself.

## Recommendation
1. **Track rsETH price age separately**: Add a `uint256 rsETHPriceUpdatedAt` storage variable in `LRTOracle` and set it to `block.timestamp` every time `rsETHPrice` is written (line 313). In `RSETHPriceFeed.latestRoundData()`, return `min(ethUsdUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt`.
2. **Revert when paused**: Add a check in `RSETHPriceFeed.latestRoundData()` — if `RS_ETH_ORACLE` is paused, revert (e.g., `revert OraclePaused()`), forcing downstream consumers to handle unavailability explicitly rather than silently consuming stale data.
3. **Allow price write-through during pause**: Consider permitting `_updateRsETHPrice()` to write the new (lower) price even when paused, so `rsETHPrice` always reflects reality even if deposits/withdrawals remain blocked.
