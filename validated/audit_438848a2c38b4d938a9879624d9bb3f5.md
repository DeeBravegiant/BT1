Audit Report

## Title
Missing Chainlink `latestRoundData()` Validation Allows Zero/Stale Price to Corrupt rsETH Minting and Withdrawal Calculations - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validation fields and performs no check on the returned `price`. A zero price propagates directly into rsETH minting (yielding 0 rsETH for real deposited assets) and withdrawal unlock calculations (division by zero, reverting all unlocks for the affected asset). The sibling contract `ChainlinkOracleForRSETHPoolCollateral` already implements all three required checks, confirming the protocol is aware of the requirement.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at L49–55 destructures only the `answer` field from `latestRoundData()` and casts it directly to `uint256` with no guards:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

There is no check for `price <= 0`, `answeredInRound < roundId`, or `updatedAt == 0`. By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` performs all three checks before using the price: [2](#0-1) 

`LRTOracle.getAssetPrice()` delegates directly to this oracle with no additional validation: [3](#0-2) 

**Deposit path:** `getRsETHAmountToMint` computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`. If `getAssetPrice` returns 0, this evaluates to 0. The slippage guard `if (rsethAmountToMint < minRSETHAmountExpected)` only protects users who pass a non-zero minimum; users passing 0 (a valid input) receive 0 rsETH while their assets are transferred into the protocol. [4](#0-3) 

**Withdrawal path:** `getUnderlyingAmountByRsETH` computes `underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)`. A zero return from `getAssetPrice` causes a division-by-zero revert, freezing all withdrawal unlock operations for the affected asset without any victim mistake required. [5](#0-4) 

**TVL path:** `_getTotalEthInProtocol` accumulates `totalAssetAmt.mulWad(assetER)` for each supported asset. A zero `assetER` silently zeroes out that asset's ETH contribution, causing `rsETHPrice` to be computed below its true value and enabling subsequent depositors to mint rsETH at an artificially cheap rate. [6](#0-5) 

For a negative `price`, Solidity 0.8.27's checked arithmetic causes `uint256(price) * 1e18` to revert (the cast itself is a bitwise reinterpretation yielding a near-`2^256` value, and the subsequent multiplication overflows), producing a DoS on all price-dependent operations.

## Impact Explanation
**Critical – Direct loss of deposited user funds:** A user depositing with `minRSETHAmountExpected = 0` (a valid and common call pattern) receives 0 rsETH while their LST assets are permanently transferred into the protocol. This is a direct, irreversible loss of deposited funds matching the Critical impact class.

**Medium – Temporary freezing of funds:** The withdrawal unlock path unconditionally divides by `getAssetPrice(asset)`. A zero return causes a revert for every withdrawal unlock attempt for the affected asset, freezing pending withdrawals until the feed recovers. No victim mistake is required for this path.

## Likelihood Explanation
Low. Requires a Chainlink price feed registered for a supported LST asset to return `price = 0` or a stale answer. This can occur during feed initialization, aggregator migration, or a prolonged network outage where `answeredInRound < roundId`. Chainlink's own documentation explicitly warns that `latestRoundData` can return stale data and recommends staleness checks. The protocol already applies these checks in `ChainlinkOracleForRSETHPoolCollateral`, confirming awareness of the failure mode. The condition is externally triggered but is a documented, non-negligible failure mode of the oracle infrastructure.

## Recommendation
Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Optionally add a heartbeat staleness check: `if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();`.

## Proof of Concept
**Withdrawal freeze (no victim mistake required):**
1. Deploy a mock Chainlink aggregator for a supported LST asset that returns `price = 0` from `latestRoundData()`.
2. Register it via `ChainlinkPriceOracle.updatePriceFeedFor(asset, mockFeed)`.
3. A user with a pending withdrawal calls `LRTWithdrawalManager.unlockQueue(asset, ...)`.
4. Internally, `getUnderlyingAmountByRsETH` calls `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice(asset)` → returns `0`.
5. `underlyingToReceive = amount * rsETHPrice / 0` → division-by-zero revert.
6. All withdrawal unlocks for that asset are frozen until the feed is corrected.

**Deposit loss (user passes zero slippage):**
1. Same mock feed returning `price = 0`.
2. User calls `LRTDepositPool.depositAsset(asset, 10e18, 0, "")`.
3. `getRsETHAmountToMint` returns `(10e18 * 0) / rsETHPrice = 0`.
4. Slippage check `0 < 0` is false; deposit proceeds.
5. `IERC20(asset).safeTransferFrom(user, depositPool, 10e18)` executes; `IRSETH.mint(user, 0)` mints nothing.
6. User's 10 LST tokens (~10 ETH value) are permanently lost.
