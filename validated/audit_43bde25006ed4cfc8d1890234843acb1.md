### Title
Unchecked Attacker-Controlled Index into `summaries` List During Weight Proof Validation Causes Unhandled `IndexError` — (`File: chia/full_node/weight_proof.py`)

### Summary
In `_validate_sub_epoch_segments`, the `sub_epoch_n` value extracted from a peer-supplied weight proof's `sub_epoch_segments` is used directly as an index into the locally-reconstructed `summaries` list without any bounds check. A malicious peer can craft a weight proof where a segment's `sub_epoch_n` exceeds the length of `summaries`, triggering an unhandled Python `IndexError` that propagates out of the validation path and can crash the syncing node or wallet process.

### Finding Description

During weight proof validation, `_validate_sub_epoch_segments` iterates over `segments_by_sub_epoch`, whose keys are the `sub_epoch_n` values taken directly from `SubEpochChallengeSegment` objects in the peer-supplied weight proof: [1](#0-0) 

These keys are then used without bounds checking to index into `summaries`, a list whose length is fixed by the number of `sub_epoch_data` entries in the same proof: [2](#0-1) 

The same unchecked pattern appears inside `__get_rc_sub_slot`, called from the same loop: [3](#0-2) 

`summaries` is built from `weight_proof.sub_epochs` via `_map_sub_epoch_summaries` and has length `len(weight_proof.sub_epochs)`: [4](#0-3) 

An attacker can craft a proof where `sub_epochs` contains N entries (so `summaries` has length N) while `sub_epoch_segments` contains a segment with `sub_epoch_n >= N`. When the loop reaches that segment, `summaries[sub_epoch_n]` raises `IndexError`.

Neither `_validate_sub_epoch_segments` nor its callers (`validate_weight_proof_single_proc`, `validate_weight_proof_inner`) wrap this call in a `try/except`: [5](#0-4) [6](#0-5) 

The wallet's `WalletWeightProofHandler.validate_weight_proof` also calls `validate_weight_proof_inner` without catching `IndexError`: [7](#0-6) 

### Impact Explanation

An unprivileged peer on the network can send a crafted `WeightProof` message. The receiving full node or wallet calls `validate_weight_proof` / `validate_weight_proof_single_proc` on it. The unhandled `IndexError` propagates through the call stack, crashing the sync task. If the exception is not caught at the top-level handler, it terminates the node or wallet process. Even if caught, the node is forced to abort sync and retry, and a persistent attacker controlling multiple peers can keep the victim unable to sync — matching the "long-lived inability for honest nodes/wallets to process sync updates" High impact criterion.

### Likelihood Explanation

Weight proofs are exchanged over the peer-to-peer protocol during initial sync and are accepted from any connected peer without prior authentication. Any unprivileged network participant can connect and send a crafted proof. The crafting requires only setting `sub_epoch_n` in a `SubEpochChallengeSegment` to a value larger than the number of `sub_epoch_data` entries in the same proof — a trivial manipulation of the serialized data structure.

### Recommendation

Add an explicit bounds check before indexing `summaries` with `sub_epoch_n` in `_validate_sub_epoch_segments`:

```python
if sub_epoch_n >= len(summaries):
    log.error(f"sub_epoch_n {sub_epoch_n} out of range for summaries (len={len(summaries)})")
    return None
```

Apply the same guard in `__get_rc_sub_slot` before `summaries[uint32(segment.sub_epoch_n - 1)]`. Additionally, wrap the entire body of `_validate_sub_epoch_segments` in a `try/except (IndexError, Exception)` that returns `None` on any unexpected access error, consistent with the function's existing error-return contract.

### Proof of Concept

1. Connect to a victim full node or wallet as a peer.
2. Craft a `WeightProof` where:
   - `sub_epochs` contains exactly 2 `SubEpochData` entries (so `summaries` will have length 2 after `_map_sub_epoch_summaries`).
   - `sub_epoch_segments` contains one `SubEpochChallengeSegment` with `sub_epoch_n = 5` (any value ≥ 2).
   - `recent_chain_data` is populated with enough valid-looking header blocks to pass `_validate_summaries_weight` and `_get_last_ses_hash`.
3. Send the crafted proof in response to a `RequestProofOfWeight` message.
4. The victim calls `_validate_sub_epoch_segments`; the loop reaches `sub_epoch_n = 5`; `summaries[5]` raises `IndexError`; the exception propagates unhandled through `validate_weight_proof_single_proc` or `validate_weight_proof_inner`, crashing the sync task. [8](#0-7)

def map_segments_by_sub_epoch(
    sub_epoch_segments: list[SubEpochChallengeSegment],
) -> dict[int, list[SubEpochChallengeSegment]]:
    segments: dict[int, list[SubEpochChallengeSegment]] = {}
    curr_sub_epoch_n = -1
    for idx, segment in enumerate(sub_epoch_segments):
        if curr_sub_epoch_n < segment.sub_epoch_n:
            curr_sub_epoch_n = segment.sub_epoch_n
            segments[curr_sub_epoch_n] = []
        segments[curr_sub_epoch_n].append(segment)
    return segments
