Audit Report

## Title
Missing Oracle Registration Check in `addNewSupportedAsset()` Freezes `updateRSETHPrice()`, Enabling Deposits at Stale Rate — (`contracts/LRTOracle.sol`, `contracts/LRTConfig.sol`)

## Summary

`LRTConfig.addNewSupportedAsset()` adds a new asset to `supportedAssetList` with no requirement that a corresponding price oracle exists in `LRTOracle`. Once the asset is added, every call to `updateRSETHPrice()` reverts because `_getTotalEthInProtocol()` unconditionally calls `getAssetPrice()` for every supported asset, and `getAssetPrice()` hard-reverts via `onlySupportedOracle` when no oracle is registered. During this window, `rsETHPrice` remains stale while deposits of existing assets continue to execute against it, allowing depositors to mint excess rsETH at the expense of existing holders' accrued yield.

## Finding Description

`LRTConfig._addNewSupportedAsset()` sets `isSupportedAsset[asset] = true` and pushes the asset into `supportedAssetList` with no oracle check: [1](#0-0) 

`LRTOracle._getTotalEthInProtocol()` iterates over every entry in `supportedAssetList` and calls `getAssetPrice(asset)` for each: [2](#0-1) 

`getAssetPrice()` is guarded by `onlySupportedOracle`, which hard-reverts when `assetPriceOracle[asset] == address(0)`: [3](#0-2) 

`_getTotalEthInProtocol()` is called unconditionally inside `_updateRsETHPrice()`: [4](#0-3) 

Both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` call `_updateRsETHPrice()`, so both revert for the entire window: [5](#0-4) 

Meanwhile, `LRTDepositPool.getRsETHAmountToMint()` reads the stored `rsETHPrice` directly without triggering a refresh: [6](#0-5) 

Deposits of existing assets (stETH, ETH) continue to succeed because `onlySupportedERC20Token` only checks `isSupportedAsset`, which is set for the pre-existing assets: [7](#0-6) 

The numerator `lrtOracle.getAssetPrice(asset)` in `getRsETHAmountToMint` is live (fetched from the existing asset's oracle), while the denominator `lrtOracle.rsETHPrice()` is stale-low. Any depositor calling `depositAsset()` or `depositETH()` during the window receives more rsETH than they are entitled to.

## Impact Explanation

**High — Theft of unclaimed yield.** Staking rewards accrue continuously, so the true rsETH/ETH rate rises over time. While `updateRSETHPrice()` is broken, the stored `rsETHPrice` is lower than the true rate. Depositors receive excess rsETH (numerator is live, denominator is stale-low), and this excess is minted at the direct expense of existing rsETH holders whose accrued yield is diluted. This is a concrete, quantifiable transfer of value from existing holders to new depositors.

## Likelihood Explanation

`addNewSupportedAsset()` requires `TIME_LOCK_ROLE` (a timelock contract), while `updatePriceOracleFor()` requires `onlyLRTAdmin`. These are structurally separate transactions executed by different roles. The timelock execution itself creates a **guaranteed non-zero window** between the asset entering `supportedAssetList` and the admin separately calling `updatePriceOracleFor()`. Any block mined in that window where staking rewards have accrued since the last price update is exploitable by any unprivileged depositor. No admin error or compromise is required — normal governance operations create the condition. **Likelihood: Medium.**

## Recommendation

1. **Atomic registration**: Extend `addNewSupportedAsset()` (or a governance wrapper) to atomically call `LRTOracle.updatePriceOracleFor()` in the same transaction, ensuring no asset ever enters `supportedAssetList` without a live oracle.

2. **Graceful skip**: In `_getTotalEthInProtocol()`, skip assets whose `assetPriceOracle` is `address(0)` rather than reverting, so a missing oracle degrades gracefully instead of freezing all price updates.

3. **Deposit guard**: In `_beforeDeposit()`, assert that `updateRSETHPrice()` does not revert (or call it) before computing `rsethAmountToMint`, so deposits cannot proceed against a stale price.

## Proof of Concept

1. Protocol is live; `rsETHPrice` was last updated at `T=0` (e.g., `1.02e18`).
2. At `T=1`, the timelock executes `LRTConfig.addNewSupportedAsset(newLST, limit)`. `newLST` is now in `supportedAssetList`. No oracle is registered in `LRTOracle` for `newLST`.
3. At `T=2`, staking rewards push the true rsETH rate to `1.025e18`. Every call to `updateRSETHPrice()` reverts at `getAssetPrice(newLST)` → `AssetOracleNotSupported`. `rsETHPrice` remains `1.02e18`.
4. Attacker calls `depositAsset(stETH, 100e18, 0, "")`. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.02e18 ≈ 98.04 rsETH` instead of the correct `100e18 * 1e18 / 1.025e18 ≈ 97.56 rsETH`. Attacker receives ~0.48 rsETH extra per 100 stETH deposited, diluting existing holders.
5. At `T=3`, admin calls `updatePriceOracleFor(newLST, oracle)`. `updateRSETHPrice()` now succeeds and the price jumps to `1.025e18`, but the dilution already occurred and is irreversible.

**Foundry test plan**: Fork mainnet, call `addNewSupportedAsset` from the timelock address, confirm `updateRSETHPrice()` reverts with `AssetOracleNotSupported`, then call `depositAsset(stETH, 100e18, 0, "")` and assert the minted rsETH amount exceeds `100e18 * 1e18 / trueRsETHPrice`.
