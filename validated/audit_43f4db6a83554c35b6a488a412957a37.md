### Title
Deposits Paid for Non-Executed `propose_update` Proposals Are Permanently Locked in the MPC Contract — (`File: crates/contract/src/update.rs`)

---

### Summary

When `propose_update` is called, the proposer must attach a NEAR deposit proportional to the size of the uploaded contract code or config. When any one proposal reaches threshold and `do_update` executes, it unconditionally clears **all** other pending proposals — but never refunds the deposits paid by the proposers of those cleared proposals. Because `UpdateEntry` does not record the proposer's account, there is no mechanism to issue refunds, and the deposited NEAR is permanently locked in the contract.

---

### Finding Description

`propose_update` requires a deposit calculated from the byte size of the proposed update:

```rust
// crates/contract/src/lib.rs  ~L1308-1331
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
...
let id = self.proposed_updates.propose(update);
// Refund the difference if the proposer attached more than required.
if let Some(diff) = attached.checked_sub(required)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(proposer).transfer(diff).detach();
}
```

Only the *excess* above `required` is refunded. The `required` portion stays in the contract.

`ProposedUpdates::propose` stores the update but **not the proposer's account**:

```rust
// crates/contract/src/update.rs  ~L167-173
pub fn propose(&mut self, update: Update) -> UpdateId {
    let bytes_used = bytes_used(&update);
    let id = self.id.generate();
    self.entries.insert(id, UpdateEntry { update, bytes_used });
    id
}
```

`UpdateEntry` only holds `update` and `bytes_used` — no `proposer: AccountId` field exists.

When any proposal reaches threshold, `do_update` clears **all** entries:

```rust
// crates/contract/src/update.rs  ~L195-200
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
```

Because the proposer identity was never stored, there is no way to issue refunds to the proposers of the cleared proposals. The deposited NEAR tokens are permanently locked in the contract with no withdrawal function anywhere in the codebase.

---

### Impact Explanation

Every participant who proposed an update that was not the one ultimately executed loses their full storage deposit permanently. For a large contract upload (up to ~1.5 MB), the required deposit can reach **~40 NEAR** (as evidenced by `NearToken::from_near(40)` in sandbox tests). With multiple concurrent proposals — a normal governance scenario — the total locked amount scales linearly with the number of non-winning proposals. There is no admin withdrawal function, no `refund_proposal` endpoint, and no recovery path. The funds are irrecoverably locked in the MPC contract account.

This matches the allowed Medium impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

This occurs during **normal governance operation**. Any time two or more participants each propose a different update (e.g., different code versions or config values) and one reaches threshold, all other proposers lose their deposits. The README explicitly describes this clearing behavior as intentional. The likelihood is high because concurrent proposals are expected in a multi-participant governance system.

---

### Recommendation

Store the proposer's `AccountId` inside `UpdateEntry` alongside `bytes_used`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add this
    pub(super) deposit: NearToken,    // add this
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each non-executed proposal. This mirrors the pattern already used in `require_deposit` for excess refunds.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract blob, attaching ~27 NEAR (required deposit).
2. Participant B calls `propose_update` with a different 1 MB contract blob, attaching ~27 NEAR.
3. Threshold participants vote for proposal A via `vote_update`.
4. `do_update` executes: proposal A is deployed, then `self.entries.clear()` removes proposal B.
5. Participant B's ~27 NEAR deposit is now permanently locked in the contract — no refund is issued, no withdrawal function exists, and the proposer identity was never recorded.
