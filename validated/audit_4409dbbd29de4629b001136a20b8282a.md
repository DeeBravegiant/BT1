### Title
`pve::decrypt` skips ZK proof verification that `pve::verify` enforces, allowing attacker-controlled ciphertexts to yield attacker-chosen key-share output - (File: src/cbmpc/api/pve_base_pke.cpp)

### Summary
The public API function `coinbase::api::pve::decrypt` calls the internal `pve_ct.decrypt` with `skip_verify=true`, permanently bypassing the ZK proof check that `coinbase::api::pve::verify` enforces. A caller who invokes `decrypt` directly—without first calling `verify`—will accept any attacker-crafted ciphertext and recover an attacker-chosen scalar as the key share. The same omission appears in `decrypt_batch` and `combine_ac`.

### Finding Description
The PVE (Publicly Verifiable Encryption) API exposes three sibling operations over the same ciphertext type:

- `encrypt` – creates a ciphertext with an embedded ZK proof that the encrypted scalar `x` satisfies `x·G = Q`
- `verify` – validates that ZK proof against a caller-supplied `Q_compressed`
- `decrypt` – recovers `x` from the ciphertext

`verify` (lines 210–241 of `pve_base_pke.cpp`) calls `pve_ct.verify(bridge, ek_mem, Q_expected, label)`, which validates the ZK proof. [1](#0-0) 

`decrypt` (lines 247–291) calls `pve_ct.decrypt(...)` with the flag `/*skip_verify=*/true`, which unconditionally skips that same proof. [2](#0-1) 

The API provides no enforcement that `verify` must precede `decrypt`; both are standalone public entry points. The identical pattern appears in `pve_batch_single_recipient.cpp` (`decrypt_batch`): [3](#0-2)
