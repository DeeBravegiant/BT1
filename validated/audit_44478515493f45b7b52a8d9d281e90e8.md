### Title
Cross-Chain Replay of Foreign Transaction Verification — No Spent-Request Tracking in `verify_foreign_transaction` - (File: `crates/contract/src/lib.rs`, `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` flow in the MPC contract does not include any nonce, caller identity, or spent-request tracking in the `VerifyForeignTransactionRequest` key. After a response is delivered and the pending-request entry is removed, the identical foreign-chain transaction can be re-submitted by any unprivileged caller, causing the MPC network to produce a fresh (but identical) threshold signature over the same foreign-chain payload. A bridge contract that relies on the MPC attestation as its sole proof of foreign-chain finality — without independently tracking consumed attestations — is exposed to double-spend.

---

### Finding Description

**Root cause — `VerifyForeignTransactionRequest` carries no uniqueness anchor**

`VerifyForeignTransactionRequest` is defined as:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

The struct contains only the chain-specific query, the domain, and the payload version. It carries **no caller identity, no nonce, and no sequence number**. The conversion function that builds this key from user-supplied arguments is equally bare:

```rust
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [2](#0-1) 

**Root cause — pending-request entry is removed on response, leaving no spent record**

When `respond_verify_foreign_tx` is called, `resolve_yields_for` drains and **removes** the map entry:

```rust
let resumed = requests
    .remove(request)   // entry is gone after this
    ...
``` [3](#0-2) 

After the entry is removed, the contract holds no record that this `(tx_id, chain, extractors, domain_id)` tuple was ever verified. Any caller can immediately re-submit the identical `VerifyForeignTransactionRequestArgs`, the contract enqueues a new yield, and the MPC nodes will sign the same payload again.

**The contract itself acknowledges caller-agnostic keying**

The unit test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` explicitly demonstrates that Alice and Bob submitting the same request are queued under a single key and both receive the same response:

```rust
// And: caller bob submits the identical request — a different account would today
// be blocked from receiving a response by alice's submission.
``` [4](#0-3) 

**Signed payload is deterministic for the same foreign transaction**

The MPC nodes sign `SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))`. Because the same `tx_id` on a finalized foreign chain always yields the same extracted values, the resulting `payload_hash` and ECDSA signature are identical across repeated verifications of the same transaction. [5](#0-4) 

---

### Impact Explanation

The primary stated use case for `verify_foreign_transaction` is the **Omnibridge inbound flow**: a bridge service submits a foreign-chain transaction ID, the MPC network verifies finality and returns a threshold signature, and a NEAR bridge contract releases tokens in exchange for that attestation. [6](#0-5) 

Because the MPC contract removes the pending-request entry after the first response and keeps no spent-request ledger, an attacker can:

1. Observe a legitimate bridge deposit (e.g., Bitcoin `tx_id = X` sending 1 BTC to the bridge address).
2. Submit `verify_foreign_transaction(tx_id=X)` → receive MPC signature `S` over `payload_hash(X)`.
3. Present `S` to the bridge contract → receive 1 BTC-equivalent of wrapped tokens.
4. Wait for the yield to resolve and the pending-request entry to be removed.
5. Re-submit `verify_foreign_transaction(tx_id=X)` → receive the same MPC signature `S` again.
6. Present `S` a second time to a bridge contract that does not independently track consumed attestations → receive another 1 BTC-equivalent of wrapped tokens.

This is a **cross-chain replay / double-spend** condition. The MPC contract, which is supposed to be the authoritative trust anchor for foreign-chain finality, provides no guarantee that each foreign transaction is attested at most once.

**Impact class:** High — Cross-chain replay causing double-spend conditions in bridge execution.

---

### Likelihood Explanation

- **No privilege required.** `verify_foreign_transaction` is a public payable method requiring only 1 yoctoNEAR deposit.
- **No threshold collusion required.** The attacker only submits a standard user-facing call; the MPC nodes respond honestly.
- **Deterministic outcome.** The same `tx_id` always produces the same `payload_hash` and the same ECDSA signature, so the replayed attestation is byte-for-byte identical to the original.
- **Directly reachable.** The attack path is a single on-chain call repeated after the first response is consumed.

The only mitigating factor is that a well-implemented bridge contract would track consumed `payload_hash` values. However, the MPC contract's design does not enforce or document this requirement, and the contract itself provides no mechanism to prevent re-verification.

---

### Recommendation

Introduce a spent-request ledger in the MPC contract. Two concrete options:

1. **Permanent spent-set (strongest):** After `respond_verify_foreign_tx` resolves a request, insert the `VerifyForeignTransactionRequest` key into a `LookupSet<VerifyForeignTransactionRequest>`. Reject any future `verify_foreign_transaction` call whose key is already in the set. This gives the MPC contract a hard guarantee that each foreign transaction is attested at most once.

2. **Caller-scoped key (analogous to `sign`):** Include the `predecessor_account_id` in the `VerifyForeignTransactionRequest` key (as `sign` does via the `tweak` derived from `predecessor_id`). This prevents cross-caller replay but does not prevent the same caller from re-submitting after the first response is consumed.

Option 1 is the correct fix for the bridge use case. Option 2 is insufficient because the same caller (e.g., a bridge service) could still re-submit.

Additionally, the design documentation should explicitly state that downstream bridge contracts **must** independently track consumed `payload_hash` values as a defense-in-depth measure.

### Proof of Concept

```
// Round 1
Alice calls: verify_foreign_transaction({ request: Bitcoin(tx_id=X), domain_id: D, payload_version: V1 })
  → MPC nodes verify tx X on Bitcoin, sign payload_hash(X)
  → Contract returns VerifyForeignTransactionResponse { payload_hash: H, signature: S }
  → Alice presents (H, S) to bridge contract → receives 1 WBTC

// pending_verify_foreign_tx_requests entry for key K=(Bitcoin(tx_id=X), D, V1) is now REMOVED

// Round 2 (replay)
Alice calls: verify_foreign_transaction({ request: Bitcoin(tx_id=X), domain_id: D, payload_version: V1 })
  → Contract accepts (no spent-request check), enqueues new yield under same key K
  → MPC nodes verify tx X again (still finalized), sign same payload_hash(X)
  → Contract returns VerifyForeignTransactionResponse { payload_hash: H, signature: S }  // identical
  → Alice presents (H, S) to bridge contract again → receives another 1 WBTC (double-spend)
```

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
