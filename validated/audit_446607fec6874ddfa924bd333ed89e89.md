Audit Report

## Title
`addNewSupportedAsset()` leaves `assetPriceOracle` unset, permanently freezing `updateRSETHPrice()` and enabling stale-price minting - (File: contracts/LRTOracle.sol)

## Summary
`LRTConfig._addNewSupportedAsset()` appends an asset to `supportedAssetList` without registering a corresponding entry in `LRTOracle.assetPriceOracle`. Because `LRTOracle._getTotalEthInProtocol()` unconditionally calls `getAssetPrice()` for every asset in that list, and `getAssetPrice()` reverts via the `onlySupportedOracle` modifier when `assetPriceOracle[asset] == address(0)`, the entire `updateRSETHPrice()` call chain is bricked until a separate `updatePriceOracleFor()` call is made. During this window the stored `rsETHPrice` is frozen, and any depositor can mint rsETH at the stale below-market rate, extracting accrued yield from existing holders.

## Finding Description

**Root cause — `_addNewSupportedAsset()` does not set the oracle mapping**

`LRTConfig._addNewSupportedAsset()` sets `isSupportedAsset[asset] = true`, pushes `asset` onto `supportedAssetList`, and sets `depositLimitByAsset[asset]`. It makes no call to `LRTOracle.updatePriceOracleFor()`, leaving `assetPriceOracle[asset] == address(0)`. [1](#0-0) 

**Revert chain — `_getTotalEthInProtocol()` → `getAssetPrice()` → `onlySupportedOracle`**

`_getTotalEthInProtocol()` iterates every element of `lrtConfig.getSupportedAssetList()` and calls `getAssetPrice(asset)` for each. The `onlySupportedOracle` modifier on `getAssetPrice()` reverts with `AssetOracleNotSupported` when `assetPriceOracle[asset] == address(0)`. A single asset with no oracle causes the entire loop to revert. [2](#0-1) [3](#0-2) 

**`updateRSETHPrice()` and `updateRSETHPriceAsManager()` both become permanently broken**

Both public entry points call `_updateRsETHPrice()` → `_getTotalEthInProtocol()`. Neither has a bypass path. The stored `rsETHPrice` is frozen at its last written value. [4](#0-3) 

**Deposits proceed using the stale stored price**

`getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, the frozen stored value. Deposits of any existing supported asset (ETH, stETH, ETHx) are not blocked; they succeed and mint rsETH at the outdated rate. [5](#0-4) 

**Role separation widens the window**

`addNewSupportedAsset()` requires `TIME_LOCK_ROLE`; `updatePriceOracleFor()` requires `onlyLRTAdmin` (DEFAULT_ADMIN_ROLE). These are distinct roles and may be held by separate multi-sigs or subject to independent governance queues. [6](#0-5) [7](#0-6) 

## Impact Explanation

During the window between `addNewSupportedAsset()` and `updatePriceOracleFor()`:

1. `rsETHPrice` is frozen at its last value (e.g., 1.01e18).
2. EigenLayer rewards continue to accrue, increasing the true backing per rsETH (e.g., to 1.015e18).
3. Any depositor calling `depositETH()` or `depositAsset()` for an existing supported asset mints rsETH at the stale lower price, receiving more rsETH than the current backing warrants.
4. This dilutes existing rsETH holders, transferring their accrued yield to the new depositor.

**Impact: High — Theft of unclaimed yield**, which is an explicitly allowed impact class. The theft is proportional to the price drift and the deposit size; it is not bounded by any on-chain guard.

## Likelihood Explanation

Adding a new LST is a routine governance operation. The follow-up oracle registration is a separate transaction gated by a different role. Multi-sig execution delays, governance queues, or simple operational oversight can leave the window open for hours to days. The exploit requires no special access — any depositor can act the moment the new asset appears in `supportedAssetList`. The precondition (a new asset being added) is a normal, expected protocol lifecycle event, not an edge case.

**Likelihood: Medium.**

## Recommendation

Enforce atomicity at the point of asset registration. Two options:

1. **Preferred — require the oracle at registration time**: Extend `addNewSupportedAsset()` to accept a `priceOracle` address and call `LRTOracle.updatePriceOracleFor(asset, priceOracle)` in the same transaction, preserving the invariant that every supported asset always has a valid oracle.

2. **Defensive fallback — make `_getTotalEthInProtocol()` skip unregistered assets**: Replace the unconditional `getAssetPrice(asset)` call with a check on `assetPriceOracle[asset] != address(0)` before including the asset in the TVL sum. This degrades gracefully instead of bricking the price update for all assets.

Option 1 is preferred because it eliminates the invariant violation at its source.

## Proof of Concept

```
// Foundry fork test outline

1. Fork mainnet; protocol is live with rsETHPrice = 1.01e18.

2. vm.prank(timeLockRoleHolder);
   lrtConfig.addNewSupportedAsset(newLST, depositLimit);
   // newLST is now in supportedAssetList; assetPriceOracle[newLST] == address(0)

3. vm.expectRevert(LRTOracle.AssetOracleNotSupported.selector);
   lrtOracle.updateRSETHPrice();
   // Reverts — rsETHPrice frozen at 1.01e18

4. // Simulate reward accrual: warp forward, inject ETH rewards into deposit pool
   vm.deal(address(lrtDepositPool), address(lrtDepositPool).balance + rewardAmount);
   // True backing per rsETH is now ~1.015e18, but rsETHPrice still reads 1.01e18

5. vm.prank(attacker);
   lrtDepositPool.depositETH{value: 100 ether}(0, "");
   // getRsETHAmountToMint uses rsETHPrice = 1.01e18 (stale)
   // Attacker receives ~99.01 rsETH; fair amount at 1.015e18 would be ~98.52 rsETH
   // Attacker gains ~0.49 rsETH at the expense of existing holders

6. vm.prank(lrtAdmin);
   lrtOracle.updatePriceOracleFor(newLST, oracleAddr);
   lrtOracle.updateRSETHPrice(); // now succeeds; price reflects dilution
```

        UtilLib.checkNonZeroAddress(asset);
        if (depositLimit == 0) {

        }
        if (isSupportedAsset[asset]) {
            revert AssetAlreadySupported();
        }
        isSupportedAsset[asset] = true;

        depositLimitByAsset[asset] = depositLimit;

    }
