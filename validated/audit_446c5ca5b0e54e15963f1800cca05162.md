### Title
Unchecked Multiplication in `denormalize_amount` Can Silently Overflow, Causing Irreversible Fund Lock - (File: near/omni-bridge/src/lib.rs)

### Summary
`Contract::denormalize_amount` performs `amount * 10_u128.pow(diff_decimals)` with no overflow guard. In Rust compiled to WASM (release mode), integer overflow wraps silently. Because `amount` is ultimately sourced from a user-controlled EVM `initTransfer` call, a user transferring above the overflow threshold for a given token's decimal configuration will have their origin-chain tokens locked while receiving a drastically reduced (wrapped) amount on NEAR — an irreversible fund loss.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}
``` [1](#0-0) 

It is called inside `fin_transfer_callback` to convert the amount from the origin-chain proof into the NEAR-side representation before minting or transferring tokens to the recipient:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
...
fee: Self::denormalize_fee(&init_transfer.fee, decimals),
``` [2](#0-1) 

`init_transfer.amount.0` is the raw `uint128` value emitted by the EVM `InitTransfer` event, which is set directly by the unprivileged caller of `OmniBridge::initTransfer`:

```solidity
function initTransfer(
    address tokenAddress,
    uint128 amount,
    ...
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
``` [3](#0-2) 

`denormalize_fee` also delegates to `denormalize_amount` for the fee field, so the fee path shares the same defect:

```rust
fn denormalize_fee(fee: &Fee, decimals: Decimals) -> Fee {
    Fee {
        fee: U128(Self::denormalize_amount(fee.fee.0, decimals)),
``` [4](#0-3) 

The `Decimals` struct stores both sides' precision:

```rust
pub struct Decimals {
    pub decimals: u8,
    pub origin_decimals: u8,
}
``` [5](#0-4) 

`diff_decimals = origin_decimals − decimals`. For a token registered with `diff_decimals = 18` (e.g., an origin token with 24 decimals mapped to a 6-decimal NEAR token), the overflow threshold is:

```
u128::MAX / 10^18 ≈ 3.4 × 10^20
```

In 18-decimal token units that is only **~340 tokens** — directly analogous to the `341e18` threshold in the reference report. Any user transferring more than that amount triggers silent wrapping, and the wrapped result is a tiny fraction of the correct value.

The existing test suite only exercises `denormalize_amount` with `u64::MAX` as the input amount, which stays well below the overflow boundary and therefore does not catch this defect:

```rust
Contract::denormalize_amount(
    u64::MAX.into(),
    Decimals { decimals: 9, origin_decimals: 24 }
),
u128::from(u64::MAX) * 1_000_000_000_000_000_u128
``` [6](#0-5) 

### Impact Explanation

When `denormalize_amount` wraps, `transfer_message.amount` is set to a value far smaller than the amount the user locked on the origin chain. The NEAR side then mints or transfers that reduced amount to the recipient. The origin-chain tokens remain locked with no automatic refund path visible in `fin_transfer_callback`. This constitutes an **irreversible fund lock**: the user's origin-chain value is permanently trapped while they receive a negligible NEAR-side credit.

### Likelihood Explanation

The attacker path is fully unprivileged: any user calling `OmniBridge::initTransfer` on EVM with an amount above the overflow threshold for their token's `diff_decimals` triggers the bug. For tokens registered with `diff_decimals ≥ 18`, the threshold is low enough (≤ 340 tokens for 18-decimal EVM tokens) that ordinary, non-adversarial transfers will hit it. No key compromise, operator collusion, or privileged access is required.

### Recommendation

Replace the bare multiplication with a checked variant and revert on overflow:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount
        .checked_mul(10_u128.pow(diff_decimals))
        .expect("denormalize_amount: overflow")
}
```

This converts a silent, value-corrupting wrap into an explicit, recoverable panic, preventing the bridge from crediting a wrong amount and locking user funds.

### Proof of Concept

```rust
// diff_decimals = 18 (e.g. origin_decimals=24, decimals=6)
let decimals = Decimals { decimals: 6, origin_decimals: 24 };

// 341 tokens expressed in 18-decimal origin units
let amount: u128 = 341 * 10_u128.pow(18);

// Overflows u128::MAX; wraps to a tiny value instead of panicking
let result = Contract::denormalize_amount(amount, decimals);

// result << amount  →  user receives almost nothing on NEAR
// while 341 tokens remain locked on the origin chain
assert!(result < amount);
```
