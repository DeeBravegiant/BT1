Audit Report

## Title
ETH Withdrawal Permanently Frozen When Recipient Contract Cannot Receive ETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._transferAsset` uses a push-payment `call{value: amount}("")` to deliver ETH to the withdrawal recipient. If the recipient is a smart-contract wallet whose `receive()` reverts or is absent at completion time, every call to `completeWithdrawal` and `completeWithdrawalForUser` reverts, rolling back all state mutations and leaving the request permanently stuck. No admin, operator, or user-facing rescue path exists.

## Finding Description
`_processWithdrawalCompletion` is the sole code path for finalising ETH withdrawals, reached by both `completeWithdrawal` (L183-185) and `completeWithdrawalForUser` (L192-204). [1](#0-0) [2](#0-1) 

Inside `_processWithdrawalCompletion`, state mutations (`popFront`, `delete withdrawalRequests[requestId]`, `unlockedWithdrawalsCount[asset]--`) all precede the call to `_transferAsset`. [3](#0-2) 

`_transferAsset` for ETH performs a raw low-level call and reverts with `EthTransferFailed` on failure, causing the entire transaction — including all preceding state mutations — to revert. [4](#0-3) 

Because the revert rolls back `popFront` and `unlockedWithdrawalsCount--`, the request remains in the unlocked queue indefinitely. `sweepRemainingAssets` is also blocked because `hasUnlockedWithdrawals(asset)` stays `true`. [5](#0-4) 

A grep search for `cancel`, `redirect`, `rescue`, `claimETH`, `updateRecipient`, and `changeAddress` in `LRTWithdrawalManager.sol` returns no matches, confirming there is no alternative recovery path. The rsETH was already transferred to the contract at initiation time and cannot be reclaimed. [6](#0-5) 

The developer comment on `completeWithdrawalForUser` explicitly acknowledges ETH transfer issues but dismisses them as "non-impactful," which understates the permanent-freeze scenario. [7](#0-6) 

## Impact Explanation
**Permanent freezing of funds (Critical).** Once a smart-contract wallet is reconfigured to reject ETH after `initiateWithdrawal` is called, the deposited rsETH and the corresponding ETH are irrecoverably locked inside `LRTWithdrawalManager`. Neither the user nor any privileged role can complete, cancel, or redirect the withdrawal. This matches the "Permanent freezing of funds" critical impact class exactly. [4](#0-3) 

## Likelihood Explanation
Smart-contract wallets (Gnosis Safe, account-abstraction wallets, upgradeable proxies) are widely used in DeFi. The ~8-day unlock delay between `initiateWithdrawal` and `completeWithdrawal` creates a meaningful window during which a wallet can be legitimately reconfigured — threshold changes, implementation upgrades, fallback-handler removal — without the user intending to break their withdrawal. The reconfiguration need not be malicious; an inadvertent Safe upgrade that drops the fallback handler is sufficient. Likelihood is **Low**, but the impact is **Critical**, yielding an overall **High** severity. [8](#0-7) 

## Recommendation
1. **Pull-over-push**: Record the owed ETH amount in a per-user mapping inside `_processWithdrawalCompletion` (after all state updates) and expose a separate `claimETH()` function. This decouples accounting from delivery.
2. **Wrap as WETH**: Deliver WETH via `safeTransfer` instead of raw ETH; ERC-20 transfers are independent of the recipient's `receive()` function.
3. **Admin redirect / cancel**: Add a privileged function that can redirect a stuck withdrawal to a new address supplied by the user, or cancel it and return the rsETH. [4](#0-3) 

## Proof of Concept
```
1. Alice deploys a Gnosis Safe configured with a fallback handler (accepts ETH).
2. Alice calls initiateWithdrawal(ETH_TOKEN, 1e18, "").
   → rsETH transferred to LRTWithdrawalManager (L166); withdrawal request created.
3. Alice removes the fallback handler from her Safe (Safe now has no receive()).
4. Operator calls unlockQueue(ETH_TOKEN, ...) after the 8-day delay.
   → unlockedWithdrawalsCount[ETH]++ ; request is now unlocked.
5. Alice calls completeWithdrawal(ETH_TOKEN, "").
   → _processWithdrawalCompletion: popFront, delete, unlockedWithdrawalsCount-- (L705-717).
   → _transferAsset: payable(alice).call{value: 1e18}("") → Safe rejects → sent = false.
   → EthTransferFailed revert; all state changes rolled back (L879).
6. Operator calls completeWithdrawalForUser(ETH_TOKEN, alice, "") → same revert.
7. sweepRemainingAssets(ETH_TOKEN) → reverts with PendingWithdrawalsExist (L403).
8. Alice's 1e18 ETH and rsETH are permanently frozen in LRTWithdrawalManager.

Foundry test sketch:
- Deploy a MockRejectETH contract (no receive()).
- Call initiateWithdrawal from MockRejectETH.
- Fast-forward blocks past withdrawalDelayBlocks.
- Call unlockQueue.
- Assert completeWithdrawal reverts with EthTransferFailed.
- Assert withdrawalRequests[requestId] still exists (request not consumed).
- Assert address(lrtWithdrawalManager).balance unchanged.
``` [9](#0-8) [4](#0-3)

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
```

        if (asset == LRTConstants.ETH_TOKEN) {
            (bool sent,) = payable(to).call{ value: amount }("");
