Audit Report

## Title
Accrued Aave Interest Permanently Blocks ETH Withdrawal Completion When `totalETHDepositedToAave` Falls Below `request.expectedAssetAmount` - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`_withdrawFromAave` hard-caps withdrawable ETH at `min(aaveAWETH.balanceOf, totalETHDepositedToAave)`, intentionally excluding accrued interest. When sequential prior withdrawals reduce `totalETHDepositedToAave` below a pending request's `expectedAssetAmount`, but the actual aWETH balance (principal + interest) remains sufficient, `_processWithdrawalCompletion` permanently reverts with `InsufficientLiquidityForWithdrawal`. The user's withdrawal request is frozen indefinitely with no permissionless recovery path.

## Finding Description

**Root cause — `_withdrawFromAave` (lines 905–921):**

```solidity
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
    ? aaveBalance
    : totalETHDepositedToAave;   // caps at principal tracker, not actual balance

withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
``` [1](#0-0) 

The cap is `min(aaveAWETH.balanceOf, totalETHDepositedToAave)`. Accrued interest above the principal is never accessible for user withdrawals.

**Trigger site — `_processWithdrawalCompletion` (lines 719–731):**

```solidity
if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
    uint256 contractBalance = address(this).balance;
    if (contractBalance < request.expectedAssetAmount) {
        uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
        _withdrawFromAave(amountNeeded);          // capped at principal tracker
        uint256 balanceAfter = address(this).balance;
        if (balanceAfter < request.expectedAssetAmount) {
            revert InsufficientLiquidityForWithdrawal();   // ← permanently stuck
        }
    }
}
``` [2](#0-1) 

**State changes that precede the revert (all rolled back on revert):**

- Line 705: `popFront()` removes the nonce from the user's queue
- Line 712: `delete withdrawalRequests[requestId]`
- Line 717: `unlockedWithdrawalsCount[asset]--` [3](#0-2) 

Because all of these occur before the revert in the same transaction, every subsequent call to `completeWithdrawal` re-enters the same state and hits the same revert.

**Concrete trigger state:**

| Variable | Value |
|---|---|
| `totalETHDepositedToAave` | 5 ETH |
| `aaveAWETH.balanceOf(this)` | 10 ETH (5 principal + 5 accrued interest) |
| `request.expectedAssetAmount` | 10 ETH |
| `address(this).balance` | 0 ETH |

Execution: `amountNeeded = 10` → `_withdrawFromAave(10)` → `withdrawablePrincipal = min(10, 5) = 5` → withdraws 5 ETH → `balanceAfter = 5 < 10` → **reverts**.

**No permissionless recovery path:**

- `setAaveIntegrationEnabled(false)` (lines 486–497): calls `_collectInterestToTreasury()` first (sends the 5 ETH interest to treasury), then `_withdrawFromAave(aaveBalance)` — still capped at `totalETHDepositedToAave = 5`. Only 5 ETH principal lands in the contract; user still cannot withdraw 10 ETH. [4](#0-3) 

- `_collectInterestToTreasury` (lines 945–961): actively removes the interest that could have covered the shortfall, sending it to treasury before any withdrawal attempt. [5](#0-4) 

- `depositIdleETHToAave` requires idle ETH already in the contract — unavailable in this state.
- The user has no function to call that bypasses the principal cap.

## Impact Explanation

**Critical — Permanent freezing of user funds.** The user's rsETH was already transferred to the contract at `initiateWithdrawal` time. The ETH needed to satisfy the withdrawal exists in Aave as aWETH, but `totalETHDepositedToAave` prevents its retrieval. Without new ETH deposits flowing into the protocol and being deposited to Aave (increasing `totalETHDepositedToAave`), the withdrawal is permanently frozen. This directly matches the allowed Critical impact: *Permanent freezing of funds*.

## Likelihood Explanation

All four required conditions are routine in normal protocol operation — no attacker action is needed:

1. Aave integration is enabled (intended production path).
2. Interest accrues on any non-zero deposit over time (guaranteed).
3. Prior sequential withdrawals reduce `totalETHDepositedToAave` below a later request's `expectedAssetAmount` — a normal outcome of multiple users withdrawing.
4. No idle ETH sits in the contract at the time of the stuck user's call.

Any unprivileged user can trigger this by being the last (or a later) withdrawer after others have already completed their withdrawals while interest has accrued.

## Recommendation

When `totalETHDepositedToAave` is insufficient to cover a committed withdrawal, allow `_withdrawFromAave` to draw from the full `aaveAWETH.balanceOf` (principal + interest) up to the requested amount, and set `totalETHDepositedToAave` to zero rather than underflowing. Alternatively, before sweeping interest to treasury in `_collectInterestToTreasury`, verify that `assetsCommitted[ETH_TOKEN]` is fully covered by `totalETHDepositedToAave`; if not, retain the interest in the contract to satisfy pending committed withdrawals first.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;
// forge test --match-test test_withdrawalFrozenByInterestAccrual --fork-url $ETH_RPC_URL -vvvv

contract FrozenWithdrawalPoC is Test {
    LRTWithdrawalManager withdrawalManager;
    address userA;
    address userB;

    function test_withdrawalFrozenByInterestAccrual() public {
        // 1. 100 ETH deposited to Aave → totalETHDepositedToAave = 100
        // 2. Warp 365 days → Aave accrues ~5 ETH interest → aaveAWETH.balanceOf = 105
        vm.warp(block.timestamp + 365 days);

        // 3. UserA queues withdrawal for 10 ETH (expectedAssetAmount = 10e18)
        // 4. UserB queues + completes withdrawal for 95 ETH
        //    → totalETHDepositedToAave = 5, aaveAWETH.balanceOf ≈ 10 (5 principal + 5 interest)

        // 5. UserA's completeWithdrawal permanently reverts
        vm.prank(userA);
        vm.expectRevert(ILRTWithdrawalManager.InsufficientLiquidityForWithdrawal.selector);
        withdrawalManager.completeWithdrawal(LRTConstants.ETH_TOKEN, "");

        // 6. aWETH balance IS sufficient but principal tracker is not
        assertGe(withdrawalManager.getAaveBalance(), 10e18);
        assertLt(withdrawalManager.totalETHDepositedToAave(), 10e18);

        // 7. Request still exists (revert rolled back all state)
        (,uint256 expected,,) = withdrawalManager.getUserWithdrawalRequest(
            LRTConstants.ETH_TOKEN, userA, 0
        );
        assertEq(expected, 10e18); // alive but permanently uncompletable
    }
}
```

    function _collectInterestToTreasury() internal returns (uint256 interestAmount) {
        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        uint256 principal = totalETHDepositedToAave;

        // Return 0 if no interest or balance is less than principal (accounting for rounding)
        if (aaveBalance <= principal) return 0;

        interestAmount = aaveBalance - principal;

        aaveWETHGateway.withdrawETH(aavePool, interestAmount, address(this));

        address treasury = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        (bool sent,) = payable(treasury).call{ value: interestAmount }("");
        if (!sent) revert TreasuryTransferFailed();

        emit InterestCollectedToTreasury(interestAmount, treasury);
    }
