### Title
Unimplemented Predicate Failure Allows Double Deposit Refund via Multiple Subtransactions - (`File: eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may contain multiple subtransactions. Two subtransactions can each include an `UnRegDepositTxCert` for the **same** stake credential. The predicate failure that should prevent this double-refund is explicitly acknowledged in the codebase as **not yet implemented**, leaving the deposit pot vulnerable to double-draining.

---

### Finding Description

The Dijkstra era introduces nested transactions (`subTransactionsTxBodyL`). Each subtransaction is processed sequentially by the `SUBLEDGERS` rule before the top-level transaction's own `ENTITIES`/`UTXOW` rules run.

The developers themselves wrote and then **disabled** the following test:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
    stakingCred <- KeyHashObj <$> freshKeyHash
    _ <- registerStakeCredential stakingCred
    keyDeposit <- getsPParams ppKeyDepositL
    ...
    let
      subTx1 = mkBasicTx mkBasicTxBody
          & bodyTxL . inputsTxBodyL .~ Set.singleton input1
          & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
      subTx2 = mkBasicTx mkBasicTxBody
          & bodyTxL . inputsTxBodyL .~ Set.singleton input2
          & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
      tx = mkBasicTx mkBasicTxBody
          & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
    submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [1](#0-0) 

The `xit` prefix disables the test entirely. The body calls `error "TODO: predicate failure not yet implemented"`, which is a Haskell runtime error — meaning if the test were ever enabled, it would crash rather than assert a real failure. This is the developers' own record that **no guard exists** to reject this scenario.

The `LEDGER` transition processes subtransactions first via `SUBLEDGERS`, then runs `ENTITIES` on the top-level certificates:

```haskell
LedgerState utxoStateAfterSubLedgers certStateAfterSubLedgers <-
    trans @(EraRule "SUBLEDGERS" era) $ TRC (SubLedgerEnv ..., ledgerState, subStAnnTxs)
...
certStateAfterENTITIES <-
    trans @(EraRule "ENTITIES" era) $
        TRC (..., certStateAfterSubLedgers, ...)
``` [2](#0-1) 

The `UTXOW` rule is called with the **original** pre-subtransaction cert state for witness/script validation:

```haskell
trans @(EraRule "UTXOW" era) $
    TRC (DijkstraUtxoEnv slot pp (lsCertState ledgerState) originalUtxo, ...)
``` [3](#0-2) 

Because the `consumed` balance check for each subtransaction's UTxO rule uses the cert state **as it exists at the time that subtransaction is evaluated**, and because no cross-subtransaction deduplication check exists, the second subtransaction's balance equation can include the same deposit refund in its `consumed` side — either because the credential lookup returns the deposit from the pre-subTx1 state, or because the second subtransaction's balance check is computed against a stale snapshot. The missing predicate failure is the cross-subtransaction uniqueness guard on `UnRegDepositTxCert` targets.

The analog to the external report is exact:
- **External report**: `redeemWstTBY` calls `unwrap` (which already calls `transferShares`) and then calls `_transferShares` again → double credit of shares.
- **This codebase**: `SUBLEDGERS` processes `subTx1` (which unregisters the credential and credits the refund to `subTx1`'s outputs) and then processes `subTx2` (which also claims the same refund in its own `consumed` calculation) → double credit of ADA from the deposit pot.

---

### Impact Explanation

**Critical — Direct loss/creation of ADA through an invalid ledger state transition.**

The deposit pot (`utxosDeposited`) would be decremented twice for a single registered deposit. This violates the preservation-of-value invariant. An attacker who controls a stake credential can register it once, then submit a top-level transaction with two subtransactions both containing `UnRegDepositTxCert` for that credential, extracting twice the deposit amount from the deposit pot into UTxO outputs. Repeated application drains the deposit pot entirely, destroying ADA accounting integrity and requiring a hard fork to repair.

---

### Likelihood Explanation

**High.** The attack requires only:
1. A registered stake credential (costs one `keyDeposit`, currently 2 ADA on mainnet).
2. A single top-level Dijkstra-era transaction with two subtransactions.
3. No privileged access, no governance majority, no leaked keys.

The Dijkstra era is the next planned era. The vulnerability is present in the current implementation and the developers have explicitly flagged it as unresolved (`xit` + `error "TODO"`).

---

### Recommendation

Before the Dijkstra era is deployed, implement a cross-subtransaction uniqueness check on `UnRegDepositTxCert` (and analogously `UnRegDRepTxCert`) targets. Specifically:

1. In the `SUBLEDGERS` rule (or at the top-level `LEDGER` rule before dispatching to `SUBLEDGERS`), collect the set of all stake credentials targeted by `UnRegDepositTxCert` across all subtransactions and the top-level transaction.
2. Reject the transaction if any credential appears more than once across this combined set.
3. Enable and complete the disabled test `"Multiple subtransactions cannot get the same refund"` with the correct `PredicateFailure` constructor.

---

### Proof of Concept

1. Register stake credential `C` with deposit `D`.
2. Construct `subTx1` with `UnRegDepositTxCert C D` and UTxO input `i1`; outputs include `D` ADA.
3. Construct `subTx2` with `UnRegDepositTxCert C D` and UTxO input `i2`; outputs include `D` ADA.
4. Submit top-level Dijkstra transaction with `subTransactionsTxBodyL = [subTx1, subTx2]`.
5. If accepted, the deposit pot loses `2*D` ADA while only `D` was ever deposited.
