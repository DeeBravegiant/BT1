Audit Report

## Title
Missing Sequencer Uptime Feed and Heartbeat Staleness Check Enable Stale-Rate Deposits - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` but omits both a sequencer uptime feed check and a `block.timestamp - updatedAt` heartbeat staleness check. On L2s, when the sequencer is down, Chainlink oracles freeze at their last reported price. After sequencer recovery, the oracle still returns the pre-downtime stale price until a new round is pushed, allowing an attacker to deposit collateral at an inflated rate and receive excess rsETH/wrsETH from the pool's reserves.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 26–37) performs three checks after calling `latestRoundData()`:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

Both guards are insufficient for the sequencer-downtime scenario:

1. **`answeredInRound < roundID`**: This only detects if the oracle is returning data from a prior round. When the sequencer is down, no new rounds are pushed, so `answeredInRound == roundID` holds for the frozen last round — the check passes even if that round is hours old.

2. **`timestamp == 0`**: This only catches an uninitialized round. A frozen round has a non-zero `updatedAt` timestamp from before the downtime — the check passes.

3. **No sequencer uptime feed check**: Chainlink's own documentation requires L2 deployments to query the sequencer uptime feed (e.g., `ArbitrumSequencerUptimeFeed`) before trusting `latestRoundData()`. No such check exists anywhere in this contract.

The stale rate flows directly into the deposit swap calculation in every L2 pool variant:

- `RSETHPool.viewSwapRsETHAmountAndFee(amount, token)` (lines 340–346): `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate`
- `RSETHPoolV3.viewSwapRsETHAmountAndFee(amount, token)` (lines 328–334): same formula
- `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee(amount, token)` (lines 449–452): same formula

In `RSETHPool`, the pool transfers pre-minted wrsETH from its reserves to the depositor. In `RSETHPoolV3`, the pool mints new wrsETH. In both cases, if `tokenToETHRate` is stale-inflated, the depositor receives more rsETH/wrsETH than the deposited token is worth.

## Impact Explanation
**High — Theft of unclaimed yield.**

The pool's pre-minted rsETH/wrsETH reserves represent protocol-owned yield. An attacker depositing at a stale inflated rate drains these reserves faster than they are replenished: the pool disburses rsETH valued at the stale (higher) price but receives collateral worth the true (lower) price. The shortfall is a direct loss to the protocol and honest depositors. In `RSETHPoolV3`, minting new wrsETH at an inflated rate dilutes all existing rsETH holders. The magnitude scales linearly with the price drop and deposit size.

## Likelihood Explanation
**Low.** L2 sequencer downtime is rare but has occurred on Arbitrum and Optimism. The attack requires: (1) sequencer downtime long enough for a meaningful price move in a supported collateral token, and (2) the attacker acting in the window between sequencer recovery and the next Chainlink round update. This window is narrow (seconds to minutes) but exploitable by a prepared attacker monitoring the sequencer status feed. No privileged access is required — `deposit()` is a public function callable by any EOA.

## Recommendation
1. **Add a sequencer uptime check** using the appropriate Chainlink sequencer uptime feed for each L2 (e.g., `0xFdB631F5EE196F0ed6FAa767959853A9F217697D` on Arbitrum). Revert if the sequencer is down or was restarted within a grace period (~60 seconds).

2. **Add a heartbeat staleness check**: `if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();` where `MAX_STALENESS` is set per feed (e.g., 3600 seconds for a 1-hour heartbeat feed). This replaces reliance on the deprecated `answeredInRound < roundID` check.

## Proof of Concept
1. wstETH/ETH Chainlink price on Arbitrum is `1.20e18` (last updated at T=0, `answeredInRound == roundID`).
2. L2 sequencer goes offline at T=100.
3. wstETH/ETH true price drops to `1.00e18` on L1 at T=200; Chainlink L2 oracle freezes at `1.20e18`.
4. Sequencer recovers at T=3600. `latestRoundData()` still returns `1.20e18` with `answeredInRound == roundID` and `timestamp != 0` — all three existing checks in `getRate()` pass.
5. Attacker calls `deposit(wstETH, 100e18, "")` on `RSETHPool`.
6. `viewSwapRsETHAmountAndFee(100e18, wstETH)` calls `IOracle(supportedTokenOracle[wstETH]).getRate()` → returns `1.20e18` (stale).
7. `rsETHAmount = 100e18 * 1.20e18 / rsETHToETHrate` — attacker receives rsETH equivalent to 120 ETH worth of wstETH, having deposited only 100 ETH worth.
8. Pool bridges 100 wstETH (worth 100 ETH at true price) to L1 but has disbursed rsETH worth 120 ETH.
9. Net loss to pool: 20 ETH per 100 wstETH deposited.

**Foundry fork test plan**: Fork Arbitrum mainnet at a block just after a historical sequencer outage. Mock `latestRoundData()` to return a price from before the outage with `updatedAt` set to the pre-outage timestamp. Call `deposit(wstETH, 100e18, "")` and assert that `rsETHAmount` exceeds the fair value. Confirm all three existing checks in `getRate()` pass despite the stale price.
