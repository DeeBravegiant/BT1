Audit Report

## Title
Blocked User Can Bypass rsETH Transfer Restriction by Completing a Pre-Initiated Withdrawal — (File: contracts/LRTWithdrawalManager.sol)

## Summary
`RSETH.blockUserTransfers` sets a 24-hour transfer block on a user's rsETH. However, once a user has called `LRTWithdrawalManager.initiateWithdrawal`, their rsETH is held by `LRTWithdrawalManager` and later burned by `unlockQueue`. `_processWithdrawalCompletion` never checks whether the withdrawing user is blocked before disbursing ETH or LST, allowing a blocked user to fully exit the protocol through a pre-queued withdrawal.

## Finding Description
`RSETH._enforceNotBlocked` is enforced in three places: `mint` (L238), `burnFrom` (L246), and `_transfer` (L288–289). This means a currently blocked user cannot initiate a new withdrawal, because `initiateWithdrawal` (L166) calls `safeTransferFrom`, which triggers `RSETH._transfer` and checks `_enforceNotBlocked(msg.sender)`.

However, if the user called `initiateWithdrawal` **before** being blocked, the rsETH is already held by `LRTWithdrawalManager`. When the operator later calls `unlockQueue`, the rsETH is burned from `LRTWithdrawalManager` itself (L305: `IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned)`), not from the user — so `_enforceNotBlocked` is checked against `LRTWithdrawalManager`, not the blocked user.

When the user subsequently calls `completeWithdrawal` → `_processWithdrawalCompletion` (L699–738), the function performs no block check on `user` before reaching `_transferAsset(asset, user, request.expectedAssetAmount)` (L734). `_transferAsset` (L876–883) sends native ETH via `call` or LST via `safeTransfer` — neither path touches `RSETH._enforceNotBlocked`.

The admin's `recoverFrozenFunds` (L206–219) can only seize rsETH held **at the blocked address** (`balanceOf(from)`, L215). Since the rsETH was moved to `LRTWithdrawalManager` before the block was applied, `recoverFrozenFunds` finds a zero balance and recovers nothing. There is no on-chain mechanism to cancel or redirect a queued withdrawal once the rsETH has been burned.

## Impact Explanation
The blocking mechanism's purpose is to freeze a flagged user's economic position. A user who queued a withdrawal before being blocked can still receive ETH or LST through `completeWithdrawal`, fully exiting the protocol while the block is active. The admin has no on-chain path to intercept this exit once the rsETH has been burned by `unlockQueue`. This constitutes a **temporary (and in practice permanent for that position) freezing bypass**.

**Impact: Medium — Temporary freezing of funds.**

## Likelihood Explanation
The default `withdrawalDelayBlocks` is `8 days / 12 seconds` (~57,600 blocks), creating an 8-day window between `initiateWithdrawal` and `completeWithdrawal`. Any compliance event that triggers a block during this window leaves the queued withdrawal unrecoverable. No special permissions or front-running are required; the user simply waits for the delay to pass and calls `completeWithdrawal`. The admin can re-apply the block daily, but `completeWithdrawal` ignores it entirely.

**Likelihood: Medium.**

## Recommendation
Add a blocked-user check inside `_processWithdrawalCompletion` before disbursing assets:

```solidity
// In _processWithdrawalCompletion, before _transferAsset:
IRSETH rseth = IRSETH(lrtConfig.rsETH());
uint256 blockedUntil = rseth.transfersBlockedUntil(user);
if (blockedUntil != 0 && block.timestamp < blockedUntil && !rseth.isPermanentlyExempt(user)) {
    revert UserTransfersBlocked(user);
}
```

Alternatively, expose a dedicated `isBlocked(address)` view on `RSETH` and call it here. This mirrors the pattern already used in `mint` and `burnFrom`. Additionally, consider whether `unlockQueue` should skip or defer burning rsETH for requests belonging to currently blocked users, so that `recoverFrozenFunds` retains the ability to seize those tokens.

## Proof of Concept
1. Alice holds rsETH and calls `LRTWithdrawalManager.initiateWithdrawal(ETH, amount, "")`. Her rsETH is transferred to `LRTWithdrawalManager` (L166). Her own rsETH balance is now 0.
2. Admin detects suspicious activity and calls `RSETH.blockUserTransfers([Alice])`, setting `transfersBlockedUntil[Alice] = block.timestamp + 1 days` (L162–176).
3. Admin calls `RSETH.recoverFrozenFunds(Alice)` — `balanceOf(Alice)` is 0 (L215), so `super._transfer(Alice, custodyAddress, 0)` recovers nothing (L218).
4. Operator calls `unlockQueue` — Alice's rsETH is burned from `LRTWithdrawalManager` via `burnFrom(address(this), ...)` (L305). The block check in `burnFrom` is against `LRTWithdrawalManager`, not Alice.
5. After 8 days, Alice calls `completeWithdrawal(ETH, "")`. `_processWithdrawalCompletion` performs no block check and calls `_transferAsset(ETH, Alice, amount)` (L734), sending ETH to Alice via `payable(Alice).call{value: amount}("")` (L878).
6. Alice has exited the protocol with her underlying ETH despite being blocked. The admin has no remaining on-chain recourse.
