### Title
Single Attested Participant Can Submit Arbitrary CKD Response for `AppPublicKey` Requests, Bypassing Threshold Requirement — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs no cryptographic output verification when the pending request uses the `AppPublicKey` variant. A single Byzantine attested participant (strictly below the signing threshold) can call `respond_ckd` with any arbitrary `CKDResponse`, which the contract unconditionally accepts, removes from the pending-request map, and delivers to every queued caller. This unilaterally determines the confidential-key-derivation output without the threshold of participants that the protocol requires.

---

### Finding Description

In `crates/contract/src/lib.rs` at lines 675–682, `respond_ckd` branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV` requests the contract runs a BLS12-381 pairing check (`ckd_output_check`, defined in `crates/contract/src/primitives/ckd.rs` lines 80–102) that cryptographically binds the response `(big_y, big_c)` to the MPC network's public key and the user's app public key. For `AppPublicKey` requests the arm is empty — the contract performs **zero** verification of the response values.

After the match block, `resolve_yields_for` is called unconditionally:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

This removes the pending-request entry from `pending_ckd_requests` and resumes every queued yield with the (potentially fabricated) response bytes. The request lifecycle is fully completed — the pending-request accounting state is updated — but the response content is never validated.

The `AppPublicKey` variant is the legacy, privately-verifiable form. It is the variant used in the sandbox test at line 3404 and in the e2e test at `crates/e2e-tests/tests/ckd_verification.rs` line 63. It is the default path for callers that do not need public verifiability.

---

### Impact Explanation

The CKD protocol is designed so that no single participant can determine the output; the correct `(big_y, big_c)` pair requires a threshold of participants to contribute their key shares. For `AppPublicKey` requests the contract enforces none of this: one attested participant can submit `big_y = G` and `big_c = G` (or any other points) and the contract will accept and deliver them. Every caller whose yield is queued under that request key receives the fabricated confidential key. The user's derived secret is therefore controlled by the attacker, not by the MPC network.

This matches the allowed Critical impact: **confidential key derivation output without the required participant authorization** and **bypass of threshold-signature requirements**.

---

### Likelihood Explanation

The attacker must be an attested participant (enforced by `assert_caller_is_attested_participant_and_protocol_active` at line 666). A single Byzantine participant below the threshold satisfies this requirement. No collusion, no leaked key, and no privileged operator access beyond normal participant status is needed. The `AppPublicKey` variant is the most common CKD path. The attack is a single direct contract call with arbitrary `big_y`/`big_c` byte arrays.

---

### Recommendation

Add a cryptographic output check for `AppPublicKey` requests analogous to the one already present for `AppPublicKeyPV`. For the non-publicly-verifiable variant the contract cannot run a pairing check (it lacks the user's private key), so the recommended mitigation is to require that `respond_ckd` be called only after a threshold of participants have submitted matching partial responses off-chain and the leader aggregates them — or, alternatively, deprecate the `AppPublicKey` variant in favour of `AppPublicKeyPV`, which is verifiable on-chain.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)` and attaches 1 yoctoNEAR. The contract stores the yield index in `pending_ckd_requests`.

2. Malicious attested participant (participant index 0, for example) calls `respond_ckd` with the correct `CKDRequest` key and a fabricated response:
   ```json
   { "big_y": "<48 bytes of attacker's choice>",
     "big_c": "<48 bytes of attacker's choice>" }
   ```

3. `respond_ckd` passes the empty `AppPublicKey` match arm, calls `resolve_yields_for`, removes the entry from `pending_ckd_requests`, and resumes the user's yield with the fabricated bytes.

4. The user's transaction resolves with the attacker-chosen `(big_y, big_c)`. The user's derived confidential key is now under the attacker's control, not the MPC network's.

5. Any legitimate `respond_ckd` call from the honest nodes for the same request now returns `Err(RequestNotFound)` because the entry was already drained — the honest response is silently discarded.
