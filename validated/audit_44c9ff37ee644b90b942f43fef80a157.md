Based on my research, I have found a valid analog. The `DelegateAction` (meta-transaction) mechanism in nearcore has the same front-running vulnerability class as the `applyPermit()` issue: a signed authorization can be extracted from the mempool and re-submitted by an attacker, consuming the nonce and causing the original relayer's transaction to fail.

### Title
SignedDelegateAction Can Be Front-Run to Force Relayer Transaction Failure - (File: `runtime/runtime/src/actions.rs`)

### Summary
Any unprivileged actor who observes a pending transaction containing a `SignedDelegateAction` in the mempool can extract it and re-submit it inside their own outer transaction. If the attacker's transaction is included first, the delegate nonce is consumed, and the original relayer's transaction fails with `DelegateActionInvalidNonce`. This is the exact analog of the `safePermit()` front-running pattern: a signed, replayable authorization is consumed by a third party before the intended submitter's transaction executes.

### Finding Description

In NEAR's meta-transaction (NEP-366) design, a user (Alice) signs a `DelegateAction` off-chain and hands it to a relayer. The relayer wraps it in an outer `SignedTransaction` (signed by the relayer) and submits it to the network. The `SignedDelegateAction` is a self-contained, user-signed object that does **not** bind to the identity of the outer transaction's signer.

`validate_delegate_action_key()` in `runtime/runtime/src/actions.rs` performs the nonce check:

```rust
if delegate_nonce.nonce() <= current_nonce {
    result.result = Err(ActionErrorKind::DelegateActionInvalidNonce {
        delegate_nonce: delegate_nonce.nonce(),
        ak_nonce: current_nonce,
    }.into());
    return Ok(());
}
``` [1](#0-0) 

On success, the nonce is immediately written back to state:

```rust
DelegateNonceUpdate::AccessKey => {
    access_key.nonce = delegate_nonce.nonce();
    set_access_key(state_update, sender_id.clone(), public_key.clone(), &access_key);
}
``` [2](#0-1) 

Because the `SignedDelegateAction` is not bound to the outer transaction's signer, an attacker who sees the relayer's pending transaction can:

1. Extract the `SignedDelegateAction` from the relayer's transaction body.
2. Construct their own outer `SignedTransaction` (signed by the attacker, receiver = Alice) containing the same `SignedDelegateAction`.
3. Submit it with a higher gas price to be included first.
4. The attacker's transaction executes, consuming Alice's nonce.
5. The relayer's original transaction is then processed and fails with `DelegateActionInvalidNonce` because `delegate_nonce.nonce() <= current_nonce` is now true.

The `DelegateAction` struct itself confirms the nonce is the only replay-prevention mechanism and is not bound to a specific outer signer:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    /// Nonce to ensure that the same delegate action is not sent twice by a
    /// relayer and should match for given account's `public_key`.
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
``` [3](#0-2) 

The meta-transaction documentation explicitly acknowledges that the relayer is compensated through inner actions and that some level of trust is required: [4](#0-3) 

### Impact Explanation

- The relayer's outer transaction fails with `DelegateActionInvalidNonce`; the relayer loses the gas fees paid for the outer transaction with no compensation.
- The user's inner actions still execute (via the attacker's transaction), so the relayer's FT-based compensation payment (embedded in the inner actions) still goes to the relayer's account — but the relayer's outer transaction outcome is `Failure`, breaking any integration that checks the outer transaction's execution status.
- External relayer integrations that poll the outcome of the submitted transaction hash and expect `SuccessReceiptId` will observe a failure, potentially triggering incorrect retry logic, double-execution attempts, or financial accounting errors.
- The corrupted protocol value is the **execution outcome** of the relayer's transaction (recorded under the relayer's transaction hash in the outcome trie): it records `DelegateActionInvalidNonce` failure instead of success.

### Likelihood Explanation

Low. The attacker gains no direct financial benefit (they pay gas to execute Alice's action without receiving any reward). However, a competing relayer or a targeted griefing actor can reliably trigger this by monitoring the public mempool. The attack requires no special privileges — only the ability to submit a valid transaction, which any account holder can do.

### Recommendation

Analogous to the EVM fix (checking allowance before calling `safePermit()`), `validate_delegate_action_key()` should check whether the delegate nonce has already been consumed **and** whether the action was already successfully applied before returning a hard failure. A practical mitigation is for relayers to query the on-chain nonce for Alice's access key before submitting, and skip submission if the nonce is already ≥ the delegate nonce. At the protocol level, consider binding the `SignedDelegateAction` to the outer transaction's signer public key, or introducing an idempotency check: if `delegate_nonce.nonce() <= current_nonce` and the outcome for this delegate action already exists as a success, treat it as a no-op rather than a failure.

### Proof of Concept

1. Alice signs a `DelegateAction` with `nonce = N+1` and sends it to Relayer.
2. Relayer constructs `outer_tx = SignedTransaction { signer: relayer, receiver: alice, actions: [Delegate(signed_da)], nonce: R }` and broadcasts it.
3. Attacker observes `outer_tx` in the mempool, extracts `signed_da`.
4. Attacker constructs `attack_tx = SignedTransaction { signer: attacker, receiver: alice, actions: [Delegate(signed_da)], nonce: A }` and broadcasts with higher gas price.
5. `attack_tx` is included first; `validate_delegate_action_key()` passes (`N+1 > N`), Alice's access key nonce is set to `N+1`.
6. `outer_tx` is included next; `validate_delegate_action_key()` fails: `delegate_nonce.nonce() (N+1) <= current_nonce (N+1)` → `DelegateActionInvalidNonce`.
7. Relayer's transaction outcome is recorded as failure; relayer loses gas fees. [5](#0-4)

```rust
fn validate_delegate_action_key(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    delegate_action: VersionedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let sender_id = delegate_action.sender_id();
    let public_key = delegate_action.public_key();
    // 'sender_id' account existence must be checked by a caller
    let mut access_key = match get_access_key(state_update, sender_id, public_key)? {
        Some(access_key) => access_key,
        None => {
            result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
                InvalidAccessKeyError::AccessKeyNotFound {
                    account_id: sender_id.clone(),
                    public_key: public_key.clone().into(),
                },
            )
            .into());
            return Ok(());
        }
    };

    // A plain nonce advances the single access_key.nonce and forbids gas keys;
    // a gas key nonce advances one of the gas key's nonces selected by
    // nonce_index.
    let delegate_nonce = delegate_action.nonce();
    let (current_nonce, nonce_update) = match delegate_nonce {
        TransactionNonce::Nonce { .. } => {
            if access_key.gas_key_info().is_some() {
                result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
                    InvalidAccessKeyError::DelegateActionRequiresNonGasKey,
                )
                .into());
                return Ok(());
            }
            (access_key.nonce, DelegateNonceUpdate::AccessKey)
        }
        TransactionNonce::GasKeyNonce { nonce_index, .. } => {
            let Some(gas_key_info) = access_key.gas_key_info() else {
                result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
                    InvalidAccessKeyError::DelegateActionRequiresGasKey,
                )
                .into());
                return Ok(());
            };
            if nonce_index >= gas_key_info.num_nonces {
                result.result = Err(ActionErrorKind::DelegateActionInvalidNonceIndex {
                    nonce_index,
                    num_nonces: gas_key_info.num_nonces,
                }
                .into());
                return Ok(());
            }
            // The index is range-checked above and gas keys initialize every
            // nonce row at creation, so a missing row is inconsistent state.
            let current_nonce =
                get_gas_key_nonce(state_update, sender_id, public_key, nonce_index)?.ok_or_else(
                    || {
                        StorageError::StorageInconsistentState(format!(
                            "gas key nonce row missing for {} {} at in-range index {nonce_index} (num_nonces {})",
                            sender_id, public_key, gas_key_info.num_nonces,
                        ))
                    },
                )?;
            (current_nonce, DelegateNonceUpdate::GasKey { nonce_index })
        }
    };

    if delegate_nonce.nonce() <= current_nonce {
        result.result = Err(ActionErrorKind::DelegateActionInvalidNonce {
            delegate_nonce: delegate_nonce.nonce(),
            ak_nonce: current_nonce,
        }
        .into());
        return Ok(());
    }
```

In the example visualized above, the payment is done using \$FT. Together with
the transfer to John, Alice also adds an action to pay 0.1 \$FT to the relayer.
The relayer checks the content of the `SignedDelegateAction` and only processes
it if this payment is included as the first action. In this way, the relayer
will be paid in the same transaction as John.

Note that the payment to the relayer is still not guaranteed. It could be that
Alice does not have sufficient $FT and the transfer fails. To mitigate, the
relayer should check the $FT balance of Alice first.

Unfortunately, this still does not guarantee that the balance will be high
enough once the meta transaction executes. The relayer could waste NEAR gas
without compensation if Alice somehow reduces her \$FT balance in just the right
moment. Some level of trust between the relayer and its user is therefore
required.

The vision here is that there will be mostly application-specific relayers. A
general-purpose relayer is difficult to implement with just the MVP. See
limitations below.
