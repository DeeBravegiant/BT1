### Title
Stale `congestion_gas` in `StateStoredReceiptMetadata` Causes Incorrect `CongestionInfo` After Protocol Fee Upgrade — (`runtime/runtime/src/congestion_control.rs`)

### Summary

When the `use_state_stored_receipt` protocol feature is active, receipts stored in the delayed-receipt queue or outgoing buffer carry a pre-computed `congestion_gas` value in `StateStoredReceiptMetadata`. This value is computed once — at storage time — using the action fees from the current `RuntimeConfig`. After a protocol upgrade that changes action fees, `receipt_congestion_gas()` returns the stale stored value verbatim instead of recomputing it with the new fee schedule. The stale values accumulate in `CongestionInfo.delayed_receipts_gas` and `CongestionInfo.buffered_receipts_gas`, which are part of the chunk header and drive all congestion-control decisions for the shard.

For a plain `Receipt`, the gas is **recomputed** with the current config. For a `StateStoredReceipt`, the **stale stored value** is returned directly, ignoring the `config` parameter entirely. The `StateStoredReceiptMetadata` struct that holds this stale value is: [5](#0-4) 

The stale `congestion_gas` values are then subtracted from `CongestionInfo` when receipts are popped: [6](#0-5) 

And the accumulated `delayed_receipts_gas` / `buffered_receipts_gas` fields drive congestion level computation: [7](#0-6) 

These fields are stored in the `CongestionInfo` embedded in every chunk header: [8](#0-7) 

### Impact Explanation

After a protocol upgrade that changes action fees (e.g., `AccountCostIncrease` which modifies `create_account_cost` and introduces `account_creation_charge`), receipts stored before the upgrade carry stale `congestion_gas` values computed under the old fee schedule. These stale values are used to maintain `CongestionInfo.delayed_receipts_gas` and `CongestionInfo.buffered_receipts_gas` in chunk headers.

The corrupted protocol values are:
- `CongestionInfo.delayed_receipts_gas` — understated if fees increased, overstated if fees decreased
- `CongestionInfo.buffered_receipts_gas` — same

Downstream effects:
- `incoming_congestion()` and `outgoing_congestion()` return wrong levels, causing incorrect transaction admission (too many or too few transactions accepted per chunk)
- `outgoing_gas_limit()` returns wrong forwarding allowances between shards, corrupting cross-shard receipt flow control
- All nodes agree on the same wrong value (no consensus split), but the congestion control invariant — that `delayed_receipts_gas` accurately reflects the gas cost of queued work under the current fee schedule — is broken [9](#0-8) 

### Likelihood Explanation

The trigger requires two conditions to coincide: (1) `use_state_stored_receipt` is enabled (a protocol feature introduced in a recent version), and (2) a subsequent protocol upgrade changes action fees while receipts from the previous epoch remain in the delayed queue or outgoing buffer. Both conditions are realistic in the NEAR upgrade cadence. An unprivileged user can deliberately fill the delayed receipt queue by submitting many cross-shard function calls just before a known protocol upgrade, maximizing the number of stale-metadata receipts that survive into the new epoch.

### Recommendation

When retrieving a `StateStoredReceipt` from state, recompute `congestion_gas` using the current `RuntimeConfig` rather than returning the stored value, or store a protocol-version tag alongside the metadata and recompute on version mismatch. Alternatively, document that `congestion_gas` metadata is only valid for the protocol version under which it was stored and bootstrap `CongestionInfo` fresh (via `bootstrap_congestion_info`) at every epoch boundary where fees change. [10](#0-9) 

### Proof of Concept

1. At protocol version N (with `use_state_stored_receipt = true`), a user submits many cross-shard function-call transactions. The resulting receipts cannot be immediately forwarded and are buffered via `buffer_receipt()`, storing `congestion_gas` computed with version-N fees.
2. A protocol upgrade to version N+1 changes action fees (e.g., `action_receipt_creation` execution fee increases).
3. In the first chunk after the upgrade, `DelayedReceiptQueueWrapper::pop()` calls `receipt_congestion_gas()` on the buffered `StateStoredReceipt` objects. The function returns the stale version-N `congestion_gas` values.
4. `apply_congestion_changes()` subtracts these stale values from `CongestionInfo`, leaving `delayed_receipts_gas` lower than the true gas cost of remaining queued work.
5. `incoming_congestion()` computes a lower-than-actual congestion level, causing the shard to accept more new transactions than it should, worsening the backlog. [11](#0-10)

    fn buffer_receipt(
        &mut self,

        Ok(())

fn action_receipt_congestion_gas(
    receipt: &Receipt,
    config: &RuntimeConfig,
    action_receipt: VersionedActionReceipt,
) -> Result<Gas, IntegerOverflowError> {
    let prepaid_exec_gas =
        total_prepaid_exec_fees(config, &action_receipt.actions(), receipt.receiver_id())?
            .gas
            .checked_add(config.fees.fee(ActionCosts::new_action_receipt).exec_fee().gas)
            .ok_or(IntegerOverflowError)?;
    // account for gas guaranteed to be used for creating new receipts
    let prepaid_send_cost = total_prepaid_send_fees(config, &action_receipt.actions())?;
    let prepaid_gas = prepaid_exec_gas.checked_add_result(prepaid_send_cost.gas)?;

    // account for gas potentially used for dynamic execution
    let gas_attached_to_fns = total_prepaid_gas(&action_receipt.actions())?;
    let gas = gas_attached_to_fns.checked_add_result(prepaid_gas)?;

    Ok(gas)

    pub(crate) fn pop(
        &mut self,
        trie_update: &mut TrieUpdate,
        config: &RuntimeConfig,
    ) -> Result<Option<ReceiptOrStateStoredReceipt<'_>>, RuntimeError> {
        // While processing receipts, we need to keep track of the gas and bytes
        // even for receipts that may be filtered out due to a resharding event
        loop {
            // Check proof size limit before each receipt is popped.
            if trie_update.trie.check_proof_size_limit_exceed() {
                break;
            }
            let Some(receipt) = self.queue.pop_front(trie_update)? else {
                break;
            };
            let delayed_gas = receipt_congestion_gas(&receipt, &config)?;
            let delayed_bytes = receipt_size(&receipt)? as u64;
            self.removed_delayed_gas =
                self.removed_delayed_gas.checked_add(delayed_gas).ok_or(IntegerOverflowError)?;
            self.removed_delayed_bytes = self
                .removed_delayed_bytes
                .checked_add(delayed_bytes)
                .ok_or(IntegerOverflowError)?;

            // Track gas and bytes for receipt above and return only receipt that belong to the shard.
            if self.receipt_filter_fn(&receipt) {
                return Ok(Some(receipt));
            }
        }
        Ok(None)
    }
