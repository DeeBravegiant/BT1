Audit Report

## Title
Division by Zero in `getRsETHAmountToMint` When `rsETHPrice` Is Uninitialized — (File: contracts/LRTDepositPool.sol)

## Summary
`LRTOracle.rsETHPrice` is a `uint256` state variable that defaults to `0` and is never set in `initialize`. Until `updateRSETHPrice()` is called at least once, every call to `LRTDepositPool.depositETH` or `depositAsset` flows through `getRsETHAmountToMint`, which divides by `lrtOracle.rsETHPrice()` with no zero-guard, causing a Solidity 0.8 division-by-zero panic and reverting the transaction. No user funds are lost, but the deposit interface is completely non-functional during this window.

## Finding Description
`LRTOracle` declares `rsETHPrice` as a plain storage variable: [1](#0-0) 

`initialize` sets only `lrtConfig` and emits an event — it never touches `rsETHPrice`: [2](#0-1) 

The only path that writes `rsETHPrice` is `_updateRsETHPrice()`, called by the public `updateRSETHPrice()`: [3](#0-2) 

`_updateRsETHPrice` sets `rsETHPrice = 1 ether` only when `rsethSupply == 0`: [4](#0-3) 

Until that call is made, `rsETHPrice == 0`. Every deposit flows through `_beforeDeposit` → `getRsETHAmountToMint`: [5](#0-4) 

which executes the unguarded division: [6](#0-5) 

There is no `require(rsETHPrice > 0)` or equivalent check anywhere in this path. The same unguarded pattern exists in `LRTWithdrawalManager.getExpectedAssetAmount` (divides by `lrtOracle.getAssetPrice(asset)`) and `_calculatePayoutAmount` (divides by `assetPrice`): [7](#0-6) [8](#0-7) 

## Impact Explanation
All calls to `depositETH` and `depositAsset` revert with a division-by-zero panic while `rsETHPrice == 0`. No user funds are lost, but the deposit interface is completely non-functional until the price is initialized. This matches the **Low** impact tier: *"Contract fails to deliver promised returns, but doesn't lose value."*

## Likelihood Explanation
The vulnerable window opens at deployment and closes after the first successful `updateRSETHPrice()` call. Because `updateRSETHPrice()` is public and permissionless, any actor — including the first depositor — can close the window themselves. However, a depositor who does not know to call the price-update function first will receive a guaranteed revert with no clear error message. Likelihood is **Low**.

## Recommendation
Add an explicit zero-guard before each division that uses a price value:

```solidity
uint256 price = lrtOracle.rsETHPrice();
require(price > 0, "rsETHPrice not initialized");
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / price;
```

Apply the same pattern in `getExpectedAssetAmount` and `_calculatePayoutAmount`. Additionally, call `_updateRsETHPrice()` (or directly set `rsETHPrice = 1 ether`) inside `initialize` so the contract is safe from block zero.

## Proof of Concept
1. Deploy `LRTOracle` and `LRTDepositPool`; do **not** call `updateRSETHPrice()`.
2. Confirm `LRTOracle.rsETHPrice() == 0`.
3. Call `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. Transaction reverts with a Solidity panic (division by zero) because `getRsETHAmountToMint` executes `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` where `rsETHPrice == 0`.
5. Call `LRTOracle.updateRSETHPrice()` (public, no role required); this sets `rsETHPrice = 1 ether` because `rsethSupply == 0`.
6. Repeat step 3 — deposit succeeds.
