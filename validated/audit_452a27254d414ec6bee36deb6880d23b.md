Audit Report

## Title
Missing Sequencer Uptime Feed Check in `ProtectedPriceProviderL2` Allows Stale-Price Swaps During and After L2 Sequencer Downtime — (`smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`, `smart-contracts-poc/contracts/PriceProviderL2.sol`)

## Summary

`ProtectedPriceProviderL2` is the canonical L2 price provider for production pools on Arbitrum, Base, Optimism, and other L2 deployments. Its sole L2 safety gate is a `refTime`-based staleness check (`_isStale`). Neither it nor `PriceProviderL2` queries a Chainlink sequencer uptime feed or enforces a post-restart grace period. During L2 sequencer downtime, oracle `refTime` freezes at the pre-downtime value; `_isStale` returns `false` for up to `MAX_TIME_DELTA` seconds (up to 7 days), and swaps execute at the frozen stale price. The registry ABI confirms the intended production interface includes `sequencerUptimeFeed()` and `GRACE_PERIOD()`, which are absent from the source.

## Finding Description

`ProtectedPriceProviderL2._computeBidAsk` performs a single staleness check:

```solidity
// ProtectedPriceProviderL2.sol lines 206-209
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
```

`_isStale` only compares `refTime` age against `MAX_TIME_DELTA`:

```solidity
// ProtectedPriceProviderL2.sol lines 138-153
return (nowTs - refTime) > maxDelta;
```

There is no `sequencerUptimeFeed.latestRoundData()` call and no `GRACE_PERIOD` enforcement anywhere in either contract. The constructor accepts no `_sequencerUptimeFeed` parameter. The registry ABI (`registry.json` lines 632–636, 722–732, 984–994) shows the intended deployed interface exposes `sequencerUptimeFeed()` (returning `AggregatorV3Interface`) and `GRACE_PERIOD()`, and the factory's `createPriceProvider` in the registry ABI (lines 2001–2005) includes `_sequencerUptimeFeed` as a required parameter — all absent from the actual source. `PriceProviderFactoryL2.createPriceProvider` (lines 41–79) passes no sequencer feed to the deployed `PriceProviderL2`.

When the L2 sequencer goes down, the offchain oracle (Pyth/Chainlink) stops receiving updates; `refTime` freezes. For any downtime shorter than `MAX_TIME_DELTA`, `_isStale` returns `false` and `getBidAndAskPrice()` returns the frozen pre-downtime bid/ask. Transactions submitted via the L1 rollup inbox bypass the sequencer and reach the pool directly. When the sequencer restarts, oracle data remains stale (from before downtime) but within `MAX_TIME_DELTA`, so swaps immediately resume at the pre-downtime price before the oracle refreshes.

The full call path is: `MetricOmmPool.swap` → `_getBidAndAskPriceX64` → `IPriceProvider(activePriceProvider).getBidAndAskPrice()` → `ProtectedPriceProviderL2.getBidAndAskPrice` → `_getBidAndAskPrice` → `_computeBidAsk` → `_isStale` (sole gate, passes on frozen price).

## Impact Explanation

The pool is a pure oracle market maker — its solvency guarantee rests entirely on the oracle price being current. A stale bid/ask returned by `getBidAndAskPrice()` directly corrupts every swap output: the pool settles trades at the pre-downtime price while the real market has moved. An attacker who knows the real price has dropped can buy the underpriced asset from the pool at the stale ask, extracting the full price difference from LP funds. This is a direct loss of LP principal, matching the "bad-price execution: stale bid/ask quote reaches a pool swap" and "pool insolvency" allowed impact categories.

## Likelihood Explanation

Arbitrum and Optimism sequencers have documented downtime events. The protocol explicitly deploys `PriceProviderFactoryL2` on Arbitrum, Base, Optimism, Avalanche, BSC, Polygon, and others (registry deployments, lines 5800–5836). No privileged action is required: any public trader can submit a swap via the L1 rollup inbox during downtime, or execute immediately after sequencer restart before the oracle refreshes. `MAX_TIME_DELTA` is configurable up to 7 days, creating a very wide exploitation window. The attack is repeatable across every sequencer downtime event.

## Recommendation

Add a sequencer uptime feed check to `ProtectedPriceProviderL2` (and `PriceProviderL2`) consistent with the registry ABI's intended interface:

```solidity
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

function _isSequencerUp() internal view returns (bool) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer != 0) return false;
    if (block.timestamp - startedAt < GRACE_PERIOD) return false;
    return true;
}
```

Call `_isSequencerUp()` at the top of `_computeBidAsk` / `_getBidAndAskPrice` and return `(0, type(uint128).max)` if it returns `false`. Pass `_sequencerUptimeFeed` through `PriceProviderFactoryL2.createPriceProvider` matching the registry ABI's intended signature.

## Proof of Concept

1. Deploy `ProtectedPriceProviderL2` on Arbitrum with `MAX_TIME_DELTA = 3600`.
2. Oracle publishes price `P = 2000 USDC/ETH` at `T = 0`; `refTime = T`.
3. Arbitrum sequencer goes down at `T = 1`. Oracle stops updating; `refTime` stays at `T`.
4. Real market price drops to `P' = 1000 USDC/ETH` during downtime.
5. At `T = 1800` (30 min into downtime), attacker submits a swap via the L1 rollup inbox.
6. `_isStale(T, T+1800, 3600, ...)` → `1800 ≤ 3600` → `false` (not stale).
7. `getBidAndAskPrice()` returns bid/ask derived from `P = 2000`.
8. Attacker buys ETH at the stale 2000 USDC price; LP loses 1000 USDC per ETH traded.
9. Sequencer restarts at `T = 3000`. Attacker front-runs the first oracle update, repeating the swap before `refTime` refreshes.

Foundry test: mock the oracle to return a fixed `refTime = T`, `vm.warp(T + MAX_TIME_DELTA - 1)`, call `getBidAndAskPrice()`, assert it returns a non-stale quote at the frozen price. Then add a mock sequencer feed returning `answer = 0, startedAt = T + 1` and assert the call reverts `FeedStalled` — demonstrating the missing guard.
