### Title
Irreversible Eviction of Valid Mempool Transactions on Rejected Admission — (`crates/apollo_mempool/src/mempool.rs`)

### Summary

`Mempool::try_make_space` permanently removes stuck-account transactions from the pool before the capacity check for the incoming transaction is confirmed to succeed. When the incoming transaction is still too large after eviction, `handle_capacity_overflow` returns `MempoolError::MempoolFull` but the evicted transactions are never restored. An unprivileged submitter can craft a single oversized transaction to permanently drain stuck-but-valid transactions from the mempool at zero cost.

### Finding Description

The admission path in `add_tx_validations` calls `handle_capacity_overflow` when the pool is full: [1](#0-0) 

`handle_capacity_overflow` calls `try_make_space`, which **immediately removes** transactions from the pool and the queue: [2](#0-1) 

If `try_make_space` returns `false` (freed bytes still insufficient), `handle_capacity_overflow` returns `Err(MempoolError::MempoolFull)`: [3](#0-2) 

There is no rollback. The evicted transactions are gone. The test explicitly documents this as the current behavior: [4](#0-3) 

The analog to the external report is exact: in the DeFi bug, reserve funds are **added** to pool collateral before liquidation collateral is **subtracted**, causing a capacity overflow that aborts the operation while leaving the state mutated. Here, eviction (the subtraction) is **committed** before the capacity check is confirmed to succeed, and the operation is aborted while leaving the state mutated.

The `creating_gap` guard only blocks eviction when the incoming transaction itself would create a nonce gap: [5](#0-4) 

An attacker using a fresh account (no existing pool transactions, nonce = 0) bypasses this guard entirely, because `creating_gap` evaluates to `false` for a gap-free submission.

### Impact Explanation

An unprivileged user submits a single invoke transaction with a large signature (e.g., 20+ `Felt` elements, as demonstrated in the test at line 1895–1938). The mempool evicts all stuck-account transactions to attempt to make room. If the incoming transaction is still too large, it is rejected with `MempoolFull`, but the evicted transactions are permanently gone. The attacker pays nothing (the transaction is rejected before sequencing). Legitimate users whose stuck transactions were evicted must resubmit, and the attack is repeatable.

This matches the allowed impact: **High — Mempool admission rejects valid transactions before sequencing.** [6](#0-5) 

### Likelihood Explanation

The mempool is reachable from the public HTTP gateway. No special account state or privilege is required beyond submitting a transaction whose byte size exceeds the combined size of all stuck-account transactions currently in the pool. The `creating_gap` guard is trivially bypassed with a fresh account at nonce 0. The attack requires only one RPC call per eviction cycle.

### Recommendation

Perform a dry-run capacity check before mutating pool state. Compute `total_evictable_bytes` by summing the sizes of all evictable accounts without removing them. Only if `total_evictable_bytes >= required_space` should `try_make_space` proceed with actual removal. Alternatively, collect evicted transactions into a temporary buffer and restore them if the incoming transaction is ultimately rejected.

### Proof of Concept

The existing test `rejects_or_accepts_tx_based_on_freed_space` (case `insufficient_eviction_space`) already demonstrates the invariant violation: [6](#0-5) 

1. Fill the mempool with `tx1` and `tx2` from address `"0x1"` (nonce 1 and 2, account nonce 0 → nonce gap → evictable).
2. Submit `large_tx` from address `"0x0"` (nonce 0, account nonce 0 → no gap → eviction path taken).
3. `try_make_space` removes `tx1` and `tx2` from the pool.
4. `large_tx` is still too large → `MempoolError::MempoolFull` returned.
5. `tx1` and `tx2` are permanently absent: `assert!(!mempool.tx_pool.contains_account(contract_address!("0x1")))` passes.

The attacker's transaction costs nothing (rejected), yet two valid transactions are permanently removed from the mempool.

#[rstest]
#[case::insufficient_eviction_space(20, false)]
#[case::sufficient_eviction_space(8, true)]
fn rejects_or_accepts_tx_based_on_freed_space(
    #[case] signature_size: usize,
    #[case] expect_success: bool,
) {
    let tx1 = add_tx_input!(tx_hash: 1, address: "0x1", tx_nonce: 1, account_nonce: 0);
    let tx2 = add_tx_input!(tx_hash: 2, address: "0x1", tx_nonce: 2, account_nonce: 0);

    let large_signature = vec![felt!("0x0"); signature_size];
    let large_tx = AddTransactionArgs {
        tx: internal_invoke_tx(invoke_tx_args!(
            tx_hash: tx_hash!(3),
            signature: TransactionSignature(large_signature.into())
        )),
        account_state: AccountState { address: contract_address!("0x0"), nonce: nonce!(0) },
    };

    let capacity = tx1.tx.total_bytes() + tx2.tx.total_bytes();
    let mut mempool = Mempool::new(
        MempoolConfig {
            static_config: MempoolStaticConfig {
                capacity_in_bytes: capacity,
                ..Default::default()
            },
            ..Default::default()
        },
        Arc::new(FakeClock::default()),
    );

    for tx in [&tx1, &tx2] {
        add_tx(&mut mempool, tx);
    }

    if expect_success {
        add_tx(&mut mempool, &large_tx);
        assert!(mempool.tx_pool.get_by_tx_hash(large_tx.tx.tx_hash()).is_ok());
    } else {
        add_tx_expect_error(&mut mempool, &large_tx, MempoolError::MempoolFull);
    }

    // Transactions tx1 and tx2 are evicted regardless of whether large_tx is accepted or rejected.
    // We do not revert the eviction attempt even if adding large_tx ultimately fails.
    assert!(!mempool.tx_pool.contains_account(contract_address!("0x1")));
