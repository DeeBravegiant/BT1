### Title
STRK Native Fees Collected in `init_transfer` Are Permanently Locked With No Recovery Path — (`starknet/src/omni_bridge.cairo`)

### Summary

The Starknet `OmniBridge.init_transfer` pulls STRK tokens from the caller as a `native_fee` and deposits them into the contract. The contract has no function to distribute, claim, or withdraw those STRK tokens. They are permanently locked.

### Finding Description

In `starknet/src/omni_bridge.cairo`, the public `init_transfer` function accepts an optional `native_fee` denominated in STRK:

```cairo
if native_fee > 0 {
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer_from(caller, get_contract_address(), native_fee.into());
    assert(success, 'ERR_FEE_TRANSFER_FAILED');
}
``` [1](#0-0) 

The STRK tokens are transferred into the contract (`get_contract_address()`). The contract's entire public interface is:

```cairo
fn log_metadata(...)
fn deploy_token(...)
fn fin_transfer(...)
fn init_transfer(...)
fn upgrade_token(...)
fn set_pause_flags(...)
fn pause_all(...)
fn get_token_address(...)
fn is_bridge_token(...)
fn is_transfer_finalised(...)
``` [2](#0-1) 

None of these functions distribute or withdraw STRK. The `fin_transfer` function only handles `payload.token_address` tokens (bridge tokens via `mint` or native tokens via ERC20 `transfer`) — it has no branch for releasing accumulated STRK native fees:

```cairo
fn fin_transfer(...) {
    ...
    if self.is_bridge_token(payload.token_address) {
        IBridgeTokenDispatcher { contract_address: payload.token_address }
            .mint(payload.recipient, payload.amount.into());
    } else {
        let success = IERC20Dispatcher { contract_address: payload.token_address }
            .transfer(payload.recipient, payload.amount.into());
        assert(success, 'ERR_TRANSFER_FAILED');
    }
    ...
}
``` [3](#0-2) 

There is no `claim_fee`, no admin rescue, and no `withdraw` function anywhere in the contract. The contract storage has no accounting for accumulated native fees: [4](#0-3) 

**Contrast with EVM**: On EVM, the `SECURITY.md` explicitly documents that native fees (ETH) are recovered via `finTransfer` with `tokenAddress=address(0)`, which sends ETH to the recipient. No equivalent mechanism exists on Starknet. [5](#0-4) 

### Impact Explanation

Every call to `init_transfer` with `native_fee > 0` permanently locks STRK tokens in the Starknet OmniBridge contract. The intended relayer never receives the fee. There is no recovery path — no admin function, no claim function, no upgrade path that could release these tokens without a full contract replacement. This is an irreversible fund lock of user-paid bridge fees.

**Impact class**: Critical — Irreversible fund lock / permanently unclaimable user value in bridge fee flows.

### Likelihood Explanation

`init_transfer` is a fully public, permissionless function. Any user who follows the documented bridge protocol and pays a STRK native fee triggers the lock. The README explicitly documents `native_fee` as a supported fee mechanism. Likelihood is high — every user paying a native fee on Starknet is affected.

### Recommendation

Add a fee distribution mechanism to the Starknet contract. The simplest fix mirrors the EVM approach: in `fin_transfer`, when `payload.fee_recipient` is non-empty and the transfer message carried a `native_fee`, transfer the corresponding STRK amount to `payload.fee_recipient`. Alternatively, add an admin-callable `claim_native_fee(recipient, amount)` function, or track accumulated fees per relayer and add a `claim_fee` entrypoint analogous to the NEAR bridge's `claim_fee`.

### Proof of Concept

1. User calls `init_transfer(token_address, 1000, 0, 500, "near:recipient.near", "")` on the Starknet OmniBridge, where `500` is the STRK `native_fee`.
2. The contract executes `transfer_from(caller, get_contract_address(), 500)` — 500 STRK enters the contract.
3. The relayer observes the `InitTransfer` event and submits the transfer on NEAR.
4. NEAR eventually calls `fin_transfer` on Starknet with the signed payload.
5. `fin_transfer` mints/transfers `payload.amount` of `payload.token_address` to `payload.recipient`. The 500 STRK sitting in the contract is never touched.
6. The 500 STRK is permanently locked. No function in the contract can release it.
