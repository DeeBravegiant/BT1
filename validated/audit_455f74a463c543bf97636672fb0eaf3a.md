Audit Report

## Title
FIFO Queue Head-of-Line Blocking via Oversized Withdrawal Request - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`initiateWithdrawal` validates the requested amount against the full protocol TVL via `getAvailableAssetAmount`, while `_unlockWithdrawalRequests` sources available funds exclusively from `unstakingVault.balanceOf(asset)` — the vault's idle balance only. Because the loop breaks immediately on the first request it cannot cover, an attacker can queue a single large request that permanently stalls `nextLockedNonce`, blocking every subsequent user from completing their withdrawal until the vault accumulates enough idle assets to cover the attacker's request.

## Finding Description
**Admission check (L170):** `initiateWithdrawal` calls `getAvailableAssetAmount`, which computes `totalAssets - assetsCommitted[asset]` where `totalAssets = lrtDepositPool.getTotalAssetDeposits(asset)`. [1](#0-0)  `getTotalAssetDeposits` sums all protocol locations including EigenLayer-staked and unstaking assets. [2](#0-1)  In a live restaking protocol, this value is orders of magnitude larger than the vault's idle balance.

**Unlock gate (L849):** `_getUnlockParams` sets `totalAvailableAssets` to `unstakingVault.balanceOf(asset)` — only the vault's idle balance. [3](#0-2) 

**FIFO break (L800):** The unlock loop breaks immediately when the front-of-queue request's `payoutAmount` exceeds `availableAssetAmount`. [4](#0-3)  There is no `continue` path, no skip mechanism, and no way to process smaller requests behind a large one.

**Nonce freeze (L815):** `nextLockedNonce[asset]` is only written after the loop exits, so if the loop never advances past nonce N, the nonce is frozen at N indefinitely. [5](#0-4) 

**Victim lock (L707):** `completeWithdrawal` reverts with `WithdrawalLocked()` for any user whose nonce is ≥ `nextLockedNonce[asset]`. [6](#0-5)  Since victims' rsETH was already transferred in at `initiateWithdrawal` time, they cannot recover it. [7](#0-6) 

**Exploit flow:**
1. Protocol TVL = 1000 stETH; vault idle balance = 10 stETH; `assetsCommitted = 0`.
2. Attacker holds rsETH ≈ 900 stETH, calls `initiateWithdrawal(stETH, rsETH_for_900_stETH)`. Check passes: `900 < 1000 - 0`. Attacker receives nonce N.
3. 100 victims each queue 1 stETH withdrawals, receiving nonces N+1…N+100.
4. Operator calls `unlockQueue`. `totalAvailableAssets = 10 stETH`. Loop hits nonce N: `900 > 10` → `break`. `nextLockedNonce` stays at N.
5. All 100 victims revert with `WithdrawalLocked()`. Resolution requires the vault to accumulate 900 stETH.

## Impact Explanation
This is **Temporary Freezing of Funds** (Medium). Victims' rsETH is held in the contract and cannot be recovered until the vault accumulates enough idle assets to cover the attacker's request. Depending on the attacker's request size relative to the vault's accumulation rate, this freeze can persist for an extended and operationally disruptive period. The freeze is not permanent only because the protocol can, in principle, unstake validators to refill the vault — but this requires extraordinary operational measures and is not guaranteed to be timely.

## Likelihood Explanation
The attack is permissionless: any rsETH holder can call `initiateWithdrawal`. The attacker does not lose funds — their rsETH remains claimable once the vault eventually accumulates enough assets. The attacker's only cost is locking their own rsETH alongside the victims'. No oracle manipulation, governance capture, or admin compromise is required. The attack is repeatable: after the vault finally accumulates enough to cover the first blocking request, the attacker (or another actor) can immediately queue another oversized request.

## Recommendation
1. **Skip-on-insufficient-funds:** Replace `break` with `continue` at L800 so smaller requests behind a large one can still be processed. Track skipped nonces in a separate data structure for revisiting when more assets arrive.
2. **Cap withdrawal amount at unlock time:** Enforce that `expectedAssetAmount` at `initiateWithdrawal` cannot exceed a configurable fraction of the vault's current idle balance, not the total TVL.
3. **Per-request maximum:** Add a protocol-level cap on the maximum single withdrawal amount relative to the vault's current balance to prevent any single request from monopolizing the queue.
