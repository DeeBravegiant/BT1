Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` + `LRTOracle.updateRSETHPrice()` Enables Yield Theft via Sandwich Attack - (`contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary

`FeeReceiver.sendFunds()` has no access control, allowing any caller to flush accumulated MEV/execution-layer rewards into `LRTDepositPool` at an attacker-chosen moment. Because `LRTOracle.updateRSETHPrice()` is also permissionless, an attacker can atomically: deposit at the stale pre-reward price, flush rewards, update the price, and lock in a withdrawal at the inflated price — extracting a proportional share of the reward from existing depositors.

## Finding Description

**Root cause:** Two permissionless state-changing functions create an exploitable ordering gap.

`FeeReceiver.sendFunds()` carries no role check: [1](#0-0) 

Once called, the ETH lands in `LRTDepositPool` and is immediately reflected in `getETHDistributionData()` via `address(this).balance`: [2](#0-1) 

`LRTOracle.updateRSETHPrice()` is `public whenNotPaused` — no role required: [3](#0-2) 

Deposits mint rsETH using the **cached** `rsETHPrice`: [4](#0-3) 

`initiateWithdrawal` locks `expectedAssetAmount` at the **current** `rsETHPrice` at initiation time: [5](#0-4) [6](#0-5) 

At unlock time, `_calculatePayoutAmount` pays the **minimum** of the locked `expectedAssetAmount` and the current return — meaning the attacker's locked-in high price is a ceiling, not a floor, protecting them from downside while capturing the full inflated amount: [7](#0-6) 

**Exploit flow (single atomic bundle):**

1. Attacker calls `depositETH{value: X}` → receives `rsETH_amount = X / rsETHPrice_old` (stale, pre-reward price).
2. Attacker calls `FeeReceiver.sendFunds()` → R ETH of rewards move to `LRTDepositPool`; TVL increases by R.
3. Attacker calls `LRTOracle.updateRSETHPrice()` → `rsETHPrice_new = (TVL_old + X + R) / rsETH_supply_new > rsETHPrice_old`.
4. Attacker calls `initiateWithdrawal(ETH, rsETH_amount)` → `expectedAssetAmount = rsETH_amount * rsETHPrice_new / 1e18 > X`.

After `withdrawalDelayBlocks`, when an operator calls `unlockQueue`, the attacker's `expectedAssetAmount` (locked at the inflated price) is paid out. Profit ≈ `rsETH_amount * (rsETHPrice_new - rsETHPrice_old)` ≈ attacker's TVL share × R.

**Why existing guards fail:**

The `pricePercentageLimit` check in `_updateRsETHPrice` only blocks the call if the price increase exceeds the configured threshold **and** the caller lacks `MANAGER` role: [8](#0-7) 

This guard fails because: (a) `pricePercentageLimit` can be zero (no limit at all); (b) for small R relative to TVL, the price increase stays within any configured limit; (c) the attacker can front-run a legitimate `sendFunds()` call and back-run with `updateRSETHPrice()` + `initiateWithdrawal()`.

The `getAvailableAssetAmount` check in `initiateWithdrawal` does not prevent the attack — after `sendFunds()`, the pool's balance includes R, so the attacker's inflated `expectedAssetAmount` passes the availability check: [9](#0-8) [10](#0-9) 

## Impact Explanation

**High — Theft of unclaimed yield.** The attacker extracts MEV/execution-layer rewards proportional to their deposit size relative to total TVL. Existing depositors receive correspondingly less yield. The attack is repeatable every reward accumulation cycle and requires no privileged access.

## Likelihood Explanation

**Medium.** Requirements: (1) monitor `FeeReceiver` balance for accumulated rewards; (2) capital to deposit (returned after the withdrawal delay minus gas); (3) call two permissionless functions atomically with deposit and withdrawal initiation via a standard MEV bundle. No privileged access is needed. The attack is feasible for any sophisticated on-chain actor and is repeatable.

## Recommendation

1. **Restrict `FeeReceiver.sendFunds()`** to a trusted role (e.g., `MANAGER`) so rewards cannot be flushed at an attacker-chosen moment:
   ```solidity
   function sendFunds() external onlyRole(LRTConstants.MANAGER) { ... }
   ```
2. **Alternatively**, call `_updateRsETHPrice()` atomically inside `receiveFromRewardReceiver()` so the price is updated before any subsequent deposit or withdrawal in the same block can exploit the gap.
3. **Or**, when computing `expectedAssetAmount` in `initiateWithdrawal`, snapshot the price and use the **lower** of the deposit-time price and the withdrawal-initiation price, preventing a user from locking in a price that was just inflated in the same block.

## Proof of Concept

```
// Foundry fork test outline
function testYieldTheftSandwich() public {
    // Setup: FeeReceiver has accumulated R ETH in rewards
    // Record rsETHPrice_old = lrtOracle.rsETHPrice()

    // tx1: attacker deposits X ETH
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: X}(0, "");
    uint256 rsETHBalance = rsETH.balanceOf(attacker);

    // tx2: flush rewards
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // tx3: update price
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();
    uint256 rsETHPrice_new = lrtOracle.rsETHPrice();
    // assert rsETHPrice_new > rsETHPrice_old

    // tx4: initiate withdrawal at inflated price
    rsETH.approve(address(lrtWithdrawalManager), rsETHBalance);
    vm.prank(attacker);
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHBalance, "");
    // expectedAssetAmount = rsETHBalance * rsETHPrice_new / 1e18 > X

    // advance blocks past withdrawalDelayBlocks
    vm.roll(block.number + withdrawalDelayBlocks + 1);

    // operator unlocks queue
    vm.prank(operator);
    lrtWithdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, 0, type(uint256).max, 0, type(uint256).max);

    // attacker completes withdrawal
    uint256 balanceBefore = attacker.balance;
    vm.prank(attacker);
    lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    uint256 received = attacker.balance - balanceBefore;

    // assert received > X (profit = attacker's share of R)
    assertGt(received, X);
}
```
