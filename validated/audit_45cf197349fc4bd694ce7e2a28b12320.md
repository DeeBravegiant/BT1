### Title
PromiseYield Timeout Queue Ordering Assumption Broken by `yield_timeout_length_in_blocks` Protocol Upgrade - (`File: runtime/runtime/src/lib.rs`)

### Summary

The `resolve_promise_yield_timeouts` function in `runtime/runtime/src/lib.rs` processes the `PromiseYieldTimeout` queue by iterating from the front and **breaking on the first entry whose `expires_at` exceeds the current block height**. This is correct only if the queue is strictly ordered by `expires_at`. However, if the `yield_timeout_length_in_blocks` runtime parameter is **increased** via a protocol upgrade while yields are already queued, newer entries will have a larger `expires_at` than older ones — preserving order. But if it is **decreased**, newer entries will have a *smaller* `expires_at` than older entries already in the queue, violating the monotonicity assumption. The loop then stops at the first old (not-yet-expired) entry and never reaches the newer entries that have already expired, permanently blocking their timeout callbacks from firing until the older entries expire.

### Finding Description

When a contract calls `promise_yield_create`, the runtime enqueues a `PromiseYieldTimeout` entry with:

```
expires_at = apply_state.block_height + config.wasm_config.limit_config.yield_timeout_length_in_blocks
``` [1](#0-0) 

Entries are appended to the back of the queue in FIFO order. The timeout resolution loop in `resolve_promise_yield_timeouts` reads from the front and breaks as soon as it finds an entry that has not yet expired:

```rust
// Queue entries are ordered by expires_at
if queue_entry.expires_at > apply_state.block_height {
    break;
}
``` [2](#0-1) 

The comment `// Queue entries are ordered by expires_at` is the explicit invariant. This invariant holds only when `yield_timeout_length_in_blocks` is constant or increases. If a protocol upgrade **decreases** `yield_timeout_length_in_blocks`, a yield created after the upgrade will have a smaller `expires_at` than yields created before the upgrade that are still in the queue. The queue is no longer sorted, and the loop will stop at the first old entry, skipping all newer entries with earlier deadlines.

### Impact Explanation

When `yield_timeout_length_in_blocks` is reduced via a protocol upgrade and yields are already queued with the old (larger) timeout:

1. New yields created after the upgrade have `expires_at = current_height + new_shorter_timeout`.
2. Old yields in the queue have `expires_at = old_height + old_longer_timeout`, which is larger.
3. The queue is now **out of order**: old entries at the front have larger `expires_at` than new entries behind them.
4. The loop breaks at the first old entry, never reaching the new entries.
5. New yields whose timeout has passed are **never resolved by the timeout mechanism** until all older entries ahead of them in the queue expire.

The concrete corrupted protocol value is the **execution outcome** of the `PromiseYield` receipt: the timeout callback is never triggered, the `PromiseYield` receipt remains stuck in state, and the contract's callback is permanently delayed. This is a **runtime state mismatch** — the protocol guarantees that a yield will be resolved (either by resume or timeout) within `yield_timeout_length_in_blocks` blocks, but this guarantee is violated.

The current value of `yield_timeout_length_in_blocks` is 200 blocks across all protocol versions in the config store. [6](#0-5) 

### Likelihood Explanation

The trigger requires a protocol upgrade that decreases `yield_timeout_length_in_blocks`. This is a governance-level change (protocol version bump voted by validators), not an unprivileged user action. However, the **attacker-controlled entry path** is: any unprivileged user can call `promise_yield_create` on a deployed contract to enqueue a `PromiseYieldTimeout` entry. If a protocol upgrade then reduces `yield_timeout_length_in_blocks`, the user's own yield (or any yield created after the upgrade) will be stuck. The user does not need any special privilege to create the yield entries that become stuck.

Given that `yield_timeout_length_in_blocks` has been 200 blocks since introduction and no reduction has occurred, the likelihood of a reduction is low but non-zero — protocol upgrades routinely adjust runtime parameters, and a future upgrade could reduce this value. The impact when it occurs is a permanent DoS of yield timeout resolution for all yields created after the upgrade until older yields drain from the queue.

### Recommendation

Remove the `break` on the first non-expired entry and instead **continue** past it, processing all expired entries regardless of their position in the queue. Alternatively, maintain the queue sorted by `expires_at` at insertion time (insertion sort or a priority queue), or scan the entire queue each block. The simplest fix matching the Andromeda recommendation pattern:

```rust
// Instead of:
if queue_entry.expires_at > apply_state.block_height {
    break;
}
// Use:
if queue_entry.expires_at > apply_state.block_height {
    promise_yield_indices.first_index += 1;
    continue;
}
```

A cleaner fix is to sort entries by `expires_at` at enqueue time so the invariant is always maintained.

### Proof of Concept

1. Protocol is at version N with `yield_timeout_length_in_blocks = 200`.
2. At block height 1000, contract A calls `promise_yield_create`. Entry enqueued: `expires_at = 1200`.
3. Protocol upgrades to version N+1 with `yield_timeout_length_in_blocks = 50`.
4. At block height 1050, contract B calls `promise_yield_create`. Entry enqueued: `expires_at = 1100`.
5. Queue state (front to back): `[{A, expires_at=1200}, {B, expires_at=1100}]`.
6. At block height 1100, `resolve_promise_yield_timeouts` runs:
   - Reads front entry: `{A, expires_at=1200}`. Since `1200 > 1100`, **breaks**.
   - Entry B (`expires_at=1100`) is never reached.
7. Contract B's yield callback is never triggered at block 1100 as the protocol guarantees.
8. B's yield remains stuck until block 1200 when A's entry finally expires and drains from the front, allowing B to be processed.
