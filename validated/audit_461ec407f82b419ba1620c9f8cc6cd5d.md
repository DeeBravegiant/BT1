### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Allows Adversarial Queue Saturation, Blocking Legitimate Bridge Verification Requests - (File: `crates/contract/src/dto_mapping.rs`)

### Summary
The `verify_foreign_transaction` endpoint stores pending requests under a caller-agnostic key (`VerifyForeignTransactionRequest` = `{request, domain_id, payload_version}`), while `sign()` uses a caller-specific key (tweak derived from `predecessor_id + path`). Because the fan-out queue cap (`MAX_PENDING_REQUEST_FAN_OUT = 128`) is shared across all callers for the same foreign transaction, an adversary can saturate the queue with 128 cheap submissions from distinct accounts, causing every subsequent legitimate caller to receive `PendingRequestQueueFull` and be unable to submit the same verification request.

### Finding Description
`sign()` constructs its storage key by baking the caller's account ID into the tweak:

```rust
// crates/near-mpc-crypto-types/src/sign.rs:118-125
impl SignatureRequest {
    pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
        let tweak = crate::kdf::derive_tweak(predecessor_id, path);
        SignatureRequest { domain_id: domain, tweak, payload }
    }
}
```

Each caller therefore gets a distinct map entry and a private queue slot budget of 128.

`verify_foreign_transaction()` converts its args to a storage key that contains **no caller identity**:

```rust
// crates/contract/src/dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

All callers submitting the same `(domain_id, request, payload_version)` tuple share one queue entry. The shared queue is capped at 128:

```rust
// crates/contract/src/pending_requests.rs:51-58
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull {
            limit: MAX_PENDING_REQUEST_FAN_OUT,
        }
        .to_string(),
    );
}
```

An adversary who submits 128 `verify_foreign_transaction` calls for the same foreign-chain transaction (from 128 distinct NEAR accounts, each paying only 1 yoctoNEAR) fills the queue to capacity. Every subsequent legitimate caller receives `PendingRequestQueueFull` and cannot enqueue their request until MPC nodes respond and drain the queue.

The contract's own test acknowledges the caller-agnostic sharing:

```rust
// crates/contract/src/lib.rs:3242-3243
// And: caller bob submits the identical request — a different account would today
// be blocked from receiving a response by alice's submission.
```

### Impact Explanation
In the Omnibridge inbound flow (foreign chain → NEAR), a bridge service calls `verify_foreign_transaction` to attest that a specific foreign-chain deposit finalized. If an adversary saturates the queue for that specific `tx_id` before the bridge service can submit, the bridge service's call is rejected. If the bridge enforces a claim window (common in cross-chain bridges to prevent replay), the user's foreign-chain funds may be permanently unrecoverable. This is a request-lifecycle manipulation that breaks the production safety invariant that any caller with a valid deposit and gas should be able to enqueue a verification request.

### Likelihood Explanation
The attack requires only 128 NEAR transactions, each with 1 yoctoNEAR deposit and ~10 TGas. The total cost is negligible. The attacker needs to know the target `tx_id` (publicly visible on the foreign chain) and front-run the bridge service's submission. Front-running is straightforward on NEAR because mempool transactions are observable. The attack is repeatable: after MPC nodes drain the queue, the attacker can re-saturate it before the bridge service retries.

### Recommendation
Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` storage key, mirroring the `sign()` design. This gives each caller a private queue slot budget and eliminates cross-caller queue contention. Alternatively, enforce a per-account submission limit for `verify_foreign_transaction` requests targeting the same foreign transaction.

### Proof of Concept
1. Attacker observes a target Bitcoin `tx_id` on-chain that a bridge service will soon verify.
2. Attacker submits 128 `verify_foreign_transaction` calls with `{request: BitcoinRpcRequest{tx_id: TARGET, ...}, domain_id: X, payload_version: V1}` from 128 distinct NEAR accounts.
3. `push_pending_yield` is called 128 times; the queue for that request key reaches `MAX_PENDING_REQUEST_FAN_OUT`.
4. Bridge service submits the same request; `push_pending_yield` panics with `PendingRequestQueueFull`.
5. Bridge service cannot enqueue the verification; if the claim window expires before MPC nodes respond and the bridge service retries successfully, the user's bridged funds are lost.
