Audit Report

## Title
Division-by-Zero in `viewSwapAgETHAmountAndFee` Blocks All Deposits Before First Cross-Chain Rate Message - (File: contracts/agETH/AGETHPoolV3.sol)

## Summary
`CrossChainRateReceiver.rate` defaults to `0` and is only populated by a successful `lzReceive` call. Before the first LayerZero message arrives, every call to `AGETHPoolV3.deposit()` (both ETH and token variants) reverts with a Solidity 0.8.x division-by-zero panic (`Panic(0x12)`), making the pool non-functional during the deployment window. The impact is **Low** — deposits revert and all ETH/tokens are returned to callers; no funds are permanently lost or frozen in the contract.

## Finding Description
`CrossChainRateReceiver` declares `rate` as a plain `uint256` with no initializer, so it defaults to `0`: [1](#0-0) 

`getRate()` returns this value unconditionally with no zero-guard: [2](#0-1) 

`rate` is only ever written inside `lzReceive`, which requires a valid LayerZero message from the trusted source chain: [3](#0-2) 

`AGETHRateReceiver`'s constructor sets configuration fields but never initializes `rate`: [4](#0-3) 

Both `viewSwapAgETHAmountAndFee` overloads divide by `agETHToETHrate` without a zero-check: [5](#0-4) [6](#0-5) 

`addSupportedToken` already applies a zero-rate guard for token oracles, demonstrating the pattern is known but was not applied to `agETHOracle`: [7](#0-6) 

No equivalent guard exists in `initialize()` or in either `deposit()` overload for the `agETHOracle` rate. [8](#0-7) 

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

All ETH and token `deposit()` calls revert during the window between contract deployment and the first successful `lzReceive`. Because Solidity 0.8.x reverts the entire transaction on `Panic(0x12)`, no ETH or tokens are retained by the contract — callers receive their assets back. The service is temporarily unavailable, but no funds are frozen inside the contract and no value is destroyed. This does not meet the "Temporary freezing of funds" (Medium) threshold, which requires funds already held in the contract to be inaccessible.

## Likelihood Explanation
The condition is inherent to the deployment lifecycle and requires no attacker action. Any user who calls `deposit()` between contract deployment and the first `lzReceive` message will trigger the revert. The window duration depends on LayerZero bridge latency and operator scheduling, ranging from seconds to hours. `isEthDepositEnabled` is set to `true` in `initialize()`, so ETH deposits are live immediately. [9](#0-8) 

## Recommendation
Add a zero-rate guard in `viewSwapAgETHAmountAndFee` (mirroring the existing guard in `addSupportedToken`):

```solidity
uint256 agETHToETHrate = getRate();
if (agETHToETHrate == 0) revert OracleRateNotInitialized();
```

Alternatively, initialize `rate` to `1e18` in the `AGETHRateReceiver` constructor as a safe sentinel, or require a privileged setter to push a valid rate before enabling deposits.

## Proof of Concept

```solidity
// 1. Deploy AGETHRateReceiver(srcChainId, rateProvider, lzEndpoint)
//    → rate == 0 (default uint256)

// 2. Deploy AGETHPoolV3, initialize with agETHOracle = address(AGETHRateReceiver)
//    → isEthDepositEnabled = true immediately

// 3. Call AGETHPoolV3.deposit{value: 1 ether}("")
//    → viewSwapAgETHAmountAndFee(1 ether)
//    → agETHToETHrate = IOracle(agETHOracle).getRate() == 0
//    → agETHAmount = amountAfterFee * 1e18 / 0  ← Panic(0x12), revert
//    → ETH returned to caller (no freeze, no loss)

// 4. Simulate lzReceive: call from lzEndpoint with rate = 1e18
//    → rate is now 1e18

// 5. Call AGETHPoolV3.deposit{value: 1 ether}("") again → succeeds
```

Reproducible on a local Hardhat/Foundry fork with no admin compromise required — only the uninitialized `rate` state is needed.
