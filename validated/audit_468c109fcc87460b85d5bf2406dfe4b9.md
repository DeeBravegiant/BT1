Audit Report

## Title
Stale Chainlink Price Accepted in `getAssetPrice` Enables rsETH Over-Minting via Cross-Asset Price Skew - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt` and `answeredInRound`, accepting any stale price without a heartbeat or round-completeness check. Because `updateRSETHPrice()` is public and unrestricted, an unprivileged attacker can call it during a Chainlink staleness event to write a deflated `rsETHPrice` to storage, then immediately deposit a fresh-feed asset to receive more rsETH than the deposited value warrants, diluting all existing rsETH holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` binds only the second return value of `latestRoundData()`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Neither `updatedAt` nor `answeredInRound` is read, so there is no heartbeat check and no round-completeness check. This price flows into `LRTOracle._getTotalEthInProtocol()`:

```solidity
// contracts/LRTOracle.sol L339-343
uint256 assetER = getAssetPrice(asset);
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```

That total is used by `_updateRsETHPrice()` to compute and store `rsETHPrice`. The entry point is public and unrestricted:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The stored `rsETHPrice` is then the denominator in the minting formula:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Exploit path:**
1. A Chainlink LST feed (e.g. stETH/ETH) goes stale, reporting a price slightly below the real value.
2. Attacker calls `updateRSETHPrice()`. `_getTotalEthInProtocol()` uses the stale-low stETH price, deflating `totalETHInProtocol` and therefore `rsETHPrice`.
3. Attacker calls `depositAsset(cbETH, ...)`. The numerator uses the live cbETH price; the denominator is the deflated `rsETHPrice`. The attacker receives more rsETH than the deposited value warrants.
4. When the feed recovers and `updateRSETHPrice()` is called again, `rsETHPrice` rises back to fair value. The attacker's excess rsETH is now worth more ETH than was deposited, at the expense of all prior holders.

**Why existing guards are insufficient:**

`_updateRsETHPrice()` contains a `pricePercentageLimit` circuit-breaker: if `newRsETHPrice` drops more than `pricePercentageLimit` below `highestRsethPrice`, the protocol pauses. However:
- If `pricePercentageLimit == 0` (the default after initialization), the check is explicitly skipped (`pricePercentageLimit > 0 &&` short-circuits).
- Even when set, the guard only catches deviations *above* the configured threshold. Any staleness event causing a price drop *within* the limit passes through silently, still allowing profitable over-minting proportional to the deviation.
- The guard compares against `highestRsethPrice` (all-time high), not the previous price, so a feed that has been stale for a long time can cause a large absolute deviation while still being within the percentage limit if the all-time high is much higher.

By contrast, `ChainlinkOracleForRSETHPoolCollateral` in the same repository already performs staleness checks:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

This confirms the team is aware of the pattern but did not apply it to `ChainlinkPriceOracle`.

## Impact Explanation

Existing rsETH holders suffer dilution — their proportional claim on the underlying ETH pool shrinks without compensation. This is direct, at-rest fund loss for every holder at the time of the exploit, matching the **Critical** impact class: *direct theft of any user funds, whether at-rest or in-motion*. The magnitude scales with (a) the stale asset's TVL share and (b) the degree of price deviation before the feed recovers. The attack is repeatable across every staleness event on any supported LST feed.

## Likelihood Explanation

Chainlink feeds have historically gone stale during periods of network congestion or node operator issues. The protocol supports multiple LST assets, each with its own feed; the probability that at least one feed lags during a high-congestion period is non-trivial. The attacker requires no special role, no front-running, and no oracle manipulation — only the ability to observe on-chain that one feed's `updatedAt` is old and to call two public functions (`updateRSETHPrice` then `depositAsset`). **Likelihood: Low-to-Medium.**

## Recommendation

1. **Add staleness and round-completeness checks in `ChainlinkPriceOracle.getAssetPrice()`**, mirroring the pattern already present in `ChainlinkOracleForRSETHPoolCollateral`, and additionally enforce a configurable heartbeat window:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(answeredInRound >= roundId,  "Stale price: incomplete round");
require(updatedAt != 0,              "Stale price: round not complete");
require(block.timestamp - updatedAt <= maxStaleness[asset], "Stale price: heartbeat exceeded");
require(price > 0,                   "Invalid price");
```

2. **Store a per-feed `maxStaleness` value** (e.g. 3600 s for a 1-hour heartbeat feed) set by the LRT manager alongside `updatePriceFeedFor`.

3. **Apply the same fix to `RSETHPriceFeed.latestRoundData()`**, which forwards the raw ETH/USD answer without staleness validation.

## Proof of Concept

**Setup:** Protocol holds 1 000 stETH (stETH/ETH feed = 0.999 stale, real = 1.001) and 1 000 cbETH (cbETH/ETH feed = 1.05, live). rsETH supply = 2 049 rsETH. Real `rsETHPrice` ≈ 1.001 ETH. `pricePercentageLimit` = 0 (default) or set to 1%.

**Step 1 — Feed goes stale.** stETH/ETH Chainlink feed stops updating. Last reported price = 0.999; real price = 1.001.

**Step 2 — Attacker calls `updateRSETHPrice()` (public, no role required).**
- `_getTotalEthInProtocol()` computes: `1000 × 0.999 + 1000 × 1.05 = 2049 ETH` (stale stETH price used).
- Real total: `1000 × 1.001 + 1000 × 1.05 = 2051 ETH`.
- `rsETHPrice` written as `2049 / 2049 = 1.000 ETH` instead of real `≈ 1.001 ETH`.
- Price drop is ~0.1%, within a 1% `pricePercentageLimit` → no pause triggered.

**Step 3 — Attacker calls `depositAsset(cbETH, 100e18, ...)`.**
- `rsethAmountToMint = (100 × 1.05) / 1.000 = 105 rsETH`
- Correct amount: `(100 × 1.05) / 1.001 ≈ 104.895 rsETH`
- Attacker receives ~0.105 rsETH excess per 100 cbETH deposited.

**Step 4 — Feed recovers.** `updateRSETHPrice()` is called with correct prices. `rsETHPrice` rises back to ~1.001. The attacker's 105 rsETH is now worth more ETH than deposited, at the expense of all prior holders.
