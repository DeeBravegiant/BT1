Audit Report

## Title
Unprivileged ETH Donation via Open `receive()` Inflates `address(this).balance` and Temporarily Freezes ETH Deposits - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool` exposes an unrestricted `receive()` fallback that allows any caller to donate ETH directly to the contract. Because `getETHDistributionData()` reports `address(this).balance` as the canonical ETH held in the pool, donated ETH is indistinguishable from legitimately deposited ETH. An attacker who donates just enough ETH to push `address(this).balance` above `depositLimitByAsset(ETH)` causes every subsequent `depositETH()` call to revert with `MaximumDepositLimitReached`, temporarily freezing all ETH deposits until an operator manually drains the pool to a NodeDelegator.

## Finding Description
`LRTDepositPool` has an open, unrestricted receive fallback at line 58:

```solidity
receive() external payable { }
```

`getETHDistributionData()` at line 480 sets the pool's ETH holding to the raw contract balance:

```solidity
ethLyingInDepositPool = address(this).balance;
```

`getTotalAssetDeposits(ETH)` at lines 385–397 aggregates `assetLyingInDepositPool` (plus NDC, EigenLayer, converter, and unstaking vault balances) into the total ETH figure used for limit enforcement.

`_checkIfDepositAmountExceedesCurrentLimit` at lines 676–682 enforces the cap for ETH as:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
```

Because `depositETH` is `payable`, `msg.value` is already reflected in `address(this).balance` before `_beforeDeposit` runs, so the check intentionally omits adding `amount` again. This same design means ETH sent directly via `receive()` is silently counted in `totalAssetDeposits` with no way to distinguish it from protocol deposits. If donated ETH pushes `address(this).balance` to or above the deposit limit, the `>` comparison returns `true` and `_beforeDeposit` reverts with `MaximumDepositLimitReached` for every subsequent `depositETH()` call.

## Impact Explanation
**Medium — Temporary freezing of funds.** All ETH deposits are blocked until an operator calls `transferETHToNodeDelegator`, reducing `address(this).balance` below the limit. No user can deposit ETH during this window, and the attack can be repeated each time the operator drains the pool. Funds already in the protocol are not at risk of theft, but the deposit mechanism is rendered non-functional for an indefinite period controlled by the attacker's repeat donations.

## Likelihood Explanation
The attack requires only a plain ETH transfer, executable by any EOA or contract with no special role. The cost equals the donated ETH, which is not recoverable by the attacker but remains in the protocol. If the deposit limit is nearly full, a 1 wei donation suffices. The attack is repeatable: each time the operator drains the pool to a NodeDelegator, the attacker can re-donate to re-trigger the freeze.

## Recommendation
Replace `address(this).balance` in `getETHDistributionData()` with an internal ledger variable (e.g., `uint256 private _ethInPool`) that is incremented only through the authorised deposit paths (`depositETH`, `receiveFromRewardReceiver`, `receiveFromLRTConverter`, `receiveFromNodeDelegator`) and decremented when ETH is transferred out. Alternatively, restrict `receive()` to known callers (reward receiver, converter, node delegators) and revert on unexpected senders.

## Proof of Concept
1. Protocol is live; ETH deposit limit is 1,000 ETH; current `address(this).balance` is 999 ETH.
2. Eve sends 1 ETH directly to `LRTDepositPool` via a plain transfer (hits `receive()`). `address(this).balance` becomes 1,000 ETH.
3. Alice calls `depositETH{value: 1 ether}(minRSETH, "")`.
   - `msg.value` is received → `address(this).balance = 1,001 ETH`.
   - `getETHDistributionData()` returns `ethLyingInDepositPool = 1,001 ETH`.
   - `getTotalAssetDeposits(ETH)` returns 1,001 ETH.
   - `_checkIfDepositAmountExceedesCurrentLimit`: `1001 > 1000` → `true`.
   - Transaction reverts: `MaximumDepositLimitReached`.
4. All ETH deposits are frozen until an operator calls `transferETHToNodeDelegator`, reducing `address(this).balance` below 1,000 ETH.
5. Eve repeats step 2 to re-freeze deposits.

**Foundry test plan:** Deploy `LRTDepositPool` on a fork, set deposit limit to `X ETH`, seed the pool to `X - 1 ETH` via `depositETH`, call `address(pool).transfer(1 ether)` from an unprivileged address, then assert that a subsequent `depositETH{value: 1 ether}` reverts with `MaximumDepositLimitReached`.
