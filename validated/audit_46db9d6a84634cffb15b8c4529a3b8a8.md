### Title
`respond_verify_foreign_tx` Accepts Attacker-Supplied `payload_hash` Without Binding It to the Pending `request` - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies the MPC signature against `response.payload_hash` — a value supplied entirely by the responding node — without checking that this hash was actually derived from the pending `request`. A single Byzantine leader node can replay a valid `(payload_hash, signature)` pair produced for a *different* foreign transaction and have the contract accept it as a legitimate response to the current request, delivering a forged attestation to the waiting caller.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs the following signature check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← attacker-controlled

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← wrong variable: should be derived from `request`
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The correct value to verify against is a hash computed deterministically from the pending `request` and the observed foreign-chain values — i.e., `SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <observed> }))`. The protocol defines this canonical construction explicitly: [2](#0-1) 

`ForeignTxSignPayloadV1` binds the `ForeignChainRpcRequest` (including `tx_id`) together with the extracted values: [3](#0-2) 

The node-side code correctly computes `payload_hash` from the actual payload before signing: [4](#0-3) 

But the contract never recomputes or constrains `response.payload_hash` — it accepts whatever the responding node provides. Because the contract does not have access to the `extracted_values` at response time, it cannot independently reconstruct the hash. This leaves the binding between `request` and `payload_hash` entirely unenforced on-chain.

The analog to the external report is direct: the external report uses `position_amount` (the wrong, larger variable) instead of `_reduce_by_amount` (the correct variable) in a minimum-amount check, causing the check to always fail. Here, `response.payload_hash` (the wrong, attacker-controlled variable) is used instead of a hash derived from `request` (the correct, deterministic variable) in the signature-binding check, causing the check to be bypassable.

### Impact Explanation

A malicious leader node replays a valid `(payload_hash_A, signature_A)` pair — produced by the MPC network for a *previous* transaction A — as the response to a pending request for transaction B. The contract's only check is `verify_ecdsa_signature(signature_A, payload_hash_A, root_pk)`, which passes because the signature is genuinely valid over `payload_hash_A`. The contract then resolves the yield for request B with `{payload_hash: payload_hash_A, signature: signature_A}`.

The caller (e.g., an Omnibridge contract) receives a `VerifyForeignTransactionResponse` bearing a valid MPC root-key signature, but the `payload_hash` encodes transaction A's `tx_id` and extracted values, not transaction B's. Any bridge logic that does not independently recompute the expected hash from the original request will treat this as a valid attestation of transaction B, enabling:

- **Cross-request replay / double-spend**: the same foreign-chain event (transaction A) can be used to authorize multiple distinct bridge operations by replaying its `(payload_hash, signature)` against different pending requests.
- **Forged foreign-chain verification**: the contract attests that transaction B was verified when it was not, breaking the core security guarantee of the `verify_foreign_transaction` flow.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

The attacker is a single Byzantine MPC node operating below the signing threshold. No threshold collusion is required. The node participates in normal signing operations and accumulates valid `(payload_hash, signature)` pairs as a natural byproduct of the protocol. It then calls `respond_verify_foreign_tx` — a function any attested participant may call — with a recycled pair targeting any pending request on the same domain. The entry path is fully reachable from an unprivileged (but attested) participant account.

### Recommendation

The contract must enforce that `response.payload_hash` is bound to the pending `request`. Since the contract cannot recompute the full hash (it lacks `extracted_values`), the binding must be established at signing time. Two complementary mitigations:

1. **Include `request` in the signed payload's domain tag**: ensure `ForeignTxSignPayloadV1` encodes the full `VerifyForeignTransactionRequest` (including `domain_id` and `payload_version`) so that a hash produced for request A is cryptographically incompatible with request B even if both share the same chain type.

2. **Bind `payload_hash` to `request` in the contract**: store a commitment to the expected `payload_hash` prefix (the portion derivable from `request` alone, i.e., the Borsh encoding of `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: [] })` as a lower-bound prefix check) alongside the pending yield, and reject responses whose `payload_hash` does not start from the correct request encoding. Alternatively, require the responding node to also submit the `extracted_values` so the contract can recompute and verify the hash in full.

### Proof of Concept

```
1. MPC network processes request_A = {domain_id: D, request: Bitcoin{tx_id: [0xAA;32], ...}}
   → produces payload_hash_A = SHA-256(borsh(ForeignTxSignPayloadV1{request: Bitcoin{tx_id:[0xAA;32],...}, values:[BlockHash([0x11;32])]}))
   → produces signature_A = ECDSA_sign(root_sk, payload_hash_A)

2. User submits request_B = {domain_id: D, request: Bitcoin{tx_id: [0xBB;32], ...}}
   → contract stores pending yield under key request_B

3. Byzantine leader node calls:
   respond_verify_foreign_tx(
     request  = request_B,                                    // correct pending key
     response = { payload_hash: payload_hash_A,              // ← replayed from tx A
                  signature:    signature_A }                 // ← valid over payload_hash_A
   )

4. Contract check: verify_ecdsa_signature(signature_A, payload_hash_A, root_pk) → OK
   (signature is genuinely valid; contract never checks payload_hash_A encodes tx_id=[0xAA;32])

5. Contract resolves yield for request_B with {payload_hash_A, signature_A}

6. Caller receives VerifyForeignTransactionResponse attesting tx_id=[0xBB;32] was verified,
   but payload_hash_A encodes tx_id=[0xAA;32] — forged attestation delivered.
``` [5](#0-4)
