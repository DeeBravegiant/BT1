Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Withdrawal Initiations — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`maxNumberOfWithdrawalsPerUser` is declared as a storage variable but never assigned in `initialize()`, leaving it at its default value of `0`. The guard in `initiateWithdrawal()` compares `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, which evaluates to `0 >= 0 → true` for every caller, causing every withdrawal attempt to revert with `WithdrawalLimitReached`. All staked KERNEL tokens are frozen until an admin separately calls `setMaxNumberOfWithdrawalsPerUser()`.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared at L108 but never assigned in `initialize()` (L259–271): [1](#0-0) [2](#0-1) 

The guard in `initiateWithdrawal()` at L323 is:
```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
``` [3](#0-2) 

Because `uint256` array `.length` is always `>= 0`, and `maxNumberOfWithdrawalsPerUser == 0` post-deployment, this condition is unconditionally `true` for every caller. The setter at L614–616 explicitly rejects `0` as an invalid value, confirming the default state is operationally broken: [4](#0-3) 

`stake()` and `stakeFor()` accept deposits without restriction, so users can lock tokens into the contract but cannot retrieve them via `initiateWithdrawal()` until the admin calls `setMaxNumberOfWithdrawalsPerUser()`.

## Impact Explanation
Every KERNEL staker's principal is frozen in the contract from the moment of deployment until an admin manually calls `setMaxNumberOfWithdrawalsPerUser()`. This is a concrete **temporary freezing of funds** affecting all users — matching the "Medium. Temporary freezing of funds" allowed impact tier. The freeze is not hypothetical: it is the default on-chain state of every fresh deployment.

## Likelihood Explanation
The broken state is the default state of every deployment. No attacker action is required — any user who stakes and then calls `initiateWithdrawal()` before the admin's separate configuration call will have their funds frozen. The exposure window is the entire period between contract deployment and the admin's remediation call, which has no on-chain enforcement or deadline.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... existing setup ...
    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
}
``` [2](#0-1) 

## Proof of Concept
1. Deploy and call `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` is `0`.
2. Alice calls `stake(100e18)`. `balanceOf[Alice] = 100e18`. Succeeds.
3. Alice calls `initiateWithdrawal(100e18)`.
4. Check: `userWithdrawalIds[Alice].length >= maxNumberOfWithdrawalsPerUser` → `0 >= 0` → `true`.
5. Transaction reverts with `WithdrawalLimitReached`.
6. Alice's 100e18 KERNEL remain locked. No recovery path exists for Alice until admin calls `setMaxNumberOfWithdrawalsPerUser(n)` for `n > 0`.

Foundry test sketch:
```solidity
function test_withdrawalBlockedOnFreshDeploy() public {
    kernelDepositPool.initialize(admin, address(kernelToken), address(rewardToken));
    kernelToken.mint(alice, 100e18);
    vm.prank(alice);
    kernelToken.approve(address(kernelDepositPool), 100e18);
    vm.prank(alice);
    kernelDepositPool.stake(100e18);
    vm.prank(alice);
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    kernelDepositPool.initiateWithdrawal(100e18);
}
```
