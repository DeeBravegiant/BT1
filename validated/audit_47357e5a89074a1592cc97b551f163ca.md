Audit Report

## Title
ETH Deposit Limit Check Missing Incoming Amount in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric check: for ERC20 assets it correctly tests `totalAssetDeposits + amount > limit`, but for ETH it only tests `totalAssetDeposits > limit`, omitting the incoming deposit amount. This renders the ETH deposit cap functionally unenforceable whenever the current total is at or below the limit, allowing any depositor to push ETH holdings arbitrarily above the configured cap in a single transaction.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676ŌĆō682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ŌåÉ amount never added
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

The ETH branch at line 679 evaluates only whether the **already-deposited** total exceeds the limit; the incoming `amount` is never included. The ERC20 branch at line 681 correctly sums both. As a result, whenever `totalAssetDeposits Ōēż depositLimitByAsset(ETH_TOKEN)`, the function unconditionally returns `false` for any ETH deposit, no matter how large `amount` is.

The call path is fully public and unprivileged:

1. `depositETH()` (lines 76ŌĆō93) ŌĆö `external payable`, no role restriction beyond `whenNotPaused` and `onlySupportedAsset`. [2](#0-1) 

2. `_beforeDeposit()` (lines 648ŌĆō670) ŌĆö calls `_checkIfDepositAmountExceedesCurrentLimit`; reverts with `MaximumDepositLimitReached` only if it returns `true`. [3](#0-2) 

Because the ETH branch never returns `true` while `totalAssetDeposits Ōēż limit`, the revert is never triggered for ETH deposits under that condition, regardless of deposit size.

## Impact Explanation
The ETH deposit limit is a protocol safety mechanism controlling how much ETH is accepted and subsequently deployed into EigenLayer strategies. With the check broken, rsETH can be minted beyond the intended cap, and ETH can accumulate idle in the deposit pool if EigenLayer capacity is the reason for the cap, reducing yield for all rsETH holders. The protocol's accounting invariant (`totalAssetDeposits Ōēż depositLimitByAsset`) is violated. No deposited ETH is directly lost or stolen. This matches **Low ŌĆö contract fails to deliver promised returns, but doesn't lose value**. [4](#0-3) 

## Likelihood Explanation
Any unprivileged ETH depositor can trigger this in a single call to `depositETH()` whenever `totalAssetDeposits Ōēż depositLimitByAsset(ETH_TOKEN)`. No special role, governance access, front-running, or multi-step setup is required. The condition `totalAssetDeposits Ōēż limit` is the normal operating state of the protocol. **Likelihood: High.** [2](#0-1) 

## Recommendation
Include the incoming deposit amount in the ETH branch, consistent with the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000e18`.
2. Current state: `getTotalAssetDeposits(ETH_TOKEN)` returns `999e18`.
3. Attacker calls `depositETH{value: 500e18}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500e18)` evaluates `999e18 > 1000e18` ŌåÆ `false` ŌåÆ deposit proceeds, minting rsETH for 500 ETH.
5. Protocol now holds 1499 ETH worth of rsETH minted, 499 ETH above the intended cap.
6. For comparison, an ERC20 depositor in the same state with `amount = 500e18` is correctly rejected: `999e18 + 500e18 > 1000e18` ŌåÆ `true` ŌåÆ `MaximumDepositLimitReached`.

Foundry test plan: deploy `LRTDepositPool` with a mock config returning `depositLimitByAsset = 1000e18`; seed `totalAssetDeposits` to `999e18`; call `depositETH{value: 500e18}`; assert the call succeeds and `getTotalAssetDeposits(ETH_TOKEN)` returns `1499e18`, exceeding the limit. [1](#0-0)
