### Title
Missing x_share-to-Q Binding Check in `blob_to_keyset` Enables Root Public Key Substitution — (`src/cbmpc/api/hd_keyset_eddsa_2p.cpp`)

### Summary

`blob_to_keyset` in `src/cbmpc/api/hd_keyset_eddsa_2p.cpp` accepts a keyset blob without verifying any binding between the stored `x_share` scalar and `root_Q_compressed`. An attacker supplying a crafted blob can inject an arbitrary valid curve point as the root public key Q. This Q is accepted verbatim and propagated through `extract_root_public_key_compressed`, `refresh`, and `derive_eddsa_2p_keys`, all without detection.

### Finding Description

`blob_to_keyset` performs the following checks on deserialization: [1](#0-0) 

It validates role, curve, and that `x_share`/`k_share` are in `[0, q)`, and that `root_Q_compressed` decodes to a valid curve point. It does **not** verify that `x_share * G` is consistent with `root_Q_compressed` in any way.

By contrast, `eddsa_mp.cpp`'s `deserialize_key_blob` explicitly enforces this binding at line 158: [2](#0-1) 

The multi-party blob stores per-party `Qis_compressed` (individual public shares), enabling the check `x_share * G == Qi_self`. The 2-party HD keyset blob (`keyset_blob_v1_t`) stores only the aggregate Q and one party's scalar share — it contains no `Qi_self` field — so no equivalent binding check is performed: [3](#0-2) 

After deserialization, `refresh` copies `key.root.Q` verbatim into the new keyset without re-deriving or re-checking it: [4](#0-3) 

And `extract_root_public_key_compressed` returns whatever Q was stored in the blob: [5](#0-4) 

Similarly, `derive_eddsa_2p_keys` uses `key.root.Q` directly to compute `Q_derived`: [6](#0-5) 

All derived eddsa2pc key blobs inherit the attacker-controlled Q value.

### Impact Explanation

An attacker who can supply a crafted keyset blob (malicious serialized-input provider, compromised blob storage, or Byzantine party controlling their own blob) can:

1. **Key substitution via `extract_root_public_key_compressed`**: The API returns an attacker-chosen curve point as the root public key. Any application logic that uses this value to identify the key, derive a wallet address, or register a public key will operate on the wrong key.

2. **Derived key corruption via `derive_eddsa_2p_keys`**: All derived eddsa2pc key blobs will carry a Q value derived from the attacker-controlled root Q. Signatures produced with these keys will fail verification against the substituted Q, or the application will associate signatures with the wrong public identity.

3. **Refresh propagation**: After `refresh`, the new keyset blob carries the attacker-controlled Q verbatim (line 61 of `hd_keyset_eddsa_2p.cpp` protocol file), while x_shares are updated. The new keyset is internally inconsistent: `(x_share_p1 + r_x)*G + (x_share_p2 - r_x)*G = real_Q ≠ attacker_Q`.

The attacker cannot produce valid signatures under the substituted Q without knowing the honest peer's x_share, so outright signature forgery is not directly enabled. However, the accepted-bad-output (wrong Q returned from a public API) constitutes a key substitution vulnerability per the High impact criteria.

### Likelihood Explanation

Any caller who can supply a keyset blob to `extract_root_public_key_compressed`, `refresh`, or `derive_eddsa_2p_keys` can trigger this. The blob is described as an opaque persisted byte string, making it a natural target for tampering. The attack requires no interaction with the peer and no cryptographic computation — just crafting a valid-format blob with a chosen Q.

### Recommendation

Extend `keyset_blob_v1_t` to include a `Qi_self_compressed` field (the serialized `x_share * G` for this party). In `blob_to_keyset`, after parsing `x_share`, compute `x_share * G` and verify it equals the stored `Qi_self_compressed`. This mirrors the binding check already present in `eddsa_mp.cpp`'s `deserialize_key_blob` at line 158. The aggregate Q cannot be checked against x_share alone in the 2-party case (since Q = x_share_p1*G + x_share_p2*G and the peer's share is absent), but binding x_share to its own public share Qi_self is sufficient to prevent substitution of an arbitrary Q.

### Proof of Concept

```
// Attacker-crafted blob:
//   version = 1
//   role    = 0  (p1)
//   curve   = ed25519
//   root_Q_compressed  = compressed encoding of 2*G   ← attacker-chosen
//   root_K_compressed  = compressed encoding of 2*G   ← attacker-chosen
//   x_share = 1   (valid scalar, in [0, q))
//   k_share = 1   (valid scalar, in [0, q))

buf_t crafted_blob = build_keyset_blob(/*x_share=*/1, /*k_share=*/1,
                                        /*Q=*/2*G, /*K=*/2*G);

assert(rv == SUCCESS);   // passes — no x_share*G == Q check

assert(rv == SUCCESS);
assert(out_Q == compressed(2*G));   // returns attacker-chosen Q
assert(out_Q != compressed(1*G));   // ≠ x_share * G
```

`blob_to_keyset` returns `SUCCESS` because all format checks pass (valid role, valid curve, `x_share=1` is in `[0,q)`, `2*G` is a valid curve point). `extract_root_public_key_compressed` then returns `2*G` — an attacker-controlled value that does not correspond to the stored `x_share`.

static error_t blob_to_keyset(const keyset_blob_v1_t& blob, coinbase::mpc::key_share_eddsa_hdmpc_2p_t& keyset) {
  if (blob.role > 1) return coinbase::error(E_FORMAT, "invalid keyset blob role");
  if (static_cast<curve_id>(blob.curve) != curve_id::ed25519)
    return coinbase::error(E_FORMAT, "invalid keyset blob curve");

  keyset.party_index = static_cast<coinbase::mpc::party_idx_t>(blob.role);
  keyset.curve = coinbase::crypto::curve_ed25519;

  const coinbase::crypto::mod_t& q = keyset.curve.order();
  if (!q.is_in_range(blob.x_share)) return coinbase::error(E_FORMAT, "invalid keyset blob");
  if (!q.is_in_range(blob.k_share)) return coinbase::error(E_FORMAT, "invalid keyset blob");

  keyset.root.x_share = blob.x_share;
  keyset.root.k_share = blob.k_share;

  error_t rv = keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed);
  if (rv) return rv;
  return keyset.root.K.from_bin(keyset.curve, blob.root_K_compressed);
}
