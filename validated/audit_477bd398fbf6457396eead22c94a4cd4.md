### Title
Attached NEAR Deposit Not Refunded on Intermediate Callback Failure in WalletContract — (`runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary

`WalletContract::rlp_execute` is `#[payable]` and accepts an attached NEAR deposit from any caller (typically a relayer). The contract tracks this as a `CallerDeposit` and is supposed to refund it if the underlying action fails. However, in two intermediate callbacks — `nep_141_storage_balance_callback` and `address_check_callback` — early-return error paths discard the `caller_deposit` without issuing a refund. The attached NEAR stays permanently in the wallet contract (the user's eth-implicit account), causing the relayer to lose their attached funds.

### Finding Description

`rlp_execute` is marked `#[payable]`: [1](#0-0) 

Inside `inner_rlp_execute`, the attached deposit is captured as a `CallerDeposit`: [2](#0-1) 

`CallerDeposit` is only created for external (non-self) callers with a non-zero deposit: [3](#0-2) 

The final callback `rlp_execute_callback` correctly refunds `caller_deposit` on failure: [4](#0-3) 

However, `nep_141_storage_balance_callback` has two early-return error paths that do **not** refund `caller_deposit`: [5](#0-4) 

Similarly, `address_check_callback` has early-return paths that drop `caller_deposit` without refunding: [6](#0-5) 

The `caller_deposit` parameter is received by both callbacks but is silently dropped in these error branches.

### Impact Explanation

A relayer who attaches NEAR to `rlp_execute` (e.g., to fund a cross-contract call) will permanently lose that NEAR if an intermediate callback fails early. The NEAR remains in the wallet contract (the user's eth-implicit account) with no mechanism to recover it. The corrupted value is the relayer's account balance (a concrete on-chain balance entry).

### Likelihood Explanation

An attacker who controls a wallet contract can deploy a malicious NEP-141 token contract whose `storage_balance_of` always fails or returns malformed JSON. The attacker then crafts an ERC-20 transfer Ethereum transaction targeting this malicious token and induces a relayer to submit it with an attached NEAR deposit. The `storage_balance_of` call fails, `nep_141_storage_balance_callback` returns early, and the relayer's deposit is not refunded. This is reachable via a normal public RPC transaction submitted by any unprivileged user acting as a relayer.

### Recommendation

In every early-return error path inside `nep_141_storage_balance_callback` and `address_check_callback`, issue a refund to the caller before returning, mirroring the pattern already used in `rlp_execute_callback`:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund_promise = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(
        refund_promise,
        NearToken::from_yoctonear(yocto_near.into()),
    );
}
```

This should be applied to all `return PromiseOrValue::Value(...)` error branches in both callbacks.

### Proof of Concept

1. Attacker deploys a malicious NEP-141 contract at `evil-token.near` whose `storage_balance_of` always panics.
2. Attacker (wallet contract owner) signs an Ethereum ERC-20 transfer transaction: `erc20_transfer(to=<some_address>, value=<amount>)` targeting `evil-token.near`.
3. Relayer calls `rlp_execute("evil-token.near", <tx_bytes>)` with `attached_deposit = 5 NEAR`.
4. `inner_rlp_execute` creates `CallerDeposit { account_id: relayer, yocto_near: 5e24 }`.
5. The ERC-20 path calls `evil-token.near::storage_balance_of(...)` then chains to `nep_141_storage_balance_callback`.
6. `storage_balance_of` panics → `PromiseResult::Failed` → early return at line 204–209 without refunding `caller_deposit`.
7. Relayer's 5 NEAR remains in the attacker's wallet contract. Relayer balance decreases by 5 NEAR with no recourse. [7](#0-6)
