Audit Report

## Title
Precision loss in `rewardPerToken()` combined with unconditional `updatedAt` advancement permanently freezes unclaimed yield for all stakers - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`rewardPerToken()` performs integer division that truncates to zero when `rewardRate * timeDelta * DECIMAL_PRECISION < totalKernelStaked`. The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when no reward increment was recorded. Because `getReward()` is permissionless and requires no stake, any external caller can repeatedly consume time intervals without any reward accrual, permanently locking the corresponding `rewardsToken` balance in the contract with no recovery mechanism.

## Finding Description

`rewardPerToken()` (L408–413) computes:

```solidity
rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

When `rewardRate * timeDelta * 1e18 < totalKernelStaked`, Solidity integer division truncates the increment to zero, leaving `rewardPerTokenStored` unchanged.

The `updateReward` modifier (L232–242) always executes:

```solidity
rewardPerTokenStored = rewardPerToken();   // unchanged (truncated to 0 increment)
updatedAt = lastTimeRewardApplicable();    // advances unconditionally
```

`updatedAt` advances regardless of whether `rewardPerTokenStored` changed. The elapsed time is permanently consumed — no staker can ever recover rewards for that interval.

`getReward()` (L382–390) has no access control, no minimum-stake requirement, and no minimum-elapsed-time guard. Any address with zero stake can call it, triggering `updateReward`, advancing `updatedAt`, and silently discarding the time window.

Existing guards are insufficient: `nonReentrant` prevents re-entrancy within a single call but does not prevent repeated calls across blocks. There is no check that `rewardPerToken()` actually increased before advancing `updatedAt`.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Each successful attacker call within the truncation window permanently destroys the yield that should have accrued during that interval. The `rewardsToken` balance corresponding to the lost intervals remains locked in the contract; there is no admin sweep, no recovery function, and no mechanism to redistribute it. Sustained over a full reward period, 100% of distributed rewards can be rendered permanently inaccessible to all stakers.

The attacker receives nothing — this is not theft to the attacker — so the precise allowed impact is **permanent freezing of unclaimed yield** (Medium), not theft of unclaimed yield.

## Likelihood Explanation

The attack is unconditionally reachable by any unprivileged external caller with zero capital. The truncation window is determined by:

```
timeDelta < totalKernelStaked / (rewardRate * DECIMAL_PRECISION)
```

With a standard 18-decimal reward token and realistic staking amounts, the window is on the order of seconds. On L2 chains (Arbitrum, Base, Optimism) where block times are 0.25–2 s and gas costs are negligible, a bot can sustain the attack for the entire reward period with no stake and minimal cost. The condition worsens with low-decimal reward tokens (e.g., USDT/USDC), where the truncation window widens to ~6 seconds per the PoC.

## Recommendation

1. **Conditionally advance `updatedAt`** — only update it when `rewardPerToken()` strictly increases:

```solidity
modifier updateReward(address _account) {
    uint256 newRewardPerToken = rewardPerToken();
    if (newRewardPerToken > rewardPerTokenStored) {
        rewardPerTokenStored = newRewardPerToken;
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

2. **Scale `rewardRate` by `DECIMAL_PRECISION` at storage time** in `notifyRewardAmount` so the numerator is always large enough to avoid truncation for any realistic `timeDelta`.

3. **Restrict `getReward()` to callers with a non-zero staked balance**, eliminating the zero-cost attack vector entirely.

## Proof of Concept

```
Setup:
  rewardsToken      = USDT (6 decimals)
  duration          = 604800 s (7 days)
  reward            = 100e6 USDT
  rewardRate        = 100e6 / 604800 ≈ 165
  totalKernelStaked = 1_000e18 KERNEL
  DECIMAL_PRECISION = 1e18

Truncation condition:
  165 * timeDelta * 1e18 / 1_000e18 = 0  when timeDelta ≤ 6 s

Attack (bot, zero KERNEL stake):
  t=0:   notifyRewardAmount() → updatedAt = 0
  t=5:   attacker calls getReward()
           rewardPerToken() = (165 * 5 * 1e18) / 1_000e18 = 0 (truncated)
           updatedAt → 5, rewardPerTokenStored unchanged
  t=10:  attacker calls getReward()
           rewardPerToken() = (165 * 5 * 1e18) / 1_000e18 = 0 (truncated)
           updatedAt → 10
  ... repeated every 5 s for 604800 s ...

Result:
  rewardPerTokenStored = 0 for entire reward period
  All stakers earn 0 rewards
  100e6 USDT permanently locked in contract
```

Foundry invariant test plan: deploy `KernelDepositPool` with USDT as `rewardsToken`, stake 1000e18 KERNEL from a legitimate staker, call `notifyRewardAmount(100e6)`, then fuzz-loop `getReward()` from a zero-stake address with `timeDelta ∈ [1, 6]` seconds between calls.
