Audit Report

## Title
Native ETH Sent with Non-WETH ERC20 Swap Is Silently Stranded and Stealable via `refundETH()` - (File: `metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
All four swap entry points in `MetricOmmSimpleRouter` are declared `payable` but impose no guard requiring `msg.value == 0` when `tokenIn` is a plain ERC20. When a caller sends native ETH alongside such a swap, `PeripheryPayments.pay()` silently ignores the ETH and settles via `safeTransferFrom`, leaving the ETH stranded on the router. Any unprivileged address can immediately drain the stranded ETH by calling the public `refundETH()`, which transfers the full router ETH balance to `msg.sender` with no attribution to the original depositor.

## Finding Description
`PeripheryPayments.pay()` (L69–88) branches on `token`:

```
if (payer == address(this))  → ERC20 safeTransfer
else if (token == WETH)      → consume native balance, then safeTransferFrom
else                         → safeTransferFrom(payer, recipient, value)  // msg.value ETH untouched
```

When `token != WETH`, the `else` branch at L85–87 runs `safeTransferFrom` and never references `address(this).balance`. Any `msg.value` already on the contract is left stranded.

All four public swap functions — `exactInputSingle` (L67), `exactInput` (L92), `exactOutputSingle` (L130), `exactOutput` (L154) — are `payable` and set the callback payer to `msg.sender` with the caller-supplied `tokenIn`. None check `msg.value == 0` when `tokenIn != WETH`.

The `receive()` guard at L32–34 (`if (msg.sender != WETH) revert NotWETH()`) does not protect against this: it only fires for bare ETH transfers (no calldata), not for ETH attached to a function call.

`refundETH()` at L58–63 then transfers the entire router ETH balance to `msg.sender`:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

There is no access control, no attribution to the original depositor, and no minimum-balance check. Any address calling `refundETH()` after the victim's swap receives the full stranded amount.

## Impact Explanation
Direct loss of user principal. A user who accidentally sends ETH alongside a non-WETH ERC20 swap loses the full `msg.value` to any frontrunner or MEV bot that calls `refundETH()` in the same or a subsequent block. The swap itself succeeds (ERC20 is pulled correctly via `safeTransferFrom`), so the user receives no revert signal and no indication of loss. The stolen value equals the full `msg.value` sent. This satisfies the "Critical/High/Medium direct loss of user principal" impact gate.

## Likelihood Explanation
Medium. The Uniswap v3 multicall pattern of attaching ETH to a swap and calling `refundETH()` afterward is well-known; users familiar with it may accidentally attach `msg.value` to a non-WETH swap. The interface NatSpec at L11 of `IMetricOmmSimpleRouter.sol` states "ERC-20 routes only. No native ETH" but this is documentation only — there is no on-chain enforcement. The four swap functions being `payable` actively invites ETH attachment. The theft requires no special privilege and is repeatable by any observer of the mempool.

## Recommendation
Add a guard in the `else` branch of `pay()` to revert if `msg.value != 0`:

```solidity
} else {
    if (msg.value != 0) revert NativeETHNotAccepted();
    IERC20(token).safeTransferFrom(payer, recipient, value);
}
```

Alternatively, add a modifier to each swap entry point:

```solidity
modifier noNativeUnlessWETH(address tokenIn) {
    if (tokenIn != WETH && msg.value != 0) revert NativeETHNotAccepted();
    _;
}
```

Apply it to `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput`. The `pay()` approach is preferable as it is a single enforcement point covering all current and future callers.

## Proof of Concept
```solidity
// Victim calls exactInputSingle with accidental msg.value:
router.exactInputSingle{value: 1 ether}(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),
        tokenIn:          address(USDC),   // NOT WETH
        tokenOut:         address(WETH),
        zeroForOne:       false,
        amountIn:         1_000e6,
        amountOutMinimum: 0,
        recipient:        victim,
        deadline:         block.timestamp + 60,
        priceLimitX64:    type(uint128).max,
        extensionData:    ""
    })
);
// pay() hits the else branch → safeTransferFrom(USDC, victim, pool, 1000e6)
// 1 ETH is now stranded on the router.
// Victim receives WETH output; no revert; no indication of loss.

// Attacker calls refundETH() in the same block:
router.refundETH();
// _transferETH(msg.sender, 1 ether) → attacker receives victim's 1 ETH.
```

Foundry test skeleton: deploy `MetricOmmSimpleRouter` with a USDC/WETH pool, fund victim with USDC allowance, call `exactInputSingle{value: 1 ether}` with `tokenIn = USDC`, assert `address(router).balance == 1 ether`, then call `router.refundETH()` from attacker address and assert attacker balance increased by 1 ether and router balance is zero.
