Audit Report

## Title
Unset `queuedWithdrawalsBuffer` Allows Instant Withdrawals to Drain Vault Assets Reserved for Queued Withdrawal Claimants — (`contracts/LRTUnstakingVault.sol`)

## Summary
`getAssetsAvailableForInstantWithdrawal` uses `queuedWithdrawalsBuffer[asset]` as its sole guard, but this mapping defaults to `0` and is only set by a separate, non-atomic operator call. When the buffer is unset, the entire vault balance is available for instant withdrawals, allowing an attacker to drain assets that are owed to pending queued withdrawal users. If the vault is emptied, `unlockQueue` reverts on `AmountMustBeGreaterThanZero` and queued users — whose rsETH is already locked in the `WithdrawalManager` — cannot complete their withdrawals.

## Finding Description
`queuedWithdrawalsBuffer` is declared as a mapping that defaults to `0`: [1](#0-0) 

`setQueuedWithdrawalsBuffer` is the only write path and requires an explicit operator call: [2](#0-1) 

`getAssetsAvailableForInstantWithdrawal` computes availability solely from this buffer: [3](#0-2) 

When `reservedBuffer == 0`, `availableAmount == vaultBalance` — the entire vault is exposed. `instantWithdrawal` enforces only this single check before calling `unstakingVault.redeem`: [4](#0-3) 

There is no cross-check against `assetsCommitted[asset]` in the instant withdrawal path. `assetsCommitted` is only consulted in `getAvailableAssetAmount`, which gates new queued withdrawal initiations — not instant withdrawals: [5](#0-4) 

`unlockQueue` reads the raw vault balance via `_createUnlockParams`: [6](#0-5) 

And immediately reverts if that balance is zero: [7](#0-6) 

Queued users' rsETH is transferred into the `WithdrawalManager` at `initiateWithdrawal` time and is only burned during `unlockQueue`. If `unlockQueue` cannot execute, their rsETH remains locked and their asset claim is unserviceable: [8](#0-7) 

## Impact Explanation
**High — Theft of unclaimed yield.** The vault accumulates staking yield on top of principal as assets are returned from EigenLayer. Queued withdrawal users are entitled to their pro-rata share of that yield (locked in at `initiateWithdrawal` time via `expectedAssetAmount`). An instant withdrawer holding sufficient rsETH can consume the entire vault balance — principal plus accrued yield — before `unlockQueue` is called, taking yield that belongs to queued claimants. The queued users have already surrendered their rsETH to the `WithdrawalManager`; they cannot reclaim it and cannot receive their assets until the vault is replenished externally.

## Likelihood Explanation
**Medium.** The three required preconditions are:
1. `isInstantWithdrawalEnabled[asset] == true` — a normal operational state enabled by an operator.
2. `queuedWithdrawalsBuffer[asset] == 0` — the **default** state for every asset; requires no attacker action.
3. The vault holds assets from a completed EigenLayer unstaking cycle while queued withdrawals are pending — a routine operational window.

Condition 2 is the default for every asset. An operator who enables instant withdrawal without also calling `setQueuedWithdrawalsBuffer` (a separate, non-atomic call) leaves the system vulnerable. No privileged access is required by the attacker; any rsETH holder can trigger the exploit via the public `instantWithdrawal` function.

## Recommendation
1. **Enforce the buffer automatically.** When `unlockQueue` is called, treat `assetsCommitted[asset]` (converted to vault-asset terms) as the minimum reserved buffer, independent of the manually set value.
2. **Alternatively**, in `instantWithdrawal`, subtract `assetsCommitted[asset]` from the available amount so the vault always retains enough for pending queued claims.
3. **At minimum**, require that `setQueuedWithdrawalsBuffer` is called atomically with `setInstantWithdrawalEnabled` (e.g., a combined setter), or add a check in `setInstantWithdrawalEnabled` that reverts if the buffer is still zero when enabling.

## Proof of Concept
```solidity
// Setup:
// - queuedWithdrawalsBuffer[ETH] = 0 (default, never set)
// - isInstantWithdrawalEnabled[ETH] = true
// - Vault holds 100 ETH (50 ETH owed to queued withdrawal users, 50 ETH yield)
// - victim has a pending queued withdrawal for 50 ETH (rsETH already in WithdrawalManager)
// - attacker holds rsETH worth 100 ETH at current oracle price

// Step 1: attacker calls instantWithdrawal
// getAssetsAvailableForInstantWithdrawal returns 100 ETH (buffer = 0)
// check passes: 100 ETH <= 100 ETH
withdrawalManager.instantWithdrawal(ETH_TOKEN, attackerRsETH, "");
// vault is now empty

// Step 2: operator calls unlockQueue for victim
// _createUnlockParams: totalAvailableAssets = unstakingVault.balanceOf(ETH) = 0
// reverts: AmountMustBeGreaterThanZero
// victim cannot complete withdrawal — rsETH locked, assets frozen

assertEq(address(unstakingVault).balance, 0);
vm.expectRevert(ILRTWithdrawalManager.AmountMustBeGreaterThanZero.selector);
withdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, minAsset, minRsETH, maxAsset, maxRsETH);
```
