### Title
Block Producer Can Front-Run Treasury Withdrawal Enactment via Stable-Sort Submission-Order Dependency - (File: eras/conway/impl/src/Cardano/Ledger/Conway/Governance/Internal.hs)

### Summary
The `reorderActions` function uses a stable sort that preserves submission order for same-priority governance actions. All `TreasuryWithdrawals` proposals share priority 5. The `RATIFY` rule processes them sequentially while updating the evolving `ensTreasury` balance after each enactment. A block producer can order transactions within their block to control which `TreasuryWithdrawals` proposal is inserted first into the `Proposals` OMap, determining which proposals succeed when the treasury is insufficient for all ratified withdrawals.

### Finding Description
`reorderActions` in `Cardano.Ledger.Conway.Governance.Internal` sorts governance actions by `actionPriority` using `sortOn`, which is Haskell's stable sort:

```haskell
reorderActions = SS.fromList . sortOn (actionPriority . gasAction) . toList
```

All `TreasuryWithdrawals` actions receive priority 5:

```haskell
actionPriority TreasuryWithdrawals {} = 5
```

Because the sort is stable, multiple `TreasuryWithdrawals` proposals maintain their original insertion order from the `Proposals` OMap. That insertion order is determined by the order transactions were included in blocks.

In `ratifyTransition`, actions are processed head-first from the signal sequence. When a `TreasuryWithdrawals` action passes all checks and is enacted, the `ENACT` rule reduces `ensTreasury`:

```haskell
TreasuryWithdrawals wdrls _ ->
  ...
  ensTreasury = ensTreasury st <-> wdrlsAmount
```

The updated `EnactState` (with reduced `ensTreasury`) is threaded into the recursive call for the remaining actions. Each subsequent `TreasuryWithdrawals` proposal is checked against the already-depleted treasury via `withdrawalCanWithdraw`:

```haskell
withdrawalCanWithdraw (TreasuryWithdrawals m _) treasury =
  Map.foldr' (<+>) zero m <= treasury
```

A block producer controlling a slot can order the submission transactions of competing `TreasuryWithdrawals` proposals within their block, ensuring their own proposal is inserted into the `Proposals` OMap before a competing proposal. At the epoch boundary, `runConwayRatify` calls `reorderActions` on the pulser's snapshot, and the stable sort preserves the block-producer-controlled submission order for all priority-5 actions. The block producer's proposal is processed first, depleting the treasury, and the competing proposal fails `withdrawalCanWithdraw`.

### Impact Explanation
A block producer below consensus threshold can cause a governance-ratified `TreasuryWithdrawals` proposal to fail enactment by front-running it with their own competing proposal at the transaction-ordering level. The victim's proposal, despite having received sufficient DRep/SPO/committee votes, is silently skipped at the epoch boundary because the treasury was already depleted by the attacker's earlier-ordered proposal. This modifies treasury withdrawals outside the intended governance outcome: the governance process approved both proposals, but the block producer unilaterally determines which one is enacted. This matches the **Medium** impact: attacker-controlled blocks modify treasury withdrawals outside design parameters.

### Likelihood Explanation
The scenario requires: (1) a block producer slot, (2) two or more `TreasuryWithdrawals` proposals that both achieve ratification thresholds in the same epoch, and (3) a treasury balance insufficient to cover all of them simultaneously. Condition (3) is the binding constraint in practice, but treasury-constrained epochs are plausible during periods of high governance activity or low reserves. Condition (1) is available to any registered stake pool operator who wins a slot, which is below the consensus threshold. Likelihood is **medium-low** but non-negligible.

### Recommendation
Replace the stable sort in `reorderActions` with a sort that breaks ties among same-priority actions by a deterministic, manipulation-resistant key such as `GovActionId` (which encodes the submitting transaction ID and output index). This removes the block-producer-controlled submission-order dependency:

```haskell
reorderActions =
  SS.fromList
    . sortOn (\gas -> (actionPriority (gasAction gas), gasId gas))
    . toList
```

Alternatively, document explicitly that submission order determines enactment priority among same-priority proposals and that block producers can exploit this ordering when the treasury is constrained.

### Proof of Concept
1. Treasury = 100 ADA. Two `TreasuryWithdrawals` proposals are pending in the mempool: victim's proposal B (60 ADA) and attacker's proposal A (60 ADA).
2. The block producer includes the submission transaction for A before B in their block. A is inserted first into the `Proposals` OMap.
3. Both proposals accumulate sufficient DRep, SPO, and committee votes over the epoch.
4. At the epoch boundary, `runConwayRatify` calls `reorderActions` on the pulser snapshot. Both proposals have priority 5; stable sort preserves A before B.
5. `ratifyTransition` processes A first: `withdrawalCanWithdraw` passes (60 ≤ 100), `ENACT` fires, `ensTreasury` → 40 ADA.
6. `ratifyTransition` processes B next: `withdrawalCanWithdraw` fails (60 > 40). B is not enacted and remains in proposals until expiry.
7. The block producer's withdrawal is credited; the victim's governance-approved withdrawal silently fails.
