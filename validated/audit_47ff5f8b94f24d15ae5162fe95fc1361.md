### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Validating It Against the Original Request — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over the **caller-supplied** `response.payload_hash`, but never checks that `response.payload_hash` is the correct hash derived from the original pending `VerifyForeignTransactionRequest`. A single Byzantine attested participant can therefore resolve a pending foreign-chain verification request with a `payload_hash` that corresponds to a *different* transaction, delivering a forged verification response to the original caller.

---

### Finding Description

When a user calls `verify_foreign_transaction`, the contract stores the full `VerifyForeignTransactionRequest` (containing the `ForeignChainRpcRequest`, `domain_id`, and `payload_version`) as the key in `pending_verify_foreign_tx_requests` and parks the caller's execution via a yield-resume promise.

When a node calls `respond_verify_foreign_tx`, the contract:

1. Looks up the pending yield by the supplied `request` key.
2. Retrieves the domain's root public key.
3. Verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root key.
4. Resolves all queued yields with the raw `response` bytes. [1](#0-0) 

The critical gap is step 3: the contract verifies the signature over **whatever hash the responding node supplies**, but never reconstructs the expected hash from the original request and compares it to `response.payload_hash`.

The expected hash is `SHA-256(borsh(ForeignTxSignPayload { request: original_request, values: extracted_values }))`. The contract has all the information needed to enforce this (the original `request` is the lookup key), but it does not. [2](#0-1) 

By contrast, the `respond` path for ordinary `sign` requests derives the expected public key from the request's `tweak` field and verifies the signature against that derived key — a tighter binding between request parameters and the verification step. [3](#0-2) 

The node-side code confirms the intended invariant: nodes compute `payload_hash = ForeignTxSignPayload::V1{request, values}.compute_msg_hash()` and sign that hash with a zero tweak. [4](#0-3) 

The contract never enforces this invariant on the response path.

---

### Impact Explanation

**High — Forged foreign-chain verification causing invalid bridge execution.**

A Byzantine attested participant can:

1. Participate honestly in the MPC signing round for pending request **A** (transaction `tx_A`), obtaining a valid threshold signature `sig_A` over `hash_A`.
2. While request **B** (transaction `tx_B`) is also pending, call `respond_verify_foreign_tx(request_B, { payload_hash: hash_A, signature: sig_A })`.
3. The contract accepts the call: `sig_A` is a valid signature over `hash_A` under the root key, and `request_B` exists in the pending map.
4. The caller of request **B** receives `{ payload_hash: hash_A, signature: sig_A }` — a response that attests to `tx_A`, not `tx_B`.

Any bridge contract that does not independently reconstruct and compare the expected `payload_hash` will accept this as proof that `tx_B` was verified by the MPC network, enabling invalid bridge execution or double-spend conditions. The documentation explicitly states callers are expected to validate the hash themselves, but the contract — the trust anchor — provides no on-chain enforcement. [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The attacker must be an attested MPC participant (a real operational requirement, not a privileged role in the threat model). No threshold collusion is needed — a single Byzantine node that participates in one legitimate signing round can obtain a valid signature and immediately reuse it against a different pending request. In a production environment where multiple `verify_foreign_transaction` requests are pending simultaneously (the fan-out design explicitly supports this), the preconditions are routinely met. [6](#0-5) 

---

### Recommendation

In `respond_verify_foreign_tx`, reconstruct the expected `payload_hash` from the original request before accepting the response. Concretely:

1. Compute `expected_hash = ForeignTxSignPayload::V1 { request: request.request.clone(), values: response.extracted_values }.compute_msg_hash()`.
2. Assert `response.payload_hash == expected_hash` before proceeding to signature verification.

This requires the `VerifyForeignTransactionResponse` to include the `extracted_values` used to build the payload (or the contract must store them at request time). Alternatively, store the expected `payload_hash` in the pending-request map at enqueue time once nodes have agreed on it, and compare on respond.

The `respond` path for ordinary signatures provides the correct model: the verification key is derived deterministically from the stored request parameters, leaving no room for the responder to substitute a different value. [3](#0-2) 

---

### Proof of Concept

**Setup:** Two `verify_foreign_transaction` requests are pending — `req_A` (for Bitcoin tx `[0xAA; 32]`) and `req_B` (for Bitcoin tx `[0xBB; 32]`).

**Attack:**

```
// Step 1: MPC network signs payload for req_A
payload_A = ForeignTxSignPayload::V1 { request: req_A.request, values: [BlockHash([0x11;32])] }
hash_A    = SHA-256(borsh(payload_A))
sig_A     = MPC_threshold_sign(hash_A)   // Byzantine node participates honestly here

// Step 2: Byzantine node submits sig_A as the response for req_B
respond_verify_foreign_tx(
    request  = req_B,                          // matches a real pending entry
    response = { payload_hash: hash_A,         // hash of tx_A, not tx_B
                 signature:    sig_A }         // valid sig over hash_A
)

// Contract check (lib.rs:729-734):
//   verify_ecdsa_signature(sig_A, hash_A, root_pk)  → OK  ✓
//   (no check that hash_A == expected_hash_for_req_B)

// Result: caller of req_B receives { payload_hash: hash_A, sig_A }
// Bridge contract interprets this as "MPC verified tx_BB" — forged.
```

The contract's only guard is the signature check at lines 729–734; the `payload_hash` binding to the original request is entirely absent. [7](#0-6)

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
