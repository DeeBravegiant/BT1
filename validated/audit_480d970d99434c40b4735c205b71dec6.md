Audit Report

## Title
Zero rsETH Output on Sub-Threshold Deposits Silently Retains User Funds — (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

## Summary
All three L2 pool contracts compute `rsETHAmount` via integer division that truncates to zero when the deposited amount is below the current rsETH/ETH exchange rate threshold. No guard checks that `rsETHAmount > 0` before the contract accepts the user's ETH or tokens and proceeds to mint/transfer zero wrsETH/rsETH. The deposited value is permanently retained by the pool with no revert and no on-chain indication of failure.

## Finding Description
In all three pool contracts, the ETH deposit path computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

and the token deposit path computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

With `rsETHToETHrate ≈ 1.05e18` (growing over time), any `amountAfterFee < rsETHToETHrate / 1e18` (i.e., ≤ 1 wei for ETH at current rates) causes Solidity integer division to truncate `rsETHAmount` to zero.

The deposit functions then proceed unconditionally:

- `RSETHPoolV3.sol` line 262: `wrsETH.mint(msg.sender, rsETHAmount)` — mints 0, no revert
- `RSETHPoolNoWrapper.sol` line 241: `rsETH.safeTransfer(msg.sender, rsETHAmount)` — transfers 0, no revert
- `RSETHPoolV3ExternalBridge.sol` line 381: `wrsETH.mint(msg.sender, rsETHAmount)` — mints 0, no revert

OpenZeppelin's `_mint(to, 0)` and `safeTransfer(to, 0)` both succeed silently. For ETH deposits, `msg.value` is already held by the contract before the computation. For token deposits, `safeTransferFrom` pulls the tokens before `rsETHAmount` is computed, so the tokens are already in the pool when the zero transfer occurs.

The only existing guard is `if (amount == 0) revert InvalidAmount()`, which does not protect against a nonzero `amount` that produces a zero output after division. There is no `minRSETHAmountExpected` slippage parameter in any of these deposit signatures. By contrast, `LRTDepositPool._beforeDeposit` enforces both a `minAmountToDeposit` floor and a `minRSETHAmountExpected` slippage check, reverting with `MinimumAmountToReceiveNotMet()` if the output is insufficient.

Affected code paths:
- `RSETHPoolV3.sol` lines 256–262 (ETH deposit) and lines 282–290 (token deposit)
- `RSETHPoolV3.sol` lines 299–308 (`viewSwapRsETHAmountAndFee`)
- `RSETHPoolNoWrapper.sol` lines 235–241 (ETH deposit) and lines 260–268 (token deposit)
- `RSETHPoolNoWrapper.sol` lines 277–286 (`viewSwapRsETHAmountAndFee`)
- `RSETHPoolV3ExternalBridge.sol` lines 375–381 (ETH deposit) and lines 401–409 (token deposit)
- `RSETHPoolV3ExternalBridge.sol` lines 418–427 (`viewSwapRsETHAmountAndFee`)

## Impact Explanation
**Low — Contract fails to deliver promised returns.**

A depositor who sends any ETH or token amount below the truncation threshold receives zero wrsETH/rsETH while their deposited value is permanently retained by the pool. The `SwapOccurred` event is emitted with `rsETHAmount = 0`, giving no on-chain indication of failure. The practical loss per transaction is at most a few wei today for ETH, but the threshold grows as the rsETH exchange rate appreciates. For token deposits with lower-precision oracles, the truncation threshold can be meaningfully larger. Integrators or smart-contract callers that do not inspect the emitted event value will silently lose funds.

## Likelihood Explanation
Any unprivileged depositor can trigger this by sending a sufficiently small ETH value (e.g., 1 wei) to `deposit(string)` on any of the three contracts. No front-running, oracle manipulation, or admin action is required. The condition is deterministic and reproducible at any time. All affected entry points are public and payable. The threshold is currently ~1 wei for ETH but grows monotonically as restaking rewards accrue.

## Recommendation
Add a zero-output guard in each `deposit()` function immediately after computing `rsETHAmount`:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert ZeroRsETHMinted();
```

Apply this to all ETH and token deposit overloads in `RSETHPoolV3`, `RSETHPoolNoWrapper`, and `RSETHPoolV3ExternalBridge`. Optionally, add a `minRSETHAmountExpected` parameter (as `LRTDepositPool._beforeDeposit` does at line 667) to give callers explicit slippage control.

## Proof of Concept
Assume `rsETHToETHrate = 1.05e18`, `feeBps = 0`:

1. Alice calls `RSETHPoolV3.deposit{value: 1}("")` (1 wei ETH).
2. `viewSwapRsETHAmountAndFee(1)` computes:
   - `fee = 1 * 0 / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
3. `feeEarnedInETH += 0`
4. `wrsETH.mint(Alice, 0)` — succeeds silently; Alice receives 0 wrsETH
5. Alice's 1 wei ETH is held by the pool with no corresponding wrsETH issued
6. `SwapOccurred(Alice, 0, 0, "")` is emitted — no revert, no failure indication

**Foundry fuzz test plan:**
```solidity
function testFuzz_zeroOutputDeposit(uint256 amount) public {
    vm.assume(amount > 0 && amount < rsETHPool.getRate() / 1e18 + 1);
    uint256 balanceBefore = wrsETH.balanceOf(alice);
    vm.deal(alice, amount);
    vm.prank(alice);
    rsETHPool.deposit{value: amount}("");
    assertEq(wrsETH.balanceOf(alice), balanceBefore); // alice received 0 wrsETH
    assertGt(address(rsETHPool).balance, 0);          // pool retained alice's ETH
}
```

The same path applies to token deposits in all three pool contracts via `deposit(address token, uint256 amount, string referralId)`.
