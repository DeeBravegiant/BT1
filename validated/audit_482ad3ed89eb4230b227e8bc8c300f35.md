Audit Report

## Title
`account != msg.sender` Guard in `_processClaim` Permanently Freezes Unclaimed KERNEL Yield for Smart Contract Recipients - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

## Summary
`KernelMerkleDistributor._processClaim` unconditionally reverts with `Unauthorized()` when `account != msg.sender`, blocking any caller other than the exact recipient address from triggering a claim. Smart contract addresses included in the merkle tree that lack an internal function to call `claim` or `claimAndStake` cannot self-claim and cannot be claimed on their behalf, permanently stranding their allocated KERNEL yield in the distributor with no admin recovery path.

## Finding Description
Both `claim` and `claimAndStake` delegate to `_processClaim` at lines 261 and 280 respectively. Inside `_processClaim` (lines 311–313):

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

This check is unconditional. The merkle proof at lines 320–323 already cryptographically binds `index`, `account`, and `cumulativeAmount` — only a caller possessing a valid proof for a specific `account` can succeed — so the `msg.sender` identity check provides no additional security. Yet it blocks every caller that is not `account` itself.

The `IMerkleDistributor` interface NatSpec at line 122 describes the `account` parameter as "The address to send the token to," and `claimAndStake` calls `IKernelDepositPool.stakeFor(account, amountToStake)` at line 282, both of which imply third-party claiming is a supported design intent. The guard contradicts this intent.

For a smart contract recipient (vault, pool, multisig, reward router) that has no function to call `claim` or `claimAndStake`:
1. Any EOA calling `claim(index, address(contractRecipient), amount, proof)` hits `account != msg.sender` → `revert Unauthorized()`.
2. The contract recipient itself cannot self-call unless it implements a dedicated claim function.
3. `userClaims[account]` is never updated, so the allocation is never marked claimed.
4. No admin function exists to sweep or redirect the stranded tokens; the only admin state-change functions are `setKernelDepositPool`, `setProtocolTreasury`, `setFeeInBPS`, `setMerkleRoot`, `pause`, and `unpause` — none of which transfer unclaimed tokens to an alternate address.

The KERNEL tokens remain locked in `KernelMerkleDistributor` indefinitely.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

KERNEL tokens allocated to a smart contract recipient in the merkle tree are irrecoverably stranded if that contract cannot self-call `claim`. This matches the allowed impact class "Medium. Permanent freezing of unclaimed yield." The loss is bounded to the affected recipient's allocation rather than all protocol funds, justifying Medium rather than Critical severity.

## Likelihood Explanation
Medium. The KERNEL ecosystem explicitly supports staking on behalf of other addresses via `KernelDepositPool.stakeFor` with `STAKE_FOR_ROLE`, and `claimAndStake` is designed to stake directly for `account`, indicating protocol-level addresses are expected participants. Off-chain merkle tree construction is controlled by the protocol and can include any address — including vaults, liquidity pools, or reward routers — without any on-chain enforcement preventing it. No attacker action is required; the freeze occurs automatically the moment such an address is included in a merkle root and no bespoke claim function exists in that contract.

## Recommendation
Remove the `account != msg.sender` guard from `_processClaim`. The merkle proof verification at lines 320–323 already ensures only a caller holding a valid proof for `account` can succeed; the identity check is redundant for security. This allows any address to permissionlessly claim on behalf of a recipient, which is safe because tokens are always sent to `account` (line 263) or staked for `account` (line 282), never to `msg.sender`.

Alternatively, introduce a delegated-claimer mapping (analogous to EigenLayer's `claimerFor`) so accounts can pre-authorize a third-party address to claim on their behalf, preserving access control while enabling delegation.

## Proof of Concept
1. Protocol builds a merkle tree including `address(vaultContract)` with `cumulativeAmount = X`.
2. Owner calls `KernelMerkleDistributor.setMerkleRoot(root)`.
3. EOA calls `kernelMerkleDistributor.claim(index, address(vaultContract), X, proof)`.
4. `_processClaim` evaluates `address(vaultContract) != msg.sender (EOA)` → `revert Unauthorized()`.
5. `vaultContract` has no function to call `claim`, so it also cannot self-claim.
6. `userClaims[address(vaultContract)]` remains `{lastClaimedIndex: 0, cumulativeAmount: 0}` forever.
7. KERNEL tokens equal to `X` (minus fee) remain locked in `KernelMerkleDistributor` with no recovery path.

Foundry test sketch:
```solidity
function test_smartContractRecipientFrozen() public {
    // Deploy a contract with no claim function
    address vault = address(new VaultWithNoClaim());
    // Build merkle tree with vault as recipient, set root
    // ...
    vm.prank(address(0xBEEF)); // any EOA
    vm.expectRevert(IMerkleDistributor.Unauthorized.selector);
    distributor.claim(1, vault, 1e18, proof);
    // vault itself also cannot call claim — allocation is frozen
}
```
