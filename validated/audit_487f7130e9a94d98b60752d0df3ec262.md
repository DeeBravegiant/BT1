### Title
Stale Decimal Registration via `MintCloseAuthority` on Token-2022 Mints in `log_metadata` - (File: solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs)

### Summary

The `log_metadata` instruction accepts Token-2022 mints without checking for the `MintCloseAuthority` extension. An attacker who controls a mint's close authority can register a mint with high decimals (e.g., 18), then close and reinitialize the mint at the same address with lower decimals (e.g., 6). NEAR's `TokenExists` guard permanently locks in the stale high-decimal registration, while all subsequent `init_transfer` calls use the actual low-decimal mint. This creates a permanent decimals accounting divergence between Solana and NEAR.

### Finding Description

The `log_metadata` instruction in `solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs` reads `self.mint.decimals` at call time and posts it to NEAR via Wormhole. The only constraint on the mint is that its `mint_authority` must not be the bridge's authority PDA: [1](#0-0) 

No check is performed for the `MintCloseAuthority` Token-2022 extension. The vault is created with `init_if_needed`, meaning `log_metadata` can be called multiple times for the same mint: [2](#0-1) 

The decimals are serialized into the Wormhole payload and sent to NEAR: [3](#0-2) 

On the NEAR side, `add_token` stores the decimals with a `TokenExists` guard that permanently prevents re-registration: [4](#0-3) 

When `init_transfer` is later called with the reinitialized (low-decimal) mint, `transfer_checked` uses the current on-chain decimals correctly on Solana: [5](#0-4) 

But NEAR's accounting uses the stale registered decimals, producing a `normalize_amount` / `denormalize_amount` calculation with the wrong `origin_decimals`: [6](#0-5) 

### Impact Explanation

A permanent mismatch between the decimals registered on NEAR and the actual on-chain mint decimals causes every cross-chain transfer for that token to apply the wrong normalization factor. For example, if NEAR has `origin_decimals=18` registered but the actual mint has `decimals=6`, a user transferring 1 token (1e6 units) would receive 1e6 units of an 18-decimal NEAR token — worth 10^-12 of the intended value. This is a decimals/balance-accounting divergence that breaks backing guarantees, matching the "High" impact tier.

### Likelihood Explanation

Low. The attacker must be the creator of the Token-2022 mint (holding the close authority), must register the token with the bridge before closing and reinitializing it, and must attract real user transfers to the manipulated token. The technical steps are trivial, but the social engineering component reduces practical likelihood.

### Recommendation

In the `LogMetadata` account validation, add an explicit check that the Token-2022 `MintCloseAuthority` extension is absent (or that the close authority is `None`) before accepting the mint for registration. For example, when `token_program == token_2022::ID`, unpack the mint extensions and assert `get_extension::<MintCloseAuthority>()` either fails or returns a `None` close authority. This mirrors the recommendation in the external report and prevents the mint address from being recycled with different decimals after registration.

### Proof of Concept

1. Attacker creates a Token-2022 mint at address `M` with `MintCloseAuthority` set to attacker-controlled key `K`, and `decimals = 18`. Supply = 0.
2. Attacker calls `log_metadata` with mint `M`. The vault PDA `[VAULT_SEED, M]` is created (`init_if_needed`). A Wormhole message is posted with `decimals = 18`.
3. NEAR processes the Wormhole VAA via the prover. `add_token` registers `M` with `origin_decimals = 18`.
4. Attacker uses key `K` to invoke the Token-2022 `CloseAccount` instruction on mint `M` (supply is 0, so this succeeds). Mint account at `M` is deleted.
5. Attacker reinitializes a new Token-2022 mint at address `M` with `decimals = 6` (no `MintCloseAuthority` needed this time).
6. Attacker calls `log_metadata` again. The vault already exists (`init_if_needed` is a no-op). A new Wormhole message is posted with `decimals = 6`. NEAR rejects it with `TokenExists`.
7. NEAR permanently holds `origin_decimals = 18` for mint `M`. The actual on-chain mint has `decimals = 6`.
8. A victim calls `init_transfer` transferring 1 token (1e6 units, 6-decimal mint). Solana correctly locks 1e6 units. NEAR normalizes using `origin_decimals = 18`, crediting the victim with 1e6 units of an 18-decimal token — a 10^12 factor loss.
