### Title
Presignature Pool Exhaustion via Unbounded Distinct `sign()` Requests Causes Temporary DoS of Core Signing Functionality - (File: `crates/contract/src/lib.rs`, `crates/node/src/providers/ecdsa/sign.rs`)

---

### Summary

The `sign()` contract function accepts an unbounded number of **distinct** signature requests for only 1 yoctoNEAR each. Every distinct request, when picked up by a node leader, irreversibly consumes one presignature from the finite, slowly-replenished presignature pool via `presignature_store.take_owned().await`. An unprivileged attacker can flood the contract with distinct requests (varying payload or path), draining the presignature pool faster than nodes can regenerate it, causing all subsequent legitimate sign requests to block indefinitely and eventually time out.

---

### Finding Description

**Entry point — `sign()` in `crates/contract/src/lib.rs`:**

The only admission controls on `sign()` are:
- A minimum deposit of 1 yoctoNEAR (`MINIMUM_SIGN_REQUEST_DEPOSIT`)
- A minimum gas attachment of 10 Tgas
- A valid domain and payload format [1](#0-0) [2](#0-1) 

The `MAX_PENDING_REQUEST_FAN_OUT = 128` cap in `crates/contract/src/pending_requests.rs` only limits the fan-out queue for **duplicate** requests sharing the same `(caller, domain, payload, path)` key. [3](#0-2) [4](#0-3) 

There is **no global cap** on the number of distinct pending requests. The `pending_signature_requests` map is a `LookupMap<SignatureRequest, Vec<YieldIndex>>` that grows without bound as distinct `(caller, domain, payload, path)` tuples are submitted. [5](#0-4) 

**Resource consumption — `make_signature_leader()` in `crates/node/src/providers/ecdsa/sign.rs`:**

When a node leader picks up any pending sign request, it immediately and irreversibly calls `presignature_store.take_owned().await`, consuming one presignature from the finite pool: [6](#0-5) 

The same pattern applies to `RobustEcdsaSignatureProvider::make_signature_leader` and `VerifyForeignTxProvider::make_verify_foreign_tx_leader`: [7](#0-6) [8](#0-7) 

**The finite, slowly-replenished pool:**

Presignatures are pre-computed in the background. Each presignature requires a pair of Beaver triples, and triple generation is the bottleneck — heavy OT-based cryptographic computation done in batches of 64. The pool is bounded by `desired_presignatures_to_buffer` per domain. [9](#0-8) [10](#0-9) 

When the pool is empty, `take_owned()` **blocks indefinitely** until a new presignature is generated:

> "Eventually the node will exhaust its owned presignatures. At that point `take_owned()` blocks indefinitely (waiting for a presignature that will never arrive), and the node stops being able to lead signature computations." [11](#0-10) 

**Request expiration:**

Pending requests expire after `REQUEST_EXPIRATION_BLOCKS = 200` blocks. If a presignature is not available within that window, the request times out and the user receives `RequestError::Timeout`. [12](#0-11) 

---

### Impact Explanation

An attacker submitting N distinct sign requests (varying the 32-byte ECDSA payload) causes N presignatures to be consumed. Once the pool is exhausted, all subsequent legitimate sign requests block in `take_owned()` and time out after ~200 blocks. This is a temporary but complete DoS of the core signing functionality — the primary service the MPC network provides. The invariant that legitimate, properly-deposited sign requests are processed within the yield-resume window is broken. This maps to the **Medium** allowed impact: *request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.*

---

### Likelihood Explanation

The attack cost is approximately `N × 0.001 NEAR` (1 yoctoNEAR deposit + ~7 Tgas gas fee per request). On mainnet, `desired_presignatures_to_buffer` is on the order of hundreds of presignatures per domain. An attacker can exhaust the pool for well under 1 NEAR. The attack is fully automatable with a simple script that submits sign requests with incrementing payloads. No privileged access, no collusion, and no special knowledge is required — any NEAR account can call `sign()`.

---

### Recommendation

1. **Increase the minimum deposit** for `sign()` to a value that makes bulk exhaustion economically infeasible relative to the cost of presignature generation.
2. **Implement a global cap** on the total number of distinct pending sign requests across all keys, rejecting new submissions when the cap is reached.
3. **Implement per-account rate limiting** on sign requests to prevent a single account from monopolizing the presignature pool.
4. **Decouple presignature consumption from request acceptance**: do not consume a presignature until the request has been validated against some admission criterion (e.g., a stake or bond).

### Proof of Concept

```
# Attacker script (pseudocode)
for i in 0..POOL_SIZE:
    near call v1.signer sign \
        '{"request": {"payload_v2": {"Ecdsa": "<32-byte hash with i encoded>"}, "path": "attack", "domain_id": 0}}' \
        --deposit 0.000000000000000000000001 \   # 1 yoctoNEAR
        --gas 10000000000000                     # 10 Tgas
        --account-id attacker.near
```

1. Attacker submits `POOL_SIZE` distinct sign requests (each with a unique 32-byte ECDSA payload), paying ~1 yoctoNEAR + gas per request.
2. Node leaders pick up each request and call `presignature_store.take_owned().await`, consuming one presignature each.
3. The presignature pool is exhausted.
4. All subsequent legitimate sign requests from real users block in `take_owned()`.
5. After ~200 blocks, legitimate requests expire with `RequestError::Timeout`.
6. Nodes begin regenerating presignatures (slow: requires triple generation), but the attacker can re-flood as soon as the pool partially refills. [13](#0-12) [6](#0-5) [14](#0-13)

```rust
/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;
```

```rust
pub(crate) fn push_pending_yield<K>(

    K: BorshSerialize + BorshDeserialize + Clone + Ord,

    let queue = requests.entry(request).or_default();
```
