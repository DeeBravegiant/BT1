### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Creates Unbacked Wrapped Supply — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter in both the cross-chain event and the Wormhole message without measuring the actual tokens received. For fee-on-transfer (FOT) ERC-20 tokens, the bridge retains `amount − fee` but signals `amount` to the destination chain, which mints the full `amount` to the recipient. The backing ratio is permanently broken on every such deposit.

---

### Finding Description

In `OmniBridge.initTransfer`, when the token is neither a bridge-issued token nor a custom-minter token, the bridge pulls tokens from the caller with:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← requested amount, not received amount
);
``` [1](#0-0) 

Immediately after, the function emits the cross-chain event using the same caller-supplied `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← not the actual received balance
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

`OmniBridgeWormhole.initTransferExtension` then publishes this same `amount` into the Wormhole VAA that the NEAR side consumes:

```solidity
Borsh.encodeUint128(amount),
``` [3](#0-2) 

The NEAR bridge reads the proof and calls `fin_transfer_callback`, which denormalizes and credits the full `amount` to the recipient:

```solidity
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [4](#0-3) 

No balance snapshot (`balanceOf(address(this))` before/after) is taken anywhere in the EVM path to reconcile what was actually received.

---

### Impact Explanation

Every `initTransfer` call with a FOT token creates a permanent deficit: the bridge holds `amount − transferFee` on EVM but has authorized minting of `amount` on NEAR. Repeated deposits accumulate an unbacked surplus of wrapped tokens. When users redeem on EVM, the bridge's actual reserves are insufficient to cover all outstanding claims, causing either failed withdrawals (irreversible fund lock) or, if the bridge pays from reserves, unauthorized release of other users' locked assets.

This maps directly to the allowed critical impact: **unauthorized creation of unbacked wrapped bridge assets through accounting failure**.

---

### Likelihood Explanation

The entry point is the public `initTransfer` function — no privilege required. Any user can call it with any ERC-20 token address. FOT tokens (e.g., tokens with built-in reflection or tax mechanisms) are common on EVM chains. The attacker does not need to exploit any external dependency; they simply supply a FOT token address and a large `amount`. The deficit compounds with every deposit.

---

### Recommendation

Replace the static `amount` with a before/after balance measurement for the non-bridge-token path:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use `received` (cast to uint128) in place of `amount` for all downstream accounting
```

The `received` value must then be propagated into `initTransferExtension` and the emitted event so the cross-chain message reflects the actual locked collateral.

---

### Proof of Concept

1. Deploy or use any ERC-20 token that deducts a 1% transfer fee.
2. Call `OmniBridge.initTransfer(tokenAddress, 1000e18, 0, 0, "near:recipient.near", "")`.
3. The bridge receives `990e18` tokens (`balanceOf(bridge)` increases by 990, not 1000).
4. The emitted `InitTransfer` event and the Wormhole VAA both carry `amount = 1000e18`.
5. The NEAR relayer submits the proof; `fin_transfer_callback` mints `1000e18` wrapped tokens to `recipient.near`.
6. The bridge now has a 10-token deficit per deposit. After N deposits the bridge is insolvent by `N × 10e18` tokens, and the last redeemers cannot withdraw.
