### Title
Wrong `receiver_id` used in `DeterministicStateInit` validation inside `DelegateAction` makes the feature permanently unusable from meta-transactions — (`runtime/runtime/src/action_validation.rs`)

---

### Summary

`validate_delegate_action()` passes the **outer** transaction's `receiver` to `validate_actions_with_mode()` instead of `delegate_action.receiver_id()` (the inner action's actual target). Because `validate_deterministic_state_init()` checks that the derived deterministic account ID equals the supplied `receiver_id`, and the outer receiver is virtually never a deterministic account, every `DeterministicStateInit` action wrapped inside a `DelegateAction` (meta-transaction) fails validation. The feature is completely broken for any unprivileged user who submits such a meta-transaction.

---

### Finding Description

`validate_delegate_action()` is called for every `Action::Delegate` and `Action::DelegateV2` action. It must validate the inner actions of the delegate payload against the **inner** receiver (the account the delegate action targets), not the outer transaction receiver.

The buggy path:

```rust
fn validate_delegate_action(
    limit_config: &LimitConfig,
    delegate_action: VersionedDelegateActionRef<'_>,
    receiver: &AccountId,          // ← outer tx receiver
    current_protocol_version: ProtocolVersion,
    mode: ValidateReceiptMode,
) -> Result<(), ActionsValidationError> {
    let actions = delegate_action.get_actions();
    let inner_receiver =
        if ProtocolFeature::FixDelegatedDeterministicStateInit.enabled(current_protocol_version) {
            delegate_action.receiver_id()   // correct
        } else {
            receiver                        // BUG: outer receiver used
        };
    validate_actions_with_mode(limit_config, &actions, inner_receiver, ...)?;
    Ok(())
}
``` [1](#0-0) 

`validate_actions_with_mode()` dispatches to `validate_deterministic_state_init()` for `Action::DeterministicStateInit`:

```rust
fn validate_deterministic_state_init(
    limit_config: &LimitConfig,
    action: &DeterministicStateInitAction,
    receiver_id: &AccountId,
) -> Result<(), ActionsValidationError> {
    let derived_id = derive_near_deterministic_account_id(&action.state_init);
    if derived_id != *receiver_id {          // ← compared against outer receiver
        return Err(ActionsValidationError::InvalidDeterministicStateInitReceiver { ... });
    }
    ...
}
``` [2](#0-1) 

A legitimate meta-transaction has:
- `outer_tx.receiver_id` = the relayer's target (e.g. `relayer.near`)
- `delegate_action.receiver_id` = the deterministic account (e.g. `0s69284a5453e7be5632b28b6a01baecf6c12c156d`)

The check `derived_id != relayer.near` always fails, so the transaction is rejected at the mempool admission stage before it ever reaches execution.

The codebase itself documents this as a bug:

> "This is a bug fixed with `FixDelegatedDeterministicStateInit` that validated against the wrong id. This makes it impossible to initialize deterministic accounts from meta transactions." [3](#0-2) 

The fix is gated behind `ProtocolFeature::FixDelegatedDeterministicStateInit`, which is listed near the end of the feature enum and has not yet been assigned a stable protocol version in this snapshot of the codebase. [4](#0-3) 

---

### Impact Explanation

Any unprivileged user who submits a `DelegateAction` (meta-transaction / relayed transaction) whose inner payload contains `DeterministicStateInit` will have their transaction rejected at the `validate_verify_and_charge_transaction` stage with `ActionsValidationError::InvalidDeterministicStateInitReceiver`. The entire `DeterministicStateInit`-via-meta-transaction feature (NEP-616 sharded contracts) is non-functional. The corrupted protocol value is the **action validation result** for this action class: it is always `Err` when it should be `Ok`.

---

### Likelihood Explanation

Any user who attempts to create a deterministic account through a meta-transaction (a standard, publicly documented pattern) triggers the bug. No special privileges are required. The entry path is a normal signed transaction submitted via public RPC.

The integration test `test_deterministic_state_init_meta_tx_receiver_check_pre_fix` confirms the pre-fix behavior: [5](#0-4) 

---

### Recommendation

Replace `receiver` with `delegate_action.receiver_id()` unconditionally in `validate_delegate_action()`, removing the protocol-version branch entirely (or accelerating the activation of `FixDelegatedDeterministicStateInit`). The correct value is already identified in the fix branch:

```rust
let inner_receiver = delegate_action.receiver_id();
```

---

### Proof of Concept

1. User constructs `state_init` for a deterministic account `det_account = derive_near_deterministic_account_id(&state_init)`.
2. User wraps `Action::DeterministicStateInit { state_init }` inside a `DelegateAction { receiver_id: det_account, ... }`.
3. A relayer submits the outer `SignedTransaction { receiver_id: relayer_target, actions: [Delegate(...)] }`.
4. `validate_delegate_action()` is called with `receiver = relayer_target`.
5. `validate_actions_with_mode(..., inner_receiver = relayer_target, ...)` is called.
6. `validate_deterministic_state_init(state_init, relayer_target)` computes `derived_id = det_account` and checks `det_account != relayer_target` → `true` → returns `Err(InvalidDeterministicStateInitReceiver)`.
7. The transaction is rejected at mempool admission; the deterministic account is never created. [6](#0-5)

/// Set up the exploit scenario and return the result of submitting the exploit tx.
///
/// `det_account_b` is deployed as a deterministic account and given an access key so
/// it can act as meta_tx_sender. The exploit tx wraps `state_init_b` inside a delegate
/// action whose `receiver_id` is `det_account_a` (wrong target). With the fix this is
/// caught at tx validation; without it, tx validation passes but the receipt fails.
fn try_meta_tx_deterministic_receiver_exploit(
    protocol_version: ProtocolVersion,
) -> Result<FinalExecutionOutcomeView, InvalidTxError> {
    let mut env = TestEnv::setup_with_version(Balance::from_near(100), protocol_version);
    env.deploy_global_contract(GlobalContractDeployMode::AccountId);

    let (_state_init_a, det_account_a) = env.new_deterministic_account_with_data(small());
    let (state_init_b, det_account_b) = env.new_deterministic_account_with_data(big());
    assert_ne!(det_account_a, det_account_b);

    // Deploy det_account_b and add a full-access key so it can act as meta_tx_sender.
    let user_signer = create_user_test_signer(&env.user_account());
    let storage_balance = env.balance_for_storage(state_init_b.clone());
    let deploy_tx = SignedTransaction::deterministic_state_init(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        env.get_tx_block_hash(),
        state_init_b.clone(),
        storage_balance,
    );
    env.run_tx(deploy_tx);

    let meta_tx_sender_signer = create_user_test_signer(&det_account_b);
    let pk_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&meta_tx_sender_signer.public_key()).unwrap(),
    );
    let add_key_args = serde_json::json!([
        { "batch_create": { "account_id": det_account_b.as_str() }, "id": 0 },
        {
            "action_add_key_with_full_access": {
                "promise_index": 0,
                "public_key": pk_base64,
                "nonce": 0
            },
            "id": 0,
            "return": true
        }
    ]);
    let add_key_tx = SignedTransaction::call(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        Balance::from_near(2),
        "call_promise".to_owned(),
        serde_json::to_vec(&add_key_args).unwrap(),
        Gas::from_teragas(300),
        env.get_tx_block_hash(),
    );
    env.run_tx(add_key_tx);

    // Craft the exploit: outer_tx.receiver = det_account_b = derive(state_init_b).
    // Old check: det_account_b == derive(state_init_b) passes.
    // The delegate action targets det_account_a, which is the wrong account.
    // In no protocol version can this ever be allowed to be executed successfully.
    let relayer = env.independent_account();
    let relayer_signer = create_user_test_signer(&relayer);
    let inner_action = Action::DeterministicStateInit(Box::new(DeterministicStateInitAction {
        state_init: state_init_b,
        deposit: Balance::ZERO,
    }));
    let delegate_nonce = env.next_nonce_for(&det_account_b);
    let delegate_action = DelegateAction {
        sender_id: det_account_b.clone(),
        receiver_id: det_account_a,
        actions: vec![NonDelegateAction::try_from(inner_action).unwrap()],
        nonce: delegate_nonce,
        max_block_height: 1_000_000,
        public_key: meta_tx_sender_signer.public_key(),
    };
    let signed_delegate_action =
        SignedDelegateAction::sign(&meta_tx_sender_signer, delegate_action);
    let tx = SignedTransaction::from_actions(
        env.next_nonce(),
        relayer,
        det_account_b,
        &relayer_signer,
        vec![Action::Delegate(Box::new(signed_delegate_action))],
        env.get_tx_block_hash(),
    );
    env.try_execute_tx(tx)
}
