Audit Report

## Title
Polynomial Gas Complexity in `updateRSETHPrice()` via Nested Loops Over Assets × NDCs × Queued Withdrawals × Strategies — (`contracts/LRTOracle.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is a public, permissionless function whose gas cost scales as O(assets × NDCs × queued-withdrawals × strategies) due to nested iteration across live EigenLayer state. The protocol's own code acknowledges the function breaks above ~120 uncompleted withdrawals, but the per-asset multiplication means the effective safe limit is far lower than the team's comment implies. If the function consistently reverts out-of-gas, `rsETHPrice` becomes permanently stale, breaking all deposit minting and withdrawal accounting.

## Finding Description

**Call chain (confirmed in code):**

`updateRSETHPrice()` (public, `whenNotPaused`) calls `_updateRsETHPrice()`, which calls `_getTotalEthInProtocol()`.

`_getTotalEthInProtocol()` opens **Loop 1** over `supportedAssets` and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` per asset: [1](#0-0) 

`getTotalAssetDeposits` delegates to `getAssetDistributionData`, which opens **Loop 2** over `nodeDelegatorQueue` and calls `getAssetUnstaking(asset)` per NDC: [2](#0-1) 

For ETH, `getAssetDistributionData` delegates to `getETHDistributionData()`, which also calls `getAssetUnstaking(ETH_TOKEN)` per NDC — adding another full NDC-loop pass: [3](#0-2) 

`NodeDelegator.getAssetUnstaking` opens **Loops 3 and 4**: it fetches all queued withdrawals from EigenLayer's `DelegationManager` (one external call per invocation) and iterates over every strategy inside each withdrawal: [4](#0-3) 

**Why existing guards are insufficient:**

`maxUncompletedWithdrawalCount` is capped at 80 by `setMaxUncompletedWithdrawalCount`, with the comment explicitly acknowledging the function breaks above ~120: [5](#0-4) 

However, this cap is a *global* count across all NDCs, and `getAssetUnstaking` is called **N × M** times (once per asset per NDC). The total inner-loop iterations are `N × total_withdrawals × S`, not just `total_withdrawals`. With 5 assets, 10 NDCs, 80 withdrawals, and 5 strategies, this is 5 × 80 × 5 = 2,000 inner iterations plus 50 external `getQueuedWithdrawals` calls — all in a single transaction.

`maxNodeDelegatorLimit` has no hard upper bound; the admin can raise it arbitrarily: [6](#0-5) 

Furthermore, EigenLayer operator-forced undelegations create queued withdrawals directly in `DelegationManager` without going through the protocol's `increaseUncompletedWithdrawalCount` path, so the actual on-chain withdrawal count can exceed `maxUncompletedWithdrawalCount`. The team's own comment acknowledges this: *"Need buffer for theoretical operator forced undelegations (ndc count * asset count = 15)"*. [7](#0-6) 

## Impact Explanation

**Medium — Unbounded gas consumption.** If `updateRSETHPrice()` consistently reverts out-of-gas, `rsETHPrice` is never updated. Every deposit (`getRsETHAmountToMint`) and every withdrawal (`getExpectedAssetAmount`) reads the stale stored price: [8](#0-7) 

Users minting or completing withdrawals receive incorrect rsETH amounts based on the last successfully stored price, constituting incorrect share/asset accounting for all users. The secondary effect of a permanently stale price also matches **Low — Contract fails to deliver promised returns**, but the primary classification is Medium due to the gas exhaustion vector.

## Likelihood Explanation

`updateRSETHPrice()` is public and requires no privileges. The gas cost is determined entirely by protocol state (number of supported assets, NDCs, and EigenLayer queued withdrawals) that grows with normal protocol operation. No attacker action is required — the protocol reaching its own configured limits is sufficient. Forced undelegations by EigenLayer operators (an external, non-protocol party) can push the withdrawal count above the protocol's own cap without any admin action. The team's comment confirms awareness that the function breaks above ~120 withdrawals, and the per-asset multiplication means the true breaking point is lower than that figure.

## Recommendation

1. **Decouple unstaking accounting from live EigenLayer queries**: Maintain an on-chain `assetUnstaking[asset]` accumulator updated at queue/complete time (in `initiateUnstaking`, `undelegate`, `completeUnstaking`) instead of re-deriving it from `getQueuedWithdrawals` on every price update. This eliminates Loops 3 and 4 entirely from the price update path.
2. **Consolidate NDC iteration**: `getAssetDistributionData` and `getETHDistributionData` both loop over `nodeDelegatorQueue` independently. A single pass per NDC that accumulates all asset balances would halve the NDC-loop count.
3. **Hard-cap `maxNodeDelegatorLimit`**: Add an enforced upper bound (e.g., 10) in `updateMaxNodeDelegatorLimit` analogous to the cap on `maxUncompletedWithdrawalCount`.
4. **Account for the per-asset multiplier in the withdrawal cap**: The comment's "120 withdrawals" threshold does not account for the N-asset multiplication. The effective safe limit is `120 / N_assets`, which should be reflected in the cap logic.

## Proof of Concept

**Minimal call sequence (no privileges required):**

```
// Any EOA calls:
LRTOracle.updateRSETHPrice()
  → _updateRsETHPrice()
    → _getTotalEthInProtocol()
      → for each asset in supportedAssetList:              // Loop 1: N assets
          LRTDepositPool.getTotalAssetDeposits(asset)
            → getAssetDistributionData(asset)
              → for each NDC in nodeDelegatorQueue:        // Loop 2: M NDCs
                  NodeDelegator.getAssetUnstaking(asset)
                    → DelegationManager.getQueuedWithdrawals(ndc)  // external call
                    → for each queued withdrawal:          // Loop 3: W withdrawals
                        for each strategy:                 // Loop 4: S strategies
                            strategy.sharesToUnderlyingView(shares)
```

**Foundry fork test plan:**

1. Fork mainnet at a block where the protocol has ≥3 supported assets and ≥5 NDCs.
2. Use `vm.store` to set `uncompletedWithdrawalCount` to 75 and `maxUncompletedWithdrawalCount` to 80.
3. Queue 15 withdrawals across 5 NDCs (3 per NDC, each with 5 strategies) via `initiateUnstaking`.
4. Call `LRTOracle.updateRSETHPrice()` with `gas: 30_000_000` and assert it does not revert.
5. Repeat with 80 total withdrawals across 10 NDCs and 5 assets; assert the call reverts with out-of-gas.
6. Alternatively, use `vm.mockCall` to simulate `getQueuedWithdrawals` returning 80 withdrawals per NDC for each of 10 NDCs, and measure gas via `gasleft()` before and after the call.

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
