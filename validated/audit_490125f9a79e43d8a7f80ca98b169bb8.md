Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a Synthetix-style reward accumulator where `updatedAt` always advances to `lastTimeRewardApplicable()` in the `updateReward` modifier, even when `totalKernelStaked == 0` and `rewardPerTokenStored` cannot increase. Any time interval during which no tokens are staked permanently consumes the reward clock without attributing rewards to anyone, locking `rewardRate × gap_seconds` tokens in the contract with no recovery path. The contract's own NatSpec explicitly acknowledges this and relies solely on off-chain operational guarantees.

## Finding Description
The `updateReward` modifier unconditionally sets `updatedAt = lastTimeRewardApplicable()` on every state-changing call:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // frozen when totalKernelStaked == 0
    updatedAt = lastTimeRewardApplicable();    // always advances
    ...
}
``` [1](#0-0) 

`rewardPerToken()` returns `rewardPerTokenStored` unchanged when supply is zero, so the time delta is consumed without any accumulation:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    ...
}
``` [2](#0-1) 

`initiateWithdrawal` applies `updateReward(msg.sender)` before decrementing `totalKernelStaked`, so the last staker's rewards are correctly captured up to the withdrawal moment. After the decrement, `totalKernelStaked == 0`. Any subsequent call to `updateReward` (e.g., a new staker calling `stake`) will advance `updatedAt` to the current time without increasing `rewardPerTokenStored`, permanently discarding all rewards for the empty interval: [3](#0-2) 

The `notifyRewardAmount` guard only prevents starting a new period with zero supply; it does not prevent all stakers from exiting mid-period: [4](#0-3) 

The contract's own NatSpec confirms the unmitigated risk and the reliance on off-chain coordination: [5](#0-4) 

There is no sweep, rescue, or redistribution function for stranded reward tokens.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens deposited by the admin via `notifyRewardAmount` and intended for stakers are permanently locked in the contract for any interval where `totalKernelStaked == 0`. No on-chain recovery mechanism exists. The magnitude is `rewardRate × zero_supply_duration`, which can be substantial over a multi-day or multi-week reward window.

## Likelihood Explanation
Any staker can call `initiateWithdrawal` at any time during an active reward period — no privileged access is required. A coordinated or organic full exit (e.g., due to a better yield opportunity, market downturn, or loss of confidence) reduces `totalKernelStaked` to zero. The contract developers explicitly acknowledge this scenario is possible and are relying on social/operational coordination rather than on-chain enforcement to prevent it.

## Recommendation
Freeze the reward clock when supply is zero. In `rewardPerToken()`, also return early without updating `updatedAt` when `totalKernelStaked == 0`. Concretely, move the `updatedAt` assignment out of the modifier and into a separate helper that only advances `updatedAt` when `totalKernelStaked > 0`, or store a `lastNonZeroTimestamp` and use it as the lower bound of the accumulator calculation. This is the canonical fix applied in referenced Synthetix-style audits and preserves full user freedom to withdraw at any time.

## Proof of Concept
```
Setup:
  duration = 10,000s, rewardAmount = 100e18, Alice stakes 1e18 KERNEL
  Admin calls notifyRewardAmount(100e18)
  → finishAt = T+10000, updatedAt = T, rewardRate = 1e16/s

T+5000: Alice calls initiateWithdrawal(1e18)
  → updateReward(Alice): rewardPerTokenStored += 5e19, updatedAt = T+5000
  → totalKernelStaked = 0

T+10000: Bob calls stake(1e18)
  → updateReward(Bob): totalKernelStaked==0 → rewardPerTokenStored unchanged (5e19)
  → updatedAt = T+10000 (gap consumed)
  → userRewardPerTokenPaid[Bob] = 5e19

Bob calls getReward(): earned(Bob) = 1e18*(5e19-5e19)/1e18 = 0
Alice calls getReward(): rewards[Alice] = 5e19 (first half only)

Result: 50e18 tokens permanently locked, unclaimable by any address.
```

Foundry invariant test plan: assert that `rewardsToken.balanceOf(address(pool)) >= sum of all claimable rewards` never holds false after a full-exit/re-entry sequence within an active reward window.
