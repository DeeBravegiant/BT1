### Title
Malicious peer can set `n_transactions=0` in `SignedBlockHeader` to force empty `BlockBody` storage — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary
`HeaderStreamBuilder::parse_data_for_block` accepts any `n_transactions` value from a peer without validation. `TransactionStreamFactory::parse_data_for_block` uses that stored value as the loop bound, so `n_transactions=0` causes an empty `BlockBody` to be committed for a block that actually contains transactions.

### Finding Description
The header sync path stores a peer-supplied `SignedBlockHeader` verbatim after only checking block number ordering and signature vector length. The `n_transactions` field is never validated against any trusted source (L1, block hash, or `concatenated_counts`). The transaction sync path then reads `n_transactions` from the stored header as `target_transaction_len` and uses it as the exclusive upper bound of the collection loop. When `target_transaction_len == 0`, the loop body is skipped entirely and `Ok(Some((BlockBody::default(), block_number)))` is returned, which `write_to_storage` commits via `append_body`. The actual `FullTransaction` messages sent by the peer remain unconsumed in the stream, potentially corrupting subsequent block parsing.

### Impact Explanation
The syncing node permanently stores an empty `BlockBody` for a block that has real transactions. All RPC queries for that block's transactions return empty results. The p2p sync server re-serves the corrupted empty body to downstream peers. The stored `transaction_commitment` and `receipt_commitment` in the header (peer-supplied) are inconsistent with the empty body, but no post-storage check detects this.

### Likelihood Explanation
Any peer that can establish a p2p sync connection can trigger this. No special privileges are required. The peer only needs to send a syntactically valid `SignedBlockHeader` with `n_transactions=0` and a valid-length signature vector.

### Recommendation
In `HeaderStreamBuilder::parse_data_for_block`, validate `n_transactions` against the `concatenated_counts` field (as the TODO at line 140–141 already acknowledges). Alternatively, in `TransactionStreamFactory::parse_data_for_block`, after collecting transactions, verify that the count matches the header's `n_transactions` and that the transaction hashes produce a commitment matching the header's `transaction_commitment` before calling `write_to_storage`.

### Proof of Concept
1. Connect a malicious peer to a syncing node.
2. Send a `SignedBlockHeader` for block 0 with `n_transactions=0`, one valid signature, and arbitrary `transaction_commitment`/`receipt_commitment` values.
3. Send a `DataOrFin(None)` (Fin) for the transaction stream.
4. Assert `get_block_transactions(BlockNumber(0))` returns an empty vec.
5. Assert the stored `transaction_commitment` in the header does not match `calculate_transaction_commitment` of the actual transactions for that block.
