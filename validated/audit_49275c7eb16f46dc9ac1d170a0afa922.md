Audit Report

## Title
Missing Chainlink Oracle Staleness Check Enables Stale Price Exploitation During Deposit - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all staleness-detection fields, accepting any price regardless of age. This stale price feeds directly into rsETH minting calculations via `LRTDepositPool.depositAsset()`, allowing a depositor to receive excess rsETH when a Chainlink feed is stale with an inflated price, diluting existing rsETH holders' claims on the underlying pool.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol`, `getAssetPrice()` fetches the Chainlink price but discards all staleness fields:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The `updatedAt` timestamp and `answeredInRound` fields are completely ignored. There is no `block.timestamp - updatedAt > MAX_STALENESS` check and no `answeredInRound >= roundId` check.

This is in direct contrast to `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`, which ships in the same codebase and performs partial staleness validation:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
``` [2](#0-1) 

The stale price propagates through the following call chain:

1. `LRTOracle.getAssetPrice(asset)` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)` [3](#0-2) 
2. `LRTDepositPool.getRsETHAmountToMint()` computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` [4](#0-3) 
3. `LRTDepositPool.depositAsset()` is publicly callable by any user [5](#0-4) 

The `pricePercentageLimit` guard in `_updateRsETHPrice()` does not protect the deposit path: `depositAsset()` uses the stored `rsETHPrice` directly and never calls `updateRSETHPrice()`. An attacker exploits the window where the Chainlink feed is stale-inflated but `rsETHPrice` still reflects the true value, minting excess rsETH at the expense of existing holders. [6](#0-5) 

## Impact Explanation
**Critical — Direct theft of user funds.** When a Chainlink LST/ETH feed goes stale with a price above the true market price, an attacker deposits the LST and receives `(amount * stalePriceInflated) / rsETHPrice` rsETH — more than the deposited assets are worth. This excess rsETH represents a dilution of all existing rsETH holders' proportional claims on the underlying ETH pool. When the attacker later redeems their rsETH through the withdrawal mechanism, they extract more ETH than they deposited, at the direct expense of other holders. This is a concrete, quantifiable theft of user funds, not merely a theoretical accounting error.

## Likelihood Explanation
**Medium.** Chainlink LST/ETH feeds (e.g., stETH/ETH) have heartbeat periods up to 24 hours and only update on a 0.5% deviation threshold. During network congestion, oracle node downtime, or low-volatility periods, feeds can remain at a stale price for extended periods. An attacker monitoring on-chain `updatedAt` values can detect when the last reported price is above the true market price and execute the deposit atomically. No privileged access is required; the attack is repeatable whenever the staleness condition recurs.

## Recommendation
Add staleness validation in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Where `MAX_STALENESS` is a configurable per-feed constant set to slightly above the feed's documented heartbeat period.

## Proof of Concept
**Exploit path:**
1. Chainlink stETH/ETH feed goes stale; last reported price = 1.05 ETH (true price = 1.00 ETH). `updatedAt` is 25+ hours old.
2. `rsETHPrice` was last updated when prices were accurate (e.g., 1.00 ETH per rsETH).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint` computes `(1000e18 * 1.05e18) / 1e18 = 1050 rsETH` — 50 rsETH more than deserved.
5. Attacker redeems 1050 rsETH through the withdrawal manager, extracting ~1050 ETH worth of value while only depositing ~1000 ETH worth of stETH.
6. The ~50 ETH difference is extracted from existing rsETH holders' pool share.

**Foundry test plan:**
- Deploy `ChainlinkPriceOracle` with a mock Chainlink feed.
- Mock `latestRoundData()` to return `updatedAt = block.timestamp - 2 days` and `answeredInRound = roundId - 1`.
- Call `getAssetPrice()` and assert it returns the stale price without reverting.
- In a fork test: advance time past the heartbeat, verify `depositAsset()` mints excess rsETH compared to a deposit made with a fresh price, and confirm the rsETH/ETH ratio is diluted for pre-existing holders.
