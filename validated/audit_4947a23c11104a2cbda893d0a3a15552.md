Audit Report

## Title
Sequential Integer Division Truncation Causes Systematic Treasury Fee Underpayment - (File: contracts/LRTOracle.sol)

## Summary
`_updateRsETHPrice()` computes the protocol fee mint amount through three chained integer divisions. Each division truncates its remainder, and both truncation errors act in the same direction on every invocation, guaranteeing the treasury is always minted fewer rsETH tokens than the exact mathematical fee entitles it to. The per-call shortfall is bounded by a small number of wei but is strictly monotonic and accumulates over the protocol's lifetime.

## Finding Description
The fee computation in `_updateRsETHPrice()` proceeds in three steps, each introducing a downward truncation error:

**Step 1** — `protocolFeeInETH` is truncated downward:
```solidity
// contracts/LRTOracle.sol L246
protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
```
Let F = exact fee, F' = floor(F). Then F' ≤ F, with a deficit of up to `(10_000 − 1) / 10_000` wei.

**Step 2** — `newRsETHPrice` is computed using the already-truncated `protocolFeeInETH`:
```solidity
// contracts/LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```
Because F' ≤ F, the subtrahend is too small, so `(totalETHInProtocol − F')` ≥ `(totalETHInProtocol − F)`, inflating `newRsETHPrice` above its true value.

**Step 3** — `rsethAmountToMintAsProtocolFee` divides the deflated numerator by the inflated denominator:
```solidity
// contracts/LRTOracle.sol L301
uint256 rsethAmountToMintAsProtocolFee = protocolFeeInETH.divWad(newRsETHPrice);
```
Both errors compound in the same direction: numerator too small, denominator too large → minted amount is strictly less than `F * rsethSupply / (totalETHInProtocol − F)` on every call where rewards have accrued.

The same two-division pattern appears in `RSETHPool.viewSwapRsETHAmountAndFee` (L311–319) and `RSETHPoolV3.viewSwapRsETHAmountAndFee` (L299–307), where `fee` is truncated downward and the inflated `amountAfterFee` is then divided by the rate, causing depositors to receive marginally more rsETH than they should at the expense of the fee pool.

No existing guard prevents this: the `_checkAndUpdateDailyFeeMintLimit` check (L303) only enforces an upper cap on minting; it does not correct the downward truncation.

## Impact Explanation
On every `updateRSETHPrice()` call where `totalETHInProtocol > previousTVL`, the treasury is minted fewer rsETH tokens than the protocol's fee percentage entitles it to. The per-call shortfall is at most a few wei, but it is strictly one-directional and accumulates monotonically. No user funds are at risk; the loss is borne entirely by the protocol treasury in the form of permanently uncollectable protocol yield. This matches the allowed Low impact: **Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
`updateRSETHPrice()` is a public, permissionless function callable by any address. It is expected to be invoked frequently by keepers or any participant. Every invocation where staking rewards have accrued (i.e., `totalETHInProtocol > previousTVL`) triggers the three-step division chain. No special role or precondition is required beyond the protocol being unpaused and TVL having grown. Likelihood is **High**.

## Recommendation
Consolidate all multiplications before the final division in `_updateRsETHPrice()` to eliminate intermediate truncations. Compute `rsethAmountToMintAsProtocolFee` directly from primary inputs in a single formula:

```solidity
// Multiply first, divide once
uint256 rsethAmountToMintAsProtocolFee =
    (rewardAmount * lrtConfig.protocolFeeInBPS() * rsethSupply)
    / (10_000 * (totalETHInProtocol - (rewardAmount * lrtConfig.protocolFeeInBPS() / 10_000)));
```

For `RSETHPool` and `RSETHPoolV3`, replace the two-step fee subtraction with a single combined division:
```solidity
rsETHAmount = amount * (10_000 - feeBps) * 1e18 / (10_000 * rsETHToETHrate);
```

## Proof of Concept
Using the values from the report:
- `rewardAmount = 1e18 + 3`, `protocolFeeInBPS = 1000`, `totalETHInProtocol = 100_001e18 + 3`, `rsethSupply = 95_000e18`

**Current (truncated):**
- F' = `((1e18 + 3) * 1000) / 10_000 = 1e17` (3000 wei truncated → 0)
- `newRsETHPrice = (100_001e18 + 3 − 1e17).divWad(95_000e18)` → inflated by the missing 0.3 wei
- `rsethAmountToMintAsProtocolFee = 1e17.divWad(inflated_price)` → strictly less than correct value

**Correct (exact):**
- True F = `1e17 + 0.3 wei` → rounds to `1e17 + 1` after proper rounding
- `rsethAmountToMintAsProtocolFee` = `(1e17 + 1) * 95_000e18 / (100_001e18 + 3 − (1e17 + 1))` → 1–2 wei more than current

A Foundry fuzz test parameterizing `rewardAmount`, `feeBPS`, `totalETHInProtocol`, and `rsethSupply` and asserting `mintedAmount_current ≤ mintedAmount_exact` will confirm the invariant holds universally, and `mintedAmount_current < mintedAmount_exact` whenever `(rewardAmount * feeBPS) % 10_000 != 0`.
