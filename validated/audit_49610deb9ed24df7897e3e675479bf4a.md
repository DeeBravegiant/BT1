### Title
User-Settable Finality Threshold Bypasses Foreign-Chain Verification Safety — (`File: crates/near-mpc-contract-interface/src/types/foreign_chain.rs`, `crates/contract/src/lib.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

### Summary

The `verify_foreign_transaction` endpoint accepts user-supplied finality parameters (`confirmations` for Bitcoin, `finality` for EVM chains) that are used verbatim as the security threshold during MPC node verification. No on-chain policy enforces a minimum finality requirement. An unprivileged caller can set `confirmations: 1` (or `EvmFinality::Latest`) to obtain a valid MPC-signed attestation for a foreign-chain transaction that has not actually finalized, enabling double-spend attacks in bridge scenarios.

### Finding Description

`BitcoinRpcRequest` carries a user-supplied `confirmations: BlockConfirmations` field, and `EvmRpcRequest` carries a user-supplied `finality: EvmFinality` field. Both are part of the public `VerifyForeignTransactionRequestArgs` struct accepted by `verify_foreign_transaction()`. [1](#0-0) 

The contract's `verify_foreign_transaction()` validates only that the `domain_id` maps to a `ForeignTx` domain and that the chain is in the supported set. It performs **no validation** of the user-supplied finality threshold against any on-chain policy: [2](#0-1) 

The `args_into_verify_foreign_tx_request` conversion passes the user-supplied values through unchanged: [3](#0-2) 

On the node side, `BitcoinInspector::extract()` uses the caller-supplied `block_confirmations_threshold` directly as the acceptance threshold: [4](#0-3) 

The signed payload is `ForeignTxSignPayload::V1 { request, values }` where `request` is the full `ForeignChainRpcRequest` including the attacker-chosen `confirmations` value. The MPC network signs this without any floor check. [5](#0-4) 

### Impact Explanation

This is a **forged foreign-chain verification** issue. A bridge contract consuming the MPC attestation receives a valid threshold signature over `(request={confirmations:1}, observed_values=[block_hash])`. If the bridge contract does not independently re-validate the `confirmations` field (which it has no reason to, since the MPC network is the trusted oracle), it will process the inbound bridge transfer. If the Bitcoin transaction is subsequently reorganized out of the canonical chain, the attacker has already received funds on NEAR while the Bitcoin transaction no longer exists — a classic double-spend.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass … that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

- The entry path is fully unprivileged: any account can call `verify_foreign_transaction()` with a 1 yoctoNEAR deposit.
- Bitcoin 1-confirmation transactions are routinely reorganized; 6 confirmations is the industry standard for finality.
- The Omnibridge inbound flow is the primary stated use case, making this directly exploitable in production.
- No existing contract-level guard prevents `confirmations: 0` or `confirmations: 1`.

### Recommendation

Store a per-chain minimum finality policy in the on-chain foreign chain configuration (the `ChainEntry` / `ForeignChainsConfig` already voted in by participants). During `verify_foreign_transaction()`, retrieve the minimum required confirmations (or minimum finality level for EVM) from that on-chain record and reject any request whose user-supplied value falls below it — analogous to how `usdt.decimals()` should be read from the token contract rather than accepted from the caller.

### Proof of Concept

1. Operator votes in Bitcoin as a supported foreign chain via `register_foreign_chains_config`.
2. Attacker broadcasts a Bitcoin transaction `T` and waits for 1 confirmation.
3. Attacker calls:
   ```json
   verify_foreign_transaction({
     "request": {
       "Bitcoin": {
         "tx_id": "<T>",
         "confirmations": 1,
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": 1
   })
   ```
   with 1 yoctoNEAR deposit.
4. MPC nodes call `BitcoinInspector::extract(T, threshold=1, ...)`. The check `1 <= 1` passes.
5. Nodes sign `ForeignTxSignPayload::V1 { request: Bitcoin{tx_id:T, confirmations:1, ...}, values: [BlockHash(H)] }` and call `respond_verify_foreign_tx`.
6. The NEAR bridge contract receives a valid MPC signature and releases NEAR-side funds.
7. Attacker double-spends `T` on Bitcoin (e.g., via a selfish-mining reorg or RBF on an unconfirmed parent). The MPC attestation remains valid on NEAR while the Bitcoin transaction no longer exists on the canonical chain.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
