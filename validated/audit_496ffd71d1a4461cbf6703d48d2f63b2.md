### Title
Sub-Transaction Execution Units Bypass `maxTxExUnits` and `maxBlockExUnits` Limits - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, sub-transactions embedded inside a top-level transaction are validated through the `SUBUTXO` rule, which does **not** call `validateExUnitsTooBigUTxO`. Additionally, the block-level `validateExUnits` check in `dijkstraBbodyTransition` only sums `totExUnits` over top-level transactions, completely ignoring sub-transaction execution units. An unprivileged transaction sender can therefore embed sub-transactions with arbitrarily large declared Plutus execution unit budgets, bypassing both the per-transaction (`maxTxExUnits`) and per-block (`maxBlockExUnits`) resource limits. This allows a single block to contain far more Plutus script execution work than the protocol parameters intend to permit.

---

### Finding Description

**Vulnerability class:** Resource-limit bypass — a limit check that should apply to all code paths that trigger Plutus script execution is only enforced on the top-level transaction path, not on the sub-transaction path.

**Analogous pattern to the CLOB report:** Just as `amend()` bypasses `incrementLimitsPlaced()` by creating new order book entries through a different code path, Dijkstra sub-transactions bypass execution unit limits by going through `SUBUTXO` instead of the main `UTXO` rule.

#### Path 1 — Top-level transaction (limit IS enforced)

In `dijkstraUtxoTransition` (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`):

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
``` [1](#0-0) 

This correctly rejects a top-level transaction whose declared redeemer execution units exceed `maxTxExUnits`.

#### Path 2 — Sub-transaction (limit is NOT enforced)

In `dijkstraSubUtxoTransition` (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`), the entire transition rule runs without any call to `validateExUnitsTooBigUTxO`: [2](#0-1) 

There is no `maxTxExUnits` check anywhere in this transition. Sub-transactions can declare redeemers with execution units of any magnitude.

#### Path 3 — Block-level check also misses sub-transaction execution units

In `dijkstraBbodyTransition` (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`):

```haskell
let txs = blockBody ^. txSeqBlockBodyL
...
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
``` [3](#0-2) 

`validateExUnits` is defined as:

```haskell
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax ...
``` [4](#0-3) 

`totExUnits tx` sums only the redeemers in the top-level transaction's own witness set. Sub-transactions carry their own witness sets with their own redeemers, which `totExUnits` never touches. The block-level budget check therefore sees only the top-level execution units and is blind to all sub-transaction execution units.

#### Contrast: Reference script size IS correctly aggregated

The reference script size check uses `batchNonDistinctRefScriptsSize`, which explicitly aggregates across sub-transactions:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
``` [5](#0-4) 

This asymmetry confirms the execution unit aggregation is an oversight, not a design choice.

---

### Impact Explanation

**Allowed impact category:** *Medium — Attacker-controlled transactions exceed intended validation limits.*

The `maxTxExUnits` and `maxBlockExUnits` protocol parameters exist to bound the CPU and memory cost of Plutus script evaluation per transaction and per block, protecting nodes from excessive validation work. By embedding sub-transactions with large declared execution unit budgets, an attacker can force nodes to run the Plutus interpreter with a total budget far exceeding what the protocol parameters permit. A single block could contain execution work equivalent to many multiples of `maxBlockExUnits`, causing all honest nodes to spend disproportionate CPU and memory validating that block. This degrades block validation throughput and can cause nodes to fall behind the chain tip under sustained attack.

---

### Likelihood Explanation

Any unprivileged transaction sender can submit a Dijkstra-era top-level transaction containing sub-transactions. No special keys, governance roles, or privileged access are required. The attacker only needs to:
1. Construct a top-level transaction with a minimal (valid) redeemer budget.
2. Embed N sub-transactions each with redeemers declaring execution units near or above `maxTxExUnits`.
3. Submit the transaction to the mempool.

The attack is cheap (only normal transaction fees apply to the top-level tx) and repeatable across blocks.

---

### Recommendation

1. **Add a per-sub-transaction execution unit check** in `dijkstraSubUtxoTransition` analogous to the check in `dijkstraUtxoTransition`:
   ```haskell
   runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
   ```

2. **Fix the block-level check** in `dijkstraBbodyTransition` to aggregate execution units across all sub-transactions. Define a `batchTotExUnits` function analogous to `batchNonDistinctRefScriptsSize`:
   ```haskell
   batchTotExUnits tx =
     totExUnits tx
       <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
   ```
   and use it in `validateExUnits` at the BBODY level.

---

### Proof of Concept

1. Set `maxTxExUnits = ExUnits { exUnitsMem = 14_000_000, exUnitsSteps = 10_000_000_000 }` and `maxBlockExUnits = ExUnits { exUnitsMem = 62_000_000, exUnitsSteps = 20_000_000_000 }` (current mainnet values).

2. Construct a Dijkstra top-level transaction with:
   - A trivial top-level redeemer with `ExUnits 1 1` (well under `maxTxExUnits`).
   - 10 sub-transactions each with a redeemer declaring `ExUnits { exUnitsMem = 14_000_000, exUnitsSteps = 10_000_000_000 }` (equal to `maxTxExUnits`).

3. Submit the transaction. The top-level UTXO check passes (1 ≤ maxTxExUnits). The SUBUTXO rule runs for each sub-transaction without any execution unit check. The BBODY check sees only `ExUnits 1 1` from the top-level tx and passes.

4. The Plutus interpreter is invoked with a total budget of `ExUnits { 140_000_001, 100_000_000_001 }` — more than **5× the intended block limit** — for a single top-level transaction, with no predicate failure raised. [2](#0-1) [6](#0-5) [7](#0-6) [5](#0-4)

-- | Validate that total execution units (all transactions) do not exceed block limit.
-- ∑(tx ∈ txs)(totExunits tx) ≤ maxBlockExUnits pp
validateExUnits ::
  forall era.
  ( AlonzoEraTx era
  , InjectRuleFailure "BBODY" AlonzoBbodyPredFailure era
  ) =>
  StrictSeq.StrictSeq (Tx TopTx era) ->
  -- | Max block exunits protocol parameter.
  ExUnits ->
  Rule (EraRule "BBODY" era) 'Transition ()
validateExUnits txs ppMax =
  let txTotal = foldMap totExUnits txs
   in pointWiseExUnits (<=) txTotal ppMax
        ?! injectFailure
          ( TooManyExUnits $
              Mismatch
                { mismatchSupplied = txTotal
                , mismatchExpected = ppMax
                }
          )
