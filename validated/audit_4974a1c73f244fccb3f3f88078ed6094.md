### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Reward via Unvalidated `farmer_reward_address_override` in `RespondSignatures` - (File: `chia/farmer/farmer_api.py`)

### Summary
A malicious harvester can redirect the farmer's 1/8 block reward (XCH) to an arbitrary address by setting `farmer_reward_address_override` in a `RespondSignatures` message. The farmer's `_process_respond_signatures()` function accepts this field unconditionally, with no fee-quality check and no enforcement of the CHIP-22 convention. The only guard (`notify_farmer_reward_taken_by_harvester_as_fee`) is triggered only by the earlier `NewProofOfSpace` message, not by `RespondSignatures`, and even then it only logs a warning — it never rejects the override.

### Finding Description

CHIP-22 introduced `farmer_reward_address_override` as an opt-in fee mechanism: a harvester that meets a deterministic fee-quality threshold may redirect the farmer reward to itself as payment. The protocol defines two message fields carrying this override:

1. `harvester_protocol.NewProofOfSpace.farmer_reward_address_override` — sent when the harvester reports a proof.
2. `harvester_protocol.RespondSignatures.farmer_reward_address_override` — sent when the harvester returns signatures.

**Path 1 — `NewProofOfSpace` (partial guard, logging only):**

In `FarmerAPI.new_proof_of_space()`, when `farmer_reward_address_override` is set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

That function computes `fee_quality` and compares it to `fee_threshold`, but on failure it only emits a `log.warning` — it never returns a rejection signal and the block-building flow continues regardless: [2](#0-1) 

**Path 2 — `RespondSignatures` (no guard at all):**

In `_process_respond_signatures()`, the farmer unconditionally replaces its own configured `farmer_target` with whatever address the harvester supplies in `RespondSignatures.farmer_reward_address_override`: [3](#0-2) 

There is no fee-quality check here. A harvester can send `NewProofOfSpace` with `farmer_reward_address_override=None` (triggering no logging at all), wait for the farmer to request signatures, then return `RespondSignatures` with `farmer_reward_address_override=<attacker_address>`. The farmer accepts it silently.

The resulting `DeclareProofOfSpace` carries the attacker's address as `farmer_reward_address`: [4](#0-3) 

This address is embedded in the block's `FoliageBlockData.farmer_reward_puzzle_hash`, which determines where the farmer coinbase coin is created: [5](#0-4) 

The `RespondSignatures` protocol message definition confirms `farmer_reward_address_override` is an optional field the harvester controls: [6](#0-5) 

### Impact Explanation

Every block the farmer wins while connected to a malicious harvester results in the farmer's 1/8 block reward (currently 0.25 XCH per block) being paid to the attacker's puzzle hash instead of the farmer's configured address. The farmer receives nothing for those blocks. This is a direct, on-chain, irreversible XCH reward diversion.

### Likelihood Explanation

Any entity that can act as a harvester peer to a farmer can exploit this. Third-party harvester services are a supported and common deployment model (CHIP-22 was designed for them). A malicious harvester operator needs only to:
1. Accept connections from farmers (or connect to a farmer).
2. Return valid proofs of space with `farmer_reward_address_override=None` in `NewProofOfSpace`.
3. Return `RespondSignatures` with `farmer_reward_address_override` set to the attacker's address.

No key compromise, no admin access, and no cryptographic break is required. The farmer has no way to detect this without monitoring on-chain reward destinations.

### Recommendation

In `_process_respond_signatures()`, before accepting `response.farmer_reward_address_override`, enforce the CHIP-22 fee-quality convention: compute `fee_quality` from the stored proof and the signage point challenge, verify that `fee_quality <= response.fee_info.applied_fee_threshold`, and reject (return `None`) if the check fails or if `fee_info` is absent. The same enforcement should be applied in `new_proof_of_space()` — the current logging-only behavior in `notify_farmer_reward_taken_by_harvester_as_fee` must be upgraded to a hard rejection that prevents the block-building flow from continuing. [7](#0-6) 

### Proof of Concept

1. Attacker runs a harvester service and accepts a farmer connection (or connects to a farmer).
2. Harvester finds a valid proof of space for a signage point.
3. Harvester sends `NewProofOfSpace` with `farmer_reward_address_override=None` and `fee_info=None` — no warning is logged.
4. Farmer calls `request_signatures` back to the harvester.
5. Harvester returns `RespondSignatures` with `farmer_reward_address_override=<attacker_puzzle_hash>`.
6. `_process_respond_signatures()` at line 917–918 replaces `farmer_reward_address` with the attacker's address without any check.
7. Farmer broadcasts `DeclareProofOfSpace` with `farmer_reward_address=<attacker_puzzle_hash>`.
8. Full node creates the farmer coinbase coin at the attacker's puzzle hash.
9. Farmer's 0.25 XCH block reward is permanently lost to the attacker. [8](#0-7)

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
