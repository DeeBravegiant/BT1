### Title
Receipt Size Hard Limit Bypassed via Post-Validation `output_data_receivers` Mutation - (File: `runtime/runtime/src/lib.rs`)

### Summary
The runtime validates a newly-created receipt's size against `max_receipt_size` before appending the parent receipt's `output_data_receivers` to it. When a contract uses `promise_return`, the parent's data-receiver list is grafted onto the child receipt **after** the size check passes, producing a receipt that silently exceeds the hard limit and enters the forwarding pipeline unchecked.

### Finding Description
The vulnerability class is identical to the GenesisGroup report: a boundary check is performed at one point in time, but the guarded value is mutated past the limit immediately afterward, with no re-validation.

**Step 1 – Size check in `validate_receipt`**

`validate_receipt` in `ValidateReceiptMode::NewReceipt` mode measures the receipt's borsh-serialized length and rejects it if it exceeds `max_receipt_size` (4 MiB). [1](#0-0) 

**Step 2 – Post-validation mutation in `apply_action_receipt`**

After all new receipts have been validated, `apply_action_receipt` in `lib.rs` checks whether the executing receipt used `promise_return` (i.e., `ReturnData::ReceiptIndex`). If so, it appends the **parent** receipt's `output_data_receivers` directly onto the child receipt that was just validated: [2](#0-1) 

This mutation happens unconditionally and without any subsequent size re-check. A contract can craft receipt C at exactly `max_receipt_size`, call `promise_return(C)` from within a receipt that carries one or more `output_data_receivers`, and the appended receivers push C's serialized size above the limit.

**Step 3 – Oversized receipt forwarded without rejection**

The mutated receipt is immediately handed to `receipt_sink.forward_or_buffer_receipt`, which routes it to another shard. The congestion-control layer contains an explicit workaround that clamps the *accounting* size to `max_receipt_size` to prevent the receipt from getting stuck, but the actual receipt bytes remain oversized: [3](#0-2) 

**Step 4 – Receiving side uses lenient validation**

When the oversized receipt arrives at the destination shard it is validated with `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size check to "handle them gracefully until the receipt size limit bug is fixed": [4](#0-3) 

The bug is acknowledged in the codebase and tracked as issue #12606. The test suite documents both the `promise_return` and `value_return` variants: [5](#0-4) 

### Impact Explanation
The `max_receipt_size` hard limit is a load-bearing invariant for the `ChunkStateWitness` size budget. The design document states the combined limits are intended to keep the uncompressed witness under 17 MiB. An oversized receipt (up to `max_receipt_size` + size of appended `output_data_receivers`) directly inflates the witness beyond its intended ceiling. Concretely:

- The **receipt** stored in the outgoing receipt proof and included in the `ChunkStateWitness` exceeds `max_receipt_size`.
- The **congestion-control size accounting** for that receipt is wrong (clamped to `max_receipt_size` instead of the true size), so bandwidth grants are consumed incorrectly.
- Chunk validators applying the witness re-execute the receipt under `ExistingReceipt` mode and produce the same outcome, so there is no consensus split — but the witness size invariant that protects network distribution is violated.

### Likelihood Explanation
The trigger is fully unprivileged: any account can deploy a contract, craft a receipt at exactly `max_receipt_size` using `generate_large_receipt`-style host calls, and invoke `promise_return` from a chained promise. The nearcore test suite already demonstrates this path works end-to-end. No validator or operator cooperation is required.

### Recommendation
After appending `output_data_receivers` to the child receipt (lines 1028–1035 of `lib.rs`), re-measure the receipt's serialized size and return a `NewReceiptValidationError::ReceiptSizeExceeded` action error if the post-mutation size exceeds `max_receipt_size`. The same re-check should be applied to the `value_return` path where a `DataReceipt` wrapping a large return value is constructed (lines 1044–1056).

### Proof of Concept
1. Deploy a contract that exposes `max_receipt_size_promise_return_method1(args_size)`.
2. Compute `args_size = max_receipt_size − base_receipt_size` so that receipt C serializes to exactly `max_receipt_size`.
3. Submit a transaction calling `max_receipt_size_promise_return_method1` from an account that is itself the `output_data_receiver` of a parent promise (i.e., the call is chained with `.then`).
4. Observe that the transaction succeeds (no `ReceiptSizeExceeded` error), and that an incoming receipt with `borsh::object_length > max_receipt_size` appears in the chain — exactly as asserted by `assert_oversized_receipt_occurred` in the existing test. [6](#0-5)

#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
