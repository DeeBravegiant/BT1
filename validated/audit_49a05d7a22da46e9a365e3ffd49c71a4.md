Audit Report

## Title
Zero rsETH Minted on Dust Deposits Due to Integer Division Truncation - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
Every L2 pool contract computes `rsETHAmount` via integer division in `viewSwapRsETHAmountAndFee`. When the deposit amount is small enough that `amountAfterFee * 1e18 < rsETHToETHrate`, the result truncates to zero. No post-calculation guard checks `rsETHAmount > 0` before proceeding, so the deposit executes, consuming the user's ETH or ERC-20 tokens, while minting or transferring zero rsETH/wrsETH to the depositor.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee`:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) 

Solidity integer division truncates toward zero. `rsETHToETHrate` is sourced from `getRate()` and equals the ETH price of rsETH scaled by 1e18. Since rsETH accrues yield, this rate starts at `1e18` and only increases. Once the rate exceeds `1e18` (i.e., after any yield accrual), a deposit of `1 wei` produces:

```
amountAfterFee = 1
rsETHAmount = 1 * 1e18 / rsETHToETHrate = 0  (integer truncation)
```

The ETH-path `deposit` function then unconditionally calls:

```solidity
wrsETH.mint(msg.sender, rsETHAmount);  // mints 0
``` [2](#0-1) 

The only existing guard is `if (amount == 0) revert InvalidAmount()`, which checks the raw input, not the computed output. [3](#0-2) 

The same pattern is confirmed in every pool variant:

- `RSETHPoolV3ExternalBridge`: `wrsETH.mint(msg.sender, rsETHAmount)` after unchecked division [4](#0-3) 

- `RSETHPoolNoWrapper`: `rsETH.safeTransfer(msg.sender, rsETHAmount)` — transferring 0 tokens does not revert [5](#0-4) 

- `RSETHPool`: `IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount)` — same [6](#0-5) 

- `RSETHPoolV2`: identical division, no output guard [7](#0-6) 

For the token path, the same truncation applies via `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate`. [8](#0-7) 

For `LRTDepositPool`, the L1 pool has a `minRSETHAmountExpected` slippage guard, but it only reverts when `rsethAmountToMint < minRSETHAmountExpected`. If the caller passes `0` as the minimum, a zero-mint deposit is silently accepted. [9](#0-8) 

## Impact Explanation
A depositor sends ETH or ERC-20 tokens and receives zero wrsETH/rsETH. The deposited ETH is absorbed into the pool's balance but is not credited to `feeEarnedInETH` (only the `fee` portion is, which is also 0 for dust amounts), leaving the funds permanently untracked and unrecoverable by the user. This matches **Low — contract fails to deliver promised returns, but doesn't lose value** from the allowed impact scope: the protocol retains the ETH, but the user receives nothing for their deposit.

## Likelihood Explanation
No special privileges or oracle manipulation are required. Any unprivileged user calling `deposit{value: 1}("")` triggers the condition as soon as `rsETHToETHrate > 1e18`, which occurs after any yield accrual. The threshold grows over time: at 10% yield (`rsETHToETHrate = 1.1e18`), any deposit of 1 wei produces zero rsETH. Accidental dust deposits from wallets, contract integrations that do not enforce a minimum output, or rounding in upstream calculations can all trigger this silently. No victim mistake beyond sending a small amount is required.

## Recommendation
1. Add a post-calculation zero-output guard in every `deposit` function (or inside `viewSwapRsETHAmountAndFee`):
   ```solidity
   if (rsETHAmount == 0) revert InvalidAmount();
   ```
2. For `LRTDepositPool._beforeDeposit`, enforce `minRSETHAmountExpected > 0` or add an explicit `rsethAmountToMint > 0` check independent of the caller-supplied minimum.
3. Document the minimum deposit threshold so integrators can enforce it off-chain.

## Proof of Concept
Assume `rsETHToETHrate = 1.001e18` (0.1% yield accrued — realistic after a few months).

1. User calls `RSETHPoolV3.deposit{value: 1}("")`.
2. `fee = 1 * feeBps / 10_000 = 0` (for any `feeBps < 10_000`).
3. `amountAfterFee = 1`.
4. `rsETHAmount = 1 * 1e18 / 1_001_000_000_000_000_000 = 0` (integer truncation).
5. `feeEarnedInETH += 0`.
6. `wrsETH.mint(msg.sender, 0)` executes without revert.
7. User's 1 wei is permanently held in the pool with no accounting entry; user holds zero wrsETH.

**Foundry fuzz test sketch:**
```solidity
function testFuzz_zeroMintOnDustDeposit(uint256 rate) public {
    vm.assume(rate > 1e18 && rate < 2e18);
    // set rsETHToETHrate = rate via oracle mock
    uint256 balanceBefore = wrsETH.balanceOf(user);
    pool.deposit{value: 1}("");
    assertEq(wrsETH.balanceOf(user), balanceBefore); // user received 0
}
```
