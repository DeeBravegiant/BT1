Audit Report

## Title
Missing Chainlink Staleness Check Allows Oracle Price Abuse in Deposits and Withdrawals - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt` and `answeredInRound`, performing no staleness validation. This stale price propagates directly into `LRTDepositPool.getRsETHAmountToMint()` and `LRTWithdrawalManager.getExpectedAssetAmount()`, both reachable by any unprivileged user, enabling theft of unclaimed yield from existing rsETH holders when a Chainlink feed freezes.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` destructures the Chainlink response as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
```

`updatedAt` (position 4) and `answeredInRound` (position 5) are silently discarded. No check of the form `block.timestamp - updatedAt > maxStaleness` or `answeredInRound < roundId` is performed.

This price is returned through `LRTOracle.getAssetPrice()` (line 156–158) and consumed in two critical public paths:

**Path 1 – Deposit minting** (`LRTDepositPool.getRsETHAmountToMint`, line 520):
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
Called from `depositAsset()` and `depositETH()`, both publicly accessible with no privilege requirement.

**Path 2 – Withdrawal asset amount** (`LRTWithdrawalManager.getExpectedAssetAmount`, line 593):
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```
Called from `initiateWithdrawal()`, publicly accessible.

`ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 30–31) does check `answeredInRound < roundID` and `timestamp == 0`, but still omits the critical time-based bound (`block.timestamp - timestamp > heartbeat`), leaving it partially vulnerable to the same class of issue.

The SECURITY.md exclusion for "Incorrect data supplied by third-party oracles" does not apply here. This is a protocol-level failure to validate data freshness — a code bug in the protocol's own contracts — not a case of an oracle node providing wrong data. The SECURITY.md explicitly notes: "This does not exclude oracle manipulation/flash-loan attacks."

## Impact Explanation
**High – Theft of unclaimed yield.**

When a Chainlink LST/ETH feed freezes at a price above the real market price, a depositor calling `depositAsset()` receives more rsETH than the fair value of their deposit (numerator `getAssetPrice(asset)` is inflated). This dilutes all existing rsETH holders, constituting theft of their accrued yield. The rsETHPrice used as the denominator is a stored value updated separately via `updateRSETHPrice()`; if it was last updated at a correct price, the mismatch is direct and unmitigated.

## Likelihood Explanation
**Medium.** Chainlink LST/ETH feeds on mainnet have documented heartbeat windows (e.g., 24 hours). During periods of low price volatility — where the deviation threshold is not triggered — feeds can remain stale for extended periods within their heartbeat. An attacker can monitor `updatedAt` on-chain, identify a stale feed, and call `depositAsset()` without any privileged access. This is a known, recurring condition, not a theoretical edge case.

## Recommendation
In `ChainlinkPriceOracle.getAssetPrice()`, capture `updatedAt` and revert if the price is stale:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(answeredInRound >= roundId, "Stale round");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= MAX_ORACLE_DELAY, "Stale price");
```

`MAX_ORACLE_DELAY` should be set per-feed based on its documented heartbeat (e.g., 86400 seconds for a 24-hour heartbeat feed, with a small buffer). Store it alongside `assetPriceFeed` in the mapping.

Apply the same time-based check to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which already checks `answeredInRound` and `timestamp == 0` but is missing the `block.timestamp - timestamp` bound.

## Proof of Concept
1. Assume stETH/ETH Chainlink feed last updated 25 hours ago (past its 24-hour heartbeat) at stale price `1.05e18` while the real market price is `0.98e18`. The rsETHPrice stored in `LRTOracle` was last updated when prices were correct.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
3. `getRsETHAmountToMint(stETH, 100e18)` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.05e18`.
4. `rsethAmountToMint = (100e18 * 1.05e18) / rsETHPrice` — attacker receives rsETH priced as if stETH is worth 1.05 ETH, not 0.98 ETH (~7% excess).
5. The excess rsETH minted dilutes all existing rsETH holders proportionally, constituting theft of their accrued yield.
6. No privileged access is required; the only precondition is a stale Chainlink feed, which is an observable on-chain condition via `latestRoundData().updatedAt`.

**Foundry fork test plan:** Fork mainnet, mock a Chainlink aggregator returning a `latestRoundData()` with `updatedAt = block.timestamp - 25 hours`. Call `depositAsset()` with a normal user. Assert that `rsethAmountToMint` exceeds the fair value computed against the real market price. Confirm no revert occurs.
