### Title
Single Byzantine Participant Can Forge `AppPublicKey` CKD Responses, Bypassing Threshold Requirement — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_ckd` in the MPC contract accepts `CKDResponse` values for the `AppPublicKey` variant with **zero cryptographic verification**. Unlike ECDSA/EdDSA `respond` (which verifies the signature against the derived public key) and the `AppPublicKeyPV` variant (which runs `ckd_output_check`), the `AppPublicKey` branch is a no-op. Any single attested participant can submit arbitrary `big_y`/`big_c` values and the contract will resolve the pending yield, delivering a forged confidential-key-derivation output to the requesting user.

### Finding Description

In `respond_ckd` the response-validation block is:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKey` arm the contract immediately falls through to:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

Compare this with `respond` (ECDSA) and `respond_verify_foreign_tx`, both of which verify the submitted value against a public key before resolving the yield: [3](#0-2) [4](#0-3) 

The only gate standing between an attested participant and a successful forged `respond_ckd` call is:

1. `assert_caller_is_signer()` — signer == predecessor
2. `is_running_or_resharing()` — protocol state check
3. `accept_requests` — TEE gate
4. `assert_caller_is_attested_participant_and_protocol_active()` — participant + attestation check [5](#0-4) 

None of these checks verify that the submitted `CKDResponse` is the output of a genuine threshold computation. For ECDSA/EdDSA the signature itself is the proof; for `AppPublicKey` CKD there is no equivalent proof.

### Impact Explanation

A single Byzantine participant (strictly below the signing threshold) who holds a valid TEE attestation can:

1. Observe a pending `request_app_private_key` call whose `app_public_key` is the `AppPublicKey` variant.
2. Immediately call `respond_ckd` with attacker-chosen `big_y` and `big_c` values — no off-chain threshold protocol needed.
3. The contract resolves the yield and delivers the forged `CKDResponse` to the requesting user.

The user receives key-derivation material that the attacker fully controls. Depending on how the application uses the derived key (encryption, signing, authentication), this enables secret recovery or forgery — matching the "bypass of threshold-signature requirements … that materially enables forgery or secret recovery" impact class.

### Likelihood Explanation

Every attested participant is a potential attacker. The TEE attestation proves the node runs the whitelisted image, but:

- A compromised node operator who can influence the attested binary (e.g., via a supply-chain compromise of the allowed image hash) can exploit this without any physical TEE attack.
- The window of opportunity is the entire lifetime of a pending `AppPublicKey` CKD request — the attacker simply races the honest nodes.
- No collusion is required; a single participant suffices.

### Recommendation

Add on-chain verification for the `AppPublicKey` CKD response analogous to what `AppPublicKeyPV` already does via `ckd_output_check`. If the `AppPublicKey` variant is intentionally unverifiable on-chain (e.g., because the app public key alone is insufficient to check the output), the contract should either:

- Require threshold-many identical `respond_ckd` calls before resolving the yield (mirroring the off-chain threshold), or
- Deprecate the `AppPublicKey` variant in favour of `AppPublicKeyPV`, which carries the proof material needed for on-chain verification.

At minimum, document the trust assumption explicitly: `AppPublicKey` CKD security relies entirely on every responding participant being honest, not on the threshold property.

### Proof of Concept

```
1. User calls `request_app_private_key` with AppPublicKey(pk) and domain_id pointing to a BLS12-381 domain.
   → Contract stores the pending CKD request and yields.

2. Malicious attested participant P (one of N, threshold = t < N) calls:
     respond_ckd(ckd_request, CKDResponse { big_y: ATTACKER_POINT, big_c: ATTACKER_POINT })

3. Contract executes:
     match &request.app_public_key {
         AppPublicKey(_) => {}   // no check — falls through
         ...
     }
     resolve_yields_for(...)     // yield resolved with attacker-chosen values

4. User's `request_app_private_key` call returns CKDResponse { big_y: ATTACKER_POINT, big_c: ATTACKER_POINT }.
   The attacker knows the discrete log of ATTACKER_POINT and therefore the derived key material.
``` [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
