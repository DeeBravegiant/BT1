Audit Report

## Title
`RsETHTokenWrapper` 1:1 cross-token mint/burn with no per-token accounting enables fund theft when multiple altRsETH tokens are active - (File: contracts/L2/RsETHTokenWrapper.sol)

## Summary
`RsETHTokenWrapper` supports multiple allowed altRsETH tokens and mints/burns `wrsETH` at a strict 1:1 ratio against any of them with no exchange rate check and no per-token accounting. Once a second token is added via `reinitialize` or `addAllowedToken`, any unprivileged user can deposit the cheaper altRsETH token and immediately withdraw the more valuable one at 1:1, extracting the price difference from other depositors. The `maxAmountToDepositBridgerAsset` function compounds the issue by comparing the aggregate `totalSupply()` against the balance of only one token, inflating the deposit ceiling by the supply backed by all other tokens.

## Finding Description
The contract maintains `mapping(address => bool) public allowedTokens` and exposes three paths to register additional tokens:

- `reinitialize(address _altRsETH)` — `DEFAULT_ADMIN_ROLE`, explicitly designed as the upgrade path for a second token. [1](#0-0) 
- `addAllowedToken(address _asset)` — `TIMELOCK_ROLE`. [2](#0-1) 

`_deposit` mints exactly `_amount` wrsETH for exactly `_amount` of any allowed token — no exchange rate, no per-token ledger: [3](#0-2) 

`_withdraw` burns exactly `_amount` wrsETH and transfers exactly `_amount` of **whichever** allowed token the caller specifies — again no rate check: [4](#0-3) 

Both `withdraw` and `withdrawTo` are fully public with no role restriction. [5](#0-4) 

`maxAmountToDepositBridgerAsset` uses `totalSupply()` (all wrsETH from all tokens) minus the balance of one specific asset, so wrsETH minted against token B inflates the apparent headroom for token A: [6](#0-5) 

The identical pattern exists in `AGETHTokenWrapper`: [7](#0-6) [8](#0-7) [9](#0-8) 

No existing guard prevents a caller from depositing token A and withdrawing token B. The only check in `_withdraw` is `allowedTokens[_asset]`, which passes for any registered token. [10](#0-9) 

## Impact Explanation
**Critical — direct theft of user funds.** Once two allowed tokens trade at any non-zero price differential (a routine condition for bridge-issued representations of the same underlying asset), any holder of wrsETH can atomically swap the cheaper token for the more expensive one at 1:1, extracting the difference from other depositors. The stolen value is bounded only by the balance of the more valuable token held by the wrapper. No flash loan, no special role, and no victim mistake is required.

## Likelihood Explanation
**Medium.** The precondition is a legitimate governance action (`reinitialize` or `addAllowedToken`) that the contract explicitly provisions for. Once a second token is live, the exploit is immediately available to any unprivileged user. LayerZero OFT representations and native-bridge representations of the same L1 token routinely trade at small but non-zero discounts/premiums on secondary markets, providing the price differential needed to profit. The attack is repeatable until the more valuable token's balance is drained.

## Recommendation
1. **Per-token accounting**: add `mapping(address => uint256) public wrsETHMintedFor` and update it in `_deposit`/`_withdraw`. Use `wrsETHMintedFor[_asset]` instead of `totalSupply()` in `maxAmountToDepositBridgerAsset`.
2. **Enforce same-token redemption**: record which token each wrsETH unit was minted against (e.g., via a per-user or per-mint receipt) and require `_withdraw` to redeem against the same token, or introduce an oracle-based exchange rate before permitting cross-token redemption.
3. Apply the same fixes to `AGETHTokenWrapper`.

## Proof of Concept
**Setup**: Admin calls `reinitialize(altRsETH_B)`. `altRsETH_B` trades at 1.05× the value of `altRsETH_A` (realistic bridge premium).

```
// Step 1 — victim deposits the valuable token
Alice.deposit(altRsETH_B, 100e18)
  → wrapper receives 100e18 altRsETH_B
  → Alice receives 100e18 wrsETH

// Step 2 — attacker deposits the cheap token
Bob.deposit(altRsETH_A, 100e18)
  → wrapper receives 100e18 altRsETH_A
  → Bob receives 100e18 wrsETH

// Step 3 — attacker withdraws the valuable token (no restriction)
Bob.withdraw(altRsETH_B, 100e18)
  → burns 100e18 wrsETH
  → Bob receives 100e18 altRsETH_B (worth 105e18 altRsETH_A equivalent)
```

**Result**: Bob spent 100e18 altRsETH_A and received 100e18 altRsETH_B, netting a 5 ETH-equivalent profit. Alice's 100e18 altRsETH_B is gone; the wrapper holds only 100e18 altRsETH_A against Alice's 100e18 wrsETH, which she cannot redeem at fair value.

**Foundry invariant test**: assert `∀ allowed token t: ERC20(t).balanceOf(wrapper) >= wrsETHMintedFor[t]` after arbitrary sequences of `deposit`/`withdraw` calls across two allowed tokens. The invariant breaks immediately in step 3 above.

    function maxAmountToDepositBridgerAsset(address _asset) public view returns (uint256) {
        if (!allowedTokens[_asset]) return 0;

        // get totalSupply of wrsETH minted
        uint256 wrsETHSupply = totalSupply();
        // balance of _asset with the contract
        uint256 balanceOfAssetInWrapper = ERC20Upgradeable(_asset).balanceOf(address(this));

        if (balanceOfAssetInWrapper > wrsETHSupply) return 0;

        return wrsETHSupply - balanceOfAssetInWrapper;
    }
