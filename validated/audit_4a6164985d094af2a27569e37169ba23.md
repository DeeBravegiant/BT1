Audit Report

## Title
Missing Chainlink Oracle Staleness and Validity Checks Allow Stale Price Acceptance - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `answer`, performing no validation on staleness (`updatedAt`), round completeness (`answeredInRound` vs `roundId`), or price sign. The same repository already implements all three guards in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, making the omission a clear inconsistency. A stale inflated price flows directly into rsETH mint calculations, enabling an attacker to deposit a depegged asset at the stale rate and extract value from existing rsETH holders.

## Finding Description
In `contracts/oracles/ChainlinkPriceOracle.sol`, `getAssetPrice()` reads from a Chainlink feed as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values of `latestRoundData()` are available but only `answer` is used. The following checks are entirely absent:
1. **Staleness check**: `block.timestamp - updatedAt > threshold`
2. **Stale round check**: `answeredInRound < roundId`
3. **Non-negative price check**: `price <= 0`

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` correctly implements all three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle.getAssetPrice()` is consumed in two critical paths:

**Deposit minting** — `LRTDepositPool.getRsETHAmountToMint()`:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

**rsETH price update** — `LRTOracle._getTotalEthInProtocol()`:
```solidity
uint256 assetER = getAssetPrice(asset);
...
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [4](#0-3) 

**Exploit path:**
1. A Chainlink feed for a supported LST (e.g., stETH) goes stale during network congestion or L2 sequencer downtime, freezing the on-chain price at `P_stale`.
2. The asset depegs on the open market to `P_market < P_stale`, but `rsETHPrice` was last updated when the price was still `P_stale` (or close to it).
3. Attacker purchases the depegged asset at `P_market` on the open market.
4. Attacker calls `depositAsset()`. `getRsETHAmountToMint()` computes `rsethAmountToMint = (amount * P_stale) / rsETHPrice`, minting rsETH at the inflated stale rate.
5. The attacker receives rsETH representing more ETH value than they deposited, diluting existing holders.
6. When the oracle updates and `rsETHPrice` is recalculated to reflect the true lower asset value, existing holders' rsETH is worth less — the attacker extracted the difference.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` only applies to the rsETH price update call, not to the deposit minting path, and triggers only after the damage is done. [5](#0-4) 

The `updatePriceOracleForValidated` sanity check validates the price only at oracle registration time, not at each use. [6](#0-5) 

## Impact Explanation
**High — Theft of unclaimed yield.** An attacker who deposits a depegged LST at a stale inflated price receives more rsETH than the fair ETH value they contributed. When the oracle corrects and `rsETHPrice` is recalculated, the protocol's total ETH per rsETH decreases, transferring yield that had accrued to existing rsETH holders to the attacker. This is a direct, concrete theft of unclaimed yield from existing holders, not merely a theoretical dilution. A zero or negative price return additionally causes a revert (temporary freezing of deposits) or catastrophic `uint256` wraparound.

## Likelihood Explanation
Chainlink feeds go stale during low-volatility periods (heartbeat not triggered), network congestion, or L2 sequencer downtime. The protocol is deployed on Arbitrum, Optimism, Base, and Scroll, where sequencer outages are a documented, recurring risk. No privileged role is required; any depositor can exploit the window by calling `depositAsset()` with a standard ERC-20 transfer. The attack is repeatable for as long as the feed remains stale.

## Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`STALENESS_THRESHOLD` should be configurable per feed based on the feed's documented heartbeat interval (e.g., 3600 seconds for a 1-hour heartbeat feed).

## Proof of Concept

**Fork test outline (Foundry):**

```solidity
// 1. Fork mainnet/Arbitrum at a block where stETH/ETH feed is live.
// 2. Deploy or reference ChainlinkPriceOracle with the stETH/ETH feed.
// 3. Warp block.timestamp forward by > heartbeat interval (e.g., 2 hours).
//    The feed's updatedAt is now stale; latestRoundData() still returns the old price.
// 4. Record stalePrice = chainlinkPriceOracle.getAssetPrice(stETH).
// 5. Simulate a market depeg: the attacker acquires stETH at 0.97 ETH.
// 6. Call lrtDepositPool.depositAsset(stETH, amount, 0, "").
// 7. Assert rsethMinted == amount * stalePrice / rsETHPrice  (inflated vs fair value).
// 8. Call lrtOracle.updateRSETHPrice() to recalculate with the now-corrected price.
// 9. Assert rsETHPrice decreased, proving existing holders were diluted.
// 10. Confirm the attacker's rsETH redeems for more ETH than they deposited.
```

The test demonstrates that without staleness validation, step 6 succeeds and step 9 confirms value extraction from existing holders. With the recommended fix applied, step 6 reverts with `StalePrice()`.
