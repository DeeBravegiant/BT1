Let me look at the `respond_verify_foreign_tx` function more carefully and compare it with the `respond` function for regular signatures to confirm the analog.

### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Pending Request — Cross-Request Signature Replay Enables Forged Foreign-Chain Verification - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function verifies that a signature is valid over a caller-supplied `payload_hash`, but never verifies that `payload_hash` was derived from the pending `request`. A single malicious attested participant (below the signing threshold) can replay a legitimate MPC threshold signature produced for one foreign-tx verification request to resolve a completely different pending request, forging on-chain verification for a transaction that was never actually confirmed on the foreign chain.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–754), the contract receives a `VerifyForeignTransactionResponse` containing two caller-supplied fields: `payload_hash` and `signature`. The contract verifies that `signature` is valid over `payload_hash` using the MPC root public key, then resolves the pending request:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract then resolves the pending request with the full response:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

**The missing check:** The contract never verifies that `response.payload_hash` was derived from the pending `request`. The `ForeignTxSignPayload` that nodes sign encodes both the original request (tx_id, chain, extractors) and the extracted values from the foreign chain:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
``` [3](#0-2) 

**Contrast with `respond` for regular signatures** (lines 563–651), where the payload is taken directly from the on-chain pending request — there is no caller-supplied payload hash:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← taken from the on-chain request, not from the caller
    &expected_public_key,
)
``` [4](#0-3) 

The SDK's `ForeignChainSignatureVerifier` (used by bridge integrators) does perform this check:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [5](#0-4) 

But the on-chain contract — the authoritative verification layer — does not enforce this invariant.

---

### Impact Explanation

A single malicious attested participant (below the signing threshold) can execute a **cross-request signature replay**:

1. Submit `verify_foreign_transaction` for a real, confirmed transaction (request A). The MPC nodes legitimately produce threshold signature `sig_A` over `payload_hash_A = SHA256(borsh(ForeignTxSignPayload { request: A, values: [real_block_hash_A] }))`.
2. Separately submit `verify_foreign_transaction` for a non-existent or unconfirmed transaction (request B) — any attested participant can also call user-facing functions.
3. Call `respond_verify_foreign_tx(request=B, response={payload_hash=payload_hash_A, signature=sig_A})`.

The contract accepts this because:
- Request B is pending ✓
- `sig_A` is a valid MPC threshold signature over `payload_hash_A` ✓
- The contract never checks that `payload_hash_A` encodes request B ✗

Request B is resolved with a forged `VerifyForeignTransactionResponse`. Bridge services that do not independently re-verify the payload_hash (using the SDK's `ForeignChainSignatureVerifier`) will receive an on-chain "verified" signal for a transaction that was never confirmed on the foreign chain. This enables **invalid bridge execution** — e.g., releasing bridged funds for a deposit transaction that never occurred — matching the "High" impact category of forged foreign-chain verification causing double-spend conditions.

---

### Likelihood Explanation

**Medium.** The attacker must be a single malicious attested participant (below threshold). No threshold collusion is required — the attacker only needs to observe a legitimate threshold signature produced for one request (which they can do as a participant in the MPC protocol) and replay it against a different pending request. The attacker can also submit the target request B themselves as a normal user. The only constraint is that both requests must be pending simultaneously, which is achievable by timing the submissions.

---

### Recommendation

In `respond_verify_foreign_tx`, the contract must verify that `response.payload_hash` is bound to the pending `request`. Since the contract does not know the extracted values, the most robust fix is to require the responding node to also submit the `Vec<ExtractedValue>` alongside the response, allowing the contract to re-derive and compare the payload hash:

```rust
// Proposed: re-derive expected payload hash from request + submitted values
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),
});
let expected_hash = expected_payload.compute_msg_hash()?;
require!(
    expected_hash == response.payload_hash,
    RespondError::PayloadHashMismatch
);
```

This mirrors the pattern used by the SDK's `ForeignChainSignatureVerifier` and closes the cross-request replay path entirely.

// payload_hash_A = SHA256(borsh(ForeignTxSignPayload::V1 {
//     request: BitcoinRpcRequest { tx_id: [0xAA;32], ... },
//     values: [BitcoinExtractedValue::BlockHash([0xCC;32])]
// }))
// sig_A = threshold_sign(payload_hash_A)  // valid MPC signature

// Step 2: Malicious attested participant calls:
respond_verify_foreign_tx(
    request = VerifyForeignTransactionRequest {
        request: BitcoinRpcRequest { tx_id: [0xBB;32], ... },  // request B
        domain_id: ...,
        payload_version: V1,
    },
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,   // hash for request A, not B
        signature: sig_A,               // valid signature over payload_hash_A
    }
)

// Contract checks:
// 1. assert_caller_is_attested_participant_and_protocol_active() → PASS
// 2. verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → PASS
// 3. resolve_yields_for(pending_verify_foreign_tx_requests, request_B, response) → RESOLVES

// Result: request B is resolved with payload_hash_A.
