Audit Report

## Title
Missing Time-Based Staleness Check in Chainlink Oracle Allows Stale Collateral Prices - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` omits a time-based heartbeat check (`block.timestamp - updatedAt > heartbeat`), relying solely on `answeredInRound < roundID` and `timestamp == 0`. On modern Chainlink OCR feeds deployed on L2s, `answeredInRound` always equals `roundId`, making the round-based check a no-op. A price that has not been refreshed for hours or days passes all three guards, causing L2 pool deposits to mint incorrect wrsETH amounts.

## Finding Description
`getRate()` at line 26–37 of `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` calls `latestRoundData()` and applies three checks:

```solidity
if (answeredInRound < roundID) revert StalePrice();   // line 30 – no-op on OCR feeds
if (timestamp == 0) revert IncompleteRound();          // line 31 – only catches uninitialised rounds
if (ethPrice <= 0) revert InvalidPrice();              // line 32 – sign check only
```

On Chainlink OCR aggregators (the standard on Arbitrum, Optimism, Base, etc.), `answeredInRound` is always set equal to `roundId` by the aggregator contract, so the first check never reverts. The second check only catches a completely uninitialised round (`updatedAt == 0`). No check of the form `block.timestamp - timestamp > HEARTBEAT` is present.

The returned `normalizedPrice` is then consumed by every L2 pool variant (`RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV2NBA`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolNoWrapper`) through their shared pattern:

```solidity
uint256 rsETHToETHrate = getRate();                          // stale price accepted
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;        // wrong mint amount
wrsETH.mint(msg.sender, rsETHAmount);
```

If the feed has not been updated (e.g., L2 sequencer downtime, Chainlink node outage, deviation threshold never crossed during low-volatility periods), the stale price is used verbatim. Because rsETH is a yield-bearing token whose ETH rate monotonically increases, a stale (older, lower) rate causes the division to yield a larger `rsETHAmount`, minting more wrsETH than the deposited ETH warrants. Existing wrsETH holders are diluted.

## Impact Explanation
**Low – Contract fails to deliver promised returns, but doesn't lose value** (with potential escalation to High in prolonged outage scenarios).

Under normal drift (rsETH rate increases slowly), a stale price causes depositors to receive slightly more wrsETH than they are entitled to, diluting existing holders' redemption value. This is a concrete, non-hypothetical deviation from the protocol's promised exchange rate. In a prolonged feed outage (hours to days, which has occurred on L2 Chainlink deployments), the magnitude of over-minting grows proportionally, escalating toward theft of unclaimed yield from existing wrsETH holders.

## Likelihood Explanation
Any unprivileged depositor calling `deposit()` on any L2 pool during a period when the Chainlink feed has not been updated within its heartbeat window triggers the issue. No attacker capability is required beyond a normal ETH deposit. Chainlink feed outages on L2s are documented historical events (sequencer downtime, node issues). The condition is passively triggered, not actively manufactured by the depositor.

## Recommendation
Add a configurable heartbeat constant and enforce it in `getRate()`:

```solidity
uint256 public constant HEARTBEAT = 3600; // set per-feed (e.g. 1 h or 24 h)

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 updatedAt, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

The heartbeat value should match the Chainlink feed's documented update frequency for each deployed chain.

## Proof of Concept
**Foundry fork test outline:**

1. Fork the target L2 (e.g., Arbitrum) at a recent block.
2. Deploy or reference the live `ChainlinkOracleForRSETHPoolCollateral` pointing at a real Chainlink feed.
3. Use `vm.warp(block.timestamp + 25 hours)` to simulate a feed that has not updated.
4. Call `getRate()` — it returns the 25-hour-old price without reverting, demonstrating all three guards pass.
5. Call the pool's `viewSwapRsETHAmountAndFee(1 ether)` before and after the warp; show the returned `rsETHAmount` is identical despite the elapsed time, confirming the stale price is accepted and would produce an inflated mint.
