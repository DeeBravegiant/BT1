### Title
Direct Deposits Missing from Produced Value Calculation Enables ADA Creation Out of Thin Air — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in the transaction body that allows ADA to be deposited directly into account balances. However, neither `dijkstraProducedValue` nor `dijkstraSubTxProducedValue` includes the direct deposits amount in the produced value. Because the value conservation check (`consumed == produced`) does not account for direct deposits on either side, an attacker can include arbitrarily large direct deposits in a transaction without funding them from UTxO inputs, causing ADA to be created out of thin air.

---

### Finding Description

The Dijkstra era adds `dtbrDirectDeposits :: !DirectDeposits` to both `DijkstraTxBodyRaw TopTx` and `DijkstraTxBodyRaw SubTx`: [1](#0-0) [2](#0-1) 

These deposits are applied to account balances in `dijkstraEntitiesTransition` via `applyDirectDeposits`: [3](#0-2) 

`applyDirectDeposits` unconditionally adds coin to account balances with no funding check: [4](#0-3) 

The produced value for a top-level Dijkstra transaction is computed by `dijkstraProducedValue`, which delegates to `conwayProducedValue` (outputs + fees + cert deposits + treasury donation + burned assets) and aggregates sub-transaction produced values — **no direct deposits term is added**: [5](#0-4) 

For sub-transactions, `dijkstraSubTxProducedValue` similarly omits direct deposits: [6](#0-5) 

The consumed value (`getConsumedDijkstraValue`) also does not include direct deposits: [7](#0-6) 

The `EraUTxO DijkstraEra` instance wires these functions into the preservation-of-value check: [8](#0-7) 

The `validateValueNotConservedUTxO` check (inherited from Shelley, present in the Dijkstra UTXO rule as evidenced by `ValueNotConservedUTxO` in `DijkstraUtxoPredFailure`) computes:

```
consumed pp certState utxo txBody == produced pp certState txBody
``` [9](#0-8) 

Because direct deposits appear in neither `consumed` nor `produced`, the check is blind to them. An attacker can set `directDepositsTxBodyL` to deposit X ADA into their account while only funding `outputs + fees + cert deposits` from inputs. The conservation check passes (`consumed == produced`), and `applyDirectDeposits` then inflates the account balance by X ADA with no corresponding deduction anywhere in the ledger state.

The only validation applied to direct deposits is a network-ID check and a missing-accounts check: [10](#0-9) [11](#0-10) 

Neither check verifies that the deposited ADA is funded from transaction inputs.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An unprivileged transaction author can craft a transaction that passes all validation rules while causing the ledger to credit an arbitrary amount of ADA to any registered account. This violates the fundamental preservation-of-value invariant and constitutes unbounded ADA inflation. Every honest node would accept the transaction and apply the same invalid state transition, so there is no divergence — the inflation is universally accepted, making it permanent and unrecoverable without a hard fork.

---

### Likelihood Explanation

The Dijkstra era is the current development era. Any user who can submit a Dijkstra-era transaction can exploit this. The attack requires only: (1) a registered target account, (2) a valid transaction body with `directDeposits` set to the desired amount, and (3) inputs sufficient to balance outputs + fees + cert deposits (not the direct deposits). No privileged access, key compromise, or consensus majority is required.

---

### Recommendation

Add the total direct deposits coin to the produced value in both `dijkstraProducedValue` (for top-level transactions) and `dijkstraSubTxProducedValue` (for sub-transactions). Concretely, in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`:

```haskell
-- In dijkstraProducedValue, add direct deposits from the top-level tx body:
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> inject (fold . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
    <> foldMap'
         (getProducedValue pp isRegPoolId . view bodyTxL)
         (txBody ^. subTransactionsTxBodyL)

-- In dijkstraSubTxProducedValue, add direct deposits from the sub-tx body:
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject ( getTotalDepositsTxBody pp isRegPoolId txBody
             <> txBody ^. treasuryDonationTxBodyL
             <> fold (unDirectDeposits $ txBody ^. directDepositsTxBodyL) )
    <> burnedMultiAssets txBody
```

This ensures that any ADA credited to accounts via direct deposits must be funded from transaction inputs, restoring the preservation-of-value invariant.

---

### Proof of Concept

1. Register account `A` (e.g., via a `RegTxCert` certificate in a prior transaction).
2. Construct a Dijkstra-era transaction `tx` with:
   - `directDepositsTxBodyL = DirectDeposits { A → Coin 1_000_000_000 }` (1000 ADA)
   - Inputs that balance only `outputs + fee` (no extra ADA for the direct deposit)
   - No outputs to account `A`
3. Submit `tx`. The `validateValueNotConservedUTxO` check passes because neither `consumed` nor `produced` includes the 1000 ADA direct deposit.
4. `dijkstraEntitiesTransition` calls `applyDirectDeposits`, crediting account `A` with 1000 ADA.
5. The total ADA in circulation has increased by 1000 ADA with no corresponding deduction from any pot (UTxO, reserves, treasury, fees, or deposits). [5](#0-4) [6](#0-5) [3](#0-2) [4](#0-3)
