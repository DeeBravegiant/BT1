### Title
Attacker-Controlled `authorized_providers` Injected via On-Chain REMARK Condition Causes Irreversible CATWallet-to-CRCATWallet Conversion — (`chia/wallet/wallet_state_manager.py`, `chia/wallet/vc_wallet/cr_cat_wallet.py`)

---

### Summary

A permissionless attacker who holds any amount of a CAT token (matching the victim's `CATWallet` asset_id) can submit a crafted on-chain CAT spend that includes a REMARK condition encoding attacker-chosen `authorized_providers` and `proofs_checker`. When the victim's wallet syncs, `handle_cat` calls `CRCAT.get_next_from_coin_spend` on the parent spend, blindly reads those attacker-supplied values, and then permanently converts the victim's `CATWallet` to a `CRCATWallet` with attacker-controlled authorization parameters. No signature, key, or privileged access is required beyond owning a small amount of the same CAT.

---

### Finding Description

**Call chain:**

```
peer notification (hint subscription)
  → _add_coin_states
    → determine_coin_type (line 917: coin_spend = fetch_coin_spend_for_coin_state(parent_coin_state, peer))
      → handle_cat
        → CRCAT.get_next_from_coin_spend(coin_spend)   # reads REMARK condition
          → CRCATWallet.convert_to_cr(found_cat_wallet, crcat.authorized_providers, ...)
```

**Step 1 – Attacker crafts a valid on-chain CAT spend.**

`CRCAT.get_next_from_coin_spend` has a branch for when the parent spend is *not* a CR-CAT: [1](#0-0) 

If the inner puzzle is not wrapped in `CREDENTIAL_RESTRICTION`, the function looks for a REMARK condition (opcode `1`) and reads `authorized_providers` and `proofs_checker` directly from it — with no validation of their content. An attacker crafts a standard CAT spend whose inner puzzle outputs:

```
(1  <victim_inner_puzzle_hash>  <attacker_providers_list>  <attacker_proofs_checker>)
```

and creates an output coin with puzzle hash = `CRCAT_puzzle(asset_id, attacker_providers, attacker_proofs_checker, victim_inner_puzzle_hash)`, hinted to `victim_inner_puzzle_hash`.

**Step 2 – `fetch_coin_spend` does not block this.** [2](#0-1) 

`fetch_coin_spend` only validates that the returned puzzle hashes to the coin's actual `puzzle_hash`. Because the attacker created a real on-chain coin with the CRCAT puzzle hash, this check passes. The puzzle reveal is authentic; only the REMARK condition's payload is attacker-controlled.

**Step 3 – Inner puzzle ownership check passes.** [3](#0-2) 

The wallet verifies it controls `crcat.inner_puzzle_hash`. Since the attacker set `inner_puzzle_hash` to the victim's own derivation puzzle hash (obtained from on-chain hints), this check passes.

**Step 4 – No existing CRCAT wallet → finds existing CATWallet → converts it.** [4](#0-3) 

Because the victim has a `CATWallet` (not yet a `CRCATWallet`) for this `asset_id`, the code falls through to the conversion branch and calls: [5](#0-4) 

`convert_to_cr` overwrites the wallet record in the database with `WalletType.CRCAT` and stores the attacker-supplied `authorized_providers` and `proofs_checker` permanently.

---

### Impact Explanation

1. **Existing CAT coins become inaccessible.** After conversion, the wallet is a `CRCATWallet`. Its coin-selection queries use `CoinType.CRCAT`, but the victim's pre-existing coins are stored as `CoinType.CAT`. They are no longer returned by `get_unspent_coins_for_wallet`, making them unspendable through the wallet UI.

2. **Future CRCAT coins require attacker-chosen authorization.** Any subsequent CRCAT coins received by this wallet will be governed by the attacker's `authorized_providers`, meaning the attacker controls which VCs can authorize spends.

3. **Irreversible without manual DB intervention.** `convert_to_cr` writes directly to the user store; there is no undo path in the wallet.

---

### Likelihood Explanation

- The attacker needs only a small amount of the target CAT (publicly purchasable for any traded token).
- The victim's puzzle hash is visible on-chain from prior transactions.
- The attack requires one standard blockchain transaction (normal fees, no special privileges).
- The victim's wallet triggers automatically on sync with no user interaction required.

---

### Recommendation

1. **Validate `authorized_providers` against a known-good source before conversion.** Do not accept CR parameters from an arbitrary on-chain REMARK condition as grounds to mutate an existing wallet. At minimum, require explicit user confirmation before converting a `CATWallet` to `CRCATWallet`.

2. **Separate "wallet discovery" from "wallet mutation."** `handle_cat` should only *create* a new CRCAT wallet for a previously unknown asset_id, never silently *convert* an existing one based on peer-supplied data.

3. **Ignore REMARK-sourced CR params for conversion.** The conversion path at lines 1297–1312 should be gated on a user-initiated action or a trusted configuration source, not on data extracted from an arbitrary coin spend.

### Proof of Concept

```python
# Attacker holds some CAT coins with asset_id matching victim's CATWallet.
# Victim's inner puzzle hash (p2_puzzle_hash) is known from on-chain hints.

attacker_providers = [bytes32(b"\xde\xad" * 16)]  # attacker-controlled
attacker_proofs_checker = PROOF_FLAGS_CHECKER.curry([])  # trivially permissive

# Construct the CRCAT output puzzle hash
crcat_puzzle = construct_cat_puzzle(
    CAT_MOD,
    asset_id,
    construct_cr_layer(attacker_providers, attacker_proofs_checker, victim_inner_puzzle_hash),
)
crcat_puzzle_hash = crcat_puzzle.get_tree_hash()

# Inner puzzle of the attacker's CAT coin outputs:
#   (CREATE_COIN crcat_puzzle_hash amount [victim_inner_puzzle_hash])  <- hint
#   (REMARK victim_inner_puzzle_hash attacker_providers attacker_proofs_checker)
inner_solution = Program.to([
    [51, crcat_puzzle_hash, amount, [victim_inner_puzzle_hash]],
    [1, victim_inner_puzzle_hash, attacker_providers, attacker_proofs_checker],
])

# Submit this CAT spend to the blockchain.
# When victim's wallet syncs:
#   - hint subscription fires for victim_inner_puzzle_hash
#   - determine_coin_type → handle_cat → get_next_from_coin_spend reads REMARK
#   - convert_to_cr called with attacker_providers
#   - victim's CATWallet is now CRCATWallet with attacker-controlled providers
```

**Assertion:** After sync, `wallet_state_manager.wallets[cat_wallet_id]` is a `CRCATWallet` with `info.authorized_providers == attacker_providers`, and the victim's original CAT coins are no longer returned by `get_spendable_coins_for_wallet`.
