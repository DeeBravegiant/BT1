### Title
Proposer Deposits Permanently Locked When Non-Winning Update Proposals Are Cleared - (File: `crates/contract/src/update.rs`)

### Summary

When `do_update` executes a winning contract-upgrade proposal, it bulk-clears all other pending proposals via `self.entries.clear()` without refunding the storage-staking deposits paid by the proposers of those non-winning entries. Because `UpdateEntry` does not record the proposer's account ID or the deposit amount, there is no recovery path: the NEAR tokens are permanently locked in the contract balance.

### Finding Description

`propose_update` in `lib.rs` requires each proposer to attach a deposit covering the storage cost of their proposal:

```rust
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// excess is refunded, but the required portion stays in the contract
``` [1](#0-0) 

The deposit is consumed by the contract to stake the storage used by the `UpdateEntry`. The entry itself stores only the update payload and `bytes_used`—not the proposer's identity or the deposit amount:

```rust
self.entries.insert(id, UpdateEntry { update, bytes_used });
``` [2](#0-1) 

When threshold votes are reached and `do_update` fires, it removes the winning entry and then unconditionally clears every remaining entry:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
``` [3](#0-2) 

No refund promise is issued for the cleared entries. Because `UpdateEntry` never recorded who paid or how much, the contract has no information with which to issue refunds even if the code were patched post-hoc. The freed storage stake flows back into the contract's own balance and is irrecoverable by the original proposers.

There is no `withdraw_proposal` or equivalent function anywhere in the contract, and no governance path to drain the contract balance back to depositors.

### Impact Explanation

Every participant who proposed a non-winning update loses their full required deposit. For a full WASM binary proposal the deposit can reach tens of NEAR (sandbox tests use `NearToken::from_near(40)` for a near-maximum-size contract). For config-only proposals the deposit is smaller (~0.1 NEAR) but still non-trivial. Because multiple proposals can coexist simultaneously (the codebase explicitly supports this), a single `vote_update` execution can lock the deposits of every other proposer in one atomic sweep, with no recovery mechanism.

This breaks the production accounting invariant that storage-staking deposits are returned when the staked storage is freed.

### Likelihood Explanation

The scenario requires only that two or more participants independently propose different updates—a normal operational pattern during governance (e.g., one participant proposes a code upgrade while another proposes a config change). No collusion, no privileged access, and no external dependency is needed. The loss is triggered automatically by the threshold-vote execution path that every participant can reach.

### Recommendation

1. Add a `proposer: AccountId` and `deposit: NearToken` field to `UpdateEntry` at proposal time.
2. In `do_update`, before calling `self.entries.clear()`, iterate over the remaining entries and issue a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one.
3. Alternatively, expose a `withdraw_proposal(id)` method that lets a proposer reclaim their deposit and remove their own entry before a competing proposal is executed.

### Proof of Concept

1. Participant A calls `propose_update` with a WASM binary, attaching 10 NEAR as the required deposit. `UpdateEntry { update: wasm_A, bytes_used: N }` is stored; A's 10 NEAR is now in the contract balance.
2. Participant B calls `propose_update` with a different WASM binary, attaching 10 NEAR. A second `UpdateEntry` is stored; B's 10 NEAR is now in the contract balance.
3. Threshold participants vote for A's proposal. `vote_update` calls `do_update(&id_A, gas)`.
4. `do_update` removes entry A, then calls `self.entries.clear()` — entry B is deleted with no refund promise.
5. B's 10 NEAR is permanently locked in the contract. There is no function to retrieve it. [4](#0-3) [5](#0-4)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
