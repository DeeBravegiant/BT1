Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks All Withdrawal Initiations Until Admin Intervention - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal` evaluates `0 >= 0 = true` and reverts for every caller, blocking all withdrawal initiations. Any user who stakes KERNEL tokens before the admin calls `setMaxNumberOfWithdrawalsPerUser` will have their funds temporarily frozen.

## Finding Description
`KernelDepositPool.initialize()` (L259–271) sets `kernelToken`, `rewardsToken`, and access-control roles but never assigns `maxNumberOfWithdrawalsPerUser`. The state variable therefore holds its zero default. Every call to `initiateWithdrawal` (L320–338) hits the guard at L323:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

For any fresh caller, `userWithdrawalIds[msg.sender].length == 0` and `maxNumberOfWithdrawalsPerUser == 0`, so `0 >= 0` is immediately `true` and the function reverts with `WithdrawalLimitReached`. Meanwhile, `stake()` (L281–289) has no analogous guard and accepts any non-zero amount, so users can lock real value into the contract immediately after deployment. The only exit path — `initiateWithdrawal` — is unconditionally blocked. The setter `setMaxNumberOfWithdrawalsPerUser` (L614) explicitly rejects `0` as invalid (`_maxNumberOfWithdrawalsPerUser == 0` → revert), confirming that `0` is not a valid operational value, yet the initializer leaves the contract in exactly that invalid state.

## Impact Explanation
Any user who stakes KERNEL tokens via `stake()` or `stakeFor()` cannot recover their principal until the admin calls `setMaxNumberOfWithdrawalsPerUser` with a non-zero value. The staked balance is held inside the contract and the only exit path is unconditionally blocked. This constitutes **temporary freezing of funds** (Medium).

## Likelihood Explanation
The freeze is automatic and affects every user from the moment the contract is deployed. No attacker action is required; any honest user who stakes before the admin sets the limit will find their funds locked. The likelihood is certain for any deployment that does not call `setMaxNumberOfWithdrawalsPerUser` as an immediate post-deployment step, and there is no on-chain enforcement requiring that call.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER;
```

This mirrors the pattern used in `LRTDepositPool` where analogous limits are set during initialization, and ensures the contract is never deployed in an operationally invalid state.

## Proof of Concept
1. Deploy `KernelDepositPool` and call `initialize(admin, kernelToken, rewardToken)`. Do **not** call `setMaxNumberOfWithdrawalsPerUser`.
2. Any user calls `stake(1e18)` — succeeds; `balanceOf[user] = 1e18`.
3. Same user calls `initiateWithdrawal(1e18)`.
4. Execution reaches `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser` → `0 >= 0` → `true` → reverts with `WithdrawalLimitReached`.
5. The user's `1e18` KERNEL is inaccessible until the admin calls `setMaxNumberOfWithdrawalsPerUser(n)` where `n > 0`.
