Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Bypass of the ETH Deposit Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for LST assets it evaluates `totalAssetDeposits + amount > depositLimit`, but for ETH it evaluates only `totalAssetDeposits > depositLimit`, omitting `amount`. Any unprivileged caller can therefore deposit arbitrarily large ETH amounts in a single `depositETH` call, pushing total ETH deposits well beyond the protocol-configured cap while rsETH is minted for the full amount.

## Finding Description
The sole deposit-limit gate is `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682). The ETH branch returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)`, which is `false` whenever the running total is at or below the cap — regardless of how large `amount` is. The LST branch correctly adds `amount` before comparing. `_beforeDeposit` (lines 648–670) calls this function and reverts with `MaximumDepositLimitReached` only when it returns `true`; because the ETH branch never returns `true` for a deposit that merely *reaches* the cap, the revert never fires. `depositETH` (lines 76–93) is `external payable` with no role restriction, so any caller can supply an arbitrarily large `msg.value`, pass the check, and receive rsETH for the full amount.

## Impact Explanation
The deposit cap for ETH is the primary risk-management control on how much ETH the protocol accepts and routes to EigenLayer. Bypassing it allows rsETH to be minted beyond the intended cap, breaking the protocol's stated invariant that deposits are bounded. This matches the allowed impact **Low – Contract fails to deliver promised returns**: the cap constraint is not enforced for ETH, so the protocol does not deliver the risk-bounding behavior it promises. If the cap was set as a hard risk limit against EigenLayer slashing exposure, exceeding it escalates toward protocol insolvency.

## Likelihood Explanation
The entry point `depositETH` is fully permissionless (`external payable`, no role modifier). The bypass condition — `totalAssetDeposits <= depositLimit` — is the normal operating state of the protocol before the cap is reached. No special role, flash loan, front-running, or external dependency is required. Any ETH holder can trigger this at any time.

## Recommendation
Add `amount` to the ETH branch to mirror the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
-       return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+       return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Deploy/fork with `depositLimitByAsset(ETH_TOKEN) = 1000 ether` and seed `totalAssetDeposits(ETH_TOKEN) = 999 ether`.
2. Call `depositETH{value: 500 ether}(0, "")` from any EOA.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999 ether > 1000 ether` → `false` → no revert.
4. `_mintRsETH` mints rsETH for 500 ETH; total ETH deposits become 1499 ETH, 49.9% above the cap.
5. Repeat from step 2 to push deposits to any multiple of the cap.
