### Title
`vote_reshared` Does Not Reset `accept_requests` After Successful Resharing — (`File: crates/contract/src/lib.rs`)

---

### Summary

When `verify_tee()` sets `self.accept_requests = false` because kicking out expired-attestation participants would violate the threshold relation, and participants subsequently trigger a manual resharing via `vote_new_parameters` that completes via `vote_reshared`, the `accept_requests` flag is never restored to `true`. The contract re-enters `Running` state with a valid participant set but permanently rejects all user `sign`, `request_app_private_key`, `verify_foreign_transaction`, `respond`, `respond_ckd`, and `respond_verify_foreign_tx` calls with `TeeValidationFailed`.

---

### Finding Description

`verify_tee()` sets `self.accept_requests = false` on line 1737 when it detects that the surviving participants with valid attestations are too few to satisfy the threshold relation: [1](#0-0) 

This is a deliberate safety precaution. However, `vote_reshared()` — the function that transitions the contract from `Resharing` back to `Running` after a successful key resharing — never touches `accept_requests`: [2](#0-1) 

The only code paths that restore `accept_requests = true` are both inside `verify_tee()` itself (lines 1710 and 1743): [3](#0-2) 

`vote_new_parameters` and `vote_reshared` both use `voter_or_panic()` / `assert_caller_is_attested_participant_and_protocol_active()` for authorization, neither of which checks `accept_requests`. This means a resharing can be proposed and completed while `accept_requests = false`, and the flag is never cleared on the successful `Running` re-entry.

After `vote_reshared` completes, every user-facing request path hits the `accept_requests` guard: [4](#0-3) 

And every node response path hits the same guard independently: [5](#0-4) 

---

### Impact Explanation

After a successful resharing that resolves the participant-set problem, the contract is permanently locked: no signature requests can be submitted, no CKD requests can be processed, and no MPC node can deliver a response. The MPC network is operationally dead until an operator discovers the root cause and manually calls `verify_tee()` again. This breaks the production safety invariant that a successfully reshared contract with a valid participant set must accept requests.

**Impact class:** Medium — contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

The trigger sequence is realistic in production:

1. One or more participants' TEE attestations expire naturally (7-day renewal cycle per the design doc).
2. The expired participants happen to be the ones whose removal would push the surviving set below the threshold relation (e.g., a 5-of-5 or near-threshold configuration).
3. Any participant calls `verify_tee()` — this is a routine scheduled call — setting `accept_requests = false`.
4. Operators respond by proposing a new participant set via `vote_new_parameters` (adding a replacement node or adjusting the threshold).
5. Resharing completes via `vote_reshared`.
6. The contract is now Running with a valid set but `accept_requests = false`.

Steps 3–5 are the normal operational recovery procedure. No adversarial action beyond waiting for attestation expiry is required.

---

### Recommendation

In `vote_reshared`, after assigning the new Running state, unconditionally reset `accept_requests = true`:

```rust
if let Some(new_state) = self.protocol_state.vote_reshared(key_event_id)? {
    self.protocol_state = new_state;
    self.accept_requests = true;   // ← add this line
    self.recompute_available_foreign_chains();
    // ... cleanup promises ...
}
```

Apply the same fix to `vote_cancel_resharing` when it reverts to the previous `RunningContractState`, since that path also transitions back to Running without touching `accept_requests`.

---

### Proof of Concept

```
State: Running, N=5, GovernanceThreshold=5, accept_requests=true
       (all 5 participants needed; one domain with ReconstructionThreshold=5)

Step 1: Participant P5's attestation expires.

Step 2: Any participant calls verify_tee().
        → reverify_and_cleanup_participants returns Partial{4 valid participants}
        → validate_governance_against_reconstruction fails (5 > 4)
        → accept_requests = false                          [lib.rs:1737]
        → protocol_state stays Running

Step 3: Operators add a new participant P6 and call vote_new_parameters
        with {P1,P2,P3,P4,P6}, threshold=5.
        → vote_new_parameters does NOT check accept_requests
        → threshold votes accumulate → Resharing state entered

Step 4: Resharing completes; all new participants call vote_reshared.
        → protocol_state = RunningContractState{P1..P4,P6}  [lib.rs:1172]
        → accept_requests is NEVER set to true              [lib.rs:1170-1236]

Step 5: User calls sign(...).
        → check_request_preconditions hits:
            if !self.accept_requests { panic!("TeeValidationFailed") }
        → ALL sign/ckd/verify_foreign_tx calls fail permanently.

Step 6: MPC node calls respond(...).
        → if !self.accept_requests { return Err(TeeValidationFailed) }
        → ALL respond/respond_ckd/respond_verify_foreign_tx calls fail.

Recovery requires an operator to discover the issue and call verify_tee()
again — an action not part of any automated post-resharing flow.
```

        if let Some(new_state) = self.protocol_state.vote_reshared(key_event_id)? {
