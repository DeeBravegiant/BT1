### Title
Unchecked `beneficiary_id` in `action_delete_account` Causes Permanent Token Burn When Beneficiary Does Not Exist - (File: runtime/runtime/src/actions.rs)

### Summary

`action_delete_account` unconditionally emits a `Receipt::new_balance_refund` to the caller-supplied `beneficiary_id` without verifying that the beneficiary account exists on-chain. Because the resulting refund receipt carries `predecessor_id = "system"`, any failure during its execution causes the transferred balance to be permanently burnt rather than returned. An unprivileged user who specifies a nonexistent (or deleted) named account as `beneficiary_id` will have their entire account balance destroyed with no recovery path.

### Finding Description

In `action_delete_account` (`runtime/runtime/src/actions.rs`, lines 349–354), after all guard checks pass, the function unconditionally pushes a balance-refund receipt addressed to `delete_account.beneficiary_id`:

```rust
let account_balance = account_ref.amount();
if account_balance > Balance::ZERO {
    result
        .new_receipts
        .push(Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance));
}
``` [1](#0-0) 

`Receipt::new_balance_refund` constructs a system receipt (`predecessor_id = "system"`) containing a single `Transfer` action directed at `beneficiary_id`: [2](#0-1) 

When this system receipt is later applied, `apply_action_receipt` reaches `check_account_existence` for the `Transfer` action. If the account does not exist, `check_transfer_to_nonexisting_account` is called: [3](#0-2) 

The function explicitly notes that refund receipts do **not** create implicit accounts:

```
// - Refunds don't automatically create accounts, because refunds are free and
// we don't want some type of abuse.
// - Account deletion with beneficiary creates a refund, so it'll not create a
// new account.
``` [4](#0-3) 

So the refund receipt fails with `AccountDoesNotExist`. Back in `apply_action_receipt`, the branch for system-predecessor receipts handles this failure by burning the deposit:

```rust
let gas_refund_result = if receipt.predecessor_id().is_system() {
    // If the refund fails tokens are burned.
    if result.result.is_err() {
        stats.balance.other_burnt_amount = safe_add_balance(
            stats.balance.other_burnt_amount,
            total_deposit(&action_receipt.actions())?,
        )?
    }
    GasRefundResult::default()
``` [5](#0-4) 

The protocol documentation confirms this: "If the execution of a refund fails, the refund amount is burnt." [6](#0-5) 

The account has already been removed from state at line 356 before the refund receipt is emitted: [7](#0-6) 

There is no rollback of the account deletion when the beneficiary refund fails. The tokens are permanently destroyed.

The only validation performed on `beneficiary_id` is a syntactic account-ID format check inside `validate_action_receipt`: [8](#0-7) 

No existence check is performed.

### Impact Explanation

The exact corrupted protocol value is the **account balance** of the deleted account: it should be credited to `beneficiary_id` but is instead permanently burnt (removed from total supply). The state root after the block will reflect a lower total balance than expected, with no corresponding validator reward or fee. The user's funds are irrecoverably destroyed.

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a `DeleteAccount` transaction with a `beneficiary_id` that:
- Names a non-existent named account (e.g., a typo, a future account that was never created, or an account that was deleted between signing and execution), or
- Names any account that is deleted in the same block before the refund receipt is processed.

A contract calling `promise_batch_action_delete_account` with a computed or user-supplied `beneficiary_id` faces the same risk. The scenario is realistic: users routinely specify beneficiary accounts by name, and cross-shard timing means the beneficiary could be deleted between the `DeleteAccount` receipt being created and the refund receipt being applied.

### Recommendation

Before emitting the balance-refund receipt, validate that `beneficiary_id` exists in the current state, or alternatively reject the `DeleteAccount` action at validation time if `beneficiary_id` is not a currently-existing account. A lighter-weight mitigation is to fall back to the `predecessor_id` (the receipt sender) when the beneficiary does not exist, mirroring how deposit refunds already fall back to `predecessor_id`.

### Proof of Concept

1. Alice has account `alice.near` with 10 NEAR balance.
2. Alice submits `DeleteAccount { beneficiary_id: "nonexistent.near" }` where `nonexistent.near` does not exist.
3. Runtime executes `action_delete_account`: removes `alice.near` from state, emits `Receipt::new_balance_refund("nonexistent.near", 10 NEAR)`.
4. The refund receipt (system predecessor) is applied on the shard hosting `nonexistent.near`.
5. `check_account_existence` returns `AccountDoesNotExist` for the `Transfer` action.
6. `apply_action_receipt` detects `predecessor_id().is_system()` and `result.result.is_err()`, adds 10 NEAR to `other_burnt_amount`.
7. Alice's 10 NEAR is permanently burnt. `nonexistent.near` is never created. No recovery is possible.
