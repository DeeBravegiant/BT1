## Title
Missing Minimum Output Protection on All L2 Pool Deposit Functions â€” (File: contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

## Summary
Every L2 pool `deposit()` function computes the output wrsETH/rsETH amount using a live oracle rate at execution time but accepts no `minRSETHAmountExpected` parameter, so a depositor cannot set a floor on the tokens they receive. The L1 `LRTDepositPool` enforces this protection via `_beforeDeposit`; the L2 pools do not. Any oracle rate update between transaction submission and execution silently delivers fewer tokens to the depositor with no on-chain recourse.

## Finding Description
`LRTDepositPool._beforeDeposit` enforces a slippage floor at lines 667â€“669:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [1](#0-0) 

This check is absent from every L2 pool variant. In `RSETHPoolV3.viewSwapRsETHAmountAndFee`, the output is computed as:

```solidity
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [2](#0-1) 

`getRate()` is fetched live from the oracle at execution time. The ETH deposit path in `RSETHPool.deposit` transfers the computed amount directly with no floor check: [3](#0-2) 

The same pattern is present in `RSETHPoolV3.deposit` (ETH and token overloads): [4](#0-3)

**Exploit path:**
1. Depositor observes `rsETHToETHrate = R` and submits `deposit{value: X}`.
2. Before the transaction is mined, the protocol oracle updates the rate to `R' > R`.
3. Transaction executes: `rsETHAmount = X * 1e18 / R'`, which is strictly less than `X * 1e18 / R`.
4. Depositor receives fewer tokens than expected with no revert and no on-chain protection.

No existing guard prevents this. The `nonReentrant` and `whenNotPaused` modifiers and the `limitDailyMint` modifier in V3 variants address entirely different concerns and do not bound the output amount.

## Impact Explanation
**Low â€” Contract fails to deliver promised returns, but does not lose value.** The deposited ETH or LST is retained by the pool (to be bridged to L1), so no funds are destroyed. However, the depositor receives fewer liquid restaking tokens than the protocol implicitly promised at submission time. This matches the allowed impact: "Low. Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
**Medium.** The rsETH oracle rate is updated periodically as part of normal protocol operation â€” no attacker action is required. On L2 chains with public mempools or sequencer ordering, a pending oracle update is observable, but even without adversarial ordering, routine rate updates during normal block production are sufficient to trigger the shortfall. The risk compounds as new tokens and oracles are added via `addSupportedToken`, since each new oracle introduces an additional rate-drift surface with no slippage floor. Any depositor on any of the six affected L2 pool contracts is exposed on every deposit.

## Recommendation
Add a `minRSETHAmountExpected` parameter to every `deposit()` overload across all six L2 pool contracts and revert if the computed output falls below it, mirroring the existing L1 pattern:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same change to all token deposit overloads. The error `MinimumAmountToReceiveNotMet` already exists in the L1 codebase and can be reused.

## Proof of Concept
1. Oracle reports `rsETHToETHrate = 1.05e18`.
2. User submits `RSETHPoolV3.deposit{value: 1 ether}("ref")`, expecting `â‰ˆ 0.952 wrsETH`.
3. Before the transaction is mined, the oracle updates to `rsETHToETHrate = 1.10e18`.
4. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 â‰ˆ 0.909 wrsETH`.
5. User receives `0.909 wrsETH` â€” a ~4.5% shortfall â€” with no revert.

**Foundry fork test plan:**
- Fork the target L2 chain.
- Deploy or reference the live `RSETHPoolV3` and its oracle.
- Record `getRate()` before deposit.
- Prank the oracle updater to increase the rate.
- Call `deposit{value: 1 ether}("ref")` and assert the received wrsETH balance is less than `1e18 * 1e18 / initialRate`.
- Confirm no revert occurs and no minimum-output check is enforced.
- Repeat for `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV2ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`.
