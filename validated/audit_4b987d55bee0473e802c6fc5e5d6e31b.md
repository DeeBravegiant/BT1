Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Incoming Amount in Limit Check - (File: `contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison: for LST assets it correctly checks `totalAssetDeposits + amount > limit`, but for ETH it checks only `totalAssetDeposits > limit`, omitting the incoming `amount`. This allows any depositor to bypass the ETH deposit cap entirely in a single transaction, depositing an arbitrarily large amount of ETH as long as the running total has not already exceeded the limit.

## Finding Description
The root cause is in `_checkIfDepositAmountExceedesCurrentLimit` at lines 676–682 of `contracts/LRTDepositPool.sol`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch never incorporates `amount` into the comparison. The check only returns `true` (triggering `MaximumDepositLimitReached`) when the cumulative total has *already* surpassed the limit from a prior deposit. Any deposit arriving while `totalAssetDeposits ≤ limit` passes unconditionally, regardless of how large `amount` is.

The call chain is: `depositETH` (L76–93) → `_beforeDeposit` (L648–670, passes `msg.value` as `depositAmount`) → `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, depositAmount)`. The guard at L661–663 that reverts on `true` is never reached because the ETH branch always returns `false` when the total is at or below the limit.

## Impact Explanation
The ETH deposit limit is a risk-management invariant intended to cap the protocol's EigenLayer exposure. Because the check is broken for ETH, the invariant is never enforced: a single depositor can push `totalAssetDeposits` to an arbitrary multiple of the intended ceiling in one transaction, receiving proportional rsETH. This matches the allowed impact **Low — contract fails to deliver promised returns** (the deposit-cap guarantee is not upheld). The scenario can escalate toward protocol insolvency if the limit was sized to bound slashing exposure, but the concrete, directly provable impact is the broken invariant.

## Likelihood Explanation
The entry point is the public, permissionless `depositETH` function. No privileged role, flash loan, front-running, or multi-step setup is required. The condition `totalAssetDeposits ≤ depositLimit` is trivially satisfied at protocol launch and after any reset, making the bypass available to any depositor at any time the cap has not already been breached.

## Recommendation
Include the incoming `amount` in the ETH branch, mirroring the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Set `depositLimitByAsset[ETH_TOKEN] = 1_000e18`; `getTotalAssetDeposits(ETH_TOKEN)` returns `0`.
2. Call `LRTDepositPool.depositETH{value: 100_000e18}(0, "")`.
3. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 100_000e18)`.
4. ETH branch evaluates `0 > 1_000e18` → `false` → no revert.
5. `_mintRsETH` mints rsETH proportional to 100 000 ETH.
6. `getTotalAssetDeposits(ETH_TOKEN)` now returns `100_000e18` — 100× the intended cap.

Foundry fuzz test plan: fuzz `amount` over `[depositLimit+1, type(uint128).max]` with `totalAssetDeposits = 0`; assert that `_checkIfDepositAmountExceedesCurrentLimit` returns `true` for all such inputs — the current code will falsify this assertion for every fuzzed value, confirming the bypass.
