Audit Report

## Title
Stale `rsETHPrice` silently propagated through `RSETHPriceFeed.latestRoundData` with misleading `updatedAt` timestamp — (`contracts/oracles/RSETHPriceFeed.sol`)

## Summary

`RSETHPriceFeed.latestRoundData` computes a composite rsETH/USD price by multiplying a stored `rsETHPrice` (from `LRTOracle`) by a live ETH/USD Chainlink price, but returns `updatedAt` sourced exclusively from the ETH/USD feed. Because `LRTOracle` stores no timestamp for when `rsETHPrice` was last written, the composite price can embed a stale rsETH/ETH rate while `updatedAt` appears fresh to any downstream consumer. This is exacerbated during protocol pauses, when the public `updateRSETHPrice()` path is blocked by `whenNotPaused`.

## Finding Description

In `contracts/oracles/RSETHPriceFeed.sol` lines 63–70, `latestRoundData` fetches all five return values — including `updatedAt` — from `ETH_TO_USD.latestRoundData()`, then overwrites only `answer` with the composite price:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

`RS_ETH_ORACLE.rsETHPrice()` reads the storage variable `uint256 public override rsETHPrice` declared at `contracts/LRTOracle.sol` line 28. This variable is written only inside `_updateRsETHPrice()` at line 313 (`rsETHPrice = newRsETHPrice`), which is reached via:

- `updateRSETHPrice()` (line 87) — public but gated by `whenNotPaused`
- `updateRSETHPriceAsManager()` (line 94) — manager-only, no pause gate

No timestamp is stored alongside `rsETHPrice`. The `updatedAt` returned by `latestRoundData` therefore reflects only the ETH/USD Chainlink heartbeat (typically ~1 hour), not the age of the rsETH/ETH component.

The pause-on-deviation logic at lines 277–281 calls `_pause()` when the price drops beyond `pricePercentageLimit`, which sets `paused = true`. While paused, `updateRSETHPrice()` reverts, freezing `rsETHPrice` at its pre-pause value. During this window, `latestRoundData` continues returning a fresh `updatedAt` from the ETH/USD feed while the rsETH/ETH rate is frozen.

## Impact Explanation

A downstream lending protocol using `RSETHPriceFeed` as a Chainlink-compatible oracle will call `latestRoundData()`, receive a fresh `updatedAt` (from the ETH/USD feed), pass its own staleness check (e.g., `require(block.timestamp - updatedAt < 3600)`), and use the returned `answer` as the rsETH/USD price — which silently embeds a potentially hours-to-days-old rsETH/ETH rate. The contract fails to deliver the promised accurate composite price with correct staleness metadata. This matches **Low: contract fails to deliver promised returns, but doesn't lose value** — the `RSETHPriceFeed` contract's core promise is to expose an accurate, Chainlink-compatible rsETH/USD price with valid staleness signaling; it fails to do so.

## Likelihood Explanation

Under normal operation, `updateRSETHPrice()` is public and callable by any keeper or bot, limiting staleness. However, the pause-on-deviation path (lines 270–281) is an on-chain, automatically triggered condition that blocks the public update path for the duration of the pause. A pause of hours to days is realistic given the existing logic. Even outside a pause, keeper infrastructure failure causes silent staleness accumulation with no on-chain signal. No privileged attacker is required; the condition arises from normal protocol operation.

## Recommendation

`LRTOracle` should record the timestamp of the last `rsETHPrice` update. Concretely:

1. Add `uint256 public rsETHPriceUpdatedAt` to `LRTOracle` and expose it via `IRSETHOracle`.
2. In `_updateRsETHPrice()`, set `rsETHPriceUpdatedAt = block.timestamp` alongside `rsETHPrice = newRsETHPrice` (line 313).
3. In `RSETHPriceFeed.latestRoundData` (and `getRoundData`), override `updatedAt` with `min(updatedAt, RS_ETH_ORACLE.rsETHPriceUpdatedAt())` so downstream consumers see the true staleness of the composite price.

## Proof of Concept

```solidity
// Local fork test — no mainnet interaction required
function test_staleRsETHPriceUpdatedAt() public {
    // Snapshot updatedAt before any warp
    (, , , uint256 updatedAtBefore, ) = rsETHPriceFeed.latestRoundData();

    // Warp 48 hours without calling updateRSETHPrice
    vm.warp(block.timestamp + 48 hours);

    // Mock ETH/USD feed to return a fresh updatedAt (block.timestamp)
    // (Chainlink heartbeat would have fired multiple times in 48h)
    mockEthUsdFeed.setUpdatedAt(block.timestamp);

    (, , , uint256 updatedAt, ) = rsETHPriceFeed.latestRoundData();

    // updatedAt appears fresh (within 1 hour), passes any standard staleness check
    assertLt(block.timestamp - updatedAt, 3600, "updatedAt appears fresh");

    // But rsETHPrice embedded in answer is 48h old — no on-chain signal of staleness
    uint256 storedRsETHPrice = lrtOracle.rsETHPrice();
    // storedRsETHPrice has not changed despite 48h elapsed
    assertEq(storedRsETHPrice, rsETHPriceAtStart, "rsETHPrice is stale");
}
```

The test demonstrates that `updatedAt` passes a standard 1-hour staleness check while the rsETH/ETH component embedded in `answer` is 48 hours old, with no on-chain mechanism to detect the divergence.
