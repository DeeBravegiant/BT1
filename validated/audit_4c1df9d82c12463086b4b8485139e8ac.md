### Title
IEEE 754 `Double` Arithmetic in `likelihood`/`leaderProbability` Produces Non-Deterministic `NonMyopic` State Serialized into Consensus `EpochState` - (File: `eras/shelley/impl/src/Cardano/Ledger/Shelley/PoolRank.hs`)

---

### Summary

The `likelihood` and `leaderProbability` functions use `Double` (IEEE 754 floating-point) arithmetic to compute `Likelihood` values that are stored in `NonMyopic`, which is a serialized field of `EpochState`. The code itself contains a `TODO` comment acknowledging floating-point precision concerns. The project's own `non-integer-calculations` specification document explicitly states that IEEE 754 floating-point is unsuitable for Cardano because it is non-deterministic across implementations and compiler settings. Since `EpochState` (including `esNonMyopic`) is CBOR-serialized and transmitted between nodes, divergent `Double` computations across different implementations or compiler configurations can produce different serialized ledger states, causing deterministic disagreement between honest nodes.

---

### Finding Description

**Step 1 — `leaderProbability` converts exact `Rational` stake to lossy `Double`:**

`leaderProbability` takes a `Rational` relative stake and converts it to `Double` via `realToFrac`, then performs IEEE 754 exponentiation:

```haskell
leaderProbability activeSlotCoeff relativeStake decentralizationParameter =
  (1 - (1 - asc) ** s) * (1 - d')
  where
    asc = realToFrac . unboundRational . activeSlotVal $ activeSlotCoeff
    s = realToFrac relativeStake   -- Rational → Double, lossy
``` [1](#0-0) 

**Step 2 — `likelihood` uses `Double` `log` with an explicit TODO about precision:**

The result `t :: Double` from `leaderProbability` is fed into `likelihood`, which computes 100 log-weight sample points using IEEE 754 `log`. The developer left an explicit unresolved concern:

```haskell
l x = n * log x + m * log (1 - t * x)
-- TODO: worry more about loss of floating point precision
``` [2](#0-1) 

**Step 3 — `Likelihood` values are stored in `NonMyopic`, which is a serialized field of `EpochState`:**

`NonMyopic` holds `likelihoodsNM :: VMap (KeyHash StakePool) Likelihood`, where each `Likelihood` is a sequence of 100 `LogWeight` values computed from `Double`. `NonMyopic` is a strict field of `EpochState`:

```haskell
data EpochState era = EpochState
  { ...
  , esNonMyopic :: !NonMyopic
  -- ^ This field, esNonMyopic, does not appear in the formal spec
  -- and is not a part of the protocol. It is only used for providing
  -- data to the stake pool ranking calculation ...
  }
``` [3](#0-2) 

Despite the comment, `esNonMyopic` is **explicitly included** in the `EncCBOR` instance for `EpochState`:

```haskell
encCBOR EpochState {esChainAccountState, esLState, esSnapshots, esNonMyopic} =
  encode $ Rec EpochState
    !> To esChainAccountState
    !> To esLState
    !> To esSnapshots
    !> To esNonMyopic   -- ← serialized into the ledger state
``` [4](#0-3) 

**Step 4 — `newLikelihoods` computed from `Double` are written into the reward update and then into `NonMyopic`:**

In `startStep`, `makeLikelihoods` calls `leaderProbability` (returning `Double`) and `likelihood` (consuming `Double`), producing `newLikelihoods` stored in `rewLikelihoods` of `RewardSnapShot`, which is later applied to update `esNonMyopic`: [5](#0-4) 

**Step 5 — The project's own specification document explicitly forbids IEEE 754 for this purpose:**

`docs/non-integer-calculations/non-integer-calculations.tex` states:

> "In all these cases, it is important that all distributed nodes calculate the same values, since otherwise there might be a disagreement about what should be included in the blockchain and forks might then result."

It then explicitly identifies IEEE 754 floating-point as unsuitable because elementary functions are not standardized, results vary with compiler settings, and rounding modes differ across implementations. [6](#0-5) [7](#0-6) 

The consensus-critical leader check (`checkLeaderNatValue`) correctly uses `FixedPoint` arithmetic via `taylorExpCmp` from `libs/non-integral`. The `likelihood`/`leaderProbability` path does not. [8](#0-7) 

---

### Impact Explanation

**Impact: High — Deterministic disagreement between honest nodes from ledger rule evaluation, era transition, or serialization.**

`EpochState` (including `esNonMyopic`) is CBOR-serialized and transmitted between nodes. If two honest nodes running different implementations (e.g., a future alternative client, or even the same GHC binary compiled with different optimization flags affecting floating-point intermediate register use) compute different `Double` values for `log` or `**`, they produce different `Likelihood` sequences, different `NonMyopic` values, and therefore different serialized `EpochState` bytes. Any mechanism that compares serialized ledger states — including snapshot validation, state synchronization, or ledger state hashing — would observe a divergence. The contradiction between the comment ("not a part of the protocol") and the `EncCBOR` instance (which serializes it unconditionally) is itself a latent correctness hazard: future code that hashes or compares the full `EpochState` would silently include this non-deterministic field.

---

### Likelihood Explanation

**Likelihood: Medium.**

Within a single GHC-compiled Haskell binary on a fixed platform, IEEE 754 `Double` arithmetic is deterministic. The risk materializes concretely when: (a) an alternative ledger implementation is developed (the project explicitly anticipates multiple implementations in the non-integer-calculations document); (b) GHC optimization levels or LLVM backend settings change the order of floating-point operations; or (c) the `esNonMyopic` field is included in any future ledger state hash. The `TODO` comment in the production code is direct evidence that the developers themselves recognized this as an unresolved concern.

---

### Recommendation

1. Replace `Double` arithmetic in `leaderProbability` and `likelihood` with `FixedPoint` (already used in `checkLeaderNatValue`) or exact `Rational` arithmetic, consistent with the project's own non-integer-calculations specification.
2. If `esNonMyopic` is genuinely not part of the protocol, remove it from the `EncCBOR` instance for `EpochState` to prevent it from contaminating any future ledger state hash.
3. Add deterministic unit tests for `likelihood` with fixed inputs and expected outputs to catch regressions from arithmetic changes, analogous to the golden tests recommended in the non-integer-calculations document.

---

### Proof of Concept

```
Node A (GHC -O2, SSE2):
  leaderProbability asc sigma d  →  t_A :: Double
  likelihood n t_A epochSize     →  Likelihood [LogWeight w_A_0, ..., w_A_99]

Node B (alternative implementation or -O0):
  leaderProbability asc sigma d  →  t_B :: Double  (t_B ≠ t_A due to FP rounding)
  likelihood n t_B epochSize     →  Likelihood [LogWeight w_B_0, ..., w_B_99]

EpochState serialization:
  encCBOR esNonMyopic_A  ≠  encCBOR esNonMyopic_B
  → different serialized EpochState bytes
  → deterministic disagreement on any comparison of serialized ledger state
```

The entry path is fully attacker-independent: it is triggered by normal epoch-boundary reward computation for any registered stake pool, requiring no special privileges.

\subsection{IEEE~754 Floating Point}
\label{sec:ieee-754-floating}

A straightforward approach is to use IEEE~754 floating-point arithmetic.  This is
widely supported by common programming languages and CPU/GPU architectures.
%
The basic arithmetic operations of IEEE~754 floating-point are well specified,
and they are usually very efficient due to direct HW implementation.
%
Double-precision IEEE~754 floating-point numbers (\emph{binary64}) provide 53 bits (~15.99 decimal digits) of
precision.  This is slightly less than is needed to represent the
required fraction of 1 lovelace: there are $4.5\cdot10^{16}$ possible fractions.
Moreover, the  elementary functions that are required for Cardano are not standardized and there can be
subtle differences in different implementations, with results even varying when different compiler settings are used
on the same implementation\footnote{e.g. maintaining intermediate representations within internal registers \emph{vs.} writing to memory.}. It is also worth noting that some architectures
provide excess precision~\footnote{The original x87 provided 80 bit
  floating-point}, which can result in greater accuracy, but can also give slight differences in results
between implementations. Some programming languages, e.g., D, exploit this by default. Many
languages support excess precision, but default to IEEE~754 64 bits. In
particular on the AMD64 architecture most language implementations use SSE floating-point arithmetic.
A final problem is that IEEE~754 supports different rounding modes\footnote{round to nearest (with even digits breaking ties or ties rounding away from zero), round down, round up, round to zero},  which will obviously give
different results in some cases.  Some programming languages allow the rounding mode to be changed, e.g. C99 onwards, but others do
not, e.g., Haskell.

checkLeaderNatValue bn σ f =
  if activeSlotVal f == maxBound
    then -- If the active slot coefficient is equal to one,
    -- then nearly every stake pool can produce a block every slot.
    -- In this degenerate case, where ln (1-f) is not defined,
    -- we let the VRF leader check always succeed.
    -- This is a testing convenience, the active slot coefficient should not
    -- bet set above one half otherwise.
      True
    else case taylorExpCmp 3 recip_q x of
      ABOVE _ _ -> False
      BELOW _ _ -> True
      MaxReached _ -> False
  where

    recip_q =
      case nonZero . toInteger $ certNatMax - certNat of
        Just d -> fromRational (toInteger certNatMax %. d)
        Nothing -> fromIntegral @Natural @FixedPoint certNatMax
    x = -fromRational σ * c
