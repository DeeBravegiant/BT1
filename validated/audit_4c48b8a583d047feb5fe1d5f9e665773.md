### Title
Single Byzantine Participant Can Permanently Block All `verify_foreign_transaction` Requests via Empty Foreign-Chain Registration — (`crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which computes the **strict intersection** of every active participant's registered chain set. A single participant calling `register_foreign_chain_support` with an empty set immediately removes every chain from the supported set, causing all subsequent `verify_foreign_transaction` calls to revert. No threshold collusion is required; one Byzantine participant below the signing threshold is sufficient to permanently block the entire bridge verification flow.

### Finding Description

`get_supported_foreign_chains()` iterates over every active participant's registered chain set and returns only chains that **all** participants have registered:

```rust
// crates/contract/src/lib.rs  ~2203-2217
foreign_chain_to_node_mapping
    .into_iter()
    .filter_map(|(foreign_chain, nodes_supporting_chain)| {
        let all_active_nodes_supports_chain =
            nodes_supporting_chain.is_superset(&active_participant_account_ids);
        if all_active_nodes_supports_chain {
            Some(foreign_chain)
        } else {
            None
        }
    })
    ...
```

`verify_foreign_transaction` calls this function and panics if the requested chain is absent:

```rust
// crates/contract/src/lib.rs  ~533-542
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported { ... }.to_string(),
    );
}
```

Any participant can overwrite their own entry at any time via `register_foreign_chain_support`, which performs no threshold check:

```rust
// crates/contract/src/lib.rs  ~972-983
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);
    Ok(())
}
```

Submitting an empty `SupportedForeignChains` set causes `is_superset` to fail for every chain, collapsing the supported set to empty. The project's own design document explicitly acknowledges this:

> "A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down."
> — `docs/design/calculating-supported-foreign-chains.md`

The proposed fix (`get_available_foreign_chains()`, threshold-based) exists in the codebase but `verify_foreign_transaction` has **not** been migrated to use it; it still calls `get_supported_foreign_chains()`.

### Impact Explanation

Every `verify_foreign_transaction` call for any foreign chain reverts with `ForeignChainNotSupported`. Bridge inbound flows (foreign chain → NEAR) that depend on MPC-attested verification are permanently blocked. Funds locked in bridge contracts on foreign chains that require a signed MPC attestation to be released cannot be unlocked, constituting a permanent freezing of funds flowing through the bridge. The attack is persistent: the malicious participant can re-submit an empty registration whenever honest nodes restore their entries.

**Impact: Medium** — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

### Likelihood Explanation

Any single registered participant (one of N nodes, well below the signing threshold) can trigger this with a single on-chain call. No key material, no threshold collusion, and no special privilege beyond being a current participant is required. The only recovery path is a full resharing to remove the malicious participant, which itself requires threshold cooperation and takes time, during which the bridge remains blocked.

### Recommendation

Replace the `get_supported_foreign_chains()` gate in `verify_foreign_transaction` with `get_available_foreign_chains()`, which already implements the threshold-based availability check (`≥ signing_threshold` participants must cover the chain). This is the fix already designed and documented in `docs/design/calculating-supported-foreign-chains.md` — it just needs to be wired into the `verify_foreign_transaction` guard.

### Proof of Concept

1. Network is running with N participants, all having registered Bitcoin support. Bridge users can call `verify_foreign_transaction` for Bitcoin successfully.
2. Participant P₁ (a single node, no threshold needed) calls `register_foreign_chain_support` with an empty `SupportedForeignChains` set.
3. `get_supported_foreign_chains()` now computes `is_superset({P₁, …, Pₙ})` against the Bitcoin node set, which no longer contains P₁ → Bitcoin is dropped from the result.
4. Any subsequent `verify_foreign_transaction` call for Bitcoin hits the `!supported_chains.contains(&requested_chain)` check and panics with `ForeignChainNotSupported`.
5. All pending and future bridge verification requests for Bitcoin (and every other chain P₁ dropped) are permanently rejected until a resharing removes P₁ — a threshold-level operation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
