Audit Report

## Title
Share Price Inflation via Direct ETH Transfer Enables Zero-Mint Deposit, Stealing Victim Funds — (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary
An unprivileged attacker donates ETH directly to `LRTDepositPool` (via its open `receive()`) and calls the public `updateRSETHPrice()`. Because `pricePercentageLimit` defaults to zero, the price guard short-circuits and the stored `rsETHPrice` is inflated by an arbitrary factor in a single transaction. A subsequent victim deposit with `minRSETHAmountExpected = 0` computes `rsethAmountToMint = 0` (integer underflow from division), passes the slippage check silently, and the victim's ETH is permanently absorbed into the protocol while the attacker retains 100% of the rsETH supply.

## Finding Description

**Root cause 1 — raw balance accounting.**
`getETHDistributionData()` reads `address(this).balance` directly: [1](#0-0) 

`getAssetDistributionData()` reads `IERC20(asset).balanceOf(address(this))`: [2](#0-1) 

Both are inflatable by anyone because the contract has an open `receive()`: [3](#0-2) 

**Root cause 2 — public, uncapped price update.**
`updateRSETHPrice()` is callable by any address: [4](#0-3) 

The only guard against an unbounded price increase is: [5](#0-4) 

When `pricePercentageLimit == 0` (the storage default, set only by an admin call to `setPricePercentageLimit`), the condition short-circuits to `false` and the price update proceeds unconditionally. The new price is: [6](#0-5) 

With `rsethSupply = 1 wei` and `totalETHInProtocol ≈ 1.5e18`, `divWad` (= `x * 1e18 / y`) yields `newRsETHPrice ≈ 1.5e36`.

**Root cause 3 — no zero-mint floor.**
Mint amount is computed by dividing by the stored (now inflated) price: [7](#0-6) 

With `rsETHPrice ≈ 1.5e36` and a victim deposit of `1.5e18` wei ETH: `(1.5e18 × 1e18) / (1.5e36 + 1e18) = 1.5e36 / (1.5e36 + 1e18) = 0` (integer division, numerator < denominator).

`_beforeDeposit` only checks `rsethAmountToMint < minRSETHAmountExpected`; when both are zero the check passes: [8](#0-7) 

`RSETH.mint(to, 0)` also succeeds — `checkDailyMintLimit(0)` evaluates `0 + 0 > maxMintAmountPerDay` which is false for any value of `maxMintAmountPerDay`: [9](#0-8) 

**Fee minting does not block the attack.** Even when `protocolFeeInBPS > 0`, the fee in rsETH is `protocolFeeInETH.divWad(newRsETHPrice) = ~1.5e16 * 1e18 / 1.5e36 = 0`, so `_checkAndUpdateDailyFeeMintLimit(0)` is called and passes: [10](#0-9) 

## Impact Explanation

**Critical — direct theft of user funds.**
The victim deposits real ETH and receives zero rsETH. Their ETH is permanently held by the protocol. The attacker, holding the only non-zero rsETH balance (1 wei), owns 100% of the protocol's shares and can redeem all deposited funds — including the victim's — through the withdrawal system. This matches the allowed impact: *"Direct theft of any user funds, whether at-rest or in-motion."*

## Likelihood Explanation

**Medium.** `pricePercentageLimit` is a storage variable that defaults to zero and requires an explicit admin call to set. `minAmountToDeposit` also defaults to zero, allowing a 1-wei seed deposit. Front-end integrations and direct contract callers routinely pass `minRSETHAmountExpected = 0`. The attacker needs capital equal to the victim's deposit but recovers it (plus the victim's funds) upon redemption. The attack is atomic and repeatable.

## Recommendation

1. **Reject zero-mint deposits.** Add `require(rsethAmountToMint > 0, "ZeroMintAmount")` in `_beforeDeposit` or `getRsETHAmountToMint`.
2. **Use internal accounting instead of raw balances.** Track deposited ETH and LST amounts in storage variables rather than reading `address(this).balance` and `balanceOf`, so direct transfers cannot inflate the price.
3. **Enforce a non-zero `pricePercentageLimit` at initialization.** Require the admin to set it before `updateRSETHPrice()` is callable, or default it to a safe value (e.g., 1% = `1e16`).
4. **Seed the protocol with a non-trivial initial deposit** (dead-address mint) to make the rsETH supply large enough that a donation cannot meaningfully shift the price.

## Proof of Concept

```
State: rsethSupply=0, rsETHPrice=1e18, pricePercentageLimit=0, minAmountToDeposit=0

1. Attacker calls depositETH{value: 1 wei}(0, "")
   → rsethAmountToMint = (1 * 1e18) / 1e18 = 1
   → Attacker holds 1 wei rsETH; rsethSupply = 1

2. Attacker sends 1.5 ETH directly to LRTDepositPool (receive())
   → address(LRTDepositPool).balance = 1 + 1.5e18

3. Attacker calls LRTOracle.updateRSETHPrice()
   → totalETHInProtocol = 1 + 1.5e18
   → previousTVL = 1 * 1e18 / 1e18 = 1
   → protocolFeeInETH rounds to 0 (fee in rsETH = 0 at this price)
   → newRsETHPrice = (1.5e18 + 1) * 1e18 / 1 ≈ 1.5e36
   → pricePercentageLimit == 0 → no revert
   → rsETHPrice stored = ~1.5e36

4. Victim calls depositETH{value: 1.5 ETH}(0, "")
   → rsethAmountToMint = (1.5e18 * 1e18) / (1.5e36 + 1e18)
                       = 1.5e36 / (1.5e36 + 1e18) = 0 (integer division)
   → minRSETHAmountExpected = 0 → 0 < 0 is false → no revert
   → RSETH.mint(victim, 0) → victim receives 0 rsETH
   → Victim's 1.5 ETH is absorbed into the protocol

5. Attacker holds 1 wei rsETH = 100% of supply
   → Attacker redeems ~3 ETH (own deposit + donation + victim's deposit)
   → Net profit ≈ 1.5 ETH
```

Foundry test plan: deploy `LRTDepositPool` + `LRTOracle` + `RSETH` on a local fork with `pricePercentageLimit = 0`; execute steps 1–5 above; assert `rsETH.balanceOf(victim) == 0` and `rsETH.balanceOf(attacker) == 1`.
