### Title
Harvester Can Unconditionally Override Farmer Block Reward Address Without Fee Quality Enforcement - (File: `chia/farmer/farmer_api.py`)

### Summary
A harvester connected to a farmer can redirect the farmer's block reward (XCH) to any arbitrary puzzle hash by setting `farmer_reward_address_override` in a `RespondSignatures` message. The farmer accepts this override unconditionally with no enforcement of the CHIP-22 fee quality threshold. The only check that exists is advisory logging, which does not block the reward diversion.

### Finding Description

CHIP-22 defines a fee convention where a third-party harvester may take a farmer's block reward as a fee, but **only** when a deterministic fee quality threshold is met (`fee_quality <= applied_fee_threshold`). The farmer is supposed to enforce this gate.

In `_process_respond_signatures` in `chia/farmer/farmer_api.py`, when the farmer receives a `RespondSignatures` message from a harvester, it unconditionally replaces its own configured reward address with whatever the harvester supplies: [1](#0-0) 

There is no check here that:
1. The fee quality threshold was actually met.
2. The `farmer_reward_address_override` in `RespondSignatures` matches what was declared in `NewProofOfSpace`.
3. Any authorization from the farmer is required.

The only fee quality check exists in `notify_farmer_reward_taken_by_harvester_as_fee`, called from `new_proof_of_space`: [2](#0-1) 

But that function only **logs a warning** — it never rejects the override or prevents the block signing flow from proceeding: [3](#0-2) 

Furthermore, a harvester can set `farmer_reward_address_override=None` in `NewProofOfSpace` (bypassing even the warning log) and then set it to an attacker-controlled address in `RespondSignatures`, since the two fields are checked independently with no cross-validation.

The `RespondSignatures` protocol message carries the override field: [4](#0-3) 

### Impact Explanation

When a harvester wins a block, the farmer constructs a `DeclareProofOfSpace` message using `farmer_reward_address` derived from the harvester's override: [5](#0-4) 

This `farmer_reward_address` is the puzzle hash that receives the farmer's block reward (XCH). By setting it to an attacker-controlled address, the harvester permanently diverts the farmer's block reward for every block won while the harvester is connected. This is a direct, irreversible loss of XCH for the farmer.

**Impact class:** High — unauthorized payout redirection of XCH block rewards by a linked entity (harvester) that bypasses the intended authorization gate (fee quality threshold).

### Likelihood Explanation

The attacker must operate a harvester that is connected to a victim farmer. This is a common and expected operational setup (remote harvesters, third-party harvester software, co-located harvesters). The attacker needs no leaked keys, no admin access, and no cryptographic break — only a valid network connection to the farmer's harvester port and plots with valid keys. The attack triggers automatically on every block won.

### Recommendation

In `_process_respond_signatures`, before accepting `response.farmer_reward_address_override`, enforce the CHIP-22 fee quality convention: compute `fee_quality = calculate_harvester_fee_quality(pospace.proof, response.challenge_hash)` and verify that `fee_quality <= applied_fee_threshold` from the corresponding `NewProofOfSpace.fee_info`. If the threshold is not met or `fee_info` is absent, reject the override and use `self.farmer.farmer_target` instead. Additionally, validate that the override address in `RespondSignatures` matches the one declared in `NewProofOfSpace` for the same proof.

### Proof of Concept

1. Attacker operates a harvester with valid plots connected to a victim farmer.
2. When the harvester finds a proof of space that wins a block, it sends `NewProofOfSpace` with `farmer_reward_address_override=None` and `fee_info=None` (no warning logged).
3. The farmer sends `RequestSignatures` back to the harvester.
4. The harvester responds with `RespondSignatures` where `farmer_reward_address_override` is set to the attacker's puzzle hash.
5. In `_process_respond_signatures`, the farmer executes:
   ```python
   farmer_reward_address = self.farmer.farmer_target  # farmer's address
   if response.farmer_reward_address_override is not None:
       farmer_reward_address = response.farmer_reward_address_override  # attacker's address, no check
   ```
6. The farmer broadcasts `DeclareProofOfSpace` with `farmer_reward_address = attacker_puzzle_hash`.
7. The block is accepted by the network; the farmer's block reward (XCH) is paid to the attacker's address.
8. The farmer receives nothing. The attack repeats for every subsequent block won. [6](#0-5) [7](#0-6)

```python
    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
```
