Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` Operand in Bounds Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric bounds check: the ETH branch at line 679 evaluates `totalAssetDeposits > depositLimit` (omitting `amount`), while the ERC-20 branch at line 681 correctly evaluates `totalAssetDeposits + amount > depositLimit`. As a result, any unprivileged caller can push the protocol's ETH TVL arbitrarily above the configured deposit cap in a single `depositETH` call, causing rsETH to be minted beyond the intended safety ceiling.

## Finding Description
The sole deposit-cap guard is `_checkIfDepositAmountExceedesCurrentLimit` ( [1](#0-0) ). Its ETH branch checks only whether the **current** total already exceeds the limit, not whether the **post-deposit** total would:

```solidity
// L679 — ETH branch (buggy)
return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));

// L681 — ERC-20 branch (correct)
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

This guard is called unconditionally from `_beforeDeposit` ( [2](#0-1) ), which is invoked by every `depositETH` call ( [3](#0-2) ). No other check enforces the cap before rsETH is minted. Because `amount` is never added to `totalAssetDeposits` in the ETH branch, the check passes as long as the current total is still below the limit, regardless of how large the incoming deposit is.

## Impact Explanation
**Low — Contract fails to deliver promised returns.**

The deposit cap is an explicit protocol constraint. When `totalAssetDeposits = 999 ETH` and `depositLimit = 1 000 ETH`, a depositor can send 500 ETH in one transaction: `999 > 1 000` is `false`, so no revert occurs, rsETH is minted for the full 500 ETH, and `totalAssetDeposits` becomes 1 499 ETH — 49.9 % above the cap. The overshoot is bounded only by the depositor's balance. The protocol's risk exposure to EigenLayer strategies exceeds the operator-configured maximum, and `getAssetCurrentLimit` returns 0 after the fact, but the excess minting has already occurred.

## Likelihood Explanation
`depositETH` is public and permissionless — no role, whitelist, or special condition is required. Any depositor who observes `totalAssetDeposits` approaching `depositLimit` on-chain can exploit this in a single transaction. The condition is easy to monitor and the exploit is repeatable until the admin raises or removes the limit.

## Recommendation
Add `amount` to the ETH branch, mirroring the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1_000 ether`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 999 ether` through normal usage.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999 > 1_000` → `false` → no revert. ( [4](#0-3) )
5. `_mintRsETH` mints rsETH for the full 500 ETH. ( [5](#0-4) )
6. `getTotalAssetDeposits(ETH)` now returns `1_499 ether`, 49.9 % above the cap.
7. The ERC-20 path for the same scenario evaluates `999 + 500 > 1_000` → `true` → `MaximumDepositLimitReached` revert, correctly blocking the deposit. ( [6](#0-5) )

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    vm.deal(attacker, 500 ether);
    // precondition: 999 ETH already deposited
    // action
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 500 ether}(0, "");
    // assertion
    assertGt(lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN), 1_000 ether);
}
```
