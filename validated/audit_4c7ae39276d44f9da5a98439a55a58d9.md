Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Deposits Beyond the Configured Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses a strict `>` comparison without adding the incoming `amount` for the ETH branch, while the ERC-20 branch correctly adds `amount` before comparing. When `totalAssetDeposits == depositLimit`, the ETH check returns `false`, allowing `depositETH` to proceed and silently breach the governance-approved cap.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the function has two branches:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ETH branch omits `amount`. When `totalAssetDeposits == depositLimit`, the expression `totalAssetDeposits > depositLimit` evaluates to `false`, so `_beforeDeposit` (lines 648–670) does not revert at line 661–663, and `depositETH` (lines 76–93) mints rsETH for the full `msg.value`. After the call, `totalAssetDeposits` becomes `depositLimit + msg.value`, exceeding the cap. No privileged role or special condition is required beyond the cap being exactly reached, which is a normal operational state.

## Impact Explanation
The deposit limit is the protocol's primary risk-management ceiling on ETH restaked. Bypassing it allows the protocol to accumulate more ETH than the governance-approved cap, increasing EigenLayer slashing exposure and violating off-chain risk commitments. No funds are directly stolen. This maps to **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The precondition (`totalAssetDeposits == depositLimit`) is a routine operational state. Any depositor monitoring on-chain state can observe it via `getAssetCurrentLimit` returning `0` and immediately call `depositETH` with any `msg.value > 0`. No special role, timing window, or front-running is required. Likelihood is **High**.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC-20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = X`.
2. Deposits accumulate until `getTotalAssetDeposits(ETH_TOKEN) == X`; `getAssetCurrentLimit(ETH_TOKEN)` returns `0`.
3. Attacker calls `depositETH{value: Y}(0, "")` for any `Y > 0`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, Y)` evaluates `X > X` → `false` → no revert.
5. `_mintRsETH` mints rsETH for `Y` ETH; `getTotalAssetDeposits(ETH_TOKEN)` is now `X + Y`, exceeding the cap.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    uint256 limit = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN);
    // fill deposits to exactly the limit via prior deposits
    // assert getAssetCurrentLimit == 0
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), limit);
}
