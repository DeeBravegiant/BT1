Audit Report

## Title
Chainlink `minAnswer`/`maxAnswer` Circuit Breaker Not Validated — Inflated LST Price Enables Over-Minting of rsETH - (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but never validates the returned price against the Chainlink aggregator's built-in `minAnswer`/`maxAnswer` circuit breakers. If a supported LST crashes below `minAnswer`, the oracle silently returns the floor price, causing `LRTDepositPool` to mint rsETH at an inflated rate. The protocol's downside-protection pause only activates when `updateRSETHPrice()` is called externally, leaving the deposit path unprotected during the attack window.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` performs a bare `latestRoundData()` call with no bounds check:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Chainlink aggregators clamp their output to `[minAnswer, maxAnswer]`. If the real market price of a supported LST falls below `minAnswer`, the aggregator still reports `minAnswer`, not the real price.

This price flows through `LRTOracle.getAssetPrice()` (L156–158) directly into `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The downside-protection pause in `LRTOracle._updateRsETHPrice()` (L270–281) only fires when `updateRSETHPrice()` is called. `LRTOracle.getAssetPrice()` carries no `whenNotPaused` guard and is a pure view function — it returns the inflated price regardless of pause state. The deposit path in `LRTDepositPool.depositAsset()` does carry `whenNotPaused`, but the pool is only paused after `updateRSETHPrice()` is called by an external actor, creating an exploitable window between the crash and the pause.

The same unchecked pattern exists in `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L27–34), which checks for staleness and zero price but not for `minAnswer`/`maxAnswer` clamping.

## Impact Explanation

**Critical — Direct theft of user funds / Protocol insolvency.**

An attacker who acquires a crashed LST cheaply on the open market and deposits it into `LRTDepositPool` receives rsETH computed at the inflated `minAnswer` price rather than the real market price. The over-minted rsETH can be redeemed for ETH, extracting value from all existing rsETH holders. This is a direct, concrete theft from depositors and constitutes protocol insolvency.

## Likelihood Explanation

**Low.** The attack requires a supported LST (e.g., stETH, cbETH) to suffer a severe price crash that triggers the Chainlink circuit breaker — a rare but historically documented event (Venus/LUNA on BSC, 2022). The attacker requires no special privileges; any external user can call `depositAsset()`. The attack is repeatable within the window between the crash and the protocol pause being triggered by an external `updateRSETHPrice()` call. The SECURITY.md exclusion for "depegging of an external stablecoin" does not apply here because LSTs are ETH-denominated liquid staking tokens, not stablecoins, and the root cause is a protocol integration defect, not incorrect data from the oracle.

## Recommendation

In `ChainlinkPriceOracle.getAssetPrice()`, store per-feed `minAnswer`/`maxAnswer` bounds (readable from the aggregator's `minAnswer()`/`maxAnswer()` accessors or set by the admin at feed registration time) and revert if the returned price falls outside them:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
if (price <= int256(minAnswer[asset]) || price >= int256(maxAnswer[asset])) revert InvalidPrice();
```

Apply the same guard in `ChainlinkOracleForRSETHPoolCollateral.getRate()`. Additionally, consider adding a `whenNotPaused` guard to `LRTOracle.getAssetPrice()` so that a paused oracle state also blocks the deposit pricing path.

## Proof of Concept

1. Assume stETH is a supported asset; its Chainlink feed has `minAnswer = 0.5e18` (0.5 ETH). Real market price crashes to `0.05e18` (0.05 ETH) due to a slashing/depeg event.
2. Attacker purchases 1 000 stETH at 0.05 ETH each on the open market (cost: 50 ETH).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, minRSETH, "")`.
4. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → `latestRoundData()` returns `0.5e18` (clamped to `minAnswer`).
5. `rsethAmountToMint = (1000e18 × 0.5e18) / rsETHPrice` — 10× more rsETH than the deposited value warrants.
6. Attacker redeems the over-minted rsETH for ETH, extracting ~450 ETH of value from existing depositors.
7. `updateRSETHPrice()` has not yet been called, so the protocol is not paused; the attack completes before any guardian can react.

**Foundry fork test plan:** Fork mainnet, mock a Chainlink aggregator that returns `minAnswer` for stETH, call `depositAsset` as an unprivileged address, assert that `rsethAmountToMint` exceeds the fair value by the ratio `minAnswer / realPrice`, and confirm no revert occurs.
