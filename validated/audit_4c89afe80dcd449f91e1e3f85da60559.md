Looking at the external bug (missing `stopTime > block.timestamp` check in stream creation), I need to find an analog in the Sequencer where a time-based parameter lacks a lower-bound sanity check against the current time, allowing a past/stale value to be accepted and committed.

I traced the full proposal validation path: `validate_proposal` → `is_proposal_init_valid` in `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, and found the analog.

---

### Title
Missing lower-bound timestamp check in `is_proposal_init_valid` allows a Byzantine proposer to set `ProposalInit.timestamp` to a past value, causing the validator to accept stale L1 gas prices — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` enforces an **upper** bound on `ProposalInit.timestamp` (`timestamp ≤ now + window`) but has **no symmetric lower bound** against the current time. A Byzantine proposer can set `timestamp` to any past value ≥ `last_block_timestamp`. Because the validator then queries L1 gas prices using the **proposer-supplied** timestamp, both sides compute the same historical prices, the margin check passes, and the block is committed with artificially low L1 gas prices.

---

### Finding Description

In `is_proposal_init_valid` the timestamp validation is:

```rust
let now: u64 = clock.unix_now();
let last_block_timestamp =
    proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
if init_proposed.timestamp < last_block_timestamp {          // lower bound: only vs. previous block
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
    return Err(ValidateProposalError::InvalidProposalInit(...));  // upper bound: future window
}
``` [1](#0-0) 

There is **no check** of the form `timestamp >= now - block_timestamp_window_seconds`. The only lower bound is `last_block_timestamp`, which can be arbitrarily far in the past (e.g., after a chain pause or network partition).

Immediately after the timestamp bounds check, the validator queries L1 gas prices using the **proposer-supplied** timestamp:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,   // ← attacker-controlled past timestamp
    proposal_init_validation.previous_proposal_init.as_ref(),
    gas_price_params,
)
.await;
``` [2](#0-1) 

Because both the proposer and the validator call `get_l1_prices_in_fri_and_wei` with the **same** (past) timestamp, they obtain the same historical prices. The subsequent `within_margin` check then passes: [3](#0-2) 

The proposal is accepted, and the block is committed with a stale timestamp and the corresponding historical (potentially lower) L1 gas prices embedded in the block header.

The `ProposalInit.timestamp` field is defined as a plain `u64` with no protocol-level floor: [4](#0-3) 

The same asymmetric pattern (upper bound only, no lower bound against `now`) also appears in `try_sync`: [5](#0-4) 

---

### Impact Explanation

**Critical — Incorrect L1 gas price effect with economic impact.**

A Byzantine proposer sets `timestamp = T_past` where L1 gas prices were lower. The validator computes the same historical prices for `T_past`, the margin check passes, and the block is finalized with artificially low `l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, and `l1_data_gas_price_wei`. Every transaction in that block pays fees computed against the stale (lower) prices, under-charging users and under-compensating the sequencer for actual L1 costs. The committed block header carries the wrong timestamp, which also corrupts any time-dependent logic downstream (L1 handler cooldowns, oracle lookups keyed to block time, etc.).

---

### Likelihood Explanation

**Medium.** The attack requires the proposer to be Byzantine (one of the consensus participants). In a BFT system tolerating `f` faults, any of the `f` Byzantine nodes that wins a proposer slot can execute this. The L1 gas price provider maintains a sliding window of recent prices; as long as the chosen past timestamp falls within that window and prices were lower then, the attack succeeds without any fallback triggering. Normal block intervals (seconds to tens of seconds) keep `last_block_timestamp` recent, but after any chain pause the window widens significantly.

---

### Recommendation

Add a symmetric lower-bound check mirroring the existing upper-bound check:

```rust
// Existing upper bound:
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
    return Err(ValidateProposalError::InvalidProposalInit(..., "Timestamp is in the future"));
}
// Missing lower bound (add this):
if init_proposed.timestamp < now.saturating_sub(proposal_init_validation.block_timestamp_window_seconds) {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "Timestamp is too far in the past: now={}, block_timestamp_window_seconds={}, proposed={}",
            now,
            proposal_init_validation.block_timestamp_window_seconds,
            init_proposed.timestamp
        ),
    ));
}
``` [6](#0-5) 

This mirrors the `block_timestamp_window_seconds` window symmetrically, bounding the timestamp to `[now − window, now + window]` (intersected with `[last_block_timestamp, ∞)`).

---

### Proof of Concept

1. Chain is running normally; the previous block has `last_block_timestamp = T_prev` (e.g., 30 s ago). L1 gas prices at `T_prev` were 20 % lower than at `T_now`.
2. Byzantine proposer constructs `ProposalInit` with `timestamp = T_prev`.
3. Proposer queries `get_l1_prices_in_fri_and_wei(provider, T_prev, ...)` → obtains the lower historical prices and embeds them in `ProposalInit`.
4. Validator receives the proposal. `is_proposal_init_valid` checks:
   - `T_prev >= last_block_timestamp` → **passes** (equal).
   - `T_prev <= T_now + window` → **passes** (T_prev < T_now).
   - No lower-bound check against `T_now` → **no rejection**.
5. Validator calls `get_l1_prices_in_fri_and_wei(provider, T_prev, ...)` → same historical prices.
6. `within_margin` check passes; `is_proposal_init_valid` returns `Ok(())`.
7. `initiate_validation` → batcher executes the block with `block_timestamp = T_prev` and the lower gas prices.
8. Block is finalized; all transactions in the block are charged fees based on the 20 %-lower L1 gas prices. [7](#0-6) [8](#0-7)

pub(crate) async fn get_l1_prices_in_fri_and_wei_and_conversion_rate(
    l1_gas_price_provider_client: Arc<dyn L1GasPriceProviderClient>,
    timestamp: u64,
    previous_proposal_init: Option<&PreviousProposalInitInfo>,
    gas_price_params: &GasPriceParams,
) -> (L1PricesInFri, L1PricesInWei, u128) {
    // One of these paths should fill the return values:
    // 1. Both L1 gas price and eth/strk rate are Ok, use those.
    // 2. Otherwise, use previous block info.
    // 3. If that isn't available either, use min gas prices and default eth/strk rate.

    // Get the eth to fri rate from the oracle, and the L1 gas price (in wei) from the provider.
    let (eth_to_fri_rate, price_info) = tokio::join!(
        l1_gas_price_provider_client.get_rate(timestamp),
        l1_gas_price_provider_client.get_price_info(BlockTimestamp(timestamp))
    );
    if price_info.is_err() {
        warn!("Failed to get l1 gas price from provider: {:?}", price_info);
        CONSENSUS_L1_GAS_PRICE_PROVIDER_ERROR.increment(1);
    }
    if eth_to_fri_rate.is_err() {
        warn!("Failed to get eth to fri rate from oracle: {:?}", eth_to_fri_rate);
    }
    if let (Ok(eth_to_fri_rate), Ok(mut price_info)) = (eth_to_fri_rate, price_info) {
        // Both L1 prices and rate are Ok, so we can use them.
        info!(
            "raw eth_to_fri_rate (from oracle): {eth_to_fri_rate}, raw l1 gas price wei (from \
             provider): {price_info:?}"
        );
        apply_fee_transformations(&mut price_info, gas_price_params);
        let prices_in_wei = L1PricesInWei {
            l1_gas_price: price_info.base_fee_per_gas,
            l1_data_gas_price: price_info.blob_fee,
        };
        // Apply the eth/strk rate to get prices in fri.
        let l1_gas_prices_fri_result =
            L1PricesInFri::convert_from_wei(&prices_in_wei, eth_to_fri_rate);
        // If conversion fails, leave return_value=None to try backup methods.
        if let Ok(prices_in_fri) = l1_gas_prices_fri_result {
            return (prices_in_fri, prices_in_wei, eth_to_fri_rate);
        } else {
            warn!(
                "Failed to convert L1 gas prices to FRI: {:?}",
                l1_gas_prices_fri_result.clone().err()
            );
        }
    }
