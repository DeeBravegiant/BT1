Audit Report

## Title
`_withdrawFromAave` silently returns zero when `totalETHDepositedToAave` is exhausted but residual Aave interest remains, causing ETH withdrawal DoS — (`contracts/LRTWithdrawalManager.sol`)

## Summary
When all tracked principal (`totalETHDepositedToAave`) has been consumed by prior withdrawals while residual accrued interest still sits in Aave, `_withdrawFromAave` computes `withdrawablePrincipal = min(aaveBalance, 0) = 0` and silently returns `0`. Any subsequent `completeWithdrawal` call for ETH that requires Aave liquidity then reverts with `InsufficientLiquidityForWithdrawal`, blocking users who have already burned their rsETH from completing their withdrawals.

## Finding Description
In `_withdrawFromAave` at [1](#0-0) , the withdrawable amount is capped to `min(aaveBalance, totalETHDepositedToAave)`. When `totalETHDepositedToAave == 0`, this collapses to `0` regardless of the actual `aaveBalance`, and the function returns `0` silently via the `if (withdrawnAmount == 0) return 0` guard.

This state is reached naturally: when a withdrawal exactly drains the remaining principal (e.g., `amountNeeded == totalETHDepositedToAave`), line 918 sets `totalETHDepositedToAave = 0` while Aave still holds accrued interest as `aaveBalance > 0`. [2](#0-1) 

In `_processWithdrawalCompletion`, no interest collection is performed before calling `_withdrawFromAave`. [3](#0-2)  After `_withdrawFromAave` returns `0`, the post-withdrawal balance check at line 728 fails and reverts with `InsufficientLiquidityForWithdrawal`.

`_collectInterestToTreasury` routes the residual `aaveBalance` to the treasury rather than back to the contract, so the interest is inaccessible to the withdrawal flow. [4](#0-3)  Disabling the Aave integration does not resolve the stuck state for already-queued requests.

## Impact Explanation
**Medium — Temporary freezing of funds.** Users who have already burned rsETH via `initiateWithdrawal` cannot complete their ETH withdrawals once this state is reached. Their position is frozen until an external ETH inflow (e.g., from the unstaking vault) replenishes the contract's idle balance. This matches the allowed impact class "Temporary freezing of funds."

## Likelihood Explanation
The trigger condition — `totalETHDepositedToAave` reaching exactly zero while `aaveBalance > 0` — arises in normal operation whenever a withdrawal drains the last tracked principal while any interest has accrued. No attacker action is required; this is a routine operational state. Any user whose withdrawal is processed after this point and whose `expectedAssetAmount` exceeds the contract's idle ETH balance will be blocked. The condition is repeatable and self-sustaining until external ETH is deposited.

## Recommendation
In `_withdrawFromAave`, when `totalETHDepositedToAave == 0` but `aaveBalance > 0`, treat the full `aaveBalance` as reclaimable liquidity:

```solidity
uint256 withdrawablePrincipal = totalETHDepositedToAave == 0
    ? aaveBalance
    : (aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave);
```

Alternatively, call `_collectInterestToTreasury()` inside `_processWithdrawalCompletion` before `_withdrawFromAave` so that residual interest is swept to the treasury first, bringing `aaveBalance` back in line with `totalETHDepositedToAave` before the principal withdrawal is attempted. [5](#0-4) 

## Proof of Concept
1. Aave integration enabled; `totalETHDepositedToAave = 100 ETH`, `aaveBalance = 101 ETH` (1 ETH accrued interest), contract idle balance = 0.
2. User A calls `completeWithdrawal(ETH)` for `100 ETH`.
3. `_withdrawFromAave(100)` → `withdrawablePrincipal = min(101, 100) = 100`; withdraws 100 ETH; `totalETHDepositedToAave = 0`, `aaveBalance = 1 ETH`.
4. User A receives 100 ETH; contract idle balance returns to 0.
5. User B calls `completeWithdrawal(ETH)` for any `amount > 0`; contract idle balance = 0.
6. `_withdrawFromAave(amount)` → `withdrawablePrincipal = min(1, 0) = 0`; `withdrawnAmount = 0`; returns 0.
7. `balanceAfter = 0 < request.expectedAssetAmount` → `revert InsufficientLiquidityForWithdrawal()`.
8. User B's withdrawal is permanently blocked until external ETH is deposited.

Foundry test plan: deploy with mock Aave WETH gateway and aWETH token; seed `totalETHDepositedToAave = 100e18`, mock `aaveAWETH.balanceOf` returning `101e18`; call `_processWithdrawalCompletion` for 100 ETH (succeeds); then call again for any nonzero amount and assert revert with `InsufficientLiquidityForWithdrawal`. [5](#0-4)

```text
    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
```
