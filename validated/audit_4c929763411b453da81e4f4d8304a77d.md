Audit Report

## Title
No Cancellation Path for Pending Withdrawal Requests Causes Permanent rsETH Freeze - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager` transfers rsETH from the user into the contract on `initiateWithdrawal` but provides no user-callable or admin-callable function to cancel a pending (not yet unlocked) request and return the rsETH. If `unlockQueue` can never successfully advance `nextLockedNonce[asset]` — because the unstaking vault balance is zero or the first queued request's payout permanently exceeds available vault assets — every affected user's rsETH is irretrievably locked in the contract with no recovery path. The contract's own NatSpec acknowledges the slashing edge case but implements no mitigation.

## Finding Description

**Withdrawal lifecycle:**

1. `initiateWithdrawal` (L150–178): rsETH is pulled from the user into the contract via `safeTransferFrom`; a `WithdrawalRequest` is stored and `assetsCommitted[asset]` is incremented.
2. `unlockQueue` (L268–320): operator-only; advances `nextLockedNonce[asset]` in FIFO order and burns rsETH.
3. `completeWithdrawal` (L183–185): user receives the underlying asset; requires `usersFirstWithdrawalRequestNonce < nextLockedNonce[asset]`.

**Blocking path 1 — vault balance zero:**
`_createUnlockParams` (L837–851) sets `totalAvailableAssets = unstakingVault.balanceOf(asset)`. If EigenLayer slashing drains the unstaking vault for an asset, `unlockQueue` always reverts at L297:
```solidity
if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero();
```

**Blocking path 2 — FIFO head exceeds available assets:**
Even with a non-zero vault balance, the inner loop in `_unlockWithdrawalRequests` (L790–814) breaks without advancing `nextLockedNonce_` the moment the first queued request's payout exceeds available assets:
```solidity
if (availableAssetAmount < payoutAmount) break;
```
`nextLockedNonce[asset]` is written back unchanged (L815), so every subsequent user's `completeWithdrawal` call reverts at L707:
```solidity
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

**No recovery path:**
There is no `cancelWithdrawal`, no `emergencyRefund`, and no admin function that returns rsETH to depositors. `sweepRemainingAssets` (L395–414) sweeps LST assets to the treasury — it does not touch rsETH held in the contract. The rsETH burned in `unlockQueue` (L305) is the only exit for rsETH; if `unlockQueue` never succeeds, the rsETH balance accumulates permanently. The contract's own NatSpec at L148–149 acknowledges: *"There is an edge case were the user withdraws last underlying asset and that asset gets slashed"* — but no mitigation is implemented.

## Impact Explanation

**Critical — Permanent freezing of funds.**

Users who have called `initiateWithdrawal` have already surrendered their rsETH to the contract. If `nextLockedNonce[asset]` can never advance, every affected user's rsETH is irretrievably locked. There is no on-chain mechanism — user-callable or admin-callable — to return the rsETH to its original owners. This matches the "Permanent freezing of funds" Critical impact class exactly.

## Likelihood Explanation

**Low-to-Medium.** The EigenLayer slashing scenario (vault balance → 0) is a tail-risk event but is an explicitly acknowledged protocol risk: `NodeDelegator` integrates directly with EigenLayer strategies where slashing is a designed feature, not an external compromise. The FIFO head-blocking scenario (one large request whose payout permanently exceeds vault liquidity) requires no slashing — only a sustained liquidity shortfall, which becomes permanent if the asset is later deprecated. Both paths are reachable without any privileged action by the attacker; a normal user calling `initiateWithdrawal` is sufficient to place funds at risk.

## Recommendation

Add a user-callable `cancelWithdrawal(address asset)` function that:
1. Verifies the caller's oldest nonce is still in the locked state (`nonce >= nextLockedNonce[asset]`).
2. Removes the nonce from `userAssociatedNonces[asset][msg.sender]` and deletes `withdrawalRequests[requestId]`.
3. Decrements `assetsCommitted[asset]` by `request.expectedAssetAmount`.
4. Returns `request.rsETHUnstaked` of rsETH to `msg.sender` via `safeTransfer`.

Optionally, add an admin-callable `emergencyRefundWithdrawals(address asset, address[] calldata users)` callable when the contract is paused, to handle bulk recovery in the slashing scenario.

## Proof of Concept

**EigenLayer slashing scenario (step-by-step call sequence):**

1. User calls `initiateWithdrawal(stETH, 10 ether, "")`. rsETH is transferred to `LRTWithdrawalManager`; `assetsCommitted[stETH] += expectedStETH`.
2. EigenLayer slashing event reduces all stETH held in strategies to zero; `unstakingVault.balanceOf(stETH)` returns 0.
3. Operator calls `unlockQueue(stETH, ...)`. Execution reaches L297: `if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero()` — reverts permanently.
4. User calls `completeWithdrawal(stETH, "")`. Execution reaches L707: `if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked()` — reverts permanently.
5. No `cancelWithdrawal` exists. User's 10 ether rsETH is permanently locked in `LRTWithdrawalManager`.

**Foundry test plan:**
- Fork mainnet; deploy `LRTWithdrawalManager` with a mock `LRTUnstakingVault` whose `balanceOf` returns 0.
- Call `initiateWithdrawal` as a normal user; assert rsETH balance of contract increases.
- Call `unlockQueue`; assert it reverts with `AmountMustBeGreaterThanZero`.
