Audit Report

## Title
Missing Zero-Output Guard in L2 Pool Deposit Functions Allows Dust Deposits to Be Silently Consumed - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All L2 pool `deposit()` functions compute `rsETHAmount` via integer division that can truncate to zero for dust inputs, yet unconditionally proceed to mint or transfer 0 wrsETH/rsETH. A depositor sending a dust amount of ETH or tokens whose `amountAfterFee * 1e18 < rsETHToETHrate` will have their deposit accepted, receive nothing in return, and hold no claim on the retained funds. The L1 `LRTDepositPool` has an equivalent `minRSETHAmountExpected` guard; no such protection exists on L2.

## Finding Description
In `RSETHPoolV3.deposit()`, the only input guard is `if (amount == 0) revert InvalidAmount()` at line 256. After that, `viewSwapRsETHAmountAndFee` computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // line 307
```

If `amountAfterFee * 1e18 < rsETHToETHrate` (e.g., 1 wei ETH when rate = 1.05e18), Solidity integer division yields `rsETHAmount = 0`. The function then calls `wrsETH.mint(msg.sender, 0)` at line 262 — OpenZeppelin's `_mint` permits zero-amount calls and returns silently. The deposited ETH is retained in the pool with no corresponding liability to the user.

The same pattern is present in `RSETHPoolNoWrapper.deposit()` at line 241 (`rsETH.safeTransfer(msg.sender, 0)` — SafeERC20 also permits zero transfers), and in the ERC20-token deposit overloads of both contracts.

The `limitDailyMint` modifier in `RSETHPoolV3` adds 0 to `dailyMintAmount` when `rsETHAmount = 0`, so the daily cap is not consumed and the transaction passes through silently, emitting a `SwapOccurred` event with `rsETHAmount = 0`.

By contrast, `LRTDepositPool._beforeDeposit()` at lines 667–669 explicitly reverts if `rsethAmountToMint < minRSETHAmountExpected`, providing the protection that is absent on L2.

## Impact Explanation
**Low. Contract fails to deliver promised returns.**

A user depositing a dust amount (e.g., 1 wei ETH when `rsETHToETHrate > 1e18`) will have their ETH accepted by the pool, receive 0 wrsETH/rsETH, and hold no on-chain claim on the deposited ETH. The ETH is eventually bridged to L1 as protocol liquidity. The loss per transaction is bounded by the dust threshold (sub-wei to a few wei), but the contract silently fails to deliver its core promise — minting wrsETH proportional to the deposited asset.

## Likelihood Explanation
**Low.** rsETH naturally appreciates above 1 ETH over time as yield accrues, so any deposit of less than `rsETHToETHrate / 1e18` wei (currently ~1 wei) after fee will trigger this. No oracle manipulation or privileged access is required. The condition arises from ordinary integer division on dust inputs submitted by any unprivileged external caller. Users are not warned and the transaction does not revert.

## Recommendation
Add a zero-output guard immediately after computing `rsETHAmount` in all L2 pool deposit functions:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, introduce a `minRSETHAmountExpected` parameter (mirroring `LRTDepositPool._beforeDeposit()`) so callers can enforce their own slippage tolerance. Apply the fix to both ETH and ERC20 deposit paths in `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`.

## Proof of Concept
1. `rsETHToETHrate = 1.05e18` (realistic after yield accrual).
2. Attacker/user calls `RSETHPoolV3.deposit{value: 1}("")` (1 wei ETH).
3. `viewSwapRsETHAmountAndFee(1)`:
   - `fee = 1 * feeBps / 10_000 = 0`
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation)
4. `limitDailyMint` adds 0 to `dailyMintAmount` — no revert.
5. `wrsETH.mint(msg.sender, 0)` executes — user receives 0 wrsETH.
6. 1 wei ETH is retained in the pool; `SwapOccurred` emits with `rsETHAmount = 0`.
7. Same sequence applies to `RSETHPoolNoWrapper.deposit{value: 1}("")` → `rsETH.safeTransfer(msg.sender, 0)`.

**Foundry fuzz test sketch:**
```solidity
function testFuzz_dustDepositReceivesZero(uint256 amount) public {
    vm.assume(amount > 0 && amount < rsETHToETHrate / 1e18 + 1);
    uint256 balanceBefore = wrsETH.balanceOf(user);
    vm.prank(user);
    pool.deposit{value: amount}("");
    assertEq(wrsETH.balanceOf(user), balanceBefore); // user received 0
    assertGt(address(pool).balance, 0);              // pool retained ETH
}
