Audit Report

## Title
Stale `rsETHPrice` in `instantWithdrawal()` Enables Over-Extraction of Assets Before Oracle Update — (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTOracle` stores `rsETHPrice` as a state variable updated only when `updateRSETHPrice()` is explicitly called. `instantWithdrawal()` computes the asset payout directly from this stored price with no freshness check, while the regular withdrawal path applies a `min(expectedAmount, currentReturn)` guard that `instantWithdrawal()` lacks. Any rsETH holder can exploit the window between a real TVL decrease and the next oracle update to extract more assets than their proportional share, with the shortfall socialized among remaining holders.

## Finding Description

`LRTOracle.updateRSETHPrice()` is public and permissionless:

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_updateRsETHPrice()` computes the new price from live on-chain TVL (`_getTotalEthInProtocol()`) and writes it to the `rsETHPrice` state variable. Until this function is called, `rsETHPrice` reflects the pre-loss value.

`instantWithdrawal()` computes the payout via `getExpectedAssetAmount()`:

```solidity
// LRTWithdrawalManager.sol L228
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
```

```solidity
// LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

This reads the stored `rsETHPrice` with no freshness check and no call to `updateRSETHPrice()` before execution.

The regular withdrawal path (`_unlockWithdrawalRequests` → `_calculatePayoutAmount`) applies a protective cap:

```solidity
// LRTWithdrawalManager.sol L833-834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

`instantWithdrawal()` has no equivalent guard. The downside-protection pause in `_updateRsETHPrice()` (L270-282) only triggers when the oracle is updated — it does not protect against exploitation in the window before the update.

**Exploit path:**
1. A TVL-reducing event (e.g., EigenLayer slashing, asset depeg) occurs on-chain. Stored `rsETHPrice` = `P_high`; actual price = `P_low < P_high`.
2. Attacker observes the on-chain state change. `updateRSETHPrice()` has not been called.
3. Attacker calls `instantWithdrawal(asset, rsETHAmount, "")`. Payout = `rsETHAmount * P_high / assetPrice` — computed at the stale inflated price.
4. rsETH is burned; attacker receives excess assets from `LRTUnstakingVault`.
5. `updateRSETHPrice()` is eventually called. Price drops to `P_low`. Remaining rsETH holders now hold tokens backed by fewer assets per token; the attacker's excess `rsETHAmount * (P_high - P_low) / assetPrice` is a direct loss to them.

## Impact Explanation

**High — Theft of unclaimed yield.** The attacker extracts assets in excess of their true proportional entitlement. The shortfall is borne by all remaining rsETH holders when the oracle is updated, reducing the value backing their tokens. This is a concrete, quantifiable loss to other protocol participants, not a hypothetical one.

## Likelihood Explanation

**Medium.** The attack window opens whenever actual TVL decreases before `updateRSETHPrice()` is called. EigenLayer slashing events and asset balance changes are observable on-chain without any privileged access. The attacker does not need to front-run any specific transaction — they only need to act within the window. The only prerequisite is that `instantWithdrawal` is enabled for the target asset (`onlyInstantWithdrawalAllowed` modifier), which is a manager-controlled flag expected to be active in normal operation. The attack is repeatable across any such event.

## Recommendation

1. **Atomically refresh the price inside `instantWithdrawal()`** by calling `updateRSETHPrice()` (or the internal `_updateRsETHPrice()`) before computing `assetAmountUnlocked`, ensuring the price is always current at execution time.
2. Alternatively, apply the same `min(expectedAmount, currentReturn)` guard used in `_calculatePayoutAmount()` to `instantWithdrawal()`, computing `currentReturn` from live TVL rather than the stored price.
3. As a defense-in-depth measure, require that `updateRSETHPrice()` was called within a recent block window before any instant withdrawal is processed.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// 1. Fork mainnet at a block where rsETHPrice = P_high.
// 2. Simulate a TVL decrease: mock getEffectivePodShares() to return a lower value,
//    or directly reduce an asset balance in LRTUnstakingVault.
// 3. Assert: lrtOracle.rsETHPrice() still returns P_high (not yet updated).
// 4. Attacker calls instantWithdrawal(asset, rsETHAmount, "").
//    Record attacker's asset balance before and after.
// 5. Call updateRSETHPrice(). Record new price P_low.
// 6. Assert: attacker received rsETHAmount * P_high / assetPrice assets,
//    which exceeds rsETHAmount * P_low / assetPrice — the correct entitlement.
// 7. Assert: remaining rsETH holders' per-token asset backing decreased by the excess.
```
