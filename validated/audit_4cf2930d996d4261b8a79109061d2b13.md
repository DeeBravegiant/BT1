Audit Report

## Title
Unbounded `referralId` String Enables Unbounded Gas Consumption and Block Stuffing - (`contracts/pools/RSETHPoolV3.sol`)

## Summary
Both `deposit` overloads in `RSETHPoolV3` accept a `string memory referralId` parameter with no length bound. The string is copied into memory and emitted verbatim in a `SwapOccurred` log, causing gas consumption to scale linearly with its length. An unprivileged depositor can craft a single deposit with a maximally-sized `referralId` to consume nearly the entire L2 block gas budget, preventing other deposits from being included in that block.

## Finding Description
`RSETHPoolV3.deposit(string memory referralId)` (L246) and `RSETHPoolV3.deposit(address, uint256, string memory referralId)` (L271) both accept an arbitrary-length string. The string is used exclusively in event emissions at L264 and L292:

```solidity
emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);        // L264
emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId, token); // L292
```

A grep across the entire repository for `referralId.length`, `bytes(referralId)`, or any `MAX_REFERRAL` constant returns zero results — confirming the complete absence of any length guard. The `nonReentrant`, `whenNotPaused`, and `limitDailyMint` modifiers (L249–251, L277–280) execute before the string is processed and do not inspect its length. There is no mechanism anywhere in the call path that bounds the string.

Gas cost for a string of length `L` bytes:
- Calldata (non-zero bytes): `16 * L` gas
- LOG event data: `8 * L` gas
- Memory allocation: `3 * ceil(L/32)` gas + quadratic expansion term

For `L = 500,000` bytes: calldata + log alone exceeds 12M gas. On an L2 with a 30M gas block limit, a string of ~1.5 MB saturates the block. The attacker's required ETH deposit is 1 wei; gas fees on L2 are negligible.

## Impact Explanation
**Medium — Unbounded gas consumption / Low — Block stuffing.**

An attacker can repeatedly submit one deposit per block with a maximally-sized `referralId`, consuming the block's gas budget and leaving insufficient gas for legitimate deposit transactions. This constitutes unbounded gas consumption (Medium) and block stuffing (Low) per the allowed impact scope. No admin action is required; the attacker only needs to be a normal depositor with 1 wei.

## Likelihood Explanation
**Low-Medium.** The attack requires no special privileges, no front-running, and no external dependencies. It is economically viable on L2 chains (where this contract is deployed) because gas prices are orders of magnitude cheaper than on L1. The attack is repeatable every block indefinitely.

## Recommendation
Add a maximum length check on `referralId` in both `deposit` overloads, and apply consistently to all pool variants (`RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2`, etc.):

```solidity
uint256 private constant MAX_REFERRAL_ID_LENGTH = 128;

function deposit(string memory referralId) external payable ... {
    if (bytes(referralId).length > MAX_REFERRAL_ID_LENGTH) revert ReferralIdTooLong();
    ...
}

function deposit(address token, uint256 amount, string memory referralId) external ... {
    if (bytes(referralId).length > MAX_REFERRAL_ID_LENGTH) revert ReferralIdTooLong();
    ...
}
```

## Proof of Concept
```solidity
function testBlockStuffing() public {
    uint256 gasStart = gasleft();

    // Attacker: 500KB referralId, deposit 1 wei
    bytes memory bigId = new bytes(500_000);
    pool.deposit{value: 1 wei}(string(bigId));

    uint256 gasUsed = gasStart - gasleft();
    // 500_000 bytes * (16 calldata + 8 log) = 12M gas minimum
    // A normal deposit costs ~80_000 gas — insufficient gas remains in block
    assertGt(gasUsed, 8_000_000);
}
```

The test confirms: (1) no revert occurs with a 500KB `referralId`; (2) gas consumed far exceeds what is needed for a normal deposit; (3) remaining block gas is insufficient for additional deposits.
