Audit Report

## Title
Block Stuffing Causes `stake32EthValidated()` to Revert via Stale `expectedDepositRoot`, Temporarily Preventing ETH Staking — (`contracts/NodeDelegator.sol`)

## Summary

`stake32EthValidated()` performs an exact-equality check between an operator-supplied `expectedDepositRoot` and the live `IETHPOSDeposit.get_deposit_root()` at execution time. Because the ETH2 deposit contract root advances with every new validator deposit globally, an attacker can perform block stuffing to delay the operator's transaction until the root changes, causing a guaranteed revert with `InvalidDepositRoot`. The 32 ETH remains idle in the NDC and `stakedButUnverifiedNativeETH` is never incremented for the duration of the attack.

## Finding Description

`stake32EthValidated` at lines 186–200 of `contracts/NodeDelegator.sol` is declared `external` with no access control modifier of its own. It reads the live deposit root and compares it to the caller-supplied `expectedDepositRoot`:

```solidity
IETHPOSDeposit depositContract = _getEigenPodManager().ethPOS();
bytes32 actualDepositRoot = depositContract.get_deposit_root();
if (expectedDepositRoot != actualDepositRoot) {
    revert InvalidDepositRoot(expectedDepositRoot, actualDepositRoot);
}
stake32Eth(pubkey, signature, depositDataRoot);
```

The operator reads `get_deposit_root()` offchain, encodes it as `expectedDepositRoot`, and submits the transaction. If any deposit to the globally shared, permissionless ETH2 deposit contract lands between that offchain read and the transaction's inclusion, the root advances and the call reverts. Block stuffing amplifies this: the attacker fills consecutive blocks with high-gas-price transactions, preventing the operator's transaction from being included while organic validator onboarding activity continues to advance the root. The attacker does not need to make deposits themselves.

When the revert occurs, `stake32Eth` is never reached, so `stakedButUnverifiedNativeETH` is not incremented and the 32 ETH remains idle in the NDC. The operator must recompute `expectedDepositRoot` and resubmit, but the attacker can repeat the stuffing cycle indefinitely.

The `IPubkeyRegistry.hasPubkey` guard inside `stake32Eth` (lines 159–163) and the `onlyLRTOperator` modifier on `stake32Eth` (line 157) do not mitigate this: they are only reached after the deposit root check passes, and neither prevents the root from changing between the operator's offchain snapshot and on-chain execution.

## Impact Explanation

The 32 ETH is not lost, but it fails to earn EigenLayer restaking yield for the duration of each stuffing cycle. This concretely matches two allowed impact classes: **Low — Block stuffing** (the attack mechanism is block stuffing) and **Low — Contract fails to deliver promised returns, but doesn't lose value** (staking yield is not earned while ETH sits idle in the NDC).

## Likelihood Explanation

Block stuffing on Ethereum mainnet requires outbidding all competing transactions for full blocks, making sustained attacks expensive. However, the vulnerability is structurally present: the deposit root check creates a race condition that any block-stuffing actor can exploit. The cost scales with block base fee and the number of blocks stuffed, not with the value at risk, making the attack more plausible during low-fee periods. The attacker does not need to control the ETH2 deposit contract or make any deposits themselves — organic validator onboarding activity (which is continuous on mainnet) is sufficient to advance the root during the stuffed window.

## Recommendation

Replace the exact-equality deposit root check with a mechanism that does not create a revert-on-stale-root condition:

1. **Remove `stake32EthValidated` entirely** and rely solely on `stake32Eth` with offchain monitoring of the pubkey registry to detect front-running. The `IPubkeyRegistry.hasPubkey` guard in `stake32Eth` already prevents pubkey reuse and provides the primary front-running protection.

2. **Accept a root window**: allow the operator to supply a list of acceptable roots (e.g., the last N roots), or check that the deposit count has not advanced beyond a configurable threshold since the operator's snapshot, rather than requiring an exact match.
