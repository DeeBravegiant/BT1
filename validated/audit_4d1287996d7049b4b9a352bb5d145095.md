### Title
Missing Minimum `BlockConfirmations` Bound Allows MPC Signing of Unconfirmed Bitcoin Transactions - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The `verify_foreign_transaction` contract method accepts `BlockConfirmations(0)` without any minimum-value enforcement. Because the node-side inspector only checks `threshold <= actual_confirmations`, a threshold of zero trivially passes for every transaction — including mempool-only (unconfirmed) ones. The MPC network will then produce a valid threshold signature over a payload that attests to a transaction that has never been included in a block, enabling a double-spend against any bridge contract that relies on the MPC attestation.

---

### Finding Description

`BlockConfirmations` is a plain `u64` newtype with no lower-bound constraint:

```rust
pub struct BlockConfirmations(pub u64);
```

The ABI schema explicitly permits `0` (`"minimum": 0.0`). The contract's `verify_foreign_transaction` entry point performs no validation on this field before enqueuing the request:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(
        request.domain_id,
        DomainPurpose::ForeignTx,
        ...
    );
    // No check on request.request.confirmations
    ...
    self.enqueue_yield_request(...);
}
```

The node-side `BitcoinInspector::extract` enforces only:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
if !enough_block_confirmations {
    return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations { ... });
}
```

When `block_confirmations_threshold` is `0`, the predicate `0 <= N` is always `true` for any `N ≥ 0`, including `N = 0` (a mempool transaction). The node then proceeds to extract values and sign the payload. The signed `ForeignTxSignPayloadV1` embeds the original request (including `confirmations: 0`) and the extracted block hash, and `respond_verify_foreign_tx` on the contract validates only the ECDSA signature over the payload hash — it does not re-examine the confirmation count:

```rust
let signature_is_valid = match (&response.signature, public_key) {
    (dtos::SignatureResponse::Secp256k1(sig), PublicKeyExtended::Secp256k1 { near_public_key }) => {
        near_mpc_signature_verifier::verify_ecdsa_signature(sig, &payload_hash, &secp_pk).is_ok()
    }
    ...
};
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
// No check that the embedded confirmations value meets any minimum
pending_requests::resolve_yields_for(...)
```

There is no per-chain minimum confirmation floor anywhere in the contract, the node, or the `ChainEntry` whitelist structure.

---

### Impact Explanation

This maps to the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

The core safety invariant of `verify_foreign_transaction` is that the MPC network attests only to **finalized** foreign-chain events. Accepting `BlockConfirmations(0)` breaks this invariant: the network produces a cryptographically valid threshold signature over a payload that includes an unconfirmed (or even mempool-only) Bitcoin transaction. Any bridge contract that trusts the MPC attestation without independently re-checking the embedded confirmation count will release funds against a transaction that may subsequently be double-spent or never confirmed. The signed payload does embed `confirmations: 0`, but bridge contracts that treat the MPC signature as a sufficient finality proof — the intended usage pattern — are directly exposed.

---

### Likelihood Explanation

The attack requires no privileges. `verify_foreign_transaction` is a public payable method callable by any NEAR account. Submitting `BlockConfirmations(0)` is a single-field change to a normal user request. The MPC nodes will independently query the RPC, observe that `0 ≤ actual_confirmations`, and each produce a signature share. No threshold collusion, no operator access, and no key material is needed beyond the small deposit required by `check_request_preconditions`.

---

### Recommendation

Enforce a per-chain minimum confirmation floor at the contract level inside `verify_foreign_transaction`, before the request is enqueued:

```rust
// Example: reject Bitcoin requests with fewer than MIN_BITCOIN_CONFIRMATIONS
if let ForeignChainRpcRequest::Bitcoin(ref btc) = request.request {
    if btc.confirmations.0 < MIN_BITCOIN_CONFIRMATIONS {
        env::panic_str("BlockConfirmations below minimum for Bitcoin");
    }
}
```

Alternatively, store per-chain minimum confirmation requirements inside `ChainEntry` (alongside `quorum`) so that governance controls the floor, and validate against it in `verify_foreign_transaction`. The node-side inspector already enforces the user-supplied threshold correctly; the missing piece is a contract-level lower bound that prevents the threshold from being set to zero.

---

### Proof of Concept

1. Attacker broadcasts a Bitcoin transaction `T` to the mempool (0 confirmations).
2. Attacker calls `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "Bitcoin": {
         "tx_id": "<T's txid>",
         "confirmations": 0,
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": <foreign_tx_domain>,
     "payload_version": "V1"
   }
   ```
3. Each MPC node calls `getrawtransaction` for `T`. Bitcoin returns `confirmations: 0` (mempool). The inspector evaluates `0 <= 0` → `true`; extraction proceeds and returns the (null/pending) block hash.
4. The MPC network produces a threshold ECDSA signature over `SHA-256(borsh(ForeignTxSignPayloadV1 { request: {confirmations:0,...}, values: [...] }))`.
5. A node calls `respond_verify_foreign_tx`; the contract verifies the signature and resolves the yield, returning the signed response to the attacker.
6. Attacker presents the MPC-signed response to a bridge contract to claim NEAR-side funds.
7. Attacker double-spends `T` on Bitcoin. The bridge has already released funds against an unconfirmed transaction.
