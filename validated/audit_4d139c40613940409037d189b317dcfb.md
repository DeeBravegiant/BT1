### Title
`ForeignTxSignPayload` Missing NEAR Contract Address and Network Binding Enables Cross-Contract Signature Replay - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

### Summary

`ForeignTxSignPayloadV1`, the canonical payload the MPC network signs for foreign-chain transaction verification, contains only `(request, extracted_values)`. It omits the NEAR MPC contract's account ID, the NEAR network identifier, and any expiry/deadline. A `VerifyForeignTransactionResponse` obtained from one MPC contract deployment is therefore cryptographically indistinguishable from one intended for any other deployment that shares the same MPC key, and it never expires. Any downstream bridge contract (e.g., Omnibridge) that calls `ForeignChainSignatureVerifier::verify_signature` from the `near-mpc-sdk` will accept a replayed attestation, enabling double-spend or unauthorized bridge execution.

### Finding Description

`ForeignTxSignPayloadV1` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [1](#0-0) 

The 32-byte `msg_hash` that the MPC network signs is:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [2](#0-1) 

The payload contains no:
- NEAR MPC contract account ID (`env::current_account_id()`)
- NEAR network/chain identifier (mainnet vs. testnet)
- Caller account ID
- Expiry timestamp or nonce

`respond_verify_foreign_tx` on the contract verifies only that the ECDSA signature is valid against the MPC domain's public key and that a pending request exists: [3](#0-2) 

The response is then serialized and returned to the caller via NEAR's yield/resume mechanism. The downstream SDK verifier (`ForeignChainSignatureVerifier::verify_signature`) reconstructs the expected payload from `(request, expected_extracted_values)` and checks the signature — again with no contract-address or network binding: [4](#0-3) 

The contract itself explicitly supports multiple callers receiving the same signed response for the same request (fan-out): [5](#0-4) 

### Impact Explanation

**High — cross-contract replay enabling double-spend / invalid bridge execution.**

A `VerifyForeignTransactionResponse` is a portable, signed attestation that a specific foreign-chain transaction occurred with specific extracted values. Because the payload is not bound to any NEAR contract address or network:

1. **Cross-contract replay (same network):** Two bridge contracts (e.g., Omnibridge v1 and v2, or two independent bridge deployments) both use the MPC network. An attacker obtains a valid attestation for Bitcoin tx X via contract A, then submits the identical `(payload_hash, signature)` to contract B. Contract B's call to `ForeignChainSignatureVerifier::verify_signature` succeeds — the signature is valid — and contract B releases funds for the same Bitcoin tx, causing a double-spend.

2. **Cross-network replay:** If the same MPC key material is ever shared between NEAR mainnet and testnet deployments (e.g., during migration or key refresh), an attestation obtained cheaply on testnet is valid on mainnet.

3. **Temporal replay:** Attestations never expire. A signature obtained for a transaction that was later reorganized or invalidated on the foreign chain remains valid indefinitely.

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow, where the attestation directly gates fund release. A successful replay directly causes loss of funds.

### Likelihood Explanation

**Medium-High.** The MPC network is explicitly designed to serve multiple downstream bridge contracts simultaneously. The fan-out behavior (multiple callers receiving the same response) is a documented feature. Any two bridge contracts that independently call `verify_foreign_transaction` for the same foreign tx will receive identical, interchangeable attestations. An attacker who is a user of one bridge contract automatically possesses a valid attestation replayable against any other bridge contract on the same NEAR network that uses the same MPC key domain.

### Recommendation

Include the NEAR MPC contract's account ID, the NEAR network identifier, and the caller's account ID (or a request-specific nonce) in `ForeignTxSignPayloadV1`. A versioned, domain-separated payload analogous to EIP-712 would be appropriate:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub mpc_contract_id: AccountId,   // env::current_account_id() at request time
    pub near_chain_id: String,        // "mainnet" | "testnet"
    pub request_id: VerifyForeignTxId, // unique per-request nonce
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

This ensures a signature produced for one contract/network/request cannot be replayed in any other context.

### Proof of Concept

1. Deploy two bridge contracts (`BridgeA`, `BridgeB`) on NEAR mainnet, both using the same MPC contract.
2. `BridgeA` calls `verify_foreign_transaction` for Bitcoin tx `X` and receives `VerifyForeignTransactionResponse { payload_hash, signature }`.
3. Attacker extracts the response and calls `BridgeB`'s claim function, passing the same `(payload_hash, signature)`.
4. `BridgeB` calls `ForeignChainSignatureVerifier::verify_signature` with the MPC public key — it succeeds because the payload `SHA-256(borsh(ForeignTxSignPayloadV1 { request: X, values: [...] }))` is identical and the signature is valid.
5. `BridgeB` releases funds for Bitcoin tx `X` a second time — double-spend achieved.

The fan-out test in the contract confirms that the same `(payload_hash, signature)` is intentionally delivered to multiple callers, making the attestation trivially portable: [6](#0-5)

    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
