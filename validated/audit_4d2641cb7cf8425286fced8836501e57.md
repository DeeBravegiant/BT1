### Title
Unrestricted `init()` Allows Any Caller to Seize Control of the MPC Network Before Legitimate Initialization — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `init()` function of the MPC contract carries no access-control guard. Any NEAR account can call it before the legitimate deployer does. Because the deployment documentation shows contract deployment and initialization as two separate transactions, a race window exists in which an attacker can front-run the `init()` call, supply their own participant set, and become the sole controller of the MPC network.

---

### Finding Description

`init()` is decorated with `#[init]` (which only prevents *re*-initialization once state exists) but carries no `#[private]` attribute and no owner/predecessor check:

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

By contrast, the sibling initializer `init_running` is correctly guarded:

```rust
#[private]
#[init]
#[handle_result]
pub fn init_running(...)
``` [2](#0-1) 

The deployment documentation explicitly performs deployment and initialization as two separate CLI transactions:

```
near contract call-function as-transaction mpc-contract.test.near init \
  file-args /tmp/init_args.json ...
``` [3](#0-2) 

Between the `DeployContract` transaction and the `init` transaction there is at least one block (~1 s on NEAR mainnet). During that window any account can submit its own `init()` call with an attacker-controlled `ThresholdParameters`. The only validation inside `init()` is that the supplied parameters are internally consistent (non-empty participant set, valid threshold); there is no check on who the caller is. [4](#0-3) 

If the attacker's call is included first, the contract enters `Running` state with the attacker's participant set. All subsequent legitimate `init()` calls revert because state already exists.

---

### Impact Explanation

**Critical.** An attacker who wins the initialization race:

1. Becomes the sole participant (or controls the threshold) of the MPC network.
2. Can call `vote_add_domains()` to register signing domains, then drive key generation (`start_keygen_instance`, `vote_pk`) to completion — all without any honest node involvement.
3. Can then call `respond()` / `respond_ckd()` to issue threshold signatures and confidential key derivations for arbitrary payloads, because `assert_caller_is_attested_participant_and_protocol_active()` will pass for the attacker's own account. [5](#0-4) 

This satisfies the allowed impact: **Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization.**

---

### Likelihood Explanation

- The MPC contract account name is publicly known before deployment (named accounts on NEAR are chosen in advance).
- NEAR does not have a traditional public mempool, but deployment transactions are immediately visible on-chain; an attacker monitoring the chain can submit their `init()` call in the very next block.
- The deployment guide separates `near deploy` and `near ... init` into distinct manual steps, making the window reliably present in production deployments.
- No special privilege, validator collusion, or TEE access is required — a single unprivileged NEAR account suffices.

---

### Recommendation

Add `#[private]` to `init()`, mirroring the protection already applied to `init_running`:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> { ... }
```

`#[private]` in NEAR SDK enforces `predecessor_account_id() == current_account_id()`, so only the contract account itself (i.e., the deployer using a batched `DeployContract + FunctionCall` action) can call it. Alternatively, perform deployment and initialization atomically in a single transaction using NEAR's batched-action API, eliminating the race window entirely.

---

### Proof of Concept

1. Attacker learns the target contract account ID (e.g., `v2.multichain-mpc.near`) before deployment.
2. Attacker monitors the NEAR blockchain for a `DeployContract` action on that account.
3. In the immediately following block, attacker submits:
   ```
   near contract call-function as-transaction v2.multichain-mpc.near init \
     json-args '{"parameters": {"threshold": 1, "participants": {"next_id": 1, "participants": [["attacker.near", 0, {"tls_public_key": "ed25519:<attacker_key>", "url": "https://attacker.example.com"}]]}}}' \
     sign-as attacker.near ...
   ```
4. If included before the legitimate `init()`, the contract is now in `Running` state with `attacker.near` as the sole participant.
5. Attacker calls `vote_add_domains(...)` (threshold = 1, so one vote suffices), then `start_keygen_instance` / `vote_pk` to complete key generation.
6. Attacker can now call `respond(request, forged_signature)` for any pending `sign()` request, issuing unauthorized threshold signatures. [6](#0-5) [7](#0-6)
