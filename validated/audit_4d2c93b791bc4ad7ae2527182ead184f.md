Audit Report

## Title
Missing Staleness Validation on Chainlink `updatedAt` Enables Over-Minting of rsETH at Stale Prices - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and silently discards the `updatedAt` return value, accepting arbitrarily stale prices with no elapsed-time guard. Because this oracle feeds directly into `LRTDepositPool.getRsETHAmountToMint()`, a stale Chainlink feed allows any depositor to mint rsETH at a price that no longer reflects the current LST/ETH exchange rate, diluting all existing rsETH holders.

## Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` destructures `latestRoundData()` keeping only `price` and discarding all other return values including `updatedAt`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

There is no check of the form `require(block.timestamp - updatedAt <= MAX_STALENESS)`, no `require(price > 0)`, and no `answeredInRound >= roundId` check. The price returned may be hours or days old.

This oracle is wired into `LRTOracle.getAssetPrice()` (line 157), which is called by `LRTDepositPool.getRsETHAmountToMint()` (line 520):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`getRsETHAmountToMint()` is called from `_beforeDeposit()` (line 665), which is invoked by both `depositETH()` and `depositAsset()` — fully public, permissionless entry points.

For contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 30–32) does check `answeredInRound < roundID` and `timestamp == 0`, demonstrating the project is aware of oracle validation patterns but failed to apply them to the core deposit path oracle.

## Impact Explanation

**High — Theft of unclaimed yield.**

When a Chainlink feed for a supported LST becomes stale and the last recorded price is higher than the current market price (e.g., during a depeg event the feed has not yet reflected), `getRsETHAmountToMint()` computes `(amount * inflatedPrice) / rsETHPrice`, minting more rsETH than the deposited assets are worth. The excess rsETH represents an unbacked claim on protocol TVL, directly diluting all existing rsETH holders' proportional share of the underlying ETH. This constitutes theft of unclaimed yield from rsETH holders. If the price deviation is large enough, the impact escalates toward protocol insolvency.

## Likelihood Explanation

**Medium.** Chainlink feeds become stale due to RPC/node outages, network congestion, or rapid price movements that do not cross the deviation threshold before the heartbeat fires. An attacker requires no special permissions — only the ability to monitor on-chain oracle state and call `depositAsset()` or `depositETH()` during a staleness window. The attack is repeatable across any supported LST feed and requires no victim interaction.

## Recommendation

Add a configurable `MAX_STALENESS` constant and validate `updatedAt` in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
require(price > 0, "Invalid price");
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`MAX_STALENESS` should be set per-feed based on the Chainlink heartbeat (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer). Apply the same fix to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which checks round completeness but not elapsed time since `timestamp`.

## Proof of Concept

1. Chainlink's stETH/ETH feed last updated at `T-2h` with price `1.001e18`.
2. stETH depegs to `0.98e18` on-chain; the Chainlink feed has not yet updated (heartbeat not triggered, deviation threshold not crossed).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `getRsETHAmountToMint()` computes `(1000e18 * 1.001e18) / rsETHPrice` using the stale `1.001e18` price instead of the real `0.98e18`.
5. Attacker receives ~2.1% more rsETH than the deposited stETH is currently worth.
6. All existing rsETH holders are diluted; the attacker holds excess rsETH backed by no real value.

**Foundry fork test plan:** Fork mainnet, mock a Chainlink aggregator returning a price with `updatedAt = block.timestamp - 2 hours`, call `depositAsset()` with a known stETH amount, assert that `rsethAmountToMint` exceeds the fair value computed using the current on-chain stETH/ETH rate, and assert that existing holder share of TVL decreases.
