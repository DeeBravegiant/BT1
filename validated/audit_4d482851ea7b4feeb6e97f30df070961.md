### Title
Byzantine CKD Coordinator Delivers Attacker-Controlled Secret via Unvalidated `AppPublicKey` Response — (`crates/contract/src/lib.rs`, `respond_ckd`)

---

### Summary

`respond_ckd` applies a pairing-based cryptographic output check only for the `AppPublicKeyPV` variant. For the `AppPublicKey` (legacy, privately-verifiable) variant, the contract performs **zero validation** of `big_y` or `big_c` before resolving the yield. A single Byzantine coordinator node can substitute arbitrary G1 points — including the identity — into the response, causing the TEE app to derive a fully attacker-predicted secret.

---

### Impact Explanation

**Attack arithmetic:** The attacker (Byzantine coordinator) chooses `big_y = k·G` for a known scalar `k`, and sets `big_c` to any value they control (e.g., `identity`, or `k·G` for a known `k'`).

The TEE app computes:
```
secret = unmask(a) = big_c − big_y · a
```

If `big_c = identity` and `big_y = k·G`:
```
secret = 0 − k·G·a = −k·A
```

where `A = a·G` is the app's **public** key. The attacker knows `k` (they chose it) and `A` (it is public), so they can compute `−k·A` exactly. The TEE app's derived secret is fully known to the attacker before the app uses it.

The attacker can therefore:
- Impersonate the TEE app in any protocol that uses this derived key.
- Decrypt any data encrypted to the TEE app's derived key.

This is **unauthorized confidential key derivation output** — the TEE app receives a secret that was not produced by the honest MPC protocol, and the attacker knows it.

---

### Likelihood Explanation

- The `AppPublicKey` variant is the documented legacy variant and is still fully supported in production.
- The attack requires **one** Byzantine coordinator node — strictly below the threshold. The coordinator role rotates per request, so any single compromised participant can act as coordinator for some requests.
- The coordinator calls `respond_ckd` directly; no other participant can detect or prevent the substitution.
- The contract's existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` explicitly passes `[1u8; 48]` and `[2u8; 48]` as `big_y`/`big_c` (not valid curve points) and the contract accepts them, confirming the absence of any validation. [7](#0-6) 

---

### Recommendation

Add a cryptographic output check for the `AppPublicKey` variant in `respond_ckd`. Since `AppPublicKey` does not provide `pk2` (a G2 point), the pairing check used for `AppPublicKeyPV` cannot be applied directly. Options:

1. **Preferred:** Require all new CKD requests to use `AppPublicKeyPV`, and deprecate `AppPublicKey`. The PV variant already has both node-side (`aggregated_output_check`) and contract-side (`ckd_output_check`) validation.
2. **Interim:** Add a non-identity check on `big_y` and `big_c` (reject the identity point) and validate that both points are in the prime-order subgroup via `bls12381_p1_decompress` + pairing check against a known reference, or require the coordinator to provide a Schnorr proof of knowledge of the discrete log of `big_y`.

Also add `aggregated_output_check` to the non-PV coordinator path in `protocol.rs` (analogous to what `protocol_pv.rs` already does at line 66).

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
