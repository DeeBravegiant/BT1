### Title
Unvalidated `sender_address` in Invoke Transactions Bypasses Signature Verification via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function skips the `__validate__` entry-point call (the only place where an account's signature is checked) for any invoke transaction whose `sender_address` has a pending deploy-account in the mempool. Because `sender_address` is a free, user-supplied field that is never bound to the submitter's identity, an unprivileged attacker can impersonate any victim account that has a deploy-account in flight and inject an unsigned/arbitrarily-signed invoke transaction into the victim's nonce-1 slot in the mempool.

---

### Finding Description

`skip_stateful_validations` fires when three conditions are simultaneously true:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed). [1](#0-0) 

When those conditions hold, the function calls `account_tx_in_pool_or_recent_block(tx.sender_address())` and, if it returns `true`, returns `skip_validate = true`. [2](#0-1) 

`account_tx_in_pool_or_recent_block` simply checks whether the mempool's internal state or pool contains any transaction for that address — it does not verify that the caller owns that address. [3](#0-2) 

Back in `run_validate_entry_point`, the returned boolean is used to set `execution_flags.validate = !skip_validate`. When `validate = false`, `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling the account's `__validate__` entry point. [4](#0-3) [5](#0-4) 

`perform_pre_validation_stage` only checks nonce bounds and fee bounds — it does not verify the signature.

The transaction is then forwarded to the mempool via `mempool_client.add_tx(add_tx_args)` with no further signature check. [6](#0-5) 

---

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit an `invoke` transaction with:

- `sender_address = victim_address`
- `nonce = 1`
- Any arbitrary (invalid) signature

The gateway admits this transaction because `skip_validate = true` suppresses the only signature check. The invalid transaction occupies the victim's nonce-1 slot in the mempool. The victim's legitimate nonce-1 invoke must then either pay a higher fee to displace it (fee escalation) or wait until the attacker's transaction is eventually rejected during batcher execution and the slot is freed.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The trigger is fully unprivileged and observable on-chain/in-mempool. Any time a new account submits a `deploy_account` transaction, the window is open. The attacker needs only to:

1. Monitor the mempool for `deploy_account` transactions.
2. Immediately submit a crafted `invoke` with the victim's address and `nonce = 1`.

No special keys, permissions, or network access beyond normal RPC submission are required.

---

### Recommendation

The `skip_stateful_validations` path was designed as a UX convenience for users who broadcast `deploy_account + invoke` atomically. The invariant it must preserve is: **the invoke's `sender_address` must be the same account whose deploy-account is in the mempool, and the invoke must have been submitted by the same party.**

Two complementary fixes:

1. **Bind the skip to a verified deploy-account hash.** When a `deploy_account` is admitted to the mempool, record its `contract_address` → `tx_hash` mapping. In `skip_stateful_validations`, require that the mempool entry for `sender_address` is specifically a `deploy_account` transaction (not just any transaction), and that the invoke's signature is at least structurally non-empty before skipping.

2. **Perform a lightweight signature-format check before skipping.** Even without running the full `__validate__` entry point, the gateway can reject transactions whose signature field is empty or whose transaction hash does not match the expected format for the claimed `sender_address`, raising the cost of the attack.

---

### Proof of Concept

POST /gateway/add_transaction
{ type: "DEPLOY_ACCOUNT", sender_address: V, nonce: 0, signature: [valid_r, valid_s], ... }
# → Mempool now contains V's deploy_account; account_tx_in_pool_or_recent_block(V) == true.

# Step 2 – Attacker submits invoke impersonating V.
POST /gateway/add_transaction
{
  type: "INVOKE",
  sender_address: V,       # victim's address, freely supplied
  nonce: 1,                # triggers skip_stateful_validations
  signature: [0xdead, 0xbeef],  # arbitrary, never checked
  calldata: [<attacker_calldata>],
  resource_bounds: { l2_gas: { max_amount: 1, max_price_per_unit: 1 } },
  ...
}

# Gateway evaluation:
#   tx.nonce() == 1  ✓
#   account_nonce == 0  ✓  (V not deployed yet)
#   account_tx_in_pool_or_recent_block(V) == true  ✓
#   → skip_validate = true
#   → __validate__ NOT called
#   → transaction admitted to mempool at (V, nonce=1)

# Result: V's nonce-1 slot is occupied by attacker's invalid transaction.
# V's legitimate nonce-1 invoke is blocked or must pay escalated fee.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
