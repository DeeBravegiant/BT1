### Title
Single Byzantine Participant Can Corrupt `ckd_pv` Aggregate and Prevent `respond_ckd`, Causing Deposit Loss — (`crates/threshold-signatures/src/confidential_key_derivation/protocol_pv.rs`)

---

### Summary

The `ckd_pv` coordinator aggregates all participant shares by blind addition and then runs a single `aggregated_output_check` on the result. There is no per-share verification. A single Byzantine participant (below threshold) can send a random `(big_y, big_c)` pair, corrupt the aggregate, and cause `aggregated_output_check` to fail. The resulting `ProtocolError::AssertionFailed` propagates through `CKDComputation::compute` and prevents `respond_ckd` from being called. Combined with the NEAR yield-resume timeout, this can cause the user's deposit to be consumed.

---

### Finding Description

In `do_ckd_coordinator` (`protocol_pv.rs`), the coordinator collects one share from every selected participant via `recv_from_others`, adds them together unconditionally, and only then verifies the aggregate:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}

let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);

if !aggregated_output_check(&ckd_output, app_pk, &key_pair.public_key, &hash_point) {
    return Err(ProtocolError::AssertionFailed(
        "CKD output failed to verify".to_string(),
    ));
}
``` [1](#0-0) 

`aggregated_output_check` verifies `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` via `multi_miller_loop`. If any single participant sends a random point pair, the pairing equation fails and the function returns `false`. [2](#0-1) 

The coordinator selects exactly `threshold.value()` participants — the minimum required — so every selected participant's share is mandatory: [3](#0-2) 

The `ProtocolError` propagates through `run_protocol("ckd_pv", ...)` in `CKDComputation::compute`: [4](#0-3) 

And then through `perform_leader_centric_computation` via `?` in `make_ckd_leader`, preventing any call to `respond_ckd`: [5](#0-4) 

On the contract side, if `respond_ckd` is never called, the yield-resume callback fires with `Err(PromiseError::Failed)`, pops the pending request, and calls `fail_on_timeout` — consuming the deposit: [6](#0-5) 

---

### Impact Explanation

A retry mechanism exists (the pending-request queue re-issues the attempt after `CHECK_EACH_REQUEST_INTERVAL` when the previous attempt is dropped). However:

1. The Byzantine participant is re-eligible for selection on every retry, since `select_random_active_participants_including_me` draws `threshold` participants at random from all running participants.
2. For a 3-of-5 deployment, the Byzantine participant is selected with probability 3/5 = 60% per attempt. The expected number of retries before a clean attempt is ~2.5, but variance is high and the Byzantine participant can be selected in many consecutive retries.
3. Each failed attempt consumes the full `ckd.timeout_sec` before the retry fires.
4. If the NEAR yield-resume window is exhausted before a clean attempt completes, the deposit is permanently consumed and the request is never resolved.

The coordinator also has no way to identify which participant sent the bad share (no per-share proof), so it cannot blacklist the Byzantine participant between retries.

---

### Likelihood Explanation

The attacker must be a registered MPC participant. This is a meaningful precondition, but registered participants are exactly the entities the protocol must be Byzantine-fault-tolerant against below threshold. The attack requires no special tooling: the Byzantine node simply sends random G1 points instead of its computed share. The attack is repeatable on every CKD request the Byzantine participant is selected for.

---

### Recommendation

Add per-share verification before aggregation in `do_ckd_coordinator`. Each participant should attach a zero-knowledge proof (e.g., a Schnorr proof of knowledge of `y` such that `big_y = y·G` and `big_c = x_i·H + y·pk1`) alongside their `(big_y, big_c)` share. The coordinator verifies each proof individually before adding the share to the aggregate. A participant whose proof fails is excluded and the computation continues with the remaining honest participants (provided enough honest participants remain). This is the standard approach in verifiable secret sharing and threshold protocols.

---

### Proof of Concept

A unit test that demonstrates the issue:

```rust
// In protocol_pv.rs test module:
// Run ckd_pv with one participant sending random (big_y, big_c).
// Assert the coordinator returns ProtocolError::AssertionFailed.
// This proves a single Byzantine participant below threshold breaks the protocol.

let mut rng = MockCryptoRng::seed_from_u64(42);
let (app_id, _app_sk, app_pk) = generate_ckd_app_package(&mut rng);
let participants = generate_participants(3); // threshold = 2, total = 3
let coordinator = participants[0];
let (f, pk) = generate_test_keys(1, &mut rng); // degree 1 = threshold-1

// Byzantine participant (participants[1]) sends random points
// instead of compute_signature_share output.
// The coordinator aggregates and aggregated_output_check fails.
// Expected: run_protocol returns Err(ProtocolError::AssertionFailed).
```

The path is: malformed share → blind aggregation in `do_ckd_coordinator` → `aggregated_output_check` returns `false` → `ProtocolError::AssertionFailed` → `CKDComputation::compute` error → no `respond_ckd` → NEAR yield timeout → deposit consumed.

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`
fn aggregated_output_check(
    output: &CKDOutput,
    app_pk: &PublicVerificationKey,
    public_key: &VerifyingKey,
    hash_point: &ElementG1,
) -> bool {
    if !check_valid_point_g1(output.big_c.into()) || !check_valid_point_g1(output.big_y.into()) {
        return false;
    }
    multi_miller_loop(&[
        (output.big_c, -ElementG2::generator()),
        (output.big_y, app_pk.pk2),
        (*hash_point, public_key.to_element()),
    ])
}
