### Title
Proposer Deposits Permanently Locked When a Competing Update Is Executed — (File: `crates/contract/src/update.rs`)

---

### Summary

`propose_update` collects a storage-staking deposit from every proposer, but `do_update` clears all competing entries without refunding those deposits. Any NEAR attached by proposers whose updates are superseded is permanently frozen in the contract with no recovery path.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` is `#[payable]` and enforces a deposit proportional to the serialized size of the proposed update: [1](#0-0) 

The required amount is computed by `ProposedUpdates::required_deposit`, which multiplies `bytes_used` (struct overhead + 128 × `AccountId` size + code/config bytes) by the per-byte storage cost: [2](#0-1) 

Only the excess above `required` is refunded to the proposer at call time: [3](#0-2) 

The `required` portion stays in the contract. When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then **bulk-clears every other entry and all votes**: [4](#0-3) 

`UpdateEntry` stores only `update` and `bytes_used`; it records neither the proposer's `AccountId` nor the deposit amount: [5](#0-4) 

Because the proposer identity and deposit value are never persisted, `do_update` has no information with which to issue refunds. The NEAR tokens paid by every non-winning proposer are permanently locked in the contract.

---

### Impact Explanation

Every participant who calls `propose_update` with a contract binary (potentially hundreds of KB) pays a deposit on the order of several NEAR. When any other update reaches threshold and executes, `entries.clear()` silently discards all competing proposals and their associated deposits. There is no `cancel_update`, no `withdraw_deposit`, and no post-execution sweep that returns funds. The loss is permanent and proportional to the size of the proposed binary.

This constitutes permanent freezing of funds controlled by the chain-signature contract, matching the Critical/Medium allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The scenario requires only that two or more participants independently propose different updates — a routine governance situation. No adversarial intent is needed; the loss occurs as a side-effect of normal threshold voting. A Byzantine participant strictly below the signing threshold can also deliberately trigger the loss by proposing a large competing update before the legitimate update reaches threshold, causing honest proposers to lose their deposits when the legitimate update executes.

---

### Recommendation

Extend `UpdateEntry` to record the proposer's `AccountId` and the exact deposit amount at proposal time. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::transfer` back to each proposer. This mirrors the refund pattern already used in `propose_update` itself and in `submit_participant_info`.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 400 KB contract binary. `bytes_used` ≈ 400 000 + overhead bytes; at NEAR's storage cost (~10^19 yoctoNEAR/byte) the required deposit is ≈ 4 NEAR. A pays exactly 4 NEAR; no excess is refunded. [6](#0-5) 

2. Participant B calls `propose_update` with a different 400 KB binary. B also pays ≈ 4 NEAR.

3. Threshold participants vote for A's update via `vote_update`. Once the threshold is reached, `do_update` is invoked: [7](#0-6) 

4. Inside `do_update`, after removing A's entry, `self.entries.clear()` discards B's entry: [8](#0-7) 

5. B's 4 NEAR deposit is now held by the contract with no code path that can return it. `proposed_updates()` returns an empty map; there is no `withdraw` or `cancel` endpoint. B's funds are permanently frozen.

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
