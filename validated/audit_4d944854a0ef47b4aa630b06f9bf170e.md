### Title
No Explicit Emergency Pause Mechanism Despite `accept_requests` Guards on All Critical Functions — (File: `crates/contract/src/lib.rs`)

---

### Summary

`MpcContract` contains an `accept_requests: bool` field that gates every user-facing and node-facing critical function. However, there is no governance-controlled `pause()` function that participants can invoke to set `accept_requests = false` in an emergency. The only code path that sets `accept_requests = false` is an indirect side-effect inside `verify_tee()` when TEE validation fails below threshold. This is the structural analog to the reported "Emergency Pause Functionality Not Implemented" finding.

---

### Finding Description

`MpcContract` declares `accept_requests: bool` as a contract-state field and enforces it as a gate in every critical execution path:

- `check_request_preconditions()` — called by `sign`, `request_app_private_key`, and `verify_foreign_transaction` [1](#0-0) 
- `respond()` [2](#0-1) 
- `respond_ckd()` [3](#0-2) 
- `respond_verify_foreign_tx()` [4](#0-3) 

The field is initialized to `true` in both `init()` and `init_running()`: [5](#0-4) 

The **only** code that ever writes `false` to `accept_requests` is inside `verify_tee()`, and only when TEE validation finds fewer than `threshold` valid participants: [6](#0-5) 

`verify_tee()` also writes `true` back when validation passes: [7](#0-6) 

The `update_config()` function is `#[private]` (self-call only) and updates `Config`, not `accept_requests`. [8](#0-7) 

There is no `pause()` or equivalent governance-voted function that participants can call to set `accept_requests = false` directly. The contract's "pause" state is exclusively a side-effect of TEE attestation validation, not a manually triggerable emergency control.

---

### Impact Explanation

If a critical exploit is discovered while TEE validation is passing (i.e., all nodes hold valid attestations), participants have no on-chain mechanism to halt new `sign`, `request_app_private_key`, `verify_foreign_transaction`, or `respond*` calls. The only remediation path is `propose_update` → `vote_update` → contract upgrade, which requires threshold votes and multiple blocks of latency. During that window, an active exploit (e.g., a bug in signature verification logic, a compromised node submitting forged responses, or a malicious foreign-chain input producer) can continue to drain or corrupt pending requests. This maps to the **Medium** allowed impact: *contract execution-flow manipulation that breaks production safety/accounting invariants*.

---

### Likelihood Explanation

Medium. The contract is deployed on mainnet handling real cross-chain funds. TEE attestations are expected to remain valid during normal operation, meaning `verify_tee()` will not naturally set `accept_requests = false` during an active exploit. The governance upgrade path (`propose_update` / `vote_update`) is the only alternative, and it is slow by design. Any window between exploit discovery and successful upgrade is unmitigated.

---

### Recommendation

Add an explicit threshold-voted emergency pause and unpause mechanism:

```rust
// Requires threshold votes to prevent abuse
pub fn vote_pause(&mut self) -> Result<(), Error> { ... }
pub fn vote_unpause(&mut self) -> Result<(), Error> { ... }
```

These should accumulate votes (similar to `vote_cancel_resharing`) and flip `accept_requests` only when the governance threshold is reached, giving participants a direct, auditable, and fast-acting emergency control independent of TEE validation state.

---

### Proof of Concept

1. Contract is `Running`, all nodes have valid TEE attestations → `verify_tee()` sets `accept_requests = true`. [7](#0-6) 

2. A critical bug is discovered in `respond()` signature verification logic. Participants want to halt the contract immediately.

3. No participant can call any function to set `accept_requests = false` — `verify_tee()` will return `true` (all attestations valid) and set `accept_requests = true`. [9](#0-8) 

4. Participants must call `propose_update()` + gather threshold `vote_update()` calls to deploy a patched contract. [10](#0-9) 

5. During the multi-block governance window, `sign` and `respond*` remain fully open, and the exploit continues unimpeded through the unguarded `accept_requests = true` path. [1](#0-0)

    pub fn vote_update(&mut self, id: UpdateId) -> Result<bool, Error> {
        log!(
            "vote_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        let ProtocolContractState::Running(running_state) = &self.protocol_state else {
            env::panic_str("protocol must be in running state");
        };

        let threshold = self.threshold()?;

        let voter = self.voter_or_panic();
        if self.proposed_updates.vote(&id, voter).is_none() {
            return Err(InvalidParameters::UpdateNotFound.into());
        }

        // Filter votes to only count current participants voting for this specific update.
        // This ensures correctness even if the cleanup promise in MpcContract::vote_reshared() fails.
        let valid_votes_count = running_state
            .parameters
            .participants()
            .participants()
            .iter()
            .filter(|(account_id, _, _)| {
                self.proposed_updates
                    .vote_by_participant
                    .get(account_id)
                    .is_some_and(|voted_id| *voted_id == id)
            })
            .count();

        // Not enough votes from current participants, wait for more.
        if (valid_votes_count as u64) < threshold.value() {
            return Ok(false);
        }

        let update_gas_deposit = Gas::from_tgas(self.config.contract_upgrade_deposit_tera_gas);

        let Some(_promise) = self.proposed_updates.do_update(&id, update_gas_deposit) else {
            return Err(InvalidParameters::UpdateNotFound.into());
        };

        Ok(true)
    }
