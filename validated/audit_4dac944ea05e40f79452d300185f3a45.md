### Title
Exponential Reference Script Fee Tier Bypassed by Splitting Usage Across Multiple Transactions - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs`)

### Summary

The Conway/Dijkstra era's exponential tiered pricing for reference scripts (`tierRefScriptFee`) is enforced per-transaction. An unprivileged transaction sender can bypass the exponential pricing entirely by splitting reference script usage across many small transactions within the same block, each staying within the first (cheapest) pricing tier, paying only the base linear rate for the full block-level reference script budget instead of the intended exponential rate.

### Finding Description

The Conway era introduced exponential tiered pricing for reference scripts to deter DDoS attacks (ADR 009). The pricing function `tierRefScriptFee` applies an exponential multiplier (`1.2` per 25KiB stride) to the total reference script size within a single transaction: [1](#0-0) 

The minimum fee for a transaction is computed by `getConwayMinFeeTxUtxo`, which passes only the per-transaction reference script size to `tierRefScriptFee`: [2](#0-1) 

The block-level check in `validateBodyRefScriptsSizeTooBig` enforces only a hard cap (`maxRefScriptSizePerBlock`, default 1MiB) on the total reference script size across all transactions in a block — it does **not** apply exponential pricing across transactions: [3](#0-2) 

Because the exponential pricing resets to the base rate for every new transaction, an attacker can split their reference script usage into many transactions each exactly at the first-tier boundary (25,600 bytes), paying only the base rate for each.

**Concrete fee comparison** (with default parameters: stride=25,600 bytes, multiplier=1.2, base=15 lovelace/byte):

- **5 transactions × 200KiB each** (filling 1MiB block): each pays the full 8-tier exponential fee ≈ 6,334,648 lovelace → total ≈ **31,673,240 lovelace**
- **40 transactions × 25KiB each** (same 1MiB total): each pays only the first-tier fee = 25,600 × 15 = 384,000 lovelace → total = **15,360,000 lovelace**

The attacker pays approximately **half** the intended cost for the same total reference script load on the network. The savings grow further as the multiplier increases or as the attacker uses even smaller per-transaction chunks.

In the Dijkstra era, `batchNonDistinctRefScriptsSize` is used for the per-transaction size limit check (including sub-transactions), but the fee calculation still uses `getConwayMinFeeTxUtxo` → `txNonDistinctRefScriptsSize` (top-level only), introducing an additional dimension where sub-transaction reference scripts are size-checked but not exponentially priced: [4](#0-3) [5](#0-4) 

### Impact Explanation

This falls under the **Medium** allowed impact: *"Attacker-controlled transactions... modify fees... outside design parameters."*

The explicit design goal of the exponential pricing (documented in ADR 009) was to make large reference script usage expensive enough to deter DDoS attacks, while keeping small usage affordable. By splitting across transactions, an attacker can submit the maximum block-level reference script load (1MiB) at roughly the base linear rate, paying significantly less than the protocol designers intended. This weakens the economic deterrent against reference-script-based DDoS attacks that was specifically introduced after the June 2024 mainnet attack. [6](#0-5) 

### Likelihood Explanation

**High likelihood.** Any transaction sender can exploit this by constructing multiple small transactions. No privileged access, key compromise, or consensus majority is required. The attacker only needs to:
1. Know the `refScriptCostStride` parameter (publicly readable from protocol parameters)
2. Submit multiple transactions per block, each referencing ≤ `refScriptCostStride` bytes of reference scripts

Block producers have no mechanism to reject such a split — each individual transaction is valid and pays its correct per-transaction minimum fee.

### Recommendation

Apply the exponential pricing cumulatively across all transactions in a block, analogous to how `totalRefScriptSizeInBlock` already tracks the cumulative total for the hard-cap check. Specifically:

1. Track a running cumulative reference script size at the block level (already computed in `totalRefScriptSizeInBlock`).
2. Compute each transaction's reference script fee as `tierRefScriptFee(cumulativeSize + txSize) - tierRefScriptFee(cumulativeSize)` — i.e., the marginal cost at the current cumulative position in the pricing curve.
3. Alternatively, introduce a block-level exponential fee that is distributed across transactions (e.g., via a block-level fee floor).

This mirrors the recommendation in the external report: limit fee-tier resets per block rather than per transaction.

### Proof of Concept

Using the existing `tierRefScriptFee` function (exported from `Cardano.Ledger.Conway.Tx`):

```haskell
-- Default parameters (Conway/Dijkstra mainnet)
stride     = 25600  -- bytes
multiplier = 1.2
baseFee    = 15     -- lovelace/byte

-- Cost of one 200KiB transaction (8 tiers)
singleTxCost = tierRefScriptFee multiplier stride baseFee (8 * stride)
-- ≈ 6,334,648 lovelace

-- Cost of 40 transactions of 25KiB each (each resets to tier 1)
splitTxCost = 40 * tierRefScriptFee multiplier stride baseFee stride
-- = 40 * 384,000 = 15,360,000 lovelace

-- 5 large transactions vs 40 small transactions for the same 1MiB block:
largeTxTotal = 5 * singleTxCost   -- ≈ 31,673,240 lovelace
smallTxTotal = 40 * splitTxCost   -- = 15,360,000 lovelace

-- Attacker saves: 31,673,240 - 15,360,000 = 16,313,240 lovelace (~51% discount)
-- for identical total reference script load on the network
```

The block-level hard cap (`maxRefScriptSizePerBlock = 1MiB`) is respected in both cases; only the fee paid differs. The `validateBodyRefScriptsSizeTooBig` check passes for both scenarios. [7](#0-6) [8](#0-7)

tierRefScriptFee ::
  HasCallStack =>
  -- | Growth factor or step multiplier
  Rational ->
  -- | Increment size in which price grows linearly according to the price
  Int ->
  -- | Base fee. Currently this is customizable by `ppMinFeeRefScriptCostPerByteL`
  Rational ->
  -- | Total RefScript size in bytes
  Int ->
  Coin
tierRefScriptFee multiplier sizeIncrement
  | multiplier <= 0 || sizeIncrement <= 0 = error "Size increment and multiplier must be positive"
  | otherwise = go 0
  where
    go !acc !curTierPrice !n
      | n < sizeIncrement =
          Coin $ floor (acc + toRational n * curTierPrice)
      | otherwise =
          go (acc + sizeIncrementRational * curTierPrice) (multiplier * curTierPrice) (n - sizeIncrement)
    sizeIncrementRational = toRational sizeIncrement

totalRefScriptSizeInBlock ::
  (AlonzoEraTx era, BabbageEraTxBody era) => ProtVer -> StrictSeq (Tx TopTx era) -> UTxO era -> Int
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      let updatedUtxo = accUtxo `Map.union` unUTxO toAdd
          toAdd
            | IsValid True <- tx ^. isValidTxL = txouts $ tx ^. bodyTxL
            | otherwise = collOuts $ tx ^. bodyTxL
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)

Unfortunately things did not go exactly as planned, because on [June 25th 2024 an attack like this actually took place](https://cardanospot.io/news/ddos-attack-on-the-cardano-blockchain-mRIKAzZTNnzq5NGd). This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.

## Decision

Linear pricing was either too expensive when the multiplier was set too high or was an inadequate deterrent when the multiplier was set too low. Therefore, we needed to implement a pricing mechanism that would be very expensive for usage with large quantities of large plutus scripts, while keeping the pricing reasonably low for the most common use case of a total size of reference scripts of at most 25KiB per transaction. One of the constraints we had to operate under was inability to add any new protocol parameters, since that was a bit too late in the release cycle of the Conway era. In other words we had to hard code some values, which will be turned into proper protocol parameters in the next era.
