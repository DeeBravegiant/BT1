**`vote_code_hash`** — the vote count returned by `tee_state.vote()` is compared directly to the threshold: [1](#0-0)

`vote_code_hash` has **neither** of these defenses. The code comment on `remove_non_participant_update_votes` explicitly says "MpcContract::vote_update uses filtering to ensure correctness even if this cleanup fails" — but no equivalent fix was applied to `vote_code_hash`.

## Verdict

### Title
Stale TEE Code-Hash Votes from Removed Participants Count Toward Threshold in `vote_code_hash` — (`crates/contract/src/lib.rs`)

### Summary
`CodeHashesVotes::count_votes` counts all stored votes without filtering by current participants. If the detached `clean_tee_status` promise fails after a resharing, stale votes from removed participants persist in `tee_state.votes` and are counted by `vote_code_hash` toward the threshold, allowing `whitelist_tee_proposal` to fire with fewer than threshold current-participant votes.

### Finding Description
After `vote_reshared` completes a resharing that removes participants, it spawns `clean_tee_status` as a detached promise: [7](#0-6) 

If this promise fails (e.g., insufficient `clean_tee_status_tera_gas`, or the contract is not in Running state when the promise executes), stale votes from removed participants remain in `tee_state.votes.proposal_by_account`.

When a current participant subsequently calls `vote_code_hash`, `AuthenticatedParticipantId::new` correctly validates the caller is a current participant: [8](#0-7) 

But `CodeHashesVotes::count_votes` counts every entry in `proposal_by_account` regardless of whether the voter is still a participant: [9](#0-8) 

The returned count includes stale votes, so the threshold check `votes >= self.threshold()?.value()` can pass with fewer than threshold current-participant votes, triggering `whitelist_tee_proposal`.

### Impact Explanation
A TEE Docker image hash is whitelisted with fewer than threshold current-participant votes. This breaks the TEE security invariant: the whitelist is supposed to require threshold agreement from the active participant set. A whitelisted malicious image could be used by MPC nodes, potentially enabling key-share extraction or unauthorized signing.

### Likelihood Explanation
The code explicitly acknowledges that the cleanup promise can fail — the comment on `remove_non_participant_update_votes` says "MpcContract::vote_update uses filtering to ensure correctness even if this cleanup fails." The `clean_tee_status_tera_gas` is a configurable value; if set too low, the promise fails silently. The detached promise pattern provides no failure signal. The scenario requires a resharing that removes participants who previously voted for a target hash, plus cleanup failure — realistic in a network that undergoes participant rotation.

### Recommendation
Apply the same defensive pattern used in `vote_update`: at vote-counting time in `vote_code_hash`, filter `tee_state.votes.proposal_by_account` against the current participant set before comparing to the threshold. This makes the function correct regardless of whether `clean_tee_status` ran. Alternatively, add inline participant filtering to `CodeHashesVotes::count_votes` (accepting a `&Participants` argument), mirroring `TeeVerifierVotes::vote`: [6](#0-5) 

### Proof of Concept
1. Set up a contract with N=5, T=3 participants (P0–P4).
2. P0 and P1 vote for a target `code_hash` via `vote_code_hash`.
3. Perform a resharing removing P0 and P1; new set is {P2, P3, P4}, T=3.
4. Do **not** execute the detached `clean_tee_status` promise (simulate failure by skipping it in the test environment).
5. P2 calls `vote_code_hash` for the same `code_hash`.
6. `count_votes` returns 3 (P0 + P1 stale + P2 fresh).
7. Assert `whitelist_tee_proposal` was called — only 1 current-participant vote was cast.

The existing unit test `test_clean_non_participant_votes_removes_stale_votes` confirms the stale votes exist before cleanup runs: [10](#0-9)

    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
    }
