Audit Report

## Title
Pusher-delegation consent signature is replayable before deadline, allowing creator to reinstate revoked oracle write authority — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

## Summary
`allowPushers` verifies a pusher's EIP-191 consent signature bound only to `(chainid, address(this), deadline, pusher, creator)` with no nonce and no action field. After a pusher self-revokes via `revokePusher()`, the creator can replay the original signature bytes at any time before the deadline to overwrite `namespaceRemapping[pusher]` back to themselves, undoing the revocation. A compromised pusher key can then push arbitrary slot words into the creator's namespace, producing bad-price execution in any pool consuming that oracle feed.

## Finding Description
`allowPushers` constructs the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

There is no nonce, no boolean action field, and no used-signature registry. `_ensureDeadline` only enforces `block.timestamp <= deadline`: [2](#0-1) 

`revokePusher` clears the mapping but records nothing on-chain about the revocation:

```solidity
function revokePusher() external {
    address creator = namespaceRemapping[msg.sender];
    if (creator == address(0) || creator == msg.sender) revert NoSelfRemapping();
    namespaceRemapping[msg.sender] = address(0);
    emit PusherRevoked(msg.sender, creator);
}
``` [3](#0-2) 

Because the signed payload is identical for every `allowPushers` call with the same `(deadline, pusher, creator)` triple, the creator can call `allowPushers` again with the original signature bytes before the deadline passes, restoring `namespaceRemapping[pusher] = creator` and nullifying the pusher's self-revocation. The code comment at L186–191 acknowledges the deadline as a partial mitigation but does not address pre-deadline replay. [4](#0-3) 

## Impact Explanation
The `CompressedOracle` feeds bid/ask prices into `MetricOmmPool.swap` via `_getBidAndAskPriceX64()`, which are consumed directly by `SwapMath.midAndSpreadFeeX64FromBidAsk` and passed to `_executeSwap`. [5](#0-4) 

If a pusher revokes because their signing key was compromised, the creator's replay re-establishes the delegation. The attacker holding the compromised key can then push arbitrary `mid`/`spread` values into the creator's namespace slots. Those values reach live swap execution, satisfying the **bad-price execution** and **pool insolvency** impact gates: traders receive more output than the oracle permits, or the pool receives less input than owed, directly draining LP principal.

## Likelihood Explanation
- The creator is not a trusted party (oracle admin holds `ADMIN_ROLE`; the creator is a public namespace owner equivalent to a public oracle pusher).
- The original `allowPushers` calldata is permanently public on-chain; the creator needs no additional capability to extract the signature bytes.
- The pusher's revocation is a unilateral self-service action, giving the creator a clear motive to replay when the pusher revokes against their wishes.
- The replay window equals the full deadline duration chosen by the creator when calling `allowPushers`; nothing in the contract caps this value.
- No special privileges beyond being the original creator are required; the attack is repeatable as many times as desired before the deadline.

## Recommendation
**Short term:** Add a per-pusher nonce to the signed digest and increment it on every successful `allowPushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender, pusherNonce[pusher]
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonce[pusher]++;
namespaceRemapping[pusher] = msg.sender;
```

A replayed old signature recovers a stale nonce and fails verification.

**Long term:** Bind an explicit `allow`/`revoke` action field into the signed payload so a consent signature cannot be repurposed as a re-delegation after revocation.

## Proof of Concept

```
1. Pusher signs consent for Creator with deadline = block.timestamp + 365 days.
   digest = keccak256(abi.encode(chainid, oracle, deadline, pusher, creator))
   sig = pusher.sign(digest)

2. Creator calls allowPushers(deadline, [pusher], [sig])
   → namespaceRemapping[pusher] = creator  ✓

3. Pusher's key is compromised; pusher calls revokePusher()
   → namespaceRemapping[pusher] = address(0)  ✓

4. Creator calls allowPushers(deadline, [pusher], [sig])  ← same sig, same deadline
   → _ensureDeadline passes (block.timestamp <= deadline still holds)
   → ECDSA.recover returns pusher  ← sig is cryptographically valid
   → namespaceRemapping[pusher] = creator  ← revocation undone ✗

5. Attacker (holding compromised pusher key) calls fallback with manipulated slot words
   → namespaceRemapping[msg.sender] = creator → writes into creator's namespace
   → Pool reads manipulated bid/ask → bad-price swap execution → LP loss
```
