### Title
Gateway Admission Validates Against Point-in-Time `l2_gas_price` Instead of `next_l2_gas_price` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `validate_resource_bounds` reads the **current block's** `l2_gas_price` from a cached `BlockInfo` snapshot to threshold-check incoming transactions. However, every admitted transaction will be executed in the **next** block, whose L2 gas price is already stored as `next_l2_gas_price` in the same block header. Using the stale point-in-time price instead of the already-known next-block price causes the gateway to admit transactions that will fail at execution (when the price rises) and to reject transactions that would succeed (when the price falls). A developer TODO in the production code explicitly acknowledges the wrong field is being read.

---

---

### Impact Explanation

**Case 1 — price rises between blocks (over-admission):**  
`next_l2_gas_price > l2_gas_price`. A transaction whose `max_l2_gas_price` satisfies `threshold_old ≤ max_l2_gas_price < threshold_new` passes the gateway check but will be rejected at blockifier execution with `MaxGasPriceTooLow` (or `InsufficientResourceBounds`). The gateway has admitted an invalid transaction.

**Case 2 — price falls between blocks (under-admission):**  
`next_l2_gas_price < l2_gas_price`. A transaction whose `max_l2_gas_price` satisfies `threshold_new ≤ max_l2_gas_price < threshold_old` is rejected by the gateway even though it would succeed in the next block. A valid transaction is silently dropped.

Both cases match the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The EIP-1559 L2 gas price adjusts every block based on gas usage. Under normal load the price drifts by up to `1/gas_price_max_change_denominator` per block. The `min_gas_price_percentage` config (default non-zero) means the threshold window is a fraction of the current price, so any block-to-block price movement creates a gap between the stale threshold and the correct one. The condition is reachable whenever `validate_resource_bounds = true` (the production setting) and the L2 gas price changes between consecutive blocks, which is the common case.

---

### Recommendation

Replace the read of `block_header.l2_gas_price.price_in_fri` with `block_header.next_l2_gas_price` in `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, or expose a dedicated accessor that returns `next_l2_gas_price` and use it in `validate_resource_bounds`. This resolves the TODO comment and aligns the admission threshold with the price that will actually govern execution.

---

### Proof of Concept

1. Block N is committed with `l2_gas_price = 100 FRI` and `next_l2_gas_price = 130 FRI` (30 % increase due to high usage).
2. Gateway creates `GatewayFixedBlockSyncStateClient` for block N; `block_info_cache` stores `l2_gas_price = 100`.
3. User submits a V3 transaction with `max_l2_gas_price = 70 FRI` and `min_gas_price_percentage = 50`.
4. `threshold = 50 % × 100 = 50`. Check: `70 ≥ 50` → **admitted**.
5. Batcher executes the transaction in block N+1 at `l2_gas_price = 130 FRI`. Check: `70 < 130` → **`MaxGasPriceTooLow`**, transaction fails.
6. Conversely, a transaction with `max_l2_gas_price = 55 FRI` would be rejected at the gateway (`55 ≥ 50` passes, but if price had fallen to 80 and threshold_new = 40, a transaction with `max_l2_gas_price = 45` would be rejected at gateway threshold_old=50 even though 45 ≥ 40 = threshold_new).

The root cause is the single line `l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?` in `get_block_info_from_sync_client` — it should read `block_header.next_l2_gas_price` instead. [7](#0-6) [8](#0-7) [9](#0-8)

pub fn calculate_next_l2_gas_price_for_fin(

    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,

    if let Some(override_value) = override_l2_gas_price_fri {

    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);

        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
