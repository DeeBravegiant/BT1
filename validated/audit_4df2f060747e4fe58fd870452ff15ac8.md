### Title
Single-Participant Empty-Set Registration Permanently Blocks All Foreign-Chain Verification — (File: `crates/contract/src/lib.rs`)

### Summary

`register_foreign_chain_support` accepts an empty `SupportedForeignChains` set from any single participant and stores it verbatim in `node_foreign_chain_support.foreign_chain_support_by_node`. Because `get_supported_foreign_chains()` requires the supporting-node set to be a **superset of every active participant**, one participant registering an empty set immediately collapses the returned chain list to empty. `verify_foreign_transaction` gates on this result, so all foreign-chain verification requests are rejected from that moment forward.

### Finding Description

`register_foreign_chain_support` performs no validation on the content of the supplied set:

```rust
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);   // empty set accepted
    Ok(())
}
``` [1](#0-0) 

`get_supported_foreign_chains()` then computes a strict intersection: a chain is returned only when the set of nodes that registered it is a **superset** of all active participants.

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain {
    Some(foreign_chain)
} else {
    None
}
``` [2](#0-1) 

If participant `P` registers `{}`, `P`'s account ID appears in `active_participant_account_ids` but contributes no chain to `foreign_chain_to_node_mapping`. No chain's supporting-node set can therefore be a superset of all active participants, so the function returns empty. The project's own design document confirms the consequence:

> "A single node that registers an empty list (or hasn't registered yet) drops **every** chain — one operator can take the whole feature down."



The structural parallel to H-03 is exact: in the Solidity case `delete` writes `address(0)` into an array slot; here `insert(account_id, {})` writes an empty-set entry into the map. In both cases the "null" entry is never rejected at write time, and downstream iteration over the collection produces a broken result.

### Impact Explanation

`verify_foreign_transaction` rejects any request whose target chain is absent from `get_supported_foreign_chains()`. After one participant registers an empty set, every `verify_foreign_transaction` call fails immediately. This permanently freezes the foreign-chain verification flow — all pending and future cross-chain requests are denied — until the participant re-registers a non-empty set. This matches the **Medium** impact class: *contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration*.

### Likelihood Explanation

Any single active participant (one of N, strictly below the signing threshold) can trigger this with a single on-chain call. No collusion, no key material, and no privileged access beyond normal participant status is required. The call is cheap and reversible by the attacker, making it usable as a targeted griefing tool against the foreign-chain bridge feature.

### Recommendation

1. **Reject empty registrations at write time** — add a guard in `register_foreign_chain_support` that returns an error if `foreign_chain_support.is_empty()`.
2. **Adopt the threshold-based availability model** — the project's own design document proposes replacing the strict-intersection rule with `get_available_foreign_chains()`, which requires only ≥ `signing_threshold` participants to cover a chain. This eliminates the single-participant veto entirely. [3](#0-2) 

### Proof of Concept

1. Contract is Running with participants `{A, B, C}`, threshold = 2. All three have registered `{Bitcoin, Ethereum}`. `get_supported_foreign_chains()` returns `{Bitcoin, Ethereum}`.
2. Participant `A` calls `register_foreign_chain_support({})` (empty set). The map now contains `A → {}`, `B → {Bitcoin, Ethereum}`, `C → {Bitcoin, Ethereum}`.
3. `get_supported_foreign_chains()` iterates: Bitcoin is supported by `{B, C}`, Ethereum by `{B, C}`. Neither set is a superset of `{A, B, C}`. Return value: `{}`.
4. Any call to `verify_foreign_transaction(Bitcoin, …)` is now rejected because Bitcoin is not in the supported set.
5. The network's foreign-chain bridge is fully blocked by a single participant acting below the signing threshold. [4](#0-3)

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }
