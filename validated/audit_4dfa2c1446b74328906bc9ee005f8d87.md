Audit Report

## Title
Pusher Consent Signature Replay Allows Creator to Re-establish Revoked Delegation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol::allowPushers`)

## Summary
`allowPushers` verifies a pusher's EIP-191 consent signature but stores no record of used signatures and increments no nonce on revocation. A creator who holds a valid, non-expired consent signature can call `allowPushers` again with the identical `(deadline, sig)` pair immediately after the pusher calls `revokePusher()`, silently re-writing `namespaceRemapping[pusher] = creator` and forcing the pusher's bot to continue feeding the creator's namespace against the pusher's will. The contract's own NatSpec comment acknowledges the deadline as the sole mitigation, but the deadline only bounds the replay window — it does not prevent replay within it.

## Finding Description
`allowPushers` signs over `(chainid, oracle, deadline, pusher, creator)` with no nonce and no used-signature registry:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;   // written every call, no guard
``` [1](#0-0) 

`revokePusher` clears the mapping but does not invalidate any outstanding signature:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [3](#0-2) 

Because `ECDSA.recover` is deterministic and the signed tuple contains no nonce or revocation counter, the same `(deadline, sig)` pair passes verification identically on every replay. The contract's own NatSpec explicitly acknowledges this risk — "an undated signature could re-establish a delegation AFTER the pusher revoked it" — but treats the deadline as the only mitigation: [4](#0-3) 

The deadline is creator-chosen and can be set months or years in the future, making the replay window arbitrarily large.

## Impact Explanation
A pusher who discovers a creator is acting maliciously calls `revokePusher()` to withdraw consent. The creator immediately replays the original consent signature via `allowPushers`, re-writing `namespaceRemapping[pusher] = creator`. The pusher's bot, still running during the operational gap between revocation and shutdown, continues to push data into the creator's namespace. The creator's `IPriceProvider` reads from that namespace and supplies bid/ask quotes to `MetricOmmPool.swap`. The pusher believes they have revoked consent and their data is no longer feeding the creator's pool, but it is — the revocation invariant is broken. This constitutes bad-price execution: oracle data the pusher intended to stop supplying continues to reach live pool swaps, and the creator retains full control over the namespace (including the ability to overwrite slots with manipulated values between the pusher's updates), causing incorrect bid/ask quotes to reach traders.

## Likelihood Explanation
- The creator trivially retains the original consent signature — they received it when they first called `allowPushers`.
- Production deployments commonly use multi-month or multi-year deadlines (the deadline is caller-chosen with no protocol cap).
- The pusher's bot realistically continues running for some time after `revokePusher()` is called (operational gap between on-chain revocation and off-chain bot shutdown).
- No privileged access is required; `allowPushers` is a standard public call.
- The replay is free and can be repeated as many times as needed within the deadline window.

## Recommendation
Add a per-pusher revocation nonce to the signed message and increment it on every successful `revokePusher` or `removePushers` call:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // bind to current nonce
    ))
);

// In revokePusher / removePushers:
pusherNonce[pusher]++;        // invalidate all prior signatures
namespaceRemapping[pusher] = address(0);
```

This ensures that any signature issued before a revocation is permanently invalidated, regardless of whether the deadline has expired.

## Proof of Concept
```
1. Pusher P signs:
   sig = sign(keccak256(abi.encode(chainid, oracle, deadline=now+365days, P, C)))

2. Creator C calls allowPushers(deadline, [P], [sig])
   → namespaceRemapping[P] = C  ✓

3. Pusher P calls revokePusher()
   → namespaceRemapping[P] = address(0)  ✓ (P believes they are free)

4. Creator C calls allowPushers(deadline, [P], [sig])  ← SAME sig, deadline still valid
   → ECDSA.recover produces P  ✓ (no nonce, no used-sig check)
   → namespaceRemapping[P] = C  ← re-established without P's knowledge

5. P's bot continues pushing → all writes land in C's namespace via fallback
   → C's pool price provider reads C's namespace → quotes reach swap()
   → P's revocation has no effect until the original deadline expires
```
