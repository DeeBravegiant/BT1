### Title
Missing Zero-Deposit Guard in `submit_participant_info` Re-Submission Path Permanently Freezes Caller's NEAR Tokens - (File: `crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is marked `#[payable]` and is the single entry point for both **initial** attestation submissions (which require a storage deposit) and **re-submissions** by existing participants (which do not). When the caller is an existing participant performing a re-submission, the deposit-handling block is skipped entirely with no check that `attached_deposit == 0`. Any NEAR tokens attached to such a call are permanently frozen inside the contract.

---

### Finding Description

`submit_participant_info` computes a boolean flag to decide whether to enforce and refund a deposit:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check deposit >= cost, refund excess ...
}
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` — i.e., the caller **is** an existing participant **and** the insertion is an update, not a new entry — the entire block is bypassed. There is no `else` branch asserting `attached_deposit == 0`. Because the function is `#[payable]`, the NEAR runtime accepts any attached value without complaint, and the contract has no code path to return it.

The function signature and `#[payable]` annotation are identical for both the new-submission and re-submission cases: [2](#0-1) 

A node operator who habitually attaches a small deposit (e.g., to cover storage "just in case," mirroring the pattern required on first submission) will silently lose those tokens on every subsequent re-attestation call.

---

### Impact Explanation

Any NEAR tokens attached to a re-submission call by an existing participant are transferred into the contract's balance with no mechanism for recovery. The contract has no withdrawal function for such stranded funds. This is a **permanent, irreversible loss** of the caller's NEAR tokens. The impact matches: *Balance or contract execution-flow manipulation that breaks production safety/accounting invariants.*

---

### Likelihood Explanation

The likelihood is non-negligible for the following reasons:

1. **Same function, two semantically different paths.** `submit_participant_info` is the sole endpoint for both initial and renewal attestation. Operators who learned to attach a deposit for the first call have no obvious signal that re-submissions must carry zero deposit.
2. **`#[payable]` signals deposit acceptance.** The attribute tells callers and tooling that deposits are valid, increasing the probability of accidental attachment.
3. **Periodic re-attestation is a normal operational event.** Nodes must refresh their TEE quotes on a schedule (e.g., after a TEE upgrade or launcher hash change), making this code path exercised repeatedly in production.

---

### Recommendation

Add an `else` branch that asserts `attached_deposit == 0` when the deposit is not consumed, mirroring the fix recommended in the original M-05 report:

```rust
if attestation_storage_must_be_paid_by_caller {
    let storage_used = env::storage_usage().saturating_sub(initial_storage);
    let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
    let attached = env::attached_deposit();
    if attached < cost {
        return Err(/* InsufficientDeposit */);
    }
    if let Some(diff) = attached.checked_sub(cost) && diff > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(diff).detach();
    }
} else {
    // Re-submission by existing participant: no deposit is consumed.
    // Reject non-zero deposits to prevent accidental fund loss.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        return Err(InvalidParameters::UnexpectedDeposit {
            attached: attached.as_yoctonear(),
        }.into());
    }
}
```

Alternatively, refund any attached deposit unconditionally in the `else` branch rather than rejecting the call, to preserve usability.

---

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time with `attached_deposit = 10_000 yoctoNEAR` (covering storage). The deposit check runs; excess is refunded. `alice.near` is now a participant.

2. After a TEE upgrade, `alice.near` calls `submit_participant_info` again to refresh her attestation, attaching `10_000 yoctoNEAR` out of habit.

3. Inside the function:
   - `attestation_insertion_result` is `ParticipantInsertion::UpdatedParticipant` → `is_new_attestation = false`.
   - `self.voter_account()` succeeds (alice is a participant) → `caller_is_not_participant = false`.
   - `attestation_storage_must_be_paid_by_caller = false || false = false`.
   - The `if` block is skipped entirely.

4. The call succeeds. `alice.near` loses `10_000 yoctoNEAR` permanently; the contract balance increases by that amount with no recovery path. [3](#0-2)
