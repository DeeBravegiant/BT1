Audit Report

## Title
L2 Price Providers Lack Sequencer Uptime Check, Enabling Stale-Price Swaps After Restart - (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly designed for L2 deployment (both carry `FUTURE_TOLERANCE` to handle sequencer clock skew) but neither accepts a `sequencerUptimeFeed` parameter nor performs any sequencer liveness check before quoting bid/ask prices. When an L2 sequencer goes offline, oracle prices freeze on-chain. Upon restart, if the frozen price is younger than `MAX_TIME_DELTA`, the staleness check passes and an attacker can immediately execute a swap against the stale pre-outage price, extracting value from LPs.

## Finding Description

Both contracts implement `_isStale` as a pure time-delta check with no sequencer awareness:

```solidity
// PriceProviderL2.sol L135-L150
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) return (refTime - nowTs) > futureTol;
    return (nowTs - refTime) > maxDelta;
}
```

`MAX_TIME_DELTA` is bounded only at `> 0` and `<= 7 days`, meaning a price frozen during a sequencer outage of any duration shorter than `MAX_TIME_DELTA` will pass the staleness check upon restart.

The full `_getBidAndAskPrice()` path in `PriceProviderL2` performs no sequencer liveness check before accepting the oracle price and computing bid/ask. The `_computeBidAsk()` path in `ProtectedPriceProviderL2` is identical in this regard — it checks staleness and price guards but has no sequencer uptime gate.

A `grep` across all `.sol` files in the repository confirms that neither `sequencerUptimeFeed` nor `GRACE_PERIOD` appears anywhere in the production Solidity source. The `PriceProviderFactoryL2.createPriceProvider()` function deploys `PriceProviderL2` without any sequencer feed parameter. By contrast, `ChainlinkVerifierL2` (present in the registry ABI) does carry `sequencerUptimeFeed` and `GRACE_PERIOD`, confirming the system designers are aware of the pattern and applied it selectively.

Exploit flow:
1. `PriceProviderL2` deployed with `MAX_TIME_DELTA = 1 days`.
2. Oracle last updated at T=0 with price = 100.
3. L2 sequencer goes offline at T=0. Oracle price frozen at 100.
4. True market price moves to 110 (+10%) during the 4-hour outage.
5. Sequencer restarts at T=4h. Stored `refTime` is T=0, age = 4h < 1 day → `_isStale` returns `false`.
6. Attacker calls `pool.swap()` before any oracle keeper pushes a fresh price.
7. `_getBidAndAskPrice()` returns bid/ask around 100 (stale).
8. Attacker buys token1 at price 100, immediately sells on a DEX at 110 → ~10% profit.
9. LPs receive token0 worth 100 but gave up token1 worth 110. Pool balances no longer cover fair-value LP claims.

## Impact Explanation

Direct loss of LP principal through bad-price execution. The stale bid/ask (price 100 when market is 110) reaches the pool swap path, allowing the attacker to extract the full price-move delta from LP reserves. Pool token balances no longer cover fair-value LP claims at current market prices, satisfying the "pool insolvency" and "bad-price execution" impact gates. Severity is High given the magnitude of potential loss (bounded only by pool liquidity and price move during outage).

## Likelihood Explanation

- Both contracts are deployed on Arbitrum, Base, Avalanche, BSC, Berachain, and other L2s via `PriceProviderFactoryL2`.
- L2 sequencer outages are documented historical events (Arbitrum ~7-hour outage in 2022; Base has had brief outages).
- The attack requires no special privilege — any EOA can call `swap()` immediately after sequencer restart.
- The vulnerability window is bounded only by oracle keeper response time after restart, which can be minutes.
- `createPriceProvider` is permissionless, so pools using these providers are widely deployable.

## Recommendation

Add a Chainlink sequencer uptime check with a grace period to both `PriceProviderL2._getBidAndAskPrice()` and `ProtectedPriceProviderL2._computeBidAsk()`, mirroring the pattern already present in `ChainlinkVerifierL2`:

```solidity
// Add immutable:
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

// Add to _getBidAndAskPrice() / _computeBidAsk() before the staleness check:
if (address(sequencerUptimeFeed) != address(0)) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer != 0 || block.timestamp - startedAt < GRACE_PERIOD) {
        return (0, type(uint128).max); // fail closed
    }
}
```

Also update `PriceProviderFactoryL2.createPriceProvider()` to accept and forward `_sequencerUptimeFeed`.

```
    // At T=4h: _isStale(T=0, T=4h, 1 day) → false (4h < 1 day)
    // Call pool.swap() → getBidAndAskPrice() returns bid/ask around 100
    // Assert attacker receives token1 at stale price 100 while market is 110
    // Assert LP balances are insufficient to cover fair-value claims

```

1. Deploy `PriceProviderL2` with `MAX_TIME_DELTA = 1 days`, `FUTURE_TOLERANCE = 10`.
2. Set oracle `refTime = block.timestamp`, `mid = 100e8`.
3. `vm.warp(block.timestamp + 4 hours)` without updating oracle (simulates sequencer outage).
4. Call `provider.getBidAndAskPrice()` — returns valid bid/ask around 100 (not stale).
5. Confirm swap executes at price 100 while true market price is 110.
