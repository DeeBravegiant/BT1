Audit Report

## Title
Unbounded withdrawal queue flooding via uncapped `initiateWithdrawal` requests causes temporary freezing of legitimate user funds - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` imposes no per-user cap on pending requests and `minRsEthAmountToWithdraw[asset]` defaults to `0` for every asset until an admin explicitly sets it. An attacker can flood the global FIFO queue with dust requests, forcing the operator to exhaust the entire backlog before `nextLockedNonce` advances to a legitimate user's entry, temporarily freezing that user's already-transferred rsETH.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a plain mapping with no initialization in `initialize()`: [1](#0-0) [2](#0-1) 

The guard in `initiateWithdrawal` is:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
``` [3](#0-2) 

When `minRsEthAmountToWithdraw[asset] == 0`, the second condition `rsETHUnstaked < 0` is vacuously false for `uint256`, so any non-zero amount (including 1 wei) is accepted. The `ExceedAmountToWithdraw` check at line 170 does not prevent flooding because 1-wei rsETH maps to ~1-wei of expected asset, a negligible commitment relative to total available assets.

Each accepted call appends a new entry to the global sequential nonce: [4](#0-3) 

`_unlockWithdrawalRequests` advances `nextLockedNonce` strictly in order — it cannot skip entries: [5](#0-4) 

A legitimate user whose request lands at nonce `N+k` cannot have their request unlocked until `nextLockedNonce` has advanced past all nonces `0..N+k-1`. There is no mechanism to skip or bypass attacker-inserted entries. There is also no per-user cap anywhere in `initiateWithdrawal` or `_addUserWithdrawalRequest`. [6](#0-5) 

## Impact Explanation
The legitimate user's rsETH is transferred into the contract at `initiateWithdrawal` time (line 166) and cannot be recovered until their request is unlocked. With a large attacker-controlled backlog, the operator must issue proportionally many `unlockQueue` batches before `nextLockedNonce` reaches the victim's entry. This constitutes **temporary freezing of funds** (Medium impact).

## Likelihood Explanation
The attack requires no privileged access, no oracle manipulation, and no governance capture. Any rsETH holder can call `initiateWithdrawal` with 1-wei amounts. On Arbitrum, gas costs per call are low, making thousands of dust requests economically feasible. The attacker recovers their rsETH (minus gas) by completing their own withdrawals after the delay. The condition is reachable by any unprivileged external caller.

## Recommendation
1. **Short term:** Set a non-zero `minRsEthAmountToWithdraw` for every supported asset at initialization time (e.g., `0.01 ether` worth of rsETH). Add a per-user cap on simultaneously pending requests (e.g., `require(userAssociatedNonces[asset][msg.sender].length() < MAX_PENDING_PER_USER)` in `_addUserWithdrawalRequest`).
2. **Long term:** Consider a per-asset global queue depth limit or a fee-per-request mechanism to make flooding economically prohibitive.

## Proof of Concept
1. Confirm `minRsEthAmountToWithdraw[stETH] == 0` (never set after deployment).
2. Attacker acquires 10,000 wei of rsETH (e.g., via `RSETHPool` on Arbitrum).
3. Attacker calls `initiateWithdrawal(stETH, 1, "")` 10,000 times, occupying global nonces `0..9999`.
4. Alice calls `initiateWithdrawal(stETH, 1e18, "")`, receiving nonce `10000`.
5. After `withdrawalDelayBlocks` pass, operator calls `unlockQueue(stETH, 500, ...)` repeatedly. Each call advances `nextLockedNonce` by at most 500 entries.
6. After 20 full batches, `nextLockedNonce` reaches `10000` and Alice's request is finally unlocked. Alice's funds were frozen for the entire duration of those operator transactions, a delay the attacker can extend arbitrarily by increasing the flood size.

**Foundry test sketch:**
```solidity
function testQueueFlood() public {
    // attacker mints 10_000 wei rsETH, calls initiateWithdrawal 10_000 times
    for (uint i = 0; i < 10_000; i++) {
        vm.prank(attacker);
        withdrawalManager.initiateWithdrawal(stETH, 1, "");
    }
    // Alice queues a real withdrawal
    vm.prank(alice);
    withdrawalManager.initiateWithdrawal(stETH, 1e18, "");
    // advance blocks past withdrawalDelayBlocks
    vm.roll(block.number + withdrawalDelayBlocks + 1);
    // operator must call unlockQueue 20+ times before Alice's nonce is reached
    for (uint batch = 0; batch < 20; batch++) {
        vm.prank(operator);
        withdrawalManager.unlockQueue(stETH, nextLockedNonce + 500, ...);
    }
    // assert Alice's nonce is still locked after fewer than 20 batches
}
```
