Audit Report

## Title
Shared `dailyMintAmount` Counter Across All Deposit Tokens Allows One Token's Deposits to Block All Others - (File: `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

## Summary
Both `RSETHPoolV3` and `RSETHPoolV3ExternalBridge` maintain a single contract-level `dailyMintAmount` accumulator shared across native ETH and every supported ERC20 token. The `limitDailyMint` modifier, applied to both the ETH and ERC20 `deposit` overloads, reads and writes this shared counter regardless of which token is being deposited. A sufficiently large deposit of any one token exhausts the shared counter, causing every subsequent `deposit()` call for any other token to revert with `DailyMintLimitExceeded` for the remainder of the calendar day.

## Finding Description
`RSETHPoolV3` declares three contract-level scalar state variables:

```solidity
uint256 public dailyMintLimit;   // L51
uint256 public dailyMintAmount;  // L54
uint256 public lastMintDay;      // L57
```

The `limitDailyMint` modifier (L96–125) accumulates all minted rsETH — regardless of the source token — into the single `dailyMintAmount` slot:

```solidity
modifier limitDailyMint(uint256 amount, address token) {
    ...
    if (currentDay > lastMintDay) {
        lastMintDay = currentDay;
        dailyMintAmount = 0;          // single reset for all tokens
    }
    if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
        revert DailyMintLimitExceeded();
    }
    dailyMintAmount += rsETHAmount;   // single accumulator for all tokens
    _;
}
```

Both deposit entry points apply this modifier:

- `deposit(string)` (ETH path, L246–265): `limitDailyMint(msg.value, ETH_IDENTIFIER)`
- `deposit(address, uint256, string)` (ERC20 path, L271–293): `limitDailyMint(amount, token)`

The identical pattern is present in `RSETHPoolV3ExternalBridge` at L66–73 (state variables) and L130–159 (modifier), with both deposit overloads at L366–412.

There is no per-token mapping; the `token` parameter passed to `limitDailyMint` is used only to select the correct price oracle for computing `rsETHAmount`, never to key the counter. Once `dailyMintAmount` reaches `dailyMintLimit` via any token, every subsequent deposit for every other token reverts until the next day's reset.

Existing guards are insufficient: `nonReentrant` prevents reentrancy but does not isolate per-token accounting; `onlySupportedToken` gates which ERC20s are accepted but does not give each token its own quota; `whenNotPaused` is an admin-controlled kill switch unrelated to the accounting flaw.

## Impact Explanation
**Medium — Temporary freezing of funds.**

When `dailyMintAmount` reaches `dailyMintLimit` through deposits of token A (e.g., native ETH), all depositors of token B (e.g., wstETH) are locked out of the pool for up to 24 hours. Their tokens remain in their own wallets but cannot be deposited into the protocol until the next day's counter reset. The freeze is repeatable every day and requires no privileged access.

## Likelihood Explanation
**Medium.** The `deposit()` functions are publicly callable with no role requirement. Any user — including one acting adversarially — who deposits enough of one token to approach `dailyMintLimit` will block all other token depositors. On chains where ETH liquidity is deep and `dailyMintLimit` is set conservatively, a single large transaction is sufficient. The 24-hour reset makes this repeatable daily.

## Recommendation
Replace the three shared scalar variables with per-token mappings keyed on the token address (using `ETH_IDENTIFIER` for native ETH):

```solidity
mapping(address token => uint256) public dailyMintAmount;
mapping(address token => uint256) public lastMintDay;
```

Update `limitDailyMint` to read and write `dailyMintAmount[token]` and `lastMintDay[token]`. Optionally retain a global cap as a secondary aggregate guard. Apply the same fix to both `RSETHPoolV3` and `RSETHPoolV3ExternalBridge`.

## Proof of Concept
1. Deploy `RSETHPoolV3` with `dailyMintLimit = 1000e18` (1 000 rsETH) and both ETH and wstETH deposits enabled.
2. Alice calls `deposit{value: X}("")` where `viewSwapRsETHAmountAndFee(X)` returns ≥ 1 000 rsETH. After the call, `dailyMintAmount == dailyMintLimit`.
3. Bob calls `deposit(wstETH, amount, "")` for any non-zero `amount` in the same calendar day. Inside `limitDailyMint`: `currentDay == lastMintDay` (no reset), so `dailyMintAmount + rsETHAmount > dailyMintLimit` is immediately true → reverts with `DailyMintLimitExceeded`.
4. Bob and all other wstETH/LST depositors are blocked until the next day's reset, despite their deposits being entirely independent of Alice's ETH deposit.

**Foundry test sketch:**
```solidity
function test_sharedDailyMintLimitBlocks() public {
    // Alice exhausts the daily limit with ETH
    vm.deal(alice, largeAmount);
    vm.prank(alice);
    pool.deposit{value: largeAmount}("");

    // Bob's wstETH deposit reverts
    vm.prank(bob);
    vm.expectRevert(RSETHPoolV3.DailyMintLimitExceeded.selector);
    pool.deposit(wstETH, smallAmount, "");
}
```
