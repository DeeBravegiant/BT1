Audit Report

## Title
ETH Withdrawal Permanently Frozen When Recipient Contract Cannot Receive ETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTWithdrawalManager._transferAsset()` pushes native ETH to the withdrawal recipient via `.call{value:}` and unconditionally reverts on failure. If the recipient is a smart contract without a `receive()`/`fallback()` function, `completeWithdrawal` will always revert. Because rsETH is burned from the withdrawal manager in a prior `unlockQueue` transaction, the user's ETH becomes permanently frozen in the contract with no recovery path.

## Finding Description

The withdrawal lifecycle spans two separate transactions:

**Transaction 1 — `unlockQueue`:** rsETH is irreversibly burned from the withdrawal manager and ETH is redeemed into it. [1](#0-0) 

**Transaction 2 — `completeWithdrawal` / `completeWithdrawalForUser`:** Both call `_processWithdrawalCompletion`, which attempts to push ETH to the user via `_transferAsset`. [2](#0-1) 

If `to` is a contract that cannot receive ETH, `.call{value:}` returns `sent = false` and `EthTransferFailed` is thrown, reverting the entire transaction. Because Solidity reverts unwind all state changes in that transaction, the `popFront()`, `delete withdrawalRequests[requestId]`, and `unlockedWithdrawalsCount[asset]--` at lines 705, 712, and 717 are all rolled back. [3](#0-2) 

The result is a permanent loop: the withdrawal request is preserved in the queue, every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` hits the same revert, and the ETH remains locked in the contract. The rsETH burned in the prior `unlockQueue` transaction is not recoverable. The `sweepRemainingAssets` admin function is blocked as long as `unlockedWithdrawalsCount[asset] > 0`, which remains true because the decrement is always rolled back. [4](#0-3) 

## Impact Explanation

**Critical — Permanent freezing of funds.** After `unlockQueue` executes, the user's rsETH is irreversibly burned and the corresponding ETH sits in the withdrawal manager. Neither `completeWithdrawal` nor `completeWithdrawalForUser` can deliver the ETH to a non-ETH-receiving contract. No admin sweep or per-user recovery function exists. The user permanently loses their rsETH value with no recourse.

## Likelihood Explanation

**Medium.** Smart contracts routinely hold and manage rsETH — DeFi vaults, multisigs, aggregators, and proxy contracts are common depositors. Many such contracts intentionally omit a `receive()` function (e.g., pure ERC20 vaults, Gnosis Safe modules, proxy contracts with no ETH handling). No privileged access is required; any such contract that initiates an ETH withdrawal triggers this freeze. The path is reachable by any depositor and requires no attacker — it is a self-inflicted but unrecoverable loss.

## Recommendation

Replace the push-ETH pattern with a **pull-payment** model: store failed ETH deliveries in a per-user claimable mapping and expose a separate `claimEth()` function. Alternatively, wrap ETH to WETH before delivery so the transfer is always an ERC20 `safeTransfer`, which succeeds for any recipient. At minimum, avoid reverting on a failed push:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) {
    pendingEthClaims[to] += amount; // allow user to pull later
}
```

## Proof of Concept

1. Deploy `Victim` — a contract with no `receive()` function that holds rsETH.
2. `Victim` calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH is burned from the withdrawal manager (line 305), ETH is redeemed into it (line 307). This transaction succeeds and is final.
4. `Victim` calls `completeWithdrawal(ETH_TOKEN, "")` → `_processWithdrawalCompletion` → `_transferAsset(ETH_TOKEN, Victim, amount)` → `.call{value:}` returns `false` → `revert EthTransferFailed()`. Entire transaction reverts; withdrawal request state is preserved.
5. Step 4 reverts on every retry. rsETH is permanently gone. ETH is permanently locked in the withdrawal manager.

A Foundry test can confirm this by deploying a no-`receive()` contract, running the full withdrawal lifecycle on a fork, and asserting that `completeWithdrawal` always reverts while the withdrawal manager's ETH balance remains unchanged.
