### Title
Missing minimum-length guard on custom KEM shared secret allows trivially weak AES-GCM key derivation — (`src/cbmpc/c_api/pve_base_pke.cpp`)

### Summary
`kem_encap_shim` accepts any non-negative `kem_ss` size from a caller-supplied `cbmpc_pve_kem_encap_fn` callback without enforcing a minimum length. A malicious callback returning a 1-byte (or 0-byte) shared secret causes `kem_aead_ciphertext_t::seal` to derive the AES-GCM-256 key via `HKDF(IKM=1 byte)`, bounding the key's entropy to ≤8 bits. The resulting ciphertext is trivially decryptable by exhaustive search over 256 (or 1) possible keys.

### Finding Description
The call chain `cbmpc_pve_encrypt_with_kem` → `c_base_kem_adapter_t::encrypt` → `kem_policy_runtime_kem_t::encapsulate` → `kem_encap_shim` → user callback → `kem_aead_ciphertext_t::seal` contains no guard on the size of `kem_ss` returned by the callback. The shim only rejects `kem_ss.size < 0` or `(kem_ss.size > 0 && !kem_ss.data)`. A callback returning `{size=1, data=<ptr>}` or `{size=0, data=nullptr}` passes both checks. The resulting `buf_t` is passed directly as the IKM to `hkdf_extract_sha256`, which produces a 32-byte PRK whose entropy is bounded by the IKM entropy (≤8 bits for 1-byte IKM, 0 bits for empty IKM). The 32-byte AES-GCM key derived from this PRK is therefore trivially brute-forceable.

This is in direct contrast to the two HSM-backed KEM paths in the same file:
- `kem_policy_rsa_oaep_hsm_t::decapsulate` enforces `kem_ss.size() == 32`
- `kem_policy_ecdh_p256_hsm_t::decapsulate` enforces `dh.size() == 32`

The custom KEM path has no equivalent enforcement, and the public API documentation (`cbmpc_pve_kem_encap_fn`) does not document any minimum size requirement, so callers have no notice of this constraint.

### Impact Explanation
In a system where the KEM callback is provided by a separate component (e.g., an HSM service or plugin) that is distinct from the component generating the plaintext scalar `x`: a malicious callback provider returns a known 1-byte `kem_ss`. The ciphertext is stored. The callback provider (or any party they share the 1-byte value with) can later brute-force the AES-GCM key in ≤256 decryption attempts to recover the private scalar `x` without possessing `dk`. `cbmpc_pve_verify_with_kem` does not catch this because it re-invokes the same malicious callback deterministically, producing a consistent (but weak) ciphertext.

### Likelihood Explanation
Exploitability requires the callback to be provided by a component that is separate from and adversarial to the component supplying `x`. This is a realistic architecture for HSM-backed or plugin-based KEM integrations. The missing guard is a single missing bounds check, and the attack requires no cryptographic sophistication — just returning a short buffer.

### Recommendation
In `kem_encap_shim`, after the callback returns successfully, add a minimum size check on `kem_ss` before accepting it:

```cpp
// Enforce a minimum shared-secret size (e.g., 16 bytes for 128-bit security).
constexpr int KEM_SS_MIN_SIZE = 16;
if (kem_ss.size < KEM_SS_MIN_SIZE) {
    if (kem_ct.data) cbmpc_cmem_free(kem_ct);
    if (kem_ss.data) cbmpc_cmem_free(kem_ss);
    return E_FORMAT;
}
```

Also update the `cbmpc_pve_kem_encap_fn` documentation to state the minimum `out_kem_ss` size requirement explicitly.

  cmem_t kem_ss{nullptr, 0};
