Audit Report

## Title
Unvalidated `owner` in `addLiquidity` Permanently Locks Deposited Principal — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no zero-address guard or capability check at the pool level. Because `removeLiquidity` enforces `msg.sender == owner` as a hard gate, any position minted under `address(0)` or under a contract incapable of calling `removeLiquidity` results in permanently irrecoverable LP shares and the underlying deposited tokens. The periphery's zero-address guard is bypassable by calling the pool directly.

## Finding Description
`MetricOmmPool.addLiquidity` (L182–196) accepts `owner` as a raw caller-supplied parameter and passes it directly to `LiquidityLib.addLiquidity`, which writes shares to `_positionBinShares[keccak256(owner, salt, binIdx)]` and pulls tokens from `msg.sender` via the `IMetricOmmModifyLiquidityCallback` at `LiquidityLib.sol` L147–148. No validation of `owner` is performed at the pool level.

`MetricOmmPool.removeLiquidity` (L199–212) enforces `if (msg.sender != owner) revert NotPositionOwner()` at L206. This is the sole ownership gate and it is absolute — no delegation, no approval mechanism, no admin override exists.

**Exploit path 1 — `address(0)` via direct pool call:**
Any EOA or contract that implements `IMetricOmmModifyLiquidityCallback` can call `pool.addLiquidity(address(0), salt, deltas, callbackData, "")` directly, bypassing the periphery entirely. Tokens are pulled from `msg.sender` and shares are recorded under `address(0)`. Since `msg.sender` can never equal `address(0)` in the EVM, `removeLiquidity(address(0), ...)` always reverts with `NotPositionOwner`. Funds are permanently locked.

**Exploit path 2 — non-zero incapable contract via periphery:**
`MetricOmmPoolLiquidityAdder._validateOwner` (L247–249) only rejects `address(0)`. Passing a non-zero contract address that has no `removeLiquidity` call path (e.g., a multisig, vault, or yield optimizer) passes this check. The periphery calls `pool.addLiquidity(owner, ...)`, tokens are pulled from `msg.sender`, and the position is locked under the incapable contract.

The existing guard is insufficient because it lives only in the periphery and only covers the zero-address case, while the pool itself is a public, permissionless contract.

## Impact Explanation
Direct, irreversible loss of user principal. Tokens transferred from `msg.sender` during the `metricOmmModifyLiquidityCallback` settlement are permanently locked in the pool. The pool's `binTotals` and `_binStates` correctly account for the tokens, but no address can ever satisfy `msg.sender == owner` to call `removeLiquidity`. This constitutes broken core LP withdrawal functionality and direct loss of deposited assets, meeting the Critical/High threshold under the allowed impact gate.

## Likelihood Explanation
Exploit path 1 requires only that the attacker (or mistaken integrator) implement `IMetricOmmModifyLiquidityCallback` and call the pool directly — no privileges required, repeatable by any address. Exploit path 2 is reachable by any caller of the periphery who supplies a non-zero contract `owner` that cannot interact with the pool, a realistic scenario for protocol integrators (yield optimizers, vaults, airdrop distributors) using the documented operator pattern. Both paths are unprivileged and require no special setup.

## Recommendation
1. Add a zero-address guard directly inside `MetricOmmPool.addLiquidity` before any state changes:
   ```solidity
   require(owner != address(0), InvalidOwner());
   ```
2. Consider splitting `addLiquidity` into a self-owned variant (forces `owner = msg.sender`) and an explicit operator variant (`addLiquidityFor(address owner, ...)`), so the dangerous operator path is opt-in and clearly documented.
3. Document that callers using the operator path are solely responsible for ensuring `owner` can invoke `removeLiquidity`, and consider an on-chain ERC-165 check or allowlist for contract receivers.

## Proof of Concept
```solidity
// Path 1: direct pool call with address(0) — bypasses periphery guard entirely
pool.addLiquidity(
    address(0),   // owner: no msg.sender can ever equal address(0)
    salt,
    deltas,
    callbackData, // attacker contract implements IMetricOmmModifyLiquidityCallback
    ""
);
// Tokens pulled from attacker via callback, locked forever.
// pool.removeLiquidity(address(0), ...) always reverts NotPositionOwner.

// Path 2: periphery with non-zero incapable contract owner
liquidityAdder.addLiquidityExactShares(
    pool,
    address(vaultWithNoPoolInteraction), // non-zero, passes _validateOwner
    salt,
    deltas,
    maxAmount0,
    maxAmount1,
    ""
);
// Tokens pulled from msg.sender. Position locked under vaultWithNoPoolInteraction.
// vaultWithNoPoolInteraction cannot call pool.removeLiquidity — funds irrecoverable.
```
