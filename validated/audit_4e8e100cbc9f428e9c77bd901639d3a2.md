Audit Report

## Title
ETH Deposit Limit Check Omits Incoming `amount`, Allowing Limit Bypass - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric condition: the ETH branch checks only whether the existing total already exceeds the limit (`totalAssetDeposits > limit`), while the ERC20 branch correctly adds the incoming deposit (`totalAssetDeposits + amount > limit`). Any unprivileged caller can invoke `depositETH` with a `msg.value` large enough to push the running total past the configured cap, and the guard will not fire.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, `_checkIfDepositAmountExceedesCurrentLimit` branches on `asset == LRTConstants.ETH_TOKEN`:

```solidity
// line 679 — ETH branch: amount is never added
return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));

// line 681 — ERC20 branch: correct
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
``` [1](#0-0) 

The ETH branch evaluates `false` whenever `totalAssetDeposits <= limit`, regardless of how large `amount` is. The function is the sole pre-deposit guard, called from `_beforeDeposit` at line 661, which is itself called by the public `depositETH` at line 87 with `msg.value` as the amount. [2](#0-1) [3](#0-2) 

`getAssetCurrentLimit` (lines 402–408) confirms the limit is intended as a hard ceiling: it returns `depositLimit - totalAssetDeposits` and 0 when already exceeded, making the asymmetric ETH check a clear deviation from the protocol's own invariant. [4](#0-3) 

No other guard in `depositETH` or `_beforeDeposit` compensates for the missing `amount` term. [5](#0-4) 

## Impact Explanation
The `depositLimitByAsset` cap is a risk-management control. When bypassed for ETH, the protocol accepts and stakes more ETH in EigenLayer than the configured ceiling allows, violating the protocol's own invariant. This maps to **Low — Contract fails to deliver promised returns, but doesn't lose value**: the deposit cap guarantee is broken, but no direct theft or permanent freeze of funds occurs.

## Likelihood Explanation
No special role or privilege is required. Any ordinary depositor calling `depositETH{value: X}()` where `totalAssetDeposits + X > depositLimit` but `totalAssetDeposits <= depositLimit` will silently bypass the cap. The condition is triggered on every such deposit and is fully repeatable.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC20 branch:

```diff
- return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+ return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
``` [6](#0-5) 

## Proof of Concept
```
State:
  depositLimitByAsset(ETH) = 100 ETH
  totalAssetDeposits(ETH)  =  95 ETH

Attacker calls: depositETH{value: 10 ether}(0, "")

ETH branch evaluates: 95 > 100  →  false  →  no revert
New total: 105 ETH  →  exceeds limit by 5 ETH
rsETH minted and ETH staked in EigenLayer

ERC20 branch (same numbers): 95 + 10 > 100  →  true  →  MaximumDepositLimitReached
```

Foundry test plan:
1. Deploy `LRTDepositPool` with a mock `LRTConfig` setting `depositLimitByAsset(ETH_TOKEN) = 100 ether`.
2. Seed `getTotalAssetDeposits(ETH_TOKEN)` to return 95 ether (e.g., via a prior deposit).
3. Call `depositETH{value: 10 ether}(0, "")` from an unprivileged address.
4. Assert the call succeeds (no revert) and that `getTotalAssetDeposits(ETH_TOKEN)` returns 105 ether, exceeding the 100 ether limit.
5. Confirm the equivalent ERC20 path reverts with `MaximumDepositLimitReached` under identical numeric conditions.
