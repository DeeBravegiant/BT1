Audit Report

## Title
ETH Deposit Limit Not Enforced at Exact Boundary Due to Strict Inequality - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses a strict `>` comparison for the ETH branch, meaning when `totalAssetDeposits == depositLimitByAsset(ETH_TOKEN)` the function returns `false` and the deposit is not blocked. This allows ETH deposits to exceed the configured cap at the exact boundary, while the public view `getAssetCurrentLimit` already reports zero remaining capacity at that same state.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)`. When `totalAssetDeposits` equals the limit exactly, this expression is `false`, so `_beforeDeposit` (L661–663) does not revert and the deposit proceeds. The ERC-20 branch correctly includes the incoming `amount` in the comparison (`totalAssetDeposits + amount > limit`), blocking any non-zero deposit once the limit is reached. The ETH branch has no equivalent inclusion of `amount`, and the off-by-one in the inequality means the boundary state is not treated as exceeded. `getAssetCurrentLimit` (L402–409) uses the same `>` but returns `depositLimit - totalAssetDeposits = 0` at equality, so the public view correctly signals zero capacity while the enforcement gate does not enforce it.

## Impact Explanation
An unprivileged depositor can call `depositETH` and mint rsETH beyond the protocol's configured ETH deposit ceiling. No funds are lost or stolen, but the protocol fails to enforce its stated deposit constraint. This matches **Low — contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The boundary condition is reachable in two ways: (a) an admin sets `depositLimitByAsset(ETH_TOKEN)` to exactly the current TVL to freeze new deposits — immediately exploitable by any depositor; or (b) a sequence of deposits naturally lands the cumulative total at the limit. Both paths are realistic. Likelihood is **Low** but non-negligible, particularly in scenario (a) which is a common operational pattern for pausing inflows.

## Recommendation
Change the strict inequality to non-strict in the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit` (L679):

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits >= lrtConfig.depositLimitByAsset(asset));
}
```

This aligns the enforcement gate with the semantics of `getAssetCurrentLimit`, which already treats `totalAssetDeposits == depositLimit` as zero remaining capacity.

## Proof of Concept
1. Admin calls `setDepositLimitByAsset(ETH_TOKEN, X)` where `X` equals the current `getTotalAssetDeposits(ETH_TOKEN)`.
2. `getAssetCurrentLimit(ETH_TOKEN)` returns `0` — cap is reported as full.
3. Attacker calls `depositETH{value: 1 ether}(0, referralId)`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether)` evaluates `X > X` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the attacker; total ETH deposits become `X + 1 ether`, exceeding the configured limit.
