### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Submitted `request` — Cross-Request Replay Enables Forged Foreign-Chain Verification - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that the MPC signature in the response is cryptographically valid over `response.payload_hash` using the root public key, but it never checks that `response.payload_hash` is the correct hash for the supplied `request`. A single Byzantine participant (below threshold) can replay a legitimately produced response from one pending foreign-transaction verification request onto a completely different pending request, causing the contract to resolve that second request with fabricated verification data.

### Finding Description

The `respond_verify_foreign_tx` entry point in `crates/contract/src/lib.rs` performs the following checks before resolving the pending yield queue: [1](#0-0) 

1. The caller is an attested participant and the protocol is active.
2. The signature in `response` is valid over `response.payload_hash` using the root public key for `request.domain_id`.
3. A pending entry for `request` exists in `pending_verify_foreign_tx_requests`.

What is **absent** is any check that `response.payload_hash` equals the canonical hash of `(request, extracted_values, observed_at)` for the supplied `request`. The contract treats `payload_hash` as an opaque blob whose only constraint is that the MPC signature over it is valid.

Compare this with the regular `respond` path for ECDSA signatures: there the signature is verified against a **derived** public key that is computed from the specific request's derivation path and predecessor account, so a signature produced for request A is cryptographically invalid under the derived key for request B. No such binding exists for `respond_verify_foreign_tx`; all foreign-tx domains share a single root key, and the payload hash is not reconstructed from the request inside the contract.

The `ForeignTxSignPayload` that MPC nodes are supposed to hash is:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

The contract never receives `values` and therefore cannot reconstruct or verify the hash. [2](#0-1) 

The pending-request resolution helper simply drains the queue for whatever `request` key is passed in: [3](#0-2) 

### Impact Explanation

A single malicious participant (strictly below the signing threshold) can:

1. Submit `verify_foreign_transaction(request_B)` as a user for a fraudulent or non-existent foreign-chain transaction. Honest MPC nodes will fail to verify it and will never produce a response.
2. Observe any legitimately produced `respond_verify_foreign_tx(request_A, response_A)` call on-chain (all NEAR transactions are public).
3. Immediately submit `respond_verify_foreign_tx(request_B, response_A)` — the contract accepts it because `sig_A` is a valid MPC signature over `payload_hash_A` under the root key.
4. The pending yield for `request_B` is resolved with `response_A = {payload_hash_A, sig_A}`.

The caller of `verify_foreign_transaction(request_B)` — e.g., a bridge contract — receives a `VerifyForeignTransactionResponse` containing a valid MPC signature. Because the response does not include the `extracted_values` or `observed_at` that were used to produce `payload_hash_A`, the receiving contract has no way to detect that the hash corresponds to a different transaction. If the bridge contract checks only that the signature is valid over the returned `payload_hash` (the natural check), it will accept the response as proof that the fraudulent transaction was verified, enabling an invalid inbound bridge execution or double-spend.

This matches the **High** allowed impact: *"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

- The attacker must be an attested participant — a meaningful barrier, but the threat model explicitly includes Byzantine participants below threshold.
- The attacker needs a pending `request_B` in the queue — trivially achieved by calling `verify_foreign_transaction` themselves as a user (1 yoctoNEAR deposit).
- The attacker needs a valid `response_A` — all `respond_verify_foreign_tx` calls are public NEAR transactions; the attacker simply monitors the chain.
- No cryptographic capability beyond reading public chain state is required.
- The race window is generous: the attacker only needs to submit before the yield for `request_B` times out (the yield timeout is configurable but typically many blocks).

### Recommendation

The contract must verify that `response.payload_hash` is the canonical hash of the supplied `request` before resolving the pending queue. Two complementary approaches:

1. **Include `extracted_values` in the response**: Change `VerifyForeignTransactionResponse` to carry the `Vec<ExtractedValue>` alongside the hash. The contract recomputes `payload_hash = H(request, values)` and asserts equality with `response.payload_hash` before calling `resolve_yields_for`.

2. **Commit the request hash inside the payload**: Define `payload_hash = H(H(request) || H(values))`. The contract can verify the first component independently: assert that `payload_hash` is consistent with `H(request)` by requiring the responder to supply `H(values)` separately, then checking `payload_hash == H(H(request) || h_values)`.

Either approach closes the gap by making the signature cryptographically bound to the specific `request` key used to look up the pending queue, mirroring the binding that already exists for regular ECDSA `respond` via derived keys.

### Proof of Concept

```
Setup:
  - Alice submits verify_foreign_transaction(request_A) for a real Bitcoin tx.
  - Eve (malicious participant) submits verify_foreign_transaction(request_B)

Step 1: Honest MPC nodes verify tx_A, produce payload_hash_A = H(request_A, values_A),
        sign it, and submit respond_verify_foreign_tx(request_A, {payload_hash_A, sig_A}).
        Alice's contract receives the correct response.

Step 2: Eve observes {payload_hash_A, sig_A} in the NEAR transaction log.

Step 3: Eve (as a participant) submits:
          respond_verify_foreign_tx(request_B, {payload_hash_A, sig_A})

Step 4: Contract check — verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK.
        No check that payload_hash_A == H(request_B, ...).

        It verifies verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK.
        It cannot reconstruct payload_hash_B (no extracted_values available).
        It accepts the response as proof that the fraudulent tx_B was verified.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
