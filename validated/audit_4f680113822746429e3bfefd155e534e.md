Audit Report

## Title
Unrestricted `sendETHToL1ViaBridge` Enables Block Stuffing — (`contracts/bridges/UnichainMessenger.sol`)

## Summary
`UnichainMessenger.sendETHToL1ViaBridge` is a public function with no access control and no minimum value threshold. Any external caller can invoke it with 1 wei repeatedly, consuming L2 block gas and temporarily delaying the `BRIDGER_ROLE`'s `bridgeAssetsViaNativeBridge` transaction. The same pattern exists across `BaseMessenger`, `ArbitrumMessenger`, and `ScrollMessenger`.

## Finding Description
`UnichainMessenger.sendETHToL1ViaBridge` enforces only `msg.value == value` and `nonReentrant`: [1](#0-0) 

There is no `onlyRole`, no minimum ETH threshold, and no per-sender rate limit. The function directly invokes `IUnichainMessenger(l2bridge).bridgeETHTo` with `DEFAULT_GAS_LIMIT = 200_000`: [2](#0-1) 

By contrast, `LineaMessenger` enforces a meaningful minimum via `minimumFeeInWei`: [3](#0-2) 

The legitimate caller, `RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge`, is correctly gated behind `onlyRole(BRIDGER_ROLE)`: [4](#0-3) 

An attacker bypasses the pool entirely and calls the messenger directly. The same unguarded pattern is present in `BaseMessenger` and `ArbitrumMessenger`: [5](#0-4) [6](#0-5) 

## Impact Explanation
**Low — Block stuffing.** Each attacker call with 1 wei triggers a bridge operation consuming ~50k–100k L2 gas. On Unichain (OP Stack, ~30M block gas limit), an attacker can fill a block with ~300–600 such calls. During the stuffing window, the `BRIDGER_ROLE`'s `bridgeAssetsViaNativeBridge` transaction is excluded from the block, temporarily preventing ETH held in `RSETHPoolNoWrapper` from being bridged to `L1Vault`. The delay is temporary and repeatable, matching the "Low. Block stuffing." allowed impact class.

## Likelihood Explanation
- The function is fully public with no role check, reachable by any external address.
- Cost per call is 1 wei + L2 gas; on Unichain L2, gas fees are fractions of a cent per transaction.
- No protocol-level defense exists in the messenger contract.
- The pattern is replicated across `BaseMessenger`, `ArbitrumMessenger`, and `ScrollMessenger`, broadening the attack surface to all supported L2 chains.
- No privileged access, leaked keys, or victim mistakes are required.

## Recommendation
Add an access control modifier (e.g., `onlyRole(BRIDGER_ROLE)`) to `sendETHToL1ViaBridge` in all messenger contracts (`UnichainMessenger`, `BaseMessenger`, `ArbitrumMessenger`, `ScrollMessenger`). Alternatively, enforce a meaningful minimum ETH threshold (analogous to `LineaMessenger`'s `minimumFeeInWei` check) to make spam economically infeasible. The `LineaMessenger` pattern is the reference implementation already present in the codebase.
