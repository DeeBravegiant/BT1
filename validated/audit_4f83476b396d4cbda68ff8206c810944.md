### Title
Dev OS Measurements Compiled Into Production Builds Allow TEE Security Bypass - (File: `crates/mpc-attestation/src/attestation.rs`)

### Summary

The `default_measurements()` function permanently compiles both production and **dev** OS image measurements into the MPC node binary and seeds them into the on-chain contract during migration. A node operator can deploy a dev OS image (which exposes SSH access and debug capabilities) and still pass TEE attestation, bypassing the TEE security boundary that is supposed to prevent operators from accessing key shares.

### Finding Description

`default_measurements()` in `crates/mpc-attestation/src/attestation.rs` returns a static array of two `ExpectedMeasurements` entries:

```rust
pub fn default_measurements() -> &'static [ExpectedMeasurements] {
    static MEASUREMENTS: [ExpectedMeasurements; 2] = [
        include_measurements!("assets/tcb_info.json"),
        // TODO(#1433): Security - remove dev measurements from production builds after testing is complete
        include_measurements!("assets/tcb_info_dev.json"),
    ];
    &MEASUREMENTS
}
``` [1](#0-0) 

These measurements are seeded into the contract's `AllowedMeasurements` list on every migration. The dev measurements (`tcb_info_dev.json`) correspond to a `dstack-dev-*` OS image, which — as confirmed by the deployment env files — enables SSH access (`EXTERNAL_SSH_PORT`) and other debug capabilities not present in the production image. [2](#0-1) [3](#0-2) 

The `tcb_info_dev.json` and `tcb_info.json` files have distinct RTMR values (different `rtmr2`, `compose_hash`, etc.), confirming they represent different OS environments. The dev measurements are compiled in via the `include_measurements!` proc macro at build time and are indistinguishable from production measurements during on-chain attestation verification. [4](#0-3) 

The contract's `verify_measurements` path accepts any measurement in the allowed list: [5](#0-4) 

The `README` for test assets explicitly acknowledges the risk: *"If they are stale, nodes running a newer OS image will fail attestation until operators vote in the correct measurements"* and notes this is a bootstrap mechanism that was supposed to be removed after release 3.8. [6](#0-5) 

### Impact Explanation

The TEE security model's core guarantee is that **even the node operator cannot access key shares** — the OS image is measured and the sealing key is derived from those measurements. By allowing dev OS images to pass attestation, an operator can:

1. Deploy a `dstack-dev-*` image with SSH access enabled.
2. Pass `submit_participant_info` attestation verification (dev measurements are in the allowed list).
3. Be accepted as a valid MPC participant with a stored `VerifiedAttestation`.
4. SSH into the running CVM and extract their key share from memory or the encrypted RocksDB store.

This constitutes **unauthorized access to MPC key shares** — the TEE boundary that is supposed to make key shares inaccessible to operators is bypassed. A single compromised key share, combined with social engineering or collusion with one additional participant below the threshold, can enable partial key recovery or targeted signing attacks.

**Impact: Critical** — Bypass of TEE attestation security guarantees enabling unauthorized access to MPC key share material.

### Likelihood Explanation

The dev measurements are compiled into every production binary and seeded on every migration. Any operator who knows the dev image is accepted (visible from the open-source codebase) can deliberately deploy a dev image. The TODO comment (`#1433`) has been present since at least release 3.0.4 (CHANGELOG entry for PR #1428), indicating this has not been remediated across multiple production releases. [7](#0-6) 

### Recommendation

- **Short term:** Remove `include_measurements!("assets/tcb_info_dev.json")` from `default_measurements()` in production builds. Use a compile-time feature flag (e.g., `#[cfg(feature = "dev")]`) to gate dev measurements out of release binaries. Immediately vote to remove the dev measurements from the on-chain `AllowedMeasurements` list on all deployed contracts.
- **Long term:** Resolve TODO `#1433`. Enforce that the `default_measurements()` function used during contract migration only ever contains production OS measurements. Add a CI check that fails if dev measurements are present in a release build.

### Proof of Concept

1. Observe that `default_measurements()` includes `tcb_info_dev.json` unconditionally.
2. On contract migration, `default_measurements()` is used to seed `AllowedMeasurements` — both prod and dev entries are added to the on-chain allowed list.
3. An operator deploys a CVM using `OS_IMAGE=dstack-dev-0.5.8` (as shown in `deployment/cvm-deployment/configs/kms.env` line 44).
4. The dev image's RTMR measurements match `tcb_info_dev.json`; `submit_participant_info` passes `verify_measurements` because the dev entry is in the allowed list.
5. The operator SSHes into the running dev CVM (SSH port is exposed), accesses the MPC node process, and extracts the key share from the encrypted RocksDB or from memory. [8](#0-7) [9](#0-8)

#[proc_macro]
pub fn include_measurements(input: TokenStream) -> TokenStream {
    let input = parse_macro_input!(input as LitStr);
    let file_path = input.value();

    try_include_measurements(&file_path).unwrap_or_compile_error(input.span())
}

fn try_include_measurements(file_path: &str) -> anyhow::Result<proc_macro2::TokenStream> {
    let manifest_dir = env::var("CARGO_MANIFEST_DIR")
        .context("CARGO_MANIFEST_DIR environment variable not set")?;
    let full_path = Path::new(&manifest_dir).join(file_path);

    let json_content = fs::read_to_string(&full_path)
        .with_context(|| format!("Failed to read file '{}'", full_path.display()))?;

    generate_measurements_tokens(&json_content).with_context(|| {
        format!(
            "Failed to generate measurements from file '{}'",
            full_path.display()
        )
    })
}

impl AllowedMeasurements {
    /// Adds a new measurement set to the allowed list.
    /// Returns `false` if the measurement is already in the list.
    pub fn add(&mut self, measurement: ContractExpectedMeasurements) -> bool {
        if self.entries.contains(&measurement) {
            log!("measurement already in allowed list");
            return false;
        }
        self.entries.push(measurement);
        true
    }
