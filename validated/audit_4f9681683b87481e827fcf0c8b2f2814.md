### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root MPC public key, but never checks that `response.payload_hash` was actually derived from the `request` argument. A single attested participant (below threshold) can replay any previously produced `(payload_hash, signature)` pair — publicly visible on-chain — against any currently pending `VerifyForeignTransactionRequest`, causing the contract to resolve that request with a forged verification result.

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs two independent operations that are never cryptographically bound to each other:

1. It verifies `response.signature` over `response.payload_hash` against the root public key.
2. It resolves the pending queue entry keyed by `request`.

The `request` parameter is used only as a map lookup key. The `response.payload_hash` — which encodes the actual foreign-chain transaction data that was supposedly verified — is never checked to equal `SHA256(borsh(ForeignTxSignPayloadV1 { request, values }))`. The contract accepts any `payload_hash` for which a valid root-key signature exists, regardless of whether it corresponds to the pending `request`.

```
// crates/contract/src/lib.rs lines 726-752
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← attacker-controlled; never verified against `request`
    &secp_pk,
)
.is_ok()
...
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,                              // ← used only as map key
    serde_json::to_vec(&response).unwrap(), // ← contains the unbound payload_hash
)
```

By contrast, the SDK-side verifier (`crates/near-mpc-sdk/src/foreign_chain.rs`) correctly recomputes the expected hash from the request and checks `expected_payload_hash == response.payload_hash`. This check is absent from the on-chain contract.

The `ForeignTxSignPayload` structure binds the hash to a specific `(request, extracted_values)` pair:

```
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs lines 1499-1509
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> { ... }
}
```

The contract never calls `compute_msg_hash` on the incoming `request` to compare against `response.payload_hash`.

### Impact Explanation

A single Byzantine attested participant (below signing threshold) can:

1. Observe any previously produced `(payload_hash_B, signature_B)` pair from a past legitimate `respond_verify_foreign_tx` call — these are publicly visible on-chain.
2. Wait for a new pending `VerifyForeignTransactionRequest` (`request_A`) to appear in the contract.
3. Call `respond_verify_foreign_tx(request_A, { payload_hash: payload_hash_B, signature: signature_B })`.
4. The contract accepts the call (signature is valid for `payload_hash_B` under the root key), drains the entire fan-out queue for `request_A`, and delivers `{ payload_hash_B, signature_B }` to every waiting caller.

Every caller waiting on `request_A` receives a response that attests to a **different** foreign-chain transaction (`request_B`'s data). Bridge contracts that consume this response without re-running the SDK-level `ForeignChainSignatureVerifier` check will treat the forged attestation as proof that `request_A`'s transaction was confirmed, enabling invalid bridge execution (e.g., releasing funds for a transaction that was never finalized, or double-spending a previously processed transaction). Even callers that do re-verify are permanently denied a legitimate response because the pending queue entry is irrevocably drained.

This matches: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

- The attacker is a single attested participant, a realistic Byzantine adversary model explicitly in scope.
- No threshold collusion is required; one participant suffices.
- The replay material (`payload_hash`, `signature`) is publicly available on-chain from any prior `respond_verify_foreign_tx` transaction.
- The attack requires no special timing beyond the existence of a pending request, which is a normal operational state.
- Bridge contracts that rely on `verify_foreign_transaction` for inbound fund releases are the primary deployment target, making exploitation directly profitable.

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected payload hash from the `request` argument and assert it equals `response.payload_hash` before accepting the response:

```rust
// Derive the expected hash from the on-chain request.
// Note: the contract does not store `extracted_values`, so the node must
// include them in the response, and the contract must verify the binding.
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // add this field to the response
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, store the expected `payload_hash` in the pending-request map at submission time (if the contract can compute it deterministically without the extracted values) or require the response to include the extracted values so the contract can verify the full binding. The SDK's `ForeignChainSignatureVerifier::verify_signature` already implements the correct check and should serve as the reference.

### Proof of Concept

1. The MPC network legitimately processes `request_B` (e.g., Bitcoin tx `[0xBB; 32]`) and a participant calls `respond_verify_foreign_tx(request_B, { payload_hash_B, sig_B })`. This transaction is recorded on-chain.

2. User Alice submits `verify_foreign_transaction(request_A)` (e.g., Bitcoin tx `[0xAA; 32]`), creating a pending entry in `pending_verify_foreign_tx_requests`.

3. The malicious participant reads `payload_hash_B` and `sig_B` from the on-chain history of step 1.

4. The malicious participant calls:
   ```
   respond_verify_foreign_tx(
       request = request_A,          // pending request for tx 0xAA
       response = {
           payload_hash: payload_hash_B,  // hash of tx 0xBB's data
           signature: sig_B,              // valid root-key sig over payload_hash_B
       }
   )
   ```

5. The contract at line 729 verifies `sig_B` over `payload_hash_B` — valid. At line 749 it drains the queue for `request_A` and delivers `{ payload_hash_B, sig_B }` to Alice.

6. Alice's bridge contract receives a response claiming the MPC network attested to `payload_hash_B` (tx `0xBB`'s data) for her request about tx `0xAA`. If Alice's contract does not re-run the SDK verifier, it releases funds based on a forged attestation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
