### Title
NFT Royalty Bypass via Integer Division Truncation to Zero - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py` uses Python integer floor division (`//`) to calculate NFT royalty payments. When the offered fungible amount is small enough, the division truncates to zero, causing the wallet to skip the royalty payment coin entirely. An unprivileged taker can deliberately structure an offer with a sufficiently small fungible amount to transfer a royalty-bearing NFT while paying zero royalties to the creator.

### Finding Description
`compute_royalty_amount` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

`MAX_ROYALTY_BASIS_POINTS` is 10000. For any offer where `abs(offered_amount) // royalty_split * percentage < 10000`, the result is zero. The codebase explicitly tests and accepts this: [2](#0-1) 

When `compute_royalty_amount` returns zero, `make_nft1_offer` explicitly skips creating the royalty payment coin: [3](#0-2) 

Additionally, the `trade_prices` list passed to the NFT spend (which the transfer program puzzle uses to enforce royalties) is filtered to exclude entries where the royalty would be zero: [4](#0-3) 

This means both the royalty payment coin and the on-chain trade price enforcement are omitted when the amount truncates to zero.

### Impact Explanation
An unprivileged attacker (offer maker or taker) can create or respond to an offer with a fungible amount small enough to produce a zero royalty. For a royalty of `P` basis points, any offer below `10000 // P` mojos results in zero royalty. For CATs (1000 mojos per unit), a 1% royalty (100 basis points) is bypassed for any offer below 100 CAT mojos (0.1 CAT). The NFT creator's royalty address receives nothing. This is an unauthorized accounting change affecting NFT offer settlement — the royalty-bearing NFT is transferred with the royalty obligation silently voided. [5](#0-4) 

### Likelihood Explanation
CATs have 1000 mojos per unit, making small-amount offers practical. For a 1% royalty, any CAT offer under 0.1 CAT triggers zero royalty. A seller wishing to avoid paying royalties to the NFT creator can deliberately price the NFT at a dust CAT amount, transfer it, and then separately compensate the buyer off-chain. The `royalty_calculation` static method used for display also has the same truncation: [6](#0-5) 

This means even the royalty preview shown to users before accepting an offer will display zero, giving no warning.

### Recommendation
In `compute_royalty_amount`, after computing `amount`, raise a `ValueError` (or return a minimum of 1 mojo) if `percentage > 0` but `amount == 0`, preventing offers that would silently zero out a non-zero royalty. Alternatively, enforce a minimum offer amount of `10000 // percentage` mojos when royalty-bearing NFTs are involved. [1](#0-0) 

### Proof of Concept
1. Mint an NFT with `royalty_percentage = 100` (1%).
2. Create an offer: sell the NFT for 50 CAT mojos (0.05 CAT).
3. `compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)` → `50 // 1 * 100 // 10000 = 5000 // 10000 = 0`.
4. `make_nft1_offer` skips the royalty payment coin (`payment_sum == 0`).
5. The NFT transfers; the royalty address receives 0 mojos.

This is confirmed by the existing test: [7](#0-6)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
