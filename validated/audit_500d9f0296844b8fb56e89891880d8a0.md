Audit Report

## Title
Cross-Token Withdrawal Allows Fund Theft When Multiple Tokens Are Allowed - (File: contracts/L2/RsETHTokenWrapper.sol)

## Summary
`RsETHTokenWrapper._withdraw` validates only that the caller-supplied `_asset` is in `allowedTokens`, but never enforces that the withdrawn token matches the token originally deposited. Because the contract explicitly supports multiple allowed tokens via `reinitialize` and `addAllowedToken`, any `wrsETH` holder can burn shares and receive whichever allowed token they choose, enabling direct theft of funds from other depositors when a price discrepancy exists between allowed tokens.

## Finding Description
The `_deposit` function at lines 134–141 mints `wrsETH` 1:1 for any allowed token without recording which token was deposited. The `_withdraw` function at lines 120–128 burns `wrsETH` and transfers any caller-specified allowed token, with no per-token accounting or deposit-withdrawal pairing:

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();  // sole guard
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    emit Withdraw(_asset, msg.sender, _to, _amount);
}
```

The contract is explicitly designed to hold multiple allowed tokens simultaneously: `initialize` adds the first token, `reinitialize` (lines 47–49) adds a second, and `addAllowedToken` (lines 174–176) adds further tokens. The `wrsETH` supply therefore represents aggregate collateral across all allowed tokens, but withdrawal is unrestricted as to which token is redeemed. The public entry points `withdraw` (line 84) and `withdrawTo` (line 92) impose no additional restrictions. No per-token reserve tracking exists anywhere in the contract.

## Impact Explanation
**Critical — direct theft of user funds.** When two allowed tokens trade at different prices (e.g., due to bridge latency, liquidity imbalance, or market conditions between L2 representations of rsETH), an attacker deposits the cheaper token, receives `wrsETH` at 1:1, then withdraws the more expensive token at 1:1. The attacker captures the price spread, and remaining depositors are left holding only the cheaper token. Repeated at scale, this drains the entire balance of the more valuable token. This matches the allowed Critical impact: "Direct theft of any user funds."

## Likelihood Explanation
**Medium.** Two preconditions are required: (1) at least two tokens must be in `allowedTokens` simultaneously — a supported and documented contract state via `reinitialize` and `addAllowedToken`; and (2) a price discrepancy must exist between the two tokens. Alt-rsETH bridge representations on L2 chains routinely trade at small but non-zero spreads due to bridge latency and liquidity differences. No special role or permission is needed once both conditions are met — any `wrsETH` holder can execute the attack.

## Recommendation
Introduce per-token reserve accounting: maintain a `mapping(address => uint256) public tokenReserves` that is incremented on `_deposit` and decremented on `_withdraw`. Enforce in `_withdraw` that the contract's tracked reserve for `_asset` is sufficient to cover `_amount`, preventing cross-token substitution. Alternatively, restrict the contract to a single canonical withdrawal token, or require that each `wrsETH` mint records the deposited token and only allows redemption of that same token.

## Proof of Concept
**Setup**: `tokenA` (price 0.99 rsETH) and `tokenB` (price 1.00 rsETH) are both in `allowedTokens`. The contract holds 1000 `tokenB` deposited by legitimate users.

**Attack sequence**:
```
1. Attacker acquires 1000 tokenA (cost: ~990 USD equivalent).
2. Attacker calls deposit(tokenA, 1000)
   → _deposit checks allowedTokens[tokenA] == true ✓
   → transfers 1000 tokenA into contract
   → mints 1000 wrsETH to attacker
3. Attacker calls withdraw(tokenB, 1000)
   → _withdraw checks allowedTokens[tokenB] == true ✓  (sole guard passes)
   → burns 1000 wrsETH from attacker
   → transfers 1000 tokenB to attacker (~1000 USD)
4. Net profit: ~10 USD per 1000 units. Scales linearly.
   Legitimate tokenB depositors now hold only tokenA.
```

**Foundry test plan**: Deploy `RsETHTokenWrapper`, call `initialize` with `tokenA`, call `reinitialize` with `tokenB`, seed the contract with 1000 `tokenB` via `depositBridgerAssets`, execute the above sequence as an unprivileged attacker address, and assert that the attacker's `tokenB` balance equals 1000 and the contract's `tokenB` balance equals 0.
