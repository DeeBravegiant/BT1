Audit Report

## Title
stETH `transferFrom` 1-2 Wei Share-Rounding Causes rsETH Over-Minting in `depositAsset` - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.depositAsset` computes `rsethAmountToMint` via `_beforeDeposit` using the caller-supplied `depositAmount` before the stETH transfer occurs. Because stETH uses share-based accounting, `safeTransferFrom` delivers `depositAmount - 1` or `depositAmount - 2` wei to the contract, yet rsETH is minted for the full `depositAmount`. This creates a systematic, accumulating deficit between actual stETH held and rsETH supply, fractionally diluting all existing rsETH holders.

## Finding Description
In `LRTDepositPool.depositAsset` (L109–117):

```solidity
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
```

`_beforeDeposit` (L648–670) calls `getRsETHAmountToMint(asset, depositAmount)` (L506–521), which computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This uses the user-supplied `depositAmount` as the basis. stETH's `transferFrom` internally converts the token amount to shares (rounding down) and back, so the contract receives `depositAmount - 1` or `depositAmount - 2` wei. The rsETH minted is still based on the original `depositAmount`.

`getTotalAssetDeposits` (L385–396) uses `IERC20(asset).balanceOf(address(this))` for `assetLyingInDepositPool`, which reflects the actual (lower) balance. The rsETH price oracle therefore computes a slightly lower `totalETHValue / totalRsETHSupply` ratio than warranted, because `totalRsETHSupply` is inflated by 1–2 wei per deposit while `totalETHValue` reflects actual holdings. No existing guard prevents this: `_checkIfDepositAmountExceedesCurrentLimit` also uses `depositAmount` (not actual received), so the deposit cap check is also off by 1–2 wei per deposit.

## Impact Explanation
Every stETH deposit via `depositAsset` mints rsETH for 1–2 wei of stETH the protocol never received. Over a large number of deposits this accumulates: the rsETH price (`totalETHValue / totalRsETHSupply`) is fractionally lower than it should be, and existing rsETH holders receive slightly less than the promised backing when redeeming. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
stETH is a core supported asset (registered as `LRTConstants.ST_ETH_TOKEN` in `LRTConfig.initialize`). The rounding is a structural property of every stETH `transferFrom` call, not a conditional edge case. Any unprivileged user calling `depositAsset` with stETH triggers it. Likelihood is **High**.

## Recommendation
Use a balance-before/balance-after pattern to determine the actual received amount and base rsETH minting on that:

```solidity
uint256 balanceBefore = IERC20(asset).balanceOf(address(this));
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
uint256 actualReceived = IERC20(asset).balanceOf(address(this)) - balanceBefore;
uint256 rsethAmountToMint = _beforeDeposit(asset, actualReceived, minRSETHAmountExpected);
_mintRsETH(rsethAmountToMint);
```

Alternatively, use Lido's `transferShares` so the transferred amount is exact.

## Proof of Concept
1. Alice holds 1e18 stETH.
2. Alice calls `LRTDepositPool.depositAsset(stETH, 1e18, minRsETH, "")`.
3. `_beforeDeposit` computes `rsethAmountToMint` based on `1e18`.
4. `safeTransferFrom` executes; due to stETH share rounding, the contract receives `1e18 - 1` wei.
5. `_mintRsETH(rsethAmountToMint)` mints rsETH for `1e18` stETH worth of value.
6. The protocol holds `1e18 - 1` stETH but has issued rsETH backed by `1e18` stETH — a 1 wei deficit.
7. Repeated across N deposits, the deficit grows to N–2N wei; `rsETHPrice = totalETHValue / totalRsETHSupply` is fractionally lower, diluting all existing rsETH holders.

**Foundry fork test plan:** Fork mainnet, deploy/configure `LRTDepositPool` with real stETH, call `depositAsset` with a round stETH amount, assert `IERC20(stETH).balanceOf(depositPool) < depositAmount` and `rsETH.totalSupply()` corresponds to `depositAmount` rather than the actual received balance.     function _beforeDeposit(
```text
        address asset,
        uint256 depositAmount,
        uint256 minRSETHAmountExpected
    )
        private
        view
        returns (uint256 rsethAmountToMint)
    {
        if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
            revert InvalidAmountToDeposit();
        }

        if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
            revert MaximumDepositLimitReached();
        }

        rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

        if (rsethAmountToMint < minRSETHAmountExpected) {
            revert MinimumAmountToReceiveNotMet();
        }
    }
```
