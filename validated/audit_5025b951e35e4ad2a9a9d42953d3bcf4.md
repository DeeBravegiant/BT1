Audit Report

## Title
Unprotected `updateRate()` Allows Anyone to Drain Contract ETH, Permanently Disrupting Cross-Chain Rate Propagation - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary
`MultiChainRateProvider.updateRate()` has no access control and draws LayerZero fees from the contract's own ETH balance without verifying that `msg.value` covers the total cost. Any unprivileged caller can invoke it repeatedly with `msg.value = 0`, consuming the contract's ETH across all registered receiver chains. Once drained, all future rate updates revert, leaving every connected L2 pool permanently relying on a stale rsETH/ETH exchange rate.

## Finding Description
`updateRate()` is declared `external payable nonReentrant` with no ownership or role guard:

```solidity
// contracts/cross-chain/MultiChainRateProvider.sol L108
function updateRate() external payable nonReentrant {
```

Inside the loop, for each registered receiver the function calls:

```solidity
// L124-129
(uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
    .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
    dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
);
```

`{ value: estimatedFee }` draws from `address(this).balance`, not exclusively from `msg.value`. There is no assertion that `msg.value >= sum(estimatedFees)`. The NatSpec comment at L105-107 explicitly documents that the contract is intended to hold ETH:

> "This function is set to payable to pay for gas on execute lzReceive (on the receiver contract) on the destination chain."

An attacker calling with `msg.value = 0` silently consumes the contract's ETH across all receivers on every call. `RSETHMultiChainRateProvider` inherits this logic unchanged and adds no additional guard.

## Impact Explanation
Once the contract's ETH is exhausted, every call to `updateRate()` reverts at the first `ILayerZeroEndpoint.send`. All L2 rate receivers stop receiving fresh rsETH/ETH prices. L2 pools (`RSETHPool`/`RSETHPoolV3` variants) continue to mint `wrsETH` against a stale rate, meaning depositors receive incorrect amounts of `wrsETH` relative to the true rsETH price. This maps to **Low — Contract fails to deliver promised returns**: the rate-provider contract permanently fails to deliver its core service (fresh cross-chain rates) until the contract is re-funded and patched, with secondary risk of incorrect wrsETH issuance on all connected L2 chains.

## Likelihood Explanation
The attack requires only a public call with `msg.value = 0` — no privileges, no capital at risk, no complex setup. With multiple registered receivers, each call drains one LayerZero fee per receiver. The barrier to entry is zero, and the attack is repeatable until the balance reaches zero.

## Recommendation
1. **Add access control**: restrict `updateRate()` to `onlyOwner` or a dedicated `UPDATER_ROLE`.
2. **Alternatively, require the caller to cover fees**: assert `msg.value >= estimateTotalFee()` at the top of the function so the contract never subsidises callers from its own balance.
3. **Refund excess `msg.value`** after the loop to prevent ETH accumulation that could be drained in future calls.

## Proof of Concept
1. Protocol deploys `RSETHMultiChainRateProvider`, funds it with 1 ETH to cover automated updates across 3 L2 chains (~0.03 ETH per call).
2. Attacker calls `updateRate()` with `msg.value = 0` in a loop (~33 iterations).
3. Each call draws `estimatedFee × 3` from the contract balance; attacker pays only L1 gas.
4. After ~33 calls the contract balance reaches 0.
5. The next legitimate `updateRate()` call reverts with insufficient balance on the first `ILayerZeroEndpoint.send`.
6. All registered L2 rate receivers stop updating; L2 pools mint `wrsETH` at a permanently stale rsETH/ETH rate.
