Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Deposit Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` evaluates `totalAssetDeposits > limit` for ETH instead of `totalAssetDeposits + amount > limit`, meaning the incoming deposit value is never factored into the cap check. Any unprivileged caller can invoke `depositETH` with an arbitrarily large `msg.value` and bypass the ETH deposit cap in a single transaction, inflating rsETH supply beyond the protocol's intended risk ceiling.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676ŌĆō682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ŌåÉ amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ŌåÉ correct
}
```

The ETH branch checks whether the cap is already exceeded *before* the deposit, not whether the deposit *would* exceed it. The ERC20 branch correctly adds `amount`. The function is the sole guard called by `_beforeDeposit` (line 661), which is invoked by `depositETH` (line 87) before minting rsETH. As long as `totalAssetDeposits Ōēż depositLimit` (the normal operating state), the check always returns `false` regardless of how large `msg.value` is, and the deposit proceeds unconditionally.

## Impact Explanation
**Critical ŌĆō Protocol insolvency.** The deposit cap is the primary risk-management control limiting ETH concentration in EigenLayer strategies. A single depositor can push total ETH holdings arbitrarily far above the cap in one transaction. The protocol mints rsETH at the current exchange rate for the full over-limit amount. If EigenLayer strategies cannot absorb or redeem the excess, the protocol holds more ETH obligations than it can service, degrading the rsETH/ETH backing ratio for all holders and driving the protocol toward insolvency.

## Likelihood Explanation
**High.** `depositETH` is `external payable`, requires no role or whitelist, and is the primary user-facing deposit entry point. The broken branch is exercised on every ETH deposit while `totalAssetDeposits Ōēż depositLimit`, which is the normal operating state. No preconditions beyond possessing ETH are required; the exploit is repeatable and atomic.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Preconditions:
- `depositLimitByAsset(ETH_TOKEN) = 1000 ether`
- `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`

Steps:
1. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
2. `_beforeDeposit(ETH_TOKEN, 10_000 ether, 0)` ŌåÆ `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)`.

6. Protocol now holds 10 999 ETH against a cap of 1 000 ETH; rsETH supply inflated by 10 000 ETH worth of tokens.
