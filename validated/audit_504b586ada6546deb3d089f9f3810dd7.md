Audit Report

## Title
ETH Withdrawal Permanently Frozen When Recipient Contract Rejects ETH â€” No Cancellation Path Exists - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._processWithdrawalCompletion` transfers ETH via a low-level `call` that reverts if the recipient contract cannot accept ETH. Because rsETH is burned in a prior, separate transaction (`unlockQueue`), and no cancellation or redirect mechanism exists, the ETH becomes permanently locked inside `LRTWithdrawalManager` with no admin recovery path available while the stuck withdrawal remains outstanding.

## Finding Description
The withdrawal lifecycle spans two separate transactions:

**Transaction 1 â€” `unlockQueue`:** Burns rsETH via `IRSETH.burnFrom` and pulls ETH from `LRTUnstakingVault` into `LRTWithdrawalManager`. This is irreversible once mined. [1](#0-0) 

**Transaction 2 â€” `completeWithdrawal` / `completeWithdrawalForUser`:** Calls `_processWithdrawalCompletion`, which pops the nonce, deletes the request, decrements `unlockedWithdrawalsCount`, then calls `_transferAsset`. [2](#0-1) 

`_transferAsset` for ETH:
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
``` [3](#0-2) 

If `to` is a contract with no `receive()` or one that reverts on ETH receipt, `sent == false` and the entire Transaction 2 reverts atomically â€” restoring the nonce, the `WithdrawalRequest`, and `unlockedWithdrawalsCount`. Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` identically reverts.

**Why existing checks fail:**

- `completeWithdrawalForUser` (operator-assisted path) calls the same `_processWithdrawalCompletion` with the same `user` address â€” it sends ETH to the same rejecting contract. [4](#0-3) 
  The NatDoc comment at line 191 even acknowledges: *"Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH"* â€” confirming no ETH-specific redirect exists.

- `sweepRemainingAssets` is gated by `hasUnlockedWithdrawals(asset) == false`. While the stuck withdrawal remains outstanding, `unlockedWithdrawalsCount[asset] > 0`, so this call reverts with `PendingWithdrawalsExist`. [5](#0-4) [6](#0-5) 

- There is no `cancelWithdrawal`, no `redirectWithdrawal`, and no re-mint path anywhere in the contract.

## Impact Explanation
After `unlockQueue` executes, the user's rsETH is irreversibly burned and the corresponding ETH is held in `LRTWithdrawalManager`. If `completeWithdrawal` can never succeed, the ETH is permanently frozen inside the contract and the user has lost both their rsETH and their ETH. No admin function can recover the ETH without a contract upgrade, because `sweepRemainingAssets` is blocked by the outstanding unlocked withdrawal count.

**Impact: Permanent freezing of user funds â€” Critical per allowed scope.**

## Likelihood Explanation
Smart contracts that interact with DeFi protocols (multisigs, vaults, aggregators, proxy wallets) frequently lack a plain `receive()` function or have one that conditionally reverts. Any such contract that calls `initiateWithdrawal` for the ETH asset will trigger this path. The condition is realistic and not uncommon in DeFi composability contexts. **Likelihood: Low-Medium.**

## Recommendation
Add a withdrawal cancellation or redirect mechanism. At minimum, implement a manager-callable `redirectWithdrawal(address asset, address fromUser, address toRecipient)` that completes the transfer to an alternative address when the original recipient cannot accept ETH. Alternatively, allow the user (or an operator on their behalf) to cancel a stuck unlocked withdrawal and re-mint the equivalent rsETH value, restoring the user's position.

## Proof of Concept
1. Deploy a smart contract `NoReceive` with no `receive()` function.
2. From `NoReceive`, call `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, amount, "")`. rsETH is transferred to the manager.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned; ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager`. This transaction succeeds and is final.
4. `NoReceive` calls `completeWithdrawal(ETH_TOKEN, "")`. The low-level `call` to `NoReceive` returns `false`; the function reverts with `EthTransferFailed`. All state changes in this transaction roll back.
5. Repeat step 4 indefinitely â€” always reverts.
6. Call `sweepRemainingAssets(ETH_TOKEN)` â€” reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.
7. The ETH is permanently locked in `LRTWithdrawalManager`; the rsETH is permanently burned.

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
```
