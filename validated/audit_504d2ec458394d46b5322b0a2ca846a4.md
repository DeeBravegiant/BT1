### Title
Deposit Refund Double-Counted in Balance Check for Dijkstra Sub-Transactions - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary
In the Dijkstra era, `getConsumedDijkstraValue` computes the consumed value for a top-level transaction by aggregating consumed values from both the top-level body and all sub-transactions, using the **same pre-transaction** deposit-lookup functions for every sub-transaction. If multiple sub-transactions each contain a deregistration certificate for the same credential, the deposit refund is counted once per sub-transaction in the balance check, inflating the `consumed` side and allowing the transaction to balance with outputs that exceed the legitimately available funds.

### Finding Description

`getConsumedDijkstraValue` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` is:

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
``` [1](#0-0) 

The `lookupStakingDeposit` and `lookupDRepDeposit` closures are derived from the **original** `certState` (before any sub-transactions are applied) and are passed unchanged into every sub-transaction's consumed-value computation. `getConsumedMaryValue` calls `getTotalRefundsTxBody`, which calls `shelleyTotalRefundsTxCerts` / `conwayDRepRefundsTxCerts`. Each of these functions independently queries `lookupDeposit cred` against the same original state:

```haskell
| Just deposit <- lookupDeposit cred -> (regCreds, totalRefunds <+> deposit)
``` [2](#0-1) 

Because the lookup always returns the deposit from the **pre-transaction** state, two sub-transactions each containing `UnRegDepositTxCert stakingCred D` will each independently find `D` and each add `D` to the consumed total, yielding `consumed = inputs + 2D` instead of `inputs + D`.

The preservation-of-value check in `dijkstraUtxoTransition` then enforces `consumed = produced`:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [3](#0-2) 

With `consumed = inputs + 2D`, the check allows `outputs = inputs + 2D - fees`, meaning the transaction can balance while producing `D - fees` more ADA in outputs than was legitimately available.

The developers are aware of this gap. A test explicitly marked `xit` ("skipped/pending") reads:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
    ...
    submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [4](#0-3) 

The `error "TODO: predicate failure not yet implemented"` confirms that the guard preventing this double-refund has not been implemented. The test is skipped precisely because the expected rejection does not yet exist.

The same structural issue applies to DRep deposit refunds via `conwayDRepRefundsTxCerts`, which also uses the same pre-transaction `lookupDRepDeposit` across all sub-transactions. [5](#0-4) 

### Impact Explanation

If a transaction with two sub-transactions each deregistering the same credential passes the balance check and is accepted by the ledger, the attacker extracts `D` lovelace from the deposit pot that was never legitimately theirs. This constitutes a direct, attacker-controlled creation of ADA value through an invalid ledger state transition — matching the **Critical** impact class: *"Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition."*

The deposit pot is a shared ledger resource; draining it harms all future deregistrants who are owed refunds.

### Likelihood Explanation

The Dijkstra era is the newest era and sub-transactions are a novel feature. The `xit` test and `TODO` comment confirm the protection is absent. An unprivileged transaction author can craft a top-level transaction with two sub-transactions containing identical deregistration certificates. The only remaining question is whether the Dijkstra `SUBCERTS`/`SUBLEDGER` rules independently reject the second deregistration before the balance check is satisfied; if those rules process sub-transactions against the evolving (post-sub-tx-1) state, the second deregistration fails and the transaction is rejected for a different reason. However, the skipped test and missing predicate failure indicate this path has not been hardened, making the likelihood of a successful exploit non-trivial once the era is live.

### Recommendation

1. In `getConsumedDijkstraValue`, track which credentials have already been counted for a refund across sub-transactions and suppress duplicate lookups, analogous to how `shelleyTotalRefundsTxCerts` tracks intra-tx registrations via `regCreds`.
2. Implement the missing predicate failure referenced by the `xit` test so that a transaction containing duplicate deregistration certificates across sub-transactions is explicitly rejected at the balance-check stage.
3. Apply the same deduplication to DRep refunds in `conwayDRepRefundsTxCerts` when called from `getConsumedDijkstraValue`.

### Proof of Concept

```
1. Register stakingCred on-chain (deposit D paid).
2. Craft a top-level Dijkstra transaction with no certificates in the top-level body.
3. Add sub-tx1: inputs = {utxo1}, certs = [UnRegDepositTxCert stakingCred D], outputs = [addr ← (value1 + D)]
4. Add sub-tx2: inputs = {utxo2}, certs = [UnRegDepositTxCert stakingCred D], outputs = [addr ← (value2 + D)]
5. Balance check computes:
     consumed = value1 + value2 + D (sub-tx1 refund) + D (sub-tx2 refund) = value1 + value2 + 2D
     produced = (value1 + D) + (value2 + D) + fees = value1 + value2 + 2D + fees
   → balance satisfied if fees are covered by the top-level tx inputs.
6. If the SUBCERTS rule does not reject the second deregistration, the transaction is accepted
   and the attacker receives D lovelace more than they deposited.
``` [6](#0-5) [4](#0-3)
