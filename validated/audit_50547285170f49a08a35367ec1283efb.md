Audit Report

## Title
Permissionless Share Injection Enables Permanent Griefing of LP Withdrawals via `MinimalLiquidity` Guard — (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
`addLiquidity` in `MetricOmmPool` accepts an arbitrary `owner` address with no requirement that `msg.sender == owner`, allowing any caller to inject shares into any existing position. `removeLiquidity` in `LiquidityLib` reverts when the post-removal share count falls in the range `(0, minimalMintableLiquidity)`. An attacker who observes a victim's `removeLiquidity` call can frontrun it by injecting a single dust share, shifting the post-removal remainder from zero into the forbidden range and causing a permanent, repeatable revert. Because there is no "remove all" escape hatch, the victim's funds can be locked indefinitely.

## Finding Description

**Permissionless deposit to any position.**
`MetricOmmPool.addLiquidity` accepts an `owner` parameter with no `msg.sender == owner` check:

```solidity
// MetricOmmPool.sol L182-196
function addLiquidity(
    address owner, uint80 salt, LiquidityDelta calldata deltas,
    bytes calldata callbackData, bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    ...
    _beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
    (amount0Added, amount1Added) = LiquidityLib.addLiquidity(
        _liquidityContext(), owner, salt, deltas, ...
    );
```

By contrast, `removeLiquidity` enforces `if (msg.sender != owner) revert NotPositionOwner()` at L206, but `addLiquidity` has no equivalent guard.

**`MinimalLiquidity` guard in `removeLiquidity`.**
`LiquidityLib.removeLiquidity` at L199-202 reverts when the post-removal share count is a non-zero dust value:

```solidity
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

**`addLiquidity` minimum check is on the new total, not the delta.**
`LiquidityLib.addLiquidity` at L76-79 checks `newUserShares = existing + delta >= minimalMintableLiquidity`. For a position already holding ≥ `minimalMintableLiquidity - 1` shares, adding exactly 1 share passes this guard unconditionally.

**Attack sequence:**
1. Alice holds exactly `minimalMintableLiquidity` (e.g., 1000) shares in bin 4.
2. Alice submits `removeLiquidity(1000)` → `newUserShares = 0` → passes guard.
3. Bob frontruns: calls `addLiquidity(owner=Alice, shares=1)` → Alice now has 1001 shares. `newUserShares = 1001 >= 1000` passes the add-liquidity guard.
4. Alice's original tx executes: `newUserShares = 1001 - 1000 = 1`. Since `1 > 0 && 1 < 1000`, `MinimalLiquidity(1, 1000)` reverts.
5. Alice re-queries and submits `removeLiquidity(1001)`. Bob repeats step 3. Indefinitely.

**Cost to attacker.**
The token cost for 1 share is `Math.ceilDiv(binState.token0BalanceScaled * 1, binTotalSharesVal)` scaled units, then converted to external units via `Math.ceilDiv(scaledAmount, token0ScaleMultiplier)`. For tokens with fewer than 18 decimals (large `token0ScaleMultiplier`) or bins with low scaled balance relative to total shares, the external cost rounds to 0 or 1 wei per injection — negligible for sustained griefing.

**No escape hatch.**
`removeLiquidity` requires the caller to specify an exact share count. There is no `type(uint256).max` sentinel or "burn all" variant that bypasses the `MinimalLiquidity` guard when `newUserShares` would be zero.

## Impact Explanation
An LP's entire position can be permanently locked. The victim cannot withdraw principal without access to a private mempool. This constitutes a broken core pool functionality (unusable withdraw flow) and direct loss of user principal, meeting Sherlock High/Medium thresholds. The exact corrupted value is `positionBinShares[posKey]`, which is permanently stranded in the forbidden range `(0, minimalMintableLiquidity)` from the victim's perspective.

## Likelihood Explanation
- No special privilege is required; any EOA-backed contract implementing `IMetricOmmModifyLiquidityCallback` can call `addLiquidity` for any `owner`.
- The attack cost is near-zero (1 wei or 0 tokens per injection depending on bin state and token decimals).
- Mempool monitoring is standard for MEV bots.
- The `DepositAllowlistExtension` mitigates this only when explicitly configured on the pool; the base pool has no such guard.
- The attack is fully repeatable: every time the victim re-queries and resubmits, the attacker can frontrun again.

## Recommendation
**Option A (preferred):** Allow `sharesToRemove == type(uint256).max` to mean "burn all shares," bypassing the `MinimalLiquidity` check when `newUserShares` would be exactly zero. This preserves the operator pattern while giving LPs an atomic escape hatch.

**Option B:** Enforce `msg.sender == owner` inside `addLiquidity` (remove the operator pattern). This eliminates the injection vector entirely but breaks legitimate operator use-cases.

**Option C:** Apply the `MinimalLiquidity` guard to the *delta* being added (not the new total), so that adding fewer than `minimalMintableLiquidity` shares to any position is rejected. This raises the per-injection cost to at least `minimalMintableLiquidity` shares worth of tokens, making sustained griefing economically unattractive.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

function testFrontrunRemoveLiquidity() public {
    // Alice adds minimalMintableLiquidity (1000) shares to bin 4.
    vm.prank(alice);
    pool.addLiquidity(alice, SALT, _delta(4, 1000), abi.encode(KIND_PAY), "");
    // Alice's position: 1000 shares. removeLiquidity(1000) → newUserShares=0 → passes.

    // Bob frontruns: injects 1 share into Alice's position.
    // newUserShares for add = 1001 >= 1000 → passes addLiquidity guard.
    vm.prank(bob);
    pool.addLiquidity(alice, SALT, _delta(4, 1), abi.encode(KIND_PAY), "");
    // Alice now has 1001 shares.

    // Alice's original tx: remove 1000 shares.
    // newUserShares = 1001 - 1000 = 1; 1 > 0 && 1 < 1000 → MinimalLiquidity revert.
    vm.prank(alice);
    vm.expectRevert(
        abi.encodeWithSelector(IMetricOmmPoolActions.MinimalLiquidity.selector, 1, 1000)
    );
    pool.removeLiquidity(alice, SALT, _delta(4, 1000), "");

    // Bob repeats every time Alice resubmits. Alice's funds are permanently locked.
}
```

The root cause is confirmed at:
- `LiquidityLib.addLiquidity` L76-79: guard on new total, not delta
- `LiquidityLib.removeLiquidity` L199-202: `MinimalLiquidity` guard on remainder
- `MetricOmmPool.addLiquidity` L182-196: no `msg.sender == owner` check
