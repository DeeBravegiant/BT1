Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Deposit Cap Bypass ‚ÄĒ (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric check: the ETH branch tests only `totalAssetDeposits > depositLimit` (line 679), omitting the incoming `amount`, while the ERC20 branch correctly tests `totalAssetDeposits + amount > depositLimit` (line 681). Any unprivileged caller can push ETH deposits arbitrarily beyond the configured cap in a single transaction, leaving excess ETH idle in the pool and diluting the rsETH exchange rate for all holders.

## Finding Description [1](#0-0) 

The public entry point `depositETH` (line 76) passes `msg.value` as `depositAmount` to `_beforeDeposit` (line 87), which calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` at line 661 and reverts with `MaximumDepositLimitReached` only when the function returns `true`. For ETH, the function returns `totalAssetDeposits > depositLimit` ‚ÄĒ a condition that is already `false` during normal protocol operation (i.e., while the cap has not yet been reached). The incoming `amount` is never added, so the guard fires only after the limit is already exceeded, not when a deposit would push it over. The ERC20 path correctly adds `amount` before comparing. [2](#0-1) 

## Impact Explanation
The deposit limit is the protocol's primary mechanism for capping ETH exposure per asset. Because the check is broken for ETH, any depositor can push the total ETH held in `LRTDepositPool` arbitrarily beyond the configured cap. Excess ETH that cannot be deployed to EigenLayer strategies sits idle in the pool, earning no yield, and dilutes the rsETH exchange rate for all existing holders. No principal is lost, but the protocol fails to enforce its own promised cap and all rsETH holders receive lower-than-expected yield on the excess idle ETH.

**Impact: Low** ‚ÄĒ Contract fails to deliver promised returns (deposit limit not enforced for ETH), but no direct loss of principal.

## Likelihood Explanation
The vulnerable path is the public, permissionless `depositETH` function. [3](#0-2) 

No special role, privilege, or precondition is required beyond having ETH to deposit. The exploitable condition (`totalAssetDeposits(ETH) ‚Č§ depositLimit`) is the normal operating state of the protocol. The issue is repeatable by any depositor at any time before the limit is organically exceeded. **Likelihood: High.**

## Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ‚Üź add amount
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH) = 1000 ether`.
2. Current `getTotalAssetDeposits(ETH) = 999 ether`.
3. Depositor calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ether)` evaluates `999 ether > 1000 ether` ‚Üí `false`.
5. Deposit is accepted; total ETH in protocol becomes `1499 ether` ‚ÄĒ 499 ETH over the intended cap.
6. The 499 ETH excess sits idle in `LRTDepositPool`, not deployed to EigenLayer, earning no yield, diluting rsETH price for all holders.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 1000 ether);
    // seed pool to 999 ether via prior deposits ...
    vm.deal(attacker, 500 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 500 ether}(0, "");
    assertGt(lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
```
