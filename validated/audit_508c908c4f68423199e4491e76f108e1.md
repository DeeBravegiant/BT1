### Title
Unpermissioned `init` Allows Any Caller to Seize Control of the MPC Participant Set During Deployment Window - (File: crates/contract/src/lib.rs)

### Summary
The `MpcContract::init` function carries no access-control guard. The production deployment flow deploys the contract binary first (`without-init-call`) and calls `init` in a separate, later transaction. Any unprivileged NEAR account that observes the deployment transaction can race to call `init` with an attacker-controlled participant set before the legitimate operator does, permanently seizing governance of the MPC network and its threshold-signing capability.

### Finding Description
`MpcContract::init` is decorated only with `#[init]` (the NEAR SDK one-time-initialization guard) and `#[handle_result]`. It carries **no** `#[private]` attribute and performs **no** check on `env::predecessor_account_id()` or `env::signer_account_id()`. [1](#0-0) 

Compare with `init_running`, which is correctly gated with `#[private]`: [2](#0-1) 

The production deployment script (`deploy-tee-cluster.sh`) deploys the contract binary in one step using `without-init-call`: [3](#0-2) 

Then calls `init` in a separate, later step: [4](#0-3) 

The same two-step pattern appears in `scripts/launch-localnet.sh`: [5](#0-4) 

Between the `deploy` phase and the `near_init` phase there is a publicly observable window (multiple blocks, potentially minutes) during which the contract is live on-chain but uninitialized. Because `#[init]` only prevents a *second* call — it does not restrict *who* makes the first call — any NEAR account can submit a valid `init` transaction during this window.

### Impact Explanation
An attacker who wins the race calls `init` with a `ThresholdParameters` struct that lists only their own account(s) as participants and sets `threshold = 1`. The contract transitions immediately to `Running` state with the attacker as the sole participant. [6](#0-5) 

From that position the attacker can:
- Vote to add signing domains (`vote_add_domains`) and trigger key generation, producing MPC key shares under their control.
- Issue threshold signatures for arbitrary foreign-chain transactions (Bitcoin, Ethereum, etc.) via `sign()` — constituting **unauthorized threshold signature issuance**.
- Permanently block the legitimate operator: because `#[init]` prevents a second call, the legitimate `init` will fail, and the only recovery path is redeploying the contract (losing any state).

This maps directly to the allowed Critical impact: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization.*

### Likelihood Explanation
The deployment window is observable by anyone monitoring the NEAR blockchain. The `deploy_contract` and `init_contract` steps are separated by multiple script phases (key collection, `init_args` generation, key registration) that can take minutes to hours in a real deployment. A front-running bot watching for new contract deployments with no state can trivially detect and exploit this window. No privileged access, leaked key, or collusion is required — only a funded NEAR account and knowledge of the `init` ABI (which is public).

### Recommendation
Add `#[private]` to `init`, mirroring the guard already present on `init_running`:

```rust
#[handle_result]
#[init]
#[private]          // ← add this
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

`#[private]` in the NEAR SDK asserts `predecessor_account_id() == current_account_id()`, meaning only the contract account itself (i.e., the deployer signing as the contract account, which is exactly what the deployment scripts do) can call `init`. This closes the race window without changing the legitimate deployment flow.

Alternatively, deploy the contract and call `init` atomically in a single transaction using NEAR's `deploy + init` combined call, eliminating the window entirely.

### Proof of Concept

1. Operator broadcasts: `near contract deploy mpc-contract.test.near use-file mpc_contract.wasm without-init-call ...`
2. Attacker observes the deployment transaction on-chain (contract now has code, no state).
3. Before the operator's `init_contract` step executes, attacker broadcasts:
   ```
   near contract call-function as-transaction mpc-contract.test.near init \
     json-args '{
       "parameters": {
         "threshold": 1,
         "participants": {
           "next_id": 1,
           "participants": [["attacker.near", 0, {"tls_public_key": "ed25519:<attacker_key>", "url": "https://attacker.example.com"}]]
         }
       }
     }' prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near network-config mainnet sign-with-keychain send
   ```
4. `init` succeeds (no access check, contract was uninitialized). Contract is now in `Running` state with `attacker.near` as the sole participant.
5. Operator's subsequent `init` call fails: `"The contract has already been initialized"`.
6. Attacker calls `vote_add_domains` to add a Secp256k1 signing domain, triggers key generation, and can now issue threshold signatures for any foreign-chain transaction. [7](#0-6)

// Contract developer helper API
#[near]
impl MpcContract {
    #[handle_result]
    #[init]
    pub fn init(
        parameters: dtos::ThresholdParameters,
        init_config: Option<dtos::InitConfig>,
    ) -> Result<Self, Error> {
        let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
        // Log participant count and hash - full parameters exceed NEAR's 16KB log limit at ~100 participants
        let params_hash = env::sha256_array(borsh::to_vec(&parameters).unwrap());
        log!(
            "init: signer={}, num_participants={}, parameters_hash={:?}, init_config={:?}",
            env::signer_account_id(),
            parameters.participants().len(),
            params_hash,
            init_config,
        );
        parameters.validate()?;

        // TODO(#1087): Every participant must have a valid attestation, otherwise we risk
        // participants being immediately kicked out once contract transitions into running.
        let initial_participants = parameters.participants();
        let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);

        Ok(Self {
            protocol_state: ProtocolContractState::Running(RunningContractState::new(
                DomainRegistry::default(),
                Keyset::new(EpochId::new(0), Vec::new()),
                parameters,
                AddDomainsVotes::default(),
            )),
            pending_signature_requests: LookupMap::new(StorageKey::PendingSignatureRequestsV4),
            pending_ckd_requests: LookupMap::new(StorageKey::PendingCKDRequestsV3),
            pending_verify_foreign_tx_requests: LookupMap::new(
                StorageKey::PendingVerifyForeignTxRequestsV2,
            ),
            proposed_updates: ProposedUpdates::default(),
            config: init_config.map(Into::into).unwrap_or_default(),
            tee_state,
            accept_requests: true,
            node_migrations: NodeMigrations::default(),
            metrics: Default::default(),
            node_foreign_chain_support: Default::default(),
            foreign_chains: Lazy::new(
                StorageKey::ForeignChainMetadata,
                ForeignChainsMetadata::default(),
            ),
            tee_verifier_account_id: None,
            tee_verifier_votes: TeeVerifierVotes::default(),
        })
    }
