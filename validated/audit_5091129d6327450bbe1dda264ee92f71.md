Audit Report

## Title
Unprivileged Caller Can Trigger Circuit-Breaker via Transient swETH Rate Dip, Temporarily Freezing All Deposits and Withdrawals — (`contracts/LRTOracle.sol`)

## Summary

`updateRSETHPrice()` is an unrestricted public function. Its downside circuit-breaker fires unconditionally for any caller when `newRsETHPrice` drops below `highestRsethPrice` by more than `pricePercentageLimit`, immediately pausing `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle`. Because `SwETHPriceOracle` passes through `swETH.getRate()` raw with no smoothing, any unprivileged caller who times their call to a block where swETH's spot rate has transiently dipped can freeze the protocol until an admin manually unpauses each contract.

## Finding Description

**No access control on entry point:**
`updateRSETHPrice()` is `public whenNotPaused` with no role restriction — any EOA or contract can call it at any block. [1](#0-0) 

**Raw spot rate with no smoothing:**
`SwETHPriceOracle.getAssetPrice()` returns `ISwETH(swETHAddress).getRate()` directly, a spot value that can transiently dip during rebase or rate-update windows. [2](#0-1) 

**Asymmetric access control — the root cause:**
For a price *increase* above the threshold, a non-manager caller gets a hard revert (`PriceAboveDailyThreshold`): [3](#0-2) 

For a price *decrease* below the threshold, there is **no role check** — the circuit-breaker fires unconditionally for any caller, pausing all three contracts and returning: [4](#0-3) 

**`highestRsethPrice` is the all-time high, never reset downward:**
It only ever increases, so over time even a modest transient dip in swETH rate can push `newRsETHPrice` below `highestRsethPrice` by more than `pricePercentageLimit`. [5](#0-4) 

**Exploit flow:**
1. Attacker observes a block where `swETH.getRate()` is transiently low enough that the computed `newRsETHPrice < highestRsethPrice` by more than `pricePercentageLimit`.
2. Attacker calls `oracle.updateRSETHPrice()` with no privileges and no capital.
3. `isPriceDecreaseOffLimit` evaluates to `true`; `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()` are called in the same transaction.
4. Protocol is frozen. `updateRSETHPrice()` is now blocked by `whenNotPaused`, so the attacker cannot self-recover. Admin must manually call `unpause()` on each contract.
5. After admin unpauses, the attacker can repeat at the next rebase window.

## Impact Explanation

When the circuit-breaker fires: `LRTDepositPool` is paused (all deposits revert), `LRTWithdrawalManager` is paused (all withdrawal initiations revert), and `LRTOracle` itself is paused. No funds are lost, but all user-facing deposit and withdrawal flows are frozen until admin intervention. This is **Temporary freezing of funds** — a Medium-severity impact within the allowed scope.

## Likelihood Explanation

- `updateRSETHPrice()` requires zero privileges and zero capital.
- swETH's `getRate()` is a documented spot value that can dip transiently during rebase windows; no oracle manipulation is required.
- The attacker only needs to observe the on-chain rate and submit a transaction at the right block — no front-running needed.
- `pricePercentageLimit` is admin-configurable; even a 1% limit (`1e16`) is realistic and achievable by a natural transient dip.
- The attack is repeatable after each admin unpause.

## Recommendation

1. **Add a role check before the downside pause**, mirroring the existing upside check: only allow the circuit-breaker to fire when `msg.sender` holds `MANAGER` role; for unprivileged callers, emit an event or revert instead of pausing.
2. **Restrict `updateRSETHPrice()` to a privileged role** (e.g., `MANAGER` or a dedicated keeper), or add a per-caller cooldown.
3. **Use a TWAP or rate-smoothing layer** in `SwETHPriceOracle` instead of passing `swETH.getRate()` raw.
4. **Consider comparing against `previousPrice` instead of `highestRsethPrice`** for the downside circuit-breaker, so only a drop relative to the last recorded price — not the all-time high — triggers a pause.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Fork test at a block where swETH.getRate() is transiently below
// highestRsethPrice by > pricePercentageLimit (e.g., immediately after a rebase)

contract GriefTest is Test {
    LRTOracle oracle = LRTOracle(ORACLE_ADDR);
    LRTDepositPool pool = LRTDepositPool(POOL_ADDR);
    LRTWithdrawalManager wm = LRTWithdrawalManager(WM_ADDR);

    function testGrief() external {
        // Pre-condition: pricePercentageLimit > 0 (e.g. 1e16 = 1%)
        // Pre-condition: swETH.getRate() at this block causes newRsETHPrice
        //                to be below highestRsethPrice by > pricePercentageLimit

        assertFalse(pool.paused());
        assertFalse(wm.paused());
        assertFalse(oracle.paused());

        // Unprivileged caller triggers the circuit-breaker
        vm.prank(address(0xdead));
        oracle.updateRSETHPrice();

        // Protocol is now frozen — no funds lost, but all flows blocked
        assertTrue(pool.paused());
        assertTrue(wm.paused());
        assertTrue(oracle.paused());

        // Rate recovers next block, but protocol stays paused — requires admin action
        vm.roll(block.number + 1);
        assertTrue(pool.paused());
    }
}
```

No special privileges, no capital, and no front-running are required — only knowledge of a block where swETH's spot rate is transiently below the protocol's all-time-high rsETH price by more than `pricePercentageLimit`.
