Audit Report

## Title
Unvalidated `l2bridge` Parameter Allows Gas-Bomb Block Stuffing — (`contracts/bridges/UnichainMessenger.sol`)

## Summary
`UnichainMessenger.sendETHToL1ViaBridge` is a public, permissionless function that accepts a fully caller-controlled `l2bridge` address and makes an unbounded-gas external call to it. An attacker can supply a gas-consuming contract as `l2bridge` with only 1 wei, saturating Unichain blocks and delaying legitimate `BRIDGER_ROLE` bridge operations.

## Finding Description
`UnichainMessenger.sendETHToL1ViaBridge` imposes no access control and no validation on the `l2bridge` parameter: [1](#0-0) 

The only guard is `msg.value != value`. The external call on line 26 does **not** cap gas at `DEFAULT_GAS_LIMIT`; that constant is passed as the `_minGasLimit` *parameter* to `bridgeETHTo` (an L1 hint), not as an EVM gas stipend. Under EIP-150, the call forwards `(63/64) * gasleft()` to the callee: [2](#0-1) 

A contract implementing `IUnichainMessenger.bridgeETHTo` with an infinite loop will consume nearly the entire block gas budget per transaction. The pool contracts that legitimately call this messenger (`RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge`) are gated by `onlyRole(BRIDGER_ROLE)` and pass a stored, validated `l2Bridge` state variable: [3](#0-2) 

But `UnichainMessenger` itself has no such restriction — any EOA or contract can call it directly with an arbitrary `l2bridge`.

## Impact Explanation
An attacker can repeatedly call `sendETHToL1ViaBridge{value: 1 wei}(gasBombAddress, anyTarget, 1)` to saturate Unichain blocks. On OP Stack chains, gas prices are extremely low, making sustained block stuffing economically viable. This delays or prevents the `BRIDGER_ROLE` operator from executing `bridgeAssetsViaNativeBridge`, temporarily disrupting the ETH bridging flow from Unichain to L1. This matches the allowed impact: **Low — Block stuffing**.

## Likelihood Explanation
The attack requires only deploying a contract with a gas-consuming `bridgeETHTo` implementation (trivial) and calling the public function with 1 wei. No role, approval, or special state is required. The function is unconditionally callable by any address on Unichain, and the cost per block-stuffing transaction on OP Stack is negligible.

## Recommendation
Restrict `sendETHToL1ViaBridge` to only be callable by authorized pool contracts via an `onlyRole` check, or make `l2bridge` an immutable set at deployment rather than a caller-supplied argument — mirroring how pool contracts already store `l2Bridge` as a validated state variable. Alternatively, add an explicit allowlist for the `l2bridge` parameter.

## Proof of Concept
```solidity
contract GasBomb {
    function bridgeETHTo(address, uint32, bytes memory) external payable {
        uint256 i;
        while (true) { unchecked { i++; } }
    }
    receive() external payable {}
}

contract PoC {
    function exploit(address messenger) external payable {
        GasBomb bomb = new GasBomb();
        UnichainMessenger(messenger).sendETHToL1ViaBridge{value: 1}(
            address(bomb), address(0xdead), 1
        );
    }
}
```
Deploy on a local Unichain fork, call `exploit` with `value = 1 wei`, and observe the transaction consuming the full block gas limit. The `BRIDGER_ROLE` operator's subsequent `bridgeAssetsViaNativeBridge` call cannot be included until the attacker stops stuffing blocks.
