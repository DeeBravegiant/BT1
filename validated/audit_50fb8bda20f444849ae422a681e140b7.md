Audit Report

## Title
Reward Tokens Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. Because `rewardPerToken()` short-circuits and returns `rewardPerTokenStored` unchanged during a zero-stake window, the elapsed time is silently discarded when the next user action occurs, permanently freezing any rewards that accrued during that window. There is no admin rescue or sweep function to recover these tokens.

## Finding Description
The `updateReward` modifier performs two independent assignments on every call:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // (A) — no-op when totalKernelStaked == 0
    updatedAt = lastTimeRewardApplicable();    // (B) — always advances
    ...
}
``` [1](#0-0) 

Assignment (A) calls `rewardPerToken()`, which returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    ...
}
``` [2](#0-1) 

Assignment (B) always advances `updatedAt` regardless. When the next user action occurs after a zero-stake gap, the `rewardRate * (lastTimeRewardApplicable() - updatedAt)` term in `rewardPerToken()` starts from the new `updatedAt`, permanently erasing the rewards that accrued during the empty window.

The `initiateWithdrawal` function immediately reduces `totalKernelStaked`, making it trivially possible for the last staker to drain it to zero: [3](#0-2) 

The `notifyRewardAmount` guard only prevents *starting* a new reward period with zero stake; it does not prevent stake from draining to zero after a period has already started: [4](#0-3) 

The contract's own NatSpec explicitly acknowledges this behavior but relies solely on an off-chain operational promise: [5](#0-4) 

There is no on-chain enforcement of the invariant, and no admin-callable sweep or recovery function exists anywhere in the contract.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

Reward tokens transferred into the contract via `notifyRewardAmount` and accrued during any zero-stake window are permanently unclaimable. The `rewardsToken` balance of the contract will exceed the sum of all claimable `rewards[user]` values by exactly the amount frozen during the empty period. No path exists to recover these tokens.

## Likelihood Explanation
The trigger is a normal, permissionless user action (`initiateWithdrawal`). It requires only that the last remaining staker withdraws while a reward period is active — a realistic scenario during low-participation periods, market stress, or protocol migration. No attacker capability is required; any ordinary user can trigger this unintentionally. The condition is repeatable across multiple reward periods.

## Recommendation
Decouple the `updatedAt` advancement from the zero-stake short-circuit. Only advance `updatedAt` when `totalKernelStaked > 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This preserves the unaccumulated time so that when stakers return, the rewards for the empty window are distributed to them rather than frozen. Alternatively, add an admin-callable `recoverUnallocatedRewards()` function that sweeps any surplus `rewardsToken` balance exceeding the sum of all pending rewards.

## Proof of Concept
**Setup:** `duration = 7 days`, `rewardRate = 1000 / 7 days ≈ 1.653e-3 tokens/sec`.

1. **T0** — Admin calls `notifyRewardAmount(1000)`. `finishAt = T0 + 7 days`, `updatedAt = T0`.
2. **T0** — Alice calls `stake(100)`. `totalKernelStaked = 100`.
3. **T0 + 1 day** — Alice calls `initiateWithdrawal(100)`.
   - `updateReward(Alice)` fires: `rewardPerTokenStored` accumulates 1 day of rewards; `updatedAt = T0 + 1 day`. Alice's earned rewards are saved.
   - `totalKernelStaked = 0`.
4. **T0 + 2 days** — Bob calls `stake(100)`.
   - `updateReward(Bob)` fires: `rewardPerToken()` returns `rewardPerTokenStored` unchanged (zero-stake branch). **`updatedAt` is advanced to `T0 + 2 days`**, silently discarding 1 day of rewards (~143 tokens).
   - `totalKernelStaked = 100`.
5. **T0 + 3 days** — Bob calls `getReward()`.
   - Bob earns rewards only for the window `T0+2 → T0+3` (1 day), not 2 days.
   - The ~143 tokens for the window `T0+1 → T0+2` are permanently frozen in the contract with no recovery path.

**Foundry test plan:** Deploy `KernelDepositPool`, call `notifyRewardAmount`, have Alice stake and then fully withdraw, warp 1 day, have Bob stake, warp another day, call `getReward` for Bob, and assert that `rewardsToken.balanceOf(contract) > rewards[Bob]` by the frozen amount. Confirm no function exists to recover the surplus balance.
