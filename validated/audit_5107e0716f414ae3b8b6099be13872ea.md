Audit Report

## Title
`updateRSETHPrice()` DoS via Missing Oracle for Newly Added Supported Asset — (`contracts/LRTOracle.sol`)

## Summary
`LRTOracle._getTotalEthInProtocol()` unconditionally calls `getAssetPrice(asset)` for every address in `supportedAssetList`. If any asset lacks a configured oracle (`assetPriceOracle[asset] == address(0)`), the call reverts via the `onlySupportedOracle` modifier. Because `LRTConfig.addNewSupportedAsset()` (gated by `TIME_LOCK_ROLE`) and `LRTOracle.updatePriceOracleFor()` (gated by `onlyLRTAdmin`) are separate transactions with different role requirements, a window exists between them during which every call to the public `updateRSETHPrice()` reverts, freezing `rsETHPrice` at its last cached value.

## Finding Description
`_getTotalEthInProtocol()` (lines 331–349, `LRTOracle.sol`) iterates over every address returned by `lrtConfig.getSupportedAssetList()` and calls `getAssetPrice(asset)` (line 339) for each one. `getAssetPrice` (line 156) is decorated with `onlySupportedOracle(asset)` (lines 40–44), which unconditionally reverts with `AssetOracleNotSupported` when `assetPriceOracle[asset] == address(0)`.

`LRTConfig.addNewSupportedAsset()` (lines 99–101) delegates to `_addNewSupportedAsset()` (lines 106–118), which sets `isSupportedAsset[asset] = true` and appends the asset to `supportedAssetList` with no check that a corresponding oracle entry exists in `LRTOracle`. The oracle is configured separately via `updatePriceOracleFor()` (lines 113–119), which requires `onlyLRTAdmin` — a distinct role from `TIME_LOCK_ROLE`.

Exploit path:
1. `TIME_LOCK_ROLE` calls `LRTConfig.addNewSupportedAsset(newAsset, limit)` — `newAsset` is now in `supportedAssetList`.
2. `LRTAdmin` has not yet called `LRTOracle.updatePriceOracleFor(newAsset, oracle)` — `assetPriceOracle[newAsset] == address(0)`.
3. Any unprivileged caller invokes `LRTOracle.updateRSETHPrice()`.
4. Execution reaches `_getTotalEthInProtocol()` → `getAssetPrice(newAsset)` → `onlySupportedOracle` → revert `AssetOracleNotSupported`.
5. `rsETHPrice` is frozen at its last stored value for the entire duration of the window.

No existing guard prevents this: `updateRSETHPrice()` has only a `whenNotPaused` check, and `_getTotalEthInProtocol()` performs no zero-address skip on the oracle mapping.

## Impact Explanation
While `rsETHPrice` is stale, `getRsETHAmountToMint()` (deposits) and `getExpectedAssetAmount()` (withdrawals) compute exchange rates against an outdated TVL figure. If true TVL has grown since the last successful update, depositors receive more rsETH than warranted, diluting existing holders' yield. The contract fails to deliver its promised accurate exchange rate for the duration of the DoS window. This matches the allowed **Low** impact: *contract fails to deliver promised returns, but doesn't lose value*.

## Likelihood Explanation
Asset onboarding is a standard operational procedure. The two required calls are controlled by different roles and must be separate transactions, making the window inherent to the protocol's design rather than an edge case. Any deployment that follows the natural multi-step onboarding sequence will exhibit this window. The window closes only when `LRTAdmin` executes `updatePriceOracleFor`, but until then every call to `updateRSETHPrice()` by any caller reverts.

## Recommendation
Enforce atomicity between asset registration and oracle configuration. Either:
- Require that `assetPriceOracle[asset]` is already set in `LRTOracle` before `addNewSupportedAsset` completes (cross-contract check), or
- Skip assets with no oracle in `_getTotalEthInProtocol()` instead of reverting:

```solidity
// contracts/LRTOracle.sol — _getTotalEthInProtocol()
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    address asset = supportedAssets[assetIdx];
    if (assetPriceOracle[asset] == address(0)) {
        unchecked { ++assetIdx; }
        continue;
    }
    uint256 assetER = getAssetPrice(asset);
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    totalETHInProtocol += totalAssetAmt.mulWad(assetER);
    unchecked { ++assetIdx; }
}
```

## Proof of Concept
1. Deploy protocol with `stETH` and `ETHx` in `supportedAssetList`, both with oracles configured.
2. `TIME_LOCK_ROLE` calls `LRTConfig.addNewSupportedAsset(newLST, 100_000 ether)`.
3. Confirm `LRTOracle.assetPriceOracle[newLST] == address(0)` (oracle not yet set).
4. Call `LRTOracle.updateRSETHPrice()` from any EOA — transaction reverts with `AssetOracleNotSupported`.
5. Confirm `rsETHPrice` is unchanged from its pre-step-2 value; all deposit/withdrawal exchange rates are now computed against a stale price.
6. Call `LRTOracle.updatePriceOracleFor(newLST, oracleAddr)` as `LRTAdmin` — `updateRSETHPrice()` now succeeds.
