### Title
Single Byzantine Attested Participant Can Fabricate AppPublicKey CKD Responses and Preempt All Fan-Out Yields — (`crates/contract/src/lib.rs`, `crates/contract/src/pending_requests.rs`)

### Summary

`respond_ckd` performs no cryptographic verification of the `CKDResponse` when the request uses the `AppPublicKey` variant. A single Byzantine attested participant can call `respond_ckd` with an arbitrary fabricated response, which `resolve_yields_for` immediately uses to drain the entire fan-out queue. All callers who submitted duplicate requests receive attacker-controlled key material, and every subsequent honest `respond_ckd` call fails with `RequestNotFound`.

### Finding Description

**Missing verification for `AppPublicKey` variant.**

In `respond_ckd`, the match on `app_public_key` is:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKeyPV` arm runs a BLS12-381 pairing check (`ckd_output_check`) that cryptographically binds the response to the network public key and the app's key pair. The `AppPublicKey` arm is an empty no-op — any `CKDResponse` is accepted unconditionally.

**Single-participant authorization.**

The only gate before the match is:

```rust
self.assert_caller_is_attested_participant_and_protocol_active();
``` [2](#0-1) 

This requires the caller to be *one* attested participant, not a threshold of them. There is no on-chain aggregation or quorum check.

**`resolve_yields_for` drains the entire queue atomically.**

```rust
let resumed = requests
    .remove(request)          // removes the entire Vec<YieldIndex>
    .unwrap_or_default()
    .into_iter()
    .map(|YieldIndex { data_id }| {
        env::promise_yield_resume(&data_id, response_bytes.clone());
    })
    .count();

if resumed > 0 { Ok(()) } else { Err(InvalidParameters::RequestNotFound.into()) }
``` [3](#0-2) 

The map entry is removed before any yield is resumed. After the first `respond_ckd` call completes, the queue is gone. Every subsequent honest call finds `resumed == 0` and returns `RequestNotFound`.

**Attack sequence:**

1. Up to `MAX_PENDING_REQUEST_FAN_OUT` (128) callers submit identical `AppPublicKey` CKD requests (same `app_public_key`, `app_id`, `domain_id`). Each is queued as a `YieldIndex` in `pending_ckd_requests`.
2. A single Byzantine attested participant observes the request on-chain and immediately calls `respond_ckd` with fabricated `big_c` and `big_y` values — arbitrary G1 points of the attacker's choosing.
3. No pairing check fires. `resolve_yields_for` removes the entire queue and calls `promise_yield_resume` for all 128 yields with the fabricated payload.
4. All 128 callers receive the attacker-chosen `CKDResponse`.
5. Every honest node's subsequent `respond_ckd` call returns `Err(RequestNotFound)`.

### Impact Explanation

The attacker fully controls the `CKDResponse` (`big_c`, `big_y`) delivered to every caller. Callers use this material as their derived key output. An attacker who chooses `big_c = G1 * s` for a known scalar `s` learns the "derived secret" for every affected caller, enabling key recovery or forged operations that depend on the derived key. This is unauthorized key derivation output delivered without threshold authorization, matching the Critical scope: *bypass of threshold-signature requirements or unauthorized access to secret material that materially enables forgery or secret recovery*.

### Likelihood Explanation

The race is trivially won: the attacker only needs to submit one transaction before any honest node does. On NEAR, block times are ~1 second and the attacker can monitor the mempool or on-chain state. No threshold collusion is required — a single Byzantine participant suffices. The `AppPublicKey` variant is the legacy/simpler path and is actively used (it is the default deserialization target per the interface types).

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` arm, or require a threshold quorum of `respond_ckd` calls before resolving yields (analogous to how signing aggregates shares off-chain before a single on-chain submission). At minimum, the `AppPublicKey` arm must not be a silent no-op when the response carries secret key material.
