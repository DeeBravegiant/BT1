Audit Report

## Title
Unconditional Zero-Fee `safeTransfer` to Treasury Permanently Freezes Claims for Zero-Transfer-Reverting Tokens — (`contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` even when `feeInBPS == 0`, producing a zero-value transfer. ERC-20 tokens that revert on zero-value transfers (a known class of non-standard but deployed tokens) will cause every `claim()` call to revert, permanently preventing all users from claiming their merkle-distributed yield. The sibling contract `KernelTop100MerkleDistributor.sol` already applies the correct guard, confirming the fix is known to the codebase.

## Finding Description
In `MerkleDistributor.sol`, `feeInBPS` is validated only with an upper-bound check (`> MAX_FEE_IN_BPS`), making `feeInBPS = 0` an explicitly valid configuration both at initialization and via `setFeeInBPS(0)`. When `claim()` executes with `feeInBPS == 0`:

1. `fee = (claimableAmount * 0) / 10_000 = 0` (line 138)
2. `IERC20(token).safeTransfer(account, amountToSend)` succeeds (line 141)
3. `IERC20(token).safeTransfer(protocolTreasury, 0)` is called unconditionally (line 144)

For any token that reverts on zero-value transfers, step 3 reverts the entire transaction. Because the state update (`userClaims[account]`) occurs at lines 134–135 before the transfers, the revert rolls back the state — but the call will always revert again on retry, making the claim permanently inaccessible. The `token` address is freely configurable via `setToken()`, and zero-transfer-reverting tokens are a documented class of non-standard ERC-20s in production. `KernelTop100MerkleDistributor.sol` lines 332–334 already demonstrates the correct pattern: `if (fee > 0) { kernel.safeTransfer(protocolTreasury, fee); }`.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** All users of a `MerkleDistributor` instance configured with `feeInBPS = 0` and a zero-transfer-reverting token are completely unable to claim their rewards. The contract holds their tokens but every `claim()` call reverts. Setting `feeInBPS` back to non-zero would unblock claims but would incorrectly charge users a fee they were never owed, constituting a separate accounting violation. This matches the allowed impact: *Medium — Permanent freezing of unclaimed yield*.

## Likelihood Explanation
- `feeInBPS = 0` requires no special exploit — it is a valid owner-set configuration with no lower-bound guard.
- `token` is configurable to any ERC-20; zero-transfer-reverting tokens exist in production.
- Any unprivileged user calling `claim()` triggers the revert — no special role, timing, or front-running is required.
- The combination is realistic: a protocol deploying this distributor for a zero-fee distribution with a strict ERC-20 token hits this unconditionally.

## Recommendation
Guard the treasury transfer with a zero-check, mirroring the pattern already used in `KernelTop100MerkleDistributor.sol`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy `MerkleDistributor` with a token that reverts on zero-value `transfer()` calls and `feeInBPS = 0`.
2. Owner calls `setMerkleRoot(root)` with a leaf for `alice` at `cumulativeAmount = 100e6`.
3. Alice calls `claim(1, alice, 100e6, proof)`.
4. Line 138: `fee = (100e6 * 0) / 10_000 = 0`.
5. Line 141: `safeTransfer(alice, 100e6)` succeeds.
6. Line 144: `safeTransfer(protocolTreasury, 0)` reverts (token rejects zero-value transfer).
7. Entire transaction reverts. Alice's claim state is rolled back. Every subsequent call by Alice reverts identically — her tokens are permanently frozen in the contract.
