### Title
Congestion Admission Bypass via Epoch-Layout Mismatch in `spice_shard_congestion_info` During Resharding Transition — (`chain/client/src/rpc_handler.rs`, `chain/chain/src/spice/chunk_application.rs`)

### Summary
During the Spice resharding transition window — when the consensus head is in the new epoch but the last certified block is still in the old epoch — `process_tx_internal` computes `receiver_shard` using the **new** shard layout while passing the **old-epoch** certified header to `spice_shard_congestion_info`. The function constructs a `ShardUId` with the new layout's version and looks it up against `ChunkExtra` entries stored under the old layout's version. The lookup fails; the fallback also fails because the child shard ID did not exist in the old epoch. The function returns `None`, and `validate_tx` skips the `ShardCongested`/`ShardStuck` rejection entirely.

### Finding Description
`process_tx_internal` derives `shard_layout` from the current epoch (line 177) and `receiver_shard` from that layout (lines 178–179). It then calls `spice_shard_congestion_info` with that same `shard_layout` but with `certified_header` from the previous epoch (line 185, 193–198). Inside `spice_shard_congestion_info`, `ShardUId::from_shard_id_and_layout(shard_id, shard_layout)` embeds the **new** layout version into the key (line 273). The `get_chunk_extra` call at the old block hash with the new-version key finds nothing. The fallback `get_execution_result_from_store` also finds nothing because the child shard ID is absent from the old epoch's execution results. The function returns `None`. Back in `validate_tx`, the `if let Some(congestion_info)` guard is not entered, so no `ShardCongested` or `ShardStuck` error is produced.

### Impact Explanation
An unprivileged user can submit transactions to a newly created child shard (post-resharding) during the certification-lag window and receive `ValidTx` instead of `InvalidTx(ShardCongested)`. The transaction enters the mempool and is forwarded to chunk producers. This bypasses the RPC-level congestion admission gate, which is the primary mechanism for protecting congested shards from new transaction load at the network edge.

### Likelihood Explanation
The scenario requires: (a) Spice enabled, (b) a resharding epoch boundary, (c) the node's last certified block still in the old epoch (a normal Spice operating condition). All three are expected production states. No special privileges are needed — any user with RPC access can submit a transaction to a receiver account that maps to a child shard in the new layout. The window lasts for the duration of the certification lag (potentially multiple blocks).

### Recommendation
In `process_tx_internal`, derive the shard layout from the **certified header's epoch** when calling `spice_shard_congestion_info`, and translate `receiver_shard` accordingly (using the parent shard mapping if the certified epoch predates the resharding). Alternatively, `spice_shard_congestion_info` should accept both the current and certified epoch's shard layouts and perform the parent-shard translation internally when they differ.

### Proof of Concept
Extend `test-loop-tests/src/tests/spice/resharding.rs`:
1. Set up a resharding boundary with `execution_delay = 2` (as in the existing test).
2. After the new epoch starts, verify via `get_last_certified_block_header` that the certified block is still in the old epoch.
3. Mark the child shard as congested by injecting max `delayed_receipts_gas` into its `ChunkExtra`.
4. Submit a transaction whose `receiver_id` maps to the congested child shard in the new layout.
5. Assert the RPC returns `ValidTx` (bypass) rather than `InvalidTx(ShardCongested)`.

The existing test at lines 76–87 already confirms the prerequisite: the first block of the resharded epoch has its last certified block in the previous epoch with a different shard layout. [6](#0-5) [7](#0-6) [3](#0-2)

pub fn spice_shard_congestion_info(
    chain_store: &ChainStoreAdapter,
    shard_layout: &ShardLayout,
    block_header: &BlockHeader,
    shard_id: ShardId,
) -> Option<ExtendedCongestionInfo> {
    let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, shard_layout);
    let chunk_store = chain_store.chunk_store();
    let congestion_info =
        if let Ok(chunk_extra) = chunk_store.get_chunk_extra(block_header.hash(), &shard_uid) {
            chunk_extra.congestion_info()
        } else {
            get_execution_result_from_store(chain_store, block_header.hash(), shard_id)?
                .chunk_extra
                .congestion_info()
        };
    Some(ExtendedCongestionInfo::new(congestion_info, 0))
}
