### Title
Solana `deploy_token` Bypasses Pause State — Missing `DEPLOY_TOKEN_PAUSED` Flag - (File: `solana/programs/bridge_token_factory/src/lib.rs`)

---

### Summary

The Solana bridge's `deploy_token` function has no pause check. The `ALL_PAUSED` bitmask in `constants.rs` only covers `INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED` — there is no `DEPLOY_TOKEN_PAUSED` bit defined or enforced. Every other chain in the codebase (EVM, Starknet, Aptos) defines and enforces a `PAUSE_DEPLOY_TOKEN` flag inside `deploy_token`. When the Solana bridge is fully paused in response to a security incident, an attacker holding a legitimately-obtained but not-yet-submitted MPC-signed `DeployTokenPayload` can still call `deploy_token` and create a new wrapped token on Solana, bypassing the operator's emergency halt.

---

### Finding Description

**Root cause — `constants.rs`:**

```rust
pub const INIT_TRANSFER_PAUSED: u8 = 1 << 0;
pub const FINALIZE_TRANSFER_PAUSED: u8 = 1 << 1;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED;
// No DEPLOY_TOKEN_PAUSED constant exists
``` [1](#0-0) 

**Root cause — `lib.rs` `deploy_token`:**

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
```

No pause check is present. Compare with `finalize_transfer` and `init_transfer` in the same file, which both gate on the pause bitmask: [2](#0-1) [3](#0-2) [4](#0-3) 

**Contrast with every other chain:**

- EVM `deployToken`: `whenNotPaused(PAUSED_DEPLOY_TOKEN)` modifier
- Starknet `deploy_token`: `assert(!_is_paused(@self, PAUSE_DEPLOY_TOKEN), 'ERR_DEPLOY_TOKEN_PAUSED')`
- Aptos `deploy_token`: `assert!((state.pause_flags & PAUSE_DEPLOY_TOKEN) == 0, E_DEPLOY_TOKEN_PAUSED)` [5](#0-4) [6](#0-5) [7](#0-6) 

The Solana `ALL_PAUSED` constant is structurally incomplete: it covers only the two transfer operations and has no bit reserved for token deployment. [1](#0-0) 

---

### Impact Explanation

When the Solana bridge is paused via `pause()` (which sets `ALL_PAUSED`), `deploy_token` remains callable. An attacker who holds a valid, legitimately-obtained MPC-signed `DeployTokenPayload` — one that was signed before the pause was triggered — can submit it to Solana and create a new wrapped token on-chain. This constitutes unauthorized creation of a wrapped bridge asset through deployment while the operator's emergency halt is active. If the pause was triggered precisely because of a vulnerability in the token-deployment path (e.g., a malicious or incorrectly-mapped token was being deployed), the pause provides no protection on Solana.

---

### Likelihood Explanation

MPC signatures for `deploy_token` are generated asynchronously: the NEAR side signs the payload and the result is returned to the caller, who then submits it to Solana in a separate transaction. There is a natural window between signature issuance and on-chain submission. An attacker who initiated a legitimate `log_metadata` → NEAR MPC signing flow before the pause can hold the signed payload and submit it after the bridge is paused. No privileged access, leaked key, or compromised guardian is required — only a valid signature obtained through the public protocol flow.

---

### Recommendation

1. Add a `DEPLOY_TOKEN_PAUSED` constant to `constants.rs`:
   ```rust
   pub const DEPLOY_TOKEN_PAUSED: u8 = 1 << 2;
   pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED | DEPLOY_TOKEN_PAUSED;
   ```
2. Add a pause guard to `deploy_token` in `lib.rs`, consistent with `finalize_transfer` and `init_transfer`:
   ```rust
   require!(
       ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
       error::ErrorCode::Paused
   );
   ```
3. Consider adding the same guard to `log_metadata`, which is also missing a pause check and is fully permissionless. [8](#0-7) 

---

### Proof of Concept

1. Attacker calls `log_metadata` on Solana for a target token (permissionless, no pause check).
2. NEAR side processes the Wormhole message; NEAR MPC signs a `DeployTokenPayload` for the token.
3. Attacker receives the signed payload but does not submit it yet.
4. A security incident is detected; operator calls `pause()` on Solana, setting `ALL_PAUSED = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED`.
5. Attacker calls `deploy_token(signed_payload)` on Solana. Because `deploy_token` has no pause check and `ALL_PAUSED` has no `DEPLOY_TOKEN_PAUSED` bit, the call succeeds.
6. A new wrapped token is created on Solana and registered in the bridge's token mapping, despite the operator's emergency halt — bypassing the intended state-machine restriction. [2](#0-1) [1](#0-0)
