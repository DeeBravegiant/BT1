Looking at the actual code in `crates/contract/src/tee/proposal.rs`:

### Title
Off-by-one in `valid_entries()` always includes the last-expired image hash in the allowed set — (`crates/contract/src/tee/proposal.rs`)

---

### Summary

`AllowedDockerImageHashes::valid_entries()` uses `rposition` to locate the rightmost entry whose grace period has elapsed, then returns the slice starting **at** that index. Because `rposition` returns the index of the expired entry itself, the slice `[cutoff_index..]` always includes that expired entry. Any node running the most-recently-expired image hash therefore passes TEE authorization indefinitely, regardless of how far past its deadline the current block time is.

---

### Finding Description

`valid_entries()` is the single gate that decides which image hashes are accepted for attestation verification:

```
fn valid_entries(&self, tee_upgrade_deadline_duration: Duration) -> Vec<AllowedMpcDockerImage> {
    let current_time = Timestamp::now();
    let cutoff_index = self
        .allowed_tee_proposals
        .iter()
        .rposition(|allowed_docker_image| {
            ...
            grace_period_deadline < current_time   // true  ⟺  entry is EXPIRED
        })
        .unwrap_or(0);

    self.allowed_tee_proposals
        .get(cutoff_index..)          // ← starts AT the expired entry
        .unwrap_or(&[])
        .to_vec()
}
``` [1](#0-0) 

`rposition` returns the index of the **last entry that satisfies the predicate**, i.e. the last expired entry. The slice `[cutoff_index..]` therefore begins with that expired entry and includes it in the returned "valid" set.

**Concrete two-entry trace:**

| Index | Hash | Deadline | State at `T_attack` |
|-------|------|----------|---------------------|
| 0 | `hash_A` | `T1 + grace` | EXPIRED (`T_attack > T1 + grace`) |
| 1 | `hash_B` | `T2 + grace` | VALID (`T_attack < T2 + grace`) |

`rposition` scans right-to-left:
- index 1: `deadline > current_time` → `false`
- index 0: `deadline < current_time` → `true` → returns `Some(0)`

`cutoff_index = 0` → `proposals[0..]` = `[hash_A, hash_B]`

`hash_A` is expired but is present in the returned slice.

The existing test `test_clean_expired` does **not** catch this because it sets the block timestamp 1 nanosecond past `hash_B`'s deadline as well, making **both** entries expired. In that case `rposition` returns `Some(1)`, `proposals[1..]` = `[hash_B]`, and the test passes — but it is testing the "always keep at least one" fallback, not the partial-expiry scenario. ### Impact Explanation

A node operator who refuses to upgrade after the grace period expires can continue submitting valid TEE attestations using the old image hash. Because `valid_entries()` always includes the last-expired hash, `attestation.verify_locally()` and `re_verify()` both accept it. The node remains an attested participant and retains the ability to call `respond` / `respond_ckd`, contributing threshold signature shares and CKD outputs. This directly violates the invariant that an expired image hash must not authorize any node for signing or key derivation.

---

### Likelihood Explanation

The condition is reached whenever the whitelist contains at least two entries and the oldest one has passed its deadline while the newest has not — the normal state during any rolling upgrade window. No special privileges are required beyond already being a participant (or being able to submit an attestation). The bug is permanent: it persists for the entire interval between the first and second entry's deadlines, which is bounded only by the gap between successive `vote_tee_proposal` calls.

---

### Recommendation

Change the slice start from `cutoff_index` to `cutoff_index + 1` when `rposition` returns `Some`, preserving the "always keep at least one" fallback only when the result would otherwise be empty:

```rust
let first_valid = self
    .allowed_tee_proposals
    .iter()
    .rposition(|entry| {
        entry.added.checked_add(tee_upgrade_deadline_duration)
            .map_or(true, |deadline| deadline < current_time)
    })
    .map(|i| i + 1)   // exclude the expired entry itself
    .unwrap_or(0);

// Always return at least the last entry even if everything is expired.
let start = if first_valid >= self.allowed_tee_proposals.len() {
    self.allowed_tee_proposals.len().saturating_sub(1)
} else {
    first_valid
};

self.allowed_tee_proposals.get(start..).unwrap_or(&[]).to_vec()
```

Add a dedicated test that sets the block time strictly between `T1 + grace` and `T2 + grace` and asserts that `hash_A` is **absent** from the result.

### Proof of Concept

```rust
#[test]
fn expired_hash_must_not_appear_in_valid_entries() {
    let mut allowed = AllowedDockerImageHashes::default();
    const GRACE: Duration = Duration::from_secs(10 * 24 * 60 * 60); // 10 days
    const NS: u64 = 1_000_000_000;

    // T1 = 1 s: insert hash_A
    testing_env!(VMContextBuilder::new().block_timestamp(NS).build());
    allowed.insert(NodeImageHash::from([1u8; 32]), GRACE);

    // T2 = 2 s: insert hash_B
    testing_env!(VMContextBuilder::new().block_timestamp(2 * NS).build());
    allowed.insert(NodeImageHash::from([2u8; 32]), GRACE);

    // Advance past hash_A's deadline but NOT hash_B's:
    // T_attack = 1 s + 10 days + 1 ns  (hash_A expired, hash_B still valid)
    let t_attack = NS + GRACE.as_nanos() as u64 + 1;
    testing_env!(VMContextBuilder::new().block_timestamp(t_attack).build());

    let hashes = allowed.get_image_hashes(GRACE);

    // hash_A must NOT be present — currently FAILS due to the off-by-one
    assert!(
        !hashes.contains(&NodeImageHash::from([1u8; 32])),
        "expired hash_A must not appear in valid_entries"
    );
    // hash_B must still be present
    assert!(hashes.contains(&NodeImageHash::from([2u8; 32])));
}
```
