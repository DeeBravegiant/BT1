Audit Report

## Title
Double-Truncation in `_updateRsETHPrice` Silently Decrements `rsETHPrice` on No-Reward Updates — (`contracts/LRTOracle.sol`)

## Summary
`_updateRsETHPrice` computes `previousTVL` via `mulWad` (floor) and then `newRsETHPrice` via `divWad` (floor). The round-trip `floor(floor(S·P / WAD) · WAD / S)` is provably `≤ P`, and strictly less than `P` whenever `(S·P) mod WAD ∈ (0, S)`. For any supply `S ≥ WAD` and a non-WAD-aligned price, this condition is always satisfied, so every no-reward oracle update silently decrements `rsETHPrice` by 1 wei. Because `updateRSETHPrice()` is public and unrestricted, any caller can repeatedly drive the price downward during periods of no reward accrual.

## Finding Description
Both `WadMath.mulWad` and `WadMath.divWad` truncate (floor) via OpenZeppelin's `mulDiv`: [1](#0-0) 

Inside `_updateRsETHPrice`, the two truncating operations are: [2](#0-1) [3](#0-2) 

When `totalETHInProtocol == previousTVL` (no new rewards), `protocolFeeInETH = 0` and:

```
newRsETHPrice = floor( floor(S·P / WAD) · WAD / S )
```

Let `r = (S·P) mod WAD`. Then `floor(S·P/WAD) = (S·P − r)/WAD`, so:

```
newRsETHPrice = floor( P − r/S )
```

- `r = 0` → result = P (stable)
- `0 < r < S` → result = **P − 1** (1-wei drop)

When `S ≥ WAD` (at least 1 rsETH), `r < WAD ≤ S`, so `r < S` is always true. The price decreases by 1 wei on every no-reward call whenever `(S·P) mod WAD ≠ 0`.

The downside protection at lines 270–291 only pauses the protocol if the price drop exceeds `pricePercentageLimit`; a 1-wei decrease never triggers it: [4](#0-3) 

The price is unconditionally written to storage afterward: [5](#0-4) 

`updateRSETHPrice()` has no access control — it is callable by anyone: [6](#0-5) 

## Impact Explanation
Every public call to `updateRSETHPrice()` with no new rewards silently reduces `rsETHPrice` by 1 wei. Existing rsETH holders redeem for slightly less ETH than they are owed. No principal is destroyed (the underlying ETH remains in the protocol), but the exchange rate drifts below the true value — a failure to deliver promised returns. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The condition is met under normal operation: any supply `S ≥ 1e18` combined with any non-integer-ETH price triggers it. `updateRSETHPrice()` is public with no role requirement. Periods of no reward accrual (e.g., between EigenLayer reward distributions) are routine. An adversary can call the function repeatedly at negligible cost to walk the price from `1e18 + k` down to `1e18` in `k` transactions.

## Recommendation
Add a guard after computing `newRsETHPrice` to prevent the price from decreasing when no fee was taken:

```solidity
// After computing newRsETHPrice (line 250):
if (protocolFeeInETH == 0 && newRsETHPrice < rsETHPrice) {
    newRsETHPrice = rsETHPrice; // preserve price when no reward accrued
}
```

Alternatively, add a `mulDivRoundUp` variant to `WadMath` and use it for the `newRsETHPrice` computation, or only write `rsETHPrice` when `newRsETHPrice >= rsETHPrice` (absent a genuine loss event).

## Proof of Concept
Concrete arithmetic (no fuzzing needed):

```
S = 3e18 + 1   (rsethSupply: 3 rsETH + 1 wei)
P = 1e18 + 1   (rsETHPrice:  1 ETH + 1 wei)

S·P = (3e18+1)·(1e18+1) = 3e36 + 4e18 + 1
r   = (3e36 + 4e18 + 1) mod 1e18 = 1
r < S  →  condition met

previousTVL        = floor((3e36 + 4e18 + 1) / 1e18) = 3e18 + 4
totalETHInProtocol = 3e18 + 4   (no new rewards)
protocolFeeInETH   = 0

newRsETHPrice = floor((3e18 + 4)·1e18 / (3e18 + 1))
              = floor((3e36 + 4e18) / (3e18 + 1))
              = 1e18   <   1e18 + 1  ✗
```

A Foundry invariant test can confirm: deploy with `rsethSupply = 3e18+1`, `rsETHPrice = 1e18+1`, mock `_getTotalEthInProtocol()` to return `previousTVL`, call `updateRSETHPrice()` repeatedly, and assert `rsETHPrice` never decreases — the invariant will break on the first call.
