### Title
`verify_foreign_transaction` Enforces All-Participant Intersection Instead of Signing-Threshold Coverage — (`File: crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which requires **every** active participant to have registered a chain. A single Byzantine participant below the signing threshold can register an empty chain list, causing the intersection to collapse and blocking **all** foreign-chain verification requests — even when a full signing threshold of honest nodes is available and ready to serve them.

---

### Finding Description

`verify_foreign_transaction` calls `get_supported_foreign_chains()` to decide whether to accept a request: [1](#0-0) 

`get_supported_foreign_chains()` computes the **strict intersection** of every active participant's registered chains: [2](#0-1) 

The filter condition `nodes_supporting_chain.is_superset(&active_participant_account_ids)` means a chain is only "supported" if **all** active participants have registered it. One participant registering an empty list (or omitting a chain) drops that chain — or all chains — from the supported set entirely.

A separate, threshold-aware implementation already exists: `update_available_chains_config_cache` in `ForeignChainsMetadata` counts per-chain supporters and compares against the signing threshold: [3](#0-2) 

The contract also exposes `get_available_foreign_chains()` (exercised in tests) which uses this threshold-based cache. However, `verify_foreign_transaction` never calls it — it calls the stricter all-participant intersection function instead.

The project's own design document explicitly identifies this as a bug: [4](#0-3) 

The analog to the external report is direct:

| External Report | NEAR MPC |
|---|---|
| `createMarketOrder` enforces **initial margin** (all collateral) even for reduce operations | `verify_foreign_transaction` enforces **all-participant intersection** even when threshold coverage suffices |
| Trader below initial margin but above maintenance margin is blocked from reducing exposure | Bridge user is blocked from submitting a valid foreign-tx request even when ≥ threshold nodes cover the chain |
| Fix: check maintenance margin for reduce operations | Fix: gate on `get_available_foreign_chains()` (threshold-based) instead of `get_supported_foreign_chains()` |

---

### Impact Explanation

**Medium.** A single Byzantine participant (strictly below the signing threshold) can call `register_foreign_chain_config` with an empty or chain-omitting payload. This causes `get_supported_foreign_chains()` to return an empty set (or exclude the target chain), and `verify_foreign_transaction` will reject every subsequent request for that chain with `ForeignChainNotSupported`. The entire foreign-chain bridge verification flow is halted for all users, breaking the production request-lifecycle invariant that the bridge should be serviceable whenever ≥ threshold honest nodes cover the chain. No network-level DoS or operator misconfiguration is required — the attack is a single on-chain transaction.

---

### Likelihood Explanation

Any one of the `n` active participants can execute this attack. The attacker needs only to be a registered participant (a role that is governance-controlled but finite in number). The call `register_foreign_chain_config` is a normal participant operation with no additional privilege requirement beyond being a participant: [5](#0-4) 

The attack is persistent: the empty registration remains until the participant re-registers with a non-empty list, giving the attacker sustained control over bridge liveness.

---

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which gates on `signing_threshold` coverage rather than unanimous participation:

```rust
// Before (all-participant intersection — one node can block all chains):
let supported_chains = self.get_supported_foreign_chains();

// After (threshold-based — tolerates up to n − threshold non-covering nodes):
let supported_chains = self.get_available_foreign_chains();
```

This matches the design intent documented in `docs/design/calculating-supported-foreign-chains.md` and the existing `update_available_chains_config_cache` implementation. [6](#0-5) 

---

### Proof of Concept

1. Deploy the contract with `n` participants and signing threshold `t < n`.
2. All `n` participants register Bitcoin as a supported chain. `get_supported_foreign_chains()` returns `{Bitcoin}`.
3. One Byzantine participant (participant `i`, where `i` is any single participant) calls `register_foreign_chain_config` with an empty `ForeignChainConfiguration`.
4. `get_supported_foreign_chains()` now returns `{}` because `nodes_supporting_chain.is_superset(&active_participant_account_ids)` fails — participant `i` is no longer in the Bitcoin supporters set.
5. Any user calling `verify_foreign_transaction` for Bitcoin receives `ForeignChainNotSupported` and the transaction reverts, even though `n − 1 ≥ t` honest nodes are fully capable of verifying and signing the request.

The existing sandbox test `register_foreign_chain_config__returns_empty_when_not_all_registered` demonstrates step 4 directly: [7](#0-6)

```rust
    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
```

```markdown
- **Available** is computed dynamically from the per-node config reports: `C` is available iff
  ≥ `signing_threshold` active participants cover `C`. `available ⊆ whitelisted` always.

`verify_foreign_transaction(C)` is **rejected unless `C` is available**: the contract fails fast
instead of accepting a request that can't reach the signing threshold and letting it time out. The
rejection is temporary — `C` becomes serviceable again as soon as enough nodes report coverage.

The legacy `get_supported_foreign_chains()` (the intersection rule) is **to be deprecated** in favour
of the two views above.
```
