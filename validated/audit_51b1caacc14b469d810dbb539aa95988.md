Audit Report

## Title
Missing Staleness and Validity Checks on Chainlink `latestRoundData()` Return Values - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `roundId`, `updatedAt`, and `answeredInRound`, performing no staleness, completeness, or sign checks on the returned price. A stale Chainlink price propagates directly into rsETH mint calculations via `LRTDepositPool.getRsETHAmountToMint()`, enabling incorrect rsETH issuance. The same repository already applies all three required checks in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the omission is unintentional.

## Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at L52, `getAssetPrice()` destructures `latestRoundData()` with a wildcard that captures only `price`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

None of the following checks are present:
- `price > 0` — a zero price silently returns 0, causing `getRsETHAmountToMint` to compute 0 rsETH for any deposit amount; a negative price causes an overflow revert.
- `updatedAt != 0` — a zero timestamp signals an incomplete round.
- `answeredInRound >= roundId` — a stale round is accepted as current.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` at L30–32 applies all three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The deposit path that consumes the unvalidated price is:

1. `LRTDepositPool.depositAsset()` (L99) → `_beforeDeposit()` (L648) → `getRsETHAmountToMint()` (L506)
2. `getRsETHAmountToMint` computes: `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`
3. `lrtOracle.getAssetPrice(asset)` (L156) delegates to `ChainlinkPriceOracle.getAssetPrice()` live on every deposit call.
4. `lrtOracle.rsETHPrice()` is a stored value updated discretely via `updateRSETHPrice()`.

Because `getAssetPrice()` is called live while `rsETHPrice` is a stored snapshot, a stale inflated Chainlink price inflates the numerator without a corresponding adjustment to the denominator, causing over-minting of rsETH. A stale deflated or zero price causes under-minting or minting of 0 rsETH (depositor loses tokens with no revert if `minRSETHAmountExpected == 0`).

The `minRSETHAmountExpected` slippage guard in `_beforeDeposit` (L667) protects only the depositor from receiving too little rsETH; it does not protect existing rsETH holders from dilution caused by over-minting.

## Impact Explanation

**Medium — Temporary freezing of funds / incorrect rsETH issuance.**

- **Zero price path**: `getAssetPrice` returns 0 → `getRsETHAmountToMint` returns 0 → deposit succeeds but mints 0 rsETH, permanently losing the depositor's LST tokens with no revert (if `minRSETHAmountExpected` is 0). This constitutes temporary freezing of depositor funds.
- **Stale inflated price path**: over-minting dilutes the backing per rsETH for all existing holders, constituting share mis-accounting. At sufficient scale this approaches theft of unclaimed yield from existing rsETH holders.

## Likelihood Explanation

Chainlink feeds update on heartbeat (up to 24 h for some feeds) or deviation threshold. During low-volatility windows, `updatedAt` can be many hours old and `answeredInRound < roundId` can hold. No attacker action is required — any unprivileged depositor calling `LRTDepositPool.depositAsset()` during such a window triggers the vulnerable path. The condition is passive, repeatable, and requires no privileged access.

## Recommendation

Apply the same pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(price > 0, "Chainlink price <= 0");
    require(updatedAt != 0, "Incomplete round");
    require(answeredInRound >= roundId, "Stale price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, apply equivalent checks in `RSETHPriceFeed.latestRoundData()` (L68–69) before multiplying the raw ETH/USD answer by the rsETH/ETH rate.

## Proof of Concept

1. Chainlink's stETH/ETH feed enters a heartbeat-only update window; `updatedAt` is 23 h old and `answeredInRound < roundId`.
2. Any user calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")`.
3. `_beforeDeposit` calls `getRsETHAmountToMint(stETH, amount)`.
4. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → `latestRoundData()` returns the stale price with no revert.
5. If the stale price is inflated: `rsethAmountToMint` is over-computed; the depositor receives excess rsETH, diluting all existing holders.
6. If the stale price is 0: `rsethAmountToMint = 0`; with `minRSETHAmountExpected = 0` the deposit succeeds, the LST is transferred in, and 0 rsETH is minted — the depositor's funds are effectively frozen.

**Foundry fork test plan**: fork mainnet, mock the stETH/ETH Chainlink aggregator to return `(roundId=2, price=0, startedAt=0, updatedAt=0, answeredInRound=1)`, call `depositAsset` with `minRSETHAmountExpected=0`, assert `rsETH.balanceOf(depositor) == 0` while `stETH.balanceOf(depositPool) > 0`.
