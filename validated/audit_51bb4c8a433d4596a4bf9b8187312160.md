### Title
Propose-Update Deposits Are Permanently Locked When Any Update Is Executed — (`crates/contract/src/update.rs`)

---

### Summary

The `propose_update` function requires participants to deposit NEAR tokens proportional to the size of the proposed update (up to ~40 NEAR for a full contract binary). When any update reaches threshold and `do_update` is called, **all** pending proposals are cleared without refunding the deposits paid by their proposers. The `UpdateEntry` struct does not track proposer account IDs, making refunds structurally impossible. There is no cancel or withdraw mechanism. Deposited NEAR is permanently locked in the contract's balance with no recovery path.

---

### Finding Description

In `crates/contract/src/update.rs`, `ProposedUpdates::do_update` clears every pending entry after executing the winning update: [1](#0-0) 

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
```

In `crates/contract/src/lib.rs`, `propose_update` accepts a deposit from the proposer, refunds only the excess, and keeps the required amount: [2](#0-1) 

The required deposit is calculated as `env::storage_byte_cost() * bytes_used`: [3](#0-2) [4](#0-3) 

The `UpdateEntry` struct stores no proposer account ID or deposit amount: [5](#0-4) 

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
```

This makes it **structurally impossible** to issue refunds even if `do_update` were patched, because the proposer's identity is not recorded anywhere in the stored entry.

The only other deposit-touching path is `remove_update_vote`, which removes a vote but does not touch the proposal entry or refund anything: [6](#0-5) 

There is no `cancel_proposal`, `withdraw_proposal`, or any other function that returns a deposit to a proposer.

---

### Impact Explanation

Every participant who calls `propose_update` loses their deposit permanently the moment any update is executed via `vote_update`. This includes:

- **The proposer of the winning update** — their entry is removed first, then all others are cleared; no refund is issued to anyone.
- **All other proposers** — their entries are bulk-cleared by `self.entries.clear()` with no refund.

For a full contract binary, the deposit can reach ~40 NEAR (confirmed by the sandbox test using `NearToken::from_near(40)`): [7](#0-6) 

The deposited NEAR remains in the contract's balance. The NEAR runtime returns freed storage cost to the contract's balance (not to the proposer), so the proposer's deposit is permanently absorbed by the contract. This breaks the production accounting invariant that deposits are refundable when the purpose for which they were paid is no longer served — directly analogous to the M-15 pattern where a governance action causes funds to become permanently inaccessible to their rightful owner.

**Impact class:** Medium — balance/accounting invariant violation causing permanent loss of participant-deposited NEAR funds.

---

### Likelihood Explanation

This occurs **every time** a governance update is executed, which is a routine and expected operation. In any scenario where multiple participants each propose their preferred update (a natural occurrence given that each participant may have a different view of the correct upgrade), all but one proposer will lose their full deposit. The loss is not contingent on any adversarial action — it is a guaranteed consequence of the normal governance lifecycle.

---

### Recommendation

1. Add `proposer: AccountId` and `deposit: NearToken` fields to `UpdateEntry` to record who paid and how much.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each.
3. Alternatively, expose a `cancel_proposal(id: UpdateId)` function that allows the original proposer to withdraw their proposal and receive a full refund, callable only by the proposer before the proposal is executed.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 40 NEAR deposit for a contract code update (e.g., a 1.5 MB binary). The deposit is accepted; only excess is refunded. A's 40 NEAR is now held by the contract.
2. Participant B calls `propose_update` with their own deposit for a different update.
3. Threshold participants call `vote_update` for B's proposal. `do_update` is triggered.
4. Inside `do_update`: B's entry is removed, then `self.entries.clear()` deletes A's entry. No refund is issued to A or B.
5. A's 40 NEAR is permanently locked in the contract's balance. A has no `cancel_proposal`, no `withdraw`, and no other recovery path. The `UpdateEntry` for A's proposal no longer exists, and even if it did, it never contained A's account ID. [8](#0-7) [9](#0-8)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }
