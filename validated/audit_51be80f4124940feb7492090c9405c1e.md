### Title
Duplicate `input_data_ids` in `ActionReceipt` Causes Permanent Receipt Lockup via Incorrect `PendingDataCount` — (`runtime/runtime/src/lib.rs`)

### Summary

`process_action_receipt` iterates over `action_receipt.input_data_ids()` without deduplication. When the same `data_id` appears more than once, `pending_data_count` is incremented for each occurrence, but only a single `TrieKey::PostponedReceiptId` trie entry is written (the second write silently overwrites the first with the same value). The resulting mismatch between the stored count and the number of unique pending entries permanently locks the postponed receipt in state — it can never execute and its `PendingDataCount` can never reach zero.

### Finding Description

**Root cause — `process_action_receipt`:** [1](#0-0) 

The loop increments `pending_data_count` once per occurrence of each `data_id` that is not yet in state, and writes `TrieKey::PostponedReceiptId { data_id }` for each. If `input_data_ids = [D, D]` and `D` is absent from state:

- Iteration 1: `pending_data_count = 1`, writes `PostponedReceiptId[D] = receipt_id`
- Iteration 2: `has_received_data` still returns `false` (the previous `set` wrote `PostponedReceiptId`, not `ReceivedData`), so `pending_data_count = 2`, overwrites `PostponedReceiptId[D] = receipt_id` (same key, same value — net effect: one entry)

The receipt is postponed with `PendingDataCount = 2` but only **one** `PostponedReceiptId[D]` entry.

**What happens when the DataReceipt for `D` arrives:** [2](#0-1) 

1. Reads `PendingDataCount = 2`; since `2 ≠ 1`, decrements to `1` and removes `PostponedReceiptId[D]`
2. Now `PendingDataCount = 1` but **no** `PostponedReceiptId[D]` remains
3. No future DataReceipt can decrement the count — the receipt is permanently stuck

`ReceivedData[D]` is also orphaned in the trie (saved but never consumed).

**Missing validation — `validate_action_receipt`:** [3](#0-2) 

The only check is that `input_data_ids.len() ≤ max_number_input_data_dependencies`. There is no uniqueness check, mirroring exactly the external report's pattern where only a count/length is validated.

**Secondary impact — `apply_action_receipt` with data already present:** [4](#0-3) 

If `D` is already in state when the receipt arrives, `apply_action_receipt` reads and removes `ReceivedData[D]` on the first iteration, then on the second iteration finds it absent and returns `StorageError::StorageInconsistentState`. This propagates as a `RuntimeError` from `apply`, causing chunk application to fail with a storage inconsistency error.

### Impact Explanation

- **Permanent receipt lockup / trie state corruption**: `TrieKey::PendingDataCount` and `TrieKey::PostponedReceiptId` entries are orphaned; the postponed receipt never executes. The affected account's cross-contract call is permanently frozen with no recovery path.
- **`StorageError` on chunk apply** (data-already-present path): chunk application returns an error, which is a runtime state mismatch that can cause consensus divergence between nodes depending on how the error is surfaced.
- **Corrupted state root**: the orphaned trie entries alter the state root for the affected shard, permanently embedding the corruption.

### Likelihood Explanation

An unprivileged user deploys a contract that calls `promise_and` (or equivalent promise-combination host functions) with the same promise index appearing twice. The NEAR VM logic passes the resulting `input_data_ids` — which will contain the same `data_id` twice — directly into the `ActionReceipt`. Neither the VM-level receipt construction nor `validate_action_receipt` rejects duplicate entries. This is a straightforward, low-cost attack requiring only a deployed contract and a single transaction.

### Recommendation

1. In `validate_action_receipt` (`runtime/runtime/src/verifier.rs`), add a uniqueness check on `input_data_ids` and return `ReceiptValidationError` if duplicates are found.
2. As a defense-in-depth measure, deduplicate `input_data_ids` at the start of `process_action_receipt` before computing `pending_data_count`.

### Proof of Concept

```
Deploy contract C on account "alice":
  fn call():
    let p = promise_create("bob", "noop", ...)
    // pass the same promise index twice to promise_and
    let combined = promise_and([p, p])
    promise_then(combined, "alice", "callback", ...)

Alice sends a transaction calling C::call().

Runtime produces ActionReceipt R for "alice::callback" with:
  input_data_ids = [data_id_D, data_id_D]   // same hash twice

process_action_receipt(R):
  pending_data_count = 2
  PostponedReceiptId[D] = R.receipt_id       // written twice, one entry

DataReceipt for D arrives:
  PendingDataCount[R] = 2 → 1
  PostponedReceiptId[D] removed

State after:
  PendingDataCount[R] = 1  (stuck forever)
  ReceivedData[D]          (orphaned)
  PostponedReceipt[R]      (orphaned, never executes)
```
