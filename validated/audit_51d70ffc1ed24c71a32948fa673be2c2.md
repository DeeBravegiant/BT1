Audit Report

## Title
Oracle-Price vs. Actual-ETH Mismatch Leaves Permanent Residual in `ethValueInWithdrawal`, Inflating `rsETHPrice` and Freezing Unclaimed Yield — (`contracts/LRTConverter.sol`)

## Summary

`transferAssetFromDepositPool` books the ETH value of stETH into `ethValueInWithdrawal` using the oracle price at transfer time. `_sendEthToDepositPool`, called from `claimStEth`, decrements `ethValueInWithdrawal` by the actual ETH received from Lido. When actual ETH is less than the booked oracle value — due to slashing or oracle drift over the multi-day withdrawal window — a non-zero residual is permanently stranded in `ethValueInWithdrawal`. This phantom value propagates through `getETHDistributionData` → `getTotalAssetDeposits` → `_getTotalEthInProtocol` → `rsETHPrice`, permanently inflating the price and freezing the over-counted yield in the accounting layer where it can never be claimed.

## Finding Description

**Step 1 — Booking at oracle price**

`transferAssetFromDepositPool` increments `ethValueInWithdrawal` by `stETHAmount × P_oracle` where `P_oracle` is the stETH/ETH oracle price at the moment of transfer:

```solidity
// LRTConverter.sol L140
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
```

**Step 2 — Partial decrement by actual ETH received**

`claimStEth` calls `_sendEthToDepositPool(address(this).balance)`, passing the actual ETH returned by Lido. The decrement logic is:

```solidity
// LRTConverter.sol L255-259
if (ethValueInWithdrawal > _amount) {
    ethValueInWithdrawal -= _amount;   // residual = booked - actual
} else {
    ethValueInWithdrawal = 0;
}
```

When `stETHAmount × P_oracle > actualETH`, the `if`-branch fires and a residual equal to `stETHAmount × P_oracle − actualETH` is left in `ethValueInWithdrawal` with no mechanism to clear it. `transferAssetToDepositPool` can decrement it, but only while stETH is still held in the converter — after `unstakeStEth` the stETH is gone, so that path is unavailable.

**Step 3 — Residual propagates to rsETHPrice**

`getETHDistributionData` reads `ethValueInWithdrawal` directly as `ethLyingInConverter`:

```solidity
// LRTDepositPool.sol L498-499
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
```

`getTotalAssetDeposits(ETH)` sums all components including `assetLyingInConverter`. `_getTotalEthInProtocol` calls `getTotalAssetDeposits` for every supported asset, so the phantom residual is included in `totalETHInProtocol`. `rsETHPrice` is then computed from this inflated total:

```solidity
// LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

**Step 4 — New depositors receive fewer rsETH**

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

An inflated `rsETHPrice` denominator means every depositor after the residual is created receives fewer rsETH than they are entitled to. The phantom ETH is double-counted: the real ETH was already sent to the deposit pool (correctly reflected in `address(depositPool).balance`), but `ethValueInWithdrawal` still counts it a second time.

**No mechanism to clear the residual**

There is no admin or operator function to reset `ethValueInWithdrawal`. Once the last `claimStEth` is processed and the residual remains, it is permanent.

## Impact Explanation

The residual is phantom ETH: the real ETH was already sent to the deposit pool, but `ethValueInWithdrawal` still counts it a second time. This permanently inflates `rsETHPrice` by `residual / rsethSupply`. Every depositor after the event receives proportionally fewer rsETH than they should. The over-counted yield can never be claimed by anyone — it is frozen in the accounting layer.

**Impact: Medium — Permanent freezing of unclaimed yield.**

## Likelihood Explanation

Lido withdrawal requests are finalized over a period of days to weeks. During that window:

- **Slashing**: A validator slashing event reduces the ETH/share ratio, so the actual ETH received at finalization is less than the oracle price at transfer time.
- **Oracle drift**: The stETH/ETH oracle price can move between `transferAssetFromDepositPool` and `claimStEth`. Even a 0.01% downward drift on a 10,000 stETH unstake leaves a 1 ETH residual.

Slashing is rare but non-zero; oracle drift over a multi-day window is routine. The vulnerability is triggered on any normal operator-driven unstake cycle where the oracle price at transfer time exceeds the actual ETH/stETH ratio at claim time. No malicious behavior is required — the operator is following the normal protocol flow.

## Recommendation

Replace the partial-decrement logic in `_sendEthToDepositPool` with a mechanism that tracks the booked value per withdrawal request and zeroes it out exactly when the corresponding claim is processed:

1. In `unstakeStEth`, record the booked ETH value for each Lido request ID in a `mapping(uint256 requestId => uint256 bookedValue)`.
2. In `claimStEth`, subtract exactly the booked value for that request ID from `ethValueInWithdrawal` (not the actual ETH received), and separately account for any slashing loss as a realized loss.
3. Alternatively, add an admin-callable `syncEthValueInWithdrawal()` that recomputes `ethValueInWithdrawal` from the set of still-pending Lido request IDs, allowing the operator to correct any residual after all requests are finalized.

## Proof of Concept

Foundry fork test on mainnet (`--fork-url $ETH_RPC`):

```solidity
// 1. Operator calls transferAssetFromDepositPool(stETH, 10_000e18)
//    Oracle price at this moment: 0.9995e18 ETH/stETH
//    => ethValueInWithdrawal = 10_000 * 0.9995e18 / 1e18 = 9_995e18 (booked)

// 2. Operator calls unstakeStEth(10_000e18)
//    Lido withdrawal request submitted (requestId = X)

// 3. Simulate slashing: Lido finalizes at 0.9985 ETH/stETH
//    => actualETH = 9_985e18

// 4. Operator calls claimStEth(X, hint)
//    _sendEthToDepositPool(9_985e18) is called
//    ethValueInWithdrawal (9_995e18) > _amount (9_985e18)
//    => ethValueInWithdrawal = 9_995e18 - 9_985e18 = 10e18  ← PERMANENT RESIDUAL

assertEq(converter.ethValueInWithdrawal(), 10e18);

// 5. No more stETH in converter; no further claims possible.
//    Residual persists forever.

// 6. rsETHPrice is inflated by 10e18 / rsethSupply.
//    New depositor sending 1 ETH receives:
//    rsethMinted = 1e18 * 1e18 / inflatedRsETHPrice  < fair amount
```

The residual equals `(P_oracle_at_transfer − P_actual_at_claim) × stETHAmount / 1e18` and inflates `rsETHPrice` by `residual / rsethSupply`, causing every subsequent depositor to receive proportionally fewer rsETH.
