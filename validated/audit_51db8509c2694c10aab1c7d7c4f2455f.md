Audit Report

## Title
Free Share Minting in Drained Bins Dilutes Existing LP Claims — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
In `LiquidityLib.addLiquidity`, when a bin has existing shares (`binTotalSharesVal > 0`) but both token balances have been drained to zero by swaps, the proportional share-price calculation yields zero for both token amounts. Share accounting is written unconditionally to storage while the token-payment callback is gated on a non-zero aggregate amount, so it is skipped entirely. An attacker mints an arbitrary number of shares at zero cost and later redeems them for real tokens when the price reverses and refills the bin, directly stealing from existing LPs.

## Finding Description
In `LiquidityLib.addLiquidity`, the per-bin amount calculation takes the proportional branch when `binTotalSharesVal > 0`:

```solidity
// LiquidityLib.sol L108–111
} else {
    amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
    amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
}
``` [1](#0-0) 

If both `token0BalanceScaled` and `token1BalanceScaled` are zero (bin fully drained by swaps), both `amount0Scaled` and `amount1Scaled` evaluate to `ceilDiv(0, binTotalSharesVal) = 0`. The subsequent balance updates are guarded by `if (amount0Scaled > 0)` / `if (amount1Scaled > 0)`, so `totalToken0ToAddScaled` and `totalToken1ToAddScaled` remain zero. However, share state is written unconditionally:

```solidity
// LiquidityLib.sol L120–121
binTotalShares[binIdx] = binTotalSharesVal + sharesToAdd;
positionBinShares[posKey] = newUserShares;
``` [2](#0-1) 

After the loop, the token-payment callback is only invoked when the aggregate owed amount is non-zero:

```solidity
// LiquidityLib.sol L144–155
if (amount0Added > 0 || amount1Added > 0) {
    IMetricOmmModifyLiquidityCallback(msg.sender)
        .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
    ...
}
``` [3](#0-2) 

Since both aggregates are zero, the callback is never fired and no tokens are transferred. The attacker's shares are committed to storage for free. Swap execution never modifies `_binTotalShares`, so a drained bin permanently retains its pre-drain share count, making this state reachable and persistent. [4](#0-3) 

## Impact Explanation
When the oracle price reverses and `zeroForOne` swaps refill the drained bin with token0, `removeLiquidity` computes each LP's entitlement proportionally to total shares. The attacker's free shares dilute every legitimate LP's claim. Existing LPs receive fewer tokens than they deposited; the attacker withdraws tokens they never paid for. This is a direct theft of LP principal with loss scaling linearly with the ratio of free shares minted to pre-existing shares. This meets the Critical/High threshold for direct loss of user principal under the allowed impact gate. [5](#0-4) 

## Likelihood Explanation
Bins being fully drained is a routine market event in any active pool — it occurs whenever the oracle price moves far enough that all liquidity in a bin is consumed by swaps. The attacker only needs to monitor on-chain `BinSwapped` events, detect when a bin's balance reaches zero while `_binTotalShares` remains positive, and call `addLiquidity` before the price reverses. No privileged access is required; `addLiquidity` is permissionless and accepts any `owner` address. The drained-bin state is persistent and detectable by any observer. [6](#0-5) 

## Recommendation
Add a guard in `LiquidityLib.addLiquidity` inside the `else` branch (when `binTotalSharesVal > 0`) that reverts if both computed amounts are zero:

```solidity
} else {
    amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
    amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
    if (amount0Scaled == 0 && amount1Scaled == 0) revert DrainedBin(binIdx);
}
```

This enforces the invariant that shares in a non-empty bin must always represent a non-zero token claim, mirroring the behavior of the `binTotalSharesVal == 0` branch which always computes a positive amount via `initialScaledToken0/1PerShareE18`. [7](#0-6) 

## Proof of Concept
1. Pool deployed with bins `[-1, 0, 1]`. Alice calls `addLiquidity` for bin `1` (above current price), depositing token0; `_binTotalShares[1] = 10_000`, `_binStates[1].token0BalanceScaled = X > 0`.
2. A series of `!zeroForOne` swaps fully consume bin `1`: `_binStates[1].token0BalanceScaled = 0`, `_binStates[1].token1BalanceScaled = 0`. `_binTotalShares[1]` remains `10_000` (swaps never touch share accounting).
3. Bob calls `addLiquidity(owner=Bob, salt=0, deltas={binIdxs:[1], shares:[10_000]}, ...)`.
   - `binTotalSharesVal = 10_000 > 0` → proportional branch.
   - `amount0Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - `amount1Scaled = ceilDiv(0 * 10_000, 10_000) = 0`.
   - `totalToken0ToAddScaled = 0`, `totalToken1ToAddScaled = 0` → callback skipped.
   - `_binTotalShares[1]` updated to `20_000`; `_positionBinShares[key(Bob,0,1)]` set to `10_000`. **No tokens paid.**
4. Price reverses; `zeroForOne` swaps refill bin `1` with 1_000 units of token0 (scaled).
5. Bob calls `removeLiquidity` for his 10_000 shares: receives `1_000 * 10_000 / 20_000 = 500` units of token0.
6. Alice removes her 10_000 shares: receives only 500 units instead of 1_000. Alice loses 500 units of token0 she legitimately deposited. [8](#0-7)
