Audit Report

## Title
Stale Cross-Chain rsETH/ETH Rate Enables Over-Minting of rsETH on L2 Pools — (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally without checking `lastUpdated`, despite the contract recording when the rate was last received via LayerZero. All L2 deposit pools (`RSETHPoolV3` and variants) call this function to compute rsETH mint amounts. When the rate is stale and lower than the true current L1 rate, depositors receive more rsETH shares than their ETH contribution warrants, diluting the accrued yield of all existing rsETH holders.

## Finding Description
`CrossChainRateReceiver` stores both `rate` and `lastUpdated`: [1](#0-0) 

`lzReceive` sets `lastUpdated = block.timestamp` on each rate update: [2](#0-1) 

However, `getRate()` returns `rate` with no staleness check against `lastUpdated`: [3](#0-2) 

`RSETHPoolV3.viewSwapRsETHAmountAndFee` calls `getRate()` and computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`: [4](#0-3) 

`deposit()` calls this function and mints rsETH directly to the depositor: [5](#0-4) 

`updateRate()` on `MultiChainRateProvider` is permissionless but requires the caller to pay LayerZero relay fees, with no on-chain enforcement of a maximum update interval: [6](#0-5) 

Since rsETH appreciates monotonically, any delay in propagating the updated rate means the stale rate is lower than the true rate. A lower denominator in `amountAfterFee * 1e18 / rsETHToETHrate` produces a larger `rsETHAmount`, minting excess shares to the depositor at the expense of existing holders' accrued yield.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders have accrued yield reflected in the rising L1 rate. When a depositor mints using a stale (lower) rate, they receive more rsETH shares than their ETH is worth at the true current rate. This excess dilutes the proportional claim of all existing holders on the underlying ETH, directly transferring their accrued yield to the new depositor. No privileged access is required; any user calling the public `deposit()` function during a staleness window exploits this automatically.

## Likelihood Explanation
**Medium.** LayerZero message delivery is not instantaneous, and `updateRate()` requires the caller to pay native gas and relay fees with no keeper or on-chain incentive. Rate updates are expected to be periodic (daily or on-demand), making staleness windows of hours to days realistic. Any depositor on any L2 pool automatically benefits from the stale rate during such a window, and the effect compounds across every deposit until the rate is refreshed.

## Recommendation
Add a configurable `maxStaleness` threshold to `CrossChainRateReceiver` and revert in `getRate()` if the rate is stale:

```solidity
uint256 public maxStaleness; // e.g., 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes all L2 pool deposits to revert when the oracle is stale, preventing over-minting until the rate is refreshed. Pair this with an automated keeper or on-chain incentive to call `updateRate()` regularly to avoid unnecessary deposit downtime.

## Proof of Concept
1. At T=0, L1 rsETH price is 1.05 ETH/rsETH. L2 `CrossChainRateReceiver` holds `rate = 1.05e18`, `lastUpdated = T`.
2. Staking rewards accrue on L1. At T=48h, the true rsETH price is 1.06 ETH/rsETH, but no one has called `updateRate()`. The L2 oracle still returns `1.05e18`.
3. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("")`.
4. `viewSwapRsETHAmountAndFee(100e18)` computes: `rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.238 rsETH`.
5. At the true rate of 1.06, the attacker should receive: `100e18 / 1.06e18 ≈ 94.340 rsETH`.
6. The attacker receives ~0.898 rsETH more than entitled, diluting existing holders' accrued yield by that amount.

**Foundry fork test plan:**
- Fork an L2 where `RSETHPoolV3` is deployed.
- Warp time forward 48 hours without calling `updateRate()`.
- Record existing holder rsETH balance and total supply.
- Call `deposit` with 100 ETH as attacker.
- Assert attacker received more rsETH than `100e18 * 1e18 / trueRate`.
- Assert existing holder's proportional ETH claim decreased.
