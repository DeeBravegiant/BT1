Audit Report

## Title
`_updateRsETHPrice` price-increase guard bypassed when rsETH price is below all-time high, enabling front-running of price updates — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle._updateRsETHPrice()` applies the `pricePercentageLimit` rate-limit guard only when `newRsETHPrice > highestRsethPrice`. When the rsETH price is recovering from a prior dip (still below `highestRsethPrice`), the guard block is never entered, allowing an arbitrarily large price increase to proceed unchecked. Because both `depositETH()` and `updateRSETHPrice()` are public and permissionless, an attacker can deposit at the stale low price, trigger the price update themselves, and capture the accrued yield that belongs to existing rsETH holders.

## Finding Description
In `_updateRsETHPrice()`, the sole rate-limit mechanism is:

```solidity
// contracts/LRTOracle.sol lines 252–267
if (newRsETHPrice > highestRsethPrice) {
    uint256 priceDifference = newRsETHPrice - highestRsethPrice;
    bool isPriceIncreaseOffLimit =
        pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
    if (isPriceIncreaseOffLimit) {
        if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
            revert PriceAboveDailyThreshold();
        }
    }
}
``` [1](#0-0) 

The outer `if` condition gates the entire check on `newRsETHPrice > highestRsethPrice`. When the current price is below the ATH (e.g., after a dip that was within `pricePercentageLimit` and did not trigger the pause), this block is never entered. The price is still written unconditionally at line 313: [2](#0-1) 

The entry path is fully permissionless:

1. `updateRSETHPrice()` — public, no access control: [3](#0-2) 

2. `depositETH()` — public, callable by any EOA: [4](#0-3) 

3. `getRsETHAmountToMint` uses the **stored** `lrtOracle.rsETHPrice()`, not a freshly computed value: [5](#0-4) 

**Exploit flow:**
- State: `highestRsethPrice = 1.05e18`, `rsETHPrice = 1.03e18` (prior dip of 1.9%, within limit, no pause), `pricePercentageLimit = 1e16` (1%), price update not called for 2 days so accumulated rewards push `newRsETHPrice` to `1.049e18`.
- Attacker calls `depositETH{value: 100 ether}()` → receives `100e18 / 1.03e18 ≈ 97.087 rsETH` at the stale low price.
- Attacker calls `updateRSETHPrice()` → `1.049e18 < 1.05e18 = highestRsethPrice`, guard block is skipped, `rsETHPrice` is set to `1.049e18`.
- Attacker exits (secondary market or withdrawal) → `97.087 × 1.049 ≈ 101.84 ETH`, profit ≈ 1.84 ETH. The 1% `pricePercentageLimit` would have blocked this (1.84% increase), but was never evaluated.

## Impact Explanation
**High — Theft of unclaimed yield.** The attacker captures staking rewards that should have accrued proportionally to existing rsETH holders. The surplus is bounded only by accumulated rewards since the last price update and the size of the attacker's deposit. The theft is direct and repeatable whenever the preconditions are met.

## Likelihood Explanation
**Medium.** The required preconditions — a prior dip within `pricePercentageLimit` (so no pause was triggered) and a multi-day gap between price updates — are both realistic and fully observable on-chain before the attacker acts. The attack requires no privileged access, no victim mistakes, and no external protocol compromise. It is repeatable any time the conditions recur.

## Recommendation
Replace the comparison against `highestRsethPrice` with a comparison against the **previous stored price** (`rsETHPrice`) so the rate-limit is enforced on every upward price move, regardless of whether the new price is above or below the ATH:

```solidity
if (newRsETHPrice > rsETHPrice) {
    uint256 priceDifference = newRsETHPrice - rsETHPrice;
    bool isPriceIncreaseOffLimit =
        pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(rsETHPrice);
    if (isPriceIncreaseOffLimit) {
        if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
            revert PriceAboveDailyThreshold();
        }
    }
}
```

This ensures the daily rate-limit is applied to all price increases, not only those that exceed the ATH.
