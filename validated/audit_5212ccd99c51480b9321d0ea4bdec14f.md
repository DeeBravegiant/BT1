Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check — (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric condition: for ERC-20 assets it checks `totalAssetDeposits + amount > depositLimit`, but for ETH it checks only `totalAssetDeposits > depositLimit`, omitting the incoming deposit amount. Any unprivileged depositor can therefore push ETH holdings arbitrarily beyond the configured cap in a single transaction, violating the protocol's deposit ceiling invariant.

## Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` ( [1](#0-0) ):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));  // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch at line 679 evaluates whether the **current** total already exceeds the limit, not whether the **post-deposit** total would. As long as `totalAssetDeposits ≤ depositLimit`, the function returns `false` regardless of `amount`.

The call chain is: `depositETH` (public, payable) [2](#0-1)  → `_beforeDeposit` [3](#0-2)  → `_checkIfDepositAmountExceedesCurrentLimit`. The `MaximumDepositLimitReached` revert is never triggered for ETH deposits as long as the pre-deposit total is within the limit, so `_mintRsETH` proceeds unconditionally for any `msg.value`.

No other guard in `depositETH` or `_beforeDeposit` enforces the deposit cap for ETH. [4](#0-3) 

## Impact Explanation
**Low — Contract fails to deliver promised returns.** `depositLimitByAsset[ETH_TOKEN]` is the protocol's primary risk-management cap on ETH exposure. Bypassing it allows unbounded rsETH minting beyond the intended ceiling. If excess ETH cannot be fully restaked into EigenLayer strategies (e.g., strategy capacity is reached), it sits idle in the deposit pool, degrading yield for all rsETH holders. The invariant `getTotalAssetDeposits(ETH) ≤ depositLimit` is broken. No direct theft of funds occurs, placing this squarely in the **Low** impact class.

## Likelihood Explanation
The entry point is the public, payable `depositETH` function — no special role or privilege is required. The precondition (`totalAssetDeposits ≤ depositLimit`) is the normal operating state of the protocol. Any depositor can trigger this in a single transaction and repeat it indefinitely as long as the pre-deposit total remains at or below the limit.

## Recommendation
Add `amount` to the ETH branch to match the ERC-20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Assume `depositLimitByAsset[ETH_TOKEN] = 1000 ETH` and `totalAssetDeposits = 999 ETH`.

1. Attacker calls `depositETH{value: 500 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500e18)`.
3. ETH branch evaluates `999e18 > 1000e18` → `false`. Guard does not revert.
4. `_mintRsETH` mints rsETH for 500 ETH, pushing total ETH deposits to 1499 ETH — 49.9% above the configured limit.
5. The attacker (or any depositor) can repeat this in subsequent transactions as long as `totalAssetDeposits ≤ depositLimit` at the start of each call.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    // Setup: set ETH deposit limit to 1000 ether, seed pool with 999 ether
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 1000 ether);
    // ... seed 999 ether via prior deposits ...

    // Attacker deposits 500 ether — should revert but does not
    vm.deal(attacker, 500 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 500 ether}(0, "");

    // Assert limit is exceeded
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
```
