### Title
Hint store permanently skips hints for committed blocks when `ingest_batch` raises on partial-batch error — (`chia/full_node/full_node.py`)

### Summary
During `sync_from_fork_point`, a malicious peer can serve a batch where blocks 0..K-1 are valid (committed to the coin store) and block K is invalid. `add_prevalidated_blocks` returns `(agg_state_change_summary, err)` with both non-None. `ingest_batch` raises `ValueError` before `hint_store.add_hints()` is called. On re-sync, those blocks return `ALREADY_HAVE_BLOCK` and their hints are never recovered, permanently corrupting the hint index.

### Finding Description
`add_prevalidated_blocks` (lines 1683–1758 of `chia/full_node/full_node.py`) iterates over a batch of blocks. For each block that produces `NEW_PEAK`, it accumulates `agg_state_change_summary`. When block K fails (pre-validation error at line 1692, or `INVALID_BLOCK`/`DISCONNECTED_BLOCK` at line 1749), it returns `(agg_state_change_summary, err)` — both non-None.

`ingest_batch` (lines 1416–1418) checks `if err is not None` and immediately raises `ValueError`, discarding `state_change_summary`. The `hint_store.add_hints()` call at line 1437 is never reached.

The coin records for blocks 0..K-1 are durably committed (each `blockchain.add_block()` uses its own DB transaction). On re-sync, those blocks return `ALREADY_HAVE_BLOCK`, which is not handled in the `add_prevalidated_blocks` loop — `agg_state_change_summary` is never populated for them, so hints are never written.

### Impact Explanation
Wallet subscriptions via puzzle-hash hints (`register_for_ph_updates`) rely on `hint_store.get_coin_ids_multi()`. Missing hint entries mean wallets permanently miss coins (CATs, NFTs, DIDs, offer coins) that were created with hints pointing to their puzzle hash during the affected sync window. This is a durable corruption of the hint index with direct wallet sync impact.

### Likelihood Explanation
Any peer the victim connects to during sync can trigger this. The attacker serves real chain blocks (publicly available) as blocks 0..K-1, then appends a crafted invalid block K (e.g., a real block with a corrupted PoSpace field). No special privileges are required. The victim's node bans the peer but the hint gap is already permanent.

### Recommendation
In `ingest_batch`, when `err is not None` but `state_change_summary is not None`, still call `hint_store.add_hints()` for the successfully committed blocks before raising:

```python
state_change_summary, err = await self.add_prevalidated_blocks(...)
# Always persist hints for any blocks that were successfully committed
if state_change_summary is not None:
    hints_to_add, _ = get_hints_and_subscription_coin_ids(
        state_change_summary,
        self.subscriptions.has_coin_subscription,
        self.subscriptions.has_puzzle_subscription,
    )
    await self.hint_store.add_hints(hints_to_add)
if err is not None:
    await peer.close(CONSENSUS_ERROR_BAN_SECONDS)
    raise ValueError(...)
```

### Proof of Concept
1. Intercept or simulate a sync peer.
2. Serve a batch of K valid blocks (real chain blocks containing hinted `CREATE_COIN` outputs), followed by block K+1 with a corrupted PoSpace.
3. Observe that `add_prevalidated_blocks` returns `(summary_for_0_to_K, Err.INVALID_POSPACE)`.
4. Observe that `ingest_batch` raises before `hint_store.add_hints()`.
5. Allow re-sync with an honest peer.
6. Query `hint_store.get_coin_ids(hint)` for hints from blocks 0..K-1 — assert returns empty.
7. Query `coin_store.get_coin_record(coin_id)` for the same coins — assert they exist (coin records committed).
