### Title
Unguarded `init` Allows Any Caller to Front-Run MPC Contract Initialization and Seize Full Participant Control - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `init` function of `MpcContract` carries the `#[init]` attribute but **not** `#[private]`. In NEAR SDK, `#[init]` only prevents re-initialization once state exists; it places no restriction on *who* may call the function. Any external account can therefore race the legitimate deployer and call `init` first, supplying attacker-controlled participant accounts and an attacker-chosen threshold. The contract immediately enters `Running` state under the attacker's governance, with no recovery path.

---

### Finding Description

`MpcContract::init` is declared as:

```rust
// crates/contract/src/lib.rs  lines 1924-1929
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

There is no `#[private]` attribute and no `predecessor_account_id == current_account_id` assertion inside the function body. The function body only validates the supplied `parameters` and writes the resulting state: [2](#0-1) 

By contrast, the sibling initializer `init_running` is explicitly marked `#[private]`, which restricts it to self-calls only:

```rust
// crates/contract/src/lib.rs  lines 1976-1979
#[private]
#[init]
#[handle_result]
pub fn init_running(...)
``` [3](#0-2) 

A sandbox test even asserts that `init_running` rejects external callers with `"Method init_running is private"`: [4](#0-3) 

No equivalent protection exists for `init`. The deployment workflow (documented in the localnet guide) shows that deployment and initialization are two separate transactions, creating a front-runnable window: [5](#0-4) 

---

### Impact Explanation

An attacker who calls `init` first can supply:

- A `participants` map containing only attacker-controlled NEAR accounts.
- A `threshold` of 1 (or any value the attacker satisfies alone).

The contract immediately transitions to `ProtocolContractState::Running` with those parameters: [6](#0-5) 

From that point the attacker's accounts are the sole legitimate participants. They can:

1. Call `vote_add_domains` to register signing domains.
2. Drive DKG (`start_keygen_instance` / `vote_pk`) as the only participant, obtaining full control of every generated key share.
3. Issue threshold signatures (`sign`, `request_app_private_key`, `verify_foreign_transaction`) for arbitrary cross-chain payloads — Bitcoin, Ethereum, Solana — without any honest-node involvement.

Because `#[init]` prevents re-initialization, the legitimate deployer cannot overwrite the attacker's state. The only remediation after the fact is redeployment to a new contract account, which requires migrating all users and infrastructure.

This satisfies: **Critical — Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization.**

---

### Likelihood Explanation

NEAR is a public blockchain. Every transaction, including contract deployments, is visible in real time. An attacker running a simple indexer that watches for deployments of the known MPC contract WASM hash can submit the malicious `init` call in the same block or the immediately following block. No privileged access, leaked key, or collusion is required — only the ability to submit a NEAR transaction.

---

### Recommendation

Add `#[private]` to `init`, making it callable only by the contract account itself (i.e., the deployer must use a batch transaction that deploys and initializes atomically):

```rust
#[handle_result]
#[init]
#[private]          // ← add this
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

Alternatively, assert inside the function body that `env::predecessor_account_id() == env::current_account_id()`. Either approach matches the protection already applied to `init_running` and `migrate`.

---

### Proof of Concept

1. Attacker monitors the NEAR blockchain for a deployment of the MPC contract WASM to a known account (e.g., `v1.signer`).
2. In the same block (or the next), attacker submits:
   ```
   near contract call-function as-transaction v1.signer init \
     json-args '{"parameters": {"threshold": 1, "participants": {"next_id": 1, "participants": [["attacker.near", 0, {"tls_public_key": "<attacker_key>", "url": "https://attacker.example"}]]}}}' \
     sign-as attacker.near
   ```
3. Contract state is now `Running` with `attacker.near` as the sole participant and threshold 1.
4. Attacker calls `vote_add_domains` to register a Secp256k1 Sign domain.
5. Attacker drives DKG alone (`start_keygen_instance` → `vote_pk`), obtaining the sole key share for the domain.
6. Attacker calls `sign(...)` with any payload and any derivation path; the MPC node under attacker control responds with a valid threshold signature.
7. The resulting signature is valid on the target chain (Bitcoin, Ethereum, etc.), enabling arbitrary unauthorized cross-chain transactions.

async fn init_running_rejects_external_callers_pre_initialization() {

    let number_of_participants = 2;
    let (accounts, participants) = gen_accounts(&worker, number_of_participants).await;

    let threshold_parameters = ThresholdParameters::new(

    let init_running_args = serde_json::json!({
            "domains": [],
            "next_domain_id": 0,
            "keyset": Keyset::new(EpochId::new(2), vec![]),
            "parameters": threshold_parameters,
    });

        .call(contract.id(), method_names::INIT_RUNNING)
        .max_gas()
        .args_json(init_running_args)
        .transact()
        .await
        .unwrap()
        .into_result()
        .expect_err("method is private and not callable from participant account.");
