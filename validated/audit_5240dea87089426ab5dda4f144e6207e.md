### Title
Unguarded `init` Allows Any Caller to Seize Full Control of the MPC Network Before Legitimate Initialization — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `MpcContract::init` function carries no caller restriction. Any external account can call it on a freshly deployed (uninitialized) contract and supply attacker-controlled `ThresholdParameters`, installing themselves as the sole participant with threshold 1. Because the production deployment workflow separates the `deploy` transaction from the `init` transaction, a front-running attacker can seize the entire MPC network before the legitimate operator initializes it.

---

### Finding Description

`MpcContract::init` is decorated with `#[init]` (NEAR SDK), which only prevents double-initialization — it does **not** restrict the caller. The function accepts arbitrary `parameters` (participant set + threshold) and immediately transitions the contract into `Running` state with those parameters.

```rust
// crates/contract/src/lib.rs  lines 1924-1973
#[handle_result]
#[init]                          // ← prevents re-init, does NOT restrict caller
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    // No predecessor / signer check whatsoever
    let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
    parameters.validate()?;
    ...
    Ok(Self {
        protocol_state: ProtocolContractState::Running(RunningContractState::new(
            DomainRegistry::default(),
            Keyset::new(EpochId::new(0), Vec::new()),
            parameters,          // ← fully attacker-controlled
            ...
        )),
        ...
    })
}
```

Compare with `init_running`, which is correctly guarded:

```rust
// crates/contract/src/lib.rs  lines 1976-1979
#[private]   // ← only the contract itself may call this
#[init]
pub fn init_running(...) -> Result<Self, Error> { ... }
```

The production deployment scripts issue `deploy` and `init` as two separate transactions, creating an exploitable window:

```bash
# Step 1 – contract is live but uninitialized
near contract deploy "$MPC_CONTRACT_ACCOUNT" use-file "$MPC_CONTRACT_PATH" \
    without-init-call ...

# Step 2 – attacker front-runs this call
near contract call-function as-transaction "$MPC_CONTRACT_ACCOUNT" init \
    file-args "$INIT_ARGS_JSON" ...
```

---

### Impact Explanation

An attacker who wins the race calls `init` with:
- `participants`: a single entry pointing to an attacker-controlled NEAR account
- `threshold`: 1

The contract enters `Running` state with the attacker as the only recognized participant. From that position the attacker can:

1. Call `vote_add_domains` to register signing domains.
2. Call `start_keygen_instance` / `vote_pk` to complete DKG with a key they generated unilaterally (threshold-1 DKG is trivial).
3. Respond to every user `sign` / `request_app_private_key` / `verify_foreign_transaction` call with signatures produced from their own key material.
4. Propose and self-approve contract upgrades via `propose_update` + `vote_update`.

This constitutes **unauthorized threshold signature issuance** and **bypass of threshold-signature requirements** — the entire security guarantee of the MPC network is nullified. Users receive signatures they believe are backed by a distributed threshold network but are in fact produced by a single attacker.

**Impact class:** Critical — unauthorized transaction execution and threshold signature issuance without required participant authorization.

---

### Likelihood Explanation

- The deployment scripts (`localnet/tee/scripts/rust-launcher/deploy-tee-cluster.sh`, `docs/localnet/localnet.md`) universally use a two-step deploy-then-init pattern with no atomicity guarantee.
- NEAR transactions are public; an attacker monitoring the chain detects the deploy transaction in the same block or the next block and submits a competing `init` call.
- No privileged access, leaked secrets, or TEE capabilities are required — only a standard NEAR account and the ability to submit a transaction.
- The attack is a single transaction and succeeds deterministically if it lands before the operator's `init`.

---

### Recommendation

Add a predecessor/signer check inside `init` so only the contract account itself (or a pre-designated deployer account) can call it:

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    // Restrict to the contract account itself (mirrors #[private] semantics)
    assert_eq!(
        env::predecessor_account_id(),
        env::current_account_id(),
        "init may only be called by the contract account"
    );
    ...
}
```

Alternatively, add `#[private]` (which enforces `predecessor == current_account_id` automatically in NEAR SDK) and invoke `init` as a self-call chained to the deploy transaction, eliminating the window entirely.

---

### Proof of Concept

1. Operator broadcasts: `near contract deploy mpc-contract.near use-file mpc_contract.wasm without-init-call ...`
2. Attacker observes the deploy transaction on-chain (block N).
3. Attacker broadcasts in block N or N+1:
   ```bash
   near contract call-function as-transaction mpc-contract.near init \
     json-args '{
       "parameters": {
         "participants": {
           "next_id": 1,
           "participants": [["attacker.near", 0, {"url":"http://attacker.com","tls_public_key":"ed25519:<attacker_key>"}]]
         },
         "threshold": 1
       },
       "init_config": null
     }' \
     prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near ...
   ```
4. `init` succeeds; `mpc-contract.near` is now in `Running` state with `attacker.near` as the sole participant.
5. Operator's subsequent `init` call panics: `"The contract has already been initialized"`.
6. Attacker runs DKG (trivial with n=t=1), registers a key, and begins responding to all user signing requests with attacker-generated signatures. [1](#0-0) [2](#0-1) [3](#0-2)

  local keys_file="$1"
  log "Adding node keys to NEAR accounts using $keys_file"
  [ -f "$keys_file" ] || { err "Missing keys file at $keys_file. Run collect phase first."; exit 1; }

  jq -c '.[]' "$keys_file" | while read -r row; do
    local acct signer responder
    acct="$(echo "$row" | jq -r .account)"
    signer="$(echo "$row" | jq -r .signer_pk)"
    responder="$(echo "$row" | jq -r .responder_pk)"

    log "$acct: add signer key"
    near_add_key_skip_if_exists "$acct" "$signer" "signer"

    log "$acct: add responder key"
    near_add_key_skip_if_exists "$acct" "$responder" "responder"
  done
}
