Audit Report

## Title
Unbounded Withdrawal Queue Spam via Zero `minRsEthAmountToWithdraw` Enables Temporary Freezing of Legitimate Withdrawals — (`File: contracts/LRTWithdrawalManager.sol`)

## Summary

`minRsEthAmountToWithdraw[asset]` defaults to `0` for any asset not explicitly configured by the admin. When this value is `0`, the minimum-amount guard in `initiateWithdrawal` reduces to only rejecting `rsETHUnstaked == 0`, allowing any amount ≥ 1 wei. An attacker holding rsETH can flood the FIFO withdrawal queue with arbitrarily many dust requests, forcing the operator to process all spam entries before any legitimate user's request behind them can be unlocked, temporarily freezing those users' funds.

## Finding Description

**Root cause — ineffective minimum check when `minRsEthAmountToWithdraw[asset] == 0`:**

```solidity
// L162-163
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

`minRsEthAmountToWithdraw` is a `mapping(address => uint256)` (L35) that Solidity initialises to `0` for every key. `setMinRsEthAmountToWithdraw` (L330-332) is the only way to configure it, and `initialize` (L90-98) never calls it. Therefore, for any freshly supported asset, `minRsEthAmountToWithdraw[asset] == 0`, and the condition `rsETHUnstaked < 0` is always `false` for a `uint256`. The only effective guard is `rsETHUnstaked == 0`, so any amount ≥ 1 wei is accepted.

**FIFO queue with no skip mechanism:**

`_addUserWithdrawalRequest` (L744-759) assigns monotonically increasing nonces. `_unlockWithdrawalRequests` (L770-816) starts from `nextLockedNonce[asset]` and advances sequentially:

```solidity
// L790-815
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
```

There is no mechanism to skip or reorder entries. Legitimate users whose requests were queued after the spam entries cannot be unlocked until every preceding spam entry has been processed. The operator can limit the range per call via `firstExcludedIndex`, but must still advance through every spam entry in order before reaching legitimate requests.

**Exploit path:**

1. Admin deploys and adds support for an asset; `minRsEthAmountToWithdraw[asset]` remains `0`.
2. Attacker holds rsETH and calls `initiateWithdrawal(asset, 1, "")` repeatedly across many blocks. Each call passes the check, commits ~1 wei of assets, and pushes a nonce into the global queue.
3. Legitimate users queue their withdrawals at higher nonces.
4. Operator calls `unlockQueue`. The loop must advance through all spam nonces before reaching legitimate users' nonces. Even with batched calls (small `firstExcludedIndex`), the operator must make O(spam_count) calls before any legitimate withdrawal is unlocked.
5. Legitimate users' funds are frozen for the duration of spam processing.

**Existing checks that fail:**

- `ExceedAmountToWithdraw` check (L170): only prevents committing more than the vault's available balance; with 1-wei requests this is trivially satisfied.
- `withdrawalDelayBlocks` (L795): only delays when spam entries become eligible; it does not prevent queue flooding.
- `firstExcludedIndex` parameter: allows batching but does not allow skipping spam entries already in the queue.

## Impact Explanation

**Medium — Temporary freezing of funds.** Legitimate users' withdrawal requests are queued behind spam entries and cannot be unlocked until all preceding spam entries are processed. The operator must make O(spam_count) batched `unlockQueue` calls to clear the backlog. During this period, legitimate users cannot complete their withdrawals and their rsETH remains locked in the contract. This matches the allowed impact "Medium. Temporary freezing of funds."

## Likelihood Explanation

Any rsETH holder can trigger this without any privileged access. The attacker's rsETH is returned when their spam requests are eventually processed, so the only sustained cost is gas per `initiateWithdrawal` call. The attack is most effective against newly supported assets or fresh deployments where `minRsEthAmountToWithdraw` has not been set. The attack can be repeated continuously: once the operator clears a batch of spam, the attacker can re-queue new spam entries ahead of the next wave of legitimate requests.

## Recommendation

1. **Set a non-zero default in `initialize`**: Call `setMinRsEthAmountToWithdraw` for all initially supported assets during `initialize`, or add a storage variable for a global default minimum that is enforced when the per-asset value is `0`.
2. **Guard against zero minimum at withdrawal time**: In `initiateWithdrawal`, add `if (minRsEthAmountToWithdraw[asset] == 0) revert MinAmountNotConfigured();` to prevent withdrawals on assets where the minimum has not been explicitly set.
3. **Per-user pending request cap**: Reject `initiateWithdrawal` if the caller already has more than `N` pending (unlocked) requests in `userAssociatedNonces[asset][msg.sender]`.
