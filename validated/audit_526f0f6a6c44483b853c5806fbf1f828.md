Audit Report

## Title
Missing Chainlink Price Feed Staleness Check Allows Stale Asset Prices to Corrupt rsETH Valuation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all freshness fields, accepting stale prices without any validation. Because `updateRSETHPrice()` is permissionless, an attacker can commit a stale-price-derived `rsETHPrice` on-chain, enabling them to deposit at a mispriced rate and extract value from existing rsETH holders. The same repository already implements the correct staleness pattern in `ChainlinkOracleForRSETHPoolCollateral`, confirming this is an unintentional omission.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` reads only the `price` field from `latestRoundData()`, silently discarding `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository performs both the `answeredInRound < roundID` check and the `timestamp == 0` check: [2](#0-1) 

The full permissionless call chain is:

```
updateRSETHPrice() [public whenNotPaused]
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()
            └─ getAssetPrice(asset)  ← LRTOracle.sol:339
                 └─ ChainlinkPriceOracle.getAssetPrice()  ← no staleness check
``` [3](#0-2) [4](#0-3) 

The `pricePercentageLimit` guard in `_updateRsETHPrice()` is not a substitute: it compares the new price against `highestRsethPrice` (an all-time high watermark), not against a freshness window. A stale price that falls within the configured percentage band passes through unchecked. If `pricePercentageLimit` is 0 (its default until explicitly set), there is no guard at all. [5](#0-4) 

## Impact Explanation
When a Chainlink feed is in a stale round and returns a price lower than the true market price, `_getTotalEthInProtocol()` underestimates total ETH, producing a depressed `rsETHPrice`. An attacker who calls `updateRSETHPrice()` to commit this price and then immediately deposits receives more rsETH than their deposit is worth, diluting all existing rsETH holders. This constitutes **theft of unclaimed yield** (High) from existing holders. In the reverse direction (stale price higher than true), depositors are shortchanged, mapping to **contract fails to deliver promised returns** (Low). The High-impact path is the primary concern. [6](#0-5) [7](#0-6) 

## Likelihood Explanation
Chainlink feeds enter stale rounds during L1 gas spikes, node outages, and network congestion — documented real-world events on mainnet. The attacker requires no special privileges: `updateRSETHPrice()` is `public whenNotPaused`, callable by any EOA. The attacker only needs to monitor Chainlink round data off-chain and time the call when `answeredInRound < roundId` or `updatedAt` is outside the heartbeat window. The attack is repeatable every time a feed goes stale. [3](#0-2) 

## Recommendation
Apply the same staleness pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Recommended: per-feed heartbeat check
    // if (block.timestamp - updatedAt > maxStaleness[asset]) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Add a configurable `maxStaleness` mapping per asset to handle feeds with different heartbeat intervals (e.g., 1 hour for ETH/USD vs. 24 hours for LST/ETH feeds). [1](#0-0) 

## Proof of Concept
1. Monitor Chainlink's stETH/ETH feed until `answeredInRound < roundId` (stale round condition).
2. Call `LRTOracle.updateRSETHPrice()` — permissionless, no prerequisites.
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale (lower) price without reverting.
4. `rsETHPrice` is set to a depressed value on-chain.
5. Immediately call `LRTDepositPool.depositAsset(stETH, largeAmount)` — `getRsETHAmountToMint()` uses the depressed `rsETHPrice`, minting excess rsETH to the attacker.
6. When the Chainlink feed recovers and `updateRSETHPrice()` is called again, `rsETHPrice` corrects upward, and the attacker's excess rsETH is now worth more ETH than deposited — extracted from existing holders' share of TVL.
