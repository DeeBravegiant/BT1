Audit Report

## Title
Stale Cross-Chain Oracle Rate Causes Structural rsETH/wrsETH Over-Issuance and Pool Insolvency - (`contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

## Summary

All three L2 deposit pools compute the rsETH/wrsETH amount to issue using a cross-chain oracle rate that is always ≤ the actual current L1 rate, because rsETH appreciates continuously while the oracle is updated periodically. When the deposited ETH is later bridged to L1 and deposited into `LRTDepositPool`, the L1 mints fewer rsETH than the L2 pool already issued. This structural rate mismatch creates a cumulative, unbounded deficit: `RSETHPoolNoWrapper`'s rsETH reserve depletes monotonically, and the `RsETHTokenWrapper` backing wrsETH for `RSETHPoolV3`/`RSETHPoolV3ExternalBridge` becomes undercollateralized, causing last redeemers to receive less rsETH than their wrsETH entitles them to.

## Finding Description

**Root cause — stale oracle with no freshness enforcement:**

`CrossChainRateReceiver.lzReceive()` stores the rate when a LayerZero message arrives but `getRate()` returns it unconditionally with no staleness check:

```solidity
rate = _rate;
lastUpdated = block.timestamp;
``` [1](#0-0) 

```solidity
function getRate() external view returns (uint256) {
    return rate;
}
``` [2](#0-1) 

`InterimRSETHOracle` has no timestamp at all — rate is set manually by `MANAGER_ROLE` with no on-chain staleness bound: [3](#0-2) 

**Exploit path — RSETHPoolNoWrapper (Arbitrum/Unichain):**

`deposit()` transfers pre-loaded rsETH from the pool's own balance to the user at the stale L2 rate: [4](#0-3) 

The rsETH amount is computed as `amountAfterFee * 1e18 / rsETHToETHrate` where `rsETHToETHrate` is the stale oracle value: [5](#0-4) 

The deposited ETH is later bridged to L1 via `bridgeAssets()`/`bridgeAssetsViaNativeBridge()`, where `L1Vault.depositETHForL1VaultETH()` deposits it into `LRTDepositPool` at the **current L1 rate**: [6](#0-5) 

Since `R_L1 ≥ R_L2` (rsETH appreciates monotonically), `L1` mints `ETH * 1e18 / R_L1` rsETH, which is strictly less than the `ETH * 1e18 / R_L2` rsETH already transferred to the user. The deficit per deposit is `ETH * (R_L1 - R_L2) / (R_L2 * R_L1)`. The pool's rsETH reserve depletes on every deposit cycle with no replenishment mechanism.

**Exploit path — RSETHPoolV3 / RSETHPoolV3ExternalBridge:**

These pools mint wrsETH to users immediately via `wrsETH.mint(msg.sender, rsETHAmount)` at the stale L2 rate: [7](#0-6) [8](#0-7) 

The `RsETHTokenWrapper` tracks the undercollateralization gap via `maxAmountToDepositBridgerAsset()`: [9](#0-8) 

When rsETH is bridged back from L1 and deposited into the wrapper via `depositBridgerAssets()`, the amount received (`ETH * 1e18 / R_L1`) is less than the wrsETH already minted (`ETH * 1e18 / R_L2`). The wrapper's rsETH balance never fully covers the wrsETH supply. Users who call `withdraw()` to redeem wrsETH for rsETH will find insufficient rsETH in the wrapper once the cumulative deficit exceeds the initial seed balance. [10](#0-9) 

**No existing guard prevents this:** The pool contracts call `getRate()` without checking `lastUpdated`. There is no rate cap, no deficit accounting, no stability reserve, and no circuit breaker that halts deposits when the pool's rsETH reserve falls below a threshold.

## Impact Explanation

**Protocol insolvency (Critical).** For `RSETHPoolNoWrapper`, the pool's rsETH reserve depletes monotonically — eventually the pool cannot fulfill new deposits. For `RSETHPoolV3`/`RSETHPoolV3ExternalBridge`, wrsETH becomes structurally undercollateralized: the wrapper holds less rsETH than the outstanding wrsETH supply, so last redeemers cannot withdraw their full rsETH entitlement. At $100M TVL, 5% APR, and a 24-hour oracle lag, the daily deficit is approximately $13,700 (~$5M/year). No contract mechanism covers this shortfall. This matches the "Protocol insolvency" Critical impact class.

## Likelihood Explanation

**Medium-High.** The deficit is structural and continuous — it occurs on every single deposit by any ordinary user, requiring no attacker and no adversarial conditions. The `InterimRSETHOracle` path (manual updates) creates larger staleness windows. The `CrossChainRateReceiver` path depends on LayerZero message frequency, which can be delayed by network congestion or operator inaction. The deficit is small per block but the cumulative effect is material at protocol scale and grows without bound.

## Recommendation

1. **Enforce oracle freshness in `getRate()`**: Add a `maxStaleness` parameter and revert if `block.timestamp - lastUpdated > maxStaleness` in `CrossChainRateReceiver`. For `InterimRSETHOracle`, add a `lastUpdated` timestamp and enforce the same check.
2. **Rate cap on deposit**: When computing `rsETHAmount`, use a conservative rate floor — e.g., apply a small discount to the oracle rate to account for expected appreciation between oracle updates and L1 deposit.
3. **Deficit accounting and deposit circuit breaker**: Track cumulative rsETH deficit in `RSETHPoolNoWrapper` and halt new deposits when the deficit exceeds a configurable threshold relative to the pool's rsETH reserve.
4. **Stability reserve**: Allocate a portion of collected fees to a reserve fund that covers the rate-lag deficit, ensuring the wrapper remains fully collateralized.

## Proof of Concept

**Setup**: `RSETHPoolNoWrapper` on Arbitrum. `InterimRSETHOracle` last set at `R_L2 = 1.050e18`. Actual L1 rate at time of L1 deposit: `R_L1 = 1.055e18` (24 hours of ~4.7% APR accrual).

**Step 1 — User calls `deposit{value: 100 ether}("")` on L2:**
```
fee = 100e18 * feeBps / 10_000  (assume feeBps = 0 for clarity)
rsETHAmount = 100e18 * 1e18 / 1.050e18 = 95.238 rsETH
```
Pool transfers 95.238 rsETH to user from its pre-loaded balance.

**Step 2 — Bridger calls `bridgeAssets()`, ETH arrives at `L1Vault`:**
`depositETHForL1VaultETH()` calls `lrtDepositPool.depositETH(100 ETH)` at current L1 rate `1.055e18`:
```
rsETHMinted = 100e18 * 1e18 / 1.055e18 = 94.787 rsETH
```

**Step 3 — `bridgeRsETHToL2()` sends 94.787 rsETH back to L2 pool:**
Pool receives 94.787 rsETH but gave out 95.238 rsETH.

**Deficit per cycle: 0.451 rsETH (~$1,350 at $3,000/ETH) on a single 100 ETH deposit.**

Repeat for every deposit. The pool's rsETH reserve decreases by 0.451 rsETH per 100 ETH deposited. At $100M TVL with daily bridging, the annualized deficit is ~$5M. No contract mechanism covers this shortfall. The pool's rsETH reserve depletes monotonically until it cannot fulfill new deposits.

**Foundry fork test plan**: Fork Arbitrum mainnet. Deploy/use existing `RSETHPoolNoWrapper` with `InterimRSETHOracle`. Record pool rsETH balance. Simulate 10 deposit cycles of 100 ETH each with a fixed oracle rate while advancing the L1 rate by 0.05% per cycle. Assert that `rsETH.balanceOf(pool)` decreases by the expected deficit amount after each cycle. Confirm the deficit is non-zero and cumulative.
