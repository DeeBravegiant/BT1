### Title
Proposal Deposits Permanently Locked When `do_update` Clears All Competing Proposals — (`crates/contract/src/update.rs`)

### Summary

When `do_update` executes a winning contract-upgrade proposal, it unconditionally clears **all** other pending proposals and their associated storage-staking deposits. Because `UpdateEntry` never records the proposer's account ID, there is no mechanism to refund those deposits. Any participant who proposed a competing update loses their deposit permanently — it is absorbed into the contract's balance with no recovery path.

### Finding Description

`propose_update` requires a deposit proportional to the size of the proposed WASM binary or config blob: [1](#0-0) 

The deposit is accepted and the entry is stored, but the proposer's account ID is **not** recorded in `UpdateEntry`: [2](#0-1) 

When `do_update` is triggered by a threshold vote, it removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal: [3](#0-2) 

No refund is issued for the cleared entries. The contract has no `defund`-style withdrawal function for its own balance, so the deposits are permanently locked.

The `propose_update` function does refund excess deposit at submission time, but it does not store the proposer's identity for a future refund: [4](#0-3) 

### Impact Explanation

Any participant who proposed a competing update loses their full storage-staking deposit when a different proposal reaches threshold. For a 1 MB WASM binary the deposit is on the order of 10 NEAR (storage byte cost × bytes_used). The funds are absorbed into the contract's balance with no recovery path — a permanent, irreversible loss matching the "permanent freezing of funds" criterion.

### Likelihood Explanation

This occurs in any normal governance round where more than one participant proposes a different update. It requires no malicious intent: participants voting for their preferred update inadvertently destroy the deposits of all other proposers. The test suite confirms the clearing behaviour is intentional and exercised: [5](#0-4) 

### Recommendation

Store the proposer's `AccountId` inside `UpdateEntry` alongside `bytes_used`. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and schedule a `Promise::new(entry.proposer).transfer(refund_amount)` for each one. This mirrors the pattern already used in `propose_update` for excess-deposit refunds.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB WASM binary, attaching ~10 NEAR deposit. Entry `id=0` is stored.
2. Participant B calls `propose_update` with a different WASM binary, attaching ~10 NEAR deposit. Entry `id=1` is stored.
3. Threshold participants call `vote_update(id=0)`. `do_update(id=0)` fires.
4. Inside `do_update`: entry `id=0` is removed and deployed; then `self.entries.clear()` wipes entry `id=1`.
5. Participant B's ~10 NEAR deposit is now part of the contract's balance. `UpdateEntry` for `id=1` carried no `proposer` field, so no refund is possible. The funds are permanently locked. [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }
