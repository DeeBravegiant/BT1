### Title
`verify_foreign_transaction` Accepts Empty `extractors` List, Enabling MPC Signature Over Unverified Foreign-Chain State — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` contract entry-point does not validate that the `extractors` field inside the chain-specific RPC request is non-empty. When a user submits a request with an empty extractor list, MPC nodes honestly process it, extract nothing, and collectively sign a canonical payload whose `values` field is an empty vector. The contract then resolves the yield and returns a valid threshold signature to the caller. The result is a genuine MPC signature that appears to attest to a verified foreign-chain transaction but actually carries zero extracted on-chain state — a direct analog to `BlockhashRegistry.recreateBlockheaders` succeeding with an empty `_blockheaders` array and emitting `LogBlockhashAdded` without having added anything.

---

### Finding Description

**Root cause — missing non-empty check on `extractors`**

`verify_foreign_transaction` in `crates/contract/src/lib.rs` performs several precondition checks (gas, deposit, chain whitelist) but never asserts that the per-chain extractor list is non-empty:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);          // gas / deposit / domain
    let requested_chain = request.request.chain();
    if !supported_chains.contains(&requested_chain) { env::panic_str(...); }
    // ← NO check: request.request.extractors().is_empty()
    let request = args_into_verify_foreign_tx_request(request);
    self.enqueue_yield_request(...);
}
``` [1](#0-0) 

The per-chain request structs (`BitcoinRpcRequest`, `EvmRpcRequest`, `SolanaRpcRequest`, etc.) all carry `extractors: Vec<…>` with no minimum-length constraint enforced anywhere in the contract: [2](#0-1) 

**Node-side behavior with empty extractors**

The node-side inspector's `extract` function is explicitly tested to succeed and return an empty vector when no extractors are provided:

```rust
let extracted_values = inspector
    .extract(tx_id, threshold, Vec::new())   // ← empty extractor list
    .await
    .expect("extract should succeed");
assert_eq!(expected_extractions, vec![]);    // ← returns empty, no error
``` [3](#0-2) 

The node then constructs `ForeignTxSignPayloadV1 { request, values: vec![] }` and signs `SHA-256(borsh(payload))`. The `values` field is an unconstrained `Vec<ExtractedValue>`: [4](#0-3) 

**Contract-side response validation**

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` using the root public key. It does not verify that `payload_hash` was derived from a payload containing at least one extracted value, nor that the number of values matches the number of extractors in the original request:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response, &payload_hash, &secp_pk,
).is_ok()
// ← no check: response.values.is_empty() or values.len() == request.extractors.len()
``` [5](#0-4) 

The yield is then resolved and the caller receives the response as if a full verification with extracted values had occurred: [6](#0-5) 

**Parallel to the original bug**

| `BlockhashRegistry` | NEAR MPC |
|---|---|
| `_blockheaders` is empty | `extractors` is empty |
| `reCalculateBlockheaders` loop skipped, returns `currentBlockhash` unchanged | Inspector loop skipped, returns `values: []` |
| Same blockhash stored, `LogBlockhashAdded` emitted | Empty-values payload signed, yield resolved as success |
| Caller believes a new blockhash was added | Caller receives a valid MPC signature that appears to attest to verified foreign-chain state |

---

### Impact Explanation

A caller receives a genuine threshold signature from the MPC network over `SHA-256(borsh(ForeignTxSignPayload { request, values: [] }))`. Any NEAR bridge or omnibridge contract that:

1. Calls `verify_foreign_transaction` to gate an inbound bridge action, and
2. Does not independently assert that the returned `values` vector is non-empty and contains the expected typed observations

will accept this signature as proof that the MPC network verified the foreign transaction, even though the network attested to nothing about the transaction's content (amount, recipient, block hash, etc.). This enables an attacker to obtain a valid "verification" signature for a transaction without the MPC network having extracted any on-chain state, potentially triggering invalid bridge execution or double-spend conditions.

**Allowed impact matched**: High — forged foreign-chain verification / light-client-style verification bypass that causes invalid bridge execution.

---

### Likelihood Explanation

The attack requires only a 1-yoctoNEAR deposit and a call to the public `verify_foreign_transaction` entry-point with an empty extractor list. No special role, no collusion, no key material. The honest MPC nodes will process the request correctly (verifying finality) but sign an empty-values payload because the protocol does not prohibit it. Likelihood is **High**.

---

### Recommendation

1. **In `verify_foreign_transaction`**: reject requests whose extractor list is empty before enqueuing the yield:
   ```rust
   if request.request.extractors_is_empty() {
       env::panic_str("at least one extractor must be specified");
   }
   ```

2. **In `respond_verify_foreign_tx`**: assert that `response.values` (or equivalently, that `payload_hash` was derived from a payload with `values.len() == request.extractors.len()`) is non-empty before resolving the yield. Because the contract stores the original request, it can reconstruct the expected extractor count and compare.

3. **In bridge/consumer contracts**: independently verify that the `values` vector in the returned `VerifyForeignTransactionResponse` is non-empty and contains the expected typed observations before acting on the signature.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
