### Title
Unverified Peer-Supplied CoinSpend Solution Enables Pool Payout Redirection via Crafted Pool Singleton State Transition - (`chia/wallet/util/wallet_sync_utils.py`, `chia/pools/pool_wallet.py`, `chia/wallet/wallet_state_manager.py`)

---

### Summary

When a pool wallet singleton coin is spent on-chain, the wallet fetches the coin's puzzle and solution from the connected peer via `fetch_coin_spend`. The puzzle hash is verified against the coin's committed puzzle hash, but the **solution is never verified** against any on-chain commitment (no Merkle proof, no CLVM re-execution). A malicious peer can return the correct puzzle with a crafted solution encoding a different `target_puzzle_hash`, causing `apply_state_transition` to persist a forged pool state and `update_pool_config` to overwrite the farmer's `payout_instructions` with an attacker-controlled address.

---

### Finding Description

**Entry point — `fetch_coin_spend` in `wallet_sync_utils.py`:** [1](#0-0) 

The function requests the puzzle and solution from the peer via `request_puzzle_solution`. It validates that `puzzle.get_tree_hash() == coin.puzzle_hash`, but the **solution field is accepted verbatim** with no further verification. A malicious peer can return the correct puzzle (so the hash check passes) paired with an arbitrary crafted solution.

**State update path — `_add_coin_states` in `wallet_state_manager.py`:** [2](#0-1) 

When a `POOLING_WALLET` coin is spent, the code calls `fetch_coin_spend_for_coin_state(curr_coin_state, peer)` and immediately passes the result to `apply_state_transition`. There is no re-execution of the CLVM puzzle with the returned solution, and no Merkle proof tying the solution to the block's transaction root.

**Missing guard — `apply_state_transition` in `pool_wallet.py`:** [3](#0-2) 

The only check performed is that `spent_coin_name == new_state.coin.name()` (the coin ID matches the expected tip). No CLVM execution is performed to verify the solution is consistent with the puzzle. The crafted `CoinSpend` is stored directly into `pool_store` and then passed to `update_pool_config`.

**Config overwrite — `update_pool_config` in `pool_wallet.py`:** [4](#0-3) 

`update_pool_config` derives `target_puzzle_hash` from the pool state extracted from the (attacker-supplied) solution and writes it as `payout_instructions` into the local `PoolingShareState` YAML. The harvester reads this file and sends `payout_instructions` to the pool during share submission. The pool then pays block rewards to whatever address is in `payout_instructions`.

---

### Impact Explanation

An unprivileged attacker operating as a peer the wallet connects to can redirect all future pool farming rewards to an attacker-controlled puzzle hash. The corruption persists in the local `PoolingShareState` config file until the farmer manually resyncs from a trusted peer. This constitutes **pool payout redirection** and **corruption of pool membership state with direct security impact**, matching the High impact category.

---

### Likelihood Explanation

The wallet connects to untrusted peers by default. The coin state for the pool singleton passes `validate_received_state_from_peer` (which validates coin existence via Merkle proofs against the block header), but the subsequent `fetch_coin_spend` call is made to the **same peer** with no independent verification of the returned solution. Any peer the wallet connects to can mount this attack the moment a pool singleton coin is spent on-chain (e.g., during a normal pool state transition such as joining or leaving a pool).

---

### Recommendation

After fetching the `CoinSpend` from the peer, re-execute the puzzle with the returned solution using the CLVM interpreter and verify that the resulting conditions are consistent with the known on-chain outcome (e.g., the child coin created matches the coin ID already confirmed on-chain). Specifically:

1. In `fetch_coin_spend` (`wallet_sync_utils.py`), after verifying the puzzle hash, run `puzzle.run(solution)` and confirm the `CREATE_COIN` conditions produce the known child coin.
2. In `apply_state_transition` (`pool_wallet.py`), before calling `pool_store.add_spend`, execute the puzzle with the solution and assert the resulting child coin matches `get_most_recent_singleton_coin_from_coin_spend(new_state)`.

---

### Proof of Concept

1. Farmer wallet connects to a malicious peer (untrusted mode).
2. A legitimate pool singleton spend occurs on-chain (e.g., farmer initiates `pw_join_pool`).
3. Malicious peer sends a valid `CoinState` for the spent singleton coin (coin exists on-chain, passes `validate_received_state_from_peer`).
4. `_add_coin_states` detects `WalletType.POOLING_WALLET` and calls `fetch_coin_spend_for_coin_state(curr_coin_state, peer)`.
5. Malicious peer responds to `request_puzzle_solution` with the correct puzzle (hash matches) but a crafted solution whose `PoolState` encodes `target_puzzle_hash = attacker_address`.
6. `apply_state_transition` checks only `spent_coin_name == new_state.coin.name()` — passes — and stores the forged spend.
7. `update_pool_config` writes `payout_instructions = attacker_address` to the local `PoolingShareState`.
8. Harvester reads the config and submits `payout_instructions = attacker_address` to the pool on every share.
9. Pool pays all block rewards to the attacker's address.

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(
        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)
    )
    if solution_response is None or not isinstance(solution_response, RespondPuzzleSolution):
        raise PeerRequestException(f"Was not able to obtain solution {solution_response}")
    coin_id = coin.name()
    if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
        raise PeerRequestException(f"Peer returned wrong puzzle hash for coin {coin_id}")
    if solution_response.response.coin_name != coin_id:
        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )

    async def apply_state_transition(
        self, new_state: CoinSpend, block_height: uint32, action_scope: WalletActionScope
    ) -> bool:
        """
        Updates the Pool state (including DB) with new singleton spends.
        The DB must be committed after calling this method. All validation should be done here. Returns True iff
        the spend is a valid transition spend for the singleton, False otherwise.
        """
        tip: tuple[uint32, CoinSpend] = await self.get_tip()
        tip_spend = tip[1]

        tip_coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(tip_spend)
        assert tip_coin is not None
        spent_coin_name: bytes32 = tip_coin.name()

        if spent_coin_name != new_state.coin.name():
            history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
            if new_state.coin.name() in [sp.coin.name() for _, sp in history]:
                self.log.info(f"Already have state transition: {new_state.coin.name().hex()}")
            else:
                self.log.warning(
                    f"Failed to apply state transition. tip: {tip_coin} new_state: {new_state} height {block_height}"
                )
            return False

        await self.wallet_state_manager.pool_store.add_spend(self.wallet_id, new_state, block_height)
        tip_spend = (await self.get_tip())[1]
        self.log.info(f"New PoolWallet singleton tip_coin: {tip_spend} farmed at height {block_height}")

        # If we have reached the target state, resets it to None. Loops back to get current state
        for _, added_spend in reversed(
            await self.wallet_state_manager.pool_store.get_spends_for_wallet(self.wallet_id)
        ):
            latest_state: PoolState | None = solution_to_pool_state(added_spend)
            if latest_state is not None:
                if self.target_state == latest_state:
                    self.target_state = None
                    self.next_transaction_fee = uint64(0)
                    self.next_tx_config = DEFAULT_TX_CONFIG
                break

        await self.update_pool_config(action_scope)
        return True
