### Title
Duplicate `UnRegDepositTxCert` Across Sub-Transactions Allows Double-Claiming of Stake Deposit Refund — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubLedgers.hs`, `SubLedger.hs`)

---

### Summary

In the Dijkstra era, a top-level batch transaction may contain multiple sub-transactions. The `SUBLEDGERS` rule processes them sequentially via `foldM`, threading the updated `LedgerState` between iterations. There is no cross-sub-transaction check preventing two sub-transactions from each including an `UnRegDepositTxCert` for the **same** staking credential. The developers have explicitly documented this gap with a disabled (`xit`) test whose expected predicate failure is marked `error "TODO: predicate failure not yet implemented"`. This is the direct Cardano analog of the external report's double-counting bug: a positive value (the deposit refund) is counted in the first sub-transaction's balance equation, and the second sub-transaction attempts to claim the same refund again without a guard that rejects it.

---

### Finding Description

The `SUBLEDGERS` transition rule iterates over sub-transactions with a simple `foldM`:

```haskell
foldM
  ( \ls subTx ->
      trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
  )
  ledgerState
  subTxs
``` [1](#0-0) 

Inside each `SUBLEDGER` invocation, the `SUBUTXOW` rule receives the `certState` that was current **before** this sub-transaction's `SUBENTITIES` processing:

```haskell
utxoStateAfterSubUtxow <-
  trans @(EraRule "SUBUTXOW" era) $
    TRC
      ( SubUtxoEnv slot pp certState originalUtxo topIsValid
      , utxoStateBeforeSubUtxow
      , stAnnTx
      )
``` [2](#0-1) 

The `SUBUTXO` rule calls `Shelley.updateUTxOStateNoFees`, which uses that `certState` to compute `certsTotalRefundsTxBody` for the sub-transaction's balance accounting. For the **first** sub-transaction, the credential is still registered, so the refund is legitimately counted. After the first sub-transaction completes, the credential is removed from `certState`. However, no rule checks whether a subsequent sub-transaction in the same batch also carries an `UnRegDepositTxCert` for the same credential, nor whether the deposit has already been claimed.

The developers have explicitly acknowledged this gap with a disabled test:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = ... & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [3](#0-2) 

The `xit` prefix means the test is skipped entirely; the `error "TODO: predicate failure not yet implemented"` is a placeholder confirming that the rejection logic does not yet exist. Because `submitFailingTx` expects the transaction to be rejected, the `xit` implies the transaction currently **succeeds** when it should fail.

---

### Impact Explanation

If both sub-transactions succeed and each sub-transaction's UTxO update includes the deposit refund in its outputs, the deposit pot is debited once (the credential is deregistered once) while the UTxO gains the refund amount twice. This is a direct **creation of ADA** through an invalid ledger state transition — the preservation-of-value invariant is violated. The deposit pot would be under-debited or the UTxO would be over-credited, depending on which accounting path `updateUTxOStateNoFees` takes for sub-transactions that lack a fee/balance check.

This matches the **Critical** impact class: *Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition.*

---

### Likelihood Explanation

The attack requires only an unprivileged transaction sender with a registered staking credential. Constructing a Dijkstra batch transaction with two sub-transactions carrying duplicate `UnRegDepositTxCert` certificates is straightforward and requires no special privilege, leaked key, or governance majority. The attack is deterministic and reproducible.

---

### Recommendation

1. In the `SUBLEDGERS` or `SUBLEDGER` rule, collect all `UnRegDepositTxCert` credential hashes across all sub-transactions before processing begins and reject the batch if any credential appears more than once.
2. Alternatively, implement the missing predicate failure in `SUBENTITIES`/`SUBCERTS`/`SUBDELEG` that detects an attempt to deregister a credential that was already deregistered by an earlier sub-transaction in the same batch.
3. Enable and complete the `xit "Multiple subtransactions cannot get the same refund"` test once the predicate failure is implemented.

---

### Proof of Concept

The disabled test in `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` (lines 53–75) is itself the proof of concept. An attacker submits:

```
TopTx {
  subTransactions: [
    SubTx1 { certs: [UnRegDepositTxCert stakingCred deposit], inputs: [input1], outputs: [addr1 ← value1 + deposit] },
    SubTx2 { certs: [UnRegDepositTxCert stakingCred deposit], inputs: [input2], outputs: [addr2 ← value2 + deposit] }
  ]
}
```

Sub-tx1 deregisters `stakingCred` and claims `deposit` in its UTxO balance. Sub-tx2 also carries `UnRegDepositTxCert stakingCred deposit`; because no cross-sub-transaction guard exists, it is not rejected. If `updateUTxOStateNoFees` does not enforce `consumed = produced` for sub-transactions, sub-tx2's outputs also include `deposit`, and the UTxO gains `deposit` ADA that was never backed by a corresponding debit from the deposit pot — violating preservation of value.
