Audit Report

## Title
Missing Minimum Return Check in L2 Pool Deposit Functions Exposes Users to Unbounded Slippage - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV2.sol)

## Summary
The `deposit` functions in `RSETHPoolV3` and `RSETHPoolV2` accept no `minRSETHAmountExpected` parameter, so users cannot express a minimum acceptable wrsETH output. The minted amount is computed from a live oracle rate at execution time, meaning any oracle update between a user's preview call and transaction inclusion silently reduces the wrsETH minted with no on-chain revert. The L1 counterpart `LRTDepositPool._beforeDeposit` enforces exactly this check, confirming the protocol intended slippage protection that was omitted on L2.

## Finding Description
`RSETHPoolV3.deposit(string)` at L246–265 and `RSETHPoolV3.deposit(address,uint256,string)` at L271–293 both call `viewSwapRsETHAmountAndFee`, which divides by the live `getRate()` oracle value, and immediately mint the result with no floor check:

```solidity
// contracts/pools/RSETHPoolV3.sol L258–262
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
```

```solidity
// contracts/pools/RSETHPoolV3.sol L286–290
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
```

`RSETHPoolV2.deposit(string)` at L207–219 has the identical gap:

```solidity
// contracts/pools/RSETHPoolV2.sol L212–216
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // no minimum check
```

The rate used in both pools is fetched from a permissionlessly-updatable oracle:

```solidity
// contracts/pools/RSETHPoolV3.sol L304–307
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

By contrast, `LRTDepositPool._beforeDeposit` at L648–669 enforces:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

There is no equivalent guard anywhere in the L2 deposit paths. The `limitDailyMint` modifier also calls `viewSwapRsETHAmountAndFee` independently before the deposit body executes, so the oracle rate is read twice per transaction with no guarantee of consistency, compounding the exposure.

## Impact Explanation
A user who previews their expected wrsETH output via `viewSwapRsETHAmountAndFee` and then submits a deposit will silently receive fewer wrsETH than shown if the oracle rate ticks upward before inclusion. The user has no mechanism to express a minimum acceptable output and no on-chain recourse. This matches the allowed Low impact: **Contract fails to deliver promised returns, but does not lose value** — the user's ETH value is approximately preserved (each wrsETH is worth proportionally more), but the token quantity delivered is less than the quantity the contract's own view function indicated at preview time.

## Likelihood Explanation
The rsETH oracle rate is updated permissionlessly as staking rewards accrue and is propagated to L2 oracles on a regular cadence. No privileged actor, front-runner, or attacker is required; any routine oracle update between a user's preview and their transaction's inclusion triggers the discrepancy. The affected entry points are the primary user-facing deposit functions on every supported L2 chain. The condition is repeatable and requires no special setup.

## Recommendation
Add a `minRSETHAmountExpected` parameter to all `deposit` overloads in `RSETHPoolV3` and `RSETHPoolV2`, mirroring `LRTDepositPool._beforeDeposit`:

```solidity
function deposit(string memory referralId, uint256 minRSETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRSETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

Apply the same pattern to the ERC-20 token overload in `RSETHPoolV3` and to `RSETHPoolV2.deposit`.

## Proof of Concept
1. User calls `RSETHPoolV3.viewSwapRsETHAmountAndFee(1 ether)` when `getRate()` returns `1.05e18`; preview shows `≈ 0.9524 wrsETH`.
2. Before the user's transaction is included, the L2 rsETH oracle is updated to `1.06e18` via a routine staking-reward propagation (no privileged action required beyond the normal oracle update flow).
3. User's `deposit{value: 1 ether}(referralId)` executes. `viewSwapRsETHAmountAndFee` now computes `1e18 * (1 ether - fee) / 1.06e18 ≈ 0.9434 wrsETH` — approximately 0.94% fewer than previewed.
4. `wrsETH.mint(msg.sender, rsETHAmount)` mints the reduced amount with no revert and no slippage event.
5. The same scenario on L1 via `LRTDepositPool.depositETH` with `minRSETHAmountExpected = 0.9524e18` would revert with `MinimumAmountToReceiveNotMet`; the L2 pools have no equivalent guard.

A Foundry fork test can reproduce this by: (a) deploying or forking the L2 pool, (b) calling `viewSwapRsETHAmountAndFee` to record the expected amount, (c) using `vm.mockCall` or direct oracle state manipulation to increment `getRate()` by a small amount, and (d) asserting that `deposit` succeeds and mints strictly less than the previewed amount with no revert.
