Audit Report

## Title
Consent signature replay in `allowPushers` allows creator to re-establish delegation after pusher revocation - (File: smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol)

## Summary
`allowPushers` verifies a pusher's EIP-191 consent signature and enforces a deadline, but never records that a signature has been consumed. After a pusher calls `revokePusher()` to clear their delegation, the creator can immediately replay the original consent signature (with the same `deadline` value) to re-establish `namespaceRemapping[pusher] = creator`, defeating the revocation. The code comment explicitly acknowledges the deadline is meant to prevent exactly this scenario, but a deadline check alone is insufficient — it only rejects expired signatures, not already-used ones.

## Finding Description
`_ensureDeadline` in `OracleBase` only checks `block.timestamp <= deadline` and returns; neither `OracleBase` nor `CompressedOracleV1` maintains any used-signature set or per-pusher nonce. The signed message is `keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))`. All five fields are static for a given consent act, so the same `(deadline, pusher, creator, sig)` tuple is unconditionally re-accepted on every call to `allowPushers` as long as `block.timestamp <= deadline`.

Exploit path:
1. Pusher signs consent for creator with `deadline = T + 1 day`.
2. Creator calls `allowPushers(deadline, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator calls `allowPushers(deadline, [pusher], [sig])` again with the identical arguments → `namespaceRemapping[pusher] = creator` is restored.
5. Steps 3–4 can repeat until `block.timestamp > deadline`.

Existing guards reviewed and found insufficient:
- `_ensureDeadline` (OracleBase line 124–126): only a timestamp comparison, no invalidation.
- ECDSA recovery (CompressedOracle line 207): verifies authenticity but not uniqueness of use.
- `NoSelfRemapping` check (line 200–202): unrelated guard.

## Impact Explanation
A pusher whose private key is compromised calls `revokePusher()` to stop the attacker from writing bad prices into the creator's namespace. The creator (unaware of the compromise, or acting maliciously) replays the original consent signature, restoring delegation. The attacker now continues pushing arbitrary oracle data into the creator's namespace. That data is consumed by `price()` → pool `swap` → `_getBidAndAskPriceX64`, producing bad bid/ask quotes that reach live swaps. This is a bad-price execution impact: traders receive incorrect swap amounts, constituting direct loss of user principal above Sherlock thresholds.

## Likelihood Explanation
The replay requires only the creator to re-submit the original calldata — no new signature, no privileged key, no special setup. The window is the full deadline duration (up to whatever the pusher agreed to, commonly hours to days). Any creator who has stored the original transaction can perform this replay. The pusher has no on-chain mechanism to close the window early.

## Recommendation
Record each consumed signature hash in a `mapping(bytes32 => bool) private _usedConsents` in `CompressedOracleV1`. Before writing `namespaceRemapping`, compute `bytes32 consentHash = keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))`, require `!_usedConsents[consentHash]`, then set `_usedConsents[consentHash] = true`. This makes each consent one-time-use regardless of deadline expiry, matching the stated design intent.

## Proof of Concept
```solidity
// Foundry test
function testReplayAfterRevoke() public {
    uint256 deadline = block.timestamp + 1 days;
    bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

    address[] memory pushers = new address[](1);
    pushers[0] = pusher;
    bytes[] memory sigs = new bytes[](1);
    sigs[0] = sig;

    // Step 1: establish delegation
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs);
    assertEq(oracle.namespaceRemapping(pusher), creator);

    // Step 2: pusher revokes
    vm.prank(pusher);
    oracle.revokePusher();
    assertEq(oracle.namespaceRemapping(pusher), address(0));

    // Step 3: creator replays the SAME signature — should revert but does not
    vm.prank(creator);
    oracle.allowPushers(deadline, pushers, sigs); // succeeds — replay accepted
    assertEq(oracle.namespaceRemapping(pusher), creator); // delegation restored
}
```
