### Title
Incorrect DRep Expiry Check in Ratification Omits `numDormantEpochs`, Artificially Deflating the Denominator — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

`dRepAcceptedRatio` in the Conway RATIFY rule checks DRep expiry using only the raw stored `drepExpiry` field, without adding `vsNumDormantEpochs`. The correct expiry check — used everywhere else in the codebase — is `numDormantEpochs + storedExpiry < currentEpoch`. Because the ratification logic omits `numDormantEpochs`, DReps that are **not actually expired** are silently excluded from the denominator of the yes/total ratio. This artificially deflates the denominator, lowering the effective threshold for governance ratification and enabling governance actions to be enacted with less DRep support than the protocol requires.

---

### Finding Description

`dRepAcceptedRatio` iterates over the DRep stake distribution and, for each `DRepCredential`, looks up the DRep's state and checks:

```haskell
| reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
``` [1](#0-0) 

`drepExpiry drepState` is the **raw stored expiry**, computed by `computeDRepExpiry` as:

```
storedExpiry = currentEpochAtRegistration + drepActivity - numDormantEpochsAtRegistration
``` [2](#0-1) 

The **actual** expiry at any later point in time is `storedExpiry + currentNumDormantEpochs`. This is precisely what `vsActualDRepExpiry` computes and what `isDRepExpired` uses:

```haskell
binOpEpochNo (+) (vState ^. vsNumDormantEpochsL) (drep' ^. drepExpiryL) < currentEpoch
``` [3](#0-2) 

`RatifyEnv` does not carry a `reNumDormantEpochs` field, so `dRepAcceptedRatio` has no way to apply the correct check. The result is that any DRep satisfying:

```
storedExpiry < currentEpoch   AND   storedExpiry + numDormantEpochs >= currentEpoch
```

is treated as expired in ratification while it is **not actually expired** by the protocol's own definition. [4](#0-3) 

---

### Impact Explanation

The denominator of `dRepAcceptedRatio` is `totalExcludingAbstainStake` — the sum of stake of all non-abstaining, non-expired DReps. When DReps are incorrectly excluded from this sum, the denominator shrinks. Because the threshold check is `yesStake / totalExcludingAbstainStake >= threshold`, a smaller denominator means the same number of yes-votes clears the threshold more easily.

Concretely: a governance action that requires, say, 67% DRep approval could be enacted with materially fewer yes-votes than the protocol intends, whenever `numDormantEpochs` is non-zero. This can affect any governance action type — treasury withdrawals, protocol-parameter changes, committee updates, hard-fork initiations, or constitutional changes — all of which go through `dRepAccepted` → `dRepAcceptedRatio`. [5](#0-4) 

**Allowed impact matched:** *Critical — Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.*

---

### Likelihood Explanation

Dormant epochs accumulate naturally whenever no governance proposals are submitted for one or more epochs. An unprivileged DRep can observe the current `numDormantEpochs` on-chain and choose to submit a proposal precisely when the accumulated dormant count is large enough to exclude a decisive fraction of the DRep stake from the denominator. No privileged access, key compromise, or supermajority collusion is required — only DRep registration (open to anyone) and a deposit-backed proposal submission.

---

### Recommendation

Pass `vsNumDormantEpochs` into `RatifyEnv` (e.g., as a new field `reNumDormantEpochs :: EpochNo`) and update the expiry guard in `dRepAcceptedRatio` to:

```haskell
| reCurrentEpoch > binOpEpochNo (+) reNumDormantEpochs (drepExpiry drepState) ->
    (yes, tot) -- drep is expired, so we don't consider it
```

This aligns the ratification expiry check with the canonical `isDRepExpired` / `vsActualDRepExpiry` logic used everywhere else in the codebase. [6](#0-5) 

---

### Proof of Concept

**Setup:**
- `drepActivity = 5`, `currentEpoch = 10`, `numDormantEpochs = 4`
- DRep D registered at epoch 1 when `numDormantEpochs = 0`
  - `storedExpiry = 1 + 5 - 0 = 6`

**Actual expiry check (`isDRepExpired`):**
```
numDormantEpochs + storedExpiry < currentEpoch
4 + 6 = 10 < 10  →  False  (D is NOT expired)
```

**Ratification check (`dRepAcceptedRatio`):**
```
reCurrentEpoch > drepExpiry drepState
10 > 6  →  True  (D is treated as expired — INCORRECT)
```

D's stake is excluded from `totalExcludingAbstainStake`. If D controls 20% of DRep stake and would have voted No, the effective threshold drops from 67% of 100% to 67% of 80% = 53.6% of total stake. An attacker holding 54% of total DRep stake can now enact any governance action unilaterally, despite the protocol requiring 67%. [7](#0-6) [8](#0-7)

dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
