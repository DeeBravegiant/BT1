### Title
User-Controlled `BlockConfirmations` Bypasses Bitcoin Finality Check, Enabling Signature Over Unconfirmed Transactions - (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` contract endpoint accepts a user-supplied `confirmations: BlockConfirmations` field inside `BitcoinRpcRequest`. No minimum value is enforced anywhere — in the contract, in the inspector, or in any shared validation helper. A caller who sets `confirmations: 0` will always pass the finality gate, causing the MPC network to produce a threshold signature over a Bitcoin transaction that has zero on-chain confirmations and may be double-spent.

---

### Finding Description

`verify_foreign_transaction` queues a `VerifyForeignTransactionRequest` after checking only that the domain exists, has `ForeignTx` purpose, and the chain is in the supported set. [1](#0-0) 

The `BitcoinRpcRequest` carries a caller-controlled `confirmations` field:

```rust
pub struct BitcoinRpcRequest {
    pub tx_id: BitcoinTxId,
    pub confirmations: BlockConfirmations,   // ← fully user-supplied, no floor
    pub extractors: Vec<BitcoinExtractor>,
}
```

Inside `BitcoinInspector::extract`, the sole finality gate is:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [2](#0-1) 

When `block_confirmations_threshold == 0`, the inequality `0 <= actual_confirmations` is trivially true for every transaction — including mempool transactions that have zero confirmations — so the guard never fires. The inspector proceeds to extract values and the MPC nodes collectively sign the payload.

The signed payload is `ForeignTxSignPayloadV1 { request, values }`, where `request` is the full `ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest { confirmations: 0, … })`. The MPC network therefore issues a threshold signature that cryptographically attests "this Bitcoin transaction satisfies a 0-confirmation finality requirement", which is meaningless as a security guarantee.

---

### Impact Explanation

**High.** The foreign-chain verification feature exists precisely to let bridge contracts on NEAR react to finalized foreign-chain events without a trusted relayer. By setting `confirmations: 0`, an attacker obtains a valid MPC threshold signature over an unconfirmed (or even mempool-only) Bitcoin transaction. A bridge contract that relies on this signature to release NEAR-side funds will pay out before the Bitcoin transaction is confirmed, and the attacker can then double-spend the Bitcoin input. This is a direct cross-chain verification bypass leading to double-spend conditions — matching the "High" impact tier.

---

### Likelihood Explanation

**High.** The attack requires only a standard NEAR account, a 1 yoctoNEAR deposit, and the ability to craft a `BitcoinRpcRequest` with `confirmations: 0`. No privileged role, no threshold collusion, and no special tooling are needed. The contract performs no range check on `BlockConfirmations` before queuing the request, and the inspector performs no range check before using it as a threshold. Any user of a bridge built on this feature can exploit it immediately.

---

### Recommendation

Enforce a protocol-level minimum on `BlockConfirmations` at the contract boundary, before the request is queued. A sensible floor for Bitcoin is 1 (transaction must be mined); production bridges typically require 6. The fix mirrors the mitigation in the referenced report — cap/floor the user-supplied value rather than trusting it blindly:

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Bitcoin(ref req) = request.request {
    if req.confirmations < MIN_BITCOIN_CONFIRMATIONS {
        env::panic_str("Bitcoin confirmations below minimum");
    }
}
```

Alternatively, define `BlockConfirmations` as a validated newtype whose constructor rejects values below the protocol minimum, so the invariant is enforced at the type level everywhere the field is used.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
