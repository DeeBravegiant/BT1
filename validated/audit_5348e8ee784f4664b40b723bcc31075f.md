Audit Report

## Title
Withdrawal Payout Capped at Initiation-Time Value Causes Systematic Loss of Yield Accrued During Delay - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`_calculatePayoutAmount()` returns `min(expectedAssetAmount, currentReturn)`, capping the payout at the stale initiation-time price even when rsETH has appreciated. Because the full `rsETHUnstaked` is always burned regardless of which branch is taken, every ordinary withdrawer who experiences any rsETH price appreciation during the delay window permanently loses the yield that accrued on their position.

## Finding Description

At initiation, `expectedAssetAmount` is recorded using the rsETH price at that moment:

```solidity
// L168
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
// L173
assetsCommitted[asset] += expectedAssetAmount;
```

At unlock time, `_calculatePayoutAmount` (L824–835) computes the current fair value but discards it when it exceeds the initiation-time estimate:

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

`_unlockWithdrawalRequests` (L802–807) then burns the **full** rsETH while only unlocking the minimum:

```solidity
assetsCommitted[asset] -= request.expectedAssetAmount;
request.expectedAssetAmount = payoutAmount;   // overwritten with min
rsETHAmountToBurn += request.rsETHUnstaked;   // full rsETH burned
assetAmountToUnlock += payoutAmount;           // only min unlocked
```

The burn and vault redeem are executed at L305–307:

```solidity
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
unstakingVault.redeem(asset, assetAmountUnlocked);
```

The delta between `currentReturn` and `payoutAmount` remains in the unstaking vault and is never returned to the user. No existing guard compensates the user for the discarded yield; the `min` logic is the only branch, and it unconditionally favours the protocol over the withdrawer whenever the price has risen.

## Impact Explanation

rsETH is a yield-bearing token whose price is designed to increase monotonically as EigenLayer staking rewards accrue. Every withdrawal where `updateRSETHPrice()` has been called between initiation and unlock — the normal, expected case — results in the user receiving less than the current fair value of their burned rsETH. The discarded amount is accrued yield that belongs to the withdrawer. This maps directly to the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation

No attacker capital, privileged access, or special conditions are required. Any unprivileged user who calls `initiateWithdrawal()` followed by `completeWithdrawal()` after the default 8-day delay (57,600 blocks, L94) is affected whenever the rsETH price has increased in the interim. Because operators are incentivised to call `updateRSETHPrice()` regularly, this condition holds for virtually every withdrawal. The loss scales linearly with withdrawal size and delay duration (up to the 16-day maximum).

## Recommendation

Replace the `min` with the current fair value so the user receives the full worth of their burned rsETH:

```solidity
function _calculatePayoutAmount(
    WithdrawalRequest storage request,
    uint256 rsETHPrice,
    uint256 assetPrice
) private view returns (uint256) {
    return (request.rsETHUnstaked * rsETHPrice) / assetPrice;
}
```

If a slippage cap is desired for operational reasons (e.g., to bound vault exposure), apply it symmetrically — for example, cap at `expectedAssetAmount * (1 + maxSlippage)` — rather than hard-capping at the stale initiation-time value. Any cap must be paired with a partial rsETH refund proportional to the unclaimed yield, so the user is not penalised for normal price appreciation.

## Proof of Concept

1. rsETH price at `t=0`: 1.05 ETH/rsETH. User calls `initiateWithdrawal(stETH, 100e18)`. `expectedAssetAmount = 100 × 1.05 / 1.0 = 105 stETH`. 100 rsETH transferred to contract.
2. 8 days pass (57,600 blocks). Operator calls `updateRSETHPrice()`; rsETH price is now 1.06 ETH/rsETH (routine yield accrual).
3. Operator calls `unlockQueue`. `_calculatePayoutAmount` computes `currentReturn = 100 × 1.06 / 1.0 = 106 stETH`. Since `105 < 106`, returns `105`.
4. `rsETHAmountToBurn += 100e18` (full burn); `assetAmountToUnlock += 105e18` (capped payout).
5. `burnFrom(address(this), 100e18)` executes; `unstakingVault.redeem(stETH, 105e18)` executes.
6. User calls `completeWithdrawal` and receives 105 stETH. The 1 stETH of yield accrued during the delay remains in the vault and is never disbursed to the user.
