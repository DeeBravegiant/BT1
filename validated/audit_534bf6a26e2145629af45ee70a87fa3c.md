Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Per-Deposit Amount Validation — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch comparison, checking only whether the running total already exceeds the limit rather than whether `totalAssetDeposits + amount` would exceed it. Any unprivileged caller can invoke `depositETH` with an arbitrarily large `msg.value` and push total ETH deposits well beyond `depositLimitByAsset[ETH]` in a single transaction. The deposit cap — the protocol's primary risk-control mechanism for ETH — is never enforced at deposit time.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the function reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC20
}
``` [1](#0-0) 

The ETH branch at line 679 evaluates `totalAssetDeposits > limit`, which returns `false` (deposit allowed) whenever the running total is at or below the limit — regardless of how large `amount` is. The ERC20 branch at line 681 correctly evaluates `totalAssetDeposits + amount > limit`.

The call chain is:

`depositETH` (line 87) → `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, ...)` → `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)` [2](#0-1) 

`msg.value` is passed as `amount` but is silently discarded inside the ETH branch. No other guard in `depositETH` or `_beforeDeposit` enforces the cap. After the deposit succeeds, `getAssetCurrentLimit` (lines 402–409) retroactively returns `0`, but the limit was never enforced at deposit time. [3](#0-2) 

## Impact Explanation
**Low — Contract fails to deliver promised returns.**

The deposit limit is the protocol's stated mechanism for controlling ETH exposure. Bypassing it means `getTotalAssetDeposits(ETH)` can exceed `depositLimitByAsset[ETH]` by an arbitrary margin in one call, violating the invariant the rest of the system (e.g., `getAssetCurrentLimit`) assumes. Deposited funds are not stolen or frozen, but the protocol accepts more ETH than operators configured it to accept, potentially exceeding safe delegation capacity to EigenLayer strategies.

## Likelihood Explanation
`depositETH` is `external payable` with no access control beyond `whenNotPaused` and `onlySupportedAsset`. [4](#0-3) 

The only precondition is that `totalAssetDeposits[ETH] ≤ depositLimitByAsset[ETH]` at call time, which is the normal operating state. No flash loans, front-running, or privileged keys are required. Any user with sufficient ETH can trigger this.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Configure `depositLimitByAsset[ETH_TOKEN] = 100_000 ether`; assume `getTotalAssetDeposits(ETH_TOKEN) = 99_999 ether`.
2. Attacker calls `depositETH{value: 50_000 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50_000 ether)` evaluates `99_999 ether > 100_000 ether` → `false` → deposit proceeds.
4. `_mintRsETH` mints rsETH for the full 50,000 ETH.
5. Post-call: `getTotalAssetDeposits(ETH_TOKEN) = 149_999 ether` — 49,999 ETH above the configured limit. `getAssetCurrentLimit(ETH_TOKEN)` returns `0`, confirming the limit was breached but was never enforced.

**Foundry fuzz test sketch:**
```solidity
function testFuzz_ethDepositBypassesLimit(uint96 excess) public {
    vm.assume(excess > 0);
    uint256 limit = lrtConfig.depositLimitByAsset(ETH_TOKEN);
    // bring deposits to exactly the limit via prior deposits
    // then deposit `excess` more
    depositPool.depositETH{value: excess}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), limit);
}
```
