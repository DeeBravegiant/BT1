### Title
Missing Output Verification for `AppPublicKey` CKD Variant Allows Single Byzantine Participant to Submit Arbitrary CKD Response — (`crates/contract/src/lib.rs`, `crates/threshold-signatures/src/confidential_key_derivation/protocol.rs`)

### Summary

The `respond_ckd` contract function performs a cryptographic pairing-based output check only for the `AppPublicKeyPV` variant. For the `AppPublicKey` (non-PV, legacy) variant the match arm is explicitly empty. Simultaneously, `do_ckd_coordinator` in the non-PV protocol has no `aggregated_output_check` call. Any single attested participant can therefore call `respond_ckd` with a fully crafted `(big_y, big_c)` pair for any pending `AppPublicKey` CKD request and the contract will resolve the yield with that attacker-controlled output, bypassing the threshold security guarantee entirely.

### Finding Description

**Contract-level gap — `respond_ckd`**

In `crates/contract/src/lib.rs` the match on `app_public_key` is:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

The `AppPublicKey` arm is a no-op. The only guards before this point are:
- `assert_caller_is_signer()` — caller must be the transaction signer
- `assert_caller_is_attested_participant_and_protocol_active()` — caller must be an attested participant [2](#0-1) 

There is no check that the caller is the designated leader/coordinator for the request. Any single attested participant can call `respond_ckd` with arbitrary `(big_y, big_c)` for any pending `AppPublicKey` request.

**Protocol-level gap — `do_ckd_coordinator` (non-PV)**

In `crates/threshold-signatures/src/confidential_key_derivation/protocol.rs`, after aggregating participant shares the coordinator simply returns the result with no verification:

```rust
async fn do_ckd_coordinator(...) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) = compute_signature_share(...)?;
    let waitpoint = chan.next_waitpoint();
    for (_, participant_output) in recv_from_others::<CKDOutput>(...).await? {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))   // ← no aggregated_output_check
}
``` [3](#0-2) 

Compare with the PV variant (`protocol_pv.rs`), which explicitly calls `aggregated_output_check` before returning:

```rust
if !aggregated_output_check(&ckd_output, app_pk, &key_pair.public_key, &hash_point) {
    return Err(ProtocolError::AssertionFailed("CKD output failed to verify".to_string()));
}
``` [4](#0-3) 

**Why the check is structurally absent for `AppPublicKey`**

`ckd_output_check` requires a G2 key (`app_pk2`) to compute the pairing `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`: [5](#0-4) 

The `AppPublicKey` variant carries only a G1 point — no G2 key is available — so the pairing check is structurally impossible. The `AppPublicKeyPV` variant carries both `pk1` (G1) and `pk2` (G2), enabling the check. This is the documented design difference ("Privately verifiable (legacy)" vs. "Publicly verifiable"). [6](#0-5) 

The consequence is that for `AppPublicKey` requests the contract has no mechanism to enforce that the submitted `(big_y, big_c)` is the correct threshold-aggregated output.

**Existing test confirms the gap**

The existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` passes `[1u8; 48]` and `[2u8; 48]` as `big_y`/`big_c` — clearly invalid curve points — and the contract accepts them without complaint: [7](#0-6) 

### Impact Explanation

A single Byzantine attested participant can:
1. Observe any pending `AppPublicKey` CKD request on-chain.
2. Call `respond_ckd` with a crafted `CKDResponse { big_y: X, big_c: Y }` of their choice.
3. The contract resolves the yield and delivers the attacker-controlled `(big_y, big_c)` to the requesting application.

The application then computes `big_c − app_sk · big_y`, which is an attacker-controlled G1 point rather than the correct `msk · H(pk ‖ app_id)`. The threshold security guarantee — that at least `t` honest participants must contribute — is completely bypassed for this variant. This constitutes unauthorized confidential key derivation output without the required participant authorization.

### Likelihood Explanation

- The attacker needs only to be a single attested participant (legitimate MPC node), which is below the signing threshold.
- No complex cryptographic forgery is required; the attacker simply calls a public contract function with arbitrary byte values.
- The `AppPublicKey` variant is the legacy/default format and is actively used in production (the CLI example defaults to it when `--publicly-verifiable` is not passed).
- The attack is race-condition-free: the first `respond_ckd` call for a given request wins; a Byzantine participant can front-run the honest coordinator.

### Recommendation

1. **Deprecate and remove `AppPublicKey`** in favour of `AppPublicKeyPV`, which has a sound on-chain output check. Migrate all callers to the PV variant.
2. If `AppPublicKey` must be retained for backwards compatibility, add a coordinator-binding mechanism: record the identity of the node that first claimed the request (e.g., via a `claim_ckd` step) and enforce in `respond_ckd` that only that node may submit the response.
3. Alternatively, require callers of `request_app_private_key` with `AppPublicKey` to also supply a G2 component (upgrading them silently to the PV path), or reject `AppPublicKey` requests at the contract level.

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
