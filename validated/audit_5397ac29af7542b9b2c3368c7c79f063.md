### Title
`#[payable]` `rlp_execute` silently absorbs attached deposits on early-return paths, permanently locking caller NEAR — (`runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

---

### Summary

The `rlp_execute` entry point of the ETH-implicit wallet contract is marked `#[payable]`, so any caller may attach a NEAR deposit. Two code paths return `PromiseOrValue::Value(...)` (a successful function-call outcome) without ever refunding the deposit: (1) the `has_in_flight_tx` guard at the top of the function, and (2) the catch-all `Err(e)` arm that handles user errors and unregistered-relayer errors. In both cases the deposit is silently credited to the wallet contract's balance and is irrecoverable by the original caller.

---

### Finding Description

`rlp_execute` is declared `#[payable]`:

```rust
#[payable]
pub fn rlp_execute(&mut self, target: AccountId, tx_bytes_b64: String)
    -> PromiseOrValue<ExecuteResponse>
```

**Path 1 — `has_in_flight_tx` early return (lines 97-105)**

```rust
if self.has_in_flight_tx {
    return PromiseOrValue::Value(ExecuteResponse {
        success: false,
        ...
    });
}
```

The function returns a value (not a panic), so the NEAR runtime does **not** auto-refund the deposit. No transfer back to `predecessor_account_id` is issued. The deposit is absorbed into the wallet contract's balance.

**Path 2 — `Err(e)` catch-all (line 126)**

```rust
Err(e) => PromiseOrValue::Value(e.into()),
```

This arm is reached for `Error::User(_)` (e.g. wrong nonce, unsupported action, invalid key, `AccountNonceExhausted`) and for `Error::Relayer(_)` when the signer is not the wallet account itself. Again, a value is returned, not a panic, so no automatic deposit refund occurs.

The `CallerDeposit` / `rlp_execute_callback` refund mechanism is only reachable when `inner_rlp_execute` returns `Ok(promise)` **and** the downstream cross-contract call subsequently fails. It is never reached in either early-return path above.

`CallerDeposit::new` is constructed inside `inner_rlp_execute` (line 345), which is never called in Path 1, and whose result is discarded in Path 2 because the `Err` is propagated before any promise is scheduled.

---

### Impact Explanation

An external caller who attaches any non-zero NEAR deposit to `rlp_execute` and hits either early-return path loses that deposit permanently (from their perspective). The tokens are credited to the wallet contract's balance. The wallet contract exposes no withdrawal function; only the wallet owner (via a signed Ethereum transaction through `rlp_execute` itself) can move those funds out. The caller has no recourse.

The corrupted protocol value is the caller's **NEAR account balance**: it is decremented by the attached deposit with no corresponding credit or refund receipt.

---

### Likelihood Explanation

**Path 1** is reachable by any unprivileged external caller with no special knowledge. During normal wallet operation, `has_in_flight_tx` is `true` for the entire duration of any in-flight cross-contract call. A caller who attaches a deposit and submits a transaction during that window loses their funds. This window is predictable and observable on-chain.

**Path 2** is reachable by any caller who submits a transaction that triggers a user-level error (wrong nonce, unsupported action type, malformed RLP, exhausted nonce space). These conditions are easy to trigger accidentally or deliberately.

---

### Recommendation

1. **Remove `#[payable]`** from `rlp_execute` if attached deposits from external callers are not a required feature for the early-return paths. The NEAR SDK will then reject any call that attaches a non-zero deposit before execution begins, preventing the loss entirely.

2. If `#[payable]` must be retained, add an explicit refund in every early-return path:

```rust
if self.has_in_flight_tx {
    let deposit = env::attached_deposit();
    if deposit > NearToken::from_yoctonear(0) {
        Promise::new(env::predecessor_account_id()).transfer(deposit);
    }
    return PromiseOrValue::Value(ExecuteResponse { ... });
}
```

Apply the same pattern to the `Err(e)` arm and to all early-return branches in `address_check_callback` and `nep_141_storage_balance_callback` that receive a `caller_deposit` argument but do not forward it.

---

### Proof of Concept

1. Deploy the wallet contract (ETH-implicit account, e.g. `0xabc...def.near`).
2. Submit a valid signed Ethereum transaction via `rlp_execute` that triggers a long-running cross-contract call (e.g. an ERC-20 transfer requiring a `storage_balance_of` check). `has_in_flight_tx` is now `true`.
3. From a second account, call `rlp_execute` on the same wallet contract with `attached_deposit = 5 NEAR` and any valid arguments.
4. The function hits the `has_in_flight_tx` guard (line 97) and returns `PromiseOrValue::Value(...)`.
5. Inspect the second account's balance: it has decreased by 5 NEAR. Inspect the wallet contract's balance: it has increased by 5 NEAR. No refund receipt is produced.
