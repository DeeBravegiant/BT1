### Title
Queued Future-Round Proposals Bypass Timestamp Lower-Bound Validation, Allowing Stale Block Timestamps and L1 Gas Prices to Be Committed — (File: `crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

---

### Summary

When `validate_proposal` receives a `ProposalInit` for a **future round**, it stores the proposal in `queued_proposals` with no timestamp validation. When the round eventually advances in `set_height_and_round`, the stale proposal is dequeued and validated. The timestamp check in `is_proposal_init_valid` only verifies that the timestamp is not too far **in the future** (`<= now + block_timestamp_window_seconds`); it has **no lower bound against the current wall-clock time**. A malicious proposer can set `timestamp = last_block_timestamp + 1` (the minimum valid value), queue the proposal for a future round, and have it accepted minutes later with stale L1 gas prices and a stale block timestamp embedded in the committed block.

---

### Finding Description

**Queuing path — no timestamp check:**

In `validate_proposal`, proposals for future rounds are stored without any timestamp validation:

```rust
std::cmp::Ordering::Greater => {
    trace!("Queueing proposal for future round.");
    self.queued_proposals
        .insert(init.round, ((init, timeout, content_receiver), fin_sender));
    fin_receiver
}
``` [1](#0-0) 

**Dequeue path — only upper-bound timestamp check:**

When `set_height_and_round` advances to the queued round, the proposal is dequeued and `is_proposal_init_valid` is called. That function enforces:

```rust
if init_proposed.timestamp < last_block_timestamp { ... }   // lower bound: >= previous block
if init_proposed.timestamp > now + block_timestamp_window_seconds { ... }  // upper bound: not too far in future
``` [2](#0-1) 

There is **no check** of the form `init_proposed.timestamp >= now - staleness_window`. A proposal with `timestamp = last_block_timestamp + 1` (set minutes ago) passes both checks when dequeued.

**L1 gas price validation uses the stale timestamp:**

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,   // ← stale timestamp, not current time
    proposal_init_validation.previous_proposal_init.as_ref(),
    gas_price_params,
)
.await;
``` [3](#0-2) 

Because both proposer and validator look up L1 prices at the same stale timestamp, the price margin check passes. The block is then committed with the stale timestamp and stale L1 gas prices.

**Production configuration confirms the window is 1 second (upper-bound only):** [4](#0-3) 

The parameter description says "Maximum allowed deviation (seconds) of a proposed block's timestamp from the current time," but the implementation only enforces the upper half of that deviation.

**Dequeue site in `set_height_and_round` constructs `ProposalInitValidation` with the same missing lower bound:** [5](#0-4) 

---

### Impact Explanation

The committed block carries:

1. **Stale `block_timestamp`** — Smart contracts using the `get_block_timestamp()` syscall receive a value that may be minutes behind wall-clock time. Time-dependent logic (auctions, vesting schedules, expiry checks) executes incorrectly. This is a wrong execution result from blockifier/syscall logic for accepted input.

2. **Stale L1 gas prices** (`l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, `l1_data_gas_price_wei`) — All transactions in the block are charged fees based on prices from the stale timestamp. If L1 gas has spiked since the proposal was created, users are systematically undercharged for L1 data availability costs, causing an incorrect fee/L1 gas price effect with economic impact.

Both the block timestamp and the gas prices are inputs to `PartialBlockHashComponents` and the final block hash, so the committed state root and block hash reflect the stale values. [6](#0-5) 

---

### Likelihood Explanation

Any validator selected as proposer for a future round can trigger this:

1. Craft a `ProposalInit` with `timestamp = last_block_timestamp + 1` (minimum valid value, maximally stale).
2. Send it for round R+1 while the network is at round R.
3. The proposal is queued without timestamp validation.
4. When round R times out (~9 seconds in production, `timeouts.proposal.base = 9.1`), `set_height_and_round(H, R+1)` is called.
5. The stale proposal is dequeued and accepted.

No special privilege beyond being a scheduled proposer is required. The attack is deterministic and requires no race condition.

---

### Recommendation

Add a staleness lower-bound check inside `is_proposal_init_valid` in `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`:

```rust
// Reject timestamps that are too far in the past relative to the current time.
if now > proposal_init_validation.block_timestamp_window_seconds
    && init_proposed.timestamp < now - proposal_init_validation.block_timestamp_window_seconds
{
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "Timestamp is too stale: now={}, window={}, proposed={}",
            now,
            proposal_init_validation.block_timestamp_window_seconds,
            init_proposed.timestamp
        ),
    ));
}
```

Alternatively, validate the timestamp at **queuing time** (before inserting into `queued_proposals`) using the current clock, and reject proposals whose timestamps are already outside the acceptable window. This mirrors the "deadline parameter" fix recommended in the original DeFi report.

### Proof of Concept

1. Network is at height H, round R. Last committed block has timestamp `T_prev`. Current wall-clock time is `T_now = T_prev + 60` (60 seconds have elapsed).
2. Malicious validator is scheduled as proposer for round R+1.
3. Proposer crafts `ProposalInit { timestamp: T_prev + 1, l1_gas_price_fri: <price at T_prev+1>, ... }`.
4. Proposer sends the proposal for round R+1 to all validators.
5. Each validator calls `validate_proposal(init, timeout, content_receiver)` with `init.round = R+1 > current_round = R`.
6. The proposal is inserted into `queued_proposals` **without timestamp validation** (line 875–876).
7. Round R times out (~9 seconds). `set_height_and_round(H, R+1)` is called.
8. The queued proposal is dequeued. `is_proposal_init_valid` is called with `now = T_prev + 69`.
9. Check 1: `T_prev + 1 >= T_prev` ✓
10. Check 2: `T_prev + 1 <= (T_prev + 69) + 1` ✓ — passes because the stale timestamp is well below `now + 1`.
11. L1 gas prices fetched at `T_prev + 1` match the proposer's prices ✓ — both sides use the same stale timestamp.
12. Block H is committed with `block_timestamp = T_prev + 1` and L1 gas prices from `T_prev + 1`.
13. All transactions in block H are charged fees based on L1 gas prices from 69 seconds ago. If L1 gas spiked during that window, users are systematically undercharged.
14. Smart contracts in block H receive `get_block_timestamp() = T_prev + 1`, which is ~69 seconds behind wall-clock time, causing incorrect behavior in any time-dependent contract logic.
