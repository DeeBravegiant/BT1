Audit Report

## Title
Unbounded Per-User Withdrawal Queue Entries Enable Temporary Freezing of Subsequent Users' Funds — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`initiateWithdrawal` imposes no cap on how many times a single address may call it, allowing any rsETH holder to flood the global FIFO withdrawal queue with arbitrarily many dust-sized entries. Because `_unlockWithdrawalRequests` advances `nextLockedNonce[asset]` strictly in sequence with no skip mechanism, every legitimate user whose request lands after the attacker's entries has their rsETH locked in the contract until the operator exhausts every preceding entry, one batch at a time.

## Finding Description
`initiateWithdrawal` (L150–178) transfers rsETH from the caller and appends a new entry to the global queue via `_addUserWithdrawalRequest`. The only admission guard is a minimum-amount check against `minRsEthAmountToWithdraw[asset]`, which is a plain mapping that defaults to `0`. When it is zero, the effective floor is 1 wei of rsETH (the `rsETHUnstaked == 0` branch). There is no per-user request count limit.

Each call increments the global `nextUnusedNonce[asset]` counter and stores a `WithdrawalRequest` keyed by `getRequestId(asset, nonce)`. The operator's `unlockQueue` path calls `_unlockWithdrawalRequests`, which iterates from `nextLockedNonce[asset]` to `firstExcludedIndex` in strict ascending order. The loop has two early-exit conditions — withdrawal delay not elapsed and insufficient available assets — but neither allows skipping a nonce; `nextLockedNonce[asset]` is only advanced past entries that are fully processed. There is no administrative function to cancel, skip, or reorder queue entries.

A user with even a small rsETH balance can submit thousands of 1-wei requests across many transactions. Each request commits a negligible `expectedAssetAmount` to `assetsCommitted[asset]` (L173), and the attacker recovers that asset when their requests are eventually processed, so the net cost is gas only. Any legitimate user who calls `initiateWithdrawal` after the attacker receives a nonce beyond all attacker entries and cannot call `completeWithdrawal` (which checks `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]` at L707) until the operator has advanced `nextLockedNonce[asset]` past every preceding dust entry.

## Impact Explanation
**Medium — Temporary freezing of funds.** Legitimate users' rsETH is transferred into the contract at `initiateWithdrawal` time (L166) and cannot be recovered until `completeWithdrawal` succeeds. With a large number of preceding dust entries, the operator must issue many batched `unlockQueue` calls (each bounded by block gas limit) before `nextLockedNonce[asset]` reaches the legitimate user's nonce. During this period the user's rsETH and the corresponding underlying asset are inaccessible. The duration is attacker-controlled and proportional to the number of dust entries inserted.

## Likelihood Explanation
**Medium.** The attack is permissionless — any rsETH holder qualifies. `minRsEthAmountToWithdraw` is unset at deployment (defaults to 0), making the minimum 1 wei. Even after an admin sets a non-zero minimum, a participant with a moderate rsETH balance can create thousands of entries. The attacker's capital is fully returned when their requests are processed; the only sunk cost is gas. The attack can be repeated across multiple assets and is not detectable until the queue is already bloated.

## Recommendation
1. **Enforce a non-zero `minRsEthAmountToWithdraw` at initialization** and add a setter guard preventing it from being set to zero.
2. **Cap pending requests per user per asset**: add `require(userAssociatedNonces[asset][msg.sender].length() < MAX_REQUESTS_PER_USER)` inside `initiateWithdrawal`.
3. **Add an operator skip/cancel function** that can advance `nextLockedNonce[asset]` past a specific nonce (returning the committed rsETH to the requester or burning it), so the queue cannot be permanently stalled by malicious entries.

## Proof of Concept
1. Deploy `LRTWithdrawalManager`; `minRsEthAmountToWithdraw[ETH]` is `0` by default.
2. Attacker holds N wei of rsETH; calls `initiateWithdrawal(ETH, 1, "")` N times across multiple transactions, each advancing `nextUnusedNonce[ETH]` by 1 and committing ~1 wei of `assetsCommitted[ETH]`.
3. Legitimate user calls `initiateWithdrawal(ETH, 1 ether, "")` — succeeds, placed at nonce N.
4. Operator calls `unlockQueue(ETH, batchSize, ...)` repeatedly; each call advances `nextLockedNonce[ETH]` by at most `batchSize` entries, all belonging to the attacker.
5. Legitimate user calls `completeWithdrawal(ETH, "")` — reverts with `WithdrawalLocked` because `N >= nextLockedNonce[ETH]` until all N attacker entries are processed.
6. Foundry fuzz/invariant test: fuzz the number of attacker `initiateWithdrawal` calls before a legitimate user's call; assert that `nextLockedNonce[ETH]` must equal N before `completeWithdrawal` can succeed for the legitimate user.
