Audit Report

## Title
Missing Staleness Check in `CrossChainRateReceiver.getRate()` Allows Stale Rate to Drive rsETH Minting — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver` stores a `lastUpdated` timestamp that is written on every `lzReceive()` call but is never read for any validation. `getRate()` unconditionally returns the stored `rate` regardless of age. All pool contracts (`RSETHPoolV2`, `RSETHPoolV3`) delegate their rate lookup to this function with no additional freshness guard, meaning deposits can mint rsETH at a rate that is arbitrarily out of date.

## Finding Description

`CrossChainRateReceiver.lzReceive()` is the sole update path for the stored rate. It sets both `rate` and `lastUpdated` when a LayerZero message arrives: [1](#0-0) 

`getRate()` returns `rate` with no reference to `lastUpdated`: [2](#0-1) 

A grep across the entire repo confirms `lastUpdated` is **written** in `CrossChainRateReceiver.sol` (line 16, line 97) and `CrossChainRateProvider.sol` (line 17, line 92) but **never read** for any validation — it is purely informational. [3](#0-2) 

`RSETHPoolV2.getRate()` and `RSETHPoolV3.getRate()` both delegate directly to `IOracle(rsETHOracle).getRate()` with no additional freshness check: [4](#0-3) [5](#0-4) 

The deposit flow in `RSETHPoolV2`: [6](#0-5) 

calls `viewSwapRsETHAmountAndFee()`, which computes: [7](#0-6) 

The formula `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` uses the stale rate directly. The same pattern exists in `RSETHPoolV3.viewSwapRsETHAmountAndFee()`: [8](#0-7) 

No existing guard in any of these contracts checks `block.timestamp - lastUpdated` before using the rate.

## Impact Explanation

rsETH is a liquid staking token whose ETH exchange rate monotonically increases as staking rewards accrue. A stale (lower) rate causes `amountAfterFee * 1e18 / rsETHToETHrate` to mint **more** rsETH per ETH than the current backing warrants, diluting existing holders. This matches the allowed impact: **Low — contract fails to deliver promised returns, but doesn't lose value**. No direct ETH is drained from the pool, but depositors receive an incorrect rsETH amount relative to the protocol's promised exchange rate.

## Likelihood Explanation

LayerZero message delivery is not guaranteed. Network congestion, dropped messages, or the off-chain keeper failing to call `updateRate()` on the provider are all realistic operational conditions. No privileged access is required — any depositor calling `deposit()` while the rate is stale triggers the impact. The longer the gap since the last `lzReceive()`, the larger the rate deviation. This is an expected operational scenario for any cross-chain oracle system.

## Recommendation

Add a configurable `maxStaleness` threshold and revert in `getRate()` if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(lastUpdated != 0, "Rate not initialized");
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
``` [2](#0-1) 

## Proof of Concept

```solidity
// Foundry fuzz test
function testStaleRateMintsBadAmount(uint256 elapsed) public {
    elapsed = bound(elapsed, 7 days, 365 days);

    // Deploy RSETHRateReceiver; lzReceive sets rate = 1.05e18 at t=0
    // No further lzReceive calls are made
    vm.warp(block.timestamp + elapsed);

    // Rate is still 1.05e18 — no revert, no staleness check
    uint256 rate = receiver.getRate();
    assertEq(rate, 1.05e18);

    // Pool mints at stale rate; real rate may be 1.10e18 after rewards
    (uint256 rsETHAmount,) = pool.viewSwapRsETHAmountAndFee(1 ether);
    // rsETHAmount computed from 1.05e18 instead of 1.10e18
    // depositor receives ~4.5% more rsETH than current backing warrants
}
```

The test requires no privileged setup: deploy `RSETHRateReceiver`, call `lzReceive` once to initialize the rate, warp forward, and call `getRate()` — it returns the stale value without reverting. Connecting this to `RSETHPoolV2.deposit()` on a local fork confirms the over-minting. [9](#0-8)
