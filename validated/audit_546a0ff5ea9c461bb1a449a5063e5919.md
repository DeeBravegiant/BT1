Audit Report

## Title
Permanently inflated `stakedButUnverifiedNativeETH` after beacon-chain slashing of unverified validators causes protocol insolvency — (File: `contracts/NodeDelegator.sol`)

## Summary
When a validator is staked via `stake32Eth()`, `stakedButUnverifiedNativeETH` is incremented by 32 ETH. The sole decrement path is `verifyWithdrawalCredentials()`, which EigenPod explicitly forbids for any validator that already has an exit epoch set. Beacon-chain slashing forces an exit epoch before credential verification is possible, permanently trapping 32 ETH in `stakedButUnverifiedNativeETH` with no admin override or correction mechanism. This inflates `getEffectivePodShares()`, propagates through `_getTotalEthInProtocol()`, and permanently overstates `rsETHPrice`, constituting protocol insolvency.

## Finding Description

**Root cause — only one decrement path exists:**

`stake32Eth()` unconditionally increments the counter: [1](#0-0) 

The only decrement is inside `verifyWithdrawalCredentials()`: [2](#0-1) 

A grep across the entire repository confirms no other write path to `stakedButUnverifiedNativeETH` exists. [3](#0-2) 

**EigenPod blocks credential verification for slashed validators:**

The interface NatSpec states explicitly: *"Validators proven via this method MUST NOT have an exit epoch set already."* [4](#0-3) 

The corresponding revert error `ValidatorIsExitingBeaconChain` is defined in the same interface: [5](#0-4) 

Beacon-chain slashing immediately sets an exit epoch on the validator. Therefore `verifyWithdrawalCredentials()` will always revert for a slashed-but-unverified validator, and `stakedButUnverifiedNativeETH` is never decremented for it.

**Inflated counter propagates to `rsETHPrice`:**

`getEffectivePodShares()` sums `stakedButUnverifiedNativeETH` with `withdrawableShare`: [6](#0-5) 

`getETHDistributionData()` calls `getEffectivePodShares()` across all NDCs: [7](#0-6) 

`_getTotalEthInProtocol()` sums all asset values including this inflated ETH figure: [8](#0-7) 

`_updateRsETHPrice()` computes `rsETHPrice` from the inflated TVL: [9](#0-8) 

**Additional compounding factor:** When the slashed validator's remaining ETH (e.g. 28 ETH) eventually arrives at the EigenPod and a checkpoint is completed, EigenPod credits "any ETH in the pod not already awarded shares" as new shares. This means `withdrawableShare` also increases by ~28 ETH. `getEffectivePodShares()` then returns `32 (stuck) + 28 (checkpoint-credited) = 60 ETH` against a real backing of 28 ETH — a 32 ETH overstatement per slashed validator, larger than the submitted claim's 4 ETH estimate.

**No correction mechanism exists:** There is no admin-callable function to adjust `stakedButUnverifiedNativeETH`, no checkpoint-based reconciliation, and no decay function. The existing downside price-protection circuit breaker in `_updateRsETHPrice()` only triggers on price *decreases*; it does not address upward inflation of `rsETHPrice`. [10](#0-9) 

## Impact Explanation

**Critical — Protocol insolvency.** For each validator slashed before credential verification, the protocol permanently overstates its ETH TVL. The inflated `rsETHPrice` causes new depositors to receive fewer rsETH tokens than the actual backing warrants (they overpay), and allows withdrawers to redeem at an inflated rate, draining real ETH faster than it is backed. With multiple validators, the overstatement compounds with no automatic correction, leaving later withdrawers unable to redeem at par — a classic insolvency cascade matching the allowed impact class.

## Likelihood Explanation

**Medium.** The window between `stake32Eth()` and `verifyWithdrawalCredentials()` spans multiple days (beacon chain activation queue plus proof submission latency). Beacon-chain slashing events (double-signing, surround voting) are rare but have occurred on mainnet. The protocol operates many validators across multiple NodeDelegators, increasing aggregate exposure. No attacker action is required — a single organic slashing event during the unverified window permanently corrupts the accounting.

## Recommendation

1. Add an admin-callable function that decrements `stakedButUnverifiedNativeETH` for a specific validator pubkey after confirming via the EigenPod's `validatorPubkeyToInfo()` that the validator is slashed and its exit epoch is set while its EigenPod status remains `INACTIVE`.
2. Alternatively, track per-validator state (pubkey → 32 ETH entry) so that individual slashed-but-unverified validators can be zeroed out without affecting other unverified validators.
3. Consider reconciling `stakedButUnverifiedNativeETH` against the EigenPod's actual credited shares during checkpoint completion to detect and correct divergence automatically.

## Proof of Concept

1. Operator calls `stake32Eth(pubkey, sig, root)` for validator V → `stakedButUnverifiedNativeETH += 32 ether`.
2. Before `verifyWithdrawalCredentials()` is called, validator V is slashed on the beacon chain (e.g. double-signing). V receives an exit epoch immediately.
3. Operator attempts `verifyWithdrawalCredentials(...)` for V → EigenPod reverts with `ValidatorIsExitingBeaconChain`.
4. `stakedButUnverifiedNativeETH` remains at 32 ETH permanently.
5. V's remaining balance (~28 ETH) eventually arrives at the EigenPod. A subsequent `startCheckpoint` / `verifyCheckpointProofs` cycle credits this ETH as `withdrawableShare` (pod ETH not already awarded shares).
6. `getEffectivePodShares()` now returns `32 + 28 = 60 ETH` for this NDC; correct backing is 28 ETH.
7. `getETHDistributionData()` → `getTotalAssetDeposits(ETH)` → `_getTotalEthInProtocol()` is inflated by 32 ETH.
8. `updateRSETHPrice()` is called (permissionless) → `rsETHPrice` is computed from inflated TVL.
9. New depositors receive fewer rsETH per ETH than warranted; withdrawers redeem at the inflated price, draining real ETH; later withdrawers cannot redeem at par.
