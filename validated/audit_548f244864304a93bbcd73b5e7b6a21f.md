Audit Report

## Title
Token Donation Inflates `getTotalAssetDeposits()` via Raw `balanceOf()`, Enabling Deposit-Limit DoS — (File: `contracts/LRTDepositPool.sol`)

## Summary
`getAssetDistributionData()` measures protocol-held LST balances using raw `IERC20(asset).balanceOf()` calls on the deposit pool and each NodeDelegator. Because ERC-20 transfers are permissionless, any attacker can donate a small amount of a supported LST directly to `LRTDepositPool` or a registered NodeDelegator, inflating `getTotalAssetDeposits()`. This inflated total causes `_checkIfDepositAmountExceedesCurrentLimit()` to permanently return `true`, reverting every subsequent `depositAsset()` call for that token with `MaximumDepositLimitReached` until governance raises the cap.

## Finding Description
`getAssetDistributionData()` computes the protocol balance for a supported LST as:

```solidity
// LRTDepositPool.sol L444
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
// L448
assetLyingInNDCs += IERC20(asset).balanceOf(nodeDelegatorQueue[i]);
```

These values are summed in `getTotalAssetDeposits()` (L385–397) and consumed by `_checkIfDepositAmountExceedesCurrentLimit()`:

```solidity
// L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    // ...
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This check is the sole guard in `_beforeDeposit()` (L661), which gates every `depositAsset()` call. There is no internal accounting variable — the protocol has no way to distinguish protocol-controlled inflows from unsolicited transfers. There is also no sweep function in `LRTDepositPool` for unsolicited ERC-20 tokens; the only outbound ERC-20 path (`transferAssetToNodeDelegator`) moves tokens to a NodeDelegator, which is still counted in `getTotalAssetDeposits()`, providing no relief.

Secondary impact: `LRTOracle._getTotalEthInProtocol()` (L341) also calls `getTotalAssetDeposits()`. A large enough donation inflates `newRsETHPrice` past `highestRsethPrice * (1 + pricePercentageLimit)`, causing `updateRSETHPrice()` to revert for any non-manager caller (L263–264).

## Impact Explanation
**Medium — Temporary freezing of funds.** Once an attacker donates enough of a supported LST to push `totalAssetDeposits` above `depositLimitByAsset`, every `depositAsset()` call for that token reverts. The donated tokens cannot be ejected by the protocol, so the DoS persists until an admin raises the deposit cap. The attacker can immediately re-donate after each admin adjustment, making the freeze effectively continuous at low cost (only the donated amount, which can be as small as 1 wei above the remaining headroom).

## Likelihood Explanation
**High.** Any holder of a supported LST (stETH, ETHx, sfrxETH, etc.) can execute this attack permissionlessly with a single `transfer()` call. No special role, flash loan, or complex setup is required. The attack is repeatable at negligible cost.

## Recommendation
Replace raw `balanceOf()` reads in `getAssetDistributionData()` with an internal accounting variable (`depositedBalance[asset]`) that is incremented exclusively inside `depositAsset()` and `swapAssetForETHWithinDepositPool()`, and decremented on outflows (`transferAssetToNodeDelegator`, `swapETHForAssetWithinDepositPool`). Use this tracked balance instead of `balanceOf(address(this))` and `balanceOf(nodeDelegatorQueue[i])`. This mirrors the ERC-4626 pattern of maintaining a separate `totalAssets` counter rather than relying on `balanceOf(address(this))`.

## Proof of Concept
1. Assume `depositLimitByAsset(stETH) = 10_000e18` and current `getTotalAssetDeposits(stETH) = 9_999e18` (1 stETH of headroom).
2. Attacker calls `stETH.transfer(address(lrtDepositPool), 2e18)`.
3. `getAssetDistributionData()` now returns `assetLyingInDepositPool = 2e18` extra → `getTotalAssetDeposits(stETH) = 10_001e18`.
4. Any user calling `depositAsset(stETH, amount, ...)` hits `_checkIfDepositAmountExceedesCurrentLimit()` → `10_001e18 + amount > 10_000e18` → `true` → revert `MaximumDepositLimitReached`.
5. All stETH deposits are blocked. The donated 2 stETH remain in the pool with no protocol mechanism to eject them.

**Foundry test sketch:**
