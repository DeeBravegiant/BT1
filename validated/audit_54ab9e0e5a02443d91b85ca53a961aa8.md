### Title
`compute_receipt_congestion_gas` Returns Zero for `PromiseResume` Receipts in Delayed Queue, Causing Underestimation of Shard Congestion - (File: `runtime/runtime/src/congestion_control.rs`)

### Summary

`compute_receipt_congestion_gas` unconditionally returns `Gas::ZERO` for `PromiseResume` receipts, even though the code explicitly acknowledges these receipts can land in the delayed receipts queue. This causes `delayed_receipts_gas` in `CongestionInfo` to undercount the true gas load, making the shard appear less congested than it is. Downstream, `incoming_congestion`, `congestion_level`, `outgoing_gas_limit`, and `shard_accepts_transactions` all derive from this underestimated value, so other shards forward more receipts to the affected shard than the protocol intends, and the shard accepts more new transactions than it should.

### Finding Description

`compute_receipt_congestion_gas` in `runtime/runtime/src/congestion_control.rs` dispatches on receipt type and returns `Gas::ZERO` for every non-`Action` variant:

```rust
VersionedReceiptEnum::PromiseResume(_) => {
    // The congestion control MVP does not account for resuming a promise.
    // Unlike `PromiseYield`, it is possible that a promise-resume ends
    // up in the delayed receipts queue.
    // But similar to a data receipt, it would be difficult to find the cost
    // of it without expensive state lookups.
    Ok(Gas::ZERO)
}
```

The comment itself confirms the broken invariant: a `PromiseResume` receipt **can** enter the delayed receipts queue, yet its gas is counted as zero. [1](#0-0) 

Every time a `PromiseResume` receipt is pushed into the delayed queue, `DelayedReceiptQueueWrapper::push` calls `compute_receipt_congestion_gas` to obtain the gas to add to `new_delayed_gas`:

```rust
let gas = compute_receipt_congestion_gas(&receipt, &config)?;
...
self.new_delayed_gas = self.new_delayed_gas.checked_add(gas).ok_or(IntegerOverflowError)?;
``` [2](#0-1) 

The accumulated `new_delayed_gas` is later applied to `CongestionInfo::delayed_receipts_gas`. [3](#0-2) 

`CongestionInfo::incoming_congestion` is computed directly from `delayed_receipts_gas`:

```rust
pub fn incoming_congestion(&self, config: &CongestionControlConfig) -> f64 {
    clamped_f64_fraction(
        self.delayed_receipts_gas(),
        config.max_congestion_incoming_gas.as_gas(),
    )
}
``` [4](#0-3) 

`congestion_level` takes the max of all four sub-metrics including `incoming_congestion`:

```rust
pub fn congestion_level(&self) -> f64 {
    incoming_congestion.max(outgoing_congestion).max(memory_congestion).max(missed_chunks_congestion)
}
``` [5](#0-4) 

`outgoing_gas_limit` and `shard_accepts_transactions` both consume `congestion_level`:

```rust
pub fn outgoing_gas_limit(&self, sender_shard: ShardId) -> Gas {
    let congestion = self.congestion_level();
    if Self::is_fully_congested(congestion) {
        ...Gas::ZERO...
    } else {
        mix_gas(self.config.max_outgoing_gas, self.config.min_outgoing_gas, congestion)
    }
}
``` [6](#0-5) 

When `PromiseResume` receipts with large attached gas accumulate in the delayed queue, `delayed_receipts_gas` is underestimated, `incoming_congestion` is underestimated, and therefore `outgoing_gas_limit` returns a value that is too high and `shard_accepts_transactions` returns `Yes` when it should return `No`.

### Impact Explanation

**Impact: Medium**

Other shards compute their forwarding allowance to the affected shard using `outgoing_gas_limit`. An underestimated congestion level causes them to forward more receipts than the protocol intends. Simultaneously, the affected shard accepts more new transactions than it should. The `receipt_bytes`-based `memory_congestion` metric is still accurate (byte sizes of `PromiseResume` receipts are counted correctly), so the growth is eventually bounded by the memory threshold — but the gas-based incoming congestion signal fails to trigger backpressure in time. The concrete corrupted protocol value is `CongestionInfo::delayed_receipts_gas` stored in the chunk header, which propagates to every shard that reads this shard's congestion info to make forwarding decisions.

### Likelihood Explanation

**Likelihood: Medium**

A `PromiseResume` receipt enters the delayed queue whenever the shard is busy enough that it cannot be processed in the same chunk it arrives. An unprivileged user can deliberately trigger this by:
1. Deploying a contract that calls `promise_yield_create` with high attached gas.
2. Calling `promise_yield_resume` to generate `PromiseResume` receipts targeting a shard that is already under load.

The attacker pays gas proportional to the receipts created, but the underestimation of congestion persists for as long as `PromiseResume` receipts remain in the delayed queue, which can span many chunks under sustained load.

### Recommendation

For each `PromiseResume` receipt pushed into the delayed queue, look up the gas of the corresponding `PromiseYield` receipt (stored under `TrieKey::PromiseYieldReceipt`) and use that value as the congestion gas contribution. If the lookup is considered too expensive, an alternative is to store the expected callback gas in the `PromiseResume` receipt itself at creation time (similar to how `StateStoredReceiptMetadata::congestion_gas` is used for buffered receipts), so that `compute_receipt_congestion_gas` can return the correct value without a trie read.

### Proof of Concept

1. Deploy a contract on shard S that calls `promise_yield_create` attaching 300 Tgas to the callback.
2. Submit enough transactions to shard S to fill its delayed queue (so new receipts are deferred).
3. Call `promise_yield_resume` from another account; the resulting `PromiseResume` receipt is pushed into the delayed queue with `compute_receipt_congestion_gas` returning `Gas::ZERO`.
4. Observe that `CongestionInfo::delayed_receipts_gas` for shard S does not increase despite 300 Tgas of pending work.
5. Other shards query `outgoing_gas_limit(S)` and receive a value higher than they would if the 300 Tgas were counted, forwarding more receipts to S than the protocol intends. [7](#0-6) [5](#0-4) [6](#0-5)

pub(crate) fn compute_receipt_congestion_gas(
    receipt: &Receipt,
    config: &RuntimeConfig,
) -> Result<Gas, IntegerOverflowError> {
    match receipt.versioned_receipt() {
        VersionedReceiptEnum::Action(action_receipt) => {
            // account for gas guaranteed to be used for executing the receipts
            action_receipt_congestion_gas(receipt, config, action_receipt.into())
        }
        VersionedReceiptEnum::Data(_data_receipt) => {
            // Data receipts themselves don't cost gas to execute, their cost is
            // burnt at creation. What we should count, is the gas of the
            // postponed action receipt. But looking that up would require
            // reading the postponed receipt from the trie.
            // Thus, the congestion control MVP does not account for data
            // receipts or postponed receipts.
            Ok(Gas::ZERO)
        }
        VersionedReceiptEnum::PromiseYield(_) => {
            // The congestion control MVP does not account for yielding a
            // promise. Yielded promises are confined to a single account, hence
            // they never cross the shard boundaries. This makes it irrelevant
            // for the congestion MVP, which only counts gas in the outgoing
            // buffers and delayed receipts queue.
            Ok(Gas::ZERO)
        }
        VersionedReceiptEnum::PromiseResume(_) => {
            // The congestion control MVP does not account for resuming a promise.
            // Unlike `PromiseYield`, it is possible that a promise-resume ends
            // up in the delayed receipts queue.
            // But similar to a data receipt, it would be difficult to find the cost
            // of it without expensive state lookups.
            Ok(Gas::ZERO)
        }
        VersionedReceiptEnum::GlobalContractDistribution(_) => Ok(Gas::ZERO),
    }
}

    pub fn congestion_level(&self) -> f64 {
        let incoming_congestion = self.incoming_congestion();
        let outgoing_congestion = self.outgoing_congestion();
        let memory_congestion = self.memory_congestion();
        let missed_chunks_congestion = self.missed_chunks_congestion();

            .max(memory_congestion)
            .max(missed_chunks_congestion)
