Audit Report

## Title
Permanent Freezing of Unclaimed Yield When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` implements a Synthetix-style staking rewards model where `rewardPerToken()` freezes accumulation when `totalKernelStaked == 0`, but the `updateReward` modifier unconditionally advances `updatedAt` to the current timestamp. Any staker holding 100% of `totalKernelStaked` can call `initiateWithdrawal()` for their full balance during an active reward period, causing all rewards accrued during the resulting zero-staked interval to be permanently locked in the contract with no recovery path.

## Finding Description
The root cause is a mismatch between `rewardPerToken()` and the `updateReward` modifier:

`rewardPerToken()` (L408–414) short-circuits to return the frozen `rewardPerTokenStored` when `totalKernelStaked == 0`, accumulating no new rewards. However, the `updateReward` modifier (L232–242) unconditionally sets `updatedAt = lastTimeRewardApplicable()` regardless of whether `totalKernelStaked` is zero. This means the time gap during which no tokens are staked is silently consumed and skipped — the contract advances its internal clock past the zero-staked interval without distributing any rewards for it.

The exploit path:
1. Alice is the sole staker (`totalKernelStaked = 100e18`) during an active reward period (`block.timestamp < finishAt`).
2. Alice calls `initiateWithdrawal(100e18)`. The `updateReward(Alice)` modifier fires first: since `totalKernelStaked` is still 100e18 at that moment, Alice correctly receives her accrued rewards up to this point. Then `balanceOf[Alice] -= 100e18` and `totalKernelStaked -= 100e18` execute (L325–326), reducing `totalKernelStaked` to 0.
3. During the zero-staked interval, every call to any `updateReward`-gated function (e.g., Bob calling `stake()`) fires the modifier with `totalKernelStaked == 0`, freezing `rewardPerTokenStored` and advancing `updatedAt` past the gap.
4. The rewards corresponding to `rewardRate × (duration of zero-staked gap)` are permanently locked in the contract.

The only existing guard is in `notifyRewardAmount()` (L570), which reverts if `totalKernelStaked == 0` at the time of starting a new reward period. This does not prevent `totalKernelStaked` from reaching zero *during* an active period. There is no `recoverERC20` or equivalent function anywhere in the contract, making the locked tokens irrecoverable. The NatSpec comment at L18–22 explicitly acknowledges this behavior and states the only mitigation is an off-chain operational promise — there is zero on-chain enforcement.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens transferred into the contract via `notifyRewardAmount()` that correspond to the zero-staked interval are permanently locked. No staker — past or future — can ever claim them. The loss is proportional to `rewardRate × (duration of zero-staked gap)`, which can be substantial (e.g., 15 days of rewards in the PoC scenario). This matches the allowed impact class "Medium. Permanent freezing of unclaimed yield."

## Likelihood Explanation
**Medium.** The scenario requires a single staker holding 100% of `totalKernelStaked` to call `initiateWithdrawal()` for their full balance during an active reward period. This is realistic in early-stage deployment (as the NatSpec itself acknowledges). The staker is exercising a legitimate, permissionless function — no coordination, no privileged access, and no victim mistake is required. The contract's own documentation confirms the risk exists and that the only mitigation is operational.

## Recommendation
1. **On-chain guard in `initiateWithdrawal()`**: Revert if the withdrawal would reduce `totalKernelStaked` to zero while `block.timestamp < finishAt`.
2. **Admin recovery function**: Add a `recoverRewards()` function callable only after `finishAt` that allows the admin to reclaim reward tokens in excess of what is owed to current stakers.
3. **Alternatively**: In `updateReward`, only advance `updatedAt` when `totalKernelStaked > 0`, so the zero-staked interval is not consumed and rewards can be redistributed in the next period.

## Proof of Concept
```
State: duration = 30 days, rewardRate = 1e18/s, finishAt = T+30d

T=0:   Admin calls notifyRewardAmount(2592000e18)
       totalKernelStaked = 100e18 (Alice is the only staker)

T=15d: Alice calls initiateWithdrawal(100e18)
       → updateReward(Alice) fires: rewardPerTokenStored updated correctly,
         updatedAt = T+15d, rewards[Alice] = 15d × rewardRate (correct)
       → balanceOf[Alice] = 0, totalKernelStaked = 0

T=15d–T+20d: totalKernelStaked == 0
       → Any call to updateReward-gated function freezes rewardPerTokenStored
         and advances updatedAt, consuming the gap silently

T=20d: Bob calls stake(100e18)
       → updateReward(Bob) fires with totalKernelStaked == 0
       → rewardPerTokenStored unchanged, updatedAt = T+20d
       → 5-day gap rewards (5d × 1e18/s = 432000e18 tokens) permanently lost

T=30d: finishAt reached
       Rewards for T+15d → T+30d (15 days × 1e18/s = 1296000e18 tokens)
       permanently locked in contract with no recovery path.
```

Foundry test plan: deploy `KernelDepositPool`, call `notifyRewardAmount` with Alice staked, `warp` to mid-period, have Alice call `initiateWithdrawal` for full balance, `warp` forward, have Bob stake, `warp` to `finishAt`, assert `rewardsToken.balanceOf(contract) > rewards[Bob]` and that the excess is unclaimable by any address.
