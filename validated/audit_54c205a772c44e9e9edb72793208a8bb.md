Audit Report

## Title
Deposit/Withdrawal Asymmetry Allows Temporary Freezing of User Funds - (`contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTConfig.addNewSupportedAsset` registers an asset as supported without setting an EigenLayer strategy. `LRTDepositPool.depositAsset` accepts deposits for any supported ERC20 token regardless of strategy configuration, minting rsETH to the user. However, `LRTWithdrawalManager.initiateWithdrawal` enforces `onlySupportedStrategy(asset)`, which reverts when `assetStrategy[asset] == address(0)`. Users who deposit during the window between asset listing and strategy configuration receive rsETH but cannot initiate any withdrawal for that asset until an admin acts.

## Finding Description
`LRTConfig._addNewSupportedAsset` sets `isSupportedAsset[asset] = true` and pushes to `supportedAssetList`, but never touches `assetStrategy[asset]`, leaving it as `address(0)`.

`LRTDepositPool.depositAsset` is gated only by `onlySupportedERC20Token(asset)`, which checks `isSupportedAsset[asset] == true` and `asset != ETH_TOKEN`. No strategy check exists. A deposit for a newly-listed asset with no strategy succeeds and mints rsETH.

`LRTWithdrawalManager.initiateWithdrawal` applies both `onlySupportedAsset(asset)` and `onlySupportedStrategy(asset)`. The `onlySupportedStrategy` modifier evaluates:

```solidity
if (asset != LRTConstants.ETH_TOKEN && lrtConfig.assetStrategy(asset) == address(0)) {
    revert StrategyNotSupported();
}
```

This unconditionally reverts for any non-ETH asset whose strategy has not yet been set. `instantWithdrawal` is identically gated. There is no alternative withdrawal path for a user holding rsETH backed by a strategy-less asset.

`updateAssetStrategy` requires `UtilLib.checkNonZeroAddress(strategy)`, so once set, a strategy cannot be zeroed out. The only window where `assetStrategy[asset] == address(0)` for a live, depositable asset is between `addNewSupportedAsset` (TIME_LOCK_ROLE) and the first `updateAssetStrategy` (DEFAULT_ADMIN_ROLE) call — two separate governance transactions with no on-chain ordering enforcement.

## Impact Explanation
Users who deposit a newly-listed asset before its strategy is configured receive rsETH but every call to `initiateWithdrawal` or `instantWithdrawal` for that asset reverts with `StrategyNotSupported`. The deposited asset is locked in the protocol with no user-accessible exit path. This constitutes **temporary freezing of funds** (Medium severity). If the admin never sets a strategy or the asset is deprecated before one is assigned, the freeze becomes indefinite, escalating toward permanent freezing of funds.

## Likelihood Explanation
The scenario requires no malicious actor. `addNewSupportedAsset` and `updateAssetStrategy` are separate governance operations with different role requirements (TIME_LOCK_ROLE vs DEFAULT_ADMIN_ROLE), making an atomic same-transaction setup impossible. Any user who deposits during the gap — which may span multiple blocks or longer depending on governance latency — is affected. The deposit succeeds silently with no warning to the user.

## Recommendation
Add `onlySupportedStrategy(asset)` to `LRTDepositPool.depositAsset` so deposits are rejected for assets without a configured strategy, eliminating the asymmetry. Alternatively, remove `onlySupportedStrategy` from `initiateWithdrawal` and gate only the `unlockQueue` operator path on strategy presence, allowing users to always queue a withdrawal regardless of strategy state.

## Proof of Concept
1. Admin calls `LRTConfig.addNewSupportedAsset(tokenX, depositLimit)` — `isSupportedAsset[tokenX] = true`, `assetStrategy[tokenX] = address(0)`.
2. Alice calls `LRTDepositPool.depositAsset(tokenX, amount, minRSETH, "")` — passes `onlySupportedERC20Token`, rsETH minted to Alice.
3. Alice calls `LRTWithdrawalManager.initiateWithdrawal(tokenX, rsETHAmount, "")` — execution reaches `onlySupportedStrategy(tokenX)`, evaluates `lrtConfig.assetStrategy(tokenX) == address(0)` → `true` → reverts with `StrategyNotSupported`.
4. Alice's deposited `tokenX` is locked in the protocol. Her rsETH remains in her wallet but cannot be redeemed for `tokenX` until admin calls `updateAssetStrategy(tokenX, strategyAddr)`.

Foundry test outline:
```solidity
function test_depositWithoutStrategy_blocksWithdrawal() public {
    // 1. Add asset without strategy
    vm.prank(timeLockRole);
    lrtConfig.addNewSupportedAsset(address(tokenX), 1000 ether);

    // 2. Alice deposits
    vm.startPrank(alice);
    tokenX.approve(address(lrtDepositPool), 100 ether);
    lrtDepositPool.depositAsset(address(tokenX), 100 ether, 0, "");
    uint256 rsETHBalance = rsETH.balanceOf(alice);
    assertGt(rsETHBalance, 0);

    // 3. Alice tries to withdraw — must revert
    rsETH.approve(address(lrtWithdrawalManager), rsETHBalance);
    vm.expectRevert(ILRTWithdrawalManager.StrategyNotSupported.selector);
    lrtWithdrawalManager.initiateWithdrawal(address(tokenX), rsETHBalance, "");
    vm.stopPrank();
}
```
