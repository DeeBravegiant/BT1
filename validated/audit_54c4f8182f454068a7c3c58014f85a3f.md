Audit Report

## Title
Stale `rsETHPrice` Read in Deposit Flow Enables Yield Dilution of Existing rsETH Holders — (`contracts/LRTDepositPool.sol`)

## Summary
`LRTOracle.rsETHPrice` is a cached state variable updated only when `updateRSETHPrice()` is explicitly called. Neither `depositAsset()` nor `depositETH()` triggers a price refresh before computing the rsETH mint amount via `getRsETHAmountToMint()`. When staking rewards have accrued since the last price update, the stale (lower) price causes depositors to receive excess rsETH, directly diluting the unclaimed yield of existing holders.

## Finding Description
`LRTOracle` stores the rsETH/ETH exchange rate in the state variable `rsETHPrice` (LRTOracle.sol L28). This value is only updated when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is called explicitly (LRTOracle.sol L87–96). Between calls, `rsETHPrice` is stale.

The deposit flow is:

```
depositAsset() / depositETH()
  → _beforeDeposit()          [private view, LRTDepositPool.sol L648–670]
    → getRsETHAmountToMint()  [LRTDepositPool.sol L506–521]
      → lrtOracle.rsETHPrice()  ← reads stale cached value
```

`_beforeDeposit` is declared `private view` (LRTDepositPool.sol L648–655), so it cannot itself call the state-mutating `updateRSETHPrice()`. Neither `depositAsset()` (L99–118) nor `depositETH()` (L76–93) calls `updateRSETHPrice()` before invoking `_beforeDeposit()`. The mint formula at L520 is:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` fetches a live price from the underlying price oracle, but `lrtOracle.rsETHPrice()` returns the last-written cached value. When rewards have accrued and the true TVL has risen, the denominator is understated, minting more rsETH than the depositor's contribution warrants. No existing check prevents this: `minRSETHAmountExpected` is a depositor-side slippage guard and does not protect existing holders.

## Impact Explanation
When ETH staking rewards accrue between `updateRSETHPrice()` calls, the true rsETH/ETH price rises but `rsETHPrice` remains at its prior (lower) value. A depositor who deposits in this window receives rsETH computed against the stale-low denominator — more rsETH than their ETH contribution is worth at the true price. Upon the next `updateRSETHPrice()` call, the new price is set using the full TVL (including the attacker's deposit) divided by the inflated supply, permanently reducing the price that existing holders can redeem at. The excess rsETH minted to the attacker represents a direct transfer of accrued-but-not-yet-reflected yield from existing holders to the depositor.

**Impact: High — Theft of unclaimed yield**, matching the allowed scope.

## Likelihood Explanation
`updateRSETHPrice()` is called off-chain by keepers on a periodic schedule; it is not atomically enforced within every deposit. The window between reward accrual (e.g., stETH rebase, EigenLayer reward distribution) and the next keeper call is routine and predictable. `updateRSETHPrice()` is `public` and callable by anyone, so an attacker can verify on-chain that the stored price is stale (by comparing `rsETHPrice` against a freshly computed value) and deposit immediately before the keeper updates it. No special role or privilege is required. The attack is repeatable every reward cycle.

## Recommendation
Remove the `view` modifier from `_beforeDeposit` and call `updateRSETHPrice()` at its start, before `getRsETHAmountToMint()` is invoked:

```solidity
function _beforeDeposit(
    address asset,
    uint256 depositAmount,
    uint256 minRSETHAmountExpected
)
    private
    returns (uint256 rsethAmountToMint)
{
    // Refresh price before computing mint amount
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();

    if (depositAmount == 0 || depositAmount < minAmountToDeposit) {
        revert InvalidAmountToDeposit();
    }
    if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
        revert MaximumDepositLimitReached();
    }
    rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
    if (rsethAmountToMint < minRSETHAmountExpected) {
        revert MinimumAmountToReceiveNotMet();
    }
}
```

This ensures every deposit uses a price that reflects current TVL, eliminating the stale-price window.

## Proof of Concept
**State at time T:**
- `rsETHPrice = 1.00e18` (last updated), TVL = 1000 ETH, rsETH supply = 1000.

**Step 1 — Rewards accrue:**
- stETH rebases; true TVL = 1010 ETH. `rsETHPrice` is NOT updated (keeper has not run). True price = 1.01e18.

**Step 2 — Attacker calls `depositAsset(stETH, 10e18, 0, "")`:**
- `getRsETHAmountToMint` computes: `(10e18 * 1e18) / 1.00e18 = 10e18` rsETH minted.
- Correct amount at true price: `10e18 / 1.01e18 ≈ 9.9009e18` rsETH.
- Attacker receives `≈ 0.099e18` excess rsETH.

**Step 3 — Keeper calls `updateRSETHPrice()`:**
- TVL = 1020 ETH (1010 + 10 deposited), supply = 1010 rsETH.
- New price = `1020e18 / 1010 ≈ 1.00990e18`.
- Without the attack, price would be `1010e18 / 1009.9 ≈ 1.01e18`.
- Existing 1000 rsETH holders now hold `1000 * 1.00990 = 1009.90 ETH` instead of `1010 ETH` — a loss of `≈ 0.099 ETH` of accrued yield transferred to the attacker.

**Foundry fork test outline:**
1. Fork mainnet at a block just after a stETH rebase but before the keeper's `updateRSETHPrice()` tx.
2. Assert `lrtOracle.rsETHPrice()` is below the value returned by a simulated `_getTotalEthInProtocol() / rsETH.totalSupply()`.
3. Call `depositAsset` as an unprivileged address.
4. Call `updateRSETHPrice()`.
5. Assert the attacker's rsETH balance, when redeemed, exceeds their deposited ETH value, and that existing holders' redemption value decreased relative to the pre-deposit TVL share.
