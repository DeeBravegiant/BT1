### Title
Deposits from Non-Winning `propose_update` Proposals Are Permanently Lost When Any Update Executes — (`crates/contract/src/update.rs`)

---

### Summary

When `do_update` executes a winning contract/config proposal, it unconditionally clears **all** other pending proposals from storage without refunding the NEAR deposits those proposers paid. Because `UpdateEntry` does not record the proposer's account ID or deposit amount, there is no recovery path. Any participant who proposed a competing update permanently loses their deposit.

---

### Finding Description

`propose_update` requires a deposit proportional to the size of the proposed update:

```rust
// lib.rs ~1308-1330
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
// ...
let id = self.proposed_updates.propose(update);
// Refund only the *excess* above required; the required amount stays in the contract.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
```

The required deposit is computed as:

```rust
// update.rs ~278-298
fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;
    match update {
        Update::Contract(code) => { bytes_used += code.len() as u128; }
        ...
    }
    bytes_used
}
fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
```

For a ~1.5 MB contract WASM (the maximum allowed, as tested in `test_propose_contract_max_size_upload`), this deposit is on the order of **15+ NEAR**.

When `vote_update` reaches threshold and calls `do_update`:

```rust
// update.rs ~195-200
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    // Clear all entries as they might be no longer valid
    self.entries.clear();          // ← all competing proposals deleted
    self.vote_by_participant.clear();
    ...
}
```

All competing proposals are deleted. The `UpdateEntry` struct stores only `update` and `bytes_used` — **not** the proposer's account ID or the deposit amount:

```rust
// update.rs ~132-135
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
```

There is no `cancel_proposal`, `reclaim_deposit`, or any other refund path. The deposits from all non-winning proposals are permanently absorbed into the contract's balance.

---

### Impact Explanation

This is a **Medium** impact: it breaks the production accounting invariant that deposits paid to the contract are either consumed for their stated purpose or refunded. Participants who propose updates that are superseded by a competing proposal suffer a direct, permanent, and unrecoverable loss of NEAR tokens. For contract-code proposals, this loss can be 15+ NEAR per proposal. The contract's balance grows by the sum of all non-winning deposits with no mechanism to redistribute or return them.

This maps to the allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The scenario is a normal operational one. Multiple participants can independently propose different updates simultaneously (the test `test_propose_update_contract_many` explicitly exercises this). Whenever any update reaches threshold, all other proposals are silently cleared. Because participants are incentivized to propose their preferred update and the protocol does not serialize proposals, concurrent proposals are expected in practice.

---

### Recommendation

1. Store the proposer's `AccountId` inside `UpdateEntry` alongside `bytes_used`.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each.
3. Alternatively, add a `cancel_proposal(id: UpdateId)` endpoint that allows the original proposer to withdraw their deposit while the proposal is still pending.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1.5 MB WASM, attaching ~15 NEAR deposit. Proposal ID = 0.
2. Participant B calls `propose_update` with a different 1.5 MB WASM, attaching ~15 NEAR deposit. Proposal ID = 1.
3. Threshold participants vote for proposal 1. `vote_update` calls `do_update(&1, ...)`.
4. `do_update` removes entry 1, then calls `self.entries.clear()` — entry 0 (Participant A's proposal) is deleted with no refund.
5. Participant A's ~15 NEAR is permanently locked in the contract. There is no function to reclaim it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

fn bytes_used(update: &Update) -> u128 {

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;
