Audit Report

## Title
FIFO Withdrawal Queue Break Causes Temporary Freezing of Subsequent Users' rsETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`initiateWithdrawal` admits requests using `getAvailableAssetAmount`, which is based on `lrtDepositPool.getTotalAssetDeposits` (protocol-wide, including EigenLayer-deployed assets), while `unlockQueue` processes requests using only `unstakingVault.balanceOf` (physically returned assets). When the head-of-queue request cannot be covered by vault balance, `_unlockWithdrawalRequests` breaks the loop and leaves `nextLockedNonce` unchanged, permanently blocking all subsequent users from completing their withdrawals until the blocking request is serviced. No `cancelWithdrawal` function exists, leaving affected users with no recovery path.

## Finding Description
**Admission vs. unlock metric mismatch:**
`getAvailableAssetAmount` at line 599–603 computes availability as `lrtDepositPool.getTotalAssetDeposits(asset) - assetsCommitted[asset]`, which includes assets still deployed in EigenLayer strategies. This allows large withdrawal requests to be admitted even when the `LRTUnstakingVault` holds far less.

`_createUnlockParams` at line 849 sets `totalAvailableAssets = unstakingVault.balanceOf(asset)` — only what has been physically returned from EigenLayer. This value is passed as `availableAssetAmount` into `_unlockWithdrawalRequests`.

**FIFO break behavior:**
The unlock loop at lines 790–815 iterates from `nextLockedNonce_` in strict order. At line 800, `if (availableAssetAmount < payoutAmount) break` exits the entire loop when the head-of-queue request cannot be covered. `nextLockedNonce[asset]` is only advanced on successful unlock (line 812). A single large stuck request at nonce N prevents all requests at nonces N+1, N+2, … from ever being unlocked in that call.

**Completion check:**
`_processWithdrawalCompletion` at line 707 reverts with `WithdrawalLocked` when `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]`. If `nextLockedNonce` is stuck at 0, any user with nonce ≥ 1 satisfies this condition and cannot complete.

**No cancel path:**
A grep across the entire repository confirms no `cancelWithdrawal` function exists. Users cannot reclaim their rsETH (transferred into the contract at line 166 during `initiateWithdrawal`) through any public call.

**Exploit flow:**
1. Protocol has 100 ETH total assets: 80 ETH in EigenLayer, 20 ETH in vault.
2. User A calls `initiateWithdrawal` for rsETH worth 80 ETH. Admission passes (`80 < 100 - 0`). `assetsCommitted = 80`. Nonce = 0.
3. User B calls `initiateWithdrawal` for rsETH worth 10 ETH. Admission passes (`10 < 100 - 80`). `assetsCommitted = 90`. Nonce = 1.
4. Operator calls `unlockQueue`; vault has 20 ETH. Loop evaluates nonce 0: `20 < 80` → `break`. `nextLockedNonce` stays at 0.
5. User B calls `completeWithdrawal`: `popFront()` returns nonce 1; `1 >= 0` → `WithdrawalLocked` revert.
6. User B's rsETH remains locked in `LRTWithdrawalManager` with no exit.

## Impact Explanation
User B's rsETH is transferred into `LRTWithdrawalManager` at `initiateWithdrawal` and cannot be recovered until User A's larger request is eventually serviced by the operator unstaking sufficient assets from EigenLayer. The freeze duration is unbounded from User B's perspective and entirely outside their control. This constitutes **temporary freezing of funds** (Medium severity).

## Likelihood Explanation
No adversarial intent is required. The gap between `getTotalAssetDeposits` (admission) and `unstakingVault.balanceOf` (unlock) is the normal operational state of the protocol — the majority of assets are routinely deployed in EigenLayer. Any large withdrawal request followed by one or more smaller requests from other users triggers this condition under ordinary concurrent usage. The condition is persistent and repeatable.

## Recommendation
1. **Add a `cancelWithdrawal` function** allowing a user to remove their own still-locked (nonce < `nextLockedNonce` not yet reached) withdrawal request and receive their rsETH back, reversing the `assetsCommitted` accounting.
2. **Alternatively**, change the `break` at line 800 to `continue` so that smaller subsequent requests are not blocked by a single large head-of-queue entry that cannot currently be covered. This requires tracking skipped nonces separately to maintain accounting integrity.
3. **Additionally**, align the admission check to use `unstakingVault.balanceOf` (or a conservative estimate) rather than `getTotalAssetDeposits`, so that requests are only admitted when they can realistically be serviced.

// Foundry fork test outline

function test_FIFOQueueBlocking() public {
    // Setup: 100 ETH total assets, 80 ETH in EigenLayer, 20 ETH in unstakingVault

    // Step 1: User A initiates withdrawal for rsETH worth 80 ETH
    // L162-176: admission check passes (80 < getTotalAssetDeposits=100 - assetsCommitted=0)
    // rsETH transferred to LRTWithdrawalManager; assetsCommitted[ETH] = 80; nonce 0 assigned
    vm.prank(userA);
    withdrawalManager.initiateWithdrawal(ETH, rsETH_80, "");

    // Step 2: User B initiates withdrawal for rsETH worth 10 ETH
    // admission check passes (10 < 100 - 80 = 20)
    // rsETH transferred; assetsCommitted[ETH] = 90; nonce 1 assigned
    vm.prank(userB);
    withdrawalManager.initiateWithdrawal(ETH, rsETH_10, "");

    // Step 3: Operator calls unlockQueue; vault only has 20 ETH
    // _createUnlockParams: totalAvailableAssets = unstakingVault.balanceOf(ETH) = 20 ETH (L849)
    // _unlockWithdrawalRequests loop:
    //   nonce 0: payoutAmount=80 ETH > 20 ETH → break (L800)
    //   nextLockedNonce[ETH] remains 0
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH, 2, minAsset, minRsETH, maxAsset, maxRsETH);
    assertEq(withdrawalManager.nextLockedNonce(ETH), 0);

    // Step 4: User B attempts to complete withdrawal
    // _processWithdrawalCompletion: popFront() = 1; 1 >= nextLockedNonce(0) → WithdrawalLocked (L707)
    vm.prank(userB);
    vm.expectRevert(LRTWithdrawalManager.WithdrawalLocked.selector);
    withdrawalManager.completeWithdrawal(ETH, "");

    // User B's rsETH is locked in the contract with no cancel path
    // Confirmed: no cancelWithdrawal function exists anywhere in the contract
}
