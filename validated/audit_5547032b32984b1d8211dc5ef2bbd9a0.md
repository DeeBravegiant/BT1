Audit Report

## Title
Stale Cached `rsETHPrice` Enables Share Price Manipulation via Deposits and Instant Withdrawals - (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTOracle.rsETHPrice` is a persistent state variable updated only when `updateRSETHPrice()` is explicitly called. Both `LRTDepositPool.getRsETHAmountToMint()` and `LRTWithdrawalManager.getExpectedAssetAmount()` divide by this stale value while simultaneously reading live Chainlink prices via `getAssetPrice()`. An unprivileged attacker can time deposits or instant-withdrawals within the staleness window to receive more rsETH or more underlying assets than their fair share, directly stealing value from other protocol participants.

## Finding Description

**Root cause:** `rsETHPrice` is a cached state variable (`uint256 public override rsETHPrice`) updated only via `_updateRsETHPrice()`, which is never called atomically inside any user-facing operation.

**Deposit path** (`LRTDepositPool.getRsETHAmountToMint`, line 520):
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
`getAssetPrice(asset)` fetches a live Chainlink price on every call; `rsETHPrice()` returns the stale cached value. When underlying LST prices rise between two `updateRSETHPrice()` calls, the true rsETHPrice is higher than stored. A depositor acting before the update receives more rsETH than their deposit warrants, diluting all existing holders.

**Instant-withdrawal path** (`LRTWithdrawalManager.getExpectedAssetAmount`, line 593):
```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```
When LST prices fall and `rsETHPrice` is stale-high, an attacker calling `instantWithdrawal` before `updateRSETHPrice()` receives more underlying assets than their rsETH is truly worth. Unlike the queued-withdrawal path, `instantWithdrawal` applies **no** `min(expectedAmount, currentReturn)` guard — it transfers the full stale-inflated amount immediately.

**Why existing guards are insufficient:**
- The downside auto-pause in `_updateRsETHPrice()` (lines 270–282) only fires when `updateRSETHPrice()` is explicitly called. The entire attack window exists precisely *before* that call.
- The `pricePercentageLimit` check bounds the magnitude of a single update but does not prevent exploitation during the staleness window.
- `isInstantWithdrawalEnabled[asset]` gates the instant-withdrawal path but is a manager-controlled toggle, not a per-transaction freshness check.
- The deposit path has no analogous gate and is always open when the protocol is unpaused.

## Impact Explanation

**Critical — direct theft of user funds.**

- **Instant-withdrawal scenario:** Attacker burns rsETH and receives more underlying LST than the rsETH is worth at the true current price. The excess is drawn directly from the pool, reducing the redemption value for all remaining depositors.
- **Deposit scenario:** Attacker mints more rsETH than their deposit warrants. After `updateRSETHPrice()` is called, the attacker's rsETH is worth more than deposited; the dilution is borne by all existing holders as a direct, quantifiable reduction in their rsETH backing.

Both paths result in concrete, measurable loss to other protocol participants, satisfying the "direct theft of any user funds" criterion.

## Likelihood Explanation

**Medium.** `updateRSETHPrice()` is public and called by off-chain bots, but there is no on-chain freshness enforcement. Any block between two update calls is a valid attack window. LST prices (stETH/ETH, ETHx/ETH) fluctuate continuously via Chainlink; even 0.1–0.5% deviations over a multi-block window are sufficient for a profitable attack at scale. The attacker requires no special role — any rsETH holder or ETH depositor can execute the deposit-side path at any time the protocol is unpaused. The instant-withdrawal path additionally requires `isInstantWithdrawalEnabled[asset] == true`, which is manager-controlled but is a feature intended to be active in normal operation.

## Recommendation

1. **Atomically refresh `rsETHPrice` before every deposit and withdrawal.** Call `_updateRsETHPrice()` (or an equivalent view-only live computation) inside `_beforeDeposit` and `getExpectedAssetAmount` rather than reading the cached state variable.
2. **Alternatively, compute the share price on-the-fly** using `_getTotalEthInProtocol() / rsethSupply` at the point of use, eliminating the cached value entirely for user-facing operations.
3. **Add a staleness guard** on `rsETHPrice` (e.g., a `lastUpdatedTimestamp` that must be within N blocks) and revert deposits/withdrawals if the price is stale.

## Proof of Concept

**Instant-withdrawal theft (requires `isInstantWithdrawalEnabled[stETH] == true`):**

1. Block B: stETH/ETH Chainlink = 1.01. `updateRSETHPrice()` called; `rsETHPrice` = 1.01. Pool holds 10,000 stETH, 9,901 rsETH outstanding.
2. Block B+10: stETH/ETH Chainlink drops to 0.99. True rsETHPrice ≈ 0.99. Stored `rsETHPrice` = 1.01 (stale).
3. Attacker (holds 1,000 rsETH) calls `instantWithdrawal(stETH, 1000)`:
   - `getExpectedAssetAmount(stETH, 1000)` = `1000 * 1.01 / 0.99` ≈ **1020.2 stETH**
   - Fair value: `1000 * 0.99 / 0.99` = **1000 stETH**
   - Attacker receives **~20.2 stETH excess**, stolen from remaining depositors.
4. `updateRSETHPrice()` subsequently called; remaining depositors' rsETH is now backed by fewer assets.

**Deposit-side dilution (always available):**

1. Block B: stETH/ETH = 0.99. `rsETHPrice` = 0.99 (updated).
2. Block B+5: stETH/ETH rises to 1.01 (live Chainlink). True rsETHPrice ≈ 1.01. Stored `rsETHPrice` = 0.99 (stale).
3. Attacker deposits 1,000 stETH:
   - `getRsETHAmountToMint(stETH, 1000)` = `1000 * 1.01 / 0.99` ≈ **1020.2 rsETH**
   - Fair amount: `1000 * 1.01 / 1.01` = **1000 rsETH**
   - Attacker receives **~20.2 rsETH excess** at existing holders' expense.
4. `updateRSETHPrice()` called; attacker's rsETH is now worth more than deposited.

**Foundry fork test plan:** Fork mainnet, deploy with a mock Chainlink feed, call `updateRSETHPrice()`, advance the feed price by 0.5%, call `depositAsset` / `instantWithdrawal` without calling `updateRSETHPrice()` again, assert that the minted rsETH / received assets exceed the fair-value amount computed from the live price ratio.
