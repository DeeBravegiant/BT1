Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Unlimited ETH Deposits Beyond Protocol Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric check: for LST assets it correctly evaluates `totalAssetDeposits + amount > depositLimit`, but for ETH it evaluates only `totalAssetDeposits > depositLimit`, silently dropping the incoming `amount`. Any unprivileged depositor can therefore push ETH TVL arbitrarily beyond the configured cap and receive a full rsETH mint for the over-limit amount, breaking the protocol's primary risk-management invariant.

## Finding Description
The root cause is a missing `+ amount` in the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit`: [1](#0-0) 

Line 679 returns `totalAssetDeposits > depositLimit` — a pre-deposit comparison that ignores the incoming `amount` entirely. Line 681 (the LST path) correctly includes `amount`. The guard is invoked unconditionally in `_beforeDeposit`: [2](#0-1) 

Which is called by the public, payable `depositETH` entry point: [3](#0-2) 

Because the check passes, `getRsETHAmountToMint` is called with the full `depositAmount` and rsETH is minted for the entire over-limit value via `_mintRsETH`: [4](#0-3) 

No other guard in the call chain compensates for the missing `amount` in the ETH branch.

## Impact Explanation
The deposit limit is the protocol's primary TVL cap for ETH. Bypassing it means the pool accumulates ETH far beyond what EigenLayer strategies can absorb. Excess ETH sits idle in the deposit pool earning no restaking yield, while rsETH holders expect restaking returns. This concretely matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.** No direct theft occurs because the depositor sends real ETH, but the protocol's yield-delivery guarantee is broken for all existing rsETH holders.

## Likelihood Explanation
`depositETH` is public and payable with no access restriction beyond `whenNotPaused`. The precondition is simply that `getTotalAssetDeposits(ETH_TOKEN) <= depositLimit` — a normal operating state. Any depositor can observe this on-chain and immediately send an arbitrarily large ETH deposit. No special role, flash loan, or multi-step setup is required. The exploit is repeatable until the pre-deposit total itself exceeds the cap (which requires a prior over-limit deposit to have already occurred).

## Recommendation
Add `amount` to the ETH branch to mirror the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Assume `depositLimitByAsset(ETH_TOKEN) = 100 ether` and `getTotalAssetDeposits(ETH_TOKEN) = 99 ether`.

1. Attacker calls `depositETH{value: 1000 ether}(0, "")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1000 ether)`.
3. Check evaluates `99 ether > 100 ether` → `false` → no revert.
4. `getRsETHAmountToMint(ETH_TOKEN, 1000 ether)` returns rsETH for the full 1000 ETH.
5. `_mintRsETH` mints that rsETH to the attacker.
6. Pool now holds 1099 ETH against a 100 ETH cap; the excess 999 ETH cannot be deployed to EigenLayer strategies, permanently degrading yield for all rsETH holders.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    // set depositLimit = 100 ether, seed pool with 99 ether of prior deposits
    vm.deal(attacker, 1000 ether);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1000 ether}(0, "");
    assertGt(lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN), 100 ether); // cap violated
}
```

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
