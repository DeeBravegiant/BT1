### Title
`ForeignTxSignPayload.compute_msg_hash()` Omits `domain_id` and Contract Address, Enabling Cross-Context Signature Replay — (`File: crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

The hash that MPC nodes sign for `verify_foreign_transaction` requests is computed solely from `(ForeignChainRpcRequest, Vec<ExtractedValue>)`. Neither the `domain_id` nor the MPC contract's NEAR account address is included. Because `build_signature_request` also applies a zero tweak (no key derivation), the signed digest is identical for any two contexts that share the same key material and the same foreign-chain request. A signature produced for one domain or contract deployment can therefore be replayed verbatim on another, causing the second context to emit a forged foreign-chain verification attestation.

---

### Finding Description

**Root cause — missing context binding in the signed hash.**

`ForeignTxSignPayloadV1` contains only the raw chain request and the extracted values:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

The hash is `SHA-256(borsh(ForeignTxSignPayload))` — no `domain_id`, no contract address, no NEAR-chain context:

```rust
pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
    let mut hasher = sha2::Sha256::new();
    borsh::BorshSerialize::serialize(self, &mut hasher)?;
    Ok(Hash256(hasher.finalize().into()))
}
``` [1](#0-0) 

**Root cause — zero tweak for all foreign-tx signing.**

`build_signature_request` hard-codes `Tweak::new([0u8; 32])`, so the signing key is always the raw root key of the domain with no per-request derivation:

```rust
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),
    ...
    domain: request.domain_id,
})
``` [2](#0-1) 

**Verification gap in `respond_verify_foreign_tx`.**

The contract verifies the signature against `response.payload_hash` and the domain's public key, but never checks that the payload hash encodes the domain or contract context. Any `(payload_hash, signature)` pair that is valid under the domain key is accepted:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
``` [3](#0-2) 

The `domain_id` is present in `VerifyForeignTransactionRequest` and is used only to look up the public key — it is never committed into the signed digest. [4](#0-3) 

---

### Impact Explanation

During a contract migration (v1 → v2) where the same key material is carried over, both contract instances share the same domain public key. A signature `S` produced by the MPC network for a `verify_foreign_transaction` call on v1 is cryptographically indistinguishable from a valid response for the identical request on v2, because the signed digest contains no contract-address or domain binding. An attacker who observes `S` on-chain (it is submitted publicly via `respond_verify_foreign_tx`) can immediately replay it against v2's pending request for the same foreign-chain transaction. The bridge contract consuming v2's response would then incorrectly treat the foreign transaction as freshly attested by the MPC network for v2, enabling a double-spend or invalid bridge execution.

This matches the **High** allowed impact: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

NEAR MPC key migration is an explicitly designed and documented operational procedure (the threat-model document discusses key migration CVMs and seed material transfer). Any migration window during which both the old and new contract are live — and the new contract holds the same domain key — opens the replay window. The attacker requires no privileged access: the only inputs needed are (a) the publicly observable `respond_verify_foreign_tx` transaction on the old contract and (b) the ability to call `respond_verify_foreign_tx` on the new contract, which is open to any attested participant. The foreign-chain request itself is also public (submitted on-chain via `verify_foreign_transaction`).

---

### Recommendation

Include `domain_id` and the MPC contract's NEAR account ID in `ForeignTxSignPayloadV1` so that the signed digest is bound to a specific domain and contract deployment:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub contract_id: AccountId,   // NEAR account of the MPC contract
    pub domain_id: DomainId,
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

This is the direct analog of the ERC-7739 defensive rehashing recommendation: bind the signed message to the verifying context so that a signature valid in one context cannot be accepted in another.

---

### Proof of Concept

1. MPC contract **v1** is deployed at `v1.chain-signatures.near` with `ForeignTx` domain `D`, root key `K`.
2. MPC contract **v2** is deployed at `v2.chain-signatures.near` during migration; domain `D` is initialised with the same key `K`.
3. A bridge service calls `verify_foreign_transaction` on **v1** for Bitcoin tx `T` (domain `D`).
4. MPC nodes compute `H = SHA256(borsh(ForeignTxSignPayload::V1 { request: Bitcoin(T), values: V }))` and produce ECDSA signature `S` under `K`. `S` is submitted on-chain to v1 via `respond_verify_foreign_tx` and is publicly visible.
5. Attacker calls `verify_foreign_transaction` on **v2** for the same Bitcoin tx `T` (domain `D`), paying the 1 yoctoNEAR deposit.
6. Attacker immediately calls `respond_verify_foreign_tx` on **v2** supplying `request = {domain_id: D, request: Bitcoin(T), ...}` and `response = {payload_hash: H, signature: S}`.
7. **v2** looks up key `K` for domain `D`, verifies `S` over `H` — verification passes because `H` is identical (no domain/contract binding) and `K` is the same key.
8. **v2** resolves the pending yield and returns `VerifyForeignTransactionResponse` to the bridge contract, which processes the Bitcoin deposit a second time. [1](#0-0) [5](#0-4) [6](#0-5)

fn build_signature_request(

    foreign_tx_payload: &dtos::ForeignTxSignPayload,

    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
