Audit Report

## Title
Missing Chainlink `updatedAt` Staleness Check Allows Stale Collateral Price Acceptance - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` retrieves Chainlink's `updatedAt` timestamp but never validates it against `block.timestamp`. The two existing guards (`answeredInRound < roundID` and `timestamp == 0`) do not detect a round that was answered hours or days ago. Any collateral token priced via this oracle can be deposited at a stale (inflated) rate, causing the depositor to receive more rsETH/wrsETH than the current market value warrants, at the expense of existing holders.

## Finding Description
In `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` lines 26–37, `getRate()` calls `latestRoundData()` and captures `timestamp` (`updatedAt`) but performs no comparison against `block.timestamp`:

```solidity
function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();   // only catches unanswered rounds
    if (timestamp == 0) revert IncompleteRound();          // only catches zero timestamp
    if (ethPrice <= 0) revert InvalidPrice();
    // ← no: block.timestamp - timestamp > MAX_STALENESS check
    ...
}
```

This oracle is registered as the `supportedTokenOracle[token]` for collateral LSTs in the RSETHPool family. When a user calls `deposit(token, amount, referralId)`, the pool calls `viewSwapRsETHAmountAndFee(amount, token)`, which fetches `tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate()` and computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate`. A stale, inflated `tokenToETHRate` directly inflates the rsETH minted per unit of collateral deposited.

The `answeredInRound < roundID` guard only detects rounds that were never answered. A round that was answered 12 hours ago will have `answeredInRound == roundID` and a non-zero `timestamp`, passing all three guards while returning a price that no longer reflects market reality.

## Impact Explanation
**High — Theft of unclaimed yield / potential Critical — Direct theft of user funds.**

An attacker who deposits collateral at a stale inflated price receives more rsETH/wrsETH than the current market value of their deposit. This dilutes the backing ratio for all existing rsETH holders (theft of yield). At sufficient scale, or if the attacker immediately redeems rsETH for ETH from the pool, this constitutes direct extraction of other users' deposited funds. Both impacts are within the allowed scope.

## Likelihood Explanation
L2 sequencer outages on Arbitrum, Optimism, and Base are documented, recurring events. During an outage, Chainlink feeds cannot post new rounds; when the sequencer resumes, there is a window before the feed updates during which the stale price passes all guards. An unprivileged attacker monitoring the sequencer status can submit a `deposit(token, amount, referralId)` call immediately upon sequencer resumption, before the oracle updates, requiring no privileged access and no victim mistake. The attack is repeatable across any sequencer downtime event.

## Recommendation
Add a configurable maximum staleness threshold and validate `updatedAt` against `block.timestamp` in `getRate()`:

```solidity
uint256 public constant MAX_STALENESS = 3600; // configure per feed heartbeat

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 updatedAt, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice(); // ← add this
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

The staleness threshold should be set per-feed based on the Chainlink heartbeat (e.g., 3600s for 1-hour heartbeat feeds, 86400s for 24-hour feeds), and should also incorporate a sequencer uptime feed check for L2 deployments.

## Proof of Concept
1. Deploy or fork an L2 (e.g., Arbitrum) where `RSETHPoolV3` or `RSETHPool` has a supported collateral token (e.g., wstETH) with `ChainlinkOracleForRSETHPoolCollateral` as its oracle.
2. Simulate sequencer downtime: advance `block.timestamp` by 2 hours without advancing the Chainlink feed (mock `latestRoundData()` to return `updatedAt = block.timestamp - 7200`, `answeredInRound == roundID`, and `answer = 1.2e8` while the true market rate has dropped to `1.0e8`).
3. Call `deposit(wstETH, 1e18, "")` on the pool. `ChainlinkOracleForRSETHPoolCollateral.getRate()` returns `1.2e18` — all three guards pass.
4. Observe that `rsETHAmount = 1e18 * 1.2e18 / rsETHToETHrate` — 20% more rsETH than the current market rate warrants.
5. Confirm that without the staleness check, the transaction succeeds and the attacker holds excess rsETH at the expense of existing holders.

Foundry fork test: fork Arbitrum mainnet at a block just after a sequencer resumption, warp `block.timestamp` forward past the Chainlink heartbeat without mining new oracle rounds, then call `deposit` and assert the minted rsETH exceeds the fair-value amount.
