Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Calculations Enables Theft of Unclaimed Yield — (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary

`LRTOracle` stores the rsETH/ETH exchange rate in a public state variable `rsETHPrice` that is only updated when `updateRSETHPrice()` is explicitly called. `LRTDepositPool.getRsETHAmountToMint` and `LRTWithdrawalManager.getExpectedAssetAmount` both read this stored value directly with no atomic refresh. When staking rewards accrue between oracle updates, the stored price lags behind the true price, allowing depositors to receive more rsETH than they deserve — diluting existing holders' accrued yield.

## Finding Description

`LRTOracle` stores the exchange rate in a public state variable updated only via `_updateRsETHPrice()`:

```solidity
// contracts/LRTOracle.sol L28
uint256 public override rsETHPrice;
```

The true current price is computed inside the `private` function `_getTotalEthInProtocol()` (L331–349), which sums all staked assets at their current oracle prices. This logic is not exposed through any public view function, and `ILRTOracle` exposes only the stored value:

```solidity
// contracts/interfaces/ILRTOracle.sol L31
function rsETHPrice() external view returns (uint256);
```

**Deposit path**: `getRsETHAmountToMint` (L506–521 of `LRTDepositPool.sol`) divides by the stored `rsETHPrice`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This is called unconditionally from `_beforeDeposit` (L665), which is invoked by `depositAsset` and `depositETH` with no prior call to `updateRSETHPrice()`.

**Withdrawal path**: `getExpectedAssetAmount` (L593 of `LRTWithdrawalManager.sol`) multiplies by the stored `rsETHPrice`:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

Called from both `initiateWithdrawal` (L168) and `instantWithdrawal` (L228) with no prior price refresh.

**Why existing guards fail**: The `minRSETHAmountExpected` slippage parameter in `_beforeDeposit` (L667) protects the depositor from receiving too little, not existing holders from dilution. The `pricePercentageLimit` check in `_updateRsETHPrice` (L256–266) limits how much the price can increase per update call, but does not prevent the stale price from being used in deposit/withdrawal calculations. An attacker deliberately avoids calling `updateRSETHPrice()` before depositing to exploit the stale price.

## Impact Explanation

**High — Theft of unclaimed yield.**

When staking rewards accrue, the true rsETH/ETH rate rises above the stored `rsETHPrice`. A depositor calling `depositAsset` with a stale (lower) denominator receives more rsETH than their deposit is worth at the true price. After `updateRSETHPrice()` is called, the attacker's excess rsETH is already minted and permanently dilutes all prior holders' share of the protocol's assets. The accrued yield that belonged to existing rsETH holders is transferred to the attacker. This is direct, irreversible theft of unclaimed yield, not merely a best-practice concern.

## Likelihood Explanation

`updateRSETHPrice()` is permissionless but is not called atomically with deposits. Oracle updates are triggered off-chain on a periodic schedule (bots/managers), creating a predictable, observable staleness window. The gap between the stored price and the true price is computable entirely from on-chain data (total asset balances × asset oracle prices ÷ rsETH supply). No special access, flash loans, or governance control is required. Any external depositor can exploit this on every oracle update cycle, making it continuously repeatable.

## Recommendation

1. **Preferred**: Add a public view function to `LRTOracle` that computes the current rsETH price on-the-fly (replicating `_getTotalEthInProtocol()` logic) without writing state. Expose it through `ILRTOracle`. Replace `lrtOracle.rsETHPrice()` in `getRsETHAmountToMint` and `getExpectedAssetAmount` with this live-price function.
2. **Alternative**: Call `updateRSETHPrice()` atomically at the start of `depositAsset`, `depositETH`, and `initiateWithdrawal` before any price-dependent calculation. This ensures the stored price is always fresh at the point of use.

## Proof of Concept

**Setup**: Fork mainnet with `rsETHPrice` stored = `1.000e18` (last updated 12 hours ago). Staking rewards have accrued; true price (computed from on-chain balances) = `1.005e18`.

**Steps**:
1. Attacker calls `LRTDepositPool.depositAsset(stETH, 1_000e18, 0, "")`.
2. `getRsETHAmountToMint` computes: `(1_000e18 × 1e18) / 1.000e18 = 1_000 rsETH`.
3. Fair amount at true price: `(1_000e18 × 1e18) / 1.005e18 ≈ 995.02 rsETH`.
4. Attacker receives `≈ 4.98 rsETH` excess.
5. Anyone calls `updateRSETHPrice()`; price updates to `1.005e18`.
6. Attacker's excess rsETH is already minted. All prior holders' rsETH is now worth proportionally less — their accrued yield has been diluted.

**Foundry test plan**: Write a fork test that (a) records `rsETHPrice`, (b) advances time to simulate reward accrual (or directly manipulates asset balances to increase TVL), (c) calls `depositAsset` without calling `updateRSETHPrice()` first, (d) asserts that `rsethAmountToMint > (depositAmount × assetPrice) / trueLivePrice`, and (e) calls `updateRSETHPrice()` and asserts the attacker's rsETH balance exceeds fair value.
