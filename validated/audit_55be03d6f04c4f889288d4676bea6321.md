Audit Report

## Title
Missing Slippage Protection in L2 Pool `deposit()` Functions Allows Users to Receive Fewer rsETH Tokens Than Expected - (File: contracts/pools/RSETHPoolNoWrapper.sol)

## Summary
The L2 deposit pool contracts (`RSETHPoolNoWrapper`, `RSETHPoolV2`, `RSETHPoolV3`, and variants) compute the rsETH output amount from a live oracle rate at execution time but accept no `minRsETHAmountOut` parameter. Any user whose transaction is pending across an oracle rate update will silently receive fewer rsETH/wrsETH tokens than observed at submission time, with no on-chain mechanism to revert. The L1 `LRTDepositPool` implements this protection via `minRSETHAmountExpected` and `_beforeDeposit()`, confirming the protocol recognizes its necessity.

## Finding Description
In `RSETHPoolNoWrapper.sol`, both `deposit(string)` (ETH) and `deposit(address,uint256,string)` (token) call `viewSwapRsETHAmountAndFee()`, which reads `getRate()` from the oracle at execution time and computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. The result is immediately transferred to the caller with no minimum-output guard:

```solidity
// RSETHPoolNoWrapper.sol L231-244
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
rsETH.safeTransfer(msg.sender, rsETHAmount);
```

The same pattern exists in `RSETHPoolV2.sol` (`wrsETH.mint(msg.sender, rsETHAmount)` with no check) and in `RSETHPoolV3.sol` and its variants.

By contrast, `LRTDepositPool._beforeDeposit()` explicitly reverts with `MinimumAmountToReceiveNotMet` if `rsethAmountToMint < minRSETHAmountExpected`, demonstrating the protocol's own standard for this protection.

The rsETH/ETH rate (`rsETHToETHrate`) increases monotonically as staking rewards accrue. Any pending transaction that spans an oracle update will execute at a higher rate, yielding strictly fewer rsETH tokens. No attacker is required; the structural design alone causes the shortfall.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

A user who previews the expected output via `viewSwapRsETHAmountAndFee()` and submits a deposit transaction may receive fewer rsETH/wrsETH tokens than observed, with no on-chain protection to revert. The protocol does not lose value; only the depositor suffers a shortfall relative to their expectation. This matches the allowed Low impact: "Contract fails to deliver promised returns, but doesn't lose value."

## Likelihood Explanation
Every unprivileged user calling `deposit()` on any deployed L2 pool contract is affected. Oracle rate updates are periodic and routine. On congested L2 networks, transaction inclusion delays are common. No special attacker capability is required — the loss occurs passively whenever a deposit transaction is pending across an oracle update. The condition is repeatable and affects all depositors.

## Recommendation
Add a `minRsETHAmountOut` parameter to all `deposit()` overloads in the L2 pool contracts, mirroring the existing `minRSETHAmountExpected` pattern in `LRTDepositPool`. Revert if the computed `rsETHAmount` falls below the caller-specified minimum:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountOut) external payable nonReentrant whenNotPaused {
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountOut) revert SlippageExceeded();
    ...
}
```

Apply the same change to the token-deposit overload and to all pool contract variants (`RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolV2NBA`).

## Proof of Concept
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` on `RSETHPoolNoWrapper` and observes `rsETHAmount = X` at oracle rate `R`.
2. User submits `deposit{value: 1 ether}(referralId)`.
3. Before the transaction is mined, the oracle rate is updated from `R` to `R' > R` (rsETH appreciated due to staking rewards).
4. The transaction executes: `rsETHAmount = (1 ether - fee) * 1e18 / R'`, which is strictly less than `X`.
5. `rsETH.safeTransfer(msg.sender, rsETHAmount)` executes with no minimum check; user receives fewer tokens than expected with no revert.

**Foundry fork test plan:**
- Fork an L2 where `RSETHPoolNoWrapper` is deployed.
- Record `viewSwapRsETHAmountAndFee(1 ether)` → `expectedAmount`.
- Use `vm.store` to advance the oracle rate by a small increment (simulating a periodic update).
- Call `deposit{value: 1 ether}("")` and assert `rsETH.balanceOf(user) < expectedAmount`.
- Confirm no revert occurs.
