Audit Report

## Title
O(A×N) Unbounded External Calls in `updateRSETHPrice()` Can Permanently Freeze rsETH Price — (`contracts/LRTOracle.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is a public, permissionless function whose internal call chain makes A×N separate external calls to `DelegationManager.getWithdrawableShares()`, where A is the number of supported assets and N is the number of NodeDelegators. Neither dimension has a hard on-chain ceiling, so as the protocol scales, the cumulative gas cost can exceed the block gas limit, permanently preventing any price update.

## Finding Description

The confirmed call chain through unmodified production code:

```
updateRSETHPrice()                          [public, whenNotPaused only]
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()          [loops over A assets — LRTOracle.sol L336]
            └─ getTotalAssetDeposits(asset)
                 └─ getAssetDistributionData(asset)
                      └─ [loops over N NDCs — LRTDepositPool.sol L447]
                           ├─ getAssetBalance(asset)           [L450 — one external call per NDC]
                           │    └─ NodeDelegatorHelper.getAssetBalance()
                           │         └─ getWithdrawableShare()
                           │              └─ getWithdrawableShares()  [wraps single strategy in array]
                           │                   └─ DelegationManager.getWithdrawableShares()  ← external
                           └─ getAssetUnstaking(asset)         [L451 — additional external call per NDC]
```

**Root cause:** `NodeDelegatorHelper.getWithdrawableShare()` (L52-65) wraps a single strategy in a one-element array and calls `getWithdrawableShares()` (L41-50), which in turn calls `DelegationManager.getWithdrawableShares()`. This means one external call is made per (NDC, asset) pair instead of batching all strategies for a given NDC into a single call.

**No hard cap on either dimension:**
- `maxNodeDelegatorLimit` initializes to 10 (`LRTDepositPool.sol` L49) but `updateMaxNodeDelegatorLimit()` (L290-297) only enforces `maxNodeDelegatorLimit_ >= nodeDelegatorQueue.length` — no upper bound.
- `_addNewSupportedAsset()` in `LRTConfig.sol` (L115) pushes to `supportedAssetList` with no length cap.

**Existing guards are insufficient:** The only guard on `updateRSETHPrice()` is `whenNotPaused` (L87). There is no gas guard, no batching, and no cap that would prevent the A×N call count from growing without bound.

## Impact Explanation

**Medium — Unbounded gas consumption / permanent freezing of price update.**

If A×N `getWithdrawableShares` calls (plus surrounding loop overhead, SLOADs, and CALL setup) exceed the block gas limit (~30M gas on Ethereum mainnet), every invocation of `updateRSETHPrice()` reverts with out-of-gas. Since `rsETHPrice` is only written through `_updateRsETHPrice()` (L313), the stored price becomes permanently stale. Downstream flows that read `rsETHPrice` directly (e.g., `getRsETHAmountToMint()` at `LRTDepositPool.sol` L520) will silently use an outdated rate, and the protocol loses the ability to correct the price without a contract upgrade. This matches the allowed impact: **Medium — Unbounded gas consumption**.

## Likelihood Explanation

- `updateRSETHPrice()` is callable by any EOA with no role requirement — only `whenNotPaused`.
- The protocol already supports multiple LST assets (stETH, ETHx, etc.) and is designed to add more via governance.
- `maxNodeDelegatorLimit` has no on-chain ceiling; operators routinely increase it as the protocol scales.
- No gas-limit guard or batching exists anywhere in the price-update path.
- At A=10, N=10: 100 cold external calls to EigenLayer. At A=20, N=20: 400 calls. At A=30, N=30: 900 calls — these are reachable configurations given the absence of any hard cap.

## Recommendation

1. **Batch EigenLayer queries per NDC**: Instead of calling `getWithdrawableShares` once per (NDC, asset) pair, call it once per NDC with all strategies in a single array. `NodeDelegatorHelper.getWithdrawableShares()` (L41-50) already accepts `IStrategy[] memory strategies` — use it at the NDC level to fetch all asset balances in one external call per NDC, reducing the call count from A×N to N.

2. **Enforce a hard cap** on both `maxNodeDelegatorLimit` and `supportedAssetList.length` at values that keep worst-case gas well below the block limit (e.g., max 10 NDCs × 10 assets = 100 calls maximum).

3. **Add a gas guard** in `updateRSETHPrice()` (e.g., `require(gasleft() > MIN_GAS_REQUIRED)`) to fail fast with a meaningful error rather than an opaque out-of-gas revert.
