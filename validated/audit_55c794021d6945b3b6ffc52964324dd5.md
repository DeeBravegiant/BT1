Audit Report

## Title
ETH Deposit Limit Check Missing `amount` Allows Deposit Cap Bypass - (File: `contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric deposit-limit check: for ERC20 assets it correctly tests `totalAssetDeposits + amount > limit`, but for ETH it omits `amount` entirely, testing only `totalAssetDeposits > limit`. Any unprivileged ETH depositor can therefore push the protocol past its configured ETH deposit cap in a single `depositETH` call, enabling excess rsETH minting beyond the protocol's intended backing.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682), the ETH branch compares only the pre-deposit total against the limit, ignoring the incoming `amount`:

```solidity
// contracts/LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC20
}
```

This function is called from `_beforeDeposit` (lines 661–663), which reverts with `MaximumDepositLimitReached` only when it returns `true`. `_beforeDeposit` is invoked by `depositETH` (line 87) with `msg.value` as the deposit amount. Because the ETH branch never adds `amount` to `totalAssetDeposits`, the check passes whenever `totalAssetDeposits ≤ limit`, regardless of how large `msg.value` is. The deposit then proceeds to `_mintRsETH`, minting rsETH for the full `msg.value` even when it pushes the pool over the cap.

**Concrete example:**
- `depositLimitByAsset(ETH_TOKEN)` = 100 ETH
- `totalAssetDeposits(ETH)` = 99 ETH
- Alice calls `depositETH{value: 10 ether}`
- Check: `99 > 100` → `false` → deposit allowed
- Post-deposit total: **109 ETH** (9 ETH over cap); ERC20 path would have correctly returned `true` (`99 + 10 > 100`) and reverted.

## Impact Explanation
The ETH deposit limit is a protocol-level safety cap. Bypassing it allows excess rsETH to be minted beyond what the protocol intends to back. If the cap was sized to match EigenLayer strategy capacity or liquidity constraints, exceeding it leaves the protocol unable to honour redemptions. This constitutes a concrete path to **protocol insolvency** (a Critical allowed impact), or at minimum a contract failing to deliver promised returns (Low). Given that the cap is a deliberate protocol invariant and rsETH is minted against the excess ETH, the realistic impact is **excess rsETH minting enabling protocol insolvency**.

## Likelihood Explanation
The vulnerable condition — `totalAssetDeposits ≤ limit` but `totalAssetDeposits + depositAmount > limit` — is a normal operating state whenever the pool is near its cap. No special privileges, setup, or victim mistakes are required. Any unprivileged user can call `depositETH` with a sufficiently large `msg.value` to trigger it. The exploit is repeatable until the pool is paused by an admin.

## Recommendation
Add `amount` to the ETH branch, mirroring the ERC20 logic:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 100 ether`.
2. Protocol accumulates `99 ether` of ETH deposits (via prior `depositETH` calls).
3. Alice calls `depositETH{value: 10 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10 ether)` evaluates `99 > 100` → `false`.
5. `_mintRsETH` executes; Alice receives rsETH; protocol ETH total becomes 109 ETH — 9 ETH above the configured cap.
6. Any depositor can repeat until the cap is arbitrarily exceeded.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 100 ether);
    // fill pool to 99 ETH via prior deposits ...
    vm.deal(alice, 10 ether);
    vm.prank(alice);

}
