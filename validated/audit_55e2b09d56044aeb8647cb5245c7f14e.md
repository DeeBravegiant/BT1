### Title
Excess Deposit Silently Absorbed for Existing-Participant Re-submissions in `submit_participant_info` — (`File: crates/contract/src/lib.rs`)

### Summary
In `submit_participant_info`, the deposit-check-and-refund block is gated on `attestation_storage_must_be_paid_by_caller`. When an existing participant updates their attestation (`UpdatedExistingParticipant` + caller is a voter), that flag is `false` and the entire block is skipped. Because the function is `#[payable]`, any deposit the caller attaches is silently absorbed by the contract — never charged, never refunded — breaking the accounting invariant that excess deposits must be returned to the sender.

### Finding Description
`submit_participant_info` computes a boolean to decide whether to enforce a deposit:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;   // line 823-824

if attestation_storage_must_be_paid_by_caller {
    // ... cost check, InsufficientDeposit error, and refund of excess ...
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();  // line 847
    }
}
```

When the caller is an existing participant re-submitting an updated attestation:
- `is_new_attestation` = `false` (`UpdatedExistingParticipant` branch of `add_participant`)
- `caller_is_not_participant` = `false` (`voter_account()` succeeds)
- Therefore `attestation_storage_must_be_paid_by_caller` = `false`

The `if` block is skipped entirely. There is no other refund path in the function. Any `attached_deposit` is silently retained by the contract.

The function is marked `#[payable]`, so the NEAR runtime accepts any deposit amount without restriction. The contract's own design elsewhere (e.g., `require_deposit`, `propose_update`) always refunds excess; this path does not.

### Impact Explanation
This breaks the production accounting invariant that "excess deposit is automatically refunded" — stated explicitly in the contract README and enforced in every other payable entry point. An existing participant who attaches a non-zero deposit when re-submitting (e.g., following the operator guide's instruction to attach `--deposit` for first-time joins, then repeating the same call on a TEE upgrade) permanently loses those NEAR tokens to the contract with no error, no log, and no recourse. The contract balance grows by the absorbed amount; the caller's balance shrinks by the same amount. This is the direct NEAR-MPC analog of M-06: the "unspent" deposit is silently taken by the contract instead of being returned.

### Likelihood Explanation
The `mpc-node` daemon attaches `0` by default on periodic re-submissions, so automated re-attestation is unaffected. However, the operator guide explicitly instructs first-time joiners to call `submit_participant_info` manually with `--deposit` once. An operator who follows this instruction and then re-runs the same manual command on a subsequent TEE upgrade (a documented, expected lifecycle event) will silently lose the attached deposit. The scenario is realistic and requires no adversary — only a legitimate participant following documented procedure.

### Recommendation
Add an unconditional refund of any excess deposit after the `if attestation_storage_must_be_paid_by_caller` block, mirroring the pattern used in `require_deposit` and `propose_update`:

```rust
// After the conditional block, always refund any deposit not consumed.
let cost_charged = if attestation_storage_must_be_paid_by_caller { cost } else { NearToken::from_yoctonear(0) };
if let Some(diff) = env::attached_deposit().checked_sub(cost_charged)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
```

Alternatively, reject non-zero deposits when no deposit is required, so the caller receives an immediate error rather than a silent loss.

### Proof of Concept
1. Alice is an existing participant with a stored attestation (e.g., after initial onboarding).
2. A TEE upgrade occurs; Alice must re-submit her attestation.
3. Alice follows the operator guide and calls `submit_participant_info` manually with `--deposit 1NEAR`.
4. Inside the contract: `add_participant` returns `UpdatedExistingParticipant`; `voter_account()` succeeds.
5. `is_new_attestation = false`, `caller_is_not_participant = false` → `attestation_storage_must_be_paid_by_caller = false`.
6. The deposit-check block is skipped; no refund is scheduled.
7. The function returns `Ok(())`.
8. Alice's 1 NEAR is permanently absorbed by the contract. Alice's balance decreases; the contract balance increases. No log entry, no error. [1](#0-0) [2](#0-1) [3](#0-2)
