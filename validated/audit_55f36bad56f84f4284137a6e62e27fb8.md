### Title
Byzantine Participant Can Corrupt Non-PV CKD Output via Unverified Aggregation in `protocol.rs::do_ckd_coordinator` — (`crates/threshold-signatures/src/confidential_key_derivation/protocol.rs`)

---

### Summary

The non-publicly-verifiable CKD protocol (`protocol.rs`) aggregates participant contributions `(norm_big_y_i, norm_big_c_i)` in `do_ckd_coordinator` without any post-aggregation correctness check. A single Byzantine participant (strictly below the signing threshold) can send an arbitrary pair over the P2P channel. The coordinator blindly sums all contributions and submits the corrupted `(Y, C)` to the contract. The contract's `respond_ckd` explicitly skips verification for the `AppPublicKey` variant. The app therefore receives a key that is not `msk·H(pk, app_id)`, and the attacker can freely choose the additive offset applied to that key.

---

### Finding Description

**Root cause 1 — missing aggregation check in `protocol.rs`:**

`do_ckd_coordinator` in `protocol.rs` aggregates all participant shares and immediately returns the result:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

The PV sibling `protocol_pv.rs::do_ckd_coordinator` performs the identical aggregation loop but then calls `aggregated_output_check`, which verifies the pairing equation `e(C, G2) = e(Y, app_pk2) · e(H(pk,app_id), msk_pk)` and returns `ProtocolError` on failure:

```rust
if !aggregated_output_check(&ckd_output, app_pk, &key_pair.public_key, &hash_point) {
    return Err(ProtocolError::AssertionFailed(
        "CKD output failed to verify".to_string(),
    ));
}
``` [2](#0-1) 

No equivalent guard exists in `protocol.rs`.

**Root cause 2 — contract skips verification for `AppPublicKey` variant:**

`respond_ckd` in the contract explicitly does nothing for the non-PV variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(...) { env::panic_str(...); }
    }
}
``` [3](#0-2) 

**Root cause 3 — node dispatches the unprotected protocol for `AppPublicKey` requests:**

`CKDComputation::compute` routes `AppPublicKey` requests to `protocol.rs::ckd` and `AppPublicKeyPV` requests to `protocol_pv.rs::ckd_pv`: [4](#0-3) 

There is therefore no layer — neither inside the MPC protocol nor on-chain — that can detect or reject a corrupted non-PV CKD output.

---

### Impact Explanation

Let honest contributions sum to `(Y_h, C_h)` such that `C_h - a·Y_h = msk·H(pk, app_id)`. A Byzantine participant substitutes `(Y'_i, C'_i)` for its honest share. The coordinator computes:

```
Y_final = Y_h + Y'_i
C_final = C_h + C'_i
```

The app decrypts: `key = C_final - a·Y_final = msk·H(pk,app_id) + (C'_i - a·Y'_i)`.

If the attacker sets `Y'_i = G1::identity()` and `C'_i = δ` (any chosen G1 point), the `a`-dependent term vanishes and the app receives `msk·H(pk,app_id) + δ`. The attacker freely controls the additive offset `δ` without knowing the app's secret key `a`. The app has no reference value to compare against and cannot detect the substitution. The output is therefore an attacker-chosen key that is not the intended `msk·H(pk,app_id)`, constituting unauthorized CKD output.

---

### Likelihood Explanation

The attacker must be one of the `threshold`-many participants selected for the CKD round. Participant selection is random from the active set (`select_random_active_participants_including_me`). A single compromised node in the participant set is sufficient — no threshold-level collusion is required. The `AppPublicKey` (non-PV) variant is a production-supported request type, not gated behind any feature flag. [5](#0-4) 

---

### Recommendation

1. **Add `aggregated_output_check` to `protocol.rs::do_ckd_coordinator`** — mirror the guard already present in `protocol_pv.rs`. For the non-PV variant the check degenerates to verifying `e(C, G2) = e(Y, app_pk2) · e(H(pk,app_id), msk_pk)` using the G1-only app public key (treating it as `pk1` with an implicit `pk2 = a·G2` unknown to the coordinator). Note that without `pk2` the coordinator cannot run the pairing check, which is precisely why the PV variant was introduced. The correct fix is therefore to **deprecate the non-PV variant** or require all callers to supply `AppPublicKeyPV`.

2. **Add on-chain verification for `AppPublicKey` in `respond_ckd`** — or reject `AppPublicKey` requests entirely and require `AppPublicKeyPV`, which already has the on-chain `ckd_output_check` guard.

---

### Proof of Concept

```rust
// Integration test sketch (non-PV protocol, one Byzantine participant)
#[test]
fn byzantine_participant_corrupts_non_pv_ckd_output() {
    let mut rng = OsRng;
    let app_id = AppId::try_from(b"Near App").unwrap();
    let app_sk = Scalar::random(&mut rng);
    let app_pk = G1Projective::generator() * app_sk;

    let threshold = 2;
    let participants = generate_participants(3);
    let keys = run_keygen(&participants, threshold.into());
    let public_key = keys[&participants[0]].public_key;
    let coordinator = participants[0];

    // Honest participants run the real protocol; Byzantine participant
    // sends (G1::identity(), G1::identity()) as its contribution.
    // The coordinator aggregates without checking and returns a corrupted output.
    // Assert: ckd.unmask(app_sk) ≠ msk·H(pk, app_id)
    // (verify_signature should fail on the corrupted output)
}
```

The test mirrors `test_ckd` in `crates/threshold-signatures/tests/ckd.rs` but replaces one participant's `do_ckd_participant` with a stub that sends `CKDOutput::new(G1::identity(), G1::identity())`. The coordinator's `do_ckd_coordinator` will accept and aggregate the identity contributions, producing a corrupted `(Y, C)`. The final `verify_signature` call will fail, confirming the unauthorized output. [6](#0-5)

fn test_ckd() {
    let mut rng = OsRng;

    // Create the app necessary items
    let app_id = AppId::try_from(b"Near App").unwrap();
    let app_sk = Scalar::random(&mut rng);
    let app_pk = G1Projective::generator() * app_sk;

    // create participants
    let threshold = 2;
    let participants = generate_participants(3);

    let keys = run_keygen(&participants, threshold.into());

    assert!(keys.len() == participants.len());

    let public_key = keys.get(&participants[0]).unwrap().public_key;
    let coordinator = choose_coordinator_at_random(&participants);

    let mut protocols: GenProtocol<CKDOutputOption> = Vec::with_capacity(participants.len());

    for p in &participants {
        let key_pair = keys.get(p).unwrap();

        let protocol = ckd(
            &participants,
            coordinator,
            *p,
            key_pair.clone(),
            app_id.clone(),
            app_pk,
            OsRng,
        )
        .unwrap();

        protocols.push((*p, Box::new(protocol)));
    }

    let result = run_protocol(protocols).unwrap();

    // test one single some for the coordinator
    let mut some_iter = result.into_iter().filter(|(_, ckd)| ckd.is_some());

    let ckd = some_iter
        .next()
        .map(|(_, c)| c.unwrap())
        .expect("Expected exactly one Some(CKDCoordinatorOutput)");
    assert!(
        some_iter.next().is_none(),
        "More than one Some(CKDCoordinatorOutput)"
    );

    // compute msk . H(app_id)
    let confidential_key = ckd.unmask(app_sk);
    assert!(verify_signature(&public_key, &app_id, &confidential_key).is_ok());
