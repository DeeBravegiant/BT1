### Title
Unbounded `fee_proposal_fri` Accepted During Startup Window Corrupts Future Gas-Price Accounting — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

During the first `fee_proposal_window_size` blocks (startup / near-genesis), `is_proposal_init_valid` explicitly skips all bounds enforcement on the proposer-supplied `fee_proposal_fri` field. A malicious proposer can therefore inject an arbitrary value (e.g., `u128::MAX`) into the `ProposalInit`, have it accepted by every validator, committed to consensus, stored in the block header, and recorded in `fee_proposals_window`. Once the window fills, `compute_fee_actual` returns a median that includes the poisoned value, permanently distorting the L2 base-fee for all subsequent blocks.

### Finding Description

`is_proposal_init_valid` in `validate_proposal.rs` validates `fee_proposal_fri` in two steps:

1. **Presence check** (lines 370–394): rejects `None` when `starknet_version >= V0_14_3` and `Some(_)` when below it.
2. **Bounds check** (lines 396–416): enforces that `fee_proposal_fri` lies within `fee_proposal_bounds(fee_actual, margin_ppt)`.

The bounds check is guarded by:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    ...
}
``` [1](#0-0) 

When `fee_actual` is `None` — which is the case for every block in `[0, window_size)` — the inner body is never reached and **any** `fee_proposal_fri` value passes validation.

The honest proposer's fallback during this window is `l2_gas_price`:

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
``` [2](#0-1) 

But the validator never checks that the received `fee_proposal_fri` equals (or is close to) `l2_gas_price_fri`, which **is** validated in the same function:

```rust
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
``` [3](#0-2) 

The accepted `fee_proposal_fri` is then bound into the `ProposalCommitment` via Poseidon:

Because the validator computes `batcher_block_commitment` using the **received** `fee_proposal_fri` (not a locally-derived value), the commitment always matches the proposer's `ProposalFin.proposal_commitment` regardless of the injected value:

```rust

The poisoned `fee_proposal_fri` is then stored in the block header and later read back by `initialize_fee_proposals_window`:

```rust

    block_number,
    block.block_header_without_hash.fee_proposal_fri,

If a malicious proposer controls a majority of blocks in the startup window, the median becomes the injected extreme value, permanently distorting `fee_actual` and therefore `l2_gas_price` for all subsequent blocks.

### Impact Explanation

`fee_proposal_fri` is accepted without any bounds check during the startup window. An injected `u128::MAX` (or near-zero) value is committed to consensus, stored on-chain, and fed into the sliding-window median that drives the L2 base fee. Once the window fills with poisoned values, `fee_actual` becomes extreme, causing `l2_gas_price` to spike to `u128::MAX` (blocking all transactions) or collapse to near-zero (enabling fee-free spam). This is a direct, permanent economic impact on the network's fee market.

**Matching impact scope**: *Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.*

### Likelihood Explanation

The attack requires being the block proposer for more than half of the `fee_proposal_window_size` blocks during the startup window. In Tendermint the proposer rotates proportionally to stake, so a validator with >50% of stake during genesis/restart can execute this unilaterally. Even a minority proposer can skew the median by controlling a smaller fraction of startup blocks, shifting the base fee in their favor. The startup window is a predictable, finite period that an attacker can plan around.

### Recommendation

During the startup window (when `fee_actual` is `None`), validate `fee_proposal_fri` against the locally-known `l2_gas_price_fri` using the same `fee_proposal_bounds` helper, substituting `l2_gas_price_fri` as the reference:

```rust
// Startup fallback: no fee_actual yet, but still bound fee_proposal to l2_gas_price.
if let (None, Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        proposal_init_validation.l2_gas_price_fri,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
```

This mirrors the honest proposer's own fallback (`freeze at l2_gas_price`) and closes the window without breaking the startup path for honest nodes.

### Proof of Concept

1. Network starts at block 0 (`fee_proposal_window_size = 10`, so `fee_actual = None` for blocks 0–9).
2. Malicious proposer is selected for block 0. It constructs a `ProposalInit` with `fee_proposal_fri = Some(GasPrice(u128::MAX))`.
3. `is_proposal_init_valid` reaches the bounds check at line 398; the `if let` pattern fails to match because `proposal_init_validation.fee_actual` is `None`. The check is skipped entirely.
4. `initiate_validation` forwards the `ProposalInit` to the batcher unchanged.
5. `handle_proposal_part` computes `batcher_block_commitment = Poseidon(partial_block_hash, u128::MAX)`.
6. The proposer's `ProposalFin.proposal_commitment` is also `Poseidon(partial_block_hash, u128::MAX)`. The comparison at line 244 passes.
7. Consensus reaches decision; `fee_proposal_fri = u128::MAX` is stored in the block header.
8. Repeat for blocks 1–9 (if the attacker controls enough proposer slots).
9. At block 10, `compute_fee_actual` computes the median of `[u128::MAX, ...]`; if the attacker controlled ≥6 slots, the median is `u128::MAX`.
10. All subsequent `l2_gas_price` values are derived from `fee_actual = u128::MAX`, making every transaction unaffordable and effectively halting the network. [1](#0-0) [8](#0-7) [9](#0-8)

```rust
pub fn compute_fee_actual(
    fee_proposals_window: &BTreeMap<BlockNumber, Option<GasPrice>>,
    height: BlockNumber,
    window_size: u64,
) -> Option<GasPrice> {
    let Some(start) = height.0.checked_sub(window_size) else {
        warn!(
            "Cannot compute fee_actual for height {height}: height is below window_size \
             ({window_size})"
        );
        return None;
    };
    let window_size_usize = usize::try_from(window_size).expect("window_size fits in usize");
    let mut window = Vec::with_capacity(window_size_usize);
    for source_height in (start..height.0).map(BlockNumber) {
        match fee_proposals_window.get(&source_height) {
            Some(Some(price)) => window.push(*price),
            Some(None) | None => {
                warn!(
                    "Cannot compute fee_actual for height {height}: fee_proposals_window has no \
                     recorded fee_proposal for height {source_height}"
                );
                return None;
            }
        }
    }
    window.sort();
    let mid = window_size_usize / 2;
    let median = if window_size_usize.is_multiple_of(2) {
        // Even: average of the two middle values, rounded down.
        // Overflow-safe averaging: a + (b - a) / 2 (safe because sorted, so b >= a).
        GasPrice(window[mid - 1].0 + (window[mid].0 - window[mid - 1].0) / 2)
    } else {
        window[mid]
    };
    Some(median)
}
```
