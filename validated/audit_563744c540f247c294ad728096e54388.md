### Title
Reentrancy in `initTransfer` / `initTransfer1155` Causes Duplicate `originNonce` in Emitted Bridge Messages — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer` and `OmniBridge.initTransfer1155` increment `currentOriginNonce` at the top of the function but then pass the **live storage variable** `currentOriginNonce` — not a local snapshot — to `initTransferExtension` and the `InitTransfer` event after making an external token call. A reentrant call during that external call increments `currentOriginNonce` again, so both the outer and the reentrant invocation publish a Wormhole message carrying the same `originNonce`. The NEAR bridge receives two distinct VAAs that both claim the same origin nonce, which can produce double-credit of bridged assets.

### Finding Description

In `initTransfer` the execution order is:

1. `currentOriginNonce += 1` — storage written to value **N**
2. External token call (`safeTransferFrom` for plain ERC-20, or `burn` for bridge tokens / custom minters)
3. `initTransferExtension(…, currentOriginNonce, …)` — reads storage again; if reentered, value is now **N+1**
4. `emit InitTransfer(…, currentOriginNonce, …)` — same stale read [1](#0-0) [2](#0-1) [3](#0-2) 

The identical pattern exists in `initTransfer1155`: [4](#0-3) [5](#0-4) [6](#0-5) 

Neither function carries a `nonReentrant` modifier, and the contract does not inherit `ReentrancyGuard`. [7](#0-6) 

**Reentrancy vectors (unprivileged):**

- **ERC-20 path** (`initTransfer`, non-bridge-token branch): `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` calls into an arbitrary token. An ERC-777 token or any token with a `_beforeTokenTransfer` / `tokensToSend` hook can re-enter `initTransfer` before the outer call reaches `initTransferExtension`. [8](#0-7) 

- **ERC-1155 path** (`initTransfer1155`): `IERC1155(tokenAddress).safeTransferFrom(…)` calls into an arbitrary ERC-1155 token. A malicious token can fire a `_beforeTokenTransfer` hook that re-enters `initTransfer1155` before the outer call's `initTransferExtension` executes. [5](#0-4) 

**Collision mechanics:**

| Step | `currentOriginNonce` | Outer call nonce used | Reentrant call nonce used |
|------|---------------------|-----------------------|--------------------------|
| Outer enters, increments | N | — | — |
| External call triggers reentry | — | — | — |
| Reentrant enters, increments | N+1 | — | — |
| Reentrant calls `initTransferExtension` | N+1 | — | **N+1** |
| Outer resumes, calls `initTransferExtension` | N+1 | **N+1** | — |

Both Wormhole messages carry `originNonce = N+1`. Nonce N is never published. The NEAR bridge receives two VAAs with identical `originNonce` values.

### Impact Explanation

Two Wormhole messages with the same `originNonce` reach the NEAR bridge. If the NEAR side deduplicates by Wormhole VAA sequence number (not by `originNonce`), both are accepted and the recipient is credited twice for a single on-chain token lock/burn — producing unbacked wrapped supply or a double-release of locked native tokens. Even if NEAR rejects the second, nonce N is permanently skipped, breaking the monotonic nonce invariant that relayers and indexers rely on for settlement ordering. This maps directly to the allowed impact: **replayable cross-chain settlement producing double-credit or unbacked supply**.

### Likelihood Explanation

The attack requires only that the bridged token supports a transfer hook (ERC-777, or any ERC-20/ERC-1155 with `_beforeTokenTransfer`). Such tokens are common and the bridge accepts arbitrary token addresses for the non-bridge-token path. No privileged role is needed; any user can deploy or use a hookable token and call `initTransfer` / `initTransfer1155` directly.

### Recommendation

1. **Snapshot the nonce locally** before any external call and pass the snapshot to `initTransferExtension` and `emit`:

```solidity
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 originNonce = currentOriginNonce; // snapshot
    ...
    // external token call
    ...
    initTransferExtension(msg.sender, tokenAddress, originNonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, originNonce, ...);
}
```

2. Alternatively, add OpenZeppelin's `ReentrancyGuard` (`nonReentrant` modifier) to both `initTransfer` and `initTransfer1155`.

### Proof of Concept

```solidity
// Malicious ERC-777-style token
contract HookableToken is ERC20 {
    OmniBridge public bridge;
    bool private _entered;

    function _beforeTokenTransfer(address, address, uint256) internal override {
        if (!_entered) {
            _entered = true;
            // Re-enter initTransfer with a second transfer
            bridge.initTransfer(address(this), 1e18, 0, 0, "near:attacker.near", "");
        }
    }
}

// Attack:
// 1. Deploy HookableToken, approve bridge for 2e18.
// 2. Call bridge.initTransfer(hookableToken, 1e18, ...).
//    - currentOriginNonce becomes N.
//    - safeTransferFrom triggers _beforeTokenTransfer hook.
//    - Hook re-enters initTransfer: currentOriginNonce becomes N+1.
//    - Reentrant initTransferExtension publishes Wormhole msg with nonce N+1.
//    - Outer initTransferExtension publishes Wormhole msg with nonce N+1.
// Result: two VAAs with originNonce=N+1; nonce N never emitted.
// NEAR processes both → double credit.
```
