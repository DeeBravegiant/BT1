### Title
Single Byzantine Participant Can Deliver Forged Confidential Key Derivation Output via Unverified `respond_ckd` for `AppPublicKey` Variant - (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_ckd` contract method performs cryptographic output verification **only** for the `AppPublicKeyPV` variant of a CKD request. For the `AppPublicKey` variant, the contract performs **zero** cryptographic checks on the submitted `CKDResponse` before resolving the pending yield and delivering the result to the requesting user. Any single attested participant — a Byzantine node strictly below the signing threshold — can race the legitimate leader and submit an arbitrary forged CKD response that the contract unconditionally accepts, bypassing the threshold requirement entirely.

---

### Finding Description

In `respond_ckd`, after the caller passes the attestation gate (`assert_caller_is_attested_participant_and_protocol_active`), the contract branches on the request's `app_public_key` variant:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKey`, the empty arm `{}` means the contract immediately proceeds to `resolve_yields_for` with whatever `CKDResponse` the caller supplied. There is no signature check, no commitment check, and no threshold-vote requirement. The contrast with the `AppPublicKeyPV` arm — which calls `ckd_output_check` against the domain's BLS12-381 public key — makes the omission structural, not incidental.

The `respond` method for ordinary signatures does not share this flaw: it always derives the expected public key from the stored root key and the request's tweak, then verifies the submitted ECDSA/EdDSA signature before resolving the yield. [2](#0-1) 

The `respond_ckd` attestation gate is: [3](#0-2) 

Passing this gate requires only that the caller is an attested participant — a property held by every node in the network, including a single Byzantine one.

The `args_into_verify_foreign_tx_request` helper confirms that the `VerifyForeignTransactionRequest` key used for fan-out does not bind the caller's account ID, but the primary root cause here is the missing verification in `respond_ckd`. [4](#0-3) 

---

### Impact Explanation

A single Byzantine participant (strictly below the signing threshold) can:

1. Monitor the contract for a pending `request_app_private_key` call that uses the `AppPublicKey` variant.
2. Construct an arbitrary `CKDResponse` containing attacker-controlled key material.
3. Call `respond_ckd(request, forged_response)` before the legitimate leader.
4. The contract passes the attestation check, skips all verification for `AppPublicKey`, and calls `resolve_yields_for` — delivering the forged key material to the user.

Because the attacker chose the key material, they know the derived key. Any data the user subsequently encrypts with that key is readable by the attacker; any system that trusts the derived key as a user credential is compromised. This constitutes **unauthorized confidential key derivation output without the required participant authorization**, matching the Critical impact tier.

---

### Likelihood Explanation

The attack requires only one attested participant to act maliciously and to observe a pending `AppPublicKey` CKD request on-chain. Both conditions are realistic: the contract's pending-request map is public state, and a single node in a production MPC cluster is a plausible Byzantine actor. No collusion, no key leakage, and no network-level capability is needed.

---

### Recommendation

Apply the same cryptographic output check to `AppPublicKey` responses that is already applied to `AppPublicKeyPV` responses. If a verifiable commitment to the derived key is not available for the `AppPublicKey` variant by design, require threshold-many participants to submit identical responses before the yield is resolved, mirroring the threshold requirement that governs the underlying MPC signing protocol.

---

### Proof of Concept

```
1. User calls request_app_private_key({ app_public_key: AppPublicKey(pk), domain_id, ... })
   → contract enqueues CKDRequest in pending_ckd_requests, yields execution.

2. Byzantine participant B observes the pending CKDRequest on-chain.

3. B constructs forged_response = CKDResponse { ... attacker-controlled key bytes ... }.

4. B calls respond_ckd(request, forged_response):
   a. assert_caller_is_signer()                          → passes (B signs directly)
   b. is_running_or_resharing()                          → passes
   c. accept_requests                                    → passes
   d. assert_caller_is_attested_participant_...          → passes (B is attested)
   e. match AppPublicKey(_) => {}                        → NO verification, falls through
   f. resolve_yields_for(pending_ckd_requests, request,
        serde_json::to_vec(&forged_response))            → yield resolved, user receives forged key

5. Legitimate leader's subsequent respond_ckd call finds no pending yield and fails silently.

6. User holds attacker-known key material; attacker can decrypt or impersonate.
``` [5](#0-4)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
