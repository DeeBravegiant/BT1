### Title
Missing CKD Output Verification in `respond_ckd` for `AppPublicKey` Variant Allows Single Attested Participant to Deliver Forged Key Derivation — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` enforces a cryptographic correctness check on CKD responses only for the `AppPublicKeyPV` (publicly verifiable) variant. For the `AppPublicKey` (privately verifiable) variant the match arm is intentionally empty — no on-chain verification is performed. A single attested participant, acting strictly below the signing threshold, can call `respond_ckd` with an arbitrary forged `CKDResponse` for any live `AppPublicKey` CKD request and the contract will accept and deliver it to the requesting user.

---

### Finding Description

The TOCTOU structure mirrors the reported bug exactly:

**Check at request time** — `request_app_private_key` validates the domain exists, has a BLS key, and the caller paid the deposit. The user implicitly expects the threshold CKD protocol to run before any response is accepted.

**Gap** — The threshold CKD protocol runs off-chain. No slot or participation proof is reserved on-chain.

**Action at response time** — `respond_ckd` accepts the response and resolves the yield without re-checking that the threshold protocol was actually executed.

In `respond_ckd`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the response is cryptographically consistent with the domain's BLS public key and the user's app key pair. For `AppPublicKey`, the match arm is empty — any `CKDResponse` with any `big_y` and `big_c` values is accepted verbatim.

The only gate protecting `respond_ckd` is `assert_caller_is_attested_participant_and_protocol_active()`, which confirms the caller holds a valid TEE attestation and is in the participant set. [2](#0-1) 

This does **not** verify that a threshold of participants agreed on the response — it only verifies that the single caller is attested.

---

### Impact Explanation

A single attested participant (one compromised TEE node or one malicious operator) can:

1. Monitor `pending_ckd_requests` for live `AppPublicKey` CKD requests.
2. Craft a `CKDResponse` where `big_c` is an ElGamal-style encryption of an attacker-controlled private key under the user's `app_public_key`, and `big_y` is the corresponding attacker-controlled public key.
3. Call `respond_ckd` before the request times out.
4. The contract resolves the yield and delivers the forged `(big_y, big_c)` to the user.
5. The user decrypts `big_c` with their app private key and obtains the attacker-controlled private key, believing it to be their legitimate derived key.
6. The attacker also knows that private key and can sign on behalf of the user or drain any assets deposited to the derived address.

This is **confidential key derivation output delivered without the required threshold participant authorization** — a critical impact under the allowed scope.

---

### Likelihood Explanation

**Low.** Exploitation requires a single attested participant to be malicious (compromised TEE enclave or a colluding operator). TEE attestation raises the bar, but a single node below the signing threshold is sufficient — no collusion at or above threshold is needed. The attack window is bounded by the yield timeout, but the attacker controls the timing of their `respond_ckd` call.

---

### Recommendation

Add on-chain verification for the `AppPublicKey` variant in `respond_ckd`, analogous to the existing `ckd_output_check` for `AppPublicKeyPV`. If the privately verifiable variant structurally cannot be verified on-chain (because the contract does not hold the user's private key), consider:

- Requiring the response to be accompanied by a threshold-signed proof of participation (e.g., a multi-sig over the response hash from ≥ t participants).
- Switching `AppPublicKey` CKD to the `AppPublicKeyPV` path, which already has on-chain verification.
- At minimum, documenting the trust assumption explicitly and providing client-side tooling that lets users independently verify the CKD output before using the derived key.

---

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
