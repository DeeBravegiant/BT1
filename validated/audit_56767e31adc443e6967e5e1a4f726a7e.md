### Title
Unbounded Synchronous Iteration Over All Registered DReps in `updateDormantDRepExpiry` Triggered by Any Governance Proposal Transaction - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

The `updateDormantDRepExpiry` function performs a full `Map.map` traversal over every registered DRep in `vsDReps` synchronously during transaction validation whenever a transaction includes at least one governance proposal procedure. Because there is no protocol-enforced cap on the number of registered DReps and the DRep deposit is fully refundable upon deregistration, an attacker can register an arbitrarily large number of DReps, allow dormant epochs to accumulate, and then submit a single proposal-bearing transaction to force O(N) synchronous work inside the CERTS/LEDGER rule for every validating node. This is the direct Cardano analog of the StakePet unbounded-list-iteration class.

---

### Finding Description

**Root cause — `updateDormantDRepExpiry`**

`updateDormantDRepExpiry` unconditionally maps over the entire `vsDReps` map:

```haskell
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry   -- O(|vsDReps|), no bound
``` [1](#0-0) 

**Trigger — `updateDormantDRepExpiries`**

This is called inside `conwayCertsTransition` (the base case of the CERTS rule) whenever the transaction body contains at least one proposal procedure:

```haskell
updateDormantDRepExpiries tx currentEpoch =
  let hasProposals = not . OSet.null $ tx ^. bodyTxL . proposalProceduresTxBodyL
   in if hasProposals
        then certVStateL %~ updateDormantDRepExpiry currentEpoch
        else id
``` [2](#0-1) 

The CERTS transition rule calls this in its base case:

```haskell
pure $
  certState
    & updateDormantDRepExpiries tx currentEpoch
    & updateVotingDRepExpiries tx currentEpoch (pp ^. ppDRepActivityL)
    & certDStateL . accountsL %~ drainAccounts withdrawals
``` [3](#0-2) 

**No cap on registered DReps**

DRep registration (`ConwayRegDRep`) inserts into `vsDReps` with no upper-bound check:

```haskell
pure $
  certState
    & certVStateL . vsDRepsL %~ Map.insert cred drepState
``` [4](#0-3) 

The only cost is the `drepDeposit` protocol parameter, which is fully refundable on deregistration (`ConwayUnRegDRep`). [5](#0-4) 

**Contrast with the pulsed DRep stake computation**

The developers explicitly recognized that iterating over all accounts/DReps is expensive and designed the `DRepPulser` to spread that work across `4k` blocks:

```haskell
pulseSize = max 1 (fromIntegral numAccounts %. (knownNonZero @4 `mulNonZero` toIntegerNonZero k))
``` [6](#0-5) 

`updateDormantDRepExpiry` receives no such protection — it is a single synchronous `Map.map` executed inline during block validation.

---

### Impact Explanation

Every validating node must execute `Map.map updateExpiry` over the full `vsDReps` map for every block that contains a proposal-bearing transaction while `numDormantEpochs > 0`. The work is O(N) in the number of registered DReps with no protocol-level bound. At sufficient scale this inflates per-block validation wall-clock time beyond the slot window, causing honest block producers to miss their leadership slots and degrading chain liveness. Because the computation is deterministic, all nodes are equally affected, so this does not produce ledger divergence — the impact category is:

**Medium — attacker-controlled transactions exceed intended validation limits.**

---

### Likelihood Explanation

The `drepDeposit` on mainnet is 500 ADA and is fully refundable. An attacker who can temporarily lock up capital can register an unbounded number of DReps, wait for dormant epochs to accumulate naturally (no proposals need to be submitted by anyone during that window), and then submit a single `InfoAction` proposal to trigger the traversal. The attack is repeatable across epochs. The capital requirement scales linearly with N but is recovered in full after deregistration, making this a loan-style griefing attack rather than a permanent capital loss for the attacker.

---

### Recommendation

1. **Cap registered DReps** — introduce a protocol parameter `maxDReps` (analogous to `maxCollateralInputs`) and enforce it in `ConwayRegDRep`.
2. **Pulse `updateDormantDRepExpiry`** — move the dormant-epoch expiry bump into the existing `DRepPulser` so it is spread across blocks rather than executed synchronously per transaction.
3. **Raise the non-refundable component of `drepDeposit`** — even a small non-refundable fee makes mass registration economically irrational.

---

### Proof of Concept

1. Register N DReps from attacker-controlled credentials, each paying `drepDeposit` ADA (refundable). No governance threshold is required; any credential may register.
2. Allow ≥1 dormant epoch to elapse (no proposal submitted by anyone), incrementing `vsNumDormantEpochs` to a non-zero value.
3. Submit a single transaction containing one `InfoAction` proposal procedure (paying `govActionDeposit`).
4. During CERTS rule evaluation, `updateDormantDRepExpiries` detects `hasProposals = True` and calls `updateDormantDRepExpiry`, which executes `Map.map updateExpiry` over all N entries in `vsDReps`.
5. Every validating node performs O(N) synchronous work inside block validation for this transaction. Deregister all N DReps to recover the deposited ADA.

updateDormantDRepExpiry ::
  -- | Current Epoch
  EpochNo ->
  VState era ->
  VState era
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
  where
    numDormantEpochs = vState ^. vsNumDormantEpochsL
    updateExpiry =
      drepExpiryL
        %~ \currentExpiry ->
          let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
           in if actualExpiry < currentEpoch
                then currentExpiry
                else actualExpiry
