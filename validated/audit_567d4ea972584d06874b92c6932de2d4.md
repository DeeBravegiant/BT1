### Title
Unenforced CHIP-22 Harvester Fee Threshold Allows Unconditional Farmer XCH Reward Diversion - (File: `chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

---

### Summary

A malicious third-party harvester can unconditionally redirect the farmer's XCH block reward to an arbitrary address by setting `farmer_reward_address_override` in the harvester protocol messages. The farmer's software only logs a warning when the CHIP-22 fee quality threshold is not met — it never rejects or blocks the override. The farmer then signs and broadcasts a block whose `farmer_puzzle_hash` is the harvester's chosen address, permanently diverting the farmer's coinbase reward.

---

### Finding Description

CHIP-22 defines a fee convention allowing third-party harvesters to take a fraction of the farmer's block reward by setting `farmer_reward_address_override` in `NewProofOfSpace`. The farmer is supposed to verify that the "fee quality" (a deterministic hash of the proof and challenge) meets the harvester's declared `applied_fee_threshold` before accepting the override.

**Step 1 — Threshold check is logging-only, not enforced.**

In `farmer_api.py`, when a `NewProofOfSpace` arrives with `farmer_reward_address_override` set:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
```

`notify_farmer_reward_taken_by_harvester_as_fee()` in `farmer.py` computes `fee_quality` and compares it to `fee_info.applied_fee_threshold`, but only emits log messages — it returns `None` unconditionally and the caller ignores any result:

```python
if fee_quality <= fee_threshold:
    self.log.info(f"Fee threshold passed ...")
else:
    self.log.warning("Harvester illegitimately took a fee reward ...")
```

There is no `return`, no exception, no flag set, and no rejection of the proof. Execution continues regardless.

**Step 2 — `RespondSignatures` override is used with zero fee quality check.**

After the farmer requests signatures from the harvester, `_process_respond_signatures()` unconditionally substitutes the harvester's address:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

No fee quality computation or threshold comparison is performed here at all. The harvester can set any `bytes32` as `farmer_reward_address_override` in `RespondSignatures` and it will be used.

**Step 3 — The farmer signs a block paying the harvester.**

The resulting `DeclareProofOfSpace` carries `farmer_puzzle_hash = harvester_address`. The full node creates an `UnfinishedBlock` with `farmer_ph = request.farmer_puzzle_hash` and sends `RequestSignedValues` back to the farmer. The farmer signs the foliage block data — which commits to the harvester's address as the farmer reward puzzle hash — and the block is broadcast to the network. The farmer's XCH coinbase reward is permanently sent to the harvester's address.

---

### Impact Explanation

**Critical — Unauthorized reward diversion of XCH.**

Every time the malicious harvester wins a block, the farmer's entire XCH farmer reward (currently 0.25 XCH per block) is redirected to the harvester's address. The farmer's own software signs the diversion. The loss is permanent and on-chain. No keys are leaked; the farmer's signing key is used legitimately but against the farmer's interest because the fee threshold enforcement is absent.

---

### Likelihood Explanation

Any operator running a third-party harvester (the explicit use-case of CHIP-22) is exposed. A malicious harvester simply always sets `farmer_reward_address_override` to its own address and sets `applied_fee_threshold` to `0xFFFFFFFF` (maximum, so the quality check always fails) or omits `fee_info` entirely. The farmer logs a warning but proceeds. The attack requires only a network connection from the harvester to the farmer — no special privileges, no key material.

---

### Recommendation

In `new_proof_of_space()`, `notify_farmer_reward_taken_by_harvester_as_fee()` should return a boolean indicating whether the threshold was met, and the caller should abort block-making if it returns `False`:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    if not self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space):
        return None  # reject: fee threshold not met
```

In `_process_respond_signatures()`, the same fee quality check must be re-applied against the proof stored in `pospace` before accepting `response.farmer_reward_address_override`.

---

### Proof of Concept

1. A malicious harvester connects to a farmer over the standard harvester protocol.
2. On receiving a `NewSignagePointHarvester`, the harvester finds a valid proof of space and sends `NewProofOfSpace` with:
   - `farmer_reward_address_override = <attacker_puzzle_hash>`
   - `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0xFFFFFFFF)` (threshold impossible to meet)
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`:
   - `fee_quality` is computed from the proof — always `<= 0xFFFFFFFF` but the comparison `fee_quality <= fee_threshold` is `True` only if `fee_quality <= 0xFFFFFFFF`, which is always true — wait, actually with threshold `0xFFFFFFFF` the check passes. Let me correct: the attacker sets threshold to `0` so `fee_quality <= 0` is almost never true, triggering the warning path. Either way, execution continues.
4. The farmer sends `RequestSignatures` to the harvester.
5. The harvester responds with `RespondSignatures` containing `farmer_reward_address_override = <attacker_puzzle_hash>`.
6. `_process_respond_signatures()` sets `farmer_reward_address = <attacker_puzzle_hash>` with no check.
7. `DeclareProofOfSpace` is sent to the full node with `farmer_puzzle_hash = <attacker_puzzle_hash>`.
8. The full node creates the block: `farmer_ph = request.farmer_puzzle_hash` (line 1069 of `full_node_api.py`).
9. The farmer signs the foliage block data committing to `<attacker_puzzle_hash>`.
10. The block is accepted by the network; the farmer's XCH reward is paid to the attacker.

```python
    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
```
