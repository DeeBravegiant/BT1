Audit Report

## Title
Missing Staleness Check in `CrossChainRateReceiver.getRate()` Allows Stale Rate to Price Pool Swaps — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver` stores both `rate` and `lastUpdated` but `getRate()` returns `rate` unconditionally without ever reading `lastUpdated`. If LayerZero message delivery halts for any reason, the stored rate silently ages and every pool contract pointing its `rsETHOracle` at a `CrossChainRateReceiver` derivative will continue pricing deposits using an arbitrarily old rsETH/ETH rate, over-issuing rsETH to depositors and diluting existing holders.

## Finding Description
`CrossChainRateReceiver` declares `rate` and `lastUpdated` as state variables ( [1](#0-0) ) and writes `lastUpdated = block.timestamp` only inside `lzReceive()` when a cross-chain message arrives. [2](#0-1) 

`getRate()` returns `rate` with no reference to `lastUpdated`: [3](#0-2) 

`RSETHPoolV2.getRate()` delegates directly to this function: [4](#0-3) 

That rate is then used as the denominator in `viewSwapRsETHAmountAndFee()`, which computes the rsETH minted per deposit: [5](#0-4) 

The same pattern is present in `RSETHPool.viewSwapRsETHAmountAndFee()`: [6](#0-5) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` explicitly reverts on stale data, confirming the team is aware of the pattern: [7](#0-6) 

No existing guard in any pool variant checks `lastUpdated` before accepting a deposit. The `paused` flag and `dailyMintLimit` are the only circuit-breakers, neither of which is triggered automatically by oracle staleness.

## Impact Explanation
rsETH's ETH-denominated rate increases monotonically. A stale (lower) rate causes `amountAfterFee * 1e18 / rsETHToETHrate` to yield a larger rsETH amount than the depositor is entitled to at the true current rate. Depositors transacting during a stale-rate window receive more rsETH than their ETH is worth, diluting existing holders. The pool continues to accept deposits and mint tokens throughout the outage with no revert or warning.

**Impact: Low — contract fails to deliver promised returns, but doesn't lose value.** The pool's ETH balance is not directly drained; the harm is incorrect rsETH issuance relative to the true current rate.

## Likelihood Explanation
The trigger requires `lzReceive` to go uncalled for an extended period. This is realistic: LayerZero relayer outages have occurred on live networks; `updateRate()` on the provider side is permissionless but requires a caller to pay the cross-chain fee, so if no one calls it the rate ages indefinitely. No admin compromise, governance capture, or victim mistake is required — any depositor calling `deposit()` during the stale window triggers the impact.

## Recommendation
Add a configurable `maxStaleness` parameter and revert in `getRate()` if `block.timestamp - lastUpdated > maxStaleness`:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(
        lastUpdated != 0 && block.timestamp - lastUpdated <= maxStaleness,
        "Rate is stale"
    );
    return rate;
}
```

This mirrors the staleness guard already present in `ChainlinkOracleForRSETHPoolCollateral`. [8](#0-7) 

## Proof of Concept
1. Deploy `RSETHRateReceiver` (concrete subclass of `CrossChainRateReceiver`) with a mock LayerZero endpoint.
2. Call `lzReceive` with `rate = 1.05e18` at `T=0`; confirm `lastUpdated == T`.
3. Warp `block.timestamp` forward by 7 days.
4. Call `getRate()` — returns `1.05e18` (stale, no revert).
5. Call `RSETHPoolV2.viewSwapRsETHAmountAndFee(1 ether)`:
   - Stale: `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.9524 rsETH`
   - Current (e.g. `1.07e18`): `rsETHAmount = 1e18 * 1e18 / 1.07e18 ≈ 0.9346 rsETH`
   - Delta: ~0.0178 rsETH over-issued per ETH (~1.9%)
6. Assert `lastUpdated` is 7 days old and pool output diverges from the expected output at the current oracle price.

No privileged access is required — only a time warp after a normal `lzReceive` call, followed by a public `deposit()`.
