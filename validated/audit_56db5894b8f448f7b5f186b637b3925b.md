### Title
`respond_verify_foreign_tx` Accepts Forged `payload_hash` Uncorrelated with the Pending Request — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over the submitted `response.payload_hash`, but never verifies that `payload_hash` is actually derived from the pending `request`. A single malicious attested participant (strictly below the signing threshold) can replay a valid signature from any prior on-chain signing round to resolve any pending `verify_foreign_transaction` request with a forged `payload_hash`, causing the bridge contract to receive an invalid attestation.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract performs exactly two semantic checks before resolving the yield:

1. The caller is an attested participant.
2. The ECDSA signature is valid over `response.payload_hash` under the root public key. [1](#0-0) 

There is no check that `response.payload_hash` is actually derived from `request`. The intended invariant is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload { request: request.request, values: <extracted_values> }))
```

as documented and implemented in the SDK-side verifier: [2](#0-1) 

and as the node builds it before signing: [3](#0-2) 

The contract, however, never enforces this binding. It accepts any `(payload_hash, signature)` pair where the signature is valid over `payload_hash` under the root key, regardless of whether `payload_hash` has any relationship to the `request` being resolved.

This is the direct structural analog to the beacon-kit bug: just as `blkDeposits` (D) was never verified against the deposit store even though `blkDepositRoot` (R) was, here `response.payload_hash` is never verified against `request` even though `response.signature` is verified against `response.payload_hash`. [4](#0-3) 

---

### Impact Explanation

**Cross-request replay / forged foreign-chain verification (High).**

A single malicious attested participant can:

1. Observe any prior `respond_verify_foreign_tx` call on-chain and extract a valid `(payload_hash_Y, sig_Y)` pair (the signature is a public transaction argument).
2. Identify any currently pending `verify_foreign_transaction` request X.
3. Call `respond_verify_foreign_tx(request=X, response={payload_hash=payload_hash_Y, signature=sig_Y})`.
4. The contract accepts because `verify_ecdsa_signature(sig_Y, payload_hash_Y, root_pk)` passes.
5. The yield for request X is resolved and removed from `pending_verify_foreign_tx_requests`. [5](#0-4) 

The user's bridge contract then receives `response={payload_hash_Y, sig_Y}` — a response attesting to a completely different foreign-chain transaction (Y) than the one requested (X).

- **If the bridge contract uses the SDK's `ForeignChainSignatureVerifier::verify_signature`**, it detects `expected_payload_hash_X ≠ payload_hash_Y` and rejects the response. The pending request is permanently consumed (DoS — the user cannot retry because the yield is already resolved).
- **If the bridge contract does not use the SDK verifier** (a realistic scenario for custom bridge contracts that only check the signature), it is fooled into believing a different foreign-chain transaction was verified, enabling unauthorized bridge execution or double-spend conditions.

---

### Likelihood Explanation

- Requires only a **single** malicious attested participant — strictly below the signing threshold.
- The replayed `(payload_hash, signature)` pair is **publicly observable** on-chain from any prior `respond_verify_foreign_tx` transaction.
- No cryptographic capability beyond reading on-chain data is needed.
- The attack is mechanically straightforward and requires no coordination with other nodes.

---

### Recommendation

The contract must bind the response to the specific request. Since the contract does not know the extracted values (and therefore cannot recompute `payload_hash` from `request` alone), the recommended fix is to include a commitment to the `request` inside the signed payload at the node level, so that a signature produced for request Y is cryptographically unusable for request X. Concretely:

- Include a hash of the `VerifyForeignTransactionRequest` (or its unique identifier) as an additional field in `ForeignTxSignPayloadV1`, so that `payload_hash` is request-specific.
- Alternatively, the contract can store a per-request commitment (e.g., `SHA-256(borsh(request))`) at enqueue time and require that `payload_hash` starts with or is derived from that commitment, rejecting any response where the binding is absent. [3](#0-2) 

---

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(request=X) on-chain.
   → pending_verify_foreign_tx_requests[X] = [yield_id_alice]

2. Attacker (single attested participant) scans on-chain history and finds
   a prior respond_verify_foreign_tx call for request Y:
     payload_hash_Y = <observed from tx args>
     sig_Y          = <observed from tx args>
   Both are valid: verify_ecdsa_signature(sig_Y, payload_hash_Y, root_pk) = OK.

3. Attacker calls:
     respond_verify_foreign_tx(
       request  = X,                                    // matches Alice's pending request
       response = { payload_hash: payload_hash_Y,       // forged — derived from request Y
                    signature:    sig_Y }               // valid over payload_hash_Y
     )

4. Contract check (lib.rs:729-733):
     verify_ecdsa_signature(sig_Y, payload_hash_Y, root_pk) → OK  ✓
   No check that payload_hash_Y relates to request X.

5. resolve_yields_for(request=X, serialized(response)) fires:
   → Alice's yield is resolved with {payload_hash_Y, sig_Y}.
   → pending_verify_foreign_tx_requests[X] is removed.

6. Alice's bridge contract receives {payload_hash_Y, sig_Y}.
   - SDK verifier: detects payload_hash_Y ≠ expected_payload_hash_X → rejects.
     Alice's request is permanently consumed; she cannot retry. (DoS)
   - Custom verifier checking only the signature: accepts the forged attestation.
     Bridge executes as if foreign tx X was verified. (Forgery / double-spend)
```
