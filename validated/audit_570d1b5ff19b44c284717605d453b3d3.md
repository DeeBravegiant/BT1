### Title
Attached NEAR Permanently Locked in `submit_participant_info` When Existing Participant Re-Submits Attestation — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `submit_participant_info` function is marked `#[payable]` but only executes deposit-handling logic in one of its two execution branches. When an existing participant re-submits their attestation (e.g., to refresh it), the deposit-handling block is skipped entirely. Any NEAR attached to such a call is permanently locked in the contract with no retrieval mechanism.

---

### Finding Description

`submit_participant_info` (line 758) is declared `#[payable]`, signalling to callers that a NEAR deposit may be required. The function computes a boolean gate:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
``` [1](#0-0) 

When this evaluates to `true` (first-time submission or non-participant), the function correctly charges for storage and refunds any excess:

```rust
if let Some(diff) = attached.checked_sub(cost)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(account_id).transfer(diff).detach();
}
``` [2](#0-1) 

However, when the condition is `false` — i.e., the caller is an **existing participant** re-submitting an updated attestation — the entire deposit-handling block is skipped. There is no `else` branch, no refund, and no assertion that `env::attached_deposit() == 0`. Any NEAR attached to the call is silently absorbed into the contract balance with no mechanism to recover it. [3](#0-2) 

The function's `#[payable]` attribute at line 758 is the root cause: it unconditionally permits deposits regardless of which execution path will be taken. [4](#0-3) 

Compare with the analogous refund logic used correctly in `require_deposit` for `sign`/`request_app_private_key`/`verify_foreign_transaction`, which always refunds excess: [5](#0-4) 

---

### Impact Explanation

Any NEAR attached by an existing participant when re-submitting their attestation is permanently locked in the contract. The contract has no withdrawal function, no sweep mechanism, and no owner-controlled drain. The locked NEAR is irrecoverable. This breaks the accounting invariant that all deposited NEAR must either be consumed for storage or refunded to the caller — matching the **Medium** allowed impact: *"Balance… manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

Participants are sophisticated actors who run MPC nodes. However:

1. The function is `#[payable]`, which by NEAR convention signals that a deposit is expected or required.
2. The **first** call to `submit_participant_info` from a new account **does** require a deposit (for storage). A participant who re-submits (e.g., after a TEE upgrade) may reasonably attach NEAR again, following the same pattern as their initial call.
3. Tooling, scripts, or SDKs that hardcode a deposit for this method will silently lock funds on every re-submission.
4. There is no on-chain or documentation guard that warns callers not to attach NEAR on re-submissions.

Likelihood is **Low-Medium**: the scenario requires a participant to attach NEAR unnecessarily, but the API design actively encourages this mistake.

---

### Recommendation

Add an unconditional refund of any attached deposit in the branch where `attestation_storage_must_be_paid_by_caller` is `false`:

```rust
} else {
    // Existing participant re-submission: no storage cost, refund any deposit.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, assert `env::attached_deposit() == NearToken::from_yoctonear(0)` in that branch to make the invariant explicit and fail fast rather than silently locking funds.

---

### Proof of Concept

1. Participant `alice.near` calls `submit_participant_info` for the first time, attaching 0.01 NEAR for storage. The deposit is correctly charged and excess refunded.
2. After a TEE upgrade, `alice.near` re-submits her attestation. Her tooling attaches 0.01 NEAR again (same call pattern).
3. Because `is_new_attestation = false` and `caller_is_not_participant = false`, `attestation_storage_must_be_paid_by_caller = false`.
4. The deposit-handling block is skipped. The 0.01 NEAR is absorbed into the contract balance.
5. There is no function in the contract to withdraw or recover this NEAR. It is permanently locked. [6](#0-5)
