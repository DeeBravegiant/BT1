Audit Report

## Title
`protocolUnpausePool` Sets Admin-Paused (Level 1) Instead of Active (Level 0), Permanently Blocking Swaps — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`protocolUnpausePool` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). Because `protocolPausePool` accepts level-0 pools as a valid source, any active pool driven through a protocol pause/unpause cycle ends in admin-paused state. Swaps remain blocked and the protocol owner has no unilateral path to restore level 0; only the pool admin can do so via `unpausePool`.

## Finding Description
The pause state machine has three levels: 0 = active, 1 = admin-paused, 2 = protocol-paused.

`protocolPausePool` (L392–396) accepts `cur == 0` or `cur == 1` and sets level 2:
```solidity
if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
IMetricOmmPoolFactoryActions(pool).setPause(2);
```

`protocolUnpausePool` (L399–403) checks `cur == 2` but then sets level **1**, not 0:
```solidity
if (cur != 2) revert InvalidPauseTransition(cur, 1);
IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← bug: should be 0
```

`_checkNotPaused` (L643–644) rejects any non-zero level:
```solidity
if (pauseLevel != 0) revert PoolPaused();
```

`unpausePool` (L467–471) is the only function that sets level 0, and it is gated to `onlyPoolAdmin`:
```solidity
if (cur != 1) revert InvalidPauseTransition(cur, 0);
IMetricOmmPoolFactoryActions(pool).setPause(0);
```

There is no factory-owner path to set level 0 directly. After the cycle `0 → 2 → 1`, the pool is stuck in admin-paused state that it was never in before, and all swap calls revert with `PoolPaused`.

## Impact Explanation
After a single protocol pause/unpause cycle on any live (level-0) pool, every `swap` call reverts with `PoolPaused`. LP fee accrual stops. The protocol owner — who performed the "unpause" — cannot restore the pool unilaterally. If the pool admin key is lost or unresponsive, the pool is permanently frozen. This is broken core pool functionality causing unusable swap flows, meeting the allowed impact gate.

## Likelihood Explanation
Triggered on every protocol pause/unpause cycle applied to a pool at level 0, which is the normal operational state for any live pool. No special attacker setup is required; the protocol owner exercising their documented authority is sufficient. The condition is repeatable and deterministic.

## Recommendation
Change `protocolUnpausePool` to call `setPause(0)` and update the transition guard to match:
```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0); // restore to active
}
```
Alternatively, store the pre-pause level in factory state when `protocolPausePool` is called and restore it here.

## Proof of Concept
1. Pool deployed and active: `pauseLevel = 0`.
2. Protocol owner calls `protocolPausePool(pool)` — allowed (`cur == 0`). Pool: `pauseLevel = 2`.
3. Protocol owner calls `protocolUnpausePool(pool)` — sets `pauseLevel = 1`.
4. Any user calls `swap(...)` → reverts `PoolPaused` (`pauseLevel != 0`).
5. Protocol owner has no further lever; only pool admin can call `unpausePool` (1 → 0).
6. If `poolAdmin[pool]` is unresponsive, pool is permanently paused despite the protocol owner having "unpaused" it.
