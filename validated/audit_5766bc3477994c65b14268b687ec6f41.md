Audit Report

## Title
LP Full-Withdrawal DoS via Griefing `addLiquidity` on Victim's Position — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`addLiquidity` permits any `msg.sender` to mint shares into an arbitrary `owner`'s position without requiring `msg.sender == owner`. `removeLiquidity` enforces a dust floor that reverts when remaining shares fall strictly between 0 and `MINIMAL_MINTABLE_LIQUIDITY`. An attacker can front-run a victim's full-withdrawal by adding a single dust share to the victim's position, causing the victim's transaction to revert indefinitely at negligible per-attempt cost.

## Finding Description
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` parameter and imposes no `msg.sender == owner` check. [1](#0-0) 

The only pre-call guard is `_beforeAddLiquidity`, which dispatches to extension hooks only when `BEFORE_ADD_LIQUIDITY_ORDER != 0`. In the default pool configuration (no `DEPOSIT_ALLOWLIST_PROVIDER` extension), this is a no-op. [2](#0-1) 

The interface NatSpec explicitly documents this as intentional: *"msg.sender pays but need not equal owner (operator pattern)"*. [3](#0-2) 

`LiquidityLib.removeLiquidity` then enforces a dust floor on the remaining share count: [4](#0-3) 

This check fires whenever `0 < newUserShares < MINIMAL_MINTABLE_LIQUIDITY`, with no exception for the case where the caller intended a full exit (`sharesToRemove == userShares` before the front-run).

**Exploit path:**
1. Alice holds `N ≥ MINIMAL_MINTABLE_LIQUIDITY` shares in bin `B`, salt `S`, and submits `removeLiquidity(alice, S, [{B, N}])`.
2. Bob observes the pending calldata, deploys a contract implementing `IMetricOmmModifyLiquidityCallback`, and front-runs with `addLiquidity(alice, S, [{B, 1}])`, paying the proportional token cost for 1 share. The `addLiquidity` dust check passes because `N + 1 ≥ MINIMAL_MINTABLE_LIQUIDITY`. [5](#0-4) 
3. Alice's transaction executes: `userShares = N+1`, `sharesToRemove = N`, `newUserShares = 1`. The check `1 > 0 && 1 < MINIMAL_MINTABLE_LIQUIDITY` is true → `revert MinimalLiquidity(1, MINIMAL_MINTABLE_LIQUIDITY)`.
4. Bob repeats on every resubmission. Alice cannot escape without discovering the exact inflated share count and submitting a new transaction — which Bob can front-run again.

The callback requirement (`IMetricOmmModifyLiquidityCallback`) is not a meaningful barrier; the attacker deploys a trivial contract that transfers the required dust token amount. [6](#0-5) 

## Impact Explanation
The victim's `removeLiquidity` flow is rendered permanently unusable for the targeted position as long as the attacker repeats the front-run. This matches the allowed impact criterion: **broken core pool functionality / unusable withdraw flow**. The victim's principal is not stolen but is inaccessible, which constitutes a material loss of access to LP assets.

## Likelihood Explanation
- No privilege is required; any EOA or contract can call `addLiquidity` with an arbitrary `owner`.
- The victim's `(owner, salt, binIdx)` tuple is fully visible in pending transaction calldata.
- The attacker's per-attempt cost is `1 / binTotalShares` of the bin's token balance — negligible for any reasonably sized bin.
- All pools without a `DEPOSIT_ALLOWLIST_PROVIDER` extension (the default) are fully exposed. [7](#0-6) 

## Recommendation
**Minimal fix in `LiquidityLib.removeLiquidity`:** skip the dust check when the caller is performing a full exit (`sharesToRemove == userShares`), since leaving 0 shares is always valid:

```solidity
uint256 newUserShares = userShares - sharesToRemove;
// Only enforce dust floor on partial removals.
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

Alternatively, allow callers to pass `type(uint256).max` as `sharesToRemove` to mean "remove all", resolving the exact count at execution time. Optionally, restrict `addLiquidity` to require `msg.sender == owner` unless an explicit operator approval exists, eliminating the griefing vector entirely.

## Proof of Concept
```
Setup:
  MINIMAL_MINTABLE_LIQUIDITY = 1000
  Alice holds 10_000 shares in bin 4, salt = 42.

Step 1 – Alice submits:
  pool.removeLiquidity(alice, 42, [{binIdx: 4, shares: 10_000}], "")

Step 2 – Bob front-runs (higher gas, same block):
  // Bob's contract implements IMetricOmmModifyLiquidityCallback
  pool.addLiquidity(alice, 42, [{binIdx: 4, shares: 1}], callbackData, "")
  // addLiquidity dust check: 10_000 + 1 = 10_001 >= 1000 → passes
  // Alice's positionBinShares[key(alice,42,4)] = 10_001

Step 3 – Alice's transaction executes:
  userShares     = 10_001
  sharesToRemove = 10_000
  newUserShares  = 1
  Check: 1 > 0 && 1 < 1000 → true → revert MinimalLiquidity(1, 1000)

Bob repeats on every resubmission at ~1/10_000 of bin token balance per attempt.
```

  }
