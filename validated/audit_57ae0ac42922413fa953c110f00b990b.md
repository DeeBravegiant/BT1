Audit Report

## Title
Missing Deadline Guard on `MetricOmmPoolLiquidityAdder` Allows Stale Liquidity Deposits at Shifted Oracle Prices - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary
`MetricOmmPoolLiquidityAdder` exposes four public liquidity entry points with no `deadline` parameter and no `_checkDeadline` guard, unlike `MetricOmmSimpleRouter` which calls `_checkDeadline(params.deadline)` as the first statement in every swap entry point. A pending transaction can sit in the mempool and execute arbitrarily late, after the oracle price has moved significantly, deploying the user's tokens into bins at a price the user never intended to accept.

## Finding Description
`MetricOmmSimpleRouter` calls `_checkDeadline(params.deadline)` at lines 68, 93, 131, and 155 as the very first statement in every swap entry point. The `_checkDeadline` helper is defined in `MetricOmmSwapRouterBase` (lines 91–94) and reverts with `TransactionExpired` if `block.timestamp > deadline`.

`MetricOmmPoolLiquidityAdder` inherits only from `IMetricOmmPoolLiquidityAdder` and `PeripheryPayments` — not from `MetricOmmSwapRouterBase` — and has no equivalent guard on any of its four entry points (`addLiquidityExactShares` × 2, `addLiquidityWeighted` × 2).

The `addLiquidityWeighted` path is the most dangerous variant. It performs a live probe at execution time (lines 106–112 and 139–145) to determine the current `need0`/`need1` token ratio from the pool's live oracle state, then scales the user's weight vector to that ratio via `_scaleWeightsToShares` (lines 226–243) and pulls tokens from the user. If the transaction executes after the oracle price has moved, the probe reflects the new price, and the user's tokens are deployed into bins calibrated to the new price.

The `_validateBinAndBinPosition` check (lines 263–286) provides only coarse protection: it reverts only if the pool cursor has moved outside the caller-supplied `[minimalCurBin, maximalCurBin]` range. A user who supplies a range of e.g. `[-1, 1]` gets no protection against significant price drift within that range.

`addLiquidityExactShares` (lines 56–68 and 71–81) has no cursor-bounds check at all — it calls `_addLiquidity` directly after only validating array lengths and the owner address.

## Impact Explanation
A user submits `addLiquidityWeighted` when the oracle price is at P₀. The transaction is delayed in the mempool. The oracle price moves to P₁. When the transaction executes, the probe runs at P₁, producing a completely different `need0`/`need1` composition. The user's tokens are pulled (up to `maxAmountToken0`/`maxAmountToken1`) and deposited into bins calibrated to P₁. If the price reverts toward P₀, the LP position immediately suffers impermanent loss relative to the price the user intended to enter at. The `maxAmountToken0`/`maxAmountToken1` caps bound the nominal token outflow but do not protect against the economic loss from deploying capital at the wrong price point. This constitutes a direct, bounded loss of user principal — a Medium-severity impact under Sherlock thresholds.

## Likelihood Explanation
Any LP using the standard periphery contract is exposed with no privileged access required. The `addLiquidityWeighted` variant is the primary user-facing path for LPs who do not want to compute exact shares off-chain. Mempool delays of minutes to hours are routine on Ethereum mainnet and Base during high-activity periods. The condition is triggered by ordinary network congestion, not by any attacker action.

## Recommendation
Add a `deadline` parameter to all four public entry points in `MetricOmmPoolLiquidityAdder` and call `_checkDeadline(deadline)` as the first statement, mirroring the pattern already established in `MetricOmmSimpleRouter`. Since `MetricOmmPoolLiquidityAdder` does not inherit from `MetricOmmSwapRouterBase`, it should either inherit from that base or duplicate the one-liner:

```solidity
function _checkDeadline(uint256 deadline) internal view {
    if (block.timestamp > deadline) revert TransactionExpired(deadline, block.timestamp);
}
```

Apply this to all four entry points: both overloads of `addLiquidityExactShares` and both overloads of `addLiquidityWeighted`.

## Proof of Concept
1. User approves `MetricOmmPoolLiquidityAdder` for up to 10,000 USDC and 5 WETH.
2. User calls `addLiquidityWeighted(pool, salt, weights, 10000e6, 5e18, -1, 0, 1, 1e18, "")` when oracle mid = $2,000/ETH.
3. Transaction is stuck in mempool for 30 minutes; oracle price moves to $2,400/ETH (still within the `[-1, 1]` cursor range, so `_validateBinAndBinPosition` does not revert).
4. Transaction executes: the probe at lines 106–112 runs at $2,400, `need0`/`need1` ratio reflects the new price. `_scaleWeightsToShares` computes a scale factor against the user's caps. Tokens are pulled and deposited into bins around $2,400.
5. Oracle price reverts to $2,000 within the hour. The LP position is now out-of-range on the token0 side; the user has effectively bought ETH at $2,400 with their USDC leg.
6. User removes liquidity and receives fewer tokens in aggregate than they would have at their intended entry price — a direct economic loss bounded by `maxAmountToken0`/`maxAmountToken1` but not zero.
