Audit Report

## Title
Stale Oracle Rate in `CrossChainRateReceiver` Enables Over-Minting of wrsETH, Causing Protocol Insolvency — (`contracts/pools/RSETHPoolV2ExternalBridge.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness check despite `lastUpdated` being recorded on every update. `RSETHPoolV2ExternalBridge.deposit()` uses this rate to compute and mint wrsETH. When the rate is stale (lower than the true rsETH/ETH exchange rate), every depositor receives more wrsETH than the deposited ETH can back on L1, creating systemic undercollateralization and protocol insolvency.

## Finding Description

**Root cause — `getRate()` has no freshness enforcement.**

`CrossChainRateReceiver` records `lastUpdated` on every `lzReceive` call but never reads it in `getRate()`: [1](#0-0) 

`getRate()` unconditionally returns the last stored value: [2](#0-1) 

**Minting math divides by the stale rate.**

`viewSwapRsETHAmountAndFee` calls `getRate()` and computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. When `rsETHToETHrate` is lower than the true rate, the quotient is inflated: [3](#0-2) 

**`deposit()` mints the inflated amount unconditionally.** [4](#0-3) 

**`limitDailyMint` does not prevent the invariant break.**

The daily cap is denominated in rsETH units computed using the same stale rate, so the cap itself is inflated. It limits the scale of over-minting within a single day but resets every 24 hours and does not prevent per-deposit undercollateralization: [5](#0-4) 

**Exploit path:**
1. LayerZero message is delayed (congestion, rate provider not calling `updateRate()`, etc.) — no privileged action required.
2. `rate` in `CrossChainRateReceiver` is now lower than the true rsETH/ETH exchange rate.
3. Any unprivileged user calls `deposit()` with ETH.
4. `viewSwapRsETHAmountAndFee` divides by the stale (low) rate, producing an inflated `rsETHAmount`.
5. `wrsETH.mint(msg.sender, rsETHAmount)` mints the excess.
6. The deposited ETH is bridged to L1, but the L1 vault cannot mint enough rsETH to redeem all outstanding wrsETH.

## Impact Explanation

**Critical — Protocol insolvency.** rsETH is a yield-bearing token whose exchange rate grows monotonically. A stale rate is always lower than the true rate. Every deposit during a staleness window mints `(trueRate / staleRate − 1) × depositAmount` excess wrsETH. The pool bridges only the deposited ETH to L1; the L1 vault cannot cover the shortfall. Accumulated across depositors and multiple daily windows, this produces a structural deficit: total wrsETH outstanding exceeds the ETH/rsETH backing on L1.

Concrete example (no fees, staleRate = 1e18, trueRate = 1.05e18):
- 1 ETH deposited → `1e18 * 1e18 / 1e18 = 1e18` wrsETH minted
- Correct amount: `1e18 * 1e18 / 1.05e18 ≈ 0.952e18` wrsETH
- Excess per deposit: ~0.048 wrsETH (~5%)
- 100 depositors × 1 ETH = 100 ETH bridged, but 100 wrsETH outstanding (worth 105 ETH at true rate)

## Likelihood Explanation

No privileged actor needs to be compromised. The condition arises from normal operational variance: LayerZero congestion, rate provider not calling `updateRate()` frequently enough, or any cross-chain messaging delay. The `lastUpdated` field already present in the contract confirms the developers anticipated tracking freshness but omitted enforcement. Any unprivileged depositor triggers the over-minting simply by calling `deposit()` during a staleness window.

## Recommendation

Add a staleness threshold check in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

Alternatively, enforce the check inside `RSETHPoolV2ExternalBridge.getRate()` before consuming the oracle value. The threshold should be calibrated to the expected update frequency of the rate provider.

## Proof of Concept

```solidity
// Foundry fork test
function testStaleOracleOverMint() public {
    // 1. Deploy mock oracle returning stale rate = 1e18 (rate has not been updated)
    MockOracle staleOracle = new MockOracle(1e18);
    // Admin sets oracle (setup step, not attacker action)
    vm.prank(admin);
    pool.setRSETHOracle(address(staleOracle));

    // 2. True rate has grown to 1.05e18 but oracle is stale
    uint256 trueRate = 1.05e18;
    uint256 depositAmount = 1 ether;

    // 3. Unprivileged depositor calls deposit()
    uint256 balanceBefore = wrsETH.balanceOf(alice);
    vm.prank(alice);
    pool.deposit{value: depositAmount}("ref");
    uint256 minted = wrsETH.balanceOf(alice) - balanceBefore;

    // 4. Assert over-minting: minted > correct amount
    uint256 correctAmount = depositAmount * 1e18 / trueRate; // ~0.952e18
    assertGt(minted, correctAmount, "wrsETH minted exceeds ETH backing at true rate");

    // 5. Scale: repeat for N depositors, assert total wrsETH supply > total ETH deposited / trueRate
}
```
