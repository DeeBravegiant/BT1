### Title
Completed `verify_foreign_transaction` requests carry no replay guard, enabling cross-chain double-spend — (`File: crates/contract/src/pending_requests.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` flow has no "already-completed" tracking. After `respond_verify_foreign_tx` drains the pending queue, the identical `VerifyForeignTransactionRequest` can be re-submitted by any unprivileged caller, causing MPC nodes to produce a second valid threshold signature over the same foreign-chain event. A bridge contract that uses MPC attestation to release NEAR-side funds for a foreign-chain deposit can therefore be drained twice for the same deposit.

---

### Finding Description

**Root cause — no completed-request set**

`resolve_yields_for` removes the map entry on success but writes nothing to a "seen" set:

```rust
// crates/contract/src/pending_requests.rs  lines 74-87
let resumed = requests
    .remove(request)          // entry deleted; nothing recorded as "done"
    .unwrap_or_default()
    ...
``` [1](#0-0) 

`push_pending_yield`, called on every new `verify_foreign_transaction`, checks only the fan-out cap — it never consults a completed-request set because none exists:

```rust
// crates/contract/src/pending_requests.rs  lines 43-59
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) ...
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) { ... }
    queue.push(YieldIndex { data_id });
}
``` [2](#0-1) 

**Root cause — caller-agnostic request key**

`VerifyForeignTransactionRequest`, the map key, contains only `(request, domain_id, payload_version)`. The caller's account ID is absent:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [3](#0-2) 

This means any caller submitting the same foreign-chain tx produces the same map key. After the first `respond_verify_foreign_tx` removes the entry, any caller can re-enqueue the identical request:

```rust
// crates/contract/src/lib.rs  lines 549-556
let request = args_into_verify_foreign_tx_request(request);
let callback_args = serde_json::to_vec(&(&request,)).unwrap();
self.enqueue_yield_request(
    method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
    callback_args,
    callback_gas,
    move |this, id| this.add_verify_foreign_tx_request(request, id),
);
``` [4](#0-3) 

**Node-level confirmation**

Each re-submission creates a new NEAR receipt, so the node-side `VerifyForeignTxRequest` receives a fresh `id` and `receipt_id`. The nodes have no "already signed this foreign-chain tx" guard either, so they process each re-submission as an independent new job:

```rust
// crates/node/src/types.rs  lines 157-168
pub struct VerifyForeignTxRequest {
    pub id: VerifyForeignTxId,      // unique per receipt
    pub receipt_id: CryptoHash,
    pub request: dtos::ForeignChainRpcRequest,
    ...
}
``` [5](#0-4) 

**Analogy to H-2**

| Optimism H-2 | NEAR MPC analog |
|---|---|
| `successfulMessages` wiped during migration → replay check always passes | No completed-request set ever exists → replay check is absent entirely |
| Legacy withdrawal re-relayed after migration | Foreign-chain tx re-signed after first completion |
| Double-spend of bridged ETH/ERC20 | Double-spend of NEAR-side bridged assets |

---

### Impact Explanation

A bridge contract (e.g., Omnibridge inbound) that calls `verify_foreign_transaction` to obtain an MPC attestation before releasing NEAR-side funds will receive a second valid threshold signature for the same foreign-chain deposit if the request is re-submitted. Any bridge that checks only signature validity — not whether the foreign-chain tx was already attested — will release funds twice. This is a direct, unprivileged double-spend of bridged assets controlled by the MPC network's signing capability.

Impact category: **High — Cross-chain replay causing invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

- Requires no special role, no collusion, no leaked key.
- Cost: 1 yoctoNEAR deposit + gas per re-submission.
- The attacker only needs to observe the NEAR chain for a successful `respond_verify_foreign_tx` receipt and then immediately re-submit the identical `verify_foreign_transaction` arguments.
- The contract accepts the re-submission unconditionally.

---

### Recommendation

1. **Add a completed-request set** to `MpcContract`:
   ```rust
   completed_verify_foreign_tx_requests: LookupSet<VerifyForeignTransactionRequest>,
   ```
2. **Guard `verify_foreign_transaction`**: reject any request whose key is already in the completed set.
3. **Populate the set in `respond_verify_foreign_tx`** (or in `return_verify_foreign_tx_and_clean_state_on_success` on the success branch) before removing the pending entry.
4. Alternatively, include a caller-supplied nonce or the NEAR block height in `VerifyForeignTransactionRequest` so that each submission is inherently unique, and enforce uniqueness at the contract level.

---

### Proof of Concept

```
1. Alice calls verify_foreign_transaction({Bitcoin, tx_id=0xABC, ...}, domain_id=0, V1)
   → pending_verify_foreign_tx_requests[key] = [yield_1]

2. MPC nodes verify Bitcoin tx 0xABC, call respond_verify_foreign_tx(key, sig_1)
   → resolve_yields_for removes key from map; yield_1 resumed with sig_1
   → Bridge contract receives sig_1, releases 1 BTC worth of NEAR tokens to Alice

3. Attacker calls verify_foreign_transaction({Bitcoin, tx_id=0xABC, ...}, domain_id=0, V1)
   → push_pending_yield: no completed-set check; key re-inserted
   → pending_verify_foreign_tx_requests[key] = [yield_2]

4. MPC nodes see new pending request (new receipt_id → new VerifyForeignTxRequest.id),
   re-verify Bitcoin tx 0xABC (still finalized), call respond_verify_foreign_tx(key, sig_2)
   → yield_2 resumed with sig_2

5. Attacker presents sig_2 to bridge contract
   → Bridge contract verifies sig_2 (valid threshold signature), releases another
     1 BTC worth of NEAR tokens — double-spend achieved.
``` [6](#0-5) [7](#0-6)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
