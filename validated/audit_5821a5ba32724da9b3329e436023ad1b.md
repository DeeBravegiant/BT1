Audit Report

## Title
ETH Deposit Limit Check Omits Incoming `msg.value`, Allowing Cap to Be Exceeded - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison for ETH: it tests only whether the pre-deposit total already exceeds the limit, without adding the incoming `amount`. The ERC-20 branch correctly adds `amount` before comparing. Any depositor can push ETH TVL past the configured cap whenever the running total is at or near the limit.

## Finding Description
`depositETH` passes `msg.value` as `depositAmount` to `_beforeDeposit`, which forwards it to `_checkIfDepositAmountExceedesCurrentLimit`: [1](#0-0) [2](#0-1) 

Inside the check, the ETH branch drops `amount` entirely:

```solidity
// L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← ERC-20: correct
``` [3](#0-2) 

When `totalAssetDeposits == depositLimit`, the ETH check evaluates `limit > limit` → `false`, so no revert is triggered and the deposit proceeds, minting rsETH and incrementing the total beyond the cap. No existing guard compensates for this omission.

## Impact Explanation
The ETH deposit cap is a protocol-level risk-management invariant. Bypassing it allows rsETH to be minted beyond the intended ceiling. No funds are directly stolen or frozen; the contract simply fails to enforce the promised deposit limit. This matches the allowed impact: **Low — contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The condition is trivially reachable by any unprivileged depositor. No special role, timing window, or front-running is required. Once `totalAssetDeposits(ETH) >= depositLimit`, every subsequent `depositETH` call bypasses the cap. The condition is repeatable indefinitely.

## Recommendation
Include the incoming ETH amount in the cap check, consistent with the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. Protocol accumulates `totalAssetDeposits(ETH) = 1000 ether` (exactly at cap).
3. Attacker calls `depositETH{value: 100 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 100 ether)` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 100 ETH; `totalAssetDeposits(ETH)` becomes 1100 ether — 10% above the configured cap.

**Foundry test sketch:**
```solidity
function test_ethCapBypass() public {
    // set cap to 1000 ether, fill to cap via legitimate deposits
    // then call depositETH{value: 1 ether} and assert totalAssetDeposits > cap
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), lrtConfig.depositLimitByAsset(ETH_TOKEN));
}
```
