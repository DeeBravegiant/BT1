### Title
Unauthenticated Future-Proposal Cache Poisoning Allows Any Peer to Pre-empt the Legitimate Proposer for Upcoming Heights — (`crates/apollo_consensus/src/manager.rs`)

---

### Summary

`MultiHeightManager::handle_proposal` checks that `init.proposer` equals the committee-derived expected proposer, but this check compares a **self-reported field in the wire message** against the committee lookup — it does not verify that the actual network sender is the proposer. Any peer can therefore craft a `ProposalInit` with `proposer` set to the legitimate proposer's address and inject it into the future-proposal cache. Because `cache_future_proposal` uses `or_insert` (first-writer-wins), the legitimate proposer's real proposal for that `(height, round)` slot is silently discarded when it arrives later. The code itself acknowledges this in a TODO comment.

---

### Finding Description

**Root cause — `cache_future_proposal` is first-writer-wins with no sender authentication:** [1](#0-0) 

```rust
fn cache_future_proposal(
    &mut self,
    init: ProposalInit,
    content_receiver: mpsc::Receiver<ContextT::ProposalPart>,
) {
    self.future_proposals_cache
        .entry(init.height)
        .or_default()
        .entry(init.round)
        .or_insert((init, content_receiver));   // ← first writer wins; later writers silently dropped
}
```

**The only guard before caching is a field-equality check, not a sender-identity check:** [2](#0-1) 


`init.proposer` is a field inside the protobuf message; any peer can set it to any value. There is no cryptographic signature over `ProposalInit` that would bind the message to the actual sender. The committee lookup (`get_proposer_for_height`) is deterministic and public — an attacker can compute the expected proposer for any future `(height, round)` pair and forge the field.

**Cached proposals are replayed unconditionally at height start:** [3](#0-2) 

```rust
let cached_proposals = self.cache.get_current_height_proposals(height);
for (init, content_receiver) in cached_proposals {
    let new_requests =
        self.handle_proposal_known_proposal_init(height, shc, init, content_receiver).await;
    pending_requests.extend(new_requests);
}
```

Because `get_current_height_proposals` removes and returns the cached entry, the legitimate proposer's real proposal — which arrives after the fake one was cached — is rejected by `or_insert` and never stored. When the height starts, only the attacker's fake proposal is replayed.

**Height/round admission window (`should_cache_proposal`):** [4](#0-3) 

The `should_cache_msg` function limits caching to `future_height_limit` heights ahead and `future_height_round_limit` rounds. This bounds the attack window but does not prevent it — the attacker simply targets every `(height, round)` slot within the window.

---

### Impact Explanation

When the fake proposal is replayed at height start, `handle_proposal_known_proposal_init` stores the attacker's content receiver in `current_height_proposals_streams` and triggers `SMRequest::StartValidateProposal`. [5](#0-4) 

The orchestrator then calls `validate_proposal`, which calls `is_proposal_init_valid` and `initiate_validation` (batcher). The fake content stream either delivers no `ProposalFin` (channel closed immediately) or delivers a `ProposalFin` whose `proposal_commitment` mismatches the batcher's computed hash, causing `ValidateProposalError::ProposalFinMismatch` or `ValidationTimeout`. [6](#0-5) 

The validator therefore sends a Nil prevote for round 0. The round advances. The attacker can repeat for round 1, 2, … up to `future_height_round_limit`, stalling block production for the targeted height. Because the proposer check is purely field-based, the cost to the attacker is only the ability to send P2P messages — no stake, no fee, no privileged access.

**Matching impact category:** *High — consensus admission accepts an invalid proposal (fake `ProposalInit` with correct proposer field but invalid content) and rejects the legitimate proposal before sequencing.*

---

### Likelihood Explanation

- The proposer schedule is deterministic and publicly computable from the committee.
- Any P2P-connected peer (not necessarily a validator) can send a `ProposalStreamMessage`.
- The attack requires only that the fake proposal arrives at the victim before the legitimate proposer's proposal — straightforward with a low-latency connection or by targeting the first message of a new height.
- The `future_height_limit` window means the attacker can pre-poison multiple upcoming heights in a single burst.

---

### Recommendation

**Short term:** Before caching a future proposal, verify that the message was received from the peer whose network identity corresponds to the expected proposer (e.g., using the authenticated sender metadata already available in `BroadcastedMessageMetadata`). Alternatively, require a cryptographic signature over `ProposalInit` from the proposer's key and verify it before caching.

**Long term:** Remove the first-writer-wins `or_insert` policy. If a second proposal arrives for the same `(height, round)` from the authenticated proposer (e.g., a retransmit), it should replace the cached entry rather than being silently dropped. The TODO comment at line 872 already flags this for version 1.0.

---

### Proof of Concept

1. Attacker connects to a validator node as a P2P peer.
2. At current height `H`, attacker queries the committee to determine the expected proposer `P` for height `H+1`, round 0.
3. Attacker constructs a `ProposalInit` with `height = H+1`, `round = 0`, `proposer = P`, and sends it wrapped in a `ProposalStreamMessage` with an immediately-closed content channel (no transactions, no `ProposalFin`).
4. `handle_proposal` passes the `proposer != init.proposer` check (both equal `P`) and calls `cache_future_proposal`, storing the fake entry under `(H+1, 0)`.
5. The legitimate proposer `P` later sends its real proposal for `(H+1, 0)`. `cache_future_proposal` calls `or_insert`, which is a no-op because the slot is already occupied. The real proposal is discarded.
6. When height `H+1` starts, `get_current_height_proposals` returns the fake entry. Validation times out or returns `ProposalFinMismatch`. The validator votes Nil for round 0.
7. Repeat for rounds 1, 2, … to stall the height indefinitely. [7](#0-6) [1](#0-0) [3](#0-2)

```rust
    fn should_cache_msg(
        &self,
        current_height: &BlockNumber,
        current_round: Round,
        msg_height: BlockNumber,
        msg_round: Round,
        msg_description: &str,
    ) -> bool {
        let limits = &self.future_msg_limit;
        let height_diff = msg_height.saturating_sub(current_height.0);

        let should_cache = height_diff <= limits.future_height_limit.into()
            // For current height, check against current round + future_round_limit
            && (height_diff == 0 && msg_round <= current_round + limits.future_round_limit
                // For future heights, check absolute round limit
                || height_diff > 0 && msg_round <= limits.future_height_round_limit);

        if !should_cache {
            warn!(
                "Dropping {} for height={} round={} when current_height={} current_round={} - \
                 limits: future_height={}, future_height_round={}, future_round={}",
                msg_description,
                msg_height,
                msg_round,
                current_height,
                current_round,
                limits.future_height_limit,
                limits.future_height_round_limit,
                limits.future_round_limit
            );
        }

        should_cache
    }
```
