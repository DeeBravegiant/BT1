Audit Report

## Title
Share Inflation Attack via `rsETHPrice` Manipulation on First Deposit - (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary

When `rsethSupply` is 1 wei, an attacker can directly donate LST tokens to `LRTDepositPool` and call the public, permissionless `updateRSETHPrice()` to inflate `rsETHPrice` to an arbitrarily large value. Because `pricePercentageLimit` is never set in `initialize()` and defaults to `0`, the price-increase guard is permanently disabled for unprivileged callers. A subsequent depositor who passes `minRSETHAmountExpected = 0` receives 0 rsETH while their assets are absorbed into the TVL, which the attacker's 1-wei rsETH position fully represents.

## Finding Description

**Root cause 1 — `updateRSETHPrice()` is public with no access control:** [1](#0-0) 

Any unprivileged caller can trigger a price update at any time.

**Root cause 2 — `pricePercentageLimit` defaults to `0`, permanently disabling the price-increase guard:**

`initialize()` never sets `pricePercentageLimit`. [2](#0-1) 

The guard condition short-circuits on `pricePercentageLimit > 0`, so when it is `0` (the default), `isPriceIncreaseOffLimit` is always `false` and no revert ever fires for an unprivileged caller: [3](#0-2) 

**Root cause 3 — TVL reads live `balanceOf`, making it inflatable by direct donation:**

`getAssetDistributionData` uses `IERC20(asset).balanceOf(address(this))` for the deposit-pool component: [4](#0-3) 

This value is summed by `_getTotalEthInProtocol()` via `getTotalAssetDeposits`: [5](#0-4) 

**Root cause 4 — Minting uses the stored (manipulable) `rsETHPrice`:** [6](#0-5) 

**Root cause 5 — `_beforeDeposit` does not enforce a non-zero rsETH output:**

The slippage check `rsethAmountToMint < minRSETHAmountExpected` passes when both sides are `0`: [7](#0-6) 

**Exploit flow:**

1. Attacker deposits 1 wei stETH → receives 1 wei rsETH; `rsethSupply = 1`.
2. Attacker directly transfers `D = 1,000,000e18` stETH to `LRTDepositPool` (no protocol function needed).
3. Attacker calls `updateRSETHPrice()`. `totalETHInProtocol ≈ D + 1`. `newRsETHPrice = (D + 1) * 1e18 / 1 = (D + 1) * 1e18`. Because `pricePercentageLimit == 0`, `isPriceIncreaseOffLimit = false`; price is accepted and stored.
4. Victim calls `depositAsset(stETH, V=1,000,000e18, minRSETHAmountExpected=0, ...)`. `rsethAmountToMint = (V * 1e18) / ((D+1) * 1e18) = V / (D+1) = 1e24 / (1e24 + 1) = 0` (integer division). Victim receives 0 rsETH; their `V` stETH is absorbed into TVL.
5. Attacker calls `updateRSETHPrice()` again. `totalETHInProtocol ≈ D + V + 1`. `newRsETHPrice ≈ (D + V + 1) * 1e18`. Attacker's 1 wei rsETH now represents 100% of supply.
6. Attacker initiates withdrawal. `getExpectedAssetAmount` computes `1 * rsETHPrice / assetPrice ≈ D + V + 1 ≈ 2,000,000e18` stETH. Attacker extracts the victim's `V` stETH as profit. [8](#0-7) 

## Impact Explanation

**Critical — Direct theft of any user funds.**

The victim deposits a large LST amount and receives 0 rsETH. Their assets are permanently absorbed into the protocol TVL. The attacker's 1-wei rsETH position represents 100% of the rsETH supply and therefore 100% of the TVL (including the victim's deposit). The attacker redeems at the inflated price and extracts the victim's full deposit. The attack is repeatable against every subsequent depositor who passes `minRSETHAmountExpected = 0`.

## Likelihood Explanation

**Medium-to-High.** The attack requires only: (1) being the first depositor or acting when `rsethSupply` is near zero; (2) holding enough LST to donate (donation `D ≥ V` forces `V / (D+1) = 0`); (3) calling the public `updateRSETHPrice()`. The attack is front-runnable: the attacker observes the victim's pending deposit transaction in the mempool, front-runs with steps 1–3, and back-runs with the withdrawal. The only natural mitigation (`pricePercentageLimit`) is unset by default and requires an admin action to activate.

## Recommendation

1. **Set `pricePercentageLimit` to a non-zero value at initialization** (e.g., `1e16` for 1%) so that a sudden price spike from a donation triggers `PriceAboveDailyThreshold` for non-manager callers.
2. **Seed the protocol with a non-trivial initial rsETH mint to `address(0)`** on first deployment so `rsethSupply` is never 1 wei.
3. **Snapshot TVL at deposit time** rather than reading live `balanceOf`, or use a time-weighted price to prevent single-block manipulation.
4. **Enforce a non-zero minimum rsETH output** in `_beforeDeposit` regardless of the caller-supplied `minRSETHAmountExpected` (e.g., revert if `rsethAmountToMint == 0`).

## Proof of Concept

```
Initial state:
  rsETHPrice = 1e18 (set by updateRSETHPrice() when rsethSupply == 0)
  rsethSupply = 0

Step 1 — Attacker deposits 1 wei stETH:
  rsethAmountToMint = (1 * 1e18) / 1e18 = 1 wei rsETH
  rsethSupply = 1

Step 2 — Attacker directly transfers D = 1,000,000e18 stETH to LRTDepositPool:
  IERC20(stETH).balanceOf(depositPool) = D + 1

Step 3 — Attacker calls updateRSETHPrice() (public, no access control):
  totalETHInProtocol = (D + 1) * 1e18 / 1e18 = D + 1          [mulWad]
  newRsETHPrice      = (D + 1) * 1e18 / 1     = (D+1)*1e18    [divWad]
  pricePercentageLimit == 0 → isPriceIncreaseOffLimit = false → no revert
  rsETHPrice = (1,000,000e18 + 1) * 1e18

Step 4 — Victim deposits V = 1,000,000e18 stETH (minRSETHAmountExpected = 0):
  rsethAmountToMint = (V * 1e18) / ((D+1) * 1e18)
                    = V / (D+1)
                    = 1,000,000e18 / (1,000,000e18 + 1)
                    = 0  ← integer division rounds down
  Victim receives 0 rsETH. Funds absorbed into TVL.

Step 5 — Attacker calls updateRSETHPrice() again:
  totalETHInProtocol ≈ D + V + 1 ≈ 2,000,000e18
  rsETHPrice ≈ 2,000,000e18 * 1e18

Step 6 — Attacker redeems 1 wei rsETH via LRTWithdrawalManager:
  underlyingToReceive = 1 * rsETHPrice / assetPrice
                      ≈ 2,000,000e18 stETH
  Attacker profit ≈ 1,000,000e18 stETH (victim's deposit)
```

**Foundry test plan:** Deploy `LRTConfig`, `LRTOracle`, `LRTDepositPool`, `LRTWithdrawalManager`, and a mock stETH. Initialize with `pricePercentageLimit = 0` (default). Execute steps 1–6 above in a single test function. Assert `rsethAmountToMint == 0` for the victim and that the attacker's withdrawal amount equals approximately `D + V`.
