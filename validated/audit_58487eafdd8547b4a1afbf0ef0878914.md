Audit Report

## Title
Ineffective ETH Deposit Limit Check Allows Depositors to Exceed Protocol Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool.sol` omits the incoming `amount` from the ETH branch comparison, evaluating only `totalAssetDeposits > depositLimit` instead of `totalAssetDeposits + amount > depositLimit`. As a result, any depositor can push ETH deposits arbitrarily above the configured cap as long as the total has not already exceeded it, violating the protocol's deposit limit invariant.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)`, while the ERC20 branch correctly returns `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. Because `amount` is absent from the ETH branch, the guard returns `false` (i.e., "not exceeded") for any deposit made while `totalAssetDeposits <= depositLimit`, regardless of how large `amount` is. The check only triggers after the cap is already breached — it cannot prevent the breach itself.

This function is called from `_beforeDeposit` (L661–663), which gates every `depositETH` call (L76–93). Because `_beforeDeposit` is `private` and `view`, no other path bypasses it. The root cause is purely the missing `+ amount` in the ETH branch.

## Impact Explanation
The ETH deposit cap is not enforced. Any depositor can mint rsETH beyond the protocol's intended ceiling, violating the deposit limit invariant. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.** No funds are directly stolen; the protocol simply mints more rsETH than the cap permits.

## Likelihood Explanation
`depositETH` is a public, permissionless, payable function. No special role, flash loan, or external dependency is required. The vulnerable condition (`totalAssetDeposits <= depositLimit`) is the normal operating state of the protocol. Any depositor can trigger this on every deposit until the cap is already exceeded. Likelihood is **High**.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Deploy/fork with `depositLimitByAsset(ETH) = 1000 ether` and `getTotalAssetDeposits(ETH) = 999 ether`.
2. Any EOA calls `depositETH{value: 500 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 500 ether)` evaluates `999 ether > 1000 ether` → `false`.
4. `_beforeDeposit` does not revert; rsETH is minted for 500 ETH.
5. `getTotalAssetDeposits(ETH)` is now `1499 ether`, 49.9% above the cap.
6. The same call can be repeated by any depositor in the same block with no bound on overshoot.
