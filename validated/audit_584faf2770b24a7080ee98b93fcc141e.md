### Title
Caller-Supplied `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies the MPC threshold signature against a `payload_hash` value that is **entirely supplied by the responding node**, without checking that the hash encodes the `ForeignChainRpcRequest` stored in the pending request. A single malicious attested participant can reuse a legitimately-produced threshold signature from one foreign-chain verification request as the response to a different pending request, delivering a forged `VerifyForeignTransactionResponse` to the original caller.

### Finding Description

In `respond_verify_foreign_tx`, the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

`response.payload_hash` is taken verbatim from the caller's argument. The contract verifies only that the signature is a valid MPC signature over *some* 32-byte hash — it never verifies that the hash encodes `ForeignTxSignPayload::V1 { request: <the pending ForeignChainRpcRequest>, values: ... }`.

The canonical payload hash is defined as:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

The `request` field inside `ForeignTxSignPayloadV1` is the `ForeignChainRpcRequest` (chain, tx_id, extractors, finality). The contract stores the pending `VerifyForeignTransactionRequest` (which contains this `ForeignChainRpcRequest`) as the map key: [3](#0-2) 

But it never reconstructs or checks that `response.payload_hash` is consistent with the stored request. The `values` (extracted chain data) are not submitted to the contract, so the contract cannot independently compute the expected hash — but it also does not even verify the `ForeignChainRpcRequest` portion, which it *does* have.

**Attack path (cross-request replay):**

1. Two pending requests exist: **R1** (tx_id = T1) and **R2** (tx_id = T2).
2. The MPC protocol runs for R2. The leader and threshold-many followers each independently query the foreign chain for T2, compute `hash(R2, values2)`, and produce a valid threshold signature `sig` over it.
3. The leader (or any participant who received `sig`) calls:
   ```
   respond_verify_foreign_tx(
       request = R1,                          // the pending request for T1
       response = {
           payload_hash = hash(R2, values2),  // hash of T2's data
           signature    = sig                 // valid MPC sig over hash(R2, values2)
       }
   )
   ```
4. The contract checks: caller is attested ✓, `sig` is valid over `response.payload_hash` ✓, R1 is in `pending_verify_foreign_tx_requests` ✓ — and resolves all yields for R1 with this response.
5. The caller of `verify_foreign_transaction(R1)` receives `VerifyForeignTransactionResponse { payload_hash: hash(R2, values2), signature: sig }` — a response whose hash does not correspond to T1.

No threshold collusion is required. The attacker reuses a legitimately-produced threshold signature; they need only be a single attested participant who observed the completed signature for R2.

### Impact Explanation

Any NEAR smart contract that calls `verify_foreign_transaction` and then acts on the response (e.g., an omnibridge releasing funds after verifying an inbound foreign-chain deposit) may receive a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to the transaction it requested. If the caller does not independently recompute the expected hash from its own copy of the request and extracted values and compare it to `response.payload_hash`, it will accept the forged attestation. This enables invalid bridge execution: funds could be released for a transaction that was never actually verified, or verified data (block hash, log, program ID) from a different transaction could be injected into on-chain logic.

The `near-mpc-sdk`'s `ForeignChainSignatureVerifier::verify_signature` does perform the payload-hash check: [4](#0-3) 

However, the contract itself — the trust anchor — does not enforce this binding, leaving callers that implement their own verification or omit the hash check fully exposed.

### Likelihood Explanation

- Requires a single malicious attested participant (below signing threshold) — no threshold collusion.
- The attacker needs only to have observed a completed threshold signature for any other pending request, which is a normal outcome of participating in the MPC protocol.
- Two concurrent pending requests for different transactions are a routine production condition.
- The attack is silent: the contract emits no error, and the forged response is indistinguishable from a legitimate one at the contract level.

### Recommendation

In `respond_verify_foreign_tx`, verify that `response.payload_hash` is consistent with the `ForeignChainRpcRequest` stored in the pending request. Since the contract does not know the extracted `values`, the node should submit them alongside the response so the contract can recompute and verify the hash:

```rust
// Proposed: node submits values; contract verifies binding
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),  // from pending map
    values: response.values.clone(),   // newly submitted
});
let expected_hash = expected_payload.compute_msg_hash()?;
require!(expected_hash == response.payload_hash, "payload_hash mismatch");
```

Alternatively, at minimum, the contract should verify that the `ForeignChainRpcRequest` encoded inside `response.payload_hash` matches the pending request — though this requires the node to submit the pre-image or a commitment scheme.

### Proof of Concept

**Setup:** Two pending requests, R1 (Bitcoin tx_id = `[0x01; 32]`) and R2 (Bitcoin tx_id = `[0x02; 32]`), both in `pending_verify_foreign_tx_requests`.

**Step 1:** MPC protocol completes for R2. Leader obtains `sig` = valid threshold signature over `hash(R2, values2)`.

**Step 2:** Malicious leader calls (pseudocode):
```rust
contract.respond_verify_foreign_tx(
    request = VerifyForeignTransactionRequest { request: BitcoinRpcRequest { tx_id: [0x01;32], ... }, ... },
    response = VerifyForeignTransactionResponse {
        payload_hash: hash(ForeignTxSignPayload::V1 { request: BitcoinRpcRequest { tx_id: [0x02;32], ... }, values: values2 }),
        signature: sig,  // valid MPC sig over hash(R2, values2)
    }
)
```

**Step 3:** Contract accepts (signature valid over submitted hash, R1 is pending). Caller of `verify_foreign_transaction(R1)` receives the forged response.

The root cause is at: [5](#0-4) 

where `response.payload_hash` is used directly as the message to verify without any binding check against the stored `VerifyForeignTransactionRequest`.
