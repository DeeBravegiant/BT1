Audit Report

## Title
rsETH Price Inflation via Unrestricted ETH Donation to LRTDepositPool â€” Direct Theft of Depositor Funds - (File: contracts/LRTDepositPool.sol / contracts/LRTOracle.sol)

## Summary
`LRTDepositPool` exposes an unrestricted `receive()` fallback that allows any caller to inflate `address(this).balance`. Because `getETHDistributionData()` uses the raw contract balance as the canonical ETH accounting figure, a subsequent call to the public `updateRSETHPrice()` permanently inflates `rsETHPrice`. With `pricePercentageLimit` defaulting to `0` at deployment, the price-increase guard is unconditionally bypassed, and depositors who call `depositETH` with `minRSETHAmountExpected = 0` receive zero rsETH tokens for their ETH, which is captured by the attacker on redemption.

## Finding Description

**Root cause â€” raw balance used as canonical ETH accounting:**

`getETHDistributionData()` reads `address(this).balance` directly:

```solidity
// LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

Any ETH sent via the open fallback is indistinguishable from legitimately deposited ETH.

**Open ETH receiver:**

```solidity
// LRTDepositPool.sol L58
receive() external payable { }
```

No access control, no accounting update.

**Price derivation from inflated balance:**

`_getTotalEthInProtocol()` (L331â€“349) calls `getTotalAssetDeposits(ETH_TOKEN)` â†’ `getAssetDistributionData()` â†’ `getETHDistributionData()`, summing the inflated balance. `_updateRsETHPrice()` then computes:

```solidity
// LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

With `rsethSupply = 1 wei` and `totalETHInProtocol â‰ˆ 1000e18`, `newRsETHPrice â‰ˆ 1000e36` (divWad multiplies by 1e18).

**Price-increase guard is disabled by default:**

`initialize()` never sets `pricePercentageLimit`, so it defaults to `0`. The guard short-circuits unconditionally:

```solidity
// LRTOracle.sol L256-257
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```

`isPriceIncreaseOffLimit` is always `false`, so `updateRSETHPrice()` (public, L87) accepts any price increase without restriction.

**Minting uses the stored inflated price:**

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

With `rsETHPrice â‰ˆ 1000e36`, a victim depositing 999 ETH computes `rsethAmountToMint = 999e36 / 1000e36 = 0`.

**No protocol-level guard against zero-rsETH mints:**

`_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`:

```solidity
// LRTDepositPool.sol L667-669
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

When `minRSETHAmountExpected = 0` (the default for callers who omit slippage protection), `0 < 0` is false â€” no revert. The victim's ETH is accepted, 0 rsETH is minted, and the ETH is permanently absorbed into the pool controlled by the attacker's single rsETH token.

## Impact Explanation

**Critical â€” Direct theft of user funds.**

A victim who calls `depositETH{value: 999 ether}(0, "")` after the price manipulation receives 0 rsETH. Their 999 ETH is absorbed into the pool. The attacker, holding the only 1 wei of rsETH, redeems it and recovers the entire pool balance (~1999 ETH), netting ~999 ETH profit. The protocol provides no contract-level guard against minting zero shares for a non-zero ETH deposit; the only protection (`minRSETHAmountExpected`) is opt-in and defaults to 0 in direct contract calls and many frontend integrations.

## Likelihood Explanation

The attack requires no privileged access. The attacker needs only:
1. A small initial deposit to obtain 1 wei rsETH (public `depositETH`).
2. A direct ETH transfer to `LRTDepositPool` via `receive()` (no access control).
3. A call to the public `updateRSETHPrice()`.
4. A victim who calls `depositETH` with `minRSETHAmountExpected = 0`.

Steps 1â€“3 are fully attacker-controlled. Step 4 is realistic: `minRSETHAmountExpected = 0` is the natural default for direct contract callers, scripts, and many frontend integrations that do not compute expected output before submission. The attack is repeatable and the withdrawal queue delay does not prevent the harm to victims.

## Recommendation

1. **Replace `address(this).balance` with internal accounting.** Introduce a `totalDepositedETH` storage variable incremented only through authorised entry points (`depositETH`, `receiveFromRewardReceiver`, `receiveFromNodeDelegator`, `receiveFromLRTConverter`) and use it in `getETHDistributionData()` instead of `address(this).balance`.

2. **Set a non-zero `pricePercentageLimit` at initialisation.** This caps per-update price increases and forces large manipulations to span multiple cycles, giving operators time to detect and respond.

3. **Enforce `rsethAmountToMint > 0` unconditionally in `_beforeDeposit`.** Add `require(rsethAmountToMint > 0, "ZeroSharesMinted")` regardless of the caller-supplied `minRSETHAmountExpected`, eliminating the zero-share deposit path entirely.

4. **Seed the protocol with an initial deposit at deployment** so that the ratio of donated ETH to existing TVL is never large enough to cause meaningful rounding loss.
