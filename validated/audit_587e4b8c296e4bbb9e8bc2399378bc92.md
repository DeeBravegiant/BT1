### Title
Future-Height Proposal Cache Poisoning via Unauthenticated `ProposalInit.proposer` — (`crates/apollo_consensus/src/manager.rs`)

### Summary

`ConsensusCache::cache_future_proposal` stores the **first** proposal received for any `(height, round)` pair using `.or_insert()`. Because `ProposalInit` carries no cryptographic signature from the proposer, any network peer can craft a message with `proposer` set to the legitimate proposer's address and inject it before the real proposer, permanently occupying the cache slot. The legitimate proposal is then silently discarded, causing consensus to fail for that height/round. The code itself acknowledges this attack vector with an open TODO.

### Finding Description

`ConsensusCache::cache_future_proposal` in `crates/apollo_consensus/src/manager.rs`:

```rust
fn cache_future_proposal(
    &mut self,
    init: ProposalInit,
    content_receiver: mpsc::Receiver<ContextT::ProposalPart>,
) {
    self.future_proposals_cache
        .entry(init.height)
        .or_default()
        .entry(init.round)
        .or_insert((init, content_receiver)); // first-writer wins; all later proposals silently dropped
}
``` [1](#0-0) 

The caller in `handle_proposal` checks that `init.proposer` matches the committee-derived expected proposer for `(height, round)`:

```rust
if proposer != init.proposer {
    warn!("Invalid proposer for height {} and round {}: expected {:?}, got {:?}", ...);
    return Ok(VecDeque::new());
}
``` [2](#0-1) 

However, `ProposalInit` contains **no cryptographic signature** from the proposer — `proposer` is a plain `ContractAddress` field that any peer can set to any value: [3](#0-2) 

The code explicitly acknowledges the resulting attack:

```rust
// Note: new proposals with the same height/round will be ignored.
//
// TODO(matan): This only work for trusted peers. In the case of
// possibly malicious peers this is a
// possible DoS attack (malicious
// users can insert invalid/bad/malicious proposals before
// "good" nodes can propose).
//
// When moving to version 1.0 make sure this is addressed.
self.cache.cache_future_proposal(init, content_receiver);
``` [4](#0-3) 

The `StarkAuthentication` mechanism (`sign_identification` / `verify_identity`) covers peer-identity handshakes, not proposal-content signing. No path in `handle_proposal` verifies that the network sender's key corresponds to the `init.proposer` address before caching. [5](#0-4) 

### Impact Explanation

A malicious peer that is part of the P2P network can:

1. Observe the committee to learn the legitimate proposer for height `H+1`, round `0`.
2. Craft a `ProposalInit` with `proposer = <legitimate_proposer_address>`, `height = H+1`, `round = 0`, and a content stream that will fail validation (e.g., wrong gas prices, invalid transactions, or an immediately-closed channel).
3. Send this stream to victim nodes before the legitimate proposer does.
4. The proposer-address check passes; `cache_future_proposal` stores the spoofed entry via `.or_insert()`.
5. When the legitimate proposer sends its valid proposal for the same `(H+1, 0)`, `or_insert` is a no-op — the slot is already occupied and the valid proposal is silently discarded.
6. When height `H+1` is reached, `get_current_height_proposals` returns the spoofed entry; it fails validation, forcing a round advance.
7. The attacker repeats for round `1`, `2`, … causing indefinite consensus liveness failure and preventing any block from being finalized.

This is the direct sequencer analog of M-03: just as an attacker stakes dust on behalf of a receiver to lock in an unintended `lockPeriod`, a malicious peer injects a spoofed `ProposalInit` on behalf of the legitimate proposer to lock in an invalid proposal for every future `(height, round)`.

### Likelihood Explanation

**Medium.** The attacker must be a connected P2P peer, but the network is permissionless for validators. The attack is cheap (a single crafted message per round), requires no stake or special privilege, and can be repeated indefinitely across all future heights and rounds.

### Recommendation

**Option 1 (preferred):** Require the proposer to sign the `ProposalInit` with its consensus key. Verify the signature before calling `cache_future_proposal`. Reject unsigned or incorrectly-signed proposals.

**Option 2:** Bind the proposal to the network sender's authenticated peer identity. Verify that the sender's libp2p `PeerId` maps to the expected proposer's Starknet address before caching.

**Option 3 (partial mitigation):** Change `or_insert` to an unconditional `insert` so the legitimate proposer can overwrite a spoofed entry. This alone does not prevent the attack if the attacker can keep re-injecting, but combined with peer-reputation/banning it reduces the window.
