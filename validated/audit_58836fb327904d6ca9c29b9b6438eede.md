Audit Report

## Title
Missing Staleness and Validity Checks on Chainlink `latestRoundData()` Return Values - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, with no staleness, incomplete-round, or non-negative price validation. A stale or zero price propagates into `LRTOracle._updateRsETHPrice()`, which is callable by any unprivileged address via the public `updateRSETHPrice()` function. A sufficiently stale low price can trigger the downside-protection logic, pausing `LRTDepositPool` and `LRTWithdrawalManager` and temporarily freezing all user funds.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at line 52 discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The missing checks are:
- `answeredInRound >= roundId` — stale round detection
- `updatedAt != 0` — incomplete round detection
- `updatedAt >= block.timestamp - MAX_DELAY` — heartbeat miss detection
- `price > 0` — zero/negative answer detection

This price is consumed by `LRTOracle._getTotalEthInProtocol()` (lines 339–343), which multiplies each asset's total deposits by the oracle price to compute `totalETHInProtocol`. That value feeds directly into `_updateRsETHPrice()` (line 250), which computes `newRsETHPrice`.

The downside-protection logic at lines 270–281 checks whether `newRsETHPrice` has fallen below `highestRsethPrice` by more than `pricePercentageLimit`. If so, it calls `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()`, freezing all deposits and withdrawals.

`updateRSETHPrice()` at line 87 is `public` with only a `whenNotPaused` guard — any unprivileged address can call it at any time.

The contrast with `ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 30–32) is direct: that contract applies all three checks (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`), confirming the team is aware of the pattern and chose not to apply it in `ChainlinkPriceOracle`.

A secondary instance exists in `RSETHPriceFeed.latestRoundData()` (lines 68–69), which forwards the raw ETH/USD answer to external consumers without any validation.

## Impact Explanation
**Medium — Temporary freezing of funds.**

If a supported LST's Chainlink feed returns a stale price that is sufficiently below the current market price (e.g., a missed heartbeat during a price spike), `totalETHInProtocol` is understated, `newRsETHPrice` drops below `highestRsethPrice` by more than `pricePercentageLimit`, and the protocol automatically pauses `LRTDepositPool` and `LRTWithdrawalManager`. All user deposits and withdrawals are frozen until an admin manually unpauses. This is a concrete, in-scope impact.

A secondary impact exists: a stale high price inflates `rsETHPrice`, causing depositors to receive more rsETH than their assets are worth, diluting existing holders.

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for ETH/stETH). During network congestion, oracle operator issues, or L2 sequencer downtime, feeds can go stale while still returning the last cached answer with an outdated `updatedAt`. This is a historically observed failure mode, not a theoretical one. Because `updateRSETHPrice()` is public and callable by any address, an attacker can monitor for a stale price and trigger the update at the exact moment it is live, reliably inducing the pause. The attack requires no special privileges and is repeatable.

## Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(answeredInRound >= roundId, "Stale price");
require(updatedAt != 0, "Incomplete round");
require(price > 0, "Invalid price");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
```

Apply the same pattern inside `RSETHPriceFeed.latestRoundData()` before forwarding the ETH/USD answer to external consumers. Define `MAX_STALENESS` per feed based on its documented heartbeat interval.

## Proof of Concept
1. A supported LST (e.g., stETH) has its Chainlink feed go stale — `updatedAt` is 2+ hours old and `answeredInRound < roundId`, meaning the answer is carried from a prior round at a price 5% below current market.
2. Any address calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale low price without reverting.
4. `totalETHInProtocol` is understated by ~5% of the stETH TVL.
5. `newRsETHPrice` falls below `highestRsethPrice` by more than `pricePercentageLimit`.
6. `_updateRsETHPrice()` calls `lrtDepositPool.pause()` and `withdrawalManager.pause()`, freezing all user deposits and withdrawals until an admin manually unpauses.

**Foundry fork test plan:** Fork mainnet, set `block.timestamp` to `updatedAt + heartbeat + 1` for the stETH/ETH feed while keeping the feed's stored answer unchanged, call `LRTOracle.updateRSETHPrice()` from an unprivileged EOA, and assert that `lrtDepositPool.paused()` and `withdrawalManager.paused()` return `true`.
