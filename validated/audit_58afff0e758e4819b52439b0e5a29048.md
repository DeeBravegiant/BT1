### Title
`remove_last_txs()` Does Not Purge L1 Handler Transactions from `consumed_l1_handler_tx_hashes` When Validator Truncates Execution Data — (`crates/apollo_batcher/src/block_builder.rs`)

---

### Summary

In the validator path of `BlockBuilder::build_block_inner`, when the proposer's `ProposalFin.executed_transaction_count` is less than the number of transactions the validator has already executed, `remove_last_txs` is called to strip the surplus execution records. However, `remove_last_txs` omits `consumed_l1_handler_tx_hashes` from its cleanup. Any L1 handler transaction that falls in the truncated suffix remains in `consumed_l1_handler_tx_hashes` and is subsequently reported to the L1 events provider as consumed at `decision_reached`, permanently suppressing those L1→L2 messages from ever being re-queued.

---

### Finding Description

**Step 1 – L1 handler transactions are inserted unconditionally on execution.**

In `collect_execution_results_and_stream_txs`, every L1 handler transaction is inserted into `consumed_l1_handler_tx_hashes` the moment its execution result arrives, regardless of whether it will ultimately be included in the final block: [1](#0-0) 

**Step 2 – Validator truncation path calls `remove_last_txs` on the surplus transactions.**

After the loop in `build_block_inner`, when `final_n_executed_txs` (received from the proposer's `ProposalFin`) is less than `self.n_executed_txs`, the validator computes the surplus hashes and calls `remove_last_txs`: [2](#0-1) 

**Step 3 – `remove_last_txs` does not touch `consumed_l1_handler_tx_hashes`.**

The function removes surplus entries from `execution_infos_and_signatures`, `rejected_tx_hashes`, and `proof_facts_block_numbers`, but the `consumed_l1_handler_tx_hashes` field is never cleaned: [3](#0-2) 

**Step 4 – The stale hashes flow into `decision_reached` and are committed to the L1 provider.**

`decision_reached` passes `block_execution_artifacts.execution_data.consumed_l1_handler_tx_hashes` directly to `commit_proposal_and_block`, which forwards them to the L1 events provider as definitively consumed: [4](#0-3) [5](#0-4) 

**Concrete trigger scenario (no malicious actor required):**

1. Proposer streams transactions `[T1, T2, T3, T4_L1H, T5_L1H]` to validators via `output_content_sender` before knowing the block will close early.
2. Proposer's executor hits the deadline or bouncer limit after committing only `T1–T3`; it sends `ProposalFin` with `executed_transaction_count = 3`.
3. Validator has already executed all five transactions; `T4_L1H` and `T5_L1H` are in `consumed_l1_handler_tx_hashes`.
4. `remove_last_txs([T4_L1H, T5_L1H])` removes them from `execution_infos_and_signatures` / `rejected_tx_hashes` but leaves them in `consumed_l1_handler_tx_hashes`.
5. At `decision_reached`, the validator's batcher reports `T4_L1H` and `T5_L1H` as consumed; the L1 provider marks them committed and never re-queues them.

---

### Impact Explanation

**Critical – Wrong L1 message / wrong state from execution logic.**

L1 handler transactions that were never included in the committed block are permanently marked as consumed by the L1 events provider. The corresponding L1→L2 messages are silently dropped: they will not appear in any block's receipt, will not update any contract storage, and cannot be replayed. This constitutes an authoritative wrong state (incorrect consumed-transaction set) and a wrong L1 message result, matching the critical impact category.

---

### Likelihood Explanation

The trigger is a normal operational condition: the proposer streams transactions to validators before the block closes, and the block closes before all streamed transactions are executed. This happens whenever the proposer hits a deadline or bouncer limit mid-stream. L1 handler transactions are included in the stream whenever there are pending L1 events. No adversarial action is required; the bug fires on any validator node that executes surplus L1 handler transactions in a round where the proposer truncates the final count.

---

### Recommendation

Add `consumed_l1_handler_tx_hashes` to the cleanup loop inside `remove_last_txs`:

```rust
fn remove_last_txs(&mut self, tx_hashes: &[TransactionHash]) {
    for tx_hash in tx_hashes.iter().rev() {
        remove_last_map(&mut self.execution_infos_and_signatures, tx_hash);
        remove_last_set(&mut self.rejected_tx_hashes, tx_hash);
        remove_last_map(&mut self.proof_facts_block_numbers, tx_hash);
        // FIX: also remove from consumed_l1_handler_tx_hashes
        remove_last_set(&mut self.consumed_l1_handler_tx_hashes, tx_hash);
    }
}
```

Additionally, add a test that places an L1 handler transaction in the truncated suffix and asserts it is absent from `consumed_l1_handler_tx_hashes` after `remove_last_txs`.
