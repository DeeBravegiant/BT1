### Title
`get_proposer()` Weighted Selection Uses Full Committee Weight Including Ineligible Proposers, Causing Wrong Proposer Binding - (`crates/apollo_staking/src/staking_manager.rs`)

### Summary

When `use_only_actual_proposer_selection = false` (the production default), `get_proposer()` generates a random value bounded by `self.total_weight`, which is the sum of **all** committee members' weights — including those with `can_propose = false`. The `choose_proposer()` function then selects from all committee members using those cumulative weights. Ineligible proposers occupy a weight-proportional slice of the selection range and can be returned as the authoritative proposer for a height/round, binding the wrong signer to the consensus proposal slot.

### Finding Description

In `fetch_and_build_committee()`, `cumulative_weights` and `total_weight` are built from the full `committee_members` set without filtering for eligibility:

```rust
let cumulative_weights: Vec<u128> = committee_members
    .iter()
    .scan(0, |acc, staker| {
        *acc = u128::checked_add(*acc, staker.weight.0)...
        Some(*acc)
    })
    .collect();
let total_weight = *cumulative_weights.last().unwrap_or(&0);
``` [1](#0-0) 

`eligible_proposers` is computed separately but is **only** used by `get_actual_proposer()`: [2](#0-1) 

In `get_proposer()`, when `use_only_actual_proposer_selection = false`, the random value is drawn from `[0, self.total_weight)` — the full committee weight — and `choose_proposer()` iterates over all `committee_members`:

```rust
let random_value = self.random_generator.generate(height, round, self.total_weight);
let proposer = self.choose_proposer(random_value);
``` [3](#0-2) 

`choose_proposer()` walks `cumulative_weights` which covers all members, not just eligible ones: [4](#0-3) 

The production default for `use_only_actual_proposer_selection` is `false`: [5](#0-4) [6](#0-5) 

The result returned by `get_proposer()` is consumed directly by the consensus manager as the authoritative proposer identity for the height/round: [7](#0-6) 

It is also used in `handle_finished_validation()` to bind the proposer to a validated proposal: [8](#0-7) 

### Impact Explanation

This is a direct analog of M-05: the denominator (`total_weight`) includes ineligible members, so eligible proposers receive a smaller effective selection share than their weight warrants, and ineligible proposers are selected with probability proportional to their weight.

**Concrete example:** Committee = {STAKER_A: weight 5000, `can_propose = true`; STAKER_B: weight 5000, `can_propose = false`}. `total_weight = 10000`. Random values `[0, 4999]` select STAKER_A (eligible); `[5000, 9999]` select STAKER_B (ineligible). 50% of rounds, `get_proposer()` returns an ineligible proposer. The consensus manager binds that ineligible address as the expected block proposer for the round. The ineligible node does not propose; the round times out. This matches the "wrong signer" impact: `get_proposer()` binds the wrong signer to the proposal slot.

### Likelihood Explanation

- `use_only_actual_proposer_selection = false` is the shipped default in `config_schema.json` and `StakingManagerStaticConfig::default()`.
- Any deployment where the `CommitteeConfig` marks some stakers with `can_propose = false` (as demonstrated in the test fixtures `get_actual_proposer_some_eligible` and `test_get_actual_proposer_epoch_changes`) triggers the bug on every call to `get_proposer()` where the random value falls in an ineligible staker's weight range.
- No unprivileged action is required; the bug is structural and fires deterministically given the committee composition.

### Recommendation

In `get_proposer()`, restrict both the random value range and the selection pool to eligible proposers only. Build a separate `eligible_cumulative_weights` and `eligible_total_weight` from `committee_members` filtered by `eligible_proposers`, and pass `eligible_total_weight` to the random generator and use `eligible_cumulative_weights` in `choose_proposer()`. Alternatively, when `eligible_proposers` is non-empty, always delegate `get_proposer()` to `get_actual_proposer()` regardless of the flag, since the flag's purpose is to unify the two paths, not to introduce a third broken path.

### Proof of Concept

Using the existing test infrastructure in `crates/apollo_staking/src/staking_manager_test.rs`, configure a committee where STAKER_1 (`weight=1000`, `can_propose=false`), STAKER_2 (`weight=2000`, `can_propose=true`), STAKER_3 (`weight=3000`, `can_propose=true`), STAKER_4 (`weight=4000`, `can_propose=false`) with `use_only_actual_proposer_selection = false`. `total_weight = 10000`. Random values `[0, 3999]` (40%) select STAKER_4 (ineligible) and `[9000, 9999]` (10%) select STAKER_1 (ineligible). `get_proposer()` returns an ineligible proposer 50% of the time. The consensus manager at `get_proposer_for_height()` binds that address as the virtual proposer, the round stalls, and consensus liveness degrades proportionally to the ineligible weight fraction — exactly mirroring the M-05 denominator-mismatch pattern.     // Eligibility is determined by the `can_propose` field in the corresponding ConfiguredStaker.

    fn calculate_eligible_proposers(
        &self,
        committee_members: &StakerSet,
        config_stakers: &[ConfiguredStaker],
    ) -> Vec<ContractAddress> {
        let eligible_stakers: HashSet<ContractAddress> = config_stakers
            .iter()
            .filter(|staker| staker.can_propose)
            .map(|staker| staker.address)
            .collect();

        // like in the committee.
        committee_members
            .iter()
            .filter(|staker| eligible_stakers.contains(&staker.address))
            .map(|staker| staker.address)
            .collect()
    }

impl CommitteeTrait for Committee {
    fn get_proposer(&self, height: BlockNumber, round: Round) -> CommitteeResult<ContractAddress> {
        if self.use_only_actual_proposer_selection {
            return Ok(self.get_actual_proposer(height, round));
        }

        if self.committee_members.is_empty() {
            return Err(CommitteeError::EmptyCommittee);
        }

        // Check if we can return from cache.
        if let Some(address) = self
            .proposer_cache
            .lock()
            .expect("Mutex poisoned")
            .filter(|(h, r, _)| *h == height && *r == round)
            .map(|(_, _, addr)| addr)
        {
            return Ok(address);
        }

        // Generate a pseudorandom value in the range [0, total_weight) based on the height, round,
        // and the block hash stored in the generator.
        let random_value = self.random_generator.generate(height, round, self.total_weight);

        // Select a proposer from the committee using the generated random and update the cache.
        let proposer = self.choose_proposer(random_value);
        *self.proposer_cache.lock().expect("Mutex poisoned") = Some((height, round, proposer));

        Ok(proposer)
    }

    fn choose_proposer(&self, random: u128) -> ContractAddress {
        let total_weight = self.total_weight;
        assert!(
            random < self.total_weight,
            "Invalid random value {random}: exceeds total weight limit of {total_weight}."
        );

        // Iterates over stakers and selects staker `i` if `random < cumulative_weights[i]`.
        // Each staker occupies a range of values proportional to their weight, defined as:
        //     [cumulative_weights[i - 1], cumulative_weights[i])
        // Since we iterate in order, the first staker whose cumulative weight exceeds `random`
        // is the one whose range contains it.
        for (i, cum_weight) in self.cumulative_weights.iter().enumerate() {
            if random < *cum_weight {
                return self
                    .committee_members
                    .get(i)
                    .expect(
                        "Inconsistent committee data; cumulative_weights and committee_members \
                         are not the same length.",
                    )
                    .address;
            }
        }

        // We should never reach this point.
        panic!("Inconsistent committee data; cumulative_weights inconsistent with total weight.")
    }

/// Fetches the committee for the given height and returns the proposer for that height and round.
async fn get_proposer_for_height(
    committee_provider: &Arc<dyn CommitteeProvider>,
    height: BlockNumber,
    round: Round,
) -> Result<ValidatorId, ConsensusError> {
    let committee = committee_provider
        .get_committee(height)
        .await
        .map_err(|e| ConsensusError::CommitteeError(e.to_string()))?;
    committee.get_proposer(height, round).map_err(|e| ConsensusError::CommitteeError(e.to_string()))
}
