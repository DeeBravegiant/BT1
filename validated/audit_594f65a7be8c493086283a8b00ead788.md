### Title
Receipt `max_receipt_size` Limit Bypassed via `promise_return` + `output_data_receivers` Extension — (File: `runtime/runtime/src/lib.rs`)

---

### Summary

The `max_receipt_size` limit is enforced when a receipt is first created (`ValidateReceiptMode::NewReceipt`), but the runtime has a second code path — triggered by `promise_return` — that appends `output_data_receivers` to an already-validated receipt **after** the size check has passed. This allows any unprivileged contract caller to produce receipts that exceed the protocol's `max_receipt_size` invariant and commit them to the canonical chain. The codebase explicitly acknowledges this as an open bug (issue #12606) and has added a workaround in congestion control to prevent receipts from getting permanently stuck, but the root cause is unresolved.

---

### Finding Description

**Limit check — one path only:**

`validate_receipt` in `runtime/runtime/src/verifier.rs` enforces the size cap only when `mode == ValidateReceiptMode::NewReceipt`:

```rust
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

**Bypass path — `promise_return` appends `output_data_receivers` after validation:**

When a contract calls `promise_return(receipt_index)`, the runtime modifies the already-validated receipt by extending its `output_data_receivers` field with the parent receipt's data receivers. This happens in `apply_action_receipt` in `runtime/runtime/src/lib.rs`:

```rust
if !action_receipt.output_data_receivers().is_empty() {
    if let Ok(ReturnData::ReceiptIndex(receipt_index)) = result.result {
        match result.new_receipts.get_mut(receipt_index as usize)... {
            ReceiptEnum::Action(new_action_receipt) | ... =>
                new_action_receipt.output_data_receivers
                    .extend_from_slice(&action_receipt.output_data_receivers()),
``` [2](#0-1) 

A contract can craft a receipt whose initial size equals exactly `max_receipt_size` (passing `NewReceipt` validation), then use `promise_return` to cause `output_data_receivers` to be appended, pushing the receipt over the limit. The receipt is never re-validated after this mutation.

**Incoming receipt processing skips the size check:**

When the oversized receipt arrives at the destination shard, it is validated with `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size check:

```rust
validate_receipt(
    &processing_state.apply_state.config.wasm_config.limit_config,
    receipt,
    protocol_version,
    ValidateReceiptMode::ExistingReceipt,  // size check skipped
)
``` [3](#0-2) 

The `ExistingReceipt` mode was introduced specifically to tolerate these oversized receipts:

```rust
/// 2) There is a bug which allows to create receipts that are above the size limit. Runtime has
///    to handle them gracefully until the receipt size limit bug is fixed.
///    See https://github.com/near/nearcore/issues/12606 for details.
ExistingReceipt,
``` [4](#0-3) 

**Congestion control workaround — clamps size to prevent receipts getting stuck:**

Without a workaround, an oversized receipt would exceed the outgoing size limit and never be forwarded, permanently bricking the shard. The congestion control code in `runtime/runtime/src/congestion_control.rs` clamps the reported size to `max_receipt_size` to prevent this:

```rust
// There is a bug which allows to create receipts that are above the size limit. Receipts
// above the size limit might not fit under the maximum outgoing size limit. Let's pretend
// that all receipts are at most `max_receipt_size` to avoid receipts getting stuck.
if size > max_receipt_size {
    size = max_receipt_size;
}
``` [5](#0-4) 

The workaround prevents the worst-case bricking, but the underlying invariant violation remains: oversized receipts enter the canonical chain, and congestion control accounting uses a clamped (incorrect) size, corrupting the congestion info stored in state.

**The bug is confirmed by integration tests:**

The test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` explicitly demonstrates the bypass and asserts that an oversized receipt does appear in the chain:

---

### Impact Explanation

The `max_receipt_size` protocol invariant is violated: any unprivileged contract caller can produce receipts that exceed the limit and commit them to the canonical chain. The congestion control state stored in the trie uses a clamped (incorrect) size for these receipts, meaning the `congestion_size` field in `StateStoredReceipt` metadata diverges from the actual serialized receipt size. This corrupts the congestion info component of the state root. Without the clamping workaround, the oversized receipt would be permanently stuck in the outgoing buffer, bricking the affected shard and trapping all funds routed through it — the direct analog to the Hats Protocol safe-bricking scenario.

---

### Likelihood Explanation

Any account that can deploy a contract (i.e., any NEAR account with sufficient balance) can trigger this. The attacker deploys a contract that:
1. Creates a receipt whose initial size equals exactly `max_receipt_size` (e.g., via large `FunctionCall` args).
2. Chains it with `promise_batch_then` so the parent receipt has non-empty `output_data_receivers`.
3. Calls `promise_return` on the large receipt, causing the runtime to append `output_data_receivers` post-validation.

This requires no special privileges and is reachable via a standard public RPC transaction.

---

### Recommendation

Re-validate the receipt size **after** `output_data_receivers` are appended in `apply_action_receipt`. Specifically, after the `extend_from_slice` call at `runtime/runtime/src/lib.rs` lines 1030–1035, recompute the receipt size and return an error (or abort the `promise_return` substitution) if the new size exceeds `max_receipt_size`. This mirrors the fix recommended in the Hats Protocol report: add a post-execution check that ensures the invariant holds after all mutations, not just at initial creation.

---

### Proof of Concept

1. Deploy a contract with a method `exploit()` that:
   - Creates promise A targeting itself with args padded to `max_receipt_size - base_receipt_size` bytes.
   - Creates promise B (a `.then` on A) targeting itself.
   - Calls `promise_return(A)` so that when A executes, B's `output_data_receivers` are appended to A's receipt.
2. Submit a transaction calling `exploit()`.
3. Observe (as confirmed by `test_max_receipt_size_promise_return`) that a receipt with `size > max_receipt_size` appears in the incoming receipts of the next block, violating the protocol invariant and corrupting the congestion info stored in the shard state.
