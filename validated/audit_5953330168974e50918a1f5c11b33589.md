### Title
Sync Peer Can Inject Arbitrary Transaction Hash via Unauthenticated P2P, Corrupting Storage and Causing `starknet_getTransactionByHash` to Return Hash-Body Mismatch — (`crates/apollo_protobuf/src/converters/transaction.rs`)

---

### Summary

`TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` blindly trusts the `transaction_hash` field supplied by a sync peer without recomputing or verifying it against the transaction body. The injected hash flows through the p2p sync client into persistent storage and is served verbatim by `starknet_getTransactionByHash`, violating the invariant `stored_hash == hash(transaction_body, chain_id)`.

---

### Finding Description

**Step 1 — Protobuf conversion, no hash verification.**

`TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` reads `transaction_hash` directly from the wire message and returns it paired with the decoded transaction body, with no recomputation: [1](#0-0) [2](#0-1) 

The chain `protobuf::TransactionWithReceipt -> FullTransaction` calls this impl and propagates the injected hash into `FullTransaction.transaction_hash`: [3](#0-2) 

**Step 2 — P2P sync client, acknowledged missing validation.**

`TransactionStreamFactory::parse_data_for_block` receives the `FullTransaction` from the peer, destructures it, and pushes the unverified hash directly into `block_body.transaction_hashes`. The TODO comment is the codebase's own acknowledgment of the gap: [4](#0-3) 

**Step 3 — Storage write, injected hash persisted as canonical.**

`write_transactions` stores the peer-supplied hash in both the lookup index (`transaction_hash_to_idx_table`) and in `TransactionMetadata.tx_hash`, with no recomputation: [5](#0-4) 

**Step 4 — RPC, hash-body mismatch served to clients.**

`get_transaction_by_hash` looks up the transaction by the injected hash, retrieves the transaction body, and returns `TransactionWithHash { transaction, transaction_hash }` where `transaction_hash` is the caller-supplied injected value — not a recomputed hash: [6](#0-5) 

The real hash of the transaction body is never indexed, so `starknet_getTransactionByHash(real_hash)` returns `TRANSACTION_HASH_NOT_FOUND` while `starknet_getTransactionByHash(injected_hash)` returns the body with the injected hash in the `transaction_hash` field.

---

### Impact Explanation

- **RPC authority corruption**: `starknet_getTransactionByHash` returns an authoritative-looking response where the `transaction_hash` field does not match `hash(transaction_body, chain_id)`. Any client that re-derives the hash from the returned body will detect a mismatch, breaking trust in the node's RPC.
- **Index corruption**: The real transaction hash is absent from `transaction_hash_to_idx_table`; the injected hash occupies its slot. All hash-keyed lookups (`getTransactionReceipt`, `getTransactionStatus`, `getTransactionByHash`) are affected for the corrupted transactions.
- **Invariant violation**: The storage invariant `stored_hash == hash(stored_body, chain_id)` is broken for every transaction whose hash a malicious peer injects.

This maps to the allowed impact: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"* and *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."*

---

### Likelihood Explanation

The attacker only needs to operate a node that responds to p2p sync queries with crafted `TransactionWithReceipt` messages. No validator, sequencer, or admin privileges are required. Any node syncing from peers (the normal operating mode for full nodes) is exposed. The codebase itself flags the missing check with `// TODO(eitan): Validate transaction hash from untrusted sources`. [7](#0-6) 

---

### Recommendation

In `parse_data_for_block`, after decoding each `FullTransaction`, recompute the transaction hash from the body and `chain_id` using `get_transaction_hash` / `validate_transaction_hash` (already present in `starknet_api::transaction_hash`) and reject the peer (return `ParseDataError::BadPeer`) if the recomputed hash does not match the peer-supplied hash. The conversion layer in `TryFrom<protobuf::TransactionInBlock>` should either drop the `transaction_hash` field entirely (forcing callers to compute it) or treat it as an unverified hint only.

fn injected_hash_accepted_without_verification() {
    use starknet_types_core::felt::Felt;
    use crate::protobuf;

    let arbitrary_hash = Felt::from(0xdeadbeef_u64);

    // Build a minimal InvokeV1 body
    let invoke_v1 = protobuf::transaction_in_block::InvokeV1 {
        sender: Some(protobuf::Address { elements: Some(Felt::ONE.into()) }),
        max_fee: Some(Felt::ZERO.into()),
        signature: Some(protobuf::AccountSignature { parts: vec![] }),
        calldata: vec![],
        nonce: Some(Felt::ZERO.into()),
    };

    let msg = protobuf::TransactionInBlock {
        transaction_hash: Some(arbitrary_hash.into()),
        txn: Some(protobuf::transaction_in_block::Txn::InvokeV1(invoke_v1)),
    };

    let (_, returned_hash) =
        <(starknet_api::transaction::Transaction,
          starknet_api::transaction::TransactionHash)>::try_from(msg).unwrap();

    // Asserts the injected value is returned verbatim — no recomputation
    assert_eq!(returned_hash.0, arbitrary_hash);
    // The actual hash of the InvokeV1 body with chain_id would differ from 0xdeadbeef
}
