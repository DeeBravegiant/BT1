### Title
User-Controlled `EvmFinality::Latest` Allows MPC Attestation of Non-Finalized (Reorg-able) Foreign Transactions - (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` flow accepts a caller-supplied `finality` field in `EvmRpcRequest`. When set to `EvmFinality::Latest`, the MPC network verifies and signs an attestation against the instantaneous chain tip — a block that has not reached economic finality and can be reorganized away. No minimum finality level is enforced at the contract or node level. This is the direct structural analog to the `slot0` vulnerability: both use an instantaneous, manipulable state value instead of a settled/verified one for a critical signing decision.

---

### Finding Description

The `EvmRpcRequest` struct exposes a `finality` field that is fully caller-controlled:

```rust
pub struct EvmRpcRequest {
    pub tx_id: EvmTxId,
    pub extractors: Vec<EvmExtractor>,
    pub finality: EvmFinality,   // <-- caller sets this
}

pub enum EvmFinality {
    Latest,    // instantaneous chain tip — NOT final
    Safe,
    Finalized,
}
``` [1](#0-0) 

The on-chain entry point `verify_foreign_transaction` performs no validation of the finality level — it only checks that the target chain is in the supported set:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);
    let requested_chain = request.request.chain();
    let supported_chains = self.get_supported_foreign_chains();
    if !supported_chains.contains(&requested_chain) { env::panic_str(...); }
    // finality is never inspected — request is enqueued as-is
    self.enqueue_yield_request(...);
}
``` [2](#0-1) 

On the node side, `EvmInspector::verify_finality_level` faithfully translates `Latest` to `FinalityTag::Latest` and queries `eth_getBlockByNumber("latest", ...)`. The check only asserts that the head block number is ≥ the receipt's block number — it does not require the block to be safe or finalized:

```rust
async fn verify_finality_level(
    &self,
    receipt_block_number: U64,
    finality: EthereumFinality,
) -> Result<(), ForeignChainInspectionError> {
    let finality_tag = match finality {
        EthereumFinality::Finalized => FinalityTag::Finalized,
        EthereumFinality::Safe      => FinalityTag::Safe,
        EthereumFinality::Latest    => FinalityTag::Latest,  // <-- accepted
    };
    let head: GetBlockByNumberResponse = self.client.request(GET_BLOCK_BY_NUMBER_METHOD, &args).await?;
    if head.number < receipt_block_number {
        return Err(ForeignChainInspectionError::NotFinalized);
    }
    Ok(())
}
``` [3](#0-2) 

The signed payload commits to the full `ForeignChainRpcRequest` (including the `finality` field), but the MPC network is the trusted verifier — the receiving bridge contract is expected to trust the MPC signature without re-checking finality semantics. [4](#0-3) 

---

### Impact Explanation

The MPC network issues a threshold signature attesting that a foreign-chain transaction was observed and verified. If the transaction was verified at `Latest` finality, the attested block can be reorganized away after the signature is issued. The signature remains cryptographically valid and can be presented to a bridge contract (e.g., Omnibridge inbound flow) to trigger fund release or state changes for a transaction that no longer exists on the canonical chain. This constitutes **forged foreign-chain verification causing invalid bridge execution** — a double-spend or fund-theft condition.

---

### Likelihood Explanation

- Any unprivileged caller can submit `verify_foreign_transaction` with `finality: Latest` — no special role required.
- EVM L2s (Arbitrum, Base, Abstract, HyperEVM, Polygon, BNB) have shorter block times and meaningful reorg windows at the `latest` tag.
- The attacker controls the finality parameter directly in the request struct; no manipulation of the MPC nodes or threshold collusion is needed.
- The attack is profitable whenever the bridge releases value (tokens, wrapped assets) on the NEAR side in exchange for the MPC-attested foreign transaction.

---

### Recommendation

1. **Enforce a minimum finality level at the contract layer**: Reject any `EvmRpcRequest` with `finality == Latest` inside `verify_foreign_transaction`. Only `Safe` or `Finalized` should be accepted for bridge-critical operations.
2. **Enforce at the node layer as a defense-in-depth**: In `EvmInspector::verify_finality_level`, treat `EthereumFinality::Latest` as an error for `ForeignTx`-domain requests.
3. **For Bitcoin**: similarly enforce a minimum `BlockConfirmations` value (e.g., ≥ 6) at the contract level rather than trusting the caller-supplied value.

---

### Proof of Concept

1. Attacker submits a transaction `T` on Ethereum (or any EVM L2) that, when verified, would entitle them to bridge funds on NEAR.
2. Attacker immediately calls `verify_foreign_transaction` with:
   ```json
   {
     "request": {
       "Ethereum": {
         "tx_id": "<T's hash>",
         "finality": "Latest",
         "extractors": ["BlockHash"]
       }
     },
     "domain_id": "<ForeignTx domain>",
     "payload_version": "V1"
   }
   ```
3. The contract enqueues the request without inspecting `finality`. [5](#0-4) 
4. MPC nodes call `EvmInspector::extract(tx_id, EthereumFinality::Latest, ...)`, which queries `eth_getBlockByNumber("latest")` and passes the finality check as long as the latest block number ≥ the receipt's block number. [6](#0-5) 
5. The MPC network produces a threshold signature over `(request, extracted_values)` and calls `respond_verify_foreign_tx`. [7](#0-6) 
6. Attacker receives the valid MPC signature and presents it to the bridge contract to claim funds.
7. The Ethereum block containing `T` is subsequently reorganized. `T` never existed on the canonical chain, but the MPC signature is irrevocable and the bridge has already released funds.

        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",

            request
        );

        self.check_request_preconditions(

        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();

                    requested: requested_chain,
                }
                .to_string(),

            callback_args,
            callback_gas,

        );

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

    async fn extract(
        &self,
        transaction: Chain::TransactionHash,
        finality: EthereumFinality,
        extractors: Vec<EvmExtractor>,
    ) -> Result<Vec<EvmExtractedValue<Chain>>, ForeignChainInspectionError> {
        let get_transaction_receipt_args = GetTransactionReceiptARgs {
            transaction_hash: H256(transaction.into()),
        };
        let transaction_receipt: GetTransactionReceiptResponse = self
            .client
            .request(
                GET_TRANSACTION_RECEIPT_METHOD,
                &get_transaction_receipt_args,
            )
            .await?;

        self.verify_finality_level(transaction_receipt.block_number, finality)
            .await?;
        self.verify_block_is_canonical(
            transaction_receipt.block_number,
            transaction_receipt.block_hash,
        )
        .await?;

        let transaction_success = transaction_receipt.status == U64::one();

        if !transaction_success {
            return Err(ForeignChainInspectionError::TransactionFailed);
        }

        extractors
            .iter()
            .map(|extractor| extractor.extract_value(&transaction_receipt))
            .collect()
    }
