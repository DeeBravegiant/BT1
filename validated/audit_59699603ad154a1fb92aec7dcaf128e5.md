### Title
Unelected Committee Member Can Pre-Cast Binding Governance Votes via Block-Producer Bypass of MEMPOOL Check at Protocol Version 10 — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

---

### Summary

At Conway protocol version 10, the `GOV` rule does not enforce the unelected-committee-voter restriction that the `MEMPOOL` rule does. Because block producers apply the `LEDGER`/`GOV` rule directly without passing through `MEMPOOL`, a block producer can include a transaction in which a cold credential that is only in a pending (not yet enacted) `UpdateCommittee` proposal casts a committee vote. The `GOVCERT` rule already permits such a cold credential to authorize a hot key. Once the `UpdateCommittee` proposal is later enacted, the pre-cast vote is retroactively counted by `committeeAcceptedRatio` in the `RATIFY` rule, potentially tipping a governance action over its ratification threshold without legitimate committee authorization.

---

### Finding Description

**Root-cause chain:**

**Step 1 — GOVCERT permits hot-key registration for unelected cold credentials.**

`checkAndOverwriteCommitteeMemberState` in `GovCert.hs` allows a cold credential that appears in any pending `UpdateCommittee` proposal (`isPotentialFutureMember`) to register a hot key via `ConwayAuthCommitteeHotKey`. No enactment is required. [1](#0-0) 

**Step 2 — GOV rule has no unelected-voter guard at PV 10.**

`conwayGovTransition` in `Gov.hs` only applies the `UnelectedCommitteeVoters` check when `hardforkConwayDisallowUnelectedCommitteeFromVoting` is `True`, i.e., PV ≥ 11. At PV 10 the guard is absent; the GOV rule accepts committee votes from any hot credential registered in `csCommitteeCreds`, regardless of whether the corresponding cold credential is in the enacted committee. [2](#0-1) 

**Step 3 — MEMPOOL check is bypassable by block producers.**

The `mempoolTransition` in `Mempool.hs` does apply the unelected-committee check at PV 10, but the MEMPOOL rule is only invoked for transactions entering the mempool. Block producers apply the `LEDGER` rule (which calls `GOV`) directly when building a block, bypassing `MEMPOOL` entirely. The code comment itself acknowledges the gap: [3](#0-2) 

**Step 4 — Pre-cast vote is retroactively counted after enactment.**

`committeeAcceptedRatio` in `Ratify.hs` iterates over enacted committee members, looks up each member's hot key in `csCommitteeCreds`, then looks up that hot key in `gasCommitteeVotes`. If the pre-cast vote was stored in `gasCommitteeVotes` before enactment, it is counted once the cold credential enters the enacted committee. [4](#0-3) 

**Step 5 — `authorizedElectedHotCommitteeCredentials` confirms the vote counts post-enactment.** [5](#0-4) 

**`hardforkConwayDisallowUnelectedCommitteeFromVoting` definition:** [6](#0-5) 

**Full exploit path:**

1. Attacker (any ADA holder) submits an `UpdateCommittee` proposal adding their cold credential `C` (deposit required, no governance threshold needed to *propose*).
2. Attacker submits a `ConwayAuthCommitteeHotKey C H` certificate in the same or a subsequent transaction; `GOVCERT` accepts it because `C` is in a pending proposal (`isPotentialFutureMember = True`).
3. Attacker, acting as a block producer, crafts a transaction containing `CommitteeVoter H` voting `VoteYes` on a target governance action `G` and includes it directly in a block. The `GOV` rule at PV 10 has no `UnelectedCommitteeVoters` check; the vote is stored in `gasCommitteeVotes G`.
4. DReps and SPOs (possibly including the attacker) vote to enact the `UpdateCommittee` proposal through normal governance.
5. At the epoch boundary, `C` enters the enacted committee. `committeeAcceptedRatio` now counts the pre-cast `VoteYes` from `H`.
6. If this vote tips `G` over the committee threshold, `G` is ratified and enacted.

---

### Impact Explanation

A governance action (`TreasuryWithdrawals`, `HardForkInitiation`, `ParameterChange`, `NewConstitution`) can be ratified and enacted using a committee vote that was cast before the voter was legitimately elected to the committee. This constitutes an unauthorized governance action being enacted, matching the allowed Critical impact: *"Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted."*

---

### Likelihood Explanation

The attacker must (a) be a registered stake pool operator capable of producing a block (a "block producer below consensus threshold," which is an explicitly allowed attacker profile), and (b) arrange for the `UpdateCommittee` proposal to be enacted through normal DRep + SPO voting. The block-producer requirement is satisfied by any registered SPO; the governance requirement is non-trivial but achievable by a well-resourced attacker who controls sufficient DRep or SPO stake. The window is limited to PV 10 (before the PV 11 hard fork activates the GOV-rule guard). Likelihood is **Low-Medium**.

---

### Recommendation

Apply the `hardforkConwayDisallowUnelectedCommitteeFromVoting` guard unconditionally in `conwayGovTransition` (or equivalently, always check for unelected committee voters regardless of protocol version), rather than relying solely on the MEMPOOL rule for PV 10. The comment in `Mempool.hs` already acknowledges this gap ("This check can completely be removed once mainnet switches to protocol version 11, since the same check has been implemented in the GOV rule"), but the GOV rule check is only active at PV ≥ 11, leaving PV 10 blocks unprotected against direct block inclusion.
