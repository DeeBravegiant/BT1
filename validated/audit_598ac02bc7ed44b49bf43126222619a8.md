### Title
Permanently Unrecoverable Funds When Recipient Is Blacklisted in a Token During Cross-Chain Finalization — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates an outbound transfer from NEAR to an EVM chain (or Starknet/Aptos) for a token with a blacklist mechanism (e.g., USDC, USDT), the origin-chain tokens are burned or locked before the destination-chain `finTransfer` is executed. If the recipient address is subsequently blacklisted in the token contract, `finTransfer` will always revert, and there is no cancellation or refund path on the NEAR side. The user's tokens are permanently unrecoverable.

### Finding Description

**Step 1 — Origin chain burns/locks tokens irreversibly.**

In `near/omni-bridge/src/lib.rs`, `init_transfer_internal` burns deployed tokens and locks native tokens before storing the transfer message: [1](#0-0) 

Once `burn_tokens_if_needed` is called, the tokens are gone from the user's balance and from the bridge's supply. The transfer message is stored in `pending_transfers`.

**Step 2 — Destination-chain `finTransfer` sends to a fixed, MPC-signed recipient.**

In `evm/src/omni-bridge/contracts/OmniBridge.sol`, `finTransfer` marks the nonce used and then attempts to transfer tokens to `payload.recipient`: [2](#0-1) 

The `payload.recipient` is part of the Borsh-encoded message that was signed by the NEAR MPC. It cannot be changed without invalidating the signature. If `safeTransfer(payload.recipient, payload.amount)` reverts (e.g., because `payload.recipient` is blacklisted in USDC), the entire transaction reverts — including the nonce marking at line 287. The nonce is therefore not permanently consumed, but the relayer can never produce a valid payload with a different recipient, so every retry will fail identically.

**Step 3 — No cancellation or refund path exists on NEAR.**

`remove_transfer_message` is only called inside `claim_fee_callback`, which requires proof of a successful `finTransfer` on the destination chain: [3](#0-2) 

There is no public `cancel_transfer`, `refund_transfer`, or equivalent function. The transfer message remains in `pending_transfers` indefinitely, and the burned/locked tokens are permanently unrecoverable.

The same pattern exists in Starknet's `fin_transfer`: [4](#0-3) 

And in Aptos's `fin_transfer`: [5](#0-4) 

### Impact Explanation

This is a **Critical — Irreversible fund lock**. For any token with a blacklist (USDC, USDT, and others), if the recipient's address is blacklisted between the time `init_transfer` is called on NEAR and the time `finTransfer` is executed on the destination chain, the user's tokens are permanently locked in the bridge with no recovery path. The MPC signature binds the recipient address, and the NEAR contract has no mechanism to cancel the pending transfer and refund the user.

### Likelihood Explanation

**Medium.** Blacklistable tokens (USDC, USDT) are among the most commonly bridged assets. Blacklisting events do occur (regulatory compliance, sanctions enforcement). The window between `init_transfer` and `finTransfer` is typically short, but for high-value transfers or during network congestion, the window can be extended. The scenario requires no attacker — it is triggered by a normal token-admin action (blacklisting) that is entirely external to the bridge.

### Recommendation

Add a cancellation/refund path on the NEAR side that allows a pending outbound transfer to be cancelled and the locked tokens returned (or burned tokens re-minted) to the sender, callable after a timeout or after a proven failed finalization. Alternatively, allow the MPC to sign an updated payload with a different recipient address when the original recipient is provably unable to receive the token.

### Proof of Concept

1. Alice holds 1000 USDC on NEAR (as a deployed bridge token).
2. Alice calls `ft_transfer_call` on the USDC token contract, triggering `init_transfer` on the NEAR bridge with `recipient = alice_evm_address`.
3. `init_transfer_internal` burns Alice's 1000 USDC on NEAR and stores the transfer in `pending_transfers`.
4. Before the relayer calls `finTransfer` on Ethereum, Circle blacklists `alice_evm_address`.
5. The relayer calls `finTransfer` on `OmniBridge.sol`. The call reaches line 351: `IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount)`. USDC reverts because `alice_evm_address` is blacklisted. The entire transaction reverts.
6. The relayer retries — same result every time. The MPC signature is bound to `alice_evm_address` and cannot be changed.
7. Alice's 1000 USDC are permanently burned on NEAR. There is no `cancel_transfer` function. `remove_transfer_message` is never called. Alice's funds are permanently lost. [6](#0-5) [7](#0-6)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
