### Title
Unverified `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Replay of Foreign-Chain Attestations — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` accepts `payload_hash` directly from the caller-submitted response and only checks that the signature is valid for that hash under the root public key. It never verifies that `payload_hash` was actually derived from the pending `request` being resolved. A single attested MPC leader node can replay a threshold signature obtained for one foreign-chain request to resolve a completely different pending request, delivering a forged attestation to the waiting user.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract extracts `payload_hash` from the caller-supplied `response` struct and verifies only that the ECDSA signature is valid for that hash under the domain's root public key:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract never recomputes `payload_hash` from the `request` argument, nor does it verify that `payload_hash` encodes the same `request` that is being resolved. The `request` is only used to look up the pending yield queue entry:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

Contrast this with the regular `respond` function, where `payload_hash` is taken from the on-chain `request.payload` (stored at submission time), not from the caller-supplied response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [3](#0-2) 

The signed payload for foreign-chain verification is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [4](#0-3) 

with `payload_hash = SHA256(borsh(ForeignTxSignPayload))`. [5](#0-4) 

Because the contract never checks that the submitted `payload_hash` encodes the correct `request`, a malicious leader can present any previously obtained threshold signature as a valid response to any pending request.

---

### Impact Explanation

**Impact: High — Forged foreign-chain verification causing invalid bridge execution.**

Attack scenario:

1. User A submits `verify_foreign_transaction(request_A)` for Bitcoin tx X — enters pending queue.
2. User B submits `verify_foreign_transaction(request_B)` for Bitcoin tx Y — enters pending queue.
3. The malicious leader node drives the threshold signing protocol for request A, obtaining a valid signature `sig_A` over `payload_hash_A = SHA256(borsh(ForeignTxSignPayload{request_A, values_A}))`.
4. The malicious leader calls `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
5. The contract verifies `sig_A` is valid for `payload_hash_A` under the root public key — **passes**.
6. The contract resolves request B's yield with the forged response.
7. User B's contract receives `{ payload_hash: payload_hash_A, signature: sig_A }` — a signature attesting to Bitcoin tx X's block hash, delivered as the answer to a query about Bitcoin tx Y.

Any bridge contract that trusts the returned `payload_hash` without independently recomputing it from the expected request and extracted values will accept this forged attestation. The SDK's `ForeignChainSignatureVerifier` does perform this check:

```rust
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { ... });
}
``` [6](#0-5) 

but the on-chain contract — the authoritative trust anchor — performs no such check, leaving every caller that does not use the SDK exposed.

---

### Likelihood Explanation

A single attested MPC participant acting as the signing leader for any foreign-chain request can execute this attack without any collusion above the threshold. The leader is the only node that assembles the final threshold signature and submits `respond_verify_foreign_tx`. The followers independently verify the foreign-chain RPC result and contribute their signing shares, but they do not control which pending request the leader resolves with the resulting signature. The leader can legitimately obtain `sig_A` for request A and then submit it against request B in a separate transaction. Both requests can be simultaneously pending (the fan-out design explicitly supports multiple concurrent requests). [7](#0-6) 

---

### Recommendation

The contract should recompute `payload_hash` from the `request` and the caller-supplied `values`, then assert it matches `response.payload_hash`. This requires the responder to also submit the `values` (extracted chain data) alongside the hash, mirroring how the SDK verifier works:

```rust
// Require the node to submit the extracted values
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(), // new field
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This binds the `payload_hash` to the specific `request` being resolved, eliminating the cross-request replay vector. The `values` are already bounded in size by the extractor definitions, so adding them to the response does not introduce unbounded storage.

---

### Proof of Concept

1. Deploy the contract with two pending `verify_foreign_transaction` requests: `request_A` (Bitcoin tx `[0xAA; 32]`) and `request_B` (Bitcoin tx `[0xBB; 32]`).
2. As the leader, complete the threshold signing protocol for `request_A`, obtaining `sig_A` over `payload_hash_A`.
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
4. Observe that the contract accepts the call (signature check passes against the root key).
5. Observe that request B's yield resolves with `payload_hash_A` — the hash of Bitcoin tx X's block data — even though request B asked about Bitcoin tx Y.
6. A bridge contract that does not call `ForeignChainSignatureVerifier::verify_signature` will accept this as a valid attestation for Bitcoin tx Y. [8](#0-7) [9](#0-8)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
