Audit Report

## Title
Zero rsETH Minted on Deposit Due to Integer Division Truncation — (`contracts/LRTDepositPool.sol`)

## Summary

`getRsETHAmountToMint` performs integer division that silently truncates to zero when the deposit amount is small relative to the rsETH price. `_beforeDeposit` contains no independent zero-output guard; the sole protection is the caller-supplied `minRSETHAmountExpected` parameter. When a caller passes `0` for that parameter — a common default — assets are permanently transferred to the protocol while zero rsETH is minted, leaving the depositor with no redemption token and no recovery path.

## Finding Description

`getRsETHAmountToMint` at line 520 of `LRTDepositPool.sol` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

This is plain Solidity integer division. When `amount * assetPrice < rsETHPrice`, the result truncates to zero.

`_beforeDeposit` (lines 648–670) computes `rsethAmountToMint` and then checks only:

```solidity
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
``` [2](#0-1) 

There is no unconditional `if (rsethAmountToMint == 0) revert` guard. When `minRSETHAmountExpected = 0`, the comparison `0 < 0` evaluates to `false` and the function returns `0` without reverting.

`depositAsset` then unconditionally executes the asset transfer and mint:

```solidity
uint256 rsethAmountToMint = _beforeDeposit(asset, depositAmount, minRSETHAmountExpected);
IERC20(asset).safeTransferFrom(msg.sender, address(this), depositAmount);
_mintRsETH(rsethAmountToMint);
``` [3](#0-2) 

`_mintRsETH` calls `IRSETH.mint(msg.sender, 0)`, issuing zero tokens to the depositor. [4](#0-3) 

The oracle sanity check permits asset prices as low as `1e16`: [5](#0-4) 

`rsETHPrice` initialises at `1e18` and grows monotonically as yield accrues: [6](#0-5) 

For `assetPrice = 1e16` and `rsETHPrice = 1e18`, any deposit of fewer than 100 asset units rounds to zero. For ETH-pegged assets, the threshold is 1 wei once `rsETHPrice` exceeds `1e18`.

The `minAmountToDeposit` check (`depositAmount == 0 || depositAmount < minAmountToDeposit`) does not close the gap: `minAmountToDeposit` defaults to `0`, so any non-zero deposit passes, and even a configured minimum (e.g., 50 units) still allows amounts in the range `[minAmountToDeposit, threshold)` to produce zero rsETH. [7](#0-6) 

## Impact Explanation

A depositor who receives zero rsETH has permanently lost their deposited assets. All withdrawal paths (`initiateWithdrawal`, `instantWithdrawal`) require the caller to supply a non-zero rsETH balance:

```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
``` [8](#0-7) 

With zero rsETH, the depositor cannot initiate any withdrawal. The deposited assets are irrecoverable by the user. This is **permanent freezing of funds** (Critical) and constitutes a direct, permanent loss of user funds.

## Likelihood Explanation

The condition requires two realistic, independently common states:

1. A deposit amount small enough that `amount * assetPrice < rsETHPrice`. This is trivially true for 1-wei deposits of any asset, for low-value assets (price `1e16`) with deposits under 100 units, or for any asset once `rsETHPrice` has grown above `assetPrice`.
2. The caller passes `minRSETHAmountExpected = 0`. This is the default in scripts, integration wrappers, and front-ends that omit explicit slippage protection, and is a well-documented failure mode across DeFi.

No privileged role is required. Any external account can trigger this on every supported asset.

## Recommendation

Add an unconditional zero-output revert inside `_beforeDeposit`, independent of the caller-supplied slippage parameter:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);

if (rsethAmountToMint == 0) revert ZeroRsETHMinted();

if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

This ensures the protocol never silently accepts a deposit that produces no rsETH, regardless of what the caller supplies for `minRSETHAmountExpected`.

## Proof of Concept

1. Deploy with a supported asset whose oracle price is `1e16` (within the permitted range `[1e16, 1e19]`). `rsETHPrice = 1e18` (initial value).
2. Call `depositAsset(asset, 99, 0, "")` from any EOA.
3. `getRsETHAmountToMint` computes `(99 * 1e16) / 1e18 = 0` (integer truncation).
4. `_beforeDeposit` checks `0 < 0` → `false` → returns `0` without reverting.
5. `safeTransferFrom` moves 99 units from the caller to the contract.
6. `_mintRsETH(0)` mints zero rsETH to the caller.
7. Caller holds 0 rsETH; `initiateWithdrawal` reverts with `InvalidAmountToWithdraw` on any attempt to recover the 99 units.
8. The 99 units are permanently locked in the protocol.

A Foundry fuzz test parameterising `depositAmount` over `[1, threshold)` and `minRSETHAmountExpected = 0` will reproduce the zero-mint outcome for all values in that range.
