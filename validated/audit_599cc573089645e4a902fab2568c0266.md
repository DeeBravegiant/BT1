Audit Report

## Title
Stale `rsETHPrice` Enables Depositors to Capture stETH Rebase Yield from Existing Holders — (`contracts/LRTOracle.sol` / `contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle.rsETHPrice` is a lazily-updated state variable that is never refreshed atomically during deposits. Because stETH rebases increase the protocol's live TVL without triggering a price update, any depositor can call `depositETH` or `depositAsset` between a rebase and the next `updateRSETHPrice()` invocation, receiving more rsETH than the post-rebase NAV warrants. After the price is updated, the attacker's position is worth more than they deposited — at the direct expense of pre-existing rsETH holders whose stETH yield was diluted.

## Finding Description

`getRsETHAmountToMint` in `LRTDepositPool` computes the mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`rsETHPrice` is a plain storage variable in `LRTOracle`: [2](#0-1) 

It is only written inside `_updateRsETHPrice()`, which is invoked exclusively by the two explicit update functions — neither of which is called from `depositETH` or `depositAsset`: [3](#0-2) 

When `_updateRsETHPrice()` does run, it computes live TVL by reading `IERC20(asset).balanceOf(...)` in real time through `getTotalAssetDeposits`: [4](#0-3) [5](#0-4) 

stETH is a rebasing token: its `balanceOf` increases automatically at each daily rebase with no on-chain transaction. The moment a rebase occurs, `_getTotalEthInProtocol()` would return a higher value, but `rsETHPrice` remains at its last stored value. There is no staleness timestamp, no freshness guard, and no automatic update triggered by `_beforeDeposit`: [6](#0-5) 

**Exploit path (no special role required):**

1. stETH rebases → `IERC20(stETH).balanceOf(depositPool)` increases by δ; `rsETHPrice` remains at pre-rebase value `P_old = TVL / S`.
2. Attacker calls `depositETH(1 ether)` → receives `1e18 / P_old` rsETH.
3. Correct post-rebase mint would be `1e18 / P_new` where `P_new = (TVL + δ) / S > P_old`.
4. `updateRSETHPrice()` is called → `rsETHPrice` rises to `P_new`.
5. Attacker's excess rsETH is now worth more than 1 ETH; the difference `≈ δ / TVL` per unit deposited is extracted from existing holders' accrued yield.

Note: the `pricePercentageLimit` guard in `_updateRsETHPrice` does not mitigate this — it only throttles the price update itself, and if the rebase is large enough to trigger `PriceAboveDailyThreshold`, the stale window is extended further, worsening the attack surface. [7](#0-6) 

## Impact Explanation

Existing rsETH holders' unclaimed stETH rebase yield is diluted. The attacker receives rsETH at a discount relative to the true post-rebase NAV; after the price update their position is worth more than they deposited. This is a direct, quantifiable transfer of yield from existing holders to the depositor. Impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation

stETH rebases once per day on Ethereum mainnet. `updateRSETHPrice()` is not called atomically with deposits and depends on an off-chain keeper. The stale window is predictable and can last minutes to hours. No special role, no private key, and no governance action is required — only a standard `depositETH` or `depositAsset` call from any EOA. The attack is repeatable every rebase cycle. Likelihood: **High**.

## Recommendation

Call `_updateRsETHPrice()` (or an equivalent inline computation) at the start of `_beforeDeposit` before computing `getRsETHAmountToMint`, so the mint ratio always reflects the current live TVL. Alternatively, compute the mint ratio on-the-fly from live TVL and rsETH total supply rather than from the cached `rsETHPrice` state variable. As a weaker mitigation, record a `lastPriceUpdateTimestamp` and revert deposits when `rsETHPrice` has not been refreshed within a configurable staleness window (e.g., 1 hour).

## Proof of Concept

**Foundry fork test outline:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;
import "forge-std/Test.sol";

contract StaleRsETHPricePoC is Test {
    ILRTDepositPool depositPool = ILRTDepositPool(<DEPOSIT_POOL>);
    ILRTOracle      oracle      = ILRTOracle(<ORACLE>);
    IRSETH          rseth       = IRSETH(<RSETH>);
    address         steth       = <STETH>;

    function testStaleYieldCapture() external {
        // 1. Record pre-rebase price
        uint256 priceBefore = oracle.rsETHPrice();

        // 2. Simulate stETH rebase: advance time by 1 day so Lido oracle triggers rebase
        //    vm.warp(block.timestamp + 1 days);
        //    (or vm.store stETH totalShares/totalPooledEther to simulate balance increase)

        // 3. Confirm rsETHPrice is still stale
        assertEq(oracle.rsETHPrice(), priceBefore);

        // 4. Deposit 1 ETH at stale price
        uint256 rsethBefore = rseth.balanceOf(address(this));
        depositPool.depositETH{value: 1 ether}(0, "");
        uint256 rsethMinted = rseth.balanceOf(address(this)) - rsethBefore;

        // 5. Update price to reflect rebase
        oracle.updateRSETHPrice();
        uint256 priceAfter = oracle.rsETHPrice();
        assertGt(priceAfter, priceBefore);

        // 6. Attacker's position is worth more than 1 ETH
        uint256 attackerValueETH = rsethMinted * priceAfter / 1e18;
        assertGt(attackerValueETH, 1 ether);
    }
}
```

The `rsethMinted` exceeds what the post-rebase price would have issued, and after `updateRSETHPrice()` the attacker's position is worth more than the deposited ETH — at the expense of pre-existing rsETH holders whose stETH yield was diluted.
