### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Reuse to Forge Foreign-Chain Attestations - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over `response.payload_hash`, but never validates that `response.payload_hash` is actually derived from the `request` stored on-chain. A single malicious attested participant can reuse a valid MPC signature obtained from any prior legitimate `verify_foreign_transaction` round to resolve a completely different pending request, delivering a forged attestation to the caller.

---

### Finding Description

The vulnerability class from the external report is: **a proof root is accepted without being anchored to a trusted on-chain state**. In `withdrawBySnapshot()`, `extraProof.root` is never validated against `massExitBlock.blockRid`. The direct analog here is that `response.payload_hash` in `respond_verify_foreign_tx` is never validated against the `request` stored in `pending_verify_foreign_tx_requests`.

In `respond_verify_foreign_tx` (lines 691–754 of `crates/contract/src/lib.rs`), the contract performs these checks:

1. Caller is an attested participant.
2. The signature in `response.signature` is valid over `response.payload_hash` using the ForeignTx domain root public key.
3. The `request` key exists in `pending_verify_foreign_tx_requests` (via `resolve_yields_for`). [1](#0-0) 

What is **never** checked is whether `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific `request` being resolved. The `payload_hash` is completely decoupled from the `request` key used to look up the pending yield.

The canonical payload structure is: [2](#0-1) 

`ForeignTxSignPayloadV1` embeds the full `ForeignChainRpcRequest` (including `tx_id`, `extractors`, `finality`) alongside the `ExtractedValue` list. The `compute_msg_hash` function hashes the entire Borsh-serialized payload. The contract never recomputes or validates this binding. [3](#0-2) 

On the node side, the leader correctly computes `payload_hash` from the actual foreign-chain inspection result and submits it: [4](#0-3) 

But the contract performs no corresponding on-chain re-derivation or binding check.

---

### Impact Explanation

**Impact: High — Forged foreign-chain verification / invalid bridge execution.**

An attacker who is a single attested participant can:

1. **Observe** any prior legitimate `respond_verify_foreign_tx` call on-chain (NEAR transactions are public) and extract `(hash_A, sig_A)` — a valid MPC signature over `SHA-256(borsh(ForeignTxSignPayload{tx_id_X, values_X}))`.
2. **Submit** a new `verify_foreign_transaction` request for a different `tx_id_Y` (any chain, any transaction ID), paying the small deposit. This creates a pending yield entry keyed by `request_B`.
3. **Call** `respond_verify_foreign_tx(request=request_B, response=(payload_hash=hash_A, signature=sig_A))`.
4. The contract verifies `sig_A` over `hash_A` → **valid** (it is a genuine MPC signature).
5. The contract resolves the yield for `request_B` and returns `(hash_A, sig_A)` to the caller of `verify_foreign_transaction(tx_id_Y)`.

The caller receives a response whose `payload_hash` encodes `tx_id_X`'s block hash, not `tx_id_Y`'s. Any downstream bridge contract (e.g., the omnibridge inbound flow described in the design doc) that does not independently reconstruct and compare the expected `payload_hash` will accept this as a valid attestation for `tx_id_Y`, enabling invalid bridge execution or double-spend conditions. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Medium.**

- Requires the attacker to be a single attested participant (TEE-verified). TEE attestation is a defense-in-depth measure but does not eliminate the contract-level missing check.
- No threshold cooperation is needed: the attacker reuses an already-produced signature from public on-chain data.
- The attacker only needs to pay the `MINIMUM_SIGN_REQUEST_DEPOSIT` to create the target pending request.
- The attack is silent — the contract emits no error and the yield resolves normally.

---

### Recommendation

In `respond_verify_foreign_tx`, require the caller to also submit the full `ForeignTxSignPayload` (not just the hash). The contract should then:

1. Assert `payload.request == request` (the pending request's `ForeignChainRpcRequest` matches the submitted payload's embedded request).
2. Recompute `expected_hash = SHA-256(borsh(payload))`.
3. Assert `response.payload_hash == expected_hash`.
4. Verify the signature over `expected_hash`.

This anchors `payload_hash` to the on-chain `request`, closing the analog of the `extraProof.root`-not-validated-against-`massExitBlock.blockRid` vulnerability.

---

### Proof of Concept

```
// Setup: attacker is an attested participant.

// Step 1: Observe a legitimate response for tx_id_X on-chain.
//   hash_A = SHA-256(borsh(ForeignTxSignPayload::V1 { request: Bitcoin(tx_id_X, ...), values: [BlockHash(H1)] }))
//   sig_A  = valid MPC Secp256k1 signature over hash_A

// Step 2: Attacker creates a pending request for tx_id_Y.
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: tx_id_Y,          // different transaction
        confirmations: 1.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
});

// Step 3: Attacker calls respond with request=B but response from request A.
contract.respond_verify_foreign_tx(
    request_B,                                    // pending request for tx_id_Y
    VerifyForeignTransactionResponse {
        payload_hash: hash_A,                     // hash bound to tx_id_X
        signature: sig_A,                         // valid signature over hash_A
    },
);
// Contract checks: verify_ecdsa_signature(sig_A, hash_A, root_pk) → OK
// Contract resolves yield for request_B → caller of verify_foreign_transaction(tx_id_Y)
//   receives (hash_A, sig_A) — a forged attestation for tx_id_Y.
```

The contract's signature check at line 729 passes because `sig_A` is a genuine MPC signature over `hash_A`. The missing check — that `hash_A` is derived from `request_B` — is never performed. [6](#0-5) [7](#0-6)

    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
