### Title
Unvalidated `ProposalInit.builder` field allows a malicious proposer to redirect all block fees to an arbitrary sequencer address — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The `ProposalInit.builder` field is proposer-supplied over the consensus P2P network and is passed directly into `BlockInfo.sequencer_address` without any validation in `is_proposal_init_valid`. Because `sequencer_address` is the address that receives all transaction fees during block execution and is committed into the block hash, a malicious proposer can set `builder` to any address — including their own — causing every validator to execute the block with the wrong sequencer address, crediting all fees to the attacker and committing a block with a permanently wrong sequencer address on-chain.

### Finding Description

**Root cause — `builder` is never validated**

`is_proposal_init_valid` in `validate_proposal.rs` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, `timestamp`, L1 gas prices (within margin), and `fee_proposal_fri`. The `ProposalInitValidation` struct that carries the locally-trusted reference values contains no `builder` field at all:

```rust
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub block_timestamp_window_seconds: u64,
    pub previous_proposal_init: Option<PreviousProposalInitInfo>,
    pub l1_da_mode: L1DataAvailabilityMode,
    pub l2_gas_price_fri: GasPrice,
    pub starknet_version: StarknetVersion,
    pub fee_actual: Option<GasPrice>,
}
``` [1](#0-0) 

The `builder` field is never compared against the locally-configured `builder_address` (stored in `ContextStaticConfig`).

**Propagation — `builder` becomes `sequencer_address` in `BlockInfo`**

`convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address`:

```rust
Ok(starknet_api::block::BlockInfo {
    block_number: init.height,
    block_timestamp: BlockTimestamp(init.timestamp),
    sequencer_address: init.builder,   // ← proposer-controlled, unvalidated
    ...
})
``` [2](#0-1) 

This `BlockInfo` is passed to the batcher for both proposal building and validation: [3](#0-2) 

**Fee collection uses `sequencer_address`**

During execution, `add_fee_to_sequencer_balance` credits all transaction fees to `block_context.block_info.sequencer_address`: [4](#0-3) 

**`sequencer_address` is committed into the block hash**

`PartialBlockHashComponents::new` includes `sequencer_address` in the partial block hash, which is then chained into the final block hash:

```rust
sequencer: SequencerContractAddress(block_info.sequencer_address),
``` [5](#0-4) 

```rust
.chain(&partial_block_hash_components.sequencer.0)
``` [6](#0-5) 

**Why the `ProposalCommitment` comparison does not catch this**

At the end of `validate_proposal`, the validator compares `built_block` (its own batcher output) against `received_fin.proposal_commitment`. Both sides computed their commitment using the same proposer-supplied `init.builder`, so the comparison passes even when `builder` is spoofed: [7](#0-6) 

### Impact Explanation

A malicious proposer sets `ProposalInit.builder` to an attacker-controlled address. Every validator node accepts the proposal (no `builder` check), executes all transactions with the spoofed `sequencer_address`, and credits all block fees to the attacker's address. The resulting block hash permanently encodes the wrong sequencer address on-chain. This is a **Critical** impact: "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."

Additionally, any contract that calls `get_execution_info()` and branches on `sequencer_address` (e.g., fee-discount logic, sequencer-only gates) will behave incorrectly for the entire block.

### Likelihood Explanation

Any consensus participant with proposing rights can trigger this. The `builder_address` is a plain node-config field with no on-chain or protocol-level binding to the proposer's identity. The `assume_no_malicious_validators: true` flag in integration tests confirms the system currently relies on social trust rather than code enforcement. [8](#0-7) 

### Recommendation

**Short term**: Add `builder` to `ProposalInitValidation` and check it in `is_proposal_init_valid`:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
```

The `expected_builder` value should come from the node's locally-configured `ContextStaticConfig.builder_address`, not from the network.

**Long term**: Never derive security-sensitive execution parameters (sequencer address, fee recipient) from proposer-supplied payload fields without authenticating them against a locally-trusted source. The `builder` address should either be fixed by protocol (derived from the proposer's identity) or enforced by a signed commitment that validators can independently verify.

### Proof of Concept

1. Attacker controls a validator node with proposing rights.
2. Attacker modifies their node to set `ProposalInit.builder` to `ATTACKER_WALLET` instead of the legitimate `builder_address`.
3. When selected as proposer, the node broadcasts a `ProposalInit` with `builder = ATTACKER_WALLET`.
4. Each validator calls `is_proposal_init_valid` — no check on `builder` — and proceeds.
5. `initiate_validation` calls `convert_to_sn_api_block_info(init)`, producing `BlockInfo { sequencer_address: ATTACKER_WALLET, ... }`.
6. The batcher executes all transactions; `add_fee_to_sequencer_balance` credits every fee to `ATTACKER_WALLET`.
7. The batcher returns a `partial_block_hash` that includes `ATTACKER_WALLET` as sequencer.
8. The proposer's `ProposalFin.proposal_commitment` was computed with the same `ATTACKER_WALLET`, so `built_block == received_fin.proposal_commitment` — the proposal is accepted.
9. `decision_reached` commits the block; `sequencer_address = ATTACKER_WALLET` is stored in the block header and block hash permanently.
10. All fees from the block are in `ATTACKER_WALLET`'s balance; the legitimate sequencer receives nothing.
