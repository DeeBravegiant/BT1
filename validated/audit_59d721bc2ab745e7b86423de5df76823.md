### Title
Unguarded `assert` in `solution_to_pool_state` Allows Malicious Peer to Permanently Halt Pool Wallet — (`chia/pools/pool_puzzles.py`, `chia/pools/pool_wallet.py`)

---

### Summary

A malicious peer can return a `CoinSpend` with a structurally malformed solution (e.g., an inner solution with 4 elements) that passes all checks in `add_spend` and `apply_state_transition`, gets committed to the DB, and then causes an unguarded `AssertionError` in `solution_to_pool_state` on every subsequent call to `get_current_state`. Because `rewind` calls `get_current_state` *before* `rollback`, even a reorg cannot recover the wallet. The result is a permanently halted pool wallet requiring manual DB intervention.

---

### Finding Description

**1. The unguarded assertion**

`solution_to_pool_state` contains a hard `assert` with no surrounding `try/except`:

```python
num_args = len(inner_solution.as_python())
assert num_args in {2, 3}   # line 413 — raises AssertionError if num_args == 4
``` [1](#0-0) 

**2. `add_spend` stores spends with zero solution validation**

`WalletPoolStore.add_spend` only checks height ordering and parent-chain continuity. It performs no CLVM execution and no solution structure validation: [2](#0-1) 

**3. `apply_state_transition` stores the spend *before* calling `solution_to_pool_state`**

At line 286 the malformed spend is committed to the DB. The loop at lines 291–300 then calls `solution_to_pool_state` on every stored spend in reverse order — hitting the malformed spend first — and the `AssertionError` propagates uncaught: [3](#0-2) 

**4. `get_current_state` has no exception handling**

Every subsequent call to `get_current_state` iterates stored spends and calls `solution_to_pool_state` with no `try/except`, so the `AssertionError` propagates permanently: [4](#0-3) 

**5. `rewind` calls `get_current_state` *before* `rollback` — recovery is impossible**

Even a blockchain reorg cannot fix the broken state. `rewind` calls `get_current_state()` at line 314, which raises `AssertionError`, so `rollback()` at line 315 is never reached. The `except Exception` swallows the error and returns `False`, leaving the malformed spend in the DB: [5](#0-4) 

---

### Impact Explanation

Once the malformed spend is committed, every operation that calls `get_current_state` — including `apply_state_transition`, `update_pool_config`, `join_pool`, `self_pool`, `claim_pool_rewards`, `new_peak`, and `rewind` — raises `AssertionError`. The user permanently loses the ability to leave the pool, claim rewards, or self-pool. Manual SQLite intervention is required to delete the malformed row from `pool_state_transitions`.

This matches the **High** impact criterion: *"Permanent or long-lived inability for honest wallets to process pool actions under normal network assumptions."*

---

### Likelihood Explanation

The attacker only needs to operate a reachable Chia full-node peer. When the victim's wallet detects that the pool singleton coin has been spent (via a `CoinState` update), it issues a `RequestPuzzleSolution` to its connected peer. A malicious peer can return any solution bytes it chooses. The only guard in `apply_state_transition` is the coin-name check (line 276), which the attacker satisfies trivially because the real tip coin's `parent_coin_info`, `puzzle_hash`, and `amount` are all public on-chain data. [6](#0-5) 

---

### Recommendation

1. **Replace the bare `assert` with a guarded return.** In `solution_to_pool_state`, replace `assert num_args in {2, 3}` with `if num_args not in {2, 3}: return None`. This makes the function safe to call on any peer-supplied spend.

2. **Validate solution structure in `add_spend` or `apply_state_transition`** before committing to the DB — call `solution_to_pool_state` on the spend and reject it if it raises or returns an unexpected result.

3. **Fix the ordering bug in `rewind`**: call `rollback` before `get_current_state`, or wrap `get_current_state` in its own `try/except` so a broken spend can be rolled back during a reorg.

---

### Proof of Concept

```python
# Craft a CoinSpend whose inner_solution has 4 elements
malformed_solution = Program.to([
    lineage_proof,          # outer wrapper element 1
    singleton_amount,       # outer wrapper element 2
    [1, 2, 3, 4],           # inner_solution with 4 args → triggers assert
])
malformed_spend = make_spend(tip_coin, tip_puzzle, malformed_solution)

# Step 1: add_spend accepts it (no solution validation)
await pool_store.add_spend(wallet_id, malformed_spend, block_height)

# Step 2: get_current_state now raises AssertionError on every call
try:
    await pool_wallet.get_current_state()
except AssertionError:
    print("Wallet permanently halted")  # always reached

# Step 3: rewind cannot recover — get_current_state is called before rollback
result = await pool_wallet.rewind(block_height - 1, action_scope)
assert result is False  # exception swallowed, rollback never executed
```
