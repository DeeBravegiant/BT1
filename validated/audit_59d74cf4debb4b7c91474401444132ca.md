### Title
Pool Exit Timer Reset via Absorb Spend in `LEAVING_POOL` State — (`File: chia/pools/pool_puzzles.py`)

### Summary
The `create_absorb_spend` function permits reward-absorb spends while a pool singleton is in the `LEAVING_POOL` (waiting room) state. Each absorb spend destroys the current singleton coin and creates a new one, resetting the `ASSERT_HEIGHT_RELATIVE` (`relative_lock_height`) timer that guards pool exit. A malicious pool operator — or any unprivileged third party who observes the chain — can repeatedly submit zero-fee absorb spends against incoming p2_singleton reward coins, permanently preventing the farmer from completing the exit.

### Finding Description

The pool-exit flow requires two on-chain state transitions:

1. `FARMING_TO_POOL` → `LEAVING_POOL` (waiting room): farmer signals intent to leave.
2. `LEAVING_POOL` → `SELF_POOLING`: farmer completes exit after `relative_lock_height` blocks have elapsed since the waiting-room coin was last confirmed.

`create_absorb_spend` in `chia/pools/pool_puzzles.py` explicitly handles both the `FARMING_TO_POOL` and `LEAVING_POOL` inner puzzles: [1](#0-0) 

When the singleton is in the waiting room, the absorb path (spend type `0`) is used. This spend destroys the current singleton coin and creates a new singleton coin at the current block height. Because `ASSERT_HEIGHT_RELATIVE` is evaluated against the coin's own `confirmed_block_index`, the timer restarts from the block in which the absorb was confirmed.

The wallet's `self_pool` method enforces the lock by reading `history[-1][0]` — the height of the **most recent** singleton spend — and adding `relative_lock_height`: [2](#0-1) 

Every absorb spend advances `history[-1][0]`, so the exit deadline is pushed forward by exactly `relative_lock_height` blocks each time.

The protocol itself confirms absorb spends are valid in the waiting room — the lifecycle test explicitly exercises this path and expects success: [3](#0-2) 

Absorb spends require **no BLS signature** when `fee=0` (the pool reward coin's puzzle enforces only that the singleton inner puzzle hash matches). Any party who knows the singleton's current state can construct and broadcast a valid absorb spend bundle.

### Impact Explanation

A farmer in `LEAVING_POOL` state continues farming with plots that still point to the pool's p2_singleton address, so new coinbase reward coins arrive at predictable, deterministic addresses. An attacker who submits an absorb spend for each incoming reward coin keeps the singleton coin perpetually "fresh," resetting the `relative_lock_height` countdown indefinitely. The farmer can never satisfy the height check in `self_pool` and is permanently locked in the pool. Their farming rewards continue to flow to the pool's `target_puzzle_hash` rather than to their own wallet.

This satisfies the **High** impact criterion: *permanent or long-lived inability for honest farmers to process valid pool actions under normal network assumptions*.

### Likelihood Explanation

The pool operator is the most motivated attacker: they receive the absorbed rewards and lose a member if the farmer exits. The attack cost is near-zero — absorb spends carry no mandatory fee and the attacker receives the reward coin's value in return. A `relative_lock_height` of 1440 blocks (~24 hours) means the attacker needs only one absorb per ~24 hours to maintain the lock. Any third party who monitors the chain for `LEAVING_POOL` singletons can also execute this attack at negligible cost.

### Recommendation

The waiting-room puzzle should not allow the singleton coin to be re-created (and thus the timer reset) via absorb spends. Two complementary mitigations:

1. **Record the entry height, not the last-spend height.** Store the block height at which the singleton first entered `LEAVING_POOL` state (e.g., in the puzzle or in a separate on-chain memo) and enforce `ASSERT_HEIGHT_ABSOLUTE` against that fixed height rather than `ASSERT_HEIGHT_RELATIVE` against the current coin.

2. **Disallow absorb spends in the waiting room.** Remove the `is_pool_waitingroom_inner_puzzle` branch from `create_absorb_spend` and reject absorb spends at the puzzle level when the singleton is in `LEAVING_POOL` state. Any unclaimed p2_singleton rewards accumulated during the waiting period can be swept after the exit completes.

### Proof of Concept

```
Block N:   Farmer calls self_pool() → singleton enters LEAVING_POOL state.
           relative_lock_height = 1440 blocks.
           Exit allowed at block N + 1440.

Block N+1: Farmer's plot wins a block → reward coin at p2_singleton address.
           Attacker calls create_absorb_spend(travel_coinsol, LEAVING_POOL_state, ...)
           → new singleton coin confirmed at block N+1.
           Exit now requires block N+1 + 1440 = N+1441.

Block N+1440: Farmer's plot wins again → new reward coin.
              Attacker absorbs → new singleton at N+1440.
              Exit now requires N+2880.

... (repeats indefinitely at zero net cost to attacker) ...
```

The `self_pool` check at `history[-1][0] + relative_lock_height` is always satisfied by the attacker's most recent absorb, so the farmer's exit transaction is perpetually rejected with `"Cannot self pool until height X"`. [4](#0-3) [5](#0-4)

def create_absorb_spend(
    last_coin_spend: CoinSpend,
    current_state: PoolState,
    launcher_coin: Coin,
    height: uint32,
    genesis_challenge: bytes32,
    delay_time: uint64,
    delay_ph: bytes32,
) -> list[CoinSpend]:
    inner_puzzle: Program = pool_state_to_inner_puzzle(
        current_state, launcher_coin.name(), genesis_challenge, delay_time, delay_ph
    )
    reward_amount: uint64 = calculate_pool_reward(height)
    if is_pool_member_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol: Program = Program.to([reward_amount, height])
    elif is_pool_waitingroom_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, destination_puzhash, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol = Program.to([0, reward_amount, height])
    else:
        raise ValueError
    # full sol = (parent_info, my_amount, inner_solution)
    coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(last_coin_spend)
    assert coin is not None

    if coin.parent_coin_info == launcher_coin.name():
        parent_info: Program = Program.to([launcher_coin.parent_coin_info, launcher_coin.amount])
    else:
        p = Program.from_bytes(bytes(last_coin_spend.puzzle_reveal))
        last_coin_spend_inner_puzzle: Program | None = get_inner_puzzle_from_puzzle(p)
        assert last_coin_spend_inner_puzzle is not None
        parent_info = Program.to(
            [
                last_coin_spend.coin.parent_coin_info,
                last_coin_spend_inner_puzzle.get_tree_hash(),
                last_coin_spend.coin.amount,
            ]
        )
    full_solution: SerializedProgram = SerializedProgram.to([parent_info, last_coin_spend.coin.amount, inner_sol])
    full_puzzle: SerializedProgram = create_full_puzzle(inner_puzzle, launcher_coin.name()).to_serialized()
    assert coin.puzzle_hash == full_puzzle.get_tree_hash()

    reward_parent: bytes32 = pool_parent_id(height, genesis_challenge)
    p2_singleton_puzzle = create_p2_singleton_puzzle(
        SINGLETON_MOD_HASH, launcher_coin.name(), delay_time, delay_ph
    ).to_serialized()
    reward_coin: Coin = Coin(reward_parent, p2_singleton_puzzle.get_tree_hash(), reward_amount)
    p2_singleton_solution = SerializedProgram.to([inner_puzzle.get_tree_hash(), reward_coin.name()])
    assert p2_singleton_puzzle.get_tree_hash() == reward_coin.puzzle_hash
    assert full_puzzle.get_tree_hash() == coin.puzzle_hash
    assert get_inner_puzzle_from_puzzle(Program.from_bytes(bytes(full_puzzle))) is not None

    coin_spends = [
        CoinSpend(coin, full_puzzle, full_solution),
        CoinSpend(reward_coin, p2_singleton_puzzle, p2_singleton_solution),
    ]
    return coin_spends
