Audit Report

## Title
`tokenFeeBps` Never Initialized in `addSupportedToken`, Causing Zero-Fee ERC20 Token Deposits - (File: `contracts/pools/RSETHPool.sol`)

## Summary
`RSETHPool.sol` declares a per-token fee mapping `tokenFeeBps` at line 88, but `addSupportedToken` (lines 637–656) never sets it. Because Solidity mappings default to zero, every ERC20 token deposit via `deposit(token, amount, referralId)` computes `fee = 0`, and the protocol collects no fee revenue from token deposits until the admin separately calls `setTokenFeeBps`. This contrasts with ETH deposits, where `feeBps` is explicitly initialized in `initialize()`.

## Finding Description
`RSETHPool.sol` declares the per-token fee mapping at line 88:
```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```
The token deposit fee is computed in `viewSwapRsETHAmountAndFee(uint256 amount, address token)` at lines 335–336:
```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```
`addSupportedToken` (lines 637–656) sets `supportedTokenOracle[token]`, `tokenBridge[token]`, and pushes to `supportedTokenList`, but never assigns `tokenFeeBps[token]`. The only way to set a non-zero fee is via the separate `setTokenFeeBps` function (lines 583–594), which requires `DEFAULT_ADMIN_ROLE` and is entirely optional with no enforcement. The public `deposit(address token, uint256 amount, string referralId)` function (lines 284–305) is callable by any unprivileged user and uses the zero-defaulted fee immediately after a token is added. By contrast, the ETH deposit path uses `feeBps` which is set at construction time via `initialize()` (line 248), demonstrating the protocol's intent to charge fees from the start.

## Impact Explanation
Every ERC20 token deposit through `deposit(token, amount, referralId)` is processed with `fee = 0`, meaning `feeEarnedInToken[token]` remains zero and the protocol collects no fee revenue from token deposits. Depositors receive the full rsETH equivalent with no fee deducted. This is a concrete instance of **Low — Contract fails to deliver promised returns (fee collection), but no user funds are lost**: the fee mechanism exists and is designed to work, but silently produces zero output for all tokens in their default state.

## Likelihood Explanation
This condition is the default for every token added via `addSupportedToken`. Any unprivileged depositor can call `deposit(token, amount, referralId)` immediately after a token is listed and pay zero fees indefinitely if the admin does not separately call `setTokenFeeBps`. No special attacker capability is required — normal deposit behavior is sufficient to trigger the zero-fee path. The window of exposure begins at token listing and ends only if/when the admin explicitly calls `setTokenFeeBps`.

## Recommendation
Add a `_feeBps` parameter to `addSupportedToken` and set `tokenFeeBps[token]` atomically during token registration:
```solidity
function addSupportedToken(address token, address oracle, address bridge, uint256 _feeBps)
    external onlyRole(TIMELOCK_ROLE)
{
    // ... existing checks ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    emit AddSupportedToken(token, oracle, bridge);
}
```
This mirrors how `feeBps` is set at initialization for ETH deposits and ensures no token can be listed without an explicit fee assignment.

## Proof of Concept
1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0` (Solidity default).
2. Any user calls `deposit(wstETH, 1e18, "ref")` (lines 284–305).
3. `viewSwapRsETHAmountAndFee(1e18, wstETH)` (lines 326–347) computes:
   - `feeBpsForToken = tokenFeeBps[wstETH]` → `0`
   - `fee = 1e18 * 0 / 10_000` → `0`
   - `amountAfterFee = 1e18`
4. `feeEarnedInToken[wstETH] += 0` — protocol collects nothing.
5. User receives full rsETH with zero fee deducted.

**Foundry test plan:**
```solidity
function test_zeroFeeOnTokenDeposit() public {
    // Setup: add wstETH as supported token (no feeBps set)
    vm.prank(timelockAdmin);
    pool.addSupportedToken(address(wstETH), address(oracle), address(bridge));
    
    // Verify tokenFeeBps is zero
    assertEq(pool.tokenFeeBps(address(wstETH)), 0);
    
    // User deposits 1e18 wstETH
    vm.startPrank(user);
    wstETH.approve(address(pool), 1e18);
    pool.deposit(address(wstETH), 1e18, "ref");
    vm.stopPrank();
    
    // Fee earned is zero
    assertEq(pool.feeEarnedInToken(address(wstETH)), 0);
}
