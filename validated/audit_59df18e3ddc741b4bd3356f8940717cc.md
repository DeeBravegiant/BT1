Audit Report

## Title
Permissionless `updateRate` with No Staleness Guard Enables Block-Stuffing to Exploit Stale Cross-Chain rsETH Rate — (`contracts/cross-chain/RSETHMultiChainRateProvider.sol`)

## Summary

`MultiChainRateProvider.updateRate()` is a permissionless `external payable` function with no access control. `CrossChainRateReceiver.getRate()` returns the stored rate unconditionally with no maximum-age check. L2 pools (`RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`) price every swap directly against this stored rate. An attacker who stuffs Ethereum mainnet blocks during an `rsETHPrice` increase window can prevent rate propagation to L2, then extract excess rsETH from the pool at the stale-low rate, constituting theft of unclaimed yield from existing rsETH holders.

## Finding Description

**Rate propagation path (L1 → L2):**

`RSETHMultiChainRateProvider.getLatestRate()` reads `ILRTOracle(rsETHPriceOracle).rsETHPrice()` live from L1. [1](#0-0) 

`MultiChainRateProvider.updateRate()` is fully permissionless — no `onlyOwner`, no role check, no caller whitelist. [2](#0-1) 

It sends the encoded rate via LayerZero to every registered `RateReceiver`. [3](#0-2) 

**Rate storage on L2 (no expiry):**

`CrossChainRateReceiver.lzReceive()` writes `rate` and `lastUpdated`, but `getRate()` returns `rate` unconditionally — `lastUpdated` is stored and never read by any pricing path. [4](#0-3) 

**Pool pricing (no staleness guard):**

`RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee()` calls `getRate()` directly and divides by it. A stale-low rate yields more rsETH per ETH than the current fair value. [5](#0-4) 

`RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee()` is structurally identical. [6](#0-5) 

**The gap:** `lastUpdated` is present in both provider and receiver but is **never consulted** by any swap pricing function. No circuit-breaker reverts or discounts swaps when the rate is older than any threshold. [7](#0-6) 

## Impact Explanation

When `rsETHPrice` rises on L1 (e.g., after EigenLayer reward accrual updates `LRTOracle`) and the L2 receiver still holds the pre-increase rate `R_old < R_new`, every depositor receives:

```
rsETHAmount = amountAfterFee * 1e18 / R_old   >   amountAfterFee * 1e18 / R_new
```

The excess rsETH is funded by the pool's pre-loaded rsETH inventory, which represents yield belonging to existing rsETH holders. Arbitrageurs extract this as profit — **High. Theft of unclaimed yield**. The enabling mechanism is **Low. Block stuffing**, both of which are explicitly listed in the allowed impact scope.

## Likelihood Explanation

`updateRate()` is permissionless, so the attacker must prevent all callers from landing a transaction on Ethereum mainnet via block stuffing (filling consecutive blocks with high-gas-price filler transactions). This is expensive (~30 M gas × base fee per block), but:

- The attack window only needs to span the blocks between the `rsETHPrice` oracle update and the next successful `updateRate()` call.
- The attacker can monitor the mempool for the oracle update transaction, then immediately begin stuffing.
- Profit scales with pool liquidity and the magnitude of the price jump; a large EigenLayer reward event could make the economics favorable.
- No admin compromise, no private key leak, and no governance capture is required.

Likelihood is **Low** due to the cost of mainnet block stuffing, but impact is **High** (theft of unclaimed yield).

## Recommendation

1. **Add a staleness guard in `CrossChainRateReceiver.getRate()`**: revert (or return a sentinel) if `block.timestamp - lastUpdated > MAX_RATE_AGE`. This prevents pools from pricing swaps against an arbitrarily old rate regardless of the cause of the delay.

2. **Add the same guard in the L2 pools**: `RSETHPoolNoWrapper` and `RSETHPoolV3ExternalBridge` should independently validate rate freshness before executing a swap, so a lagging oracle cannot silently drain the pool.

3. **Consider a keeper with EIP-1559 tip floor**: ensure `updateRate()` is called with a competitive `maxPriorityFeePerGas` so that block stuffing becomes prohibitively expensive relative to any realistic arbitrage profit.

## Proof of Concept

Foundry fork test (fork of Ethereum mainnet + target L2):

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

function testBlockStuffingStaleRate() public {
    uint256 oldRate = 1.05e18;  // pre-reward rate stored in receiver
    uint256 newRate = 1.08e18;  // post-reward rate now live on L1 oracle

    // 1. vm.store(lrtOracle, rsETHPriceSlot, bytes32(newRate))
    //    — simulates EigenLayer reward accrual updating LRTOracle
    // 2. vm.roll(block.number + N) without calling updateRate()
    //    — simulates block stuffing; receiver still holds oldRate
    // 3. Call pool.viewSwapRsETHAmountAndFee(1 ether)
    //    — pool prices swap using stale oldRate

    (uint256 rsETHAmount,) = pool.viewSwapRsETHAmountAndFee(1 ether);
    uint256 trueAmount = 1 ether * 1e18 / newRate;

    // Excess ~0.026e18 rsETH per ETH deposited extracted from pool inventory
    assertGt(rsETHAmount, trueAmount, "stale rate yields excess rsETH");
}
```

The test requires no privileged access: it only needs the oracle storage slot updated (simulating normal reward accrual) and `updateRate()` withheld (simulating block stuffing). The assertion passes on unmodified code whenever `newRate > oldRate`.
