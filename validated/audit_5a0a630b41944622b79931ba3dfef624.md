Audit Report

## Title
Permissionless `sendFunds()` Enables Atomic Deposit-Before-Reward-Flush to Steal Unclaimed Yield - (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` carries no access-control modifier, allowing any caller to move accumulated MEV/execution-layer rewards into the deposit pool at will. Because `LRTOracle._getTotalEthInProtocol()` never reads `FeeReceiver.balance` directly, an attacker can deposit rsETH at the artificially low stored price, then atomically flush the rewards and update the price, extracting yield that belongs to pre-existing rsETH holders.

## Finding Description

`getETHDistributionData()` explicitly excludes `FeeReceiver` balance from TVL accounting: [1](#0-0) 

The function reads `address(this).balance`, NDC balances, EigenLayer pod shares, unstaking vault balance, and converter value — but never `FeeReceiver.balance`: [2](#0-1) 

`_getTotalEthInProtocol()` in `LRTOracle` calls `getTotalAssetDeposits()` → `getAssetDistributionData()` → `getETHDistributionData()`, so the FeeReceiver balance is absent from the price computation: [3](#0-2) 

The rsETH price is computed from this incomplete TVL: [4](#0-3) 

`FeeReceiver.sendFunds()` has no access-control modifier — any EOA or contract can call it: [5](#0-4) 

`updateRSETHPrice()` is similarly public: [6](#0-5) 

Deposit minting uses the **stored** `rsETHPrice` (set at the last `updateRSETHPrice()` call), which was computed without FeeReceiver balance: [7](#0-6) 

**Exploit sequence (single transaction or two sequential txs):**
1. FeeReceiver accumulates N ETH in MEV rewards (not yet flushed).
2. Attacker calls `depositETH{value: X}(...)` — minting is priced at the stale, understated `rsETHPrice`.
3. Attacker calls `FeeReceiver.sendFunds()` — N ETH moves to the deposit pool, increasing TVL.
4. Attacker calls `LRTOracle.updateRSETHPrice()` — price rises to reflect the new TVL.
5. Attacker's rsETH is now worth more than they paid; the gain comes from diluting existing holders' accrued yield.

The `pricePercentageLimit` guard at L252–266 only blocks the call if the price increase exceeds the configured threshold **and** `pricePercentageLimit > 0`. For typical MEV reward sizes relative to total TVL, the per-update price delta is small and will not breach the threshold. If `pricePercentageLimit == 0` the guard is entirely disabled. [8](#0-7) 

## Impact Explanation

This is **theft of unclaimed yield** (High). Rewards that have accrued to existing rsETH holders but not yet been flushed into TVL are captured by the attacker. The attacker mints rsETH at a price that does not reflect those rewards, then forces the price correction, diluting every pre-existing holder's share of the accumulated yield. The gain scales linearly with the size of the FeeReceiver balance at the time of the attack.

## Likelihood Explanation

- MEV and execution-layer rewards flow into `FeeReceiver` continuously; a non-zero balance exists between every validator proposal and the next operator flush.
- Both `sendFunds()` and `updateRSETHPrice()` are fully permissionless — no role, key, or governance action is required.
- The attack requires only monitoring `FeeReceiver.balance` on-chain and submitting two transactions (or one via a helper contract).
- It is repeatable every reward cycle with no cooldown.

**Likelihood: Medium** (requires mempool monitoring and acting before operators flush, but is otherwise unconstrained).

## Recommendation

Either:
1. **Include `FeeReceiver.balance` in `getETHDistributionData()`** so TVL always reflects true backing regardless of flush timing. Store the `FeeReceiver` address in `LRTConfig` and add `ethLyingInFeeReceiver = IFeeReceiver(feeReceiver).balance` to the sum returned by `getETHDistributionData()`.
2. **Restrict `sendFunds()` to a privileged role** (e.g., `MANAGER`) and require operators to call it atomically before every `updateRSETHPrice()` invocation, eliminating the permissionless flush window.

Option 1 is preferred as it closes the gap structurally without relying on operational discipline.

## Proof of Concept

**Minimal Foundry fork test outline:**

```solidity
// Setup: fork mainnet, ensure FeeReceiver has accumulated 10 ETH
uint256 feeReceiverBal = address(feeReceiver).balance; // e.g. 10 ETH

uint256 priceBefore = lrtOracle.rsETHPrice();

// Step 1: attacker deposits 100 ETH at understated price
vm.prank(attacker);
lrtDepositPool.depositETH{value: 100 ether}(0, "");
uint256 attackerShares = rsETH.balanceOf(attacker);

// Step 2: flush FeeReceiver
vm.prank(attacker);
feeReceiver.sendFunds();

// Step 3: update price
vm.prank(attacker);
lrtOracle.updateRSETHPrice();

uint256 priceAfter = lrtOracle.rsETHPrice();
assert(priceAfter > priceBefore);

// Step 4: attacker's rsETH is worth more than deposited
uint256 attackerValueAfter = attackerShares * priceAfter / 1e18;
assert(attackerValueAfter > 100 ether); // profit extracted from existing holders
```

The test demonstrates that `attackerValueAfter > 100 ether` while the total ETH added to the pool by the attacker was exactly 100 ETH, with the surplus coming from pre-existing holders' unclaimed MEV yield.
