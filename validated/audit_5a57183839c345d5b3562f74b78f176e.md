### Title
Malicious Connected Harvester Can Unconditionally Redirect Farmer Block Reward to Attacker-Controlled Address — (`File: chia/farmer/farmer_api.py`)

### Summary
A connected harvester (e.g., a third-party harvester under CHIP-22) can set `farmer_reward_address_override` in a `RespondSignatures` message to any arbitrary address it controls. The farmer's `_process_respond_signatures` function accepts this override unconditionally — with zero fee-quality enforcement — and uses it as the `farmer_reward_puzzle_hash` in the `DeclareProofOfSpace` message sent to the full node. Every block won while the malicious harvester is connected results in the 1/8 farmer block reward being paid to the attacker's address instead of the farmer's configured address.

### Finding Description

CHIP-22 introduced `farmer_reward_address_override` to allow third-party harvesters to take a portion of the farmer reward as a service fee, gated by a `fee_quality` / `applied_fee_threshold` convention. The farmer is supposed to enforce this threshold and only honour overrides that pass the quality check.

**Path 1 — via `NewProofOfSpace`:**

In `new_proof_of_space` (farmer_api.py line 128-129), when the harvester sends `farmer_reward_address_override`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`. That function checks the fee quality and emits a log warning if the threshold is not met — but it **does not return an error, does not reject the proof, and does not prevent the block-making flow from continuing**. [1](#0-0) [2](#0-1) 

**Path 2 — via `RespondSignatures` (no warning at all):**

In `_process_respond_signatures` (farmer_api.py lines 916-918), the farmer reads `response.farmer_reward_address_override` from the `RespondSignatures` message and unconditionally substitutes it for `self.farmer.farmer_target`. There is **no fee-quality check, no logging, and no rejection** at this point:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [3](#0-2) 

The resulting `DeclareProofOfSpace` carries the attacker's `farmer_reward_puzzle_hash`: [4](#0-3) 

A malicious harvester can exploit Path 2 without even triggering the Path 1 warning: send `NewProofOfSpace` with `farmer_reward_address_override=None` (no log), then send `RespondSignatures` with `farmer_reward_address_override` set to the attacker's address. The farmer silently redirects the reward.

The `RespondSignatures` protocol message definition confirms the field is freely settable by the harvester: [5](#0-4) 

### Impact Explanation

Every block won while the malicious harvester is connected results in the farmer's 1/8 block reward (currently 0.25 XCH per block) being paid to the attacker's address. The farmer receives nothing for those blocks. This is a direct, on-chain, irreversible theft of XCH via payout redirection — matching the **High** impact category: *"Bypass of wallet, daemon, keychain, RPC, pool, or Data Layer authorization that enables unauthorized signing, key use, coin control, payout redirection."*

### Likelihood Explanation

The attacker must be a connected harvester. Under CHIP-22, third-party harvester services (e.g., DrPlotter) are an explicitly supported and growing use case. A farmer operator who connects to a third-party harvester service is exposed to this attack for every block that harvester finds. The attacker needs no keys, no admin access, and no cryptographic break — only a valid TLS connection as a harvester peer, which is the normal operational state for third-party harvesters.

### Recommendation

In `_process_respond_signatures`, before accepting `response.farmer_reward_address_override`, perform the same fee-quality check that `notify_farmer_reward_taken_by_harvester_as_fee` performs, and **reject** (return `None`) if the quality threshold is not met or if `fee_info` was not provided in the original `NewProofOfSpace`. Additionally, store the override and `fee_info` from `NewProofOfSpace` in the farmer's in-memory proof-of-space cache and validate in `_process_respond_signatures` that the override in `RespondSignatures` matches the one declared in `NewProofOfSpace` and that the fee quality passed.

### Proof of Concept

1. Attacker operates a third-party harvester service and convinces (or is already) connected to a farmer.
2. When the harvester finds a valid proof of space, it sends `NewProofOfSpace` with `farmer_reward_address_override=None` and `fee_info=None` (no warning triggered on the farmer).
3. The farmer sends `RequestSignatures` back to the harvester.
4. The harvester responds with `RespondSignatures` where `farmer_reward_address_override` is set to the attacker's own puzzle hash.
5. `_process_respond_signatures` (farmer_api.py line 917-918) accepts the override without any check.
6. `DeclareProofOfSpace` is sent to the full node with the attacker's puzzle hash as `farmer_reward_puzzle_hash`.
7. The full node creates the farmer reward coin paying to the attacker's address.
8. The farmer's configured `farmer_target` receives nothing for that block.
