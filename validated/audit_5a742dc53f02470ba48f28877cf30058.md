### Title
Single Attested Participant Can Deliver Forged CKD Response for `AppPublicKey` Variant Without Threshold Authorization - (File: `crates/contract/src/lib.rs`)

### Summary
The `respond_ckd` function in the MPC contract enforces a cryptographic output check only for the `AppPublicKeyPV` variant of a CKD request. For the `AppPublicKey` (legacy, single G1 point) variant, **no verification of the response is performed**. This allows any single attested participant — strictly below the signing threshold — to call `respond_ckd` with an arbitrary forged `CKDResponse` and have it accepted and delivered to the waiting user, bypassing the threshold authorization requirement that is the entire security premise of the MPC network.

### Finding Description

In `respond_ckd` at [1](#0-0) , the contract branches on the `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which performs a BLS12-381 pairing check verifying `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`. [2](#0-1) 

For `AppPublicKey`, the arm is a no-op. Any `(big_y, big_c)` pair is accepted unconditionally.

After this branch, `resolve_yields_for` is called, which **removes the pending request from the map** and resumes all queued yield promises with the (unverified) response bytes. [3](#0-2) 

The state-update analogy to the reported bug is exact: just as `redeemObligation` failed to mark the position as redeemed (allowing repeated extraction), `respond_ckd` fails to enforce the invariant that the delivered CKD output is cryptographically bound to the MPC network's key material. The pending-request entry is consumed on the first call, but the content of that single call is never validated, so the attacker's forged payload is irrevocably delivered.

The `AppPublicKey` variant is not deprecated; it is the legacy single-G1-point form still accepted by the contract and exercised in production e2e tests. [4](#0-3) 

The caller gate `assert_caller_is_attested_participant_and_protocol_active` requires only that the signer is a current attested participant. [5](#0-4)  It does not require threshold agreement on the response content. A single participant satisfies this gate.

### Impact Explanation

A single attested MPC participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary `CKDResponse{big_y: ..., big_c: ...}` for any pending `AppPublicKey` CKD request. The contract accepts the call, removes the pending entry, and resumes the user's yield promise with the forged payload. The user's NEAR transaction receives a `(big_y, big_c)` pair that is not the output of the threshold CKD protocol and is not bound to the MPC network's BLS key. This constitutes **confidential key derivation output delivered without the required threshold participant authorization** — the core security guarantee of the CKD service is violated by a single Byzantine participant.

Mapped impact: **Critical — confidential key derivation output without the required participant authorization.**

### Likelihood Explanation

The attack requires only that the adversary is an attested participant in the current epoch. No cryptographic break, no threshold collusion, and no TEE bypass is needed. The attacker calls `respond_ckd` directly from their NEAR account with hand-crafted `big_y`/`big_c` values. Because `resolve_yields_for` removes the map entry on the first successful call, the race is won by whichever participant submits first; a malicious participant who monitors the mempool for `request_app_private_key` transactions can front-run honest nodes. The `AppPublicKey` variant is the legacy default path and is actively used.

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` arm. Because `AppPublicKey` provides only `pk1` (a G1 point) without a paired G2 point, the existing `ckd_output_check` cannot be applied directly. The options are:

1. **Require callers to use `AppPublicKeyPV`** and reject `AppPublicKey` requests at the `request_app_private_key` entry point (deprecate the unverifiable variant).
2. **Collect threshold-many identical responses** before resolving the yield, analogous to how `vote_pk` and `vote_reshared` require all-participant agreement before state transitions. [6](#0-5) 
3. **Add a G2 companion key** to `AppPublicKey` requests so the existing pairing check can be applied, effectively migrating callers to `AppPublicKeyPV` semantics.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
