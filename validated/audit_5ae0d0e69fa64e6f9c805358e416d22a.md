Audit Report

## Title
Reward Tokens Permanently Frozen When `totalKernelStaked` Drops to Zero Mid-Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` implements a Synthetix-style staking rewards mechanism that halts reward accumulation when `totalKernelStaked` reaches zero. Because `initiateWithdrawal` immediately decrements `totalKernelStaked` (before the withdrawal delay expires), the last staker's withdrawal can bring the total to zero mid-period, permanently stranding all reward tokens that would have accrued from that point to `finishAt`. No on-chain recovery path exists for these tokens.

## Finding Description
`rewardPerToken()` at L408-414 returns the frozen `rewardPerTokenStored` whenever `totalKernelStaked == 0`, halting all accumulation. The `updateReward` modifier at L232-242 snaps `updatedAt = lastTimeRewardApplicable()` on every call, so when the last staker calls `initiateWithdrawal` (L320-327), `updatedAt` is set to the current timestamp and `totalKernelStaked` is set to zero in the same transaction. All reward tokens that would have accrued from that moment to `finishAt` are never attributed to any user.

When `notifyRewardAmount` is eventually called again (after someone re-stakes), the guard at L570 (`if (totalKernelStaked == 0) revert NoStakedTokens()`) prevents calling it while supply is zero. Once supply is restored and `block.timestamp >= finishAt`, the calculation at L579-580 uses only `receivedAmount / duration` — the stranded excess balance sitting in the contract is not rolled in. The contract has no `Recoverable` inheritance and no admin function to extract stuck `rewardsToken` balance, confirming the tokens are permanently frozen.

The contract's own NatSpec at L18-22 explicitly acknowledges this risk and describes an off-chain operational mitigation (ensuring tokens are always staked), but provides no on-chain safeguard against mid-period zero-supply.

## Impact Explanation
Reward tokens accruing during any zero-supply interval within an active reward period are permanently frozen inside `KernelDepositPool` with no on-chain extraction path. This matches the allowed impact: **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
Low. Requires `totalKernelStaked` to reach zero during an active reward window, which demands all stakers to call `initiateWithdrawal` concurrently. This is realistic in a low-participation pool, during a market stress event, or when a single large staker holds the entire stake. The trigger is fully within the control of unprivileged stakers via the public `initiateWithdrawal` function.

## Recommendation
Add an admin-only rescue function that transfers only the excess `rewardsToken` balance — defined as `rewardsToken.balanceOf(address(this))` minus the sum of all legitimately owed `rewards[user]` entries (tracked via an aggregate `totalPendingRewards` counter). This prevents recovery of user-owed rewards while allowing stranded tokens to be reclaimed. Alternatively, modify `notifyRewardAmount` to detect and roll in any excess balance beyond what the current `rewardRate` accounts for.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `totalKernelStaked = 100e18`. `rewardRate = 1_000e18 / duration`, `finishAt = block.timestamp + duration`.
2. At the halfway point, the sole staker calls `initiateWithdrawal(100e18)`. The `updateReward` modifier snaps `updatedAt = block.timestamp` and `totalKernelStaked` becomes `0`.
3. For the remaining half of the period, `rewardPerToken()` returns the frozen `rewardPerTokenStored`. Approximately `500e18` reward tokens accrue to no address.
4. After `finishAt`, a new user stakes 1 wei. Admin calls `notifyRewardAmount(newAmount)`. Since `block.timestamp >= finishAt`, `rewardRate = newAmount / duration` — the `500e18` stranded tokens are excluded.
5. The `500e18` tokens remain in the contract permanently. No function exists to extract them.

**Foundry test sketch:**
```solidity
function test_rewardsStrandedOnZeroSupply() public {
    // Setup: stake, start reward period
    pool.stake(100e18);
    rewardToken.approve(address(pool), 1000e18);
    pool.notifyRewardAmount(1000e18);

    // Fast-forward to midpoint
    vm.warp(block.timestamp + duration / 2);

    // Last staker withdraws — totalKernelStaked → 0
    pool.initiateWithdrawal(100e18);
    assertEq(pool.totalKernelStaked(), 0);

    // Fast-forward past finishAt
    vm.warp(pool.finishAt() + 1);

    // Re-stake and start new period
    pool.stake(1);
    rewardToken.approve(address(pool), 100e18);
    pool.notifyRewardAmount(100e18);

    // ~500e18 tokens remain stranded — no function to recover them
    uint256 stranded = rewardToken.balanceOf(address(pool)) - pool.rewards(address(this));
    assertGt(stranded, 400e18); // approximately 500e18 stranded
}
```
