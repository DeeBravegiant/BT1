Audit Report

## Title
Unguarded `msg.value` on Non-WETH Swap and Liquidity Functions Allows Accidental ETH to Be Stolen via `refundETH()` — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
All four swap entry-points in `MetricOmmSimpleRouter` and all four liquidity entry-points in `MetricOmmPoolLiquidityAdder` are declared `payable` but contain no `msg.value == 0` guard for non-WETH token paths. Any ETH sent with such a call is silently retained in the contract because `pay()` ignores `address(this).balance` when `token != WETH`. The permissionless `refundETH()` then allows any caller to drain the full contract ETH balance, causing direct, irrecoverable loss of the victim's ETH.

## Finding Description
All eight entry-points are confirmed `payable` in the production code:

- `MetricOmmSimpleRouter`: `exactInputSingle` (L67), `exactInput` (L92), `exactOutputSingle` (L130), `exactOutput` (L154) — none contain a `msg.value == 0` check.
- `MetricOmmPoolLiquidityAdder`: both overloads of `addLiquidityExactShares` (L56, L71) and both overloads of `addLiquidityWeighted` (L88, L123) — none contain a `msg.value == 0` check.

The internal `pay()` function in `PeripheryPayments.sol` (L69–88) branches on `token`:
- `payer == address(this)` → ERC-20 `safeTransfer`
- `token == WETH` → wraps `address(this).balance` first, then pulls remainder via `safeTransferFrom`
- `else` → plain `safeTransferFrom`; **`address(this).balance` is never touched**

When `token != WETH`, any ETH attached to the call is silently retained. The `receive()` guard at L32–34 (`if (msg.sender != WETH) revert NotWETH()`) only applies to plain ETH transfers with no calldata; it does **not** block ETH arriving via a `payable` function call.

`refundETH()` (L58–63) has no access control and forwards `address(this).balance` to `msg.sender`:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

A secondary path exists: if ETH is sitting in the router when a different user performs a WETH swap, `pay()` (L74–77) consumes that ETH first to satisfy the second user's WETH obligation, transferring the victim's ETH to the pool on behalf of a third party.

The interface documentation at `IMetricOmmSimpleRouter.sol` L11 explicitly states: `"ERC-20 routes only. No native ETH, WETH wrap/unwrap…"`, confirming native ETH is not an intended input for these paths, making the `payable` modifier an unintended exposure.

## Impact Explanation
Direct loss of user principal. A user who mistakenly attaches ETH to a non-WETH swap or liquidity call permanently loses that ETH: it is either stolen by any MEV bot or observer calling `refundETH()` in the same or a subsequent block, or misappropriated by a subsequent WETH swap that consumes it to pay another user's obligation. The victim has no on-chain recourse. This meets the Sherlock threshold for direct loss of user funds.

## Likelihood Explanation
All affected functions are `payable`, so wallets, aggregators, and multicall bundles can forward ETH without any EVM-level rejection. Users frequently confuse native ETH with WETH, especially in multicall bundles mixing `unwrapWETH9`/`refundETH` with swap calls. MEV bots routinely monitor for unprotected `refundETH`-style helpers; exploitation requires no special privilege, no setup, and can occur in the same block as the victim's transaction.

## Recommendation
Add a `msg.value == 0` guard at the top of every swap and liquidity entry-point that does not intentionally accept native ETH:

```solidity
require(msg.value == 0, "Router: unexpected ETH");
```

Apply this to `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput` in `MetricOmmSimpleRouter`, and both overloads of `addLiquidityExactShares` and `addLiquidityWeighted` in `MetricOmmPoolLiquidityAdder`. Alternatively, remove the `payable` modifier from these functions entirely, since the interface explicitly scopes them to ERC-20 only.

## Proof of Concept
```solidity
// Step 1 – victim calls exactInputSingle with 1 ETH attached (USDC→DAI, neither is WETH)
router.exactInputSingle{value: 1 ether}(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),   // USDC/DAI pool
        tokenIn:          USDC,
        tokenOut:         DAI,
        zeroForOne:       true,
        amountIn:         1_000e6,
        amountOutMinimum: 990e18,
        recipient:        victim,
        deadline:         block.timestamp + 60,
        priceLimitX64:    0,
        extensionData:    ""
    })
);
// Swap succeeds. pay() calls safeTransferFrom(USDC, victim, pool, 1000e6).
// 1 ETH is silently retained in the router.

// Step 2 – attacker calls refundETH in the same or next block
router.refundETH(); // attacker receives 1 ETH; victim has no recourse
```
