### Title
Stale `CertState` in Dijkstra Batch-Transaction Value-Conservation Check Causes Permanent Deposit-Pot Inflation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`)

---

### Summary

In the Dijkstra era's `dijkstraLedgerTransition`, the original `CertState` (captured before any sub-transaction is processed) is passed to the `UTXOW` rule. The value-conservation check and the deposit-accounting update inside `UTXOW` both use this stale `CertState` to compute deposits and refunds for the **entire batch** (top-level tx + all sub-transactions). Because sub-transactions are processed first and can register pools or stake credentials, the stale `CertState` causes the deposit pot (`utxosDeposited`) to be inflated beyond the sum of obligations actually tracked in `CertState`, permanently locking ADA.

---

### Finding Description

**Execution flow in `dijkstraLedgerTransition`:**

1. `originalUtxo` and `lsCertState ledgerState` (the original `CertState`) are captured.
2. `SUBLEDGERS` processes all sub-transactions, producing `certStateAfterSubLedgers` and `utxoStateAfterSubLedgers`. Sub-transactions can register pools, stake credentials, or DReps, mutating `CertState`.
3. `ENTITIES` processes the top-level transaction's certificates using `certStateAfterSubLedgers`, producing `certStateFinal`.
4. `UTXOW` is invoked with `DijkstraUtxoEnv slot pp (lsCertState ledgerState) originalUtxo` — the **original, pre-sub-transaction** `CertState`. [1](#0-0) 

Inside `dijkstraUtxoTransition`, the value-conservation check is:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [2](#0-1) 

`certState` here is the original, stale one. `consumed` calls `conwayConsumed`, which calls `getConsumedValue = getConsumedDijkstraValue`, which aggregates consumed value across **both the top-level tx and all sub-transactions**: [3](#0-2) 

Similarly, `produced` calls `getProducedValue = getProducedDijkstraValue`, which aggregates produced value across the top-level tx and all sub-transactions, using `isRegPoolId` derived from the original `CertState`: [4](#0-3) 

After the check passes, `Babbage.updateUTxOStateByTxValidity` is called with the same stale `certState`: [5](#0-4) 

This calls `updateUTxOStateNoFees`, which uses `certsTotalDepositsTxBody pp certState txBody` to compute the deposit change and updates `utxosDeposited`: [6](#0-5) 

**Concrete exploit — pool registration double-deposit:**

| Step | Action | `utxosDeposited` | Pool deposit in `CertState` |
|---|---|---|---|
| Initial | Pool P not registered | D | 0 |
| SUBLEDGERS: sub-tx registers P | `updateUTxOStateNoFees` adds `D_pool` | D + D_pool | D_pool |
| ENTITIES: top-level tx registers P | Treated as UPDATE (P already registered) | D + D_pool | D_pool (unchanged) |
| UTXOW: `updateUTxOStateByTxValidity` | Original certState says P not registered → adds `D_pool` | **D + 2·D_pool** | D_pool |

After pool P retires, `POOLREAP` refunds only `D_pool` (from `spsDeposit`): [7](#0-6) 

The extra `D_pool` in `utxosDeposited` is never refunded. The invariant `potEqualsObligation` is permanently violated: [8](#0-7) 

The same pattern applies to stake credential registrations (if a sub-tx registers a credential that the top-level tx also registers) and DRep registrations.

---

### Impact Explanation

**High — Permanent freezing of funds where recovery requires a hard fork.**

A transaction author crafts a batch transaction where a sub-transaction registers pool P and the top-level transaction also registers pool P. The deposit pot is inflated by `D_pool` (currently 500 ADA on mainnet). This ADA is permanently locked in `utxosDeposited` with no mechanism to recover it short of a hard fork. The attacker spends `D_pool` to cause the freeze (the value-conservation check requires them to supply the extra `D_pool` in inputs), making this a deliberate griefing attack against the deposit accounting invariant.

---

### Likelihood Explanation

**Medium.** The Dijkstra era (protocol version 12) is implemented in this codebase and is the next planned era. Any unprivileged transaction author can craft the described batch transaction. No special privileges, keys, or governance majority are required. The only cost to the attacker is the `D_pool` (or `D_key`) they must supply to satisfy the value-conservation check, which they permanently lose.

---

### Recommendation

In `dijkstraLedgerTransition`, pass `certStateFinal` (the post-ENTITIES certState) to `UTXOW` instead of the original `lsCertState ledgerState`. This ensures that both the value-conservation check and `updateUTxOStateByTxValidity` use the certState that reflects all sub-transaction and top-level certificate effects, keeping `utxosDeposited` in sync with the actual obligations tracked in `CertState`.

Alternatively, the deposit accounting in `updateUTxOStateNoFees` for the top-level transaction should use `certStateFinal` to correctly classify pool registrations as new (deposit required) vs. updates (no deposit).

---

### Proof of Concept

```
Batch transaction (Dijkstra era):
  sub-transactions:
    subTx1:
      inputs:  [utxo_A]          -- contains V_A lovelace
      outputs: [addr_X: V_A - D_pool]
      certs:   [RegPoolTxCert poolP params1]
  top-level:
    inputs:  [utxo_B]            -- contains V_B lovelace
    outputs: [addr_Y: V_B - D_pool - fee]
    fee:     fee
    certs:   [RegPoolTxCert poolP params2]  -- same pool, different params
```

**Trace:**
1. SUBLEDGERS processes `subTx1`: pool P registered, `utxosDeposited += D_pool`.
2. ENTITIES processes top-level certs: pool P already registered → UPDATE, `spsDeposit` unchanged.
3. UTXOW value-conservation check (original certState, P not registered):
   - `produced` = `(V_B - D_pool - fee) + fee + D_pool` (top-level) + `(V_A - D_pool) + D_pool` (sub-tx) = `V_A + V_B` ✓
   - `consumed` = `V_A + V_B` ✓ → check passes.
4. `updateUTxOStateByTxValidity` (original certState, P not registered): `utxosDeposited += D_pool`.
5. Final state: `utxosDeposited` increased by `2·D_pool`, but `CertState` only tracks `D_pool` for pool P.
6. Pool P retires: `POOLREAP` refunds `D_pool`. Extra `D_pool` in `utxosDeposited` is permanently locked.
