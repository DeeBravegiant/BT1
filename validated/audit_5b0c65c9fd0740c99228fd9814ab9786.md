### Title
Pool Wallet State Corruption via Unvalidated Peer-Provided Solution in `apply_state_transition` - (File: chia/pools/pool_wallet.py)

### Summary

`PoolWallet.apply_state_transition` accepts a `CoinSpend` fetched from a peer without validating the spend's **solution** against the blockchain. A malicious peer can supply the correct puzzle (matching the on-chain puzzle hash) paired with a crafted fake solution that encodes an attacker-controlled `PoolState`. The wallet stores this spend, derives a corrupted pool state from it, and persists the corrupted state to the local pool config — redirecting the farmer's pool membership and payout target.

### Finding Description

**Step 1 — Coin spend fetch with no solution validation**

In `_add_coin_states`, when a `POOLING_WALLET` singleton coin is observed as spent, the wallet fetches the coin spend from the connected peer: [1](#0-0) 

`fetch_coin_spend` validates only the puzzle hash and coin name — the **solution is never validated** against the blockchain: [2](#0-1) 

A malicious peer can return the correct puzzle (satisfying the puzzle-hash check) paired with an arbitrary fake solution.

**Step 2 — `apply_state_transition` performs no pool-state validation**

The docstring of `apply_state_transition` states "All validation should be done here," yet the only guard is a coin-name equality check: [3](#0-2) 

After the coin-name check passes, the spend (with the fake solution) is unconditionally stored: [4](#0-3) 

No validation of the decoded `PoolState`, `owner_pubkey`, or `target_puzzle_hash` is performed.

**Step 3 — Corrupted state propagates to pool config**

`apply_state_transition` immediately calls `update_pool_config`, which reads the pool state from the stored (fake) spend and writes it to the persistent pool config file: [5](#0-4) [6](#0-5) 

The fields written include `target_puzzle_hash`, `pool_url`, and `owner_public_key` — all sourced from the attacker-controlled solution.

**Step 4 — Farmer uses corrupted config**

The farmer reads `PoolingShareState` to communicate with the pool. The `target_puzzle_hash` field is what the farmer presents to the pool as the payout address during pool protocol registration: [7](#0-6) 

### Impact Explanation

An attacker who controls a peer the wallet connects to can corrupt the wallet's pool membership state:

- **`target_puzzle_hash` overwritten** → farmer registers with the pool using an attacker-controlled payout address; pool sends XCH block rewards to the attacker.
- **`pool_url` overwritten** → farmer is directed to a rogue pool endpoint, enabling further protocol-level attacks.
- **`owner_public_key` overwritten** → farmer authentication with the pool breaks.

This matches the allowed High impact: *"Corruption of… pool membership state… with direct security impact"* and potentially Critical: *"reward diversion… affecting XCH… pool wallets."*

### Likelihood Explanation

- Any node on the Chia network can act as a full-node peer to a wallet.
- The wallet's `validate_received_state_from_peer` validates block inclusion and the removals Merkle root — confirming the coin was spent — but does **not** validate the solution returned by `fetch_coin_spend`.
- The attacker only needs to wait for a legitimate singleton spend to occur on-chain, then serve the correct puzzle with a crafted solution. No key material or privileged access is required.

### Recommendation

1. **Short-term:** In `fetch_coin_spend`, after receiving the puzzle and solution from the peer, execute the puzzle against the solution using the CLVM and verify the output conditions match the expected singleton output (i.e., the child coin). Reject the spend if execution fails or produces unexpected conditions.
2. **Long-term:** In `apply_state_transition`, validate that the decoded `PoolState` from the new spend represents a legal transition from the current state (e.g., `owner_pubkey` is unchanged, state machine transitions are valid). The docstring already acknowledges this is where validation belongs.

### Proof of Concept

1. Attacker runs a malicious full node and gets the victim wallet to connect to it (standard peer discovery).
2. A legitimate pool singleton coin is spent on-chain (e.g., the farmer performs a normal state transition).
3. Attacker reports the coin as spent at the correct block height — `validate_received_state_from_peer` passes because the coin is genuinely in the block's removals.
4. Wallet calls `fetch_coin_spend_for_coin_state` → `fetch_coin_spend`. Attacker returns the correct puzzle (puzzle hash matches) but a crafted solution encoding `PoolState` with `target_puzzle_hash = attacker_address`.
5. `fetch_coin_spend` accepts the response (puzzle hash check passes, coin name check passes, solution unchecked).
6. `apply_state_transition` stores the fake spend (coin name matches tip), calls `update_pool_config`.
7. `update_pool_config` writes `attacker_address` as `target_puzzle_hash` to `pooling_share_state.yaml`.
8. Farmer reads the corrupted config, registers with the pool using `attacker_address` as payout destination.
9. Pool sends XCH block rewards to the attacker's address.

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(
        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)
    )
    if solution_response is None or not isinstance(solution_response, RespondPuzzleSolution):
        raise PeerRequestException(f"Was not able to obtain solution {solution_response}")
    coin_id = coin.name()
    if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
        raise PeerRequestException(f"Peer returned wrong puzzle hash for coin {coin_id}")
    if solution_response.response.coin_name != coin_id:
        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )

    async def apply_state_transition(
        self, new_state: CoinSpend, block_height: uint32, action_scope: WalletActionScope
    ) -> bool:
        """
        Updates the Pool state (including DB) with new singleton spends.
        The DB must be committed after calling this method. All validation should be done here. Returns True iff
        the spend is a valid transition spend for the singleton, False otherwise.
        """
        tip: tuple[uint32, CoinSpend] = await self.get_tip()
        tip_spend = tip[1]

        tip_coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(tip_spend)
        assert tip_coin is not None
        spent_coin_name: bytes32 = tip_coin.name()

        if spent_coin_name != new_state.coin.name():
            history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
            if new_state.coin.name() in [sp.coin.name() for _, sp in history]:
                self.log.info(f"Already have state transition: {new_state.coin.name().hex()}")
            else:
                self.log.warning(
                    f"Failed to apply state transition. tip: {tip_coin} new_state: {new_state} height {block_height}"
                )
            return False

        await self.wallet_state_manager.pool_store.add_spend(self.wallet_id, new_state, block_height)
        tip_spend = (await self.get_tip())[1]
        self.log.info(f"New PoolWallet singleton tip_coin: {tip_spend} farmed at height {block_height}")

        # If we have reached the target state, resets it to None. Loops back to get current state
        for _, added_spend in reversed(
            await self.wallet_state_manager.pool_store.get_spends_for_wallet(self.wallet_id)
        ):
            latest_state: PoolState | None = solution_to_pool_state(added_spend)
            if latest_state is not None:
                if self.target_state == latest_state:
                    self.target_state = None
                    self.next_transaction_fee = uint64(0)
                    self.next_tx_config = DEFAULT_TX_CONFIG
                break

        await self.update_pool_config(action_scope)
        return True
