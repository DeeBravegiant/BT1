### Title
Missing Caller Binding in `verify_foreign_transaction` Enables Attestation Frontrunning — (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` does not bind the request to the caller's identity. Unlike `sign()` and `request_app_private_key()`, which derive a caller-specific tweak from `(predecessor_id, path)` and embed it in the request key, `verify_foreign_transaction` silently discards the caller's account ID. The resulting `VerifyForeignTransactionRequest` key contains only `(domain_id, ForeignChainRpcRequest, payload_version)`. Any unprivileged caller can submit an identical request and receive the same MPC attestation via the fan-out mechanism, enabling a frontrunning adversary to obtain and use an attestation intended for another party to claim bridge funds.

---

### Impact Explanation

The attestation produced by `verify_foreign_transaction` is not bound to the submitting caller. Any adversary who observes a pending `verify_foreign_transaction` call in the NEAR mempool can submit an identical call and receive the same root-key attestation. Because the attestation carries no caller identity, a bridge contract that releases funds to whoever presents a valid attestation first cannot distinguish the legitimate requester from the frontrunner. The adversary presents the attestation before the original caller and claims the bridged funds, while the original caller's subsequent claim is rejected as already fulfilled.

This maps directly to the **High** allowed impact: *attestation authorization bypass that causes invalid bridge execution or double-spend conditions*.

---

### Likelihood Explanation

- The NEAR mempool is publicly observable; `verify_foreign_transaction` call arguments are fully visible before inclusion.
- Reproducing the call requires only copying the exact same `(domain_id, ForeignChainRpcRequest, payload_version)` tuple — no privileged knowledge.
- The attacker does not need to front-run in the strict sense; submitting after Alice also works because the fan-out delivers the same attestation to both. The attacker only needs to present the attestation to the bridge contract before Alice does.
- No threshold collusion, TEE access, or operator privilege is required.

---

### Recommendation

1. Add a `derivation_path: String` field to `VerifyForeignTransactionRequestArgs`.
2. In `verify_foreign_transaction`, capture `predecessor` from `check_request_preconditions` (currently discarded).
3. Derive a caller-specific tweak using a foreign-tx-specific prefix (as described in the design doc) from `(predecessor, derivation_path)`.
4. Include the derived tweak in `VerifyForeignTransactionRequest` so the request key is unique per caller.
5. Update the node-side `build_signature_request` to use the derived tweak instead of `Tweak::new([0u8; 32])`.
6. Update `respond_verify_foreign_tx` to verify the signature against the derived public key (not the root key), mirroring the pattern in `respond`.

## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
