### Title
Fee Underpayment via Sub-Transaction Reference Scripts in Dijkstra Batch — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary
In the Dijkstra era, `validateFeeTooSmallUTxO` computes the minimum fee using only the top-level transaction's reference scripts (`txNonDistinctRefScriptsSize`), while the batch ref-script size limit uses `batchNonDistinctRefScriptsSize` which includes sub-transaction reference scripts. An attacker can craft a batch where sub-transactions reference large scripts (up to `ppMaxRefScriptSizePerTxG`) that are priced at zero in the fee calculation, underpaying for the actual validation cost of the batch.

### Finding Description
`EraUTxO DijkstraEra` sets `getMinFeeTxUtxo = getConwayMinFeeTxUtxo`, which internally calls `txNonDistinctRefScriptsSize utxo tx`. This function only iterates over the top-level transaction's `inputsTxBodyL ∪ referenceInputsTxBodyL`; it does not recurse into `subTransactionsTxBodyL`. The Dijkstra-specific `batchNonDistinctRefScriptsSize` function, which does include sub-transaction ref scripts, is only used in `Ledger.hs` for the `DijkstraTxRefScriptsSizeTooBig` size-cap check. There is no corresponding use of `batchNonDistinctRefScriptsSize` (or any equivalent) in the fee validation path.

Because sub-transactions carry no fee field, the single `dtbrFee` in the top-level body is the only fee paid for the entire batch. The minimum fee enforced by `validateFeeTooSmallUTxO` is therefore blind to any ref-script overhead introduced by sub-transactions.

### Impact Explanation
An attacker can submit a batch where the top-level transaction has zero or minimal reference scripts (paying a low minimum fee) while embedding sub-transactions that each reference large scripts. The sub-tx ref scripts are processed and validated by the node but are not priced into the fee. This allows systematic underpayment for ref-script processing cost across the batch, violating the fee-covers-validation-cost invariant. Impact is **Medium**: attacker-controlled transactions modify fees outside design parameters.

### Likelihood Explanation
The attack requires only the ability to submit a valid Dijkstra-era batch transaction, which is an unprivileged operation. No governance keys, stake majority, or special roles are needed. The discrepancy is structural and deterministic — every node enforces the same under-pricing — so it is reliably exploitable on any Dijkstra-era network.

### Recommendation
Replace `getConwayMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance with a Dijkstra-specific implementation that passes `batchNonDistinctRefScriptsSize utxo tx` (instead of `txNonDistinctRefScriptsSize utxo tx`) to `getMinFeeTx`. This ensures the declared fee must cover the ref-script cost of the entire batch, consistent with the existing size-limit accounting.

### Proof of Concept
```

--   top-level tx: no reference inputs (txNonDistinctRefScriptsSize = 0)
--   sub-tx 1..N: each references a large script S of size K bytes
--                (total batch size = N*K ≤ ppMaxRefScriptSizePerTxG)
--
-- Fee set to: getConwayMinFeeTxUtxo pp topTx utxo
--           = getMinFeeTx pp topTx 0
--           = alonzoMinFeeTx pp topTx   (no ref-script surcharge)
--
-- batchNonDistinctRefScriptsSize utxo topTx = N*K  (passes size limit)
-- validateFeeTooSmallUTxO passes because minFee ignores sub-tx ref scripts
--
-- Differential: getConwayMinFeeTxUtxo(topTx) vs
--               getMinFeeTx pp topTx (batchNonDistinctRefScriptsSize utxo topTx)
-- The second is strictly larger for any batch with sub-tx ref scripts.
```
