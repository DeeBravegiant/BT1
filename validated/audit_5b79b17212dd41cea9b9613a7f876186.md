### Title
Missing On-Chain Pairing Check for `AppPublicKey` CKD Responses Allows Byzantine Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC smart contract performs a BLS12-381 pairing check on the CKD response **only** when the request uses the `AppPublicKeyPV` variant. When the request uses the plain `AppPublicKey` variant (the default, simpler path), the response `(big_c, big_y)` is accepted and returned to the caller with **zero cryptographic verification**. A single Byzantine attested participant can call `respond_ckd` with an arbitrary forged response before honest nodes, causing the TEE app to receive an attacker-controlled key instead of the correct MPC-derived secret.

---

### Finding Description

In `respond_ckd`, the match on `request.app_public_key` has an empty arm for the `AppPublicKey` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` enforces the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk‖app_id), mpc_pk)`, which cryptographically binds the response to the MPC network's master secret key. [2](#0-1) 

For the `AppPublicKey` variant, no equivalent check exists. Any `(big_c, big_y)` pair — including the identity point, a random point, or a point encoding an attacker-chosen scalar — is accepted unconditionally and forwarded to the waiting yield.

The `AppPublicKey` variant is the default path shown in all documentation and examples: [3](#0-2) 

The design documentation states that for this variant "verification is performed by the app after decryption," but the contract does not enforce that the app ever performs this check, and once the yield is resolved with a forged response, the app has no mechanism to request a replacement. [4](#0-3) 

---

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without required participant authorization.**

The CKD protocol is designed so that the correct output `s = H(pk‖app_id) · msk` requires threshold cooperation of MPC nodes. A single Byzantine attested participant can bypass this threshold requirement entirely for the `AppPublicKey` variant by:

1. Racing to call `respond_ckd` with a forged `(big_c, big_y)` before honest nodes submit their response.
2. The contract resolves the yield with the forged response (no pairing check blocks it).
3. The TEE app receives `big_c_forged` and decrypts to obtain `s_forged = big_c_forged - a · big_y_forged`, which is an attacker-controlled value rather than the true `H(pk‖app_id) · msk`.

If the attacker sets `big_c = a · G1` and `big_y = G1` (so `big_c_forged = a · big_y_forged`), the app decrypts to `s = 0`. If the attacker sets `big_c = r · G1` for a known scalar `r` and `big_y = identity`, the app decrypts to `s = r · G1`, a key the attacker knows. In either case:

- The app's derived key is **not** the MPC-protected secret — threshold authorization is bypassed.
- If the app uses this key to protect assets (e.g., encrypt data, derive signing keys), those assets are compromised.
- The app cannot retry: the yield is already resolved.

---

### Likelihood Explanation

**Medium.** The attacker must be an attested MPC participant (satisfying `assert_caller_is_attested_participant_and_protocol_active`), which is within the stated threat model of a Byzantine participant strictly below the signing threshold. [5](#0-4) 

The `AppPublicKey` variant is the default path used by most callers (no G2 key required), making it the common attack surface. The attacker simply needs to observe a pending CKD request on-chain and submit a forged response before honest nodes.

---

### Recommendation

**Short term:** Remove the empty `AppPublicKey` arm or add an equivalent pairing check. Since `AppPublicKey` only provides a G1 point (no G2 component), the contract cannot perform the full `ckd_output_check` as-is. The simplest fix is to **deprecate the `AppPublicKey` variant** and require all callers to use `AppPublicKeyPV`, which provides the G2 component needed for on-chain verification.

**Long term:** Whenever a new CKD or signing response path is added, ensure that every variant of the request type has a corresponding cryptographic binding check before the yield is resolved. Apply the same principle used in `respond` (ECDSA/EdDSA signature verification) and `respond_ckd` (PV pairing check) uniformly across all response handlers.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk1)` on a BLS12-381 CKD domain.
2. Attacker (Byzantine attested participant) observes the pending request on-chain.
3. Attacker calls `respond_ckd(ckd_request, CKDResponse { big_y: identity_point, big_c: arbitrary_g1_point })`.
4. The contract executes the `AppPublicKey(_) => {}` arm — no check — and calls `resolve_yields_for`, resolving the yield with the forged response.
5. The TEE app receives `(big_c_forged, big_y_forged)` and decrypts: `s = big_c_forged - a · identity = big_c_forged`, an attacker-chosen G1 point, not the true `H(pk‖app_id) · msk`.
6. The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates this: it passes `big_y: [1u8; 48]` and `big_c: [2u8; 48]` (invalid points) and the contract accepts them without error for the `AppPublicKey` variant. [6](#0-5)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
```
