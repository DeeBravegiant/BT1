Audit Report

## Title
First-Depositor Inflation Attack via Bootstrap `rsETHPrice` Bypass and Disabled Price Ceiling — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle._updateRsETHPrice()` unconditionally sets `rsETHPrice = 1 ether` and returns early when `rsethSupply == 0`, bypassing all price-validation guards. Because `pricePercentageLimit` is never initialized in `initialize()` (defaults to `0`), the price-increase ceiling guard is permanently disabled until an admin explicitly sets it. An unprivileged attacker can chain these two gaps with an ETH donation to inflate `rsETHPrice` to an arbitrary value immediately after deployment, causing subsequent depositors who omit a slippage guard to receive 0 rsETH for real ETH — a direct theft of funds.

## Finding Description

**Root cause 1 — bootstrap bypass in `_updateRsETHPrice()`**

When `rsethSupply == 0`, the function sets `rsETHPrice = 1 ether` and `highestRsethPrice = 1 ether`, then returns immediately. Every downstream guard — the `pricePercentageLimit` ceiling, fee minting, and the downside auto-pause — is skipped entirely. [1](#0-0) 

**Root cause 2 — `pricePercentageLimit` defaults to `0`**

`initialize()` never sets `pricePercentageLimit`, so it starts at `0`. [2](#0-1) 

The guard that enforces a ceiling on price increases evaluates `pricePercentageLimit > 0` first; since it is `0`, `isPriceIncreaseOffLimit` is always `false` and no price-increase limit is enforced until an admin calls `setPricePercentageLimit()`. [3](#0-2) 

**Root cause 3 — open ETH donation path**

Anyone can send ETH directly to `LRTDepositPool` via the unrestricted `receive()` fallback. [4](#0-3) 

That ETH is immediately counted in `getETHDistributionData()` as `address(this).balance`, inflating `totalETHInProtocol` used by the oracle. [5](#0-4) 

**Root cause 4 — zero-rsETH mint is silently accepted**

`_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`. A victim who passes `minRSETHAmountExpected = 0` (common default) receives 0 rsETH while their ETH is accepted, because `0 < 0` is `false`. [6](#0-5) 

**Full exploit chain:**

1. Attacker calls `updateRSETHPrice()` when `rsethSupply == 0` → bootstrap bypass sets `rsETHPrice = 1e18`.
2. Attacker calls `depositETH{value: 1 wei}(0, "")` → mints 1 wei rsETH (100% of supply).
3. Attacker sends 1000 ETH to `LRTDepositPool` via `receive()` → `address(this).balance` inflated to ~1000 ETH.
4. Attacker calls `updateRSETHPrice()` again → `newRsETHPrice = (1000e18 + 1) * 1e18 / 1 ≈ 1000e36`. Since `pricePercentageLimit == 0`, `isPriceIncreaseOffLimit = false`, no revert.
5. Victim calls `depositETH{value: 999e18}(0, "")` → `rsethAmountToMint = 999e18 * 1e18 / 1000e36 = 0`. No revert. Victim loses 999 ETH with 0 rsETH issued.
6. Attacker holds 100% of rsETH supply and can redeem for the entire TVL (~1999 ETH), recovering the 1000 ETH donation and netting ~999 ETH profit.

## Impact Explanation

**Critical — direct theft of depositor funds.** The victim's ETH is permanently absorbed into the protocol with no rsETH issued. The attacker holds 100% of the rsETH supply and can redeem it for the entire TVL. The donation is fully recovered; the victim's ETH is the attacker's profit. This matches the allowed Critical impact: "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation

The attack window opens at deployment and closes only when an admin explicitly calls `setPricePercentageLimit()` with a non-zero value. There is no on-chain enforcement that this must happen before the first deposit. `updateRSETHPrice()` is a public, permissionless function callable by anyone. The only cost to the attacker is the ETH donation, which is recovered in full. Any depositor who omits a slippage guard (`minRSETHAmountExpected = 0`) is a viable victim. The attack is executable by any unprivileged external user with no special access.

## Recommendation

1. **Set a non-zero `pricePercentageLimit` in `initialize()`** (e.g., `1e16` for 1%) so the price-increase ceiling is active from block 0.
2. **Revert in `_beforeDeposit` if `rsethAmountToMint == 0`** to prevent silent fund loss regardless of oracle state.
3. **Seed the protocol with a non-trivial initial deposit** (or use a virtual-offset pattern as in OpenZeppelin ERC-4626 inflation-attack documentation) so the first-depositor attack surface does not exist.
4. Consider restricting `receive()` to known callers (node delegators, reward receivers) to eliminate the open donation vector.

## Proof of Concept

```
State at deployment:
  rsETHPrice           = 0   (uninitialized)
  rsethSupply          = 0
  pricePercentageLimit = 0   (never set in initialize())

Step 1 — Attacker triggers bootstrap bypass:
  attacker calls updateRSETHPrice()
    → rsethSupply == 0 → early return
    → rsETHPrice = 1e18, highestRsethPrice = 1e18
    (all validation skipped)

Step 2 — Attacker acquires a foothold:
  attacker calls depositETH{value: 1 wei}(minRSETHAmountExpected=0, "")
    → rsethAmountToMint = (1 * 1e18) / 1e18 = 1 wei rsETH
    → attacker holds 1 wei rsETH (100% of supply)

Step 3 — Attacker donates ETH to inflate TVL:
  attacker sends 1000 ETH to LRTDepositPool via receive()
    → address(this).balance = 1000e18 + 1 wei
    → totalETHInProtocol ≈ 1000e18

Step 4 — Attacker inflates rsETHPrice (no limit check):
  attacker calls updateRSETHPrice()
    → rsethSupply = 1 wei
    → newRsETHPrice = (1000e18 + 1) * 1e18 / 1 ≈ 1000e36
    → pricePercentageLimit == 0 → isPriceIncreaseOffLimit = false → no revert
    → rsETHPrice ≈ 1000e36

Step 5 — Victim deposits with no slippage guard:
  victim calls depositETH{value: 999e18}(minRSETHAmountExpected=0, "")
    → rsethAmountToMint = (999e18 * 1e18) / 1000e36 = 0 (rounds down)
    → 0 < 0 is false → no revert
    → 0 rsETH minted; 999 ETH accepted into pool
    → victim loses 999 ETH

Step 6 — Attacker redeems:
  attacker redeems 1 wei rsETH (100% of supply)
    → entitled to entire TVL ≈ 1999 ETH
    → net profit ≈ 999 ETH stolen
```

**Foundry test plan:** Deploy `LRTOracle` and `LRTDepositPool` on a local fork with `rsethSupply = 0` and `pricePercentageLimit = 0`. Execute steps 1–6 above. Assert that after step 5, the victim's rsETH balance is 0 and the pool's ETH balance includes the victim's 999 ETH. Assert that after step 6, the attacker's ETH balance has increased by ~999 ETH net of the donation.
