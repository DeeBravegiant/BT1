### Title
Unbounded `fee_proposal_fri` During V0_14_3 Startup Window Permanently Inflates `fee_actual` Gas-Price Floor — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

In `is_proposal_init_valid`, the bounds check on `fee_proposal_fri` is entirely skipped whenever `fee_actual` is `None`. `fee_actual` is `None` for the first `window_size` (= 10) blocks after V0_14_3 activates, because every pre-V0_14_3 block stored in the sliding window carries `fee_proposal_fri = None`, causing `compute_fee_actual` to return `None`. During this predictable, one-time window a proposer who controls a majority of those 10 slots can publish an arbitrarily large `fee_proposal_fri` that is accepted by every validator, permanently poisoning the median `fee_actual` that serves as the EIP-1559 gas-price floor for all subsequent blocks.

---

### Finding Description

**Root cause — unconditional skip of bounds enforcement when `fee_actual` is `None`:** [1](#0-0) 

```rust
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
```

The `if let` pattern silently drops the check when `fee_actual` is `None`. There is no fallback cap — no hardcoded `MAX_FEE`, no comparison against `l2_gas_price`, nothing. Any `fee_proposal_fri` value, including `u128::MAX`, passes validation.

**Why `fee_actual` is `None` for the first 10 blocks after V0_14_3:**

`compute_fee_actual` returns `None` the moment any slot in the `[height - window_size, height - 1]` range is missing or holds `None`. [2](#0-1) 

Pre-V0_14_3 blocks are stored with `fee_proposal_fri = None` (the field did not exist). `initialize_fee_proposals_window` faithfully loads those `None` values from state-sync: [3](#0-2) 

So for heights `[H, H+9]` (where H is the first V0_14_3 block), every validator computes `fee_actual = None` and the bounds check is skipped for all 10 blocks.

**Validators also skip the check — the bypass is network-wide:**

`validate_proposal` passes `fee_actual` from the same `compute_fee_actual` call: [4](#0-3) 

Every honest validator has the same `None` result, so a proposal with `fee_proposal_fri = u128::MAX` is accepted by 2/3+ of the network and committed.

**The inflated value is permanently recorded and propagates:**

After `decision_reached`, `record_fee_proposal` writes the proposer-supplied value into the sliding window: [5](#0-4) 

Once the window fills (block H+10), `compute_fee_actual` returns the median of those 10 values. If 6 of the 10 are inflated, the median is inflated. That median then becomes `effective_min` in the EIP-1559 calculation: [6](#0-5) 

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
```

With `effective_min = u128::MAX`, `calculate_next_base_gas_price` enters the "price below minimum" branch and increases the gas price by ~0.3 % every block (doubling in ≈ 231 blocks / ≈ 10 minutes), continuously until it reaches `u128::MAX`.

**The inflated value is also bound into the `ProposalCommitment`:** [7](#0-6) 

```rust
ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
```

The commitment that 2/3+ of validators sign over includes the attacker-chosen `fee_proposal_fri`, making the manipulation irreversible on-chain.

**Versioned constants confirm `window_size = 10` and `margin_ppt = 2` (0.2 %/block):** [8](#0-7) 

The 0.2 %/block cap means once `fee_actual` is inflated it takes thousands of blocks to recover.

---

### Impact Explanation

A proposer who controls ≥ 6 of the 10 startup blocks can set `fee_actual` to any value up to `u128::MAX`. This inflated floor is then enforced by `calculate_next_l2_gas_price_for_fin` on every subsequent block, causing the L2 gas price to increase exponentially. All user transactions pay continuously inflating fees; at the extreme the gas price reaches `u128::MAX` and every transaction fails for insufficient gas. The corrupted value is committed into the `ProposalCommitment` signed by 2/3+ of validators and stored in `BlockHeaderWithoutHash.fee_proposal_fri`, making it permanent. This matches **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."**

---

### Likelihood Explanation

The attack window is exactly 10 blocks at a single, publicly known, one-time event (the V0_14_3 protocol upgrade). An attacker who controls ≥ 6 of those 10 proposer slots (i.e., > 50 % of voting power during that window) can execute the attack. In a BFT system with equal-weight validators this requires controlling a majority of validators, which is outside the standard 1/3-fault assumption. However:

- The window is predictable and short, making targeted stake concentration feasible.
- A validator with > 50 % stake (e.g., a dominant staking pool) can execute this unilaterally.
- Even a minority attacker who controls 1–5 slots can shift the median upward by a smaller but non-zero amount, degrading fee accuracy for all users.

---

### Recommendation

Add a hardcoded `MAX_FEE_PROPOSAL_FRI` constant (analogous to the `MAX_FEE` the original report's judge recommended) and enforce it unconditionally in `is_proposal_init_valid`, regardless of whether `fee_actual` is `Some` or `None`:

```rust
const MAX_FEE_PROPOSAL_FRI: u128 = /* e.g. 10 * current_max_expected_price */;

// Always enforce an absolute ceiling, even during the startup window.
if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    if fee_proposal.0 > MAX_FEE_PROPOSAL_FRI {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}

// Existing relative-to-fee_actual check (only when window is full).
if let (Some(fee_actual), Some(fee_proposal)) = (...) { ... }
```

Alternatively, during the startup window, bound `fee_proposal_fri` against `l2_gas_price` (the honest proposer's own fallback value) with the same `fee_proposal_margin_ppt` margin, since `l2_gas_price` is always available.

---

### Proof of Concept

1. V0_14_3 activates at height H. The `fee_proposals_window` for `[H-10, H-1]` contains 10 `None` entries (pre-V0_14_3 blocks). `compute_fee_actual` returns `None` for all heights `[H, H+9]`.
2. A malicious proposer controls 6 of the 10 proposer slots in `[H, H+9]`. For each of those 6 blocks it sets `fee_proposal_fri = GasPrice(u128::MAX)` in `ProposalInit`.
3. `is_proposal_init_valid` reaches the bounds check: `fee_actual = None`, so the `if let` body is skipped. The proposal passes validation and is committed.
4. `record_fee_proposal` stores `Some(u128::MAX)` for each of the 6 blocks.
5. At height H+10, `compute_fee_actual` sorts the window: `[honest×4, u128::MAX×6]`. The median (average of 5th and 6th values) = `u128::MAX`.
6. `calculate_next_l2_gas_price_for_fin` sets `effective_min = u128::MAX`. The gas price begins increasing by 0.3 %/block, doubling every ≈ 231 blocks (≈ 10 minutes at 2.6 s/block).
7. All user transactions pay exponentially increasing fees. The `fee_proposal_margin_ppt = 2` (0.2 %/block) cap means the fee can only decrease by 0.2 %/block once honest proposers resume, requiring thousands of blocks to recover.
