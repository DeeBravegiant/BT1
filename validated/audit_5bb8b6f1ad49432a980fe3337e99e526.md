### Title
Deposits for Non-Executed `propose_update` Entries Are Permanently Locked — (`File: crates/contract/src/update.rs`)

### Summary
When a participant calls `propose_update`, a NEAR storage-staking deposit is collected and held in the contract. If a different proposal is subsequently executed via `do_update`, all other proposal entries are bulk-cleared without refunding their associated deposits. Because `UpdateEntry` never records the proposer's account ID, there is no mechanism to return these funds. The locked NEAR tokens constitute an irrecoverable balance-accounting invariant violation.

### Finding Description
`propose_update` in `crates/contract/src/lib.rs` collects a deposit sized to cover on-chain storage for the proposal payload: [1](#0-0) 

Only the *excess* above the required amount is refunded immediately. The required portion stays in the contract. The deposit amount and the proposer's identity are never stored inside `UpdateEntry`: [2](#0-1) 

`UpdateEntry` holds only `update` and `bytes_used` — no `proposer: AccountId`, no `deposit: NearToken`. When `do_update` executes a winning proposal it bulk-clears every other entry: [3](#0-2) 

Because the proposer identity was never persisted, there is no information available to issue refunds to the owners of the cleared entries. The NEAR tokens are permanently absorbed into the contract's balance.

There is no `remove_update_proposal` endpoint, and `remove_update_vote` / `remove_non_participant_update_votes` only remove *votes*, not proposal entries: [4](#0-3) 

### Impact Explanation
Every participant whose proposal is cleared by a competing `do_update` permanently loses the storage-staking deposit they paid. For a 1 MB WASM payload the required deposit is approximately 10 NEAR (NEAR charges ~10 NEAR/MB for storage staking). With multiple concurrent proposals this loss compounds. The funds are not stolen by an external party — they are frozen inside the contract with no recovery path, breaking the production accounting invariant that deposits must be returned when the storage they cover is freed.

This maps to the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation
The scenario is a routine operational event: participants routinely propose competing updates (code upgrades, config changes). Whenever threshold votes converge on one proposal and `vote_update` triggers `do_update`, every other live proposal is silently cleared. No adversarial setup is required — normal governance activity is sufficient to trigger the loss.

### Recommendation
Store the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before clearing the remaining entries, iterate over them and schedule a `Promise::new(entry.proposer).transfer(entry.deposit)` for each. This mirrors the refund pattern already used in `require_deposit` and `submit_participant_info`.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB WASM payload, attaching ~10 NEAR deposit. Entry `id=0` is stored; 10 NEAR is held by the contract.
2. Participant B calls `propose_update` with a config change, attaching a smaller deposit. Entry `id=1` is stored.
3. A threshold of participants call `vote_update(id=1)`. `do_update` executes:
   - Removes entry `id=1` (the winning proposal).
   - Calls `self.entries.clear()` — entry `id=0` is deleted.
   - Calls `self.vote_by_participant.clear()`.
   - No refund is issued for entry `id=0`.
4. Participant A's ~10 NEAR is permanently locked in the contract. There is no endpoint to recover it. [5](#0-4) [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
