Looking at the exact code paths, I need to verify:
1. Whether `dlog::verify` / `dlogeq::verify` can return `Err(IdentityElement)` from attacker-controlled input
2. Whether that error propagates as a hard abort in `do_generation_many`
3. Whether a Byzantine participant below threshold can actually craft such input

### Title
Byzantine Participant Can Abort Triple Generation for All Honest Nodes via Crafted Identity-Commitment Proof — (`crates/threshold-signatures/src/crypto/proofs/dlog.rs` + `dlogeq.rs`, `triples/generation.rs`)

---

### Summary

`dlog::verify` and `dlogeq::verify` return `Err(ProtocolError::IdentityElement)` — a hard error — when the reconstructed commitment point is the group identity. A Byzantine participant who knows their own secret scalar (which they always do, having generated it themselves) can trivially craft a proof `(e, s)` that forces this identity result. The `?` operator at every call site in `do_generation_many` propagates this hard error upward, aborting the entire triple generation protocol for all honest nodes. A single participant below threshold can repeat this indefinitely, permanently blocking ECDSA signing.

---

### Finding Description

**Root cause — `dlog::verify`:** [1](#0-0) 

```rust
let big_k = C::Group::generator() * proof.s.0 - *statement.public * proof.e.0;
// Raises error if the big_k turned out to be the identity element
let ser = C::Group::serialize(&big_k).map_err(|_| ProtocolError::IdentityElement)?;
```

If `big_k` is the group identity, `serialize` fails and `Err(IdentityElement)` is returned — **before** the transcript challenge is ever checked.

**Root cause — `dlogeq::verify`:** [2](#0-1) 

```rust
let big_k0 = phi0 - *statement.public0 * proof.e.0;
let big_k1 = phi1 - *statement.public1 * proof.e.0;
let enc = encode_two_points::<C>(&big_k0, &big_k1)?;
```

`encode_two_points` returns `Err(IdentityElement)` if either `big_k0` or `big_k1` is the identity. [3](#0-2) 

**How the attacker crafts the identity commitment:**

In Round 3 (`wait2`), the verifier checks the Byzantine participant's own dlog proof against their own polynomial commitment: [4](#0-3) 

```rust
let statement0 = dlog::Statement::<C> {
    public: &their_big_e.eval_at_zero()?.value(),  // = e_j(0) * G
};
if !dlog::verify(..., statement0, their_phi_proof0)? {  // ? propagates Err
```

`statement.public = e_j(0) * G` where `e_j(0)` is the Byzantine participant's **own** secret scalar — they know it because they generated it. The attacker sets:

- `proof.e = 1` (any nonzero scalar)
- `proof.s = e_j(0)` (i.e., `e * e_j(0)` with `e = 1`)

The verifier then computes:

```
big_k = s*G - e*X = e_j(0)*G - 1*(e_j(0)*G) = identity
```

No discrete log needs to be solved. No cryptographic assumption is broken. The attacker uses only their own secret.

**The same attack applies at `wait4` (dlogeq) and `wait5` (dlog):** [5](#0-4) [6](#0-5) 

At `wait4`, `statement.public0 = big_e_j_zero.index(from)?.value()` is again the Byzantine participant's own commitment, so the same `(e=1, s=e_j(0))` trick forces `big_k0 = identity`.

**Error propagation:**

The `?` operator at all three call sites propagates `Err(IdentityElement)` directly out of `do_generation_many`, which is the core async function driving the entire protocol: [7](#0-6) 

The error is not caught or attributed to a specific participant — the `from` variable identifying the Byzantine sender is lost in the propagation. All honest nodes abort their protocol instance.

---

### Impact Explanation

Triple generation is the prerequisite for all ECDSA presigning. Without triples, no signatures can be produced. A single Byzantine participant below threshold can abort every triple generation attempt indefinitely. Because the error does not identify the culprit, honest nodes cannot automatically exclude the attacker. All funds controlled by the MPC network are effectively frozen until operators manually identify and remove the Byzantine participant — which may not be straightforward given the opaque error.

---

### Likelihood Explanation

The attack requires only that the Byzantine participant is one of the N participants in a triple generation session (a normal, expected role) and that they know their own secret scalar (guaranteed — they generated it). No special access, no threshold collusion, no cryptographic break. The crafted proof `(e=1, s=x)` is trivial to compute. The attacker can repeat it on every retry.

---

### Recommendation

In both `dlog::verify` and `dlogeq::verify`, treat a reconstructed identity commitment as a **verification failure** (`Ok(false)`) rather than a hard error (`Err`). The identity commitment is simply an invalid proof — it is not an internal invariant violation:

```rust
// dlog::verify — replace:
let ser = C::Group::serialize(&big_k).map_err(|_| ProtocolError::IdentityElement)?;
// with:
let Ok(ser) = C::Group::serialize(&big_k) else { return Ok(false); };
```

Apply the same change in `encode_two_points` when called from `dlogeq::verify` (or add an explicit identity check before calling `encode_two_points` and return `Ok(false)`).

Additionally, consider attributing the error to the sending participant at the call sites in `do_generation_many` so that Byzantine participants can be identified and excluded.

### Proof of Concept

```rust
#[test]
fn test_dlog_identity_commitment_returns_ok_false_not_err() {
    use frost_secp256k1::Secp256K1Sha256;
    use k256::{ProjectivePoint, Scalar};
    use frost_core::serialization::SerializableScalar;

    // x is the Byzantine participant's own secret
    let x = Scalar::from(42u64);
    let big_x = ProjectivePoint::GENERATOR * x; // their public key

    let statement = dlog::Statement::<Secp256K1Sha256> { public: &big_x };

    // Craft (e=1, s=x): big_k = s*G - e*X = x*G - x*G = identity
    let proof = dlog::Proof {
        e: SerializableScalar(Scalar::from(1u64)),
        s: SerializableScalar(x),
    };

    let transcript = Transcript::new(b"protocol");
    let result = dlog::verify(&mut transcript.fork(b"party", &[1]), statement, &proof);

    // Currently returns Err(IdentityElement) — should return Ok(false)
    assert_eq!(result, Ok(false), "got: {:?}", result);
}
```

The test currently fails with `Err(IdentityElement)` instead of `Ok(false)`, confirming the bug. The same pattern applies to `dlogeq::verify` with `(e=1, s=e_j(0))` targeting `public0`.
