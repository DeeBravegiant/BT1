### Title
Unpermissioned `directDeposits` Griefs Legacy-Mode Withdrawal Transactions in Dijkstra Era — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

The Dijkstra era introduces `directDeposits`, a new transaction field that lets any unprivileged user add ADA to any registered account without the account owner's signature. When a transaction uses PlutusV1–V3 scripts ("legacy mode"), the ledger requires withdrawals to exactly equal the account balance. A griefer can submit a `directDeposits` transaction adding 1 lovelace to a target account, changing its balance and causing any in-flight legacy-mode withdrawal transaction (signed with the old balance) to fail with `IncompleteWithdrawals`.

---

### Finding Description

**Root cause — no witness required for `directDeposits` target:**

`applyDirectDeposits` in `libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs` adds ADA to any registered account with no ownership check:

```haskell
-- /Note/ - There are no checks that direct deposits mention only registered accounts.
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
```

The only validation applied to `directDeposits` in the UTXO rule is a network-ID check (`validateWrongNetworkInDirectDeposit`). No witness for the target account is required.

**Root cause — legacy mode enforces exact balance match:**

`validateWithdrawals` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs` branches on `legacyMode` (set to `True` when PlutusV1–V3 scripts are present):

```haskell
if legacyMode
  then do
    let (missingWithdrawals, incompleteWithdrawals) =
          case withdrawalsThatDoNotDrainAccounts withdrawals network accounts of
            Nothing -> (Map.empty, Map.empty)
            Just (missing, incomplete) -> (unWithdrawals missing, incomplete)
    failOnNonEmptyMap incompleteWithdrawals IncompleteWithdrawals
    ...
  else do
    let (missingWithdrawals, exceededWithdrawals) =
          case withdrawalsThatExceedAccountBalance withdrawals network accounts of
            ...
```

`withdrawalsThatDoNotDrainAccounts` (in `libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs`) requires:

```haskell
withdrawalAmount == fromCompact (account ^. balanceAccountStateL)
```

Any deviation — even +1 lovelace — triggers `IncompleteWithdrawals` and rejects the entire transaction.

**Attack chain:**

1. Alice holds `B` lovelace in her reward account.
2. Alice constructs and signs a transaction containing a PlutusV2 script and a withdrawal of exactly `B` lovelace. The signed transaction body commits to the withdrawal amount `B`.
3. Alice broadcasts the transaction; it sits in the mempool.
4. Mallory submits a transaction with `directDeposits = { Alice's account → 1 lovelace }`. No witness for Alice's account is needed.
5. Mallory's transaction is included in a block before Alice's.
6. Alice's account balance is now `B + 1`.
7. Alice's transaction is processed: `validateWithdrawals` (legacy mode) calls `withdrawalsThatDoNotDrainAccounts`, finds `B ≠ B + 1`, and emits `IncompleteWithdrawals`. The transaction is rejected.
8. Mallory can repeat step 4 each time Alice resubmits, sustaining the DoS indefinitely.

The test in `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertsSpec.hs` (lines 93–106) explicitly confirms that in legacy mode a zero-amount withdrawal (i.e., any amount that does not equal the exact balance) fails with `IncompleteWithdrawals`, while the same withdrawal succeeds in non-legacy mode (lines 108–113).

---

### Impact Explanation

**Impact: Medium** — Attacker-controlled transactions modify reward account balances outside design parameters, causing legitimate withdrawal transactions to fail. A sustained attack prevents a victim from ever successfully withdrawing rewards while the attacker continues depositing. The victim loses transaction fees on each failed attempt; the attacker's cost per grief is 1 lovelace + transaction fees (non-zero but low). This matches the allowed Medium impact: *"Attacker-controlled transactions… modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."*

---

### Likelihood Explanation

**Likelihood: Medium.** The attack requires:
- The victim to use a PlutusV1–V3 script in a transaction that also includes a withdrawal (common for DeFi protocols interacting with staking rewards).
- The attacker to observe the victim's transaction in the mempool and submit a `directDeposits` transaction first — straightforward given Cardano's public mempool.

No privileged access, leaked keys, or consensus majority is required. Any unprivileged user can submit a `directDeposits` transaction targeting any registered account.

---

### Recommendation

1. **Require a witness for `directDeposits` targets.** Add a check in the ENTITIES or UTXO rule that each account address in `directDeposits` is accompanied by a valid witness (key signature or script) from the account owner, analogous to how withdrawals require a witness.
2. **Alternatively**, document that `directDeposits` can change account balances and advise protocol authors using PlutusV1–V3 scripts to migrate to PlutusV4 (non-legacy mode), where partial withdrawals (`≤ balance`) are accepted and the exact-match requirement does not apply.

### Proof of Concept

```
State:
  Alice's reward account balance = 100_000_000 lovelace (100 ADA)

Step 1 – Alice builds and signs:
  tx_alice = {
    inputs:    [some UTxO],
    withdrawals: { Alice's account → 100_000_000 },
    scripts:   [PlutusV2 script],   -- triggers legacyMode = True
    ...
  }
  Alice broadcasts tx_alice to the mempool.

Step 2 – Mallory observes tx_alice and submits:
  tx_mallory = {
    inputs:    [Mallory's UTxO covering fee + 1 lovelace],
    outputs:   [change back to Mallory],
    directDeposits: { Alice's account → 1 },   -- no witness needed
    ...
  }
  tx_mallory is included in block N.

Step 3 – tx_alice is processed in block N+1:
  validateWithdrawals (legacyMode=True):
    withdrawalsThatDoNotDrainAccounts:
      Alice's withdrawal amount = 100_000_000
      Alice's actual balance    = 100_000_001   -- changed by Mallory
      100_000_000 ≠ 100_000_001  →  IncompleteWithdrawals
  tx_alice is REJECTED.

Mallory repeats step 2 each time Alice resubmits.
```
