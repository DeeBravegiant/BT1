### Title
Sub-Transaction ExUnits Excluded from Per-Transaction and Per-Block Resource Limit Checks in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

### Summary

In the Dijkstra era, top-level transactions may embed an arbitrary number of sub-transactions, each carrying its own Plutus redeemers with independent ExUnits budgets. The `totExUnits` function only sums redeemers from the **top-level** transaction's witness set. As a result, neither the per-transaction ExUnits cap (`maxTxExUnits`) nor the per-block ExUnits cap (`maxBlockExUnits`) accounts for the ExUnits declared in sub-transactions. An unprivileged transaction sender can craft a batch whose aggregate script execution budget far exceeds both protocol-parameter limits, and a block producer can include such a batch in a valid block.

---

### Finding Description

**`totExUnits` is blind to sub-transaction redeemers.**

`totExUnits` is defined as:

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

It folds only over the redeemers stored in the **top-level** transaction's witness set. [1](#0-0) 

Sub-transactions in the Dijkstra era carry their own witness sets (including their own redeemers), but these are never visited by `totExUnits`. [2](#0-1) 

**Per-transaction check misses sub-tx ExUnits.**

`dijkstraUtxoTransition` enforces:

```haskell
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

Because `totExUnits tx` ignores sub-transaction redeemers, a top-level transaction with zero top-level redeemers but N sub-transactions each carrying `maxTxExUnits` worth of redeemers passes this check with a reported total of `ExUnits 0 0`. [3](#0-2) 

**Per-block check misses sub-tx ExUnits.**

`dijkstraBbodyTransition` enforces:

```haskell
Alonzo.validateExUnits @era txs $ pp ^. ppMaxBlockExUnitsL
```

`validateExUnits` computes `txTotal = foldMap totExUnits txs` over the sequence of top-level transactions. Since `totExUnits` is sub-tx-blind, the block-level sum is also sub-tx-blind. [4](#0-3) [5](#0-4) 

**`SUBUTXO` rule has no ExUnits check at all.**

`dijkstraSubUtxoTransition` performs validity-interval, input-set, output-size, network-ID, and UTxO-update checks, but contains no call to `validateExUnitsTooBigUTxO` or any equivalent. The `DijkstraSubUtxoPredFailure` type has no `ExUnitsTooBigUTxO` constructor, and the mapping function explicitly marks that failure as `error "Impossible"`:

```haskell
ExUnitsTooBigUTxO _ -> error "Impossible: `ExUnitsTooBigUTxO` for SUBUTXO"
``` [6](#0-5) [7](#0-6) 

---

### Impact Explanation

The `maxTxExUnits` and `maxBlockExUnits` protocol parameters exist to bound the CPU and memory required to validate any single transaction and any single block, respectively. Their purpose is stated explicitly in the formal spec:

> "The parameters are used to ensure that the time and memory that are required to verify a block are bounded." [8](#0-7) 

By excluding sub-transaction ExUnits from both checks, the Dijkstra era allows a transaction batch whose actual aggregate script execution budget is `(N_sub_txs × maxTxExUnits_per_sub_tx)` — unbounded by any protocol parameter — to be accepted as valid. This directly violates the intended resource-limit design parameters, matching the **Medium** impact class: *"Attacker-controlled transactions... exceed intended validation limits."*

---

### Likelihood Explanation

The attack requires only an unprivileged transaction sender who can construct a Dijkstra-era top-level transaction embedding multiple sub-transactions with large Plutus redeemer budgets, and a block producer willing to include it (or a block producer who is themselves the attacker). No privileged access, key compromise, or consensus majority is needed. The Dijkstra era is the newest era in this repository and sub-transactions are a novel feature, making this class of omission plausible.

---

### Recommendation

1. **Extend `totExUnits` (or introduce a Dijkstra-specific variant)** to recursively sum ExUnits across all sub-transactions embedded in a top-level transaction:

   ```haskell
   totExUnitsBatch tx =
     totExUnits tx
       <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
   ```

2. **Replace the call in `dijkstraUtxoTransition`** at the `validateExUnitsTooBigUTxO` site with a batch-aware check using the above function.

3. **Replace the call in `dijkstraBbodyTransition`** (`Alonzo.validateExUnits @era txs`) with a version that folds `totExUnitsBatch` over the top-level transaction sequence.

4. **Add a per-sub-transaction ExUnits cap** in `dijkstraSubUtxoTransition` analogous to `validateExUnitsTooBigUTxO`, so each individual sub-transaction is also bounded by `maxTxExUnits`.

---

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction `txTop` with:
   - Zero redeemers in its own witness set (`totExUnits txTop = ExUnits 0 0`).
   - `K` sub-transactions `[subTx_1 … subTx_K]`, each with redeemers summing to `maxTxExUnits = ExUnits M S`.

2. Submit `txTop`. The UTXO check `validateExUnitsTooBigUTxO pp txTop` evaluates `ExUnits 0 0 ≤ maxTxExUnits` — passes. [3](#0-2) 

3. A block producer includes `txTop` in a block. The BBODY check `Alonzo.validateExUnits txs maxBlockExUnits` computes `foldMap totExUnits [txTop] = ExUnits 0 0 ≤ maxBlockExUnits` — passes. [4](#0-3) 

4. The `SUBLEDGERS` → `SUBLEDGER` → `SUBUTXOW` → `SUBUTXO` pipeline processes each `subTx_i`. No ExUnits check is performed at any point in this pipeline. [6](#0-5) 

5. Actual script execution for the block totals `K × ExUnits M S`, which can be made arbitrarily larger than `maxBlockExUnits` by increasing `K`, while every ledger check passes.
