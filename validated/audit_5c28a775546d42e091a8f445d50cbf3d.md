### Title
Unaccounted `directDeposits` in Dijkstra Era Bypasses ADA Preservation-of-Value Check, Enabling Arbitrary ADA Creation - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in the transaction body that allows a transaction to add ADA directly to registered account balances. However, the `directDeposits` amount is not included in either the `consumed` or `produced` side of the preservation-of-value check (`validateValueNotConservedUTxO`). As a result, an unprivileged transaction author can include arbitrary `directDeposits` in their transaction body, causing ADA to be credited to account balances without being deducted from any UTxO input. This is direct, attacker-controlled creation of ADA from nothing.

---

### Finding Description

**Vulnerability class**: Funds/accounting bug — balance state modified directly without going through the proper conservation accounting path (direct analog to the ERC20 `_balances` direct-write bug where `_totalSupply` is not updated).

**Root cause — three cooperating files:**

**1. `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs` lines 211–216**

`applyDirectDeposits` is called unconditionally after certificate processing. The only pre-check is `directDepositsMissingAccounts`, which only verifies that target accounts are registered — it does not verify that the deposited ADA was paid for by the transaction. [1](#0-0) 

**2. `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubEntities.hs` lines 182–187**

The same pattern is repeated for sub-transactions: `applyDirectDeposits` is applied to the sub-transaction's `directDeposits` field with no conservation accounting. [2](#0-1) 

**3. `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs` lines 93–106 and `eras/mary/impl/src/Cardano/Ledger/Mary/UTxO.hs` lines 69–86**

`dijkstraProducedValue` delegates to `conwayProducedValue` (UTxO outputs + fee + deposits + treasury donation) and does not add `directDeposits`. Similarly, `getConsumedMaryValue` (called by `getConsumedDijkstraValue`) only sums UTxO inputs, minted multi-assets, refunds, and withdrawals — no `directDeposits` term. [3](#0-2) [4](#0-3) 

**4. `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` line 381**

`dijkstraUtxoTransition` calls `validateValueNotConservedUTxO` using `consumed = conwayConsumed` (imported from Conway, which predates `directDeposits`) and `produced` (which calls `getProducedDijkstraValue`). Neither side includes `directDeposits`. [5](#0-4) 

**The conservation invariant that is violated:**

The Shelley/Mary/Conway/Dijkstra preservation-of-value property requires:

> `consumed(pp, utxo, txb) = produced(pp, certState, txb)`

For withdrawals (the symmetric operation), the withdrawn amount is included in `consumed` so that the UTxO outputs can absorb it. For `directDeposits` (the reverse: ADA flowing from UTxO into account balances), the deposited amount must appear in `produced`. It does not. [6](#0-5) [7](#0-6) 

**`applyDirectDeposits` itself:**

The function directly increments `balanceAccountStateL` for each entry in the `DirectDeposits` map with no conservation guard. [8](#0-7) 

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

An attacker submits a Dijkstra-era transaction with:
- UTxO inputs and outputs that balance normally (conservation check passes)
- A `directDeposits` field crediting an arbitrary amount of ADA to their own registered account

Because `directDeposits` appears in neither `consumed` nor `produced`, the conservation check passes. `applyDirectDeposits` then unconditionally adds the specified ADA to the target account balance. The attacker can subsequently withdraw this ADA in a following transaction. The total ADA supply increases without any corresponding deduction from reserves, treasury, or UTxO.

This matches the allowed impact: *"Critical. Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition."*

---

### Likelihood Explanation

**High.** The entry path requires only the ability to submit a valid Dijkstra-era transaction. No privileged key, governance majority, or special role is needed. The `directDeposits` field is a standard part of the `DijkstraTxBodyRaw` structure, serializable by any transaction author. The only prerequisite is that the target account address is registered — which the attacker can satisfy by registering their own staking credential first. [9](#0-8) [10](#0-9) 

---

### Recommendation

Include `directDeposits` in the `produced` side of the preservation-of-value calculation, mirroring how `withdrawals` is included in `consumed`. Concretely:

- In `dijkstraProducedValue` (and `getProducedDijkstraValue`), add the sum of all `directDeposits` amounts from the top-level transaction body and each sub-transaction body to the produced value.
- Equivalently, add a `directDeposits` term to `getConsumedDijkstraValue` on the consumed side (with the opposite sign convention), consistent with the existing treatment of `withdrawals`.
- Add a corresponding property-based test asserting that the total ADA in (UTxO + accounts + fees + deposits + treasury + reserves) is unchanged after applying a Dijkstra transaction that contains `directDeposits`.

The fix is symmetric to the existing withdrawal accounting:

```
consumed = UTxO_inputs + mint + refunds + withdrawals
produced = UTxO_outputs + fee + deposits + treasury_donation + directDeposits  ← add this
```

### Proof of Concept

1. Register a staking credential `K` on the Dijkstra ledger (standard `RegDepositTxCert`).
2. Construct a Dijkstra top-level transaction `tx` with:
   - `spendInputs`: any UTxO input worth `F` lovelace
   - `outputs`: one output returning `F - fee` lovelace to any address
   - `fee`: `fee` lovelace (satisfies `validateFeeTooSmallUTxO`)
   - `directDeposits`: `{ AccountAddress(K) → 1_000_000_000_000 }` (1 million ADA)
   - All other fields empty/default
3. Submit `tx`. `validateValueNotConservedUTxO` computes:
   - `consumed = F` (UTxO input only; `directDeposits` absent)
   - `produced = (F - fee) + fee = F` (UTxO output + fee; `directDeposits` absent)
   - Check passes: `F == F`.
4. `dijkstraEntitiesTransition` calls `applyDirectDeposits`, crediting 1 million ADA to account `K`.
5. In a subsequent transaction, submit a withdrawal for `K`'s full balance. 1 million ADA is now spendable in the UTxO — created from nothing. [11](#0-10) [12](#0-11) [13](#0-12)

```haskell
-- | Ensure that value consumed and produced matches up exactly
--
-- > consumed pp utxo txb = produced pp poolParams txb
validateValueNotConservedUTxO ::
  (EraUTxO era, EraCertState era) =>
  PParams era ->
  UTxO era ->
  CertState era ->
  TxBody TopTx era ->
  Test (ShelleyUtxoPredFailure era)
validateValueNotConservedUTxO pp utxo certState txBody =
  failureUnless (consumedValue == producedValue) $
    ValueNotConservedUTxO Mismatch {mismatchSupplied = consumedValue, mismatchExpected = producedValue}
  where
    consumedValue = consumed pp certState utxo txBody
    producedValue = produced pp certState txBody
```
