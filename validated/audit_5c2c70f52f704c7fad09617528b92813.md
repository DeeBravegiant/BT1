Audit Report

## Title
Chainlink Price Feed Staleness Not Validated, Enabling Oracle Rate Abuse — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` and `answeredInRound` return values, accepting any price the feed returns regardless of age. A stale price propagates directly into the deposit mint-ratio calculation (`getRsETHAmountToMint`) and into the rsETH price update (`_updateRsETHPrice`), allowing depositors to receive systematically too many or too few rsETH tokens relative to the true asset value.

## Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at line 52, `getAssetPrice()` destructures only the `answer` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`updatedAt` (position 4) and `answeredInRound` (position 5) are discarded. No `require(block.timestamp - updatedAt <= threshold)` and no `require(answeredInRound >= roundId)` check exists anywhere in the file or in its callers.

The stale price propagates through the following on-chain call chain:

1. `ChainlinkPriceOracle.getAssetPrice(asset)` returns the stale price.
2. `LRTOracle._getTotalEthInProtocol()` (lines 336–343) multiplies the stale price by `getTotalAssetDeposits(asset)` for every supported LST to compute `totalETHInProtocol`.
3. `LRTOracle._updateRsETHPrice()` (line 250) derives `newRsETHPrice` from the stale TVL.
4. `LRTDepositPool.getRsETHAmountToMint()` (line 520) computes `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` — the numerator uses the live (stale) Chainlink price at deposit time, while the denominator uses the stored `rsETHPrice`.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (lines 252–266) only fires when `newRsETHPrice` exceeds `highestRsethPrice` by more than the configured threshold. A stale price that was already near the current price at the time of staleness, or one that drifts gradually, bypasses this guard entirely. The downside-protection pause (lines 270–281) is reactive and does not prevent the mispricing from affecting deposits that occur before the pause is triggered.

## Impact Explanation

**High — Theft of unclaimed yield**

*Inflated stale price scenario:* `_getTotalEthInProtocol()` overstates TVL → `rsETHPrice` is computed too high → new depositors receive fewer rsETH than their deposit value entitles them to → existing rsETH holders are enriched at depositors' expense (theft of yield from depositors).

*Deflated stale price scenario:* `_getTotalEthInProtocol()` understates TVL → `rsETHPrice` is computed too low → new depositors receive excess rsETH relative to their deposit value → all existing rsETH holders are diluted (theft of yield from existing holders). If the deflation is severe enough and the `pricePercentageLimit` guard does not trigger (e.g., because the stale price is within the threshold), this can compound into protocol insolvency.

Both scenarios are triggered without any privileged action.

## Likelihood Explanation

**Medium**

Chainlink feeds can go stale due to network congestion preventing oracle updates, node operator failures, or a feed being deprecated or paused without the protocol updating its configuration. The protocol supports multiple LST assets (stETH, ETHx, rETH, sfrxETH, swETH), each with its own independent price feed. The probability that at least one feed experiences a staleness event over the protocol's lifetime is non-trivial. No privileged action is required — any depositor calling `depositAsset()` or `depositETH()` during a staleness window triggers the mispricing.

## Recommendation

Add staleness and round-completeness validation in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= STALENESS_THRESHOLD, "Stale price");
```

Set `STALENESS_THRESHOLD` per feed based on the feed's documented heartbeat (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer). Consider storing per-asset thresholds in the contract alongside `assetPriceFeed`.
