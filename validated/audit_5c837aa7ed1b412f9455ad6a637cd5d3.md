Audit Report

## Title
Unchecked Caller in `claim()` Enables Forced Fee Extraction via Front-Running — (`File: contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary

`MerkleDistributor.claim()` accepts an arbitrary `account` parameter with no `msg.sender == account` check, allowing any caller to trigger a claim on behalf of any address. When `feeInBPS > 0`, an attacker can front-run a `setFeeInBPS` reduction by force-claiming for all eligible addresses at the current high fee rate, permanently extracting the fee difference from victims' allocations. The sister contract `KernelMerkleDistributor` explicitly prevents this with an `account != msg.sender` revert; `MerkleDistributor` does not.

## Finding Description

`MerkleDistributor.claim()` is fully permissionless and accepts `account` as a caller-supplied parameter with no identity check:

```solidity
// contracts/utils/MerkleDistributor/MerkleDistributor.sol L97-106
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
```

The entire function body (L107–L147) contains zero caller-identity validation. The fee is deducted and the claim state is permanently updated:

```solidity
// L134-144
userClaims[account].lastClaimedIndex = index;
userClaims[account].cumulativeAmount = cumulativeAmount;

uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);
```

Once `userClaims[account].lastClaimedIndex` is set, `isClaimed()` returns `true` and any subsequent claim attempt reverts with `AlreadyClaimed`. The fee extraction is irreversible.

By contrast, `KernelMerkleDistributor._processClaim()` (L311–313) enforces:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

The root cause is the deliberate omission of this guard in `MerkleDistributor`.

**Concrete front-running exploit path:**
1. `feeInBPS = 1000` (10%). Owner submits `setFeeInBPS(0)`.
2. Attacker observes the pending tx in the mempool.
3. Attacker front-runs: calls `claim(index, victimAddress, cumulativeAmount, proof)` for every eligible address.
4. Each victim receives 90% of their allocation; 10% is sent to `protocolTreasury`. `userClaims[victim].lastClaimedIndex` is set.
5. Owner's `setFeeInBPS(0)` confirms. Victims attempt to claim — all revert with `AlreadyClaimed`.
6. Victims permanently lose the 10% they would have received in full had they claimed after the fee reduction.

## Impact Explanation

**High — Theft of unclaimed yield.**

Victims are entitled to their full allocation at the fee rate in effect when they choose to claim. The forced claim at a higher fee rate permanently reduces their receivable yield by up to 10% (`MAX_FEE_IN_BPS = 1000`). The loss is irreversible: the claim slot is consumed, the fee is transferred to `protocolTreasury`, and no recovery path exists. This directly matches the "High. Theft of unclaimed yield." impact class.

## Likelihood Explanation

**High.** The attack requires no role, no signature, and no prior state. All inputs (`index`, `account`, `cumulativeAmount`, `merkleProof`) are derivable from public merkle tree data. The attack is a single transaction costing only gas. Fee-reduction governance actions are observable in the mempool, making front-running straightforward and repeatable across all eligible addresses in one block.

## Recommendation

Add the same caller check already present in `KernelMerkleDistributor._processClaim()`:

```solidity
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
+   if (account != msg.sender) revert Unauthorized();
    ...
}
```

Also add `Unauthorized` to the `IMerkleDistributor` interface error list for consistency.
