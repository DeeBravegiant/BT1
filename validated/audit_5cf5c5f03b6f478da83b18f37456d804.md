Audit Report

## Title
Stale `rsETHPrice` in `LRTOracle` Allows Depositors to Mint Excess rsETH at the Expense of Existing Holders — (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides a live asset price by the stored `rsETHPrice` state variable, which is only updated when `updateRSETHPrice()` is explicitly called — never atomically inside `depositETH()` or `depositAsset()`. When underlying LSTs appreciate between price updates, any depositor can mint more rsETH than the true exchange rate warrants, permanently diluting existing holders' accrued yield.

## Finding Description
`getRsETHAmountToMint()` at [1](#0-0)  computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`getAssetPrice(asset)` reads live from an external price fetcher, while `rsETHPrice` is a stored state variable [2](#0-1)  that is only refreshed inside `_updateRsETHPrice()`, which is only reachable via the public `updateRSETHPrice()` [3](#0-2)  or the manager-only `updateRSETHPriceAsManager()`.

Neither `depositETH()` [4](#0-3)  nor `depositAsset()` [5](#0-4)  calls `updateRSETHPrice()` or `_updateRsETHPrice()` before computing the mint amount. Both simply call `_beforeDeposit()` → `getRsETHAmountToMint()` with the cached price.

The stored price is computed as: [6](#0-5) 

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

When LSTs accrue staking rewards, `totalETHInProtocol` rises immediately but `rsETHPrice` remains at its last stored value. The denominator in `getRsETHAmountToMint()` is artificially low, so the depositor receives more rsETH than the true exchange rate warrants.

The same stale price is used in `LRTWithdrawalManager.getExpectedAssetAmount()` [7](#0-6) :

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

called directly from `instantWithdrawal()` [8](#0-7)  before burning rsETH, creating a symmetric stale-high withdrawal path.

An additional amplifier: when price appreciation exceeds `pricePercentageLimit`, non-manager callers of `updateRSETHPrice()` receive a `PriceAboveDailyThreshold` revert [9](#0-8) , blocking the public update path and extending the stale window indefinitely until a manager acts.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders accumulate yield as underlying LSTs appreciate. A depositor acting while `rsETHPrice` is stale captures a portion of that accrued yield by receiving excess rsETH. After `updateRSETHPrice()` is called, the new price is permanently lower than it would have been without the dilutive deposit, reducing the value of every pre-existing rsETH token. The attack is repeatable at every price-update cycle and scales linearly with deposit size. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation
No privileged access is required. `updateRSETHPrice()` is never called inside any deposit path, so a non-zero stale window always exists after any LST yield accrual. Any depositor can observe on-chain that `rsETHPrice` is stale by comparing `lrtOracle.rsETHPrice()` against a freshly computed TVL/supply ratio, then deposit without front-running any specific transaction. The `pricePercentageLimit` guard can further widen the window by blocking public updates when appreciation is large, making the attack more profitable and more persistent.

## Recommendation
1. Call `_updateRsETHPrice()` atomically at the start of `depositETH()`, `depositAsset()`, and `instantWithdrawal()` before computing the exchange rate, so the mint/burn always uses a fresh price.
2. Alternatively, derive the rsETH amount directly from the live TVL and current supply rather than from the cached `rsETHPrice` state variable.
3. Ensure the `pricePercentageLimit` guard does not permanently block public updates; consider a time-based override so that a sufficiently long elapsed time always permits a non-manager update.

## Proof of Concept
**Setup:**
- `totalETHInProtocol` = 1,010 ETH (stETH appreciated 1%)
- `rsethSupply` = 1,000 rsETH
- `rsETHPrice` (stale) = 1.000 ETH (true value = 1.010 ETH)

**Steps:**
1. Attacker calls `depositETH{value: 100 ether}(0, "")` before `updateRSETHPrice()` is called.
2. `getRsETHAmountToMint` returns `100e18 * 1e18 / 1.000e18 = 100 rsETH`. Correct amount: `100e18 * 1e18 / 1.010e18 ≈ 99.01 rsETH`. Attacker receives ~0.99 rsETH excess.
3. After deposit: `totalETH = 1,110`, `rsethSupply = 1,100`, true price = `1,110 / 1,100 ≈ 1.009 ETH`.
4. `updateRSETHPrice()` is called; `rsETHPrice` set to 1.009 ETH.
5. Attacker's 100 rsETH is worth `100 × 1.009 = 100.9 ETH` — a 0.9 ETH profit on a 100 ETH deposit.
6. Existing 1,000 rsETH holders collectively hold `1,000 × 1.009 = 1,009 ETH` instead of the 1,010 ETH they were entitled to — a 1 ETH loss extracted by the attacker.

**Foundry fork test plan:** Fork mainnet, set `rsETHPrice` to a value below the live TVL/supply ratio (simulating a stale price after LST yield accrual), call `depositETH` as an unprivileged address, then call `updateRSETHPrice`, and assert that the attacker's rsETH balance × new price exceeds their deposit value while pre-existing holders' proportional claim has decreased.
