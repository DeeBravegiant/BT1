### Title
Unverified `respond_ckd` Response for `AppPublicKey` Variant Allows Any Attested Participant to Corrupt Pending CKD Requests - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in `MpcContract` performs cryptographic output verification **only** for the `CKDAppPublicKey::AppPublicKeyPV` variant. For the `CKDAppPublicKey::AppPublicKey` variant, no verification is performed on the submitted `CKDResponse`. Any single attested participant — a Byzantine node strictly below the signing threshold — can call `respond_ckd` with a fabricated `(big_y, big_c)` pair targeting any pending `AppPublicKey` CKD request, permanently resolving all queued yields with garbage data and preventing legitimate nodes from ever delivering the correct confidential key.

---

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs:653–689`), after confirming the caller is an attested participant, the contract conditionally verifies the response:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKey` requests, the branch is a no-op: any `big_y` and `big_c` values pass through. `resolve_yields_for` then **removes** the entire pending queue for that request key and resumes every queued yield with the attacker-supplied bytes:

```rust
let resumed = requests
    .remove(request)          // ← permanently removes the entry
    .unwrap_or_default()
    .into_iter()
    .map(|YieldIndex { data_id }| {
        env::promise_yield_resume(&data_id, response_bytes.clone());
    })
    .count();
``` [2](#0-1) 

After this call, the request no longer exists in `pending_ckd_requests`. Any subsequent legitimate `respond_ckd` from the honest MPC nodes returns `Err(RequestNotFound)`.

Contrast with `respond` (ECDSA/EdDSA signatures), which always verifies the signature before resolving:

```rust
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
pending_requests::resolve_yields_for(...)
``` [3](#0-2) 

And contrast with `respond_ckd` for `AppPublicKeyPV`, which runs `ckd_output_check` — a pairing-based check that verifies `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`: [4](#0-3) 

The `AppPublicKey` variant is "privately verifiable" by design (only the holder of the app secret key can verify correctness), so no equivalent on-chain check exists. This design asymmetry is not mitigated by any access control that restricts *which* attested participant may respond to *which* request.

The attacker can reconstruct the exact `CKDRequest` map key from the on-chain `request_app_private_key` transaction (which exposes `predecessor_id`, `derivation_path`, `app_public_key`, and `domain_id`), and can confirm the request is still pending via the public `get_pending_ckd_request` view method. [5](#0-4) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe a pending `request_app_private_key` call using the `AppPublicKey` variant on-chain.
2. Reconstruct the `CKDRequest` key from the transaction arguments.
3. Call `respond_ckd` with arbitrary `big_y` and `big_c` values before the honest MPC nodes respond.
4. The contract accepts the garbage response and resolves all queued yields with it.
5. The user receives a cryptographically invalid confidential key (`big_c` and `big_y` are not a valid CKD output for their `app_public_key`).
6. Legitimate MPC nodes can no longer respond — the request is gone from `pending_ckd_requests`.
7. Any data the user subsequently encrypts to the expected derived key is permanently inaccessible; any data already encrypted to it cannot be decrypted.

This breaks the production safety invariant that only the threshold-honest MPC network can produce a valid CKD response for a user's request.

**Impact: Medium** — request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attacker needs only to be a single attested participant — one Byzantine node below the signing threshold suffices.
- All information needed to construct the attack (the `CKDRequest` key) is publicly visible on-chain.
- The attacker must race the honest MPC nodes, but since they can submit immediately upon observing the `request_app_private_key` receipt, this race is winnable, especially if the attacker is monitoring the chain in real time (as all MPC nodes do via the indexer).
- The `AppPublicKey` variant is the legacy/simpler variant and is actively used in production (the `AppPublicKeyPV` variant is newer).

---

### Recommendation

1. **Restrict `respond_ckd` to a designated leader per request.** The node coordinator already computes a per-request leader selection order (`QueuedRequest::leader_selection_order`). Enforce on-chain that only the current leader (or a threshold-agreed set) may call `respond_ckd`, analogous to how signing sessions are coordinated off-chain.

2. **Alternatively, require all new CKD requests to use `AppPublicKeyPV`.** Since `AppPublicKeyPV` responses are publicly verifiable on-chain via `ckd_output_check`, they are immune to this attack. Deprecate `AppPublicKey` for new requests.

3. **At minimum, document the trust assumption** that `AppPublicKey` CKD responses are unverifiable on-chain and that the security of this variant depends entirely on all participants being honest — which contradicts the Byzantine-fault-tolerant threat model.

         AppPublicKey(some_g1_point),

         predecessor_id="alice.near",
         derivation_path="my/path",

            &mut self.pending_ckd_requests,
            &request,

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
