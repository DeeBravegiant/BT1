Audit Report

## Title
Unvalidated Chainlink `latestRoundData()` Return Values Enable Stale-Price-Triggered Protocol Freeze and Incorrect rsETH Minting - (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and silently discards `roundId`, `updatedAt`, and `answeredInRound`, performing no staleness, incomplete-round, or non-negative price checks. A stale or zero price returned by any supported asset's Chainlink feed propagates into two critical paths: the public `updateRSETHPrice()` function, which can trigger an automatic protocol-wide pause (temporary fund freeze), and `depositAsset()`/`depositETH()`, which can cause incorrect rsETH minting amounts.

## Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at L49–55:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

All of `roundId`, `updatedAt`, and `answeredInRound` are discarded. No check is made that `price > 0`, `answeredInRound >= roundId`, or `updatedAt > 0`.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` at L30–32 applies all three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

This inconsistency within the same repository confirms the omission in `ChainlinkPriceOracle` is an oversight.

**Exploit path — temporary fund freeze:**

1. A supported asset's Chainlink feed enters a stale round (`answeredInRound < roundId`) and returns `price = 0` or a significantly outdated value.
2. Any unprivileged caller invokes `LRTOracle.updateRSETHPrice()` (public, no access control — L87).
3. `_updateRsETHPrice()` calls `_getTotalEthInProtocol()` (L231), which iterates over all supported assets and calls `getAssetPrice(asset)` (L339), delegating to `ChainlinkPriceOracle.getAssetPrice()`.
4. A zero price for any asset causes `totalETHInProtocol` to be understated by that asset's full ETH contribution.
5. `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)` (L250) drops below `highestRsethPrice`.
6. If the drop exceeds `pricePercentageLimit`, the downside-protection logic at L277–281 executes: `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()` are called — all deposits and withdrawals are frozen until an admin manually unpauses.

**Exploit path — incorrect rsETH minting:**

`LRTDepositPool.getRsETHAmountToMint()` at L520 computes:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
A stale non-zero price (e.g., 10% above actual) causes depositors to receive proportionally more rsETH than they should, diluting existing holders. A stale price below actual causes depositors to receive less rsETH than owed.

**Existing guards are insufficient:** `updatePriceOracleForValidated()` (L101–108) performs a one-time sanity check at oracle registration time (`price > 1e16 && price < 1e19`) but provides no protection against prices that become stale after registration.

## Impact Explanation

**Primary — Medium: Temporary freezing of funds.** Any public caller can invoke `updateRSETHPrice()` at any time a supported Chainlink feed is stale. If the resulting computed price drop exceeds `pricePercentageLimit`, `LRTDepositPool` and `WithdrawalManager` are automatically paused, freezing all user deposits and withdrawals until an admin manually unpauses. This is a concrete, externally triggerable temporary fund freeze.

**Secondary — Low: Contract fails to deliver promised returns.** A stale non-zero price causes `depositAsset()`/`depositETH()` to mint an incorrect rsETH amount — either over-minting (diluting existing holders) or under-minting (loss to the depositor) — without any error or revert.

## Likelihood Explanation

Chainlink feeds can return stale data during network congestion, oracle node downtime, or feed deprecation — all documented, real-world failure modes requiring no privileged access or oracle operator compromise. `updateRSETHPrice()` has no access control and can be called by any EOA or contract. `depositAsset()` and `depositETH()` are standard user-facing entry points. The vulnerability is continuously exploitable whenever any supported feed is stale, and requires no special setup beyond the feed entering a stale state.

## Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.sol` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    require(price > 0, "ChainlinkPriceOracle: price <= 0");
    require(answeredInRound >= roundId, "ChainlinkPriceOracle: stale price");
    require(updatedAt > 0, "ChainlinkPriceOracle: incomplete round");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, consider a per-feed heartbeat check (`block.timestamp - updatedAt <= MAX_DELAY`) to guard against feeds that are technically complete but temporally stale.

## Proof of Concept

**Foundry fork test outline (temporary freeze path):**

1. Fork mainnet at a block where a supported asset's Chainlink feed has `answeredInRound < roundId` (or mock `latestRoundData()` to return `price = 0`).
2. Deploy/use the existing `ChainlinkPriceOracle` with `pricePercentageLimit > 0` set on `LRTOracle`.
3. Call `LRTOracle.updateRSETHPrice()` from an unprivileged EOA.
4. Assert that `LRTDepositPool.paused() == true` and `WithdrawalManager.paused() == true`.
5. Confirm that a subsequent `depositAsset()` call reverts with `Pausable: paused`.

**Foundry unit test outline (incorrect minting path):**

1. Mock `latestRoundData()` to return a stale price 10% above the true price (with `answeredInRound < roundId`).
2. Call `depositAsset()` with a known `depositAmount`.
