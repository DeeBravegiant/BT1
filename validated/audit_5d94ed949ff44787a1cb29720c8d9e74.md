Audit Report

## Title
Missing `try/catch` on `_withdrawFromAave` in `_processWithdrawalCompletion` Causes Temporary Freezing of User Funds — (File: contracts/LRTWithdrawalManager.sol)

## Summary
When Aave integration is enabled, `_processWithdrawalCompletion` calls `_withdrawFromAave` at line 724 without any `try/catch` guard. If Aave's pool is paused or frozen, the call reverts and the user cannot complete their withdrawal. Because rsETH was already burned during `unlockQueue` (Phase 1), the user holds no rsETH and cannot re-initiate; their ETH is locked in Aave until the pool resumes. This is an asymmetry with the Aave deposit path in `unlockQueue`, which correctly uses `try/catch`.

## Finding Description
The withdrawal lifecycle has two phases. In Phase 1, `unlockQueue` burns rsETH and deposits ETH to Aave using a `try/catch`:

```solidity
// LRTWithdrawalManager.sol L305
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
``` [1](#0-0) 

In Phase 2, `_processWithdrawalCompletion` calls `_withdrawFromAave` with no error handling:

```solidity
// LRTWithdrawalManager.sol L724
_withdrawFromAave(amountNeeded);   // ← no try/catch
``` [2](#0-1) 

If `_withdrawFromAave` reverts (e.g., Aave pool paused), the entire `completeWithdrawal` transaction reverts. Because Solidity reverts undo all state changes, the `delete withdrawalRequests[requestId]` at line 712 and the `popFront()` at line 705 are also rolled back — the withdrawal request survives. However, the rsETH burn from Phase 1 is permanent and already committed in a prior transaction. The user holds no rsETH and cannot re-initiate a new withdrawal; their ETH is inaccessible in Aave for the duration of the pause.

Note: the PoC's claim that "the request is permanently deleted" is incorrect — the revert restores it. The actual harm is the temporary freeze: rsETH is gone, ETH is in Aave, and `completeWithdrawal` will keep reverting until Aave resumes. [3](#0-2) 

## Impact Explanation
**Medium — Temporary freezing of funds.** Users whose withdrawal requests were unlocked while Aave integration was active cannot retrieve their ETH for as long as Aave remains paused. Their rsETH is already burned and irrecoverable. Recovery requires either Aave resuming or an admin calling `emergencyWithdrawFromAave` (PAUSER_ROLE gated) and then disabling Aave integration, which is not guaranteed to be timely. [4](#0-3) 

## Likelihood Explanation
Requires: (1) `isAaveIntegrationEnabled = true`, an operator-configured state; (2) Aave's guardian pausing the pool after `unlockQueue` runs but before users call `completeWithdrawal`. Aave V3 has a documented guardian-controlled emergency pause that has been activated on mainnet. The time gap between `unlockQueue` and `completeWithdrawal` (enforced by `withdrawalDelayBlocks`) creates a window during which this can occur. No attacker action is required — the freeze is triggered by a normal user calling a public function during an Aave pause.

## Recommendation
Mirror the `try/catch` pattern already used for the Aave deposit in `unlockQueue`. Wrap `_withdrawFromAave` in `_processWithdrawalCompletion` with a `try/catch`; on failure, revert with a descriptive error (e.g., `AaveWithdrawalFailed`) so the user can retry later when Aave resumes. Alternatively, add an admin path to disable Aave integration and flush aWETH back to the contract so pending withdrawals can be completed without Aave. [5](#0-4) 

## Proof of Concept
1. `isAaveIntegrationEnabled = true`.
2. Alice calls `initiateWithdrawal(ETH, 1 ether rsETH, ...)`. rsETH is transferred to the contract.
3. Operator calls `unlockQueue(ETH, ...)`. Alice's rsETH is burned (line 305); 1 ETH is redeemed from the unstaking vault and deposited into Aave via `try this.depositToAaveExternal(...)` (lines 310–317). Both succeed.
4. Aave's guardian pauses the pool (external, no protocol action required).
5. Alice calls `completeWithdrawal(ETH, ...)`. `_processWithdrawalCompletion` is entered; `contractBalance = 0 < 1 ETH`, so `_withdrawFromAave(1 ether)` is called at line 724. Aave's `withdraw` reverts because the pool is paused.
6. The entire transaction reverts. Alice's withdrawal request is restored (revert semantics). However, Alice's rsETH is permanently burned from step 3. She cannot re-initiate a withdrawal and cannot call `completeWithdrawal` successfully until Aave unpauses. Her 1 ETH is frozen in Aave.
7. Repeat step 5 on every block — it reverts every time until Aave resumes or an admin intervenes. [2](#0-1)
