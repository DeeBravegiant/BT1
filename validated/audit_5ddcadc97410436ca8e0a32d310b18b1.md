### Title
Missing Identity-Point Validation in CKD App Public Key Check Exposes Derived Key Material On-Chain - (File: crates/contract/src/primitives/ckd.rs)

### Summary

The `app_public_key_check` function in the Confidential Key Derivation (CKD) flow does not reject the BLS12-381 identity (zero) points as an app public key. Because `e(identity_G1, G2) = 1 = e(G1, identity_G2)`, the pairing equation is trivially satisfied by the identity pair. When a caller submits `(pk1=G1_identity, pk2=G2_identity)`, the encryption term in the CKD protocol vanishes entirely, and the MPC network's response `C` becomes the raw, unencrypted BLS signature `msk · H(pk, app_id)` — the derived secret key — posted in plaintext on-chain for any observer to read.

### Finding Description

`app_public_key_check` verifies `e(pk1, G2) = e(G1, pk2)` but performs no check that either point is non-identity:

```rust
// crates/contract/src/primitives/ckd.rs:62-74
pub(crate) fn app_public_key_check(app_public_key: &dtos::CKDAppPublicKeyPV) -> bool {
    let pk1 = env::bls12381_p1_decompress(&app_public_key.pk1);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pairing_input = [
        pk1.as_slice(), MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        G1_GENERATOR_UNCOMPRESSED.as_slice(), pk2.as_slice(),
    ].concat();
    env::bls12381_pairing_check(&pairing_input)
}
```

The codebase itself documents this gap with a test that asserts the identity pair is **accepted**:

```rust
// crates/contract/src/primitives/ckd.rs:479-495
/// Documents the pre-existing behavior that identity key pairs satisfy
/// the pairing equation and are accepted.
fn app_public_key_check__should_accept_identity_key_pair() {
    let app_pk = dtos::CKDAppPublicKeyPV {
        pk1: dtos::Bls12381G1PublicKey(G1Projective::identity().to_compressed()),
        pk2: dtos::Bls12381G2PublicKey(G2Projective::identity().to_compressed()),
    };
    let accepted = app_public_key_check(&app_pk);
    assert!(accepted);   // ← passes: identity pair is not rejected
}
```

The CKD protocol encrypts the derived key as `C = msk·H(pk, app_id) + a·Y`, where `a` is the caller's private scalar and `Y` is a random blinding point. When `pk1 = a·G1 = identity`, it follows that `a = 0`. Every node computes `C_i = S_i + y_i·identity = S_i`, so the coordinator aggregates `C = msk·H(pk, app_id)` — the raw BLS signature with no encryption. The `ckd_output_check` also passes with `pk2 = identity` because the equation reduces to standard BLS signature verification, which `C` satisfies by construction:

```rust
// crates/contract/src/primitives/ckd.rs:80-102
pub(crate) fn ckd_output_check(...) -> bool {
    // With pk2=identity: e(C,-G2)·e(Y,identity)·e(H,pk) = 1
    // simplifies to: e(C,G2) = e(H(pk,app_id), pk)  ← BLS verify, always true for C=msk·H
    env::bls12381_pairing_check(&pairing_input)
}
```

### Impact Explanation

The derived key `msk·H(pk, app_id)` is the secret output of the CKD protocol — it is the BLS signature that the TEE app uses to derive its private key via HKDF. With `a = 0`, the response `(Y, C)` posted on-chain contains `C` equal to this secret in plaintext. Any blockchain observer can read `C` directly from the transaction receipt without knowing any private key. This completely defeats the confidentiality guarantee of CKD: the secret material derived from the MPC master key share is disclosed to the public.

This matches the allowed critical impact: **unauthorized access to MPC key derivation output / secret material that materially enables secret recovery**.

### Likelihood Explanation

The attack requires only a single unprivileged NEAR account call to `request_app_private_key` with a crafted identity-point pair. No threshold collusion, TEE access, or privileged role is needed. The codebase already contains a passing test that confirms the identity pair is accepted, meaning the vulnerable path is reachable with zero cryptographic effort.

### Recommendation

Add an explicit identity-point rejection in `app_public_key_check` before the pairing check:

```rust
pub(crate) fn app_public_key_check(app_public_key: &dtos::CKDAppPublicKeyPV) -> bool {
    // Reject identity points: a=0 would strip all encryption from the CKD output.
    let pk1_bytes = app_public_key.pk1.as_ref();
    let pk2_bytes = app_public_key.pk2.as_ref();
    if is_bls12381_g1_identity(pk1_bytes) || is_bls12381_g2_identity(pk2_bytes) {
        return false;
    }
    // ... existing pairing check ...
}
```

The companion test `app_public_key_check__should_accept_identity_key_pair` must be updated to assert `false`.

### Proof of Concept

1. Caller invokes `request_app_private_key` with:
   - `pk1 = G1Projective::identity().to_compressed()` (the compressed BLS12-381 G1 identity point)
   - `pk2 = G2Projective::identity().to_compressed()` (the compressed BLS12-381 G2 identity point)
2. `app_public_key_check` returns `true` — confirmed by the existing test at line 483.
3. The CKD request is stored on-chain; MPC nodes pick it up and compute `C_i = x_i · H(pk, app_id)` (no blinding, since `y_i · identity = identity`).
4. The coordinator aggregates `C = msk · H(pk, app_id)` and `Y = Σ(y_i · G1)`.
5. `ckd_output_check` returns `true` because `e(C, G2) = e(H(pk, app_id), pk)` is exactly BLS signature verification, which holds for `C = msk · H(pk, app_id)`.
6. The contract posts `(Y, C)` on-chain. `C` is the raw derived key in plaintext.
7. Any observer reads `C` from the NEAR transaction receipt and uses it directly as the derived BLS key, bypassing the intended confidentiality of the CKD protocol.
