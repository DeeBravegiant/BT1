Audit Report

## Title
No Slippage Protection in L2 Pool `deposit` Functions Allows Users to Receive Fewer rsETH Than Expected - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

## Summary
All four L2 deposit pool contracts expose public `deposit` entry points that compute the rsETH/wrsETH output amount entirely from a live oracle rate at execution time, with no caller-supplied minimum output parameter. The L1 `LRTDepositPool` correctly guards both deposit paths with `minRSETHAmountExpected`. Because the L2 oracle rate is updated periodically by operators, a user's transaction can be mined after a rate update, silently delivering fewer rsETH tokens than the user observed when constructing the transaction, with no on-chain recourse.

## Finding Description
In `RSETHPoolV3.sol`, both ETH and token deposit paths compute output via `viewSwapRsETHAmountAndFee`, which reads the live oracle rate with no floor check:

```solidity
// RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();          // live oracle read
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}

**Exploit flow:**
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain at oracle rate R1 and observes expected output X = `1e18 / R1`.
2. User submits `deposit{value: 1 ether}("ref")` to `RSETHPoolV3`.
3. Before the transaction is included, the operator updates the oracle to rate R2 > R1 (rsETH has accrued yield; each rsETH is now worth more ETH).
4. User's transaction executes: `viewSwapRsETHAmountAndFee` returns Y = `1e18 / R2 < X`.
5. User receives Y wrsETH instead of X, with no revert and no recourse.

No existing guard prevents this: `nonReentrant` prevents reentrancy, `whenNotPaused` prevents execution when paused, and `limitDailyMint` caps total daily volume â€” none of these check the per-user minimum output.

## Impact Explanation
The depositor receives fewer rsETH/wrsETH tokens than they observed and intended to transact at. Their deposited collateral (ETH or LST) is fully transferred to the pool and will be bridged to L1; the shortfall in rsETH token count is unrecoverable. This matches **Low: Contract fails to deliver promised returns, but doesn't lose value** â€” the deposited asset's ETH value is preserved in the rsETH received, but the user receives fewer rsETH tokens than the rate they observed, with no ability to revert.

## Likelihood Explanation
**Medium.** The rsETH oracle rate increases monotonically as the protocol accrues yield and is updated periodically by operators. On L2s with short block times (Arbitrum, Optimism, Base, Unichain), an oracle update transaction and a pending user deposit can land in the same or adjacent blocks with no attacker involvement. Any user depositing a non-trivial amount is exposed on every transaction. No privileged attacker, no front-running, and no victim mistake is required â€” the rate drift is a natural consequence of the oracle update cadence.

## Recommendation
Add a `minRsETHAmountExpected` parameter to all L2 pool `deposit` functions, mirroring the pattern already used in `LRTDepositPool._beforeDeposit`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert SlippageExceeded();
    wrsETH.mint(msg.sender, rsETHAmount);
    ...
}
```

Apply the same fix to the token-deposit overload in all four affected pool contracts: `RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, and `RSETHPoolV3WithNativeChainBridge`.

## Proof of Concept
**Foundry fork test plan:**
1. Fork an L2 (e.g., Arbitrum) at a block where `RSETHPool` is deployed.
2. Record `rate1 = RSETHPool.getRate()` and compute `expectedRsETH = 1e18 * 1e18 / rate1` (for 1 ETH deposit, zero fee).
3. Simulate an oracle update: call `rsETHOracle.setRate(rate1 * 101 / 100)` (1% increase) from the operator address.
4. Call `RSETHPool.deposit{value: 1 ether}("ref")` from a user address.
5. Assert `wrsETH.balanceOf(user) < expectedRsETH` â€” the user received fewer tokens than observed pre-update.
6. Assert the transaction did not revert â€” confirming no on-chain protection exists.
7. Repeat for the token-deposit overload and for `RSETHPoolV3`, `RSETHPoolNoWrapper`, `RSETHPoolV3WithNativeChainBridge`.
