### Title
Foreign-Tx Signing Always Inherits Presignature Participants Chosen for Liveness, Not Chain Coverage — (`File: crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

`make_verify_foreign_tx_leader` unconditionally reuses the participant set embedded in a pre-generated presignature when assembling the signing group for a `verify_foreign_transaction` request. Those presignature participants were selected for network liveness, not for whether they can actually query the requested foreign chain. Any participant that does not cover the chain produces no signature share, causing the signing session to stall and the presignature to be consumed without producing a result. The accepted request then silently times out.

---

### Finding Description

When the leader node handles a `verify_foreign_transaction` request, it calls `make_verify_foreign_tx_leader`. The first thing it does is dequeue a presignature from the domain's presignature store and extract the participant list that was baked into that presignature at generation time:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs  lines 63-71
let (presignature_id, presignature) = domain_data.presignature_store.take_owned().await;
let participants = presignature.participants.clone();          // ← liveness-based set
let channel = self.ecdsa_signature_provider.new_channel_for_task(
    VerifyForeignTxTaskId::VerifyForeignTx { id, presignature_id },
    participants,                                              // ← used verbatim
)?;
```

Presignatures are generated by `run_background_presignature_generation`, which selects participants via `select_random_active_participants_including_me` — a purely liveness-based random draw from online nodes. There is no chain-coverage filter at presignature generation time, and no coverage check at signing time before the presignature is consumed.

The project's own design document explicitly identifies this as a production gap:

> "Foreign-tx signing must elect participants that **cover** the requested chain (report ≥ `rpc_quorum(C)` providers for `C`), not merely online ones — a non-covering participant produces no share and can stall the request. **Implementation requirement, not current behavior**: today the signing set is inherited from a presignature, whose participants were chosen for liveness, not chain coverage."
> — `docs/design/calculating-supported-foreign-chains.md` lines 71–76

The analog to M-06 is exact: just as `calcMint` always returned `poolId=0 / idx=0` regardless of which pool was optimal, `make_verify_foreign_tx_leader` always returns the presignature's embedded participant set regardless of which participants actually cover the requested chain.

---

### Impact Explanation

**Medium.** The contract-level availability check (`get_supported_foreign_chains` / `get_available_foreign_chains`) guarantees that ≥ `signing_threshold` active participants cover the requested chain before a request is accepted. However, the signing-set selection ignores this guarantee: the presignature's participants may include nodes that do not cover the chain. Those nodes call `execute_foreign_chain_request`, fail (no configured inspector or RPC quorum not met), and produce no signature share. The signing session cannot reach threshold, the presignature is permanently consumed, and the request silently times out. This breaks the request-lifecycle safety invariant — an accepted, fee-paid request that the network is contractually capable of fulfilling is instead dropped — without any explicit on-chain failure signal to the caller.

---

### Likelihood Explanation

Realistic in any heterogeneous deployment where not every participant covers every whitelisted chain. The design document explicitly anticipates this scenario ("incomplete coverage" is expected to be transient but is not prevented). Because presignatures are generated continuously in the background with liveness-only participant selection, a non-covering participant will routinely appear in presignature sets for chains it does not support. Any unprivileged user submitting a `verify_foreign_transaction` request for such a chain triggers the failure path.

---

### Recommendation

Before consuming a presignature for a `verify_foreign_transaction` request, filter the presignature pool to only those whose embedded participant set is a subset of nodes that cover the requested chain (as reported by `get_available_foreign_chain_by_node` / `register_available_foreign_chain_config`). If no such presignature exists, defer rather than consuming an incompatible one. Alternatively, generate chain-specific presignature pools whose participants are selected with a coverage filter applied at generation time, as described in the design document.

---

### Proof of Concept

**Step 1 — Setup.** Deploy a 4-node MPC cluster. Configure nodes 1–3 to cover Bitcoin; configure node 4 to cover only Ethereum. The signing threshold is 3. Bitcoin is "available" (3 nodes cover it ≥ threshold).

**Step 2 — Presignature generation.** Background presignature generation runs `select_random_active_participants_including_me` with no chain filter. A presignature is generated with participants `{node1, node3, node4}` — a valid liveness-based set, but node 4 does not cover Bitcoin.

**Step 3 — Request submission.** An unprivileged user calls `verify_foreign_transaction` for Bitcoin. The contract accepts it (Bitcoin is available).

**Step 4 — Leader execution.** The leader dequeues the presignature with participants `{node1, node3, node4}` at `sign.rs:63`. It opens a signing channel to all three. Node 4 reaches `execute_foreign_chain_request` for Bitcoin, finds no configured Bitcoin inspector, and returns an error — producing no signature share.

**Step 5 — Outcome.** The signing session cannot reach threshold (only 2 of 3 required shares). The presignature is permanently consumed. The request times out with no on-chain failure signal. The user's deposit is not refunded.
