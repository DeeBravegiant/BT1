### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the responder-supplied `response.payload_hash` is correctly signed, but never checks that `payload_hash` was derived from the transaction ID embedded in the matched `request`. A single Byzantine attested participant (below threshold) can recycle a `(payload_hash, signature)` pair from any previously settled `verify_foreign_transaction` response and inject it as the answer to a completely different pending request, causing the victim caller to receive a cryptographically valid but semantically forged attestation.

### Finding Description

`respond_verify_foreign_tx` performs two independent checks and then resolves the pending yield:

1. **Signature validity** – it verifies `response.signature` over `response.payload_hash` against the root public key.
2. **Request existence** – it looks up `request` in `pending_verify_foreign_tx_requests` and drains the queue.

What it never does is bind `response.payload_hash` to the transaction identifier carried inside `request`. The comment on line 728 even acknowledges the check is only "against the root public key":

```rust
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,   // ← taken verbatim from the response, never compared to request
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

After the signature check passes, the full `response` (including the attacker-chosen `payload_hash`) is serialised and delivered to every waiting yield:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

`resolve_yields_for` only checks that the `request` key exists in the map; it does not inspect the response payload at all. [3](#0-2) 

### Impact Explanation

A victim user calls `verify_foreign_transaction` for transaction T2. The contract stores the request and yields execution. A Byzantine participant calls `respond_verify_foreign_tx` supplying:

- `request` = the victim's pending request (for T2, which exists in the map)
- `response.payload_hash` = H1 (the hash that was signed for a completely different, previously settled transaction T1)
- `response.signature` = S1 (the valid MPC signature over H1, already public on-chain from the earlier settlement)

The contract verifies S1 over H1 — which is valid — and resolves the victim's yield with the forged response. The victim receives a `VerifyForeignTransactionResponse` asserting that their transaction T2 was verified and signed, but the embedded `payload_hash` and signature actually correspond to T1. Any downstream bridge logic that trusts this response will execute against the wrong transaction hash, enabling double-spend or invalid bridge execution.

**Allowed impact matched:** *High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.*

### Likelihood Explanation

- The attacker needs only to be a single attested participant — strictly below the signing threshold.
- The recycled `(payload_hash, signature)` pair is fully public: every `respond_verify_foreign_tx` call is an on-chain transaction whose arguments are visible to all participants and observers.
- No key material, TEE break, or collusion is required. The attacker simply replays data already broadcast by the honest majority.
- The window is open for every `verify_foreign_transaction` request that has not yet been answered by an honest node.

### Recommendation

Before accepting the response, derive the expected payload hash from the transaction identifier in `request` (using the same derivation the MPC nodes perform off-chain) and assert equality:

```rust
let expected_hash = hash_of_tx_id(&request); // same derivation as nodes use
assert_eq!(response.payload_hash.0, expected_hash,
    "payload_hash does not match the transaction in the request");
```

Alternatively, store the expected `payload_hash` alongside the pending request at submission time (inside `verify_foreign_transaction`) and compare it in `respond_verify_foreign_tx`.

### Proof of Concept

1. Honest nodes settle request R1 (transaction T1). The on-chain call to `respond_verify_foreign_tx` is public; extract `payload_hash = H1` and `signature = S1`.
2. User submits `verify_foreign_transaction` for transaction T2. Contract stores request R2 in `pending_verify_foreign_tx_requests`.
3. Byzantine participant calls `respond_verify_foreign_tx(request=R2, response={payload_hash: H1, signature: S1})`.
4. Contract executes line 726–734: verifies S1 over H1 against root key → **valid**.
5. Contract executes lines 749–753: removes R2 from the map and resumes the yield with the forged response.
6. User's callback receives `VerifyForeignTransactionResponse { payload_hash: H1, signature: S1 }` — a cryptographically valid attestation for the wrong transaction. [4](#0-3) [2](#0-1)

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
