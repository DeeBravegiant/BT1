### Title
Pool Operator Can Indefinitely Reset Farmer's Waiting-Room Heightlock via Permissionless `forward_pool_reward` — (File: chia/pools/plotnft_drivers.py)

---

### Summary

In Chia's PlotNFT v2 pool protocol, a pool operator can permanently prevent a farmer from leaving the pool by repeatedly calling `forward_pool_reward` while the farmer is in the waiting-room state. Each call spends the current singleton coin and creates a new one at the current block height, resetting the `ASSERT_HEIGHT_RELATIVE` (heightlock) timer. The spend requires no signature, so the pool can execute this indefinitely while continuing to collect farming rewards.

---

### Finding Description

When a farmer declares intent to leave a pool, the singleton transitions to the waiting-room state (`exiting=True, pool_config != None`). Exiting the waiting room requires the delegated puzzle to include `ASSERT_HEIGHT_RELATIVE(heightlock)`, enforced by the `Heightlock` wrapper in `user_restriction()`:

```python

    return ValidatorStackRestriction(

            ...

        if not self.exiting
        else [Heightlock(self.guaranteed_pool_config.heightlock), SendMessageBanned()]

``` [1](#0-0) 

`ASSERT_HEIGHT_RELATIVE` is measured from the **creation height of the current singleton coin**. Every time the singleton is spent and recreated, the reference height resets.

`forward_pool_reward` is callable in the waiting-room state because its only guard is `if not self.pooling`, and `pooling` is `True` whenever `pool_config is not None` — which remains true in the waiting room:

```python

---

### Impact Explanation

A malicious pool operator can permanently trap a farmer in the waiting room. The pool continues to receive farming rewards (forwarded to `pool_puzzle_hash`) with each absorb, while the farmer's exit countdown never completes. This is a **permanent inability for an honest farmer to execute a valid pool exit action**, matching the High impact category: *"Permanent or long-lived inability for honest… farmers… to process valid… pool actions under normal network assumptions."*

The farmer's XCH farming rewards are continuously diverted to the pool's puzzle hash for the duration of the attack, constituting ongoing unauthorized reward diversion.

---

### Likelihood Explanation

The pool operator has a direct financial incentive: each `forward_pool_reward` call both resets the exit timer and delivers a farming reward to the pool. The attack requires only that the pool farm blocks (their normal operation) and submit permissionless spend bundles. No special access, leaked keys, or cryptographic breaks are needed. Any pool operator who wishes to retain a farmer indefinitely can execute this continuously at negligible cost.

---

### Recommendation

Replace the relative height lock (`ASSERT_HEIGHT_RELATIVE`) with an **absolute height lock** (`ASSERT_HEIGHT_ABSOLUTE`) anchored to the block height at which the farmer first entered the waiting room. This height should be recorded in the singleton state at the moment of the waiting-room transition and used as the immutable reference, so that subsequent absorb/reward-forward spends cannot advance the reference point.

Alternatively, the waiting-room puzzle should be redesigned so that absorb spends do not recreate the singleton coin (e.g., by separating the reward-forwarding path from the singleton lineage), preserving the original creation height for the purpose of the timelock.

---

### Proof of Concept

1. Farmer joins pool with `heightlock = 1440`.
2. Farmer calls `exit_to_waiting_room` at block height H. Singleton coin S₀ is created at height H. Farmer needs block H + 1440 to exit.
3. Pool farms a block at H+1, producing a `PoolReward` coin at the p2_singleton puzzle hash.
4. Pool submits `forward_pool_reward(reward)` with `G2Element()` (no signature). Singleton S₀ is spent; new coin S₁ is created at height H+1. Pool collects the farming reward. Farmer now needs block H+1+1440.
5. Pool repeats step 3–4 every block. Each iteration advances the required exit height by one block.
6. The farmer can never accumulate `heightlock` blocks of "silence" — the exit is permanently deferred.

This is directly confirmed by `test_plotnft_pooling_claim` (parametrized over `"waiting_room"`) and the lifecycle test at line 618–645 of `test_plotnft_wallet.py`, which shows that after a single `forward_pool_reward` while leaving, the farmer must wait an additional `heightlock + 2` blocks: [7](#0-6)
