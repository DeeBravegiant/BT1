### Title
Direct ADA Creation via Unaccounted `directDeposits` in UTxO Conservation Check — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary

The Dijkstra era introduces a new `directDeposits` transaction body field. `applyDirectDeposits` adds these amounts to account balances in the ENTITIES rule. However, neither `dijkstraProducedValue` nor `getConsumedDijkstraValue` includes `directDepositsTxBodyL` in the UTxO conservation check. An unprivileged attacker can include a `directDeposit` of arbitrary amount X to any registered account and have X ADA credited to that account without deducting X from UTxO outputs, creating ADA out of thin air.

### Finding Description

The `validateValueNotConservedUTxO` check enforces:

```
consumed(pp, certState, utxo, txBody) == produced(pp, certState, txBody)
```

For DijkstraEra:
- `consumed = conwayConsumed → getConsumedDijkstraValue → getConsumedMaryValue` (UTxO inputs + withdrawals + cert refunds)
- `produced = getProducedDijkstraValue → dijkstraProducedValue → conwayProducedValue → getProducedMaryValue` (UTxO outputs + fees + cert deposits + treasury donation)

`directDepositsTxBodyL` appears in **neither** side. [4](#0-3) [8](#0-7) 

The ENTITIES rule then unconditionally applies `applyDirectDeposits` after the `directDepositsMissingAccounts` registration check passes: [9](#0-8) 

The unregister/re-register sequence described in the question is one valid path, but the simpler attack requires only a registered account:

**Simpler exploit:**
1. Account `cred` is registered with any balance ≥ 0.
2. Attacker submits a transaction with `directDeposits = {cred → X}` for any X > 0.
3. `directDepositsMissingAccounts` passes (account is registered).
4. `validateValueNotConservedUTxO` passes (X is absent from both sides).
5. `applyDirectDeposits` credits X to the account balance.
6. X ADA is created from nothing.

**The unregister/re-register path** (as described in the question) additionally demonstrates that the `checkStakeKeyHasZeroRewardBalance` guard in `ConwayUnRegCert` is bypassed by first draining the balance via `applyWithdrawals`, then unregistering, re-registering, and finally applying the direct deposit to the freshly registered account: [7](#0-6) 

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

An attacker can mint unbounded ADA into any registered account in a single transaction. The only cost is the transaction fee. This directly violates the total ADA supply invariant (`sumUTxO + sumObligations + sumBalances + fees + treasury + reserves = constant`). Every honest node would accept the transaction and apply the inflated account balance, causing permanent ledger divergence from any node that correctly enforces conservation.

### Likelihood Explanation

Any account holder in the Dijkstra era can exploit this immediately. No privileged access, governance majority, or key compromise is required. The attacker only needs to construct a valid transaction body with a non-empty `directDeposits` field targeting a registered account. The attack is deterministic and locally testable.

### Recommendation

Include `directDeposits` in the `produced` side of the UTxO conservation check. Specifically, `dijkstraProducedValue` (and `dijkstraSubTxProducedValue` for sub-transactions) must add `inject (sum (unDirectDeposits (txBody ^. directDepositsTxBodyL)))` to the produced value, mirroring how withdrawals are included on the consumed side. [4](#0-3)
