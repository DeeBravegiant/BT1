### Title
`verify_foreign_transaction` Gates on Stale All-Participant Intersection Instead of Threshold-Based `available_foreign_chains` Cache — (`File: crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction()` uses `get_supported_foreign_chains()` — the legacy all-participant intersection rule — to decide whether to accept a foreign-chain signing request. The contract already maintains a correctly-updated, threshold-based `available_foreign_chains` cache (via `recompute_available_foreign_chains()`), but the gate function was never switched over. A single participant (strictly below the signing threshold) can exploit this mismatch to permanently block all `verify_foreign_transaction` requests by registering an empty chain-support set, breaking the threshold-safety invariant the protocol is designed to enforce.

---

### Finding Description

`verify_foreign_transaction()` calls `self.get_supported_foreign_chains()` at line 534 to decide whether to accept the request:

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...)
}
```

`get_supported_foreign_chains()` (lines 2176–2218) computes the **strict intersection** of every active participant's registered chains from the legacy `node_foreign_chain_support` storage:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
```

This means **every** active participant must appear in the supporter set for a chain to pass. If even one participant has not registered, or registers an empty set via `register_foreign_chain_support()`, the intersection is empty and every `verify_foreign_transaction` call panics with `ForeignChainNotSupported`.

Meanwhile, the contract already has a correctly-maintained threshold-based cache: `available_foreign_chains` inside `ForeignChainsMetadata`, updated by `recompute_available_foreign_chains()` (lines 1028–1055) on every `register_foreign_chains_config`, `vote_reshared`, and `vote_update_foreign_chain_providers` call. This cache requires only ≥ `reconstruction_threshold` participants to cover a chain. The design document (`docs/design/calculating-supported-foreign-chains.md`, lines 29–34) explicitly states:

> `verify_foreign_transaction(C)` is **rejected unless `C` is available** … The legacy `get_supported_foreign_chains()` (the intersection rule) is **to be deprecated**.

The code was never updated to use `get_available_foreign_chains()` in the gate.

**Attack path:**

1. Attacker controls one participant node (strictly below threshold).
2. Attacker calls `register_foreign_chain_support` with an empty `SupportedForeignChains` set. This is a public participant-authenticated endpoint (line 972) that writes directly to `node_foreign_chain_support.foreign_chain_support_by_node`.
3. `get_supported_foreign_chains()` now returns an empty set (intersection with empty = empty).
4. Every subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported`, regardless of how many other participants correctly registered their chains.
5. The `available_foreign_chains` cache (the correct threshold-based state) is unaffected and still shows chains as available — but it is never consulted.

---

### Impact Explanation

This breaks the production safety invariant that foreign-chain availability is threshold-governed: a single participant (below the signing threshold) can permanently block all `verify_foreign_transaction` requests. The foreign-chain verification feature — which gates bridge inbound flows and cross-chain attestations — becomes completely non-functional until the attacker re-registers. This is a **request-lifecycle and contract execution-flow manipulation that breaks production safety/accounting invariants** (Medium per the allowed scope).

---

### Likelihood Explanation

The `register_foreign_chain_support` endpoint is callable by any active participant with no additional authorization beyond being a voter. A newly-joined participant who has not yet registered also triggers the same condition automatically (their absence from the supporter map causes the intersection to fail). The attack requires no threshold collusion, no key material, and no network-level access — only a single participant-level NEAR transaction.

---

### Recommendation

Replace the `get_supported_foreign_chains()` call in `verify_foreign_transaction()` with `get_available_foreign_chains()`:

```rust
// Before (line 534):
let supported_chains = self.get_supported_foreign_chains();

// After:
let supported_chains = self.get_available_foreign_chains();
```

This aligns the gate with the threshold-based cache that is already correctly maintained and updated on every relevant state change.

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
