Audit Report

## Title
`setPoolAdminFeeDestination` Fails to Flush Accrued Fees Before Overwriting Destination, Diverting Owed Admin Fees to New Recipient - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`setPoolAdminFeeDestination` overwrites `poolAdminFeeDestination[pool]` without first calling `collectFees` on the old destination. Any spread-fee surplus that accumulated under the old destination is permanently redirected to the new destination on the next `collectPoolFees` call. The old destination has no recourse to recover these tokens.

## Finding Description
`setPoolAdminFees` (L408–435) correctly calls `collectFees` with the current `poolAdminFeeDestination[pool]` before updating fee rates, ensuring the old destination receives all accrued fees: [1](#0-0) 

`setPoolAdminFeeDestination` (L438–447) performs no such flush — it directly overwrites the destination: [2](#0-1) 

`collectPoolFees` reads `poolAdminFeeDestination[pool]` at call time, so any surplus accumulated before the destination change is sent to the new destination: [3](#0-2) 

The surplus is computed inside `collectFees` as live pool balance minus bin totals minus notional fees, and is transferred directly to `adminFeeDestination_`: [4](#0-3) [5](#0-4) 

Because `collectPoolFees` is permissionless, any third party can call it immediately after the destination change, guaranteeing the old destination receives nothing.

## Impact Explanation
The old admin fee destination loses all admin-share spread fees that accrued between the last `collectPoolFees` call and the `setPoolAdminFeeDestination` call. These are real ERC-20 token balances held by the pool as surplus. The new destination receives tokens it did not earn. The loss is irreversible once the next collection executes. This constitutes a direct, permanent loss of owed protocol/admin fee assets to a legitimate recipient, meeting the allowed impact gate for medium/high severity.

## Likelihood Explanation
The pool admin is a semi-trusted actor who legitimately calls `setPoolAdminFeeDestination` during treasury rotations. The asymmetry with `setPoolAdminFees` (which enforces collection automatically) means the guard is absent by design omission, not by intent. Any treasury rotation without a prior manual `collectPoolFees` call triggers the loss. Because `collectPoolFees` is permissionless, a third party (keeper, bot, or adversary) can front-run or immediately follow the destination change to sweep accumulated surplus to the new destination, making the race window reliably exploitable without any special privilege.

## Recommendation
Mirror the pattern in `setPoolAdminFees` by flushing accrued fees to the old destination before overwriting it:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to old destination before switching
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

## Proof of Concept
1. Pool is created with `adminFeeDestination = oldTreasury`.
2. Swaps occur; spread-fee surplus accumulates (`balance > binTotals + notionalFees`).
3. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)` without a prior `collectPoolFees`.
4. `poolAdminFeeDestination[pool]` is now `newTreasury`.
5. Anyone calls `collectPoolFees(pool)`.
6. `collectFees` computes the full accumulated surplus and sends the admin share to `newTreasury`.
7. `oldTreasury` receives nothing despite having earned the fees during steps 1–3.

A Foundry test can confirm this by: deploying a pool, executing swaps to build surplus, calling `setPoolAdminFeeDestination` to a new address, then calling `collectPoolFees` and asserting `oldTreasury.balance == 0` while `newTreasury.balance > 0`.
