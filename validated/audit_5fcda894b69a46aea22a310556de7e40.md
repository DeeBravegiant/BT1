### Title
Deposits from Cleared Update Proposals Are Permanently Lost When `do_update` Executes - (`crates/contract/src/update.rs`)

---

### Summary

When `vote_update` reaches the signing threshold and `do_update` is called, it clears **all** pending update proposals from contract storage. However, the NEAR deposits paid by the proposers of those cleared proposals are never refunded. The contract permanently retains these funds, causing direct, unrecoverable loss for every participant whose proposal was swept away by a competing update execution.

---

### Finding Description

**Deposit collection in `propose_update`:**

In `crates/contract/src/lib.rs`, `propose_update` requires each proposer to attach a deposit proportional to the size of the update payload:

```rust
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
if attached < required {
    return Err(InvalidParameters::InsufficientDeposit { ... }.into());
}
let id = self.proposed_updates.propose(update);
// Only the *excess* above `required` is refunded; the required amount is kept.
if let Some(diff) = attached.checked_sub(required) && diff > NearToken::from_yoctonear(0) {
    Promise::new(proposer).transfer(diff).detach();
}
``` [1](#0-0) 

The `required` deposit is computed as `storage_byte_cost × bytes_used`, where `bytes_used` includes the full serialized contract binary or config blob plus a fixed overhead for 128 participant-vote slots:

```rust
fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;
    match update {
        Update::Contract(code) => { bytes_used += code.len() as u128; }
        Update::Config(config) => { bytes_used += serde_json::to_vec(&config).unwrap().len() as u128; }
    }
    bytes_used
}
fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
``` [2](#0-1) 

For a 1 MB contract binary this deposit is approximately 10 NEAR (NEAR charges ~1 NEAR per 100 KB of storage).

**Silent clearing of all proposals in `do_update`:**

When the threshold is reached, `do_update` removes the winning entry and then unconditionally clears every remaining entry and every vote:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;

    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
``` [3](#0-2) 

No refund is issued to the proposers of the cleared entries. Furthermore, `UpdateEntry` does not store the proposer's `AccountId`, so the contract has no record of whom to refund even if it tried:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    // ← no proposer field
}
``` [4](#0-3) 

The test `test_proposed_updates_do_update_clears_all_state` explicitly confirms that all entries are wiped, with no refund logic present: [5](#0-4) 

---

### Impact Explanation

Every participant who proposed a competing update loses their full required deposit — permanently and irrecoverably — the moment any other update reaches the signing threshold. For a contract-code proposal carrying a 1 MB binary, this is ~10 NEAR per proposer. With N competing proposals, N−1 proposers each lose their full deposit. The contract balance grows by the sum of those deposits with no mechanism to recover them. This is a direct, on-chain loss of funds controlled by the MPC contract, matching the **Medium** allowed impact: *"Balance … manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The governance flow explicitly supports multiple simultaneous proposals (the sandbox test `test_propose_update_contract_many` proposes two updates and votes for the last one, confirming all are cleared). In any real upgrade cycle where participants disagree on which binary or config to adopt, multiple proposals will coexist. The loss is triggered by the normal, intended execution of `vote_update` reaching threshold — no adversarial action is required beyond a participant proposing a competing update before the threshold is reached.

---

### Recommendation

1. Add a `proposer: AccountId` field to `UpdateEntry` so the contract knows whom to refund.
2. In `do_update`, before calling `self.entries.clear()`, iterate over the remaining entries and issue a `Promise::new(entry.proposer).transfer(required_deposit(entry.bytes_used))` for each one.
3. Alternatively, track a `deposits: LookupMap<UpdateId, (AccountId, NearToken)>` alongside `entries` and drain it on clear.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR. `UpdateEntry { update: Contract([...]), bytes_used: ~1_000_000 }` is stored; 10 NEAR is retained by the contract.
2. Participant B calls `propose_update` with a different 1 MB binary, attaching ~10 NEAR. A second entry is stored; another 10 NEAR is retained.
3. Threshold participants vote for proposal A via `vote_update`. `do_update` is called: proposal A is executed, proposal B's entry is removed via `self.entries.clear()`, and `self.vote_by_participant.clear()` wipes all votes.
4. Participant B's 10 NEAR deposit is now held by the contract with no refund path. The contract has no record of B's `AccountId` in the cleared `UpdateEntry`, so the funds are permanently stranded. [6](#0-5) [7](#0-6)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
