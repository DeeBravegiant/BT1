Audit Report

## Title
Stale Cross-Chain Rate Consumed Without Freshness Check Allows Excess rsETH Minting — (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary

`CrossChainRateReceiver` stores the rsETH/ETH exchange rate updated asynchronously via LayerZero and exposes a `lastUpdated` timestamp, but neither `getRate()` nor any caller ever validates that timestamp against a maximum staleness bound. Both `RSETHPoolV3` and `RSETHPoolNoWrapper` divide deposited ETH by this potentially stale rate to compute rsETH output, allowing any depositor to exploit the window between an L1 price increase and the corresponding L2 rate update to receive more rsETH than the current rate entitles them to, extracting yield from existing rsETH holders.

## Finding Description

`CrossChainRateReceiver.lzReceive()` stores the incoming rate and records `lastUpdated`:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L95-97
rate = _rate;
lastUpdated = block.timestamp;
```

`getRate()` returns the stored value unconditionally with no freshness check:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

`RSETHPoolV3.getRate()` and `RSETHPoolNoWrapper.getRate()` both delegate directly to this oracle:

```solidity
// contracts/pools/RSETHPoolV3.sol L235-237
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

Both pools' `viewSwapRsETHAmountAndFee()` divide by this rate to compute rsETH output:

```solidity
// contracts/pools/RSETHPoolV3.sol L304-307
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

```solidity
// contracts/pools/RSETHPoolNoWrapper.sol L282-285
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

`deposit()` in both pools calls `viewSwapRsETHAmountAndFee()` and immediately mints/transfers rsETH to the caller with no staleness guard:

```solidity
// contracts/pools/RSETHPoolV3.sol L258-262
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);
```

```solidity
// contracts/pools/RSETHPoolNoWrapper.sol L237-241
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
rsETH.safeTransfer(msg.sender, rsETHAmount);
```

The `dailyMintLimit` in `RSETHPoolV3` caps total daily issuance but does not prevent the stale-rate exploit within that cap — it only bounds the magnitude of loss per day, not the existence of the vulnerability. `RSETHPoolNoWrapper` has no such cap at all. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

## Impact Explanation

When the L1 rsETH price rises (reflecting accrued restaking rewards) but the L2 rate has not yet been updated, a depositor receives `amountAfterFee * 1e18 / staleRate` rsETH instead of the correct `amountAfterFee * 1e18 / currentRate`. Because `staleRate < currentRate`, the depositor receives excess rsETH. This excess represents a claim on more ETH than was deposited. The attacker bridges the excess rsETH to L1 (via the wrsETH wrapper in `RSETHPoolV3`, or directly as an OFT in `RSETHPoolNoWrapper`) and redeems at the current L1 rate, extracting ETH from the protocol's TVL. The extracted value is yield that belongs to existing rsETH holders, who are diluted. This matches **High — Theft of unclaimed yield**. [5](#0-4) [6](#0-5) 

## Likelihood Explanation

The rate update is entirely off-chain: a keeper or bot must send a LayerZero message from L1 to L2. Any delay — network congestion, keeper downtime, or deliberate inaction — creates an exploitable window. Because rsETH accrues restaking rewards continuously, the L1 price rises monotonically between updates, making the L2 rate perpetually slightly stale. No special permissions are required; `deposit()` is callable by any address. An attacker monitoring L1 oracle updates and L2 `lastUpdated` events can reliably identify and exploit these windows repeatedly. [7](#0-6) 

## Recommendation

Expose `lastUpdated` through the oracle interface and enforce a maximum staleness threshold in both pool contracts before accepting the rate. For example, add to `RSETHPoolV3.viewSwapRsETHAmountAndFee()` and `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee()`:

```solidity
interface IOracle {
    function getRate() external view returns (uint256);
    function lastUpdated() external view returns (uint256);
}

uint256 constant MAX_RATE_AGE = 1 hours; // or protocol-defined value

uint256 rsETHToETHrate = getRate();
require(
    block.timestamp - IOracle(rsETHOracle).lastUpdated() <= MAX_RATE_AGE,
    "Stale rate"
);
```

`lastUpdated` is already a public state variable in `CrossChainRateReceiver` and requires no new storage; it only needs to be consumed by callers. [8](#0-7) 

## Proof of Concept

**Setup (fork test against deployed L2):**

1. Deploy or fork a chain with `CrossChainRateReceiver` at a known address, `RSETHPoolV3` (or `RSETHPoolNoWrapper`) pointing to it as `rsETHOracle`.
2. Confirm `rate = 1.05e18`, `lastUpdated = T`.
3. Simulate L1 price rising to `1.10e18` by **not** sending a new LayerZero update (rate remains `1.05e18`).

**Attack sequence:**

```solidity
// Attacker deposits 100 ETH at stale rate
pool.deposit{value: 100 ether}("ref");
// rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.238e18 rsETH

// Correct amount at current rate:
// 100e18 * 1e18 / 1.10e18 ≈ 90.909e18 rsETH

// Excess: ≈ 4.329 rsETH ≈ 4.762 ETH at current rate

// Attacker bridges excess rsETH to L1 and redeems
```

**Foundry invariant test plan:**

```solidity
function invariant_noExcessRsETHMinted() public {
    // Assert: for any deposit of `amount` ETH,
    // rsETHMinted <= amount * 1e18 / currentL1Rate
    // where currentL1Rate is fetched from the L1 oracle directly
    // Violation: rsETHMinted > amount * 1e18 / currentL1Rate
    //            when L2 rate is stale (lastUpdated + MAX_AGE < block.timestamp)
}
```

A fork test calling `deposit()` after warping `block.timestamp` past the last `lastUpdated` by more than the expected update interval will demonstrate the excess minting concretely. [3](#0-2) [4](#0-3)
