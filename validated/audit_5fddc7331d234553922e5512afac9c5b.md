### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Allows Byzantine Coordinator to Deliver Forged Foreign-Chain Verification Response - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never validates that `response.payload_hash` is consistent with the `request` (i.e., that it equals `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))`). A single Byzantine attested MPC coordinator node can reuse a valid signature obtained from one foreign-chain verification request to resolve a completely different pending request with a mismatched `payload_hash`, causing the contract to deliver a forged verification response to the caller.

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following checks:

1. Caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` using the root public key for the domain.
3. `request` matches a pending entry in `pending_verify_foreign_tx_requests`.

What is **not** checked: that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload{request, extracted_values}` for any valid set of extracted values consistent with `request`. [1](#0-0) 

The `payload_hash` is taken directly from the caller-supplied `response` and used as the message to verify the signature against. The contract has no way to recompute the expected hash because the extracted values are not stored on-chain, but it also makes no attempt to bind the hash to the `request` in any way.

Compare this to the regular `respond` function, where the payload hash is taken from the stored `request` (not from the response), making it impossible for the responder to substitute a different hash: [2](#0-1) 

The `VerifyForeignTransactionResponse` struct carries both `payload_hash` and `signature` as independent, caller-supplied fields: [3](#0-2) 

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does perform the consistency check — it recomputes the expected hash and compares it to `response.payload_hash` — but this check lives entirely off-chain in the caller's code, not in the contract: [4](#0-3) 

The contract itself resolves the pending yield and returns the response to the original caller without any such check: [5](#0-4) 

### Impact Explanation

A Byzantine MPC coordinator node (a single attested participant, strictly below the signing threshold) can:

1. Act as coordinator for a legitimate `verify_foreign_transaction(request_A)` and obtain the final threshold signature `sig_A` over `payload_hash_A = SHA-256(borsh(ForeignTxSignPayload{request_A, actual_values_A}))`.
2. Observe a different pending request `request_B` (e.g., a bridge service waiting for verification of a high-value transaction).
3. Call `respond_verify_foreign_tx(request_B, {payload_hash: payload_hash_A, signature: sig_A})`.
4. The contract accepts: `sig_A` is a valid signature over `payload_hash_A` under the root public key. The `request_B` key exists in `pending_verify_foreign_tx_requests`. Both checks pass.
5. The contract resolves `request_B` and delivers `{payload_hash_A, sig_A}` to the caller of `verify_foreign_transaction(request_B)`.

The caller receives a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to their request. Any downstream bridge contract or application that does not independently recompute and verify the `payload_hash` will accept this as a valid verification of transaction B, when in fact the MPC network only verified transaction A. This enables forged foreign-chain verification and potential invalid bridge execution.

**Impact class:** High — forged foreign-chain verification causing invalid bridge execution.

### Likelihood Explanation

The attack requires exactly one Byzantine attested MPC participant who is selected as coordinator for at least one legitimate signing session (to obtain a reusable signature). Coordinator selection is deterministic and rotates across participants, so any single Byzantine participant will eventually be coordinator. No threshold collusion, no key leakage, and no network-level attack is required. The attacker-controlled entry point is the public `respond_verify_foreign_tx` contract method, callable by any attested participant.

### Recommendation

The contract must bind `response.payload_hash` to `request` before accepting the response. The most direct fix is to include the `extracted_values` in the response so the contract can recompute and verify the hash:

```rust
// In respond_verify_foreign_tx, after verifying the signature:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // add to response DTO
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

If including extracted values in the response is undesirable (e.g., size constraints), an alternative is to store a commitment to the expected `payload_hash` in the pending-request map at request time — but this is not possible since the extracted values are unknown at request submission time. The cleanest solution is to include extracted values in the response and perform the recomputation on-chain, mirroring the approach used by `ForeignChainSignatureVerifier::verify_signature` in the SDK.

### Proof of Concept

```rust
// Setup: Byzantine coordinator obtains sig_A from a legitimate signing of request_A
let request_A = VerifyForeignTransactionRequest { /* tx_id_A, ... */ };
let payload_hash_A = /* SHA-256(borsh(ForeignTxSignPayload{request_A, actual_values_A})) */;
let sig_A = /* valid MPC threshold signature over payload_hash_A */;

// Victim: a pending request for a different transaction
let request_B = VerifyForeignTransactionRequest { /* tx_id_B, ... */ };
// request_B is in pending_verify_foreign_tx_requests

// Attack: Byzantine coordinator responds to request_B with sig_A / payload_hash_A
contract.respond_verify_foreign_tx(
    request_B,
    VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,  // wrong hash for request_B
        signature: sig_A,              // valid signature, but over payload_hash_A
    },
);
// Contract accepts: sig_A is valid over payload_hash_A under root public key ✓
// Contract accepts: request_B is in pending map ✓
// Contract resolves request_B with {payload_hash_A, sig_A} — forged response delivered
```

The contract's verification at lines 726–734 passes because it only checks `verify_ecdsa_signature(sig_A, payload_hash_A, root_pk)`, which is true. It never checks `payload_hash_A == SHA-256(borsh(ForeignTxSignPayload{request_B, ...}))`. [6](#0-5)
