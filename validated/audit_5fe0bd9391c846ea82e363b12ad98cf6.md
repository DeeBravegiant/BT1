Audit Report

## Title
Block Stuffing Allows Attacker to Monopolize Daily Mint Limit, Temporarily Freezing Deposits for All Other Users - (File: contracts/pools/RSETHPoolV2ExternalBridge.sol)

## Summary
The permissionless `deposit` function in `RSETHPoolV2ExternalBridge` enforces a single shared `dailyMintLimit` with no per-depositor cap. An attacker can use block stuffing on a low-cost L2 to exclude competing depositors, then land a single deposit that exhausts the entire remaining daily limit, preventing all other users from depositing for up to 24 hours while the attacker receives the full daily rsETH allocation.

## Finding Description
The `deposit` function carries no role check and is callable by any address:

```solidity
// L289
function deposit(string memory referralId) external payable nonReentrant whenNotPaused limitDailyMint(msg.value) {
```

The `limitDailyMint` modifier maintains a single shared counter `dailyMintAmount` against `dailyMintLimit`. Once the limit is reached, every subsequent call reverts:

```solidity
// L119-125
if (dailyMintAmount + rsETHAmount > dailyMintLimit) {
    revert DailyMintLimitExceeded();
}
dailyMintAmount += rsETHAmount;
```

There is no per-address tracking, no maximum deposit per user per day, and no anti-monopolization guard anywhere in the modifier or the `deposit` function body.

`remainingDailyMintLimit()` is a public view function that exposes the exact remaining capacity:

```solidity
// L380-385
function remainingDailyMintLimit() external view returns (uint256) {
    uint256 effectiveDailyMintAmount = (getCurrentDay() > lastMintDay) ? 0 : dailyMintAmount;
    return dailyMintLimit - effectiveDailyMintAmount;
}
```

**Attack path:**
1. Attacker calls `remainingDailyMintLimit()` at the start of a new day to observe the full available capacity.
2. Attacker broadcasts a batch of high-gas dummy transactions with a gas price sufficient to fill the block gas limit for several consecutive L2 blocks, excluding competing depositors.
3. Attacker's own `deposit` call, sized to consume the full remaining daily limit, is included in the next available slot.
4. `dailyMintAmount` reaches `dailyMintLimit`; all subsequent `deposit` calls revert with `DailyMintLimitExceeded` until the next day.

The existing `whenNotPaused` and `nonReentrant` guards are irrelevant to this attack. The `WHITELISTED_USER_ROLE` exists in the contract but is not applied to `deposit`.

## Impact Explanation
All legitimate depositors are unable to call `deposit` for the remainder of the current 24-hour window. The attacker receives the full daily rsETH allocation. This constitutes a **temporary freezing of deposits** for all other users, matching the allowed impact: **Low. Block stuffing**.

## Likelihood Explanation
The attack is feasible on any L2 chain (Arbitrum, Optimism, Base, etc.) where this contract is deployed. Gas costs on L2s are orders of magnitude lower than mainnet, making block stuffing economically viable. The attacker also receives the full daily rsETH allocation as compensation, making the attack economically self-funding. The `deposit` function is permissionless, the daily limit is a shared resource with no per-user cap, and `remainingDailyMintLimit()` gives the attacker precise targeting information. The attack is repeatable every day.

## Recommendation
1. **Per-user deposit cap**: Introduce a `mapping(address => uint256) public dailyMintAmountPerUser` and enforce a maximum deposit amount per address per day to prevent any single depositor from monopolizing the daily limit.
2. **Commit-reveal or time-delay**: Require a two-step deposit (commit then reveal after N blocks) to make block stuffing ineffective.
3. **Access control on `deposit`**: Restrict `deposit` to whitelisted addresses using the existing `WHITELISTED_USER_ROLE` to raise the bar for attackers.

## Proof of Concept
```solidity
function testBlockStuffingMonopolizesDailyLimit() public {
    uint256 remaining = pool.remainingDailyMintLimit();
    assertGt(remaining, 0, "limit already exhausted");

    // Attacker stuffs blocks (simulated) then deposits the full remaining limit
    uint256 ethNeeded = remaining * pool.getRate() / 1e18;
    vm.deal(attacker, ethNeeded + 1 ether);
    vm.prank(attacker);
    pool.deposit{value: ethNeeded}("ref");

    // Victim's deposit now reverts
    vm.deal(victim, 1 ether);
    vm.prank(victim);
    vm.expectRevert(RSETHPoolV2ExternalBridge.DailyMintLimitExceeded.selector);
    pool.deposit{value: 1 ether}("ref");

    // Limit resets next day
    assertEq(pool.remainingDailyMintLimit(), 0);
    vm.warp(block.timestamp + 1 days);
    assertGt(pool.remainingDailyMintLimit(), 0, "limit resets next day");
}
```

The Foundry fork test above (run against an L2 fork) demonstrates that a single depositor can exhaust the daily limit and cause all subsequent `deposit` calls to revert with `DailyMintLimitExceeded` for the remainder of the 24-hour window. The block stuffing component (broadcasting high-gas dummy transactions) is the off-chain mechanism that ensures the attacker's transaction is included before victims' transactions on the L2.
