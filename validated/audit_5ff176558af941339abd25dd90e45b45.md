### Title
Unverified `respond_ckd` Response for `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Confidential Key Derivation Output — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC contract performs a cryptographic output check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`. For the `AppPublicKey` (legacy) variant, **no verification of the response is performed**. Any single attested MPC participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary `CKDResponse`, and the contract will accept it and deliver the forged key material to the requesting user.

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically proves the response encodes the correct MPC-derived secret. [2](#0-1) 

For `AppPublicKey`, the arm is a no-op (`{}`). The contract then unconditionally calls `resolve_yields_for`, which resumes all queued yield promises with the unverified `response` bytes, delivering them to the original caller. [3](#0-2) 

The only gate on `respond_ckd` is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a single attested participant — not a threshold of them. [4](#0-3) 

---

### Impact Explanation

The CKD protocol is designed so that the user provides an ephemeral public key `pk1 = a·G1`, the MPC network computes `big_s = H(app_id)·msk` (the actual secret), picks random `y`, and returns `(big_y = y·G1, big_c = big_s + y·pk1)`. The user recovers `big_s = big_c − a·big_y` using their private key `a`.

If a Byzantine participant submits arbitrary `(big_y, big_c)`, the user computes a wrong `big_s` that has no relation to the actual MPC master secret. The attacker can choose values such that the "secret" the user derives is known to the attacker (e.g., set `big_y = G1·y_fake` and `big_c = G1·s_fake + pk1·y_fake` for attacker-chosen `s_fake, y_fake`). The user then derives `big_s = G1·s_fake`, a value the attacker knows.

This constitutes **unauthorized confidential key derivation output without the required threshold participant authorization**: a single Byzantine node below the signing threshold forges the CKD response, bypassing the threshold requirement entirely. The user's confidential key is permanently corrupted or disclosed to the attacker.

**Allowed impact matched:** *Critical — Unauthorized confidential key derivation output without the required participant authorization.*

---

### Likelihood Explanation

- The attacker must be an attested MPC participant (has a valid TEE attestation and is in the active participant set). This is a Byzantine participant strictly below the signing threshold — exactly the attacker profile the system is designed to tolerate.
- The attack requires only one on-chain transaction (`respond_ckd`) after observing a pending `AppPublicKey` CKD request.
- `AppPublicKey` is the default/legacy variant used by all existing integrations (e.g., `CKDRequestTest::new` in tests uses `AppPublicKey`).
- No special timing, mempool access, or off-chain capability is needed beyond being an attested participant. [5](#0-4) 

---

### Recommendation

Add a cryptographic output check for the `AppPublicKey` variant in `respond_ckd`. Since `AppPublicKey` provides only `pk1` (a G1 point) without a G2 counterpart, the pairing-based `ckd_output_check` cannot be applied directly. Options:

1. **Require `AppPublicKeyPV` for all new requests** and deprecate `AppPublicKey` at the contract level, rejecting new `AppPublicKey` submissions.
2. **Add a weaker consistency check** for `AppPublicKey`: verify that `big_y` and `big_c` are valid, non-identity G1 points (preventing trivially degenerate forgeries), and document that full verifiability requires `AppPublicKeyPV`.
3. **Require threshold-signed responses**: introduce a mechanism where `respond_ckd` only accepts a response that carries threshold-many signatures from participants, analogous to how the off-chain MPC protocol already requires threshold participation.

The `AppPublicKeyPV` path is the correct model — the `AppPublicKey` path should either be removed or receive an equivalent security guarantee.

---

### Proof of Concept

1. Alice calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)` where `pk1 = a·G1` for her private scalar `a`.
2. The contract stores the pending request in `pending_ckd_requests`.
3. Mallory, a single attested MPC participant, observes the pending request on-chain.
4. Mallory picks arbitrary scalars `y_fake, s_fake` and calls:
   ```
   respond_ckd(
     request = <Alice's CKDRequest>,
     response = CKDResponse {
       big_y: G1 * y_fake,
       big_c: G1 * s_fake + pk1 * y_fake,
     }
   )
   ```
5. The contract executes lines 675–688: the `AppPublicKey` arm is a no-op, `resolve_yields_for` resumes Alice's yield with the forged response.
6. Alice receives `(big_y, big_c)` and computes `big_s = big_c − a·big_y = G1·s_fake`. Mallory knows `s_fake`, so Mallory knows Alice's derived secret.
7. The actual MPC-derived secret `H(app_id)·msk` is never delivered to Alice. [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
