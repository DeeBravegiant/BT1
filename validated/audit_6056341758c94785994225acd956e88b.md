Audit Report

## Title
Stale rsETH Exchange Rate in `CrossChainRateReceiver.getRate()` Enables Over-Minting of wrsETH After L2 Sequencer Downtime - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally with no check against `lastUpdated`. Because the rate is only refreshed when a LayerZero message arrives from L1, any L2 sequencer downtime causes the stored rate to become arbitrarily stale. When the sequencer recovers, all L2 pool contracts immediately resume using this stale (lower) rate to price deposits, allowing any depositor to receive more rsETH/wrsETH than the current fair value, diluting existing holders' accrued yield.

## Finding Description
`CrossChainRateReceiver` stores the rsETH/ETH exchange rate and a `lastUpdated` timestamp, but `getRate()` ignores `lastUpdated` entirely:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L102-105
function getRate() external view returns (uint256) {
    return rate;  // No staleness check
}
``` [1](#0-0) 

The rate is only updated inside `lzReceive()`, which requires a live L2 sequencer to relay LayerZero messages: [2](#0-1) 

Both pool variants consume this rate directly. `RSETHPoolV3.viewSwapRsETHAmountAndFee()`: [3](#0-2) 

`RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee()`: [4](#0-3) 

Both `deposit()` functions are public and call these view functions with no additional rate-freshness guard. Since rsETH accrues staking rewards monotonically, a stale rate is always lower than the true current rate. The formula `rsETHAmount = amountAfterFee * 1e18 / staleRate` produces more rsETH than `amountAfterFee * 1e18 / trueRate` whenever `staleRate < trueRate`.

The existing `paused` / `whenNotPaused` modifier in `RSETHPoolV3` and `PausableUpgradeable` in `RSETHPoolNoWrapper` are not automatically triggered by sequencer downtime; they require manual admin action. The `dailyMintLimit` in `RSETHPoolV3` caps total daily minting but does not prevent individual over-minting at a stale rate within that cap. [5](#0-4) 

## Impact Explanation
rsETH's ETH price increases monotonically as staking rewards accrue (~4% APY). During sequencer downtime, the stored rate falls behind the true rate. Upon recovery, every depositor who calls `deposit()` before a fresh LayerZero message arrives receives excess rsETH/wrsETH proportional to the accrued-but-undelivered rate delta. This excess is extracted from the pool of accrued yield belonging to existing wrsETH holders, constituting **theft of unclaimed yield** — a High-severity impact within the allowed scope.

## Likelihood Explanation
All target L2 chains (Scroll, Optimism, Base, Linea, Arbitrum) use centralized sequencers with documented historical downtime. No special permissions are required: any address can call `deposit()` immediately after sequencer recovery. The attacker only needs to monitor sequencer status and submit a deposit before the next LayerZero rate update arrives. The attack is repeatable for the entire window between sequencer recovery and the next successful `lzReceive()` call.

## Recommendation
1. **Add a staleness check in `CrossChainRateReceiver.getRate()`**:
   ```solidity
   uint256 public constant MAX_RATE_STALENESS = 24 hours;

   function getRate() external view returns (uint256) {
       require(block.timestamp - lastUpdated <= MAX_RATE_STALENESS, "Rate is stale");
       return rate;
   }
   ```
2. **Add a Chainlink L2 Sequencer Uptime Feed check and a time-based heartbeat staleness bound in `ChainlinkOracleForRSETHPoolCollateral.getRate()`** per [Chainlink's official L2 guidance](https://docs.chain.link/data-feeds/l2-sequencer-feeds), including a grace period (e.g., 1 hour) after sequencer recovery before accepting prices. [6](#0-5) 

## Proof of Concept
1. L1 pushes `rate = 1.05e18` to `CrossChainRateReceiver` via LayerZero; `lastUpdated = T`.
2. L2 sequencer goes down for 6 hours. rsETH accrues rewards; true rate rises to `1.06e18`. No `lzReceive()` can execute.
3. Sequencer recovers. `CrossChainRateReceiver.rate` is still `1.05e18`.
4. Attacker calls `RSETHPoolV3.deposit{value: 100 ether}("")` (or `RSETHPoolNoWrapper.deposit`).
5. Pool computes: `rsETHAmount = 100e18 * 1e18 / 1.05e18 ≈ 95.238 wrsETH`.
6. Fair amount at true rate: `100e18 * 1e18 / 1.06e18 ≈ 94.340 wrsETH`.
7. Attacker receives **~0.898 excess wrsETH** per 100 ETH, extracted from existing holders' accrued yield.
8. Attack repeats until the next `lzReceive()` corrects the rate.

**Foundry fork test plan**: Fork the target L2 at a block after a known sequencer outage; warp `block.timestamp` forward by the outage duration without calling `lzReceive()`; call `deposit()` and assert `rsETHAmount > expectedFairAmount` using the true post-outage rate fetched from L1.
