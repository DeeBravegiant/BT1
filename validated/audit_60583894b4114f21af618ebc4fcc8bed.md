### Title
Propose-Update Storage Deposit Permanently Locked in Contract with No Refund Mechanism — (File: `crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

### Summary

The `propose_update` function requires participants to attach a NEAR deposit to cover storage costs for the proposed update payload. Neither the execution path (`do_update`) nor any cleanup path refunds this deposit to the original proposer. The `UpdateEntry` struct does not even record the proposer's identity, making a future refund structurally impossible without a state migration. NEAR tokens are permanently locked in the contract's balance with no recovery mechanism.

### Finding Description

When a participant calls `propose_update`, the contract computes a required deposit: [1](#0-0) 

```rust
pub fn required_deposit(update: &Update) -> NearToken {
    required_deposit(bytes_used(update))
}
```

`bytes_used` includes the full WASM binary size plus a fixed overhead for 128 assumed vote entries: [2](#0-1) 

For a 1 MB WASM binary this is approximately 10+ NEAR at current storage byte costs. The exact required amount is kept by the contract; only the excess is refunded: [3](#0-2) 

When the update reaches threshold and `do_update` is called, **all entries are cleared without refunding any deposit**: [4](#0-3) 

The `UpdateEntry` struct stores only the update payload and `bytes_used`; the proposer's `AccountId` is never recorded: [5](#0-4) 

This means even if a refund were desired, the contract has no record of who to refund. The same applies when entries are cleared after resharing via `remove_non_participant_votes`: [6](#0-5) 

No `cancel_update`, `withdraw_update_deposit`, or equivalent rescue function exists anywhere in the contract. The contract has no general balance-withdrawal mechanism.

### Impact Explanation

Every `propose_update` call permanently locks NEAR tokens in the contract's balance. For a typical contract upgrade (1 MB WASM), this is ~10 NEAR per proposal. Over the lifetime of the MPC network — which undergoes regular upgrades — multiple proposals accumulate, each locking funds with no recovery path. The contract's balance grows monotonically from these deposits, but neither the proposer nor any governance action can retrieve them. This breaks the production accounting invariant that storage deposits are refunded when the storage they cover is freed (the pattern used correctly in `sign`, `request_app_private_key`, and `submit_participant_info`).

**Impact: Medium** — Balance accounting invariant broken; NEAR permanently frozen in the chain-signature contract.

### Likelihood Explanation

Contract upgrades are a routine, recurring operation for the MPC network. Every upgrade cycle involves at least one `propose_update` call. The deposit loss is deterministic and cumulative — it occurs on every proposal regardless of whether the update is executed or abandoned. No special conditions or adversarial behavior are required; normal operational use triggers the issue.

### Recommendation

Store the proposer's `AccountId` and the deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before clearing entries, iterate over all entries and schedule `Promise::new(entry.proposer).transfer(entry.deposit)` for each one. This mirrors the refund pattern already used in `propose_update` for excess deposits and in `submit_participant_info` for storage deposits.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB WASM binary, attaching ~10 NEAR (the required deposit).
2. The contract stores the entry and keeps the full deposit; only excess is refunded.
3. Scenario A — update executed: threshold votes are cast, `vote_update` calls `do_update`, which clears all entries. The ~10 NEAR remains in the contract's balance; Participant A receives nothing.
4. Scenario B — update abandoned: participants disagree; after resharing, `remove_non_participant_update_votes` removes votes but leaves entries. The ~10 NEAR remains locked indefinitely.
5. In both scenarios, there is no contract method that Participant A (or anyone else) can call to recover the deposit. [7](#0-6) [4](#0-3)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

    /// Propose update to either code or config, but not both of them at the same time.
    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }
