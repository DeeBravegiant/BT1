### Title
Hardcoded `boundX = 3` in `taylorExpCmp` Produces Incorrect Slot-Leader Decisions When Active Slot Coefficient Exceeds ~0.95 - (File: libs/cardano-protocol-tpraos/src/Cardano/Protocol/TPraos/BHeader.hs)

---

### Summary

The Praos slot-leader eligibility check in `checkLeaderNatValue` calls `taylorExpCmp 3 recip_q x`, where the first argument (`3`) is the Lagrange error bound `M` for the Taylor series remainder. This bound is only valid when `|ln(1-f)| ≤ 3`, i.e., when the active slot coefficient `f ≤ ~0.95`. The protocol allows `f` up to `1` (exclusive), and the code does **not** clamp or validate `f` against this mathematical precondition before using the hardcoded bound. When `f` is set above ~0.95, the error bound is too small, causing `taylorExpCmp` to return premature `ABOVE`/`BELOW` decisions based on an incorrect remainder estimate, which makes the leader check return wrong results — either accepting blocks from ineligible producers or rejecting blocks from eligible ones.

---

### Finding Description

The slot-leader check computes whether a block producer is eligible by verifying:

```
1/q < exp(-σ · ln(1-f))
```

This is implemented via `taylorExpCmp`:

```haskell
-- libs/cardano-protocol-tpraos/src/Cardano/Protocol/TPraos/BHeader.hs, line 388
case taylorExpCmp 3 recip_q x of
  ABOVE _ _ -> False
  BELOW _ _ -> True
  MaxReached _ -> False
```

The first argument `3` is the `boundX` parameter — the Lagrange error bound `M` used to estimate the Taylor remainder:

```haskell
-- libs/non-integral/src/Cardano/Ledger/NonIntegral.hs, lines 196-209
taylorExpCmp :: RealFrac a => a -> a -> a -> CompareResult a
taylorExpCmp boundX cmp x = go 1000 0 x 1 1
  where
    go maxN n err acc divisor
      ...
      where
        errorTerm = abs (err' * boundX)
```

The design document explicitly states:

> "a good integral bound for the error estimation is the maximal value of |ln(1-f)| ≤ M for f ∈ [0, 0.95], that is M = 3. **For larger values of f, a different value of M would need to be chosen.**"

The `activeSlotLog` function computes `ln(1-f)` and stores it as a `FixedPoint`:

```haskell
-- libs/cardano-ledger-core/src/Cardano/Ledger/BaseTypes.hs, lines 741-755
mkActiveSlotCoeff v =
  ActiveSlotCoeff
    { unActiveSlotVal = v
    , unActiveSlotLog =
        if v == maxBound
          then 0
          else floor (fpPrecision * ln' ((1 :: FixedPoint) - fromRational (unboundRational v)))
    }
```

When `f > 0.95`, `|ln(1-f)| > 3`. The hardcoded `boundX = 3` then **underestimates** the true Taylor remainder, causing `taylorExpCmp` to declare `ABOVE` or `BELOW` prematurely with an error bound that is too tight. The comparison result is therefore incorrect.

The `PositiveUnitInterval` type allows any value in `(0, 1)`, and the only guard in `checkLeaderNatValue` is the degenerate `f == 1` case. No check prevents `f` from being set to a value where `|ln(1-f)| > 3`.

---

### Impact Explanation

**Impact: High — Deterministic disagreement between honest nodes from ledger rule evaluation.**

When `f` is set to a value above ~0.95 via a protocol parameter update:

- `taylorExpCmp 3 recip_q x` uses an incorrect error bound, producing wrong `ABOVE`/`BELOW` answers.
- `checkLeaderNatValue` returns `True` (eligible) or `False` (ineligible) incorrectly.
- Different implementations (Haskell node, alternative clients) may compute different results for the same block header, causing **deterministic ledger divergence**: some nodes accept a block while others reject it.
- This matches the allowed impact: *"Deterministic disagreement between honest nodes from ledger rule evaluation, era transition, serialization, or script/witness validation."*

---

### Likelihood Explanation

**Likelihood: Medium.**

- The active slot coefficient `f` is a governance-controlled protocol parameter. A governance action (parameter update) setting `f` above ~0.95 would trigger the bug.
- The current mainnet value is `f = 0.05`. However, the parameter is updatable via on-chain governance (Conway era), and the ledger code does not enforce an upper bound below 1 on `f` for this mathematical precondition.
- An attacker who can pass a governance proposal (or a misconfigured update) could trigger this. Under Conway governance, a constitutional committee or DRep majority could enact such a change.
- The bug is also reachable in testnets or private networks where `f` is set high for testing purposes.

---

### Recommendation

Replace the hardcoded `3` with a dynamically computed bound derived from the actual `activeSlotLog` value:

```haskell
-- Compute M = ceiling(|ln(1-f)|) + 1 as the error bound
let boundM = ceiling (abs c) + 1
case taylorExpCmp (fromIntegral boundM) recip_q x of
```

This matches the design document's guidance: *"M can easily be estimated as the smallest integer that is larger than the exponential function that is evaluated at the upper bound of the interval of the domain of x."*

Alternatively, enforce a protocol parameter constraint that `f ≤ 0.95` (or the appropriate threshold) at the ledger validation layer.

---

### Proof of Concept

**Root cause chain:**

1. A governance proposal sets `f = 0.99` (valid `PositiveUnitInterval`, passes all ledger checks).
2. `mkActiveSlotCoeff` computes `unActiveSlotLog = floor(fpPrecision * ln(1 - 0.99)) = floor(fpPrecision * ln(0.01)) ≈ floor(fpPrecision * (-4.605))`.
3. `activeSlotLog f` returns approximately `-4.605` as a `FixedPoint`.
4. In `checkLeaderNatValue`, `x = -σ * c ≈ σ * 4.605`. For `σ = 1`, `x ≈ 4.605`.
5. `taylorExpCmp 3 recip_q 4.605` is called. The true bound `M` should be `≥ 5` (since `|x| ≈ 4.605`), but `boundX = 3` is used.
6. The error term `abs(err' * 3)` is smaller than the true remainder `abs(err' * 5)`, so the function declares `ABOVE` or `BELOW` at an iteration where the partial sum has not yet converged to within the true error.
7. `checkLeaderNatValue` returns an incorrect boolean — either a non-leader is accepted (`True`) or a valid leader is rejected (`False`).
8. Nodes running different implementations or with different floating-point behavior may disagree, causing a chain fork.

```haskell
taylorExpCmp :: RealFrac a => a -> a -> a -> CompareResult a
taylorExpCmp boundX cmp x = go 1000 0 x 1 1
  where
    go maxN n err acc divisor
      | maxN == n = MaxReached n
      | cmp >= acc' + errorTerm = ABOVE acc' (n + 1)
      | cmp < acc' - errorTerm = BELOW acc' (n + 1)
      | otherwise = go maxN (n + 1) err' acc' divisor'
      where
        errorTerm = abs (err' * boundX)
        divisor' = divisor + 1
        nextX = err
        err' = (err * x) / divisor'
        acc' = acc + nextX
```
