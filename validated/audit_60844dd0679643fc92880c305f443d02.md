### Title
Attestation Expiry Computed from `block.timestamp` Instead of Certificate's Actual Validity — (`crates/mpc-attestation/src/attestation.rs`)

### Summary
The NEAR MPC contract computes a stored attestation's `expiry_timestamp_seconds` as `block.timestamp + DEFAULT_EXPIRATION_DURATION_SECONDS` (1 day) rather than extracting the actual expiry from the TDX certificate itself. This mirrors the external report's pattern exactly: a fixed offset from the current block time is used instead of the semantically correct value. As a result, a node whose TDX certificate expires before the 1-day window continues to be treated as a valid TEE participant and can participate in threshold signing operations without re-attesting.

### Finding Description

In `crates/mpc-attestation/src/attestation.rs`, `AcceptedAttestation::dstack()` stamps the stored expiry as:

```rust
// TODO(#1639): extract timestamp from certificate itself
let expiration_timestamp_seconds =
    current_timestamp_seconds + DEFAULT_EXPIRATION_DURATION_SECONDS;
``` [1](#0-0) 

where `DEFAULT_EXPIRATION_DURATION_SECONDS = 60 * 60 * 24` (1 day): [2](#0-1) 

The `current_timestamp_seconds` is the NEAR block timestamp at the moment of attestation submission. The resulting `ValidatedDstackAttestation` stores this computed expiry: [3](#0-2) 

All subsequent re-verification in the contract checks only this stored value, not the certificate's actual validity period:

```rust
let attestation_has_expired = *expiration_timestamp_seconds < timestamp_seconds;
``` [4](#0-3) 

The `Timestamp::now()` helper in the contract confirms the source is `near_sdk::env::block_timestamp()`: [5](#0-4) 

The codebase's own TODO comment acknowledges the correct fix: `// TODO(#1639): extract timestamp from certificate itself`.

### Impact Explanation

A TDX certificate carries its own validity window (issued by Intel's PCS/PCK infrastructure). If a node submits a certificate whose actual validity ends in, say, 2 hours, the contract nevertheless grants it a full 1-day validity window. During the remaining 22 hours the node:

- Passes `re_verify` checks in `reverify_participants` and `reverify_and_cleanup_participants`
- Is not evicted by `clean_invalid_attestations`
- Continues to be counted as a valid TEE participant for `verify_tee` and resharing decisions

This breaks the production safety invariant that only nodes currently provably running inside a trusted TEE may participate in threshold signing. A node whose hardware attestation has lapsed can co-sign threshold signatures for up to 1 day without re-attesting, weakening the TEE enforcement that is the system's primary trust anchor.

**Impact: Medium** — participant-state manipulation that breaks production safety/accounting invariants (TEE enforcement) without requiring network-level DoS or operator misconfiguration.

### Likelihood Explanation

The scenario is reachable by any node operator: submit a valid attestation shortly before the certificate's natural expiry. No collusion, no privileged access, and no threshold-level compromise is required. The node simply does not re-attest after its certificate expires and continues operating. The 1-day window is long enough to participate in multiple signing rounds. **Likelihood: Medium.**

### Recommendation

Replace the fixed-offset computation with the actual expiry extracted from the TDX certificate (as the existing TODO#1639 already prescribes). The contract should enforce an upper bound on the caller-supplied expiry (e.g., reject any expiry more than `DEFAULT_EXPIRATION_DURATION_SECONDS` beyond the current block time) so that a malicious node cannot supply an arbitrarily large expiry either.

```rust
// Correct approach: extract from certificate
let expiration_timestamp_seconds = certificate.not_after_unix_seconds();
// Enforce upper bound
let max_allowed = current_timestamp_seconds + DEFAULT_EXPIRATION_DURATION_SECONDS;
let expiration_timestamp_seconds = expiration_timestamp_seconds.min(max_allowed);
```

### Proof of Concept

1. Node A holds a TDX certificate valid until `T + 2h`.
2. At time `T`, node A calls `submit_participant_info`. The contract verifies the raw quote (DCAP accepts it — it is not yet expired) and stores `expiry_timestamp_seconds = T + 86400` (1 day).
3. At time `T + 2h` the certificate's actual validity lapses. Intel's PCS would now reject a fresh quote from this hardware.
4. At time `T + 3h`, any caller invokes `verify_tee`. The contract calls `reverify_participants`, which calls `re_verify` with `timestamp_seconds = T + 3h`. The check `expiry_timestamp_seconds (T + 86400) < timestamp_seconds (T + 10800)` is **false** — node A is still considered valid.
5. Node A participates in threshold signing rounds, contributing a key share, despite its TEE certificate having expired. [6](#0-5) [7](#0-6)

impl VerifiedAttestation {
    pub fn re_verify(
        &self,
        timestamp_seconds: u64,
        allowed_mpc_docker_image_hashes: &[NodeImageHash],
        allowed_launcher_docker_compose_hashes: &[LauncherDockerComposeHash],
        allowed_measurements: &[ExpectedMeasurements],
    ) -> Result<(), VerificationError> {
        match self {
            Self::Dstack(ValidatedDstackAttestation {
                mpc_image_hash,
                launcher_compose_hash,
                expiry_timestamp_seconds: expiration_timestamp_seconds,
                measurements,
            }) => {
                let attestation_has_expired = *expiration_timestamp_seconds < timestamp_seconds;

                if attestation_has_expired {
                    return Err(VerificationError::Custom(format!(
                        "The attestation expired at t = {:?}, time_now = {:?}",
                        expiration_timestamp_seconds, timestamp_seconds
                    )));
                }

                let () = verify_mpc_hash(mpc_image_hash, allowed_mpc_docker_image_hashes)?;
                let () = verify_launcher_compose_hash(
                    launcher_compose_hash,
                    allowed_launcher_docker_compose_hashes,
                )?;

                verify_measurements(measurements, allowed_measurements)?;

                Ok(())
            }
            Self::Mock(mock_attestation) => mock_attestation.verify_constraints(
                timestamp_seconds,
                allowed_mpc_docker_image_hashes,
                allowed_launcher_docker_compose_hashes,
                allowed_measurements,
            ),
        }
    }
