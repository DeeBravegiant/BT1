### Title
Unsigned `farmer_reward_address_override` in `RespondSignatures` Enables Malicious Harvester to Redirect Farmer Block Rewards - (File: chia/farmer/farmer_api.py)

### Summary
A malicious third-party harvester can redirect the farmer's block reward (250 XCH per block) to any arbitrary address by setting `farmer_reward_address_override` in the `RespondSignatures` protocol message. This field is not cryptographically signed by the harvester and is not subject to any enforced fee-quality gate, creating a direct analog to the SGX `addInstances()` bug: a security-critical parameter is excluded from the signed payload, allowing arbitrary substitution.

### Finding Description
In `_process_respond_signatures` in `chia/farmer/farmer_api.py`, the farmer unconditionally accepts `response.farmer_reward_address_override` from the harvester's `RespondSignatures` message and uses it as the farmer reward destination:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [1](#0-0) 

The harvester's cryptographic contribution in `RespondSignatures` covers only the `message_signatures` list — the signage-point hashes (`challenge_chain_sp`, `reward_chain_sp`) or foliage block data hashes. The `farmer_reward_address_override` field in `RespondSignatures` is a plain, unsigned field:

```python
class RespondSignatures(Streamable):
    ...
    message_signatures: list[tuple[bytes32, G2Element]]
    include_source_signature_data: bool
    farmer_reward_address_override: bytes32 | None   # ← NOT signed
``` [2](#0-1) 

The farmer verifies the BLS signatures over the signage-point hashes at lines 882–893, but performs zero verification on `farmer_reward_address_override`: [3](#0-2) 

The accepted `farmer_reward_address` is then forwarded verbatim as `farmer_puzzle_hash` in `DeclareProofOfSpace`: [4](#0-3) 

The full node's `declare_proof_of_space` handler trusts this value without independent verification: [5](#0-4) 

It is passed directly into `create_unfinished_block` → `create_foliage` → `FoliageBlockData.farmer_reward_puzzlehash`, which determines where the 250 XCH farmer coinbase reward is paid: [6](#0-5) 

**Second gap — fee-quality check is not enforced and does not cover `RespondSignatures`.**

The CHIP-22 fee-quality check is applied only to `farmer_reward_address_override` in the earlier `NewProofOfSpace` message (line 128), and even there it only emits a log warning — it never rejects the block: [7](#0-6) [8](#0-7) 

The `farmer_reward_address_override` in `RespondSignatures` is never checked against the fee-quality threshold at all. A malicious harvester can therefore send `NewProofOfSpace` with `farmer_reward_address_override = None` (bypassing even the log-only check) and then inject an arbitrary address in `RespondSignatures`.

### Impact Explanation
A malicious third-party harvester — a role explicitly supported by CHIP-22 — can redirect every farmer block reward (250 XCH per block) to an attacker-controlled address. This is unauthorized payout redirection of XCH coinbase rewards, matching the **High** impact category: "Bypass of … pool, or Data Layer authorization that enables … payout redirection."

### Likelihood Explanation
CHIP-22 explicitly enables third-party harvester services. Any farmer that connects to an external harvester (e.g., a GPU-accelerated or compressed-plot service) is exposed. The attacker requires no keys, no admin access, and no cryptographic break — only a network connection accepted by the farmer's SSL handshake.

### Recommendation
1. **Include `farmer_reward_address_override` in the signed payload.** The farmer should include the intended override address in the `RequestSignatures` messages it sends to the harvester, and the harvester must sign over it. The farmer then verifies that the returned `farmer_reward_address_override` matches what was signed.
2. **Enforce the fee-quality check, not just log it.** If the fee quality does not meet the harvester's declared threshold, the farmer should drop the override rather than log a warning and proceed.
3. **Bind `RespondSignatures.farmer_reward_address_override` to `NewProofOfSpace.farmer_reward_address_override`.** If the two values differ, the farmer should reject the response.

### Proof of Concept
1. Attacker operates a malicious harvester service; farmer connects to it (standard CHIP-22 third-party harvester flow).
2. Malicious harvester sends `NewProofOfSpace` with `farmer_reward_address_override = None` — no fee-quality log is triggered.
3. Farmer sends `RequestSignatures` asking the harvester to sign `[challenge_chain_sp, reward_chain_sp]`.
4. Malicious harvester produces valid BLS signatures over those hashes (it has the plot key) and returns `RespondSignatures` with `farmer_reward_address_override = attacker_puzzle_hash`.
5. `_process_respond_signatures` verifies the BLS signatures (they are valid), then sets `farmer_reward_address = attacker_puzzle_hash` with no further check.
6. `DeclareProofOfSpace` is sent to the full node with `farmer_puzzle_hash = attacker_puzzle_hash`.
7. Full node calls `create_unfinished_block` → `create_foliage` → block is created with the 250 XCH farmer coinbase coin locked to `attacker_puzzle_hash`.
8. Block is finalized and confirmed on-chain; the farmer receives 0 XCH, the attacker receives 250 XCH.
