### Title
`respond_verify_foreign_tx` Accepts Mismatched `payload_hash` — Cross-Request Replay Within Same Domain — (`crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain root key. It performs **no check** that `response.payload_hash` is actually derived from the `request` parameter supplied in the same call. A single Byzantine attested-participant node that possesses a valid `(H1, sig(H1))` pair produced by the MPC network for request R1 can submit it as the response for a different pending request R2. The contract accepts the call, resolves R2's yield with R1's payload, and delivers to R2's caller a signed observation that encodes a completely different foreign transaction.

### Finding Description

In `crates/contract/src/lib.rs`, `respond_verify_foreign_tx` (lines 691–754) performs the following checks:

1. Caller is an attested participant.
2. Protocol is running and `accept_requests` is true.
3. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key.
4. `request` exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What is **absent**: any check that `response.payload_hash` is the SHA-256 of `borsh(ForeignTxSignPayload::V1 { request: request.request, values: ... })`. The contract cannot compute this independently because it does not know the `values` (the observed foreign-chain data), but it also does not reject a `payload_hash` that encodes a completely different `ForeignChainRpcRequest`. [2](#0-1) 

After the signature check passes, `resolve_yields_for` drains R2's entire yield queue and delivers the full `response` (containing R1's `payload_hash` and signature) to every waiter on R2. [3](#0-2) 

### Impact Explanation

R2's caller receives `VerifyForeignTransactionResponse { payload_hash: H1, signature: sig(H1) }` where `H1 = SHA256(borsh(ForeignTxSignPayload::V1 { request: R1.request, values: observed_R1 }))`. The response is a genuine MPC-network signature, but it attests to R1's transaction data, not R2's. Any bridge contract that does not independently reconstruct the expected hash from its own request parameters and compare it to `response.payload_hash` will accept this as a valid attestation of R2, enabling invalid bridge execution.

The SDK helper `ForeignChainSignatureVerifier::verify_signature` in `crates/near-mpc-sdk/src/foreign_chain.rs` does perform this check: [4](#0-3) 

However, this check is **caller-side only** and is not enforced by the contract. The contract-level invariant — that the response delivered to R2's yield corresponds to R2's request — is broken.

### Likelihood Explanation

**Preconditions required:**
- A single Byzantine attested-participant MPC node (no threshold collusion needed).
- Two different `verify_foreign_transaction` requests R1 and R2 pending simultaneously for the same domain.
- The Byzantine node must possess a valid `(H1, sig(H1))` for R1. This is obtainable by: (a) participating in the threshold signing protocol for R1 and learning the assembled signature, or (b) observing R1's response submitted on-chain and replaying it for R2 while R2 is still pending.

The `assert_caller_is_attested_participant_and_protocol_active()` guard requires only that the caller is an attested participant — a single node, not a threshold quorum. [5](#0-4) 

### Recommendation

The contract cannot independently compute the expected `payload_hash` because it does not receive the `values`. Two mitigations are possible:

1. **Protocol change**: Require the responder to include the `values` in the response. The contract then computes `expected_hash = SHA256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: response.values }))` and asserts `expected_hash == response.payload_hash` before accepting.
2. **Caller enforcement**: Mandate (via documentation and SDK) that every consumer of `VerifyForeignTransactionResponse` calls `ForeignChainSignatureVerifier::verify_signature` before acting on the response. This is already the documented intent but is not enforced on-chain.

```rust
// Unit test sketch (within the existing test harness in crates/contract/src/lib.rs)
fn cross_request_replay_within_same_domain() {
    let mut rng = rand::rngs::StdRng::from_seed([42u8; 32]);
    let (context, mut contract, secret_key) =
        basic_setup_with_protocol(Protocol::CaitSith, DomainPurpose::ForeignTx, &mut rng);
    register_supported_chains(&mut contract, [dtos::ForeignChain::Bitcoin]);
    let SharedSecretKey::Secp256k1(sk) = secret_key else { unreachable!() };

    // (1) Submit R1 (tx_id = [1u8;32]) and R2 (tx_id = [2u8;32])
    let make_request = |tx_id: [u8; 32]| VerifyForeignTransactionRequestArgs {
        domain_id: DomainId::default().0.into(),
        payload_version: ForeignTxPayloadVersion::V1,
        request: dtos::ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
            tx_id: tx_id.into(),
            confirmations: 2.into(),
            extractors: vec![BitcoinExtractor::BlockHash],
        }),
    };
    testing_env!(VMContextBuilder::new()
        .predecessor_account_id("alice.near".parse().unwrap())
        .current_account_id(context.current_account_id.clone())
        .attached_deposit(NearToken::from_yoctonear(1))
        .build());
    contract.verify_foreign_transaction(make_request([1u8; 32]));
    contract.verify_foreign_transaction(make_request([2u8; 32]));

    let r1 = args_into_verify_foreign_tx_request(make_request([1u8; 32]));
    let r2 = args_into_verify_foreign_tx_request(make_request([2u8; 32]));

    // (2) Build a valid response for R1
    let payload_r1 = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
        request: r1.request.clone(),
        values: vec![ExtractedValue::BitcoinExtractedValue(
            BitcoinExtractedValue::BlockHash([0xAAu8; 32].into()),
        )],
    });
    let h1 = payload_r1.compute_msg_hash().unwrap();
    let sk_ec = elliptic_curve::SecretKey::<Secp256k1>::from_bytes(&sk.to_bytes()).unwrap();
    let signing_key = SigningKey::from_bytes(&sk_ec.to_bytes()).unwrap();
    let (sig, rec) = signing_key.sign_prehash_recoverable(&h1.0).unwrap();
    let response_r1 = VerifyForeignTransactionResponse {
        payload_hash: h1.clone(),
        signature: dtos::SignatureResponse::Secp256k1(
            dtos::K256Signature::from_ecdsa_recoverable(&sig, rec),
        ),
    };

    // (3) Submit R1's response for R2 — cross-request replay
    with_active_participant_and_attested_context(&contract);
    let result = contract.respond_verify_foreign_tx(r2.clone(), response_r1);

    // (4) Contract ACCEPTS — R2's yield is resolved with R1's payload_hash
    assert!(result.is_ok(), "contract should reject cross-request replay but accepted it");
    // R2's caller now holds a signed attestation for R1's tx_id=[1u8;32],
    // not R2's tx_id=[2u8;32].
}
```

The test demonstrates that the contract accepts the call and resolves R2's yield with R1's `payload_hash`. R2's caller receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes a different transaction than the one they requested. A bridge contract that does not call `ForeignChainSignatureVerifier::verify_signature` would treat this as a valid attestation of R2, enabling invalid bridge execution. [6](#0-5) [7](#0-6)

```rust
    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
```

```rust
        Ok(Hash256(hasher.finalize().into()))
```

```rust
impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
```
