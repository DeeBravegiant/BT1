### Title
Fee-on-Transfer Token Accounting Divergence in `initTransfer` Creates Unbacked Cross-Chain Supply - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter for both the `safeTransferFrom` call and the emitted `InitTransfer` event without measuring the actual balance received. When a fee-on-transfer ERC20 is used, the bridge locks `amount - fee` but the cross-chain message carries `amount`, causing the destination chain to mint or release more tokens than are actually held in the bridge vault.

### Finding Description
In `OmniBridge.initTransfer`, for native (non-bridge, non-custom-minter) ERC20 tokens, the code path is:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← requested amount, not actual received amount
);
```

followed immediately by:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,         // ← same caller-supplied value, not balance delta
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) [2](#0-1) 

There is no balance-before/balance-after check. The `amount` in the emitted event is the parameter value, not the actual tokens received. The NEAR-side prover reads this event and constructs an `InitTransferMessage` with `amount` as the credited value. [3](#0-2) 

The NEAR bridge then processes this message, minting or releasing `amount` tokens to the recipient on the destination chain. [4](#0-3) 

There is no token whitelist enforced in `initTransfer` for the native ERC20 path — any ERC20 address can be passed. [5](#0-4) 

### Impact Explanation
For every `initTransfer` call with a fee-on-transfer token, the bridge vault receives `amount - transferFee` but the cross-chain message commits to `amount`. The destination chain mints or releases `amount` of wrapped tokens. The bridge is immediately undercollateralized by `transferFee` per transaction. Repeated calls drain the backing reserve, eventually allowing redemptions that cannot be honored — a permanent, irreversible backing guarantee break. This falls under **Critical: unauthorized creation of unbacked wrapped bridge assets through settlement failure**.

### Likelihood Explanation
- No privileged role is required; `initTransfer` is a public, unpaused entrypoint callable by any EOA.
- No token whitelist exists in the native ERC20 branch of `initTransfer`.
- Fee-on-transfer tokens (e.g., USDT with fees enabled, STA, PAXG) are real, deployed ERC20s. If any such token is registered with the bridge's NEAR-side token mappings, every bridge user inadvertently triggers the discrepancy.
- An attacker can also deploy a custom fee-on-transfer ERC20, register it through the normal metadata/deploy flow, and systematically drain the bridge's backing for that token.

### Recommendation
Measure the actual balance change rather than trusting the caller-supplied `amount`:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "Fee-on-transfer tokens not supported");
```

Alternatively, explicitly whitelist supported tokens and reject any token whose transfer results in a balance delta less than `amount`. Document that fee-on-transfer and rebasing tokens are unsupported.

### Proof of Concept
1. Deploy a fee-on-transfer ERC20 `FeeToken` that deducts 1% on every transfer.
2. Register `FeeToken` with the Omni Bridge (log metadata, deploy wrapped token on NEAR).
3. Call `initTransfer(feeTokenAddress, 1_000_000, 0, 0, "near:attacker.near", "")`.
4. `safeTransferFrom` moves 1,000,000 units but the bridge receives only 990,000 (1% fee taken).
5. The emitted `InitTransfer` event carries `amount = 1_000_000`.
6. The NEAR prover reads the event; `InitTransferMessage.amount = U128(1_000_000)`. [6](#0-5) 

7. NEAR bridge mints 1,000,000 wrapped `FeeToken` to `attacker.near`.
8. The EVM bridge vault holds only 990,000 `FeeToken`.
9. Attacker redeems 1,000,000 wrapped tokens on NEAR, triggering `finTransfer` on EVM for 1,000,000 — but the vault only has 990,000, causing the final redemption to fail or drain other users' funds.
10. Repeat to widen the undercollateralization gap. [7](#0-6)
