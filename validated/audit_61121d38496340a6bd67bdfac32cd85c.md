### Title
Caller-Supplied `payload_hash` in `respond_verify_foreign_tx` Enables Forged Foreign-Chain Verification Response - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies the MPC signature against a `payload_hash` that is entirely caller-supplied in the response body, without ever recomputing or binding that hash to the on-chain pending `request`. A single malicious attested participant can replay any previously observed `(payload_hash, signature)` pair from a past call to satisfy a new, unrelated pending verification request, causing the contract to deliver a forged `VerifyForeignTransactionResponse` to the waiting user.

### Finding Description

In `respond_verify_foreign_tx`, the signature validity check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← caller-supplied
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract only checks that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key. It never recomputes the expected hash from the pending `request` (i.e., `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))`), so there is no binding between the `payload_hash` in the response and the actual foreign-chain transaction the user submitted.

Contrast this with the regular `respond` function, where the payload to verify against is taken directly from the immutable on-chain request:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [2](#0-1) 

The `VerifyForeignTransactionResponse` struct carries both fields as opaque caller-supplied data: [3](#0-2) 

After the signature check passes, `resolve_yields_for` drains the pending queue for the `request` key and delivers the raw serialized `response` bytes (including the attacker-chosen `payload_hash`) to every waiting yield: [4](#0-3) 

**Exploit path:**

1. Attacker is a single malicious attested participant (below threshold).
2. A legitimate `respond_verify_foreign_tx` call for request A is observed on-chain; its `payload_hash_A` and `signature_A` are public.
3. A user submits `verify_foreign_transaction` for a different request B (different `tx_id`, different extracted values, different expected `payload_hash_B`).
4. The attacker calls `respond_verify_foreign_tx(request_B, {payload_hash: payload_hash_A, signature: signature_A})`.
5. The contract verifies `verify_ecdsa_signature(signature_A, payload_hash_A, root_pk)` → passes (the signature is genuinely valid over `payload_hash_A`).
6. The contract resolves the yield for request B, delivering `{payload_hash: payload_hash_A, signature: signature_A}` to the user.
7. The user's bridge application receives a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to their actual transaction.

### Impact Explanation

A bridge application that trusts the contract's response without independently recomputing the expected `payload_hash` (as the `near-mpc-sdk`'s `ForeignChainSignatureVerifier::verify_signature` does) will accept a forged attestation. This enables invalid bridge execution: the application believes a specific foreign-chain transaction (e.g., a Bitcoin deposit) was verified by the MPC network, when in fact a completely different transaction's hash was signed. This can cause double-spend conditions or unauthorized asset releases on the bridge side. [5](#0-4) 

The SDK-level check (`expected_payload_hash == response.payload_hash`) is the only guard, but it is off-chain and optional — the contract itself provides no enforcement.

### Likelihood Explanation

The attacker must be a single attested MPC participant (Byzantine participant below the signing threshold). All past `(payload_hash, signature)` pairs are permanently visible on-chain. No threshold collusion is required — one rogue node can execute this attack independently on any pending `verify_foreign_transaction` request.

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the on-chain `request` fields and the `extracted_values` embedded in the response, then assert it equals `response.payload_hash` before accepting the signature. Specifically:

```rust
// Recompute expected hash from the request and the values the node claims to have extracted
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // extracted_values must be added to the response DTO
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include the `extracted_values` in the `VerifyForeignTransactionResponse` DTO so the contract can independently verify the hash binding, mirroring how `respond` derives the payload from the immutable on-chain request. [6](#0-5) 

### Proof of Concept

```rust
// Step 1: Observe a past respond_verify_foreign_tx call on-chain for request_A.
// payload_hash_A and signature_A are public.

// Step 2: User submits verify_foreign_transaction for request_B (different tx_id).
contract.verify_foreign_transaction(request_args_B); // pending queue now has request_B

// Step 3: Malicious participant replays old (payload_hash_A, signature_A) for request_B.
let forged_response = VerifyForeignTransactionResponse {
    payload_hash: payload_hash_A,   // hash of a DIFFERENT transaction
    signature: signature_A,          // valid signature over payload_hash_A
};
// Contract checks: verify_ecdsa_signature(signature_A, payload_hash_A, root_pk) → OK
// Contract does NOT check: payload_hash_A == SHA256(borsh({request_B, extracted_values}))
contract.respond_verify_foreign_tx(request_B, forged_response).unwrap();

// Result: user's yield is resolved with payload_hash_A, not the hash of their actual tx.
// Bridge application receives a forged attestation for request_B.
``` [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

        Ok(Hash256(hasher.finalize().into()))

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
