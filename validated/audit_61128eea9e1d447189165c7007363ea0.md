### Title
Multiple Sub-Transactions Can Claim the Same Deposit Refund, Creating ADA Out of Thin Air - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

In the Dijkstra era, the `SUBUTXO` rule processes each sub-transaction independently using the **original, unmodified `CertState`** from the environment. Because `certsTotalRefundsTxBody` is evaluated against this frozen `certState` for every sub-transaction, two sub-transactions that each include an `UnRegDepositTxCert` for the **same** staking credential will each independently compute a full `keyDeposit` refund and each decrement `utxosDeposited` by that amount. The deposit pot is drained twice for a single credential registration, creating ADA that was never deposited.

---

### Finding Description

The `dijkstraSubUtxoTransition` rule in `SubUtxo.hs` calls `Shelley.updateUTxOStateNoFees` for each sub-transaction:

```haskell
dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  ...
  if isValid
    then do
      newState <-
        Shelley.updateUTxOStateNoFees
          pp
          utxoState
          txBody
          certState          -- ← same original certState for every sub-tx
          ...
``` [1](#0-0) 

Inside `updateUTxOStateNoFees`, the deposit accounting is:

```haskell
totalRefunds = certsTotalRefundsTxBody pp certState txBody
totalDeposits = certsTotalDepositsTxBody pp certState txBody
depositChange = totalDeposits <-> totalRefunds
...
utxosDeposited = utxosDeposited <> depositChange
``` [2](#0-1) 

Because `certState` is **never updated** between sub-transactions (the `SUBUTXO` rule only returns a new `UTxOState`, not a new `CertState`), if sub-transaction 1 and sub-transaction 2 both contain `UnRegDepositTxCert stakingCred keyDeposit` for the same credential:

- Sub-tx 1: `certsTotalRefundsTxBody` sees credential registered → `utxosDeposited -= keyDeposit`
- Sub-tx 2: `certsTotalRefundsTxBody` sees credential **still** registered (same frozen `certState`) → `utxosDeposited -= keyDeposit` again

The deposit pot is decremented by `2 × keyDeposit` while only `1 × keyDeposit` was ever deposited.

The top-level `validateValueNotConservedUTxO` check only validates the **top-level transaction body**, not sub-transactions:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [3](#0-2) 

The `SUBUTXO` rule has no `validateValueNotConservedUTxO` check of its own, so the double-refund passes all validation.

The developers are aware of this gap. The test suite contains a disabled test (`xit`) that explicitly names the problem but marks the predicate failure as unimplemented:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
    ...
    submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [4](#0-3) 

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

An attacker who registers one staking credential (paying `keyDeposit` once) can construct a top-level transaction containing two sub-transactions that each deregister the same credential. The `utxosDeposited` pot is decremented by `2 × keyDeposit`, but the outputs of the sub-transactions collectively receive `2 × keyDeposit` in refunds. The extra `keyDeposit` is ADA that was never deposited — it is created from nothing. Repeated at scale (many credentials, many sub-transactions), this drains the deposit pot and inflates the ADA supply, violating the preservation-of-value invariant.

---

### Likelihood Explanation

Any unprivileged transaction sender can construct this transaction. No special role, key, or governance majority is required. The only prerequisite is registering a staking credential (a routine, low-cost operation). The Dijkstra era is experimental and not yet deployed on mainnet, which limits immediate exposure, but the code path is reachable in any testnet or future mainnet deployment of the Dijkstra era.

---

### Recommendation

Before processing each sub-transaction's certificates in `dijkstraSubUtxoTransition`, the rule must track which credentials have already been deregistered by earlier sub-transactions in the same batch and exclude them from the refund calculation. Concretely:

1. Thread a **mutable or accumulated `CertState`** through the sub-transaction loop so that each sub-transaction sees the post-deregistration state of all prior sub-transactions.
2. Alternatively, collect all certificates from all sub-transactions and compute a single, deduplicated `totalRefunds` at the batch level before distributing it, mirroring the existing `validateBatchWithdrawals` pattern that already aggregates withdrawals across the batch.

The existing `validateBatchWithdrawals` function in `Utxo.hs` demonstrates the correct pattern for batch-level aggregation:

```haskell
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
``` [5](#0-4) 

The same aggregation-then-validate approach must be applied to deposit refunds.

---

### Proof of Concept

1. Register staking credential `C`, paying `keyDeposit = D` ADA. `utxosDeposited` increases by `D`.
2. Construct a top-level transaction with two sub-transactions:
   - `subTx1`: inputs `{utxo1}`, certs `[UnRegDepositTxCert C D]`, outputs `[addr1 ← D]`
   - `subTx2`: inputs `{utxo2}`, certs `[UnRegDepositTxCert C D]`, outputs `[addr2 ← D]`
3. Submit the transaction.
4. `dijkstraSubUtxoTransition` processes `subTx1`:
   - `certsTotalRefundsTxBody` with original `certState` → refund = `D`
   - `utxosDeposited -= D`; `addr1` receives `D`
5. `dijkstraSubUtxoTransition` processes `subTx2`:
   - `certsTotalRefundsTxBody` with **same** original `certState` → refund = `D` again
   - `utxosDeposited -= D`; `addr2` receives `D`
6. Net result: `utxosDeposited` decreased by `2D`, but only `D` was ever deposited. `addr1 + addr2` hold `2D`. The extra `D` ADA was created from the deposit pot.

The disabled test at `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53–75` confirms this scenario is reachable and that no predicate failure currently prevents it. [4](#0-3) [6](#0-5) [7](#0-6)

updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let UTxOState {utxosUtxo, utxosDeposited, utxosFees, utxosDonation} = utxos
      UTxO utxo = utxosUtxo
      !utxoAdd = txouts txBody -- These will be inserted into the UTxO
      {- utxoDel  = txins txb ◁ utxo -}
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      {- newUTxO = (txins txb ⋪ utxo) ∪ outs txb -}
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      deletedUTxO = UTxO utxoDel
      totalRefunds = certsTotalRefundsTxBody pp certState txBody
      totalDeposits = certsTotalDepositsTxBody pp certState txBody
      depositChange = totalDeposits <-> totalRefunds
  depositChangeEvent depositChange
  txUtxODiffEvent deletedUTxO utxoAdd
  pure $!
    UTxOState
      { utxosUtxo = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees = utxosFees
      , utxosGovState = govState
      , utxosInstantStake =
          deleteInstantStake deletedUTxO (addInstantStake utxoAdd (utxos ^. instantStakeL))
      , utxosDonation = utxosDonation
