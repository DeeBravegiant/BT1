### Title
Proposal Deposits Permanently Locked on `do_update` — No Refund Path for Non-Executed Proposals - (`crates/contract/src/update.rs`)

---

### Summary

When `do_update` executes a winning proposal it calls `entries.clear()` and `vote_by_participant.clear()`, wiping every competing proposal from storage. Because `UpdateEntry` never records the proposer's account ID or the deposit amount, and because no cancel/withdraw endpoint exists, the NEAR tokens attached to every non-executed proposal are permanently locked in the contract with no refund path.

---

### Finding Description

`propose_update` requires an attached deposit sized to cover storage for the proposal: [1](#0-0) 

The deposit is kept in the contract balance; only the *excess* above the required amount is returned to the caller: [2](#0-1) 

`UpdateEntry` stores only the update payload and its byte size — the proposer's `AccountId` and the deposit amount are never recorded: [3](#0-2) 

When threshold is reached, `do_update` removes the winning entry and then unconditionally clears **all** remaining entries and votes: [4](#0-3) 

There is no `cancel_update`, `withdraw_proposal`, or any other mechanism that would return deposits to proposers of cleared entries. The grep for `cancel_update|withdraw_proposal|refund.*deposit` in `crates/contract/src/**/*.rs` returns zero hits in `update.rs`.

---

### Impact Explanation

Every participant who called `propose_update` for a proposal that was not the one executed loses their deposit permanently. The contract balance increases by the sum of all non-executed proposal deposits. For a contract binary update the deposit is proportional to the WASM size (`env::storage_byte_cost() × (8 KB overhead + wasm_len)`); a 1 MB binary costs roughly 10 NEAR per proposal. For a config update the cost is roughly 85 milliNEAR per proposal. With N concurrent proposals, N-1 deposits are irrecoverably locked.

This breaks the balance/accounting invariant: funds paid into the contract for a specific purpose (storage reservation for a proposal) are never returned when that purpose is voided by a competing execution.

---

### Likelihood Explanation

The scenario is not adversarial — it is the normal operating mode. The README explicitly describes the multi-proposal workflow ("each participant propose a new update"), and the existing sandbox test `test_propose_update_config` already has every participant submit a separate proposal with a 100 milliNEAR deposit before voting one to threshold: [5](#0-4) 

That test does not assert that the other proposers receive refunds, confirming the loss is silent and currently untested.

---

### Recommendation

1. **Store the proposer and deposit in `UpdateEntry`**: add `proposer: AccountId` and `deposit: NearToken` fields.
2. **Refund on clear**: in `do_update`, before calling `entries.clear()`, iterate the remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each.
3. **Alternatively**, add a `cancel_update(id: UpdateId)` endpoint callable only by the original proposer that removes the entry and refunds the deposit, and issue bulk refunds in `do_update` for any entries that remain after the winning proposal executes.

---

### Proof of Concept

```
// Sandbox test outline (3 participants, threshold = 2)
// 1. participant_0 proposes config_A with 100 milliNEAR deposit  → proposal_0
// 2. participant_1 proposes config_B with 100 milliNEAR deposit  → proposal_1
// 3. participant_2 proposes config_C with 100 milliNEAR deposit  → proposal_2
//
// 4. participant_0 votes proposal_0
// 5. participant_1 votes proposal_0  → threshold reached, do_update fires
//    entries.clear() wipes proposal_1 and proposal_2
//
// 6. Assert: participant_1.balance == initial_balance - 100 milliNEAR  (deposit lost)
// 7. Assert: participant_2.balance == initial_balance - 100 milliNEAR  (deposit lost)
// 8. Assert: contract.balance increased by 200 milliNEAR
//
// All three assertions pass because do_update issues no refund transfers.
```

The existing test `test_propose_update_config` already exercises steps 1–5 with exactly this setup; adding balance assertions at step 6–8 would demonstrate the loss. [6](#0-5)
