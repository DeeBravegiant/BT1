### Title
Block-Level Reference Script Size Limit Bypassed via Sub-Transaction Reference Scripts in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the block-level reference script size check (`validateBodyRefScriptsSizeTooBig`) reuses the Conway-era `totalRefScriptSizeInBlock` function, which only iterates over top-level transaction inputs. It does not account for reference scripts in sub-transactions (`dtbrSubTransactions`). The per-transaction check correctly uses `batchNonDistinctRefScriptsSize` (which includes sub-transactions), but the per-block check uses `txNonDistinctRefScriptsSize` (which does not). An attacker can craft a block whose top-level transactions carry zero or minimal top-level reference scripts while embedding large reference scripts exclusively in sub-transactions, causing the actual total reference script deserialization work per block to exceed `maxRefScriptSizePerBlock` by an unbounded multiple.

---

### Finding Description

The Dijkstra era introduces nested transactions (`subTransactionsTxBodyL`, type `OMap TxId (Tx SubTx era)`). Each sub-transaction has its own `dstbrSpendInputs` and `dstbrReferenceInputs`, and those inputs can resolve to UTxO entries carrying reference scripts.

**Per-transaction check (correct):** `dijkstraLedgerTransition` calls `validateAllRefScriptSize`, which uses `batchNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs:264-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

This correctly sums reference script sizes across the top-level tx and all sub-transactions.

**Per-block check (incomplete):** `dijkstraBbodyTransition` calls `Conway.validateBodyRefScriptsSizeTooBig`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs:363
Conway.validateBodyRefScriptsSizeTooBig @era pp blockBody (ls ^. utxoL)
```

`validateBodyRefScriptsSizeTooBig` calls `totalRefScriptSizeInBlock`, which calls `txNonDistinctRefScriptsSize` for each top-level transaction:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs:357-370
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      ...
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
```

`txNonDistinctRefScriptsSize` only looks at the top-level transaction's own inputs:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs:183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

Sub-transaction inputs are never visited. The block-level accumulator therefore sees only the top-level reference scripts, not those in any sub-transaction.

---

### Impact Explanation

The `maxRefScriptSizePerBlock` protocol parameter was introduced specifically to cap the total reference script deserialization work per block and prevent DDoS attacks (as documented in ADR-9 and confirmed by the June 2024 mainnet attack). By placing reference scripts exclusively in sub-transactions, an attacker can submit a block where the block-level check reports 0 bytes while the actual deserialization burden is `N × maxRefScriptSizePerTx` (where N is the number of top-level transactions in the block). With default parameters (`maxRefScriptSizePerTx = 200 KiB`, `maxRefScriptSizePerBlock = 1 MiB`), a block with 10 top-level transactions each carrying 200 KiB of sub-transaction reference scripts would impose 2 MiB of deserialization work while passing the block-level check. This exceeds the intended validation limit, matching the **Medium** impact: attacker-controlled transactions exceed intended validation limits.

---

### Likelihood Explanation

This is reachable by any transaction author in the Dijkstra era (protocol version 12+). No privileged access is required. The attacker only needs to:
1. Produce UTxO entries containing large reference scripts.
2. Craft top-level Dijkstra transactions with sub-transactions whose inputs reference those UTxO entries.
3. Keep the top-level transaction's own reference inputs empty (or small) so the per-tx batch check passes.
4. Submit a block containing multiple such transactions.

The per-tx limit (`validateAllRefScriptSize` using `batchNonDistinctRefScriptsSize`) still applies per transaction, so the excess per top-level tx is bounded by `maxRefScriptSizePerTx`. However, across a full block the total can be a multiple of `maxRefScriptSizePerBlock`.

---

### Recommendation

Replace the call to `Conway.validateBodyRefScriptsSizeTooBig` in `dijkstraBbodyTransition` with a Dijkstra-specific block-level check that uses `batchNonDistinctRefScriptsSize` (or an equivalent block-level accumulator) to sum reference script sizes across both top-level transactions and all their sub-transactions. Concretely, a new `totalRefScriptSizeInBlockDijkstra` function should replace `txNonDistinctRefScriptsSize` with `batchNonDistinctRefScriptsSize` in the per-transaction accumulation step.

---

### Proof of Concept

**Root cause chain:**

1. `dijkstraBbodyTransition` delegates the block-level ref-script check to the Conway implementation: [1](#0-0) 

2. `validateBodyRefScriptsSizeTooBig` accumulates sizes using `totalRefScriptSizeInBlock`: [2](#0-1) 

3. `totalRefScriptSizeInBlock` calls `txNonDistinctRefScriptsSize` per top-level tx — sub-transactions are never visited: [3](#0-2) 

4. `txNonDistinctRefScriptsSize` only unions the top-level tx's own `referenceInputsTxBodyL` and `inputsTxBodyL`: [4](#0-3) 

5. By contrast, the per-tx check in `dijkstraLedgerTransition` correctly uses `batchNonDistinctRefScriptsSize`, which folds over sub-transactions: [5](#0-4) [6](#0-5) 

6. Sub-transactions are stored in `dtbrSubTransactions` of the top-level `DijkstraTxBodyRaw`, and each sub-transaction has its own independent input sets: [7](#0-6) [8](#0-7) 

**Attack scenario:**

- Attacker pre-populates the UTxO with outputs carrying large reference scripts (e.g., 200 KiB each).
- Attacker crafts top-level Dijkstra transactions with empty `referenceInputsTxBodyL` and `inputsTxBodyL` at the top level, but with sub-transactions whose `dstbrReferenceInputs` reference those large-script UTxO entries.
- `validateAllRefScriptSize` (per-tx) passes because `batchNonDistinctRefScriptsSize` ≤ `maxRefScriptSizePerTx`.
- `validateBodyRefScriptsSizeTooBig` (per-block) passes because `totalRefScriptSizeInBlock` sees 0 bytes from the top-level inputs.
- The block is accepted with actual reference script deserialization work far exceeding `maxRefScriptSizePerBlock`.
