Audit Report

## Title
`maxUncompletedWithdrawalCount` Cap Causes Temporary Freezing of User rsETH in `LRTWithdrawalManager` — (`contracts/NodeDelegator.sol`, `contracts/LRTUnstakingVault.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary

`NodeDelegator.initiateUnstaking()` enforces a hard cap of 80 on concurrent in-flight EigenLayer withdrawals. When this cap is saturated through normal operations, operators cannot queue new EigenLayer withdrawals, preventing assets from reaching `LRTUnstakingVault`. Because `LRTWithdrawalManager.unlockQueue()` requires a non-zero vault balance to proceed, user withdrawal requests — for which rsETH has already been irrevocably transferred to `LRTWithdrawalManager` — cannot be serviced. No cancellation path exists, leaving user rsETH frozen for up to 7 days per EigenLayer withdrawal cycle.

## Finding Description

The withdrawal lifecycle requires assets to flow: EigenLayer strategies → `LRTUnstakingVault` → `LRTWithdrawalManager` → user. Step 1 of this flow is gated by a hard cap check in `NodeDelegator.initiateUnstaking()`:

```solidity
// contracts/NodeDelegator.sol L304-306
if (_getUnstakingVault().uncompletedWithdrawalCount() >= _getUnstakingVault().maxUncompletedWithdrawalCount()) {
    revert MaxUncompletedWithdrawalsReached();
}
``` [1](#0-0) 

`maxUncompletedWithdrawalCount` is set by the LRT Manager and is hard-capped at 80:

```solidity
// contracts/LRTUnstakingVault.sol L153-154
if (_maxUncompletedWithdrawalCount > 80) {
    revert MaxUncompletedWithdrawalCountTooHigh();
}
``` [2](#0-1) 

When the cap is saturated, `unlockQueue` cannot proceed because `_createUnlockParams` reads `unstakingVault.balanceOf(asset)` as `totalAvailableAssets`: [3](#0-2) 

And `unlockQueue` immediately reverts if this is zero: [4](#0-3) 

Meanwhile, `initiateWithdrawal` irrevocably transfers the user's rsETH to the contract at call time: [5](#0-4) 

A search of the codebase confirms no `cancelWithdrawal` function exists anywhere in `LRTWithdrawalManager.sol`. Users have no mechanism to reclaim their rsETH once a withdrawal request is queued.

## Impact Explanation

This is a **Medium — Temporary freezing of funds**. User rsETH is held in `LRTWithdrawalManager` with no cancellation path. When the cap is saturated, no new EigenLayer unstaking can be initiated, no assets flow into `LRTUnstakingVault`, and no queued withdrawal requests can be unlocked or completed. The freeze duration is bounded by EigenLayer's withdrawal delay (up to 7 days per batch), but can compound across multiple cycles if the cap remains saturated. The freeze is temporary, not permanent, which places this squarely in the Medium category.

## Likelihood Explanation

During periods of elevated withdrawal demand (market stress, large rsETH holders exiting), operators will naturally saturate the cap of 80 through normal, non-adversarial operations. Once saturated, every new user withdrawal request enters a queue that cannot be serviced until an existing EigenLayer withdrawal completes. The cap is a fixed ceiling with no dynamic adjustment mechanism. This is a realistic, non-adversarial scenario requiring no special attacker capability — only high protocol utilization.

## Recommendation

1. **Add a withdrawal cancellation path** — allow users to cancel a queued withdrawal and reclaim their rsETH if the request has not yet been unlocked (`nonce >= nextLockedNonce[asset]`), preventing indefinite rsETH lockup when the cap is saturated.
2. **Maintain a liquid buffer in `LRTUnstakingVault`** — keep a portion of assets outside EigenLayer to service withdrawal requests without requiring new `initiateUnstaking` calls.
3. **Expose cap utilization** — add a view function returning `uncompletedWithdrawalCount / maxUncompletedWithdrawalCount` so operators can proactively manage the queue before saturation.
4. **Consider raising the hard cap** — the comment in `setMaxUncompletedWithdrawalCount` notes 120 is the gas-safe upper bound; the current ceiling of 80 leaves limited headroom. [6](#0-5) 

## Proof of Concept

1. Protocol reaches `uncompletedWithdrawalCount == maxUncompletedWithdrawalCount == 80` through normal operator operations.
2. Alice calls `LRTWithdrawalManager.initiateWithdrawal(stETH, 10e18, "")` — her rsETH is transferred to `LRTWithdrawalManager` at L166, `assetsCommitted[stETH]` increases.
3. Operator attempts `NodeDelegator.initiateUnstaking([stETHStrategy], [shares])` to move stETH from EigenLayer to `LRTUnstakingVault` — **reverts** with `MaxUncompletedWithdrawalsReached` at L305.
4. `LRTUnstakingVault.balanceOf(stETH) == 0` (all stETH is in EigenLayer strategies).
5. Operator attempts `LRTWithdrawalManager.unlockQueue(stETH, ...)` — **reverts** with `AmountMustBeGreaterThanZero` at L297.
6. Alice's rsETH remains locked in `LRTWithdrawalManager`. No `cancelWithdrawal` exists. She must wait up to 7 days for one of the 80 existing EigenLayer withdrawals to complete, freeing a slot for a new `initiateUnstaking` call.
