Audit Report

## Title
Missing Chainlink Price Feed Staleness Validation Allows Stale Price Exploitation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `answer`, performing no staleness, round-completion, or validity checks. A stale or invalid Chainlink price flows directly into rsETH mint calculations, allowing an attacker to mint excess rsETH during a staleness window at the expense of existing holders. The same protocol already applies the missing checks in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the omission is unintentional.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at line 52 silently discards `roundId`, `startedAt`, `updatedAt`, and `answeredInRound`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No check is made for:
- `answeredInRound >= roundId` (stale round detection)
- `updatedAt != 0` (incomplete round detection)
- `price > 0` (invalid/negative price detection)
- `block.timestamp - updatedAt <= heartbeat` (feed freshness)

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` — used for pool collateral pricing in the same protocol — explicitly validates all three conditions before returning:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The unchecked price propagates through `LRTOracle.getAssetPrice()` into `LRTDepositPool.getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

**Exploit flow:**
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale at a price above the true market rate.
2. Attacker calls `depositAsset(asset, amount, 0, "")` — a fully public, permissionless function.
3. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(asset)` returns the stale inflated price.
4. `rsethAmountToMint = (amount * inflatedPrice) / rsETHPrice` mints more rsETH than the deposit is worth.
5. Attacker redeems or sells the excess rsETH, extracting ETH value from existing holders.

No existing guard prevents this at deposit time. The `pricePercentageLimit` check in `LRTOracle._updateRsETHPrice()` only fires when a privileged manager calls `updateRSETHPrice()`, not during user deposits.

## Impact Explanation
**Critical — Direct theft of user funds (at-rest).**

Excess rsETH minted against a stale inflated price represents a claim on more ETH than was deposited. When redeemed, this ETH comes from the pool backing existing holders, permanently diluting their redemption value. The rsETH supply is inflated relative to actual ETH backing, constituting direct, irreversible theft from at-rest depositors.

## Likelihood Explanation
**Medium.** Chainlink LST/ETH feeds on Ethereum mainnet carry 24-hour heartbeats and 0.5% deviation thresholds. Staleness events occur during prolonged network congestion, Chainlink keeper outages, or feed deprecation. The risk is elevated for L2 deployments (the repository contains `contracts/L2/`) where sequencer downtime can prevent oracle updates entirely. The exploitation window is bounded by staleness duration, but no on-chain mechanism blocks deposits during that window. The protocol's own use of staleness checks in `ChainlinkOracleForRSETHPoolCollateral` confirms the team is aware of this risk class.

## Recommendation
Mirror the validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral` by adding staleness and validity checks to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optionally: if (block.timestamp - updatedAt > HEARTBEAT) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

## Proof of Concept
**Foundry fork test outline:**

```solidity
// Fork mainnet at a block where a stETH/ETH Chainlink feed is stale
// (or mock the feed to return a stale round with inflated price)
function testStaleChainlinkPriceMintExcess() public {
    // 1. Deploy/fork with ChainlinkPriceOracle pointing to a mock feed
    MockChainlinkFeed feed = new MockChainlinkFeed();
    // Set stale: answeredInRound < roundId, updatedAt = block.timestamp - 25 hours
    feed.setRoundData(10, 1.05e18 /* inflated */, 0, block.timestamp - 25 hours, 9);

    // 2. Attacker deposits 1 stETH
    uint256 rsethBefore = rsETH.balanceOf(attacker);
    vm.prank(attacker);
    depositPool.depositAsset(stETH, 1e18, 0, "");
    uint256 rsethAfter = rsETH.balanceOf(attacker);

    // 3. Assert attacker received more rsETH than fair value
    // rsethAfter - rsethBefore > fair rsETH for 1 stETH at true price
    assertGt(rsethAfter - rsethBefore, fairRsETHAmount);
}
```

The test demonstrates that without staleness checks, `getAssetPrice` returns the stale inflated value, `getRsETHAmountToMint` computes an excess mint amount, and `depositAsset` mints it to the attacker — all through a fully public call path with no privilege requirements.
