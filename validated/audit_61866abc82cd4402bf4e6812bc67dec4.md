### Title
Permanently Locked Bridge Assets When EVM Recipient Is Blacklisted by Token — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
When a user initiates a NEAR→EVM bridge transfer, their tokens are locked or burned on NEAR at `init_transfer` time. The MPC then signs a payload that cryptographically binds the EVM recipient address. If that recipient address is subsequently blacklisted by the EVM token (e.g., USDT, USDC, or any token with a transfer blacklist), every call to `finTransfer` on the EVM side will revert. Because the MPC signature is bound to the blacklisted recipient, no valid alternative finalization is possible, and there is no cancel or refund path on the NEAR side. The locked/burned tokens are permanently unclaimable.

### Finding Description

**Step 1 — Tokens locked/burned on NEAR at `init_transfer` time.**

In `near/omni-bridge/src/lib.rs`, `init_transfer_internal` irreversibly locks or burns the user's tokens before any cross-chain confirmation:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [1](#0-0) 

There is no user-callable cancel or refund function. `remove_transfer_message` is only invoked internally in specific callback paths.

**Step 2 — MPC signature cryptographically binds the recipient.**

The MPC signs the transfer payload including the exact EVM recipient address. The signed payload cannot be altered — a different recipient would require a new MPC signature, which is not a standard protocol path available to users.

**Step 3 — `finTransfer` on EVM reverts if recipient is blacklisted.**

In `evm/src/omni-bridge/contracts/OmniBridge.sol`, `finTransfer` marks the nonce consumed first, then calls `safeTransfer` to the recipient:

```solidity
completedTransfers[payload.destinationNonce] = true;
// ...
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [2](#0-1) 

If `payload.recipient` is blacklisted by the token contract, `safeTransfer` reverts. Because Solidity reverts roll back all state changes, `completedTransfers[nonce]` is also rolled back — the nonce is not consumed. However, the MPC-signed payload still encodes the blacklisted recipient, so every retry of `finTransfer` with the same valid signature will revert identically. No alternative finalization is possible.

**Step 4 — No recovery path exists.**

The NEAR bridge has a `fin_transfer_send_tokens_callback` that handles failed token sends when NEAR is the *destination* chain — it reverts lock actions and removes the fin-transfer record: [3](#0-2) 

But when EVM is the destination, the EVM `finTransfer` is called by the relayer independently. There is no cross-chain callback from EVM back to NEAR to signal failure and trigger a refund. The NEAR-side transfer record remains in `pending_transfers` indefinitely, and the locked/burned tokens are unrecoverable.

The same pattern exists in the Starknet bridge's `fin_transfer`: [4](#0-3) 

### Impact Explanation

This matches the **Critical** impact category: *Irreversible fund lock, frozen redemption path, or permanently unclaimable user value in bridge flows.*

Any user who initiates a NEAR→EVM transfer of a token that implements address-level transfer restrictions (USDT, USDC, and many others) and whose EVM recipient address is blacklisted between `init_transfer` and `finTransfer` will have their tokens permanently locked in the NEAR bridge contract (or permanently burned if the token is a deployed bridge token), with no protocol-level recovery path.

### Likelihood Explanation

Moderate. The Omni Bridge explicitly supports arbitrary ERC-20 tokens, including well-known tokens with blacklists (USDT, USDC). Blacklisting of addresses by token issuers is a real, documented occurrence. The window between `init_transfer` on NEAR and `finTransfer` on EVM can span multiple blocks or longer if the relayer is delayed. A user whose address is blacklisted during this window loses their funds permanently.

### Recommendation

1. **Add a cancel/refund mechanism on NEAR.** Allow a user (or DAO) to cancel a pending outbound transfer after a timeout, reversing the lock or re-minting burned tokens to the original sender.
2. **Implement a cross-chain failure acknowledgment path.** If `finTransfer` on EVM fails for a non-replay reason (e.g., blacklisted recipient), allow the relayer to submit a signed failure proof back to NEAR to trigger a refund.
3. **Consider a two-step EVM finalization.** Separate nonce consumption from token delivery so that a failed delivery does not permanently block the transfer but allows an alternative recipient or refund to be specified via a privileged (multisig/timelock) recovery call.

### Proof of Concept

1. Alice holds 10,000 USDT on NEAR and calls `ft_transfer_call` → `init_transfer` targeting her EVM address `0xAlice`. The NEAR bridge locks 10,000 USDT. [5](#0-4) 

2. The MPC signs a `TransferMessagePayload` with `recipient = 0xAlice` and `tokenAddress = USDT_EVM`.

3. Before the relayer submits `finTransfer`, USDT's issuer blacklists `0xAlice` on EVM.

4. The relayer calls `OmniBridge.finTransfer(signatureData, payload)`. The call reaches:
   ```solidity
   IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
   ```
   USDT reverts because `0xAlice` is blacklisted. The entire transaction reverts; `completedTransfers[nonce]` is rolled back. [6](#0-5) 

5. Every subsequent retry with the same MPC-signed payload fails identically. No alternative payload can be constructed without a new MPC signature.

6. Alice's 10,000 USDT remain locked in the NEAR bridge contract forever. There is no cancel function, no timeout refund, and no cross-chain failure acknowledgment mechanism.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
