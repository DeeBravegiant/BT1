### Title
Wallet Sync Crash via Hintless CAT Coin Spend Triggering Unguarded `assert` in `handle_cat` â€” (`chia/wallet/wallet_state_manager.py`)

### Summary

An unprivileged attacker who holds any amount of a DEFAULT_CATS token (e.g., Spacebucks) can craft a valid CAT spend that produces an output coin with no hint. When a victim's wallet â€” which has an existing CAT wallet for that asset_id â€” processes the resulting coin state, it calls `handle_cat`, which unconditionally asserts `hinted_coin.hint is not None` at line 1247. Because `compute_spend_hints_and_additions` legitimately returns `hint=None` for hintless coins, this assertion fires, raising `AssertionError` and halting coin state processing for the affected block.

### Finding Description

**`compute_spend_hints_and_additions`** (in `chia/wallet/util/compute_hints.py`) explicitly models hint as optional: [1](#0-0) 

`hint` is initialized to `None` and only set if the CREATE_COIN condition contains a valid 32-byte third argument. A CAT spend that omits the hint argument in its CREATE_COIN condition is fully valid at the consensus level â€” hints are a wallet convention, not a consensus rule.

**`handle_cat`** (in `chia/wallet/wallet_state_manager.py`) then does: [2](#0-1) 

This is a hard `assert`, not a guarded `if`/`return None`. If `hinted_coin.hint` is `None`, Python raises `AssertionError`. There is no `try/except` wrapping this call in the visible code path.

The identical pattern also exists in `handle_did` at line 1386: [3](#0-2) 

**Attack path:**

1. Victim has a CAT wallet for a DEFAULT_CATS asset_id (e.g., Spacebucks). The wallet subscribes to the corresponding CAT puzzle hashes (derived from the victim's inner puzzle keys + asset_id).
2. Attacker acquires a small amount of that DEFAULT_CAT token (publicly available on DEXes).
3. Attacker crafts a CAT spend that sends the token to the victim's CAT puzzle hash **without** including a hint in the CREATE_COIN condition.
4. The spend is valid at consensus â€” no signature from the victim is required; the attacker signs their own spend.
5. The spend is included in a block. The victim's full node delivers the new coin state to the wallet (via puzzle hash subscription, since the victim's wallet is already subscribed to that puzzle hash).
6. The wallet calls `handle_cat` â†’ `compute_spend_hints_and_additions` returns `HintedCoin(coin, None)` â†’ `assert hinted_coin.hint is not None` â†’ `AssertionError`.
7. Coin state processing for the block halts. On retry, the same block is re-processed, the same assertion fires again â€” permanent sync failure for the affected wallet.

**DEFAULT_CATS relevance:** The `default_cats` check at line 1325 is reached only *after* the assert at line 1247. The DEFAULT_CATS asset_ids are relevant because they are the most likely asset_ids for which victim wallets already have CAT wallets (and thus puzzle hash subscriptions), making the attack broadly applicable without requiring the victim to have any unusual configuration. ### Impact Explanation

This falls under **High: Corruption of wallet sync state with direct security impact** â€” specifically, permanent or long-lived inability for an honest wallet to process valid blocks. The wallet cannot skip the offending coin state and will re-encounter the same `AssertionError` on every sync retry, causing a permanent sync stall until the user manually intervenes or the software is patched.

### Likelihood Explanation

- Attacker needs only a trivial amount of any DEFAULT_CATS token (purchasable on-chain).
- No victim key material, admin access, or cryptographic break is required.
- The victim only needs to have previously created a CAT wallet for the targeted asset_id â€” a common configuration for any user who has interacted with DEFAULT_CATS tokens.
- The malicious spend is indistinguishable from a normal spend at the consensus level.

### Recommendation

Replace the `assert` statements in `handle_cat` (line 1247) and `handle_did` (line 1386) with graceful `None` checks:

```python
# Instead of:
assert hinted_coin.hint is not None, f"hint missing for coin {hinted_coin.coin}"

# Use:
if hinted_coin.hint is None:
    self.log.warning(f"No hint for coin {hinted_coin.coin}, skipping")
    return None
```

This matches the existing pattern used elsewhere in the function (e.g., `return None` on unrecognized puzzle types at lines 1250â€“1252).
