Audit Report

## Title
`RSETHPriceFeed.getRoundData` Returns Corrupted Historical Prices by Mixing Historical ETH/USD with Current rsETH Rate - (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed` implements `AggregatorV3Interface` and is deployed as a production Chainlink-compatible RSETH/USD price feed. Its `getRoundData` function retrieves the historical ETH/USD price for a requested round but always multiplies by the **current** rsETH/ETH rate from `RS_ETH_ORACLE.rsETHPrice()`, producing a fabricated answer that is neither the historical nor the current RSETH/USD price. Any external DeFi protocol that calls `getRoundData` for historical deviation or staleness validation will receive a corrupted value, potentially triggering false circuit-breakers that freeze rsETH-collateralized positions.

## Finding Description
In `contracts/oracles/RSETHPriceFeed.sol` lines 53–61:

```solidity
function getRoundData(uint80 _roundId)
    external view
    returns (uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound)
{
    (roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.getRoundData(_roundId);
    answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
}
```

`ETH_TO_USD.getRoundData(_roundId)` correctly returns the historical ETH/USD price for round `_roundId`. However, `RS_ETH_ORACLE.rsETHPrice()` — defined via the `IRSETHOracle` interface at line 23 — exposes only a single current-rate getter with no historical lookup parameter. There is no mechanism in the contract or its dependencies to retrieve the rsETH/ETH rate that was valid at the time of the queried round. The result is:

```
returned_answer = current_rsETH/ETH × historical_ETH/USD
```

instead of:

```
correct_answer = historical_rsETH/ETH × historical_ETH/USD
```

The `latestRoundData` path (lines 63–70) is unaffected because both components are current there. The discrepancy between the returned and correct historical answer grows monotonically as staking rewards cause rsETH/ETH to drift upward over time.

## Impact Explanation
`RSETHPriceFeed` is deployed in production (confirmed as `RSETHPriceFeed (Morph)` in the README) specifically to be registered as a Chainlink-compatible feed in external lending and derivatives protocols. Any such protocol that calls `getRoundData` to validate that the latest price has not deviated excessively from a recent historical round will receive a fabricated answer. Because rsETH/ETH increases monotonically with staking rewards, the returned "historical" price will be systematically inflated relative to the true historical value. This inflated historical price, compared against the correct current price, produces an artificially large apparent price drop, which can trigger false circuit-breakers or staleness reverts in integrating protocols — **temporarily freezing rsETH-collateralized positions** held by rsETH holders. This matches the allowed Medium impact: **Temporary freezing of funds**.

## Likelihood Explanation
`getRoundData` is a standard, permissionlessly callable function of `AggregatorV3Interface`. Protocols that register a Chainlink-compatible feed routinely call `getRoundData` for historical deviation checks as a best-practice safety pattern. The `RSETHPriceFeed` contract is deployed precisely to be consumed by such protocols. The rsETH/ETH rate increases continuously as staking rewards accrue, so the magnitude of the corrupted answer grows over time, making false circuit-breaker triggers increasingly likely the longer the feed is in use.

## Recommendation
1. **Preferred**: Revert `getRoundData` with an explicit `NotSupported()` error so integrating protocols cannot silently consume incorrect data.
2. **Alternative**: Store historical rsETH/ETH prices on-chain keyed by ETH/USD `roundId` at each price update, and use the stored value in `getRoundData`.
3. At minimum, document prominently that `RSETHPriceFeed` must only be registered in protocols that exclusively use `latestRoundData` and never `getRoundData`.

## Proof of Concept
1. At time T0: rsETH/ETH = 1.05, ETH/USD round R = 2000. True RSETH/USD at R = 2100.
2. At time T1: rsETH/ETH = 1.10 (staking rewards accrued), ETH/USD latest round R+N = 1800. Current RSETH/USD = 1980.
3. External protocol calls `RSETHPriceFeed.getRoundData(R)` to check historical deviation.
4. Returned answer: `1.10 × 2000 = 2200` (inflated; correct value is 2100).
5. Protocol computes deviation: `(2200 − 1980) / 2200 ≈ 10%` — exceeds a typical 5–8% circuit-breaker threshold.
6. Protocol reverts or pauses, freezing rsETH-collateralized positions.

**Foundry fork test plan**: Fork mainnet/Morph at a block where rsETH/ETH has drifted from an earlier block. Deploy or reference the live `RSETHPriceFeed`. Call `getRoundData` with a round ID from the earlier block. Assert that the returned answer differs from `historical_rsETH_per_ETH × historical_ETH_USD` and matches `current_rsETH_per_ETH × historical_ETH_USD`, confirming the corruption.
