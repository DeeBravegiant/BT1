### Title
Single-connection DNS seeder promotion via `is_reliable()` bootstrap branch — (`chia/seeder/peer_record.py`)

### Summary
`PeerReliability.is_reliable()` contains a bootstrap branch that returns `True` after a single successful connection (`tries=1, successes=1`). Because the Chia crawler calls `peer_connected` (and thus `reliability.update(True, ...)`) upon receiving any `NewPeak` message with sufficient height, an attacker who accepts one crawler connection and sends a single valid-height `NewPeak` is immediately promoted to the `good_peers` DNS table, with no requirement for sustained availability across multiple time windows.

### Finding Description

`PeerReliability.is_reliable()` has six branches. The first:

```python
if self.tries > 0 and self.tries <= 3 and self.successes * 2 >= self.tries:
    return True
``` [1](#0-0) 

With `tries=1` and `successes=1` (one successful connection), this evaluates to `True` immediately. The remaining five branches require exponentially weighted moving average statistics accumulated over 2h, 8h, 1d, 1w, and 1m windows with minimum observation counts — none of which apply here.

The `update()` method increments `tries` and `successes` on every call:

```python
self.tries += 1
if is_reachable:
    self.successes += 1
``` [2](#0-1) 

`peer_connected` calls `reliability.update(True, ...)` whenever a peer sends a `NewPeak` with `height >= minimum_height`: [3](#0-2) [4](#0-3) 

`load_reliable_peers_to_db` then writes every peer where `is_reliable()` is True into the `good_peers` table, which the DNS server reads: [5](#0-4) 

### Impact Explanation

The `good_peers` table is the sole source of IPs served by the Chia DNS introducer to bootstrapping nodes. An attacker who can insert their IP into this table can redirect new nodes to attacker-controlled peers. With enough attacker-controlled IPs in the DNS response pool, a bootstrapping node may connect exclusively to attacker peers, enabling an eclipse attack. An eclipsed node can be fed a forked chain, causing consensus divergence for that node.

The attack requires only:
- Getting the attacker's IP gossiped into the crawler's peer list (trivial via any connected node's `RespondPeers`)
- Accepting one TCP connection from the crawler and sending one `NewPeak` message with `height >= minimum_height` (no signature or chain proof required)

This can be repeated every crawl cycle (every ~15 seconds per `asyncio.sleep(15)` in `crawl()`) to maintain presence in `good_peers`. [6](#0-5) 

### Likelihood Explanation

The attack is trivially automatable. The attacker needs no special resources beyond a publicly reachable IP and the ability to speak the Chia peer protocol (open source). The crawler actively connects outbound to all known peers, so the attacker does not need to initiate the connection — they only need to respond correctly once.

### Recommendation

Remove or harden the bootstrap branch. Options:
- **Remove the first branch entirely** and require peers to accumulate statistics in at least one time window (e.g., `stat_2h` with `count > 2`) before being considered reliable.
- **Raise the minimum tries threshold** to a value that requires observations across multiple crawl cycles (e.g., `tries >= 8`) before the bootstrap branch can fire.
- **Require a minimum time span** between first and last observation before any branch of `is_reliable()` returns True.

### Proof of Concept
