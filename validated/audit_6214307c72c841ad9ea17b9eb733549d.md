Audit Report

## Title
Unbounded Nested Loop in `NodeDelegator.getAssetUnstaking()` Causes DoS on All Deposit and Price-Update Paths - (File: contracts/NodeDelegator.sol)

## Summary
`NodeDelegator.getAssetUnstaking()` performs a nested loop over all EigenLayer-queued withdrawals and their strategies, making multiple external calls per iteration. This view function is invoked on every call to `depositETH()`, `depositAsset()`, and `updateRSETHPrice()`. As the number of NDCs, queued withdrawals, and strategies grows through normal protocol scaling, the cumulative gas cost can exceed the block gas limit, permanently reverting all deposits and price updates.

## Finding Description
`NodeDelegator.getAssetUnstaking()` at L405–427 fetches all queued withdrawals from EigenLayer's `DelegationManager` and iterates over them with a nested loop, making external calls to `strategy.underlyingToken()` and `strategy.sharesToUnderlyingView()` inside the inner loop:

```solidity
// contracts/NodeDelegator.sol L405-427
function getAssetUnstaking(address asset) external view returns (uint256 amount) {
    (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
        _getDelegationManager().getQueuedWithdrawals(address(this));
    for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
        for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
            // external calls: strategy.underlyingToken(), strategy.sharesToUnderlyingView()
        }
    }
}
```

This function is called inside `LRTDepositPool.getAssetDistributionData()` (L446–456) for every NDC in `nodeDelegatorQueue`. `getAssetDistributionData()` feeds into `getTotalAssetDeposits()`, which is called by `_checkIfDepositAmountExceedesCurrentLimit()` inside `_beforeDeposit()` (L661), which is called by both `depositETH()` (L87) and `depositAsset()` (L111).

Additionally, `LRTOracle._getTotalEthInProtocol()` (L336–348) calls `getTotalAssetDeposits()` for **every supported asset** in a loop, and `_getTotalEthInProtocol()` is called by the public `updateRSETHPrice()` (L87–89).

The effective gas cost scales as:
> `supportedAssets.length × nodeDelegatorQueue.length × queuedWithdrawals.length × strategies.length`

All four dimensions are configurable: `maxNodeDelegatorLimit` is raiseable by admin via `updateMaxNodeDelegatorLimit()` (L290–297); `maxUncompletedWithdrawalCount` is set in `LRTUnstakingVault`; strategies per withdrawal are operator-supplied in `initiateUnstaking()`. No on-chain guard caps the product of these dimensions.

## Impact Explanation
When the product of the four loop dimensions grows large enough to exhaust the block gas limit, every call to `depositETH()`, `depositAsset()`, and `updateRSETHPrice()` reverts. This constitutes **temporary (potentially permanent) freezing of the deposit path** and a stale rsETH price. Both "Unbounded gas consumption" (Medium) and "Temporary freezing of funds" (Medium) are concrete, in-scope impacts. Existing deposited funds are not directly stolen, but the protocol becomes inoperable until queued withdrawals are completed to reduce state.

## Likelihood Explanation
The preconditions are normal protocol operations: admin deploys multiple NDCs (up to `maxNodeDelegatorLimit`), and operators legitimately call `initiateUnstaking()` with multiple strategies across multiple NDCs. No malicious intent is required. As the protocol scales organically, the combined loop depth grows without any single action crossing a clear safety threshold. The trigger itself — calling `depositETH()`, `depositAsset()`, or `updateRSETHPrice()` — requires no privileges whatsoever.

## Recommendation
1. **Decouple accounting from the hot path**: Store `assetUnstaking` as a cached state variable updated lazily on `initiateUnstaking()` and `completeUnstaking()` rather than recomputing it on every read.
2. **Remove `getAssetUnstaking()` from `getAssetDistributionData()`**: Return it only in off-chain view calls, not in the deposit/price-update path.
3. **Cap strategies per withdrawal** in `initiateUnstaking()` to bound the inner loop.
4. **Cap `maxUncompletedWithdrawalCount`** at a value that, combined with `maxNodeDelegatorLimit` and `supportedAssets.length`, provably stays within safe gas bounds.

## Proof of Concept
1. Admin raises `maxNodeDelegatorLimit` to 10 and deploys 10 NDCs.
2. Operators call `initiateUnstaking()` on each NDC with 5 strategies, repeated up to `maxUncompletedWithdrawalCount` times (e.g., 50 pending withdrawals per NDC).
3. Protocol supports 5 LST assets.
4. Any user calls `depositETH()`. The call chain reaches `getAssetUnstaking()` for each (asset × NDC) pair: `5 × 10 × 50 × 5 = 12,500` inner iterations, each involving external storage reads from EigenLayer. The transaction reverts with out-of-gas.
5. `updateRSETHPrice()` similarly reverts, freezing the rsETH price.

**Foundry fork test plan**: Fork mainnet, deploy 10 NDCs, queue 50 withdrawals with 5 strategies each on each NDC, then call `depositETH{value: 1 ether}()` and assert it reverts with out-of-gas, or measure `gasleft()` delta to confirm it exceeds the block gas limit.
