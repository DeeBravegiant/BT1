### Title
Unbounded Pool HTTP Response Body Read Enables Farmer Process DoS - (File: chia/farmer/farmer.py)

### Summary
The Chia farmer reads HTTP response bodies from pool servers using `await resp.text()` with no size limit across all pool protocol endpoints. A malicious pool operator can return an arbitrarily large response body, causing the farmer process to exhaust memory (OOM crash) or block the async event loop, resulting in a long-lived inability to process valid blocks and earn farming rewards.

### Finding Description
In `chia/farmer/farmer.py`, every pool HTTP handler reads the full response body into memory without any size guard:

- `_pool_get_pool_info` — `json.loads(await resp.text())` [1](#0-0) 
- `_pool_get_farmer` — `json.loads(await resp.text())` [2](#0-1) 
- `_pool_post_farmer` — `json.loads(await resp.text())` [3](#0-2) 
- `_pool_put_farmer` — `json.loads(await resp.text())` [4](#0-3) 

In `chia/farmer/farmer_api.py`, the POST `/partial` response (submitted on every new proof of space) is also read without a size limit: [5](#0-4) 

A grep for `read_limit`, `max_size`, `content_length`, or `read_bufsize` in `chia/farmer/*.py` returns zero matches, confirming no size guard exists anywhere in the farmer's pool HTTP stack. [6](#0-5) 

None of the `aiohttp.ClientSession()` constructors in these methods pass a `read_limit` or `read_bufsize` parameter. [7](#0-6) 

The `update_pool_state` loop calls `_pool_get_pool_info` and `_pool_get_farmer` on a recurring timer, and `new_proof_of_space` in `farmer_api.py` calls the `/partial` endpoint on every eligible proof. [8](#0-7) 

### Impact Explanation
A malicious pool operator can respond to any of these requests with a multi-gigabyte body. Because `aiohttp.ClientResponse.text()` buffers the entire body before returning, the farmer process will:

1. **OOM crash** — loading a sufficiently large body exhausts the process heap, killing the farmer. The farmer can no longer process proofs of space, submit partials, or farm blocks until manually restarted.
2. **Event-loop stall** — even before OOM, the synchronous `json.loads(...)` call on a very large string blocks the single-threaded asyncio event loop, preventing the farmer from responding to harvester messages or signage-point notifications for the duration of the parse.

Because `update_pool_state` is called on a fixed interval and `/partial` is called on every proof, a pool that consistently returns oversized bodies will cause repeated crashes or stalls, constituting a **long-lived inability for an honest farmer to process valid blocks under normal network assumptions**.

### Likelihood Explanation
Any pool operator the farmer is configured to use can trigger this with a trivial HTTP server change — no cryptographic material, no Chia protocol privilege, and no access to the farmer's keys is required. The farmer initiates the connection; the pool only needs to return a large body. The attack fires automatically on the next `update_pool_state` tick or the next proof of space.

### Recommendation
Apply a response-body size cap before reading. For example:

```python
MAX_POOL_RESPONSE_BYTES = 65_536  # 64 KiB

raw = await resp.content.read(MAX_POOL_RESPONSE_BYTES + 1)
if len(raw) > MAX_POOL_RESPONSE_BYTES:
    raise ValueError("Pool response body too large")
response = json.loads(raw)
```

Alternatively, pass `read_bufsize` / enforce `Content-Length` checks, or use `aiohttp.ClientSession(read_limit=MAX_POOL_RESPONSE_BYTES)` (available in aiohttp ≥ 3.8).

### Proof of Concept
A malicious pool server at `pool_url` returns the following to any request:

```
HTTP/1.1 200 OK
Content-Type: application/json

{"x": "<1 GB of repeated data>"}
```

The farmer's `_pool_get_pool_info` (or any other handler) calls `await resp.text()`, which buffers the entire gigabyte into a Python `str`. On a typical farmer machine this triggers the OOM killer, terminating the farmer process. Because `update_pool_state` is called every `UPDATE_POOL_INFO_INTERVAL` seconds, the farmer will crash again on every restart as long as it remains configured to use that pool URL, producing a sustained farming outage.         try:
            async with aiohttp.ClientSession(trust_env=True) as session:
                url = f"{pool_config.pool_url}/pool_info"
                async with session.get(url, ssl=ssl_context_for_root(get_mozilla_ca_crt(), log=self.log)) as resp:
                    if resp.ok:
                        response: dict[str, Any] = json.loads(await resp.text())
                        self.log.info(f"GET /pool_info response: {response}")
                        if "error_code" in response:
                            self.handle_failed_pool_response(
                                pool_config.p2_singleton_puzzle_hash,
                                f"Error in GET /pool_info {pool_config.pool_url}, {response}",
                            )
                            return None
                        try:
                            pool_info = GetPoolInfoResponse.from_json_dict(response)
                        except Exception as e:
                            self.handle_failed_pool_response(
                                pool_config.p2_singleton_puzzle_hash,
                                f"Invalid GET /pool_info response {pool_config.pool_url}, {e}",
                            )
                            return None
                        new_pool_url: str | None = None
                        response_url_str = f"{resp.url}"
                        if (
                            response_url_str != url
                            and len(resp.history) > 0
                            and all(r.status in {301, 308} for r in resp.history)
                        ):
                            new_pool_url = response_url_str.replace("/pool_info", "")

                        return GetPoolInfoResult(pool_info=pool_info, new_pool_url=new_pool_url)
                    else:
                        self.handle_failed_pool_response(
                            pool_config.p2_singleton_puzzle_hash,
                            f"Error in GET /pool_info {pool_config.pool_url}, {resp.status}",
                        )

            self.handle_failed_pool_response(
                pool_config.p2_singleton_puzzle_hash, f"Exception in GET /pool_info {pool_config.pool_url}, {e}"
            )

        return None
