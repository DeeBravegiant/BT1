Audit Report

## Title
Stale `rsETHPrice` Exploitable via Public `updateRSETHPrice()` to Steal Unclaimed Yield from rsETH Holders - (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTOracle.rsETHPrice` is a stored state variable that lags behind real-time Chainlink LST prices. Because `updateRSETHPrice()` is publicly callable and `ChainlinkPriceOracle.getAssetPrice()` performs no staleness validation, an attacker can deposit at the stale (lower) rsETH price, trigger a price update, and withdraw at the new (higher) price ã extracting accrued yield that belongs to existing rsETH holders. The attack is fully permissionless and amplifiable with a flash loan.

## Finding Description

**Root cause 1 ã `ChainlinkPriceOracle.getAssetPrice()` discards all staleness fields:**

`contracts/oracles/ChainlinkPriceOracle.sol` L52 calls `latestRoundData()` but silently drops `updatedAt` and `answeredInRound`:
```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
``` [1](#0-0) 

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` (the L2 path) explicitly checks both:
```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
``` [2](#0-1) 

**Root cause 2 ã `updateRSETHPrice()` is permissionlessly callable:**

Any address can trigger a price update at any time: [3](#0-2) 

The `pricePercentageLimit` guard at lines 252ã266 only blocks updates that exceed the configured threshold ã it does not prevent updates within the limit, and is entirely absent when `pricePercentageLimit == 0`. [4](#0-3) 

**Root cause 3 ã `getRsETHAmountToMint()` mixes a live Chainlink price with the stale stored price:**

At deposit time, `lrtOracle.getAssetPrice(asset)` fetches the current Chainlink price while `lrtOracle.rsETHPrice()` reads the stale stored state variable. When LST prices have risen since the last `updateRSETHPrice()` call, the denominator is stale-low, so the depositor receives more rsETH than the protocol's current NAV justifies: [5](#0-4) 

**Root cause 4 ã `getExpectedAssetAmount()` reads the current `rsETHPrice` at execution time:**

Both `instantWithdrawal` and `initiateWithdrawal` price the withdrawal using the live `rsETHPrice` at the moment of the call: [6](#0-5) 

After the attacker triggers `updateRSETHPrice()`, the stored price rises to the true NAV, and the withdrawal is priced at the new higher rate.

**Attack flow (single transaction when `instantWithdrawal` is enabled):**

1. Observe `rsETHPrice` is stale-low (Chainlink LST prices have risen since last update).
2. Call `LRTDepositPool.depositETH()` ã receives `amount / rsETHPrice_stale` rsETH (more than fair value).
3. Call `LRTOracle.updateRSETHPrice()` ã stored price rises to `rsETHPrice_new`.
4. Call `LRTWithdrawalManager.instantWithdrawal()` ã receives `rsETHAmount û rsETHPrice_new / assetPrice` ETH.

**Attack flow (no `instantWithdrawal` required):**

Steps 1ã3 are identical. In step 4, the attacker calls `initiateWithdrawal()` immediately after the price update; `expectedAssetAmount` is locked in at the new higher price and paid out in full after the withdrawal delay. [7](#0-6) 

**Existing guards are insufficient:**

- `pricePercentageLimit`: only blocks price increases above the configured threshold; attacks within the threshold window (e.g., a 0.5% gap when the limit is 1%) proceed unimpeded. When `pricePercentageLimit == 0` there is no guard at all.
- `onlyInstantWithdrawalAllowed`: only gates the instant path; the `initiateWithdrawal` path is always open.
- `nonReentrant` / `whenNotPaused`: do not prevent the multi-step sequence across separate transactions.

## Impact Explanation

**High ã Theft of unclaimed yield.** The attacker extracts ETH representing accrued yield that belongs to existing rsETH holders. The profit is proportional to the price gap: a 1% stale gap on a 1,000 ETH deposit yields ã 9.52 ETH of risk-free profit taken directly from the pool's NAV. With a flash loan the attack scales linearly (100,000 ETH ã ã 952 ETH profit per execution). This matches the allowed impact "High ã Theft of unclaimed yield."

## Likelihood Explanation

**Medium.** `rsETHPrice` is naturally stale between keeper calls. Chainlink LST/ETH feeds update on a 24-hour heartbeat or a 0.5% deviation threshold; any period where feeds have moved but `rsETHPrice` has not been refreshed opens the window. No special permissions are required. The attack is fully permissionless, repeatable on every staleness window, and amplifiable with a flash loan.

## Recommendation

1. **Add staleness validation to `ChainlinkPriceOracle.getAssetPrice()`** ã check `answeredInRound >= roundId` and `updatedAt > block.timestamp - maxStaleness`, mirroring `ChainlinkOracleForRSETHPoolCollateral.getRate()`.
2. **Restrict `updateRSETHPrice()` to trusted keepers** ã add a role check (e.g., `onlyLRTManager` or a dedicated `KEEPER_ROLE`) so an attacker cannot trigger a price update on demand.
3. **Snapshot `rsETHPrice` at deposit time** ã record the `rsETHPrice` used to mint rsETH and cap the withdrawal payout to that snapshot, preventing the attacker from benefiting from a price update they triggered.
