Audit Report

## Title
ETH Withdrawal Permanently Frozen When Recipient Contract Rejects ETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._processWithdrawalCompletion` uses a push-based ETH transfer via `_transferAsset`. If the withdrawal initiator is a contract without a `receive()` function, `completeWithdrawal` will always revert. Because rsETH is burned in a prior `unlockQueue` transaction, the ETH is permanently frozen in `LRTWithdrawalManager` with no recovery path.

## Finding Description
The withdrawal lifecycle spans two separate transactions:

**Transaction 1 — `unlockQueue` (operator-called):** Burns the user's rsETH held by the contract and redeems ETH from `LRTUnstakingVault` into `LRTWithdrawalManager`. This burn is permanent and cannot be rolled back. [1](#0-0) 

**Transaction 2 — `completeWithdrawal` (user-called):** Calls `_processWithdrawalCompletion(asset, msg.sender, referralId)`, which pops the nonce, deletes the request, decrements `unlockedWithdrawalsCount`, and then calls `_transferAsset`. [2](#0-1) [3](#0-2) 

`_transferAsset` uses a push-based call:
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
``` [4](#0-3) 

If `to` is a contract without a `receive()` function, `sent` is `false` and the entire `_processWithdrawalCompletion` call reverts. Solidity atomicity rolls back the nonce pop, request deletion, and count decrement — so the withdrawal request persists. However, the rsETH burn from `unlockQueue` (Transaction 1) is in a separate, already-finalized transaction and is **not** rolled back.

The result: the user's rsETH is permanently burned, the ETH is locked in `LRTWithdrawalManager`, and every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` will revert identically because both route through the same `_processWithdrawalCompletion(asset, user, ...)` pushing ETH to the same non-receiving address. [5](#0-4) 

`LRTWithdrawalManager` does not inherit `Recoverable` and has no `recoverETH` or redirect function. The only potential escape valve, `sweepRemainingAssets`, is permanently blocked because `unlockedWithdrawalsCount[asset]` remains non-zero (the stuck request is never completed): [6](#0-5) 

The contract class declaration confirms no `Recoverable` inheritance: [7](#0-6) 

## Impact Explanation
**Critical — Permanent freezing of funds.** The user loses both their rsETH (burned in `unlockQueue`) and the corresponding ETH (frozen in `LRTWithdrawalManager`). There is no admin function to redirect the ETH to an alternate address, and no pull-based fallback. The stuck `unlockedWithdrawalsCount` also permanently blocks `sweepRemainingAssets` for the affected asset.

## Likelihood Explanation
**Medium.** Smart contract wallets (Gnosis Safe configurations with custom modules), protocol-owned treasury addresses, DeFi vaults, and account-abstraction wallets that do not implement a `receive()` function are common in the ecosystem. Any such address that calls `initiateWithdrawal` will have its funds permanently frozen once the operator runs `unlockQueue`. The condition is self-contained within the protocol — no external oracle, governance, or attacker action is required beyond the user's own contract lacking ETH receipt capability.

## Recommendation
Replace the push-based ETH transfer in `_processWithdrawalCompletion` with a pull-based pattern. Record owed ETH in a mapping during completion processing, and expose a separate `claimETH(address recipient)` function that allows the user to pull funds to any address they control:

```solidity
mapping(address => uint256) public pendingETHWithdrawals;

// In _processWithdrawalCompletion, replace _transferAsset for ETH:
if (asset == LRTConstants.ETH_TOKEN) {
    pendingETHWithdrawals[user] += request.expectedAssetAmount;
} else {
    _transferAsset(asset, user, request.expectedAssetAmount);
}

// New claim function:
function claimETH(address recipient) external nonReentrant {
    uint256 amount = pendingETHWithdrawals[msg.sender];
    if (amount == 0) revert NothingToClaim();
    pendingETHWithdrawals[msg.sender] = 0;
    (bool sent,) = payable(recipient).call{ value: amount }("");
    if (!sent) revert EthTransferFailed();
}
```

This decouples accounting from delivery, ensuring a non-receiving address cannot cause permanent fund loss.

## Proof of Concept

1. Alice deploys `VaultContract` (no `receive()` function) and calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, ...)` from it. rsETH is transferred to `LRTWithdrawalManager`. [8](#0-7) 

2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. Alice's rsETH is burned from the contract; ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`. This transaction finalizes — the burn is irreversible. [1](#0-0) 

3. Alice calls `completeWithdrawal(ETH_TOKEN, ...)` from `VaultContract`. `_processWithdrawalCompletion` reaches `_transferAsset(ETH_TOKEN, VaultContract, amount)`. [9](#0-8) 

4. `payable(VaultContract).call{ value: amount }("")` returns `false`. `EthTransferFailed` is thrown; the entire transaction reverts. The withdrawal request is restored in storage, but the rsETH burn from step 2 is not. [10](#0-9) 

5. Every subsequent call — whether Alice calls `completeWithdrawal` or an operator calls `completeWithdrawalForUser` — reverts identically. Alice's rsETH is permanently burned. The ETH remains in `LRTWithdrawalManager` forever. `unlockedWithdrawalsCount[ETH_TOKEN]` stays elevated, also blocking `sweepRemainingAssets`.

**Foundry test sketch:**
```solidity
contract NoReceive {} // no receive() or fallback

function test_ethFrozenForNonReceivingContract() public {
    NoReceive vault = new NoReceive();
    // Fund vault with rsETH, approve LRTWithdrawalManager
    vm.prank(address(vault));
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
    
    // Operator unlocks — rsETH burned here (permanent)
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, ...);
    
    uint256 rsETHAfterBurn = rsETH.balanceOf(address(withdrawalManager));
    assertEq(rsETHAfterBurn, 0); // burned
    
    // completeWithdrawal always reverts
    vm.prank(address(vault));
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);
    withdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    
    // ETH stuck in contract
    assertGt(address(withdrawalManager).balance, 0);
}
```

    function _processWithdrawalCompletion(address asset, address user, string calldata referralId) internal {
        if (userAssociatedNonces[asset][user].empty()) {
            revert NoWithdrawalRequests(user, asset);
        }

        // Retrieve and remove the oldest withdrawal request for the user.
        uint256 usersFirstWithdrawalRequestNonce = userAssociatedNonces[asset][user].popFront();
        // Ensure the request is already unlocked.
        if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();

        bytes32 requestId = getRequestId(asset, usersFirstWithdrawalRequestNonce);
        WithdrawalRequest memory request = withdrawalRequests[requestId];

        delete withdrawalRequests[requestId];

        // Check that the withdrawal delay has passed since the request's initiation.
        if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) revert WithdrawalDelayNotPassed();

        unlockedWithdrawalsCount[asset]--;

        // If Aave integration is enabled and asset is ETH, withdraw from Aave if needed
        if (isAaveIntegrationEnabled && asset == LRTConstants.ETH_TOKEN) {
            uint256 contractBalance = address(this).balance;
            if (contractBalance < request.expectedAssetAmount) {
                uint256 amountNeeded = request.expectedAssetAmount - contractBalance;
                _withdrawFromAave(amountNeeded);

                // Verify we have sufficient balance after withdrawal
                uint256 balanceAfter = address(this).balance;
                if (balanceAfter < request.expectedAssetAmount) {
                    revert InsufficientLiquidityForWithdrawal();
                }
            }
        }

        _transferAsset(asset, user, request.expectedAssetAmount);

        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(user, asset, request.rsETHUnstaked, request.expectedAssetAmount);
    }
```
