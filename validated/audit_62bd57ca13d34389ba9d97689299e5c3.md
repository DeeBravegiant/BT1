Audit Report

## Title
`getAvailableAssetAmount` Counts All Protocol Assets While `unlockQueue` Can Only Service `LRTUnstakingVault` Balance, Causing Temporary Freezing of User rsETH — (`contracts/LRTWithdrawalManager.sol`)

## Summary

`initiateWithdrawal` gates user commitment using `getAvailableAssetAmount`, which aggregates assets across all protocol locations via `getTotalAssetDeposits`. However, `unlockQueue` can only service withdrawals from the `LRTUnstakingVault` balance. When assets reside primarily in EigenLayer (the protocol's normal operating state), users can commit rsETH against assets that are unreachable by `unlockQueue`, freezing their rsETH in `LRTWithdrawalManager` until an operator completes the full multi-step EigenLayer unstaking cycle.

## Finding Description

**Commitment gate — uses total protocol assets:**

`initiateWithdrawal` transfers the user's rsETH to the contract and calls `getAvailableAssetAmount` to gate the commitment: [1](#0-0) 

`getAvailableAssetAmount` delegates to `getTotalAssetDeposits`: [2](#0-1) 

`getTotalAssetDeposits` sums every protocol location — DepositPool, NDCs, EigenLayer staked, EigenLayer unstaking, Converter, and UnstakingVault: [3](#0-2) 

**Unlock gate — uses only `LRTUnstakingVault` balance:**

`_createUnlockParams` sets `totalAvailableAssets` exclusively to `unstakingVault.balanceOf(asset)`: [4](#0-3) 

`_unlockWithdrawalRequests` hard-stops if the vault balance is insufficient to cover the next payout: [5](#0-4) 

**No cancellation path exists:**

A search for `cancelWithdrawal` returns no matches anywhere in the codebase. Once rsETH is transferred into `LRTWithdrawalManager` and `assetsCommitted` is incremented, there is no mechanism for the user to recover their rsETH until `unlockQueue` succeeds. [6](#0-5) 

## Impact Explanation

A user's rsETH is irrecoverably locked in `LRTWithdrawalManager` from the moment `initiateWithdrawal` succeeds until the operator completes the full EigenLayer unstaking cycle and calls `unlockQueue`. During this window the user cannot sell, transfer, or reclaim their rsETH. This is a concrete **Medium — Temporary freezing of funds** impact.

## Likelihood Explanation

The protocol's primary purpose is restaking into EigenLayer. In normal operation the vast majority of assets reside in EigenLayer strategies, not in `LRTUnstakingVault`. Any user who calls `initiateWithdrawal` when the vault holds less than their `expectedAssetAmount` will have their rsETH frozen. This is a routine, predictable protocol state requiring no attacker — any ordinary withdrawer triggers it. Likelihood is **medium-high**.

## Recommendation

Align the commitment gate with the execution gate. The simplest fix is to restrict `getAvailableAssetAmount` to return only the `LRTUnstakingVault` balance minus already-committed assets:

```solidity
function getAvailableAssetAmount(address asset) public view override returns (uint256) {
    ILRTUnstakingVault unstakingVault =
        ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
    uint256 vaultBalance = unstakingVault.balanceOf(asset);
    return vaultBalance > assetsCommitted[asset] ? vaultBalance - assetsCommitted[asset] : 0;
}
```

This ensures users can only commit against assets that `unlockQueue` can immediately service. Alternatively, extend `unlockQueue` to draw from DepositPool and NDC balances, but this is significantly more complex and introduces additional risk.

## Proof of Concept

1. Protocol state: 900 ETH staked in EigenLayer, 100 ETH in `LRTUnstakingVault`. `assetsCommitted[ETH] = 0`.
2. `getTotalAssetDeposits(ETH)` returns 1000 ETH. `getAvailableAssetAmount(ETH)` returns 1000 ETH.
3. User calls `initiateWithdrawal(ETH, X_rsETH)` where `expectedAssetAmount = 1000 ETH`. Check at L170 passes (`1000 <= 1000`). User's rsETH is transferred to `LRTWithdrawalManager`. `assetsCommitted[ETH] = 1000`.
4. Operator calls `unlockQueue(ETH, ...)`. `_createUnlockParams` sets `totalAvailableAssets = unstakingVault.balanceOf(ETH) = 100 ETH`.
5. Inside `_unlockWithdrawalRequests`: `payoutAmount ≈ 1000 ETH`, `availableAssetAmount = 100 ETH`. Line 800 fires: `break`. Zero withdrawals are unlocked.
6. User's rsETH remains locked in `LRTWithdrawalManager` with no recourse — `cancelWithdrawal` does not exist.
7. Operator must call `initiateUnstaking` on a NodeDelegator, wait ~7 days for EigenLayer's withdrawal delay, call `completeUnstaking` to move assets to `LRTUnstakingVault`, then call `unlockQueue` again — only then is the withdrawal unlocked.
