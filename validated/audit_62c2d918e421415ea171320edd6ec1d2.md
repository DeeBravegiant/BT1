Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Calculations Allows Theft of Unclaimed Yield - (`contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary

`LRTOracle.rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is explicitly called. Neither `depositETH()` nor `depositAsset()` refreshes this price before computing the rsETH mint amount. When staking rewards accrue and the stored price is stale (lower than true), any depositor receives more rsETH than deserved, diluting existing holders and stealing their unclaimed yield.

## Finding Description

`rsETHPrice` is a plain storage variable, not computed on-the-fly: [1](#0-0) 

`updateRSETHPrice()` is public and permissionless but is never called atomically with any deposit: [2](#0-1) 

`depositETH()` calls `_beforeDeposit()` → `getRsETHAmountToMint()`, which divides by the stored stale price: [3](#0-2) 

Neither `depositETH()` nor `depositAsset()` calls `updateRSETHPrice()` before this computation: [4](#0-3) [5](#0-4) 

The `minRSETHAmountExpected` slippage guard only protects the depositor from receiving *fewer* rsETH than expected — it does not prevent receiving *more* than deserved and provides no protection against this attack.

The `pricePercentageLimit` guard inside `_updateRsETHPrice()` reverts non-manager callers if the price increase exceeds the threshold: [6](#0-5) 

This guard does not protect against the stale-price deposit attack. The attacker never needs to call `updateRSETHPrice()` — they simply deposit *before* the keeper does. The guard is irrelevant to the exploit path.

**Note on withdrawals:** The report's withdrawal claim is directionally incorrect. With a stale lower `rsETHPrice`, `initiateWithdrawal()` computes `amount * staleLowerPrice / assetPrice`, which gives the withdrawer *less* underlying than deserved — it harms withdrawers, not benefits them. The deposit vector alone is sufficient for the stated impact. [7](#0-6) 

## Impact Explanation

**High — Theft of unclaimed yield.**

When staking rewards accrue (EigenLayer pod balances increase, stETH rebases, etc.), `totalETHInProtocol` rises but `rsETHPrice` remains at its last stored value. A depositor in this window receives:

```
rsethAmountToMint = (depositAmount × assetPrice) / staleLowerPrice
```

This is more rsETH than the true exchange rate warrants. When `updateRSETHPrice()` is eventually called, the inflated rsETH supply dilutes the price, and existing holders receive less yield than they earned. The excess rsETH held by the attacker represents a direct, concrete transfer of accrued yield from existing holders to the attacker. This matches the allowed impact: **Theft of unclaimed yield**.

## Likelihood Explanation

**High.** `updateRSETHPrice()` is called by off-chain keepers on a schedule, not atomically with every deposit. A stale-price window always exists between reward accrual events (e.g., EigenLayer pod balance increases, stETH rebases) and the next keeper call. No special privileges are required — any depositor can exploit this by monitoring on-chain reward accrual and timing their deposit to precede the keeper's `updateRSETHPrice()` call. The attack is repeatable every reward cycle.

## Recommendation

Call `updateRSETHPrice()` (or an internal equivalent) at the start of `depositETH()` and `depositAsset()` before any mint amount is computed:

```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
    ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    emit ETHDeposit(msg.sender, msg.value, rsethAmountToMint, referralId);
}
```

Apply the same pattern to `depositAsset()`. If `pricePercentageLimit` would cause `updateRSETHPrice()` to revert for large reward accruals, consider using an internal price-refresh path that does not revert on threshold breaches but still updates the stored price, or pause deposits when the price is stale beyond a configurable tolerance.

## Proof of Concept

1. Protocol state: 1000 ETH in assets, 1000 rsETH supply → stored `rsETHPrice = 1.0 ETH/rsETH`.
2. 100 ETH in staking rewards accrue. True price = `1100/1000 = 1.1`. Stored price remains `1.0`.
3. Attacker calls `depositETH{value: 100 ETH}()` before the keeper calls `updateRSETHPrice()`.
4. `getRsETHAmountToMint` computes: `100e18 × 1e18 / 1.0e18 = 100 rsETH` minted. Correct amount at true price: `≈ 90.9 rsETH`. Attacker received ~9.1 extra rsETH.
5. Keeper calls `updateRSETHPrice()`. New supply = 1100 rsETH, total ETH = 1200 ETH. New price = `1200/1100 ≈ 1.0909`.
6. Attacker's 100 rsETH is worth `≈ 109 ETH` — a profit of ~9 ETH on a 100 ETH deposit.
7. Original 1000 rsETH holders now hold `1000/1100 × 1200 ≈ 1090.9 ETH` instead of the 1100 ETH they earned. They lost ~9.1 ETH of yield.

**Foundry fork test outline:**
- Fork mainnet, deploy/configure protocol.
- Simulate reward accrual by directly increasing EigenLayer pod balance or stETH rebase.
- Assert `lrtOracle.rsETHPrice()` has not yet updated.
- Call `depositETH` as attacker; record rsETH minted.
- Call `updateRSETHPrice()` as keeper.
- Assert attacker's rsETH value exceeds deposit amount; assert original holders' ETH-equivalent decreased relative to pre-reward state.
