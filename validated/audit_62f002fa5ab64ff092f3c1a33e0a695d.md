Audit Report

## Title
Stale `rsETHPrice` Allows Over-Minting of rsETH When `pricePercentageLimit` Blocks Public Price Updates â€” (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary
`LRTOracle.rsETHPrice` is a stored state variable used as the denominator in rsETH minting. When the true rsETH price rises beyond `pricePercentageLimit` above `highestRsethPrice`, public calls to `updateRSETHPrice()` revert with `PriceAboveDailyThreshold` before the stored price is updated. During the resulting stale-price window, any depositor calling `depositETH` or `depositAsset` receives more rsETH than their deposit is worth, diluting existing holders' accumulated yield.

## Finding Description
`LRTDepositPool.getRsETHAmountToMint` computes rsETH to mint as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.rsETHPrice()` is a public getter for the stored state variable `rsETHPrice` (not a live computation). It is only updated when `_updateRsETHPrice()` completes successfully.

Inside `_updateRsETHPrice()`, when `newRsETHPrice > highestRsethPrice` and the difference exceeds `pricePercentageLimit.mulWad(highestRsethPrice)`, the function reverts for any non-manager caller (`PriceAboveDailyThreshold`). This revert occurs at line 264, before the assignment `rsETHPrice = newRsETHPrice` at line 313. The stored price is therefore never updated.

The deposit entry points `depositETH` and `depositAsset` both call `_beforeDeposit`, which calls `getRsETHAmountToMint` using the stale stored price. The only guard in `_beforeDeposit` is `minRSETHAmountExpected`, which protects the depositor from receiving *too little* rsETH â€” it does not prevent over-minting. No freshness check, no inline price recomputation, and no revert-on-stale-price logic exists in the deposit path.

**Exploit flow:**
1. `pricePercentageLimit = 1e16` (1%); `highestRsethPrice = 1.00e18`; `rsETHPrice = 1.00e18`.
2. Staking rewards accumulate; true price becomes `1.02e18` (2% increase).
3. Any user calls `updateRSETHPrice()` â†’ reverts (`0.02e18 > 0.01e18`).
4. Stored `rsETHPrice` remains `1.00e18`.
5. Attacker calls `depositETH(0, "")` with 100 ETH.
6. `getRsETHAmountToMint` returns `100e18 * 1e18 / 1.00e18 = 100 rsETH` instead of the correct `â‰ˆ98.04 rsETH`.
7. Attacker receives â‰ˆ1.96 excess rsETH at no cost.
8. Manager calls `updateRSETHPriceAsManager()` â†’ price updates to `â‰ˆ1.019e18` (diluted by the excess supply).
9. Attacker's rsETH is redeemable for more ETH than deposited; existing holders' rsETH is worth less.

## Impact Explanation
The excess rsETH minted during the stale-price window increases total supply without a corresponding increase in TVL. When the manager subsequently updates the price, `newRsETHPrice = totalETHInProtocol / rsethSupply` is lower than it would have been absent the over-minting. Every existing rsETH holder's token is worth less â€” their accumulated staking yield is transferred to the attacker. This is a concrete, quantifiable **theft of unclaimed yield** (High severity).

## Likelihood Explanation
`pricePercentageLimit` is a live protocol parameter intended to be set conservatively (e.g., 1%). Staking rewards accumulate continuously; if `updateRSETHPrice()` is not called at least once per reward cycle, the true price can exceed the limit before the next public call. The exploit requires no special permissions, no front-running, and no oracle manipulation â€” only a deposit transaction during the stale window. The window persists until a manager manually calls `updateRSETHPriceAsManager()`. Any number of depositors can exploit it simultaneously, amplifying the dilution.

## Recommendation
1. **Compute price on-the-fly in `getRsETHAmountToMint`**: Replace `lrtOracle.rsETHPrice()` with an inline `totalETHInProtocol / rsethSupply` calculation so the minting rate always reflects current state.
2. **Atomic price update before deposit**: Call `_updateRsETHPrice()` (or a view equivalent) inside `_beforeDeposit` and revert if the price cannot be updated due to the threshold check.
3. **Restrict `pricePercentageLimit` to downward moves only**: The limit is a safety guard against manipulation. Blocking upward updates creates the stale-price window. Apply the limit only to price decreases (which already trigger a pause) and allow unrestricted upward updates.
