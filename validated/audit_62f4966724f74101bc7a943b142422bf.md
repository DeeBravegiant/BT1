### Title
Deposit Refund Misdirection in Meta Transactions Allows Unprivileged User to Drain Relayer Balance - (File: runtime/runtime/src/actions.rs)

### Summary
When a `DelegateAction` inner receipt fails on the receiver's shard, the deposit refund is sent to the delegate sender (Alice), not the relayer who actually paid the deposit. An unprivileged user can exploit this by crafting a meta transaction with a large attached deposit that is designed to fail, causing the relayer to permanently lose the deposit while the user gains it.

### Finding Description
In `apply_delegate_action`, the inner receipt spawned from a `DelegateAction` is constructed with `predecessor_id: sender_id.clone()` — i.e., Alice's account ID — not the relayer's. [1](#0-0) 

When this inner receipt fails on the receiver's shard, `refund_unspent_gas_and_deposits` computes the deposit refund and routes it via `receipt.balance_refund_receiver()`: [2](#0-1) 

`balance_refund_receiver()` returns `refund_to` if set, otherwise falls back to `predecessor_id`: [3](#0-2) 

The inner receipt uses `ReceiptEnum::Action` (not `ActionV2`), so `refund_to` is always `None`: [4](#0-3) 

Therefore the deposit refund unconditionally goes to `predecessor_id` = Alice. The code comment at the construction site explicitly acknowledges this asymmetry: [5](#0-4) 

The architecture documentation confirms the financial attack surface: [6](#0-5) 

### Impact Explanation
The corrupted protocol value is the **balance** of both parties: the relayer's `amount` field in the state trie is reduced by the full deposit, while Alice's `amount` is increased by the same deposit via the refund receipt. This is a direct, irreversible balance transfer from the relayer to Alice triggered by a deliberately failing meta transaction. The relayer paid the deposit at transaction submission time; there is no recovery path once the inner receipt fails.

### Likelihood Explanation
Any unprivileged user who can submit a `SignedDelegateAction` to a relayer can trigger this. The attack requires:
1. Alice attaches a large deposit to an inner `FunctionCall` action targeting a contract that will reject the call (e.g., a method that panics, a non-existent method, or a contract whose state Alice can manipulate to cause failure after the relayer's pre-flight check).
2. The relayer, acting correctly, wraps and submits the transaction.
3. The inner receipt fails on the receiver's shard.
4. The deposit refund flows to Alice.

The relayer cannot prevent this through pre-flight checks alone because Alice can front-run the submission by changing the receiver contract's state between the relayer's check and the block in which the receipt executes. This is the exact front-running/sandwiching pattern described in the external report.

### Recommendation
The inner receipt created in `apply_delegate_action` should use `ReceiptEnum::ActionV2` with `refund_to` set to the relayer's account ID (`action_receipt.signer_id()`), so that deposit refunds from failed inner receipts are returned to the party that paid them. Alternatively, the protocol specification should be updated to explicitly define this as a protocol-level invariant violation and provide a migration path.

### Proof of Concept
1. Alice creates a `SignedDelegateAction` with `receiver_id = some_contract`, inner action = `FunctionCall { method_name: "transfer", deposit: 10 NEAR, ... }`.
2. Alice ensures `some_contract` will reject the call (e.g., by withdrawing her FT balance from the contract just before the relayer submits, or by targeting a method that panics under a specific state Alice controls).
3. Alice submits the `SignedDelegateAction` to the relayer.
4. The relayer pre-flight checks pass (state looks valid at check time), wraps the action in a transaction, and submits it. The relayer's balance is debited 10 NEAR for the deposit.
5. On Alice's shard, `apply_delegate_action` creates the inner receipt with `predecessor_id = Alice`.
6. On `some_contract`'s shard, the inner receipt fails. `refund_unspent_gas_and_deposits` calls `Receipt::new_balance_refund(receipt.balance_refund_receiver(), 10 NEAR)` → `balance_refund_receiver()` returns `predecessor_id` = Alice.
7. Alice receives 10 NEAR. The relayer's 10 NEAR is permanently lost.
