Audit Report

## Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of Actual Caller, Allowing Allowlist Bypass — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

## Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards its first parameter (the actual caller forwarded by the pool) and gates access solely on the `owner` parameter (the LP share recipient). Because `MetricOmmPool.addLiquidity` imposes no `msg.sender == owner` requirement, any address not on the allowlist can call `addLiquidity` with an allowlisted `owner`, pass the extension check, pay tokens themselves via the callback, and credit LP shares to the allowlisted address — fully bypassing the pool admin's deposit restriction.

## Finding Description

`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` with no identity check: [1](#0-0) 

By contrast, `removeLiquidity` enforces `msg.sender == owner` at line 206: [2](#0-1) 

The pool forwards `(msg.sender, owner, ...)` to the extension hook. Inside `DepositAllowlistExtension.beforeAddLiquidity`, the first parameter — the actual caller — is unnamed and discarded; only `owner` is checked against the allowlist: [3](#0-2) 

Inside `LiquidityLib.addLiquidity`, the token-pull callback is invoked on `msg.sender` (the actual payer): [4](#0-3) 

While LP shares are credited to `owner` via the position key `keccak256(owner, salt, bin)`: [5](#0-4) 

The extension's own NatSpec states it "Gates `addLiquidity` by depositor address" — but the depositor is `msg.sender`, not `owner`. The gate is evaluated against the wrong identity.

## Impact Explanation

The `DepositAllowlistExtension` is the pool admin's mechanism to restrict which addresses may deposit (e.g., for permissioned or regulated pools). The bypass allows any unprivileged address to inject tokens into the pool and credit LP shares to an allowlisted address, violating the admin-configured access boundary. This matches the allowed impact gate: **admin-boundary break — factory/pool-admin role checks bypassed by an unprivileged path**. Secondary effects include the allowlisted victim receiving unwanted LP shares in a position they did not initiate, and the attacker being able to manipulate specific bin balances affecting proportional share redemption for other LPs in that bin.

## Likelihood Explanation

Medium. The precondition is that the pool has `DepositAllowlistExtension` registered as a `beforeAddLiquidity` hook and at least one allowlisted address is publicly known (observable from on-chain events emitted by `setAllowedToDeposit`). The attacker must hold enough tokens to fund the callback but gains nothing financially — the attack is a compliance bypass and griefing vector. Both preconditions are readily observable on-chain, making the attack straightforwardly repeatable.

## Recommendation

`DepositAllowlistExtension.beforeAddLiquidity` should check the actual caller (first parameter) rather than `owner`:

```solidity
function beforeAddLiquidity(address caller, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][caller]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Alternatively, `MetricOmmPool.addLiquidity` should enforce `msg.sender == owner` (mirroring `removeLiquidity`) when a restrictive extension is active, so that the payer and position owner are always the same address.

## Proof of Concept

1. Pool is deployed with `DepositAllowlistExtension` as a `beforeAddLiquidity` hook.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)`. Bob is not allowlisted.
3. Bob (not allowlisted) calls:
   ```solidity
   pool.addLiquidity(alice, salt, deltas, callbackData, extensionData);
   ```
4. Pool calls `extension.beforeAddLiquidity(bob, alice, salt, deltas, extensionData)`.
5. Extension checks `allowedDepositor[pool][alice]` → `true` → passes. Bob's identity is never checked.
6. `LiquidityLib.addLiquidity` calls `IMetricOmmModifyLiquidityCallback(bob).metricOmmModifyLiquidityCallback(...)` — Bob pays tokens.
7. `positionBinShares[keccak256(alice, salt, bin)]` is incremented — Alice receives LP shares.
8. Bob (not allowlisted) has successfully deposited into the restricted pool, violating the admin's access control invariant.
