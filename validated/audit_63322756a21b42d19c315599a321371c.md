### Title
Unbounded `drepDelegs` Set Enables Attacker-Controlled Permanent Freezing of DRep Deposit via `ConwayUnRegDRep` - (File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs)

---

### Summary

The `DRepState` type stores an unbounded `Set (Credential Staking)` called `drepDelegs` tracking every stake credential delegated to a given DRep. When a DRep unregisters via the `ConwayUnRegDRep` certificate, the `GOVCERT` rule iterates over the entire `drepDelegs` set to clear each delegator's back-pointer. Because there is no protocol-level cap on the number of delegators per DRep, an unprivileged attacker can register arbitrarily many stake credentials, delegate them all to a victim DRep, and make the unregistration transaction's ledger-rule evaluation cost grow without bound. Once the set is large enough, no block producer can include the `ConwayUnRegDRep` certificate within the block-validation time budget, permanently freezing the DRep's deposit.

---

### Finding Description

**Root cause — `clearDRepDelegations` in `conwayGovCertTransition`**

In `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/GovCert.hs`, the `ConwayUnRegDRep` branch of `conwayGovCertTransition` performs:

```haskell
let
  certState' =
    certState & certVStateL . vsDRepsL %~ Map.delete cred
  clearDRepDelegations delegs accountsMap =
    foldr (Map.adjust (dRepDelegationAccountStateL .~ Nothing)) accountsMap delegs
pure $
  case mDRepState of
    Nothing -> certState'
    Just dRepState ->
      certState'
        & certDStateL . accountsL . accountsMapL
          %~ clearDRepDelegations (drepDelegs dRepState)
``` [1](#0-0) 

`clearDRepDelegations` is a `foldr` over `drepDelegs dRepState`. Each iteration calls `Map.adjust` on the global accounts map, which is O(log a) where `a` is the number of registered accounts. The total cost is therefore **O(d × log a)** where `d` is the cardinality of `drepDelegs`. There is no bound on `d`.

**The unbounded field**

`DRepState` is defined in `libs/cardano-ledger-core/src/Cardano/Ledger/DRep.hs`:

```haskell
data DRepState = DRepState
  { drepExpiry :: !EpochNo
  , drepAnchor :: !(StrictMaybe Anchor)
  , drepDeposit :: !(CompactForm Coin)
  , drepDelegs :: !(Set (Credential Staking))
  }
``` [2](#0-1) 

`drepDelegs` is a plain `Set` with no size constraint anywhere in the protocol parameters. The Conway `PParams` record (`ConwayPParams`) contains `cppDRepDeposit`, `cppDRepActivity`, `cppGovActionDeposit`, etc., but no field that caps the number of delegators per DRep. [3](#0-2) 

**How delegations are added**

`unDelegReDelegDRep` in `VState.hs` is called whenever a stake credential registers with a DRep delegation or changes its delegation target. It calls `Set.insert stakeCred` on the target DRep's `drepDelegs`:

```haskell
addNewDelegation =
  case mNewDRep of
    Just (DRepCredential dRepCred) ->
      Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
    _ -> id
``` [4](#0-3) 

Each stake credential that delegates to a DRep adds one entry to `drepDelegs`. Stake credentials do not expire; they remain in `drepDelegs` until they voluntarily unregister or redelegate. The DRep has no mechanism to force delegators to leave.

**Attack path**

1. Attacker registers N stake credentials (each costs `ppKeyDeposit` as a refundable deposit plus a small transaction fee).
2. Each credential is delegated to the victim DRep via `RegDepositDelegTxCert` or `DelegTxCert`, inserting it into `drepDelegs`.
3. The attacker keeps all credentials registered and delegated indefinitely; the only ongoing cost is the initial transaction fees (deposits are fully refundable on unregistration).
4. When the victim DRep submits a `ConwayUnRegDRep` certificate, the ledger rule evaluates `clearDRepDelegations` over N entries. For large N this computation exceeds the block-validation time budget; no block producer can include the transaction.
5. The DRep's `ppDRepDeposit` (500 ADA on mainnet) is permanently locked in the protocol state.

---

### Impact Explanation

**Allowed impact matched**: *High — Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork.*

The DRep deposit (`ppDRepDeposit`) is only returned upon successful processing of `ConwayUnRegDRep`. If that certificate can never be included in a block because its ledger-rule evaluation cost is unbounded, the deposit is frozen for the lifetime of the chain. Restoring the DRep's ability to unregister would require a hard fork that either (a) caps `drepDelegs` size, (b) removes the `clearDRepDelegations` step from `ConwayUnRegDRep`, or (c) introduces a lazy/incremental clearing mechanism.

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender. No governance majority, leaked key, or consensus threshold is required. The attacker's net cost is only transaction fees (deposits are refundable). With `ppKeyDeposit` = 2 ADA and typical fees of ~0.17 ADA per transaction, registering and delegating 100,000 credentials costs roughly 17,000 ADA in non-refundable fees — economically feasible for a motivated attacker targeting a high-profile DRep. The attack is one-shot and permanent: once `drepDelegs` is large enough, no further action by the attacker is needed.

---

### Recommendation

1. **Remove `clearDRepDelegations` from `ConwayUnRegDRep`**: Instead of eagerly clearing all delegators' back-pointers at unregistration time, treat a missing DRep credential in `vsDReps` as an implicit "no delegation" during reward/ratification computation. The `dRepDelegationAccountStateL` field in each account can be lazily invalidated on next access or at epoch boundary.

2. **Alternatively, cap `drepDelegs` size**: Introduce a protocol parameter `ppMaxDRepDelegators :: Word32` and reject `DelegVote` certificates that would push a DRep's delegator count above the cap. This bounds the worst-case cost of `clearDRepDelegations`.

3. **Incremental clearing**: Spread the clearing work across multiple transactions or epoch-boundary pulses, analogous to how `DRepPulser` spreads stake-distribution computation.

### Proof of Concept

```
-- Setup
ppKeyDeposit  = 2_000_000 lovelace  (2 ADA)
ppDRepDeposit = 500_000_000 lovelace (500 ADA)

-- Step 1: Victim registers as DRep
Tx1: ConwayRegDRep victimCred 500_000_000 SNothing
-- victimCred now in vsDReps; drepDelegs victimCred = {}

-- Step 2: Attacker registers N=200_000 stake credentials
-- and delegates each to victimCred (spread across many txs)
for i in 1..200_000:
  TxN: [ RegDepositDelegTxCert attackerCred_i
           (DelegVote (DRepCredential victimCred))
           2_000_000 ]
-- After: |drepDelegs victimCred| = 200_000
-- Attacker cost: ~34_000 ADA in fees; 400_000 ADA in refundable deposits

-- Step 3: Victim attempts to unregister
TxVictim: ConwayUnRegDRep victimCred 500_000_000
-- Ledger evaluates:
--   clearDRepDelegations (Set of 200_000 creds) accountsMap
--   = foldr (Map.adjust ...) accountsMap [200_000 entries]
--   ≈ 200_000 × O(log 200_000) ≈ 3_400_000 Map operations
-- Block producers cannot include TxVictim within slot time budget.
-- victimCred remains in vsDReps; 500 ADA deposit is permanently frozen.

-- Step 4: Attacker recovers deposits
for i in 1..200_000:
  TxR: UnRegDepositTxCert attackerCred_i 2_000_000
-- Attacker recoups 400_000 ADA; net cost = ~34_000 ADA in fees.
```
