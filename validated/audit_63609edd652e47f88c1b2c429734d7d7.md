Audit Report

## Title
`instantWithdrawal` can drain `LRTUnstakingVault` assets reserved for queued withdrawals due to unsynchronized `queuedWithdrawalsBuffer` - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`initiateWithdrawal()` records future asset obligations in `assetsCommitted[asset]` inside `LRTWithdrawalManager`, but the corresponding protection in `LRTUnstakingVault` — `queuedWithdrawalsBuffer[asset]` — is a static operator-set value that is never automatically updated when new queued withdrawals are added. When instant withdrawals are enabled, any rsETH holder can drain the vault to zero, causing `unlockQueue()` to revert unconditionally and temporarily freezing funds for all users who already queued withdrawals.

## Finding Description

**Root cause:** Two separate accounting systems track the same obligation but are never synchronized.

`initiateWithdrawal()` increments `assetsCommitted[asset]` at `LRTWithdrawalManager.sol:173`:
```solidity
assetsCommitted[asset] += expectedAssetAmount;
```
This check uses `getAvailableAssetAmount()` which aggregates across the entire protocol (deposit pool + node delegators + EigenLayer strategies), not just the vault balance.

`instantWithdrawal()` at `LRTWithdrawalManager.sol:231` checks only the vault's static buffer:
```solidity
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
```

`getAssetsAvailableForInstantWithdrawal()` at `LRTUnstakingVault.sol:235-237`:
```solidity
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
```

`queuedWithdrawalsBuffer[asset]` defaults to zero and is only updated by the operator via `setQueuedWithdrawalsBuffer()` at `LRTUnstakingVault.sol:199-208`. There is no code path that increments it when `assetsCommitted` grows.

**Exploit path:**
1. Vault holds 200 ETH. `queuedWithdrawalsBuffer[ETH]` = 0. Total protocol assets = 1000 ETH.
2. User A calls `initiateWithdrawal()`. Check passes (1000 - 0 ≥ 100). `assetsCommitted[ETH]` = 100 ETH. User A's rsETH is now held in `LRTWithdrawalManager`.
3. User B (any rsETH holder) calls `instantWithdrawal()`. `getAssetsAvailableForInstantWithdrawal()` returns `200 - 0 = 200 ETH`. Check passes. `unstakingVault.redeem(ETH, 200 ETH)` executes. Vault balance = 0.
4. Operator calls `unlockQueue()`. `_createUnlockParams()` at `LRTWithdrawalManager.sol:849` reads `unstakingVault.balanceOf(asset)` = 0. Line 297 reverts: `AmountMustBeGreaterThanZero`.
5. User A's queued withdrawal is frozen. Their rsETH is locked in `LRTWithdrawalManager` with no recourse until the vault is refilled.

**Why existing checks fail:** The `onlyInstantWithdrawalAllowed` modifier only checks whether the feature is enabled — it does not account for outstanding queued obligations. The buffer check in `getAssetsAvailableForInstantWithdrawal()` is only as accurate as the last manual operator update, which can be arbitrarily stale.

## Impact Explanation
Users who called `initiateWithdrawal()` have already transferred their rsETH to `LRTWithdrawalManager` and cannot reclaim it. The operator cannot call `unlockQueue()` because the vault balance is zero, causing a hard revert at `LRTWithdrawalManager.sol:297`. Funds are frozen until the vault is refilled via EigenLayer withdrawal completions or new deposits. This is **temporary freezing of funds** (Medium severity).

## Likelihood Explanation
Two conditions must hold: (1) `isInstantWithdrawalEnabled[asset]` is `true` — a normal operational state once the feature is live; (2) `queuedWithdrawalsBuffer[asset]` is zero or stale — the default is zero, and the buffer is never automatically updated. The window of exposure exists continuously between any `initiateWithdrawal()` call and the operator's next manual `setQueuedWithdrawalsBuffer()` call. Both `initiateWithdrawal()` and `instantWithdrawal()` are permissionless. No privileged access, oracle manipulation, or governance capture is required.

## Recommendation
Replace the manually-managed `queuedWithdrawalsBuffer` with a dynamic check that reads `assetsCommitted[asset]` from `LRTWithdrawalManager` directly inside `getAssetsAvailableForInstantWithdrawal()`. Alternatively, expose a callback from `LRTWithdrawalManager` to `LRTUnstakingVault` that atomically increments/decrements the buffer whenever `assetsCommitted` changes — on `initiateWithdrawal()` (increment), `_unlockWithdrawalRequests()` (decrement at `LRTWithdrawalManager.sol:802`), and any cancellation paths. This ensures the two accounting systems remain synchronized without relying on manual operator intervention.

## Proof of Concept
Foundry fork test outline:
1. Deploy/fork with vault holding 200 ETH, `queuedWithdrawalsBuffer[ETH]` = 0, `isInstantWithdrawalEnabled[ETH]` = true.
2. User A calls `initiateWithdrawal(ETH, rsETHFor100ETH, "")`. Assert `assetsCommitted[ETH]` = 100 ETH. Assert User A's rsETH is now in `LRTWithdrawalManager`.
3. User B calls `instantWithdrawal(ETH, rsETHFor200ETH, "")`. Assert call succeeds. Assert `unstakingVault.balanceOf(ETH)` = 0.
4. Advance blocks past `withdrawalDelayBlocks`.
5. Operator calls `unlockQueue(ETH, nextUnusedNonce, ...)`. Assert call reverts with `AmountMustBeGreaterThanZero`.
6. Assert User A cannot call `completeWithdrawal(ETH, "")` (withdrawal is still locked, never unlocked).
