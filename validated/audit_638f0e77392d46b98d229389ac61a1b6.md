### Title
GovernanceThreshold == n Deadlock: Single Offline Participant Permanently Freezes MPC Signing — (`crates/contract/src/primitives/thresholds.rs`, `crates/contract/src/lib.rs`)

### Summary

When the GovernanceThreshold equals the total participant count (`t = n`, which is explicitly permitted by the upper-bound function), a single participant going offline (TEE attestation expiry) permanently freezes the MPC network. The `verify_tee` path sets `accept_requests = false` and cannot auto-reshare, while the `validate_incoming_proposal` check simultaneously blocks the remaining `n-1` participants from proposing a recovery resharing, because it requires at least `threshold = n` old participants in any new proposal. No escape path exists.

### Finding Description

The `governance_threshold_upper_relative_bound` function sets the maximum allowed GovernanceThreshold to `n` (100% of participants):

```rust
// crates/contract/src/primitives/thresholds.rs, lines 19-25
pub(crate) fn governance_threshold_upper_relative_bound(n: u64) -> u64 {
    n
}
```

The code comment even acknowledges the risk: *"would be a discussion subject to drop this upper bound down not to have problems with smart contract being locked if t = n and if an operator stops voting."*

When `t = n` and one participant's TEE attestation expires, `verify_tee` reaches the `TeeValidationResult::Partial` branch with `remaining = n-1`. It calls `validate_governance_against_reconstruction(n-1, GovernanceThreshold=n, ...)`, which fails because `n > n-1`, so the contract sets `accept_requests = false` and stays Running — it cannot auto-reshare:

```rust
// crates/contract/src/lib.rs, lines 1727-1738
if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
    u64::try_from(remaining).expect("participant count fits in u64"),
    current_params.threshold(),
    max_reconstruction_threshold,
) {
    // ...
    self.accept_requests = false;
    return Ok(false);
}
```

The remaining `n-1` participants then attempt to vote for a new parameter set that excludes the offline participant. This proposal contains `n-1` old participants. `validate_incoming_proposal` enforces:

```rust
// crates/contract/src/primitives/thresholds.rs, line 166
if n_old < self.threshold().value() {
    return Err(InvalidCandidateSet::InsufficientOldParticipants.into());
}
```

Since `n-1 < n = threshold`, every resharing proposal is rejected with `InsufficientOldParticipants`. The contract upgrade path (`vote_update`) also requires threshold votes, so with one participant offline and `t = n`, no upgrade can be approved either. The contract is permanently deadlocked.

### Impact Explanation

The MPC network controls signing keys for all foreign-chain assets (Bitcoin, Ethereum, etc.) managed through the chain-signatures contract. With `accept_requests = false` and no path to resharing or contract upgrade, no threshold signatures can ever be issued again. All funds on foreign chains whose only signing authority is this MPC network are permanently frozen. This matches the allowed Critical impact: *"permanent freezing of funds controlled by the MPC network."*

### Likelihood Explanation

- `t = n` is a valid, deployable configuration (the upper bound explicitly allows it, and the code comment acknowledges operators may choose it).
- A single participant going offline — hardware failure, network partition, or a Byzantine participant deliberately allowing their TEE attestation to expire — is a realistic, low-bar event.
- A Byzantine participant strictly below the signing threshold (any single node when `t = n`) can deliberately trigger this by refusing to renew their TEE attestation, causing `verify_tee` to freeze the network with no recovery path.
- No privileged access, key leakage, or threshold-level collusion is required.

### Recommendation

Lower the `governance_threshold_upper_relative_bound` to strictly less than `n` (e.g., `n-1` or `ceil(0.8*n)`) so that at least one participant can be offline while the remaining set still satisfies the threshold relation and can propose a recovery resharing. Alternatively, add an emergency escape path (e.g., a time-locked resharing that requires only `threshold-1` old participants after a prolonged freeze) that does not itself require threshold votes to initiate.

### Proof of Concept

1. Deploy the contract with `n = 5` participants and `GovernanceThreshold = 5` (`t = n`, valid per `governance_threshold_upper_relative_bound`).
2. Participant P5 deliberately stops renewing their TEE attestation; their attestation expires.
3. Any participant calls `verify_tee()`.
   - `reverify_and_cleanup_participants` returns `Partial { participants_with_valid_attestation: [P1,P2,P3,P4] }` (remaining = 4).
   - `validate_governance_against_reconstruction(4, Threshold(5), ...)` → `MaxRequirementFailed { max: 4, found: 5 }`.
   - `accept_requests = false`; function returns `Ok(false)`.
4. P1–P4 attempt `vote_new_parameters` with a proposal containing only P1–P4 (4 old participants).
   - `validate_incoming_proposal`: `n_old = 4 < threshold = 5` → `Err(InsufficientOldParticipants)`.
5. P1–P4 attempt `vote_update` to deploy a patched contract.
   - Quorum requires 5 votes; only 4 participants can vote → quorum never reached.
6. The contract is permanently frozen: no signatures issued, no resharing possible, no upgrade possible. All foreign-chain funds are inaccessible. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
