### Title
Gateway and Mempool Silently Accept Zero-Fee Transactions When `validate_resource_bounds` Is Disabled — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`, `crates/apollo_gateway/src/stateless_transaction_validator.rs`, `crates/apollo_mempool_config/src/config.rs`)

---

### Summary

A single shared configuration flag, `validate_resource_bounds`, simultaneously disables L2 gas price and resource-bound validation across the stateless gateway, stateful gateway, and mempool. When set to `false` — a mode explicitly documented as intended for "system bootstrap" — the system continues to operate without any error or warning, silently admitting transactions with zero resource bounds into the **priority queue** of the mempool. There is no guard preventing this configuration from persisting into or being re-enabled during production operation.

---

### Finding Description

The `validate_resource_bounds` pointer target in `node_config.rs` fans out to three validators simultaneously:

```
gateway_config.static_config.stateful_tx_validator_config.validate_resource_bounds
gateway_config.static_config.stateless_tx_validator_config.validate_resource_bounds
mempool_config.static_config.validate_resource_bounds
``` [1](#0-0) 

The config schema description reads: *"Indicates that validations related to resource bounds are applied. It should be set to false during a system bootstrap."* [2](#0-1) 

**Stateless validator** — when the flag is `false`, the function immediately returns `Ok(())` before checking for zero resource bounds or minimum gas price:

```rust
fn validate_resource_bounds(&self, tx: &RpcTransaction) -> ... {
    if !self.config.validate_resource_bounds {
        return Ok(());   // ← entire check skipped silently
    }
    // zero-bounds check, min_gas_price check, max_l2_gas_amount check
}
``` [3](#0-2) 

**Stateful validator** — the L2 gas price threshold check is wrapped in the same flag:

```rust
async fn validate_resource_bounds(&self, executable_tx: &ExecutableTransaction) -> ... {
    // Skip this validation during the systems bootstrap phase.
    if self.config.validate_resource_bounds {
        // ... threshold check ...
    }
    Ok(())   // ← returns Ok regardless
}
``` [4](#0-3) 

**Mempool** — the `MempoolStaticConfig` field description confirms the effect: *"If false, all transactions are inserted into the priority queue."* This means a zero-gas-price transaction bypasses the pending queue and lands directly in the priority queue, ahead of fee-paying transactions. [5](#0-4) 

The mempool test `test_add_bootstrap_tx_depends_on_config` explicitly verifies this behavior: with `validate_resource_bounds = false`, a transaction with `max_l2_gas_price: 0` is placed in the **priority queue** (not the pending queue): [6](#0-5) 

---

### Impact Explanation

When `validate_resource_bounds = false`:

1. **Stateless gateway**: A transaction with all resource bounds set to zero (zero max-fee equivalent) passes the `ZeroResourceBounds` check and the `MaxGasPriceTooLow` check, entering the pipeline.
2. **Stateful gateway**: The L2 gas price threshold comparison against the previous block's price is skipped entirely.
3. **Mempool**: The zero-gas-price transaction is placed in the **priority queue**, meaning it is returned to the batcher before legitimate fee-paying transactions.

The broken invariant is: *every transaction admitted to the priority queue must have an L2 gas price at or above the configured threshold*. With the flag disabled, this invariant is silently violated. Zero-fee transactions are sequenced ahead of paying ones, constituting an incorrect fee/resource accounting effect with direct economic impact on sequencer revenue and transaction ordering fairness.

Impact scope: **High — Mempool/gateway admission accepts invalid transactions before sequencing**; and potentially **Critical — Incorrect fee/resource accounting with economic impact** if zero-resource-bound transactions execute without fee collection.

---

### Likelihood Explanation

The configuration is explicitly documented as a bootstrap mode, meaning operators are expected to toggle it. There is no programmatic guard (e.g., an environment check, a startup assertion, or a warning log) that prevents `validate_resource_bounds = false` from persisting into or being re-enabled during production. Any operator misconfiguration, deployment script error, or deliberate bootstrap-mode re-activation exposes the live sequencer to this condition. An attacker who can observe the node's admission behavior (e.g., by submitting probe transactions) can detect when the flag is disabled and flood the priority queue with zero-fee transactions.

---

### Recommendation

**Short term**: Add a startup assertion or warning log that rejects (or loudly warns) if `validate_resource_bounds = false` is detected outside of an explicitly declared development/bootstrap environment. At minimum, emit a `warn!` or `error!` log at node startup when this flag is `false`, analogous to how the external report recommends returning a 500/503 when Redis is disabled.

**Long term**: Rename the flag to something that makes the insecurity explicit (e.g., `bootstrap_mode_disable_resource_bound_validation`) and enforce that it cannot be set to `true` (disabled) unless a separate `allow_insecure_bootstrap` flag is also set. This mirrors the external report's recommendation to make insecure configuration options impossible to enable outside of development.

---

### Proof of Concept

1. Deploy the sequencer with `validate_resource_bounds = false` (bootstrap mode, as documented).
2. Submit an `InvokeV3` transaction with `AllResourceBounds { l1_gas: 0, l2_gas: ResourceBounds { max_amount: N, max_price_per_unit: 0 }, l1_data_gas: 0 }`.
3. Observe that the stateless validator returns `Ok(())` at `stateless_transaction_validator.rs:60-62` without checking zero bounds.
4. Observe that the stateful validator returns `Ok(())` at `stateful_transaction_validator.rs:228-242` without comparing against the previous block's L2 gas price.
5. Observe (via `test_add_bootstrap_tx_depends_on_config`) that the mempool places this zero-gas-price transaction in the **priority queue**, ahead of all fee-paying transactions.
6. The batcher pulls this transaction first; it is sequenced with zero fee declared, violating the fee-accounting invariant. [4](#0-3) [3](#0-2) [7](#0-6) [1](#0-0)

fn test_add_bootstrap_tx_depends_on_config(#[values(true, false)] allow_bootstrap: bool) {
    let mut builder = MempoolTestContentBuilder::new();
    builder.config.static_config.validate_resource_bounds = !allow_bootstrap;
    builder.gas_price_threshold = GasPrice(7);
    let mut mempool = builder.build_full_mempool();

    let zero_bounds_tx = add_tx_input!(
        tx_hash: 1,
        address: "0x0",
        tx_nonce: 0,
        account_nonce: 0,
        tip: 0,
        max_l2_gas_price: 0
    );
    add_tx(&mut mempool, &zero_bounds_tx);

    let txs = vec![TransactionReference::new(&zero_bounds_tx.tx)];
    let (expected_priority, expected_pending) =
        if allow_bootstrap { (txs, vec![]) } else { (vec![], txs) };
    let expected_mempool_content = MempoolTestContentBuilder::new()
        .with_pending_queue(expected_pending)
        .with_priority_queue(expected_priority)
        .build();
    expected_mempool_content.assert_eq(&mempool.content());
