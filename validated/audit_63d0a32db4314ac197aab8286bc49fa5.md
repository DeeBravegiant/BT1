### Title
Governance Update Proposal Deposits Permanently Locked With No Cancellation or Refund Path - (File: `crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

### Summary
`propose_update` requires a storage-staking deposit (up to ~40 NEAR for a full contract binary). There is no expiration, cancellation, or refund mechanism for proposals that are never executed. When `do_update` fires for a competing proposal, it clears all other entries without refunding their depositors. The deposit is permanently locked in the contract with no recovery path.

### Finding Description
`propose_update` collects a deposit sized to cover storage staking for the uploaded WASM or config blob: [1](#0-0) 

The deposit is accepted and the entry is stored. Only the *excess* above the required amount is refunded to the proposer at call time. The required portion stays in the contract.

The only paths that remove entries from `proposed_updates` are:

1. `do_update`, which clears **all** entries and votes when any single proposal crosses threshold: [2](#0-1) 

2. `remove_non_participant_update_votes`, which removes only *votes*, not entries or deposits: [3](#0-2) 

Neither path refunds the deposits of the cleared proposals. There is no `cancel_update`, no expiration timestamp on entries, and no withdrawal function. The `UpdateEntry` struct stores `bytes_used` for accounting but no depositor address to refund: [4](#0-3) 

### Impact Explanation
A participant who calls `propose_update` with a full contract binary (up to ~1.5 MB) must attach up to ~40 NEAR as a storage-staking deposit. If a competing proposal is executed by threshold vote, `do_update` clears all entries including the first proposer's, but the deposit is never returned. The NEAR tokens remain in the contract's balance with no on-chain mechanism to recover them. This breaks the accounting invariant that governance deposits are returned when the associated storage is freed.

**Impact: Medium** — balance/accounting invariant broken; real NEAR tokens permanently locked in the contract.

### Likelihood Explanation
This occurs in any normal governance scenario where multiple proposals are live simultaneously and one reaches threshold before the others. It also occurs when a participant is removed via resharing after having proposed an update — their deposit is stranded. No adversarial action is required; routine governance operations trigger the loss.

### Recommendation
Store the depositor's `AccountId` alongside each `UpdateEntry` and refund the deposit when an entry is removed — whether by `do_update` clearing all entries, or by a new explicit `cancel_update` function callable by the original proposer. Alternatively, add an expiration block height to each proposal and refund on expiry, mirroring the 7-day grace-period fix applied in the referenced MerkleLockup report.

### Proof of Concept

1. Participant A calls `propose_update` with a 1.5 MB WASM binary, attaching 40 NEAR.
2. Participant B calls `propose_update` with a different WASM binary, attaching 40 NEAR.
3. Threshold participants vote for B's proposal via `vote_update`.
4. `do_update` is triggered for B's proposal: [5](#0-4) 

5. A's entry is removed from `entries`. No refund is issued to A. A's 40 NEAR is permanently locked in the contract balance.
6. There is no function in the contract that A can call to recover the deposit. [6](#0-5)

impl ProposedUpdates {
    pub fn required_deposit(update: &Update) -> NearToken {
        required_deposit(bytes_used(update))
    }

    /// Propose an update given the new contract code and/or config.
    pub fn propose(&mut self, update: Update) -> UpdateId {
        let bytes_used = bytes_used(&update);

        let id = self.id.generate();
        self.entries.insert(id, UpdateEntry { update, bytes_used });

        id
    }
