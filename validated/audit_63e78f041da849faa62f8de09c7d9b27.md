Audit Report

## Title
Unguarded Division by Zero in `getRsETHAmountToMint` When `rsETHPrice` Is Uninitialized — (`contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` with no zero-check. `rsETHPrice` in `LRTOracle` is a plain `uint256` storage variable that defaults to `0` and is never set in `initialize()`. Until `updateRSETHPrice()` is explicitly called, every call to `depositETH()` and `depositAsset()` reverts with a division-by-zero panic, freezing all deposits.

## Finding Description
`LRTOracle.rsETHPrice` is declared as a bare storage variable with no initialization:

```solidity
// contracts/LRTOracle.sol L28
uint256 public override rsETHPrice;
```

`LRTOracle.initialize()` sets only `lrtConfig` and emits an event — it never calls `_updateRsETHPrice()` or assigns `rsETHPrice`:

```solidity
// contracts/LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

`_updateRsETHPrice()` is the only writer of `rsETHPrice` (line 313), and it is only reachable via the public `updateRSETHPrice()`. Until that call is made, `rsETHPrice == 0`.

`LRTDepositPool.getRsETHAmountToMint()` performs an unchecked division by this value:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

Both primary deposit entry points call `_beforeDeposit()`, which calls `getRsETHAmountToMint()`:

- `depositETH()` → `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, ...)` (L87)
- `depositAsset()` → `_beforeDeposit(asset, depositAmount, ...)` (L111)

When `rsETHPrice == 0`, Solidity 0.8's checked arithmetic causes a `Panic(0x12)` revert on every deposit call. No existing guard prevents this: neither `_beforeDeposit` nor `getRsETHAmountToMint` checks the oracle return value before dividing.

**Secondary path — silent zero return in `getExpectedAssetAmount`:**
`LRTWithdrawalManager.getExpectedAssetAmount()` uses `rsETHPrice` as the numerator:

```solidity
// contracts/LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

When `rsETHPrice == 0` this silently returns `0`. `instantWithdrawal()` burns the caller's rsETH (L229) before the availability check (L231); with `assetAmountUnlocked == 0` the check `0 > available` is always false, so execution continues and the user receives nothing. `initiateWithdrawal()` similarly locks rsETH in the contract and records `expectedAssetAmount = 0`, yielding zero on completion. This path requires `rsETHPrice` to reach `0` while rsETH is in circulation — possible if `pricePercentageLimit == 0` (its default) and `totalETHInProtocol` drops to `0`, because the downside-protection circuit-breaker at lines 273-274 is gated on `pricePercentageLimit > 0` and therefore does not fire.

## Impact Explanation
**Primary (Medium — Temporary freezing of funds):** All deposits via `depositETH()` and `depositAsset()` revert with division-by-zero for the entire window in which `rsETHPrice == 0`. The freeze lifts only when a privileged operator calls `updateRSETHPrice()`.

**Secondary (Critical — Direct theft of user funds):** If `rsETHPrice` is written to `0` by `_updateRsETHPrice()` (complete-slashing scenario with `pricePercentageLimit == 0`), any user calling `instantWithdrawal()` has their rsETH burned and receives `0` underlying assets — a total loss of principal.

## Likelihood Explanation
**Deposit freeze:** Near-certain on every fresh deployment. `rsETHPrice` is `0` from contract construction until `updateRSETHPrice()` is first called. Any deposit attempted in this window reverts. No attacker action is required; the condition is structural.

**Withdrawal fund loss:** Low probability but reachable without privileged action once the state is reached. `pricePercentageLimit` defaults to `0` (no downside guard), so a complete slashing event that drives `totalETHInProtocol` to `0` causes `_updateRsETHPrice()` to write `rsETHPrice = 0` without pausing the protocol. Any rsETH holder who then calls `instantWithdrawal()` loses their entire balance.

## Recommendation
Add an explicit zero-guard in `getRsETHAmountToMint()`:

```solidity
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert InvalidRsETHPrice();
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / currentRsETHPrice;
```

Add the same guard in `getExpectedAssetAmount()`:

```solidity
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert InvalidRsETHPrice();
underlyingToReceive = amount * currentRsETHPrice / lrtOracle.getAssetPrice(asset);
```

`LRTOracle.initialize()` should call `_updateRsETHPrice()` (or unconditionally set `rsETHPrice = 1 ether`) so the variable is never `0` after deployment.

## Proof of Concept
**Deposit freeze:**
1. Deploy `LRTOracle` and `LRTDepositPool`; do not call `updateRSETHPrice()`.
2. `rsETHPrice` is `0` (default storage value).
3. Any user calls `depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `getRsETHAmountToMint` → `(1e18 * assetPrice) / 0` → Solidity 0.8 `Panic(0x12)`.
5. All deposits are blocked until an operator calls `updateRSETHPrice()`.

**Withdrawal fund loss:**
1. Set `pricePercentageLimit = 0` (default; no privileged action needed to leave it there).
2. Drive `totalETHInProtocol` to `0` (complete slashing).
3. Call `updateRSETHPrice()`; `_updateRsETHPrice()` computes `newRsETHPrice = 0` and writes it (downside-protection branch skipped because `pricePercentageLimit > 0` is false).
4. User holding rsETH calls `instantWithdrawal(asset, rsETHAmount, "")`.
5. `getExpectedAssetAmount` returns `0`; `burnFrom` burns the user's rsETH; `_transferAsset` sends `0` assets.
6. User loses their entire rsETH balance with no recourse.
