Audit Report

## Title
Stale `rsETHPrice` in `instantWithdrawal` Allows Over-Redemption at Expense of Remaining rsETH Holders - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`instantWithdrawal` computes the asset payout using the cached `rsETHPrice` state variable from `LRTOracle` alongside the live `getAssetPrice(asset)` return. Because `rsETHPrice` is only updated on explicit calls to `updateRSETHPrice()`, any window where the live LST price has fallen but `rsETHPrice` has not yet been refreshed allows a caller to redeem rsETH for more underlying assets than their tokens are worth. The resulting shortfall is silently absorbed by all remaining rsETH holders when the price is eventually updated.

## Finding Description

**Root cause 1 — `rsETHPrice` is a cached state variable.**

`LRTOracle` stores `rsETHPrice` as a plain `uint256` state variable updated only inside `_updateRsETHPrice()`, which is called only when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is explicitly invoked:

- `LRTOracle.sol:28` — `uint256 public override rsETHPrice;`
- `LRTOracle.sol:87-89` — `updateRSETHPrice()` is the sole public entry point.
- `LRTOracle.sol:313` — `rsETHPrice = newRsETHPrice;` is the only write.

Between calls, `rsETHPrice` can diverge from the true protocol value whenever underlying LST prices move.

**Root cause 2 — `instantWithdrawal` mixes the stale cached price with a live oracle price.**

`getExpectedAssetAmount` (called unconditionally from `instantWithdrawal` at `LRTWithdrawalManager.sol:228`) computes:

```
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```
(`LRTWithdrawalManager.sol:593`)

`lrtOracle.rsETHPrice()` is the stale cached value; `lrtOracle.getAssetPrice(asset)` reads the live price fetcher (`LRTOracle.sol:156-158`). When the live asset price has dropped but `rsETHPrice` has not yet been reduced, the division yields a larger-than-correct asset amount.

`instantWithdrawal` performs no price refresh before this call (`LRTWithdrawalManager.sol:212-253`).

**Why existing guards are insufficient.**

The downside-protection logic in `_updateRsETHPrice` (`LRTOracle.sol:270-282`) pauses the protocol only when `updateRSETHPrice()` is actually called and the drop exceeds `pricePercentageLimit`. It provides zero protection during the stale window before that call.

The queued-withdrawal path is protected: `_calculatePayoutAmount` (`LRTWithdrawalManager.sol:824-835`) caps the payout at the minimum of the originally expected amount and the current return, so a price drop after `initiateWithdrawal` harms the withdrawer, not remaining holders. `instantWithdrawal` has no equivalent cap.

**Exploit path.**

1. Protocol holds 100 stETH; rsETH supply = 100; `rsETHPrice` = 1.00 ETH (stored).
2. Live LST oracle drops to 0.95 ETH for stETH; `rsETHPrice` has not been refreshed.
3. Attacker calls `instantWithdrawal(stETH, 10e18, "")`:
   - `assetAmountUnlocked = 10e18 * 1.00e18 / 0.95e18 = 10.526e18 stETH`
   - Fair value of 10 rsETH = 9.50 ETH; attacker extracts ~0.526 stETH excess.
4. `updateRSETHPrice()` is called: remaining 90 rsETH now backs 89.474 stETH × 0.95 = 84.99 ETH → new price = 0.9443 ETH instead of the correct 0.95 ETH.
5. Passive holders absorb the ~0.057 ETH/rsETH shortfall.

No mempool access or special privilege is required; the attacker only needs to monitor `lrtOracle.getAssetPrice(asset)` on-chain and call `instantWithdrawal` before `updateRSETHPrice()` is called.

## Impact Explanation

**High — Theft of unclaimed yield.**

Each `instantWithdrawal` executed during a stale-price window extracts more underlying assets than the burned rsETH entitles the caller to. The deficit is transferred to all remaining rsETH holders via a lower price on the next `updateRSETHPrice()` call. The economic incentive is to exit before the update and re-enter afterward at the lower price, compounding losses for passive holders. This matches the "Theft of unclaimed yield" impact class exactly.

## Likelihood Explanation

- `instantWithdrawal` requires `isInstantWithdrawalEnabled[asset]` to be set by a manager, but once enabled the attack surface is permanently open to any unprivileged caller.
- `LRTUnstakingVault` must hold sufficient assets (`getAssetsAvailableForInstantWithdrawal`), which is the normal operating condition.
- LST price movements (stETH, ETHx) and EigenLayer slashing events are realistic, recurring triggers; the stale-price window spans the interval between oracle bot runs (minutes to hours).
- No technical sophistication beyond reading a public view function is required.

## Recommendation

Force a price refresh at the top of `instantWithdrawal` before `getExpectedAssetAmount` is called:

```solidity
ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
```

Alternatively, compute the payout directly from live oracle prices (mirroring `_getTotalEthInProtocol` logic) rather than reading the cached `rsETHPrice`, eliminating the stale-price window entirely. A secondary mitigation is a same-block deposit/withdrawal cooldown to prevent atomic round-trip exploitation.

## Proof of Concept

```
Foundry fork test outline:
1. Fork mainnet; deploy protocol with 100 stETH deposited, rsETH supply = 100,
   rsETHPrice = 1.00e18 (set via updateRSETHPrice()).
2. Mock the stETH price oracle to return 0.95e18 (simulating a depeg).
   Do NOT call updateRSETHPrice() — rsETHPrice remains 1.00e18.
3. Enable instant withdrawal for stETH (vm.prank(manager)).
4. Attacker calls instantWithdrawal(stETH, 10e18, "").
   Assert: attacker receives > 10e18 * 0.95e18 / 1e18 = 9.5e18 stETH.
   Concretely: 10e18 * 1.00e18 / 0.95e18 ≈ 10.526e18 stETH.
5. Call updateRSETHPrice().
   Assert: new rsETHPrice < 0.95e18 (should be ~0.9443e18).
   Assert: remaining 90 rsETH holders' pro-rata ETH value < 90 * 0.95e18.
6. Difference between step 4 payout and fair value (9.5e18 stETH) quantifies
   the theft from remaining holders.
```
