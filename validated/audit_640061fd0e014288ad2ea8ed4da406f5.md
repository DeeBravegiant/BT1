Audit Report

## Title
`_processClaim` Enforces `account == msg.sender`, Permanently Blocking Unclaimed Yield for Smart Contract Recipients - (File: contracts/KERNEL/KernelMerkleDistributor.sol)

## Summary

`KernelMerkleDistributor._processClaim` contains a hard `account != msg.sender` guard that rejects any caller who is not the exact `account` address in the merkle leaf. Because the merkle proof already cryptographically binds the claim to `account`, this guard is redundant but actively prevents smart contract addresses (e.g., vesting contracts, protocol-owned treasuries, or any contract without a generic `execute` method) that are included in the merkle tree from ever claiming their KERNEL yield, permanently freezing it.

## Finding Description

`_processClaim` at line 311–313 unconditionally reverts when `account != msg.sender`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Both `claim()` (L261) and `claimAndStake()` (L280) delegate to `_processClaim`, so neither entry point can be called by a third party on behalf of `account`.

The `IMerkleDistributor` interface (L125) declares `claim` with a distinct `account` parameter, signalling that the caller and the beneficiary may differ. The merkle leaf at L320 is:

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount));
```

Any valid proof is already bound to `account`; a third-party caller cannot redirect tokens to themselves. The `account != msg.sender` guard therefore provides zero additional security while breaking the third-party execution model.

**Affected smart contract recipients:** Any address in the merkle tree that is a smart contract without a generic `execute` capability (e.g., a vesting contract, a protocol treasury with no arbitrary-call method, or a reward-splitter contract) cannot call `claim()` with itself as `msg.sender`. Because the merkle root is periodically updated and cumulative amounts only increase, the unclaimed yield for such addresses accumulates and is permanently inaccessible.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Smart contract addresses included in the merkle tree that lack a generic external-call mechanism have no viable path to invoke `claim()` or `claimAndStake()` with `msg.sender == account`. Their KERNEL yield accumulates in the distributor and can never be extracted. This matches the allowed impact "Medium. Permanent freezing of unclaimed yield."

## Likelihood Explanation

Merkle distributions in DeFi routinely include smart contract addresses: protocol-owned treasuries, vesting contracts, DAO multisigs with restricted execution modules, and reward-splitter contracts. Any such address in the tree that cannot directly dispatch a `claim()` call is permanently affected. The condition is reachable by any unprivileged external observer who notices the affected address is not claiming; no privileged access is required to trigger or observe the freeze.

## Recommendation

Remove the `account != msg.sender` guard from `_processClaim`. The merkle proof is the sole and sufficient authorization mechanism:

```solidity
// Delete these lines:
// if (account != msg.sender) {
//     revert Unauthorized();
// }
```

Any caller supplying a valid proof for `account` can only direct tokens to `account`, so no theft vector is introduced.

## Proof of Concept

1. Deploy a minimal vesting contract `VestingVault` that holds KERNEL on behalf of a beneficiary but exposes no `execute(address, bytes)` method.
2. Include `address(VestingVault)` in the merkle tree with a non-zero `cumulativeAmount`.
3. Call `setMerkleRoot` to activate the distribution.
4. Attempt `claim(index, address(VestingVault), amount, proof)` from any EOA — `msg.sender != account` → `revert Unauthorized()`.
5. Attempt to call `claim` from inside `VestingVault` — no such method exists in the contract.
6. Observe that `userClaims[address(VestingVault)].cumulativeAmount` remains 0 across all future merkle root updates; the yield is permanently frozen.

Foundry test sketch:
```solidity
function test_smartContractRecipientCannotClaim() public {
    address vault = address(new VestingVault()); // no execute()
    // build merkle tree with vault as leaf
    bytes32[] memory proof = buildProof(vault, amount);
    vm.prank(address(0xRelayer));
    vm.expectRevert(IMerkleDistributor.Unauthorized.selector);
    distributor.claim(index, vault, amount, proof);
    // vault itself also cannot call claim — no code path exists
}
```
