Audit Report

## Title
Stale `rsETHPrice` Used in Deposit and Instant-Withdrawal Calculations Without Prior Oracle Update â€” (`contracts/LRTDepositPool.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTOracle` stores the rsETH/ETH exchange rate in the public storage variable `rsETHPrice`, which is only updated when `updateRSETHPrice()` is explicitly called. Neither the deposit path (`depositETH()`/`depositAsset()`) nor `instantWithdrawal()` trigger this update before reading the cached price, allowing any unprivileged caller to mint excess rsETH against a stale (lower-than-true) price, diluting the accrued yield of all existing holders.

## Finding Description
`rsETHPrice` is a plain storage variable in `LRTOracle`: [1](#0-0) 

It is only refreshed inside `_updateRsETHPrice()`, which is reached via the public `updateRSETHPrice()` (no access-control modifier, only `whenNotPaused`): [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` reads `lrtOracle.rsETHPrice()` directly with no preceding update call: [3](#0-2) 

This function is invoked unconditionally by `_beforeDeposit()`, which is called by both `depositETH()` and `depositAsset()`: [4](#0-3) 

`LRTWithdrawalManager.instantWithdrawal()` calls `getExpectedAssetAmount()`, which also reads the cached price without refreshing it: [5](#0-4) 

No existing guard in any of these paths forces an oracle update. The `minRSETHAmountExpected` slippage parameter in `depositETH()` does not protect against this because the attacker *benefits* from the stale lower price and would set a permissive minimum. The `pricePercentageLimit` check inside `_updateRsETHPrice()` can actually extend the staleness window: if the accumulated price increase exceeds the configured threshold, non-manager callers receive `PriceAboveDailyThreshold` and cannot force an update at all, leaving the stale price in place until a manager acts. [6](#0-5) 

## Impact Explanation
**High â€” Theft of unclaimed yield.**

`rsETHPrice` increases monotonically as staking rewards accrue. Between oracle updates the stored price is lower than the true current value. A depositor calling `depositETH()` with a stale price receives:

```
rsethAmountToMint = (depositAmount Ã— assetPrice) / rsETHPrice
```

A lower denominator inflates `rsethAmountToMint`. The new depositor receives more rsETH shares than the current protocol value justifies. Because rsETH is a share token backed by a fixed pool of assets, over-minting dilutes every existing holder's proportional claim â€” their accrued yield is transferred to the late depositor. This is a concrete, quantifiable financial loss to existing holders matching the "High â€” Theft of unclaimed yield" impact class.

For `instantWithdrawal()`, if `rsETHPrice` is stale and *higher* than the true value (e.g., after a slashing event not yet reflected), the user redeems rsETH for more underlying assets than entitled, directly draining the unstaking vault.

## Likelihood Explanation
`updateRSETHPrice()` is a separate transaction with no atomic coupling to any deposit or withdrawal. Staking rewards accrue continuously; the oracle is updated periodically by an off-chain keeper or manual call. There is always a window â€” potentially hours or days â€” during which `rsETHPrice` is stale. Any depositor can exploit this window without any special privilege, simply by depositing when the price has not been recently refreshed. The attack is repeatable every time the oracle lags behind true accrued value.

## Recommendation
Call `updateRSETHPrice()` (or an equivalent internal helper) at the start of `depositETH()`, `depositAsset()`, and `instantWithdrawal()` before any exchange-rate-dependent calculation is performed. This ensures the price used for minting and redemption always reflects the current protocol state. Alternatively, expose an internal `_updateRsETHPrice()` call that can be embedded atomically in these flows without the public `whenNotPaused` wrapper if pausing semantics need to differ.

## Proof of Concept
1. Staking rewards accrue for 24 hours; `updateRSETHPrice()` has not been called. Suppose the true rsETH price is `1.05 ETH` but the stored `rsETHPrice` is `1.00 ETH`.
2. Attacker calls `depositETH{ value: 100 ether }(0, "")`.
3. `getRsETHAmountToMint` computes: `(100e18 Ã— 1e18) / 1.00e18 = 100 rsETH`.
4. True fair amount at current price: `100e18 / 1.05e18 â‰ˆ 95.24 rsETH`.
5. Attacker receives `â‰ˆ 4.76 rsETH` more than entitled â€” representing the yield accrued by all existing holders since the last oracle update.
6. After `updateRSETHPrice()` is called, the attacker's rsETH is worth `100 Ã— 1.05 = 105 ETH` while they only deposited `100 ETH`, extracting `â‰ˆ 5 ETH` of yield from existing holders.
