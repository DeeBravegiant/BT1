### Title
Unconditional `burn_tokens_if_needed` in `resolve_fast_transfer` destroys relayer tokens on failed delivery — (`near/omni-bridge/src/lib.rs`)

### Summary

`resolve_fast_transfer` calls `burn_tokens_if_needed` **before** checking `is_refund_required`. For deployed (wrapped) tokens with a non-empty `msg`, this dispatches a detached burn of `amount_without_fee` from the bridge's balance regardless of whether the delivery succeeded. When the recipient rejects the transfer (`ft_on_transfer` returns the full amount → `is_refund_required = true`), the function returns `amount` to trigger a NEP-141 refund to the relayer, but the bridge's token balance was already burned by the detached receipt, so the refund transfer fails and the relayer's tokens are permanently destroyed.

### Finding Description

**Call sequence:**

1. Relayer calls `ft_transfer_call` on the token contract → bridge's `ft_on_transfer` → `fast_fin_transfer` → `fast_fin_transfer_to_near_callback`.
2. `fast_fin_transfer_to_near_callback` calls `send_tokens(token, recipient, amount_without_fee, msg)`. For a deployed token with non-empty `msg`, `send_tokens` calls `mint(recipient, amount_without_fee, Some(msg))` on the token contract, minting new tokens to the recipient and invoking `ft_on_transfer` on the recipient contract.
3. `resolve_fast_transfer` is called as the callback.
4. **Line 908:** `burn_tokens_if_needed(token, amount_without_fee)` — dispatches `ext_token::burn(amount_without_fee).detach()` unconditionally. This schedules a burn receipt against the bridge's balance on the token contract.
5. **Line 910:** `is_refund_required(true)` reads promise result 0. If the recipient's `ft_on_transfer` returned the full amount (rejected), the mint contract burned the minted tokens and returned `U128(0)` used → `is_refund_required = true`.
6. The function returns `amount_without_fee` to the NEP-141 callback chain, signalling the relayer's token contract to refund `amount_without_fee` from the bridge's balance to the relayer.
7. **Receipt ordering:** The detached burn receipt (created in step 4) was enqueued before the refund receipt (created by the relayer's token contract after receiving the return value). The burn executes first, reducing the bridge's balance by `amount_without_fee`. The subsequent refund receipt fails because the bridge no longer holds those tokens.

The relayer's `amount_without_fee` tokens are permanently destroyed.

**Contrast with the correct pattern** in `fin_transfer_send_tokens_callback`: [1](#0-0) 

There, `burn_tokens_if_needed` is called **inside** the `if is_refund_required` block — only on failure, and only to discard tokens that were already minted to the bridge but could not be delivered. `resolve_fast_transfer` inverts this logic.

**Root cause:** `resolve_fast_transfer` was written with the burn unconditionally before the refund check: [2](#0-1) 

whereas `burn_tokens_if_needed` dispatches a detached, irrevocable receipt: [3](#0-2) 

### Impact Explanation

For any fast transfer to NEAR over a deployed (wrapped) token where `msg` is non-empty and the recipient contract rejects the delivery, the relayer permanently loses `amount_without_fee` tokens. The bridge's backing invariant is also violated: tokens are burned without a corresponding successful cross-chain settlement.

### Likelihood Explanation

Any recipient contract that can reject `ft_on_transfer` (e.g., a paused DeFi protocol, a contract with an allowlist, or a deliberately crafted contract) triggers this path. The relayer has no way to predict or prevent recipient-side rejection. The condition is reachable through normal, unprivileged bridge usage.

### Recommendation

Move `burn_tokens_if_needed` into the success branch, mirroring `fin_transfer_send_tokens_callback`:

```rust
pub fn resolve_fast_transfer(...) -> U128 {
    if Self::is_refund_required(is_ft_transfer_call) {
        self.remove_fast_transfer(fast_transfer_id);
        amount
    } else {
        // Only burn on success to prevent double-minting on finalization
        self.burn_tokens_if_needed(token_id.clone(), amount);
        U128(0)
    }
}
```

### Proof of Concept

1. Deploy a wrapped token (deployed token) on NEAR via the bridge.
2. Register a recipient contract whose `ft_on_transfer` always returns the full `amount` (rejects all transfers).
3. As a relayer, call `ft_transfer_call` on the token contract with a `FastFinTransferMsg` targeting that recipient and a non-empty `msg`.
4. Observe that `resolve_fast_transfer` is called with `is_ft_transfer_call = true`.
5. The mint to the recipient is rejected; promise result 0 is `U128(0)`.
6. `burn_tokens_if_needed` fires unconditionally (detached burn receipt).
7. `is_refund_required` returns `true`; `resolve_fast_transfer` returns `amount_without_fee`.
8. The NEP-141 refund receipt executes after the burn receipt and fails (bridge balance insufficient).
9. Verify: relayer's balance is reduced by `amount_without_fee`; tokens are gone. [4](#0-3) [5](#0-4) [6](#0-5)
