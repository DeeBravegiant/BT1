Audit Report

## Title
Stale `rsETHPrice` Allows Depositors to Dilute Existing Holders' Accumulated Yield - (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary
`LRTOracle.rsETHPrice` is a stored value updated only when `updateRSETHPrice()` is explicitly called. Because `LRTDepositPool` mints rsETH using this stored price without refreshing it first, any depositor can deposit at a below-true-value price during the staleness window and then trigger a price update, capturing a disproportionate share of yield that accumulated for existing holders. The attack is permissionless, repeatable, and requires no privileged access.

## Finding Description
`LRTOracle.updateRSETHPrice()` is declared `public` with no access control: [1](#0-0) 

The stored `rsETHPrice` is only updated when this function is called. Between calls, as underlying LSTs (e.g. stETH) rebase or appreciate, the stored price lags behind the true protocol value.

`LRTDepositPool.getRsETHAmountToMint` computes the rsETH to mint using the stored (potentially stale) `rsETHPrice`: [2](#0-1) 

`_beforeDeposit`, called by both `depositETH` and `depositAsset`, invokes `getRsETHAmountToMint` with no prior call to `updateRSETHPrice()`: [3](#0-2) 

**Exploit path:**
1. stETH rebases (daily, ~0.01%/day). The stored `rsETHPrice` is now lower than the true protocol value.
2. Attacker calls `depositAsset` with a large amount. rsETH is minted at the stale (lower) price — more rsETH than the true fair share.
3. Attacker calls `updateRSETHPrice()`. The price rises to reflect the accumulated appreciation. Because the attacker's deposit increased `rsethSupply` more than proportionally, the new price is lower than it would have been without the attack.
4. Attacker's rsETH is worth more than deposited. The shortfall is borne by existing holders whose accumulated yield was diluted.

**Concrete example:**
- State: 100 ETH in protocol, 100 rsETH outstanding, stored `rsETHPrice = 1.00e18` (stale). True value: 101 ETH.
- Attacker deposits 100 ETH → mints `100 / 1.00 = 100 rsETH`.
- `updateRSETHPrice()` called: new price = `201 / 200 = 1.005 ETH`.
- Attacker's 100 rsETH = 100.5 ETH (0.5 ETH profit).
- Original holders' 100 rsETH = 100.5 ETH instead of 101 ETH (0.5 ETH yield stolen).

The `pricePercentageLimit` guard only reverts if the price increase exceeds the configured threshold: [4](#0-3) 

A normal daily stETH rebase (~0.01%) is well within any reasonable limit, so the attacker can freely call `updateRSETHPrice()` themselves after depositing.

## Impact Explanation
**High — Theft of unclaimed yield.** Yield accumulated by existing rsETH holders between oracle updates is partially captured by the attacker. The attacker profits directly at the expense of long-term depositors. The loss per attack cycle is proportional to the accumulated yield and the attacker's deposit size relative to the existing TVL. This matches the allowed impact: *High. Theft of unclaimed yield.*

## Likelihood Explanation
**High.** stETH rebases every ~24 hours, creating a predictable, recurring staleness window. Both `depositAsset`/`depositETH` and `updateRSETHPrice()` are permissionless — no special role, key, or governance action is required. An MEV bot can monitor the stETH rebase schedule on-chain and execute the attack atomically. The attack is repeatable every oracle update cycle, making cumulative losses significant.

## Recommendation
Call `_updateRsETHPrice()` (or `updateRSETHPrice()`) atomically at the beginning of `_beforeDeposit` before computing `rsethAmountToMint`. This ensures every deposit uses the freshest possible price, eliminating the staleness window. Alternatively, introduce a deposit fee or a time-weighted average price (TWAP) for the rsETH rate to make the attack economically unattractive.

## Proof of Concept
**Foundry fork test outline:**

```solidity
function testFrontRunStalePrice() public {
    // Fork mainnet after a stETH rebase but before updateRSETHPrice() is called
    // 1. Record oracle.rsETHPrice() — stale (lower than true value)
    uint256 stalePrice = oracle.rsETHPrice();

    // 2. Compute true price by calling _getTotalEthInProtocol() off-chain or via a view
    // Confirm stalePrice < truePrice

    // 3. Attacker deposits large stETH amount at stale price
    vm.startPrank(attacker);
    stETH.approve(address(pool), type(uint256).max);
    pool.depositAsset(address(stETH), LARGE_AMOUNT, 0, "");
    uint256 attackerRsETH = rsETH.balanceOf(attacker);

    // 4. Trigger price update
    oracle.updateRSETHPrice();
    uint256 newPrice = oracle.rsETHPrice();

    // 5. Assert attacker's rsETH value > deposited ETH value
    uint256 attackerValue = attackerRsETH * newPrice / 1e18;
    assertGt(attackerValue, LARGE_AMOUNT * stalePrice / 1e18);

    // 6. Assert existing holders received less yield than without the attack
    // (compare rsETHPrice with and without the attacker's deposit)
    vm.stopPrank();
}
```

The test is reproducible on a mainnet fork by selecting a block immediately after a stETH rebase where `oracle.rsETHPrice()` has not yet been updated.
