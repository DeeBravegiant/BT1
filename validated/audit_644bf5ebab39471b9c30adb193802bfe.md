### Title
Relayer Loses Attached Deposit When Meta Transaction Inner Action Fails — (`runtime/runtime/src/actions.rs`)

---

### Summary

In NEAR's meta transaction (DelegateAction) flow, the relayer pays all costs upfront — including any NEAR deposit attached to inner actions. When the inner action receipt fails on the receiver's shard, the deposit refund is sent to the **sender** (Alice, the `predecessor_id` of the inner receipt), not to the relayer who actually funded it. An unprivileged user can exploit this by crafting a `DelegateAction` with a large deposit attached to an inner action that is designed to fail, causing the relayer to permanently lose the deposited NEAR.

---

### Finding Description

In `apply_delegate_action`, the inner receipt is constructed with `predecessor_id` set to `sender_id` (Alice), not the relayer: [1](#0-0) 

The code comment at lines 471–475 explicitly acknowledges this: [2](#0-1) 

When the inner receipt fails on Bob's shard, `refund_unspent_gas_and_deposits` in `runtime/runtime/src/lib.rs` calls `receipt.balance_refund_receiver()` to determine where to send the deposit refund: [3](#0-2) 

`balance_refund_receiver()` returns `predecessor_id()` when no `refund_to` override is set: [4](#0-3) 

For the inner receipt, `predecessor_id` is Alice — so the deposit refund goes to Alice, not the relayer. Gas refunds, by contrast, correctly go to the signer (the relayer): [5](#0-4) 

The `ActionReceiptV2` struct has a `refund_to` field that could redirect the refund, but it is not populated for inner receipts generated by `apply_delegate_action`: [6](#0-5) 

The docs confirm this behavior and acknowledge the financial incentive for Alice to exploit it: [7](#0-6) 

---

### Impact Explanation

The relayer's NEAR balance is reduced by the full deposit amount attached to the inner action, while Alice's NEAR balance is increased by the same amount. This is a direct, concrete NEAR token loss for the relayer — not a theoretical or performance-only issue. The corrupted protocol value is the relayer's account balance (a state entry in the trie).

---

### Likelihood Explanation

Alice can trigger this with a valid, well-formed `DelegateAction` submitted through the public RPC. The simplest attack: Alice crafts a `DelegateAction` with a large deposit attached to a function call on a contract she controls. She sends it to the relayer. After the relayer submits the outer transaction (paying the deposit), Alice redeploys her contract to panic on that method. The inner action fails, and the deposit refund flows to Alice. No privileged access is required. The relayer's only defense is off-chain trust and pre-submission checks, which cannot prevent race conditions.

---

### Recommendation

Populate the `refund_to` field in the inner receipt generated by `apply_delegate_action` with the relayer's account ID (the `signer_id` of the outer action receipt). This would cause `balance_refund_receiver()` to return the relayer instead of Alice when the inner action fails, matching the gas refund behavior. The `ActionReceiptV2.refund_to` field already exists for exactly this purpose. [8](#0-7) 

---

### Proof of Concept

1. Alice deploys a contract `alice-contract.near` with a method `drain` that currently succeeds.
2. Alice crafts a `DelegateAction` targeting `alice-contract.near::drain` with a deposit of 100 NEAR, signs it, and sends it to the relayer off-chain.
3. The relayer validates the action (the method exists and appears valid), wraps it in a transaction, and submits it — paying 100 NEAR as the deposit.
4. Before the transaction executes (within the same or next block), Alice redeploys `alice-contract.near` with a version of `drain` that panics immediately.
5. The inner receipt reaches `alice-contract.near`, the function call fails, and `refund_unspent_gas_and_deposits` issues a balance refund to `receipt.balance_refund_receiver()` — which is Alice (`predecessor_id` of the inner receipt).
6. Alice receives 100 NEAR. The relayer receives only the gas refund (a small fraction of the total cost). Net loss to the relayer: ~100 NEAR.

        match self.receipt() {
            ReceiptEnum::Action(_)
            | ReceiptEnum::Data(_)
            | ReceiptEnum::PromiseYield(_)
            | ReceiptEnum::PromiseResume(_)
            | ReceiptEnum::GlobalContractDistribution(_) => &None,
            ReceiptEnum::ActionV2(action_receipt_v2)
            | ReceiptEnum::PromiseYieldV2(action_receipt_v2) => &action_receipt_v2.refund_to,
        }
    }

    pub fn balance_refund_receiver(&self) -> &AccountId {
        self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
    }

```markdown
## Balance refunds in meta transactions

Unlike gas refunds, the protocol sends balance refunds to the predecessor
(a.k.a. sender) of the receipt. This makes sense, as we deposit the attached
balance to the receiver, who has to explicitly reattach a new balance to new
receipts they might spawn.

In the world of meta transactions, this assumption is also challenged. If an
inner action requires an attached balance (for example a transfer action) then
this balance is taken from the relayer.

The relayer can see what the cost will be before submitting the meta transaction
and agrees to pay for it, so nothing wrong so far. But what if the transaction
fails execution on Bob's shard? At this point, the predecessor is `Alice` and
therefore she receives the token balance refunded, not the relayer. This is
something relayer implementations must be aware of since there is a financial
incentive for Alice to submit meta transactions that have high balances attached
but will fail on Bob's shard.
```
