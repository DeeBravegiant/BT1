Audit Report

## Title
Router's Stranded Native ETH Consumed to Settle Subsequent WETH-Input Swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` uses `address(this).balance` unconditionally when settling a WETH-input swap, with no check that the ETH originated from the current caller's `msg.value`. Because `exactInputSingle`, `exactOutputSingle`, `exactInput`, and `exactOutput` are all `payable` and impose no requirement that `msg.value` equals `amountIn`, any ETH left in the router by a prior user who omitted `refundETH` is silently consumed to pay for a subsequent attacker's swap. The attacker receives real pool output tokens while contributing zero ETH or WETH.

## Finding Description

In `pay()`, the WETH-input branch reads the router's full native balance and, if it covers the required `value`, wraps and forwards it without referencing the `payer` argument at all:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();   // uses router's own ETH
        IERC20(WETH).safeTransfer(recipient, value);
    }
``` [1](#0-0) 

The `payer` identity stored in transient storage (set to `msg.sender` at entry) is completely ignored in this branch. The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` in any of the four `payable` swap entry points. [2](#0-1) 

`exactInputSingle` stores `msg.sender` as payer and immediately calls `pool.swap`, which re-enters via `metricOmmSwapCallback` → `_justPayCallback` → `pay()`. There is no assertion that `msg.value >= params.amountIn` when `tokenIn == WETH`: [3](#0-2) 

The pool's `IncorrectDelta` guard only verifies that its own balance increased by the required amount; it is indifferent to whether the WETH was wrapped from the attacker's ETH or from a prior user's stranded ETH: [4](#0-3) 

The test suite documents the intended safe pattern — pairing every over-sent ETH swap with an explicit `refundETH` call in a multicall — but this is not enforced at the contract level: [5](#0-4) 

## Impact Explanation

A prior user's stranded ETH is consumed to settle the attacker's WETH-input swap. The attacker receives real pool output tokens (token0 or token1) without transferring any asset. The victim permanently loses the ETH they left in the router. Loss magnitude equals `min(router.balance, attacker's amountIn)` and is bounded only by how much ETH the victim forgot to reclaim. This is a direct loss of user principal, meeting the Critical/High threshold under the allowed impact gate.

## Likelihood Explanation

Users routinely over-send ETH to guarantee a swap succeeds and rely on `refundETH` to recover the surplus. Omitting `refundETH` is a realistic user error, especially when calling `exactInputSingle` directly rather than via multicall. The router's ETH balance is publicly readable on-chain, so an attacker can monitor it and exploit it atomically in the next block. No special privileges are required; any unprivileged address can call `exactInputSingle{value: 0}` with `tokenIn = WETH`.

## Recommendation

Track the ETH that belongs to the current call context. Inside `pay()`, when `token == WETH`, only consume native ETH up to the `msg.value` of the originating call (passed as a parameter or stored in transient storage at entry). Any shortfall must be pulled from the payer via `safeTransferFrom`. Alternatively, enforce that `address(this).balance == 0` at the start of every non-multicall payable entry point, or require callers to always pair WETH-input swaps with `refundETH` in a multicall and enforce this as a hard invariant at the contract level.

## Proof of Concept

```
1. Victim calls router.exactInputSingle{value: 10_000}(tokenIn=WETH, amountIn=1_000, ...)
   → swap uses 1_000 ETH worth of WETH; 9_000 ETH remains in router.
   → Victim does NOT call refundETH.

2. Attacker calls router.exactInputSingle{value: 0}(tokenIn=WETH, amountIn=9_000, ...)
   → Pool executes swap, calls metricOmmSwapCallback on router.
   → Router calls _justPayCallback → pay(WETH, attacker, pool, 9_000).
   → nativeBalance (9_000) >= value (9_000) → branch at line 75 taken.
   → payer (attacker) is never referenced; router wraps its own 9_000 ETH.
   → WETH transferred to pool; pool's IncorrectDelta check passes.
   → Attacker receives pool output tokens.

Result: Attacker paid 0 ETH/WETH; victim lost 9_000 ETH.
```

Foundry test plan: deploy router with WETH, fund victim with 10_000 wei, call `exactInputSingle{value: 10_000}(amountIn=1_000)` without `refundETH`, assert `address(router).balance == 9_000`, then call `exactInputSingle{value: 0}(amountIn=9_000)` as attacker with no WETH approval, assert attacker received output tokens and `address(router).balance == 0`.

    _checkDeadline(params.deadline);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);

      .swap(

        priceLimitX64,
        "",
        params.extensionData
      );

    amountOut = MetricOmmSwapInputs.int128ToUint128(out);

    _clearExpectedCallbackPool();
  }
