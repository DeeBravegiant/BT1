Audit Report

## Title
Stale Cross-Chain rsETH/ETH Rate Used in L2 Pool Deposits Without Staleness Check — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver` stores `lastUpdated` on every rate update but `getRate()` returns `rate` unconditionally without checking it. All L2 pool contracts (`RSETHPoolV2NBA`, `RSETHPoolV2`, `RSETHPoolV3`, etc.) delegate pricing entirely to this call and use the returned rate to compute wrsETH mint amounts. If the off-chain bot or LayerZero delivery fails, a depositor can exploit the stale (lower) rate to receive more wrsETH than the current backing justifies, diluting accrued yield for existing holders.

## Finding Description
`CrossChainRateReceiver.lzReceive()` records both `rate` and `lastUpdated` on every update:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L95-97
rate = _rate;
lastUpdated = block.timestamp;
```

However, `getRate()` returns `rate` unconditionally with no reference to `lastUpdated`:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L102-105
function getRate() external view returns (uint256) {
    return rate;
}
```

Every L2 pool contract delegates pricing entirely to this call. `RSETHPoolV2NBA.viewSwapRsETHAmountAndFee()` uses the returned rate directly:

```solidity
// contracts/pools/RSETHPoolV2NBA.sol L124-133
uint256 rsETHToETHrate = getRate();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The same pattern is present in `RSETHPoolV3.viewSwapRsETHAmountAndFee()` (L299-308). The rate update path requires two off-chain dependencies: an external bot calling `CrossChainRateProvider.updateRate()` (L85-101) and LayerZero delivering the message. Either can fail. By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L26-37) already implements staleness guards (`answeredInRound < roundID`, `timestamp == 0`), demonstrating the protocol is aware of the pattern but did not apply it to `CrossChainRateReceiver`.

## Impact Explanation
rsETH accrues staking yield continuously, so its ETH value increases over time. When the L2 receiver's `rate` is stale and lower than the true current value, the formula `rsETHAmount = amountAfterFee * 1e18 / staleRate` mints more wrsETH per ETH than the actual backing justifies. Once the rate updates, the attacker's wrsETH is fully backed at the correct rate, having permanently extracted accrued yield from prior depositors. This constitutes **theft of unclaimed yield — High severity** per the allowed impact scope.

## Likelihood Explanation
The exploit requires no privileged access. Any unprivileged depositor can call `RSETHPoolV2NBA.deposit()` (or equivalent pool functions) at any time. The attacker only needs to monitor `CrossChainRateReceiver.lastUpdated` on-chain and compare it against the live mainnet `LRTOracle.rsETHPrice`. rsETH yield accrues every Ethereum epoch (~6.4 minutes), so even a few hours of bot downtime, gas exhaustion, or LayerZero congestion creates a measurable and exploitable gap. The attack is repeatable across all deployed L2 chains (Arbitrum, Optimism, Polygon zkEVM, Blast, Mode, Scroll).

## Recommendation
Add a configurable `maxStaleness` threshold to `CrossChainRateReceiver` and revert in `getRate()` if the rate is stale:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

error StaleRate();

function getRate() external view returns (uint256) {
    if (block.timestamp - lastUpdated > maxStaleness) revert StaleRate();
    return rate;
}
```

This mirrors the staleness guard already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L30-31).

## Proof of Concept
1. Mainnet `LRTOracle.rsETHPrice` is `1.05e18` (updated regularly on L1).
2. Off-chain bot fails to call `CrossChainRateProvider.updateRate()` for 12 hours. `CrossChainRateReceiver.rate` on Arbitrum remains at `1.03e18`.
3. Attacker calls `RSETHPoolV2NBA.deposit{value: 100 ether}("")`.
4. `viewSwapRsETHAmountAndFee(100e18)` computes (assuming 0 fee): `rsETHAmount = 100e18 * 1e18 / 1.03e18 ≈ 97.09 wrsETH`.
5. Correct amount at live rate: `100e18 * 1e18 / 1.05e18 ≈ 95.24 wrsETH`.
6. Attacker receives `≈1.85 excess wrsETH` (~1.9% over-mint) at the expense of existing holders.
7. When the bot resumes and rate updates to `1.05e18`, the attacker's wrsETH is fully backed, having extracted yield from prior depositors permanently.

**Foundry fork test plan**: Fork an L2 (e.g., Arbitrum) at a block where `RSETHRateReceiver.lastUpdated` is recent. Warp `block.timestamp` forward by 12+ hours without triggering `lzReceive`. Call `RSETHPoolV2NBA.deposit` and assert the minted wrsETH amount exceeds what the current mainnet rate would justify, confirming the over-mint.
