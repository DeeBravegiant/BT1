### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies only that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the MPC root key. It never checks that `response.payload_hash` is actually the canonical hash of `ForeignTxSignPayload::V1 { request: request.request, values: <observed_values> }`. A single Byzantine attested participant (below signing threshold) can therefore replay a legitimately-obtained MPC signature from one foreign-tx verification request to resolve a completely different pending request, delivering a forged `payload_hash` to every waiting caller.

### Finding Description

In `respond_verify_foreign_tx` the contract performs:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // fully attacker-controlled
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // root public key (zero-tweak path)
)
.is_ok()
``` [1](#0-0) 

The `payload_hash` field comes entirely from the caller-supplied `response` argument; the contract never recomputes it from `request.request` and the extracted values. Compare this with `respond`, which derives the expected public key from `request.tweak` and verifies the signature against the payload stored inside the `SignatureRequest` itself — the payload is part of the request key, not the response. [2](#0-1) 

The node-side code confirms that foreign-tx signatures are produced with a **zero tweak** (root key):

```rust
tweak: Tweak::new([0u8; 32]),
``` [3](#0-2) 

Because the tweak is zero, any valid `(payload_hash, signature)` pair produced for *any* foreign-tx request is a valid signature under the root key and will pass the contract's check when submitted against *any other* pending request.

The `VerifyForeignTransactionRequest` key stored in `pending_verify_foreign_tx_requests` contains only `(domain_id, payload_version, ForeignChainRpcRequest)` — it does not include the requester's account ID: [4](#0-3) 

`args_into_verify_foreign_tx_request` confirms no caller binding is added at enqueue time: [5](#0-4) 

### Impact Explanation

A single malicious attested participant (strictly below the signing threshold) can:

1. Submit `verify_foreign_transaction` for a foreign-chain transaction **T_Y** they control (e.g., a small deposit).
2. Receive the legitimate MPC response `{ payload_hash: H_Y, signature: σ_root(H_Y) }` where `H_Y = SHA-256(borsh(ForeignTxSignPayload::V1 { request_Y, values_Y }))`.
3. Observe a bridge's pending `verify_foreign_transaction` for a high-value transaction **T_X** in the NEAR mempool.
4. Front-run the honest leader node's `respond_verify_foreign_tx(request_X, honest_response)` by submitting `respond_verify_foreign_tx(request_X, { payload_hash: H_Y, signature: σ_root(H_Y) })`.
5. The contract accepts the call (signature is valid over `H_Y` under the root key; `request_X` is in the pending map) and drains all queued yields for `request_X` with the forged response. [6](#0-5) 

Every caller waiting on `request_X` — including bridge contracts — receives `{ payload_hash: H_Y, signature: σ_root(H_Y) }`. A bridge that only checks signature validity (not that `payload_hash` reconstructs from the expected transaction) will accept this as proof that **T_X** was verified with the extracted values of **T_Y**, enabling forged attestation and potential double-spend or invalid inbound bridge execution.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

The attacker must be an attested participant (one Byzantine node below threshold). They need only one prior legitimate `verify_foreign_transaction` response for any transaction they control on the target foreign chain — a realistic precondition for a bridge operator who is also an MPC node. The NEAR mempool is observable by all participants, making front-running straightforward. No threshold collusion is required.

### Recommendation

Recompute `payload_hash` inside `respond_verify_foreign_tx` from the request content rather than trusting the caller-supplied value. Specifically, the contract should:

1. Require the caller to supply the full `ForeignTxSignPayload` (or at minimum the `extracted_values`).
2. Compute `expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values }))` on-chain.
3. Assert `response.payload_hash == expected_hash` before accepting the signature.

Alternatively, bind the `payload_hash` into the pending-request key at enqueue time (analogous to how `SignatureRequest` embeds the payload), so a response carrying a mismatched hash simply fails the map lookup.

### Proof of Concept

```rust
// Attacker is an attested participant.
// Step 1: submit a legitimate verify_foreign_transaction for tx_Y (attacker-controlled).
contract.verify_foreign_transaction(request_args_Y);
// ... MPC network responds with honest response_Y = { payload_hash: H_Y, sig: σ(H_Y) }

// Step 2: observe bridge's pending request for high-value tx_X.
// Step 3: front-run the honest respond call:
contract.respond_verify_foreign_tx(
    request_X,                          // the bridge's pending request
    VerifyForeignTransactionResponse {
        payload_hash: response_Y.payload_hash,  // H_Y — wrong transaction
        signature:    response_Y.signature,     // σ_root(H_Y) — valid under root key
    },
);
// Contract accepts: sig is valid over payload_hash under root key.
// Bridge receives forged attestation for tx_X claiming values from tx_Y.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
