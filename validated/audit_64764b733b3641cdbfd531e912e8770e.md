### Title
Missing Contract-Identity Binding in `ForeignTxSignPayloadV1` Enables Cross-Contract Signature Replay - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`ForeignTxSignPayloadV1.compute_msg_hash()` produces a hash that contains only the foreign-chain request parameters and extracted values. It omits any NEAR-side binding — no MPC contract account ID, no domain ID, no NEAR network identifier. Because the MPC network is explicitly designed to serve multiple bridge callers with the same public key, a `VerifyForeignTransactionResponse` produced for one bridge contract is cryptographically valid for any other bridge contract that uses the same MPC domain key and presents the same foreign-chain request parameters.

---

### Finding Description

The canonical signed payload is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
``` [1](#0-0) 

The resulting `msg_hash = SHA-256(borsh(ForeignTxSignPayload))` binds only to `(chain_type, tx_id, extractors, finality, extracted_values)`. It does **not** bind to:

- The NEAR MPC contract account ID (`env::current_account_id()`)
- The domain ID that selected the signing key
- The NEAR network identifier (mainnet vs. testnet)

The client-side verifier in the SDK mirrors this omission:

```rust
impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });
        let expected_payload_hash = expected_payload.compute_msg_hash()...;
        let payload_is_correct = expected_payload_hash == response.payload_hash;
        ...
        near_mpc_signature_verifier::verify_ecdsa_signature(
            k256_signature,
            &expected_payload_hash,
            secp256k1_public_key,
        )
    }
}
``` [2](#0-1) 

The verifier checks only that the signature is valid over the payload hash using the MPC public key. It does not check which contract, domain, or network the signature was produced for.

On the contract side, `respond_verify_foreign_tx` verifies the signature against `response.payload_hash` and the domain's root public key (zero tweak), with no contract-identity check: [3](#0-2) 

The node-side `build_signature_request` confirms the zero tweak for all foreign-tx requests: [4](#0-3)

### Impact Explanation

The MPC network is explicitly designed to serve multiple bridge callers. Any bridge contract that calls `verify_foreign_transaction` on the MPC contract and receives a `VerifyForeignTransactionResponse` produces a signature that is valid for **any other bridge contract** using the same MPC domain public key, provided the foreign-chain request parameters and extracted values are identical.

**Attack scenario — cross-bridge double-spend:**

1. Bridge A (e.g., `omnibridge-v1.near`) calls `verify_foreign_transaction` for Bitcoin tx X with extractor `BlockHash`, receiving response `{payload_hash: H, signature: S}`.
2. Bridge A uses this to unlock funds for its user.
3. An attacker presents the same `{H, S}` to Bridge B (`omnibridge-v2.near`), which uses the same MPC domain public key.
4. Bridge B's `ForeignChainSignatureVerifier.verify_signature()` accepts the response — the signature is valid over the same `H = SHA-256(borsh(ForeignTxSignPayload{Bitcoin(X), [block_hash_Y]}))`.
5. Bridge B unlocks funds for the attacker without the MPC nodes ever verifying the transaction for Bridge B.

This is a direct analog to the ERC20Permit chainID omission: the signed message does not account for which contract instance (or NEAR network) it was produced for, enabling replay across any contract that trusts the same MPC key.

**Allowed impact match:** High — forged foreign-chain verification / participant authorization bypass causing invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

The MPC network is designed to serve multiple callers simultaneously. The `verify_foreign_transaction` API is a public, permissionless endpoint. Any bridge contract that integrates the MPC SDK and uses the same domain public key is a valid replay target. An attacker only needs to:

1. Observe a valid `VerifyForeignTransactionResponse` on-chain (it is returned as a NEAR promise result, publicly visible).
2. Present it to a second bridge contract that accepts the same foreign-chain event.

No threshold collusion, no key compromise, and no privileged access is required. The attacker's only constraint is finding two bridge contracts that both react to the same foreign-chain transaction using the same MPC public key — which is the intended multi-caller use case.

---

### Recommendation

**Short term:** Include the MPC contract account ID and domain ID in the signed payload. Extend `ForeignTxSignPayloadV1` with a `contract_id: AccountId` and `domain_id: DomainId` field, populated by the contract at request time (not supplied by the caller), so the hash binds to the specific contract instance and domain:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub contract_id: AccountId,   // env::current_account_id() at request time
    pub domain_id: DomainId,
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

**Long term:** Document the replay-binding properties of every signature schema produced by the MPC network. Require that all future payload versions include a contract-identity field. Audit all SDK consumers (`ForeignChainSignatureVerifier`) to ensure they reconstruct the full binding context, including contract ID and domain ID, before accepting a response.

### Proof of Concept

**Step 1 — Observe a valid response for Bridge A:**

Bridge A calls `verify_foreign_transaction` on the MPC contract for Bitcoin tx `[0xbb; 32]` with extractor `BlockHash`. The MPC nodes verify and call `respond_verify_foreign_tx`. The contract resumes and returns:

```json
{
  "payload_hash": "<H>",
  "signature": { "scheme": "Secp256k1", "big_r": "...", "s": "...", "recovery_id": 0 }
}
```

`H = SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: Bitcoin([0xbb;32], 1, [BlockHash]), values: [block_hash_Y] })))` [5](#0-4) 

**Step 2 — Replay to Bridge B:**

Bridge B constructs a `ForeignChainSignatureVerifier` with the same request and expected values, then calls:

```rust
verifier.verify_signature(&response_from_bridge_a, &mpc_public_key)
``` [6](#0-5) 

**Step 3 — Verification passes:**

`expected_payload_hash == response.payload_hash` is true (same request, same values, same hash function). The ECDSA signature `S` is valid over `H` under the MPC domain public key. `verify_signature` returns `Ok(())`. Bridge B accepts the foreign-chain event and executes its bridge logic — without the MPC nodes ever verifying the transaction for Bridge B. [7](#0-6)

        response: &VerifyForeignTransactionResponse,

        let payload_is_correct = expected_payload_hash == response.payload_hash;
