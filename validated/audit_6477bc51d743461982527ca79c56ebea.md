Audit Report

## Title
ETH Deposit Limit Check Omits `amount`, Allowing Deposits Beyond Protocol Cap - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison: for ERC-20 assets it adds the incoming `amount` to `totalAssetDeposits` before comparing against the cap, but for ETH it drops `amount` entirely. As a result, any depositor can push ETH holdings arbitrarily above the configured limit in a single `depositETH` call, as long as the running total has not yet strictly exceeded the cap.

## Finding Description
The root cause is in `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount absent
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
``` [1](#0-0) 

The ETH branch returns `false` (limit not exceeded) whenever `totalAssetDeposits <= depositLimit`, regardless of how large `amount` is. `_beforeDeposit` (lines 648–670) calls this function and only reverts on `true`, so the guard is bypassed for any ETH deposit made while the running total is at or below the cap. [2](#0-1) 

`getAssetCurrentLimit` (lines 402–409) correctly computes remaining capacity and would already return `0` at the limit, but it is not used as the enforcement guard. [3](#0-2) 

## Impact Explanation
The deposit limit is the protocol's primary mechanism for capping total ETH exposure relative to EigenLayer strategy capacity. Bypassing it allows over-minting of rsETH beyond the intended ceiling. Excess ETH that cannot be deployed into EigenLayer strategies sits idle, earning no restaking yield, so rsETH holders receive lower returns than the protocol promises. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The entry point is the public, permissionless `depositETH()` function. No special role or precondition is required beyond holding ETH. The vulnerable condition (`totalAssetDeposits <= depositLimit`) is the normal operating state of the protocol for most of its lifetime. Any depositor can trigger this at any time, making likelihood **High**.

## Recommendation
Apply the same `+ amount` pattern used for ERC-20 assets to the ETH branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100_000 ether`.
2. Protocol accumulates exactly `100_000 ether`; `getAssetCurrentLimit(ETH)` returns `0`.
3. Attacker calls `depositETH{value: 50_000 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 50_000 ether)`:
   - `totalAssetDeposits = 100_000 ether`
   - Check: `100_000 ether > 100_000 ether` → `false` → guard passes.
5. rsETH is minted for `50_000 ether` beyond the cap.
6. Protocol holds `150_000 ether` against a `100_000 ether` limit; the excess `50_000 ether` cannot be deployed into EigenLayer, suppressing yield for all rsETH holders.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock config setting `depositLimitByAsset[ETH_TOKEN] = 100_000 ether`. Seed `totalAssetDeposits` to exactly `100_000 ether`. Call `depositETH{value: 50_000 ether}` from an unprivileged address and assert it does not revert and that `getTotalAssetDeposits(ETH_TOKEN)` returns `150_000 ether`, confirming the limit was bypassed.
