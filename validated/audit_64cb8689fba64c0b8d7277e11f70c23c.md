Audit Report

## Title
Excess ETH Permanently Locked in `updateRate()` Due to Missing Refund and No Recovery Path - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary

`MultiChainRateProvider.updateRate()` is an unrestricted `external payable` function that forwards exactly `estimatedFee` per receiver to the LayerZero endpoint but neither enforces `msg.value == totalFee` nor refunds any surplus. The contract inherits only `Ownable` and `ReentrancyGuard` and contains no `receive()`, `withdraw()`, or sweep function, so any ETH sent above the consumed total is permanently locked with no recovery path for any party.

## Finding Description

`updateRate()` is callable by any address with no access control: [1](#0-0) 

Inside the loop, `estimateFees` is queried at execution time and exactly that amount is forwarded per receiver: [2](#0-1) 

There is no pre-loop accumulation check (`require(msg.value == totalFee)`), no post-loop refund (`msg.sender.call{value: msg.value - consumed}("")`), and the contract body contains no ETH recovery mechanism whatsoever: [3](#0-2) 

`RSETHMultiChainRateProvider` inherits this behavior directly and adds no override or guard: [4](#0-3) 

By contrast, every other payable bridge entry point in the codebase enforces exact-fee equality before proceeding: [5](#0-4) 

The `estimateTotalFee()` view helper exists but is a separate call; between that call and the `updateRate()` execution, the owner may add or remove receivers, making the off-chain estimate stale and the correct value non-trivially computable atomically: [6](#0-5) 

## Impact Explanation

Any ETH sent in excess of the sum of per-receiver `estimatedFee` values at execution time is irrecoverably locked in `MultiChainRateProvider` / `RSETHMultiChainRateProvider`. There is no `receive()` fallback, no `withdraw()`, and no sweep function anywhere in the inheritance chain. This constitutes **Critical — Permanent freezing of funds**: the locked ETH belongs to the caller and can never be retrieved by any party.

## Likelihood Explanation

`updateRate()` carries no access control, so any EOA, keeper bot, or integrating contract can call it. Callers must estimate the correct total fee off-chain by summing `estimateFees()` across all receivers. Because the receiver list can change between the off-chain query and the on-chain execution, and because defensive over-estimation is a standard pattern for gas/fee payments, callers will routinely send more ETH than consumed. The loss is repeatable on every call where `msg.value > totalConsumedFee`.

## Recommendation

Add a pre-loop accumulation and equality check mirroring the pattern used in `L1Vault.bridgeRsETHToL2`:

```solidity
uint256 totalFee;
for (uint256 i; i < rateReceiversLength;) {
    uint16 dstChainId = uint16(rateReceivers[i]._chainId);
    (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
        .estimateFees(dstChainId, address(this), _payload, false, bytes(""));
    totalFee += estimatedFee;
    unchecked { ++i; }
}
require(msg.value == totalFee, "Incorrect ETH amount");
// then execute sends in a second loop
```

Alternatively, refund `msg.value - totalConsumed` to `msg.sender` after the send loop.

## Proof of Concept

1. Owner configures two `rateReceivers` on `RSETHMultiChainRateProvider`.
2. Caller queries `estimateTotalFee()` off-chain; result is `0.01 ETH`.
3. Caller calls `updateRate{ value: 0.02 ETH }()` (over-estimating defensively).
4. The loop consumes exactly `0.01 ETH` across the two LayerZero `send` calls.
5. The remaining `0.01 ETH` remains in the contract balance.
6. No function exists to withdraw it; the `0.01 ETH` is permanently locked.
