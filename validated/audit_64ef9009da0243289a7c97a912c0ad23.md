### Title
Single-Provider Verification Bypass via Transient-Error Suppression in `FanOut` — (`File: crates/foreign-chain-inspector/src/lib.rs`)

---

### Summary

The `FanOut` inspector, which fans out foreign-chain verification queries to multiple RPC providers, silently discards all transient errors and accepts a single successful response as the final verdict — with no minimum quorum enforcement. An adversary who controls or can compromise one whitelisted RPC provider, and can cause all other providers to return transient errors (rate-limit responses, transport timeouts, 5xx), can forge a foreign-chain verification result and obtain a valid MPC threshold signature over a fabricated observation.

---

### Finding Description

`FanOut::extract()` in `crates/foreign-chain-inspector/src/lib.rs` queries all configured inspectors concurrently and partitions results into three buckets: successes, non-transient errors, and transient errors.

```rust
// crates/foreign-chain-inspector/src/lib.rs  lines 96–116
for (idx, result) in join_set.join_all().await {
    match result {
        Ok(values) => successes.push((idx, values)),
        Err(err) if err.is_transient() => {
            tracing::warn!(...);
            first_transient_error.get_or_insert(err);  // silently dropped
        }
        Err(err) => {
            non_transient_errors.push((idx, err));
        }
    }
}
```

Transient errors are classified by `is_transient()`:

```rust
// lines 266–274
pub fn is_transient(&self) -> bool {
    matches!(
        self,
        Self::ClientError(_)
            | Self::RpcRequestFailed(_)   // transport error, timeout, rate-limit, 5xx
            | Self::NotFinalized
            | Self::NotEnoughBlockConfirmations { .. }
    )
}
```

After partitioning, if any successes exist and no non-transient errors exist, the first success is returned unconditionally — regardless of how many providers failed transiently:

```rust
// lines 130–142
if let Some(first_values) = successes.first() {
    let all_successes_agree = successes.iter().all(|(_, v)| v == &first_values.1);
    if !all_successes_agree { return Err(InspectorResponseMismatch); }
    let (_, first) = successes.into_iter().next().expect("checked non-empty");
    return Ok(first);   // ← single provider's result accepted
}
```

The integration test `fan_out__should_tolerate_transient_when_only_one_inspector_succeeds` explicitly encodes this as intended behavior: two providers fail with `NotFinalized`, one succeeds, and the fan-out returns that single result.

The design documentation explicitly states the intended invariant that is violated here:

> "Every node partaking in a foreign signature verification request queries all its locally configured RPC providers for the relevant chain, independently of other nodes. **A quorum of those RPC providers must agree on the verification; if fewer than the quorum agree, the node errors out that foreign-tx validation and does not retry the request.**"
> — `docs/design/allowing-per-node-foreign-chain-rpc-configuration.md`, line 27

The `ChainEntry.quorum` field is voted in on-chain alongside the provider list, but `FanOut` never reads or enforces it. The design document itself acknowledges this gap:

> "This sub-quorum outcome must be terminal — the leader must not re-attempt the request. **Implementation requirement, not current behavior.**"
> — `docs/design/calculating-supported-foreign-chains.md`, line 62–63

**Attack path:**

1. Attacker controls or compromises one whitelisted RPC provider for a target chain (e.g., Ethereum, Starknet).
2. Attacker submits a `verify_foreign_transaction` request referencing a transaction that did not succeed on the foreign chain (or a fabricated transaction).
3. Attacker floods all other whitelisted providers for that chain with requests, triggering HTTP 429 rate-limit responses. `RpcRequestFailed` is classified as transient, so `FanOut` silently discards those results.
4. Only the attacker-controlled provider responds with `Ok(forged_extracted_values)`.
5. `FanOut` returns the forged values as the verified observation.
6. The MPC node produces a signature share over `(request, forged_values, observed_at)`.
7. Once a threshold of nodes are similarly deceived (or the attacker controls enough providers across nodes), the contract's `respond_verify_foreign_tx` is called with a valid threshold signature over a fabricated observation, enabling unauthorized bridge execution.

---

### Impact Explanation

A successful attack produces a valid MPC threshold signature over a forged foreign-chain observation. For the Omnibridge inbound flow, this means the MPC network attests that a foreign deposit finalized when it did not, enabling the attacker to mint bridged assets without a corresponding deposit — a direct theft of funds from the bridge. This matches the allowed impact: **forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

---

### Likelihood Explanation

- **Provider compromise**: RPC providers are whitelisted by threshold vote, but a provider that was legitimately voted in can later be compromised (key leak, supply-chain attack on the provider's infrastructure). The attacker needs only one such provider per node.
- **Transient suppression**: Triggering HTTP 429 rate-limit responses on public RPC endpoints is trivially achievable by flooding them with requests. `RpcRequestFailed` (which covers 429) is classified as transient and silently discarded.
- **Cross-node coordination**: Each node independently selects providers via deterministic hashing. An attacker who controls one provider and can rate-limit others can repeat the attack across enough nodes to reach the signing threshold.

Likelihood is **moderate**: it requires a compromised whitelisted provider, but the transient-suppression step (rate limiting) is low-cost and the quorum gap is a structural implementation deficiency, not a configuration error.

---

### Recommendation

Enforce the on-chain `ChainEntry.quorum` value inside `FanOut::extract()`. After collecting all results, count the number of providers that returned the same successful value. If that count is below `quorum`, return a non-transient sub-quorum error instead of accepting the result. Concretely:

```rust
// Proposed change in FanOut::extract()
if successes.len() < required_quorum {
    return Err(ForeignChainInspectionError::SubQuorumResult {
        got: successes.len(),
        required: required_quorum,
    });
}
```

`SubQuorumResult` should be classified as **non-transient** so the leader does not retry the request (per the design requirement in `docs/design/calculating-supported-foreign-chains.md` line 62). The `FanOut` struct should carry the quorum value, read from the on-chain `ChainEntry.quorum` at construction time.

### Proof of Concept

The existing test `fan_out__should_tolerate_transient_when_only_one_inspector_succeeds` in `crates/foreign-chain-inspector/tests/fanout.rs` lines 503–515 directly demonstrates the vulnerable behavior:

```rust
async fn fan_out__should_tolerate_transient_when_only_one_inspector_succeeds() {
    let succeeding = mock_returning(ok(vec![99]));
    let make_transient = || mock_returning(err(|| ForeignChainInspectionError::NotFinalized));
    let fan_out = fan_out_of(vec![make_transient(), succeeding, make_transient()]);

    let result = fan_out.extract((), (), vec![]).await;

    assert_eq!(result.unwrap(), vec![99]);  // single provider's result accepted
}
```

Replace `NotFinalized` with `RpcRequestFailed` (HTTP 429) and `ok(vec![99])` with a forged extracted value to reproduce the attack scenario. The fan-out returns the forged value with no quorum check. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

```rust
/// Combines multiple inspectors that target the same chain into a single inspector.
///
/// All inner inspectors are queried concurrently. The fan-out treats every
/// non-transient outcome (success or non-transient error, see
/// [`ForeignChainInspectionError::is_transient`]) as a substantive verdict that must
/// agree across inspectors. Transient errors (network issues, finality not yet reached,
/// etc.) are tolerated so that a single unavailable RPC does not take the whole node
/// out of signing.
///
/// Outcomes:
/// * All substantive verdicts are `Ok` with the same extracted values → returns those values.
/// * All substantive verdicts are non-transient errors of the same variant → returns one of
///   them (e.g. all inspectors agree the transaction failed).
/// * Substantive verdicts disagree (`Ok` vs. non-transient error, two different non-transient
///   error variants, or two different success values) → returns
///   [`ForeignChainInspectionError::InspectorResponseMismatch`].
/// * Every inspector returned a transient error → the first such error is propagated.
///
/// Variant-level comparison is used for non-transient errors, so inspectors that all report
/// the same failure mode (e.g. `NonCanonicalBlock`) are considered to agree even if the
/// inner fields differ.
```
