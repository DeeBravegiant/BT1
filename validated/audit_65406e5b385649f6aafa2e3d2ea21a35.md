Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked` is zero, while `rewardPerToken()` short-circuits and returns the stale `rewardPerTokenStored` in that case. Any rewards emitted during a zero-staked interval are silently skipped and permanently locked in the contract, with no mechanism for recovery or redistribution.

## Finding Description
The root cause is a mismatch between `rewardPerToken()` and `updateReward`:

`rewardPerToken()` (L408–414) returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`, correctly avoiding a division-by-zero. However, `updateReward` (L232–241) always executes `updatedAt = lastTimeRewardApplicable()` unconditionally. When `totalKernelStaked` is zero, this advances the accounting timestamp without accumulating any rewards for the elapsed interval.

The exploit path:
1. An active reward window is running (`rewardRate > 0`, `block.timestamp < finishAt`).
2. The last staker calls `initiateWithdrawal()`. The `updateReward` modifier fires first, correctly snapshotting accrued rewards up to that point. Then `totalKernelStaked -= _amount` (L326) drops it to zero.
3. Time passes. `rewardRate` continues to emit rewards, but `rewardPerTokenStored` is frozen because `rewardPerToken()` short-circuits on every call.
4. A new staker calls `stake()`. `updateReward` fires before `totalKernelStaked` is incremented (L284–285). At this moment `totalKernelStaked` is still 0, so `rewardPerToken()` returns the stale stored value, and `updatedAt` is advanced to the present. The entire reward budget for the idle interval is discarded.
5. `totalKernelStaked` is then incremented, but the skipped interval can never be recovered.

The `notifyRewardAmount` guard at L570 (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new reward period with zero staked; it does not prevent stakers from exiting after a period has started. The contract's own NatSpec (L18–22) acknowledges this behavior and states the mitigation is purely operational, with no on-chain enforcement preventing the last staker from withdrawing during an active window.

## Impact Explanation
Rewards emitted during the zero-staked interval are permanently locked in `KernelDepositPool`. No staker — present or future — can ever claim them. The reward tokens remain in the contract balance but are irrecoverable, as there is no admin sweep or rescue function for stranded reward tokens. This constitutes **permanent freezing of unclaimed yield**, matching the Medium severity tier in the allowed impact scope.

## Likelihood Explanation
Any unprivileged staker can trigger this by being the last to call `initiateWithdrawal()` during an active reward window. No special role, governance action, or coordination is required. The `withdrawalDelay` (up to 30 days) only delays the token transfer; `totalKernelStaked` is reduced immediately at `initiateWithdrawal()` time (L326), so the accounting damage occurs regardless of whether the withdrawal is ever claimed. Market conditions, a better yield opportunity, or deliberate griefing are all realistic triggers.

## Recommendation
Do not advance `updatedAt` when `totalKernelStaked` is zero. This preserves the unallocated interval so it is correctly distributed once staking resumes:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This is the standard fix applied in audited Synthetix forks and ensures that rewards emitted during a zero-staked interval are retroactively distributed to the next staker who resumes staking.

## Proof of Concept
Foundry test sequence:

1. Deploy `KernelDepositPool`. Admin calls `notifyRewardAmount(1000e18)` with `duration = 100 days`. `rewardRate = 10e18/day`. Alice stakes 1000 KERNEL — she is the only staker.
2. Warp 10 days. Alice calls `initiateWithdrawal(1000)`. `updateReward` snapshots 10 days × 10e18 = 100e18 rewards correctly. `totalKernelStaked` drops to 0. `updatedAt = block.timestamp`.
3. Warp 50 more days. Assert: `rewardPerToken()` returns the same stale value (no accumulation). Assert: `rewardRate * 50 days = 500e18` worth of rewards have been emitted but are unallocated.
4. Bob calls `stake(1000)`. `updateReward` fires: `totalKernelStaked == 0` → `rewardPerToken()` returns stale value; `updatedAt` advances 50 days. Then `totalKernelStaked = 1000`.
5. Assert: `earned(Bob) == 0` immediately after staking. Warp to `finishAt`. Assert: Bob earns only rewards from day 60 onward (≈ 400e18), not the 500e18 from the idle window. Assert: contract holds ≥ 500e18 reward tokens that can never be claimed by any address.
