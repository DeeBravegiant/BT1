### Title
`ForeignTxSignPayload::compute_msg_hash()` Omits Contract Address and Network Identifier, Enabling Cross-Network Signature Replay — (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`ForeignTxSignPayload::compute_msg_hash()` hashes only the foreign-chain request and extracted values. It does not bind the signature to the MPC contract address or the NEAR network identifier. This is the direct structural analog of the `LibTokenPermit::_buildDomainSeparator` bug: just as that function used a static `CHAIN_ID` constant instead of `block.chainid`, this function produces a hash that is identical across any NEAR deployment sharing the same MPC key — including both sides of a NEAR hard fork. A valid signature produced on one fork can be replayed verbatim on the other.

---

### Finding Description

`ForeignTxSignPayload::compute_msg_hash()` is defined as:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:1504-1509
impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
}
```

The serialized type is:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:1499-1502
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

Neither the MPC contract account ID (`v1.signer` on mainnet, `v1.signer-prod.testnet` on testnet) nor any NEAR network identifier is included. The hash is therefore identical for the same foreign-chain request and extracted values regardless of which NEAR network or contract deployment produced it.

`respond_verify_foreign_tx` verifies the signature only against `response.payload_hash` and the domain's stored public key:

```rust
// crates/contract/src/lib.rs:715-747
let domain = request.domain_id;
let public_key = self.public_key_extended(domain.0.into())?;
...
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

There is no check that binds the signature to the specific contract instance or NEAR network. If the same MPC key exists on two NEAR contexts (e.g., both sides of a fork), a signature accepted on one is accepted on the other.

---

### Impact Explanation

**Impact class: High — Cross-chain replay causing invalid bridge execution or double-spend conditions.**

When NEAR forks (as Ethereum did into ETH/ETHPoW), both forks inherit the same contract state, including the same MPC domain public key. The MPC nodes will follow one fork; the other fork retains the key but receives no new honest responses.

Attack path:
1. NEAR forks into fork-A (followed by MPC nodes) and fork-B (abandoned).
2. Both forks share the same `ForeignTx` domain public key.
3. Attacker submits `verify_foreign_transaction` on fork-A for Ethereum tx `X`.
4. MPC nodes sign `ForeignTxSignPayload::V1 { request: Ethereum(tx_X), values: [block_hash_Y] }` and call `respond_verify_foreign_tx` on fork-A.
5. Attacker observes the on-chain `respond_verify_foreign_tx` call and extracts `(payload_hash, signature)`.
6. Attacker replays the identical `respond_verify_foreign_tx` call on fork-B against any pending request for the same foreign transaction.
7. Fork-B's contract accepts the response — the key is the same, the payload hash is the same, the signature is valid.
8. Any bridge contract on fork-B that gates fund release on a valid MPC foreign-tx attestation releases funds a second time, producing a double-spend.

The `ForeignChainSignatureVerifier::verify_signature` in the SDK confirms this: it verifies only the payload hash and the signature against the MPC public key — no contract address, no network ID.

```rust
// crates/near-mpc-sdk/src/foreign_chain.rs:53-88

let payload_is_correct = expected_payload_hash == response.payload_hash;
...
near_mpc_signature_verifier::verify_ecdsa_signature(k256_signature, &expected_payload_hash, secp256k1_public_key)
```

---

### Likelihood Explanation

NEAR has not forked to date, so the immediate likelihood is low. However:
- The structural gap is present today and requires no code change to exploit once a fork occurs.
- The ETH/ETHPoW precedent shows that L1 forks do happen and are exploited within hours.
- The MPC contract is explicitly deployed on both mainnet (`v1.signer`) and testnet (`v1.signer-prod.testnet`); if a key migration error ever caused the same key material to appear on both networks, the replay window would be open immediately without any fork.

---

### Recommendation

Include the MPC contract account ID (or a static NEAR network label) in `ForeignTxSignPayloadV1` so that the signed hash is bound to a specific deployment:

```diff
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
+   /// MPC contract account ID — binds the signature to a specific deployment.
+   pub contract_id: AccountId,
}
```

Populate `contract_id` from `env::current_account_id()` inside `verify_foreign_transaction` before the payload is constructed and signed by the nodes. This mirrors the recommended fix in the external report (replacing a static constant with a dynamic runtime value) and is the standard EIP-712 domain-separator pattern applied to NEAR.

---

### Proof of Concept

```
1. NEAR hard-forks → fork-A (live) and fork-B (minority).
   Both forks: contract state identical, ForeignTx domain key = K.

2. Attacker on fork-A calls:
     verify_foreign_transaction({ request: Bitcoin(tx=[0xAA;32], ...), domain_id: 1, payload_version: V1 })

3. MPC nodes (fork-A only) compute:
     payload = ForeignTxSignPayload::V1 { request: Bitcoin(tx=[0xAA;32],...), values: [BlockHash([0xBB;32])] }
     msg_hash = SHA-256(borsh(payload))   // no contract_id, no network_id
     sig = ECDSA_sign(K_private, msg_hash)
   and call respond_verify_foreign_tx(request, { payload_hash: msg_hash, signature: sig }) on fork-A.

4. Attacker extracts (msg_hash, sig) from fork-A's transaction history.

5. On fork-B, attacker submits:
     verify_foreign_transaction(same args)   // creates a pending request
     respond_verify_foreign_tx(same request, { payload_hash: msg_hash, signature: sig })

6. Fork-B contract checks:
     verify_ecdsa_signature(sig, msg_hash, K_public)  → OK  (same key, same hash)
   → resolves the pending request, returns the attestation to the caller.

7. Bridge contract on fork-B receives a "valid" MPC attestation for Bitcoin tx [0xAA;32]
   and releases funds → double-spend.
