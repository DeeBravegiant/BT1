Audit Report

## Title
Block Stuffing Suppresses rsETH Price by Preventing `startCheckpoint`, Causing Dilution of Existing Holders — (`contracts/NodeDelegator.sol`, `contracts/LRTOracle.sol`)

## Summary
ETH that accumulates in the EigenPod via beacon-chain partial-withdrawal sweeps and execution-layer rewards is not reflected in `getEffectivePodShares()` until `startCheckpoint` is called and a checkpoint is fully proven. Because `startCheckpoint` is gated behind `onlyLRTOperator`, an attacker who block-stuffs can prevent the operator's transaction from landing, holding `rsETHPrice` below its true backing for the duration of the stuffing window. During this window, new depositors mint rsETH at the suppressed price, diluting existing holders.

## Finding Description

**Root cause — structural accounting gap in `getEffectivePodShares()`:**

`NodeDelegator.getEffectivePodShares()` returns only `stakedButUnverifiedNativeETH + withdrawableShare`:

```solidity
// contracts/NodeDelegator.sol L556-562
function getEffectivePodShares() external view override returns (uint256 ethStaked) {
    uint256 withdrawableShare =
        NodeDelegatorHelper.getWithdrawableShare(lrtConfig, IStrategy(lrtConfig.beaconChainETHStrategy()));
    return stakedButUnverifiedNativeETH + withdrawableShare;
}
```

`NodeDelegatorHelper.getWithdrawableShare` delegates to `DelegationManager.getWithdrawableShares()`, which reflects only finalized checkpoint shares — not raw ETH sitting in the pod awaiting a checkpoint. ETH that arrives via the EigenPod's `receive()` fallback (partial-withdrawal sweeps, EL rewards) is therefore invisible to this function until `startCheckpoint` → `verifyCheckpointProofs` → checkpoint finalization.

**Gap propagates to TVL and price:**

`LRTDepositPool.getETHDistributionData()` accumulates `getEffectivePodShares()` as `ethStakedInEigenLayer` across all NDCs:

```solidity
// contracts/LRTDepositPool.sol L487
ethStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getEffectivePodShares();
```

`LRTOracle._updateRsETHPrice()` derives `totalETHInProtocol` from this sum and computes:

```solidity
// contracts/LRTOracle.sol L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

Uncheckpointed pod ETH is absent from the numerator, so `rsETHPrice` is lower than the true per-token backing.

**`startCheckpoint` is operator-only; block stuffing blocks it:**

```solidity
// contracts/NodeDelegator.sol L259-261
function startCheckpoint(bool revertIfNoBalance) external onlyLRTOperator {
    eigenPod.startCheckpoint(revertIfNoBalance);
}
```

An attacker who fills every block with high-gas transactions prevents the operator's `startCheckpoint` call from landing. The operator has no on-chain bypass; they can only wait or use a private relay.

**Existing guards are insufficient:**

The `PriceAboveDailyThreshold` guard in `LRTOracle.sol` L252-266 only fires when the price *increases* beyond `pricePercentageLimit`. It does not prevent the price from being suppressed during the stuffing window, and it actually compounds the harm: when stuffing ends and the checkpoint is finalized, the price jump may exceed the threshold, requiring a manager call to push the price through and further delaying correct price discovery.

## Impact Explanation

**Low — Block stuffing.** While block stuffing is active, `rsETHPrice` is set below the true backing. New depositors minting rsETH at the suppressed price receive more rsETH per ETH than they should, diluting existing holders. When stuffing ends and the checkpoint is finalized, the price corrects upward, but the dilution to existing holders is permanent. This maps directly to the allowed impact "Low. Block stuffing."

## Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive but economically rational if the attacker can profit from minting rsETH at a suppressed price and redeeming after the price corrects. The structural gap is permanent and requires no special preconditions beyond ETH accruing to the pod, which happens continuously via beacon-chain rewards. The attack is repeatable as long as the accounting gap exists.

## Recommendation

1. **Include raw pod ETH in `getEffectivePodShares()`**: add `address(eigenPod).balance` (minus `withdrawableRestakedExecutionLayerGwei` already counted as shares) to the return value so uncheckpointed ETH is always reflected in TVL, eliminating the structural gap regardless of whether `startCheckpoint` has been called.
2. **Alternatively**, read `IEigenPod.currentCheckpoint().podBalanceGwei` and add it to the share count when a checkpoint is in progress.
3. Use a private/protected mempool relay (e.g., Flashbots `eth_sendPrivateTransaction`) for the operator's `startCheckpoint` call to make block stuffing ineffective.

## Proof of Concept

```solidity
// Foundry fork test (mainnet fork with active NodeDelegator + EigenPod)
function testBlockStuffingPriceSuppression() public {
    // 1. Simulate beacon-chain partial withdrawal arriving at EigenPod
    vm.deal(address(eigenPod), address(eigenPod).balance + 1 ether);

    // 2. Verify getEffectivePodShares() does NOT include the 1 ETH
    uint256 sharesBefore = nodeDelegator.getEffectivePodShares();
    // sharesBefore does not reflect the 1 ETH sitting in eigenPod

    // 3. Record rsETH price while ETH is uncheckpointed (block stuffing window)
    lrtOracle.updateRSETHPrice();
    uint256 priceSuppressed = lrtOracle.rsETHPrice();

    // 4. Attacker mints rsETH at suppressed price — gets more tokens than fair share
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 10 ether}(...);

    // 5. Operator finalizes checkpoint (stuffing ends)
    vm.prank(operator);
    nodeDelegator.startCheckpoint(false);
    // ... verifyCheckpointProofs ...

    // 6. Price corrects upward
    lrtOracle.updateRSETHPrice();
    uint256 priceAfter = lrtOracle.rsETHPrice();

    assertGt(priceAfter, priceSuppressed); // price was suppressed during stuffing
    // Attacker's rsETH is now worth more per token; existing holders are diluted
}
```
