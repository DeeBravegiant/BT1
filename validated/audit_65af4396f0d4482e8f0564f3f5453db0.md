Audit Report

## Title
Shared `RSETH.checkDailyMintLimit` Cap Allows Depositor to Block Protocol Fee Minting — (`contracts/RSETH.sol`, `contracts/LRTOracle.sol`)

## Summary
`RSETH.mint()` applies the `checkDailyMintLimit` modifier to every caller, including `LRTOracle` when it mints protocol fees. A depositor who fills `maxMintAmountPerDay` within a period causes every subsequent `updateRSETHPrice()` call to revert, preventing `rsETHPrice` from being updated and permanently blocking protocol fee yield for that period. The attack is repeatable every 24 hours at the cost of gas only, with no capital loss to the attacker.

## Finding Description
There are two independent daily-limit systems in the protocol. **System 1** (`RSETH.sol`): `currentPeriodMintedAmount` and `maxMintAmountPerDay` are shared by every caller of `RSETH.mint()`, with no exemption for privileged callers. The `checkDailyMintLimit` modifier at `RSETH.sol` L42–56 reverts with `DailyMintLimitExceeded` if `currentPeriodMintedAmount + amount > maxMintAmountPerDay`. **System 2** (`LRTOracle.sol`): `_checkAndUpdateDailyFeeMintLimit` at L197–210 is a separate, independent cap governing only the oracle's own fee accounting.

The fee mint path in `_updateRsETHPrice()` (L298–311) first passes the oracle's own fee-limit check at L303, then calls `IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee)` at L306. This re-enters `RSETH.mint()`, which unconditionally applies `checkDailyMintLimit`. If a depositor has already filled `maxMintAmountPerDay` via `LRTDepositPool._mintRsETH()` → `IRSETH.mint()`, the modifier reverts, causing the entire `updateRSETHPrice()` transaction to revert. `rsETHPrice` is never updated and the fee is never minted for that period. There is no priority ordering, role-based bypass, or exemption for the oracle's fee mint path.

## Impact Explanation
When `maxMintAmountPerDay` is set to a finite non-zero value and a depositor fills the cap: (1) `updateRSETHPrice()` reverts — `rsETHPrice` becomes stale; (2) protocol fee rsETH is not minted to the treasury for that period. If the attacker repeats the fill every period, the treasury fee is perpetually blocked — yield accrues in the TVL but is never extracted as rsETH to the treasury. This constitutes **permanent freezing of unclaimed yield**, which is an explicitly allowed Medium impact in the target scope.

## Likelihood Explanation
- Requires `maxMintAmountPerDay` to be set to a finite non-zero value by the manager — a valid and intended operational state.
- Requires the attacker to deposit enough ETH/LST to fill the remaining daily cap — no special role needed, only capital, which is returned as rsETH (no capital loss).
- `updateRSETHPrice()` is a public function; the attacker can time the fill to occur just before a price update.
- The attack costs only gas per period and is trivially repeatable.

Likelihood: **Medium**.

## Recommendation
Decouple protocol fee minting from the user-deposit daily cap. Add a separate internal `_mintFee(address to, uint256 amount)` function in `RSETH` that retains the `MINTER_ROLE` check and `whenNotPaused` guard but skips `checkDailyMintLimit`, relying solely on `LRTOracle._checkAndUpdateDailyFeeMintLimit` for fee-specific rate limiting. Update `LRTOracle._updateRsETHPrice()` to call this new function instead of the public `mint()`. This cleanly separates the two concerns without affecting user deposit limits.
