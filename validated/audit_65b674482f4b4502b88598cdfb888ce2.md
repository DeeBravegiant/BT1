Audit Report

## Title
`AnchoredPriceProvider._isStale` Hard-Rejects Future `refTime`, Permanently DoS-ing Swaps on L2 Deployments — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary
`AnchoredPriceProvider`, explicitly documented as "the one standard provider for public pools," contains an L1-only staleness check that unconditionally returns `true` whenever `refTime > block.timestamp`. On L2 chains (Base, HyperEVM) where sequencer `block.timestamp` routinely lags behind oracle publication time, this causes every call to `getBidAndAskPrice()` to revert with `FeedStalled`, making all swaps on pools using this provider permanently unusable. The protocol already ships `PriceProviderL2` and `ProtectedPriceProviderL2` with a `futureTol`-aware staleness check, but no equivalent `AnchoredPriceProviderL2` exists.

## Finding Description
`AnchoredPriceProvider._isStale` (L222–230) unconditionally returns `true` for any `refTime > nowTs`:

```solidity
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta)
    internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) return true;   // hard-rejects any future timestamp
    return (nowTs - refTime) > maxDelta;
}
```

This is called from `_readLeg` (L283), which is the sole path through which `_getBidAndAskPrice` reads oracle data:

```solidity
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS))
    return (mid, spreadBps, refTime, false);
```

When `ok == false`, `_getBidAndAskPrice` (L260) returns the sentinel `(0, type(uint128).max)`, and `getBidAndAskPrice` (L216) reverts with `FeedStalled`. `MetricOmmPool._getBidAndAskPriceX64` (L806–811) catches this and reverts with `PriceProviderFailed`, blocking every swap.

In contrast, `PriceProviderL2._isStale` (L135–150) accepts a `futureTol` parameter and only rejects future timestamps that exceed the tolerance:

```solidity
if (refTime > nowTs) {
    return (refTime - nowTs) > futureTol;   // tolerates clock skew
}
```

No `AnchoredPriceProviderL2` variant exists in the repository. The existing L2-aware providers (`PriceProviderL2`, `ProtectedPriceProviderL2`) do not replicate the anchored-band logic of `AnchoredPriceProvider`, so there is no drop-in substitute for public pools on L2.

## Impact Explanation
Every swap on any pool whose `priceProvider` is an `AnchoredPriceProvider` deployed on an L2 chain will revert at `_getBidAndAskPriceX64` → `PriceProviderFailed` whenever the oracle's `refTime` exceeds `block.timestamp` by even 1 second. Because L2 sequencer timestamps routinely lag real-world time, this is the normal operating state, not an edge case. The result is a complete, persistent DoS on all swap flows for affected pools — matching the allowed impact: *"Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows."*

## Likelihood Explanation
High. The protocol explicitly targets Base and HyperEVM (both L2s). The existence of `PriceProviderL2` and `ProtectedPriceProviderL2` proves the team is aware of L2 clock-skew. `AnchoredPriceProvider` is the designated standard provider for public pools (NatSpec line 11: "the one standard provider for public pools"). Any public pool on L2 using it will be broken under normal oracle update cadence. No privileged access is required — any unprivileged trader calling `swap` triggers the revert.

## Recommendation
Add an `AnchoredPriceProviderL2` variant that accepts a `FUTURE_TOLERANCE` immutable (bounded to ≤ 1 hour, matching existing L2 providers) and overrides `_isStale` with the four-argument form identical to `PriceProviderL2._isStale`. Alternatively, merge the L2-aware staleness logic directly into `AnchoredPriceProvider` with `FUTURE_TOLERANCE = 0` as the L1 default, set via a constructor parameter.

## Proof of Concept
1. Deploy `AnchoredPriceProvider` on Base with a Pyth Lazer or Chainlink Data Streams oracle.
2. Push an oracle update whose `refTime` is 2 seconds ahead of the current Base sequencer `block.timestamp` (normal for off-chain pushers).
3. Call `pool.swap(...)` on any pool using this provider.
4. Execution path: `swap` → `_getBidAndAskPriceX64` (L804) → `getBidAndAskPrice` (L214) → `_getBidAndAskPrice` (L258) → `_readLeg` (L277) → `_isStale(refTime=T+2, nowTs=T, maxDelta=...)` returns `true` → `ok = false` → returns `(0, type(uint128).max)` → `getBidAndAskPrice` reverts `FeedStalled` → pool reverts `PriceProviderFailed`.
5. Every subsequent swap reverts identically until the sequencer timestamp catches up — which may never happen if oracle updates are continuous.
