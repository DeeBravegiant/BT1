### Title
Missing `max_l2_gas_amount` Admission Check for Declare Transactions in Gateway Stateless Validator — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces an upper bound on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions but explicitly skips that check for `Declare` transactions. An attacker can submit a `Declare` transaction with `l2_gas.max_amount = u64::MAX` and the gateway will admit it, bypassing the intended admission-control limit.

---

### Finding Description

In `validate_resource_bounds`, the check reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check at all
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production config sets `max_l2_gas_amount = 1_210_000_000`: [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly confirms the bypass — a Declare with `max_amount: GasAmount(200)` passes when the limit is `100`: [3](#0-2) 

By contrast, the same test for `Invoke` and `DeployAccount` asserts a rejection: [4](#0-3) 

The full `validate` entry point calls `validate_resource_bounds` for every transaction type, so the gap is reachable on every Declare submission: [5](#0-4) 

---

### Impact Explanation

A Declare transaction carrying `l2_gas.max_amount = u64::MAX` passes the stateless validator and enters the mempool. This violates the gateway's admission invariant that no transaction with `max_l2_gas_amount > config.max_l2_gas_amount` should be sequenced. Downstream effects include:

1. **Wrong RPC fee estimate** — `starknet_estimateFee` and `starknet_simulateTransactions` derive the per-transaction gas limit from `resource_bounds.l2_gas.max_amount`. With `u64::MAX` as the bound, the simulator's gas accounting is driven by an unbounded value, producing an authoritative-looking wrong result (matches the "High. RPC execution, fee estimation … returns an authoritative-looking wrong value" impact).
2. **Mempool admission of invalid transactions** — the gateway accepts a transaction that should have been rejected, matching the "High. Mempool/gateway/RPC admission accepts invalid transactions" impact.

---

### Likelihood Explanation

Triggering the gap requires only a well-formed Declare transaction with an oversized `l2_gas.max_amount`. No privileged access, no malicious peer, no special timing is needed. Any unprivileged user can craft and submit such a transaction via the public HTTP endpoint.

---

### Recommendation

Apply the same upper-bound check to Declare transactions and remove the TODO:

```rust
// In validate_resource_bounds:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

This mirrors the existing check for Invoke/DeployAccount and closes the inconsistency.

---

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)` and a valid Sierra class.
2. Submit it to the gateway HTTP endpoint (`starknet_addDeclareTransaction`).
3. Observe that the stateless validator returns `Ok(())` — the transaction is admitted to the mempool despite `max_l2_gas_amount = 1_210_000_000` in the production config.
4. Call `starknet_estimateFee` on the same transaction and observe that the returned fee is computed against the `u64::MAX` gas bound rather than the configured limit.

The existing test already demonstrates step 3 in isolation: [6](#0-5)
