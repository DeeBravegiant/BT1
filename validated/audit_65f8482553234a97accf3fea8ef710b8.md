### Title
Missing Attestation Validity Check in `is_caller_an_attested_participant` Allows Expired/Revoked-Image Nodes to Contribute to Threshold Signatures - (File: `crates/contract/src/tee/tee_state.rs`)

---

### Summary

`is_caller_an_attested_participant()` — the gating check called by every `respond*` method — verifies only that a node's identity record exists in `stored_attestations` (account ID and key match). It never calls `re_verify` to confirm the stored attestation is still temporally valid or that the node's image hash is still on the approved list. As a result, a node whose TEE attestation has expired, or whose image hash has been rotated out of the allowed set, can continue to call `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` and have its threshold-signature contributions accepted by the contract — bypassing the TEE enforcement that is the system's primary hardware-level security guarantee.

---

### Finding Description

The `respond()` family of methods enforces two runtime guards before accepting a node's signature contribution:

1. The coarse-grained `accept_requests` flag (set/cleared only when `verify_tee()` is explicitly called).
2. `assert_caller_is_attested_participant_and_protocol_active()`, which delegates to `is_caller_an_attested_participant()`.

`is_caller_an_attested_participant()` performs only an **identity** check:

```rust
// crates/contract/src/tee/tee_state.rs  ~L469-L498
pub(crate) fn is_caller_an_attested_participant(
    &self,
    participants: &Participants,
) -> Result<(), AttestationCheckError> {
    let signer_account_pk = env::signer_account_pk();
    let signer_id = env::signer_account_id();

    let info = participants
        .info(&signer_id)
        .ok_or(AttestationCheckError::CallerNotParticipant)?;

    let attestation = self
        .stored_attestations
        .get(&info.tls_public_key)
        .ok_or(AttestationCheckError::AttestationNotFound)?;

    if attestation.node_id.account_id != signer_id {
        return Err(AttestationCheckError::AttestationOwnerMismatch);
    }

    let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
        .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
    if attestation.node_id.account_public_key != signer_ed25519 {
        return Err(AttestationCheckError::AttestationKeyMismatch);
    }

    Ok(())   // ← no expiry check, no image-hash re-verification
}

`verify_tee()` is not called automatically; it requires an explicit participant transaction. The documentation states it is expected to be triggered "every 2 days." Between calls, any node whose attestation has expired or whose image hash has been removed from the allowed list remains in `stored_attestations` and passes `is_caller_an_attested_participant` without issue.

The `accept_requests` flag provides no per-node granularity: it is set to `false` only when the number of valid participants drops **below threshold** — meaning a single expired/revoked node among an otherwise healthy set leaves `accept_requests = true` and the expired node's `respond()` calls succeed.

---

### Impact Explanation

**Attestation authorization bypass enabling unauthorized threshold-signature contribution.**

The TEE attestation system exists to guarantee that every node contributing a key-share to a threshold signature is running an approved, unmodified binary inside a genuine Intel TDX enclave. When an image hash is rotated out of the allowed set (e.g., because a vulnerability was found in the old binary), nodes running that binary should immediately lose the ability to contribute to signing. Instead, they retain full signing capability until `verify_tee()` is manually triggered.

A node running a deprecated or known-vulnerable image can:
- Contribute its key share to threshold signatures for any pending `sign()` request.
- Contribute to `respond_verify_foreign_tx()`, potentially approving foreign-chain transactions under a compromised verification path.
- If the deprecated image contains a key-share exfiltration vulnerability, the window between image-hash rotation and `verify_tee()` execution is the precise window in which the vulnerability is exploitable on-chain.

This maps to the allowed impact: **High — participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions**, and potentially **Critical — unauthorized transaction execution or threshold signature issuance without the required participant authorization** if the deprecated node software is actively malicious.

---

### Likelihood Explanation

- Image hash rotations are a normal operational event (software updates, security patches). Every rotation opens a window.
- The window lasts up to ~2 days (the documented `verify_tee()` cadence), but there is no on-chain enforcement of that cadence — it can be longer.
- The attack requires no collusion above threshold; a single expired/revoked node among an otherwise valid set is sufficient to exploit the gap.
- The node operator does not need to do anything special: simply not renewing the attestation (or running the old image after a hash rotation) is enough.
- The entry path is the standard `respond()` call, callable by any current participant.

---

### Recommendation

Add an inline validity check inside `is_caller_an_attested_participant` (or in `assert_caller_is_attested_participant_and_protocol_active`) that calls `re_verify` on the retrieved `VerifiedAttestation` using the current block timestamp and the current allowed hash/measurement sets:

```rust
// After retrieving `attestation` from stored_attestations:
let timestamp_seconds = env::block_timestamp() / 1_000_000_000;
attestation.verified_attestation.re_verify(
    timestamp_seconds,
    &self.allowed_docker_image_hashes.hashes(),
    &self.allowed_launcher_images.compose_hashes(),
    &self.allowed_measurements.measurements(),
).map_err(|e| AttestationCheckError::AttestationExpiredOrRevoked(e.to_string()))?;
```

This mirrors the check already performed in `reverify_participants` and ensures that every `respond*` call is gated on a live validity check, not just a stale identity lookup.

---

### Proof of Concept

1. Deploy the contract with 3 participants (threshold 2). All three submit valid attestations.
2. Governance votes to remove the old image hash from `allowed_docker_image_hashes` (simulating a security patch rollout). Participant 3 has not yet resubmitted with the new image hash.
3. **Do not call `verify_tee()`.**
4. A user calls `sign()`. Participant 3 (running the old, now-disallowed image) calls `respond()` with a valid signature share.
5. `is_caller_an_attested_participant` finds Participant 3 in `stored_attestations`, verifies account ID and key — returns `Ok(())`.
6. `accept_requests` is still `true` (participants 1 and 2 are valid, count ≥ threshold).
7. The contract accepts Participant 3's contribution. The threshold signature is produced with input from a node running non-approved software.
8. Only after `verify_tee()` is called does Participant 3 get evicted — but all signatures produced in the interim are already on-chain and irrevocable.
