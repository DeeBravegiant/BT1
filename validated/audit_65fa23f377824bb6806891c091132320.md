Audit Report

## Title
Missing Slippage Protection in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPool.sol, RSETHPoolNoWrapper.sol, RSETHPoolV2.sol, RSETHPoolV2ExternalBridge.sol, RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol)

## Summary
All L2 pool `deposit()` functions compute the rsETH/wrsETH output using a live oracle rate at execution time but accept no `minRsETHAmountExpected` parameter from the caller. If the oracle rate changes between transaction submission and execution, the user receives fewer rsETH tokens than anticipated with no ability to revert. The L1 `LRTDepositPool` contract already implements this protection, confirming the L2 omission is unintentional.

## Finding Description
Every L2 pool `deposit()` function calls `viewSwapRsETHAmountAndFee()`, which fetches `rsETHToETHrate` live via `getRate()` and computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. No minimum-output guard is applied before transferring or minting tokens to the caller.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.** A depositor receives fewer rsETH/wrsETH tokens than they expected at submission time when the oracle rate increases before their transaction is mined. The user's ETH/LST is consumed and the transaction succeeds; there is no revert path. The shortfall is relative to the user's expectation, not to the on-chain state at execution time, so no funds are stolen — but the contract fails to deliver the return the user was promised when they signed the transaction.

## Likelihood Explanation
No attacker capability is required. The oracle rate (`rsETHToETHrate`) is updated periodically on every L2 chain where these pools are deployed. Any deposit transaction pending in the mempool when an oracle update is mined will execute at the new, less favorable rate. This is a routine occurrence and affects any ordinary depositor.

## Recommendation
Add a `minRsETHAmountExpected` parameter to all `deposit()` functions in the L2 pool contracts, mirroring `LRTDepositPool._beforeDeposit()`:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected)
    external payable nonReentrant whenNotPaused
{
    ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    ...
}
```

The existing `getMinAmount(uint256 amount, uint256 slippageTolerance)` helper can be used off-chain to compute the appropriate `minRsETHAmountExpected` before submitting.

## Proof of Concept
1. Oracle reports `rsETHToETHrate = 1.05e18`.
2. User submits `RSETHPoolV2ExternalBridge.deposit{value: 1 ether}("ref")`, expecting `≈0.952 wrsETH` after fee.
3. Before the tx is mined, oracle updates to `rsETHToETHrate = 1.10e18`.
4. Deposit executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 wrsETH`.
5. User receives `≈0.909 wrsETH` — a ~4.5% shortfall — with no revert.

**Foundry fork test plan:**
- Fork an L2 where `RSETHPoolV2ExternalBridge` is deployed.
- Record the current `getRate()` value.
- Prank the oracle updater to increase the rate by ~5%.
- Call `deposit{value: 1 ether}("ref")` as a normal user.
- Assert that `wrsETH.balanceOf(user)` is less than the amount computed at the pre-update rate, confirming the shortfall with no revert.
