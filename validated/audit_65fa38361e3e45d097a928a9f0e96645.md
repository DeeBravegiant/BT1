### Title
Dev Firmware TEE Measurements Accepted in Production Participant Authorization — (`crates/mpc-attestation/src/attestation.rs`)

### Summary

`default_measurements()` statically compiles both production (`assets/tcb_info.json`) and development (`assets/tcb_info_dev.json`) measurement sets into every build. These are seeded into the on-chain `allowed_measurements` list on contract migration. As a result, a node operator running dev firmware can submit a valid DCAP-attested quote whose MRTD/RTMR0-2 match only the dev set, pass `verify_any_measurements`, and be accepted as a valid participant in the production signing network.

### Finding Description

`default_measurements()` in `crates/mpc-attestation/src/attestation.rs` returns a static array of two `ExpectedMeasurements`: [1](#0-0) 

The comment on line 262 explicitly acknowledges this as a security issue (`TODO(#1433): Security - remove dev measurements from production builds after testing is complete`).

The contract's attestation verification path reads the allowed measurements from the **on-chain** `allowed_measurements` list via `get_accepted_measurements()`: [2](#0-1) 

This on-chain list is populated in two ways:
1. **Seeded on migration** — the `AllowedMeasurements` doc comment states: *"seeded from `default_measurements()` on migration"*, and the testnet setup guide explicitly instructs operators to vote in both `tcb_info.json` and `tcb_info_dev.json`.
2. **Via `vote_add_os_measurement`** — requires threshold participant votes. [3](#0-2) 

When a node submits attestation via `submit_participant_info`, the contract calls: [4](#0-3) 

`verify_any_measurements` iterates over `accepted_measurements` and returns `Ok` on the **first match**: [5](#0-4) 

If the dev measurement set is in the on-chain allowed list, a node whose MRTD/RTMR0-2 match only the dev set will pass this check.

### Impact Explanation

A node operator running dev firmware (which typically has debug capabilities, relaxed security policies, or unaudited code paths) can:
1. Generate a valid DCAP quote from genuine Intel TDX hardware running dev firmware.
2. Submit it via `submit_participant_info`.
3. Pass `verify_any_measurements` because the dev `ExpectedMeasurements` is in the on-chain allowed list.
4. Be accepted as a valid participant, participate in threshold signing, and potentially access key share material during the MPC protocol.

This directly violates the invariant that only nodes running audited production firmware may join the production signing set.

### Likelihood Explanation

**Moderate.** The attacker must:
- Possess genuine Intel TDX hardware (a real barrier, but node operators already have this).
- Run dev firmware on that hardware (accessible to any operator).
- Have the approved MPC docker image hash and launcher compose hash in the on-chain allowed list (already voted in for legitimate nodes).
- Be an existing participant or be voted in via resharing.

The most realistic path is an **existing participant** switching from prod to dev firmware and resubmitting their attestation — the contract accepts it without any governance action required.

### Recommendation

1. Remove `assets/tcb_info_dev.json` from the `default_measurements()` array in production builds (resolve TODO #1433).
2. Ensure the migration seeding logic does not include dev measurements in the on-chain allowed list for production deployments.
3. Existing participants should vote to remove dev measurements from the on-chain `allowed_measurements` list via `vote_remove_os_measurement`.

### Proof of Concept

Build an integration test that:
1. Seeds the contract's `allowed_measurements` with `default_measurements()` (both prod and dev).
2. Constructs a `DstackAttestation` whose MRTD/RTMR0-2 match only the dev `ExpectedMeasurements` from `assets/tcb_info_dev.json`.
3. Calls `verify_any_measurements` with this report.
4. Asserts it returns `Ok(dev_measurements)` — confirming a dev-firmware node passes production attestation verification.

The test helper `setup_approved_measurements` in the contract test suite already demonstrates this pattern by seeding both prod and dev measurements: [6](#0-5)

    fn verify_any_measurements(
        &self,
        report_data: &TDReport10,
        tcb_info: &TcbInfo,
        accepted_measurements: &[ExpectedMeasurements],
    ) -> Result<ExpectedMeasurements, VerificationError> {
        for expected in accepted_measurements {
            if self
                .verify_static_rtmrs(report_data, tcb_info, expected)
                .is_ok()
                && self
                    .verify_key_provider_digest(tcb_info, &expected.key_provider_event_digest)
                    .is_ok()
            {
                return Ok(*expected); // found a valid match
            }
        }

        Err(VerificationError::WrongHash {
            name: "expected_measurements",
            expected: "one of the embedded TCB info sets (prod or dev)".into(),
            found: "none matched".into(),
        })
    }
