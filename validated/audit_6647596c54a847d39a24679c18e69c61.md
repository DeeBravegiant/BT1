Audit Report

## Title
Missing User-Controlled Slippage Protection in L2 Pool `deposit()` Functions - (File: contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol)

## Summary
All four L2 pool contracts expose `deposit()` entry points that compute rsETH output entirely from a live oracle rate with no caller-supplied minimum output floor. A user who submits a deposit transaction cannot enforce a minimum acceptable rsETH amount, so any oracle rate movement between transaction submission and on-chain execution silently reduces the tokens received. The L1 `LRTDepositPool` correctly enforces a `minRSETHAmountExpected` parameter, making this an explicit gap in the L2 contracts.

## Finding Description
Every L2 pool `deposit()` function computes `rsETHAmount` via `viewSwapRsETHAmountAndFee`, which calls `getRate()` (backed by `IOracle.getRate()`) at execution time:

- `RSETHPool.deposit(string)` and `RSETHPool.deposit(address,uint256,string)` — lines 265–305: no `minRsETHAmount` parameter; output is `amountAfterFee * 1e18 / rsETHToETHrate`.
- `RSETHPoolNoWrapper.deposit(string)` and `RSETHPoolNoWrapper.deposit(address,uint256,string)` — lines 231–271: identical pattern.
- `RSETHPoolV2ExternalBridge.deposit(string)` — line 289: identical pattern.
- `RSETHPoolV3ExternalBridge.deposit(string)` and `RSETHPoolV3ExternalBridge.deposit(address,uint256,string)` — lines 366–412: identical pattern; rate computation at lines 418–453.

None of these functions accept or check a caller-supplied minimum output. The only input validation is `if (amount == 0) revert InvalidAmount()`.

By contrast, `LRTDepositPool.depositETH` (line 76) and `LRTDepositPool.depositAsset` (line 99) both accept `minRSETHAmountExpected` and enforce it via `_beforeDeposit`, reverting if the computed mint amount falls below the caller's floor.

The oracle rate (`IOracle.getRate()`) updates continuously. On L2 chains with sub-second to ~2-second block times, the rate seen at signing routinely differs from the rate at execution. No attacker action is required; ordinary block progression is sufficient to trigger the discrepancy.

## Impact Explanation
A depositor receives fewer rsETH/wrsETH tokens than they anticipated at signing. Because rsETH is a yield-bearing token, the deposited ETH value is not destroyed, but the user receives a smaller protocol share than intended. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The condition is triggered by any oracle rate increase between transaction submission and mining — a normal, continuous occurrence on every supported L2 chain. No privileged access, no attacker, and no special network conditions are required. Every deposit on every L2 pool is affected on every block where the oracle rate has moved upward.

## Recommendation
Add a `minRsETHAmount` parameter to all `deposit()` overloads in all four L2 pool contracts and revert if the computed `rsETHAmount` falls below it, mirroring the existing `LRTDepositPool` pattern:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmount)
    external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmount) revert InsufficientOutputAmount();
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same pattern to all ERC-20 token `deposit()` overloads across all four contracts.

## Proof of Concept
1. Alice calls `RSETHPoolV3ExternalBridge.deposit{value: 1 ether}("ref")` when `rsETHToETHrate = 1.05e18`. Expected rsETH ≈ 0.952.
2. Before Alice's transaction is mined, the oracle updates to `rsETHToETHrate = 1.10e18` (normal yield accrual).
3. Alice's transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 rsETH`.
4. Alice receives ~4.5% fewer rsETH tokens than expected with no on-chain revert.
