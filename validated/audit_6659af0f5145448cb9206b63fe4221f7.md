### Title
`accept_requests` Flag Not Reset After `vote_reshared` / `vote_cancel_resharing` Transition — (`File: crates/contract/src/lib.rs`)

---

### Summary

After `verify_tee()` sets `self.accept_requests = false` (because kicking out participants with expired attestations would break the threshold relation), the contract stays `Running`. Participants can then initiate and complete a resharing via `vote_new_parameters` + `vote_reshared()`. When `vote_reshared()` transitions the contract back to `Running` with a new, fully-attested participant set, it **never resets `accept_requests` to `true`**. The same omission exists in `vote_cancel_resharing()`. The result is that the contract permanently refuses all user-facing requests (`sign`, `respond`, `respond_ckd`, `respond_verify_foreign_tx`) after a successful resharing, until an additional `verify_tee()` call is made — a call that is not automatically spawned and is not documented as required.

---

### Finding Description

`MpcContract` holds a top-level boolean field `accept_requests` that gates every user-facing signing path. [1](#0-0) 

`verify_tee()` sets this flag to `false` when the surviving valid-attestation set is too small to maintain the threshold relation, and the contract stays `Running`: [2](#0-1) 

At this point participants may call `vote_new_parameters` to propose a new, fully-attested participant set and drive the contract into `Resharing`. When resharing completes, `vote_reshared()` transitions the contract back to `Running`: [3](#0-2) 

No line in `vote_reshared()` resets `accept_requests` to `true`. The six cleanup promises spawned afterward also do not touch the flag: [4](#0-3) 

The same omission exists in `vote_cancel_resharing()`, which also transitions back to `Running` without touching `accept_requests`.

Every user-facing request path then hits the guard: [5](#0-4) 

and every node `respond*` path hits: [6](#0-5) 

Both return `TeeError::TeeValidationFailed` for as long as `accept_requests` remains `false`.

---

### Impact Explanation

After a successful resharing that replaces the participant set with nodes that all have valid TEE attestations, the contract is in `Running` state with `accept_requests = false`. Every call to `sign`, `request_app_private_key`, `verify_foreign_transaction`, `respond`, `respond_ckd`, and `respond_verify_foreign_tx` is rejected with `TeeValidationFailed`. No threshold signature can be issued and no pending yield can be resolved. This is a **request-lifecycle and contract execution-flow break** that violates the production safety invariant: a `Running` contract with a fully-attested participant set must accept requests. The freeze persists until an operator manually calls `verify_tee()` — a call that is not automatically spawned by `vote_reshared()` and is not documented as a post-resharing requirement.

This matches the **Medium** allowed impact: *"request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

The trigger sequence is entirely realistic in production:

1. One or more participants' TEE attestations expire (a scheduled, expected event).
2. Any participant calls `verify_tee()`. If the surviving valid set is below the threshold relation, `accept_requests` is set to `false` and the contract stays `Running`.
3. Participants vote via `vote_new_parameters` to replace the expired nodes — a standard governance operation.
4. Resharing completes via `vote_reshared()`.
5. The contract is now `Running` with a healthy participant set but `accept_requests = false`.

No privileged access, no collusion above threshold, and no network-level attack is required. The sequence follows the documented operational runbook for handling expired attestations.

---

### Recommendation

In `vote_reshared()`, after `self.protocol_state = new_state;`, add:

```rust
self.accept_requests = true;
```

Apply the same fix in `vote_cancel_resharing()` after the state transition back to `Running`. Alternatively, spawn a `verify_tee()` promise (analogous to the existing cleanup promises) so the flag is re-evaluated automatically. Add a test that verifies `accept_requests` is `true` after a resharing that was preceded by a `verify_tee()` call that set it to `false`.

---

### Proof of Concept

```
1. Deploy contract with 5 participants, GovernanceThreshold = 5,
   reconstruction_threshold = 5 (all participants required).

2. Expire attestations for 2 participants (leaving 3 valid, below threshold of 5).

3. Call verify_tee():
   → TeeValidationResult::Partial, validate_governance_against_reconstruction fails
   → accept_requests = false, contract stays Running.

4. All 5 participants call vote_new_parameters with a new 5-participant set
   (all with fresh valid attestations).
   → Contract enters Resharing.

5. New participants drive resharing to completion via
   start_reshare_instance + vote_reshared (threshold votes).
   → vote_reshared() transitions contract to Running.
   → accept_requests is still false (never reset).

6. Any user calls sign(...):
   → check_request_preconditions hits `if !self.accept_requests`
   → panics with TeeError::TeeValidationFailed.
   → Signing is permanently blocked despite a healthy new participant set.
``` [7](#0-6) [5](#0-4)
