### Title
Single Participant Empty Foreign-Chain Registration Blocks All `verify_foreign_transaction` Requests — (`crates/contract/src/foreign_chain_rpc.rs`)

---

### Summary

`get_supported_foreign_chains()` computes the **strict intersection** of every active participant's registered foreign-chain set. A single participant that registers an empty list — or that has not yet registered at all — collapses the intersection to the empty set, causing every subsequent `verify_foreign_transaction` call to be rejected at the contract level, regardless of how many other participants correctly cover the requested chain.

---

### Finding Description

The on-chain contract exposes `verify_foreign_transaction`, which early-rejects any request whose target chain is absent from `get_supported_foreign_chains()`.

The current production implementation of `get_supported_foreign_chains()` returns the **strict intersection** of all active participants' per-node registered chain sets. This is explicitly documented as the current behavior in the production design document:

> "Today, `get_supported_foreign_chains()` returns the **strict intersection** of every active participant's registered chains, and `verify_foreign_transaction` rejects any request whose target chain is not in it. **A single node that registers an empty list (or hasn't registered yet) drops every chain** — one operator can take the whole feature down." [1](#0-0) 

The `verify_foreign_transaction` entry point enforces this check synchronously before enqueuing the yield-resume promise: [2](#0-1) 

If `get_supported_foreign_chains()` returns an empty set (because one participant registered nothing), the `!supported_chains.contains(&requested_chain)` guard fires and `env::panic_str` is called, reverting the entire transaction. Every caller — regardless of which chain they requested — is blocked.

The proposed fix (switching from strict intersection to a threshold-based availability rule) is documented as **"Status: Proposed"** and has not yet been merged into the production contract: [3](#0-2) 

---

### Impact Explanation

A single participant calling `register_foreign_chain_config` with an empty list — or a newly reshared-in participant that has not yet submitted any registration — causes `get_supported_foreign_chains()` to return `∅`. Every `verify_foreign_transaction` call then hits the `ForeignChainNotSupported` panic path and reverts, regardless of which chain is requested and regardless of how many other participants correctly cover it.

This is a **contract execution-flow manipulation that breaks the request-lifecycle invariant** of the `verify_foreign_transaction` feature: a request for a chain that is covered by `n − 1` participants (well above the signing threshold) is permanently rejected as long as one participant's registration is empty. Users who have already paid the deposit and are waiting for a yield-resume response will time out and lose their deposit.

This matches the allowed Medium impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

The trigger requires one participant to hold an empty foreign-chain registration. This occurs naturally in two scenarios:

1. **Resharing adds a new participant** who has not yet called `register_foreign_chain_config`. During the window between the resharing completing and the new node's first registration, the intersection is empty.
2. **An adversarial participant** deliberately submits an empty registration to suppress the feature.

Neither scenario requires threshold collusion. A single participant below the signing threshold is sufficient.

---

### Recommendation

Replace the strict-intersection rule in `get_supported_foreign_chains()` with the threshold-based availability rule already described in the design document: a chain is **available** if and only if at least `signing_threshold` active participants report coverage for it. This is the fix the team has already designed: [4](#0-3) 

Until the fix is deployed, `verify_foreign_transaction` should also check availability against the threshold-based count rather than the strict intersection, so that a single non-registering participant cannot suppress the entire feature.

---

### Proof of Concept

1. Network is in `Running` state with participants `[P0, P1, P2]`, threshold = 2. All three have registered `{Bitcoin, Ethereum}`.
2. A resharing adds participant `P3`. `P3` has not yet called `register_foreign_chain_config`.
3. `get_supported_foreign_chains()` computes `{Bitcoin, Ethereum} ∩ {Bitcoin, Ethereum} ∩ {Bitcoin, Ethereum} ∩ {} = {}`.
4. Any user calling `verify_foreign_transaction` for Bitcoin hits:

```rust
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported { requested: requested_chain }.to_string(),
    );
}
``` [5](#0-4) 

5. The transaction reverts. The user's deposit is consumed. The request is never enqueued. All `verify_foreign_transaction` requests for all chains fail until `P3` registers a non-empty chain list.

The design document explicitly names this as the root cause and the motivation for the proposed fix: [1](#0-0)
