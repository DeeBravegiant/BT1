### Title
Unsigned Routing Metadata in `ChunkEndorsementV2` Allows Cache-Poisoning to Suppress Validator Endorsements - (File: `core/primitives/src/stateless_validation/chunk_endorsement.rs`)

### Summary

`ChunkEndorsementV2` carries two independently-signed payloads — `inner` (containing only `chunk_hash`) and `metadata` (containing `account_id`, `shard_id`, `epoch_id`, `height_created`) — with no cryptographic binding between them. An unprivileged network peer that observes two valid endorsements from the same validator can splice the `inner`+`signature` from one with the `metadata`+`metadata_signature` from another. The resulting forged endorsement passes all signature checks and is accepted into the endorsement cache under the target chunk's routing key, but with the wrong `chunk_hash`. Because the cache deduplicates by `(ChunkProductionKey, account_id)`, the legitimate endorsement that arrives later is silently dropped as "already received," suppressing that validator's vote for the target chunk.

### Finding Description

`ChunkEndorsementV2` is defined as:

```rust
pub struct ChunkEndorsementV2 {
    inner: ChunkEndorsementInnerV1,       // signed: chunk_hash only
    signature: Signature,                  // sig over inner
    metadata: ChunkEndorsementMetadata,   // shard_id, epoch_id, height_created, account_id
    metadata_signature: Signature,         // sig over metadata
}
``` [1](#0-0) 

The `verify()` method checks both signatures independently against the same public key:

```rust
fn verify(&self, public_key: &PublicKey) -> bool {
    let inner = borsh::to_vec(&self.inner).unwrap();
    let metadata = borsh::to_vec(&self.metadata).unwrap();
    self.signature.verify(&inner, public_key)
        && self.metadata_signature.verify(&metadata, public_key)
}
``` [2](#0-1) 

`inner` contains only `chunk_hash` + `signature_differentiator`; `metadata` contains `shard_id`, `epoch_id`, `height_created`, `account_id`. Neither payload commits to the other's fields. The `chunk_hash` is not included in the metadata signature, and the routing metadata is not included in the inner signature. [3](#0-2) 

All routing and validator-assignment decisions are driven exclusively from `metadata`:

### Impact Explanation

An attacker constructs a forged `ChunkEndorsementV2` by splicing:
- `inner` + `signature` from validator V's endorsement of chunk A (`chunk_hash_A`)
- `metadata` + `metadata_signature` from validator V's endorsement of chunk B (`shard_B`, `epoch_B`, `height_B`)

Both signatures verify against V's public key, so `verify()` returns `true`. `validate_chunk_endorsement_signature` passes. The forged endorsement is stored in the cache under key `(shard_B, epoch_B, height_B, validator_V)` with `chunk_hash_A`.

When the legitimate endorsement E_B for chunk B from validator V arrives, the deduplication check finds the key already present and silently drops E_B. When the block producer calls `collect_chunk_endorsements` for chunk B, validator V's cache entry has `chunk_hash_A ≠ chunk_hash_B` and is filtered out. Validator V's endorsement for chunk B is effectively suppressed. [8](#0-7) 

If enough validators are targeted, the chunk fails to reach the endorsement threshold, corrupting the `ChunkEndorsementsState` (the `is_endorsed` decision) and potentially stalling block production.

### Likelihood Explanation

An unprivileged network peer connected to the P2P network can receive broadcast endorsement messages from validators. The attacker needs two valid endorsements from the same validator (for different chunks at different heights or shards), which are routinely produced as the validator endorses chunks across epochs. The forged message must arrive at the block producer before the legitimate one — achievable by a well-connected peer. No validator key or privileged access is required; only passive observation of P2P traffic and message construction.

### Recommendation

Bind `inner` and `metadata` into a single signed payload so that neither can be substituted independently. Replace the two-signature scheme with a single signature over a combined hash:

```rust
fn sign_combined(chunk_hash: &ChunkHash, metadata: &ChunkEndorsementMetadata, signer: &ValidatorSigner) -> Signature {
    let combined = borsh::to_vec(&(chunk_hash, metadata)).unwrap();
    signer.sign_bytes(&combined)
}
```

Alternatively, include the `chunk_hash` inside `ChunkEndorsementMetadata` so that the metadata signature commits to the specific chunk being endorsed, making cross-endorsement splicing impossible.

### Proof of Concept

1. Attacker runs a network node and connects to validator V as a peer.
2. Attacker observes `ChunkEndorsementV2` E_A from V for chunk A at height H1, shard S1.
3. Attacker observes `ChunkEndorsementV2` E_B from V for chunk B at height H2, shard S2.
4. Attacker constructs forged M: `{ inner: E_A.inner, signature: E_A.signature, metadata: E_B.metadata, metadata_signature: E_B.metadata_signature }`.
5. Attacker forwards M to the block producer node before E_B arrives.
6. Block producer calls `process_chunk_endorsement(M)`: `verify()` passes (both sigs valid under V's key), `validate_chunk_relevant_as_validator` passes (V is assigned to chunk B's slot), M is stored under `(chunk_B_key, V)` with `chunk_hash_A`.
7. E_B arrives; deduplication check finds `(chunk_B_key, V)` already present → E_B is silently dropped.
8. `collect_chunk_endorsements` for chunk B filters out V's entry (`chunk_hash_A ≠ chunk_hash_B`); V's stake is not counted toward endorsement. [9](#0-8) [10](#0-9)
