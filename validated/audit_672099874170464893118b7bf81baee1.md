### Title
Gas Undercharge in `bls12381_impl!` Macro When Input Is Read from a Register - (`File: runtime/near-vm-runner/src/logic/bls12381.rs`)

### Summary
The `bls12381_impl!` macro, which generates eight BLS12-381 host functions (`bls12381_p1_sum`, `bls12381_p2_sum`, `bls12381_g1_multiexp`, `bls12381_g2_multiexp`, `bls12381_p1_decompress`, `bls12381_p2_decompress`, `bls12381_map_fp_to_g1`, `bls12381_map_fp2_to_g2`), computes the per-element gas charge from `value_len` **before** reading the actual data. When a contract passes input via a register (`value_ptr == u64::MAX`), `value_len` is the register ID, not the data length. The macro divides the register ID by `ITEM_SIZE` to get `elements_count`, which is almost always zero for small register IDs, so the expensive per-element gas is never charged even though the full computation runs on all bytes in the register.

### Finding Description
NEAR's host-function calling convention allows a contract to pass data either from linear memory (`value_ptr` = memory address, `value_len` = byte length) or from a register (`value_ptr == u64::MAX`, `value_len` = register ID). The `bls12381_impl!` macro does not distinguish between these two modes when computing gas:

```rust
// runtime/near-vm-runner/src/logic/bls12381.rs  lines 43-48
self.result_state.gas_counter.pay_base($bls12381_base)?;

let elements_count = value_len / $ITEM_SIZE;          // ← value_len may be a register ID
self.result_state.gas_counter.pay_per($bls12381_element, elements_count as u64)?;

let data = get_memory_or_register!(self, value_ptr, value_len)?;  // ← actual data read here
```

When `value_ptr == u64::MAX`, `value_len` is the register ID (e.g., `0`, `1`, `2`). Dividing a small register ID by `ITEM_SIZE` (97, 128, 193, 224, 48, or 96) yields zero, so `pay_per` charges for zero elements. The data is then read from the register and passed to the implementation function (e.g., `p1_sum`), which calls `check_input_size` and, if the register contents are a valid multiple of `ITEM_SIZE`, processes every element without any per-element gas having been charged.

By contrast, `bls12381_pairing_check` (which is **not** generated by the macro) correctly reads the data first and then computes `elements_count` from `data.len()`:

```rust
// runtime/near-vm-runner/src/logic/logic.rs  lines 1436-1439
let data = get_memory_or_register!(self, value_ptr, value_len)?;
let elements_count = data.len() / ITEM_SIZE;          // ← correct: uses actual length
self.result_state.gas_counter.pay_per(bls12381_pairing_element, elements_count as u64)?;
```

The same macro-level defect is duplicated in the wasmtime runner:

```rust
// runtime/near-vm-runner/src/wasmtime_runner/logic.rs  lines 53-54
let elements_count = value_len / $ITEM_SIZE;
ctx.result_state.gas_counter.pay_per($bls12381_element, elements_count as u64)?;
```

### Impact Explanation
The per-element gas parameters for BLS12-381 operations are the dominant cost of those host functions. A contract that stores large BLS12-381 data in a register (produced by any prior host call that writes to a register) and then invokes one of the eight affected functions in register mode pays only the flat base cost, regardless of how many elements are processed. This breaks the gas accounting invariant that every unit of validator compute must be paid for. The corrupted protocol value is the **gas charged** for BLS12-381 host function execution: it is systematically under-reported relative to actual CPU work performed.

### Likelihood Explanation
Any unprivileged contract author can trigger this by:
1. Populating a register with valid BLS12-381-encoded data (e.g., by calling `sha256` or any other host function that writes to a register, or by reading from contract memory into a register).
2. Calling `bls12381_p1_sum(0, u64::MAX, output_register)` (or any of the seven other affected functions) with register ID `0` as `value_len`.

No special privileges are required. The technique is straightforward and the savings in per-element gas grow linearly with the number of elements in the register.

### Recommendation
Move the `elements_count` computation to after the data is read, mirroring the correct pattern already used in `bls12381_pairing_check`:

```rust
// Corrected bls12381_impl! macro body
self.result_state.gas_counter.pay_base($bls12381_base)?;

let data = get_memory_or_register!(self, value_ptr, value_len)?;  // read first

let elements_count = data.len() / $ITEM_SIZE;                     // then compute count
self.result_state.gas_counter.pay_per($bls12381_element, elements_count as u64)?;
```

Apply the same fix to the wasmtime runner's copy of the macro in `runtime/near-vm-runner/src/wasmtime_runner/logic.rs`.

### Proof of Concept
```rust
// Inside a NEAR contract:
pub fn exploit(&mut self) {
    // Step 1: write 1000 valid BLS12-381 G1 points (97 bytes each = 97,000 bytes)
    // into register 0 via any host call that produces output in a register.
    // For example, a prior call to bls12381_p1_sum with memory input writes its
    // result to a register; or the contract can use input() + read_register().
    // Here we assume register 0 already holds 97,000 bytes of valid p1_sum input.

    // Step 2: call bls12381_p1_sum with value_ptr = u64::MAX (register mode),
    //         value_len = 0 (register ID 0).
    // Gas charged: pay_base + pay_per(element, 0/97) = pay_base + 0
    // Actual work: processes all 1000 elements in register 0.
    env::bls12381_p1_sum(
        0,           // value_len = register ID 0
        u64::MAX,    // value_ptr = sentinel for register mode
        1,           // output register
    );
    // 1000 expensive BLS12-381 additions performed; zero per-element gas paid.
}
```
