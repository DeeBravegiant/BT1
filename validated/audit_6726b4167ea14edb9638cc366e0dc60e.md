### Title
Backup-CLI `secrets.json` Written Without Restrictive File Permissions, Exposing Key Material and Keyshare Decryption Key - (File: crates/backup-cli/src/adapters/secrets_storage.rs)

### Summary
The `backup-cli`'s `JsonSecretsStorage::open_write()` creates `secrets.json` using `tokio::fs::File::options()` without setting an explicit file mode. On Linux, this defaults to the process umask (typically `0o022`), producing a world-readable `0o644` file. The file contains the backup service's `p2p_private_key` (used for mTLS authentication with MPC nodes), `near_signer_key`, and `local_storage_aes_key` (used to decrypt locally stored keyshares). Separately, the permanent and temporary keyshare files written by `LocalPermanentKeyStorageBackend::store()` and `PendingKeyshareStorageHandle::commit_keyshare()` are also created without explicit permissions, making the AES-encrypted keyshare blobs world-readable. An attacker with local filesystem access can read `secrets.json` to obtain `local_storage_aes_key`, then read and decrypt the world-readable keyshare files, obtaining a complete MPC participant's key share.

### Finding Description
`JsonSecretsStorage::<File>::open_write()` in `crates/backup-cli/src/adapters/secrets_storage.rs` opens the file as:

```rust
File::options()
    .create(true)
    .truncate(true)
    .write(true)
    .open(file_path)
    .await
```

No `.mode(0o600)` call is made. [1](#0-0) 

The file written is `secrets.json`, which serializes `types::PersistentSecrets` containing `p2p_private_key`, `near_signer_key`, and `local_storage_aes_key`. [2](#0-1) 

This is called from `backup::run_command` during the `GenerateKeys` subcommand, which is the documented first step for any node operator setting up the backup service. [3](#0-2) 

The MPC node's own `secrets.json` correctly uses `write_secret_file` which explicitly sets `0o600` and calls `fs::set_permissions` afterward. The backup-cli has no equivalent protection. [4](#0-3) 

Compounding this, the permanent keyshare files stored by the backup-cli (via `LocalPermanentKeyStorageBackend::store()`) are also created with `tokio::fs::File::create_new()` without any explicit mode, making the AES-128-GCM encrypted blobs world-readable. [5](#0-4) 

Temporary keyshare files written by `PendingKeyshareStorageHandle::commit_keyshare()` have the same problem. [6](#0-5) 

The `local_storage_aes_key` in `secrets.json` is exactly the key used to decrypt these stored keyshare blobs. [7](#0-6) 

### Impact Explanation
An attacker with read access to the backup-cli operator's home directory can:
1. Read the world-readable `secrets.json` and extract `local_storage_aes_key`.
2. Read the world-readable encrypted keyshare files under `permanent_keys/`.
3. Decrypt the keyshare blobs using AES-128-GCM with the extracted key, recovering a complete MPC participant's `KeyshareData` (ECDSA `private_share`, EdDSA share, or CKD share).

Additionally, the exposed `p2p_private_key` is the credential used for mTLS authentication with MPC nodes. Since the backup service's public key is registered in the MPC smart contract, an attacker holding this private key can impersonate the registered backup service and request keyshares from MPC nodes over the migration endpoint. [8](#0-7) 

This constitutes unauthorized access to MPC key shares and secret material. While a single participant's share is below the signing threshold, it is a direct, unauthenticated read of secret key material that is a necessary component of the distributed key and materially advances secret recovery when combined with shares from other compromised participants.

### Likelihood Explanation
The backup-cli is a production tool documented for use in node migration on mainnet/testnet. [9](#0-8) 

The default Linux umask of `0o022` produces `0o644` files, making this world-readable on any standard deployment. No special attacker capability is required beyond local user access to the same machine — a realistic threat on shared infrastructure, container escape, or any co-tenant scenario. The MPC node's own `secrets.json` was explicitly hardened with `0o600`, demonstrating awareness of the risk, but the backup-cli was not given the same treatment.

### Recommendation
**Short term:** Replace the `File::options()` call in `JsonSecretsStorage::<File>::open_write()` with an `OpenOptionsExt::mode(0o600)` call (using `std::os::unix::fs::OpenOptionsExt`) to create `secrets.json` with owner-only permissions. Apply the same fix to `LocalPermanentKeyStorageBackend::store()` and `PendingKeyshareStorageHandle::commit_keyshare()` for the keyshare files, and to `LocalPermanentKeyStorageBackend::new()` for the `permanent_keys/` directory (`0o700`).

**Long term:** On startup, the backup-cli should verify the permissions of `secrets.json` and refuse to proceed (or emit a hard warning) if the file is readable by group or others, mirroring the long-term recommendation from the original report.

### Proof of Concept
```
# 1. Operator runs: backup-cli --home-dir /home/operator/backup generate-keys
#    This creates /home/operator/backup/secrets.json with mode 0o644 (world-readable).

# 2. Attacker (different local user) reads the file:
cat /home/operator/backup/secrets.json
# Output: {"p2p_private_key":"...","near_signer_key":"...","local_storage_aes_key":[...]}

# 3. Operator later runs: backup-cli get-keyshares ...
#    This stores encrypted keyshares at /home/operator/backup/permanent_keys/<id>
#    with mode 0o644 (world-readable).

# 4. Attacker reads the encrypted keyshare blob and decrypts it using
#    the extracted local_storage_aes_key (AES-128-GCM), recovering the
#    full KeyshareData (private_share scalar for ECDSA, etc.) for that
#    MPC participant.
```

The root cause is the absence of `.mode(0o600)` in `JsonSecretsStorage::<File>::open_write()` at `crates/backup-cli/src/adapters/secrets_storage.rs:57-62`, and the absence of equivalent permission-setting in `LocalPermanentKeyStorageBackend::store()` at `crates/node/src/keyshare/local.rs:49` and `PendingKeyshareStorageHandle::commit_keyshare()` at `crates/node/src/keyshare/temporary.rs:141`.

/// Writes data to a file with owner-only permissions (0o600).
/// Permissions are set explicitly after writing to ensure correctness
/// even if the file already exists with different permissions.
fn write_secret_file(path: &Path, data: &[u8]) -> anyhow::Result<()> {
    let mut file = fs::OpenOptions::new()
        .write(true)
        .create(true)
        .truncate(true)
        .mode(0o600)
        .open(path)
        .with_context(|| format!("failed to create secret file {}", path.display()))?;
    file.write_all(data)
        .with_context(|| format!("failed to write secret file {}", path.display()))?;
    fs::set_permissions(path, fs::Permissions::from_mode(0o600)).with_context(|| {
        format!(
            "failed to set permissions on secret file {}",
            path.display()
        )
    })?;
    Ok(())
}

## Step 1: Setup the Backup CLI

First, you'll need to set up the backup CLI tool and generate keys for the backup service.

### Install backup-cli

Install the backup-cli tool using cargo (run from the repository root):

```bash
cargo install --path crates/backup-cli --locked
```

This installs the `backup-cli` binary to your cargo bin directory (typically `~/.cargo/bin`), which should be in your `PATH`.

### Generate Backup Service Keys

Create a home directory for the backup-cli and generate its keys:

```bash
export BACKUP_HOME_DIR=/path/to/backup/home
mkdir -p $BACKUP_HOME_DIR

backup-cli \
  --home-dir $BACKUP_HOME_DIR \
  generate-keys
```

This creates a `secrets.json` file in your backup home directory containing:
- `p2p_private_key`: Used for mutual TLS authentication with MPC nodes
- `local_storage_aes_key`: Used to encrypt keyshares stored locally

**Important:** Keep the `secrets.json` file secure. Anyone with access to this file can authenticate as your backup service and decrypt any keyshares stored locally.
