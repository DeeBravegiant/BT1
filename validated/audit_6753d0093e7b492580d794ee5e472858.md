Audit Report

## Title
Block Stuffing Accumulates Multi-Day Yield Into Single Fee Mint, Triggering `DailyFeeMintLimitExceeded` and Freezing Price Updates — (`contracts/LRTOracle.sol`)

## Summary
An attacker can use block stuffing to suppress calls to `updateRSETHPrice()` for multiple days. When the update is finally executed, the entire accumulated yield is treated as a single-period TVL increase, and the resulting protocol fee mint amount can exceed `maxFeeMintAmountPerDay`. Both the public `updateRSETHPrice()` and the manager-only `updateRSETHPriceAsManager()` share the same internal path through `_checkAndUpdateDailyFeeMintLimit`, so both revert simultaneously, freezing `rsETHPrice` until the manager separately raises the cap.

## Finding Description
`updateRSETHPrice()` is a permissionless public function that delegates to `_updateRsETHPrice()`. [1](#0-0) 

`updateRSETHPriceAsManager()` provides no alternative internal path — it calls the identical `_updateRsETHPrice()`: [2](#0-1) 

Inside `_updateRsETHPrice()`, the TVL delta since the last update is treated as the reward for the current period, with no accounting for how many days have elapsed: [3](#0-2) 

The resulting fee is passed to `_checkAndUpdateDailyFeeMintLimit`. That function correctly resets `currentPeriodMintedFeeAmount` to zero when a new day begins, but then immediately checks the full accumulated fee against the single-day cap: [4](#0-3) 

If N days of yield are presented at once, the fee is N× the expected daily fee, and the check at line 205 reverts with `DailyFeeMintLimitExceeded`. The call site at line 303 is reached by both entry points: [5](#0-4) 

The live asset rate is sourced from `SwETHPriceOracle.getAssetPrice` → `ISwETH.getRate()`, which accumulates continuously regardless of whether `updateRSETHPrice()` is called: [6](#0-5) 

Recovery requires the manager to first call `setMaxFeeMintAmountPerDay` to raise the cap, then retry — a two-step out-of-band intervention: [7](#0-6) 

## Impact Explanation
`rsETHPrice` is not updated while both entry points are bricked. All downstream consumers (deposit pool pricing, withdrawal valuations, pool exchange rates) read a stale price. This matches the explicitly allowed impact: **Low — Block stuffing**.

## Likelihood Explanation
Block stuffing on Ethereum mainnet is expensive but economically rational if the attacker profits from stale pricing (e.g., holds a large short position on rsETH or exploits a downstream protocol reading `rsETHPrice`). The attack requires no privileged access and is fully permissionless. The vulnerability is triggered whenever `maxFeeMintAmountPerDay` is set conservatively relative to the protocol's TVL and yield rate — a realistic configuration. The attack is repeatable: after the manager raises the cap and the update succeeds, the attacker can stuff blocks again.

## Recommendation
In `_checkAndUpdateDailyFeeMintLimit`, instead of reverting when the fee exceeds the daily cap, cap the minted fee at the remaining daily allowance and carry forward the uncollected fee, or skip fee minting for the excess. Alternatively, give `updateRSETHPriceAsManager()` its own internal path that bypasses the daily fee limit check (minting the full fee regardless of the cap), so the manager can always recover from a stuffed-block scenario without needing a separate `setMaxFeeMintAmountPerDay` call first.

## Proof of Concept
```solidity
// Foundry unit test — no mainnet required
function testBlockStuffingDailyFeeLimitDOS() public {
    // 1. Set a small daily fee cap
    lrtOracle.setMaxFeeMintAmountPerDay(1e15);

    // 2. Simulate 7 days of block stuffing — no updateRSETHPrice called
    vm.warp(block.timestamp + 7 days);

    // 3. swETH rate has accrued 7 days of ~4% APR yield
    mockSwETH.setRate(1e18 + 7 * 1.1e13);

    // 4. Public call reverts
    vm.expectRevert(ILRTOracle.DailyFeeMintLimitExceeded.selector);
    lrtOracle.updateRSETHPrice();

    // 5. Manager call also reverts — same internal path
    vm.prank(manager);
    vm.expectRevert(ILRTOracle.DailyFeeMintLimitExceeded.selector);
    lrtOracle.updateRSETHPriceAsManager();
}
```

    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
