### Title
`respond_verify_foreign_tx` Signature Check Does Not Bind `payload_hash` to Request Content, Enabling Cross-Request Replay - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over the caller-supplied `response.payload_hash`, but never checks that `payload_hash` is actually derived from the pending `request`'s content. A single Byzantine attested participant who holds any previously produced root-key signature can replay it as a valid response to an unrelated pending foreign-transaction verification request, delivering a forged attestation to the waiting caller.

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The check only asserts: *"this signature is a valid ECDSA signature over this hash under the root key."* It does **not** assert: *"this hash is the canonical `ForeignTxSignPayload` hash derived from the pending `request`."*

Contrast this with the regular `respond` path, where `payload_hash` is taken directly from the immutable `request.payload` field — the content the user originally committed to:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
).is_ok()
``` [2](#0-1) 

For `verify_foreign_transaction`, the expected hash would be `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: <observed> }).compute_msg_hash()`. The contract never computes or compares this; it blindly trusts whatever hash the responding participant supplies. [3](#0-2) 

The `VerifyForeignTransactionResponse` struct carries both `payload_hash` and `signature` as free caller-supplied fields with no on-chain binding to the originating request.

### Impact Explanation

A single Byzantine attested participant who has previously participated in any legitimate `verify_foreign_transaction` signing round possesses a valid `(H_old, sig_old)` pair — a root-key signature over the hash of some past foreign-chain observation. They can submit:

```
respond_verify_foreign_tx(
    request  = <any currently pending request B>,
    response = { payload_hash: H_old, signature: sig_old }
)
```

The contract will:
1. Confirm the caller is an attested participant ✓
2. Confirm request B is pending ✓
3. Confirm `sig_old` is a valid root-key signature over `H_old` ✓
4. Resolve request B's yield with `{ payload_hash: H_old, signature: sig_old }` ✓ [4](#0-3) 

The caller of request B receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to a completely different foreign-chain observation (request A's data). Any bridge or NEAR contract that does not independently recompute the expected hash and compare it will accept this as a valid attestation of request B's transaction — which was never actually verified. This enables forged foreign-chain verification and invalid bridge execution (e.g., crediting a deposit that never occurred on the foreign chain).

### Likelihood Explanation

The attacker is a single Byzantine attested MPC participant — strictly below the signing threshold. No threshold collusion is required. The participant only needs to have been present for one prior legitimate signing round to obtain a reusable `(H, sig)` pair. There is no nonce, no per-request binding, and no replay-prevention mechanism in the contract. The attack is deterministic and requires no special timing or network conditions.

### Recommendation

In `respond_verify_foreign_tx`, recompute the expected `payload_hash` from the `request` fields and the `values` embedded in the response payload, then assert equality before accepting the response. Concretely, the response should carry the full `ForeignTxSignPayload` (not just its hash), and the contract should call `payload.compute_msg_hash()` and compare it to `response.payload_hash` before the signature check. This mirrors how `respond` binds `payload_hash` to `request.payload` rather than trusting the caller-supplied value.

### Proof of Concept

1. Participant P participates in a legitimate signing round for request A (Bitcoin tx `0xAA…`), obtaining `(H_A, sig_A)` where `H_A = ForeignTxSignPayload::V1 { request: A, values: [BlockHash(0x11…)] }.compute_msg_hash()`.
2. User submits `verify_foreign_transaction` for request B (Bitcoin tx `0xBB…`). Request B is now pending.
3. P calls `respond_verify_foreign_tx(request=B, response={ payload_hash: H_A, signature: sig_A })`.
4. Contract checks: B is pending ✓, `sig_A` valid over `H_A` under root key ✓. Resolves B's yield.
5. User of request B receives `{ payload_hash: H_A, signature: sig_A }` — an attestation that `0xAA…` was observed, delivered as if it were an attestation for `0xBB…`.
6. A bridge contract that only checks `verify_ecdsa_signature(sig_A, H_A, root_pk)` without recomputing the expected hash for `0xBB…` accepts this as proof that `0xBB…` was finalized, and credits the corresponding deposit.
