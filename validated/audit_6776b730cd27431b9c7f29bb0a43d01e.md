Audit Report

## Title
RSETH Shared Daily Mint Limit Blocks Oracle Price Updates When Exhausted by User Deposits — (`contracts/LRTOracle.sol`)

## Summary
`RSETH.mint` applies `checkDailyMintLimit` to every mint unconditionally, including oracle fee mints. When user deposits exhaust `RSETH.maxMintAmountPerDay` within a period, the subsequent call to `IRSETH(rsETHTokenAddress).mint` inside `_updateRsETHPrice()` reverts, rolling back the entire transaction and leaving `rsETHPrice` at its previous stale value until the RSETH period resets (~24 h).

## Finding Description
`_updateRsETHPrice()` sequences three operations when a fee is due:

```
303: _checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee); // oracle limit — passes
306: IRSETH(rsETHTokenAddress).mint(treasury, rsethAmountToMintAsProtocolFee); // RSETH limit — may revert
313: rsETHPrice = newRsETHPrice; // never reached if line 306 reverts
```

`RSETH.mint` carries `checkDailyMintLimit(amount)` with no exemption path for oracle fee mints. The modifier reverts with `DailyMintLimitExceeded` when `currentPeriodMintedAmount + amount > maxMintAmountPerDay`. Because `rsETHPrice = newRsETHPrice` (line 313) is placed after the mint call, a revert at line 306 rolls back the price update entirely. The oracle's own `currentPeriodMintedFeeAmount` increment (from line 303) is also rolled back, so the oracle's internal accounting is self-consistent but the price remains stale.

The deposit-exhaustion trigger requires no privileged action: user deposits throughout the day each call `RSETH.mint`, incrementing `currentPeriodMintedAmount`. Once the cumulative deposit mints reach `maxMintAmountPerDay`, any subsequent call to `updateRSETHPrice` that would mint a non-zero fee will revert. The two limits (`RSETH.maxMintAmountPerDay` and `LRTOracle.maxFeeMintAmountPerDay`) are set independently with no on-chain cross-validation, so the RSETH limit can be exhausted by deposits while the oracle's own limit still has capacity.

## Impact Explanation
**Low — Contract fails to deliver promised returns.** The price oracle cannot update for up to ~24 h (until the RSETH period resets). During this window, new depositors receive slightly more rsETH than they should (diluting existing holders), and protocol fee revenue for that period is permanently lost. No user funds are at direct risk of theft or permanent loss; the protocol is over-collateralised, not insolvent.

## Likelihood Explanation
Moderate. The deposit-exhaustion path requires only that cumulative user deposits within a single RSETH period reach `maxMintAmountPerDay`. On a popular protocol with significant deposit volume, this is plausible without any privileged key compromise or coordination. The condition is repeatable every period.

## Recommendation
Decouple the fee mint from the price update. If `IRSETH.mint` would exceed the RSETH daily limit, skip the fee mint (emit an event) but still execute `rsETHPrice = newRsETHPrice`. One concrete approach: wrap the mint call in a try/catch or pre-check `RSETH.remainingDailyMintLimit()` before attempting the mint, and proceed with the price update regardless. Alternatively, grant the oracle contract a separate minting path in RSETH that bypasses the shared daily cap, or enforce on-chain that `RSETH.maxMintAmountPerDay` accounts for expected oracle fee mints in addition to user deposit mints.

## Proof of Concept
```solidity
// 1. Set RSETH daily limit to a small but non-zero value
rseth.setMaxMintAmountPerDay(100 ether);

// 2. Simulate user deposits that exhaust the limit
// (depositAsset calls RSETH.mint internally)
for (uint i = 0; i < N; i++) {
    lrtDepositPool.depositAsset(asset, amount, minRsETH, referral);
}
// After deposits, currentPeriodMintedAmount >= maxMintAmountPerDay

// 3. Ensure TVL has grown so protocolFeeInETH > 0
// (e.g. asset price increase between periods)

// 4. Call updateRSETHPrice — expect revert from RSETH.checkDailyMintLimit
uint256 previousPrice = lrtOracle.rsETHPrice();
vm.expectRevert(
    abi.encodeWithSelector(RSETH.DailyMintLimitExceeded.selector, ...)
);
lrtOracle.updateRSETHPrice();

// 5. Assert price was NOT updated
assertEq(lrtOracle.rsETHPrice(), previousPrice);
```

Revert path: `updateRSETHPrice` → `_updateRsETHPrice` → `IRSETH.mint` → `checkDailyMintLimit` → `revert DailyMintLimitExceeded`.

        // Check if we need to reset the period if it has been more than 24 hours
        if (block.timestamp >= periodStartTime + 1 days) {
            currentPeriodMintedAmount = 0;
            periodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit

        _;
    }
