### Title
Cross-wallet derivation-path poisoning via `set_used_up_to` missing `wallet_id` filter — (`chia/wallet/wallet_puzzle_store.py`)

### Summary

`WalletPuzzleStore.set_used_up_to` issues an `UPDATE` with no `wallet_id` predicate. Any call triggered for wallet_id=1 (standard wallet) at index N silently marks all other wallets' (e.g., CAT wallet_id=2) derivation paths at indices 0..N as `used=1`, permanently advancing their "used" pointer without those wallets ever receiving a coin.

### Finding Description

`set_used_up_to` at line 157 executes:

```sql
UPDATE derivation_paths SET used=1 WHERE derivation_index<=?
``` [3](#0-2) 

The `derivation_paths` table holds records for all wallets. Different wallet IDs share the same integer derivation index space (e.g., both wallet_id=1 and wallet_id=2 have rows at index 0, 1, 2, …). The PRIMARY KEY is `(puzzle_hash, wallet_id)`, confirming co-existence of multiple wallets at the same index. [4](#0-3) 

When the wallet sync path processes a coin state for a puzzle_hash belonging to wallet_id=1 at derivation_index=N and calls `set_used_up_to(N)`, the UPDATE touches every wallet's rows at indices 0..N, not just wallet_id=1's rows.

The read-side queries (`get_unused_derivation_path_for_wallet`, `get_current_derivation_record_for_wallet`) do correctly filter by `wallet_id`: [5](#0-4) 

This means after the poisoning write, those reads will correctly see wallet_id=2's paths as used — the damage is real and persistent.

### Impact Explanation

After poisoning, `get_unused_derivation_path_for_wallet(wallet_id=2)` returns the first index **above** N, or `None` if no paths beyond N were pre-generated. The CAT wallet (or any secondary wallet) stops issuing puzzle hashes from index 0 onward, causing it to miss all incoming CAT coins sent to those addresses. Wallet sync state for the affected wallet is permanently corrupted until the user manually re-derives paths or restores from seed. This matches the **High** impact category: corruption of wallet sync state with direct security impact (missed/untracked coins).

### Likelihood Explanation

The trigger path is reachable from an unprivileged remote peer:

1. A malicious peer (or a peer relaying attacker-crafted data) sends a `CoinState` for a coin whose `puzzle_hash` is a known derivation of the victim wallet's standard wallet (wallet_id=1) at a high index N. The wallet's puzzle hashes are derived from its public key, which is observable on-chain.
2. The wallet's sync handler (`_add_coin_states` in `wallet_state_manager.py`) processes the coin state, looks up the derivation record, and calls `set_used_up_to(N)`.
3. The unfiltered UPDATE marks all wallet_id=2 paths at 0..N as used.

No key material, admin access, or broken cryptography is required. The attacker only needs to know the victim's public key (public) and send a valid-looking coin state via the peer protocol.

### Recommendation

Add a `wallet_id` parameter to `set_used_up_to` and include it in the WHERE clause:

```python
async def set_used_up_to(self, index: uint32, wallet_id: uint32) -> None:
    async with self.db_wrapper.writer_maybe_transaction() as conn:
        await conn.execute_insert(
            "UPDATE derivation_paths SET used=1 WHERE derivation_index<=? AND wallet_id=?",
            (index, wallet_id),
        )
```

All call sites in `wallet_state_manager.py` and `wsm_apis.py` must be updated to pass the correct `wallet_id`. [1](#0-0) 

### Proof of Concept

1. Create a wallet with wallet_id=1 (standard) and wallet_id=2 (CAT), both pre-deriving paths 0..10.
2. Assert wallet_id=2 has `used=0` for all paths 0..10.
3. Trigger `set_used_up_to(uint32(5))` (as would happen when a coin state for wallet_id=1 at index 5 is processed).
4. Query `SELECT used FROM derivation_paths WHERE wallet_id=2 AND derivation_index<=5`.
5. All rows return `used=1` — wallet_id=2's paths are poisoned despite never receiving a coin.
6. `get_unused_derivation_path_for_wallet(uint32(2))` now returns 6 instead of 0, confirming the wallet will skip addresses 0–5 for future receives.
