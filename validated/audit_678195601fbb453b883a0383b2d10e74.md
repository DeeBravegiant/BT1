Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` applies asymmetric validation logic: for LST assets it checks `totalAssetDeposits + amount > limit`, but for ETH it checks only `totalAssetDeposits > limit`, entirely discarding the incoming `amount`. Any unprivileged caller can invoke `depositETH` with an arbitrarily large `msg.value` and bypass the ETH deposit cap, causing the protocol to accept ETH deposits far beyond the configured ceiling and mint rsETH for the excess.

## Finding Description
The root cause is in `_checkIfDepositAmountExceedesCurrentLimit` at `contracts/LRTDepositPool.sol` lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // `amount` ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch evaluates only whether the cap has **already been exceeded** by prior deposits; it never considers the size of the incoming deposit. The LST branch correctly adds `amount` to `totalAssetDeposits` before comparing.

The call path is: `depositETH` (L76–93) → `_beforeDeposit` (L648–670) → `_checkIfDepositAmountExceedesCurrentLimit`. The only guard that could block an oversized ETH deposit is this check; `_beforeDeposit` reverts with `MaximumDepositLimitReached` only when the function returns `true`, which it never does while `totalAssetDeposits ≤ limit`. No other check in `depositETH` or `_beforeDeposit` constrains `msg.value` relative to the configured limit.

`depositLimitByAsset` is a live, admin-settable mapping in `LRTConfig` (L23) and is the sole mechanism for per-asset deposit caps.

## Impact Explanation
The protocol fails to enforce its own ETH deposit limit, a core risk-management control. Any depositor can push ETH holdings arbitrarily beyond the configured ceiling in a single transaction, and rsETH is minted for the full over-limit amount. This maps to **Low — Contract fails to deliver promised returns, but doesn't lose value**: the deposit cap is a protocol-promised constraint that is silently not honored for ETH, while no direct theft or freezing of funds occurs as a direct consequence of the bypass itself.

## Likelihood Explanation
Exploitable by any unprivileged external caller with no special setup. The only precondition is that `totalAssetDeposits(ETH_TOKEN) ≤ depositLimitByAsset(ETH_TOKEN)`, which is the normal operating state of the protocol. The attacker needs only to call `depositETH{value: X}(0, "")` with `X` exceeding the cap. Repeatable at any time the cap has not already been breached.

## Recommendation
Remove the ETH-specific branch and apply the same amount-inclusive check uniformly:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This ensures the incoming deposit amount is always evaluated against the remaining capacity, regardless of asset type.

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether` via `LRTConfig.updateAssetDepositLimit`.
2. Protocol starts fresh: `getTotalAssetDeposits(ETH_TOKEN) == 0`.
3. Attacker calls `LRTDepositPool.depositETH{value: 10_000 ether}(0, "")`.
4. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)`.
5. ETH branch executes: `return (0 > 100 ether)` → `false`; no revert.
6. `_mintRsETH` mints rsETH for the full `10_000 ether`; the ETH cap of `100 ether` is bypassed by 100×.

**Foundry test sketch:**
```solidity
function test_ethDepositLimitBypass() public {
    // set limit to 100 ether
    vm.prank(manager);
    lrtConfig.updateAssetDepositLimit(LRTConstants.ETH_TOKEN, 100 ether);

    address attacker = makeAddr("attacker");
    vm.deal(attacker, 10_000 ether);

    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 10_000 ether}(0, "");

    // assert total ETH deposits now exceed the limit
    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), 100 ether);
}
```
