### Title
Cross-chain replay of `DelegateAction` meta-transactions via missing chain ID in `SignableMessage` — (`File: core/primitives/src/signable_message.rs`)

---

### Summary

The `SignableMessage` scheme used to sign `DelegateAction` (meta-transactions, NEP-366) and `DelegateActionV2` (NEP-611) does not include a chain ID or genesis hash in the signed payload. A malicious relayer can capture a user's signed `DelegateAction` from one NEAR network (e.g., testnet) and replay it on another network (e.g., mainnet) or a forked chain by wrapping it in a fresh outer `SignedTransaction` with a valid block hash on the target chain. The user's inner signature remains valid because the signed bytes are identical across chains.

---

### Finding Description

`DelegateAction::get_nep461_hash()` computes the hash that the user signs:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
``` [1](#0-0) 

`SignableMessage` is defined as:

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub msg: &'a T,
}
``` [2](#0-1) 

The `discriminant` is only a NEP number (`366` for `DelegateAction`, `611` for `DelegateActionV2`):

```rust
SignableMessageType::DelegateAction =>
    MessageDiscriminant::new_on_chain(NEP_366_META_TRANSACTIONS).unwrap()
``` [3](#0-2) 

There is **no chain ID, genesis hash, or network identifier** anywhere in the signed payload. The `DelegateAction` struct itself contains only `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key`: [4](#0-3) 

The outer `SignedTransaction` that carries the `DelegateAction` does contain a `block_hash` field (which is chain-specific), but **this field is provided and signed by the relayer, not the user**. The user only signs the inner `DelegateAction`. A malicious relayer can therefore:

1. Receive a user's signed `DelegateAction` intended for testnet.
2. Construct a new outer `SignedTransaction` on mainnet with a valid mainnet `block_hash`.
3. Embed the original signed `DelegateAction` unchanged inside it.
4. Submit to mainnet — the inner signature verifies correctly because the signed bytes are network-agnostic.

Verification in `SignedDelegateAction::verify()` only checks the NEP discriminant and the action payload, never the chain:

```rust
pub fn verify(&self) -> bool {
    let hash = delegate_action.get_nep461_hash();
    self.signature.verify(hash.as_ref(), public_key)
}
``` [5](#0-4) 

The `max_block_height` field is a block **height** (not a hash), so it does not bind the action to a specific chain — both mainnet and testnet share the same block height namespace.

---

### Impact Explanation

A malicious relayer can replay a user's `DelegateAction` on a different NEAR network or a forked chain, causing the user's actions (token transfers, contract calls, `AddKey`, `DeleteKey`, `DeleteAccount`) to execute on the wrong chain. The corrupted protocol value is the **account state** (balance, nonce, access keys) on the target chain — specifically, the nonce is advanced and the action's effects are applied without the user's consent on that chain.

---

### Likelihood Explanation

Many NEAR users have accounts with the same account ID on both mainnet and testnet, often using the same key pair. The nonce on the target chain only needs to be strictly less than the nonce embedded in the replayed `DelegateAction` (monotonic mode) or exactly one less (strict mode). An attacker who operates a relayer service is in a privileged position to observe and selectively replay signed delegate actions. The `max_block_height` provides a time window but not chain binding.

---

### Recommendation

Include the chain ID (genesis hash or the `chain_id` string from genesis config) in the `SignableMessage` payload or in the `MessageDiscriminant` computation, so that a signature produced on one NEAR network is cryptographically invalid on any other network.

---

### Proof of Concept

1. User on testnet signs a `DelegateAction`: transfer 100 NEAR to `bob.testnet`, `nonce=5`, `max_block_height=10_000_000`, using key `ed25519:XYZ`.
2. The same user has account on mainnet with key `ed25519:XYZ` and current nonce `3` (so nonce `5` is valid).
3. Malicious relayer captures the signed `DelegateAction` from testnet.
4. Relayer constructs a new outer `SignedTransaction` on mainnet:
   - `block_hash` = a recent valid mainnet block hash (relayer's own)
   - `actions` = `[Action::Delegate(signed_delegate_action)]` (the captured one, unchanged)
5. Relayer signs and submits the outer transaction to mainnet.
6. Mainnet runtime calls `SignedDelegateAction::verify()` → passes (signature is over chain-agnostic bytes).
7. The transfer of 100 NEAR executes on mainnet, draining the user's mainnet account.

The root cause is in `core/primitives/src/signable_message.rs` (`SignableMessage` struct) and `core/primitives/src/action/delegate.rs` (`get_nep461_hash`), which together produce a chain-agnostic signed digest for all meta-transactions. [6](#0-5) [5](#0-4)

#[derive(BorshSerialize)]
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub msg: &'a T,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash)]
#[non_exhaustive]
pub enum SignableMessageType {
    /// A delegate action, intended for a relayer to included it in an action list of a transaction.
    DelegateAction,
    /// A delegate action with gas key support, intended for a relayer to include it in an action
    /// list of a transaction.
    DelegateActionV2,
}

#[derive(thiserror::Error, Debug)]
#[non_exhaustive]
pub enum ReadDiscriminantError {
    #[error("does not fit any known categories")]
    UnknownMessageType,
    #[error("NEP {0} does not have a known on-chain use")]
    UnknownOnChainNep(u32),
    #[error("NEP {0} does not have a known off-chain use")]
    UnknownOffChainNep(u32),
    #[error("discriminant is in the range for transactions")]
    TransactionFound,
}

#[derive(thiserror::Error, Debug)]
#[non_exhaustive]
pub enum CreateDiscriminantError {
    #[error("nep number {0} is too big")]
    NepTooLarge(u32),
}

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
}
