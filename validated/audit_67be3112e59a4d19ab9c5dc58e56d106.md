Audit Report

## Title
Unguarded `receive()` in `LRTUnstakingVault` Enables Forced Protocol-Fee Minting, Stealing Unclaimed Yield from rsETH Holders — (`contracts/LRTUnstakingVault.sol`)

## Summary
`LRTUnstakingVault.receive()` accepts ETH from any caller with no access control. Because `LRTDepositPool.getETHDistributionData()` reads `lrtUnstakingVault.balance` directly, any ETH sent to the vault is immediately reflected in the protocol's TVL. When the permissionless `LRTOracle.updateRSETHPrice()` is subsequently called, the inflated TVL is treated as earned yield, causing the protocol to levy its fee on ETH that was never staking yield — minting excess rsETH to the treasury and permanently diverting yield that should have accrued to existing rsETH holders.

## Finding Description

**Root cause — unguarded `receive()`:**

`LRTUnstakingVault.receive()` has no role check, whitelist, or guard: [1](#0-0) 

**Raw `.balance` read in TVL accounting:**

`LRTDepositPool.getETHDistributionData()` reads the vault's balance directly from `lrtUnstakingVault.balance`: [2](#0-1) 

This value flows into `getTotalAssetDeposits(ETH)` as `assetLyingUnstakingVault`: [3](#0-2) 

**Permissionless price update uses inflated TVL:**

`LRTOracle.updateRSETHPrice()` is callable by anyone when not paused: [4](#0-3) 

Inside `_updateRsETHPrice()`, `_getTotalEthInProtocol()` calls `getTotalAssetDeposits` for every supported asset, picking up the inflated vault balance: [5](#0-4) 

**Fee logic treats any TVL increase as earned yield:**

Any increase over `previousTVL` (computed as `rsethSupply × rsETHPrice`) triggers fee collection: [6](#0-5) 

The resulting `protocolFeeInETH` is converted to rsETH and minted to the treasury: [7](#0-6) 

**Why existing guards are insufficient:**

- `pricePercentageLimit`: Reverts if the price increase exceeds the configured threshold, but the attacker can bypass this by keeping each donation small enough to stay under the threshold, repeating across multiple calls.
- `maxFeeMintAmountPerDay`: Caps daily fee minting, but this is a protocol-operational parameter set by the manager to allow normal fee accrual — it is not a security boundary. The attacker can repeat the attack across multiple days.
- `_checkAndUpdateDailyFeeMintLimit`: Only reverts if `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`. In normal operation, `maxFeeMintAmountPerDay` is set to a non-zero value, so small donations pass through.

## Impact Explanation

**High — Theft of unclaimed yield.**

Every ETH sent by the attacker to `LRTUnstakingVault` causes the protocol to levy its fee on that ETH as if it were staking yield. At a 10% protocol fee, sending 100 ETH causes 10 ETH worth of rsETH to be minted to the treasury rather than accruing to existing holders. The donated ETH is real backing, so the rsETH price does rise — but a portion of that rise is captured as protocol fee instead of accruing to existing holders. Existing rsETH holders permanently lose yield equal to `donatedETH × protocolFeeInBPS / 10_000`. This is a concrete, on-chain diversion of yield from existing rsETH holders to the treasury, matching the "Theft of unclaimed yield" impact class.

## Likelihood Explanation

The attack requires only two permissionless actions: a plain ETH transfer to `LRTUnstakingVault` and a call to `LRTOracle.updateRSETHPrice()`. No privileged role, no front-running dependency, and no external protocol assumption is needed. The attacker's only cost is the donated ETH (which is permanently lost), but the harm to rsETH holders is real and irreversible. The `pricePercentageLimit` check can be bypassed by keeping each donation small. The `maxFeeMintAmountPerDay` limit can be bypassed by spreading donations across multiple days. Any external actor with ETH can execute this attack repeatedly.

## Recommendation

1. **Restrict `receive()` in `LRTUnstakingVault`** to only accept ETH from known, trusted senders (e.g., `LRTDepositPool`, `NodeDelegator`, EigenLayer withdrawal contracts). Revert on unexpected senders.
2. **Track deposited ETH internally** rather than relying on raw `address(this).balance`. Maintain an accounting variable incremented only by authorised `receive*` functions, and use that variable in `getETHDistributionData()` instead of `lrtUnstakingVault.balance`.
3. Apply the same fix to `LRTDepositPool.getETHDistributionData()` line 480, which also reads `address(this).balance` directly.

## Proof of Concept

```solidity
// Fork test (Foundry, mainnet fork)
function testAttackInflatesTVLAndStealsFee() public {
    uint256 rsethSupplyBefore  = rsETH.totalSupply();
    uint256 treasuryBalBefore  = rsETH.balanceOf(treasury);
    uint256 priceBefore        = lrtOracle.rsETHPrice();

    // Step 1: Attacker sends 10 ETH directly to LRTUnstakingVault (no role required)
    vm.deal(attacker, 10 ether);
    vm.prank(attacker);
    (bool ok,) = address(lrtUnstakingVault).call{value: 10 ether}("");
    require(ok, "ETH transfer failed");

    // Step 2: Anyone calls updateRSETHPrice() — no role required
    lrtOracle.updateRSETHPrice();

    // Step 3: Treasury received rsETH it should not have
    uint256 treasuryMinted = rsETH.balanceOf(treasury) - treasuryBalBefore;
    assertGt(treasuryMinted, 0, "treasury received excess rsETH from donated ETH");

    // Step 4: rsETH price rose, but a portion of the donated ETH was taken as fee
    // rather than accruing entirely to existing holders
    uint256 priceAfter = lrtOracle.rsETHPrice();
    assertGt(priceAfter, priceBefore);

    // Existing holders' share of the donated ETH was diluted by treasuryMinted rsETH
}
```
