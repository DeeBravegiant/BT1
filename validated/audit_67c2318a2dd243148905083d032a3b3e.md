### Title
Unvalidated `builder` Field in `ProposalInit` Allows Byzantine Proposer to Inject Arbitrary Sequencer Address into Execution Context and Block Hash — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` validates many fields of `ProposalInit` (height, timestamp, starknet_version, l1_da_mode, l2_gas_price_fri, all four L1 gas prices, version_constant_commitment, fee_proposal_fri) but **never validates the `builder` field**. The `builder` field is passed directly to `convert_to_sn_api_block_info` as `sequencer_address`, which flows into the blockifier's `BlockInfo` for every transaction executed in the block, into the `get_sequencer_address` syscall result, into fee collection, and into `PartialBlockHashComponents` (and therefore the final block hash). A Byzantine consensus proposer can set `builder` to any arbitrary address; validators will accept the proposal, execute all transactions with the wrong sequencer address, and commit a block hash that encodes that address.

### Finding Description

`ProposalInit` carries a `builder: ContractAddress` field: [1](#0-0) 

`is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, all four L1 gas prices, and `fee_proposal_fri`, but contains no check on `builder`: [2](#0-1) 

`ProposalInitValidation` — the struct that carries the validator's locally-derived reference values — has no `builder` field at all: [3](#0-2) 

After `is_proposal_init_valid` passes, `initiate_validation` calls `convert_to_sn_api_block_info(&init)`, which maps `init.builder` directly to `sequencer_address`: [4](#0-3) 

That `BlockInfo` is forwarded to the batcher via `ValidateBlockInput`: [5](#0-4) 

The batcher uses `sequencer_address` for every transaction's execution context (including the `get_sequencer_address` syscall) and for fee collection. It also feeds into `PartialBlockHashComponents`: [6](#0-5) 

which is hashed into the final block hash: [7](#0-6) 

The end-of-validation commitment check (`built_block != received_fin.proposal_commitment`) does **not** catch this because both the proposer and the validator compute the commitment using the same attacker-supplied `init.builder` value: [8](#0-7) 

### Impact Explanation

A Byzantine consensus proposer sets `builder` to an arbitrary address `X`. Every validator:
1. Passes `is_proposal_init_valid` (no `builder` check).
2. Executes all transactions with `sequencer_address = X` — every `get_sequencer_address` syscall returns `X` instead of the legitimate sequencer address.
3. Collects sequencer fees to `X`.
4. Computes `PartialBlockHashComponents` with `sequencer = X`, producing a block hash that encodes the wrong sequencer address.
5. The `ProposalFin` commitment check passes because both sides used `X`.

Consensus reaches agreement on a block with a corrupted sequencer address, wrong syscall results, and misdirected fees. This matches:
- **Critical**: wrong syscall result (`get_sequencer_address`) from execution logic for accepted input.
- **Critical**: incorrect fee/balance effect with economic impact (fees to wrong address).

### Likelihood Explanation

Requires a Byzantine consensus proposer — a validator that is a legitimate participant in the consensus set but acts maliciously. In a multi-validator BFT deployment (the direction of the codebase), up to `f < n/3` Byzantine validators are assumed. Any one of them, when selected as proposer, can trigger this with a single crafted `ProposalInit`. No other preconditions are needed; the attack is a single-message injection.

### Recommendation

Add `builder` to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`:

```rust
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub builder: ContractAddress,   // add this
    // ... existing fields
}
```

```rust
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The validator's `builder` reference value should be sourced from the node's own configuration (the same source used by `initiate_build` when constructing `ProposalInit`): [9](#0-8) 

### Proof of Concept

1. Byzantine node is selected as proposer for height H.
2. It constructs `ProposalInit` with `builder = ContractAddress(0xdeadbeef)` (any address other than the legitimate sequencer).
3. It sends this `ProposalInit` to all validators.
4. Each validator calls `is_proposal_init_valid` — all checks pass (no `builder` check exists).
5. `initiate_validation` calls `convert_to_sn_api_block_info(&init)` → `sequencer_address = 0xdeadbeef`.
6. The batcher executes all transactions with `sequencer_address = 0xdeadbeef`; any contract calling `get_sequencer_address` receives `0xdeadbeef`.
7. Sequencer fees are transferred to `0xdeadbeef`.
8. `PartialBlockHashComponents` encodes `sequencer = 0xdeadbeef`; the block hash is computed with this value.
9. The `ProposalFin` commitment check passes (both sides used `0xdeadbeef`).
10. Consensus commits the block with the corrupted sequencer address, wrong syscall results, and stolen fees.
