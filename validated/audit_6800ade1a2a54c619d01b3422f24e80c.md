### Title
Hardcoded `ADDRESS_REGISTRAR_ACCOUNT_ID` (`address-map.near`) in Wallet Contract Is Wrong on Testnet/Localnet, Breaking Relayer Accountability - (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/ADDRESS_REGISTRAR_ACCOUNT_ID`)

---

### Summary

The ETH wallet contract bakes `ADDRESS_REGISTRAR_ACCOUNT_ID = "address-map.near"` into every compiled WASM variant (mainnet, testnet, localnet) at build time. The build script correctly parameterises `CHAIN_ID` per network but never updates `ADDRESS_REGISTRAR_ACCOUNT_ID`. On testnet and localnet `address-map.near` does not exist (or is an unrelated contract), so every cross-contract call to the address registrar fails. This silently disables the relayer-accountability mechanism: a faulty relayer that routes a user's transaction to the wrong target cannot be detected or penalised, and its access key is never revoked.

---

### Finding Description

**Root cause – build script omits per-network `ADDRESS_REGISTRAR_ACCOUNT_ID`**

`runtime/near-wallet-contract/build.rs` builds three WASM artefacts. Before each build it writes the correct `CHAIN_ID` into the source file:

```rust
std::fs::write(&chain_id_path, chain_id.to_string().into_bytes())?;
``` [1](#0-0) 

It never touches `ADDRESS_REGISTRAR_ACCOUNT_ID`. That file is committed as:

```
address-map.near
``` [2](#0-1) 

The wallet contract reads it at compile time:

```rust
const ADDRESS_REGISTRAR_ACCOUNT_ID: &str = std::include_str!("ADDRESS_REGISTRAR_ACCOUNT_ID");
``` [3](#0-2) 

So the testnet WASM (chain ID 398) and localnet WASM (chain ID 399) both call `address-map.near` for every address-registrar lookup, even though that account does not exist on those networks.

**How the address check is used**

When a relayer submits an ETH-emulated base-token transfer whose `tx.to` is an ETH-implicit address, the wallet contract must verify that the relayer looked up whether that address is registered to a named account. It does so by calling `address-map.near::lookup`:

```rust
let address_registrar = {
    let account_id = ADDRESS_REGISTRAR_ACCOUNT_ID
        .trim()
        .parse()
        .unwrap_or_else(|_| env::panic_str("Invalid address registrar"));
    ext_registrar::ext(account_id).with_static_gas(REGISTRAR_LOOKUP_GAS)
};
address_registrar.lookup(address).then(ext.address_check_callback(...))
``` [4](#0-3) 

**Callback silently swallows the failure without revoking the relayer key**

When the call to `address-map.near` fails (account does not exist on testnet), `address_check_callback` returns early with an error response and does **not** revoke the relayer's access key:

```rust
PromiseResult::Failed => {
    return PromiseOrValue::Value(ExecuteResponse {
        success: false,
        success_value: None,
        error: Some("Call to Address Registrar contract failed".into()),
    });
}
``` [5](#0-4) 

Key revocation only happens in `rlp_execute` when `Error::Relayer` is returned synchronously; it never happens inside the callback. The nonce is also not incremented in this path, leaving the user's transaction permanently stuck.

**Contrast with the correct chain-ID handling**

The `CHAIN_ID` constant is correctly set per network (397 mainnet, 398 testnet, 399 localnet) and is enforced in `validate_tx_relayer_data`:

```rust
if tx.chain_id != Some(CHAIN_ID) {
    return Err(Error::Relayer(RelayerError::InvalidChainId));
}
``` [6](#0-5) 

`ADDRESS_REGISTRAR_ACCOUNT_ID` receives no equivalent per-network treatment.

---

### Impact Explanation

On testnet and localnet every call to `address-map.near` fails. Consequently:

1. **Relayer accountability is broken.** A malicious relayer that holds a `FunctionCall` access key on a user's ETH-implicit wallet account can route an ERC-20 or base-token transfer to the wrong target (the raw ETH-implicit address instead of the registered named account). The wallet contract attempts to detect this via the registrar lookup, but the lookup always fails, so the relayer's key is **never revoked**. The corrupted protocol value is the access key state of the wallet contract account: a key that should be deleted remains valid.

2. **User transactions are permanently blocked.** Any transaction that triggers the address-check path will always fail with "Call to Address Registrar contract failed." Because the nonce is not incremented on this failure path, the user cannot advance past the stuck nonce without the relayer cooperating, and the relayer has no incentive to do so.

---

### Likelihood Explanation

Every ETH-implicit account on testnet that interacts with a named account whose address is registered in the testnet address registrar is affected. The testnet wallet contract is actively deployed and used. The bug is deterministic: `address-map.near` does not exist on testnet, so the failure occurs on every invocation of the address-check path.

---

### Recommendation

Update the build script to write the correct per-network registrar account ID into `ADDRESS_REGISTRAR_ACCOUNT_ID` before each build, mirroring the existing `CHAIN_ID` handling:

```rust
// e.g. "address-map.near" for mainnet, "address-map.testnet" for testnet, etc.
std::fs::write(&registrar_id_path, registrar_account_id.as_bytes())?;
``` [7](#0-6) 

Alternatively, expose the registrar account ID as a runtime parameter rather than a compile-time constant so it can be updated without recompiling the WASM.

---

### Proof of Concept

1. On testnet, deploy a named account `alice.testnet` and register its address in the testnet address registrar (at whatever the correct testnet account is).
2. Create an ETH-implicit wallet account and fund it.
3. A malicious relayer registers an access key on the wallet account.
4. The user signs an Ethereum transaction whose `tx.to` is the ETH-implicit address corresponding to `alice.testnet`.
5. The malicious relayer submits `rlp_execute` with `target` set to the raw ETH-implicit address (instead of `alice.testnet`).
6. The wallet contract calls `address-map.near::lookup` — this call fails because `address-map.near` does not exist on testnet.
7. `address_check_callback` returns `{ success: false, error: "Call to Address Registrar contract failed" }` without revoking the relayer's key.
8. The relayer's access key remains valid; the user's nonce is unchanged; the transaction is permanently stuck.

fn build_contract(
    dir: &str,
    contract_name: &str,
    output: &str,
    chain_id: u64,
) -> anyhow::Result<()> {
    let wasm_target_path = format!("./res/{}.wasm", output);
    if Path::new(&wasm_target_path).exists() {
        // Skip building if an artifact is already present
        return Ok(());
    }

    let absolute_dir = Path::new(dir).canonicalize()?;

    let chain_id_path = absolute_dir.join("wallet-contract/src/CHAIN_ID");
    let chain_id_content = std::fs::read(&chain_id_path).context("Failed to read CHAIN_ID file")?;

    // Update the chain id before building
    std::fs::write(&chain_id_path, chain_id.to_string().into_bytes())?;
    docker_build(absolute_dir.to_str().expect("path should be valid UTF-8"))?;

    // Restore chain id file to original value after building
    std::fs::write(&chain_id_path, chain_id_content)?;

    let build_artifact_path =
        format!("target/wasm32-unknown-unknown/release/{}.wasm", contract_name);
    let src = absolute_dir.join(build_artifact_path);

    std::fs::copy(&src, &wasm_target_path)
        .with_context(|| format!("Failed to copy `{}` to `{}`", src.display(), wasm_target_path))?;
    Ok(())
}
