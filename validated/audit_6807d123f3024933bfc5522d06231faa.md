### Title
`respond_verify_foreign_tx` Accepts Arbitrary `payload_hash` Without Binding It to the Request, Enabling Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function verifies that the submitted signature is valid over the node-supplied `response.payload_hash`, but never checks that `payload_hash` is the correct hash for the given `request`. A single malicious attested participant that already holds a valid threshold signature from any prior `verify_foreign_transaction` signing round can replay that signature — with its associated hash — as the response to a *different* pending request. The contract accepts the substituted hash and delivers a forged attestation to the waiting caller.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following steps:

1. Retrieves the root public key for the domain.
2. Verifies `response.signature` over `response.payload_hash` using that root key.
3. If the signature is valid, resolves the pending yield keyed on `request` with the full `response` (including the attacker-chosen `payload_hash`). [1](#0-0) 

The `payload_hash` field is entirely attacker-controlled: the contract never derives or recomputes the expected hash from the `request` fields (chain, tx\_id, confirmations, extractors). This is structurally different from `respond`, where the expected public key is derived from `request.tweak`, cryptographically binding the signature to the specific request: [2](#0-1) 

Because the threshold signature produced for request B is the output of the cooperative MPC protocol and is available to every node that participated in that round, a single malicious participant can:

1. Participate honestly in the signing round for request B, obtaining the threshold signature `sig_B` over `payload_hash_B`.
2. Race to call `respond_verify_foreign_tx(request_A, { payload_hash: payload_hash_B, signature: sig_B })` for a *different* pending request A.
3. The contract checks only `verify_ecdsa_signature(sig_B, payload_hash_B, root_key)` — which passes — and resolves request A with the wrong hash. [3](#0-2) 

The analog to the LibSwap bug is exact: just as the Solidity `call` returned `true` for a non-existent contract without the caller checking whether the target was real, `respond_verify_foreign_tx` accepts any `(hash, signature)` pair that is internally consistent, without checking whether the hash is real for the submitted request.

---

### Impact Explanation

The caller of `verify_foreign_transaction` receives `VerifyForeignTransactionResponse { payload_hash, signature }` and uses it as a cryptographic attestation that the MPC network verified a specific foreign-chain event. If `payload_hash` has been silently substituted, the caller holds a valid MPC signature over the *wrong* transaction. Any bridge or settlement layer that trusts this attestation without independently recomputing the hash from the original request data will authorize execution for a transaction that was never verified — enabling forged foreign-chain verification and potentially invalid bridge execution or double-spend conditions. This matches the **High** allowed impact: *forged foreign-chain verification that causes invalid bridge execution*.

---

### Likelihood Explanation

- Only a **single** malicious attested participant is required — well below the signing threshold.
- The attacker needs no special privilege beyond being an active participant: it participates in the honest signing round for request B (obtaining `sig_B`), then races to submit that signature for request A.
- Multiple `verify_foreign_transaction` requests are routinely pending simultaneously, so the precondition (two concurrent pending requests) is a normal production state, not an edge case.
- The race window is the interval between the threshold signature being produced and the honest coordinator submitting the response — a window that exists in every signing round.

---

### Recommendation

The contract must independently derive the expected `payload_hash` from the `request` fields and assert equality with `response.payload_hash` before accepting the response. If the full hash computation is too expensive on-chain, a commitment to the expected hash should be stored alongside the pending yield at request-submission time (analogous to how `request.tweak` binds `respond` to its request), so that `respond_verify_foreign_tx` can compare against it.

---

### Proof of Concept

1. Alice calls `verify_foreign_transaction` for Bitcoin `tx_id = [1u8; 32]` → **request A** enters `pending_verify_foreign_tx_requests`.
2. Bob calls `verify_foreign_transaction` for Bitcoin `tx_id = [2u8; 32]` → **request B** enters `pending_verify_foreign_tx_requests`.
3. MPC nodes cooperate to produce `payload_hash_B` and threshold signature `sig_B` for request B.
4. Malicious node M (one of the t participants in step 3) now holds `(payload_hash_B, sig_B)`.
5. M calls `respond_verify_foreign_tx(request_A, { payload_hash: payload_hash_B, signature: sig_B })`.
6. Contract executes: `verify_ecdsa_signature(sig_B, payload_hash_B, root_key)` → **passes**.
7. `resolve_yields_for(&mut pending_verify_foreign_tx_requests, &request_A, serialize(response))` resolves Alice's yield with the forged response.
8. Alice's caller receives `{ payload_hash: payload_hash_B, signature: sig_B }` — a valid MPC attestation for Bob's transaction, not hers.
9. Any downstream bridge contract that checks only signature validity (not hash-to-request binding) authorizes execution based on the forged attestation. [4](#0-3)

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
