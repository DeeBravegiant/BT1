### Title
Harvester Can Unconditionally Divert Farmer Block Rewards via Unenforced `farmer_reward_address_override` in `RespondSignatures` — (File: `chia/farmer/farmer_api.py`)

### Summary

The CHIP-22 fee convention allows third-party harvesters to redirect a fraction of farmer block rewards as a fee by setting `farmer_reward_address_override` in protocol messages. However, the farmer's `_process_respond_signatures` function in `chia/farmer/farmer_api.py` unconditionally accepts any `farmer_reward_address_override` value from a harvester's `RespondSignatures` message without enforcing the fee quality threshold or verifying the override was present in the original `NewProofOfSpace`. The only check that exists (`notify_farmer_reward_taken_by_harvester_as_fee`) is a log-only warning with no enforcement, and it is not even called from the `respond_signatures` code path. A malicious connected harvester can redirect 100% of the farmer's XCH block rewards to an arbitrary address.

### Finding Description

**Root cause — `_process_respond_signatures` in `chia/farmer/farmer_api.py`:**

When the farmer receives a `RespondSignatures` message from a harvester, it calls `_process_respond_signatures`. At lines 916–919, the farmer unconditionally replaces its configured `farmer_target` reward address with whatever `farmer_reward_address_override` the harvester provides:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [1](#0-0) 

This `farmer_reward_address` is then placed directly into the `DeclareProofOfSpace` message sent to the full node, which determines where the block reward is paid: [2](#0-1) 

**The only "check" is a log warning, not enforcement:**

The function `notify_farmer_reward_taken_by_harvester_as_fee` in `chia/farmer/farmer.py` computes a fee quality value and compares it to the harvester-supplied threshold. When the threshold is invalid, it emits a `log.warning` — but **does not block the override from being applied**: [3](#0-2) 

**The fee check is not called from the `respond_signatures` path at all:**

`notify_farmer_reward_taken_by_harvester_as_fee` is only called from `new_proof_of_space` when `new_proof_of_space.farmer_reward_address_override is not None`: [4](#0-3) 

The `respond_signatures` handler calls only `_process_respond_signatures`, which has no fee quality check: [5](#0-4) 

**Two exploit paths exist:**

1. **Overt path**: Harvester sends `NewProofOfSpace` with `farmer_reward_address_override` set to attacker address. The log warning fires but does not block. The override propagates through `RequestSignatures` → `RespondSignatures` → `DeclareProofOfSpace`.

2. **Stealthy path**: Harvester sends `NewProofOfSpace` with `farmer_reward_address_override=None` (no log warning fires, no fee check). When the farmer sends `RequestSignatures`, the harvester responds with `RespondSignatures` containing `farmer_reward_address_override=<attacker_address>`. `_process_respond_signatures` has no record of what the original `NewProofOfSpace` contained, so it accepts the injected override unconditionally. No warning is logged.

The `RespondSignatures` message type explicitly carries `farmer_reward_address_override: bytes32 | None`: [6](#0-5) 

### Impact Explanation

**Impact: High — Unauthorized farmer reward diversion.**

A malicious third-party harvester (a legitimate peer role explicitly supported by CHIP-22 and the Chia ecosystem) can redirect 100% of the farmer's XCH block rewards to an arbitrary puzzle hash of the harvester's choosing. The farmer's configured `farmer_target` address is silently replaced in every winning block. The farmer loses all block rewards for every block won using plots managed by the malicious harvester. This is a direct, unauthorized movement of XCH from the farmer to the attacker.

### Likelihood Explanation

Third-party harvesters are a supported and common deployment pattern in Chia (DrPlotter, etc.). CHIP-22 was specifically designed to enable this. Any operator of a third-party harvester service who connects to a victim farmer can exploit this. The harvester connects using the private SSL CA, which is a prerequisite for the harvester role — not an additional privilege. No leaked keys, admin access, or cryptographic breaks are required beyond being a connected harvester peer.

### Recommendation

- **Short term**: In `_process_respond_signatures`, enforce the fee quality threshold before accepting `farmer_reward_address_override`. If the threshold is not met (or `fee_info` is absent), reject the override and use `self.farmer.farmer_target`. Additionally, track whether the original `NewProofOfSpace` for the given `sp_hash`/`plot_identifier` had an override set, and reject any override in `RespondSignatures` that was not present in the corresponding `NewProofOfSpace`.
- **Long term**: Treat `farmer_reward_address_override` as a security-sensitive field. Add integration tests that verify a harvester cannot divert rewards beyond the agreed fee threshold, and that injecting an override only in `RespondSignatures` (not in `NewProofOfSpace`) is rejected.

### Proof of Concept

1. Operate a harvester connected to a victim farmer (standard third-party harvester setup).
2. When the farmer sends `RequestSignatures` for a winning proof, intercept the response.
3. Return a `RespondSignatures` message with `farmer_reward_address_override` set to the attacker's puzzle hash (and `farmer_reward_address_override=None` in the preceding `NewProofOfSpace` to avoid logging).
4. The farmer's `_process_respond_signatures` accepts the override at lines 917–918 without any validation.
5. The farmer sends `DeclareProofOfSpace` to the full node with `farmer_reward_address = attacker_puzzle_hash`.
6. The full node includes the block with the attacker's address as the farmer reward recipient.
7. The 0.25 XCH farmer reward is paid to the attacker's address instead of the farmer's configured address.

This is confirmed by the test in `chia/_tests/farmer_harvester/test_third_party_harvesters.py` which demonstrates that `farmer_reward_address_override` injected in `RespondSignatures` is respected by the farmer without any threshold enforcement: [7](#0-6)
