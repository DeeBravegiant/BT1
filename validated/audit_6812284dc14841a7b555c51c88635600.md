Audit Report

## Title
Missing L2 Sequencer Uptime Check Enables Stale Price Exploitation During Arbitrum Outages - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls Chainlink's `latestRoundData()` with no check for whether the Arbitrum L2 sequencer is active, no staleness guard on `updatedAt`, and no `price > 0` validation. The protocol is explicitly deployed on Arbitrum (confirmed by `RSETHPool.sol` NatSpec). During a sequencer outage, Chainlink feeds freeze at the last reported price, allowing an attacker to deposit assets at a stale favorable rate and mint more rsETH than the real value of their deposit warrants, diluting existing rsETH holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the asset/ETH exchange rate and returns it directly:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

Three deficiencies are simultaneously present:
1. **No L2 sequencer uptime check** — no call to a Chainlink sequencer uptime feed.
2. **No staleness check** — `updatedAt` is silently discarded.
3. **No price validity check** — `price` is cast directly to `uint256` without verifying `price > 0`.

The full call chain from a public entry point:

```
LRTDepositPool.depositAsset() / depositETH()
  → _beforeDeposit()
    → getRsETHAmountToMint()
      → LRTOracle.getAssetPrice(asset)
        → IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)
          → ChainlinkPriceOracle.getAssetPrice()  ← stale price returned here
```

`getRsETHAmountToMint()` computes the mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

`lrtOracle.rsETHPrice()` is a stored value updated by a separate keeper call (`updateRSETHPrice`), so it does not self-correct during a sequencer outage. The numerator (asset price from Chainlink) is fetched live and will be stale.

Arbitrum deployment is explicitly confirmed in `RSETHPool.sol`:

```solidity
/// @notice This contract is the pool contract for the rsETH pool on *Arbitrum*
``` [3](#0-2) 

A secondary instance exists in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which performs `answeredInRound < roundID` and `timestamp == 0` checks but no sequencer uptime check — these checks do not protect against sequencer downtime since Chainlink continues to report the last round as valid while the sequencer is offline. [4](#0-3) 

No sequencer uptime feed reference exists anywhere in the production contracts.

## Impact Explanation

**High — Theft of unclaimed yield.**

When the Arbitrum sequencer goes down and the true market price of a supported LST moves (e.g., depeg or appreciation), an attacker can deposit at the stale favorable price and receive more rsETH than the real value of their deposit warrants. When the sequencer comes back online and `updateRSETHPrice()` is called, the rsETH price adjusts downward, diluting all existing rsETH holders. This constitutes direct theft of yield/value from existing holders, as their proportional claim on the protocol's ETH is reduced by the inflated rsETH supply.

## Likelihood Explanation

Arbitrum is the confirmed deployment target. Arbitrum sequencer outages have occurred historically (December 2021, June 2023). The attack requires only that the sequencer be down and that the market price has moved during the outage — both are realistic, independently occurring conditions. No privileged access is required; any unprivileged depositor can call `depositAsset()` or `depositETH()` directly.

## Recommendation

Follow the [Chainlink L2 sequencer uptime feed documentation](https://docs.chain.link/data-feeds/l2-sequencer-feeds#example-code). Add a sequencer uptime check, a staleness guard, and a `price > 0` check in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
AggregatorV3Interface internal sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600;
uint256 public constant MAX_STALENESS = 3600;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    (, int256 sequencerAnswer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (sequencerAnswer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();

    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
    require(price > 0, "Invalid price");
    require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Apply the same fix to `ChainlinkOracleForRSETHPoolCollateral.getRate()`.

## Proof of Concept

1. Arbitrum sequencer goes offline. Chainlink oracle for stETH/ETH freezes at price `P_stale`.
2. True market price of stETH drops to `P_real < P_stale` (e.g., due to a depeg event).
3. Attacker calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")`.
4. `getRsETHAmountToMint()` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `P_stale` (stale, inflated).
5. Attacker receives `rsethAmountToMint = amount * P_stale / rsETHPrice` — more rsETH than the real value of their deposit warrants.
6. When sequencer comes back online and `updateRSETHPrice()` is called, the rsETH price drops, diluting all existing holders.

**Foundry fork test plan:**
- Fork Arbitrum mainnet at a block just before a known sequencer outage.
- Mock the Chainlink feed to return a stale price (simulating frozen feed).
- Call `depositAsset()` as an unprivileged attacker with a stale-favorable price.
- Assert that `rsethAmountToMint` exceeds the fair-value amount computed using the true market price.
- Call `updateRSETHPrice()` after restoring the correct price and assert that existing holders' proportional ETH claim has decreased.
