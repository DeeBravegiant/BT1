Audit Report

## Title
ETH Deposit DoS via Unsolicited `receive()` Balance Inflation in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

## Summary
`getETHDistributionData()` computes total protocol ETH using raw `address(this).balance`, `nodeDelegatorQueue[i].balance`, and `lrtUnstakingVault.balance`. All three contracts expose an open `receive()` function that any external caller can use to inflate these balances. Because `_checkIfDepositAmountExceedesCurrentLimit` for ETH uses a plain `totalAssetDeposits > limit` comparison (with no separate addition of `msg.value`), pushing the raw balance sum above the configured limit causes every subsequent `depositETH` call to revert with `MaximumDepositLimitReached`, freezing ETH deposits for all users until an admin raises the limit.

## Finding Description

`getETHDistributionData()` reads raw native-ETH balances directly:

```solidity
// LRTDepositPool.sol L480, L485, L496
ethLyingInDepositPool = address(this).balance;
ethLyingInNDCs += nodeDelegatorQueue[i].balance;
ethLyingInUnstakingVault = lrtUnstakingVault.balance;
```

These feed into `getTotalAssetDeposits(ETH)` via `getAssetDistributionData` → `getETHDistributionData`, which is then consumed by `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
// LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch uses `totalAssetDeposits > limit` (no `+ amount`) because `msg.value` is already reflected in `address(this).balance` at call time. This means if the raw balance sum already exceeds the limit before any new deposit arrives, the check returns `true` for every caller regardless of their deposit size, including zero-value calls.

All three balance sources accept arbitrary ETH from any sender:
- `LRTDepositPool.sol` L58: `receive() external payable { }`
- `LRTUnstakingVault.sol` L81–83: `receive() external payable { emit EthReceived(...); }`
- `NodeDelegator.sol` L81–83: `receive() external payable { emit ETHReceived(...); }`

An attacker sends `gap + 1 wei` (where `gap = depositLimitByAsset(ETH) - getTotalAssetDeposits(ETH)`) to any of these addresses. The inflated balance is immediately reflected in `getETHDistributionData()`, causing `_beforeDeposit` to revert at L661–663 for every subsequent `depositETH` call.

## Impact Explanation

**Medium — Temporary freezing of funds.** All ETH deposits via `depositETH` are blocked. Users cannot mint rsETH with ETH. The freeze persists until an admin raises `depositLimitByAsset(ETH)`. Because the attacker's ETH remains in the protocol and is counted in the next `getTotalAssetDeposits` reading, the admin must raise the limit by at least the inflated amount, or the DoS is immediately re-triggered at negligible marginal cost. This is a concrete, user-facing disruption of a core protocol function, not merely a theoretical or griefing-only concern.

## Likelihood Explanation

**Medium.** The attack requires no special role or permission — only the ability to call a public `receive()` function with ETH. The cost equals the gap between the current total deposits and the configured limit. Protocols routinely operate near their deposit caps during high-demand periods, making the gap small. The attacker's ETH is not recoverable, but the attack can be repeated each time the admin raises the limit, making sustained griefing cheap relative to the disruption caused.

## Recommendation

Replace raw `address.balance` reads in `getETHDistributionData()` with an internal accounting variable (e.g., `uint256 private _ethAccounted`) that is incremented only through controlled entry points (`receiveFromNodeDelegator`, `receiveFromRewardReceiver`, `receiveFromLRTConverter`, and `depositETH`). Unsolicited ETH arriving via the bare `receive()` function should either be rejected (revert) or tracked in a separate variable excluded from the deposit-limit calculation. This ensures that only ETH entering through protocol-sanctioned paths is counted against the deposit cap.

## Proof of Concept

1. Read `D = getTotalAssetDeposits(LRTConstants.ETH_TOKEN)` and `L = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN)` where `D < L`.
2. Attacker executes: `(bool ok,) = address(lrtDepositPool).call{value: L - D + 1}("");`
3. `getETHDistributionData()` now returns `ethLyingInDepositPool = D_old + (L - D + 1)`, so `getTotalAssetDeposits(ETH) = L + 1`.
4. Any user calling `depositETH(minRSETH, referralId)` triggers `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit(ETH, msg.value)` → `(L + 1 > L)` → `true` → `revert MaximumDepositLimitReached()`.
5. ETH deposits are frozen for all users.

**Foundry test sketch:**
```solidity
function testDepositDoS() public {
    uint256 limit = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN);
    uint256 current = lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN);
    uint256 gap = limit - current;

    // Attacker inflates balance
    vm.deal(attacker, gap + 1);
    vm.prank(attacker);
    (bool ok,) = address(lrtDepositPool).call{value: gap + 1}("");
    assertTrue(ok);

    // Legitimate user deposit now reverts
    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
}
```
