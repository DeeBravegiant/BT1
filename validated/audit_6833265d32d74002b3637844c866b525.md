Audit Report

## Title
ETH Deposit Limit Bypass via Missing Amount in Boundary Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` contains an asymmetric ETH branch that omits the incoming `amount` from the limit comparison and uses strict `>` instead of `>=`. As a result, the ETH deposit cap is entirely unenforced: any depositor can push the protocol's ETH holdings arbitrarily beyond the configured limit in a single transaction, as long as the current total has not yet crossed the cap.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the function receives `amount` as a parameter but the ETH branch never uses it:

```solidity
// line 678-679 — ETH branch: amount is ignored
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
// line 681 — LST branch: amount is correctly included
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

Two compounding defects:
1. **Missing `amount`**: The check only asks "has the limit already been exceeded?" not "would this deposit exceed the limit?"
2. **Strict `>` instead of `>=`**: Even when `totalAssetDeposits == depositLimitByAsset`, the condition is `false` and the deposit proceeds.

The call chain is: `depositETH` (line 87) → `_beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, ...)` (line 87) → `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` (line 661), which reverts only when the function returns `true`. Because the ETH branch never incorporates `amount`, it never returns `true` for a deposit that would exceed the cap — only for one that already has.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but doesn't lose value.**

The ETH deposit cap is a protocol-level risk parameter set by the admin to bound exposure. With the ETH branch broken, any depositor can mint rsETH against an arbitrarily large ETH deposit in a single call, as long as `totalAssetDeposits ≤ depositLimitByAsset` at the moment of the call. The protocol accumulates ETH far beyond the intended ceiling, undermining the risk management guarantee the limit is meant to provide. No direct theft occurs, but the contract fails to enforce its own stated constraint.

## Likelihood Explanation
High. The entry point is the public, permissionless `depositETH` function. No special role, flash loan, or oracle manipulation is required. Any depositor who observes that the current ETH total has not yet crossed the limit can immediately exploit this by sending a single large ETH deposit.

## Recommendation
Add `amount` to the ETH branch and use `>=` to match the semantics of the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount >= lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH] = 100 ether`.
2. Current `getTotalAssetDeposits(ETH) = 50 ether`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit(ETH, 10_000 ether)` (line 676):
   - `totalAssetDeposits = 50 ether`
   - ETH branch (line 679): `return (50 ether > 100 ether)` → `false`
   - `amount = 10_000 ether` is never evaluated.
5. `_beforeDeposit` (line 661) does not revert; rsETH is minted for 10,000 ETH.
6. After the call, `getTotalAssetDeposits(ETH) = 10,050 ether` — 100× the intended cap.
