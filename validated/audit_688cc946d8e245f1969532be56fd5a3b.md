Audit Report

## Title
Rewards Permanently Stuck When `totalKernelStaked` Drops to Zero Mid-Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` checks that `totalKernelStaked > 0` only at the moment `notifyRewardAmount` is called, but provides no on-chain enforcement that stake remains non-zero for the entire reward period. When all stakers exit mid-period, `rewardPerToken()` freezes while `updatedAt` is advanced by the `updateReward` modifier, permanently losing all rewards emitted during the zero-staked window with no recovery mechanism.

## Finding Description
Three code paths interact to produce the loss:

**1. `rewardPerToken()` short-circuits when `totalKernelStaked == 0`:** [1](#0-0) 
When no tokens are staked, `rewardPerTokenStored` is returned unchanged — no new rewards are accrued.

**2. `updateReward` always advances `updatedAt` to `lastTimeRewardApplicable()`:** [2](#0-1) 
This happens unconditionally, regardless of whether `totalKernelStaked` is zero. The timestamp advances even when no rewards are being distributed.

**3. `initiateWithdrawal` decrements `totalKernelStaked` immediately:** [3](#0-2) 
The `withdrawalDelay` only delays the token transfer; the accounting reduction is instant.

**Exploit flow:**
- At time T, Alice is the sole staker. Admin calls `notifyRewardAmount` — passes the `totalKernelStaked > 0` guard. [4](#0-3) 
- At T+3600, Alice calls `initiateWithdrawal`. The `updateReward` modifier checkpoints her rewards and sets `updatedAt = T+3600`. Then `totalKernelStaked = 0`.
- From T+3600 to T+7200, no one is staked.
- At T+7200, Bob stakes. `updateReward(Bob)` fires: `rewardPerToken()` sees `totalKernelStaked == 0` → returns `rewardPerTokenStored` unchanged, but `updatedAt` is set to `T+7200`. Bob's `userRewardPerTokenPaid` is set to the unchanged `rewardPerTokenStored`.
- The 3600-second gap is consumed. `rewardRate × 3600` reward tokens are permanently locked in the contract — Bob earns only from T+7200 onward, and Alice already claimed up to T+3600.

The contract's own NatSpec explicitly acknowledges this behavior but relies on an off-chain operational assumption ("ensuring there are always some tokens staked... for the entire duration of the reward period") that is not enforced by any on-chain guard: [5](#0-4) 

There is no admin sweep, rescue function, or redistribution mechanism anywhere in the contract.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens corresponding to any zero-staked window are irrecoverable. The loss equals `rewardRate × (duration of zero-staked window)`. This matches the allowed impact scope exactly.

## Likelihood Explanation
Any staker can call `initiateWithdrawal` at any time without restriction. A single large staker holding the majority of stake can unilaterally drive `totalKernelStaked` to zero. This can occur naturally (all stakers exit) or be triggered deliberately. No special privileges, coordination, or external conditions are required beyond being a normal depositor.

## Recommendation
Enforce the invariant on-chain rather than relying on operational assumptions. Options include:
- Add a check in `initiateWithdrawal` that reverts if `totalKernelStaked - _amount == 0` while `block.timestamp < finishAt`.
- Implement an admin recovery function callable after `finishAt` to reclaim rewards that were never distributed (calculable as `rewardRate × zero-staked duration`).
- Track the zero-staked duration and roll unallocated rewards into the next period via `notifyRewardAmount`.

## Proof of Concept
Foundry test sequence:
1. Deploy contract, set `duration = 86400`.
2. Alice stakes 1000 KERNEL. `totalKernelStaked = 1000`.
3. Admin calls `notifyRewardAmount(86400e18)`. `rewardRate = 1e18/s`, `finishAt = T+86400`, `updatedAt = T`.
4. Warp to `T+3600`. Alice calls `initiateWithdrawal(1000)`. Assert `totalKernelStaked == 0`, `updatedAt == T+3600`, Alice's `rewards[Alice] == 3600e18`.
5. Warp to `T+7200`. Bob calls `stake(1000)`. Assert `rewardPerTokenStored` is unchanged from step 4, `updatedAt == T+7200`.
6. Warp to `T+86400`. Bob calls `getReward()`. Assert Bob received `(86400 - 7200) × 1e18 = 79200e18` tokens, not `(86400 - 3600) × 1e18 = 82800e18`.
7. Assert `rewardsToken.balanceOf(address(pool)) >= 3600e18` (stuck rewards), with no function available to recover them.
