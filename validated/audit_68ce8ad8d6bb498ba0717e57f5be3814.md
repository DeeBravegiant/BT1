### Summary

The gateway's stateful validator increments the latest committed block number by one (`block_info.block_number.unchecked_next()`) when building the `BlockContext` for proof-facts validation, but it reads the block-hash contract from the **current committed state** (block N), which does not yet contain the hash written by `pre_process_block` for block N+1. This creates a one-block window where `validate_proof_block_number` permits `proof_block_number = N âˆ’ STORED_BLOCK_HASH_BUFFER + 1` (= Nâˆ’9 when `STORED_BLOCK_HASH_BUFFER = 10`), while `validate_proof_block_hash` reads `Felt::ZERO` for that slot and rejects the transaction. The batcher, which calls `pre_process_block` before executing transactions, would accept the same transaction. The result is a systematic false rejection of valid proof-carrying Invoke V3 transactions at the gateway admission boundary.

---

### Finding Description

**`STORED_BLOCK_HASH_BUFFER = 10`** is the Starknet analog of Ethereum's 256-block `blockhash()` window. [1](#0-0) 

`pre_process_block` writes the hash of block `next_block_number âˆ’ STORED_BLOCK_HASH_BUFFER` into the block-hash contract each time a new block is processed. At committed state N, the contract contains hashes for blocks 0 â€¦ Nâˆ’10. [2](#0-1) 

The gateway's `run_validate_entry_point` builds a `BlockContext` with `block_number = N+1`: [3](#0-2) 

`validate_proof_block_number` uses that `N+1` as `current_block_number`, so it permits `proof_block_number â‰¤ N+1âˆ’10 = Nâˆ’9`: [4](#0-3) 

`validate_proof_block_hash` then reads the block-hash contract from the **state at block N**. Block Nâˆ’9's hash is not yet written (it is written during block N+1's `pre_process_block`), so `get_storage_at` returns `Felt::ZERO`. Because the user's `proof_block_hash` is the correct non-zero hash, the check `stored_block_hash != proof_block_hash` (0 â‰  correct) fires and the transaction is rejected: [5](#0-4) 

The batcher, by contrast, calls `pre_process_block` before executing transactions in block N+1, so its state already contains the hash of Nâˆ’9 and the same transaction passes.

### Impact Explanation

Any unprivileged user who submits an Invoke V3 transaction carrying SNOS proof facts with `proof_block_number = latest_committed_block âˆ’ STORED_BLOCK_HASH_BUFFER + 1` (i.e., exactly at the boundary the block-number check allows) will have their transaction rejected by the gateway with `InvalidProofFacts("Block hash mismatch â€¦")`. The transaction is valid by protocol rules and would be accepted by the batcher. This is a **false rejection at gateway admission** â€” matching the "High" impact category: *Mempool/gateway/RPC admission rejects valid transactions before sequencing*.

---

### Likelihood Explanation

The window is narrow (exactly one block depth) but deterministic and reproducible. Any client that generates a proof anchored to the most-recent allowed block and submits it before the next block is committed will hit this rejection every time. The condition is unprivileged and requires no special access.

---

### Recommendation

In `run_validate_entry_point`, use the committed block number (N) rather than `N+1` as `current_block_number` when constructing the `BlockContext` for proof-facts validation, **or** apply the stricter bound `max_allowed = current_block_number âˆ’ STORED_BLOCK_HASH_BUFFER âˆ’ 1` so that only block hashes already present in the committed state are accepted. Either change makes the gateway's admission decision consistent with the state it actually reads.

---

### Proof of Concept

1. Let the latest committed block be N = 100. `STORED_BLOCK_HASH_BUFFER = 10`.
2. Construct an Invoke V3 transaction with `proof_block_number = 91` (= 100 âˆ’ 10 + 1) and `proof_block_hash = actual_hash_of_block_91`.
3. Submit to the gateway.
4. Gateway builds `BlockContext` with `block_number = 101`.
5. `validate_proof_block_number`: `max_allowed = 101 âˆ’ 10 = 91`; `91 â‰¤ 91` â†’ **passes**.
6. `validate_proof_block_hash`: reads `block_hash_contract[91]` from state at block 100 â†’ returns `Felt::ZERO` (hash of block 91 is written during block 101 processing, not yet committed).
7. `Felt::ZERO â‰  actual_hash_of_block_91` â†’ **rejected** with `InvalidProofFacts("Block hash mismatch for block 91 â€¦")`.
8. Wait for block 101 to be committed; resubmit the identical transaction â†’ **accepted** (hash of block 91 is now in state).

The batcher building block 101 would have accepted the transaction in step 3 because `pre_process_block` writes `block_hash_contract[91]` before executing transactions.

        current_block_number: BlockNumber,

    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }
