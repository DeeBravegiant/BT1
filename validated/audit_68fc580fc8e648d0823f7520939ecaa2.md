### Title
Missing Transaction Hash Validation in P2P Sync Allows RPC `get_transaction_by_hash` to Serve Wrong Transaction Body — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

The P2P sync client accepts `FullTransaction` objects from untrusted peers and stores the peer-supplied `transaction_hash` directly without verifying it matches `transaction.calculate_transaction_hash(chain_id)`. The RPC handler `get_transaction_by_hash` then serves the stored transaction body paired with the queried hash, with no re-computation or consistency check. A malicious peer can inject a `FullTransaction` where `transaction_hash` is a valid hash from the block's `transaction_commitment` but `transaction` is a different (wrong) body, causing the RPC to authoritatively return the wrong transaction body for that hash.

---

### Finding Description

**Step 1 — Missing validation in P2P sync ingestion**

In `parse_data_for_block`, each `FullTransaction` received from a peer is destructured and its fields are pushed directly into `block_body` without any check that `transaction.calculate_transaction_hash(chain_id) == transaction_hash`: [1](#0-0) 

The comment at line 88 is an explicit acknowledgment that this validation is absent:
```
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
```

**Step 2 — Storage stores hash and body independently, no cross-check**

`write_transactions` in `body/mod.rs` stores the transaction body in a file and the hash in the `transaction_hash_to_idx` table and `TransactionMetadata`, but never verifies consistency between them: [2](#0-1) 

**Step 3 — RPC serves the stored body paired with the queried hash, no re-computation**

`get_transaction_by_hash` looks up the `transaction_index` by the requested hash, fetches the transaction body at that index, and returns them together — it never recomputes the hash from the body to verify they match: [3](#0-2) 

The returned `TransactionWithHash` pairs the caller-supplied `transaction_hash` with whatever body was stored at that index, with no integrity check.

---

### Impact Explanation

A malicious P2P peer sends a `FullTransaction` where:
- `transaction_hash = H` is a legitimate hash that satisfies the block's `transaction_commitment` (so the block header passes any commitment check)
- `transaction = T_wrong` is a different transaction body whose actual hash is not `H`

The node stores `H → index` and `index → T_wrong`. Any caller of `starknet_getTransactionByHash(H)` receives `{ transaction: T_wrong, transaction_hash: H }` — an authoritative-looking response pairing the correct hash with the wrong transaction body. This corrupts the RPC view used by wallets, explorers, and downstream production node code.

This falls under: **High — RPC returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

The P2P sync path is reachable by any node that connects as a peer. The missing validation is explicitly documented as a TODO. No existing guard prevents a peer from sending a `FullTransaction` with a mismatched body. The block's `transaction_commitment` commits to transaction hashes, not to transaction bodies, so a peer can supply valid hashes (satisfying the commitment) paired with arbitrary wrong bodies.

---

### Recommendation

In `parse_data_for_block` (or immediately before calling `append_body`), recompute the transaction hash from the received transaction body and reject the peer if it does not match the supplied `transaction_hash`:

```rust
let computed_hash = transaction.calculate_transaction_hash(&chain_id)
    .map_err(|_| ParseDataError::BadPeer(BadPeerError::InvalidTransactionHash { ... }))?;
if computed_hash != transaction_hash {
    return Err(ParseDataError::BadPeer(BadPeerError::InvalidTransactionHash { ... }));
}
block_body.transaction_hashes.push(transaction_hash);
```

This resolves the explicit TODO at [4](#0-3)  and closes the gap between the stored hash and the stored body before it reaches the RPC layer.

---

### Proof of Concept

1. Construct a valid `BlockBody` for block N where `transaction_hashes[0] = H_real` (a hash that satisfies the block's `transaction_commitment`) but `transactions[0] = T_wrong` (a different transaction whose `calculate_transaction_hash()` returns `H_other ≠ H_real`).
2. Call `storage_writer.begin_rw_txn()?.append_body(BlockNumber(N), block_body)?.commit()?` — this succeeds with no error.
3. Call `get_transaction_by_hash(H_real)` via the RPC.
4. Assert the response contains `T_wrong` as the transaction body — demonstrating the RPC returns the wrong body for the given hash without detecting the mismatch.

This is exactly the pattern already present in the existing RPC test at [5](#0-4) , which stores a block with an arbitrary `transaction_hash` (line 2287: `block.body.transaction_hashes[0] = tx_hash!(random::<u64>())`) and asserts the RPC returns that body — confirming the storage and RPC layers accept and serve mismatched hash/body pairs without any complaint.
