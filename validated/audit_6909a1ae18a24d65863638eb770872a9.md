### Title
Stale post-resharing votes in `CodeHashesVotes` allow sub-threshold TEE image hash whitelisting when `clean_tee_status` detached promise fails — (`crates/contract/src/tee/proposal.rs`, `crates/contract/src/lib.rs`)

### Summary
When a resharing completes, `vote_reshared` spawns `clean_tee_status` as a **detached** promise to purge stale votes from removed participants. Because the promise is detached, its failure is silent and does not revert the resharing. If it fails, stale `CodeHashesVotes` entries from removed participants persist. `vote_code_hash` then counts those stale entries alongside new votes without any participant-set filter, allowing the governance threshold to be satisfied with fewer than the required number of current participants.

### Finding Description
`vote_reshared` in `lib.rs` (line 1186) spawns `clean_tee_status` with `.detach()`. A detached NEAR promise failure is silent — the parent transaction succeeds regardless. If the gas budget (`self.config.clean_tee_status_tera_gas`) is exhausted or the call panics, `tee_state.votes.proposal_by_account` retains entries keyed by `AuthenticatedParticipantId` values belonging to removed participants.

`vote_code_hash` (line 1418) calls `self.tee_state.vote(code_hash, &participant)`, which delegates to `CodeHashesVotes::vote`. That function inserts the new vote and then calls `count_votes`, which iterates **all** values in `proposal_by_account` matching the hash — including stale entries from removed participants. The returned count is compared directly against `self.threshold()?.value()` (line 1425). If stale votes + new votes ≥ threshold, `whitelist_tee_proposal` is triggered.

The developer comment on the analogous `remove_non_participant_update_votes` promise (line 1176) explicitly acknowledges this failure mode and notes that `vote_update` uses runtime filtering as a fallback. No such fallback exists in `vote_code_hash`.

### Impact Explanation
A malicious TEE image hash is added to `allowed_docker_image_hashes`. Nodes running that image can then pass `submit_participant_info` attestation verification, be accepted as attested participants, and — if subsequently voted in via `vote_new_parameters` — participate in threshold signing. This is an attestation authorization bypass enabling non-genuine nodes to enter the signing participant set.

### Likelihood Explanation
Requires: (a) removed participants having voted for the target hash before resharing (achievable by colluding ex-participants, who are below the current threshold), and (b) `clean_tee_status` failing. The failure condition is realistic: the gas budget is a configurable integer (`clean_tee_status_tera_gas`), and NEAR detached promises fail silently on out-of-gas. The codebase's own unit test (`test_clean_non_participant_votes_removes_stale_votes`) demonstrates the stale-vote counting behavior when cleanup is skipped.

### Recommendation
Mirror the pattern used for `vote_update`: add a participant-set filter inside `vote_code_hash` itself so correctness does not depend on the cleanup promise succeeding. Specifically, before calling `count_votes`, filter `proposal_by_account` to only entries whose `AuthenticatedParticipantId` is in the current participant set. Alternatively, make `clean_tee_status` a chained (non-detached) promise so its failure reverts the resharing transition, or add a retry/idempotent re-call path.

### Proof of Concept
```
Setup: N=5 participants, threshold T=3.
Step 1: P1 and P2 (who will be removed) call vote_code_hash(MALICIOUS_HASH).
        → proposal_by_account = {P1→MALICIOUS, P2→MALICIOUS}, count=2 < 3.
Step 2: vote_new_parameters removes P1,P2. New set {P3,P4,P5}, threshold T'=2.
Step 3: Resharing completes via vote_reshared. clean_tee_status detached promise fails.
        → proposal_by_account still = {P1→MALICIOUS, P2→MALICIOUS} (stale).
Step 4: P3 (current participant) calls vote_code_hash(MALICIOUS_HASH).
        → proposal_by_account = {P1→MALICIOUS, P2→MALICIOUS, P3→MALICIOUS}.
        → count_votes returns 3 >= T'=2.
        → whitelist_tee_proposal(MALICIOUS_HASH) is called.
Result: MALICIOUS_HASH is whitelisted with only 1 current-participant vote.
```

    pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
        log!(
            "vote_code_hash: signer={}, code_hash={:?}",
            env::signer_account_id(),
            code_hash,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let votes = self.tee_state.vote(code_hash, &participant);

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        // If the vote threshold is met and the new Docker hash is allowed by the TEE's RTMR3,
        // update the state
        if votes >= self.threshold()?.value() {
            self.tee_state
                .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
        }

        Ok(())
    }

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
    }
