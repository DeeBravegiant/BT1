### Title
Caller-Controlled `AppPublicKey` Variant Bypasses On-Chain CKD Output Verification, Enabling a Single Byzantine Node to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract contains a structural analog to the reported `confirm=false` bypass. The user-controlled `app_public_key` field in `CKDRequestArgs` selects between two variants: `AppPublicKey` (legacy) and `AppPublicKeyPV` (publicly verifiable). When `AppPublicKey` is chosen, the contract's `respond_ckd` handler takes an empty match arm and performs **zero cryptographic verification** of the returned `(big_y, big_c)` ciphertext. A single attested MPC participant can therefore call `respond_ckd` with an arbitrary forged response and the contract will accept and deliver it to the user, bypassing the threshold-signature requirement for CKD output integrity.

---

### Finding Description

In `respond_ckd`, the contract branches on the `app_public_key` variant stored in the pending `CKDRequest`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← empty: no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
pending_requests::resolve_yields_for(...)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically proves the response was computed using the MPC master secret key. [2](#0-1) 

For `AppPublicKey`, **no such check exists**. The response is accepted unconditionally and forwarded to the user. The variant is set by the user (or a dapp acting on their behalf) at request time in `request_app_private_key`: [3](#0-2) 

The `AppPublicKey` variant is the default legacy format, accepted as a plain G1 string, and is the format shown in the contract README's primary example: [4](#0-3) 

The existing unit test confirms that completely arbitrary byte arrays are accepted as `big_y`/`big_c` when `AppPublicKey` is used — no point validity or protocol correctness check is applied: [5](#0-4) 

---

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without the required participant authorization.**

A single Byzantine attested MPC node can call `respond_ckd` with an arbitrary `CKDResponse`. By choosing `big_y = G1_identity` and `big_c = k·G1` for any known scalar `k`, the attacker causes the user to decrypt `k·G1 - ephemeral_sk · G1_identity = k·G1`. The attacker knows `k·G1` and can derive the identical application key via the same HKDF step the user applies. The user receives a key that appears valid but is fully known to the attacker, who can then access any asset or secret the user protects with that key.

This bypasses the threshold requirement: the `ckd_output_check` pairing equation is the only on-chain mechanism that proves the response required cooperation of a threshold of honest nodes. Removing it for `AppPublicKey` reduces the effective security of CKD from t-of-n to 1-of-n.

---

### Likelihood Explanation

**Medium-High.** The `AppPublicKey` (legacy) format is the default and most-documented path. Any attested MPC participant — a single node that has turned malicious — can race to submit a forged `respond_ckd` before honest nodes. The NEAR yield-resume mechanism resolves on the first valid `respond_ckd` call, so the first forged submission wins. No threshold collusion is required; one Byzantine node strictly below the signing threshold is sufficient. [6](#0-5) 

---

### Recommendation

Remove the `AppPublicKey` variant's empty match arm and apply an equivalent output check. Since `AppPublicKey` provides only `pk1` (a single G1 point, no `pk2`), the publicly verifiable pairing check cannot be applied directly. The options are:

1. **Deprecate `AppPublicKey` for new requests** and require `AppPublicKeyPV` for all new CKD submissions, so the pairing check is always enforced on-chain.
2. **Alternatively**, require nodes to also submit a zero-knowledge proof of correct computation when responding to `AppPublicKey` requests, verified in the `respond_ckd` handler.

The contract should never allow the caller to silently opt out of the cryptographic output integrity check.

---

### Proof of Concept

1. User (or dapp) submits a CKD request using the legacy `AppPublicKey` format:
   ```json
   { "request": { "derivation_path": "mykey", "app_public_key": "bls12381g1:<pk1>", "domain_id": 2 } }
   ```
   This stores a `CKDRequest` with `app_public_key = AppPublicKey(pk1)`. [7](#0-6) 

2. A single malicious attested node constructs a forged response. Setting `big_y = G1_identity` and `big_c = k·G1` for attacker-chosen scalar `k`:
   ```rust
   let forged = CKDResponse {
       big_y: Bls12381G1PublicKey(G1_IDENTITY_COMPRESSED),
       big_c: Bls12381G1PublicKey((G1::generator() * k).to_compressed()),
   };
   contract.respond_ckd(ckd_request, forged);
   ```

3. `respond_ckd` hits the empty `AppPublicKey(_) => {}` arm — no check — and calls `resolve_yields_for`, delivering the forged `(big_y, big_c)` to the user. [1](#0-0) 

4. The user decrypts: `big_c - ephemeral_sk · big_y = k·G1 - ephemeral_sk · 0 = k·G1`. The attacker knows `k·G1` and derives the identical application key. The existing test already demonstrates this acceptance of arbitrary byte arrays: [8](#0-7)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
