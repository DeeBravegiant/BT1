### Title
Stale Removed-Participant Votes Counted Toward TEE Governance Thresholds in Post-Resharing Window — (`File: crates/contract/src/tee/proposal.rs`, `crates/contract/src/tee/measurements.rs`)

---

### Summary

`CodeHashesVotes::count_votes()`, `LauncherHashVotes::count_votes()`, and `MeasurementVotes::count_votes()` iterate over all stored votes without filtering by the current participant set. After a resharing, the cleanup (`clean_tee_status`) is dispatched as a **detached promise** and executes in a separate receipt — creating a window where stale votes from removed participants are still present and counted. Any current participant who calls `vote_code_hash`, `vote_add_launcher_hash`, or `vote_add_os_measurement` during this window will trigger a `count_votes()` that includes removed participants' votes, potentially crossing the threshold with fewer current-participant votes than required.

---

### Finding Description

Three vote-counting structures share the same root cause:

**`CodeHashesVotes::count_votes()`** (used by `vote_code_hash`):
```rust
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64   // ← no participant-membership filter
}
``` [1](#0-0) 

**`MeasurementVotes::count_votes()`** (used by `vote_add_os_measurement` / `vote_remove_os_measurement`):
```rust

            .values()
            .filter(|a| *a == action)

Contrast this with `ThresholdParametersVotes::n_votes()`, which correctly filters by current participants before counting: [4](#0-3) 

The cleanup that removes stale votes is `clean_tee_status`, which calls `tee_state.clean_non_participant_votes(participants)`: [5](#0-4) 

This cleanup is spawned from `vote_reshared` as a **detached promise** (`.detach()`), meaning it executes in a separate receipt, not atomically with the state transition: [6](#0-5) 

Between the `vote_reshared` transaction completing (contract now in Running state with new participants) and the `clean_tee_status` receipt executing, there is a window of at least one block where:
- The contract is in Running state with the new participant set.
- Stale votes from removed participants remain in `CodeHashesVotes`, `LauncherHashVotes`, and `MeasurementVotes`.
- Any current participant can call `vote_code_hash`, `vote_add_launcher_hash`, or `vote_add_os_measurement`.

The `vote_add_os_measurement` handler reads the count directly from `vote_measurement()` (which calls `count_votes()`) and compares it to the threshold without any participant filtering: [7](#0-6) 

The `vote_remove_os_measurement` handler compares the unfiltered count against `total_participants` (the new, smaller set), making it even easier to satisfy the unanimity requirement with stale votes: [8](#0-7)

### Impact Explanation

**Concrete attack scenario (N=5, T=3):**

1. Before resharing: participants P1–P5. P1 and P2 vote for malicious MPC Docker image hash `H` via `vote_code_hash`.
2. Resharing removes P1 and P2. New set: {P3, P4, P5}, threshold=3.
3. `clean_tee_status` is scheduled as a detached promise but has not yet executed.
4. P3 (a current participant, possibly the attacker or a colluding node) calls `vote_code_hash(H)`.
5. `CodeHashesVotes::count_votes(H)` returns 3 (P1 + P2 + P3). Threshold met. `H` is whitelisted.

Only 1 current participant voted, but the threshold of 3 was satisfied by stale votes. The malicious Docker image hash `H` is now in the TEE allowlist. A node running image `H` can submit a valid attestation and be admitted to the MPC network, gaining access to key shares and signing capability.

For `vote_remove_os_measurement` (unanimity required): if removed participants had voted for removal, the unanimity check `votes >= total_participants` uses the new (smaller) `total_participants` count but the unfiltered vote count, making it trivially satisfiable.

This breaks the threshold-signature admission invariant: unauthorized TEE images can be whitelisted with sub-threshold current-participant support, enabling unauthorized nodes to participate in key generation and signing.

---

### Likelihood Explanation

- The window is one NEAR block (~1 second), but NEAR processes transactions asynchronously and the detached promise executes in a separate receipt. Any transaction submitted in the same block as `vote_reshared` (or the immediately following block) can exploit this.
- The attacker only needs to be a current participant (below threshold) and know that stale votes exist — this is observable on-chain.
- No privileged access, TEE attack, or threshold collusion is required. A single current participant can trigger the issue if removed participants had previously voted for the same proposal.
- The codebase's own test comments acknowledge this pattern as a known risk for `vote_update` (which was fixed with participant filtering), but the TEE vote counters were not similarly hardened. [9](#0-8) 

---

### Recommendation

Apply the same participant-membership filter used in `ThresholdParametersVotes::n_votes()` to all three `count_votes()` methods. Each should accept a `&Participants` argument and filter `vote_by_account` / `proposal_by_account` to only count entries whose key is a current participant before comparing against the threshold. This eliminates the dependency on the asynchronous cleanup promise for correctness.

Alternatively, make `clean_tee_status` a synchronous inline call within `vote_reshared` before returning, rather than a detached promise, so the state is always consistent when the new Running state becomes visible.

---

### Proof of Concept

**Step-by-step (N=5, T=3, targeting `vote_add_os_measurement`):**

1. Pre-resharing: P1 and P2 call `vote_add_os_measurement(M)`. `MeasurementVotes.vote_by_account` now contains `{P1→Add(M), P2→Add(M)}`. Count = 2, threshold not met.
2. Resharing completes via `vote_reshared`. New Running state: {P3, P4, P5}, T=3. `clean_tee_status` is dispatched as a detached promise (separate receipt, not yet executed).
3. In the same block (or before `clean_tee_status` executes), P3 calls `vote_add_os_measurement(M)`.
4. `MeasurementVotes::vote()` inserts P3's vote and calls `count_votes(Add(M))`.
5. `count_votes` iterates all values: P1→Add(M), P2→Add(M), P3→Add(M) → returns 3.
6. `3 >= threshold(3)` → `tee_state.add_measurement(M)` is called. Measurement `M` is whitelisted.
7. `clean_tee_status` eventually executes and removes P1/P2's votes — but `M` is already in `allowed_measurements`.

The measurement `M` was whitelisted with only 1 current-participant vote (P3), violating the threshold-3 requirement. [10](#0-9) [11](#0-10)
