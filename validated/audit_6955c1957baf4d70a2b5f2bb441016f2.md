### Title
Inconsistent On-Chain Response Verification for Legacy vs. PV CKD Requests Allows Single Byzantine Participant to Forge Key Derivation Output — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` applies full cryptographic output verification only for `AppPublicKeyPV` (publicly-verifiable) CKD requests. For the legacy `AppPublicKey` variant, **no verification is performed at all**. Because a single attested participant can call `respond_ckd` and immediately resolve all queued yields, a single Byzantine participant (strictly below the signing threshold) can submit an arbitrary forged CKD response for any legacy request and the contract will deliver it to the user without challenge.

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the encrypted output is correctly derived from the master BLS key and properly encrypted to the app's ephemeral public key. For the legacy `AppPublicKey` variant, the arm is a no-op — the response bytes are accepted unconditionally and immediately fanned out to all waiting callers via `resolve_yields_for`. [2](#0-1) 

Compare this with `respond`, which always verifies the ECDSA/EdDSA signature against the derived public key before resolving: [3](#0-2) 

The inconsistency is structural: the newer `AppPublicKeyPV` path has a strong on-chain integrity guarantee; the legacy `AppPublicKey` path has none.

---

### Impact Explanation

**Critical.** A single Byzantine attested participant (one node, strictly below the signing threshold) can call `respond_ckd` with an arbitrary, attacker-chosen `CKDResponse` for any pending legacy `AppPublicKey` request. The contract resolves all queued yields with the forged payload. The victim application receives key material that the attacker already knows (because the attacker crafted it), enabling:

- Decryption of all messages the victim app encrypts to the "derived" key.
- Impersonation of the victim app's derived identity on any downstream protocol.

This constitutes **unauthorized confidential key derivation output without the required participant authorization** — the threshold of honest nodes never agreed to this output.

---

### Likelihood Explanation

**Medium.** The attack requires exactly one attested MPC participant to be Byzantine. TEE attestation raises the bar, but:

- Only one of the n participants needs to be compromised or malicious.
- The attacker does not need to break the threshold (t-of-n) — a single node suffices because `respond_ckd` is a single-caller endpoint.
- Legacy `AppPublicKey` requests are still accepted by the contract (backwards-compatibility path), so the attack surface is live in production.

---

### Recommendation

Apply the same `ckd_output_check` (or an equivalent binding check) to the `AppPublicKey` branch. If the legacy format does not carry enough public information to verify the output on-chain, the contract should either:

1. Reject new `AppPublicKey` requests (deprecate the legacy path), or
2. Require the responding node to include a zero-knowledge proof or a BLS signature over the response that the contract can verify against the stored master public key, analogous to the ECDSA/EdDSA verification in `respond`.

---

### Proof of Concept

1. Alice submits `request_app_private_key` with `app_public_key: AppPublicKey(alice_g1_point)` (legacy format). A `CKDRequest` is stored and a yield is parked.
2. Mallory, a single attested participant, constructs a `CKDResponse` containing a BLS ciphertext she controls (i.e., encrypted to a key she knows).
3. Mallory calls `respond_ckd(request, forged_response)`. The contract reaches the `AppPublicKey(_) => {}` arm — no check fires.
4. `resolve_yields_for` resumes Alice's yield with the forged bytes. [4](#0-3) 

5. Alice's callback receives the forged CKD output. Mallory, knowing the plaintext of the ciphertext she crafted, can decrypt everything Alice subsequently encrypts to the "derived" key.
