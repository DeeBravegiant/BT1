Audit Report

## Title
Unbounded Gas Consumption in `_getTotalEthInProtocol()` Due to Uncapped `supportedAssetList` Iteration — (`contracts/LRTOracle.sol`)

## Summary
`LRTOracle._getTotalEthInProtocol()` iterates over the entire `supportedAssetList` from `LRTConfig`, and for each asset calls `LRTDepositPool.getTotalAssetDeposits()`, which itself iterates over `nodeDelegatorQueue`. `supportedAssetList` has no enforced maximum length, making the gas cost of the publicly callable `updateRSETHPrice()` grow without bound as the protocol adds assets, eventually causing a permanent OOG revert that freezes price updates, fee minting, and the downside circuit-breaker.

## Finding Description
`LRTConfig._addNewSupportedAsset()` pushes to `supportedAssetList` with no length cap: [1](#0-0) 

`LRTOracle._getTotalEthInProtocol()` fetches the full array and loops over every element, calling `getTotalAssetDeposits()` per asset: [2](#0-1) 

`getTotalAssetDeposits()` delegates to `getAssetDistributionData()`, which loops over the entire `nodeDelegatorQueue` making three external calls per NDC (`balanceOf`, `getAssetBalance`, `getAssetUnstaking`): [3](#0-2) 

`_getTotalEthInProtocol()` is called unconditionally inside `_updateRsETHPrice()`: [4](#0-3) 

which is exposed via the access-control-free public function `updateRSETHPrice()`: [5](#0-4) 

By contrast, `nodeDelegatorQueue` is bounded by `maxNodeDelegatorLimit` (default 10): [6](#0-5) 

No analogous cap exists for `supportedAssetList`. The total gas cost is **O(supportedAssets × nodeDelegators × 3 external calls)**, all of which execute on every `updateRSETHPrice()` invocation.

## Impact Explanation
**Medium — Unbounded gas consumption** (explicitly listed in allowed impact scope).

When the accumulated asset count × NDC count causes `updateRSETHPrice()` to exceed the block gas limit:
1. `rsETHPrice` becomes permanently stale — all new deposits mint rsETH at an incorrect rate.
2. Protocol fee minting via `IRSETH.mint(treasury, ...)` inside `_updateRsETHPrice()` can no longer execute — permanent freezing of unclaimed yield.
3. The downside price-protection circuit-breaker (which pauses the protocol on a price drop) stops working, as it depends on `_updateRsETHPrice()` completing. [7](#0-6) 

## Likelihood Explanation
`addNewSupportedAsset` requires `TIME_LOCK_ROLE` and is a routine governance operation — not an attack vector. The protocol already initializes with 2 assets (stETH, ETHx) and is explicitly designed to expand its LST/LRT asset list over time. `maxNodeDelegatorLimit` can be raised by admin with no upper cap. At a realistic scale of 20–30 supported assets and 15–20 NDCs, the function executes 900–1,800 external calls. At ~5,000 gas per external call plus overhead, this approaches or exceeds the block gas limit on L2s where the protocol operates. No attacker action is required — normal protocol growth triggers the condition.

## Recommendation
1. Enforce a maximum length on `supportedAssetList` inside `_addNewSupportedAsset()`, analogous to the `maxNodeDelegatorLimit` guard on `nodeDelegatorQueue`.
2. Consider caching per-asset TVL and updating it incrementally (e.g., on deposit/withdrawal events) rather than recomputing the full sum on every `updateRSETHPrice()` call.
3. Alternatively, introduce a paginated or keeper-assisted pattern for `_getTotalEthInProtocol()` so the computation can be split across multiple transactions.

## Proof of Concept
**Call chain:**
```
updateRSETHPrice()                          [public, no access control]
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()
            └─ for each asset in supportedAssetList:          [unbounded]
                 └─ getTotalAssetDeposits(asset)
                      └─ getAssetDistributionData(asset)
                           └─ for each NDC in nodeDelegatorQueue:  [up to maxNodeDelegatorLimit]
                                └─ IERC20.balanceOf()           [~5k gas]
                                └─ INodeDelegator.getAssetBalance()  [~5k gas]
                                └─ INodeDelegator.getAssetUnstaking() [~5k gas]
```

**Foundry fork test plan:**
1. Deploy the protocol on a fork.
2. Via governance, call `addNewSupportedAsset()` 28 additional times (reaching 30 total assets), each with a mock ERC20 and price oracle.
3. Via admin, raise `maxNodeDelegatorLimit` to 20 and add 20 NDC contracts.
4. Call `updateRSETHPrice()` with a gas limit matching the target chain's block gas limit.
5. Observe OOG revert; confirm `rsETHPrice` is not updated and fee minting does not occur.

At 30 assets × 20 NDCs × 3 calls × ~5,000 gas = 9,000,000 gas, exceeding Ethereum mainnet's ~30M limit is achievable at higher counts, and L2 sequencer gas limits (often 10–15M) are exceeded at this scale.

**File:** contracts/LRTOracle.sol (L331-349)

```text
    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
```
