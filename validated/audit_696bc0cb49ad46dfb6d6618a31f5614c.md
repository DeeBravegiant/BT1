### Title
Missing `payload_hash` Binding to Stored Request in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` accepts a caller-supplied `response.payload_hash` and only checks that the attached signature is valid for that hash under the root public key. It never verifies that `response.payload_hash` actually corresponds to the stored pending `request`. By contrast, `respond` derives the payload hash directly from the stored `SignatureRequest`, making it impossible for the caller to supply a mismatched hash. A single malicious attested MPC participant (strictly below threshold) can replay a threshold signature produced for one foreign-tx request to resolve a different pending request with a fabricated `payload_hash`, constituting forged foreign-chain verification.

---

### Finding Description

**In `respond`**, the payload hash used for signature verification is taken directly from the stored request — the MPC node cannot supply a different value:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [1](#0-0) 

**In `respond_verify_foreign_tx`**, the payload hash is taken from the response provided by the calling MPC node, not from the stored request:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
``` [2](#0-1) 

The contract only checks that the signature is valid for `response.payload_hash` against the root key. It does **not** check that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayloadV1 { stored_request, extracted_values }))`. [3](#0-2) 

The `payload_hash` is computed by the MPC nodes off-chain as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [4](#0-3) 

The node-side code confirms the zero-tweak signing (root key, no derivation):

```rust
tweak: Tweak::new([0u8; 32]),
``` [5](#0-4) 

**Attack scenario:**

1. Request A (`tx_id_A`) and Request B (`tx_id_B`) are both pending in `pending_verify_foreign_tx_requests`.
2. The MPC network produces `payload_hash_A = SHA-256(borsh(ForeignTxSignPayloadV1 { request_A, values_A }))` and threshold signature `sig_A`.
3. A single malicious attested participant calls `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
4. The contract checks: is `request_B` pending? ✓ Is `sig_A` valid for `payload_hash_A` against root key? ✓
5. The contract resolves request B with `{ payload_hash: payload_hash_A, signature: sig_A }` — the hash of a completely different transaction's data. [6](#0-5) 

---

### Impact Explanation

The user who submitted request B receives a `VerifyForeignTransactionResponse` containing `payload_hash_A` (the hash of request A's data and extracted values) paired with a valid threshold signature. Any bridge or application that does not independently recompute and verify the `payload_hash` against the expected request data will accept this as a valid on-chain attestation of request B, when in fact request B was never verified by the MPC network. This constitutes **forged foreign-chain verification** and can cause invalid bridge execution or double-spend conditions. The documentation notes that callers *should* verify the hash locally, but the contract does not enforce this, and many integrations will not. [7](#0-6) 

---

### Likelihood Explanation

The attack requires only a single malicious attested MPC participant (strictly below the signing threshold) and two concurrently pending foreign-tx requests. In a production bridge environment, multiple concurrent requests are routine. The attacker needs access to any previously produced threshold signature for any foreign-tx request — these are submitted on-chain and are publicly observable. No threshold collusion is required; the malicious node simply replays an already-produced signature for a different pending request. [8](#0-7) 

---

### Recommendation

Include the `extracted_values` in the `VerifyForeignTransactionResponse` and have the contract independently compute `expected_payload_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request, extracted_values }))`, then assert `response.payload_hash == expected_payload_hash` before accepting the response. If promise data size is a concern, the signing protocol should instead sign `hash(request_canonical_bytes || payload_hash)` so that a signature for one request cannot be replayed for another. At minimum, the contract should reject any `respond_verify_foreign_tx` call where `response.payload_hash` cannot be bound to the specific stored `request`.

---

### Proof of Concept

```rust
// Setup: two pending foreign-tx requests
contract.verify_foreign_transaction(request_args_A); // request A pending
contract.verify_foreign_transaction(request_args_B); // request B pending

// MPC network produces a valid response for request A
let payload_A = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request_A.request.clone(),
    values: extracted_values_A,
});
let payload_hash_A = payload_A.compute_msg_hash().unwrap();
let (sig_A, recovery_id) = signing_key.sign_prehash_recoverable(&payload_hash_A.0).unwrap();
let response_A = VerifyForeignTransactionResponse {
    payload_hash: payload_hash_A,
    signature: SignatureResponse::Secp256k1(
        K256Signature::from_ecdsa_recoverable(&sig_A, recovery_id)
    ),
};

// Malicious attested participant replays sig_A to resolve request B.
// respond_verify_foreign_tx checks:
//   (1) is request_B pending?          -> YES
//   (2) is sig_A valid for payload_hash_A against root key? -> YES
// It does NOT check:
//   (3) does payload_hash_A correspond to request_B?        -> MISSING
contract.respond_verify_foreign_tx(request_B, response_A)
    .expect("succeeds — no binding check between payload_hash and stored request");

// Request B is now resolved with payload_hash_A (hash of request A's data).
// The user of request B receives a forged verification attestation.
assert!(contract.get_pending_verify_foreign_tx_request(&request_B).is_none());
``` [9](#0-8)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
