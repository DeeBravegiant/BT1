### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in `TransactionConverter` Discards Actual L1 Fee, Bypassing Fee Enforcement for All Consensus-Path L1 Handler Transactions — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`TransactionConverter::convert_consensus_l1_handler_to_internal_l1_handler` unconditionally injects `paid_fee_on_l1 = Fee(1)` when converting every `ConsensusTransaction::L1Handler` into an executable `L1HandlerTransaction`. The actual ETH fee paid on L1 is never carried through the consensus wire format and is permanently discarded at the conversion boundary. Because the blockifier's fee guard only checks `paid_fee != Fee(0)`, the hardcoded `Fee(1)` satisfies that guard for every L1 handler transaction regardless of what was actually paid, producing wrong fee accounting in every executed receipt and making the fee-sufficiency invariant unenforceable.

---

### Finding Description

**The intermediary that loses context.** `FeeWrapper.rubicall` was the intermediary that replaced `msg.sender` with the wrapper's own address. Here, `TransactionConverter` is the intermediary that replaces the real `paid_fee_on_l1` with a sentinel.

The consensus wire type `ConsensusTransaction::L1Handler` carries only `transaction::L1HandlerTransaction` (no fee field). When a validator receives a proposal containing L1 handler transactions, `handle_proposal_part` calls `transaction_converter.convert_consensus_tx_to_internal_consensus_tx`, which dispatches to:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs:473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

The resulting `executable_transaction::L1HandlerTransaction` carries `paid_fee_on_l1 = Fee(1)` into the blockifier. The blockifier's fee guard is:

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs:103-113
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
``` [2](#0-1) 

`Fee(1) != Fee(0)` is always true, so the guard is permanently satisfied for every L1 handler transaction that arrives through consensus, regardless of the actual ETH value attached to the originating `sendMessageToL2` call on L1.

The `executable_transaction::L1HandlerTransaction` struct stores `paid_fee_on_l1` as a first-class field:

```rust
// crates/starknet_api/src/executable_transaction.rs:380-396
pub struct L1HandlerTransaction {
    pub tx: crate::transaction::L1HandlerTransaction,
    pub tx_hash: TransactionHash,
    pub paid_fee_on_l1: Fee,
}
``` [3](#0-2) 

This field is the authoritative fee value used throughout execution. Hardcoding it to `Fee(1)` means every validator node executes L1 handler transactions with a fabricated fee value, producing receipts and state diffs that record `Fee(1)` instead of the real amount.

The conversion is invoked on every incoming proposal transaction batch:

```rust

### Impact Explanation

Every L1 handler transaction processed through the consensus validation path is executed with `paid_fee_on_l1 = Fee(1)`:

1. **Wrong receipt/state value.** The fee recorded in the execution receipt is `Fee(1)` rather than the actual ETH paid. Any downstream consumer of receipts (RPC `starknet_getTransactionReceipt`, provers, L1 verifiers) sees a fabricated fee.

2. **Fee-sufficiency check permanently bypassed.** The blockifier's guard `paid_fee == Fee(0)` is the only enforcement point. With `Fee(1)` hardcoded, a proposer can include L1 handler transactions whose originating L1 call paid an arbitrarily small fee (e.g., 1 wei), and every validator will accept and execute them, producing committed blocks with under-priced L1 handler transactions.

3. **Future guard breakage.** The TODO comment acknowledges the intent to enforce a real fee check. When that check is added, all existing consensus-path L1 handler transactions will retroactively fail, because the stored `paid_fee_on_l1 = Fee(1)` will be below any realistic execution cost threshold.

This matches the impact category: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

---

### Likelihood Explanation

- Every L1 handler transaction that enters the sequencer through the consensus proposal path is affected — this is not a corner case.
- No special attacker capability is required. Any user who sends a `sendMessageToL2` call on L1 with a minimal ETH value will have their transaction accepted and executed by all validators with `paid_fee_on_l1 = Fee(1)`.
- The proposer node itself applies the same hardcoded value when building a block (the `BlockBuilderFactory` creates a `TransactionConverter` with the same code path).
- The bug is self-confirming: the TODO comment in production code explicitly acknowledges the placeholder is not the real value.

---

### Recommendation

1. Carry `paid_fee_on_l1` through the consensus wire format. Add a `paid_fee_on_l1: Fee` field to `ConsensusTransaction::L1Handler` (and the corresponding protobuf message) so the actual ETH value is transmitted from proposer to validator.

2. Remove the hardcoded `Fee(1)` in `convert_consensus_l1_handler_to_internal_l1_handler` and use the value from the wire message.

3. Strengthen the blockifier fee guard from `paid_fee != Fee(0)` to `paid_fee >= receipt.fee` once the real value is available, as the TODO intends.

---

### Proof of Concept

**Setup:** A user sends `sendMessageToL2` on L1 with `value = 1 wei` (minimum non-zero ETH). The Starknet L1 contract accepts it and emits a message. The sequencer picks it up as an L1 handler transaction.

**Proposer path:**
- `BlockBuilder` calls `TransactionConverter::convert_internal_consensus_tx_to_executable_tx` → `convert_consensus_l1_handler_to_internal_l1_handler` → `paid_fee_on_l1 = Fee(1)`.
- The blockifier executes the transaction. The fee check `Fee(1) != Fee(0)` passes.
- The block is committed with `paid_fee_on_l1 = Fee(1)` in the receipt, regardless of actual execution cost.

**Validator path:**
- `handle_proposal_part` receives `ConsensusTransaction::L1Handler(tx)`.
- `convert_consensus_tx_to_internal_consensus_tx` → `convert_consensus_l1_handler_to_internal_l1_handler` → `paid_fee_on_l1 = Fee(1)`.
- The blockifier executes the transaction. The fee check `Fee(1) != Fee(0)` passes.
- The validator's `batcher_block_commitment` matches the proposer's, so `ProposalFinMismatch` is not triggered.
- The block is accepted with wrong fee accounting.

**Observed invariant violation:** `receipt.paid_fee_on_l1 = Fee(1)` for all L1 handler transactions, regardless of actual ETH paid. The fee-sufficiency check `paid_fee >= actual_execution_cost` is never enforced. [1](#0-0) [5](#0-4) [6](#0-5) [7](#0-6)
