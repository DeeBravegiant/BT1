### Title
Queue-Filling Front-Run Blocks Legitimate `verify_foreign_transaction` Callers via Caller-Agnostic Request Key - (File: crates/contract/src/lib.rs)

### Summary
The `verify_foreign_transaction` endpoint uses a caller-agnostic request key (no predecessor account ID), meaning all callers share a single fan-out queue slot per `(domain_id, payload_version, rpc_request)` tuple. An unprivileged attacker can front-run a victim's submission by filling that shared queue to the 128-entry cap (`MAX_PENDING_REQUEST_FAN_OUT`), causing the victim's transaction to panic with `PendingRequestQueueFull` and permanently blocking their request until the attacker's queue entries are drained.

### Finding Description

The `sign` endpoint binds its pending-request queue key to the caller's account ID via `predecessor`:

```rust
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // caller-specific — each account has its own queue slot
    &request.path,
);
``` [1](#0-0) 

The `verify_foreign_transaction` endpoint does **not** include the caller:

```rust
let request = args_into_verify_foreign_tx_request(request);
// ... no predecessor bound into the key
self.enqueue_yield_request(..., move |this, id| this.add_verify_foreign_tx_request(request, id));
``` [2](#0-1) 

The resulting `VerifyForeignTransactionRequest` key is `(domain_id, payload_version, rpc_request)` — shared across every caller who submits the same foreign-chain transaction. The codebase itself acknowledges this in a test comment:

> "a different account would today be blocked from receiving a response by alice's submission." [3](#0-2) 

`push_pending_yield` enforces a hard cap of 128 entries per key and panics if exceeded:

```rust
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
}
``` [4](#0-3) 

`pop_oldest_pending_yield` removes entries in FIFO order on timeout, meaning the attacker's entries occupy the queue until the MPC network responds (draining all 128 at once) or until each times out individually. [5](#0-4) 

### Impact Explanation

An attacker who submits 128 identical `verify_foreign_transaction` calls for a specific foreign-chain transaction (e.g., a Bitcoin withdrawal tx) fills the shared queue to capacity. Any subsequent submission by a legitimate user for the same transaction panics immediately. The MPC network then processes the attacker's 128 queued requests and delivers the signed attestation to all 128 of the attacker's yields. The victim's request was never enqueued and receives nothing. The victim must wait for the attacker to stop refilling the queue before they can participate. In a bridge context, this can permanently block a specific cross-chain withdrawal from being attested, breaking the request-lifecycle safety invariant for that transaction.

**Impact class:** Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

### Likelihood Explanation

The attack requires only an unprivileged NEAR account and the ability to observe a pending `verify_foreign_transaction` submission in the mempool (or simply submit preemptively for a known tx ID). The cost is 128 × `MINIMUM_SIGN_REQUEST_DEPOSIT` (1 yoctoNEAR each per the test setup), which is negligible. The attacker can sustain the block indefinitely by refilling the queue after each drain cycle. No privileged access, key material, or threshold collusion is required.

### Recommendation

Bind the `verify_foreign_transaction` pending-request queue key to the caller's `predecessor_account_id`, mirroring the design of `sign`. This gives each caller an independent queue slot and eliminates cross-caller interference. Alternatively, enforce a per-account submission rate limit for `verify_foreign_transaction` requests.

### Proof of Concept

1. Alice observes (or predicts) that Bob will call `verify_foreign_transaction` for Bitcoin tx `T` on domain `D`.
2. Alice submits 128 identical `verify_foreign_transaction` calls for `(D, V1, T)` with `max_gas` and `1 yoctoNEAR` each, filling `pending_verify_foreign_tx_requests[(D,V1,T)]` to `MAX_PENDING_REQUEST_FAN_OUT = 128`.
3. Bob submits `verify_foreign_transaction` for the same `(D, V1, T)`. The call reaches `push_pending_yield`, finds `queue.len() == 128`, and panics with `PendingRequestQueueFull`. Bob's deposit is returned but his request is not enqueued.
4. The MPC network processes the request and calls `respond_verify_foreign_tx`, which invokes `resolve_yields_for` and drains all 128 of Alice's yields. Alice receives 128 signed attestations; Bob receives none.
5. Alice immediately resubmits 128 calls to re-fill the queue, sustaining the block indefinitely. [6](#0-5) [7](#0-6) [8](#0-7)

            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),

pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

pub(crate) fn pop_oldest_pending_yield<K>(requests: &mut LookupMap<K, Vec<YieldIndex>>, request: &K)
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let Some(queue) = requests.get_mut(request) else {
        return;
    };
    if queue.is_empty() {
        requests.remove(request);
        return;
    }
    queue.remove(0);
    if queue.is_empty() {
        requests.remove(request);
    }
