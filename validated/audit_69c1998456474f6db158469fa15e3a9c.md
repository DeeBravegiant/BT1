Audit Report

## Title
Unvalidated Chainlink `latestRoundData()` Output Enables Stale/Zero Price to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `AggregatorV3Interface.latestRoundData()` and discards all return values except `price`, with no staleness, completeness, or sign checks. This function sits directly in the rsETH minting path: `LRTDepositPool.depositAsset()` → `getRsETHAmountToMint()` → `LRTOracle.getAssetPrice()` → `ChainlinkPriceOracle.getAssetPrice()`. A stale, zero, or negative price from Chainlink will silently corrupt the rsETH mint amount, causing protocol insolvency or depositor fund loss. The same codebase already implements all three required checks in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the omission is a code-level defect.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`) are available but only `answer` is consumed. No checks are performed. [1](#0-0) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` in the same repository correctly validates all three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The minting path is:

1. `LRTDepositPool.depositAsset()` (public, no privilege) calls `_beforeDeposit()`.
2. `_beforeDeposit()` calls `getRsETHAmountToMint(asset, depositAmount)`.
3. `getRsETHAmountToMint` computes: `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`.
4. `lrtOracle.getAssetPrice(asset)` delegates to `ChainlinkPriceOracle.getAssetPrice()`. [3](#0-2) 

The stored `rsETHPrice` is updated separately via `updateRSETHPrice()` and is not refreshed during a deposit. If the Chainlink feed is stale at deposit time, the live `getAssetPrice()` call returns the stale value while `rsETHPrice` reflects the last valid state, corrupting the ratio.

`RSETHPriceFeed.latestRoundData()` has the same omission — it forwards `ETH_TO_USD.latestRoundData()` with no validation before multiplying by `rsETHPrice`. [4](#0-3) 

The `pricePercentageLimit` guard in `LRTOracle._updateRsETHPrice()` does not protect deposits because it only runs when `updateRSETHPrice()` is explicitly called, not during the deposit flow. [5](#0-4) 

## Impact Explanation

- **Stale price (inflated):** `getAssetPrice` returns a higher-than-true value → depositor receives more rsETH than the deposited asset is worth → existing rsETH holders are diluted → **protocol insolvency (Critical)**.
- **Stale price (deflated):** depositor receives fewer rsETH tokens than owed → **direct theft of depositor value (Critical)**.
- **Zero price:** `uint256(0)` → `rsethAmountToMint = 0`; if `minRSETHAmountExpected = 0`, the deposit succeeds, the depositor's LST is transferred in, and zero rsETH is minted → **permanent loss of depositor funds (Critical)**.
- **Negative price:** `uint256(negative_int256)` wraps to a near-`type(uint256).max` value → wildly inflated rsETH mint → **protocol insolvency (Critical)**.

All four scenarios map to allowed Critical impacts: direct theft of user funds, permanent freezing of funds, or protocol insolvency.

## Likelihood Explanation

Chainlink feeds return stale data during network congestion preventing heartbeat updates, sequencer downtime on L2 deployments (no sequencer uptime check is present), and Chainlink node failures or feed deprecation. These are historically observed, non-theoretical conditions. No special privileges are required: any user calling `LRTDepositPool.depositAsset()` during a stale-feed window triggers the vulnerable path. The attack is repeatable for the duration of the staleness window.

## Recommendation

Apply the same three-check pattern already used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundID, int256 price,, uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(answeredInRound >= roundID, "Chainlink Price Stale");
require(price > 0, "Chainlink Malfunction");
require(updatedAt != 0, "Incomplete round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
```

Apply equivalent checks in `RSETHPriceFeed.latestRoundData()` before multiplying `answer` by `rsETHPrice`.

## Proof of Concept

**Foundry fork test outline:**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed's `answeredInRound < roundId` (simulate by deploying a mock `AggregatorV3Interface` that returns a stale round with an artificially low price and `answeredInRound = roundId - 1`).
2. Register the mock feed via `ChainlinkPriceOracle.updatePriceFeedFor(stETH, mockFeed)`.
3. Call `LRTDepositPool.depositAsset(stETH, 1e18, 0, "")` as an unprivileged EOA.
4. Observe that `getRsETHAmountToMint` returns an inflated value (stale low price → inflated ratio) and rsETH is minted without revert.
5. Assert `rsethMinted > expectedRsethAtTruePrice`, confirming over-issuance.

For the zero-price case: set mock to return `answer = 0`, call `depositAsset` with `minRSETHAmountExpected = 0`, and assert the depositor's stETH is transferred in while zero rsETH is minted.
