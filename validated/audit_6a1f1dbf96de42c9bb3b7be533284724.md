### Title
Missing Transaction Hash Validation in P2P Sync Allows Malicious Peer to Corrupt Stored Transaction Hashes — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `transaction_hash` directly from an unauthenticated p2p peer's `FullTransaction` message and stores it in the node's body storage without any validation that the hash actually corresponds to the transaction body. The code itself acknowledges this with an explicit TODO comment.

---

### Finding Description

In `parse_data_for_block`, each `FullTransaction { transaction, transaction_output, transaction_hash }` is received from a peer-controlled `ClientResponsesManager`. The only checks performed are:
- Count: that the number of transactions matches `n_transactions` from the stored header.
- Protobuf decode validity.

No check is made that `transaction_hash` is the canonical hash of `transaction`. The peer-supplied hash is pushed directly:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [1](#0-0) 

This `BlockBody` is then written to persistent storage unconditionally via `write_to_storage`:

```rust
storage_writer.begin_rw_txn()?.append_body(self.1, self.0)?.commit()?;
``` [2](#0-1) 

Inside `write_transactions`, the peer-supplied `tx_hash` is inserted as the key in `transaction_hash_to_idx_table` and stored in `TransactionMetadata`:

```rust
transaction_hash_to_idx_table.insert(txn, tx_hash, &transaction_index)?;
transaction_metadata_table.append(txn, &transaction_index,
    &TransactionMetadata { tx_location, tx_output_location, tx_hash: *tx_hash })?;
``` [3](#0-2) 

This applies to **all** transaction types, including `Transaction::L1Handler`.

---

### Impact Explanation

The corrupted storage has the following concrete effects:

1. **`starknet_getTransactionByHash(correct_hash)` → `TRANSACTION_HASH_NOT_FOUND`**: The correct canonical hash is not in the index, so the RPC returns "not found" for a transaction that exists in the block. [4](#0-3) 

2. **`starknet_getTransactionByHash(wrong_hash)` → returns the transaction**: The wrong hash maps to the transaction, so the node serves an authoritative-looking but incorrect response.

3. **`starknet_getTransactionByBlockIdAndIndex` returns wrong hash**: The stored `tx_hash` in `TransactionMetadata` is returned directly. [5](#0-4) 

4. **`starknet_getTransactionReceipt` returns wrong `transaction_hash`**: The receipt is keyed by the wrong hash. [6](#0-5) 

**Clarification on the "L1 handler message anchor" claim:** The `msg_hash` field in L1Handler receipts is computed on-the-fly from the transaction body via `l1_handler_tx.calc_msg_hash()`, not from the stored `transaction_hash`. The sequencer's L1 events transaction manager (`apollo_l1_events`) gets its data from the L1 scraper, not from p2p sync. Therefore, the specific claim that this corrupts the "L1 handler message anchor used by downstream L1 finality checks" is **overstated** — the L1 finality path is not affected. The actual impact is corrupted RPC responses and a corrupted transaction hash index, which is **High** (not Critical).

---

### Likelihood Explanation

Any node that participates in p2p sync and connects to a malicious peer is vulnerable. The p2p sync client is a production code path used by full nodes. The attacker only needs to respond to a `TransactionQuery` with crafted `TransactionsResponse` protobuf messages containing mismatched `transaction_hash` fields. No authentication or privilege is required.

---

### Recommendation

In `parse_data_for_block`, after receiving each `FullTransaction`, recompute the canonical hash from the transaction body and `chain_id` using `validate_transaction_hash` (which already exists in `crates/starknet_api/src/transaction_hash.rs`) and reject the peer with `BadPeerError` if the hash does not match. [7](#0-6) 

The `block_number` needed for `validate_transaction_hash` is already available in `parse_data_for_block`.

---

### Proof of Concept

1. Attacker node responds to a `TransactionQuery` for block N.
2. For each `Transaction::L1Handler(body)` in the block, the attacker sends a `TransactionsResponse` with the correct `body` but sets `transaction_hash` to an arbitrary value (e.g., `Felt::from(0xdeadbeef)`).
3. The victim node's `parse_data_for_block` accepts all N transactions (count matches `n_transactions` from the header).
4. `write_to_storage` commits the corrupted `BlockBody` to storage.
5. `starknet_getTransactionByHash(canonical_hash)` on the victim returns `TRANSACTION_HASH_NOT_FOUND`.
6. `starknet_getTransactionByHash(0xdeadbeef)` returns the L1Handler transaction body.

pub fn validate_transaction_hash(
    transaction: &Transaction,
    block_number: &BlockNumber,
    chain_id: &ChainId,
    expected_hash: TransactionHash,
    transaction_options: &TransactionOptions,
) -> Result<bool, StarknetApiError> {
    let mut possible_hashes = get_deprecated_transaction_hashes(
        chain_id,
        block_number,
        transaction,
        transaction_options,
    )?;
    possible_hashes.push(get_transaction_hash(transaction, chain_id, transaction_options)?);
    Ok(possible_hashes.contains(&expected_hash))
}
