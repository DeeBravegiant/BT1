Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables Persistent DoS of Both Oracle Price Update Paths via `DailyFeeMintLimitExceeded` - (File: contracts/FeeReceiver.sol / contracts/LRTOracle.sol)

## Summary
`FeeReceiver.sendFunds()` has no access control, allowing any caller to flush arbitrary ETH into `LRTDepositPool`. This inflates the protocol's TVL, causing `_updateRsETHPrice()` to compute a `rsethAmountToMintAsProtocolFee` that exceeds `maxFeeMintAmountPerDay`, which triggers an unconditional revert in `_checkAndUpdateDailyFeeMintLimit()`. Both `updateRSETHPrice()` and `updateRSETHPriceAsManager()` share the same `_updateRsETHPrice()` code path with no bypass for the daily limit check, so both are blocked. Because `rsETHPrice` is never updated while the DoS is active, the inflated TVL persists across 24-hour period resets, requiring privileged admin intervention to resolve.

## Finding Description

**Root cause — no access control on `sendFunds()`:**

`FeeReceiver.sendFunds()` carries no role modifier: [1](#0-0) 

`LRTDepositPool.receiveFromRewardReceiver()` is equally unrestricted: [2](#0-1) 

**Injected ETH is immediately reflected in TVL:**

`getETHDistributionData()` reads the raw ETH balance of the deposit pool: [3](#0-2) 

`_getTotalEthInProtocol()` aggregates this via `getTotalAssetDeposits(ETH_TOKEN)`: [4](#0-3) 

**Inflated TVL produces an oversized fee:**

When `totalETHInProtocol > previousTVL`, a protocol fee is computed: [5](#0-4) 

The resulting `rsethAmountToMintAsProtocolFee` is passed to `_checkAndUpdateDailyFeeMintLimit()`: [6](#0-5) 

`_checkAndUpdateDailyFeeMintLimit()` reverts unconditionally when the amount exceeds the cap: [7](#0-6) 

**Both price update paths are blocked:**

`updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` (manager-only) both call the same `_updateRsETHPrice()`: [8](#0-7) 

The manager role check at line 263 only gates the `PriceAboveDailyThreshold` revert path; it does not bypass `_checkAndUpdateDailyFeeMintLimit()`. There is no escape hatch for the manager: [9](#0-8) 

**DoS persists across 24-hour resets:**

`_checkAndUpdateDailyFeeMintLimit()` resets `currentPeriodMintedFeeAmount` to 0 at the start of each new period: [10](#0-9) 

However, because `rsETHPrice` is never updated while the DoS is active, `previousTVL = rsethSupply.mulWad(rsETHPrice)` remains stale. The attacker's ETH remains in the deposit pool, so `totalETHInProtocol` still exceeds `previousTVL` on every subsequent period. The fee computation produces the same oversized amount each period, and the revert repeats indefinitely without further attacker action.

## Impact Explanation

All protocol operations that depend on a current rsETH price — deposits, withdrawals, and `unlockQueue` operations — are blocked for as long as the DoS is active. The DoS is not self-healing at the 24-hour boundary; it requires a privileged admin to increase `maxFeeMintAmountPerDay` (via `setMaxFeeMintAmountPerDay()`) to a value large enough to accommodate the inflated fee. This constitutes **Medium — Temporary freezing of funds**.

## Likelihood Explanation

The attack requires only ETH and a single permissionless call to `sendFunds()`. No tokens, approvals, or special roles are needed. `maxFeeMintAmountPerDay` is a finite value set by the manager, and the feature exists precisely to cap fee minting, making it a natural DoS vector. The attacker's ETH is not lost — it increases TVL and therefore the value of any rsETH the attacker holds, making the net cost of the attack negligible. The attack is repeatable and requires no ongoing attacker participation. Likelihood: **High**.

## Recommendation

1. **Add access control to `sendFunds()`** — restrict it to `MANAGER` or `OPERATOR_ROLE` so only trusted parties can flush rewards into the deposit pool.
2. **Cap rather than revert in `_checkAndUpdateDailyFeeMintLimit()`** — instead of reverting when the fee exceeds the daily limit, mint only up to the remaining limit and carry the excess to the next period. This eliminates the daily limit as a DoS vector.
3. **Provide a manager bypass for the daily limit check** — `updateRSETHPriceAsManager()` should be able to proceed even when the daily limit is exceeded, ensuring the manager can always recover the price oracle.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

// Preconditions:
//   maxFeeMintAmountPerDay = 1 wei
//   rsETHPrice = 1e18, rsethSupply > 0, protocolFeeInBPS > 0
//   feePeriodStartTime set within last 24 h

function testDoS() public {
    // 1. Seed FeeReceiver with 1 ETH (permissionless receive())
    vm.deal(attacker, 1 ether);
    vm.prank(attacker);
    (bool ok,) = address(feeReceiver).call{value: 1 ether}("");
    require(ok);

    // 2. Flush ETH into deposit pool (permissionless sendFunds())
    vm.prank(attacker);
    feeReceiver.sendFunds();
    // deposit pool ETH balance now inflated by 1 ETH
    // totalETHInProtocol > previousTVL => protocolFeeInETH > 0
    // rsethAmountToMintAsProtocolFee >> maxFeeMintAmountPerDay (1 wei)

    // 3. Public price update reverts
    vm.expectRevert(abi.encodeWithSelector(ILRTOracle.DailyFeeMintLimitExceeded.selector));
    lrtOracle.updateRSETHPrice();

    // 4. Manager price update also reverts (no bypass for daily limit check)
    vm.prank(manager);
    vm.expectRevert(abi.encodeWithSelector(ILRTOracle.DailyFeeMintLimitExceeded.selector));
    lrtOracle.updateRSETHPriceAsManager();

    // 5. After 24 hours, DoS persists (rsETHPrice was never updated)
    vm.warp(block.timestamp + 1 days + 1);
    vm.expectRevert(abi.encodeWithSelector(ILRTOracle.DailyFeeMintLimitExceeded.selector));
    lrtOracle.updateRSETHPrice();
}
```
