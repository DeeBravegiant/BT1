Audit Report

## Title
Missing Chainlink Price Staleness and Validity Checks in `ChainlinkPriceOracle.getAssetPrice()` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all metadata, performing no staleness check on `updatedAt`, no round-completeness check on `answeredInRound`, and no validity check on `price`. The same protocol applies all three guards in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming awareness of the requirement. The unguarded oracle feeds directly into rsETH minting and withdrawal calculations reachable by any unprivileged user.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at [1](#0-0)  destructures `latestRoundData()` discarding all fields except `price`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The five return values are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. `updatedAt` and `answeredInRound` are silently discarded, meaning:

1. **No staleness check**: `updatedAt` is never compared against `block.timestamp`. A feed that has not been updated for hours or days returns its last stored price without reverting.
2. **No round completeness check**: `answeredInRound < roundId` (indicating an in-progress round) is never tested.
3. **No price validity check**: `price <= 0` is never tested. A zero `int256` cast to `uint256` produces zero; a negative `int256` cast to `uint256` produces a value near `type(uint256).max`, causing overflow and revert in Solidity 0.8.x (DoS).

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` enforces all three guards: [2](#0-1) 

The unguarded price flows through `LRTOracle.getAssetPrice()` [3](#0-2)  into `LRTDepositPool.getRsETHAmountToMint()`: [4](#0-3)  and `LRTWithdrawalManager.getExpectedAssetAmount()`: [5](#0-4) 

The `_getTotalEthInProtocol()` function used to compute `rsETHPrice` also calls `getAssetPrice()` for every supported asset, meaning a stale price also corrupts the rsETH price denominator. [6](#0-5) 

## Impact Explanation
**High — Theft of unclaimed yield.** If a Chainlink feed for a supported LST (stETH, ETHx, rETH, etc.) stops updating and the stale price is lower than the true current price, depositors receive more rsETH than they are entitled to, diluting the yield accrued by existing rsETH holders. Conversely, if the stale price is higher than the true price, withdrawal requesters receive more underlying asset than they should, draining protocol reserves. Both paths constitute theft of unclaimed yield from existing rsETH holders or direct protocol asset loss. The zero-price path causes `rsethAmountToMint = 0`, which triggers a revert at the `minRSETHAmountExpected` check for any non-zero slippage parameter — a temporary freeze of deposits (Medium). The SECURITY.md exclusion for "incorrect data supplied by third-party oracles" does not apply here: the bug is in the protocol's own validation code, not in the oracle's data, as evidenced by the protocol already implementing the same checks in `ChainlinkOracleForRSETHPoolCollateral`. [7](#0-6) 

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (1 hour for ETH/USD, up to 24 hours for some LST feeds). During Ethereum network congestion, oracle keeper failures, or L2 sequencer issues, feeds can lag well beyond their heartbeat — a historically observed condition (e.g., March 2023 USDC depeg). No attacker action beyond submitting a standard deposit or withdrawal transaction during the stale window is required. The exposure is passive and activates automatically whenever any supported LST feed drifts past its heartbeat. [8](#0-7) 

## Recommendation
Apply the same three guards already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`, and add a configurable `MAX_PRICE_AGE` staleness bound per feed:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_PRICE_AGE) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_PRICE_AGE` should be set per-feed based on the Chainlink heartbeat for each asset (e.g., 3600 seconds for 1-hour heartbeat feeds, with a reasonable buffer). [9](#0-8) 

## Proof of Concept
1. A supported LST Chainlink feed (e.g., stETH/ETH) stops updating. Its `updatedAt` is now 25 hours old.
2. The true stETH/ETH rate has moved from 0.9990 to 1.0010 (a 0.2% increase).
3. An attacker calls `LRTDepositPool.depositAsset(stETH, 1_000e18, 0, "")`.
4. `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns the stale 0.9990 price without reverting.
5. `rsethAmountToMint = (1_000e18 * 0.9990e18) / rsETHPrice` — the attacker receives rsETH priced against the stale (lower) rate.
6. When `updateRSETHPrice()` is next called with the correct price, the rsETH price adjusts upward, and the attacker's position is worth more than they paid, at the expense of existing holders.
7. No privileged action is required; the attacker only needs to observe that the feed is stale and submit a standard deposit.

**Foundry fork test plan**: Fork mainnet, use `vm.mockCall` to make the stETH/ETH Chainlink feed return a stale `updatedAt` (e.g., `block.timestamp - 26 hours`) with a price 0.5% below current. Call `depositAsset`, then call `updateRSETHPrice`, and assert that the attacker's rsETH balance is worth more ETH than deposited, while existing holder NAV decreased. [1](#0-0)
