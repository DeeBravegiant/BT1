### Title
Unvalidated `ProposalInit.builder` Allows Malicious Proposer to Redirect All Block Transaction Fees to Arbitrary Address — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`ProposalInit.builder` is the field that becomes `sequencer_address` in the executed block's `BlockInfo`. Transaction fees for every transaction in a block are charged to `sequencer_address`. The validator's `is_proposal_init_valid` function checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, L1 gas prices, and `fee_proposal_fri` — but never checks `builder`. A malicious proposer can therefore set `builder` to any attacker-controlled address, and every validator will accept the proposal, execute the block with the wrong fee recipient, and reach consensus on a block where all transaction fees flow to the attacker.

---

### Finding Description

`convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address`:

```rust
// crates/apollo_consensus_orchestrator/src/utils.rs:332
sequencer_address: init.builder,
```

The OS Cairo code charges fees to `block_context.block_info_for_execute.sequencer_address`:

```cairo
// crates/apollo_starknet_os_program/.../transaction_impls.cairo:130
local calldata: TransferCallData = TransferCallData(
    recipient=block_context.block_info_for_execute.sequencer_address,
    amount=Uint256(low=low_actual_fee, high=0),
);
```

The proposer constructs `ProposalInit` with `builder: args.builder_address` (a local config value). A malicious proposer substitutes any address:

```rust
// crates/apollo_consensus_orchestrator/src/build_proposal.rs:174
builder: args.builder_address,  // proposer-controlled, no constraint
```

`is_proposal_init_valid` validates many fields but has no check for `builder`:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:312-321
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
{
    return Err(...);
}
// ... L1 gas price checks, fee_proposal checks — no builder check anywhere
```

`ProposalInitValidation` does not even carry an expected `builder` value:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:75-85
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub block_timestamp_window_seconds: u64,
    pub previous_proposal_init: Option<PreviousProposalInitInfo>,
    pub l1_da_mode: L1DataAvailabilityMode,
    pub l2_gas_price_fri: GasPrice,
    pub starknet_version: StarknetVersion,
    pub fee_actual: Option<GasPrice>,
    // no builder / sequencer_address field
}
```

Because both the proposer and every validator call `convert_to_sn_api_block_info(init)` with the same (attacker-supplied) `init.builder`, the batcher on both sides executes the block with the same malicious `sequencer_address`. The resulting `partial_block_hash` is identical on both sides, so the `ProposalFinMismatch` guard at line 244 does not fire. Consensus reaches agreement on a block whose fee recipient is the attacker.

---

### Impact Explanation

Every transaction fee in the block is transferred to `sequencer_address` (= `init.builder`). A malicious proposer sets this to an attacker wallet. All honest validators accept and finalize the block. The legitimate sequencer receives zero fees for that block; the attacker receives all of them. This is a direct, quantifiable economic loss matching the "Incorrect fee … with economic impact" criterion.

---

### Likelihood Explanation

Any validator that wins the proposer slot — which happens in normal rotation — can trigger this. No special privilege beyond being selected as proposer is required. The attack is silent: the block is valid in every other respect, passes all existing checks, and reaches finality.

---

### Recommendation

Add `builder` (expected sequencer address) to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`. Until the committee-derived builder address is available (see the `TODO(Asmaa)` comment), the validator should at minimum reject any `init.builder` that does not match the locally configured `builder_address`, or derive the expected value from the committee roster the same way the proposer does.

---

### Proof of Concept

1. Malicious node wins proposer slot for height H.
2. In `initiate_build`, it sets `builder: attacker_address` instead of `self.config.static_config.builder_address`.
3. It broadcasts `ProposalPart::Init(init)` with `init.builder = attacker_address`.
4. Every validator calls `is_proposal_init_valid` — passes (no `builder` check).
5. Every validator calls `initiate_validation` → `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_address`.
6. Batcher executes all transactions; OS Cairo charges each fee to `attacker_address`.
7. Batcher returns `partial_block_hash` computed with `attacker_address` as sequencer.
8. Proposer's `partial_block_hash` matches (same `builder` used on both sides).
9. `ProposalFinMismatch` check passes; `validate_proposal` returns `Ok(built_block)`.
10. Consensus commits the block; all fees for height H are in the attacker's account.

**Corrupted value:** `BlockInfo.sequencer_address` for the committed block equals `attacker_address` instead of the legitimate sequencer address. All fee transfers for that block go to the wrong recipient. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
