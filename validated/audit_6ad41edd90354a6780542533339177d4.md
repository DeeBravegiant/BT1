Audit Report

## Title
Zero Fee Applied to ERC-20 Token Deposits Due to Uninitialized `tokenFeeBps` Mapping - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps` that Solidity initializes to `0` for any key never explicitly written. `addSupportedToken` does not set `tokenFeeBps[token]`, and there is no guard preventing deposits when the per-token fee is zero. Any depositor can call `deposit(token, amount, referralId)` immediately after a token is added and receive the full rsETH equivalent with no fee deducted, permanently denying the protocol treasury fee revenue for that window.

## Finding Description
`RSETHPool.sol` declares two fee variables:

- `feeBps` (L43) — global basis-point fee for native ETH deposits.
- `tokenFeeBps` (L88) — per-token mapping for ERC-20 deposits; Solidity default is `0` for any unset key.

`addSupportedToken` (L637–656) registers a token by writing `supportedTokenOracle[token]` and `tokenBridge[token]`, but never touches `tokenFeeBps[token]`. A separate admin call to `setTokenFeeBps` (L583–594) is required to configure the fee.

The token deposit path at L335–336 reads directly from the uninitialized mapping:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];   // 0 if never set
fee = amount * feeBpsForToken / 10_000;        // fee == 0
```

`deposit(token, amount, referralId)` (L284–305) calls `viewSwapRsETHAmountAndFee(amount, token)`, accumulates `fee` into `feeEarnedInToken[token]`, and transfers `rsETHAmount` to the caller. When `fee == 0`, the full token value is converted to rsETH with no protocol cut. There is no on-chain check that `tokenFeeBps[token] > 0` before accepting deposits. The ETH deposit path (L311–313) has no equivalent gap because `feeBps` is set at initialization.

## Impact Explanation
Protocol fee revenue on ERC-20 token deposits is permanently lost for every deposit made while `tokenFeeBps[token] == 0`. The fees that should have been collected can never be retroactively recovered. This constitutes **theft of unclaimed yield** (High): yield owed to the protocol treasury is silently zeroed out by a structural gap in the contract, exploitable by any unprivileged depositor.

## Likelihood Explanation
Every time `addSupportedToken` is called, `tokenFeeBps[token]` starts at `0`. The `addSupportedToken` and `setTokenFeeBps` calls are separate transactions, creating a guaranteed window — even if only one block wide — during which zero-fee deposits are possible. A sophisticated user monitoring the mempool for `AddSupportedToken` events can front-run or immediately follow the admin transaction with a large deposit. The pool already supports wstETH on Arbitrum, confirming the multi-token path is live. The condition is trivially reachable with no special privileges.

## Recommendation
Initialize `tokenFeeBps[token]` inside `addSupportedToken`, requiring the caller to supply the intended fee in basis points as an additional parameter. Alternatively, add a fallback in `viewSwapRsETHAmountAndFee(amount, token)` that uses the global `feeBps` when `tokenFeeBps[token] == 0`, mirroring the ETH deposit path. A stricter option is to revert in `viewSwapRsETHAmountAndFee(amount, token)` if `tokenFeeBps[token]` has never been explicitly set (using a separate `bool` or sentinel value to distinguish "set to zero" from "never set").

## Proof of Concept
1. Admin calls `addSupportedToken(tokenX, oracleX, bridgeX)` — `tokenFeeBps[tokenX]` remains `0` (L637–656, no write to `tokenFeeBps`).
2. Before admin calls `setTokenFeeBps(tokenX, N)`, attacker calls `deposit(tokenX, 1_000e18, "ref")`.
3. `viewSwapRsETHAmountAndFee(1_000e18, tokenX)` executes L335–336: `feeBpsForToken = 0`, `fee = 0`.
4. `feeEarnedInToken[tokenX] += 0` — treasury accrues nothing.
5. Attacker receives rsETH equivalent to the full `1_000e18` token value.
6. Contrast: an ETH depositor sending equivalent value pays `feeBps` (e.g., 5 bps) via L312.

Foundry test plan: deploy `RSETHPool` with a mock oracle and mock token; call `addSupportedToken`; immediately call `deposit(token, 1e18, "")` without calling `setTokenFeeBps`; assert `feeEarnedInToken[token] == 0` and that the depositor received the full rsETH amount.
