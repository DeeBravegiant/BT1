Audit Report

## Title
No Staleness Check on Chainlink `latestRoundData()` Allows Stale Price to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt`, `answeredInRound`, and does not validate that `price > 0`. A stale or zero price propagates into `LRTOracle._getTotalEthInProtocol()`, corrupting the on-chain `rsETHPrice` used for all deposits and withdrawals. The public, permissionless `updateRSETHPrice()` function allows any attacker to opportunistically lock in a favorable stale price.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` silently drops every return value except the raw `price`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The sibling contract `ChainlinkOracleForRSETHPoolCollateral` in the same repository applies all three guards that are absent here:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L27-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0)            revert IncompleteRound();
if (ethPrice <= 0)             revert InvalidPrice();
``` [2](#0-1) 

The stale price flows into `_getTotalEthInProtocol()`, which sums `assetPrice × assetBalance` for every supported LST: [3](#0-2) 

The corrupted total is then used to compute and persist `rsETHPrice`: [4](#0-3) 

`updateRSETHPrice()` is a public function with no access control beyond `whenNotPaused`, allowing any external caller to trigger the price update at will: [5](#0-4) 

The `pricePercentageLimit` guard at L252–266 is an insufficient mitigation: it only fires when `newRsETHPrice > highestRsethPrice` by more than the configured band, and it is entirely bypassed when `pricePercentageLimit == 0` (the default after initialization). Stale prices that fall within the band — which is the normal operating range during a Chainlink heartbeat window — pass through unchecked. [6](#0-5) 

## Impact Explanation

**Scenario A — Stale price is lower than actual (deflated):** `rsETHPrice` is understated. An attacker calls `updateRSETHPrice()` to lock in the deflated price, deposits LST collateral, and receives more rsETH shares than the true value warrants. When the price corrects, the attacker redeems at the true rate, extracting value from existing rsETH holders. This maps to **High: Theft of unclaimed yield**.

**Scenario B — Stale price is higher than actual (inflated):** `rsETHPrice` is overstated. An attacker calls `updateRSETHPrice()` to lock in the inflated price, then initiates a withdrawal priced at the inflated rate. When the price corrects, the attacker's claim is settled at the inflated rate, extracting excess ETH from the pool. This maps to **Critical: Direct theft of user funds**.

## Likelihood Explanation

Chainlink feeds operate on a heartbeat (up to 24 hours for some LST feeds) plus a deviation threshold (typically 0.5–1%). During the heartbeat window, the on-chain price can diverge from the true market price by the full deviation band without triggering an update. This is a known, recurring condition. Because `updateRSETHPrice()` is publicly callable with no access control, an attacker can monitor off-chain prices and call the function at the exact moment the stale price is most favorable, requiring no special privileges and no victim mistakes.

## Recommendation

1. **Short term:** Add staleness and validity guards to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:
   ```solidity
   (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
       priceFeed.latestRoundData();
   require(answeredInRound >= roundId, "Stale price");
   require(updatedAt != 0, "Incomplete round");
   require(price > 0, "Invalid price");
   require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
   ```
2. **Long term:** Restrict `updateRSETHPrice()` to trusted callers (e.g., a keeper with `MANAGER` role) or add a circuit-breaker that refuses to update the stored price if any individual feed is stale, preventing opportunistic timing attacks.

## Proof of Concept

1. Chainlink's stETH/ETH feed has a 24-hour heartbeat. At hour 23, the true stETH price has dropped 1% but the on-chain feed still reports the old (higher) price — within the `pricePercentageLimit` band, so no revert occurs.
2. Attacker observes the discrepancy off-chain.
3. Attacker calls `LRTOracle.updateRSETHPrice()` (no access control). `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the stale (inflated) price. `rsETHPrice` is set above its true value.
4. Attacker immediately initiates a withdrawal via `LRTWithdrawalManager`, receiving a claim priced at the inflated `rsETHPrice`.
5. When Chainlink updates at hour 24, the true price is reflected. The attacker's withdrawal claim was locked in at the inflated rate, extracting excess ETH from the pool at the expense of remaining rsETH holders.

The entire attack path is permissionless: steps 3–4 require only standard ERC-20 and deposit-pool interactions available to any unprivileged user.
