Audit Report

## Title
Push-Pattern ETH Transfer in `_processWithdrawalCompletion` Permanently Freezes Funds for Non-ETH-Receiving Contracts - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._transferAsset` uses a push pattern to deliver ETH to withdrawing users. If the recipient is a contract that cannot accept ETH, the transfer reverts and the withdrawal becomes permanently uncompletable. Because rsETH is burned and ETH is pulled from the unstaking vault in a prior `unlockQueue` transaction, the ETH equivalent is irreversibly stranded in `LRTWithdrawalManager` with no admin recovery path.

## Finding Description
`_processWithdrawalCompletion` is the single code path for both `completeWithdrawal` (user-initiated) and `completeWithdrawalForUser` (operator-initiated). It terminates with a push transfer:

```solidity
// L876-L883
function _transferAsset(address asset, address to, uint256 amount) internal {
    if (asset == LRTConstants.ETH_TOKEN) {
        (bool sent,) = payable(to).call{ value: amount }("");
        if (!sent) revert EthTransferFailed();
    } else {
        IERC20(asset).safeTransfer(to, amount);
    }
}
``` [1](#0-0) 

The call at L734 inside `_processWithdrawalCompletion` pushes to `user`: [2](#0-1) 

Because all state mutations in `_processWithdrawalCompletion` (nonce pop at L705, request deletion at L712, `unlockedWithdrawalsCount` decrement at L717) occur in the same transaction as the push, a revert on the ETH transfer rolls back all of them. The withdrawal request is restored to the queue, and every subsequent attempt produces the same revert. [3](#0-2) 

The irreversible damage occurs in `unlockQueue`, which is a separate, already-committed transaction:
- rsETH held by the contract is burned at L305.
- ETH equivalent is pulled from the unstaking vault at L307. [4](#0-3) 

After `unlockQueue` commits, the rsETH is gone and the ETH sits in `LRTWithdrawalManager`. If the user address cannot receive ETH, the ETH is permanently frozen.

**No admin recovery path exists:**
- `sweepRemainingAssets` requires `!hasUnlockedWithdrawals(asset)` (L403), which is false while the stuck request holds `unlockedWithdrawalsCount[asset] > 0`.
- `completeWithdrawalForUser` still pushes to the same `user` address; its own NatSpec acknowledges it is "Not expected to be used for ETH."
- There is no function to redirect an ETH withdrawal to an alternative address or cancel a completed-unlock request. [5](#0-4) [6](#0-5) 

## Impact Explanation
**Critical — Permanent freezing of funds.**

After `unlockQueue` executes, the rsETH is irreversibly burned and the ETH equivalent is held in `LRTWithdrawalManager`. Because the only delivery mechanism is the push call that always reverts for a non-ETH-receiving contract, and because no admin function can redirect or recover the ETH, the funds are permanently frozen. This matches the allowed Critical impact: *Permanent freezing of funds*.

## Likelihood Explanation
**Medium.** No attacker action is required; the user simply needs to be a smart contract without a `receive()` function or with a reverting fallback. This is realistic for custom yield aggregators, vaults, or any contract that holds rsETH and initiates an ETH withdrawal. The user initiates the withdrawal themselves via the public `initiateWithdrawal` function; the freeze is triggered by the normal protocol lifecycle once `unlockQueue` is called by an operator.

## Recommendation
Replace the push pattern with a pull pattern for ETH withdrawals:

1. In `_processWithdrawalCompletion`, instead of calling `_transferAsset(asset, user, ...)` for ETH, record the claimable amount in a mapping: `pendingETHWithdrawals[user] += amount`.
2. Expose a separate `claimETH(address recipient)` function that allows the user (or any caller on their behalf) to withdraw to an arbitrary recipient address they control.

This eliminates the dependency on the recipient's ability to accept a pushed ETH transfer and ensures no ETH can be permanently stranded.

## Proof of Concept
Minimal call sequence demonstrating the freeze:

1. Deploy a contract `NoReceiveETH` (no `receive()` function) that holds rsETH.
2. `NoReceiveETH` calls `initiateWithdrawal(ETH_TOKEN, amount, "")` — rsETH is transferred to `LRTWithdrawalManager`.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH is burned (L305), ETH is pulled from the unstaking vault into `LRTWithdrawalManager` (L307). This transaction commits; rsETH is gone.
4. `NoReceiveETH` calls `completeWithdrawal(ETH_TOKEN, "")` — `_transferAsset` attempts `payable(NoReceiveETH).call{value: amount}("")`, which returns `false` because `NoReceiveETH` has no `receive()`. `EthTransferFailed` is thrown; the transaction reverts.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, address(NoReceiveETH), "")` — same revert.
6. Operator attempts `sweepRemainingAssets(ETH_TOKEN)` — reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.

Result: rsETH is burned, ETH is permanently locked in `LRTWithdrawalManager`, and no recovery function exists. [1](#0-0) [7](#0-6)
