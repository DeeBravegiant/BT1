Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` implements a Synthetix-style staking rewards model where `rewardPerToken()` short-circuits to return the stored value when `totalKernelStaked == 0`. If all stakers call `initiateWithdrawal` during an active reward period, the remaining undistributed reward tokens ‚Äî computed as `rewardRate √ó (finishAt ‚àí updatedAt)` ‚Äî become permanently locked in the contract with no on-chain recovery path. The contract's own NatSpec acknowledges this risk but relies entirely on off-chain coordination that is not enforced by any contract guard.

## Finding Description
`rewardPerToken()` at L408‚Äì414 returns `rewardPerTokenStored` unchanged whenever `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    ...
}
```

`initiateWithdrawal` at L320‚Äì338 decrements `totalKernelStaked` immediately (before the withdrawal delay elapses), with no check preventing it from reaching zero during an active reward window:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
```

The `updateReward` modifier at L232‚Äì241 snapshots `rewardPerTokenStored` and `updatedAt` at the moment of the last withdrawal. After that call, `totalKernelStaked == 0`, and all reward tokens that would have accrued from `updatedAt` to `finishAt` are stranded ‚Äî `rewardPerToken()` will never advance past the snapshot, so no staker can ever claim them.

The only existing guard is in `notifyRewardAmount` at L569‚Äì570:
```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
```
This prevents *starting* a reward period with zero stakers but does nothing to prevent all stakers from exiting *during* an active period. No admin function (`setRewardsDuration`, `setWithdrawalDelay`, `setMaxNumberOfWithdrawalsPerUser`) provides a token recovery path. The contract explicitly acknowledges this at L18‚Äì22 but defers entirely to off-chain coordination.

## Impact Explanation
Reward tokens (ERC-20 `rewardsToken`) are permanently frozen inside `KernelDepositPool` with no on-chain path to recovery. This is **permanent freezing of unclaimed yield**, a Medium-severity impact in the allowed scope. The magnitude is `rewardRate √ó (finishAt ‚àí block.timestamp)` tokens at the time the last staker exits.

## Likelihood Explanation
The scenario requires all stakers to exit during an active reward period. This is realistic when: (a) the staker set is small (early deployment, single large staker); (b) market conditions incentivize mass unstaking. The trigger is a permissionless call to `initiateWithdrawal` ‚Äî no privileged access is required. The lock-in occurs the moment the last staker initiates withdrawal, not when they claim, because `totalKernelStaked` is decremented immediately at L326.

## Recommendation
Add a permissioned `recoverLockedRewards()` function that transfers stranded reward tokens to the treasury when `totalKernelStaked == 0` and the reward period has ended (or is active with zero stakers). Alternatively, track elapsed zero-staking time and exclude it from the reward window so `rewardRate` is effectively paused, resuming when staking returns. A minimal fix is to add a check in `initiateWithdrawal` that reverts (or pauses rewards) if the withdrawal would bring `totalKernelStaked` to zero during an active reward period.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` while Alice is the sole staker with 100 KERNEL. `rewardRate = 1_000e18 / duration`, `finishAt = block.timestamp + duration`.
2. After half the period, Alice calls `initiateWithdrawal(100)`.
   - `updateReward(alice)` fires: `rewardPerTokenStored` and `updatedAt` are snapshotted at the current timestamp.
   - `totalKernelStaked` is decremented to 0 at L326.
3. For the remaining half-period, `rewardPerToken()` always returns `rewardPerTokenStored` (unchanged), so `rewardRate √ó (finishAt ‚àí updatedAt) ‚âà 500e18` reward tokens accumulate in the contract but are never attributed to any staker.
4. No function in `KernelDepositPool` can extract these tokens. They are permanently locked.

**Foundry test sketch:**
```solidity
function test_rewardsLockedOnZeroStake() public {
    // Setup: Alice stakes, admin notifies rewards
    vm.prank(alice); pool.stake(100e18);
    vm.prank(admin); pool.notifyRewardAmount(1_000e18);
    uint256 rewardRate = pool.rewardRate();
    uint256 finishAt = pool.finishAt();

    // Advance half the period
    vm.warp(block.timestamp + pool.duration() / 2);

    // Alice withdraws all
    vm.prank(alice); pool.initiateWithdrawal(100e18);
    assertEq(pool.totalKernelStaked(), 0);

    // Advance to end of period
    vm.warp(finishAt + 1);

    // Stranded rewards = rewardRate * (finishAt - updatedAt)
    uint256 stranded = rewardRate * (finishAt - pool.updatedAt());
    assertGt(stranded, 0);

    // No function can recover them
    assertEq(pool.rewardsToken().balanceOf(address(pool)), stranded + alice_earned);
}
```
