### Title
Dijkstra Sub-Transaction Double-Claim of Deposit Refund Allows Direct ADA Creation - (`eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` / `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may contain multiple fee-free sub-transactions. The ledger currently lacks a predicate failure preventing two sub-transactions within the same top-level transaction from each deregistering the **same** staking credential and each independently claiming the same deposit refund. This allows an attacker to extract the same deposit twice from `utxosDeposited`, creating ADA from nothing and violating the preservation-of-value invariant.

---

### Finding Description

**Vulnerability class:** Funds/accounting bug — fee-free operations that manipulate a shared accounting pot, causing direct ADA creation.

**Root cause — `updateUTxOStateNoFees`:**

Sub-transactions are processed by `updateUTxOStateNoFees` (the fee-free variant of `updateUTxOState`), defined in `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/Utxo.hs`:

```haskell
-- | Like 'updateUTxOState', but does not collect fees. This is used for sub-transactions
-- where fees are not applicable.
updateUTxOStateNoFees :: ... => ... -> m (UTxOState era)
updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  ...
  totalRefunds = certsTotalRefundsTxBody pp certState txBody
  totalDeposits = certsTotalDepositsTxBody pp certState txBody
  depositChange = totalDeposits <-> totalRefunds
  ...
  pure $! UTxOState { utxosDeposited = utxosDeposited <> depositChange, ... }
``` [1](#0-0) 

The refund amount is computed from `certState` — the snapshot of the delegation state at the time the sub-transaction is evaluated. If two sub-transactions both contain `UnRegDepositTxCert stakingCred keyDeposit` for the **same** credential, and the `certState` passed to each is not updated to reflect the first sub-transaction's deregistration, both sub-transactions independently compute a non-zero refund for the same credential, and `utxosDeposited` is decremented twice.

**Acknowledged missing protection:**

The Dijkstra test suite explicitly acknowledges this gap with a disabled (`xit`) test and a `TODO` comment:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  stakingCred <- KeyHashObj <$> freshKeyHash
  _ <- registerStakeCredential stakingCred
  keyDeposit <- getsPParams ppKeyDepositL
  ...
  let
    subTx1 = mkBasicTx mkBasicTxBody
      & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = mkBasicTx mkBasicTxBody
      & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = mkBasicTx mkBasicTxBody
      & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [2](#0-1) 

The `xit` prefix marks the test as pending/disabled, and `error "TODO: predicate failure not yet implemented"` confirms the guard against this scenario does not yet exist in the production rules.

**Analog to external report:**

| External Report (Flayer) | Cardano Ledger (Dijkstra) |
|---|---|
| `deposit`/`redeem` are fee-free | Sub-transactions are fee-free (`updateUTxOStateNoFees`) |

| Affects utilization rate → interest rate | Directly creates ADA by double-claiming refund |

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

If two sub-transactions both successfully claim the same deposit refund:
- `utxosDeposited` is decremented by `2 × keyDeposit`
- Only `1 × keyDeposit` was ever deposited
- The attacker's UTxO outputs receive `2 × keyDeposit` in value
- The preservation-of-value invariant (`Circulation + Deposits + Fees + Rewards = constant`) is broken

The formal property from `eras/shelley/formal-spec/Properties.md` states:

> **Property** The value (Circulation + Deposits + Fees + Rewards) is the same before and after the transition. [3](#0-2) 

This property is violated: `Deposits` decreases by `2 × keyDeposit` while `Circulation` increases by `2 × keyDeposit`, but only `1 × keyDeposit` was legitimately deposited.

---

### Likelihood Explanation

- **Attacker-controlled entry path**: Any unprivileged user can craft a Dijkstra-era top-level transaction containing two sub-transactions with identical `UnRegDepositTxCert` certificates for the same credential.
- **No special privileges required**: Only requires owning a registered staking credential and the signing key for it.
- **Explicitly unprotected**: The developers themselves have documented the missing predicate failure with a `TODO` comment in the test suite.
- **Dijkstra is a production-bound era** in the repository's era progression (Byron → … → Conway → Dijkstra).

---

### Recommendation

1. **Implement the missing predicate failure** in the Dijkstra sub-ledger/sub-certs rules: before processing each sub-transaction's certificates, verify that the staking credential being deregistered has not already been deregistered by a prior sub-transaction within the same top-level transaction. Maintain a running set of credentials deregistered by earlier sub-transactions and reject any subsequent sub-transaction that attempts to deregister the same credential.

2. **Thread the updated `certState`** (post-deregistration) from each sub-transaction into the evaluation of the next sub-transaction, so that the second sub-transaction sees the credential as already unregistered and fails with `StakeKeyNotRegisteredDELEG`.

3. **Enable the `xit` test** once the predicate failure is implemented, to prevent regression.

---

### Proof of Concept

1. Register staking credential `C` with deposit `D` (paying `D` ADA into `utxosDeposited`).
2. Craft a top-level Dijkstra transaction `TX` containing two sub-transactions:
   - `subTx1`: contains `UnRegDepositTxCert C D` + UTxO input `I1`
   - `subTx2`: contains `UnRegDepositTxCert C D` + UTxO input `I2`
3. Submit `TX`.
4. Without the missing predicate failure, both sub-transactions independently compute `totalRefunds = D` from the same `certState` (where `C` is still registered), and `utxosDeposited` is decremented by `2D`.
5. The attacker's outputs receive `2D` ADA while only `D` was deposited — `D` ADA is created from nothing. [2](#0-1) [1](#0-0)

-- | Like 'updateUTxOState', but does not collect fees. This is used for sub-transactions
-- where fees are not applicable.
updateUTxOStateNoFees ::
  ( EraTxBody era
  , EraStake era
  , EraCertState era
  , Monad m
  ) =>
  PParams era ->
  UTxOState era ->
  TxBody l era ->
  CertState era ->
  GovState era ->
  (Coin -> m ()) ->
  (UTxO era -> UTxO era -> m ()) ->
  m (UTxOState era)
updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let UTxOState {utxosUtxo, utxosDeposited, utxosFees, utxosDonation} = utxos
      UTxO utxo = utxosUtxo
      !utxoAdd = txouts txBody -- These will be inserted into the UTxO
      {- utxoDel  = txins txb ◁ utxo -}
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      {- newUTxO = (txins txb ⋪ utxo) ∪ outs txb -}
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      deletedUTxO = UTxO utxoDel
      totalRefunds = certsTotalRefundsTxBody pp certState txBody
      totalDeposits = certsTotalDepositsTxBody pp certState txBody
      depositChange = totalDeposits <-> totalRefunds
  depositChangeEvent depositChange
  txUtxODiffEvent deletedUTxO utxoAdd
  pure $!
    UTxOState
      { utxosUtxo = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees = utxosFees
      , utxosGovState = govState
      , utxosInstantStake =
          deleteInstantStake deletedUTxO (addInstantStake utxoAdd (utxos ^. instantStakeL))
      , utxosDonation = utxosDonation
      }
