Audit Report

## Title
Reward Sniping via Stake-Before-Distribution / Unstake-After-Claim Bypasses Long-Term Staker Yield - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` implements a Synthetix-style continuous reward distribution with no minimum staking period and no lock on reward claiming. An attacker can stake a large amount immediately before `notifyRewardAmount` is called, accrue a disproportionate share of rewards, claim them via `getReward()`, and initiate withdrawal — all before the `withdrawalDelay` expires. The `withdrawalDelay` only delays return of principal KERNEL tokens; it does not delay reward collection. Long-term stakers are permanently diluted of yield they were owed.

## Finding Description
Three confirmed code properties combine to enable the attack:

**1. No minimum staking period.** `stake()` immediately credits `balanceOf[msg.sender]` and `totalKernelStaked`, making the staker eligible for rewards from the very next second with no warm-up: [1](#0-0) 

**2. `initiateWithdrawal()` snapshots rewards before decrementing stake.** The `updateReward(msg.sender)` modifier executes first, writing the full accrued amount into `rewards[msg.sender]`, and only then are `balanceOf` and `totalKernelStaked` decremented: [2](#0-1) 

**3. `getReward()` is unconditionally callable.** There is no staking-duration requirement; it transfers whatever `rewards[msg.sender]` holds, independently of any pending withdrawal: [3](#0-2) 

The `withdrawalDelay` enforced in `claimWithdrawal()` only gates the return of principal KERNEL tokens: [4](#0-3) 

It has no effect on `getReward()`, which can be called immediately after `initiateWithdrawal()` locks in the reward snapshot.

The `updateReward` modifier used by both `stake()` and `initiateWithdrawal()` correctly checkpoints `rewardPerTokenStored` and `userRewardPerTokenPaid`, but provides no protection against transient staking: [5](#0-4) 

**Exploit path (does not require front-running; staking before any reward period suffices):**
1. Attacker calls `stake(9_000 KERNEL)` before or concurrent with `notifyRewardAmount`.
2. Admin calls `notifyRewardAmount(100_000 rewardTokens)` — attacker now owns 90% of `totalKernelStaked`.
3. Attacker waits T seconds, accruing `≈ rewardRate * T * 0.90` reward tokens.
4. Attacker calls `initiateWithdrawal(9_000)` — `updateReward` snapshots full accrued rewards into `rewards[attacker]`; stake is decremented.
5. Attacker immediately calls `getReward()` — receives ~90% of distributed rewards.
6. After `withdrawalDelay`, attacker calls `claimWithdrawal()` — receives 9,000 KERNEL principal back.

Original 1,000-KERNEL stakers received ~10% of rewards instead of 100%. The extracted rewards are permanently gone.

## Impact Explanation
**High — Theft of unclaimed yield.** This maps exactly to the allowed impact class "Theft of unclaimed yield." The attacker extracts reward tokens that were economically owed to committed long-term stakers. The rewards already transferred to the attacker via `rewardsToken.safeTransfer` are irrecoverable: [6](#0-5) 

## Likelihood Explanation
**Medium.** The attack requires capital sufficient to represent a meaningful fraction of `totalKernelStaked` and the reward amount must exceed the opportunity cost of locking KERNEL for `withdrawalDelay` (up to `MAX_WITHDRAWAL_DELAY = 30 days`): [7](#0-6) 

The attack is permissionless, requires no privileged access, and is repeatable on every `notifyRewardAmount` call. For large reward distributions relative to the existing staked pool, it is highly profitable even with a multi-day delay. Mempool monitoring (optional, for optimal timing) is standard MEV tooling.

## Recommendation
1. **Minimum staking period before reward eligibility.** Track each user's `stakeTimestamp` and revert `getReward()` if `block.timestamp - stakeTimestamp < withdrawalDelay`.
2. **Reward vesting / time-weighted accrual.** Only count rewards earned after a configurable warm-up window following a stake event, preventing same-period entry/exit exploitation.
3. **Tie reward claiming to withdrawal delay.** Prevent `getReward()` from being called while a withdrawal is pending, or require rewards to vest over the same `withdrawalDelay` period as principal.

## Proof of Concept
Foundry test plan (local fork or unit test):

```solidity
function test_rewardSniping() public {
    // Setup: existing stakers hold 1_000 KERNEL
    vm.prank(existingStaker);
    pool.stake(1_000e18);

    // Attacker stakes 9_000 KERNEL before notifyRewardAmount
    vm.prank(attacker);
    kernelToken.approve(address(pool), 9_000e18);
    vm.prank(attacker);
    pool.stake(9_000e18);
    // totalKernelStaked = 10_000; attacker = 90%

    // Admin notifies reward
    vm.prank(admin);
    pool.notifyRewardAmount(100_000e18);

    // Advance 1 day
    vm.warp(block.timestamp + 1 days);

    // Attacker snapshots rewards and initiates withdrawal
    vm.prank(attacker);
    pool.initiateWithdrawal(9_000e18);

    // Attacker claims rewards immediately (no delay)
    uint256 balBefore = rewardsToken.balanceOf(attacker);
    vm.prank(attacker);
    pool.getReward();
    uint256 attackerReward = rewardsToken.balanceOf(attacker) - balBefore;

    // Assert attacker received ~90% of 1-day rewards
    // existingStaker received only ~10%
    assertGt(attackerReward, 0);
    // After withdrawalDelay, attacker recovers principal
    vm.warp(block.timestamp + pool.withdrawalDelay());
    vm.prank(attacker);
    pool.claimWithdrawal(1); // withdrawalId = 1
}
```
