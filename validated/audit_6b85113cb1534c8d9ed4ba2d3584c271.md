### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Binding It to the Pending Request, Enabling Cross-Request Replay by a Single Malicious Participant — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`, but never verifies that `payload_hash` is actually derived from the specific pending `request`. A single malicious attested participant (below threshold) can replay a valid `{payload_hash, signature}` pair from any previously completed foreign-tx verification to resolve any pending request, causing the contract to return a forged verification attestation to the waiting caller.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–754), the contract performs the following steps:

1. Asserts the caller is an attested participant.
2. Fetches the root public key for the domain.
3. Verifies `response.signature` is valid over `response.payload_hash` using the **root public key** (no tweak, no binding to the request).
4. Resolves all pending yields for `request` with the response.

The critical missing step: the contract **never verifies** that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific `request` being responded to. The `payload_hash` field is entirely caller-controlled.

```rust
// crates/contract/src/lib.rs:722-734
let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
    .expect("Secp256k1 variant always has a secp256k1 key");

let payload_hash: [u8; 32] = response.payload_hash.0;  // ← caller-supplied, unverified binding

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,   // ← root key, no tweak, no request binding
)
.is_ok()
``` [1](#0-0) 

Contrast this with the regular `respond` function, which correctly derives the expected public key from `request.tweak` before verifying, cryptographically binding the signature to the specific request:

```rust
// crates/contract/src/lib.rs:597-608
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,  // ← derived key, bound to request
)
``` [2](#0-1) 

The design document specifies that `msg_hash = SHA-256(borsh(ForeignTxSignPayload))` where `ForeignTxSignPayload::V1` contains both the `ForeignChainRpcRequest` and the `Vec<ExtractedValue>`. The contract is supposed to be the authoritative verifier of this binding, but it delegates that responsibility entirely to the caller. [3](#0-2) 

The test comment at line 3694 even acknowledges this explicitly: `// simulate signature with the root key (no tweak for foreign tx)`, confirming the design intent — but the contract never enforces that the hash being signed over actually encodes the correct request. [4](#0-3) 

---

### Impact Explanation

A single malicious attested participant (below the signing threshold) can:

1. Observe any previously completed `respond_verify_foreign_tx` call on-chain and extract the valid `{payload_hash_A, sig_A}` pair (all on-chain data is public).
2. Wait for a new `verify_foreign_transaction(request_B)` to be submitted — for example, a bridge inbound transfer for a transaction that did **not** actually finalize on the foreign chain.
3. Call `respond_verify_foreign_tx(request_B, {payload_hash: payload_hash_A, signature: sig_A})`.
4. The contract checks: is `sig_A` a valid ECDSA signature over `payload_hash_A` under the root key? **Yes** — it was produced by the threshold network for a different request.
5. The contract resolves `request_B` and returns `{payload_hash_A, sig_A}` to the waiting caller.

The caller of `request_B` receives a response that carries a genuine MPC root-key signature, but the `payload_hash` encodes a **different** transaction's data. Any bridge contract that does not independently reconstruct and verify the `payload_hash` binding will accept this as a valid attestation of `request_B`'s foreign transaction, enabling invalid bridge execution or double-spend.

This maps directly to the allowed impact: **"Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."**

---

### Likelihood Explanation

The attacker must be a single attested participant in the MPC network — a realistic adversary model explicitly within scope (Byzantine participant strictly below the signing threshold). No threshold collusion is required. The attack requires only:

- Observing a prior valid response on-chain (trivially available).
- Submitting a single `respond_verify_foreign_tx` transaction with the replayed response.

The window of opportunity exists for every pending `verify_foreign_transaction` request. The attack is deterministic and requires no cryptographic capability beyond holding a valid TEE attestation.

---

### Recommendation

The contract must enforce that `response.payload_hash` is consistent with the pending `request`. Since the contract cannot independently compute the extracted `values`, the minimum viable fix is to verify that the `payload_hash` encodes the correct `ForeignChainRpcRequest`:

- Require the responder to include the full `ForeignTxSignPayload` (not just the hash) in the response, so the contract can verify `SHA-256(borsh(payload)) == response.payload_hash` **and** `payload.request == request.request`.
- Alternatively, store a commitment to the expected request data when the request is enqueued, and verify the submitted `payload_hash` encodes that same request before accepting the response.

This mirrors the fix pattern from the ERC-3156 analog: the enforcing function must actively verify the invariant itself rather than trusting the caller to have satisfied it.

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(request_A) for a legitimate Bitcoin tx (tx_id = 0xAA...).
   → MPC nodes sign payload_hash_A = SHA256(borsh({request_A, [BlockHash(0xBB...)]}))
   → respond_verify_foreign_tx(request_A, {payload_hash_A, sig_A}) is called on-chain.
   → {payload_hash_A, sig_A} is now publicly observable in NEAR transaction history.

2. Bob submits verify_foreign_transaction(request_B) for a Bitcoin tx (tx_id = 0xCC...)
   that has NOT finalized (e.g., a fraudulent bridge deposit).

3. Malicious attested participant Eve calls:
   respond_verify_foreign_tx(request_B, {payload_hash: payload_hash_A, signature: sig_A})

4. Contract execution (lib.rs:718-734):
   - secp_pk = root public key for domain
   - verify_ecdsa_signature(sig_A, payload_hash_A, secp_pk) → Ok(())  ✓ (valid signature)
   - signature_is_valid = true
   - resolve_yields_for(pending_verify_foreign_tx_requests, request_B, {payload_hash_A, sig_A})

5. Bob's bridge contract receives VerifyForeignTransactionResponse {
       payload_hash: payload_hash_A,   // encodes tx_id=0xAA, not 0xCC
       signature: sig_A,               // genuine MPC root-key signature
   }
   → Bridge contract verifies signature is valid: YES.
   → Bridge contract does not independently verify payload_hash encodes tx_id=0xCC.
   → Bridge releases funds for an unfinalized foreign transaction.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }

pub fn verify_ecdsa_signature(
    signature: &K256Signature,
    message: &[u8; 32],
    public_key: &Secp256k1PublicKey,
) -> Result<(), VerificationError> {
    let sig_bytes = signature.to_ecrecover_bytes();

    // ecrecover with malleability_flag=true validates r < n and s < n/2,
    // then recovers the public key from the signature.
    let recovered = near_sdk::env::ecrecover(message, &sig_bytes, signature.recovery_id, true)
        .ok_or(VerificationError::FailedToRecoverSignature)?;

    if recovered != public_key.0 {
        return Err(VerificationError::RecoveredPkDoesNotMatchExpectedKey);
    }
    Ok(())
}
