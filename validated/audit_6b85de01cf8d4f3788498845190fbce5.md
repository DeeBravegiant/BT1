Audit Report

## Title
Nested O(assets × NDCs × withdrawals) Loop in `updateRSETHPrice()` Causes Unbounded Gas Consumption, Risking Oracle Freeze — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is publicly callable with no role restriction. Its gas cost scales as O(numSupportedAssets × numNDCs × numQueuedWithdrawalsPerNDC) because `NodeDelegator.getAssetUnstaking()` calls `getQueuedWithdrawals(NDC_i)` once per (asset, NDC) pair rather than once per NDC. The protocol's own comments acknowledge a hard ceiling of ~120 total queued withdrawals; forced EigenLayer operator undelegations can push the actual queued-withdrawal count above the soft cap of 80 without triggering the `maxUncompletedWithdrawalCount` guard, causing `updateRSETHPrice()` to exceed the block gas limit and permanently freeze fee minting.

## Finding Description

**Call chain:**

`updateRSETHPrice()` (public, `whenNotPaused`, no role check) → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` loops over every supported asset:

```solidity
// LRTOracle.sol L336-348
for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
    uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
    ...
}
```

`getTotalAssetDeposits(asset)` → `getAssetDistributionData(asset)` loops over every NDC and calls `getAssetUnstaking(asset)` on each:

```solidity
// LRTDepositPool.sol L446-456
for (uint256 i; i < ndcsCount;) {
    assetUnstakingFromEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getAssetUnstaking(asset);
    ...
}
```

`NodeDelegator.getAssetUnstaking()` makes a fresh external call to EigenLayer's `getQueuedWithdrawals` and iterates over every queued withdrawal and every strategy within it:

```solidity
// NodeDelegator.sol L405-427
function getAssetUnstaking(address asset) external view returns (uint256 amount) {
    (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
        _getDelegationManager().getQueuedWithdrawals(address(this));
    for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
        for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
            ...
        }
    }
}
```

`getQueuedWithdrawals(NDC_i)` is therefore called **numAssets × numNDCs** times per `updateRSETHPrice()` invocation — once per (asset, NDC) pair — even though the result is identical for all assets sharing the same NDC. Total work is O(numAssets × numNDCs × numQueuedWithdrawalsPerNDC).

**Soft cap and its bypass:**

`LRTUnstakingVault.setMaxUncompletedWithdrawalCount` enforces a ceiling of 80 (hard-coded max), with the comment explicitly acknowledging the gas ceiling:

```solidity
// LRTUnstakingVault.sol L150-158
// 120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price
// Need buffer for theoretical operator forced undelegations (ndc count * asset count = 15)
if (_maxUncompletedWithdrawalCount > 80) {
    revert MaxUncompletedWithdrawalCountTooHigh();
}
```

The buffer of 40 (80→120) is sized for `ndc_count × asset_count = 15`. However, `maxNodeDelegatorLimit` is initialized to 10 (`LRTDepositPool.sol` L49), and with 5 supported assets that product is 50 — exceeding the buffer. More critically, EigenLayer operator-forced undelegations create queued withdrawals in EigenLayer directly, bypassing `increaseUncompletedWithdrawalCount()`. The tracked `uncompletedWithdrawalCount` becomes stale; the only resync path is `setUncompletedWithdrawalCount()` (manager-only), which itself iterates all NDCs and fetches all queued withdrawals. Until the manager resyncs, `initiateUnstaking` and `undelegate` continue to check the stale (lower) counter, allowing further withdrawals to be queued above the effective gas ceiling.

Both `updateRSETHPrice()` (public) and `updateRSETHPriceAsManager()` (manager-only) call the same `_updateRsETHPrice()` with identical gas cost. If the gas cost exceeds the block gas limit, both are uncallable until queued withdrawals are completed — a process gated by EigenLayer's withdrawal delay (~7 days per batch).

## Impact Explanation

**Permanent freezing of unclaimed yield (Medium).** Protocol fee minting occurs exclusively inside `_updateRsETHPrice()`. If the function is uncallable, all fee minting halts for the duration of the freeze. Yield accrued during the freeze period may be partially unrecoverable due to the `maxFeeMintAmountPerDay` cap: when the oracle eventually unfreezes, the accumulated fee amount may exceed the daily limit, causing `_checkAndUpdateDailyFeeMintLimit` to revert and preventing full fee recovery.

**Unbounded gas consumption (Medium).** The gas cost of `updateRSETHPrice()` is unbounded with respect to the number of queued EigenLayer withdrawals, number of NDCs, and number of supported assets — all of which are protocol-controlled parameters that can grow over time.

## Likelihood Explanation

- `updateRSETHPrice()` is publicly callable with no access control.
- The protocol's own comment confirms the gas ceiling is real and was already hit at 120 withdrawals in testing.
- `maxNodeDelegatorLimit = 10` and 5 supported assets yields a multiplicative factor of 50 external EigenLayer calls per `updateRSETHPrice()` invocation, exceeding the buffer of 40 built into the soft cap.
- Forced undelegations by EigenLayer operators are an acknowledged operational scenario (explicitly referenced in the contract comment) and bypass the `maxUncompletedWithdrawalCount` guard entirely, pushing the actual queued-withdrawal count above the soft cap without any on-chain enforcement preventing it.

## Recommendation

Refactor `getAssetUnstaking` calls out of the per-asset loop. Cache `getQueuedWithdrawals(NDC_i)` once per NDC and reuse the result across all assets, reducing external calls from O(assets × NDCs) to O(NDCs). Concretely, add a function to `NodeDelegator` that returns unstaking amounts for all supported assets in a single `getQueuedWithdrawals` call, and invoke it once per NDC inside `getAssetDistributionData`. Additionally, harden the soft cap to account for the actual multiplicative factor (`ndc_count × asset_count`) rather than a fixed constant of 15.

## Proof of Concept

Call trace for `updateRSETHPrice()` with 5 assets and 10 NDCs each holding 8 queued withdrawals (total = 80, within the soft cap):

```
updateRSETHPrice()                          // public, no role check
└─ _updateRsETHPrice()
   └─ _getTotalEthInProtocol()
      └─ for asset in [ETH, stETH, ETHx, sfrxETH, X]:   // 5 iterations
            getTotalAssetDeposits(asset)
            └─ getAssetDistributionData(asset)
               └─ for NDC in [NDC0..NDC9]:               // 10 iterations
                     getAssetUnstaking(asset)             // called 50 times total
                     └─ getQueuedWithdrawals(NDC_i)       // 50 cold external EigenLayer calls
                        └─ for withdrawal in [0..7]:      // 8 × 50 = 400 inner iterations
```

To demonstrate the forced-undelegation bypass: deploy a fork test, set `maxUncompletedWithdrawalCount = 80`, simulate an EigenLayer operator calling `forceUndelegation` on all 10 NDCs (each holding 5 strategies → 10 new queued withdrawals per NDC = 100 additional withdrawals), then call `updateRSETHPrice()` and observe OOG revert. The `uncompletedWithdrawalCount` remains at its pre-undelegation value throughout, confirming the guard is bypassed.
