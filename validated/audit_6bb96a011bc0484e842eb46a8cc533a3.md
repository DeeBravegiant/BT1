Audit Report

## Title
ETH Push-Transfer to Non-Payable Contract Permanently Freezes Withdrawal Funds - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`LRTWithdrawalManager` uses a push-payment pattern to deliver ETH to withdrawing users in `_processWithdrawalCompletion`. Because rsETH is burned in a prior, already-finalized transaction (`unlockQueue`), any user whose address is a contract without a `receive()` or `fallback()` function will have their rsETH permanently destroyed while the corresponding ETH becomes irrecoverable inside the withdrawal manager, with no on-chain rescue path.

## Finding Description
The withdrawal lifecycle spans three separate transactions:

**Step 1 — `initiateWithdrawal` (L166):** rsETH is pulled from the user into the withdrawal manager via `safeTransferFrom`.

**Step 2 — `unlockQueue` (L305–307):** rsETH held by the manager is burned via `burnFrom`, and the corresponding ETH is redeemed from `LRTUnstakingVault`. This transaction is irreversible and fully independent of Step 3.

**Step 3 — `completeWithdrawal` → `_processWithdrawalCompletion` (L699–738):** The nonce is popped (L705), the request is deleted (L712), `unlockedWithdrawalsCount` is decremented (L717), and then ETH is pushed to the user via `_transferAsset` (L734).

Inside `_transferAsset` (L877–879):
```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `to` is a contract without a `receive()` or `fallback()` function, the low-level call returns `false`, `EthTransferFailed` is thrown, and the entire Step 3 transaction reverts — restoring the nonce, the request record, and the count. However, the rsETH burned in Step 2 is **not** restored; it was destroyed in a prior, already-committed transaction.

The result is a permanent deadlock:
- The withdrawal request remains in storage (Step 3 always reverts).
- `unlockedWithdrawalsCount[asset]` never reaches zero, so `hasUnlockedWithdrawals(asset)` remains `true`.
- `sweepRemainingAssets` is blocked by the `if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist()` guard (L403).
- The operator path `completeWithdrawalForUser` (L192–203) also calls `_processWithdrawalCompletion(asset, user, referralId)` and pushes ETH to the same non-payable `user` address, so it fails identically.
- No per-user rescue or re-routing function exists anywhere in the contract.

The ETH is permanently locked inside `LRTWithdrawalManager` with no on-chain path to recover it for that user.

## Impact Explanation
**Critical — Permanent freezing of funds.** After `unlockQueue` executes, the user's rsETH is irreversibly gone. If the ETH push to the user's contract address always reverts, the ETH is permanently locked inside `LRTWithdrawalManager`. The user suffers a total loss of their staked position, matching the "Permanent freezing of funds" Critical impact class.

## Likelihood Explanation
**Low–Medium.** The trigger requires the withdrawing address to be a smart contract without a `receive()` or `fallback()` function. No attacker action is required; the freeze is triggered accidentally by the affected user's own contract architecture. Realistic scenarios include protocol-owned treasury or DAO contracts holding rsETH, automated vault or strategy contracts that deposit into KelpDAO on behalf of users, and any contract that calls `initiateWithdrawal` without verifying its ETH-receive capability. The condition is uncommon but entirely plausible in a DeFi composability context.

## Recommendation
1. **Pull-payment pattern:** Instead of pushing ETH to the user in `_processWithdrawalCompletion`, record a claimable balance mapping and let the user pull it with a separate `claimETH()` call. This decouples delivery failure from withdrawal accounting.
2. **Pre-flight check:** Before accepting an ETH withdrawal request in `initiateWithdrawal`, verify that the caller can receive ETH (e.g., `require(msg.sender.code.length == 0 || <test-send passes>)`).
3. **Admin rescue:** Add a privileged function that can re-route stuck ETH for a specific request ID to an alternate address supplied by the user, callable only after a timeout, without requiring the original non-payable address to receive ETH.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// A contract with NO receive() function — cannot accept raw ETH pushes
contract NoReceiveWallet {
    function initiateWithdraw(address withdrawalManager, uint256 rsETHAmount) external {
        // rsETH approved to withdrawal manager beforehand
        ILRTWithdrawalManager(withdrawalManager)
            .initiateWithdrawal(ETH_TOKEN, rsETHAmount, "ref");
    }

    function tryComplete(address withdrawalManager) external {
        // ALWAYS reverts with EthTransferFailed — no receive() present
        ILRTWithdrawalManager(withdrawalManager)
            .completeWithdrawal(ETH_TOKEN, "ref");
    }
}
```

**Attack trace:**
1. `NoReceiveWallet.initiateWithdraw(...)` — rsETH transferred to `LRTWithdrawalManager`. ✓
2. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH **burned** (L305), ETH redeemed into manager (L307). ✓ (irreversible)
3. `NoReceiveWallet.tryComplete(...)` — `_transferAsset` pushes ETH to `NoReceiveWallet`, call returns `false`, reverts with `EthTransferFailed`. Withdrawal request restored in storage, but rsETH is already gone.
4. Step 3 can be retried infinitely; it will always revert. `unlockedWithdrawalsCount` never decrements. `sweepRemainingAssets` is permanently blocked. ETH is permanently frozen.
