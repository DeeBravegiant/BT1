### Title
No Cancellation Path for Pending NEAR-Origin Transfers Permanently Locks User Tokens and Native Fee - (`near/omni-bridge/src/lib.rs`)

### Summary
The NEAR bridge contract has no function to cancel a pending outbound transfer. Once a user initiates a transfer via `ft_transfer_call` → `ft_on_transfer` → `init_transfer_internal`, their tokens are burned/locked and their `native_fee` is irrevocably deducted from their storage balance. If the transfer is never finalized on the destination chain (i.e., `claim_fee` is never called), the `native_fee` — and the bridged token amount — are permanently unclaimable by the user with no on-chain recovery path.

### Finding Description

**Step 1 — Tokens and native_fee are locked atomically in `init_transfer_internal`.**

`init_transfer_internal` computes `required_storage_balance` as the sum of the storage bytes cost **plus** the full `native_fee` amount, then deducts it from the user's `available` storage balance: [1](#0-0) 

Tokens are then burned (bridge tokens) or locked (native tokens): [2](#0-1) 

**Step 2 — `remove_transfer_message` refunds only storage bytes, never the `native_fee`.**

When a pending transfer is removed (e.g., in `sign_transfer_callback` when `fee.is_zero()`), only the freed storage bytes are credited back to the owner's `available` balance. The `native_fee` portion that was deducted in step 1 is silently discarded: [3](#0-2) 

**Step 3 — The only release path for `native_fee` is `claim_fee`, callable only by the relayer.**

`claim_fee` is gated behind `#[trusted_relayer]` and requires a valid proof from the destination chain. It sends the `native_fee` to the relayer, not back to the user: [4](#0-3) 

**Step 4 — No cancellation function exists.**

A grep for `fn cancel`, `fn withdraw_transfer`, `fn abort`, and `fn revert_transfer` in `near/omni-bridge/src/lib.rs` returns zero matches. There is no public function that allows a user to cancel a pending transfer and recover their tokens or `native_fee`.

**Step 5 — `update_transfer_fee` cannot reduce the fee.**

The only fee-mutation path enforces that fees can only increase, not decrease, so the user cannot zero out the `native_fee` to mitigate the lock: [5](#0-4) 

**Step 6 — `storage_withdraw` cannot reach the locked amount.**

`storage_withdraw` only releases the `available` portion of the storage balance. Because the `native_fee` was deducted from `available` at init time and is never restored, the user cannot withdraw it: [6](#0-5) 

### Impact Explanation

If a pending transfer is never finalized — because the relayer goes offline, the MPC signing network is temporarily unavailable, or the destination chain is unreachable — the user suffers two simultaneous permanent losses:

1. **Bridged tokens**: burned (bridge tokens) or locked (native tokens) in `init_transfer_internal` with no reversal path.
2. **Native fee (NEAR)**: deducted from the user's storage balance and irrecoverable because `remove_transfer_message` does not restore it and no cancellation function exists.

This matches the **Critical** criterion: *irreversible fund lock / permanently unclaimable user value in bridge flows*.

### Likelihood Explanation

Any scenario in which a signed transfer is not followed by a `claim_fee` call triggers the lock: relayer downtime, MPC latency spikes, destination-chain congestion, or a user who initiated a transfer to an unsupported or misconfigured recipient address. The `ft_transfer_call` entry point is fully public and requires no special role. The vulnerability is latent in every transfer that carries a non-zero `native_fee`.

### Recommendation

1. **Add a `cancel_transfer` function** callable by the transfer `sender` (or `storage_owner`) that:
   - Removes the entry from `pending_transfers`.
   - Restores the `native_fee` to the caller's `available` storage balance (or transfers it directly).
   - Unlocks/re-mints the bridged token amount back to the sender.
2. Alternatively, modify `remove_transfer_message` to also refund the `native_fee` stored in the `TransferMessage` to the owner's `available` balance whenever a transfer is removed for any reason.

### Proof of Concept

```
1. Alice calls ft_transfer_call(locker, 1000 tokens, msg={native_fee: 5 NEAR, fee: 10, recipient: eth:0xABCD})
2. ft_on_transfer → init_transfer → init_transfer_internal:
     - 1000 tokens burned/locked
     - Alice's available storage balance reduced by (storage_bytes_cost + 5 NEAR)
     - TransferMessage inserted into pending_transfers with owner = Alice
3. Relayer calls sign_transfer → MPC signs → sign_transfer_callback fires
     - fee.is_non_zero() → transfer NOT removed from pending_transfers
4. Relayer goes offline; claim_fee is never called.
5. Alice calls storage_withdraw(5 NEAR) → fails: available balance is 0 (already deducted).
6. Alice has no cancel_transfer to call.
7. 1000 tokens and 5 NEAR are permanently locked with no on-chain recovery.
``` [7](#0-6) [8](#0-7)

            return transfer_message.amount;

                &token_id,

            return transfer_message.amount;

        assert_one_yocto();

        let mut storage = self
            .storage_balance_of(&account_id)
            .near_expect(StorageError::AccountNotRegistered(account_id.clone()));

                requested: to_withdraw,
