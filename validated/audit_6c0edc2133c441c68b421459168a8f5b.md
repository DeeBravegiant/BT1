### Title
Missing Aggregated-Output Verification in Non-PV CKD Coordinator Allows Byzantine Participant to Corrupt Derived Key - (`crates/threshold-signatures/src/confidential_key_derivation/protocol.rs`)

---

### Summary

The non-publicly-verifiable `ckd()` protocol in `protocol.rs` has no per-share or post-aggregation cryptographic verification. A single Byzantine participant (below threshold) can send arbitrary `CKDOutput` elliptic-curve points to the coordinator, which blindly aggregates them. The resulting `CKDOutput` is cryptographically invalid for the current `(app_id, app_pk, participants)` tuple. The contract's `respond_ckd` also performs no verification for the `AppPublicKey` variant, so the corrupted output is accepted on-chain and returned to the caller.

---

### Finding Description

**Root cause — `do_ckd_coordinator` in `protocol.rs`:** [1](#0-0) 

The coordinator iterates over every `CKDOutput` received from participants and unconditionally adds the raw G1 points:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

`recv_from_others` only checks that the message arrived from a known participant at the correct waitpoint — it performs no cryptographic binding check: [2](#0-1) 

`CKDOutput` is two bare G1 points with no session, `app_id`, or `app_pk` binding: [3](#0-2) 

There is no post-aggregation check in the non-PV coordinator — the function returns the corrupted `CKDOutput` directly: [4](#0-3) 

**Contrast with the PV variant (`protocol_pv.rs`):**

The PV coordinator runs `aggregated_output_check` after aggregation, verifying `e(C, g2) = e(Y, app_pk2) · e(H(pk‖app_id), msk_pk)` via a pairing: [5](#0-4) [6](#0-5) 

This check is entirely absent from the non-PV path.

**Production dispatch — `node/src/providers/ckd/sign.rs`:**

Both variants are dispatched in production based on the `app_public_key` type: [7](#0-6) 

The `AppPublicKey` branch calls the unprotected `ckd()`.

**Contract-level verification — `contract/src/lib.rs`:**

`respond_ckd` also skips verification for the `AppPublicKey` case: [8](#0-7) 

So the corrupted output passes through the contract unchecked and is returned to the caller.

---

### Impact Explanation

A Byzantine participant sends arbitrary G1 points as their `CKDOutput`. The coordinator aggregates them, producing:

```
C_bad = msk·H(pk‖app_id) + Σ_{j≠b} λ_j·y_j·app_pk + C_b_arbitrary
Y_bad = Σ_{j≠b} λ_j·y_j·G + Y_b_arbitrary
```

When the client unmasks with `C_bad − a·Y_bad`, the result is not `msk·H(pk‖app_id)`. The app derives a wrong key. The invariant that each CKD request produces a deterministic output bound to its `(app_id, app_pk, participants)` tuple is broken. This is a **request-lifecycle manipulation** breaking a production safety invariant.

---

### Likelihood Explanation

- Any single participant in the non-PV CKD path can trigger this — no threshold collusion required.
- The attacker is a legitimate (but Byzantine) MPC participant, which is an explicitly modeled threat.
- The `AppPublicKey` variant is a live production code path.
- No special network access or operator misconfiguration is needed.

---

### Recommendation

Apply the same `aggregated_output_check` pairing verification to the non-PV coordinator after aggregation. Since the non-PV variant uses a single G1 `app_pk` (not a G1/G2 pair), the check must be adapted: require callers to also supply a G2 encoding of the same secret (i.e., migrate `AppPublicKey` to `AppPublicKeyPV`), or add an independent per-share zero-knowledge proof of correct computation (e.g., a Schnorr proof that `C_i − Y_i·app_pk = x_i·H(pk‖app_id)` for the participant's known public share `x_i·G2`).

### Proof of Concept

```rust
// In an integration test:
// 1. Run a legitimate CKD for app_id_A, capture participant P_b's CKDOutput.
// 2. Start a new CKD for app_id_B with the same participants.
// 3. Have P_b inject the CKDOutput from step 1 instead of computing a fresh share.
// 4. The coordinator aggregates without error.
// 5. Assert: ckd_output.unmask(app_sk_B) != msk * hash_app_id_with_pk(&pk, &app_id_B)
//    (the output is wrong for app_id_B).
// 6. Confirm the PV variant rejects the same injection via ProtocolError::AssertionFailed.
```

The non-PV coordinator has no mechanism to detect the injected share; the PV coordinator's `aggregated_output_check` would catch it immediately. [9](#0-8)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`
fn aggregated_output_check(
    output: &CKDOutput,
    app_pk: &PublicVerificationKey,
    public_key: &VerifyingKey,
    hash_point: &ElementG1,
) -> bool {
    if !check_valid_point_g1(output.big_c.into()) || !check_valid_point_g1(output.big_y.into()) {
        return false;
    }
    multi_miller_loop(&[
        (output.big_c, -ElementG2::generator()),
        (output.big_y, app_pk.pk2),
        (*hash_point, public_key.to_element()),
    ])
}
