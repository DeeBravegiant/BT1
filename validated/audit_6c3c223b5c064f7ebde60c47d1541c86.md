### Title
Single Malicious Attested Participant Can Deliver Arbitrary CKD Output for `AppPublicKey` Requests — (File: crates/contract/src/lib.rs)

### Summary

The `respond_ckd` contract method performs no cryptographic verification of the CKD response when the request carries an `AppPublicKey` variant. Any single attested participant can call `respond_ckd` with a completely fabricated `CKDResponse` and the contract will accept it, resume all waiting yields, and deliver the forged key material to the user — entirely bypassing the threshold-MPC requirement.

### Finding Description

In `respond_ckd` (lines 654–688 of `crates/contract/src/lib.rs`), the contract branches on the request's `app_public_key` field:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant the contract calls `ckd_output_check`, which verifies the MPC-produced ciphertext against the user-supplied public key and the domain's BLS12-381 root key. For the `AppPublicKey` variant the match arm is intentionally empty — zero verification is performed. The function then unconditionally calls `resolve_yields_for`, which drains the entire fan-out queue and resumes every waiting yield with whatever bytes the caller supplied:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

`resolve_yields_for` removes the map entry on the first successful call, so the first `respond_ckd` call wins and all subsequent honest-node calls return `RequestNotFound`:

```rust
let resumed = requests
    .remove(request)          // ← entry gone after first call
    .unwrap_or_default()
    ...
``` [3](#0-2) 

The only gate in front of `respond_ckd` is `assert_caller_is_attested_participant_and_protocol_active()` — any single attested participant satisfies it. No leader check, no quorum check, no response-correctness check exists for `AppPublicKey` requests. [4](#0-3) 

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe a pending `AppPublicKey` CKD request on-chain.
2. Construct an arbitrary `CKDResponse { big_y, big_c }` — no MPC protocol participation required.
3. Call `respond_ckd` before the honest leader submits the real result.
4. The contract accepts the fabricated response, resumes all queued yields, and delivers the forged key material to every waiting caller.

The user receives a key that was never produced by the threshold MPC network. Depending on how the derived key is used (wallet derivation, encryption of assets, cross-chain bridge authorization), this directly enables theft or permanent loss of funds controlled by the derived key. This matches the allowed critical impact: **unauthorized confidential key derivation output without the required participant authorization**.

### Likelihood Explanation

Medium. The attacker must be an attested participant (TEE-attested, registered in the contract). However, no collusion with other participants is needed — a single rogue node suffices. The attack window is the time between a CKD request appearing on-chain and the honest leader submitting the real response. Because the attacker does not need to run any MPC computation, they can submit immediately upon seeing the request, consistently winning the race.

### Recommendation

Apply the same on-chain output verification to `AppPublicKey` requests that is already applied to `AppPublicKeyPV` requests. If the `AppPublicKey` variant genuinely cannot be verified with a single on-chain check (because no verifiable public-key commitment is available), require a threshold quorum of participants to submit identical responses before the yield is resolved — mirroring the threshold guarantee that the MPC protocol itself provides.

### Proof of Concept

```
1. User calls request_app_private_key({
       app_public_key: AppPublicKey(some_bls_key),
       domain_id: <CKD domain>,
       derivation_path: "m/0"
   })
   → contract stores CKDRequest R in pending_ckd_requests, creates yield Y.

2. Attacker (any single attested participant) immediately calls:
   respond_ckd(
       request = R,
       response = CKDResponse { big_y: ATTACKER_POINT, big_c: ATTACKER_CIPHERTEXT }
   )
   → contract enters respond_ckd:
       assert_caller_is_attested_participant_and_protocol_active() → passes
       match AppPublicKey(_) => {}   ← no verification
       resolve_yields_for(R, serialize(ATTACKER_RESPONSE))
           → removes R from pending_ckd_requests
           → resumes yield Y with forged bytes

3. Honest MPC nodes finish the real computation and call respond_ckd(R, real_response)
   → resolve_yields_for returns Err(RequestNotFound) — entry already gone.

4. User's callback receives ATTACKER_CIPHERTEXT as the CKD output.
   Any asset or key derived from this output is under attacker control.
```

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
