### Title
Proposal Deposits Permanently Locked in Contract When Any Update Is Executed — (`crates/contract/src/update.rs`)

### Summary

When `do_update` executes a winning governance proposal, it unconditionally clears **all** pending proposals from storage. The NEAR deposits paid by the proposers of the cleared (non-executed) proposals are absorbed into the contract's account balance with no refund path and no record of who paid them.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a deposit proportional to the size of the proposed update (contract binary or config blob). The required amount is computed by `ProposedUpdates::required_deposit`, which multiplies `env::storage_byte_cost()` by the estimated bytes the entry will occupy. Only the excess above the required amount is refunded to the proposer; the required portion is retained by the contract to cover storage staking. [1](#0-0) 

The `UpdateEntry` struct that is stored for each proposal contains only the update payload and a `bytes_used` field — **the proposer's `AccountId` is never recorded**. [2](#0-1) 

When threshold votes are reached, `vote_update` calls `do_update`, which removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()` to wipe every other pending proposal. [3](#0-2) 

Because the proposer's identity was never stored, there is no way to issue refunds for the cleared proposals. The freed storage staking is released back into the contract's own balance, and the original depositors have no mechanism to recover their NEAR.

### Impact Explanation

Every participant who proposed an update that was not the one ultimately executed permanently loses their deposit. For a large contract binary (e.g., a 1 MB WASM), the required deposit is approximately `1,000,000 bytes × 10 yoctoNEAR/byte = 10,000,000,000,000,000,000,000 yoctoNEAR ≈ 10 NEAR`. With multiple competing proposals, the total locked amount scales linearly with the number of non-winning proposals. The funds are irrecoverably absorbed into the contract's balance — there is no admin withdrawal function, no refund path, and no record of who is owed what.

This matches the **Medium** allowed impact: *balance or accounting invariant broken without relying on network-level DoS or operator misconfiguration*.

### Likelihood Explanation

The scenario is a normal governance operation. Multiple participants can independently propose updates (e.g., two participants propose different contract binaries at the same time). When one reaches threshold and is executed, all others are silently cleared. This is not a contrived attack — it is the documented behavior of the update system and will occur in any real deployment where governance is active.

### Recommendation

1. Store the proposer's `AccountId` inside `UpdateEntry` alongside `bytes_used`.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(refund_amount)` for each one, where `refund_amount` is `env::storage_byte_cost().saturating_mul(entry.bytes_used)`.
3. Alternatively, refund deposits lazily: keep cleared entries in a separate "refundable" map and expose a `claim_update_deposit(id)` endpoint that any former proposer can call.

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, attaching ~5 NEAR deposit. The deposit is retained by the contract.
2. Participant B calls `propose_update` with a different 500 KB binary, attaching ~5 NEAR deposit. The deposit is retained by the contract.
3. Threshold participants vote for B's proposal. `vote_update` calls `do_update(&B_id, gas)`.
4. Inside `do_update`: `self.entries.remove(&B_id)` removes B's entry; `self.entries.clear()` removes A's entry. `self.vote_by_participant.clear()` removes all votes.
5. A's ~5 NEAR is now permanently absorbed into the contract's balance. There is no function to recover it, and A's `AccountId` was never stored in the entry. [4](#0-3) [5](#0-4)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
