### Title
`try_sync` Accepts Spoofed `fee_proposal_fri` for Pre-V0_14_3 Blocks, Corrupting `fee_proposals_window` Median — (`crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

---

### Summary

`try_sync` unconditionally records `fee_proposal_fri` from a synced block into `fee_proposals_window` without checking whether the block's `starknet_version` is compatible with a non-`None` value. Because `fee_proposal_fri` is explicitly **not** part of the block hash (per the P2P proto comment), a malicious unauthenticated P2P peer can inject `Some(large_value)` for a pre-V0_14_3 block. This bypasses the `None`-sentinel that `compute_fee_actual` relies on to detect pre-V0_14_3 entries, corrupting the sliding-window median and causing every subsequent `fee_proposal_bounds` check to use a wrong `fee_actual`.

---

### Finding Description

**Step 1 — `try_sync` records without version validation.**

`try_sync` reads a `SyncBlock` from the internal state-sync component and immediately calls `record_fee_proposal` with whatever `fee_proposal_fri` the block carries:

```rust
self.record_fee_proposal(height, sync_block.block_header_without_hash.fee_proposal_fri);
```

There is no check that `sync_block.block_header_without_hash.starknet_version >= StarknetVersion::V0_14_3` before accepting a `Some(...)` value. The developer-acknowledged TODO on line 1061 confirms this gap:

```rust
// TODO(Asmaa): validate starknet_version and parent_hash when they are stored.
``` [1](#0-0) 

**Step 2 — `record_fee_proposal` inserts unconditionally.**

```rust
fn record_fee_proposal(&mut self, height: BlockNumber, fee_proposal_fri: Option<GasPrice>) {
    self.fee_proposals_window.insert(height, fee_proposal_fri);
}
``` [2](#0-1) 

**Step 3 — `compute_fee_actual` treats `Some(None)` as a sentinel but not `Some(Some(X))`.**

The function returns `None` (triggering the safe fallback) only when an entry is `Some(None)` or absent. If the entry is `Some(Some(large_value))`, it is pushed into the median window:

```rust
Some(Some(price)) => window.push(*price),
Some(None) | None => {
    warn!(...);
    return None;
}
``` [3](#0-2) 

**Step 4 — `fee_proposal_fri` is NOT part of the block hash (P2P proto).**

The P2P sync proto carries an explicit warning:

```
// WARNING: this field is currently not part of the block hash, so the value must be trusted.
optional Uint128 fee_proposal_fri = 22;
``` [4](#0-3) 

Because the field is outside the signed block hash, any unauthenticated P2P peer can send a `SignedBlockHeader` for a pre-V0_14_3 block with `fee_proposal_fri = Some(large_value)`. The block-hash signature check passes; the spoofed value is stored to local storage; `state_sync_client.get_block()` returns it; `try_sync` records it.

**Step 5 — Wrong `fee_actual` corrupts `fee_proposal_bounds` in `validate_proposal`.**

`validate_proposal` computes `fee_actual` from the same `fee_proposals_window` and uses it to enforce bounds on incoming proposals:

```rust
fee_actual: compute_fee_actual(
    &self.fee_proposals_window,
    init.height,
    VersionedConstants::latest_constants().fee_proposal_window_size,
),
``` [5](#0-4) 

```rust
if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
``` [6](#0-5) 

A corrupted `fee_actual` shifts both bounds, causing honest proposals to be rejected or dishonest proposals to be accepted.

**Step 6 — Same gap in `initialize_fee_proposals_window`.**

The window-initialization path has the identical missing check:

```rust
Ok(block) => self.record_fee_proposal(
    block_number,
    block.block_header_without_hash.fee_proposal_fri,
),
``` [7](#0-6) 

---

### Impact Explanation

A corrupted `fee_proposals_window` causes `compute_fee_actual` to return a wrong median. This wrong `fee_actual` is used in:

1. **`fee_proposal_bounds` in `validate_proposal`** — nodes that synced the spoofed block will reject honest proposals or accept out-of-range proposals, causing consensus disagreement between synced and non-synced nodes.
2. **`calculate_next_l2_gas_price`** — the next block's L2 gas price is derived from `fee_actual`; a corrupted value produces a wrong `next_l2_gas_price`, directly affecting transaction fee economics. [8](#0-7) 

---

### Likelihood Explanation

- Any unauthenticated P2P peer can craft a `SignedBlockHeader` with a spoofed `fee_proposal_fri` for a historical pre-V0_14_3 block. No validator or operator privilege is required.
- The P2P proto explicitly documents that this field is outside the block hash, so no cryptographic barrier exists.
- The attack is effective during the window immediately after the V0_14_3 upgrade, when the sliding window still contains pre-V0_14_3 heights.

---

### Recommendation

In both `try_sync` and `initialize_fee_proposals_window`, validate that `fee_proposal_fri` is consistent with `starknet_version` before calling `record_fee_proposal`:

```rust
let fee_proposal = if sync_block.block_header_without_hash.starknet_version
    >= StarknetVersion::V0_14_3
{
    sync_block.block_header_without_hash.fee_proposal_fri
} else {
    // Pre-V0_14_3: fee_proposal must be None regardless of what the peer sent.
    None
};
self.record_fee_proposal(height, fee_proposal);
```

This mirrors the existing check in `validate_proposal` (`is_proposal_init_valid`) and closes the gap that the TODO on line 1061 acknowledges. [9](#0-8)
