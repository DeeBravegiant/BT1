### Title
`respond_verify_foreign_tx` Accepts Replayed Stale Foreign-Chain Attestations — No Binding Between `payload_hash` and the Pending Request — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` on-chain entry point verifies only that the submitted ECDSA signature is valid over `response.payload_hash` using the MPC root key. It does **not** verify that `payload_hash` is the hash of `ForeignTxSignPayload{request, values}` for the specific pending request being answered. Because `ForeignTxSignPayloadV1` contains no timestamp, NEAR block height, or nonce, any previously produced MPC signature over `(request, old_values)` is permanently valid and can be replayed by a single Byzantine attested participant against a new pending request for the same foreign-chain transaction, causing the network to attest to stale or reorg-invalidated foreign-chain state.

---

### Finding Description

**Root cause — missing `payload_hash` ↔ request binding in `respond_verify_foreign_tx`**

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` (lines 691–754) performs two checks:

1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the MPC root key. [1](#0-0) 

It then resolves all queued yields for `request` with the raw `response` bytes: [2](#0-1) 

There is **no check** that `response.payload_hash == SHA-256(borsh(ForeignTxSignPayload{request, values}))` for any specific `values`. The contract never sees the extracted values; it only sees the hash and the signature over it.

**Root cause — `ForeignTxSignPayloadV1` carries no temporal binding**

The struct that nodes sign is: [3](#0-2) 

It contains only `request` (the foreign-chain query) and `values` (the extracted observations). There is no NEAR block height, no wall-clock timestamp, and no per-request nonce. The design document explicitly states the payload should include `observed_at`: [4](#0-3) 

but the implementation omits it. A valid signature over `(request, values)` is therefore valid forever and for any future pending request that carries the same `request` key.

**How the pending-request map is keyed**

`VerifyForeignTransactionRequest` is keyed on `(request, domain_id, payload_version)` — no caller identity, no NEAR receipt ID, no timestamp: [5](#0-4) 

After a request is resolved the map entry is removed. When the same foreign-chain transaction is submitted again (e.g., after a reorg), a fresh map entry is created under the identical key, and the old signature is still cryptographically valid against it.

**Node-side caching makes the old signature trivially available**

The leader node caches the computed `VerifyForeignTransactionRespondArgs` in memory and resubmits it on every retry without re-querying the foreign chain: [6](#0-5) 

The full assembled signature is also broadcast on-chain in the `respond_verify_foreign_tx` transaction, making it publicly observable by any attested participant.

---

### Impact Explanation

A single Byzantine attested participant (below the signing threshold) who observed or led a prior signing round for request `R` possesses a valid MPC threshold signature `sig_old` over `payload_hash_old = SHA-256(borsh({R, V_old}))`. When a new pending request for `R` is created (e.g., after a foreign-chain reorg that invalidated the transaction), that participant calls:

```
respond_verify_foreign_tx(request=R, response={payload_hash_old, sig_old})
```

The contract accepts the call (signature is valid), resolves all queued yields with `payload_hash_old`, and every waiting caller receives an MPC attestation that the foreign-chain transaction was observed with values `V_old` — values that may correspond to a block that no longer exists on the canonical chain.

For the primary use case (Omnibridge inbound flow), this enables a bridge service to claim NEAR-side funds for a foreign-chain deposit that was reorganized away, constituting a **forged foreign-chain verification leading to invalid bridge execution or double-spend**.

This matches the allowed High impact: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

**Medium-High.** The preconditions are:

1. The same foreign-chain transaction is submitted twice (realistic after any reorg or retry).
2. A single attested participant has access to the old `(payload_hash, signature)` pair — either because they were the signing leader (and the node cached it) or because they read it from the on-chain `respond_verify_foreign_tx` transaction, which is publicly visible.
3. The attacker calls `respond_verify_foreign_tx` before honest nodes submit a fresh response for the new request.

No threshold collusion is required. The attacker is one Byzantine participant acting alone.

---

### Recommendation

1. **Bind `payload_hash` to the pending request on-chain.** Include the extracted `values` in `VerifyForeignTransactionResponse` and have `respond_verify_foreign_tx` recompute `SHA-256(borsh(ForeignTxSignPayload{request, values}))` on-chain, rejecting any response where the recomputed hash does not match `response.payload_hash`.

2. **Add a temporal or per-instance nonce to `ForeignTxSignPayloadV1`.** Include the NEAR block height (or the NEAR receipt ID of the originating `verify_foreign_transaction` call) in the signed payload. This makes each signature valid only for the specific request instance that triggered it, preventing cross-instance replay.

3. **Implement the `observed_at` field** that the design document already specifies but the implementation omits. [4](#0-3) 

---

### Proof of Concept

```
T1: User calls verify_foreign_transaction(request=R)
    → pending_verify_foreign_tx_requests[R] = [yield_id_1]

T1: MPC nodes query foreign chain, get V1 = [BlockHash(H1)]
    payload_hash_1 = SHA-256(borsh(ForeignTxSignPayload::V1{R, V1}))
    MPC threshold signs payload_hash_1 → sig_1
    Leader (Byzantine) saves (payload_hash_1, sig_1)

T1: Leader calls respond_verify_foreign_tx(R, {payload_hash_1, sig_1})
    → contract verifies sig_1 over payload_hash_1 ✓
    → yield_id_1 resolved with payload_hash_1
    → pending_verify_foreign_tx_requests[R] removed

    [Foreign chain reorg: block H1 is orphaned; tx R is no longer canonical]

T2: User calls verify_foreign_transaction(request=R) again
    → pending_verify_foreign_tx_requests[R] = [yield_id_2]

T2: Byzantine participant calls respond_verify_foreign_tx(R, {payload_hash_1, sig_1})
    → contract verifies sig_1 over payload_hash_1 ✓  (no freshness check)
    → yield_id_2 resolved with stale payload_hash_1
    → User receives MPC attestation for the reorg-invalidated state V1=[BlockHash(H1)]
    → Bridge contract accepts attestation → invalid funds released
```

The contract has no mechanism to detect that `payload_hash_1` was produced for a different request instance or that the foreign-chain state it encodes is no longer canonical. [7](#0-6) [8](#0-7) [6](#0-5)
