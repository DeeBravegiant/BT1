### Title
Wallet Accepts Forged Coin State Due to Missing Per-Coin Check in `validate_additions` When `proofs=None` - (File: chia/wallet/util/wallet_sync_utils.py)

### Summary
In `validate_additions`, when a peer responds with `proofs=None`, the function only verifies that the Merkle root recomputed from the returned coin list matches the block's `additions_root`. It never checks that the specific puzzle hash the wallet requested is actually present in the returned coins. A malicious untrusted peer can exploit this by responding with `proofs=None` and the real coins of any canonical block (so the root check passes), while the wallet's specific coin is absent. The wallet then incorrectly accepts a forged `CoinState`, corrupting its coin records and balance.

### Finding Description

`validate_received_state_from_peer` in `chia/wallet/wallet_node.py` validates an untrusted peer's `CoinState` by calling `request_and_validate_additions`: [1](#0-0) 

`request_and_validate_additions` sends `RequestAdditions(height, header_hash, [puzzle_hash])` â€” a request for a **specific** puzzle hash â€” and passes the peer's response directly to `validate_additions`: [2](#0-1) 

`validate_additions` has two branches. When `proofs is not None`, each puzzle hash is individually proven against the Merkle root. When `proofs is None`, the function only recomputes the root from the returned coin list and compares it to the block's `additions_root`: [3](#0-2) 

**The missing check**: when `proofs is None`, the function never verifies that the specific `puzzle_hash` the wallet requested is actually present in the returned `coins` list. It returns `True` solely because the aggregate root matches.

A legitimate full node always returns proofs when specific puzzle hashes are requested: [4](#0-3) 

But a malicious peer is free to respond with `proofs=None` and any coin list whose Merkle root equals the canonical block's `additions_root` â€” i.e., the real coins of that block. `validate_additions` returns `True`, and the caller never inspects whether the wallet's coin is actually in the list: [5](#0-4) 

The result is cached and the forged `CoinState` is accepted: [6](#0-5) 

The accepted state is then committed to the wallet's coin store via `add_coin_states`. [7](#0-6) 

### Impact Explanation

A malicious untrusted peer can inject an arbitrary `CoinState` â€” with any `puzzle_hash`, `parent_coin_info`, and `amount` â€” into the wallet's coin records. The wallet's displayed balance is inflated by the forged coin's amount. If a user is awaiting a payment and the malicious peer injects a forged coin state matching the expected amount, the user may believe the payment arrived and release goods or services. This constitutes payment forgery with direct financial impact. The wallet's coin records are persistently corrupted until a resync from a honest peer.

### Likelihood Explanation

Any peer the wallet connects to in untrusted mode (i.e., any third-party full node) can mount this attack. The wallet's untrusted-mode sync path is the standard path for wallets not connected to their own node. No cryptographic capability is required â€” the attacker only needs to serve a well-formed `RespondAdditions` message with `proofs=None` and the real coin list of any canonical block.

### Recommendation

After `validate_additions` returns `True` in the `proofs is None` branch, the caller must verify that the specific puzzle hash it requested is actually present in the returned coin list. Alternatively, modify `validate_additions` to accept the requested puzzle hash as a parameter and enforce its presence when `proofs is None`. The proof-based branch already performs this check correctly via `confirm_included_already_hashed` / `confirm_not_included_already_hashed`. [8](#0-7) 

### Proof of Concept

1. Wallet (untrusted mode) subscribes to puzzle hash `PH_wallet`.
2. Malicious peer sends `CoinState(coin=Coin(parent=RAND, puzzle_hash=PH_wallet, amount=1_000_000_000_000), created_height=H, spent_height=None)`.
3. Wallet calls `request_and_validate_additions(peer, cache, H, header_hash_H, PH_wallet, additions_root_H)`.
4. Malicious peer responds: `RespondAdditions(height=H, header_hash=header_hash_H, coins=<real coins of block H>, proofs=None)`.
5. `validate_additions` recomputes root from real coins â†’ matches `additions_root_H` â†’ returns `True`. `PH_wallet` is never checked for membership.
6. `validate_block_inclusion` passes (block H is canonical).
7. Wallet commits the forged coin to its coin store. Balance inflated by 1 XCH.
8. User, seeing the expected balance, releases goods. The coin cannot be spent on-chain; the payment was never made.

async def request_and_validate_additions(
    peer: WSChiaConnection,
    peer_request_cache: PeerRequestCache,
    height: uint32,
    header_hash: bytes32,
    puzzle_hash: bytes32,
    additions_root: bytes32,
) -> bool | None:
    if peer_request_cache.in_additions_in_block(header_hash, puzzle_hash):
        return True
    additions_request = RequestAdditions(height, header_hash, [puzzle_hash])
    additions_res: RespondAdditions | RejectAdditionsRequest | None = await peer.call_api(
        FullNodeAPI.request_additions, additions_request
    )
    if additions_res is None or isinstance(additions_res, RejectAdditionsRequest):
        log.info(
            f"Failed to obtain additions for height {height} header hash {header_hash} "
            f"puzzle hash {puzzle_hash} from peer {peer.peer_node_id} / {peer.peer_info.host} "
            f"version {peer.version} response: {additions_res}"
        )
        return None
    result: bool = validate_additions(
        additions_res.coins,
        additions_res.proofs,
        additions_root,
    )
    peer_request_cache.add_to_additions_in_block(header_hash, puzzle_hash, height)
    return result
