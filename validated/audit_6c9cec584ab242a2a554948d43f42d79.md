Audit Report

## Title
`getAssetUnstaking` Uses Raw `scaledShares` Without Slashing Adjustment, Causing `getAvailableAssetAmount` to Overestimate Withdrawable ETH — (File: contracts/NodeDelegator.sol)

## Summary

`NodeDelegator.getAssetUnstaking` returns raw EigenLayer `scaledShares` for the beacon-chain ETH strategy without multiplying by `maxMagnitude × beaconChainScalingFactor`. After a slashing event, this inflates `getTotalAssetDeposits`, which inflates `getAvailableAssetAmount`, allowing users to commit more ETH to the withdrawal queue than EigenLayer will ever deliver. When `unlockQueue` runs against the actual (slashed) vault balance, it can only service a subset of committed requests, leaving the remainder permanently or temporarily frozen.

## Finding Description

`NodeDelegator.getAssetUnstaking` iterates over queued EigenLayer withdrawals and sums the withdrawable amount per asset:

```solidity
// NodeDelegator.sol lines 421-424
uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
amount += strategyAsset == LRTConstants.ETH_TOKEN
    ? sharesToUnstake                              // ← raw scaledShares, no slashing factor
    : strategy.sharesToUnderlyingView(sharesToUnstake);
```

`withdrawalShares` is the second return value of `IDelegationManager.getQueuedWithdrawals`, which exposes the `scaledShares` field of the `Withdrawal` struct. The EigenLayer interface explicitly documents at lines 123–133 of `IDelegationManager.sol`:

> "Note that these scaledShares need to be multiplied by the operator's maxMagnitude and beaconChainScalingFactor at completion to include slashing occurring during the queue withdrawal delay."

For LST strategies the code delegates to `strategy.sharesToUnderlyingView`, which internally applies the strategy's exchange rate and reflects slashing. For the beacon-chain ETH strategy the code uses `sharesToUnstake` directly — slashing is silently ignored. A grep for `getWithdrawableShares`, `maxMagnitude`, and `beaconChainScalingFactor` across the entire repository returns zero matches, confirming no compensating adjustment exists anywhere.

The inflated value propagates through the following chain:

1. `LRTDepositPool.getTotalAssetDeposits` (lines 394–396) sums `assetUnstakingFromEigenLayer` (sourced from `getAssetUnstaking`) into `totalAssetDeposit`.
2. `LRTWithdrawalManager.getAvailableAssetAmount` (lines 600–602) computes `totalAssets - assetsCommitted[asset]` using the inflated total.
3. `initiateWithdrawal` (lines 170–173) gates new requests on this inflated available amount and increments `assetsCommitted[asset]`.

When `unlockQueue` is later called, it uses the **actual** vault balance:

```solidity
// LRTWithdrawalManager.sol line 849
totalAvailableAssets: unstakingVault.balanceOf(asset)
```

If slashing reduced the ETH that arrived in the vault, `unlockQueue` exits early at line 800 (`if (availableAssetAmount < payoutAmount) break`) once the real balance is exhausted. Requests beyond that point remain locked indefinitely, while the rsETH for those requests was already transferred to the withdrawal manager at `initiateWithdrawal` time (line 166).

## Impact Explanation

**Temporary (potentially permanent) freezing of user funds.** Users who initiated withdrawal requests after slashing occurred — while `getAssetUnstaking` still reported the pre-slashing amount — will have their requests locked in the queue. The vault will never receive enough ETH to cover the over-committed `assetsCommitted[ETH]` from the slashed withdrawal alone. If no subsequent ETH inflows (new deposits, other operator completions) arrive, the freeze is permanent. Affected users cannot complete their withdrawals and cannot reclaim their rsETH. This maps to **Critical: Permanent freezing of funds** in the worst case, and **Medium: Temporary freezing of funds** if future inflows eventually cover the shortfall.

## Likelihood Explanation

EigenLayer slashing is an explicit, documented risk of restaking. The withdrawal delay (7–14 days) creates a window during which slashing can reduce the actual ETH delivered. No special role or front-running is required — any unprivileged user can call `initiateWithdrawal` while the inflated `getAvailableAssetAmount` is visible on-chain. The protocol itself acknowledges the slashing edge case in a comment at line 149 of `LRTWithdrawalManager.sol`. The condition is reachable on any deployment where an operator is slashed during an active withdrawal queue.

## Recommendation

Replace the raw `sharesToUnstake` path for the beacon-chain ETH strategy with a call that applies the operator's current `maxMagnitude` and `beaconChainScalingFactor`, mirroring how EigenLayer computes the withdrawable amount at completion time. Alternatively, use `IDelegationManager.getWithdrawableShares` (if available in the deployed EigenLayer version), which already returns the slashing-adjusted value, and use that result in `getAssetUnstaking` instead of the raw `scaledShares`.

## Proof of Concept

1. Operator stakes 100 ETH via `NodeDelegator.stake32Eth` and verifies credentials.
2. Operator calls `NodeDelegator.initiateUnstaking` — 100 ETH worth of `scaledShares` enters the EigenLayer withdrawal queue.
3. EigenLayer slashing event reduces the operator's `maxMagnitude` by 50%; actual ETH to be received drops to 50 ETH.
4. `NodeDelegator.getAssetUnstaking(ETH)` still returns 100 ETH (raw `scaledShares`, no adjustment).
5. `LRTDepositPool.getTotalAssetDeposits(ETH)` returns 100 ETH.
6. `LRTWithdrawalManager.getAvailableAssetAmount(ETH)` returns 100 ETH.
7. User A calls `initiateWithdrawal(ETH, rsETH_A)` → commits 60 ETH; `assetsCommitted[ETH]` = 60.
8. User B calls `initiateWithdrawal(ETH, rsETH_B)` → commits 40 ETH; `assetsCommitted[ETH]` = 100.
9. Operator calls `NodeDelegator.completeUnstaking` — only 50 ETH arrives in `LRTUnstakingVault`.
10. Operator calls `LRTWithdrawalManager.unlockQueue(ETH, ...)` — `totalAvailableAssets` = 50 ETH; User A's 60 ETH request cannot be covered, loop breaks immediately, neither user is unlocked (or only a partial unlock occurs).
11. User B (and potentially User A) calls `completeWithdrawal` → reverts with `WithdrawalLocked` indefinitely; their rsETH is permanently held by the withdrawal manager.

**Foundry fork test plan:** Deploy against a mainnet/testnet EigenLayer fork. Queue a beacon-chain ETH withdrawal, simulate a slashing event reducing `maxMagnitude`, call `getAssetUnstaking` and assert it returns the pre-slashing value, then call `initiateWithdrawal` for the full inflated amount, complete the EigenLayer withdrawal, call `unlockQueue`, and assert that `completeWithdrawal` reverts for the over-committed user.
