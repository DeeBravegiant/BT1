### Title
`init` Initializer Can Be Front-Run by Any Unprivileged Caller — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `init` function of the NEAR MPC contract imposes no access-control check on its caller. Because NEAR contract deployment and initialization are separate on-chain transactions, any observer can race to call `init` before the legitimate deployer does, seeding the MPC network with an attacker-controlled participant set and threshold, and thereby gaining exclusive authority to issue threshold signatures.

---

### Finding Description

The `init` function is the sole entry point for bootstrapping the MPC contract from a blank state. It accepts an arbitrary `ThresholdParameters` struct (participant set + threshold) and writes it directly into `ProtocolContractState::Running`:

```rust
// crates/contract/src/lib.rs  lines 1924-1973
#[handle_result]
#[init]                          // ← prevents re-init, but does NOT restrict the caller
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    // ... no predecessor / signer check ...
    Ok(Self {
        protocol_state: ProtocolContractState::Running(RunningContractState::new(
            DomainRegistry::default(),
            Keyset::new(EpochId::new(0), Vec::new()),
            parameters,          // ← attacker-supplied
            AddDomainsVotes::default(),
        )),
        accept_requests: true,   // ← immediately open for signing
        ...
    })
}
```

The `#[init]` attribute only prevents a second call once state exists; it does **not** restrict who may make the first call. There is no `#[private]` attribute, no `assert_eq!(env::predecessor_account_id(), env::current_account_id())`, and no check against the contract's upgrade authority.

By contrast, the sibling initializers `init_running` and `migrate` are both correctly gated:

```rust
// line 1976
#[private]   // predecessor must equal current_account_id
#[init]
pub fn init_running(...) -> Result<Self, Error> { ... }

// line 2060
#[private]
#[init(ignore_state)]
pub fn migrate() -> Result<Self, Error> { ... }
```

The deployment workflow (documented in `docs/localnet/localnet.md` and `docs/testnet/setup-guide-for-testnet-with-tee-support.md`) is a **two-step process**: first deploy the Wasm binary, then call `init` in a separate transaction. The window between those two transactions is exploitable.

---

### Impact Explanation

An attacker who front-runs `init` can supply:
- A participant set containing only accounts they control.
- A threshold of `1`.

The contract immediately enters `Running` state with `accept_requests: true`. From that point:

1. The attacker's node is the only recognized participant; it can unilaterally start key-generation (`start_keygen_instance`) and vote its own public key to completion (`vote_pk`), producing a live MPC keyset under attacker control.
2. All subsequent `sign`, `request_app_private_key`, and `verify_foreign_transaction` calls are routed to the attacker's node, which can issue threshold signatures for arbitrary foreign-chain transactions without any legitimate participant authorization.
3. The legitimate deployer cannot recover: `init` will revert on a second call because state already exists, and `migrate` / `init_running` require `#[private]` (predecessor == contract account), which the deployer cannot satisfy from outside.

This satisfies the **Critical** impact category: *unauthorized threshold signature issuance without the required participant authorization*.

---

### Likelihood Explanation

- NEAR transactions are publicly visible in the mempool and on-chain before finality; a monitoring bot can detect a `DeployContract` action and immediately submit a competing `init` call.
- No special privilege, key material, or collusion is required — only the ability to submit a NEAR transaction.
- The attack is a single transaction and requires no gas beyond the standard 300 Tgas used in the documented deployment flow.

---

### Recommendation

Add a caller-identity check at the top of `init` that restricts invocation to the contract's own account (i.e., the deployer, who controls the account before handing it over):

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    // Only the contract account itself (i.e., the deployer) may initialize.
    assert_eq!(
        env::predecessor_account_id(),
        env::current_account_id(),
        "init: caller must be the contract account"
    );
    // ... rest of function unchanged
}
```

Alternatively, apply the `#[private]` attribute (which performs the same check via the NEAR SDK macro), consistent with how `init_running` and `migrate` are already protected.

---

### Proof of Concept

1. Observe a `DeployContract` action targeting the MPC contract account on-chain.
2. Immediately submit a transaction calling `init` with attacker-controlled parameters:

```json
{
  "parameters": {
    "threshold": 1,
    "participants": {
      "next_id": 1,
      "participants": [
        [
          "attacker.near",
          0,
          {
            "tls_public_key": "<attacker_ed25519_key>",
            "url": "https://attacker-node.example.com:13001"
          }
        ]
      ]
    }
  }
}
```

3. If this transaction is included before the legitimate deployer's `init`, the contract enters `Running` state with the attacker as the sole participant.
4. The attacker's node calls `vote_add_domains` (threshold = 1, so one vote suffices), then `start_keygen_instance` / `vote_pk` to generate a keyset under their control.
5. Any user calling `sign(...)` on the contract will have their request fulfilled by the attacker's node, which can produce arbitrary signatures or selectively deny service. [1](#0-0) [2](#0-1) [3](#0-2)
