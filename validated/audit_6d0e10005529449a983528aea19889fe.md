## Vulnerability Analysis

I need to trace the full validation chain for P1's Paillier modulus N in the ECDSA-2PC DKG, then determine whether a three-prime N = p·q·r can pass all checks.

### Step 1: `create_pub(N)` validation [1](#0-0) 

`is_valid_modulus` checks only: `m > 1 && m.get_bits_count() <= MAX_MODULUS_BITS && m.is_odd()`. No primality or factor-count check. [2](#0-1) 

`create_pub` adds only a bit-size upper bound. **No two-prime structure check.**

### Step 2: `valid_paillier_interactive_t::verify()` validation [3](#0-2) 

The verifier runs two checks:
1. `check_integer_with_small_primes(N, param::alpha)` — rejects N divisible by any prime ≤ alpha = 2^13 = 8192.
2. `sigma[i].pow_mod(N, N) != rho[i]` — the N-th root proof: verifies σᵢᴺ ≡ ρᵢ (mod N). [4](#0-3) [5](#0-4) 

**No two-prime structure check anywhere in the chain.**

### Step 3: Can a three-prime N satisfy the N-th root proof?

The proof requires: σᵢ = ρᵢ^{N⁻¹ mod φ(N)} mod N, so that σᵢᴺ = ρᵢ^{N·N⁻¹} = ρᵢ¹ mod N.

For N = p·q·r with all three primes large (> 8192):
- φ(N) = (p−1)(q−1)(r−1)
- gcd(N, φ(N)) = 1 (since p∤(p−1), q∤(q−1), r∤(r−1) for large distinct primes)
- Therefore N⁻¹ mod φ(N) **exists**, and the N-th root proof is satisfiable.

Byzantine P1 computes φ(N) = (p−1)(q−1)(r−1) directly (they know all three factors), computes N_inv = N⁻¹ mod φ(N), and crafts valid σᵢ values. The production `paillier_t::prove()` uses `paillier.get_phi_N()` which only stores (p−1)(q−1), but P1 is Byzantine and can craft the `prover_msg_t` sigma array directly without using the production prover. [6](#0-5) 

### Step 4: DKG accepts the three-prime key [7](#0-6) 

`step4_p2_output` calls `create_pub(N)`, `verify_cipher(c_key)`, `valid.verify(...)`, `pdl.verify(...)`, `equal.verify(...)`, `range.verify(...)`. None of these check for two-prime structure. All pass for a three-prime N with large factors.

### Step 5: Critical impact during signing

During signing, P2 encrypts under P1's Paillier key N: [8](#0-7) 

P2 sends `c_tag = Enc_N(k₂⁻¹·m + k₂⁻¹·x₂·r + ρ·q)`. Since P1 knows the factorization p, q, r of N, they can decrypt c_tag to recover `v = k₂⁻¹·(m + x₂·r) mod q`.

Combined with the public signature `s = (k₁·k₂)⁻¹·(m + (x₁+x₂)·r) mod q` and P1's known values (k₁, x₁, m, r, s):

- `s·k₁ − v = k₂⁻¹·x₁·r mod q` (non-zero with overwhelming probability)
- `C = v·x₁·r·(s·k₁ − v)⁻¹ mod q = m + x₂·r mod q`
- `x₂ = (C − m)·r⁻¹ mod q`

P1 recovers x₂ from **a single signing session**, then computes the full private key x = x₁ + x₂ mod q.

### Title
Missing Two-Prime Structure Check in Paillier Key Validation Allows Byzantine P1 to Recover Full ECDSA Private Key — (`src/cbmpc/protocol/ecdsa_2p.cpp`, `src/cbmpc/zk/zk_paillier.cpp`)

### Summary
`valid_paillier_interactive_t::verify()` and `create_pub()` do not verify that N is a product of exactly two primes. A Byzantine P1 can supply N = p·q·r (three large primes), craft a valid N-th root proof using φ(N) = (p−1)(q−1)(r−1), pass all DKG validation, and then decrypt P2's signing ciphertext to recover P2's secret share in a single signing session.

### Finding Description
The validation chain in `step4_p2_output` (`src/cbmpc/protocol/ecdsa_2p.cpp:54–83`) calls:
1. `paillier.create_pub(N)` — checks only: odd, ≤ 2048 bits.
2. `valid.verify(paillier, prover_pid, valid_m2)` — checks only: no small prime factors (≤ 8192), N-th root proof, coprimality of ρ-product.

For N = p·q·r with all three primes > 8192, all checks pass. The N-th root proof is satisfiable because gcd(N, φ(N)) = 1 for large distinct primes, so N⁻¹ mod φ(N) exists and σᵢ = ρᵢ^{N⁻¹} satisfies σᵢᴺ ≡ ρᵢ (mod N). Byzantine P1 computes φ(N) = (p−1)(q−1)(r−1) directly and crafts the `prover_msg_t::sigma` array without using the production prover (which would use the wrong φ(N) = (p−1)(q−1)).

### Impact Explanation
After DKG completes, P2 holds a key blob with P1's three-prime Paillier key. During signing, P2 encrypts `k₂⁻¹·(m + x₂·r) mod q` (plus a blinding term ρ·q) under N. P1 decrypts using the known factorization, extracts `v = k₂⁻¹·(m + x₂·r) mod q`, and solves for x₂ using the public signature and P1's own nonce k₁. This recovers P2's secret share x₂ and thus the full ECDSA private key x = x₁ + x₂ mod q from a single signing session.

### Likelihood Explanation
The attack requires P1 to: (1) generate three large primes summing to 2048 bits, (2) compute φ(N) and N⁻¹ mod φ(N), (3) craft sigma values matching the DRBG-derived challenges. All steps are straightforward bignum operations. No threshold collusion is required — P1 alone is sufficient.

### Recommendation
Add an explicit two-prime structure check in `valid_paillier_interactive_t::verify()` (and/or `create_pub`). The standard approach is to verify that N has exactly two prime factors, e.g., by requiring the prover to supply a proof that N = p·q with p, q prime (e.g., via a Miller-Rabin primality proof on p and q, or by extending the N-th root proof to bind the factor count). Alternatively, require the prover to supply p and q and verify p·q = N and both are prime.

error_t valid_paillier_interactive_t::verify(const crypto::paillier_t& paillier, const crypto::mpc_pid_t& prover_pid,
                                             const prover_msg_t& prover_msg) {
  crypto::vartime_scope_t vartime_scope;

  error_t rv = UNINITIALIZED_ERROR;
  const mod_t& N = paillier.get_N();
  buf128_t k = crypto::ro::hash_string(kV, N, prover_pid).bitlen128();
  crypto::drbg_aes_ctr_t drbg(k);

  if (N <= 0) return coinbase::error(E_CRYPTO);
  if (paillier_no_small_factors == zk_flag::unverified) {
    if (rv = check_integer_with_small_primes(N, param::alpha)) return rv;
    paillier_no_small_factors = zk_flag::verified;
  }

  bn_t rho_prod = 1;
  for (int i = 0; i < param::t; i++) {
    bn_t rho = drbg.gen_bn(N);
    MODULO(N) rho_prod *= rho;

    if (prover_msg.sigma[i] <= 0) return coinbase::error(E_CRYPTO);
    if (prover_msg.sigma[i].pow_mod(N, N) != rho) return coinbase::error(E_CRYPTO);
  }

  if (!mod_t::coprime(rho_prod, N)) return coinbase::error(E_CRYPTO);
  paillier_valid_key = zk_flag::verified;
  return SUCCESS;
}

                                                    const bn_t& c_key, const crypto::mpc_pid_t& prover_pid, mem_t sid) {
  error_t rv = UNINITIALIZED_ERROR;
  ecurve_t curve = Q1.get_curve();
  const mod_t& q = curve.order();
  const int N_bits = N.get_bits_count();
  if (N_bits < crypto::paillier_t::bit_size) return coinbase::error(E_CRYPTO);
  if (N_bits > crypto::paillier_t::bit_size)
    return coinbase::error(E_CRYPTO, "unsupported Paillier modulus size from counterparty");
  if (N_bits < 3 * q.get_bits_count() + 3 * SEC_P_STAT + SEC_P_COM + 1)
    return coinbase::error(E_CRYPTO, "length of N < 3lg q+ 3 stat-sec-param + com-sec-param + 1");
  if (rv = paillier.create_pub(N)) return coinbase::error(E_CRYPTO, "invalid Paillier modulus from counterparty");

  // Potential optimization: both `verify_cipher` and pdl.verify perform GCDs. These can be merged into a single GCD by
  // multiplying them together. See the notes in the spec.
  if (rv = paillier.verify_cipher(c_key)) return rv;

  if (rv = valid.verify(paillier, prover_pid, valid_m2)) return rv;

  pdl.paillier_valid_key = valid.paillier_valid_key;
  pdl.paillier_no_small_factors = valid.paillier_no_small_factors;
  pdl.paillier_range_exp_slack_proof = zk::zk_flag::skip;
  if (rv = pdl.verify(c_key, paillier, Q1, sid, 0)) return rv;

  equal.paillier_valid_key = valid.paillier_valid_key;
  equal.paillier_no_small_factors = valid.paillier_no_small_factors;
  if (rv = equal.verify(paillier, c_key, q, Com)) return rv;
  if (rv = range.verify(Com, q)) return rv;
  return SUCCESS;
}
