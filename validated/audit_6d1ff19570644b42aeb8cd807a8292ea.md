Audit Report

## Title
Synthetic Ratio Precision Loss in `_getBidAndAskPrice` Silently Produces Zero Mid, Permanently Blocking Pool Swaps — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary
In `AnchoredPriceProvider._getBidAndAskPrice()`, the synthetic ratio `Math.mulDiv(mid, ORACLE_DECIMALS, mid2)` floors to zero when the base feed price is less than `1e-8` of the quote feed price. No post-division zero-check exists, so `_computeBidAsk(0, spreadBps)` is called, which returns the stall sentinel `(0, type(uint128).max)`. `getBidAndAskPrice()` then reverts with `FeedStalled`, and the pool re-throws as `PriceProviderFailed`, completely blocking all swaps for as long as the price ratio remains extreme.

## Finding Description
In the two-feed synthetic mode (`quoteFeedId != bytes32(0)`), `_getBidAndAskPrice()` computes the ratio at line 267:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);  // ORACLE_DECIMALS = 1e8
```

Integer division floors to zero whenever `mid * 1e8 < mid2`. There is no guard for `mid == 0` after this line; execution falls through to `_computeBidAsk(0, spreadBps)` at line 271.

Inside `_computeBidAsk`, `_bandEdge(0, edgeFactor, rounding)` evaluates to `Math.mulDiv(0, Q64 * edgeFactor, STEP_DENOM, rounding) = 0`, so `refBid == 0` at line 311 triggers the stall sentinel return `(0, type(uint128).max)`.

`getBidAndAskPrice()` checks `if (bid == 0 || ask == type(uint128).max) revert FeedStalled()` at line 216. The pool's `_getBidAndAskPriceX64()` catches this revert and re-throws as `PriceProviderFailed`. Since `swap()` calls `_getBidAndAskPriceX64()` as its very first action (line 228), every swap reverts for the duration of the extreme price ratio.

The existing guard at line 265 (`if (!ok2 || mid2 == 0)`) only protects against a zero denominator before the division, not against a zero result after it.

## Impact Explanation
All swaps on any pool using `AnchoredPriceProvider` in synthetic ratio mode are completely and permanently blocked whenever the base/quote price ratio falls below `1e-8`. This is broken core pool functionality: the primary user-facing action (swap) becomes unusable, LPs cannot rebalance through swaps, and traders cannot execute. This satisfies the "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows" impact gate.

## Likelihood Explanation
The condition `mid * 1e8 < mid2` is reachable without any privileged action through normal market price movement. Example: base = SHIB at $0.00001 → `mid = 1000`; quote = BTC at $60,000 → `mid2 = 6_000_000_000_000`. Then `Math.mulDiv(1000, 1e8, 6e12) = 0`. Any pool using the synthetic ratio mode with a sufficiently asymmetric pair (e.g., micro-cap token vs. BTC) is vulnerable. No attacker input is required — the condition arises from valid oracle data reflecting real market prices.

## Recommendation
Add a zero-check for the synthetic ratio result immediately after the division at line 267:

```solidity
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);
if (mid == 0) return (0, type(uint128).max); // ratio underflows 8-decimal precision
```

This fails closed (same behavior as a stale feed) rather than silently producing a zero price that propagates through `_computeBidAsk`.
