Audit Report

## Title
Pool Admin Permanently Locks Protocol Fees via Reverting `adminFeeDestination` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`setPoolAdminFeeDestination` accepts any non-zero address with no timelock or capability check. If the pool admin sets a reverting contract as the destination, every call to `collectFees` reverts before the protocol leg executes, permanently locking all accrued protocol and admin fees inside the pool. Because `setPoolProtocolFee` calls `collectFees` as its first step, the factory owner also loses the ability to adjust protocol fee rates on the affected pool.

## Finding Description
`setPoolAdminFeeDestination` stores any non-zero address immediately with no timelock and no validation that the address can receive tokens:

```solidity
// MetricOmmPoolFactory.sol:438-447
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
``` [1](#0-0) 

Inside `MetricOmmPool.collectFees`, the admin transfer executes **before** the protocol transfer. `transferToken0/1` use `safeTransfer`, which propagates any revert from the destination:

```solidity
// MetricOmmPool.sol:416-430
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // reverts here
}
...
if (totalFee0ToProtocol > 0) {
    transferToken0(FACTORY, totalFee0ToProtocol);             // never reached
}
notionalFeeToken0Scaled = 0;
notionalFeeToken1Scaled = 0;
``` [2](#0-1) [3](#0-2) 

Every factory entry-point that collects fees passes `poolAdminFeeDestination[pool]` unchanged and has no alternative path:

- `collectPoolFees` (permissionless) — lines 379–389
- `setPoolProtocolFee` (`onlyOwner`) — lines 327–335
- `setPoolAdminFees` (`onlyPoolAdmin`) — lines 418–425 [4](#0-3) [5](#0-4) 

Because `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` are cleared **after** the transfers, a revert leaves the accumulators intact but permanently unclaimable — there is no separate protocol-only collection path and no factory override that supplies an alternative destination.

`createPool` is permissionless; any caller can deploy a pool and hold the pool admin role. The attack therefore requires only a single post-deployment transaction from the pool admin with no preconditions.

## Impact Explanation
All accrued spread fees (held as pool balance surplus) and notional fees (`notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`) are permanently locked inside the pool with no recovery mechanism. The factory owner loses the ability to adjust protocol fee rates on the affected pool because `setPoolProtocolFee` calls `collectFees` as its first step and will always revert. This is a direct, irrecoverable loss of protocol fee revenue and a broken core governance function, satisfying the "Admin-boundary break: pool admin exceeds caps" and "Critical/High direct loss of protocol fees" allowed impacts. The pool admin is semi-trusted and constrained by caps and timelocks on other actions, but `setPoolAdminFeeDestination` has neither, allowing the admin to affect protocol-owned revenue that is outside their intended authority.

## Likelihood Explanation
`createPool` is permissionless — any address can deploy a pool and become its admin. The attack requires a single call to `setPoolAdminFeeDestination` with a reverting contract address, executable at any time after pool creation. No preconditions beyond holding the pool admin role are needed. The only on-chain signal is the `PoolAdminFeeDestinationUpdated` event, which is easy to miss before fees accumulate. A malicious or compromised pool admin can execute this silently and irreversibly.

## Recommendation
Separate the admin and protocol fee transfers so a failure in the admin leg cannot block the protocol leg. Concretely:

1. **Split `collectFees` into two independent functions** — one for the protocol share (callable by the factory regardless of admin destination state) and one for the admin share — so the protocol can always collect its portion.
2. **Alternatively**, wrap the admin transfer in a `try/catch` and emit an event on failure, leaving admin fees claimable via a pull pattern while always completing the protocol transfer.
3. At minimum, add a timelock to `setPoolAdminFeeDestination` (consistent with the price-provider rotation pattern already used in the codebase) so the factory owner has a window to react before a malicious destination takes effect.

## Proof of Concept

1. Deploy a `RevertOnReceive` contract whose fallback/`transfer` hook reverts unconditionally.
2. Any address calls `factory.createPool(...)` with themselves as `admin`.
3. Pool admin calls `factory.setPoolAdminFeeDestination(pool, address(revertOnReceive))`.
4. Swaps accumulate spread surplus and `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`.
5. Anyone calls `factory.collectPoolFees(pool)` → `pool.collectFees(...)` → `transferToken0(revertOnReceive, ...)` → **revert**. Protocol receives nothing.
6. Factory owner calls `factory.setPoolProtocolFee(pool, newFee, 0)` → same revert path → **cannot update fee rates**.
7. Fees remain locked in the pool indefinitely with no recovery mechanism available to the protocol.
