### Title
`verify_foreign_transaction` Uses All-Participant Intersection Rule Instead of Signing-Threshold Rule, Enabling Single-Participant DoS of Foreign Chain Verification - (File: `crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which requires **every** active participant to have registered a chain (strict intersection). The design intent — documented explicitly and implemented in `get_available_foreign_chains()` — is to require only `signing_threshold` participants. A single Byzantine participant below the signing threshold can permanently block all foreign chain verification requests by simply not registering any chains.

### Finding Description

`verify_foreign_transaction` calls `get_supported_foreign_chains()` to check whether the requested chain is serviceable:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...)
}
```

`get_supported_foreign_chains()` computes the **strict intersection** of all active participants' registered chains — a chain is included only if **every** active participant has registered it:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain { Some(foreign_chain) } else { None }
```

The design document and the separately implemented `get_available_foreign_chains()` / `recompute_available_foreign_chains()` use a different, correct rule: a chain is available if **at least `signing_threshold`** active participants cover it. The contract even caches this threshold-based set and exposes it via `get_available_foreign_chains()`, but `verify_foreign_transaction` never consults it.

The design document states explicitly:

> `verify_foreign_transaction(C)` is **rejected unless `C` is available** — the contract fails fast instead of accepting a request that can't reach the signing threshold.

The implementation contradicts this: it uses the legacy all-participant intersection rule, not the threshold-based available set.

### Impact Explanation

A single Byzantine participant (strictly below the signing threshold) can block **all** foreign chain verification requests by not calling `register_foreign_chain_support` (or calling it with an empty set). Because `get_supported_foreign_chains()` requires every active participant to be in the supporter set for a chain, one absent registration empties the entire supported-chain set. Every subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported`, regardless of how many honest participants have registered the chain.

Additionally, as the system migrates to the new `register_foreign_chains_config` path (which writes to `foreign_chains`, not `node_foreign_chain_support`), `get_supported_foreign_chains()` will return an empty set for all chains, permanently breaking `verify_foreign_transaction` for all callers.

This breaks the request-lifecycle invariant: foreign chain verification requests are rejected even when a threshold of participants can service them, violating the core safety property of the threshold design.

### Likelihood Explanation

The attacker is a single MPC participant (below threshold). No collusion, no privileged access, no key material is required. The participant simply omits calling `register_foreign_chain_support`. This is reachable in production: any one of the `n` participants can trigger it unilaterally. The system is also naturally exposed during any migration to the new registration path.

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which already implements the correct threshold-based rule. The available-chain cache is maintained by `recompute_available_foreign_chains()` and is updated on every `register_foreign_chains_config` call, so no additional computation is needed at request time.

### Proof of Concept

Setup: 4 participants, signing threshold 3, Bitcoin whitelisted. Participants 0, 1, 2 register Bitcoin via `register_foreign_chain_support`. Participant 3 (Byzantine, below threshold) registers nothing.

- `get_available_foreign_chains()` → `{Bitcoin}` (3 ≥ threshold of 3) ✓
- `get_supported_foreign_chains()` → `{}` (participant 3 absent from supporter set) ✗
- `verify_foreign_transaction(Bitcoin)` → panics with `ForeignChainNotSupported`

The existing unit test `get_available_foreign_chains__should_include_chain_when_at_least_threshold_participants_cover_it` confirms that 3-of-4 coverage is sufficient for availability, yet `verify_foreign_transaction` would reject the same request because it uses the stricter intersection rule.

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
