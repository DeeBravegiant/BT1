### Title
Missing `payload_hash`-to-`request` Binding Verification in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Byzantine Participant — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function verifies that the submitted ECDSA signature is valid over `response.payload_hash` using the domain's root public key, but never verifies that `response.payload_hash` is actually the hash of a `ForeignTxSignPayload` whose embedded `request` field matches the `request` argument. A single malicious MPC participant below the signing threshold can therefore recycle a legitimately-produced threshold signature from a previously completed request B and submit it as the response to a different pending request A. The contract accepts the call, resolves all queued yields for request A with the wrong payload hash, and delivers the forged response to every caller waiting on request A.

---

### Finding Description

`respond_verify_foreign_tx` (lines 691–754 of `crates/contract/src/lib.rs`) performs four checks before resolving pending yields:

1. Caller is an attested participant.
2. Protocol is running or resharing.
3. `accept_requests` is true.
4. `verify_ecdsa_signature(response.signature, response.payload_hash, root_public_key)` succeeds. [1](#0-0) 

It then calls `pending_requests::resolve_yields_for` with the raw `response` bytes: [2](#0-1) 

**What is absent**: there is no check that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: <anything> })))`. The contract knows `request.request` at call time but never uses it to constrain the hash.

The signed payload type is: [3](#0-2) 

Because `request` is embedded inside the Borsh-serialised payload, `payload_hash_B = SHA-256(borsh(ForeignTxSignPayload { request: request_B, values: values_B }))` is cryptographically bound to `request_B`, not to `request_A`. The contract never enforces this binding.

Contrast this with the regular `respond` path, where the payload is taken directly from `request.payload` (a field the contract already holds), so the signature is always verified over the correct, request-specific data: [4](#0-3) 

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does perform the binding check: [5](#0-4) 

But this check lives in the **caller's** contract, not in the MPC contract itself. The MPC contract delivers the forged response unconditionally; whether the downstream contract catches it depends entirely on whether it uses the SDK correctly.

---

### Impact Explanation

**Attack path (single Byzantine participant, below threshold):**

1. The MPC network legitimately processes request B for some user → produces `response_B` with a valid threshold signature over `payload_hash_B`.
2. The malicious participant observes `response_B` (it is broadcast to all nodes during the signing protocol).
3. A victim submits `verify_foreign_transaction(request_A)` → a pending yield is queued under `request_A`.
4. The malicious participant calls `respond_verify_foreign_tx(request_A, response_B)`.
5. The contract passes all four checks (signature is valid over `payload_hash_B` using the root key; `request_A` is pending) and resolves every queued yield for `request_A` with `response_B`.
6. Every caller waiting on `request_A` receives `response_B`.

**Downstream consequences:**

- If the caller's contract uses `ForeignChainSignatureVerifier` correctly, it rejects the response (hash mismatch) → **DoS / forced transaction failure** for every user queued under `request_A`, including fan-out duplicates from different callers.
- If the caller's contract (e.g., a bridge) only checks that the signature is valid over the returned `payload_hash` without re-deriving the expected hash from its own `request_A`, it accepts `payload_hash_B` as proof of a foreign-chain event that was never observed for `request_A` → **forged foreign-chain verification**, enabling invalid bridge execution (e.g., minting tokens on NEAR for a deposit that was never made on the foreign chain, or for a deposit made to a different address/amount encoded in `request_B`).

This matches the **High** allowed impact: *"forged foreign-chain verification … that causes invalid bridge execution."*

---

### Likelihood Explanation

**Medium.** The preconditions are:

- One Byzantine MPC participant below the signing threshold (the system explicitly tolerates up to `threshold − 1` malicious nodes).
- The MPC network must have previously produced at least one valid `response_B` for any request B on the same domain — a condition satisfied during normal operation.
- A victim must have a pending `request_A` on the same domain.

No key material needs to be leaked, no threshold collusion is required, and no network-level attack is needed. The malicious participant simply reuses a legitimately-produced threshold signature in a different call.

---

### Recommendation

The MPC contract must bind the accepted `payload_hash` to the `request` it is resolving. Two complementary approaches:

1. **Protocol-level fix (preferred):** Change the signing payload so that the MPC nodes sign `SHA-256(borsh(request) ‖ SHA-256(borsh(values)))` in two separable parts, and have the contract verify the first part against `request.request`. This makes the binding verifiable on-chain without knowing `values`.

2. **Contract-level partial fix:** Require the caller to also supply the `values` vector alongside the response. The contract recomputes `expected_hash = SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values })))` and asserts `expected_hash == response.payload_hash` before accepting the call. This adds on-chain gas cost but closes the gap entirely.

In `respond_verify_foreign_tx`, the check should be inserted immediately after the existing signature verification:

```rust
// Verify payload_hash is bound to this specific request
let expected_prefix_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: supplied_values,   // new parameter
}).compute_msg_hash()?;
if expected_prefix_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```
