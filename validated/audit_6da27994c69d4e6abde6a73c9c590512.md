### Title
Error State Overwritten by Subsequent Validation Check in `validate_delegate_action_key` — (File: `runtime/runtime/src/actions.rs`)

### Summary
In `validate_delegate_action_key`, when a `DelegateAction` signed with a `FunctionCallPermission` access key carries a nonzero deposit **and** a mismatched receiver, the `DepositWithFunctionCall` error written to `result.result` is silently overwritten by the subsequent `ReceiverMismatch` check. The final execution outcome records the wrong error variant, corrupting the outcome root committed in the chunk header.

### Finding Description
`validate_delegate_action_key` in `runtime/runtime/src/actions.rs` validates the access key used to sign a `DelegateAction`. When the key has `FunctionCallPermission`, the function checks (in order): deposit > 0, receiver mismatch, method name mismatch.

Before `ProtocolFeature::FixDelegateActionDepositWithFunctionCallError` (protocol version 85), the deposit check sets `result.result = Err(DepositWithFunctionCall)` but does **not** return early:

```rust
if function_call.deposit > Balance::ZERO {
    result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
        InvalidAccessKeyError::DepositWithFunctionCall,
    ).into());
    // No early return here before the fix
}
if delegate_action.receiver_id() != &function_call_permission.receiver_id {
    result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
        InvalidAccessKeyError::ReceiverMismatch { ... },
    ).into());
    return Ok(());
}
```

When both conditions hold, the `ReceiverMismatch` assignment at line 652 overwrites the `DepositWithFunctionCall` error set at line 638. The fix — a conditional early return gated on the protocol feature — is present but only activates at version ≥ 85. [1](#0-0) 

The protocol feature and its version assignment: [2](#0-1) [3](#0-2) 

### Impact Explanation
The execution outcome for the delegate action receipt records the wrong `ActionError` variant (`ReceiverMismatch` instead of `DepositWithFunctionCall`). Because outcomes are content-addressed and committed via the **outcome root** in the chunk header, a different error variant produces a different outcome hash and therefore a different outcome root. Any node replaying the chunk at protocol version < 85 computes a different outcome root than a node at version ≥ 85, constituting a **consensus divergence** on the outcome root — a concrete protocol value.

The action itself fails in both cases (no balance is incorrectly transferred), but the committed outcome root is wrong for the pre-fix version, breaking the invariant that the outcome root is deterministic and matches the first validation failure encountered.

### Likelihood Explanation
Triggering the bug requires a `DelegateAction` (meta-transaction) where:
1. The signing key has `FunctionCallPermission`.
2. The inner `FunctionCall` action has `deposit > 0`.
3. The `receiver_id` in the delegate action does not match the access key's `receiver_id`.

Any unprivileged user can craft and submit such a transaction via the public RPC. The bug is active on nodes running protocol version < 85. Since the fix is at version 85 (the current stable version), the live network is protected, but the old code path is preserved for historical replay.

### Recommendation
The fix is already applied: `FixDelegateActionDepositWithFunctionCallError` adds an early `return Ok(())` after setting the `DepositWithFunctionCall` error, preventing the subsequent receiver check from overwriting it. [4](#0-3) 

The pattern to enforce going forward: whenever `result.result` is set to an error inside a multi-check validation function, always return immediately to prevent later checks from silently overwriting the first error.

### Proof of Concept
1. Create account `alice` with a `FunctionCallPermission` access key restricted to `receiver = bob.near`.
2. Construct a `DelegateAction` signed by that key where the inner `FunctionCall` has `deposit = 1 yoctoNEAR` and `receiver_id = carol.near` (mismatches both the deposit rule and the receiver).
3. Submit via `broadcast_tx_async` on a node running protocol version < 85.
4. Observe the execution outcome: the recorded error is `ReceiverMismatch` instead of `DepositWithFunctionCall`.
5. The outcome hash differs from what a version-85 node would compute for the same receipt, producing a different outcome root in the chunk header — confirmed by the test `test_delegate_deposit_with_function_call_reports_receiver_mismatch_before_fix`. [5](#0-4)
