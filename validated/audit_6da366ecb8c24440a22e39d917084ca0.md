### Title
Cross-Request Response Injection in `respond_verify_foreign_tx` — No Payload-to-Request Binding Check - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` accepts two independent parameters — a `VerifyForeignTransactionRequest` (used as the pending-queue map key) and a `VerifyForeignTransactionResponse` (containing `payload_hash` and `signature`) — but never verifies that `response.payload_hash` was derived from `request.request` (the actual foreign-chain transaction). A single Byzantine attested participant who holds any previously-produced valid root-key signature can inject that signature as the response to a completely different pending request, causing users to receive a fraudulent foreign-chain verification attestation.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs two independent checks:

1. **Signature validity**: `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's **root** public key (no derivation/tweak).
2. **Queue lookup**: `request` exists as a key in `pending_verify_foreign_tx_requests`. [1](#0-0) 

Neither check binds `response.payload_hash` to `request.request` (the `ForeignChainRpcRequest` containing the specific `tx_id` the user submitted). The contract has no assertion of the form:

```
expected_hash = hash(request.request, extracted_values)
assert response.payload_hash == expected_hash
```

The pending-request map is keyed on the full `VerifyForeignTransactionRequest` struct, so the queue lookup succeeds for any request that was previously submitted, regardless of what `payload_hash` the response carries. [2](#0-1) 

This is structurally identical to the reported `removeVault(address vault_, bytes32 vaultId_)` bug: two parameters that must correspond are accepted without cross-validation, so the wrong entity (here: the wrong pending request) is resolved with wrong data.

---

### Impact Explanation

A user submits `verify_foreign_transaction` for Bitcoin tx **T2** and waits for a yield-resume response. A Byzantine attested participant who previously participated in a signing round for a different request **R1** (covering tx **T1**) holds a valid root-key signature **S1** over `hash(T1)`. They call:

```
respond_verify_foreign_tx(
    request = R2,          // pending queue entry for T2
    response = { payload_hash: hash(T1), signature: S1 }
)
```

The contract:
- Verifies S1 over `hash(T1)` with the root key → **PASS** (signature is mathematically valid)
- Finds R2 in `pending_verify_foreign_tx_requests` → **FOUND**
- Drains R2's yield queue, delivering `{ payload_hash: hash(T1), signature: S1 }` to the user

The user receives a signed attestation that **T1** was verified, not T2. If the downstream bridge protocol trusts the contract's response without independently re-deriving the expected payload hash from the original transaction, this constitutes a **forged foreign-chain verification** that can enable invalid bridge execution or double-spend conditions.

This maps to the allowed impact: **High — forged foreign-chain verification / participant authorization bypass that causes invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

- The attacker must be a single **attested participant** (below signing threshold — no collusion required).
- They need one valid root-key signature over any payload hash, which they naturally accumulate by participating in any prior `verify_foreign_tx` signing round as leader or follower.
- `respond_verify_foreign_tx` has no caller restriction beyond `assert_caller_is_attested_participant_and_protocol_active`; any attested node may call it at any time with any request/response pair. [3](#0-2) 

In a production environment processing many bridge requests, a Byzantine participant will have accumulated multiple valid signatures and can target any concurrently-pending request.

---

### Recommendation

Before calling `resolve_yields_for`, recompute the expected payload hash from the request and compare it to `response.payload_hash`. Because the contract cannot independently query the foreign chain, the response should include the extracted values so the contract can verify:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, include a cryptographic commitment to the request (e.g., hash of the `VerifyForeignTransactionRequest`) inside the signed payload so the signature itself binds the response to the specific request.

---

### Proof of Concept

1. Alice submits `verify_foreign_transaction` for Bitcoin tx **T2** → pending entry R2 is created.
2. Byzantine node **B** is the leader for a prior signing round for request R1 (tx T1). After the round completes, B holds valid signature **S1** over `hash(T1)` under the root key.
3. B calls:
   ```
   respond_verify_foreign_tx(
       request  = R2,
       response = { payload_hash: hash(T1), signature: S1 }
   )
   ```
4. Contract verifies S1 over `hash(T1)` → valid. Finds R2 in map → found. Drains R2's yield queue.
5. Alice's transaction resumes with `{ payload_hash: hash(T1), signature: S1 }` — a signed attestation for T1, not T2.
6. Alice (or a bridge contract) uses this attestation to claim funds for T2, which was never actually verified. [4](#0-3) [2](#0-1)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
