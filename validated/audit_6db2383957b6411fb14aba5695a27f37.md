### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Enables Fan-Out Queue Exhaustion, Blocking Foreign Transaction Verifications — (File: `crates/contract/src/pending_requests.rs`)

---

### Summary

The `verify_foreign_transaction` contract method stores pending requests under a key that contains **no caller identity**. Combined with the hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` entries per key, any unprivileged caller can fill the queue for a specific foreign transaction, causing every subsequent legitimate submission for that same transaction to revert with `PendingRequestQueueFull`. The attacker can continuously refill the queue after it drains, sustaining the denial of service at near-zero cost.

---

### Finding Description

**Root cause — caller-agnostic request key**

`args_into_verify_foreign_tx_request` constructs the map key from only `domain_id`, `request` (chain + tx_id + extractors), and `payload_version`:

```rust
// crates/contract/src/dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

No `predecessor_account_id` is included. The `VerifyForeignTransactionRequest` struct itself confirms this:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs:124-128
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

This is explicitly confirmed by the unit test comment:

> "both yields are queued under the single (caller-agnostic) request key"

This contrasts with `sign()`, where `SignatureRequest::new` takes `&predecessor` as a parameter, making each caller's request key unique.

**Bounded queue cap**

`push_pending_yield` enforces a hard cap and panics (reverting the transaction) when the queue is full:

```rust
// crates/contract/src/pending_requests.rs:51-58
if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull {
            limit: MAX_PENDING_REQUEST_FAN_OUT,
        }
        .to_string(),
    );
}
```

`MAX_PENDING_REQUEST_FAN_OUT = 128`.

**Attack path**

1. Attacker submits 128 `verify_foreign_transaction` calls for a specific foreign transaction (same `domain_id`, `request`, `payload_version`). The minimum deposit is `MINIMUM_SIGN_REQUEST_DEPOSIT = 1 yoctoNEAR` — essentially free.
2. The queue for that request key is now full.
3. Any legitimate user submitting the same foreign transaction verification receives `PendingRequestQueueFull` and their transaction reverts.
4. MPC nodes process the 128 attacker requests and call `respond_verify_foreign_tx`, draining the queue.
5. Attacker immediately submits 128 more requests.
6. Steps 3–5 repeat indefinitely.

The attacker can preemptively fill the queue for any known foreign transaction (e.g., a specific Bitcoin `tx_id`) before legitimate bridge users submit their requests.

---

### Impact Explanation

This breaks the request-lifecycle for specific foreign transaction verifications. Bridge operations for targeted foreign transactions can be blocked indefinitely. Legitimate users pay gas for transactions that revert. In a bridge context, this can strand user funds on the foreign chain if the verification window is time-sensitive (e.g., confirmation-count requirements). The impact maps to "request-lifecycle manipulation that breaks production safety/accounting invariants."

---

### Likelihood Explanation

Any unprivileged NEAR account can execute this attack. The cost per 128-request cycle is approximately 128 × (gas cost per transaction) + 128 yoctoNEAR deposit — negligible on NEAR's L1. The attacker can monitor the NEAR blockchain for `verify_foreign_transaction` receipts and front-run or preemptively target specific foreign transactions. No privileged access, TEE bypass, or threshold collusion is required.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring the approach used by `sign()` where `SignatureRequest::new` takes `&predecessor`. This makes each caller's queue slot independent, preventing cross-caller queue exhaustion. If fan-out across callers for the same foreign transaction is a desired feature, add a per-submission nonce or timestamp to the key so each submission occupies a distinct slot.

---

### Proof of Concept

```rust
// Fill the queue for a specific Bitcoin transaction
for _ in 0..128 {
    contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
        domain_id: foreign_tx_domain_id,
        payload_version: ForeignTxPayloadVersion::V1,
        request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
            tx_id: TARGET_TX_ID.into(),
            confirmations: 1.into(),
            extractors: vec![BitcoinExtractor::BlockHash],
        }),
    });
}

// Any subsequent legitimate call for the same tx now panics:
// "Pending-request queue is full for this request key (limit: 128)."
contract.verify_foreign_transaction(same_args); // reverts
```

The existing unit test `add_signature_request__should_panic_when_pending_queue_is_full` already demonstrates the panic path for the signature queue; the same mechanism applies identically to `pending_verify_foreign_tx_requests` via the shared `push_pending_yield` helper.

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
