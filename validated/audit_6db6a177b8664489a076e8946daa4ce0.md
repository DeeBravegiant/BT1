Audit Report

## Title
`updatedAt` unconditionally advances when `totalKernelStaked == 0`, permanently freezing reward yield — (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
The `updateReward` modifier unconditionally sets `updatedAt = lastTimeRewardApplicable()` even when `totalKernelStaked == 0`. Because `rewardPerToken()` short-circuits and returns the stored value unchanged during a zero-supply window, the time elapsed is consumed without distributing any rewards. Any rewards that accrued during that window are permanently locked in the contract with no recovery path.

## Finding Description
The `updateReward` modifier at L232–242 always advances `updatedAt`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // unconditional
    ...
}
```

`rewardPerToken()` at L408–414 correctly short-circuits when supply is zero:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // no new rewards distributed
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

However, `updatedAt` is still advanced to `lastTimeRewardApplicable()` in the modifier regardless of supply. When a new staker later calls `stake()`, the modifier fires again: `rewardPerToken()` now computes `rewardRate * (lastTimeRewardApplicable() - updatedAt)`, but `updatedAt` already reflects the end of the zero-supply gap, so the entire `T1→T2` reward window is silently skipped.

`initiateWithdrawal()` at L320–338 allows any staker to reduce `totalKernelStaked` at any time with no on-chain guard preventing it from reaching zero during an active reward period. There is no `rescueTokens` or equivalent admin function in the contract; `notifyRewardAmount` uses `safeTransferFrom` to pull a new amount and cannot recycle the already-stuck balance.

The contract's own NatDoc at L18–22 acknowledges this behavior but relies entirely on an off-chain operational assumption ("ensuring there are always some tokens staked … for the entire duration of the reward period"), which is not enforced on-chain.

## Impact Explanation
Rewards accrued during any window where `totalKernelStaked == 0` are permanently frozen in the contract. No staker can ever claim them, and no admin function can recover them. This is a concrete, irreversible loss of yield tokens matching **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
Medium. The scenario requires all active stakers to call `initiateWithdrawal` during an active reward period. Reward periods are admin-configurable and can span weeks. Stakers have full autonomy to unstake at any time. A gradual drain to zero — or a coordinated exit — is sufficient. No privileged compromise is required; only normal user actions trigger the loss.

## Recommendation
Only advance `updatedAt` when rewards are actually being distributed (i.e., when `totalKernelStaked > 0`):

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures time elapsed while no tokens are staked is not consumed, so rewards remain claimable when stakers return.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1000e18)` with `totalKernelStaked = 100e18`. `rewardRate`, `finishAt`, and `updatedAt = T0` are set.
2. At time `T1`, all stakers call `initiateWithdrawal(all)`. The last call triggers `updateReward`: `rewardPerTokenStored` is updated correctly for `T0→T1`, `updatedAt = T1`, then `totalKernelStaked = 0`.
3. Time advances to `T2` (e.g., 7 days later). Rewards worth `rewardRate * (T2 - T1)` have accrued but cannot be distributed.
4. A new staker calls `stake(1e18)`. `updateReward` fires: `rewardPerToken()` returns `rewardPerTokenStored` unchanged (supply was 0), but `updatedAt` is set to `T2`.
5. `totalKernelStaked = 1e18`. Future rewards are calculated from `T2` onward only.
6. All rewards for the `T1→T2` window (`rewardRate * (T2 - T1)` tokens) are permanently stuck in the contract with no recovery path.

**Foundry test sketch:**
```solidity
function test_frozenRewardsDuringZeroSupply() public {
    // Setup: stake, notify rewards
    vm.prank(admin); pool.notifyRewardAmount(1000e18);
    // All stakers withdraw
    vm.prank(alice); pool.initiateWithdrawal(aliceBalance);
    // Advance time 7 days (mid-period)
    vm.warp(block.timestamp + 7 days);
    // New staker enters
    vm.prank(bob); pool.stake(1e18);
    // Advance to end of period
    vm.warp(finishAt);
    vm.prank(bob); pool.getReward();
    // Assert: bob's rewards only cover T2→finishAt, not T1→T2
    // Assert: rewardsToken.balanceOf(address(pool)) > 0 (stuck)
}
```
