Audit Report

## Title
`initiateWithdrawal()` Always Reverts Due to Uninitialized `maxNumberOfWithdrawalsPerUser` — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal()` evaluates `userWithdrawalIds[msg.sender].length >= 0`, which is unconditionally `true` for any `uint256`, causing every withdrawal attempt to revert with `WithdrawalLimitReached`. Users can stake freely but are completely unable to withdraw until an admin manually calls `setMaxNumberOfWithdrawalsPerUser()`.

## Finding Description
`initialize()` sets `kernelToken`, `rewardsToken`, and roles, but never assigns `maxNumberOfWithdrawalsPerUser`: [1](#0-0) 

The variable therefore holds its Solidity default of `0`: [2](#0-1) 

Inside `initiateWithdrawal()`, the guard at line 323 is:
```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```
With `maxNumberOfWithdrawalsPerUser == 0`, this reduces to `length >= 0`, which is always `true` for `uint256`. Every call reverts immediately. [3](#0-2) 

The admin setter explicitly rejects `0` as a value, confirming `0` is an invalid operational state — yet it is the state the contract starts in: [4](#0-3) 

Meanwhile, `stake()` has no corresponding guard and accepts deposits freely: [5](#0-4) 

This creates an asymmetry: users can deposit but cannot exit until an admin intervenes.

## Impact Explanation
Every staker is unable to call `initiateWithdrawal()` after a fresh deployment that omits the post-init `setMaxNumberOfWithdrawalsPerUser()` call. Staked principal is locked in the contract with no exit path. This constitutes **temporary freezing of funds** (Medium), as the admin can eventually unblock withdrawals by calling the setter, but users have no recourse in the interim.

## Likelihood Explanation
The contract is upgradeable and `initialize()` is the sole setup entry point. There is no on-chain enforcement requiring `setMaxNumberOfWithdrawalsPerUser()` to be called before staking opens. Any deployment — including proxy upgrades that re-initialize — that omits this post-init step leaves the pool in the broken state. The condition is triggered by any ordinary user calling `initiateWithdrawal()`, requiring no special privileges or unusual assumptions.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`:

```diff
 function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
     ...
     kernelToken = IERC20(_kernelToken);
     rewardsToken = IERC20(_rewardToken);
+    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER;
 }
```

This mirrors the existing constant `MAX_WITHDRAWALS_PER_USER = 100` already defined in the contract: [6](#0-5) 

## Proof of Concept
1. Deploy `KernelDepositPool` behind a proxy and call `initialize(admin, kernelToken, rewardToken)` — do **not** call `setMaxNumberOfWithdrawalsPerUser()`.
2. Verify: `maxNumberOfWithdrawalsPerUser == 0`.
3. User approves and calls `stake(100e18)`. Succeeds; `balanceOf[user] == 100e18`.
4. User calls `initiateWithdrawal(100e18)`.
5. Execution reaches line 323: `0 >= 0` → `true` → reverts with `WithdrawalLimitReached`.
6. User's 100 KERNEL are frozen with no exit path.
7. Admin calls `setMaxNumberOfWithdrawalsPerUser(10)`. Step 4 now succeeds — confirming the root cause is the missing initialization, not a logic error in the guard itself.
