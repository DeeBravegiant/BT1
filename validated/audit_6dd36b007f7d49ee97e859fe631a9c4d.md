Audit Report

## Title
`maxUncompletedWithdrawalCount` Cap Never Enforced in `increaseUncompletedWithdrawalCount`, Enabling Unbounded Gas Consumption in `updateRSETHPrice` - (File: `contracts/LRTUnstakingVault.sol`)

## Summary
`LRTUnstakingVault` declares `maxUncompletedWithdrawalCount` and provides `setMaxUncompletedWithdrawalCount` to cap in-flight EigenLayer withdrawals, with an explicit protocol comment acknowledging ~120 uncompleted withdrawals will break `updateRSETHPrice()`. However, `increaseUncompletedWithdrawalCount()` — the sole incrementor — contains no check against this cap, making the limit dead configuration. As `uncompletedWithdrawalCount` grows past the safe threshold through normal restaking operations, `updateRSETHPrice()` exhausts block gas on every call, permanently freezing the price oracle and halting deposits and withdrawals.

## Finding Description
The state variable and its setter exist and enforce a hard ceiling of 80:

```solidity
// LRTUnstakingVault.sol L40
uint256 public maxUncompletedWithdrawalCount;

// L150-159
function setMaxUncompletedWithdrawalCount(uint256 _maxUncompletedWithdrawalCount) external onlyLRTManager {
    // 120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price
    if (_maxUncompletedWithdrawalCount > 80) {
        revert MaxUncompletedWithdrawalCountTooHigh();
    }
    maxUncompletedWithdrawalCount = _maxUncompletedWithdrawalCount;
    emit MaxUncompletedWithdrawalCountSet(_maxUncompletedWithdrawalCount);
}
```

The incrementor contains no guard:

```solidity
// L184-186
function increaseUncompletedWithdrawalCount() external onlyLRTNodeDelegator {
    uncompletedWithdrawalCount++;
}
```

`maxUncompletedWithdrawalCount` is written and emitted but **never read** anywhere in the increment path. Every EigenLayer unstaking operation queued by any NodeDelegator increments the counter without bound.

The gas-exhaustion sink is the call chain: `LRTOracle.updateRSETHPrice()` → `_getTotalEthInProtocol()` → `ILRTDepositPool.getTotalAssetDeposits(asset)` (per supported asset) → `getAssetDistributionData()` → `INodeDelegator.getAssetUnstaking(asset)` (per NDC). `getAssetUnstaking` calls EigenLayer's `DelegationManager.getQueuedWithdrawals`, whose iteration cost scales linearly with the number of pending withdrawals per staker. With up to 10 NDCs and multiple supported assets, the total iteration count across all `getQueuedWithdrawals` calls equals `uncompletedWithdrawalCount × assets`, which grows without any on-chain brake.

## Impact Explanation
When `uncompletedWithdrawalCount` exceeds the protocol-acknowledged safe threshold (~120), every call to `updateRSETHPrice()` reverts out-of-gas. `rsETHPrice` becomes permanently stale. This directly causes:
- **Unbounded gas consumption** (Medium): `_getTotalEthInProtocol` iterates unboundedly over EigenLayer queued withdrawal state.
- **Temporary freezing of funds** (Medium): `getRsETHAmountToMint()` and `getExpectedAssetAmount()` both depend on a live `rsETHPrice`; a stale price means deposits mint at an incorrect rate and withdrawals compute incorrect redemption amounts. The downside-protection auto-pause in `_updateRsETHPrice()` also stops functioning.

## Likelihood Explanation
No malicious actor is required. Normal restaking lifecycle activity — operators queuing EigenLayer unstaking across multiple NodeDelegators and assets — accumulates `uncompletedWithdrawalCount`. With `maxNodeDelegatorLimit = 10` and multiple supported LSTs, the counter can reach the danger zone through routine operations. Because no on-chain check enforces the cap, there is no revert or signal to warn operators before the threshold is crossed. The manager's `setUncompletedWithdrawalCount()` admin escape hatch can resync the counter, but it does not prevent re-accumulation since the root cause (missing guard) remains.

## Recommendation
Enforce the cap inside `increaseUncompletedWithdrawalCount`:

```solidity
function increaseUncompletedWithdrawalCount() external onlyLRTNodeDelegator {
    if (maxUncompletedWithdrawalCount > 0 &&
        uncompletedWithdrawalCount >= maxUncompletedWithdrawalCount) {
        revert MaxUncompletedWithdrawalCountReached();
    }
    uncompletedWithdrawalCount++;
}
```

The `maxUncompletedWithdrawalCount > 0` guard preserves the ability to disable the cap (zero = unlimited) while ensuring the limit is actually applied when configured.

## Proof of Concept
1. Manager calls `setMaxUncompletedWithdrawalCount(50)`. `maxUncompletedWithdrawalCount` is stored as `50`.
2. Operators initiate 51+ EigenLayer unstaking operations across NodeDelegators. Each NodeDelegator calls `increaseUncompletedWithdrawalCount()`. No revert occurs; `uncompletedWithdrawalCount` reaches `51` and beyond.
3. Continue until `uncompletedWithdrawalCount` exceeds ~120.
4. Any caller invokes `LRTOracle.updateRSETHPrice()`. The call chain reaches `_getTotalEthInProtocol()` → `getTotalAssetDeposits(asset)` → `getAssetUnstaking(asset)` per NDC, which calls `DelegationManager.getQueuedWithdrawals` for each NDC. Total iteration cost is proportional to `uncompletedWithdrawalCount × supportedAssets`. The transaction reverts out-of-gas.
5. `rsETHPrice` is permanently stale. Deposits and withdrawals use an incorrect exchange rate; the downside-protection auto-pause cannot trigger.

**Foundry fork test plan**: Fork mainnet, deploy with 10 NDCs and 3 supported assets, call `setMaxUncompletedWithdrawalCount(50)`, simulate 130 `increaseUncompletedWithdrawalCount()` calls from NDC addresses, then call `updateRSETHPrice()` with a fixed gas limit matching the current block gas limit and assert it reverts.
