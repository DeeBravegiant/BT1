Audit Report

## Title
`_calculatePayoutAmount` min-cap silently strips accrued yield from withdrawing users — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`_calculatePayoutAmount` returns `min(expectedAssetAmount, currentReturn)`, where `expectedAssetAmount` is snapshotted at withdrawal initiation time. Because rsETH is a restaking token whose price monotonically increases with EigenLayer rewards, `currentReturn > expectedAssetAmount` is the normal post-delay state, causing the cap to fire on virtually every standard withdrawal. The user's full rsETH is burned at the stale initiation-time price, and the accrued yield delta is silently retained in the vault and redistributed to remaining rsETH holders.

## Finding Description
**Code path:**

1. `initiateWithdrawal` (L166–175): user's rsETH is transferred to the contract; `expectedAssetAmount = rsETHUnstaked * rsETHPrice / assetPrice` is snapshotted and stored in `WithdrawalRequest`.

2. Days/weeks pass (EigenLayer withdrawal delay). rsETH price rises as restaking rewards accrue.

3. Operator calls `unlockQueue` → `_unlockWithdrawalRequests` (L790–815) → `_calculatePayoutAmount` (L824–835):

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn)
    ? request.expectedAssetAmount   // cap fires; user loses appreciation
    : currentReturn;
```

4. `payoutAmount = expectedAssetAmount` (stale, lower value). `request.expectedAssetAmount` is overwritten with this value (L804), and the full `rsETHUnstaked` is queued for burning (L805).

5. `completeWithdrawal` → `_processWithdrawalCompletion` (L699–737) transfers `request.expectedAssetAmount` — the stale amount — to the user (L734).

**Why existing checks fail:** The `availableAssetAmount` guard (L800) only prevents over-payment relative to vault liquidity; it does not prevent under-payment. The natspec at L818–819 explicitly documents the `min` behavior as intentional, but this design causes the user's rsETH — held by the contract from initiation until burn — to accrue value that is never returned to the user. The delta `(currentReturn − expectedAssetAmount)` is never distributed; it inflates the share value for all remaining rsETH holders at the expense of the withdrawing user.

## Impact Explanation
Every user who initiates a standard withdrawal loses the yield that accrued on their locked rsETH during the waiting period. The lost amount is concrete and measurable: at a 4–6% annualised restaking yield and a multi-day EigenLayer delay, the gap is non-trivial on any meaningful withdrawal size. The accrued yield is not destroyed — it is redistributed to remaining rsETH holders, constituting **theft of unclaimed yield (High)** within the allowed impact scope.

## Likelihood Explanation
No attacker action is required. The condition `currentReturn > expectedAssetAmount` is the normal, expected state for any withdrawal that completes after rewards have accrued. The EigenLayer withdrawal delay is measured in days; even a modest annualised restaking yield produces a measurable gap. Every withdrawing user is affected automatically, making this a systemic, high-likelihood issue.

## Recommendation
At `unlockQueue` time, pay `currentReturn` unconditionally. The slashing-protection intent (preventing over-payment when rsETH price drops) is preserved by the downside case already handled by `currentReturn < expectedAssetAmount`:

```solidity
// corrected — always pay current fair value
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return currentReturn;
```

Alternatively, if the protocol intentionally locks in the initiation-time price, the rsETH should **not** be transferred to the contract until `completeWithdrawal` is called, so the user retains economic exposure to their rsETH during the waiting period.

## Proof of Concept
**Setup**: rsETH price = 1.05 ETH, stETH price = 1.02 ETH at initiation.

1. Alice calls `initiateWithdrawal(stETH, 1e18 rsETH)`.
   - `expectedAssetAmount = 1e18 * 1.05e18 / 1.02e18 ≈ 1.0294 stETH`
   - Alice's 1 rsETH is held by the contract.

2. Two weeks pass. rsETH price rises to 1.08 ETH (restaking rewards accrued).

3. Operator calls `unlockQueue(stETH, ...)`.
   - `currentReturn = 1e18 * 1.08e18 / 1.02e18 ≈ 1.0588 stETH`
   - `payoutAmount = min(1.0294, 1.0588) = 1.0294 stETH` ← cap fires
   - Alice's 1 rsETH is burned; she is credited 1.0294 stETH.

4. Alice calls `completeWithdrawal(stETH)` and receives **1.0294 stETH**.

5. Alice's 1 rsETH was worth **1.0588 stETH** at unlock time. She lost **~0.0294 stETH** (~2.8%) of yield. That value remains in the vault.

**Foundry test plan**: Deploy `LRTWithdrawalManager` on a local fork. Call `initiateWithdrawal` with 1e18 rsETH. Advance blocks past `withdrawalDelayBlocks`. Manually update the mock oracle to return a higher `rsETHPrice`. Call `unlockQueue` then `completeWithdrawal`. Assert that `balanceOf(alice, stETH)` equals `expectedAssetAmount` (stale) rather than `currentReturn` (fair value), confirming the yield loss.
