Audit Report

## Title
Stale Chainlink Price Feed Used Without Validation Enables Excess rsETH Minting — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validity fields and performs no checks on the returned price. A stale or zero price is silently accepted and flows directly into rsETH minting calculations. This allows a depositor to receive excess rsETH when a supported LST has depegged but its Chainlink feed has not yet updated, diluting existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at L52 discards all five return values except `price`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No staleness check (`block.timestamp - updatedAt > heartbeat`), no incomplete-round check (`answeredInRound >= roundId`), and no non-positive price check (`price > 0`) are performed.

This is in direct contrast to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which is the protocol's own Chainlink wrapper and explicitly validates all three conditions:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price flows into the deposit path: `LRTDepositPool.getRsETHAmountToMint()` computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`, where `lrtOracle.getAssetPrice(asset)` delegates directly to `ChainlinkPriceOracle.getAssetPrice()`. [3](#0-2) 

`lrtOracle.rsETHPrice()` is a stored value updated separately by `updateRSETHPrice()`. The deposit calculation does not call `updateRSETHPrice()`, so the `pricePercentageLimit` guard in `_updateRsETHPrice()` does not protect the deposit path. An inflated stale asset price in the numerator directly inflates `rsethAmountToMint` with no circuit breaker. [4](#0-3) 

The stale price also flows into `_getTotalEthInProtocol()`, which iterates all supported assets and calls `getAssetPrice(asset)` for each, inflating `totalETHInProtocol` and subsequently `newRsETHPrice`, potentially triggering excess protocol fee rsETH minting to the treasury. [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.**

When a supported LST depegs but its Chainlink feed is stale (reflecting the pre-depeg price), a depositor receives more rsETH than the deposited LST is actually worth. The excess rsETH represents value extracted from existing rsETH holders, whose proportional claim on the backing TVL is diluted. The magnitude scales with the size of the depeg and the deposit limit. This matches the allowed impact "Theft of unclaimed yield."

## Likelihood Explanation
**Medium.** Chainlink feeds go stale in documented, recurring scenarios: L2 sequencer downtime, network congestion, and feed heartbeat windows (up to 24 hours for some LST/ETH feeds). LST depegs, even temporary ones, are a known occurrence in the ecosystem. The combination is realistic and non-hypothetical. The exploit requires no special privileges — any unprivileged user can call `depositAsset()`. [6](#0-5) 

## Recommendation
Add the same validation checks already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    // Optional: configurable per-feed heartbeat staleness check
    // if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, configure a per-feed maximum staleness threshold (heartbeat + buffer) and revert if `block.timestamp - updatedAt` exceeds it. [7](#0-6) 

## Proof of Concept
1. Chainlink's stETH/ETH feed last updated at `T-25h` with price `1.0 ETH` (within its 24h heartbeat, so no on-chain circuit breaker fires). stETH has since depegged to `0.97 ETH` on-market.
2. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
3. `_beforeDeposit` calls `getRsETHAmountToMint(stETH, 1000e18)`:
   ```
   rsethAmountToMint = (1000e18 * getAssetPrice(stETH)) / rsETHPrice
                     = (1000e18 * 1.0e18) / 1.0e18   ← stale price used
                     = 1000 rsETH
   ```
   Correct amount at actual price: `970 rsETH`.
4. Attacker receives `1000 rsETH` for `1000 stETH` worth only `970 ETH`, extracting `30 rsETH` (~3%) of value from existing holders.
5. No revert occurs because `ChainlinkPriceOracle.getAssetPrice()` performs zero validation on the returned price data.

**Foundry fork test plan:** Fork mainnet at a block where a stETH/ETH Chainlink feed is within its heartbeat but stETH has depegged. Deploy a mock `AggregatorV3Interface` returning a stale `updatedAt` and inflated price. Register it via `updatePriceFeedFor`. Call `depositAsset` and assert `rsethAmountToMint > (depositAmount * actualPrice / rsETHPrice)`. [8](#0-7)
