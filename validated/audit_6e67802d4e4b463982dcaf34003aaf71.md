### Title
Expired/Revoked TEE Attestation Not Validated in `is_caller_an_attested_participant()`, Allowing Revoked Nodes to Submit Threshold Signatures — (File: `crates/contract/src/tee/tee_state.rs`)

---

### Summary

`is_caller_an_attested_participant()` verifies only that a stored attestation *exists* for the calling participant, but never checks whether that attestation is still *valid* (not expired, image hash still allowlisted, measurements still accepted). This is the direct analog of the `transferFrom()` blacklist bypass: the `from`-party check is present (caller is in the participant set and has a stored entry) but the status check (is that entry still valid?) is absent. A participant whose TEE image hash has been revoked from the allowlist, or whose attestation has expired, can continue to call `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` until `clean_invalid_attestations()` is explicitly invoked.

---

### Finding Description

`is_caller_an_attested_participant()` performs four checks:

1. Signer is in the active participants list.
2. A `stored_attestations` entry exists for the participant's registered TLS key.
3. The stored `account_id` matches the signer.
4. The stored `account_public_key` matches the signer's on-chain key. [1](#0-0) 

It does **not** call `reverify_participants()`, which is the function that actually checks expiry, allowed MPC docker image hashes, allowed launcher compose hashes, and accepted measurements: [2](#0-1) 

`reverify_participants()` is only invoked from `reverify_and_cleanup_participants()` and `clean_invalid_attestations()` — never from the participant-authentication path used by the respond methods.

All three respond entry points gate on `assert_caller_is_attested_participant_and_protocol_active()`, which in turn calls only `is_caller_an_attested_participant()`: [3](#0-2) 

The three affected methods: [4](#0-3) [5](#0-4) [6](#0-5) 

The global `accept_requests` flag is checked, but it is a network-wide gate, not a per-participant validity gate. As long as enough *other* participants have valid attestations, `accept_requests` remains `true` and the revoked node's calls are not blocked by it. [7](#0-6) 

---

### Impact Explanation

The TEE security model's invariant is: **only nodes running an approved, attested image may participate in threshold signing**. When an operator discovers a vulnerability in a deployed image and removes its hash from the allowlist, the expectation is that nodes running that image are immediately barred from submitting signatures. Because `is_caller_an_attested_participant()` never calls `reverify_participants()`, a node whose image hash has been revoked (or whose attestation has expired) retains full `respond*` access until `clean_invalid_attestations()` is explicitly called. During that window the revoked node can:

- Continue participating in the off-chain P2P threshold protocol (other nodes still treat it as a valid participant because it remains in the on-chain participant set).
- Submit the resulting threshold signature via `respond()` / `respond_ckd()` / `respond_verify_foreign_tx()`, which the contract accepts because the cryptographic signature check passes and the stale attestation entry still satisfies `is_caller_an_attested_participant()`.

This breaks the participant/attestation authorization invariant: a node that the governance layer has effectively revoked continues to contribute to — and complete — threshold signing operations.

Impact classification: **Medium** — participant-state manipulation that breaks a production safety invariant (`respond*` access should be gated on a *valid* attestation, not merely a *present* one) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

`clean_invalid_attestations()` is not called automatically on every block; it is triggered manually or as a post-resharing promise. The gap between an image-hash revocation vote crossing threshold and the next `clean_invalid_attestations()` sweep can span many hours or days. During that entire window any revoked-but-still-stored participant can call `respond*`. The attack requires no special privilege beyond being a current on-chain participant — an unprivileged adversarial node that has had its image revoked is the exact attacker model.

---

### Recommendation

Add a per-call validity check inside `is_caller_an_attested_participant()` (or inside `assert_caller_is_attested_participant_and_protocol_active()`) by invoking `reverify_participants()` on the resolved `NodeId` and returning an error if the result is not `TeeQuoteStatus::Valid`. This mirrors the fix in the reference report: apply the full status check to every relevant party, not just verify existence.

```rust
// After the existing key-binding checks in is_caller_an_attested_participant():
let node_id = &attestation.node_id;
let tee_upgrade_deadline = /* read from config */;
match self.reverify_participants(node_id, tee_upgrade_deadline) {
    TeeQuoteStatus::Valid => Ok(()),
    TeeQuoteStatus::Invalid(reason) => Err(AttestationCheckError::AttestationInvalid(reason)),
}
```

### Proof of Concept

1. **Setup**: Node A is an active participant with a stored, currently-valid attestation. `accept_requests` is `true`.
2. **Revocation**: Governance votes to remove Node A's MPC docker image hash from the allowlist. `reverify_participants()` would now return `TeeQuoteStatus::Invalid` for Node A.
3. **No cleanup yet**: `clean_invalid_attestations()` has not been called; Node A's entry remains in `stored_attestations`.
4. **Off-chain protocol**: A user submits a `sign()` request. Node A is still in the on-chain participant set, so other nodes include it in the P2P threshold protocol and a valid threshold signature is produced.
5. **Bypass**: Node A calls `respond(request, signature)`. The contract executes `assert_caller_is_attested_participant_and_protocol_active()` → `is_caller_an_attested_participant()`. The function finds Node A's entry in `stored_attestations`, confirms key binding, and returns `Ok(())` — **without ever calling `reverify_participants()`**. The cryptographic signature check passes. The response is accepted and the yield is resolved.
6. **Result**: A node running a revoked TEE image has successfully submitted a threshold signature, bypassing the TEE security model.

    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();
        let allowed_measurements = self.get_accepted_measurements();

        let participant_attestation = self.stored_attestations.get(&node_id.tls_public_key);
        let Some(participant_attestation) = participant_attestation else {
            return TeeQuoteStatus::Invalid("participant has no attestation".to_string());
        };

        // Verify the attestation quote
        let time_stamp_seconds = Self::current_time_seconds();
        match participant_attestation.verified_attestation.re_verify(
            time_stamp_seconds,
            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
