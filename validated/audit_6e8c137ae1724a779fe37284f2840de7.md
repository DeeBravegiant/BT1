### Title
Unprotected `init` Function Allows Front-Running to Hijack MPC Network Participant Set - (File: `crates/contract/src/lib.rs`)

### Summary
The `init` function of the MPC contract has no caller access control. Because the standard deployment flow deploys the contract code first and calls `init` in a separate transaction, any unprivileged NEAR account that observes the deployment on-chain can race to call `init` with attacker-controlled participant parameters before the legitimate operator does, permanently seizing control of the MPC network.

### Finding Description
The `init` function is the sole entry point for bootstrapping the MPC contract's participant set, governance threshold, and configuration. It is decorated with NEAR's `#[init]` macro, which only guarantees the function runs exactly once (when contract state is absent). It carries no `#[private]` guard, no owner check, and no signer-identity restriction of any kind. [1](#0-0) 

By contrast, the sibling initializer `init_running` is explicitly marked `#[private]`, meaning only the contract itself can invoke it. [2](#0-1) 

Every documented deployment flow — localnet, testnet, and the automated TEE cluster script — separates contract deployment from initialization into two distinct on-chain transactions: [3](#0-2) [4](#0-3) 

Between the deployment block and the initialization block, the contract is live on-chain but uninitialized. Any account that monitors the chain and detects the deployment can immediately submit an `init` call with attacker-chosen `parameters` (participant list, threshold, TLS keys, URLs) and win the race.

### Impact Explanation
A successful front-run lets the attacker call `init` with a `ThresholdParameters` struct that names only attacker-controlled accounts as participants and sets `threshold = 1`. The contract transitions directly into `ProtocolContractState::Running` with the attacker's participant set. [5](#0-4) 

From that point:
- All governance votes (`vote_add_domains`, `vote_new_parameters`, `vote_pk`, etc.) require being an attested participant — a set the attacker fully controls.
- The attacker can add signing domains, run key generation, and become the sole holder of MPC key shares.
- Every subsequent `sign`, `request_app_private_key`, and `verify_foreign_transaction` call will be processed by the attacker's node, enabling unauthorized threshold signature issuance over any cross-chain payload.
- The legitimate operators cannot recover without a full contract redeployment, which itself is subject to the same race.

This matches the allowed Critical impact: **Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization.**

### Likelihood Explanation
NEAR does not expose a public mempool in the Ethereum sense, but the deployment transaction is finalized and publicly visible on-chain within seconds. The window between the deployment block and the initialization block is non-zero in every real deployment (the scripts show them as separate CLI invocations, often with human review steps in between). An attacker running a NEAR indexer or polling the RPC can detect the uninitialized contract and submit a competing `init` transaction in the very next block. The attack requires no special privilege, no key material, and no collusion — only the ability to submit a NEAR transaction.

### Recommendation
Add an explicit caller restriction to `init`. The simplest approach consistent with the rest of the contract is to require that the signer is the contract account itself (i.e., the deployer), mirroring the `#[private]` pattern already used on `init_running`:

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    // Restrict to the contract account (deployer) only
    assert_eq!(
        env::predecessor_account_id(),
        env::current_account_id(),
        "init can only be called by the contract account"
    );
    // ... rest of function
}
```

Alternatively, combine deployment and initialization into a single atomic transaction using NEAR's `deploy_and_init` pattern, eliminating the race window entirely.

### Proof of Concept

1. Operator deploys the MPC contract without an init call (standard flow):
   ```
   near contract deploy mpc-contract.test.near use-file mpc_contract.wasm without-init-call ...
   ```
2. Attacker's indexer detects the new contract account on-chain (state = `NotInitialized`).
3. Attacker immediately submits:
   ```json
   near contract call-function as-transaction mpc-contract.test.near init \
     json-args '{
       "parameters": {
         "threshold": 1,
         "participants": {
           "next_id": 1,
           "participants": [["attacker.near", 0, {
             "tls_public_key": "<attacker_p2p_key>",
             "url": "https://attacker.example.com"
           }]]
         }
       }
     }' ...
   ```
4. If the attacker's transaction is included before the operator's `init` call, the contract initializes with `attacker.near` as the sole participant and threshold = 1.
5. The operator's subsequent `init` call fails with "contract already initialized" (NEAR `#[init]` guard).
6. The attacker's node runs key generation, acquires the MPC key shares, and can sign arbitrary cross-chain payloads unilaterally. [6](#0-5)
