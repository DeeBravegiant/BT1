### Title
Wrong Receiver ID Check in `validate_delegate_action` Blocks `DeterministicStateInit` Inside Meta-Transactions - (File: `runtime/runtime/src/action_validation.rs`)

### Summary

`validate_delegate_action` validates inner actions of a `DelegateAction` (meta-transaction) against the **outer transaction's `receiver_id`** (the user's own account, where the delegate action is unpacked) instead of `delegate_action.receiver_id()` (the actual target of the inner actions). This is structurally identical to the external report: a check that should use the identity of the **final target** instead uses the identity of the **intermediary**, blocking all legitimate use of `DeterministicStateInit` inside meta-transactions.

### Finding Description

In `validate_delegate_action`, the `inner_receiver` variable used to validate inner actions is set to `receiver` — the outer transaction's receiver, which is the user's own account (where the `DelegateAction` is unpacked) — rather than `delegate_action.receiver_id()`, which is the actual account the inner actions target. [1](#0-0) 

When a `DeterministicStateInit` inner action is present, `validate_deterministic_state_init` is called with this wrong `receiver_id`: [2](#0-1) 

The check `derived_id != *receiver_id` compares the deterministic account ID derived from the `state_init` payload against the **outer transaction's receiver** (the user's own account), not the deterministic account the inner action is actually targeting. Since the user's account is never equal to the derived deterministic ID, this check always fails, and the transaction is rejected with `InvalidDeterministicStateInitReceiver`.

The parallel to the external report is exact:
- **External report**: `BribeRewarder::deposit()` checks `msg.sender == NFT owner`, but `msg.sender` is the voter contract (the intermediary), so all `vote()` calls revert.
- **Nearcore**: `validate_delegate_action` checks `derive(state_init) == outer_tx.receiver_id`, but the correct check is against `delegate_action.receiver_id()` (the inner target), so all `DeterministicStateInit` actions inside meta-transactions are rejected.

The codebase itself documents this as a bug: [3](#0-2) 

The protocol feature `FixDelegatedDeterministicStateInit` gates the corrected code path: [4](#0-3) 

### Impact Explanation

Any user who submits a meta-transaction (`DelegateAction`) containing a `DeterministicStateInit` inner action — even with a perfectly valid payload and the correct `delegate_action.receiver_id` — will have their transaction rejected at tx-validation with `InvalidDeterministicStateInitReceiver`. The validity decision for a legitimate transaction is corrupted: a valid transaction is treated as invalid. This makes the combination of `DeterministicStateInit` + meta-transactions completely non-functional on protocol versions before `FixDelegatedDeterministicStateInit`.

The test suite confirms this behavior: [5](#0-4) 

### Likelihood Explanation

Any unprivileged user who attempts to deploy a deterministic account via a meta-transaction (a documented and intended use case, as shown by `test_deterministic_state_init_via_meta_tx`) will trigger this rejection. The entry path is a standard public RPC transaction submission. No special privileges are required. [6](#0-5) 

### Recommendation

Use `delegate_action.receiver_id()` as the `inner_receiver` unconditionally in `validate_delegate_action`, removing the protocol-version branch. The fix is already present behind the `FixDelegatedDeterministicStateInit` feature flag and should be the only code path.

### Proof of Concept

1. Deploy a global contract and derive a deterministic account ID `det_account` from a `state_init`.
2. Construct a `DelegateAction` with `sender_id = user_account`, `receiver_id = det_account`, and inner action `DeterministicStateInit { state_init, deposit: 0 }`.
3. Wrap it in a transaction signed by a relayer with `receiver_id = user_account` (the standard meta-tx structure).
4. Submit via RPC.
5. On protocol versions before `FixDelegatedDeterministicStateInit`, the transaction is rejected at tx-validation with `InvalidDeterministicStateInitReceiver { receiver_id: user_account, derived_id: det_account }`, even though the inner action is perfectly valid.

The exact exploit scenario and the pre-fix rejection path are demonstrated in: [7](#0-6)

fn try_meta_tx_deterministic_receiver_exploit(
    protocol_version: ProtocolVersion,
) -> Result<FinalExecutionOutcomeView, InvalidTxError> {
    let mut env = TestEnv::setup_with_version(Balance::from_near(100), protocol_version);
    env.deploy_global_contract(GlobalContractDeployMode::AccountId);

    let (_state_init_a, det_account_a) = env.new_deterministic_account_with_data(small());
    let (state_init_b, det_account_b) = env.new_deterministic_account_with_data(big());
    assert_ne!(det_account_a, det_account_b);

    // Deploy det_account_b and add a full-access key so it can act as meta_tx_sender.
    let user_signer = create_user_test_signer(&env.user_account());
    let storage_balance = env.balance_for_storage(state_init_b.clone());
    let deploy_tx = SignedTransaction::deterministic_state_init(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        env.get_tx_block_hash(),
        state_init_b.clone(),
        storage_balance,
    );
    env.run_tx(deploy_tx);

    let meta_tx_sender_signer = create_user_test_signer(&det_account_b);
    let pk_base64 = near_primitives_core::serialize::to_base64(
        &borsh::to_vec(&meta_tx_sender_signer.public_key()).unwrap(),
    );
    let add_key_args = serde_json::json!([
        { "batch_create": { "account_id": det_account_b.as_str() }, "id": 0 },
        {
            "action_add_key_with_full_access": {
                "promise_index": 0,
                "public_key": pk_base64,
                "nonce": 0
            },
            "id": 0,
            "return": true
        }
    ]);
    let add_key_tx = SignedTransaction::call(
        env.next_nonce(),
        env.user_account(),
        det_account_b.clone(),
        &user_signer,
        Balance::from_near(2),
        "call_promise".to_owned(),
        serde_json::to_vec(&add_key_args).unwrap(),
        Gas::from_teragas(300),
        env.get_tx_block_hash(),
    );
    env.run_tx(add_key_tx);

    // Craft the exploit: outer_tx.receiver = det_account_b = derive(state_init_b).
    // Old check: det_account_b == derive(state_init_b) passes.
    // The delegate action targets det_account_a, which is the wrong account.
    // In no protocol version can this ever be allowed to be executed successfully.
    let relayer = env.independent_account();
    let relayer_signer = create_user_test_signer(&relayer);
    let inner_action = Action::DeterministicStateInit(Box::new(DeterministicStateInitAction {
        state_init: state_init_b,
        deposit: Balance::ZERO,
    }));
    let delegate_nonce = env.next_nonce_for(&det_account_b);
    let delegate_action = DelegateAction {
        sender_id: det_account_b.clone(),
        receiver_id: det_account_a,
        actions: vec![NonDelegateAction::try_from(inner_action).unwrap()],
        nonce: delegate_nonce,
        max_block_height: 1_000_000,
        public_key: meta_tx_sender_signer.public_key(),
    };
    let signed_delegate_action =
        SignedDelegateAction::sign(&meta_tx_sender_signer, delegate_action);
    let tx = SignedTransaction::from_actions(
        env.next_nonce(),
        relayer,
        det_account_b,
        &relayer_signer,
        vec![Action::Delegate(Box::new(signed_delegate_action))],
        env.get_tx_block_hash(),
    );
    env.try_execute_tx(tx)
}
