Audit Report

## Title
First Depositor Captures Entire Reward Distribution via 1-Wei Stake — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` uses a Synthetix-style `rewardPerToken` accumulator that divides by `totalKernelStaked`. Because `stake()` enforces only `_amount != 0` and `notifyRewardAmount()` enforces only `totalKernelStaked != 0`, an attacker who stakes 1 wei before any legitimate user becomes the sole beneficiary of the entire reward stream from the moment `notifyRewardAmount` is called until other stakers arrive. The protocol's own NatSpec explicitly acknowledges 1-wei staking as an acceptable operational pattern, providing no on-chain guard against this.

## Finding Description

The accumulator in `rewardPerToken()` divides by `totalKernelStaked`:

```solidity
// L408-L413
return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

When `totalKernelStaked = 1`, the full reward rate accumulates into `rewardPerTokenStored` at `rewardRate * elapsed * 1e18` per second. The per-user `earned()` at L421-L423 then returns `1 * (rewardPerTokenStored - 0) / 1e18 = full reward amount` for the 1-wei position.

`stake()` at L281-L288 has no minimum amount check beyond `_amount != 0`. `notifyRewardAmount()` at L566-L570 only requires `totalKernelStaked != 0` before setting `rewardRate = receivedAmount / duration`. There is no on-chain check that `totalKernelStaked` represents a meaningful amount relative to the reward being distributed.

The NatSpec at L14-L22 explicitly states: *"staking just 1 wei by any address will ensure that the contract never has any unallocated rewards"* — framing 1-wei staking as an acceptable operational pattern, which means no off-chain guard is expected to prevent it either.

**Exploit path:**
1. Attacker calls `stake(1)` on a freshly deployed or post-period contract → `totalKernelStaked = 1`, `balanceOf[attacker] = 1`.
2. Admin calls `notifyRewardAmount(R)` → `rewardRate = R / duration`. The `totalKernelStaked != 0` check passes.
3. Time elapses. `rewardPerToken()` accumulates to `R * 1e18`.
4. Attacker calls `getReward()` → `earned(attacker) = 1 * (R * 1e18) / 1e18 = R`. Attacker receives all `R` reward tokens.
5. Legitimate stakers who join after step 2 receive rewards only from their stake time onward; rewards accrued in the window `[notifyRewardAmount → first legitimate stake]` are permanently captured by the attacker.

## Impact Explanation

**High — Theft of unclaimed yield.** The attacker directly receives reward tokens deposited by the admin for distribution to legitimate stakers. If no legitimate staker joins during the full period, 100% of the reward distribution is stolen. Even with later stakers, all yield accrued before their arrival is permanently captured. This is a concrete, quantifiable token loss matching the "Theft of unclaimed yield" impact class.

## Likelihood Explanation

**Medium.** No front-running is required — the attacker only needs to call `stake(1)` before any legitimate user on a new or reset contract, which is trivially achievable. The cost is 1 wei of KERNEL plus gas. The protocol's NatSpec explicitly endorses 1-wei staking as a valid operational pattern, so no off-chain monitoring is expected to flag it. The admin's `notifyRewardAmount` call is a routine operation with no on-chain signal distinguishing a 1-wei attacker position from a legitimate dust stake.

## Recommendation

1. **Enforce a minimum stake** in `stake()`: `require(_amount >= MIN_STAKE)` where `MIN_STAKE` is set to a value that makes 1-wei positions economically negligible relative to expected reward distributions.
2. **Alternatively**, add a meaningful threshold check in `notifyRewardAmount()`: require `totalKernelStaked >= MIN_STAKED_THRESHOLD` before allowing reward distribution to begin.
3. **Or**, initialize `totalKernelStaked` with a protocol-owned dead-share (analogous to Uniswap v2's minimum liquidity burn) so that any single user's share is always bounded by `userBalance / (userBalance + deadShare)`.

## Proof of Concept

**Parameters:** `duration = 30 days`, `R = 1_000_000e18` reward tokens.

```
1. attacker.stake(1)
   → totalKernelStaked = 1, balanceOf[attacker] = 1

2. admin.notifyRewardAmount(1_000_000e18)
   → rewardRate = 1_000_000e18 / 2_592_000 ≈ 3.858e17 tokens/sec
   → finishAt = block.timestamp + 30 days

3. warp(+30 days)

4. rewardPerToken()
   = 0 + (3.858e17 * 2_592_000 * 1e18) / 1
   = 1_000_000e18

5. attacker.getReward()
   earned(attacker) = 1 * (1_000_000e18 - 0) / 1e18 = 1_000_000e18
   → attacker receives 1_000_000 reward tokens

6. Any legitimate user who staked after step 2 receives 0 rewards for this period.
```

Foundry test plan: deploy `KernelDepositPool`, call `stake(1)` from attacker address, call `notifyRewardAmount` from admin, `vm.warp` by `duration`, call `getReward()`, assert `rewardsToken.balanceOf(attacker) == R`.
