### Title
Double `PromiseResume` Receipt Creation via Repeated `promise_yield_resume` Calls — (`runtime/runtime/src/ext.rs`)

### Summary

`submit_promise_resume_data` in `runtime/runtime/src/ext.rs` uses `has_promise_yield_status` as a guard against double-resume, but that function only checks whether the trie key **exists** — not what value it holds. After the first call writes `PromiseYieldStatus::ResumeInitiated`, the key still exists, so the second call within the same function execution passes the guard again and appends a second `PromiseResume` receipt for the same `data_id`.

### Finding Description

The guard logic in `submit_promise_resume_data` is:

```rust
if has_yield_receipt_in_state || has_yield_status_in_state {
    self.receipt_manager.create_promise_resume_receipt(data_id, data);
    set_promise_yield_status(..., PromiseYieldStatus::ResumeInitiated);
    return Ok(true);
}
``` [1](#0-0) 

`has_promise_yield_receipt` checks whether `TrieKey::PromiseYieldReceipt` exists in the trie: [2](#0-1) 

`has_promise_yield_status` checks whether `TrieKey::PromiseYieldStatus` exists in the trie: [3](#0-2) 

Neither check inspects the **value** of the status — only its presence.

**First call** to `submit_promise_resume_data(data_id, ...)`:
- `has_promise_yield_receipt` → `true` (yield receipt is in the trie)
- `has_promise_yield_status` → `true` (status is `Yielded`)
- Guard passes → `create_promise_resume_receipt` appends receipt #1, status overwritten to `ResumeInitiated`

**Second call** to `submit_promise_resume_data(data_id, ...)` within the same WASM execution:
- `has_promise_yield_receipt` → still `true` — the yield receipt is **not removed** during resume initiation; it is only removed when the receipt is actually processed by the runtime
- `has_promise_yield_status` → still `true` — the key still exists, now holding `ResumeInitiated`; `has_promise_yield_status` does not distinguish `Yielded` from `ResumeInitiated`
- Guard passes again → `create_promise_resume_receipt` appends receipt #2 for the same `data_id` [4](#0-3) 

The `TrieUpdate` overlay makes pending writes immediately visible to subsequent reads within the same execution, so the `ResumeInitiated` write from call #1 is visible to call #2 — but it does not prevent the guard from passing because the check is existence-only.

### Impact Explanation

A malicious contract that calls `promise_yield_resume` twice for the same `data_id` in a single function execution produces two `PromiseResume` (data) receipts in the outgoing receipt set. This:

1. **Violates the invariant** that each `data_id` produces at most one resume receipt.
2. **Produces a divergent receipt set** — the outgoing receipts differ from what the protocol intends, causing a different state root after the chunk is applied.
3. **May cause double-execution** of the yield callback when both receipts are processed, leading to incorrect balance changes, nonce state, or storage mutations depending on the callback logic.

### Likelihood Explanation

Any unprivileged user can deploy a contract that calls `promise_yield_resume` twice for the same `data_id` in a single function body. No validator or admin privileges are required. The attacker only needs to pay gas for two `promise_yield_resume` calls. The path is directly reachable via a public `FunctionCall` transaction.

### Recommendation

Replace the existence check with a value check. The guard should only allow resume when the status is specifically `Yielded`, not `ResumeInitiated`:

```rust
let yield_status = get_promise_yield_status(self.trie_update, &self.account_id, data_id)
    .map_err(wrap_storage_error)?;

let is_resumable = has_yield_receipt_in_state
    || matches!(yield_status, Some(PromiseYieldStatus::Yielded));

if is_resumable {
    self.receipt_manager.create_promise_resume_receipt(data_id, data);
    set_promise_yield_status(..., PromiseYieldStatus::ResumeInitiated);
    return Ok(true);
}
```

This ensures that once `ResumeInitiated` is written, subsequent calls for the same `data_id` within the same execution return `false` without creating a second receipt.

### Proof of Concept

A Rust unit test calling `submit_promise_resume_data` twice for the same `data_id` on the same `Ext` instance (backed by a `TrieUpdate` that has a `PromiseYieldReceipt` for that `data_id`) would show that `receipt_manager.data_receipts` contains two entries with `is_promise_resume: true` for the same `data_id` after both calls return `Ok(true)`.     fn submit_promise_resume_data(
        &mut self,
        data_id: CryptoHash,
        data: Vec<u8>,
    ) -> Result<bool, VMLogicError> {
        let has_yield_receipt_in_state =
            has_promise_yield_receipt(self.trie_update, self.account_id.clone(), data_id)
                .map_err(wrap_storage_error)?;
        let has_yield_status_in_state =
            has_promise_yield_status(self.trie_update, &self.account_id, data_id)
                .map_err(wrap_storage_error)?;

        if has_yield_receipt_in_state || has_yield_status_in_state {
            self.receipt_manager.create_promise_resume_receipt(data_id, data);
            set_promise_yield_status(
                &mut self.trie_update,
                &self.account_id,
                data_id,
                PromiseYieldStatus::ResumeInitiated,
            );
            return Ok(true);
        }

        Ok(false)
    }
