### Title
Unelected Committee Members Can Cast Governance Votes Accepted by the GOV Rule Before Protocol Version 11 — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

---

### Summary

In the Conway era GOV ledger rule, the voter-existence check uses `authorizedHotCommitteeCredentials`, which returns hot credentials for **all** cold credentials in `CommitteeState`, including those that are **not** in the currently elected committee. A separate, correctly scoped function `authorizedElectedHotCommitteeCredentials` exists but is not used for this check. Before protocol version 11, the GOV rule contains no guard against unelected committee votes; the only guard lives in the MEMPOOL rule, which a block producer can bypass. Votes cast by unelected committee members are stored in `gasCommitteeVotes` and will count toward ratification the moment the cold credential is elected, enabling unauthorized enactment of governance actions.

---

### Finding Description

**Two functions with critically different semantics share a misleadingly similar name:**

`authorizedHotCommitteeCredentials` (in `libs/cardano-ledger-core/src/Cardano/Ledger/State/CertState.hs`) collects every hot credential from `CommitteeState.csCommitteeCreds`, regardless of whether the corresponding cold credential appears in the currently elected `Committee`:

```haskell
authorizedHotCommitteeCredentials :: CommitteeState era -> Set.Set (Credential HotCommitteeRole)
authorizedHotCommitteeCredentials CommitteeState {csCommitteeCreds} =
  let toHotCredSet acc = \case
        CommitteeHotCredential hotCred -> Set.insert hotCred acc
        CommitteeMemberResigned {} -> acc
   in F.foldl' toHotCredSet Set.empty csCommitteeCreds
``` [1](#0-0) 

`authorizedElectedHotCommitteeCredentials` (in `eras/conway/impl/src/Cardano/Ledger/Conway/Governance.hs`) correctly intersects `csCommitteeCreds` with the elected committee's member map before extracting hot credentials:

```haskell
authorizedElectedHotCommitteeCredentials committee committeeState =
  case committee of
    SNothing -> Set.empty
    SJust electedCommiteee ->
      authorizedHotCommitteeCredentials $
        CommitteeState $
          csCommitteeCreds committeeState `Map.intersection` committeeMembers electedCommiteee
``` [2](#0-1) 

**In `conwayGovTransition`, the voter-existence set is built with the broader, unscoped function:**

```haskell
knownCommitteeMembers = authorizedHotCommitteeCredentials committeeState
``` [3](#0-2) 

This set is used at line 594 to intern/validate `CommitteeVoter` credentials. Any hot credential authorized by *any* cold credential in `CommitteeState` — elected or not — passes the `VotersDoNotExist` check.

**The guard against unelected voters is version-gated and absent from the GOV rule before PV 11:**

```haskell
when (hardforkConwayDisallowUnelectedCommitteeFromVoting $ pp ^. ppProtocolVersionL) $
    failOnNonEmpty
      (unelectedCommitteeVoters committee committeeState gsVotingProcedures)
      (injectFailure . UnelectedCommitteeVoters)
``` [4](#0-3) 

```haskell
hardforkConwayDisallowUnelectedCommitteeFromVoting :: ProtVer -> Bool
hardforkConwayDisallowUnelectedCommitteeFromVoting pv = pvMajor pv > natVersion @10
``` [5](#0-4) 

At protocol versions 9–10 (the entire deployed Conway era), the GOV rule imposes no restriction on unelected committee voters.

**The MEMPOOL rule does check, but only at the mempool layer:**

```haskell
unless (hardforkConwayDisallowUnelectedCommitteeFromVoting protVer) $
  -- This check can completely be removed once mainnet switches to protocol
  -- version 11, since the same check has been implemented in the GOV rule.
  --
  -- Disallow votes by unelected committee members
  ...
  failOnNonEmpty
    ( unelectedCommitteeVoters ... )
    (ConwayMempoolFailure . addPrefix . T.pack . show . NE.toList)
``` [6](#0-5) 

A block producer can include a transaction directly in a block, bypassing the MEMPOOL rule entirely. The GOV rule then accepts the unelected committee vote.

**Accepted votes persist and count upon election.** `committeeAcceptedRatio` iterates over elected cold credentials, resolves their hot credential via `CommitteeState`, and looks up that hot credential in `gasCommitteeVotes`:

```haskell
Just (CommitteeHotCredential hotKey) ->
  case Map.lookup hotKey votes of
    Nothing -> (yes, tot + 1) -- member hasn't voted, vote "no"
    ...
    Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
``` [7](#0-6) 

A pre-cast vote stored under the hot credential is found and counted the moment the cold credential appears in the elected committee.

---

### Impact Explanation

An unelected committee member's vote, injected by a block producer into a block at PV 9–10, is accepted by the GOV rule and stored in `gasCommitteeVotes`. When that cold credential is subsequently elected via an `UpdateCommittee` action, the pre-cast vote immediately counts in `committeeAcceptedRatio` without any re-vote. This can tip the committee acceptance ratio over the required threshold, causing a governance action — including `ParameterChange`, `HardForkInitiation`, `TreasuryWithdrawals`, `NewConstitution`, or `UpdateCommittee` — to be ratified and enacted without the committee member having reviewed the action post-election.

**Allowed impact matched:** *Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted* (Critical); or at minimum, *attacker-controlled transactions exceed intended validation limits* (Medium).

---

### Likelihood Explanation

- Any party can register a cold credential and authorize a hot credential — no privilege required.
- A block producer (explicitly listed as a valid attacker in the scope) can include the vote transaction directly in a block, bypassing the MEMPOOL guard.
- The pre-cast vote only counts if the cold credential is later elected, which requires a separate governance action. However, a candidate who knows they are about to be elected (e.g., because an `UpdateCommittee` proposal has already passed the DRep/SPO threshold and is pending enactment) can pre-cast votes on multiple governance actions in the same epoch, maximizing impact.
- The window is the entire PV 9–10 Conway era (currently deployed on mainnet).

---

### Recommendation

Replace `authorizedHotCommitteeCredentials` with `authorizedElectedHotCommitteeCredentials` at line 474 of `Gov.hs` when building `knownCommitteeMembers`, so that only hot credentials belonging to elected cold credentials are considered "known" voters:

```haskell
-- Before (line 474):
knownCommitteeMembers = authorizedHotCommitteeCredentials committeeState

-- After:
knownCommitteeMembers = authorizedElectedHotCommitteeCredentials committee committeeState
```

This makes the voter-existence check consistent with the ratification logic and eliminates the need for the separate version-gated `unelectedCommitteeVoters` guard in the GOV rule (which can then be removed). The MEMPOOL guard can also be removed once this fix is in place.

---

### Proof of Concept

The existing test at `eras/conway/impl/testlib/Test/Cardano/Ledger/Conway/Imp/LedgerSpec.hs:281–318` already demonstrates the divergence:

```haskell
it "Unelected Committee voting" $ whenPostBootstrap $ do
  ...
  pv <- getProtVer
  if hardforkConwayDisallowUnelectedCommitteeFromVoting pv
    then
      submitFailingTx tx [injectFailure $ UnelectedCommitteeVoters [ccHot]]
    else do
      txFixed <-
        submitFailingMempoolTx "unallowed votes" tx $
          NonEmpty.singleton . injectFailure . ConwayMempoolFailure $
            "Unelected committee members are not allowed to cast votes: " <> ...
      withNoFixup $ submitTx_ txFixed   -- ← succeeds at the LEDGER level
``` [8](#0-7) 

`withNoFixup $ submitTx_ txFixed` applies the LEDGER rule directly (simulating block inclusion) and **succeeds** at PV 9–10, confirming that the GOV rule accepts the unelected committee vote. A block producer exploiting this would:

1. Register cold credential `C` and authorize hot credential `H`.
2. Craft a transaction voting `VoteYes` as `CommitteeVoter H` on a target governance action `G`.
3. Include the transaction directly in a produced block (bypassing MEMPOOL).
4. The GOV rule accepts the vote; `H → VoteYes` is stored in `gasCommitteeVotes G`.
5. Submit or wait for an `UpdateCommittee` action that adds `C` to the elected committee.
6. Once enacted, `committeeAcceptedRatio` finds `H`'s pre-cast `VoteYes` and counts it, potentially ratifying `G` without `C` having reviewed it post-election.
