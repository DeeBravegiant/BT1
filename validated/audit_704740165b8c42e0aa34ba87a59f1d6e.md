Audit Report

## Title
stETH Rebasing Rewards Unaccounted During LRTConverter Transit â€” (File: contracts/LRTConverter.sol)

## Summary

`LRTConverter` records the ETH value of transferred stETH in a fixed counter (`ethValueInWithdrawal`) at the moment of transfer. Because stETH is a rebasing token, its balance grows daily while held in the converter, but `ethValueInWithdrawal` is never updated to reflect this growth. The delta â€” accumulated rebasing rewards â€” is invisible to the rsETH price calculation for the entire duration stETH sits in the converter, and if the operator unstakes only the originally tracked amount, the excess stETH remains in the converter with no automatic accounting or routing mechanism.

## Finding Description

**Root cause â€” fixed counter set at transfer time:**

In `transferAssetFromDepositPool` (L140), the ETH value is snapshotted once:

```solidity
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
```

This counter is never incremented again, regardless of how much stETH rebases.

**Accounting path that excludes live balance:**

`getAssetDistributionData` (L460) explicitly zeroes the converter's contribution for non-ETH assets:

```solidity
assetLyingInConverter = 0; // assets in converter are accounted in their eth value => getETHDistributionData()
```

`getETHDistributionData` (L499) then reads only the static counter:

```solidity
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
```

Every other location stETH can reside (deposit pool, NDCs, unstaking vault) uses live `IERC20(asset).balanceOf(...)` calls, so rebasing is captured automatically there. The converter is the sole exception.

**Orphaned stETH after partial unstake:**

`unstakeStEth` (L170â€“177) passes an operator-supplied `amountToUnstake` to `_unstakeStEth` (UnstakeStETH.sol L48â€“57), which calls `withdrawalQueue.requestWithdrawals(amounts, address(this))` for exactly that amount. If the operator supplies the originally tracked amount rather than the full rebased balance, the rebasing delta remains as stETH in the converter. After `claimStEth` is called, `_sendEthToDepositPool` (L252â€“263) zeroes `ethValueInWithdrawal`:

```solidity
if (ethValueInWithdrawal > _amount) {
    ethValueInWithdrawal -= _amount;
} else {
    ethValueInWithdrawal = 0;
}
```

The orphaned stETH now has no accounting entry. `transferAssetToDepositPool` (L149â€“166) exists and is callable by the asset-transfer role, providing a manual recovery path, but it requires explicit operator awareness and action â€” there is no automatic mechanism.

**Why existing checks are insufficient:**

`withinUnstakeLimits` (L58â€“77) only enforces that `amountToUnstake` does not exceed withdrawal allowances; it does not enforce that the full rebased balance is unstaked. No function in `LRTConverter` reads `stETH.balanceOf(address(this))` to compare against `ethValueInWithdrawal`.

## Impact Explanation

**Medium â€” Permanent freezing of unclaimed yield.**

For the entire period stETH sits in the converter, `getTotalAssetDeposits` underreports total protocol assets by the accumulated rebasing rewards. This understates the rsETH exchange rate, diluting existing rsETH holders in favour of any depositor who mints rsETH during that window. If the operator unstakes only the tracked amount, the rebasing-reward stETH remains in the converter with no automatic routing back to the protocol. While `transferAssetToDepositPool` provides a manual escape hatch, it requires the operator to identify and act on the discrepancy; absent that action, the yield is frozen in the converter indefinitely.

## Likelihood Explanation

stETH is a first-class supported asset (`ST_ETH_TOKEN`). `LRTConverter` is the designated unstaking path for stETH. Lido's accounting oracle rebases every ~24 hours unconditionally. Any stETH that transits through `LRTConverter` â€” the normal operational flow â€” will accumulate unaccounted rewards. No attacker action is required; the accounting gap opens automatically the moment stETH enters the converter and closes only when the operator manually intervenes.

## Recommendation

Replace the fixed ETH-value counter for stETH with a shares-based representation using `IStETH.getSharesByPooledEth` / `IStETH.getPooledEthByShares`, so that `ethValueInWithdrawal` reflects the live rebased balance at query time. Alternatively, enforce in `unstakeStEth` that `amountToUnstake == stETH.balanceOf(address(this))` (i.e., always unstake the full balance), and update `ethValueInWithdrawal` to the live ETH value of that full balance before submitting the withdrawal request.

## Proof of Concept

1. Operator calls `LRTConverter.transferAssetFromDepositPool(stETH, 1000e18)`.
   - `ethValueInWithdrawal` = `1000e18` (assuming 1:1 price).
2. Lido oracle runs the next day. `stETH.balanceOf(converter)` becomes `1000.04e18`.
   - `ethValueInWithdrawal` is still `1000e18`.
3. A user calls `LRTDepositPool.depositAsset(stETH, ...)`. `getTotalAssetDeposits(stETH)` â†’ `getETHDistributionData()` â†’ `ethLyingInConverter = 1000e18`. The `0.04e18` rebasing reward is invisible. rsETH price is understated; the depositor receives more rsETH than they should, diluting existing holders.
4. Operator calls `unstakeStEth(1000e18)`. `0.04e18` stETH remains in `LRTConverter`.
5. Lido finalises the withdrawal. Operator calls `claimStEth`. ETH goes to deposit pool. `ethValueInWithdrawal` is zeroed.
6. `0.04e18` stETH remains in `LRTConverter` with no accounting entry. It is recoverable only if the operator explicitly calls `transferAssetToDepositPool(stETH, 0.04e18)`; otherwise it is frozen indefinitely.
