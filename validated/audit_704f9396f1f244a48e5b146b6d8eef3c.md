Audit Report

## Title
`_updateRsETHPrice` Resets Price to `1 ether` When `rsethSupply == 0`, Enabling Theft of Accrued Yield — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle._updateRsETHPrice()` unconditionally resets `rsETHPrice` and `highestRsethPrice` to `1 ether` and returns early when `rsethSupply == 0`, without ever calling `_getTotalEthInProtocol()`. Any ETH remaining in the protocol at that moment — including ETH mid-unstake from EigenLayer subject to the ~7-day withdrawal delay — is silently orphaned. The first depositor after the reset mints rsETH at `1 ether`, then captures the orphaned ETH as yield when the price is next updated.

## Finding Description

In `contracts/LRTOracle.sol` at lines 218–222, `_updateRsETHPrice()` contains:

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

The function returns before reaching line 231 (`uint256 totalETHInProtocol = _getTotalEthInProtocol()`), which aggregates ETH across `LRTDepositPool`, all `NodeDelegator` contracts, EigenLayer pod shares (`getEffectivePodShares()`), `LRTUnstakingVault`, and `LRTConverter` (see `LRTDepositPool.getETHDistributionData()`, lines 467–500).

EigenLayer enforces a mandatory withdrawal delay (~7 days). When the last rsETH is burned via `unlockQueue()`, ETH that is mid-unstake from EigenLayer is still tracked by `getAssetUnstaking()` and `getEffectivePodShares()` but is not yet redeemable. If `updateRSETHPrice()` — which is `public whenNotPaused` and callable by anyone — is called at this moment, the zero-supply branch fires, resetting the price anchor to `1 ether` and discarding the in-flight ETH from the accounting.

A secondary consequence: `highestRsethPrice` is also reset to `1 ether`. The downside-protection circuit-breaker at lines 270–282 uses `highestRsethPrice` as its reference. Resetting it from, e.g., `1.1 ether` to `1 ether` permanently lowers the reference, allowing a larger absolute price drop before the pause triggers in the next cycle.

## Impact Explanation

**High — Theft of unclaimed yield.** Previous rsETH holders accrued staking rewards that are still in-flight inside EigenLayer. After the price reset to `1 ether`, those rewards are not reflected in the price. The first depositor mints rsETH at `1 ether`. When the EigenLayer withdrawal completes and the ETH lands in the protocol, the next `updateRSETHPrice()` call incorporates it, raising `rsETHPrice` above `1 ether`. The first depositor withdraws at the higher price, capturing yield that belonged to the previous holders.

## Likelihood Explanation

`rsethSupply` reaching zero is a natural protocol event: it occurs when every user completes the full withdrawal lifecycle (`initiateWithdrawal` → `unlockQueue` → `completeWithdrawal`). EigenLayer's mandatory ~7-day withdrawal delay routinely leaves ETH in-flight during this window. `updateRSETHPrice()` is callable by any unprivileged external account while the protocol is unpaused, so an attacker only needs to observe the zero-supply condition and deposit before the orphaned ETH is accounted for.

## Recommendation

Before returning early in the zero-supply branch, call `_getTotalEthInProtocol()` and check whether it is non-zero. If it is, either:
1. Revert, preventing the price reset until all in-flight ETH has settled, or
2. Preserve the current `rsETHPrice` and `highestRsethPrice` rather than resetting them to `1 ether`.

Additionally, do not unconditionally reset `highestRsethPrice` to `1 ether`; preserve the historical peak so the circuit-breaker reference is not weakened on protocol restart.

## Proof of Concept

1. Protocol runs normally; `rsETHPrice = 1.1 ether`, `highestRsethPrice = 1.1 ether`. ~0.05 ETH of staking rewards is mid-unstake inside EigenLayer (tracked by `getEffectivePodShares()` / `getAssetUnstaking()`).
2. All users call `initiateWithdrawal(ETH, ...)`. rsETH is held in `LRTWithdrawalManager`.
3. Operator calls `unlockQueue(ETH, ...)`. All rsETH is burned; `rsethSupply = 0`. The 0.05 ETH is still in EigenLayer's withdrawal queue and cannot yet be redeemed.
4. Anyone calls `updateRSETHPrice()`. The zero-supply branch at lines 218–222 fires: `rsETHPrice = 1 ether`, `highestRsethPrice = 1 ether`. The 0.05 ETH is never read.
5. Attacker calls `LRTDepositPool.depositETH{value: 1 ether}(0, "")`. `getRsETHAmountToMint` (line 520) computes `1e18 * 1e18 / 1e18 = 1e18` rsETH minted at `rsETHPrice = 1 ether`.
6. EigenLayer withdrawal completes; 0.05 ETH lands in the protocol. `_getTotalEthInProtocol()` now returns `1.05 ether`.
7. Anyone calls `updateRSETHPrice()`. `newRsETHPrice = 1.05e18 / 1e18 = 1.05 ether`.
8. Attacker calls `initiateWithdrawal` + `unlockQueue` + `completeWithdrawal`, receiving `1.05 ETH` for their `1 rsETH`. Net gain: `0.05 ETH` of yield that belonged to the previous depositors.
