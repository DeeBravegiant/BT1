### Title
`only_query` Hardcoded to `false` in `BroadcastedDeclareV1Transaction` Simulation Path Returns Wrong Execution Info to Contracts — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary
The `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput` conversion in `api/mod.rs` hardcodes `only_query = false` with an explicit TODO comment, causing `starknet_estimateFee` and `starknet_simulateTransactions` to execute DeclareV1 transactions as if they are real sequenced transactions rather than query-mode simulations.

### Finding Description

In the `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput` implementation, the `only_query` field is hardcoded to `false`: [1](#0-0) 

This value flows directly into `CommonAccountFields.only_query` via `create_tx_info()`: [2](#0-1) 

The `only_query` flag controls what the `get_execution_info` syscall returns to the executing contract. When `only_query=true`, the transaction version is modified (the high bit `2^128` is set, per `signed_tx_version`), signaling to the contract that it is running in simulation/query mode. With `only_query=false` hardcoded, the version returned is the plain real version (`1` for DeclareV1), indistinguishable from an actual sequenced transaction. [3](#0-2) 

### Impact Explanation

Any account contract whose `__validate_declare__` entry point calls `get_execution_info` and branches on whether the transaction version has the query-mode high bit set will receive the wrong version during `estimateFee` or `simulateTransactions`. The simulation result — including fee estimate, execution trace, and revert status — will reflect non-query semantics, producing an authoritative-looking but incorrect RPC response. This falls squarely under: **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

The trigger path is fully unprivileged: any caller can submit a `BroadcastedDeclareV1Transaction` to `starknet_estimateFee` or `starknet_simulateTransactions`. The bug is unconditional — it fires for every such call. The impact is conditional on the account contract inspecting `get_execution_info` for query-mode detection, which is a documented and supported pattern in the Starknet spec.

### Recommendation

Replace the hardcoded `false` at `api/mod.rs:507` with `true` for all simulation/estimation paths (`estimateFee`, `simulateTransactions`). The TODO comment already acknowledges this is unresolved. The correct value for simulation endpoints is `only_query = true`. [4](#0-3) 

### Proof of Concept

1. Deploy an account contract whose `__validate_declare__` calls `get_execution_info` and checks whether `tx_info.version >= 2^128` (query mode). If in query mode, it returns success; otherwise it reverts.
2. Call `starknet_estimateFee` with a `BroadcastedDeclareV1Transaction` signed by that account.
3. Observe the simulation reverts (because `only_query=false` causes version `1` to be returned, not `2^128 + 1`), even though the same transaction submitted for real execution would succeed.
4. Alternatively: assert in a Rust integration test that `get_execution_info.version` inside `__validate_declare__` during `estimateFee` equals `2^128 + 1`, not `1`.
