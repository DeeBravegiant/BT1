### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Original Request — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over the caller-supplied `response.payload_hash`, but never checks that `payload_hash` is actually derived from the `request` being resolved. A single Byzantine attested participant (the leader node) can take a legitimately produced threshold signature for one pending foreign-chain verification request and use it to resolve a *different* pending request, delivering a forged `payload_hash` to the waiting caller.

---

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following checks before resolving pending yields:

1. Caller is an attested participant.
2. Protocol is running.
3. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key. [1](#0-0) 

What it does **not** check is whether `response.payload_hash` is the canonical hash of `ForeignTxSignPayload::V1 { request: request.request, values: ... }` for the specific `request` being resolved.

The signed payload is defined as:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }))
``` [2](#0-1) 

The `ForeignChainRpcRequest` (including `tx_id`) is embedded in the hash, so `payload_hash_A` for request A is cryptographically distinct from `payload_hash_B` for request B. However, the contract never reconstructs the expected hash from `request.request` and compares it to `response.payload_hash`. It only checks that the signature is valid over *whatever* `payload_hash` the caller provides.

**Attack path:**

1. Two requests are pending simultaneously:
   - Request A: verify Bitcoin tx X (`tx_id = [0xAA; 32]`)
   - Request B: verify Bitcoin tx Y (`tx_id = [0xBB; 32]`)
2. The MPC network legitimately produces `(payload_hash_A, signature_A)` for request A.
3. The Byzantine leader calls `respond_verify_foreign_tx(request_B, response_A)` — passing request B as the key but response A as the payload.
4. The contract checks: is `signature_A` valid over `payload_hash_A`? **Yes** — it is a valid threshold signature.
5. The contract resolves request B's pending yields with `response_A`.
6. The caller who submitted request B receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_A, signature: signature_A }`.

The caller receives a response with a valid MPC signature, but `payload_hash_A` encodes Bitcoin tx X's data, not Bitcoin tx Y's. [3](#0-2) 

The design documentation explicitly states that callers are expected to reconstruct the expected hash locally and compare:

> "The response contains the hash of the sign payload, so callers can verify the signature by checking it against the expected hash they reconstruct locally."

But the contract itself provides no on-chain enforcement of this binding, so any bridge contract that only verifies the signature (without re-deriving the expected `payload_hash` from the original request) will silently accept the forged attestation.

The `ForeignChainSignatureVerifier` helper in the SDK does perform this check client-side: [4](#0-3) 

But this is an off-chain SDK helper, not an on-chain invariant enforced by the contract.

---

### Impact Explanation

**High — Forged foreign-chain verification causing invalid bridge execution.**

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a bridge contract mints or releases tokens on NEAR only after the MPC network attests that a specific foreign-chain transaction finalized. A Byzantine leader can substitute a valid attestation for transaction X in place of the expected attestation for transaction Y. Any bridge contract that does not independently re-derive and compare `payload_hash` will accept the forged attestation and execute the bridge action (e.g., mint tokens) for a transaction that was never verified — or was verified with different parameters (different amount, recipient, or block hash).

---

### Likelihood Explanation

- Requires exactly **one** Byzantine attested participant acting as the signing leader — strictly below the threshold.
- Requires two concurrent pending requests (routine in any active bridge deployment).
- The threshold signature for request A is produced by honest nodes; the Byzantine leader merely redirects it to resolve request B.
- Bridge contracts that follow the SDK's `ForeignChainSignatureVerifier` pattern are protected, but contracts that only call `verify_ecdsa_signature` on the returned signature without checking `payload_hash` content are vulnerable — a realistic mistake given that the contract itself does not enforce the binding.

---

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, recompute the expected `payload_hash` from `request.request` and `request.payload_version` and assert it equals `response.payload_hash`:

```rust
// After signature verification:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: /* extracted from response or re-derived */,
}).compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Because the extracted `values` are not stored on-chain (only the request key is), the simplest fix is to include the `ForeignChainRpcRequest` in the hash check: verify that `response.payload_hash` decodes (via the SDK's `ForeignTxSignPayload`) to a payload whose `.request` field equals `request.request`. Alternatively, store the expected `payload_hash` in the pending-request map at submission time (computed from the request key alone, without the extracted values) and enforce it at response time.

// Two pending requests exist simultaneously:
// request_A: Bitcoin tx_id = [0xAA; 32]
// request_B: Bitcoin tx_id = [0xBB; 32]

// MPC nodes legitimately sign payload_hash_A for request_A.
let payload_a = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: [0xAA; 32].into(), confirmations: 1.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
    values: vec![ExtractedValue::BitcoinExtractedValue(
        BitcoinExtractedValue::BlockHash([0x01; 32].into()),
    )],
});
let payload_hash_a = payload_a.compute_msg_hash().unwrap();
let (sig_a, rec_a) = signing_key.sign_prehash_recoverable(&payload_hash_a.0).unwrap();
let response_a = VerifyForeignTransactionResponse {
    payload_hash: payload_hash_a,
    signature: SignatureResponse::Secp256k1(
        K256Signature::from_ecdsa_recoverable(&sig_a, rec_a),
    ),
};

// Byzantine leader resolves request_B with response_A:
contract.respond_verify_foreign_tx(request_b, response_a)
    .expect("contract accepts forged response — no payload_hash binding check");

// Caller of request_B now holds a valid MPC signature over Bitcoin tx 0xAA,
// not the 0xBB tx they requested. A bridge contract that only checks the
// signature (not the payload_hash content) will accept this as a valid
// attestation of tx 0xBB.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
