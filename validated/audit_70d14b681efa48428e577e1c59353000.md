Audit Report

## Title
Zero-rate oracle causes deposit revert on fresh deployment — (`contracts/agETH/AGETHPoolV3.sol`, `contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.rate` initializes to `0` and `getRate()` returns it unconditionally. Before the first LayerZero `lzReceive` call, `AGETHPoolV3.viewSwapAgETHAmountAndFee` divides by this zero value, causing every `deposit()` call to revert with a division-by-zero panic. The contract sets `isEthDepositEnabled = true` during `initialize()` but cannot fulfill deposits until the oracle receives its first cross-chain update.

## Finding Description
`CrossChainRateReceiver.rate` is declared as `uint256 public rate` with no initialization, defaulting to `0`. [1](#0-0) 

`getRate()` returns `rate` with no zero-guard. [2](#0-1) 

`AGETHPoolV3.viewSwapAgETHAmountAndFee(uint256)` fetches the rate and uses it as a divisor at line 168: [3](#0-2) 

The token deposit path at line 194 has the same issue: [4](#0-3) 

Both `initialize()` and `setAgETHOracle()` accept any non-zero oracle address without verifying `getRate() != 0`: [5](#0-4) [6](#0-5) 

This is inconsistent with `addSupportedToken()`, which explicitly rejects a zero-rate oracle: [7](#0-6) 

The rate is only set when `lzReceive` is called by the LayerZero endpoint: [8](#0-7) 

## Impact Explanation
All ETH and token deposits to `AGETHPoolV3` revert during the window between contract deployment and the first successful `lzReceive` call. The contract sets `isEthDepositEnabled = true` in `initialize()` but cannot fulfill deposits. No funds are lost — reverts return ETH to callers. **Impact: Low — contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
This occurs on every fresh deployment. The LayerZero relayer introduces a non-zero latency between deployment and the first cross-chain rate message. Any user who calls `deposit()` in that window receives a confusing division-by-zero panic revert. No attacker action is required; the condition is a natural consequence of the deployment state.

## Recommendation
Add a zero-rate guard in `initialize()` and `setAgETHOracle()`, mirroring the existing check in `addSupportedToken()`:

```solidity
if (IOracle(_agETHOracle).getRate() == 0) revert UnsupportedOracle();
```

Alternatively, add a zero-check inside `viewSwapAgETHAmountAndFee` before dividing and revert with a descriptive error such as `OracleNotInitialized()`.

## Proof of Concept

```solidity
// 1. Deploy AGETHRateReceiver (inherits CrossChainRateReceiver; rate == 0, lzReceive never called)
// 2. Deploy AGETHPoolV3 with agETHOracle = address(rateReceiver)
//    initialize() sets isEthDepositEnabled = true
// 3. Any user calls:
vm.expectRevert(); // division-by-zero panic (Solidity 0.8 arithmetic check)
pool.deposit{value: 1 ether}("ref");
// Reverts at: agETHAmount = amountAfterFee * 1e18 / agETHToETHrate (agETHToETHrate == 0)
```
