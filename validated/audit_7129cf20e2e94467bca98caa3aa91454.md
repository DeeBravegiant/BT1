### Title
NFT Royalty Payment Bypass via Integer Truncation to Zero - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
`compute_royalty_amount` uses integer floor division that silently truncates to zero for small offered amounts. When the computed royalty is zero, the offer construction skips the royalty coin entirely and also excludes the trade price from the `trade_prices_list` passed to the NFT ownership-layer CLVM puzzle, meaning the puzzle itself does not enforce any royalty payment. An unprivileged buyer can craft an offer for a sufficiently small fungible amount to bypass royalty payments owed to the NFT creator.

### Finding Description
`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

`MAX_ROYALTY_BASIS_POINTS` is 10000. For any `offered_amount` where `abs(offered_amount) * percentage < 10000`, the result is 0. For example, with a 1% royalty (`percentage=100`), any offer of fewer than 100 mojos produces a royalty of 0. The project's own test explicitly documents this:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

When `compute_royalty_amount` returns 0, two downstream effects occur in `make_nft1_offer`:

**Effect 1 – Royalty coin skipped entirely.** The royalty payment sum is 0, so the royalty spend is skipped:

```python
if sum(p.amount for _, p in payments) == 0:
    continue
``` [3](#0-2) 

**Effect 2 – Trade price excluded from the CLVM puzzle's `trade_prices_list`.** The filter at line 896 removes any trade price whose royalty would be zero:

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [4](#0-3) 

The NFT ownership-layer CLVM puzzle enforces royalties only based on the `TRADE_PRICES_LIST` embedded in the spend solution. If that list is empty, the puzzle imposes no royalty obligation. The offer is therefore valid on-chain with zero royalty paid.

Additionally, the announcement guard that would bind the royalty coin to the offer bundle is also skipped (only added when `p.amount > 0`):

```python
for launcher_id, p in payments
    if p.amount > 0
``` [5](#0-4) 

### Impact Explanation
An unprivileged buyer can create a valid, on-chain-accepted offer to purchase a royalty-bearing NFT while paying zero royalty to the creator. The royalty creator — a third party distinct from the NFT owner — receives nothing despite having embedded a non-zero royalty percentage in the NFT at mint time. This is an unauthorized accounting change affecting NFT offer settlement: the royalty obligation encoded in the singleton is silently voided for small-amount trades.

### Likelihood Explanation
Any wallet user can construct such an offer. No special privileges, leaked keys, or cryptographic breaks are required. The threshold below which royalties are bypassed depends on the royalty percentage: for a 1% royalty (`percentage=100`), any offer below 100 mojos bypasses royalties; for a 0.01% royalty (`percentage=1`), any offer below 10,000 mojos bypasses royalties. CAT mojos (which have a different unit scale) make this threshold practically reachable in real trades.

### Recommendation
Add a guard in `compute_royalty_amount` (or at the call site in `make_nft1_offer`) that rejects or warns when a non-zero `percentage` produces a zero royalty amount, analogous to the fix described in the reference report. At minimum, document the truncation behavior clearly so NFT creators understand that royalties are not enforced for small-denomination trades.

```python

### Proof of Concept
1. Mint an NFT with `royalty_percentage=100` (1%).
2. Create an offer requesting the NFT and offering 50 mojos of XCH.
3. `compute_royalty_amount(-50, 1, 100)` → `50 * 100 // 10000` = **0**.
4. `payment_sum == 0` → royalty coin spend is skipped (line 911–912).
5. `50 * 100 // 10000 == 0` → trade price excluded from `trade_prices_list` (line 896).
6. The NFT ownership-layer puzzle is solved with an empty `trade_prices_list`; no royalty is enforced.
7. The NFT owner accepts the offer; the NFT transfers for 50 mojos; the royalty creator receives 0 mojos. [6](#0-5) [7](#0-6)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
