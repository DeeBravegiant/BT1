### Title
Deposit Refund Misdirection in `DelegateAction` Inner Receipts Allows Unprivileged Users to Drain Relayer Balance - (File: runtime/runtime/src/actions.rs)

### Summary

When a `DelegateAction` inner receipt fails on the receiver's shard, the protocol unconditionally sends the deposit refund to the `predecessor_id` of the inner receipt — which is the DelegateAction **sender** (Alice), not the **relayer** who actually paid the deposit. An unprivileged user can exploit this by crafting a `DelegateAction` with a large deposit attached to a function call designed to fail, causing the relayer to permanently lose the deposited NEAR tokens while Alice receives them.

### Finding Description

In `apply_delegate_action`, the inner receipt spawned from a `DelegateAction` is constructed with `predecessor_id` set to `sender_id` (Alice): [1](#0-0) 

The code comment at lines 471–475 explicitly acknowledges the consequence:

> "If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction. Gas is refunded to the signer, this is Relayer." [2](#0-1) 

When the inner receipt fails on the receiver's shard, `refund_unspent_gas_and_deposits` in `lib.rs` issues the deposit refund to `receipt.balance_refund_receiver()`, which resolves to `predecessor_id` (Alice): [3](#0-2) 

`balance_refund_receiver()` returns `predecessor_id` unless an explicit `refund_to` override is set (only available in `ActionReceiptV2`). The inner receipt created by `apply_delegate_action` uses `ReceiptEnum::Action` (V0), so no override is possible: [4](#0-3) 

The architecture documentation confirms this is a protocol-enforced behavior with a concrete financial incentive for abuse: [5](#0-4) 

### Impact Explanation

The corrupted protocol value is the **relayer's NEAR token account balance**. The relayer pays the deposit when the outer transaction is processed (gas is purchased from the relayer's account at `verify_and_charge_transaction`). When the inner receipt fails, the deposit — which can be arbitrarily large, up to the relayer's full balance — is transferred to Alice's account, not returned to the relayer. The relayer's `amount` field in the trie state is permanently reduced by the deposit amount; Alice's `amount` is permanently increased by the same amount. This is a direct, irreversible token transfer from relayer to attacker.

### Likelihood Explanation

Any unprivileged user with a NEAR account can:
1. Create a `DelegateAction` targeting a contract method that is guaranteed to fail (e.g., a non-existent method, a method that panics, or a contract that has been deleted).
2. Attach a large `deposit` to the inner `FunctionCallAction`.
3. Submit the `SignedDelegateAction` to any relayer offering meta transaction services.

The relayer has no on-chain mechanism to prevent this. The only mitigation is off-chain pre-validation by the relayer, but this is not enforced by the protocol. The attack is cheap for Alice (she only needs to sign a message) and costly for the relayer (loses the full deposit). The attack can be repeated indefinitely until the relayer's balance is exhausted.

### Recommendation

1. **Protocol-level fix**: Modify `apply_delegate_action` to create the inner receipt using `ReceiptEnum::ActionV2` with `refund_to: Some(action_receipt.signer_id().clone())` (the relayer), so that deposit refunds on failure are returned to the relayer rather than Alice.
2. **Interim mitigation**: Relayer implementations must validate that the inner actions' deposits are zero, or that the target contract and method are known to succeed, before accepting a `DelegateAction`.

### Proof of Concept

1. Alice creates a `DelegateAction` with `receiver_id = some_contract`, `actions = [FunctionCall { method_name: "nonexistent", deposit: 10_000 NEAR, ... }]`, signed with her key.
2. Alice submits the `SignedDelegateAction` to a relayer.
3. The relayer wraps it in a transaction (paying 10,000 NEAR deposit + gas) and submits to the chain.
4. On Alice's shard, `apply_delegate_action` creates an inner receipt with `predecessor_id = alice`, `signer_id = relayer`.
5. On `some_contract`'s shard, the function call fails (`MethodNotFound`).
6. `refund_unspent_gas_and_deposits` issues `Receipt::new_balance_refund(receipt.balance_refund_receiver(), 10_000 NEAR)` — `balance_refund_receiver()` returns `predecessor_id = alice`.
7. Alice's balance increases by 10,000 NEAR; the relayer's balance is permanently reduced by 10,000 NEAR. [6](#0-5) [3](#0-2) [4](#0-3)
