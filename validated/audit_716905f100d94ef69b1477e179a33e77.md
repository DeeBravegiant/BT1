### Title
Unenforced `farmer_reward_address_override` Allows Any Connected Harvester to Redirect Farmer Block Rewards — (`File: chia/farmer/farmer_api.py`)

### Summary
The farmer unconditionally accepts `farmer_reward_address_override` from any connected harvester in both `NewProofOfSpace` and `RespondSignatures` protocol messages, redirecting the 2 XCH farmer block reward to an arbitrary attacker-controlled address. The CHIP-22 fee-quality "check" that is supposed to gate this override only emits a log warning and never blocks the action — a direct analog to the Solidity pattern where a validation exists but does not prevent the unauthorized state change.

### Finding Description

**Vulnerability class:** Auth bypass / insufficient guard — a check is present but non-enforcing.

**Two entry points, same root cause:**

**Entry point 1 — `NewProofOfSpace.farmer_reward_address_override`**

In `FarmerAPI.new_proof_of_space`, when a harvester sends a `NewProofOfSpace` with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`: [1](#0-0) 

That function computes a `fee_quality` and compares it to the harvester-supplied `applied_fee_threshold`, but **only logs a warning** when the threshold is violated — it never returns an error, raises an exception, or prevents the proof from proceeding: [2](#0-1) 

The proof-of-space flow continues regardless, and `RequestSignatures` is sent to the harvester.

**Entry point 2 — `RespondSignatures.farmer_reward_address_override`**

In `_process_respond_signatures`, the `farmer_reward_address_override` field from the harvester's `RespondSignatures` message is accepted with **zero validation** — no fee-quality check, no log, nothing: [3](#0-2) 

The overridden address is then placed directly into `DeclareProofOfSpace.farmer_reward_address` and sent to the full node: [4](#0-3) 

The `RespondSignatures` dataclass confirms `farmer_reward_address_override` is a first-class protocol field: [5](#0-4) 

### Impact Explanation

When a block is won, the farmer reward coin (2 XCH) is created at `farmer_reward_address` as specified in `DeclareProofOfSpace`. By overriding this field, a malicious harvester permanently diverts the farmer's block reward to an attacker-controlled puzzle hash. The farmer receives nothing; the loss is irreversible on-chain.

This falls squarely within: **High — bypass of farmer authorization that enables payout redirection**.

### Likelihood Explanation

- CHIP-22 explicitly enables third-party harvesters, making this a realistic and common deployment scenario.
- The attacker only needs a valid TLS certificate to connect as a harvester (which the farmer operator issues when onboarding a third-party harvester).
- The exploit requires setting a single field in a standard protocol message — no cryptographic material, no key compromise, no special timing.
- The farmer has no mechanism to detect or prevent this at runtime; the only signal is a log line that most operators will never see.

### Recommendation

The farmer must enforce the fee-quality convention, not merely log it. Specifically:

1. In `new_proof_of_space`: if `farmer_reward_address_override` is set and `fee_quality > applied_fee_threshold` (or `fee_info` is absent), **drop the proof** and return `None` rather than continuing.
2. In `_process_respond_signatures`: apply the same fee-quality gate before accepting `response.farmer_reward_address_override`. If the override was not pre-authorized by a passing `NewProofOfSpace` fee check, reject it.
3. Consider tracking, per `(sp_hash, plot_identifier)`, whether the fee-quality check passed during `new_proof_of_space`, and only honour the `RespondSignatures` override when that flag is set.

### Proof of Concept

A malicious harvester connected to a victim farmer:

1. Receives a `NewSignagePointHarvester` from the farmer.
2. Finds a valid proof of space for the signage point.
3. Sends `NewProofOfSpace` with `farmer_reward_address_override = attacker_puzzle_hash` and `fee_info = None` (or any `applied_fee_threshold`).
4. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`, logs a warning, and continues — sending `RequestSignatures` back to the harvester.
5. The harvester responds with `RespondSignatures` (also with `farmer_reward_address_override = attacker_puzzle_hash`).
6. `_process_respond_signatures` unconditionally sets `farmer_reward_address = attacker_puzzle_hash`.
7. `DeclareProofOfSpace` is sent to the full node with the attacker's address as the farmer reward destination.
8. The block is farmed; the 2 XCH farmer reward is paid to the attacker.

The farmer operator sees only a log warning (if `fee_info` was provided) or nothing at all (if `fee_info` is `None` and the override is only in `RespondSignatures`). [6](#0-5) [7](#0-6)

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
