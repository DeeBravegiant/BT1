### Title
`ForeignTxSignPayloadV1.compute_msg_hash()` Omits NEAR Network/Contract Binding, Enabling Cross-Deployment Replay of Foreign Transaction Verification Signatures — (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`)

---

### Summary

`ForeignTxSignPayloadV1` — the struct whose Borsh-serialized hash the MPC network signs for every `verify_foreign_transaction()` request — contains only `{request, values}` with no binding to the NEAR MPC contract account ID, NEAR network ID, or domain ID. If the same MPC key material exists on two NEAR deployments (e.g., after a NEAR hard fork, or during a contract migration that preserves key shares), a `ForeignTxSignPayload` signature obtained on one deployment is cryptographically valid on the other. A downstream bridge contract using `ForeignChainSignatureVerifier` cannot distinguish between signatures from different deployments, enabling cross-deployment replay and double-spend conditions.

---

### Finding Description

**Root cause — static, context-free payload hash:**

`ForeignTxSignPayloadV1` is defined as:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

and `compute_msg_hash()` computes:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload { V1 { request, values } }))
``` [1](#0-0) 

The hash includes no NEAR contract account ID, no NEAR network ID (mainnet/testnet), and no domain ID. This is the direct analog to ERC20Permit's `DOMAIN_SEPARATOR` being fixed at deployment time without a dynamic `block.chainid` check.

**Compounding factor — zero tweak (root key signing):**

`build_signature_request()` in the node's ForeignTx provider hardcodes `Tweak::new([0u8; 32])`:

```rust
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),   // root key, no derivation
    domain: request.domain_id,
})
``` [2](#0-1) 

Every `verify_foreign_transaction()` request is signed with the ForeignTx domain's **root key** (no per-request key derivation). This means the same root public key is used for all ForeignTx signatures, making the absence of a network/contract binding in the payload hash the sole protection against cross-deployment replay — and that protection is absent.

**Verification side — no network binding checked:**

`ForeignChainSignatureVerifier.verify_signature()` reconstructs the payload from `{request, values}` and verifies the ECDSA signature against a caller-supplied public key. No NEAR contract account ID or network ID is included in the reconstructed payload:

```rust
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: self.request,
    values: self.expected_extracted_values,
});
let expected_payload_hash = expected_payload.compute_msg_hash()...;
``` [3](#0-2) 

A bridge contract on fork B that trusts the MPC public key and calls `verify_signature()` will accept a `(payload_hash, signature)` pair obtained on fork A, because the hash is identical on both forks for the same `{request, values}`.

---

### Impact Explanation

**Impact: High — Cross-chain replay / forged foreign-chain verification enabling invalid bridge execution or double-spend conditions.**

After a NEAR hard fork where both forks retain the same MPC contract state (and therefore the same distributed key shares), the MPC root public key is identical on both forks. An attacker who obtains a valid `VerifyForeignTransactionResponse` on fork A (e.g., proving that a Bitcoin transaction with `tx_id = X` was confirmed) can submit the same `(payload_hash, signature)` on fork B. A bridge contract on fork B that calls `ForeignChainSignatureVerifier::verify_signature()` will accept it as a valid MPC attestation, because:

1. The `msg_hash` is `SHA-256(borsh({request, values}))` — identical on both forks for the same inputs.
2. The signature was produced with the root key, which is the same on both forks.
3. The verifier checks only `(payload_hash, signature, public_key)` — no fork/network binding.

This enables a double-spend: the same foreign-chain transaction is accepted as "verified" on both forks, allowing an attacker to claim bridge funds twice.

The same scenario applies during a contract migration that preserves key shares (e.g., migrating the MPC contract to a new NEAR account ID while keeping the same distributed key material).

---

### Likelihood Explanation

**Likelihood: Medium.**

NEAR has undergone protocol upgrades and the MPC contract is a long-lived, critical piece of infrastructure. A hard fork of NEAR (whether contentious or accidental) would leave both forks with identical MPC key shares and an identical `ForeignTxSignPayload` hash space. Contract migrations that preserve key shares are also a realistic operational scenario. The attacker-controlled entry path is straightforward: call `verify_foreign_transaction()` on one fork, collect the `VerifyForeignTransactionResponse`, and replay it on the other fork. No threshold collusion or privileged access is required — the attacker only needs to be an unprivileged caller on one fork.

---

### Recommendation

**Short term:** Include the NEAR MPC contract account ID and NEAR network ID in `ForeignTxSignPayloadV1` (or in a wrapper struct used by `compute_msg_hash()`), so the signed hash is bound to a specific deployment:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub contract_id: AccountId,   // e.g., "v1.signer.near"
    pub network_id: String,       // e.g., "mainnet"
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
```

Alternatively, include the `domain_id` in the payload so that at minimum signatures cannot be replayed across domains.

**Long term:** Audit all other signed payloads (e.g., `SignatureRequest`, CKD outputs) for the same missing binding. Document the risks associated with NEAR hard forks and contract migrations for the MPC key infrastructure, and develop mitigation strategies (e.g., key rotation after a fork).

---

### Proof of Concept

1. Deploy the MPC contract on NEAR mainnet. The ForeignTx domain DKG produces root public key `PK`.
2. NEAR mainnet is hard-forked. Both fork A and fork B retain the same MPC contract state, including the same key shares, so both have root public key `PK`.
3. On fork A, Alice calls `verify_foreign_transaction({ request: Bitcoin(tx_id=X, ...), domain_id: D, payload_version: V1 })`.
4. The MPC nodes on fork A query Bitcoin, confirm `tx_id=X`, construct `ForeignTxSignPayloadV1 { request: Bitcoin(tx_id=X,...), values: [BlockHash(H)] }`, compute `msg_hash = SHA-256(borsh(payload))`, and sign it with the root key of domain D. Alice receives `VerifyForeignTransactionResponse { payload_hash: msg_hash, signature: sig }`.
5. On fork B, Alice submits the same `(payload_hash, signature)` to a bridge contract that calls `ForeignChainSignatureVerifier::verify_signature(response, PK)`.
6. The verifier reconstructs `ForeignTxSignPayloadV1 { request: Bitcoin(tx_id=X,...), values: [BlockHash(H)] }`, computes the same `msg_hash` (no network binding), and verifies `sig` against `PK` — which succeeds because `PK` is identical on fork B.
7. The bridge contract on fork B accepts the attestation and releases funds, even though the MPC nodes on fork B never independently verified `tx_id=X`. [4](#0-3) [5](#0-4) [6](#0-5)

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }

    foreign_tx_payload: &dtos::ForeignTxSignPayload,

    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =

        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
