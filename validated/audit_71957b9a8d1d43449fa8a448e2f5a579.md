Let me look at the exact `respond_verify_foreign_tx` validation logic and the Borsh encoding of `ForeignChainRpcRequest` to confirm the discriminant separation.

### Title
`respond_verify_foreign_tx` Accepts Any Valid Signature Without Binding `payload_hash` to the Submitted `request` — Cross-Chain EVM Replay via Single Byzantine Node — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted signature is cryptographically valid over `response.payload_hash`. It never reconstructs the expected payload hash from the `request` parameter and compares the two. A single Byzantine MPC node (below signing threshold) that has already obtained a legitimate MPC signature over a Polygon payload can submit that signature as the response for a pending Arbitrum request with the same inner `EvmRpcRequest`. The contract accepts it, resolves the Arbitrum yield, and returns a `VerifyForeignTransactionResponse` whose `payload_hash` attests to a Polygon observation — not an Arbitrum one.

---

### Finding Description

**Missing invariant in `respond_verify_foreign_tx`**

The function performs two checks:

1. The ECDSA signature is valid over `response.payload_hash` using the domain's root public key.
2. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

There is no third check: **the contract never verifies that `response.payload_hash` is the hash of a payload whose embedded `ForeignChainRpcRequest` matches the `request` argument**. The two values are completely decoupled.

**Why the Borsh discriminants matter**

`ForeignChainRpcRequest` carries no `#[repr(u8)]` / `#[borsh(use_discriminant = true)]`, so Borsh assigns sequential discriminants by declaration order. `Arbitrum` is variant 7 and `Polygon` is variant 8. [2](#0-1) 

Therefore `hash(ForeignTxSignPayloadV1 { request: Arbitrum(R), values: V })` ≠ `hash(ForeignTxSignPayloadV1 { request: Polygon(R), values: V })`. The two payloads are distinct, but the contract never checks which one was actually signed.

**Why a single Byzantine node suffices**

Followers independently fetch the stored request and build the payload themselves: [3](#0-2) 

So a Byzantine leader cannot trick followers into co-signing a Polygon payload for an Arbitrum request — the followers would build the Arbitrum payload and the MPC round would fail. The attack does not require corrupting the signing round at all.

**Concrete attack path (single Byzantine node, below threshold)**

1. Both Arbitrum and Polygon are whitelisted. User A submits `verify_foreign_transaction` for Polygon tx T. The MPC network legitimately signs `ForeignTxSignPayload::V1 { request: Polygon(evm_req), values: [EvmExtractedValue::BlockHash(H)] }`, producing `(polygon_hash, sig)`.
2. User B submits `verify_foreign_transaction` for Arbitrum tx T (same `EvmRpcRequest` inner value). The contract stores the pending request keyed by `VerifyForeignTransactionRequest { request: Arbitrum(evm_req), ... }`.
3. The Byzantine node calls `respond_verify_foreign_tx(request = Arbitrum_key, response = { payload_hash: polygon_hash, signature: sig })`.
4. The contract checks: is `sig` valid over `polygon_hash`? **Yes** (legitimately produced). Is `Arbitrum_key` in pending? **Yes**. Both pass.
5. The contract resolves the Arbitrum yield with the Polygon-attested response. User B receives `VerifyForeignTransactionResponse { payload_hash: polygon_hash, signature: sig }`. [4](#0-3) 

**Partial mitigation in the SDK (not in the contract)**

The SDK's `ForeignChainSignatureVerifier::verify_signature` does reconstruct the expected payload hash from the original request and compares it to `response.payload_hash`: [5](#0-4) 

Bridge consumers using the SDK would catch the mismatch. However, the contract itself is the authoritative on-chain source of truth. It accepted and forwarded a mismatched response. Bridge consumers that trust the contract's acceptance, implement their own verification, or omit the SDK check are directly exploitable.

---

### Impact Explanation

A single Byzantine MPC node can make the NEAR contract attest that an Arbitrum transaction was verified when the actual signed payload attests to a Polygon observation. Any bridge consumer that relies on the contract's acceptance (rather than independently re-verifying `payload_hash` against the original request) will process an invalid bridge event — enabling cross-chain replay and potential double-spend conditions.

---

### Likelihood Explanation

Requires one Byzantine attested MPC participant (below signing threshold) and both Arbitrum and Polygon to be whitelisted — exactly the precondition stated. No threshold collusion, no key leakage, no network-level attack. The Byzantine node only needs to capture a legitimately produced Polygon signature and submit it against a pending Arbitrum request with the same inner `EvmRpcRequest`.

---

### Recommendation

In `respond_verify_foreign_tx`, after verifying the signature, reconstruct the expected payload hash from the `request` parameter and assert it equals `response.payload_hash`. Because the contract does not know the extracted `values`, the simplest fix is to commit to the `ForeignChainRpcRequest` separately — e.g., hash only the request portion and include it as a domain separator in the signed payload, or store a request-hash commitment at enqueue time and verify it on respond. At minimum, the contract should reject any response whose `payload_hash` does not begin with a prefix that encodes the correct chain discriminant.

```rust
// Unit test sketch (no external deps needed)
// 1. Enqueue an Arbitrum request
let arbitrum_req = ForeignChainRpcRequest::Arbitrum(EvmRpcRequest {
    tx_id: EvmTxId([0xbb; 32]),
    extractors: vec![EvmExtractor::BlockHash],
    finality: EvmFinality::Finalized,
});
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    domain_id, payload_version: ForeignTxPayloadVersion::V1, request: arbitrum_req.clone(),
});

// 2. Build a Polygon payload (same inner EvmRpcRequest) and sign it with the root key
let polygon_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: ForeignChainRpcRequest::Polygon(EvmRpcRequest {
        tx_id: EvmTxId([0xbb; 32]),
        extractors: vec![EvmExtractor::BlockHash],
        finality: EvmFinality::Finalized,
    }),
    values: vec![ExtractedValue::EvmExtractedValue(
        EvmExtractedValue::BlockHash(Hash256([0xcc; 32])),
    )],
});
let polygon_hash = polygon_payload.compute_msg_hash().unwrap();
let (sig, rec) = root_signing_key.sign_prehash_recoverable(&polygon_hash.0).unwrap();

// 3. Submit the Polygon-signed response against the Arbitrum request key
let result = contract.respond_verify_foreign_tx(
    VerifyForeignTransactionRequest { domain_id, payload_version: V1, request: arbitrum_req },
    VerifyForeignTransactionResponse {
        payload_hash: polygon_hash,   // ← Polygon hash, not Arbitrum
        signature: wrap_sig(sig, rec),
    },
);

// Expected (current behaviour): Ok(()) — contract accepts the mismatched payload
// Expected (correct behaviour): Err(InvalidSignature) or similar
assert!(result.is_ok(), "contract accepted Polygon-attested hash for Arbitrum request");
```
