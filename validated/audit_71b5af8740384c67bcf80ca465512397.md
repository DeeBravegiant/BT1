### Title
Single Byzantine Participant Can Deliver Forged CKD Output for `AppPublicKey` Variant — (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` contract method performs **no on-chain cryptographic verification** of the CKD response when the request uses the legacy `AppPublicKey` variant. A single Byzantine attested participant, strictly below the signing threshold, can submit an arbitrary forged `CKDResponse` that the contract accepts and delivers to the requesting user, bypassing the threshold-agreement requirement for confidential key derivation.

### Finding Description
In `respond_ckd`, the match on `request.app_public_key` has an empty arm for the `AppPublicKey` variant — no verification of the response is performed:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

Every other response path in the contract enforces on-chain correctness:

- `respond` verifies the ECDSA/EdDSA signature against the derived public key before resolving the yield. [2](#0-1) 
- `respond_verify_foreign_tx` verifies the ECDSA signature against the root public key. [3](#0-2) 
- `respond_ckd` with `AppPublicKeyPV` calls `ckd_output_check` to verify the BLS12-381 output against the domain public key. [4](#0-3) 

Only the `AppPublicKey` branch is unguarded. After the participant check passes, `resolve_yields_for` is called unconditionally with whatever `big_y`/`big_c` the caller supplied: [5](#0-4) 

The `AppPublicKey` variant is still publicly callable via `request_app_private_key`: [6](#0-5) 

### Impact Explanation
A single malicious attested participant (one node, strictly below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request on-chain.
2. Call `respond_ckd` with arbitrary `big_y` and `big_c` BLS12-381 G1 points.
3. The contract accepts the forged response — no invariant check exists for this variant — and delivers it to the user through the yield-resume mechanism.

The user receives a wrong confidential key derivation output. If the derived key protects funds or sensitive material, the user loses access to it. If the attacker can craft `big_y`/`big_c` such that they know the resulting plaintext key (e.g., by encrypting a key they control to the user's public key), this escalates to direct theft of assets protected by that derived key.

This breaks the core safety invariant that CKD outputs require threshold-many honest participants to agree off-chain before any result is delivered on-chain — an invariant that is enforced for every other response type.

### Likelihood Explanation
The `AppPublicKey` variant is still accepted by the contract (described as "legacy" but not gated or removed). Any single attested participant can exploit this with a straightforward on-chain call. No special cryptographic knowledge is required beyond being an attested participant and observing the pending request map.

### Recommendation
1. **Immediate**: Deprecate and reject `AppPublicKey` requests at the `request_app_private_key` entry point, directing callers to `AppPublicKeyPV`.
2. **If the variant must be kept**: Require threshold-many participants to submit identical responses before resolving the yield, or add an equivalent on-chain binding check.
3. **Consistency**: Align `respond_ckd` with `respond` and `respond_verify_foreign_tx` so that every response path has an on-chain correctness check before `resolve_yields_for` is called.

### Proof of Concept

```
1. User calls request_app_private_key({app_public_key: AppPublicKey(pk), ...})
   → pending CKD request stored on-chain

2. Malicious attested participant calls:
   respond_ckd(request, CKDResponse { big_y: [attacker_chosen], big_c: [attacker_chosen] })

3. Contract execution path:
   - assert_caller_is_signer()                          ✓ (participant is signer)
   - is_running_or_resharing()                          ✓
   - accept_requests                                    ✓
   - assert_caller_is_attested_participant_...          ✓ (participant is attested)
   - match AppPublicKey(_) => {}                        ← NO VERIFICATION
   - resolve_yields_for(forged_response)                ← forged data delivered

4. User's yield resumes with attacker-supplied big_y / big_c.
   User derives a wrong or attacker-controlled confidential key.
``` [7](#0-6)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
