Audit Report

## Title
Stale `rsETHPrice` Enables Yield Theft via Deposit Before `updateRSETHPrice()` Call - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is a public, permissionless function that updates the stored `rsETHPrice` state variable. Between updates, the stored price becomes stale while underlying LST assets accrue yield. Because `LRTDepositPool.getRsETHAmountToMint()` uses the stale stored `rsETHPrice` as the denominator while using a live Chainlink price as the numerator, an attacker can deposit at the artificially low stale price to receive more rsETH than fair value, then trigger or wait for a price update — extracting yield that belongs to existing rsETH holders.

## Finding Description
`updateRSETHPrice()` at `LRTOracle.sol:87` is `public whenNotPaused` with no access control:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`getRsETHAmountToMint()` at `LRTDepositPool.sol:520` computes the mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` reads a **live** Chainlink price, while `lrtOracle.rsETHPrice()` returns the **stale** cached value. When yield has accrued since the last update, `rsETHPrice` is lower than the true current rate, making the denominator too small and minting excess rsETH to the depositor.

The `pricePercentageLimit` guard at `LRTOracle.sol:252-266` only blocks a non-manager from calling `updateRSETHPrice()` if the price increase **exceeds** the configured threshold. For normal daily staking yield (~0.01% per day), the increase is well within any reasonable threshold, so the attacker can call `updateRSETHPrice()` themselves. Even if the threshold were exceeded, the attacker's over-minted rsETH is already locked in — they simply wait for the manager's routine update. The `minRSETHAmountExpected` slippage parameter in `depositETH`/`depositAsset` is set by the attacker themselves and provides no protection.

The exploit path is:
1. Observe that `rsETHPrice` is stale (live TVL > `rsethSupply * rsETHPrice`).
2. Call `LRTDepositPool.depositETH{value: X}(0, "")` — minted rsETH is inflated because the denominator `rsETHPrice` is below true value.
3. Call `LRTOracle.updateRSETHPrice()` (or wait for the routine update) — price rises to reflect accrued yield.
4. Redeem/sell the over-minted rsETH at the updated higher price.

The dilution is borne by all existing rsETH holders whose entitled price appreciation is partially captured by the attacker's excess shares.

## Impact Explanation
**High — Theft of unclaimed yield.** Each attack cycle extracts a fraction of the yield accrued since the last price update. The attack is repeatable every update interval. Existing holders receive a smaller price increase than they are entitled to because the attacker's over-minted shares dilute the supply before the price is corrected.

## Likelihood Explanation
**Medium.** No privileged role, leaked key, or oracle compromise is required. The attacker only needs to monitor on-chain state (live Chainlink prices vs. stored `rsETHPrice`) and submit two public transactions. The attack is fully self-contained, requires no victim interaction, and is repeatable on every update cycle. The per-cycle profit is bounded by the yield accumulated in the interval, but is economically meaningful at scale.

## Recommendation
1. **Atomic price refresh on deposit:** Call `_updateRsETHPrice()` at the start of `depositETH` and `depositAsset` (or enforce a staleness bound) before computing `rsethAmountToMint`, so the mint price always reflects current TVL.
2. **Compute mint amount from live TVL directly:** Replace the stored `rsETHPrice` denominator in `getRsETHAmountToMint()` with an on-the-fly calculation using `_getTotalEthInProtocol()` and `rsETH.totalSupply()`, bypassing the stale cache entirely.
3. **Deposit fee:** Introduce a small entry fee to make the marginal yield-extraction attack economically unattractive.

## Proof of Concept
**State:** `rsETHPrice = 1.010e18` (last updated 24 h ago); live stETH/ETH Chainlink rate has drifted up so true rsETH price is `1.012e18`. Total rsETH supply = 1 000 rsETH.

1. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(0, "")`.
   - `getAssetPrice(ETH_TOKEN)` → `1e18` (live).
   - `rsETHPrice()` → `1.010e18` (stale).
   - Minted: `100e18 * 1e18 / 1.010e18 ≈ 99.0099 rsETH`.
   - Fair mint at true price: `100e18 * 1e18 / 1.012e18 ≈ 98.8142 rsETH`.
   - **Excess: ≈ 0.1957 rsETH.**

2. Attacker calls `LRTOracle.updateRSETHPrice()`.
   - New TVL includes attacker's 100 ETH; new supply = 1 099.0099 rsETH.
   - `rsETHPrice` updates to `≈ 1.01182e18` (slightly diluted from true `1.012e18`).

3. Attacker redeems `99.0099 rsETH` at `≈ 1.01182e18` → recovers `≈ 100.18 ETH`, profiting `≈ 0.18 ETH` at the expense of the original 1 000 rsETH holders who receive a smaller price increase than they earned.

**Foundry fork test plan:** Fork mainnet, snapshot `rsETHPrice`, advance time by 1 day (allowing Chainlink prices to drift), execute the deposit + `updateRSETHPrice()` sequence, assert that the attacker's rsETH redemption value exceeds their deposit and that the rsETH price increase for pre-existing holders is smaller than it would have been without the attacker's deposit.
