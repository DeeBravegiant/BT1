Audit Report

## Title
Minimum-Progress Rule + Proportional Scaling Desynchronize `curPosInBin` from `token0BalanceScaled`, Stranding LP Token0 and Causing Zero-Output Swaps — (`metric-core/contracts/libraries/SwapMath.sol`)

## Summary

When `curPosInBin = MAX_POS_BIN - 1` and a trader submits a small exact-input buy-token0 swap with `amountSpecifiedRemainingScaled` in the range `[price, price*(1+fee))` (in internal units), the minimum-progress guard in `computeAnalyticalTargetPosForBuyToken0` forces `targetPos = MAX_POS_BIN`, while the proportional scaling correction in `buyToken0InBinSpecifiedIn` floors `out0Scaled` to zero. The pool consumes the trader's token1 input, outputs zero token0, and permanently marks the bin as exhausted for all future buy-token0 swaps — even though the bin's full token0 balance remains intact.

## Finding Description

**Root cause 1 — Minimum-progress rule forces `targetPos = MAX_POS_BIN`**

In `computeAnalyticalTargetPosForBuyToken0`, when `currBinPos = MAX_POS_BIN - 1`, `maxSubCurrPos = 1`. For a small input, the quadratic formula rounds `deltaPos` to zero, and the minimum-progress guard unconditionally sets `deltaPos = 1`: [1](#0-0) 

The clamp `if (deltaPos > maxSubCurrPos)` does not fire because `1 > 1` is false: [2](#0-1) 

So `targetPos = MAX_POS_BIN - 1 + 1 = MAX_POS_BIN`.

**Root cause 2 — Proportional scaling correction floors `out0Scaled` to zero**

Back in `buyToken0InBinSpecifiedIn`, with `targetPos = MAX_POS_BIN` and `currBinPos = MAX_POS_BIN - 1`:

```
out0Scaled = calculateOutputToken0FromBinPosition(token0Balance, MAX_POS_BIN-1, MAX_POS_BIN)
           = token0Balance * 1 / 1 = token0Balance  (full balance)
``` [3](#0-2) 

`totalIn1Scaled ≈ token0Balance * price * (1+fee)` is computed for this full output. Since `totalIn1Scaled > amountSpecifiedRemainingScaled`, the scaling correction fires: [4](#0-3) 

- `delta = 1`, so `scaledDelta = ceilDiv(1 * amountSpecifiedRemainingScaled, totalIn1Scaled) = 1` → `targetPos` stays at `MAX_POS_BIN`.
- `out0Scaled = (token0Balance * amountSpecifiedRemainingScaled) / totalIn1Scaled`. When `amountSpecifiedRemainingScaled < price * (1+fee)` (which holds in the attack window), this floors to **zero**.

**Root cause 3 — Rounding desynchronization between the two functions**

`computeAnalyticalTargetPosForBuyToken0` computes `priceAtCurrPosX64` with floor rounding: [5](#0-4) 

While `buyToken0InBinSpecifiedIn` computes `startingPriceX64` with ceil rounding: [6](#0-5) 

This makes the analytical function slightly over-optimistic, contributing to the divergence.

**State corruption**

After the swap, `binState.token0BalanceScaled -= 0` (unchanged), but `curPosInBin = MAX_POS_BIN` is persisted: [7](#0-6) [8](#0-7) 

Every subsequent buy-token0 swap checks: [9](#0-8) 

`curPosInBinCache = MAX_POS_BIN = type(uint104).max` satisfies the second condition, so the bin is permanently skipped despite holding its full token0 balance.

**Existing guard is insufficient**

The early-exit guard at line 550 (`(amountSpecifiedRemainingScaled << 64) < startingPriceX64`) only blocks inputs strictly below the price of 1 internal token0 unit. The attack window `[price, price*(1+fee))` sits above this threshold, so the guard does not fire. [10](#0-9) 

## Impact Explanation

1. **Swap conservation failure**: The trader pays `amountSpecifiedRemainingScaled` units of token1 and receives zero token0. The pool accepts the input and emits no output — a direct loss to the trader.
2. **Stranded LP token0**: The bin's entire token0 balance becomes inaccessible to all future buy-token0 swaps. LPs can only recover it via `removeLiquidity`, not through normal trading — a broken core pool functionality causing loss of LP-owed assets.
3. **Bad-price execution for subsequent traders**: Future buy-token0 swaps skip the stranded bin and fill from the next (higher-priced) bin, paying a worse price than the pool's actual liquidity would permit.

Severity: **High** — direct loss of trader funds (zero output for non-zero input), stranded LP principal, and bad-price execution, all triggered by an unprivileged caller with no special access.

## Likelihood Explanation

- `curPosInBin = MAX_POS_BIN - 1` is a natural state reached by any buy-token0 swap that nearly exhausts a bin.
- The attacker submits a swap with `amountSpecifiedRemainingScaled` in the range `[price, price*(1+fee))` in internal units. For a pool with 18-decimal token1 and ETH/USDC price ~3000, this means providing 3000–3008 wei of token1 — a trivially small and feasible amount.
- No privileged access, no special token behavior, no oracle manipulation required.
- The attack is repeatable: after the bin is stranded, the attacker can move to the next bin and repeat.

## Recommendation

1. **Guard against zero output after scaling correction**: After the block at lines 624–634, if `out0Scaled == 0`, return `(currBinPos, 0, 0, 0, 0)` rather than consuming input and advancing the cursor to `MAX_POS_BIN`.
2. **Bound the minimum-progress rule**: Before applying `deltaPos = 1`, verify that the implied `out0Scaled` for a 1-step advance is non-zero. If it would be zero, return early with no swap.
3. **Synchronize rounding**: Use the same rounding direction (floor or ceil) for `priceAtCurrPosX64` in `computeAnalyticalTargetPosForBuyToken0` and `startingPriceX64` in `buyToken0InBinSpecifiedIn`.

## Proof of Concept

**Setup**:
- Pool with 18-decimal token0 (e.g. WETH) and 18-decimal token1.
- `upperPriceX64 > ONE_X64` (price of token0 > 1 in internal units).
- `curPosInBin = MAX_POS_BIN - 1` (reached naturally by a prior buy-token0 swap).
- `binState.token0BalanceScaled = T > 0`.
- Fee = 0.3% (`feeX64 = 0.003 * ONE_X64`).

**Attack** (Foundry unit test):
```solidity
// price = 3000 in internal units → priceX64 = 3000 * ONE_X64
// Attack window: amountSpecified in [3000, 3009) internal units
swap(attacker, zeroForOne=false, amountSpecified=3000, priceLimitX64=type(uint128).max, ...)
```

**Trace**:
1. `amountSpecifiedRemainingScaled = 3000`.
2. Guard at L550: `3000 * 2^64 >= startingPriceX64 ≈ 3000 * 2^64` → does not fire.
3. `computeAnalyticalTargetPosForBuyToken0(MAX_POS_BIN-1, MAX_POS_BIN, 3000, T, ...)`: quadratic gives `deltaPos = 0` → minimum progress → `deltaPos = 1` → `targetPos = MAX_POS_BIN`.
4. `out0Scaled = T * 1 / 1 = T` (full balance).
5. `totalIn1Scaled ≈ T * 3000 * 1.003 ≈ 3009*T`.
6. Scaling correction: `scaledDelta = ceilDiv(1 * 3000, 3009*T) = 1` → `targetPos = MAX_POS_BIN`; `out0Scaled = (T * 3000) / (3009*T) = 0`.
7. `_saveBinState`: `token0BalanceScaled = T` (unchanged).
8. `_finalizeSwap`: `curPosInBin = MAX_POS_BIN`.
9. Trader receives 0 token0, pays 3000 units of token1.
10. All subsequent buy-token0 swaps: `curPosInBinCache >= type(uint104).max` → bin skipped, `T` units of token0 permanently stranded.

**Assertions**:
- `out0Scaled == 0` while `totalIn1Scaled == 3000 > 0` — conservation violated.
- `curPosInBin == MAX_POS_BIN` while `token0BalanceScaled == T > 0` — cursor/balance invariant violated.
- Next buy-token0 swap fills from a higher-priced bin — bad-price execution.
