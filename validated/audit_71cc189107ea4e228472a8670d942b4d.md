Audit Report

## Title
`onlyBeforeInitialize2` Modifier Always Reverts Due to Missing `isInitialized2()` Implementation — (`contracts/utils/UnlockedWithdrawalsInitializer.sol`)

## Summary

`UnlockedWithdrawalsInitializer` declares `isInitialized2()` in its local `ILRTWithdrawalManagerView` interface but this function is never implemented in `LRTWithdrawalManager`. The `onlyBeforeInitialize2` modifier's `catch` block unconditionally reverts with `WithdrawalManagerAlreadyInitialized2()` for any error, including the revert caused by calling a non-existent selector. This permanently disables `processChunk` and `finalizeInitialize2`, preventing `unlockedWithdrawalsCount` from being seeded for existing deployments and causing arithmetic underflow reverts when users attempt to complete pre-existing unlocked withdrawals.

## Finding Description

`ILRTWithdrawalManagerView` declares `isInitialized2() external view returns (bool)` at line 19 of `UnlockedWithdrawalsInitializer.sol`, but a grep across the entire codebase confirms this function exists **only** in that interface — it is never implemented in `LRTWithdrawalManager`.

The `onlyBeforeInitialize2` modifier (lines 61–68) calls `_withdrawalManager().isInitialized2()` inside a `try/catch`. When `LRTWithdrawalManager` receives the call, it has no matching selector and no `fallback()`, so the EVM reverts with empty return data. The `catch` block fires and unconditionally reverts with `WithdrawalManagerAlreadyInitialized2()`. There is no code path that allows the modifier to pass through.

Both `processChunk` (line 81) and `finalizeInitialize2` (line 112) carry this modifier, making them permanently unreachable. `initialize2` on `LRTWithdrawalManager` (lines 109–121) is therefore never called through the helper. For existing deployments, `unlockedWithdrawalsCount[asset]` remains `0`. When a user calls `completeWithdrawal` for a pre-existing unlocked request, `_processWithdrawalCompletion` executes `unlockedWithdrawalsCount[asset]--` (line 717) on a zero value, triggering a Solidity 0.8 checked-arithmetic underflow revert.

## Impact Explanation

Users holding pre-existing unlocked withdrawal requests cannot complete them — every call to `completeWithdrawal` reverts with an underflow. This constitutes **temporary freezing of funds** (Medium), because an admin who holds or can acquire the `UNLOCKED_WITHDRAWAL_INITIALIZER` role can still call `initialize2` directly on `LRTWithdrawalManager`, bypassing the broken helper. If no such remediation is taken, the freeze becomes permanent for all affected users.

Additionally, `hasUnlockedWithdrawals` returns `false` when the count is `0`, which allows `sweepRemainingAssets` to drain the contract balance to the treasury while real withdrawal obligations exist, compounding user losses.

## Likelihood Explanation

The failure is immediate and total: the very first call to `processChunk` or `finalizeInitialize2` reverts with no conditional path around it. Any deployment that relies on `UnlockedWithdrawalsInitializer` — the contract's stated purpose — is fully affected. No attacker action is required; the broken modifier fires on normal operator/manager usage.

## Recommendation

1. **Implement `isInitialized2()` in `LRTWithdrawalManager`**: add a `bool private _initialized2` storage flag set to `true` at the end of `initialize2`, and expose it as a public view function. This is the minimal fix.

2. **Alternatively, invert the catch logic**: a revert from `isInitialized2()` means the function does not exist, i.e., `initialize2` has not yet been called. Change the modifier so the `catch` block is a no-op (allowing execution to proceed) and only revert inside the `returns` branch when `isInitialized == true`.

3. **Add an integration test** that calls `processChunk` against the actual `LRTWithdrawalManager` to catch interface/implementation mismatches before deployment.

   → EVM calls LRTWithdrawalManager with isInitialized2() selector.
