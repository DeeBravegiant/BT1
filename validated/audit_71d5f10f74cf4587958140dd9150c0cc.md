Audit Report

## Title
Uninitialized `minRsEthAmountToWithdraw` Enables Dust-Entry Queue Flooding, Temporarily Freezing Legitimate Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`minRsEthAmountToWithdraw` is declared as a per-asset mapping in `LRTWithdrawalManager` but is never initialized in `initialize()`, defaulting to `0` for every asset. Combined with `minAmountToDeposit` also defaulting to `0` in `LRTDepositPool`, an unprivileged attacker can cheaply acquire dust rsETH and flood the withdrawal queue with arbitrarily many 1-wei entries. Because the queue is strictly FIFO and `_unlockWithdrawalRequests` must traverse all preceding entries before reaching any legitimate request, legitimate withdrawals are temporarily frozen until the operator exhausts the dust backlog.

## Finding Description

**Root cause — uninitialized minimum:**

`minRsEthAmountToWithdraw` is declared at line 35 of `LRTWithdrawalManager.sol` but `initialize()` (lines 90–98) only sets `withdrawalDelayBlocks` and never touches this mapping, leaving every asset's minimum at the Solidity default of `0`.

The guard in `initiateWithdrawal` (lines 162–164) is:
```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```
When `minRsEthAmountToWithdraw[asset] == 0`, the second condition `rsETHUnstaked < 0` is always false for a `uint256`, so any call with `rsETHUnstaked >= 1` passes unchecked.

**Dust rsETH acquisition:**

`LRTDepositPool.minAmountToDeposit` is similarly never initialized (line 30, `initialize()` lines 45–52), so `_beforeDeposit` (line 657) only blocks a zero-value deposit. An attacker deposits a tiny amount of ETH, receives dust rsETH via `getRsETHAmountToMint`, and repeats.

**Queue flooding:**

Each accepted `initiateWithdrawal` call appends a `WithdrawalRequest` and increments `nextUnusedNonce[asset]` (lines 751–757). The only economic bound is `assetsCommitted[asset]` (line 173), which grows by `expectedAssetAmount ≈ 1 wei` per dust entry — allowing up to `totalAvailableAssets` (in wei) entries, an astronomically large number for any real deployment.

**FIFO freeze mechanism:**

`_unlockWithdrawalRequests` (lines 790–815) iterates from `nextLockedNonce` up to the caller-supplied `firstExcludedIndex`. The loop contains two early-exit conditions:
1. `if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;` — if the front-of-queue entry has not yet passed the 8-day delay, the loop exits immediately and `nextLockedNonce` does not advance.
2. `if (availableAssetAmount < payoutAmount) break;` — insufficient assets.

Because the queue is strictly ordered by nonce, dust entries inserted before legitimate requests act as a mandatory prefix. During the 8-day delay window the loop breaks on the very first dust entry, preventing any advancement of `nextLockedNonce`. After the delay passes, the operator must iterate through every dust entry (in batches bounded by `firstExcludedIndex`) before `nextLockedNonce` reaches any legitimate request. With 50,000 dust entries this requires 50+ separate `unlockQueue` transactions, each consuming significant gas, during which all legitimate withdrawals remain locked.

The admin setter `setMinRsEthAmountToWithdraw` (lines 330–333) exists but is never called during deployment, leaving the window open from the first block the contract is live.

## Impact Explanation

**Medium — Temporary freezing of funds.**

Honest users who have already burned their rsETH via `initiateWithdrawal` cannot call `completeWithdrawal` until their nonce is below `nextLockedNonce[asset]` (line 707). With a large dust prefix, `nextLockedNonce` stalls, and all legitimate pending withdrawals are frozen for the duration of the backlog-clearing period. Funds are not permanently lost — the admin can set a non-zero minimum and the operator can drain the dust queue in batches — but the freeze is concrete and user-impactful.

## Likelihood Explanation

**High.** The attack requires no privileged role, no oracle manipulation, and no front-running. The only prerequisites are:
1. `minAmountToDeposit == 0` (default, confirmed in code).
2. `minRsEthAmountToWithdraw[asset] == 0` (default, confirmed in code).
3. A negligible ETH balance to fund dust deposits (gas cost dominates).

The attack is executable from the first block after deployment and is fully repeatable.

## Recommendation

1. **Initialize `minRsEthAmountToWithdraw` in `initialize()`** for each supported asset to a protocol-meaningful floor (e.g., equivalent to 0.001 ETH in rsETH terms).
2. **Initialize `minAmountToDeposit` in `LRTDepositPool.initialize()`** to a non-zero value to prevent dust rsETH minting.
3. **Enforce a non-zero minimum** in `setMinRsEthAmountToWithdraw` (revert if `_minAmount == 0`), consistent with the pattern in `KernelVaultETH.setMinDeposit`.
4. **Add a per-user cap** on pending withdrawal requests to bound queue growth per address.

## Proof of Concept

```solidity
// Preconditions (both confirmed as defaults):
//   minRsEthAmountToWithdraw[ETH_TOKEN] == 0
//   minAmountToDeposit == 0

for (uint i = 0; i < 50_000; i++) {
    // Acquire dust rsETH (minAmountToDeposit == 0 allows 1-wei deposit)
    lrtDepositPool.depositETH{value: smallAmount}(0, "");

    // Queue a dust withdrawal (minRsEthAmountToWithdraw == 0 allows 1-wei)
    rsETH.approve(address(lrtWithdrawalManager), 1);
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");
}

// State after attack:
// nextUnusedNonce[ETH_TOKEN] == 50_000 (all dust)
// Legitimate user's nonce == 50_000+
//
// During 8-day delay: unlockQueue breaks on nonce 0 (delay not passed),
//   nextLockedNonce stays at 0, legitimate request at nonce 50_000+ is frozen.
//
// After 8-day delay: operator must call unlockQueue ~50 times (1000 entries/batch)
//   before nextLockedNonce reaches the legitimate request.
//   All legitimate withdrawals remain frozen throughout.
```

A Foundry fork test can confirm this by: (1) deploying with default state, (2) executing the loop above, (3) asserting that `unlockQueue` with `firstExcludedIndex = nextUnusedNonce` reverts OOG or requires many batched calls, and (4) asserting that a legitimate user's `completeWithdrawal` reverts with `WithdrawalLocked` throughout.
