Audit Report

## Title
Strict `msg.value` Equality Check Causes Revert on LayerZero Fee Fluctuation - (File: contracts/bridges/TACWETHBridge.sol)

## Summary
`TACWETHBridge.bridgeTokenToL1()` re-derives the LayerZero native fee at execution time via `getNativeFee()` and enforces strict equality `msg.value != nativeFee`. Because LayerZero fees are dynamic and change with destination-chain gas prices, any fee movement between a user's off-chain quote and on-chain execution causes an unconditional revert, rendering the bridge unusable during periods of normal network activity.

## Finding Description
In `contracts/bridges/TACWETHBridge.sol` at lines 108–113, `bridgeTokenToL1()` calls `getNativeFee(amount, recipient)` internally, which in turn calls `wethOFT.quoteSend(sendParam, false)` (lines 159–161) to obtain the current LayerZero fee. The function then enforces:

```solidity
if (msg.value != nativeFee) {
    revert InvalidNativeFee();
}
```

There is no tolerance band, no `>=` check, and no excess-refund path. The caller must supply exactly the fee computed at the moment of execution — a value they cannot know in advance because `quoteSend` is re-evaluated inside the call. The standard user flow (call `getNativeFee` off-chain → submit tx with that value) is broken whenever the fee changes by even 1 wei between the two calls. The `safeTransferFrom` at line 116 occurs after the fee check, so the revert is clean, but the bridge operation fails entirely. No privileged role is required to trigger this; any user calling the permissionless `bridgeTokenToL1` is affected.

## Impact Explanation
Users on TAC who need to move WETH to Ethereum L1 are temporarily unable to do so during any period of LayerZero fee movement. Their WETH remains on TAC and cannot be bridged until they retry with a freshly-quoted fee that may itself be stale by execution time. This constitutes **Medium — Temporary freezing of funds**: the assets are not lost, but the only bridge path for WETH from TAC to L1 is rendered non-functional during normal network volatility.

## Likelihood Explanation
LayerZero fees are derived from destination-chain gas prices and oracle-reported asset conversion rates, both of which update on every block. No attacker action is required; any period of moderate Ethereum mainnet activity (e.g., a popular NFT mint, market event, or airdrop) is sufficient to shift the fee between quote and execution. This is a routine occurrence, making likelihood **High**.

## Recommendation
Replace the strict equality check with a minimum-fee check and refund any surplus `msg.value` to the caller:

```solidity
uint256 nativeFee = getNativeFee(amount, recipient);
if (msg.value < nativeFee) {
    revert InvalidNativeFee();
}
uint256 surplus = msg.value - nativeFee;
MessagingFee memory fee = MessagingFee({ nativeFee: nativeFee, lzTokenFee: 0 });
(, OFTReceipt memory oftReceipt) = wethOFT.send{ value: nativeFee }(sendParam, fee, msg.sender);
if (surplus > 0) {
    (bool ok,) = payable(msg.sender).call{ value: surplus }("");
    require(ok, "refund failed");
}
```

This is the standard pattern for LayerZero OFT integrations and ensures the transaction succeeds as long as the user over-quotes slightly.

## Proof of Concept
1. Deploy `TACWETHBridge` on a local fork of TAC with a live LayerZero endpoint.
2. Call `getNativeFee(1 ether, recipient)` → records `fee0`.
3. Advance one block or mock `quoteSend` to return `fee0 + 1 wei`.
4. Submit `bridgeTokenToL1{value: fee0}(recipient, 1 ether)`.
5. Observe revert with `InvalidNativeFee()`.
6. Confirm WETH remains in the caller's wallet (no transfer occurred).

A Foundry fuzz test varying the return value of a mock `wethOFT.quoteSend` by ±1 wei relative to `msg.value` will reproduce the revert on any non-zero delta, confirming the strict equality is the sole root cause. [1](#0-0) [2](#0-1)
