### Title
`respond_ckd` Skips Cryptographic Verification for Legacy `AppPublicKey` CKD Responses, Enabling Single-Participant Forged Key Delivery - (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` function in the MPC contract performs **no cryptographic verification** of the CKD response when the request uses the legacy `CKDAppPublicKey::AppPublicKey` variant. Any single attested participant (strictly below signing threshold) can call `respond_ckd` with an arbitrary `CKDResponse` for any pending legacy CKD request, delivering a forged derived key to the user without threshold consensus. The `AppPublicKeyPV` variant has a pairing-based output check; the legacy variant has none.

### Finding Description

In `respond_ckd`, the contract branches on the `app_public_key` variant:

```rust
// crates/contract/src/lib.rs:675-681
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, cryptographically binding the response to the MPC master key and the user's ephemeral key pair. [2](#0-1) 

For the legacy `AppPublicKey` variant, the branch is a no-op — the response `(big_y, big_c)` is accepted unconditionally and immediately delivered to the user via `resolve_yields_for`. [3](#0-2) 

By contrast, the `respond` function for threshold signatures **always** verifies the signature cryptographically before resolving the yield: [4](#0-3) 

The only gate on `respond_ckd` is `assert_caller_is_attested_participant_and_protocol_active()` — a single participant below threshold satisfies this. [5](#0-4) 

**Attack path:**
1. User calls `request_app_private_key` with a legacy `AppPublicKey` request (a single BLS12-381 G1 point `A`).
2. A single malicious attested participant calls `respond_ckd` with attacker-chosen `big_y = G1` (generator) and `big_c = s * G1` for any scalar `s` the attacker knows.
3. The contract accepts the response (no verification for legacy variant) and delivers it to the user.
4. The user computes their derived key as `big_c - a * big_y = s*G1 - a*G1 = (s-a)*G1`. The attacker knows `s` and can recover the user's derived secret if they also know `a`, or they can set `big_y = identity` and `big_c = s*G1` so the user's derived key is exactly `s` — a value the attacker controls.
5. Any assets the user places under this key are accessible to the attacker.

The legacy variant is still actively used and documented as the default in the contract README: [6](#0-5) 

### Impact Explanation

**Critical.** A single Byzantine participant (below signing threshold) can deliver a forged CKD response to any user who submitted a legacy CKD request. The user receives a derived private key that the attacker controls, enabling theft of any assets placed under that key on any foreign chain. This directly bypasses the threshold requirement for CKD: the protocol requires threshold consensus to produce the correct response, but the contract does not enforce this for legacy requests. The impact matches: *"Bypass of threshold-signature requirements … or secret material that materially enables forgery or secret recovery."*

### Likelihood Explanation

**High.** Any single attested participant can perform this attack with no additional preconditions. The attacker only needs to call `respond_ckd` before the legitimate coordinator does (a front-run on NEAR). The legacy `AppPublicKey` variant is the default and widely used. There is no rate-limiting, no coordinator identity check, and no cryptographic binding between the response and the threshold computation.

### Recommendation

1. **Immediate:** Deprecate the legacy `AppPublicKey` variant and reject new requests that use it, requiring all callers to use `AppPublicKeyPV` which has the pairing-based output check.
2. **If legacy must be kept:** Add a threshold-signed commitment to the response (e.g., require the response to carry a threshold signature over `(app_id, big_y, big_c)`) so the contract can verify threshold consensus without the G2 key.
3. **Minimum:** Emit a clear on-chain warning that legacy CKD responses carry no on-chain integrity guarantee, and document the trust assumption explicitly.

### Proof of Concept

```rust
// 1. User submits a legacy CKD request
contract.request_app_private_key(CKDRequestArgs {
    derivation_path: "my-key".to_string(),
    app_public_key: CKDAppPublicKey::AppPublicKey(user_g1_point), // legacy variant
    domain_id: bls_domain_id,
});

// 2. Attacker (single attested participant) front-runs the legitimate coordinator
//    and submits a forged response where big_y = identity, big_c = attacker_secret * G1
let forged_response = CKDResponse {
    big_y: G1_IDENTITY,                          // zeroes out the randomness term
    big_c: attacker_secret_times_g1,             // attacker-chosen key
};
// No cryptographic check fires — the contract accepts this unconditionally
contract.respond_ckd(ckd_request, forged_response); // succeeds

// 3. User receives (big_y=0, big_c=attacker_secret*G1)
//    Derived key = big_c - a * big_y = attacker_secret*G1 - a*0 = attacker_secret*G1
//    User's private key = attacker_secret  ← attacker knows this
```

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

### Submitting a confidential key derivation (ckd) request

Users can submit a ckd request to the MPC network via the
`request_app_private_key` endpoint of this contract. A **deposit of 1
yoctonear is required** (see [Deposit requirement](#deposit-requirement)).

The ckd request takes the following arguments:

- `derivation_path` (String): the derivation path (used to derive different keys from the same account).
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
