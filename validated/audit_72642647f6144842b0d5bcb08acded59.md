### Title
Unguarded `assert len(new_coin_state) == 1` in Pool Wallet Singleton State-Tracking Loop Allows Malicious Peer to Permanently Corrupt Pool Wallet State — (`File: chia/wallet/wallet_state_manager.py`)

---

### Summary

Inside `WalletStateManager._add_coin_states`, when processing a spent `POOLING_WALLET` singleton coin, the code calls `self.wallet_node.get_coin_state(...)` to fetch the next singleton coin's state and then immediately asserts the result has exactly one element. A malicious (or misbehaving) peer can return a valid but empty `RespondToCoinUpdates` response, causing `get_coin_state` to return `[]`. The bare `assert` then raises `AssertionError`, which is caught by the outer handler but classified as a non-retryable error, permanently dropping the pool wallet state transition from the retry store and leaving the wallet's pool wallet state corrupted.

---

### Finding Description

In `chia/wallet/wallet_state_manager.py`, the `POOLING_WALLET` branch of `_add_coin_states` walks a chain of singleton state transitions in a `while` loop:

```python
new_coin_state: list[CoinState] = await self.wallet_node.get_coin_state(
    [new_singleton_name], peer=peer, fork_height=fork_height
)
assert len(new_coin_state) == 1          # ← bare assert, no length guard
curr_coin_state = new_coin_state[0]
``` [1](#0-0) 

`get_coin_state` (in `wallet_node.py`) sends a `RegisterForCoinUpdates` message to the peer. For an **untrusted** peer it filters results through `validate_received_state_from_peer` and returns only validated states:

```python
if not self.is_trusted(peer):
    valid_list = []
    for coin in coin_state.coin_states:
        ...
        valid = await self.validate_received_state_from_peer(...)
        if valid:
            valid_list.append(coin)
    return valid_list
return coin_state.coin_states
``` [2](#0-1) 

A peer that returns a valid `RespondToCoinUpdates` with an **empty** `coin_states` list (a perfectly legal protocol response meaning "I have no record of this coin") causes `get_coin_state` to return `[]`. No `PeerRequestException` is raised — the function returns normally with an empty list.

The `assert len(new_coin_state) == 1` then fires an `AssertionError`. This propagates out of the `async with self.db_wrapper.writer():` block (rolling back the DB transaction) and is caught by the outer handler:

```python
except Exception as e:
    self.log.exception(f"Failed to add coin_state: {coin_state}, error: {e}")
    if rollback_wallets is not None:
        self.wallets = rollback_wallets
    if isinstance(e, (PeerRequestException, aiosqlite.Error)):
        await self.retry_store.add_state(coin_state, peer.peer_node_id, fork_height)
    else:
        await self.retry_store.remove_state(coin_state)   # ← AssertionError goes here
    continue
``` [3](#0-2) 

Because `AssertionError` is not a `PeerRequestException` or `aiosqlite.Error`, the state is **removed from the retry store** rather than queued for retry. The pool wallet singleton state transition is permanently dropped.

---

### Impact Explanation

The pool wallet singleton's state transition chain is aborted mid-walk and never retried. The wallet's in-memory and on-disk pool wallet state diverges from the actual on-chain state. Concretely:

- The wallet loses track of the current singleton coin ID and pool state.
- Subsequent calls to `claim_pool_rewards` / `pw_absorb_rewards` operate on stale state, producing invalid spend bundles that will be rejected by the mempool.
- The farmer cannot absorb pool rewards until the wallet is fully re-synced from a trusted peer.

This matches the allowed High impact: *"Permanent or long-lived inability for honest… wallets… to process… pool actions"* and *"Corruption of… pool membership state… with direct security impact."*

---

### Likelihood Explanation

Any unprivileged node on the Chia network can act as a full-node peer. The wallet connects to untrusted peers by default during sync. The attacker only needs to:

1. Accept the wallet's `RegisterForCoinUpdates` request for the new singleton coin.
2. Reply with a valid `RespondToCoinUpdates` whose `coin_states` field is an empty list.

This is a single, trivially crafted protocol message. No key material, admin access, or cryptographic break is required. The attack is triggered whenever the victim's pool wallet singleton undergoes any state transition (joining/leaving a pool, absorbing rewards) while connected to the malicious peer.

---

### Recommendation

Replace the bare `assert` with an explicit length check that raises a `PeerRequestException` (so the state is queued for retry) or gracefully breaks the loop:

```python
new_coin_state: list[CoinState] = await self.wallet_node.get_coin_state(
    [new_singleton_name], peer=peer, fork_height=fork_height
)
if len(new_coin_state) != 1:
    raise PeerRequestException(
        f"Expected 1 coin state for singleton {new_singleton_name.hex()}, "
        f"got {len(new_coin_state)}"
    )
curr_coin_state = new_coin_state[0]
```

Raising `PeerRequestException` ensures the outer handler routes the failure into `retry_store.add_state(...)` instead of `retry_store.remove_state(...)`, so the state transition is retried when the wallet reconnects to a different peer.

---

### Proof of Concept

**Attacker setup:** Run a modified Chia full node that, upon receiving `RegisterForCoinUpdates` for any coin ID, responds with:

```python
RespondToCoinUpdates(coin_names=[], height=uint32(0), coin_states=[])
```

**Victim setup:** A wallet with an active `POOLING_WALLET` that connects to the attacker's node as an untrusted peer.

**Trigger:** Cause the pool wallet singleton to be spent on-chain (e.g., by the pool forwarding a reward). The victim wallet receives a `CoinStateUpdate` for the spent singleton, enters `_add_coin_states`, reaches the `POOLING_WALLET` branch, calls `get_coin_state` for the successor singleton coin, receives `[]` from the attacker, hits `assert len(new_coin_state) == 1`, and permanently drops the state transition.

**Observable result:** The wallet's pool wallet state is frozen at the pre-transition state. `pw_status` returns stale data. `pw_absorb_rewards` produces spend bundles referencing a spent singleton coin, which are rejected by the mempool. Pool rewards cannot be claimed until the wallet is fully re-synced from a trusted peer. [4](#0-3) [5](#0-4) [3](#0-2)

        coin_state: RespondToCoinUpdates | None = await peer.call_api(FullNodeAPI.register_for_coin_updates, msg)
        if coin_state is None or not isinstance(coin_state, RespondToCoinUpdates):
            raise PeerRequestException(f"Was not able to get states for {coin_names}")

        if not self.is_trusted(peer):
            valid_list = []
            for coin in coin_state.coin_states:
                if coin.coin.name() not in coin_names:
                    await peer.close(9999)
                    self.log.warning(f"Peer {peer.peer_node_id} sent us an unrequested coin state. Banning.")

                valid = await self.validate_received_state_from_peer(
                    coin, peer, self.get_cache_for_peer(peer), fork_height
                )

                    valid_list.append(coin)
