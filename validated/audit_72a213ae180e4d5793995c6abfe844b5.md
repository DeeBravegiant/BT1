Audit Report

## Title
Daily Fee Mint Cap Hard Revert Permanently Blocks Price Updates and Freezes Unclaimed Yield - (File: contracts/LRTOracle.sol)

## Summary
`_checkAndUpdateDailyFeeMintLimit` unconditionally reverts when the accumulated daily fee mint amount reaches `maxFeeMintAmountPerDay`. Because this check is embedded in `_updateRsETHPrice()` before the price write, every subsequent call to `updateRSETHPrice()` and `updateRSETHPriceAsManager()` reverts for the remainder of the 24-hour period. Protocol fee rsETH that would have been minted to the treasury for the blocked window is permanently lost and cannot be recovered after the period resets.

## Finding Description
`_updateRsETHPrice()` computes `rsethAmountToMintAsProtocolFee` and passes it to `_checkAndUpdateDailyFeeMintLimit` at line 303. Inside that function (lines 205–207), if `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay`, the function reverts with `DailyFeeMintLimitExceeded`. This revert unwinds the entire call stack before `rsETHPrice = newRsETHPrice` (line 313) executes, leaving the price stale. Both public entry points — `updateRSETHPrice()` (public, line 87) and `updateRSETHPriceAsManager()` (manager-only, line 94) — call `_updateRsETHPrice()` with no conditional bypass around the fee check. There is no code path that allows the price to be updated while skipping the fee mint check. Once the cap is hit within a period, all price update calls revert until `block.timestamp >= feePeriodStartTime + 1 days` triggers the reset at line 199–202. Any fee that would have accrued during the blocked window is permanently lost — the reset only zeroes `currentPeriodMintedFeeAmount` for the new period; it does not retroactively mint missed fees.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Protocol fee rsETH that would have been minted to the treasury for the remainder of the blocked day is permanently lost. This matches the allowed impact "Medium. Permanent freezing of unclaimed yield." Additionally, `rsETHPrice` and `highestRsethPrice` are not updated during the blocked window, causing any protocol component that reads `rsETHPrice` (deposits, withdrawals, cross-chain rate pushes) to operate on a stale value for up to 24 hours.

## Likelihood Explanation
`updateRSETHPrice()` is a public function callable by anyone. The cap being hit is a normal operational condition: as TVL grows, the cumulative `rsethAmountToMintAsProtocolFee` across multiple calls within a single 24-hour window can exceed `maxFeeMintAmountPerDay`. No attacker action is required — the condition arises from normal protocol operation. The likelihood increases monotonically with TVL growth and call frequency.

## Recommendation
Instead of reverting when the cap is exceeded, cap the fee at the remaining daily allowance and allow the price update to proceed:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal returns (uint256 cappedFee) {
    if (block.timestamp >= feePeriodStartTime + 1 days) {
        currentPeriodMintedFeeAmount = 0;
        feePeriodStartTime = getCurrentPeriodStartTime();
    }
    uint256 remaining = maxFeeMintAmountPerDay > currentPeriodMintedFeeAmount
        ? maxFeeMintAmountPerDay - currentPeriodMintedFeeAmount
        : 0;
    cappedFee = feeAmount > remaining ? remaining : feeAmount;
    currentPeriodMintedFeeAmount += cappedFee;
}
```

Use the returned `cappedFee` in `_updateRsETHPrice()` for the actual mint, so the price update always proceeds regardless of the fee cap state.

## Proof of Concept
1. Manager sets `maxFeeMintAmountPerDay = X` rsETH.
2. During a high-yield day, `updateRSETHPrice()` is called multiple times. After several calls, `currentPeriodMintedFeeAmount` reaches `X`.
3. TVL continues to grow. The next call computes `rsethAmountToMintAsProtocolFee > 0`.
4. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` hits line 205: `currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay` → reverts with `DailyFeeMintLimitExceeded`.
5. `rsETHPrice` is **not updated** (line 313 is never reached). `highestRsethPrice` is **not updated**.
6. Every subsequent call to `updateRSETHPrice()` or `updateRSETHPriceAsManager()` reverts identically until `block.timestamp >= feePeriodStartTime + 1 days`.
7. All fee rsETH that would have been minted to the treasury for the remainder of the day is permanently lost — the period reset at lines 199–202 only resets the counter for the new period.

Foundry test plan: deploy `LRTOracle` on a fork, set `maxFeeMintAmountPerDay` to a small value, call `updateRSETHPrice()` until `currentPeriodMintedFeeAmount` reaches the cap, then assert that the next call reverts with `DailyFeeMintLimitExceeded` and that `rsETHPrice` remains unchanged.
