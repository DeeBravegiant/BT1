### Title
Missing Low-S Enforcement in `ecurve_secp256k1_t::verify` Allows Signature Malleability ‚Äî (`src/cbmpc/crypto/base_ecc_secp256k1.cpp`)

---

### Summary

`ecurve_secp256k1_t::verify` delegates to `ossl_ecdsa_verify`, a custom OpenSSL-backed verifier that only checks `0 < s < q`. It never checks `s <= q/2`. Because standard ECDSA math accepts both `(r, s)` and `(r, q‚àís)` as valid for the same message and key, any caller of this API will accept both forms. The Bitcoin secp256k1 library (`secp256k1_ecdsa_verify`) is never invoked in the production `src/` path for verification.

---

### Finding Description

**Call chain:**

`ecurve_secp256k1_t::verify` ‚Üí `to_ossl_point` ‚Üí `ossl_ecdsa_verify` ‚Üí `ecdsa_signature_t::from_der` ‚Üí scalar range check ‚Üí ECDSA math ‚Üí `SUCCESS` [1](#0-0) 

`ecurve_secp256k1_t::verify` converts the secp256k1 public key to an OpenSSL `EC_POINT` and calls `ossl_ecdsa_verify`. [2](#0-1) 

Inside `ossl_ecdsa_verify`, the only scalar guard is:

```cpp
if (r <= 0 || r >= q || s <= 0 || s >= q)
    return coinbase::error(E_CRYPTO, "ossl_ecdsa_verify: invalid scalar");
``` [3](#0-2) 

This accepts any `s` in `(0, q)`. There is no `s > q/2` rejection. The verification then computes:

```
w = s‚Åª¬π mod q
u1 = e¬∑w,  u2 = r¬∑w
R = u1¬∑G + u2¬∑Q
accept if R.x mod q == r
``` [4](#0-3) 

**Why both `(r, s)` and `(r, q‚àís)` pass:**

For the negated signature `s' = q ‚àí s`:
- `w' = (q‚àís)‚Åª¬π = ‚àís‚Åª¬π mod q`
- `u1' = ‚àíe¬∑s‚Åª¬π`, `u2' = ‚àír¬∑s‚Åª¬π`
- `R' = u1'¬∑G + u2'¬∑Q = ‚àí(u1¬∑G + u2¬∑Q) = ‚àíR`
- Negating an elliptic curve point preserves the x-coordinate, so `R'.x = R.x`
- Therefore `R'.x mod q == r` ‚Üí `SUCCESS`

The Bitcoin secp256k1 library's `secp256k1_ecdsa_verify` (which enforces low-S and would reject `s > q/2`) is confirmed absent from all production `src/` verification paths. [5](#0-4) 

---

### Impact Explanation

Any caller of `ecurve_secp256k1_t::verify` (or `ecc_pub_key_t::verify` on a secp256k1 key) will accept both `(r, s)` and `(r, q‚àís)` as valid signatures for the same message under the same key. This is classic ECDSA signature malleability. In the cb-mpc context, this means:

- A Byzantine peer or external attacker who possesses one valid signature can trivially produce a second distinct "valid" signature without knowledge of the private key.
- Any protocol step that uses `verify` as a gate (e.g., in `ecdsa_mp.cpp`, `ecdsa_2p.cpp`) can be passed with either form, breaking the assumption that a verified signature uniquely identifies a signing event.
- Downstream consumers (e.g., Bitcoin transaction pipelines) that expect low-S-normalized signatures will observe divergent behavior between what cb-mpc accepts and what the Bitcoin network accepts.

---

### Likelihood Explanation

The attacker needs only a single valid signature `(r, s)` ‚Äî which is the normal output of any signing operation ‚Äî and can compute `(r, q‚àís)` with trivial arithmetic. No secret material is required. The path is directly reachable through the public `ecc_pub_key_t::verify` API on any secp256k1 key.

---

### Recommendation

Add a low-S check inside `ossl_ecdsa_verify` (or in `ecurve_secp256k1_t::verify` before delegating) for secp256k1:

```cpp
// After the existing range check, for secp256k1:
if (curve == curve_secp256k1) {
    bn_t half_q = (q.value() + 1) / 2;  // (q+1)/2
    if (s > half_q)
        return coinbase::error(E_CRYPTO, "ossl_ecdsa_verify: high-S rejected");
}
```

Alternatively, route secp256k1 verification through `secp256k1_ecdsa_verify` (which already enforces low-S) instead of the custom `ossl_ecdsa_verify` path.

error_t ossl_ecdsa_verify(const EC_GROUP* group, EC_POINT* point, mem_t hash, mem_t signature) {
  error_t rv = UNINITIALIZED_ERROR;
  ecurve_t curve = ecurve_t::find(group);
  if (!curve) return coinbase::error(E_CRYPTO, "ossl_ecdsa_verify: unsupported curve");

  ecdsa_signature_t sig;
  if (rv = sig.from_der(curve, signature)) return rv;

  const mod_t& q = curve.order();
  const bn_t& r = sig.get_r();
  const bn_t& s = sig.get_s();
  if (r <= 0 || r >= q || s <= 0 || s >= q) return coinbase::error(E_CRYPTO, "ossl_ecdsa_verify: invalid scalar");

  int curve_size = curve.size();
  if (hash.size >= curve_size) hash.size = curve_size;
  const bn_t e = bn_t::from_bin(hash);

  buf_t oct(curve.point_bin_size());
  cb_assert(EC_POINT_point2oct(group, point, POINT_CONVERSION_UNCOMPRESSED, oct.data(), oct.size(),
                               bn_t::thread_local_storage_bn_ctx()) > 0);

  ecc_point_t Q;
  if (rv = Q.from_oct(curve, oct)) return rv;
  if (rv = curve.check(Q)) return rv;

  bn_t u1, u2;
  MODULO(q) {
    const bn_t w = q.inv(s);
    u1 = e * w;
    u2 = r * w;
  }

  vartime_scope_t vartime_scope;
  const ecc_point_t R = curve.mul_add(u1, Q, u2);
  if (R.is_infinity()) return coinbase::error(E_CRYPTO, "ossl_ecdsa_verify: R is infinity");
  if (q.mod(R.get_x()) != r) return coinbase::error(E_CRYPTO, "ossl_ecdsa_verify: invalid signature");
  return SUCCESS;
}
```
