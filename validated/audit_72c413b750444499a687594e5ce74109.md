Audit Report

## Title
Missing Staleness, Round-Completeness, and Non-Positive Price Guards in `ChainlinkPriceOracle.getAssetPrice()` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `roundId`, `updatedAt`, and `answeredInRound`, and casts `price` directly to `uint256` without a positivity check. A stale or zero price propagates into `rsETHPrice` via the permissionless `LRTOracle.updateRSETHPrice()`, allowing any caller to set an incorrect exchange rate and enable depositors to extract value from existing rsETH holders.

## Finding Description
At `contracts/oracles/ChainlinkPriceOracle.sol` L52–54, the oracle discards all validity signals from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is made that:
- `answeredInRound >= roundId` (round completeness)
- `updatedAt != 0` (non-zero timestamp)
- `price > 0` (non-negative/non-zero price)

By contrast, the sibling contract `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L30–32 applies all three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The exploit path is:
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale due to network congestion or a circuit-breaker event.
2. Any address calls `LRTOracle.updateRSETHPrice()` (public, no access control beyond `whenNotPaused`, L87–89).
3. `_updateRsETHPrice()` calls `_getTotalEthInProtocol()` (L231), which iterates over supported assets and calls `getAssetPrice(asset)` (L339), which delegates to `ChainlinkPriceOracle.getAssetPrice()`.
4. The stale (artificially low) price is returned without revert, reducing `totalETHInProtocol`.
5. `newRsETHPrice` is set lower than the true value (L250) and stored as `rsETHPrice` (L313).
6. A depositor calls `LRTDepositPool.depositAsset()`, which computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` (L520). With `rsETHPrice` artificially depressed, the depositor receives more rsETH than their deposit is worth, extracting value from all existing rsETH holders.

The downside-protection guard at `LRTOracle.sol` L270–281 only triggers a pause if the price drop exceeds `pricePercentageLimit`. If `pricePercentageLimit` is zero (unset) or the stale price deviation is within the configured limit, no pause occurs and the corrupted price is committed.

## Impact Explanation
A stale or zero asset price fed into `_getTotalEthInProtocol()` produces an incorrect `totalETHInProtocol`, which directly sets `rsETHPrice`. An artificially low `rsETHPrice` causes new depositors to receive more rsETH than the true ETH value of their deposit, diluting all existing rsETH holders — constituting direct theft of user funds and potential protocol insolvency. This matches the **Critical** impact class: *Direct theft of any user funds* and *Protocol insolvency*.

## Likelihood Explanation
Chainlink feeds go stale during network congestion, sequencer downtime, or when asset prices move outside the deviation threshold. `updateRSETHPrice()` is permissionless — any unprivileged address can call it. An attacker can monitor Chainlink round data off-chain and call `updateRSETHPrice()` precisely when a feed is stale, then immediately deposit to capture the mispriced rsETH. No special privileges, governance access, or key compromise is required.

## Recommendation
Apply the same guards already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Optionally, add a heartbeat check: `require(block.timestamp - updatedAt <= MAX_DELAY)`.

## Proof of Concept
**Foundry fork test outline:**
1. Fork mainnet at a block where the stETH/ETH Chainlink feed has not updated for >1 hour (or mock `latestRoundData()` to return a stale `updatedAt` and a price 10% below current).
2. Call `LRTOracle.updateRSETHPrice()` from an unprivileged EOA — confirm it succeeds and `rsETHPrice` is set to the stale-derived value.
3. Call `LRTDepositPool.depositAsset(stETH, amount, 0, "")` from the same EOA — confirm `rsethAmountToMint` is inflated relative to the true price.
4. Assert that the rsETH minted exceeds the fair share, demonstrating value extraction from existing holders.
5. Confirm the same call reverts when `ChainlinkOracleForRSETHPoolCollateral.getRate()` is used (proving the fix works).
