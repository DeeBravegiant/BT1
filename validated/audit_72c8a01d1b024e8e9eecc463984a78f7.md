### Title
Unvalidated CKD Response for `AppPublicKey` Variant Allows Single Malicious Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

`MpcContract::respond_ckd` performs a cryptographic output check (`ckd_output_check`) only when the CKD request uses the `AppPublicKeyPV` variant. When the request uses the `AppPublicKey` variant, the match arm is empty and the response is accepted and delivered to the user with **zero cryptographic verification**. A single malicious attested participant can call `respond_ckd` with an arbitrary, attacker-chosen `CKDResponse`, delivering a fully attacker-controlled key to the user and bypassing the threshold requirement entirely.

---

### Finding Description

In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
// crates/contract/src/lib.rs:675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk, app_id), mpc_pk)`, which cryptographically binds the response to the MPC network's master secret key and the user's app public key:

```rust
// crates/contract/src/primitives/ckd.rs:80-101
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool { ... }
``` [2](#0-1) 

For `AppPublicKey`, **no equivalent check exists**. The `AppPublicKey` variant carries only a G1 point (`Bls12381G1PublicKey`) and lacks the G2 counterpart needed for the pairing equation, but the contract does not substitute any alternative verification — it simply skips all output validation. [3](#0-2) 

The `AppPublicKey` variant is a real, production-used path. The sandbox test utility `create_response_ckd` constructs requests using it: [4](#0-3) 

The `request_app_private_key` entry point similarly skips any check for `AppPublicKey`: [5](#0-4)

### Impact Explanation

The CKD protocol computes `big_c = msk·H(pk, app_id) + y·app_pk` and `big_y = y·G1`. The user decrypts with their private key `a`: `sig = big_c − a·big_y = msk·H(pk, app_id)`. This is the confidential derived key.

Because `respond_ckd` accepts any `(big_y, big_c)` pair for `AppPublicKey` requests without verification, a single malicious attested participant can submit:
- `big_y = identity (0)`, `big_c = attacker_chosen_point`

The user then decrypts `sig = attacker_chosen_point − a·0 = attacker_chosen_point`, receiving a fully attacker-controlled key. The attacker knows the key they injected, breaking confidentiality entirely.

This bypasses the threshold requirement: only **one** attested participant is needed to forge the response, not `t` of `n`. The contract's `assert_caller_is_attested_participant_and_protocol_active()` check only verifies the caller is any active participant, not that they participated in the specific CKD computation. [6](#0-5) 

**Allowed impact matched:** *Critical — confidential key derivation output without the required participant authorization.*

---

### Likelihood Explanation

Any single attested MPC participant (Byzantine node strictly below the signing threshold) can exploit this. The attacker needs only to:
1. Observe a pending `AppPublicKey` CKD request in `pending_ckd_requests`.
2. Call `respond_ckd` with an arbitrary `CKDResponse` before the honest nodes respond.

No collusion, no key leakage, no TEE attack, and no network-level DoS is required. The `AppPublicKey` variant is the simpler, non-publicly-verifiable path and is actively used in production flows.

---

### Recommendation

Apply the same cryptographic output verification to `AppPublicKey` requests. Since `AppPublicKey` provides only a G1 point (no G2 counterpart for the pairing check), the contract should either:

1. **Require `AppPublicKeyPV` for all on-chain CKD requests**, removing the `AppPublicKey` variant from the public API, or
2. **Derive a surrogate G2 key** from the G1 key (if the protocol allows it) and apply an equivalent binding check, or
3. **Verify the response off-chain** and have the contract accept only a threshold-signed attestation of the output rather than a single participant's unverified claim.

The simplest safe fix is option 1: enforce `AppPublicKeyPV` in `respond_ckd` and reject `AppPublicKey` responses at the contract level, consistent with how `ckd_output_check` is already defined and tested. [7](#0-6)

### Proof of Concept

```rust
// Attacker is one attested participant.
// A victim has submitted a CKD request using AppPublicKey variant.
// The attacker calls respond_ckd with a forged response:

let forged_response = CKDResponse {
    big_y: Bls12381G1PublicKey::from(&ElementG1::identity()),  // zero blinding
    big_c: Bls12381G1PublicKey::from(&attacker_chosen_point),  // attacker-controlled key
};

// respond_ckd accepts this without any cryptographic check:
contract.respond_ckd(victim_ckd_request, forged_response)
    .expect("accepted with no output verification");

// The victim's yield is resolved with the forged response.
// The victim decrypts: sig = big_c - a * big_y = attacker_chosen_point - a * 0
//                         = attacker_chosen_point  (fully attacker-controlled)
```

The attacker knows the derived key the victim will use, breaking the confidentiality guarantee of the CKD protocol. The threshold of `t` honest participants is completely bypassed.
