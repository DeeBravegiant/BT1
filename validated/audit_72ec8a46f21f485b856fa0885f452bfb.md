### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Validating It Against the Stored Request — (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` accepts `response.payload_hash` as a free parameter supplied by the calling MPC participant and only checks that the signature is valid over that hash. It never verifies that `payload_hash` was actually derived from `ForeignTxSignPayload { request, values }` for the pending request being resolved. A single malicious leader node — a Byzantine participant strictly below the signing threshold — can obtain a valid threshold signature for one pending foreign-tx request and submit it as the response to a *different* pending request, causing the contract to emit a valid MPC attestation over a hash that does not correspond to the actual foreign-chain verification result.

### Finding Description

**Root cause — `respond_verify_foreign_tx` (crates/contract/src/lib.rs:691-754)**

The function performs two checks:
1. The caller is an attested participant.
2. The ECDSA signature in `response` is valid over `response.payload_hash` against the domain's root public key.

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
```

It does **not** verify that `response.payload_hash == ForeignTxSignPayload::V1 { request: request.request, values: <observed> }.compute_msg_hash()`. The contract stores only the `VerifyForeignTransactionRequest` (chain, tx_id, extractors, domain, payload_version) in `pending_verify_foreign_tx_requests`; the extracted `values` are never stored on-chain, so the contract has no way to recompute the expected hash on its own.

**Analog to SP1Blobstream bug:** In SP1Blobstream, `trusted_block_height` and `target_block_height` were passed as separate inputs and committed to state without being validated against the actual heights inside the light-block headers. The fix was to derive those values from the verified data structure instead of accepting them as free inputs. Here, `payload_hash` is the analogous free input: it should be derived from `ForeignTxSignPayload { request, values }` but is instead accepted verbatim from the caller.

**Attack path (single malicious leader, below threshold):**

The leader node is the only participant that calls `respond_verify_foreign_tx`. Followers only participate in the threshold signing round; they never submit the on-chain response.

1. Two concurrent pending requests exist: **Request A** (tx_id = A) and **Request B** (tx_id = B).
2. The malicious leader is elected leader for Request A's signing session.
3. Followers independently query the foreign chain for Request A, compute `hash_A = SHA256(borsh(ForeignTxSignPayload { request_A, values_A }))`, and contribute signing shares over `hash_A`.
4. The leader assembles the threshold signature `sig_A` over `hash_A` — a fully valid MPC signature.
5. Instead of calling `respond_verify_foreign_tx(request_A, { payload_hash: hash_A, sig_A })`, the leader calls `respond_verify_foreign_tx(request_B, { payload_hash: hash_A, sig_A })`.
6. The contract finds Request B in pending state ✓, verifies `sig_A` over `hash_A` ✓, and resolves Request B's yield with `{ payload_hash: hash_A, signature: sig_A }`.

Request B's caller receives a valid MPC signature, but over `hash_A` — the attestation of a completely different foreign transaction.

**Node-side flow confirming the leader's exclusive control over the on-chain call:** [1](#0-0) 

The leader computes `payload_hash = response.0.0.compute_msg_hash()` and passes it directly into `VerifyForeignTransactionRespondArgs::from_signature`. Nothing prevents it from substituting a hash from a different session. [2](#0-1) 

**Contract verification gap:** [3](#0-2) 

The contract verifies only `sig_A` over `payload_hash` (line 729-734). There is no call to `ForeignTxSignPayload::compute_msg_hash()` and no comparison against a hash recomputed from the stored request. [4](#0-3) 

### Impact Explanation

The contract emits a valid threshold signature over a hash that does not correspond to the actual foreign-chain verification result for the resolved request. Any bridge contract (e.g., Omnibridge inbound flow) that does not independently reconstruct and compare the expected `payload_hash` from the request and the expected extracted values will accept this forged attestation as proof that a specific foreign-chain event occurred, when it did not. This enables a malicious leader to forge foreign-chain verification attestations, potentially triggering invalid bridge execution or double-spend conditions.

The `near-mpc-sdk`'s `ForeignChainSignatureVerifier::verify_signature()` does perform the hash check client-side: [5](#0-4) 

However, this is a client-side SDK helper, not a contract-enforced invariant. Bridge contracts that do not use the SDK, or that do not know the expected extracted values at verification time, are directly vulnerable. The contract itself — the trust anchor — does not enforce the binding.

### Likelihood Explanation

The attacker must be a single attested MPC participant who acts as the leader for a signing session. Leader election is not privileged — any participant can become the leader for a given request. The attack requires two concurrent pending `verify_foreign_transaction` requests, which is realistic in any production bridge deployment with non-trivial throughput. No threshold collusion, leaked keys, or physical TEE attacks are required.

### Recommendation

Require the responding participant to supply the extracted `values` alongside the response, and have the contract recompute `ForeignTxSignPayload::V1 { request: request.request, values }.compute_msg_hash()`, asserting it equals `response.payload_hash` before accepting the response. This is the direct analog of the SP1Blobstream fix: derive the committed value from the verified data structure rather than accepting it as a free input.

```rust
// In respond_verify_foreign_tx, after retrieving the stored request:
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // caller must now supply values
})
.compute_msg_hash()
.expect("hash computation must not fail");

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

### Proof of Concept

1. Submit two concurrent `verify_foreign_transaction` requests: Request A (Bitcoin tx_id = `[0xAA; 32]`) and Request B (Bitcoin tx_id = `[0xBB; 32]`).
2. The malicious leader node runs the threshold signing session for Request A, obtaining `sig_A` over `hash_A = SHA256(borsh(ForeignTxSignPayload { request_A, values_A }))`.
3. The leader calls `respond_verify_foreign_tx` with `request = request_B` and `response = { payload_hash: hash_A, signature: sig_A }`.
4. The contract at lines 718-747 verifies `sig_A` over `hash_A` — which passes — and resolves Request B's yield with `{ payload_hash: hash_A, signature: sig_A }`.
5. Request B's caller receives a valid MPC signature, but `hash_A` encodes the verification of Bitcoin tx `[0xAA; 32]`, not `[0xBB; 32]`. A bridge contract that does not reconstruct and compare the expected hash will accept this as proof that tx `[0xBB; 32]` was verified. [6](#0-5) [7](#0-6) [8](#0-7)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
