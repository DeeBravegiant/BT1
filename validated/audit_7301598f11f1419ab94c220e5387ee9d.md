After thorough analysis of the NEAR MPC contract, I found a valid analog.

---

### Title
Unverified CKD Response for `AppPublicKey` Variant Allows Single Byzantine Leader to Deliver Fabricated Key Derivation Output — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` skips all cryptographic output verification when the CKD request uses the `AppPublicKey` variant. A single Byzantine attested participant acting as the protocol leader can call `respond_ckd` with completely fabricated `big_y`/`big_c` values, bypassing the threshold-authorization requirement and delivering a wrong derived key to the user.

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, the contract enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically ties the response to the MPC master key and the user's app public key. [2](#0-1) 

For `AppPublicKey`, the contract performs **no such check** and immediately proceeds to `resolve_yields_for`, delivering whatever `CKDResponse` the caller supplied. [3](#0-2) 

The access guard `assert_caller_is_attested_participant_and_protocol_active` only confirms the caller is an attested participant; it does not require threshold cooperation for the response values themselves. [4](#0-3) 

Therefore, a single Byzantine leader can:
1. Observe a pending `AppPublicKey` CKD request.
2. Skip the off-chain MPC CKD protocol entirely.
3. Call `respond_ckd` with arbitrary `big_y` and `big_c` byte arrays.
4. The contract accepts the call and resumes the user's yield with the fabricated output.

The existing test at lines 4513–4521 inadvertently confirms this: it uses `[1u8; 48]` / `[2u8; 48]` as response values and expects success, demonstrating that any byte sequence is accepted for `AppPublicKey`. [5](#0-4) 

Contrast this with `respond` (for signatures), which always verifies the submitted signature against the MPC public key and rejects invalid ones: [6](#0-5) 

---

### Impact Explanation

The user's `request_app_private_key` call resolves with a fabricated CKD output. The user cannot recover the correct MPC-derived key from this output. This constitutes **unauthorized confidential key derivation output delivered without the required threshold-participant authorization** — a single Byzantine participant strictly below the signing threshold suffices to corrupt the result. This maps directly to the allowed Critical impact: *"confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

The attacker must be an attested participant (requires a valid TEE attestation in production). Once a participant, leader selection is deterministic per request, so the attacker can predict which requests they will lead. Any `AppPublicKey` CKD request assigned to a Byzantine leader is silently corrupted with no on-chain detection mechanism.

---

### Recommendation

1. **Require `AppPublicKeyPV` for all production CKD requests** and deprecate `AppPublicKey`, since only `AppPublicKeyPV` has an on-chain verifiable pairing check.
2. **Or** add an equivalent cryptographic binding for `AppPublicKey` responses so the contract can verify the output is consistent with the MPC master key before resolving the yield.
3. At minimum, document clearly that `AppPublicKey` provides no on-chain integrity guarantee and must not be used in security-sensitive contexts.

---

### Proof of Concept

```
1. Attacker is an attested participant (account: attacker.near).
2. User calls:
     request_app_private_key({
       app_public_key: AppPublicKey(some_bls_pk),
       derivation_path: "m/0",
       domain_id: <CKD domain>
     })
   Contract stores CKDRequest in pending_ckd_requests and yields.

3. Attacker (as leader) calls:
     respond_ckd(
       request,
       CKDResponse { big_y: [0u8; 48], big_c: [0u8; 48] }
     )

4. Contract executes:
     match &request.app_public_key {
       AppPublicKey(_) => {}   // no check
     }
     resolve_yields_for(...)   // user's promise resumes with fabricated output

5. User receives wrong CKD output; the correct MPC-derived key is unrecoverable.
   No on-chain error is raised. No threshold cooperation was required.
```

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
