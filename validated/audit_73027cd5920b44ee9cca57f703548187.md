### Title
Unconstrained `fee_proposal_fri` During Startup Window Allows Malicious Proposer to Inject Arbitrary Values into Consensus Commitment and Future Fee Computation — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `is_proposal_init_valid` function in the proposal validation path only enforces bounds on `fee_proposal_fri` when **both** `fee_actual` and `fee_proposal_fri` are `Some`. When `fee_actual` is `None` — which occurs for the first `window_size` blocks after V0_14_3 activation — the bounds check is silently skipped. The proposer is expected to freeze `fee_proposal_fri` at `l2_gas_price` in this state, but the validator never enforces this constraint. A malicious proposer can therefore set `fee_proposal_fri` to any value (including `u128::MAX`) during the startup window, have it accepted by all validators, committed into the `ProposalCommitment`, and recorded in the `fee_proposals_window` that drives future L2 gas prices.

---

### Finding Description

**Structural analog to Vether4 M-12:** In Vether4, the fee-exclusion check fires when *either* sender *or* recipient is excluded, so routing through an excluded address bypasses fees for everyone. Here, the bounds check fires only when *both* `fee_actual` and `fee_proposal_fri` are `Some`; when `fee_actual` is `None` (the startup window), the check is bypassed entirely — regardless of what value the proposer supplies.

**Exact code path:**

In `is_proposal_init_valid` the bounds check is:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(...);
    }
}
``` [1](#0-0) 

When `proposal_init_validation.fee_actual` is `None`, the entire `if let` body is skipped. The validator does **not** fall back to checking `fee_proposal_fri == l2_gas_price_fri`, even though `l2_gas_price_fri` is present in `ProposalInitValidation` and the proposer is documented to freeze there:

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
``` [2](#0-1) 

`fee_actual` is `None` whenever `compute_fee_actual` cannot fill the sliding window:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
``` [3](#0-2) 

This `None` propagates directly into `ProposalInitValidation.fee_actual`: [4](#0-3) 

**Commitment path:** The accepted `fee_proposal_fri` is hashed into the `ProposalCommitment` that consensus signs over:

```rust
ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
``` [5](#0-4) 

It is then stored in `fee_proposals_window` at `decision_reached`, feeding `compute_fee_actual` for all subsequent blocks: [6](#0-5) 

---

### Impact Explanation

1. **Wrong `ProposalCommitment` accepted by consensus.** The validator computes `batcher_block_commitment = proposal_commitment_from(partial_block_hash, fee_proposal_fri)` using the attacker-supplied `fee_proposal_fri`. Because the proposer's `ProposalFin` was constructed with the same extreme value, the `built_block == received_fin.proposal_commitment` check passes. Consensus signs over a commitment that encodes an arbitrary fee value.

2. **Persistent skew of future L2 gas prices.** The injected extreme `fee_proposal_fri` is recorded in `fee_proposals_window`. Once the window fills, `compute_fee_actual` computes the median over these values. A proposer controlling ≥ ⌈window_size/2⌉ + 1 blocks during the startup window can set the median to any desired value, causing all subsequent blocks to use a wrong `fee_actual` as the floor for the EIP-1559-style L2 gas price adjustment. This is an incorrect fee/gas effect with direct economic impact on every transaction in the network.

---

### Likelihood Explanation

The startup window occurs at genesis and at every V0_14_3 protocol upgrade. In a BFT network with rotating proposers, a single malicious validator controls roughly `1/n` of the startup blocks. For small `window_size` values or small validator sets, controlling the majority of startup blocks is feasible. The attack requires no special privileges beyond being a validator and proposing during the startup window — an unprivileged trigger.

---

### Recommendation

When `fee_actual` is `None`, the validator should enforce the same fallback the proposer uses: `fee_proposal_fri` must equal `l2_gas_price_fri` (the value in `ProposalInitValidation`). Replace the silent skip with an explicit equality check:

```rust
match (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri) {
    (Some(fee_actual), Some(fee_proposal)) => {
        // existing bounds check
    }
    (None, Some(fee_proposal)) => {
        // Startup window: proposer must freeze at l2_gas_price
        if fee_proposal != proposal_init_validation.l2_gas_price_fri {
            return Err(ValidateProposalError::InvalidProposalInit(...));
        }
    }
    _ => {}
}
```

### Proof of Concept

1. Network activates V0_14_3. `fee_proposals_window` is empty; `compute_fee_actual` returns `None` for all heights `< window_size`.
2. Malicious validator is selected as proposer for block `height = 1` (within the startup window).
3. Proposer constructs `ProposalInit` with `fee_proposal_fri = Some(GasPrice(u128::MAX))` instead of the expected `l2_gas_price`.
4. Validator calls `is_proposal_init_valid`. `proposal_init_validation.fee_actual` is `None`. The `if let (Some(fee_actual), Some(fee_proposal))` arm does not match. No error is returned.
5. `initiate_validation` is called; the batcher executes the block normally.
6. `handle_proposal_part` computes `batcher_block_commitment = proposal_commitment_from(partial, Some(u128::MAX))`. The proposer's `ProposalFin` carries the same commitment. The `built_block == received_fin.proposal_commitment` check passes.
7. `valid_proposals.insert_proposal` stores the block with `fee_proposal_fri = u128::MAX`.
8. At `decision_reached`, `record_fee_proposal(height, Some(u128::MAX))` is called, inserting `u128::MAX` into `fee_proposals_window`.
9. Once the window fills, `compute_fee_actual` returns a median that is heavily skewed toward `u128::MAX` if the attacker controlled the majority of startup blocks, causing all future L2 gas prices to be set at an extreme value. [1](#0-0) [7](#0-6) [5](#0-4)

```rust
pub fn compute_fee_actual(
    fee_proposals_window: &BTreeMap<BlockNumber, Option<GasPrice>>,
    height: BlockNumber,
    window_size: u64,
) -> Option<GasPrice> {
    let Some(start) = height.0.checked_sub(window_size) else {
        warn!(
            "Cannot compute fee_actual for height {height}: height is below window_size \
             ({window_size})"
        );
        return None;
    };
    let window_size_usize = usize::try_from(window_size).expect("window_size fits in usize");
    let mut window = Vec::with_capacity(window_size_usize);
    for source_height in (start..height.0).map(BlockNumber) {
        match fee_proposals_window.get(&source_height) {
            Some(Some(price)) => window.push(*price),
            Some(None) | None => {
                warn!(
                    "Cannot compute fee_actual for height {height}: fee_proposals_window has no \
                     recorded fee_proposal for height {source_height}"
                );
                return None;
            }
        }
    }
    window.sort();
    let mid = window_size_usize / 2;
    let median = if window_size_usize.is_multiple_of(2) {
        // Even: average of the two middle values, rounded down.
        // Overflow-safe averaging: a + (b - a) / 2 (safe because sorted, so b >= a).
        GasPrice(window[mid - 1].0 + (window[mid].0 - window[mid - 1].0) / 2)
    } else {
        window[mid]
    };
    Some(median)
}
