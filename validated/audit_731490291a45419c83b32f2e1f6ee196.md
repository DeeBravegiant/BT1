Audit Report

## Title
Precision Loss in `rewardPerToken()` Permanently Freezes Unclaimed Yield When `totalKernelStaked` Is Large - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`KernelDepositPool` uses a Synthetix-style reward accumulator with `DECIMAL_PRECISION = 1e18`. When `totalKernelStaked` is sufficiently large relative to `rewardRate * timeDelta * 1e18`, the per-interval increment to `rewardPerTokenStored` truncates to zero. Critically, the `updateReward` modifier advances `updatedAt` to the current timestamp even when the increment is zero, permanently discarding the rewards for that interval with no recovery path.

## Finding Description

`rewardPerToken()` computes the accumulator increment as:

```solidity
(rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION) / totalKernelStaked
``` [1](#0-0) 

The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` after computing `rewardPerTokenStored`:

```solidity
rewardPerTokenStored = rewardPerToken();
updatedAt = lastTimeRewardApplicable();
``` [2](#0-1) 

When `rewardRate * timeDelta * 1e18 < totalKernelStaked`, the division truncates to zero. Because `updatedAt` still advances, the time window `[updatedAt_old, now]` is consumed without allocating any rewards. Those reward-wei are permanently locked in the contract — no user can claim them and no admin function can recover them.

`notifyRewardAmount` only adds new rewards; it does not account for previously unaccumulated amounts: [3](#0-2) 

`rewardRate` itself is truncated on assignment:

```solidity
rewardRate = receivedAmount / duration;
``` [4](#0-3) 

A second truncation occurs in `earned()` where `balanceOf[_account] * delta / DECIMAL_PRECISION` can also round down for small balances: [5](#0-4) 

## Impact Explanation

Rewards deposited via `notifyRewardAmount` that are lost to per-interval rounding are permanently locked in the contract. No user can claim them and no admin rescue function exists. This constitutes **permanent freezing of unclaimed yield**, matching the Medium impact class in the allowed scope.

## Likelihood Explanation

KERNEL has a 1 billion token supply (`1e27` wei): [6](#0-5) 

With 10% of supply staked (`totalKernelStaked = 1e26`), `receivedAmount = 100e18`, and `duration = 365 days`:
- `rewardRate ≈ 3_170_979`
- Per-block numerator: `3_170_979 * 12 * 1e18 ≈ 3.8e25`
- Increment: `3.8e25 / 1e26 = 0` — every 12-second interval loses all rewards

The threshold for zero-increment at 12-second blocks is `totalKernelStaked > ~3.8e25`, i.e., ~38 million KERNEL (3.8% of supply) — trivially achievable. Any unprivileged staker calling `stake()` can push the system into the precision-loss regime: [7](#0-6) 

## Recommendation

1. **Increase `DECIMAL_PRECISION`**: Replace `1e18` with `1e36` so the numerator `rewardRate * timeDelta * DECIMAL_PRECISION` stays above `totalKernelStaked` across realistic staking levels.
2. **Track dust**: Accumulate the remainder `(rewardRate * timeDelta * DECIMAL_PRECISION) % totalKernelStaked` and carry it forward to the next update interval.
3. **Enforce minimum reward rate**: In `notifyRewardAmount`, revert if `rewardRate * DECIMAL_PRECISION < totalKernelStaked / expectedMinTimeDelta` to prevent configurations that guarantee precision loss from the start.

## Proof of Concept

**Setup:**
- `totalKernelStaked = 1e26` (100 million KERNEL, 10% of supply)
- `receivedAmount = 100e18`, `duration = 31_536_000` (365 days)
- `rewardRate = 100e18 / 31_536_000 ≈ 3_170_979`

**Foundry invariant test sketch:**

```solidity
function invariant_noRewardsLostToRounding() public {
    // After full reward period elapses with frequent interactions (every 12s),
    // sum of all claimable rewards should equal notified reward amount.
    // With totalKernelStaked = 1e26, this invariant breaks:
    // totalClaimable < notifiedAmount  ← rewards permanently frozen
}
```

**Concrete per-block check:**
```
numerator = 3_170_979 * 12 * 1e18 = 3.805e25
totalKernelStaked = 1e26
increment = 3.805e25 / 1e26 = 0  ← updatedAt still advances, rewards lost
```

Over 365 days with 12-second blocks: `2,628,000` intervals × `38,051,748` reward-wei lost each = effectively all 100 reward tokens permanently frozen while stakers receive zero yield.

    function notifyRewardAmount(uint256 _amount) external onlyRole(DEFAULT_ADMIN_ROLE) updateReward(address(0)) {
        if (_amount == 0) revert AmountZero();

        // Prevent starting a reward period when no tokens are staked to avoid unallocated rewards
        if (totalKernelStaked == 0) revert NoStakedTokens();

        // Transfer reward tokens into the contract
        uint256 balanceBefore = rewardsToken.balanceOf(address(this));
        rewardsToken.safeTransferFrom(msg.sender, address(this), _amount);
        uint256 balanceAfter = rewardsToken.balanceOf(address(this));
        // Calculate the actual amount of tokens received in case of a transfer fee (tax)
        uint256 receivedAmount = balanceAfter - balanceBefore;

        if (block.timestamp >= finishAt) {
            rewardRate = receivedAmount / duration;
        } else {
            uint256 remaining = (finishAt - block.timestamp) * rewardRate;
            rewardRate = (receivedAmount + remaining) / duration;
        }

        if (rewardRate == 0) revert RewardRateZero();

        finishAt = block.timestamp + duration;
        updatedAt = block.timestamp;

        emit NotifyRewardAmount(receivedAmount, finishAt);
    }
```
