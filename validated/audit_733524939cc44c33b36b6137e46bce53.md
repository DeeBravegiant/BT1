### Title
Unconditional acceptance of `farmer_reward_address_override` in `RespondSignatures` allows malicious harvester to divert XCH farmer block reward — (`chia/farmer/farmer_api.py`)

### Summary
A malicious harvester peer can set `farmer_reward_address_override` to an attacker-controlled puzzle hash in a `RespondSignatures` message. The farmer's `_process_respond_signatures` accepts this override without any fee quality validation, causing `DeclareProofOfSpace.farmer_reward_address` to be set to the attacker's address instead of `self.farmer.farmer_target`.

### Finding Description
CHIP-22 introduces `farmer_reward_address_override` as a mechanism for third-party harvesters to redirect the farmer reward as a fee, gated on a `fee_info.applied_fee_threshold` that must meet a quality threshold. However, the enforcement is incomplete:

1. `fee_info` exists only in `NewProofOfSpace` (harvester→farmer), not in `RespondSignatures`.
2. `_process_respond_signatures` (lines 916–918) applies the override from `RespondSignatures` with no validation whatsoever.
3. A malicious harvester can send `NewProofOfSpace` with `farmer_reward_address_override=None` (bypassing the `notify_farmer_reward_taken_by_harvester_as_fee` call entirely), then respond to `RequestSignatures` with `RespondSignatures.farmer_reward_address_override` set to any attacker address.

The resulting `DeclareProofOfSpace` carries the attacker's puzzle hash as `farmer_reward_address`, which the full node uses to assign the block reward coin. [5](#0-4) 

### Impact Explanation
Every block won by the farmer while connected to the malicious harvester will have its farmer reward (currently 0.25 XCH per block) permanently diverted to the attacker's address. This is an irreversible on-chain transfer. The farmer receives nothing and has no recourse.

### Likelihood Explanation
CHIP-22 is explicitly designed for third-party harvester services. Farmers who use such services connect to harvesters they do not fully control. The attack requires only that the harvester has a valid plot key (obtained legitimately through the normal `HarvesterHandshake` flow) and finds a winning proof of space — both are normal operating conditions.

### Recommendation
In `_process_respond_signatures`, before applying `farmer_reward_address_override`, validate that:
1. The corresponding `NewProofOfSpace` carried a non-None `fee_info` with a valid `applied_fee_threshold` (store this alongside the proof in `self.farmer.proofs_of_space`).
2. The `applied_fee_threshold` meets or exceeds the computed fee quality for the proof (`calculate_harvester_fee_quality`).
3. The override address matches the one declared in `NewProofOfSpace` (preventing substitution at the `RespondSignatures` stage).

If any check fails, reject the override and use `self.farmer.farmer_target`.

```python

    sp_hash=valid_sp_hash,
    local_pk=valid_local_pk,

assert result.farmer_reward_address == attacker_address      # True
```
