### Title
Missing Response-to-Request Binding in `respond_verify_foreign_tx` Allows Byzantine Participant to Resolve Wrong Yield — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid signature over `response.payload_hash` using the domain key, but **never verifies that `response.payload_hash` encodes `request.request`**. A Byzantine attested participant can therefore submit a cryptographically valid response for request_A while passing request_B as the lookup key, causing Bob's yield (queued under request_B) to be irrevocably resolved with Alice's response data.

The "structural identity" framing in the question is a red herring — no type confusion is needed. The exploit is a straightforward missing binding check.

---

### Finding Description

`VerifyForeignTransactionRequestArgs` and `VerifyForeignTransactionRequest` are indeed byte-for-byte identical in both JSON and Borsh: [1](#0-0) 

Both carry `{ request: ForeignChainRpcRequest, domain_id: DomainId, payload_version: ForeignTxPayloadVersion }` with no additional discriminating field. However, the actual vulnerability is in `respond_verify_foreign_tx`: [2](#0-1) 

The contract:
1. Fetches the public key for `request.domain_id`
2. Verifies `response.signature` is valid over `response.payload_hash` using that key
3. Calls `resolve_yields_for(&pending_verify_foreign_tx_requests, &request, response_bytes)`

**What is never checked**: that `response.payload_hash` equals `hash(ForeignTxSignPayload::V1 { request: request.request, values: ... })`. The contract only verifies the signature is valid over *some* hash using the domain key — it does not bind that hash to the `request` parameter passed in.

The `payload_hash` is defined as:

```
H(ForeignTxSignPayload::V1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> })
```

The contract cannot reconstruct this hash at respond-time because it does not know `values` (the extracted foreign-chain data). This is by design — but the consequence is that the contract cannot enforce the binding, and it does not reject a response whose `payload_hash` encodes a different `ForeignChainRpcRequest` than the one in `request`.

`resolve_yields_for` then drains the entire queue for the submitted key and fires `promise_yield_resume` for every queued yield: [3](#0-2) 

---

### Impact Explanation

A Byzantine attested participant who holds a valid MPC response for request_A (tx_id_A, domain_D) can call:

```
respond_verify_foreign_tx(request_B, response_A)
```

where request_B is a different pending request (tx_id_B, same domain_D). The signature check passes (valid sig over `H(tx_id_A, values_A)` using domain_D's key). `resolve_yields_for` then drains the queue for request_B, delivering `response_A` to Bob's yield. Bob's transaction completes with a `VerifyForeignTransactionResponse` whose `payload_hash` encodes Alice's transaction data, not Bob's.

Consequences:
- **Bob's yield is irrevocably consumed** with wrong data. His original transaction completes (with wrong response); he cannot "un-resolve" a yield.
- **Bob receives a response that does not correspond to his request.** If his bridge application uses `ForeignChainSignatureVerifier::verify_signature` from the SDK, it will detect the mismatch (the reconstructed expected hash won't match `response.payload_hash`). If it does not re-verify, it may accept Alice's transaction data as proof of Bob's transaction.
- **Alice's yield is unaffected** — the attacker can still (or separately) call `respond_verify_foreign_tx(request_A, response_A)` to resolve Alice's yield.

The contract-level invariant — *respond_verify_foreign_tx resolves exactly the yields queued for the submitted request key, with a response that corresponds to that key* — is broken.

---

### Likelihood Explanation

**Precondition**: The attacker must be an attested participant (TEE-attested, Byzantine below threshold). This is a meaningful barrier but is explicitly within the Byzantine-participant threat model.

**Opportunity**: The attacker needs a valid MPC response for any request on the same domain. As a participant, they process all requests and have access to all responses before submitting them on-chain. They can trivially hold back response_A and submit it against request_B.

**Trigger**: Two different callers must have concurrent pending requests on the same domain. This is a normal production condition for any active bridge.

---

### Recommendation

Add a binding check in `respond_verify_foreign_tx` that verifies `response.payload_hash` encodes `request.request`. Since the contract does not know `values`, it cannot reconstruct the full `ForeignTxSignPayload` hash. Two options:

1. **Include the `ForeignChainRpcRequest` hash as a separate commitment in the response** — MPC nodes sign `H(request || values)` but also include `H(request)` separately, and the contract verifies `H(request)` matches `request.request`.
2. **Embed the request key inside `payload_hash`** — change `ForeignTxSignPayload` so that `H(request)` is a prefix or domain separator, allowing the contract to verify the request component without knowing `values`.

Either approach closes the gap without requiring the contract to know the extracted values.

### Proof of Concept

```rust
// Unit test sketch (contract test harness)
// 1. Setup: two pending requests on the same domain
let request_args_A = VerifyForeignTransactionRequestArgs { request: rpc_request_A, domain_id, payload_version: V1 };
let request_args_B = VerifyForeignTransactionRequestArgs { request: rpc_request_B, domain_id, payload_version: V1 };
let request_A = args_into_verify_foreign_tx_request(request_args_A.clone());
let request_B = args_into_verify_foreign_tx_request(request_args_B.clone());

contract.verify_foreign_transaction(request_args_A); // Alice's yield queued under request_A
contract.verify_foreign_transaction(request_args_B); // Bob's yield queued under request_B

// 2. MPC produces response_A (for request_A)
let payload_A = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: rpc_request_A, values: values_A });
let response_A = sign_with_domain_key(payload_A, &secret_key);

// 3. Byzantine participant submits response_A against request_B
with_active_participant_and_attested_context(&contract);
let result = contract.respond_verify_foreign_tx(request_B.clone(), response_A.clone());

// 4. Contract accepts (signature is valid over payload_A's hash using domain key)
assert!(result.is_ok(), "contract incorrectly accepts mismatched response");

// 5. Bob's yield is now consumed with Alice's response data
assert!(contract.pending_verify_foreign_tx_requests.get(&request_B).is_none(),
    "Bob's yield was drained with wrong response");
// Alice's yield is still pending
assert!(contract.pending_verify_foreign_tx_requests.get(&request_A).is_some(),
    "Alice's yield is unaffected");
```

The test would pass at step 4 (contract accepts) and confirm at step 5 that Bob's yield was resolved with Alice's response — demonstrating the invariant violation.

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
```
