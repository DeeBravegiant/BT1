Audit Report

## Title
Donation Attack via Open `receive()` Inflates `rsETHPrice`, Enabling Direct Theft from Depositors - (File: contracts/LRTOracle.sol)

## Summary
`LRTDepositPool.receive()` is open, allowing anyone to donate ETH without minting rsETH. Because `getETHDistributionData()` uses the raw `address(this).balance`, a donation inflates `totalETHInProtocol`. Since `pricePercentageLimit` is never set in `initialize()` and defaults to `0`, the price-cap guard in `_updateRsETHPrice()` is permanently disabled, allowing an arbitrarily large single-block price increase. Subsequent depositors receive near-zero rsETH for their ETH, while the attacker—holding the only existing rsETH—redeems at the inflated price, extracting the difference.

## Finding Description

**Root cause chain:**

**Step 1 — Raw balance used in TVL accounting.**
`LRTOracle._getTotalEthInProtocol()` calls `ILRTDepositPool.getTotalAssetDeposits(asset)`, which calls `getAssetDistributionData()`, which for ETH delegates to `getETHDistributionData()`:

```solidity
// LRTDepositPool.sol line 480
ethLyingInDepositPool = address(this).balance;
```

For ERC-20 LSTs (line 444):
```solidity
assetLyingInDepositPool = IERC20(asset).balanceOf(address(this));
```

Both use the raw contract balance, which includes any tokens or ETH sent directly without going through `depositAsset`/`depositETH`.

**Step 2 — Open `receive()` enables costless balance inflation.**

```solidity
// LRTDepositPool.sol line 58
receive() external payable { }
```

Any actor can send ETH to `LRTDepositPool` without minting rsETH.

**Step 3 — Price-cap guard is permanently disabled at deployment.**

`initialize()` (lines 64–68) does not set `pricePercentageLimit`, so it defaults to `0`. The guard in `_updateRsETHPrice()`:

```solidity
// LRTOracle.sol lines 256–257
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```

When `pricePercentageLimit == 0`, `isPriceIncreaseOffLimit` is always `false`. The revert at line 264 is never reached by an unprivileged caller.

**Step 4 — Price update is public.**

```solidity
// LRTOracle.sol line 87
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

**Step 5 — Minting formula uses rsETHPrice as denominator.**

```solidity
// LRTDepositPool.sol line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

An inflated `rsETHPrice` causes integer division to round down to near-zero rsETH for subsequent depositors.

**Step 6 — Redemption uses rsETHPrice as numerator.**

```solidity
// LRTWithdrawalManager.sol line 593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

The attacker's rsETH is redeemed at the inflated price.

**Exploit flow (concrete numbers):**

| Step | rsethSupply | totalETHInProtocol | rsETHPrice (divWad result) |
|---|---|---|---|
| Attacker deposits 1 wei ETH | 1 wei | 1 wei | 1e18 |
| Attacker donates 1 000 ETH via `receive()` | 1 wei | 1 000e18 + 1 | — |
| `updateRSETHPrice()` called | 1 wei | 1 000e18 + 1 | `(1000e18+1)*1e18/1` ≈ 1000e36 |
| Victim deposits 1 999 ETH, `minExpected=0` | 2 wei | 2 999e18 + 1 | — |
| `updateRSETHPrice()` called | 2 wei | 2 999e18 + 1 | `(2999e18+1)*1e18/2` ≈ 1499.5e36 |
| Attacker initiates withdrawal of 1 wei rsETH | — | — | receives ≈ 1499.5 ETH |

Attacker spent ≈ 1 000 ETH, receives ≈ 1 499.5 ETH → **net profit ≈ 499.5 ETH**.
Victim deposited 1 999 ETH, receives rsETH worth ≈ 1 499.5 ETH → **net loss ≈ 499.5 ETH**.

*(Note: the submitted claim's table shows rsETHPrice ≈ 1000e18 rather than the correct ≈ 1000e36, but this is a presentation error; the profit arithmetic and mechanism are correct.)*

**Why existing checks fail:**

- `_beforeDeposit` slippage guard (line 667) only triggers if `rsethAmountToMint < minRSETHAmountExpected`; a victim passing `minRSETHAmountExpected = 0` receives no protection.
- The `pricePercentageLimit` guard (lines 256–257) is short-circuited to `false` when `pricePercentageLimit == 0`.
- `withdrawalDelayBlocks` (8 days) delays but does not prevent the attack; the attacker simply waits.
- `unlockQueue` is operator-gated, but operators are expected to process the queue regularly; `instantWithdrawal` bypasses the delay entirely if enabled.

## Impact Explanation

**Critical — Direct theft of user funds.**

The attacker extracts real ETH from subsequent depositors proportional to the donation size. The loss is not yield; it is principal deposited by victims. This matches the allowed impact: *"Direct theft of any user funds, whether at-rest or in-motion, other than unclaimed yield."*

## Likelihood Explanation

- `pricePercentageLimit` is `0` at deployment and remains `0` unless an admin explicitly calls `setPricePercentageLimit`. No deployment script or `initialize()` sets it.
- `updateRSETHPrice()` is permissionless; the attacker controls the timing.
- `LRTDepositPool.receive()` is unconditionally open; ETH donation costs only gas plus the donated amount.
- The attack is repeatable: after each victim deposit, the attacker can re-inflate and re-redeem.
- No privileged access, no oracle compromise, no governance capture required.

## Recommendation

1. **Initialize `pricePercentageLimit` to a non-zero value in `initialize()`** (e.g., `1e16` for 1% per update) so the guard is active from deployment.
2. **Snapshot-based accounting**: track deposited balances in a storage variable incremented only through `depositETH`/`depositAsset`, rather than reading raw `address(this).balance` or `IERC20.balanceOf`. Donated amounts would then be excluded from TVL until explicitly swept by an operator.
3. **Alternatively**, add a `minRSETHAmountExpected > 0` enforcement in `_beforeDeposit` to prevent zero-rsETH deposits.

## Proof of Concept

Minimal Foundry fork test outline:

```solidity
function test_donationAttack() public {
    // 1. Attacker deposits 1 wei ETH → gets 1 wei rsETH
    vm.prank(attacker);
    depositPool.depositETH{value: 1}(0, "");
    assertEq(rsETH.balanceOf(attacker), 1);

    // 2. Attacker donates 1000 ETH directly
    vm.deal(attacker, 1000 ether + 1);
    vm.prank(attacker);
    (bool ok,) = address(depositPool).call{value: 1000 ether}("");
    assertTrue(ok);

    // 3. Inflate price (pricePercentageLimit == 0, no revert)
    oracle.updateRSETHPrice();
    uint256 inflatedPrice = oracle.rsETHPrice();
    assertGt(inflatedPrice, 1000e18); // ~1000e36

    // 4. Victim deposits 1999 ETH, gets ~1 wei rsETH
    vm.deal(victim, 1999 ether);
    vm.prank(victim);
    depositPool.depositETH{value: 1999 ether}(0, "");
    assertLe(rsETH.balanceOf(victim), 2); // near-zero rsETH

    // 5. Attacker initiates withdrawal at inflated price
    vm.prank(attacker);
    rsETH.approve(address(withdrawalManager), 1);
    vm.prank(attacker);
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, 1, "");

    // expectedAssetAmount ≈ 1499.5 ETH — attacker profit ≈ 499.5 ETH
    (,uint256 expected,,) = withdrawalManager.getUserWithdrawalRequest(ETH_TOKEN, attacker, 0);
    assertGt(expected, 1400 ether);
}
```
