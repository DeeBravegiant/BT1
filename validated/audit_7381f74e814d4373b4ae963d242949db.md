### Title
Missing Value Conservation Check in Sub-Transaction UTxO Validation Enables ADA Creation - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

### Summary

The Dijkstra era introduces sub-transactions (batched transactions nested inside a top-level transaction). The sub-transaction UTxO validation rule (`dijkstraSubUtxoTransition`) applies all standard structural checks but omits the `validateValueNotConservedUTxO` check that every other era's UTXO rule enforces. Because no balance invariant is verified for sub-transactions, an unprivileged sender can craft a sub-transaction whose outputs exceed its inputs by an arbitrary amount, permanently injecting that excess ADA into the UTxO set.

### Finding Description

Every era from Shelley through Conway enforces value conservation in its UTXO transition rule. For example, the top-level Dijkstra UTXO rule calls:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [1](#0-0) 

The sub-transaction UTxO transition (`dijkstraSubUtxoTransition`) runs a series of structural checks — validity interval, forecast, output size, input existence, network IDs, max tx size, execution units — but never calls `validateValueNotConservedUTxO`:

```haskell
dijkstraSubUtxoTransition = do
  TRC (SubUtxoEnv slot pp certState originalUtxo (IsValid isValid), utxoState, stAnnTx) <-
    judgmentContext
  ...
  runTest $ Shelley.validateInputSetEmptyUTxO txBody
  runTest $ Shelley.validateBadInputsUTxO originalUtxo (inputs `Set.union` refInputs)
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxoState) inputs
  -- *** validateValueNotConservedUTxO is absent ***
  if isValid
    then do
      newState <-
        Shelley.updateUTxOStateNoFees pp utxoState txBody certState ...
      pure $ newState & utxosDonationL <>~ txBody ^. treasuryDonationTxBodyL
    else pure utxoState
``` [2](#0-1) 

`updateUTxOStateNoFees` unconditionally removes the sub-transaction's inputs from the UTxO and inserts its outputs, with no prior balance check:

```haskell
updateUTxOStateNoFees pp utxos txBody certState govState ... = do
  let ...
      !utxoAdd = txouts txBody          -- outputs inserted unconditionally
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      ...
``` [3](#0-2) 

The top-level transaction's balance check cannot compensate for this gap. It uses `originalUtxo` and the top-level `txBody` only; it has no visibility into sub-transaction inputs or outputs:

```haskell
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [1](#0-0) 

Furthermore, the top-level rule enforces that its own inputs must exist in `originalUtxo`, so the top-level transaction cannot spend sub-transaction outputs in the same batch. The inflated outputs simply persist in the UTxO state and are spendable in any subsequent transaction.

Sub-transaction bodies have no fee field (`DijkstraSubTxBodyRaw` omits `dtbrFee`), so `updateUTxOStateNoFees` is the correct state-update path — but it was designed for the case where inputs and outputs already balance, relying on a prior conservation check that was never added. [4](#0-3) 

### Impact Explanation

An attacker can permanently inject an arbitrary quantity of ADA into the UTxO by submitting a Dijkstra-era transaction containing a sub-transaction whose outputs exceed its inputs. The excess ADA is indistinguishable from legitimately created UTxO entries and can be spent freely in subsequent transactions. This constitutes direct, unbounded creation of ADA through an invalid ledger state transition — the most severe class of ledger vulnerability.

**Impact: Critical — Direct creation of ADA through an invalid ledger state transition.**

### Likelihood Explanation

The Dijkstra era is a production implementation target in this repository. The sub-transaction feature is reachable by any unprivileged transaction sender with access to a Dijkstra-era node. No special role, key, or governance threshold is required. The attacker needs only to construct a valid sub-transaction body (all structural checks pass) with outputs summing to more than its inputs. The attack is deterministic and repeatable.

### Recommendation

Add `validateValueNotConservedUTxO` (or an equivalent batch-aware conservation check) to `dijkstraSubUtxoTransition`, immediately before the `updateUTxOStateNoFees` call, using the sub-transaction's own UTxO slice and cert state:

```haskell
-- Inside dijkstraSubUtxoTransition, before updateUTxOStateNoFees:
runTest $ Shelley.validateValueNotConservedUTxO pp (utxosUtxo utxoState) certState txBody
```

The check should use the current (post-prior-sub-tx) UTxO state rather than `originalUtxo`, because earlier sub-transactions in the same batch may have already consumed some inputs. Regression tests should verify that a sub-transaction with `sum(outputs) > sum(inputs) + refunds - deposits` is rejected with `ValueNotConservedUTxO`.

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction containing one sub-transaction.
2. The sub-transaction spends a single UTxO worth 1 ADA and produces a single output worth 1,000,000 ADA. All structural checks pass (non-empty inputs, inputs exist in UTxO, output above minimum, valid network ID, etc.).
3. Submit the transaction. `dijkstraSubUtxoTransition` runs all checks, finds no violation, and calls `updateUTxOStateNoFees`, which removes the 1 ADA input and inserts the 1,000,000 ADA output.
4. The top-level transaction's `validateValueNotConservedUTxO` checks only the top-level body against `originalUtxo`; it sees no imbalance.
5. The 1,000,000 ADA output is now a valid UTxO entry. In the next transaction, the attacker spends it, realising 999,999 ADA of created value.
6. The attack is repeatable without limit.

updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let UTxOState {utxosUtxo, utxosDeposited, utxosFees, utxosDonation} = utxos
      UTxO utxo = utxosUtxo
      !utxoAdd = txouts txBody -- These will be inserted into the UTxO
      {- utxoDel  = txins txb ◁ utxo -}
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      {- newUTxO = (txins txb ⋪ utxo) ∪ outs txb -}
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      deletedUTxO = UTxO utxoDel
      totalRefunds = certsTotalRefundsTxBody pp certState txBody
      totalDeposits = certsTotalDepositsTxBody pp certState txBody
      depositChange = totalDeposits <-> totalRefunds
  depositChangeEvent depositChange
  txUtxODiffEvent deletedUTxO utxoAdd
  pure $!
    UTxOState
      { utxosUtxo = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees = utxosFees
      , utxosGovState = govState
      , utxosInstantStake =
          deleteInstantStake deletedUTxO (addInstantStake utxoAdd (utxos ^. instantStakeL))
      , utxosDonation = utxosDonation
      }
