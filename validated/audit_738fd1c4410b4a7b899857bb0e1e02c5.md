### Title
`accept_requests = false` Blocks Completion of Already-In-Flight Signature Requests - (`File: crates/contract/src/lib.rs`)

### Summary

When `verify_tee()` determines that fewer than threshold participants have valid TEE attestations, it sets `self.accept_requests = false`. This flag is then checked in **both** new-request submission paths **and** the `respond`/`respond_ckd`/`respond_verify_foreign_tx` completion paths. As a result, signature requests that were already enqueued in the yield-resume queue before the flag was set can never be completed by MPC nodes, permanently freezing those in-flight requests until NEAR's yield timeout expires.

### Finding Description

`verify_tee()` sets `accept_requests = false` when TEE validation fails: [1](#0-0) 

This flag is correctly checked in `check_request_preconditions`, which gates all new user-facing request submissions (`sign`, `request_app_private_key`, `verify_foreign_transaction`): [2](#0-1) 

However, the same flag is also checked inside the node-facing **response** methods that complete already-queued requests:

- `respond()`: [3](#0-2) 

- `respond_ckd()`: [4](#0-3) 

- `respond_verify_foreign_tx()`: [5](#0-4) 

Pending requests are stored in `pending_signature_requests`, `pending_ckd_requests`, and `pending_verify_foreign_tx_requests` as yield-resume entries: [6](#0-5) 

Once `accept_requests` is `false`, MPC nodes calling `respond*` receive `TeeError::TeeValidationFailed` and cannot resolve any of those queued yields. The requests remain stuck until NEAR's yield-resume timeout fires, at which point callers receive a failure response rather than their signature.

### Impact Explanation

Any signature, CKD, or foreign-transaction-verification request that was submitted before `accept_requests` was set to `false` is permanently frozen. Users cannot obtain their signatures and must wait for the yield timeout to receive a failure. This breaks the request-lifecycle safety invariant: a request that was validly accepted by the contract (deposit paid, yield created) can never be fulfilled, even if the MPC nodes are fully capable of producing the signature. This matches the **Medium** allowed impact: "request-lifecycle … manipulation that breaks production safety/accounting invariants."

### Likelihood Explanation

`verify_tee()` is callable by any current participant (voter): [7](#0-6) 

TEE attestations expire naturally over time, and governance can remove image hashes via `vote_remove_launcher_hash` or `vote_remove_os_measurement`, both of which cause previously valid attestations to fail re-verification. Either event triggers `TeeValidationResult::Partial` and sets `accept_requests = false`. Because `verify_tee()` is a routine operational call (nodes are expected to call it periodically), the window where in-flight requests are blocked is a normal operational condition, not an exotic edge case.

### Recommendation

Decouple the two concerns. The `accept_requests` flag should only gate **new** request submissions. Remove the `if !self.accept_requests` guard from `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. Those methods already enforce that the caller is an attested participant and that the protocol is running or resharing, which is sufficient authorization for completing an already-accepted request.

Analogously to the Isomorph fix: just as `closeLoan`/`callLiquidation` were decoupled from `collateralValid`, the `respond*` paths must be decoupled from `accept_requests`.

### Proof of Concept

1. User calls `sign(request)` — a yield-resume entry is created and stored in `pending_signature_requests`. [8](#0-7) 

2. Before MPC nodes call `respond`, a participant calls `verify_tee()`. Attestations have expired, so `TeeValidationResult::Partial` is returned and `accept_requests` is set to `false`. [9](#0-8) 

3. MPC nodes compute the threshold signature and call `respond(request, response)`. The method immediately returns `Err(TeeError::TeeValidationFailed)` at the `accept_requests` check before reaching `resolve_yields_for`. [3](#0-2) 

4. `resolve_yields_for` is never called; the yield in `pending_signature_requests` is never resumed. [10](#0-9) 

5. The user's request times out and fails. The signature is lost despite the MPC network being capable of producing it.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
