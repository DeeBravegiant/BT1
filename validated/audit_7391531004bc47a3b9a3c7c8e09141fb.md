Audit Report

## Title
Division by Zero in `getRsETHAmountToMint` When `rsETHPrice` Is Uninitialized — (File: contracts/LRTDepositPool.sol)

## Summary
`LRTOracle.rsETHPrice` is a `uint256` state variable that defaults to `0` and is never set in `initialize`. Until `updateRSETHPrice()` is called at least once, every call to `depositETH` or `depositAsset` flows through `getRsETHAmountToMint`, which divides by `lrtOracle.rsETHPrice()` without a zero-guard, causing a Solidity 0.8 division-by-zero panic and reverting the transaction. No user funds are lost, but the deposit interface is completely non-functional during this window.

## Finding Description
`LRTOracle` declares `rsETHPrice` as a plain storage variable with no initialization in `initialize`:

```solidity
// LRTOracle.sol L28
uint256 public override rsETHPrice;

// LRTOracle.sol L64-68 — initialize() never sets rsETHPrice
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

`rsETHPrice` is only written by `_updateRsETHPrice()`, which is triggered by the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. When `rsethSupply == 0` (i.e., before any deposits), `_updateRsETHPrice()` correctly sets `rsETHPrice = 1 ether`, but this requires an explicit call.

`LRTDepositPool.getRsETHAmountToMint` performs an unguarded division:

```solidity
// LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The call chain for any deposit is: `depositETH` / `depositAsset` → `_beforeDeposit` (L648–670) → `getRsETHAmountToMint` → division by zero panic when `rsETHPrice == 0`.

No existing check in `_beforeDeposit` guards against a zero `rsETHPrice`; the only checks are for `depositAmount == 0`, `depositAmount < minAmountToDeposit`, and the deposit limit.

## Impact Explanation
All calls to `depositETH` and `depositAsset` revert with a Solidity panic while `rsETHPrice == 0`. No user funds are lost or frozen (deposits simply fail), but the contract cannot deliver its core promised function — accepting deposits and minting rsETH. This matches the **Low** impact tier: *"Contract fails to deliver promised returns, but doesn't lose value."*

## Likelihood Explanation
The vulnerable window exists from deployment until the first successful `updateRSETHPrice()` call. Because `updateRSETHPrice()` is public and permissionless (only gated by `whenNotPaused`), any actor — including the first depositor — can close the window by calling it. In practice the freeze is short-lived, but it is a guaranteed revert for any depositor who does not know to call the price-update function first. Likelihood is **Low**.

## Recommendation
Add an explicit zero-guard before the division in `getRsETHAmountToMint`:

```solidity
uint256 price = lrtOracle.rsETHPrice();
require(price > 0, "rsETHPrice not initialized");
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / price;
```

Additionally, call `_updateRsETHPrice()` (or directly set `rsETHPrice = 1 ether`) inside `LRTOracle.initialize` so the contract is safe from block zero without requiring a separate transaction.

## Proof of Concept
1. Deploy `LRTOracle` and `LRTDepositPool`; do **not** call `updateRSETHPrice()`.
2. Confirm `LRTOracle.rsETHPrice() == 0`.
3. Call `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. Transaction reverts with a Solidity panic (division by zero) at `LRTDepositPool.sol` L520 because `lrtOracle.rsETHPrice() == 0`.
5. Call `LRTOracle.updateRSETHPrice()` (public, no role required; sets `rsETHPrice = 1 ether` since `rsethSupply == 0`).
6. Repeat step 3 — deposit succeeds.
