### Title
Byzantine Equivocation via `echo_ready_thresholds(n≤3)=(0,0)` Causes Honest-Node Divergence on DKG Public Key — (`crates/threshold-signatures/src/protocol/echo_broadcast.rs`)

---

### Summary

`echo_ready_thresholds` returns `(0, 0)` for any `n ≤ 3`, collapsing the delivery condition in `reliable_broadcast_receive_all` to `count > 0`. A single self-simulated Ready vote is sufficient to deliver a value. A Byzantine participant can therefore send different commitments to different honest participants in the same broadcast session, causing them to derive different master public keys after DKG — with no abort triggered.

---

### Finding Description

**Root cause — `echo_ready_thresholds`** [1](#0-0) 

For `n ≤ 3`, both `echo_t` and `ready_t` are `0`. The Ready-delivery condition is: [2](#0-1) 

`count > 2 * 0` means `count > 0`. The self-simulated Ready vote (injected immediately after the Echo phase) always satisfies this, so delivery happens before any message from a peer is needed.

**Equivocation trace (n=3: honest P1, honest P2, Byzantine B)**

*Round 3 of `do_keyshare` — commitment broadcast via `do_broadcast`:* [3](#0-2) 

B sends `Send(commitment_B_A)` to P1 and `Send(commitment_B_B)` to P2.

- P1 receives `Send(commitment_B_A)` → echoes it → simulates own Echo (count=1 > `echo_t`=0) → sends `Ready(commitment_B_A)` → simulates own Ready (count=1 > `2*ready_t`=0) → **delivers `commitment_B_A`**.
- P2 receives `Send(commitment_B_B)` → same path → **delivers `commitment_B_B`**.

Each honest party delivers a different value for B's broadcast. The consistency property of reliable broadcast is broken.

**Why no abort is triggered**

*Round 2 commitment hash (plain `send_many`, no broadcast protection):* [4](#0-3) 

B pre-computes two matching hashes: `H(B, commitment_B_A, session_id)` → P1, `H(B, commitment_B_B, session_id)` → P2.

*Round 4 `verify_commitment_hash`:* [5](#0-4) 

P1 checks its received commitment against its received hash → passes. P2 does the same → passes. Neither aborts.

*Round 4 `verify_proof_of_knowledge`:* B knows the discrete log behind both commitments and can produce a valid Schnorr proof for each. Both honest parties accept. [6](#0-5) 

*Round 5 `validate_received_share`:* B sends shares consistent with each commitment privately. Both pass.

*`broadcast_success`:* B was honest in Round 1, so P1 and P2 share the same `session_id`. Both broadcast `(true, session_id)`. The check `all(|(_, sid)| sid == &session_id)` passes for both. [7](#0-6) 

**Divergent output**

```
P1: public_key = commitment_P1(0) + commitment_P2(0) + commitment_B_A(0)
P2: public_key = commitment_P1(0) + commitment_P2(0) + commitment_B_B(0)
``` [8](#0-7) 

Both `KeygenOutput` values are returned without error. The MPC network has no agreed-upon master public key.

---

### Impact Explanation

Honest nodes hold irreconcilably different views of the master public key after a completed DKG run. Signatures produced under one node's key share are invalid from the other node's perspective. B can selectively collude with whichever honest node holds the key share consistent with B's chosen commitment, enabling unauthorized threshold signature issuance without the full required participant set. This maps to: **Critical — unauthorized threshold signature issuance / honest-node divergence on key state**.

---

### Likelihood Explanation

The vulnerability is triggered whenever `n ≤ 3`. `assert_key_invariants` only requires `participants.len() >= 2` and `threshold >= 2`, so `n=3` is a fully valid, library-permitted configuration. [9](#0-8) 

The attack requires exactly one Byzantine participant — strictly below the signing threshold — and no special access beyond the ability to send protocol messages. The equivocation is deterministic and requires no brute force.

---

### Recommendation

The `(0, 0)` special case must be removed. For `n ≤ 3`, the BFT broadcast threshold `f = floor((n-1)/3)` is `0`, meaning the protocol tolerates zero Byzantine faults. The correct behavior is to require **all** `n` Echo votes before sending Ready (echo threshold = `n-1`) and **all** `n` Ready votes before delivery (ready threshold = `n-1`), or alternatively reject `n ≤ 3` as an unsupported configuration at the `assert_key_invariants` level. [1](#0-0)

### Proof of Concept

```
Participants: P1 (honest), P2 (honest), B (Byzantine), n=3, threshold=2

Round 1: B broadcasts my_session_id honestly → session_id is identical for P1 and P2.

Round 2 (send_many, no broadcast):
  B → P1: commitment_hash_A = H(B, C_A, session_id)
  B → P2: commitment_hash_B = H(B, C_B, session_id)   // C_A ≠ C_B

Round 3 (do_broadcast, vulnerable):
  B → P1: Send(C_A)   // P1 echoes C_A, self-simulates Echo (count=1>0),
                       // sends Ready(C_A), self-simulates Ready (count=1>0),
                       // delivers C_A — DONE, finish_ready=true
  B → P2: Send(C_B)   // P2 delivers C_B by same path

Round 4:
  P1: verify H(B, C_A, session_id) == commitment_hash_A  ✓
  P2: verify H(B, C_B, session_id) == commitment_hash_B  ✓
  Both proof-of-knowledge checks pass (B knows secrets for both C_A and C_B).

Round 5 (private):
  B → P1: f_B(P1) consistent with C_A  ✓
  B → P2: f_B(P2) consistent with C_B  ✓

broadcast_success: same session_id for all → passes.

Output:
  P1.public_key = C_P1(0) + C_P2(0) + C_A(0)
  P2.public_key = C_P1(0) + C_P2(0) + C_B(0)   ← divergence, no error
```

Assert: `P1.public_key != P2.public_key` — DKG consistency invariant violated.

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
