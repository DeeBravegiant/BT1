### Title
Attacker-Controlled `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies the MPC signature against `response.payload_hash`, a value supplied entirely by the calling node. The contract never independently derives the expected hash from the pending request. A single Byzantine participant who has previously led a legitimate signing session can replay that session's final signature — paired with its original `payload_hash` — as the response to a *different* pending request, causing the contract to deliver a forged verification result to the waiting user.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following check:

```rust
// crates/contract/src/lib.rs  lines 726-734
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← attacker-supplied

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← verified against itself, not against the request
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The `payload_hash` field is taken verbatim from the node-supplied `VerifyForeignTransactionResponse`. The contract checks only that the signature is self-consistent with that hash; it never verifies that the hash was actually derived from the pending `request` (i.e., from `SHA-256(borsh(ForeignTxSignPayload { request, extracted_values }))`). [2](#0-1) 

By contrast, the off-chain SDK's `ForeignChainSignatureVerifier::verify_signature` *does* independently recompute the expected hash from the request and the extracted values, then asserts `expected_payload_hash == response.payload_hash`:

```rust
// crates/near-mpc-sdk/src/foreign_chain.rs  lines 53-63
let expected_payload_hash = expected_payload.compute_msg_hash()?;
let payload_is_correct = expected_payload_hash == response.payload_hash;
if !payload_is_correct {
    return Err(VerifyForeignChainError::IncorrectPayloadSigned { … });
}
``` [3](#0-2) 

The on-chain contract performs no equivalent check. After the signature check passes, the contract immediately resolves all queued yields for the pending request with the unverified response:

```rust
// crates/contract/src/lib.rs  lines 749-753
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [4](#0-3) 

---

### Impact Explanation

**Impact: High** — Forged foreign-chain verification / cross-request signature replay causing invalid bridge execution.

A Byzantine leader who has completed a legitimate signing session for request A holds the final ECDSA signature `sig_A` over `payload_hash_A`. It can call:

```
respond_verify_foreign_tx(
    request  = request_B,          // a different, legitimately pending request
    response = { payload_hash: payload_hash_A, signature: sig_A }
)
```

The contract will:
1. Confirm `sig_A` is a valid ECDSA signature over `payload_hash_A` under the root key — **true**.
2. Confirm `request_B` is in `pending_verify_foreign_tx_requests` — **true**.
3. Resolve all yields for `request_B` with the forged response.

The user waiting on `request_B` receives `{ payload_hash: payload_hash_A, signature: sig_A }`. `payload_hash_A` encodes the extracted values for a *different* foreign transaction (request A). A bridge service that does not independently re-verify the payload hash (or that cannot, because it does not know the expected extracted values) will treat this as a valid attestation of request B's foreign-chain state, enabling double-spend or invalid bridge execution.

---

### Likelihood Explanation

**Likelihood: High.**

- Any single attested participant who has ever been the signing leader holds a reusable final signature.
- The attack requires no threshold collusion: one Byzantine leader suffices.
- The contract is designed for continuous operation with many concurrent pending requests, so a target request is almost always available.
- The attacker needs only to submit a single well-formed transaction; no special tooling is required.

---

### Recommendation

The contract must bind the signature to the specific pending request. Since the contract does not know the extracted values, the recommended fix is to embed the request's canonical identifier into the signed payload so the contract can verify the binding without knowing the extracted values. Concretely:

1. **Include a request-derived commitment in the signed payload.** For example, prepend `SHA-256(borsh(ForeignChainRpcRequest))` to the data that nodes sign, and have the contract verify this prefix against the known request before accepting the response.

2. **Alternatively, have nodes include extracted values in the response** and have the contract recompute and verify the payload hash on-chain (matching what the SDK already does in `ForeignChainSignatureVerifier::verify_signature`).

The minimal diff to the contract check:

```rust
// Derive the request-binding prefix the contract CAN compute independently
let request_hash = env::sha256_array(
    borsh::to_vec(&request.request).expect("borsh serialization must succeed")
);
// Verify the response payload_hash starts with (or commits to) request_hash
// before accepting the signature.
```

### Proof of Concept

1. Honest nodes complete a signing session for `request_A` (Bitcoin tx `[0xAA; 32]`). The leader obtains `(payload_hash_A, sig_A)`.
2. A user submits `verify_foreign_transaction` for `request_B` (Bitcoin tx `[0xBB; 32]`). The contract stores it in `pending_verify_foreign_tx_requests`.
3. The Byzantine leader calls:
   ```
   respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })
   ```
4. The contract at line 729–734 verifies `sig_A` over `payload_hash_A` — valid. It does not check that `payload_hash_A` encodes `request_B`'s data.
5. The contract resolves `request_B`'s yield with the forged response (line 749–753).
6. The user's callback receives `{ payload_hash: payload_hash_A, signature: sig_A }` — a valid MPC attestation for the *wrong* transaction — and any bridge logic acting on it executes under false premises. [5](#0-4) [6](#0-5)

        response: &VerifyForeignTransactionResponse,

        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
