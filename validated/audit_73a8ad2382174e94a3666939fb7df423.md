Audit Report

## Title
ETH Deposit Limit Check Missing Deposit Amount Allows Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison for ETH versus LST assets. The ETH branch evaluates `totalAssetDeposits > depositLimit`, omitting the incoming `amount`, while the LST branch correctly evaluates `totalAssetDeposits + amount > depositLimit`. When `totalAssetDeposits == depositLimit` the ETH check returns `false`, allowing a deposit to proceed past the configured cap.

## Finding Description
At [1](#0-0)  the ETH branch on line 679 reads:

```solidity
return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
```

whereas the LST branch on line 681 reads:

```solidity
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

When `totalAssetDeposits == depositLimit`, the ETH expression evaluates to `false`, so `_beforeDeposit` at [2](#0-1)  does not revert with `MaximumDepositLimitReached`, and the deposit proceeds. This is internally inconsistent: `getAssetCurrentLimit` at [3](#0-2)  already returns `0` when `totalAssetDeposits >= depositLimit`, confirming the protocol considers the cap reached, yet the guard still passes.

## Impact Explanation
Any depositor can push ETH holdings above the configured `depositLimitByAsset` cap by depositing when `totalAssetDeposits == depositLimit`. rsETH is minted for the excess deposit and the protocol takes on more ETH/EigenLayer exposure than governance intended. No funds are directly stolen, but the contract fails to enforce its own deposit cap — matching the allowed impact: **Low — Contract fails to deliver promised returns (deposit cap enforcement) but does not lose value**.

## Likelihood Explanation
`depositETH()` is a public payable function requiring no special privileges. The condition `totalAssetDeposits == depositLimit` is reachable in normal operation as deposits accumulate toward a governance-set round number. Any depositor monitoring on-chain state can observe the exact moment the total reaches the limit and submit a transaction in the same or next block. No victim cooperation, governance capture, or external dependency is required.

## Recommendation
Add `amount` to the ETH branch to match the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Governance sets `depositLimitByAsset[ETH_TOKEN] = 10_000 ether`.
2. Deposits accumulate until `getTotalAssetDeposits(ETH_TOKEN) == 10_000 ether`; `getAssetCurrentLimit(ETH_TOKEN)` returns `0`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `10_000 ether > 10_000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the attacker; `totalAssetDeposits` becomes `10_500 ether`, 5% above the cap.

Foundry test plan: deploy a fork, set the limit, fill deposits to exactly the limit via `vm.store` or repeated deposits, then call `depositETH{value: 1 ether}()` and assert it does **not** revert — demonstrating the bypass — then apply the fix and assert it **does** revert.
