Audit Report

## Title
ETH-Value Accounting Underflow in `transferAssetToDepositPool` Silently Zeros `ethValueInWithdrawal` While LSTs Remain in Converter, Enabling Below-Fair-Value rsETH Minting — (`contracts/LRTConverter.sol`)

## Summary
`transferAssetToDepositPool` debits `ethValueInWithdrawal` at the current oracle price, but the variable was credited at the oracle price at the time of deposit. When the oracle price rises between the two operations, the debit for a partial return can exceed the entire accumulated credit, flooring `ethValueInWithdrawal` to zero even though other LST batches remain physically in the converter. Because `getAssetDistributionData` sets `assetLyingInConverter = 0` for every LST and delegates converter accounting entirely to `ethValueInWithdrawal`, the remaining tokens become invisible to the TVL calculation. Any caller can then invoke the public `updateRSETHPrice()` to bake the deflated TVL into the stored rsETH price, allowing a deposit at below-fair-value and a profit when the true value is later recognized.

## Finding Description

**Root cause — `LRTConverter.sol` line 163:**
```solidity
ethValueInWithdrawal = ethValueInWithdrawal > assetValue ? ethValueInWithdrawal - assetValue : 0;
``` [1](#0-0) 

The credit side records ETH value at the oracle price at deposit time:
```solidity
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
``` [2](#0-1) 

If two batches A and B are deposited at price P1 (`ethValueInWithdrawal = (A+B)*P1`), the price rises to P2, and batch A is returned first, the debit is `A*P2`. When `A*P2 > (A+B)*P1` (i.e., price appreciation ≥ `B/A * 100%`), the floor-to-zero fires and `ethValueInWithdrawal = 0` even though batch B (worth `B*P2`) still sits in the converter.

**Why the remaining tokens become invisible:**

`getAssetDistributionData` explicitly sets `assetLyingInConverter = 0` for every LST, delegating converter accounting entirely to `getETHDistributionData`, which reads only `ethValueInWithdrawal`: [3](#0-2) [4](#0-3) 

**Why the deflated price is exploitable:**

`updateRSETHPrice()` is public — any caller can trigger it when the protocol is not paused: [5](#0-4) 

`_getTotalEthInProtocol` sums `getTotalAssetDeposits` for every supported asset, which for ETH routes through `getETHDistributionData` → `ethValueInWithdrawal`. A zeroed `ethValueInWithdrawal` directly reduces `totalETHInProtocol` and therefore `newRsETHPrice`: [6](#0-5) 

`getRsETHAmountToMint` divides by the stored `rsETHPrice`, so a deflated price mints more rsETH per unit deposited: [7](#0-6) 

**Existing guard — downside price protection:**

`_updateRsETHPrice` contains a downside protection that pauses the protocol if the price drop exceeds `pricePercentageLimit`: [8](#0-7) 

This guard is insufficient for two reasons: (1) if `pricePercentageLimit == 0` (its default unset value), the condition `pricePercentageLimit > 0` is false and the pause never fires; (2) even when set, the guard only triggers if the price drop exceeds the configured threshold — when the converter holds a small fraction of total TVL, the resulting price drop is proportionally small and stays within the limit, allowing the deflated price to be committed and deposits to proceed.

## Impact Explanation

**High — Theft of unclaimed yield.**

When `ethValueInWithdrawal` is zeroed while batch B of stETH remains in the converter, the TVL is understated by `B * P2`. `rsETHPrice` is deflated proportionally. An attacker deposits at the deflated price and receives excess rsETH. When batch B is eventually returned via `transferAssetToDepositPool`, TVL is restored, rsETH price rises, and the attacker redeems at a profit — extracting the yield accrued by the converter's LST holdings from existing rsETH holders.

## Likelihood Explanation

The triggering condition `A * P2 >= (A + B) * P1` simplifies to a price increase of at least `B/A * 100%`. In practice:
- If the operator moves a large batch A first and a small residual batch B second (a common operational pattern), even a 1–2% stETH price appreciation (routine staking yield over days) satisfies the condition.
- The operator is performing entirely legitimate operations; no compromise or collusion is required.
- `updateRSETHPrice()` is permissionless, so the attacker can atomically trigger the price update and deposit in the same block.
- The attack is repeatable every time the operator performs a partial return of converter assets during a period of LST price appreciation.

## Recommendation

Replace the floor-to-zero ETH-value debit with per-asset token-amount tracking. Store the raw token amounts deposited into the converter and compute the ETH value live from the current oracle price on read:

```solidity
// Instead of a single ethValueInWithdrawal, track token amounts per asset:
mapping(address => uint256) public assetAmountInWithdrawal;

// On transferAssetFromDepositPool:
assetAmountInWithdrawal[_asset] += _amount;

// On transferAssetToDepositPool:
assetAmountInWithdrawal[_asset] -= _amount; // reverts on underflow (Solidity 0.8+)

// ethValueInWithdrawal becomes a view function:
function ethValueInWithdrawal() external view returns (uint256 total) {
    address[] memory assets = lrtConfig.getSupportedAssetList();
    ILRTOracle oracle = ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE));
    for (uint256 i; i < assets.length; ++i) {
        if (assets[i] == LRTConstants.ETH_TOKEN) continue;
        total += (assetAmountInWithdrawal[assets[i]] * oracle.getAssetPrice(assets[i])) / 1e18;
    }
}
```

This ensures the reported ETH value always reflects the actual tokens held, regardless of price movements between deposit and return, and the Solidity 0.8 underflow revert provides an additional safety net against over-debiting.

## Proof of Concept

```solidity
// Setup: A = 100e18 stETH, B = 10e18 stETH, P1 = 1.00e18, P2 = 1.12e18

// Step 1: operator moves two batches to converter at P1
transferAssetFromDepositPool(stETH, 100e18);  // ethValueInWithdrawal = 100e18
transferAssetFromDepositPool(stETH,  10e18);  // ethValueInWithdrawal = 110e18

// Step 2: stETH oracle price rises to 1.12e18 (12% appreciation)

// Step 3: operator returns first batch at P2
// assetValue = 100e18 * 1.12e18 / 1e18 = 112e18 > 110e18
transferAssetToDepositPool(stETH, 100e18);    // ethValueInWithdrawal = 0  ← BUG
// 10e18 stETH (worth 11.2 ETH) still in converter, completely untracked

// Step 4: attacker calls updateRSETHPrice()
// totalETHInProtocol is understated by 11.2 ETH
// newRsETHPrice is deflated; if pricePercentageLimit == 0 or drop < limit, price is committed

// Step 5: attacker deposits ETH, receives excess rsETH at deflated price

// Step 6: operator returns second batch
transferAssetToDepositPool(stETH, 10e18);     // TVL restored

// Step 7: attacker calls updateRSETHPrice() again, rsETHPrice rises
// Attacker redeems rsETH at profit — yield stolen from existing holders

// Invariant that should hold but is violated after Step 3:
assert(ethValueInWithdrawal >= 10e18 * 1.12e18 / 1e18);  // FAILS: 0 >= 11.2e18
```

A Foundry fork test can confirm this by: (1) deploying against a mainnet fork with stETH as a supported asset, (2) simulating the two `transferAssetFromDepositPool` calls via the asset transfer role, (3) warping time and updating the stETH oracle price to P2, (4) calling `transferAssetToDepositPool` for batch A, (5) asserting `ethValueInWithdrawal == 0` while `IERC20(stETH).balanceOf(converter) == 10e18`, and (6) measuring the rsETH minted before and after the price restoration to quantify the attacker's profit.
