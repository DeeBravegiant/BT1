Audit Report

## Title
ETH Deposit Amount Excluded from Limit Check Allows Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` silently discards the `amount` parameter for the ETH branch, checking only the pre-deposit total against the cap rather than `totalAssetDeposits + amount`. This allows any depositor to push ETH TVL arbitrarily above `depositLimitByAsset(ETH)` in a single transaction, breaking the protocol's primary ETH deposit cap invariant.

## Finding Description
At `contracts/LRTDepositPool.sol` L676-682, the function receives `amount` but ignores it for ETH:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC20
}
```

The caller at L661-663 passes `depositAmount = msg.value`:

```solidity
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
```

Because the ETH branch evaluates only the pre-deposit state (`totalAssetDeposits > limit`), the check passes whenever the current total has not yet crossed the cap — regardless of how large `msg.value` is. The parallel view function `getAssetCurrentLimit` (L402-409) correctly computes `limit - totalAssetDeposits`, confirming the intended invariant is that `totalAssetDeposits + incomingAmount` must not exceed the cap. The ETH branch violates this invariant.

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's primary mechanism for controlling ETH restaked into EigenLayer strategies. Bypassing it causes excess ETH to sit idle in the NodeDelegator (unable to be deposited into EigenLayer if the strategy cap is also reached), diluting rsETH holders who receive minted rsETH backed by non-productive ETH. This matches the allowed impact: **Low — Contract fails to deliver promised returns** (deposit cap invariant broken; excess ETH may not earn restaking rewards, diluting rsETH value).

## Likelihood Explanation
The exploit requires no special role, no oracle manipulation, no front-running, and no victim mistake. Any address can call `depositETH` with an arbitrarily large `msg.value`. The vulnerable condition (`totalAssetDeposits < depositLimitByAsset(ETH)`) is the normal operating state of the protocol. The exploit is repeatable until the ETH branch is patched.

## Recommendation
Add the incoming deposit amount to the pre-deposit total for the ETH branch, matching the ERC20 branch:

```diff
 if (asset == LRTConstants.ETH_TOKEN) {
-    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
 }
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH) = 100 ether`.
2. Current `getTotalAssetDeposits(ETH) = 99.9 ether`.
3. Attacker calls `depositETH{value: 50 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 50 ether)` evaluates `99.9 ether > 100 ether` → `false` → no revert.
5. 50 ETH is accepted; total ETH becomes 149.9 ETH — 49.9 ETH above the cap.
6. `getAssetCurrentLimit(ETH)` now returns `0`; the cap has been exceeded by ~1.5×.

**Foundry test plan:**
```solidity
function testETHDepositBypassesLimit() public {
    uint256 limit = 100 ether;
    lrtConfig.setDepositLimitByAsset(LRTConstants.ETH_TOKEN, limit);
    // Bring total to just below cap
    depositPool.depositETH{value: 99.9 ether}(0, "");
    // Single call exceeds cap — should revert but does not
    depositPool.depositETH{value: 50 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), limit);
}
