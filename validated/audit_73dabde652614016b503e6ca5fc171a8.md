### Title
Cross-Request Replay of Foreign-Chain Verification Response Enables Forged Bridge Attestation - (File: `crates/contract/src/lib.rs`)

### Summary
`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid for the submitted `payload_hash` under the root public key, but never verifies that `payload_hash` actually corresponds to the foreign transaction described in the `request` parameter. A single Byzantine MPC participant (below threshold) can replay a previously observed legitimate `(payload_hash, signature)` pair against a different pending `verify_foreign_transaction` request, causing the contract to deliver a forged verification attestation to the caller.

### Finding Description

In `respond_verify_foreign_tx`, the contract performs two independent operations:

**1. Signature check** — verifies `response.signature` is valid for `response.payload_hash` under the domain's root public key: [1](#0-0) 

The comment "Check the signature is correct against the **root** public key" is the tell: no tweak is applied, so the check is entirely decoupled from the specific request.

**2. Yield resolution** — resolves all pending yields keyed by `request` with the serialized `response`: [2](#0-1) 

There is **no check** that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the specific `request` submitted. The hash is accepted from the response as-is.

Contrast with `respond()` for regular sign requests, which verifies the signature against a **derived key** computed from `request.tweak` (encoding the caller's account ID and path), cryptographically binding the response to the specific request: [3](#0-2) 

The `VerifyForeignTransactionRequest` struct contains no caller account ID — only `request`, `domain_id`, and `payload_version`: [4](#0-3) 

This is by design (the fan-out feature allows different callers to share a response), but it means the pending-request map is keyed purely on foreign-transaction parameters, so any attested participant can resolve any pending request with any valid `(payload_hash, signature)` pair.

The node-side signing uses a **zero tweak**, confirming the root key is used with no request binding: [5](#0-4) 

**Attack path:**

1. Legitimate request A (`verify_foreign_transaction(bitcoin_tx_A)`) is processed; the MPC network produces `(payload_hash_A, sig_A)` and `respond_verify_foreign_tx(request=A, response=(payload_hash_A, sig_A))` is submitted on-chain (publicly observable).
2. A bridge contract submits new request B (`verify_foreign_transaction(bitcoin_tx_B)`).
3. The malicious attested participant calls `respond_verify_foreign_tx(request=B, response=(payload_hash_A, sig_A))`.
4. The contract checks: is `sig_A` valid for `payload_hash_A` under the root key? **Yes** — passes.
5. The contract resolves request B's yield with `(payload_hash_A, sig_A)`.
6. The bridge contract receives a valid MPC signature attesting to `bitcoin_tx_A` as the response to its request for `bitcoin_tx_B`.

The existing test explicitly confirms that any attested participant can submit a response for any pending request without restriction: [6](#0-5) 

### Impact Explanation

A bridge contract (e.g., Omnibridge inbound flow) that calls `verify_foreign_transaction` to gate fund releases receives a valid MPC-signed attestation for the wrong foreign transaction. If the bridge contract does not independently recompute and verify `payload_hash` against the expected transaction — a step the MPC contract's design does not enforce or document as required — it will accept the forged attestation and release funds for a transaction that was never verified. This is **forged foreign-chain verification causing invalid bridge execution**, matching the High-severity allowed impact.

### Likelihood Explanation

The attack requires a single malicious attested MPC participant (Byzantine node strictly below the signing threshold). The attacker needs only to observe a previously submitted legitimate `respond_verify_foreign_tx` call (all on-chain transactions are public) and submit a new `verify_foreign_transaction` request to target. No threshold collusion, no key material beyond being an attested participant, and no privileged access is required. The `assert_caller_is_attested_participant_and_protocol_active` check is the only gate: [7](#0-6) 

### Recommendation

Bind the response to the request. Two options:

1. **Include extracted values in the response**: Have MPC nodes also submit the `extracted_values` alongside `payload_hash`, allowing the contract to recompute `SHA-256(borsh(ForeignTxSignPayload { request, values }))` and verify it equals `response.payload_hash`.
2. **Include a request-specific nonce in the signed payload**: Incorporate the on-chain receipt ID or yield data ID into the signed message so that a signature produced for one request cannot be replayed for another.

### Proof of Concept

```rust
// Step 1: Observe legitimate response for request A on-chain (public)
// payload_hash_A = SHA256(borsh(ForeignTxSignPayload { request: bitcoin_tx_A, values: [block_hash_A] }))
// sig_A = MPC_root_key.sign(payload_hash_A)

// Step 2: Bridge contract submits new request B
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: bitcoin_tx_B,  // different transaction
        confirmations: 2.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
});

// Step 3: Malicious attested node replays response A against pending request B
contract.respond_verify_foreign_tx(
    // request key points to B's pending yield
    request = VerifyForeignTransactionRequest { request: bitcoin_tx_B, domain_id, payload_version },
    response = VerifyForeignTransactionResponse {
        payload_hash: payload_hash_A,  // hash of bitcoin_tx_A, not bitcoin_tx_B
        signature: sig_A,              // valid MPC signature for payload_hash_A
    }
);
// Contract: sig_A valid for payload_hash_A under root key? YES → resolves B's yield
// Bridge contract receives attestation for bitcoin_tx_A instead of bitcoin_tx_B
// → invalid bridge execution if bridge does not recompute and verify payload_hash
```

```rust
                );
```
