Audit Report

## Title
Division by Zero in `getRsETHAmountToMint` When `rsETHPrice` Is Uninitialized ‚Äî (`contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle.rsETHPrice` is a `uint256` storage variable that defaults to `0` on deployment and is only assigned a non-zero value after `updateRSETHPrice()` is explicitly called. `LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()` with no zero-guard, causing an EVM division-by-zero panic revert on any deposit attempt made before that initialization call. All deposit entry points (`depositETH`, `depositAsset`) are blocked until the price is initialized.

## Finding Description

`LRTOracle` declares `rsETHPrice` as a plain `uint256` with no initialization in `initialize()`:

```solidity
// contracts/LRTOracle.sol line 28
uint256 public override rsETHPrice;
```

`LRTOracle.initialize()` (lines 64‚Äì68) only sets `lrtConfig` and emits an event ‚Äî it never calls `_updateRsETHPrice()`. The only paths that assign a non-zero value to `rsETHPrice` are inside `_updateRsETHPrice()`: `rsETHPrice = 1 ether` when `rsethSupply == 0` (line 219), or `rsETHPrice = newRsETHPrice` (line 313). Both paths are reachable only via `updateRSETHPrice()` (public, `whenNotPaused`, line 87) or `updateRSETHPriceAsManager()` (manager-only, line 94) ‚Äî neither is called automatically.

`LRTDepositPool.getRsETHAmountToMint()` performs an unchecked division:

```solidity
// contracts/LRTDepositPool.sol line 520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This is called unconditionally by `_beforeDeposit()` (line 665), which is called by both `depositETH()` (line 87) and `depositAsset()` (line 111). No caller checks whether `rsETHPrice` is zero before the division. When `rsETHPrice == 0`, the EVM triggers a division-by-zero panic (Solidity 0.8.x), reverting the transaction.

## Impact Explanation

Any call to `depositETH()` or `depositAsset()` before `updateRSETHPrice()` has been executed reverts unconditionally. All deposit functionality is frozen for every user during this window. This matches **Medium ‚Äî Temporary freezing of funds**: deposits are completely blocked, though the condition is self-correctable once any account calls the public `updateRSETHPrice()`.

## Likelihood Explanation

The vulnerable window opens at deployment and closes on the first successful `updateRSETHPrice()` call. Because `updateRSETHPrice()` is public and callable by anyone, the window is short in practice ‚Äî but the code provides no enforcement that it is called before deposits are opened. A fresh deployment or a redeployment without a scripted initialization sequence leaves the protocol in this state. No privileged access is required to trigger the revert; any unprivileged user attempting a deposit will hit it.

## Recommendation

Add a zero-guard in `getRsETHAmountToMint()` before the division:

```solidity
uint256 currentRsETHPrice = lrtOracle.rsETHPrice();
if (currentRsETHPrice == 0) revert RsETHPriceNotInitialized();
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / currentRsETHPrice;
```

Alternatively, call `_updateRsETHPrice()` (or enforce `updateRSETHPrice()`) as part of `LRTOracle.initialize()`, guaranteeing `rsETHPrice` is non-zero before any deposit can be attempted.

## Proof of Concept

1. Deploy `LRTOracle` and `LRTDepositPool` on a local fork. Confirm `LRTOracle.rsETHPrice() == 0`.
2. Do **not** call `updateRSETHPrice()`.
3. As any EOA, call `depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `_beforeDeposit()` ‚Üí `getRsETHAmountToMint()` ‚Üí line 520: `/ lrtOracle.rsETHPrice()` ‚Üí `/ 0` ‚Üí EVM panic revert (error code `0x12`).
5. Repeat for `depositAsset()` ‚Äî same result.
6. Call `updateRSETHPrice()` from any account. Confirm `rsETHPrice == 1 ether` (since `rsethSupply == 0`).
7. Retry `depositETH` ‚Äî succeeds. Confirms the freeze is temporary and self-correctable.
