### Title
NFT Royalty Silently Truncates to Zero on Small Offers, Enabling Royalty Bypass in Offer Settlement - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
`compute_royalty_amount` uses sequential integer floor division that can silently produce zero when the offered fungible amount is small. Because the offer-building code in `make_nft1_offer` skips the royalty coin spend entirely when the computed royalty sum is zero, an NFT owner can craft a dust-amount offer, accept it from a second wallet they control, and transfer the NFT with zero royalty paid to the creator — a complete royalty bypass through the offer settlement path.

### Finding Description

`compute_royalty_amount` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

With `MAX_ROYALTY_BASIS_POINTS = 10000`, any offered amount where `abs(offered_amount) * percentage < 10000` truncates to zero. The project's own unit test explicitly documents this:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

In `make_nft1_offer`, the zero-royalty result is stored in `royalty_payments` as a `CreateCoin` with `amount=0`: [3](#0-2) 

The announcement-assertion loop then filters out all zero-amount payments, so no royalty announcement is committed to the spend bundle: [4](#0-3) 

Finally, the royalty coin spend is skipped entirely when the total royalty sum is zero: [5](#0-4) 

The same truncation exists in the `royalty_calculation` static method used by the RPC endpoint: [6](#0-5) 

### Impact Explanation

An NFT owner can bypass the royalty obligation encoded in the NFT's transfer program by constructing an offer where the fungible side (XCH or CAT) is small enough that `compute_royalty_amount` returns zero. The offer is valid on-chain because no royalty announcement is asserted and no royalty coin is created. The NFT creator receives zero royalty despite having set a non-zero `royalty_percentage` at mint time. This constitutes unauthorized reward diversion and corrupted offer settlement affecting NFT-controlled assets.

### Likelihood Explanation

Any NFT owner can execute this unilaterally: create an offer offering 1–9999 mojos of XCH for their own NFT (held in a second wallet), accept it, and the NFT transfers with zero royalty. No cooperation from a third party is required. The threshold amount below which royalties truncate to zero scales with the royalty percentage: for a 1% royalty (100 basis points), any offered amount below 10,000 mojos (< 0.00000001 XCH) produces zero royalty.

### Recommendation

1. Add a minimum-amount guard in `compute_royalty_amount`: if the computed royalty is zero but `percentage > 0`, either raise a `ValueError` or return a minimum of 1 mojo.
2. In `make_nft1_offer`, reject or warn when the offered fungible amount is so small that all royalties truncate to zero while the NFT has a non-zero royalty percentage.
3. Alternatively, enforce a minimum fungible offer amount proportional to `MAX_ROYALTY_BASIS_POINTS / percentage` to guarantee at least 1 mojo of royalty.

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
