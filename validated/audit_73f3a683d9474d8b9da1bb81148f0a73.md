### Title
Missing Batch-Level Deposit Refund Validation Allows Double-Claiming Across Sub-Transactions - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces sub-transactions (`subTransactionsTxBodyL`). The UTXO rule validates batch-level withdrawals via `validateBatchWithdrawals`, which aggregates all withdrawal amounts across the top-level transaction and all sub-transactions and checks them against the original account balances. However, **no analogous batch-level check exists for deposit refunds**. The `getConsumedDijkstraValue` function sums refund amounts from every sub-transaction independently against the original (pre-batch) cert state, allowing multiple sub-transactions to each claim the same deposit refund. The developers have explicitly acknowledged this gap with a disabled test (`xit`) whose expected predicate failure is marked `error "TODO: predicate failure not yet implemented"`.

---

### Finding Description

**Root cause — `getConsumedDijkstraValue` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`:**

```haskell
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
``` [1](#0-0) 

Each sub-transaction's consumed value is computed by calling `getConsumedMaryValue` with the **same** `lookupStakingDeposit` closure derived from the **original** cert state. If two sub-transactions both contain `UnRegDepositTxCert stakingCred D`, each independently resolves the deposit `D` from the original state, and the total consumed value includes `2D` in refunds.

**Contrast with the withdrawal check in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`:**

```haskell
runTest $ validateBatchWithdrawals accounts tx
``` [2](#0-1) 

`validateBatchWithdrawals` explicitly aggregates all withdrawal amounts across the batch and rejects any that exceed the original account balance:

```haskell
let allWithdrawals =
      Map.unionsWith (<>) $
        unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
          : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
            | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
            ]
``` [3](#0-2) 

No equivalent `validateBatchRefunds` function exists. The `validateValueNotConservedUTxO` call uses the original cert state:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [4](#0-3) 

**Developer acknowledgment — `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`:**

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  stakingCred <- KeyHashObj <$> freshKeyHash
  _ <- registerStakeCredential stakingCred
  keyDeposit <- getsPParams ppKeyDepositL
  ...
  let
    subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
    tx = ... & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
``` [5](#0-4) 

The test is disabled (`xit`) because the rejection predicate does not yet exist. The `error "TODO: predicate failure not yet implemented"` is the placeholder for the missing failure constructor.

A second disabled test covers the related case where the deposit amount changes between registration and the sub-transaction refund: [6](#0-5) 

**`dijkstraSubTxProducedValue` does not include refunds on the produced side**, confirming that refunds are only accounted for on the consumed side:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody <> txBody ^. treasuryDonationTxBodyL)
    <> burnedMultiAssets txBody
``` [7](#0-6)

### Impact Explanation

The `validateValueNotConservedUTxO` balance check passes when two sub-transactions each claim refund `D` for the same credential, because the consumed side totals `2D` in refunds while the attacker supplies sufficient UTxO inputs to balance the equation. Whether the deposit pot is actually over-drained depends on whether the SUBCERTS/CERTS rules process sub-transactions against the original cert state or sequentially:

- **If sub-transactions are validated against the original cert state** (consistent with the comment "this is the original Accounts, before any transactions were applied" at line 347–348 of `Utxo.hs`): both `UnRegDepositTxCert` certificates succeed, the deposit pot is reduced by `2D`, and the attacker extracts `D` ADA that was never deposited. This is a **Critical** direct loss of ADA through an invalid ledger state transition.
- **If sub-transactions are processed sequentially with state carry-over**: the second sub-transaction fails with `StakeKeyNotRegisteredDELEG`, the transaction is rejected, and no ADA is lost. The impact is then **Medium** — the balance check incorrectly accepts a transaction body with inflated refund amounts, constituting a validation logic error that modifies the effective deposit accounting outside design parameters.

In either case the missing check is a reachable, attacker-controlled entry path requiring no privileged access: any user who has registered a stake credential can construct such a batch transaction. [8](#0-7) 

---

### Likelihood Explanation

- The Dijkstra era is production-targeted code in the repository scope.
- The attack requires only a registered stake credential (no governance keys, no privileged access).
- The developers have explicitly identified the scenario and left it unguarded (`xit` + `error "TODO"`).
- The analogous withdrawal check (`validateBatchWithdrawals`) demonstrates the correct pattern; its absence for refunds is a structural omission, not an intentional design choice.

---

### Recommendation

Add a `validateBatchRefunds` function mirroring `validateBatchWithdrawals`. It should:

1. Aggregate all `UnRegDepositTxCert` and `UnRegDRepTxCert` refund claims across the top-level transaction and every sub-transaction.
2. For each credential, verify that the total claimed refund does not exceed the deposit recorded in the original cert state.
3. Reject the transaction with a new predicate failure (e.g., `RefundsExceedDepositBalance`) if any credential's total claimed refund exceeds its stored deposit.

This check should be inserted in `dijkstraUtxoTransition` immediately after `validateBatchWithdrawals`, using the same `accounts`/`certState` snapshot that is already available at that point. [9](#0-8)

### Proof of Concept

```
Setup:
  stakingCred registered with deposit D = ppKeyDepositL

Attack transaction:
  TopTx {
    subTransactions: [
      SubTx1 { certs: [UnRegDepositTxCert stakingCred D], inputs: [utxo1] }
      SubTx2 { certs: [UnRegDepositTxCert stakingCred D], inputs: [utxo2] }
    ]
  }

validateValueNotConservedUTxO evaluation (using original certState):
  consumed = utxo1 + utxo2 + D (subTx1 refund) + D (subTx2 refund)
           = utxo1 + utxo2 + 2D
  produced = outputs + fees
  → balance equation satisfied if utxo1 + utxo2 = outputs + fees - 2D

Result:
  validateValueNotConservedUTxO passes.
  If SUBCERTS processes against original state: deposit pot -= 2D, attacker nets D ADA.
  If SUBCERTS processes sequentially: second cert fails, tx rejected — but balance check
  was already incorrectly satisfied with 2D in refunds.
```

The disabled test at `CertSpec.hs:53–75` is the developers' own reproduction of this exact scenario, left unresolved pending implementation of the missing predicate failure. [5](#0-4)

dijkstraUtxoTransition = do
  TRC (DijkstraUtxoEnv slot pp certState originalUtxo, utxos, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG
  -- this is the original Accounts, before any transactions were applied
  let accounts = certState ^. certDStateL . accountsL

  let txBody = tx ^. bodyTxL

  {- inInterval (SlotOf Γ) (ValidIntervalOf txTop) -}
  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo

  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  {- SpendInputs ≠ ∅ -}
  runTestOnSignal $ Shelley.validateInputSetEmptyUTxO txBody

  let allInputs = txBody ^. allInputsTxBodyF
      inputs = txBody ^. inputsTxBodyL

  {- SpendInputsOf txTop ∪ RefInputsOf txTop ∪ CollInputsOf txTop ⊆ dom(utxo₀) -}
  runTest $ Shelley.validateBadInputsUTxO originalUtxo allInputs

  validate $ validateBatchCollateral pp tx originalUtxo
