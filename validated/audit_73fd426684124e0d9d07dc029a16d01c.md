### Title
Single Attested Participant Can Forge CKD Responses for `AppPublicKey` Requests, Bypassing Threshold Requirement - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` in the MPC contract performs no cryptographic verification of the `CKDResponse` when the request uses the `AppPublicKey` variant. A single Byzantine attested participant can call `respond_ckd` with an entirely fabricated `(big_y, big_c)` pair for any pending `AppPublicKey` CKD request, bypassing the threshold requirement and delivering a malicious key to the user.

### Finding Description

The `respond_ckd` function applies a cryptographic pairing check only for the `AppPublicKeyPV` variant:

```rust
// crates/contract/src/lib.rs lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk, app_id), mpc_pk)`, which proves the response was computed using the MPC network's master secret key and required threshold participation. [2](#0-1) 

For `AppPublicKey`, there is no equivalent check. The contract unconditionally accepts any `(big_y, big_c)` pair submitted by any single attested participant and resolves all pending yields with it.

The `AppPublicKey` variant is the default, non-publicly-verifiable variant used in production (e.g., the e2e test `ckd_response__passes_cryptographic_verification` uses it). [3](#0-2) 

The off-chain node protocol does require threshold participation to compute the CKD output: [4](#0-3) 

But the on-chain contract enforces no such requirement for `AppPublicKey` requests. A malicious participant can skip the off-chain protocol entirely and call `respond_ckd` directly on-chain with fabricated values.

### Impact Explanation

The CKD protocol computes `big_c = msk·H(pk, app_id) + y·A` where `A = a·G1` is the user's app public key. The user recovers the key as `sig = big_c − a·big_y = msk·H(pk, app_id)`.

If an attacker submits `big_y = G1_identity` (the identity point) and `big_c = attacker_chosen_point`, the user computes `sig = attacker_chosen_point − a·0 = attacker_chosen_point`. The attacker chose this point, so they know the resulting `sig` and can compute `s = HKDF(sig)` — the user's derived secret key. This is a complete key disclosure: the attacker learns the user's application-specific private key.

Additionally, the forged response is fanned out to **all** callers who queued under the same request key (the duplicate-request fan-out mechanism), so a single forged `respond_ckd` call compromises every user who submitted the same CKD request. [5](#0-4) 

This maps to the allowed critical impact: **bypass of threshold-signature requirements and unauthorized access to secret material that materially enables secret recovery**.

### Likelihood Explanation

The attacker must be a single attested participant — a Byzantine participant strictly below the signing threshold. This is a realistic adversary model explicitly in scope. The attack requires only one on-chain transaction (`respond_ckd`) with crafted arguments; no off-chain coordination or key-share access is needed. The `AppPublicKey` variant is the default and most commonly used CKD variant, so pending requests of this type will be present in production.

### Recommendation

Apply the same pairing-based output check to `AppPublicKey` requests. Since `AppPublicKey` provides only `a·G1` (not `a·G2`), the existing `ckd_output_check` cannot be used directly. Options:

1. **Require `AppPublicKeyPV` for all production CKD requests** and deprecate the unverifiable `AppPublicKey` variant.
2. **Add a threshold-count check**: require that `respond_ckd` can only be called after a quorum of participants have submitted matching `(big_y, big_c)` values off-chain, with the contract only accepting the response once threshold agreement is reached on-chain.
3. **Introduce a BLS signature over the response**: have the MPC network produce a BLS aggregate signature over `(app_id, big_y, big_c)` that the contract can verify against the stored BLS public key, proving threshold participation.

### Proof of Concept

1. User Alice submits `request_app_private_key` with `AppPublicKey(a·G1)` for derivation path `"mykey"`.
2. The request is stored in `pending_ckd_requests`.
3. Attacker (a single attested participant) calls `respond_ckd` with:
   - `request` = Alice's pending `CKDRequest`
   - `response.big_y` = BLS12-381 G1 identity point (48 bytes of the compressed identity encoding)
   - `response.big_c` = any attacker-chosen G1 point `P`
4. The contract's `match` arm for `AppPublicKey` executes the empty branch `{}` — no check is performed.
5. `resolve_yields_for` resolves Alice's yield with the forged response.
6. Alice receives `(big_y=0, big_c=P)` and computes `sig = P − a·0 = P`.
7. Alice derives `s = HKDF(P)`. The attacker, who chose `P`, also computes `s = HKDF(P)` and now knows Alice's application-specific private key.

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates that an arbitrary `CKDResponse` with dummy byte values `([1u8; 48], [2u8; 48])` is accepted without error for the `AppPublicKey` variant, confirming the absence of any output validation.     #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
