### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Submitted `request`, Enabling Cross-Request Payload Substitution - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function in the MPC contract verifies that the submitted signature is cryptographically valid over `response.payload_hash` using the root public key, but it never verifies that `response.payload_hash` is actually derived from the specific `request` that is being resolved. A single malicious attested participant (strictly below the signing threshold) can call `respond_verify_foreign_tx` for a pending request B by supplying a `payload_hash` and signature that were legitimately produced by the MPC network for a completely different request A. The contract accepts this, drains request B's pending yields, and delivers the forged response to all of request B's callers.

---

### Finding Description

In `respond_verify_foreign_tx`, the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract only checks two things: (1) the signature is a valid ECDSA signature over `response.payload_hash` under the root key, and (2) the `request` key exists in `pending_verify_foreign_tx_requests`. It does **not** verify that `response.payload_hash` is the canonical hash of `SHA-256(borsh(ForeignTxSignPayload{request, values}))` for the specific `request` being resolved. [2](#0-1) 

Contrast this with the regular `respond` function, where the payload is fixed at submission time inside `SignatureRequest` and the signature is verified against that exact stored payload with a caller-specific tweak:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [3](#0-2) 

For `verify_foreign_transaction`, there is no `tweak` field in `VerifyForeignTransactionRequest` (unlike `SignatureRequest`), and the signature is over the root key with no caller-specific binding: [4](#0-3) 

The `payload_hash` is defined as `SHA-256(borsh(ForeignTxSignPayload{request, values}))` where `values` are the extracted foreign-chain observations (block hashes, log entries, etc.) that only the MPC nodes observe: [5](#0-4) 

Because the contract cannot recompute `payload_hash` from `request` alone (it does not have `values`), it cannot enforce the binding. The design relies entirely on the participant being honest.

---

### Impact Explanation

**High — Forged foreign-chain verification causing invalid bridge execution.**

A single malicious attested participant (the leader, or any participant who observes a completed signing round) can:

1. Participate honestly in signing request A (e.g., verifying Bitcoin tx_id A), obtaining a valid `(payload_hash_A, signature_A)` pair signed by the MPC network.
2. Observe that request B (e.g., for a different Bitcoin tx_id B) is pending in `pending_verify_foreign_tx_requests`.
3. Call `respond_verify_foreign_tx(request_B, {payload_hash_A, signature_A})`.
4. The contract passes both checks (valid signature over `payload_hash_A`; `request_B` exists), drains request B's yields, and delivers `{payload_hash_A, signature_A}` to all of request B's callers.

Request B's callers (e.g., an Omnibridge contract) receive a response that carries a valid MPC threshold signature, but the `payload_hash` encodes tx_id A's data, not tx_id B's. Bridge contracts that only verify the signature validity (which is the natural check — the bridge does not know the extracted values a priori and cannot recompute `expected_payload_hash`) will accept this as a valid attestation of tx_id B. This enables invalid bridge execution: a bridge operation keyed to tx_id B is unlocked by an attestation that actually covers tx_id A.

Additionally, request B's pending yields are permanently drained. Even if the bridge detects the mismatch, it cannot retry — the request is gone and must be resubmitted with a new deposit.

The `ForeignChainSignatureVerifier` in the SDK does check `expected_payload_hash == response.payload_hash`, but it requires the caller to supply `expected_extracted_values`: [6](#0-5) 

Bridge contracts that do not know the expected extracted values in advance (the common case for "did this tx finalize?" queries) cannot use this check and are therefore vulnerable.

---

### Likelihood Explanation

**Medium.** The attacker must be a single attested MPC participant — below the signing threshold. This is explicitly within scope as a "Byzantine participant strictly below the signing threshold." The participant needs to:

- Be an active attested participant (required to call `respond_verify_foreign_tx`).
- Have access to a `(payload_hash, signature)` pair produced by the MPC network for any prior request on the same domain.
- Observe a pending request in `pending_verify_foreign_tx_requests` to target.

All three conditions are routinely satisfied by any active node. The attack requires no collusion, no key leakage, and no network-level interference. The only constraint is that the attacker must act before the honest leader submits the legitimate response for request B.

---

### Recommendation

The contract must bind `response.payload_hash` to the submitted `request`. Since the contract cannot recompute the full `payload_hash` (it lacks `values`), the recommended fix is to include a commitment to the `request` in the signed payload that the contract *can* verify independently. For example:

- Change the signing payload to `SHA-256(borsh(request) || SHA-256(borsh(values)))` and have the contract verify the first component against the stored `request`.
- Or include a `request_hash = SHA-256(borsh(request))` field in `VerifyForeignTransactionResponse` and have the contract verify `request_hash == SHA-256(borsh(request))` before accepting the response.
- Alternatively, add a `tweak` derived from the request (analogous to how `SignatureRequest` uses a caller/path tweak) so that a signature produced for request A is cryptographically invalid for request B.

---

### Proof of Concept

```
1. Attacker (attested participant P) observes two pending requests:
   - request_A = {chain: Bitcoin, tx_id: [0xAA; 32], ...}  (pending)
   - request_B = {chain: Bitcoin, tx_id: [0xBB; 32], ...}  (pending, target)

2. P participates honestly in the MPC signing round for request_A.
   The MPC network produces:
     payload_hash_A = SHA-256(borsh(ForeignTxSignPayload{request_A, values_A}))
     signature_A    = ECDSA_sign(root_key, payload_hash_A)

3. Before the honest leader submits respond_verify_foreign_tx(request_A, ...),
   P calls:
     respond_verify_foreign_tx(
       request  = request_B,          // ← different request
       response = {
         payload_hash: payload_hash_A, // ← hash from request_A
         signature:    signature_A,    // ← signature from request_A
       }
     )

4. Contract checks:
   a. verify_ecdsa_signature(signature_A, payload_hash_A, root_pk) → OK
   b. pending_verify_foreign_tx_requests.get(request_B) → exists
   → Contract resolves request_B's yields with {payload_hash_A, signature_A}.

5. Bridge contract for request_B receives {payload_hash_A, signature_A}.
   It verifies: verify_ecdsa_signature(signature_A, payload_hash_A, root_pk) → OK.
   It cannot verify that payload_hash_A encodes tx_id [0xBB;32] (it doesn't know values_A).
   → Bridge accepts the forged attestation and executes the bridge operation for tx_id B.
``` [7](#0-6) [8](#0-7)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
