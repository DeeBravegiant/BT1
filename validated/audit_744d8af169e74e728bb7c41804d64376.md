Audit Report

## Title
Rebasing stETH Token Amounts Stored as Fixed Values in Withdrawal Queue Lead to Permanent Fund Freeze - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTWithdrawalManager` stores fixed stETH token amounts in `request.expectedAssetAmount` after `unlockQueue()` transfers stETH from the vault and burns rsETH. Because stETH is a rebasing token, a negative rebase between `unlockQueue()` and `completeWithdrawal()` reduces the contract's stETH balance below the sum of committed amounts. Users who call `completeWithdrawal()` last receive nothing, and because their rsETH was already burned in `unlockQueue()`, their funds are permanently unrecoverable without manual admin intervention.

## Finding Description

**Stage 1 – `initiateWithdrawal()`**: A fixed `expectedAssetAmount` is computed from oracle prices and stored in `withdrawalRequests`, and `assetsCommitted[asset]` is incremented by the same fixed token amount. [1](#0-0) 

**Stage 2 – `unlockQueue()`**: `_createUnlockParams` reads the vault's stETH balance via `unstakingVault.balanceOf(asset)`, which calls `IERC20(asset).balanceOf(address(this))` — a rebasing token amount, not shares. [2](#0-1) 

`_unlockWithdrawalRequests` overwrites each request's `expectedAssetAmount` with a fixed payout amount, then `unstakingVault.redeem(asset, assetAmountUnlocked)` transfers the exact stETH token amount into `LRTWithdrawalManager`. Critically, rsETH is burned at this point — before any user calls `completeWithdrawal()`. [3](#0-2) 

The per-request `expectedAssetAmount` is overwritten with the fixed payout: [4](#0-3) 

**Stage 3 – `completeWithdrawal()`**: The contract transfers exactly `request.expectedAssetAmount` stETH using a plain `IERC20.safeTransfer` with no share-based accounting: [5](#0-4) [6](#0-5) 

**Root cause**: After Stage 2, `LRTWithdrawalManager` holds a fixed stETH token balance equal to the sum of all unlocked `expectedAssetAmount` values. A negative rebase (e.g., validator slashing) autonomously reduces the contract's stETH balance without any transfer. The contract has no mechanism to detect or handle this discrepancy. The first users to call `completeWithdrawal()` drain the available balance; subsequent users' calls revert because `safeTransfer` fails. Since rsETH was already burned in `unlockQueue()`, there is no on-chain path for affected users to recover their funds.

The `sweepRemainingAssets` function cannot be used to recover the situation because it requires `hasUnlockedWithdrawals(asset) == false`, but `unlockedWithdrawalsCount[asset]` is only decremented inside `_processWithdrawalCompletion` — which never completes for the blocked users. [7](#0-6) 

A secondary issue exists between Stage 1 and Stage 2: `assetsCommitted[asset]` tracks token amounts while `getAvailableAssetAmount` computes `totalAssets - assetsCommitted[asset]` using the live rebasing balance. A negative rebase causes `assetsCommitted` to exceed `totalAssets`, making `getAvailableAssetAmount()` return 0 and blocking all new withdrawal initiations until the operator processes the queue. [8](#0-7) 

The protocol's own code comment in `initiateWithdrawal` acknowledges this edge case ("There is an edge case were the user withdraws last underlying asset and that asset gets slashed") but provides no mitigation. [9](#0-8) 

## Impact Explanation

**Critical – Permanent freezing of funds**: Users whose `completeWithdrawal()` calls revert after a negative rebase have their rsETH permanently burned (at `unlockQueue()` time) with no on-chain recovery path. The loss is concentrated on the last users to claim rather than being shared proportionally. Without manual admin intervention (directly sending stETH to the contract), affected users permanently lose their funds. This matches the "Permanent freezing of funds" critical impact class.

## Likelihood Explanation

stETH is an explicitly supported and actively used asset, confirmed by the `ST_ETH_TOKEN` constant, the `initialize2` function seeding `unlockedWithdrawalsCount` for stETH, and dedicated stETH unstaking logic in `LRTConverter`. [10](#0-9) [11](#0-10) 

The withdrawal delay is 8 days, creating a substantial window during which stETH sits in `LRTUnstakingVault` with `assetsCommitted` tracking a fixed amount. After unlock, stETH sits in `LRTWithdrawalManager` until each user individually calls `completeWithdrawal()`. Ethereum validator slashing events are realistic and have occurred historically. Any slashing event during either window triggers the vulnerability. No attacker action is required — the trigger is a natural blockchain event.

## Recommendation

Store and transfer stETH in **shares** rather than token amounts throughout the withdrawal lifecycle:

1. In `initiateWithdrawal()`, convert `expectedAssetAmount` to stETH shares using `IStETH(stETH).getSharesByPooledEth(expectedAssetAmount)` and store the shares value.
2. In `_unlockWithdrawalRequests()`, compute `payoutAmount` in shares and store shares in `request.expectedAssetAmount`.
3. In `_processWithdrawalCompletion()`, use `IStETH(stETH).transferShares(user, sharesAmount)` instead of `IERC20.safeTransfer`.
4. Update `assetsCommitted[stETH]` to track shares, and convert to token amounts only when computing `getAvailableAssetAmount()` using `IStETH.getPooledEthByShares()`.

This ensures all accounting is rebase-invariant and each user receives their proportional share of the stETH balance regardless of rebasing events between initiation and claim.

## Proof of Concept

1. Alice and Bob each call `initiateWithdrawal(stETH, ...)`. Their `expectedAssetAmount` values (e.g., 90 stETH and 10 stETH) are stored as fixed token amounts; `assetsCommitted[stETH] = 100e18`.
2. The operator calls `unlockQueue(stETH, ...)` after the 8-day delay. The vault's 100 stETH is transferred to `LRTWithdrawalManager` via `unstakingVault.redeem`. Both requests are unlocked with `expectedAssetAmount` = 90e18 and 10e18 respectively. rsETH is burned for both users at this step.
3. A validator slashing event causes a 5% negative rebase. `LRTWithdrawalManager` now holds 95 stETH (was 100).
4. Alice calls `completeWithdrawal()` and receives 90 stETH — her full amount. Contract now holds 5 stETH.
5. Bob calls `completeWithdrawal()` — the `safeTransfer` of 10 stETH reverts because the contract only holds 5 stETH. Bob's rsETH is already burned and unrecoverable on-chain.
6. `unlockedWithdrawalsCount[stETH]` remains 1 (Bob's request was never completed), so `sweepRemainingAssets` cannot be called. The 5 stETH remaining in the contract is inaccessible without admin intervention.
