Audit Report

## Title
Stale Cross-Chain Rate Used for wrsETH Minting With No Staleness Guard - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally, with no check against `lastUpdated`. All L2 pool contracts use this rate as the sole pricing input for wrsETH minting. Because rsETH/ETH accrues staking rewards monotonically, any stale snapshot is always lower than the true rate, causing depositors to receive excess wrsETH proportional to the staleness gap ‚Äî constituting theft of unclaimed yield.

## Finding Description
`CrossChainRateReceiver` stores the rsETH/ETH exchange rate in `rate` and records the update time in `lastUpdated`. The `getRate()` view function returns `rate` with no staleness check:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L102-105
function getRate() external view returns (uint256) {
    return rate;
}
```

`lastUpdated` is written on every `lzReceive()` call but is never read inside `getRate()`. [1](#0-0) 

Every L2 pool contract resolves the minting rate through `IOracle(rsETHOracle).getRate()`, which delegates to `CrossChainRateReceiver.getRate()`:

- `RSETHPoolV3ExternalBridge.viewSwapRsETHAmountAndFee()` at L423: `uint256 rsETHToETHrate = getRate();` ‚Üí `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;` [2](#0-1) 

- `RSETHPoolV3WithNativeChainBridge.viewSwapRsETHAmountAndFee()` at L340 does the same. [3](#0-2) 

The pool's `getRate()` simply proxies to the oracle: [4](#0-3) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` enforces `answeredInRound < roundID` and `timestamp == 0` guards, demonstrating the protocol is aware of staleness risks for other oracle types but omitted the same protection for the cross-chain rate. [5](#0-4) 

The exploit path:
1. `rate` is last updated at time T with value `R_stale`.
2. True rate rises to `R_true > R_stale` as staking rewards accrue; no keeper calls `updateRate()`.
3. Attacker calls `deposit{value: 1 ether}()` on any L2 pool.
4. `viewSwapRsETHAmountAndFee` calls `getRate()` ‚Üí returns `R_stale`.
5. `rsETHAmount = amountAfterFee * 1e18 / R_stale` ‚Äî larger than fair value because `R_stale < R_true`.
6. Attacker receives excess wrsETH, bridges back to L1, unwraps at the true rate, and extracts more ETH than deposited.

No privileged access is required. The `deposit()` function is fully public and permissionless. [6](#0-5) 

## Impact Explanation
**High ‚Äî Theft of unclaimed yield.** Every depositor during a stale window receives wrsETH whose face value at the true rate exceeds the ETH they supplied. When redeemed on L1 at the true rate, the surplus is extracted from the protocol's pre-minted rsETH reserve or dilutes other holders. Accumulated over many deposits during a prolonged stale window, this constitutes direct theft of unclaimed yield. The SECURITY.md exclusion for "incorrect data supplied by third-party oracles" does not apply here ‚Äî the data itself is correct when received; the vulnerability is the absence of an on-chain maximum-age guard in the protocol's own receiver contract.

## Likelihood Explanation
`updateRate()` on `RSETHMultiChainRateProvider` is permissionless but requires the caller to supply native ETH for LayerZero messaging fees. During high gas prices, network congestion, or keeper inactivity, updates can be delayed for hours or longer. The `lastUpdated` field confirms the design anticipates gaps between updates, yet no on-chain guard enforces a maximum gap. Any depositor can read `lastUpdated` on-chain and time a deposit to exploit the stale window. The attack is repeatable across all deployed L2 pool contracts simultaneously.

**Likelihood: Medium.**

## Recommendation
Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 24 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This mirrors the staleness-check pattern already used in `ChainlinkOracleForRSETHPoolCollateral`. [7](#0-6)
