### Title
Fee-on-Transfer Token Accounting Divergence Creates Unbacked Wrapped Supply and Fund Lock - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initTransfer` records and emits the caller-supplied `amount` rather than the actual tokens received by the contract. For fee-on-transfer ERC20 tokens, the bridge takes custody of `amount ŌłÆ fee` but credits the full `amount` cross-chain, creating unbacked wrapped supply on NEAR and an eventual irreversible fund lock for the last withdrawers on EVM.

### Finding Description

In `OmniBridge.initTransfer`, when the token is neither a bridge token nor a custom-minter token, the contract pulls tokens from the caller and then emits an event using the raw caller-supplied `amount`:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 406-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ŌåÉ requested amount, not actual received
    );
}
```

Immediately after, the event is emitted with the same unverified `amount`:

```solidity
// lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // ŌåÉ emitted as-is; no balance-before/after check
    fee,
    nativeFee,
    recipient,
    message
);
```

For a fee-on-transfer token (e.g., one that deducts 1 % on every transfer), `safeTransferFrom` deposits only `amount ├Ś 0.99` into the bridge, but the `InitTransfer` event records the full `amount`. The NEAR bridge's `fin_transfer_callback` reads the proof of this event and creates a `TransferMessage` crediting the full `amount` to the user. [1](#0-0) [2](#0-1) 

On the NEAR side, `fin_transfer_callback` denormalizes and stores the amount from the proof without any independent verification of what the EVM bridge actually received: [3](#0-2) 

When the user later bridges back from NEAR to EVM, `sign_transfer` signs a payload for the full credited amount, and `finTransfer` on EVM calls `safeTransfer(recipient, amount)`. Because the bridge holds only `amount ŌłÆ fee`, the last user(s) to redeem will find the contract insolvent. [4](#0-3) 

There is no token allowlist or compatibility gate in `initTransfer`; any ERC20 address is accepted in the else-branch. The `logMetadata` function similarly accepts any token address, so a fee-on-transfer token can be fully registered and used without any privileged action. [5](#0-4) 

### Impact Explanation

**High ŌĆö balance-accounting divergence that breaks backing guarantees and causes irreversible fund lock.**

Each `initTransfer` with a fee-on-transfer token inflates the credited supply on NEAR relative to the actual EVM custody. Over multiple transfers the deficit compounds. When redemptions exceed actual custody, the bridge cannot honour withdrawals, permanently locking the shortfall for the last redeemers. If the bridge also holds other tokens in the same contract, those are not at direct risk, but the accounting integrity of the affected token is permanently broken.

### Likelihood Explanation

**Medium.** Fee-on-transfer tokens (e.g., USDT on some chains, SafeMoon-style tokens, tokens with built-in tax mechanisms) are common in the wild. No privileged role is required: any user can call `logMetadata` to register such a token and then call `initTransfer`. The attacker does not need to exploit any key or oracle; the divergence accumulates passively with normal usage.

### Recommendation

1. **Measure actual received amount** using a balance-before/after pattern in `initTransfer`:
   ```solidity
   uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
   IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
   uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
   require(actualReceived == amount, "Fee-on-transfer token not supported");
   ```
   Emit `InitTransfer` with `actualReceived` instead of `amount`.

2. **Alternatively**, maintain a token allowlist and explicitly reject fee-on-transfer tokens during registration (`logMetadata` / `deployToken`).

3. Publish a Token Integration Checklist (as suggested in the external report) that explicitly flags fee-on-transfer, rebasing, and pausable tokens as incompatible with the bridge's lock-and-mint accounting model.

### Proof of Concept

1. Deploy or identify a fee-on-transfer ERC20 token `FoT` (1 % fee on every transfer).
2. Call `OmniBridge.logMetadata(FoT)` ŌĆö succeeds, emits `LogMetadata`; NEAR bridge registers `FoT` with its decimals.
3. Approve `OmniBridge` for 1 000 `FoT`.
4. Call `OmniBridge.initTransfer(FoT, 1000, 0, 0, "near:alice.near", "")`.
   - `safeTransferFrom` deposits **990** `FoT` into the bridge.
   - `InitTransfer` event is emitted with `amount = 1000`.
5. NEAR bridge processes the proof and credits **1 000** `FoT`-wrapped tokens to `alice.near`.
6. Alice bridges back: NEAR signs a transfer for **1 000**; EVM `finTransfer` calls `safeTransfer(alice_evm, 1000)`.
   - Bridge only holds **990** ŌåÆ transaction reverts.
   - Alice's funds are permanently locked; the 10-token deficit is unrecoverable.
