### Title
Stale `target_shard_index` from pre-resharding epoch used to index post-resharding `outcome_roots`, producing an invalid light-client proof — (`chain/client/src/view_client_actor.rs`)

---

### Summary

`GetExecutionOutcome` computes `target_shard_index` once from the **original** epoch's shard layout, then calls `get_next_block_hash_with_new_chunk`, which can silently return a **child** shard ID from a **new** epoch after resharding. The `outcome_roots` vector is then built from the new block (new shard layout), but the stale `target_shard_index` — valid only in the old layout — is used to select the Merkle sibling path. In V2/V3 layouts where shard ID ≠ shard index, this indexes the wrong element, producing an `outcome_root_proof` that fails `verify_path`.

---

### Finding Description

**Step 1 — `target_shard_index` is frozen to the old epoch.** [1](#0-0) 

`epoch_id` is the epoch of the block where the receipt was executed. `shard_layout` and `target_shard_index` are derived from that epoch and never updated.

**Step 2 — `get_next_block_hash_with_new_chunk` can cross an epoch boundary and return a different shard ID.** [2](#0-1) 

When the epoch changes and the shard layout changes, `shard_ids` is replaced with the children of the original shard. The function then returns `(block_hash, child_shard_id)` — a shard ID that exists only in the **new** layout.

**Step 3 — `outcome_roots` is built from the new block; the stale index is applied.** [3](#0-2) 

`outcome_roots` has one entry per chunk in block `h` (new epoch, new shard layout). `merklize(&outcome_roots).1[target_shard_index]` uses the old index, which points to the wrong position in the new layout.

**Why V2/V3 layouts make this concrete.** [4](#0-3) 

In V0/V1, shard ID equals shard index, so the old index accidentally stays valid. In V2/V3, `get_shard_index` does a map lookup; the child shard's index in the new layout is independent of the parent's index in the old layout.

**V2 concrete example** (from the test layout): [5](#0-4) 

Old layout (V1): shards `[0,1,2,3]`, `target_shard_id=1`, `target_shard_index=1`.  
New layout (V2): `shard_ids=[3,8,4,7]` — shard 7 at index 3, shard 8 at index 1.  
`get_next_block_hash_with_new_chunk` iterates children `[7,8]` and returns shard 7 (first child with a new chunk in the first post-resharding block).  
`outcome_roots[1]` = outcome root for shard 8 (index 1 in new layout).  
Correct target is shard 7 at index 3. The Merkle sibling path is for the wrong leaf → `verify_path` fails.

**V3 concrete example:** [6](#0-5) 

Parent shard `B` at index `k` is spliced into `[max+1, max+2]`. First child `max+1` lands at index `k` (same as parent), second child `max+2` at index `k+1`. If the first child's chunk is missing and the second child's chunk is present, `get_next_block_hash_with_new_chunk` returns `max+2`. The stale `target_shard_index=k` selects `outcome_roots[k]` = outcome root for `max+1`, not `max+2`. Proof is wrong.

---

### Impact Explanation

Any unprivileged RPC caller that requests `EXPERIMENTAL_tx_status` or `light_client_proof` for a receipt executed in the epoch immediately before a resharding event receives a structurally malformed `outcome_root_proof`. The Merkle path is for the wrong shard index. Any light client or bridge relayer that calls `verify_path(outcome_root, outcome_root_proof, chunk_outcome_root)` will reject the proof, making the receipt unprovable. For cross-chain bridges this is a liveness failure: locked assets cannot be released because the proof of execution cannot be verified.

---

### Likelihood Explanation

Resharding is a production protocol feature actively used on NEAR mainnet. The V2 case triggers on the **first block of the new epoch** (the common path, no missing chunks required) whenever the first child shard's index in the new layout differs from the parent's index in the old layout — which is the normal case for V2 layouts. The V3 case requires a missing chunk for the first child, which is less common but possible under normal validator churn. No special privileges are needed; any account can send a transaction, wait for it to execute near a resharding boundary, and then call the public RPC.

---

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, new_target_shard_id)`, recompute `target_shard_index` from the **new** block's epoch and the **new** shard ID:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    // Recompute index using the new epoch's layout

    let new_shard_layout = self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
```

```rust
    pub fn get_shard_index(&self, shard_id: ShardId) -> Result<ShardIndex, ShardLayoutError> {
        match self {
            // In V0 & V1 the shard id and shard index are the same.
            Self::V0(_) | Self::V1(_) => Ok(shard_id.into()),
            // In V2 & V3 the shard id and shard index are **not** the same.
            Self::V2(v2) => v2.get_shard_index(shard_id),
            Self::V3(v3) => v3.get_shard_index(shard_id),
        }
    }
```

```rust
    fn derive_impl(
        mut shard_ids: Vec<ShardId>,
        mut boundary_accounts: Vec<AccountId>,
        new_boundary_account: AccountId,
        mut shards_split_map: ShardsSplitMapV3,
    ) -> Result<Self, ShardLayoutError> {
        let Err(new_boundary_idx) = boundary_accounts.binary_search(&new_boundary_account) else {
            return Err(ShardLayoutError::DuplicateBoundaryAccount {
                account_id: new_boundary_account,
            });
        };
        boundary_accounts.insert(new_boundary_idx, new_boundary_account);

        let max_shard_id =
            *shard_ids.iter().max().expect("there should always be at least one shard");
        let new_shards = vec![max_shard_id + 1, max_shard_id + 2];

        let [last_split] = shard_ids
            .splice(new_boundary_idx..new_boundary_idx + 1, new_shards.clone())
            .collect_array()
            .expect("should only splice one shard");
        shards_split_map.insert(last_split, new_shards);

        Ok(Self::new(boundary_accounts, shard_ids, shards_split_map, last_split))
    }
```
