### Title
Stale `accept_requests` Flag and Non-Reverifying Attestation Check Allow Expired-TEE Nodes to Submit Threshold Signatures - (File: `crates/contract/src/lib.rs`)

### Summary

The `respond`, `respond_ckd`, and `respond_verify_foreign_tx` functions gate signature submission on two checks that both use stale/cached state: the global `accept_requests` boolean (set only when `verify_tee()` is last called) and `is_caller_an_attested_participant` (which checks only that an attestation *exists* in storage, never that it is still valid). A node whose TEE attestation has expired since the last `verify_tee()` invocation can still call any `respond*` endpoint and have its response accepted, bypassing the TEE security invariant that only nodes provably running in a trusted execution environment may participate in threshold signing.

### Finding Description

`verify_tee()` is the only function that re-verifies attestation validity against the current block timestamp and updates `accept_requests`. Per the design documentation it is called roughly every 7 days. Between calls, attestations can expire.

The three `respond*` functions check:

1. `self.accept_requests` — a cached boolean last written by `verify_tee()`.
2. `self.assert_caller_is_attested_participant_and_protocol_active()` → `is_caller_an_attested_participant()` — which only confirms an entry exists in `stored_attestations`; it never calls `reverify_participants` or checks the attestation's expiry timestamp.

Neither check re-verifies the *current* validity of the caller's attestation at respond time. [1](#0-0) [2](#0-1) [3](#0-2) 

The live re-verification path (`reverify_participants`) is only exercised inside `reverify_and_cleanup_participants`, which is called exclusively from `verify_tee()` and `vote_new_parameters()`. [4](#0-3) 

`clean_invalid_attestations` can remove expired entries from `stored_attestations`, but it is a separate, explicitly-invoked call and does not update `accept_requests`. [5](#0-4) 

### Impact Explanation

A node whose attestation has expired (meaning it can no longer prove it is running inside a trusted TEE) retains its key shares and can continue to participate in the off-chain threshold signing protocol. When it calls `respond` with a cryptographically valid signature, the contract accepts the response because:

- `accept_requests` is still `true` (set by the last `verify_tee()` call before expiry).
- `is_caller_an_attested_participant` finds the (now-expired) entry in `stored_attestations` and returns `Ok(())`.

The TEE security invariant — that only nodes provably running in a trusted environment may contribute to threshold signatures — is broken for the entire window between attestation expiry and the next `verify_tee()` call (up to ~7 days by design). This constitutes **participant-state manipulation that breaks a production safety invariant**: the contract's TEE-gated signing model.

### Likelihood Explanation

TEE attestations carry expiry timestamps and expire naturally. The design document explicitly states `verify_tee` is called every 7 days. Any node whose attestation expires mid-cycle can exploit this window without any special privilege — it only needs to be a current participant with an expired attestation and a valid threshold signature to submit. [6](#0-5) 

### Recommendation

Perform a live per-caller attestation validity check inside each `respond*` function before accepting the response. Specifically, call `reverify_participants` (or an equivalent single-entry validity check) for the calling node's stored attestation at respond time, rather than relying on the cached `accept_requests` flag and the existence-only `is_caller_an_attested_participant` check.

### Proof of Concept

1. Contract is `Running` with 3 participants; all attestations are valid; `verify_tee()` is called → `accept_requests = true`.
2. Participant P's attestation expires (block timestamp advances past `expiry_timestamp_seconds`).
3. `verify_tee()` is **not** called again (within the 7-day window).
4. P participates in the off-chain threshold signing protocol (other nodes still communicate with it).
5. P calls `respond(request, valid_signature)`.
6. Contract checks `accept_requests` → `true` (stale). Checks `is_caller_an_attested_participant` → finds P's expired entry in `stored_attestations`, returns `Ok(())` (no expiry check). Signature is cryptographically valid → `resolve_yields_for` resolves the pending request and returns the signature to the user.
7. A node that can no longer prove TEE integrity has successfully submitted a threshold signature, bypassing the TEE security model. [7](#0-6)

    pub fn reverify_and_cleanup_participants(
        &mut self,
        participants: &Participants,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeValidationResult {
        self.allowed_docker_image_hashes
            .cleanup_expired_hashes(tee_upgrade_deadline_duration);

        let participants_with_valid_attestation: Vec<_> = participants
            .participants()
            .iter()
            .filter(|(_, _, participant_info)| {
                // Use the stored NodeId (keyed by TLS public key) so the real
                // `account_public_key` participates in re-verification. If
                // there is no stored attestation for this TLS key, the
                // participant is invalid.
                let Some(node_id) = self.find_node_id_by_tls_key(&participant_info.tls_public_key)
                else {
                    return false;
                };

                let tee_status =
                    self.reverify_participants(&node_id, tee_upgrade_deadline_duration);

                matches!(tee_status, TeeQuoteStatus::Valid)
            })
            .cloned()
            .collect();

        if participants_with_valid_attestation.len() != participants.len() {
            let participants_with_valid_attestation =
                Participants::init(participants.next_id(), participants_with_valid_attestation);

            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            }
        } else {
            TeeValidationResult::Full
        }
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
