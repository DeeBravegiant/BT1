### Title
Asymmetric `total_stake` vs `exempted_stake` Entity Sets in Validator Kickout Protection Inflates Exemption Threshold - (File: chain/epoch-manager/src/lib.rs)

### Summary
In `compute_validators_to_reward_and_kickout`, `total_stake` is accumulated over **all** validators in the current epoch including those already in `prev_validator_kickout`, while `compute_exempted_kickout` explicitly **skips** those same previously-kicked-out validators when accumulating `exempted_stake`. The `min_keep_stake` threshold is derived from the inflated `total_stake`, making it systematically impossible for `exempted_stake` to reach the correct proportion. This causes more validators to be exempted from kickout than `validator_max_kickout_stake_perc` intends, leaving underperforming validators in the set for an additional epoch.

### Finding Description

In `compute_validators_to_reward_and_kickout`, the loop over `epoch_info.validators_iter()` unconditionally adds every validator's stake to `total_stake`, including validators whose `account_id` is present in `prev_validator_kickout`: [1](#0-0) 

`total_stake` is then passed to `compute_exempted_kickout` as the denominator for `min_keep_stake`: [2](#0-1) 

Inside `compute_exempted_kickout`, the loop that accumulates `exempted_stake` explicitly skips any validator present in `prev_validator_kickout`: [3](#0-2) 

The invariant the protocol intends to enforce is: *"the total stake of validators kicked out this epoch must not exceed `validator_max_kickout_stake_perc`% of total stake."* The complement is `exempt_perc = 100 - validator_max_kickout_stake_perc`, and `min_keep_stake = total_stake * exempt_perc / 100`.

When previously-kicked-out validators (those in `prev_validator_kickout`) hold stake S_prev, `total_stake` is inflated by S_prev, so `min_keep_stake` is inflated by `S_prev * exempt_perc / 100`. But `exempted_stake` can never include S_prev (those validators are skipped). The algorithm must therefore exempt additional non-kicked-out validators to compensate, exempting more stake than the protocol intends.

### Impact Explanation

The corrupted protocol value is the **validator set for epoch T+2**: validators that should be kicked out for failing block/chunk/endorsement production thresholds are instead exempted and remain active validators. This is a direct analog to the original report's asymmetry: one calculation (`total_stake`) counts all entities including ineligible ones, while the paired calculation (`exempted_stake`) counts only eligible ones, producing a miscalibrated threshold.

The `validator_max_kickout_stake_perc` is set to 30% on both mainnet and testnet: [4](#0-3) 

When previously-kicked-out validators hold, say, 20% of total stake, the effective kickout cap is reduced from 30% to approximately `30% * (1 - 0.20) = 24%` of the eligible validator stake, meaning underperforming validators that collectively hold up to 6% of total stake escape kickout they should receive.

**Impact:** Low — does not corrupt state roots or allow invalid blocks, but incorrectly retains underperforming validators in the active set, degrading chunk/block production reliability for one epoch.

### Likelihood Explanation

**High** — the condition triggers whenever any validator is in `prev_validator_kickout` with non-zero stake, which is a routine occurrence in production (validators are kicked out every epoch for insufficient block/chunk production). The mainnet epoch config has `validator_max_kickout_stake_perc = 30` and kickout thresholds of 80%, so this miscalculation is active in every epoch where at least one validator was kicked out in the prior epoch.

### Recommendation

Exclude previously-kicked-out validators from `total_stake` to match the set used in `compute_exempted_kickout`:

```rust
// In compute_validators_to_reward_and_kickout, line ~469:
- total_stake = total_stake.checked_add(v.stake()).unwrap();
+ if !prev_validator_kickout.contains_key(account_id) {
+     total_stake = total_stake.checked_add(v.stake()).unwrap();
+ }
```

This ensures `min_keep_stake = total_stake * exempt_perc / 100` is computed over the same set of validators that can actually contribute to `exempted_stake`, restoring the intended `validator_max_kickout_stake_perc` semantics.

### Proof of Concept

Consider 5 validators with equal stake 1000 each (total = 5000). `validator_max_kickout_stake_perc = 30`, so `exempt_perc = 70`, `min_keep_stake = 5000 * 70 / 100 = 3500`.

- `prev_validator_kickout` = {test3} (stake 1000, kicked last epoch)
- test0, test1, test2, test4 all fail production thresholds this epoch

`total_stake` = 5000 (includes test3's 1000). `min_keep_stake` = 3500.

`compute_exempted_kickout` iterates by online ratio (descending), skipping test3. It must accumulate 3500 from {test0, test1, test2, test4} (each stake 1000). It exempts test0 (1000), test1 (2000), test2 (3000), test4 (4000 ≥ 3500 → stops after test4 is added). All four are exempted; none are kicked out.

**Correct behavior** (excluding test3 from `total_stake`): `total_stake` = 4000, `min_keep_stake` = 2800. The algorithm exempts test0 (1000), test1 (2000), test2 (3000 ≥ 2800 → stops). test4 is NOT exempted and IS kicked out. The protocol's 30% cap is correctly enforced.

The existing test `test_max_kickout_stake_ratio` in `chain/epoch-manager/src/tests/mod.rs` demonstrates the scenario with `prev_validator_kickout` containing test3, but does not assert that the inflated `total_stake` causes incorrect exemptions: [5](#0-4)

/// Test that the stake of validators kicked out in an epoch doesn't exceed the max_kickout_stake_ratio
fn test_max_kickout_stake_ratio() {

        .for_protocol_version(PROTOCOL_VERSION);

        ("test1".parse().unwrap(), Balance::from_yoctonear(1000)),
        ("test2".parse().unwrap(), Balance::from_yoctonear(1000)),
        ("test3".parse().unwrap(), Balance::from_yoctonear(1000)),
        ("test4".parse().unwrap(), Balance::from_yoctonear(1000)),
    ];
    let epoch_info = epoch_info(
        0,
        accounts,
        vec![0, 1, 2, 3],
        vec![vec![0, 1], vec![2, 4]],
        PROTOCOL_VERSION,
        ShardLayout::multi_shard(num_shards, 0),
    );
    let block_stats = HashMap::from([
        (0, ValidatorStats { produced: 50, expected: 100 }),
        // here both test1 and test2 produced the most number of blocks, we made that intentionally
        // to test the algorithm to pick one deterministically to save in this case.

        // validator 3 doesn't need to produce any block or chunk
        (3, ValidatorStats { produced: 0, expected: 0 }),
    ]);
    let chunk_stats_tracker = HashMap::from([
        (

            HashMap::from([
                (0, ChunkStats::new_with_production(0, 100)),
                (1, ChunkStats::new_with_production(0, 100)),
            ]),
        ),
        (
            ShardId::new(1),
            HashMap::from([
                (2, ChunkStats::new_with_production(100, 100)),
                (4, ChunkStats::new_with_production(50, 100)),
            ]),
        ),
    ]);
    let prev_validator_kickout =
        HashMap::from([("test3".parse().unwrap(), ValidatorKickoutReason::Unstaked)]);
    let (validator_stats, kickouts) = EpochManager::compute_validators_to_reward_and_kickout(
        &epoch_config,
        &epoch_info,
        &block_stats,
        &chunk_stats_tracker,
        &HashMap::new(),
        &prev_validator_kickout,
    );

        HashMap::from([
            ("test0".parse().unwrap(), NotEnoughBlocks { produced: 50, expected: 100 }),
            ("test2".parse().unwrap(), NotEnoughBlocks { produced: 70, expected: 100 }),
            ("test4".parse().unwrap(), NotEnoughChunks { produced: 50, expected: 100 }),
