### Title
Negative `n` in `TimelockMOf` Native Script Bypasses Signature Validation — (File: `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs`)

---

### Summary

The `evalTimelock` function's `isValidMOf` helper uses a signed `Int` counter that can be attacker-supplied as a negative value via CBOR. When `n < 0`, the loop's boundary condition `n <= 0` is satisfied immediately, causing the script to return `True` without evaluating any sub-scripts or signatures. This is the direct analog of the external report's loop-condition underflow: instead of a `uint256` wrapping below zero, a signed `Int` is supplied below zero from the start, causing the loop to skip all validation.

---

### Finding Description

`TimelockRaw` is the raw representation of Allegra-era native scripts. Its `TimelockMOf` constructor stores a signed `Int` as the required count:

```haskell
| TimelockMOf !Int !(StrictSeq (Timelock era))
-- Note that the Int may be negative in which case (TimelockMOf (-2) [..]) is always True
``` [1](#0-0) 

The `DecCBOR` instance for `TimelockRaw` decodes this `Int` directly from CBOR with no non-negativity check:

```haskell
decRaw 3 = Ann (SumD TimelockMOf) <*! Ann From <*! D (sequence <$> decCBOR)
``` [2](#0-1) 

The evaluator is:

```haskell
isValidMOf n SSeq.Empty = n <= 0
isValidMOf n (ts SSeq.:<| tss) =
  n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
``` [3](#0-2) 

The loop's termination condition is `n <= 0`. For any `n < 0` supplied via CBOR, this condition is satisfied on the very first call — before any sub-script or key-hash is examined — and the function returns `True` unconditionally. The loop never iterates; it exits immediately with a passing result.

This is structurally identical to the external report's bug: the loop is supposed to count down from a positive value and stop at zero, but an invalid starting value (negative here, underflowed-to-huge there) causes the boundary condition to be satisfied at the wrong point, skipping all intended checks.

The `EncCBOR` instance encodes `m` via `To m` using the standard `EncCBOR Int` instance, which encodes signed CBOR integers, so negative values round-trip through serialization without error:

```haskell
TimelockMOf m xs -> Sum TimelockMOf 3 !> To m !> To xs
``` [4](#0-3) 

---

### Impact Explanation

A script author submits a transaction containing a `TimelockMOf (-1) [keyHash_A, keyHash_B]` native script as a witness. The ledger calls `evalTimelock`, which calls `isValidMOf (-1) [...]`. Because `-1 <= 0` is immediately `True`, the script validates without checking any key hashes or time bounds. Any UTxO whose address is the hash of such a script can be spent by anyone who possesses the script, with no signatures required.

This exceeds the intended validation limits of the native script system: a `RequireMOf` script is designed to enforce that at least `m` of the listed conditions are satisfied. A negative `m` silently removes that enforcement entirely.

**Matched impact**: *Medium — Attacker-controlled scripts exceed intended validation limits.*

---

### Likelihood Explanation

The attack path is fully attacker-controlled and requires no privileged access:

1. Attacker serializes `TimelockMOf (-1) [keyHash_victim, keyHash_attacker]` to CBOR — a trivial operation.
2. Attacker presents the script to a victim as a "2-of-2 multi-sig" (the script hash is indistinguishable from a legitimate one without inspecting the raw script).
3. Victim sends ADA or native assets to the script address.
4. Attacker submits a spending transaction providing the script as a witness; no signatures are needed.
5. The ledger accepts the transaction because `evalTimelock` returns `True`.

The victim must inspect the raw CBOR of the script to detect the negative `n`. Wallets and tooling that display scripts in human-readable form may show "2 of 2" without revealing the sign of the integer.

---

### Recommendation

Add a non-negativity guard in the `DecCBOR` instance for `TimelockRaw` when decoding the `TimelockMOf` count:

```haskell
decRaw 3 = do
  n <- decCBOR
  when (n < 0) $ fail "TimelockMOf: required count must be non-negative"
  Ann (SumD (TimelockMOf n)) <*! D (sequence <$> decCBOR)
```

Alternatively, change the field type from `Int` to `Word` or `Natural`, which makes negative values unrepresentable at the type level and eliminates the class of bug entirely.

---

### Proof of Concept

Construct the following CBOR (CBOR array tag `3`, signed integer `-1`, empty list):

```
83 03 20 80
```

Decoded: `[3, -1, []]` → `TimelockMOf (-1) []`

Submit a transaction spending a UTxO locked at `hash(TimelockMOf (-1) [])` with this script as the native-script witness and no key witnesses. The ledger evaluates:

```
isValidMOf (-1) Empty
  = (-1) <= 0
  = True
```

The script passes. The transaction is accepted. No signatures were provided or required. [5](#0-4)

```haskell
evalTimelock ::
  (AllegraEraScript era, NativeScript era ~ Timelock era) =>
  Set.Set (KeyHash Witness) ->
  ValidityInterval ->
  NativeScript era ->
  Bool
evalTimelock vhks (ValidityInterval txStart txExp) = go
  where
    -- The important part of this validator is that it will stop as soon as it reaches the
    -- required number of valid scripts
    isValidMOf n SSeq.Empty = n <= 0
    isValidMOf n (ts SSeq.:<| tss) =
      n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
    go = \case
      RequireTimeStart lockStart -> lockStart `lteNegInfty` txStart
      RequireTimeExpire lockExp -> txExp `ltePosInfty` lockExp
      RequireSignature hash -> hash `Set.member` vhks
      RequireAllOf xs -> all go xs
      RequireAnyOf xs -> any go xs
      RequireMOf m xs -> isValidMOf m xs
      _ -> error "Impossible: All NativeScripts should have been accounted for"
```
