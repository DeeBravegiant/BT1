### Title
Double `PromiseResume` Receipt Emission via Repeated `promise_yield_resume` on Same `data_id` — (`runtime/runtime/src/ext.rs`)

### Summary

`RuntimeExt::submit_promise_resume_data` guards against duplicate resume using `has_yield_receipt_in_state || has_yield_status_in_state`, but neither condition is cleared after the first successful call within the same function execution. A contract can call `promise_yield_resume` twice with the same `data_id` in one execution, causing two `PromiseResume` receipts with the same `data_id` to be appended to `result.new_receipts`, violating the one-resume-per-`data_id` invariant and producing a different state root and receipt set.

### Finding Description

The guard in `submit_promise_resume_data` is:

```rust
if has_yield_receipt_in_state || has_yield_status_in_state {
    self.receipt_manager.create_promise_resume_receipt(data_id, data);
    set_promise_yield_status(
        &mut self.trie_update,
        &self.account_id,
        data_id,
        PromiseYieldStatus::ResumeInitiated,
    );
    return Ok(true);
}
``` [1](#0-0) 

After the first call:

1. `has_promise_yield_receipt` checks `TrieKey::PromiseYieldReceipt` — the yield receipt is **not removed** from the trie by `submit_promise_resume_data`, so `has_yield_receipt_in_state` remains `true`.
2. `set_promise_yield_status` writes `PromiseYieldStatus::ResumeInitiated` into `trie_update` — so `has_yield_status_in_state` also remains `true` (it checks for key existence, not for a specific status value). [2](#0-1) 

`has_promise_yield_status` uses `trie.contains_key(...)` — it returns `true` for **any** status value, including `ResumeInitiated`: [2](#0-1) 

`TrieUpdate` reflects in-memory writes immediately (the `set_promise_yield_status` write is visible to the next `contains_key` call within the same execution): [3](#0-2) 

On the **second call** with the same `data_id`, both `has_yield_receipt_in_state` and `has_yield_status_in_state` are still `true`, so `create_promise_resume_receipt` is called again: [4](#0-3) 

This unconditionally pushes a second `DataReceiptMetadata { data_id, is_promise_resume: true }` into `data_receipts` with no deduplication check.

### Impact Explanation

Two `PromiseResume` receipts with the same `data_id` are emitted in `result.new_receipts`. This:

- Produces a **different state root** (the receipt set differs from the intended one-receipt-per-`data_id` invariant).
- Potentially causes the yield callback to execute **twice** (double spend of attached deposit, double state mutation), depending on how the runtime processes the second `PromiseResume` after the yield receipt has been consumed.
- Is deterministic and reproducible, so all nodes applying the same chunk will agree on the corrupted state — meaning this is a consensus-level state corruption, not a fork.

### Likelihood Explanation

Any unprivileged user can deploy a WASM contract that calls `promise_yield_resume` twice with the same `data_id` in a single function body. No validator or admin privileges are required. The precondition (a `PromiseYieldReceipt` in state for the `data_id`) is satisfied by the normal yield/resume flow: the contract first calls `promise_yield_create`, which stores the yield receipt in the trie, and then in a subsequent execution calls `promise_yield_resume` twice.

### Recommendation

In `submit_promise_resume_data`, after the first successful resume, the guard must distinguish `ResumeInitiated` from `Yielded`. The check should be changed to verify the status is specifically `Yielded` (not merely present), or the yield receipt should be removed from the trie on the first resume call. For example:

```rust
// Instead of: has_yield_receipt_in_state || has_yield_status_in_state
// Check that status is Yielded specifically:
let status = get_promise_yield_status(...)?;
if status == Some(PromiseYieldStatus::Yielded) || has_yield_receipt_in_state {
    // proceed
}
``` [5](#0-4) 

### Proof of Concept

1. Deploy a contract with a method that:
   - Calls `promise_yield_create` to obtain a `data_id` (stores `PromiseYieldReceipt` in trie).
   - In a subsequent receipt execution, calls `promise_yield_resume(data_id, b"first")` and then `promise_yield_resume(data_id, b"second")` in the same function body.
2. Apply the chunk via `Runtime::apply`.
3. Assert that `result.new_receipts` contains **two** `PromiseResume` receipts with the same `data_id` — violating the invariant that exactly one is emitted per `data_id`. [1](#0-0) [6](#0-5)

        &mut self,
        data_id: CryptoHash,

        let has_yield_receipt_in_state =
