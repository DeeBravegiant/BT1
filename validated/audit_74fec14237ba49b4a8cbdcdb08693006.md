Audit Report

## Title
Stale `rsETHPrice` Enables Yield Extraction from Existing rsETH Holders During stETH Rebase Windows ŌĆö (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle` stores `rsETHPrice` as a state variable updated only when `updateRSETHPrice()` is explicitly called. Because stETH is a rebase token whose `balanceOf` increases automatically every ~24 hours, a window exists between each rebase and the next price update during which `rsETHPrice` is stale-low relative to the live TVL. Any depositor who calls `depositAsset(stETH, ...)` inside this window receives more rsETH than the true exchange rate warrants, permanently diluting the yield that should accrue to existing rsETH holders.

## Finding Description

`rsETHPrice` is a stored state variable in `LRTOracle`: [1](#0-0) 

It is only written inside `_updateRsETHPrice()`, which is invoked by the public `updateRSETHPrice()` or the manager-gated `updateRSETHPriceAsManager()`: [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` divides by this stored value directly: [3](#0-2) 

Meanwhile, `_getTotalEthInProtocol()` ŌĆö called inside `_updateRsETHPrice()` ŌĆö computes TVL from live on-chain balances via `getTotalAssetDeposits`, which reads `IERC20(asset).balanceOf(...)`: [4](#0-3) [5](#0-4) 

stETH is a rebase token: its `balanceOf` increases automatically every ~24 hours without any transaction. After a rebase event of size ╬┤ ETH:

- Live TVL = `T + ╬┤`
- Stored `rsETHPrice` = `T / S` (stale, not yet updated)
- True price = `(T + ╬┤) / S`

An attacker depositing `D` stETH (Ōēł D ETH) during this window receives:

```
rsETH_minted = D / rsETHPrice_stale = D * S / T
```

The correct amount at the true price would be:

```
rsETH_correct = D * S / (T + ╬┤)
```

The attacker receives `D * S * ╬┤ / (T * (T + ╬┤))` excess rsETH.

When `updateRSETHPrice()` is subsequently called, `previousTVL` is computed as `rsethSupply * rsETHPrice`: [6](#0-5) 

With the attacker's inflated supply, `previousTVL` accounts for the attacker's deposit at the stale price, so `rewardAmount = totalETHInProtocol - previousTVL` still equals ╬┤ (the rebase). However, the new price is computed over the now-inflated supply: [7](#0-6) 

This yields a lower `newRsETHPrice` than would result without the attacker's deposit. Algebraically, for existing holders' rsETH:

- Without attacker: price = `(T + ╬┤ ŌłÆ fee) / S`
- With attacker: price = `T * (T + ╬┤ + D ŌłÆ fee) / (S * (T + D))`

The difference is positive (`D * (╬┤ ŌłÆ fee) / (S * (T + D))`), confirming existing holders receive less ETH per rsETH. The attacker's excess rsETH represents a direct claim on the rebase yield that belonged to existing holders.

No guard in `_beforeDeposit` forces a price refresh before minting: [8](#0-7) 

The `pricePercentageLimit` check only gates callers who are not managers when the new price exceeds `highestRsethPrice` by more than the threshold ŌĆö a normal daily stETH rebase (Ōēł0.011%) is far below any reasonable limit, so the public `updateRSETHPrice()` path is unblocked and the window closes only when someone calls it: [9](#0-8) 

## Impact Explanation
**High ŌĆö Theft of unclaimed yield.** Existing rsETH holders are entitled to the full benefit of each stETH rebase. A depositor who times their deposit into the stale-price window extracts a portion of that yield proportional to their deposit share of TVL. For a 0.011% daily rebase and an attacker depositing 10% of TVL, approximately 10% of the daily rebase yield is redirected to the attacker. At protocol scale (hundreds of millions of TVL), this is material and repeatable every ~24 hours.

## Likelihood Explanation
**LowŌĆōMedium.** The stETH rebase window is predictable and recurring (~24 hours). `updateRSETHPrice()` is public, so the window closes as soon as any party calls it, but a sophisticated attacker can monitor the stETH oracle/rebase event and front-run the price update transaction. No privileged access is required ŌĆö any unprivileged depositor can trigger this by calling `depositAsset`. The attack is repeatable every rebase cycle.

## Recommendation
1. **Pull-based price freshness**: In `_beforeDeposit`, call `updateRSETHPrice()` (or an internal equivalent) before computing `rsethAmountToMint`, ensuring the price used for minting always reflects the current on-chain TVL.
2. **Alternatively**, compute `rsethAmountToMint` using a live TVL snapshot (`_getTotalEthInProtocol() / rsethSupply`) rather than the stored `rsETHPrice`, eliminating the staleness window entirely.
3. **At minimum**, enforce a maximum age on `rsETHPrice` (e.g., revert deposits if the price has not been updated within the last N blocks).

## Proof of Concept
1. At block B, stETH rebases: the protocol's stETH balance increases by ╬┤. `_getTotalEthInProtocol()` would now return `T + ╬┤`, but `rsETHPrice` remains `T / S`.
2. Attacker calls `depositAsset(stETH, D, 0, "")` at block B+1. `getRsETHAmountToMint` computes `D * getAssetPrice(stETH) / rsETHPrice_stale`, minting more rsETH than the true price warrants.
3. At block B+2, anyone calls `updateRSETHPrice()`. The new price is computed over the inflated `rsethSupply`, yielding a lower `newRsETHPrice` than would have resulted without step 2.
4. Existing holders' rsETH is now worth less ETH than it should be. The difference ŌĆö proportional to the attacker's deposit share ŌĆö has been transferred to the attacker as excess rsETH.
5. After `withdrawalDelayBlocks`, the attacker redeems their rsETH via `LRTWithdrawalManager`, receiving more stETH than they deposited at fair value.
