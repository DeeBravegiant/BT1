### Title
FanOut Inspector Accepts Single-Provider Response Without Enforcing On-Chain RPC Quorum — (File: `crates/foreign-chain-inspector/src/lib.rs`)

---

### Summary

The `FanOut<Inspector>` implementation in `crates/foreign-chain-inspector/src/lib.rs` returns a successful foreign-chain verification result as soon as **any single** RPC provider succeeds, without checking whether the number of agreeing providers meets the on-chain `ChainEntry.quorum` threshold voted in by participants. A single compromised or manipulated whitelisted RPC provider — combined with transient failures on the remaining providers — is sufficient to cause every MPC node to sign a forged foreign-chain verification.

---

### Finding Description

The `FanOut<Inspector>` struct fans out a foreign-chain query to all configured inspectors concurrently and is intended to enforce an RPC response quorum before accepting a result. However, the `extract` method contains no quorum check: [1](#0-0) 

```rust
if let Some(first_values) = successes.first() {
    let all_successes_agree = successes.iter().all(|(_, v)| v == &first_values.1);
    if !all_successes_agree {
        return Err(ForeignChainInspectionError::InspectorResponseMismatch);
    }
    let (_, first) = successes.into_iter().next().expect("checked non-empty");
    return Ok(first);
}
```

The method returns `Ok` whenever `successes.len() >= 1` and all successes agree — regardless of how many providers actually responded. The `FanOut` struct itself accepts no `quorum` parameter: [2](#0-1) 

The on-chain `ChainEntry` stores a governance-voted `quorum` field specifying the minimum number of providers that must agree before a node accepts a verification result: [3](#0-2) 

This quorum is voted in by participants via `vote_update_foreign_chain_providers` and requires the protocol signing threshold to apply: [4](#0-3) 

The design documentation explicitly acknowledges the gap:

> "A node considers the foreign TX verified iff at least the per-chain RPC quorum of providers agreed. If fewer than the RPC quorum agree, the node errors out and produces no signature share. **This is an implementation requirement, not current behavior.**" [5](#0-4) 

The node-side `execute_foreign_chain_request` calls the inspector and uses its result directly with no post-hoc quorum check: [6](#0-5) 

The transient-error classification in `ForeignChainInspectionError::is_transient` includes `ClientError`, `RpcRequestFailed`, and `NotFinalized`: [7](#0-6) 

These are exactly the error classes an attacker can induce on honest providers (network disruption, rate-limit exhaustion, timeout) while keeping a single compromised provider responsive.

---

### Impact Explanation

**High — forged foreign-chain verification leading to invalid bridge execution.**

The `verify_foreign_transaction` flow is the trust anchor for the Omnibridge inbound path: a NEAR contract reacts to a foreign-chain event only after the MPC network signs an attestation that the transaction finalized. If a single compromised RPC provider can satisfy the `FanOut` check, the MPC network will produce a valid threshold signature over a fabricated `ForeignTxSignPayload`, attesting to a transaction that never occurred or was never finalized. The resulting `VerifyForeignTransactionResponse` is returned on-chain and accepted by any NEAR contract that trusts it, enabling double-spend or theft of bridged assets. [8](#0-7) 

---

### Likelihood Explanation

**Medium.** The attacker must control or compromise one whitelisted RPC provider for the target chain. The whitelist is governance-voted, so the provider set is known and stable. Causing transient failures on the remaining providers (network-level disruption, rate-limit exhaustion against specific API keys, or simply targeting a moment when other providers are degraded) is realistic for a well-resourced adversary. No threshold-level collusion among MPC participants is required — the attack is entirely external to the MPC node set.

---

### Recommendation

Pass the `ChainEntry.quorum` value into `FanOut` at construction time and enforce it inside `extract`:

```rust
pub struct FanOut<Inspector> {
    inspectors: NonEmptyVec<Inspector>,
    quorum: usize,  // minimum number of agreeing successes required
}

// In extract():
if successes.len() < self.quorum {
    return Err(ForeignChainInspectionError::SubQuorumResponse {
        required: self.quorum,
        got: successes.len(),
    });
}
```

The `quorum` value should be read from the on-chain `ChainEntry.quorum` field (already voted in by participants) and wired through the node's inspector construction path. The design documentation already specifies this as the intended behavior; the fix closes the gap between specification and implementation. [9](#0-8) 

---

### Proof of Concept

**Setup**: A chain (e.g., Ethereum) is whitelisted with `quorum = 2` and three providers: `alchemy`, `infura`, `ankr`.

**Attack**:
1. Attacker compromises or controls `alchemy`'s RPC endpoint for the target chain.
2. Attacker causes `infura` and `ankr` to return transient errors for the target node (e.g., via rate-limit exhaustion of those API keys, or network-level disruption).
3. Attacker submits `verify_foreign_transaction` for a fabricated Bitcoin `tx_id` that does not exist on-chain.
4. Each MPC node's `FanOut` fires all three inspectors concurrently.
5. `alchemy` returns a crafted `GetRawTransactionVerboseResponse` with sufficient confirmations and a plausible block hash; `infura` and `ankr` return `ClientError` (transient).
6. `FanOut::extract` sees `successes = [(0, forged_values)]`, `non_transient_errors = []`, `first_transient_error = Some(...)`. The split check passes (`non_transient_errors` is empty). `all_successes_agree = true`. Returns `Ok(forged_values)`.
7. `build_signature_request` hashes the forged `ForeignTxSignPayload` and the node proceeds to produce a signature share.
8. Once threshold nodes have been similarly deceived, `respond_verify_foreign_tx` is called with a valid threshold signature over the forged payload.
9. The NEAR contract resumes the caller's yield with a `VerifyForeignTransactionResponse` attesting to a transaction that never occurred. [10](#0-9) [11](#0-10)

```rust
    /// pair. `protocol_threshold` is the protocol's signing threshold (the same one
    /// that gates threshold signatures), not to be confused with `ChainEntry.quorum`,
    /// which is the RPC response quorum nodes use when querying the listed providers.
    ///
    /// The input batch is a `NonEmptyBTreeMap<ForeignChain, ChainEntry>`, so two
    /// invariants are enforced at borsh-deserialize time and don't need to be re-checked
    /// here: the batch is non-empty, and each chain appears at most once.
    ///
    /// Returns the chains applied this call.
    pub fn vote(
        &mut self,
        participant: AuthenticatedParticipantId,
        votes: NonEmptyBTreeMap<ForeignChain, dtos::ChainEntry>,
        threshold_parameters: &ThresholdParameters,
    ) -> Result<Vec<ForeignChain>, Error> {
        let mut applied: Vec<ForeignChain> = Vec::new();
        let votes: BTreeMap<ForeignChain, dtos::ChainEntry> = votes.into();
        for (chain, entry) in votes {
            let entry: ChainEntry = entry.try_into()?;
            let hash = ProposalHash::from(entry.clone());
            if self
                .votes
                .vote(chain, hash, participant.clone(), threshold_parameters)?
            {
                self.entries.replace(chain, entry);
                applied.push(chain);
            }
        }
        Ok(applied)
    }
}
```

```markdown
### Individual node quorum of RPC providers for verification requests

When a foreign TX verification request is processed by a set of nodes, every node individually queries its locally-configured RPC providers for that chain. A node considers the foreign TX verified iff at least the per-chain RPC quorum of providers agreed. If fewer than the RPC quorum agree, the node errors out and produces no signature share. This sub-quorum outcome must be **terminal** for the request — the leader does not re-attempt it. This is an implementation requirement, not current behavior. See [Calculating the whitelisted and available foreign-chain sets](calculating-supported-foreign-chains.md#verification-behavior).

The quorum value comes from on-chain `ChainEntry.quorum`, voted in as part of the same `ChainVote` that voted the chain's provider list — so participants agree on both "which providers are trusted" and "how many of them must concur" in one round.
```
