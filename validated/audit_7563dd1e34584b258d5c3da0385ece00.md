### Title
Missing HTTP Timeout on Pool Protocol Requests Enables Indefinite Farmer Blocking - (File: chia/farmer/farmer_api.py, chia/farmer/farmer.py)

### Summary
Multiple pool-facing HTTP calls in the Chia farmer are made without any request timeout. A malicious or unresponsive pool server can cause the farmer's coroutines to hang indefinitely, permanently blocking partial submission and pool state updates, causing the farmer to lose all pool rewards.

### Finding Description

Four functions in `chia/farmer/farmer.py` and one in `chia/farmer/farmer_api.py` open `aiohttp.ClientSession()` without a timeout and issue HTTP requests without a `timeout=` argument:

**`chia/farmer/farmer_api.py` — `new_proof_of_space` (POST /partial):** [1](#0-0) 

No timeout is passed to either `ClientSession()` or `session.post()`. This is called for every valid proof of space the harvester finds.

**`chia/farmer/farmer.py` — `_pool_get_pool_info` (GET /pool_info):** [2](#0-1) 

**`chia/farmer/farmer.py` — `_pool_get_farmer` (GET /farmer):** [3](#0-2) 

**`chia/farmer/farmer.py` — `_pool_post_farmer` (POST /farmer):** [4](#0-3) 

**`chia/farmer/farmer.py` — `_pool_put_farmer` (PUT /farmer):** [5](#0-4) 

By contrast, the Data Layer's `get_downloader` and `get_uploaders` correctly pass `timeout=self.client_timeout` to every pool plugin request: [6](#0-5) 

### Impact Explanation

`new_proof_of_space` is invoked by the farmer event loop for every proof delivered by a harvester. When the pool server deliberately stalls the TCP connection (sends SYN-ACK but never sends an HTTP response), the `async with session.post(...)` context manager awaits indefinitely. Because asyncio cooperative scheduling is used, the coroutine is suspended at the `await` point and never resumes. Each subsequent proof spawns another hung coroutine. The farmer accumulates unbounded pending tasks, all waiting on the pool server, and no partial is ever acknowledged. The farmer loses all pool rewards for the duration of the hang.

The `update_pool_state` path (`_pool_get_pool_info` → `_pool_get_farmer` → `_pool_post_farmer` / `_pool_put_farmer`) is called on a periodic timer. A hang there prevents the farmer from refreshing pool difficulty and authentication tokens, causing all subsequent partials to be rejected as stale or invalid even after the hang resolves. [7](#0-6) 

### Likelihood Explanation

Any pool operator the farmer has joined can trigger this by simply not responding to HTTP requests. No cryptographic material, key compromise, or host access is required. The pool URL is an external HTTP endpoint controlled by a third party. A network-level attacker capable of intercepting pool traffic (DNS hijack, BGP hijack, MITM on non-TLS pool URLs) can also trigger this without being the pool operator.

### Recommendation

Pass an explicit `aiohttp.ClientTimeout` to every pool HTTP call. A consistent pattern already exists in the codebase:

```python
# farmer.py / farmer_api.py — apply to all five call sites
POOL_REQUEST_TIMEOUT = aiohttp.ClientTimeout(total=30)

async with aiohttp.ClientSession(timeout=POOL_REQUEST_TIMEOUT) as session:
    async with session.post(
        f"{pool_url}/partial",
        json=post_partial_request.to_json_dict(),
        ssl=ssl_context_for_root(get_mozilla_ca_crt(), log=self.farmer.log),
        timeout=POOL_REQUEST_TIMEOUT,
        headers={...},
    ) as resp:
        ...
```

Catch `asyncio.TimeoutError` alongside the existing `Exception` handler at each call site so that a timeout is logged and counted as a pool error rather than silently hanging.

### Proof of Concept

1. Farmer joins a pool at `https://pool.example.com`.
2. Pool operator configures their server to accept TCP connections on port 443 but never send an HTTP response (e.g., `iptables` DROP on outbound after the TLS handshake, or a custom server that reads the request and then sleeps forever).
3. Farmer's harvester finds a valid proof of space and calls `new_proof_of_space`.
4. `session.post(f"{pool_url}/partial", ...)` is awaited with no timeout — the coroutine suspends indefinitely.
5. Every subsequent proof spawns another hung coroutine. No partial is ever submitted. The farmer earns zero pool rewards for the entire duration.
6. Simultaneously, `update_pool_state` hangs in `_pool_get_pool_info`, preventing pool difficulty refresh. Even if the pool later responds, stale authentication tokens cause partials to be rejected.
