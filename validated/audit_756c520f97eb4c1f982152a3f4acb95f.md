### Title
Missing `curve.check()` on `root_Q`/`root_K` in ECDSA-2PC HD Keyset Blob Parser Allows Infinity-Point Injection on P256 - (`src/cbmpc/api/hd_keyset_ecdsa_2p.cpp`)

---

### Summary

`blob_to_keyset` in `src/cbmpc/api/hd_keyset_ecdsa_2p.cpp` calls `ecc_point_t::from_bin` on `root_Q_compressed` and `root_K_compressed` but never calls `curve.check()` afterward. For the P256 curve, `ecurve_ossl_t::from_bin` explicitly accepts the all-zero infinity-point encoding. An attacker who controls the keyset blob can inject `root_Q = ∞`, which passes deserialization, survives the VRF-DH protocol step (which only uses `root_K`), and causes `derive_ecdsa_2p_keys` to emit derived ECDSA-2PC key blobs whose public key is `delta*G + non_hard_delta*G` instead of `Q + delta*G + non_hard_delta*G`. Those derived blobs pass `ecdsa_2p::blob_to_key` validation (which does call `curve.check()` on the derived Q, but that point is a valid non-infinity point). The result is accepted invalid cryptographic output.

---

### Finding Description

**Vulnerable function:** `blob_to_keyset` in `src/cbmpc/api/hd_keyset_ecdsa_2p.cpp`

```cpp
error_t rv = keyset.root.Q.from_bin(icurve, blob.root_Q_compressed);
if (rv) return rv;
return keyset.root.K.from_bin(icurve, blob.root_K_compressed);
```

No `curve.check()` is called after either `from_bin`. [1](#0-0) 

**Contrast with `ecdsa_2p::blob_to_key`**, which explicitly guards the same operation:

```cpp
if (const error_t rv = key.Q.from_bin(curve, blob.Q_compressed)) return rv;
if (curve.check(key.Q)) return coinbase::error(E_FORMAT, "invalid key blob");
``` [2](#0-1) 

**Why `from_bin` accepts infinity on P256:** `ecurve_ossl_t::from_bin` contains an explicit infinity-handling branch: if the first byte is `0x00` and all bytes are zero, it truncates to 1 byte and passes it to `EC_POINT_oct2point`, which sets the point to infinity and returns success.

```cpp
if (bin.size > 0 && bin[0] == 0)  // infinity
{
  if (bin.size != 1 + size() && bin.size != 1 + size() * 2) return coinbase::error(E_FORMAT);
  for (int i = 0; i < bin.size; i++)
    if (bin[i]) return coinbase::error(E_CRYPTO);
  bin.size = 1;
}
if (0 >= EC_POINT_oct2point(...)) { return openssl_error(...); }
return SUCCESS;
``` [3](#0-2) 

**Why `curve.check()` would catch this:**

```cpp
error_t ecurve_t::check(const ecc_point_t& point) const {
  if (!point.valid()) return crypto::error("EC-point invalid");
  if (point.get_curve() != *this) return crypto::error("EC-point of wrong curve");
  if (!point.is_in_subgroup()) return crypto::error("EC-point is not on curve");
  if (!thread_local_store_allow_ecc_infinity) {
    if (point.is_infinity()) return crypto::error("EC-point is infinity");
  }
  return SUCCESS;
}
``` [4](#0-3) 

**Why secp256k1 is NOT affected:** `ecurve_secp256k1_t::from_bin` delegates to `secp256k1_eckey_pubkey_parse`, which only accepts bytes starting with `0x02` or `0x03` (compressed) or `0x04`/`0x06`/`0x07` (uncompressed/hybrid). There is no encoding for the infinity point in this format; any attempt returns 0 (failure). [5](#0-4) [6](#0-5) 

**Why the DH proof in `derive_keys` does not catch this:** The VRF-DH step uses `root.K` (not `root.Q`) for its proof. `K_share = k_share * G` and `other_K_share = K - k_share * G`. If `root_K` is a valid non-infinity point, `other_K_share` is also valid, and `dh_t::verify` calls `curve.check(A)` on `other_K_share`, which passes. `root_Q` is only used as `Q = key.root.Q` and then added to `delta_G`; no check is performed on it. [7](#0-6) 

**Propagation to derived key blob:** With `root_Q = ∞`:

```cpp
ecc_point_t Q_derived = Q + delta_G;  // = ∞ + delta*G = delta*G
derived_keys[i].Q = Q_derived + non_hard_delta_G;  // = (delta + non_hard_delta)*G
``` [8](#0-7) 

The derived `Q` is a valid non-infinity point, so `serialize_ecdsa2pc_key_blob` succeeds and the output blob passes `ecdsa_2p::blob_to_key` validation. The public key in the blob is wrong (it does not equal `x_share_p1 * G + x_share_p2 * G`), but this is not detected.

---

### Impact Explanation

An attacker who can supply a crafted P256 keyset blob to `derive_ecdsa_2p_keys` (or `refresh`) obtains derived ECDSA-2PC key blobs that:
- Pass all API-level validation
- Contain a public key that does not correspond to the stored private key shares
- Will produce signatures that fail external verification

This is accepted invalid cryptographic output at the public API boundary, matching the "High" impact tier: *"Public API reachable validation bypass … creates accepted invalid cryptographic output."*

---

### Likelihood Explanation

The keyset blob is an opaque persisted byte string. Any party that can write to the blob store (storage-layer attacker, compromised application tier, or a party that receives the blob over an unauthenticated channel) can inject this. The crafted blob requires only a valid Paillier key, a valid `k_share`, a valid `root_K` point, and 33 zero bytes for `root_Q_compressed`. The attack is limited to P256 (not secp256k1).

---

### Recommendation

Add `curve.check()` calls immediately after both `from_bin` calls in `blob_to_keyset`:

```cpp
error_t rv = keyset.root.Q.from_bin(icurve, blob.root_Q_compressed);
if (rv) return rv;
if (icurve.check(keyset.root.Q)) return coinbase::error(E_FORMAT, "invalid keyset blob: root_Q");

rv = keyset.root.K.from_bin(icurve, blob.root_K_compressed);
if (rv) return rv;
if (icurve.check(keyset.root.K)) return coinbase::error(E_FORMAT, "invalid keyset blob: root_K");
return SUCCESS;
```

Apply the same fix to the EdDSA HD keyset parser (`src/cbmpc/api/hd_keyset_eddsa_2p.cpp` lines 78–80), which has the identical omission. [9](#0-8) 

---

### Proof of Concept

1. Construct a `keyset_blob_v1_t` for P256 with:
   - `version = 1`, `role = 0` (p1), `curve = <P256 curve_id>`
   - `root_Q_compressed` = 33 zero bytes (infinity encoding)
   - `root_K_compressed` = valid compressed P256 point (e.g., the generator)
   - `k_share` = any value in `[1, q)`
   - `paillier` = valid 2048-bit Paillier key pair
   - `x_share` = any value in `Z_N`
   - `c_key` = `paillier.encrypt(x_share)`
2. Serialize with `coinbase::convert(blob)`.
3. Call `coinbase::api::hd_keyset_ecdsa_2p::derive_ecdsa_2p_keys(job, crafted_blob, hard_path, non_hard_paths, sid, out)`.
4. Assert return value is `SUCCESS`.
5. Call `coinbase::api::ecdsa_2p::get_public_key_compressed(out[0], pub)` — assert `SUCCESS`.
6. Verify that `pub` equals `(delta + non_hard_delta[0]) * G` (not `Q_real + delta*G + non_hard_delta[0]*G`).
7. Attempt to sign a message with the derived blob and verify the signature against `pub` — the signature will be invalid because the private key shares do not correspond to `pub`.
