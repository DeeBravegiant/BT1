## Assessment

The claim's attack path is **partially correct** but the conclusion overstates the exploitability. Here is the precise analysis:

**What the validation DOES protect:**
- `validate_received_state_from_peer` cryptographically proves (via Merkle proofs against the weight-proof-anchored chain) that the DL singleton coin was actually spent at the claimed height. An untrusted peer cannot fabricate a coin state for a coin that was not actually spent on-chain.

**What the validation does NOT protect:**
- After the coin state is validated, `fetch_coin_spend_for_coin_state(coin_state, peer)` is called back to the **same untrusted peer**. The peer can return the correct puzzle (puzzle hash is checked) but an **arbitrary solution**. `singleton_removed` then runs `run_with_cost(puzzle, ..., solution)` with the attacker-supplied solution and extracts `root` from the resulting `CREATE_COIN` conditions. [8](#0-7) 

**The real constraint:** For the attack to reach `singleton_removed`, the DL singleton coin must have **actually been spent on-chain**. The attacker cannot fabricate a spend for a coin that was never spent — the Merkle proof validation would reject it. The attacker can only corrupt the wallet's *interpretation* of a real, on-chain spend by substituting a different solution (e.g., encoding a different root).

**Concrete impact:** For a DL singleton that was legitimately spent (root A → root B on-chain), an untrusted peer can cause the wallet to record root C instead of root B. This corrupts the wallet's local DL store view — it will subscribe to a wrong child coin, record the wrong root, and may accept/reject DL inclusion proofs incorrectly.

**Verdict:**

### Title
Untrusted Peer Can Corrupt DataLayer Singleton Root via Unvalidated Solution in `fetch_coin_spend_for_coin_state` — (`chia/wallet/wallet_state_manager.py`)

### Summary
After `validate_received_state_from_peer` cryptographically confirms a DL singleton coin was spent on-chain, `_add_coin_states` calls `fetch_coin_spend_for_coin_state(coin_state, peer)` back to the same untrusted peer. Only the puzzle hash is validated; the solution is accepted as-is. `singleton_removed` runs the puzzle with the attacker-supplied solution and writes the resulting root to the DL store.

### Finding Description
In `_add_coin_states` at line 2125, when `record.wallet_type == WalletType.DATA_LAYER` and `coin_state.spent_height is not None`, the code calls `fetch_coin_spend_for_coin_state(coin_state, peer)` where `peer` is the originating (potentially untrusted) peer. `fetch_coin_spend` validates only that `puzzle.get_tree_hash() == coin.puzzle_hash` and `coin_name` matches — the solution is unchecked. `singleton_removed` then executes `run_with_cost(puzzle, MAX_BLOCK_COST_CLVM, solution)` with the attacker-controlled solution and extracts `root = bytes32(condition[3][1])` from the output `CREATE_COIN` condition, writing it to `dl_store.add_singleton_record`. [5](#0-4) [6](#0-5) [7](#0-6) 

### Impact Explanation
The wallet's local DL store root is corrupted to an attacker-chosen value. Downstream effects include: subscribing to a wrong child coin ID, recording a false root in `SingletonRecord`, and potentially accepting/rejecting DL inclusion proofs against the wrong root. This is "Corruption of Data Layer root/store state with direct security impact" per the High scope.

### Likelihood Explanation
Requires: (1) an untrusted peer connection (default for public nodes), (2) a real DL singleton coin that was actually spent on-chain (publicly observable), (3) the ability to serve a modified `RespondPuzzleSolution`. All three are achievable by any network-level attacker without any key material.

### Recommendation
After fetching the `CoinSpend` from an untrusted peer, validate the solution against the on-chain removals data. Specifically, re-derive the expected output conditions from the known puzzle and compare the `CREATE_COIN` puzzle hash against the actual child coin's puzzle hash (which can be fetched and validated via `fetch_children` + Merkle proof). Alternatively, fetch the puzzle/solution from a trusted peer or from the local full node when available.

### Proof of Concept
1. Observe a real DL singleton coin `C` spent at height `H` (root A → root B on-chain).
2. Connect to the victim wallet as an untrusted peer.
3. Send `CoinStateUpdate(height=H, fork_height=H-1, peak_hash=<real>, items=[CoinState(C, spent_height=H, created_height=...)])`.
4. Send `NewPeakWallet` with the real peak; wallet validates the coin state via Merkle proofs (passes, coin was actually spent).
5. Wallet calls back `request_puzzle_solution(C.name(), H)` to the attacker peer.
6. Respond with the correct puzzle reveal but a solution encoding root C (arbitrary).
7. Assert: `dl_store.get_latest_singleton(launcher_id).root == C` (not B).

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(
        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)
    )
    if solution_response is None or not isinstance(solution_response, RespondPuzzleSolution):
        raise PeerRequestException(f"Was not able to obtain solution {solution_response}")
    coin_id = coin.name()
    if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
        raise PeerRequestException(f"Peer returned wrong puzzle hash for coin {coin_id}")
    if solution_response.response.coin_name != coin_id:
        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )
