Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero Mid-Period - (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` distributes ERC-20 reward tokens over a fixed window set by `notifyRewardAmount`. When all stakers call `initiateWithdrawal` during an active reward period, `totalKernelStaked` immediately drops to zero, causing `rewardPerToken()` to freeze its accumulator for the remainder of the window. All reward tokens allocated to the zero-staked interval are permanently locked in the contract with no recovery path.

## Finding Description

`initiateWithdrawal` immediately decrements `totalKernelStaked` at initiation time, not at claim time: [1](#0-0) 

`rewardPerToken()` handles the zero-staked case by returning the frozen stored accumulator, silently discarding `rewardRate * elapsed` for every second `totalKernelStaked` remains zero: [2](#0-1) 

The `updateReward` modifier snapshots `rewardPerToken()` before every state change, so the frozen value propagates into every subsequent user checkpoint: [3](#0-2) 

`notifyRewardAmount` guards against starting a new period with zero stakers: [4](#0-3) 

But there is no symmetric guard in `initiateWithdrawal` preventing `totalKernelStaked` from reaching zero while `block.timestamp < finishAt`. The contract's own NatDoc explicitly acknowledges this consequence and relies entirely on off-chain operational discipline: [5](#0-4) 

There is no admin sweep, rescue, or re-allocation function anywhere in the contract for stranded reward tokens.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** Any reward tokens allocated to the zero-staked interval are permanently unclaimable. `rewardPerTokenStored` is frozen at the value it held when `totalKernelStaked` last became zero. When new stakers arrive after `finishAt`, `rewardRate` is no longer applied. The tokens remain in the contract balance but can never be distributed or recovered.

## Likelihood Explanation

The trigger requires no privileged role, oracle manipulation, or external compromise. Any ordinary staker exercising their normal right to withdraw can cause this. A single large staker holding the majority of `totalKernelStaked` can unilaterally trigger it by calling `initiateWithdrawal` for their full balance. The scenario is especially plausible during long reward periods (e.g., 30 days) when market conditions change and rational stakers exit after a price drop in the reward token.

## Recommendation

Add a check in `initiateWithdrawal` that, when `totalKernelStaked` would drop to zero mid-period, sweeps the remaining unallocated rewards (`rewardRate * (finishAt - block.timestamp)`) to a recoverable treasury address and resets `rewardRate` and `finishAt`:

```solidity
// After: totalKernelStaked -= _amount;
if (totalKernelStaked == 0 && block.timestamp < finishAt) {
    uint256 remainingRewards = rewardRate * (finishAt - block.timestamp);
    rewardRate = 0;
    finishAt = block.timestamp;
    if (remainingRewards > 0) {
        rewardsToken.safeTransfer(treasury, remainingRewards);
    }
}
```

Alternatively, prevent `totalKernelStaked` from reaching zero while a reward period is active by reverting in that case.

## Proof of Concept

1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 30 days`; `rewardRate = 1_000e18 / 30 days`.
2. Alice is the only staker: `balanceOf[Alice] = totalKernelStaked = 100e18`.
3. After 1 day, Alice calls `initiateWithdrawal(100e18)`.
   - `updateReward(Alice)` correctly snapshots 1 day of rewards into `rewards[Alice]`.
   - `totalKernelStaked` is set to `0`.
4. For the remaining 29 days, every call to `rewardPerToken()` returns the frozen `rewardPerTokenStored`; `rewardRate * 29 days ≈ 966e18` tokens accumulate as permanently unclaimable.
5. Alice calls `getReward()` and receives only her ~33e18 (1-day share).
6. No address can ever claim the remaining ~966e18 tokens; no admin rescue function exists in the contract. [6](#0-5) [7](#0-6)
