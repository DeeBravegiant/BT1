Audit Report

## Title
Missing `owner != address(0)` Validation Permanently Locks LP Tokens - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.addLiquidity` accepts `address(0)` as the `owner` argument without any guard. Because `removeLiquidity` enforces `msg.sender == owner` and `msg.sender` can never equal `address(0)`, any tokens deposited under an `address(0)` owner are permanently locked inside the pool. The periphery helper `MetricOmmPoolLiquidityAdder` carries this check, but the core pool is a standalone public contract that any caller can reach directly.

## Finding Description
`MetricOmmPool.addLiquidity` (lines 182–196) performs only two input checks before delegating to `LiquidityLib.addLiquidity`: it rejects an empty `binIdxs` array and a length mismatch between `binIdxs` and `shares`. There is no check that `owner != address(0)`.

Inside `LiquidityLib.addLiquidity`, the position key is computed at line 72 using the supplied `owner` (which may be `address(0)`), shares are credited to `positionBinShares[posKey]` at line 121, and `binTotalShares[binIdx]` is incremented at line 120. Tokens are then pulled from `msg.sender` via the modify-liquidity callback at lines 147–148. All three storage slots (`positionBinShares`, `binTotalShares`, `binState.token0/1BalanceScaled`) are permanently updated.

The withdrawal path at `removeLiquidity` line 206 enforces `if (msg.sender != owner) revert NotPositionOwner()`. Because `msg.sender` can never equal `address(0)` in the EVM, the shares credited to the zero-address position key can never be burned, and the deposited tokens are irrecoverable.

The periphery contract `MetricOmmPoolLiquidityAdder._validateOwner` (lines 247–249) does carry this guard, but the core pool is a public contract; any EOA or contract can call `addLiquidity` directly, bypassing the periphery entirely.

## Impact Explanation
Direct loss of user principal: tokens transferred into the pool via the modify-liquidity callback are irrecoverable. Additionally, `binTotalShares[binIdx]` is permanently inflated, which dilutes the share-to-token ratio for all subsequent LPs adding liquidity to the same bin (line 109–110 of `LiquidityLib.sol` uses `binTotalSharesVal` as the denominator). The locked tokens are never redistributed. This constitutes a direct loss of user principal and broken core pool functionality meeting Sherlock Medium/High thresholds.

## Likelihood Explanation
Low. The periphery `MetricOmmPoolLiquidityAdder` blocks this path for standard users. Exploitation requires a caller to interact with the core pool directly — either through a custom integration that omits the zero-address check, or through user/operator error. No privileged role is required; any address can call `addLiquidity` on the pool. The scenario is realistic for integrators building directly on top of the core pool.

## Recommendation
Add a zero-address guard at the top of `MetricOmmPool.addLiquidity`, mirroring the check already present in the periphery:

```solidity
function addLiquidity(
    address owner,
    ...
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
    if (owner == address(0)) revert InvalidPositionOwner();   // ← add this
    if (deltas.binIdxs.length == 0) return (0, 0);
    ...
}
```

Alternatively, place the guard inside `LiquidityLib.addLiquidity` so it is enforced regardless of which entry point is used.

## Proof of Concept
```solidity
contract PoC is IMetricOmmModifyLiquidityCallback {
    IERC20 token0; IERC20 token1; IMetricOmmPool pool;

    function metricOmmModifyLiquidityCallback(
        uint256 a0, uint256 a1, bytes calldata
    ) external {
        if (a0 > 0) token0.transfer(msg.sender, a0);
        if (a1 > 0) token1.transfer(msg.sender, a1);
    }

    function attack() external {
        int256[] memory bins   = new int256[](1); bins[0]   = 0;
        uint256[] memory shares = new uint256[](1); shares[0] = 1_000_000;
        LiquidityDelta memory d = LiquidityDelta({binIdxs: bins, shares: shares});

        // owner == address(0): no revert, tokens pulled, shares locked forever
        pool.addLiquidity(address(0), 0, d, "", "");

        // Attempt to recover: reverts with NotPositionOwner because msg.sender != address(0)
        pool.removeLiquidity(address(0), 0, d, "");
    }
}
```

After `attack()`: tokens transferred into the pool are unrecoverable, `binTotalShares[0]` is permanently inflated, and `binState.token0/1BalanceScaled` and `binTotals.scaledToken0/1` are permanently inflated by the deposited scaled amount.
