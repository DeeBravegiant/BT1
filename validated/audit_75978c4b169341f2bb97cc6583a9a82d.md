Audit Report

## Title
`instantWithdrawal` Drains `LRTUnstakingVault` Below `assetsCommitted`, Temporarily Freezing Queued Withdrawal Users - (File: contracts/LRTWithdrawalManager.sol / contracts/LRTUnstakingVault.sol)

## Summary
`LRTWithdrawalManager.instantWithdrawal()` gates vault outflows using a static, manually-set `queuedWithdrawalsBuffer` rather than the live `assetsCommitted[asset]` mapping. When instant withdrawals are enabled and the buffer is zero (the Solidity default) or set below the current committed amount, any rsETH holder can drain `LRTUnstakingVault` below the amount already promised to pending queued-withdrawal requests. This causes `unlockQueue()` to stall, temporarily freezing queued users' funds for at least one full EigenLayer withdrawal cycle (â‰¥7 days).

## Finding Description
`initiateWithdrawal()` transfers the caller's rsETH to the withdrawal manager and records the owed asset amount in `assetsCommitted[asset]` ( [1](#0-0) ). This mapping is the protocol's authoritative record of vault liquidity already spoken for; it is only decremented inside `unlockQueue()` when requests are actually processed ( [2](#0-1) ).

`unlockQueue()` determines processable capacity by reading the vault's raw balance via `unstakingVault.balanceOf(asset)` ( [3](#0-2) ) and breaks the processing loop the moment that balance falls below the next request's payout ( [4](#0-3) ).

`instantWithdrawal()` pulls directly from the vault, gated only by `getAssetsAvailableForInstantWithdrawal()` ( [5](#0-4) ). That function subtracts the static `queuedWithdrawalsBuffer[asset]` â€” not `assetsCommitted[asset]` â€” from the vault balance ( [6](#0-5) ). The buffer defaults to `0` and is set manually by operators via `setQueuedWithdrawalsBuffer()` ( [7](#0-6) ). There is no on-chain enforcement that `queuedWithdrawalsBuffer[asset] >= assetsCommitted[asset]`, so instant withdrawals can consume vault liquidity already promised to queued users without any correction to `assetsCommitted`.

## Impact Explanation
**Medium â€” Temporary freezing of funds.** Queued-withdrawal users transfer their rsETH to the withdrawal manager at `initiateWithdrawal()` time and cannot reclaim it. If instant withdrawals drain the vault below `assetsCommitted`, `unlockQueue()` cannot advance the queue. Affected users cannot call `completeWithdrawal()` (their request remains locked) and must wait for operators to complete a new EigenLayer withdrawal cycle (â‰¥7 days) to replenish the vault. Funds are not permanently lost but are frozen for an indeterminate period beyond the promised withdrawal delay.

## Likelihood Explanation
**Low.** Two conditions must hold simultaneously: (1) the operator has called `setInstantWithdrawalEnabled(asset, true)` â€” disabled by default; and (2) `queuedWithdrawalsBuffer[asset]` is zero (the default) or set below the current `assetsCommitted[asset]`. Both are realistic in normal operation: instant withdrawal is a documented protocol feature, and the buffer is a static value that operators must manually keep in sync with a continuously changing `assetsCommitted`. A lag period where the buffer is insufficient is expected during normal activity. The exploit itself requires no special privilege â€” any rsETH holder can call `instantWithdrawal()`.

## Recommendation
Replace the static `queuedWithdrawalsBuffer` lookup in `getAssetsAvailableForInstantWithdrawal()` with a dynamic read of `assetsCommitted` from `LRTWithdrawalManager`:

```solidity
// LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal
function getAssetsAvailableForInstantWithdrawal(address asset) external view returns (uint256) {
    uint256 vaultBalance = balanceOf(asset);
    uint256 committed = ILRTWithdrawalManager(withdrawalManager).assetsCommitted(asset);
    return committed >= vaultBalance ? 0 : vaultBalance - committed;
}
```

This ensures the vault always reserves exactly the amount already promised to queued-withdrawal users before permitting instant outflows.

## Proof of Concept
**Setup**: Instant withdrawal enabled for ETH; `queuedWithdrawalsBuffer[ETH]` = 0 (default); `LRTUnstakingVault` holds 100 ETH.

1. **Alice** calls `initiateWithdrawal(ETH, rsETHAmount)` where `getExpectedAssetAmount` returns 80 ETH.
   - `assetsCommitted[ETH]` â†’ 80 ETH. Alice's rsETH is locked in the withdrawal manager.
2. **Bob** (any rsETH holder) calls `instantWithdrawal(ETH, rsETHAmount2)` where `getExpectedAssetAmount` returns 80 ETH.
   - `getAssetsAvailableForInstantWithdrawal(ETH)` = `100 âˆ’ 0` = 100 ETH â†’ check passes.
   - Vault balance drops to 20 ETH.
3. Operator calls `unlockQueue(ETH, ...)`.
   - `totalAvailableAssets` = `unstakingVault.balanceOf(ETH)` = 20 ETH.
   - Alice's request requires 80 ETH â†’ `availableAssetAmount (20) < payoutAmount (80)` â†’ loop breaks immediately.
4. Alice cannot call `completeWithdrawal()`. Her funds are frozen until operators complete a new EigenLayer withdrawal cycle (â‰¥7 days).

**Foundry test plan**: Deploy `LRTWithdrawalManager` and `LRTUnstakingVault` on a local fork. Enable instant withdrawal for ETH, leave `queuedWithdrawalsBuffer` at 0. Fund the vault with 100 ETH. Call `initiateWithdrawal` as Alice (80 ETH equivalent), then `instantWithdrawal` as Bob (80 ETH equivalent). Assert `unstakingVault.balanceOf(ETH) < assetsCommitted[ETH]`. Call `unlockQueue` and assert Alice's request nonce remains in the locked state.
