Audit Report

## Title
No Staleness Check on Chainlink Price Feed Allows Stale Asset Prices to Be Used for rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt`, `answeredInRound`, and `roundId`, performing zero staleness or validity validation. This stale price flows directly into rsETH minting math via `LRTDepositPool.getRsETHAmountToMint()`, allowing any depositor to receive inflated rsETH when a Chainlink feed lags behind a real-world asset depeg, diluting all existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches the price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All four non-price return values (`roundId`, `startedAt`, `updatedAt`, `answeredInRound`) are silently discarded. There is no `block.timestamp - updatedAt <= maxAge` check, no `answeredInRound >= roundId` check, and no `price > 0` guard. The interface declares the full five-value return signature but the implementation ignores four of them. [2](#0-1) 

`LRTOracle.getAssetPrice()` delegates directly to this oracle with no additional validation: [3](#0-2) 

`LRTDepositPool.getRsETHAmountToMint()` uses this price in the minting formula:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [4](#0-3) 

`lrtOracle.rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is called externally — it is not recomputed on-the-fly during deposits. If a supported LST depegs on secondary markets but the Chainlink feed has not yet updated (still within its heartbeat interval or deviation threshold), `getAssetPrice(asset)` returns the stale inflated price while `rsETHPrice` reflects the last stored value. The attacker deposits assets worth less than the oracle reports and receives rsETH computed at the inflated rate, directly diluting all existing rsETH holders.

The public `depositAsset()` function is the entry point, callable by any unprivileged user: [5](#0-4) 

The `_beforeDeposit` path has no price-freshness guard — only a deposit amount minimum and a deposit cap check: [6](#0-5) 

The `pricePercentageLimit` downside-protection in `_updateRsETHPrice()` only triggers when `updateRSETHPrice()` is called and only if the Chainlink feed has already reflected the depeg — it provides no protection against a stale feed during live deposits. [7](#0-6) 

## Impact Explanation
When a supported LST depegs but the Chainlink feed lags, any depositor can mint rsETH at the stale inflated price. The deposited assets back fewer ETH than the minted rsETH represents, reducing the backing ratio of all outstanding rsETH. Existing holders suffer a proportional loss of value. Sustained exploitation can push the protocol toward insolvency. This matches **High — Theft of unclaimed yield** (dilution of existing holders' accrued value) and is on the boundary of **Critical — Protocol insolvency** under repeated or large-scale exploitation.

## Likelihood Explanation
Chainlink LST/ETH feeds on Ethereum mainnet have heartbeat intervals of 1–24 hours and deviation thresholds of 0.5–1%. During any period where an asset's price moves within the deviation band or network congestion delays keeper updates, the feed can be stale for hours. No special role or privilege is required — any address can call `depositAsset()`. The window of exposure is the entire duration between Chainlink updates, which is a routine operating condition.

## Recommendation
Add staleness and validity checks in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= maxStaleness[asset], "Stale price");
```

`maxStaleness` should be configurable per asset by the admin role (e.g., 3600 seconds for ETH/USD, up to 86400 seconds for less-liquid LST feeds), mirroring the partial fix already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()`. [8](#0-7) 

## Proof of Concept
1. stETH/ETH Chainlink feed last updated 20 hours ago (within its 24-hour heartbeat; no update triggered). stETH has depegged 5% on secondary markets due to a slashing event, but the on-chain feed still reports the pre-depeg price (e.g., `1.0e18`).
2. `rsETHPrice` was last updated when stETH was at `1.0e18`, so it correctly reflects the old TVL.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
4. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.0e18`.
5. `rsethAmountToMint = (100e18 * 1.0e18) / rsETHPrice` — attacker receives rsETH as if depositing 100 ETH worth of stETH, but the actual market value is only 95 ETH.
6. Attacker sells or redeems rsETH, extracting ~5 ETH of value from existing holders.
7. Repeat until the deposit cap is reached or the Chainlink feed updates.

**Foundry fork test outline:**
- Fork mainnet at a block where a stETH/ETH feed update is overdue.
- Mock `latestRoundData()` to return a price with `updatedAt = block.timestamp - 23 hours`.
- Call `depositAsset` and assert `rsethAmountToMint > (depositAmount * trueMarketPrice) / rsETHPrice`.
