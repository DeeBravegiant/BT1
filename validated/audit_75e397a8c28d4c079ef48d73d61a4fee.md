Audit Report

## Title
Stale Rate in `CrossChainRateReceiver.getRate()` Enables Over-Minting of wrsETH — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness check. All pool contracts call this oracle unconditionally during `deposit`. If the LayerZero rate update pipeline stalls, the stale (lower) rate causes `deposit` to over-mint wrsETH relative to the true current rate, breaking the backing invariant and leading to protocol insolvency.

## Finding Description

`CrossChainRateReceiver` stores the last received rate and the timestamp it was received: [1](#0-0) 

`lzReceive` updates both fields when a LZ message arrives: [2](#0-1) 

`getRate()` returns `rate` with **no staleness check**: [3](#0-2) 

Every pool variant delegates its oracle call to this function without any freshness guard. In `RSETHPoolV2`: [4](#0-3) 

The minting math in `viewSwapRsETHAmountAndFee` divides by the oracle rate directly: [5](#0-4) 

The same pattern is present in `RSETHPoolV3`: [6](#0-5) 

And in `RSETHPoolV2ExternalBridge`: [7](#0-6) 

The `updateRate()` function on the provider side is permissionless but requires the caller to supply ETH for LZ fees: [8](#0-7) 

There is no on-chain circuit-breaker, no heartbeat requirement, and no admin alert. The `lastUpdated` field is stored but never read by `getRate()` or any pool contract.

## Impact Explanation

rsETH is a liquid staking token whose ETH-denominated rate monotonically increases over time. A stale rate is therefore always **lower** than the current true rate. Because `rsETHAmount = amountAfterFee * 1e18 / staleRate`, a lower denominator produces a **larger** wrsETH mint. Every depositor during the stale window receives more wrsETH than the deposited ETH can back at the true rate.

This directly causes **protocol insolvency** (a Critical allowed impact). The secondary path — `rate == 0` before the first LZ message is ever delivered — causes a division-by-zero revert on every `deposit` call, constituting **temporary freezing of funds** (a Medium/Critical allowed impact).

## Likelihood Explanation

The off-chain keeper calling `updateRate()` must supply ETH for LZ fees on every call. Infrastructure failure, key loss, or budget exhaustion silently ages the stored rate with no on-chain detection. Any unprivileged depositor can trigger the over-mint simply by calling `deposit()` while the rate is stale — no attacker capability is required beyond being a normal user. The longer the gap, the larger the over-mint per deposit. This is a realistic operational failure mode.

## Recommendation

Add a `MAX_STALENESS` constant and enforce it in `getRate()`:

```solidity
// CrossChainRateReceiver.sol
uint256 public constant MAX_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate is stale");
    return rate;
}
```

Alternatively, enforce the check at the pool level inside `viewSwapRsETHAmountAndFee`, or expose `lastUpdated` through the `IOracle` interface so pools can validate freshness before minting.

## Proof of Concept

```solidity
// Local fork test (no mainnet):
// 1. Deploy CrossChainRateReceiver (RSETHRateReceiver) with a mock LZ endpoint.
// 2. Simulate one lzReceive call setting rate = 1.05e18, lastUpdated = block.timestamp.
// 3. Warp block.timestamp forward by 30 days.
// 4. Call getRate() — returns 1.05e18 (stale, no revert).
// 5. Deploy RSETHPoolV2 pointing to the receiver as rsETHOracle.
// 6. Call pool.deposit{value: 1 ether}("ref").
//    → rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.952e18  (stale, lower rate)
// 7. Simulate fresh lzReceive setting rate = 1.10e18 (true current rate).
// 8. Repeat deposit{value: 1 ether}.
//    → rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909e18  (correct)
// 9. Assert: minted amount in step 6 > minted amount in step 8.
//    Difference (≈ 0.043e18 wrsETH per ETH) is unbacked — protocol insolvency.
//
// Zero-rate path:
// 10. Deploy fresh receiver (rate == 0, no lzReceive yet).
// 11. Call pool.deposit{value: 1 ether} → reverts with division-by-zero.
//     All deposits are frozen until a valid LZ message arrives.
```
