### Title
Failed Receipt Logs Processed as Valid NEP-141 FT Events in Rosetta RPC — (File: `chain/rosetta-rpc/src/adapters/nep141.rs`)

### Summary

The Rosetta RPC's NEP-141 event collection pipeline processes `EVENT_JSON:` logs from **all** receipt execution outcomes, including those with `ExecutionStatusView::Failure`, without checking the outcome's execution status. An unprivileged user can deploy a contract that emits a well-formed `ft_transfer` NEP-141 event log and then causes the receipt to revert, injecting phantom fungible-token transfer operations into the Rosetta `/block` response.

### Finding Description

In `ExecutionToReceipts::for_block`, all execution outcomes for a block are collected without any status filter:

```rust

    .map_err(...)?
    .into_iter()
    .flat_map(|(_k, v)| v)
    .collect();                          // includes Failure outcomes

    &execution_outcomes,                 // passed unfiltered

    currencies,

`collect_nep141_events` then iterates over every outcome and calls `extract_events`, which reads `outcome.outcome.logs` with no check on `outcome.outcome.status`:

```rust

Note that the `statuses` map is built correctly (it records `Failure` for failed outcomes), but it is only used later for `TransactionMetadata` — it is **not** consulted before `collect_nep141_events` runs. The event collection and the status recording are two separate passes over `execution_outcomes`, and the event pass has no guard. [4](#0-3) 

### Impact Explanation

The corrupted value is the `FungibleTokenEvent` list embedded in the Rosetta `/block` response. Exchanges and indexers that use the Rosetta Data API for NEP-141 balance reconciliation will observe phantom `ft_transfer` operations that never actually moved tokens on-chain. This can cause:

- Incorrect credit/debit entries in exchange accounting systems that rely on Rosetta for balance reconciliation.
- False positive transfer events in indexers, polluting historical FT transfer records.
- Potential for targeted manipulation: an attacker can craft a failing receipt that appears to transfer an arbitrary amount from any `old_owner_id` to any `new_owner_id`, as long as the contract is registered in the Rosetta node's `currencies` list.

The `FungibleTokenEvent.status` field is set to `"FAILURE"` (via `get_status`), but the event is still emitted into the operations list of the Rosetta transaction, meaning consumers that do not filter on `status` will treat it as a real transfer. [5](#0-4) 

### Likelihood Explanation

Any unprivileged NEAR account can deploy a smart contract. The contract only needs to:
1. Emit a log line starting with `EVENT_JSON:` containing a valid NEP-141 `ft_transfer` payload.
2. Panic (or otherwise revert) after emitting the log.

NEAR's runtime records logs even for failed receipts — the `ExecutionOutcome.logs` field is populated regardless of the final `ExecutionStatus`. This is by design for debugging, but the Rosetta RPC does not account for it. The attack requires no special privileges, no validator access, and no coordination. It is repeatable at will for any block.

### Recommendation

In `collect_nep141_events` (or in its caller before passing `execution_outcomes`), skip any outcome whose status is not a success variant:

```rust
for outcome in receipt_execution_outcomes {
    match &outcome.outcome.status {
        ExecutionStatusView::SuccessValue(_)
        | ExecutionStatusView::SuccessReceiptId(_) => {}
        _ => continue,   // skip failed/unknown outcomes
    }
    let events = extract_events(outcome);
    ...
}
```

Alternatively, filter `execution_outcomes` before passing to `collect_nep141_events` in `ExecutionToReceipts::for_block`.

### Proof of Concept

1. Deploy a contract on a NEAR network whose Rosetta RPC is configured with a `currencies` entry for contract `token.near`:
   ```rust
   pub fn exploit(&self) {
       near_sdk::env::log_str(
           r#"EVENT_JSON:{"standard":"nep141","version":"1.0.0","event":"ft_transfer","data":[{"old_owner_id":"victim.near","new_owner_id":"attacker.near","amount":"1000000000000000000000000"}]}"#
       );
       panic!("revert");   // receipt fails
   }
   ```
2. Call `exploit()` on the deployed contract.
3. Query the Rosetta `/block` endpoint for the block containing that receipt.
4. Observe that the response includes a `FungibleTokenEvent` with `old_owner_id=victim.near`, `new_owner_id=attacker.near`, `amount=1000000000000000000000000`, even though the receipt failed and no tokens moved. [6](#0-5) [7](#0-6)

pub(crate) fn extract_events(
    execution_outcome: &ExecutionOutcomeWithIdView,
) -> Vec<crate::models::Nep141Event> {
    let prefix = "EVENT_JSON:";
    execution_outcome
        .outcome
        .logs
        .iter()
        .filter_map(|untrimmed_log| {
            let log = untrimmed_log.trim();
            if !log.starts_with(prefix) {
                return None;
            }

            match serde_json::from_str::<'_, crate::models::Nep141Event>(log[prefix.len()..].trim())
            {
                Ok(result) => Some(result),
                Err(_err) => None,
            }
        })
        .collect()
}
