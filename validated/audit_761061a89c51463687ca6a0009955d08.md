### Title
`MockAttestation::Valid` Permanently Bypasses TEE Attestation Verification for Any Participant - (File: `crates/mpc-attestation/src/attestation.rs`)

### Summary
The `MockAttestation::Valid` variant unconditionally passes all TEE attestation verification checks — analogous to a zero-address signer that trivially satisfies every comparison. Any registered participant can submit this mock attestation via the public `submit_participant_info` endpoint. Once stored, it permanently passes `re_verify`, meaning the participant can never be evicted by `verify_tee`. This breaks the TEE security invariant that gates all attested-participant operations (`respond`, `vote_pk`, `vote_reshared`, etc.).

### Finding Description

`MockAttestation::Valid.verify_constraints` in `crates/mpc-attestation/src/attestation.rs` is:

```rust
MockAttestation::Valid => Ok(()),
```

It returns `Ok(())` unconditionally, ignoring every policy input: allowed image hashes, launcher compose hashes, OS measurements, and expiry timestamps. [1](#0-0) 

`TeeState::add_participant` in `crates/contract/src/tee/tee_state.rs` calls `attestation.verify_locally(...)` with no guard that rejects mock variants in production. The call succeeds and stores `VerifiedAttestation::Mock(MockAttestation::Valid)` in `stored_attestations`. [2](#0-1) 

`reverify_participants` (called by `verify_tee` → `reverify_and_cleanup_participants`) calls `re_verify` on the stored attestation. For `MockAttestation::Valid`, `re_verify` always returns `Ok(())` regardless of the current allowlists or measurements, so the entry is never removed. [3](#0-2) 

The sandbox test `test_submit_participant_info_succeeds_with_mock_attestation` confirms that `MockAttestation::Valid` is accepted by the live contract API: [4](#0-3) 

`is_caller_an_attested_participant` — the gate for `respond`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, and `start_reshare_instance` — only checks that a stored attestation exists with matching `account_id` and `account_public_key`. It does **not** call `re_verify`; it accepts any stored entry, including a permanently-valid mock. [5](#0-4) 

### Impact Explanation

A Byzantine participant (strictly below signing threshold) can:

1. Call `submit_participant_info` with `Attestation::Mock(MockAttestation::Valid)` — accepted with no production guard.
2. Operate entirely outside a genuine TDX enclave while passing every `is_caller_an_attested_participant` check.
3. Never be evicted by `verify_tee` because `MockAttestation::Valid` always passes `re_verify` regardless of image-hash rotations, launcher-hash changes, or measurement updates.

This permanently breaks the production safety invariant that all active MPC participants must run inside an approved TEE. The participant's key share is unprotected, and the governance mechanism designed to enforce TEE compliance (`verify_tee`) is rendered ineffective for that participant. This satisfies the **Medium** allowed impact: participant-state manipulation that breaks production safety/accounting invariants.

### Likelihood Explanation

The attack path requires only that the caller be a registered participant (governance-approved). No threshold collusion, no privileged operator access, and no external dependency is needed. The attacker simply serializes `Attestation::Mock(MockAttestation::Valid)` — a publicly documented variant — and calls `submit_participant_info`. The sandbox test already demonstrates this succeeds against the real contract.

### Recommendation

1. Add a production guard in `TeeState::add_participant` (or in `submit_participant_info`) that rejects any `Attestation::Mock(_)` variant when the contract is not in a test/devnet context. A compile-time feature flag or an explicit runtime check (`if matches!(attestation, Attestation::Mock(_)) { return Err(...) }`) would close the path.
2. Similarly, `re_verify` for `VerifiedAttestation::Mock(MockAttestation::Valid)` should fail in production rather than unconditionally returning `Ok(())`, so that existing mock entries are evicted on the next `verify_tee` call after the guard is deployed.

### Proof of Concept

```
1. Attacker is a registered participant (account: attacker.near).
2. Attacker calls:
     submit_participant_info(
         proposed_participant_attestation = Attestation::Mock(MockAttestation::Valid),
         tls_public_key = <attacker's real TLS key>,
     )
   with attached deposit to cover storage.
3. Contract calls add_participant → attestation.verify_locally(...) → MockAttestation::Valid.verify() → Ok(()) unconditionally.
4. stored_attestations[attacker_tls_key] = NodeAttestation { ..., verified_attestation: Mock(Valid) }.
5. Attacker calls respond(...) — is_caller_an_attested_participant passes (stored entry exists, keys match).
6. Governance rotates image hashes; verify_tee is called.
   reverify_participants → re_verify → MockAttestation::Valid → Ok(()) — attacker is NOT evicted.

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
