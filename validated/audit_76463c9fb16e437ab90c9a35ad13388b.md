### Title
`verify_tee` Uses GovernanceThreshold Instead of ReconstructionThreshold to Prematurely Block Signing Requests - (`crates/contract/src/lib.rs`)

### Summary

The `verify_tee` function incorrectly uses the GovernanceThreshold (a governance-weighted value) instead of the per-domain ReconstructionThreshold (the actual signing threshold) to determine whether to set `accept_requests = false`. This can cause all signing requests to be permanently blocked even when the remaining TEE-valid participants are sufficient to produce valid threshold signatures — an exact structural analog to M-8's weighted-value check prematurely triggering a blocking state.

### Finding Description

In `verify_tee` (`crates/contract/src/lib.rs:1693–1770`), when some participants have invalid or expired TEE attestations, the function checks whether the remaining valid participants satisfy `validate_governance_against_reconstruction`. This check fails — and sets `accept_requests = false` — when `remaining < GovernanceThreshold`. However, the actual ability to produce threshold signatures depends on the per-domain ReconstructionThreshold, not the GovernanceThreshold.

The protocol enforces `GovernanceThreshold >= max(ReconstructionThreshold)` across all domains (see `validate_governance_against_reconstruction` at `crates/contract/src/primitives/thresholds.rs:92–108`). This means there is always a valid configuration range where:

- `ReconstructionThreshold <= remaining < GovernanceThreshold`
- The network **can** still produce valid threshold signatures
- But `accept_requests = false` is incorrectly set

The problematic check is:

```rust
// crates/contract/src/lib.rs:1727-1738
if let Err(err) = ThresholdParameters::validate_governance_against_reconstruction(
    u64::try_from(remaining).expect("participant count fits in u64"),
    current_params.threshold(),   // GovernanceThreshold — not the signing threshold
    max_reconstruction_threshold,
) {
    self.accept_requests = false;
    return Ok(false);
}
``` [1](#0-0) 

When `accept_requests = false`, all user-facing request paths are blocked:

- `sign()` panics with `TeeError::TeeValidationFailed` [2](#0-1) 
- `respond()` returns `TeeError::TeeValidationFailed` [3](#0-2) 
- `respond_ckd()` and `respond_verify_foreign_tx()` return the same error [4](#0-3) 

Critically, `verify_tee` returns `Ok(false)` **without triggering resharing**, leaving the network in a state where signing is blocked and no automatic recovery is initiated. The log message explicitly acknowledges this requires "manual intervention" — even though the network is still capable of signing. [5](#0-4) 

The GovernanceThreshold is validated to be at least 60% of participants (`governance_threshold_lower_relative_bound`), while the ReconstructionThreshold can be as low as 2 (the absolute minimum). [6](#0-5) 

### Impact Explanation

**Medium.** This is a request-lifecycle manipulation that breaks the production safety invariant: the contract stops accepting signing requests even though the MPC network has enough TEE-valid participants to produce valid threshold signatures. All pending and new signing requests are blocked until manual intervention restores the state. This matches the allowed Medium impact: *"request-lifecycle... manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

The `accept_requests` flag is a global gate on all signing activity. [7](#0-6) 

### Likelihood Explanation

**Moderate.** The condition requires:

1. A domain with `ReconstructionThreshold < GovernanceThreshold` — a common and valid configuration (e.g., GovernanceThreshold = 4, ReconstructionThreshold = 3 with 5 participants). The protocol explicitly allows and enforces this gap. [8](#0-7) 
2. Enough participants' TEE attestations to expire such that `remaining < GovernanceThreshold` but `remaining >= ReconstructionThreshold`. TEE attestations expire naturally via `reverify_and_cleanup_participants`. [9](#0-8) 
3. Any participant to call `verify_tee` — a routine operation callable by any current participant. [10](#0-9) 

A Byzantine participant strictly below the signing threshold can deliberately let their attestation expire to trigger this condition. No privileged access or above-threshold collusion is required.

### Recommendation

Change the blocking condition in `verify_tee` to use the ReconstructionThreshold instead of the GovernanceThreshold. Specifically, `accept_requests = false` should only be set when `remaining < max(ReconstructionThreshold)` across all domains — i.e., when the network genuinely cannot produce signatures. The GovernanceThreshold check should only gate resharing eligibility, not signing capability.

```rust
// Proposed fix: block signing only when signing is actually impossible
let max_recon = max_reconstruction_threshold(running_state.domains.domains());
if let Some(recon) = max_recon {
    if (remaining as u64) < recon.inner() {
        self.accept_requests = false;
        return Ok(false);
    }
}
// GovernanceThreshold check: only gates resharing, not signing
```

### Proof of Concept

1. Deploy contract with 5 participants, GovernanceThreshold = 4, ReconstructionThreshold = 3 (valid: 4 ≥ 3, 4 ≥ 60% of 5).
2. Let 2 participants' TEE attestations expire (remaining = 3).
3. Any participant calls `verify_tee`.
4. `reverify_and_cleanup_participants` returns `Partial { participants_with_valid_attestation: [3 participants] }`. [11](#0-10) 
5. `validate_governance_against_reconstruction(3, GovernanceThreshold=4, max_reconstruction=3)` → Error (`4 > 3`). [12](#0-11) 
6. `accept_requests = false` is set; `verify_tee` returns `Ok(false)` without triggering resharing. [13](#0-12) 
7. All `sign()` calls now panic with "TEE validation failed".
8. But 3 participants with valid TEE can still produce signatures (ReconstructionThreshold = 3 is met).
9. The network is stuck: signing blocked, no resharing triggered, manual intervention required — even though the signing threshold is satisfied.

        if k.value() > n_shares {

                required: lower_relative_bound,
                found: k.value(),
            }
            .into());

            return Err(InvalidThreshold::MaxRelRequirementFailed {
                max: upper_relative_bound,
                found: k.value(),
            }
            .into());
        }
        Ok(())
    }

    pub fn validate_governance_against_reconstruction(
        num_participants: u64,
        governance: Threshold,
        max_reconstruction_threshold: Option<ReconstructionThreshold>,
    ) -> Result<(), Error> {
        Self::validate_threshold(num_participants, governance)?;
        if let Some(max_reconstruction_threshold) = max_reconstruction_threshold
            && governance.value() < max_reconstruction_threshold.inner()
        {
            return Err(InvalidThreshold::BelowReconstructionThreshold {
                reconstruction_threshold: max_reconstruction_threshold.inner(),
                governance_threshold: governance.value(),
            }
            .into());
        }
        Ok(())
    }
