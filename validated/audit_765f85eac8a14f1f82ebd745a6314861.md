### Title
`GatewayDynamicConfig` (`native_classes_whitelist`) Cannot Be Retrieved from `ConfigManager`, Causing Permanent Gateway/Batcher Divergence After Any Dynamic Config Update - (File: `crates/apollo_config_manager_types/src/communication.rs`)

### Summary

`NodeDynamicConfig` stores a `gateway_dynamic_config` field (containing `native_classes_whitelist`) that is updated by `ConfigManagerRunner` whenever the config file changes. Every other component — batcher, class manager, consensus, context, http server, mempool, state sync, staking manager — has a corresponding `get_*_dynamic_config` method in `ConfigManagerClient` and polls for updates on each request. The gateway has no such method: `GetGatewayDynamicConfig` is absent from `ConfigManagerRequest`, `ConfigManagerResponse`, and `ConfigManagerClient`. After any live config update, the batcher applies the new `native_classes_whitelist` on every block-building request, while the gateway is permanently frozen at the startup value. This creates a persistent divergence in which contract classes are executed natively vs. via the Cairo VM between the two components.

### Finding Description

`NodeDynamicConfig` declares `gateway_dynamic_config: Option<GatewayDynamicConfig>` alongside all other per-component dynamic configs. [1](#0-0) 

`NodeDynamicConfig::from(&SequencerNodeConfig)` correctly populates `gateway_dynamic_config` from `gateway_config.dynamic_config`. [2](#0-1) 

`ConfigManagerRunner::update_config` calls `set_node_dynamic_config` with the freshly loaded `NodeDynamicConfig` (including the updated `gateway_dynamic_config`) whenever the config file changes. [3](#0-2) 

`ConfigManager::set_node_dynamic_config` atomically replaces the stored `NodeDynamicConfig`, so the new `gateway_dynamic_config` is now live inside the manager. [4](#0-3) 

However, `ConfigManagerClient` exposes getters for every component **except** the gateway. There is no `get_gateway_dynamic_config` method, no `GetGatewayDynamicConfig` request variant, and no `GetGatewayDynamicConfig` response variant. [5](#0-4) [6](#0-5) 

The batcher, by contrast, calls `get_batcher_dynamic_config()` on **every** incoming request and immediately applies the result, so its `native_classes_whitelist` is always current. [7](#0-6) 

The `native_classes_whitelist` pointer is declared as shared between `batcher_config.dynamic_config.native_classes_whitelist` and `gateway_config.dynamic_config.native_classes_whitelist`, making the intent clear that both should track the same value. [8](#0-7) 

After a live config update (e.g., restricting native execution to `Limited([class_A])` to mitigate a native-execution bug), the batcher executes non-whitelisted classes via the Cairo VM while the gateway continues to execute them natively. The two components now apply different execution semantics to the same class hashes.

### Impact Explanation

`native_classes_whitelist` is consumed by `StatefulTransactionValidatorFactory::instantiate_validator` to construct a `StateReaderAndContractManager` for each gateway validation. [9](#0-8) 

The gateway runs the account's `__validate__` entry point through this manager. If the whitelist is updated to exclude a class because native execution produces wrong results for it (the primary operational reason to restrict the whitelist), the gateway continues to validate that class natively while the batcher executes it via VM. This divergence causes:

- **Gateway accepts a transaction the batcher will reject**: native execution incorrectly passes `__validate__` (e.g., wrong signature check, wrong nonce, wrong fee bound) → transaction enters the mempool and is streamed to the batcher → batcher re-executes with VM and gets the correct (failing) result → block building is disrupted.
- **Gateway rejects a valid transaction**: native execution incorrectly fails `__validate__` → valid transaction is silently dropped at the admission layer and never reaches the batcher.

Both outcomes match the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The trigger is an operator updating `native_classes_whitelist` in the config file — a documented, supported, and operationally expected action (the mainnet deployment already pins a specific class hash). [10](#0-9) 

The divergence is permanent (not a transient race): the gateway never recovers the new value without a restart. The scenario is realistic: the whitelist exists precisely to allow operators to disable native execution for a class when a bug is found, which is exactly when the execution-semantic divergence between gateway and batcher becomes security-relevant.

### Recommendation

Add `get_gateway_dynamic_config` to `ConfigManagerClient`, `ConfigManagerRequest`, `ConfigManagerResponse`, and `ConfigManager`, mirroring the pattern used for every other component. The gateway's `handle_request` (or equivalent per-request hook) should call `get_gateway_dynamic_config()` and apply the result before processing each transaction, exactly as the batcher does with `get_batcher_dynamic_config()`. [5](#0-4) [11](#0-10) 

### Proof of Concept

1. Start the sequencer with `native_classes_whitelist = "All"` (default). Both gateway and batcher execute all classes natively.
2. Suppose class `0xABCD` has a known native-execution bug that causes `__validate__` to return success even for an invalid signature.
3. Operator updates the config file: `native_classes_whitelist = '["0x1234"]'` (excludes `0xABCD`).
4. `ConfigManagerRunner` detects the change and calls `set_node_dynamic_config` with the new `NodeDynamicConfig` (including `gateway_dynamic_config.native_classes_whitelist = Limited([0x1234])`).
5. On the next batcher request, `get_batcher_dynamic_config()` returns the new config; the batcher now executes `0xABCD` via VM.
6. The gateway has no `get_gateway_dynamic_config()` call; its `native_classes_whitelist` remains `All`. It continues to execute `0xABCD` natively.
7. An attacker submits a transaction from an account whose class is `0xABCD` with an invalid signature. The gateway's native `__validate__` incorrectly returns success → transaction is admitted to the mempool. The batcher's VM `__validate__` correctly returns failure → the transaction cannot be included in any block, but it has already consumed gateway and mempool resources and may displace valid transactions.

    pub(crate) async fn set_node_dynamic_config(
        &self,
        node_dynamic_config: NodeDynamicConfig,
    ) -> ConfigManagerResult<()> {
        info!("ConfigManager: updating node dynamic config");
        if let Err(errors) = node_dynamic_config.validate() {
            error!(
                "ConfigManager: dynamic config update rejected: {errors}. Keeping previous config."
            );
            return Err(ConfigManagerError::InvalidConfig(errors.to_string()));
        }
        let mut config = self.latest_node_dynamic_config.write().await;
        *config = node_dynamic_config;
        Ok(())
    }
