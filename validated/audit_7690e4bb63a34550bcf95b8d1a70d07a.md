### Title
Attacker-Controlled `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay â€” (File: crates/contract/src/lib.rs)

---

### Summary

`respond_verify_foreign_tx` accepts a caller-supplied `response.payload_hash` and only checks that the submitted signature is valid over that hash. It never verifies that `payload_hash` is the correct hash of `ForeignTxSignPayload{request, extracted_values}` for the *pending* request. A single malicious attested participant can replay any previously observed on-chain `(payload_hash, signature)` pair against any pending foreign-transaction verification request, causing the contract to resolve that request with a forged attestation.

---

### Finding Description

In `respond_verify_foreign_tx`, the payload hash used for signature verification is taken directly from the caller-supplied response DTO, not derived from the pending request:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

By contrast, the regular `respond` function correctly derives the payload from the request itself, making it impossible to substitute a foreign hash:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
.is_ok()
``` [2](#0-1) 

The `VerifyForeignTransactionResponse` struct carries only `payload_hash` and `signature` â€” no `extracted_values` are included, so the contract has no material with which to reconstruct and verify the expected hash:

```rust
pub struct VerifyForeignTransactionResponse {
    pub payload_hash: Hash256,
    pub signature: SignatureResponse,
}
``` [3](#0-2) 

The correct hash is defined off-chain as `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))`, where `request` is the `ForeignChainRpcRequest` that is part of the pending map key: [4](#0-3) 

Because every `respond_verify_foreign_tx` call is a public on-chain transaction, any `(payload_hash_X, signature_X)` pair produced for a past transaction X is permanently observable by any network participant. The pending-request map is keyed by `VerifyForeignTransactionRequest` (which includes the foreign-chain RPC request), but the contract never checks that the submitted `payload_hash` is consistent with that key: [5](#0-4) 

**Attack steps (single Byzantine participant, strictly below signing threshold):**

1. Attacker observes a legitimate on-chain response `(payload_hash_X, signature_X)` for a past transaction X â€” no threshold collusion required, this is public data.
2. Attacker (an attested participant) submits a new `verify_foreign_transaction` request for a different transaction Y, paying the 1 yoctoNEAR minimum deposit.
3. Attacker calls `respond_verify_foreign_tx(request_Y, {payload_hash: payload_hash_X, signature: signature_X})`.
4. The contract checks `assert_caller_is_attested_participant_and_protocol_active()` â€” passes, attacker is a participant.
5. The contract checks `verify_ecdsa_signature(signature_X, payload_hash_X, root_pk)` â€” passes, the old signature is genuinely valid.
6. The contract calls `resolve_yields_for(&mut self.pending_verify_foreign_tx_requests, &request_Y, serialized_response)`, resolving the user's yield with the forged response.
