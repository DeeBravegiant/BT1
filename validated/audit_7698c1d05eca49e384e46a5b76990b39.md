### Title
Deposit Funds Permanently Locked When `do_update` Clears All Pending Proposals Without Refund - (File: `crates/contract/src/update.rs`)

### Summary

When a contract update is executed via `do_update`, all competing pending update proposals are cleared from storage, but the NEAR deposits paid by their proposers are never refunded. Those funds are permanently locked in the contract with no recovery path.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a deposit proportional to the bytes consumed by the proposed update (code or config), computed by `ProposedUpdates::required_deposit`. [1](#0-0) 

The deposit is accepted and the proposal is stored: [2](#0-1) 

When any update reaches threshold votes and `do_update` is called, it removes the winning entry and then unconditionally clears **all** remaining entries and all votes: [3](#0-2) 

The `entries.clear()` and `vote_by_participant.clear()` calls at lines 199–200 destroy every competing proposal's storage record. However, there is no corresponding refund of the deposits those proposers attached. The `UpdateEntry` struct stores `bytes_used` (the basis for the deposit calculation) but that value is discarded along with the entry: [4](#0-3) 

The `ProposedUpdates` struct has no deposit ledger — deposits are accepted in `lib.rs` but the `update.rs` module has no mechanism to return them: [5](#0-4) 

This is structurally identical to the `safeRagequit` pattern: a "push" exit path (here: `do_update` clearing all proposals) silently discards funds belonging to other participants, with no pull-based recovery mechanism.

The only refund path in the contract is the excess-deposit refund in `require_deposit`, which uses a detached fire-and-forget promise and only covers the difference above the minimum — it does not cover the minimum deposit itself: [6](#0-5) 

### Impact Explanation

Every participant who proposed a competing update loses their full deposit permanently when any other update is executed. The funds remain in the contract balance with no on-chain mechanism to claim them. This breaks the production accounting invariant that deposits are held only for the duration of storage use and must be returned when that storage is freed. Impact: **permanent freezing of funds** controlled by the MPC contract.

### Likelihood Explanation

Contract upgrades are a routine governance operation. Every upgrade cycle that has more than one competing proposal (a normal governance scenario where participants disagree on which update to apply) will silently destroy the deposits of all losing proposers. No attacker capability beyond being a participant and proposing an update is required — the loss is triggered by the normal execution of any other proposal reaching threshold.

### Recommendation

`do_update` should iterate over all entries being cleared, compute the refund owed to each proposer (using the stored `bytes_used` and `env::storage_byte_cost()`), and schedule `Promise::new(proposer).transfer(refund)` for each before clearing. This requires either storing the proposer's `AccountId` in `UpdateEntry`, or maintaining a separate deposit ledger keyed by `UpdateId`. The fix mirrors the "pull pattern" resolution described in the original report: funds must be explicitly returned to their owners rather than silently discarded.

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary. Required deposit ≈ `500_000 * storage_byte_cost`. Deposit is accepted and stored.
2. Participant B calls `propose_update` with a different 500 KB binary. Same deposit accepted.
3. Threshold participants vote for B's proposal and call `vote_update` → `do_update(B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry is destroyed.
5. A's deposit (potentially several NEAR) remains in the contract balance forever. No function exists to reclaim it. [3](#0-2) [7](#0-6)

    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }
