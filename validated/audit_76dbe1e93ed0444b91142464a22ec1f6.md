Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Incoming Amount in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its limit check, comparing only `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. This allows any caller to deposit an arbitrarily large amount of ETH in a single transaction as long as the running total has not already exceeded the cap, rendering the ETH deposit limit entirely unenforceable.

## Finding Description
`_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682) contains an asymmetric guard:

```solidity
// ETH branch — amount is ignored
return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));

// LST branch — amount is correctly included
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

The public `depositETH` function (lines 76–93) is `payable` and passes `msg.value` as `depositAmount` to `_beforeDeposit` (lines 648–670), which calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` and reverts only when the function returns `true`. Because the ETH branch never adds `amount` to `totalAssetDeposits`, the check only fires when a *prior* deposit has already pushed the total past the limit. Any deposit arriving while `totalAssetDeposits ≤ limit` passes unconditionally, regardless of how large `msg.value` is. No existing guard compensates for this omission.

## Impact Explanation
The deposit limit is the protocol's primary risk-management control bounding EigenLayer slashing and liquidity exposure. With the check broken, the ETH cap is never enforced on the incoming deposit, so the protocol can accumulate ETH far beyond the intended ceiling. The direct, concrete impact is that the contract fails to deliver its promised deposit-cap invariant. This matches the allowed Low impact: **"Contract fails to deliver promised returns, but doesn't lose value."**

## Likelihood Explanation
`depositETH` is a public, permissionless, payable function. No role, privilege, front-running, or multi-step setup is required. Any depositor with sufficient ETH can trigger the bypass in a single transaction at any time `totalAssetDeposits ≤ depositLimit`, which is the normal operating state of the protocol.

## Recommendation
Mirror the LST branch in the ETH branch by including `amount`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Set `depositLimitByAsset[ETH_TOKEN] = 1_000e18`; `getTotalAssetDeposits(ETH_TOKEN)` returns `0`.
2. Call `LRTDepositPool.depositETH{value: 100_000e18}(0, "")` from any EOA.
3. `_beforeDeposit` invokes `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 100_000e18)`.
4. ETH branch evaluates `0 > 1_000e18` → `false` → no revert.
5. `_mintRsETH` mints rsETH proportional to 100 000 ETH.
6. `getTotalAssetDeposits(ETH_TOKEN)` now returns `100_000e18`, 100× the configured limit.

Foundry test plan: deploy `LRTDepositPool` with a mock config returning `depositLimitByAsset = 1_000e18`, call `depositETH{value: 100_000e18}`, assert no revert and that `getTotalAssetDeposits` exceeds the limit. [1](#0-0) [2](#0-1) [3](#0-2)
