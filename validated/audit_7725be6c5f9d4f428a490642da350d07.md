Audit Report

## Title
`ChainlinkPriceOracle.getAssetPrice` Ignores Chainlink `latestRoundData` Validity Indicators, Enabling Stale-Price rsETH Minting - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice` calls `priceFeed.latestRoundData()` but silently discards every validity field, accepting stale or invalid prices without reversion. The stale price propagates directly into `LRTDepositPool.getRsETHAmountToMint`, causing any depositor to receive an incorrect rsETH amount. The sibling oracle `ChainlinkOracleForRSETHPoolCollateral` in the same repository already implements all three required guards, confirming the omission is unintentional.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice` reads only the raw `answer` field:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol line 52
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No check is performed for:
- `answeredInRound < roundId` (stale round)
- `updatedAt == 0` (incomplete round)
- `price <= 0` (invalid/negative answer)

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate` in the same codebase performs all three guards:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol lines 30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The stale price flows through the following confirmed call chain:

1. `LRTOracle.getAssetPrice` delegates to `IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)` [3](#0-2) 
2. `LRTOracle._getTotalEthInProtocol` accumulates per-asset ETH value using that price [4](#0-3) 
3. `LRTDepositPool.getRsETHAmountToMint` uses `lrtOracle.getAssetPrice(asset)` and `lrtOracle.rsETHPrice()` to compute rsETH to mint [5](#0-4) 
4. Both `depositETH` and `depositAsset` call `_beforeDeposit → getRsETHAmountToMint` then mint rsETH [6](#0-5) 

The `pricePercentageLimit` guard in `_updateRsETHPrice` does not mitigate this: it only fires when `updateRSETHPrice()` is called explicitly, not during deposits. During a deposit, `getAssetPrice` is called directly and the stale price is used without any threshold check. [7](#0-6) 

## Impact Explanation

When a Chainlink LST/ETH feed goes stale with an inflated price, every depositor calling `depositAsset` or `depositETH` receives more rsETH than the deposited assets are worth. This directly dilutes existing rsETH holders and, at scale, leads to **protocol insolvency** — the total rsETH supply becomes unbacked by real assets. If the stale price is deflated, depositors are under-minted rsETH, constituting a loss of promised returns. Both directions match allowed Critical/Low impacts; the inflated-price direction constitutes **Critical — protocol insolvency**.

## Likelihood Explanation

Chainlink feeds go stale during periods of low volatility, network congestion, or feed deprecation. The `answeredInRound < roundId` condition triggers whenever a round fails to complete. No privileged access is required; any user calling `depositAsset` or `depositETH` during a stale-price window triggers the issue. The condition is realistic and well-documented in production Chainlink deployments.

## Recommendation

Apply the same three guards already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Optionally add a configurable `stalePriceThreshold` (e.g., `block.timestamp - updatedAt > threshold`) to enforce a maximum age on the price.

## Proof of Concept

1. A Chainlink LST/ETH feed used by `ChainlinkPriceOracle` goes stale (e.g., `answeredInRound < roundId`, or `updatedAt` is 25+ hours in the past). The feed returns an inflated last-recorded price.
2. An attacker calls `LRTDepositPool.depositAsset(lstToken, amount, 0, "")`.
3. `_beforeDeposit` calls `getRsETHAmountToMint(lstToken, amount)`.
4. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(lstToken)`, which calls `ChainlinkPriceOracle.getAssetPrice`, which calls `priceFeed.latestRoundData()` and returns the stale inflated price without reverting.
5. The inflated asset price causes `getRsETHAmountToMint` to return a larger rsETH amount than the deposit warrants: `rsethAmountToMint = (amount * stalePriceInflated) / rsETHPrice`.
6. `_mintRsETH` mints the excess rsETH to the depositor.
7. The depositor redeems the excess rsETH for more underlying assets than they deposited, draining the protocol.

**Foundry fork test plan:**
- Fork mainnet at a block where a supported LST/ETH Chainlink feed has a stale round (`answeredInRound < roundId`).
- Deploy or use the existing `ChainlinkPriceOracle` pointed at that feed.
- Call `depositAsset` with a known amount and record rsETH minted.
- Assert that rsETH minted exceeds the fair value implied by the true market price, confirming over-minting without revert.
