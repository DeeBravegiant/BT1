Audit Report

## Title
No Recovery Path When Aave V3 Pool Is Paused Freezes User ETH Withdrawals - (File: contracts/LRTWithdrawalManager.sol)

## Summary
When `isAaveIntegrationEnabled` is `true` and the Aave V3 pool is paused, users with unlocked ETH withdrawal requests cannot complete them because `_processWithdrawalCompletion` calls `_withdrawFromAave`, which unconditionally invokes `aaveWETHGateway.withdrawETH` and reverts. Every admin escape hatch intended to disable the integration or recover funds also calls `_withdrawFromAave` before it can clear the flag, leaving no code path that sets `isAaveIntegrationEnabled = false` without first successfully withdrawing from Aave.

## Finding Description
`_processWithdrawalCompletion` at lines 720–731 checks whether the contract's native ETH balance is sufficient to cover a user's unlocked withdrawal. If not, it calls `_withdrawFromAave(amountNeeded)` unconditionally. [1](#0-0) 

`_withdrawFromAave` at line 917 calls `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))` with no error handling. If the Aave V3 pool is paused, this call reverts, propagating the revert through `completeWithdrawal` and `completeWithdrawalForUser`. [2](#0-1) 

The three intended recovery paths share the same flaw:

1. **`emergencyWithdrawFromAave`** (lines 551–563) calls `_withdrawFromAave(amount)` directly and reverts if Aave is paused. [3](#0-2) 

2. **`setAaveIntegrationEnabled(false)`** (lines 486–503) calls `_withdrawFromAave(aaveBalance)` before reaching `isAaveIntegrationEnabled = enabled` at line 503. If Aave is paused and `aaveBalance > 0`, the function reverts before the flag is cleared. [4](#0-3) 

3. **`configureAaveIntegration`** (lines 438–453) calls `_withdrawFromAave(aaveBalance)` on the old pool before accepting new addresses. Same revert path. [5](#0-4) 

There is no function in the contract that sets `isAaveIntegrationEnabled = false` without first successfully executing a withdrawal from Aave. The `pause()` function pauses the LRT contract itself but does not clear the Aave integration flag and does not unblock `emergencyWithdrawFromAave` (which lacks `whenNotPaused`).

## Impact Explanation
**Medium — Temporary freezing of funds; escalates to Critical (permanent freezing) if Aave is deprecated.**

All ETH deposited into Aave via `unlockQueue → depositToAaveExternal` is inaccessible while Aave is paused. Every user whose unlocked ETH withdrawal requires a drawdown from Aave has their funds frozen for the duration of the pause. If Aave is permanently paused or deprecated, those funds are permanently locked and all pending ETH withdrawal requests become uncompletable, matching the "Permanent freezing of funds" critical impact class.

## Likelihood Explanation
Aave V3 has a documented guardian-controlled pause mechanism and has been paused on Ethereum mainnet in response to external protocol incidents (e.g., March 2023 during the Euler Finance exploit). The Aave integration is an explicit, enabled feature of `LRTWithdrawalManager`. The trigger is a realistic, precedented external event, not a theoretical edge case. No attacker action is required; the freeze occurs automatically once Aave is paused and a user attempts to complete a withdrawal whose ETH is held in Aave.

## Recommendation
1. In `_processWithdrawalCompletion`, wrap `_withdrawFromAave` in a `try/catch`; on failure, revert with a descriptive error (e.g., `AaveWithdrawalFailed`) so operators know to disable the integration before retrying.
2. Separate the "disable flag" step from the "withdraw funds" step in `setAaveIntegrationEnabled(false)`. Set `isAaveIntegrationEnabled = false` unconditionally first, then attempt the withdrawal. This allows `completeWithdrawal` to fall back to native ETH balance immediately.
3. Add a `forceDisableAaveIntegration()` function callable by `PAUSER_ROLE` that sets `isAaveIntegrationEnabled = false` without attempting any Aave withdrawal, to be used when Aave is paused.

## Proof of Concept
1. Manager calls `configureAaveIntegration(...)` and `setAaveIntegrationEnabled(true)`.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)` — ETH is unlocked and auto-deposited to Aave via `depositToAaveExternal`. `address(this).balance` is now 0; all ETH is in Aave.
3. User calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, ...)` — request is queued.
4. Operator calls `unlockQueue` again — user's request is unlocked.
5. Aave V3 pool is paused by Aave governance.
6. User calls `completeWithdrawal(ETH_TOKEN, referralId)`.
7. `_processWithdrawalCompletion`: `address(this).balance < request.expectedAssetAmount` → true.
8. `_withdrawFromAave(amountNeeded)` → `aaveWETHGateway.withdrawETH(aavePool, ...)` reverts (pool paused).
9. `completeWithdrawal` reverts. User funds frozen.
10. Pauser calls `emergencyWithdrawFromAave(type(uint256).max)` → calls `_withdrawFromAave` → reverts.
11. Manager calls `setAaveIntegrationEnabled(false)` → `aaveBalance > 0` → calls `_withdrawFromAave` → reverts before `isAaveIntegrationEnabled = false` is reached.
12. Manager calls `configureAaveIntegration(newPool, ...)` → `aaveBalance > 0` → calls `_withdrawFromAave` on old pool → reverts.
13. No recovery path exists. ETH remains locked in Aave until Aave is unpaused.

```text
    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
```
