### Title
`respond_verify_foreign_tx` Accepts Attacker-Controlled `payload_hash` Without Binding It to the Pending Request — (`File: crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` contract method verifies only that the submitted ECDSA signature is valid over the caller-supplied `response.payload_hash`. It never checks that `response.payload_hash` is the canonical hash of `ForeignTxSignPayload{request, extracted_values}` for the matching pending `request`. A Byzantine MPC leader node (a single participant, strictly below the signing threshold) can therefore initiate a threshold-signing round over an arbitrary fake payload hash, collect honest co-signers' shares (who sign whatever hash the leader presents), and submit the resulting valid signature together with the fake hash. The contract accepts the response, drains the pending-request queue, and delivers a forged `VerifyForeignTransactionResponse` to every waiting caller.

### Finding Description

In `respond_verify_foreign_tx` the signature check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // caller-supplied

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // verified only against itself
    &secp_pk,
)
.is_ok()
```

`response.payload_hash` is taken verbatim from the responding node's argument; the contract never recomputes `SHA-256(borsh(ForeignTxSignPayload { request, values }))` from the stored `request` and checks equality. Contrast this with the regular `respond` path, where the payload is embedded in the `SignatureRequest` itself and is therefore immutable once the request is queued.

The node-side leader constructs the `SignatureRequest` payload from whatever `ForeignTxSignPayload` it chooses:

```rust
fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,   // leader-controlled
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    ...
    Ok(SignatureRequest { payload: Payload::Ecdsa(payload_bytes), ... })
}
```

Non-leader nodes participate in the threshold-signing protocol for whatever `SignatureRequest` the leader presents; they sign the hash without independently re-verifying the foreign chain. A Byzantine leader therefore controls what hash is signed, and the contract has no mechanism to detect the substitution.

### Impact Explanation

A Byzantine leader node can make the MPC network issue a valid threshold signature over a fabricated `ForeignTxSignPayload` (e.g., attesting that a Bitcoin transaction has 6 confirmations and a specific block hash when it does not, or that an EVM log contains specific bridge-deposit data when it does not). The contract resolves the pending request and returns the forged `VerifyForeignTransactionResponse` to all callers. Any bridge or NEAR contract that relies on `ForeignChainSignatureVerifier::verify_signature` from the SDK to gate fund releases will accept the forged attestation, enabling invalid bridge execution or double-spend conditions. This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

### Likelihood Explanation

The attack requires only one Byzantine participant to be selected as the signing-round leader for a given `verify_foreign_transaction` request. Leader selection is deterministic and rotates across participants; a Byzantine node will eventually be leader. No threshold-level collusion is needed: honest co-signers blindly sign the hash the leader presents. The entry point (`respond_verify_foreign_tx`) is callable by any attested participant, and the contract performs no binding check between the response hash and the stored request.

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected payload hash from the stored request and the extracted values included in the response, then assert equality before accepting the signature:

```rust
// Reconstruct the canonical hash from the request and the node-reported values
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),   // add this field to the response DTO
});
let expected_hash = expected_payload.compute_msg_hash()
    .map_err(|_| RespondError::PayloadHashComputationFailed)?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, have each non-leader node independently inspect the foreign chain and verify the payload hash before contributing its signing share, so a Byzantine leader cannot obtain a valid threshold signature over a fake hash.

### Proof of Concept

**Step 1 — User submits a legitimate request:**
```
verify_foreign_transaction({ domain_id, payload_version: V1,
    request: Bitcoin { tx_id: [0x07;32], confirmations: 6,
                       extractors: [BlockHash] } })
```
Contract stores the request in `pending_verify_foreign_tx_requests`.

**Step 2 — Byzantine leader constructs a fake payload:**
```rust
let fake_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: bitcoin_request.clone(),
    values: vec![BitcoinExtractedValue::BlockHash([0xff; 32].into())],
    //                                             ^^^^ fabricated block hash
});
let fake_hash = fake_payload.compute_msg_hash().unwrap();
```

**Step 3 — Leader initiates MPC signing of `fake_hash`:**
Honest co-signers receive a `SignatureRequest { payload: fake_hash, ... }` and contribute their shares without querying the Bitcoin RPC.

**Step 4 — Leader submits the forged response:**
```
respond_verify_foreign_tx(
    request = { tx_id: [0x07;32], ... },
    response = { payload_hash: fake_hash, signature: threshold_sig_over_fake_hash }
)
```

**Step 5 — Contract accepts:**
`verify_ecdsa_signature(sig, fake_hash, root_pk)` passes because the signature is genuinely valid over `fake_hash`. The pending request is drained and the forged `VerifyForeignTransactionResponse` is delivered to the caller.

**Step 6 — Bridge contract is deceived:**
The caller's bridge contract calls `ForeignChainSignatureVerifier::verify_signature` with `expected_extracted_values = [BlockHash([0xff;32])]` (values it learned from the response), which matches `fake_hash`, so verification passes and funds are released for a transaction that may not exist or may not have the required confirmations.

        let expected_payload_hash = expected_payload
            .compute_msg_hash()

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,

                &expected_payload_hash,

        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
