Audit Report

## Title
Stale Cross-Chain Rate in `CrossChainRateReceiver.getRate()` Enables Over-Minting of agETH — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/agETH/AGETHPoolV3.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` without any check against `lastUpdated`, meaning a rate that has not been refreshed via `lzReceive` for an arbitrary duration is served as current. `AGETHPoolV3.deposit()` uses this rate unconditionally to compute the agETH mint amount. Because agETH appreciates monotonically, a stale (lower) rate causes the pool to mint more agETH per ETH than the current L1 backing justifies, diluting existing holders.

## Finding Description

`CrossChainRateReceiver` records `lastUpdated` on every `lzReceive` call ( [1](#0-0) ) but `getRate()` ignores it entirely, returning `rate` unconditionally: [2](#0-1) 

`AGETHPoolV3.getRate()` delegates directly to this oracle: [3](#0-2) 

`deposit()` calls `viewSwapAgETHAmountAndFee()`, which computes the mint amount as `agETHAmount = amountAfterFee * 1e18 / agETHToETHrate`: [4](#0-3) 

The minted amount is then sent to the depositor with no further rate validation: [5](#0-4) 

Because agETH appreciates over time, a stale rate is always lower than the current L1 rate. A lower denominator in the division produces a larger `agETHAmount`, so every depositor during a stale window receives more agETH than the current collateral backing justifies. There is no `maxStaleness` guard, no circuit-breaker, and no admin-only pause that would prevent this automatically.

The SECURITY.md exclusion for "incorrect data supplied by third-party oracles" does not apply here: the rate data was correct when received; the defect is the contract's own failure to enforce a freshness invariant on data it already holds.

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.** The deposited ETH remains in the pool; no funds are directly drained. However, the excess agETH minted represents a claim on more ETH than was deposited, diluting the redemption value of all existing agETH holders. The protocol's collateral invariant (total agETH supply backed 1:1 at the current rate by pooled ETH) is broken for the duration of any stale window.

## Likelihood Explanation

LayerZero relayer delays are a documented operational reality (network congestion, gas spikes, relayer downtime). The `updateRate()` call on the provider side must be triggered externally and is permissionless; if no keeper calls it, the receiver rate can be arbitrarily old. No special privilege is required — any depositor passively benefits by calling `deposit()` during a stale window. The condition is reachable by any unprivileged external user through the normal `deposit()` flow.

## Recommendation

Add a configurable `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if the stored rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 24 hours, set by owner

function getRate() external view returns (uint256) {
    require(lastUpdated != 0 && block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

This causes `AGETHPoolV3.deposit()` to revert rather than mint at a stale rate, protecting the collateral invariant. `maxStaleness` should be owner-configurable to allow operational tuning without a contract upgrade.

## Proof of Concept

```solidity
// Foundry fork/local test
// 1. Deploy AGETHRateReceiver with a mock LZ endpoint.
// 2. Call lzReceive with rate = 1.01e18 (agETH slightly above peg).
// 3. vm.warp(block.timestamp + 7 days) — no further lzReceive calls.
// 4. Assume L1 agETH rate has appreciated to 1.05e18 (currentL1Rate).
// 5. Call AGETHPoolV3.deposit{value: 1 ether}("").
// 6. Assert:
//    agETHMinted  == 1e18 * 1e18 / 1.01e18  ≈ 0.9901e18  (stale, over-minted)
//    correctMint  == 1e18 * 1e18 / 1.05e18  ≈ 0.9524e18
//    agETHMinted > correctMint               // invariant broken, no revert
```

The test passes on unmodified code, confirming over-minting without any privileged action.
