### Title
Completed Sign Requests Never Removed from Node-Side `SignRequestStorage`, Enabling Unauthorized Threshold Signature Replay by a Byzantine Leader - (File: `crates/node/src/storage.rs`)

---

### Summary

The node-side `SignRequestStorage` (and its siblings `CKDRequestStorage`, `VerifyForeignTransactionRequestStorage`) in `crates/node/src/storage.rs` only ever **inserts** requests into RocksDB and never removes them — not on completion, not on timeout, not on epoch change. A single Byzantine MPC participant acting as signing leader can exploit this by opening a new MPC signing channel using the `SignatureId` of an already-completed, on-chain-resolved request. Honest follower nodes validate the request by looking it up in their local `SignRequestStorage`, find the stale entry, and participate in the signing protocol, producing an unauthorized threshold signature for the original payload without any corresponding on-chain request.

---

### Finding Description

`SignRequestStorage` exposes only two methods: `add()` and `get()`. There is no `remove()` or `delete()` method. [1](#0-0) 

Every `SignatureRequest` indexed from the chain is written to `DBCol::SignRequest` in RocksDB and stays there permanently. The in-memory `PendingRequests` queue in `crates/node/src/requests/queue.rs` does correctly remove completed and expired requests from its own `HashMap`, but this is a separate, ephemeral structure. The persistent `SignRequestStorage` is never updated to reflect completion.

The codebase itself acknowledges this gap with an explicit TODO comment in `monitor_block_updates`: [2](#0-1) 

The follower signing path for all three signature schemes (OT-based ECDSA, Robust ECDSA, EdDSA) validates a signing request by calling `sign_request_store.get(id)` and blocking until the entry is found: [3](#0-2) [4](#0-3) [5](#0-4) 

Because completed requests are never evicted, a stale entry for a resolved `SignatureId` satisfies this check indefinitely, even after the corresponding on-chain yield has been resolved and the contract's `pending_signature_requests` map has been cleaned up by `resolve_yields_for`. [6](#0-5) 

---

### Impact Explanation

A single Byzantine MPC node acting as leader can:

1. Select the `SignatureId` of any previously completed signature request (still present in every honest node's `SignRequestStorage`).
2. Open a new `NetworkTaskChannel` for `EcdsaTaskId::Signature { id: stale_id, presignature_id: fresh_id }`.
3. Honest follower nodes receive the channel, call `sign_request_store.get(stale_id)`, find the stale entry, and proceed to participate in the full MPC signing protocol.
4. A valid threshold ECDSA/EdDSA signature is produced for the original payload under the original derived key — **without any on-chain authorization**.

The resulting signature is cryptographically valid and usable on the target foreign chain (Ethereum, Bitcoin, Solana, etc.) to replay the original transaction if the target chain's replay protection has been consumed or is absent. Additionally, each unauthorized signing attempt consumes a presignature from honest nodes' stores, a scarce resource that requires significant MPC computation to replenish.

The same issue applies to `CKDRequestStorage`: a stale CKD request can be replayed to produce an unauthorized confidential key derivation output. [7](#0-6) 

**Impact class:** Unauthorized threshold signature issuance without required participant authorization — **Critical**.

---

### Likelihood Explanation

- Requires exactly **one** Byzantine MPC participant to act as leader. This is strictly below the signing threshold and is the minimal adversarial assumption for MPC security.
- The stale entries accumulate permanently; the attack window is unbounded and grows with network uptime.
- The attacker needs no special privileges beyond being a registered MPC participant and knowing a past `SignatureId` (which is observable from on-chain events indexed by every node).
- No network-level DoS, TEE attack, or key leakage is required.

---

### Recommendation

1. Add a `remove(id: SignatureId)` method to `SignRequestStorage` (and its siblings).
2. In `monitor_block_updates`, after `pending_signatures.notify_new_block(signature_requests)` processes `completed_signatures`, call `sign_request_store.remove(id)` for each finalized `SignatureId`. This resolves the acknowledged TODO at line 289 of `mpc_client.rs`.
3. On epoch change (resharing), call `delete_all(DBCol::SignRequest)` in the same atomic RocksDB batch used by `delete_stale_triples_and_presignatures`, since all prior request IDs are invalidated by the new epoch's yield namespace. [8](#0-7) 

---

### Proof of Concept

**Setup:** MPC network with `n=5`, `t=3`. Attacker controls node `P1` (one Byzantine participant, below threshold).

1. **Legitimate flow:** User calls `sign(payload, path)` on-chain. Contract creates yield with `data_id = H`. All nodes index `SignatureRequest { id: H, payload, tweak, ... }` into their `SignRequestStorage`. Honest leader `P2` runs the protocol; `P2` submits `respond(request, sig)` to the contract. Contract calls `resolve_yields_for`, removing the entry from `pending_signature_requests`. User receives the signature.

2. **Attack:** Byzantine `P1` opens a new `NetworkTaskChannel` with task `EcdsaTaskId::Signature { id: H, presignature_id: fresh_presig }` to honest followers `P3`, `P4`, `P5`.

3. **Follower validation:** Each follower calls `sign_request_store.get(H)`. Because `SignRequestStorage` never removed the entry after step 1, `get(H)` returns immediately with the original `SignatureRequest`.

4. **Unauthorized signing:** Followers proceed with `make_signature_follower_given_request`, consuming their unowned presignature shares and running the full Cait-Sith/FROST protocol. A valid threshold signature `σ'` is produced for `payload` under the derived key — with no on-chain authorization.

5. **Result:** `σ'` is a valid ECDSA signature for the original transaction payload. `P1` cannot submit it to the MPC contract (the yield is resolved), but can broadcast it directly to the target foreign chain. [2](#0-1) [9](#0-8) [3](#0-2)

pub struct SignRequestStorage {
    db: Arc<SecretDB>,
    add_sender: broadcast::Sender<SignatureId>,
}

impl SignRequestStorage {
    pub fn new(db: Arc<SecretDB>) -> anyhow::Result<Self> {
        let (tx, _) = tokio::sync::broadcast::channel(500);
        Ok(Self { db, add_sender: tx })
    }

    /// If given request is already in the database, returns false.
    /// Otherwise, inserts the request and returns true.
    pub fn add(&self, request: &SignatureRequest) -> bool {
        let key = borsh::to_vec(&request.id).unwrap();
        if self
            .db
            .get(DBCol::SignRequest, &key)
            .expect("Unrecoverable error reading from database")
            .is_some()
        {
            return false;
        }
        let value_ser = serde_json::to_vec(&request).unwrap();
        let mut update = self.db.update();
        update.put(DBCol::SignRequest, &key, &value_ser);
        update
            .commit()
            .expect("Unrecoverable error writing to database");
        let _ = self.add_sender.send(request.id);
        true
    }

    /// Returns when a signature request with given id is present, then returns it.
    /// This behavior is necessary because a peer might initiate computation for a signature
    /// request before our indexer has caught up to the request. We need proof of the request
    /// from on-chain in order to participate in the computation.
    pub async fn get(&self, id: SignatureId) -> Result<SignatureRequest, anyhow::Error> {
        let key = borsh::to_vec(&id)?;
        let mut rx = self.add_sender.subscribe();
        if let Some(request_ser) = self.db.get(DBCol::SignRequest, &key)? {
            return Ok(serde_json::from_slice(&request_ser)?);
        }
        loop {
            let added_id = match rx.recv().await {
                Ok(added_id) => added_id,
                Err(e) => match e {
                    broadcast::error::RecvError::Closed => {
                        metrics::SIGN_REQUEST_CHANNEL_FAILED.inc();
                        return Err(anyhow::anyhow!("Error in sign_request channel recv, {e}"));
                    }
                    broadcast::error::RecvError::Lagged(msg_n) => {
                        tracing::info!("{msg_n} messages lagged during sign_request channel recv");
                        continue;
                    }
                },
            };
            if added_id == id {
                break;
            }
        }
        let request_ser = self.db.get(DBCol::SignRequest, &key)?.unwrap();
        Ok(serde_json::from_slice(&request_ser)?)
    }

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
