Audit Report

## Title
Uninitialized `tokenFeeBps` Causes Zero Fees on All ERC20 Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains a separate per-token fee mapping `tokenFeeBps[token]` for ERC20 deposits. Because `addSupportedToken` never initializes this mapping, it defaults to `0` for every newly added token. Any user depositing a supported ERC20 token (e.g., wstETH) immediately after it is added will pay zero fees, while ETH depositors correctly pay `feeBps`. The protocol permanently loses fee revenue on all token deposits until an admin separately calls `setTokenFeeBps`.

## Finding Description
`addSupportedToken` (L637–656) sets `supportedTokenOracle[token]` and `tokenBridge[token]` but never sets `tokenFeeBps[token]`:

```solidity
// L651-653
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never set → remains 0
```

The token deposit path at L284–305 calls `viewSwapRsETHAmountAndFee(amount, token)` (L326–347), which reads `tokenFeeBps[token]`:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token]; // == 0
fee = amount * feeBpsForToken / 10_000;      // == 0
```

The result: `feeEarnedInToken[token] += 0` and the user receives wrsETH equivalent to the full token amount with no fee deducted. By contrast, the ETH deposit path (L265–278) uses the global `feeBps` set at initialization, so ETH deposits correctly charge fees. The setter `setTokenFeeBps` (L583–594) exists but requires a separate privileged call after token addition — the default state is the vulnerable state. No existing guard prevents a deposit at `tokenFeeBps == 0`.

## Impact Explanation
Every token deposit made while `tokenFeeBps[token] == 0` results in zero protocol fee revenue. The fees that should accrue to the protocol treasury are instead retained by depositors as excess rsETH. This is a direct, quantifiable loss of protocol yield — matching **High: Theft of unclaimed yield** from the allowed impact scope. The loss is permanent for all deposits made before `setTokenFeeBps` is called, and the window is open from the moment `addSupportedToken` is called.

## Likelihood Explanation
The vulnerability is active immediately upon `addSupportedToken` being called. No special attacker capability is required — `deposit(address token, uint256 amount, string referralId)` is a fully public, permissionless function. Any user who deposits a supported ERC20 token before the admin calls `setTokenFeeBps` triggers the fee-free path. There is no race condition or front-running requirement; the default mapping value is the vulnerable state for the entire lifetime of the token until explicitly corrected.

## Recommendation
Initialize `tokenFeeBps[token]` inside `addSupportedToken`, either by accepting a `_feeBps` parameter or by defaulting to the global `feeBps`:

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    // ... existing checks ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    // ...
}
```

Alternatively, fall back to `feeBps` in `viewSwapRsETHAmountAndFee` when `tokenFeeBps[token] == 0`, mirroring the ETH deposit behavior.

## Proof of Concept
1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0`.
2. User calls `deposit(wstETH, 10 ether, "ref")`.
3. `viewSwapRsETHAmountAndFee(10 ether, wstETH)` computes `feeBpsForToken = 0`, `fee = 10 ether * 0 / 10_000 = 0`.
4. `feeEarnedInToken[wstETH] += 0` — no fee recorded.
5. User receives wrsETH for the full `10 ether` of wstETH with zero fee deducted.
6. Equivalent ETH deposit via `deposit(referralId)` computes `fee = 10 ether * feeBps / 10_000 > 0` and correctly deducts it.

Foundry test plan: deploy `RSETHPool` with a non-zero `feeBps`, call `addSupportedToken` for a mock ERC20 with a mock oracle, deposit tokens, assert `feeEarnedInToken[token] == 0` and that the user received the full rsETH amount. Confirm the same deposit after `setTokenFeeBps` correctly records a non-zero fee.
