Looking at the actual `_withdrawFromAave` implementation and its callers in `LRTWithdrawalManager.sol`.

Audit Report

## Title
`_withdrawFromAave` Silent No-Op When `totalETHDepositedToAave == 0` Causes ETH Withdrawal Reverts — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`_withdrawFromAave` caps the withdrawable amount at `totalETHDepositedToAave` (tracked principal). When this counter reaches zero after all principal has been withdrawn, the function silently returns 0 even if aWETH balance (accrued interest) remains. Any subsequent call to `completeWithdrawal` for ETH that relies on this function to top up the contract balance will revert with `InsufficientLiquidityForWithdrawal`, temporarily freezing user ETH withdrawals until operator intervention.

## Finding Description

**Root cause — `_withdrawFromAave` (lines 905–921):**

```solidity
uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
    ? aaveBalance
    : totalETHDepositedToAave;          // == 0 when all principal is gone

withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
if (withdrawnAmount == 0) return 0;     // silent no-op
```

When `totalETHDepositedToAave == 0`, `withdrawablePrincipal = 0`, so `withdrawnAmount = 0` and the function returns without touching Aave, regardless of the actual aWETH balance. [1](#0-0) 

**Exploit path — `_processWithdrawalCompletion` (lines 719–732):**

```solidity
_withdrawFromAave(amountNeeded);   // returns 0, no ETH pulled

uint256 balanceAfter = address(this).balance;
if (balanceAfter < request.expectedAssetAmount) {
    revert InsufficientLiquidityForWithdrawal();   // always fires
}
``` [2](#0-1) 

**Preconditions → trigger → bad result:**

1. Aave integration is enabled and ETH has been deposited over time (`totalETHDepositedToAave > 0`).
2. Interest accrues continuously; `aaveAWETH.balanceOf(address(this)) > totalETHDepositedToAave`.
3. Users complete ETH withdrawals; each completion calls `_withdrawFromAave`, decrementing `totalETHDepositedToAave` until it reaches 0. The aWETH balance is now purely accrued interest (e.g., 0.5 ETH).
4. The contract's idle ETH balance is 0 (all ETH was in Aave).
5. Any user calls `completeWithdrawal(ETH, ...)`. The contract balance is insufficient, `_withdrawFromAave(amountNeeded)` is called, returns 0, and the call reverts with `InsufficientLiquidityForWithdrawal`. [3](#0-2) 

**Existing guards are insufficient:**

- `emergencyWithdrawFromAave` first calls `_collectInterestToTreasury()`, which withdraws all remaining aWETH to the treasury (not the contract). Then it calls `_withdrawFromAave(amount)`, which finds `aaveBalance == 0` and reverts with `InsufficientAaveBalance`. The ETH is now in the treasury, not available for user withdrawals. [4](#0-3) 

- `setAaveIntegrationEnabled(false)` follows the same pattern: collects interest to treasury, then skips `_withdrawFromAave` because `aaveBalance == 0` after collection. Integration is disabled but user withdrawals still revert. [5](#0-4) 

## Impact Explanation

**Temporary freezing of funds (Medium).** Unlocked ETH withdrawal requests that have passed the delay period cannot be completed. Users' rsETH has already been burned (at `unlockQueue` time) and their ETH is locked in the contract/Aave until an operator manually sends ETH to the contract or takes other remedial action. The freeze persists until operator intervention; no user action can resolve it.

## Likelihood Explanation

This is a natural end-state of normal protocol operation whenever Aave integration is active. As users progressively complete ETH withdrawals, `totalETHDepositedToAave` monotonically decreases to zero. Interest accrual is continuous and guaranteed. No attacker capability is required — any ordinary user completing a withdrawal after the counter hits zero triggers the revert. The condition is repeatable and predictable.

## Recommendation

Remove the artificial cap at `totalETHDepositedToAave` inside `_withdrawFromAave`, or fall back to withdrawing the full aWETH balance when `totalETHDepositedToAave == 0` but `aaveBalance > 0`. For example:

```solidity
uint256 withdrawablePrincipal = totalETHDepositedToAave == 0
    ? aaveBalance   // all remaining balance is interest; allow full withdrawal
    : (aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave);
```

Alternatively, update `totalETHDepositedToAave` accounting to track net deposits minus withdrawals correctly so it never underflows relative to the actual aWETH balance. Also audit `emergencyWithdrawFromAave` and `setAaveIntegrationEnabled(false)` to ensure interest collected goes to the contract (not treasury) when pending user withdrawals exist.

    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
```
