### Title
Unvalidated `builder` field in `ProposalInit` allows designated proposer to commit blocks with arbitrary sequencer address — (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

### Summary

The `is_proposal_init_valid` function validates many fields of `ProposalInit` (height, l1_da_mode, l2_gas_price_fri, starknet_version, version_constant_commitment, all four L1 gas prices, fee_proposal_fri) but never validates the `builder` field. The `builder` field is passed verbatim to `convert_to_sn_api_block_info` and becomes the `sequencer_address` inside `BlockInfo`, which is then hashed into `PartialBlockHashComponents.sequencer` and used for every `get_sequencer_address` syscall in the block. A malicious designated proposer can set `builder` to any address; the validator's batcher re-executes with the same spoofed `builder`, so both sides compute the same `partial_block_hash`, the `ProposalFinMismatch` check passes, and the block is committed with the wrong sequencer address.

### Finding Description

`ProposalInitValidation` carries the reference values the validator uses to check a received `ProposalInit`: [1](#0-0) 

`builder` is absent from this struct. Inside `is_proposal_init_valid`, the equality/margin checks cover `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, all four L1 prices, and `fee_proposal_fri`: [2](#0-1) [3](#0-2) 

`builder` is never mentioned. After `is_proposal_init_valid` passes, `initiate_validation` converts the full `ProposalInit` — including the unchecked `builder` — into a `ValidateBlockInput`: [4](#0-3) 

`convert_to_sn_api_block_info(&init)` maps `init.builder` to `BlockInfo.sequencer_address`. That `BlockInfo` is then used by the batcher to execute all transactions in the block and to compute `PartialBlockHashComponents`: [5](#0-4) 

The `sequencer` slot in `PartialBlockHashComponents` feeds directly into `PartialBlockHash::from_partial_block_hash_components` and therefore into `ProposalCommitment`: [6](#0-5)
