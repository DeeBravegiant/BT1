### Title
Double-Claiming of Deposit Refunds via Multiple Subtransactions in Dijkstra Era - (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces subtransactions: a single top-level transaction can embed multiple sub-transactions (`subTransactionsTxBodyL`). The production validation code in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` implements a batch-level check (`validateBatchWithdrawals`) that aggregates withdrawals across all subtransactions and verifies the total does not exceed the account balance. No equivalent batch-level check exists for deposit refunds. An attacker can include `UnRegDepositTxCert` for the same staking credential in two or more subtransactions within the same top-level transaction, potentially claiming the same deposit refund multiple times and creating ADA out of the deposit pot.

---

### Finding Description

The Dijkstra era's `UTXO` rule in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` validates a batch of subtransactions. For reward withdrawals, a dedicated batch-level guard exists: [1](#0-0) 

`validateBatchWithdrawals` aggregates all withdrawals across the top-level transaction and every subtransaction, then checks the total against the pre-batch account balance. This prevents a subtransaction from withdrawing a balance that was already consumed by a sibling subtransaction.

No analogous guard exists for deposit refunds. Deposit refunds are computed per-subtransaction inside `updateUTxOStateNoFees` via `certsTotalRefundsTxBody`: [2](#0-1) 

The `lookupDeposit` function passed to `shelleyTotalRefundsTxCerts` / `conwayTotalRefundsTxCerts` queries the **live** `certState` at the moment each subtransaction is processed. If the `SUBLEDGERS` rule does not propagate the updated `certState` between sibling subtransactions (i.e., if it evaluates them against the same initial state), both subtransactions would see the credential as registered and both would compute a non-zero refund, allowing the deposit to be claimed twice.

The developers have explicitly acknowledged this gap. The test file for the Dijkstra era contains: [3](#0-2) 

The test is marked `xit` (skipped/pending) and the expected predicate failure is `error "TODO: predicate failure not yet implemented"` — a Haskell runtime error that explicitly documents the missing production check. The parallel `validateBatchWithdrawals` function for withdrawals was added precisely because the same class of double-spend is possible for withdrawals; the deposit-refund analogue was not.

---

### Impact Explanation

If two subtransactions both successfully claim `UnRegDepositTxCert stakingCred keyDeposit` for the same credential, the deposit pot (`utxosDeposited`) is decremented twice for a single deposit, and the attacker's outputs receive twice the deposit amount. This constitutes direct creation of ADA through an invalid ledger state transition: the total ADA in circulation increases by `keyDeposit` without any corresponding source, violating the global ADA preservation invariant.

This maps to: **Critical — Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition.**

---

### Likelihood Explanation

The Dijkstra era is the current development frontier. Any user who can submit a valid top-level transaction can embed multiple subtransactions. No privileged role, governance majority, or external dependency is required. The attacker only needs to:
1. Register a staking credential (paying `keyDeposit`).
2. Construct a top-level transaction with two subtransactions, each containing `UnRegDepositTxCert` for the same credential.
3. Submit the transaction.

The attack is fully self-contained within a single transaction and requires no mempool observation or timing coordination.

---

### Recommendation

Add a batch-level deposit-refund check in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, mirroring `validateBatchWithdrawals`:

1. Before processing subtransactions, collect all `UnRegDepositTxCert` (and `UnRegDRepTxCert`, `RetirePoolTxCert`) certificates across the top-level transaction and all subtransactions.
2. Verify that no staking credential, DRep credential, or pool key appears in more than one deregistration certificate across the batch.
3. Alternatively, ensure `SUBLEDGERS` always threads the updated `certState` sequentially between sibling subtransactions, so the second deregistration of the same credential fails with `StakeKeyNotRegisteredDELEG` rather than silently succeeding.

The `validateBatchWithdrawals` function at lines 249–275 of `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` is the correct structural template for this fix.

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL
