### Title
Puzzle-Hash-Level Additions Cache Bypass Allows Fabricated Unspent Coin Acceptance — (`chia/wallet/util/wallet_sync_utils.py`, `chia/wallet/util/peer_request_cache.py`)

### Summary

The `_additions_in_block` cache in `PeerRequestCache` is keyed by `(header_hash, puzzle_hash)`. Once a legitimate coin with puzzle hash P is validated in block H, any subsequent coin with the same puzzle hash P at the same block H bypasses the Merkle inclusion proof entirely. The additions Merkle proof cryptographically binds a specific set of coin IDs to a puzzle hash, but the cache discards that binding. A malicious peer can inject a fabricated unspent coin with the same puzzle hash as a real coin in a real block, and the wallet will accept it as valid.

### Finding Description

**What `validate_additions` actually proves:**

When coin A (`parent=X, ph=P, amount=1`) is validated, the peer returns the full coin list for puzzle hash P and the proof validates two things: [1](#0-0) 

1. `hash_coin_ids([coin_A.name()])` is included in the additions Merkle root
2. `puzzle_hash_P` is included in the additions Merkle root

This cryptographically proves that the **complete set** of coins with puzzle hash P in block H is exactly `{coin_A}`. If coin B also had puzzle hash P, the hash would be `hash_coin_ids([coin_A.name(), coin_B.name()])` — a different value.

**What the cache records:**

After successful validation, the cache stores only `(header_hash, puzzle_hash)`: [2](#0-1) 

The specific coin IDs that were proven are discarded.

**The bypass path:**

When coin B (`parent=Y, ph=P, amount=999`) arrives for the same block H: [3](#0-2) 

`in_additions_in_block(header_hash_H, P)` returns `True` immediately, skipping the `RequestAdditions` round-trip and the Merkle proof entirely.

Back in `validate_received_state_from_peer`, after the cache hit returns `True`: [4](#0-3) 

For an unspent coin (`spent_height is None`), only `validate_block_inclusion` is called — which checks that block H is in the weight-proof chain, not that coin B is in block H. No removals check is performed. The coin is then added to `_states_validated` and the function returns `True`. [5](#0-4) 

### Impact Explanation

The wallet's coin store is written with a fabricated coin that does not exist on-chain. The wallet's displayed XCH/CAT balance is inflated. For CAT, NFT, DID, or singleton wallets, the wallet may make incorrect state decisions based on the corrupted coin record. This is an unauthorized accounting change affecting XCH and any asset type whose coins share a puzzle hash with a real on-chain coin.

The fabricated coin cannot be spent (the full node would reject the spend bundle), but the wallet's local state is persistently corrupted until a resync.

### Likelihood Explanation

- The attacker must be an untrusted peer the wallet connects to (standard network peer, no privilege required).
- The cache is per-peer (`get_cache_for_peer`), so the attacker controls both steps from the same connection. [6](#0-5) 
- The attacker needs one real on-chain coin with puzzle hash P in any block H (public blockchain data).
- The attack is deliverable via the standard `CoinStateUpdate` protocol message path (`state_update_received` → `add_states_from_peer` → `validate_received_state_from_peer`). [7](#0-6) 

### Recommendation

The cache key must include the specific coin ID, not just the puzzle hash. Change `_additions_in_block` from `LRUCache[tuple[bytes32, bytes32], uint32]` (keyed by `(header_hash, puzzle_hash)`) to `LRUCache[tuple[bytes32, bytes32, bytes32], uint32]` (keyed by `(header_hash, puzzle_hash, coin_id)`). Alternatively, store the full proven coin-ID set per `(header_hash, puzzle_hash)` and check membership of the specific coin ID on cache hit.
