### Title
Cross-Request Signature Replay in `respond_verify_foreign_tx` via Unbound `payload_hash` — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that a supplied signature is valid for a caller-supplied `payload_hash`, but never checks that `payload_hash` was actually derived from the `VerifyForeignTransactionRequest` being answered. A single Byzantine MPC participant can therefore replay a previously observed valid `(payload_hash, signature)` pair to forge a verification response for an entirely different pending foreign-chain transaction request.

### Finding Description

The `respond_verify_foreign_tx` function performs two independent checks before resolving pending yields:

1. The ECDSA signature in `response` is valid for `response.payload_hash` against the **root** public key (no per-user tweak).
2. The `request` key exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

Critically, the contract never verifies that `response.payload_hash` was computed from `request`. The `payload_hash` is a caller-supplied field inside `VerifyForeignTransactionResponse`; the contract accepts whatever value the responding node provides, as long as the signature over it is valid. [2](#0-1) 

In the legitimate flow, the MPC nodes compute `payload_hash` as `ForeignTxSignPayload::V1 { request, values }.compute_msg_hash()`, binding the hash to the specific foreign-chain RPC request and its extracted values. The contract, however, enforces none of this binding. [3](#0-2) 

Additionally, `VerifyForeignTransactionRequest` keys are **caller-agnostic**: requests from different accounts for the same foreign-chain transaction are queued under the same map entry and all drained by a single `respond_verify_foreign_tx` call. This is confirmed by the fan-out design and tests. [4](#0-3) 

### Impact Explanation

A single malicious MPC node (Byzantine participant strictly below the signing threshold) can:

1. Observe any on-chain `respond_verify_foreign_tx` call that carried a valid `(payload_hash_A, signature_A)` pair for a legitimate foreign-chain transaction.
2. Submit a new `verify_foreign_transaction` request for a **different** (possibly fraudulent) foreign-chain transaction (`request_B`).
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: signature_A })`.
4. The contract accepts the call — the signature is valid for `payload_hash_A`, and `request_B` exists in the pending map — and delivers `{ payload_hash_A, signature_A }` to every caller waiting on `request_B`.

The callers of `request_B` receive a signed response whose `payload_hash` does not correspond to their submitted transaction. Any downstream bridge or application that trusts this response as proof of foreign-chain state is presented with a forged verification. This enables invalid bridge execution and potential double-spend conditions, matching the **High** impact category: *"Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

Exploitation requires the attacker to be an attested MPC participant (`assert_caller_is_attested_participant_and_protocol_active` is enforced). However, only **one** such participant needs to be Byzantine — no threshold collusion is required. The attacker needs only to observe a prior legitimate `respond_verify_foreign_tx` transaction (publicly visible on-chain) to obtain a replayable `(payload_hash, signature)` pair. The fan-out design (caller-agnostic request keys) means the attacker does not even need to be the original submitter of `request_B`.

### Recommendation

Bind `response.payload_hash` to `request` inside `respond_verify_foreign_tx`. Before accepting the response, recompute the expected payload hash from the `request` fields and the extracted values, and assert that `response.payload_hash` equals the expected value. Alternatively, require the MPC nodes to include the `request` identifier inside the signed payload so the contract can verify the binding cryptographically.

The analogous fix in the reference report was to move the nonce increment **before** the external call; here the fix is to verify the payload-hash binding **before** resolving the yields.

### Proof of Concept

```
1. Legitimate flow (observed on-chain):
   respond_verify_foreign_tx(
     request = { tx_id: 0xAAAA, chain: Bitcoin, ... },
     response = { payload_hash: H_A, signature: S_A }
   )
   → H_A = hash(ForeignTxSignPayload { request: 0xAAAA, values: [...] })
   → S_A = ECDSA_sign(root_key, H_A)   ← valid, publicly visible

2. Attacker submits a fraudulent request:
   verify_foreign_transaction({ tx_id: 0xBBBB, chain: Bitcoin, ... })
   → queued in pending_verify_foreign_tx_requests under key K_B

3. Attacker (malicious MPC node) calls:
   respond_verify_foreign_tx(
     request = { tx_id: 0xBBBB, ... },   ← K_B exists in pending map ✓
     response = { payload_hash: H_A, signature: S_A }  ← replayed from step 1
   )

4. Contract checks:
   - verify_ecdsa_signature(S_A, H_A, root_key) → OK  ✓  (no binding to request)
   - pending_verify_foreign_tx_requests.get(K_B) → Some(...) ✓
   → resolve_yields_for(K_B, serialize({ H_A, S_A }))

5. Callers waiting on tx_id=0xBBBB receive { payload_hash: H_A, signature: S_A }
   — a signed attestation whose hash does not correspond to their transaction.
``` [5](#0-4)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
