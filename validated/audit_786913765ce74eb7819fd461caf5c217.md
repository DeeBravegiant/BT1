### Title
`respond_ckd` Skips Output Verification for `AppPublicKey` Variant, Enabling Single-Participant CKD Forgery - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs a cryptographic pairing check (`ckd_output_check`) only for the `AppPublicKeyPV` variant of CKD requests. For the `AppPublicKey` variant, the contract executes an **empty match arm with no output validity check**. A single malicious attested participant can submit an arbitrary `CKDResponse` for any pending `AppPublicKey` CKD request, bypassing the threshold requirement and delivering a forged derived key to the user.

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant: [1](#0-0) 

- **`AppPublicKeyPV`**: calls `ckd_output_check(...)`, which verifies `e(big_c, G2) = e(big_y, app_pk2) · e(hash_point, public_key)` via a BLS12-381 pairing check.
- **`AppPublicKey`**: empty arm — **no check whatsoever**. [2](#0-1) 

For `AppPublicKey`, the user provides only `pk1 = x·G1` (no `pk2`), so the pairing equation used in `AppPublicKeyPV` cannot be applied directly. However, the contract makes no attempt to verify the response at all — not even a basic point-validity or non-identity check. Any attested participant can call `respond_ckd` with an arbitrary `(big_y, big_c)` pair and the contract will accept it and resolve all pending yields for that request. [3](#0-2) 

Contrast with `respond()` for regular signatures, which **always** verifies the signature against the derived public key before accepting it: [4](#0-3) 

The analog to M-17 is direct: just as `LienToken.createLien` checked only whether an auction existed (an external trigger) without checking the underlying liquidation condition, `respond_ckd` checks only whether the caller is an attested participant without checking whether the CKD output is cryptographically valid for the `AppPublicKey` variant.

---

### Impact Explanation

A single malicious attested participant (strictly below the signing threshold) can:

1. Submit `big_y = identity_point` and `big_c = attacker_chosen_point` for any pending `AppPublicKey` CKD request.
2. The user computes their derived key as `big_c − x · big_y = big_c − x · 0 = big_c`, a point **fully controlled by the attacker**.
3. The attacker knows the user's derived key material, enabling theft of any assets the user subsequently controls with that key.

This bypasses the threshold requirement: the MPC protocol requires `t+1` participants to cooperate to produce a valid CKD output, but a single malicious participant can forge the contract-level response for `AppPublicKey` variant without any threshold cooperation. This matches the **Critical** allowed impact: *"Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery."* [5](#0-4) 

---

### Likelihood Explanation

Any single attested participant in the MPC network can exploit this. The `AppPublicKey` variant is used in production, confirmed by the e2e test `ckd_response__passes_cryptographic_verification`: [6](#0-5) 

The attacker only needs to be an attested participant and observe a pending `AppPublicKey` CKD request in the contract state (which is public on-chain). No collusion, no privileged access, and no network-level attack is required.

---

### Recommendation

1. **Require `AppPublicKeyPV` for all CKD requests** to enable full on-chain pairing verification, and deprecate the unverifiable `AppPublicKey` variant.
2. If `AppPublicKey` must be retained, add at minimum a non-identity and curve-membership check on `big_y` and `big_c` before resolving yields, and document clearly that this variant provides weaker on-chain guarantees.
3. Align `respond_ckd` with `respond()` by rejecting responses that cannot be verified against the domain's public key.

---

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(pk1)` where `pk1 = x·G1`.
2. Malicious attested participant calls:
   ```
   respond_ckd(request, CKDResponse { big_y: G1_IDENTITY, big_c: attacker_point })
   ```
3. Contract reaches line 676 (`AppPublicKey(_) => {}`), executes the empty arm, skips all checks, and calls `resolve_yields_for` at line 684.
4. User receives `(G1_IDENTITY, attacker_point)` and computes derived key = `attacker_point − x · G1_IDENTITY = attacker_point`.
5. Attacker knows `attacker_point` and thus the user's derived key, enabling full control over any assets protected by it. [7](#0-6)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Verify that a CKD response (AppPublicKey variant) is mathematically correct.
#[tokio::test]
#[expect(non_snake_case)]
async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),
        "CKD response failed cryptographic verification"
    );
}
