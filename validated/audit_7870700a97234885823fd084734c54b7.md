Audit Report

## Title
Missing Chainlink Price Feed Staleness Validation in `getAssetPrice()` Allows Stale Prices to Inflate rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all validation fields (`updatedAt`, `answeredInRound`, `roundId`), silently accepting stale or incomplete Chainlink rounds. The unvalidated price flows directly into the rsETH minting formula in `LRTDepositPool`, allowing any depositor to mint excess rsETH at a stale inflated price, diluting the share value of all existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches the Chainlink price without any validation:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values of `latestRoundData()` are available but only `answer` is used. No check is performed on `updatedAt` (staleness), `answeredInRound` vs `roundId` (round completeness), or `price <= 0` (validity).

The same codebase already implements the correct pattern in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

**Exploit path:**

1. A Chainlink LST/ETH feed (e.g. stETH/ETH) goes stale at an inflated last-known price (e.g. during a temporary depeg recovery or oracle downtime within the 24-hour heartbeat window).
2. An unprivileged attacker calls `depositAsset(asset, depositAmount, 0, "")`.
3. `_beforeDeposit()` calls `getRsETHAmountToMint()`: [3](#0-2) 
4. `lrtOracle.getAssetPrice(asset)` delegates to `ChainlinkPriceOracle.getAssetPrice()`, returning the stale inflated price.
5. The attacker receives `rsethAmountToMint = (depositAmount × staleInflatedPrice) / rsETHPrice` — more rsETH than their deposit is worth.

The `pricePercentageLimit` guard in `_updateRsETHPrice()` does **not** protect the deposit path: it only fires when `updateRSETHPrice()` is called, and `depositAsset()` uses the stored `rsETHPrice` without triggering a price update. [4](#0-3) 

The stale price also propagates into `_getTotalEthInProtocol()`, inflating the computed TVL and causing excess fee rsETH to be minted to the treasury: [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.**

Excess rsETH minted to the attacker at a stale inflated price dilutes the share value of all existing rsETH holders. Each existing holder's rsETH now represents a smaller fraction of the protocol's true ETH TVL, constituting theft of their unclaimed yield. The same stale price fed into `_updateRsETHPrice()` causes excess fee rsETH to be minted to the treasury, further diluting holders. This is not excluded by the SECURITY.md "incorrect data supplied by third-party oracles" clause — the oracle is supplying correct staleness metadata (`updatedAt`, `answeredInRound`) that the protocol is ignoring; the bug is in the protocol's own code.

## Likelihood Explanation
Chainlink LST/ETH feeds (stETH/ETH, cbETH/ETH, rETH/ETH) have 24-hour heartbeat intervals. During low-volatility periods, the feed legitimately does not update for up to 24 hours. If the last reported price is higher than the true current price (e.g., after a temporary depeg event where the price has since fallen), the stale price is exploitable. No special role, setup, or privileged access is required — any user can call `depositAsset()` at any time. [6](#0-5) 

## Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
// Optionally: if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
``` [7](#0-6) 

## Proof of Concept
**Foundry fork test outline:**

1. Fork mainnet at a block where a supported LST Chainlink feed (e.g. stETH/ETH) has not been updated for several hours (within its 24-hour heartbeat).
2. Warp `block.timestamp` forward so `block.timestamp - updatedAt > 1 hours` while keeping `answeredInRound == roundId` (simulating a heartbeat-stale but not incomplete round).
3. Alternatively, use `vm.mockCall` on the feed's `latestRoundData()` to return `answeredInRound < roundId` with an inflated `answer`.
4. Call `depositAsset(stETH, amount, 0, "")` as an unprivileged address.
5. Assert that `rsETH.balanceOf(attacker) > amount` (in ETH-equivalent terms using the true price), confirming excess rsETH was minted.
6. Assert that existing holders' rsETH price (via `lrtOracle.rsETHPrice()` after `updateRSETHPrice()`) has decreased relative to a baseline run without the stale price, confirming yield dilution.
