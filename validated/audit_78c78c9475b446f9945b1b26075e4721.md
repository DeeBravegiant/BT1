Audit Report

## Title
Zero-Initialized Rate in `CrossChainRateReceiver` Causes Deposit DoS via Division-by-Zero — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.rate` is a `uint256` storage variable that defaults to `0` at deployment. `getRate()` returns it unconditionally. Both overloads of `viewSwapRsETHAmountAndFee` divide by this value without a zero-guard, causing a Solidity 0.8.x checked arithmetic panic (revert) on every deposit call until the first `lzReceive` message sets a non-zero rate. No funds are minted or lost — all deposits simply revert during this window.

## Finding Description

`CrossChainRateReceiver.rate` is declared as a plain storage variable with no initialization: [1](#0-0) 

`RSETHRateReceiver`'s constructor sets `rateInfo`, `srcChainId`, `rateProvider`, and `layerZeroEndpoint`, but never sets `rate`, leaving it at `0`: [2](#0-1) 

`getRate()` returns the value unconditionally with no zero-guard: [3](#0-2) 

`RSETHPoolV3.initialize` accepts the oracle address with only a non-zero address check — it never calls `getRate()` to validate the returned value: [4](#0-3) 

The `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee` before the `dailyMintLimit` check, so the division-by-zero panic fires first: [5](#0-4) 

The ETH-path division has no zero-guard: [6](#0-5) 

The token-path division also has no zero-guard: [7](#0-6) 

By contrast, `viewSwapAssetToPremintedRsETH` explicitly guards against zero rate, confirming developer awareness of the risk: [8](#0-7) 

`addSupportedToken` and `setSupportedTokenOracle` both guard against zero rate for collateral token oracles, but the main rsETH oracle path in `initialize` and `setRSETHOracle` does not: [9](#0-8) [10](#0-9) 

The same unguarded division pattern exists across all pool variants (`RSETHPool`, `RSETHPoolV2`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`): [11](#0-10) 

## Impact Explanation

**Medium — Temporary freezing of funds (deposits).** Under Solidity 0.8.x, division by zero is a checked arithmetic panic that reverts the transaction. No tokens are minted, no funds are transferred, and no state is corrupted. The effect is that all deposit calls revert until the first cross-chain `lzReceive` message delivers a non-zero rate. This is a temporary, self-resolving DoS window that matches the allowed impact class "Medium. Temporary freezing of funds."

## Likelihood Explanation

The DoS window is bounded by the deployment-to-first-`lzReceive` gap. For `RSETHPoolV3`, `reinitialize` enforces a future `startTimestamp`, so the effective window is between when `startTimestamp` passes and when the first `lzReceive` arrives. For older pool variants without `limitDailyMint`, the window begins immediately after `initialize`. No attacker action is required — any user who deposits during this window loses gas and receives nothing. The likelihood of at least one user hitting this window is moderate given that pools are publicly accessible immediately after deployment.

## Recommendation

1. In `RSETHPoolV3.initialize` (and equivalent initializers in all pool variants), add a rate validation analogous to `addSupportedToken`:
   ```solidity
   if (IOracle(_rsETHOracle).getRate() == 0) revert UnsupportedOracle();
   ```
2. Apply the same guard in `setRSETHOracle`.
3. In both overloads of `viewSwapRsETHAmountAndFee`, add an explicit zero-check before division:
   ```solidity
   if (rsETHToETHrate == 0) revert UnsupportedOracle();
   ```
4. Consider requiring the deployer to seed an initial rate in `RSETHRateReceiver`'s constructor, or add a one-time `setInitialRate` function callable before the pool goes live.

## Proof of Concept

```solidity
function testDepositRevertsBeforeFirstLzReceive() public {
    // 1. Deploy RSETHRateReceiver — rate == 0 by default
    RSETHRateReceiver receiver = new RSETHRateReceiver(
        srcChainId, rateProvider, lzEndpoint
    );
    assertEq(receiver.getRate(), 0);

    // 2. Initialize RSETHPoolV3 pointing at the fresh receiver
    pool.initialize(admin, bridger, wrsETH, feeBps, address(receiver), true);

    // 3. Reinitialize with a past startTimestamp so limitDailyMint doesn't block on timestamp
    pool.reinitialize(100 ether, block.timestamp); // startTimestamp = now

    // 4. Any deposit attempt reverts with Panic(0x12) — division by zero
    vm.expectRevert(); // Solidity 0.8 division-by-zero panic
    pool.deposit{value: 1 ether}("ref");

    // 5. After first lzReceive sets rate, deposit succeeds
    vm.prank(address(lzEndpoint));
    receiver.lzReceive(
        srcChainId,
        abi.encodePacked(rateProvider),
        0,
        abi.encode(uint256(1.05e18))
    );
    assertGt(receiver.getRate(), 0);

    pool.deposit{value: 1 ether}("ref"); // succeeds
}
```
