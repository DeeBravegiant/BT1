Audit Report

## Title
Uninitialized `tokenFeeBps` Mapping Causes Zero Fees on All ERC20 Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains a per-token fee mapping `tokenFeeBps` that is never initialized when a token is added via `addSupportedToken`. Because Solidity mappings default to zero, every token deposit (e.g., wstETH) computes a fee of zero, transferring the full wrsETH equivalent to the depositor with no fee deducted. The protocol collects zero fee revenue on all ERC20 token deposits while correctly collecting fees on ETH deposits.

## Finding Description
`addSupportedToken` (L637–656) sets `supportedTokenOracle[token]` and `tokenBridge[token]` but never sets `tokenFeeBps[token]`:

```solidity
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never set — defaults to 0
```

The token deposit path in `deposit(address token, uint256 amount, string memory referralId)` (L284–305) calls `viewSwapRsETHAmountAndFee(amount, token)` (L326–347), which reads the uninitialized mapping:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token]; // == 0
fee = amount * feeBpsForToken / 10_000;      // == 0
```

The result: `feeEarnedInToken[token] += 0` and the depositor receives wrsETH equivalent to the full token amount with no fee deducted. The setter `setTokenFeeBps` (L583–594) exists but requires a separate privileged call after token addition — there is no enforcement that it is ever called. The ETH deposit path (L311–320) uses the global `feeBps` set at initialization, so it is unaffected.

## Impact Explanation
Every token depositor receives more wrsETH than they are entitled to — the fee that should accrue to the protocol treasury is instead retained by the depositor in the form of excess wrsETH. This is a continuous, permanent loss of protocol fee revenue on all ERC20 token deposits from the moment any token is added. This matches **High — Theft of unclaimed yield**: yield the protocol is designed and expected to collect is diverted to depositors due to the uninitialized fee state.

## Likelihood Explanation
The vulnerable state is the default state. Any unprivileged user calling `deposit(token, amount, referralId)` on any supported token immediately after `addSupportedToken` triggers the zero-fee path. No special conditions, front-running, or attacker setup is required. The function is public, permissionless, and callable by any address. The vulnerability persists indefinitely unless the admin separately calls `setTokenFeeBps` for every supported token.

## Recommendation
Initialize `tokenFeeBps[token]` inside `addSupportedToken`, either by accepting a `_feeBps` parameter or by defaulting to the global `feeBps`:

```solidity
function addSupportedToken(address token, address oracle, address bridge, uint256 _feeBps)
    external onlyRole(TIMELOCK_ROLE)
{
    // ... existing checks ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    tokenFeeBps[token] = _feeBps;
    // ...
}
```

Alternatively, fall back to `feeBps` in `viewSwapRsETHAmountAndFee` when `tokenFeeBps[token] == 0`, mirroring the ETH deposit behavior.

## Proof of Concept
1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0` (L651–653).
2. User calls `deposit(wstETH, 10 ether, "ref")` (L284).
3. `viewSwapRsETHAmountAndFee(10 ether, wstETH)` executes: `feeBpsForToken = tokenFeeBps[wstETH] = 0`, `fee = 10 ether * 0 / 10_000 = 0` (L335–336).
4. `feeEarnedInToken[wstETH] += 0` — no fee recorded (L300).
5. User receives wrsETH equivalent to the full `10 ether` of wstETH with zero fee deducted (L302).
6. Contrast: an ETH deposit of equivalent value via `deposit(referralId)` computes `fee = amount * feeBps / 10_000 > 0` (L312) and correctly deducts it.

Foundry test plan: deploy `RSETHPool` with a mock oracle and mock wrsETH, call `addSupportedToken` without a subsequent `setTokenFeeBps`, call `deposit(token, amount, "ref")`, and assert `feeEarnedInToken[token] == 0` and the user's wrsETH balance equals the full unfeeed amount. Compare against a run where `setTokenFeeBps` is called first to confirm the fee is correctly deducted.
