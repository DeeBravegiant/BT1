### Title
Sub-Transaction Native Script Validation Bypass via Aggregated Script Presence Check — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxow.hs`)

### Summary

In the Dijkstra era, the `UTXOW` rule performs an **aggregated** script-presence check across the top-level transaction and all its sub-transactions, but performs **per-level** script execution (phase-1 validation). A native script required to authorize an action (withdrawal, certificate, governance vote) in a sub-transaction can be supplied in the top-level transaction's witness set. Because the top-level `validateFailedBabbageScripts` only validates scripts against the top-level tx's own needed hashes, and the sub-level `SUBUTXOW` only validates scripts present in the sub-tx's own witness set, the script is **never executed at all**. An unprivileged attacker can therefore drain rewards from any native-script-locked stake account by embedding an unauthorized withdrawal in a sub-transaction.

---

### Finding Description

The `dijkstraUtxowTransition` function in `Utxow.hs` performs two distinct checks on scripts:

**1. Aggregated presence check** (line 261):
```haskell
let witnessScripts =
      Map.keysSet (tx ^. witsTxL . scriptTxWitsL)
        <> foldMap (Map.keysSet . (^. witsTxL . scriptTxWitsL)) subTxs
    ...
runTest $ Babbage.babbageMissingScripts pp allScriptHashesNeeded refScripts witnessScripts
```
`witnessScripts` is the **union** of scripts from all levels. `allScriptHashesNeeded` is the **union** of scripts needed by all levels. The check passes as long as the union of provided scripts covers the union of needed scripts — a script needed by a sub-tx can be satisfied by a script present in the top-level tx's witness set.

**2. Per-level phase-1 execution** (line 244):
```haskell
-- Per-level: phase-1 script validation is per-tx (script execution)
runTest $ Babbage.validateFailedBabbageScripts tx scriptsProvided topScriptHashesNeeded
```
`scriptsProvided` is only the top-level tx's scripts. `topScriptHashesNeeded` is only the top-level tx's needed script hashes. A script needed exclusively by a sub-tx is **not in `topScriptHashesNeeded`**, so it is not validated here even if it is present in `scriptsProvided`.

In `SUBUTXOW` (`SubUtxow.hs`, line 267–270):
```haskell
let scriptsNeeded = scriptsNeededStAnnTx stAnnTx
    scriptHashesNeeded = getScriptsHashesNeeded scriptsNeeded
runTest $ Babbage.validateFailedBabbageScripts tx scriptsProvided scriptHashesNeeded
```
`scriptsProvided` here is the **sub-tx's own** scripts. If the script was placed in the top-level tx's witness set, it is absent from the sub-tx's `scriptsProvided`, so `validateFailedBabbageScripts` does not check it here either. There is **no** `babbageMissingScripts` call in `SUBUTXOW` — that check exists only at the top level.

**Result**: A native script needed by a sub-transaction but provided in the top-level transaction's witness set is **never executed**. The aggregated presence check passes, but the script's conditions (timelock slot range, multisig threshold, etc.) are never evaluated.

---

### Impact Explanation

**Critical — Direct loss of ADA through an invalid ledger state transition.**

The most direct attack targets reward accounts locked by native scripts (timelocks or multisig). In Cardano, a withdrawal from a script-locked reward account requires the script to be provided and to validate against the transaction. With this bug:

1. An attacker identifies a reward account whose stake credential is a native script hash (e.g., a multisig requiring M-of-N signatures, or a timelock).
2. The attacker constructs a top-level transaction containing:
   - The native script in the top-level tx's witness set.
   - A sub-transaction whose body includes a withdrawal from the victim's reward account.
   - Outputs directing the withdrawn ADA to the attacker's address.
3. The aggregated `babbageMissingScripts` check passes (the script hash is present in the union of provided scripts).
4. The script is never executed against the sub-tx — neither in the top-level `validateFailedBabbageScripts` (script not in `topScriptHashesNeeded`) nor in the sub-level `validateFailedBabbageScripts` (script not in sub-tx's `scriptsProvided`).
5. The withdrawal is accepted, and the victim's entire reward balance is redirected to the attacker.

The same bypass applies to certificates (unauthorized delegation, DRep registration/deregistration) and governance votes submitted inside sub-transactions that are protected by native scripts.

---

### Likelihood Explanation

- The Dijkstra era introduces sub-transactions; this vulnerability is specific to that era.
- Native-script-locked stake credentials are used in production for multisig staking arrangements and protocol-controlled accounts.
- The attack requires no privileged access, no key compromise, and no governance majority — only the ability to submit a well-formed transaction.
- The attacker needs only to know the victim's reward address (publicly visible on-chain) and the script preimage (also publicly visible, as it must be submitted on-chain to be used legitimately).

---

### Recommendation

The `SUBUTXOW` rule must perform its own **per-level** missing-scripts check, ensuring that every script needed by the sub-transaction is present in the sub-transaction's own witness set (or its reference inputs). The aggregated check in the top-level `UTXOW` should be removed or restricted to only verify that no extraneous scripts are provided across levels, not that sub-level needs are satisfied by top-level witnesses.

Concretely, add a `babbageMissingScripts`-equivalent call inside `dijkstraSubUtxowTransition` using only the sub-tx's own `scriptsProvided` and `scriptHashesNeeded`, mirroring the per-level pattern already used for `validateFailedBabbageScripts`, `hasExactSetOfRedeemers`, and `checkScriptIntegrityHash`.

---

### Proof of Concept

**Setup**: Alice holds 100 ADA in rewards. Her stake credential is a native multisig script `S` requiring signatures from keys `K1` and `K2`. The script hash `H(S)` is her stake credential.

**Attack**:

1. Attacker constructs `topTx` with:
   - `topTx.wits.scripts = { H(S) → S }` (script provided at top level)
   - `topTx.body.subTransactions = { subTx }` where:
     - `subTx.body.withdrawals = { Alice's reward address → 100 ADA }`
     - `subTx.wits.scripts = {}` (no scripts in sub-tx witness set)
   - `topTx.body.outputs = [{ attacker's address, 100 ADA }]`

2. Ledger evaluation:
   - Top-level `babbageMissingScripts`: `witnessScripts = {H(S)}`, `allScriptHashesNeeded = {H(S)}` (from sub-tx withdrawal) → **passes**.
   - Top-level `validateFailedBabbageScripts`: `topScriptHashesNeeded = {}` (top-level tx has no script-locked inputs/withdrawals) → **S is not checked**.
   - Sub-level `validateFailedBabbageScripts`: `scriptsProvided = {}` (sub-tx has no scripts) → **S is not checked**.
   - Sub-level `validateNeededWitnesses`: withdrawal credential is a script hash, not a key hash → **no VKey required**.

3. Transaction is accepted. Alice's 100 ADA rewards are transferred to the attacker. The multisig condition was never evaluated. [1](#0-0) [2](#0-1) [3](#0-2)
