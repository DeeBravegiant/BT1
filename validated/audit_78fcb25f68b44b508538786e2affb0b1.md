### Title
Unconstrained `farmer_reward_address_override` Allows Malicious Harvester to Divert Farmer Block Rewards - (File: `chia/farmer/farmer_api.py`)

### Summary
In `_process_respond_signatures`, the farmer unconditionally accepts any `farmer_reward_address_override` value from a connected harvester's `RespondSignatures` message. Although `notify_farmer_reward_taken_by_harvester_as_fee` logs a warning when the fee quality threshold is not met, it does not block the override from being used. A malicious third-party harvester can set this field to any arbitrary puzzle hash and redirect the farmer's entire XCH block reward (1/8 of block reward) to an attacker-controlled address.

### Finding Description
The CHIP-22 mechanism allows third-party harvesters to take the farmer's block reward as a fee by setting `farmer_reward_address_override` in `NewProofOfSpace` and `RespondSignatures`. In `FarmerAPI._process_respond_signatures` (`chia/farmer/farmer_api.py`), the farmer unconditionally replaces `farmer_reward_address` with whatever puzzle hash the harvester provides:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override  # no validation
    include_source_signature_data = True
``` [1](#0-0) 

This overridden address is then passed directly into `DeclareProofOfSpace` as `farmer_puzzle_hash`: [2](#0-1) 

The full node uses this value verbatim when constructing the unfinished block's `FoliageBlockData.farmer_reward_puzzle_hash`: [3](#0-2) 

The `notify_farmer_reward_taken_by_harvester_as_fee` function, called when the override is present, only logs a warning when the fee quality threshold is not met â€” it does not return early, raise an exception, or prevent the override from being used: [4](#0-3) 

The farmer subsequently signs the `FoliageBlockData` hash (which now contains the attacker's puzzle hash) without verifying the embedded `farmer_reward_puzzle_hash`. Consensus validation only checks that the foliage block data signature is valid against the plot public key â€” it does not constrain what address `farmer_reward_puzzle_hash` may contain: [5](#0-4) 

The `farmer_reward_address_override` field is defined in the harvester protocol with no restrictions on its value: [6](#0-5) 

### Impact Explanation
**Critical â€” Unauthorized reward diversion affecting XCH.** A malicious third-party harvester can redirect the farmer's 1/8 block reward to any arbitrary puzzle hash on every block the farmer wins. The farmer has no on-chain or protocol-level recourse: the block is valid, the signature is valid (the plot key is a 2-of-2 between farmer and harvester, and the harvester controls its half), and consensus does not constrain `farmer_reward_puzzle_hash` to any farmer-owned address. The farmer loses their entire reward for every block farmed through the malicious harvester.

### Likelihood Explanation
**Medium.** The attack requires the farmer to connect to a malicious third-party harvester. CHIP-22 was explicitly designed to support third-party harvester services, making this a realistic deployment scenario. The attacker needs no special privileges beyond operating a harvester that the farmer connects to.

### Recommendation
1. In `new_proof_of_space`, if the fee quality check in `notify_farmer_reward_taken_by_harvester_as_fee` fails (i.e., `fee_quality > fee_threshold`), the farmer should **reject** the proof and not proceed with block creation rather than merely logging a warning.
2. The farmer should verify that `farmer_reward_address_override` matches the harvester's pre-registered address (if any fee agreement exists), not accept an arbitrary puzzle hash.
3. When `include_signature_source_data` is `True` and the full node provides the actual `FoliageBlockData` in `RequestSignedValues`, the farmer should verify that `foliage_block_data.farmer_reward_puzzle_hash` matches the expected value before signing.

### Proof of Concept
1. Attacker operates a third-party harvester service and a farmer connects to it.
2. When a valid proof of space is found, the harvester sends `NewProofOfSpace` with `farmer_reward_address_override = attacker_puzzle_hash` and `fee_info` set to an arbitrarily high threshold.
3. `FarmerAPI.new_proof_of_space` calls `notify_farmer_reward_taken_by_harvester_as_fee`, which logs a warning but does not block execution.
4. The farmer sends `RequestSignatures` to the harvester for the SP hashes.
5. The harvester responds with `RespondSignatures` containing `farmer_reward_address_override = attacker_puzzle_hash`.
6. `_process_respond_signatures` unconditionally sets `farmer_reward_address = attacker_puzzle_hash` and emits `DeclareProofOfSpace` to the full node.
7. The full node constructs an unfinished block with `farmer_reward_puzzle_hash = attacker_puzzle_hash` and sends `RequestSignedValues` back to the farmer.
8. The farmer signs the foliage block data hash (which embeds the attacker's address) without inspecting it.
9. The full node finalizes and propagates the block. The farmer's 1/8 XCH block reward is paid to the attacker's address.
