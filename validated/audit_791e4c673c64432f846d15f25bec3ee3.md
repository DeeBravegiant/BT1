### Title
Cross-Request Replay of `VerifyForeignTransactionResponse` via Unbound `payload_hash` — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is valid over `response.payload_hash` using the **root** public key, but never verifies that `response.payload_hash` actually corresponds to the foreign-chain transaction described in `request`. A single attested MPC participant (below threshold) can replay a legitimately produced `(payload_hash, signature)` pair from one resolved request to drain a *different* pending request, causing its callers to receive a verification result for the wrong foreign-chain transaction.

### Finding Description

In `respond` (regular signatures) the contract derives the expected verification key from the request's tweak:

```rust
// crates/contract/src/lib.rs ~line 598
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response, payload_hash, &expected_public_key,
)
```

Because the tweak encodes the caller, derivation path, and payload, a signature produced for request A is cryptographically invalid for request B.

`respond_verify_foreign_tx` performs no such binding:

```rust
// crates/contract/src/lib.rs ~line 723-734
let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
    .expect("Secp256k1 variant always has a secp256k1 key");
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &secp_pk,
).is_ok()
```

The signature is checked only against the **root** key with no tweak, and `response.payload_hash` is accepted verbatim — the contract never recomputes the expected hash from `request` to confirm they match.

`ForeignTxSignPayload` is designed to bind the hash to the request:

```rust
// crates/contract/tests/sandbox/foreign_chain_request.rs ~line 76-80
let (payload, response) = sign_foreign_tx_response(
    &verify_request.request,
    extracted_values,
    foreign_tx_key.as_secp256k1(),
);
```

But that binding is enforced only off-chain by honest nodes; the on-chain contract never re-derives or checks it.

After signature verification passes, the response is fanned out to all queued callers:

```rust
// crates/contract/src/lib.rs ~line 749-753

    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)

### Impact Explanation

A single attested MPC participant executes the following replay:

1. Request A (`verify_foreign_transaction(tx_A)`) is submitted and legitimately resolved; the participant observes the on-chain `respond_verify_foreign_tx` call and extracts `(payload_hash_A, signature_A)`.
2. Request B (`verify_foreign_transaction(tx_B)`) is pending.
3. The attacker calls `respond_verify_foreign_tx(request_B, {payload_hash_A, signature_A})`.
4. The contract looks up `request_B` in `pending_verify_foreign_tx_requests` — found.
5. It verifies `signature_A` over `payload_hash_A` against the root key — **valid** (the signature was produced by the MPC network).
6. It resolves all yields for `request_B` with `{payload_hash_A, signature_A}`.
7. Every caller of `request_B` receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to `tx_A`, not `tx_B`.

Callers that use the returned `(payload_hash, signature)` pair to authorize a bridge operation will execute against the wrong foreign-chain transaction, enabling forged foreign-chain verification and potential double-spend or invalid bridge execution.

Impact: **High** — matches "Cross-chain replay, forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."

### Likelihood Explanation

The attacker must be a single attested MPC participant — no threshold collusion is required. All on-chain `respond_verify_foreign_tx` calls are public; any participant can observe and extract a prior `(payload_hash, signature)` pair. The attack is therefore trivially executable by any one compromised or malicious node.

### Recommendation

Inside `respond_verify_foreign_tx`, recompute the expected payload hash from `request` (mirroring the off-chain `ForeignTxSignPayload::compute_msg_hash` logic) and assert that it equals `response.payload_hash` before proceeding to signature verification. This binds the response to the specific request, making cross-request replay cryptographically impossible — the same defence that the `request.tweak` provides in `respond`.

### Proof of Concept

```
Alice  → contract: verify_foreign_transaction(tx_A)   // pending request A
Bob    → contract: verify_foreign_transaction(tx_B)   // pending request B

// Honest nodes process tx_A; attacker observes the respond call:
Honest → contract: respond_verify_foreign_tx(request_A, {hash_A, sig_A})
// request_A is now drained; hash_A and sig_A are visible on-chain.

// Attacker replays the same response for request_B:
Attacker → contract: respond_verify_foreign_tx(request_B, {hash_A, sig_A})
// Contract checks: sig_A valid over hash_A with root key? YES.
// Contract resolves request_B → Bob receives {hash_A, sig_A}.

// Bob's bridge contract now holds a valid MPC signature over hash_A
// (tx_A's data) while believing it verified tx_B.
```

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
