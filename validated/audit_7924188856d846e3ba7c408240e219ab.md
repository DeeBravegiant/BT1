### Title
Unbounded while-loop in pool wallet singleton sync controlled by peer-provided data causes wallet hang - (File: chia/wallet/wallet_state_manager.py)

### Summary
In `chia/wallet/wallet_state_manager.py`, the `_add_coin_states()` method contains an unbounded `while curr_coin_state.spent_height is not None:` loop (lines 2089–2123) that processes pool wallet singleton state transitions. The loop's termination condition is entirely controlled by data returned by the connected peer. A malicious peer can keep returning coin states with `spent_height is not None`, causing the wallet's sync coroutine to loop indefinitely with no iteration cap.

### Finding Description
When `_add_coin_states()` processes a spent pool wallet singleton coin (identified by `record.wallet_type is WalletType.POOLING_WALLET` and `coin_state.coin.amount == uint64(1)`), it enters the following loop:

```python
while curr_coin_state.spent_height is not None:
    cs = await fetch_coin_spend_for_coin_state(curr_coin_state, peer)
    success = await singleton_wallet.apply_state_transition(cs, ...)
    if not success:
        break
    new_singleton_coin = get_most_recent_singleton_coin_from_coin_spend(cs)
    if new_singleton_coin is None:
        break
    new_singleton_name = new_singleton_coin.name()
    ...
    new_coin_state = await self.wallet_node.get_coin_state(
        [new_singleton_name], peer=peer, fork_height=fork_height
    )
    assert len(new_coin_state) == 1
    curr_coin_state = new_coin_state[0]   # ← peer-controlled
``` [1](#0-0) 

Every iteration fetches `curr_coin_state` from the peer via `get_coin_state()`. There is no iteration counter, no maximum depth, and no timeout on the loop body itself. The only exit paths are:
1. `apply_state_transition` returns `False`
2. `get_most_recent_singleton_coin_from_coin_spend` returns `None`
3. The peer eventually returns a coin state with `spent_height is None`

A malicious peer controls all three: it supplies the coin spend bytes via `fetch_coin_spend_for_coin_state` and the next coin state via `get_coin_state`. By crafting responses that pass `apply_state_transition` and always returning a new coin state with `spent_height is not None`, the peer keeps the loop alive indefinitely.

A structurally identical unbounded loop exists in `get_latest_singleton_coin_spend()`:

```python
while coin_state.spent_height is not None:
    coin_state_list = await self.wallet_node.fetch_children(coin_state.coin.name(), peer=peer)
    ...
    coin_state = odd_coin   # ← peer-controlled
``` [2](#0-1) 

### Impact Explanation
The wallet's async sync task is blocked inside the infinite loop. Because `_add_coin_states` holds the `db_wrapper` writer lock during this loop, no other wallet database writes can proceed. The wallet cannot process further coin state updates, confirm transactions, update pool membership state, or respond to RPC calls that require the lock. This constitutes a **long-lived inability for honest wallets to process sync updates and pool actions** — a High-severity impact under the allowed scope. Additionally, repeated calls to `apply_state_transition` with peer-crafted coin spends may corrupt the local pool wallet state (pool membership, payout target, singleton lineage), directly affecting pool reward accounting.

### Likelihood Explanation
The attacker must be the peer the wallet connects to. In the Chia network, wallets connect to full nodes that are not inherently trusted. An attacker who operates a full node (or performs a BGP/DNS hijack to redirect the wallet's connection) can trigger this path whenever the wallet has a pool wallet and the attacker sends a coin state update for the pool singleton with `spent_height is not None`. No key material, admin access, or cryptographic break is required.

### Recommendation
Add an explicit iteration cap to the loop, analogous to how the mempool uses `MAX_SKIPPED_ITEMS` and block building uses a timeout:

```python
MAX_SINGLETON_SYNC_DEPTH = 1000
depth = 0
while curr_coin_state.spent_height is not None:
    if depth >= MAX_SINGLETON_SYNC_DEPTH:
        log.warning("Pool wallet singleton sync depth exceeded limit, aborting")
        break
    depth += 1
    ...
```

The same fix should be applied to `get_latest_singleton_coin_spend()` and `DataLayerWallet._track_new_launcher_id()`.

### Proof of Concept
1. Attacker runs a malicious full node.
2. Victim wallet with a pool wallet connects to the attacker's node.
3. Attacker sends a `CoinState` update for the pool singleton with `spent_height = 100`, `amount = 1`.
4. Wallet enters `_add_coin_states` → pool wallet branch → the unbounded while-loop.
5. Each iteration: attacker returns a crafted `CoinSpend` that passes `apply_state_transition` (e.g., a valid-looking pool state transition to a new singleton coin), and returns a new `CoinState` for the derived singleton with `spent_height = 101, 102, …` indefinitely.
6. The wallet's sync coroutine never exits the loop; the `db_wrapper` writer lock is held permanently; all wallet operations requiring the lock are blocked. [3](#0-2)
