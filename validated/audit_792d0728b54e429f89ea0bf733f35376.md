Audit Report

## Title
Stale `rsETHPrice` Used in `getRsETHAmountToMint` Allows Depositors to Capture Accrued Yield From Existing Holders - (`contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle` stores the rsETH/ETH exchange rate in a cached storage variable `rsETHPrice` that is only updated when `updateRSETHPrice()` is explicitly called. `LRTDepositPool.getRsETHAmountToMint()` divides by this cached value, and neither `depositETH()` nor `depositAsset()` refreshes the price before computing the mint amount. When staking rewards accrue between price updates, the stale (lower) denominator causes new depositors to receive more rsETH than they are entitled to, diluting the yield that belongs to existing holders.

## Finding Description

`LRTOracle` stores the exchange rate as a public state variable:

```solidity
// LRTOracle.sol:28
uint256 public override rsETHPrice;
```

This variable is only written inside `_updateRsETHPrice()`, which is invoked exclusively through `updateRSETHPrice()` (public, keeper-driven) or `updateRSETHPriceAsManager()` (manager-only). Neither is called atomically with deposits.

The deposit flow is:

`depositETH()` / `depositAsset()` → `_beforeDeposit()` → `getRsETHAmountToMint()`:

```solidity
// LRTDepositPool.sol:520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The true current price is computed inside `_updateRsETHPrice()` by summing all TVL across NodeDelegators via `_getTotalEthInProtocol()` (a `private view`) and dividing by total rsETH supply. Between two consecutive keeper calls, ETH staking yield and EigenLayer AVS rewards increase the real TVL while `rsETHPrice` in storage remains at the pre-reward value.

An unprivileged depositor who deposits during this window uses the stale (lower) denominator, receiving more rsETH than the true price would entitle them to. When `updateRSETHPrice()` is eventually called, the new price reflects the diluted supply, and existing holders receive less yield than they earned.

Additionally, the `pricePercentageLimit` guard in `_updateRsETHPrice()` can cause non-manager callers to revert if the accrued reward pushes the price above the daily threshold, potentially extending the staleness window and worsening the discrepancy.

No existing check in `depositETH()` or `depositAsset()` forces a price refresh before minting.

## Impact Explanation

**High — Theft of unclaimed yield.**

Concrete example:
- TVL = 1000 ETH, supply = 1000 rsETH → `rsETHPrice = 1.00`
- 10 ETH staking rewards accrue → true price = 1.01, stored price = 1.00
- Attacker deposits 100 ETH → receives `100 / 1.00 = 100 rsETH` instead of `100 / 1.01 ≈ 99.01 rsETH`
- After price update: total ETH = 1110, total supply = 1100, new price ≈ 1.009
- Original holders' 1000 rsETH is worth ≈ 1009 ETH instead of the 1010 ETH they earned
- Attacker's 100 rsETH is worth ≈ 100.9 ETH on a 100 ETH deposit — capturing ~0.9 ETH of others' yield

The excess rsETH represents a direct claim on TVL that was earned by prior holders. This is repeatable on every deposit made while the price is stale.

## Likelihood Explanation

`updateRSETHPrice()` is driven by an off-chain keeper on a periodic schedule, not per-block. The protocol's own `pricePercentageLimit` guard and daily fee-minting logic confirm that price updates are intentionally periodic. Any deposit in the window between reward accrual and the next keeper call exploits this. A sophisticated attacker can monitor `getTotalAssetDeposits()` and the asset oracle prices on-chain to compute the true TVL, compare it to `rsETHPrice * totalSupply`, and time deposits to maximize the discrepancy. No privileged access is required; only a standard `depositETH{value: ...}()` call is needed.

## Recommendation

At the start of `depositETH()` and `depositAsset()`, call `updateRSETHPrice()` (or expose `_updateRsETHPrice()` as an internal call reachable from `LRTDepositPool`) before computing the mint amount. This ensures the denominator in `getRsETHAmountToMint` always reflects the current TVL. Alternatively, compute the mint amount using a freshly calculated price derived directly from `_getTotalEthInProtocol()` divided by total rsETH supply, bypassing the cached storage variable entirely.

## Proof of Concept

1. Deploy/fork with: TVL = 1000 ETH, rsETH supply = 1000, `rsETHPrice = 1e18`.
2. Simulate 10 ETH of staking rewards accruing to the EigenPod (increasing `getEffectivePodShares()` by 10 ETH). `rsETHPrice` in storage remains `1e18`.
3. Call `LRTDepositPool.depositETH{value: 100 ether}(0, "")` as an unprivileged attacker.
4. Observe: `getRsETHAmountToMint` computes `(100e18 * 1e18) / 1e18 = 100e18` rsETH minted.
5. Call `LRTOracle.updateRSETHPrice()`. New price = `(1110e18) / 1100e18 ≈ 1.009e18`.
6. Assert: attacker's 100 rsETH is worth `100 * 1.009 = 100.9 ETH`; original holders' 1000 rsETH is worth `1009 ETH` instead of `1010 ETH`.
7. Difference of ~1 ETH transferred from existing holders to attacker, confirming theft of unclaimed yield.

Foundry invariant test: assert that for any deposit, `rsETH minted * lrtOracle.rsETHPrice() <= deposit value in ETH * 1e18` — this invariant will break whenever `rsETHPrice` is stale relative to true TVL.
