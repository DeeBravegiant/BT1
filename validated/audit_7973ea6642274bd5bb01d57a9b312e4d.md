Audit Report

## Title
Permanent ETH Freeze in `LRTWithdrawalManager` When Withdrawing User Is a Contract Without `receive()` — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

The withdrawal lifecycle in `LRTWithdrawalManager` is split across two separate transactions: `unlockQueue` (which permanently burns rsETH and moves ETH into the manager) and `completeWithdrawal` (which delivers ETH to the user). If the withdrawing address is a smart contract without a `receive()` or `fallback()` function, the ETH delivery in `_transferAsset` always reverts via `revert EthTransferFailed()`, but the rsETH burned in the prior `unlockQueue` call is not recoverable. The ETH remains permanently locked in `LRTWithdrawalManager` with no on-chain escape hatch.

## Finding Description

**Step 1 — `unlockQueue` (operator-triggered):**

`_unlockWithdrawalRequests` marks requests as unlocked and increments `unlockedWithdrawalsCount[asset]`. After the loop, rsETH held by the manager is permanently burned and the corresponding ETH is redeemed from the unstaking vault into the manager's balance: [1](#0-0) 

At this point rsETH is gone and ETH sits in `LRTWithdrawalManager`.

**Step 2 — `completeWithdrawal` / `completeWithdrawalForUser` (user or operator-triggered):**

`_processWithdrawalCompletion` pops the nonce from the user's queue, deletes the request, decrements `unlockedWithdrawalsCount`, and then calls `_transferAsset`: [2](#0-1) 

`_transferAsset` for ETH uses a low-level `call`: [3](#0-2) 

If the recipient contract has no `receive()` or `fallback()`, `call{value: amount}("")` returns `false`, triggering `revert EthTransferFailed()`. Because this revert unwinds the entire transaction, all state mutations in `_processWithdrawalCompletion` — `popFront()`, `delete withdrawalRequests[requestId]`, and `unlockedWithdrawalsCount[asset]--` — are rolled back. The withdrawal request remains in the unlocked queue indefinitely.

**No recovery path:**

- `sweepRemainingAssets` is gated by `hasUnlockedWithdrawals(asset)`, which returns `true` as long as `unlockedWithdrawalsCount[asset] > 0`. Since the revert prevents the decrement, sweeping is permanently blocked: [4](#0-3) 

- `completeWithdrawalForUser` (operator path) calls the same `_processWithdrawalCompletion` and hits the same revert: [5](#0-4) 

- There is no admin function to redirect a withdrawal to a different recipient or cancel an already-unlocked request.

## Impact Explanation

**Critical — Permanent freezing of funds.** The user's rsETH is irreversibly burned during `unlockQueue`. The corresponding ETH is trapped in `LRTWithdrawalManager` with no on-chain mechanism to release it. The user loses their entire withdrawal amount permanently, and the locked ETH cannot be swept to treasury either, as `sweepRemainingAssets` is blocked by the non-zero `unlockedWithdrawalsCount`.

## Likelihood Explanation

**Low.** The affected user must be a smart contract that: (1) holds rsETH and approves `LRTWithdrawalManager`, (2) calls `initiateWithdrawal` with `asset == ETH_TOKEN`, and (3) does not implement a `receive()` or `fallback()` function. This is realistic for protocol integrations, vaults, or smart contract wallets that hold rsETH but were not designed to receive raw ETH. The developer comment at line 191 — *"Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH"* — shows awareness of ETH-delivery edge cases but incorrectly dismisses the permanent-freeze scenario, as the gas grief framing misses the irreversible rsETH burn that precedes the failed delivery. [6](#0-5) 

## Recommendation

Validate that the withdrawing address can receive ETH **before** accepting the withdrawal request in `initiateWithdrawal`. Options include:

1. **EOA-only check:** At the start of `initiateWithdrawal` for `asset == ETH_TOKEN`, require `msg.sender.code.length == 0`.
2. **Zero-value probe:** Attempt a zero-value ETH transfer to `msg.sender` at initiation time and revert if it fails.
3. **Admin escape hatch:** Add a privileged function that can redirect an already-unlocked withdrawal to an alternate recipient when the primary recipient is unable to receive ETH, allowing recovery without a contract upgrade.

## Proof of Concept

```solidity
// Contract that holds rsETH but has no receive() function
contract NoReceiveWallet {
    function initiateETHWithdrawal(
        address withdrawalManager,
        address rsETH,
        uint256 rsETHAmount
    ) external {
        IERC20(rsETH).approve(withdrawalManager, rsETHAmount);
        ILRTWithdrawalManager(withdrawalManager)
            .initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
    }
}
```

**Sequence:**
1. `NoReceiveWallet.initiateETHWithdrawal(...)` — rsETH transferred to `LRTWithdrawalManager`, withdrawal request created.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH **permanently burned** from `LRTWithdrawalManager`; ETH moved from `LRTUnstakingVault` into `LRTWithdrawalManager`.
3. `NoReceiveWallet` calls `completeWithdrawal(ETH_TOKEN, "")` — `_transferAsset` attempts `payable(NoReceiveWallet).call{value: amount}("")`; call returns `false`; `revert EthTransferFailed()` fires; entire transaction reverts; withdrawal request remains in queue.
4. Operator calls `completeWithdrawalForUser(ETH_TOKEN, address(NoReceiveWallet), "")` — same revert.
5. Manager calls `sweepRemainingAssets(ETH_TOKEN)` — reverts because `hasUnlockedWithdrawals(ETH_TOKEN) == true`.
6. **ETH is permanently locked in `LRTWithdrawalManager`. rsETH is permanently burned.**

**Foundry test plan:** Deploy `NoReceiveWallet` on a mainnet fork, seed it with rsETH, call `initiateETHWithdrawal`, advance blocks past `withdrawalDelayBlocks`, call `unlockQueue`, then assert that both `completeWithdrawal` and `completeWithdrawalForUser` revert with `EthTransferFailed`, and that `address(withdrawalManager).balance` equals the locked ETH amount with no path to recover it.
