Audit Report

## Title
ETH Deposit Limit Check Omits Incoming `amount`, Allowing Limit Bypass - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric condition: the ETH branch checks only whether the existing total already exceeds the limit (`totalAssetDeposits > limit`), while the ERC20 branch correctly includes the incoming deposit (`totalAssetDeposits + amount > limit`). Any depositor can push ETH deposits past the configured `depositLimitByAsset` cap by submitting a deposit that, when added to the current total, would exceed the limit — but the guard never performs that addition for ETH.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` ( [1](#0-0) ), the ETH branch at line 679 evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` — omitting `amount`. The ERC20 branch at line 681 correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`.

This function is the sole limit guard, called from `_beforeDeposit` at line 661 ( [2](#0-1) ), which is invoked by `depositETH` passing `msg.value` as the amount ( [3](#0-2) ).

When `totalAssetDeposits` is at or below the limit, the ETH branch always returns `false` regardless of `amount`, so `_beforeDeposit` never reverts with `MaximumDepositLimitReached` for ETH. `getAssetCurrentLimit` confirms the limit is intended as a hard cap by computing remaining capacity as `depositLimit - totalAssetDeposits` ( [4](#0-3) ), making the asymmetry a clear implementation defect.

## Impact Explanation
The `depositLimitByAsset` cap is a risk-management invariant. When bypassed for ETH, the protocol accepts and stakes more ETH in EigenLayer than the configured ceiling permits, violating the protocol's own accounting guarantee. No funds are directly stolen, but the contract fails to deliver the promised deposit-cap guarantee.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
No special role or privilege is required. Any ordinary depositor calling `depositETH` with a `msg.value` large enough that `totalAssetDeposits + msg.value > depositLimit` — while `totalAssetDeposits <= depositLimit` — will silently bypass the cap. The condition is triggered on every such ETH deposit and is fully repeatable.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC20 branch:

```diff
- return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+ return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

## Proof of Concept
```
State:
  depositLimitByAsset(ETH) = 100 ETH
  totalAssetDeposits(ETH)  =  95 ETH

Attacker calls: depositETH{value: 10 ether}(0, "")

ETH branch check: 95 > 100  →  false  →  no revert
New total:        105 ETH    →  exceeds limit by 5 ETH
rsETH minted, ETH staked in EigenLayer

ERC20 equivalent: 95 + 10 > 100  →  true  →  MaximumDepositLimitReached revert
```

Foundry fuzz test plan: fuzz `depositAmount` over `[1, type(uint128).max]` with `totalAssetDeposits` set to `depositLimit - 1`; assert that `depositETH{value: depositAmount}` reverts with `MaximumDepositLimitReached` whenever `depositAmount > 1`. The current code will fail this assertion for any `depositAmount` in that range.
