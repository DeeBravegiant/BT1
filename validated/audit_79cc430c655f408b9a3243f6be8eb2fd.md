### Title
Peer-Controlled FAILED Acks Permanently Suppress Unconfirmed Spend Bundles via `increment_sent` / `is_valid()` — (`chia/wallet/wallet_transaction_store.py`)

### Summary

Six attacker-controlled full-node peers, each legitimately receiving a wallet's spend bundle and returning `MempoolInclusionStatus.FAILED` with a non-fee error, can cause `increment_sent` to mark the transaction `confirmed=True, confirmed_at_height=0`. Once in that state, `get_not_sent` never resubmits it (`WHERE confirmed=0`), and `rollback_to_block` never deletes it (`DELETE WHERE confirmed_at_height > height` skips height 0). The spend bundle is permanently suppressed: coins remain unspent on-chain while the wallet reports the transaction as confirmed, creating an irrecoverable accounting divergence.

---

### Finding Description

**`is_valid()` threshold and the marking path**

`TransactionRecordOld.is_valid()` returns `False` once `sent_to` accumulates `minimum_send_attempts = 6` entries, none of which is a SUCCESS or a fee-related error: [1](#0-0) [2](#0-1) 

When `is_valid()` returns `False`, `increment_sent` immediately writes `confirmed=True, confirmed_at_height=0` to the DB: [3](#0-2) 

**The `transaction_ack` guard — necessary but insufficient**

The wallet does enforce that acks must correspond to in-flight sends: [4](#0-3) 

This prevents purely unsolicited acks. However, it does **not** prevent a legitimately connected peer from receiving the spend bundle (which the wallet sends proactively) and then responding with `FAILED`. Six distinct attacker-controlled peers, each receiving one send, each returning one FAILED ack, satisfy the guard and accumulate six entries in `sent_to`.

**`rollback_to_block` does not recover height-0 records** [5](#0-4) 

`DELETE WHERE confirmed_at_height > height` evaluates `0 > 0` as `False` for any `height ≥ 0`. A reorg to any block height leaves the poisoned record intact.

**`get_not_sent` permanently excludes the record** [6](#0-5) 

`WHERE confirmed=0` excludes the record once `confirmed=True`, so the wallet never resubmits it.

**`add_transaction_record` cache inconsistency**

When `increment_sent` calls `add_transaction_record` with `confirmed=True`, the `unconfirmed_txs` in-memory cache is not pruned (the removal only happens in `set_confirmed`): [7](#0-6) [8](#0-7) 

The stale cache entry does not provide a recovery path because `get_not_sent` queries the DB, not the cache.

---

### Impact Explanation

- Coins referenced by the spend bundle remain unspent on-chain.
- The wallet permanently displays the transaction as confirmed (at height 0), causing accounting divergence.
- No standard wallet operation (`rollback_to_block`, `get_not_sent`, `get_all_unconfirmed`) recovers or resubmits the bundle.
- Affected transaction types: any `OUTGOING_TX`, `OUTGOING_CAT`, clawback, or offer-settlement spend bundle.

This matches **High: Corruption of wallet sync state / offer settlement state with direct security impact**.

---

### Likelihood Explanation

An attacker must run 6 full-node peers and get the victim wallet to connect to all of them — a Sybil attack. Chia wallets connect to up to ~8 peers discovered via DNS seeding or `wallet_peers`. An attacker who seeds 6 malicious nodes into the peer discovery pool (or who operates on a network segment where the victim has limited peer choices) can achieve this without any key compromise or privileged access. Each malicious peer simply receives the spend bundle normally and replies with `FAILED` + any non-fee `Err` code (e.g., `GENERATOR_RUNTIME_ERROR`).

---

### Recommendation

1. **Do not use `confirmed=True` to represent "permanently failed"** — introduce a distinct `TransactionStatus` enum value (e.g., `FAILED_PERMANENTLY`) so that `get_not_sent`, `rollback_to_block`, and UI layers can distinguish it from genuine on-chain confirmation.
2. **Fix `rollback_to_block`** to also reset or delete records with `confirmed=True AND confirmed_at_height=0` that were not confirmed by a block event.
3. **Cap peer influence on `sent_to`**: require at least one SUCCESS or PENDING from a *trusted* peer before accepting FAILED verdicts from untrusted peers as authoritative, or require the FAILED count to come from peers whose node IDs are independently verified.
4. **Sync `unconfirmed_txs` cache** in `increment_sent` when marking confirmed, mirroring the removal done in `set_confirmed`.

---

### Proof of Concept

```python
# Pseudocode unit test
store = await WalletTransactionStore.create(db_wrapper, config)
await store.add_transaction_record(pending_tx)  # confirmed=False

# Simulate 6 distinct peers each returning FAILED
for i in range(6):
    await store.increment_sent(
        pending_tx.name, f"peer{i}", MempoolInclusionStatus.FAILED, Err.GENERATOR_RUNTIME_ERROR
    )

# Assert tx is now "confirmed" at height 0
record = await store.get_transaction_record(pending_tx.name)
assert record.confirmed is True
assert record.confirmed_at_height == 0

# Assert get_not_sent never returns it
not_sent = await store.get_not_sent()
assert pending_tx.name not in [r.name for r in not_sent]

# Assert rollback_to_block(0) does NOT delete it
await store.rollback_to_block(0)
record = await store.get_transaction_record(pending_tx.name)
assert record is not None  # still present — coins permanently lost from wallet perspective
```

    def is_valid(self) -> bool:
        if len(self.sent_to) < minimum_send_attempts:
            # we haven't tried enough peers yet
            return True
        if any(x[1] == MempoolInclusionStatus.SUCCESS.value for x in self.sent_to):

            return True
        return False

    async def set_confirmed(self, tx_id: bytes32, height: uint32):
        """
        Updates transaction to be confirmed.
        """
        current: TransactionRecord | None = await self.get_transaction_record(tx_id)
        if current is None:
            return None
        if current.confirmed_at_height == height:
            return
        tx: TransactionRecord = dataclasses.replace(current, confirmed_at_height=height, confirmed=True)
        await self.add_transaction_record(tx)
        self.unconfirmed_txs.remove(get_light_transaction_record(current))
