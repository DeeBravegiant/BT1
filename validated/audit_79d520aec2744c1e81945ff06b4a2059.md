### Title
Permissionless `directDeposits` Enables Griefing of Stake Credential Deregistration, Locking Key Deposits — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in the transaction body that allows **any transaction author** to push ADA into any registered stake account without authorization from the recipient. Because deregistration of a stake credential requires a zero account balance, an attacker can front-run a victim's deregistration transaction with a tiny direct deposit, causing the deregistration to fail. This can be repeated indefinitely, permanently locking the victim's key deposit for as long as the attacker is willing to spend minimal lovelace.

---

### Finding Description

The Dijkstra era adds `dtbrDirectDeposits :: !DirectDeposits` to the top-level and sub-transaction body: [1](#0-0) 

`DirectDeposits` is a `Map AccountAddress Coin` — a mapping from any registered reward account address to an ADA amount to be credited: [2](#0-1) 

In `dijkstraEntitiesTransition`, after certificates are processed, direct deposits are validated and applied:

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts

pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [3](#0-2) 

The only validation performed is `directDepositsMissingAccounts`, which checks only that the target account is **registered** — there is no authorization check from the recipient: [4](#0-3) 

`applyDirectDeposits` unconditionally adds the deposited amount to the account balance: [5](#0-4) 

Separately, the `ConwayUnRegCert` deregistration rule (reused in Dijkstra via `Cardano.Ledger.Dijkstra.Rules.Deleg`) enforces that the account balance must be zero before deregistration is permitted: [6](#0-5) 

A user wishing to deregister with a non-zero balance must include a withdrawal in the same transaction to drain the account first. The withdrawal validation (`withdrawalsThatDoNotDrainAccounts`) requires the withdrawal amount to **exactly equal** the account balance: [7](#0-6) 

An attacker who front-runs the victim's transaction with a 1-lovelace direct deposit causes the victim's withdrawal amount to no longer match the new balance, triggering `ShelleyIncompleteWithdrawals`. The deregistration fails, and the key deposit remains locked.

The same attack applies to sub-transactions in the Dijkstra era, where `dijkstraSubEntitiesTransition` applies the same pattern: [8](#0-7) 

---

### Impact Explanation

**Medium. Attacker-controlled transactions modify deposits and withdrawals outside design parameters.**

An unprivileged attacker can continuously front-run any victim's stake credential deregistration transaction by including a `directDeposits` entry targeting the victim's account. Each front-run costs the attacker only 1 lovelace plus a transaction fee, while the victim's key deposit (currently 2 ADA on mainnet) remains locked indefinitely. The victim cannot deregister as long as the attacker persists. The victim's withdrawal amount must be updated after each failed attempt, but the attacker can always invalidate it again before it lands on-chain.

For script-based stake credentials whose withdrawal scripts enforce a strict balance condition (e.e., the script checks the withdrawal equals a specific expected amount), a single attacker deposit permanently prevents the script from authorizing any valid withdrawal, causing a hard permanent freeze of the account balance.

---

### Likelihood Explanation

The attack requires no special privileges — any Dijkstra-era transaction author can include `directDeposits` targeting any registered account. The cost per griefing attempt is minimal (1 lovelace + fees). The victim's account address is publicly visible on-chain. Mempool monitoring to identify deregistration transactions is straightforward. The attack is therefore realistic for any motivated adversary targeting a specific user's key deposit.

---

### Recommendation

Add a recipient-authorization check to `directDeposits` processing. Specifically, require that each target `AccountAddress` in `directDeposits` is either:

1. Authorized by a witness for the target stake credential (key or script), or
2. Explicitly opted-in via a prior on-chain registration flag.

Alternatively, if push-payments to accounts are intentional by design, the deregistration rule should be updated to allow deregistration even with a non-zero balance (returning the balance to the owner), so that permissionless deposits cannot block the deregistration path.

---

### Proof of Concept

1. Victim registers stake credential `cred` and earns 0 rewards (or withdraws all rewards). Account balance = 0.
2. Victim constructs transaction `T_victim`:
   - `certsTxBodyL = [ConwayUnRegCert cred SNothing]`
   - `withdrawalsTxBodyL = Withdrawals [(accountAddress cred, Coin 0)]`
3. Attacker observes `T_victim` in the mempool.
4. Attacker constructs transaction `T_attack` with higher fee (to be processed first):
   - `directDepositsTxBodyL = DirectDeposits [(accountAddress cred, Coin 1)]`
5. `T_attack` is applied first: victim's account balance becomes `Coin 1`.
6. `T_victim` is applied next: withdrawal of `Coin 0` does not drain the account (balance is `Coin 1`), triggering `ShelleyIncompleteWithdrawals`. The `ConwayUnRegCert` is never reached.
7. Victim's key deposit remains locked. Attacker repeats step 4 for every subsequent deregistration attempt by the victim. [9](#0-8) [10](#0-9) [11](#0-10)

-- | Returns `Nothing` iff every credential targeted by the supplied
-- `DirectDeposits` is a registered account. Otherwise it returns the subset of
-- direct deposits whose target credential is not registered.
directDepositsMissingAccounts ::
  EraAccounts era =>
  DirectDeposits ->
  Accounts era ->
  Maybe DirectDeposits
directDepositsMissingAccounts (DirectDeposits dds) accounts
  | Map.foldrWithKey' checkRegistered True dds = Nothing
  | otherwise = Just $ DirectDeposits $ Map.foldrWithKey' collectMissing Map.empty dds
  where
    isRegistered (AccountAddress _ (AccountId credential)) =
      isAccountRegistered credential accounts
    checkRegistered addr _ acc = acc && isRegistered addr
    collectMissing addr amount acc
      | isRegistered addr = acc
      | otherwise = Map.insert addr amount acc
```
