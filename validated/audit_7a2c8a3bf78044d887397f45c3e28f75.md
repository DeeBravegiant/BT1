Audit Report

## Title
Token Decimal Mismatch in `viewSwapRsETHAmountAndFee` Causes Near-Total Loss of Deposited Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The token-deposit variant of `viewSwapRsETHAmountAndFee` in five pool contracts computes rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate`, implicitly assuming the deposited token has 18 decimals. When a non-18-decimal token (e.g., USDC with 6 decimals) is added as supported collateral via the intended admin operation `addSupportedToken`, any user who deposits that token receives `10^(18 - tokenDecimals)` times less rsETH than owed, while their full token balance is transferred to the pool and bridged to L1.

## Finding Description
**Root cause — dimensional mismatch in the rate formula:**

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalises the Chainlink answer to a 1e18-scaled price representing the ETH value of **one full human-readable token unit**:

```solidity
// ChainlinkOracleForRSETHPoolCollateral.sol L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

`rsETHToETHrate` is similarly 1e18-scaled. The pool formula is:

```solidity
// RSETHPoolV3.sol L334
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

For an 18-decimal token `amountAfterFee` is already in 1e18 units, so the formula is dimensionally consistent. For a 6-decimal token, `amountAfterFee` is in 1e6 units, making the result 1e12 times too small.

**Exploit path:**

1. Admin legitimately calls `addSupportedToken(USDC, oracle)` — an intended, non-malicious configuration operation gated by `TIMELOCK_ROLE`.
2. Any unprivileged user calls `deposit(USDC, 1_000e6, referralId)`.
3. `safeTransferFrom` moves 1 000 USDC from the user to the pool.
4. `viewSwapRsETHAmountAndFee` returns `rsETHAmount ≈ 3.8e5 wei` instead of `≈ 3.8e17 wei`.
5. `wrsETH.mint(msg.sender, rsETHAmount)` mints the negligible amount.
6. The pool's bridger subsequently bridges the full USDC balance to L1, permanently removing it from the user's reach.

**Existing checks are insufficient:**

- `limitDailyMint` uses the already-deflated `rsETHAmount`, so the daily cap is never approached and does not block the deposit.
- `addSupportedToken` only validates that the oracle returns a non-zero rate; it performs no decimal check on the token.
- No slippage or minimum-output parameter exists in `deposit`.

The same root cause is present identically in `RSETHPoolNoWrapper.sol` L311, `RSETHPool.sol` L346, `RSETHPoolV3ExternalBridge.sol` L452, and `RSETHPoolV3WithNativeChainBridge.sol` L370.

## Impact Explanation
**Critical — direct theft of user funds.** The deposited tokens are irrevocably transferred to the pool and bridged to L1 (accruing to the protocol), while the depositor receives a negligible rsETH credit worth `10^(tokenDecimals - 18)` of the deposited value. For USDC/USDT (6 decimals) this is a 99.9999999999 % loss per deposit. This matches the allowed impact: *"Direct theft of any user funds, whether at-rest or in-motion."*

## Likelihood Explanation
The prerequisite is an admin adding a non-18-decimal token via `addSupportedToken`. This is a legitimate, intended protocol operation — not a malicious act. USDC and USDT are the most natural L2 collateral candidates, and the contracts contain no code-level guard preventing their addition. Once added, every subsequent deposit by any unprivileged user triggers the loss automatically and repeatably. The admin need not be aware of the bug.

## Recommendation
Retrieve the token's `decimals()` and normalise `amountAfterFee` to 1e18 before applying the rate formula in all five pool variants:

```solidity
uint256 tokenDecimals = IERC20Metadata(token).decimals();
rsETHAmount = amountAfterFee * tokenToETHRate * 1e18
              / (rsETHToETHrate * 10 ** tokenDecimals);
```

Alternatively, enforce in `addSupportedToken` that only 18-decimal tokens may be added, but this would unnecessarily restrict the protocol's stated design.

## Proof of Concept
**Parameters:**
- USDC (6 decimals) added as supported token
- `tokenToETHRate` = 4e14 (1 USDC ≈ 0.0004 ETH, 1e18-normalised)
- `rsETHToETHrate` = 1.05e18
- User deposits 1 000 USDC → `amount = 1_000e6`, `feeBps = 0`

**Actual execution (`RSETHPoolV3.sol` L334):**
```
rsETHAmount = 1_000e6 * 4e14 / 1.05e18
            = 4e23 / 1.05e18
            ≈ 380_952  (~3.8e5 wei rsETH)
```

**Correct result:**
```
rsETHAmount = (1_000e6 * 1e18 / 1e6) * 4e14 / 1.05e18
            ≈ 3.8e17 wei rsETH  (~0.38 rsETH)
```

**Ratio:** user receives `3.8e5 / 3.8e17 = 1e-12` of the correct amount.
