Audit Report

## Title
`initiateWithdrawal()` permanently reverts post-deployment due to uninitialized `maxNumberOfWithdrawalsPerUser` — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal()` evaluates `userWithdrawalIds[msg.sender].length >= 0`, which is always `true` for a `uint256` array length, causing every call to revert with `WithdrawalLimitReached`. All staked KERNEL tokens are frozen until an admin manually calls `setMaxNumberOfWithdrawalsPerUser`.

## Finding Description
`initialize()` sets `kernelToken`, `rewardsToken`, and the admin role, but never assigns `maxNumberOfWithdrawalsPerUser`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L259-271
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    // maxNumberOfWithdrawalsPerUser left at default: 0
}
```

`initiateWithdrawal()` at L323 checks:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

With `maxNumberOfWithdrawalsPerUser == 0`, this becomes `length >= 0`. Since `length` is a `uint256` it is always `≥ 0`, so the condition is unconditionally `true` and every call reverts. The setter `setMaxNumberOfWithdrawalsPerUser` (L610-620) explicitly rejects `0` as an input, confirming that `0` is an invalid operational value — yet initialization leaves it there. `claimWithdrawal` requires a prior `initiateWithdrawal` record, so the entire withdrawal lifecycle is blocked.

## Impact Explanation
Every user who stakes KERNEL tokens cannot initiate a withdrawal until the admin calls `setMaxNumberOfWithdrawalsPerUser`. This constitutes a **temporary freezing of funds** for all stakers — a concrete, deterministic impact matching the allowed Medium scope. The freeze is not hypothetical; it is the default state of every fresh deployment.

## Likelihood Explanation
The condition triggers on the very first call to `initiateWithdrawal` after deployment, by any staker, with no special setup required. It is 100% reproducible and requires no attacker — any normal user attempting to exit their position hits it. The window of exposure lasts from deployment until the admin explicitly calls the setter.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a valid non-zero value inside `initialize()`:

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    ...
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
+   maxNumberOfWithdrawalsPerUser = 10; // any value in (0, MAX_WITHDRAWALS_PER_USER]
}
```

## Proof of Concept
1. Deploy `KernelDepositPool` via proxy; call `initialize(admin, kernelToken, rewardToken)`.
2. Confirm: `maxNumberOfWithdrawalsPerUser == 0`.
3. User calls `stake(1e18)` — succeeds; `balanceOf[user] == 1e18`.
4. User calls `initiateWithdrawal(1e18)`.
5. Execution reaches L323: `userWithdrawalIds[user].length >= maxNumberOfWithdrawalsPerUser` → `0 >= 0` → `true`.
6. Transaction reverts with `WithdrawalLimitReached`.
7. User's KERNEL is locked; `claimWithdrawal` is unreachable until admin calls `setMaxNumberOfWithdrawalsPerUser`.

**Foundry test sketch:**
```solidity
function test_initiateWithdrawal_blockedAtDeployment() public {
    // pool deployed and initialized without setting maxNumberOfWithdrawalsPerUser
    vm.startPrank(user);
    kernelToken.approve(address(pool), 1e18);
    pool.stake(1e18);
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(1e18);
    vm.stopPrank();
}
```
