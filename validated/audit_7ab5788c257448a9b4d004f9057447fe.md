### Title
Sub-Transaction Plutus Script ExUnits Excluded from Fee Calculation and Block Execution Limits in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`, `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Tx.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction may embed an arbitrary number of sub-transactions (`dtbrSubTransactions`), each of which can carry its own Plutus script witnesses and redeemers. The `totExUnits` function and every downstream check (`validateExUnitsTooBigUTxO`, `validateExUnits` in BBODY) only sum the ExUnits declared in the **top-level** transaction's witness set. Sub-transaction ExUnits are never counted. Because the minimum-fee formula uses `totExUnits` of the top-level transaction, and because sub-transactions carry no fee field at all, an unprivileged submitter can embed arbitrarily expensive Plutus scripts in sub-transactions and force every validating node to execute them at zero marginal cost. This is the direct Cardano analog of the gas-siphoning pattern described in the external report: the protocol provides "free" computation (validator CPU/memory) that an attacker can exploit by supplying attacker-controlled scripts through the sub-transaction mechanism.

---

### Finding Description

**`totExUnits` only reads top-level witnesses.** [1](#0-0) 

```haskell
totExUnits tx = foldMap snd $ tx ^. witsTxL . rdmrsTxWitsL . unRedeemersL
```

`witsTxL` on a `DijkstraTx TopTx` returns only the top-level witnesses. Sub-transactions are stored in `dtbrSubTransactions` inside the body, not in the top-level witness set, so their redeemers and ExUnits are invisible to `totExUnits`.

**The per-transaction ExUnits cap uses `totExUnits`.** [2](#0-1) 

`validateExUnitsTooBigUTxO` compares `totExUnits tx` against `ppMaxTxExUnitsL`. Because sub-transaction ExUnits are excluded, the check can pass even when the aggregate script work across the batch far exceeds the protocol limit.

**The per-block ExUnits cap uses the same function.** [3](#0-2) 

`validateExUnits` folds `totExUnits` over all transactions in the block. Sub-transaction ExUnits are again excluded, so the block-level cap `ppMaxBlockExUnitsL` can be silently exceeded.

**The minimum fee uses `totExUnits`.** [4](#0-3) 

`alonzoMinFeeTx` adds `txscriptfee (pp ^. ppPricesL) allExunits` where `allExunits = totExUnits tx`. Sub-transaction script fees are therefore zero.

**Sub-transactions are explicitly fee-free.** [5](#0-4) 

The comment on `updateUTxOStateNoFees` confirms: "This is used for sub-transactions where fees are not applicable." There is no compensating mechanism that rolls sub-transaction ExUnits into the top-level fee.

**Collateral is triggered by sub-transaction redeemers but sized against the top-level fee only.** [6](#0-5) 

`validateBatchCollateral` correctly detects that any sub-transaction has redeemers and requires collateral, but the collateral amount is computed from `pp ^. bodyTxL . feeTxBodyL` of the **top-level** transaction — which does not include sub-transaction script fees. Collateral therefore provides no deterrent proportional to the actual sub-transaction execution cost.

**The Dijkstra UTXO transition applies the ExUnits check only to the top-level transaction.** [7](#0-6) 

```
{- totExunits tx ≤ maxTxExUnits pp -}
runTest $ Alonzo.validateExUnitsTooBigUTxO pp tx
```

`tx` here is the top-level transaction; sub-transactions are processed separately through `SUBLEDGERS` without an equivalent ExUnits cap.

---

### Impact Explanation

An unprivileged transaction submitter can craft a top-level transaction with zero or minimal top-level ExUnits (paying only the byte-size fee) and embed N sub-transactions each carrying Plutus scripts budgeted at `maxTxExUnits`. Every validating node must execute all sub-transaction scripts. The aggregate CPU/memory consumed per block can be a multiple of `maxBlockExUnits` with no additional fee charged and no block-level cap enforced. This matches the **Medium** allowed impact: *"Attacker-controlled transactions… exceed intended validation limits or modify fees… outside design parameters."* It also approaches the **High** threshold of deterministic disagreement if nodes with different resource limits diverge on whether to accept a block, analogous to the reference-script DDoS that occurred on mainnet in June 2024. [8](#0-7) 

---

### Likelihood Explanation

The Dijkstra era is experimental but its ledger rules are production code in this repository. Any transaction submitter — no privileged role required — can construct a `DijkstraTxBody` with `dtbrSubTransactions` containing Plutus-script-bearing sub-transactions. The entry path is a single serialized transaction submitted to the mempool. No key compromise, governance majority, or Sybil attack is needed.

---

### Recommendation

1. **Extend `totExUnits` (or introduce a Dijkstra-specific variant)** to recursively sum ExUnits across all sub-transactions:
   ```haskell
   totExUnitsDijkstra tx =
     totExUnits tx
       <> foldMap totExUnits (tx ^. bodyTxL . subTransactionsTxBodyL)
   ```
2. **Use the extended sum in `validateExUnitsTooBigUTxO`** for Dijkstra-era transactions so the per-transaction cap covers the full batch.
3. **Use the extended sum in `alonzoMinFeeTx`** (or the Dijkstra override) so the minimum fee includes script fees for all sub-transaction redeemers.
4. **Use the extended sum in `validateExUnits` (BBODY)** so the per-block cap is enforced over the full execution budget consumed by a block.
5. **Size collateral against the full batch fee** (including sub-transaction script fees) rather than only the top-level fee field.

### Proof of Concept

```
Attacker constructs:
  TopTx {
    fee = minByteFee(topTx),          -- no script fee component
    subTransactions = [
      SubTx { redeemers = [(ptr, (rdmr, ExUnits maxMem maxSteps))] },
      SubTx { redeemers = [(ptr, (rdmr, ExUnits maxMem maxSteps))] },
      ... (N sub-transactions)
    ]
  }

Ledger checks:
  totExUnits topTx = 0                -- sub-tx ExUnits not counted
  validateExUnitsTooBigUTxO: 0 ≤ maxTxExUnits  ✓ (passes)
  validateFeeTooSmallUTxO:   fee ≥ minByteFee   ✓ (passes, no script fee)
  validateBatchCollateral:   collateral ≥ collateralPercent * fee  ✓ (passes, fee is tiny)

Node execution:
  Each sub-transaction's Plutus scripts are evaluated at full ExUnits budget.
  Total execution = N × maxTxExUnits >> maxBlockExUnits.
  Fee paid by attacker ≈ byte-size fee only.
``` [9](#0-8) [10](#0-9)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability. This attack forced us to make a quick decision on implementing a definite prevention of such attacks, while trying to not have a drastic impact on the common use case that DApp developers rely on so much.
