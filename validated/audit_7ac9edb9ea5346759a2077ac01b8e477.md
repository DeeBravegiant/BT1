### Title
Attacker-Controlled Peer Can Corrupt Data Layer Wallet Root Hash via Crafted Solution in Retry Path — (`chia/wallet/wallet_state_manager.py`, `chia/data_layer/data_layer_wallet.py`)

### Summary

When a DL singleton `CoinState` is placed in `retry_store` and the original peer disconnects, `_retry_failed_states` silently substitutes any available full-node peer. That substitute peer's `RespondPuzzleSolution` response is validated only for puzzle hash correctness — the **solution** is completely unconstrained. `singleton_removed` then executes the puzzle with the attacker-supplied solution and reads the new Data Layer root hash directly from the CREATE_COIN condition's **hints** (memos), without verifying that those hints are consistent with the actual puzzle hash of the new singleton coin. An attacker-controlled peer can therefore inject an arbitrary root hash into the DL wallet's persistent `SingletonRecord` store.

### Finding Description

**Step 1 — Retry path selects a substitute peer without restriction.**

In `_retry_failed_states`:

```python
if len(matching_peer) == 0:
    peer = self.get_full_node_peer()   # any connected full-node peer
``` [1](#0-0) 

Any peer that is currently connected and advertises `NodeType.FULL_NODE` is eligible. No trust check is performed here.

**Step 2 — `fetch_coin_spend` validates the puzzle hash but not the solution.**

```python

    raise PeerRequestException(...)
# solution is accepted unconditionally
return make_spend(coin, solution_response.response.puzzle, solution_response.response.solution)
``` [2](#0-1) 

The puzzle must hash to `coin.puzzle_hash` (a fixed, public value), but the **solution** field is returned verbatim with no validation.

**Why the attacker can craft a valid-looking solution:**

A DL singleton puzzle is `SINGLETON_TOP_LAYER_MOD` curried over an `NFT_STATE_LAYER_MOD` layer. The singleton top layer's CLVM enforces lineage proof validity and that the output coin has an odd amount, but it does **not** enforce the content of the CREATE_COIN hints. `AGG_SIG_ME` is merely an output condition in CLVM — it is not enforced during `run_with_cost` (signature validation is a consensus-layer concern). The attacker therefore can supply:

- A valid lineage proof (all fields are public blockchain data)
- An inner solution that produces a CREATE_COIN with the correct `full_puzzle_hash` (matching the real on-chain child coin) but with `condition[3]` (hints) containing an attacker-chosen root hash

The CLVM execution succeeds, `match_dl_singleton` returns `True` (the puzzle structure is genuine), and the corrupted `SingletonRecord` is written to the database. [6](#0-5) 

### Impact Explanation

The DL wallet's `SingletonRecord` for the affected launcher ID is permanently overwritten with an attacker-chosen `root` and `inner_puzzle_hash`. Downstream consequences:

- `get_spendable_singleton_info` returns the corrupted lineage proof, causing all subsequent `create_update_state_spend` calls to construct invalid spends (wrong lineage proof → rejected by consensus).
- DL offer settlement (`finish_graftroot_solutions`, `verify_offer`) relies on the wallet's stored root to match proofs of inclusion; a corrupted root breaks offer integrity checks.
- Any Data Layer client querying the wallet's view of the store root receives an attacker-controlled value, corrupting Data Layer ownership and root hash state.

### Likelihood Explanation

The attacker must be a peer that can connect as `NodeType.FULL_NODE`. This is an unprivileged network position. The trigger condition (a DL coin_state in `retry_store`) is reachable by the attacker themselves: connect as peer A, respond to the initial `request_puzzle_solution` with a rejection (`RejectPuzzleSolution`), causing `PeerRequestException` → `retry_store.add_state`. Then disconnect peer A, reconnect as peer B (attacker-controlled), and wait for the `coin_state_retry_seconds` timer.

### Recommendation

In `singleton_removed`, after extracting `full_puzzle_hash`, `root`, and `inner_puzzle_hash` from the conditions, verify that the hints are consistent with the actual puzzle hash:

```python
expected_ph = create_host_fullpuz(inner_puzzle_hash, root, singleton_record.launcher_id).get_tree_hash_precalc(inner_puzzle_hash)
if full_puzzle_hash != expected_ph:
    self.log.warning("DL singleton spend has inconsistent hints vs puzzle hash, ignoring")
    return
```

This ensures that even if a peer supplies a crafted solution with wrong hints, the wallet will reject it because the hints do not match the cryptographically committed puzzle hash of the new singleton coin.

### Proof of Concept

1. Set up a wallet with a tracked DL singleton (launcher_id `L`, current root `R_real`).
2. Connect attacker peer A. When `_add_coin_states` calls `fetch_coin_spend_for_coin_state` for the spent DL singleton coin, peer A returns `RejectPuzzleSolution`. This raises `PeerRequestException`, causing `retry_store.add_state(coin_state, peer_A.peer_node_id, fork_height)`.
3. Disconnect peer A. Connect attacker peer B.
4. Wait for `_retry_failed_states` to fire. It calls `get_full_node_peer()` → peer B.
5. Peer B responds to `request_puzzle_solution` with:
   - `puzzle`: the correct DL singleton puzzle (hash matches `coin.puzzle_hash`) ✓
   - `solution`: a crafted solution whose CLVM execution produces a CREATE_COIN condition with `condition[3] = [launcher_id, R_fake, inner_puzzle_hash_fake]`
6. `singleton_removed` runs, extracts `root = R_fake`, and writes `SingletonRecord(root=R_fake, ...)` to `dl_store`.
7. Assert: `dl_store.get_latest_singleton(L).root == R_fake` (corrupted), not `R_real`.

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:

```python
def match_dl_singleton(puzzle: Program | SerializedProgram) -> tuple[bool, Iterator[Program]]:
    """
    Given a puzzle test if it's a CAT and, if it is, return the curried arguments
    """
    mod, singleton_curried_args = uncurry(puzzle)
    if mod == SINGLETON_TOP_LAYER_MOD:
        mod, dl_curried_args = uncurry(singleton_curried_args.at("rf"))
        if mod == NFT_STATE_LAYER_MOD and dl_curried_args.at("rrf") == ACS_MU_PH:
            launcher_id = singleton_curried_args.at("frf")
            root = dl_curried_args.at("rff")
            innerpuz = dl_curried_args.at("rrrf")
            return True, iter((innerpuz, root, launcher_id))

    return False, iter(())
