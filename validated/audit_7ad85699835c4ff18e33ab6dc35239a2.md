Audit Report

## Title
Stale Cached `rsETHPrice` Allows Depositors to Mint Excess rsETH, Diluting Existing Holders' Accrued Yield - (`contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle` stores `rsETHPrice` as a persistent state variable that must be manually updated via `updateRSETHPrice()`. `LRTDepositPool.getRsETHAmountToMint()` divides by this cached value to compute how many rsETH tokens to mint. Neither `depositETH()` nor `depositAsset()` calls `updateRSETHPrice()` before minting, so any depositor who acts while the stored price is below the true current value receives excess rsETH, permanently diluting existing holders' accrued staking yield.

## Finding Description

`rsETHPrice` is a persistent state variable in `LRTOracle`: [1](#0-0) 

It is only updated when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is explicitly called: [2](#0-1) 

`getRsETHAmountToMint()` divides by this cached value: [3](#0-2) 

Neither `depositETH()` nor `depositAsset()` triggers a price update before calling `_beforeDeposit()` → `getRsETHAmountToMint()`: [4](#0-3) [5](#0-4) 

As staking rewards accrue, the true rsETH/ETH rate rises above the stored `rsETHPrice`. Because the mint formula is `rsethAmountToMint = (amount * assetPrice) / rsETHPrice`, a stale (lower) denominator inflates the mint amount. The depositor receives more rsETH than their deposit warrants, and when `updateRSETHPrice()` is eventually called, the inflated supply permanently reduces the per-token ETH backing for all prior holders.

A guaranteed multi-block staleness window exists when `pricePercentageLimit` is configured: if the true price increase exceeds the threshold, `updateRSETHPrice()` reverts for any non-manager caller with `PriceAboveDailyThreshold`: [6](#0-5) 

During this window the price is frozen at the stale value and only a manager can advance it, while deposits remain open and continue to mint at the stale rate.

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders have accrued yield reflected in the true rsETH/ETH rate being higher than the stored `rsETHPrice`. A depositor who deposits before `updateRSETHPrice()` is called receives rsETH at the stale (cheaper) rate. When the price is eventually updated, the total rsETH supply is inflated relative to the protocol's ETH, permanently reducing the redemption value per token for all prior holders. The attacker can exit at the true (higher) price on secondary markets, capturing yield that belonged to existing holders. The magnitude scales with deposit size and staleness duration.

## Likelihood Explanation

**Moderate-to-High.** Staking rewards accrue continuously on-chain, so any gap between `updateRSETHPrice()` calls creates an exploitable window. `updateRSETHPrice()` is public but there is no on-chain enforcement requiring it before deposits; MEV bots have a direct financial incentive to skip calling it before their own deposits. The `pricePercentageLimit` guard creates a guaranteed, potentially multi-block window where non-managers are blocked from updating the price while deposits remain open, making exploitation deterministic in that configuration. [7](#0-6) 

## Recommendation

Call `updateRSETHPrice()` (or inline the live price computation) at the start of `depositETH()` and `depositAsset()` before computing `rsethAmountToMint`:

```solidity
function depositETH(uint256 minRSETHAmountExpected, string calldata referralId)
    external payable nonReentrant whenNotPaused onlySupportedAsset(LRTConstants.ETH_TOKEN)
{
+   ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
    uint256 rsethAmountToMint = _beforeDeposit(LRTConstants.ETH_TOKEN, msg.value, minRSETHAmountExpected);
    _mintRsETH(rsethAmountToMint);
    emit ETHDeposit(msg.sender, msg.value, rsethAmountToMint, referralId);
}
```

Apply the same change to `depositAsset()`. This ensures the rsETH price always reflects current accrued yield before any new shares are minted. Note that the `pricePercentageLimit` revert path must also be handled (e.g., allow the manager-gated update to proceed, or cap the mint amount to the highest permissible price).

## Proof of Concept

1. Staking rewards accrue in EigenLayer over time. True rsETH price rises from `P_stale = 1.00e18` to `P_true = 1.05e18`. `updateRSETHPrice()` has not been called (or is blocked by `pricePercentageLimit`).
2. Attacker calls `depositETH{value: 100 ether}(0, "")`.
3. `getRsETHAmountToMint` computes: `100e18 * 1e18 / 1.00e18 = 100e18 rsETH` instead of the correct `100e18 * 1e18 / 1.05e18 ≈ 95.24e18 rsETH`.
4. Attacker receives `≈ 4.76 rsETH` in excess.
5. Attacker sells excess rsETH on a secondary market at `P_true`, capturing `≈ 4.76 * 1.05 ≈ 5 ETH` of yield belonging to existing holders.
6. When `updateRSETHPrice()` is eventually called, the inflated supply permanently dilutes all prior holders.

**Foundry fork test plan:**
- Fork mainnet; deploy/configure the contracts with a non-zero `pricePercentageLimit`.
- Simulate EigenLayer reward accrual (advance time, increase pod balance) so `_getTotalEthInProtocol()` returns a value above `previousTVL`.
- Confirm `updateRSETHPrice()` reverts for a non-manager caller with `PriceAboveDailyThreshold`.
- Call `depositETH` as the attacker; record `rsethAmountToMint`.
- Have the manager call `updateRSETHPriceAsManager()`; record the new `rsETHPrice`.
- Assert that `attacker_rsETH * newRsETHPrice > 100 ether` (attacker received more ETH-equivalent than deposited).
- Assert that existing holder redemption value per rsETH decreased after the price update.
