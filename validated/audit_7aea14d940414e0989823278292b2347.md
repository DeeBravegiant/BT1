### Title
Unprivileged `directDeposits` Forces Non-Zero Balance on Victim's Stake Account, Blocking Deregistration and Freezing Deposit — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

In the Dijkstra era, the `directDeposits` field of a transaction body allows any unprivileged submitter to add ADA to any registered stake account without requiring a witness from the account owner. Because the `ConwayUnRegCert` (stake deregistration) rule enforces a strict zero-balance precondition, an attacker can permanently block a victim from deregistering their stake credential — and reclaiming their stake deposit — by repeatedly forcing small direct deposits into the victim's account. For script-credential accounts whose script does not permit withdrawals, the freeze is permanent and requires a hard fork to recover.

---

### Finding Description

**Root cause 1 — No witness required for `directDeposits`.**

`dijkstraEntitiesTransition` in `Entities.hs` reads `directDeposits` from the transaction body and applies them unconditionally after processing certificates:

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```

The only guard is that the target account is registered. No witness from the recipient is required. [1](#0-0) 

**Root cause 2 — `UTXOW` witness check does not cover `directDeposits`.**

`dijkstraUtxowTransition` calls `Shelley.validateNeededWitnesses`, which collects required witnesses for UTxO inputs, withdrawals, and certificates — but has no branch for `directDeposits`:

```haskell
{- witsVKeyNeeded utxo tx genDelegs ⊆ witsKeyHashes -}
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody
``` [2](#0-1) 

**Root cause 3 — `ConwayUnRegCert` requires zero account balance.**

The Conway DELEG rule (reused by Dijkstra) enforces a hard zero-balance precondition before allowing stake deregistration:

```haskell
checkStakeKeyHasZeroRewardBalance = do
  accountState <- mAccountState
  let balanceCompact = accountState ^. balanceAccountStateL
  guard (balanceCompact /= mempty)
  Just $ fromCompact balanceCompact
failOnJust
  checkStakeKeyHasZeroRewardBalance
  (injectFailure . StakeKeyHasNonZeroAccountBalanceDELEG)
``` [3](#0-2) 

This precondition is also confirmed in the Shelley formal spec: *"There is a precondition that the credential has been registered and that the reward balance is zero."* [4](#0-3) 

**`applyDirectDeposits` adds to balance with no authorization check:** [5](#0-4)

### Impact Explanation

**For key-credential accounts (Medium → High griefing):** The victim can include a withdrawal in the same transaction as the `UnRegCert`, because `dijkstraEntitiesTransition` applies withdrawals *before* processing certificates. This means a single-transaction recovery is possible. However, the attacker can front-run every deregistration attempt by submitting a new direct deposit in the same block (processed before the victim's tx), forcing the victim to perpetually retry. The victim's stake deposit is effectively held hostage at the cost of the attacker's transaction fees.

**For script-credential accounts (High — permanent freeze):** If the victim's staking script does not permit withdrawals (a valid design, e.g., a script that only authorises delegation and deregistration), the victim cannot drain the forced balance. The zero-balance precondition in `ConwayUnRegCert` then permanently blocks deregistration, freezing the victim's stake deposit. Recovery requires a protocol change (hard fork) to either remove the zero-balance precondition or add a bypass for script-credential accounts.

This matches the allowed High impact: *"Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork."*

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction submitter on mainnet. The attacker only needs to know the victim's stake account address (publicly visible on-chain) and pay the transaction fee plus the deposited amount (which can be as small as 1 lovelace). The `directDeposits` field is part of the Dijkstra transaction body CDDL (field index 25) and is fully serializable by any wallet or tool. [6](#0-5) 

The attack is also available inside sub-transactions via the `SUBENTITIES` rule, which applies the same `applyDirectDeposits` logic without recipient authorization. [7](#0-6) 

---

### Recommendation

Require a witness from the recipient account owner for each entry in `directDeposits`, analogous to how withdrawals require a stake-credential witness. Concretely, `getWitsVKeyNeeded` (and its script counterpart) should be extended to include the credentials of all `directDeposits` targets, so that `validateNeededWitnesses` enforces recipient authorization. Alternatively, the `ConwayUnRegCert` zero-balance precondition should be relaxed to allow deregistration when the non-zero balance is solely attributable to unsolicited direct deposits, with the balance returned to the owner on deregistration.

---

### Proof of Concept

1. Victim registers a script-credential stake account (script permits delegation and deregistration but not withdrawals). Victim pays the key deposit.
2. Attacker constructs a valid Dijkstra transaction with `directDeposits = { victim_account_address => 1 }`, spending 1 lovelace from their own UTxO. No witness from the victim is needed.
3. Transaction is accepted: `dijkstraEntitiesTransition` passes `directDepositsMissingAccounts` (account is registered) and calls `applyDirectDeposits`, setting victim's balance to 1 lovelace.
4. Victim submits a transaction containing only `DijkstraUnRegCert`. The Conway DELEG rule evaluates `checkStakeKeyHasZeroRewardBalance`, finds `balanceCompact ≠ mempty`, and fails with `StakeKeyHasNonZeroAccountBalanceDELEG`.
5. Victim cannot include a withdrawal because their script rejects it. The victim's stake deposit is permanently locked.
6. Attacker can repeat step 2 indefinitely at minimal cost to prevent any recovery attempt.
