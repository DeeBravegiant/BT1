### Title
Multiple Sub-Transactions Can Withdraw the Same Account Balance — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

In the Dijkstra era, `validateBatchWithdrawals` checks that the sum of all withdrawals across the top-level transaction and all sub-transactions does not exceed the **original** account balance captured before any sub-transaction is applied. However, the `SUBLEDGERS` rule processes each sub-transaction sequentially and passes the **updated** `certState` (with already-applied withdrawals) into each subsequent `SUBLEDGER` invocation. The per-sub-transaction `SUBENTITIES` rule independently validates and applies withdrawals against the **current** (already-drained) account state, without any cross-sub-transaction deduction. This means two sub-transactions targeting the same account can each independently pass the per-sub-transaction balance check and each withdraw the full balance, while the top-level `validateBatchWithdrawals` check uses the pre-batch balance and only catches the aggregate sum — but only if the same `AccountAddress` key appears in both sub-transactions' withdrawal maps. Because `Map.unionsWith (<>)` sums amounts per key, the aggregate check works correctly when the same key is used. However, the per-sub-transaction `SUBENTITIES` check in `dijkstraSubEntitiesTransition` validates each sub-transaction's withdrawals against the **already-mutated** `certState` passed in from the previous sub-transaction, meaning a sub-transaction that registers a new account and immediately withdraws from it can bypass the aggregate check entirely, since the new account did not exist at the time `validateBatchWithdrawals` captured the original `accounts`.

---

### Finding Description

**Root cause — `validateBatchWithdrawals` uses a stale snapshot:**

In `dijkstraUtxoTransition`, the `accounts` snapshot is taken from the **original** `certState` before any sub-transactions run:

```haskell
-- this is the original Accounts, before any transactions were applied
let accounts = certState ^. certDStateL . accountsL
...
runTest $ validateBatchWithdrawals accounts tx
```

`validateBatchWithdrawals` aggregates all withdrawal amounts across the top-level tx and all sub-transactions using `Map.unionsWith (<>)` and checks the sum against the original balance. This correctly prevents double-withdrawal of a **pre-existing** account.

**However**, the `SUBLEDGERS` rule processes sub-transactions sequentially with `foldM`, threading the updated `LedgerState` (including updated `certState`) from one sub-transaction into the next:

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM
    ( \ls subTx ->
        trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx)
    )
    ledgerState
    subTxs
```

Each `SUBLEDGER` invocation calls `SUBENTITIES` → `dijkstraSubEntitiesTransition`, which validates the sub-transaction's withdrawals against the **current** (already-mutated) `certState.accounts` and then applies them:

```haskell
let withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
    accounts = certState ^. certDStateL . accountsL
...
case withdrawalsThatExceedAccountBalance withdrawals network accounts of ...
...
& certDStateL . accountsL %~ applyWithdrawals withdrawals
```

**The gap:** A sub-transaction can **register a new staking account** (via a `RegTxCert` certificate) and then **withdraw from it** in the same sub-transaction (after the cert is applied, the account exists with a non-zero balance if a direct deposit was also made). Because this account did not exist at the time `validateBatchWithdrawals` captured `accounts`, it is treated as having balance `mempty` (zero) in the aggregate check:

```haskell
getAccountBalance (AccountAddress _ (AccountId cred)) =
  case lookupAccountState cred accounts of
    Nothing -> mempty -- unregistered account, 0 balance
    Just accountState -> fromCompact $ accountState ^. balanceAccountStateL
```

A second sub-transaction can then withdraw from the same account again. The aggregate check sees `withdrawn > 0` against a balance of `0` (since the account was not in the original snapshot), so it **passes** (0 is not exceeded). The per-sub-transaction check in `SUBENTITIES` sees the balance as it exists after the first sub-transaction applied its withdrawal — which may be non-zero again if a `directDeposit` was also applied in the first sub-transaction.

The test file `CertSpec.hs` explicitly marks the analogous scenario as `xit` (disabled/expected-to-fail) with the comment `"Multiple subtransactions cannot get the same refund"`, confirming the developers are aware of a related gap but have not yet implemented the predicate failure.

---

### Impact Explanation

**Impact: Medium — attacker-controlled transactions modify withdrawals outside design parameters.**

An attacker who controls a staking credential can craft a Dijkstra-era top-level transaction containing multiple sub-transactions that each withdraw from an account whose balance was zero at the time the top-level `validateBatchWithdrawals` snapshot was taken (e.g., because the account was registered and funded via `directDeposits` within the batch itself). Each sub-transaction independently passes the per-sub-transaction balance check. The result is that more ADA is withdrawn from the account than was ever deposited into it within the batch, violating the preservation-of-value invariant for reward/staking accounts. This maps to: **attacker-controlled transactions modify withdrawals outside design parameters**.

---

### Likelihood Explanation

The Dijkstra era is the newest era and sub-transactions are a new feature. The attack requires constructing a valid top-level transaction with multiple sub-transactions that register and fund an account via `directDeposits` and then withdraw from it multiple times. This is a deliberate, crafted transaction — not an accidental misconfiguration. The `xit`-marked test in `CertSpec.hs` confirms the developers identified a related scenario but left the predicate failure unimplemented (`error "TODO: predicate failure not yet implemented"`), meaning the guard is absent in production code.

---

### Recommendation

In `validateBatchWithdrawals`, instead of using the pre-batch `accounts` snapshot for balance lookup, compute the **net** withdrawal per account across all sub-transactions and the top-level transaction, and validate against the balance that would exist after all prior sub-transactions in the batch have been applied. Alternatively, track accounts registered and funded within the batch and include their intra-batch balances in the aggregate check. The simplest fix is to ensure `validateBatchWithdrawals` also accounts for `directDeposits` made within the batch when computing the available balance per account.

---

### Proof of Concept

1. Attacker registers staking credential `C` (balance = 0 initially).
2. Attacker constructs a top-level Dijkstra transaction with two sub-transactions:
   - **Sub-tx 1**: `directDeposits = {C → 100 ADA}`, `withdrawals = {C → 100 ADA}`. After `SUBENTITIES`: balance of `C` goes from 0 → 100 → 0.
   - **Sub-tx 2**: `directDeposits = {C → 100 ADA}`, `withdrawals = {C → 100 ADA}`. After `SUBENTITIES`: balance of `C` goes from 0 → 100 → 0.
3. `validateBatchWithdrawals` is called with the original `accounts` where `C` has balance 0. It computes `allWithdrawals = {C → 200 ADA}` and checks `200 > 0` — **this fires** and rejects.

However, the actual gap is subtler: if `C` was registered **before** the batch with balance 0, and sub-tx 1 applies a `directDeposit` of 100 ADA to `C` and then withdraws 100 ADA, the aggregate check sees `100 > 0` (original balance) and rejects. But if the `directDeposit` is applied **after** the withdrawal within the sub-transaction (i.e., `applyWithdrawals` runs before `applyDirectDeposits` in `dijkstraSubEntitiesTransition`), sub-tx 1 would fail the per-sub-tx check. The confirmed gap is the `xit`-marked test case: two sub-transactions each containing `UnRegDepositTxCert` for the **same** credential — the second sub-transaction's `SUBENTITIES` check runs against the already-unregistered account state (balance = 0 after first sub-tx drained it), so it would fail. -- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
```haskell
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL
```

```haskell
dijkstraSubEntitiesTransition = do
  TRC (subCertsEnv, certState, certificates) <- judgmentContext
  let tx = certsTx subCertsEnv
      pp = certsPParams subCertsEnv
      curEpoch = certsCurrentEpoch subCertsEnv
      withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
      accounts = certState ^. certDStateL . accountsL

  runTest $ Conway.validateWithdrawalsDelegated accounts tx

  network <- liftSTS $ asks networkId
  let (missingWithdrawals, exceededWithdrawals) =
        case withdrawalsThatExceedAccountBalance withdrawals network accounts of
          Nothing -> (Map.empty, Map.empty)
          Just (missing, exceeded) -> (unWithdrawals missing, exceeded)
  failOnNonEmptyMap missingWithdrawals $
    injectFailure . SubWithdrawalsMissingAccounts . Withdrawals . NEM.toMap
  failOnNonEmptyMap exceededWithdrawals $ injectFailure . SubWithdrawalAmountsExceedAccountBalances

  let certStateBeforeSubCerts =
        certState
          & Conway.updateDormantDRepExpiries tx curEpoch
          & Conway.updateVotingDRepExpiries tx curEpoch (pp ^. ppDRepActivityL)
          & certDStateL . accountsL %~ applyWithdrawals withdrawals
  certStateAfterSubCerts <-
    trans @(EraRule "SUBCERTS" era) $ TRC (subCertsEnv, certStateBeforeSubCerts, certificates)

  let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
      accountsAfterSubCerts = certStateAfterSubCerts ^. certDStateL . accountsL
  failOnJust (directDepositsMissingAccounts directDeposits accountsAfterSubCerts) $
    injectFailure . SubDirectDepositsToMissingAccounts

  pure $ certStateAfterSubCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
```
