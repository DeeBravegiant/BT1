### Title
`respond_verify_foreign_tx` accepts any valid signature over any `payload_hash` without binding it to the submitted request — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is cryptographically valid over `response.payload_hash`, but never verifies that `response.payload_hash` is actually derived from the submitted `request`. A single Byzantine attested participant (strictly below the signing threshold) can replay a signature produced for a previous, legitimately-processed foreign-chain request and use it to resolve a completely different pending request, delivering a forged verification result to the caller.

### Finding Description

The contract's `respond_verify_foreign_tx` function performs two actions:

1. Verifies `response.signature` is a valid ECDSA signature over `response.payload_hash` using the domain's root public key.
2. Resolves all queued yields for `request` with the full `response` (including the unverified `payload_hash`). [1](#0-0) 

What is **never checked** is whether `response.payload_hash` is the canonical hash of `ForeignTxSignPayload::V1 { request: <the submitted request>, values: <extracted values> }`. The contract has the submitted `request` in hand (it is the key used to look up the pending yield), but it never uses it to constrain what `payload_hash` may be.

The `ForeignTxSignPayload` that nodes are supposed to sign is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))
``` [2](#0-1) 

Because the contract only checks `verify_ecdsa_signature(signature, payload_hash, root_pk)`, any `(payload_hash, signature)` pair that was ever legitimately produced by the MPC network for **any** prior request satisfies the check.

Compare with `respond`, which correctly derives the per-request key before verifying: [3](#0-2) 

`respond_verify_foreign_tx` skips the analogous binding step entirely. [4](#0-3) 

### Impact Explanation

A single Byzantine attested participant (one node, well below the signing threshold) can:

1. Observe a legitimately produced response `(H1, S1)` for a past request `R1` (e.g., Bitcoin tx T1). This response is submitted on-chain and is publicly visible.
2. Wait for a new `verify_foreign_transaction` request `R2` (e.g., Bitcoin tx T2) to enter the pending queue.
3. Call `respond_verify_foreign_tx(R2, { payload_hash: H1, signature: S1 })`.
4. The contract accepts the call: `S1` is a valid signature over `H1` under the root key, and `R2` is a live pending request.
5. The caller of `R2` receives `(H1, S1)` — a response that cryptographically attests to T1, not T2.

Any bridge contract or NEAR smart contract that consumes the `VerifyForeignTransactionResponse` without independently recomputing the expected `payload_hash` from its own request parameters will treat this as proof that T2 was verified on the foreign chain, enabling invalid bridge execution (e.g., releasing funds for a transaction that was never confirmed, or for the wrong transaction).

The SDK does provide `ForeignChainSignatureVerifier::verify_signature` for client-side verification: [5](#0-4) 

However, this is an off-chain/client-side helper. The contract itself provides no such guarantee, so any caller that omits this step — or any future integration that trusts the contract's acceptance as sufficient — is vulnerable.

### Likelihood Explanation

- Requires only **one** Byzantine attested participant, far below the signing threshold.
- No new cryptographic material needs to be forged; the attacker simply replays an existing on-chain response.
- The attacker can choose the timing: wait for a high-value `verify_foreign_transaction` request, then immediately replay a previously observed `(payload_hash, signature)` pair before any honest node responds.
- The attack is silent — the contract emits no error, and the caller receives a well-formed `VerifyForeignTransactionResponse`.

### Recommendation

In `respond_verify_foreign_tx`, require the caller to also supply the `extracted_values` that were observed, then verify on-chain that:

```
response.payload_hash == SHA-256(borsh(ForeignTxSignPayload::V1 {
    request: request.request,   // the submitted request, already in scope
    values: submitted_values,   // newly required parameter
}))
```

This binds the `payload_hash` to the specific request and its observed values, making replay across different requests impossible. The signature check then becomes meaningful as a proof that the MPC network attested to exactly those values for exactly that request.

### Proof of Concept

```
// Setup: R1 was previously processed; (H1, S1) is on-chain.
// R2 is a new pending request for a different foreign tx.

// Attacker (single Byzantine participant) calls:
contract.respond_verify_foreign_tx(
    R2,                                    // pending request for tx T2
    VerifyForeignTransactionResponse {
        payload_hash: H1,                  // hash derived from tx T1
        signature: S1,                     // valid signature over H1
    }
)
// → contract accepts (S1 valid over H1, R2 is pending)
// → caller of R2 receives (H1, S1)
// → caller believes T2 was verified; actually T1 was verified
// → bridge releases funds for T2 based on forged attestation
```

The root cause is directly analogous to the `reduce_position` bug: `respond_verify_foreign_tx` updates one piece of state (resolves the yield for `request`) but omits the corresponding invariant check (that `payload_hash` is bound to `request`), breaking the accounting invariant that every resolved foreign-tx yield carries a hash that commits to the request that triggered it. [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
