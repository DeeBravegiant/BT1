### Title
Unbounded List Index Access in Weight Proof Block Pre-Validation Crashes Node Sync - (File: chia/consensus/multiprocess_validation.py)

### Summary
In `chia/consensus/multiprocess_validation.py`, the function `pre_validate_block` accesses `wp_summaries` using an index derived from `block.height` without any bounds check. A malicious peer can craft a weight proof whose `sub_epochs` list is shorter than the index computed from the recent chain's block heights, triggering an unhandled `IndexError` that crashes the pre-validation worker and halts the syncing node's ability to process valid blocks.

### Finding Description

Inside `pre_validate_block`, when a block includes a sub-epoch summary and `wp_summaries` is not `None`, the code computes an index directly from the block's height and uses it to index into the `wp_summaries` list:

```python
if block_rec.sub_epoch_summary_included is not None and wp_summaries is not None:
    next_ses = wp_summaries[int(block.height / constants.SUB_EPOCH_BLOCKS) - 1]
``` [1](#0-0) 

There is no guard of the form `if computed_index < len(wp_summaries)` before this access. The `wp_summaries` list is populated from the `sub_epochs` field of the weight proof sent by a peer:

```python
summaries, sub_epoch_weight_list = _validate_sub_epoch_summaries(self.constants, weight_proof)
``` [2](#0-1) 

The weight proof validation (`_validate_sub_epoch_summaries` → `_map_sub_epoch_summaries`) reconstructs `summaries` from the attacker-controlled `weight_proof.sub_epochs` list. Its length is bounded only by what the attacker includes. The validation checks that the last summary hash matches the recent chain, but does **not** enforce that `len(summaries) >= max(block.height // SUB_EPOCH_BLOCKS)` for every block in `recent_chain_data`. [3](#0-2) 

If an attacker crafts a weight proof where `recent_chain_data` contains blocks at heights requiring index `K` but `len(wp_summaries) <= K`, Python raises an unhandled `IndexError` inside the pre-validation worker.

### Impact Explanation

`pre_validate_block` is dispatched into a `ProcessPoolExecutor` via `prevalidate_blocks`. An unhandled exception in the worker propagates as a failed `Future`, which causes `add_block_batch` to fail, halting the syncing node's ability to advance its peak. A persistent attacker controlling one or more peers can repeatedly serve the crafted weight proof, causing a long-lived inability for the honest node to sync valid blocks. [4](#0-3) 

This matches the **High** impact category: *Permanent or long-lived inability for honest nodes to process valid blocks under normal network assumptions.*

### Likelihood Explanation

Any unprivileged peer participating in the Chia P2P network can serve a weight proof during sync. The weight proof format is attacker-controlled. No special keys, admin access, or cryptographic break is required. The only prerequisite is that the crafted proof passes `_validate_sub_epoch_summaries`, which checks hash chaining and weight but not the relationship between `len(summaries)` and the maximum block height in `recent_chain_data`.

### Recommendation

Add an explicit bounds check before the index access in `pre_validate_block`:

```python
if block_rec.sub_epoch_summary_included is not None and wp_summaries is not None:
    ses_idx = int(block.height / constants.SUB_EPOCH_BLOCKS) - 1
    if ses_idx < 0 or ses_idx >= len(wp_summaries):
        return return_error(Err.INVALID_SUB_EPOCH_SUMMARY)
    next_ses = wp_summaries[ses_idx]
    if not block_rec.sub_epoch_summary_included.get_hash() == next_ses.get_hash():
        ...
```

Additionally, `_validate_sub_epoch_summaries` (or its callers) should verify that `len(summaries)` is sufficient to cover all blocks in `recent_chain_data` that include a sub-epoch summary.

### Proof of Concept

1. Attacker connects to a syncing honest node as a peer.
2. Attacker serves a crafted `WeightProof` where `sub_epochs` contains only `N` entries, but `recent_chain_data` contains a block at height `H` such that `int(H / SUB_EPOCH_BLOCKS) - 1 >= N` and that block has `sub_epoch_summary_included is not None`.
3. The crafted proof passes `_validate_sub_epoch_summaries` because the hash chain of the `N` summaries is internally consistent and the last summary hash matches the `subepoch_summary_hash` embedded in the recent chain's sub-slot.
4. The honest node calls `add_block_batch` → `prevalidate_blocks` → `pre_validate_block` with `wp_summaries` of length `N`.
5. At line 263, `wp_summaries[int(H / SUB_EPOCH_BLOCKS) - 1]` raises `IndexError` (index out of range).
6. The worker future fails; `add_block_batch` returns an error; the node cannot advance its peak and cannot sync. [1](#0-0)

def _map_sub_epoch_summaries(
    sub_blocks_for_se: uint32,
    ses_hash: bytes32,
    sub_epoch_data: list[SubEpochData],
    curr_difficulty: uint64,
) -> tuple[list[SubEpochSummary], uint128, list[uint128]]:
    total_weight: uint128 = uint128(0)
    summaries: list[SubEpochSummary] = []
    sub_epoch_weight_list: list[uint128] = []
    for idx, data in enumerate(sub_epoch_data):
        ses = SubEpochSummary(
            ses_hash,
            data.reward_chain_hash,
            data.num_blocks_overflow,
            data.new_difficulty,
            data.new_sub_slot_iters,
            data.challenge_merkle_root,
        )

        if idx < len(sub_epoch_data) - 1:
            delta = 0
            if idx > 0:
                delta = data.num_blocks_overflow
            log.debug(f"sub epoch {idx} start weight is {total_weight + curr_difficulty} ")
            sub_epoch_weight_list.append(uint128(total_weight + curr_difficulty))
            total_weight = uint128(
                total_weight
                + curr_difficulty * (sub_blocks_for_se + sub_epoch_data[idx + 1].num_blocks_overflow - delta)
            )

        # if new epoch update diff and iters
        if data.new_difficulty is not None:
            curr_difficulty = data.new_difficulty

        # add to dict
        summaries.append(ses)
        ses_hash = std_hash(ses)
    # add last sub epoch weight
    sub_epoch_weight_list.append(uint128(total_weight + curr_difficulty))
    return summaries, total_weight, sub_epoch_weight_list
```
