### Title
`respond_verify_foreign_tx` Hardcodes Secp256k1-Only Signature Verification, Permanently Breaking Ed25519 `ForeignTx` Domain Requests - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` contract method hardcodes support for exactly one signature scheme (`Secp256k1`) when verifying MPC-produced signatures for foreign-chain transactions. The contract allows `ForeignTx` domains to be configured with any protocol (including `Protocol::Frost` / Ed25519), but the respond path has no branch for Ed25519. Any yield request submitted against an Ed25519 `ForeignTx` domain is permanently unresolvable: MPC nodes compute a valid signature, call `respond_verify_foreign_tx`, and the contract always returns `SignatureSchemeMismatch`, breaking the request lifecycle for every user of that domain.

---

### Finding Description

`verify_foreign_transaction` accepts requests for any domain whose purpose is `DomainPurpose::ForeignTx`, with no restriction on the domain's underlying `Protocol`. [1](#0-0) 

The corresponding respond path, `respond_verify_foreign_tx`, then performs signature verification with a `match` that only handles the `Secp256k1` arm: [2](#0-1) 

Any other signature type (e.g., `Ed25519` produced by a `Protocol::Frost` domain) falls into the catch-all arm and returns `RespondError::SignatureSchemeMismatch`. There is no Ed25519 branch, unlike the regular `respond` method which handles both schemes: [3](#0-2) 

Domains are added via `vote_add_domains`, which accepts any `DomainConfig` including `Protocol::Frost` with `DomainPurpose::ForeignTx`, with no cross-validation that `ForeignTx` domains must use Secp256k1: [4](#0-3) 

---

### Impact Explanation

**Medium.** Once an Ed25519 `ForeignTx` domain exists, every `verify_foreign_transaction` call against it creates a yield request that can never be resolved. MPC nodes will compute a correct Ed25519 signature and call `respond_verify_foreign_tx`, which will always reject it with `SignatureSchemeMismatch`. The yield eventually times out, the user's callback receives an error, and the request lifecycle invariant is permanently broken for the entire domain. No funds are directly stolen, but the `ForeignTx` signing capability for that domain is completely non-functional, breaking a core production safety invariant.

---

### Likelihood Explanation

**Medium.** Adding a `ForeignTx` domain requires a threshold governance vote (`vote_add_domains`), so it is not directly triggerable by a single unprivileged caller. However, the contract imposes no validation preventing a `ForeignTx` + `Protocol::Frost` combination from being voted in. Once such a domain is live — whether by operator error or intentional configuration — every user request to it is permanently broken with no on-chain recovery path short of contract redeployment.

---

### Recommendation

Add an Ed25519 verification branch to `respond_verify_foreign_tx` mirroring the one in `respond`, or add a validation in `vote_add_domains` (or `check_request_preconditions`) that rejects `ForeignTx` domains whose protocol is not `Secp256k1` until full multi-scheme support is implemented. The inconsistency between `respond` and `respond_verify_foreign_tx` should be explicitly documented and enforced.

---

### Proof of Concept

1. Participants vote via `vote_add_domains` to add a domain with `purpose = ForeignTx` and `protocol = Protocol::Frost` (Ed25519). No contract validation blocks this.
2. A user calls `verify_foreign_transaction` referencing this domain. The call passes `check_request_preconditions` (only checks `DomainPurpose::ForeignTx`, not the protocol), and a yield request is enqueued.
3. MPC nodes observe the request, compute a valid Ed25519 threshold signature, and call `respond_verify_foreign_tx` with an `Ed25519`-variant `SignatureResponse`.
4. Inside `respond_verify_foreign_tx`, the `match` on `(&response.signature, public_key)` finds no `Ed25519` arm and falls into the catch-all, returning `Err(RespondError::SignatureSchemeMismatch { ... })`.
5. The yield is never resolved. After the NEAR yield-resume timeout, the user's callback fires with an error. The request is permanently unresolvable, and the entire `ForeignTx` domain is non-functional.

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
