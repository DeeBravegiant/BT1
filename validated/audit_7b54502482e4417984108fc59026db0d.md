Audit Report

## Title
Stale Cross-Chain Rate Allows L2 Depositors to Mint Excess rsETH, Diluting Existing Holders' Accrued Yield - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness check. All L2 pool variants call this function to determine how much rsETH to mint per ETH deposited. When the true L1 rsETH/ETH rate has risen due to staking rewards but the L2 receiver has not been updated (because no one has paid LayerZero fees to push an update), depositors receive more rsETH than their ETH entitles them to, diluting the accrued yield of all existing rsETH holders.

## Finding Description

`CrossChainRateReceiver` stores `rate` and `lastUpdated` but `getRate()` performs no age validation:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L102-105
function getRate() external view returns (uint256) {
    return rate;   // no staleness check against lastUpdated
}
```

The `rate` is only updated when `lzReceive` is triggered by a LayerZero message originating from `MultiChainRateProvider.updateRate()`, which is a permissionless but fee-bearing call — anyone can call it, but must pay the LayerZero messaging fee. During periods of high gas prices or low keeper activity, the L2 rate can lag by hours or days.

The L1 rate source is `RSETHMultiChainRateProvider.getLatestRate()` → `ILRTOracle(rsETHPriceOracle).rsETHPrice()`, which is itself a cached value updated by `LRTOracle.updateRSETHPrice()`. Both ends of the pipeline are cached with no on-chain freshness enforcement.

Every L2 pool variant uses the stale rate directly in its minting formula. In `RSETHPoolV2.viewSwapRsETHAmountAndFee()`:

```solidity
uint256 rsETHToETHrate = getRate();   // potentially stale
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The same pattern is confirmed in `RSETHPoolV3` (L299-308), `RSETHPoolV2ExternalBridge` (L307-316), and other pool variants. A grep for `MAX_RATE_AGE`, `staleness`, `stale`, or `lastUpdated` across all contract files returns zero matches, confirming no staleness protection exists anywhere in the minting path.

## Impact Explanation

**High — Theft of unclaimed yield.**

rsETH accrues staking rewards continuously from underlying LSTs (stETH, ETHx, etc.), reflected as a rising rsETH/ETH price in `LRTOracle.rsETHPrice`. This accrued-but-not-yet-distributed appreciation is the "unclaimed yield" of existing rsETH holders.

When the L2 rate is stale at 1.00 while the true rate is 1.05:
- A depositor sending 100 ETH receives `100e18 * 1e18 / 1.00e18 = 100 rsETH`
- Correct entitlement: `100e18 * 1e18 / 1.05e18 ≈ 95.24 rsETH`
- The excess ~4.76 rsETH is minted without corresponding ETH backing at the true rate

This excess rsETH represents a claim on the protocol's assets that was funded by diluting the accrued yield of existing holders. The `wrsETH.mint(msg.sender, rsETHAmount)` call issues real, redeemable tokens backed by the yield that existing holders had earned.

## Likelihood Explanation

- `updateRate()` is permissionless but requires paying LayerZero fees; no on-chain mechanism forces periodic updates.
- rsETH/ETH rate drifts upward predictably and continuously as underlying LSTs earn staking rewards.
- An attacker can monitor the L1 `rsETHPrice` vs. the L2 `CrossChainRateReceiver.rate` off-chain and deposit on L2 precisely when the gap is largest, before anyone pushes an update.
- No special privileges are required; any unprivileged depositor can exploit this.
- The `dailyMintLimit` in `RSETHPoolV2`/`RSETHPoolV3` provides partial mitigation by capping the total rsETH minted per day, but does not prevent the rate-based over-minting within that cap.

## Recommendation

Add a maximum staleness check in `CrossChainRateReceiver.getRate()`:

```solidity
uint256 public constant MAX_RATE_AGE = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_RATE_AGE, "Rate is stale");
    return rate;
}
```

This causes deposits to revert when the rate is stale, protecting existing holders until a fresh rate is pushed. Alternatively, integrate a Chainlink or other on-chain oracle as a sanity bound on the L2 side, or implement an automated keeper that pushes rate updates on a fixed schedule.

## Proof of Concept

1. At T=0, L1 `rsETHPrice = 1.00e18`. Rate is pushed to L2 via LayerZero. `CrossChainRateReceiver.rate = 1.00e18`, `lastUpdated = T`.
2. Over 48 hours, staking rewards accrue. L1 `rsETHPrice` rises to `1.05e18`. No one pays to push the update to L2 (high gas, no keeper).
3. At T+48h, attacker calls `RSETHPoolV2.deposit{value: 100 ether}("")`.
4. `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee(100e18)`, which calls `getRate()` → returns `1.00e18` (stale).
5. `rsETHAmount = 100e18 * 1e18 / 1.00e18 = 100e18` rsETH minted via `wrsETH.mint(msg.sender, 100e18)`.
6. Correct amount at true rate: `100e18 * 1e18 / 1.05e18 ≈ 95.24e18` rsETH.
7. Attacker receives ~4.76 excess rsETH, representing yield stolen from existing holders.
8. All existing rsETH holders are diluted proportionally — their rsETH now represents a smaller share of the protocol's total ETH backing.

**Foundry fork test plan:** Fork an L2 where the pool is deployed. Deploy a mock `CrossChainRateReceiver` with `rate = 1.00e18` and `lastUpdated = block.timestamp - 48 hours`. Point the pool's `rsETHOracle` to this receiver. Call `deposit{value: 100 ether}`. Assert that `rsETHAmount > 100e18 * 1e18 / 1.05e18`, confirming over-minting relative to the true rate.
