### Title
Silent Empty-String Metadata Bypass in Token-2022 `log_metadata` Permanently Poisons NEAR Token Mapping — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`LogMetadata::process` / `parse_metadata_account` silently returns `("", "")` for any Token-2022 mint whose `MetadataPointer` points to a non-zero address not owned by the Metaplex program. No error is raised. The resulting Wormhole message carries empty `name` and `symbol`. NEAR's `deploy_token_callback` accepts this message without validation and permanently registers the Solana mint address with empty metadata. Because `deploy_token_internal` enforces uniqueness with a hard `require!`, no subsequent legitimate `log_metadata` call can overwrite the mapping.

---

### Finding Description

**Step 1 — Silent fallback in `parse_metadata_account`** [1](#0-0) 

When the `metadata` account's owner is not `MetaplexID`, the function returns `(String::default(), String::default())` — i.e., `("", "")` — with `Ok(...)`. No error code, no revert.

**Step 2 — Token-2022 path reaches this fallback for any attacker-controlled pointer** [2](#0-1) 

When `metadata_pointer.metadata_address.0` is non-zero and not equal to the mint itself, `parse_metadata_account` is called with the pointer address. The attacker controls this address because they created the mint. They pass the pointed-to account (e.g., a system-program-owned account) as the `metadata` optional account. The key-equality check at lines 78–82 passes because the attacker chose the pointer to match the account they pass.

**Step 3 — No access control on `log_metadata`** [3](#0-2) 

The only constraint on the mint is that its mint authority must not be the bridge's authority PDA. An attacker-created Token-2022 mint trivially satisfies this.

**Step 4 — Wormhole message emitted with empty strings** [4](#0-3) 

The payload is posted to Wormhole with `name: ""` and `symbol: ""`.

**Step 5 — NEAR `deploy_token_callback` has no name/symbol validation** [5](#0-4) 

`deploy_token_callback` verifies only the factory emitter address, then passes `metadata.name` and `metadata.symbol` directly to `deploy_token_internal` without any non-empty check. (Contrast: the NEAR-native `log_metadata_callback` at line 341 does check `!metadata.name.is_empty() && !metadata.symbol.is_empty()`, but that guard is absent here.)

**Step 6 — Mapping is permanently poisoned** [6](#0-5) 

`deploy_token_internal` inserts the token into `deployed_tokens` and panics if it already exists. There is no admin function to remove a deployed token entry. Once the attacker's empty-metadata registration succeeds, the Solana mint address is permanently bound to a NEAR token with empty name and symbol, and all future `deploy_token` calls for the same mint will revert with `TokenExists`.

---

### Impact Explanation

This is a **token-mapping poisoning** attack. For any Solana Token-2022 mint that has not yet been registered on NEAR, an unprivileged attacker can:

1. Permanently bind that mint's canonical NEAR token ID to a contract with empty `name`/`symbol`.
2. Block all future legitimate registrations of that mint.
3. If the mint is a real asset with existing holders, those holders can never bridge it through the canonical path — their redemption path is permanently frozen.

This falls under **Critical — Irreversible fund lock / frozen redemption path** and **High — Asset-identity / token-mapping divergence**.

---

### Likelihood Explanation

The attack is fully permissionless:
- Creating a Token-2022 mint with a custom `MetadataPointer` is a standard SPL Token-2022 operation costing a few lamports.
- `log_metadata` has no signer restriction beyond the payer.
- Submitting the Wormhole VAA to NEAR's `deploy_token` is also permissionless.

The attacker can front-run any legitimate `log_metadata` call for a new Token-2022 token, or pre-poison any mint address before it gains adoption.

---

### Recommendation

1. **Reject empty metadata at the Solana emission point.** In `process`, after resolving `(name, symbol)`, add:
   ```rust
   require!(!name.trim().is_empty() && !symbol.trim().is_empty(), ErrorCode::EmptyTokenMetadata);
   ```

2. **Reject empty metadata at the NEAR ingestion point.** In `deploy_token_callback`, mirror the guard already present in `log_metadata_callback`:
   ```rust
   require!(
       !metadata.name.is_empty() && !metadata.symbol.is_empty(),
       BridgeError::InvalidMetadata
   );
   ```

3. **Treat non-MetaplexID-owned third-party metadata as an error**, not a silent fallback. In `parse_metadata_account`, replace the `else` branch with an explicit error code (e.g., `ErrorCode::InvalidTokenMetadataOwner`).
