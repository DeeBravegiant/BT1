### Title
`SyncStateReader::get_compiled_class_hash` Is a `todo!()` Stub That Panics During Gateway Stateful Validation of Declare Transactions - (File: `crates/apollo_gateway/src/sync_state_reader.rs`)

### Summary

`SyncStateReader`, the production `BlockifierStateReader` implementation used by the gateway for stateful transaction validation, leaves `get_compiled_class_hash` as an unimplemented `todo!()` stub. Any blockifier code path that reads the compiled class hash from the underlying state reader — which occurs during Declare transaction validation — will panic at runtime, causing the gateway to crash the validation task and reject the transaction.

### Finding Description

`SyncStateReader` implements the `BlockifierStateReader` trait (`StateReader` from the blockifier crate) for use in the gateway's stateful validation path. Every other method in the trait is fully implemented, but `get_compiled_class_hash` is left as `todo!()`: [1](#0-0) 

The trait's own documentation states the expected contract: "Returns `CompiledClassHash::default()` if no v1_class is found for the given class hash." [2](#0-1) 

The sibling `GenesisStateReader` in the same file correctly implements this method by returning `Ok(CompiledClassHash::default())`: [3](#0-2) 

The `SyncOrGenesisStateReader` enum delegates `get_compiled_class_hash` to whichever variant is active: [4](#0-3) 

When the `Sync` variant is active (i.e., at least one block exists), the call reaches the `todo!()` and panics.

The blockifier's `CachedState` calls through to the underlying reader's `get_compiled_class_hash` whenever the compiled class hash for a given class hash is not already in the write-cache. This occurs during:
1. State diff computation (`to_state_diff()`) after executing a Declare transaction — the diff logic reads the old compiled class hash from the underlying reader to determine what changed.
2. Any direct lookup of an existing class's compiled class hash from state.

The gateway's stateful validation of a Declare transaction exercises this path: the blockifier executes `__validate__`, writes the new compiled class hash into the `CachedState` write-set, and then computes the state diff to check resource bounds and fees, at which point it reads the *prior* compiled class hash from the underlying `SyncStateReader`. [5](#0-4) 

### Impact Explanation

A `todo!()` in Rust panics unconditionally at runtime. When the gateway's stateful validation task panics, the validation fails with an unhandled error, and the Declare transaction is rejected. Depending on whether the panic is caught by a `catch_unwind` boundary (none is visible in this path), it may also crash the validation worker thread entirely, causing a denial-of-service for all subsequent transactions processed by that worker.

This matches the allowed impact: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

### Likelihood Explanation

Any user submitting a valid Declare transaction (V0, V1, V2, or V3) to a running sequencer node that has processed at least one block (so `SyncStateReader` is used rather than `GenesisStateReader`) will trigger this panic. Declare transactions are a standard, publicly documented transaction type. No special privilege or knowledge is required.

### Recommendation

Implement `get_compiled_class_hash` in `SyncStateReader` by querying the state sync client, mirroring the pattern used for `get_class_hash_at` and `get_nonce_at`. If the state sync client does not expose a `get_compiled_class_hash` endpoint, return `Ok(CompiledClassHash::default())` as the trait contract specifies for missing entries, consistent with `GenesisStateReader`.

```rust
fn get_compiled_class_hash(&self, class_hash: ClassHash) -> StateResult<CompiledClassHash> {
    // Query state sync or return default per trait contract.
    Ok(CompiledClassHash::default())
}
```

### Proof of Concept

1. Start a sequencer node that has synced at least one block (so `SyncStateReader` is the active variant, not `GenesisStateReader`).
2. Submit any valid Declare transaction (e.g., a V3 Declare) to the gateway's HTTP endpoint.
3. The gateway calls `SyncStateReaderFactory::get_blockifier_state_reader_and_gateway_fixed_block_from_latest_block`, which returns a `SyncOrGenesisStateReader::Sync(SyncStateReader)`. [6](#0-5) 

4. The blockifier executes the Declare transaction's `__validate__` entrypoint and then computes the state diff, calling `CachedState::get_compiled_class_hash` on the underlying `SyncStateReader`.
5. `SyncStateReader::get_compiled_class_hash` hits `todo!()` and panics.
6. The gateway rejects the Declare transaction with an internal error; the transaction never reaches the mempool. [1](#0-0)
