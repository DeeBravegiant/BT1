### Title
`respond_verify_foreign_tx` Accepts Arbitrary `payload_hash` Unbound to the Original Request, Enabling Cross-Request Response Substitution - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` is actually the hash of `ForeignTxSignPayload::V1(request.request, …)`. A single Byzantine attested participant can replay any previously-published valid MPC response for foreign-tx request Y as the answer to a completely different pending request X, causing the contract to fan-out a forged attestation to every caller waiting on X.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs three checks before resolving all queued yields:

1. Caller is an attested participant.
2. The ECDSA signature in `response` is valid over `response.payload_hash` against the root public key.
3. `request` exists in `pending_verify_foreign_tx_requests`. [1](#0-0) 

What is **absent** is any binding between `response.payload_hash` and `request.request`. The contract never reconstructs `ForeignTxSignPayload::V1(request.request, values)` and never asserts that `SHA-256(borsh(payload)) == response.payload_hash`. [2](#0-1)

Because the contract never enforces this binding, a Byzantine participant can:

1. Observe any previously-accepted `respond_verify_foreign_tx(request_Y, response_Y)` call on-chain (all NEAR transactions are public). `response_Y` contains a valid root-key signature over `payload_hash_Y`.
2. While a different request X is pending in `pending_verify_foreign_tx_requests`, call `respond_verify_foreign_tx(request_X, response_Y)`.
3. The contract passes check 2 (`sig_Y` is valid over `payload_hash_Y`) and check 3 (`request_X` is pending), then calls `resolve_yields_for`, which **automatically fans out `response_Y` to every caller waiting on request X**. [4](#0-3) 

Every caller that submitted `verify_foreign_transaction` for request X now holds a `VerifyForeignTransactionResponse` whose `payload_hash` and `signature` attest to the data from request Y, not X. The self-triggered fan-out (the analog of Licredity's back-run) is the `resolve_yields_for` drain: once the attacker fires the single `respond_verify_foreign_tx` call, the contract automatically delivers the forged attestation to all queued callers without any further action.

The attack can be repeated: the attacker can use the same `response_Y` (or any other previously-published response) to poison any number of subsequently-pending requests, one per call.

### Impact Explanation

Bridge contracts (e.g., Omnibridge) call `verify_foreign_transaction` to obtain an MPC attestation that a specific foreign-chain deposit occurred. If the attacker substitutes a response for a different transaction, the bridge contract receives a structurally valid signature that does not attest to the deposit it requested. Any bridge logic that does not independently recompute `SHA-256(borsh(ForeignTxSignPayload::V1(expected_request, expected_values)))` and compare it to `response.payload_hash` will accept the forged attestation and may release funds for a deposit that never happened, enabling a double-spend or invalid bridge execution.

The `ForeignChainSignatureVerifier` in the SDK does perform this check: [5](#0-4) 

However, the contract itself provides no such guarantee, so any caller that trusts the contract's acceptance of the response without re-verifying the payload hash is vulnerable.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
