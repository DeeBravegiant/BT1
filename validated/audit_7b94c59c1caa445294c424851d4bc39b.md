### Title
Detached Refund Promises Silently Fail, Permanently Locking Excess Deposits — (File: `crates/contract/src/lib.rs`)

### Summary

The MPC contract schedules excess-deposit refunds using `Promise::new(account).transfer(amount).detach()` in three separate code paths. The `.detach()` call fires the promise and discards its result entirely. If the transfer receipt fails — for example because the recipient account no longer exists at execution time — the failure is silently swallowed, the funds are never delivered, and no recovery path exists. This is the direct NEAR analog of the Solidity `transfer()` anti-pattern flagged in M-01.

### Finding Description

Three production refund sites in `crates/contract/src/lib.rs` share the same pattern:

**Site 1 — `require_deposit` (called by every `sign` and `request_app_private_key` invocation):** [1](#0-0) 

**Site 2 — `submit_participant_info` (TEE participant registration):** [2](#0-1) 

**Site 3 — `propose_update` (governance update proposal):** [3](#0-2) 

In all three cases the pattern is identical:

```rust
Promise::new(account_id).transfer(diff).detach();
```

`.detach()` schedules the transfer as an independent receipt and drops the `Promise` handle, meaning the contract never inspects whether the receipt succeeded or failed. In NEAR, a transfer receipt to a non-existent account fails; the runtime does not automatically re-credit the contract, and no callback is registered to detect or handle the failure.

**Concrete attack path (Site 1 / `require_deposit`):**

1. Attacker constructs a batch transaction containing two actions: `FunctionCall { method: "sign", deposit: D }` (where `D` > 1 yoctoNEAR) followed by `DeleteAccount { beneficiary: attacker2 }`.
2. The `FunctionCall` receipt executes first: `require_deposit` computes `diff = D − 1` and schedules `Promise::new(predecessor).transfer(diff).detach()`.
3. The `DeleteAccount` action executes in the same receipt, removing the predecessor account.
4. The refund receipt fires in the next execution step; the target account no longer exists; the receipt fails silently.
5. `diff` yoctoNEAR is permanently locked inside the contract with no recovery mechanism.

The same path applies to Sites 2 and 3 with the respective callers (`account_id` / `proposer`).

### Impact Explanation

Excess deposits attached to `sign`, `request_app_private_key`, `submit_participant_info`, and `propose_update` calls can be permanently frozen inside the contract. The contract holds no withdrawal or sweep function for stranded balances, so locked funds are irrecoverable. This breaks the explicit accounting invariant documented in the code ("refund excess deposit … to predecessor") and constitutes permanent freezing of user funds controlled by the MPC contract.

This maps to: **Medium — balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration.**

### Likelihood Explanation

The `DeleteAccount` action is a standard NEAR protocol primitive available to any account owner. Batch transactions combining a function call with `DeleteAccount` are valid and routinely used in NEAR tooling. No privileged access, threshold collusion, or TEE bypass is required. Any unprivileged caller can trigger this against their own account. The minimum deposit is 1 yoctoNEAR, so the exploitable surplus is whatever the caller chose to attach above that floor; for `propose_update` this can be meaningful (WASM storage costs are on the order of 1–2 NEAR).

### Recommendation

Replace every `.detach()` refund with a chained callback that verifies the transfer succeeded, or — simpler and idiomatic in NEAR — use `#[callback_result]` on a dedicated `on_refund` method. At minimum, avoid `.detach()` on any promise that carries user funds:

```rust
// Instead of:
Promise::new(account_id.clone()).transfer(diff).detach();

// Use a then-callback to detect failure and log/re-credit:
Promise::new(account_id.clone())
    .transfer(diff)
    .then(
        Self::ext(env::current_account_id())
            .on_refund(account_id, diff)
    );
```

Alternatively, adopt a pull-payment pattern: credit the refund amount to a per-account claimable balance stored in contract state, and let users withdraw it explicitly. This eliminates the push-transfer failure mode entirely.

### Proof of Concept

```
1. alice.near has balance B.
2. alice.near submits batch tx:
     Action[0]: FunctionCall("sign", payload, deposit=10_000_000 yN)
     Action[1]: DeleteAccount(beneficiary=bob.near)
3. Contract receipt executes require_deposit:
     diff = 10_000_000 - 1 = 9_999_999 yN
     Promise::new("alice.near").transfer(9_999_999).detach()  ← scheduled
4. DeleteAccount executes: alice.near is removed.
5. Refund receipt fires: alice.near does not exist → receipt fails silently.
6. 9_999_999 yN remains in mpc-contract with no recovery path.

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(

        Some(diff) => {

            }
        }
    }
}
