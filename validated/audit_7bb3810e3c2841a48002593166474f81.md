Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Deposits Beyond Protocol Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric comparison for ETH vs ERC20 assets: the ETH branch tests only whether the running total already exceeds the limit, without adding the incoming `amount`. Any unprivileged caller can therefore deposit arbitrarily large ETH amounts past the configured cap, receiving rsETH for the full value while the excess ETH sits idle and earns no yield, permanently diluting yield for all rsETH holders.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676-682), the ETH branch is:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // missing + amount
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC20
``` [1](#0-0) 

The guard returns `true` (blocking the deposit) only when the running total *already* exceeds the limit. It never tests whether the new `msg.value` would push the total over the limit. The ERC20 path correctly adds `amount` before comparing.

`depositETH` (L76-93) calls `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`, then unconditionally calls `_mintRsETH(rsethAmountToMint)` where `rsethAmountToMint` is computed from the full `msg.value`. [2](#0-1) 

Because the guard never fires for a deposit that would *cross* the limit (only for one made *after* the limit is already exceeded), rsETH is minted for the entire deposited ETH regardless of how far it exceeds the remaining capacity. [3](#0-2) 

## Impact Explanation
The deposit limit exists to match EigenLayer strategy capacity. ETH deposited beyond that capacity cannot be delegated to EigenLayer and sits idle in the DepositPool/NDCs earning no yield. Because rsETH is minted for the full surplus amount, the yield generated by legitimately deployed assets is spread across a larger rsETH supply, permanently suppressing the yield accrued by all existing rsETH holders. This is **Permanent freezing of unclaimed yield (Medium)**.

## Likelihood Explanation
`depositETH` is fully permissionless — any address can call it with arbitrary `msg.value`. The vulnerable condition (`getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`) is the normal operating state of the protocol. No role, signature, or special precondition is required. Likelihood is **High**.

## Recommendation
Mirror the ERC20 logic in the ETH branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
1. ETH deposit limit is set to 1 000 ETH; `getTotalAssetDeposits(ETH)` = 999 ETH.
2. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999 > 1000` → `false` → deposit is not blocked.
4. `getRsETHAmountToMint(ETH, 10_000 ether)` returns rsETH proportional to 10 000 ETH.
5. `_mintRsETH` mints that full amount to the attacker.
6. Protocol now holds 10 999 ETH — 9 999 ETH over the cap — with rsETH minted for the entire surplus. The excess ETH cannot be deployed to EigenLayer, permanently suppressing yield for all rsETH holders.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    // set limit to 1000 ETH, deposit 999 ETH to approach limit
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 1000 ether);
    depositPool.depositETH{value: 999 ether}(0, "");
    // now deposit 10_000 ETH — should revert but does not
    depositPool.depositETH{value: 10_000 ether}(0, "");
    // assert total deposits >> limit
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
``` [1](#0-0)
