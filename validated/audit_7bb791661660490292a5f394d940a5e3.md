### Title
CKD Response Accepted Without Output Validation for `AppPublicKey` Variant — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs cryptographic output validation **only** for the `AppPublicKeyPV` variant of CKD requests. For the `AppPublicKey` variant, no output check is performed at all. This mirrors the Router M-04 pattern exactly: just as the Router accepted a partial swap without verifying the output amount, the MPC contract accepts a CKD response without verifying its correctness. A single Byzantine attested participant — strictly below the signing threshold — can submit a forged CKD response that the contract will accept and deliver to the user.

---

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`, lines 653–689), the contract branches on the `app_public_key` variant of the request:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(C, g2) = e(Y, app_pk2) · e(H(app_id), pk)`, which cryptographically binds the response to the threshold-computed output. For `AppPublicKey`, the arm is a no-op — the response bytes are accepted unconditionally.

The only gate before this branch is `assert_caller_is_attested_participant_and_protocol_active()`, which requires the caller to be **one** attested participant, not a threshold-many quorum: [2](#0-1) 

After the (absent) check, `resolve_yields_for` immediately resumes every queued yield with the submitted response bytes: [3](#0-2) 

The fan-out helper in `pending_requests.rs` drains the entire queue and delivers the response to all waiting callers without any further validation: [4](#0-3) 

---

### Impact Explanation

A single Byzantine attested participant — one node, well below the signing threshold — can call `respond_ckd` for any pending `AppPublicKey` CKD request and supply an arbitrary `CKDResponse`. The contract will:

1. Accept the forged response (no output check).
2. Resume every yield queued for that request with the forged bytes.
3. Return the forged confidential key derivation output to the user(s).

Because the CKD output is used to derive application-specific private keys (the whole purpose of the `request_app_private_key` flow), a forged output means the user receives a key that the attacker controls or knows, enabling theft of any assets protected by that derived key. This satisfies the **Critical** impact tier: *confidential key derivation output without the required participant authorization*.

---

### Likelihood Explanation

The attacker must be an attested MPC participant — a realistic adversary model explicitly in scope (Byzantine participant strictly below the signing threshold). No threshold collusion is required; a single compromised or malicious node suffices. The `AppPublicKey` variant is a documented, production-facing API path (noted in `request_app_private_key`'s docstring), so real users will exercise it. The attack requires only one malicious `respond_ckd` call, which is a standard node operation.

---

### Recommendation

Apply the same pairing-based output check to `AppPublicKey` requests, or — if the app public key is intentionally withheld from the contract — require threshold-many independent `respond_ckd` submissions that must agree before the yield is resolved (analogous to how Uniswap v3 enforces full-fill when `sqrtPriceLimitX96 == 0`). At minimum, document that `AppPublicKey` requests carry no on-chain output integrity guarantee and warn integrators accordingly.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk)` and attaches the required deposit. The contract enqueues a yield and stores the `CKDRequest`.
2. Byzantine attested participant (one node) constructs an arbitrary `CKDResponse` containing attacker-chosen `big_c` and `big_y` values.
3. Byzantine node calls `respond_ckd(request, forged_response)`. The contract passes `assert_caller_is_attested_participant_and_protocol_active()`, enters the `AppPublicKey` arm (no-op), and calls `resolve_yields_for`, resuming the user's yield with the forged bytes.
4. The user's `request_app_private_key` call returns the forged CKD output. The user derives a private key from it — a key the attacker already knows — and uses it to protect assets. The attacker can now spend those assets.

No threshold collusion, no TEE break, no network-level attack required.

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
```
