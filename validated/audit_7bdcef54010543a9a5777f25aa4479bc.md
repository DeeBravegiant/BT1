Audit Report

## Title
Missing Access Control on `sendETHToL1ViaBridge` Allows Any Caller to Exhaust Linea Bridge Period Limit, Temporarily Freezing Pool ETH - (File: contracts/bridges/LineaMessenger.sol)

## Summary

`LineaMessenger.sendETHToL1ViaBridge` is declared `external payable` with only a `nonReentrant` guard and no role check or caller whitelist. Any unprivileged attacker can call it with the real Linea bridge address and their own ETH, consuming the bridge's global `limitInWei` period quota. Once exhausted, the pool's `bridgeAssetsViaNativeBridge` call reverts, freezing all accumulated pool ETH until the next period resets.

## Finding Description

`LineaMessenger.sendETHToL1ViaBridge` (line 31) has no access control:

```solidity
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value) external payable nonReentrant {
```

The only guards are non-zero address checks, a value match check, and a minimum fee check — none of which restrict the caller. The function accepts an arbitrary `l2bridge` address and forwards the caller's ETH directly to `ILineaMessageService(l2bridge).sendMessage`.

The Linea bridge enforces a **global** `limitInWei` per period (exposed via `ILineaMessageService.limitInWei()` and `currentPeriodAmountInWei()`). Because this limit is global and not per-sender, any call to `sendMessage` — regardless of who initiates it — counts against the shared quota.

An attacker calls `sendETHToL1ViaBridge{value: remaining}(LINEA_BRIDGE, attacker_l1_addr, remaining)` where `remaining = limitInWei - currentPeriodAmountInWei`. The attacker's ETH is bridged to their own L1 address, recovering principal minus `minimumFee`.

After exhaustion, when `BRIDGER_ROLE` calls any of the following:
- `RSETHPoolV2ExternalBridge.bridgeAssetsViaNativeBridge` (line 477)
- `RSETHPoolV2.bridgeAssets` (line 294)
- `RSETHPoolV3ExternalBridge.bridgeAssetsViaNativeBridge` (line 659)

...each routes through `IL2Messenger(messenger).sendETHToL1ViaBridge`, which calls `sendMessage` on the Linea bridge and reverts with a period-limit error. Pool ETH remains locked until the next period resets.

## Impact Explanation

**Medium. Temporary freezing of funds.** All ETH accumulated in the Linea pool(s) is frozen for the duration of each Linea bridge rate-limit period. Users who deposited ETH expecting it to be bridged to L1 experience delayed settlement. If the attacker repeats the attack at every period reset (cost: only `minimumFee` per period, principal recovered on L1), the freeze becomes effectively permanent.

## Likelihood Explanation

High. The preconditions are minimal: (1) `sendETHToL1ViaBridge` is publicly callable with no access control — confirmed; (2) the attacker needs ETH equal to the remaining period quota, which is fully recoverable on L1 minus fees; (3) no front-running, privileged access, or victim mistakes required. The attack is a single transaction and is repeatable every period reset.

## Recommendation

Restrict `sendETHToL1ViaBridge` to only be callable by the authorized pool contract(s). The simplest fix is to add an access control check using the inherited `Recoverable` admin pattern or an explicit allowlist:

```solidity
address public authorizedCaller;

modifier onlyAuthorizedCaller() {
    if (msg.sender != authorizedCaller) revert Unauthorized();
    _;
}

function sendETHToL1ViaBridge(...) external payable nonReentrant onlyAuthorizedCaller { ... }
```

Alternatively, store the canonical `l2bridge` address in the contract at deployment and remove it as a caller-supplied parameter, eliminating the ability for arbitrary callers to route ETH through the contract entirely.

## Proof of Concept

```solidity
// Fork Linea mainnet
// LineaMessenger deployed at 0x838686d23521435B528F68Cde6404C07ae007299

ILineaMessageService bridge = ILineaMessageService(LINEA_BRIDGE);
LineaMessenger messenger = LineaMessenger(0x838686d23521435B528F68Cde6404C07ae007299);

uint256 remaining = bridge.limitInWei() - bridge.currentPeriodAmountInWei();

// Step 1: attacker exhausts the period limit using their own ETH
messenger.sendETHToL1ViaBridge{value: remaining}(LINEA_BRIDGE, attacker, remaining);

// Step 2: assert period limit is now exhausted
assertEq(bridge.limitInWei() - bridge.currentPeriodAmountInWei(), 0);

// Step 3: BRIDGER_ROLE attempts to bridge pool ETH — reverts
vm.prank(BRIDGER_ROLE);
vm.expectRevert(); // Linea bridge: RateLimitExceeded
RSETHPoolV2ExternalBridge(LINEA_POOL).bridgeAssetsViaNativeBridge(poolBalance);
// Pool ETH is now frozen until next period reset
```
