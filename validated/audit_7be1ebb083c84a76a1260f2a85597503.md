### Title
Single-Participant Empty-Registration Freezes All Foreign-Chain Verification — (`crates/contract/src/lib.rs`)

### Summary
`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which requires **every** active participant to have registered a given chain. A single participant calling `register_foreign_chain_support({})` (empty set) drops the entire supported set to empty, causing every subsequent `verify_foreign_transaction` call to panic with `ForeignChainNotSupported`. The threshold-based fix (`get_available_foreign_chains()`) exists in the codebase but is not yet wired to `verify_foreign_transaction`.

### Finding Description
`verify_foreign_transaction` calls `get_supported_foreign_chains()` at line 534, which iterates `node_foreign_chain_support` and filters chains where `nodes_supporting_chain.is_superset(&active_participant_account_ids)` — i.e., only chains that **all** active participants have registered. If any single participant inserts an empty entry via `register_foreign_chain_support`, no chain can satisfy the superset condition, so the returned set is empty. The check at line 535 then panics for any requested chain.

The contract already has a correct threshold-based path: `recompute_available_foreign_chains()` counts per-chain supporters and compares against the `ForeignTx` domain's `reconstruction_threshold`, stored in `foreign_chains.available_foreign_chains`. `verify_foreign_transaction` ignores this and reads from the legacy `node_foreign_chain_support` store instead.

### Impact Explanation
Any authenticated participant (a single node, below signing threshold) can persistently block all `verify_foreign_transaction` calls by repeatedly submitting an empty chain registration. Bridge users receive `ForeignChainNotSupported` panics for every chain. Recovery requires either the attacker to re-register with chains, or a governance action to remove the attacker from the participant set — neither of which is automatic.

### Likelihood Explanation
The attacker only needs to be a participant. The call is permissioned only by `voter_or_panic()` (signer == predecessor, must be a participant). No collusion, no threshold, no special privilege beyond participant membership is required. The attack is a single on-chain transaction.

### Recommendation
Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with a check against `get_available_foreign_chains()` (the threshold-based cached set). The infrastructure is already present: `recompute_available_foreign_chains()` correctly uses the `ForeignTx` domain's `reconstruction_threshold` and is called on every `register_foreign_chains_config` invocation. The migration plan in `docs/design/calculating-supported-foreign-chains.md` already describes this transition; it should be completed before the next production deployment.

### Proof of Concept
```rust
// 1. All n participants register Bitcoin
for participant in &participants {
    contract.register_foreign_chain_support(BTreeSet::from([ForeignChain::Bitcoin]).into());
}
// get_supported_foreign_chains() returns {Bitcoin}

// 2. One participant registers empty set
contract.register_foreign_chain_support(BTreeSet::new().into());

// 3. get_supported_foreign_chains() now returns {}
assert!(contract.get_supported_foreign_chains().is_empty());

// 4. verify_foreign_transaction panics
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    chain: ForeignChain::Bitcoin, ..
}); // panics: ForeignChainNotSupported
```

        &mut self,

Today, `get_supported_foreign_chains()` returns the **strict intersection** of every
active participant's registered chains, and `verify_foreign_transaction` rejects any
request whose target chain is not in it. A single node that registers an empty list
(or hasn't registered yet) drops **every** chain — one operator can take the whole
feature down. That is what this proposal fixes.
