Audit Report

## Title
Unrestricted `sendFunds()` in `FeeReceiver` Enables Stale-Price Deposit to Steal Yield from rsETH Holders - (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` carries no access-control modifier, allowing any external caller to flush accumulated MEV/EL rewards into `LRTDepositPool` at will. Because `LRTOracle.updateRSETHPrice()` is also publicly callable and enforces a `pricePercentageLimit` guard that reverts on large TVL spikes, an attacker can sequence these two calls to leave `rsETHPrice` stale at its pre-reward value, then deposit at that artificially low price — capturing yield that belongs to existing rsETH holders.

## Finding Description

**Root cause — no access control on `sendFunds()`:**

`FeeReceiver.sendFunds()` is declared `external` with no role modifier: [1](#0-0) 

`LRTDepositPool.receiveFromRewardReceiver()` is equally unrestricted: [2](#0-1) 

Once ETH lands in the deposit pool, `getETHDistributionData()` counts it immediately via `address(this).balance`: [3](#0-2) 

**Price guard causes revert, leaving `rsETHPrice` stale:**

`updateRSETHPrice()` is publicly callable: [4](#0-3) 

Inside `_updateRsETHPrice()`, if the computed new price exceeds `highestRsethPrice` by more than `pricePercentageLimit`, and the caller is not a MANAGER, the function reverts — leaving `rsETHPrice` unchanged: [5](#0-4) 

**Deposits use the stale stored price:**

`getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, which is the stored state variable — not a live recalculation: [6](#0-5) 

A lower stale denominator means the attacker receives more rsETH per ETH than the true exchange rate warrants.

**Exploit path:**
1. `FeeReceiver` accumulates a large MEV reward balance over days/weeks.
2. Attacker calls `FeeReceiver.sendFunds()` — no role check, succeeds for any caller.
3. ETH moves to `LRTDepositPool`; TVL spikes by the full reward amount.
4. Attacker calls `LRTOracle.updateRSETHPrice()` — the spike exceeds `pricePercentageLimit`, the call reverts with `PriceAboveDailyThreshold`, and `rsETHPrice` is NOT updated.
5. Attacker calls `LRTDepositPool.depositETH{value: X}(minRSETH, "")` — minting uses the stale (lower) `rsETHPrice`, yielding more rsETH than the true rate.
6. Manager eventually calls `updateRSETHPriceAsManager()` — price updates to the true higher value, and the attacker's inflated rsETH balance is now worth more than it should be, at the expense of all pre-existing holders.

**Why existing guards are insufficient:**
- The `pricePercentageLimit` guard is the mechanism that makes the stale-price window exploitable; it is designed to protect against oracle manipulation but here it is weaponized.
- `receiveFromRewardReceiver()` has no caller check, so the ETH injection cannot be blocked at the deposit pool level.
- `depositETH` has `nonReentrant` and `whenNotPaused` but neither prevents a deposit at a stale price.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders are entitled to the price appreciation that accumulated MEV rewards represent. By depositing at the stale pre-reward price, the attacker's rsETH balance is inflated relative to the true TVL, permanently diluting every pre-existing holder's proportional claim on the underlying assets. The stolen value equals the attacker's excess rsETH multiplied by the true rsETH price after the manager update.

## Likelihood Explanation
**Medium.** `pricePercentageLimit` is explicitly configurable via `setPricePercentageLimit()` and the guard code is written to be enforced in production. MEV/EL rewards accumulate continuously; over days or weeks the balance can be large enough to push a single-block TVL spike past the configured threshold. The attack requires only two public calls (`sendFunds()` + `updateRSETHPrice()`) and a subsequent deposit — no special privileges, no flash loan, no front-running of a specific transaction. It is repeatable each time rewards accumulate past the threshold.

## Recommendation
Restrict `sendFunds()` to an authorized role so only a trusted keeper or manager can flush rewards:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Additionally, restrict `receiveFromRewardReceiver()` in `LRTDepositPool` to only accept calls from the registered `FeeReceiver` address, preventing direct ETH injection from arbitrary callers.

## Proof of Concept

```
Setup:
  - FeeReceiver holds 500 ETH in MEV rewards.
  - rsETHPrice = 1.05e18, highestRsethPrice = 1.05e18.
  - pricePercentageLimit = 1e16 (1%).
  - rsETH totalSupply = S.

Step 1: attacker.call(FeeReceiver.sendFunds())
  → 500 ETH transferred to LRTDepositPool (no access control).
  → LRTDepositPool.balance += 500 ETH.

Step 2: attacker.call(LRTOracle.updateRSETHPrice())
  → _getTotalEthInProtocol() returns previousTVL + 500 ETH.
  → newRsETHPrice = (previousTVL + 500e18) / S >> 1.05e18 * 1.01.
  → isPriceIncreaseOffLimit = true, caller is not MANAGER.
  → REVERT: PriceAboveDailyThreshold().
  → rsETHPrice remains at 1.05e18 (stale).

Step 3: attacker.call(LRTDepositPool.depositETH{value: X}(minRSETH, ""))
  → getRsETHAmountToMint uses rsETHPrice = 1.05e18 (stale).
  → attacker receives X / 1.05e18 rsETH instead of the correct lower amount.

Step 4: manager.call(LRTOracle.updateRSETHPriceAsManager())
  → rsETHPrice updated to true higher value (e.g., 1.08e18).
  → attacker's rsETH is now redeemable at 1.08e18, having paid only 1.05e18.
  → excess rsETH value = (1.08e18 - 1.05e18) / 1.08e18 * X, stolen from existing holders.

Foundry fork test plan:
  1. Fork mainnet at a block where FeeReceiver holds significant ETH.
  2. Assert sendFunds() succeeds from address(0xdead).
  3. Assert updateRSETHPrice() reverts with PriceAboveDailyThreshold.
  4. Record attacker rsETH balance after depositETH.
  5. Call updateRSETHPriceAsManager() as manager.
  6. Assert attacker rsETH value > X (deposited ETH), proving yield theft.
```
