Audit Report

## Title
Unconditional `msg.sender` Gate in `_processClaim` Permanently Freezes Unclaimed KERNEL Yield for Non-Self-Executing Contract Recipients â€” (`contracts/KERNEL/KernelMerkleDistributor.sol`)

## Summary

`KernelMerkleDistributor._processClaim` enforces `account == msg.sender` unconditionally at lines 311â€“313. Any contract address included in the merkle tree that lacks a mechanism to dispatch external calls (no `execute`, `delegatecall`, or equivalent) can never satisfy this check. Because the contract has no admin token-rescue function, the KERNEL yield allocated to such addresses is permanently unclaimable and locked in the distributor.

## Finding Description

`_processClaim` contains a hard gate:

```solidity
// KernelMerkleDistributor.sol L311-313
if (account != msg.sender) {
    revert Unauthorized();
}
``` [1](#0-0) 

Both public entry points route exclusively through this function with no bypass:

- `claim` at line 261: [2](#0-1) 
- `claimAndStake` at line 280: [3](#0-2) 

For an EOA or a smart contract wallet with a generic `execute` function (e.g., Gnosis Safe), the check is satisfiable because the account itself becomes `msg.sender`. However, for any contract address that cannot initiate external calls â€” a pure receiver, a simple vault, a yield aggregator with no generic dispatch â€” there is no code path that makes `msg.sender == account`. Every call to `claim` or `claimAndStake` for such an address reverts with `Unauthorized`, regardless of who calls it.

The merkle proof already cryptographically binds `(index, account, cumulativeAmount)` to the tree root, and `kernel.safeTransfer(account, amountToSend)` always sends to the proven `account`. The `msg.sender` guard provides no additional security against fund redirection, yet it creates an irrecoverable freeze for non-self-executing contract recipients.

Critically, the admin functions in `KernelMerkleDistributor` (lines 348â€“424) include only `setKernelDepositPool`, `setProtocolTreasury`, `setFeeInBPS`, `setMerkleRoot`, `pause`, and `unpause` â€” there is no token-rescue function: [4](#0-3) 

This contrasts directly with `KernelTop100MerkleDistributor`, which provides a `withdrawTokens` owner escape hatch: [5](#0-4) 

## Impact Explanation

Any KERNEL yield allocated in the merkle tree to a contract address that cannot self-execute is permanently unclaimable. The tokens remain locked in the distributor with no admin rescue path. This matches the allowed impact: **Medium â€” Permanent freezing of unclaimed yield**.

## Likelihood Explanation

The merkle tree is constructed off-chain. Protocol participants who interact via contract addresses â€” DAO treasuries, simple escrow contracts, yield aggregator vaults without generic `execute` functions â€” are realistic recipients if they held rsETH or otherwise qualified for the distribution. The protocol has no on-chain mechanism to detect or exclude such addresses before the root is set. Once a root is set and the distribution round advances, the freeze is irreversible for any affected address.

## Recommendation

1. **Remove or relax the `msg.sender` check.** The merkle proof cryptographically binds `(index, account, cumulativeAmount)` to the root, and all transfers go to the proven `account`. The guard provides no additional security and should be removed.
2. **Add an owner-controlled token rescue function** mirroring `KernelTop100MerkleDistributor.withdrawTokens` as a safety net for any future edge cases.
