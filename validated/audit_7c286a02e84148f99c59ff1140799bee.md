Audit Report

## Title
Missing Chainlink Price Feed Staleness Validation Enables Stale-Price rsETH Minting — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards all validity indicators (`updatedAt`, `answeredInRound`, `roundId`), accepting stale or frozen Chainlink feed data as current. Because `LRTOracle.updateRSETHPrice()` is an unrestricted public function that uses this oracle to compute the protocol-wide rsETH/ETH price, any caller can lock in a stale low price and immediately deposit to mint excess rsETH, diluting existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches round data but reads only the `price` field:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

`updatedAt`, `answeredInRound`, `roundId`, and `price > 0` are all unchecked. By contrast, the pool-level wrapper `ChainlinkOracleForRSETHPoolCollateral.getRate()` performs all three validity checks before returning a price: [2](#0-1) 

`LRTOracle.getAssetPrice()` delegates directly to `ChainlinkPriceOracle` for all L1 LST assets: [3](#0-2) 

That price feeds `_getTotalEthInProtocol()`, which feeds `_updateRsETHPrice()`, which sets the stored `rsETHPrice` used for all deposit minting: [4](#0-3) 

`updateRSETHPrice()` is an unrestricted public function — any EOA or contract can call it: [5](#0-4) 

**Existing guards are insufficient.** The `pricePercentageLimit` downside protection at lines 270–281 only pauses the protocol if the price drop *exceeds* the configured limit. If `pricePercentageLimit == 0` (no limit set), there is zero protection. Even when set, a stale deviation within the threshold (e.g., 0.5% stale deviation with a 1% limit) passes through silently and allows the attack to succeed. [6](#0-5) 

## Impact Explanation
**High — Theft of unclaimed yield / share dilution.**

When a Chainlink LST/ETH feed goes stale at a price below the true current price, an attacker calls `updateRSETHPrice()` to write a depressed `rsETHPrice` to storage. They then immediately deposit into `LRTDepositPool`, which mints rsETH at the artificially low price — receiving more rsETH than the deposited value warrants. When the feed recovers and `rsETHPrice` is corrected upward, the attacker's excess rsETH is backed by value extracted from all existing rsETH holders. This is a direct, quantifiable theft of unclaimed yield, matching the allowed High impact class.

## Likelihood Explanation
**Medium.** Chainlink feeds have heartbeat intervals (commonly 1 hour for LST feeds) and deviation thresholds. Staleness windows occur during Ethereum network congestion, Chainlink node outages, and feed deprecation/migration periods. The attack requires no special role, no governance capture, and no private key compromise. Any depositor can execute it permissionlessly by monitoring feed staleness and calling `updateRSETHPrice()` at the right moment. The attack is repeatable whenever a feed goes stale.

## Recommendation
Add staleness and validity guards to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be set per-asset based on the Chainlink feed's documented heartbeat interval (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer).

## Proof of Concept
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale at price `P_stale < P_true` (e.g., during congestion), with the deviation within `pricePercentageLimit` so no auto-pause triggers.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `P_stale` without any staleness check.
4. `newRsETHPrice = (stale_TVL - fee) / rsethSupply` — lower than true price — is written to `rsETHPrice`.
5. Attacker calls `LRTDepositPool.depositAsset(stETH, largeAmount)`.
6. Deposit pool mints `rsETHAmount = largeAmount * P_true / rsETHPrice_stale` — more rsETH than deserved.
7. Feed recovers; next `updateRSETHPrice()` call raises `rsETHPrice`, but attacker holds excess rsETH backed by other users' value.

**Foundry fork test plan:** Fork mainnet, mock a Chainlink stETH/ETH aggregator returning a stale answer (set `updatedAt` to `block.timestamp - 2 hours`, `answeredInRound < roundId`). Call `LRTOracle.updateRSETHPrice()` as an unprivileged EOA. Assert `rsETHPrice` is set to the stale-derived value. Then call `LRTDepositPool.depositAsset(stETH, amount)` and assert the minted rsETH exceeds what would be minted at the true price. Advance time, call `updateRSETHPrice()` again with a fresh feed, and assert the attacker's rsETH balance represents excess value extracted from the pool.
