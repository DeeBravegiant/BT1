### Title
Sub-Transaction Reference Script Costs Excluded from Minimum Fee Calculation in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

### Summary

In the Dijkstra era, the minimum fee calculation for a batch (top-level) transaction does not account for reference scripts used inside sub-transactions. The helper `batchNonDistinctRefScriptsSize` correctly aggregates reference script sizes across all sub-transactions, but the `EraUTxO` instance for `DijkstraEra` reuses `getConwayMinFeeTxUtxo`, which only measures the top-level transaction's reference scripts. An attacker can therefore craft a batch transaction whose sub-transactions carry arbitrarily large reference scripts while the enforced minimum fee is computed as if those scripts do not exist, paying far less than the validation cost they impose on every validating node.

### Finding Description

The Conway era introduced tiered reference-script pricing (`minFeeRefScriptCostPerByte`) specifically to prevent DDoS attacks where large scripts are cheap to submit but expensive to deserialize and validate. The Dijkstra era extends Conway with nested sub-transactions (`subTransactionsTxBodyL`). Each sub-transaction body (`DijkstraSubTxBodyRaw`) can carry its own `referenceInputsTxBodyL`, pointing to UTxO entries that contain reference scripts.

The Dijkstra `EraUTxO` instance delegates the minimum-fee computation directly to the Conway implementation:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getConwayMinFeeTxUtxo   -- line 141
```

`getConwayMinFeeTxUtxo` calls `txNonDistinctRefScriptsSize`, which only inspects the top-level transaction body:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx   -- line 175
```

The Dijkstra module itself defines the correct batch-aware counterpart:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)   -- lines 271-277
      )
```

`batchNonDistinctRefScriptsSize` is exported but never wired into `getMinFeeTxUtxo`. The minimum-fee check in the Dijkstra UTXO rule therefore enforces:

```
minfee pp txTop utxo₀ ≤ txfee txb
```

using only the top-level reference script size, while the actual deserialization and validation cost is proportional to the combined reference script size across the top-level transaction **and all sub-transactions**. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

**Medium — Attacker-controlled transactions exceed intended validation limits or modify fees outside design parameters.**

An attacker submits a batch transaction whose top-level body carries a small or zero-size reference script (paying a low fee) while embedding multiple sub-transactions that each reference large Plutus scripts via `referenceInputsTxBodyL`. Every honest node must deserialize all referenced scripts for every sub-transaction during validation, but the fee collected covers only the top-level script overhead. This recreates the exact DDoS vector that the `minFeeRefScriptCostPerByte` parameter was introduced to close (see ADR 9), now re-opened for the Dijkstra batch-transaction surface. The attacker pays a fee calibrated to zero sub-transaction reference scripts while imposing the full deserialization cost of all sub-transaction reference scripts on the network. [4](#0-3) 

### Likelihood Explanation

The entry path is fully unprivileged: any transaction submitter can construct a `DijkstraTxBody` with `dtbrSubTransactions` containing sub-transactions that reference large scripts. No special role, key, or governance action is required. The `DijkstraSubTxBodyRaw` structure explicitly supports `dstbrReferenceInputs`, and the `SUBUTXO` rule validates those inputs against `originalUtxo`, confirming the feature is reachable on-chain. The discrepancy between `batchNonDistinctRefScriptsSize` (defined but unused in fee logic) and `getConwayMinFeeTxUtxo` (used) is a straightforward omission. [5](#0-4) [6](#0-5) 

### Recommendation

Replace the `getMinFeeTxUtxo` binding in the `EraUTxO DijkstraEra` instance with a Dijkstra-specific implementation that passes `batchNonDistinctRefScriptsSize utxo tx` instead of `txNonDistinctRefScriptsSize utxo tx` to `getMinFeeTx`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getDijkstraMinFeeTxUtxo :: (EraTx era, BabbageEraTxBody era, DijkstraEraTxBody era)
  => PParams era -> Tx TopTx era -> UTxO era -> Coin
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx

instance EraUTxO DijkstraEra where
  ...
  getMinFeeTxUtxo = getDijkstraMinFeeTxUtxo
```

This ensures the tiered reference-script pricing applies to the full batch, consistent with the intent of ADR 9 and the existing `batchNonDistinctRefScriptsSize` helper. [7](#0-6) 

### Proof of Concept

1. Locate a UTxO entry containing a large Plutus V3 script (e.g., near the 200 KiB per-transaction limit).
2. Construct a `DijkstraTxBody` (top-level) with:
   - `dtbrSubTransactions`: an `OMap` of N sub-transactions, each with `dstbrReferenceInputs` pointing to the large-script UTxO entry.
   - `dtbrTxfee`: the minimum fee computed by `getConwayMinFeeTxUtxo` (top-level only, near-zero ref-script component).
3. Submit the transaction. The `FeeTooSmallUTxO` check passes because it uses `txNonDistinctRefScriptsSize` (top-level only). Each validating node must deserialize the large script N times (once per sub-transaction), but the fee collected reflects 0 bytes of sub-transaction reference scripts.
4. Repeat with increasing N or script size to amplify the validation cost/fee ratio beyond what the protocol intends to permit. [8](#0-7) [3](#0-2)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability.
