### Title
Unprotected `init` Function Allows Any Caller to Front-Run Contract Initialization and Seize Full MPC Network Control - (File: `crates/contract/src/lib.rs`)

### Summary

The `MpcContract::init` function lacks the `#[private]` access-control attribute present on every other initialization-class method (`init_running`, `migrate`). Because NEAR contract deployment and initialization are always separate transactions, any unprivileged account can observe the deployment transaction and race to call `init` first with attacker-controlled `ThresholdParameters`, installing themselves as the sole participant of the MPC network before the legitimate operator does.

### Finding Description

`MpcContract::init` is the primary production initialization entry point. It is decorated with `#[init]` (which prevents re-initialization once state exists) but carries **no** `#[private]` guard:

```rust
// crates/contract/src/lib.rs:1924-1926
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

By contrast, the two other `#[init]`-annotated methods are both `#[private]`:

```rust
#[private]
#[init]
pub fn init_running(...) -> Result<Self, Error> { ... }

#[private]
#[init(ignore_state)]
pub fn migrate() -> Result<Self, Error> { ... }
``` [2](#0-1) [3](#0-2) 

In NEAR, `#[private]` enforces `predecessor_account_id == current_account_id`, meaning only the contract account itself can invoke the method. Without it, `init` is callable by any NEAR account.

Every documented deployment procedure issues deployment and initialization as **separate transactions** with an observable gap between them:

- Localnet guide: `without-init-call` deploy, then a separate `init` call
- Testnet guide: `without-init-call` deploy, then a separate `init` call
- TEE cluster script: `deploy_contract` phase, then `near_init` phase [4](#0-3) [5](#0-4) [6](#0-5) 

During this window, an attacker can submit their own `init` call with `ThresholdParameters` that name only attacker-controlled accounts as participants and set `threshold = 1`. The function accepts any caller, validates only the structural correctness of the parameters, and then calls `TeeState::with_mocked_participant_attestations` on the attacker-supplied participant set — granting those participants valid mock TEE attestations:

```rust
// crates/contract/src/lib.rs:1944-1945
let initial_participants = parameters.participants();
let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);
``` [7](#0-6) [8](#0-7) 

Once `init` succeeds, the `#[init]` guard permanently blocks any subsequent call to `init`, locking the legitimate operator out. The attacker's account is now the only recognized participant with a valid mock attestation, and the contract is in `Running` state.

### Impact Explanation

With sole participant status and threshold 1, the attacker can:

1. Call `vote_add_domains` unilaterally to register signing domains (Secp256k1, Ed25519, BLS12-381).
2. Drive the entire DKG ceremony alone (`start_keygen_instance`, `vote_pk`) to generate MPC key shares under their exclusive control.
3. Respond to any `sign`, `request_app_private_key`, or `verify_foreign_transaction` request with attacker-crafted outputs — issuing threshold signatures and confidential key derivations without any legitimate participant authorization.
4. Permanently block the legitimate operator from ever initializing or recovering the contract, since `#[init]` prevents re-initialization.

This constitutes **unauthorized threshold signature issuance** and **unauthorized access to MPC signing capability**, matching the Critical impact tier: *"Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization."*

### Likelihood Explanation

The attack requires only a standard NEAR account and the ability to monitor the mempool or block explorer for the deployment transaction. No privileged access, key material, or collusion is needed. The deployment-to-initialization gap is inherent to the documented multi-step procedure and is present in every deployment script in the repository. The attack is a single transaction submitted in the window between `deploy_contract` and `init_contract`.

### Recommendation

Add `#[private]` to `init`, consistent with `init_running` and `migrate`:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

Since all deployment scripts already sign the `init` call `as` the contract account (`sign-as $MPC_CONTRACT_ACCOUNT`), adding `#[private]` does not break any existing workflow — it only prevents third-party accounts from calling `init`.

Alternatively, combine deployment and initialization into a single batch transaction using NEAR's `DeployContract` + `FunctionCall` action batch, so no uninitialized window ever exists on-chain.

### Proof of Concept

1. Operator broadcasts: `near contract deploy mpc-contract.near use-file mpc_contract.wasm without-init-call ...`
2. Attacker observes the deployment transaction in the block explorer.
3. Attacker broadcasts before the operator's `init` call:
   ```bash
   near contract call-function as-transaction mpc-contract.near init \
     json-args '{
       "parameters": {
         "threshold": 1,
         "participants": {
           "next_id": 1,
           "participants": [[
             "attacker.near", 0,
             {"tls_public_key": "<attacker_ed25519_key>", "url": "https://attacker.example.com"}
           ]]
         }
       }
     }' \
     prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
     sign-as attacker.near network-config mainnet sign-with-keychain send
   ```
4. The contract is now in `Running` state with `attacker.near` as the sole participant holding a valid mock attestation.
5. The operator's subsequent `init` call fails: `"Contract is already initialized"` (NEAR `#[init]` guard).
6. Attacker calls `vote_add_domains` (threshold = 1, only their vote needed), drives DKG, and gains exclusive signing authority over the MPC network. [9](#0-8)

    pub(crate) fn with_mocked_participant_attestations(participants: &Participants) -> Self {
        let mut tee_state = Self::default();

        for (account_id, _, participant_info) in participants.participants() {
            let tls_public_key = participant_info.tls_public_key.clone();
            // TODO(#1087): replace account_public_key with a real account public
            // key passed in by the caller. `Participants` does not currently
            // carry the operator's account public key, so a mocked entry
            // cannot record the real one and we use the TLS key as a unique
            // per-participant placeholder. The mock keeps the
            // participant from being kicked out of an empty `TeeState` until
            // a real `submit_participant_info` call replaces it (keyed by
            // TLS), but any caller-facing check that compares
            // `signer_account_pk` against the stored key will fail until
            // then. #1087 tracks threading real attestations through
            // initialization so this sentinel can go away.
            let node_id = NodeId {
                account_id: account_id.clone(),
                tls_public_key: tls_public_key.clone(),
                // Use tls_public_key as account_public_key instead of hardcoded
                // Ed25519PublicKey::from([0u8; 32]) so that same account public
                // key isn't associated with different tls keys.
                // This is not a fix for above issue: #1087, which should be
                // addressed outside this PR.
                account_public_key: tls_public_key.clone(),
            };

            tee_state.stored_attestations.insert(
                tls_public_key,
                NodeAttestation {
                    node_id,
                    verified_attestation: VerifiedAttestation::Mock(
                        attestation::MockAttestation::Valid,
                    ),
                },
            );
        }

        tee_state
    }
