### Title
Storage Deposits Paid via `propose_update` Are Permanently Locked After `do_update` Clears All Entries — (`File: crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

The `propose_update` function collects a NEAR storage deposit from the proposer to cover on-chain storage for the update entry. When `do_update` is executed, it clears **all** pending entries and votes, freeing that storage — but never refunds any of the deposits. The NEAR tokens accumulate permanently in the contract with no sweep or withdrawal mechanism, directly mirroring the LamboVEthRouter "1-wei accumulation" pattern.

---

### Finding Description

`propose_update` (lib.rs) charges the caller a deposit equal to `storage_byte_cost × bytes_used`:

```rust
// lib.rs ~1308-1316
let attached = env::attached_deposit();
let required = ProposedUpdates::required_deposit(&update);
if attached < required {
    return Err(InvalidParameters::InsufficientDeposit { ... }.into());
}
```

Only the **excess** above `required` is refunded:

```rust
// lib.rs ~1327-1331
if let Some(diff) = attached.checked_sub(required)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(proposer).transfer(diff).detach();
}
```

The `required` portion stays in the contract. When `do_update` is later called, it removes the executed entry **and clears every other pending entry**:

```rust
// update.rs ~196-200
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;
    self.entries.clear();          // all competing proposals wiped
    self.vote_by_participant.clear();
    ...
}
```

No deposit is returned to any proposer — neither the one whose update was executed nor the proposers of the competing entries that were silently cleared. `bytes_used` is stored in `UpdateEntry` but is never consulted during cleanup:

```rust
// update.rs ~132-135
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,   // recorded but never used for refund
}
```

There is no `sweep`, `withdraw`, or `refund_deposit` function anywhere in the contract.

---

### Impact Explanation

Every governance upgrade cycle permanently locks NEAR tokens in the contract:

- A contract binary upgrade proposal requires a deposit proportional to the binary size. For a 500 KB WASM blob the deposit is on the order of `500,000 × storage_byte_cost` (~0.5–5 NEAR depending on the NEAR storage price at the time).
- When `do_update` fires, **all** competing proposals are cleared simultaneously, so every proposer in that round loses their deposit.
- Over the lifetime of the contract (multiple upgrades, config changes, competing proposals) the locked balance grows monotonically with no recovery path.

This matches the allowed Medium impact: *"Balance … manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

`propose_update` is a routine governance operation. Every contract upgrade or config change triggers it. The loss is automatic and unconditional — no special attacker action is required; normal protocol operation is sufficient to accumulate locked funds. The only prerequisite is being a participant, which is the normal operational role.

---

### Recommendation

1. **Refund on `do_update`**: Before clearing entries, iterate over them and transfer each entry's `bytes_used`-derived deposit back to the original proposer. The proposer's account ID must be stored in `UpdateEntry` for this purpose.
2. **Alternatively**, record a `proposer → deposit` map and refund on explicit cancellation or on `do_update` sweep.
3. **At minimum**, add a participant-only `cancel_update(id)` that removes a single entry and refunds its deposit, so funds are not permanently stranded.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary. Required deposit ≈ 5 NEAR. Excess refunded; 5 NEAR stays in contract.
2. Participant B calls `propose_update` with a competing config update. Required deposit ≈ 0.01 NEAR. Stays in contract.
3. Threshold votes accumulate on A's proposal; `vote_update` calls `do_update(&id_A, gas)`.
4. Inside `do_update`: `entries.clear()` wipes both A's and B's entries. Neither A nor B receives any refund.
5. 5.01 NEAR is now permanently locked. Repeat for every upgrade cycle.

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
