### Title
Malicious Harvester Unconditionally Redirects Farmer Block Reward via Unenforced `farmer_reward_address_override` - (File: chia/farmer/farmer_api.py)

---

### Summary

The CHIP-22 third-party harvester fee convention allows a harvester to set `farmer_reward_address_override` in `NewProofOfSpace` and `RespondSignatures` messages to redirect the farmer's block reward as a fee. However, the farmer enforces no hard rejection when the fee quality check fails or when `fee_info` is absent — it only emits log warnings. As a result, any connected harvester peer can unconditionally redirect 100% of the farmer's block reward (0.25 XCH per block) to an arbitrary attacker-controlled address, with no cryptographic or protocol-level enforcement preventing it.

---

### Finding Description

**Root cause — two unenforced check points:**

**1. `FarmerAPI.new_proof_of_space()` — logging only, no rejection**

When a harvester sends `NewProofOfSpace` with `farmer_reward_address_override != None`, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

Inside `notify_farmer_reward_taken_by_harvester_as_fee()`, the fee quality is computed and compared against the harvester-supplied `applied_fee_threshold`. If the threshold is violated (or `fee_info` is `None`), the function **only logs a warning** and returns normally — it does not raise, does not return a sentinel, and does not abort the block-making flow: [2](#0-1) 

After this call returns, the farmer unconditionally proceeds to send `RequestSignatures` to the harvester regardless of whether the fee quality check passed or failed.

**2. `FarmerAPI._process_respond_signatures()` — unconditional override acceptance**

When the harvester responds with `RespondSignatures`, the farmer again unconditionally accepts `farmer_reward_address_override` from the response and substitutes it for the configured `farmer_target`: [3](#0-2) 

This overridden address is then placed directly into `DeclareProofOfSpace.farmer_reward_address`, which is broadcast to the full node and determines where the 0.25 XCH farmer reward coin is created on-chain: [4](#0-3) 

There is no cross-check between the `farmer_reward_address_override` in `NewProofOfSpace` and the one in `RespondSignatures`, and no enforcement that the fee quality threshold was actually satisfied before the override is applied.

---

### Impact Explanation

**High — Unauthorized payout redirection of XCH farmer rewards.**

A malicious harvester can redirect the farmer's block reward (currently 0.25 XCH per block won) to any attacker-controlled puzzle hash on every block the farmer wins. The redirection is permanent and on-chain: once `DeclareProofOfSpace` is accepted by the full node with the attacker's `farmer_reward_address`, the reward coin is created at the attacker's address and cannot be recovered by the farmer. This matches the allowed High impact: "Bypass of ... authorization that enables ... payout redirection."

---

### Likelihood Explanation

The CHIP-22 third-party harvester role is an explicitly supported, non-privileged role in the Chia ecosystem. Any operator who connects a harvester to a farmer (including remote/third-party harvesters) can exploit this. No leaked keys, admin access, or cryptographic break is required — only a valid harvester connection. The attacker simply sets `farmer_reward_address_override` to their own address in every `NewProofOfSpace` message they send. The farmer will log a warning but proceed to sign and broadcast the block with the attacker's reward address.

---

### Recommendation

The farmer must enforce the fee quality convention as a hard gate, not merely a log warning. Specifically:

1. In `FarmerAPI.new_proof_of_space()`, after calling `notify_farmer_reward_taken_by_harvester_as_fee()`, check whether the fee quality threshold was actually satisfied. If `fee_info` is absent or `fee_quality > applied_fee_threshold`, **reject the override** by either ignoring `farmer_reward_address_override` (falling back to `farmer_target`) or dropping the proof entirely.

2. In `FarmerAPI._process_respond_signatures()`, before accepting `response.farmer_reward_address_override`, verify that the corresponding `NewProofOfSpace` also carried a valid, passing fee quality check. If not, ignore the override and use `self.farmer.farmer_target`.

3. Optionally, verify that `response.farmer_reward_address_override` matches the value declared in the original `NewProofOfSpace` to prevent a harvester from switching the target address between the two messages.

---

### Proof of Concept

**Attacker preconditions:** Operate or control a harvester that is connected to the victim farmer (a standard, non-privileged role).

**Steps:**

1. Attacker's harvester receives a `NewSignagePointHarvester` from the farmer.
2. Attacker's harvester finds a valid proof of space for the signage point.
3. Attacker's harvester sends `NewProofOfSpace` to the farmer with:
   - `farmer_reward_address_override = attacker_puzzle_hash` (any arbitrary address)
   - `fee_info = None` (or any `applied_fee_threshold` value, including one that fails the quality check)
4. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`, which logs a warning but **does not abort**.
5. Farmer sends `RequestSignatures` back to the attacker's harvester.
6. Attacker's harvester responds with `RespondSignatures` containing `farmer_reward_address_override = attacker_puzzle_hash`.
7. Farmer's `_process_respond_signatures()` unconditionally sets `farmer_reward_address = attacker_puzzle_hash` and returns `DeclareProofOfSpace` with that address.
8. Full node creates the farmer reward coin (0.25 XCH) at `attacker_puzzle_hash`.
9. Farmer receives no block reward; attacker receives 0.25 XCH.

This repeats on every block the farmer wins while the malicious harvester is connected. [7](#0-6) [8](#0-7) [9](#0-8)

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
