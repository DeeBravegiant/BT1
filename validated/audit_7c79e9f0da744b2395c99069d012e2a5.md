Audit Report

## Title
Missing Zero-Amount Guard After Division-Truncated rsETH Calculation in Token Deposit Path - (File: contracts/pools/RSETHPoolV3.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol)

## Summary
All five L2 pool variants compute `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` in the token-deposit overload of `viewSwapRsETHAmountAndFee`. When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, Solidity integer division silently truncates the result to 0. The calling `deposit` functions proceed unconditionally: the user's tokens are transferred into the pool, and 0 rsETH is minted or transferred back. The deposited tokens are permanently retained by the pool with no recourse for the user.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256,address)` at line 334, `RSETHPool` at line 346, `RSETHPoolNoWrapper` at line 311, `RSETHPoolV3ExternalBridge` at line 452, and `RSETHPoolV3WithNativeChainBridge` at line 370, the token-deposit overload computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

Both `tokenToETHRate` and `rsETHToETHrate` are 1e18-precision values. When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, the division truncates to 0. This occurs concretely when `amountAfterFee = 1` and `tokenToETHRate ≤ rsETHToETHrate` (e.g., any token whose oracle rate is at or below the current rsETH/ETH rate, such as a token at 0.95e18 while rsETH is at 1.05e18, or even wstETH/rETH at ~1e18 when rsETH rate exceeds the token rate).

The ETH-deposit overload correctly scales: `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`, which cannot produce 0 for any non-zero `amountAfterFee` since `1e18 ≥ rsETHToETHrate` is not guaranteed but the formula is structurally different. The token path has no equivalent protection.

The calling deposit functions in all five contracts follow the same pattern — confirmed in `RSETHPoolV3.deposit(address,uint256,string)` at lines 282–292, `RSETHPoolNoWrapper.deposit(address,uint256,string)` at lines 260–270, `RSETHPoolV3ExternalBridge.deposit` at lines 401–411, `RSETHPoolV3WithNativeChainBridge.deposit` at lines 318–328, and `RSETHPool.deposit` at lines 294–304:

```solidity
if (amount == 0) revert InvalidAmount();                          // only guards amount, not rsETHAmount
IERC20(token).safeTransferFrom(msg.sender, address(this), amount); // tokens taken first
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);                             // mints 0 silently
```

The only existing guard is `if (amount == 0) revert InvalidAmount()`, which does not protect against a non-zero `amount` producing a zero `rsETHAmount`. The L1 `LRTDepositPool._beforeDeposit` (lines 648–669) has an analogous formula but is protected by `minRSETHAmountExpected`; the L2 token deposit path has no equivalent.

## Impact Explanation
The contract fails to deliver its promised return: a user depositing a non-zero token amount receives 0 rsETH, and the deposited tokens are permanently retained by the pool. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value** (from the pool's perspective). The absolute monetary value lost per incident is at most a few wei for currently supported 18-decimal LSTs, but the code path is structurally broken for any token whose oracle rate is at or below the rsETH/ETH rate at the time of deposit.

## Likelihood Explanation
Any unprivileged external user can trigger this by calling `deposit(token, 1, referralId)` on any of the five affected contracts. No special setup, front-running, privileged access, or oracle manipulation is required. The entry path is fully public and permissionless. For currently supported LSTs (wstETH, stETH, rETH, sfrxETH) with rates near or above 1e18, truncation to 0 occurs only at `amountAfterFee = 1` (1 wei). For any newly added collateral token with an oracle rate below the current rsETH/ETH rate, the truncation threshold is higher. Likelihood is low in practice because no rational user deposits 1 wei intentionally, but the condition is unconditionally reachable.

## Recommendation
Add a zero-amount guard immediately after computing `rsETHAmount` in the token-deposit overload of `viewSwapRsETHAmountAndFee`, or in each calling `deposit` function, across all five affected contracts:

```solidity
if (rsETHAmount == 0) revert InvalidAmount();
```

This mirrors the protection already present in `LRTDepositPool._beforeDeposit` via `minRSETHAmountExpected`, and closes the gap between the ETH and token deposit paths.

## Proof of Concept
1. Deploy or interact with any of the five affected pool contracts on a supported L2.
2. Call `deposit(token, 1, "ref")` where `token` has an oracle rate ≤ current rsETH/ETH rate (e.g., any token at 0.95e18 while rsETH is at 1.05e18, or any 18-decimal LST at exactly 1e18 when rsETH rate is 1.05e18).
3. Inside `viewSwapRsETHAmountAndFee(1, token)`:
   - `fee = 1 * feeBps / 10_000 = 0`
   - `amountAfterFee = 1`
   - `tokenToETHRate = 0.95e18`
   - `rsETHToETHrate = 1.05e18`
   - `rsETHAmount = 1 * 0.95e18 / 1.05e18 = 0`
4. `IERC20(token).safeTransferFrom(msg.sender, address(this), 1)` executes — 1 wei of token is taken.
5. `wrsETH.mint(msg.sender, 0)` (or `rsETH.safeTransfer(msg.sender, 0)` in `RSETHPool`/`RSETHPoolNoWrapper`) executes silently — user receives nothing.
6. The 1 wei of token is permanently held by the pool.

Foundry fuzz test plan: fuzz `amount` over `[1, 1e6]` and `tokenToETHRate` over `[0.5e18, 1.2e18]` with `rsETHToETHrate` fixed at `1.05e18`; assert that whenever `amount * tokenToETHRate / rsETHToETHrate == 0`, the deposit reverts rather than succeeding.
