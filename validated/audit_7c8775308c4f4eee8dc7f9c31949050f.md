### Title
Unconstrained `payload_hash` in `respond_verify_foreign_tx` Enables Forged Foreign-Chain Attestation — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` accepts a node-supplied `payload_hash` and only checks that the provided signature is valid over that hash against the root public key. It never verifies that `payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload { request, values }))` for the stored `VerifyForeignTransactionRequest`. A single malicious attested participant can therefore resolve a pending foreign-tx yield with a valid threshold signature that was produced for a *different* transaction, delivering a forged attestation to the waiting caller.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

`response.payload_hash` is a free field supplied entirely by the calling node. The contract verifies only that the signature is valid for *that* hash; it never derives the expected hash from the stored `VerifyForeignTransactionRequest` and compares.

Contrast this with the regular `respond` path, where the payload is taken directly from the stored request object — the node cannot substitute a different payload:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [2](#0-1) 

The `VerifyForeignTransactionRequest` stored as the map key contains only `{ request, domain_id, payload_version }` — no pre-committed hash: [3](#0-2) 

The `VerifyForeignTransactionResponse` that is serialised and returned to the yield-resume caller carries both `payload_hash` and `signature` verbatim from the node: [4](#0-3) 

After the signature check passes, the full `response` (including the unchecked `payload_hash`) is written into the yield-resume data and returned to the original caller:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [5](#0-4) 

---

### Impact Explanation

A downstream NEAR contract (e.g., an Omnibridge inbound handler) receives a `VerifyForeignTransactionResponse` and trusts the `payload_hash` as the MPC network's attestation of what happened on the foreign chain. If the `payload_hash` was silently swapped for a different transaction's hash, the downstream contract is presented with a valid MPC signature over fabricated foreign-chain data. This enables:

- **Forged bridge attestations**: attesting that a deposit transaction occurred when it did not, or substituting a different amount/recipient.
- **Double-spend conditions**: replaying a valid attestation for tx Y to satisfy a pending request for tx X, allowing the same foreign-chain event to be claimed twice across two separate `verify_foreign_transaction` requests.

This matches the **High** allowed impact: *"forged foreign-chain verification … that causes invalid bridge execution or double-spend conditions."*

---

### Likelihood Explanation

The attack requires exactly one malicious **attested participant** (a Byzantine node strictly below the signing threshold). No threshold collusion is needed to *submit* the forged response — only to produce the original valid signature, which the network does legitimately for other requests. Once any valid threshold signature over any `ForeignTxSignPayload` exists on-chain (i.e., after the first successful `respond_verify_foreign_tx` call), a single rogue node can reuse it against any pending request whose `VerifyForeignTransactionRequest` key it knows. The pending request map is public on-chain state, so the attacker has full visibility.

---

### Recommendation

Before accepting the response, recompute the expected payload hash on-chain from the stored request and the node-supplied extracted values, and assert equality:

```rust
// Pseudocode
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This requires adding `extracted_values` to `VerifyForeignTransactionResponse` so the contract can independently reconstruct and verify the hash, mirroring how `respond` binds the signature to the user-committed payload.

---

### Proof of Concept

1. User A calls `verify_foreign_transaction` for Bitcoin tx X. The request `R_X = { request: Bitcoin(tx_id: X, ...), domain_id, payload_version }` is stored in `pending_verify_foreign_tx_requests`.

2. The MPC network legitimately processes a separate request for Bitcoin tx Y, producing threshold signature `σ_Y` over `H_Y = SHA-256(borsh(ForeignTxSignPayload { request: Bitcoin(tx_id: Y, ...), values: [...] }))`. Node N calls `respond_verify_foreign_tx(R_Y, { payload_hash: H_Y, signature: σ_Y })` — this succeeds normally.

3. Malicious node M (a single attested participant) now calls:
   ```
   respond_verify_foreign_tx(
       request  = R_X,          // matches the pending yield for tx X
       response = { payload_hash: H_Y, signature: σ_Y }  // valid sig, wrong tx
   )
   ```

4. The contract looks up `R_X` in `pending_verify_foreign_tx_requests` ✓, verifies `verify_ecdsa_signature(σ_Y, H_Y, root_pk)` → valid ✓, and resolves the yield with `{ payload_hash: H_Y, signature: σ_Y }`.

5. User A's NEAR contract receives a `VerifyForeignTransactionResponse` carrying `H_Y` and `σ_Y` — a valid MPC signature, but attesting to tx Y's data, not tx X's. The downstream bridge contract, trusting the MPC attestation, processes a fraudulent inbound transfer. [6](#0-5) [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
