Audit Report

## Title
Unvalidated Chainlink Price Data in `ChainlinkPriceOracle.getAssetPrice()` Enables Stale Price to Corrupt rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `priceFeed.latestRoundData()` and discards all validation fields, using only the raw `price` value without checking staleness, round completeness, or sign. This unvalidated price flows directly into rsETH minting calculations in `LRTDepositPool.getRsETHAmountToMint()`. When a Chainlink feed goes stale and returns an inflated last-known price while `rsETHPrice` was set at an earlier accurate value, depositors receive excess rsETH, stealing unclaimed yield from existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` (lines 49–55) silently discards all validation fields from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No checks are performed for `answeredInRound < roundId` (stale round), `updatedAt == 0` (incomplete round), or `price <= 0`. The same codebase already implements all three validations in `ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 30–32):

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated price propagates through two paths:

**Path 1 — Direct minting:** `LRTDepositPool.getRsETHAmountToMint()` (line 520) uses the asset price directly in the mint ratio:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Path 2 — Global price update (public):** `LRTOracle.updateRSETHPrice()` (line 87) is callable by anyone and internally calls `_getTotalEthInProtocol()` (lines 331–349), which iterates all supported LSTs and calls `getAssetPrice(asset)` for each, accumulating `totalETHInProtocol` used to compute and persist `rsETHPrice`.

The critical exploit path exploits a timing mismatch: `rsETHPrice` is set at time T1 with accurate prices; the LST/ETH feed goes stale at T2 returning an inflated last-known price; a depositor deposits at T3 using the stale (inflated) asset price against the accurate (lower) `rsETHPrice`, receiving more rsETH than their deposit warrants.

## Impact Explanation
**Theft of unclaimed yield (High).** When a Chainlink LST/ETH feed goes stale and returns an inflated price, `getRsETHAmountToMint` computes a numerator inflated by the stale price while the denominator (`rsETHPrice`) reflects a prior accurate value. The depositor receives excess rsETH tokens, diluting the proportional claim of all existing rsETH holders on the underlying ETH — constituting theft of unclaimed yield. The zero-price scenario (permanent loss of deposited funds) requires the user to pass `minRSETHAmountExpected = 0`, which constitutes a victim mistake and is excluded. The negative-price scenario would overflow and revert under Solidity 0.8.x checked arithmetic.

## Likelihood Explanation
Chainlink feed staleness is a documented and recurring risk during L2 sequencer downtime, extreme network congestion, or node operator issues. No privileged access is required: `updateRSETHPrice()` is public and callable by any account. The depositor exploiting the stale price needs only to call `depositAsset()` during the staleness window — a standard user action. Likelihood is **Low** given Chainlink's general reliability, but the impact is severe enough to warrant High severity.

## Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Optionally add a heartbeat check: `if (block.timestamp - updatedAt > HEARTBEAT_INTERVAL) revert StalePrice();`

## Proof of Concept
1. Chainlink stETH/ETH feed goes stale (last updated 4+ hours ago; heartbeat is 1 hour). Feed continues returning the last known price (e.g., 1.001e18) without reverting.
2. At T1, `LRTOracle.updateRSETHPrice()` is called with accurate prices; `rsETHPrice` is set to an accurate value (e.g., 1.05e18).
3. At T2, the stETH/ETH feed goes stale, but the true stETH/ETH rate has dropped to 0.999e18. The feed still returns 1.001e18.
4. At T3, a depositor calls `LRTDepositPool.depositAsset(stETH, 1e18, 0, "")`.
5. `getRsETHAmountToMint(stETH, 1e18)` computes: `(1e18 * 1.001e18) / 1.05e18 ≈ 0.9533e18 rsETH` instead of the correct `(1e18 * 0.999e18) / 1.05e18 ≈ 0.9514e18 rsETH`.
6. The depositor receives ~0.002e18 excess rsETH, diluting all existing holders' proportional claim on the underlying ETH.

**Foundry fork test plan:**
- Fork mainnet/L2 at a block where a Chainlink LST/ETH feed is near its heartbeat expiry.
- Mock `latestRoundData()` to return a stale `updatedAt` with an inflated price.
- Call `updateRSETHPrice()` at T1 with accurate prices to set `rsETHPrice`.
- Call `depositAsset()` at T3 with the stale feed active.
- Assert that `rsethAmountToMint` exceeds the amount computed using the true current price, confirming excess minting.
