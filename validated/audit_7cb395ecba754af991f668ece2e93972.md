Audit Report

## Title
Stale `rsETHPrice` After `_updateRsETHPrice()` Revert on Daily Fee Mint Limit Allows New Depositors to Mint rsETH at Below-Market Price, Diluting Existing Holders — (`contracts/LRTOracle.sol`)

## Summary
In `LRTOracle._updateRsETHPrice()`, the assignment `rsETHPrice = newRsETHPrice` (line 313) is placed after the protocol-fee mint block. If either `_checkAndUpdateDailyFeeMintLimit` (LRTOracle, line 206) or `RSETH.checkDailyMintLimit` (RSETH, line 51) causes a revert, the price update is permanently skipped for that call while the ETH that caused the TVL increase is already settled in the deposit pool. Because both `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` are permissionless, an attacker can deliberately trigger this revert, leaving `rsETHPrice` stale, and then deposit at the artificially low price to receive excess rsETH, directly diluting all existing holders.

## Finding Description
**Step 1 — ETH is settled permissionlessly.** `FeeReceiver.sendFunds()` has no access control. It calls `LRTDepositPool.receiveFromRewardReceiver()`, a plain `payable` function, permanently increasing `address(this).balance` and thus `_getTotalEthInProtocol()`.

**Step 2 — `updateRSETHPrice()` is permissionless.** `LRTOracle.updateRSETHPrice()` is `public whenNotPaused` with no role restriction.

**Step 3 — `rsETHPrice` is written only after the fee mint block.** Inside `_updateRsETHPrice()` (LRTOracle.sol lines 299–313):
1. `newRsETHPrice` is computed (line 250).
2. `_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` is called (line 303) — **reverts** with `DailyFeeMintLimitExceeded` if the LRTOracle fee cap is breached.
3. `IRSETH(...).mint(treasury, rsethAmountToMintAsProtocolFee)` is called (line 306) — **reverts** with `DailyMintLimitExceeded` via RSETH's `checkDailyMintLimit` modifier if the RSETH daily cap is breached.
4. `rsETHPrice = newRsETHPrice` (line 313) — **only reached if neither revert fires**.

**Step 4 — Deposits use the stale price.** `LRTDepositPool.getRsETHAmountToMint()` computes `rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` (line 520). There is no oracle-freshness check in `_beforeDeposit`. A stale (lower) `rsETHPrice` yields a higher rsETH amount per ETH deposited.

**Why existing guards are insufficient:**
- `pricePercentageLimit` only guards against price *increases* above a threshold; a stale price that is *lower* than the true price is not caught.
- The `minRSETHAmountExpected` slippage parameter in `depositETH` protects the depositor, not existing holders.
- `updateRSETHPriceAsManager()` is the only privileged path that bypasses the public revert, but it requires the `MANAGER` role and is not called automatically.

## Impact Explanation
After the revert, `rsETHPrice` is frozen at the pre-reward value `P`, while the true price is `P' > P` (TVL increased but supply did not). Any depositor who calls `depositETH` during the window receives `Y / P` rsETH instead of the correct `Y / P'`. The excess `Y × (P' − P) / (P × P')` rsETH per deposited ETH is backed by no additional ETH, permanently diluting every existing holder's share of the protocol. This constitutes **direct theft of user funds (Critical)**: existing rsETH holders suffer a quantifiable, irreversible reduction in the ETH value of their holdings.

## Likelihood Explanation
Both trigger conditions are realistic and require no special privileges:
- **LRTOracle fee cap** (`maxFeeMintAmountPerDay`): A conservatively set cap combined with a large MEV reward block produces a fee rsETH amount exceeding the cap in a single call.
- **RSETH daily mint cap** (`maxMintAmountPerDay`): Heavy user deposit activity during the day can exhaust most of the RSETH daily mint budget, leaving insufficient headroom for the fee mint.

Neither `sendFunds()` nor `updateRSETHPrice()` requires any role. An attacker needs only gas to trigger the revert and then deposit at the stale price. The attack window lasts up to 24 hours (until the daily limit resets). The attack is repeatable every day the cap is hit.

## Recommendation
Move `rsETHPrice = newRsETHPrice` **before** the fee mint block so the price is always committed regardless of fee mint success. Alternatively, wrap the fee mint in a `try/catch` so that a fee-mint failure skips the mint but still commits the price update. A secondary defense is to add a `lastPriceUpdateTimestamp` and reject deposits (or apply a penalty rate) when the price is stale beyond a configurable threshold.
