Audit Report

## Title
Rewards Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`, while `rewardPerToken()` silently skips accrual during zero-stake periods. Any `rewardRate`-denominated tokens emitted during a zero-stake gap are permanently unrecoverable, as no sweep or rescue function exists in the contract.

## Finding Description
The root cause is a mismatch between `rewardPerToken()` and `updateReward`:

`rewardPerToken()` correctly freezes the stored value when supply is zero: [1](#0-0) 

But `updateReward` unconditionally advances `updatedAt` regardless of supply: [2](#0-1) 

`initiateWithdrawal` reduces `totalKernelStaked` immediately in the function body, after the modifier has already checkpointed the departing staker correctly: [3](#0-2) 

Exploit trace:
- **T=10**: Alice (sole staker) calls `initiateWithdrawal(all)`. The `updateReward` modifier fires while `totalKernelStaked > 0`, correctly checkpointing Alice. Then `totalKernelStaked` drops to 0. `updatedAt` is set to T=10.
- **T=30**: Bob calls `stake(X)`. `updateReward` fires while `totalKernelStaked == 0`: `rewardPerToken()` returns the old stored value (gap T=10→T=30 skipped), but `updatedAt` is advanced to T=30. The 20-second gap's worth of `rewardRate × 20` tokens are now permanently orphaned.
- **T=40**: Bob calls `getReward()`. He earns only T=30→T=40 rewards. The T=10→T=30 rewards remain locked in the contract forever.

The `notifyRewardAmount` guard only prevents starting a new period with zero stakers; it does not prevent all stakers from exiting during an already-active period: [4](#0-3) 

The contract's own NatSpec acknowledges this limitation and relies entirely on an off-chain operational assumption: [5](#0-4) 

There is no `rescueTokens`, `sweep`, or any other recovery function in the contract.

## Impact Explanation
Reward tokens transferred into the contract via `notifyRewardAmount` and emitted during any zero-stake gap are permanently frozen with no recovery path. This maps directly to **Medium — Permanent freezing of unclaimed yield**. The lost amount scales with `rewardRate × gap_duration`; with `MAX_WITHDRAWAL_DELAY = 30 days`, the gap can be up to 30 days of emissions. [6](#0-5) 

## Likelihood Explanation
The scenario is reachable by any single unprivileged staker who is the sole depositor — a realistic condition early in the protocol or after a mass exit. No code-level guard prevents `totalKernelStaked` from reaching zero during an active reward period. The mandatory `withdrawalDelay` (up to 30 days) structurally guarantees a window during which the gap can persist. No attacker coordination is required; a single honest user performing a normal withdrawal is sufficient to trigger the loss.

## Recommendation
Only advance `updatedAt` when `totalKernelStaked > 0` in the `updateReward` modifier:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This preserves the gap's time window so that when the next staker enters, `rewardPerToken()` replays the full gap using `lastTimeRewardApplicable() - updatedAt` (which still spans back to the last non-zero-supply timestamp), distributing the frozen rewards correctly.

## Proof of Concept

```solidity
// Setup: Alice is sole staker, admin starts reward period
kernelDepositPool.setRewardsDuration(30 days);
kernelDepositPool.notifyRewardAmount(30_000e18); // rewardRate ≈ 1000e18/day

// Alice initiates full withdrawal → totalKernelStaked = 0
vm.prank(alice);
kernelDepositPool.initiateWithdrawal(aliceStake);

// 10 days pass with zero stakers; 10_000e18 tokens emitted but unaccounted
vm.warp(block.timestamp + 10 days);

// Bob stakes; updateReward fires with totalKernelStaked==0, advances updatedAt past the gap
vm.prank(bob);
kernelDepositPool.stake(1e18);

// Another 10 days pass
vm.warp(block.timestamp + 10 days);

// Bob claims — earns only ~10_000e18 (T+10 to T+20), not ~20_000e18
vm.prank(bob);
kernelDepositPool.getReward();

// Assert: ~10_000e18 permanently stuck in contract
assertGt(rewardsToken.balanceOf(address(kernelDepositPool)), 9_000e18);
```
