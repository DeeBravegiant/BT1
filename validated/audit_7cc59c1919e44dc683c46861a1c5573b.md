### Title
Double-Deposit Inflation via Sub-Transaction Pool Re-Registration — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubPool.hs`, `SubLedger.hs`, `Ledger.hs`, `Utxo.hs`)

---

### Summary

In the Dijkstra era, the `LEDGER` rule passes the **original pre-SUBLEDGERS** `certState` to both `SUBUTXO` (for the sub-transaction) and `UTXOW`/`UTXO` (for the top-level transaction). When an attacker registers the same pool P in both a sub-transaction and the top-level transaction body, each UTxO accounting step independently counts a `ppPoolDeposit` increment (because pool P is absent from the original `certState` in both cases), while `ENTITIES` for the top-level transaction sees pool P as already registered (from the sub-ledger result) and records only a single `spsDeposit`. The result is `utxosDeposited` permanently exceeding the sum of all `spsDeposit` values by exactly `ppPoolDeposit`, violating the `potEqualsObligation` invariant.

---

### Finding Description

**Execution flow in `dijkstraLedgerTransition`** (`Ledger.hs`):

**Step 1 — SUBLEDGERS** processes the sub-transaction via `dijkstraSubLedgersTransition` (`SubLedger.hs`):

```
SUBENTITIES(certState)  →  certStateAfterSubEntities  (pool P added to psStakePools)
SUBUTXOW(SubUtxoEnv ... certState ...)  →  uses original certState
```

The critical line is: [1](#0-0) 

`SUBUTXOW` receives `certState` (the **original**, pre-SUBENTITIES state). Inside `dijkstraSubUtxoTransition`: [2](#0-1) 

`updateUTxOStateNoFees` calls `certsTotalDepositsTxBody pp certState txBody`. Since pool P is **not** in the original `certState`'s `psStakePools`, `shelleyTotalDepositsTxCerts` counts it as a new registration: [3](#0-2) 

→ `utxosDeposited += ppPoolDeposit` (sub-tx deposit #1)

After SUBLEDGERS: `certStateAfterSubLedgers.psStakePools` contains pool P with `spsDeposit = ppPoolDeposit`.

**Step 2 — ENTITIES** processes the top-level transaction's `RegPool P` cert using `certStateAfterSubLedgers`: [4](#0-3) 

Pool P is **already registered** → treated as a re-registration (parameter update). No new `spsDeposit` entry is created. `certStateAfterENTITIES.psStakePools` still has exactly one entry for P with `spsDeposit = ppPoolDeposit`.

**Step 3 — UTXOW/UTXO** for the top-level transaction uses `lsCertState ledgerState` (the **original** pre-SUBLEDGERS certState): [5](#0-4) 

Inside `dijkstraUtxoTransition`, `updateUTxOStateByTxValidity` is called with this original `certState`: [6](#0-5) 

`updateUTxOState` → `certsTotalDepositsTxBody pp certState txBody`. Pool P is **not** in the original `certState` → counted as a new registration again.

→ `utxosDeposited += ppPoolDeposit` (top-tx deposit #2)

**Final state:**
- `utxosDeposited` increased by `2 × ppPoolDeposit`
- `psStakePools` has pool P with `spsDeposit = ppPoolDeposit` (one entry)
- `potEqualsObligation` invariant violated by `ppPoolDeposit`

The deposit accounting function confirms the check: [7](#0-6) [8](#0-7) 

The `LEDGER` rule registers this assertion: [9](#0-8)

### Impact Explanation

`utxosDeposited` is permanently inflated by `ppPoolDeposit` per attack execution. The surplus ADA is locked in the deposit pot with no corresponding `spsDeposit` obligation record. It can never be refunded (no deregistration certificate will match it) and cannot be drained without a hard fork that corrects the accounting. The `potEqualsObligation` invariant is permanently broken for the affected ledger state.

---

### Likelihood Explanation

Any unprivileged user can submit a valid top-level transaction containing a sub-transaction. Both the sub-transaction and the top-level transaction body can independently include `RegPool P` for the same pool key. No governance majority, operator access, or leaked key is required. The attack is deterministic, cheap (costs one `ppPoolDeposit` from the top-tx value conservation check; the sub-tx deposit increment has no corresponding value conservation check in `SUBUTXO`), and repeatable across different pool IDs.

---

### Recommendation

In `dijkstraSubLedgersTransition` (`SubLedger.hs`), pass `certStateAfterSubEntities` (not the original `certState`) to `SUBUTXOW`:

```haskell
-- Current (buggy):
trans @(EraRule "SUBUTXOW" era) $
  TRC (SubUtxoEnv slot pp certState originalUtxo topIsValid, ...)

-- Fixed:
trans @(EraRule "SUBUTXOW" era) $
  TRC (SubUtxoEnv slot pp certStateAfterSubEntities originalUtxo topIsValid, ...)
```

Similarly, in `dijkstraLedgerTransition` (`Ledger.hs`), pass `certStateAfterSubLedgers` (not `lsCertState ledgerState`) to `UTXOW`:

```haskell
-- Current (buggy):
TRC (DijkstraUtxoEnv slot pp (lsCertState ledgerState) originalUtxo, ...)

-- Fixed:
TRC (DijkstraUtxoEnv slot pp certStateAfterSubLedgers originalUtxo, ...)
```

Both fixes ensure that deposit calculations use the certState that reflects all prior certificate processing, eliminating the double-counting.

### Proof of Concept

```haskell

it "double-deposit inflation via sub-tx + top-tx RegPool same pool" $ do
  poolKH <- freshKeyHash  -- pool operator key
  let poolParams = mkPoolParams poolKH ...
  
  -- Sub-transaction registers pool P
  let subTx = mkBasicTx mkBasicTxBody
        & bodyTxL . certsTxBodyL .~ [RegPoolTxCert poolParams]
  
  -- Top-level transaction also registers pool P (re-registration)
  let topTx = mkBasicTx mkBasicTxBody
        & bodyTxL . certsTxBodyL .~ [RegPoolTxCert poolParams]
        & bodyTxL . subTransactionsTxBodyL .~ OMap.singleton (txId subTx) subTx
  
  submitTx topTx  -- should succeed
  
  -- Check invariant violation
  ledgerState <- getLedgerState
  let utxosDeposited = ledgerState ^. lsUTxOStateL . utxosDepositedL
      obligations    = obligationCertState (ledgerState ^. lsCertStateL)
  utxosDeposited @/= sumObligation obligations  -- FAILS: inflated by ppPoolDeposit
```
