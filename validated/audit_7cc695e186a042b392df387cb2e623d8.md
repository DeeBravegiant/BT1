### Title
Gateway `validate_nonce` Uses Committed-Block Nonce Ignoring Staged Mempool Transactions, Causing Valid Declare Transactions to Be Incorrectly Rejected — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_nonce` reads the account nonce exclusively from the last committed block state (`gateway_fixed_block_state_reader.get_nonce`). It is blind to transactions that have already been staged (handed to the batcher but not yet committed). When `reject_future_declare_txs = true` (the production default), the check requires `incoming_tx_nonce == account_nonce` for Declare transactions. Because `account_nonce` is the committed-block nonce and not the staged nonce, any Declare whose nonce equals the staged nonce (the only nonce that would actually succeed in the mempool) is rejected by the gateway before the mempool is ever consulted.

---

### Finding Description

**Admission path:**

1. `extract_state_nonce_and_run_validations` fetches `account_nonce` from `get_nonce_from_state`, which delegates to `GatewayFixedBlockSyncStateClient::get_nonce` — a read against the last committed block, not the mempool's in-flight state. [1](#0-0) 

2. `run_pre_validation_checks` calls `validate_state_preconditions` (which calls `validate_nonce`) **before** `validate_by_mempool`. If `validate_nonce` returns an error, the function short-circuits and the mempool is never consulted. [2](#0-1) 

3. `validate_nonce` for Declare transactions (with `reject_future_declare_txs = true`, the production default) requires strict equality between the incoming nonce and the committed-block nonce: [3](#0-2) 

4. The mempool, by contrast, tracks a `staged` map that advances the effective account nonce for every transaction handed to the batcher. `MempoolState::resolve_nonce` returns the staged nonce when present, making the mempool's view of the account nonce higher than the committed-block nonce during block construction. [4](#0-3) 

**The gap:** While the batcher is building a block, accounts with staged transactions have an effective nonce of `committed_nonce + k` (where `k` is the number of staged transactions). A user who correctly computes their next nonce as `committed_nonce + k` and submits a Declare will be rejected by the gateway with `"Invalid transaction nonce. Expected: nonce = N, got: N+k"`. The only nonce the gateway accepts (`N`) is the one the mempool will immediately reject with `NonceTooOld` (because the staged nonce is `N+k`). The user is in a catch-22 and cannot submit a Declare while any of their transactions are staged.

`reject_future_declare_txs` is `true` by default in production: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Impact: High — Mempool/gateway admission rejects valid transactions before sequencing.**

Any account that has one or more transactions currently staged (i.e., handed to the batcher for the block in progress) cannot submit a Declare transaction until the block is committed and the staged nonces are cleared. In a continuously-producing sequencer, staged transactions are the norm, not the exception. This means Declare transactions (contract class declarations) are systematically rejected for active accounts during block construction, degrading liveness for contract deployment workflows.

---

### Likelihood Explanation

The condition is triggered whenever:
- `reject_future_declare_txs = true` (production default), **and**
- An account has at least one transaction currently staged in the mempool (i.e., `get_txs` has been called and the transaction has not yet been committed).

During normal sequencer operation, the batcher continuously calls `get_txs`, staging transactions. Any active account submitting a Declare during this window will be rejected. The window lasts for the entire block-building period.

---

### Recommendation

`validate_nonce` should consult the mempool's staged nonce (via `mempool_client`) rather than relying solely on the committed-block nonce for the Declare equality check. Concretely, the gateway should call `mempool_client.validate_tx` first (which uses `resolve_nonce` including staged state), and only apply the `reject_future_declare_txs` equality check against the resolved nonce, not the raw committed-block nonce. Alternatively, the gateway can query the mempool for the account's current effective nonce before running `validate_nonce`.

---

### Proof of Concept

```
1. Account A has committed nonce = 5.
2. Batcher calls get_txs(); account A's invoke tx (nonce=5) is staged.
   → MempoolState::staged[A] = 6.
3. User submits Declare(nonce=6) for account A.
4. Gateway calls get_nonce_from_state(A) → returns 5 (committed block).
5. validate_nonce: incoming_tx_nonce(6) != account_nonce(5) → REJECTED:
   "Invalid transaction nonce. Expected: nonce = 5, got: 6."
6. User retries with Declare(nonce=5).
7. Gateway: 5 == 5 → passes validate_nonce.
8. validate_by_mempool: resolve_nonce(A, 5) → staged nonce = 6.
   tx_nonce(5) < account_nonce(6) → REJECTED: NonceTooOld.
9. User cannot submit any Declare until the block commits and staged state clears.
```

The root cause is `get_nonce_from_state` returning only the committed-block nonce, mirroring the external report's `_getCurrentHedgedNetDeltaWithSpot()` ignoring pending position requests. [7](#0-6) [8](#0-7)

    fn resolve_nonce(&self, address: ContractAddress, incoming_account_nonce: Nonce) -> Nonce {

            .get(&address)
            .or_else(|| self.committed.get(&address))
            .copied()
