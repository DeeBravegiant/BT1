Audit Report

## Title
Zero-Amount Fee Transfer in `claim()` Freezes User Rewards When `feeInBPS == 0` - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` at line 144 even when `fee == 0`. Since `feeInBPS = 0` is a valid and reachable state, any token that reverts on zero-value transfers will cause every `claim()` call to revert, permanently freezing all unclaimed allocations in the contract.

## Finding Description
In `MerkleDistributor.sol`, the fee transfer at line 144 is unconditional:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;   // line 138 — evaluates to 0 when feeInBPS == 0
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);     // line 141

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);     // line 144 — always called, even when fee == 0
``` [1](#0-0) 

`setFeeInBPS()` only rejects values exceeding `MAX_FEE_IN_BPS`; zero is explicitly permitted:

```solidity
function setFeeInBPS(uint256 _feeInBPS) external onlyOwner {
    if (_feeInBPS > MAX_FEE_IN_BPS) {
        revert InvalidFeeInBPS();
    }
    feeInBPS = _feeInBPS;
``` [2](#0-1) 

The same applies to `initialize()`, which also only checks `_feeInBPS > MAX_FEE_IN_BPS`. [3](#0-2) 

The sibling contract `KernelTop100MerkleDistributor` correctly guards this transfer:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
``` [4](#0-3) 

`MerkleDistributor` lacks this guard entirely. Additionally, `setToken()` allows the owner to configure any ERC-20 token, including tokens from the well-documented class that revert on zero-value transfers. [5](#0-4) 

## Impact Explanation
When `feeInBPS == 0` and the configured token reverts on zero-value transfers, every call to `claim()` reverts. Since `claim()` is the sole withdrawal path in `MerkleDistributor`, all unclaimed token allocations are permanently frozen in the contract. This matches **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
- `feeInBPS = 0` is the default if initialized with `_feeInBPS = 0`, and the owner can set it to zero at any time via `setFeeInBPS(0)` — no attacker action required.
- The contract is token-agnostic; `setToken()` allows configuration with any ERC-20, including tokens that revert on zero-value transfers (a well-documented token class).
- No privileged attacker is needed; the freeze arises from normal, permitted owner configuration combined with a public `claim()` call.
- The condition is repeatable and persistent: once triggered, no user can claim until the owner changes `feeInBPS` to a non-zero value or swaps the token.

## Recommendation
Guard the fee transfer with a zero-amount check, consistent with `KernelTop100MerkleDistributor`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy `MerkleDistributor` with a token that reverts on zero-value transfers and `feeInBPS = 0` (or call `setFeeInBPS(0)` post-deployment).
2. Set a valid Merkle root containing an allocation for `alice` via `setMerkleRoot()`.
3. `alice` calls `claim(index, alice, cumulativeAmount, proof)`.
4. `fee = (claimableAmount * 0) / 10_000 = 0`.
5. `IERC20(token).safeTransfer(alice, claimableAmount)` succeeds.
6. `IERC20(token).safeTransfer(protocolTreasury, 0)` **reverts** — the entire transaction rolls back, including the state update at lines 134–135.
7. `alice`'s allocation remains locked indefinitely; no alternative claim path exists in the contract.
