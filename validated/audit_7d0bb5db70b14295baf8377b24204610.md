Audit Report

## Title
Missing Slippage Protection When Removing Liquidity - (File: `metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`MetricOmmPool.removeLiquidity` accepts no `minAmount0` or `minAmount1` parameters, and the returned token amounts are computed entirely from live bin state at execution time. Because swaps can move the price cursor across bins and flip their token composition before a pending withdrawal executes, a liquidity provider can receive zero of their deposited token with no on-chain mechanism to revert. No periphery contract adds this protection for withdrawals.

## Finding Description
`MetricOmmPool.removeLiquidity` delegates directly to `LiquidityLib.removeLiquidity`:

```solidity
function removeLiquidity(address owner, uint80 salt, LiquidityDelta calldata deltas, bytes calldata extensionData)
    external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
    returns (uint256 amount0Removed, uint256 amount1Removed)
``` [1](#0-0) 

Inside `LiquidityLib.removeLiquidity`, each bin's token amounts are computed as a proportional share of the bin's current live scaled balances:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [2](#0-1) 

After computing amounts, tokens are transferred unconditionally with no post-condition check:

```solidity
if (amount0Removed > 0) { IERC20(ctx.token0).safeTransfer(owner, amount0Removed); }
if (amount1Removed > 0) { IERC20(ctx.token1).safeTransfer(owner, amount1Removed); }
``` [3](#0-2) 

In the Metric OMM bin model, a swap that moves the price cursor through a bin converts its entire token0 balance to token1 (or vice versa). If such a swap executes between a user's transaction submission and execution, `binState.token0BalanceScaled` becomes 0 for that bin, and the user receives only token1 — silently, with no revert.

`MetricOmmPoolLiquidityAdder` provides slippage protection only for deposits via `maxAmountToken0`/`maxAmountToken1` caps enforced in the callback, but exposes no `removeLiquidity` function at all. [4](#0-3) 

`IMetricOmmSimpleRouter` handles only swaps; it has no liquidity removal path. [5](#0-4) 

There is no `minAmount` or slippage guard anywhere in the production call stack for `removeLiquidity`.

## Impact Explanation
A liquidity provider who deposited token0 into a bin above the current price cursor receives exclusively token1 (zero token0) if a swap moves the cursor past that bin before their withdrawal executes. The user receives a token they did not want, must execute an additional swap to convert back (incurring fees and further slippage), and has no on-chain mechanism to revert the transaction if the output falls below their acceptable threshold. This constitutes a direct loss of the expected asset composition and meets the Sherlock Medium/High threshold for missing slippage protection on LP withdrawal.

## Likelihood Explanation
The trigger is any permissionless swap that moves the price cursor across a bin in which the victim holds shares — a routine operation in an active pool. No privileged access is required. The victim's `removeLiquidity` transaction is visible in the mempool, making targeted front-running straightforward and repeatable by any unprivileged actor.

## Recommendation
Add `minAmount0` and `minAmount1` parameters to `removeLiquidity` in both `MetricOmmPool` and `LiquidityLib`, and revert if computed amounts fall below the caller's minimums:

```solidity
if (amount0Removed < minAmount0 || amount1Removed < minAmount1) revert SlippageExceeded();
```

Additionally, expose a `removeLiquidity` wrapper in `MetricOmmPoolLiquidityAdder` (or a dedicated periphery contract) that enforces these bounds, mirroring the existing `maxAmountToken0`/`maxAmountToken1` pattern used for deposits.

## Proof of Concept
1. Pool has bins at indices -1, 0, 1. Current price cursor is at bin 0.
2. Alice calls `addLiquidity` for bin 1 (above current price), depositing token0.
3. Alice submits `removeLiquidity` for bin 1, expecting to receive token0.
4. Bob front-runs with a large `swap(zeroForOne=false)` that moves the cursor from bin 0 to bin 1 and beyond, converting bin 1's `token0BalanceScaled` entirely to `token1BalanceScaled`.
5. Alice's `removeLiquidity` executes: `binState.token0BalanceScaled == 0`, so `amount0Scaled = 0` at line 205 of `LiquidityLib.sol`. Alice receives only token1 and zero token0.
6. The transfer block at lines 242–247 skips the token0 transfer. The transaction succeeds silently with no revert, as there is no minimum-output check anywhere in the call path. [2](#0-1) [3](#0-2)
