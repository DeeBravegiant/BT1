### Title
`new_peak` permanently triggers `ValueError` after PlotNFT exit confirms at exactly `finish_exiting_height`, closing the full-node peer on every block — (`chia/wallet/plotnft_wallet/plotnft_wallet.py`)

---

### Summary

An off-by-one in `coin_added` causes `finish_exiting_height` to never be cleared when the exit-from-waiting-room transaction is confirmed at exactly the stored `finish_height`. Every subsequent call to `new_peak` then calls `_finish_leaving_pool` on a non-exiting PlotNFT, which raises `ValueError("Cannot exit waiting room while not in it")`. The exception propagates uncaught through `WalletStateManager.new_peak` and `WalletNode.new_peak_wallet`, landing in `_process_new_subscriptions`, which closes the full-node peer connection (`peer.close(9999)`) on every new block. The wallet enters a permanent reconnect-fail-disconnect loop.

---

### Finding Description

**Step 1 — `finish_exiting_height` is set**

When `coin_added` receives an exiting PlotNFT, it stores `finish_height = waiting_room_height + heightlock`: [1](#0-0) 

**Step 2 — Off-by-one: the height is never cleared when exit confirms at exactly `finish_height`**

When `coin_added` later receives the post-exit (non-exiting) PlotNFT, it clears the stored height only if `finish_height < height` (strictly less than): [2](#0-1) 

The exit transaction carries `AssertHeightRelative(heightlock)`, so the earliest block it can be confirmed in is exactly `W + heightlock = finish_height`. When that happens, `finish_height < finish_height` is `False`, so `clear_exiting_info` is **never called**.

**Step 3 — `rollback_to_block` does not touch the height table**

`PlotNFTStore.rollback_to_block` only deletes rows from `plotnft2s` and `pool_reward2s`. The `finish_exiting_height` and `finish_exiting_info` tables are untouched: [3](#0-2) 

**Step 4 — `new_peak` fires `_finish_leaving_pool` on a non-exiting PlotNFT**

Two blocks after `finish_height`, the guard `finish_height <= height - 2` becomes true. With no unconfirmed transactions (the exit tx is already confirmed), `_finish_leaving_pool` is called: [4](#0-3) 

**Step 5 — `exit_waiting_room` raises `ValueError`**

`_finish_leaving_pool` fetches the current PlotNFT (now `exiting=False`) and calls `exit_waiting_room`, which raises unconditionally when `exiting` is `False`: [5](#0-4) 

**Step 6 — Exception propagates, peer is closed**

`WalletStateManager.new_peak` has no try/except: [6](#0-5) 

The exception reaches `_process_new_subscriptions`, which closes the full-node peer connection: [7](#0-6) 

Because `finish_exiting_height` is never cleared, this repeats on every new peak indefinitely.

---

### Impact Explanation

After a normal pool exit where the exit transaction confirms at exactly `finish_height` (the earliest valid block, which is the common case when the tx waits in the mempool until the height lock expires), the wallet enters a permanent loop:

1. Connect to full node
2. Receive `new_peak`
3. `ValueError` raised inside `new_peak`
4. Full-node peer closed (`peer.close(9999)`)
5. Reconnect → repeat

The wallet cannot maintain a stable full-node connection and cannot process new peaks, auto-claim coins, or retry fee-bumped transactions. The `finish_exiting_height` row persists in the DB across restarts, so the condition survives wallet restarts.

---

### Likelihood Explanation

This is triggered by the **normal, expected flow** of leaving a pool. No adversarial input is required. The exit tx is submitted at `finish_height - 2` (when `new_peak` first fires), sits in the mempool, and is included in the first eligible block at `finish_height`. This is the common case in any environment where blocks are produced at a steady rate. The bug is reproducible deterministically in integration tests.

---

### Recommendation

Change the strict inequality in `coin_added` from `<` to `<=`:

```python
# chia/wallet/plotnft_wallet/plotnft_wallet.py, line 462
if finish_height is not None and finish_height <= height:   # was: finish_height < height
    await self.wallet_state_manager.plotnft2_store.clear_exiting_info(wallet_id=self.id())
```

Additionally, add a defensive try/except in `PlotNFT2Wallet.new_peak` that clears `finish_exiting_height` and logs a warning if `_finish_leaving_pool` raises, so a stale height can never permanently break peak processing.

---

### Proof of Concept

```
1. Create a PlotNFT2Wallet and join a pool (heightlock = 5).
2. Call leave_pool(); confirm the exit-to-waiting-room tx at height W.
   → finish_exiting_height = W + 5 stored in DB.
3. Farm blocks until height W + 5.
   → new_peak at W + 3 fires _finish_leaving_pool, submits exit tx.
   → Exit tx confirmed at height W + 5 (= finish_height).
   → coin_added called with exiting=False at height W + 5.
   → finish_height < height → W+5 < W+5 → False → NOT cleared.
4. Farm 2 more blocks (height W + 7).
   → new_peak: finish_height (W+5) <= (W+7) - 2 = W+5 → True.
   → get_unconfirmed_for_wallet == [] → guard passes.
   → _finish_leaving_pool called.
   → get_current_plotnft() returns exiting=False PlotNFT.
   → exit_waiting_room raises ValueError("Cannot exit waiting room while not in it").
   → Exception propagates → peer.close(9999).
5. Assert: wallet disconnects from full node on every subsequent block.
   Assert: finish_exiting_height row still present in DB.
```
