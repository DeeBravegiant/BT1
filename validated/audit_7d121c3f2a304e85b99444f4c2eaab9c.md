### Title
Harvester Fee Convention (CHIP-22) Has No Enforcement — Malicious Harvester Can Unconditionally Divert Farmer Block Rewards - (File: `chia/farmer/farmer_api.py`)

### Summary

The CHIP-22 third-party harvester fee convention allows a harvester to redirect the farmer's block reward (1.75 XCH) to an arbitrary address via `farmer_reward_address_override`. The farmer's only protection is a probabilistic "fee quality" check in `notify_farmer_reward_taken_by_harvester_as_fee()`, but that check **only emits a log warning** — it never rejects the proof, raises an exception, or prevents block submission. A malicious harvester can unconditionally divert 100% of the farmer's block reward to itself with no on-chain or meaningful off-chain enforcement.

---

### Finding Description

When a harvester sends `NewProofOfSpace` with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

That function checks the CHIP-22 fee quality convention but **only logs**: [2](#0-1) 

The function has no return value, raises no exception, and the caller in `new_proof_of_space` does not check any result — execution continues unconditionally to request signatures and proceed with block construction.

Later, in `_process_respond_signatures()`, the farmer unconditionally substitutes the harvester-supplied override address as the farmer reward destination: [3](#0-2) 

The `RespondSignatures` protocol message also carries a `farmer_reward_address_override` field that the harvester controls: [4](#0-3) 

There is no on-chain puzzle or condition that enforces a maximum fee rate. The `ProofOfSpaceFeeInfo.applied_fee_threshold` field is self-reported by the harvester: [5](#0-4) 

---

### Impact Explanation

A malicious third-party harvester can:

1. Send `NewProofOfSpace` with `farmer_reward_address_override` set to the attacker's own puzzle hash.
2. Set `applied_fee_threshold = 0xFFFFFFFF` (the maximum `uint32` value, representing a 100% fee rate) so the log-only check passes silently, **or** omit `fee_info` entirely (which also only triggers a log warning at line 930–934).
3. The farmer proceeds to sign and submit `DeclareProofOfSpace` with the attacker's address as `farmer_reward_address`.
4. The block is accepted by the network; the attacker receives the full 1.75 XCH farmer block reward.

This constitutes **unauthorized reward diversion** of XCH from the farmer to the attacker. The farmer loses their entire block reward with no recourse.

---

### Likelihood Explanation

Any third-party harvester service (e.g., a commercial GPU-accelerated plotting/harvesting service) that a farmer connects to can exploit this. The farmer-harvester connection is a normal, expected, unprivileged network connection. No key compromise or admin access is required. The attacker only needs to be a connected harvester peer.

---

### Recommendation

The farmer must **reject** the proof and refuse to proceed with block construction when the fee convention check fails, rather than merely logging a warning. Specifically, `notify_farmer_reward_taken_by_harvester_as_fee()` should return a boolean (or raise), and `new_proof_of_space` should gate further processing on that result:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    if not self.farmer.validate_harvester_fee_convention(sp, new_proof_of_space):
        return None  # reject — do not proceed to sign or submit
```

Additionally, consider enforcing a farmer-configured maximum fee threshold in the local config so that harvesters advertising above that threshold are always rejected.

---

### Proof of Concept

1. Attacker operates a harvester connected to a victim farmer.
2. Attacker sends `NewProofOfSpace` for a winning proof with:
   - `farmer_reward_address_override = <attacker_puzzle_hash>`
   - `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=uint32(0xFFFFFFFF))`
3. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()` — the check `fee_quality <= 0xFFFFFFFF` is always true, so only an info log is emitted. [6](#0-5) 
4. Farmer continues to `_process_respond_signatures()`, where `response.farmer_reward_address_override` is set to the attacker's address and unconditionally used. [7](#0-6) 
5. `DeclareProofOfSpace` is submitted to the full node with `farmer_reward_address = <attacker_puzzle_hash>`.
6. Block is farmed; attacker receives 1.75 XCH farmer reward. Victim farmer receives nothing.
