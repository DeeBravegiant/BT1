Audit Report

## Title
Missing Zero-Rate Guard in `CrossChainRateReceiver.lzReceive` and `viewSwapRsETHAmountAndFee` Causes Division-by-Zero, Temporarily Blocking Deposits — (`contracts/cross-chain/CrossChainRateReceiver.sol`, `contracts/pools/RSETHPoolV3.sol`)

## Summary

`CrossChainRateReceiver.lzReceive` unconditionally stores any decoded `uint256` rate, including zero, with no validation. `CrossChainRateProvider.updateRate()` is permissionless, so any caller can broadcast the current oracle value cross-chain. If `LRTOracle.rsETHPrice` is zero (its default before `updateRSETHPrice()` is first called), a zero rate is delivered to the receiver. `RSETHPoolV3.viewSwapRsETHAmountAndFee` then divides by this zero rate, causing a Solidity panic that reverts every deposit until a valid rate is re-broadcast.

## Finding Description

**Root cause 1 — no zero-rate guard in `lzReceive`:**

`CrossChainRateReceiver.lzReceive` decodes the payload and writes it directly to storage with no validation:

```solidity
uint256 _rate = abi.decode(_payload, (uint256));
rate = _rate;   // no require(_rate > 0)
``` [1](#0-0) 

**Root cause 2 — `LRTOracle.rsETHPrice` starts at zero:**

`rsETHPrice` is a plain `uint256` state variable; its initial value is `0` until `_updateRsETHPrice()` is first invoked. [2](#0-1) 

**Root cause 3 — permissionless `updateRate()`:**

`CrossChainRateProvider.updateRate()` has no access control — any external caller can push the current oracle value cross-chain: [3](#0-2) 

`RSETHRateProvider.getLatestRate()` reads `rsETHPrice()` directly with no zero-check: [4](#0-3) 

**Root cause 4 — unchecked division in `viewSwapRsETHAmountAndFee`:**

Both overloads of `viewSwapRsETHAmountAndFee` divide by `rsETHToETHrate` without guarding against zero: [5](#0-4) [6](#0-5) 

Both `deposit(string)` and `deposit(address,uint256,string)` call these functions unconditionally, and the `limitDailyMint` modifier also calls them before the function body executes: [7](#0-6) 

**Inconsistency confirming the gap:** `viewSwapAssetToPremintedRsETH` already has the correct guard (`if (rsETHToETHrate == 0) revert UnsupportedOracle()`), but `viewSwapRsETHAmountAndFee` does not: [8](#0-7) 

**Exploit path:**
1. `LRTOracle` is deployed but `updateRSETHPrice()` has not yet been called → `rsETHPrice == 0`.
2. Attacker calls `RSETHRateProvider.updateRate{value: fee}()` — permissionless, no guard.
3. `getLatestRate()` returns `0`; LayerZero delivers `abi.encode(0)` to `CrossChainRateReceiver`.
4. `lzReceive` stores `rate = 0` with no rejection.
5. Every subsequent call to `RSETHPoolV3.deposit()` (either overload) reaches `amountAfterFee * 1e18 / 0` and panics with a Solidity division-by-zero, reverting all deposits.
6. The freeze ends only after a valid non-zero rate is re-broadcast via `updateRate()` following proper oracle initialization.

## Impact Explanation

**Medium — Temporary Freezing of Funds.** All deposit paths on the L2 pool revert with a division-by-zero panic for the duration of the zero-rate window. No funds already in the contract are lost, and the freeze resolves once a valid rate is delivered. This matches the allowed impact "Medium. Temporary freezing of funds."

## Likelihood Explanation

**Low.** The precondition requires `LRTOracle.rsETHPrice == 0`, which holds only before `updateRSETHPrice()` is first called. In a correctly sequenced deployment this window is brief. However, because `updateRate()` is permissionless, a griefing actor monitoring the mempool can race to call it during this window. The same condition recurs if the oracle is ever redeployed or reset without immediately calling `updateRSETHPrice()`. No privileged access is required; any EOA with enough ETH to pay the LayerZero fee can trigger this.

## Recommendation

1. **Guard against zero in `lzReceive`:** Add `require(_rate > 0, "Rate must be non-zero");` before `rate = _rate` in `CrossChainRateReceiver.lzReceive`. [1](#0-0) 

2. **Guard against zero in `viewSwapRsETHAmountAndFee`:** Add `require(rsETHToETHrate > 0, "Invalid rate");` before the division in both overloads, consistent with the guard already present in `viewSwapAssetToPremintedRsETH`. [5](#0-4) 

3. **Guard against zero in `updateRate()`:** Add `require(latestRate > 0, "Rate must be non-zero");` in `CrossChainRateProvider.updateRate()` before encoding the payload. [9](#0-8) 

## Proof of Concept

```solidity
function test_zeroRateFreezesDeposits() public {
    // 1. Deploy LRTOracle but do NOT call updateRSETHPrice()
    //    => rsETHPrice == 0

    // 2. Anyone calls updateRate() on RSETHRateProvider (permissionless)
    rateProvider.updateRate{value: fee}();
    //    => getLatestRate() returns 0, payload = abi.encode(0) sent via LayerZero

    // 3. Simulate LayerZero delivery to RSETHRateReceiver
    vm.prank(layerZeroEndpoint);
    rateReceiver.lzReceive(
        srcChainId,
        abi.encodePacked(address(rateProvider), address(rateReceiver)),
        0,
        abi.encode(uint256(0))
    );

    // 4. Assert rate is now 0
    assertEq(rateReceiver.getRate(), 0);

    // 5. RSETHPoolV3.deposit() reverts with division-by-zero panic
    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert(); // Panic: division or modulo by zero
    pool.deposit{value: 1 ether}("ref");
}
```
