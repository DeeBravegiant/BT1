### Title
Peer-Controlled `state_diff_length` in Stored Header Drives Unchecked State Diff Assembly — (`crates/apollo_p2p_sync/src/client/state_diff.rs`, `header.rs`)

---

### Summary

`HeaderStreamBuilder` stores peer-supplied `SignedBlockHeader` values with zero cryptographic verification of `state_diff_length` or `state_diff_commitment`. `StateDiffStreamBuilder` then reads `target_state_diff_len` directly from that stored header to decide how many state-diff chunks to consume. A single malicious p2p peer can therefore dictate the exact `ThinStateDiff` written to storage for any block it serves.

---

### Finding Description

**Step 1 — Header accepted with no content validation.**

`HeaderStreamBuilder::parse_data_for_block` performs exactly two checks before returning `Ok(Some(signed_block_header))`: [1](#0-0) 

- Block number must equal the expected value.
- `signatures.len()` must equal `ALLOWED_SIGNATURES_LENGTH` (1).

There is no verification of `block_hash`, `state_diff_commitment`, or `state_diff_length` against any trusted source. No call to `verify_block_signature` exists anywhere in `crates/apollo_p2p_sync` (confirmed by search). The entire `BlockHeader` — including the peer-supplied `state_diff_length` — is written verbatim to storage: [2](#0-1) 

**Step 2 — State diff stream uses the stored (peer-controlled) length as its termination condition.**

`StateDiffStreamBuilder::parse_data_for_block` opens a read transaction and fetches `state_diff_length` from the just-stored header: [3](#0-2) 

It then loops, consuming chunks from the same peer, until `current_state_diff_len == target_state_diff_len`: [4](#0-3) 

After the loop exits, the assembled `ThinStateDiff` is stored **without any verification against `state_diff_commitment`**: [5](#0-4) 

**Step 3 — `BlockNumberLimit::Unlimited` removes the only natural throttle.**

Because `HeaderStreamBuilder::BLOCK_NUMBER_LIMIT = BlockNumberLimit::Unlimited`, the header stream queries up to `num_headers_per_query` blocks ahead with no storage-marker ceiling: [6](#0-5) [7](#0-6) 

The state diff stream uses `BlockNumberLimit::HeaderMarker` and waits for headers to be present before querying: [8](#0-7) 

This means a peer can pre-load a batch of crafted headers (advancing `header_marker` to N+k), then serve matching state diff chunks for each block in sequence. The syncing node processes them all without any cross-check.

---

### Impact Explanation

A malicious p2p peer (unauthenticated, zero privilege required) can write an arbitrary `ThinStateDiff` for any block into the node's storage. This corrupts the node's world state: every subsequent RPC call (`starknet_getStorageAt`, `starknet_call`, fee estimation, simulation) and every blockifier execution that reads from that state will operate on attacker-chosen values. If the node is also used as a sequencer, it will build blocks on top of a fabricated state root.

---

### Likelihood Explanation

Any peer that can establish a p2p connection to the syncing node can execute this attack. There is no rate-limiting, no cryptographic gating, and no post-storage consistency check that would detect or reject the corrupted data. The attack requires only that the peer respond to a header query before a legitimate peer does.

---

### Recommendation

1. **Verify `state_diff_commitment` after assembly.** After the `while` loop in `StateDiffStreamBuilder::parse_data_for_block`, call `calculate_state_diff_hash(&result)` and compare against the `state_diff_commitment` stored in the header. Reject with `BadPeerError` on mismatch.
2. **Verify block signatures cryptographically.** Call `verify_block_signature` (already present in `starknet_api::block`) inside `HeaderStreamBuilder::parse_data_for_block` using the known sequencer public key before storing any header.
3. **Verify block hash.** Compute the block hash from the stored commitments and compare against the peer-supplied `block_hash` before accepting a header.

---

### Proof of Concept

```
1. Attacker peer receives a header query for blocks [N, N+Q).
2. For each block i in [N, N+Q):
     Send SignedBlockHeader { block_number: i, state_diff_length: CRAFTED_LEN,
                              state_diff_commitment: <any>, signatures: [dummy] }
3. header_marker advances to N+Q; all crafted headers are in storage.
4. State diff stream issues a query for blocks [N, N+Q).
5. For each block i:
     target_state_diff_len = storage.get_block_header(i).state_diff_length  // == CRAFTED_LEN
     Send CRAFTED_LEN worth of StateDiffChunk messages encoding attacker-chosen
     storage writes / deployed contracts / nonces.
6. assert storage.get_state_diff(N) == attacker_chosen_state_diff  // passes
```

The stored `ThinStateDiff` for block N now contains attacker-chosen storage slots, deployed contracts, and nonces, with no error or warning emitted by the node.
