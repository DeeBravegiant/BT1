### Title
`verify_foreign_transaction` silently fails for `ForeignChain::Ton` (and `Ethereum`/`Solana`) when those chains are enabled on-chain — (`File: crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary

The MPC node's `execute_foreign_chain_request` function explicitly bails for `ForeignChainRpcRequest::Ton`, `::Ethereum`, and `::Solana` with hard-coded `bail!("... is unsupported")` statements. The on-chain contract, however, fully accepts `verify_foreign_transaction` requests for these chains whenever they appear in the `supported_chains` set. The result is a permanent, silent request-lifecycle failure: the contract enqueues the request and locks the user's deposit, but no node ever fulfils it, so the yield times out and the deposit is consumed.

### Finding Description

`ForeignChainRpcRequest` is a `#[non_exhaustive]` enum with twelve variants, including `Ton`, `Ethereum`, and `Solana`. [1](#0-0) 

The on-chain `verify_foreign_transaction` method checks only whether the requested chain is present in `get_supported_foreign_chains()` before enqueuing the yield request and locking the deposit. [2](#0-1) 

The node-side `execute_foreign_chain_request` function mirrors this check via `chain_is_supported`, which reads the same on-chain list. If the chain is present, the check passes. The function then dispatches on the request variant — but for `Ton`, `Ethereum`, and `Solana` it immediately bails: [3](#0-2) [4](#0-3) 

The `Ton` chain has complete type infrastructure: `TonRpcRequest`, `TonExtractor`, `TonExtractedValue`, a `ForeignTxSignPayload` hash snapshot test, and a sandbox-level `verify_foreign_transaction__should_succeed` test case — all of which pass because the sandbox tests never invoke the node's `execute_foreign_chain_request`. The e2e test that exercises real node signing explicitly omits Ton, Ethereum, and Solana from its chain list, confirming the gap. [5](#0-4) [6](#0-5) 

### Impact Explanation

When `ForeignChain::Ton` (or `Ethereum`/`Solana`) is present in the on-chain supported-chains set, any user who calls `verify_foreign_transaction` with one of these variants will:

1. Have their NEAR deposit locked by the contract's yield mechanism.
2. Receive no response — every node bails before producing a signature share.
3. Have the yield time out, consuming the deposit with no recourse.

This breaks the request-lifecycle safety invariant: the contract's own acceptance gate (`supported_chains.contains(chain)`) guarantees the user that the request is serviceable, but the node unconditionally rejects it. The mismatch is not surfaced to the caller at submission time.

This maps to the **Medium** allowed impact: *"request-lifecycle… manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

`ForeignChain::Ton` is a first-class variant in the public ABI, has full DTO coverage, and passes contract-level sandbox tests. A governance vote that adds `Ton` to the supported-chains whitelist (a normal operational step as new chains are onboarded) is sufficient to trigger the failure for every subsequent Ton request. No privileged access beyond the normal governance quorum is required.

### Recommendation

Either:
- Implement the Ton (and Ethereum/Solana) inspection logic in `execute_foreign_chain_request`, analogous to the existing Bitcoin/Starknet/Aptos arms; or
- Remove `ForeignChain::Ton` (and the other unimplemented variants) from the `ForeignChain` enum and the contract's accepted set until the node-side implementation is complete, so the contract's acceptance gate and the node's execution capability remain in sync.

### Proof of Concept

1. Governance votes to add `ForeignChain::Ton` to the supported-chains whitelist.
2. User calls `verify_foreign_transaction` with a `ForeignChainRpcRequest::Ton(...)` payload and attaches the required deposit.
3. Contract passes the `supported_chains.contains(&ForeignChain::Ton)` check and enqueues the yield.
4. Every MPC node calls `execute_foreign_chain_request` → `chain_is_supported` passes → hits `bail!("ForeignChainRpcRequest::Ton is unsupported")` at line 309 of `sign.rs`.
5. No node calls `respond_verify_foreign_tx`; the yield times out; the user's deposit is consumed. [7](#0-6) [4](#0-3)
