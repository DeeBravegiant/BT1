### Title
Unaccounted `directDeposits` in Preservation-of-Value Check Enables ADA Creation from Nothing — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in both top-level and sub-transaction bodies that allows a transaction to add ADA directly to registered stake accounts. However, this amount is never included in either the `consumed` or `produced` value used by the preservation-of-value (`consumed == produced`) check in the UTXO rule. An unprivileged transaction author can therefore include a `directDeposits` entry of arbitrary size, causing the target accounts to receive ADA that was never deducted from the transaction's UTxO inputs — creating ADA from nothing.

---

### Finding Description

**New field in both tx-body variants**

`DijkstraTxBodyRaw` (top-level) and `DijkstraSubTxBodyRaw` (sub-transaction) both carry a `directDeposits :: !DirectDeposits` field. [1](#0-0) [2](#0-1) 

**State mutation: accounts are credited**

In `dijkstraEntitiesTransition` (and its sub-transaction counterpart `dijkstraSubEntitiesTransition`), `applyDirectDeposits` unconditionally adds the specified amounts to the matching account balances: [3](#0-2) [4](#0-3) 

**Preservation-of-value: `directDeposits` absent from both sides**

`getConsumedDijkstraValue` delegates to `getConsumedMaryValue` for every tx body (top-level and sub-transactions). That function sums UTxO inputs, withdrawals, stake/DRep refunds, and minted assets — no `directDeposits`: [5](#0-4) 

`dijkstraProducedValue` sums outputs, fees, deposits, and treasury donations for the top-level body and all sub-transaction bodies — again no `directDeposits`: [6](#0-5) [7](#0-6) 

The `EraUTxO` instance for `DijkstraEra` wires these functions into the standard `validateValueNotConservedUTxO` check: [8](#0-7) 

**Asymmetry with withdrawals**

Withdrawals (which *remove* ADA from accounts) are correctly included on the `consumed` side. `directDeposits` (which *add* ADA to accounts) are the mirror operation and must appear on the `produced` side, but do not. The only validation performed on `directDeposits` is a network-ID check on the target addresses: [9](#0-8) 

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

Because `directDeposits` are absent from `produced`, the preservation-of-value equation the UTXO rule enforces is:

```
inputs + withdrawals + refunds  ==  outputs + fees + deposits
```

But the actual ledger-state change is:

```
inputs + withdrawals + refunds  ==  outputs + fees + deposits + directDeposits
```

A transaction that sets `directDeposits = Y` while keeping its UTxO balance normal (inputs = outputs + fees + deposits) passes the check, yet Y lovelace materialises in the target accounts with no corresponding deduction anywhere. Repeated across many transactions, this inflates the total ADA supply without bound.

---

### Likelihood Explanation

Any unprivileged user who can submit a Dijkstra-era transaction can trigger this. No special role, key, or governance majority is required. The attacker only needs to know the `AccountAddress` of any registered stake credential (publicly observable on-chain). The attack is deterministic and repeatable.

---

### Recommendation

Add the sum of all `directDeposits` amounts to the `produced` value in both `dijkstraProducedValue` (top-level) and `dijkstraSubTxProducedValue` (sub-transaction):

```haskell
-- in dijkstraProducedValue / dijkstraSubTxProducedValue
<> inject (fold $ unDirectDeposits $ txBody ^. directDepositsTxBodyL)
```

This mirrors the treatment of `treasuryDonationTxBodyL` (also an ADA sink that is already included in `produced`) and restores the preservation-of-value invariant. [6](#0-5) [7](#0-6) 

---

### Proof of Concept

1. Alice has a registered stake account with balance 0 ADA.
2. Bob constructs a Dijkstra top-level transaction:
   - `inputs`: one UTxO worth 10 ADA
   - `outputs`: one UTxO worth 10 ADA (back to Bob, minus fees — set fees = 0 for clarity)
   - `directDeposits`: `{ Alice's AccountAddress → 1_000_000_000 lovelace }`
3. The UTXO rule evaluates:
   - `consumed = getConsumedDijkstraValue … = 10 ADA` (inputs only)
   - `produced = getProducedDijkstraValue … = 10 ADA` (outputs only; `directDeposits` absent)
   - `consumed == produced` → **check passes**
4. `dijkstraEntitiesTransition` calls `applyDirectDeposits`, crediting Alice's account with 1,000,000,000 lovelace.
5. Alice's account balance is now 1,000,000,000 lovelace; total ADA supply has increased by 1,000,000,000 lovelace with no corresponding deduction anywhere in the ledger state.

The attack is repeatable with arbitrarily large `directDeposits` values, bounded only by the `Coin` type's maximum.

```haskell
getConsumedDijkstraValue ::
  forall era l.
  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
```
