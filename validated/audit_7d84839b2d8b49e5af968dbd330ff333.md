Audit Report

## Title
Stale `rsETHPrice` Allows Depositors to Capture Accrued Yield from Existing rsETH Holders — (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary

`LRTOracle.rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is explicitly called. Between updates, as EigenLayer rewards increase the protocol's actual ETH TVL, the stored price becomes stale (lower than the true per-share value). `LRTDepositPool.getRsETHAmountToMint()` mints rsETH using this stale stored price, allowing an attacker to deposit at the artificially low price and then trigger a price update, capturing a portion of the yield that should accrue exclusively to pre-existing holders.

## Finding Description

`LRTOracle.updateRSETHPrice()` is declared `public` with no access restriction beyond `whenNotPaused`: [1](#0-0) 

`LRTDepositPool.getRsETHAmountToMint()` computes the mint amount by dividing by `lrtOracle.rsETHPrice()`, which reads the stored state variable — not a freshly computed value: [2](#0-1) 

The deposit flow calls `_beforeDeposit` → `getRsETHAmountToMint` with no price refresh step: [3](#0-2) 

The `minRSETHAmountExpected` slippage guard only protects the depositor from receiving *fewer* tokens than expected; it does not prevent the depositor from receiving *more* tokens than the true current price warrants: [4](#0-3) 

When EigenLayer rewards cause `_getTotalEthInProtocol()` to exceed `rsethSupply × rsETHPrice`, the stored price is below the actual per-share value. `_updateRsETHPrice()` computes `previousTVL` using the stored price against the *current* (post-deposit) supply, so the attacker's deposit is treated as principal rather than yield: [5](#0-4) 

The `pricePercentageLimit` guard reverts `updateRSETHPrice()` for non-managers only when the price increase exceeds the configured threshold: [6](#0-5) 

This guard does not prevent the deposit at the stale price. For reward accruals within the daily threshold, the attacker can call `updateRSETHPrice()` themselves immediately after depositing. For larger accruals, the deposit is already committed and the attacker profits when a manager eventually calls `updateRSETHPriceAsManager()`.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders earn yield as EigenLayer rewards increase the protocol's total ETH. This yield is reflected when `rsETHPrice` is updated upward. An attacker who deposits at the stale (pre-update) price dilutes the supply, capturing a fraction of the pending yield that should accrue exclusively to pre-existing holders. The attack scales with deposit size relative to TVL and with the magnitude of accrued rewards, and is repeatable every reward cycle.

## Likelihood Explanation

**Low.** The attack requires no special privileges — only the ability to monitor on-chain state (NodeDelegator ETH balances vs. stored `rsETHPrice`) and submit two sequential transactions. The attacker can compute the pending price increase off-chain by simulating `_getTotalEthInProtocol()` and comparing to `rsethSupply × rsETHPrice`. The `pricePercentageLimit` guard partially mitigates by blocking non-manager price updates above the threshold, but does not prevent the deposit at the stale price. For sub-threshold reward accruals, the full two-transaction attack is executable by any unprivileged user.

## Recommendation

1. **Refresh price before minting**: Compute the current rsETH price inline within `getRsETHAmountToMint()` (or call `_updateRsETHPrice()` at the start of `depositETH`/`depositAsset`) so deposits always use the up-to-date exchange rate.
2. **Deposit fee**: Introduce a small deposit fee exceeding the expected per-block reward accrual, making the attack economically unviable.
3. **Access-restrict price updates**: Restrict `updateRSETHPrice()` to a trusted keeper role, reducing the attacker's ability to time the update immediately after depositing.

## Proof of Concept

**Setup:**
- `rsETHPrice` (stored) = 1.05 ETH/rsETH
- `rsethSupply` = 100 rsETH
- `totalETHInProtocol` (actual) = 110 ETH (5 ETH rewards accrued, `previousTVL` = 100 × 1.05 = 105 ETH)

**Attack:**

1. Attacker calls `LRTDepositPool.depositETH{value: 10 ether}(0, "")`.
   - `getRsETHAmountToMint` computes: `10e18 × 1e18 / 1.05e18 ≈ 9.524 rsETH` minted to attacker.
   - New `rsethSupply` = 109.524 rsETH; new `totalETH` = 120 ETH.

2. Attacker calls `LRTOracle.updateRSETHPrice()`.
   - `previousTVL = 109.524 × 1.05 ≈ 115 ETH`
   - `rewardAmount = 120 − 115 = 5 ETH`
   - `newRsETHPrice = 120 / 109.524 ≈ 1.0956 ETH/rsETH`

3. Attacker's 9.524 rsETH is now worth `9.524 × 1.0956 ≈ 10.435 ETH`.
   - **Attacker profit: ≈ 0.435 ETH** (deposited 10 ETH, holds 10.435 ETH equivalent).
   - **Existing holders' loss**: their 100 rsETH is worth `100 × 1.0956 = 109.56 ETH` instead of the `110 ETH` they would have had without the attack.

**Foundry test plan**: Deploy `LRTOracle` and `LRTDepositPool` on a fork, seed the protocol with 100 rsETH / 105 ETH TVL, simulate 5 ETH of EigenLayer rewards (increase NodeDelegator balance), execute the two-transaction attack sequence, and assert that the attacker's rsETH value post-update exceeds their deposit amount while existing holders' per-share value is lower than it would have been without the attack.
