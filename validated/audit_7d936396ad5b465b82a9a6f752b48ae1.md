Audit Report

## Title
Asymmetric Downside Auto-Pause Permanently Stales `rsETHPrice`, Freezing Withdrawal Queue — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle._updateRsETHPrice()` unconditionally pauses the oracle and returns without writing `rsETHPrice` when a price drop exceeds `pricePercentageLimit`. Unlike the upside case, no manager role bypass exists for the downside branch, so neither `updateRSETHPrice()` (blocked by `whenNotPaused`) nor `updateRSETHPriceAsManager()` (re-enters the same branch and re-pauses) can advance the stored price. Because the withdrawal manager is also paused by the same trigger, `unlockQueue()` and `completeWithdrawal()` are blocked, freezing all pending user withdrawals.

## Finding Description
`_updateRsETHPrice()` contains asymmetric threshold handling:

**Upside — manager bypass exists (lines 260–266):**
```solidity
if (isPriceIncreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        revert PriceAboveDailyThreshold();
    }
}
// execution continues; rsETHPrice is written
```
A manager can call `updateRSETHPriceAsManager()` and the price is written even above the upside limit.

**Downside — no manager bypass (lines 277–282):**
```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;   // rsETHPrice is never written
}
```
There is no role check. The function pauses all three contracts and returns for every caller, including the manager.

**Both public entry points call the same internal function:**
- `updateRSETHPrice()` (line 87) has `whenNotPaused` → reverts after auto-pause.
- `updateRSETHPriceAsManager()` (line 94) has no pause guard → reaches `_updateRsETHPrice()`, re-detects `isPriceDecreaseOffLimit`, calls `_pause()` (idempotent, no-op if already paused), and hits `return` — `rsETHPrice` is still not written.

`highestRsethPrice` is only updated upward (lines 293–296), so the recovery bar is always the all-time peak. Partial recoveries that remain below `highestRsethPrice * (1 - pricePercentageLimit)` are insufficient to exit the frozen state.

There is no direct setter for `rsETHPrice` or `highestRsethPrice` anywhere in `LRTOracle.sol`. The only indirect escape hatch is for the admin to call `setPricePercentageLimit(0)`, unpause the oracle, and then call `updateRSETHPrice()` — a non-obvious multi-step sequence not documented as a recovery procedure.

`unlockQueue()` (line 279, `whenNotPaused`) and `completeWithdrawal()` (line 183, `whenNotPaused`) on `LRTWithdrawalManager` are both blocked while the withdrawal manager is paused. If the admin unpauses the withdrawal manager while the oracle price remains stale (inflated), `_createUnlockParams` reads `lrtOracle.rsETHPrice()` (line 847), which is the pre-pause inflated value, causing `unlockQueue()` to over-pay users and risk insolvency.

## Impact Explanation
Users who called `initiateWithdrawal()` have their rsETH locked inside `LRTWithdrawalManager`. While the withdrawal manager is paused, neither `unlockQueue()` nor `completeWithdrawal()` can execute. If the actual NAV never recovers above `highestRsethPrice * (1 - pricePercentageLimit)`, there is no on-chain path to update `rsETHPrice` and unfreeze the queue through the intended mechanism. This constitutes **temporary (escalating to permanent) freezing of user funds**, matching the allowed impact class. If the admin unpauses the withdrawal manager while the oracle price is stale, `unlockQueue()` uses the inflated price, over-paying users and causing **protocol insolvency**.

**Primary impact: Medium — Temporary freezing of funds. Escalates to Critical (permanent freeze / protocol insolvency) if price does not recover or admin takes the wrong recovery path.**

## Likelihood Explanation
Any unprivileged external caller can trigger the auto-pause by calling `updateRSETHPrice()` once the on-chain NAV has already fallen below the threshold — no privileged access required. EigenLayer restaking carries real correlated slashing risk across validators, making a sharp NAV drop realistic. The condition requires no victim mistake; users who initiated withdrawals before the event are passively affected. The trigger is repeatable: every time the admin unpauses the oracle and the price is still below the threshold, any caller can re-trigger the pause.

## Recommendation
Mirror the upside manager bypass for the downside case. In `_updateRsETHPrice()`, add a role check before the unconditional pause-and-return:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;
    }
    // Manager consciously accepts the downside; execution continues to write rsETHPrice
}
```

Alternatively, add an explicit `setRsETHPrice(uint256)` admin/manager function that can manually advance the stored price after a governance decision, bypassing the broken auto-pause path. Also consider adding a `setHighestRsethPrice(uint256)` admin function to allow resetting the peak after a confirmed slashing event.

## Proof of Concept
1. Deploy with `highestRsethPrice = 1.05 ether`, `pricePercentageLimit = 5e16` (5%).
2. A slashing event reduces NAV so `newRsETHPrice = 0.99 ether` (5.7% below peak → `isPriceDecreaseOffLimit = true`).
3. Any address calls `updateRSETHPrice()`. `_updateRsETHPrice()` pauses `lrtDepositPool`, `withdrawalManager`, and the oracle, then returns. `rsETHPrice` remains `1.05 ether`.
4. Admin calls `LRTOracle.unpause()`. Manager calls `updateRSETHPriceAsManager()`. Same branch fires, `_pause()` is a no-op (already paused), `return` executes — `rsETHPrice` unchanged.
5. Users with rsETH locked in `LRTWithdrawalManager` (from prior `initiateWithdrawal()`) call `completeWithdrawal()` → reverts `whenNotPaused`.
6. Operator calls `unlockQueue()` → reverts `whenNotPaused`.
7. If price never recovers to `≥ 0.9975 ether`, the withdrawal queue is permanently frozen with no direct on-chain resolution path.
