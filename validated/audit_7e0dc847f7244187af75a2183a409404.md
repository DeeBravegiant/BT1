Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Unlimited ETH Deposits — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` silently discards the `amount` parameter when the asset is ETH, checking only whether the pre-deposit total already exceeds the cap rather than whether the incoming deposit would breach it. Any unprivileged caller can therefore deposit an arbitrarily large amount of ETH in a single transaction while the pre-deposit total remains at or below the configured limit, bypassing the ETH deposit cap entirely.

## Finding Description
The asymmetry is confirmed in the deployed code:

```solidity
// contracts/LRTDepositPool.sol L676-682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
``` [1](#0-0) 

The ETH branch evaluates `totalAssetDeposits > limit`, a condition that is `false` whenever the cap has not yet been breached. The incoming `amount` (i.e. `msg.value`) is never added to `totalAssetDeposits` before the comparison, so the guard can never trigger for a single ETH deposit that arrives while the running total is still within bounds.

`depositETH` is public and payable, passes `msg.value` as `depositAmount` to `_beforeDeposit`, which calls this function and reverts only on `true`: [2](#0-1) [3](#0-2) 

Because the ETH branch never evaluates `depositAmount`, the `MaximumDepositLimitReached` revert is never triggered for any single ETH deposit so long as the pre-deposit total is ≤ the configured limit.

## Impact Explanation
The ETH deposit limit is a risk-management control capping EigenLayer exposure. With the check broken, a single depositor can push the protocol's ETH holdings arbitrarily above the intended ceiling in one call. The protocol then holds more ETH in EigenLayer strategies than governance intended, violating the promised cap. This matches the allowed impact: **Low — Contract fails to deliver promised returns** (the deposit limit guarantee is not upheld for ETH).

## Likelihood Explanation
`depositETH` is public and permissionless. No special role, flash loan, or multi-step setup is required. Any depositor who sends ETH while `getTotalAssetDeposits(ETH_TOKEN) ≤ depositLimitByAsset[ETH_TOKEN]` will bypass the cap. The condition is trivially satisfiable at any point before the limit is organically reached. Likelihood is **High**.

## Recommendation
Include the deposit amount in the ETH branch, mirroring the ERC-20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1_000 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Attacker calls `depositETH{value: 50_000 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit`: `999e18 > 1_000e18` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full 50 000 ETH.
6. Protocol now holds 50 999 ETH — 50× the intended cap — with no further check preventing it.

**Foundry test plan:**
```solidity
function test_ETHDepositBypassesLimit() public {
    // set limit to 1000 ETH, pre-seed 999 ETH
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, 1000 ether);
    depositPool.depositETH{value: 999 ether}(0, "");
    // now attempt to deposit 50_000 ETH — should revert but does not
    vm.expectRevert(LRTDepositPool.MaximumDepositLimitReached.selector);
    depositPool.depositETH{value: 50_000 ether}(0, "");
    // test will fail (no revert), confirming the bug
}
```
