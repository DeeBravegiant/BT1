Audit Report

## Title
Downside-Protection Pause Loop Permanently Freezes Protocol After Significant Price Drop — (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle._updateRsETHPrice()` pauses the protocol when the rsETH price drops more than `pricePercentageLimit` below `highestRsethPrice`, but returns without writing the new price to `rsETHPrice`. Because `rsETHPrice` retains its pre-drop value and `highestRsethPrice` is never decreased, every subsequent invocation recomputes the same excessive drop and re-triggers the pause. After an admin unpauses, any unprivileged caller can immediately re-pause all three contracts by calling the public `updateRSETHPrice()`, blocking deposits and withdrawals indefinitely.

## Finding Description
In `_updateRsETHPrice()`, the downside-protection branch at lines 277–281 pauses `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTOracle`, then returns:

```solidity
// contracts/LRTOracle.sol L277-281
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;   // exits WITHOUT writing rsETHPrice
}
```

The only write to `rsETHPrice` is at line 313, which is unreachable when the branch fires:

```solidity
rsETHPrice = newRsETHPrice;   // L313 — never reached
```

`highestRsethPrice` is only ever increased (line 294–296); there is no admin function to lower it. On every subsequent call, `previousTVL` is computed from the stale, inflated `rsETHPrice` (line 234), `newRsETHPrice` is still the lower post-drop value, and `diff` still exceeds the threshold, so the pause fires again.

The manager bypass `updateRSETHPriceAsManager()` (lines 94–96) calls the same `_updateRsETHPrice()` and provides no escape: the downside branch does not inspect `msg.sender`'s role, unlike the upside branch (line 263). `_pause()` is a no-op when already paused (lines 319–322), so the function still returns without writing `rsETHPrice`.

`updateRSETHPrice()` is gated by `whenNotPaused` (line 87), so after the first pause only `updateRSETHPriceAsManager()` is callable by the manager — and it also cannot escape the loop.

The only practical escape is for the admin to set `pricePercentageLimit = 0` (line 125–128) and call a price update, which updates `rsETHPrice` but leaves `highestRsethPrice` at the old peak. Restoring the original limit immediately re-triggers the pause on the next call because `diff = highestRsethPrice − newRsETHPrice` is unchanged. The admin must therefore permanently set `pricePercentageLimit` to a value larger than the total drop from the all-time high, or keep it at zero, to restore normal operation — effectively disabling all downside protection.

## Impact Explanation
All user-facing entry points on `LRTDepositPool` (`depositETH`, `depositAsset`) and `LRTWithdrawalManager` (`initiateWithdrawal`, `completeWithdrawal`, `instantWithdrawal`) carry `whenNotPaused`. While the three contracts are paused, all deposits and withdrawals are blocked. Because any unprivileged caller can re-trigger the pause immediately after an admin unpauses, user funds are frozen for as long as the admin has not permanently altered the safety parameters. This constitutes **Temporary freezing of funds (Medium)** at minimum; the absence of any function to lower `highestRsethPrice` makes the freeze effectively permanent until the admin disables downside protection entirely.

## Likelihood Explanation
The trigger condition — a single oracle update in which `totalETHInProtocol / rsethSupply` falls more than `pricePercentageLimit` below `highestRsethPrice` — is reachable through realistic on-chain events: EigenLayer operator slashing (reducing `getEffectivePodShares()`), a supported LST depeg (reducing `getAssetPrice()` output), or a large rapid withdrawal from the unstaking vault. `pricePercentageLimit` has no enforced minimum; even a conservative 1–5% threshold can be breached by normal market volatility. Once triggered, re-triggering requires only a public call to `updateRSETHPrice()` with no special permissions or capital.

## Recommendation
Write the new price before returning in the downside-protection branch so the condition does not persist across invocations:

```solidity
if (isPriceDecreaseOffLimit) {
    rsETHPrice = newRsETHPrice;   // update so condition clears on next call
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

Additionally, add an admin-callable function to lower `highestRsethPrice` (with appropriate access control and a floor at the current `rsETHPrice`) to allow recovery without permanently disabling the threshold.

## Proof of Concept
**Setup:** `rsETHPrice = 1.0e18`, `highestRsethPrice = 1.0e18`, `pricePercentageLimit = 5e16` (5%), `rsethSupply = 1000e18`.

1. EigenLayer slashing reduces `totalETHInProtocol` to `900e18`.
2. Anyone calls `updateRSETHPrice()`:
   - `newRsETHPrice = 900e18 / 1000 = 0.9e18`
   - `diff = 1.0e18 − 0.9e18 = 0.1e18 > 0.05 × 1.0e18` → `isPriceDecreaseOffLimit = true`
   - All three contracts paused; `return` without writing `rsETHPrice`.
   - `rsETHPrice` remains `1.0e18`.
3. Admin unpauses all three contracts.
4. Anyone calls `updateRSETHPrice()` again:
   - `previousTVL = 1000e18 × 1.0e18 = 1000e18` (stale).
   - `totalETHInProtocol = 900e18` (unchanged).
   - `newRsETHPrice = 0.9e18`, `diff = 0.1e18 > 0.05e18` → pause fires again.
5. Protocol is paused again. Steps 3–4 repeat indefinitely.

A Foundry fork test can confirm this by: (a) mocking `_getTotalEthInProtocol()` to return `900e18`, (b) calling `updateRSETHPrice()`, (c) asserting all three contracts are paused and `rsETHPrice == 1.0e18`, (d) unpausing, (e) calling `updateRSETHPrice()` again, and (f) asserting all three contracts are paused again with `rsETHPrice` still `1.0e18`.
