The bug is real and locally verifiable. Here is the full analysis.

### Title
Stale `cancel_votes` Persist Across Domain Transitions, Reducing Effective Cancel Threshold — (`crates/contract/src/state/initializing.rs`)

### Summary
`cancel_votes` is a session-level `BTreeSet` that is **never cleared** when `vote_pk` advances `generating_key` to the next domain. Any participant who cast a cancel vote for domain N has that vote silently counted for every subsequent domain in the same `InitializingContractState` session, reducing the effective cancel threshold for later domains below the protocol-intended value.

### Finding Description

When `vote_pk` completes a domain and advances to the next one, it replaces only `self.generating_key`: [1](#0-0) 

`self.cancel_votes` is never touched. It accumulates votes across all domain transitions for the lifetime of the `InitializingContractState`.

In `vote_cancel`, the only replay-prevention guard is the `next_domain_id` check: [2](#0-1) 

`self.domains.next_domain_id()` is the maximum domain ID + 1 across the entire registry. It is **fixed at session creation** and does not change between domain transitions. It is identical for every domain in the session, so it provides no protection against intra-session stale votes.

The insert-and-count logic on line 132 uses `BTreeSet::insert`, which returns `false` for a duplicate: [3](#0-2) 

When participant P tries to re-vote cancel for domain 1 after already voting for domain 0, `insert` returns `false`, the condition short-circuits, and the call returns `Ok(None)` — but P's stale vote from domain 0 is still present in the set and is counted toward the threshold for domain 1.

### Impact Explanation

**Medium.** This breaks the production safety invariant that "cancelling key generation requires a fresh threshold of votes." A coalition of `threshold − 1` participants who voted cancel during domain 0 (but failed to reach threshold because domain 0 still completed) effectively carry those votes forward. For domain 1, only **1 additional** participant needs to vote cancel to trigger cancellation — far below the intended threshold. This is a contract execution-flow manipulation that breaks an accounting invariant without requiring network-level DoS or operator misconfiguration.

The impact does **not** reach Critical: the cancel operation only reverts to `RunningContractState` with already-generated keys intact. It does not issue unauthorized signatures, expose key shares, or enable forgery or secret recovery. [4](#0-3) 

### Likelihood Explanation

Requires a multi-domain `InitializingContractState` session (triggered by `vote_add_domains` with ≥ 2 new domains) and a sub-threshold coalition that votes cancel during domain 0 while domain 0 still completes. Both conditions are reachable in normal operation without any collusion above threshold.

### Recommendation

Reset `cancel_votes` whenever `generating_key` advances to a new domain inside `vote_pk`:

```rust
self.generating_key = KeyEvent::new(...);
self.cancel_votes = BTreeSet::new(); // add this line
```

This ensures each domain starts with a clean slate of cancel votes, matching the protocol's intent.

### Proof of Concept

```rust
// Setup: 3 domains, threshold = 2, 3 participants (P0, P1, P2)
// Domain 0 is currently generating.

// Step 1: P0 votes cancel for domain 0 (below threshold).
env.set_signer(&P0);
state.vote_cancel(state.domains.next_domain_id()).unwrap(); // Ok(None)
assert_eq!(state.cancel_votes.len(), 1); // {P0}

// Step 2: All participants vote pk for domain 0 → domain advances to domain 1.
// cancel_votes is NOT reset.
for p in [P0, P1, P2] {
    env.set_signer(&p);
    state.vote_pk(key_event_id_domain0, pk.clone()).unwrap();
}
assert_eq!(state.cancel_votes.len(), 1); // still {P0}

// Step 3: P0 tries to vote cancel for domain 1.
env.set_signer(&P0);
let inserted = state.cancel_votes.insert(P0_authenticated); // returns false
assert!(!inserted); // duplicate — silently ignored
assert_eq!(state.cancel_votes.len(), 1); // still 1, not 2

// Step 4: P1 votes cancel for domain 1 — only 1 new vote needed to reach threshold=2.
env.set_signer(&P1);
let result = state.vote_cancel(state.domains.next_domain_id()).unwrap();
assert!(result.is_some()); // cancellation triggered with only 1 fresh vote
```

P0's stale vote from domain 0 counted toward domain 1's threshold, reducing the effective cancel threshold from 2 to 1 for domain 1.
