Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Withdrawals Post-Deployment - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` does not set `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal()` reverts when `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and `0 >= 0` is always true, every withdrawal attempt fails from deployment until an admin explicitly calls `setMaxNumberOfWithdrawalsPerUser`. Any user who stakes KERNEL tokens during this window has their funds temporarily frozen with no self-service remedy.

## Finding Description
`initialize()` sets only `kernelToken`, `rewardsToken`, and the admin role — `maxNumberOfWithdrawalsPerUser` is never assigned: [1](#0-0) 

The state variable is declared with no default: [2](#0-1) 

The withdrawal guard in `initiateWithdrawal()`: [3](#0-2) 

With `maxNumberOfWithdrawalsPerUser == 0`, the condition `userWithdrawalIds[msg.sender].length >= 0` is always `true` for every caller (a fresh user has 0 withdrawal IDs, and `0 >= 0` holds). Every call to `initiateWithdrawal()` reverts with `WithdrawalLimitReached`. There is no alternative unstake path — `stake()` and `stakeFor()` only increase balances, and no emergency exit exists.

The only remedy is an admin call to `setMaxNumberOfWithdrawalsPerUser`, which explicitly rejects `0` as invalid: [4](#0-3) 

This means the zero state cannot be restored once fixed, but more importantly, the contract is deployed broken and requires a separate admin transaction to become functional.

## Impact Explanation
Any user who calls `stake()` before the admin sets `maxNumberOfWithdrawalsPerUser` has their KERNEL tokens locked in the contract with no self-service exit. This is a **temporary freezing of funds** (Medium severity), matching the allowed impact scope. The freeze persists for an indeterminate period — from deployment until admin action — and is especially harmful during the early high-activity period immediately after launch.

## Likelihood Explanation
This condition is present on every fresh deployment of `KernelDepositPool`. The `initialize()` function is the sole initialization path and structurally omits the parameter. Any user who stakes before the admin completes post-deployment configuration is affected. No on-chain guard prevents staking while withdrawals are blocked, and there is no event or revert in `stake()` to warn users.

## Recommendation
Set a safe non-zero default for `maxNumberOfWithdrawalsPerUser` inside `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = 10; // or any reasonable value ≤ MAX_WITHDRAWALS_PER_USER
```

Alternatively, accept `_maxNumberOfWithdrawalsPerUser` as an `initialize()` parameter, validate it is non-zero and within bounds, and store it before the contract becomes usable.

## Proof of Concept
1. Admin deploys and calls `KernelDepositPool.initialize(admin, kernelToken, rewardToken)`.
2. `maxNumberOfWithdrawalsPerUser` is `0` (Solidity default).
3. Alice calls `stake(100e18)` — succeeds; `balanceOf[Alice] = 100e18`.
4. Alice calls `initiateWithdrawal(100e18)`:
   - Check at L323: `userWithdrawalIds[Alice].length >= maxNumberOfWithdrawalsPerUser` → `0 >= 0` → `true`
   - Reverts with `WithdrawalLimitReached`.
5. Alice's `100e18` KERNEL tokens remain locked. No other withdrawal function exists.
6. Alice must wait for admin to call `setMaxNumberOfWithdrawalsPerUser(N)` with `N > 0`.

**Foundry test sketch:**
```solidity
function test_withdrawalBlockedBeforeAdminSetsLimit() public {
    // deploy + initialize (no maxNumberOfWithdrawalsPerUser set)
    pool.initialize(admin, address(kernelToken), address(rewardToken));
    
    kernelToken.mint(alice, 100e18);
    vm.startPrank(alice);
    kernelToken.approve(address(pool), 100e18);
    pool.stake(100e18);
    
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(100e18);
    vm.stopPrank();
    
    assertEq(kernelToken.balanceOf(alice), 0); // funds locked
}
```
