Audit Report

## Title
`SwapAllowlistExtension.beforeSwap` gates the router address instead of the originating user, enabling allowlist bypass for router-mediated swaps — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension` is intended to restrict pool swaps to individually permissioned addresses. However, `MetricOmmPool.swap` passes `msg.sender` (the direct caller) as the `sender` argument to `_beforeSwap`, which is the `MetricOmmSimpleRouter` contract address when users route through it. `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[pool][router]` rather than `allowedSwapper[pool][actual_user]`, nullifying per-user access control for all router-mediated swaps.

## Finding Description
`MetricOmmPool.swap` passes `msg.sender` verbatim as `sender` to `_beforeSwap`:

```solidity
// metric-core/contracts/MetricOmmPool.sol:230-231
_beforeSwap(
  msg.sender,   // direct pool caller — router when routed
  ...
```

`ExtensionCalling._beforeSwap` forwards this unchanged to every configured extension via `abi.encodeCall`:

```solidity
// metric-core/contracts/ExtensionCalling.sol:160-176
_callExtensionsInOrder(
  BEFORE_SWAP_ORDER,
  abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, recipient, ...))
);
```

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]` where `msg.sender` is the pool and `sender` is the direct pool caller:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol:37-38
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
  revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` directly, making the router `msg.sender` inside the pool:

```solidity
// metric-periphery/contracts/MetricOmmSimpleRouter.sol:72-80
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ..., "", params.extensionData);
```

The router does not encode the originating user (`msg.sender` at the router level) into `extensionData`; it passes user-supplied `params.extensionData` directly. The extension has no mechanism to recover the real user identity. The bug is structural: there is no configuration the pool admin can apply to make the extension check the actual swapper when the router is the intermediary.

**Existing guards are insufficient:** `allowAllSwappers` and `allowedSwapper` both key on the direct pool caller. No trusted-router registry or originator field exists in the extension interface or pool interface.

## Impact Explanation
Two concrete broken invariants:

**Scenario A — Allowlist bypass (Critical):** Pool admin deploys a pool with `SwapAllowlistExtension` to restrict swaps to KYC'd users. Admin allowlists the router so legitimate users can use periphery. Any unpermissioned attacker calls `router.exactInputSingle(...)` → router calls `pool.swap(...)` → `sender = router` → `allowedSwapper[pool][router] == true` → check passes → attacker swaps freely. The entire per-user allowlist is nullified for router-mediated swaps.

**Scenario B — DoS for legitimate users (Medium):** If admin does NOT allowlist the router (only individual user addresses), every allowlisted user who swaps through the router is blocked with `NotAllowedToSwap`. They must call `pool.swap` directly, losing slippage protection, multi-hop routing, and deadline enforcement.

Both outcomes break the core invariant that `SwapAllowlistExtension` gates the economically relevant swapper. Scenario A constitutes an admin-boundary break by an unprivileged path.

## Likelihood Explanation
`MetricOmmSimpleRouter` is the primary public-facing swap interface; all ordinary users are expected to route through it. `SwapAllowlistExtension` is a production periphery contract explicitly designed for per-pool access control. The pool admin has no configuration path to fix this — the bug is structural, not a misconfiguration. Any unpermissioned user can trigger Scenario A with a single `exactInputSingle` call, requiring no special privileges or setup beyond the pool admin having allowlisted the router.

## Recommendation
The `beforeSwap` hook receives `sender` (direct pool caller) and `recipient`, but neither is the actual end user when the router is involved. The correct fix requires the router to encode the real originating user in `extensionData` and the extension to decode it, verified against a known-router registry. A minimal mitigation is to document that `SwapAllowlistExtension` gates the direct pool caller only, and require pool admins to understand that allowlisting the router grants access to all router users — but this fundamentally changes the security model the extension advertises.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` configured.
2. Pool admin calls `swapExtension.setAllowedToSwap(pool, address(router), true)` — allowlisting the router so legitimate users can use periphery.
3. Unpermissioned `attacker` calls `router.exactInputSingle({pool: pool, recipient: attacker, ...})`.
4. Router calls `pool.swap(attacker, ...)` — `msg.sender` inside pool is `router`.
5. `_beforeSwap(router, attacker, ...)` is called; `sender = router`.
6. `SwapAllowlistExtension.beforeSwap` evaluates `allowedSwapper[pool][router] == true` → passes.
7. Attacker receives output tokens despite never being individually allowlisted.
