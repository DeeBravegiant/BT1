Audit Report

## Title
Stale `rsETHPrice` Enables Sandwich Attack on `updateRSETHPrice()` to Extract Yield from Existing Holders — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is publicly callable with no access control beyond `whenNotPaused`. Because `LRTDepositPool.getRsETHAmountToMint()` prices deposits using the stored (potentially stale) `rsETHPrice`, an unprivileged attacker can deposit at the stale lower price, trigger the price update, and immediately lock in a withdrawal at the new higher price. After the mandatory delay, the attacker receives more ETH than deposited, extracting value from existing holders' accrued yield.

## Finding Description

**Root cause:** `rsETHPrice` is a lazily-updated storage variable. Deposits consume it directly, while any caller can advance it at will.

`updateRSETHPrice()` has no access control:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`getRsETHAmountToMint()` reads the stored stale value:

```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`initiateWithdrawal()` locks `expectedAssetAmount` at the current (post-update) price:

```solidity
// contracts/LRTWithdrawalManager.sol L168
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
// L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

**Exploit flow:**

1. Rewards accrue (stETH rebase, EigenLayer rewards). `rsETHPrice` is stale-low at `P_old`; actual TVL implies `P_actual > P_old`.
2. Attacker deposits `X` ETH. Because `rsETHPrice = P_old`, they receive `Y = X / P_old` rsETH — more than the fair-value amount `X / P_actual`.
3. Attacker calls `updateRSETHPrice()`. The new price is computed over the updated TVL (which now includes the attacker's `X` ETH) and the enlarged supply. Mathematically, `new_price > P_old` because the TVL grew by `rewards + X` while supply grew by `X / P_old`. The `pricePercentageLimit` guard is not triggered for normal daily reward rates (< 0.05%).
4. Attacker calls `initiateWithdrawal()`. `expectedAssetAmount = Y * new_price` is stored in the request — greater than `X`.
5. After `withdrawalDelayBlocks` (~8 days), an operator calls `unlockQueue()`. `_calculatePayoutAmount()` returns `min(expectedAssetAmount, currentReturn)`. Since the protocol price continues to rise, `currentReturn ≥ expectedAssetAmount`, so the attacker receives the full locked-in amount.
6. Attacker calls `completeWithdrawal()` and receives `> X` ETH.

**Why existing guards fail:**

- `pricePercentageLimit`: checked against `highestRsethPrice`. For sub-0.05% daily reward rates, the price increase after the attacker's deposit is within the limit, so the public call succeeds.
- `getAvailableAssetAmount` check in `initiateWithdrawal()`: the attacker's own deposit increases `totalAssets`, so the check passes.
- `_calculatePayoutAmount` min-logic: protects against price decreases after initiation, but does not prevent locking in the inflated `expectedAssetAmount` at initiation time.
- No deposit fee exists to make the sandwich unprofitable.

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders' accrued rewards are diluted. The attacker mints rsETH at a below-fair-value rate (stale price), then redeems at the updated fair-value rate. The difference is extracted from the pool of rewards that should have accrued proportionally to all existing holders. The attack is repeatable every reward period with no special permissions.

## Likelihood Explanation

**Medium-High.** Conditions are continuously met: stETH rebases daily, EigenLayer rewards are periodic, and `rsETHPrice` is always stale between keeper updates. The attacker controls the timing of the price update (public function). The only constraint — `pricePercentageLimit` — is not triggered for normal daily reward rates. No privileged access is required; any depositor can execute this.

## Recommendation

1. **Atomically update the price on deposit:** Call `_updateRsETHPrice()` inside `depositETH()`/`depositAsset()` (or compute the live price inline in `getRsETHAmountToMint()`) so deposits always use the current price, not the stored stale value.
2. **Restrict `updateRSETHPrice()` to authorized callers** (e.g., `onlyLRTManager` or a keeper role), removing the attacker's ability to control the timing of the price update.
3. **Alternatively**, introduce a deposit fee exceeding the maximum possible single-update price increase, making the sandwich unprofitable.

## Proof of Concept

```
Assumptions:
  Pool: 100,000 rsETH outstanding, TVL = 100,000 ETH (P_old = 1.000 ETH)
  Accrued rewards: 100 ETH not yet reflected → P_actual = 1.001 ETH

Step 1 — Attacker deposits 1,000 ETH:
  Y = 1,000 / 1.000 = 1,000 rsETH minted
  (fair amount at P_actual: 1,000 / 1.001 ≈ 999.001 rsETH)

Step 2 — Attacker calls updateRSETHPrice():
  new TVL    = 100,000 + 100 (rewards) + 1,000 = 101,100 ETH
  new supply = 100,000 + 1,000 = 101,000 rsETH
  new_price  = 101,100 / 101,000 ≈ 1.000990 ETH  (> P_old, within pricePercentageLimit)

Step 3 — Attacker calls initiateWithdrawal(ETH, 1,000 rsETH):
  expectedAssetAmount = 1,000 × 1.000990 = 1,000.990 ETH  (locked in storage)

Step 4 — After 8-day delay, operator calls unlockQueue(), attacker calls completeWithdrawal():
  _calculatePayoutAmount → min(1,000.990, currentReturn) = 1,000.990 ETH
  Attacker receives 1,000.990 ETH

Net profit: ≈ 0.990 ETH extracted from existing holders' accrued yield.
Repeatable every reward period. Scales linearly with deposit size.
```

Roll forward `withdrawalDelayBlocks` blocks, have an operator call `unlockQueue()`, then call `completeWithdrawal()`. Assert `address(attacker).balance > 1000 ether`.
