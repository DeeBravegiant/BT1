### Title
`ParameterChange` Enacted Mid-Ratification-Run Silently Lowers Thresholds for Subsequent `TreasuryWithdrawals` in the Same Epoch - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

### Summary

In the Conway era `RATIFY` rule, when a `ParameterChange` governance action is ratified, the `ENACT` rule immediately mutates `ensCurPParams` inside the live `EnactState`. Because `ParameterChange` has action-priority 4 and `TreasuryWithdrawals` has priority 5, the reordered ratification signal always processes parameter changes **before** treasury withdrawals in the same epoch's ratification run. Any subsequent `TreasuryWithdrawals` action in that same run is then evaluated against the **newly lowered** voting thresholds rather than the thresholds that were in effect when voters cast their votes.

### Finding Description

`reorderActions` sorts the ratification signal by `actionPriority`:

```
ParameterChange  → priority 4
TreasuryWithdrawals → priority 5
``` [1](#0-0) 

Inside `ratifyTransition`, when a `ParameterChange` action passes `acceptedByEveryone`, the code calls `ENACT`, which applies `applyPPUpdates` to `ensCurPParams` and stores the result back into `rsEnactState`:

```haskell
ParameterChange _ ppup _ ->
  st & ensCurPParamsL %~ (`applyPPUpdates` ppup)
     & ensPrevPParamUpdateL .~ SJust (GovPurposeId govActionId)
``` [2](#0-1) 

The updated `newEnactState` is immediately placed into `st'` and forwarded to the recursive call that processes the remaining actions:

```haskell
st' = st & rsEnactStateL .~ newEnactState
         & rsDelayedL    .~ delayingAction govAction
         & rsEnactedL    %~ (Seq.:|> gas)
trans @(RATIFY era) $ TRC (env, st', RatifySignal sigs)
``` [3](#0-2) 

Crucially, `ParameterChange` is **not** a delaying action, so `rsDelayed` remains `False` and subsequent actions are not blocked:

```haskell
delayingAction ParameterChange {} = False
``` [4](#0-3) 

When the next action (`TreasuryWithdrawals`) is evaluated, `dRepAccepted` and `spoAccepted` both read their thresholds from `ratifyState ^. rsEnactStateL . ensCurPParamsL` — which is now the **post-update** PParams:

```haskell
votingDRepThreshold ratifyState =
  toRatifyVotingThreshold . votingDRepThresholdInternal pp isElectedCommittee
  where
    pp = ratifyState ^. rsEnactStateL . ensCurPParamsL   -- reads mutated PParams
``` [5](#0-4) 

```haskell
votingStakePoolThreshold ratifyState =
  toRatifyVotingThreshold . votingStakePoolThresholdInternal pp isElectedCommittee
  where
    pp = ratifyState ^. rsEnactStateL . ensCurPParamsL   -- reads mutated PParams
``` [6](#0-5) 

### Impact Explanation

An attacker who controls enough DRep stake to ratify a `ParameterChange` that touches governance-group parameters (threshold: `dvtPPGovGroup`) can, in the same epoch, also have a `TreasuryWithdrawals` action ratified using the **newly lowered** `dvtTreasuryWithdrawal` threshold. The treasury withdrawal passes at a threshold that was never in effect when the votes were cast, constituting an unauthorized treasury action. This maps to the **Critical** allowed impact: *"Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted."*

### Likelihood Explanation

The attack requires:
1. Enough DRep stake to meet `dvtPPGovGroup` (the threshold for governance-group parameter changes) — this can be lower than `dvtTreasuryWithdrawal`.
2. CC approval for both actions.
3. Both actions submitted and voted on in the same epoch so they appear together in the same ratification run.

No governance supermajority is required. If `dvtPPGovGroup < dvtTreasuryWithdrawal` (a common configuration where governance-parameter changes are easier to pass than treasury withdrawals), an attacker with stake between the two thresholds can execute the attack in a single epoch. This is a realistic, reachable scenario for any sufficiently large DRep coalition.

### Recommendation

The `EnactState` used to evaluate voting thresholds for actions within a single ratification run should be **frozen** at the start of that run (i.e., the PParams snapshot taken when the pulser was initialized at the epoch boundary). The updated `ensCurPParams` should only become visible to threshold checks in the **next** epoch's ratification run. Concretely, the `RatifyEnv` (which is fixed for the entire run) should carry the threshold-relevant PParams snapshot, and `votingDRepThreshold` / `votingStakePoolThreshold` / `votingCommitteeThreshold` should read from `reEnv` rather than from the mutable `rsEnactState`. Alternatively, `ParameterChange` could be made a delaying action (like `HardForkInitiation`) so that no other action is ratified in the same epoch.

### Proof of Concept

**Setup:**
- Current `dvtPPGovGroup` = 51%, `dvtTreasuryWithdrawal` = 67%.
- Attacker controls 60% of active DRep stake.
- CC is cooperative (or attacker controls enough CC hot keys).

**Steps:**
1. Attacker submits `ParameterChange` P1 that sets `dvtTreasuryWithdrawal` to 51%.
2. Attacker submits `TreasuryWithdrawals` T1 for a large ADA amount.
3. Attacker votes `Yes` on both P1 and T1 with their 60% DRep stake; CC votes `Yes` on both.
4. At the epoch boundary, `reorderActions` places P1 (priority 4) before T1 (priority 5).
5. `ratifyTransition` processes P1: `acceptedByEveryone` passes (60% ≥ 51%), `ENACT` fires, `ensCurPParams` is updated — `dvtTreasuryWithdrawal` is now 51%.
6. `ratifyTransition` recurses with `st'` (containing the mutated `ensCurPParams`) and processes T1.
7. `dRepAccepted` reads `dvtTreasuryWithdrawal = 51%` from the now-mutated `ensCurPParams`; 60% ≥ 51% → T1 is ratified.
8. At the epoch boundary, T1 is enacted and the treasury is drained — despite the treasury withdrawal threshold being 67% when the votes were cast.

The root cause is in `enactmentTransition` mutating `ensCurPParams` in-place: [2](#0-1) 

combined with `ratifyTransition` forwarding the mutated state to subsequent actions: [7](#0-6) 

and threshold functions reading from the mutable `ensCurPParams`: [8](#0-7)
