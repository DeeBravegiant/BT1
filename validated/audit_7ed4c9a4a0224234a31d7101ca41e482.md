### Title
Unbounded Proposal Deposit Accumulation Inflates DRep/SPO Voting Stake During Governance Voting Window — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Governance/Proposals.hs`)

---

### Summary

The Conway governance system counts governance proposal deposits as active voting stake for DReps and SPOs. Because there is no per-credential cap on the number of proposals a single staking credential may submit, an unprivileged attacker can submit arbitrarily many proposals naming the same staking credential as the return address, accumulating that credential's DRep's voting weight without limit. This is the direct analog of the Wheel contract's "unlimited stake accumulation during the cooldown window": the governance voting period is the window, proposal submissions are the repeated deposits, and epoch-boundary ratification is the winner-selection step.

---

### Finding Description

**Root cause — `proposalsDeposits`**

`proposalsDeposits` iterates over every live proposal and sums deposits by staking credential with no upper bound:

```haskell
proposalsDeposits =
  F.foldl'
    ( \gasMap gas ->
        Map.insertWith
          addCompactCoin
          (gas ^. gasReturnAddrL . accountAddressCredentialL)
          (fromMaybe (CompactCoin 0) $ toCompact $ gas ^. gasDepositL)
          gasMap
    )
    mempty
    . proposalsActions
``` [1](#0-0) 

Every proposal whose `pProcReturnAddr` staking credential is `C` adds `ppGovActionDeposit` to the map entry for `C`. There is no per-credential ceiling.

**Root cause — `computeDRepDistr`**

At each epoch boundary the `DRepPulser` snapshot (`dpProposalDeposits`) is fed into `computeDRepDistr`, which adds the accumulated deposit total directly to the DRep's voting weight:

```haskell
go (!drepAccum, !poolAccum) stakeCred accountState =
  let mInstantStake    = Map.lookup stakeCred instantStakeCredentials
      mProposalDeposit = Map.lookup stakeCred proposalDeposits
      stakeAndDeposits = fold $ mInstantStake <> mProposalDeposit
   in ( addToDRepDistr accountState stakeAndDeposits drepAccum
      , addToPoolDistr accountState mProposalDeposit poolAccum
      )
``` [2](#0-1) 

`stakeAndDeposits` is the sum of UTxO stake **and** all proposal deposits for that credential. The `addToDRepDistr` helper then adds this, plus the reward balance, to the DRep's entry in `dRepDistr`:

```haskell
updatedDistr = Map.insertWith (<>) dRep (stakeAndDeposits <> balance) distr
``` [3](#0-2) 

**Snapshot timing**

The snapshot is taken at the epoch boundary inside `setFreshDRepPulsingState`:

```haskell
dpProposalDeposits = proposalsDeposits props
``` [4](#0-3) 

Any proposals submitted before the epoch boundary are included. The attacker's window is the entire preceding epoch.

**No per-credential proposal limit**

Neither the `GOV` rule nor `proposalsAddAction` enforces a per-credential ceiling on the number of live proposals. `InfoAction` proposals require no governance-tree parent, so they can be submitted in unlimited quantity across multiple transactions.

**Ratification uses the inflated distribution**

`dRepAcceptedRatio` iterates over `reDRepDistr` (which was built by `computeDRepDistr`) to compute the yes/total ratio:

```haskell
(yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
``` [5](#0-4) 

An inflated DRep entry directly raises the yes-ratio, potentially pushing it past the ratification threshold.

---

### Impact Explanation

An attacker who controls a staking credential `C` delegated to DRep `D` (which has already voted `Yes` on a target governance action `G`) can:

1. Submit `N` `InfoAction` proposals naming `C` as the return address, locking `N × ppGovActionDeposit` ADA.
2. At the next epoch boundary, `D`'s entry in `reDRepDistr` is inflated by `N × ppGovActionDeposit`.
3. The yes-ratio for `G` rises; if it crosses the `dvt*` threshold, `G` is ratified by `RATIFY`.
4. After the proposals expire (within `ppGovActionLifetime` epochs), all deposits are returned to `C`.

Governance actions that can be ratified this way include `TreasuryWithdrawals`, `ParameterChange`, `HardForkInitiation`, `UpdateCommittee`, and `NewConstitution`. This maps directly to the allowed impact:

> **Critical. Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.**

The same mechanism applies to SPO voting stake: `addToPoolDistr` adds proposal deposits to the pool's `individualTotalPoolStake`, inflating the SPO yes-ratio for security-group parameter changes and hard-fork initiations. [6](#0-5) 

---

### Likelihood Explanation

**Entry path**: Any unprivileged transaction sender who holds ADA and has a registered staking credential delegated to a DRep. No privileged role, leaked key, or consensus majority is required.

**Cost**: The attacker must lock `N × ppGovActionDeposit` ADA for up to `ppGovActionLifetime` epochs. On current mainnet parameters (`ppGovActionDeposit = 100,000 ADA`, `ppGovActionLifetime = 6 epochs ≈ 30 days`), tipping a vote that is 1 percentage point below threshold on a 5-billion-ADA active-stake pool requires locking roughly 150 million ADA for 30 days — significant but recoverable. For a vote that is already at 65% against a 67% threshold the required lock-up is proportionally smaller.

**Observability**: The attack is visible on-chain (many `InfoAction` proposals from one credential), but by the time it is noticed the epoch boundary may have already passed and the action ratified.

**Likelihood**: **Medium** — requires a well-funded attacker and a governance vote near the threshold, both of which are realistic in a contentious governance environment.

---

### Recommendation

1. **Cap per-credential proposal deposits counted toward voting stake.** In `proposalsDeposits` (or in `computeDRepDistr`), clamp the deposit contribution of any single staking credential to at most one `ppGovActionDeposit` unit, or to the credential's own UTxO stake, whichever is larger.

2. **Alternatively, cap the number of live proposals per staking credential.** Enforce in the `GOV` rule that a single staking credential may not be the return address of more than `K` simultaneously live proposals (e.g., `K = 1` or a small constant).

3. **Separate the "skin-in-the-game" signal from voting weight.** The original design intent (PR #4309) was to ensure proposers have stake at risk; this does not require that *all* of a credential's locked deposits count as unbounded voting weight.

### Proof of Concept

```
Setup:
  - Total active DRep stake: S = 1,000,000,000 ADA
  - DRep D has voted Yes on governance action G
  - Current Yes stake: 640,000,000 ADA  (64%)
  - Ratification threshold: 67%
  - ppGovActionDeposit: 100,000 ADA
  - ppGovActionLifetime: 6 epochs

Attack:
  1. Attacker registers staking credential C, delegates to DRep D.
  2. Attacker submits 1,000 InfoAction proposals, each with return address
     pointing to C.  Cost: 100,000,000 ADA locked for ≤6 epochs.
  3. At epoch boundary, proposalsDeposits(C) = 100,000,000 ADA.
  4. computeDRepDistr adds 100,000,000 ADA to D's entry in reDRepDistr.
  5. dRepAcceptedRatio for G:
       yes   = 640,000,000 + 100,000,000 = 740,000,000
       total = 1,000,000,000 + 100,000,000 = 1,100,000,000
       ratio = 740/1100 ≈ 67.3%  > 67% threshold  ✓
  6. RATIFY enacts G (e.g., a TreasuryWithdrawals action draining the treasury).
  7. After 6 epochs, all 1,000 InfoAction proposals expire and
     100,000,000 ADA is returned to C.  Net cost to attacker: ~0 ADA
     (minus opportunity cost).
```

computeDRepDistr ::
  (EraStake era, ConwayEraAccounts era) =>
  InstantStake era ->
  Map (Credential DRepRole) DRepState ->
  Map (Credential Staking) (CompactForm Coin) ->
  PoolDistr ->
  Map DRep (CompactForm Coin) ->
  Map (Credential Staking) (AccountState era) ->
  (Map DRep (CompactForm Coin), PoolDistr)
computeDRepDistr instantStake regDReps proposalDeposits poolDistr dRepDistr =
  Map.foldlWithKey' go (dRepDistr, poolDistr)
  where
    go (!drepAccum, !poolAccum) stakeCred accountState =
      let instantStakeCredentials = instantStake ^. instantStakeCredentialsL
          mInstantStake = Map.lookup stakeCred instantStakeCredentials
          mProposalDeposit = Map.lookup stakeCred proposalDeposits
          stakeAndDeposits = fold $ mInstantStake <> mProposalDeposit
       in ( addToDRepDistr accountState stakeAndDeposits drepAccum
          , addToPoolDistr accountState mProposalDeposit poolAccum
          )
    addToPoolDistr accountState mProposalDeposit distr = fromMaybe distr $ do
      stakePool <- accountState ^. stakePoolDelegationAccountStateL
      proposalDeposit <- mProposalDeposit
      ips <- Map.lookup stakePool $ distr ^. poolDistrDistrL
      pure $
        distr
          & poolDistrDistrL %~ Map.insert stakePool (ips & individualTotalPoolStakeL <>~ proposalDeposit)
          & poolDistrTotalL %~ \t -> unsafeNonZero (unNonZero t <> fromCompact proposalDeposit)
    addToDRepDistr accountState stakeAndDeposits distr = fromMaybe distr $ do
      dRep <- accountState ^. dRepDelegationAccountStateL
      let
        balance = accountState ^. balanceAccountStateL
        updatedDistr = Map.insertWith (<>) dRep (stakeAndDeposits <> balance) distr
      Just $ case dRep of
        DRepAlwaysAbstain -> updatedDistr
        DRepAlwaysNoConfidence -> updatedDistr
        DRepCredential cred
          -- TODO: Potential optimization. Avoid this membership check, since delegation is
          -- guaranteed to exist. I believe it would also be safe for PV9, but we need to verify
          -- that it is in fact true due to #4772
          | Map.member cred regDReps -> updatedDistr
          | otherwise -> distr
```

```haskell
dRepAcceptedRatio ::
  forall era.
  RatifyEnv era ->
  Map (Credential DRepRole) Vote ->
  GovAction era ->
  Rational
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
