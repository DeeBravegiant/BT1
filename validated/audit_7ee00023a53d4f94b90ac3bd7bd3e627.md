### Title
`validCommitteeTerm` in RATIFY Omits Lower-Bound Epoch Check, Allowing Enactment of Already-Expired Committee Members — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The Conway `RATIFY` rule's `validCommitteeTerm` guard only enforces an **upper-bound** on new committee-member term lengths (≤ `currentEpoch + committeeMaxTermLength`). It never re-checks the **lower bound** — that each member's expiry epoch is still in the future at ratification time. The lower-bound check (`expiry > currentEpoch`) exists only in the `GOV` rule at proposal-submission time. Because ratification can occur many epochs after submission, a member whose term was valid when the proposal was submitted may already be expired when the proposal is enacted. This is the direct Cardano analog of the reported "wrong time reference" class: the guard uses the epoch at submission time instead of the epoch at ratification time.

---

### Finding Description

**GOV rule — lower-bound check at submission time only**

When an `UpdateCommittee` proposal is submitted, the `GOV` transition rule rejects any new member whose expiry epoch is `≤ currentEpoch` (the epoch of the submitting transaction):

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs  line 555-556
let invalidMembers = Map.filter (<= currentEpoch) membersToAdd
 in failOnNonEmptyMap invalidMembers (injectFailure . ExpirationEpochTooSmall)
``` [1](#0-0) 

This check is performed once, at proposal submission. It is **not** repeated at ratification time.

**RATIFY rule — only upper-bound check at ratification time**

`validCommitteeTerm` is the sole committee-term guard in the `RATIFY` rule:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs  lines 369-381
validCommitteeTerm govAction pp currentEpoch =
  case govAction of
    UpdateCommittee _ _ newMembers _ -> withinMaxTermLength newMembers
    _ -> True
  where
    committeeMaxTermLength = pp ^. ppCommitteeMaxTermLengthL
    withinMaxTermLength = all (<= addEpochInterval currentEpoch committeeMaxTermLength)
``` [2](#0-1) 

`withinMaxTermLength` checks only `expiry ≤ currentEpoch + committeeMaxTermLength`. When `expiry < currentEpoch` (already expired), this inequality is trivially satisfied for any non-negative `committeeMaxTermLength`. There is **no** `expiry > currentEpoch` guard here.

It is called in `ratifyTransition` with `reCurrentEpoch` — the epoch at ratification time:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs  line 338
&& validCommitteeTerm govAction ensCurPParams reCurrentEpoch
``` [3](#0-2) 

**Downstream effect — expired members silently reduce active committee size**

Once enacted, an already-expired member is immediately treated as abstaining in `committeeAcceptedRatio`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs  line 152
| currentEpoch > expiry = (yes, tot) -- member is expired, vote "abstain"
``` [4](#0-3) 

And excluded from `activeCommitteeSize` in `votingCommitteeThresholdInternal`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Governance/Internal.hs  lines 472-479
isActive coldKey validUntil =
  case Map.lookup coldKey hotKeys of
    Just (CommitteeMemberResigned _) -> False
    Just _ -> currentEpoch <= validUntil
    Nothing -> False
activeCommitteeSize =
  fromIntegral . Map.size . Map.filterWithKey isActive $
    foldMap' committeeMembers committee
``` [5](#0-4) 

If `activeCommitteeSize < ppCommitteeMinSizeL`, the threshold collapses to `NoVotingThreshold`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Governance/Internal.hs  lines 461-470
threshold =
  case committeeThreshold <$> committee of
    SJust t
      | hardforkConwayBootstrapPhase (pp ^. ppProtocolVersionL)
          || activeCommitteeSize >= minSize ->
          VotingThreshold t
    _ -> NoVotingThreshold
``` [6](#0-5) 

`NoVotingThreshold` means no committee vote is required for subsequent governance actions to be ratified.

---

### Impact Explanation

An `UpdateCommittee` proposal submitted in epoch S with new members whose expiry is E (where E > S, satisfying the GOV check) can be ratified in epoch R > E. At that point `validCommitteeTerm` trivially passes (`E ≤ R + committeeMaxTermLength` is always true), and the action is enacted with already-expired committee members. If this causes `activeCommitteeSize < ppCommitteeMinSizeL`, the committee threshold becomes `NoVotingThreshold` for all subsequent governance actions — meaning hard-fork initiations, parameter changes, treasury withdrawals, and constitution updates can be ratified without any committee approval. This matches the **Critical** impact category: unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork actions could be enacted.

---

### Likelihood Explanation

The scenario requires: (1) a proposer to submit an `UpdateCommittee` action with short-term members, (2) the current committee to vote yes (negligently or collusively), and (3) the proposal to remain unratified until after those terms expire. Governance proposals routinely sit in the queue for multiple epochs awaiting sufficient DRep/SPO votes. A proposer who controls or colludes with a quorum of the current committee can deliberately engineer this. Even without collusion, an honest committee that votes yes on a proposal with near-future expiry epochs may inadvertently trigger this if ratification is delayed by a `rsDelayed` flag (from a concurrent delaying action extending all proposal lifetimes by one epoch). [7](#0-6) 

---

### Recommendation

Add a lower-bound check inside `validCommitteeTerm` using the ratification-time epoch:

```haskell
validCommitteeTerm govAction pp currentEpoch =
  case govAction of
    UpdateCommittee _ _ newMembers _ ->
      withinMaxTermLength newMembers && allFutureExpiry newMembers
    _ -> True
  where
    committeeMaxTermLength = pp ^. ppCommitteeMaxTermLengthL
    withinMaxTermLength = all (<= addEpochInterval currentEpoch committeeMaxTermLength)
    allFutureExpiry   = all (> currentEpoch)   -- NEW: expiry must still be in the future
```

This mirrors the existing `ExpirationEpochTooSmall` guard in the `GOV` rule but applies it at ratification time, closing the window between submission and enactment.

---

### Proof of Concept

1. Set `ppCommitteeMaxTermLengthL = EpochInterval 10`, `ppGovActionLifetimeL = EpochInterval 5`, `ppCommitteeMinSizeL = 1`.
2. **Epoch 0**: Submit `UpdateCommittee SNothing mempty (Map.singleton coldCred (EpochNo 2)) threshold`. GOV check: `2 > 0` ✓ — proposal accepted.
3. **Epoch 1**: DReps, SPOs, and current committee cast yes votes.
4. **Epoch 3** (boundary): RATIFY processes the proposal. `validCommitteeTerm` evaluates `2 ≤ 3 + 10 = 13` ✓. No lower-bound check. `acceptedByEveryone` passes. Proposal is enacted.
5. At enactment (epoch 3): `currentEpoch > expiry` → `3 > 2` → True. The enacted member is **immediately expired**.
6. `activeCommitteeSize = 0 < ppCommitteeMinSizeL = 1` → `votingCommitteeThresholdInternal` returns `NoVotingThreshold`.
7. **Epoch 4 onward**: Any governance action (hard fork, parameter change, treasury withdrawal) can be ratified with only DRep + SPO approval — no committee vote required.
