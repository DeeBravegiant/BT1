Audit Report

## Title
Users Can Front-Run rsETH Price Decreases via `instantWithdrawal` to Avoid Losses, Shifting Burden to Honest Stakers - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.instantWithdrawal()` redeems rsETH for underlying assets using the stored `rsETHPrice` from `LRTOracle`, which is a state variable updated only when `updateRSETHPrice()` is explicitly called. Because `updateRSETHPrice()` is permissionless and not invoked atomically within `instantWithdrawal`, an attacker can redeem rsETH at a stale (pre-slash) price after EigenLayer has already reflected a validator balance decrease, extracting more assets than the current backing warrants and shifting the shortfall onto remaining rsETH holders.

## Finding Description

`instantWithdrawal` computes the payout via `getExpectedAssetAmount`:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` is a stored state variable in `LRTOracle` that is only updated when `updateRSETHPrice()` is explicitly called:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [2](#0-1) 

`_updateRsETHPrice()` computes the new price by calling `_getTotalEthInProtocol()`, which for ETH reads `getEffectivePodShares()` on each `NodeDelegator`:

```solidity
ethStakedInEigenLayer += INodeDelegator(nodeDelegatorQueue[i]).getEffectivePodShares();
``` [3](#0-2) 

`getEffectivePodShares()` returns `stakedButUnverifiedNativeETH + withdrawableShare`, where `withdrawableShare` is read live from EigenLayer's beacon chain ETH strategy: [4](#0-3) 

After a beacon chain slashing event is processed by EigenLayer (via checkpoint proofs), `withdrawableShare` decreases immediately. However, `rsETHPrice` in `LRTOracle` remains at its last stored value until `updateRSETHPrice()` is called. This creates a staleness window.

The exploit path:
1. A slashing event occurs for a Kelp validator; EigenLayer processes the checkpoint proof, reducing `withdrawableShare`.
2. `rsETHPrice` in `LRTOracle` is still stale (pre-slash value).
3. Attacker calls `instantWithdrawal(asset, rsETHUnstaked, "")` — payout is computed at the inflated stale price.
4. Attacker (or anyone) calls `LRTOracle.updateRSETHPrice()` — price drops to reflect the slash.

The `instantWithdrawal` function is accessible to any rsETH holder when `isInstantWithdrawalEnabled[asset]` is `true`: [5](#0-4) 

The downside protection in `_updateRsETHPrice()` — which pauses the protocol if the price drop exceeds `pricePercentageLimit` — does not prevent this attack because the attacker exits at step 3 before triggering the price update at step 4: [6](#0-5) 

No existing check inside `instantWithdrawal` calls `updateRSETHPrice()` or otherwise ensures the stored price is fresh before computing the payout.

## Impact Explanation

The attacker redeems rsETH at a price higher than the post-slash true value, receiving more underlying assets (ETH or LSTs) than they are entitled to. The shortfall is absorbed by remaining rsETH holders, whose tokens are now backed by fewer assets than the protocol's accounting reflects. This constitutes **theft of unclaimed yield** (High) at the direct expense of honest stakers, matching the allowed impact scope.

## Likelihood Explanation

- Beacon chain slashing events are publicly observable via the beacon chain API and on-chain events.
- EigenLayer checkpoint proofs are permissionless; the attacker can trigger the on-chain balance update themselves.
- `updateRSETHPrice()` is permissionless, so the attacker controls the timing of the price update relative to their exit.
- `instantWithdrawal` requires only that the asset has `isInstantWithdrawalEnabled[asset] == true` and that the attacker holds rsETH — no special privileges.
- The attack is repeatable for every slashing event during which instant withdrawal is enabled.

Likelihood: **Medium** — requires a slashing event and instant withdrawal being enabled, but the attacker controls both the exit and the price update timing.

## Recommendation

1. **Call `updateRSETHPrice()` at the start of `instantWithdrawal`** to ensure the payout is computed against the current price before any rsETH is burned or assets are transferred.
2. Alternatively, introduce a **price freshness check** (e.g., a maximum age for `rsETHPrice`) and revert if the stored price is stale.
3. Consider restricting `updateRSETHPrice()` to authorized callers (e.g., a keeper/manager role) to prevent attackers from controlling the timing of price updates.

## Proof of Concept

```
Block N:   Slashing event processed by EigenLayer via checkpoint proof.
           NodeDelegator.getEffectivePodShares() now returns a lower value.
           LRTOracle.rsETHPrice = 1.05e18 (stale, pre-slash value).

Block N:   Attacker calls LRTWithdrawalManager.instantWithdrawal(ETH, 100e18, "")
           getExpectedAssetAmount: 100e18 * 1.05e18 / 1e18 = 105 ETH paid out.
           rsETH burned, attacker receives 105 ETH (minus fee).

Block N:   Attacker calls LRTOracle.updateRSETHPrice()
           _getTotalEthInProtocol() reads reduced getEffectivePodShares()
           rsETHPrice drops to 1.03e18.

Result:    Attacker received 105 ETH for 100 rsETH.
           True value at new price: 100 * 1.03 = 103 ETH.
           Attacker extracted 2 ETH excess.
           Remaining rsETH holders absorb the 2 ETH shortfall.
```

Foundry fork test plan: fork mainnet, simulate a validator balance decrease by mocking `NodeDelegatorHelper.getWithdrawableShare()` to return a reduced value (post-slash), confirm `rsETHPrice` is still at the pre-slash value, call `instantWithdrawal`, then call `updateRSETHPrice()` and assert the price dropped — demonstrating the attacker received more ETH than the post-update price would have entitled them to.
