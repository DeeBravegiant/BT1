Audit Report

## Title
Nested Unbounded Loop in `NodeDelegator.getAssetUnstaking()` Called Per-Asset Per-NDC Causes Gas Exhaustion in Deposits and Price Updates - (File: contracts/NodeDelegator.sol)

## Summary
`NodeDelegator.getAssetUnstaking()` performs a nested loop over all EigenLayer-queued withdrawals and their strategies, making multiple external calls per iteration. This function is invoked once per NDC for every supported asset inside `getAssetDistributionData()` and `getETHDistributionData()`, which are transitively called by every user deposit and by the publicly accessible `updateRSETHPrice()`. As queued withdrawals, NDCs, or supported assets grow — particularly through EigenLayer-forced undelegations that bypass the protocol's own counter — cumulative gas consumption can exceed the block gas limit, temporarily freezing deposits and price updates.

## Finding Description
`NodeDelegator.getAssetUnstaking()` (L405–427) fetches all queued withdrawals from EigenLayer's `DelegationManager` and iterates over them in a nested loop, calling `strategy.underlyingToken()` and `strategy.sharesToUnderlyingView()` per strategy per withdrawal:

```solidity
// contracts/NodeDelegator.sol L406-426
(IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
    _getDelegationManager().getQueuedWithdrawals(address(this));

for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
    for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
        address strategyAsset = ... strategy.underlyingToken() ...;
        amount += ... strategy.sharesToUnderlyingView(sharesToUnstake);
    }
}
```

This function is called once per NDC in `getAssetDistributionData()` (L451) and once per NDC in `getETHDistributionData()` (L488–489). `getTotalAssetDeposits()` (L385–397) calls `getAssetDistributionData()`. `LRTOracle._getTotalEthInProtocol()` (L336–348) calls `getTotalAssetDeposits()` for **every** supported asset in a loop. `updateRSETHPrice()` (L87) is publicly callable with no access restriction. Every user deposit also triggers `_checkIfDepositAmountExceedesCurrentLimit()` (L677) → `getTotalAssetDeposits()`.

The protocol's partial mitigation is a cap of 80 on `maxUncompletedWithdrawalCount` (L153 in `LRTUnstakingVault.sol`), with the comment explicitly acknowledging "120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price." However, this counter only tracks protocol-initiated withdrawals. EigenLayer operators can force-undelegate NDCs, creating additional queued withdrawals that bypass this counter entirely. The protocol acknowledges this gap with the `setUncompletedWithdrawalCount()` admin function (L164–180), which is a manual resync — not a real-time guard. Furthermore, the 120-withdrawal threshold was computed for a fixed number of NDCs and assets; adding more supported assets or raising `maxNodeDelegatorLimit` lowers the effective threshold proportionally.

## Impact Explanation
This matches **Medium. Unbounded gas consumption** and **Medium. Temporary freezing of funds**. If `depositETH()` or `depositAsset()` reverts out of gas, users cannot deposit. If `updateRSETHPrice()` reverts out of gas, the rsETH exchange rate becomes stale, breaking fee accrual and withdrawal pricing. There is no admin escape hatch that does not require removing NDCs or assets.

## Likelihood Explanation
The protocol already operates with multiple supported assets and up to 10 NDCs. EigenLayer operators can force-undelegate at any time, pushing the real queued-withdrawal count above the protocol's 80-withdrawal cap without any on-chain enforcement preventing it. With 6 assets, 10 NDCs, and 90 queued withdrawals (80 protocol + 10 forced), the inner loop executes approximately 6 × 10 × (1 external `getQueuedWithdrawals` + 90×2 strategy calls) external calls per `updateRSETHPrice()` invocation. This is triggerable by any external account calling `updateRSETHPrice()` or by any user attempting a deposit.

## Recommendation
1. Cache the result of `getQueuedWithdrawals()` per NDC across all asset queries within a single transaction, rather than re-fetching and re-iterating EigenLayer's queued withdrawal list once per asset per NDC.
2. Store a running `assetUnstaking` tally updated incrementally on each `initiateUnstaking` / `completeUnstaking` event, eliminating the need to iterate EigenLayer's queue on every read.
3. Enforce a hard on-chain cap on the total queued withdrawal count across all NDCs before allowing new unstaking operations, accounting for forced undelegations.

## Proof of Concept
**Setup:** 6 supported assets, 10 NDCs, 80 protocol-initiated queued withdrawals (8 per NDC, 2 strategies each), plus 10 EigenLayer-forced undelegations (1 per NDC).

**Call sequence:**
1. Any external account calls `LRTOracle.updateRSETHPrice()` (no access restriction).
2. `_updateRsETHPrice()` → `_getTotalEthInProtocol()` loops over 6 assets.
3. Per asset: `getTotalAssetDeposits()` → `getAssetDistributionData()` loops over 10 NDCs.
4. Per NDC: `getAssetUnstaking()` calls `DelegationManager.getQueuedWithdrawals()` (1 external call returning 9 withdrawals) then loops 9 × 2 = 18 strategy external calls.
5. Total external calls: 6 × 10 × (1 + 18) = **1,140 external calls** at ≥2,100 gas cold each = ~2.4M gas for external calls alone, plus ABI decoding of large withdrawal arrays, loop overhead, and storage reads.
6. A Foundry fork test against mainnet EigenLayer with 10 NDCs each holding 9 queued withdrawals would demonstrate the `updateRSETHPrice()` call reverting out of gas when the block gas limit is approached, and similarly for `depositAsset()` via `_checkIfDepositAmountExceedesCurrentLimit()`.
