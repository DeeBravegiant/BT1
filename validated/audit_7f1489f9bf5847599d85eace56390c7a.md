### Title
`valid_proposals` Poisoned with Rejected Proposal Before `ProposalFinMismatch` Check in `validate_proposal` — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

In `validate_proposal()`, the proposal is unconditionally inserted into the `valid_proposals` shared cache **before** the final commitment equality check (`built_block != received_fin.proposal_commitment`). When that check fails and `ProposalFinMismatch` is returned, the state mutation is never rolled back. The `(height, round)` slot in `valid_proposals` is permanently occupied by a rejected proposal, and the `fin_sender` is silently dropped without notifying consensus.

---

### Finding Description

`validate_proposal` in `crates/apollo_consensus_orchestrator/src/validate_proposal.rs` (lines 238–247) performs a state mutation before its final validity check:

```rust
// Update valid_proposals before sending fin to avoid a race condition
// with `repropose` being called before `valid_proposals` is updated.
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

// TODO(matan): Switch to signature validation.
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}
```

`insert_proposal` (lines 180–204 of `sequencer_consensus_context.rs`) stores the proposal keyed by `(height, round)` using the **batcher-computed** commitment derived from `finished_info.proposal_commitment.partial_block_hash` and `init.fee_proposal_fri`. It also contains a hard assertion:

```rust
assert!(
    previous.is_none(),
    "Overwriting existing proposal for height {height} round {round}; at most one \
     proposal per (height, round) is allowed"
);
```

When `built_block != received_fin.proposal_commitment` (i.e., the proposer's claimed commitment in `ProposalFin` does not match the batcher's locally-computed commitment), the function returns `Err(ProposalFinMismatch)`. The `fin_sender` inside `validate_and_send` is dropped without calling `.send()`, so the consensus state machine never receives a commitment to vote on. However, the entry for `(height, round)` is already written into `valid_proposals` with the batcher's commitment and is **not removed**.

The comment acknowledges the intentional ordering: the insert is placed before the check to prevent a race with `repropose`. But this trades one race for a broken invariant: a proposal that fails the final commitment check is permanently recorded as a valid candidate.

---

### Impact Explanation

**Broken invariant — `valid_proposals` slot poisoned:**
The `(height, round)` entry in `valid_proposals` is now occupied by a proposal whose `ProposalFin` commitment was rejected. Any subsequent attempt to insert a proposal for the same `(height, round)` — whether from a retry, a `build_proposal` call, or `update_for_reproposal` — will hit the `assert!(previous.is_none(), ...)` and **panic**, crashing the node process.

**`decision_reached` path:**
`decision_reached` (line 1002 of `sequencer_consensus_context.rs`) calls `proposals.get_proposal(&height, &round, &commitment)`, which asserts that the stored commitment equals the requested one. If consensus on the batcher's commitment is reached by 2/3+ peers (possible when a malicious proposer sends different `ProposalFin` values to different validators), the local node would find the orphaned entry and proceed to commit the block — without having voted for it. The `fin_sender` was never called, so the local node's consensus state machine is out of sync with the committed state.

**Scope match:**
This is in the proposal validation path (`validate_proposal` → `handle_proposal_part` → `ProposalFin` comparison), directly binding `ProposalCommitment` and the `valid_proposals` cache used by `decision_reached` to finalize blocks.

---

### Likelihood Explanation

Any network peer acting as proposer can send a `ProposalFin` whose `proposal_commitment` field does not match the batcher's locally-computed value. This is an unprivileged, unauthenticated wire-level input. The `ProposalFin.proposal_commitment` field is received directly from the network and compared against the local batcher result at line 244. No special privilege is required to craft a mismatched value.

The panic path additionally requires the same `(height, round)` to be re-entered, which is an edge case in normal consensus but becomes reachable if the consensus manager retries validation for the same round (e.g., after a timeout or a round-change edge case).

---

### Recommendation

Move `insert_proposal` to **after** the commitment check, so the cache is only updated when the proposal is confirmed valid:

```rust
// TODO(matan): Switch to signature validation.
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

// Only insert after the commitment is verified.
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);
```

To preserve the race-condition protection noted in the comment (preventing `repropose` from running before `valid_proposals` is updated), the lock should be held across both the check and the insert, and `fin_sender.send` should be called while the lock is still held or immediately after releasing it — ensuring `valid_proposals` is always populated before the commitment is broadcast to the consensus state machine.

---

### Proof of Concept

1. A proposer (malicious or buggy) streams a valid sequence of `ProposalPart::Transactions` batches to the validator node.
2. The proposer sends `ProposalPart::Fin` with `proposal_commitment = Y`, where `Y` differs from the batcher's locally-computed commitment `X`.
3. Inside `handle_proposal_part` (line 582–596), `batcher_block_commitment` is computed as `X`; `fin.proposal_commitment` is `Y`. `HandledProposalPart::Finished(X, fin_with_Y, finished_info)` is returned.
4. Back in `validate_proposal` (line 241), `insert_proposal` is called with `finished_info` (which encodes `X`). The entry `(height, round) → X` is written into `valid_proposals`.
5. Line 244 evaluates `X != Y` → true. `CONSENSUS_PROPOSAL_FIN_MISMATCH` is incremented and `Err(ProposalFinMismatch)` is returned.
6. In `validate_and_send`, the `?` operator propagates the error; `fin_sender` is dropped without `.send()`. The consensus state machine receives no commitment.
7. The `(height, round)` slot in `valid_proposals` is now occupied by the orphaned entry with commitment `X`.
8. If any code path subsequently calls `insert_proposal` for the same `(height, round)` (e.g., a round retry or a `build_proposal` call for the same round), the `assert!(previous.is_none(), ...)` at line 199 of `sequencer_consensus_context.rs` fires, panicking the node.
