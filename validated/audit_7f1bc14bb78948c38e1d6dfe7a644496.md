### Title
Single Participant Can Block All Foreign-Chain Verification Requests via Strict-Intersection Gate — (File: `crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which requires **every** active participant to have registered a chain. A single participant registering an empty list silently drops every chain from the supported set, causing all foreign-chain verification requests to be rejected. The threshold-aware `get_available_foreign_chains()` method exists and is correct, but is never consulted by the signing path.

### Finding Description

`verify_foreign_transaction` checks whether the requested chain is in the supported set:

```rust
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(...)
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the **strict intersection** of every active participant's registered chains — a chain is included only if `nodes_supporting_chain.is_superset(&active_participant_account_ids)`: [2](#0-1) 

The design documentation explicitly identifies this as a defect: *"A single node that registers an empty list (or hasn't registered yet) drops every chain — one operator can take the whole feature down."*



The correct replacement — `get_available_foreign_chains()` — is fully implemented. It counts per-chain supporters among active participants and requires only `≥ signing_threshold` to cover a chain: [3](#0-2) 

The design specification states: *"`verify_foreign_transaction(C)` is rejected unless `C` is available"* (threshold-based), not unless `C` is unanimously supported. [4](#0-3) 

Despite `get_available_foreign_chains()` being implemented and tested, `verify_foreign_transaction` was never updated to call it. [5](#0-4) 

### Impact Explanation

Any single active participant — even one whose defection cannot break the signing threshold — can call `register_foreign_chain_support` (or its deprecated alias `register_foreign_chain_config`) with an empty chain set. Because `get_supported_foreign_chains()` requires unanimous coverage, this immediately empties the supported-chain set. Every subsequent `verify_foreign_transaction` call panics with `ForeignChainNotSupported`, freezing the entire foreign-chain verification flow until the participant re-registers. This breaks the request-lifecycle invariant that a chain servable by ≥ threshold nodes should be accepted.

### Likelihood Explanation

The attack requires only a single participant account — no threshold collusion, no privileged key, no external dependency. The call (`register_foreign_chain_support` / `register_available_foreign_chain_config`) is a normal participant-facing contract method reachable by any active node operator. A Byzantine participant strictly below the signing threshold can execute this unilaterally and repeatedly.

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which already implements the correct threshold-based gate. The migration path is documented in the design doc and the method is fully implemented and tested.

```rust
// Before (strict intersection — one non-registering node drops all chains):
let supported_chains = self.get_supported_foreign_chains();

// After (threshold-based — matches the design specification):
let supported_chains = self.get_available_foreign_chains();
``` [1](#0-0) 

### Proof of Concept

1. Deploy the contract in `Running` state with 4 participants and signing threshold 3.
2. Participants 0, 1, 2 call `register_available_foreign_chain_config` registering `Bitcoin`. `get_available_foreign_chains()` returns `{Bitcoin}` (3 ≥ threshold).
3. Participant 3 (the Byzantine node, below threshold) calls `register_foreign_chain_support` with an empty set `{}`.
4. `get_supported_foreign_chains()` now returns `{}` because participant 3 is not a superset of `{Bitcoin}`.
5. Any user calling `verify_foreign_transaction` for Bitcoin receives `ForeignChainNotSupported` and the request is rejected — despite 3 of 4 nodes being fully capable of serving it. [6](#0-5) [7](#0-6)

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }

    pub(crate) fn update_available_chains_config_cache(
        &mut self,
        active_tls_keys: &BTreeSet<dtos::Ed25519PublicKey>,
        threshold: u64,
    ) {
        let mut chain_to_supporter_count: std::collections::BTreeMap<dtos::ForeignChain, u64> =
            std::collections::BTreeMap::new();
        for tls_key in active_tls_keys {
            let Some(chains) = self.foreign_chains_configs.get(tls_key) else {
                continue;
            };
            for chain in chains.iter() {
                if self.rpc_whitelist.entries.is_whitelisted(chain) {
                    let count = chain_to_supporter_count.entry(*chain).or_default();
                    *count = count
                        .checked_add(1)
                        .expect("supporter count bounded by participant set size");
                }
            }
        }
        self.available_foreign_chains = chain_to_supporter_count
            .into_iter()
            .filter_map(|(chain, count)| (count >= threshold).then_some(chain))
            .collect::<BTreeSet<_>>()
            .into();
    }
