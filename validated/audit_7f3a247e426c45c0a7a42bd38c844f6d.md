Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks `initiateWithdrawal()` Until Admin Intervenes — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal()` checks `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, which evaluates to `0 >= 0` (always `true`) for every caller, causing an unconditional revert with `WithdrawalLimitReached`. All stakers are blocked from unstaking until an admin explicitly calls `setMaxNumberOfWithdrawalsPerUser()`.

## Finding Description
`initialize()` sets `kernelToken`, `rewardsToken`, and roles, but never assigns `maxNumberOfWithdrawalsPerUser`: [1](#0-0) 

`maxNumberOfWithdrawalsPerUser` therefore remains `0` after deployment.

`initiateWithdrawal()` contains: [2](#0-1) 

Because `userWithdrawalIds[msg.sender].length` is a `uint256` (always `>= 0`) and `maxNumberOfWithdrawalsPerUser == 0`, the condition `length >= 0` is always `true`. Every call reverts with `WithdrawalLimitReached` regardless of how many open withdrawals the user has.

The setter explicitly forbids restoring the zero value, so the only escape is an admin action: [3](#0-2) 

Until `setMaxNumberOfWithdrawalsPerUser()` is called with a valid non-zero value, the entire withdrawal path is closed.

## Impact Explanation
Any user who has called `stake()` or `stakeFor()` and wishes to recover their KERNEL tokens is blocked. Their tokens are held in the contract with no exit path available to them. This constitutes a **temporary freezing of funds** (Medium severity) for all stakers, lasting until an admin calls `setMaxNumberOfWithdrawalsPerUser()`.

## Likelihood Explanation
The contract is deployable and fully usable for staking without the admin ever noticing the missing initialization, because `stake()` has no dependency on `maxNumberOfWithdrawalsPerUser`. A user who stakes immediately after deployment and then tries to withdraw will be blocked. The missing initialization is not enforced or documented in `initialize()`, making it a realistic oversight. Any unprivileged staker can trigger the blocked state simply by calling `initiateWithdrawal()`.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
```

This mirrors the pattern used for `withdrawalDelayBlocks` in `LRTWithdrawalManager.initialize()`. [1](#0-0) 

## Proof of Concept
1. Deploy `KernelDepositPool` (admin never calls `setMaxNumberOfWithdrawalsPerUser`).
2. Alice calls `stake(100e18)` — succeeds; `balanceOf[Alice] = 100e18`.
3. Alice calls `initiateWithdrawal(100e18)`.
4. Execution reaches:
   ```solidity
   if (userWithdrawalIds[Alice].length >= maxNumberOfWithdrawalsPerUser)
   //  0 >= 0  → true
       revert WithdrawalLimitReached();
   ```
5. Transaction reverts. Alice's 100 KERNEL tokens are locked in the contract with no recourse until an admin calls `setMaxNumberOfWithdrawalsPerUser(n)` where `1 <= n <= 100`.

**Foundry test plan:**
```solidity
function test_initiateWithdrawal_blockedWhenUninitialized() public {
    // Deploy proxy + initialize (no setMaxNumberOfWithdrawalsPerUser call)

}

    function initiateWithdrawal(uint256 _amount) external nonReentrant updateReward(msg.sender) {
        if (_amount == 0) revert AmountZero();
        if (balanceOf[msg.sender] < _amount) revert InsufficientStakedBalance();
        if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();

        balanceOf[msg.sender] -= _amount;
        totalKernelStaked -= _amount;

        // Create a withdrawal record
        uint256 withdrawalId = ++withdrawalCounter;
        uint256 unlockTime = block.timestamp + withdrawalDelay;

        withdrawals[withdrawalId] = Withdrawal({
            user: msg.sender, amount: _amount, unlockTime: unlockTime, claimed: false, withdrawalId: withdrawalId
        });
        userWithdrawalIds[msg.sender].push(withdrawalId);

        emit WithdrawalInitiated(msg.sender, _amount, withdrawalId, unlockTime);
    }
```
