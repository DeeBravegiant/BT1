### Title
Single-Participant Forged CKD Output for `AppPublicKey` Variant — No Threshold Enforcement, No On-Chain Verification (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd()` enforces no cryptographic validity check on the `CKDResponse` when the request uses the `AppPublicKey` variant. A single Byzantine attested participant can call `respond_ckd()` with arbitrary `big_y` / `big_c` values and the contract will accept and deliver the forged output to all queued callers. The crash-recovery window makes this race reliably winnable, but the vulnerability is present in normal operation too.

---

### Finding Description

In `respond_ckd()`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract runs a BLS12-381 pairing check (`ckd_output_check`) that cryptographically binds the response to the network public key and the app's public key pair. [2](#0-1) 

For `AppPublicKey`, the branch is an empty no-op. The only guards that remain are:

1. `assert_caller_is_attested_participant_and_protocol_active()` — verifies the caller is a **single** attested participant.
2. Protocol state is `Running` or `Resharing`.
3. `accept_requests` flag is set. [3](#0-2) 

There is no threshold-vote mechanism for `respond_ckd()`. One attested participant calling it is sufficient to drain the entire fan-out queue:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [4](#0-3) 

`resolve_yields_for` removes the map entry and resumes every queued yield with the attacker-supplied bytes in one pass. [5](#0-4) 

The contract-level `CKDRequest` key is fully deterministic — derived from `app_public_key`, `domain_id`, `predecessor_id`, and `derivation_path` with no entropy or nonce: [6](#0-5) 

All four fields are visible in the `request_app_private_key` transaction on-chain, so the Byzantine participant can reconstruct the exact map key without any privileged access.

---

### Impact Explanation

**Attacker-controlled CKD output.** The attacker supplies arbitrary `big_y` and `big_c`. The caller computes the derived key as `big_c − app_sk · big_y`. If the attacker sets `big_y = G1_identity` (the identity point — the contract performs no identity-point check for `AppPublicKey`) and `big_c = k·G1` for a scalar `k` of their choice, the caller derives `k·G1`, a value the attacker knows. The attacker can then reproduce the same derived key and decrypt any data the caller encrypts with it.

Even without the identity-point trick, the attacker delivers a garbage key that the caller cannot distinguish from a correct one until they attempt to use it, at which point the request has already been consumed and cannot be retried.

All callers who submitted duplicate `request_app_private_key` calls for the same key (up to the 128-slot fan-out cap) receive the same forged output simultaneously. [7](#0-6) 

---

### Likelihood Explanation

**Without a crash:** The Byzantine participant monitors the NEAR blockchain for `request_app_private_key` transactions. The off-chain CKD protocol requires at least one full round-trip among all participants before the honest coordinator can call `respond_ckd()`. The Byzantine participant needs only to submit a single transaction. In practice this is a tight race, but it is structurally possible.

**With a crash (the scenario in the question):** When the coordinator or enough participants crash mid-protocol, the honest nodes must detect the failure, elect a new coordinator, and restart the protocol. During this window — which can span many NEAR blocks — the Byzantine participant can call `respond_ckd()` unopposed. The `CKDRequestStorage` in RocksDB ensures the request survives the restart and the honest nodes will eventually re-initiate, but the contract has no mechanism to reject a response that arrives before the restarted protocol completes. [8](#0-7) 

The attacker is a legitimate attested participant (below the signing threshold), so the `assert_caller_is_attested_participant_and_protocol_active()` check passes. [9](#0-8) 

---

### Recommendation

1. **Deprecate and remove `AppPublicKey` support** in `respond_ckd()`. All new callers should use `AppPublicKeyPV`, which has an on-chain pairing check that a single Byzantine participant cannot forge.
2. **If `AppPublicKey` must be retained**, require a threshold of distinct attested participants to each submit a matching `respond_ckd()` call (analogous to a vote) before the yields are resolved, so that forging requires threshold-level collusion.
3. **Add an identity-point guard** for `AppPublicKey` responses (reject `big_y == G1_identity`) to close the trivial key-control path even before a threshold mechanism is added.

---

### Proof of Concept

```rust
// 1. User submits request_app_private_key with AppPublicKey variant.
//    The CKDRequest key is deterministic and readable from the tx receipt.

// 2. Byzantine participant (attested, below threshold) reconstructs the key:
let ckd_request = CKDRequest::new(
    CKDAppPublicKey::AppPublicKey(user_app_pk),
    domain_id,
    &user_account_id,
    &derivation_path,
);

// 3. Attacker chooses big_y = identity, big_c = k·G1 for known k.
//    The caller will derive big_c − app_sk · identity = big_c = k·G1.
let forged_response = CKDResponse {
    big_y: Bls12381G1PublicKey(G1Projective::identity().to_compressed()),
    big_c: Bls12381G1PublicKey((G1Projective::generator() * k).to_compressed()),
};

// 4. Byzantine participant calls respond_ckd() — passes all contract checks
//    for AppPublicKey variant, no pairing check is run.
contract.respond_ckd(ckd_request, forged_response);
// → resolve_yields_for drains all queued yields with attacker-controlled bytes.
// → Caller derives k·G1, which the attacker already knows.
```

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` already demonstrates that `[1u8; 48]` / `[2u8; 48]` (invalid curve points) are accepted without error for `AppPublicKey` requests, confirming zero response validation. [10](#0-9)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
