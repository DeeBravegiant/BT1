### Title
Unbounded Iteration Over `weight_proof.sub_epochs` in `_validate_sub_epoch_summaries` Blocks the Main Asyncio Event Loop During Sync - (File: `chia/full_node/weight_proof.py`)

### Summary
`_validate_sub_epoch_summaries` iterates over the attacker-controlled `weight_proof.sub_epochs` list via `_map_sub_epoch_summaries` with no upper-bound check on list length. A malicious peer can send a crafted `WeightProof` with an arbitrarily large `sub_epochs` list, blocking the full node's (and wallet's) main asyncio event loop for an extended period and preventing the node from processing blocks, transactions, or sync updates.

### Finding Description

During sync, a full node requests a `WeightProof` from a peer via `request_validate_wp` in `chia/full_node/full_node.py`. Before calling `validate_weight_proof`, only two lightweight checks are performed on the received proof: [1](#0-0) 

These checks verify only that `recent_chain_data[-1]` has the correct height and weight. They impose **no constraint on the length of `weight_proof.sub_epochs`**.

`validate_weight_proof` then calls `_validate_sub_epoch_summaries` **synchronously in the main asyncio thread**: [2](#0-1) 

`_validate_sub_epoch_summaries` immediately delegates to `_map_sub_epoch_summaries`, which iterates over every element of `weight_proof.sub_epochs` with no size guard: [3](#0-2) [4](#0-3) 

The loop at line 890 runs for every `SubEpochData` entry, computing a `SubEpochSummary` and hashing it (`std_hash(ses)`) for each. There is no check like `if len(sub_epoch_data) > MAX_ALLOWED` before entering the loop.

By contrast, `_validate_sub_epoch_segments` — which runs later in a subprocess — **does** enforce a per-sub-epoch segment cap: [5](#0-4) 

No equivalent guard exists for `sub_epochs`.

The same vulnerability exists in the wallet's weight proof handler: [6](#0-5) 

The `WeightProof` type itself places no structural limit on `sub_epochs`: [7](#0-6) 

### Impact Explanation

Blocking the main asyncio event loop prevents the node from processing any concurrent messages: block announcements, transaction propagation, peer handshakes, and all other protocol messages. During the blocked period the node cannot advance its chain tip, respond to farmers, or complete sync. An attacker with multiple Sybil peers can sustain the block indefinitely by repeatedly sending crafted proofs (each peer is banned only after validation completes, but the blocking already occurred). This satisfies the High impact criterion: **permanent or long-lived inability for honest nodes to process valid blocks or sync updates under normal network assumptions**.

### Likelihood Explanation

Any unprivileged network peer can trigger this. The attacker only needs to:
1. Connect to the victim node.
2. Advertise a chain weight slightly heavier than the victim's current peak (causing the victim to request a weight proof).
3. Respond with a `WeightProof` whose `recent_chain_data[-1]` has the correct height and weight (trivially constructed from public chain data) but whose `sub_epochs` list contains hundreds of thousands of entries.

No keys, admin access, or cryptographic breaks are required.

### Recommendation

Before calling `_map_sub_epoch_summaries`, enforce an upper bound on `len(weight_proof.sub_epochs)` proportional to the claimed peak height. The expected number of sub-epochs for a chain of height `H` is at most `H // SUB_EPOCH_BLOCKS + 1`. A check such as:

```python
max_sub_epochs = (peak_height // constants.SUB_EPOCH_BLOCKS) + 2
if len(weight_proof.sub_epochs) > max_sub_epochs:
    log.error("weight proof has too many sub epochs")
    return None, None
```

should be inserted at the top of `_validate_sub_epoch_summaries`, after `last_ses_hash` is obtained (so `peak_height` is derivable from `recent_chain_data`). The same guard should be applied in the wallet's `validate_weight_proof` path.

### Proof of Concept

1. Attacker node connects to victim full node.
2. Attacker advertises `peak_weight = victim_peak_weight + 1` and `peak_height = H`.
3. Victim calls `request_validate_wp(peak_header_hash, H, peak_weight)` and sends `RequestProofOfWeight`.
4. Attacker responds with a `RespondProofOfWeight` containing a `WeightProof` where:
   - `recent_chain_data[-1]` is a real block at height `H` with weight `peak_weight + 1` (copied from public chain data or lightly forged to pass the two pre-checks).
   - `sub_epochs` contains 500,000 `SubEpochData` entries (each ~50 bytes; total ~25 MB, within typical message size limits).
5. Victim passes the height/weight pre-checks at lines 1167–1172 of `full_node.py`.
6. Victim calls `validate_weight_proof`, which synchronously calls `_validate_sub_epoch_summaries` → `_map_sub_epoch_summaries`, iterating 500,000 times in the main event loop thread.
7. Main event loop is blocked for several seconds; all concurrent block/transaction processing stalls.
8. `_validate_summaries_weight` eventually returns `False` (computed `ses_end_height` far exceeds actual chain height), validation fails, peer is banned — but the DoS window has already elapsed.
9. Attacker repeats with a fresh peer identity. [8](#0-7) [4](#0-3) [9](#0-8)

```python
def _validate_sub_epoch_summaries(
    constants: ConsensusConstants,
    weight_proof: WeightProof,
) -> tuple[list[SubEpochSummary] | None, list[uint128] | None]:
    last_ses_hash, last_ses_sub_height = _get_last_ses_hash(constants, weight_proof.recent_chain_data)
    if last_ses_hash is None:
        log.warning("could not find last ses block")
        return None, None

    summaries, total, sub_epoch_weight_list = _map_sub_epoch_summaries(
        constants.SUB_EPOCH_BLOCKS,
        constants.GENESIS_CHALLENGE,
        weight_proof.sub_epochs,
        constants.DIFFICULTY_STARTING,
    )

    log.info(f"validating {len(summaries)} sub epochs")

    # validate weight
    if not _validate_summaries_weight(constants, total, summaries, weight_proof):
        log.error("failed validating weight")
        return None, None

    last_ses = summaries[-1]
    log.debug(f"last ses sub height {last_ses_sub_height}")
    # validate last ses_hash
    if last_ses.get_hash() != last_ses_hash:
        log.error(f"failed to validate ses hashes block height {last_ses_sub_height}")
        return None, None

    return summaries, sub_epoch_weight_list
```

```python
def _map_sub_epoch_summaries(
    sub_blocks_for_se: uint32,
    ses_hash: bytes32,
    sub_epoch_data: list[SubEpochData],
    curr_difficulty: uint64,
) -> tuple[list[SubEpochSummary], uint128, list[uint128]]:
    total_weight: uint128 = uint128(0)
    summaries: list[SubEpochSummary] = []
    sub_epoch_weight_list: list[uint128] = []
    for idx, data in enumerate(sub_epoch_data):
        ses = SubEpochSummary(
            ses_hash,
            data.reward_chain_hash,
            data.num_blocks_overflow,
            data.new_difficulty,
            data.new_sub_slot_iters,
            data.challenge_merkle_root,
        )

        if idx < len(sub_epoch_data) - 1:
            delta = 0
            if idx > 0:
                delta = data.num_blocks_overflow
            log.debug(f"sub epoch {idx} start weight is {total_weight + curr_difficulty} ")
            sub_epoch_weight_list.append(uint128(total_weight + curr_difficulty))
            total_weight = uint128(
                total_weight
                + curr_difficulty * (sub_blocks_for_se + sub_epoch_data[idx + 1].num_blocks_overflow - delta)
            )

        # if new epoch update diff and iters
        if data.new_difficulty is not None:
            curr_difficulty = data.new_difficulty

        # add to dict
        summaries.append(ses)
        ses_hash = std_hash(ses)
    # add last sub epoch weight
    sub_epoch_weight_list.append(uint128(total_weight + curr_difficulty))
    return summaries, total_weight, sub_epoch_weight_list
```
