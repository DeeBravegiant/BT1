### Title
`ProposalInit.builder` Not Validated in `is_proposal_init_valid` Allows Malicious Proposer to Corrupt Sequencer Address and Steal Block Fees â€” (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `builder` field of `ProposalInit` is passed directly as `sequencer_address` into `BlockInfo` and committed to the block hash, but `is_proposal_init_valid` never checks it against any expected value. A malicious block proposer can set `builder` to an arbitrary address; all validators will accept the proposal, execute the block with the attacker's address as sequencer (routing all transaction fees there), and commit a block hash that encodes the wrong sequencer.

---

### Finding Description

`is_proposal_init_valid` in `crates/apollo_consensus_orchestrator/src/validate_proposal.rs` validates `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices (within margin), `starknet_version`, `version_constant_commitment`, `timestamp`, and `fee_proposal_fri`. The `builder` field is never checked. [1](#0-0) 

`convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address` in `BlockInfo`: [2](#0-1) 

This `BlockInfo` is forwarded to the batcher via `ValidateBlockInput.block_info` inside `initiate_validation`: [3](#0-2) 

The batcher executes all transactions with `sequencer_address = init.builder`, routing all fee token transfers to that address. The same `BlockInfo` feeds `PartialBlockHashComponents::new`, which encodes `sequencer: SequencerContractAddress(block_info.sequencer_address)` into the final block hash: [4](#0-3) 

The only commitment check is: [5](#0-4) 

Because both proposer and validator derive `BlockInfo` from the same `ProposalInit.builder` value, the `proposal_commitment` always matches regardless of what `builder` contains. The check provides no protection against a manipulated `builder`.

The proposer sets `builder` from its own static config: [6](#0-5) 

A malicious proposer simply substitutes any address it controls.

---

### Impact Explanation

**Critical â€” Incorrect fee/balance with economic impact.** All transaction fees for the block are transferred to `sequencer_address = init.builder`. A malicious proposer sets `builder` to an attacker-controlled address, stealing every fee token paid in that block.

**Critical â€” Wrong block hash / state commitment.** `PartialBlockHashComponents` encodes the wrong `sequencer` address, producing an incorrect `PartialBlockHash` that is committed on-chain and used for proof verification: [7](#0-6) 

**High â€” Authoritative-looking wrong RPC value.** `starknet_getBlock` returns the attacker's address as `sequencer_address`, an authoritative-looking wrong value visible to all clients.

---

### Likelihood Explanation

Any validator in the consensus set can be a block proposer in its turn â€” no special privilege beyond validator membership is required. The attack is silent: the proposal passes every check in `is_proposal_init_valid`, the batcher returns a matching `proposal_commitment`, and the block is committed normally. There is no observable error on the validator side.

---

### Recommendation

Add a check inside `is_proposal_init_valid` that rejects `init_proposed.builder` values that differ from the locally expected sequencer address. The expected value can be derived from the validator's own static configuration (`builder_address`) or from a committee-assigned value, mirroring the pattern already used for `l2_gas_price_fri` and `l1_da_mode`:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder
        ),
    ));
}
```

`ProposalInitValidation` should carry the expected `builder` address, populated from `self.config.static_config.builder_address` at the call site in `sequencer_consensus_context.rs`. [8](#0-7) 

---

### Proof of Concept

1. Malicious validator becomes proposer for height H.
2. Sends `ProposalInit` with `builder = attacker_address` (any address it controls).
3. `is_proposal_init_valid` passes â€” no check on `builder`.
4. `initiate_validation` calls `convert_to_sn_api_block_info(init)` â†’ `sequencer_address = attacker_address`.
5. Batcher executes block: all fee token transfers go to `attacker_address`.
6. `PartialBlockHashComponents` encodes `sequencer = attacker_address` into the block hash.
7. `proposal_commitment` matches (both sides used the same `builder` from `ProposalInit`).
8. Block is committed with the wrong sequencer address; fees are stolen; block hash is corrupted. [9](#0-8)

        || init.l1_gas_price_wei.0 == 0

        || init.l1_data_gas_price_wei.0 == 0
        || init.l2_gas_price_fri.0 == 0

    let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
    let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
    let l2_gas_price_fri = NonzeroGasPrice::new(init.l2_gas_price_fri)?;

        })?;
    Ok(starknet_api::block::BlockInfo {

        block_timestamp: BlockTimestamp(init.timestamp),
        sequencer_address: init.builder,

                l1_gas_price: l1_gas_price_wei,

                l2_gas_price: l2_gas_price_wei,
            },
        },
        use_kzg_da: init.l1_da_mode.is_use_kzg_da(),
        starknet_version: init.starknet_version,
    })
