Audit Report

## Title
`rsETHPrice` Remains Stale When Daily Fee-Mint Cap Is Exhausted, Propagating Outdated Rate Cross-Chain — (`contracts/LRTOracle.sol`, `contracts/cross-chain/RSETHRateProvider.sol`)

## Summary

`LRTOracle._updateRsETHPrice()` calls `_checkAndUpdateDailyFeeMintLimit()` before writing `rsETHPrice = newRsETHPrice`. When the daily cap is reached, the revert unwinds the entire call and the storage slot retains its previous value. `RSETHRateProvider.getLatestRate()` is a direct storage read of that slot, so any subsequent call to `CrossChainRateProvider.updateRate()` relays the stale rate to destination chains until the 24-hour window resets.

## Finding Description

**Root cause — price write is ordered after the fee-mint check**

In `_updateRsETHPrice()`, the fee-mint limit check at line 303 precedes the price commit at line 313:

```
_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);  // line 303
...
rsETHPrice = newRsETHPrice;                                         // line 313
``` [1](#0-0) 

`_checkAndUpdateDailyFeeMintLimit` reverts unconditionally when the cap is exceeded:

```solidity
if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
    revert DailyFeeMintLimitExceeded(...);
}
``` [2](#0-1) 

The revert unwinds the entire call frame, so `rsETHPrice` is never updated.

**`updateRSETHPrice()` is public — any caller can attempt it** [3](#0-2) 

**`updateRSETHPriceAsManager()` is equally affected** — it calls the same `_updateRsETHPrice()` and hits the same revert path; the manager role does not bypass the fee-mint limit. [4](#0-3) 

**`RSETHRateProvider.getLatestRate()` is a pure storage read**

```solidity
function getLatestRate() public view override returns (uint256) {
    return ILRTOracle(rsETHPriceOracle).rsETHPrice();
}
``` [5](#0-4) 

It does not trigger a price update; it returns whatever is currently stored.

**`CrossChainRateProvider.updateRate()` has no access control**

```solidity
function updateRate() external payable nonReentrant {
    uint256 latestRate = getLatestRate();
    ...
    ILayerZeroEndpoint(layerZeroEndpoint).send{value: msg.value}(...);
}
``` [6](#0-5) 

Any caller can invoke it at any time, and it faithfully relays the stale `rsETHPrice`.

**Edge case — `maxFeeMintAmountPerDay == 0` (default)**

The default value of `maxFeeMintAmountPerDay` is `0`. There is no "disabled" guard in `_checkAndUpdateDailyFeeMintLimit`: when `maxFeeMintAmountPerDay == 0`, the condition `currentPeriodMintedFeeAmount + feeAmount > 0` is true for any `feeAmount > 0`, so `updateRSETHPrice()` reverts on every call where rewards have accrued and a protocol fee is configured — even before the cap is ever explicitly "reached." [7](#0-6) 

## Impact Explanation

**Low — Contract fails to deliver promised returns, but doesn't lose value.**

Cross-chain consumers of `RSETHRateProvider` (Balancer pools, lending markets, bridges on destination chains) receive a rate that lags behind the true protocol state. Accrued yield is not reflected in the cross-chain rate until the next 24-hour window resets `currentPeriodMintedFeeAmount`. No funds are lost, but the promised up-to-date rate is not delivered.

## Likelihood Explanation

- `updateRSETHPrice()` is public; any account can call it.
- `maxFeeMintAmountPerDay` is a live, manager-settable parameter; setting it to a small value immediately triggers the condition.
- The default value of `0` causes the same revert for any call where `protocolFeeInETH > 0` (i.e., whenever rewards have accrued and `protocolFeeInBPS > 0`), requiring no special configuration.
- No privileged compromise is required; the condition arises from normal protocol operation.

## Recommendation

Two complementary fixes:

**Option A — treat `maxFeeMintAmountPerDay == 0` as "no limit"** in `_checkAndUpdateDailyFeeMintLimit`:

```solidity
function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
    if (maxFeeMintAmountPerDay == 0) return; // no cap configured
    ...
}
```

**Option B — write `rsETHPrice` before attempting fee minting**, or cap the fee rather than reverting:

```solidity
rsETHPrice = newRsETHPrice; // commit price first

if (protocolFeeInETH > 0) {
    uint256 feeAmt = rsethAmountToMintAsProtocolFee;
    if (currentPeriodMintedFeeAmount + feeAmt > maxFeeMintAmountPerDay) {
        feeAmt = maxFeeMintAmountPerDay - currentPeriodMintedFeeAmount;
    }
    // mint feeAmt, update currentPeriodMintedFeeAmount
}
```

Decoupling the price-write from the fee-mint ensures `rsETHPrice` always reflects the current TVL even when the daily fee budget is exhausted.

## Proof of Concept

```solidity
// Foundry fork/unit test sketch
function test_stalePricePropagatedCrossChain() public {
    // 1. Manager sets a tiny daily fee mint cap
    vm.prank(manager);
    lrtOracle.setMaxFeeMintAmountPerDay(1); // 1 wei cap

    // 2. Simulate reward accrual (increase underlying TVL)
    mockEigenLayerIncreaseTVL(1 ether);

    // 3. First updateRSETHPrice call hits the cap and reverts
    vm.expectRevert(ILRTOracle.DailyFeeMintLimitExceeded.selector);
    lrtOracle.updateRSETHPrice();

    // 4. rsETHPrice is unchanged (stale)
    uint256 stalePrice = lrtOracle.rsETHPrice();
    uint256 truePrice = computeExpectedPrice(); // TVL / supply
    assertLt(stalePrice, truePrice);

    // 5. updateRate() propagates the stale price cross-chain
    rsethRateProvider.updateRate{value: 0.01 ether}();
    assertEq(rsethRateProvider.rate(), stalePrice); // not truePrice
}
```

        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
