Audit Report

## Title
LP Position Permanently Locked When Owner Is USDC-Blacklisted in `removeLiquidity` — (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary

`LiquidityLib.removeLiquidity` unconditionally pushes token0 and token1 to `owner` via `safeTransfer` after completing all internal accounting updates. If `owner` is USDC-blacklisted, every call reverts, permanently locking the LP's principal. Because `MetricOmmPool.removeLiquidity` enforces `msg.sender == owner`, no third party can withdraw on the LP's behalf, and no alternative claim path exists in the protocol.

## Finding Description

In `LiquidityLib.removeLiquidity`, after all storage mutations are committed (`binState.token0BalanceScaled`, `binState.token1BalanceScaled`, `binTotalShares`, `positionBinShares`), the function unconditionally pushes owed tokens to `owner`:

```solidity
if (amount0Removed > 0) {
    IERC20(ctx.token0).safeTransfer(owner, amount0Removed);   // L243
}
if (amount1Removed > 0) {
    IERC20(ctx.token1).safeTransfer(owner, amount1Removed);   // L246
}
``` [1](#0-0) 

USDC implements an address blacklist. When `owner` is blacklisted, `safeTransfer` reverts, rolling back the entire transaction including the accounting updates. The caller restriction in `MetricOmmPool.removeLiquidity`:

```solidity
if (msg.sender != owner) revert NotPositionOwner();
``` [2](#0-1) 

prevents any third party from calling `removeLiquidity` on behalf of the blacklisted LP. There is no pull-payment path, no rescue function, and no admin override that allows token recovery for a specific LP position. The LP's shares remain permanently recorded in `_positionBinShares` and `_binTotalShares`, but the underlying tokens can never be retrieved. [3](#0-2) 

## Impact Explanation

A USDC-blacklisted LP permanently loses access to their full token0 and token1 principal with no on-chain remedy. This is a direct, permanent loss of user principal — a High-impact outcome under the allowed impact gate ("Critical/High/Medium direct loss of user principal or owed LP assets"). The pool remains solvent for other LPs since `binTotals` still accounts for the locked tokens, but the affected LP's share is irrecoverable.

## Likelihood Explanation

USDC blacklisting is an infrequent but documented, real-world administrative action by Circle (e.g., OFAC-sanctioned addresses). USDC is the explicit quote token for every pool in the protocol's production configuration. The combination of a forced-push withdrawal pattern, a strict `msg.sender == owner` guard, and USDC as a primary pool token makes this a credible, if rare, scenario. Likelihood is Low; combined with High impact, this meets Sherlock's Medium threshold.

## Recommendation

Replace the push-transfer pattern in `removeLiquidity` with a pull (claim) pattern:

1. Instead of calling `safeTransfer` inside `removeLiquidity`, credit owed amounts to a per-owner internal balance mapping (e.g., `mapping(address => mapping(address => uint256)) pendingWithdrawals`).
2. Add a separate `claimTokens(address token)` function that lets the owner pull their credited balance at any time.

This decouples accounting from token transfer, so a blacklisted address does not block the state update, and the owner can attempt the claim later or after removal from the blacklist.

## Proof of Concept

1. Alice provides liquidity to a USDC/WBTC pool. Her shares are recorded in `_positionBinShares`.
2. Circle blacklists Alice's address on USDC.
3. Alice calls `MetricOmmPool.removeLiquidity`. The `msg.sender == owner` check passes (L206). `LiquidityLib.removeLiquidity` computes `amount1Removed > 0` (USDC owed) and reaches:
   ```solidity
   IERC20(ctx.token1).safeTransfer(owner, amount1Removed);
   ```
   USDC's `transfer` reverts with a blacklist error. The entire transaction reverts, including all accounting updates.
4. Alice cannot use any other address because `msg.sender != owner` is enforced at L206.
5. Alice's shares remain in `_positionBinShares` indefinitely. Her USDC and WBTC principal are permanently locked in the pool with no on-chain recovery path. [4](#0-3) [5](#0-4)
