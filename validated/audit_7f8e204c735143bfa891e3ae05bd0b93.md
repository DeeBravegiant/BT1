### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Reward to Arbitrary Address via Unenforced `farmer_reward_address_override` - (`File: chia/farmer/farmer_api.py`)

### Summary
The `FarmerAPI._process_respond_signatures` method unconditionally accepts the `farmer_reward_address_override` field from a harvester's `RespondSignatures` message and uses it as the `farmer_reward_address` in `DeclareProofOfSpace` sent to the full node. No fee-quality threshold check is performed at this stage. A malicious third-party harvester can set this field to any arbitrary puzzle hash, permanently redirecting the farmer's block reward (1/8 of every block reward, currently 0.25 XCH) to an address of the harvester's choosing for every block won while connected.

### Finding Description

CHIP-22 defines a voluntary fee convention: a third-party harvester may redirect the farmer reward to itself by setting `farmer_reward_address_override`, but only when the proof's fee quality meets or exceeds the harvester's declared `applied_fee_threshold`. The farmer is supposed to validate this before accepting the redirect.

The actual enforcement is split across two code paths, and neither path blocks the redirect:

**Path 1 – `new_proof_of_space` (lines 128–129):**

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
```

`notify_farmer_reward_taken_by_harvester_as_fee` (lines 888–934 of `farmer.py`) computes the fee quality and compares it to the threshold, but **only emits log warnings** when the threshold is invalid or missing. It does not return a rejection signal and does not prevent the block-making flow from continuing.

**Path 2 – `_process_respond_signatures` (lines 916–919):**

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

This is the path that actually determines the `farmer_reward_puzzle_hash` embedded in `DeclareProofOfSpace`. There is **no fee-quality check here at all**. The override from `RespondSignatures` is accepted unconditionally and forwarded to the full node, which will create the block with the harvester-chosen reward address.

The `RespondSignatures` message structure confirms the field is caller-controlled with no on-wire authentication:

```python
class RespondSignatures(Streamable):
    ...
    farmer_reward_address_override: bytes32 | None
```

### Impact Explanation

Every block won while a malicious harvester is connected will have its farmer reward (1/8 of block reward) sent to the harvester's chosen address instead of the farmer's configured `xch_target_address`. The farmer has no mechanism to detect or block this at the protocol level — the log warnings in `notify_farmer_reward_taken_by_harvester_as_fee` are advisory only and do not halt the block submission. This constitutes unauthorized payout redirection of XCH rewards, matching the **High** impact category: "Bypass of wallet, daemon, keychain, RPC, pool, or Data Layer authorization that enables unauthorized signing, key use, coin control, payout redirection."

### Likelihood Explanation

Any operator who connects to a third-party harvester service (e.g., compressed-plot services explicitly supported by CHIP-22) is exposed. The attacker only needs to be a connected harvester peer — no leaked keys, no admin access, no cryptographic break required. The harvester simply sets `farmer_reward_address_override` to its own puzzle hash in every `RespondSignatures` response. The farmer will sign and submit the block with the redirected reward address.

### Recommendation

In `_process_respond_signatures`, before accepting `response.farmer_reward_address_override`, perform the same fee-quality check that `notify_farmer_reward_taken_by_harvester_as_fee` performs, and **reject** (return `None`) if the fee quality does not meet the declared threshold or if no `fee_info` is provided. The log-only behavior in `notify_farmer_reward_taken_by_harvester_as_fee` should be upgraded to a hard rejection that prevents `DeclareProofOfSpace` from being constructed with the overridden address.

### Proof of Concept

A malicious harvester intercepts the `request_signatures` call and injects an arbitrary override address into `RespondSignatures`:

```python
# Malicious harvester: override farmer reward to attacker's address
response = RespondSignatures(
    plot_identifier=request.plot_identifier,
    challenge_hash=request.challenge_hash,
    sp_hash=request.sp_hash,
    local_pk=local_sk.get_g1(),
    farmer_pk=farmer_public_key,
    message_signatures=message_signatures,
    include_source_signature_data=False,
```
