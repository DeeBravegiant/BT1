### Title
Single Byzantine Attested Participant Can Forge CKD Output for `AppPublicKey` Requests — (`crates/contract/src/lib.rs`)

### Summary

`respond_ckd` performs a cryptographic output check (`ckd_output_check`) only for the `AppPublicKeyPV` variant. For the `AppPublicKey` variant the match arm is an unconditional no-op. A single attested participant can therefore call `respond_ckd` with any arbitrary `(big_y, big_c)` bytes, and the contract will immediately resolve the yield and deliver the forged ElGamal ciphertext to the requester — bypassing the threshold requirement entirely.

---

### Finding Description

In `respond_ckd` the variant dispatch is:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

After this match, `resolve_yields_for` is called unconditionally:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

`resolve_yields_for` performs no threshold check — it drains the entire yield queue and resumes every parked caller with the supplied bytes in a single call: [3](#0-2) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a single attested participant — not a threshold of them. [4](#0-3) 

The `AppPublicKeyPV` variant is protected because `ckd_output_check` verifies the pairing equation `e(C, G2) = e(Y, app_pk2) · e(H(pk, app_id), pk)`, which binds the response to the MPC network's public key and the app's key pair: [5](#0-4) 

No equivalent binding exists for `AppPublicKey`.

---

### Impact Explanation

The CKD output `(Y, C)` is an ElGamal encryption of the BLS signature `msk · H(pk, app_id)`. The app decrypts it with its secret scalar `a` to recover `sig = C − a·Y = msk · H(pk, app_id)`, then derives its deterministic key `s = HKDF(sig)`.

A Byzantine node that submits a forged `(Y*, C*)` where it knows the discrete log of `Y*` can:
1. Deliver a ciphertext that decrypts to an attacker-chosen BLS point instead of the true `msk · H(pk, app_id)`.
2. Because the attacker chose `y*` such that `C* = y*·A + Z` for any target `Z`, the app derives a key `s*` that the attacker already knows.
3. The attacker can then decrypt any data the app encrypts to `s*`, or impersonate the app in any protocol that uses `s*`.

This is unauthorized confidential key derivation output delivered without the required threshold of participant authorization — matching the Critical impact scope.

---

### Likelihood Explanation

The attacker must be a single attested MPC participant (TEE-attested node). The MPC threshold scheme is explicitly designed to tolerate up to `t−1` Byzantine participants; a single compromised node is the canonical below-threshold adversary. The existing unit test at lib.rs:3424–3440 already demonstrates the exact attack path — it submits `big_y=[1u8;48]` and `big_c=[2u8;48]` (invalid G1 points) against an `AppPublicKey` request and asserts the call succeeds and the yield is resolved: [6](#0-5) 

No additional preconditions are needed beyond being an attested participant.

---

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. For `AppPublicKey`, the app supplies only `pk1 = a·G1`; the check `e(C, G2) = e(H(pk, app_id), pk_mpc) · e(Y, pk1)` can be performed with only the G1 component, which is already available in the request. Alternatively, deprecate `AppPublicKey` in favour of `AppPublicKeyPV`, which already has the on-chain guard.

---

### Proof of Concept

The existing test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` (lib.rs:3403–3441) is already a complete proof of concept:

1. Submit a `CKDRequest` with `AppPublicKey` variant via `request_app_private_key`.
2. Call `respond_ckd` from a single attested participant with `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (bytes that are not valid G1 points).
3. The call returns `Ok(())` and the yield is resolved with the forged bytes.
4. `get_pending_ckd_request` returns `None`, confirming the request was consumed. [7](#0-6)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
