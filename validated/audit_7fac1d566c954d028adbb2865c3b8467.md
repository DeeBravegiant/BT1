### Title
Missing CKD Output Verification for `AppPublicKey` Variant in `respond_ckd()` — (`File: crates/contract/src/lib.rs`)

### Summary

`respond()` always performs cryptographic verification of the submitted signature before resolving a pending request. `respond_ckd()` performs the analogous `ckd_output_check` only for the `AppPublicKeyPV` variant of a CKD request, but silently skips all output verification for the `AppPublicKey` variant. A single Byzantine attested participant can therefore submit an arbitrary `CKDResponse` for any `AppPublicKey`-typed CKD request and have it accepted by the contract with no cryptographic check.

### Finding Description

`respond()` always verifies the submitted signature against the expected derived public key before resolving the yield: [1](#0-0) 

`respond_ckd()` branches on the `app_public_key` variant. For `AppPublicKeyPV` it calls `ckd_output_check`, which performs a BLS12-381 pairing check that cryptographically binds `big_c` and `big_y` to the network public key and the app identity. For `AppPublicKey` the branch body is empty — no check is performed: [2](#0-1) 

The `ckd_output_check` function verifies `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which is the only on-chain guarantee that the returned key material is the honest protocol output: [3](#0-2) 

After the branch, `resolve_yields_for` is called unconditionally, delivering whatever `response` the caller supplied to the waiting user: [4](#0-3) 

The `AppPublicKey` variant is a valid, documented request type accepted by `request_app_private_key()`: [5](#0-4) 

### Impact Explanation

A single malicious attested participant can call `respond_ckd()` with an `AppPublicKey`-typed request and supply arbitrary values for `big_c` and `big_y`. The contract will accept the response without any cryptographic verification and deliver the attacker-chosen key material to the requesting user. This constitutes unauthorized confidential key derivation output — the user receives key material that was not produced by the honest MPC protocol, breaking the core security guarantee of the CKD service. Impact: **Critical** — unauthorized/forged confidential key derivation output delivered to users.

### Likelihood Explanation

Any single attested participant (one Byzantine node below the signing threshold) can exploit this. The attacker only needs to race the honest nodes to call `respond_ckd()` first for an `AppPublicKey` request. No collusion, no threshold cooperation, and no privileged access beyond being an attested participant is required. The `AppPublicKey` variant is a supported, publicly documented request type.

### Recommendation

Add the same cryptographic output verification for the `AppPublicKey` variant in `respond_ckd()`. If the `AppPublicKey` variant structurally cannot support a pairing-based check (because it lacks the G2 component), the contract should either:

1. Reject `AppPublicKey` requests in `respond_ckd()` with an explicit error, or
2. Require callers to use `AppPublicKeyPV` for all CKD requests so that on-chain verification is always possible.

At minimum, document clearly that `AppPublicKey` responses are unverified and assess whether this is acceptable for the threat model.

### Proof of Concept

1. User calls `request_app_private_key()` with `app_public_key = AppPublicKey(some_g1_point)`.
2. The contract stores the `CKDRequest` and yields.
3. A single malicious attested participant calls `respond_ckd(request, CKDResponse { big_c: attacker_point, big_y: attacker_point })`.
4. `respond_ckd()` reaches the `AppPublicKey(_) => {}` branch — no check is executed.
5. `resolve_yields_for` is called with the attacker-supplied response.
6. The user's yield resumes and they receive the attacker-chosen `CKDResponse` as if it were the honest protocol output. [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
