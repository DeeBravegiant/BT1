### Title
Hardcoded `RLP_EXECUTE_CALLBACK_GAS` May Permanently Lock ETH Implicit Account Wallet - (File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs)

### Summary
The `WalletContract` (the system contract deployed for all ETH implicit accounts on NEAR) allocates a hardcoded `5 Tgas` budget to its terminal callback `rlp_execute_callback`. If this callback receipt fails due to gas exhaustion — analogous to EIP-1884 increasing gas costs past Ethereum's 2300-unit `transfer` limit — the `has_in_flight_tx` state flag is never reset to `false`, permanently locking the wallet. All subsequent `rlp_execute` calls are rejected with "transaction already in progress," making the user's funds in the ETH implicit account permanently inaccessible.

### Finding Description

`RLP_EXECUTE_CALLBACK_GAS` is hardcoded at `Gas::from_tgas(5)`: [1](#0-0) 

Every successful `rlp_execute` call sets `has_in_flight_tx = true` and schedules a promise chain whose final step is `rlp_execute_callback` allocated exactly this fixed budget: [2](#0-1) [3](#0-2) 

The callback's first action is to write `self.has_in_flight_tx = false`: [4](#0-3) 

In NEAR, if a receipt fails (including `GasExceeded`), all state changes from that receipt are reverted. If `rlp_execute_callback` runs out of its 5 Tgas budget before completing, the `has_in_flight_tx = false` write is rolled back. The flag remains `true` in the trie permanently.

The guard at the top of `rlp_execute` then blocks every future call: [5](#0-4) 

There is no recovery path. No method exists to reset `has_in_flight_tx` independently of a successful callback execution.

The same hardcoded constant is reused in `address_check_callback` and `nep_141_storage_balance_callback` when they re-schedule `rlp_execute_callback`: [6](#0-5) [7](#0-6) 

Additionally, `address_check_callback` and `nep_141_storage_balance_callback` themselves set `has_in_flight_tx = false` at entry and `true` at exit. If either of those callbacks fails due to gas exhaustion, the same permanent lock occurs via the same revert mechanism. [8](#0-7) [9](#0-8) 

### Impact Explanation

The corrupted DB entry is `has_in_flight_tx` in the wallet contract's storage trie for the affected ETH implicit account. Once stuck at `true`:

- Every `rlp_execute` call is rejected immediately.
- The user cannot transfer NEAR, call contracts, add/remove keys, or perform any action through the wallet.
- All NEAR tokens held by the ETH implicit account become permanently inaccessible — there is no admin key, no recovery function, and no protocol-level override.

This is a permanent, irreversible loss of access to user funds for the affected account.

### Likelihood Explanation

Currently, 5 Tgas is sufficient for the operations inside `rlp_execute_callback` under the present NEAR gas schedule. The risk materializes if NEAR protocol increases the gas cost of host functions (e.g., `storage_write`, `promise_batch_create`, `promise_batch_action_transfer`) — directly analogous to EIP-1884 raising the cost of `SLOAD` past Ethereum's 2300-unit `transfer` stipend. The NEAR protocol has historically adjusted gas costs (e.g., protocol version 52 changes to `max_gas_burnt`). Any future recalibration of host-function costs that pushes the callback's actual consumption above 5 Tgas would trigger this lock for any in-flight transaction at that moment. Likelihood is low today but non-zero over the contract's lifetime, and the consequence when triggered is total and permanent.

### Recommendation

1. Replace the hardcoded `RLP_EXECUTE_CALLBACK_GAS` with a dynamic gas weight (`GasWeight`) so the callback receives a proportional share of remaining gas rather than a fixed ceiling.
2. Alternatively, add a permissioned `reset_in_flight` method callable only by the account owner (full-access key) to recover from a stuck state.
3. At minimum, document that `has_in_flight_tx` can become permanently stuck if the callback gas budget is ever insufficient, and monitor gas cost changes in protocol upgrades.

### Proof of Concept

1. User holds an ETH implicit account (`0x…`) with the `WalletContract` deployed.
2. User signs and submits a valid `rlp_execute` transaction (e.g., a `FunctionCall` action). `rlp_execute` succeeds: `has_in_flight_tx = true`, promise chain scheduled, `rlp_execute_callback` allocated exactly `RLP_EXECUTE_CALLBACK_GAS = 5 Tgas`.
3. A NEAR protocol upgrade increases the gas cost of `storage_write` or `promise_batch_create` such that `rlp_execute_callback`'s actual gas consumption exceeds 5 Tgas.
4. The `rlp_execute_callback` receipt fails with `GasExceeded`. NEAR runtime reverts all state changes from that receipt, including `self.has_in_flight_tx = false`.
5. `has_in_flight_tx` remains `true` in the trie.
6. Every subsequent call to `rlp_execute` returns `ExecuteResponse { success: false, error: "Error: transaction already in progress, please try again later." }`.
7. The user's NEAR balance in the ETH implicit account is permanently inaccessible.
