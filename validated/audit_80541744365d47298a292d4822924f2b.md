Audit Report

## Title
`maxNumberOfWithdrawalsPerUser` Uninitialized to Zero Permanently Blocks Withdrawals Until Admin Acts - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal()` guards with `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and `0 >= 0` is always `true`, every call to `initiateWithdrawal()` reverts with `WithdrawalLimitReached` until the admin explicitly calls `setMaxNumberOfWithdrawalsPerUser()`. Any user who stakes before that admin call has their KERNEL tokens frozen in the contract with no withdrawal path.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared as a plain `uint256` storage variable: [1](#0-0) 

`initialize()` sets only `kernelToken`, `rewardsToken`, and the admin role — it never assigns `maxNumberOfWithdrawalsPerUser`: [2](#0-1) 

`stake()` imposes no precondition on `maxNumberOfWithdrawalsPerUser`, so users can deposit freely: [3](#0-2) 

`initiateWithdrawal()` then unconditionally evaluates the guard at line 323: [4](#0-3) 

With `maxNumberOfWithdrawalsPerUser == 0`, `userWithdrawalIds[msg.sender].length` (which is `0` for a fresh staker) satisfies `0 >= 0 == true`, causing an unconditional revert. The only remedy is the admin calling `setMaxNumberOfWithdrawalsPerUser()`, which itself enforces `> 0`: [5](#0-4) 

Additionally, `withdrawalDelay` is similarly uninitialized (defaults to `0`), and `setWithdrawalDelay()` enforces `> 0`: [6](#0-5) 

This means that once `maxNumberOfWithdrawalsPerUser` is fixed but before `setWithdrawalDelay()` is called, any withdrawal created will have `unlockTime = block.timestamp + 0 = block.timestamp`, making it immediately claimable in the same block — bypassing the intended delay protection: [7](#0-6) 

## Impact Explanation
**Medium — Temporary freezing of funds.** Every user who calls `stake()` before the admin calls `setMaxNumberOfWithdrawalsPerUser()` has their KERNEL tokens locked in the contract with no withdrawal path. The freeze persists until the admin acts. There is no loss of principal, but the contract fails to honour the withdrawal promise during this window. This matches the allowed impact: *Medium. Temporary freezing of funds.*

## Likelihood Explanation
The contract's own NatSpec comment explicitly states that users are expected to stake *before* the admin calls `notifyRewardAmount()`, and `notifyRewardAmount()` itself enforces `totalKernelStaked > 0`: [8](#0-7) [9](#0-8) 

This makes it a near-certainty that stakers will be present before all admin parameters are configured. The window between deployment and full admin configuration is a realistic, non-negligible period. No attacker capability is required — any ordinary user calling `stake()` followed by `initiateWithdrawal()` triggers the freeze.

## Recommendation
Initialize both `maxNumberOfWithdrawalsPerUser` and `withdrawalDelay` to safe non-zero defaults inside `initialize()`. For example:

```solidity
maxNumberOfWithdrawalsPerUser = 10;
withdrawalDelay = 7 days;
```

Alternatively, add a guard in `stake()` that reverts if `maxNumberOfWithdrawalsPerUser == 0`, mirroring the pattern used in `notifyRewardAmount()` which already guards against `totalKernelStaked == 0`.

## Proof of Concept
1. Admin deploys and calls `initialize(admin, kernelToken, rewardToken)`. `maxNumberOfWithdrawalsPerUser` remains `0`.
2. User calls `stake(100e18)`. Succeeds — 100 KERNEL transferred to contract, `balanceOf[user] = 100e18`.
3. User calls `initiateWithdrawal(100e18)`.
   - Check: `userWithdrawalIds[user].length = 0`; `maxNumberOfWithdrawalsPerUser = 0`; `0 >= 0 → true` → reverts `WithdrawalLimitReached`.
4. User's 100 KERNEL is frozen until admin calls `setMaxNumberOfWithdrawalsPerUser(N)`.
5. If admin has not yet called `setWithdrawalDelay()`, any withdrawal initiated immediately after step 4 will have `unlockTime = block.timestamp`, making it claimable in the same block — bypassing the delay entirely.

**Foundry test sketch:**
```solidity
function test_withdrawalBlockedWhenMaxNotSet() public {
    // Deploy and initialize without setting maxNumberOfWithdrawalsPerUser
    pool.initialize(admin, address(kernelToken), address(rewardToken));

    // User stakes
    vm.startPrank(user);
    kernelToken.approve(address(pool), 100e18);
    pool.stake(100e18);

    // User attempts withdrawal — should revert with WithdrawalLimitReached
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(100e18);
    vm.stopPrank();
}
```
