Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Permanently Blocks All Withdrawals - (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool.initialize()` never sets `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. Because `initiateWithdrawal` guards with `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, and any `uint256 >= 0` is always `true`, every call to `initiateWithdrawal` reverts unconditionally until an admin separately calls `setMaxNumberOfWithdrawalsPerUser`. Any tokens staked via `stake`, `stakeFor`, or `KernelMerkleDistributor.claimAndStake` during this window are permanently unwithdrawable.

## Finding Description
`KernelDepositPool.initialize()` sets `kernelToken`, `rewardsToken`, and the admin role, but leaves `maxNumberOfWithdrawalsPerUser` at `0`.

`initiateWithdrawal` enforces:
```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```
`userWithdrawalIds[msg.sender].length` is a `uint256` and is always `>= 0`, so when `maxNumberOfWithdrawalsPerUser == 0` this condition is a tautology — the revert fires for every caller regardless of their staked balance or withdrawal history.

`stakeFor` and `stake` contain no equivalent guard, so tokens can be deposited into the pool freely. `KernelMerkleDistributor.claimAndStake` calls `stakeFor` directly, crediting `balanceOf[account]` and transferring tokens into the pool. The only exit path, `initiateWithdrawal`, is sealed.

The internal contradiction is explicit: `setMaxNumberOfWithdrawalsPerUser` rejects `0` with `InvalidMaxNumberOfWithdrawalsPerUser`, confirming the developers treat `0` as an invalid operational state — yet `initialize` leaves the variable there with no enforcement to move it out and no indication to deployers that a follow-up call is required.

## Impact Explanation
Any KERNEL tokens staked while `maxNumberOfWithdrawalsPerUser == 0` are permanently frozen. There is no alternative withdrawal function; `initiateWithdrawal` is the sole exit. Affected users have no recovery path. This matches the allowed impact: **Critical — Permanent freezing of funds**.

## Likelihood Explanation
The precondition is that the admin deploys and initializes `KernelDepositPool` without subsequently calling `setMaxNumberOfWithdrawalsPerUser`. This is a realistic oversight because `initialize` accepts no parameter for this value, gives no indication it must be set separately, and the contract appears fully functional for staking immediately after initialization. The broken withdrawal path is invisible until a user attempts to withdraw. Deployment scripts or checklists that omit this step silently produce a broken contract. Likelihood is **Low** (requires admin deployment oversight), but the consequence when it occurs is irreversible for all affected users.

## Recommendation
Set a safe non-zero default for `maxNumberOfWithdrawalsPerUser` inside `initialize`, or require it as a parameter with the same validation used in `setMaxNumberOfWithdrawalsPerUser`:

```solidity
function initialize(
    address _admin,
    address _kernelToken,
    address _rewardToken,
    uint256 _maxNumberOfWithdrawalsPerUser
) external initializer {
    // ... existing checks ...
    if (_maxNumberOfWithdrawalsPerUser == 0 || _maxNumberOfWithdrawalsPerUser > MAX_WITHDRAWALS_PER_USER) {
        revert InvalidMaxNumberOfWithdrawalsPerUser();
    }
    maxNumberOfWithdrawalsPerUser = _maxNumberOfWithdrawalsPerUser;
}
```

Additionally, add a guard in `stakeFor` and `stake` that reverts if `maxNumberOfWithdrawalsPerUser == 0`, preventing tokens from entering an unwithdrawable state.

## Proof of Concept
```solidity
function test_permanentFreeze() public {
    // 1. Deploy without calling setMaxNumberOfWithdrawalsPerUser
    KernelDepositPool pool = new KernelDepositPool();
    pool.initialize(admin, address(kernelToken), address(rewardToken));
    // maxNumberOfWithdrawalsPerUser == 0

    KernelMerkleDistributor distributor = new KernelMerkleDistributor();
    distributor.initialize(address(kernelToken), address(pool), treasury, 0);
    pool.grantRole(pool.STAKE_FOR_ROLE(), address(distributor));

    // 2. User calls claimAndStake — stakeFor succeeds, no withdrawal-limit guard
    vm.prank(user);
    distributor.claimAndStake(index, user, amount, proof);
    assertGt(pool.balanceOf(user), 0);

    // 3. User attempts to withdraw — always reverts: 0 >= 0 is always true
    vm.prank(user);
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    pool.initiateWithdrawal(amount);
}
```
