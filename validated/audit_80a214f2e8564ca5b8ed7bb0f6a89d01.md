Audit Report

## Title
Missing Staleness and Round-Completeness Checks in `ChainlinkPriceOracle.getAssetPrice()` Enables Overvalued LST Deposits During Feed Outages - (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` fields, accepting any returned price as current. During a Chainlink feed staleness window (e.g., an LST depeg event where the on-chain price lags the real market price), any unprivileged depositor can call `LRTDepositPool.depositAsset()` with depegged LST and receive rsETH minted at the inflated stale price, diluting all existing rsETH holders and breaking the protocol's 1:1 ETH backing invariant.

## Finding Description

**Root cause — `contracts/oracles/ChainlinkPriceOracle.sol` L49–55:**

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

The five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. Only `answer` is captured; `updatedAt` and `answeredInRound` are discarded. No check of the form `block.timestamp - updatedAt > heartbeat` or `answeredInRound < roundId` is present. [1](#0-0) 

**Contrast with the sister contract `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` L26–36**, which correctly validates all three conditions: [2](#0-1) 

**Exploit path:**

1. `LRTDepositPool.depositAsset()` calls `_beforeDeposit()` → `getRsETHAmountToMint()`. [3](#0-2) 

2. `getRsETHAmountToMint()` computes `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`. The numerator uses the live (stale) Chainlink price; the denominator `rsETHPrice` is a stored value last updated by `updateRSETHPrice()`. [4](#0-3) 

3. `lrtOracle.getAssetPrice(asset)` delegates to `ChainlinkPriceOracle.getAssetPrice()`, returning the stale inflated price. [5](#0-4) 

**Why existing guards are insufficient:**

`LRTOracle._updateRsETHPrice()` contains a `pricePercentageLimit` downside-protection check that can pause the protocol if the computed rsETH price drops too far. [6](#0-5) 

However, this check is only evaluated when `updateRSETHPrice()` is called. During a staleness window, `getAssetPrice()` still returns the old (pre-depeg) price, so `_getTotalEthInProtocol()` computes the same TVL as before the depeg, `newRsETHPrice` does not decrease, and the pause is never triggered. The protection is entirely blind to stale-price scenarios. Deposits are not gated on a fresh `updateRSETHPrice()` call. [7](#0-6) 

## Impact Explanation

**Critical — Direct theft of user funds (dilution of existing rsETH holders) / Protocol insolvency.**

When the Chainlink stETH/ETH feed is stale at `1e18` while the real market price is `0.95e18`:

- `getAssetPrice(stETH)` returns `1e18` (stale).
- `getRsETHAmountToMint(stETH, 100e18)` mints rsETH as if 100 stETH = 100 ETH, when it is actually worth only 95 ETH.
- The attacker holds rsETH redeemable for ~100 ETH of assets, backed by only 95 ETH of real value.
- All existing rsETH holders are diluted; the protocol's backing ratio falls below 1:1.
- Repeated at scale, this constitutes protocol insolvency.

## Likelihood Explanation

- The Chainlink stETH/ETH feed has a 24-hour heartbeat and a 0.5% deviation threshold. Historical precedent: the May 2022 stETH depeg and the March 2023 USDC depeg both caused Chainlink feeds to lag real market prices by hours within their heartbeat windows.
- Any network congestion event delaying Chainlink keeper transactions creates the same window.
- No special permissions are required: `depositAsset()` is callable by any address.
- The attacker only needs to observe that the on-chain Chainlink price diverges from the real market price (e.g., via a DEX spot price) and deposit during that window.
- The attack is repeatable for the entire duration of the staleness window.

## Recommendation

Add staleness and round-completeness checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
uint256 public constant MAX_STALENESS = 25 hours; // set per-asset based on feed heartbeat

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();
    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be configurable per asset (e.g., stored in a mapping) and set to the feed's documented heartbeat plus a small buffer (e.g., 25 hours for a 24-hour heartbeat feed).

## Proof of Concept

**Setup:** stETH/ETH Chainlink feed last updated at `T-20h`; current stETH market price is 0.94 ETH (depeg event in progress). Feed still reports `1e18`. `rsETHPrice` was last updated before the depeg and reflects `1e18` per stETH.

**Steps:**

1. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
2. `_beforeDeposit` → `getRsETHAmountToMint(stETH, 100e18)`:
   - `getAssetPrice(stETH)` → `1e18` (stale, should be `0.94e18`)
   - `rsethAmountToMint = (100e18 × 1e18) / rsETHPrice`
   - Attacker receives rsETH worth ~100 ETH, backed by only 94 ETH of stETH.
3. `updateRSETHPrice()` is not called during the deposit; the `pricePercentageLimit` guard never fires because the stale Chainlink price keeps the computed TVL unchanged.
4. Attacker holds rsETH redeemable for ~100 ETH of assets for a 94 ETH deposit.
5. Net gain: ~6 ETH per 100 ETH cycled, at the expense of existing rsETH holders.

**Foundry fork test plan:** Fork mainnet at a block where the stETH/ETH Chainlink feed is within its heartbeat but the stETH spot price on Curve has dropped >1%. Mock `latestRoundData()` to return a timestamp `> heartbeat` ago with the pre-depeg price. Call `depositAsset(stETH, 100e18, 0, "")` from an unprivileged address. Assert that `rsethAmountToMint` exceeds the fair value computed using the real spot price, confirming the dilution.
