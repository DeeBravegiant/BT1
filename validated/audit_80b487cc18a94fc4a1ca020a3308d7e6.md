### Title
Any Attested Participant Can Front-Run the Elected Leader's `respond` / `respond_ckd` / `respond_verify_foreign_tx` Submission - (File: crates/contract/src/lib.rs)

### Summary
The `respond`, `respond_ckd`, and `respond_verify_foreign_tx` contract entry points accept any valid `(request, response)` pair from any attested participant. The submitter's identity is never bound to the proof. A Byzantine participant (strictly below the signing threshold) who observes the elected leader's pending NEAR transaction can copy the exact same arguments and submit an identical call first, causing the leader's transaction to fail and wasting all of its computation work. This is the direct structural analog of the `finalizeBlocks` front-running described in the external report.

### Finding Description

The NEAR MPC node uses a deterministic, off-chain leader-election algorithm to decide which participant should lead each signing request and submit the final `respond` call. [1](#0-0) 

This leader selection is enforced only at the node level. The on-chain contract enforces two things when `respond` is called:

1. The caller is an attested participant.
2. The supplied signature is cryptographically valid for the request payload. [2](#0-1) 

It does **not** enforce that the caller is the specific elected leader for that request. The `signer` variable is only logged, never stored or checked against a per-request leader record. [3](#0-2) 

The same structural gap exists in `respond_ckd` and `respond_verify_foreign_tx`. [4](#0-3) [5](#0-4) 

Once a valid `(request, response)` pair is produced by the threshold protocol and broadcast by the leader, every other attested participant who sees that NEAR transaction in the mempool possesses all the information needed to submit an identical call under their own account.

### Impact Explanation

When the front-runner's transaction is included first, `resolve_yields_for` removes the pending request from the map. [6](#0-5) 

The original leader's transaction then fails because the request entry no longer exists. The leader's entire computation — presignature consumption, multi-round threshold protocol, and NEAR transaction gas — is wasted. The node-side tx-sender confirms success by checking whether the pending request has been removed, so it will silently treat the front-runner's success as its own. [7](#0-6) 

The user's request is still resolved correctly (the correct signature is delivered), so there is no immediate fund loss. However:

- The request-lifecycle invariant — that the elected leader is the entity credited with resolving the request — is broken by any single Byzantine participant below the signing threshold.
- If any future reward or fee-distribution mechanism is added for nodes that submit `respond` (a natural evolution for a decentralized sequencer network), this becomes a direct theft path: a Byzantine node can systematically steal every reward without performing any MPC computation.
- For `respond_verify_foreign_tx`, the front-runner controls which `payload_hash` value is delivered to the user if it can race with a different (but still validly signed) hash — though producing a different valid signature requires threshold cooperation and is thus disqualified; the same-hash front-run is the realistic path.

### Likelihood Explanation

NEAR transactions are publicly visible in the mempool before finalization. Any attested participant — a role that requires only a valid TEE attestation, not threshold-level collusion — can monitor the mempool, extract the `(request, response)` arguments from the leader's pending transaction, and resubmit them. The attack requires no cryptographic capability beyond being an attested participant, and no coordination with other nodes. The node-side leader-election logic provides no on-chain protection. [8](#0-7) 

### Recommendation

Bind the responding node's identity to the proof at the contract level. Two complementary approaches:

1. **On-chain leader enforcement**: Store the elected leader's account ID alongside each pending request when it is enqueued, and assert in `respond` / `respond_ckd` / `respond_verify_foreign_tx` that `env::signer_account_id()` matches the stored leader. The leader can be derived deterministically from the request ID and the current participant set, mirroring the node-side `leader_selection_hash` logic. [9](#0-8) 

2. **Proof binding**: Include the responding node's account ID in the signed payload (analogous to the external report's recommendation to include `_sequencer` in `_verifyProof`), so that a copied response is cryptographically invalid when submitted by a different account.

### Proof of Concept

```
// Setup: Node A is the elected leader for request R.
// Node A completes the threshold signing protocol and broadcasts:
//   respond({ request: R, response: SIG })
// to the NEAR network.

// Node B (attested participant, not the leader) observes Node A's
// pending transaction in the NEAR mempool and extracts (R, SIG).

// Node B submits an identical call:
//   respond({ request: R, response: SIG })
// with higher priority (e.g., by submitting immediately after observing).

// Contract check in respond():
//   assert_caller_is_attested_participant_and_protocol_active() -> OK (B is attested)
//   verify_ecdsa_signature(SIG, payload_hash, expected_public_key) -> OK (SIG is valid)
//   resolve_yields_for(pending_signature_requests, R, SIG) -> resolves, removes R

// Node A's transaction now fails: R is no longer in pending_signature_requests.
// Node A's presignature is consumed, computation is wasted, gas is lost.
// If rewards exist, Node B receives them.
```

The contract's `assert_caller_is_attested_participant_and_protocol_active` confirms Node B is a valid caller. [10](#0-9) 

The signature check passes because the signature is valid regardless of who submits it. [11](#0-10)

    #[handle_result]
    pub fn respond(
        &mut self,
        request: SignatureRequest,
        response: dtos::SignatureResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!("respond: signer={}, request={:?}", &signer, &request);

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain)?;

        let signature_is_valid = match (&response, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                // generate the expected public key
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");
                let affine = *k256::PublicKey::try_from(&secp_pk)
                    .expect("stored key is always valid")
                    .as_affine();
                let expected_public_key =
                    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;

                let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");

                // Check the signature is correct
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    payload_hash,
                    &expected_public_key,
                )
                .is_ok()
            }
            (
                dtos::SignatureResponse::Ed25519 { signature },
                PublicKeyExtended::Ed25519 {
                    edwards_point: public_key_edwards_point,
                    ..
                },
            ) => {
                let derived_public_key_edwards_point = derive_public_key_edwards_point_ed25519(
                    &public_key_edwards_point,
                    &request.tweak,
                );
                let derived_public_key_32_bytes =
                    dtos::Ed25519PublicKey::from(derived_public_key_edwards_point.compress());

                let message = request.payload.as_eddsa().expect("Payload is not EdDSA");

                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    message,
                    &derived_public_key_32_bytes,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_signature_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
