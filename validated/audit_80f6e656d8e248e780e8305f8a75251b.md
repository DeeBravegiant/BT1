### Title
Missing `payload_hash`-to-`request` Binding in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies only that the submitted ECDSA signature is valid over the submitted `payload_hash`, but never checks that `payload_hash` was actually derived from the accompanying `request`. A single Byzantine MPC participant (below the signing threshold) can take a legitimately produced MPC signature for foreign-tx request A and replay it as the response to a different pending request B. The contract accepts it, resolves all queued yields for request B, and delivers a forged attestation to every waiting caller.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs three checks:

1. Caller is an attested participant.
2. The ECDSA signature in `response.signature` is valid over `response.payload_hash`.
3. `request` exists in `pending_verify_foreign_tx_requests`; if so, all queued yields are resolved with the raw `response` bytes. [1](#0-0) 

Step 2 verifies only the cryptographic binding between `signature` and `payload_hash`. It does **not** verify that `payload_hash` encodes the correct `request`. The canonical hash is defined as:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload { request, values }))
``` [2](#0-1) 

Because `request` is embedded inside the hashed payload, a `payload_hash` produced for request A (tx_id_A, chain_A, extractors_A, observed_values_A) is cryptographically unrelated to request B. The contract never recomputes or checks this binding; it only checks `verify_ecdsa_signature(sig, payload_hash, root_pk)`.

**Attack path (single Byzantine node below threshold):**

1. Attacker is an attested participant and acts as signing leader for a legitimately pending request A.
2. The MPC network collectively produces `(payload_hash_A, sig_A)` for request A through the normal `make_verify_foreign_tx_leader` / follower flow.
3. Attacker observes a different pending request B (submitted by a victim bridge contract or user).
4. Attacker calls `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })`.
5. The contract verifies `sig_A` over `payload_hash_A` against the root public key — this passes.
6. The contract resolves all yields queued under `request_B` with the bytes of `{ payload_hash_A, sig_A }`.
7. Every caller waiting on request B receives a response whose `payload_hash` encodes a completely different transaction (tx_id_A, chain_A, values_A). [3](#0-2) 

The `VerifyForeignTransactionResponse` type carries only `payload_hash` and `signature`; there is no field that ties the response back to the originating request. [4](#0-3) 

The SDK helper `ForeignChainSignatureVerifier::verify_signature` does perform a client-side `payload_hash` equality check, but this is entirely optional and off-chain. [5](#0-4) 

Any on-chain bridge contract that trusts the contract-delivered `payload_hash` without independently recomputing it (which requires knowing the extracted values, not available at submission time) will accept the forged attestation.

---

### Impact Explanation

A single Byzantine participant below the signing threshold can cause the MPC contract to deliver a cryptographically valid but semantically forged foreign-chain attestation to any pending `verify_foreign_transaction` caller. For the primary use case — the Omnibridge inbound flow — this means a bridge contract could be made to believe that an arbitrary foreign-chain transaction (e.g., a deposit of a specific amount to a specific address) was verified, when in fact a different transaction was verified. This enables invalid bridge execution and potential theft of bridged funds, matching the **High** impact category: *forged foreign-chain verification that causes invalid bridge execution*.

---

### Likelihood Explanation

Any single attested MPC participant can execute this attack without colluding with others. The attacker only needs to:
- Be elected as signing leader for one legitimate request (normal rotation).
- Have a second request pending simultaneously (trivially arranged by submitting one themselves or waiting for organic traffic).

No key material beyond the participant's own share is required. The attack is silent — the contract emits no error, and the victim caller has no on-chain signal that the `payload_hash` is mismatched.

---

### Recommendation

The contract must bind `payload_hash` to `request` before accepting a response. The node should include the full `ForeignTxSignPayload` (i.e., `request` + `values`) in the response, and the contract should recompute and compare:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),
}).compute_msg_hash()?;

if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This ensures that `payload_hash` is deterministically tied to the `request` the contract holds, making cross-request replay impossible regardless of which node submits the response.

---

### Proof of Concept

```
1. Alice calls verify_foreign_transaction(request_B = { tx_id: 0xBBBB…, chain: Bitcoin, … })
   → pending_verify_foreign_tx_requests[request_B] = [alice_yield_id]

2. Attacker (attested node) is elected leader for request_A = { tx_id: 0xAAAA…, chain: Bitcoin, … }
   → MPC network signs payload_hash_A = SHA-256(borsh({ request_A, [BlockHash(0xDEAD…)] }))
   → attacker holds (payload_hash_A, sig_A) — a valid MPC signature

3. Attacker calls:
   respond_verify_foreign_tx(
     request  = request_B,          // matches Alice's pending entry
     response = { payload_hash: payload_hash_A, signature: sig_A }
   )

4. Contract check: verify_ecdsa_signature(sig_A, payload_hash_A, root_pk) → OK
   (no check that payload_hash_A encodes request_B)

5. Contract resolves alice_yield_id with { payload_hash_A, sig_A }

6. Alice's bridge contract receives payload_hash_A (encodes tx_id 0xAAAA…, not 0xBBBB…)
   and a valid MPC signature — if it does not recompute the expected hash it
   accepts the attestation and releases funds for an unverified deposit.
```
