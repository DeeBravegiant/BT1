### Title
Stale Foreign-Chain Extracted Values Submitted on Retry via Unconditional `computed_response` Cache — (`crates/node/src/mpc_client.rs`)

---

### Summary

The leader node caches the `computed_response` for `verify_foreign_tx` requests and unconditionally reuses it on every retry attempt, without re-querying the foreign chain. The on-chain `respond_verify_foreign_tx` function accepts any cryptographically valid signature without checking whether the embedded extracted values (e.g., block hash, confirmation count) still reflect the current canonical state of the foreign chain. If the NEAR response transaction lands in a non-canonical block and the foreign chain reorganizes before the retry, the node submits a signed attestation of now-invalid foreign-chain state, which the contract accepts and delivers to the caller.

---

### Finding Description

**Root cause — node side (`crates/node/src/mpc_client.rs`, lines 566–630):**

When the leader node processes a `verify_foreign_tx` request, it queries the foreign-chain RPC, extracts values (e.g., Bitcoin block hash, EVM block hash, confirmation count), and computes a threshold MPC signature over those values. The result is stored in `computation_progress.computed_response`:

```rust
// Only issue an MPC verify_foreign_tx computation if we haven't computed it
// in a previous attempt.
let existing_response = verify_foreign_tx_attempt
    .computation_progress
    .lock()
    .unwrap()
    .computed_response
    .clone();
let response = match existing_response {
    None => {
        // ... query foreign chain, compute MPC signature, cache it ...
        verify_foreign_tx_attempt
            .computation_progress
            .lock()
            .unwrap()
            .computed_response = Some(response.clone());
        response
    }
    Some(response) => response,   // ← unconditionally reuse cached data
};
``` [1](#0-0) 

On every subsequent retry the `Some(response) => response` arm fires with no freshness check. The retry is triggered by `ComputationProgress::update_computation_progress` whenever `last_response_submission` is older than `MAX_LATENCY_BEFORE_EXPECTING_TRANSACTION_TO_FINALIZE` (10 s), or when the queue detects the response landed in a non-canonical NEAR block: [2](#0-1) [3](#0-2) 

The test `test_pending_requests__should_reattempt_when_response_is_non_canonical` explicitly confirms that the queue re-issues the request after a NEAR fork: [4](#0-3) 

**Root cause — contract side (`crates/contract/src/lib.rs`, lines 691–754):**

`respond_verify_foreign_tx` verifies only that the ECDSA signature is cryptographically valid against the current MPC public key. It performs no check on whether the `payload_hash` (which commits to the extracted foreign-chain values) still corresponds to the canonical state of the foreign chain:

```rust
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [5](#0-4) 

Once the signature check passes, `resolve_yields_for` is called and the response is delivered to all waiting callers with no further validation. [6](#0-5) 

---

### Impact Explanation

The `verify_foreign_tx` flow is the MPC network's mechanism for bridge protocols to obtain a threshold-signed attestation that a specific foreign-chain transaction exists and that certain values (block hash, confirmation count, log data, etc.) were extracted from it. Callers use this attestation to authorize bridge actions (e.g., minting wrapped assets, releasing locked funds).

If the cached response contains extracted values from a foreign-chain block that was subsequently reorganized away, the MPC network delivers a signed attestation of a state that no longer exists on the canonical foreign chain. A bridge protocol that trusts this attestation would process a transaction that was effectively double-spent, enabling theft of bridged funds.

**Impact class:** High — forged foreign-chain verification causing invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

Two independent events must coincide:

1. **NEAR chain fork / non-canonical response block:** This is a normal operational event. The queue explicitly handles it and re-issues the request (`AggregateResponseStatus::None` after a fork). It also retries after 10 s if the NEAR transaction is not finalized.

2. **Foreign-chain reorganization between first computation and retry:** For chains with low user-specified confirmation counts (e.g., 1–2 confirmations on Bitcoin or EVM chains), shallow reorganizations occur naturally. The `BitcoinRpcRequest` and `EvmRpcRequest` types accept a caller-controlled `confirmations` field, so an attacker can deliberately request verification at a low confirmation depth to maximize reorg probability.

The attacker controls the confirmation threshold in the request and can time the submission to coincide with a period of elevated reorg risk on the target chain. The NEAR fork that triggers the retry is not attacker-controlled but occurs naturally and frequently enough to make the window realistic.

---

### Recommendation

1. **Re-query the foreign chain on retry.** Clear `computed_response` before each retry attempt for `verify_foreign_tx` requests (unlike `sign` and `ckd` requests, whose payloads are immutable). This forces the node to re-verify that the foreign-chain state is still canonical before submitting a response.

2. **Alternatively, include a NEAR block height bound in the cached response.** Invalidate the cache if the current NEAR block height has advanced beyond a threshold since the response was computed, forcing a fresh foreign-chain query.

3. **Contract-side defense (defense-in-depth).** The contract could record the NEAR block height at which a `verify_foreign_tx` request was submitted and reject responses that embed a `payload_hash` computed more than N blocks ago, though this requires the node to include a timestamp or block-height commitment in the response.

---

### Proof of Concept

**Step-by-step:**

1. Attacker submits `verify_foreign_tx` for a Bitcoin transaction with `confirmations: 1`.
2. The leader MPC node queries Bitcoin RPC at NEAR block height H, observes the transaction in block B1 with 1 confirmation, computes `ForeignTxSignPayloadV1 { request, values: [BlockHash(B1)] }`, hashes it to `payload_hash_1`, and produces an MPC signature `sig_1` over `payload_hash_1`. This is stored in `computed_response`.
3. The `respond_verify_foreign_tx` NEAR transaction is included in a NEAR block that subsequently becomes non-canonical (a fork). The queue detects `AggregateResponseStatus::None` and schedules a retry.
4. Before the retry fires, Bitcoin reorganizes: block B1 is orphaned, the transaction is unconfirmed.
5. The retry loop in `mpc_client.rs` finds `computed_response = Some(...)` and skips the foreign-chain query entirely, submitting `(payload_hash_1, sig_1)` — the stale attestation — to `respond_verify_foreign_tx`.
6. The contract verifies `sig_1` against the current MPC public key. The signature is cryptographically valid (the key has not changed). The contract calls `resolve_yields_for`, delivering `VerifyForeignTransactionResponse { payload_hash: payload_hash_1, signature: sig_1 }` to the caller.
7. The bridge protocol receives a threshold-signed attestation that the transaction was confirmed in block B1. It releases bridged funds. The transaction is never re-confirmed on the canonical Bitcoin chain, completing the double-spend. [7](#0-6) [8](#0-7)

            ComputationProgressStatus::MaxAttemptsExceeded

            ComputationProgressStatus::Pending
        } else {
            self.attempts += 1;
            ComputationProgressStatus::NewAttempt
        }

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
