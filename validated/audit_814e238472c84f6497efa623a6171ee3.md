Audit Report

## Title
`SfrxETHPriceOracle.getAssetPrice` Returns frxETH/sfrxETH Instead of ETH/sfrxETH, Inflating rsETHPrice on frxETH Depeg ‚Äî (File: `contracts/oracles/SfrxETHPriceOracle.sol`)

## Summary
`SfrxETHPriceOracle.getAssetPrice` returns `ISfrxETH(sfrxETHContractAddress).pricePerShare()` directly, which is the ERC4626 vault's underlying-asset-per-share ratio denominated in **frxETH**, not ETH. `LRTOracle._getTotalEthInProtocol` treats this value as an ETH exchange rate. When frxETH trades at a discount to ETH, `totalETHInProtocol` is overcounted, `rsETHPrice` is set above its true ETH backing, and new depositors receive fewer rsETH than they are owed.

## Finding Description
The real sfrxETH contract is an ERC4626 vault whose underlying asset is frxETH. Its `pricePerShare()` function returns the amount of frxETH redeemable per sfrxETH share ‚Äî a frxETH/sfrxETH ratio. The NatSpec in `SfrxETHPriceOracle.sol` (L9) acknowledges this ("How much frxETH is 1E18 sfrxETH worth") but then incorrectly asserts "Price is in ETH, not USD."

`getAssetPrice` (L40) returns this value with no frxETH‚ÜíETH conversion step. `_getTotalEthInProtocol` (L339‚Äì343) reads `assetER = getAssetPrice(asset)` and computes `totalETHInProtocol += totalAssetAmt.mulWad(assetER)`, treating `assetER` as ETH-denominated.

When frxETH = 1 ETH the error is zero. When frxETH trades at a discount (e.g., 0.97 ETH) and `pricePerShare()` returns `1.05e18`, the oracle reports `1.05 ETH/sfrxETH` when the true value is `1.05 √ó 0.97 = 1.0185 ETH/sfrxETH` ‚Äî a ~3% overcount of the sfrxETH portion of TVL. This flows into `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)` (L250), inflating rsETHPrice.

The `pricePercentageLimit` circuit breaker (L252‚Äì266) only triggers on sudden large price increases relative to `highestRsethPrice`. A gradual, continuous frxETH depeg produces a proportional drift that never trips the threshold, so the guard provides no protection here.

`updateRSETHPrice()` (L87‚Äì89) is public and requires no role, so any external caller ‚Äî including a bot or MEV searcher ‚Äî can lock in the inflated price during a depeg event.

## Impact Explanation
**Low ‚Äî Contract fails to deliver promised returns, but doesn't lose value.**

New depositors who deposit ETH (or any correctly-priced asset) during a frxETH depeg receive `depositAmount / rsETHPrice` rsETH. Because `rsETHPrice` is inflated, they receive fewer rsETH tokens than the true ETH backing warrants. Existing rsETH holders benefit at new depositors' expense. No funds leave the protocol, but the protocol fails to deliver the exchange rate it promises to new depositors.

## Likelihood Explanation
frxETH has historically maintained a close but imperfect peg to ETH; any deviation (even a few basis points) causes a proportional mispricing. The condition requires no privileged access, no victim mistake, and no external protocol compromise ‚Äî it activates passively whenever the frxETH/ETH market rate diverges. The public `updateRSETHPrice()` entry point means any actor can trigger the price update at will during a depeg.

## Recommendation
Replace the direct `pricePerShare()` return with a two-step conversion in `SfrxETHPriceOracle.getAssetPrice`:
1. Fetch `sfrxETH.pricePerShare()` ‚Üí frxETH per sfrxETH.
2. Fetch the frxETH/ETH spot or TWAP price from a Chainlink or Curve oracle.
3. Return `pricePerShare() √ó frxETH_per_ETH / 1e18`.

This ensures `getAssetPrice` always returns a value denominated in ETH, consistent with the `IPriceFetcher` interface contract and the `_getTotalEthInProtocol` accounting assumption.
