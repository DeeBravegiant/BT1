Audit Report

## Title
Permanent ETH Freeze via Unrecoverable Unlocked Withdrawal When Recipient Contract Lacks `receive()` — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
When a smart contract without a `receive()` function initiates an ETH withdrawal, the operator-triggered `unlockQueue` permanently burns the user's rsETH and moves ETH into `LRTWithdrawalManager`. Any subsequent attempt to complete the withdrawal via `_transferAsset` reverts unconditionally, the withdrawal request remains stuck in the unlocked queue forever, and no on-chain mechanism exists to redirect or recover the locked ETH.

## Finding Description
The withdrawal lifecycle is split across two transactions:

**Step 1 — `unlockQueue` (operator-triggered, lines 305–307):** [1](#0-0) 
rsETH is permanently burned and ETH is redeemed from `LRTUnstakingVault` into the manager. This is irreversible.

**Step 2 — `_processWithdrawalCompletion` (lines 699–738):** [2](#0-1) 
`popFront()` (line 705), `delete withdrawalRequests[requestId]` (line 712), and `unlockedWithdrawalsCount[asset]--` (line 717) all execute before `_transferAsset` at line 734. If `_transferAsset` reverts, the entire transaction is rolled back — all three state mutations are undone, leaving the request permanently stuck.

**`_transferAsset` for ETH (lines 876–883):** [3](#0-2) 
A low-level `call{value: amount}("")` to a contract without `receive()` returns `false`, triggering `revert EthTransferFailed()`. This revert propagates up and unwinds all state changes in `_processWithdrawalCompletion`.

**No recovery path:**
- `completeWithdrawalForUser` (line 202) calls the same `_processWithdrawalCompletion` and hits the identical revert. [4](#0-3) 
- `sweepRemainingAssets` is permanently blocked because `hasUnlockedWithdrawals(asset)` returns `true` as long as `unlockedWithdrawalsCount[asset] > 0`, which is never decremented due to the revert. [5](#0-4) [6](#0-5) 
- There is no admin function to redirect an unlocked withdrawal to an alternate recipient or cancel it.

## Impact Explanation
**Critical — Permanent freezing of funds.** The user's rsETH is irreversibly burned during `unlockQueue`. The corresponding ETH is trapped in `LRTWithdrawalManager` with no on-chain mechanism to release it absent a contract upgrade. This matches the allowed Critical impact class of "Permanent freezing of funds."

## Likelihood Explanation
**Low.** The affected user must be a smart contract that: (1) holds rsETH, (2) calls `initiateWithdrawal` with `asset == ETH_TOKEN`, and (3) lacks a `receive()` or `fallback()` function. This is realistic for protocol integrations, vaults, or smart contract wallets that hold rsETH but were not designed to receive raw ETH. The developer comment at line 191 — *"Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH"* — acknowledges ETH-delivery edge cases but incorrectly dismisses the permanent-freeze scenario. [7](#0-6) 

## Recommendation
Validate that the withdrawing address can receive ETH **before** accepting the withdrawal request in `initiateWithdrawal`. Options include:
1. Attempt a zero-value ETH transfer to `msg.sender` at initiation time and revert if it fails.
2. Restrict ETH withdrawals to EOAs only (`require(msg.sender.code.length == 0)`).
3. Add an admin escape hatch that can redirect an unlocked withdrawal to an alternate recipient when the primary recipient cannot receive ETH.

## Proof of Concept
```solidity
contract NoReceiveWallet {
    function initiateETHWithdrawal(address withdrawalManager, address rsETH, uint256 amount) external {
        IERC20(rsETH).approve(withdrawalManager, amount);
        ILRTWithdrawalManager(withdrawalManager).initiateWithdrawal(ETH_TOKEN, amount, "");
        // rsETH transferred to LRTWithdrawalManager; withdrawal request created
    }
    // No receive() function
}
```

1. `NoReceiveWallet.initiateETHWithdrawal(...)` — rsETH moves to manager, request created.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)` — rsETH **burned** (line 305), ETH redeemed from vault (line 307).
3. `NoReceiveWallet` calls `completeWithdrawal(ETH_TOKEN, "")` — `_transferAsset` attempts `payable(NoReceiveWallet).call{value: amount}("")`, returns `false`, reverts with `EthTransferFailed()`. All state changes roll back.
4. Operator calls `completeWithdrawalForUser(ETH_TOKEN, address(NoReceiveWallet), "")` — same revert.
5. `sweepRemainingAssets(ETH_TOKEN)` — reverts with `PendingWithdrawalsExist()` because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.
6. **ETH is permanently locked in `LRTWithdrawalManager`. rsETH is permanently burned.**

Foundry fork test plan: deploy `NoReceiveWallet`, fund with rsETH, call `initiateWithdrawal`, advance blocks past `withdrawalDelayBlocks`, call `unlockQueue`, then assert that `completeWithdrawal` reverts with `EthTransferFailed` and that `address(withdrawalManager).balance` equals the expected ETH amount indefinitely.
