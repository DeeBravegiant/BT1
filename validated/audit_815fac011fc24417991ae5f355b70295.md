Audit Report

## Title
Withdrawal Payout Permanently Capped at Initiation-Time Oracle Price, Stripping Accrued Yield from Withdrawing Users - (File: contracts/LRTWithdrawalManager.sol)

## Summary
When a user initiates a withdrawal, `expectedAssetAmount` is computed from the rsETH oracle price at that moment and stored in the `WithdrawalRequest` struct. During the mandatory 8–16 day waiting period the rsETH sits in the contract and appreciates, but `_calculatePayoutAmount` returns `min(expectedAssetAmount_at_initiation, currentReturn_at_unlock)`. Because rsETH price monotonically increases as restaking yield accrues, `currentReturn_at_unlock` always exceeds `expectedAssetAmount_at_initiation`, so every normal withdrawal is permanently capped at the stale initiation-time value. The surplus value — yield that accrued on the user's locked rsETH — is retained by the protocol.

## Finding Description

**Initiation — price locked:** [1](#0-0) 

`getExpectedAssetAmount` computes `rsETHUnstaked * rsETHPrice / assetPrice` using the oracle price at call time and passes it to `_addUserWithdrawalRequest`, which stores it verbatim: [2](#0-1) 

The rsETH is transferred into the contract at initiation and held there: [3](#0-2) 

**Unlock — min() caps the payout:** [4](#0-3) 

`currentReturn` is recomputed with the current oracle price, but the ternary returns `request.expectedAssetAmount` whenever `currentReturn > expectedAssetAmount` — i.e., whenever the rsETH price has risen since initiation. The updated (lower) `payoutAmount` is written back into `request.expectedAssetAmount`: [5](#0-4) 

**rsETH burned at unlock, not initiation:** [6](#0-5) 

The full `rsETHUnstaked` amount (now worth `currentReturn` at the higher price) is burned, but the user receives only `expectedAssetAmount_at_initiation`. The difference — `rsETHUnstaked × (P₁ − P₀) / assetPrice` — is never disbursed; it remains as surplus assets in the protocol, benefiting remaining rsETH holders.

**No existing guard prevents this.** `_validatePrices` only enforces operator-supplied min/max bounds on the oracle prices; it does not alter the `min()` logic. `assetsCommitted` accounting is decremented by the original `expectedAssetAmount` (line 802), so the surplus is silently freed back into the pool rather than returned to the user.

## Impact Explanation

**High — Theft of unclaimed yield.**

The yield that accrues on the user's locked rsETH position during the mandatory waiting period (8 days default, up to 16 days) is a concrete, quantifiable amount that belongs to the withdrawing user but is instead absorbed by the protocol. This matches the allowed impact class "Theft of unclaimed yield." At a 4% annualised restaking APY over 8 days, a 10,000 ETH withdrawal queue loses approximately 8.8 ETH per cycle; at scale the loss is material and repeatable.

## Likelihood Explanation

**High.** rsETH price increases continuously as EigenLayer restaking rewards accrue and as execution-layer rewards are forwarded. The withdrawal delay is initialised to `8 days / 12 seconds` and capped at `16 days / 12 seconds`: [7](#0-6) [8](#0-7) 

Every withdrawal that experiences any non-zero price appreciation between initiation and unlock is affected. No special attacker capability is required — any ordinary user who calls `initiateWithdrawal` and later `completeWithdrawal` is a victim.

## Recommendation

In `_calculatePayoutAmount`, return `currentReturn` when `currentReturn >= expectedAssetAmount_at_initiation` so that users receive the full current value of their rsETH at unlock time. The `min()` guard is still correct for the slashing case (price drop), but the cap should not apply when the price has risen. Concretely, invert the ternary:

```solidity
return (currentReturn < request.expectedAssetAmount) ? currentReturn : request.expectedAssetAmount;
// should become:
return (currentReturn < request.expectedAssetAmount) ? currentReturn : currentReturn;
// i.e., simply: return currentReturn;
// with the slashing floor preserved:
return currentReturn < request.expectedAssetAmount ? currentReturn : request.expectedAssetAmount;
// → replace with:
return currentReturn; // slashing already handled: if price dropped, currentReturn < expectedAssetAmount
```

More precisely: remove the upper cap on `currentReturn` and keep only the lower bound (slashing protection). Users should receive `currentReturn` at unlock time, not `min(initiation_value, current_value)`.

## Proof of Concept

1. rsETH price = 1.10 ETH/rsETH. User calls `initiateWithdrawal(ETH, 100e18)`.
   - `expectedAssetAmount = 100e18 × 1.10 / 1.0 = 110 ETH` stored in request.
   - 100 rsETH transferred to `LRTWithdrawalManager`.

2. 8 days pass. rsETH price rises to 1.12 ETH/rsETH (≈4% APY over 8 days).

3. Operator calls `unlockQueue(ETH, ...)`.
   - `currentReturn = 100e18 × 1.12 / 1.0 = 112 ETH`
   - `_calculatePayoutAmount` returns `min(110, 112) = 110 ETH`
   - `request.expectedAssetAmount` overwritten to 110 ETH
   - 100 rsETH (worth 112 ETH) burned; only 110 ETH allocated to user

4. User calls `completeWithdrawal(ETH, ...)` and receives 110 ETH.

5. **Loss**: 2 ETH of yield accrued on the user's locked position is retained by the protocol. Foundry fork test: deploy against a mainnet fork, warp 8 days, mock the oracle to return a higher rsETH price, call `unlockQueue`, assert `withdrawalRequests[requestId].expectedAssetAmount < rsETHUnstaked * newPrice / assetPrice`.
