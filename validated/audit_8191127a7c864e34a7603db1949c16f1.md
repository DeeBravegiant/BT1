### Title
Missing `x_share * G == Q` Consistency Check in `eddsa2pc` Key Blob Deserialization Allows Signing to Succeed with Mismatched Private Share — (`src/cbmpc/api/eddsa2pc.cpp`)

### Summary

The `blob_to_key` function in `src/cbmpc/api/eddsa2pc.cpp` does not verify that `x_share * G` equals the stored public key share. An API caller who controls the key blob bytes can substitute any in-range scalar for `x_share`, pass all deserialization guards, and drive `eddsa_2p::sign` to completion — returning `SUCCESS` and a 64-byte signature that fails external Ed25519 verification against the original public key `Q`.

### Finding Description

`blob_to_key` performs three checks on the deserialized blob: [1](#0-0) 

```
if (!q.is_in_range(blob.x_share)) return error(E_FORMAT, ...);  // range only
key.Q.from_bin(...)                                               // Q is a valid point
key.curve.check(key.Q)                                           // Q is on the curve
```

There is **no** `x_share * G == Q` (or `x_share * G == Qi_self`) check. The blob format stores only `Q_compressed` and `x_share`; it does not store the individual share point `Qi`, so there is nothing to compare against without computing `x_share * G` on the fly.

By contrast, `eddsa_mp::deserialize_key_blob` explicitly performs this check: [2](#0-1) 

```cpp
if (blob.x_share * G != it_self->second) return error(E_FORMAT, "invalid key blob");
```

The `eddsa_2p::sign` API path is:

1. `coinbase::api::eddsa_2p::sign` → `deserialize_key_blob` → `blob_to_key` (no consistency check) [3](#0-2) 
2. → `coinbase::mpc::eddsa2pc::sign` → `schnorr2p::sign` (no key consistency check at the protocol layer) [4](#0-3) 

The `attach_private_scalar` API (the restore path) explicitly documents and enforces `x*G == public_share_compressed`: [5](#0-4) 

This confirms the library designers know the check is necessary — it is simply absent from the direct deserialization path used by `sign` and `refresh`.

### Impact Explanation

An API caller who holds a valid `eddsa2pc` key blob (e.g., output of `derive_eddsa_2p_keys` or `dkg`) can:

1. Deserialize the blob bytes.
2. Replace `x_share` with any value in `[0, q)` (e.g., `x_share + 1`).
3. Re-serialize and pass the tampered blob to `eddsa_2p::sign`.
4. `deserialize_key_blob` / `blob_to_key` accepts it — all guards pass.
5. The 2-party signing protocol runs to completion and returns `SUCCESS`.
6. The resulting 64-byte signature is computed under the modified `x_share` but the embedded public key `Q` is the original, so the signature **fails external Ed25519 verification**.

The API contract is violated: `sign` returns `SUCCESS` with cryptographically invalid output. The honest peer (P2) participates in the protocol without any indication that P1's share is inconsistent with the agreed public key. This fits the scope criterion: **"Public API reachable validation bypass in signing creates accepted invalid cryptographic output."**

### Likelihood Explanation

The attack requires only the ability to supply an attacker-controlled key blob to the `sign` (or `refresh`) API — a standard caller capability. No threshold collusion, no memory compromise, and no network-level attack is needed. The `eddsa_mp` counterpart already has the fix, making this a straightforward oversight in the 2-party path.

### Recommendation

Add the consistency check in `blob_to_key` in `src/cbmpc/api/eddsa2pc.cpp`, mirroring the `eddsa_mp` pattern:

```cpp
const auto& G = key.curve.generator();
if (key.x_share * G + /* other_share_point */ != key.Q) {
    return coinbase::error(E_FORMAT, "invalid key blob: x_share * G mismatch");
}
```

Because the 2-party blob does not store the individual share point `Qi`, the check must either:
- Compute `x_share * G` and store/compare it against a persisted `Qi_compressed` field added to the blob format, or
- At minimum, verify that `x_share * G` is a valid curve point that could plausibly be a share (i.e., is not the identity and is in the subgroup).

The `attach_private_scalar` API already demonstrates the correct pattern and should be the reference implementation.

### Proof of Concept

```
1. Run eddsa_2p::dkg(job, curve_id::ed25519, key_blob_1) for P1.
2. Deserialize key_blob_1 into key_blob_v1_t blob.
3. Set blob.x_share = blob.x_share + 1 (mod q).
4. Re-serialize into tampered_blob.
5. Call eddsa_2p::sign(job, tampered_blob, msg, sig) for both parties.
6. Assert: sign returns SUCCESS and sig.size() == 64.
7. Retrieve Q via get_public_key_compressed(key_blob_1, pub).
8. Assert: Ed25519 verify(pub, msg, sig) FAILS.
```

The `blob_to_key` range check (`q.is_in_range`) passes for `x_share + 1` as long as the original `x_share` is not `q-1`. The signing protocol completes normally. The signature is invalid against the original `Q`. [6](#0-5) [2](#0-1)

```cpp
static error_t blob_to_key(const key_blob_v1_t& blob, coinbase::mpc::eddsa2pc::key_t& key) {
  if (blob.role > 1) return coinbase::error(E_FORMAT, "invalid key blob role");
  if (static_cast<curve_id>(blob.curve) != curve_id::ed25519)
    return coinbase::error(E_FORMAT, "invalid key blob curve");

  key.role = static_cast<coinbase::mpc::party_t>(static_cast<int32_t>(blob.role));
  key.curve = coinbase::crypto::curve_ed25519;
  const coinbase::crypto::mod_t& q = key.curve.order();
  if (!q.is_in_range(blob.x_share)) return coinbase::error(E_FORMAT, "invalid key blob");
  key.x_share = blob.x_share;

  error_t rv = key.Q.from_bin(key.curve, blob.Q_compressed);
  if (rv) return coinbase::error(rv, "invalid key blob");
  if (key.curve.check(key.Q)) return coinbase::error(E_FORMAT, "invalid key blob");
  return SUCCESS;
}
```
