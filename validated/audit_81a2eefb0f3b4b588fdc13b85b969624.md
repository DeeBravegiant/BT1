Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Amount in Boundary Check — (`contracts/LRTDepositPool.sol`)

## Summary

`_checkIfDepositAmountExceedesCurrentLimit` uses `totalAssetDeposits > limit` for ETH but `totalAssetDeposits + amount > limit` for ERC-20 tokens. The missing `amount` operand in the ETH branch means the check passes (returns `false`) when `totalAssetDeposits == depositLimit`, allowing any depositor to push the ETH total above the configured cap by an arbitrary amount in a single transaction.

## Finding Description

In `contracts/LRTDepositPool.sol` at lines 676–682, the function reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ETH branch evaluates only the pre-deposit total against the limit. When `totalAssetDeposits == depositLimit`, the expression `totalAssetDeposits > limit` is `false`, so `_beforeDeposit` (L661–663) does not revert with `MaximumDepositLimitReached`, and the deposit proceeds. The call path is:

```
depositETH{value: X}(...)          // L76, public payable
  └─ _beforeDeposit(ETH_TOKEN, X)  // L87, L648
       └─ _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, X)  // L661, L676
            └─ returns false when totalAssetDeposits == limit  ← bug
```

No other guard in `depositETH` or `_beforeDeposit` compensates for this omission. The ERC-20 equivalent path (`depositAsset`) correctly includes `amount` and is unaffected.

## Impact Explanation

The ETH deposit cap is a risk-management control bounding the protocol's EigenLayer restaking exposure. When the running total is exactly at the limit, any caller can deposit an unbounded `msg.value` and receive rsETH, silently violating the ceiling. The overshoot persists on-chain; subsequent deposits are blocked only after the fact. This maps to **Low — contract fails to deliver promised returns (deposit limit not enforced for ETH)**: no funds are stolen and no funds are frozen, but the protocol does not honour its stated deposit ceiling.

## Likelihood Explanation

`depositETH` is a public, permissionless, payable function. The trigger condition (`totalAssetDeposits == depositLimit`) is a normal steady-state: the limit is designed to be reached. Any depositor monitoring on-chain state can exploit this at the exact moment the total hits the cap. No privileged access, oracle manipulation, or victim mistake is required. Likelihood: **Medium**.

## Recommendation

Apply the same expression used for ERC-20 tokens to the ETH branch in `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This makes both paths consistent and correctly rejects any deposit that would push the running total above the configured limit.

## Proof of Concept

1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. Normal deposits accumulate until `getTotalAssetDeposits(ETH_TOKEN) == 1000 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `1000 ether > 1000 ether` → `false` → no revert.
5. 500 ETH is accepted; `totalAssetDeposits` becomes 1500 ETH — 50% above the cap.
6. Equivalent ERC-20 call evaluates `1000 + 500 > 1000` → `true` → correctly reverted.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    vm.prank(admin);
    lrtConfig.updateAssetDepositLimit(ETH_TOKEN, 1000 ether);
    // fill to exactly the limit via prior deposits ...
    // attacker deposits 500 ETH — should revert but does not
    vm.deal(attacker, 500 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 500 ether}(0, "");
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
```
