### Title
Byzantine Participant Can Inflate `maximum_height` via Crafted `IndexerHeightMessage`, Causing Premature `RequestTimedOut` Drops for All Queued Requests — (`crates/node/src/network/indexer_heights.rs`, `crates/node/src/requests/queue.rs`)

### Summary
A single Byzantine MPC participant that is bidirectionally connected can send an `IndexerHeightMessage` with a large but non-overflowing height (e.g., `10^18`). Because `set_height` accepts any value greater than the current stored value with no upper-bound validation, the inflated height is stored permanently. `eligible_leaders_and_maximum_height` then computes `maximum_height = 10^18`, which drives `cutoff_block` to an astronomically large value. Every queued request with a realistic block height is immediately dropped as `RequestTimedOut`, and all honest participants are excluded from `eligible_leaders`, leaving the Byzantine participant as the sole eligible leader.

### Finding Description
`IndexerHeightTracker::set_height` stores any height value greater than the current one without any upper-bound check:

```rust
// crates/node/src/network/indexer_heights.rs:21-26
pub fn set_height(&self, participant: ParticipantId, height: u64) {
    let atomic = self.heights.get(&participant).unwrap();
    let current = atomic.load(std::sync::atomic::Ordering::Relaxed);
    if height > current {
        atomic.store(height, std::sync::atomic::Ordering::Relaxed);
    }
```

The message handler applies no validation before calling this:

```rust
// crates/node/src/network.rs:400-402
PeerMessage::IndexerHeight(message) => {
    indexer_heights.set_height(message.from, message.message.height);
}
```

`eligible_leaders_and_maximum_height` derives `maximum_height` as the max over all alive participants' heights, then computes:

```rust
// crates/node/src/requests/queue.rs:604-606
let cutoff_block: BlockHeight =
    (maximum_height.saturating_sub(REQUEST_EXPIRATION_BLOCKS) + 1).into();
```

With `maximum_height = 10^18` and `REQUEST_EXPIRATION_BLOCKS = 200`, `cutoff_block ≈ 10^18 - 199`. Every request with a realistic block height (~130 M) satisfies `block_height < cutoff_block` and is dropped via `is_older_than`:

```rust
// crates/node/src/requests/queue.rs:378-380
if self.is_older_than(cutoff_block) {
    return RequestStatus::Drop(DropReason::RequestTimedOut);
}
```

The `STALE_PARTICIPANT_THRESHOLD` filter simultaneously excludes all honest participants from `eligible_leaders`, making the Byzantine participant the sole eligible leader for any new requests.

**Note on `u64::MAX`:** The question's specific value of `u64::MAX` does not work because the `alive_participants` check performs `peer_height + MAX_INDEXER_HEIGHT_DIFF` which overflows in release mode, wrapping to a small number and excluding the Byzantine participant. The correct attack value is any large number that avoids this overflow (e.g., `10^18`).

### Impact Explanation
All currently queued signing/CKD/verify-foreign-tx requests are permanently dropped before they have actually expired on-chain, breaking the invariant that request expiry is determined by on-chain block height. The MPC network stops processing requests entirely. Users' signing requests are silently discarded; the on-chain contract eventually times them out. The inflated height persists for the node's lifetime and survives restarts (the Byzantine participant re-sends the message immediately). This is a request-lifecycle invariant break caused by a single sub-threshold Byzantine participant.

### Likelihood Explanation
Any single MPC participant that achieves bidirectional connectivity (the normal operating state) can trigger this with one crafted message. No threshold collusion, no key material, no network-level DoS is required. The attack is trivially repeatable after every restart.

### Recommendation
1. **Cap accepted heights**: In `set_height`, reject any height that exceeds the local node's own current height by more than a configurable sanity bound (e.g., `MAX_INDEXER_HEIGHT_DIFF`).
2. **Use `saturating_add` in `all_alive_participant_ids`**: Replace `peer_height + mpc_node_config::MAX_INDEXER_HEIGHT_DIFF` with `peer_height.saturating_add(...)` to prevent overflow-based bypass.
3. **Derive `maximum_height` only from honest-looking heights**: Cross-check reported heights against the local node's own indexed height before allowing a peer's height to dominate `maximum_height`.

### Proof of Concept
```rust
// Using the existing TestNetworkAPI in crates/node/src/requests/queue.rs
let participants = vec![p0, p1, p2]; // p1 is Byzantine
let network = Arc::new(TestNetworkAPI::new(&participants));

// All participants alive and connected

network.set_height(p2, 130_000_000); // honest
network.set_height(p1, 1_000_000_000_000_000_000u64); // Byzantine: 10^18

let mut pending = PendingRequests::<SignatureRequest, _>::new(
    clock.clone(), participants.clone(), p0, network.clone()
);

// Add a request at a realistic block height
pending.notify_new_block(RequestsUpdate {
    requests: vec![realistic_request],
    block_height: 130_000_100.into(),
    // ...
});

// Advance clock by CHECK_EACH_REQUEST_INTERVAL
let attempts = pending.get_requests_to_attempt();
// Assert: attempts is empty and the request was dropped with RequestTimedOut
// because cutoff_block = 10^18 - 199 >> 130_000_100
assert!(attempts.is_empty());
```
