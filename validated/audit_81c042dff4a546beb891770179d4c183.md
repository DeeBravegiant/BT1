Audit Report

## Title
`_updateRsETHPrice` Early Return Leaves Stale Inflated `rsETHPrice` After Unpause, Enabling Excess Asset Extraction via `instantWithdrawal` - (File: contracts/LRTOracle.sol)

## Summary
In `LRTOracle._updateRsETHPrice()`, when the downside-protection circuit-breaker fires, the function pauses all three contracts and returns at line 281 before ever executing `rsETHPrice = newRsETHPrice` at line 313. The stale, pre-drop price persists in storage. After the admin unpauses the protocol, any user can call `LRTWithdrawalManager.instantWithdrawal()` before `updateRSETHPrice()` is called, burning rsETH at the inflated stale price and extracting excess assets from the protocol's TVL.

## Finding Description

**Root cause — early return before price write:**

`_updateRsETHPrice()` computes `newRsETHPrice` at line 250, then enters the downside-protection block at lines 270–282:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;   // ← exits here
}
``` [1](#0-0) 

The assignment `rsETHPrice = newRsETHPrice` sits at line 313, which is never reached when the early return fires: [2](#0-1) 

`updateRSETHPrice()` carries a `whenNotPaused` guard, so the price cannot be corrected while the oracle is paused: [3](#0-2) 

**Exploit path:**

After the admin unpauses all three contracts, `instantWithdrawal()` becomes callable again. It reads `rsETHPrice` directly through `getExpectedAssetAmount`: [4](#0-3) 

`getExpectedAssetAmount` computes `amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)` with no staleness check: [5](#0-4) 

Because `rsETHPrice` is the pre-drop (inflated) value, `assetAmountUnlocked` is larger than the current TVL warrants. The attacker burns rsETH and receives excess assets immediately, with no recalculation step. `instantWithdrawal()` is open to any user — no role restriction applies: [6](#0-5) 

**Why existing checks are insufficient:**

- The `whenNotPaused` guard on `instantWithdrawal()` only prevents exploitation while paused; it provides no protection after unpause.
- There is no price-freshness or last-updated-timestamp check anywhere in `instantWithdrawal()` or `getExpectedAssetAmount()`.
- The `CantInstantWithdrawMoreThanAvailable` check at line 231 limits withdrawal to available vault assets, but those assets are priced at the stale rate, so the check does not prevent over-extraction per rsETH unit. [7](#0-6) 

## Impact Explanation

**Critical — direct theft of user funds.** An attacker burns rsETH at the stale (pre-drop) `rsETHPrice` and receives assets priced at that inflated rate. The excess assets come directly from the protocol's TVL, diluting all remaining rsETH holders. The magnitude per call is `(stalePriceRatio − 1) × rsETHBurned × assetPrice`. The attacker can repeat until `updateRSETHPrice()` is called and the price is corrected. This matches the allowed impact: *Critical — Direct theft of any user funds*.

## Likelihood Explanation

Three conditions must align:
1. TVL drops by more than `pricePercentageLimit` in a single oracle update (e.g., a large EigenLayer slashing event or a supported LST depeg). This is a realistic operational event.
2. The admin subsequently unpauses the protocol. This is the expected operational response after investigating the cause of the price drop.
3. An attacker calls `instantWithdrawal()` before `updateRSETHPrice()` is called post-unpause. This requires no special privileges and no front-running in the strict sense — the window may persist for multiple blocks if the admin does not atomically unpause and update the price.

Instant withdrawal must be enabled for the target asset (`isInstantWithdrawalEnabled[asset] == true`), which is a normal operational configuration. Overall likelihood is **Low–Medium**, yielding a **Critical** severity finding.

## Recommendation

Write `rsETHPrice = newRsETHPrice` before calling `_pause()` and returning in the downside-protection block, so the stored price always reflects the actual current state even when the circuit-breaker fires:

```solidity
if (isPriceDecreaseOffLimit) {
    rsETHPrice = newRsETHPrice;          // ← add this line
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    emit RsETHPriceUpdate(rsETHPrice, previousPrice);
    return;
}
```

This ensures that after unpause, any call to `instantWithdrawal()` or `unlockQueue()` that races ahead of `updateRSETHPrice()` uses the correct (post-drop) price rather than the stale inflated one.

## Proof of Concept

1. Deploy with `pricePercentageLimit = 1e16` (1%).
2. Simulate a TVL drop: manipulate or observe `_getTotalEthInProtocol()` returning a value such that `newRsETHPrice = 0.97e18` while `rsETHPrice = 1.00e18` and `highestRsethPrice = 1.00e18`.
3. Call `LRTOracle.updateRSETHPrice()`. Inside `_updateRsETHPrice()`, `isPriceDecreaseOffLimit` is `true` (diff = 0.03e18 > 0.01e18). The function pauses all three contracts and returns. `rsETHPrice` remains `1.00e18`.
4. Admin calls `unpause()` on `LRTOracle`, `LRTDepositPool`, and `LRTWithdrawalManager`.
5. Attacker (holding 100e18 rsETH) calls `LRTWithdrawalManager.instantWithdrawal(ETH, 100e18, "")`.
6. `getExpectedAssetAmount` returns `100e18 * 1.00e18 / 1e18 = 100 ETH` instead of the correct `97 ETH`.
7. Attacker receives 100 ETH, burning rsETH worth only 97 ETH at current prices — a 3 ETH gain at the expense of remaining holders.
8. Attacker repeats until `updateRSETHPrice()` is called.

**Foundry test sketch:**
```solidity
