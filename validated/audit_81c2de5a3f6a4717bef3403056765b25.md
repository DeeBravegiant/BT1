Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Calculation Allows Depositors to Steal Accrued Yield — (File: `contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, which is a plain storage variable in `LRTOracle` that is only refreshed when `updateRSETHPrice()` is explicitly called. Because the deposit flow (`depositETH` → `_beforeDeposit` → `getRsETHAmountToMint`) never calls `updateRSETHPrice()`, any deposit made while staking rewards have accrued since the last price update uses a denominator that is lower than the true price, causing the depositor to receive excess rsETH at the expense of existing holders' unclaimed yield.

## Finding Description
`rsETHPrice` is a plain state variable in `LRTOracle`:

```solidity
// LRTOracle.sol L28
uint256 public override rsETHPrice;
```

It is only written inside `_updateRsETHPrice()` (L313), which is only reachable via the public `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`. Neither is called anywhere in the deposit path.

The deposit path is:
1. `depositETH()` / `depositAsset()` → `_beforeDeposit()` (L648–670)
2. `_beforeDeposit()` calls `getRsETHAmountToMint()` (L665)
3. `getRsETHAmountToMint()` computes (L520):
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```

Between two `updateRSETHPrice()` calls, staking rewards cause TVL to grow while `totalSupply` and the stored `rsETHPrice` remain unchanged. The stored price is therefore **lower** than the true current price. Using it as the denominator inflates `rsethAmountToMint`.

`_updateRsETHPrice()` computes `newRsETHPrice` using the **pre-fee-mint** supply (L250), mints fee rsETH to treasury (L306), then stores `rsETHPrice = newRsETHPrice` (L313). This means the stored price already accounts for the fee deduction — but only as of the last update. Any rewards that accrue after that point are not reflected until the next explicit call.

No existing check in the deposit path compensates for this: `whenNotPaused` only blocks paused state, `onlySupportedAsset` checks asset support, and `minRSETHAmountExpected` is a user-supplied slippage parameter that the attacker sets to zero or a favorable value.

## Impact Explanation
Every deposit made while `rsETHPrice` is stale mints more rsETH than the deposited value warrants. The excess rsETH represents a proportional claim on the yield that has accrued since the last price update — yield that rightfully belongs to existing rsETH holders. This constitutes **theft of unclaimed yield** (High). The magnitude scales with both the size of the deposit and the duration since the last `updateRSETHPrice()` call, which in practice can span hours or days.

## Likelihood Explanation
Staking rewards accumulate every block. `updateRSETHPrice()` is a standalone public function with no automatic trigger in the deposit flow. Any unprivileged depositor can call `depositETH()` or `depositAsset()` at any time without preconditions. No special role, flash loan, or oracle manipulation is required. The exploit is repeatable on every deposit made in the interval between two `updateRSETHPrice()` calls.

## Recommendation
At the start of `_beforeDeposit()` in `LRTDepositPool.sol`, call `ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice()` before `getRsETHAmountToMint()` reads `rsETHPrice`. This ensures the price is always fresh at deposit time, mirroring the protocol's own fee-accounting intent.

## Proof of Concept
1. **Initial state**: TVL = 1 000 ETH, rsETH `totalSupply` = 1 000, stored `rsETHPrice` = 1.0 ETH. `updateRSETHPrice()` was last called here.
2. **Rewards accrue**: TVL grows to 1 010 ETH (10 ETH staking reward). `updateRSETHPrice()` is NOT called. Stored `rsETHPrice` remains 1.0 ETH.
3. **True price** (with 10% protocol fee on 10 ETH reward = 1 ETH fee): `(1010 − 1) / 1000 = 1.009 ETH`.
4. **Attacker calls** `depositETH{value: 10 ether}(0, "")`:
   - `getRsETHAmountToMint` computes: `10e18 * 1e18 / 1.0e18 = 10 rsETH`.
   - Correct amount at true price: `10e18 / 1.009e18 ≈ 9.911 rsETH`.
   - **Excess minted: ≈ 0.089 rsETH** — stolen from existing holders.
5. When `updateRSETHPrice()` is eventually called, it mints fee rsETH to treasury and updates the price, but the attacker's excess rsETH is already minted and irrecoverable.

Foundry fork test plan: fork mainnet, advance time/blocks to accumulate rewards without calling `updateRSETHPrice()`, call `depositETH` as an unprivileged address, assert that `rsETH.balanceOf(attacker) > getRsETHAmountToMint(asset, amount)` computed with the post-update price, and assert that existing holders' redemption value per rsETH has decreased relative to a baseline where `updateRSETHPrice()` was called first.
