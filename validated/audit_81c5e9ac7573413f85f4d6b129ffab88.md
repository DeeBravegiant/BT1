Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Minting Allows Depositors to Capture Accrued Yield from Existing Holders - (`contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool.depositETH` and `depositAsset` compute the rsETH mint amount using the stored `LRTOracle.rsETHPrice` state variable, which is never refreshed atomically before a deposit. As staking rewards accrue, the true rsETH price rises above the stored value, causing any depositor to receive more rsETH than their deposit warrants, diluting the yield owed to pre-existing holders.

## Finding Description
`LRTOracle.rsETHPrice` is a plain state variable (line 28) updated only by explicit calls to `_updateRsETHPrice()` (line 313). The public entry point `updateRSETHPrice()` (line 87) is unrestricted but is never invoked inside the deposit path.

The deposit flow is:
- `depositETH` / `depositAsset` → `_beforeDeposit` (line 665) → `getRsETHAmountToMint` (line 520)

`getRsETHAmountToMint` computes:
```
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
`lrtOracle.rsETHPrice()` returns the last-written stored value, not the live price. The live price is computed in `_updateRsETHPrice()` as `(totalETHInProtocol - protocolFeeInETH) / rsethSupply` (line 250), where `totalETHInProtocol` grows continuously as EigenLayer/LST staking rewards accrue.

When `rsETHPrice` is stale (lower than the true value), the division yields a larger rsETH amount than the depositor deserves. After the deposit, any caller — including the attacker — can invoke `updateRSETHPrice()` to push the price to its correct higher value. The attacker's rsETH is now backed by more ETH than they deposited, at the expense of pre-existing holders whose proportional TVL claim was diluted.

Existing guards are insufficient:
- `minRSETHAmountExpected` (line 667) only protects the depositor from receiving *less* than expected; it does not prevent over-minting.
- `pricePercentageLimit` (line 257) limits per-update price jumps but does not prevent the stale-price deposit window from existing.
- `nonReentrant` and `whenNotPaused` are orthogonal to this issue.

## Impact Explanation
**High — Theft of unclaimed yield.** Every pre-existing rsETH holder's proportional claim on protocol TVL is diluted. The attacker receives rsETH backed by more ETH than they deposited; the difference is yield that accrued to existing holders before the attacker's entry. The magnitude scales with (a) the duration since the last `updateRSETHPrice()` call and (b) the staking yield accrued in that window.

## Likelihood Explanation
`updateRSETHPrice()` is a public function callable by any EOA or contract. It is not called on-chain by any keeper or by the deposit path itself. Any gap between off-chain bot updates (network congestion, bot downtime, or deliberate delay) creates an exploitable window. MEV bots can monitor the mempool for pending `updateRSETHPrice()` transactions and front-run them with a large deposit, then back-run with a withdrawal request after the price update, making this repeatable and automatable.

## Recommendation
Call `updateRSETHPrice()` (or an equivalent internal function) at the beginning of `depositETH` and `depositAsset`, before `_beforeDeposit` computes `rsethAmountToMint`. This ensures the price used for minting always reflects the current TVL and eliminates the stale-price window entirely.

## Proof of Concept
1. Staking rewards accrue for several hours; `updateRSETHPrice()` has not been called. True rsETH price is 1.005 ETH but stored `rsETHPrice` is 1.000 ETH.
2. Attacker calls `depositETH{value: 100 ETH}(0, "")`.
   - `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.000e18 = 100 rsETH`.
   - Correct amount: `100e18 * 1e18 / 1.005e18 ≈ 99.5 rsETH`.
   - Attacker receives ~0.5 rsETH extra.
3. Attacker (or anyone) calls `updateRSETHPrice()`. Price updates to 1.005 ETH.
4. Attacker's 100 rsETH is now worth `100 × 1.005 = 100.5 ETH` — a profit of ~0.5 ETH funded by diluting existing holders.

**Foundry fork test plan:** Deploy against a mainnet fork with a known rsETH supply and TVL. Warp time forward to simulate reward accrual without calling `updateRSETHPrice()`. Record `rsETHPrice` before and after `_getTotalEthInProtocol()` to confirm divergence. Execute `depositETH` and assert that the minted rsETH exceeds `depositAmount / truePrice`. Call `updateRSETHPrice()` and assert the attacker's rsETH value exceeds their deposit, while existing holders' per-rsETH ETH value is unchanged (confirming dilution).
