### Title
Governance Action Deposit Permanently Redirected to Treasury via Unregistered Return Address During Conway Bootstrap Phase — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

---

### Summary

During the Conway bootstrap phase (protocol version 9), the `conwayGovTransition` rule in the `GOV` STS skips the `ProposalReturnAccountDoesNotExist` registration check on `pProcReturnAddr`. An unprivileged proposer can therefore submit a governance proposal whose deposit-return address is not a registered stake credential. When the proposal expires or is enacted, `returnProposalDeposits` in the `EPOCH` rule cannot credit the unregistered address and silently redirects the entire deposit to the treasury. The proposer permanently loses their deposit with no automated recovery path.

---

### Finding Description

**Root cause — missing whitelist/registration check during bootstrap**

In `conwayGovTransition` the return-address registration check is guarded by `unless (hardforkConwayBootstrapPhase ...)`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs  lines 504-508
unless (hardforkConwayBootstrapPhase $ pp ^. ppProtocolVersionL) $ do
  let refundAddress = proposal ^. pProcReturnAddrL
  isAccountRegistered (refundAddress ^. accountAddressCredentialL) (certDState ^. accountsL)
    ?! (injectFailure . ProposalReturnAccountDoesNotExist) refundAddress
```

`hardforkConwayBootstrapPhase` returns `True` exactly when `pvMajor pv == natVersion @9` (protocol version 9, the initial Conway era). During that window the entire `unless` block — including the registration check — is **not executed**. The deposit is accepted regardless of whether the return address is registered. [1](#0-0) [2](#0-1) 

The bootstrap phase still permits `ParameterChange`, `HardForkInitiation`, and `InfoAction` proposals (via `checkBootstrapProposal` / `isBootstrapAction`), so a proposer can legitimately reach the deposit-acceptance path with an unregistered return address. [3](#0-2) [4](#0-3) 

**Deposit loss path — `returnProposalDeposits` in `EPOCH`**

At every epoch boundary, `returnProposalDeposits` iterates over all removed (expired or enacted) proposals and attempts to credit each proposal's deposit to its `gasReturnAddr`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Epoch.hs  lines 187-193
processProposal gas (!accounts, !unclaimed)
  | (Just _accountState, newAccounts) <-
        updateLookupAccountState addRefund cred accounts =
      (newAccounts, unclaimed)
  | otherwise =
      (accounts, Map.insert (gasId gas) (gasDeposit gas) unclaimed)
  where
    cred = gasReturnAddr gas ^. accountAddressCredentialL
```

If `cred` is not registered, the `otherwise` branch fires and the deposit is placed in `unclaimed`. The `EPOCH` transition then adds all unclaimed amounts to the treasury:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Epoch.hs  lines 347-350
chainAccountState3 =
  chainAccountState2
    & casTreasuryL <>~ (utxoState0 ^. utxosDonationL <> fold unclaimed)
``` [5](#0-4) [6](#0-5) 

The deposit is now in the treasury. The proposer has no automated mechanism to reclaim it; recovery would require a separate `TreasuryWithdrawals` governance action ratified by a supermajority of DReps, SPOs, and the Constitutional Committee — effectively unrecoverable for an individual user.

The existing test `depositMovesToTreasuryWhenStakingAddressUnregisters` confirms this treasury-redirect behaviour when the return address is unregistered at expiry time, but it tests the post-registration-then-deregistration path, not the bootstrap-phase submission path. [7](#0-6) 

**Contrast with post-bootstrap behaviour**

Post-bootstrap the same check is enforced unconditionally, and the predicate failure `ProposalReturnAccountDoesNotExist` is returned, rejecting the transaction before any deposit is taken: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

An unprivileged proposer who submits a `ParameterChange`, `HardForkInitiation`, or `InfoAction` proposal during the bootstrap phase with an unregistered `pProcReturnAddr` will have their entire governance-action deposit permanently redirected to the treasury. On mainnet the deposit is set by `ppGovActionDepositL` (100,000 ADA at Conway launch). The loss is irreversible for the individual proposer without a separate supermajority governance action. This matches the **Medium** allowed impact: *attacker-controlled proposals modify deposits/refunds outside design parameters*. [10](#0-9) 

---

### Likelihood Explanation

The bootstrap phase is protocol version 9 (the initial Conway era). Any node processing blocks at that version is affected. A proposer who has not yet registered their stake credential — or who uses a freshly generated key as the return address without registering it — will silently lose their deposit. The error is not surfaced at submission time (no predicate failure is raised), so the proposer receives no warning. Likelihood is **Medium**: the window is bounded to the bootstrap phase, but the deposit amount is large and the failure is silent.

---

### Recommendation

Remove the `unless (hardforkConwayBootstrapPhase ...)` guard around the `isAccountRegistered` check for `pProcReturnAddr`, or apply the same check unconditionally for all protocol versions. The bootstrap restriction on *proposal types* (`checkBootstrapProposal`) is sufficient to limit what can be submitted; there is no design reason to also skip the return-address registration check during bootstrap.

---

### Proof of Concept

1. Network is at protocol version 9 (bootstrap phase).
2. Generate a fresh key pair `(vk, sk)`. Do **not** register the corresponding stake credential.
3. Construct a `ProposalProcedure` with:
   - `pProcReturnAddr = AccountAddress Mainnet (AccountId (KeyHashObj (hashKey vk)))` (unregistered)
   - `pProcGovAction = InfoAction`
   - `pProcDeposit = ppGovActionDepositL pp` (e.g., 100,000 ADA)
4. Submit the transaction. It is accepted — no `ProposalReturnAccountDoesNotExist` failure is raised because the check is skipped during bootstrap.
5. Advance epochs until `gasExpiresAfter < reCurrentEpoch`.
6. Observe: `returnProposalDeposits` hits the `otherwise` branch for this proposal; the deposit is added to `unclaimed`; `casTreasuryL` increases by the deposit amount; the proposer's balance is unchanged. [1](#0-0) [5](#0-4) [11](#0-10)
