### Title
Missing payload-hash-to-request binding in `respond_verify_foreign_tx` allows Byzantine leader to deliver forged foreign-chain attestation — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid threshold signature over `response.payload_hash`, but never verifies that `response.payload_hash` is actually derived from the `request` argument supplied in the same call. A single Byzantine MPC leader node that has obtained a valid threshold signature for one pending request can replay it verbatim to resolve a *different* pending request, causing the victim caller to receive a cryptographically valid but factually incorrect foreign-chain attestation.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs two checks and then resolves the pending request:

```
1. signature valid over response.payload_hash  ✓ (checked)
2. response.payload_hash derived from request  ✗ (NOT checked)
3. resolve_yields_for(&request, serialized_response)
``` [1](#0-0) 

The signed payload is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }))
``` [2](#0-1) 

The `ForeignChainRpcRequest` embedded inside the payload encodes the chain type (Bitcoin, Ethereum, etc.) and the transaction ID. The contract never reconstructs or checks this embedded request against the `request` argument passed to `respond_verify_foreign_tx`. Because the contract cannot know the extracted `values` (they are determined off-chain by MPC nodes), it cannot fully recompute the hash — but it could at minimum assert that the chain type and tx-id encoded in the payload hash match the `request` field. It does neither.

The `build_signature_request` helper on the node side uses a zero tweak (`Tweak::new([0u8; 32])`), so there is no per-request entropy that would cryptographically bind the threshold signature to a specific pending request key. [3](#0-2) 

---

### Impact Explanation

A Byzantine MPC leader node that participated in the threshold signing of **Request A** obtains the final signature `sig_A` over `hash_A`. It can then call:

```
respond_verify_foreign_tx(request = B, response = { payload_hash = hash_A, signature = sig_A })
```

The contract accepts this call because:
- `sig_A` is a valid threshold signature over `hash_A` → signature check passes.
- Request B exists in `pending_verify_foreign_tx_requests` → lookup succeeds. [4](#0-3) 

Request B's yield queue is drained and the serialized `{ payload_hash = hash_A, signature = sig_A }` is delivered to every caller waiting on Request B. Those callers receive a cryptographically valid attestation that actually attests to a *different* foreign-chain transaction (Request A's tx). If a bridge contract consuming this response does not independently recompute the expected payload hash and compare it to `response.payload_hash`, it will accept the forged attestation and may release funds or advance state based on the wrong transaction. Additionally, Request B's pending slot is permanently consumed — no correct response can ever be delivered for it.

---

### Likelihood Explanation

The attack requires exactly one Byzantine MPC node that acts as the signing leader for at least one request. The leader is the node that calls `respond_verify_foreign_tx` on-chain; it receives the assembled threshold signature before submitting it. Any attested participant can call `respond_verify_foreign_tx` at any time with any valid signature — there is no leader-enforcement check in the contract. The Byzantine node needs only to have been the leader for *any* prior request to obtain a replayable signature. This is a realistic single-node-below-threshold attack.

---

### Recommendation

The contract should validate that `response.payload_hash` is consistent with `request`. Two complementary mitigations:

1. **Node-side**: Include the NEAR receipt ID (or the full `VerifyForeignTransactionRequest` key) as an additional field inside `ForeignTxSignPayload` before hashing, so the threshold signature is cryptographically bound to a specific pending request and cannot be replayed for another.

2. **Contract-side**: Require the MPC node to also supply the `ForeignChainRpcRequest` that was embedded in the payload, and verify on-chain that it matches `request.request`. This does not require knowing the extracted values. [5](#0-4) 

---

### Proof of Concept

```
1. Alice calls verify_foreign_transaction(Bitcoin { tx_id=[0xaa;32] })
   → Request A queued in pending_verify_foreign_tx_requests

2. Bob calls verify_foreign_transaction(Bitcoin { tx_id=[0xbb;32] })
   → Request B queued in pending_verify_foreign_tx_requests

3. MPC network (with Byzantine leader L) signs Request A:
   hash_A = SHA256(borsh(ForeignTxSignPayload::V1 {
       request: Bitcoin { tx_id=[0xaa;32], ... },
       values: [BlockHash([0x11;32])],
   }))
   sig_A  = threshold_sign(hash_A)   ← L now holds sig_A

4. L calls respond_verify_foreign_tx(request=A, response={hash_A, sig_A})
   → Request A resolved correctly (normal flow)

5. L calls respond_verify_foreign_tx(request=B, response={hash_A, sig_A})
   → Contract checks: is sig_A valid over hash_A? YES.
   → Contract looks up Request B: found.
   → Contract resolves Request B with {hash_A, sig_A}.

6. Bob's contract receives VerifyForeignTransactionResponse {
       payload_hash = hash_A,   // attests to tx [0xaa;32], NOT [0xbb;32]
       signature    = sig_A,    // valid threshold signature
   }
   If Bob's contract only verifies the signature (not the payload hash binding),
   it accepts a forged attestation for the wrong Bitcoin transaction.
```

The root cause — `respond_verify_foreign_tx` verifying the signature over `response.payload_hash` without verifying that `response.payload_hash` is derived from `request` — is the direct analog of the external report's missing chain-ID validation in `static_validate_system_transaction`.     #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
