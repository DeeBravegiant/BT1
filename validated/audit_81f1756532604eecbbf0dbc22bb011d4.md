Audit Report

## Title
`LRTOracle._updateRsETHPrice()` Downside Price Protection Disabled by Default — (File: `contracts/LRTOracle.sol`)

## Summary
`pricePercentageLimit` is declared as a plain `uint256` storage variable that defaults to `0` and is never initialized in `initialize()`. The sole downside protection guard in `_updateRsETHPrice()` short-circuits to `false` whenever `pricePercentageLimit == 0`, meaning any computed `newRsETHPrice` — no matter how low — is unconditionally written to `rsETHPrice`. A depositor who calls `depositETH` or `depositAsset` while `rsETHPrice` is near-zero receives a disproportionately large rsETH mint, diluting all existing rsETH holders.

## Finding Description
`pricePercentageLimit` is declared at line 29 with no default value (Solidity zero-initializes it):

```solidity
uint256 public pricePercentageLimit;
```

`initialize()` (lines 64–68) never sets it:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

Inside `_updateRsETHPrice()` (lines 270–282), the only downside guard is:

```solidity
bool isPriceDecreaseOffLimit =
    pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
```

When `pricePercentageLimit == 0`, the left operand of `&&` is `false`, so `isPriceDecreaseOffLimit` is always `false` regardless of the magnitude of the price drop. Execution falls through to line 313:

```solidity
rsETHPrice = newRsETHPrice;
```

The entry point `updateRSETHPrice()` (line 87) is `public` and `whenNotPaused` — callable by any external account:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The minting formula in `LRTDepositPool.getRsETHAmountToMint()` (lines 519–520) divides by `rsETHPrice`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

If `rsETHPrice` is driven to a very small value, this division yields a disproportionately large rsETH mint for any depositor, diluting all existing rsETH holders.

The only setter for `pricePercentageLimit` is `setPricePercentageLimit()` (lines 125–128), which requires `onlyLRTAdmin` — meaning protection is never active until an explicit post-deployment admin transaction.

## Impact Explanation
**Low — Contract fails to deliver promised returns to existing rsETH holders.**

The protocol implements a `pricePercentageLimit` mechanism specifically to protect existing rsETH holders from extreme price drops. Because this mechanism is disabled by default (zero-initialized, never set in `initialize()`), the protocol fails to deliver on this protection promise. Any significant TVL reduction — from EigenLayer slashing, a supported LST oracle price drop, or an accounting anomaly in `_getTotalEthInProtocol()` — results in `rsETHPrice` being written at the reduced value with no floor. Subsequent depositors mint rsETH at the depressed price, diluting existing holders' proportional claims on protocol assets. Existing holders have no recourse once the price is written.

## Likelihood Explanation
`pricePercentageLimit` is `0` in every fresh deployment until an admin explicitly calls `setPricePercentageLimit()`. The TVL drop trigger is realistic: EigenLayer slashing is a documented protocol risk, and LST oracle price feeds can temporarily report low values. `updateRSETHPrice()` is permissionless, so any actor can crystallize a low price into storage immediately after a TVL drop. Likelihood is **Low** for catastrophic slashing but **Medium** for partial slashing or oracle-driven TVL underestimation during the window where `pricePercentageLimit` remains unset.

## Recommendation
1. Set a non-zero default for `pricePercentageLimit` directly in `initialize()` so downside protection is active from deployment without requiring a separate admin transaction.
2. Add an absolute minimum price floor inside `_updateRsETHPrice()` before writing `rsETHPrice`:

```solidity
uint256 constant MIN_RSETH_PRICE = 1e17; // e.g., 0.1 ETH — tune to protocol risk tolerance

if (newRsETHPrice < MIN_RSETH_PRICE) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;
}
```

## Proof of Concept
1. Deploy protocol. `pricePercentageLimit` is `0` (default, never set in `initialize()`).
2. EigenLayer slashing event reduces ETH value held by `NodeDelegator` contracts.
3. `_getTotalEthInProtocol()` returns a significantly reduced value (e.g., drops from 100 ETH to 10 ETH with 95 rsETH supply).
4. Any account calls `updateRSETHPrice()`.
5. `newRsETHPrice = (10e18).divWad(95e18) ≈ 0.105e18`.
6. Guard evaluates: `isPriceDecreaseOffLimit = (0 > 0) && ...` → `false`. No pause, no revert.
7. `rsETHPrice = 0.105e18` written to storage (line 313).
8. Attacker calls `depositETH{value: 1 ether}(0, "")`.
9. `rsethAmountToMint = (1e18 * 1e18) / 0.105e18 ≈ 9.52e18` — attacker receives ~9.52 rsETH for 1 ETH.
10. Attacker's rsETH now represents a claim on ~9.09% of the 11 ETH TVL ≈ 1 ETH — fair at current price, but if TVL recovers (e.g., EigenLayer compensates slashing), attacker profits at existing holders' expense with no recourse for those holders.

**Foundry test sketch:**
```solidity
function test_priceDropNoFloor() public {
    // Setup: rsETH supply = 95e18, TVL = 100e18, price = ~1.05e18
    // Simulate slashing: mock _getTotalEthInProtocol to return 10e18
    // Assert pricePercentageLimit == 0
    assertEq(lrtOracle.pricePercentageLimit(), 0);
    // Call updateRSETHPrice()
    lrtOracle.updateRSETHPrice();
    // Assert rsETHPrice dropped without pause
    assertLt(lrtOracle.rsETHPrice(), 0.2e18);
    assertFalse(lrtOracle.paused());
    // Deposit 1 ETH, assert rsETH minted >> 1e18
    uint256 minted = lrtDepositPool.getRsETHAmountToMint(ETH_TOKEN, 1e18);
    assertGt(minted, 5e18);
}
