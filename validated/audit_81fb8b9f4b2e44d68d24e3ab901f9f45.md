Audit Report

## Title
SwapAllowlistExtension Checks Router Address Instead of Original Swapper, Enabling Allowlist Bypass — (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

## Summary
`SwapAllowlistExtension.beforeSwap()` gates swaps by checking `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the pool's `msg.sender` — the router contract — when a user routes through `MetricOmmSimpleRouter`. If the pool admin adds the router to the allowlist (the only way to let allowlisted users swap via the router), every unprivileged user can bypass the allowlist by routing through the same public router, defeating the access control entirely.

## Finding Description

**Call path:**

1. User calls `MetricOmmSimpleRouter.exactInputSingle()` (or any `exact*` variant).
2. Router calls `IMetricOmmPoolActions(pool).swap(recipient, zeroForOne, amount, limit, "", extensionData)` — so `pool.msg.sender = router`.
3. `MetricOmmPool.swap()` calls `_beforeSwap(msg.sender, ...)` → `sender = router address`.
4. `ExtensionCalling._beforeSwap` encodes and calls `SwapAllowlistExtension.beforeSwap(sender=router, ...)`.
5. Inside `beforeSwap`:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` = pool, `sender` = router. The check resolves to `allowedSwapper[pool][router]`, not `allowedSwapper[pool][originalUser]`.

**Root cause:** `MetricOmmPool.swap()` passes `msg.sender` (the immediate caller) as `sender` to `_beforeSwap`. When the router intermediates, the original user's address is never forwarded to the extension.

**Exploit flow:**
- Pool admin deploys a restricted pool with `SwapAllowlistExtension` and allowlists only specific addresses (e.g., `userA`).
- `userA` attempts to swap via the router → pool sees `sender = router` → `allowedSwapper[pool][router] = false` → reverts. The router is unusable for allowlisted users.
- Pool admin adds the router to the allowlist (`allowedSwapper[pool][router] = true`) to restore router functionality for `userA`.
- Now `userB` (not allowlisted) calls `MetricOmmSimpleRouter.exactInputSingle()` → pool sees `sender = router` → `allowedSwapper[pool][router] = true` → passes. `userB` bypasses the allowlist.

**Existing guards reviewed:**
- `BaseMetricExtension.onlyPool` only verifies the caller is a registered pool — it does not validate the `sender` argument.
- There is no mechanism in the pool or router to forward the original EOA/contract address to the extension.
- `MetricOmmPool.swap()` unconditionally passes `msg.sender` as `sender` with no originator context.

## Impact Explanation
The swap allowlist — the sole access control mechanism for restricting who may trade in a pool — is rendered ineffective for any pool whose admin adds the router to the allowlist. Unauthorized users gain unrestricted swap access to pools explicitly configured to exclude them. This is an admin-boundary break: an unprivileged actor bypasses a pool-level access control gate that the pool admin intended to enforce.

## Likelihood Explanation
The precondition (router on the allowlist) is a natural and expected admin action: without it, no allowlisted user can use the router at all, making the router incompatible with allowlisted pools. Any pool admin who deploys a restricted pool and also wants router support will trigger this condition. The bypass is then trivially repeatable by any unprivileged user with no special capability beyond calling the public router.

## Recommendation
The extension must check the original user's identity, not the intermediary's. Two viable approaches:

1. **Pass originator through extension data:** Have the router encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it (requires router cooperation and extension-data convention).
2. **Check `recipient` instead of `sender`:** If the pool's design guarantees `recipient` is the economic beneficiary, gate on `recipient`. However, this may not hold for multi-hop paths.
3. **Preferred — dedicated originator field:** Add an `originator` parameter to the swap interface or extension callback so the pool can pass both the immediate caller and the original initiator, and the extension gates on the originator.

// 1. Deploy pool with SwapAllowlistExtension
// 2. Allowlist userA: allowedSwapper[pool][userA] = true
// 3. Allowlist router so userA can use it: allowedSwapper[pool][router] = true
// 4. userB (not allowlisted) calls:
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: userB,
    zeroForOne: true,
    amountIn: 1000,
    amountOutMinimum: 0,
    priceLimitX64: 0,
    deadline: block.timestamp,
    extensionData: ""
}));
// Result: swap succeeds — userB bypassed the allowlist
// Assert: pool received input tokens, userB received output tokens
