### Title
No On-Chain Enforcement of CKD Response Integrity for Legacy `AppPublicKey` Path — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` enforces on-chain correctness of the CKD output only for the `AppPublicKeyPV` variant. For the legacy `AppPublicKey` variant, the contract performs **zero validation** of the returned `CKDResponse`. A single compromised attested participant acting as the signing leader can submit an arbitrary, attacker-chosen encrypted key, which the contract accepts unconditionally and delivers to the requesting user.

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no validation whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the encrypted output is the correct MPC-derived secret for the given `app_id` and root public key. For `AppPublicKey` (the legacy, still-supported path), the arm is an empty block — the response bytes are passed directly to `resolve_yields_for` with no integrity check. [2](#0-1) 

The only gate before this branch is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be a single attested participant — not a threshold of them. [3](#0-2) 

The `AppPublicKey` path is explicitly documented as still supported (described as "legacy"): [4](#0-3) 

---

### Impact Explanation

**Critical — Theft of funds controlled by the user's MPC-derived key.**

The attacker (a single compromised attested participant acting as leader) proceeds as follows:

1. Observes a pending `request_app_private_key` call using the `AppPublicKey` variant.
2. Generates an arbitrary key `K_malicious` of their choosing.
3. Encrypts `K_malicious` under the user's `app_public_key` (the user's ephemeral BLS public key, which is public on-chain in the request).
4. Calls `respond_ckd` with this fabricated `CKDResponse`. The contract accepts it — the empty match arm fires, no check runs.
5. The user receives and decrypts the response, obtaining `K_malicious`.
6. The user deploys `K_malicious` as their app-specific secret to control assets on a foreign chain.
7. The attacker, who chose `K_malicious`, also knows the plaintext and can sweep those assets.

The user has no on-chain signal that the response was tampered with; the contract's promise resolves successfully.

---

### Likelihood Explanation

**Medium.** The attack requires exactly one compromised attested participant to win the leader role for a target CKD request. TEE attestation raises the bar, but:

- Any single attested participant can become the leader for any given request.
- The `AppPublicKey` path is still accepted by the contract (not deprecated at the protocol level).
- The attacker does not need to break the threshold — they only need to be the one node that calls `respond_ckd`.

---

### Recommendation

Apply the same `ckd_output_check` to the `AppPublicKey` arm, or reject `AppPublicKey` requests at the contract level and require all new requests to use `AppPublicKeyPV`. The publicly-verifiable variant already demonstrates that an efficient on-chain check is feasible.

---

### Proof of Concept

```
1. Deploy contract in Running state with a BLS CKD domain.
2. User calls request_app_private_key with:
     app_public_key: AppPublicKey(<user_bls_g1_pk>)
3. Attacker (single attested participant) constructs:
     K_malicious = arbitrary 32-byte secret
     fake_response = BLS-encrypt(K_malicious, user_bls_g1_pk)
4. Attacker calls respond_ckd(request, fake_response).
   → assert_caller_is_attested_participant_and_protocol_active passes (attacker is attested).
   → AppPublicKey arm: empty block, no check.
   → resolve_yields_for delivers fake_response to the user's promise.
5. User decrypts fake_response → obtains K_malicious.
6. User uses K_malicious to control a foreign-chain wallet.
7. Attacker, knowing K_malicious, drains that wallet.
```

The root cause is the empty match arm at `crates/contract/src/lib.rs` line 676, which provides no on-chain enforcement of CKD output correctness for the legacy `AppPublicKey` path — directly analogous to the reported pattern of off-chain value computation with no on-chain protection for certain interaction types.
