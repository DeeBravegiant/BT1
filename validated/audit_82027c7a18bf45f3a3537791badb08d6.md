### Title
Nonce-Free `verify_foreign_transaction` Response Signature Enables Cross-Chain Replay — (`File: crates/contract/src/lib.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary

The `verify_foreign_transaction()` flow produces a threshold ECDSA signature over `ForeignTxSignPayload` that contains no nonce, no caller binding, and no timestamp. Because the pending-request map key is also caller-agnostic, any party who obtains the `VerifyForeignTransactionResponse` can replay it indefinitely against any downstream NEAR bridge contract that relies on the MPC attestation to authorize cross-chain transfers, enabling double-spend conditions.

### Finding Description

`verify_foreign_transaction()` converts caller-supplied `VerifyForeignTransactionRequestArgs` into a `VerifyForeignTransactionRequest` via `args_into_verify_foreign_tx_request()`. The resulting key stored in `pending_verify_foreign_tx_requests` does **not** include the caller's account ID. [1](#0-0) 

The unit test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` explicitly confirms that Alice and Bob submitting identical `VerifyForeignTransactionRequestArgs` map to the **same** request key and both receive the same `VerifyForeignTransactionResponse`: [2](#0-1) 

On the node side, `build_signature_request()` constructs the inner `SignatureRequest` with a **zero tweak** (`Tweak::new([0u8; 32])`), meaning the signature is produced under the **root MPC key** with no key derivation: [3](#0-2) 

`respond_verify_foreign_tx()` verifies the signature against the root public key (no tweak applied): [4](#0-3) 

The signed payload is `SHA-256(borsh(ForeignTxSignPayload{request, values}))`. It contains:
- The `ForeignChainRpcRequest` (tx_id, extractors)
- The extracted values (e.g., block hash)

It does **not** contain:
- Any per-call nonce
- The NEAR caller's account ID
- Any block height or timestamp

The `ForeignTxSignPayload` structure is defined in the contract interface: [5](#0-4) 

The `SignatureRequest` on the contract side (`crates/near-mpc-crypto-types`) also carries no nonce: [6](#0-5) 

### Impact Explanation

The `verify_foreign_transaction()` feature is explicitly designed for bridge inbound flows ("Omnibridge inbound flow (foreign chain → NEAR)"): [7](#0-6) 

A downstream NEAR bridge contract that calls `verify_foreign_transaction()` and uses the returned `VerifyForeignTransactionResponse` to authorize a mint or unlock operation has no way to distinguish a fresh attestation from a replayed one, because:

1. The signature is over a payload that is fully determined by the foreign tx ID and extracted values — identical for every submission of the same foreign tx.
2. The signature is produced under the root key (zero tweak), so it is verifiable by anyone with the MPC public key.
3. The response is delivered on-chain in a NEAR receipt, making it publicly observable and extractable.

An attacker who observes a completed `verify_foreign_transaction()` receipt can extract the `VerifyForeignTransactionResponse` and replay it against any bridge contract that does not maintain its own used-signature registry, triggering duplicate mints or unlocks for the same foreign-chain deposit — a direct double-spend.

**Impact: High** — Cross-chain replay causing invalid bridge execution or double-spend conditions.

### Likelihood Explanation

- The feature is purpose-built for bridge use cases; bridge contracts are the primary and intended consumers.
- The response is publicly visible in NEAR transaction receipts — no privileged access is needed to extract it.
- The attacker only needs to observe one successful `verify_foreign_transaction()` call for a given foreign tx and then replay the response.
- No threshold collusion, key material, or operator access is required.

### Recommendation

1. **Include a per-call nonce or NEAR receipt ID in the signed payload.** The node-side `SignatureRequest` already carries a `receipt_id` field; including it in `ForeignTxSignPayload` would bind each attestation to a unique NEAR call. [8](#0-7) 

2. **Bind the signature to the caller's account ID.** Include `predecessor_account_id` in `ForeignTxSignPayload` so the attestation is only valid for the account that requested it.

3. **Add an expiry timestamp** to `ForeignTxSignPayload` so replayed signatures become invalid after a short window.

4. **Document replay-protection requirements** for downstream bridge contracts that consume `VerifyForeignTransactionResponse`, making clear they must maintain a used-signature registry keyed on `payload_hash`.

### Proof of Concept

```
1. Alice calls verify_foreign_transaction(Bitcoin tx T) on the MPC contract.
   → MPC nodes verify T on Bitcoin, sign ForeignTxSignPayload{T, block_hash=B}.
   → Alice receives VerifyForeignTransactionResponse{payload_hash=H, signature=S}.
   → S = ECDSA_sign(root_key, H) where H = SHA256(borsh({T, B})).

2. Alice submits S to BridgeContract.mint(proof=S) → receives 1 BTC equivalent.

3. Attacker observes S in Alice's NEAR receipt (public).

4. Attacker calls BridgeContract.mint(proof=S) again with the same S.
   → BridgeContract verifies S against MPC root public key → valid.
   → Attacker receives another 1 BTC equivalent for free (double-spend).

Root cause: S is not bound to Alice's account, any nonce, or any expiry.
No re-interaction with the MPC contract is needed for step 4.
```

The fan-out design (caller-agnostic key) further amplifies this: any of the up to `MAX_PENDING_REQUEST_FAN_OUT` = 128 callers who queued the same request all receive the identical `S`, each independently capable of replaying it. [9](#0-8)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

        let mut rng = rand::rngs::StdRng::from_seed([42u8; 32]);
        let (context, mut contract, secret_key) =
            basic_setup_with_protocol(Protocol::CaitSith, DomainPurpose::ForeignTx, &mut rng);

        let SharedSecretKey::Secp256k1(secret_key) = secret_key else {
            unreachable!();
        };

        let request_args = VerifyForeignTransactionRequestArgs {
            domain_id: DomainId::default().0.into(),
            payload_version: ForeignTxPayloadVersion::V1,
            request: dtos::ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
                tx_id: [7u8; 32].into(),
                confirmations: 2.into(),
                extractors: vec![BitcoinExtractor::BlockHash],
            }),
        };
        let request = args_into_verify_foreign_tx_request(request_args.clone());

        // When: caller alice submits the request.
        let alice = AccountId::from_str("alice.near").unwrap();
        testing_env!(
            VMContextBuilder::new()
                .signer_account_id(alice.clone())
                .predecessor_account_id(alice)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()
        );
        contract.verify_foreign_transaction(request_args.clone());

        // And: caller bob submits the identical request — a different account would today
        // be blocked from receiving a response by alice's submission.
        let bob = AccountId::from_str("bob.near").unwrap();
        testing_env!(
            VMContextBuilder::new()
                .signer_account_id(bob.clone())
                .predecessor_account_id(bob)
                .current_account_id(context.current_account_id.clone())
                .attached_deposit(NearToken::from_yoctonear(1))
                .build()

        assert_eq!(

                .pending_verify_foreign_tx_requests
                .get(&request)
                .map(|q| q.len()),
            Some(2),
            "duplicate foreign-tx requests from different callers should fan out",

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
