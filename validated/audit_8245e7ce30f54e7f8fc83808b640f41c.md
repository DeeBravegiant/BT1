### Title
Unsigned Absorb Spend While in `LEAVING_POOL` State Resets `ASSERT_HEIGHT_RELATIVE` Lock Countdown, Allowing Pool to Trap Farmers Indefinitely - (`chia/pools/pool_wallet.py`, `chia/pools/pool_puzzles.py`)

---

### Summary

The Chia pool protocol enforces a `relative_lock_height` block delay before a farmer can complete their exit from a pool. This countdown is measured from the **most recent singleton coin creation height** (i.e., the last spend of the singleton). Because absorb spends (claiming `p2_singleton` farming rewards) require **no owner signature** and create a new singleton coin at the current block height, a malicious pool or any third party can continuously submit absorb spends while the farmer is in `LEAVING_POOL` state, resetting the on-chain `ASSERT_HEIGHT_RELATIVE` countdown with each absorb. This traps the farmer indefinitely as long as they continue farming.

---

### Finding Description

**Root cause — permissionless absorb spend:**

`create_absorb_spend` in `pool_puzzles.py` constructs a spend bundle that is submitted with an empty BLS signature (`G2Element()`). The wallet code explicitly documents this:

```python
# If fee is 0, no signatures are required to absorb
``` [1](#0-0) 

The absorb spend bundle consists of two deterministic coin spends built entirely from public on-chain data (launcher ID, current state, reward height) — no private key is needed: [2](#0-1) 

The lifecycle test confirms absorb spends succeed with `G2Element()` (empty signature) while the singleton is in `LEAVING_POOL` (waiting room) state: [3](#0-2) 

**Root cause — lock countdown measured from last singleton spend:**

The wallet's `new_peak` method computes the leave height as:

```python
leave_height = tip_height + pool_wallet_info.current.relative_lock_height
```

where `tip_height` is the block height of the **most recent singleton spend** (from `get_tip()`): [4](#0-3) 

`get_tip()` returns the last entry in the pool spend history: [5](#0-4) 

The same `history[-1][0]` reference is used in both `self_pool` and `join_pool` to gate the second transition: [6](#0-5) 

**On-chain enforcement also resets:** The `pool_waiting_room_innerpuz` CLSP uses `ASSERT_HEIGHT_RELATIVE` relative to the **current singleton coin's creation height**. Each absorb spend creates a new singleton coin at the absorb block height, so the on-chain `ASSERT_HEIGHT_RELATIVE` countdown resets with every absorb. The test confirms this condition is what blocks premature exit: [7](#0-6) 

**Absorb spend while in waiting room is valid by design:**

The waiting room inner puzzle (type-0 solution path) explicitly supports absorb spends while in `LEAVING_POOL` state: [8](#0-7) 

---

### Impact Explanation

**High — Permanent or long-lived inability for honest farmers to complete valid pool exit actions.**

A malicious pool (or any third party who knows the public `p2_singleton_puzzle_hash`) can:

1. Observe that a farmer has entered `LEAVING_POOL` state.
2. Wait for the farmer's plots to produce a farming reward at `p2_singleton_puzzle_hash` (a public address derivable from the launcher ID on-chain).
3. Construct and submit an absorb spend bundle with zero fee and empty signature — no key material needed.
4. The absorb advances the singleton tip to the current block, resetting `leave_height = current_block + relative_lock_height`.
5. Repeat for every subsequent farming reward.

The farmer can never satisfy `peak_height > leave_height + 2` as long as absorb spends keep advancing the tip. The farmer's singleton-controlled assets remain under pool control indefinitely. The pool has direct economic incentive to execute this attack (retaining the farmer's hashrate).

The `p2_singleton_puzzle_hash` is fully public: [9](#0-8) 

---

### Likelihood Explanation

**Medium-High.** The attack requires legitimate coinbase reward coins to exist at the farmer's `p2_singleton_puzzle_hash` — the farmer must be actively farming. However:

- Any farmer trying to leave a pool while continuing to farm (the normal case) is vulnerable.
- The pool itself is the most motivated attacker and has full visibility into the farmer's singleton state.
- The attack costs the attacker nothing (zero fee, no signature, no tokens spent).
- The farmer has no on-chain mechanism to prevent absorb spends.

The only escape is for the farmer to stop farming entirely (removing plots from the pool), forfeiting all farming rewards during the exit period — a significant economic harm.

---

### Recommendation

1. **Track `LEAVING_POOL` entry height separately.** Store the block height at which the singleton first transitioned into `LEAVING_POOL` state and use that fixed height (not `tip_height`) for the lock calculation. The on-chain puzzle would need to embed this entry height in the waiting room state.

2. **Alternatively, disallow absorb spends while in `LEAVING_POOL` state.** The waiting room inner puzzle could reject the type-0 (absorb) solution path once in escaping state, ensuring the `ASSERT_HEIGHT_RELATIVE` countdown is never reset. Farming rewards accumulated during the exit period would be claimable after the exit completes.

3. **Require owner signature for absorb spends.** Adding an `AGG_SIG_ME` condition to the absorb path of the waiting room puzzle would prevent third parties from triggering absorb spends, though this would also prevent the pool from absorbing on the farmer's behalf during exit.

---

### Proof of Concept

```
1. Farmer joins pool with relative_lock_height = 32.
2. Farmer calls pw_self_pool() at block H → singleton enters LEAVING_POOL.
   leave_height = H + 32 = H+32.

3. Farmer's plots farm a block at H+1 → reward coin appears at p2_singleton_puzzle_hash.

4. Attacker (pool or anyone) constructs absorb spend bundle:
     coin_spends = create_absorb_spend(
         last_coin_spend=tip_spend,
         current_state=LEAVING_POOL_state,
         launcher_coin=launcher_coin,
         height=H+1,
         genesis_challenge=...,
         delay_time=...,
         delay_ph=...,
     )
     spend_bundle = SpendBundle(coin_spends, G2Element())  # no signature
   Submits to mempool → accepted.

5. New singleton tip is at block H+1.
   leave_height = H+1 + 32 = H+33.

6. Farmer's plots farm a block at H+2 → attacker absorbs again.
   leave_height = H+2 + 32 = H+34.

7. Repeat indefinitely. Farmer can never satisfy peak_height > leave_height + 2.
```

The absorb spend path for the waiting room is exercised without any signature in the existing test suite, confirming the permissionless nature of the attack: [10](#0-9)
