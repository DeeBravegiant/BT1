Audit Report

## Title
ETH Deposit Limit Bypass via Missing `+ amount` in `_checkIfDepositAmountExceedesCurrentLimit` - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` evaluates the ETH deposit limit using only the pre-deposit total (`totalAssetDeposits > depositLimit`), omitting the incoming `amount`. Any unprivileged depositor can call `depositETH` with an arbitrarily large `msg.value` while `totalAssetDeposits < depositLimit`, bypassing the cap entirely and receiving rsETH minted against ETH that cannot be deployed into EigenLayer strategies.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // ← amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [1](#0-0) 

The ETH branch never incorporates `amount` into the comparison. The function returns `false` (limit not exceeded) for any `amount` whenever `totalAssetDeposits < depositLimit`, regardless of how large `amount` is. The non-ETH branch correctly adds `amount` before comparing, making the asymmetry a clear implementation defect.

The call chain is: `depositETH` (L76–93) → `_beforeDeposit` (L648–670) → `_checkIfDepositAmountExceedesCurrentLimit`. The revert at `_beforeDeposit` L662 (`MaximumDepositLimitReached`) is only triggered when the check returns `true`, which the ETH branch never does while `totalAssetDeposits < depositLimit`. [2](#0-1) 

`depositETH` is `external payable` with no role restriction beyond `whenNotPaused` and `onlySupportedAsset`. [3](#0-2) 

## Impact Explanation
**Low – Contract fails to deliver promised returns.** The deposit limit is a protocol-enforced cap on ETH exposure to EigenLayer strategies. Bypassing it allows rsETH to be minted against ETH that exceeds strategy capacity and sits idle in the pool, earning no yield. rsETH holders who deposited under the assumption that all backing ETH is deployed into yield-bearing strategies receive a token partially backed by non-productive ETH. This directly matches the "contract fails to deliver promised returns" impact tier. Escalation to **Medium – Temporary freezing of funds** is possible if the excess ETH cannot be routed to any NodeDelegator strategy and becomes stranded, but that secondary condition is not guaranteed by the bug alone.

## Likelihood Explanation
The precondition is only that a non-zero `depositLimitByAsset` has been configured for ETH, which is the expected production state whenever a cap is desired. No privileged access, oracle manipulation, or front-running is required. Any depositor who reads on-chain state can trigger this in a single transaction, repeatably, until `totalAssetDeposits` itself exceeds the limit (which only happens after the excess deposits have already been accepted).

## Recommendation
Replace the ETH branch with the same `+ amount` pattern used for ERC-20 tokens:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This makes both paths consistent and ensures the incoming deposit is always included in the limit comparison before rsETH is minted.

## Proof of Concept
Assume `depositLimitByAsset[ETH_TOKEN] = 100 ether`, `getTotalAssetDeposits(ETH_TOKEN) = 50 ether`.

1. Attacker calls `depositETH{value: 1000 ether}("")`.
2. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1000 ether)`.
3. ETH branch evaluates `50 ether > 100 ether` → `false` → limit not exceeded, no revert.
4. `getRsETHAmountToMint` computes rsETH for 1000 ETH; `_mintRsETH` mints it to the attacker.
5. Post-deposit: `getTotalAssetDeposits(ETH_TOKEN) = 1050 ether`, 950 ETH above the cap.
6. Equivalent non-ETH call would evaluate `50 + 1000 > 100` → `true` → `MaximumDepositLimitReached` revert, confirming the asymmetry.

**Foundry test plan:** Deploy with `depositLimitByAsset[ETH_TOKEN] = 100 ether`, seed pool with 50 ETH, call `depositETH{value: 1000 ether}` from an unprivileged address, assert the call succeeds and `getTotalAssetDeposits(ETH_TOKEN) > 100 ether`. Then apply the fix and assert the same call reverts with `MaximumDepositLimitReached`.
