### Title
Participant-Set Injection in CKD Follower Allows Byzantine Leader to Produce Unauthorized Confidential Key Derivation Output — (`crates/node/src/providers/ckd/sign.rs`)

---

### Summary

A Byzantine leader node can inject arbitrary extra `ParticipantId` entries into the `MpcStartMessage.participants` field it sends to followers. Because `make_ckd_follower` and `CKDComputation::compute` blindly trust `channel.participants()` for Lagrange coefficient computation — with no check against `mpc_config.participants` — every honest follower normalizes its share using the wrong (inflated) participant set. The aggregated `(big_y, big_c)` output does not correspond to the correct threshold-BLS evaluation. For the non-PV (`AppPublicKey`) contract path, `respond_ckd` performs no on-chain cryptographic verification and accepts the malformed output unconditionally.

---

### Finding Description

**Step 1 — Channel creation (leader side)**

In `make_ckd_leader`, the honest leader selects participants from `mpc_config.participants`: [1](#0-0) 

A Byzantine leader bypasses this and calls `client.new_channel_for_task(CKDTaskId::Ckd { id }, participants_with_extra)` with an injected extra `ParticipantId`. The `new_channel_for_task` function places the caller-supplied list verbatim into `MpcStartMessage.participants` and broadcasts it: [2](#0-1) 

**Step 2 — Channel construction on the follower side**

When a follower receives the `Start` message, `sender_for` constructs the `NetworkTaskChannel` with `participants: start.participants.clone()` — directly from the attacker-controlled message, with no validation: [3](#0-2) 

**Step 3 — Follower accepts channel without participant-set validation**

`make_ckd_follower` does not check that `channel.participants()` is a subset of `self.mpc_config.participants`: [4](#0-3) 

**Step 4 — Lagrange coefficients computed over attacker-controlled set**

`CKDComputation::compute` builds `cs_participants` directly from `channel.participants()` and passes it to `ckd()`: [5](#0-4) 

Inside `compute_signature_share`, `lambda_i` is computed over this inflated list: [6](#0-5) 

With an extra participant injected (e.g., threshold=2, honest set={A,B}, injected set={A,B,C}), every node computes `λ_i({A,B,C})` instead of `λ_i({A,B})`. The Lagrange coefficients are different, so the aggregated `(big_y, big_c)` does not equal `msk · H(app_id)`.

**Step 5 — Non-PV contract path accepts without verification**

`respond_ckd` performs no cryptographic check for the `AppPublicKey` variant: [7](#0-6) 

The malformed output is stored and returned to the caller. (The PV path is protected by an on-chain pairing check at line 678, but the non-PV path has no equivalent guard.)

---

### Impact Explanation

The resulting `(big_y, big_c)` is a deterministic function of the attacker-chosen participant set. The user decrypts it with their private key `a` and obtains a value that is **not** `msk · H(app_id)` — the correct threshold-BLS evaluation. The confidential key derivation output is unauthorized: it was produced without the required threshold-authorized participant set. This satisfies the Critical impact category: *unauthorized confidential key derivation output without the required participant authorization*.

---

### Likelihood Explanation

- Requires exactly one Byzantine participant node (below threshold).
- Any participant node can act as leader for a CKD request (the leader role is self-selected by whichever node picks up the pending request first).
- The injected participant only needs to be a real node with a keyshare for the domain (e.g., another legitimate participant not originally selected for this round) so that it responds and the protocol completes rather than timing out.
- The non-PV `AppPublicKey` variant is the default/legacy path used in production (as shown in the README and e2e tests).

---

### Recommendation

In `make_ckd_follower`, validate that every participant ID in `channel.participants()` is present in `self.mpc_config.participants.participants` before constructing `CKDComputation`. Reject the channel with an error if any unknown participant is found:

```rust
// In make_ckd_follower, after receiving the channel:
let authorized_ids: HashSet<_> = self.mpc_config.participants.participants
    .iter().map(|p| p.id).collect();
for pid in channel.participants() {
    anyhow::ensure!(
        authorized_ids.contains(pid),
        "CKD channel contains unauthorized participant: {:?}", pid
    );
}
```

Additionally, add an on-chain cryptographic verification for the non-PV `AppPublicKey` path in `respond_ckd`, analogous to the existing `ckd_output_check` used for the PV path.

---

### Proof of Concept

1. Deploy a 3-node cluster (threshold=2, participants={A,B,C}), BLS12-381 CKD domain, non-PV `AppPublicKey` request.
2. Byzantine node A acts as leader. Instead of selecting participants `[A, B]`, it calls `new_channel_for_task` with `[A, B, C]`.
3. Nodes B and C each receive the `Start` message, look up the keyshare, and compute `λ_i([A,B,C])` — the Lagrange coefficient over 3 participants.
4. A aggregates the three shares and calls `respond_ckd` with the result.
5. The contract accepts the response (no verification for non-PV).
6. Assert: the returned `(big_y, big_c)` differs from an honest run with participants `[A, B]`, and `CKDOutput::unmask(app_sk)` does not satisfy `verify_signature(&mpc_vk, app_id, &key)`.
