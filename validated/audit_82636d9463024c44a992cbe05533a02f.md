### Title
MPC Docker Image Hash Cannot Be Revoked Once Whitelisted — (`crates/contract/src/lib.rs`, `crates/contract/src/tee/tee_state.rs`, `crates/contract/src/tee/proposal.rs`)

---

### Summary

The `vote_code_hash` function allows threshold participants to add an MPC Docker image hash to the TEE whitelist, but there is no corresponding removal function. Unlike launcher image hashes and OS measurements — which each have both an `add` and a `remove` governance path — whitelisted MPC Docker image hashes can only be removed by time-based expiry. If a whitelisted image hash is later found to be compromised, there is no on-chain mechanism to immediately revoke it, leaving nodes running that image as valid TEE participants for the entire grace period.

---

### Finding Description

The contract exposes three categories of TEE whitelist entries, each managed by participant voting:

**1. MPC Docker image hashes** — add only: [1](#0-0) 

When threshold votes are reached, `whitelist_tee_proposal` is called, which delegates to `AllowedDockerImageHashes::insert`: [2](#0-1) 

There is **no** `vote_remove_code_hash` function anywhere in the contract. The only removal path is time-based expiry via `cleanup_expired_hashes`, which only fires lazily on the next `insert` call and always preserves at least one entry: [3](#0-2) 

**2. Launcher image hashes** — add AND remove both exist: [4](#0-3) 

**3. OS measurements** — add AND remove both exist: [5](#0-4) 

The asymmetry is clear: launcher hashes and OS measurements have symmetric governance (add + remove), but MPC Docker image hashes have only an add path. The `AllowedDockerImageHashes` struct has no `remove` method at all: [6](#0-5) 

The whitelisted image hashes are used as the gate in `add_participant` to validate TEE attestations: [7](#0-6) 

Any node whose attestation references a whitelisted image hash is accepted as a valid TEE participant and is authorized to call `respond`, `respond_ckd`, `vote_pk`, `vote_reshared`, and other privileged node-facing methods.

---

### Impact Explanation

If a whitelisted MPC Docker image hash is later found to contain a critical vulnerability (e.g., a key-extraction bug, a signing bypass, or a memory-disclosure flaw), the network has no on-chain mechanism to immediately revoke it. Nodes running the compromised image remain valid TEE participants for the entire `tee_upgrade_deadline_duration` (a configurable value, potentially days or weeks). During this window:

- Compromised nodes can call `respond()` and submit signatures, enabling **unauthorized transaction execution**.
- Compromised nodes participate in `vote_pk` / `vote_reshared`, potentially influencing key generation or resharing outcomes.
- Compromised nodes' attestations pass `reverify_and_cleanup_participants`, so `verify_tee` does not kick them out.

This matches the allowed impact: **unauthorized transaction execution / threshold signature issuance without required participant authorization**, and **participant/attestation authorization bypass that causes invalid bridge execution**.

---

### Likelihood Explanation

Docker image vulnerabilities are routinely discovered post-deployment. The scenario — a hash legitimately voted in by threshold participants, then found to be compromised — requires no collusion and no privileged access beyond what is already granted by the governance process. The missing revocation path is a structural gap, not a theoretical edge case. The `tee_upgrade_deadline_duration` grace period (configurable, but intended to be days-to-weeks) defines the minimum exposure window with no remediation path.

---

### Recommendation

Add a `vote_remove_code_hash` function mirroring the existing `vote_remove_launcher_hash` pattern:

```rust
// In lib.rs
pub fn vote_remove_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
    self.voter_or_panic();
    let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();
    let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
    // Require ALL participants to vote for removal (same as launcher/measurement removal)
    let votes = self.tee_state.vote_remove_code_hash(code_hash, &participant);
    let total_participants = threshold_parameters.participants().len() as u64;
    if votes >= total_participants {
        self.tee_state.remove_docker_image_hash(&code_hash);
    }
    Ok(())
}
```

Add a `remove` method to `AllowedDockerImageHashes` (analogous to `AllowedLauncherImages::remove`): [8](#0-7) 

Requiring unanimity for removal (as done for launcher hashes and OS measurements) prevents a minority from disrupting the network while still providing an emergency revocation path.

---

### Proof of Concept

1. Threshold participants call `vote_code_hash(hash_A)` → `hash_A` is whitelisted.
2. A critical vulnerability is discovered in the Docker image identified by `hash_A`.
3. Participants attempt to revoke `hash_A` — **no such function exists**.
4. Nodes running the vulnerable image continue to pass `add_participant` attestation checks and `reverify_and_cleanup_participants`.
5. The compromised nodes remain authorized to call `respond()` and submit signatures for the entire `tee_upgrade_deadline_duration`, with no on-chain remediation available.
6. The only mitigation is to vote in a new `hash_B` and wait for `hash_A` to expire — but `valid_entries` always preserves at least one entry (the latest), so if `hash_A` is the only entry, it **never expires**: [9](#0-8)

pub(crate) struct AllowedDockerImageHashes {
    /// Whitelisted code hashes, sorted by when they were added (oldest first). Expired entries are
    /// lazily cleaned up during insertions and TEE validation.
    allowed_tee_proposals: Vec<AllowedMpcDockerImage>,
}

impl AllowedDockerImageHashes {
    /// Checks if a Docker image hash is still valid (not expired).
    fn valid_entries(&self, tee_upgrade_deadline_duration: Duration) -> Vec<AllowedMpcDockerImage> {
        let current_time = Timestamp::now();
        // get the index of the most recently enforced docker image
        let cutoff_index = self
            .allowed_tee_proposals
            .iter()
            .rposition(|allowed_docker_image| {
                let Some(grace_period_deadline) = allowed_docker_image
                    .added
                    .checked_add(tee_upgrade_deadline_duration)
                else {
                    log!("Error: timestamp overflowed when calculating grace_period_deadline.");
                    return true;
                };
                // if the grace period for this docker hash is in the past, then older hashes are no longer accepted
                grace_period_deadline < current_time
            })
            .unwrap_or(0);

        self.allowed_tee_proposals
            .get(cutoff_index..)
            .unwrap_or(&[])
            .to_vec()
    }

    /// Removes all expired code hashes and returns the number of removed entries.
    /// Ensures that at least one (the latest) proposal always remains in the whitelist.
    pub fn cleanup_expired_hashes(&mut self, tee_upgrade_deadline_duration: Duration) {
        let valid_entries = self.valid_entries(tee_upgrade_deadline_duration);
        self.allowed_tee_proposals = valid_entries;
    }

    /// Inserts a new code hash into the list after cleaning expired entries. Maintains the sorted
    /// order by `added` (ascending).
    pub fn insert(&mut self, code_hash: NodeImageHash, tee_upgrade_deadline_duration: Duration) {
        self.cleanup_expired_hashes(tee_upgrade_deadline_duration);

        // Remove the old entry if it exists
        if let Some(pos) = self
            .allowed_tee_proposals
            .iter()
            .position(|entry| entry.image_hash == code_hash)
        {
            self.allowed_tee_proposals.remove(pos);
        }

        let new_entry = AllowedMpcDockerImage {
            image_hash: code_hash,
            added: Timestamp::now(),
        };

        // Find the correct position to maintain sorted order by `added`
        let insert_index = self
            .allowed_tee_proposals
            .iter()
            // strictly less, `<`, such that new entries take higher precedence
            // if two entries have the exact same time stamp.
            .rposition(|entry| new_entry.added < entry.added)
            .unwrap_or(self.allowed_tee_proposals.len());

        self.allowed_tee_proposals.insert(insert_index, new_entry);
    }

    /// Returns valid hashes without cleaning expired entries (read-only). Ensures that at least
    /// one proposal (the latest) is always returned. Use [`Self::cleanup_expired_hashes`]
    /// explicitly when cleanup of the internal structure is needed.
    pub fn get(&self, tee_upgrade_deadline_duration: Duration) -> Vec<AllowedMpcDockerImage> {
        self.valid_entries(tee_upgrade_deadline_duration)
    }

    /// Returns only the image hashes of valid entries.
    pub fn get_image_hashes(&self, tee_upgrade_deadline_duration: Duration) -> Vec<NodeImageHash> {
        self.valid_entries(tee_upgrade_deadline_duration)
            .into_iter()
            .map(|entry| entry.image_hash)
            .collect()
    }
}

    /// Removes a launcher image hash and all its associated compose hashes.
    /// Returns `false` if the launcher hash was not found or if removal would leave the list empty.
    pub fn remove(&mut self, launcher_hash: &LauncherImageHash) -> bool {
        let would_remain = self
            .entries
            .iter()
            .filter(|e| &e.launcher_hash != launcher_hash)
            .count();
        if would_remain == 0 {
            return false;
        }
        let len_before = self.entries.len();
        self.entries.retain(|e| &e.launcher_hash != launcher_hash);
        self.entries.len() < len_before
    }
