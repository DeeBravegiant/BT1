### Title
Cross-Network Replay of `SignedDelegateAction` (Meta-Transaction) Signatures — (`File: core/primitives/src/signable_message.rs`)

### Summary

The `SignableMessage` signing scheme used for NEAR meta-transactions (`DelegateAction` / NEP-366) does not bind the signed payload to any chain identifier. A `SignedDelegateAction` produced on one NEAR network (e.g., testnet) can be extracted from its outer transaction and replayed on a different network (e.g., mainnet) by any unprivileged observer, causing the inner actions to execute on the wrong network under the victim's identity.

### Finding Description

Regular NEAR `SignedTransaction` objects include a `block_hash` field that is chain-specific, providing implicit replay protection across networks. `DelegateAction` (meta-transactions, NEP-366) uses a different signing scheme: `SignableMessage` wraps the payload with only a `MessageDiscriminant` (a NEP number) before hashing. [1](#0-0) 

The `sign` method hashes `borsh(discriminant || msg)` with no chain identifier: [2](#0-1) 

`DelegateAction::get_nep461_hash()` calls this with only `SignableMessageType::DelegateAction` as the discriminant: [3](#0-2) 

The `DelegateAction` struct itself contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — none of which are chain-specific: [4](#0-3) 

The same is true for `VersionedDelegateActionPayload` / `DelegateActionV2` (NEP-611): [5](#0-4) 

The runtime verifies the inner signature at `apply_delegate_action` using only `signed_delegate_action.verify()`, which calls the chain-unbound `get_nep461_hash()`: [6](#0-5) 

### Impact Explanation

An attacker who observes a valid `SignedDelegateAction` on testnet (or any NEAR network) can extract it and wrap it in a fresh outer `SignedTransaction` on mainnet. If the victim account exists on both networks with the same access key and a matching nonce, the inner actions execute on mainnet under the victim's `predecessor_id`. Concrete corrupted state: the victim's access key nonce on mainnet is incremented, and all inner actions (token transfers, key additions/deletions, contract calls) execute as if authorized by the victim. The `predecessor_id` in the spawned receipt equals `sender_id`, so any contract checking `predecessor_id` treats the call as legitimate.

### Likelihood Explanation

Many NEAR users and developers maintain the same account names and key pairs on both mainnet and testnet. The nonce on testnet is often lower than on mainnet (or vice versa), making nonce alignment achievable. The `max_block_height` field is a plain integer; mainnet and testnet block heights are in similar ranges, so a value valid on one network is frequently valid on the other. The `SignedDelegateAction` is fully public inside the outer transaction visible on-chain. No privileged access is required: the attacker only needs to monitor one network and submit a transaction on the other.

### Recommendation

Include the genesis hash or chain identifier in the `SignableMessage` payload. The `GenesisId` struct (containing both `chain_id` string and genesis block hash) is already used for network-level handshake separation: [7](#0-6) 

The `SignableMessage` struct or the `MessageDiscriminant` should incorporate the `GenesisId` (or at minimum the genesis hash) so that a signature produced on one network cannot verify on another. This mirrors how regular `SignedTransaction` objects are implicitly chain-bound via `block_hash`.

### Proof of Concept

1. Alice on testnet signs a `DelegateAction` (e.g., `ft_transfer` to Bob) with `nonce = N`, `max_block_height = H`. A relayer submits it on testnet. The outer transaction is visible on-chain.
2. Attacker extracts the `SignedDelegateAction` bytes from the testnet transaction.
3. Attacker checks mainnet: Alice has the same account and key, with access key nonce `N-1` (one behind). Mainnet block height is also below `H`.
4. Attacker constructs a new outer `SignedTransaction` on mainnet (signed by the attacker's own key), with `receiver_id = alice.near` and `actions = [Action::Delegate(extracted_signed_delegate_action)]`.
5. Mainnet runtime calls `apply_delegate_action` → `signed_delegate_action.verify()` → `get_nep461_hash()` → `SignableMessage { discriminant: NEP_366, msg: &delegate_action }`. The hash is identical to the testnet hash because no chain identifier is included. Signature verifies.
6. The `ft_transfer` executes on mainnet under Alice's identity without her consent. Alice's mainnet nonce advances and her mainnet FT balance is drained. [8](#0-7) [9](#0-8)

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}
