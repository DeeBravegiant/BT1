### Title
Single Byzantine Participant Can Abort Any CKD PV Request via Unchecked Contribution Aggregation — (`crates/threshold-signatures/src/confidential_key_derivation/protocol_pv.rs`)

---

### Summary

The `do_ckd_coordinator` function in `protocol_pv.rs` aggregates `CKDOutput` contributions from all participants with no per-participant validity check. A single Byzantine participant can send arbitrary valid G1 points as their `(big_y, big_c)` contribution, corrupting the aggregated sum so that `aggregated_output_check` fails, causing the coordinator to return `ProtocolError::AssertionFailed` and abort the CKD request. Because the participant identity is discarded during aggregation, the malicious participant cannot be identified from the protocol output, making the abort repeatable on every retry.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects one `CKDOutput` from every other participant and unconditionally adds each contribution to a running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The participant identity is discarded (`_`). No per-participant proof of correct computation is required or checked before the contribution is folded into the sum.

After aggregation, the coordinator runs a single pairing check on the final result:

```rust
if !aggregated_output_check(&ckd_output, app_pk, &key_pair.public_key, &hash_point) {
    return Err(ProtocolError::AssertionFailed(
        "CKD output failed to verify".to_string(),
    ));
}
``` [2](#0-1) 

`aggregated_output_check` only validates the aggregated output, not individual contributions:

```rust
fn aggregated_output_check(output: &CKDOutput, ...) -> bool {
    if !check_valid_point_g1(output.big_c.into()) || !check_valid_point_g1(output.big_y.into()) {
        return false;
    }
    multi_miller_loop(&[...])
}
``` [3](#0-2) 

`check_valid_point_g1` only verifies that a point is on the curve and in the prime-order subgroup:

```rust
pub(crate) fn check_valid_point_g1(p: G1Affine) -> bool {
    (p.is_on_curve() & p.is_torsion_free()).into()
}
``` [4](#0-3) 

Any valid G1 point (e.g., the generator `G`, or `2·G`) passes this check. A Byzantine participant can therefore send `big_y = G, big_c = G` — both valid, both accepted by deserialization — and the coordinator will add them to the honest sum. The resulting aggregated `(norm_big_y + G, norm_big_c + G)` does not satisfy the pairing equation `e(C, g2) = e(Y, app_pk2) · e(H, pk)`, so `aggregated_output_check` returns `false` and the coordinator aborts.

**Correction to the question's specific attack vector:** Sending `big_y = identity` and `big_c = identity` does *not* trigger the failure, because the identity element is the additive neutral — adding it to the running sum leaves the sum unchanged, so the honest aggregated result is preserved and the check passes. The effective attack uses any *non-identity* arbitrary valid G1 point.

`recv_from_others` in `helpers.rs` performs no content validation; it only deduplicates by sender: [5](#0-4) 

Because the participant identity is thrown away in the aggregation loop, the coordinator has no way to determine which participant sent the corrupting contribution. Every retry of the CKD request can be aborted by the same participant.

---

### Impact Explanation

A single participant (strictly below the signing threshold) can abort any `ckd_pv` request for any `app_id` by sending one crafted message per round. Because the protocol provides no per-participant accountability, the malicious participant cannot be identified from the protocol output alone, and every retry is equally vulnerable. Any application or user whose derived key is produced exclusively through `ckd_pv` cannot obtain that key while the attacker participates, freezing any funds or secrets that depend on it.

---

### Likelihood Explanation

The attacker needs only to be a registered participant in the MPC network — no threshold collusion, no key material, no privileged access. The crafted message is a pair of valid G1 points, trivially constructable. The attack is repeatable on every CKD request for any targeted `app_id`.

---

### Recommendation

Add a per-participant zero-knowledge proof of correct contribution before folding into the aggregated sum. Concretely, each participant should attach a proof that their `(big_y, big_c)` was computed from their actual key share (e.g., a Chaum-Pedersen / DLEQ proof over the BLS12-381 pairing). The coordinator must verify each proof before adding the contribution. Contributions that fail verification should be rejected and the offending participant identified, rather than silently corrupting the aggregate. The non-PV variant (`protocol.rs`) has the same structural gap but no final check, so it silently produces wrong output — that path warrants a separate review.

---

### Proof of Concept

```rust
// Byzantine participant sends big_y = G, big_c = G (both valid G1 points)
// instead of their honest contribution.
// The coordinator adds these to the running sum:
//   norm_big_y_total = honest_sum_y + G   (corrupted)
//   norm_big_c_total = honest_sum_c + G   (corrupted)
// aggregated_output_check verifies:
//   e(norm_big_c_total, -g2) · e(norm_big_y_total, app_pk2) · e(hash_point, pk) = 1
// With the extra G terms this equation does not hold → returns false
// → ProtocolError::AssertionFailed → CKD request aborted.
//
// Retry: same participant sends G again → same abort.
// No mechanism in the protocol identifies the sender.
```

The integration test described in the question (one participant sends a zero/crafted `CKDOutput`, verify the protocol either completes or identifies the malicious participant) would confirm the abort and the absence of attribution.
