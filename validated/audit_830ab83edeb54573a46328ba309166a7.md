Audit Report

## Title
L2 Oracle Rate Divergence Causes wrsETH Over-Minting and Permanent Wrapper Undercollateralization - (File: contracts/pools/RSETHPoolV3.sol, contracts/L1Vault.sol)

## Summary
`RSETHPoolV3.deposit()` mints wrsETH immediately using the L2 oracle rate from `InterimRSETHOracle`, which is manually updated by a privileged manager and is inherently asynchronous relative to the L1 `LRTOracle.rsETHPrice`. When the L2 oracle lags behind the L1 price, more wrsETH is minted than the rsETH that will be deposited into `RsETHTokenWrapper` to back it, creating a permanent, compounding shortfall in the wrapper's collateral.

## Finding Description

**Step 1 — L2 mint (immediate, any depositor):**

`RSETHPoolV3.deposit()` calls `viewSwapRsETHAmountAndFee()`, which computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` where `rsETHToETHrate = IOracle(rsETHOracle).getRate()`. [1](#0-0) [2](#0-1) 

The L2 oracle is `InterimRSETHOracle`, which stores a manually-set `rate` updated only by a `MANAGER_ROLE` call to `setRate()`. There is no on-chain synchronization with L1. [3](#0-2) 

wrsETH is minted to the depositor immediately. The deposited ETH sits in the pool.

**Step 2 — L1 deposit (later, privileged manager):**

A manager later calls `L1Vault.depositETHForL1VaultETH()`, which deposits the bridged ETH into `LRTDepositPool` at the **current L1 oracle rate**: [4](#0-3) 

`getRsETHAmountToMint` uses `lrtOracle.rsETHPrice()` — the live L1 price, which is higher than the stale L2 rate because rsETH accrues staking rewards continuously. This yields fewer rsETH than the wrsETH already minted.

**Step 3 — Wrapper collateral shortfall (permanent):**

The rsETH minted on L1 is bridged back and deposited into `RsETHTokenWrapper` via `depositBridgerAssets`. The wrapper enforces a cap of `wrsETHSupply - balanceOfAssetInWrapper`: [5](#0-4) 

Since only 95.238 rsETH was minted (against 97.087 wrsETH), the bridger deposits 95.238 rsETH — which is below the cap, so the call succeeds — but the wrapper is left permanently short by 1.849 rsETH. The `_withdraw` function burns wrsETH 1:1 for rsETH: [6](#0-5) 

The last ~1.849 wrsETH holders cannot redeem. The deficit compounds with every deposit made while the oracle lags.

**No existing guard prevents this:** The `limitDailyMint` modifier only caps total daily wrsETH minted; it does not compare L2 and L1 rates. There is no slippage check, no minimum rsETH-per-ETH floor, and no deferred minting mechanism. [7](#0-6) 

## Impact Explanation

**Critical — Protocol insolvency / permanent undercollateralization of the wrsETH wrapper.**

The shortfall is not self-correcting: once wrsETH is minted in excess of the rsETH that will back it, the deficit is locked in. Every subsequent deposit during a lag window adds to the deficit. Redeemers who arrive last cannot withdraw their full rsETH entitlement, constituting a direct, permanent loss of user funds absorbed by the wrapper's existing reserves.

## Likelihood Explanation

**Medium.** The `InterimRSETHOracle` is explicitly described as a manual interim solution. Any non-zero delay between L1 rsETH price appreciation and the manager calling `setRate()` on L2 creates the divergence window. The bridging delay (L2→L1 ETH transfer) and the additional delay before `depositETHForL1VaultETH()` is called further widen the window. No special attacker capability is required — any depositor benefits from the stale rate passively. The condition is not edge-case; it is the normal operating state between oracle updates.

## Recommendation

1. **Deferred minting:** Do not mint wrsETH at deposit time. Issue a non-transferable receipt and mint wrsETH only after the L1 rsETH amount is confirmed and bridged back, so the minted amount equals the actual rsETH received.
2. **Rate-bound check:** Record the L2 oracle rate at deposit time and pass it as a minimum rsETH expectation to the L1 deposit step; revert if the L1 execution rate yields fewer rsETH than the wrsETH already minted.
3. **Maximum divergence guard:** Enforce a maximum acceptable divergence between the L2 oracle rate and a freshness-checked L1 rate before allowing deposits, similar to a slippage guard, and revert deposits when the divergence exceeds a safe threshold.

## Proof of Concept

**Preconditions:**
- L1 `rsETHPrice` = 1.05e18 (current, reflects latest staking rewards)
- L2 `InterimRSETHOracle.rate` = 1.03e18 (stale, manager has not yet called `setRate()`)

**Step 1 — Depositor calls `RSETHPoolV3.deposit{value: 100 ether}("")` on L2:**
```
rsETHAmount = 100e18 * 1e18 / 1.03e18 ≈ 97.087e18 wrsETH minted to depositor
```

**Step 2 — Manager calls `L1Vault.depositETHForL1VaultETH()` after ETH is bridged:**
```
rsethAmountToMint = 100e18 * 1e18 / 1.05e18 ≈ 95.238e18 rsETH minted on L1
```

**Step 3 — Bridger calls `RsETHTokenWrapper.depositBridgerAssets(rsETH, 95.238e18)`:**
- `maxAmountToDepositBridgerAsset` = 97.087 − 0 = 97.087 → 95.238 < 97.087, call succeeds
- Wrapper state: wrsETH supply = 97.087, rsETH balance = 95.238
- **Permanent shortfall: 1.849 rsETH (~1.85% of deposit)**

**Foundry fork test plan:**
1. Fork mainnet/L2 at a block where `InterimRSETHOracle.rate` is behind the L1 `LRTOracle.rsETHPrice`.
2. Call `RSETHPoolV3.deposit{value: 100 ether}("")` and record wrsETH minted.
3. Simulate `L1Vault.depositETHForL1VaultETH()` with the same ETH amount and record rsETH minted.
4. Assert `rsETH_minted < wrsETH_minted`, confirming the structural shortfall.
5. Attempt to redeem all wrsETH via `RsETHTokenWrapper.withdraw()` and assert the last redeemer reverts due to insufficient rsETH balance.

    function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
        fee = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;

        // rate of rsETH in ETH
        uint256 rsETHToETHrate = getRate();

        // Calculate the final rsETH amount
        rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
