### Title
Caller-Controlled `BlockConfirmations(0)` Bypasses Bitcoin Finality Guarantee in `verify_foreign_transaction` — (File: `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, `crates/contract/src/lib.rs`)

---

### Summary

An unprivileged caller can submit a `verify_foreign_transaction` request for Bitcoin with `confirmations: BlockConfirmations(0)`. The contract performs no minimum-confirmation validation, and the Bitcoin inspector enforces only `threshold <= actual_confirmations`, so a threshold of zero always passes — even for a mempool-only (unconfirmed) transaction. The MPC network then produces a valid threshold signature attesting to that transaction. A bridge contract that trusts this attestation without independently re-checking the confirmation count in the signed payload can be drained via a Bitcoin double-spend.

---

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` accepts a `VerifyForeignTransactionRequestArgs` whose inner `ForeignChainRpcRequest::Bitcoin` carries a `confirmations: BlockConfirmations` field that is entirely caller-supplied. The contract performs no lower-bound check on this value before enqueuing the request. [1](#0-0) 

On the node side, `BitcoinInspector::extract` in `crates/foreign-chain-inspector/src/bitcoin/inspector.rs` enforces only:

```rust
let enough_block_confirmations =
    block_confirmations_threshold <= transaction_block_confirmation;
``` [2](#0-1) 

With `block_confirmations_threshold = 0`, the condition `0 <= actual_confirmations` is trivially true for any transaction — including one with zero on-chain confirmations (mempool). The inspector proceeds to extract values and the node signs the payload.

The signed payload is `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request, values })`, where `request` includes the caller-supplied `confirmations: 0`. [3](#0-2) 

The `BlockConfirmations` type is a plain `u64` wrapper with no invariant enforcing a minimum: [4](#0-3) 

Neither the contract nor the inspector enforces any protocol-level minimum. This is the direct analog of the Uniswap report's root cause: a security-critical routing/finality parameter is pre-accepted from the caller without validation, allowing an attacker to select the least-secure option (zero confirmations ↔ zero-liquidity pool).

---

### Impact Explanation

**High — Cross-chain double-spend / forged foreign-chain verification.**

1. Attacker broadcasts a Bitcoin transaction `T` sending funds to a bridge deposit address.
2. Before `T` is confirmed, attacker calls `verify_foreign_transaction` with `confirmations: BlockConfirmations(0)` and `tx_id = T`.
3. MPC nodes query the Bitcoin RPC. If the node's mempool is visible (standard `getrawtransaction` with `verbose=true` returns mempool txs on many nodes), `actual_confirmations = 0` satisfies `0 <= 0`. Nodes extract the block hash (or a zero-confirmation placeholder) and sign.
4. Attacker receives a valid MPC threshold signature over `(request={tx_id=T, confirmations=0}, values=[...])`.
5. Attacker presents this signature to a bridge contract that releases funds.
6. Attacker then double-spends `T` on Bitcoin (e.g., via RBF or a reorg), keeping both the bridge payout and the original Bitcoin.

Even with `confirmations: 1`, a single-confirmation Bitcoin transaction is trivially reorg-able. The protocol provides no floor, so the attacker picks the weakest possible finality guarantee.

This matches the allowed impact: **"High. Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass … that causes invalid bridge execution or double-spend conditions."**

---

### Likelihood Explanation

**High.** The entry point is the public, permissionless `verify_foreign_transaction` contract method. Any NEAR account can call it with a 1 yoctoNEAR deposit. No participant privilege, TEE access, or key material is required. The attacker only needs to know a Bitcoin transaction ID (their own) and set `confirmations: 0`. [5](#0-4) 

---

### Recommendation

Enforce a protocol-level minimum confirmation count in the contract before enqueuing the request. A reasonable floor (e.g., 6 for Bitcoin mainnet) should be a contract configuration parameter voted in by participants, analogous to how `ChainEntry.quorum` is voted in for RPC providers. The node-side inspector check is correct in structure but must not be the only gate — the contract must reject requests below the minimum before they are ever dispatched to nodes.

```rust
// In verify_foreign_transaction, before enqueue_yield_request:
if let ForeignChainRpcRequest::Bitcoin(ref btc_req) = request.request {
    let min_confirmations = self.config.bitcoin_min_confirmations; // voted-in param
    if btc_req.confirmations.0 < min_confirmations {
        env::panic_str("Bitcoin confirmations below protocol minimum");
    }
}
```
