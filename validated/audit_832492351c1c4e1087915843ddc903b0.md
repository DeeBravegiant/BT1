### Title
Harvester Can Unconditionally Redirect Farmer Block Rewards via Unenforced `farmer_reward_address_override` - (File: chia/farmer/farmer_api.py)

### Summary

The CHIP-22 fee convention allows a third-party harvester to redirect the farmer's block reward by setting `farmer_reward_address_override` in the `NewProofOfSpace` and `RespondSignatures` protocol messages. The farmer's only guard — `notify_farmer_reward_taken_by_harvester_as_fee()` — is a log-only function that emits warnings but never rejects or blocks the override. In `_process_respond_signatures()`, the override is applied unconditionally with no fee-quality enforcement whatsoever. Any connected harvester can therefore divert the farmer's 1.75 XCH block reward to an arbitrary address on every won block.

### Finding Description

**Root cause — log-only enforcement in `farmer.py`:**

`notify_farmer_reward_taken_by_harvester_as_fee()` computes a fee quality score and compares it to the harvester-supplied `applied_fee_threshold`. When the threshold is violated it emits `self.log.warning(...)` but returns normally, allowing execution to continue. [1](#0-0) 

**Root cause — unconditional override in `farmer_api.py`:**

In `_process_respond_signatures()`, the farmer replaces its own configured `farmer_target` with whatever address the harvester placed in `RespondSignatures.farmer_reward_address_override`, with no fee-quality check at all: [2](#0-1) 

The resulting `DeclareProofOfSpace` carries the attacker-controlled address as `farmer_reward_address` and is broadcast to all full nodes: [3](#0-2) 

**Root cause — log-only call-site in `new_proof_of_space`:**

When `NewProofOfSpace` arrives with `farmer_reward_address_override` set, the farmer calls the log-only function and then unconditionally proceeds to request signatures: [4](#0-3) 

**Protocol message definitions** — both `NewProofOfSpace` and `RespondSignatures` carry the optional override field that any connected harvester can populate freely: [5](#0-4) [6](#0-5) 

### Impact Explanation

Every time the farmer wins a block, the 1.75 XCH farmer reward is sent to the puzzle hash specified in `DeclareProofOfSpace.farmer_reward_puzzle_hash`. Because the farmer unconditionally accepts the harvester's override, a malicious third-party harvester can redirect 100 % of the farmer's block rewards to an attacker-controlled address on every won block. This is unauthorized reward diversion of XCH — a Critical impact under the allowed scope.

### Likelihood Explanation

CHIP-22 explicitly supports third-party harvesters connecting to a farmer. Any such harvester is an unprivileged network peer. The harvester already participates in the normal proof-of-space flow (it receives signage points, finds proofs, and signs messages), so it has all the context needed (`sp_hash`, `plot_identifier`, valid signatures) to craft a winning `RespondSignatures` with an arbitrary `farmer_reward_address_override`. No key compromise, admin access, or cryptographic break is required. The attack is silent — the farmer logs a warning but the block is accepted by the network normally.

### Recommendation

The fee-quality check must be **enforced**, not merely logged. When `farmer_reward_address_override` is present and the fee quality does not satisfy the declared threshold (or no `fee_info` is provided), the farmer should:

1. Reject the `NewProofOfSpace` message and not request signatures.
2. Reject the `RespondSignatures` message and not produce a `DeclareProofOfSpace`.

The enforcement should be added both in `new_proof_of_space()` (before requesting signatures) and in `_process_respond_signatures()` (before constructing `DeclareProofOfSpace`), so that neither path can be used to bypass the check.

### Proof of Concept

1. A third-party harvester connects to a farmer over the standard harvester protocol.
2. The farmer broadcasts a `NewSignagePointHarvester2` message.
3. The harvester finds a valid proof of space for the challenge.
4. The harvester sends `NewProofOfSpace` with `farmer_reward_address_override = <attacker_puzzle_hash>` and `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0xFFFFFFFF)` (maximum threshold, guaranteeing the quality check fails).
5. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`, logs `"Invalid fee threshold"`, and continues.
6. The farmer sends `RequestSignatures` to the harvester.
7. The harvester returns `RespondSignatures` with `farmer_reward_address_override = <attacker_puzzle_hash>`.
8. `_process_respond_signatures()` sets `farmer_reward_address = <attacker_puzzle_hash>` with no quality check.
9. The farmer broadcasts `DeclareProofOfSpace` with `farmer_reward_puzzle_hash = <attacker_puzzle_hash>` to all full nodes.
10. The full node includes the block; the 1.75 XCH farmer reward is paid to the attacker's address.
