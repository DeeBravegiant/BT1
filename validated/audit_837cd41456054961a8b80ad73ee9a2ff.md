### Title
Multiple Sub-Transactions Can Double-Claim the Same Deposit Refund — (`File: eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` / `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubCerts.hs`)

---

### Summary

In the Dijkstra era, a top-level transaction (`TopTx`) can embed multiple sub-transactions (`SubTx`), each of which may include certificates. Two sub-transactions within the same top-level transaction can both include `UnRegDepositTxCert` for the **same staking credential**, causing the deposit refund to be claimed twice. The predicate failure that should prevent this is explicitly acknowledged as unimplemented in the production codebase, meaning the double-claim currently succeeds and ADA is created from nothing.

---

### Finding Description

The Dijkstra era introduces sub-transactions (`dtbrSubTransactions :: OMap TxId (Tx SubTx era)`) embedded inside a top-level transaction body. Each sub-transaction body (`DijkstraSubTxBodyRaw`) can carry its own certificate set (`dstbrCerts`), including deregistration certificates (`UnRegDepositTxCert`).

The `DijkstraSUBCERTS` / `DijkstraSUBCERT` rules process each sub-transaction's certificates independently. There is no cross-sub-transaction guard that prevents two sub-transactions from both issuing `UnRegDepositTxCert` for the same staking credential within the same top-level transaction.

The developers are aware of this gap. The test file explicitly marks the scenario as a pending failure with the comment `"TODO: predicate failure not yet implemented"`:

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

Because the test is `xit` (skipped/pending) and the expected predicate failure is an `error` placeholder, the current production code **accepts** such a transaction rather than rejecting it.

The root cause is structurally identical to the reported batchId collision: two execution paths (sub-transaction 1 and sub-transaction 2) share the same account/deposit state, and neither path is restricted from consuming the shared resource. In the original report, `execTransaction` was not restricted from using `batchId #0`; here, sub-transaction 2 is not restricted from unregistering a credential already unregistered by sub-transaction 1 within the same atomic top-level transaction. [1](#0-0) 

The sub-transaction body type confirms both `TopTx` and `SubTx` carry independent certificate sets with no shared-state deduplication guard: [2](#0-1) 

The `subTransactionsTxBodyL` lens confirms sub-transactions are stored as an `OMap TxId (Tx SubTx era)` — keyed by body hash, not by the certificates they contain — so two sub-transactions with different inputs but the same deregistration certificate are structurally distinct and both admitted: [3](#0-2) 

---

### Impact Explanation

**Critical. Direct creation of ADA through an invalid ledger state transition.**

When two sub-transactions both successfully process `UnRegDepositTxCert stakingCred keyDeposit` for the same credential, the deposit is refunded twice. The deposit pool loses `2 × keyDeposit` ADA while only one deposit was ever locked. This is unbounded: an attacker can include N sub-transactions each claiming the same refund, multiplying the ADA extracted by N.

---

### Likelihood Explanation

Any unprivileged transaction sender can craft this transaction. No special role, key, or governance majority is required. The attacker only needs:
1. A registered staking credential (or the ability to register one).
2. Two UTxO inputs (one per sub-transaction, to make the sub-transaction bodies distinct so the `OMap` deduplication by body hash does not collapse them).
3. The ability to submit a Dijkstra-era transaction.

The attack is deterministic and requires no timing race.

---

### Recommendation

In the `DijkstraSUBCERTS` or `DijkstraSUBLEDGERS` rule, accumulate the set of staking credentials deregistered across all sub-transactions and reject the top-level transaction if any credential appears in more than one sub-transaction's deregistration certificates. Concretely, implement the missing predicate failure referenced by the `TODO` comment in `CertSpec.hs`, analogous to the fix recommended in the original report: add an explicit guard that prevents the shared resource (deposit refund) from being consumed more than once across the two execution paths. [1](#0-0)

### Proof of Concept

```
1. Register staking credential C with deposit D.
2. Obtain two UTxO inputs I1 and I2.
3. Construct:
     subTx1 = mkBasicTx mkBasicTxBody
               & bodyTxL . inputsTxBodyL  .~ {I1}
               & bodyTxL . certsTxBodyL   .~ [UnRegDepositTxCert C D]
     subTx2 = mkBasicTx mkBasicTxBody
               & bodyTxL . inputsTxBodyL  .~ {I2}
               & bodyTxL . certsTxBodyL   .~ [UnRegDepositTxCert C D]
     topTx  = mkBasicTx mkBasicTxBody
               & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
4. Submit topTx.
5. Observe: deposit D is refunded twice (net gain of D ADA from the deposit pool).
```

This is exactly the scenario the disabled test `xit "Multiple subtransactions cannot get the same refund"` describes, with the expected predicate failure explicitly marked `"TODO: predicate failure not yet implemented"`. [1](#0-0)
