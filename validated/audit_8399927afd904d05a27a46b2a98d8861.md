### Title
Excess Deposit Silently Retained by Contract in `submit_participant_info` Re-submission Path ā€” (`crates/contract/src/lib.rs`)

---

### Summary

`submit_participant_info` is marked `#[payable]` and contains a conditional deposit-check block. When an existing participant re-submits their attestation (the common periodic refresh case), the entire deposit-check block is skipped, so any NEAR tokens attached to the call are silently absorbed by the contract with no refund.

---

### Finding Description

The function computes a boolean gate:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
``` [1](#0-0) 

The deposit check and refund logic is only executed when `attestation_storage_must_be_paid_by_caller` is `true`:

```rust
if attestation_storage_must_be_paid_by_caller {
    // ...
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
``` [2](#0-1) 

When `attestation_storage_must_be_paid_by_caller` is `false` ā€” i.e., the caller is already a registered participant **and** the submission updates an existing entry rather than inserting a new one ā€” the `if` block is never entered. Because the function is `#[payable]`, any NEAR tokens attached to the call are accepted by the runtime and credited to the contract's balance. There is no `else` branch, no `env::attached_deposit()` read, and no refund promise issued. The tokens are permanently locked in the contract.

This is the direct analog of the CBridge `>=` vs `==` issue: the contract accepts any deposit amount in a specific code path but only "uses" zero of it, silently retaining the rest.

---

### Impact Explanation

Any NEAR tokens a participant node attaches when periodically refreshing its attestation are permanently lost to the contract. The contract's balance grows by the excess, but there is no withdrawal or recovery mechanism for the depositor. This breaks the accounting invariant that `#[payable]` functions must either consume or refund every attached token.

**Impact: Medium** ā€” balance/accounting invariant broken; funds permanently locked in the contract without any recovery path.

---

### Likelihood Explanation

MPC nodes call `submit_participant_info` on a recurring schedule to keep their attestation fresh. The automated node code currently attaches `0` deposit for re-submissions, so the bug is dormant under normal operation. However:

- A node operator following the manual instructions in the deployment guide (which shows attaching deposits) could trigger the loss.
- Any future code change that attaches a non-zero deposit for re-submissions would silently lose those tokens.
- The condition is reachable by any existing participant without any special privilege.

**Likelihood: Low-to-Medium** ā€” not triggered by the current automated node path, but reachable by a misconfigured or manually operated node.

---

### Recommendation

Add an `else` branch (or move the refund outside the conditional) to return any attached deposit when storage payment is not required:

```rust
if attestation_storage_must_be_paid_by_caller {
    // existing check + refund logic
} else {
    // Refund any accidentally attached deposit
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, mirror the pattern used in `require_deposit` and `propose_update`, which unconditionally refund any excess above the required amount. [3](#0-2) 

---

### Proof of Concept

1. Participant `alice.near` is already registered (appears in `voter_account()` successfully).
2. `alice.near` calls `submit_participant_info` with a refreshed attestation and attaches `1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR) ā€” e.g., following the manual deployment guide.
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` (not `NewlyInsertedParticipant`).
4. `is_new_attestation = false`, `caller_is_not_participant = false` ā†’ `attestation_storage_must_be_paid_by_caller = false`.
5. The `if` block is skipped entirely; no refund promise is created.
6. The 1 NEAR is credited to the contract's balance and is unrecoverable by `alice.near`. [4](#0-3)
