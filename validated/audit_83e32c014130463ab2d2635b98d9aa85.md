Audit Report

## Title
Zero-Rate Broadcast via Unguarded `updateRate()` on Uninitialized `rsETHPrice` — (`contracts/cross-chain/RSETHRateProvider.sol`)

## Summary
`LRTOracle.initialize()` leaves `rsETHPrice` at its default value of `0`. Because `CrossChainRateProvider.updateRate()` is permissionless and contains no zero-rate guard, any caller can invoke it before the first `updateRSETHPrice()` and broadcast `rate = 0` via LayerZero to `RSETHRateReceiver` on the destination chain. The receiver stores the value without validation, causing `getRate()` to return `0` and breaking any downstream integration that depends on a valid rate.

## Finding Description
`LRTOracle.initialize()` sets only `lrtConfig` and emits an event; `rsETHPrice` and `highestRsethPrice` are left at `0`. [1](#0-0) 

`RSETHRateProvider.getLatestRate()` reads `rsETHPrice` directly from storage with no floor check, so it returns `0` during the initialization window. [2](#0-1) 

`CrossChainRateProvider.updateRate()` carries only a `nonReentrant` guard — no access control, no `require(latestRate > 0)`. It sets `rate = latestRate` and encodes it into the LayerZero payload unconditionally. [3](#0-2) 

`CrossChainRateReceiver.lzReceive()` decodes the payload and stores it without any validation: `rate = _rate`. [4](#0-3) 

The zero-supply guard inside `_updateRsETHPrice()` (which would set `rsETHPrice = 1 ether`) only runs when that function is explicitly called; it has no effect on the stored value until invoked. [5](#0-4) 

## Impact Explanation
Once `RSETHRateReceiver.rate` is set to `0`, any downstream pool or wrapper calling `getRate()` receives `0`. Integrations that use this value as a denominator (e.g., for deposit sizing or price calculations) will revert or produce incorrect results, temporarily breaking cross-chain deposit flows. No user funds are lost or frozen; the contract simply fails to deliver its promised rate. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The vulnerable window opens at `LRTOracle` proxy deployment and closes after the first successful `updateRSETHPrice()` call. During this window `updateRate()` is callable by any EOA willing to pay LayerZero fees — a low barrier. The same window reopens if the proxy is upgraded and storage is reset. The precondition is narrow but realistic during protocol initialization or re-initialization, and no privileged access is required.

## Recommendation
Add a zero-rate guard in `CrossChainRateProvider.updateRate()`:

```solidity
uint256 latestRate = getLatestRate();
require(latestRate > 0, "Rate must be non-zero");
```

Alternatively, initialize `rsETHPrice` to `1 ether` inside `LRTOracle.initialize()`, consistent with the behavior already present in `_updateRsETHPrice()` when `rsethSupply == 0`. [6](#0-5) 

## Proof of Concept
1. Deploy `LRTOracle` via proxy. Confirm `rsETHPrice == 0`, `highestRsethPrice == 0`.
2. Do **not** call `updateRSETHPrice()`.
3. From any EOA, call `RSETHRateProvider.updateRate{value: lzFee}()`.
4. `getLatestRate()` returns `ILRTOracle.rsETHPrice()` → `0`.
5. `CrossChainRateProvider` sets `rate = 0` and sends `abi.encode(0)` via LayerZero.
6. On the destination chain, `lzReceive` decodes `0` and sets `RSETHRateReceiver.rate = 0`.
7. Assert `RSETHRateReceiver.getRate() == 0`.
8. Any pool calling `getRate()` as a denominator now reverts or misbehaves.

**Foundry test sketch:**
```solidity
function test_zeroBroadcast() public {
    // LRTOracle deployed, rsETHPrice not yet set
    assertEq(lrtOracle.rsETHPrice(), 0);
    // Mock LZ endpoint to capture payload
    vm.prank(attacker);
    rateProvider.updateRate{value: 1 ether}();
    // Simulate lzReceive on receiver with captured payload (abi.encode(0))
    vm.prank(address(lzEndpoint));
    rateReceiver.lzReceive(srcChainId, abi.encodePacked(address(rateProvider), address(rateReceiver)), 0, abi.encode(uint256(0)));
    assertEq(rateReceiver.getRate(), 0);
}
```
