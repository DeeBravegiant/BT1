Audit Report

## Title
Missing Chainlink Oracle Staleness and Validity Checks Allow Stale Price to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `answeredInRound`, and `roundId`, and does not check for a non-positive price. A stale or zero price propagates through `LRTOracle._getTotalEthInProtocol()` into `rsETHPrice`, which governs how much rsETH depositors receive. The same codebase already applies all three guards in `ChainlinkOracleForRSETHPoolCollateral.getRate()`, confirming the pattern is known and intentionally applied elsewhere.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` reads the feed as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All four non-price return values are discarded; `price` is not checked for `<= 0`. By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` performs:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

`ChainlinkPriceOracle` is the oracle registered in `LRTOracle` for supported LST assets. `LRTOracle.getAssetPrice()` delegates directly to it, and `_getTotalEthInProtocol()` calls `getAssetPrice()` for every supported asset to compute total ETH backing rsETH. [3](#0-2) 

That total drives `_updateRsETHPrice()`, which sets the stored `rsETHPrice` used to mint rsETH to depositors. [4](#0-3) 

`updateRSETHPrice()` is `public` with no access control beyond `whenNotPaused`, so any external caller can trigger a price update at will. [5](#0-4) 

The downside-protection guard at lines 270–282 partially mitigates large drops by auto-pausing when the price decrease exceeds `pricePercentageLimit`, but it does not eliminate the risk:
- When `pricePercentageLimit == 0` the guard is entirely disabled (`pricePercentageLimit > 0 && ...` is always false).
- When the stale price deviation is within the configured threshold, the deflated price is accepted and written to `rsETHPrice` without pausing. [6](#0-5) 

## Impact Explanation
**Theft of unclaimed yield (High).** When a Chainlink LST/ETH feed goes stale at a price below the true value but within the `pricePercentageLimit` threshold (or when the limit is zero), an attacker calls `updateRSETHPrice()` to write the deflated price into `rsETHPrice`. New deposits are then minted at the artificially low rate, issuing excess rsETH. When the feed recovers and the price is corrected upward, the attacker's excess rsETH is worth more ETH than deposited, at the direct expense of existing rsETH holders whose accrued yield is diluted.

**Temporary freezing of funds (Medium).** A stale price of zero (or a sufficiently large negative deviation) causes `_getTotalEthInProtocol()` to collapse, triggering the auto-pause of `LRTDepositPool` and `WithdrawalManager`, freezing all user deposits and withdrawals until an admin manually unpauses.

## Likelihood Explanation
Chainlink feeds have documented historical staleness during network congestion and sequencer outages. The protocol operates on mainnet and multiple L2s. `updateRSETHPrice()` is callable by any unprivileged address with no rate limiting, so an attacker can monitor mempool/feed state and time the call to coincide with a stale window. No keeper or access restriction prevents this. Likelihood: Medium.

## Recommendation
Apply the same validation pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```diff
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

-   (, int256 price,,,) = priceFeed.latestRoundData();
+   (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
+       priceFeed.latestRoundData();
+   if (answeredInRound < roundId) revert StalePrice();
+   if (updatedAt == 0) revert IncompleteRound();
+   if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, add a per-feed configurable `heartbeat` staleness threshold: `if (block.timestamp - updatedAt > heartbeat) revert StalePrice();`.

## Proof of Concept
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale at price `P_stale < P_real` due to a sequencer outage or feed pause, with the deviation within `pricePercentageLimit` (or `pricePercentageLimit == 0`).
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `LRTOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `P_stale` without revert.
4. `totalETHInProtocol` is undervalued → `newRsETHPrice` is set below true value and written to `rsETHPrice`.
5. Attacker immediately calls `LRTDepositPool.depositAsset(stETH, amount, minRSETH)` and receives excess rsETH at the deflated rate.
6. When the feed recovers and `rsETHPrice` is corrected upward, the attacker's excess rsETH is worth more ETH than deposited, at the expense of existing holders.

**Foundry fork test plan:** Fork mainnet, mock a Chainlink stETH/ETH aggregator to return a stale answer with `updatedAt` far in the past and `answeredInRound < roundId`. Call `updateRSETHPrice()` as an unprivileged address. Assert `rsETHPrice` is set to the stale value. Deposit stETH as the attacker. Restore the mock to the real price, call `updateRSETHPrice()` again. Assert the attacker's rsETH balance is worth more ETH than deposited, and that existing holder share is diluted.
