### Title
Detached NEAR Token Refund Promises May Silently Fail, Permanently Locking User Deposits — (File: crates/contract/src/lib.rs)

---

### Summary

Multiple refund `Promise::transfer()` calls throughout `mpc-contract` are issued with `.detach()`, meaning their execution result is never observed. If a refund transfer fails at the NEAR runtime level (e.g., the recipient account no longer exists), the excess deposit is silently retained by the contract with no recovery path. This is the direct NEAR analog of the ERC20 unchecked-return-value class: a value-transfer whose failure is structurally invisible to the contract.

---

### Finding Description

In `crates/contract/src/lib.rs`, the helper `require_deposit()` and several payable methods issue refund transfers using the pattern:

```rust
Promise::new(predecessor.clone()).transfer(diff).detach();
``` [1](#0-0) 

The same pattern appears in `submit_participant_info()`:

```rust
Promise::new(account_id).transfer(diff).detach();
``` [2](#0-1) 

And in `propose_update()`:

```rust
Promise::new(proposer).transfer(diff).detach();
``` [3](#0-2) 

The grep result confirms 13 total `.detach()` call sites in `lib.rs`. [4](#0-3) 

In NEAR's execution model, `.detach()` schedules the promise but discards its outcome receipt. If the runtime-level transfer fails — because the recipient account was deleted between transaction submission and promise execution, or because the recipient is a smart contract with no `receive` handler — the tokens are returned to the contract's balance and the failure is never surfaced. The contract's state (request enqueued, storage charged, update proposed) has already been committed in the preceding receipt and is not rolled back.

The `require_deposit()` helper is invoked from `sign()` and `request_app_private_key()`, the two primary user-facing endpoints. [4](#0-3) 

---

### Impact Explanation

**Medium.** The accounting invariant "a caller who deposits more than the required minimum receives the difference back" is broken whenever the refund promise fails silently. The excess deposit is permanently locked inside the MPC contract with no on-chain recovery mechanism. This breaks the production safety/accounting invariant documented in the contract README ("Any excess deposit is automatically refunded") without requiring network-level DoS or operator misconfiguration. [5](#0-4) 

The funds are not stolen by an adversary — they remain in the contract — but they are permanently inaccessible to the depositor, constituting a direct, irreversible loss of user funds.

---

### Likelihood Explanation

**Low-to-moderate.** The most realistic trigger is a smart-contract caller (e.g., a DeFi protocol or a cross-contract integration) that calls `sign()` or `request_app_private_key()` with an excess deposit and whose own contract does not implement a `receive`/transfer-acceptance handler, or whose account is deleted between the originating receipt and the refund receipt. NEAR accounts can be deleted at any time; the refund promise executes in a later receipt, not atomically with the originating call. Any of the 13 `.detach()` sites is reachable by an unprivileged caller with no special role.

---

### Recommendation

Replace every `.detach()` refund with a chained callback that checks the transfer result and either logs the failure or re-attempts the refund. At minimum, remove `.detach()` and return the `Promise` to the caller so the runtime propagates failure:

```rust
// Instead of:
Promise::new(predecessor.clone()).transfer(diff).detach();

// Use (return the promise so failure is observable):
return Promise::new(predecessor.clone()).transfer(diff);
// or chain a .then() callback that handles failure
```

For methods that must return a value rather than a `Promise`, use a `.then()` self-callback to handle the transfer result and emit a log or re-credit the amount from contract storage.

---

### Proof of Concept

1. Deploy a smart contract `caller.near` that has **no** `receive` or `ft_on_transfer` handler (i.e., it panics on any incoming transfer).
2. From `caller.near`, call `sign()` on the MPC contract attaching `2 yoctonear` (minimum is `1 yoctonear`).
3. The `require_deposit()` helper computes `diff = 1 yoctonear` and schedules `Promise::new(caller.near).transfer(1).detach()`.
4. The transfer receipt executes; `caller.near` panics on receipt → transfer fails.
5. The `1 yoctonear` is returned to the MPC contract's balance.
6. No error is surfaced to `caller.near`; the sign request proceeds normally.
7. `caller.near` has permanently lost `1 yoctonear` with no recovery path.

The same scenario applies to `submit_participant_info()` (storage deposit refund) and `propose_update()` (update deposit refund), where the excess amounts can be substantially larger than `1 yoctonear`. [4](#0-3) [6](#0-5) [7](#0-6)

fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    let deposit = env::attached_deposit();
    match deposit.checked_sub(minimum_deposit) {
        None => {
            env::panic_str(
                &InvalidParameters::InsufficientDeposit {
                    attached: deposit.as_yoctonear(),
                    required: minimum_deposit.as_yoctonear(),
                }
                .to_string(),
            );
        }
        Some(diff) => {
            if diff > NearToken::from_yoctonear(0) {
                log!("refund excess deposit {diff} to {predecessor}");
                Promise::new(predecessor.clone()).transfer(diff).detach();
            }
        }
    }
}
