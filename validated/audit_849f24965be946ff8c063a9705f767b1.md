Audit Report

## Title
Pool Admin Can Set Per-Bin Additional Fees Without Cap Enforcement, Bypassing Protocol Fee Limits — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no cap validation, while `setPoolAdminFees` enforces explicit caps against `maxAdminSpreadFeeE6` and `HARD_MAX_SPREAD_FEE_E6`. Because the per-bin additional fee is additive to the oracle-derived base fee in swap math, a pool admin can push the effective swap fee for any bin beyond the protocol's 20% hard ceiling, causing traders to receive fewer tokens than the protocol-sanctioned maximum permits.

## Finding Description
`setPoolAdminFees` enforces caps at [1](#0-0)  before forwarding values to the pool. In contrast, `setPoolBinAdditionalFees` performs no such validation: [2](#0-1) 

`setBinAdditionalFees` in the pool only validates the bin index, not the fee magnitude: [3](#0-2) 

`addFeeBuyE6`/`addFeeSellE6` are stored as `uint16` (max 65 535): [4](#0-3) 

In swap execution, the per-bin fee is directly additive to `baseFeeX64`: [5](#0-4)  and again in `_executeSwap`: [6](#0-5) 

The protocol's hard cap is `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%): [7](#0-6) 

`uint16` max (65 535) in E6 units = 6.5535%. When added to a global spread fee at the 20% hard cap, the effective fee reaches ~26.55%, silently exceeding the hard ceiling the protocol enforces everywhere else. The periphery data provider mirrors the same additive pattern: [8](#0-7) 

## Impact Explanation
This is an admin-boundary break: the pool admin role is semi-trusted only within caps and timelocks, but the per-bin fee path has no analogous guard. The result is swap conservation failure — traders swapping through the affected bin pay an effective fee exceeding the protocol's 20% hard ceiling, receiving fewer output tokens than the oracle/bin curve permits. The excess is retained as LP spread surplus, constituting a direct loss of trader principal beyond the protocol-sanctioned limit.

## Likelihood Explanation
The pool admin has direct, immediate access to `setPoolBinAdditionalFees` with no timelock, no governance vote, and no second step. The call path is three lines with no revert for any fee value ≤ 65 535. The active bin (highest swap traffic) is the primary target; setting its additional fee to max immediately affects all live swaps through that bin.

## Recommendation
Add cap validation in `setPoolBinAdditionalFees` consistent with `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce `globalSpreadFeeE6 + addFeeBuyE6 ≤ HARD_MAX_SPREAD_FEE_E6` at call time, mirroring the invariant already maintained for global fees.

## Proof of Concept
```
1. Pool deployed with spreadFeeE6 = 200_000 (20%, at the hard cap).
2. Pool admin calls:
       factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
   → No revert. BinState[0].addFeeBuyE6 = 65535, addFeeSellE6 = 65535.
3. Trader calls swap (zeroForOne = false, buying token0):
   Effective buy fee in swap math:
       feeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
             ≈ spreadFeeX64(20%) + 6.5535%
             ≈ 26.5535% total
4. Trader pays ~26.55% above oracle mid-price instead of the protocol-capped 20%.
   Excess ~6.55% is retained in the pool as LP surplus — trader loses real token1
   beyond what the protocol's fee ceiling permits.
```
