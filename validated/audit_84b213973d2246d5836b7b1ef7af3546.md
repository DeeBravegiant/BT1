Audit Report

## Title
Untracked ETH via `receive()` Causes TVL Undercount and Temporary Fund Freeze — (File: contracts/LRTConverter.sol)

## Summary
`LRTConverter` accepts ETH via `receive() external payable {}` but never updates `ethValueInWithdrawal` for ETH arriving through this path. `LRTDepositPool.getETHDistributionData()` reads only `ethValueInWithdrawal` — not `address(lrtConverter).balance` — so any such ETH is invisible to the rsETH price oracle. Because the only ETH exit path is through `claimStEth`/`claimSwEth`, which require live Lido/Swell withdrawal requests, ETH arriving when no such request exists is temporarily frozen. A secondary defect causes `ethValueInWithdrawal` to be zeroed prematurely when extra ETH is present at claim time, corrupting TVL accounting for remaining pending withdrawals.

## Finding Description

**Root cause — untracked ETH via `receive()`:**

`LRTConverter` declares an open payable fallback: [1](#0-0) 

`ethValueInWithdrawal` is only mutated in `transferAssetFromDepositPool` (L140), `transferAssetToDepositPool` (L163), and `_sendEthToDepositPool` (L255–258). ETH arriving via `receive()` touches none of these paths: [2](#0-1) 

`LRTDepositPool.getETHDistributionData()` reads only the storage variable, never `address(lrtConverter).balance`: [3](#0-2) 

Any ETH that arrives via `receive()` is therefore invisible to the rsETH price oracle, understating TVL and lowering `rsETHPrice()`, causing new depositors to receive more rsETH than fair value and diluting existing holders.

**No exit path when no Lido/Swell request exists:**

The only functions that move ETH out of `LRTConverter` are `claimStEth` and `claimSwEth`, both of which call `_sendEthToDepositPool(address(this).balance)`: [4](#0-3) [5](#0-4) 

There is no `sweep()`, `recoverETH()`, or equivalent. If no claimable Lido/Swell withdrawal request exists, the ETH has no exit path until the operator manually initiates a new stETH unstake cycle — a multi-day process.

**Secondary defect — premature zeroing of `ethValueInWithdrawal`:**

When a claim is eventually made, `_sendEthToDepositPool` is called with `address(this).balance`, which includes both the legitimately claimed ETH and any extra ETH. The decrement logic: [6](#0-5) 

If `address(this).balance` (claimed + extra) exceeds `ethValueInWithdrawal`, the variable is set to `0` even though other pending Lido requests remain outstanding. Those remaining pending-withdrawal ETH amounts then vanish from TVL until their respective claims are processed.

## Impact Explanation

**Medium — Temporary freezing of funds:** ETH sent to `LRTConverter.receive()` when no Lido/Swell withdrawal request is claimable has no exit path. The operator must transfer stETH into the converter, initiate a new Lido withdrawal, and wait through the unbonding period (multiple days) before the ETH can be swept. This constitutes a temporary freeze of funds matching the allowed Medium impact class.

**Low — Contract fails to deliver promised returns:** While the extra ETH is untracked, `rsETHPrice()` is understated. New depositors receive excess rsETH, diluting existing holders. The premature zeroing of `ethValueInWithdrawal` compounds this by also hiding legitimately pending withdrawal ETH from TVL.

## Likelihood Explanation

ETH can reach `LRTConverter.receive()` via accidental transfer, `selfdestruct` proceeds directed at the contract, or protocol-level airdrops — all of which are possible without any privileged access. The scenario where no Lido/Swell request is outstanding is realistic during quiet periods or immediately after all requests have been claimed. The TVL undercount is triggered by any such ETH arrival regardless of whether a request exists. Both conditions are reachable by an unprivileged external actor.

## Recommendation

1. In `_sendEthToDepositPool`, cap the amount passed to the function to the amount actually claimed (tracked separately per request), rather than using `address(this).balance`. This prevents extra ETH from corrupting `ethValueInWithdrawal`.
2. Add a permissioned `sweepETH(address recipient)` function (callable by admin/manager) that forwards `address(this).balance` to the deposit pool or treasury, providing an exit path for ETH that arrives outside the normal claim flow.
3. In `getETHDistributionData()`, consider reading `max(ethValueInWithdrawal, address(lrtConverter).balance)` or adding the converter's raw balance as a separate term so that any untracked ETH is still reflected in TVL.

## Proof of Concept

1. Protocol has 100 stETH in a pending Lido withdrawal; `ethValueInWithdrawal = 100e18`.
2. An external party calls `address(lrtConverter).transfer(10 ether)` (or uses `selfdestruct`). `ethValueInWithdrawal` remains `100e18`; `address(lrtConverter).balance = 10 ETH`. `getETHDistributionData()` returns `ethLyingInConverter = 100e18`, missing the 10 ETH. TVL is understated; rsETH price drops.
3. A depositor observes the lower rsETH price and deposits, receiving excess rsETH at the expense of existing holders.
4. Operator calls `claimStEth(requestId, hint)`. `address(this).balance = 110 ETH`. `_sendEthToDepositPool(110e18)` is called. Since `110e18 > 100e18`, `ethValueInWithdrawal` is set to `0` — even if additional Lido requests for 50 ETH remain outstanding. Those 50 ETH vanish from TVL.
5. If no Lido request exists at step 2, the 10 ETH has no exit path. The operator must call `transferAssetFromDepositPool` (stETH), `unstakeStEth`, wait days for Lido unbonding, then `claimStEth` — a multi-day freeze.

**Foundry test plan:** Deploy `LRTConverter` with a mock `LRTDepositPool` and mock Lido. Assert `ethValueInWithdrawal == 0` and `address(converter).balance == 10 ether` after a direct ETH transfer. Assert `getETHDistributionData()` returns `ethLyingInConverter == 0` (missing the 10 ETH). Then simulate a claim with `address(this).balance = 110 ETH` and assert `ethValueInWithdrawal` is incorrectly zeroed when a second pending request of 50 ETH should still be tracked.
