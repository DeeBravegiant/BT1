### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect Block Fees - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The `is_proposal_init_valid` function validates many fields of the received `ProposalInit` (height, l1_da_mode, l2_gas_price_fri, starknet_version, version_constant_commitment, all four L1 gas prices, fee_proposal_fri) but never checks the `builder` field against the locally configured `builder_address`. A malicious proposer can set `builder` to any address; every honest validator will silently accept it, execute the block with the attacker-controlled `sequencer_address`, and transfer all transaction fees in that block to the attacker.

### Finding Description

**Proposer side** (`crates/apollo_consensus_orchestrator/src/build_proposal.rs`, `initiate_build`): the proposer constructs `ProposalInit` with `builder: args.builder_address`, where `builder_address` comes from `self.config.static_config.builder_address`. [1](#0-0) 

**Validator side** (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, `is_proposal_init_valid`): the function checks height, l1_da_mode, l2_gas_price_fri, starknet_version, version_constant_commitment, all four L1 gas prices, and fee_proposal_fri. The `builder` field is never compared against any locally expected value. [2](#0-1) 

The `builder` field is then forwarded verbatim to `convert_to_sn_api_block_info` as `sequencer_address`: [3](#0-2) 

This `BlockInfo` (with the attacker-supplied `sequencer_address`) is passed to both `batcher.propose_block` and `batcher.validate_block`. Because both the proposer's batcher and every validator's batcher receive the same `builder` value from the wire, they all compute the same block commitment. The `ProposalFinMismatch` guard therefore passes: [4](#0-3) 

The `ProposalInitValidation` struct has no `builder_address` field, so there is no locally-trusted reference value to compare against: [5](#0-4) 

The static config carries the legitimate builder address, but it is only consumed by the proposer, never by the validator: [6](#0-5) 

The production deployment config confirms the legitimate builder address is `0x1176a1bd84444c89232ec27754698e5d2e7e1a7f1539f12027f28b23ec9f3d8`: [7](#0-6) 

### Impact Explanation

`sequencer_address` is the fee-recipient in the blockifier's fee-transfer logic. Every transaction in the block transfers its fee to `block_context.block_info.sequencer_address`. With an attacker-controlled `builder`, all fees in the block are redirected to the attacker's address. Additionally, `sequencer_address` is an input to the partial block hash computation (`PartialBlockHashComponents::new`), so the committed block hash encodes the wrong sequencer, corrupting the on-chain state root for that block.

This matches the allowed impact: **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact"** (Critical) and **"Wrong state … storage value … from blockifier/syscall/execution logic"** (Critical).

### Likelihood Explanation

Any validator that is legitimately selected as the round proposer can execute this attack with zero additional privilege. The proposer only needs to change one field in the `ProposalInit` it broadcasts. The attack succeeds silently every round the malicious validator proposes, because no honest validator checks the `builder` field. The TODO comment in `initiate_build` acknowledges the field is not yet committee-sourced, confirming the missing validation is a known gap rather than an intentional design: [8](#0-7) 

### Recommendation

Add `builder_address: ContractAddress` to `ProposalInitValidation` and populate it from `self.config.static_config.builder_address` at the call site in `validate_proposal` (in `sequencer_consensus_context.rs`). Then add an exact-equality check inside `is_proposal_init_valid`:

```rust
if init_proposed.builder != proposal_init_validation.builder_address {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder_address, init_proposed.builder
        ),
    ));
}
```

### Proof of Concept

1. Malicious validator `V` is selected as proposer for height H.
2. `V` constructs `ProposalInit` with `builder = attacker_address` (any address it controls) instead of the configured `0x1176a1bd84444c89232ec27754698e5d2e7e1a7f1539f12027f28b23ec9f3d8`.
3. `V`'s batcher builds the block with `sequencer_address = attacker_address` → produces commitment `C`.
4. `V` broadcasts `ProposalInit{builder: attacker_address, ...}` followed by transactions and `ProposalFin{proposal_commitment: C}`.
5. Each honest validator calls `is_proposal_init_valid` — the `builder` field is never checked → validation passes.
6. Each honest validator calls `initiate_validation` → `convert_to_sn_api_block_info` sets `sequencer_address = attacker_address` → `batcher.validate_block` executes with the attacker's address.
7. Each honest validator's batcher computes the same commitment `C` → `built_block == received_fin.proposal_commitment` → `ProposalFinMismatch` guard passes.
8. Consensus reaches decision on the block; all transaction fees in block H are transferred to `attacker_address`.
