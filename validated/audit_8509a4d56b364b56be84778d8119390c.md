Audit Report

## Title
Token Depositor Receives Zero rsETH While Tokens Are Permanently Transferred Due to Rounding Truncation - (File: contracts/pools/RSETHPool.sol)

## Summary
In all four L2 pool deposit functions, a user's ERC20 tokens are transferred to the pool before `rsETHAmount` is computed. When the deposit amount is small enough that integer division in `viewSwapRsETHAmountAndFee` truncates `rsETHAmount` to zero, the user's tokens are permanently held by the pool while the user receives nothing. No zero-output guard exists in any of the affected contracts.

## Finding Description
In `RSETHPool.sol`, `RSETHPoolV3.sol`, `RSETHPoolNoWrapper.sol`, and `RSETHPoolV3ExternalBridge.sol`, the token deposit function follows this sequence:

1. `if (amount == 0) revert InvalidAmount();` — only guards against zero input.
2. `IERC20(token).safeTransferFrom(msg.sender, address(this), amount);` — tokens leave the user irrevocably.
3. `(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);` — computes output via integer division.
4. Transfer `rsETHAmount` (possibly 0) to the user.

The critical computation in `viewSwapRsETHAmountAndFee`:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

This is integer division. When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, the result is `0`. There is no subsequent `if (rsETHAmount == 0) revert` guard. The token transfer at step 2 has already occurred and is not reversed. The user's tokens remain in the pool and are eventually bridged to L1 via `moveAssetsForBridging`, making recovery impossible.

The same truncation applies to the ETH deposit path: `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`. Once rsETH has appreciated (`rsETHToETHrate > 1e18`), a 1-wei ETH deposit yields `rsETHAmount = 0`.

Confirmed affected lines (code verified):
- `RSETHPool.sol` L296 (transfer-in), L298 (compute), L302 (transfer-out 0)
- `RSETHPoolV3.sol` L284 (transfer-in), L286 (compute), L290 (`wrsETH.mint(msg.sender, 0)`)
- `RSETHPoolNoWrapper.sol` L262 (transfer-in), L264 (compute), L268 (transfer-out 0)
- `RSETHPoolV3ExternalBridge.sol` L403 (transfer-in), L405 (compute), L409 (`wrsETH.mint(msg.sender, 0)`)

## Impact Explanation
A user who deposits a dust amount of a supported token receives 0 rsETH while permanently losing their deposited tokens to the pool. The tokens are not recoverable by the user; they accumulate in the pool's balance and are bridged to L1. This constitutes a direct, permanent loss of user funds with no recourse. The correct impact classification is **Critical — Direct theft of any user funds**, as the user's tokens are taken without any corresponding rsETH output. The submitter's "Low" classification understates the impact since the user does lose value.

## Likelihood Explanation
Any unprivileged depositor can trigger this by calling `deposit` with a dust amount. No front-running, privileged access, or external compromise is required. For a token with `tokenToETHRate = 1e15` and `rsETHToETHrate = 1.1e18`, the zero-output threshold is any deposit below ~1100 wei of that token. For ETH deposits, any 1-wei deposit when `rsETHToETHrate > 1e18` (normal as rsETH appreciates) yields 0 rsETH. Automated integrations, bots performing test transactions, or users making rounding errors are realistic victims.

## Recommendation
Add a zero-output guard immediately after computing `rsETHAmount` in every deposit function across all four pool contracts, before any state changes or transfers occur:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

This mirrors the existing `if (amount == 0) revert InvalidAmount()` guard and ensures tokens are never taken without a corresponding rsETH output. Ideally, the token transfer should also be moved to after the rsETHAmount computation to follow checks-effects-interactions ordering.

## Proof of Concept
**Setup:**
- `rsETHToETHrate = 1.1e18` (rsETH has appreciated 10% — normal over time)
- Supported token with `tokenToETHRate = 1e15` (token worth 0.001 ETH)
- `feeBps = 0`

**Steps:**
1. User calls `deposit(token, 1000, "ref")`.
2. `RSETHPool.sol` L296: `IERC20(token).safeTransferFrom(msg.sender, address(this), 1000)` — tokens leave user.
3. `viewSwapRsETHAmountAndFee(1000, token)` computes: `rsETHAmount = 1000 * 1e15 / 1.1e18 = 0` (integer truncation).
4. `RSETHPool.sol` L302: `IERC20(address(wrsETH)).safeTransfer(msg.sender, 0)` — user receives nothing.
5. Transaction succeeds without revert. User has permanently lost 1000 wei of the token.

**Foundry fuzz test plan:**
```solidity
function testFuzz_depositZeroRsETH(uint256 amount) public {
    vm.assume(amount > 0 && amount < rsETHToETHrate / tokenToETHRate);
    uint256 balanceBefore = token.balanceOf(user);
    vm.prank(user);
    pool.deposit(address(token), amount, "ref");
    assertEq(wrsETH.balanceOf(user), 0);
    assertEq(token.balanceOf(user), balanceBefore - amount); // tokens lost
}
```
