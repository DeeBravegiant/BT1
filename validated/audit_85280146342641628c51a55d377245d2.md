### Title
Stale Removed-Participant Votes Counted in `CodeHashesVotes::count_votes` During Detached-Promise Window — (`crates/contract/src/tee/tee_state.rs`, `crates/contract/src/lib.rs`)

### Summary

After resharing concludes, `vote_reshared` spawns `clean_tee_status` as a **detached** promise. In the receipt gap before that promise executes, a single new participant can call `vote_code_hash` and cause `CodeHashesVotes::count_votes` to tally stale votes from removed participants alongside the fresh vote, reaching threshold and triggering `whitelist_tee_proposal` without genuine threshold authorization from the new participant set.

### Finding Description

**`vote_reshared` spawns `clean_tee_status` as a detached promise.** [1](#0-0) 

The `.detach()` call means `clean_tee_status` executes in a subsequent receipt, not atomically with the state transition. The protocol state is already `Running` with the new participant set the moment `vote_reshared` returns.

**`clean_tee_status` calls `clean_non_participant_votes`, which is the only mechanism that purges stale votes.** [2](#0-1) [3](#0-2) 

**`CodeHashesVotes::count_votes` counts every entry in `proposal_by_account` matching the hash — it does not filter by current participants.** [4](#0-3) 

**`vote_code_hash` compares the raw count against the threshold with no participant-membership filter.** [5](#0-4) 

**`AuthenticatedParticipantId` is a newtype over a numeric `ParticipantId`.** When removed participants P1 and P2 voted before resharing, their entries were stored under their old numeric IDs (e.g., `ParticipantId(0)` and `ParticipantId(1)`). When the new set {P3, P4, P5} is constructed preserving original IDs (as done in the sandbox resharing helpers), P3 gets `ParticipantId(2)` — a distinct key — so P3's `vote()` call inserts a *third* entry rather than overwriting a stale one. [6](#0-5) 

The developers are aware that stale votes must be purged — there is even a unit test documenting the exact N=5, T=3 scenario — but that test assumes `clean_non_participant_votes` is called *before* any new vote is cast: [7](#0-6) 

The production code does not guarantee this ordering because the cleanup is detached.

### Impact Explanation

A TEE Docker image hash is whitelisted into `AllowedDockerImageHashes` without T votes from the current participant set. This breaks the invariant that only threshold-many *current* participants may authorize a new TEE image. Nodes running an unauthorized image can subsequently pass attestation verification (`add_participant` / `reverify_participants`), undermining the TEE security boundary for the MPC network.

This matches the allowed Medium impact: *"participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants."*

### Likelihood Explanation

The window is one NEAR block (the gap between the `vote_reshared` receipt and the `clean_tee_status` receipt). Any participant in the new set can submit `vote_code_hash` in that block. The precondition — T-1 removed participants having pre-voted for the target hash — can arise either from legitimate prior votes for a hash the new set no longer endorses, or from deliberate pre-voting by colluding removed participants. The attack requires no special privileges beyond being a participant in the new set.

### Recommendation

Replace the detached promise pattern with an inline call to `clean_non_participant_votes` inside `vote_reshared`, immediately after the protocol state transitions to `Running`, before returning. This eliminates the window entirely. The detached cleanup promises for other state (attestations, foreign-chain data, etc.) can remain detached since those paths already filter by current participants at query time.

### Proof of Concept

```
N=5, T=3. Old set: {P1(id=0), P2(id=1), P3(id=2), P4(id=3), P5(id=4)}.

1. P1 calls vote_code_hash(H) → proposal_by_account[AuthId(0)] = H  (1 vote)
2. P2 calls vote_code_hash(H) → proposal_by_account[AuthId(1)] = H  (2 votes)
   (threshold not met; whitelist_tee_proposal not triggered)

3. Resharing: vote_new_parameters({P3,P4,P5}, T=3) → vote_reshared concludes.
   - protocol_state transitions to Running({P3,P4,P5}).
   - clean_tee_status spawned as detached promise (not yet executed).
   - proposal_by_account still contains AuthId(0)→H and AuthId(1)→H.

4. P3 (signer=P3, new Running set) calls vote_code_hash(H):
   - AuthenticatedParticipantId::new(new_participants) → AuthId(2)  [P3's id in new set]
   - proposal_by_account.insert(AuthId(2), H)  [distinct key, no overwrite]
   - count_votes(H) = 3  (AuthId(0), AuthId(1), AuthId(2) all map to H)
   - 3 >= threshold(3) → whitelist_tee_proposal(H) executes.

5. H is now in AllowedDockerImageHashes with only 1 genuine vote from the new set.

6. clean_tee_status executes (next receipt) — too late; H is already whitelisted.
```

    /// Stale CodeHashesVotes entries from removed participants must not count toward
    /// quorum after resharing.
    ///
    /// Scenario (N=5, T=3):
    /// 1. P1 and P2 vote for malicious hash before resharing.
    /// 2. Resharing removes P1 and P2. New set: {P3, P4, P5}.
    /// 3. clean_non_participant_votes removes stale votes.
    /// 4. P3 votes for the same hash — only 1 vote, not 3.
    #[test]
    fn test_clean_non_participant_votes_removes_stale_votes() {
        // Build 5 participants
        let mut all_participants = Participants::new();
        let mut account_ids = Vec::new();
        for i in 0..5 {
            let (account_id, info) = gen_participant(i);
            account_ids.push(account_id.clone());
            all_participants.insert(account_id, info).unwrap();
        }

        let mut tee_state = TeeState::default();

        // P0 and P1 vote for a malicious hash before resharing
        let malicious_hash = NodeImageHash::from([0xAA; 32]);
        for account_id in &account_ids[0..2] {
            let mut ctx = VMContextBuilder::new();
            ctx.signer_account_id(account_id.clone());
            testing_env!(ctx.build());
            let auth_id = AuthenticatedParticipantId::new(&all_participants).unwrap();
            tee_state.votes.vote(malicious_hash, &auth_id);
        }
        assert_eq!(tee_state.votes.proposal_by_account.len(), 2);

        // Resharing removes P0 and P1. New participant set: {P2, P3, P4}.
        let new_participants = all_participants.subset(2..5);

        // Clean non-participants (as done by CLEAN_TEE_STATUS after resharing)
        tee_state.clean_non_participant_votes(&new_participants);

        // Stale votes must be removed
        assert_eq!(tee_state.votes.proposal_by_account.len(), 0);

        // P2 votes for the same malicious hash — should be only 1 vote, not 3
        let p2_account = &account_ids[2];
        let mut ctx = VMContextBuilder::new();
        ctx.signer_account_id(p2_account.clone());
        testing_env!(ctx.build());
        let auth_id = AuthenticatedParticipantId::new(&new_participants).unwrap();
        let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
        assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
    }
