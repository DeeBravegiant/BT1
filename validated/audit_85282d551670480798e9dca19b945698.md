---

### Title
Column Mismatch in `get_derivation_record`: `used` Returned as `hardened`, Causing Wrong Key Derivation — (`chia/wallet/wallet_puzzle_store.py`)

---

### Summary

`get_derivation_record` SELECTs `used` as the 6th column, but `row_to_record` unconditionally interprets `row[5]` as the `hardened` field of `DerivationRecord`. Every other caller of `row_to_record` correctly selects `hardened` as the 6th column. This is a confirmed column mismatch bug.

---

### Finding Description

**The mismatch:**

`get_derivation_record` issues:
```sql
SELECT derivation_index, pubkey, puzzle_hash, wallet_type, wallet_id, used
FROM derivation_paths WHERE derivation_index=? AND wallet_id=? AND hardened=?
``` [1](#0-0) 

So `row[5]` = `used` (0 or 1).

`row_to_record` then does:
```python
bool(row[5])   # ← passed as `hardened` to DerivationRecord
``` [2](#0-1) 

Every other method that calls `row_to_record` correctly selects `hardened` as the 6th column:
- `get_derivation_record_for_puzzle_hash` [3](#0-2) 
- `record_for_pubkey` [4](#0-3) 
- `record_for_puzzle_hash` [5](#0-4) 

Only `get_derivation_record` is wrong.

**Concrete misbehavior:**

| DB state | Caller passes | row[5]=used | Returned hardened |
|---|---|---|---|
| unhardened, used=1 | hardened=False | 1 | **True** (wrong) |
| hardened, used=0 | hardened=True | 0 | **False** (wrong) |

The WHERE clause (`AND hardened=?`) correctly finds the right row, but the returned `DerivationRecord.hardened` is corrupted.

**`get_current_derivation_record_for_wallet` is a direct trigger path:**

It queries for `MAX(derivation_index) WHERE used=1 AND hardened=0`, then calls `get_derivation_record(index, wallet_id, False)`. [6](#0-5) 

The returned record will always have `hardened=True` (because `used=1`), even though the record is unhardened.

---

### Impact Explanation

`wallet_state_manager.py` uses `record.hardened` (confirmed by grep: 1 match on `.hardened`, 10 matches on `master_sk_to_wallet_sk`) to branch between `master_sk_to_wallet_sk` (hardened path) and `master_sk_to_wallet_sk_unhardened` (unhardened path). When `record.hardened` is wrong, the wrong child private key is derived — one that does not correspond to the stored public key. Any spend bundle signed with this key will have an invalid aggregated signature and be rejected by the full node.

This causes a **permanent inability to spend coins** from any unhardened address that has been marked used, through the normal wallet signing path. The coins are not lost (the correct key can be derived manually), but the wallet software is broken for those addresses.

This meets the **High** impact criterion: *"Permanent or long-lived inability for honest wallets to process valid spend bundles under normal network assumptions."*

It does **not** meet Critical (no unauthorized spend, no key exfiltration).

---

### Likelihood Explanation

This is triggered by **normal wallet operation**: any coin received at an unhardened address and spent (or observed as spent on-chain) causes `set_used_up_to` to mark that index as used. After that, every call to `get_derivation_record` for that index returns a corrupted record. This is not attacker-gated — it happens automatically during routine wallet sync. [7](#0-6) 

---

### Recommendation

Fix `get_derivation_record` to select `hardened` instead of `used` as the 6th column:

```python
"SELECT derivation_index, pubkey, puzzle_hash, wallet_type, wallet_id, hardened FROM derivation_paths "
"WHERE derivation_index=? AND wallet_id=? AND hardened=?"
```

# Insert unhardened record at index 5, wallet_id=1

# Mark it used (simulates a coin spend)

# Retrieve it

assert record.hardened == False  # FAILS: returns True because used=1 is read as hardened
