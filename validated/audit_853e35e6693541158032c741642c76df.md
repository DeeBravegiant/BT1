### Title
`respond_verify_foreign_tx` Does Not Bind `payload_hash` to the Request, Enabling Cross-Request Signature Replay by a Single Byzantine Participant - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` contract method verifies that the submitted signature is valid over the caller-supplied `response.payload_hash`, but never verifies that `response.payload_hash` is the correct hash for the given `request`. A single Byzantine attested participant can replay a threshold signature that was legitimately produced for a prior request and submit it as the response to a different pending request, delivering a forged `VerifyForeignTransactionResponse` to the waiting caller.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–754), the contract performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← caller-supplied

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,                                            // root public key, no tweak
)
.is_ok()
``` [1](#0-0) 

The check only confirms that `signature` is a valid ECDSA signature over `payload_hash` under the root key. It does **not** confirm that `payload_hash` encodes the correct `ForeignTxSignPayload { request: request.request, values: <observed_values> }` for the pending request.

Compare this with the regular `respond` method, where the payload is taken from the on-chain request state, not from the caller:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
// payload_hash comes from the stored SignatureRequest, not from the response
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
``` [2](#0-1) 

The `ForeignTxSignPayload` that nodes sign encodes both the original request and the observed extracted values:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
// msg_hash = SHA-256(borsh(ForeignTxSignPayload))
``` [3](#0-2) 

Because the contract cannot recompute this hash (it does not know the extracted `values`), it accepts any `payload_hash` that carries a valid root-key signature — including one produced for a completely different request.

---

### Impact Explanation

A Byzantine attested participant can:

1. Observe a legitimately produced `(payload_hash_A, signature_A)` for request A (publicly visible on-chain after `respond_verify_foreign_tx` is called for A, or from any prior successful response).
2. Wait for a different request B to become pending.
3. Call `respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: signature_A })`.
4. The contract accepts the call (signature is valid over `payload_hash_A` under the root key; request B exists in the pending map).
5. `resolve_yields_for` drains the entire fan-out queue for request B, delivering `{ payload_hash_A, signature_A }` to every caller waiting on request B.

Consequences:
- **Forged attestation**: Callers whose NEAR contracts do not independently verify `payload_hash` against their expected values (using `ForeignChainSignatureVerifier::verify_signature`) will accept the forged response as proof that their transaction was verified. For bridge contracts this enables invalid fund release.
- **Permanent DoS on the legitimate response**: Once the queue is drained, the legitimate MPC response for request B can never be delivered; the pending entry is gone.

The SDK does provide `ForeignChainSignatureVerifier::verify_signature` which checks `expected_payload_hash == response.payload_hash`, but this is an off-chain SDK helper, not enforced by the contract. [4](#0-3) 

---

### Likelihood Explanation

- Requires a **single** Byzantine attested participant (strictly below the signing threshold). This is explicitly within the allowed attacker model.
- The attacker needs a previously issued `(payload_hash, signature)` pair. All `respond_verify_foreign_tx` calls are on-chain transactions; their arguments are permanently public. Any past successful response is reusable.
- No special timing, no flash loans, no network-level access required. The attack is a single on-chain transaction.
- The attack is profitable whenever a pending request B exists whose caller does not verify the payload hash — a realistic assumption for early bridge integrations.

---

### Recommendation

The contract must bind the response's `payload_hash` to the request. Since the contract cannot recompute the full hash (it does not know the extracted `values`), the minimum viable fix is to verify that the `payload_hash` is a hash of a `ForeignTxSignPayload` whose embedded `request` field matches `request.request`. This requires the responding node to also supply the `values` so the contract can recompute and compare:

```rust
// Responding node supplies values alongside the signature
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),   // new field
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, adopt the same pattern as `respond`: store the expected payload hash in the pending-request map at submission time (impossible here because values are unknown until nodes query the chain) or require a quorum of nodes to submit matching `(payload_hash, values)` before the response is accepted.

### Proof of Concept

```
1. MPC network processes request_A (Bitcoin tx X, 2 confirmations).
   Nodes sign payload_hash_A = SHA-256(borsh({ request: request_A, values: [block_hash_Y] })).
   Attested participant P1 calls respond_verify_foreign_tx(request_A, { payload_hash_A, sig_A }).
   → sig_A is now publicly visible on-chain.

2. User submits verify_foreign_transaction(request_B) for Bitcoin tx Z.
   request_B is now pending.

3. Byzantine participant P1 calls:
   respond_verify_foreign_tx(request_B, { payload_hash: payload_hash_A, signature: sig_A })

4. Contract checks:
   - P1 is an attested participant ✓
   - verify_ecdsa_signature(sig_A, payload_hash_A, root_key) → valid ✓
   - request_B exists in pending map ✓
   → resolve_yields_for drains request_B queue with { payload_hash_A, sig_A }

5. User's NEAR contract receives VerifyForeignTransactionResponse {
       payload_hash: payload_hash_A,   // encodes tx X, not tx Z
       signature: sig_A,
   }
   If the user's contract does not call ForeignChainSignatureVerifier::verify_signature,
   it accepts this as proof that tx Z was verified → invalid bridge execution.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
