### Title
`EvmFinality::Latest` Accepted Without Restriction Enables Post-Reorg Double-Spend via Stale MPC Signature — (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The MPC system accepts `EvmFinality::Latest` for EVM chain transaction verification without any restriction. Because `Latest` carries no finality guarantee, a block reorg can invalidate the verified transaction after the MPC threshold signature is already issued. When the signed payload contains no block-anchoring extracted value (i.e., the `extractors` list is empty), the issued signature remains cryptographically valid regardless of any subsequent reorg, enabling a double-spend against any bridge contract that trusts it.

---

### Finding Description

**Entry point — contract accepts any finality level without restriction**

`verify_foreign_transaction` in `crates/contract/src/lib.rs` performs only three checks before enqueuing the request: domain purpose is `ForeignTx`, the chain is in the supported set, and the deposit is sufficient. [1](#0-0) 

There is no validation of the `finality` field inside `EvmRpcRequest`. `EvmFinality::Latest` is a first-class, ABI-exposed variant: [2](#0-1) 

**Node-side — `Latest` is mapped and accepted without restriction**

Inside `EvmInspector::verify_finality_level`, `EthereumFinality::Latest` is mapped directly to `FinalityTag::Latest` and the check only asserts that the "latest" head block number is ≥ the receipt's block number: [3](#0-2) 

`verify_block_is_canonical` then confirms the block hash matches the canonical chain **at the instant of the RPC call**: [4](#0-3) 

Both checks are point-in-time. Neither provides any guarantee that the block remains canonical after the signature is issued.

**Signed payload — no block anchor when extractors list is empty**

The signed payload is:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,  // includes tx_id + finality: Latest
    pub values: Vec<ExtractedValue>,      // empty when extractors: []
}
``` [5](#0-4) 

The contract does not require any extractor to be present. When `extractors: []`, `values` is `[]`, and the resulting `msg_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request, values: [] }))` is identical before and after a reorg — the signature is permanently valid. [6](#0-5) 

Even when the `Log` extractor is used, the extracted `Log` struct contains `block_hash` and `block_number` fields that change after a reorg, but the contract imposes no requirement that consuming bridge contracts verify these fields against the current canonical chain. [7](#0-6) 

---

### Impact Explanation

An attacker who obtains a valid MPC threshold signature over `(tx_id, finality=Latest, values=[])` holds an irrevocable attestation that the transaction existed on the EVM chain at some point. If the block is subsequently reorged away:

- The EVM deposit is reversed — the attacker recovers the EVM-side funds.
- The NEAR-side bridge contract, which trusts the MPC signature, has already released (or will release) the corresponding NEAR-side tokens.
- The attacker holds assets on both sides simultaneously.

This is a direct, permanent theft of funds controlled by the bridge flow that relies on the MPC foreign-chain verification. It matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions**.

---

### Likelihood Explanation

- `EvmFinality::Latest` is a fully supported, ABI-exposed variant requiring zero privileged access to invoke.
- Any unprivileged caller can submit `verify_foreign_transaction` with `finality: Latest` and `extractors: []`.
- Block reorgs of 1–2 blocks occur naturally on EVM-compatible chains (BNB, Polygon, Abstract, HyperEVM, Base, Arbitrum are all in the supported set); an attacker with modest hash power or who can exploit a network partition can engineer a reorg.
- The window between MPC signing and bridge fund release is finite but non-zero, giving the attacker time to trigger the reorg.
- No threshold collusion, TEE attack, or operator privilege is required.

---

### Recommendation

1. **Reject `EvmFinality::Latest` at the contract level** inside `verify_foreign_transaction` for any domain with purpose `ForeignTx`. Only `Safe` and `Finalized` provide meaningful reorg resistance on Ethereum-family chains.
2. **Alternatively, mandate the `BlockHash` extractor** whenever `EvmFinality::Latest` or `EvmFinality::Safe` is requested, so the signed payload is always anchored to a specific block hash that consuming contracts can re-verify against the current canonical chain.
3. **Apply the same restriction to `SolanaFinality::Processed`**, which carries an analogous pre-finality risk.
4. **Apply a minimum `BlockConfirmations` floor for Bitcoin** (e.g., ≥ 6) at the contract level, since `BlockConfirmations(1)` is currently accepted.

### Proof of Concept

**Step 1 — Attacker deposits tokens to an EVM bridge contract and submits:**

```json
{
  "request": {
    "Bnb": {
      "tx_id": "0xDEAD...BEEF",
      "extractors": [],
      "finality": "Latest"
    }
  },
  "domain_id": 1,
  "payload_version": 1
}
```

**Step 2 — MPC nodes execute (node-side `sign.rs` → `EvmInspector::extract`):**

- `verify_finality_level`: latest BNB head ≥ receipt block → passes.
- `verify_block_is_canonical`: block hash matches at query time → passes.
- `values = []` (no extractors).
- Nodes sign `SHA-256(borsh(ForeignTxSignPayloadV1 { request: Bnb(tx_id, [], Latest), values: [] }))`. [8](#0-7) 

**Step 3 — Attacker receives the threshold signature and submits it to the NEAR bridge contract to claim tokens.**

**Step 4 — Attacker causes a 1-block reorg on BNB** (or waits for a natural one), removing the deposit transaction. The EVM deposit is reversed; the attacker reclaims EVM-side funds.

**Step 5 — The MPC signature remains valid** because `msg_hash` is computed solely from `(tx_id, finality=Latest, values=[])`, none of which changed. The bridge has already released NEAR-side tokens. Double-spend complete.

            .await?;

            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
    }
