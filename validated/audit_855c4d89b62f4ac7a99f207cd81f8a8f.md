### Title
Proposer Deposits Permanently Lost When `do_update` Clears All Pending Update Entries - (File: `crates/contract/src/update.rs`)

### Summary

The `propose_update` function in `crates/contract/src/lib.rs` requires participants to attach a NEAR deposit proportional to the storage cost of their proposed update. When any proposal reaches threshold votes and `do_update` is called, it clears **all** pending entries — including proposals from other participants — without refunding their deposits. The `bytes_used` field stored in each `UpdateEntry` was clearly intended to enable this refund, but the refund logic was never implemented. Deposits paid by non-winning proposers are permanently locked in the contract.

### Finding Description

`propose_update` charges each proposer a deposit calculated as:

```rust
// crates/contract/src/lib.rs
let required = ProposedUpdates::required_deposit(&update);
```

where `required_deposit` is:

```rust
// crates/contract/src/update.rs
fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
```

The deposit is consumed by the contract and only the **excess** above `required` is refunded to the proposer at call time. The `required` portion is retained by the contract to cover storage staking for the entry.

When `vote_update` reaches threshold and calls `do_update`:

```rust
// crates/contract/src/update.rs
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;

    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
```

All non-winning entries are dropped via `self.entries.clear()`. The storage they occupied is freed (returning NEAR to the contract's own balance), but the original depositors are never refunded. The `bytes_used` field stored in each `UpdateEntry` — the exact value needed to compute the refund — is silently discarded.

There is no `remove_proposal` method, and `remove_update_vote` only removes the vote record, not the entry or its deposit. Once a participant calls `propose_update`, their deposit is irrecoverable unless their own proposal wins.

### Impact Explanation

Every participant who proposes an update that is not the one ultimately executed permanently loses their deposit. For a contract binary update, `bytes_used` includes the full WASM blob plus overhead for 128 vote slots:

```rust
fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;
    match update {
        Update::Contract(code) => { bytes_used += code.len() as u128; }
        ...
    }
    bytes_used
}
```

At NEAR's storage byte cost (~10 yoctoNEAR/byte), a 1 MB contract binary requires a deposit of approximately 10 NEAR. In a scenario where multiple participants each propose a different contract version (a realistic governance disagreement), all but one proposer permanently lose their full deposit when the winning proposal executes. This breaks the production accounting invariant that storage-staking deposits are returned when the staked storage is freed.

### Likelihood Explanation

This triggers on every successful `vote_update` execution. In any governance round where more than one participant proposes an update — which is the normal case when participants disagree on which version to deploy — all non-winning proposers lose their deposits. The `bytes_used` field being stored but never used for refunds confirms this is an unimplemented refund path, not an intentional design choice.

### Recommendation

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue a `Promise::new(proposer_account_id).transfer(required_deposit(entry.bytes_used))` for each. The proposer's `AccountId` must be tracked in `UpdateEntry` (it is not currently stored) to enable this refund. Alternatively, store the proposer and their required deposit directly in `UpdateEntry` at proposal time.

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. Contract stores `UpdateEntry { update: Contract([...]), bytes_used: ~1_000_000 }` and retains the 10 NEAR.
2. Participant B calls `propose_update` with a different 1 MB binary, attaching ~10 NEAR. Same storage and deposit.
3. Threshold participants call `vote_update` for Participant A's proposal. `do_update` is triggered.
4. `do_update` removes A's entry, then calls `self.entries.clear()` — dropping B's entry with no refund.
5. Participant B's ~10 NEAR is permanently locked in the contract. No `remove_proposal` or refund path exists.
