Audit Report

## Title
`assetsCommitted` Decremented by Original Amount While Only `payoutAmount` Is Redeemed, Inflating `getAvailableAssetAmount` and Enabling Over-Commitment of Withdrawal Requests - (File: contracts/LRTWithdrawalManager.sol)

## Summary
In `_unlockWithdrawalRequests`, `assetsCommitted[asset]` is decremented by `request.expectedAssetAmount` (the original committed amount) at line 802, but the vault only redeems `payoutAmount` (the actual, potentially smaller payout) at line 307. This mismatch inflates `getAvailableAssetAmount` by `expectedAssetAmount − payoutAmount` per unlocked request. Subsequent callers of `initiateWithdrawal` can commit against this phantom liquidity, locking their rsETH in the contract against assets that do not exist in the vault, resulting in a temporary freeze of user funds.

## Finding Description
`_calculatePayoutAmount` (line 834) returns `min(request.expectedAssetAmount, currentReturn)` where `currentReturn = (rsETHUnstaked * rsETHPrice) / assetPrice`. When the rsETH/asset exchange rate moves unfavourably between a user's `initiateWithdrawal` call and the operator's `unlockQueue` call, `payoutAmount < request.expectedAssetAmount`.

Inside the unlock loop (line 802), the decrement uses the original amount:
```solidity
assetsCommitted[asset] -= request.expectedAssetAmount;   // line 802 — original (larger) amount
request.expectedAssetAmount = payoutAmount;              // line 804 — updated to actual (smaller) amount
```

After the loop, `unlockQueue` redeems only the sum of `payoutAmount` values from the vault (line 307):
```solidity
unstakingVault.redeem(asset, assetAmountUnlocked);
```

`getAvailableAssetAmount` (lines 599–602) computes:
```
availableAssetAmount = getTotalAssetDeposits(asset) - assetsCommitted[asset]
```

`getTotalAssetDeposits` (lines 385–396) sums `assetLyingUnstakingVault` (vault balance) plus other protocol locations, but does **not** include assets already transferred into `LRTWithdrawalManager` for pending claims. After an unlock cycle:
- Vault balance decreases by `payoutAmount`
- `assetsCommitted` decreases by `expectedAssetAmount`
- Net change to `getAvailableAssetAmount` = `+(expectedAssetAmount − payoutAmount)` — phantom liquidity

Any unprivileged user can then call `initiateWithdrawal` (lines 168–173), which passes the guard `expectedAssetAmount > getAvailableAssetAmount(asset)` using the inflated value, transfers their rsETH into the contract, and increments `assetsCommitted` against non-existent vault assets. When the operator later calls `unlockQueue`, the vault is insufficient to cover the phantom-backed request, the loop exits early at line 800 (`if (availableAssetAmount < payoutAmount) break`), and the request remains permanently locked until an operator manually replenishes the vault.

## Impact Explanation
This constitutes **temporary freezing of funds** (Medium severity). A normal user's rsETH is transferred into `LRTWithdrawalManager` and cannot be retrieved until an operator manually tops up `LRTUnstakingVault`. The phantom amount accumulates with each unlock cycle where price drift occurs, and no privileged access is required to exploit it — any user calling `initiateWithdrawal` after an unlock cycle with price drift is affected.

## Likelihood Explanation
The condition `payoutAmount < expectedAssetAmount` is triggered by any unfavourable rsETH/asset price movement between `initiateWithdrawal` and `unlockQueue`. The withdrawal delay is configurable up to 16 days, during which LST/rsETH prices fluctuate continuously. This is a normal operating condition, not an edge case. No attacker capability beyond calling a public function is required.

## Recommendation
Decrement `assetsCommitted` by `payoutAmount` rather than `request.expectedAssetAmount` so the accounting reflects the actual vault outflow:

```solidity
// line 802 — change to:
assetsCommitted[asset] -= payoutAmount;
```

This ensures `getAvailableAssetAmount` accurately reflects only assets that remain in the vault and are not yet committed to pending requests.

## Proof of Concept
1. Vault holds 100 ETH. rsETH price = 1.0 ETH/rsETH.
2. User A calls `initiateWithdrawal(ETH, 10 rsETH)` → `expectedAssetAmount = 10`, `assetsCommitted = 10`, `getAvailableAssetAmount = 90`.
3. rsETH price drops to 0.8 ETH/rsETH.
4. Operator calls `unlockQueue(ETH, ...)`:
   - `payoutAmount = min(10, 8) = 8`
   - `assetsCommitted -= 10` → `assetsCommitted = 0` (line 802)
   - Vault redeems 8 ETH → vault balance = 92 ETH (line 307)
   - `getAvailableAssetAmount = 92 − 0 = 92` ← **phantom +2 ETH**
5. User B calls `initiateWithdrawal(ETH, 92 rsETH)` → `expectedAssetAmount = 92`, guard passes (92 ≤ 92), `assetsCommitted = 92`, User B's 92 rsETH locked.
6. Operator calls `unlockQueue(ETH, ...)`: vault has 92 ETH but any further price drift causes `payoutAmount > 92`, loop breaks at line 800, User B's request remains locked.

A Foundry fork test can reproduce this by: (a) mocking `lrtOracle.rsETHPrice()` to return a lower value between the two `unlockQueue` calls, (b) asserting that `getAvailableAssetAmount` exceeds actual vault balance after step 4, and (c) asserting that User B's request is not unlocked in step 6.
