Audit Report

## Title
Missing Slippage Protection in `deposit()` Functions Allows Users to Receive Fewer rsETH Tokens Than Previewed - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPool.sol, contracts/pools/RSETHPoolNoWrapper.sol, contracts/pools/RSETHPoolV2ExternalBridge.sol)

## Summary
All `deposit()` functions across the four pool contracts compute the rsETH output amount from a live oracle rate at execution time, but none accept a caller-supplied `minAmountOut` parameter. Because the oracle rate is updated regularly by an off-chain operator, a user's transaction can execute at a materially different rate than the one returned by `viewSwapRsETHAmountAndFee` at preview time, causing the user to receive fewer rsETH tokens than the contract's own preview function indicated.

## Finding Description
Every pool contract exposes a public `viewSwapRsETHAmountAndFee` preview function that users naturally call before submitting a deposit. At execution time, each `deposit()` overload recomputes the output using `getRate()`, which reads the live oracle value:

- `RSETHPoolV3ExternalBridge.sol` `deposit(string)` (L366–384) and `deposit(address,uint256,string)` (L390–412): calls `viewSwapRsETHAmountAndFee` → `getRate()` → `IOracle(rsETHOracle).getRate()`, then mints `rsETHAmount` with no floor check.
- `RSETHPool.sol` `deposit(string)` (L265–278) and `deposit(address,uint256,string)` (L284–305): same pattern, transfers pre-held wrsETH.
- `RSETHPoolNoWrapper.sol` `deposit(string)` (L231–244) and `deposit(address,uint256,string)` (L250–271): same pattern, transfers rsETH directly.
- `RSETHPoolV2ExternalBridge.sol` `deposit(string)` (L289–301): same pattern, mints wrsETH.

The oracle rate is stored in `InterimRSETHOracle` (manually set by `MANAGER_ROLE`) or received via LayerZero cross-chain messages in `CrossChainRateReceiver`. Both update the rate at any time without any on-chain coordination with pending user transactions. Since `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`, any increase in `rsETHToETHrate` between preview and execution reduces the tokens minted to the user. No existing guard (reentrancy lock, pause, daily mint cap) prevents this outcome.

## Impact Explanation
The contract provides `viewSwapRsETHAmountAndFee` as an explicit preview mechanism, but does not enforce the previewed output at execution time. A depositor receives fewer rsETH tokens than the contract's own interface indicated they would receive. The user's ETH or token input is fully consumed; they receive some rsETH but less than promised. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The rsETH/ETH rate increases monotonically as staking rewards accrue on L1, and the oracle is refreshed periodically (every few hours or on significant rate changes). On L2 networks with sequencer-controlled ordering, any deposit transaction can be delayed relative to an oracle update. No attacker is required; the condition arises from normal protocol operation. Any unprivileged depositor is exposed on every transaction, and the exposure grows with deposit size and oracle update frequency.

## Recommendation
Add a `minRsETHAmountOut` parameter to all `deposit()` overloads and revert if the computed amount falls below it:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountOut)
    external payable nonReentrant whenNotPaused
    limitDailyMint(msg.value, ETH_IDENTIFIER)
{
    uint256 amount = msg.value;
    if (amount == 0) revert InvalidAmount();

    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountOut) revert InsufficientOutput();

    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same change to all token-deposit overloads and to `RSETHPool.sol`, `RSETHPoolNoWrapper.sol`, and `RSETHPoolV2ExternalBridge.sol`.

## Proof of Concept
1. Deploy a local fork with `InterimRSETHOracle` set to `rate = 1.05e18`.
2. Call `viewSwapRsETHAmountAndFee(1 ether)` — observe `rsETHAmount ≈ 0.952e18`.
3. Submit `deposit{value: 1 ether}("ref")` but do not mine it yet.
4. Call `InterimRSETHOracle.setRate(1.10e18)` from the `MANAGER_ROLE` account (normal operational update).
5. Mine the deposit transaction — `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909e18`.
6. Assert `rsETHAmount < 0.952e18`: the user received ~4.5% fewer tokens than previewed, with no revert and no recourse.

A Foundry fuzz test can parameterize the rate delta and deposit size to demonstrate the magnitude of the shortfall across realistic oracle update ranges.
