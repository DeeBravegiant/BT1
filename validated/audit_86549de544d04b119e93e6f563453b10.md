Audit Report

## Title
Unrestricted `FeeReceiver.sendFunds()` Enables Unprivileged Fee Extraction on Donated ETH, Diverting rsETH Holder Yield to Treasury — (`contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access-control modifier, allowing any caller to push arbitrary ETH into `LRTDepositPool`. Because `LRTOracle._updateRsETHPrice()` treats the raw `address(this).balance` of the deposit pool as TVL, any ETH increase is interpreted as staking yield and triggers protocol fee rsETH minting to `PROTOCOL_TREASURY`. An unprivileged attacker can donate ETH, call `sendFunds()`, and immediately call the public `updateRSETHPrice()` to force fee extraction on their donation, diverting a portion of the yield that should have accrued to rsETH holders to the treasury instead.

## Finding Description

**Root cause — no guard on `sendFunds()`:** [1](#0-0) 
`sendFunds()` is declared `external` with no role modifier. Any EOA or contract can call it at any time, forwarding the full ETH balance of `FeeReceiver` to the deposit pool.

**`receiveFromRewardReceiver()` is equally unrestricted:** [2](#0-1) 
No access control; the call from `sendFunds()` succeeds unconditionally, crediting the deposit pool's raw ETH balance.

**ETH TVL is the raw `address(this).balance`:** [3](#0-2) 
`getETHDistributionData()` returns `address(this).balance` directly. Donated ETH is indistinguishable from genuine staking rewards.

**`updateRSETHPrice()` is public:** [4](#0-3) 
Only gated by `whenNotPaused`; any unprivileged caller can invoke it.

**Fee is minted on any TVL increase, regardless of source:** [5](#0-4) 
`rewardAmount = totalETHInProtocol - previousTVL` treats the donated ETH as yield. The fee is computed and rsETH is minted to `PROTOCOL_TREASURY`: [6](#0-5) 

**`pricePercentageLimit` bypass:** The threshold check only fires when `pricePercentageLimit > 0` and the price increase exceeds it. [7](#0-6) 
An attacker donating small amounts relative to TVL keeps the price increase within any configured threshold, or the check is entirely absent when `pricePercentageLimit == 0`.

**`maxFeeMintAmountPerDay` prerequisite:** The daily cap check at line 205 reverts if `maxFeeMintAmountPerDay == 0` and `feeAmount > 0`, so the attack requires `maxFeeMintAmountPerDay` to be set to a non-zero value by the manager — which is the expected live-deployment state for fee minting to function at all. [8](#0-7) 

**Exploit path:**
1. Attacker sends ETH to `FeeReceiver` (via `receive()`).
2. Attacker calls `FeeReceiver.sendFunds()` — no access control, ETH is pushed to `LRTDepositPool`.
3. Attacker calls `LRTOracle.updateRSETHPrice()` — no access control beyond `whenNotPaused`.
4. Oracle computes `rewardAmount = newBalance - previousTVL`, calculates `protocolFeeInETH`, and mints rsETH to treasury.
5. rsETH holders receive a smaller price appreciation than they would have without the fee extraction.

## Impact Explanation
**High — Theft of unclaimed yield.** rsETH holders hold an expectation that ETH entering the protocol accrues to them at the prevailing rsETH price. By forcing fee extraction on a donation, an unprivileged attacker diverts `D × protocolFeeInBPS / 10_000` ETH-equivalent yield from rsETH holders to the treasury. The attacker loses the donated ETH, but the harm to rsETH holders is concrete and repeatable: each call extracts yield that would otherwise have been distributed pro-rata to all rsETH holders. The `maxFeeMintAmountPerDay` cap limits per-call damage but does not prevent the attack; it can be repeated each day.

## Likelihood Explanation
No privileged role is required. The only preconditions are that the oracle is unpaused (normal operating state) and `maxFeeMintAmountPerDay > 0` (required for the fee mechanism to function). The `pricePercentageLimit` guard is bypassable by sizing donations to stay within the threshold, or is absent when set to zero. The attack is repeatable across days up to the daily cap.

## Recommendation
1. Add `onlyRole(LRTConstants.MANAGER)` to `FeeReceiver.sendFunds()`.
2. Optionally restrict `receiveFromRewardReceiver()` to the registered `FeeReceiver` address (e.g., `require(msg.sender == lrtConfig.getContract(LRTConstants.LRT_FEE_RECEIVER))`).
3. Consider snapshotting the deposit pool's ETH balance at the time of the last price update and computing yield only against that snapshot, rather than the live raw balance, to prevent any unsanctioned ETH injection from being treated as yield.

## Proof of Concept
```solidity
// Foundry test (local fork or mock setup)
// Preconditions: protocol has 100 ETH TVL, rsETH supply = 100e18, price = 1e18
// protocolFeeInBPS = 1000 (10%), pricePercentageLimit = 0 (disabled)
// maxFeeMintAmountPerDay = 10e18 (non-zero, as required for fee minting to be active)

function testDonationFeeExploit() public {
    address attacker = makeAddr("attacker");
    vm.deal(attacker, 1 ether);

    uint256 treasuryBalanceBefore = rsETH.balanceOf(treasury);

    vm.startPrank(attacker);
    // Step 1: donate 1 ETH to FeeReceiver — no access control on receive()
    (bool ok,) = address(feeReceiver).call{value: 1 ether}("");
    require(ok);

    // Step 2: push ETH into deposit pool — sendFunds() has no access control
    feeReceiver.sendFunds();

    // Step 3: trigger fee minting — updateRSETHPrice() is public
    lrtOracle.updateRSETHPrice();
    vm.stopPrank();

    uint256 treasuryBalanceAfter = rsETH.balanceOf(treasury);

    // Treasury received ~0.1 ETH worth of rsETH minted on the 1 ETH donation
    // rsETH holders lost ~0.1 ETH worth of yield to the treasury
    assertGt(treasuryBalanceAfter, treasuryBalanceBefore, "fee minted on donated ETH");
}
```
