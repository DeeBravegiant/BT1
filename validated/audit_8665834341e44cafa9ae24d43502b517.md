### Title
`EpochManager::get_validator_info` Returns Hardcoded Zero/Empty Values for Per-Shard Chunk Stats When Queried by `EpochId` — (File: `chain/epoch-manager/src/lib.rs`)

---

### Summary

`EpochManager::get_validator_info` is the backend for the public `validators` JSON-RPC endpoint. When called with a `ValidatorInfoIdentifier::EpochId` (the path used for any completed epoch), it unconditionally returns hardcoded zeros for `num_produced_chunks_per_shard` / `num_expected_chunks_per_shard` and empty arrays for `shards_endorsed` / `num_produced_endorsements_per_shard` / `num_expected_endorsements_per_shard`, regardless of actual on-chain validator activity. The `ValidatorInfoIdentifier::BlockHash` path (live/current epoch) correctly computes all of these from the aggregator. Any unprivileged caller of the `validators` RPC endpoint who supplies an `epoch_id` receives a structurally valid but factually incorrect response.

---

### Finding Description

`EpochManager::get_validator_info` branches on the identifier type:

**`ValidatorInfoIdentifier::EpochId` path** (completed epoch, lines 1421–1490):

```rust
// TODO: Compute the set of shards validated.
let shards_endorsed = vec![];
...
// TODO: Maybe fill in the per shard info about the chunk produced for requests coming from RPC.
num_produced_chunks_per_shard: vec![0; shards_produced.len()],
num_expected_chunks_per_shard: vec![0; shards_produced.len()],
...
num_produced_endorsements_per_shard: vec![0; shards_endorsed.len()], // always []
num_expected_endorsements_per_shard: vec![0; shards_endorsed.len()], // always []
``` [1](#0-0) 

The stored `EpochSummary` (retrieved at line 1434) contains the full `validator_block_chunk_stats` map with correct aggregate chunk production counts, but the per-shard breakdown is never populated from it. The code acknowledges this with two `TODO` comments.

**`ValidatorInfoIdentifier::BlockHash` path** (current/live epoch, lines 1491–1597):

This path iterates `aggregator.shard_tracker`, builds `chunks_stats_by_shard`, and correctly fills `num_produced_chunks_per_shard`, `num_expected_chunks_per_shard`, `shards_endorsed`, and the per-shard endorsement arrays. [2](#0-1) 

The two paths are structurally identical in intent but diverge in correctness: the `EpochId` path silently returns zeros/empty arrays for five fields that the `BlockHash` path computes correctly.

---

### Impact Explanation

The corrupted RPC result is the `CurrentEpochValidatorInfo` struct returned by the public `validators` JSON-RPC method when the caller supplies `{"epoch_id": "..."}`. The five affected fields — `num_produced_chunks_per_shard`, `num_expected_chunks_per_shard`, `shards_endorsed`, `num_produced_endorsements_per_shard`, `num_expected_endorsements_per_shard` — are always zero or empty for every validator in every completed epoch, regardless of actual on-chain activity.

Applications and monitoring tools that consume these fields to evaluate per-shard validator performance (e.g., staking dashboards, slashing monitors, delegation advisors) receive structurally valid but factually incorrect data. The aggregate totals (`num_produced_chunks`, `num_expected_chunks`, `num_produced_endorsements`, `num_expected_endorsements`) are correct, so the discrepancy is not immediately obvious to callers. [3](#0-2) 

---

### Likelihood Explanation

Likelihood is high. The trigger requires no special privilege: any client can call the `validators` JSON-RPC endpoint with an `epoch_id` parameter. The `EpochId` path is the standard way to query historical epoch data. The `BlockHash` path is only usable for the current (not-yet-finalized) epoch, so all historical queries necessarily hit the broken path. [4](#0-3) 

---

### Recommendation

Populate the per-shard fields in the `EpochId` branch using the stored `EpochSummary`. The `epoch_summary.validator_block_chunk_stats` map already contains per-shard chunk production data (aggregated during epoch finalization). The `EpochId` branch should iterate the shard layout and the chunk-producers settlement (already done for `validator_to_shard`) to reconstruct `chunks_stats_by_shard` from the summary, then fill the five fields the same way the `BlockHash` branch does. [5](#0-4) 

---

### Proof of Concept

1. Let the chain run for at least one full epoch so that a completed `epoch_id` exists.
2. Call the `validators` RPC endpoint:
   ```json
   {
     "jsonrpc": "2.0",
     "id": "dontcare",
     "method": "validators",
     "params": {"epoch_id": "<completed_epoch_id>"}
   }
   ```
3. Inspect any `current_validators[i]` entry for a validator known to have produced chunks in that epoch.
4. Observe: `num_produced_chunks_per_shard` is `[0, 0, ...]`, `num_expected_chunks_per_shard` is `[0, 0, ...]`, `shards_endorsed` is `[]`, `num_produced_endorsements_per_shard` is `[]`, `num_expected_endorsements_per_shard` is `[]` — despite `num_produced_chunks` and `num_expected_chunks` showing non-zero aggregate values.
5. Repeat the same call with `{"block_id": "<last_block_hash_of_that_epoch>"}` (the `BlockHash` path) and observe that all five fields are correctly populated. [6](#0-5)
