Audit Report

## Title
ETH Deposit Limit Completely Unenforced Due to Missing `amount` in Limit Check — (`contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch comparison, making the ETH deposit cap entirely ineffective for any single deposit that does not find `totalAssetDeposits` already over the limit. The ERC20 branch correctly includes `amount`. Any unprivileged user can deposit an arbitrarily large amount of ETH in one transaction, bypassing the configured cap entirely.

## Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` ( [1](#0-0) ):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount silently discarded
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

The ETH branch returns `true` (triggering revert) only when `totalAssetDeposits` **already** exceeds the limit before the deposit. The incoming `amount` is never factored in.

The public entry point `depositETH` passes `msg.value` as `depositAmount` to `_beforeDeposit` [2](#0-1) , which calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` at [3](#0-2) . The `depositAmount` (`msg.value`) is passed in but discarded by the ETH branch.

Exploit path:
- `depositLimit = 100 ETH`, `totalAssetDeposits = 0`
- Attacker calls `depositETH{value: 10_000 ether}(...)`
- Check evaluates `0 > 100 ETH` → `false` → no revert
- 10,000 ETH is accepted; `totalAssetDeposits` becomes 10,000 ETH
- The equivalent ERC20 call with 10,000 tokens against a 100-token limit would correctly revert

## Impact Explanation
**Medium — Temporary freezing of funds.** The ETH deposit cap is the protocol's mechanism to ensure deposited ETH does not exceed what EigenLayer strategies are configured to absorb. When the cap is bypassed, the pool can accumulate ETH far beyond what NDCs and EigenLayer strategies can process. Excess ETH remains stranded in the deposit pool, temporarily inaccessible to depositors who cannot withdraw it until the operator manually routes it — constituting a temporary freeze of user funds. The MANAGER's ability to cap ETH inflows during a risk event is rendered entirely ineffective.

## Likelihood Explanation
The bug requires no special role, no front-running, and no precondition beyond `totalAssetDeposits < depositLimit` (the normal operating state). Any external user calling `depositETH` with a large `msg.value` triggers it. It is repeatable and present in unmodified production code.

## Recommendation
Mirror the ERC20 branch for the ETH path in `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
```solidity
// Foundry fork test
function test_ETHDepositLimitBypassed() public {
    // Precondition: limit is 100 ETH, nothing deposited yet
    assertEq(lrtConfig.depositLimitByAsset(ETH_TOKEN), 100 ether);
    assertEq(depositPool.getTotalAssetDeposits(ETH_TOKEN), 0);

    // Attacker deposits 10x the limit in one call — no revert
    vm.deal(attacker, 1000 ether);
    vm.prank(attacker);
    depositPool.depositETH{value: 1000 ether}(0, "");

    // Total far exceeds the 100 ETH limit
    assertGt(depositPool.getTotalAssetDeposits(ETH_TOKEN), 100 ether);

    // Equivalent ERC20 deposit correctly reverts
    vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
    depositPool.depositAsset(stETH, 1000 ether, 0, "");
}
```
