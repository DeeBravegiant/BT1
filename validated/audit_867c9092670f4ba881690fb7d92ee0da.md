### Title
Caller Deposit Not Refunded in Multiple Error Paths of `address_check_callback` and `nep_141_storage_balance_callback` — (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

---

### Summary

The `rlp_execute` entry point of the ETH-implicit wallet contract is marked `#[payable]`, accepting NEAR deposits from any external caller (relayer). A `CallerDeposit` struct is designed to track and refund that deposit if the downstream cross-contract call fails. However, several early-return error branches in `address_check_callback` and `nep_141_storage_balance_callback` return without issuing the refund, permanently transferring the caller's attached NEAR into the wallet contract with no recovery path for the relayer.

---

### Finding Description

`rlp_execute` is `#[payable]` and captures the caller's attached deposit via `CallerDeposit::new()`: [1](#0-0) [2](#0-1) 

`CallerDeposit` is only populated for external (non-self) callers with a non-zero deposit: [3](#0-2) 

The deposit is refunded **only** inside `rlp_execute_callback` on `PromiseResult::Failed`: [4](#0-3) 

**Missing refunds in `address_check_callback`** — three early-return paths receive `caller_deposit` as a parameter but never use it:

1. Registrar promise failed: [5](#0-4) 

2. Unexpected JSON from registrar: [6](#0-5) 

3. Address already registered, non-self signer (invalid target): [7](#0-6) 

**Missing refunds in `nep_141_storage_balance_callback`** — three analogous early-return paths:

1. NEP-141 `storage_balance_of` promise failed: [8](#0-7) 

2. Unexpected JSON from NEP-141: [9](#0-8) 

3. Action is not `FunctionCall` (invariant violation branch): [10](#0-9) 

In every one of these six paths the function returns `PromiseOrValue::Value(ExecuteResponse { success: false, … })` directly. Because `rlp_execute_callback` is never scheduled, the refund transfer is never created, and the caller's NEAR remains in the wallet contract balance with no mechanism to recover it.

---

### Impact Explanation

The relayer's attached NEAR deposit is irrecoverably transferred into the wallet contract (the user's ETH-implicit account) whenever any of the six error branches above is taken. The corrupted value is the **relayer's account balance** (decreased) and the **wallet contract's account balance** (increased without authorization). The wallet contract owner did not sign any transaction authorizing this transfer; it is a side-effect of a failed intermediate promise. The relayer has no `withdrawStuckNEAR`-equivalent method to recover the funds.

---

### Likelihood Explanation

The `address_check` path is taken for every EOA-to-EOA base-token transfer where the target is another eth-implicit account. The registrar `lookup` call can fail if the registrar contract panics, runs out of gas, or is temporarily unavailable. The NEP-141 path is taken for every emulated ERC-20 transfer. Any relayer that attaches NEAR to `rlp_execute` (which the contract explicitly permits via `#[payable]`) and whose intermediate promise fails in one of these branches loses their deposit. The test `test_caller_refunds` confirms the refund mechanism exists and is expected, but does not cover these intermediate-callback failure paths. [11](#0-10) 

---

### Recommendation

In every early-return branch of `address_check_callback` and `nep_141_storage_balance_callback` that currently returns `PromiseOrValue::Value(ExecuteResponse { success: false, … })` without scheduling `rlp_execute_callback`, add an explicit refund transfer before returning:

```rust
if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
    let refund = env::promise_batch_create(&account_id);
    env::promise_batch_action_transfer(refund, NearToken::from_yoctonear(yocto_near.into()));
}
return PromiseOrValue::Value(ExecuteResponse { success: false, … });
```

Alternatively, refactor the callbacks so all failure paths route through `rlp_execute_callback`, which already contains the refund logic.

---

### Proof of Concept

1. Relayer calls `rlp_execute` on an ETH-implicit wallet contract, attaching 1 NEAR, with an Ethereum transaction encoding an EOA-to-EOA base-token transfer to another eth-implicit account.
2. `inner_rlp_execute` classifies this as `EOABaseTokenTransfer { address_check: Some(address), … }` and schedules `address_registrar.lookup(address).then(address_check_callback(…, caller_deposit))`. [12](#0-11) 
3. The registrar `lookup` call fails (promise result = `Failed`).
4. `address_check_callback` hits the `PromiseResult::Failed` branch and returns immediately — `caller_deposit` (1 NEAR) is never refunded. [5](#0-4) 
5. The relayer's 1 NEAR is now part of the wallet contract's balance. The relayer's balance is permanently reduced; no recovery path exists.

// An external caller gets its deposit back if the cross-contract call fails.
#[tokio::test]
async fn test_caller_refunds() -> anyhow::Result<()> {
    let TestContext { worker, wallet_contract, wallet_sk, address_registrar, .. } =
        TestContext::new().await?;

    let caller = worker.root_account()?;
    let deposit_amount = NearToken::from_near(3);
    let create_tx = |receiver_id: &AccountId, nonce: u64| {
        let method = "register";
        let args = br#"{"account_id": "birchmd.near"}"#;
        let action = Action::FunctionCall {
            receiver_id: receiver_id.to_string(),
            method_name: method.into(),
            args: args.to_vec(),
            gas: Gas::from_tgas(10).as_gas(),
            yocto_near: 0,
        };
        utils::create_signed_transaction(
            nonce,
            receiver_id,
            Wei::new_u128(deposit_amount.as_yoctonear() / (MAX_YOCTO_NEAR as u128)),
            action,
            &wallet_sk,
        )
    };

    // External caller gets a refund when the cross-contract call fails
    let pre_tx_account_balance = caller.view_account().await?.balance;
    let receiver_id: AccountId = "fake.near".parse()?;
    let result = wallet_contract
        .rlp_execute_from(
            &caller,
            receiver_id.as_str(),
            &create_tx(&receiver_id, 0),
            deposit_amount,
        )
        .await?;
    assert!(!result.success);
    let post_tx_account_balance = caller.view_account().await?.balance;
    assert!(
        pre_tx_account_balance.as_yoctonear() - post_tx_account_balance.as_yoctonear()
            < deposit_amount.as_yoctonear()
    );
