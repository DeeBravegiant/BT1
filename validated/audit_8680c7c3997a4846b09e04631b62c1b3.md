### Title
Cross-Chain Signature Replay on `deployToken` Due to Missing Chain ID in `MetadataPayload` Hash — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `aptos/sources/bridge_types.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

### Summary

The `MetadataPayload` Borsh encoding used in `deployToken` / `deploy_token` across all Omni Bridge destination chains (EVM, Starknet, Aptos, Solana) does not include any chain-identifying field. Because the NEAR MPC-derived signer address is the same across all chains, a valid `deployToken` signature produced for one chain is cryptographically valid on every other chain. An unprivileged attacker who observes a legitimate `deployToken` transaction on any chain can replay the signature on every other chain, deploying wrapped tokens without NEAR-side authorization for those chains.

### Finding Description

Every destination chain verifies `deployToken` signatures by hashing a Borsh-encoded `MetadataPayload` and recovering the signer against `nearBridgeDerivedAddress` / `near_bridge_derived_address`. The payload encoding is:

**EVM** (`OmniBridge.sol`):
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}

Because `nearBridgeDerivedAddress` is the same Ethereum-style address on every chain (it is derived from the NEAR MPC key, not per-chain), a signature that passes `ECDSA.recover(hash, sig) == nearBridgeDerivedAddress` on Ethereum will pass the identical check on Starknet, Aptos, and Solana for the same payload.

### Impact Explanation

An attacker who observes a valid `deployToken` call on any chain (the signature is in calldata, fully public) can immediately replay it on every other chain. This:

1. **Deploys wrapped tokens on chains the NEAR bridge has not authorized** — the NEAR bridge registers factories per-chain; a token deployed via replay on an unregistered chain creates a wrapped asset with no corresponding NEAR-side accounting.
2. **Permanently blocks legitimate deployment** — the `ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED` guard prevents the authorized relayer from ever deploying the token correctly on that chain after the attacker has front-run it. The deployed token's address is deterministic from the token ID, so the bridge's token-address mapping is now permanently set to the attacker-triggered deployment.
3. **Leads to irreversible fund lock** — users who call `initTransfer` on a chain where the token was deployed via replay (but the NEAR bridge has no registered factory for that chain) will have their tokens burned/locked on the destination chain with no corresponding NEAR-side settlement path.

This matches the allowed impact: *"Acceptance of insufficiently-bound signatures that bypass execution gates"* and *"Irreversible fund lock or permanently unclaimable user value."*

### Likelihood Explanation

- The attack requires zero privilege: `deployToken` is a public, permissionless entry point on all chains.
- The signature is fully visible in on-chain calldata the moment any chain processes a legitimate `deployToken`.
- The attacker only needs to copy the signature bytes and submit them to any other chain's bridge contract.
- The NEAR MPC-derived address is the same on all chains by design, so no key material is needed.

Likelihood is **High** given the trivial execution path.

### Recommendation

Bind the `MetadataPayload` hash to the destination chain by including the chain ID in the Borsh encoding, mirroring the existing `TransferMessagePayload` pattern:

```solidity
// EVM — OmniBridge.sol deployToken
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent change to `MetadataPayloadTrait::to_borsh` (Starknet), `metadata_to_borsh` (Aptos), and `DeployTokenPayload::serialize_for_near` (Solana). The NEAR MPC signing path must be updated to include the destination chain ID when producing the metadata signature.

### Proof of Concept

1. Observe a legitimate `deployToken(signatureData, {token:"wrap.near", name:"Wrapped NEAR", symbol:"wNEAR", decimals:24})` transaction on Ethereum mainnet. Extract `signatureData` from calldata.
2. Call `deployToken(signatureData, {token:"wrap.near", name:"Wrapped NEAR", symbol:"wNEAR", decimals:24})` on the Starknet `OmniBridge` contract with the identical arguments.
3. `_verify_borsh_signature` computes `keccak256(borsh({Metadata, "wrap.near", "Wrapped NEAR", "wNEAR", 24}))` — identical bytes to step 1 — and recovers the same `omni_bridge_derived_address`. Verification passes.
4. `wNEAR` is now deployed on Starknet without NEAR-side authorization. The `near_to_starknet_token` mapping is permanently set. Any subsequent legitimate deployment attempt reverts with `ERR_TOKEN_ALREADY_DEPLOYED`. [9](#0-8) [10](#0-9)
