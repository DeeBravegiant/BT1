### Title
Direct Deposits Not Included in Value Conservation Check, Enabling Unbounded ADA Creation - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces `DirectDeposits` — a transaction body field that directly credits arbitrary `Coin` amounts to registered staking account balances. The `dijkstraSubTxProducedValue` function used for sub-transaction value conservation does not include direct deposit amounts, and the top-level Conway-inherited `produced` function likewise has no knowledge of this Dijkstra-specific field. Because `applyDirectDeposits` unconditionally adds the attacker-specified amounts to account balances after the conservation check has already passed, a transaction author can create ADA out of thin air by embedding arbitrarily large direct deposit amounts in a sub-transaction (or top-level transaction) body.

---

### Finding Description

**Vulnerability class**: Funds/accounting bug — missing upper-bound check on a credited amount, analogous to the reported `settle()` issue where `curFillAmount` is not bounded by `quoteTokenData.toTrader`.

**New Dijkstra feature — DirectDeposits**

`DirectDeposits` is a `Map AccountAddress Coin` embedded in both top-level and sub-transaction bodies (CDDL field 25). It is applied to ledger state by `applyDirectDeposits`, which unconditionally adds each specified coin amount to the matching account's balance: [1](#0-0) 

The comment explicitly states: *"There are no checks that direct deposits mention only registered accounts."* The only pre-condition enforced before calling `applyDirectDeposits` is `directDepositsMissingAccounts` — a registration check, not an amount check: [2](#0-1) 

The same pattern is repeated verbatim in the sub-transaction path (`dijkstraSubEntitiesTransition`): [3](#0-2) 

**Missing inclusion in value conservation**

The sub-transaction produced-value function `dijkstraSubTxProducedValue` accounts for UTxO outputs, deposits, treasury donations, and burned multi-assets — but **not** direct deposits: [4](#0-3) 

The top-level UTXO rule calls `Shelley.validateValueNotConservedUTxO` using the Conway-inherited `produced` function, which also has no knowledge of the Dijkstra-specific `DirectDeposits` field: [5](#0-4) 

Because the conservation check runs at the UTXO level and `applyDirectDeposits` runs later at the ENTITIES/SUBENTITIES level, the direct deposit amounts are never deducted from any input side of the ledger equation. The `DirectDeposits` field in the transaction body is attacker-controlled and unbounded.
