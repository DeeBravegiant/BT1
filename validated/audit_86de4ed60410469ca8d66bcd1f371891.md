Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` at line 676 omits the incoming `amount` from the ETH branch comparison, checking only whether the running total already exceeds the cap rather than whether the new deposit would push it over. The identical guard for ERC-20 assets correctly includes `amount`. Any unprivileged depositor can therefore push total ETH deposits arbitrarily beyond `depositLimitByAsset(ETH_TOKEN)` in a single transaction.

## Finding Description
In `contracts/LRTDepositPool.sol`, `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682) has two branches:

```solidity
// ETH branch — amount is MISSING (line 679)
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
// ERC-20 branch — amount correctly included (line 681)
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
```

The ETH branch only fires when the running total **already** exceeds the cap. It never evaluates whether the new deposit would push the total over the cap. The caller `_beforeDeposit` (line 661) passes `depositAmount` (which equals `msg.value` for ETH) into this function, but the ETH branch silently ignores it. The ERC-20 branch at line 681 correctly adds `amount` to `totalAssetDeposits` before comparing.

Exploit path:
1. `depositLimitByAsset(ETH_TOKEN) = 1000 ETH`, `getTotalAssetDeposits(ETH_TOKEN) = 999 ETH`.
2. Attacker calls `depositETH{value: 500 ETH}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ETH)` evaluates `999 ETH > 1000 ETH` → `false` → no revert.
4. `_mintRsETH` mints rsETH for 500 ETH; total ETH deposits become 1499 ETH, 49.9% above the cap.
5. The same 500-token ERC-20 deposit evaluates `999 + 500 > 1000` → `true` → `MaximumDepositLimitReached` revert.

No existing check compensates for this omission. The `nonReentrant` and `whenNotPaused` modifiers on `depositETH` are unrelated to the limit logic.

## Impact Explanation
The ETH deposit cap is a protocol-level risk control. Bypassing it breaks the invariant `getTotalAssetDeposits(ETH) ≤ depositLimitByAsset(ETH)`, allowing rsETH to be minted against more ETH collateral than the protocol intends to accept. The protocol fails to deliver its promised deposit-cap guarantee. **Impact: Low — Contract fails to deliver promised returns.**

## Likelihood Explanation
The entry point is the public, permissionless `depositETH` function. No special role, governance action, or precondition beyond holding ETH is required. The condition `totalAssetDeposits ≤ depositLimitByAsset(ETH)` is the normal operating state of the protocol. Any depositor can trigger this in a single transaction. **Likelihood: High.**

## Recommendation
Add `amount` to the ETH branch, mirroring the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Foundry unit test plan:
1. Deploy `LRTDepositPool` with `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Seed the pool so `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.
3. Call `depositETH{value: 500 ether}(0, "")` from an unprivileged address.
4. Assert the call succeeds (no revert) and `getTotalAssetDeposits(ETH_TOKEN) == 1499 ether > 1000 ether`.
5. Repeat with an ERC-20 asset under identical conditions and assert `MaximumDepositLimitReached` is reverted — confirming the asymmetry. [1](#0-0) [2](#0-1) [3](#0-2)
