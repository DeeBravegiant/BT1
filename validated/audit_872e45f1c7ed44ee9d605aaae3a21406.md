Audit Report

## Title
Donation Attack via `balanceOf`/`address.balance` in TVL Accounting Inflates `rsETHPrice`, Causing Depositors to Receive Zero rsETH - (File: contracts/LRTDepositPool.sol)

## Summary
`getAssetDistributionData` and `getETHDistributionData` compute protocol TVL using raw `IERC20.balanceOf` and `address.balance` reads, which include tokens or ETH sent directly to the contracts without going through the deposit flow. An attacker can donate assets to inflate TVL, call the public `updateRSETHPrice()` to bake the inflated TVL into `rsETHPrice`, and cause subsequent depositors to receive zero rsETH for their deposits. Because `pricePercentageLimit` is never set in `initialize` and defaults to zero, the only price-increase guard is unconditionally disabled, allowing unbounded single-step price manipulation.

## Finding Description

**Root cause — LST path (`getAssetDistributionData`):**
`assetLyingInDepositPool = IERC20(asset).balanceOf(address(this))` at `LRTDepositPool.sol:444` and `assetLyingUnstakingVault = IERC20(asset).balanceOf(lrtUnstakingVault)` at `LRTDepositPool.sol:461` count any tokens present in those contracts, regardless of whether they arrived via `depositAsset`. A direct `IERC20.transfer` to `LRTDepositPool` is silently included in TVL.

**Root cause — ETH path (`getETHDistributionData`):**
`ethLyingInDepositPool = address(this).balance` at `LRTDepositPool.sol:480`. The contract has an open `receive() external payable {}` at `LRTDepositPool.sol:58`, so ETH can be force-sent and is immediately counted as protocol TVL.

**Price propagation — `_updateRsETHPrice`:**
`_getTotalEthInProtocol` at `LRTOracle.sol:331` calls `getTotalAssetDeposits` → `getAssetDistributionData`. The inflated total is divided by `rsethSupply` at `LRTOracle.sol:250` to produce `newRsETHPrice`. `updateRSETHPrice()` at `LRTOracle.sol:87` is `public` with no access control beyond `whenNotPaused`.

**Guard disabled by default:**
The price-increase guard at `LRTOracle.sol:256-257` is:
```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```
`pricePercentageLimit` is never assigned in `initialize` (`LRTOracle.sol:64-68`), so it defaults to zero. The guard is permanently disabled until an admin explicitly calls `setPricePercentageLimit`.

**Mint calculation — `getRsETHAmountToMint`:**
`rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` at `LRTDepositPool.sol:520`. With a sufficiently inflated `rsETHPrice`, integer division truncates this to zero.

**Zero-mint deposit succeeds:**
`_beforeDeposit` at `LRTDepositPool.sol:667` only reverts if `rsethAmountToMint < minRSETHAmountExpected`. If the victim passes `minRSETHAmountExpected = 0`, the check `0 < 0` is false and the deposit proceeds. `RSETH.mint(to, 0)` passes the `checkDailyMintLimit` modifier (since `currentPeriodMintedAmount + 0 > maxMintAmountPerDay` is false) and OpenZeppelin's `_mint(to, 0)` does not revert. The victim's tokens are transferred in at `LRTDepositPool.sol:114` but zero rsETH is minted.

**Theft completion:**
The attacker holds 100% of rsETH supply. `getExpectedAssetAmount` at `LRTWithdrawalManager.sol:593` computes `amount * rsETHPrice / assetPrice`. With the inflated price, the attacker's rsETH is priced to cover the donated tokens plus the victim's deposit. `getAvailableAssetAmount` at `LRTWithdrawalManager.sol:601-602` uses `getTotalAssetDeposits` (which still includes the donated tokens), so the withdrawal request passes the availability check. The attacker initiates withdrawal locking in the inflated `expectedAssetAmount`. The victim has no rsETH and no claim on the protocol.

## Impact Explanation

**Critical — Direct theft of user funds.**

A victim who deposits with `minRSETHAmountExpected = 0` (or any value ≤ the truncated amount) receives zero rsETH while their full deposit is transferred into the pool. The attacker, as the sole rsETH holder, holds a claim on all pool assets including the victim's deposit. This constitutes direct, permanent loss of the victim's deposited funds.

## Likelihood Explanation

The attack is most effective at protocol inception when rsETH supply is small. `updateRSETHPrice()` is permissionless. `pricePercentageLimit` defaults to zero, removing the only guard. The attacker needs capital equal to the donation `D` but recovers it through the withdrawal mechanism (the withdrawal's `expectedAssetAmount` is locked in at the inflated price at `initiateWithdrawal` time). The net cost is only the initial deposit, which is also recovered. No privileged access is required. The attack is repeatable across multiple victims.

## Recommendation

1. **Replace `balanceOf` with internal accounting:** Introduce a `mapping(address => uint256) internal depositedBalance` incremented in `depositAsset`/`depositETH` and decremented on withdrawal/transfer-out. Replace `IERC20(asset).balanceOf(address(this))` with `depositedBalance[asset]` and `address(this).balance` with a tracked `ethBalance` variable.

2. **Set `pricePercentageLimit` in `initialize`:** Assign a non-zero default (e.g., `1e16` for 1%) in `LRTOracle.initialize` so the price-increase guard is active from deployment.

3. **Reject zero-rsETH mints:** In `_beforeDeposit`, add `if (rsethAmountToMint == 0) revert MinimumAmountToReceiveNotMet();` to prevent deposits that produce no rsETH regardless of the slippage parameter.

## Proof of Concept

```
Preconditions: rsETH supply = 0, rsETHPrice = 1e18, stETH assetPrice = 1e18,
               pricePercentageLimit = 0 (default), depositLimit >> 1e18 + D

Step 1 — Attacker deposits 1e18 stETH via depositAsset():
  rsethAmountToMint = 1e18 * 1e18 / 1e18 = 1e18
  rsETH supply = 1e18, attacker owns 100%

Step 2 — Attacker transfers D stETH directly to LRTDepositPool (no deposit call):
  IERC20(stETH).transfer(address(lrtDepositPool), D)
  getTotalAssetDeposits(stETH) = 1e18 + D

Step 3 — Attacker calls updateRSETHPrice() (public, no access control):
  totalETHInProtocol = (1e18 + D) * 1e18 / 1e18 = 1e18 + D
  newRsETHPrice = (1e18 + D) * 1e18 / 1e18 = 1e18 + D
  isPriceIncreaseOffLimit = (0 > 0 && ...) = false  → no revert
  rsETHPrice = 1e18 + D

Step 4 — Victim deposits B stETH via depositAsset(minRSETHAmountExpected=0):
  rsethAmountToMint = B * 1e18 / (1e18 + D)
  With D >> B: rsethAmountToMint = 0
  Check: 0 < 0 → false → no revert
  RSETH.mint(victim, 0) → succeeds, victim receives 0 rsETH
  Victim's B stETH is now in the pool.

Step 5 — Attacker calls initiateWithdrawal(stETH, 1e18 rsETH):
  expectedAssetAmount = 1e18 * (1e18 + D) / 1e18 = 1e18 + D
  getAvailableAssetAmount = getTotalAssetDeposits = 1e18 + D + B ≥ 1e18 + D → passes
  Withdrawal request locked in at 1e18 + D stETH.

Step 6 — After operator processes unlockQueue (price still inflated):
  Attacker receives 1e18 + D stETH, recovering the donation.
  Victim's B stETH remains in pool, claimable by attacker's residual rsETH position.
  Net attacker profit: B stETH (victim's entire deposit).

Foundry test plan:
  - Deploy LRTDepositPool, LRTOracle, RSETH with pricePercentageLimit = 0
  - Attacker: depositAsset(stETH, 1e18, 0)
  - Attacker: IERC20(stETH).transfer(depositPool, 1e30)
  - Attacker: lrtOracle.updateRSETHPrice()
  - Assert rsETHPrice >> 1e18
  - Victim: depositAsset(stETH, 1e18, 0)
  - Assert RSETH.balanceOf(victim) == 0
  - Assert IERC20(stETH).balanceOf(depositPool) increased by 1e18
```
