Audit Report

## Title
Fee-Before-Price Ordering in `_updateRsETHPrice` Allows Multi-Period Fee Accumulation to Revert and Freeze `rsETHPrice` via `DailyFeeMintLimitExceeded` - (File: `contracts/LRTOracle.sol`)

## Summary

In `_updateRsETHPrice()`, `_checkAndUpdateDailyFeeMintLimit()` is called at line 303 **before** `rsETHPrice` is committed at line 313. When the oracle goes unupdated for ≥2 periods (whether via block stuffing or keeper outage), the accumulated multi-period fee exceeds `maxFeeMintAmountPerDay`, causing an unconditional revert that leaves `rsETHPrice` permanently stale until the manager raises the daily limit via `setMaxFeeMintAmountPerDay()`.

## Finding Description

The execution order in `_updateRsETHPrice()` is confirmed in the code:

**Step 1 — Fee accumulates over all missed periods:**
`previousTVL` is computed as `rsethSupply.mulWad(rsETHPrice)` using the *stale* stored price. If the oracle was not updated for N days, `rewardAmount = totalETHInProtocol - previousTVL` captures N days of staking rewards, and `protocolFeeInETH` scales proportionally. [1](#0-0) 

**Step 2 — Daily limit check fires before price write:**
`_checkAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee)` is called at line 303. If `feeAmount > maxFeeMintAmountPerDay`, it reverts unconditionally with `DailyFeeMintLimitExceeded`. The period reset at lines 199–201 correctly zeroes `currentPeriodMintedFeeAmount`, but `feeAmount` passed in is the entire N-day fee, not one day's worth. [2](#0-1) 

**Step 3 — `rsETHPrice` is never reached:**
`rsETHPrice = newRsETHPrice` at line 313 is unreachable while the revert condition holds. Every subsequent call to `updateRSETHPrice()` re-computes an even larger accumulated fee (since `rsETHPrice` remains stale), making the revert self-reinforcing. [3](#0-2) 

The only escape is the manager calling `setMaxFeeMintAmountPerDay()` with a sufficiently large value. [4](#0-3) 

## Impact Explanation

**Low. Block stuffing.** `rsETHPrice` storage is frozen. `LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, so all deposit minting uses a stale (undervalued) exchange rate, over-minting rsETH for depositors and diluting existing holders. The freeze is temporary — it persists until the manager raises `maxFeeMintAmountPerDay` — but the price update mechanism is completely broken during that window. [5](#0-4) 

## Likelihood Explanation

Block stuffing on Ethereum mainnet for 2+ consecutive days is extremely capital-intensive. However, the same condition is reachable via an unintentional keeper outage (no attacker required), which is a realistic operational failure. The precondition is only 2 missed periods, and if `maxFeeMintAmountPerDay` is calibrated tightly to one day's expected fee (a natural configuration), the off-by-one condition is trivially hit. `updateRSETHPrice()` is a public function callable by anyone, so the freeze cannot be broken by a third party — only the manager can resolve it. [6](#0-5) 

## Recommendation

Move `rsETHPrice = newRsETHPrice` **before** the fee check, or replace the reverting check with a non-reverting cap that mints up to `maxFeeMintAmountPerDay` and carries forward the remainder. This ensures the price is always committed regardless of fee limit status:

```solidity
// Commit price first — always reachable
rsETHPrice = newRsETHPrice;

// Then cap fee minting, never revert
uint256 cappedFee = _capAndUpdateDailyFeeMintLimit(rsethAmountToMintAsProtocolFee);
if (cappedFee > 0) {
    IRSETH(rsETHTokenAddress).mint(treasury, cappedFee);
}
```

## Proof of Concept

```solidity
function test_multiPeriodFeeFreezesRsETHPrice() public {
    // Set daily limit to exactly one day's expected fee
    oracle.setMaxFeeMintAmountPerDay(DAILY_FEE);

    // Simulate 2 days passing without updateRSETHPrice being called
    vm.warp(block.timestamp + 2 days);

    // Underlying assets accrue 2 days of staking rewards (mock oracle returns higher TVL)
    // rewardAmount = 2 * dailyReward => protocolFee = 2 * DAILY_FEE > maxFeeMintAmountPerDay

    uint256 priceBefore = oracle.rsETHPrice();

    // First call after 2-day gap: fee = 2x daily fee > maxFeeMintAmountPerDay => revert
    vm.expectRevert(ILRTOracle.DailyFeeMintLimitExceeded.selector);
    oracle.updateRSETHPrice();

    // rsETHPrice is unchanged — frozen at stale value
    assertEq(oracle.rsETHPrice(), priceBefore);

    // Subsequent calls also revert (fee keeps growing), freeze is self-reinforcing
    vm.warp(block.timestamp + 1 days);
    vm.expectRevert(ILRTOracle.DailyFeeMintLimitExceeded.selector);
    oracle.updateRSETHPrice();
}
```

    function _checkAndUpdateDailyFeeMintLimit(uint256 feeAmount) internal {
        // Reset the period if it's unset or a day has passed
        if (block.timestamp >= feePeriodStartTime + 1 days) {
            currentPeriodMintedFeeAmount = 0;
            feePeriodStartTime = getCurrentPeriodStartTime();
        }

        // Check if minting would exceed the daily limit
        if (currentPeriodMintedFeeAmount + feeAmount > maxFeeMintAmountPerDay) {
            revert DailyFeeMintLimitExceeded(currentPeriodMintedFeeAmount + feeAmount, maxFeeMintAmountPerDay);
        }

        currentPeriodMintedFeeAmount += feeAmount;
    }
