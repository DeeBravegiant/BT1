Audit Report

## Title
Daily Minting Limit Bypass at Day Boundary via Integer Division in `getCurrentDay()` — (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV2.sol`, `contracts/pools/RSETHPoolV2ExternalBridge.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`, `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol`)

## Summary
All five L2 pool contracts compute the current day as `(block.timestamp - startTimestamp) / 1 days`, creating a hard discrete boundary. Because `limitDailyMint` resets `dailyMintAmount` only when `currentDay > lastMintDay`, an unprivileged depositor can mint up to `dailyMintLimit` at the last second of day N and immediately mint another full `dailyMintLimit` at the first second of day N+1, doubling the intended daily cap across two consecutive blocks. The concrete, self-contained impact is that the contract fails to enforce its promised daily minting limit — the circuit breaker does not function as specified.

## Finding Description
`getCurrentDay()` is defined identically in all five contracts:

```solidity
function getCurrentDay() public view returns (uint256) {
    return (block.timestamp - startTimestamp) / 1 days;
}
```

The `limitDailyMint` modifier resets state only on a strict `>` comparison:

```solidity
uint256 currentDay = getCurrentDay();
if (currentDay > lastMintDay) {
    lastMintDay = currentDay;
    dailyMintAmount = 0;
}
if (dailyMintAmount + rsETHAmount > dailyMintLimit) revert DailyMintLimitExceeded();
dailyMintAmount += rsETHAmount;
```

At `startTimestamp + N×86400 − 1`, `getCurrentDay()` returns `N−1`; at `startTimestamp + N×86400`, it returns `N`. There is no overlap. An attacker who fills `dailyMintAmount` to `dailyMintLimit` at the last second of day N can immediately submit another deposit at the first second of day N+1, triggering the reset and minting another full `dailyMintLimit`. No privileged access, no oracle manipulation, and no special preconditions are required beyond holding sufficient ETH/LST. The exploit is repeatable every day boundary.

## Impact Explanation
The daily minting limit is a protocol-level guarantee: the contract promises to cap rsETH issuance to `dailyMintLimit` per 24-hour period. The bypass allows 2× that cap to be minted in two consecutive blocks. Because the attacker deposits real ETH/LST backing the extra rsETH, there is no direct fund loss or insolvency from the bypass alone — the pool remains solvent as long as the oracle rate is accurate. The concrete, self-contained impact is therefore **Low: Contract fails to deliver promised returns (security guarantee), but does not lose value**. The Medium impact of "temporary freezing of funds" claimed in the submission requires a simultaneously stale or manipulated oracle, which is an external dependency excluded by SECURITY.md ("Incorrect data supplied by third-party oracles") and is not demonstrated by the proof of concept.

## Likelihood Explanation
Any depositor can monitor `startTimestamp` and `lastMintDay` on-chain, compute the exact day boundary, and submit two back-to-back deposits. On L2 chains with 1–2 second block times, both transactions can be submitted within seconds of each other. No special privileges are required. The exploit is repeatable at every day boundary.

## Recommendation
Replace the discrete day-counter with a continuous timestamp-based window. Store `mintPeriodStart` and reset when `block.timestamp >= mintPeriodStart + 1 days`:

```solidity
uint256 public mintPeriodStart;

// In limitDailyMint:
if (block.timestamp >= mintPeriodStart + 1 days) {
    mintPeriodStart = block.timestamp;
    dailyMintAmount = 0;
}
```

This ensures the 24-hour window is always measured from the last reset, eliminating the exploitable hard boundary.

## Proof of Concept
Assume `startTimestamp = 1_000_000`, `dailyMintLimit = 100 ether`, `lastMintDay = 0`.

| Step | `block.timestamp` | `getCurrentDay()` | `lastMintDay` | `dailyMintAmount` before | Action | `dailyMintAmount` after |
|------|-------------------|-------------------|---------------|--------------------------|--------|-------------------------|
| 1 | 1,086,399 | 0 | 0 | 0 | Deposit → mint 100 rsETH | 100 ether |
| 2 | 1,086,400 | **1** | 0 → **1** | **reset to 0** | Deposit → mint 100 rsETH | 100 ether |

**Result:** 200 rsETH minted (2× `dailyMintLimit`) across two consecutive blocks. A Foundry unit test can reproduce this by warping `block.timestamp` to `startTimestamp + 86399`, calling `deposit` to fill the limit, then warping to `startTimestamp + 86400` and calling `deposit` again — the second call must succeed rather than revert with `DailyMintLimitExceeded`.
