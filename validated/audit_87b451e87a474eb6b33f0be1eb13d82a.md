### Title
Unprivileged Front-Running of `SignedDelegateAction` Consumes Sender Nonce and Silently Invalidates Relayer's Meta-Transaction - (`runtime/runtime/src/actions.rs`)

### Summary

An unprivileged attacker who observes a pending meta-transaction in the NEAR mempool can extract the embedded `SignedDelegateAction`, wrap it in their own outer `SignedTransaction` (paying only their own gas), and submit it via the public RPC. If the attacker's transaction is included first, `validate_delegate_action_key` advances Alice's access-key nonce to the delegate nonce value. When the legitimate relayer's transaction is later processed, the same function finds `delegate_nonce <= current_nonce` and returns `DelegateActionInvalidNonce`, causing the relayer's receipt to fail. Alice's intended action is never executed, and the relayer's gas is wasted.

### Finding Description

NEAR meta-transactions (NEP-366) allow a user (Alice) to sign a `DelegateAction` off-chain and hand it to a relayer. The relayer wraps the `SignedDelegateAction` inside a regular `SignedTransaction` addressed to Alice's account and submits it via the public JSON-RPC endpoint.

The `SignedDelegateAction` struct carries Alice's cryptographic signature over the inner action payload: [1](#0-0) 

The outer transaction is publicly visible in the mempool. Any observer can deserialize it, extract the `SignedDelegateAction`, and re-wrap it in a new `SignedTransaction` of their own (with themselves as the outer signer, and Alice's account as `receiver_id`). This is a valid transaction because:

1. The inner signature is Alice's — it remains cryptographically valid regardless of who wraps it.
2. The outer transaction's `receiver_id` must equal `delegate_action.sender_id` (Alice), which is public.
3. There is no field in `DelegateAction` that binds it to a specific relayer.

When `apply_delegate_action` processes the attacker's transaction first, it calls `validate_delegate_action_key`: [2](#0-1) 

On success, the nonce is immediately written back to state: [3](#0-2) 

When the legitimate relayer's transaction is subsequently processed, `validate_delegate_action_key` finds `delegate_nonce.nonce() <= current_nonce` (the nonce was already advanced) and returns `DelegateActionInvalidNonce`. The relayer's receipt fails, and Alice's inner actions are never dispatched.

The `DelegateAction` nonce comment itself acknowledges this is the nonce's purpose — but there is no protection against a third party consuming it: [4](#0-3) 

### Impact Explanation

- **Corrupted protocol value**: Alice's access-key nonce is advanced by the attacker's transaction. The relayer's transaction produces `DelegateActionInvalidNonce` instead of the expected inner action receipt. Alice's intended on-chain action (e.g., `ft_transfer`) is never executed.
- **Relayer gas loss**: The relayer's outer transaction is charged gas even though the inner action fails.
- **User impact**: Alice must re-sign a new `DelegateAction` with a fresh nonce. A persistent attacker can repeat the attack on every re-submission, permanently blocking Alice's meta-transactions.
- **Protocol reputation**: The meta-transaction feature (NEP-366) becomes unreliable for any use case where the `SignedDelegateAction` is transmitted through a public channel (e.g., a public relayer API or any observable mempool).

### Likelihood Explanation

NEAR transactions are broadcast to the network and are observable by any node via the public RPC (`broadcast_tx_async`/`broadcast_tx_commit`). The `SignedDelegateAction` is Borsh-serialized inside the outer transaction's action list and is trivially deserializable. The attacker needs only to:
1. Monitor the mempool for transactions containing `Action::Delegate`.
2. Deserialize the `SignedDelegateAction`.
3. Construct and submit a competing transaction.

No special privileges, keys, or stake are required. The attack is cheap (attacker pays only their own outer-transaction gas) and repeatable.

### Recommendation

Bind the `SignedDelegateAction` to a specific relayer by adding an optional `relayer_id: Option<AccountId>` field to `DelegateAction`. During `apply_delegate_action`, verify that `action_receipt.signer_id == delegate_action.relayer_id` when the field is set. This prevents any party other than the designated relayer from submitting the action.

As a shorter-term mitigation (analogous to the ERC20 permit fix), `apply_delegate_action` could treat a `DelegateActionInvalidNonce` as a no-op (success with no new receipt) when the nonce has already been advanced past the delegate nonce — signalling that the action was already executed rather than that it was front-run. This avoids charging the relayer for a failure caused by a third party.

### Proof of Concept

```
// Pseudocode — all steps are achievable via the public JSON-RPC.

// Step 1: Alice signs a DelegateAction (e.g., ft_transfer to John).
let delegate_action = DelegateAction {
    sender_id: "alice.near",
    receiver_id: "ft.near",
    actions: [ft_transfer("john", 10)],
    nonce: alice_access_key_nonce + 1,   // e.g., 5
    max_block_height: current_height + 100,
    public_key: alice_pub_key,
};
let signed_da = SignedDelegateAction::sign(&alice_signer, delegate_action);

// Step 2: Alice sends signed_da to the relayer off-chain.
// Relayer wraps it and submits:
//   SignedTransaction { signer=relayer, receiver=alice, actions=[Delegate(signed_da)] }
// This transaction is now visible in the mempool.

// Step 3: Attacker observes the mempool, extracts signed_da, and submits:
//   SignedTransaction { signer=attacker, receiver=alice, actions=[Delegate(signed_da)] }
// Attacker's tx is valid — alice's signature is unchanged.

// Step 4: Attacker's tx is included first.
// validate_delegate_action_key: alice_access_key.nonce is set to 5.

// Step 5: Relayer's tx is included next.
// validate_delegate_action_key: delegate_nonce (5) <= current_nonce (5) → DelegateActionInvalidNonce.
// Alice's ft_transfer never executes. Relayer's gas is burned.
```

pub struct SignedDelegateAction {
    pub delegate_action: DelegateAction,
    pub signature: Signature,
}

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}
