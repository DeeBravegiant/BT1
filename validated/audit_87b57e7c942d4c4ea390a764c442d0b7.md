### Title
Incorrect Complete Deletion of Shared VRF Key Hash in `poolReapTransition` Allows Bypassing Post-v11 VRF Uniqueness Invariant - (File: `eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

In `poolReapTransition`, when a pool re-registers with a new VRF key, the old VRF key hash is completely removed from `psVRFKeyHashes` using `Map.withoutKeys` instead of being decremented by one occurrence. If the old VRF key hash was shared by multiple pools (which was permitted before the v11 hard fork), this complete deletion incorrectly erases the entry even though other pools still reference that VRF key. A pool operator can then register a new pool with the "freed" VRF key hash, bypassing the post-v11 uniqueness invariant.

---

### Finding Description

`PState` contains a reference-counted map `psVRFKeyHashes :: Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)` that tracks how many registered pools use each VRF key hash. Post-v11, this map enforces that no two pools share the same VRF key:

```haskell
Map.notMember sppVrf psVRFKeyHashes
  ?! injectFailure (VRFKeyHashAlreadyRegistered sppId sppVrf)
``` [1](#0-0) 

In `poolReapTransition`, two distinct operations update `psVRFKeyHashes` at epoch boundary:

**Operation 1 (correct):** `removeVRFKeyHashOccurrences retiredVRFKeyHashes` — decrements the count for each VRF key hash belonging to pools that are retiring this epoch, removing the map entry only when the count reaches zero. [2](#0-1) [3](#0-2) 

**Operation 2 (incorrect):** `Map.withoutKeys danglingVRFKeyHashes` — **completely deletes** entries for VRF key hashes that were "overwritten" by a pool re-registration with a new VRF key. [4](#0-3) 

`danglingVRFKeyHashes` is computed as the set of old VRF keys for pools that have a pending re-registration with a different VRF key. The code then calls `Map.withoutKeys danglingVRFKeyHashes` to remove these old VRF keys from the map entirely:

```haskell
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )
``` [2](#0-1) 

`Map.withoutKeys` unconditionally deletes the key regardless of its count. Before the v11 hard fork, multiple pools could share the same VRF key hash. After the v11 hard fork, `populateVRFKeyHashes` correctly initialises counts reflecting all such shared registrations: [5](#0-4) 

When one of those pools re-registers with a new VRF key, `danglingVRFKeyHashes` contains the old shared VRF key. `Map.withoutKeys` then removes the entire entry — even though the count was 2 (or more) and another pool still holds that VRF key. The correct operation is to decrement the count by 1, exactly as `removeVRFKeyHashOccurrences` does for retiring pools.

The `PState` definition confirms the reference-counted design: [6](#0-5) 

---

### Impact Explanation

After the bug fires, `psVRFKeyHashes` no longer contains the old VRF key hash even though another pool still uses it. The uniqueness check `Map.notMember sppVrf psVRFKeyHashes` then incorrectly returns `True` for that VRF key, allowing a new pool to register with it. Two pools now share the same VRF key post-v11, violating the protocol invariant introduced specifically to prevent this. The `psVRFKeyHashes` map becomes permanently inconsistent with the actual set of registered VRF keys, and the uniqueness enforcement is silently bypassed for the affected key.

This matches: **Medium — attacker-controlled pool registration certificates exceed intended validation limits (VRF key uniqueness check).**

---

### Likelihood Explanation

The precondition — multiple pools sharing the same VRF key — was explicitly permitted before the v11 hard fork and is a realistic legacy state on mainnet. Any pool operator who (a) registered a pool with a shared VRF key before v11, (b) re-registers that pool with a new VRF key after v11, and (c) then submits a new pool registration with the old VRF key can trigger the bypass. No privileged access, governance majority, or key compromise is required. The attacker controls only ordinary pool registration certificates (`RegPoolTxCert`).

---

### Recommendation

Replace the complete-deletion path for `danglingVRFKeyHashes` with the same decrement logic used for retiring pools:

```haskell
-- Before (incorrect):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . (`Map.withoutKeys` danglingVRFKeyHashes)
     )

-- After (correct):
& certPStateL . psVRFKeyHashesL
  %~ ( removeVRFKeyHashOccurrences retiredVRFKeyHashes
         . removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
     )
```

`removeVRFKeyHashOccurrences` already implements the correct semantics — it decrements the count and removes the entry only when the count reaches zero — so reusing it for `danglingVRFKeyHashes` is the minimal, consistent fix. [3](#0-2) 

---

### Proof of Concept

**Setup (before v11 hard fork):**
1. Register Pool A with `VRF1`.
2. Register Pool B with `VRF1` (same VRF key — allowed pre-v11).
3. Enact the v11 hard fork. `populateVRFKeyHashes` sets `psVRFKeyHashes = {VRF1: 2}`.

**Trigger:**
4. Pool A re-registers with `VRF2`. `psFutureStakePoolParams[A] = {vrf: VRF2}`.
5. Epoch boundary fires `poolReapTransition`.
   - `danglingVRFKeyHashes = {VRF1}` (Pool A's current VRF1 ≠ future VRF2).
   - `Map.withoutKeys {VRF1}` completely removes `VRF1` from `psVRFKeyHashes`.
   - Result: `psVRFKeyHashes = {VRF2: 1}`. **`VRF1` is gone, even though Pool B still uses it.**

**Exploit:**
6. Register Pool C with `VRF1`.
   - Check: `Map.notMember VRF1 psVRFKeyHashes` → `True` (incorrectly).
   - Pool C is accepted. Pool B and Pool C now both use `VRF1` post-v11.
   - The uniqueness invariant is violated; `psVRFKeyHashes` is permanently stale.

populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
```
