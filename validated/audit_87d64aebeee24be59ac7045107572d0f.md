### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Inject Arbitrary Sequencer Address — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` in `validate_proposal.rs` validates every security-sensitive field of `ProposalInit` **except** `builder`. The `builder` field is passed verbatim through `convert_to_sn_api_block_info` into the `BlockInfo.sequencer_address` used for block execution. Because `sequencer_address` is (a) the fee-collection target for every transaction in the block, (b) a direct input to `PartialBlockHashComponents` and therefore the `PartialBlockHash` / `ProposalCommitment`, and (c) the value returned to contracts by the `get_execution_info` syscall, an unvalidated `builder` lets the current-round proposer silently redirect all block fees and corrupt the authoritative block-hash commitment accepted by consensus.

---

### Finding Description

`ProposalInit` carries a `builder` field ("Address of the one who builds/sequences the block"): [1](#0-0) 

`is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas-price fields, `starknet_version`, `version_constant_commitment`, `timestamp`, and `fee_proposal_fri`: [2](#0-1) 

`builder` is never mentioned. After `is_proposal_init_valid` returns `Ok`, `initiate_validation` calls `convert_to_sn_api_block_info(init)` and forwards the result directly to the batcher: [3](#0-2) 

`convert_to_sn_api_block_info` maps `init.builder` → `BlockInfo.sequencer_address`. The batcher then constructs `PartialBlockHashComponents` from that `BlockInfo`: [4](#0-3) 

`sequencer_address` is hashed into `PartialBlockHash` / `ProposalCommitment`: [5](#0-4) 

The proposer's own batcher also uses `init.builder` (set from `args.builder_address`, a local static config): [6](#0-5) 

Because both sides derive `sequencer_address` from the same `ProposalInit.builder` wire value, the `PartialBlockHash` computed by the validator will always match the proposer's `ProposalFin.proposal_commitment` — regardless of what address `builder` contains. The `ProposalFinMismatch` guard at line 244 of `validate_proposal.rs` therefore provides no protection: [7](#0-6) 

The consensus-level check in `manager.rs` validates only `init.proposer`, not `init.builder`: [8](#0-7) 

---

### Impact Explanation

**Incorrect fee accounting with direct economic impact (Critical).**

`sequencer_address` is the address to which every transaction's fee is transferred during execution. A malicious proposer sets `builder` to an address they control; the validator accepts the proposal; the block is committed with that address as `sequencer_address`; all fees in the block flow to the attacker. Additionally, the committed `PartialBlockHash` (and therefore the on-chain block hash) encodes the attacker-chosen address, permanently corrupting the authoritative state commitment for that height. Contracts that call `get_execution_info` receive the spoofed sequencer address, breaking any contract-level access-control logic that gates on `sequencer_address`.

---

### Likelihood Explanation

The trigger is the current-round proposer sending a `ProposalInit` with a crafted `builder` value. No special infrastructure is required beyond being selected as proposer for one round. In a multi-validator network any validator can reach this role. The attack is silent: the `ProposalFin` commitment still matches (both sides hash the same injected address), so no existing metric or log distinguishes a legitimate from a malicious proposal.

---

### Recommendation

Add a `builder` field to `ProposalInitValidation` (populated from the local node's configured `builder_address` or from the committee) and reject any `ProposalInit` whose `builder` does not match:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.expected_builder, init_proposed.builder
        ),
    ));
}
```

The `expected_builder` value should be derived from the same source the proposer uses (`static_config.builder_address`) and agreed upon by the committee, mirroring how `l2_gas_price_fri` and `l1_da_mode` are already cross-checked.

---

### Proof of Concept

1. Malicious validator is selected as proposer for height H, round R.
2. It constructs `ProposalInit { builder: attacker_address, … }` with all other fields honest.
3. Validator node receives the proposal; `is_proposal_init_valid` passes (no `builder` check).
4. `initiate_validation` → `convert_to_sn_api_block_info(init)` → `BlockInfo { sequencer_address: attacker_address, … }` is sent to the batcher.
5. Batcher executes all transactions; fees are transferred to `attacker_address`.
6. Batcher computes `PartialBlockHash` over `PartialBlockHashComponents { sequencer: attacker_address, … }`.
7. Validator's `batcher_block_commitment` matches `ProposalFin.proposal_commitment` (both used `attacker_address`); `ProposalFinMismatch` is not triggered.
8. Consensus reaches decision; block is committed with `sequencer_address = attacker_address` and all fees redirected. [9](#0-8) [10](#0-9) [11](#0-10)

#[instrument(level = "warn", skip_all, fields(?proposal_init_validation, ?init_proposed))]
async fn is_proposal_init_valid(
    proposal_init_validation: &ProposalInitValidation,
    init_proposed: &ProposalInit,
    clock: &dyn Clock,
    l1_gas_price_provider: Arc<dyn L1GasPriceProviderClient>,
    gas_price_params: &GasPriceParams,
) -> ValidateProposalResult<()> {
    let now: u64 = clock.unix_now();
    let last_block_timestamp =
        proposal_init_validation.previous_proposal_init.as_ref().map_or(0, |info| info.timestamp);
    if init_proposed.timestamp < last_block_timestamp {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is too old: last_block_timestamp={}, proposed={}",
                last_block_timestamp, init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "Timestamp is in the future: now={}, block_timestamp_window_seconds={}, \
                 proposed={}",
                now,
                proposal_init_validation.block_timestamp_window_seconds,
                init_proposed.timestamp
            ),
        ));
    }
    if init_proposed.starknet_version != proposal_init_validation.starknet_version {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "starknet_version mismatch: expected={:?}, proposed={:?}",
                proposal_init_validation.starknet_version, init_proposed.starknet_version
            ),
        ));
    }
    // `version_constant_commitment` is proposer-supplied (network-derived). It is not yet a real
    // commitment (see `expected_version_constant_commitment`): the only valid value is the
    // sentinel, so reject anything else. Enforcing the same value the proposer emits keeps the two
    // sides in lockstep, so a real value cannot ship on one side without the other.
    let expected_commitment = expected_version_constant_commitment();
    if init_proposed.version_constant_commitment != expected_commitment {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            format!(
                "version_constant_commitment mismatch: expected={expected_commitment:?}, \
                 proposed={:?}",
                init_proposed.version_constant_commitment
            ),
        ));
    }
    if !(init_proposed.height == proposal_init_validation.height
        && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
        && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
    {
        return Err(ValidateProposalError::InvalidProposalInit(
            init_proposed.clone(),
            proposal_init_validation.clone(),
            "ProposalInit validation failed".to_string(),
        ));
    }

async fn initiate_validation(
    batcher: Arc<dyn BatcherClient>,
    state_sync_client: SharedStateSyncClient,
    init: &ProposalInit,
    proposal_id: ProposalId,
    timeout_plus_margin: Duration,
    clock: &dyn Clock,
    compare_retrospective_block_hash: bool,
) -> ValidateProposalResult<()> {
    let chrono_timeout = chrono::Duration::from_std(timeout_plus_margin)
        .expect("Can't convert timeout to chrono::Duration");

    let input = ValidateBlockInput {
        proposal_id,
        deadline: clock.now() + chrono_timeout,
        retrospective_block_hash: retrospective_block_hash(
            batcher.clone(),
            state_sync_client,
            init,
            compare_retrospective_block_hash,
        )
        .await
        .map_err(ValidateProposalError::from)?,
        block_info: convert_to_sn_api_block_info(init)?,
    };
    debug!("Initiating validate proposal: input={input:?}");
    batcher.validate_block(input.clone()).await.map_err(|err| {
        ValidateProposalError::Batcher(
            format!("Failed to initiate validate proposal {input:?}."),
            err,
        )
    })?;
    Ok(())
}
