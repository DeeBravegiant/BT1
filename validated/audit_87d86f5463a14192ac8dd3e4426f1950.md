### Title
Stale-Epoch Announcement Replay Overwrites Routing Entry via `account_peers` / `account_peers_broadcasted` Divergence — (`chain/network/src/announce_accounts/mod.rs`)

---

### Summary

A real divergence exists between `account_peers` and `account_peers_broadcasted` in `AnnounceAccountCache`. When a disk-stored entry is loaded via `get_announce` (triggered by `get_account_owner`), it is inserted only into `account_peers`, not `account_peers_broadcasted`. The epoch-freshness guard in `handle_sync_routing_table` / `ViewClientActor` consults only `account_peers_broadcasted`. When the entry is absent there, `last_epoch = None`, and the freshness check is unconditionally skipped. A peer holding a captured, valid-signature announcement from a prior epoch can replay it to overwrite the current routing entry for that `account_id`, redirecting `ForwardTx` messages to the wrong peer.

---

### Finding Description

**Step 1 — Disk load creates divergence.**

`get_announce` inserts into `account_peers` only: [1](#0-0) 

The struct comment explicitly acknowledges this: [2](#0-1) 

`get_account_owner` calls `get_announce`, so any routed message lookup (e.g., a `ForwardTx` to a validator account) triggers the disk load and creates the divergence. [3](#0-2) 

**Step 2 — `handle_sync_routing_table` reads only `account_peers_broadcasted`.** [4](#0-3) 

Because the disk-loaded entry is absent from `account_peers_broadcasted`, `old.get(&id)` returns `None`, so `last_epoch = None` for that `account_id`.

**Step 3 — Epoch-freshness guard is skipped when `last_epoch = None`.** [5](#0-4) 

The `if let Some(last_epoch)` branch is not entered; the announcement proceeds directly to `check_signature_account_announce`.

**Step 4 — Signature check passes for a replayed prior-epoch announcement.**

`check_signature_account_announce` verifies that the account *was* a validator for the given `epoch_id` and that the signature is valid. A captured legitimate announcement from a previous epoch satisfies both conditions.

**Step 5 — `add_accounts` overwrites `account_peers` with the stale entry.** [6](#0-5) 

The only guard in `add_accounts` is whether `account_peers_broadcasted` already holds the *same* `epoch_id`. Since `account_peers_broadcasted` is empty for the disk-loaded entry, the stale announcement is written to both caches, overwriting the newer `account_peers` entry.

---

### Impact Explanation

After the overwrite, `get_account_owner(account_id)` returns the stale `peer_id`. All subsequent `ForwardTx` routed messages for that validator account are forwarded to the wrong peer. Transactions submitted via RPC that rely on this routing path fail to reach the intended validator and are not included in blocks, matching the scope's "rejects valid transactions before block inclusion."

---

### Likelihood Explanation

The attacker must:
1. Be a connected peer (public P2P).
2. Have previously captured a valid `AnnounceAccount` for the target account from an earlier epoch (feasible by monitoring the gossip network).
3. Send any routed message to the target account to trigger the disk load (one message suffices).
4. Send a `SyncRoutingTable` containing the replayed stale announcement.

No validator keys, admin access, or special privileges are required. The divergence is permanent for the node's lifetime unless the account re-announces with a newer epoch that goes through `add_accounts` (which would populate `account_peers_broadcasted` and restore the guard).

---

### Recommendation

In `get_announce`, when loading from disk, also populate `account_peers_broadcasted`:

```rust
Ok(Some(stored_announce_account)) => {
    self.account_peers.put(account_id.clone(), stored_announce_account.clone());
    // Populate broadcasted cache too, so the epoch-freshness guard
    // in handle_sync_routing_table is not bypassed for disk-loaded entries.
    self.account_peers_broadcasted.put(account_id.clone(), stored_announce_account.clone());
    Some(stored_announce_account)
}
```

Alternatively, `handle_sync_routing_table` should consult `account_peers` (not just `account_peers_broadcasted`) when building the `last_epoch` map, or `add_accounts` should compare the incoming epoch against the existing `account_peers` entry before overwriting.

---

### Proof of Concept

1. Node A starts with a disk-stored `AnnounceAccount { account_id: "validator.near", epoch_id: E1, peer_id: P1 }`.
2. Attacker peer connects to Node A and sends a `Routed` message addressed to `"validator.near"` → triggers `get_account_owner` → disk load into `account_peers` only; `account_peers_broadcasted` remains empty.
3. Attacker sends `SyncRoutingTable { accounts: [AnnounceAccount { account_id: "validator.near", epoch_id: E0, peer_id: P_attacker, signature: <captured valid sig> }] }`.
4. `get_broadcasted_announcements` returns `{}` → `last_epoch = None`.
5. `ViewClientActor` skips the `compare_epoch_id` check; `check_signature_account_announce` passes (E0 was a valid epoch for this account).
6. `add_accounts` overwrites `account_peers["validator.near"]` with `(E0, P_attacker)`.
7. All subsequent `ForwardTx` for `"validator.near"` are routed to `P_attacker` instead of `P1`; the transactions are dropped.
