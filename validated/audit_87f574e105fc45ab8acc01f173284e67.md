### Title
Unverified CKD Response Accepted for `AppPublicKey` Variant Allows Single Participant to Forge Confidential Key Derivation Output — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_ckd` in the MPC contract contains a special-case branch that skips all cryptographic verification of the CKD response when the request uses the `AppPublicKey` variant. A single attested MPC participant (strictly below the signing threshold) can call `respond_ckd` with an arbitrary, self-crafted response, and the contract will accept and deliver it to the user without any check. This is the direct analog of the DEX self-trade bug: a special-case branch omits the critical paired operation (verification) that is always performed in the normal case (`AppPublicKeyPV`).

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`), after confirming the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the submitted response is consistent with the domain public key and the user's app public key. For `AppPublicKey`, the match arm is empty — the response content is never verified. The contract then unconditionally calls `resolve_yields_for`, delivering whatever bytes the caller submitted to every queued yield for that request: [2](#0-1) 

Compare this to `respond` (for threshold signatures), which always verifies the signature regardless of request type before resolving: [3](#0-2) 

The `respond_ckd` function requires only one attested participant to call it. There is no threshold check on the number of participants who must agree on the response. The MPC protocol enforces threshold participation off-chain, but the contract does not enforce it on-chain for `AppPublicKey` responses — and for `AppPublicKeyPV` it does so via `ckd_output_check`. For `AppPublicKey`, that enforcement is entirely absent.

### Impact Explanation

A single malicious MPC participant (below the signing threshold) can:

1. Observe a pending `request_app_private_key` call with `AppPublicKey` variant in `pending_ckd_requests`.
2. Immediately call `respond_ckd` with a self-crafted `CKDResponse` — any bytes — before honest nodes produce the real output.
3. The contract accepts the fraudulent response (no verification), removes the pending entry, and resumes all queued yields with the attacker's payload.
4. The user receives incorrect/attacker-controlled key material as their "confidential derived key."

This constitutes **unauthorized confidential key derivation output without the required participant authorization** — a single participant below threshold can unilaterally determine the CKD output, bypassing the threshold requirement entirely. If the user subsequently uses the derived key to protect assets or encrypt sensitive data, those assets or data are at risk.

**Allowed impact match:** Critical — "Unauthorized … confidential key derivation output without the required participant authorization."

### Likelihood Explanation

Any single attested MPC participant can execute this attack. The attacker does not need to collude with other participants, compromise any key material, or perform any off-chain cryptographic computation. The only requirement is being an active, attested participant — a role that is reachable without privileged assumptions (participants are registered via `submit_participant_info` and voted in). The attack is a simple on-chain transaction submitted before honest nodes respond.

### Recommendation

Apply the same response-verification discipline to `AppPublicKey` that is already applied to `AppPublicKeyPV`. If on-chain verification of the `AppPublicKey` variant is cryptographically impossible (because the scheme is privately verifiable only), the contract should document and enforce that `AppPublicKey` requests are only accepted from a quorum of participants (e.g., require threshold-many identical responses before resolving), or restrict `respond_ckd` to only accept `AppPublicKeyPV` requests where on-chain verification is possible. The empty match arm `AppPublicKey(_) => {}` should never silently skip verification for a security-critical operation.

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: AppPublicKey(some_pk)`, attaching 1 yoctonear deposit. The request is queued in `pending_ckd_requests`.
2. Malicious attested participant constructs a `CKDResponse` with arbitrary/attacker-chosen key material.
3. Malicious participant calls `respond_ckd(request, fraudulent_response)`.
4. Contract executes: `assert_caller_is_attested_participant_and_protocol_active()` passes; the `AppPublicKey` branch is empty — no `ckd_output_check` runs; `resolve_yields_for` resumes the user's yield with the fraudulent payload.
5. User's `request_app_private_key` promise resolves with the attacker-controlled `CKDResponse`. The user's confidential key derivation output is now under attacker control, without any threshold of honest participants having participated. [4](#0-3)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
