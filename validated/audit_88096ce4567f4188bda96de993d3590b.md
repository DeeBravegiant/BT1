### Title
Inactive DReps' Expiry Incorrectly Extended by `updateDormantDRepExpiry` When a Governance Proposal Is Submitted — (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Certs.hs`)

---

### Summary

The `updateDormantDRepExpiry` function applies the dormant-epoch bump to **all** registered DReps, including those whose stored `drepExpiry` is already in the past (i.e., DReps that would be expired if not for the dormant-epoch mechanism). When a governance proposal is submitted, this incorrectly extends the activity window of DReps that the formal specification considers inactive, allowing them to vote on governance proposals beyond the intended `ppDRepActivity` window and potentially enabling unauthorized governance actions.

---

### Finding Description

`updateDormantDRepExpiry` is invoked inside `updateDormantDRepExpiries` whenever a transaction carries at least one governance proposal:

```haskell
-- Certs.hs L263-267
updateDormantDRepExpiries tx currentEpoch =
  let hasProposals = not . OSet.null $ tx ^. bodyTxL . proposalProceduresTxBodyL
   in if hasProposals
        then certVStateL %~ updateDormantDRepExpiry currentEpoch
        else id
``` [1](#0-0) 

The inner function iterates over **every** DRep in `vsDRepsL` and applies:

```haskell
-- Certs.hs L322-328
updateExpiry =
  drepExpiryL
    %~ \currentExpiry ->
      let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
       in if actualExpiry < currentEpoch
            then currentExpiry   -- only skip truly-past-actual-expiry DReps
            else actualExpiry    -- bump everyone else, including stored-expired DReps
``` [2](#0-1) 

The guard `if actualExpiry < currentEpoch` only skips DReps where `drepExpiry + numDormantEpochs < currentEpoch`. It **does not** skip DReps where `drepExpiry < currentEpoch` but `drepExpiry + numDormantEpochs >= currentEpoch`. For those "borderline" DReps the stored expiry is bumped to `drepExpiry + numDormantEpochs`.

Immediately after the bump, `vsNumDormantEpochs` is reset to `EpochNo 0`:

```haskell
vState
  & vsNumDormantEpochsL .~ EpochNo 0   -- reset counter
  & vsDRepsL %~ Map.map updateExpiry   -- bump all DReps
``` [3](#0-2) 

After the reset, the actual expiry of a borderline DRep is just its stored `drepExpiry` (since `numDormantEpochs = 0`). The divergence is:

| | Stored `drepExpiry` after reset | `numDormantEpochs` | Actual expiry | Status |
|---|---|---|---|---|
| **Implementation** | `old_expiry + D ≥ currentEpoch` | 0 | `old_expiry + D` | **Active** |
| **Formal spec** | `old_expiry < currentEpoch` | 0 | `old_expiry` | **Expired** |

This divergence is explicitly acknowledged by a disabled conformance test:

```haskell
-- EpochSpec.hs L190
disableInConformanceIt "expiry is not updated for inactive DReps" $ do
``` [4](#0-3) 

The comment references formal-ledger-specifications issue #923, confirming the implementation diverges from the specification on this point.

The `vsNumDormantEpochs` field in `VState` is designed to track dormant periods so DReps are not penalised for inactivity when there is nothing to vote on:

```haskell
-- VState.hs L59-66
vsNumDormantEpochs :: !EpochNo
-- ^ Number of contiguous epochs in which there are exactly zero
-- active governance proposals to vote on. ... We need this counter in order to
-- bump DRep expiries through dormant periods when DReps do not have
-- an opportunity to vote on anything.
``` [5](#0-4) 

The bug is that the bump is applied to DReps that **did** have an opportunity to vote (their stored expiry already passed) but simply chose not to. This is the direct analog of the LivenessGuard resetting `lastLive` for inactive owners: the dormant-epoch mechanism, intended to protect genuinely active DReps from unfair expiry, is instead used to resurrect DReps that have already lapsed.

---

### Impact Explanation

After a governance proposal is submitted and `numDormantEpochs` is reset to zero, DReps whose stored expiry was already in the past are now considered active with a future stored expiry. They can:

1. Cast votes on the submitted proposal (and subsequent proposals) that are counted toward ratification thresholds.
2. Remain active for `numDormantEpochs` additional epochs beyond the `ppDRepActivity` window the protocol intends to enforce.

If an attacker controls DReps in the borderline state, they can extend their governance voting power indefinitely by engineering dormant periods followed by proposal submissions. In a scenario where enough such DReps are controlled, governance actions (protocol-parameter changes, treasury withdrawals, committee updates, hard-fork initiations) could be ratified with an effective DRep threshold lower than the protocol intends, constituting an unauthorized governance action.

**Applicable impact**: Medium — attacker-controlled transactions (governance proposals) cause DRep votes to exceed intended validation limits (`ppDRepActivity`). Escalates toward Critical if the attacker controls sufficient stake to cross ratification thresholds with the resurrected DReps.

---

### Likelihood Explanation

The conditions required are:

1. One or more DReps whose stored `drepExpiry < currentEpoch` but `drepExpiry + numDormantEpochs ≥ currentEpoch`. This arises naturally whenever there is a period of governance inactivity (no proposals) long enough for `numDormantEpochs` to accumulate past the gap between `currentEpoch` and the DRep's stored expiry.
2. Any participant submits a transaction containing a governance proposal. This requires only the `ppGovActionDeposit` deposit and is permissionless.

Both conditions are realistic and can be engineered by an unprivileged attacker. The likelihood is **High** for any deployment that experiences governance-inactive periods.

---

### Recommendation

`updateDormantDRepExpiry` should only bump DReps whose stored `drepExpiry` is still in the future (i.e., genuinely active DReps). DReps whose stored expiry has already passed should not receive the bump, even if their actual expiry (with dormant epochs) has not yet passed:

```haskell
updateExpiry =
  drepExpiryL
    %~ \currentExpiry ->
      if currentExpiry < currentEpoch
        then currentExpiry   -- do not resurrect already-lapsed DReps
        else binOpEpochNo (+) numDormantEpochs currentExpiry
```

This aligns the implementation with the formal specification (issue #923) and ensures that `ppDRepActivity` is enforced as an absolute upper bound on DRep inactivity, regardless of dormant-epoch accumulation.

---

### Proof of Concept

The disabled conformance test in `EpochSpec.hs` directly demonstrates the bug:

```
drepActivity = 2 epochs
Epoch 0: proposal submitted, DRep registered (stored expiry = epoch 2)
Epoch 3: numDormantEpochs = 1, stored expiry = 2, actual expiry = 3 (not expired)
Epoch 5: numDormantEpochs = 3, stored expiry = 2, actual expiry = 5 (not expired)
Epoch 5: new proposal submitted →
  updateDormantDRepExpiry called:
    actualExpiry = 2 + 3 = 5 ≥ 5 (currentEpoch) → bump to 5
    numDormantEpochs reset to 0
After reset: stored expiry = 5, numDormantEpochs = 0, actual expiry = 5
  Implementation: DRep active (5 ≥ 5)
  Formal spec:    DRep expired (stored expiry 2 < 5)
``` [4](#0-3) 

An attacker who controls a DRep in this state submits a governance proposal (permissionless, costs only the `ppGovActionDeposit`), triggering the bump. The DRep's stored expiry is extended to `currentEpoch`, `numDormantEpochs` is reset to 0, and the DRep is now active per the implementation. The attacker then votes on the proposal with the resurrected DRep. This process can be repeated: each time dormant epochs accumulate and a proposal is submitted, the DRep's expiry is extended again, indefinitely deferring expiry beyond the `ppDRepActivity` bound the protocol intends to enforce.

  if numDormantEpochs == EpochNo 0
