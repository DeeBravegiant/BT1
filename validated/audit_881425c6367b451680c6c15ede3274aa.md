Audit Report

## Title
Zero agETH Minted on Integer Truncation Permanently Strands Deposited Tokens — (`contracts/agETH/AGETHPoolV3.sol`)

## Summary
In `deposit(address,uint256,string)`, tokens are transferred into the contract before `agETHAmount` is computed. When `amountAfterFee * tokenToETHRate < agETHToETHrate`, integer division in `viewSwapAgETHAmountAndFee` returns 0, and the function proceeds to mint 0 agETH with no revert. The depositor's tokens are permanently unrecoverable by the depositor — they are swept to the bridger via `moveAssetsForBridging`, not refunded.

## Finding Description
The deposit flow in `contracts/agETH/AGETHPoolV3.sol` at lines 143–153:

1. `amount == 0` guard passes for any nonzero deposit.
2. `safeTransferFrom` moves tokens into the contract (line 145) **before** any amount calculation.
3. `viewSwapAgETHAmountAndFee(amount, token)` (line 147) computes `agETHAmount = amountAfterFee * tokenToETHRate / agETHToETHrate` (line 194) using plain integer division with no floor check.
4. When `amountAfterFee * tokenToETHRate < agETHToETHrate`, the result is 0.
5. `feeEarnedInToken[token] += fee` (line 149) increments by 0 when `feeBps == 0`.
6. `agETH.mint(msg.sender, 0)` (line 151) is called — no revert, no refund.

There is no `if (agETHAmount == 0) revert` guard anywhere in the deposit path. The only privileged functions that move token balances are `withdrawFees` (moves only `feeEarnedInToken[token]`) and `moveAssetsForBridging(token)` (moves `balance − feeEarnedInToken[token]` to the bridger). Neither returns tokens to the original depositor.

## Impact Explanation
**Critical — Permanent freezing / effective theft of user funds.**

Deposited tokens are irrecoverable by the depositor. They accumulate in the contract and are eventually swept to the bridger via `moveAssetsForBridging`, not returned. For 18-decimal tokens priced near 1 ETH (e.g., wstETH), truncation only occurs at sub-wei amounts (already blocked by `amount == 0`). For 6-decimal tokens (e.g., USDC at ~$3000/ETH, `tokenToETHRate ≈ 3.33e14`), any deposit below ~3153 token-wei (~$0.003) truncates to 0. For lower-value tokens or tokens whose oracle rate falls post-registration, the threshold grows. The condition is purely arithmetic and requires no privileged action.

## Likelihood Explanation
Any unprivileged caller can invoke `deposit(token, amount, referralId)` on any supported token. The only prerequisites are `amount > 0` (enforced) and the token being supported (enforced by `onlySupportedToken`). The `addSupportedToken` guard checks only `getRate() != 0` at registration time and does not bound the ratio `agETHToETHrate / tokenToETHRate`. The truncation condition is trivially reachable with dust-sized deposits on any supported token, and with larger deposits for low-value or low-decimal tokens. The loss is repeatable across many transactions.

## Recommendation
Add a zero-amount guard immediately after computing `agETHAmount`, before any state mutation:

```solidity
(uint256 agETHAmount, uint256 fee) = viewSwapAgETHAmountAndFee(amount, token);
if (agETHAmount == 0) revert InvalidAmount();  // ← add this
feeEarnedInToken[token] += fee;
agETH.mint(msg.sender, agETHAmount);
```

Alternatively, move `safeTransferFrom` **after** the amount calculation so that a revert never leaves tokens stranded.

## Proof of Concept
```solidity
// Preconditions:
//   agETHToETHrate = 1.05e18  (agETH oracle)
//   tokenToETHRate = 3.33e14  (USDC-like 6-dec token, ~$3000/ETH)
//   feeBps         = 0
//   amount         = 3000     (3000 token-wei, well above 0)
//
// fee           = 3000 * 0 / 10_000 = 0
// amountAfterFee = 3000
// agETHAmount   = 3000 * 3.33e14 / 1.05e18
//               = 9.99e17 / 1.05e18
//               = 0   (integer truncation)
//
// Result: 3000 token-wei transferred to contract, 0 agETH minted, no revert, no refund.

function testZeroMintTruncation() public {
    uint256 amount         = 3000;
    uint256 tokenToETHRate = 3.33e14;
    uint256 agETHToETHrate = 1.05e18;
    uint256 feeBps         = 0;

    uint256 fee            = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 agETHAmount    = amountAfterFee * tokenToETHRate / agETHToETHrate;

    assertEq(agETHAmount, 0); // passes — depositor loses 3000 token-wei with no agETH received
}
```

A Foundry fuzz test over `(uint256 amount, uint256 tokenToETHRate, uint256 agETHToETHrate)` with the constraint `amount * tokenToETHRate < agETHToETHrate && amount > 0` will confirm the assertion holds across the full reachable input space.
