### Title
Malicious Peer Triggers Premature NFT Offer Cancellation via `ASSERT_HEIGHT_ABSOLUTE_FAILED` TransactionAck — (`chia/wallet/wallet_state_manager.py`, `chia/wallet/transaction_record.py`)

### Summary

A single unprivileged full-node peer connected to a wallet can send a spoofed `TransactionAck(status=FAILED, error=ASSERT_HEIGHT_ABSOLUTE_FAILED)` for a spend bundle that uses `ASSERT_HEIGHT_ABSOLUTE`. Because `ASSERT_HEIGHT_ABSOLUTE_FAILED` is not in the fee-error exemption set, `remove_from_queue` enters the trade-cancellation path. Simultaneously, honest peers responding `PENDING` (the correct response for a height-locked bundle) accumulate entries in `sent_to`, pushing `len(sent_to)` to `minimum_send_attempts`. At that point `is_valid()` returns `False` and `fail_pending_offer` is called, permanently marking the NFT offer as `FAILED` before the assertion height is ever reached.

---

### Finding Description

**Step 1 — Entry point guard (`wallet_node_api.py:110-123`)**

The `transaction_ack` handler requires `matched_in_flight_send` to be `True` before processing an ack. This means the wallet must have already sent a `SendTransaction` to the peer. This guard is trivially satisfied: the wallet sends every pending transaction to all connected peers, so any connected peer receives the bundle and may respond with any `TransactionAck`. [1](#0-0) 

**Step 2 — `remove_from_queue` error-path gate (`wallet_state_manager.py:2506-2510`)**

The trade-cancellation path is entered when `send_status == FAILED`, `error is not None`, and `error not in {INVALID_FEE_LOW_FEE, INVALID_FEE_TOO_CLOSE_TO_ZERO}`. `ASSERT_HEIGHT_ABSOLUTE_FAILED` is not in that exemption set, so the malicious peer's ack enters this path. Honest peers responding `PENDING` (no error field) do **not** enter this path. [2](#0-1) 

**Step 3 — `is_valid()` counts PENDING entries toward the failure threshold (`transaction_record.py:82-92`)**

`is_valid()` returns `False` when `len(self.sent_to) >= minimum_send_attempts` and no `SUCCESS` entry and no fee-error entry exist. `PENDING` responses from honest peers are unconditionally appended to `sent_to` by `increment_sent` and count toward `len(sent_to)`, but they do not satisfy the `SUCCESS` exemption. So 5 `PENDING` entries from honest peers + 1 `FAILED(ASSERT_HEIGHT_ABSOLUTE_FAILED)` entry from the malicious peer = 6 total entries → `is_valid()` returns `False`. [3](#0-2) 

**Step 4 — `increment_sent` unconditionally appends every ack (`wallet_transaction_store.py:202-208`)**

Every ack (SUCCESS, PENDING, or FAILED) is appended to `sent_to`. When `is_valid()` returns `False` after the append, the tx is immediately marked `confirmed=True, confirmed_at_height=0` in the store. [4](#0-3) 

**Step 5 — `fail_pending_offer` is called (`wallet_state_manager.py:2530-2537`)**

After `increment_sent` writes the updated record, `remove_from_queue` re-reads the tx. Because `is_valid()` is now `False`, `fail_pending_offer` is called for every trade associated with the spend bundle's removed coins. [5](#0-4) 

`fail_pending_offer` sets the trade status to `TradeStatus.FAILED` and fires `offer_failed`: [6](#0-5) 

---

### Impact Explanation

An NFT offer whose spend bundle contains `ASSERT_HEIGHT_ABSOLUTE` (e.g., a time-locked NFT transfer) is permanently marked `FAILED` in the wallet's trade store before the assertion height is reached. The NFT ownership transfer is abandoned. The user must manually detect the failure and recreate the offer. This directly corrupts offer/trade settlement state, satisfying the High impact criterion.

---

### Likelihood Explanation

The attacker only needs to be **one** of the wallet's connected peers — a completely unprivileged position. The wallet connects to multiple peers by default. For a bundle with `ASSERT_HEIGHT_ABSOLUTE` at a future height, all honest peers legitimately return `PENDING` (the mempool stores the bundle in its pending cache). Five such `PENDING` responses plus one spoofed `FAILED(ASSERT_HEIGHT_ABSOLUTE_FAILED)` from the malicious peer is sufficient to trigger the cancellation in a single send round. No eclipse attack is required.

---

### Recommendation

1. Add `Err.ASSERT_HEIGHT_ABSOLUTE_FAILED`, `Err.ASSERT_HEIGHT_RELATIVE_FAILED`, `Err.ASSERT_SECONDS_ABSOLUTE_FAILED`, and `Err.MEMPOOL_CONFLICT` to the exemption set in both `remove_from_queue` and `is_valid()`, since all of these are transient conditions.
2. In `is_valid()`, count only `FAILED` entries (not `PENDING` entries) toward the `minimum_send_attempts` threshold, or add a separate `PENDING` exemption analogous to the fee-error exemption.
3. In `remove_from_queue`, before calling `fail_pending_offer`, verify that the error is not one that the mempool itself classifies as transient (i.e., height/time assertions).

### Proof of Concept

```python
# Precondition: wallet connected to 5 honest peers + 1 malicious peer.
# Bundle: NFT transfer with ASSERT_HEIGHT_ABSOLUTE = current_height + 100.

# Honest peers (5x): respond PENDING → sent_to grows to 5 PENDING entries.
# Malicious peer (1x): responds FAILED(ASSERT_HEIGHT_ABSOLUTE_FAILED)
#   → remove_from_queue enters error path (not in fee exemption set)
#   → increment_sent appends → sent_to now has 6 entries
#   → is_valid(): len(sent_to)=6 >= 6, no SUCCESS, no fee error → False
#   → fail_pending_offer(trade_id) called
#   → trade store: TradeStatus.FAILED

# Assert: trade is FAILED before block height reaches current_height + 100.
# Assert: NFT is still unspent on chain (transfer never happened).
```

```python
    def is_valid(self) -> bool:
        if len(self.sent_to) < minimum_send_attempts:
            # we haven't tried enough peers yet
            return True
        if any(x[1] == MempoolInclusionStatus.SUCCESS.value for x in self.sent_to):
            # we managed to push it to mempool at least once
            return True
        if any(x[2] in {Err.INVALID_FEE_LOW_FEE.name, Err.INVALID_FEE_TOO_CLOSE_TO_ZERO.name} for x in self.sent_to):
            # we tried to push it to mempool and got a fee error so it's a temporary error
            return True
        return False
```
