Audit Report

## Title
Missing Minimum Amount Out (Slippage Protection) in L2 Pool `deposit` Functions - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

## Summary
The `deposit` functions in `RSETHPoolV3` and `RSETHPoolV3ExternalBridge` compute the wrsETH amount to mint entirely from the live oracle rate at execution time, with no caller-supplied floor. This is a direct asymmetry with the L1 entry point `LRTDepositPool._beforeDeposit`, which enforces `minRSETHAmountExpected` on every deposit. A user who observes a rate at transaction-submission time has no on-chain mechanism to reject execution if the oracle updates before their transaction is mined, resulting in fewer wrsETH tokens than expected with no revert.

## Finding Description
In `RSETHPoolV3.deposit(string)` (L246–265) and `RSETHPoolV3.deposit(address,uint256,string)` (L271–293), the minted amount is computed as:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
wrsETH.mint(msg.sender, rsETHAmount);
```

`viewSwapRsETHAmountAndFee` divides by `IOracle(rsETHOracle).getRate()` — a single live call — with no floor the caller can specify. The identical pattern exists in `RSETHPoolV3ExternalBridge.deposit` (L366–384, L390–412).

By contrast, `LRTDepositPool._beforeDeposit` (L648–670) enforces:

```solidity
rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
if (rsethAmountToMint < minRSETHAmountExpected) {
    revert MinimumAmountToReceiveNotMet();
}
```

No equivalent guard exists in either L2 pool. If a Chainlink heartbeat or deviation-triggered update advances the oracle rate between transaction submission and inclusion, the user silently receives fewer wrsETH tokens than they observed when constructing the call, with no revert path available.

Note: the withdrawal side (`instantWithdrawal`) is not a symmetric concern — because the rsETH/ETH rate is monotonically increasing, a rate increase between submission and execution yields *more* assets to the withdrawer, which is favorable. The deposit direction is the only affected path.

## Impact Explanation
The deposited ETH/token is fully represented by the minted wrsETH at the updated rate — no ETH value is lost. However, the user receives fewer wrsETH tokens than the amount they observed and intended to receive when submitting the transaction. This matches the allowed Low impact: **Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The L2 pools are the primary retail entry point. Chainlink oracle heartbeats fire every 24 hours or on a 0.5% deviation trigger. During periods of elevated gas prices, user transactions can remain pending across one or more oracle updates. No adversary is required; this is a routine, non-adversarial scenario affecting ordinary depositors. Any user who submits a deposit during a period of mempool congestion spanning an oracle update is affected.

## Recommendation
Add a `minRsETHAmountExpected` parameter to both `deposit` overloads in `RSETHPoolV3` and `RSETHPoolV3ExternalBridge`, mirroring `LRTDepositPool._beforeDeposit`:

```solidity
function deposit(
    string memory referralId,
    uint256 minRsETHAmountExpected
) external payable nonReentrant whenNotPaused limitDailyMint(msg.value, ETH_IDENTIFIER) {
    // ...
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
    if (rsETHAmount < minRsETHAmountExpected) revert SlippageExceeded();
    wrsETH.mint(msg.sender, rsETHAmount);
}
```

Apply the same pattern to the token-path overload in both contracts.

## Proof of Concept
1. User queries `RSETHPoolV3.viewSwapRsETHAmountAndFee(1 ether)` off-chain; observes `rsETHAmount ≈ 0.952e18` at `getRate() = 1.05e18`.
2. User submits `deposit{value: 1 ether}("ref")` with no minimum-out parameter.
3. Before the transaction is mined, a Chainlink heartbeat updates `getRate()` to `1.06e18`.
4. Transaction executes: `rsETHAmount = 1e18 * 1e18 / 1.06e18 ≈ 0.943e18` — approximately 1% fewer tokens than expected, with no revert.
5. On L1, the equivalent call to `LRTDepositPool.depositETH` with `minRSETHAmountExpected = 0.952e18` would have reverted with `MinimumAmountToReceiveNotMet`.

**Foundry fork test plan:** Fork the L2 deployment. Call `RSETHPoolV3.deposit{value: 1 ether}("ref")`. Between the `vm.broadcast` submission and execution, use `vm.store` to advance the oracle rate by 0.5%. Assert that `wrsETH.balanceOf(user)` is less than the amount computed at the pre-update rate, and that no revert occurred. Confirm the L1 equivalent reverts under the same conditions when `minRSETHAmountExpected` is set to the pre-update amount.
