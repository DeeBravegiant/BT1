Audit Report

## Title
Permanent Freezing of ETH When Recipient Contract Reverts on Receive — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
Any smart contract whose `receive()` reverts can call `initiateWithdrawal` to queue an ETH withdrawal. Once the operator calls `unlockQueue` — which irreversibly burns the rsETH and moves ETH into `LRTWithdrawalManager` — neither `completeWithdrawal` nor `completeWithdrawalForUser` can succeed for that user. The ETH is permanently locked in the contract with no on-chain recovery path.

## Finding Description
`_transferAsset` (lines 876–879) pushes ETH to the recipient via a low-level call and hard-reverts on failure:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

`_processWithdrawalCompletion` (lines 699–738) calls `_transferAsset` as its final step. Because Solidity reverts roll back all state changes in the same transaction, a failed ETH push also undoes `delete withdrawalRequests[requestId]` (line 712), `userAssociatedNonces[asset][user].popFront()` (line 705), and `unlockedWithdrawalsCount[asset]--` (line 717). The request is left permanently in the "unlocked" state.

`completeWithdrawalForUser` (lines 192–204) passes the original `user` address — the reverting contract — as the recipient, so the operator path is equally blocked.

`sweepRemainingAssets` (lines 395–414) is gated on `hasUnlockedWithdrawals(asset) == false` (line 403), which checks `unlockedWithdrawalsCount[asset] > 0` (line 630). Because the stuck request keeps this counter non-zero, sweep is permanently blocked for the entire asset. No rescue, recover, or force-redirect function exists in the contract.

The irreversibility comes from the two-phase design: `unlockQueue` (lines 268–320) burns rsETH via `burnFrom` and redeems ETH from the unstaking vault before any withdrawal completion is attempted. Once that step executes, the rsETH is gone and the ETH is in the manager — but the only exit path for that ETH is through the reverting `_transferAsset` call.

## Impact Explanation
After `unlockQueue` executes, the user's rsETH is permanently burned and the corresponding ETH is permanently locked inside `LRTWithdrawalManager`. No on-chain function can move it without a contract upgrade. This is **permanent freezing of user funds** — a Critical impact under the allowed scope.

## Likelihood Explanation
The attacker only needs to: (1) acquire rsETH (available on DEXes), (2) deploy a contract with a reverting `receive()`, (3) call `initiateWithdrawal` from that contract, and (4) wait for the operator's routine `unlockQueue` call. No privileged access is required. The attack is repeatable and can be executed by any external actor. Multisigs, vaults, or other smart contracts that legitimately hold rsETH and initiate ETH withdrawals are also at risk if their `receive()` is not unconditionally payable.

## Recommendation
Replace the push-ETH pattern with a pull-payment pattern: store owed ETH amounts in a per-user mapping and expose a separate `claimETH()` function that lets users pull their own funds. Alternatively, wrap the `_transferAsset` call in a `try/catch` (or check-and-store pattern) so that a single user's revert stores the failed amount in a claimable mapping rather than reverting the entire completion transaction.

## Proof of Concept

```solidity
contract MaliciousUser {
    receive() external payable { revert("no ETH"); }

    function attack(
        address withdrawalManager,
        address rsETH,
        uint256 rsETHAmount,
        address ethToken
    ) external {
        // Step 1: queue ETH withdrawal — rsETH pulled into manager
        IERC20(rsETH).approve(withdrawalManager, rsETHAmount);
        ILRTWithdrawalManager(withdrawalManager)
            .initiateWithdrawal(ethToken, rsETHAmount, "");

        // Step 2 (operator, off-chain): unlockQueue()
        //   → rsETH burned, ETH moved into withdrawalManager (irreversible)

        // Step 3: both paths revert with EthTransferFailed
        // withdrawalManager.completeWithdrawal(ethToken, "");
        // withdrawalManager.completeWithdrawalForUser(ethToken, address(this), "");

        // Result:
        //   - rsETH permanently burned
        //   - ETH permanently locked in withdrawalManager
        //   - unlockedWithdrawalsCount[ETH] never reaches 0
        //   - sweepRemainingAssets() permanently blocked for ETH
    }
}
```

Foundry fork test plan: deploy `MaliciousUser`, fund it with rsETH, call `initiateWithdrawal`, impersonate the operator to call `unlockQueue`, then assert that both `completeWithdrawal` and `completeWithdrawalForUser` revert with `EthTransferFailed`, and that `hasUnlockedWithdrawals(ETH_TOKEN)` remains `true` indefinitely.
