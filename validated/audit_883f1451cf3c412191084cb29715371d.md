Audit Report

## Title
Permanent ETH Freeze via Reverting Recipient in Two-Phase Withdrawal - (File: contracts/LRTWithdrawalManager.sol)

## Summary
The withdrawal lifecycle is split across two independent transactions: `unlockQueue` (burns rsETH and moves ETH into `LRTWithdrawalManager`) and `completeWithdrawal` (delivers ETH to the user). If the recipient is a contract whose `receive()` reverts, the delivery transaction always reverts, but the rsETH burn from the prior `unlockQueue` transaction is already finalized and cannot be undone. The ETH is permanently locked in `LRTWithdrawalManager` with no admin escape hatch.

## Finding Description
**Step 1 – `unlockQueue` (operator-only, separate finalized tx):**
rsETH is burned and ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager`: [1](#0-0) 

**Step 2 – `_processWithdrawalCompletion`:**
`unlockedWithdrawalsCount[asset]` is decremented at line 717, then `_transferAsset` is called at line 734: [2](#0-1) 

**`_transferAsset` uses an uncapped low-level call:** [3](#0-2) 

If `payable(to).call{value: amount}("")` reverts, the entire `_processWithdrawalCompletion` transaction reverts — including the decrement of `unlockedWithdrawalsCount`. This means `unlockedWithdrawalsCount[asset]` remains > 0 indefinitely.

**`sweepRemainingAssets` is permanently blocked:** [4](#0-3) 

Because `unlockedWithdrawalsCount[asset] > 0`, `hasUnlockedWithdrawals(asset)` returns true and `sweepRemainingAssets` always reverts with `PendingWithdrawalsExist`. There is no other admin function to redirect or force-credit the stuck ETH.

The protocol's own comment at line 191 incorrectly dismisses this as a non-impactful gas-grief scenario: [5](#0-4) 

The permanent-freeze scenario is categorically different from gas grief and is not addressed by any guard in the codebase.

## Impact Explanation
After `unlockQueue` finalizes, the rsETH burn is irreversible and the corresponding ETH is held in `LRTWithdrawalManager`. If the beneficiary address cannot accept ETH, that ETH is permanently locked with no admin escape hatch. This is **Critical: Permanent freezing of funds**.

## Likelihood Explanation
Any contract-based depositor (multisig, smart-contract wallet, protocol-owned account) that lacks a payable `receive()` or fallback will trigger this freeze automatically once the operator runs `unlockQueue`. No special attacker capability is required — a malicious actor can trivially deploy a contract with `receive() external payable { revert(); }` and initiate a withdrawal. The condition is reachable by any unprivileged external caller through the standard `initiateWithdrawal` → `unlockQueue` → `completeWithdrawal` flow.

## Recommendation
1. **Pull-payment pattern**: Record owed ETH per user in a mapping and let them claim it separately via a dedicated `claimETH()` function. This decouples delivery failure from accounting state.
2. **Admin redirect**: Add an admin function that can reassign a stuck withdrawal's recipient to an alternate address when the primary recipient is provably unable to receive ETH.
3. **Capped gas + credit on failure**: Use `call{value: amount, gas: 2300}` and, on failure, credit the amount to a per-user claimable balance rather than reverting.

## Proof of Concept
1. Deploy `MaliciousWallet` with `receive() external payable { revert(); }`.
2. `MaliciousWallet` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned (line 305); ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager` (line 307). **This transaction is final.**
4. `MaliciousWallet` calls `completeWithdrawal(ETH_TOKEN, "")`. `_transferAsset` → `payable(MaliciousWallet).call{value: amount}("")` → `receive()` reverts → entire tx reverts. `unlockedWithdrawalsCount` is restored to its pre-call value.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, MaliciousWallet, "")`. Same revert.
6. Operator calls `sweepRemainingAssets(ETH_TOKEN)`. Reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.
7. ETH is permanently locked. rsETH is permanently burned. No recovery path exists.
