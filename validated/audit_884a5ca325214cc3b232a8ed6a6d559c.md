### Title
Single Byzantine Participant Can Permanently Block Removal of Compromised Launcher Image Hash — (`crates/contract/src/lib.rs`)

### Summary

`vote_add_launcher_hash` requires only a signing-threshold number of votes to add a launcher image hash to the TEE allowlist, but `vote_remove_launcher_hash` requires **all** participants (unanimity). A single Byzantine participant strictly below the signing threshold can permanently veto removal of a compromised launcher hash, leaving it in the allowed set indefinitely. Unlike MPC image hashes (which auto-expire after 7 days), launcher hashes have no expiry mechanism.

### Finding Description

The contract exposes two asymmetric governance functions for launcher image hashes:

**Add** — requires threshold votes: [1](#0-0) 

**Remove** — requires ALL participants (unanimity): [2](#0-1) 

The `AllowedLauncherImages` collection has no expiry mechanism, unlike `AllowedMpcDockerImages` which auto-expires old entries after `tee_upgrade_deadline_duration_seconds`: [3](#0-2) 

The documentation explicitly confirms this asymmetry: [4](#0-3) 

The compose hash used for attestation is derived from both the launcher hash and the MPC image hash: [5](#0-4) 

This means a node running a compromised launcher with the approved MPC binary produces a compose hash that is already in the allowed set and passes attestation: [6](#0-5) 

### Impact Explanation

Once a launcher hash is added (threshold votes), if the launcher is later found to be compromised (e.g., it injects malicious configuration into the MPC node, intercepts key-derivation I/O, or manipulates the node's runtime environment), the network must remove it. Because removal requires unanimity, a single participant below the signing threshold can permanently block removal. The compromised launcher hash remains in the allowed set with no expiry. Nodes running the compromised launcher continue to pass TEE attestation and participate in threshold signing, breaking the production safety invariant that only nodes running approved software may hold key shares and produce signature contributions.

### Likelihood Explanation

Software vulnerabilities in launcher images are routinely discovered post-deployment. A single participant going Byzantine (e.g., a node operator who is compromised, coerced, or simply offline/unresponsive) is a realistic condition that the threshold model is explicitly designed to tolerate for signing operations. The asymmetry — threshold to add, unanimity to remove — means the security bar for adding is lower than the bar for removing, inverting the expected risk profile for security-critical allowlist management.

### Recommendation

Apply the same threshold-based voting to removal as to addition. Alternatively, introduce a time-to-live (TTL) / expiry mechanism for launcher hashes analogous to the `tee_upgrade_deadline_duration_seconds` grace period already used for MPC image hashes, so that a stale or compromised launcher hash automatically expires when a successor is voted in.

### Proof of Concept

1. Threshold participants call `vote_add_launcher_hash(compromised_launcher_hash)` — hash is added to the allowed set.
2. A vulnerability in the launcher is discovered; the network initiates removal via `vote_remove_launcher_hash(compromised_launcher_hash)`.
3. All participants except one vote for removal. The check at line 1489 (`votes >= total_participants`) is never satisfied.
4. The compromised launcher hash remains in `AllowedLauncherImages` indefinitely with no expiry path.
5. A node running `compromised_launcher_hash` + any approved MPC image hash produces a compose hash already stored in `AllowedLauncherImage.compose_hashes`, passes `verify_launcher_compose_hash`, and continues to participate in threshold signing. [7](#0-6) [8](#0-7)

    /// Removes a launcher image hash and all its associated compose hashes.
    /// Returns `false` if the launcher hash was not found or if removal would leave the list empty.
    pub fn remove(&mut self, launcher_hash: &LauncherImageHash) -> bool {
        let would_remain = self
            .entries
            .iter()
            .filter(|e| &e.launcher_hash != launcher_hash)
            .count();
        if would_remain == 0 {
            return false;
        }
        let len_before = self.entries.len();
        self.entries.retain(|e| &e.launcher_hash != launcher_hash);
        self.entries.len() < len_before
    }

fn verify_launcher_compose_hash(
    launcher_compose_hash: &LauncherDockerComposeHash,
    allowed_hashes: &[LauncherDockerComposeHash],
) -> Result<(), VerificationError> {
    if allowed_hashes.is_empty() {
        return Err(VerificationError::Custom(
            "the allowed mpc launcher compose hashes list is empty".to_string(),
        ));
    }

    let launcher_compose_hash_is_allowed = allowed_hashes.contains(launcher_compose_hash);

    if !launcher_compose_hash_is_allowed {
        return Err(VerificationError::Custom(format!(
            "MPC launcher compose hash {:?} is not in the allowed hashes list",
            launcher_compose_hash
        )));
    }
