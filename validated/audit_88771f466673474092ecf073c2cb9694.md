### Title
`respond_verify_foreign_tx` Verifies Signature Against Caller-Supplied `payload_hash` Without Binding It to the Stored Request, Enabling Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` accepts a caller-supplied `response.payload_hash` and verifies the ECDSA signature against it, but never checks that this hash corresponds to the actual foreign-chain data for the given `request`. A single Byzantine attested participant (below signing threshold) can replay a valid threshold signature obtained from a previously resolved request to resolve any other pending `verify_foreign_transaction` request with wrong data, permanently consuming the yield.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 691–754), the contract performs signature verification as follows:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← caller-supplied

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← used directly from response, not from stored request
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The `payload_hash` fed into `verify_ecdsa_signature` is taken verbatim from `response.payload_hash` — a field the calling MPC node supplies freely. The contract never independently computes the expected hash from the stored `VerifyForeignTransactionRequest` and never asserts `response.payload_hash == expected_hash(request)`.

This is structurally different from the regular `respond` function, which derives the payload hash from the on-chain stored `request.payload` (set immutably at `sign()` time), not from the response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,           // ← from stored request, not from response
    &expected_public_key,
)
``` [2](#0-1) 

On the node side, `build_signature_request` constructs the signing payload as `foreign_tx_payload.compute_msg_hash()` — the SHA-256 of the Borsh-encoded `ForeignTxSignPayloadV1{request, values}`. This hash is request-specific (it encodes the `tx_id` and extracted values). However, the contract never re-derives this hash from the stored request to cross-check the submitted `response.payload_hash`. [3](#0-2) 

The `ForeignTxSignPayload::compute_msg_hash` function: [4](#0-3) 

**Attack path:**

1. The MPC network legitimately processes request **R1** (tx\_id = T1). The leader submits `respond_verify_foreign_tx(request=R1, response={payload_hash=H1, signature=σ1})` on-chain. This transaction is publicly visible on NEAR.
2. A single Byzantine attested participant extracts `(H1, σ1)` from the on-chain transaction data.
3. A new pending request **R2** (tx\_id = T2, different foreign transaction) exists in `pending_verify_foreign_tx_requests`.
4. The attacker calls `respond_verify_foreign_tx(request=R2, response={payload_hash=H1, signature=σ1})`.
5. The contract checks: is `σ1` a valid ECDSA signature over `H1` under the domain root key? **Yes** — it is a legitimate threshold signature.
6. `pending_requests::resolve_yields_for` resolves R2's yield with `{payload_hash=H1, signature=σ1}`.
7. The user who submitted R2 receives a response whose `payload_hash` corresponds to T1's data, not T2's. The yield is permanently consumed. [5](#0-4)

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
```

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
