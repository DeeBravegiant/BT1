### Title
Single Attested Participant Can Resolve Any User's CKD Request With Arbitrary Key Material — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_ckd` performs a cryptographic output check only for the `AppPublicKeyPV` variant. For the `AppPublicKey` (privately-verifiable) variant, **no output check is performed at all**. Any single attested participant — strictly below the signing threshold — can call `respond_ckd` with completely arbitrary `big_y` / `big_c` values and the contract will accept the response, resolve the user's yield, and deliver garbage key material to the requester. This is the direct analog of the OverlayToken burn vulnerability: a privileged role (attested participant ≡ burner) can act on **any** user's pending request without restriction and without threshold agreement.

---

### Finding Description

In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` on-chain, making it impossible to submit a wrong response. [2](#0-1) 

For `AppPublicKey`, there is **no equivalent check**. The contract immediately proceeds to `resolve_yields_for`, which drains the entire fan-out queue and delivers whatever `response` bytes were supplied. [3](#0-2) 

The `AppPublicKey` variant is the legacy "privately verifiable" mode: the user supplies only `pk1 = a·G1` (their app public key), and the MPC network is supposed to compute `big_s = H(pk, app_id)·msk`, then return `big_y = y·G1`, `big_c = big_s + pk1·y`. Verification requires knowing `a` (the app secret key), which only the user holds — so on-chain verification is impossible for this variant. The contract designers acknowledged this by adding `AppPublicKeyPV` with a publicly-verifiable pairing check, but left the legacy path completely unguarded. [4](#0-3) 

The `CKDRequest` key stored in `pending_ckd_requests` is `(app_public_key, app_id, domain_id)`. The `app_id` is derived from `(predecessor_id, derivation_path)`, so it is user-specific. However, any attested participant can construct the same `CKDRequest` key (all fields are public on-chain from the indexed receipt) and call `respond_ckd` with arbitrary `big_y`/`big_c`. [5](#0-4) 

The existing unit test `respond_ckd__should_succeed_when_response_is_valid_and_request_exists` explicitly demonstrates this: it passes `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (invalid curve points / garbage) and the contract accepts them without complaint. [6](#0-5) 

---

### Impact Explanation

A single malicious attested participant (strictly below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request on-chain (all fields are public).
2. Call `respond_ckd` with the correct `CKDRequest` key but with arbitrary `big_y` / `big_c` values.
3. The contract resolves the user's yield with the garbage response before the honest nodes can respond.
4. The user receives wrong encrypted key material. Their attempt to decrypt `big_c - a·big_y` yields a random group element, not the intended derived key.
5. The yield is consumed; the user must submit a new request (paying the deposit again), and the attacker can repeat the attack indefinitely.

This breaks the core safety invariant of the CKD protocol: that a CKD output delivered to a user must be the result of threshold-authorized MPC computation. A single participant below threshold is producing a CKD output without the required participant authorization.

**Impact**: Medium — request-lifecycle and contract execution-flow manipulation that breaks the threshold-authorization invariant for CKD responses, causing permanent corruption of individual CKD requests without relying on network-level DoS.

---

### Likelihood Explanation

- The attacker only needs to be an attested participant (a role that exists in every production epoch).
- All information needed to construct the `CKDRequest` key is publicly visible on-chain from the indexed receipt.
- The attack requires a single NEAR transaction and no off-chain coordination.
- The `AppPublicKey` variant is the legacy default mode documented in the contract README and used in production. [7](#0-6) 

---

### Recommendation

Since on-chain verification of the `AppPublicKey` variant is impossible without the user's secret `a`, the mitigation must be architectural:

1. **Deprecate `AppPublicKey` in favour of `AppPublicKeyPV`** — the publicly-verifiable variant already has a correct on-chain pairing check and should be the only accepted mode.
2. **Or require threshold-weighted submission**: require that `respond_ckd` for the `AppPublicKey` variant be called only after a threshold of participants have agreed off-chain (e.g., by requiring a threshold-signed attestation of the response as an additional argument).
3. **Or restrict `respond_ckd` to a designated leader**: only the elected leader for the current signing round may submit the response, reducing the attack surface to a single node per request.

---

### Proof of Concept

```
1. Alice calls request_app_private_key({
       derivation_path: "mykey",
       app_public_key: AppPublicKey(pk1),   // legacy variant
       domain_id: 2
   }) with 1 yoctoNEAR deposit.

2. The contract stores CKDRequest { app_public_key: AppPublicKey(pk1),
       app_id: derive_app_id("alice.near", "mykey"), domain_id: 2 }
   in pending_ckd_requests and yields.

3. Mallory (attested participant, not threshold) reads the receipt on-chain,
   reconstructs the identical CKDRequest key, and calls:

   respond_ckd(
       request = CKDRequest { app_public_key: AppPublicKey(pk1),
                              app_id: derive_app_id("alice.near","mykey"),
                              domain_id: 2 },
       response = CKDResponse { big_y: [0u8;48], big_c: [0u8;48] }
   )

4. The contract executes lines 675-676 (AppPublicKey arm → no-op),
   then resolve_yields_for delivers the garbage response to Alice's yield.

5. Alice's transaction resumes with big_y=0, big_c=0.
   Decryption: 0 - a·0 = 0 ≠ intended derived key.
   Alice's CKD request is permanently corrupted.
   Honest nodes' subsequent respond_ckd calls return RequestNotFound.

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(app_pk1, g2) = e(g1, app_pk2)`.
///
/// Point validation is fully delegated to the host: the decompression
/// functions abort execution on malformed or off-curve encodings, and
/// `bls12381_pairing_check` returns `false` when a point is outside its
/// prime-order subgroup.
pub(crate) fn app_public_key_check(app_public_key: &dtos::CKDAppPublicKeyPV) -> bool {
    let pk1 = env::bls12381_p1_decompress(&app_public_key.pk1);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);

    let pairing_input = [
        pk1.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        G1_GENERATOR_UNCOMPRESSED.as_slice(),
        pk2.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,

    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
