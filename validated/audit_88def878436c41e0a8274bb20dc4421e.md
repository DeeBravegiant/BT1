### Title
Unguarded `init` Allows Any Caller to Seize Control of the MPC Network Before Legitimate Initialization - (File: `crates/contract/src/lib.rs`)

### Summary

The `MpcContract::init` function carries the `#[init]` attribute but **not** `#[private]`. In NEAR, `#[init]` only prevents re-initialization after the first call; it does not restrict *who* may make that first call. The production deployment scripts deploy the contract binary in a separate transaction from the `init` call, creating a window in which any unprivileged account can front-run the legitimate operator and initialize the contract with attacker-controlled participants. Because `init` also calls `TeeState::with_mocked_participant_attestations`, the attacker's participants receive mock-valid TEE attestations immediately, bypassing real TEE verification and granting the attacker full governance and signing authority over the MPC network.

### Finding Description

`MpcContract::init` is declared as:

```rust
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
``` [1](#0-0) 

There is no `#[private]` attribute and no caller-identity check inside the function body. Compare with `init_running`, which is correctly guarded:

```rust
#[private]
#[init]
#[handle_result]
pub fn init_running(...)
``` [2](#0-1) 

The production deployment script deploys the contract binary **without** an init call in the same transaction:

```bash
near contract deploy "$MPC_CONTRACT_ACCOUNT" use-file "$MPC_CONTRACT_PATH" \
  without-init-call ...
``` [3](#0-2) 

`init` is then called in a subsequent transaction, creating a window during which any account can race to call `init` first.

Inside `init`, the attacker-supplied `parameters` are accepted after only a structural `validate()` check, and `TeeState::with_mocked_participant_attestations` is called unconditionally:

```rust
let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);
``` [4](#0-3) 

`with_mocked_participant_attestations` inserts a `VerifiedAttestation::Mock(MockAttestation::Valid)` entry for every participant supplied by the caller, and critically sets `account_public_key = tls_public_key`:

```rust
account_public_key: tls_public_key.clone(),
``` [5](#0-4) 

The attestation check `is_caller_an_attested_participant` later verifies that `attestation.node_id.account_public_key == signer_ed25519`:

```rust
if attestation.node_id.account_public_key != signer_ed25519 {
    return Err(AttestationCheckError::AttestationKeyMismatch);
}
``` [6](#0-5) 

Because the attacker controls the `tls_public_key` field inside `ParticipantInfo`, they can set it equal to their own NEAR account's Ed25519 public key. After `init`, the stored mock attestation will have `account_public_key = attacker_near_pk`, so `is_caller_an_attested_participant` passes for the attacker on every subsequent governance call.

### Impact Explanation

An attacker who wins the initialization race:

1. Becomes the sole participant (threshold = 1) with a mock-valid TEE attestation.
2. Can call `vote_add_domains` to add signing domains (only `assert_caller_is_signer` is checked, and the attacker is the only voter needed to reach threshold).
3. Can call `start_keygen_instance` and `vote_pk` — both gated by `assert_caller_is_attested_participant_and_protocol_active`, which the attacker passes — to generate MPC keys under their exclusive control.
4. Can call `respond` to resolve any user `sign` request with an attacker-produced signature, constituting **unauthorized threshold signature issuance**.
5. Permanently locks out the legitimate operator: NEAR's `#[init]` prevents any subsequent `init` call, and `init_running` is `#[private]` (only callable by the contract itself).

This is a **Critical** impact: unauthorized control of the MPC signing key and permanent freezing of the legitimate network's ability to initialize.

### Likelihood Explanation

The vulnerability is trivially reachable by any unprivileged NEAR account. The attacker only needs to monitor the chain for the contract deployment transaction and submit an `init` call in the next block. No special access, key material, or collusion is required. The deployment pattern (`without-init-call` followed by a separate `init` transaction) is used in both the TEE cluster scripts and the devnet tooling, confirming the window exists in all production deployment paths.

### Recommendation

Add `#[private]` to `init`, mirroring the protection already applied to `init_running`:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

Alternatively, combine contract deployment and initialization into a single atomic transaction using NEAR's `--initFunction` / `--initArgs` flags so no uninitialized window exists.

### Proof of Concept

```
// Attacker monitors NEAR for the MPC contract deployment.
// In the very next block, attacker submits:

near contract call-function as-transaction <mpc_contract_id> init \
  json-args '{
    "parameters": {
      "threshold": 1,
      "participants": {
        "next_id": 1,
        "participants": [[
          "attacker.near",
          0,
          {
            "tls_public_key": "<attacker_near_ed25519_pubkey>",
            "url": "https://attacker.example.com"
          }
        ]]
      }
    }
  }' \
  prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' \
  sign-as attacker.near network-config mainnet sign-with-keychain send

// Result:
// - Contract is now in Running state with attacker as sole participant.
// - Attacker's mock TEE attestation is valid (account_public_key == tls_public_key == attacker_near_pk).
// - Legitimate operator's subsequent `init` call fails: "State already exists".
// - Attacker calls vote_add_domains, start_keygen_instance, vote_pk to generate keys.
// - Attacker calls respond() to sign arbitrary user requests.
```
