### Title
Unvalidated `farmer_reward_address_override` in `RespondSignatures` Enables Unconditional Farmer Reward Diversion by Malicious Harvester - (File: `chia/farmer/farmer_api.py`)

### Summary
A malicious third-party harvester can set `farmer_reward_address_override` to any arbitrary puzzle hash in a `RespondSignatures` message. The farmer's `_process_respond_signatures()` accepts this override unconditionally — with no fee-quality enforcement — and uses it as the `farmer_reward_puzzle_hash` in `DeclareProofOfSpace`, permanently redirecting the farmer's 1/8 block reward (250 XCH per block) to the attacker's address.

### Finding Description

CHIP-22 introduced a voluntary fee convention: a third-party harvester may set `farmer_reward_address_override` in `NewProofOfSpace` to claim a portion of the farmer reward as a fee. The farmer is supposed to validate this against a `fee_quality` threshold derived from the proof. However, the enforcement is absent in two ways:

**Path 1 — `NewProofOfSpace.farmer_reward_address_override` (partial check, not enforced):**

In `FarmerAPI.new_proof_of_space()`, when `farmer_reward_address_override is not None`, `notify_farmer_reward_taken_by_harvester_as_fee()` is called: [1](#0-0) 

Inside `notify_farmer_reward_taken_by_harvester_as_fee()`, the fee quality is computed and compared to the harvester-supplied threshold. When the threshold is violated, the function **only logs a warning** and returns normally — it does not reject the override or abort the block-making flow: [2](#0-1) 

**Path 2 — `RespondSignatures.farmer_reward_address_override` (no check at all):**

After the farmer sends `RequestSignatures` to the harvester, the harvester responds with `RespondSignatures`, which carries a `farmer_reward_address_override: bytes32 | None` field: [3](#0-2) 

In `_process_respond_signatures()`, this field is accepted and applied with **zero fee-quality validation**: [4](#0-3) 

The resulting `farmer_reward_address` is then embedded directly into `DeclareProofOfSpace` and broadcast to the full node: [5](#0-4) 

A malicious harvester can therefore:
1. Set `farmer_reward_address_override` to its own puzzle hash in every `RespondSignatures` message.
2. Optionally set `applied_fee_threshold = 0xFFFFFFFF` in `fee_info` to pass the (non-enforced) quality check in Path 1.
3. The farmer will sign and broadcast a `DeclareProofOfSpace` with the attacker's address as the farmer reward destination on every won block.

### Impact Explanation

Every block won by the farmer pays a 1/8 coinbase reward (250 XCH on mainnet) to `farmer_reward_puzzle_hash`. By injecting an arbitrary address via `RespondSignatures.farmer_reward_address_override`, a malicious harvester permanently diverts 100% of the farmer's block rewards to its own address. This is an unauthorized payout redirection of XCH with direct, irreversible financial impact.

### Likelihood Explanation

Any operator of a third-party harvester service (CHIP-22 model) can exploit this. The farmer-harvester connection is authenticated by TLS, but the farmer intentionally connects to third-party harvesters. The attacker only needs to be a connected harvester peer — no key compromise or admin access is required. The attack is silent (no on-chain anomaly beyond the reward address change) and repeatable on every block.

### Recommendation

1. **Enforce the fee-quality threshold as a hard gate, not a log warning.** In `notify_farmer_reward_taken_by_harvester_as_fee()`, return a rejection signal when `fee_quality > fee_threshold` or when `fee_info` is absent, and propagate that rejection to abort the block-making flow.
2. **Apply the same fee-quality check in `_process_respond_signatures()`** before accepting `RespondSignatures.farmer_reward_address_override`. The check currently present only in the `NewProofOfSpace` path is missing entirely from the `RespondSignatures` path.
3. Consider capping the maximum claimable fraction of the farmer reward that a harvester may redirect, enforced locally by the farmer.

### Proof of Concept

A malicious harvester intercepts the `RequestSignatures` call and returns a `RespondSignatures` with `farmer_reward_address_override` set to the attacker's puzzle hash:

```python
# Malicious harvester intercepts RequestSignatures and injects attacker address
async def malicious_request_signatures(
    self: HarvesterAPI, request: harvester_protocol.RequestSignatures
) -> Message | None:
    result_msg = await HarvesterAPI.request_signatures(self, request)
    assert result_msg is not None

    # Inject attacker-controlled reward address — no fee quality check enforced
    attacker_ph = bytes32(b"\xde\xad" * 16)
    response = RespondSignatures.from_bytes(result_msg.data)
    hijacked = dataclasses.replace(response, farmer_reward_address_override=attacker_ph)
    return make_msg(ProtocolMessageTypes.respond_signatures, hijacked)
```

The farmer's `_process_respond_signatures()` will unconditionally accept `attacker_ph` as `farmer_reward_address` and broadcast `DeclareProofOfSpace` with that address to the full node. The full node embeds it in the block foliage, and the 250 XCH farmer reward is paid to the attacker on every block won by this harvester. [6](#0-5) [7](#0-6) [8](#0-7)

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
