### Title
Single Transaction Conversion Failure Aborts Entire Proposal Validation Batch — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

In `handle_proposal_part`, when a validator receives a `ProposalPart::Transactions` batch, a single transaction's conversion failure (e.g., a transient class manager or proof manager RPC error) causes the entire batch to be rejected and the in-progress proposal to be aborted. The same atomicity failure exists in the proposer's `add_txs_to_executor` and in `send_reproposal`. This is the direct sequencer analog of the external report's "single bad report halts whole batch" pattern.

### Finding Description

**Validator path — `handle_proposal_part`**

When a `ProposalPart::Transactions` batch arrives, every transaction is converted concurrently and the results are collected with a short-circuit combinator:

```rust
// validate_proposal.rs lines 602-616
let conversion_results =
    futures::future::join_all(txs.into_iter().map(|tx| {
        transaction_converter.convert_consensus_tx_to_internal_consensus_tx(tx)
    }))
    .await
    .into_iter()
    .collect::<Result<Vec<_>, _>>();   // ← first Err short-circuits the whole batch
let conversion_results = match conversion_results {
    Ok(results) => results,
    Err(e) => {
        return HandledProposalPart::Failed(format!(
            "Failed to convert transactions. Stopping the build of the current \
             proposal. {e:?}"
        ));
    }
};
```

`HandledProposalPart::Failed` propagates back through `validate_proposal`'s `select!` loop, which calls `batcher_abort_proposal` and returns `Err(ValidateProposalError::ProposalPartFailed)`. The validator never casts a vote for this proposal. [1](#0-0) 

**What can make `convert_consensus_tx_to_internal_consensus_tx` fail?**

For a `ConsensusTransaction::RpcTransaction` the converter calls `convert_rpc_tx_to_internal`, which:

- For **Declare** transactions calls `class_manager_client.add_class()` — an async RPC that returns `ClassManagerClientError` on any transport or timeout failure.
- For **Invoke** transactions carrying `proof_facts` calls `proof_manager_client.get_proof()` — an async RPC that returns `ProofManagerClientError` or `ProofNotFound`. [2](#0-1) [3](#0-2) 

Both are declared as failable in `TransactionConverterError`: [4](#0-3) 

**Proposer path — `add_txs_to_executor`**

The proposer uses `try_join_all`, which is even more aggressive: the first `Err` cancels all remaining futures and propagates immediately:

```rust
// block_builder.rs lines 524-527
let tx_convert_futures = self.block_txs[block_txs_start..]
    .iter()
    .map(|tx| convert_to_executable_blockifier_tx(&self.transaction_converter, tx.clone()));
let executor_input_chunk = futures::future::try_join_all(tx_convert_futures).await?;
```

`convert_to_executable_blockifier_tx` calls `convert_internal_consensus_tx_to_executable_tx`, which for Declare transactions calls both `get_sierra` and `get_executable` on the class manager:

```rust
// transaction_converter.rs lines 279-282
let (sierra, contract_class) = tokio::try_join!(
    self.get_sierra(tx.class_hash),
    self.get_executable(tx.class_hash)
)?;
```

A single `ClassManagerClientError` or `ClassNotFound` propagates as `BlockBuilderError::TransactionConverterError`, aborts the block builder, and the proposer produces no block for this round. [5](#0-4) [6](#0-5) 

**Reproposal path — `send_reproposal`**

The same pattern appears when a validator re-streams a previously validated proposal:

```rust
// sequencer_consensus_context.rs lines 1313-1321
for batch in txs.into_iter() {
    let transactions = futures::future::join_all(batch.into_iter().map(|tx| {
        transaction_converter.convert_internal_consensus_tx_to_consensus_tx(tx)
    }))
    .await
    .into_iter()
    .collect::<Result<Vec<_>, _>>()?;   // ← aborts the entire reproposal
    ...
}
```

For Declare transactions this calls `get_sierra`; for Invoke-with-proof-facts it calls `get_proof`. One failure silently kills the reproposal. [7](#0-6) 

### Impact Explanation

| Path | Trigger | Blast radius |
|---|---|---|
| Validator (`handle_proposal_part`) | Transient class-manager or proof-manager RPC failure during any batch | Entire proposal rejected; validator casts no vote. If a quorum of validators hit the same transient failure simultaneously, consensus cannot reach agreement and the block is skipped. |
| Proposer (`add_txs_to_executor`) | Transient class-manager RPC failure for any Declare tx in the batch | Block builder aborted; proposer produces no block; consensus round change. |
| Reproposal (`send_reproposal`) | Same as validator path | Reproposal silently fails; the node cannot act as proposer for the round it won. |

In all three cases the invariant broken is: **one transaction's infrastructure failure must not prevent all other transactions in the batch from being processed**. The external bug's "linear amplification" applies here too — blast radius grows with batch size and with the number of Declare or proof-bearing Invoke transactions in flight.

### Likelihood Explanation

- The class manager and proof manager are separate RPC components. Any network blip, rolling restart, or back-pressure event produces exactly the transient `ClassManagerClientError` / `ProofManagerClientError` that triggers this path.
- Declare transactions are common during contract deployment waves; Invoke-with-proof-facts transactions are expected to grow as client-side proving adoption increases.
- No malicious actor is required; normal operational variance is sufficient.

### Recommendation

1. **Validator path**: Replace the short-circuit `collect::<Result<Vec<_>, _>>()` with per-transaction error handling. Skip and log transactions whose conversion fails; send only the successfully converted subset to the batcher. If the subset is empty, treat the batch as a no-op (`HandledProposalPart::Continue`) rather than aborting the proposal.

2. **Proposer path**: Replace `try_join_all` with `join_all` and filter out conversion errors per transaction, logging each failure. Transactions that cannot be converted should be dropped from the executor input (and ideally evicted from the mempool) rather than aborting the entire block build.

3. **Reproposal path**: Apply the same per-batch, per-transaction error isolation in `send_reproposal`; a single missing Sierra class or proof should not prevent the remaining batches from being streamed.

### Proof of Concept

**Scenario — transient class-manager failure halts validator consensus**

1. Proposer builds a block containing N transactions: `[invoke_0, declare_1, invoke_2, …, invoke_N-1]`.
2. Proposer streams `ProposalPart::Transactions([invoke_0, declare_1, invoke_2, …])` to all validators.
3. Each validator enters `handle_proposal_part` and calls `join_all` over the batch.
4. At `i = 1` (`declare_1`), `convert_consensus_tx_to_internal_consensus_tx` calls `class_manager_client.add_class(sierra)`. The class manager is momentarily overloaded and returns `ClassManagerClientError::CommunicationFailure`.
5. `collect::<Result<Vec<_>, _>>()` short-circuits on the first `Err`; the remaining `invoke_2 … invoke_N-1` conversions are discarded.
6. `HandledProposalPart::Failed(…)` is returned; `validate_proposal` calls `batcher_abort_proposal` and returns `Err(ValidateProposalError::ProposalPartFailed)`.
7. Every validator that hit the same transient failure casts no vote. If this is a majority, consensus cannot reach quorum for this round.
8. The round times out; a new proposer is elected. The block is delayed by one full consensus round.
9. The error log contains only "Failed to convert transactions" with no indication of which transaction caused the failure or that the remaining N-1 transactions were perfectly valid.
