Audit Report

## Title
Missing Token Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Massive Under-Minting for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

## Summary

`viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes rsETH output by multiplying the raw token amount (in its native decimal precision) by the oracle rate and dividing by the rsETH/ETH rate. Because both oracle rates are 1e18-scaled and cancel each other out, the result is denominated in the token's native decimals rather than 18 decimals. For tokens with fewer than 18 decimals (e.g., USDC at 6 decimals), the minted wrsETH is `10^(18 - tokenDecimals)` times smaller than it should be, causing depositors to permanently lose virtually all deposited value. The same bug is present identically in `RSETHPoolV3.sol`, `RSETHPool.sol`, `RSETHPoolV3ExternalBridge.sol`, and `AGETHPoolV3.sol`.

## Finding Description

In `RSETHPoolV3WithNativeChainBridge.sol`, the token-deposit calculation is:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

Both `tokenToETHRate` and `rsETHToETHrate` are 1e18-scaled values. `ChainlinkOracleForRSETHPoolCollateral.getRate()` explicitly normalizes the Chainlink price to 1e18 regardless of the feed's native decimals:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [2](#0-1) 

Because both rates are 1e18-scaled and divide each other out, the formula reduces to approximately `rsETHAmount ≈ amountAfterFee` (in the token's native decimal units). For an 18-decimal token this is correct. For a 6-decimal token, `amountAfterFee` is in units of 1e6, so the minted rsETH is `1e12` times too small.

Contrast with the ETH deposit path, which correctly applies `* 1e18` to normalize the 18-decimal ETH amount:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [3](#0-2) 

The token deposit path never applies the equivalent normalization. The same pattern is confirmed in all other affected contracts: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

The exploit path is:
1. Admin/timelock adds a non-18-decimal token (e.g., USDC) via `addSupportedToken`. No decimal check exists in this function.
2. Any user calls `deposit(token, amount, referralId)`.
3. `safeTransferFrom` pulls the full token amount from the user into the pool.
4. `viewSwapRsETHAmountAndFee` returns a dust-level rsETH amount due to missing normalization.
5. `wrsETH.mint(msg.sender, rsETHAmount)` mints negligible wrsETH to the user.
6. The deposited tokens accumulate in the pool and can be extracted by `BRIDGER_ROLE` via `moveAssetsForBridging` or `bridgeTokens`. [8](#0-7) 

No existing guard checks token decimals or validates that the computed rsETH amount is reasonable relative to the deposited value. The `limitDailyMint` modifier uses the same broken calculation, so it does not prevent the loss.

## Impact Explanation

**Critical — Direct permanent loss of user funds.** The deposited tokens are irrecoverably transferred into the pool while the user receives a negligible dust amount of wrsETH. The discrepancy is `10^(18 - tokenDecimals)` — for USDC this is `10^12`. A user depositing 1,000 USDC receives approximately 285,714 wei of wrsETH (~`2.86e-13` rsETH in human terms) instead of ~`2.85e17` (~0.285 rsETH). The deposited USDC accrues to the pool balance and is accessible to the `BRIDGER_ROLE`, constituting effective theft of user funds. This matches the allowed Critical impact: "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation

The `deposit(address token, uint256 amount, string referralId)` function is publicly callable with no access restriction beyond the token being supported. The only prerequisite is that a non-18-decimal token is added to `supportedTokenOracle` by the timelock. The `addSupportedToken` function imposes no decimal restriction. USDC and USDT are the most common stablecoin collateral types for multi-asset LRT pools and are the natural candidates for this deposit path. Any user who deposits such a token after it is listed triggers the loss immediately and irreversibly on every deposit.

## Recommendation

Normalize `amountAfterFee` to 18 decimals before computing the output amount in all affected contracts:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

uint256 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * (10 ** (18 - tokenDecimals));
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the same fix to `RSETHPoolV3.sol`, `RSETHPool.sol`, `RSETHPoolV3ExternalBridge.sol`, and `AGETHPoolV3.sol`. Also fix the inverse path `viewSwapAssetToPremintedRsETH`, which has the symmetric error (over-paying tokens for rsETH).

## Proof of Concept

**Setup:**
- Token: USDC (6 decimals), ETH price = $3,333 → `tokenToETHRate = 3e14` (0.0003 ETH per USDC, 1e18-scaled)
- rsETH/ETH rate: `rsETHToETHrate = 1.05e18`
- User deposits 1,000 USDC → `amount = 1_000e6 = 1e9`, fee = 0

**Buggy calculation (current code):**
```
rsETHAmount = 1e9 * 3e14 / 1.05e18
            = 3e23 / 1.05e18
            ≈ 285,714 wei of wrsETH   // ≈ 2.86e-13 rsETH — essentially zero
```

**Correct calculation:**
```
normalizedAmount = 1e9 * 1e12 = 1e21   // scale 6-decimal to 18-decimal
rsETHAmount = 1e21 * 3e14 / 1.05e18
            = 3e35 / 1.05e18
            ≈ 2.857e17                  // ≈ 0.286 rsETH ✓
```

**Foundry test plan:**
```solidity
function test_usdc_deposit_undermints() public {
    // Deploy pool with mock USDC oracle returning 3e14 and rsETH oracle returning 1.05e18
    // Add USDC (6 decimals) as supported token
    // User approves and calls deposit(usdc, 1_000e6, "")
    // Assert wrsETH.balanceOf(user) ≈ 285_714 (dust)
    // Assert usdc.balanceOf(pool) == 1_000e6 (full amount locked)
    // Assert wrsETH.balanceOf(user) << 2.857e17 (expected correct amount)
}
```
