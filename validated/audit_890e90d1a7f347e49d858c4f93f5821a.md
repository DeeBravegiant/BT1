### Title
Missing On-Chain Cryptographic Verification of CKD Response for `AppPublicKey` Variant Allows Malicious Leader to Deliver Arbitrary Derived-Key Material - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` in the MPC contract performs a BLS12-381 pairing-based output check (`ckd_output_check`) only when the pending request carries an `AppPublicKeyPV` key. When the request carries the standard `AppPublicKey` variant — the default and most widely used path — the match arm is an empty block and the contract accepts any `CKDResponse` without any cryptographic verification. A single Byzantine MPC participant that is elected leader for a CKD round can therefore submit an arbitrary `(big_y, big_c)` pair, causing the user to receive a derived-key ciphertext that does not correspond to the MPC network's master secret.

### Finding Description

**Root cause — empty match arm in `respond_ckd`:**

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk, app_id), msk_pk)` via `ckd_output_check`: [2](#0-1) 

For `AppPublicKey`, no equivalent check exists. The response is serialised and forwarded to the waiting yield-resume callback unconditionally: [3](#0-2) 

**The `AppPublicKey` variant is the default production path.** The CLI README, e2e tests, and the existing unit test all use it: [4](#0-3) 

The unit test passes a completely fabricated response (`big_y: [1u8;48]`, `big_c: [2u8;48]`) and the call succeeds — confirming that any byte string is accepted.

**Attacker-controlled entry path:**

1. A Byzantine participant is elected leader for a `CKDAppPublicKey::AppPublicKey` request (leader election is deterministic from the request ID, so the attacker can predict or wait for their turn).
2. The leader participates in the off-chain BLS12-381 CKD threshold protocol with the other nodes, obtaining the correct `(big_y, big_c)`.
3. Instead of submitting the correct result, the leader calls `respond_ckd` with an arbitrary `CKDResponse`.
4. `assert_caller_is_attested_participant_and_protocol_active` passes (the leader is a legitimate attested participant).
5. The empty `AppPublicKey` arm is taken; no pairing check is performed.
6. `resolve_yields_for` resumes the user's yield with the forged ciphertext. [5](#0-4) 

The caller-authentication guard (`assert_caller_is_attested_participant_and_protocol_active`) only verifies that the signer is a registered, attested participant — it does not verify that the submitted cryptographic material is consistent with the MPC master public key: [6](#0-5) 

### Impact Explanation

The CKD feature is the mechanism by which NEAR accounts obtain application-specific private keys derived from the MPC network's master secret. A user who receives a forged `(big_y, big_c)` pair will compute an incorrect derived key scalar. Any wallet, credential, or on-chain asset controlled by that derived key becomes inaccessible to the user. Because the `AppPublicKey` variant is the standard (non-publicly-verifiable) path and the forged response is indistinguishable from a legitimate one at the contract level, the user has no on-chain signal that the derivation was corrupted.

This maps to **Medium** impact: request-lifecycle and contract execution-flow manipulation that breaks the production safety invariant (CKD responses must be cryptographically bound to the MPC master key) without requiring network-level DoS or operator misconfiguration. A single Byzantine participant strictly below the signing threshold is sufficient.

### Likelihood Explanation

Any participant can be elected leader for a given CKD request. Leader election is deterministic from the request ID, so a patient adversary can wait for a request whose leader slot falls to their node, or can target specific users whose `app_id` hashes map to their leader slot. No collusion with other participants is required. The attacker is already a legitimate, attested participant — the only additional step is substituting the correct response with an arbitrary one before calling `respond_ckd`. **Likelihood: Moderate.**

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. Because the contract does not hold the user's private scalar `a`, a direct pairing check is not possible for this variant. The recommended mitigations are:

1. **Deprecate `AppPublicKey` for production CKD requests** and require callers to use `AppPublicKeyPV`, which supports full on-chain verification via `ckd_output_check`. The `AppPublicKeyPV` path already exists and is verified: [7](#0-6) 

2. **If `AppPublicKey` must be retained**, require the leader to also supply a zero-knowledge proof of correct computation (e.g., a Schnorr proof that `big_c - big_y·a = H(pk,app_id)·msk` for the known network public key), or require a threshold of nodes to independently submit and agree on the response before it is accepted.

3. As a defence-in-depth measure, log a warning when `AppPublicKey` is used so that monitoring can detect unexpected usage patterns.

### Proof of Concept

The existing unit test already demonstrates the bypass — it passes a completely fabricated response and the call succeeds:

```rust

    big_c: dtos::Bls12381G1PublicKey([2u8; 48]),  // arbitrary garbage

with_active_participant_and_attested_context(&contract);
match contract.respond_ckd(ckd_request.clone(), response.clone()) {
    Ok(_) => { /* accepted without any cryptographic check */ }
    Err(_) => panic!("respond_ckd should not fail"),
}
``` [8](#0-7) 

A production exploit follows the same path: an attested participant calls `respond_ckd` with `CKDAppPublicKey::AppPublicKey` and any `big_y`/`big_c` byte strings. The contract resolves the pending yield with the forged ciphertext, and the user's application receives and attempts to use the corrupted derived-key material.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
