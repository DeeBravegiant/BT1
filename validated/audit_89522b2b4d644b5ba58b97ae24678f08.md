### Title
Stale-Pending Fallback Hardcodes `l1_da_mode=Calldata`, Causing Wrong Fee Estimates on Blob-DA Networks — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

The stale-pending fallback in `read_pending_data` constructs a synthetic `DeprecatedPendingBlock`. Because `PendingBlockOrDeprecated::Deprecated(_)` unconditionally returns `L1DataAvailabilityMode::Calldata` from `l1_da_mode()`, any call to `estimate_fee(Pending)`, `starknet_call(Pending)`, or `simulate_transactions(Pending)` during the stale window executes with `use_kzg_da = false`, even when the network operates in Blob DA mode. The result is an authoritative-looking wrong fee estimate returned to the caller.

---

### Finding Description

**Stale fallback path** — `read_pending_data` returns the live pending data only when `pending_data.block.parent_block_hash() == latest_header.block_hash`. When a new block has been committed but the pending-data cache has not yet been refreshed, the condition is false and the fallback branch fires: [1](#0-0) 

The fallback copies `l1_gas_price`, `timestamp`, `sequencer_address`, and `starknet_version` from the latest committed block header, but wraps everything in `PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock { ..Default::default() })`. `DeprecatedPendingBlock` has no `l1_da_mode` field: [2](#0-1) 

The `l1_da_mode()` accessor for the `Deprecated` variant is unconditionally hardcoded to `Calldata`: [3](#0-2) 

**Propagation to execution** — `estimate_fee(Pending)` calls `client_pending_data_to_execution_pending_data`, which faithfully copies `l1_da_mode` from the block accessor: [4](#0-3) 

The resulting `ExecutionPendingData.l1_da_mode = Calldata` is then consumed by `prepare_block_context_inner`: [5](#0-4) 

With `DONT_IGNORE_L1_DA_MODE` (= `override_kzg_da_to_false = false`), the flag is not overridden, so: [6](#0-5) 

`l1_da_mode.is_use_kzg_da()` returns `false` for `Calldata`, so `use_kzg_da = false` is placed into `BlockInfo` and used for the entire fee calculation. On a Blob-DA network the correct value is `true`.

The same path is taken by `call` (line 925) and `simulate_transactions` (line 1115): [7](#0-6) [8](#0-7) 

The latest committed block's `l1_da_mode` is available at `latest_header.block_header_without_hash.l1_da_mode` but is never read in the fallback branch.

---

### Impact Explanation

`estimate_fee(Pending)` and `simulate_transactions(Pending)` return fee estimates computed with `use_kzg_da = false` (Calldata cost model) when the network is in Blob DA mode. The DA cost component of the fee is materially different between the two modes (Blob uses KZG commitments; Calldata charges per byte of calldata). The returned value is authoritative-looking — there is no error, no warning, and no indication the DA mode is wrong. This matches the **High** impact category: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

The stale window is a **normal operational state** that occurs on every block transition: the block is committed to storage before the pending-data cache is refreshed. Any unprivileged user who calls `estimate_fee(Pending)` or `starknet_call(Pending)` during this window receives the wrong result. On a Blob-DA network (current Starknet mainnet), this window is always present and the wrong DA mode is always `Calldata`. The window is short (typically sub-second), but it is deterministic and repeatable on every block.

---

### Recommendation

In the stale fallback branch, read `l1_da_mode` from the latest committed block header and use `PendingBlock` (not `DeprecatedPendingBlock`) so the field is preserved:

```rust
// In the else branch of read_pending_data:
block: PendingBlockOrDeprecated::Current(PendingBlock {
    parent_block_hash: latest_header.block_hash,
    l1_gas_price: latest_header.block_header_without_hash.l1_gas_price,
    l1_data_gas_price: latest_header.block_header_without_hash.l1_data_gas_price,
    l2_gas_price: latest_header.block_header_without_hash.l2_gas_price,
    timestamp: latest_header.block_header_without_hash.timestamp,
    sequencer_address: latest_header.block_header_without_hash.sequencer,
    starknet_version: latest_header.block_header_without_hash.starknet_version.to_string(),
    l1_da_mode: latest_header.block_header_without_hash.l1_da_mode, // ← key fix
    ..Default::default()
}),
```

### Proof of Concept

1. Construct a storage with one committed block whose header has `l1_da_mode = Blob`.
2. Set the pending-data cache to a `PendingBlock` with `parent_block_hash` pointing to a **different** (older) block hash, so `read_pending_data` takes the stale fallback branch.
3. Call `estimate_fee(Pending)` with a transaction that has non-zero state-diff output.
4. Observe that the returned `data_gas` component is zero (Calldata path) instead of non-zero (Blob/KZG path).
5. Alternatively, inspect `BlockInfo.use_kzg_da` inside the spawned blocking task — it will be `false` despite the committed block advertising `Blob`.

The exact wrong value: `FeeEstimation.data_gas_consumed` and `data_gas_price` will reflect the Calldata cost model (`use_kzg_da = false`), undercharging data-heavy transactions relative to what the sequencer will actually charge when the block is built.
