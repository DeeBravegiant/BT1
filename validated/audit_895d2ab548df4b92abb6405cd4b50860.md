### Title
Farmer Does Not Validate `farmer_reward_puzzle_hash` in `request_signed_values` Before Signing — (`chia/farmer/farmer_api.py`)

### Summary

`FarmerAPI.request_signed_values` accepts a `RequestSignedValues` message from any connected full node and blindly forwards the full-node-supplied `foliage_block_data_hash` to the harvester for signing, then co-signs it itself, without ever verifying that the hash corresponds to a `FoliageBlockData` whose `farmer_reward_puzzle_hash` matches the farmer's configured `farmer_target`. A malicious full node can exploit this to obtain a valid farmer+harvester co-signature over a foliage block that redirects the block reward to an attacker-controlled address.

---

### Finding Description

**Entry point:** `FarmerAPI.request_signed_values` in `chia/farmer/farmer_api.py`.

The only guard in `request_signed_values` is a check that the `quality_string` is known:

```python
# chia/farmer/farmer_api.py:724-726
if full_node_request.quality_string not in self.farmer.quality_str_to_identifiers:
    self.farmer.log.error(...)
    return None
```

After that, the function unconditionally wraps the full-node-supplied `foliage_block_data` (if present) into `SignatureRequestSourceData` and passes `foliage_block_data_hash` — taken verbatim from the attacker's message — as the message to sign:

```python
# chia/farmer/farmer_api.py:749-756
request = harvester_protocol.RequestSignatures(
    plot_identifier,
    challenge_hash,
    sp_hash,
    [full_node_request.foliage_block_data_hash, full_node_request.foliage_transaction_block_hash],
    message_data=message_data,
    rc_block_unfinished=full_node_request.rc_block_unfinished,
)
```

The harvester then signs whatever hashes are in `request.messages` without any content inspection:

```python
# chia/harvester/harvester_api.py:511-513
for message in request.messages:
    signature: G2Element = AugSchemeMPL.sign(local_sk, message, agg_pk)
    message_signatures.append((message, signature))
```

Back in `_process_respond_signatures`, the farmer also co-signs the same hash:

```python
# chia/farmer/farmer_api.py:959-960
foliage_sig_farmer = AugSchemeMPL.sign(sk, foliage_block_data_hash, agg_pk)
foliage_transaction_block_sig_farmer = AugSchemeMPL.sign(sk, foliage_transaction_block_hash, agg_pk)
```

At no point does the farmer check:
1. That `foliage_block_data.farmer_reward_puzzle_hash == self.farmer.farmer_target`
2. That `foliage_block_data_hash == foliage_block_data.get_hash()` (hash/content consistency)

The test suite itself acknowledges this gap: the check that `farmer_reward_puzzle_hash` matches the expected address is performed only inside a test-only interceptor in `test_third_party_harvesters.py`, not in the production handler:

```python
# chia/_tests/farmer_harvester/test_third_party_harvesters.py:240-244
assert (
    request.foliage_block_data.farmer_reward_puzzle_hash == farmer_reward_address
    or request.foliage_block_data.farmer_reward_puzzle_hash
    == bytes32(full_node_2.constants.GENESIS_PRE_FARM_FARMER_PUZZLE_HASH)
)
```

---

### Impact Explanation

A malicious full node that the farmer is connected to can:

1. Receive a legitimate `DeclareProofOfSpace` from the farmer (with a valid proof and the farmer's real `farmer_puzzle_hash`).
2. Build a candidate `UnfinishedBlock` internally with an **attacker-controlled** `farmer_reward_puzzle_hash` in `FoliageBlockData`.
3. Send `RequestSignedValues` to the farmer with `foliage_block_data_hash = attacker_foliage.get_hash()` and a valid `quality_string`.
4. Receive back a `SignedValues` containing a valid BLS aggregate signature (farmer + harvester + taproot) over the attacker-controlled foliage hash.
5. Propagate the finished block to the network. The block passes signature verification at every honest full node because the signature is genuinely produced by the plot's key pair — only the reward address is wrong.

The farmer's block reward (2 XCH per block) is permanently diverted to the attacker. The farmer has no indication anything went wrong.

---

### Likelihood Explanation

- The farmer connects to full nodes it does not fully control (public nodes, third-party infrastructure). Any such node is a valid attacker.
- The `quality_string` precondition is trivially satisfied: the malicious full node simply waits for the farmer to send a `DeclareProofOfSpace` with a winning proof, which it receives before sending `RequestSignedValues`.
- No leaked keys, broken crypto, or phishing is required.
- The attack is silent and leaves no error in the farmer's logs.

---

### Recommendation

In `FarmerAPI.request_signed_values`, before forwarding to the harvester, add:

1. **Reward address check:** If `foliage_block_data` is present, assert `foliage_block_data.farmer_reward_puzzle_hash == self.farmer.farmer_target`.
2. **Hash consistency check:** Assert `foliage_block_data.get_hash() == full_node_request.foliage_block_data_hash`.
3. **Fallback for hash-only requests:** When `foliage_block_data` is `None`, the farmer cannot verify the content of what it is signing. Consider requiring `foliage_block_data` to always be present when signing block foliage, or maintaining a local record of the expected foliage hash derived from the farmer's own `DeclareProofOfSpace`.

### Proof of Concept

```python
# Unit test sketch — no network required
async def test_malicious_full_node_redirects_reward():
    # Setup: farmer with farmer_target = FARMER_PH
    # Attacker controls a full node peer

    attacker_ph = bytes32(b"\xaa" * 32)

    # Build attacker-controlled FoliageBlockData with attacker's reward address
    attacker_foliage = FoliageBlockData(
        unfinished_reward_block_hash=...,  # any valid hash
        pool_target=PoolTarget(POOL_PH, uint32(0)),
        pool_signature=None,
        farmer_reward_puzzle_hash=attacker_ph,  # <-- attacker address
        extension_data=bytes32.zeros,
    )

    # Craft RequestSignedValues with attacker foliage hash
    malicious_request = farmer_protocol.RequestSignedValues(
        quality_string=valid_quality_string,  # from a real DeclareProofOfSpace
        foliage_block_data_hash=attacker_foliage.get_hash(),
        foliage_transaction_block_hash=bytes32.zeros,
        foliage_block_data=attacker_foliage,
    )

    # Call production handler directly
    result = await farmer_api.request_signed_values(malicious_request)

    # Production code returns SignedValues — no rejection
    assert result is not None  # PASSES — farmer signed attacker foliage
    # Expected: result should be None (farmer should reject mismatched reward address)
```
