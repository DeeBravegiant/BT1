### Title
Fee-on-Transfer Token Accounting Divergence Breaks EVM Vault Backing — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer` records and emits the caller-supplied `amount` parameter rather than the actual tokens received by the vault. For fee-on-transfer ERC-20 tokens, the vault receives `amount − fee` while the cross-chain event broadcasts `amount`, causing NEAR to mint more wrapped tokens than are backed by real collateral on EVM.

### Finding Description
In `OmniBridge.sol`, `initTransfer` performs a `safeTransferFrom` for the caller-supplied `amount`, then unconditionally emits `InitTransfer` with that same `amount`: [1](#0-0) 

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← vault receives amount − transfer_fee
);
``` [2](#0-1) 

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,         // ← event broadcasts full amount, not actual received
    ...
);
```

There is no balance-before / balance-after check. The bridge never measures what it actually received. The function accepts any ERC-20 address with no token allowlist guard: [3](#0-2) 

The NEAR bridge processes the proof of this event and credits the full `amount` to the recipient, minting wrapped tokens: [4](#0-3) 

The `ft_on_transfer` callback on NEAR receives the amount from the prover result, which is derived from the EVM event — the inflated figure.

### Impact Explanation
Every `initTransfer` call with a fee-on-transfer token creates a permanent shortfall: the EVM vault holds `amount − fee` tokens while NEAR has minted `amount` wrapped tokens. The deficit accumulates with each transfer. When users later bridge back from NEAR to EVM, the vault cannot satisfy all redemptions — the last `Σfee` worth of wrapped tokens are permanently unbacked. This is an irreversible balance-accounting divergence that breaks the bridge's backing guarantee and can result in frozen redemptions for the last redeemers.

Impact class: **High** — "Asset-identity, token-mapping, decimals, fee-routing, refund, or balance-accounting divergence that breaks backing guarantees or sends value to the wrong party."

### Likelihood Explanation
Fee-on-transfer ERC-20 tokens exist on mainnet (e.g., tokens with deflationary mechanics, reflection tokens). The `initTransfer` function imposes no token allowlist, no registration check, and no balance-delta verification. Any unprivileged user can call it with any ERC-20 address. The attacker does not need any special role; they only need to hold a fee-on-transfer token and call the public `initTransfer` entrypoint.

### Recommendation
Replace the fixed-`amount` transfer with a balance-delta pattern:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "ZeroReceived");
uint128 bridgedAmount = uint128(actualReceived); // use this in the event
```

Emit `bridgedAmount` (the actual received amount) in the `InitTransfer` event instead of the caller-supplied `amount`. Alternatively, maintain an explicit token allowlist and reject any token that is known to be fee-on-transfer or rebasing.

### Proof of Concept

1. Deploy or use an existing fee-on-transfer ERC-20 token `FOT` that deducts 5% on every transfer.
2. Approve `OmniBridge` to spend 1000 `FOT`.
3. Call `OmniBridge.initTransfer(FOT, 1000, 0, 0, "alice.near", "")`.
4. Inside the call, `safeTransferFrom(msg.sender, address(this), 1000)` executes; the vault receives only **950** `FOT` (5% fee deducted by the token).
5. The `InitTransfer` event is emitted with `amount = 1000`.
6. A relayer submits the proof to NEAR; the NEAR bridge mints **1000** wrapped `FOT` to `alice.near`.
7. The EVM vault now holds 950 `FOT` but 1000 wrapped tokens exist on NEAR — a 50-token shortfall.
8. Repeat N times; the shortfall grows by 50 per iteration.
9. When the 20th user tries to redeem 1000 wrapped `FOT` back to EVM, the vault is empty and the redemption is permanently frozen. [5](#0-4)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
