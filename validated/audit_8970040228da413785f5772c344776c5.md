Audit Report

## Title
Unpermissioned `sendFunds()` Enables DoS on `depositETH()` via Deposit Limit Inflation — (`contracts/FeeReceiver.sol`)

## Summary
`FeeReceiver.sendFunds()` has no access control, allowing any caller to flush accumulated MEV rewards into `LRTDepositPool`. Because `getETHDistributionData()` uses the raw `address(this).balance` and `_checkIfDepositAmountExceedesCurrentLimit()` for ETH checks `totalAssetDeposits > depositLimit` (with `msg.value` already counted in the balance), an attacker can push the pool's ETH accounting over its deposit limit, causing every subsequent `depositETH()` call to revert with `MaximumDepositLimitReached` until an admin intervenes.

## Finding Description

**Root cause — `FeeReceiver.sendFunds()` is unguarded:** [1](#0-0) 

The function is `external` with no role modifier. Any EOA or contract can call it at any time, immediately transferring the full `FeeReceiver` balance to `LRTDepositPool` via `receiveFromRewardReceiver()`.

**`receiveFromRewardReceiver()` is also unguarded:** [2](#0-1) 

No caller restriction; any address can push ETH into the pool through this function.

**`getETHDistributionData()` uses raw contract balance:** [3](#0-2) 

MEV rewards transferred in via `sendFunds()` immediately inflate `ethLyingInDepositPool`, which feeds directly into `getTotalAssetDeposits(ETH)`.

**`_checkIfDepositAmountExceedesCurrentLimit()` for ETH omits `amount`:** [4](#0-3) 

For ETH, the check is `totalAssetDeposits > depositLimit` rather than `totalAssetDeposits + amount > depositLimit`. This is by design — `depositETH()` is `payable` so `msg.value` is already in `address(this).balance` before `_beforeDeposit` runs. The consequence is that if MEV rewards have already pushed `totalAssetDeposits` above the limit, the condition is `true` before any user ETH is added, and **every** `depositETH()` call reverts.

**Exploit path:**
1. Pool is near its ETH deposit limit (e.g., limit = 100 ETH, current TVL = 99 ETH).
2. `FeeReceiver` has accumulated ≥ 1 ETH in MEV rewards.
3. Attacker (any EOA) calls `feeReceiver.sendFunds()` — no role required.
4. MEV rewards are transferred to `LRTDepositPool`; `address(this).balance` now exceeds 100 ETH.
5. `getTotalAssetDeposits(ETH_TOKEN)` returns > 100 ETH.
6. Any user calling `depositETH{value: X}()` triggers `_checkIfDepositAmountExceedesCurrentLimit`, which evaluates `totalAssetDeposits > depositLimit` as `true` (the user's `msg.value` is already included in the balance, but the check was already failing before it was added).
7. Every `depositETH()` call reverts with `MaximumDepositLimitReached`.

The protocol's own comment acknowledges the accounting boundary: [5](#0-4) 

Rewards in `FeeReceiver` are excluded from TVL until `sendFunds()` moves them. An attacker exploits this boundary by triggering the transfer at a moment when the pool is near its limit.

## Impact Explanation
**Medium — Temporary freezing of funds (new ETH deposits).**

Existing user funds are not at risk; rsETH holders can still redeem. However, all new `depositETH()` calls revert until an admin either raises `depositLimitByAsset[ETH]` (via `lrtConfig`) or transfers ETH out of the pool to node delegators via `transferETHToNodeDelegator()`. This constitutes a temporary but complete DoS on the primary ETH deposit path. The impact is bounded to new deposits and is recoverable by admin action, placing it squarely in the Medium: Temporary freezing of funds category.

## Likelihood Explanation
Moderate. `sendFunds()` is a public function with no guard. A griever monitoring the mempool can call it whenever `FeeReceiver.balance` is large enough to push the pool over its limit. No capital is required by the attacker; the MEV rewards already held by `FeeReceiver` are the weapon. The attack is repeatable: after an admin raises the limit, the attacker can wait for rewards to accumulate again and repeat.

## Recommendation
1. **Add access control to `sendFunds()`** — restrict to `MANAGER` or a dedicated keeper role:
   ```solidity
   function sendFunds() external onlyRole(LRTConstants.MANAGER) { ... }
   ```
2. **Add a caller whitelist to `receiveFromRewardReceiver()`** — only the registered `FeeReceiver` address should be able to call it, preventing direct ETH injection from arbitrary callers.
3. **Exclude MEV rewards from the deposit-limit check** — track user-deposited ETH separately from reward ETH, or subtract known reward inflows from `totalAssetDeposits` before comparing to `depositLimitByAsset`.

## Proof of Concept
```solidity
// Foundry fork test
function test_sendFunds_dosDepositETH() public {
    // Setup: pool near ETH limit
    uint256 limit = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN);
    // Assume pool already has (limit - 0.5 ether) in TVL via prior deposits

    // Fund FeeReceiver with 1 ETH (simulating accumulated MEV rewards)
    vm.deal(address(feeReceiver), 1 ether);

    // Attacker (any EOA) calls sendFunds — no role required
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // Pool balance now exceeds limit
    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), limit);

    // Any user depositETH now reverts
    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
    lrtDepositPool.depositETH{value: 1 ether}(0, "");
}
```
