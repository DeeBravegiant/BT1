### Title
`respond_ckd` Omits Output Verification for `AppPublicKey` Variant, Enabling Single Byzantine Participant to Forge CKD Output — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs `ckd_output_check` only for the `AppPublicKeyPV` variant of `CKDAppPublicKey`, silently skipping all cryptographic verification for the `AppPublicKey` (legacy) variant. A single attested Byzantine participant — strictly below the signing threshold — can call `respond_ckd` with arbitrary forged `CKDResponse` values for any pending `AppPublicKey` request. The contract accepts the forged response unconditionally and delivers it to every waiting caller via the fan-out queue, constituting unauthorized confidential key derivation output without the required threshold authorization.

---

### Finding Description

In `respond_ckd` the verification branch is:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKey` requests the function falls through directly to `resolve_yields_for`, which drains the entire fan-out queue and resumes every waiting yield with the supplied response bytes — no field of `CKDResponse` (`big_y`, `big_c`) is validated against the MPC network's public key or any other invariant. [2](#0-1) 

The `AppPublicKey` variant is still accepted by the contract; the README describes it as "privately verifiable, legacy" but it is not rejected: [3](#0-2) 

By contrast, both sibling respond paths perform cryptographic verification before accepting a response:

- `respond` verifies the ECDSA signature against the MPC network's root public key before calling `resolve_yields_for`.
- `respond_verify_foreign_tx` verifies the ECDSA signature and rejects with `RespondError::InvalidSignature` on failure. [4](#0-3) 

The `respond_ckd` caller is required to be an attested participant: [5](#0-4) 

However, only **one** attested participant is needed — not a threshold — to call `respond_ckd` and win the race. `resolve_yields_for` removes the map entry on the first successful call, so subsequent honest-node responses receive `RequestNotFound`. [6](#0-5) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Monitor the NEAR blockchain for any pending `AppPublicKey` CKD request.
2. Immediately call `respond_ckd` with arbitrary `big_y` / `big_c` BLS12-381 G1 points.
3. The contract skips `ckd_output_check` and delivers the forged response to every queued caller.
4. The user receives a derived key that was **not** produced by the MPC threshold protocol.

If the user deploys funds to an address derived from the forged key, those funds are permanently inaccessible (freeze) or, if the attacker also controls the encryption, directly stealable. This is unauthorized confidential key derivation output without the required participant authorization — matching the **Critical** impact tier: *"Unauthorized … confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

- The `AppPublicKey` variant is live and accepted on mainnet (not deprecated, not rejected).
- The attacker needs to be exactly **one** attested MPC node — well below any signing threshold.
- The attack is a straightforward race: observe the pending request on-chain, submit a forged `respond_ckd` call in the next block. No special cryptographic knowledge is required; any two arbitrary G1 points suffice.
- The fan-out queue is drained on the first successful `respond_ckd`, so honest nodes cannot overwrite the forged response.

---

### Recommendation

1. **Reject `AppPublicKey` requests outright** in `respond_ckd` (and in `request_app_private_key`) and require callers to migrate to `AppPublicKeyPV`, which supports on-chain verification via `ckd_output_check`.
2. If `AppPublicKey` must remain supported, add a threshold-vote accumulation step before `resolve_yields_for` is called, analogous to how the signing protocol requires threshold participation before a signature is issued.

---

### Proof of Concept

1. Alice calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(alice_pk)`.
2. The request is stored in `pending_ckd_requests` and a yield is queued.
3. Mallory, a single attested Byzantine participant, observes the pending request on-chain.
4. Mallory calls `respond_ckd(ckd_request, CKDResponse { big_y: FORGED_POINT, big_c: FORGED_POINT })`.
5. `respond_ckd` reaches the `AppPublicKey` arm — the empty branch — and skips `ckd_output_check`.
6. `resolve_yields_for` drains the queue and resumes Alice's yield with the forged bytes.
7. Alice's `request_app_private_key` promise resolves with the forged `CKDResponse`.
8. Alice derives a foreign-chain address from the forged key and deposits funds; those funds are permanently frozen (or stolen if Mallory controls the encryption). [7](#0-6)

        let signer = Self::assert_caller_is_signer();

        if !self.accept_requests {

        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
