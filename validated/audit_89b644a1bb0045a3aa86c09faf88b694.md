### Title
Expired TEE Attestation Accepted by `vote_reshared` Due to Missing `re_verify` in `is_caller_an_attested_participant` — (`crates/contract/src/tee/tee_state.rs`)

### Summary

`vote_reshared` gates access through `assert_caller_is_attested_participant_and_protocol_active`, which delegates to `is_caller_an_attested_participant`. That function only verifies that a stored attestation entry exists and that the account/key identifiers match. It never calls `re_verify` and never checks the attestation's expiry timestamp. By contrast, `vote_new_parameters` calls `reverify_and_cleanup_participants`, which does call `re_verify` and enforces expiry. A participant whose attestation was valid at proposal time but expires before resharing completes can call `vote_reshared` successfully, bypassing the TEE authorization invariant.

### Finding Description

`is_caller_an_attested_participant` (tee_state.rs:469–498) performs three checks:

1. The signer is in the active participant set.
2. A `NodeAttestation` entry exists in `stored_attestations` keyed by the participant's TLS public key.
3. The stored `account_id` and `account_public_key` match the signer's identity. [1](#0-0) 

It does **not** call `reverify_participants` or `re_verify`. The expiry field (`expiry_timestamp_seconds`) inside the stored `ValidatedDstackAttestation` is never consulted. [2](#0-1) 

`vote_new_parameters`, by contrast, calls `reverify_and_cleanup_participants`, which calls `reverify_participants` → `re_verify` for every proposed participant, enforcing expiry at proposal time. [3](#0-2) 

`vote_reshared` uses only the weaker check: [4](#0-3) 

The same weaker check is used by `start_reshare_instance` and `vote_abort_key_event_instance`. [5](#0-4) 

### Impact Explanation

A participant whose attestation expires between the `vote_new_parameters` acceptance and the final `vote_reshared` call can:

- Call `vote_reshared` successfully, contributing their vote toward resharing completion.
- If their vote is the threshold-completing vote, trigger the on-chain transition to `Running` with themselves as a key holder in the new epoch — despite their TEE certificate being expired and their TEE environment no longer verifiable.

The TEE invariant the contract is designed to enforce is that every participant actively contributing to protocol state transitions must be running in a currently-verified TEE. This invariant is violated at the `vote_reshared` boundary.

### Likelihood Explanation

TEE certificates (TLS certificates embedded in dstack attestations) have finite lifetimes. A resharing can span multiple blocks and potentially a significant wall-clock duration. If the attestation expiry falls within that window, the scenario is reachable without any collusion or privileged access — the participant simply waits for their attestation to expire and then calls `vote_reshared`. The participant is already in the proposed set (they passed `vote_new_parameters` validation), so no additional preconditions are needed.

### Recommendation

`is_caller_an_attested_participant` should call `reverify_participants` (or inline an equivalent `re_verify` call) after confirming the attestation entry exists, so that expiry and image-hash validity are enforced at every protocol-state-transition call site, not only at proposal time. Alternatively, a dedicated `assert_caller_attestation_is_current` helper that wraps both the identity check and `re_verify` should be used uniformly across `vote_reshared`, `start_reshare_instance`, `vote_abort_key_event_instance`, and `respond`.

    /// reverifies stored participant attestations.
    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();
        let allowed_measurements = self.get_accepted_measurements();

        let participant_attestation = self.stored_attestations.get(&node_id.tls_public_key);
        let Some(participant_attestation) = participant_attestation else {
            return TeeQuoteStatus::Invalid("participant has no attestation".to_string());
        };

        // Verify the attestation quote
        let time_stamp_seconds = Self::current_time_seconds();
        match participant_attestation.verified_attestation.re_verify(
            time_stamp_seconds,
            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
