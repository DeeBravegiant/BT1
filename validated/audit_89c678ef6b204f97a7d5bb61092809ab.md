Audit Report

## Title
`swap()` Silently Succeeds with Zero Output When No Liquidity Is Available or Price Limit Is Pre-Satisfied — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.swap()` does not revert when the executed swap produces zero output. All four internal swap helpers contain early-exit or silent-cap paths that return `(0, 0, 0)` / `(0, 0, 0, 0)` without reverting. The pool then calls `metricOmmSwapCallback(0, 0, callbackData)`, transfers nothing, skips the `IncorrectDelta` guard, emits a `Swap` event with zero amounts, and returns `(0, 0)` — a silent no-op for a non-zero `amountSpecified`.

## Finding Description

**Confirmed early-exit paths in production code:**

1. `_swapToken0ForToken1SpecifiedInput` (L1148–1150): returns `(0, 0, 0)` when `params.priceLimitX64 >= initialPriceX64`.
2. `_swapToken1ForToken0SpecifiedInput` (L970–972): returns `(0, 0, 0)` when `params.priceLimitX64 <= initialPriceX64`.
3. `_swapToken0ForToken1SpecifiedOutput` (L1049–1052, L1066–1068): silently caps `amountOutScaled` to `binTotals.scaledToken1` (which may be 0), then returns `(0, 0, 0, 0)` if price limit is pre-satisfied.
4. `_swapToken1ForToken0SpecifiedOutput` (L872–875, L888–890): same cap for token0.

Additionally, for specified-input helpers, when `totalAvailableToken1Scaled == 0` (L1160–1162) or `totalAvailableToken0Scaled == 0` (L982–984), the while loop breaks immediately and returns `(amountInScaled - state.amountSpecifiedRemainingScaled, state.amountCalculatedScaled, ...) = (0, 0, 0)`.

**Consequence in `swap()` (L250–278):**
- `if (amount1Delta < 0)` → false → no token1 sent to recipient
- `metricOmmSwapCallback(0, 0, callbackData)` → caller pays nothing
- `if (amount0Delta > 0 && ...)` → false → `IncorrectDelta` guard skipped
- `emit Swap(...)` fires with zero amounts
- Returns `(0, 0)`

The sole guard at L225 (`require(amountSpecified != 0, InvalidAmount())`) checks the *requested* amount, not the *executed* amount.

## Impact Explanation

Any caller invoking `swap()` directly (custom router, aggregator, keeper) with a non-zero `amountSpecified` receives a successful transaction and `(0, 0)` return when the pool has no liquidity in the requested direction or when the price limit is already satisfied. The caller has no on-chain signal distinguishing "swap executed" from "swap silently did nothing." The callback is invoked with `(0, 0)`, so the caller is not required to pay anything. Downstream logic in custom routers that assumes a successful `swap()` delivered tokens will proceed incorrectly. This constitutes broken core pool functionality: the swap flow accepts a valid non-zero input request, executes no state change, and returns success.

Note: `MetricOmmSimpleRouter` has partial protections (`amountOutMinimum` for exact-input, `InvalidOutputAmount` check for exact-output, `InvalidSwapDeltas` for multi-hop), but these do not protect direct callers of the pool's `swap()` function.

## Likelihood Explanation

Reachable under normal, unprivileged operating conditions:
- A pool fully drained of token1 (`binTotals.scaledToken1 == 0`) silently accepts any `zeroForOne = true` swap.
- A caller passing a `priceLimitX64` already satisfied by the current oracle price triggers the early-exit on every call.
- Both conditions require no malicious setup and are reachable by any unprivileged trader.

## Recommendation

After `_executeSwap` returns, add a revert if both deltas are zero and `amountSpecified != 0`:

```solidity
if (amount0Delta == 0 && amount1Delta == 0) revert SwapResultedInZeroOutput();
```

This ensures callers always receive a clear on-chain signal when a swap produces no effect.

## Proof of Concept

1. Deploy a pool with token0/token1; add liquidity only to token0-side bins so `binTotals.scaledToken1 == 0`.
2. Call `swap(recipient, true, 1e18, 0, callbackData, "")` — exact-input, selling token0 for token1.
3. Inside `_swapToken0ForToken1SpecifiedInput`, price-limit check passes (`priceLimitX64 = 0 < initialPriceX64`), then `totalAvailableToken1Scaled == 0` causes the loop to break immediately (L1160–1162).
4. Returns `(0, 0, 0)` to `_executeSwap`.
5. `swap()` calls `metricOmmSwapCallback(0, 0, callbackData)` — caller pays nothing.
6. `emit Swap(msg.sender, recipient, true, 0, 0, ...)` fires.
7. `swap()` returns `(0, 0)` — transaction succeeds. Recipient received 0 token1; caller paid 0 token0.
