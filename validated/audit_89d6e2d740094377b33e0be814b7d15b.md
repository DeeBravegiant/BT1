### Title
Sub-transaction `directDeposits` not deducted from UTxO balance, enabling ADA creation from nothing - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs)

---

### Summary

In the Dijkstra era, sub-transactions support a `directDeposits` field (CDDL field 25) that adds ADA directly to registered reward accounts. The `SUBENTITIES` rule applies these deposits to accounts, but the `SUBUTXO` rule — which updates the UTxO — does not deduct the `directDeposits` amount from the UTxO balance and explicitly marks value-conservation failures as impossible. No other rule enforces value conservation for sub-transactions. An unprivileged transaction sender can therefore craft a sub-transaction whose inputs equal its outputs while also specifying non-zero `directDeposits`, creating ADA from nothing.

---

### Finding Description

**Vulnerability class:** Accounting/state-tracking bug — a state-changing operation (`directDeposits`) is applied to one part of the ledger state (reward accounts) while the corresponding deduction from another part (UTxO balance) is never enforced.

The Dijkstra era introduces nested sub-transactions. Each sub-transaction body may carry a `directDeposits` map (reward account → coin). Processing a sub-transaction involves two independent rules:

1. **`SUBENTITIES` / `dijkstraSubEntitiesTransition`** applies `directDeposits` to reward accounts:
   ```haskell
   pure $ certStateAfterSubCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
   ```
   The only guard is that target accounts exist (`directDepositsMissingAccounts`). No check verifies that the deposited coins are funded by the sub-transaction's inputs. [1](#0-0) 

2. **`SUBUTXO` / `dijkstraSubUtxoTransition`** updates the UTxO via `updateUTxOStateNoFees`, which removes inputs, adds outputs, and adjusts `utxosDeposited` only for certificate deposits/refunds. It does **not** subtract the `directDeposits` amount from the UTxO balance. [2](#0-1) 

`updateUTxOStateNoFees` computes `depositChange = certsTotalDepositsTxBody - certsTotalRefundsTxBody`. Neither function includes `directDeposits`. [3](#0-2) 

Critically, `SUBUTXO` explicitly marks `ValueNotConservedUTxO` as impossible:
```haskell
ValueNotConservedUTxO _ -> error "Impossible: `ValueNotConservedUTxO` for SUBUTXO"
``` [4](#0-3) 

The top-level `UTXO` rule does enforce value conservation, but it processes only the top-level transaction body using the UTxO state **after** sub-transactions have already been applied. It has no visibility into sub-transaction `directDeposits`. [5](#0-4) 

The invariant `utxosDeposited == obligationCertState` (enforced as a post-condition assertion in the top-level `LEDGER` rule) also does not cover `directDeposits` because they are not tracked in `obligationCertState`. [6](#0-5) 

---

### Impact Explanation

An attacker submits a top-level Dijkstra transaction containing a sub-transaction with:
- Inputs worth X lovelace
- Outputs worth X lovelace (UTxO-balanced)
- `directDeposits` of D lovelace to one or more registered reward accounts

After processing:
- The UTxO net change is zero (inputs = outputs).
- D lovelace is added to reward accounts with no corresponding deduction anywhere.
- Total ADA in the system increases by D lovelace.

This is **direct creation of ADA through an invalid ledger state transition** — Critical impact under the allowed scope.

---

### Likelihood Explanation

- Any unprivileged transaction sender can construct a top-level Dijkstra transaction with sub-transactions.
- The sub-transaction body is attacker-controlled serialized input; `directDeposits` is an optional CDDL field (field 25) freely settable by the submitter.
- The only validation on `directDeposits` is that target accounts are registered — a trivially satisfiable condition (the attacker can register their own stake credential first).
- No governance, privileged key, or consensus threshold is required.
- The Dijkstra era is the production target era in this repository; the code path is reachable once the era is active.

---

### Recommendation

In `dijkstraSubUtxoTransition`, add a value-conservation check for sub-transactions that includes `directDeposits`. Concretely, the sub-transaction must satisfy:

```
Σ inputs + Σ withdrawals = Σ outputs + Σ directDeposits + cert_deposits − cert_refunds
```

One approach: extend `updateUTxOStateNoFees` (or add a wrapper for the sub-transaction case) to subtract the total `directDeposits` coin from the UTxO balance check, analogous to how `certsTotalDepositsTxBody` is handled. Alternatively, re-enable `ValueNotConservedUTxO` for `SUBUTXO` with a `consumed`/`produced` calculation that includes `directDeposits` on the produced side.

---

### Proof of Concept

1. Register stake credential `C` (pays `keyDeposit`).
2. Submit a top-level Dijkstra transaction containing one sub-transaction `S`:
   - `S.inputs` = `{utxo_in}` (e.g., 1000 ADA UTXO)
   - `S.outputs` = `[utxo_out]` (1000 ADA back to attacker address)
   - `S.directDeposits` = `{reward_addr(C) → 500_000_000}` (500 ADA)
3. `SUBENTITIES` validates that `C` is registered → passes. Applies 500 ADA to `C`'s reward account.
4. `SUBUTXO` calls `updateUTxOStateNoFees`: removes `utxo_in`, adds `utxo_out`, `depositChange = 0` (no certs). No value-conservation check.
5. Top-level `UTXO` checks only the top-level transaction body → passes.

**Result:** 500 ADA has been created from nothing and credited to `C`'s reward account. The attacker can then withdraw it via a standard withdrawal transaction.
