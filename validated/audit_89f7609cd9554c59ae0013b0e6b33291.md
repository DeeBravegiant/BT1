### Title
Stale MPC Signature Remains Usable After `update_transfer_fee` Increases Fee — (`near/omni-bridge/src/lib.rs`)

### Summary

`update_transfer_fee()` mutates the stored fee for a pending transfer but does not invalidate any MPC signature already issued by `sign_transfer()`. An unprivileged sender can exploit this to finalize a transfer on the destination chain using the stale (lower-fee) signature, causing the recipient to receive more tokens than the updated fee structure permits and producing a fee-routing / balance-accounting divergence.

### Finding Description

The NEAR bridge has two independent paths that affect the same transfer state:

**Path 1 — Direct fee mutation:**
`update_transfer_fee()` overwrites `transfer.message.fee` in `pending_transfers` and re-inserts the record. It performs no check on whether an MPC signature has already been issued for this transfer. [1](#0-0) 

**Path 2 — Signature-based finalization:**
`sign_transfer()` reads the current fee, computes `amount_to_transfer = amount - fee`, and asks the MPC to sign a `TransferMessagePayload` containing that amount. The resulting signature is emitted in a public `SignTransferEvent` log. [2](#0-1) 

The signed payload commits to `amount_to_transfer` and `fee_recipient` but **not** to the current stored fee value. After the MPC signs, the transfer message is only removed from `pending_transfers` when `fee.is_zero()`; for any non-zero fee the record stays live and mutable. [3](#0-2) 

The destination-chain `finTransfer` (EVM, Starknet, Aptos, Solana) is fully permissionless: it verifies only the MPC signature and the destination nonce. It has no knowledge of the current stored fee on NEAR. [4](#0-3) [5](#0-4) 

### Impact Explanation

Concrete scenario (amounts in token units):

1. Alice initiates a transfer: `amount = 1000`, `fee = 100`.
2. Trusted relayer calls `sign_transfer` → MPC signs payload with `amount_to_transfer = 900`, `fee_recipient = relayerA`. Signature **S1** is emitted in a public event.
3. Alice calls `update_transfer_fee` to raise the fee to `200` (paying the extra native-fee difference). The stored record now has `fee = 200`.
4. Alice reads **S1** from the NEAR event log and calls `finTransfer` on the destination chain with **S1**. The destination chain verifies the valid MPC signature and mints/transfers **900** tokens to the recipient — 100 more than the updated fee structure intends.
5. The relayer later calls `claim_fee` on NEAR. `claim_fee` looks up the stored `TransferMessage` (fee = 200) and pays **200** tokens to `relayerA`.
6. Total tokens paid out: 900 (destination) + 200 (NEAR fee claim) = **1100 > 1000 locked** → unbacked token supply on the destination chain.

Even in the conservative case where `claim_fee` derives the fee from the proof rather than the stored record, the recipient still receives 100 extra tokens compared to what the updated fee structure authorizes, constituting a fee-routing and balance-accounting divergence.

The `update_transfer_fee` restriction that fee can only increase (not decrease) means the stale signature always carries a *higher* `amount_to_transfer` than the current intended value. [6](#0-5) 

### Likelihood Explanation

- `update_transfer_fee` is a public, payable function callable by the transfer sender with no special role.
- `finTransfer` on every destination chain is permissionless; anyone holding a valid MPC signature can submit it.
- The MPC signature is broadcast in a public NEAR event (`SignTransferEvent`), so Alice can trivially retrieve it.
- The only prerequisite is that a trusted relayer has already called `sign_transfer` before Alice calls `update_transfer_fee` — a normal operational sequence.

### Recommendation

Include the current fee (or a fee-version nonce) in the signed `TransferMessagePayload` so that any fee mutation automatically invalidates previously issued signatures. Concretely, add `fee: Fee` to `TransferMessagePayload` and include it in `encode_hashable()`: [7](#0-6) 

After this change, a signature produced with `fee = 100` will be rejected by the destination chain if the stored fee has been updated to `200`, because the hash will differ. This mirrors the fix applied in the referenced Farcaster report (adding the current recovery address to the typehash).

Alternatively, mark a transfer as "signed" in `pending_transfers` and reject subsequent `update_transfer_fee` calls once a signature has been issued.

### Proof of Concept

```
1. Alice: ft_transfer_call → init_transfer(amount=1000, fee=100, recipient=alice_eth)
   → pending_transfers[id] = {amount:1000, fee:100, ...}

2. Relayer: sign_transfer(transfer_id=id, fee_recipient=relayer, fee=Some({fee:100,...}))
   → MPC signs payload{dest_nonce=N, amount=900, recipient=alice_eth, fee_recipient=relayer}
   → SignTransferEvent emitted with signature S1

3. Alice: update_transfer_fee(transfer_id=id, fee={fee:200, native_fee:...})
   → pending_transfers[id].fee = 200
   (S1 is NOT invalidated)

4. Alice: reads S1 from NEAR event log
   Alice: calls EVM finTransfer(S1, {dest_nonce=N, amount=900, recipient=alice_eth, ...})
   → EVM verifies S1 against nearBridgeDerivedAddress ✓
   → EVM mints 900 tokens to alice_eth

5. Relayer: calls claim_fee on NEAR with proof of step 4
   → NEAR reads pending_transfers[id].fee = 200
   → NEAR pays 200 tokens to relayer

Result: 900 minted on EVM + 200 paid on NEAR = 1100 tokens disbursed from 1000 locked.
        100 tokens exist on EVM without NEAR backing.
```

pub struct TransferMessagePayload {
    pub prefix: PayloadType,
    pub destination_nonce: Nonce,
    pub transfer_id: TransferId,
    pub token_address: OmniAddress,
    pub amount: U128,
    pub recipient: OmniAddress,
    pub fee_recipient: Option<AccountId>,
    #[serde(default)]
    pub message: Vec<u8>,
}

impl TransferMessagePayload {
    pub fn encode_hashable(&self) -> Result<Vec<u8>, String> {
        if self.message.is_empty() {
            borsh::to_vec(&TransferMessagePayloadV1::from(self.clone())).map_err(stringify)
        } else {
            borsh::to_vec(self).map_err(stringify)
        }
    }
