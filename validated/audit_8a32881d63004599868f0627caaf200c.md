### Title
Cross-Chain Replay of `deployToken` Metadata Signature Enables Unauthorized Token Deployment and Permanent Fund Loss - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `near/omni-bridge/src/lib.rs`)

### Summary
The `MetadataPayload` signed by the NEAR MPC signer for `deployToken` does not include a destination chain identifier. Because all EVM-family OmniBridge deployments share the same `nearBridgeDerivedAddress` (derived from the same MPC key path `"bridge-1"`), a valid `deployToken` signature produced for one chain (e.g., Ethereum) can be replayed verbatim on any other EVM chain (Arbitrum, Base, BNB, Polygon, HyperEVM, Abstract). This allows an unprivileged attacker to deploy an unauthorized bridge token on an unintended chain, and any user who subsequently bridges tokens through that unauthorized deployment suffers permanent, irrecoverable fund loss.

### Finding Description
The NEAR bridge signs metadata payloads in `log_metadata_callback`:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
);
``` [1](#0-0) 

No chain identifier is included in `MetadataPayload`. The EVM `deployToken` function reconstructs and verifies this exact payload:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

The `deployToken` function is permissionless — any caller with a valid `(signatureData, metadata)` pair can invoke it: [3](#0-2) 

By contrast, `finTransfer` correctly includes `omniBridgeChainId` twice in its signed payload (for token address and recipient address), binding the signature to the specific destination chain: [4](#0-3) 

The same omission exists in the Starknet bridge. `MetadataPayload.to_borsh()` encodes only `PayloadType::Metadata`, `token`, `name`, `symbol`, and `decimals` — no chain ID: [5](#0-4) 

The NEAR MPC signer uses a single fixed path `"bridge-1"` for all chains, meaning the derived secp256k1 address (`nearBridgeDerivedAddress`) is identical across all EVM deployments: [6](#0-5) 

### Impact Explanation
An attacker who observes a legitimate `deployToken(sig, metadata)` call on Ethereum can immediately replay the identical `(sig, metadata)` on Arbitrum, Base, BNB, Polygon, HyperEVM, or Abstract. The replayed call passes signature verification (same signer address, same payload hash), satisfies the `!isBridgeToken[nearToEthToken[metadata.token]]` guard (token not yet deployed on that chain), and registers the unauthorized token in the EVM bridge's `nearToEthToken` / `ethToNearToken` / `isBridgeToken` mappings. [7](#0-6) 

Once the unauthorized token exists on chain B, any user who calls `initTransfer` with it has their tokens burned (because `isBridgeToken` is `true`). The NEAR bridge's `fin_transfer_callback` then attempts to look up token decimals for the chain-B token address, which was never registered on NEAR:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [8](#0-7) 

This panics, the NEAR callback fails, and the user's tokens are permanently burned on chain B with no redemption path. This satisfies both:
- **Critical**: Irreversible fund lock / permanently unclaimable user value in bridge flows.
- **High**: Insufficiently-bound signatures that bypass execution gates.

### Likelihood Explanation
The attack is fully unprivileged and requires only passive observation of a public blockchain. Any `deployToken` transaction on any supported EVM chain is visible on-chain. The attacker extracts `signatureData` and `metadata` from calldata and submits them to any other EVM OmniBridge. No special access, leaked keys, or privileged roles are required. The attack is deterministic and requires a single transaction.

### Recommendation
Include the destination chain identifier in the signed `MetadataPayload` on both the NEAR signing side and the EVM/Starknet verification side:

**NEAR (`log_metadata_callback`):** Add a `chain_kind: ChainKind` field to `MetadataPayload` and populate it with the intended destination chain before signing.

**EVM (`deployToken`):** Include `bytes1(omniBridgeChainId)` in the `borshEncoded` payload before hashing, mirroring the pattern already used in `finTransfer`.

**Starknet (`MetadataPayload.to_borsh`):** Append `omni_bridge_chain_id` to the borsh encoding before signature verification.

This ensures a metadata signature produced for Ethereum cannot be accepted by Arbitrum, Base, or any other chain.

### Proof of Concept
1. Admin calls `logMetadata(tokenAddress)` on NEAR for token `usdc.near`. NEAR MPC signs `keccak256(borsh(MetadataPayload{Metadata, "usdc.near", "USD Coin", "USDC", 6}))` → `sig`.
2. Relayer submits `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` to Ethereum OmniBridge → legitimate USDC bridge token deployed at `0xETH_USDC`.
3. Attacker observes the Ethereum transaction, extracts `sig` and `metadata`.
4. Attacker calls `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Arbitrum OmniBridge. Signature verification passes (same `nearBridgeDerivedAddress`, identical payload hash). Arbitrum deploys `0xARB_USDC` and sets `nearToEthToken["usdc.near"] = 0xARB_USDC`, `isBridgeToken[0xARB_USDC] = true`.
5. User calls `initTransfer(0xARB_USDC, 1000e6, ...)` on Arbitrum. `BridgeToken(0xARB_USDC).burn(user, 1000e6)` executes. Tokens are destroyed.
6. NEAR `fin_transfer_callback` receives `InitTransfer` proof with `token = OmniAddress::Arb(0xARB_USDC)`. `self.token_decimals.get(&OmniAddress::Arb(0xARB_USDC))` returns `None` → panic `TokenDecimalsNotFound`. Transfer permanently fails. User's 1000 USDC is irrecoverably lost.
