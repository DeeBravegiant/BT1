Audit Report

## Title
Insufficient `msg.value` causes full revert in `updateRate()`, leaving all receivers on stale rate — (`contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary

`MultiChainRateProvider.updateRate()` writes `rate` and `lastUpdated` before iterating over all registered `rateReceivers` and calling `ILayerZeroEndpoint.send()` for each. There is no pre-loop guard verifying `msg.value` covers the sum of all estimated fees, and no `try/catch` around each `send()`. If any single `send()` reverts due to insufficient remaining ETH balance, the entire transaction reverts atomically, rolling back the state writes and leaving every receiver on the previous stale rate.

## Finding Description

`updateRate()` is permissionless and payable: [1](#0-0) 

State is written before the loop: [2](#0-1) 

Inside the loop, `estimateFees` is called per-receiver and `send()` is called with exactly that amount, with no `try/catch`: [3](#0-2) 

If `msg.value` is less than the cumulative sum of all `estimatedFee` values, the contract's ETH balance is exhausted before the loop completes. The `send()` for the receiver that pushes the balance negative reverts, unwinding the entire transaction including the `rate` and `lastUpdated` writes. The helper `estimateTotalFee()` exists: [4](#0-3) 

but there is no on-chain enforcement that `msg.value >= estimateTotalFee()` before any `send()` is attempted.

**Concrete failure path:**
1. Two or more `RateReceiver` entries are registered.
2. Caller sends `msg.value` sufficient for receiver[0]'s fee but not the total.
3. `send()` for receiver[0] succeeds, consuming its fee from the contract's balance.
4. `send()` for receiver[1] reverts because the remaining balance is less than `estimatedFee` for that receiver.
5. EVM unwinds the entire transaction; both receivers retain the old stale rate.

## Impact Explanation

All registered `RSETHRateReceiver` instances remain on the previous stale rsETH/ETH rate. Downstream liquidity pools or rate-dependent integrations on every destination chain continue to quote the wrong rate until a subsequent successful `updateRate()` call. No funds are permanently lost, but the contract fails to deliver its core promise of propagating fresh rates cross-chain.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation

- `updateRate()` has no access control, so any external caller can trigger it.
- LayerZero message fees fluctuate with gas prices; an off-chain estimate obtained seconds before submission can be stale by the time the transaction is mined.
- With N receivers, the probability that at least one fee is under-estimated grows with N.
- The failure is silent from the caller's perspective (just a revert with no descriptive error), making it easy to miss and retry with the same insufficient value.

## Recommendation

1. **Pre-loop balance check:** compute the total estimated fee before the loop and revert early with a descriptive error (e.g., `InsufficientFee`) if `msg.value` is less than the total.
2. **`try/catch` per send:** wrap each `ILayerZeroEndpoint(...).send{...}(...)` in a `try/catch` so a single failure does not roll back successful sends.
3. **Refund excess ETH:** after the loop, return any unspent ETH to `msg.sender`.

## Proof of Concept

```solidity
// MockLZEndpoint: reverts on the second send() call to simulate insufficient balance
contract MockLZEndpoint {
    uint256 public callCount;
    function estimateFees(uint16, address, bytes calldata, bool, bytes calldata)
        external pure returns (uint256, uint256) {
        return (0.01 ether, 0);
    }
    function send(uint16, bytes calldata, bytes calldata, address payable, address, bytes calldata)
        external payable {
        callCount++;
        if (callCount == 2) revert("fee too low");
    }
}

function testStaleRateOnPartialFailure() public {
    MockLZEndpoint lz = new MockLZEndpoint();
    RSETHMultiChainRateProvider provider = new RSETHMultiChainRateProvider(oracle, address(lz));
    provider.addRateReceiver(101, receiver1);
    provider.addRateReceiver(102, receiver2);

    // msg.value covers only one receiver's fee
    vm.expectRevert();
    provider.updateRate{value: 0.01 ether}();

    // Both receivers still hold rate == 0 (stale); state write was rolled back
    assertEq(provider.rate(), 0);
    assertEq(provider.lastUpdated(), 0);
}
```
