### Title
Stale Non-Participant Votes in TEE Governance Functions Allow Sub-Threshold TEE Image Whitelisting — (File: `crates/contract/src/lib.rs`)

---

### Summary

After a resharing event, the contract spawns a detached promise to clean up stale votes from removed participants via `clean_tee_status`. Unlike `vote_update`, which explicitly re-filters votes against the current participant set to guard against cleanup-promise failure, the TEE governance functions (`vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, `vote_remove_os_measurement`) use the raw vote count returned by `tee_state.vote*()` without any such filtering. If the cleanup promise fails silently, stale votes from removed participants accumulate and can be combined with votes from a sub-threshold coalition of current participants to cross the governance threshold, whitelisting a TEE image that was never approved by a true threshold of active participants.

---

### Finding Description

**Root cause — asymmetric stale-vote protection:**

**Stale-vote accumulation path:**

After every successful resharing, `vote_reshared` spawns `clean_tee_status` as a **detached** (fire-and-forget) promise:

```rust
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        vec![],
        NearToken::from_yoctonear(0),
        Gas::from_tgas(self.config.clean_tee_status_tera_gas),
    )
    .detach();
``` [5](#0-4) 

A detached promise's failure is silent — the parent transaction succeeds regardless. If `clean_tee_status` runs out of gas or panics, the stale votes from removed participants remain in `tee_state` indefinitely.

**Attack scenario:**

1. Epoch N has participants {A, B, C, D, E} with threshold 3. Participant A votes for malicious code hash H.
2. A resharing removes A (and possibly B). Epoch N+1 has {B, C, D, E} with threshold 3. The `clean_tee_status` promise fails (e.g., gas exhaustion).
3. A's vote for H remains in `tee_state`. Now only 2 current participants (C and D) need to vote for H to reach the raw count of 3, crossing the threshold.
4. `whitelist_tee_proposal` is called for H, whitelisting a TEE image that only 2 of 4 current participants approved.

The same pattern applies to `vote_add_launcher_hash` and `vote_add_os_measurement`.

**Analog to the DyadStablecoin H-05 vulnerability:**

In DyadStablecoin, the liquidation threshold used a global `totalDeposits` denominator that could be manipulated by depositing ETH, making users liquidatable without any action on their part. Here, the TEE governance threshold uses a global raw vote count that retains stale contributions from removed participants, allowing a sub-threshold coalition of current participants to cross the threshold by leveraging votes they did not cast.

---

### Impact Explanation

A malicious TEE image (code hash, launcher hash, or OS measurement) can be whitelisted with fewer than the required threshold of current participants. Nodes running the whitelisted malicious image can then submit valid attestations via `submit_participant_info` and be proposed as participants in a future resharing. Once admitted as participants, they participate in threshold signing rounds. A TEE that is adversarially controlled can leak key shares or produce unauthorized signatures, enabling **unauthorized transaction execution or threshold signature issuance** — matching the Critical/High allowed impact scope.

At minimum, the invariant "threshold of current participants must agree before any TEE image is whitelisted" is broken, which is a production safety/accounting invariant violation matching the Medium allowed impact scope.

---

### Likelihood Explanation

- Resharing events are routine operational occurrences (participant rotation, TEE upgrades).
- Detached promise failure is a known risk explicitly acknowledged in the `vote_update` comment.
- A removed participant who previously voted for a target code hash need not take any further action — their stale vote persists automatically.
- The attacker only needs to be a participant at the time of the initial vote, not at the time of whitelisting.
- No privileged access, TEE break, or threshold collusion is required.

---

### Recommendation

Apply the same re-filtering pattern used in `vote_update` to all TEE governance vote-counting paths. Before comparing the vote count against the threshold, filter the raw vote set to include only accounts that are current participants:

```rust
// In vote_code_hash, vote_add_launcher_hash, vote_add_os_measurement:
let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();
let valid_votes = self.tee_state.count_votes_for_current_participants(
    code_hash,
    threshold_parameters.participants(),
);
if valid_votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(...);
}
```

Alternatively, make `clean_tee_status` a non-detached promise (chained with `.then()`), so that resharing only finalizes if the cleanup succeeds, or add a bounded retry mechanism.

### Proof of Concept

2. Participant A calls `vote_code_hash(malicious_hash)`. Raw vote count = 1.

    pub fn vote_code_hash(&mut self, code_hash: NodeImageHash) -> Result<(), Error> {
        log!(
            "vote_code_hash: signer={}, code_hash={:?}",
            env::signer_account_id(),
            code_hash,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let votes = self.tee_state.vote(code_hash, &participant);

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        // If the vote threshold is met and the new Docker hash is allowed by the TEE's RTMR3,
        // update the state
        if votes >= self.threshold()?.value() {
            self.tee_state
                .whitelist_tee_proposal(code_hash, tee_upgrade_deadline_duration);
        }

        Ok(())
    }
