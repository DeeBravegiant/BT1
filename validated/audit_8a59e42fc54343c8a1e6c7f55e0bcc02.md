### Title
Stale Votes from Removed Participants Counted in TEE Image-Hash Threshold Check — (File: `crates/contract/src/tee/proposal.rs`)

---

### Summary

`CodeHashesVotes::count_votes()` tallies **all** stored votes for a TEE image-hash proposal, including votes cast by participants who were subsequently removed via resharing. The authorization gate (`AuthenticatedParticipantId`) prevents removed participants from casting *new* votes, but their *old* votes remain in storage until an async cleanup promise runs. Because the cleanup promise is detached and its failure is silently ignored, a window exists in which stale votes from removed Byzantine participants are counted toward the governance threshold, allowing a malicious TEE image hash to be whitelisted with fewer live-participant votes than the threshold requires.

---

### Finding Description

**Root cause — `CodeHashesVotes::count_votes()` does not filter for current participants:**

```rust
// crates/contract/src/tee/proposal.rs
fn count_votes(&self, proposal: &NodeImageHash) -> u64 {
    self.proposal_by_account
        .values()
        .filter(|&prop| prop == proposal)
        .count() as u64   // ← counts ALL stored votes, including stale ones
}
``` [1](#0-0) 

The `vote()` method inserts the new vote and immediately returns this raw count to the caller, which then compares it against the governance threshold:

```rust
pub fn vote(&mut self, proposal: NodeImageHash, participant: &AuthenticatedParticipantId) -> u64 {
    self.proposal_by_account.insert(participant.clone(), proposal);
    let total = self.count_votes(&proposal);   // ← stale votes included
    total
}
``` [2](#0-1) 

**Contrast with the safe pattern used elsewhere:**

`TeeVerifierVotes::vote()` and `ProviderVotes::vote()` both filter inline against the *current* participant set before comparing to the threshold:

```rust
// crates/contract/src/tee/verifier_votes.rs
voter_set.count_for(|p| participants.is_participant_given_participant_id(&p.get()))
``` [3](#0-2) 

```rust
// crates/contract/src/foreign_chain_rpc.rs
voter_set.count_for(|(p, c)| {
    *c == chain && participants.is_participant_given_participant_id(&p.get())
})
``` [4](#0-3) 

`CodeHashesVotes` (and `MeasurementVotes`) lack this inline filter, making them inconsistent with the rest of the codebase.

**The cleanup is async and silently ignorable:**

After a successful resharing, `vote_reshared` spawns a detached promise to call `CLEAN_TEE_STATUS`, which eventually calls `tee_state.clean_non_participant_votes()`. Because the promise is `.detach()`-ed, any failure (gas exhaustion, panic) is silently swallowed and stale votes persist indefinitely:

```rust
// crates/contract/src/lib.rs
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        vec![],
        NearToken::from_yoctonear(0),
        Gas::from_tgas(self.config.clean_tee_status_tera_gas),
    )
    .detach();   // ← failure is silently ignored
``` [5](#0-4) 

The project's own test acknowledges the pre-cleanup count would be 3 (not 1) in the scenario below:

```rust
// crates/contract/src/tee/tee_state.rs
// P2 votes for the same malicious hash — should be only 1 vote, not 3
let vote_count = tee_state.votes.vote(malicious_hash, &auth_id);
assert_eq!(vote_count, 1, "Only the fresh vote from P2 should count");
``` [6](#0-5) 

The test only passes *after* `clean_non_participant_votes` is called. Before cleanup, the count is 3, and if the threshold is 3, the proposal passes.

---

### Impact Explanation

If a malicious TEE image hash is whitelisted with fewer live-participant votes than the threshold requires, nodes running that malicious image are admitted to the MPC network. Once admitted, they can participate in threshold signing rounds. Depending on how many Byzantine nodes are admitted, this can lead to:

- **Unauthorized threshold signature issuance** — if enough malicious nodes are whitelisted to reach the signing threshold, they can co-sign arbitrary payloads without legitimate user authorization.
- **Key-share exposure** — malicious TEE images may exfiltrate key-share material during DKG or resharing.

This matches the allowed impact: *"Critical. Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

The attack requires:
1. A resharing event that removes Byzantine participants who previously voted for a malicious hash.
2. The `CLEAN_TEE_STATUS` detached promise to fail (gas misconfiguration, contract panic, or deliberate griefing of the cleanup call).
3. At least one current participant to vote for the same hash after resharing (possibly deceived into thinking the hash is legitimate).

Resharing is a routine governance operation. Detached promise failure is a realistic operational condition. The Byzantine participants need only be below the signing threshold (the whole point of the threshold model). Likelihood is **Medium**.

---

### Recommendation

Apply the same inline participant-filter pattern used by `TeeVerifierVotes` and `ProviderVotes`:

```rust
// In CodeHashesVotes::vote(), replace count_votes() with a filtered count:
pub fn vote(
    &mut self,
    proposal: NodeImageHash,
    participant: &AuthenticatedParticipantId,
    current_participants: &Participants,   // ← pass current set
) -> u64 {
    self.proposal_by_account.insert(participant.clone(), proposal);
    let total = self.proposal_by_account
        .iter()
        .filter(|(p, prop)| {
            **prop == proposal
                && current_participants.is_participant_given_participant_id(&p.get())
        })
        .count() as u64;
    total
}
```

Apply the same fix to `MeasurementVotes::count_votes()`. This makes the threshold check consistent with `TeeVerifierVotes` and `ProviderVotes`, and eliminates the dependency on the async cleanup succeeding.

### Proof of Concept

**Setup:** N=5, T=3. Participants: {P0, P1, P2, P3, P4}.

1. **Pre-resharing:** Byzantine P0 and P1 call `vote_add_os_measurement` (or the equivalent TEE-hash vote endpoint) for a malicious image hash `H_evil`. Two entries are stored in `CodeHashesVotes.proposal_by_account`: `{P0 → H_evil, P1 → H_evil}`.

2. **Resharing:** Governance votes to remove P0 and P1. New participant set: {P2, P3, P4}, threshold still 3. `vote_reshared` completes and spawns the detached `CLEAN_TEE_STATUS` promise.

3. **Cleanup fails:** The `CLEAN_TEE_STATUS` promise runs out of gas (or is deliberately front-run with a gas-exhausting transaction). Stale entries `{P0 → H_evil, P1 → H_evil}` remain in storage.

4. **Post-resharing vote:** P2 (honest but deceived) calls the TEE-hash vote endpoint for `H_evil`. `CodeHashesVotes::vote()` inserts `{P2 → H_evil}` and calls `count_votes(H_evil)`, which returns **3** (P0 + P1 + P2).

5. **Threshold crossed:** The caller compares `3 >= threshold (3)` → `true`. `H_evil` is whitelisted.

6. **Impact:** Nodes running the malicious image `H_evil` are now admitted to the MPC network. With only 1 live-participant vote (P2), the threshold of 3 was effectively bypassed using 2 stale votes from removed Byzantine participants.
