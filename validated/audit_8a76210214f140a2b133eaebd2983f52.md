Audit Report

## Title
Missing Chainlink Heartbeat Staleness Check Enables Profitable Arbitrage Draining rsETH Reserves — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

## Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` checks only round-completeness (`answeredInRound < roundID`) but omits a time-based staleness check (`block.timestamp - updatedAt > maxStaleness`). Because the pool fee is a flat percentage of the raw token deposit amount and does not scale with `tokenToETHRate`, any staleness exceeding `feeBps / 100` percent makes it profitable for an unprivileged depositor to drain the pool's rsETH reserves by depositing tokens priced at the stale-high oracle rate.

## Finding Description

**Root cause — `ChainlinkOracleForRSETHPoolCollateral.getRate()`:**

The function performs three checks but none of them bound the age of the price data:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L30-32
if (answeredInRound < roundID) revert StalePrice();   // round-completeness only
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
// ← NO block.timestamp - timestamp > maxStaleness check
```

`answeredInRound < roundID` catches an *unanswered* round but does not catch a *heartbeat-stale* round — one that was answered but whose `updatedAt` is arbitrarily old. Chainlink feeds update on a deviation threshold (e.g. 0.5 %) or a heartbeat (e.g. 24 h). If the token's market price drops by less than the deviation threshold, the feed will not publish a new round until the next heartbeat, leaving `answeredInRound == roundID` while the price is materially stale.

**Fee accounting mismatch — `RSETHPool.viewSwapRsETHAmountAndFee`:**

```solidity
// contracts/pools/RSETHPool.sol L335-346
fee = amount * feeBpsForToken / 10_000;          // fee in token units, NOT ETH-adjusted
uint256 amountAfterFee = amount - fee;
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // ← stale-high
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;          // inflated
```

The same pattern exists in `RSETHPoolV3.viewSwapRsETHAmountAndFee` (L324-334) and `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee` (L301-311). The fee is a flat percentage of the raw token deposit and does not scale with `tokenToETHRate`. When `tokenToETHRate` is stale-high by a factor `k > 1`, `rsETHAmount` is inflated by `k` while `fee` is unchanged.

**Break-even analysis:** The attacker profits whenever:

```
(staleRate - fairRate) / fairRate > feeBps / 10_000
```

For `feeBps = 30` (0.3 %), any staleness above 0.3 % is profitable. Chainlink deviation thresholds are typically 0.5 %, meaning the feed can be up to 0.5 % stale before triggering an update — already above the break-even point.

**Exploit path:**
1. Attacker monitors `updatedAt` from `latestRoundData()` on-chain.
2. Token market price drops (e.g. 0.4 %–20 %) but the Chainlink feed has not yet published a new round (within deviation/heartbeat window).
3. Attacker calls `deposit(token, amount, referralId)` — a public, permissionless function.
4. `viewSwapRsETHAmountAndFee` calls `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which passes all existing checks and returns the stale-high rate.
5. Attacker receives inflated rsETH; the protocol receives tokens worth less than the rsETH paid out; the fee does not cover the shortfall.

## Impact Explanation

**Critical — Direct theft of protocol rsETH reserves.**

With `staleRate = 1.1e18`, `fairRate = 0.9e18`, `rsETHToETHrate = 1.05e18`, `feeBps = 30`, `amount = 1e18`:

| Quantity | Value |
|---|---|
| Fee (tokens) | `3e15` |
| rsETHAmount (stale) | `≈ 1.044e18` rsETH |
| Fair rsETHAmount | `≈ 0.854e18` rsETH |
| Excess rsETH in ETH | `≈ 0.1995e18 ETH` |
| Fee revenue in ETH | `≈ 0.0027e18 ETH` |
| Net protocol loss | `≈ 0.197 ETH per deposit` |

Repeated deposits during the staleness window drain the pool's rsETH reserves. The `limitDailyMint` guard in `RSETHPoolV3` caps per-day exposure but does not prevent the attack; it only bounds the magnitude per day.

## Likelihood Explanation

- No oracle operator compromise, governance access, or privileged role is required.
- `deposit(token, amount, referralId)` is a public, permissionless function callable by any EOA or contract.
- The attacker only needs to read `updatedAt` from `latestRoundData()` on-chain and deposit when `block.timestamp - updatedAt` is large relative to the expected heartbeat.
- Chainlink LST feeds (e.g. wstETH/ETH) have deviation thresholds of 0.5 % and heartbeats of 24 h. The break-even staleness for `feeBps = 30` is 0.3 %, meaning the attack is profitable for any staleness above 0.3 % — well within normal Chainlink update windows during volatile periods.
- The attack is repeatable across all three pool variants (`RSETHPool`, `RSETHPoolV3`, `RSETHPoolNoWrapper`) and any supported collateral token whose oracle is a `ChainlinkOracleForRSETHPoolCollateral` instance.

## Recommendation

Add a configurable `maxStaleness` parameter to `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
uint256 public immutable maxStaleness;

constructor(address _oracle, uint256 _maxStaleness) {
    oracle = _oracle;
    maxStaleness = _maxStaleness;
}

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 updatedAt, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > maxStaleness) revert StalePrice(); // ← add this
    ...
}
```

Set `maxStaleness` to slightly above the feed's documented heartbeat (e.g. heartbeat × 1.1). Deploy a new oracle instance per collateral token with the appropriate `maxStaleness` for that feed.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;
import "forge-std/Test.sol";

contract MockStaleChainlinkFeed {
    function latestRoundData() external view returns (
        uint80 roundId, int256 answer, uint256 startedAt, uint256 updatedAt, uint80 answeredInRound
    ) {
        roundId = 100;
        answeredInRound = 100;                    // passes answeredInRound < roundID check
        answer = 1.1e8;                           // stale-high: 1.1 ETH per token
        updatedAt = block.timestamp - 25 hours;   // 25h stale, no time check in contract
        startedAt = updatedAt;
    }
    function decimals() external pure returns (uint8) { return 8; }
}

contract StaleOracleTest is Test {
    function testStaleOracleBreaksInvariant() public {
        uint256 fairRate   = 0.9e18;
        uint256 staleRate  = 1.1e18;
        uint256 rsETHRate  = 1.05e18;
        uint256 feeBps     = 30;
        uint256 amount     = 1e18;

        uint256 fee            = amount * feeBps / 10_000;
        uint256 amountAfterFee = amount - fee;
        uint256 rsETHMinted    = amountAfterFee * staleRate / rsETHRate;
        uint256 fairRsETH      = amountAfterFee * fairRate  / rsETHRate;
        uint256 excessRsETH    = rsETHMinted - fairRsETH;

        uint256 feeInETH    = fee * fairRate / 1e18;
        uint256 excessInETH = excessRsETH * rsETHRate / 1e18;

        // Invariant: fee revenue must cover excess rsETH given away
        // This FAILS: 2.7e15 < 1.995e17
        assertGe(feeInETH, excessInETH, "INVARIANT BROKEN: fee does not cover excess rsETH");
    }
}
```

Deploy `ChainlinkOracleForRSETHPoolCollateral` with `MockStaleChainlinkFeed` as the oracle. The `answeredInRound < roundID` check passes (both equal 100), `updatedAt != 0`, and `ethPrice > 0`, so `getRate()` returns the stale-high price. The `assertGe` fails, confirming the invariant is broken under realistic parameters.
