Audit Report

## Title
Stale Cross-Chain Oracle Rate Allows Depositors to Extract Excess wrsETH from Pool Reserves - (`contracts/pools/RSETHPool.sol`)

## Summary
`RSETHPool.deposit()` computes wrsETH output using a rate from `CrossChainRateReceiver.getRate()` that stores `lastUpdated` but enforces no staleness check before returning `rate`. Because rsETH is a monotonically appreciating yield-bearing token, any cross-chain delivery lag produces a stale rate that is lower than the true current rate, causing the division `amountAfterFee * 1e18 / rsETHToETHrate` to yield a larger-than-fair wrsETH amount, which is transferred directly from the pool's wrsETH reserves to the depositor.

## Finding Description

`CrossChainRateReceiver.lzReceive()` records both `rate` and `lastUpdated` when a LayerZero message arrives: [1](#0-0) 

`getRate()` returns the stored rate unconditionally with no check against `lastUpdated` or `block.timestamp`: [2](#0-1) 

`RSETHPool.viewSwapRsETHAmountAndFee()` consumes this rate directly: [3](#0-2) 

`deposit()` then transfers the computed wrsETH amount from the pool's own balance to the caller: [4](#0-3) 

Since rsETH/ETH rate increases monotonically, a stale (lagging) rate is always strictly lower than the true current rate. A lower denominator in `amountAfterFee * 1e18 / rsETHToETHrate` produces a larger `rsETHAmount`. The excess wrsETH is taken from the pool's reserves, not minted, constituting a direct transfer of value from the pool to the depositor. No existing check in `deposit()`, `viewSwapRsETHAmountAndFee()`, or `getRate()` validates freshness.

The `InterimRSETHOracle` has the same structural absence of a staleness check in `getRate()`, but its rate is set manually by a privileged `MANAGER_ROLE`, making that vector an operator-trust issue rather than a permissionless exploit path. [5](#0-4) 

## Impact Explanation

**Critical — Direct theft of funds at-rest in the pool.**

The pool holds a wrsETH balance that it disburses to depositors. When the oracle rate is stale by X%, a depositor receives approximately X% more wrsETH than their ETH contribution is worth at the true rate. The attacker can immediately sell the excess wrsETH on secondary markets at the true market rate. The stolen wrsETH comes directly from the pool's reserves, not from any future yield stream. This matches the allowed impact: *"Direct theft of any user funds, whether at-rest or in-motion, other than unclaimed yield."*

## Likelihood Explanation

Cross-chain oracle lag is a routine operational condition requiring no special privileges, no front-running, and no external protocol compromise. LayerZero message delivery can be delayed by minutes to hours under normal network conditions. An attacker needs only to:
1. Read `CrossChainRateReceiver.lastUpdated` and `rate` on-chain (both public).
2. Read the live rsETH/ETH rate on L1 (publicly available).
3. Call `RSETHPool.deposit{value: amount}(referralId)` whenever a profitable staleness gap exists.

The attack is repeatable every time a staleness window opens, which is every cross-chain update cycle. No governance capture, oracle operator compromise, or victim action is required.

## Recommendation

1. **Add a staleness threshold in `CrossChainRateReceiver.getRate()`:**
   ```solidity
   uint256 public constant MAX_STALENESS = 24 hours;
   function getRate() external view returns (uint256) {
       require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate stale");
       return rate;
   }
   ```
2. **Add a minimum rate floor in `viewSwapRsETHAmountAndFee`** to revert if the oracle rate deviates significantly below an expected floor (e.g., last known rate minus a tolerance).
3. **Add a `minRsETHOut` parameter to `deposit()`** so depositors can specify slippage protection, preventing sandwich-style exploitation of any residual lag.

## Proof of Concept

```solidity
// Foundry fork test on Arbitrum
function testStaleOracleExploit() public {
    // 1. Simulate oracle staleness: oracle still reports 1.05e18,
    //    but true L1 rate has moved to 1.155e18 (~10% appreciation)
    vm.mockCall(
        address(rsETHOracle),
        abi.encodeWithSelector(IOracle.getRate.selector),
        abi.encode(1.05e18)
    );

    uint256 trueRate = 1.155e18;
    uint256 depositAmount = 1 ether;

    // 2. Attacker deposits 1 ETH (assuming feeBps = 0 for clarity)
    vm.deal(attacker, depositAmount);
    vm.prank(attacker);
    pool.deposit{value: depositAmount}("ref");

    // 3. Attacker received ~0.952e18 wrsETH (stale rate)
    //    Fair value at true rate: ~0.866e18 wrsETH
    uint256 fairAmount   = depositAmount * 1e18 / trueRate;  // ~0.866e18
    uint256 received     = wrsETH.balanceOf(attacker);       // ~0.952e18
    assertGt(received, fairAmount);

    // 4. Excess ~0.086e18 wrsETH was taken from pool reserves
    //    Attacker sells on Uniswap at true market rate for ~0.086 ETH profit
}
```
