Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Amount in Overflow-Equivalent Limit Check — (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its limit comparison, checking only whether the current total already exceeds the cap rather than whether the deposit would push it over. Any unprivileged caller can invoke `depositETH` with an arbitrarily large `msg.value` while the total is still below the limit, causing the pool to accumulate ETH far beyond the configured cap and leaving the surplus idle without earning restaking yield.

## Finding Description
In `contracts/LRTDepositPool.sol` at L676–682, the ETH branch evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` while the ERC-20 branch correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. [1](#0-0) 

`_beforeDeposit` (L661–663) calls this function and reverts with `MaximumDepositLimitReached` only when it returns `true`. [2](#0-1) 

Because the ETH branch never adds `amount`, the guard returns `false` whenever `totalAssetDeposits <= depositLimit` — the normal operating state — regardless of how large `msg.value` is. The deposit proceeds, rsETH is minted, and the total can overshoot the cap by any magnitude in a single transaction. The view function `getAssetCurrentLimit` (L402–409) correctly computes remaining capacity, confirming the omission in the guard is unintentional. [3](#0-2) 

The entry point `depositETH` (L76–93) is fully permissionless (`external payable`, no role check). [4](#0-3) 

## Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.** ETH deposited beyond the cap cannot be deployed to EigenLayer strategies and sits idle in the deposit pool. rsETH holders who deposited after the breach hold tokens backed by non-yielding ETH, causing the protocol to fail to deliver the promised restaking returns. No direct theft or permanent freeze of funds occurs.

## Likelihood Explanation
The exploit requires no special role, no coordination, and no unusual precondition. The enabling condition (`totalAssetDeposits <= depositLimit`) is the normal operating state of the protocol. Any depositor can trigger this at any time by calling `depositETH` with a `msg.value` larger than the remaining capacity.

## Recommendation
Add `amount` to the ETH branch to mirror the ERC-20 logic:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Assume `depositLimitByAsset(ETH) = 1000 ether` and `getTotalAssetDeposits(ETH) = 999 ether`.

1. Alice calls `depositETH{value: 5000 ether}(0, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 5000e18)` evaluates `999e18 > 1000e18` → `false`.
3. `_beforeDeposit` does not revert; rsETH is minted for 5000 ETH.
4. `getTotalAssetDeposits(ETH)` is now 6000 ETH — 6× the intended cap — with no protocol-level rejection.

**Foundry test sketch:**
```solidity
function test_ethDepositBypassesLimit() public {
    // set depositLimit = 1000 ether, seed pool with 999 ether
    vm.deal(alice, 5000 ether);
    vm.prank(alice);
    lrtDepositPool.depositETH{value: 5000 ether}(0, "");
    assertGt(lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN), 1000 ether);
}
```

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
