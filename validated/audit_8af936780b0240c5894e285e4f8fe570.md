### Title
Unconstrained `fee_proposal_fri` During Startup Window Poisons `fee_actual` for Subsequent Blocks - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

During the first `fee_proposal_window_size` (10) blocks, `compute_fee_actual` returns `None` because the sliding window has not yet accumulated enough entries. The `is_proposal_init_valid` function explicitly skips the `fee_proposal_fri` bounds check when `fee_actual` is `None`. A malicious proposer selected during this startup window can publish any arbitrary `fee_proposal_fri` value; validators accept it unconditionally. Those unconstrained values are then recorded in `fee_proposals_window` and become the sole inputs to `fee_actual` once the window fills, permanently skewing the fee mechanism for the blocks that follow.

### Finding Description

`compute_fee_actual` in `crates/apollo_consensus_orchestrator/src/dynamic_gas_price/mod.rs` performs a `checked_sub(window_size)` on the block height. For any height below `window_size` (currently 10), the subtraction underflows, the function logs a warning, and returns `None`. [1](#0-0) 

`is_proposal_init_valid` in `validate_proposal.rs` gates the entire bounds check on both `fee_actual` and `fee_proposal` being `Some`. When `fee_actual` is `None` — which is always the case for the first `window_size` blocks — the `if let` arm is never entered and no range enforcement occurs. [2](#0-1) 

The code comment at line 397 explicitly acknowledges this: *"During initiation (fee_actual is None, <window_size blocks), bounds are not enforced."*

An honest proposer falls back to `self.l2_gas_price` when `fee_actual` is `None`: [3](#0-2) 

A malicious proposer is under no such constraint. It can publish `fee_proposal_fri = u128::MAX` (or `0`) in `ProposalInit`, and every validator will accept the proposal because the bounds check is skipped. The value is then stored in `fee_proposals_window` via `record_fee_proposal`.

Once height reaches `window_size`, `compute_fee_actual` computes the median of the window. If the attacker controlled ≥ 6 of the first 10 blocks and set `fee_proposal_fri = u128::MAX` in each, the median is `u128::MAX`. Every subsequent honest proposer's `fee_proposal` is then clamped to within `margin_ppt` (2 per-thousand) of `u128::MAX`, keeping the fee pinned at an extreme value for hundreds of additional blocks until the window naturally drains.

The `fee_actual` value is also forwarded to `calculate_next_l2_gas_price_for_fin`, which uses it to derive the L2 gas price carried in `ProposalFin.fin_payload.l2_gas_info.next_l2_gas_price_fri`: [4](#0-3) 

The `fee_proposal_fri` is also hashed into the `ProposalCommitment` that consensus signs over: [5](#0-4) 

Because the validator recomputes the commitment using the proposer-supplied `fee_proposal_fri` from `ProposalInit`, the commitment is internally consistent regardless of the value — validators never independently bound-check it during startup.

### Impact Explanation

A staker selected as proposer for a majority of the first `fee_proposal_window_size` (10) blocks can set `fee_proposal_fri` to any value. After the window fills, `fee_actual` equals the median of those unconstrained values. `calculate_next_l2_gas_price_for_fin` uses this `fee_actual` to compute the next L2 gas price, causing the L2 gas price embedded in `ProposalFin` to be incorrect with direct economic impact on every user transaction for the following `window_size`-length period. This matches the "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" criterion.

### Likelihood Explanation

The startup window is only 10 blocks, but it occurs at the most critical moment: network genesis or a fresh deployment. Committee selection at genesis is typically concentrated among a small set of initial validators. A single validator holding sufficient stake to be selected as proposer for 6 of the first 10 blocks — plausible in a bootstrapped network — can fully control `fee_actual` for the subsequent window. No privileged off-chain access is required; the attacker only needs to be a staker and be selected by the normal consensus mechanism.

### Recommendation

Pre-populate `fee_proposals_window` with `window_size` synthetic entries equal to the configured `l2_gas_price` minimum before the first block is processed (analogous to the amAMM recommendation of assigning an initial top bid at pool creation). This ensures `compute_fee_actual` always returns `Some` and `is_proposal_init_valid` always enforces bounds, eliminating the unconstrained startup window entirely.

Alternatively, when `fee_actual` is `None`, enforce that `fee_proposal_fri` equals exactly `l2_gas_price` (the honest fallback value) rather than accepting any value.

### Proof of Concept

1. Network starts. `fee_proposals_window` is empty. `fee_proposal_window_size = 10`.
2. A malicious staker is selected as proposer for blocks 0–5 (6 of the first 10 blocks).
3. For each of those blocks, the attacker publishes `ProposalInit { fee_proposal_fri: Some(GasPrice(u128::MAX)), ... }`.
4. Each validator calls `is_proposal_init_valid`. `compute_fee_actual` returns `None` for heights 0–9. The `if let (Some(fee_actual), Some(fee_proposal))` arm is never entered. The proposal passes validation.
5. `record_fee_proposal(height, Some(GasPrice(u128::MAX)))` is called for heights 0–5; honest proposers record `GasPrice(l2_gas_price)` for heights 6–9.
6. At height 10, `compute_fee_actual` computes the median of `[u128::MAX × 6, l2_gas_price × 4]`. With 6 extreme values, the median is `u128::MAX`.
7. `calculate_next_l2_gas_price_for_fin` receives `fee_actual = Some(GasPrice(u128::MAX))` and produces an extreme next L2 gas price, which is embedded in `ProposalFin.fin_payload.l2_gas_info.next_l2_gas_price_fri` and propagated to all subsequent blocks.
8. Every user transaction from block 10 onward pays fees computed against this manipulated gas price until the window naturally drains — approximately `window_size` more blocks. [2](#0-1) [6](#0-5) [7](#0-6) [8](#0-7)

pub fn compute_fee_actual(
    fee_proposals_window: &BTreeMap<BlockNumber, Option<GasPrice>>,
    height: BlockNumber,
    window_size: u64,
) -> Option<GasPrice> {
    let Some(start) = height.0.checked_sub(window_size) else {
        warn!(
            "Cannot compute fee_actual for height {height}: height is below window_size \
             ({window_size})"
        );
        return None;
    };
    let window_size_usize = usize::try_from(window_size).expect("window_size fits in usize");
    let mut window = Vec::with_capacity(window_size_usize);
    for source_height in (start..height.0).map(BlockNumber) {
        match fee_proposals_window.get(&source_height) {
            Some(Some(price)) => window.push(*price),
            Some(None) | None => {
                warn!(
                    "Cannot compute fee_actual for height {height}: fee_proposals_window has no \
                     recorded fee_proposal for height {source_height}"
                );
                return None;
            }
        }
    }
    window.sort();
    let mid = window_size_usize / 2;
    let median = if window_size_usize.is_multiple_of(2) {
        // Even: average of the two middle values, rounded down.
        // Overflow-safe averaging: a + (b - a) / 2 (safe because sorted, so b >= a).
        GasPrice(window[mid - 1].0 + (window[mid].0 - window[mid - 1].0) / 2)
    } else {
        window[mid]
    };
    Some(median)
}
