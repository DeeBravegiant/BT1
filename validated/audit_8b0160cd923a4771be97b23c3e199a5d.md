### Title
`respond_verify_foreign_tx` Accepts Any Valid Signature Regardless of Request Binding — Cross-Request Payload Hash Replay - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that the MPC signature is valid over `response.payload_hash`, but never verifies that `response.payload_hash` was actually derived from the `request` argument supplied in the same call. A single malicious attested MPC node (below the signing threshold) can replay a legitimately-produced `(payload_hash, signature)` pair from one pending foreign-chain verification request to resolve a completely different pending request, delivering false verification data to the victim caller.

---

### Finding Description

The `respond_verify_foreign_tx` function performs two independent operations:

1. **Signature check** — verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the MPC root public key.
2. **Yield resolution** — resolves all pending yields stored under the `request` key. [1](#0-0) 

The critical missing step is that the contract never recomputes or verifies that `response.payload_hash` was derived from `request`. According to the protocol specification, the correct hash is:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1({ request, extracted_values })))
``` [2](#0-1) 

The `payload_hash` commits to both the `ForeignChainRpcRequest` (tx_id, confirmations, extractors) and the `extracted_values` observed on the foreign chain. The contract has access to `request` but never uses it to validate `payload_hash`. It simply trusts whatever hash the responding node provides, as long as the signature over it is valid. [3](#0-2) 

The `pending_verify_foreign_tx_requests` map is keyed on the full `VerifyForeignTransactionRequest` struct (tx_id, confirmations, extractors, domain_id, payload_version), so two different user requests produce two different map keys. [4](#0-3) 

---

### Impact Explanation

A single malicious attested MPC node can:

1. Observe a legitimately-produced `(payload_hash_A, sig_A)` pair that the MPC network generated for **Request A** (e.g., verifying Bitcoin `tx_id=[1..]`, extracting `block_hash=[X..]`).
2. Call `respond_verify_foreign_tx(request = B, response = {payload_hash: payload_hash_A, signature: sig_A})` where **Request B** is a different pending request (e.g., verifying Bitcoin `tx_id=[2..]`).
3. The contract checks: is `sig_A` valid over `payload_hash_A`? **Yes** — it is a genuine MPC signature.
4. The contract resolves all yields queued under Request B, delivering `{payload_hash_A, sig_A}` to the user of Request B.

The user of Request B receives a `VerifyForeignTransactionResponse` whose `payload_hash` corresponds to a completely different foreign-chain transaction than the one they requested. Any downstream contract that does not independently recompute and verify the `payload_hash` binding (using `ForeignChainSignatureVerifier`) will accept this as a valid verification of `tx_id=[2..]` when it is actually a verification of `tx_id=[1..]`. This enables forged foreign-chain verification and potential double-spend or invalid bridge execution.

Even when the downstream contract does use `ForeignChainSignatureVerifier`, the yield for Request B is permanently consumed — the user cannot obtain a correct response without resubmitting, and the attacker can repeat the replay to block every resubmission. [5](#0-4) 

---

### Likelihood Explanation

The attacker must be a single attested MPC participant — a realistic threat model explicitly listed in the scope (Byzantine participant strictly below the signing threshold). No threshold collusion is required. The attacker does not forge any cryptographic material; they only replay a legitimately-produced signature. The attack is executable in a single on-chain transaction and requires no special timing or network conditions. Any pending pair of foreign-chain verification requests with the same domain is sufficient to mount the attack.

---

### Recommendation

Inside `respond_verify_foreign_tx`, after verifying the signature, recompute the expected `payload_hash` from `request` and reject any response whose `payload_hash` does not match. Because the contract does not know `extracted_values`, the binding must be enforced differently — for example, by requiring the node to also supply the `extracted_values` in the response and having the contract recompute `SHA-256(borsh(ForeignTxSignPayload::V1({request, extracted_values})))` on-chain, then comparing it to `response.payload_hash` before accepting the response.

Alternatively, the `payload_hash` could be structured so that the first component is a commitment to the `request` alone (a hash of the request), allowing the contract to at minimum verify that the hash is bound to the correct request without knowing `extracted_values`.

---

### Proof of Concept

**Setup:**
- Two pending requests exist simultaneously:
  - **Request A**: `{tx_id: [1u8;32], confirmations: 2, extractors: [BlockHash], domain_id: 0, payload_version: V1}`
  - **Request B**: `{tx_id: [2u8;32], confirmations: 2, extractors: [BlockHash], domain_id: 0, payload_version: V1}`

**Step 1 — Honest signing for Request A:**
The MPC network produces `sig_A` over:
```
payload_hash_A = SHA-256(borsh(ForeignTxSignPayload::V1({
    request: {tx_id: [1u8;32], ...},
    values: [BlockHash([42u8;32])]
})))
```

**Step 2 — Malicious replay:**
A single attested MPC node calls:
```
respond_verify_foreign_tx(
    request = Request_B,   // tx_id=[2u8;32]
    response = {
        payload_hash: payload_hash_A,   // from tx_id=[1u8;32]
        signature: sig_A                // valid MPC signature
    }
)
```

**Step 3 — Contract accepts:**
- `verify_ecdsa_signature(sig_A, payload_hash_A, mpc_root_pk)` → `Ok(())` ✓
- `resolve_yields_for(pending_verify_foreign_tx_requests, Request_B, response)` → drains all yields for Request B [6](#0-5) 

**Step 4 — Victim receives false data:**
The user of Request B receives `{payload_hash: payload_hash_A, sig: sig_A}` — a valid MPC signature over a hash that commits to `tx_id=[1u8;32]`, not `tx_id=[2u8;32]`. Any contract that does not independently verify the `payload_hash` binding accepts this as proof that `tx_id=[2u8;32]` was verified on the foreign chain.
