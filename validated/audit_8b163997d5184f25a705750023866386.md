Audit Report

## Title
Stale Cross-Chain Rate in `getRate()` Enables Over-Issuance of `wrsETH`, Causing Pool Undercollateralization — (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns the stored `rate` with no staleness check despite recording `lastUpdated`. Both `RSETHPoolV2` and `RSETHPoolV3` delegate all mint pricing to this oracle. Because rsETH/ETH is monotonically increasing, any gap between an L1 rate update and the next LayerZero delivery causes the pools to mint more `wrsETH` per ETH than the deposited ETH can cover at the true rate, permanently undercollateralizing the pool.

## Finding Description

`CrossChainRateReceiver` stores both `rate` and `lastUpdated` but `getRate()` ignores the timestamp entirely:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L102-105
function getRate() external view returns (uint256) {
    return rate;
}
```

Both pool versions call this oracle without any additional freshness guard:

- `RSETHPoolV2.getRate()` → `IOracle(rsETHOracle).getRate()` (L201-203)
- `RSETHPoolV3.getRate()` → `IOracle(rsETHOracle).getRate()` (L235-237)

Both `viewSwapRsETHAmountAndFee` implementations compute:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

where `rsETHToETHrate` is the potentially stale value. `deposit()` mints directly from this result with no additional check.

**Exploit path:**

1. L1 rsETH/ETH rate advances from R0 → R1 (R1 > R0) due to staking rewards.
2. The LayerZero message has not yet been relayed; `CrossChainRateReceiver.rate` still holds R0.
3. Attacker calls `RSETHPoolV2.deposit{value: E}()` (or V3 equivalent).
4. Pool computes `rsETHAmount = E * 1e18 / R0 > E * 1e18 / R1`.
5. Attacker receives the excess delta: `E * 1e18 * (1/R0 − 1/R1)` extra `wrsETH`.
6. Attacker bridges `wrsETH` to L1, unwraps 1:1 to rsETH, redeems at R1.
7. Attacker extracts `E * (R1/R0 − 1)` ETH of value never deposited.

The `dailyMintLimit` modifier (RSETHPoolV2 L72-93, RSETHPoolV3 L96-125) caps per-day exposure but does not prevent the exploit; it only bounds the rate of insolvency accumulation.

## Impact Explanation

**Critical — Protocol insolvency.** The pool's ETH reserves are structurally insufficient to back all outstanding `wrsETH` at the true rsETH/ETH rate. Every deposit made during a stale-rate window permanently widens the collateral gap. The deficit is `E * (1/R0 − 1/R1)` rsETH per deposit, which is irreversible once the ETH is bridged to L1 and the wrsETH is redeemed. This matches the allowed critical impact: *Protocol insolvency*.

## Likelihood Explanation

- rsETH/ETH increases continuously (~4–5% APY); the rate is never flat.
- LayerZero relay delays are routine operational events with no on-chain SLA enforcing freshness.
- `lastUpdated` is stored but never enforced, so any delay — routine or adversarial (relayer censorship, gas spike) — opens the window.
- The attack requires no special role, no front-running, and no external protocol compromise: a single permissionless `deposit()` call suffices.
- Profit per deposit is small under normal conditions (~0.01%/day lag) but scales linearly with deposit size and stale duration, and is repeatable up to the daily limit each day.

## Recommendation

Add a configurable maximum staleness threshold and revert in `getRate()` if exceeded:

```solidity
uint256 public maxStaleness; // e.g., 24 hours

function getRate() external view returns (uint256) {
    require(
        block.timestamp - lastUpdated <= maxStaleness,
        "Rate is stale"
    );
    return rate;
}
```

This causes `deposit()` to revert automatically when the oracle is stale, eliminating the exploit window without requiring admin intervention. `maxStaleness` should be set conservatively (e.g., 24 hours) and be updatable by the owner.

## Proof of Concept

```solidity
// Foundry fork test against a chain where RSETHRateReceiver is deployed
function testStaleRateOverIssuance() external {
    uint256 R0 = rateReceiver.getRate(); // e.g., 1.05e18

    // Simulate L1 rate increase without LZ delivery
    vm.warp(block.timestamp + 7 days);
    // rateReceiver.rate is still R0; lastUpdated is 7 days old

    uint256 R1 = 1.06e18; // true rate after 7 days of rewards

    uint256 E = 10 ether;
    uint256 rsETHAtStale = E * 1e18 / R0;  // over-issued amount
    uint256 rsETHAtTrue  = E * 1e18 / R1;  // correct amount

    vm.deal(attacker, E);
    vm.prank(attacker);
    pool.deposit{value: E}("ref");

    uint256 minted = wrsETH.balanceOf(attacker);
    assertEq(minted, rsETHAtStale);
    assertGt(minted, rsETHAtTrue, "Over-issued wrsETH confirmed");

    // Pool holds E ETH, can only back rsETHAtTrue rsETH at R1.
    // Deficit = (rsETHAtStale - rsETHAtTrue) * R1 / 1e18 ETH extracted by attacker.
}
```
