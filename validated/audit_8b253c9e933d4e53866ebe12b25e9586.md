### Title
Stale/Invalidated TEE Attestation Not Re-Verified on `respond` / `vote_pk` / `vote_reshared` — (`File: crates/contract/src/tee/tee_state.rs`)

---

### Summary

`is_caller_an_attested_participant` only checks that a stored attestation **exists** and that the account/key identifiers match. It never calls `reverify_participants` to confirm the stored attestation is still valid at call time. As a result, a node whose TEE attestation has expired or whose docker-image hash has been removed from the whitelist (the "write-path invalidation") can still call `respond`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, and `start_reshare_instance` (the "use path") — exactly the same asymmetry as the NFTFloorOracle bug.

---

### Finding Description

**Write path (blocked when attestation becomes invalid):**
`submit_participant_info` calls `attestation.verify_locally(…)` which checks expiry timestamp, docker-image hash, launcher-compose hash, and OS measurements at submission time. [1](#0-0) 

If any of those checks fail, the call is rejected and the stored attestation is not updated.

**Use path (NOT re-validated):**
`is_caller_an_attested_participant` — called by every node-facing method via `assert_caller_is_attested_participant_and_protocol_active` — only verifies that an entry exists in `stored_attestations` and that the `account_id` / `account_public_key` fields match the signer. It never calls `reverify_participants`. [2](#0-1) 

`reverify_participants`, which does perform the full re-check (expiry, docker hash, launcher hash, measurements), is only invoked during `vote_new_parameters` and the post-reshare cleanup sweep: [3](#0-2) 

The `respond` function calls `assert_caller_is_attested_participant_and_protocol_active` and checks `accept_requests`, but neither of those checks re-verifies the stored attestation's current validity: [4](#0-3) 

The same gap exists in `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, and `start_reshare_instance`. [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**Medium — participant-state manipulation that breaks production safety/accounting invariants.**

The system's TEE security model requires that only nodes running a currently-approved image and holding a non-expired attestation may participate in signing and governance. Once a docker-image hash is removed from the whitelist (e.g., because a vulnerability is disclosed in that image version), the intent is that nodes running that image are locked out. However, because `is_caller_an_attested_participant` does not re-verify, those nodes retain full ability to:

- Submit `respond` / `respond_ckd` / `respond_verify_foreign_tx` — contributing threshold-signature shares.
- Cast `vote_pk` / `vote_reshared` — influencing key-generation and resharing outcomes.
- Call `start_keygen_instance` / `start_reshare_instance` — acting as protocol leader.

A node running a compromised image (whose hash was removed precisely because of a security flaw) can continue to participate in all of these operations until `clean_invalid_attestations` is eventually called and its entry is pruned. The window between whitelist removal and cleanup is unbounded in the absence of an explicit trigger.

---

### Likelihood Explanation

**Realistic.** The scenario requires:
1. A vulnerability is found in a deployed MPC node image.
2. Participants vote to remove that image hash from the whitelist.
3. Nodes running the old image can no longer re-attest (`submit_participant_info` fails), but their existing stored attestation remains and is never re-checked on the use path.

This is a normal operational event (image rotation / emergency revocation) and requires no privileged collusion — the attacker is simply a node that was already a legitimate participant but is now running an image that should be revoked.

---

### Recommendation

Add a live re-verification call inside `is_caller_an_attested_participant` (or inside `assert_caller_is_attested_participant_and_protocol_active` in `lib.rs`) that calls `reverify_participants` and rejects the caller if the result is `TeeQuoteStatus::Invalid`. This mirrors the fix recommended for the NFTFloorOracle bug: the "read/use path" must enforce the same validity gate as the "write path."

```rust
// In is_caller_an_attested_participant, after the key-match check:
let tee_status = self.reverify_participants(&attestation.node_id, tee_upgrade_deadline_duration);
if !matches!(tee_status, TeeQuoteStatus::Valid) {
    return Err(AttestationCheckError::AttestationExpiredOrRevoked);
}
```

The `tee_upgrade_deadline_duration` must be threaded through from the contract config, as it already is in `vote_new_parameters`.

---

### Proof of Concept

1. Node A submits a valid attestation for image hash `H1` → stored in `tee_state.stored_attestations`.
2. Participants vote to remove `H1` from the whitelist (e.g., `vote_node_image_hash` with a removal action). `H1` is no longer in `allowed_docker_image_hashes`.
3. Node A attempts `submit_participant_info` again → `attestation.verify_locally` fails because `H1` is not in the allowed set → rejected. ✓ Write path is blocked.
4. Node A calls `respond(request, signature)`. The contract calls `assert_caller_is_attested_participant_and_protocol_active()` → `is_caller_an_attested_participant` → finds Node A's entry in `stored_attestations`, checks `account_id` and `account_public_key` match → returns `Ok(())`. The signature is accepted. ✗ Use path is **not** blocked. [2](#0-1) [3](#0-2)

    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();
        let allowed_measurements = self.get_accepted_measurements();

        let participant_attestation = self.stored_attestations.get(&node_id.tls_public_key);
        let Some(participant_attestation) = participant_attestation else {
            return TeeQuoteStatus::Invalid("participant has no attestation".to_string());
        };

        // Verify the attestation quote
        let time_stamp_seconds = Self::current_time_seconds();
        match participant_attestation.verified_attestation.re_verify(
            time_stamp_seconds,
            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
