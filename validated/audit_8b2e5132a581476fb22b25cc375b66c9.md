### Title
FanOut Transient-Tolerance Allows Single Compromised Bitcoin RPC Provider to Bypass Confirmation Threshold and Drive Unauthorized MPC Signing — (`crates/foreign-chain-inspector/src/lib.rs`)

### Summary

`NotEnoughBlockConfirmations` is classified as a **transient** error in `ForeignChainInspectionError::is_transient`. In a two-provider `FanOut`, when provider_0 (attacker-controlled) returns `Ok(H_attacker)` and provider_1 (honest) returns `NotEnoughBlockConfirmations`, the FanOut's split-detection guard never fires — because it only checks for `successes` vs `non_transient_errors`, and the honest provider's rejection is silently discarded as transient. The FanOut returns `Ok(H_attacker)`, which flows directly into `execute_foreign_chain_request` → `ForeignTxSignPayload` → MPC signing.

### Finding Description

**Root cause — `is_transient` classification:**

`NotEnoughBlockConfirmations` is listed as transient at: [1](#0-0) 

**FanOut split-detection guard:**

The guard that detects a success/failure disagreement is: [2](#0-1) 

`inspectors_split_between_success_and_failure` is only `true` when **both** `successes` and `non_transient_errors` are non-empty. Because `NotEnoughBlockConfirmations` is transient, it is routed to `first_transient_error` (line 105), not `non_transient_errors`. So with provider_0 → `Ok(H_attacker)` and provider_1 → `NotEnoughBlockConfirmations`:

- `successes` = `[(0, [H_attacker])]`
- `non_transient_errors` = `[]`
- `first_transient_error` = `Some(NotEnoughBlockConfirmations{...})`
- `inspectors_split_between_success_and_failure` = **false**

Execution falls through to the success path: [3](#0-2) 

Returns `Ok([H_attacker])` — the honest provider's rejection is completely ignored.

**This behavior is even explicitly tested as correct** in the `tolerate_transient` module: [4](#0-3) 

The test `fan_out__should_tolerate_transient_when_only_one_inspector_succeeds` asserts that a single succeeding inspector with transient failures from all others returns `Ok`. `NotEnoughBlockConfirmations` is a transient error, so this test directly covers the attack scenario.

**Downstream signing path:**

`execute_foreign_chain_request` in the node receives the `Ok` result and constructs the payload: [5](#0-4) [6](#0-5) 

`make_verify_foreign_tx_leader` then signs this payload via the MPC threshold protocol: [7](#0-6) 

There is no downstream re-validation of the extracted block hash. The attacker-supplied `H_attacker` is embedded in the `ForeignTxSignPayload` and signed.

### Impact Explanation

A single compromised Bitcoin RPC provider can cause the MPC network to produce a valid threshold signature over a `ForeignTxSignPayload` containing an attacker-chosen block hash, for a transaction that has not reached the required confirmation threshold. This is unauthorized MPC signing — the honest provider's confirmation-threshold enforcement is bypassed entirely. Any downstream bridge or application that trusts the MPC signature as proof of finality is exposed to forged foreign-chain verification and potential double-spend.

### Likelihood Explanation

The attacker needs to control one of the configured Bitcoin RPC providers. This is realistic: RPC endpoints can be compromised, a malicious provider can be introduced via supply-chain or configuration, or an attacker operating a node can configure a malicious provider as one of the two. The honest provider's `NotEnoughBlockConfirmations` response is a normal, expected condition for any under-confirmed transaction — no special timing or race condition is required.

### Recommendation

`NotEnoughBlockConfirmations` must be treated as **non-transient** in the FanOut context. A confirmation-threshold rejection from an honest provider is a substantive security verdict, not a temporary unavailability. Reclassifying it as non-transient means the split-detection guard at line 118–128 will fire when one provider returns `Ok` and another returns `NotEnoughBlockConfirmations`, producing `InspectorResponseMismatch` instead of silently accepting the attacker's result.

Alternatively, introduce a stricter FanOut policy: any transient error from a provider that co-exists with a success from another provider should be treated as a mismatch rather than tolerated.

### Proof of Concept

Construct a FanOut with two mock inspectors (using the existing `mockall` harness in `tests/fanout.rs`):

```rust
let attacker_provider = mock_returning(ok(vec![42u32])); // H_attacker
let honest_provider = mock_returning(err(|| {
    ForeignChainInspectionError::NotEnoughBlockConfirmations {
        expected: BlockConfirmations::from(6_u64),
        got: BlockConfirmations::from(1_u64),
    }
}));
let fan_out = fan_out_of(vec![attacker_provider, honest_provider]);
let result = fan_out.extract((), (), vec![]).await;
assert_eq!(result.unwrap(), vec![42u32]); // H_attacker accepted — honest provider's rejection ignored
```

This assertion passes against the current code, confirming the vulnerability. The `NotEnoughBlockConfirmations` from the honest provider is silently discarded, and the attacker's value is returned as the canonical extracted value.
