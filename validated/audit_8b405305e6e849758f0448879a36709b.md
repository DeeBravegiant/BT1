### Title
Signature-Bypass LRU Eviction of Legitimate Pending Approvals via Unauthenticated `BlockApproval` Flood — (`chain/client/src/client.rs`)

---

### Summary

`collect_block_approval` stores incoming approvals in `pending_approvals` (an `LruCache<ApprovalInner, …>` bounded by `num_block_producer_seats`) **before** verifying the approval signature, whenever the parent block is not yet known. An unprivileged peer can flood the node with `BlockApproval` messages carrying real validator `account_id`s, garbage signatures, and distinct unknown parent hashes. Each message passes the account-ID-only validator check and occupies a unique LRU slot. After `num_block_producer_seats` such messages the cache is saturated; any subsequent legitimate approval is immediately evicted. When the real parent block later arrives, `on_block_accepted_with_optional_chunk_produce` pops an empty map for the real block hash, and the block producer's Doomslug instance never receives the honest validators' approvals.

---

### Finding Description

**Step 1 – Cache initialization**

`pending_approvals` is created with capacity exactly equal to `num_block_producer_seats` (100 on mainnet): [1](#0-0) 

**Step 2 – Pre-signature insertion path**

In `collect_block_approval`, for an `Endorsement(parent_hash)` whose parent block is not yet in the chain store, `get_epoch_id_from_prev_block` returns `DBNotFoundErr` and the code calls `handle_process_approval_error` with `check_validator = true` and **returns immediately** — the signature check at lines 2436–2444 is never reached: [2](#0-1) 

**Step 3 – Validator-only guard, no signature check**

`handle_process_approval_error` only verifies that `account_id` belongs to a current- or next-epoch validator. It does not verify the cryptographic signature before inserting into the LRU cache: [3](#0-2) 

**Step 4 – Eviction of legitimate approvals**

An attacker sends 101 `BlockApproval` messages, each with:
- `account_id` = any real validator (public information)
- `inner` = `Endorsement(unknown_hash_i)` for i = 1…101 (distinct random hashes not in the chain)
- `signature` = arbitrary garbage bytes

Each message creates a new LRU entry keyed by `Endorsement(unknown_hash_i)`. After 100 insertions the cache is full; the 101st evicts the oldest entry. If a legitimate approval for `Endorsement(real_block_hash)` was inserted before the flood, it is silently dropped.

**Step 5 – Approvals lost on block arrival**

When the real parent block arrives, `on_block_accepted_with_optional_chunk_produce` pops the cache by the real block hash and gets an empty map: [4](#0-3) 

The honest validators' approvals are gone; `collect_block_approval` is never called for them, and Doomslug never accumulates the required stake threshold.

**Step 6 – No default rate limit for `BlockApproval`**

`BlockApproval` appears in `RateLimitedPeerMessageKey` and in `get_key_and_token_cost`, but `standard_preset()` only configures limits for `EpochSyncRequest` and `EpochSyncResponse`. `BlockApproval` has **no default rate limit**: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

The block producer's `Doomslug` instance is missing the evicted approvals. Doomslug requires approvals from validators holding ≥ 2/3 of stake to produce the next block. If enough approvals are evicted, the block producer stalls at the next height until the approval timer fires and validators re-send approvals — introducing a multi-second delay per block, or a complete stall if the attacker sustains the flood.

---

### Likelihood Explanation

- Validator account IDs are public (queryable via RPC).
- Any node can join the NEAR P2P network as a T2 peer and send routed `BlockApproval` messages.
- The cache capacity is only 100 entries on mainnet; 101 messages suffice.
- No rate limit is applied to `BlockApproval` by default.
- The attack is repeatable every block interval (~1 s).

---

### Recommendation

Move the signature verification **before** any insertion into `pending_approvals`. Specifically, in `collect_block_approval`, when `get_epoch_id_from_prev_block` fails with `DBNotFoundErr`, attempt to verify the signature using the current or next epoch's validator key before calling `handle_process_approval_error`. If the signature cannot be verified (epoch not yet known), either drop the approval or use a separate, signature-keyed deduplication structure. Additionally, enable a default rate limit for `BlockApproval` in `standard_preset()`.

---

### Proof of Concept

```rust
// In a test-loop test:
// 1. Set up a node with num_block_producer_seats = N validators.
// 2. Obtain any real validator account_id from the epoch manager.
// 3. Send N+1 BlockApproval messages, each with:
//    - account_id = real_validator
//    - inner = ApprovalInner::Endorsement(CryptoHash::hash_bytes(&[i as u8]))  // distinct unknown hashes
//    - signature = Signature::default()  // garbage
// 4. Then send the legitimate approval for the real parent block hash.
// 5. Assert: client.pending_approvals.get(&ApprovalInner::Endorsement(real_block_hash)) == None
//    (the legitimate approval was evicted).
```
