Audit Report

## Title
Chainlink Price Oracle Lacks Staleness Validation, Enabling Stale-Price rsETH Dilution — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt` and `answeredInRound`, performing no staleness check. When a Chainlink feed goes stale and returns a depressed price, any unprivileged caller can invoke the public `updateRSETHPrice()` to commit the stale price as `rsETHPrice`, then immediately deposit ETH to mint excess rsETH at the expense of existing holders. The downside-protection circuit-breaker only triggers a pause when the price drop exceeds `pricePercentageLimit`; if that variable is zero (unset) or the staleness is within the configured tolerance, the protocol remains open and the exploit succeeds.

## Finding Description

**Root cause — missing staleness check:**

`ChainlinkPriceOracle.getAssetPrice()` at L52 silently discards `updatedAt` and `answeredInRound`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

**Stale price propagation path:**

1. `LRTOracle._getTotalEthInProtocol()` (L339) calls `getAssetPrice(asset)` for every supported LST, summing `totalAssetAmt * assetER`. A stale, depressed price for any large LST directly reduces `totalETHInProtocol`.
2. `_updateRsETHPrice()` (L250) computes `newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply)` — a lower `totalETHInProtocol` yields a lower `newRsETHPrice`.
3. `updateRSETHPrice()` (L87) is `public whenNotPaused` — no role check, callable by anyone.

**Downside-protection bypass:**

Lines 270–282 of `LRTOracle.sol` pause the protocol only when `pricePercentageLimit > 0` AND the price drop exceeds `pricePercentageLimit.mulWad(highestRsethPrice)`. Two bypass conditions exist:
- `pricePercentageLimit == 0` (default/unset): no protection whatsoever; any magnitude of stale-price drop is committed.
- Staleness within the configured tolerance (e.g., 1–3% drop with a 5% limit): the price is committed without a pause.

**Minting exploitation:**

`LRTDepositPool.getRsETHAmountToMint()` (L520) uses the stored `rsETHPrice`:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

ETH is priced at `1e18` (via `OneETHPriceOracle`). With a depressed `rsETHPrice`, the division yields more rsETH per ETH deposited than the fair rate.

## Impact Explanation

**High — Theft of unclaimed yield / dilution of existing rsETH holders.**

An attacker who deposits while `rsETHPrice` is artificially low receives more rsETH than the fair exchange rate entitles them to. When the Chainlink feed recovers and `rsETHPrice` is corrected upward, the attacker's excess rsETH is redeemable at the higher rate — the difference is extracted from existing holders' accrued yield. The attack is repeatable across every staleness event and scales with deposit size.

## Likelihood Explanation

Chainlink feeds go stale without any oracle operator compromise: network congestion, gas price spikes, and heartbeat gaps are documented real-world events. The attacker requires no privileged role — only the ability to observe a stale feed on-chain and call two public functions (`updateRSETHPrice()` and `depositETH()`). The exploit is passive, low-cost, and permissionless.

## Recommendation

Add staleness and sanity validation in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound)
    = priceFeed.latestRoundData();

require(price > 0, "Invalid price");
require(answeredInRound >= roundId, "Stale round");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
```

`MAX_STALENESS` should be configured per-feed based on the feed's documented heartbeat. Additionally, consider requiring `pricePercentageLimit` to be set to a non-zero value before the oracle is operational, and document that the downside circuit-breaker is not a substitute for staleness validation.

## Proof of Concept

1. Chainlink stETH/ETH feed goes stale (e.g., due to network congestion), returning a price 3% below the real rate. `pricePercentageLimit` is either 0 or set to ≥ 3%.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control). `_getTotalEthInProtocol()` uses the stale 3%-low stETH price; `rsETHPrice` is stored 3% below fair value.
3. Attacker calls `LRTDepositPool.depositETH{value: 1000 ether}(0, "")`. `rsethAmountToMint = 1000e18 * 1e18 / rsETHPrice` yields ~1030 rsETH instead of the fair ~1000 rsETH.
4. Chainlink feed recovers; a keeper calls `updateRSETHPrice()` again; `rsETHPrice` returns to fair value.
5. Attacker holds ~30 rsETH of excess value extracted from existing holders, redeemable at the corrected rate.

**Foundry fork test plan:** Fork mainnet, mock the stETH/ETH Chainlink aggregator to return a stale answer (advance `block.timestamp` past the heartbeat without updating `updatedAt`), call `updateRSETHPrice()` as an unprivileged address, record `rsETHPrice`, deposit 1000 ETH, restore the feed, call `updateRSETHPrice()` again, and assert that the attacker's rsETH balance exceeds `1000e18 * 1e18 / correctedRsETHPrice`.
