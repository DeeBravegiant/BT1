### Title
P2P Sync Client Writes Forged State Diff to Storage Without Verifying Against `state_diff_commitment` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

`parse_data_for_block` in `StateDiffStreamBuilder` assembles a `ThinStateDiff` from peer-supplied `StateDiffChunk` messages and writes it to storage via `append_state_diff` without ever computing or comparing the Poseidon hash of the assembled diff against the `state_diff_commitment` field already stored in the block header. A single unauthenticated p2p peer can therefore substitute an arbitrary `class_hash` for any contract address in `deployed_contracts`, causing every subsequent `get_class_hash_at` call for that address to return the attacker-chosen value and directing execution to the wrong contract code.

### Finding Description

`parse_data_for_block` reads `state_diff_length` from the stored header and loops until the accumulated chunk count equals that length: [1](#0-0) 

The only structural checks performed are:

- total chunk count equals `state_diff_length` (lines 72–104)
- no duplicate address/key within a single field (`ConflictingStateDiffParts`)
- no empty chunk [2](#0-1) 

`unite_state_diffs` inserts the peer-supplied `class_hash` directly into `result.deployed_contracts` with no cross-check: [3](#0-2) 

The assembled `ThinStateDiff` is then committed unconditionally: [4](#0-3) 

The block header already carries `state_diff_commitment` (a Poseidon hash over the canonical diff), but `parse_data_for_block` never reads it, never computes the hash of the assembled diff, and never compares the two. A grep for `state_diff_commitment` inside `crates/apollo_p2p_sync/src/client/state_diff.rs` returns zero matches, confirming the check is entirely absent.

The header sync path (`header.rs`) likewise performs no cryptographic verification — it only checks block number ordering and that exactly one signature is present: [5](#0-4) 

No signature validity or block-hash recomputation is performed, so a peer can also supply a header whose `state_diff_commitment` is forged. Either way — whether the header is canonical or forged — the state diff content is never authenticated.

### Impact Explanation

After `append_state_diff` commits the forged diff, `get_class_hash_at` returns the attacker-chosen class hash for the targeted contract address at every `StateNumber` from that block onward. The execution layer uses this value to select the Sierra/CASM artifact to run for calls to that contract, meaning the wrong contract code is executed. This maps directly to the Critical allowed impact: *"Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution."*

### Likelihood Explanation

Any node that participates in the p2p network and accepts sync connections is reachable. The attacker needs only to respond to a `StateDiffQuery` with chunks whose total `len()` equals `target_state_diff_len` but whose `class_hash` fields are substituted. No privileged position is required; the p2p layer is explicitly unauthenticated.

### Recommendation

After assembling the full `ThinStateDiff`, compute its Poseidon commitment (using the same `calculate_state_diff_hash` already used in `apollo_committer`) and compare it against `block_header.state_diff_commitment`. Reject with `BadPeerError` on mismatch before calling `write_to_storage`. This is the same pattern the committer already enforces via `verify_state_diff_hash`: [6](#0-5) 

The check should be inserted at line 106, immediately after `validate_deprecated_declared_classes_non_conflicting` and before `Ok(Some(...))`.

### Proof of Concept

1. Canonical block N has `state_diff_length = 1` and `deployed_contracts = { A → canonical_hash }`.
2. Attacker peer responds to the `StateDiffQuery` with one `ContractDiff { contract_address: A, class_hash: Some(attacker_hash) }` — length 1, satisfying the loop termination condition.
3. `unite_state_diffs` inserts `(A, attacker_hash)` into `result.deployed_contracts`.
4. `write_to_storage` calls `append_state_diff(N, result)` and commits.
5. Any subsequent `get_class_hash_at(A, state_number >= N)` returns `attacker_hash`.
6. Execution dispatches calls to contract A using the CASM artifact registered under `attacker_hash` instead of `canonical_hash`.
