Audit Report

## Title
Precision Loss in `rewardRate` Calculation Causes Permanent Freezing of Unclaimed Yield - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`KernelDepositPool.notifyRewardAmount()` stores `rewardRate` as a plain integer division result with no `1e18` scaling. This causes two irreversible losses: a guaranteed dust remainder locked on every reward notification, and per-checkpoint accumulator increments that round to zero whenever `rewardRate * timeDelta * 1e18 < totalKernelStaked`. Because `updatedAt` is always advanced by the `updateReward` modifier regardless of whether the increment was non-zero, any rewards that should have accrued during that interval are permanently unrecoverable.

## Finding Description

`notifyRewardAmount()` sets `rewardRate` via unscaled integer division: [1](#0-0) 

`rewardPerToken()` then multiplies `rewardRate` by `DECIMAL_PRECISION` (1e18) downstream: [2](#0-1) 

For the accumulator increment to be non-zero, the condition `rewardRate * timeDelta * 1e18 >= totalKernelStaked` must hold. When `rewardRate` is small (e.g., 1, from a low-decimal reward token like USDC with a large duration), this condition fails for any realistic `totalKernelStaked`.

The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` on every `stake`, `initiateWithdrawal`, or `getReward` call: [3](#0-2) 

This means each checkpoint that rounds to zero permanently discards the rewards that should have accrued since the last checkpoint — there is no catch-up mechanism. The `rewardRate == 0` guard at line 586 only prevents the trivially zero case and does not protect against `rewardRate = 1` with a large staked supply. [4](#0-3) 

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Two distinct losses occur:

1. **Dust loss per period**: `receivedAmount % duration` wei are permanently locked on every `notifyRewardAmount` call. For a 30-day duration (2,592,000 s) with USDC (6 decimals) and `receivedAmount = 3,000,000`: `3,000,000 % 2,592,000 = 408,000` wei ≈ 0.408 USDC locked forever per period.

2. **Per-checkpoint nullification**: With `rewardRate = 1`, `timeDelta = 1 s`, `totalKernelStaked = 1e19`: increment = `1 * 1 * 1e18 / 1e19 = 0`. `rewardPerTokenStored` never advances. After the full 30-day period, `earned()` returns 0 despite 2.592 USDC having been deposited. The reward tokens remain in the contract with no recovery path. [5](#0-4) 

## Likelihood Explanation

No attacker action is required. Any unprivileged user calling `stake`, `initiateWithdrawal`, or `getReward` triggers `updateReward`, which advances `updatedAt` and discards the zero-increment checkpoint. The scenario is directly reachable whenever the reward token has fewer than 18 decimals (USDC, USDT) or when the reward amount is small relative to `duration`. The `rewardRate == 0` revert does not prevent `rewardRate = 1` with a large `totalKernelStaked`.

## Recommendation

Store `rewardRate` with `1e18` precision at the point of assignment in `notifyRewardAmount()`:

```solidity
// New period:
rewardRate = receivedAmount * DECIMAL_PRECISION / duration;
// Renewal (remaining is already scaled since rewardRate is now scaled):
uint256 remaining = (finishAt - block.timestamp) * rewardRate;
rewardRate = (receivedAmount * DECIMAL_PRECISION + remaining) / duration;
```

Then remove the `* DECIMAL_PRECISION` factor from `rewardPerToken()`:

```solidity
return rewardPerTokenStored
    + (rewardRate * (lastTimeRewardApplicable() - updatedAt))
    / totalKernelStaked;
```

This ensures the accumulator increment is non-zero for any realistic `timeDelta` and eliminates the dust remainder loss.

## Proof of Concept

**Setup**: `duration = 2_592_000` (30 days), reward token = USDC (6 decimals), `receivedAmount = 3_000_000` (3 USDC), `totalKernelStaked = 1e19` (10 KERNEL, 18 decimals).

**Step 1 — dust loss**:
```
rewardRate = 3_000_000 / 2_592_000 = 1
locked forever = 3_000_000 - 1 * 2_592_000 = 408_000 wei ≈ 0.408 USDC
```

**Step 2 — per-checkpoint nullification**:
Any user calls `stake(0)` (or any `updateReward`-gated function) each second:
```
increment = rewardRate * timeDelta * 1e18 / totalKernelStaked
          = 1 * 1 * 1e18 / 1e19 = 0
```
`rewardPerTokenStored` never increases; `updatedAt` advances each call.

**Step 3 — funds locked**:
After 30 days, `earned(alice) = 0`. The contract holds 2.592 USDC (3 USDC minus 0.408 USDC dust) with no sweep or recovery function. `getReward()` transfers 0.

**Foundry test outline**:
```solidity
function test_rewardRatePrecisionLoss() public {
    // Deploy with USDC as rewardsToken, duration = 2_592_000
    // Stake 1e19 KERNEL for alice
    // Admin calls notifyRewardAmount(3_000_000)

    // Warp 2_592_000 seconds, call updateReward each second (or warp directly)
    // Assert earned(alice) == 0

}
```
