### Title
Permissionless Pool Reward Absorb Resets `LEAVING_POOL` Relative Lock Height, Blocking Pool Exit — (File: `chia/pools/pool_puzzles.py`)

### Summary
The pool singleton's absorb-reward spend path requires **no signature** from the owner. Any unprivileged party can submit a valid absorb spend bundle for any pool wallet singleton. When the singleton is in `LEAVING_POOL` (waiting-room) state, each absorb spend destroys the current singleton coin and creates a new one at the current block height, resetting the `ASSERT_HEIGHT_RELATIVE(relative_lock_height)` countdown. An attacker who repeatedly absorbs farming rewards as they appear can permanently prevent the farmer from completing the pool-exit transition.

---

### Finding Description

**Permissionless absorb path**

`create_absorb_spend` in `chia/pools/pool_puzzles.py` constructs a spend bundle that carries an empty aggregate signature (`G2Element()`): [1](#0-0) 

The wallet layer itself documents this explicitly: [2](#0-1) 

The lifecycle test confirms the absorb bundle is accepted on-chain with `G2Element()`: [3](#0-2) 

**Lock-height reset mechanism**

The waiting-room inner puzzle is curried with `relative_lock_height` and enforces `ASSERT_HEIGHT_RELATIVE(relative_lock_height)` when the farmer attempts to exit: [4](#0-3) 

`ASSERT_HEIGHT_RELATIVE(N)` is evaluated against the **coin being spent**: the current block height must be ≥ `coin_creation_height + N`. When an absorb spend executes, the existing singleton coin is destroyed and a new singleton coin is created at the current block height. The new coin's age starts at zero, so the farmer must wait a full `relative_lock_height` from the absorb block before they can exit — regardless of how long they had already waited.

The lifecycle test explicitly exercises absorb-while-in-waiting-room and then exits only after a very large block jump (height 3 → 10 000, far exceeding the 5 000-block lock): [5](#0-4) 

**Attacker entry path**

All information needed to construct the absorb spend is public on-chain:
- The singleton's current coin ID and puzzle reveal (observable from the UTXO set).
- The farming reward coin (`p2_singleton_puzzle_hash`, amount, parent derived from `pool_parent_id(height, genesis_challenge)`).

The attacker monitors the chain for new `p2_singleton` reward coins and submits an absorb spend bundle before (or instead of) the farmer, with no credentials required.

---

### Impact Explanation

Every time a farming reward appears and the attacker absorbs it, the singleton coin is recreated at the current height and the `relative_lock_height` countdown resets. As long as the farmer continues farming (producing rewards), the attacker can repeat this indefinitely. The farmer is permanently trapped in `LEAVING_POOL` state and cannot transition to `SELF_POOLING` or a new pool. This constitutes a **permanent or long-lived inability for an honest farmer to process valid pool actions** — specifically the pool-exit state transition — matching the High impact tier.

The `PoolWallet.new_peak` logic that auto-submits the second travel transaction also depends on the singleton tip coin: [6](#0-5) 

Each attacker-driven absorb changes the tip coin, causing the wallet's pending exit transaction to reference a spent coin, invalidating it and forcing the farmer to resubmit — only to have the countdown reset again.

---

### Likelihood Explanation

- The attacker requires no keys, no privileged access, and no prior relationship with the victim.
- The only prerequisite is the existence of a `p2_singleton` farming reward coin, which appears whenever the farmer wins a block — a routine event for any active farmer.
- The singleton state and reward coins are fully public; the attacker needs only to watch the chain and front-run (or simply submit before) the farmer's absorb.
- The `MINIMUM_RELATIVE_LOCK_HEIGHT` is 5 blocks and `MAXIMUM_RELATIVE_LOCK_HEIGHT` is 1 000 blocks, so even a modest farming operation produces rewards well within any lock window. [7](#0-6) 

---

### Recommendation

Restrict the absorb spend path so that it can only be submitted by the singleton owner (or the pool, depending on state). Concretely:

1. **Add an `AGG_SIG_ME` condition** in the waiting-room inner puzzle's absorb path, requiring the owner's public key to sign the spend. The owner pubkey is already curried into the puzzle and available for this purpose.
2. Alternatively, **require the absorb spend to be co-signed with the exit spend** when the singleton is in `LEAVING_POOL` state, so that absorbing and exiting are atomic.
3. At minimum, **document the reset behavior** and add a wallet-layer guard that refuses to enter `LEAVING_POOL` state while rewards are pending, so the farmer can drain rewards before declaring intent to leave.

---

### Proof of Concept

```
1. Farmer calls pw_self_pool / pw_join_pool → singleton enters LEAVING_POOL state at block H.
   relative_lock_height = L (e.g. 100 blocks).
   Farmer must wait until block H + L to exit.

2. At block H + 50, farmer farms a block → p2_singleton reward coin R appears on-chain.

3. Attacker observes R, constructs create_absorb_spend(last_coin_spend, LEAVING_POOL_state,
   launcher_coin, height=H+50, ...) → SpendBundle([singleton_spend, R_spend], G2Element()).
   Submits to mempool (no signature required).

4. Bundle is accepted. Singleton coin is recreated at block H+50. Farmer's pending exit
   transaction references the old (now spent) singleton coin → invalid.

5. Farmer must resubmit exit transaction. New singleton coin was created at H+50,
   so exit is now blocked until H+50+L = H+150.

6. Attacker repeats at every subsequent reward. Farmer never exits the pool.
```

The absorb spend with `G2Element()` is accepted by consensus because the pool member and waiting-room CLVM puzzles do not require a signature on the absorb code path: [8](#0-7)

```python
def create_absorb_spend(
    last_coin_spend: CoinSpend,
    current_state: PoolState,
    launcher_coin: Coin,
    height: uint32,
    genesis_challenge: bytes32,
    delay_time: uint64,
    delay_ph: bytes32,
) -> list[CoinSpend]:
    inner_puzzle: Program = pool_state_to_inner_puzzle(
        current_state, launcher_coin.name(), genesis_challenge, delay_time, delay_ph
    )
    reward_amount: uint64 = calculate_pool_reward(height)
    if is_pool_member_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol: Program = Program.to([reward_amount, height])
    elif is_pool_waitingroom_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, destination_puzhash, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol = Program.to([0, reward_amount, height])
    else:
        raise ValueError
    # full sol = (parent_info, my_amount, inner_solution)
    coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(last_coin_spend)
    assert coin is not None

    if coin.parent_coin_info == launcher_coin.name():
        parent_info: Program = Program.to([launcher_coin.parent_coin_info, launcher_coin.amount])
    else:
        p = Program.from_bytes(bytes(last_coin_spend.puzzle_reveal))
        last_coin_spend_inner_puzzle: Program | None = get_inner_puzzle_from_puzzle(p)
        assert last_coin_spend_inner_puzzle is not None
        parent_info = Program.to(
            [
                last_coin_spend.coin.parent_coin_info,
                last_coin_spend_inner_puzzle.get_tree_hash(),
                last_coin_spend.coin.amount,
            ]
        )
    full_solution: SerializedProgram = SerializedProgram.to([parent_info, last_coin_spend.coin.amount, inner_sol])
    full_puzzle: SerializedProgram = create_full_puzzle(inner_puzzle, launcher_coin.name()).to_serialized()
    assert coin.puzzle_hash == full_puzzle.get_tree_hash()

    reward_parent: bytes32 = pool_parent_id(height, genesis_challenge)
    p2_singleton_puzzle = create_p2_singleton_puzzle(
        SINGLETON_MOD_HASH, launcher_coin.name(), delay_time, delay_ph
    ).to_serialized()
    reward_coin: Coin = Coin(reward_parent, p2_singleton_puzzle.get_tree_hash(), reward_amount)
    p2_singleton_solution = SerializedProgram.to([inner_puzzle.get_tree_hash(), reward_coin.name()])
    assert p2_singleton_puzzle.get_tree_hash() == reward_coin.puzzle_hash
    assert full_puzzle.get_tree_hash() == coin.puzzle_hash
    assert get_inner_puzzle_from_puzzle(Program.from_bytes(bytes(full_puzzle))) is not None

    coin_spends = [
        CoinSpend(coin, full_puzzle, full_solution),
        CoinSpend(reward_coin, p2_singleton_puzzle, p2_singleton_solution),
    ]
    return coin_spends
```
