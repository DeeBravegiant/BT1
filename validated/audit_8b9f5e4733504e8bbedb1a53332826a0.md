Audit Report

## Title
ETH Deposit Limit Check Ignores Deposit Amount, Allowing Cap Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` evaluates the ETH deposit cap using only the pre-deposit running total, omitting the incoming `amount`. Any unprivileged caller can deposit an arbitrarily large ETH amount as long as the current total has not yet crossed the configured limit, bypassing the protocol's ETH deposit ceiling entirely. The identical check for ERC-20 assets correctly includes `amount`.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch returns `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` while the ERC-20 branch returns `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`. The `amount` parameter is silently discarded for ETH. This function is the sole deposit-limit guard invoked by `_beforeDeposit` (L661), which is called unconditionally by the public, permissionless `depositETH` entry point (L87). No other check in the call chain accounts for the magnitude of the incoming ETH deposit. Existing guards (`nonReentrant`, `whenNotPaused`, `onlySupportedAsset`) are unrelated to the limit enforcement and do not compensate for the missing `+ amount`.

## Impact Explanation
The ETH deposit limit is a protocol-level safety constraint bounding total ETH exposure. Because the check never adds the incoming deposit to the running total, the cap is unenforceable: a single transaction can push total ETH deposits to an arbitrary multiple of the configured ceiling. This matches the allowed Low impact: **Contract fails to deliver promised returns / enforce promised constraints, but does not directly lose value.**

## Likelihood Explanation
`depositETH` is fully permissionless — no role, whitelist, or special condition is required. The only prerequisite is that `getTotalAssetDeposits(ETH_TOKEN)` has not yet naturally crossed the limit, which is the normal operating state. The bug is therefore reachable by any depositor at any time before the cap is organically hit, and is repeatable across multiple transactions.

## Recommendation
Add `amount` to the ETH branch, consistent with the ERC-20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Configure `depositLimitByAsset(ETH_TOKEN) = 1_000 ether`; assume `getTotalAssetDeposits(ETH_TOKEN) = 999 ether`.
2. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 10_000 ether)` evaluates `999 > 1000` → `false`; no revert.
4. `_mintRsETH` executes; total ETH deposits become 10 999 ether — 11× the intended cap.
5. An identical deposit of any ERC-20 asset would have evaluated `999 + 10_000 > 1_000` → `true` and reverted with `MaximumDepositLimitReached`.

Foundry test plan: deploy with `depositLimitByAsset = 1_000 ether`, seed `999 ether` via a prior deposit, then call `depositETH{value: 10_000 ether}` from an unprivileged address and assert `getTotalAssetDeposits(ETH_TOKEN) > depositLimitByAsset(ETH_TOKEN)` after the call succeeds.
