### Title
Reentrancy in `initTransfer1155()` via Malicious ERC1155 Token Allows Emission of Unbacked `InitTransfer` Events - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`initTransfer1155()` in `OmniBridge.sol` increments `currentOriginNonce` at the top of the function but reads it again at the bottom when emitting the `InitTransfer` event. Between those two points, it makes an external call to an attacker-controlled ERC1155 contract via `safeTransferFrom`. A malicious ERC1155 token can reenter `initTransfer1155()` during that call, causing `currentOriginNonce` to be incremented a second time. Both the reentrant call and the outer call then emit `InitTransfer` with the same (final) nonce value, while no real tokens are ever locked in the bridge. The NEAR side treats any emitted `InitTransfer` event as proof that tokens are held, so the attacker obtains cross-chain credit for tokens that were never deposited.

### Finding Description

`initTransfer1155()` has the following execution order:

1. `currentOriginNonce += 1` — nonce incremented to N+1 [1](#0-0) 
2. `IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "")` — external call to attacker-controlled token [2](#0-1) 
3. `emit BridgeTypes.InitTransfer(msg.sender, deterministicToken, currentOriginNonce, ...)` — reads `currentOriginNonce` at emit time, not at increment time [3](#0-2) 

The nonce is **never captured into a local variable** before the external call. The `emit` statement reads the storage variable `currentOriginNonce` directly.

There is no reentrancy guard on `initTransfer1155()`. The `onERC1155Received` hook check (`operator != address(this)`) does not protect against this attack because the reentrancy path goes directly into `initTransfer1155()`, not through `onERC1155Received`. [4](#0-3) 

There is also no registration requirement — `initTransfer1155()` accepts any arbitrary `tokenAddress` without checking `multiTokens` or any allowlist, so the attacker can supply a freshly deployed malicious contract. [5](#0-4) 

The documented security invariant is violated:

> **Event–transfer atomicity**: `InitTransfer` must only be emitted in a code path where tokens have already been burned/locked in the same transaction. [6](#0-5) 

### Impact Explanation

The NEAR side treats any on-chain `InitTransfer` event as authoritative proof that tokens are held on EVM. Two `InitTransfer` events are emitted with the same `originNonce` (N+2), while nonce N+1 is permanently skipped and zero real tokens are locked. The NEAR side will attempt to credit the attacker with tokens that have no EVM backing, breaking the bridge's backing guarantee. This falls under **Critical — unauthorized creation of wrapped bridge assets through verification failure**.

### Likelihood Explanation

The entry point `initTransfer1155()` is fully public and permissionless. The attacker only needs to deploy a malicious ERC1155 contract (trivial) and call `initTransfer1155()` with it. No privileged role, leaked key, or external dependency compromise is required.

### Recommendation

Capture `currentOriginNonce` into a local variable immediately after incrementing it, before any external call, and use that local variable in all subsequent reads (`initTransferExtension`, `emit`). This is the same fix already applied correctly in `initTransfer()` — note that `initTransfer()` also reads `currentOriginNonce` after external calls, so it should receive the same fix. Additionally, add a reentrancy guard (e.g., OpenZeppelin `ReentrancyGuardUpgradeable`) to `initTransfer1155()` and `initTransfer()`.

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) nonReentrant {
    currentOriginNonce += 1;
    uint64 nonce = currentOriginNonce; // capture before external call
    ...
    IERC1155(tokenAddress).safeTransferFrom(...);
    ...
    initTransferExtension(msg.sender, deterministicToken, nonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, deterministicToken, nonce, ...);
}
```

### Proof of Concept

1. Attacker deploys `MaliciousERC1155` whose `safeTransferFrom` reenters `bridge.initTransfer1155(address(this), tokenId, amount, 0, 0, "attacker.near", "")` once, then returns normally on the second call.
2. Attacker calls `bridge.initTransfer1155(MaliciousERC1155, tokenId, amount, 0, 0, "attacker.near", "")`.
3. Bridge: `currentOriginNonce` → 1. Calls `MaliciousERC1155.safeTransferFrom(attacker, bridge, ...)`.
4. Reentrant call: `currentOriginNonce` → 2. Calls `MaliciousERC1155.safeTransferFrom(...)` again (no reentry). Emits `InitTransfer(originNonce=2)`. Returns.
5. Outer call resumes: reads `currentOriginNonce=2`. Emits `InitTransfer(originNonce=2)` again.
6. Result: two `InitTransfer` events with `originNonce=2`, nonce 1 skipped, zero tokens locked. NEAR side credits attacker for two transfers backed by nothing.
