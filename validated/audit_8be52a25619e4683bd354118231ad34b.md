Audit Report

## Title
Permissionless `sendFunds()` Combined with FeeReceiver TVL Exclusion Enables Theft of Unclaimed Yield - (File: contracts/FeeReceiver.sol)

## Summary

`FeeReceiver.sendFunds()` carries no access-control modifier, allowing any address to move accumulated MEV/execution-layer rewards into `LRTDepositPool` at will. Because `getETHDistributionData()` explicitly excludes the `FeeReceiver` balance from TVL, the stored `rsETHPrice` is stale with respect to those pending rewards. An attacker deposits ETH at the stale (lower) price, triggers `sendFunds()` to inflate the deposit pool balance, and after the price update redeems rsETH at the higher price — extracting yield that belongs to pre-existing rsETH holders.

## Finding Description

**Root cause 1 — FeeReceiver balance is structurally excluded from TVL.**

`LRTDepositPool.getETHDistributionData()` sets `ethLyingInDepositPool = address(this).balance` and sums NDC balances, EigenLayer shares, the unstaking vault, and the converter — but never reads the `FeeReceiver` address balance. The inline comment at line 465 even documents this intentional gap: *"rewards are not accounted here / it will automatically be accounted once it is moved from feeReceiver/rewardReceiver to depositPool."* [1](#0-0) 

**Root cause 2 — `sendFunds()` has no access control.**

The function is declared `external` with no role modifier, so any EOA or contract can call it at any time, instantly moving the entire `FeeReceiver` ETH balance into `LRTDepositPool`. [2](#0-1) 

**Root cause 3 — Deposits use the stored, stale `rsETHPrice`.**

`getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, which is a state variable last written by a prior `_updateRsETHPrice()` call. It does not recompute TVL on-the-fly. [3](#0-2) 

**Root cause 4 — `updateRSETHPrice()` is also permissionless.**

The public entry point carries only `whenNotPaused`; no role check exists. [4](#0-3) 

**Exploit flow:**

1. Rewards accumulate in `FeeReceiver` (e.g. 10 ETH). `rsETHPrice` is stale at `1.05 ETH/rsETH`.
2. Attacker calls `depositETH{value: 100 ETH}` → receives `≈95.24 rsETH` at the stale price.
3. Attacker calls `FeeReceiver.sendFunds()` → 10 ETH moves to `LRTDepositPool`.
4. `updateRSETHPrice()` is called (by attacker or manager) → new price reflects the extra 10 ETH, rising to `≈1.055 ETH/rsETH`.
5. After the withdrawal delay, attacker redeems `95.24 rsETH` for `≈100.48 ETH` — a profit of `≈0.48 ETH` extracted from yield owed to pre-existing holders.

**Why the `pricePercentageLimit` guard does not prevent the attack:**

If the reward jump is large enough to exceed `pricePercentageLimit`, a non-manager call to `updateRSETHPrice()` reverts with `PriceAboveDailyThreshold`. [5](#0-4) 

However, the attacker has already locked in the favourable entry price at step 2. They simply wait for the manager's routine `updateRSETHPriceAsManager()` call to finalise the price update. For smaller reward amounts that fall within the limit, the attacker can call `updateRSETHPrice()` themselves in the same transaction sequence.

## Impact Explanation

The attacker captures a portion of MEV/execution-layer rewards that should accrue exclusively to pre-existing rsETH holders. This is a direct, repeatable **theft of unclaimed yield** — a **High** severity impact under the allowed scope. The stolen amount scales with the size of the `FeeReceiver` balance at the time of the attack and the attacker's deposit size.

## Likelihood Explanation

- No privileged role is required; any EOA can execute the full sequence.
- The precondition (rewards sitting in `FeeReceiver`) is the normal, continuous operating state of the protocol.
- The attacker only needs to monitor the `FeeReceiver` balance on-chain, deposit, call `sendFunds()`, and wait for the withdrawal delay.
- The `pricePercentageLimit` guard introduces a delay for large reward jumps but does not eliminate the attack — it merely requires the attacker to wait for the manager's next price update.

Likelihood is **Medium-High**.

## Recommendation

1. **Restrict `sendFunds()`** to a trusted role (e.g. `LRTConstants.MANAGER`) so reward distribution timing cannot be controlled by an attacker.
2. **Include the `FeeReceiver` balance in `getETHDistributionData()`** so pending rewards are always reflected in the rsETH price, eliminating the price-lag window entirely.
3. Alternatively, **atomically call `_updateRsETHPrice()` inside `sendFunds()`** so the price is updated in the same transaction as the reward transfer, preventing any deposit at the stale price.

## Proof of Concept

**Minimal transaction sequence (local fork):**

```solidity
// 1. Confirm FeeReceiver has accumulated rewards (e.g. 10 ETH)
assertGt(address(feeReceiver).balance, 0);

// 2. Record stale price
uint256 stalePriceBeforeSendFunds = lrtOracle.rsETHPrice(); // e.g. 1.05e18

// 3. Attacker deposits at stale price
vm.prank(attacker);
lrtDepositPool.depositETH{value: 100 ether}(0, "");
uint256 rsethReceived = rsETH.balanceOf(attacker); // ≈ 95.24e18

// 4. Attacker (or anyone) calls sendFunds — no role check
vm.prank(attacker);
feeReceiver.sendFunds();

// 5. Price update now reflects the extra 10 ETH
lrtOracle.updateRSETHPrice();
uint256 newPrice = lrtOracle.rsETHPrice(); // > stalePriceBeforeSendFunds
assertGt(newPrice, stalePriceBeforeSendFunds);

// 6. After withdrawal delay, attacker redeems rsETH at higher price
// rsethReceived * newPrice / 1e18 > 100 ETH deposited
// => profit extracted from pre-existing holders' yield
```

**Foundry fork test plan:** Fork mainnet, seed `FeeReceiver` with ETH via `vm.deal`, execute the sequence above, assert that `attacker`'s redeemed ETH exceeds their deposit, and assert that the per-rsETH value for a pre-existing holder has decreased relative to a baseline run where `sendFunds()` is called before the attacker's deposit.
