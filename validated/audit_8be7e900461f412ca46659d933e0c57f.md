Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Mint Calculation Allows Depositors to Receive Excess rsETH at Existing Holders' Expense - (File: `contracts/LRTDepositPool.sol`)

## Summary
`LRTOracle` stores the rsETH/ETH exchange rate as a plain state variable (`rsETHPrice`) that is only updated when `updateRSETHPrice()` or `updateRSETHPriceAsManager()` is explicitly called. Neither `depositETH` nor `depositAsset` refreshes this price before computing the rsETH mint amount, meaning any depositor can exploit the staleness window between reward accrual and the next price update to receive more rsETH than their deposit is worth, permanently diluting existing holders' accrued yield.

## Finding Description
`LRTOracle.rsETHPrice` is a plain `uint256` state variable updated only by explicit calls to `updateRSETHPrice()` (public, `whenNotPaused`) or `updateRSETHPriceAsManager()` (manager-only):

```solidity
// LRTOracle.sol:28
uint256 public override rsETHPrice;

// LRTOracle.sol:87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`LRTDepositPool.getRsETHAmountToMint` reads this stored value directly:

```solidity
// LRTDepositPool.sol:520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`_beforeDeposit` is declared `private view`, making it structurally incapable of calling any state-mutating price refresh. It calls `getRsETHAmountToMint` with no price update:

```solidity
// LRTDepositPool.sol:648-670
function _beforeDeposit(...) private view returns (uint256 rsethAmountToMint) {
    ...
    rsethAmountToMint = getRsETHAmountToMint(asset, depositAmount);
    ...
}
```

Both `depositETH` (L87) and `depositAsset` (L111) invoke `_beforeDeposit` with no prior call to `updateRSETHPrice()`. The mint formula `(depositAmount Ă— assetPrice) / rsETHPrice` uses a denominator that is too small whenever staking rewards have accrued since the last price update, inflating `rsethAmountToMint`.

The `minRSETHAmountExpected` slippage guard protects depositors from receiving *less* than expected; it provides no protection to existing holders against a depositor receiving *more* than the true rate entitles them to. The `pricePercentageLimit` guard in `_updateRsETHPrice` limits the magnitude of a single price step but does not prevent deposits from occurring against a stale price. No other on-chain check closes this window.

Additionally, if the accrued yield causes the true price to exceed `pricePercentageLimit` relative to `highestRsethPrice`, a non-manager call to `updateRSETHPrice()` will revert with `PriceAboveDailyThreshold` (L260â€“265), potentially extending the staleness window further.

## Impact Explanation
**High â€” Theft of unclaimed yield.** Staking rewards (stETH rebases, EigenLayer ETH rewards) continuously increase the true rsETH/ETH rate. Every depositor who deposits while `rsETHPrice` is stale receives rsETH at a below-true-market rate, capturing a share of the yield that should have accrued to existing holders. When `updateRSETHPrice()` is eventually called, the price jumps to reflect accrued rewards, but the depositor already holds the excess rsETH. The yield is permanently transferred from existing holders to the depositor. The magnitude scales with deposit size and staleness duration; during bot outages or high-reward periods, both factors increase.

## Likelihood Explanation
**Medium.** `updateRSETHPrice()` is called by off-chain bots on a periodic schedule, creating a predictable and recurring staleness window after every reward accrual event. No privileged access is required. An attacker can passively monitor `rsETHPrice` on-chain, reconstruct the true TVL from public data (asset prices via `getAssetPrice`, total deposits via `getTotalAssetDeposits`), and deposit precisely when the stored price is most stale. The entry path is the standard public `depositAsset`/`depositETH` interface. The attack is repeatable every update cycle.

## Recommendation
Call `_updateRsETHPrice()` (the internal function, to avoid the manager-gated `PriceAboveDailyThreshold` revert that would DoS public deposits) at the start of `depositAsset` and `depositETH`, before `_beforeDeposit` is invoked. Because `_beforeDeposit` is `private view`, the price refresh must be placed in the non-`view` outer deposit functions. The same refresh should be applied to `initiateWithdrawal` for symmetry. Alternatively, restructure `_beforeDeposit` to be non-`view` and call `_updateRsETHPrice()` internally so every deposit atomically refreshes the price before minting.

## Proof of Concept
Foundry fork test outline:

1. Fork mainnet at a block where `rsETHPrice` is set and TVL > 0.
2. Advance time (e.g., `vm.warp(block.timestamp + 1 days)`) to simulate staking reward accrual; asset oracle prices (stETH/ETH) will have increased.
3. Record `rsETHPrice_stored = lrtOracle.rsETHPrice()`.
4. Compute `rsETHPrice_true = lrtOracle._getTotalEthInProtocol() / rsETH.totalSupply()` (reconstruct off-chain or via a helper).
5. Assert `rsETHPrice_true > rsETHPrice_stored` (staleness confirmed).
6. As attacker, call `lrtDepositPool.depositETH{value: 1 ether}(0, "")`.
7. Record `attackerRsETH = rsETH.balanceOf(attacker)`.
8. Assert `attackerRsETH > 1 ether * 1e18 / rsETHPrice_true` (excess minted).
9. Call `lrtOracle.updateRSETHPrice()`.
10. Compute `attackerETHValue = attackerRsETH * lrtOracle.rsETHPrice() / 1e18`.
11. Assert `attackerETHValue > 1 ether` (attacker's rsETH is now worth more than deposited, at existing holders' expense).
