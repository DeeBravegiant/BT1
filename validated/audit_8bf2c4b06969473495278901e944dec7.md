### Title
Off-by-One in `valid_entries` Cutoff Includes Expired `NodeImageHash` in Allowed Set — (`crates/contract/src/tee/proposal.rs`)

---

### Summary

`AllowedDockerImageHashes::valid_entries` uses `rposition` to find the last expired entry and then slices from that index onward with `get(cutoff_index..)`. Because the slice is **inclusive** of `cutoff_index`, the expired entry at the cutoff is returned as part of the valid set. A node running an image hash whose grace period has passed can therefore submit a fresh attestation that the contract accepts, bypassing the TEE upgrade deadline invariant.

---

### Finding Description

The logic in `valid_entries`:

```rust
let cutoff_index = self
    .allowed_tee_proposals
    .iter()
    .rposition(|allowed_docker_image| {
        ...
        grace_period_deadline < current_time   // true = expired
    })
    .unwrap_or(0);

self.allowed_tee_proposals
    .get(cutoff_index..)   // ← inclusive: cutoff entry is returned
    .unwrap_or(&[])
    .to_vec()
``` [1](#0-0) 

`rposition` returns the **rightmost** index where the closure is `true` (i.e., the most recently expired entry). The subsequent `get(cutoff_index..)` slice **includes** that expired entry.

**Concrete trace** — list `[H1(added=T1), H2(added=T2)]`, grace period `G`, current time `T` where `T1+G < T ≤ T2+G`:

| Index | Entry | `grace_period_deadline < T` |
|-------|-------|-----------------------------|
| 0 | H1 | `true` (expired) |
| 1 | H2 | `false` (valid) |

`rposition` scans right-to-left, finds `true` at index 0 → `cutoff_index = 0`.  
`get(0..)` → `[H1, H2]`. **H1 is included despite being expired.**

The intended result is `[H2]` only.

The `unwrap_or(0)` fallback is correct for the "no expired entries" case (returns everything). The "all entries expired" case also works correctly: `rposition` returns the last index, `get(last..)` keeps just the newest entry. The bug is exclusively in the **mixed** case: some entries expired, some valid.

---

### Impact Explanation

`valid_entries` feeds directly into attestation acceptance and re-verification:

- `add_participant` calls `get_allowed_mpc_docker_image_hashes` → `valid_entries` to build the allowlist passed to `attestation.verify_locally`. [2](#0-1) 

- `reverify_participants` calls the same path for ongoing re-verification. [3](#0-2) 

A node running H1 (grace period expired) can call `submit_participant_info` with a fresh attestation bound to H1. The contract will accept it because H1 appears in the returned allowlist. The node remains an active, attested participant past the upgrade deadline, violating the TEE enforcement invariant.

---

### Likelihood Explanation

The window is the interval `(T1+G, T2+G]` — the time between H1's deadline expiring and H2's deadline expiring. With a 7-day grace period and typical upgrade cadence (new hash voted in days after the previous one), this window is days-wide in production. Any node operator who has not upgraded can exploit it simply by resubmitting their attestation during this window.

---

### Recommendation

Change the slice to exclude the expired cutoff entry, while preserving the "keep at least one" fallback:

```rust
let cutoff_index = self
    .allowed_tee_proposals
    .iter()
    .rposition(|allowed_docker_image| {
        ...
        grace_period_deadline < current_time
    })
    .map(|i| {
        // Exclude the expired entry unless it is the last one
        // (disaster-recovery: always keep at least one entry).
        if i + 1 < self.allowed_tee_proposals.len() { i + 1 } else { i }
    })
    .unwrap_or(0);
```

### Proof of Concept

The existing `test_clean_expired` test does **not** catch this because it advances time past **both** entries' deadlines, so `rposition` returns the last index and the slice is `[H2]` — accidentally correct. The missing test:

```rust
#[test]
fn expired_entry_excluded_when_successor_is_valid() {
    let mut allowed = AllowedDockerImageHashes::default();
    const GRACE: Duration = Duration::from_secs(10);

    // H1 added at T=0
    testing_env!(VMContextBuilder::new().block_timestamp(0).build());
    allowed.insert(dummy_code_hash(1), GRACE);

    // H2 added at T=8s
    testing_env!(VMContextBuilder::new()
        .block_timestamp(8_000_000_000)
        .build());
    allowed.insert(dummy_code_hash(2), GRACE);

    // At T=11s: H1 deadline (0+10=10s) has passed, H2 deadline (8+10=18s) has not
    testing_env!(VMContextBuilder::new()
        .block_timestamp(11_000_000_000)
        .build());

    let hashes = allowed.get_image_hashes(GRACE);
    // BUG: current code returns [H1, H2]; correct result is [H2]
    assert_eq!(hashes, vec![dummy_code_hash(2)], "H1 must be excluded");
}
