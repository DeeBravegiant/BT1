### Title
Unprotected `init` Function Allows Any Caller to Seize Full MPC Network Control Before Legitimate Operator - (File: crates/contract/src/lib.rs)

### Summary
The `MpcContract::init` function carries no caller restriction. Any NEAR account that calls it before the legitimate deployer can install itself as the sole participant with threshold 1, gaining unconditional control over all threshold-signature issuance, key generation, and confidential key derivation for the entire MPC network.

### Finding Description
The `init` function is the primary initialization entry point for `MpcContract`. It is decorated with `#[init]` (which only prevents a second call once state exists) but is **not** decorated with `#[private]` (which would restrict the caller to `predecessor_account_id == current_account_id`). [1](#0-0) 

Compare this to `init_running` and `migrate`, both of which carry `#[private]`: [2](#0-1) [3](#0-2) 

The deployment workflow, as shown in every localnet and production script, deploys the contract binary in one transaction and calls `init` in a separate, subsequent transaction: [4](#0-3) [5](#0-4) 

This creates a race window between contract deployment and initialization. Any NEAR account that observes the deployment receipt and submits an `init` call in the next block wins the race.

Inside `init`, the attacker supplies arbitrary `parameters` (participants and threshold). The function calls `TeeState::with_mocked_participant_attestations` for the supplied participants, which directly inserts mock attestations into `tee_state.stored_attestations` without going through the normal TEE quote verification path: [6](#0-5) 

These stored mock attestations satisfy every subsequent `is_caller_an_attested_participant` check, so the attacker's account immediately passes all governance guards. The contract is placed directly into `Running` state with the attacker's chosen participant set and threshold: [7](#0-6) 

The legitimate operator's subsequent `init` call panics because state already exists (`#[init]` enforces single initialization). There is no recovery path short of redeploying the contract.

The existing TODO comment at line 1942 acknowledges that participants initialized this way bypass real attestation requirements: [8](#0-7) 

### Impact Explanation
An attacker who wins the race sets themselves as the sole participant with threshold 1. They then:
1. Call `vote_add_domains` to register signing domains.
2. Drive distributed key generation as the only participant, producing key shares they fully control.
3. Issue threshold signatures for arbitrary cross-chain payloads — Bitcoin, Ethereum, Solana — without any legitimate participant authorization.
4. Call `request_app_private_key` to obtain confidential key derivation outputs for any application.

This satisfies **Critical** impact: *Unauthorized transaction execution, threshold signature issuance, or confidential key derivation output without the required participant authorization*, and *Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery.*

### Likelihood Explanation
The deployment process is documented and scripted as two separate transactions. Any on-chain observer (MEV bot, competing operator, or malicious actor monitoring the NEAR mempool/blocks) can detect the `DeployContract` action and immediately submit a crafted `init` call. No privileged access, key material, or collusion is required — only a funded NEAR account and knowledge of the contract ABI, which is public.

### Recommendation
Add `#[private]` to `init`, consistent with `init_running` and `migrate`:

```rust
#[handle_result]
#[private]   // ← add this
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

`#[private]` in NEAR SDK enforces `predecessor_account_id == current_account_id`, meaning only the contract account itself can call `init`. The deployer then combines deployment and initialization atomically in a single transaction using `DeployContract` + `FunctionCall` actions, which is the standard safe pattern already used for `init_running` during contract upgrades.

### Proof of Concept

1. Attacker monitors the NEAR blockchain for a `DeployContract` action targeting the MPC contract account (e.g., `v1.signer`).
2. In the immediately following block, attacker submits:
   ```json
   near contract call-function as-transaction v1.signer init json-args '{
     "parameters": {
       "threshold": 1,
       "participants": {
         "next_id": 1,
         "participants": [
           ["attacker.near", 0, {"tls_public_key": "<attacker_p2p_key>", "url": "https://attacker.example.com"}]
         ]
       }
     }
   }' prepaid-gas '300.0 Tgas' attached-deposit '0 NEAR' sign-as attacker.near
   ```
3. `init` succeeds: `MpcContract` is now in `Running` state with `attacker.near` as the sole participant, threshold 1, and a mock attestation stored for the attacker's TLS key.
4. The legitimate operator's `init` call fails: `"Cannot initialize a contract that is already initialized"`.
5. Attacker calls `vote_add_domains` to register a `CaitSith` signing domain, then drives DKG as the sole participant, obtaining full key shares.
6. Attacker calls `sign(...)` for any cross-chain payload and receives a valid threshold signature, enabling unauthorized execution on Bitcoin, Ethereum, or any other supported chain. [9](#0-8)

    #[handle_result]
    #[init]
    pub fn init(
        parameters: dtos::ThresholdParameters,
        init_config: Option<dtos::InitConfig>,
    ) -> Result<Self, Error> {
        let parameters: ThresholdParameters = parameters.try_into_contract_type()?;
        // Log participant count and hash - full parameters exceed NEAR's 16KB log limit at ~100 participants
        let params_hash = env::sha256_array(borsh::to_vec(&parameters).unwrap());
        log!(
            "init: signer={}, num_participants={}, parameters_hash={:?}, init_config={:?}",
            env::signer_account_id(),
            parameters.participants().len(),
            params_hash,
            init_config,
        );
        parameters.validate()?;

        // TODO(#1087): Every participant must have a valid attestation, otherwise we risk
        // participants being immediately kicked out once contract transitions into running.
        let initial_participants = parameters.participants();
        let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);

        Ok(Self {
            protocol_state: ProtocolContractState::Running(RunningContractState::new(
                DomainRegistry::default(),
                Keyset::new(EpochId::new(0), Vec::new()),
                parameters,
                AddDomainsVotes::default(),
            )),
            pending_signature_requests: LookupMap::new(StorageKey::PendingSignatureRequestsV4),
            pending_ckd_requests: LookupMap::new(StorageKey::PendingCKDRequestsV3),
            pending_verify_foreign_tx_requests: LookupMap::new(
                StorageKey::PendingVerifyForeignTxRequestsV2,
            ),
            proposed_updates: ProposedUpdates::default(),
            config: init_config.map(Into::into).unwrap_or_default(),
            tee_state,
            accept_requests: true,
            node_migrations: NodeMigrations::default(),
            metrics: Default::default(),
            node_foreign_chain_support: Default::default(),
            foreign_chains: Lazy::new(
                StorageKey::ForeignChainMetadata,
                ForeignChainsMetadata::default(),
            ),
            tee_verifier_account_id: None,
            tee_verifier_votes: TeeVerifierVotes::default(),
        })
    }
