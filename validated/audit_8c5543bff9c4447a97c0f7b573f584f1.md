### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Submitted `request` — (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash`, but never checks that `response.payload_hash` was actually derived from the `request` parameter. A single Byzantine MPC participant (strictly below the signing threshold) can reuse a legitimately-produced MPC signature for one foreign-chain request as the response to a *different* pending request, delivering a forged verification result to the waiting caller.

### Finding Description

In `respond_verify_foreign_tx` the contract performs two independent checks:

1. The caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the root public key.

It then resolves the pending yield keyed on `request` with the full `response` blob.

```rust
// crates/contract/src/lib.rs  lines 718-753
let payload_hash: [u8; 32] = response.payload_hash.0;   // taken from response, not derived from request

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,          // ← attacker-supplied hash
    &secp_pk,
).is_ok()
...
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,                                   // ← used only as a queue key
    serde_json::to_vec(&response).unwrap(),     // ← response with mismatched payload_hash delivered to caller
)
``` [1](#0-0) 

Contrast this with the regular `respond` path, where the payload hash is **derived from the request** and never taken from the response:

```rust
// crates/contract/src/lib.rs  lines 600-608
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,           // ← derived from request, not from response
    &expected_public_key,
).is_ok()
``` [2](#0-1) 

The `ForeignTxSignPayload` that nodes are supposed to sign is `SHA-256(borsh(ForeignTxSignPayloadV1 { request, values }))`, binding the request parameters to the observed chain state. [3](#0-2) 

But the contract never recomputes this hash from `request`; it accepts whatever hash the responding node supplies, as long as the signature over that hash is valid.

### Impact Explanation

A single malicious attested MPC participant (Byzantine node strictly below the signing threshold) can execute the following substitution attack:

1. **Submit a weaker variant of the target request** as a user — e.g., `verify_foreign_transaction(Bitcoin tx X, confirmations=1)` — while a legitimate user has a pending request for `(Bitcoin tx X, confirmations=6)`.
2. **Wait for the honest MPC network to sign the weaker request.** The threshold of honest nodes produces a valid signature over `payload_hash_weak = SHA-256(borsh({request_weak, observed_values}))`.
3. **Call `respond_verify_foreign_tx(request_strong, response_weak)`.** The contract finds `request_strong` in the pending queue, verifies the signature over `payload_hash_weak` (valid), and resolves the yield for `request_strong` with `response_weak`.
4. **The legitimate user receives `response_weak`** — a valid MPC signature, but one that attests to only 1 confirmation, not the 6 they required.

The user's on-chain contract receives a structurally valid `VerifyForeignTransactionResponse` (signature checks out) but the embedded `payload_hash` does not correspond to their request. Any bridge or settlement contract that does not independently recompute the expected hash from the request parameters will accept this as proof of the stronger condition, enabling invalid bridge execution or double-spend conditions. [4](#0-3) 

### Likelihood Explanation

- Only **one** attested MPC participant needs to be Byzantine; no threshold collusion is required.
- The malicious node does not forge any cryptographic material — it reuses a legitimately produced MPC signature.
- The attacker controls both the `request` key (to target any pending yield) and the `response` (to supply any previously signed payload hash).
- The `respond_verify_foreign_tx` entry point is callable by any single attested participant with no additional guards. [5](#0-4) 

### Recommendation

Derive `payload_hash` inside the contract from the `request` parameter, mirroring the `respond` function. Because the contract does not have access to the `extracted_values` (those are off-chain), the minimal fix is to **reject any response whose `payload_hash` does not begin with a prefix that commits to the request** — or, more robustly, require the responder to also supply the `extracted_values` so the contract can recompute and verify the hash:

```rust
// Recompute expected hash from request + supplied values
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(),   // add this field to the response DTO
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This mirrors the design of `respond`, where the hash is always derived from the request, never accepted from the caller. [6](#0-5) 

### Proof of Concept

```
1. Alice submits verify_foreign_transaction(Bitcoin tx X, confirmations=6, domain_id=D)
   → pending_verify_foreign_tx_requests[request_strong] = [yield_A]

2. Mallory (attested MPC node) submits verify_foreign_transaction(Bitcoin tx X, confirmations=1, domain_id=D)
   → pending_verify_foreign_tx_requests[request_weak] = [yield_M]

3. Honest MPC network signs payload_hash_weak = SHA256(borsh({request_weak, [BlockHash=0xabc...]}))
   → valid signature sig_weak under root key

4. Mallory calls respond_verify_foreign_tx(
       request  = request_strong,   // 6-confirmation request — exists in queue
       response = { payload_hash: payload_hash_weak, signature: sig_weak }
   )
   Contract checks:
     ✓ Mallory is attested participant
     ✓ sig_weak is valid over payload_hash_weak under root key
     ✓ request_strong found in pending queue
   → resolves yield_A with { payload_hash_weak, sig_weak }

5. Alice's contract receives a valid MPC signature, but payload_hash_weak encodes
   only 1 confirmation. Any bridge contract that does not recompute the expected
   hash from Alice's original request parameters accepts this as proof of 6 confirmations.
``` [7](#0-6) [1](#0-0)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

```rust
/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
