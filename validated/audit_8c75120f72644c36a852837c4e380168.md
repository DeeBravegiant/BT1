Audit Report

## Title
Missing `limitDailyMint` Enforcement in `RSETHPoolV2NBA.deposit` Allows Unbounded Daily `wrsETH` Minting - (File: contracts/pools/RSETHPoolV2NBA.sol)

## Summary
`RSETHPoolV2NBA.deposit` calls `wrsETH.mint` without any daily mint cap check, while every other pool variant in the protocol enforces a `limitDailyMint` modifier on the same minting path. Any unprivileged depositor can call `deposit` with an arbitrarily large ETH value and mint an unbounded amount of `wrsETH` in a single day, bypassing the protocol's intended daily issuance control. The contract fails to deliver the daily mint cap guarantee it is expected to enforce.

## Finding Description
`RSETHPoolV2NBA` declares no `dailyMintLimit`, `dailyMintAmount`, `lastMintDay`, or `startTimestamp` state variables and implements no `limitDailyMint` modifier. Its `deposit` function at line 106–118 of `contracts/pools/RSETHPoolV2NBA.sol` calls `wrsETH.mint(msg.sender, rsETHAmount)` directly after computing the swap amount, with only `nonReentrant` and `whenNotPaused` guards. In contrast, `RSETHPoolV2.deposit` at line 207 of `contracts/pools/RSETHPoolV2.sol` applies `limitDailyMint(msg.value)` as a modifier, which tracks `dailyMintAmount` per day and reverts with `DailyMintLimitExceeded` when the cap is exceeded. The root cause is that `RSETHPoolV2NBA` was introduced as a "no bridging automation" variant but was never updated to include the daily mint cap mechanism that was subsequently added to all other pool variants. Any call to `RSETHPoolV2NBA.deposit` reaches `wrsETH.mint` without passing through the cap gate.

## Impact Explanation
The protocol's primary on-chain risk-management control — the daily issuance limit — is entirely absent for this pool. A depositor can mint an unbounded quantity of `wrsETH` in a single day with no per-day ceiling. Because deposited ETH backs the minted `wrsETH` at the oracle rate, no direct value is lost under normal oracle conditions; however, the contract concretely fails to deliver the promised daily mint cap guarantee. This matches the allowed Low impact: **Contract fails to deliver promised returns**.

## Likelihood Explanation
Any unprivileged external caller can trigger this by calling `RSETHPoolV2NBA.deposit{value: X}("")` with any nonzero ETH amount. No special role, no front-running, no oracle manipulation, and no preconditions beyond having ETH are required. The call path is fully public and repeatable within the same day without any revert. Likelihood is **High**.

## Recommendation
Add the `limitDailyMint` mechanism to `RSETHPoolV2NBA` consistent with `RSETHPoolV2`:
1. Add state variables: `uint256 public dailyMintLimit`, `uint256 public dailyMintAmount`, `uint256 public lastMintDay`, `uint256 public startTimestamp`.
2. Implement the `limitDailyMint(uint256 amount)` modifier mirroring `RSETHPoolV2` lines 72–94.
3. Apply `limitDailyMint(msg.value)` to `deposit`.
4. Add a `reinitializer` function (and/or a `setDailyMintLimit` admin function) to set the initial limit and start timestamp, mirroring `RSETHPoolV2.reinitialize` and `RSETHPoolV2.setDailyMintLimit`.

## Proof of Concept
1. Deploy `RSETHPoolV2NBA` with a valid `wrsETH` and oracle.
2. Call `RSETHPoolV2NBA.deposit{value: 1000 ether}("")` — `wrsETH.mint(attacker, rsETHAmount)` executes with no cap check.
3. Immediately call `RSETHPoolV2NBA.deposit{value: 1000 ether}("")` again in the same block/day — succeeds again with no revert.
4. Repeat arbitrarily within the same day; total minted `wrsETH` is unbounded.
5. For comparison: the identical sequence on `RSETHPoolV2` reverts with `DailyMintLimitExceeded` after the daily quota is exhausted (line 89 of `contracts/pools/RSETHPoolV2.sol`).
