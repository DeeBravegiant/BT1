Audit Report

## Title
ETH Donation to NodeDelegator Inflates Protocol-Wide ETH Accounting, Triggering Deposit Cap and Blocking New Deposits - (File: contracts/LRTDepositPool.sol)

## Summary
`getETHDistributionData()` measures ETH held in NodeDelegator contracts using raw `address.balance`, which any unprivileged caller can inflate by sending ETH directly to a NodeDelegator via its unrestricted `receive()` function. This inflated balance propagates through `getTotalAssetDeposits()` into `_checkIfDepositAmountExceedesCurrentLimit()`, causing every subsequent `depositETH()` call to revert with `MaximumDepositLimitReached` until an admin raises the deposit cap.

## Finding Description
`getETHDistributionData()` aggregates ETH across NodeDelegators using raw balance reads:

```solidity
// contracts/LRTDepositPool.sol L484-485
for (uint256 i; i < ndcsCount;) {
    ethLyingInNDCs += nodeDelegatorQueue[i].balance;
``` [1](#0-0) 

This value flows into `getTotalAssetDeposits()`: [2](#0-1) 

Which is consumed by `_checkIfDepositAmountExceedesCurrentLimit()`. Critically, for ETH the check is `totalAssetDeposits > limit` — with no `+ amount` term — meaning if the total already exceeds the limit (even before the new deposit), it reverts: [3](#0-2) 

This gates `_beforeDeposit()`, which is called by every `depositETH()` invocation: [4](#0-3) 

Every `NodeDelegator` accepts arbitrary ETH with no access control: [5](#0-4) 

An attacker sends ETH directly to any NDC. The `receive()` function accepts it unconditionally. `getETHDistributionData()` then reports the inflated `address.balance`, pushing `getTotalAssetDeposits(ETH)` above `depositLimitByAsset`, causing all `depositETH()` calls to revert. The donated ETH cannot be recovered by the attacker and is permanently absorbed into the protocol, so the freeze persists until an admin raises the cap.

## Impact Explanation
**Medium — Temporary freezing of funds.** All new ETH deposits are blocked for as long as the inflated balance keeps `totalAssetDeposits > depositLimitByAsset`. Existing rsETH holders and withdrawal flows are unaffected; only new `depositETH()` calls revert. The freeze is temporary only because an admin can raise the deposit limit, but the disruption window is real and the attacker's cost is only the donated ETH (permanently lost).

## Likelihood Explanation
The attack is fully permissionless and requires no special role or privilege. The only cost is the ETH donated, which is permanently lost to the attacker. Near the deposit cap, even 1 wei suffices. The attack is most attractive when the protocol is close to its cap and an adversary wishes to prevent competitors from depositing during a points or reward window. An admin can mitigate by raising the cap, but cannot prevent the attack from recurring.

## Recommendation
Replace raw `address.balance` reads in `getETHDistributionData()` with an internal accounting variable updated exclusively through protocol-controlled deposit and transfer paths. For example, maintain a `mapping(address ndc => uint256 ethTracked)` incremented in `transferETHToNodeDelegator` and decremented when ETH leaves via protocol-sanctioned paths. Alternatively, add a tolerance threshold that ignores unsolicited donations below a `maxNegligibleAmount`.

## Proof of Concept
1. Protocol has `depositLimitByAsset[ETH] = 10_000 ether` and current `getTotalAssetDeposits(ETH) = 9_999.9 ether`.
2. Attacker executes:
   ```solidity
   (bool ok,) = nodeDelegatorQueue[0].call{value: 0.2 ether}("");
   ```
   `NodeDelegator.receive()` accepts the ETH and emits `ETHReceived`.
3. `getETHDistributionData()` now returns `ethLyingInNDCs` inflated by `0.2 ether`, making `getTotalAssetDeposits(ETH) = 10_000.1 ether`.
4. Any user calling `depositETH(minRSETH, ref)` triggers:
   - `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit(ETH, amount)`
   - `totalAssetDeposits (10_000.1 ether) > depositLimitByAsset (10_000 ether)` → `true`
   - Reverts with `MaximumDepositLimitReached`.
5. ETH deposits are frozen until an admin raises the limit.

**Foundry test plan:** Deploy `LRTDepositPool` and one `NodeDelegator` in a fork test. Set `depositLimitByAsset[ETH]` to a value just above current deposits. Send ETH directly to the NDC from an unprivileged address. Assert that `depositETH{value: 1 ether}()` reverts with `MaximumDepositLimitReached`.
