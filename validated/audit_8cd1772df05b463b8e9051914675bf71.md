Audit Report

## Title
`setPoolAdminFeeDestination` Redirects Unsettled Admin Fees to New Destination Without Prior Collection — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary

`setPoolAdminFeeDestination` overwrites `poolAdminFeeDestination[pool]` without first calling `collectFees`, unlike the parallel mutators `setPoolAdminFees` and `setPoolProtocolFee` which both settle accumulated fees before any state change. All spread-fee surplus and notional fees that accrued under the old destination are silently redirected to the new destination on the next `collectPoolFees` call, causing a direct loss of owed admin fees for the previous recipient.

## Finding Description

`setPoolAdminFees` (L408–435) calls `collectFees` with the current rates and current destination before updating any state: [1](#0-0) 

`setPoolAdminFeeDestination` (L438–447) performs no such settlement: [2](#0-1) 

`collectFees` in `MetricOmmPool` computes the spread surplus as a running balance (`balance0() * TOKEN_0_SCALE_MULTIPLIER - binTotals.scaledToken0 - notionalFee0AmountScaled`) and routes the admin portion to whatever `adminFeeDestination_` is passed at call time: [3](#0-2) [4](#0-3) [5](#0-4) 

`collectPoolFees` (permissionless) reads `poolAdminFeeDestination[pool]` at collection time: [6](#0-5) 

Because the surplus is a running balance with no per-epoch snapshots, every unit of spread fee accumulated under the old destination is included in the next collection — which now uses the new destination.

## Impact Explanation

The pool admin can atomically redirect all accumulated admin spread fees and notional fees from the old destination (e.g., a DAO treasury or revenue-sharing contract) to an attacker-controlled address. The old destination receives nothing for the entire period it was the designated recipient. This is a direct loss of owed protocol/admin fees, bounded by the total admin surplus at the time of the destination change, which grows with swap volume and time since last collection. This meets the "direct loss of protocol fees" impact gate.

## Likelihood Explanation

The pool admin is a per-pool privileged role that is explicitly semi-trusted under contest rules — it must stay within caps and timelocks. There is no cap or timelock on `setPoolAdminFeeDestination`. The admin fee destination is frequently a separate entity from the pool admin key (e.g., a DAO treasury). The pool admin can execute this in a single block by bundling `setPoolAdminFeeDestination` and `collectPoolFees` (which is permissionless). No additional preconditions are required beyond the pool having accumulated non-zero surplus.

## Recommendation

Mirror the pattern used by `setPoolAdminFees`: call `collectFees` with the **current** rates and the **current** destination before overwriting `poolAdminFeeDestination[pool]`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

## Proof of Concept

```
Setup:
  - Pool deployed with adminFeeDestination = treasury (separate multisig)
  - Pool accumulates surplus over N swaps
  - adminSpreadFeeE6 = 50_000 (5%), protocolSpreadFeeE6 = 10_000 (1%)
  - surplus0Scaled = 1_000_000 units; admin share ≈ 833_333 units

Attack (single block):
  1. pool_admin calls setPoolAdminFeeDestination(pool, attacker_wallet)
     → poolAdminFeeDestination[pool] = attacker_wallet
     → NO collectFees called; full surplus remains on pool

  2. Anyone calls collectPoolFees(pool)
     → spreadFee0ToAdmin = 1_000_000 * 50_000 / 60_000 ≈ 833_333 → sent to attacker_wallet
     → spreadFee0ToProtocol ≈ 166_667 → sent to FACTORY

Result:
  treasury receives 0 (should have received ≈ 833_333)
  attacker_wallet receives ≈ 833_333 (earned under treasury's tenure)
```

Foundry test: deploy pool, execute N swaps to accumulate surplus, call `setPoolAdminFeeDestination` to attacker address, call `collectPoolFees`, assert treasury balance is 0 and attacker balance equals expected admin share.
