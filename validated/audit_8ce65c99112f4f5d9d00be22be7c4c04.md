### Title
Unelected Constitutional Committee Members Can Cast Binding Governance Votes via Block-Producer Mempool Bypass at Protocol Versions 9–10 - (File: `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs`)

### Summary

In Conway era at protocol versions 9–10, the `GOV` ledger rule does not enforce the restriction that only elected Constitutional Committee (CC) members may cast votes. The authorization check is gated behind `hardforkConwayDisallowUnelectedCommitteeFromVoting` and is therefore absent from the ledger rule at those versions. A soft guard exists only in the `MEMPOOL` rule, which block producers can bypass by including transactions directly in blocks. An unelected CC member whose cold credential appears in a pending `UpdateCommittee` proposal can pre-cast votes; once their election is enacted those votes are counted during ratification, potentially causing treasury withdrawals, hard-fork initiations, protocol-parameter changes, or new constitutions to be enacted without proper authorization.

### Finding Description

`conwayGovTransition` in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Gov.hs` contains the following conditional guard:

```haskell
when (hardforkConwayDisallowUnelectedCommitteeFromVoting $ pp ^. ppProtocolVersionL) $
    failOnNonEmpty
      (unelectedCommitteeVoters committee committeeState gsVotingProcedures)
      (injectFailure . UnelectedCommitteeVoters)
``` [1](#0-0) 

`hardforkConwayDisallowUnelectedCommitteeFromVoting` returns `True` only when `pvMajor pv > natVersion @10`, i.e., protocol version ≥ 11:

```haskell
hardforkConwayDisallowUnelectedCommitteeFromVoting :: ProtVer -> Bool
hardforkConwayDisallowUnelectedCommitteeFromVoting pv = pvMajor pv > natVersion @10
``` [2](#0-1) 

At protocol versions 9 and 10 the `when` guard is `False`, so the entire `failOnNonEmpty` check is skipped. The `GOV` rule accepts votes from any hot credential that has been authorized via `AuthCommitteeHotKeyTxCert`, regardless of whether the corresponding cold credential is in the currently enacted `Committee`.

The only protection at PV 9–10 is in the `MEMPOOL` rule:

```haskell
unless (hardforkConwayDisallowUnelectedCommitteeFromVoting protVer) $
  -- This check can completely be removed once mainnet switches to protocol
  -- version 11, since the same check has been implemented in the GOV rule.
  let addPrefix = ("Unelected committee members are not allowed to cast votes: " <>)
   in failOnNonEmpty
        ( unelectedCommitteeVoters ... )
        (ConwayMempoolFailure . addPrefix . ...)
``` [3](#0-2) 

The `MEMPOOL` rule is only invoked when a transaction enters the node's mempool. It is **not** invoked during block application. A block producer can craft a block that includes a transaction containing unelected CC votes and apply it directly via the `LEDGER` rule, which calls `GOV` without the `MEMPOOL` guard.

Once the unelected member's vote is recorded in `gasCommitteeVotes`, and their `UpdateCommittee` election is subsequently enacted, `committeeAcceptedRatio` in the `RATIFY` rule iterates over the newly elected `members` and finds the pre-cast vote:

```haskell
Just (CommitteeHotCredential hotKey) ->
  case Map.lookup hotKey votes of
    ...
    Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
``` [4](#0-3) 

The codebase's own test suite confirms this behavior at PV 10:

```
it "at protocol version 10, the vote is counted once the committee update is enacted"
``` [5](#0-4) 

And the `LedgerSpec` test explicitly demonstrates the mempool bypass path:

```haskell
txFixed <-
  submitFailingMempoolTx "unallowed votes" tx $
    NonEmpty.singleton . injectFailure . ConwayMempoolFailure $
      "Unelected committee members are not allowed to cast votes: ..."
withNoFixup $ submitTx_ txFixed   -- bypasses mempool, accepted by ledger
``` [6](#0-5) 

### Impact Explanation

An unelected CC member who is also a block producer (or who colludes with one) can pre-cast `VoteYes` on any governance action — `TreasuryWithdrawals`, `HardForkInitiation`, `ParameterChange`, `NewConstitution` — before their election is enacted. Once the `UpdateCommittee` proposal is ratified and enacted, the pre-cast vote is counted as a legitimate CC vote during the next `RATIFY` pass. This can tip the `committeeAcceptedRatio` above the required threshold, causing a governance action to be enacted that would not otherwise have passed. The result is an unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action being enacted — matching the **Critical** impact tier.

### Likelihood Explanation

The attacker must (a) control a cold credential that appears in a pending `UpdateCommittee` proposal and (b) be a block producer (or collude with one) to bypass the mempool. Both conditions are realistic: committee candidates are publicly known from on-chain proposals, and block producers are a standard attacker profile in the scope. No privileged keys, leaked secrets, or majority stake are required. The attack is deterministic and requires only a single crafted transaction included in one block.

### Recommendation

Move the `unelectedCommitteeVoters` check out of the `when hardforkConwayDisallowUnelectedCommitteeFromVoting` guard so it is enforced unconditionally in `conwayGovTransition` for all protocol versions from the start of the Conway era. The `MEMPOOL` duplicate check can then be removed as the comment already anticipates. Alternatively, if the PV 9–10 behavior is intentional by design, the `MEMPOOL` check should be promoted to a hard ledger-rule predicate failure so that block producers cannot bypass it.

### Proof of Concept

1. At Conway protocol version 10, submit an `UpdateCommittee` proposal adding `coldCred` to the committee.
2. Submit `AuthCommitteeHotKeyTxCert coldCred hotCred` — accepted because `coldCred` is in a pending proposal (`ConwayCommitteeIsUnknown` check passes).
3. Craft a transaction `tx` containing `VotingProcedures { CommitteeVoter hotCred → VoteYes on gaiTreasuryWithdrawal }`.
4. Attempt `submitTx_ tx` through the mempool — rejected with `ConwayMempoolFailure "Unelected committee members are not allowed to cast votes"`.
5. As a block producer, include `tx` directly in a block (bypassing `MEMPOOL`). The `GOV` rule accepts it because `hardforkConwayDisallowUnelectedCommitteeFromVoting PV10 = False`.
6. The vote is now stored in `gasCommitteeVotes gaiTreasuryWithdrawal`.
7. Pass an epoch so the `UpdateCommittee` proposal is enacted; `coldCred` is now an elected member.
8. `isCommitteeAccepted gaiTreasuryWithdrawal` returns `True`; the treasury withdrawal is enacted.

This is exactly the sequence demonstrated by the existing test at `eras/conway/impl/testlib/Test/Cardano/Ledger/Conway/Imp/GovCertSpec.hs` lines 210–228, with step 5 being the block-producer bypass that the mempool check cannot prevent. [1](#0-0) [7](#0-6) [8](#0-7) [9](#0-8) [5](#0-4) [10](#0-9)

```haskell
committeeAcceptedRatio members votes committeeState currentEpoch =
  yesVotes %? totalExcludingAbstain
  where
    accumVotes ::
      (Integer, Integer) ->
      Credential ColdCommitteeRole ->
      EpochNo ->
      (Integer, Integer)
    accumVotes (!yes, !tot) member expiry
      | currentEpoch > expiry = (yes, tot) -- member is expired, vote "abstain" (don't count it)
      | otherwise =
          case Map.lookup member (csCommitteeCreds committeeState) of
            Nothing -> (yes, tot) -- member is not registered, vote "abstain"
            Just (CommitteeMemberResigned _) -> (yes, tot) -- member has resigned, vote "abstain"
            Just (CommitteeHotCredential hotKey) ->
              case Map.lookup hotKey votes of
                Nothing -> (yes, tot + 1) -- member hasn't voted, vote "no"
                Just Abstain -> (yes, tot) -- member voted "abstain"
                Just VoteNo -> (yes, tot + 1) -- member voted "no"
                Just VoteYes -> (yes + 1, tot + 1) -- member voted "yes"
    (yesVotes, totalExcludingAbstain) = Map.foldlWithKey' accumVotes (0, 0) members
