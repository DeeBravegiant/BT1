### Title
`respond_ckd` Accepts Unverified CKD Output for `AppPublicKey` Variant, Allowing a Single Byzantine Participant to Deliver Attacker-Controlled Confidential Key Material - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` contract method performs **no cryptographic verification** of the `CKDResponse` when the request uses the `AppPublicKey` (privately verifiable, legacy) variant. A single attested participant — acting strictly below the signing threshold — can call `respond_ckd` with arbitrary `big_y` and `big_c` values, and the contract will accept and deliver them to the user. This bypasses the threshold requirement and allows a single Byzantine node to substitute attacker-controlled key material for the user's confidential derived key.

---

### Finding Description

In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
// crates/contract/src/lib.rs, lines 675–682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(app_id), pk)`, which cryptographically proves the response was computed using the MPC master secret key and the correct `app_id`. [2](#0-1) 

For the `AppPublicKey` variant, the match arm is **empty** — the contract unconditionally proceeds to `resolve_yields_for`, delivering whatever `big_y` and `big_c` the caller supplied. [3](#0-2) 

The `AppPublicKey` variant is the default/legacy variant, explicitly documented and used in all non-PV examples: [4](#0-3) 

By contrast, `respond` (for ECDSA/EdDSA signatures) always verifies the cryptographic output against the MPC public key before resolving the yield, making it impossible for a single node to forge a valid signature: [5](#0-4) 

---

### Impact Explanation

The CKD protocol is designed so that the derived confidential key is secret from all parties, including individual MPC nodes. The security guarantee rests on the threshold: no single node below the threshold should be able to determine or control the output.

When `AppPublicKey` is used, a single Byzantine attested participant can:

1. Intercept a pending `CKDRequest` from `pending_ckd_requests`.
2. Choose arbitrary `big_y = r·G1` for a known scalar `r`, and set `big_c = r·app_pk1` (i.e., omit the `H(app_id)·msk` term entirely, or set it to any chosen value).
3. Call `respond_ckd` with this fabricated response. The contract accepts it with no verification.
4. The user receives `(big_y, big_c)` and computes `big_c - a·big_y` as their "secret" — but this value is fully known to the attacker (it is zero in the example above, or any attacker-chosen point).

This completely breaks the confidentiality guarantee: the user's derived key is not secret from the attacker. The attacker is a **single** attested participant, strictly below the signing threshold.

This maps to the allowed impact: **Critical — confidential key derivation output without the required participant authorization (threshold).**

---

### Likelihood Explanation

- The `AppPublicKey` variant is the default and most widely used variant (all non-PV documentation examples use it).
- Any single attested participant can exploit this — no collusion above threshold is required.
- The attacker only needs to race the honest leader in calling `respond_ckd`; since the contract resolves the first valid call, a Byzantine node that calls first wins.
- The entry path is a standard NEAR transaction: `respond_ckd(request, fabricated_response)` from any attested participant account.

---

### Recommendation

1. **Require `AppPublicKeyPV` for all new CKD requests** and deprecate `AppPublicKey`. The G2 component (`pk2`) is what enables on-chain verification; without it the contract cannot verify the response.
2. As an interim measure, add a comment or runtime warning that `AppPublicKey` requests receive no on-chain output verification and rely entirely on the honesty of the responding node.
3. Consider whether the `AppPublicKey` variant should be removed from the production contract interface entirely, given that it provides no security guarantee against a single Byzantine participant.

---

### Proof of Concept

The existing test at `crates/contract/src/lib.rs:3403–3441` already demonstrates the issue: it submits a `CKDResponse` with `big_y = [1u8; 48]` and `big_c = [2u8; 48]` (arbitrary byte arrays, not a valid protocol output) for an `AppPublicKey` request, and the contract accepts it without error: [6](#0-5) 

A Byzantine attacker would:

```rust
// Attacker is an attested participant. They see a pending AppPublicKey CKD request.
// They choose r = 1 (trivial scalar), so big_y = G1, big_c = app_pk1.
// The user will compute big_c - a*big_y = a*G1 - a*G1 = 0 (identity point).
// The attacker knows the user's derived "secret" is the identity — a fully known value.
let fabricated_response = CKDResponse {
    big_y: G1_GENERATOR,   // r·G1 for known r=1
    big_c: app_pk1_bytes,  // a·G1 = app_pk1, so big_c - a*big_y = 0
};
contract.respond_ckd(ckd_request, fabricated_response)
// → Ok(()) — accepted, user receives attacker-controlled key material
```

The contract's `respond_ckd` at line 676 takes the empty branch for `AppPublicKey` and proceeds directly to `resolve_yields_for`, delivering the fabricated response to the waiting user. [7](#0-6)

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
