### Title
Blacklisted recipient permanently locks bridged funds in `finTransfer` — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.sol::finTransfer` transfers tokens directly to `payload.recipient` via `safeTransfer`. If the recipient address is blacklisted by the token contract (e.g., USDC, USDT), the transfer reverts. Because the recipient is cryptographically bound inside the MPC-signed payload and no on-chain mechanism exists to redirect delivery, the transfer can never complete. The user's funds remain permanently locked or burned on the source chain with no recovery path.

---

### Finding Description

In `OmniBridge.sol::finTransfer`, after signature verification, the contract unconditionally delivers tokens to `payload.recipient`: [1](#0-0) 

For the native-locked-token path (the `else` branch), this is a bare `safeTransfer` call. OpenZeppelin's `safeTransfer` reverts if the underlying ERC-20 `transfer` returns `false` or reverts — which is exactly what USDC and USDT do when the recipient is on their blacklist.

The nonce is marked used at line 287 **before** the transfer: [2](#0-1) 

Because the entire transaction reverts when `safeTransfer` fails, the nonce write is also rolled back. The nonce is therefore never consumed, so the relayer can retry — but the retry will always fail for the same reason.

The recipient is cryptographically committed inside the MPC-signed Borsh payload: [3](#0-2) 

On the NEAR side, `sign_transfer` constructs the payload directly from the stored `transfer_message.recipient` with no way for an unprivileged caller to substitute a different address: [4](#0-3) 

The only mutable field on a pending transfer is the fee (`update_transfer_fee`); there is no `update_transfer_recipient` or cancel-and-refund entrypoint: [5](#0-4) 

Consequently, once a transfer is initiated on the source chain (tokens burned or locked in vault), and the destination recipient is or becomes blacklisted for the bridged token, the transfer is permanently undeliverable and the funds are permanently lost.

The same pattern exists in the Starknet bridge: [6](#0-5) 

---

### Impact Explanation

**Critical — Irreversible fund lock.**

A user who bridges USDC (or any blacklist-enforcing ERC-20) from any source chain to an EVM address that is (or becomes) blacklisted will have their tokens burned/locked on the source chain with no on-chain path to recover them. The bridge holds the value but can never release it to any party. This matches the allowed impact: *"Irreversible fund lock, frozen redemption path, or permanently unclaimable user or protocol value in bridge, token, fee, vault, fast-transfer, or UTXO flows."*

---

### Likelihood Explanation

**Low-Medium.** USDC and USDT are among the most commonly bridged tokens and both maintain on-chain blacklists. OFAC sanctions, exchange compliance actions, and smart-contract exploit responses all result in addresses being blacklisted. A user whose address is blacklisted after initiating a bridge transfer — or who unknowingly bridges to a blacklisted address — triggers this condition with no recourse.

---

### Recommendation

Wrap the token delivery in a try/catch (EVM) or equivalent, and on failure escrow the funds under the recipient's key so they can be claimed later via an alternative address or mechanism. Concretely:

1. **EVM**: Replace the bare `safeTransfer` with a try/catch; on failure, credit the amount to a per-recipient claimable mapping and emit an event. Add a `claimFailed(address token, address recipient)` function.
2. **Starknet**: Same pattern — catch a failed `transfer` and store the amount in a claimable map.
3. **NEAR**: Add a `cancel_transfer(transfer_id)` entrypoint callable by the original sender that refunds the locked/burned tokens and removes the pending transfer, so users are not permanently trapped.

---

### Proof of Concept

1. Alice holds 10,000 USDC on NEAR (as a bridged token).
2. Alice calls `ft_transfer_call` on the NEAR bridge to initiate a transfer of 10,000 USDC to her EVM address `0xAlice`. Tokens are burned on NEAR; a `TransferMessage` is stored with `recipient = OmniAddress::Eth(0xAlice)`.
3. Between step 2 and finalization, `0xAlice` is added to the USDC blacklist (e.g., OFAC action).
4. A relayer calls `sign_transfer` on NEAR → MPC signs a payload with `recipient = 0xAlice`.
5. Relayer calls `OmniBridge.finTransfer(sig, payload)` on EVM.
6. `completedTransfers[nonce] = true` is written, then `USDC.safeTransfer(0xAlice, 10000e6)` reverts due to blacklist → entire transaction reverts.
7. Relayer retries indefinitely; every attempt reverts at the same line.
8. Alice has no `cancel_transfer` or `update_transfer_recipient` call available.
9. 10,000 USDC are permanently destroyed on NEAR with no EVM delivery possible. [7](#0-6) [8](#0-7)

    pub fn sign_transfer(
        &mut self,
        transfer_id: TransferId,
        fee_recipient: Option<AccountId>,
        fee: &Option<Fee>,
    ) -> Promise {
        let transfer_message = self.get_transfer_message(transfer_id);

        if let Some(fee) = &fee {
            require!(
                &transfer_message.fee == fee,
                BridgeError::InvalidFee.as_ref()
            );
        }

        let token_address = self
            .get_token_address(
                transfer_message.get_destination_chain(),
                self.get_token_id(&transfer_message.token),
            )
            .unwrap_or_else(|| {
                env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
            });

        let decimals = self
            .token_decimals
            .get(&token_address)
            .near_expect(BridgeError::TokenDecimalsNotFound);
        let amount_to_transfer = Self::normalize_amount(
            transfer_message
                .amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );

        require!(
            amount_to_transfer > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );

        let message = DestinationChainMsg::from_json(&transfer_message.msg)
            .and_then(|s| s.destination_msg())
            .unwrap_or_default();

        let transfer_payload = TransferMessagePayload {
            prefix: PayloadType::TransferMessage,
            destination_nonce: transfer_message.destination_nonce,
            transfer_id,
            token_address,
            amount: U128(amount_to_transfer),
            recipient: transfer_message.recipient,
            fee_recipient,
            message,
        };

        let payload = near_sdk::env::keccak256_array(
            transfer_payload
                .encode_hashable()
                .near_expect(BridgeError::Borsh),
        );

        ext_signer::ext(self.mpc_signer.clone())
            .with_static_gas(MPC_SIGNING_GAS)
            .with_attached_deposit(env::attached_deposit())
            .sign(SignRequest {
                payload,
                path: SIGN_PATH.to_owned(),
                key_version: 0,
            })
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(SIGN_TRANSFER_CALLBACK_GAS)
                    .sign_transfer_callback(transfer_payload, &transfer_message.fee),
            )
