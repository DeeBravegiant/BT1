Audit Report

## Title
ETH Deposit Cap Not Enforced Against Incoming Amount - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of the cap check, comparing only the pre-deposit total against the limit. As a result, any depositor can bypass the ETH deposit cap in a single transaction while `totalAssetDeposits ≤ depositLimit`, causing rsETH to be minted far beyond the configured TVL ceiling.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (L676–682), the ETH branch evaluates:

```solidity
return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // L679
```

while the LST branch correctly evaluates:

```solidity
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));  // L681
```

The ETH branch never incorporates `amount`, so the function returns `false` (not exceeded) for any deposit size as long as the pre-deposit total is at or below the limit. `_beforeDeposit` (L661–663) relies entirely on this function to gate deposits; when it returns `false`, execution proceeds to `_mintRsETH` (L90) with no further cap enforcement. The public, permissionless `depositETH` (L76–93) is the entry point. [1](#0-0) [2](#0-1) 

## Impact Explanation
The ETH deposit cap is the protocol's primary TVL risk-management control for ETH. Because the incoming amount is never checked against the remaining headroom, a single depositor can push `totalAssetDeposits` arbitrarily above the cap in one call, causing `_mintRsETH` to mint rsETH far in excess of the intended backing. This breaks the protocol's TVL invariant and constitutes **contract fails to deliver promised returns** (Low), with a credible path toward protocol insolvency if the cap is the sole guard against over-minting. [3](#0-2) 

## Likelihood Explanation
The entry point is the public, payable, permissionless `depositETH` function. No role, whitelist, or special precondition is required beyond `totalAssetDeposits ≤ depositLimit` at the time of the call — a condition that is trivially observable on-chain. Any depositor can exploit this in a single transaction. [4](#0-3) 

## Recommendation
Mirror the LST branch in the ETH branch by adding `amount`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [5](#0-4) 

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 100 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `50 ether`.
3. Attacker calls `depositETH{value: 10_000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `50 ether > 100 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 10 000 ETH; `totalAssetDeposits` becomes `10 050 ether`.
6. All subsequent `depositETH` calls revert with `MaximumDepositLimitReached`, but 9 950 ETH above the cap has already been accepted and rsETH minted for it.

Foundry fuzz test plan: fuzz `depositAmount` over `[1, type(uint128).max]` with `totalAssetDeposits` initialized to any value `≤ depositLimit`; assert that `_checkIfDepositAmountExceedesCurrentLimit` returns `true` whenever `totalAssetDeposits + depositAmount > depositLimit`. The current code will fail this assertion for all `depositAmount` values when `totalAssetDeposits ≤ depositLimit`. [1](#0-0)

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
