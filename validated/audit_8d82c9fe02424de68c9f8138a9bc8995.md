Audit Report

## Title
First-Depositor Share Inflation via Permissionless `updateRSETHPrice()` and Open `receive()` — (File: contracts/LRTOracle.sol, contracts/LRTDepositPool.sol)

## Summary

`LRTOracle.updateRSETHPrice()` is callable by any unprivileged address and `LRTDepositPool` accepts ETH from anyone via its open `receive()` function, with `getETHDistributionData()` counting `address(this).balance` directly as TVL. An attacker who is the first depositor can donate ETH to inflate the stored `rsETHPrice`, causing subsequent depositors to receive a near-zero amount of rsETH for their ETH. The attacker's rsETH then represents the overwhelming majority of the pool, allowing them to eventually redeem the victim's deposited ETH through the withdrawal mechanism.

## Finding Description

**Root cause:** Three independent weaknesses compose into a critical exploit path.

**1. Permissionless price update** — `updateRSETHPrice()` carries only a `whenNotPaused` modifier, no role check:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

**2. Zero-supply bootstrap** — `_updateRsETHPrice()` unconditionally sets `rsETHPrice = 1 ether` when `rsethSupply == 0`, giving the first depositor a guaranteed 1:1 mint rate:

```solidity
// contracts/LRTOracle.sol L218-222
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

**3. Raw balance counted as TVL** — `getETHDistributionData()` uses `address(this).balance`, which includes any ETH force-sent via `receive()`:

```solidity
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

**4. Price-increase guard disabled by default** — `pricePercentageLimit` is never set in `initialize()`, so it defaults to `0`. The upside guard short-circuits to `false` for any price jump:

```solidity
// contracts/LRTOracle.sol L256-257
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
```

**Exploit flow:**

1. Attacker calls `updateRSETHPrice()` → `rsETHPrice = 1e18` (supply is 0).
2. Attacker calls `depositETH{value: minAmountToDeposit}` → receives `minAmountToDeposit` rsETH (e.g., 1e15 wei).
3. Attacker sends 100 ETH directly to `LRTDepositPool` via `receive()` → `address(this).balance = 100.001 ETH`.
4. Attacker calls `updateRSETHPrice()` again → `totalETHInProtocol = 100.001e18`, `rsethSupply = 1e15`, `newRsETHPrice = 100_001e18`. `pricePercentageLimit == 0` so the upside guard does not fire.
5. Victim calls `depositETH{value: 1 ether}` → `rsethAmountToMint = 1e18 * 1e18 / 100_001e18 ≈ 9999 wei rsETH`. Victim's 9999 rsETH represents `9999 / (1e15 + 9999) ≈ 1e-11` of the pool — effectively zero.
6. Attacker's 1e15 rsETH represents ~99.999% of the pool (now 101.001 ETH). Attacker initiates withdrawal via `LRTWithdrawalManager.initiateWithdrawal()` or `instantWithdrawal()` at the inflated `rsETHPrice`, recovering ~101 ETH against a 100.001 ETH investment. Net profit ≈ 1 ETH (the victim's deposit).

**Why existing checks fail:**

- `_beforeDeposit` checks `minRSETHAmountExpected` only if the caller supplies a non-zero value; users who pass `0` receive no protection.
- The deposit limit check for ETH (`totalAssetDeposits > depositLimit`) does not add the incoming `amount` before comparing, so the attacker's donation does not block the victim's deposit.
- `updateRSETHPriceAsManager()` exists as a privileged alternative but does not restrict the public `updateRSETHPrice()`.

## Impact Explanation

**Critical — direct theft of user funds.** The victim deposits 1 ETH and receives rsETH worth approximately `1e-9 ETH` at the inflated price. The attacker's rsETH, backed by the victim's deposited ETH, is redeemable through `LRTWithdrawalManager` once the ETH is routed through the normal operator flow (deposit pool → node delegator → unstaking vault). The victim's principal is permanently diluted to near zero; the attacker recovers both their donation and the victim's deposit.

## Likelihood Explanation

**Medium.** The attack is most effective at protocol launch when rsETH supply is minimal. It requires the attacker to front-run the first legitimate depositor and to bear temporary ETH capital cost (recovered on redemption). No privileged role is needed; both entry points (`updateRSETHPrice()` and `receive()`) are fully open to any EOA or contract. The attack is repeatable any time the rsETH supply is very low relative to the donated amount.

## Recommendation

1. **Bootstrap the initial supply**: mint a non-trivial amount of rsETH to a dead address (e.g., `address(0xdead)`) during `initialize()` so the zero-supply 1:1 window is never reachable by an unprivileged depositor.
2. **Restrict `updateRSETHPrice()`**: require at minimum a `KEEPER_ROLE` or equivalent, or enforce a minimum time-between-updates cooldown.
3. **Track deposited ETH in a dedicated storage variable** incremented only through controlled entry points (`depositETH`, `receiveFromRewardReceiver`, etc.), preventing force-sent ETH from inflating TVL.
4. **Set a non-zero `pricePercentageLimit`** as part of the deployment/initialization sequence so the upside guard is active from day one.

## Proof of Concept

```
Precondition: protocol just deployed, rsETH totalSupply = 0, pricePercentageLimit = 0.

1. Attacker calls LRTOracle.updateRSETHPrice()
   → rsETHPrice = 1e18 (supply is 0, bootstrap branch)

2. Attacker calls LRTDepositPool.depositETH{value: 1e15}(0, "")
   → rsethAmountToMint = 1e15 * 1e18 / 1e18 = 1e15
   → Attacker holds 1e15 rsETH

3. Attacker sends 100 ETH to LRTDepositPool via receive()
   → address(LRTDepositPool).balance = 100.001 ETH

4. Attacker calls LRTOracle.updateRSETHPrice()
   → totalETHInProtocol = 100.001e18
   → rsethSupply = 1e15
   → newRsETHPrice = 100.001e18 * 1e18 / 1e15 = 100_001e18
   → pricePercentageLimit == 0 → isPriceIncreaseOffLimit = false → price accepted

5. Victim calls LRTDepositPool.depositETH{value: 1 ether}(0, "")
   → rsethAmountToMint = 1e18 * 1e18 / 100_001e18 ≈ 9999 wei rsETH
   → Victim receives ~9999 wei rsETH for 1 ETH

6. Pool now holds 101.001 ETH; total rsETH = 1e15 + 9999
   Attacker's share = 1e15 / (1e15 + 9999) ≈ 99.999%
   Attacker's rsETH redeemable value ≈ 101 ETH

   → expectedAssetAmount = 1e15 * rsETHPrice / 1e18 ≈ 101 ETH
   → After operator routes ETH through NDC → UnstakingVault, attacker completes withdrawal
   → Net profit ≈ 1 ETH (victim's deposit), minus gas

        address asset,

        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
