### Title
Forged Foreign-Chain Verification via Unbound `payload_hash` in `respond_verify_foreign_tx` - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that the submitted signature is valid over `response.payload_hash`, but never verifies that `response.payload_hash` is actually derived from the `request` being resolved. A single malicious attested participant (below signing threshold) can replay any previously observed valid MPC signature over a different transaction's hash to fraudulently resolve a pending `verify_foreign_transaction` request, returning a forged attestation to the caller.

### Finding Description

The vulnerability class from the external report is **accounting/state corruption via inclusion of unconverted/unvalidated values**: `_harvestAndReport()` includes `underlyingBalance` in `totalAssets` without verifying those tokens are actually redeemable as `asset`. The direct analog here is that `respond_verify_foreign_tx` includes `response.payload_hash` in the resolved yield without verifying it is actually derived from the `request` being resolved.

In `respond_verify_foreign_tx`: [1](#0-0) 

The contract performs:
```
verify_ecdsa_signature(signature_response, &response.payload_hash.0, &root_pk)
```

It does **not** perform:
```
assert response.payload_hash == SHA-256(borsh(ForeignTxSignPayloadV1 { request: request.request, values: <observed_values> }))
```

The contract cannot perform this check because the extracted `values` are only known to MPC nodes — they are never submitted to the contract. This is structurally identical to the external report's root cause: a value is accepted into the accounting/resolution path without being validated against the canonical expected state.

For comparison, the regular `respond` function correctly takes the payload hash **from the request** (which is stored in the pending map), not from the response: [2](#0-1) 

The `ForeignTxSignPayloadV1` struct that defines what is hashed: [3](#0-2) 

Because `payload_hash = SHA-256(borsh(ForeignTxSignPayload::V1 { request, values }))` binds the hash to a specific `(request, values)` pair, a signature over `hash(tx_B, values_B)` is a valid MPC root-key signature — but it attests to tx_B's properties, not tx_A's. The contract accepts it for tx_A's pending slot regardless.

The attacker-controlled entry path: [4](#0-3) 

### Impact Explanation

A single malicious attested participant (Byzantine, below signing threshold) can:

1. Observe any previously completed `respond_verify_foreign_tx` call on-chain, obtaining a valid `(payload_hash_B, sig_B)` pair for some foreign transaction tx_B.
2. When a new `verify_foreign_transaction` request for tx_A is pending, call `respond_verify_foreign_tx` with `request = tx_A` but `response = {payload_hash_B, sig_B}`.
3. The contract verifies `sig_B` over `payload_hash_B` against the root public key → valid. It resolves the pending yield for tx_A with the fraudulent response.
4. The caller of `verify_foreign_transaction(tx_A)` receives a `VerifyForeignTransactionResponse` whose `payload_hash` encodes the properties of tx_B (e.g., a different block hash, different extracted values), not tx_A.

For bridge use cases (the primary motivation for this feature per the design doc), a downstream contract that trusts the MPC contract's attestation — e.g., releasing funds upon receiving a `VerifyForeignTransactionResponse` — would act on a fraudulent attestation. This enables invalid bridge execution or double-spend conditions: an attacker could attest that a deposit transaction occurred on a foreign chain when it did not, or substitute the block hash of a finalized transaction with one from a different block.

Additionally, the pending request for tx_A is permanently consumed by the fraudulent response. The user cannot receive a correct response without submitting a new request.

**Impact class**: High — forged foreign-chain verification causing invalid bridge execution.

### Likelihood Explanation

- Requires the attacker to be an attested participant (a node running in a TEE). This is within the stated attacker model ("Byzantine participant strictly below the signing threshold").
- The attack requires no cryptographic capability beyond observing on-chain data. Any previously completed `respond_verify_foreign_tx` transaction provides a reusable `(payload_hash, signature)` pair.
- The attack is a single transaction and requires no coordination with other participants.
- The MPC contract is the primary trust anchor for bridge integrations; downstream contracts are likely to trust its attestation without re-verifying the payload hash binding.

### Recommendation

The contract cannot recompute the expected `payload_hash` without the extracted values. Two mitigations are possible:

1. **Include extracted values in the response**: Require MPC nodes to submit the `Vec<ExtractedValue>` alongside the signature in `respond_verify_foreign_tx`. The contract recomputes `expected_hash = SHA-256(borsh(ForeignTxSignPayloadV1 { request: request.request, values }))` and asserts `response.payload_hash == expected_hash` before accepting the response. This makes the contract the authoritative verifier.

2. **Bind the payload hash to the request at submission time**: At `verify_foreign_transaction` submission, store a commitment to the expected request structure in the pending map. At response time, verify the `payload_hash` prefix or structure matches the committed request. This is a weaker check but prevents cross-request replay.

The SDK-level `ForeignChainSignatureVerifier::verify_signature` provides caller-side verification: [5](#0-4) 

However, relying solely on caller-side verification is insufficient for a contract that is named and marketed as a verification primitive. The contract itself must enforce the binding between `request` and `response.payload_hash`.

### Proof of Concept

```
// Setup: tx_B was previously verified; its response is on-chain.
// payload_hash_B = SHA-256(borsh(ForeignTxSignPayloadV1 { request: tx_B, values: [block_hash_B] }))
// sig_B = valid MPC root-key signature over payload_hash_B

// Step 1: User submits verify_foreign_transaction for tx_A (a different transaction)
user.verify_foreign_transaction({ request: tx_A, domain_id, payload_version: V1 })
// → pending request for tx_A is queued

// Step 2: Malicious attested participant replays tx_B's response for tx_A's slot
malicious_participant.respond_verify_foreign_tx(
    request = VerifyForeignTransactionRequest { request: tx_A, domain_id, payload_version: V1 },
    response = VerifyForeignTransactionResponse { payload_hash: payload_hash_B, signature: sig_B }
)
// Contract checks: verify_ecdsa_signature(sig_B, payload_hash_B, root_pk) → OK
// Contract resolves pending yield for tx_A with { payload_hash_B, sig_B }

// Step 3: User's contract receives response attesting to tx_B's properties for tx_A
// If user's contract releases bridge funds based on this attestation → invalid bridge execution
```

The contract's signature check at line 729 passes because `sig_B` is a genuine MPC root-key signature. The missing check — that `payload_hash_B` is derived from `tx_A`'s request — is never performed. [6](#0-5)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
