### Title
Unbound `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies only that the submitted ECDSA signature is cryptographically valid over the caller-supplied `response.payload_hash`. It never checks that `response.payload_hash` is the canonical hash of the `ForeignTxSignPayload` derived from the pending `request`. A single Byzantine attested participant (the signing leader) can therefore replay a legitimately-produced MPC signature from one foreign-tx verification request to resolve a completely different pending request, delivering a forged attestation to the waiting caller.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` performs the following check:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies `sig` over `response.payload_hash`, then immediately resolves the pending yields keyed by `request`:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

There is **no step** that recomputes the expected hash from `request` and compares it to `response.payload_hash`. The canonical hash is defined as:

```
msg_hash = SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request, values })))
``` [3](#0-2) 

Because `values` (the extracted foreign-chain data) are not stored on-chain and are not part of the `VerifyForeignTransactionRequest` key, the contract has no way to reconstruct the expected hash — but it also makes no attempt to do so.
