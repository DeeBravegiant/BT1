Audit Report

## Title
Stale `rsETHPrice` Allows Depositors to Mint Excess rsETH at the Expense of Existing Holders - (File: contracts/LRTDepositPool.sol / contracts/LRTOracle.sol)

## Summary
`LRTOracle.rsETHPrice` is a cached storage variable updated only on explicit calls to `updateRSETHPrice()`. `LRTDepositPool.getRsETHAmountToMint()` divides by this stale value when computing how many rsETH shares to mint. When protocol rewards accrue and raise the true NAV above the stored price, any depositor can mint excess rsETH at a discount, then call `updateRSETHPrice()` themselves to lock in the gain — extracting yield that belongs to existing holders.

## Finding Description
`getRsETHAmountToMint` in `LRTDepositPool.sol` computes the mint amount as:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` reads the public state variable `rsETHPrice` declared in `LRTOracle`:

```solidity
uint256 public override rsETHPrice;
``` [2](#0-1) 

This variable is only written inside `_updateRsETHPrice()`, which is triggered exclusively by explicit calls to `updateRSETHPrice()` or `updateRSETHPriceAsManager()`:

```solidity
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [3](#0-2) 

The true price is computed live from TVL inside `_updateRsETHPrice()`:

```solidity
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
``` [4](#0-3) 

Between reward accrual events (EigenLayer restaking rewards, stETH rebases, etc.) and the next `updateRSETHPrice()` call, `rsETHPrice` is lower than the true NAV. Because the mint formula divides by `rsETHPrice`, a stale-low denominator yields a larger `rsethAmountToMint` than the true NAV warrants.

The `_beforeDeposit` path — called by both `depositETH` and `depositAsset` — performs no price refresh before invoking `getRsETHAmountToMint`: [5](#0-4) 

The `pricePercentageLimit` guard only blocks price increases **above** the configured threshold from non-manager callers:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
if (isPriceIncreaseOffLimit) {
    if (!IAccessControl(address(lrtConfig)).hasRole(LRTConstants.MANAGER, msg.sender)) {
        revert PriceAboveDailyThreshold();
    }
}
``` [6](#0-5) 

For routine daily reward accrual — price increases within the threshold — any EOA can call `updateRSETHPrice()` and complete the attack. The protocol fee minted during the price update slightly reduces the attacker's gain but does not eliminate it.

## Impact Explanation
Every reward cycle in which `rsETHPrice` is stale, an attacker can deposit and receive rsETH representing more ETH than they contributed. After `updateRSETHPrice()` is called, the attacker's rsETH balance is worth more than the deposited amount. The surplus is extracted from the proportional claims of existing rsETH holders, whose share of the TVL is diluted. This constitutes **theft of unclaimed yield** (High impact per the allowed scope).

## Likelihood Explanation
- No privileged role is required; any EOA with ETH or a supported LST can execute the attack.
- Reward accrual events (EigenLayer distributions, stETH rebases) are publicly observable on-chain, making the timing straightforward to monitor.
- `updateRSETHPrice()` is not called atomically with deposits; a staleness window always exists between reward accrual and the next price update.
- The `pricePercentageLimit` caps per-update profit but does not prevent the attack; it is repeatable every reward cycle.

Likelihood: **Medium** (requires on-chain monitoring but no privileged access).

## Recommendation
Force a price update before computing the mint amount. The simplest fix is to call `_updateRsETHPrice()` (or an equivalent internal refresh) at the start of `_beforeDeposit`, or to expose a non-view `getRsETHAmountToMint` that refreshes the price before reading it. Alternatively, enforce a maximum staleness window on `rsETHPrice` and revert deposits if the price has not been updated within that window.

## Proof of Concept
1. EigenLayer rewards accrue; `_getTotalEthInProtocol()` now returns a value higher than `rsethSupply * rsETHPrice` (true price > stored `rsETHPrice`).
2. Attacker calls `LRTDepositPool.depositETH{value: 100 ether}(minAmount, "")`. Because `rsETHPrice` is stale-low, the attacker receives `rsETHAmount = 100e18 * 1e18 / staledLowPrice` — more rsETH than the true NAV warrants.
3. Attacker calls `LRTOracle.updateRSETHPrice()`. The price rises to the true value; the attacker's rsETH balance now represents more ETH than they deposited.
4. Attacker initiates and completes withdrawal via `LRTWithdrawalManager`, recovering more ETH than deposited. The difference is extracted from existing holders' proportional share.

**Foundry fork test outline:**
```solidity
// 1. Fork mainnet at a block where rewards have accrued since last updateRSETHPrice()
// 2. Record rsETHPrice = lrtOracle.rsETHPrice()
// 3. Record attacker's ETH balance before deposit
// 4. attacker.depositETH{value: 100 ether}(0, "")
// 5. Record rsethMinted = rsETH.balanceOf(attacker)
// 6. lrtOracle.updateRSETHPrice() — price increases
// 7. Assert rsethMinted * newRsETHPrice > 100 ether (attacker received excess value)
// 8. Assert existing holders' rsETH * newRsETHPrice < pre-deposit equivalent (dilution confirmed)
```
