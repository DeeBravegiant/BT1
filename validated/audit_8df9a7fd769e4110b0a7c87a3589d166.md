Audit Report

## Title
Uninitialized `rsETHPrice` in `LRTOracle.initialize()` Causes Division-by-Zero on All Deposit Calls - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.initialize()` never assigns `rsETHPrice`, leaving it at the Solidity default of `0`. Every deposit path in `LRTDepositPool` calls `getRsETHAmountToMint()`, which divides by `lrtOracle.rsETHPrice()`. Until `updateRSETHPrice()` is explicitly called, all deposits revert with a division-by-zero panic, preventing the protocol from accepting any funds.

## Finding Description
`LRTOracle.initialize()` only sets `lrtConfig` and emits an event, leaving `rsETHPrice` at `0`: [1](#0-0) 

`rsETHPrice` is a plain storage variable with no default: [2](#0-1) 

`LRTDepositPool.getRsETHAmountToMint()` performs an unconditional division by `lrtOracle.rsETHPrice()`: [3](#0-2) 

This is reached from `_beforeDeposit()` on every deposit: [4](#0-3) 

Which is called unconditionally by both `depositETH()` and `depositAsset()`: [5](#0-4) [6](#0-5) 

`_updateRsETHPrice()` does seed `rsETHPrice = 1 ether` when `rsethSupply == 0`, but only when explicitly invoked: [7](#0-6) 

`updateRSETHPrice()` is `public whenNotPaused`, so any caller can trigger the fix — but it must be called first: [8](#0-7) 

There is no guard in the deposit path that checks for a zero `rsETHPrice` before dividing.

## Impact Explanation
Between contract deployment (or any proxy upgrade that resets storage) and the first call to `updateRSETHPrice()`, every invocation of `depositETH()` or `depositAsset()` reverts with an EVM division-by-zero panic. No user funds are lost because the revert prevents any state change, but the protocol entirely fails to deliver its core promised service — accepting deposits.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The failure window opens on every fresh deployment or storage-resetting upgrade and closes only after `updateRSETHPrice()` is called. Because the function is permissionless, any user who encounters the revert can self-remediate immediately. The window is short in practice but is a real, deterministic failure mode for any depositor who acts before the price is seeded.

## Recommendation
Initialize `rsETHPrice` (and `highestRsethPrice`) to `1 ether` inside `LRTOracle.initialize()`, mirroring the zero-supply logic already present in `_updateRsETHPrice()`:

```solidity
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
+   rsETHPrice = 1 ether;
+   highestRsethPrice = 1 ether;
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

## Proof of Concept
1. Deploy `LRTOracle` proxy and call `initialize(lrtConfigAddr)`. `rsETHPrice` is `0`.
2. Do **not** call `updateRSETHPrice()`.
3. Call `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `getRsETHAmountToMint()` → `(1e18 * assetPrice) / 0` → EVM division-by-zero panic → revert.
5. Call `updateRSETHPrice()` (permissionless, no role required). `rsETHPrice` is now `1 ether`.
6. Repeat step 3 — deposit succeeds.
