Audit Report

## Title
Donation-Inflated `rsETHPrice` via Public `updateRSETHPrice()` Causes Zero rsETH Minting for Depositors - (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary
`LRTOracle.updateRSETHPrice()` is public and permissionless, and `rsETHPrice` is computed from live token balances of the deposit pool. An attacker can donate ETH directly to `LRTDepositPool` via its unrestricted `receive()`, call `updateRSETHPrice()` to commit the inflated price, and cause a subsequent victim depositor to receive zero rsETH for a real deposit. The victim's funds are absorbed into the protocol and accrue to existing rsETH holders, including the attacker.

## Finding Description

**Root cause — live balance accounting in TVL:**
`LRTDepositPool.getAssetDistributionData()` reads `IERC20(asset).balanceOf(address(this))` for LST assets [1](#0-0)  and `address(this).balance` for ETH [2](#0-1) , both of which include any tokens sent directly to the contract. The deposit pool accepts raw ETH via an unrestricted `receive()`. [3](#0-2) 

**Step 1 — Donation inflates `totalETHInProtocol`:**
`LRTOracle._getTotalEthInProtocol()` calls `ILRTDepositPool.getTotalAssetDeposits(asset)` for every supported asset, which in turn reads the live balances above. [4](#0-3) 

**Step 2 — `updateRSETHPrice()` is public:**
Anyone can call it at any time to commit the inflated TVL into the stored `rsETHPrice`. [5](#0-4) 

**Step 3 — Price is computed as `totalETHInProtocol / rsethSupply`:**
Donating tokens increases the numerator without increasing `rsethSupply`, so `rsETHPrice` rises proportionally. [6](#0-5) 

**Step 4 — Minting uses integer division against the cached inflated price:**
`getRsETHAmountToMint()` computes `(amount * assetPrice) / rsETHPrice`. If `rsETHPrice` has been inflated to `N × 1e18` and the victim deposits `M` ETH where `M < N`, the result rounds down to zero. [7](#0-6) 

**Step 5 — Zero-mint is not blocked when `minRSETHAmountExpected = 0`:**
`_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`. When both are zero, the check passes and the victim receives nothing. [8](#0-7) 

**Why the `pricePercentageLimit` guard fails:**
`pricePercentageLimit` is not set in `initialize()`, so it defaults to `0`. [9](#0-8)  The guard condition is `pricePercentageLimit > 0 && ...`, meaning when `pricePercentageLimit == 0` the guard is entirely disabled and any price increase is accepted from any caller. [10](#0-9) 

## Impact Explanation

A depositor loses their entire ETH or LST deposit and receives zero rsETH in return. The deposited value is redistributed to existing rsETH holders, including the attacker. This constitutes **direct theft of user funds at rest**, matching the Critical impact class: "Direct theft of any user funds, whether at-rest or in-motion, other than unclaimed yield."

## Likelihood Explanation

The attack requires the attacker to hold a small amount of rsETH (or be the first depositor), donate an amount of ETH/LST at least equal to the victim's deposit, and call the public `updateRSETHPrice()`. Because `pricePercentageLimit` defaults to `0` at deployment, the guard is disabled from day one and no incremental stepping is needed. The attacker can front-run a victim's pending deposit transaction. The capital requirement (donation ≥ victim deposit) is the primary constraint, making likelihood **Low**, but the attack is fully permissionless and requires no privileged access.

## Recommendation

1. **Remove live `balanceOf` from TVL accounting.** Track deposited amounts in an internal accounting variable rather than reading `balanceOf(address(this))` or `address(this).balance`. Donated tokens should not affect `rsETHPrice`.
2. **Enforce a non-zero `pricePercentageLimit` at initialization.** Set a conservative default (e.g., 1% = `1e16`) in `initialize()` so the guard is active from deployment.
3. **Enforce a minimum rsETH output.** Revert in `_beforeDeposit` if `rsethAmountToMint == 0`, regardless of `minRSETHAmountExpected`.
4. **Restrict `updateRSETHPrice()`.** Consider making it callable only by a trusted keeper or the manager role, preventing an attacker from committing an inflated price on demand.

## Proof of Concept

```
Initial state:
  rsETHPrice = 1e18 (rsethSupply == 0 → price reset to 1 ether)
  pricePercentageLimit = 0 (default, guard disabled)

Step 1: Attacker calls depositETH{value: 1 wei}(0, "")
  rsethAmountToMint = (1 * 1e18) / 1e18 = 1 wei rsETH
  rsethSupply = 1 wei

Step 2: Attacker sends 1000 ETH directly to LRTDepositPool (via receive())
  address(this).balance = 1000 ETH + 1 wei

Step 3: Attacker calls LRTOracle.updateRSETHPrice()
  totalETHInProtocol ≈ 1000e18 (includes donation)
  newRsETHPrice = (1000e18 + 1) / 1 ≈ 1000e18
  rsETHPrice is now ≈ 1000e18
  (pricePercentageLimit == 0 → guard skipped, call succeeds for any caller)

Step 4: Victim calls depositETH{value: 500 ETH}(0, "")
  rsethAmountToMint = (500e18 * 1e18) / 1000e18 = 0  (rounds down)
  _beforeDeposit: 0 < 0 is false → no revert
  Victim receives 0 rsETH; 500 ETH absorbed into protocol

Step 5: Attacker holds 1 wei rsETH out of total supply ≈ 1 wei
  (treasury may hold a small fee share, but attacker controls the vast majority)
  Attacker redeems rsETH → recovers ~1500 ETH
```

Foundry test plan: deploy `LRTDepositPool` and `LRTOracle` on a local fork; execute the five steps above; assert `IRSETH(rsETH).balanceOf(victim) == 0` and `address(depositPool).balance >= 1500 ether` after victim deposit.

        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

            // assetER is in 1e18 precision (1.0 = 1e18)
