### Title
Missing Zero-Length Shared-Secret Validation in `kem_encap_shim` Allows Zero-Entropy DEM Key — (`src/cbmpc/c_api/pve_base_pke.cpp`)

### Summary

The guard in `kem_encap_shim` that validates the KEM shared-secret returned by a caller-supplied `encap` callback does not reject the `{nullptr, 0}` case. A malicious or buggy callback returning `size=0 / data=nullptr` for `out_kem_ss` passes the guard, propagates an empty `buf_t` into `kem_aead_ciphertext_t::seal`, and causes HKDF to be invoked with a zero-length IKM. OpenSSL accepts this and produces a fully deterministic, fixed AES-GCM-256 key. The library returns `CBMPC_SUCCESS` and emits a structurally valid ciphertext whose DEM key has zero entropy.

### Finding Description

**Guard in `kem_encap_shim`:** [1](#0-0) 

The condition `kem_ss.size < 0 || (kem_ss.size > 0 && !kem_ss.data)` has a gap: it rejects negative sizes and the inconsistent `(size>0, data=null)` pair, but it explicitly allows `(size=0, data=null)`. When the callback returns `{nullptr, 0}`, both sub-conditions are false, so `out_kem_ss` is assigned `coinbase::buf_t(nullptr, 0)` — an empty buffer — and the function returns `CBMPC_SUCCESS`.

**Propagation through `kem_policy_runtime_kem_t::encapsulate`:** [2](#0-1) 

The return value of the shim is forwarded directly; no minimum-size check is applied to `kem_ss` after the call.

**Key derivation in `kem_aead_ciphertext_t::seal`:** [3](#0-2) 

`hkdf_extract_sha256(mem_t(), kem_ss)` is called with an empty `kem_ss`. The `mem_t` conversion of an empty `buf_t` yields `size=0`; the HKDF implementation passes this directly to OpenSSL: [4](#0-3) 

OpenSSL's HKDF with a zero-length IKM is well-defined and succeeds, producing a fixed, fully deterministic PRK. The subsequent `hkdf_expand` and `aes_gcm_t::encrypt` calls succeed, and `seal` returns `SUCCESS`.

**Entry point:** [5](#0-4) 

`cbmpc_pve_encrypt_with_kem` is a public C API that accepts a caller-supplied `cbmpc_pve_base_kem_t` struct. No minimum shared-secret length is enforced anywhere between the public entry point and the HKDF call.

### Impact Explanation

Any caller who provides a `cbmpc_pve_base_kem_t::encap` callback that returns `{nullptr, 0}` for `out_kem_ss` — whether maliciously or due to a bug — will receive `CBMPC_SUCCESS` and a structurally valid PVE ciphertext. The AES-GCM-256 key protecting that ciphertext is derived from an empty IKM, making it a fixed, globally-known constant for anyone who knows the HKDF label `"CBMPC|KEM-AEAD|v1|KDF=HKDF-SHA256|AEAD=AES-GCM-256"`. The plaintext is effectively unprotected. The library provides no signal that the output is cryptographically invalid.

### Likelihood Explanation

The `cbmpc_pve_base_kem_t` callback interface is a public, documented API surface intended for HSM or FFI integrators. A callback implementation that returns an empty shared secret on error (instead of a non-zero error code) is a plausible implementation mistake. The library's validation is the last line of defense and it fails silently for this case.

### Recommendation

Add an explicit minimum-length check for `kem_ss` in `kem_encap_shim` (and symmetrically in `kem_decap_shim`):

```cpp
// After the existing guard:
if (kem_ss.size == 0) {
    if (kem_ct.data) cbmpc_cmem_free(kem_ct);
    return coinbase::error(E_FORMAT, "KEM shared secret must be non-empty");
}
```

A stronger fix would enforce a minimum entropy threshold (e.g., `kem_ss.size < 16`) consistent with the 128-bit security level of the AES-GCM-256 DEM.
