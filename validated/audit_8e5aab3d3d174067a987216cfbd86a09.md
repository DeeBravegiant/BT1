Audit Report

## Title
Lax ETH Deposit Limit Check Allows Deposits Beyond Protocol Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison: the ETH branch checks only `totalAssetDeposits > limit` while the ERC-20 branch correctly checks `totalAssetDeposits + amount > limit`. Any depositor can push the ETH total above the configured cap in a single `depositETH` call, causing rsETH to be minted for ETH that cannot be restaked, diluting yield for all rsETH holders.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, `_checkIfDepositAmountExceedesCurrentLimit` branches on asset type:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
}
```

For ETH, the function returns `false` (not exceeded) whenever `totalAssetDeposits ≤ limit`, regardless of `amount`. This function is the sole guard called by `_beforeDeposit` (lines 661–663), which is invoked by `depositETH` (line 87). Because `amount` is never added before the comparison, a deposit that would push the running total from `limit - ε` to `limit + X` passes the check and proceeds to `_mintRsETH`. The ERC-20 path via `depositAsset` is unaffected.

## Impact Explanation
The deposit limit is the protocol's primary mechanism for capping ETH exposure to match EigenLayer restaking capacity. Bypassing it allows unlimited ETH to be deposited and rsETH to be minted beyond the intended ceiling. Excess ETH that cannot be restaked sits idle in the deposit pool or NDCs without generating yield, meaning rsETH holders receive diluted returns on that portion. No direct theft occurs, but the contract fails to deliver the promised deposit-cap guarantee.

**Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
The condition is reachable by any unprivileged depositor calling `depositETH` with `msg.value > 0` at any time the ETH total is at or below the configured limit. No special role, timing, or front-running is required. The asymmetry is always present and the gap between the lax check and the correct check grows proportionally with deposit size.

**Likelihood: High.**

## Recommendation
Apply the same inclusive check used for ERC-20 assets to the ETH branch in `_checkIfDepositAmountExceedesCurrentLimit`:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Depositor calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `999 ether > 1000 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for 500 ETH; total ETH in protocol becomes `1499 ether`, 49.9% above the cap.
6. The same depositor (or any other) can repeat until EigenLayer capacity or other external constraints are hit.

**Foundry test plan:** Deploy `LRTDepositPool` with a mock `LRTConfig` returning `depositLimitByAsset = 1000 ether`. Pre-seed `getTotalAssetDeposits` to return `999 ether`. Call `depositETH{value: 500 ether}` and assert it does not revert and that the post-deposit total exceeds the configured limit. Confirm the same call via `depositAsset` with an ERC-20 reverts with `MaximumDepositLimitReached`. [1](#0-0) [2](#0-1) [3](#0-2)
