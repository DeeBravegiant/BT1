Audit Report

## Title
Missing L2 Sequencer Uptime and Time-Based Staleness Check Allows Stale Price Exploitation After Sequencer Recovery - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs no L2 sequencer uptime feed check and no time-based heartbeat staleness check on the Chainlink price returned by `latestRoundData()`. After an L2 sequencer outage during which a collateral token's price declined, the stale pre-outage price is accepted as valid, allowing any depositor to mint wrsETH at an inflated rate and diluting all existing rsETH/wrsETH holders.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 26–37) calls `AggregatorV3Interface(oracle).latestRoundData()` and applies only two guards:

- `answeredInRound < roundID` — detects an incomplete round
- `timestamp == 0` — detects a missing timestamp

Neither of these guards catches a price that is temporally stale. There is no comparison of `block.timestamp` against `timestamp` (heartbeat check), and no query to a Chainlink L2 sequencer uptime feed. A price last updated hours before a sequencer outage passes both checks unchanged. [1](#0-0) 

This oracle is wired into the deposit path of `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge` via `supportedTokenOracle[token]`. `viewSwapRsETHAmountAndFee` uses the returned `tokenToETHRate` to compute the wrsETH amount minted to the depositor: [2](#0-1) [3](#0-2) 

The public `deposit(address token, uint256 amount, string referralId)` function is callable by any user with no privilege requirement: [4](#0-3) 

## Impact Explanation
When the L2 sequencer recovers from downtime, Chainlink may not have yet pushed updated prices. A depositor calling `deposit()` in this window receives wrsETH computed at the stale (higher) `tokenToETHRate`. The pool receives collateral worth less than the wrsETH minted against it, permanently diluting all existing rsETH/wrsETH holders. This constitutes **theft of unclaimed yield** (High severity) from existing holders, as their proportional claim on the underlying collateral is reduced without compensation.

## Likelihood Explanation
Arbitrum sequencer outages have occurred historically and are a documented operational risk for all L2 deployments. The protocol explicitly targets Arbitrum and other L2s. No special permissions, governance access, or attacker setup are required — any unprivileged user calling `deposit()` immediately after sequencer recovery triggers the issue. The exploit window exists for as long as Chainlink has not yet published the updated price post-recovery, which can span multiple blocks.

## Recommendation
Add a Chainlink L2 sequencer uptime feed check and a time-based heartbeat staleness check to `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
address public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600;
uint256 public constant HEARTBEAT = 3600; // configure per feed

function getRate() public view returns (uint256) {
    if (sequencerUptimeFeed != address(0)) {
        (, int256 seqAnswer, uint256 startedAt,,) =
            AggregatorV3Interface(sequencerUptimeFeed).latestRoundData();
        if (seqAnswer != 0) revert SequencerDown();
        if (block.timestamp - startedAt <= GRACE_PERIOD) revert GracePeriodNotOver();
    }

    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    if (block.timestamp - timestamp > HEARTBEAT) revert StalePrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

## Proof of Concept
1. Arbitrum sequencer goes offline. wstETH/ETH Chainlink price on Arbitrum is 1.2e18.
2. During outage, wstETH price drops to 1.1e18 on mainnet; Chainlink cannot push the update to Arbitrum.
3. Sequencer recovers. `latestRoundData()` still returns the pre-outage price: `timestamp` is non-zero, `answeredInRound == roundID`, `ethPrice = 1.2e18` — all existing checks pass.
4. Attacker calls `RSETHPoolV3.deposit(wstETH, 100e18, "")`.
5. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = 100e18 * 1.2e18 / rsETHToETHrate`, minting wrsETH at the stale 1.2 rate instead of the correct 1.1 rate.
6. Attacker receives ~9% excess wrsETH relative to actual collateral value, diluting all existing holders.

**Foundry fork test plan:** Fork Arbitrum mainnet at a block just after a historical sequencer recovery event. Deploy or point to the live `ChainlinkOracleForRSETHPoolCollateral`. Manipulate `block.timestamp` to simulate the post-recovery window before Chainlink updates. Call `getRate()` and assert it returns the pre-outage stale price. Then call `RSETHPoolV3.deposit()` and assert the minted wrsETH amount exceeds what the current fair price would produce.
