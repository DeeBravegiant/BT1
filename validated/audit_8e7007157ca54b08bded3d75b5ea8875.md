Audit Report

## Title
Stale rsETH Rate Masked by Fresh ETH/USD Metadata in `RSETHPriceFeed.latestRoundData()` — (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed.latestRoundData()` computes `answer` as `rsETHPrice × ETH/USD` but returns `updatedAt` and `answeredInRound` sourced exclusively from the ETH/USD Chainlink feed. Because `LRTOracle.rsETHPrice` is a stored state variable updated only via on-chain transactions, any staleness in the rsETH/ETH rate is structurally invisible to consumers performing standard Chainlink freshness checks on the returned metadata.

## Finding Description
`RSETHPriceFeed.latestRoundData()` at lines 63–70 of `contracts/oracles/RSETHPriceFeed.sol` copies all five return values from `ETH_TO_USD.latestRoundData()` and then overwrites only `answer`:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

`RS_ETH_ORACLE.rsETHPrice()` resolves to `LRTOracle.rsETHPrice` (line 28 of `contracts/LRTOracle.sol`), a plain storage variable. It is only written at line 313 inside `_updateRsETHPrice()`, which is invoked by the `public whenNotPaused` function `updateRSETHPrice()` (lines 87–89). No timestamp of the last rsETH price update is stored or consulted anywhere.

**Exploit path (block stuffing variant):**
1. Attacker identifies a downstream consumer of `RSETHPriceFeed` that applies `require(block.timestamp - updatedAt < heartbeat)` or `require(answeredInRound >= roundId)`.
2. Attacker performs block stuffing to prevent `updateRSETHPrice()` transactions from being mined, freezing `rsETHPrice` at its last stored value.
3. Chainlink's decentralized network continues updating ETH/USD normally; `updatedAt` and `answeredInRound` from `ETH_TO_USD` remain fresh.
4. `latestRoundData()` returns a recent `updatedAt` (passes staleness check) and `answeredInRound == roundId` (passes round check), but `answer` embeds an arbitrarily old rsETH/ETH rate.
5. The downstream consumer's staleness guard passes on a price whose rsETH component is frozen.

**Simpler path (keeper failure):** If the keeper calling `updateRSETHPrice()` fails for any reason, the same metadata mismatch occurs without any attacker action — `updatedAt` stays fresh from ETH/USD while `rsETHPrice` is stale.

Existing guards are insufficient: `whenNotPaused` on `updateRSETHPrice()` only prevents calls when the contract is paused; it does not protect against the rsETH price going stale while the contract is live.

## Impact Explanation
The contract unconditionally forwards ETH/USD freshness metadata as if it represented the composite rsETH/USD price. This violates the implicit contract of `AggregatorV3Interface`: that `updatedAt` reflects when the returned `answer` was last valid. Consumers relying on standard Chainlink staleness patterns receive a false freshness guarantee. Impact maps to **Low — Contract fails to deliver promised returns, but does not directly lose value**, with block stuffing as the active enabling mechanism (also a listed Low impact).

## Likelihood Explanation
Block stuffing on Ethereum mainnet is expensive but has been executed in practice. The attacker must sustain stuffing long enough for rsETH price to diverge meaningfully, then profit via a downstream protocol. Keeper failure is a lower-cost path that requires no attacker action. The structural flaw exists unconditionally in the deployed code regardless of attack vector, making the metadata mismatch a persistent latent risk. Likelihood is low-to-medium.

## Recommendation
Store a `rsETHPriceUpdatedAt` timestamp in `LRTOracle` alongside `rsETHPrice`, written at line 313 whenever `rsETHPrice` is updated. In `RSETHPriceFeed.latestRoundData()`, return `min(ethUsdUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt`. This ensures any staleness check on `updatedAt` reflects the freshness of the weakest component of the composite price.

## Proof of Concept
```solidity
// Foundry local fork test — no mainnet interaction required
// 1. Deploy MockEthUsdFeed (AggregatorV3Interface) and MockLRTOracle
// 2. Deploy RSETHPriceFeed(mockEthUsd, mockLRTOracle, "rsETH/USD")
// 3. Set mockLRTOracle.rsETHPrice = 1.05e18; mockEthUsd.answer = 3000e8;
//    mockEthUsd.updatedAt = block.timestamp
// 4. vm.warp(block.timestamp + 2 hours)
//    — do NOT call updateRSETHPrice() (simulates keeper failure / block stuffing)
// 5. mockEthUsd.updatedAt = block.timestamp  (ETH/USD updates normally)
// 6. (roundId, answer, , updatedAt, answeredInRound) = feed.latestRoundData()
// 7. Assert: block.timestamp - updatedAt < heartbeat  => passes (ETH/USD is fresh)
//            answeredInRound == roundId               => passes (from ETH/USD)
//            answer == 1.05e18 * 3000e8 / 1e18        => stale rsETH rate, 2h old
// Invariant broken: staleness check passes, but rsETH component is 2 hours stale
```

The assertion holds on unmodified code because `latestRoundData()` has no mechanism to detect or signal rsETH price staleness — it unconditionally forwards ETH/USD metadata regardless of how long ago `rsETHPrice` was last written.
