Audit Report

## Title
Fee Bypass via Integer Truncation in Deposit Fee Calculation - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPool.sol)

## Summary
The deposit fee formula `fee = amount * feeBps / 10_000` truncates to zero in Solidity integer arithmetic whenever `amount * feeBps < 10_000`. Because the only lower-bound guard on a deposit is `amount != 0`, any unprivileged caller can split an arbitrarily large deposit into sub-threshold micro-transactions and pay zero fees on the full notional value, permanently preventing `feeEarnedInETH` / `feeEarnedInToken[token]` from accruing the protocol revenue it is owed.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256)` (ETH path) and `viewSwapRsETHAmountAndFee(uint256,address)` (ERC-20 path), the fee is computed as:

```solidity
fee = amount * feeBps / 10_000;   // RSETHPoolV3.sol L300, L324
```

The identical pattern appears in `RSETHPool.viewSwapRsETHAmountAndFee(uint256)` (L312) and `RSETHPool.viewSwapRsETHAmountAndFee(uint256,address)` (L336, using `tokenFeeBps`).

The deposit entry points enforce only:

```solidity
if (amount == 0) revert InvalidAmount();   // RSETHPoolV3.sol L256, L282; RSETHPool.sol L269, L294
```

No minimum deposit amount is enforced. When `amount * feeBps < 10_000`, Solidity truncates the result to `0`. The fee accumulator is then incremented by zero:

```solidity
feeEarnedInETH += fee;          // RSETHPoolV3.sol L260; RSETHPool.sol L273
feeEarnedInToken[token] += fee; // RSETHPoolV3.sol L288; RSETHPool.sol L300
```

**Exploit path:**
1. Attacker reads the public `feeBps` variable (e.g., `feeBps = 5`, 0.05%).
2. Computes the bypass threshold: `10_000 / 5 = 2_000 wei`. Any single deposit below 2,000 wei produces `fee = 0`.
3. Calls `deposit(referralId)` 500,000 times, each with `msg.value = 1_999 wei`.
4. Each call: `fee = 1_999 * 5 / 10_000 = 0`. `feeEarnedInETH` is never incremented.
5. Total deposited ≈ 1 ETH. Total fee paid: **0 wei**. Attacker receives full rsETH value on every micro-deposit.
6. `withdrawFees` yields nothing for this deposited value.

The same structural vulnerability is present in `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`, all of which share the identical `feeBps / 10_000` formula.

## Impact Explanation
**High — Theft of unclaimed yield.**

The `feeEarnedInETH` and `feeEarnedInToken[token]` accumulators represent protocol fee revenue that has been earned but not yet withdrawn via `withdrawFees`. By forcing fee truncation to zero on every micro-deposit, the attacker permanently prevents these accumulators from reflecting the fees owed, causing the protocol to lose all fee revenue on the exploited deposit volume. The loss is permanent and irreversible: once a deposit is processed with `fee = 0`, the missing fee revenue cannot be recovered.

## Likelihood Explanation
**High.** The attack requires no special role, no front-running, and no external dependency — any EOA or contract that can call `deposit()` can exploit this. The bypass threshold is trivially computable from the public `feeBps` state variable. On L2 networks (Arbitrum, Optimism, Base, etc.) where these contracts are deployed, gas costs per transaction are on the order of fractions of a cent, making hundreds of thousands of micro-deposits economically rational. The attack scales linearly with deposit size and is repeatable indefinitely.

## Recommendation
Replace floor division with ceiling division (protocol-favoring rounding) in all fee calculation sites:

```solidity
// Before (rounds down — vulnerable)
fee = amount * feeBps / 10_000;

// After (rounds up — protocol-favoring)
fee = (amount * feeBps + 9_999) / 10_000;
```

Alternatively, enforce a minimum deposit amount such that `amount * feeBps >= 10_000` always holds (i.e., `require(amount >= 10_000 / feeBps)`). Apply the fix uniformly across all pool variants: `RSETHPoolV3`, `RSETHPool`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, and `RSETHPoolNoWrapper`.

## Proof of Concept
**Foundry fuzz test plan:**

```solidity
function testFuzz_feeBypass(uint256 iterations, uint256 dustAmount) public {
    // Setup: feeBps = 5, rsETH/ETH rate = 1.05e18
    vm.assume(iterations > 0 && iterations <= 1_000_000);
    vm.assume(dustAmount > 0 && dustAmount < 10_000 / feeBps); // below threshold

    uint256 feesBefore = pool.feeEarnedInETH();
    for (uint256 i = 0; i < iterations; i++) {
        pool.deposit{value: dustAmount}("ref");
    }
    uint256 feesAfter = pool.feeEarnedInETH();

    // Fee accumulator never incremented despite non-zero deposits
    assertEq(feesAfter, feesBefore);

    // Compare: single deposit of equivalent total value DOES accrue fees
    uint256 totalValue = dustAmount * iterations;
    pool.deposit{value: totalValue}("ref");
    assertGt(pool.feeEarnedInETH(), feesAfter);
}
```

**Concrete sequence (no fork required):**
1. Deploy `RSETHPoolV3` with `feeBps = 5`.
2. Call `deposit{value: 1999}("ref")` 500,000 times.
3. Assert `feeEarnedInETH == 0`.
4. Call `deposit{value: 1e18}("ref")` once.
5. Assert `feeEarnedInETH == 5e14` (0.05% of 1 ETH).
6. Steps 3 and 5 together prove the bypass: identical notional value deposited, zero fees in the micro-deposit path vs. correct fees in the single-deposit path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
