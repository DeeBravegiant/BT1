Audit Report

## Title
Missing Domain Separation in Merkle Leaf Enables Cross-Chain Proof Replay — (`contracts/KERNEL/KernelTop100MerkleDistributor.sol`)

## Summary
`KernelTop100MerkleDistributor._verifyClaimProof` constructs the merkle leaf as `keccak256(abi.encodePacked(user, amount))` with no `address(this)` or `block.chainid` binding. Because the merkle root is immutable after initialization and claim state is tracked per-contract-instance, a user with a valid proof on one chain can replay the identical proof on any other chain where the same root is deployed, claiming their full allocation on each chain independently.

## Finding Description
In `_verifyClaimProof` at line 293, the leaf is constructed without any chain or contract context:

```solidity
bytes32 leaf = keccak256(abi.encodePacked(user, amount));
``` [1](#0-0) 

The merkle root is set once at initialization and there is no `setMerkleRoot` function in this contract, making it permanent for the lifetime of each deployment: [2](#0-1) 

Claim state is tracked exclusively in the per-instance `userClaims` mapping, which records `amountClaimed` only within the local contract: [3](#0-2) 

The guard in `_getUnclaimedVestedAmount` checks `userClaim.amountClaimed >= userTotalClaimableAmount` against the local mapping only: [4](#0-3) 

Because the leaf hash `keccak256(alice, 1000e18)` is identical on every EVM chain, a proof `P` valid on chain A is equally valid on chain B when the same root `R` is deployed there. The Arbitrum instance has its own independent `userClaims` storage where Alice's `amountClaimed` starts at zero, so the proof passes and the full vested amount is paid out again.

The same structural gap exists in `KernelMerkleDistributor._processClaim`: [5](#0-4) 

And in `MerkleDistributor.claim`: [6](#0-5) 

## Impact Explanation
**High — Theft of unclaimed yield.** KERNEL tokens distributed through `KernelTop100MerkleDistributor` are protocol rewards. A user entitled to `X` KERNEL on chain A can replay the identical proof on every additional chain where the same root is live and receive `X` KERNEL per chain. The excess tokens are drawn from each contract's funded balance, directly reducing the pool available to other eligible Top-100 recipients or the protocol treasury.

## Likelihood Explanation
LRT-rsETH is an explicitly multi-chain protocol with pool contracts deployed on Arbitrum, Optimism, Base, and others. Deploying the same KERNEL reward distribution (same merkle root) across multiple chains is a natural operational step for a "Top 100" cross-chain reward campaign. The attacker requires only: (1) a legitimately obtained proof, which every eligible user possesses, and (2) knowledge that the same root is live on another chain — observable on-chain. No privileged access, no leaked keys, no front-running, and no victim cooperation is required.

## Recommendation
Bind every leaf to its specific deployment context by including `address(this)` and `block.chainid`:

```solidity
// KernelTop100MerkleDistributor._verifyClaimProof
bytes32 leaf = keccak256(abi.encodePacked(address(this), block.chainid, user, amount));
```

Apply the same fix to `KernelMerkleDistributor._processClaim` and `MerkleDistributor.claim`:

```solidity
bytes32 node = keccak256(abi.encodePacked(address(this), block.chainid, index, account, cumulativeAmount));
```

Off-chain merkle tree generation must be updated to hash leaves with the same domain prefix so that existing proofs are invalidated and new chain-bound proofs are issued.

## Proof of Concept
1. Protocol generates a merkle tree for the Top-100 KERNEL distribution. Alice's leaf: `keccak256(abi.encodePacked(alice, 1000e18))`. Root = `R`.
2. Protocol deploys `KernelTop100MerkleDistributor` on Ethereum mainnet with root `R`, funded with 100,000 KERNEL.
3. Protocol deploys a second `KernelTop100MerkleDistributor` on Arbitrum with the **same root `R`**, funded with 100,000 KERNEL.
4. Alice calls `claim(1000e18, proof)` on Ethereum. Proof verifies; `userClaims[alice].amountClaimed = 1000e18` on the mainnet instance.
5. Alice calls `claim(1000e18, proof)` on Arbitrum with the **identical proof**. The Arbitrum contract's `userClaims[alice].amountClaimed` is 0. The leaf hash is identical. Proof verifies. Alice receives another ~1,000 KERNEL (minus fee).
6. Alice has claimed ~2,000 KERNEL instead of ~1,000, stealing ~1,000 KERNEL from the Arbitrum contract's balance.

**Foundry fork test plan:** Deploy two instances of `KernelTop100MerkleDistributor` on two forked chains (or two local deployments with different `block.chainid` via `vm.chainId`), initialize both with the same root `R`, fund both, generate a proof for Alice off-chain, call `claim` on both instances with the same proof, and assert Alice's total received amount exceeds her single-chain entitlement.
