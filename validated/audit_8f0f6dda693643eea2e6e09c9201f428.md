Audit Report

## Title
Permanent Fund Freeze When ETH Withdrawal Recipient Cannot Receive ETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager` implements a two-phase withdrawal where rsETH is burned by the operator in `unlockQueue()` before the user calls `completeWithdrawal()`. If the user's address is a smart contract without a `receive()` function, the ETH transfer in `_processWithdrawalCompletion()` permanently reverts with no recovery path. The burned rsETH is irrecoverable and the ETH remains locked in the contract indefinitely.

## Finding Description
The withdrawal lifecycle is split across two separate transactions:

1. **`initiateWithdrawal()`** (L166): rsETH is transferred from `msg.sender` to the contract via `safeTransferFrom`. This succeeds for any address that holds rsETH, including smart contracts with no `receive()` function.

2. **`unlockQueue()`** (L305–307): The operator burns rsETH from the contract (`burnFrom(address(this), rsETHBurned)`) and pulls the corresponding asset from `LRTUnstakingVault` into `LRTWithdrawalManager`. This is irreversible.

3. **`completeWithdrawal()`** → `_processWithdrawalCompletion()` (L699–738): At L734, `_transferAsset(asset, user, request.expectedAssetAmount)` is called. For ETH, this executes `payable(to).call{value: amount}("")` (L878). If `to` has no `receive()` function, this call returns `false`, triggering `revert EthTransferFailed()` (L879).

Because Solidity reverts all state changes on revert, the `popFront()` (L705), `delete withdrawalRequests[requestId]` (L712), and `unlockedWithdrawalsCount[asset]--` (L717) are all rolled back. The withdrawal request remains in the queue in a permanently uncompletable state.

The operator-callable `completeWithdrawalForUser()` (L192–204) also calls `_processWithdrawalCompletion(asset, user, referralId)` with the same `user` address, providing no relief.

There is no admin or operator function to cancel an unlocked withdrawal, redirect assets to an alternative address, or return assets to the unstaking vault. `sweepRemainingAssets()` (L395–414) is blocked by the `hasUnlockedWithdrawals(asset)` guard (L403) as long as `unlockedWithdrawalsCount[asset] > 0`.

## Impact Explanation
**Critical — Permanent freezing of funds.** After `unlockQueue()` burns the user's rsETH and moves ETH into `LRTWithdrawalManager`, a smart contract user whose address cannot receive ETH loses both their rsETH (burned, irrecoverable) and their underlying ETH (permanently locked in the contract). No on-chain recovery mechanism exists. `sweepRemainingAssets()` is also permanently blocked for the ETH asset as long as the stuck withdrawal exists.

## Likelihood Explanation
Smart contracts that hold rsETH (e.g., yield aggregators, vaults, multisigs) can call `initiateWithdrawal(ETH_TOKEN, amount, "")` successfully since it only requires an ERC20 `safeTransferFrom`. A contract without a `receive()` function — which is common for contracts that only interact with ERC20 tokens — will trigger this freeze. No attacker action is required; the victim contract simply needs to initiate an ETH withdrawal. The operator-controlled `unlockQueue()` step is normal protocol operation and creates the irreversible burn before the delivery failure is discovered.

## Recommendation
1. Add a `receiver` parameter to `initiateWithdrawal()` and store it in the `WithdrawalRequest` struct:
```solidity
struct WithdrawalRequest {
    uint256 rsETHUnstaked;
    uint256 expectedAssetAmount;
    uint256 withdrawalStartBlock;
    address receiver; // added
}
```
2. In `_processWithdrawalCompletion()`, transfer to `request.receiver` instead of `user`, allowing the initiating contract to specify a capable recipient at request time.
3. Add an operator function to forcibly cancel a stuck unlocked withdrawal by returning the committed asset amount to the unstaking vault (or treasury) and decrementing `unlockedWithdrawalsCount[asset]`, providing a recovery path when delivery is permanently impossible.

## Proof of Concept
1. Deploy `NoReceive` contract (no `receive()` function) holding rsETH. Approve `LRTWithdrawalManager` to spend rsETH.
2. `NoReceive` calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`. Request stored at nonce N.
3. Operator calls `unlockQueue(ETH_TOKEN, N+1, ...)`. rsETH is burned from the contract; ETH is moved from `LRTUnstakingVault` into `LRTWithdrawalManager`. `unlockedWithdrawalsCount[ETH_TOKEN]` incremented.
4. `NoReceive` calls `completeWithdrawal(ETH_TOKEN, "")`. `_transferAsset` executes `payable(NoReceive).call{value: amount}("")`, which returns `false`. Transaction reverts with `EthTransferFailed`. State fully rolled back.
5. Operator calls `completeWithdrawalForUser(ETH_TOKEN, NoReceive, "")`. Same revert.
6. `NoReceive`'s rsETH is permanently burned. ETH is permanently locked in `LRTWithdrawalManager`. `unlockedWithdrawalsCount[ETH_TOKEN]` remains non-zero, blocking `sweepRemainingAssets(ETH_TOKEN)` indefinitely.
