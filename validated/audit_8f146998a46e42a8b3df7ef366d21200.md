Audit Report

## Title
rsETH Inflation Attack via Unrestricted ETH Donation and Disabled Price Guard Yields Zero rsETH to Victim - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool` exposes a bare `receive()` fallback that lets anyone inflate the contract's ETH balance without minting rsETH. Because `LRTOracle.updateRSETHPrice()` is callable by any address and `pricePercentageLimit` is never initialized (defaults to 0, disabling the price-increase guard), an attacker can push `rsETHPrice` to an arbitrarily large value in one transaction. `getRsETHAmountToMint` uses integer floor division, so a victim's deposit against the inflated price truncates to zero rsETH minted. `_beforeDeposit` does not independently require `rsethAmountToMint > 0`, so the zero-mint deposit is silently accepted and the victim's ETH is permanently absorbed by the attacker's pre-existing position.

## Finding Description

**Root cause chain:**

1. `receive()` at `LRTDepositPool.sol:58` accepts arbitrary ETH with no accounting — no rsETH is minted, no deposit limit is checked, and the balance is immediately reflected in TVL. [1](#0-0) 

2. `getETHDistributionData()` uses `address(this).balance` as `ethLyingInDepositPool`, so donated ETH is immediately included in `totalETHInProtocol`. [2](#0-1) 

3. `updateRSETHPrice()` is `public` with only a `whenNotPaused` guard — any EOA can call it. [3](#0-2) 

4. `initialize()` in `LRTOracle` never sets `pricePercentageLimit`, so it remains at the storage default of `0`. [4](#0-3) 

5. The price-increase guard short-circuits to `false` when `pricePercentageLimit == 0`, so an arbitrarily large price jump is accepted without revert. [5](#0-4) 

6. The new price is `(totalETHInProtocol - fee) / rsethSupply`. With 1 wei rsETH supply and `V` donated wei, the price becomes `(1+V) * 1e18`. [6](#0-5) 

7. `getRsETHAmountToMint` uses plain integer division: `(amount * assetPrice) / rsETHPrice`. With the inflated price, `V / (1+V) = 0` for any realistic `V`. [7](#0-6) 

8. `_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`. When `minRSETHAmountExpected == 0` (the default), `0 < 0` is false and the zero-mint deposit proceeds. [8](#0-7) 

## Impact Explanation

**Critical — Direct theft of user funds.**

The victim deposits `V` wei ETH and receives 0 rsETH. Their ETH is permanently absorbed into the pool and is redeemable only by existing rsETH holders (the attacker). The victim holds no token representing their claim and cannot recover the deposit through any protocol mechanism. This matches the allowed impact: *Direct theft of any user funds, whether at-rest or in-motion.*

## Likelihood Explanation

Two preconditions are required, both of which are realistic defaults:

1. `pricePercentageLimit == 0` — this is the **storage default** because `LRTOracle.initialize()` never assigns it. Unless the admin explicitly calls `setPricePercentageLimit` post-deployment, the guard is permanently absent.
2. The victim passes `minRSETHAmountExpected == 0` — this is the conventional default in scripts, integrations, and front-ends that omit slippage protection.

The attacker requires capital equal to `V` (the donation) plus 1 wei (the seed deposit), and recovers `1 + 2V` upon redemption, netting the victim's entire deposit. The attack is repeatable against every subsequent victim.

## Recommendation

1. **Remove or restrict the bare `receive()` fallback.** Gate ETH receipt to known callers (`receiveFromNodeDelegator`, `receiveFromLRTConverter`, `receiveFromRewardReceiver`). Unaccounted donations are the root enabler.
2. **Require `rsethAmountToMint > 0` inside `_beforeDeposit`**, independent of `minRSETHAmountExpected`. A deposit that mints zero shares must always revert.
3. **Set a non-zero `pricePercentageLimit` in `initialize()`** so that single-block price manipulation is bounded even if the above fixes are not applied.
4. Consider adding a virtual offset (analogous to OZ ERC-4626's `_decimalsOffset`) to the rsETH price calculation to prevent trivial manipulation when supply is near zero.

## Proof of Concept

```

  pricePercentageLimit = 0 (storage default, initialize() never sets it)
  minAmountToDeposit = 0 (storage default)
  rsETHPrice = 1e18 (initial value when rsethSupply == 0)

  rsethAmountToMint = (1 * 1e18) / 1e18 = 1 wei rsETH minted
  Pool ETH balance: 1 wei | rsETH supply: 1 wei

Step 2 — Attacker sends V wei ETH directly to LRTDepositPool (receive() fallback):
  Pool ETH balance: 1+V wei | rsETH supply: 1 wei (unchanged)
  getETHDistributionData() now returns ethLyingInDepositPool = 1+V

Step 3 — Attacker calls LRTOracle.updateRSETHPrice():
  rsethSupply = 1 (non-zero, skips early return)
  totalETHInProtocol = 1+V
  previousTVL = 1 * 1e18 / 1e18 = 1
  rewardAmount = V, protocolFeeInETH = V * feeInBPS / 10000 (small, negligible)
  newRsETHPrice ≈ (1+V) * 1e18 / 1 = (1+V) * 1e18
  isPriceIncreaseOffLimit = (0 > 0) && ... = false  → no revert
  rsETHPrice stored = (1+V) * 1e18

Step 4 — Victim calls depositETH{value: V}(minRSETHAmountExpected=0):
  rsethAmountToMint = (V * 1e18) / ((1+V) * 1e18)
                    = V / (1+V) = 0  ← integer floor division
  0 < 0 is false → no revert
  Victim's V wei ETH enters pool; victim receives 0 rsETH

Step 5 — Attacker calls LRTOracle.updateRSETHPrice() again:
  totalETHInProtocol = 1+2V
  newRsETHPrice ≈ (1+2V) * 1e18
  rsETHPrice stored = (1+2V) * 1e18

Step 6 — Attacker redeems 1 wei rsETH:
  Attacker recovers ≈ 1+2V wei ETH

Net P&L:
  Spent:  1 wei (deposit) + V wei (donation) ≈ V
  Gained: 1+2V wei
  Profit: ≈ V wei  (the victim's entire deposit)
```
