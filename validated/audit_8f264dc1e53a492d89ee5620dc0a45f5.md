Audit Report

## Title
Token Depositor Receives Zero rsETH While Tokens Are Permanently Transferred Due to Rounding Truncation - (File: contracts/pools/RSETHPool.sol)

## Summary
In all L2 pool `deposit(address,uint256,string)` functions, ERC20 tokens are transferred from the user to the pool before `rsETHAmount` is computed. When the deposit amount is small enough that integer division in `viewSwapRsETHAmountAndFee` truncates `rsETHAmount` to zero, the transaction completes successfully, the user's tokens are permanently held by the pool, and the user receives nothing. No zero-output guard exists in any of the four affected contracts.

## Finding Description
In `RSETHPool.sol`, the token deposit function executes in this order:

1. `IERC20(token).safeTransferFrom(msg.sender, address(this), amount)` — tokens leave the user.
2. `(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token)` — output computed.
3. `IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount)` — user receives `rsETHAmount`, which may be 0.

The computation inside `viewSwapRsETHAmountAndFee` is:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

This is Solidity integer division. When `amountAfterFee * tokenToETHRate < rsETHToETHrate`, the result is `0`. The only existing guard is `if (amount == 0) revert InvalidAmount()`, which does not protect against a non-zero input producing a zero output.

The same truncation applies to the ETH deposit path:
```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```
Once `rsETHToETHrate > 1e18` (rsETH has appreciated), a 1-wei ETH deposit yields `rsETHAmount = 0`.

For `RSETHPoolV3` and `RSETHPoolV3ExternalBridge`, the call is `wrsETH.mint(msg.sender, 0)`, which delegates to OpenZeppelin's `_mint`, which does not revert on a zero amount. For `RSETHPool` and `RSETHPoolNoWrapper`, `safeTransfer(..., 0)` also succeeds silently. In all cases the transaction completes without revert.

Affected code confirmed in the repository:
- `RSETHPool.sol` lines 296–302: transfer-before-compute, no zero-guard.
- `RSETHPoolV3.sol` lines 284–290: same pattern.
- `RSETHPoolNoWrapper.sol` lines 262–268: same pattern.

## Impact Explanation
A user depositing a dust amount of a supported token (below the rounding threshold determined by the oracle rates) permanently loses their tokens while receiving 0 rsETH. The contract accepts the tokens, records no corresponding rsETH obligation, and the tokens are subsequently bridged to L1 — beyond the user's reach. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**: the protocol retains the deposited tokens while failing to deliver the promised rsETH output to the depositor.

## Likelihood Explanation
No privileged access or front-running is required. Any unprivileged user calling `deposit(token, amount, referralId)` with a sub-threshold amount triggers the issue. The threshold is determined by oracle rates: with `tokenToETHRate = 1e15` and `rsETHToETHrate = 1.1e18`, any deposit below ~1100 wei of that token yields `rsETHAmount = 0`. For ETH deposits, any single-wei deposit when `rsETHToETHrate > 1e18` (a normal post-launch condition) also yields 0. Users testing integrations, sending dust, or making rounding errors are realistic victims. The condition is repeatable and requires no special setup.

## Recommendation
Add a zero-output guard immediately after computing `rsETHAmount` in every affected deposit function, before any state changes or transfers:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

This mirrors the existing `if (amount == 0) revert InvalidAmount()` guard and ensures tokens are never taken from a user without a corresponding rsETH output. Apply identically to `RSETHPool.sol`, `RSETHPoolV3.sol`, `RSETHPoolNoWrapper.sol`, and `RSETHPoolV3ExternalBridge.sol`.

## Proof of Concept

**Parameters:**
- `rsETHToETHrate = 1.1e18` (rsETH has appreciated 10% — normal over time)
- Supported token with `tokenToETHRate = 1e15` (token worth 0.001 ETH)
- `feeBps = 0`

**Call sequence:**
1. User calls `RSETHPool.deposit(token, 1000, "ref")`.
2. Line 296: `IERC20(token).safeTransferFrom(msg.sender, pool, 1000)` — 1000 wei of token leaves user.
3. Line 298: `viewSwapRsETHAmountAndFee(1000, token)` computes `rsETHAmount = 1000 * 1e15 / 1.1e18 = 0` (integer truncation).
4. Line 302: `IERC20(address(wrsETH)).safeTransfer(msg.sender, 0)` — user receives nothing.
5. Transaction succeeds. User has lost 1000 wei of token with no rsETH minted.

**Foundry fuzz test plan:**
```solidity
function testFuzz_depositZeroRsETH(uint256 amount) public {
    // bound amount to sub-threshold range
    amount = bound(amount, 1, rsETHToETHrate / tokenToETHRate - 1);
    token.mint(user, amount);
    vm.prank(user);
    token.approve(address(pool), amount);
    uint256 balBefore = wrsETH.balanceOf(user);
    vm.prank(user);
    pool.deposit(address(token), amount, "ref");
    assertEq(wrsETH.balanceOf(user), balBefore); // user received 0 rsETH
    assertEq(token.balanceOf(user), 0);           // user lost all tokens
}
```
