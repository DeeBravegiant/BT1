Looking at the Dijkstra era's sub-transaction validation logic, I found a concrete analog to the external report's "scattered/fragmented validation" vulnerability class.

### Title
Silent Bypass of Plutus Script Guard Datum Validation in `validateGuardDatums` — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxow.hs`)

---

### Summary

The `validateGuardDatums` function in the Dijkstra era's `SUBUTXOW` rule contains a `Nothing -> acc` branch that silently skips datum validation for Plutus script guard credentials when the script is not found in the sub-transaction's `scriptsProvided`. Because guard scripts reside in the **top-level** transaction's witnesses — not the sub-transaction's — this branch is reached for every Plutus script guard entry in `requiredTopLevelGuards`, making the datum-presence check a structural no-op for that entire credential class.

---

### Finding Description

`dijkstraSubUtxowTransition` calls `validateGuardDatums` to enforce the invariant stated in its own doc-comment:

> "Plutus script credentials must have a datum, key/native script credentials must not."

The function iterates over `txBody ^. requiredTopLevelGuardsL` and, for each script-hash credential, looks up the script in the sub-transaction's `ScriptsProvided`:

```haskell
-- SubUtxow.hs lines 210–235
validateGuardDatums (ScriptsProvided scripts) txBody =
  failureOnNonEmptySet malformed SubMalformedGuardDatums
  where
    malformed = Map.foldlWithKey' accum mempty (txBody ^. requiredTopLevelGuardsL)
    accum acc cred mbDatum =
      case credScriptHash cred of
        Nothing ->
          case mbDatum of
            SNothing -> acc
            SJust _  -> Set.insert cred acc
        Just scriptHash ->
          case Map.lookup scriptHash scripts of
            Just script
              | isNativeScript script ->
                  case mbDatum of { SNothing -> acc; SJust _ -> Set.insert cred acc }
              | otherwise ->          -- Plutus: datum MUST be SJust
                  case mbDatum of { SJust _ -> acc; SNothing -> Set.insert cred acc }
            Nothing -> acc            -- ← BUG: silently accepts, no datum check
```

The `scriptsProvided` argument is `scriptsProvidedStAnnTx stAnnTx` — the **sub-transaction's** script witnesses and reference-script inputs. The `requiredTopLevelGuards` field, however, declares credentials whose scripts live in the **top-level** transaction's witnesses (that is the entire point of the field: it names guards the sub-tx demands the top-level tx carry). Consequently `Map.lookup scriptHash scripts` returns `Nothing` for every Plutus script guard, and the `Nothing -> acc` branch is taken unconditionally, leaving `malformed` empty and the check passing regardless of whether `mbDatum` is `SJust` or `SNothing`.

The validation is split across two separate rules:

* **`UTXOW` (top-level)** — `dijkstraUtxowTransition` checks only that required guard credentials are *present* in the top-level guards set (`MissingRequiredGuards`), and does so with the weaker `runTestOnSignal`.
* **`SUBUTXOW` (sub-tx)** — `dijkstraSubUtxowTransition` is supposed to validate datum consistency via `validateGuardDatums`, but the lookup always misses, so the check never fires for Plutus script guards.

This is the direct analog of the external report's pattern: validation requirements are scattered across multiple rules, and the sub-rule's check contains an obscure silent-accept branch that renders it ineffective. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

An unprivileged transaction author can submit a sub-transaction whose `requiredTopLevelGuards` map contains a Plutus script credential paired with `SNothing` (no datum). The `validateGuardDatums` check passes unconditionally because the script is absent from the sub-transaction's `scriptsProvided`. The ledger therefore accepts a structurally malformed guard specification that violates the stated invariant.

Downstream, when the guard mechanism is used to gate governance actions, treasury operations, or certificate authorizations in the Dijkstra era, a Plutus guard script that is supposed to receive a datum (e.g., carrying authorization parameters) can be invoked without one. Depending on the script's logic this either causes an unexpected phase-2 failure (disrupting honest use) or, for scripts that do not inspect the datum, silently succeeds — allowing the guarded action to proceed without the intended datum-encoded authorization check. Either outcome represents a transaction or governance action that bypasses intended validation limits.

The allowed impact class is: **Medium — attacker-controlled transactions exceed intended validation limits or modify governance/treasury/withdrawal behavior outside design parameters.** [4](#0-3) 

---

### Likelihood Explanation

Likelihood is **high** for the structural bypass itself: the `Nothing` branch is reached for *every* Plutus script guard in `requiredTopLevelGuards` because those scripts are never in the sub-transaction's `scriptsProvided` by design. Any sub-transaction author who includes a Plutus script guard can trivially omit the datum and pass validation. The practical security consequence scales with how widely the Dijkstra guard mechanism is adopted for high-value operations. [5](#0-4) 

---

### Recommendation

The `Nothing -> acc` branch should not silently accept. Two correct options:

1. **Conservative (treat unknown script as Plutus):** require a datum when the script type cannot be determined:
   ```haskell
   Nothing ->
     case mbDatum of
       SJust _  -> acc
       SNothing -> Set.insert cred acc
   ```

2. **Explicit error:** flag the credential as malformed when its script is not resolvable at validation time, forcing the submitter to ensure the script is reachable.

Additionally, `validateGuardDatums` should be passed a combined `ScriptsProvided` that includes scripts from both the sub-transaction *and* the top-level transaction, so that the type of every guard credential can actually be determined. [4](#0-3) 

---

### Proof of Concept

1. Deploy a Plutus script `S` as a guard credential `cred = ScriptHashObj (hashScript S)`.
2. Construct a sub-transaction `subTx` with:
   - `requiredTopLevelGuards = Map.singleton cred SNothing`  ← datum intentionally absent
   - No witness for `S` in `subTx`'s witness set (script lives only in the top-level tx).
3. Wrap `subTx` in a top-level transaction that includes `cred` in `guardsTxBodyL` and provides `S` in its own witnesses.
4. Submit. `dijkstraSubUtxowTransition` calls `validateGuardDatums (scriptsProvidedStAnnTx subStAnnTx) subTxBody`. `Map.lookup (hashScript S) subScripts` returns `Nothing`; the `Nothing -> acc` branch fires; `malformed` stays empty; `SubMalformedGuardDatums` is never raised.
5. The transaction is accepted despite the Plutus guard having no datum, violating the invariant documented in the function's own comment. [6](#0-5)

  , EraRule "SUBUTXOW" era ~ SUBUTXOW era

  TransitionRule (EraRule "SUBUTXOW" era)
dijkstraSubUtxowTransition = do
  TRC (env@(SubUtxoEnv _ pp certState originalUtxo _), utxoState, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG
      txBody = tx ^. bodyTxL
      witsKeyHashes = keyHashWitnessesTxWits (tx ^. witsTxL)
      scriptsProvided = scriptsProvidedStAnnTx stAnnTx

  {- ∀[ (vk , σ) ∈ vKeySigs ] isSigned vk (txidBytes txId) σ -}

  {- vKeyHashesNeeded ⊆ vKeyHashesProvided -}
  runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody

  {- dataHashesNeeded ⊆ mapˢ hash dataProvided -}
  runTest $ Alonzo.missingRequiredDatums scriptsProvided originalUtxo tx

  {- txADhash ≡ map hash txAuxData -}
  runTestOnSignal $ Shelley.validateMetadata pp tx

  let scriptIntegrity = mkScriptIntegrity pp tx (plutusLanguagesUsedStAnnTx stAnnTx)
  runTest $ Alonzo.checkScriptIntegrityHash tx pp scriptIntegrity

    Babbage.validateScriptsWellFormedTxOuts
