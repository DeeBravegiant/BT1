### Title
P2P Sync Client Stores Peer-Supplied `transaction_hash` Without Validation, Corrupting RPC-Served Transaction Hash Data — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

The `TransactionStreamFactory::parse_data_for_block` implementation accepts `FullTransaction` messages from unauthenticated P2P peers and pushes the peer-supplied `transaction_hash` field directly into `BlockBody::transaction_hashes` without verifying it matches the hash of the accompanying `Transaction`. This unvalidated hash is then committed to storage via `append_body`, causing `starknet_getTransactionByHash` and `starknet_getBlockWithTxHashes` to serve authoritative-looking wrong transaction hashes.

---

### Finding Description

In `parse_data_for_block`, the code destructures the peer-supplied `FullTransaction` and pushes `transaction_hash` directly into the block body: [1](#0-0) 

The developer-inserted TODO on line 88 explicitly acknowledges the missing check:

```
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
```

The only preconditions the code enforces are:
1. `block_number` is below the stored header marker (`BlockNumberLimit::HeaderMarker`)
2. The number of received transactions equals `header.n_transactions` [2](#0-1) 

Neither condition involves verifying that `transaction_hash == hash(transaction)`. A peer that satisfies the count check passes through without any hash integrity check.

The resulting `BlockBody` is then committed unconditionally: [3](#0-2) 

Inside `write_transactions`, the peer-supplied hash is stored verbatim in both the `transaction_hash_to_idx` index and the `TransactionMetadata` record: [4](#0-3) 

There is no recomputation or cross-check of the hash at the storage layer.

---

### Impact Explanation

After a malicious peer injects a `FullTransaction` with `transaction_hash = X` (where `X ≠ hash(transaction)`):

- `get_transaction_idx_by_hash(real_hash)` returns `None` — the real hash is not indexed.
- `get_transaction_idx_by_hash(X)` returns the transaction index — the injected hash is indexed.
- `starknet_getTransactionByHash(real_hash)` returns `TRANSACTION_HASH_NOT_FOUND`.
- `starknet_getBlockWithTxHashes` returns `X` as the authoritative hash for that transaction.
- `starknet_getBlockWithFullTransactions` pairs the correct transaction body with the wrong hash `X`.

The RPC layer reads hashes directly from storage without revalidation: [5](#0-4) 

This matches the allowed impact: **High — RPC returns an authoritative-looking wrong value**.

---

### Likelihood Explanation

Any node participating in the P2P sync network can serve `FullTransaction` responses. The sync client connects to peers and trusts their `transaction_hash` field unconditionally. The peer is only reported (`report_peer()`) for structural violations (wrong count, wrong ordering, missing `Fin`), not for hash content mismatch. A malicious peer that satisfies the count constraint will never be reported and its corrupted data will be committed. [6](#0-5) 

---

### Recommendation

After destructuring `FullTransaction`, recompute the transaction hash using the canonical hash function and compare it against the peer-supplied value. If they differ, treat it as a `BadPeerError` and call `report_peer()`. The fix belongs at line 88–89 of `transaction.rs`, before the push into `block_body.transaction_hashes`.

---

### Proof of Concept

Using the existing `ClientResponsesManager` mock harness in `test_utils.rs`:

1. Store a block header for `BlockNumber(0)` with `n_transactions = 1`.
2. Construct a valid `Transaction` (e.g., an invoke tx) and compute its real hash `H_real`.
3. Build a `FullTransaction` with `transaction_hash = H_fake` (any value ≠ `H_real`).
4. Feed it through `TransactionStreamFactory::parse_data_for_block`.
5. Call `write_to_storage` on the result.
6. Assert `get_block_transaction_hashes(BlockNumber(0)) == [H_fake]` — the injected wrong hash is stored.
7. Assert `get_transaction_idx_by_hash(&H_real) == None` — the real hash is not findable.
