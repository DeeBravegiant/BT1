### Title
Unbounded Pool Singleton Chain Traversal Allows Peer-Injected Pool Membership State Corruption — (`chia/wallet/wallet_state_manager.py`)

### Summary

An unprivileged peer can cause the pool wallet singleton chain traversal loop in `WalletStateManager._add_coin_states` to iterate an unbounded number of times by returning an infinite chain of spent `CoinState` objects. Because `apply_state_transition` performs no cryptographic validation of the supplied `CoinSpend`, and because `compute_additions` runs the CLVM without checking signatures or cross-coin announcements, the attacker can inject fabricated pool state transitions — including arbitrary pool URLs and target puzzle hashes — into the wallet's persistent pool store and pool config.

### Finding Description

**The unbounded loop** — `_add_coin_states` enters the following loop when processing a spent `POOLING_WALLET` singleton coin:

```python
while curr_coin_state.spent_height is not None:
    cs = await fetch_coin_spend_for_coin_state(curr_coin_state, peer)
    success = await singleton_wallet.apply_state_transition(cs, ...)
    if not success:
        break
    new_singleton_coin = get_most_recent_singleton_coin_from_coin_spend(cs)
    if new_singleton_coin is None:
        break
    ...
    new_coin_state = await self.wallet_node.get_coin_state([new_singleton_name], peer=peer, ...)
    curr_coin_state = new_coin_state[0]   # peer-controlled
```

There is no iteration cap. The loop continues as long as the peer returns a `CoinState` with `spent_height is not None`. [1](#0-0) 

**Peer controls both inputs to each iteration:**

1. `fetch_coin_spend_for_coin_state` validates only that the returned puzzle hashes to `coin.puzzle_hash`; the solution is completely attacker-controlled. [2](#0-1) 

2. `get_coin_state([new_singleton_name], peer=peer)` is a direct peer query; the peer can return any `CoinState` with `spent_height != None`. [3](#0-2) 

**`apply_state_transition` performs no cryptographic validation.** The only guard is a coin-name equality check against the current tip:

```python
if spent_coin_name != new_state.coin.name():
    return False
```

No signature verification, no puzzle-hash validation against expected pool puzzle templates, no CLVM execution. If the name matches, the spend is unconditionally written to `pool_store`. [4](#0-3) 

**`compute_additions` runs the CLVM but ignores AGG_SIG and cross-coin conditions.** It extracts only `CREATE_COIN` conditions; `ASSERT_COIN_ANNOUNCEMENT` and `AGG_SIG_ME` are silently skipped:

```python
if op in {ConditionOpcode.AGG_SIG_ME, ...}:
    cost += ConditionCost.AGG_SIG.value
    continue
if op != ConditionOpcode.CREATE_COIN.value:
    continue
``` [5](#0-4) 

**`WalletPoolStore.add_spend` only checks the parent-coin-info chain link**, not puzzle validity or signatures:

```python
if spend.coin.parent_coin_info != prev.coin.name():
    raise ValueError("New spend does not extend")
``` [6](#0-5) 

### Impact Explanation

**Attack chain:**

1. Attacker connects as a full-node peer to the victim wallet.
2. Victim wallet processes a spent `POOLING_WALLET` singleton coin and enters the traversal loop.
3. For each iteration the attacker:
   - Returns the correct puzzle (known from the coin's `puzzle_hash`) paired with a crafted absorb solution containing a valid lineage proof (computable from on-chain data) and arbitrary `pool_reward_amount`/`pool_reward_height`. The CLVM runs successfully and emits a `CREATE_COIN` condition for a child coin with an odd amount. No signature is required for absorb-type solutions.
   - Returns a `CoinState` for the child coin with `spent_height != None`.
4. Each iteration calls `apply_state_transition` → `pool_store.add_spend`, persisting the fabricated spend. `update_pool_config` is then called, overwriting the on-disk pool config with state derived from the attacker's solution (pool URL, `target_puzzle_hash`, `owner_pubkey`).
5. For travel-type fabricated spends (no signature checked by `compute_additions`), the attacker can embed an arbitrary `PoolState` (via `solution_to_pool_state`) including a malicious `pool_url` and `target_puzzle_hash`.

**Concrete consequences:**
- Pool membership state in `pool_state_transitions` DB table is corrupted with attacker-fabricated intermediate spends.
- Pool config file is overwritten with attacker-chosen `pool_url` (farmer submits partials to attacker's pool) and `target_puzzle_hash` (pool pays rewards to attacker-chosen address).
- The wallet's view of the singleton tip diverges from the real chain, breaking future legitimate state transitions.
- The loop itself holds the DB writer lock for an unbounded duration, blocking all other wallet DB operations.

### Likelihood Explanation

Any peer the wallet connects to can trigger this. The wallet connects to untrusted full nodes by default. The attacker needs only to:
- Know the singleton's puzzle (derivable from the launcher ID and pool parameters, all public on-chain).
- Compute the lineage proof for each hop (trivial from on-chain data).
- Serve crafted `RespondPuzzleSolution` and `RespondToCoinState` messages.

No leaked keys, no admin access, no broken cryptography required.

### Recommendation

1. **Add an iteration cap** to the traversal loop (e.g., `MAX_SINGLETON_HOPS = 100`).
2. **Validate the coin spend cryptographically** inside `apply_state_transition`: run the CLVM via `compute_additions` and verify the resulting child coin's puzzle hash matches an expected pool singleton template for the known `launcher_id`.
3. **Verify the puzzle is a valid pool singleton** before accepting it: check that `puzzle_reveal` uncurries to the expected singleton top-layer mod with the correct `launcher_id`.
4. **Cross-check the returned `CoinState`** against locally known coin records before trusting `spent_height`.

    async def add_spend(
        self,
        wallet_id: int,
        spend: CoinSpend,
        height: uint32,
    ) -> None:
        """
        Appends (or replaces) entries in the DB. The new list must be at least as long as the existing list, and the
        parent of the first spend must already be present in the DB. Note that this is not committed to the DB
        until db_wrapper.commit() is called. However it is written to the cache, so it can be fetched with
        get_all_state_transitions.
        """
        async with self.db_wrapper.writer_maybe_transaction() as conn:
            # find the most recent transition in wallet_id
            rows = list(
                await conn.execute_fetchall(
                    "SELECT transition_index, height, coin_spend "
                    "FROM pool_state_transitions "
                    "WHERE wallet_id=? "
                    "ORDER BY transition_index DESC "
                    "LIMIT 1",
                    (wallet_id,),
                )
            )
            serialized_spend = bytes(spend)
            if len(rows) == 0:
                transition_index = 0
            else:
                existing = list(
                    await conn.execute_fetchall(
                        "SELECT COUNT(*) FROM pool_state_transitions WHERE wallet_id=? AND height=? AND coin_spend=?",
                        (wallet_id, height, serialized_spend),
                    )
                )
                if existing[0][0] != 0:
                    # we already have this transition in the DB
                    return

                row = rows[0]
                if height < row[1]:
                    raise ValueError("Height cannot go down")
                prev = CoinSpend.from_bytes(row[2])
                if spend.coin.parent_coin_info != prev.coin.name():
                    raise ValueError("New spend does not extend")
                transition_index = row[0]

            cursor = await conn.execute(
                "INSERT OR IGNORE INTO pool_state_transitions VALUES (?, ?, ?, ?)",
                (
                    transition_index + 1,
                    wallet_id,
                    height,
                    serialized_spend,
                ),
            )
            await cursor.close()
