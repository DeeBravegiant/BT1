### Title
Missing `derivation_path` Parameter in `VerifyForeignTransactionRequestArgs` Causes All Foreign-Tx Verification Requests to Sign Under a Hardcoded Zero Tweak - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary

The `VerifyForeignTransactionRequestArgs` struct is missing the `derivation_path` field that the design specification requires. As a direct consequence, `VerifyForeignTransactionRequest` is also missing the `tweak` field, and the MPC node's signing path hardcodes `tweak: Tweak::new([0u8; 32])`. Every `verify_foreign_transaction` request — regardless of caller account or intended derivation path — is signed under the same zero-tweak derived key, breaking per-caller key isolation and the design-mandated domain separation between `sign()` and `verify_foreign_transaction()`.

### Finding Description

The design document (`docs/foreign-chain-transactions.md`) specifies the following structs:

```rust
pub struct VerifyForeignTransactionRequestArgs {
    pub request: ForeignChainRpcRequest,
    pub derivation_path: String,   // <-- required by design
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}

pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub tweak: Tweak,              // <-- required by design
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The actual implementation in `crates/near-mpc-contract-interface/src/types/foreign_chain.rs` omits both fields:

```rust
pub struct VerifyForeignTransactionRequestArgs {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    // derivation_path is ABSENT
}

pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    // tweak is ABSENT
}
``` [1](#0-0) 

Because `tweak` is absent from `VerifyForeignTransactionRequest`, the MPC node's `build_signature_request` function in `crates/node/src/providers/verify_foreign_tx/sign.rs` has no tweak to read and hardcodes it to all-zero bytes:

```rust
Ok(SignatureRequest {
    id: request.id,
    receipt_id: request.receipt_id,
    payload: Payload::Ecdsa(payload_bytes),
    tweak: Tweak::new([0u8; 32]),   // <-- hardcoded zero for every request
    entropy: request.entropy,
    timestamp_nanosec: request.timestamp_nanosec,
    domain: request.domain_id,
})
``` [2](#0-1) 

For comparison, the `sign()` path correctly derives the tweak from `(predecessor_id, path)`: [3](#0-2) 

The design document also specifies a **separate tweak-derivation prefix** for foreign-tx requests (`"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`) to prevent cross-purpose key reuse. Neither the prefix nor the `derive_foreign_tx_tweak` function exists in the production `kdf.rs`: [4](#0-3) 

The design intent is explicit:

> "The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks)."
> "`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same `(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes." [5](#0-4) 

### Impact Explanation

**All `verify_foreign_transaction` requests are signed under the same key** — the key derived from the domain root with a zero tweak — regardless of which NEAR account submitted the request or what derivation path they intended to use.

Concrete consequences:

1. **Per-caller key isolation is entirely absent.** Bridge contract A and bridge contract B both receive signatures produced under the identical derived key. Any bridge application that calls `derived_public_key` with its own account and path to obtain the expected verification key will compute a *different* key than the one actually used to sign, making on-chain signature verification fail or, if the bridge uses the root key directly, silently accept signatures intended for a different bridge.

2. **Domain separation between `sign()` and `verify_foreign_transaction()` is not implemented.** The design requires a distinct prefix so that a `sign()` key and a `verify_foreign_transaction()` key for the same `(account, path)` are always different. Because the foreign-tx tweak is always zero, any `sign()` request that happens to produce a zero tweak (negligible probability, but structurally the separation is absent) would share the key. More practically, the zero-tweak key is the same for every foreign-tx caller, so a signature produced for one caller's foreign-chain observation is valid under the same public key as any other caller's observation.

3. **Cross-bridge replay is enabled when payloads collide.** If two different bridge integrations submit `verify_foreign_transaction` for the same foreign-chain transaction (same `ForeignTxSignPayload`), the MPC network produces the same signature for both, because the key is identical and the payload hash is identical. A malicious actor who obtains the signature for one bridge's request can replay it against the other bridge's verification logic without triggering a new MPC signing round.

**Severity: High** — matches "Cross-chain replay, forged foreign-chain verification, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."

### Likelihood Explanation

Any unprivileged caller can submit a `verify_foreign_transaction` request. The missing field is structural — it is absent from the on-chain ABI and the node binary — so the zero-tweak behaviour is triggered by every single request without any special attacker action. A bridge developer who reads the design document and implements key-isolation logic based on `derivation_path` will find their isolation silently broken in production.

### Recommendation

1. **Add `derivation_path: String` to `VerifyForeignTransactionRequestArgs`** and `tweak: Tweak` to `VerifyForeignTransactionRequest`, mirroring the pattern used by `SignRequestArgs` / `SignatureRequest`.

2. **Implement `derive_foreign_tx_tweak`** in `crates/near-mpc-crypto-types/src/kdf.rs` using the design-specified prefix `"near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:"`, and call it in `args_into_verify_foreign_tx_request` (analogous to how `SignatureRequest::new` calls `derive_tweak`).

3. **Replace the hardcoded `Tweak::new([0u8; 32])` in `build_signature_request`** with the tweak read from `VerifyForeignTxRequest`.

4. **Add an ABI snapshot test** that asserts `VerifyForeignTransactionRequestArgs` contains `derivation_path`, preventing silent regression.

### Proof of Concept

**Step 1 — Submit two `verify_foreign_transaction` requests from different accounts for the same foreign-chain tx:**

```json
// Caller: alice.near
{
  "request": {
    "request": { "Bitcoin": { "tx_id": "aabbcc...", "confirmations": 1, "extractors": ["BlockHash"] } },
    "domain_id": 3,
    "payload_version": 1
  }
}

// Caller: bob.near  (identical request body)
{
  "request": {
    "request": { "Bitcoin": { "tx_id": "aabbcc...", "confirmations": 1, "extractors": ["BlockHash"] } },
    "domain_id": 3,
    "payload_version": 1
  }
}
```

**Step 2 — Observe that both requests produce the same `VerifyForeignTransactionRequest` key** (no tweak field, no predecessor binding), confirmed by the test at `crates/contract/src/lib.rs:3208-3263` which shows `args_into_verify_foreign_tx_request` produces a caller-agnostic key: [6](#0-5) 

**Step 3 — Observe that `build_signature_request` in the node uses `Tweak::new([0u8; 32])` for both**, producing signatures under the same derived key: [7](#0-6) 

**Step 4 — The signature returned for alice's request is cryptographically identical to the one returned for bob's request** (same key, same payload hash), enabling replay of alice's MPC-signed observation into any bridge contract that accepts bob's key — which is the same key.

```markdown
## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
```
