Audit Report

## Title
Block Stuffing Enables Stale-Rate Over-Issuance of rsETH from Pre-Funded Pool Balance — (`contracts/pools/oracle/InterimRSETHOracle.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`)

## Summary

`InterimRSETHOracle` stores the rsETH/ETH rate as a plain storage variable updated only by a privileged `MANAGER_ROLE` call, with no on-chain staleness guard. `RSETHPoolNoWrapper.deposit()` computes rsETH payout directly from this rate and has no daily mint cap, unlike `RSETHPoolV2`/`RSETHPoolV3`. An attacker can stuff Unichain blocks to delay a pending `setRate()` update, then repeatedly deposit ETH at the stale lower rate, receiving more rsETH from the pool's pre-funded balance than the deposited ETH justifies at the correct rate.

## Finding Description

**Root cause — no staleness protection in `InterimRSETHOracle`:**

`getRate()` returns the raw storage variable `rate` with no timestamp or heartbeat check. [1](#0-0) 

`rate` is only updated when `MANAGER_ROLE` calls `setRate()` / `_setRate()`. [2](#0-1) 

**Root cause — no daily mint cap in `RSETHPoolNoWrapper`:**

`deposit()` computes payout via `viewSwapRsETHAmountAndFee()` and immediately transfers rsETH with no rate-staleness check and no mint-volume guard. [3](#0-2) 

The payout formula `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` means a lower (stale) rate produces a larger rsETH payout. [4](#0-3) 

**Contrast with `RSETHPoolV2`**, which enforces `limitDailyMint` on every deposit: [5](#0-4) [6](#0-5) 

`RSETHPoolNoWrapper` has no equivalent modifier, no `dailyMintLimit` storage variable, and no `startTimestamp` guard.

**Exploit flow:**
1. Attacker monitors Unichain mempool for a pending `InterimRSETHOracle.setRate(newRate)` where `newRate > currentRate`.
2. Attacker submits high-gas filler transactions to saturate Unichain blocks, preventing `setRate()` from landing (block stuffing).
3. While `rate` remains at `R_old < R_correct`, attacker calls `RSETHPoolNoWrapper.deposit{value: X}()` repeatedly.
4. Each call yields `X * 1e18 / R_old` rsETH instead of the correct `X * 1e18 / R_correct`, drawing excess rsETH from the pool's pre-funded OFT balance.
5. With no daily cap, the attacker repeats until the pool's rsETH balance is exhausted.

**Existing guards are insufficient:** `nonReentrant` and `whenNotPaused` do not constrain the rate used or the volume minted per time period. [7](#0-6) 

## Impact Explanation

The pool's pre-funded rsETH OFT balance is drained faster than the ETH deposited justifies at the correct rate. The protocol receives ETH at face value but has already disbursed excess rsETH it cannot recover, violating the pool's reserve invariant. This maps to **Low — Block stuffing** and **Low — Contract fails to deliver promised returns** per the allowed impact scope.

## Likelihood Explanation

Unichain is an OP Stack L2 with low and predictable gas costs, making block stuffing economically feasible when the rate differential is sufficient. `InterimRSETHOracle` is explicitly an interim manual oracle, so rate updates are infrequent and predictable, widening the exploitable window. No privileged access is required; any EOA with ETH capital can execute the attack. The absence of a daily mint cap means the attacker is bounded only by the pool's rsETH balance and their own ETH.

## Recommendation

1. **Add a daily mint limit** to `RSETHPoolNoWrapper` analogous to the `limitDailyMint` modifier in `RSETHPoolV2`. [5](#0-4) 
2. **Add a rate-staleness check** in `InterimRSETHOracle`: record `block.timestamp` on each `_setRate()` call and expose it; revert in `RSETHPoolNoWrapper.deposit()` if the rate age exceeds a configurable threshold (e.g., 1 hour).
3. **Long-term**: replace `InterimRSETHOracle` with a push-based oracle that includes a heartbeat/staleness guard.

## Proof of Concept

```solidity
// Foundry fork test on Unichain local fork
function testBlockStuffingStaleRateDrain() public {
    // Setup: InterimRSETHOracle deployed with stale rate
    uint256 staleRate   = 1.05e18;
    uint256 correctRate = 1.06e18;
    // Manager's setRate(correctRate) tx is stuck due to block stuffing
    // (simulated by NOT calling setRate in the test)

    uint256 depositAmt = 10 ether;
    uint256 iterations = 50;
    uint256 totalRsETHReceived;

    for (uint256 i = 0; i < iterations; i++) {
        vm.deal(attacker, depositAmt);
        vm.prank(attacker);
        pool.deposit{value: depositAmt}("ref");
        totalRsETHReceived += rsETH.balanceOf(attacker);
    }

    uint256 totalETH = depositAmt * iterations;
    uint256 correctRsETH = totalETH * 1e18 / correctRate;
    // Attacker received more rsETH than correct rate justifies
    assertGt(totalRsETHReceived, correctRsETH);
    // Excess = totalRsETHReceived - correctRsETH, drained from pool's pre-funded balance
}
```
