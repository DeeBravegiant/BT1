### Title
Missing Hardened-Index Guard in `non_hard_derive` Allows BIP32 Path Semantic Mismatch — (`src/cbmpc/protocol/hd_tree_bip32.cpp`)

### Summary

`non_hard_derive` accepts `bip32_path_t` entries with indices ≥ `0x80000000` (hardened range) without any validation. The public API `derive_ecdsa_2p_keys` passes caller-supplied `non_hardened_paths` through to this function with only a duplicate check. The result is that a caller can obtain a SUCCESS return and a valid-looking ECDSA-2PC key blob derived using a hardened index value but via the non-hardened (public-key-based) HMAC method — a key that matches neither standard BIP32 non-hardened nor standard BIP32 hardened derivation for that index.

### Finding Description

**Call chain:**

```
derive_ecdsa_2p_keys (public API)
  → validate_no_duplicate_bip32_paths   ← only duplicate check, no range check
  → to_internal_bip32_path              ← verbatim copy, no range check
  → key_share_ecdsa_hdmpc_2p_t::derive_keys
      → non_hard_derive(Q_derived, chain_code, non_hardened_paths)
          → for each index: HMAC-SHA512(chain_code, Q_temp || index)
            ← index used raw, no guard against index ≥ 0x80000000
```

In `non_hard_derive`, the loop at line 18–26 reads each `uint32_t index = path[j]` and feeds it directly into the HMAC with no check that `index < 0x80000000`: [1](#0-0) 

The only validation applied to `non_hardened_paths` at the API layer is `validate_no_duplicate_bip32_paths`: [2](#0-1) 

Which only checks for duplicates: [3](#0-2) 

`to_internal_bip32_path` copies indices verbatim with no range enforcement: [4](#0-3) 

### Impact Explanation

BIP32 specifies two distinct derivation methods distinguished by the index range:
- **Non-hardened** (`i < 0x80000000`): HMAC input uses the *public* key.
- **Hardened** (`i ≥ 0x80000000`): HMAC input uses the *private* key.

When a caller passes `index = 0x80000001` in `non_hardened_paths`, `non_hard_derive` computes `HMAC(chain_code, Q_temp || 0x80000001)` — using the public key with a hardened index. This produces a key that:

1. Does **not** match standard BIP32 non-hardened derivation at `0x80000001` (wrong index range).
2. Does **not** match standard BIP32 hardened derivation at `0x80000001` (wrong HMAC input — uses public key instead of private key).
3. Is returned as `SUCCESS` with a fully serialized ECDSA-2PC key blob.

Any application that relies on BIP32 path semantics for wallet compatibility, key isolation, or path-based access control will silently receive a key that corresponds to no standard derivation path. If the application uses the "non-hardened" label to make security decisions (e.g., "it is safe to share the chain code for this branch"), the mismatch undermines those assumptions. [5](#0-4) 

### Likelihood Explanation

The public API `derive_ecdsa_2p_keys` is directly callable by any API consumer. The `non_hardened_paths` parameter is a plain `std::vector<bip32_path_t>` with no documented restriction on index values in the public header: [6](#0-5) 

No runtime guard exists anywhere in the path from the public API to `non_hard_derive`. The bug is trivially triggerable by any caller, intentionally or by mistake.

### Recommendation

Add an explicit range check in `non_hard_derive` (or at the API boundary in `derive_ecdsa_2p_keys`) that rejects any index ≥ `0x80000000` in the `non_hardened_paths` argument, returning an error such as `E_BADARG`. The check belongs at the API boundary so it is enforced before any protocol communication occurs:

```cpp
// In derive_ecdsa_2p_keys, after validate_no_duplicate_bip32_paths:
for (const auto& path : non_hardened_paths) {
  for (const uint32_t idx : path.indices) {
    if (idx >= 0x80000000u)
      return coinbase::error(E_BADARG, "non_hardened_paths must not contain hardened indices");
  }
}
```

A symmetric guard should also be added to `non_hard_derive` itself as a defense-in-depth assertion.

```text
inline error_t validate_no_duplicate_bip32_paths(const std::vector<PublicPathT>& paths) {
  // O(n^2) is fine here: typical derivation batches are small (caller-controlled).
  for (size_t i = 0; i < paths.size(); i++) {
    for (size_t j = i + 1; j < paths.size(); j++) {
      if (paths[i].indices == paths[j].indices) return coinbase::error(E_BADARG, "duplicate non_hardened_paths");
    }
  }
  return SUCCESS;
```
