Audit Report

## Title
Uninitialized `tokenFeeBps` Mapping Causes Zero Fee on All Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary
`RSETHPool.sol` maintains two separate fee paths: ETH deposits use the global `feeBps` (set at initialization), while ERC-20 token deposits use `tokenFeeBps[token]`. Because `addSupportedToken` never writes to `tokenFeeBps[token]`, every newly added token defaults to a fee rate of zero. Any unprivileged depositor can call `deposit(token, amount, referralId)` and pay zero fee, causing the protocol to collect no fee revenue from the token deposit path until an admin separately calls `setTokenFeeBps`.

## Finding Description
The ETH deposit path in `viewSwapRsETHAmountAndFee(uint256 amount)` computes `fee = amount * feeBps / 10_000` where `feeBps` is set at initialization and is non-zero in production. The token deposit path in `viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes `fee = amount * feeBpsForToken / 10_000` where `feeBpsForToken = tokenFeeBps[token]`. The `tokenFeeBps` mapping is declared at L88 but `addSupportedToken` (L637–656) only writes `supportedTokenOracle[token]` and `tokenBridge[token]`; it never writes `tokenFeeBps[token]`. A separate setter `setTokenFeeBps` (L583–594) exists but is not called atomically during token registration. The result is that `tokenFeeBps[token] == 0` for every token from the moment it is added until an admin separately intervenes. The public `deposit(address token, uint256 amount, string referralId)` function (L284–305) calls `viewSwapRsETHAmountAndFee(amount, token)`, accumulates `feeEarnedInToken[token] += fee` where `fee == 0`, and transfers the full rsETH amount to the depositor. No existing check prevents this; the `onlySupportedToken` modifier only verifies the oracle is set, not that a fee has been configured.

Note: The claim that `RSETHPoolNoWrapper`, `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge` share the same defect is inaccurate — those contracts do not use a `tokenFeeBps` mapping at all; their token deposit paths use the global `feeBps`, which is initialized and non-zero.

## Impact Explanation
Every token deposit processed while `tokenFeeBps[token] == 0` yields `feeEarnedInToken[token] += 0`. The protocol treasury receives zero fee revenue from the token deposit path for the entire window between token addition and an admin's manual `setTokenFeeBps` call. This constitutes permanent loss of unclaimed yield for the protocol on all token volume processed in that window, matching the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation
The zero-fee state is the Solidity default and requires no attacker action to establish. Any unprivileged user who calls `deposit(token, amount, referralId)` while the default persists pays zero fee. The window is open from the moment `addSupportedToken` is called until an admin separately calls `setTokenFeeBps`, a step that is neither enforced nor prompted by the contract. No special privileges, victim mistakes, or external conditions are required. The path is fully reachable by any external caller.

## Recommendation
Add a `_feeBps` parameter to `addSupportedToken` and write it atomically during token registration:

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    // ... existing checks ...
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    tokenFeeBps[token] = _feeBps;   // initialize atomically
    emit AddSupportedToken(token, oracle, bridge);
}
```

This mirrors how `feeBps` is set for ETH at initialization and eliminates the inconsistency between the two deposit paths.

## Proof of Concept

1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0` (Solidity default).
2. Any user calls `deposit(wstETH, 10 ether, "ref")`.
3. `viewSwapRsETHAmountAndFee(10 ether, wstETH)` executes: `feeBpsForToken = tokenFeeBps[wstETH] = 0`, so `fee = 10 ether * 0 / 10_000 = 0`.
4. `feeEarnedInToken[wstETH] += 0` — protocol collects nothing.
5. The same user depositing `10 ether` of native ETH via `deposit("ref")` pays `10 ether * feeBps / 10_000` (e.g., 5 bps → 0.005 ETH).
6. Repeat step 2 indefinitely until admin calls `setTokenFeeBps`; all token volume in this window is fee-free.

Foundry test sketch:
```solidity
function test_zeroFeeOnTokenDeposit() public {
    // admin adds token without calling setTokenFeeBps
    vm.prank(timelockAdmin);
    pool.addSupportedToken(address(wstETH), oracle, bridge);

    // assert tokenFeeBps is 0
    assertEq(pool.tokenFeeBps(address(wstETH)), 0);

    // user deposits
    uint256 amount = 10 ether;
    deal(address(wstETH), user, amount);
    vm.startPrank(user);
    wstETH.approve(address(pool), amount);
    pool.deposit(address(wstETH), amount, "ref");
    vm.stopPrank();

    // fee collected is zero
    assertEq(pool.feeEarnedInToken(address(wstETH)), 0);
}
```
