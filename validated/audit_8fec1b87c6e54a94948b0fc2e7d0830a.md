Audit Report

## Title
Uninitialized `maxMintAmountPerDay` Causes `RSETH.mint()` to Always Revert, Blocking All Deposits — (File: contracts/RSETH.sol)

## Summary
`RSETH.initialize()` and `reinitialize()` never set `maxMintAmountPerDay`, leaving it at its Solidity default of `0`. The `checkDailyMintLimit` modifier unconditionally evaluates `currentPeriodMintedAmount + amount > maxMintAmountPerDay`, which reduces to `amount > 0` when `maxMintAmountPerDay == 0`, causing every non-zero `mint()` call to revert with `DailyMintLimitExceeded`. This blocks all user deposits via `LRTDepositPool.depositETH()` and `depositAsset()` until an admin separately calls `setMaxMintAmountPerDay`.

## Finding Description
`RSETH.initialize()` sets only `lrtConfig` and ERC20/Pausable state; `reinitialize()` sets only `periodStartTime` and `custodyAddress`. Neither sets `maxMintAmountPerDay`, so it remains `0` after any fresh deployment or upgrade.

The `checkDailyMintLimit` modifier at lines 42–56 of `contracts/RSETH.sol`:
```solidity
if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
    revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
}
```
With `maxMintAmountPerDay == 0`, this becomes `amount > 0`, which is always true for any real deposit. The revert fires unconditionally.

`RSETH.mint()` (lines 229–240) applies this modifier. `LRTDepositPool._mintRsETH()` (lines 686–690) calls `IRSETH(rsethToken).mint(msg.sender, rsethAmountToMint)`, which is invoked by both `depositETH()` (lines 76–93) and `depositAsset()` (lines 99–118).

The inconsistency is explicit: `remainingDailyMintLimit()` (lines 265–272) short-circuits with `if (maxMintAmountPerDay == 0) return 0;`, treating zero as "no limit," but the modifier that actually guards minting has no such guard.

`setMaxMintAmountPerDay()` (lines 125–128) is a separate, manually-triggered manager call with no enforcement that it is called before deposits open.

## Impact Explanation
All calls to `LRTDepositPool.depositETH()` and `depositAsset()` revert with `DailyMintLimitExceeded(amount, 0)` from the moment of deployment or upgrade until an admin calls `setMaxMintAmountPerDay`. This constitutes a **temporary freezing of funds** (Medium): the L1 deposit pool is entirely non-functional for users, and no deposits can be accepted.

## Likelihood Explanation
The condition is triggered automatically on the first deposit attempt after any deployment or upgrade where `setMaxMintAmountPerDay` is not included in the post-deployment/upgrade script. Because `maxMintAmountPerDay` is absent from both initializers, it silently defaults to zero with no on-chain enforcement or warning. Any depositor immediately discovers the failure on their first transaction. The scenario is realistic at any contract upgrade invoking `reinitialize()` without a subsequent `setMaxMintAmountPerDay` call.

## Recommendation
Add an early-return guard in `checkDailyMintLimit` that treats `maxMintAmountPerDay == 0` as "no limit," consistent with how `remainingDailyMintLimit()` already behaves:
```solidity
modifier checkDailyMintLimit(uint256 amount) {
    if (maxMintAmountPerDay != 0) {
        if (block.timestamp >= periodStartTime + 1 days) {
            currentPeriodMintedAmount = 0;
            periodStartTime = getCurrentPeriodStartTime();
        }
        if (currentPeriodMintedAmount + amount > maxMintAmountPerDay) {
            revert DailyMintLimitExceeded(currentPeriodMintedAmount + amount, maxMintAmountPerDay);
        }
        currentPeriodMintedAmount += amount;
    }
    _;
}
```
Alternatively, set a non-zero default for `maxMintAmountPerDay` inside `reinitialize()`.

## Proof of Concept
1. Deploy `RSETH` and `LRTDepositPool`; call `RSETH.initialize()` and `RSETH.reinitialize()` without calling `setMaxMintAmountPerDay`.
2. Confirm `RSETH.maxMintAmountPerDay()` returns `0`.
3. Call `LRTDepositPool.depositETH{value: 1 ether}(0, "")`.
4. Execution reaches `RSETH.mint(user, rsethAmountToMint)` → `checkDailyMintLimit(rsethAmountToMint)`.
5. Evaluates `0 + rsethAmountToMint > 0` → `true`.
6. Transaction reverts: `DailyMintLimitExceeded(rsethAmountToMint, 0)`.
7. All deposits are blocked until admin calls `setMaxMintAmountPerDay(nonZeroValue)`.

Foundry test plan: deploy contracts in a fresh state, assert `maxMintAmountPerDay == 0`, call `depositETH` with 1 ether, and `vm.expectRevert(abi.encodeWithSelector(RSETH.DailyMintLimitExceeded.selector, rsethAmount, 0))`.
