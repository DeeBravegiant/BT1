Audit Report

## Title
Withdrawal Payout Capped at Initiation-Time Rate, Stripping Yield Accrued During Delay — (File: contracts/LRTWithdrawalManager.sol)

## Summary
`_calculatePayoutAmount()` returns `min(expectedAssetAmount, currentReturn)`, capping the user's payout at the asset amount calculated at withdrawal initiation time. When rsETH appreciates between `initiateWithdrawal` and `unlockQueue` — which is virtually guaranteed over the mandatory 8-day delay — the full rsETH is burned but the user receives only the initiation-time asset equivalent. The delta accrues to the protocol vault, constituting systematic theft of unclaimed yield from every withdrawer.

## Finding Description
**Step 1 — `initiateWithdrawal` (line 168):**
The user transfers `rsETHUnstaked` to the contract. `expectedAssetAmount` is computed at the current oracle price and stored in `WithdrawalRequest.expectedAssetAmount`; `assetsCommitted[asset]` is incremented by this T0 amount. [1](#0-0) 

**Step 2 — `unlockQueue` → `_calculatePayoutAmount` (lines 824–835):**

```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```

When rsETH has appreciated (`rsETHPrice_T1 > rsETHPrice_T0`), `currentReturn > expectedAssetAmount`, so the function returns `expectedAssetAmount` (the T0 amount). `request.expectedAssetAmount` is then overwritten with this capped value. [2](#0-1) 

**Step 3 — rsETH is burned in full (line 305):**
The entire `rsETHAmountToBurn` (including the user's rsETH) is burned, but only the T0 asset amount is redeemed from the vault. [3](#0-2) 

**Step 4 — `completeWithdrawal` → `_processWithdrawalCompletion` (line 734):**
The user receives `request.expectedAssetAmount`, which is the capped T0 amount. [4](#0-3) 

The delta (`currentReturn - expectedAssetAmount`) is never redeemed from the vault. It remains in the protocol's unstaking vault, effectively redistributed to remaining rsETH holders. No existing check prevents this: `_validatePrices` only enforces operator-supplied price bounds and does not affect the `min()` logic. [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.** The yield that accrues on the user's rsETH during the mandatory withdrawal delay is unclaimed yield belonging to the withdrawer. The user surrendered their rsETH at T0; that rsETH continues to appreciate in the contract until it is burned at T1. The `min()` cap ensures the user never receives this appreciation, while the full rsETH is burned. The delta is not destroyed — it remains in the vault, benefiting other rsETH holders at the direct expense of the withdrawer. This matches the allowed impact "High. Theft of unclaimed yield" exactly.

## Likelihood Explanation
**High.** rsETH accrues staking yield continuously (~4–5% APY). The withdrawal delay is 8 days by default (up to 16 days maximum, line 340). Over 8 days at 4.5% APY, rsETH appreciates by approximately 0.099%. This condition is triggered automatically for every withdrawal processed after any oracle price update following initiation — no attacker, no special conditions, no external dependencies required. Any normal user calling `initiateWithdrawal` followed by `completeWithdrawal` after the delay will experience this loss. [6](#0-5) [7](#0-6) 

## Recommendation
Replace the `min()` with `currentReturn` so users receive the full current value of their rsETH at unlock time:

```solidity
function _calculatePayoutAmount(
    WithdrawalRequest storage request,
    uint256 rsETHPrice,
    uint256 assetPrice
) private view returns (uint256) {
    return (request.rsETHUnstaked * rsETHPrice) / assetPrice;
}
```

If the protocol must cap payouts at `expectedAssetAmount` to protect against vault insolvency (i.e., the vault may not hold enough assets to cover the appreciated value), the excess rsETH should be returned to the user rather than burned, or the design rationale should be explicitly documented and the excess should not be silently captured by the protocol.

## Proof of Concept
1. rsETH price at T0: `1.050e18`. stETH price: `1.000e18`.
2. User calls `initiateWithdrawal(stETH, 100e18)`.
   - `expectedAssetAmount = 100e18 * 1.050e18 / 1.000e18 = 105e18` stETH stored.
   - 100 rsETH transferred to contract.
3. 8 days pass. rsETH oracle price updates to `1.051e18` (~0.095% appreciation from staking yield).
4. Operator calls `unlockQueue(stETH, ...)`.
   - `currentReturn = 100e18 * 1.051e18 / 1.000e18 = 105.1e18`.
   - `_calculatePayoutAmount` returns `min(105e18, 105.1e18) = 105e18`.
   - `request.expectedAssetAmount` set to `105e18`.
   - 100 rsETH burned in full.
5. User calls `completeWithdrawal(stETH)`.
   - User receives `105e18` stETH.
   - **Lost yield: `0.1e18` stETH** (~$250 on a $250k withdrawal at current prices).
   - `0.1e18` stETH remains in the vault, benefiting remaining rsETH holders.

**Foundry test plan:** Deploy with a mock oracle. Call `initiateWithdrawal`. Advance blocks past `withdrawalDelayBlocks`. Update mock oracle to return a higher rsETH price. Call `unlockQueue`. Call `completeWithdrawal`. Assert that `user.balance == 105e18` (not `105.1e18`) and that vault balance retains the `0.1e18` delta.
