Audit Report

## Title
ETH Deposit Limit Bypass via Missing Deposit Amount in Limit Check — (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch comparison, using `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. When the running ETH total equals the configured limit exactly, the strict-greater-than check evaluates to `false`, allowing any subsequent ETH deposit to bypass the cap and mint rsETH for ETH the protocol was never intended to hold.

## Finding Description
At `contracts/LRTDepositPool.sol` lines 676–682, the function branches on `LRTConstants.ETH_TOKEN`:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
```

The ERC-20 path correctly includes `amount` so any deposit that would push the total over the limit is rejected. The ETH path only checks whether the total **already** exceeds the limit; when `totalAssetDeposits == depositLimit` the expression `depositLimit > depositLimit` is `false`, so `_beforeDeposit` (lines 648–670) does not revert and `_mintRsETH` issues rsETH for the full over-limit deposit.

`getTotalAssetDeposits(ETH)` (lines 385–397) aggregates `address(this).balance` and all downstream balances, so the total reflects the new deposit immediately on the next call. An attacker can therefore: (1) deposit exactly the remaining gap `G` to bring the total to `depositLimit`, then (2) in the same transaction deposit any additional amount `Y`; on the second call `totalAssetDeposits == depositLimit`, the check passes, and `Y` ETH of rsETH is minted beyond the cap. No privileged role is required.

`getAssetCurrentLimit` (lines 402–409) correctly reports `0` remaining capacity at this state, so off-chain tooling sees the cap as full while `depositETH` still accepts new deposits — confirming the inconsistency is isolated to the limit-check function.

## Impact Explanation
The deposit limit is the protocol's primary on-chain risk-management gate for ETH exposure. Bypassing it allows rsETH to be minted against ETH the protocol never intended to hold, expanding protocol risk beyond the configured ceiling and diluting the rsETH exchange rate for all existing holders. This maps to **Low: contract fails to deliver promised returns** — the protocol's own enforced cap is silently violated. If the limit was sized to bound EigenLayer slashing exposure, repeated exploitation could escalate toward protocol insolvency.

## Likelihood Explanation
Any unprivileged external caller can trigger this in a single atomic transaction via a helper contract. The only precondition is that the remaining gap `G = depositLimit − currentTotal` is at least `minAmountToDeposit`, which is trivially satisfiable whenever the limit has not yet been fully reached. No oracle manipulation, governance action, or external dependency is required. The condition is stable across blocks once engineered and can be repeated if the limit is raised.

## Recommendation
Apply the same inclusive comparison for ETH that is already used for ERC-20 tokens, eliminating the branch entirely:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
```
State: depositLimit = 1000 ETH, getTotalAssetDeposits(ETH) = 999 ETH, minAmountToDeposit = 0.1 ETH

Attacker contract, single transaction:
  Step 1 — depositETH{value: 1 ETH}(0, "")
    _checkIfDepositAmountExceedesCurrentLimit:
      totalAssetDeposits = 999 ETH
      999 > 1000  →  false  →  deposit accepted
    After: totalAssetDeposits = 1000 ETH (exactly at limit)

  Step 2 — depositETH{value: 500 ETH}(0, "")
    _checkIfDepositAmountExceedesCurrentLimit:
      totalAssetDeposits = 1000 ETH
      1000 > 1000  →  false  →  deposit accepted  ← BYPASS
    rsETH minted for 500 ETH at current oracle rate
    After: totalAssetDeposits = 1500 ETH (50% over limit)

Result: attacker holds rsETH for 500 ETH the protocol was never supposed to accept.
```

Foundry invariant test plan: deploy `LRTDepositPool` on a local fork, set `depositLimit = X`, fuzz `depositETH` calls summing to `> X`, and assert `getTotalAssetDeposits(ETH) <= depositLimit` after every call. The invariant will break at `totalAssetDeposits == depositLimit` with any non-zero follow-on deposit.
