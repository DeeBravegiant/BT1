### Title
Fee/Gas Vector Inconsistency in `run_revertible` PostExecution Revert Path — (`crates/blockifier/src/transaction/account_transaction.rs`)

### Summary

When execution succeeds but `PostExecutionReport` fails with `MaxGasAmountExceeded`, the final `TransactionReceipt` is assembled by splicing `post_execution_report.recommended_fee()` (computed from the **full** validate+execute gas vector) into `get_revert_receipt()` (whose `gas` field is computed from **validate-only** state changes). This produces a receipt where `receipt.fee > get_fee_by_gas_vector(receipt.gas)` whenever the execute phase consumed L1 data gas (storage writes) or L1 gas (L1 messages) that were then reverted.

---

### Finding Description

**The splice in `run_revertible`:** [1](#0-0) 

```rust
execution_state.abort();
let tx_receipt = TransactionReceipt {
    fee: post_execution_report.recommended_fee(),  // from full validate+execute gas
    ..get_revert_receipt()                          // gas from validate-only state
};
```

**`get_revert_receipt` uses validate-only state changes:** [2](#0-1) 

The `state_changes` argument is `validate_state_cache.to_state_diff()` — only validate-phase storage writes. Execute-phase storage writes are excluded. However, execute sierra gas IS included via `execution_context.sierra_gas_revert_tracker.get_gas_consumed()` as `reverted_sierra_gas`.

**`recommended_fee` uses the full gas vector:**

`PostExecutionReport::new` is called with `tx_receipt` built from the squashed validate+execute state diff: [3](#0-2) 

The `gas` field of that receipt includes execute-phase L1 data gas. This `gas` is then passed to `FeeCheckReport::from_fee_check_error`: [4](#0-3) 

```rust
ValidResourceBounds::AllResources(all_resource_bounds) => GasVector {
    l1_gas:      min(all_resource_bounds.l1_gas.max_amount,      actual_gas.l1_gas),
    l2_gas:      min(all_resource_bounds.l2_gas.max_amount,      actual_gas.l2_gas),
    l1_data_gas: min(all_resource_bounds.l1_data_gas.max_amount, actual_gas.l1_data_gas),
},
```

`actual_gas.l1_data_gas` here is the **full** (validate + execute) L1 data gas.

**The resulting inconsistency:**

| Field | Source | Value |
|---|---|---|
| `receipt.fee` | `recommended_fee` from full gas | includes execute L1 data gas (capped at max) |
| `receipt.gas.l1_data_gas` | `validate_state_cache` only | validate L1 data gas only |
| `receipt.gas.l2_gas` | validate sierra + execute sierra (reverted) | max_l2_gas + 1 |

So `receipt.fee > get_fee_by_gas_vector(receipt.gas)` when execute makes storage writes.

---

### Impact Explanation

A user submitting an InvokeV3 AllResources transaction is **overcharged** for execute-phase state changes that were reverted. The fee deducted from their balance does not correspond to the gas vector recorded in the receipt. This is an incorrect fee with direct economic impact on users.

The overcharge magnitude is bounded by `min(max_l1_data_gas, execute_l1_data_gas) * l1_data_gas_price - 1 * l2_gas_price`. Since L1 data gas prices are typically much higher than L2 gas prices, even a few storage writes in the execute phase can produce a meaningful overcharge.

---

### Likelihood Explanation

The trigger condition requires:
1. An InvokeV3 AllResources transaction where `max_l2_gas` is set tightly (≤ validate_l2_gas + execute_l2_gas)
2. The execute phase makes at least one storage write (contributing L1 data gas)
3. The execute phase consumes enough L2 gas to exceed `max_l2_gas`

This is reachable through normal RPC transaction submission with no privileged access. Any user who sets tight L2 gas bounds on a transaction that writes storage in execute can trigger this.

---

### Recommendation

When assembling the revert receipt in the `PostExecutionReport` failure branch, the `recommended_fee` should be recomputed using the **revert receipt's own gas vector** (validate-only state), not the full validate+execute gas vector. Specifically:

```rust
execution_state.abort();
let revert_receipt = get_revert_receipt();
let tx_receipt = TransactionReceipt {
    fee: get_fee_by_gas_vector(
        &tx_context.block_context.block_info,
        // cap each component at max_amount
        cap_gas_vector_at_bounds(&revert_receipt.gas, &resource_bounds),
        &FeeType::Strk,
        tx_context.effective_tip(),
    ),
    ..revert_receipt
};
```

Alternatively, `PostExecutionReport::new` should be called with the revert receipt (validate-only state) rather than the full receipt, so that `recommended_fee` is derived from the same gas vector that ends up in the receipt.

---

### Proof of Concept

```
Setup:
  - l2_gas_price = P_l2
  - l1_data_gas_price = P_da  (P_da >> P_l2)
  - validate_l2_gas = V  (validate consumes exactly V units of L2 gas)
  - execute_l2_gas = 1   (execute consumes 1 unit of L2 gas)
  - execute_da_gas = D   (execute makes D storage writes → D units of L1 data gas)
  - max_l2_gas = V       (set exactly to validate consumption)
  - max_l1_data_gas = D  (set to cover execute writes)

Transaction flow:
  1. validate runs → consumes V L2 gas, 0 L1 data gas
  2. execute runs → consumes 1 L2 gas, D L1 data gas (storage writes)
  3. full gas = {l2_gas: V+1, l1_data_gas: D}
  4. PostExecutionReport: MaxGasAmountExceeded(L2Gas, max=V, actual=V+1)
  5. recommended_fee = min(V, V+1)*P_l2 + min(D, D)*P_da = V*P_l2 + D*P_da
  6. revert_receipt.gas = {l2_gas: V+1, l1_data_gas: 0}  (validate state only)
  7. get_fee_by_gas_vector(receipt.gas) = (V+1)*P_l2 + 0*P_da

Assert:
  receipt.fee = V*P_l2 + D*P_da
  get_fee_by_gas_vector(receipt.gas) = (V+1)*P_l2

  receipt.fee - get_fee_by_gas_vector(receipt.gas) = D*P_da - P_l2 > 0  ✓
  (since D*P_da >> P_l2 for any D ≥ 1)
```

The user is overcharged by `D * P_da - P_l2` for reverted execute storage writes.
