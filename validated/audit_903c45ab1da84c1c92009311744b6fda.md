### Title
Bare `assert` in ICC Presence Check Raises `AssertionError` Instead of `ValidationError` on Crafted Peer Block - (`chia/consensus/block_header_validation.py`)

### Summary

In `validate_unfinished_header_block`, the ICC (Infused Challenge Chain) presence/absence invariant at line 203 is enforced with a bare Python `assert` rather than a `return None, ValidationError(...)`. An unprivileged remote peer can craft a `HeaderBlock` that sets `finished_sub_slots[0].infused_challenge_chain` to a non-`None` value while referencing a previous block whose `deficit == MIN_BLOCKS_PER_CHALLENGE_BLOCK`, causing `icc_challenge_hash` to remain `None` and the `assert` to fire as `AssertionError`.

### Finding Description

`icc_challenge_hash` is initialized to `None` at line 166 and is only assigned inside the branch `if prev_b.deficit < constants.MIN_BLOCKS_PER_CHALLENGE_BLOCK:` at line 168. [1](#0-0) 

When `prev_b.deficit == constants.MIN_BLOCKS_PER_CHALLENGE_BLOCK` (the maximum deficit, meaning no ICC should exist), neither the `finished_sub_slot_n == 0` branch (line 171) nor the `elif` branch (line 189) is entered, so `icc_challenge_hash` stays `None`.

The invariant check at line 203 is then: [2](#0-1) 

If an attacker sets `sub_slot.infused_challenge_chain` to a non-`None` value, the expression evaluates to `(False) == (True)` → `False`, raising `AssertionError`. This is not a `ValidationError` and is not part of the function's declared return type `tuple[uint64 | None, ValidationError | None]`.

The callers — `validate_finished_header_block` (line 872) and `Blockchain.validate_unfinished_block_header` (line 773) — do not wrap the call in a `try/except`, so the `AssertionError` propagates up the call stack uncaught. [3](#0-2) [4](#0-3) 

### Impact Explanation

If the full node's block-processing pipeline does not catch `Exception` (or `BaseException`) at the outermost level for this code path, a single crafted peer message causes an unhandled `AssertionError` that crashes the full node process. Even if an outer handler catches it, the block is silently dropped rather than cleanly rejected with a `ValidationError`, which can cause sync divergence or silent peer banning inconsistencies. The worst-case impact is **permanent inability to process subsequent blocks until manual restart**, matching the High impact category.

### Likelihood Explanation

The precondition (`prev_b.deficit == MIN_BLOCKS_PER_CHALLENGE_BLOCK`) is the normal state at the start of any sub-slot that contains no challenge block. This is a routine, publicly observable blockchain state. The correct challenge hash needed to pass the check at line 148 is also public. An attacker needs only to:
1. Observe a block in the chain with `deficit == MIN_BLOCKS_PER_CHALLENGE_BLOCK`.
2. Construct a `finished_sub_slots` entry with the correct `challenge_chain_end_of_slot_vdf.challenge` but with `infused_challenge_chain` set to any non-`None` value.
3. Send the crafted `HeaderBlock` to any reachable full node peer.

No keys, mining power, or privileged access are required.

### Recommendation

Replace the bare `assert` at line 203 (and the subsequent bare `assert` statements at lines 205–209) with explicit `return None, ValidationError(...)` calls, consistent with the rest of the validation function:

```python
# 2e. Validate that there is not icc iff icc_challenge hash is None
if (sub_slot.infused_challenge_chain is None) != (icc_challenge_hash is None):
    return None, ValidationError(Err.SHOULD_NOT_HAVE_ICC)
``` [5](#0-4) 

### Proof of Concept

1. Sync a full node to any point where the chain tip has `deficit == MIN_BLOCKS_PER_CHALLENGE_BLOCK`.
2. Construct a `HeaderBlock` with `prev_header_hash` pointing to that tip, `finished_sub_slots[0].challenge_chain.challenge_chain_end_of_slot_vdf.challenge` set to the correct value (from `curr.finished_challenge_slot_hashes[-1]`), and `finished_sub_slots[0].infused_challenge_chain` set to any `InfusedChallengeChainSubSlot` instance.
3. Call `validate_finished_header_block` (or submit via `_validate_and_add_block`) with this block.
4. Observe `AssertionError` raised at line 203 instead of a `ValidationError` being returned. [6](#0-5)
