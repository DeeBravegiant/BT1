### Title
Proposal Storage Deposits Permanently Lost When `do_update` Clears All Entries Without Refunding Proposers - (File: `crates/contract/src/update.rs`)

### Summary

`propose_update` collects a NEAR storage deposit from each proposer proportional to the size of their proposed update. When `do_update` is triggered (via `vote_update` reaching threshold), it clears **all** pending proposals and votes without refunding any deposits to any proposer. The `bytes_used` field stored in each `UpdateEntry` is never used for refund computation, and no cancel/withdraw path exists. Deposits are permanently locked in the contract balance, breaking the storage-deposit accounting invariant — the direct analog of the reported "reward taken from users' deposits" class.

---

### Finding Description

In `propose_update`, a participant pays a deposit calculated as:

```
required = storage_byte_cost × bytes_used
```

where `bytes_used` includes the full contract binary size plus overhead for 128 participant-vote `AccountId` slots: [1](#0-0) 

The deposit is accepted and only the excess is refunded to the proposer: [2](#0-1) 

The `UpdateEntry` struct stores `bytes_used` alongside the update payload — the only information needed to compute a per-proposer refund: [3](#0-2) 

When `vote_update` reaches threshold it calls `do_update`, which removes the winning entry and then **clears every other pending entry and all votes** with no refund issued to any proposer: [4](#0-3) 

There is no `cancel_update`, `withdraw_update`, or any other path that returns a proposal deposit. The `bytes_used` field is written at proposal time and never read again — it exists in the struct but is dead code with respect to refunds.

---

### Impact Explanation

Every participant who calls `propose_update` permanently loses their deposit when any update is executed. For a realistic 1 MB contract binary:

- `bytes_used` ≈ 1,000,000 (code) + 8,192 (128 × 64-byte `AccountId` overhead) ≈ 1,008,192 bytes  
- At NEAR's storage cost of ~10 yoctoNEAR/byte, the required deposit is **≈ 10 NEAR per proposal**

When `do_update` fires, it clears all N competing proposals simultaneously. If three participants each proposed a different update, all three deposits (~30 NEAR) are permanently locked in the contract balance. The contract balance grows by the sum of all lost deposits with no mechanism to recover them.

This matches the allowed Medium impact: **balance and contract execution-flow manipulation that breaks production safety/accounting invariants** — storage deposits paid to cover storage that is subsequently freed are not returned, violating the fundamental NEAR storage-staking invariant.

---

### Likelihood Explanation

The loss is **guaranteed and deterministic** on every successful governance upgrade cycle. Contract upgrades are a routine, expected operation (the codebase includes dedicated sandbox and E2E tests for the upgrade flow). Any participant who proposes an update — even the participant whose update wins — loses their deposit. The only condition required is that the threshold votes for any update, which is the normal happy path.

A Byzantine participant strictly below the signing threshold can also trigger this: they propose a competing update (paying a deposit), the threshold votes for the legitimate update, and the Byzantine participant's deposit is lost. More importantly, the legitimate proposer's deposit is also lost regardless of Byzantine behavior.

---

### Recommendation

Track the proposer's `AccountId` alongside `bytes_used` in `UpdateEntry`. In `do_update`, before clearing entries, iterate over all entries and issue a `Promise::new(proposer).transfer(refund_amount)` for each one, computing `refund_amount = storage_byte_cost × entry.bytes_used`. This mirrors the refund pattern already used in `propose_update` for excess deposits and in `submit_participant_info` for storage cost differences.

Alternatively, add a `cancel_update` method callable by the original proposer that removes their entry and refunds their deposit before an update executes.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching the required ~10 NEAR deposit. The deposit is accepted; only excess is refunded.
2. Participant B calls `propose_update` with a different 1 MB binary, also paying ~10 NEAR.
3. Threshold participants call `vote_update` for B's `UpdateId`.
4. `vote_update` calls `proposed_updates.do_update(&id, gas)`.
5. `do_update` removes B's entry, then calls `self.entries.clear()` and `self.vote_by_participant.clear()` — A's entry is deleted with no refund.
6. Neither A nor B receives any deposit back. ~20 NEAR is permanently locked in the contract balance.
7. The `bytes_used` field that was stored in both `UpdateEntry` structs — the exact data needed to compute refunds — is silently discarded. [5](#0-4) [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

    /// Propose update to either code or config, but not both of them at the same time.
    #[payable]
    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }
