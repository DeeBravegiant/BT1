### Title
Missing Royalty Percentage Range Check in Bulk NFT Mint Path Permanently Corrupts Offer Settlement - (File: chia/wallet/nft_wallet/nft_wallet.py, chia/wallet/wallet_rpc_api.py)

### Summary

The single-NFT mint path (`generate_new_nft`) validates that the royalty percentage is within the valid range (0–10000 basis points). The bulk mint path (`mint_from_did` / `mint_from_xch`, invoked via the `nft_mint_bulk` RPC) accepts the royalty percentage from caller-supplied metadata and passes it directly into `create_ownership_layer_puzzle` without any range check. An NFT minted with a royalty percentage exceeding 10000 basis points has the invalid value permanently baked into its on-chain puzzle. Any subsequent attempt to settle an offer involving that NFT raises a `ValueError` in `compute_royalty_amount`, permanently breaking offer settlement for the asset.

This guard is hit at offer-settlement time (line 798 of `nft_wallet.py`), raising an unhandled `ValueError` that permanently prevents any offer involving the NFT from being settled. [7](#0-6) 

### Impact Explanation

An NFT minted via the bulk path with `royalty_percentage > 10000` has an invalid royalty permanently encoded in its singleton puzzle. The NFT can be freely transferred (direct spend), but every attempt to settle a standard offer for it will raise a `ValueError` in `compute_royalty_amount`. This permanently corrupts the offer/trade settlement state for the affected NFT. An attacker can mint such NFTs and airdrop them to victims, leaving them holding assets that cannot be traded through the standard offer mechanism. This fits the **High** impact category: corruption of offer/trade settlement state with direct security impact on NFT assets.

### Likelihood Explanation

The `nft_mint_bulk` RPC endpoint is a standard wallet API reachable by any wallet owner. No privileged access is required beyond owning a wallet. The `royalty_percentage` field accepts any `uint16` value (0–65535); values 10001–65535 bypass all validation. The attacker only needs to call the RPC with an out-of-range value, mint the NFT, and transfer it to a target.

### Recommendation

Add the same range check present in `generate_new_nft` to both `mint_from_did` and `mint_from_xch` before passing `metadata["royalty_pc"]` to `create_ownership_layer_puzzle`. Alternatively, enforce the check in `nft_mint_bulk` before building the metadata list:

```python
if request.royalty_percentage is not None and request.royalty_percentage > MAX_ROYALTY_BASIS_POINTS:
    raise ValueError(f"Royalty percentage {request.royalty_percentage} exceeds 100%")
```

### Proof of Concept

1. Call `nft_mint_bulk` RPC with `royalty_percentage=65535` and `mint_from_did=false`.
2. The request passes deserialization (`uint16` accepts 65535).
3. `nft_mint_bulk` builds `metadata_dict` with `"royalty_pc": 65535` — no check performed.
4. `mint_from_xch` calls `create_ownership_layer_puzzle(..., 65535, ...)`, currying 65535 into the NFT's transfer-program puzzle on-chain.
5. Transfer the minted NFT to a victim wallet.
6. Victim attempts to create an offer for the NFT; `make_nft1_offer` calls `compute_royalty_amount(amount, split, 65535)`.
7. `compute_royalty_amount` raises `ValueError("NFT royalty percentage 65535 exceeds 100%")`.
8. Offer settlement is permanently impossible for this NFT.

```python
def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
