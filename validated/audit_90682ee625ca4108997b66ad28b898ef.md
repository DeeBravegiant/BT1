Audit Report

## Title
Missing Token Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Severe rsETH Underpayment for Non-18-Decimal Token Depositors - (File: contracts/pools/RSETHPoolV3ExternalBridge.sol, RSETHPoolV3.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol)

## Summary
All four L2 pool contracts compute rsETH output via `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate`, where `amountAfterFee` is in the token's native decimal units and both rates are in 1e18 precision. For 18-decimal tokens the units cancel correctly, but for any token with fewer decimals (e.g., USDC at 6) the result is `10^(18 − decimals)` times too small. The deposited tokens are transferred into the pool and bridged to L1 while the user receives near-zero rsETH, constituting permanent, irrecoverable fund loss.

## Finding Description
The token-deposit overload of `viewSwapRsETHAmountAndFee` is identical across all four contracts:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`tokenToETHRate` is the oracle's answer to "how many ETH per 1 whole token, expressed in 1e18 precision." For the formula to be dimensionally correct, `amountAfterFee` must also be in 1e18-scaled whole-token units (i.e., the token's native amount must be normalized to 18 decimals first). The code performs no such normalization.

For a 6-decimal token (USDC, USDT) with 1 000 units deposited:
- `amountAfterFee` = `1_000e6` = `1e9`
- `tokenToETHRate` ≈ `3.33e14` (1 USDC ≈ 1/3000 ETH in 1e18 precision)
- `rsETHToETHrate` ≈ `1.05e18`
- **Computed**: `1e9 × 3.33e14 / 1.05e18 ≈ 3.17e5` raw rsETH units
- **Correct**: `1e9 × 1e12 × 3.33e14 / 1.05e18 ≈ 3.17e17` raw rsETH units

The user receives `3.17e5 / 1e18 ≈ 3.17e-13` rsETH (worth ~$0) instead of `3.17e17 / 1e18 ≈ 0.317` rsETH (worth ~$950). The 1 000 USDC is held in the pool and later bridged to L1 via `bridgeTokens`; there is no recovery path for the depositor.

The `addSupportedToken` functions in all four contracts perform no decimal validation — they only check for a non-zero oracle rate:

```solidity
if (IOracle(oracle).getRate() == 0) { revert UnsupportedOracle(); }
```

No existing guard in `deposit`, `viewSwapRsETHAmountAndFee`, or `addSupportedToken` detects or rejects non-18-decimal tokens.

## Impact Explanation
**Critical — Direct theft of user funds.** A depositor of a non-18-decimal token receives `10^(18 − decimals)` times fewer rsETH than owed. For USDC/USDT (6 decimals) this is a factor of `1e12`. The deposited tokens are irrecoverably transferred to the pool and bridged to L1. This matches the allowed impact: "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation
The `addSupportedToken` interface is explicitly designed for protocol expansion beyond ETH and wstETH. Adding USDC, USDT, or any other stablecoin is a natural and foreseeable protocol upgrade. No attacker action is required: once a non-18-decimal token is listed by the TIMELOCK_ROLE (a routine governance action), every subsequent depositor of that token suffers the full loss automatically. The depositor need only call the public `deposit(token, amount, referralId)` function.

## Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate in `viewSwapRsETHAmountAndFee`:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the same normalization to `feeEarnedInToken` accounting and `getTokenBalanceMinusFees`. Alternatively, enforce in `addSupportedToken` that only tokens with exactly 18 decimals may be listed, reverting otherwise.

## Proof of Concept
1. TIMELOCK_ROLE calls `addSupportedToken(USDC, usdcOracle, usdcBridge)` on any of the four pool contracts. `usdcOracle.getRate()` returns `3.33e14` (non-zero), so the call succeeds.
2. User calls `deposit(USDC, 1_000e6, "ref")`.
3. `IERC20(USDC).safeTransferFrom(user, pool, 1_000e6)` — 1 000 USDC leaves the user.
4. `viewSwapRsETHAmountAndFee(1_000e6, USDC)` computes:
   - `fee = 1_000e6 * feeBps / 10_000` (negligible)
   - `amountAfterFee ≈ 1_000e6`
   - `rsETHAmount = 1_000e6 * 3.33e14 / 1.05e18 ≈ 317_000`
5. `wrsETH.mint(user, 317_000)` — user receives `317_000 / 1e18 ≈ 3.17e-13` rsETH.
6. BRIDGER_ROLE later calls `bridgeTokens(USDC)` — 1 000 USDC is bridged to L1 permanently.
7. User's $1 000 is gone; they hold dust rsETH worth ~$0.
