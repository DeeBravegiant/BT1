### Title
Single Attested Participant Can Inject Arbitrary CKD Response for `AppPublicKey` Variant, Bypassing Threshold Requirement — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs **no cryptographic verification** of the `CKDResponse` when the pending request uses the `CKDAppPublicKey::AppPublicKey` (non-publicly-verifiable) variant. Any single malicious attested participant can call `respond_ckd` with an arbitrary crafted response for any pending CKD request of this type, draining the yield queue and delivering a wrong confidential key to the victim — bypassing the threshold-signature requirement entirely.

---

### Finding Description

In `respond_ckd`, the response verification is conditional on the app public key variant: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← NO VERIFICATION
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For the `AppPublicKeyPV` variant, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically proves the response is the output of a legitimate threshold computation over the correct `app_id`. [2](#0-1) 

For the `AppPublicKey` variant, **no such check exists**. The contract unconditionally accepts any `(big_c, big_y)` pair and immediately resolves all queued yields with it: [3](#0-2) 

`resolve_yields_for` drains the **entire** fan-out queue in one call, so once the attacker's garbage response is accepted, the legitimate threshold-computed response can never be submitted — the map entry is gone: [4](#0-3) 

The `respond_ckd` function is callable by **any** single attested participant — there is no leader restriction: [5](#0-4) 

By contrast, `respond` for regular signature requests always verifies the signature against the derived public key before resolving yields, making forgery impossible for a single participant: [6](#0-5) 

The `AppPublicKey` variant is the primary/simpler CKD path (the `AppPublicKeyPV` variant is the opt-in publicly-verifiable extension), so this affects the default CKD usage pattern.

---

### Impact Explanation

**Critical — Permanent loss of funds / unauthorized key derivation output.**

The CKD scheme is designed so that the user unmasks the confidential key using their ephemeral private key:

```
confidential_key = CKDOutput::new(big_y, big_c).unmask(private_key)
```

If a malicious participant injects arbitrary `(big_c, big_y)` values, the victim unmasks a garbage value and receives a wrong confidential key. Any assets sent to addresses derived from this wrong key are permanently unrecoverable. The attacker also permanently prevents the victim from ever receiving the correct key for that request (the yield is drained). This constitutes both unauthorized key derivation output and permanent freezing of funds.

---

### Likelihood Explanation

**High.** The attack requires only a single Byzantine participant strictly below the signing threshold — exactly the adversarial model the threshold design is meant to defend against. The attacker must:
1. Be an attested participant (a realistic assumption for a Byzantine node).
2. Observe a pending `AppPublicKey` CKD request on-chain (all pending requests are public).
3. Call `respond_ckd` with arbitrary `big_c`/`big_y` before the legitimate response arrives.

No collusion, no key leakage, and no privileged operator access is required. The race window is the entire duration between the user's `request_app_private_key` call and the MPC nodes' legitimate `respond_ckd` call.

---

### Recommendation

Apply the same on-chain cryptographic verification to the `AppPublicKey` variant that is already applied to `AppPublicKeyPV`. Since the `AppPublicKey` variant does not include a G2 component for pairing, an alternative is to require the response to be accompanied by a threshold-aggregated BLS proof of correct computation, or to mandate that all CKD requests use the `AppPublicKeyPV` variant (which is already verifiable on-chain). At minimum, the contract should not silently skip verification for the non-PV path.

---

### Proof of Concept

1. **Victim** calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(victim_ephemeral_pk)` and `derivation_path = "my_wallet"`. A yield is queued under the `CKDRequest` key.

2. **Attacker** (a single malicious attested participant) observes the pending request on-chain and immediately calls:
   ```
   respond_ckd(
     request = <victim's CKDRequest>,
     response = CKDResponse { big_c: [0u8; 48], big_y: [0u8; 48] }
   )
   ```

3. The contract reaches the `AppPublicKey` branch, skips verification, and calls `resolve_yields_for`, which resumes the victim's yield with the garbage response.

4. The victim's `return_ck_and_clean_state_on_success` callback fires with `Ok(CKDResponse { big_c: [0;48], big_y: [0;48] })`.

5. The victim computes `confidential_key = CKDOutput::new([0;48], [0;48]).unmask(private_key)` — a deterministic garbage value.

6. The victim's application derives a wallet address from this garbage key and receives funds there. Those funds are permanently frozen because the correct key was never delivered and can never be re-requested under the same `(account_id, derivation_path)` without the victim knowing the response was corrupted. /// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
