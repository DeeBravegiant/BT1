### Title
Sub-Transaction Witness Check Uses Stale `originalUtxo`, Bypassing Authorization for Intra-Transaction UTxO Spending — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxow.hs`)

---

### Summary

In the Dijkstra era's `SUBUTXOW` rule, the required-witness computation is performed against `originalUtxo` — the UTxO set that existed at the start of the top-level transaction — rather than the live UTxO state that accumulates outputs from previously-processed sub-transactions. Because `getShelleyWitsVKeyNeededNoGov` silently skips any input it cannot find in the supplied UTxO map, UTxOs created by an earlier sub-transaction carry **no witness requirement** when spent by a later sub-transaction. The `SUBUTXO` rule, however, operates on the live state and will accept the spend. The result is that a top-level transaction author can redirect or destroy funds sent to any address inside the same nested-transaction bundle without holding the recipient's signing key.

---

### Finding Description

**Vulnerability class:** Missing access control / witness-authorization bypass on a state-setter (direct analog to the report's `LendingPoolAddressesProvider` setter with no `onlyOwner` guard).

**Root cause — `dijkstraSubUtxowTransition`:**

```
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody
```

`originalUtxo` is the frozen snapshot delivered through `SubUtxoEnv` and never updated across sub-transactions.

`validateNeededWitnesses` delegates to `getWitsVKeyNeeded → getConwayWitsVKeyNeeded → getShelleyWitsVKeyNeededNoGov`:

```haskell
inputAuthors = foldr' accum Set.empty (txBody ^. spendableInputsTxBodyF)
  where
    accum txin !ans =
      case txinLookup txin utxo' of   -- utxo' == originalUtxo
        Just txOut -> ...              -- add payment key to required set
        Nothing    -> ans              -- ← silently skipped; no witness required
```

Any input whose `TxIn` is absent from `originalUtxo` contributes **zero** required witnesses.

**How the live state diverges from `originalUtxo`:**

In `dijkstraSubLedgersTransition`, the `LedgerState` (including `utxoState`) is threaded through successive sub-transaction applications. Each sub-transaction's outputs are added to `utxoState`. When the *n*-th sub-transaction is processed, `utxoStateBeforeSubUtxow` already contains outputs from sub-transactions 1 … n−1. `SUBUTXOW` receives this live state as its `State` argument and forwards it to `SUBUTXO`, which checks input existence against the live state — not `originalUtxo`.

```haskell
-- SubLedger.hs (dijkstraSubLedgersTransition)
utxoStateAfterSubUtxow <-
  trans @(EraRule "SUBUTXOW" era) $
    TRC
      ( SubUtxoEnv slot pp certState originalUtxo topIsValid  -- ← stale snapshot
      , utxoStateBeforeSubUtxow                               -- ← live state
      , stAnnTx
      )
```

The witness check and the UTxO-spending check therefore operate on **different UTxO sets**, breaking the authorization invariant.

---

### Impact Explanation

An attacker who authors a top-level Dijkstra transaction can:

1. **Sub-tx A** — output 1 000 ADA to `Addr _ (KeyHashObj aliceKey) _` (a legitimate protocol payment to Alice).
2. **Sub-tx B** — spend that output, redirecting funds to the attacker's address, **without providing `aliceKey`'s signature**.

`SUBUTXOW` looks up the input in `originalUtxo`, finds nothing, adds no entry to the required-witness set, and passes. `SUBUTXO` looks up the input in the live state, finds it, and executes the spend. The ledger accepts an invalid state transition in which Alice's funds are destroyed without her authorization.

**Allowed impact matched:** *Critical — Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition.*

---

### Likelihood Explanation

The Dijkstra era's nested-transaction feature is explicitly designed for protocols that distribute or route funds across multiple sub-transactions within a single top-level transaction (e.g., batched DEX settlements, multi-party payment channels, smart-contract-driven disbursements). Any such protocol is a realistic attack surface. The attacker's only precondition is authoring a valid top-level transaction — no privileged keys, no governance majority, no 51% attack required.

---

### Recommendation

Replace `originalUtxo` with the live UTxO in the witness check inside `dijkstraSubUtxowTransition`:

```haskell
-- current (vulnerable)
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState originalUtxo txBody

-- fixed
let liveUtxo = utxosUtxo utxoState
runTest $ Shelley.validateNeededWitnesses witsKeyHashes certState liveUtxo txBody
```

Apply the same correction to `Alonzo.missingRequiredDatums`, which also uses `originalUtxo` and would similarly skip datum checks for script-locked outputs created by earlier sub-transactions.

### Proof of Concept

```
Top-level Tx (Dijkstra era)
  Sub-tx 1 (outputs):
    TxOut { addr = Addr Mainnet (KeyHashObj aliceKeyHash) StakeRefNull
           , value = 1_000_000_000 lovelace }   -- TxIn = (txId_sub1, 0)

  Sub-tx 2 (inputs):
    TxIn = (txId_sub1, 0)                        -- spends Alice's output
    (no addrTxWits for aliceKeyHash)
    TxOut { addr = attackerAddr, value = 1_000_000_000 lovelace }
```

**SUBUTXOW witness check for Sub-tx 2:**
- `originalUtxo` does not contain `(txId_sub1, 0)` → `txinLookup` returns `Nothing` → `aliceKeyHash` is never added to `needed` → `validateNeededWitnesses` passes.

**SUBUTXO for Sub-tx 2:**
- Live `utxoState` contains `(txId_sub1, 0)` (created by Sub-tx 1) → input is valid → spend succeeds.

validateNeededWitnesses ::
  EraUTxO era =>
  -- | Provided witness
  Set (KeyHash Witness) ->
  CertState era ->
  UTxO era ->
  TxBody t era ->
  Test (ShelleyUtxowPredFailure era)
validateNeededWitnesses witsKeyHashes certState utxo txBody =
  let needed = getWitsVKeyNeeded certState utxo txBody
      missingWitnesses = Set.difference needed witsKeyHashes
   in failureOnNonEmptySet missingWitnesses MissingVKeyWitnessesUTXOW
