### Title
Unrestricted `clean_invalid_attestations` Allows Any Caller to Prune Active Participant Attestations During TEE Upgrade Windows - (File: `crates/contract/src/lib.rs`)

### Summary
`clean_invalid_attestations` carries no access-control guard and is explicitly documented as "Callable by anyone." During a TEE upgrade window — when the participant-set whitelist is being rotated and some nodes have not yet re-submitted attestations under the new code hash — an unprivileged attacker can call this function to prune the attestations of current participants, stripping them of the ability to call any node-API method (`respond`, `vote_pk`, `vote_reshared`, etc.) and potentially preventing the network from reaching signing threshold.

---

### Finding Description

`clean_invalid_attestations` is a public, permissionless mutating function on the MPC contract: [1](#0-0) 

The function's own doc-comment confirms the absence of any caller restriction:

> "Callable by anyone while the protocol is in `Running`."

No `assert_caller_is_signer`, `voter_or_panic`, `assert_caller_is_attested_participant_and_protocol_active`, or `#[private]` guard is present. Compare this with the analogous cleanup helpers that **do** carry access control:

- `clean_tee_status` — `#[private]` (contract-only) [2](#0-1) 

- `clean_foreign_chain_data` — `#[private]` [3](#0-2) 

- `remove_non_participant_update_votes` — restricted to self-call or current participant [4](#0-3) 

`clean_invalid_attestations` is the only mutating cleanup path that is intentionally left open to the public.

The function delegates to `tee_state.clean_invalid_attestations(tee_upgrade_deadline_duration, max_scan)`, which iterates stored attestations and removes any that fail re-verification against the current whitelist and deadline. "Fail re-verification" includes attestations whose referenced code hash has been removed from the allowed set — a condition that **legitimately applies to current participants** during the overlap window of a TEE upgrade (i.e., after the old code hash is voted out but before all nodes have re-submitted under the new hash).

The `tee_upgrade_deadline_duration` used for re-verification is read directly from `self.config`: [5](#0-4) 

An attacker has no control over this value, but they do control **when** they call the function and the `max_scan` argument, which determines how many entries are swept.

---

### Impact Explanation

The attestation stored for a node is the sole credential checked by `is_caller_an_attested_participant`: [6](#0-5) 

Every node-API method — `respond`, `respond_ckd`, `respond_verify_foreign_tx`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`, `vote_abort_key_event_instance` — gates on `assert_caller_is_attested_participant_and_protocol_active`: [7](#0-6) 

If an attacker removes the stored attestation of a current participant, that participant immediately loses the ability to call any of those methods. If the attacker removes attestations for ≥ (N − threshold + 1) participants, the network can no longer assemble a threshold-sized set of responding nodes, freezing all pending signature requests. Pending yields will time out and fail, permanently denying service to users whose requests are in flight.

**Impact class:** Medium — participant-state manipulation that breaks the production safety invariant (current participants must be able to call node-API methods) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

The attack window is bounded but predictable:

1. Participants vote to remove an old MPC code hash (e.g., after a software upgrade). This is a normal governance operation.
2. Between the vote completing and all nodes re-submitting attestations under the new hash, the old attestations fail re-verification.
3. An attacker monitoring the chain observes the whitelist change and immediately calls `clean_invalid_attestations(max_scan: <large value>)`.
4. Affected participants' attestations are deleted; they cannot call `respond` or any other node-API method until they re-submit.

The attacker needs no special privilege, no key material, and no collusion. The only requirement is monitoring the chain for whitelist-change transactions — trivially achievable with a NEAR indexer. The window may be short (seconds to minutes if nodes are automated), but the call is a single transaction and can be front-run.

---

### Recommendation

Apply the same access-control pattern used by the other cleanup helpers. Either:

1. Add `#[private]` so only the contract itself can call it (matching `clean_tee_status` and `clean_foreign_chain_data`), and have `vote_reshared` spawn it as a detached promise alongside the other cleanup calls; or
2. Restrict to current participants or self-call, matching the pattern in `remove_non_participant_update_votes`.

Option 1 is preferred because it eliminates the external attack surface entirely while preserving the automated post-reshare cleanup behaviour already established for the other helpers.

### Proof of Concept

```
# Attacker observes a vote_remove_launcher_hash / vote_code_hash transaction
# that removes an old code hash from the whitelist.
# Immediately after that transaction is finalized:

near call <mpc_contract_id> clean_invalid_attestations \
  '{"max_scan": 1000}' \
  --accountId attacker.near \
  --gas 300000000000000

# Result: stored attestations of participants still running the old code hash
# are deleted. Those participants can no longer call respond/vote_pk/vote_reshared.
# Pending sign() yields time out and fail for all in-flight requests.
```

The call succeeds because `clean_invalid_attestations` performs no caller check before invoking `tee_state.clean_invalid_attestations`. [8](#0-7)

    /// Prunes up to `max_scan` stored attestations that fail re-verification (expired or
    /// referencing stale whitelists). Returns the number of entries removed. Callable by
    /// anyone while the protocol is in `Running`.
    #[handle_result]
    pub fn clean_invalid_attestations(&mut self, max_scan: u32) -> Result<u32, Error> {
        log!(
            "clean_invalid_attestations: signer={}, max_scan={}",
            env::signer_account_id(),
            max_scan
        );
        // Running-only: keygen / resharing may reference attestations that have not yet
        // been activated, so cleanup is off-limits during those phases.
        if !matches!(self.protocol_state, ProtocolContractState::Running(_)) {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }
        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);
        Ok(self
            .tee_state
            .clean_invalid_attestations(tee_upgrade_deadline_duration, max_scan as usize))
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
