Looking at the code carefully to determine if this is a real vulnerability.

### Title
Missing subgroup check for `root_Q` and `root_K` in `blob_to_keyset` allows torsion-point keyset blobs to be accepted ‚Äî (`src/cbmpc/api/hd_keyset_eddsa_2p.cpp`)

---

### Summary

`hd_keyset_eddsa_2p::blob_to_keyset` calls `from_bin` for `root_Q_compressed` and `root_K_compressed` but never calls `curve.check()` or `is_in_subgroup()` on the decoded points. Because `from_bin` succeeds for Ed25519 torsion points (they satisfy the curve equation), an attacker who supplies a crafted `keyset_blob` with a torsion root public key passes deserialization and causes `extract_root_public_key_compressed`, `refresh`, and `derive_eddsa_2p_keys` to return `SUCCESS` with cryptographically invalid output.

---

### Finding Description

**The missing guard.** In `blob_to_keyset`:

```cpp
// src/cbmpc/api/hd_keyset_eddsa_2p.cpp lines 78-80
error_t rv = keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed);
if (rv) return rv;
return keyset.root.K.from_bin(keyset.curve, blob.root_K_compressed);
```

No `curve.check()` or `is_in_subgroup()` call follows either `from_bin`. [1](#0-0) 

**The analogous function in `eddsa2pc.cpp` does have the guard:**

```cpp
// src/cbmpc/api/eddsa2pc.cpp lines 40-42
error_t rv = key.Q.from_bin(key.curve, blob.Q_compressed);
if (rv) return coinbase::error(rv, "invalid key blob");
if (key.curve.check(key.Q)) return coinbase::error(E_FORMAT, "invalid key blob");
``` [2](#0-1) 

**`ecurve_t::check` calls `is_in_subgroup` and rejects torsion points:**

```cpp
// src/cbmpc/crypto/base_ecc.cpp line 595
if (!point.is_in_subgroup()) return crypto::error("EC-point is not on curve");
``` [3](#0-2) 

**`from_bin` succeeds for torsion points.** The existing test `CryptoEdDSA::RejectTorsionAndFixInfinityEq` explicitly confirms that `from_bin` returns `SUCCESS` for the order-2 torsion point (y = p‚àí1), while `is_in_subgroup()` returns `false` and `curve.check()` returns non-SUCCESS:

```cpp
// tests/unit/crypto/test_eddsa.cpp lines 26-30
EXPECT_EQ(P.from_bin(curve, mem_t(order2, 32)), SUCCESS);
EXPECT_TRUE(P.is_on_curve());
EXPECT_FALSE(P.is_infinity());
EXPECT_FALSE(P.is_in_subgroup());
EXPECT_NE(curve.check(P), SUCCESS);
``` [4](#0-3) 

**All three public API entry points are affected** because they all call `deserialize_keyset_blob` ‚Üí `blob_to_keyset` without any subsequent point validation:

- `extract_root_public_key_compressed` (line 195) ‚Üí returns `SUCCESS` and emits the torsion point as the root public key. [5](#0-4) 
- `refresh` (line 127) ‚Üí accepts the keyset; `key_share_eddsa_hdmpc_2p_t::refresh` copies `Q` verbatim (`new_key.root.Q = key.root.Q`), so the refreshed blob re-serializes the torsion root Q and returns `SUCCESS`. [6](#0-5) 
- `derive_eddsa_2p_keys` (line 151) ‚Üí accepts the keyset; `derive_keys` computes `Q_derived = Q + delta*G` where `Q` is the torsion point, producing a derived public key in a coset of the prime-order subgroup. [7](#0-6) 

---

### Impact Explanation

**`extract_root_public_key_compressed`:** Returns `SUCCESS` with a torsion point as the root public key. Callers that use this output to verify signatures or derive child keys receive a cryptographically wrong value with no indication of error.

**`refresh`:** Returns `SUCCESS` and a new keyset blob that still carries the torsion root Q. The torsion contamination persists across refresh cycles.

**`derive_eddsa_2p_keys`:** The derived `eddsa2pc` key blobs carry `Q_derived = torsion + delta*G`, which is not in the prime-order subgroup. When these blobs are later loaded by `eddsa_2p::blob_to_key` (which does call `curve.check`), they will fail ‚Äî but the `derive_eddsa_2p_keys` call itself returns `SUCCESS`, creating a silent divergence between what the HD keyset layer accepts and what the signing layer will accept.

The root cause is a validation asymmetry: the `eddsa_2p` blob layer enforces subgroup membership on deserialization, but the `hd_keyset_eddsa_2p` blob layer does not, so attacker-controlled blobs with torsion root points are accepted as valid.

---

### Likelihood Explanation

The `keyset_blob` is an opaque byte string that callers persist and supply back to the API. Any caller who stores or transmits keyset blobs is a potential injection point. Constructing a valid-looking `keyset_blob_v1_t` with a torsion `root_Q_compressed` requires only knowledge of the serialization format (which is straightforward from the struct definition) and the 32-byte encoding of a known torsion point. No cryptographic secret is needed.

---

### Recommendation

Add `curve.check()` (which subsumes `is_in_subgroup()`) for both `root_Q` and `root_K` immediately after their respective `from_bin` calls in `blob_to_keyset`, mirroring the pattern already used in `eddsa_2p::blob_to_key`:

```cpp
// After line 78-79 in src/cbmpc/api/hd_keyset_eddsa_2p.cpp
error_t rv = keyset.root.Q.from_bin(keyset.curve, blob.root_Q_compressed);
if (rv) return rv;
if (keyset.curve.check(keyset.root.Q)) return coinbase::error(E_FORMAT, "invalid keyset blob");

rv = keyset.root.K.from_bin(keyset.curve, blob.root_K_compressed);
if (rv) return rv;
if (keyset.curve.check(keyset.root.K)) return coinbase::error(E_FORMAT, "invalid keyset blob");
```

### Proof of Concept

A deterministic unit test (no network required):

1. Encode the Ed25519 order-2 torsion point: `order2[0]=0xec`, `order2[1..30]=0xff`, `order2[31]=0x7f` (y = p‚àí1, sign bit = 0).
2. Serialize a `keyset_blob_v1_t` with `version=1`, `role=0`, `curve=ed25519`, `root_Q_compressed=order2`, `root_K_compressed=<any valid subgroup point>`, `x_share` and `k_share` in range.
3. Call `extract_root_public_key_compressed(blob, out)` ‚Äî assert it returns `E_FORMAT` (currently returns `SUCCESS`).
4. Call `refresh(job, blob, new_blob)` ‚Äî assert it returns `E_FORMAT` (currently returns `SUCCESS`).
5. Call `derive_eddsa_2p_keys(job, blob, ...)` ‚Äî assert it returns `E_FORMAT` (currently returns `SUCCESS`).

The test confirms that `blob_to_keyset` accepts the torsion-point keyset blob without error, while the analogous `eddsa_2p::blob_to_key` would reject an equivalent torsion-point key blob.
