### Title
`respond_verify_foreign_tx` Verifies Signature Against Caller-Supplied `payload_hash` Instead of Request-Derived Hash, Enabling Cross-Request Signature Replay - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function verifies the ECDSA signature against `response.payload_hash`, a value supplied entirely by the calling MPC node, without ever checking that this hash matches the hash that should have been computed from the original `request` data. A single malicious MPC participant (strictly below the signing threshold) can replay any previously produced, on-chain-visible signature to resolve a pending foreign-transaction verification request with fabricated extracted values, bypassing the foreign-chain inspection requirement entirely.

---

### Finding Description

In `respond_verify_foreign_tx` (`crates/contract/src/lib.rs`, lines 718–743), the contract extracts `payload_hash` directly from the caller-supplied `response.payload_hash` field and passes it to the signature verifier:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← attacker-controlled

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,                                        // ← checked against wrong value
    &secp_pk,
)
.is_ok()
```

The contract never independently computes the expected payload hash from the `request` fields and compares it to `response.payload_hash`. The check therefore only proves "this signature is valid over *some* hash under the root key" — not "this signature is valid over the hash of *this specific request's* verified foreign-chain data."

**Contrast with `respond`** (lines 600–608), which correctly derives `payload_hash` from the original, user-submitted `request.payload` — a value the responding node cannot alter:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← taken from the immutable request, not the response
    &expected_public_key,
)
```

The asymmetry is the root cause. For regular `respond`, the hash is anchored to the request. For `respond_verify_foreign_tx`, the hash floats freely in the response.

**Analog to H-01:** In H-01, the invariant check `assetIn * due.collateral >= collateralOut * due.debt` is evaluated before `assetIn` and `collateralOut` are accumulated, so on the first loop iteration both are zero and the check trivially passes. Here, the invariant check (signature verification) is evaluated against an attacker-controlled value (`response.payload_hash`) instead of the value derived from the request, so the check trivially passes for any hash over which the attacker holds a valid prior signature.

---

### Impact Explanation

A malicious MPC participant can resolve any pending `verify_foreign_transaction` request with fabricated extracted values and a recycled signature. The contract emits the forged `VerifyForeignTransactionResponse` (containing the wrong `payload_hash` and the recycled signature) to the waiting user yield. Bridge services or smart contracts that consume this response without independently recomputing the expected payload hash will accept the forged attestation as proof that a foreign-chain transaction was legitimately verified, enabling invalid bridge execution or double-spend conditions.

This matches the allowed High impact: **forged foreign-chain verification that causes invalid bridge execution**.

---

### Likelihood Explanation

- The attacker must be a single attested MPC participant — strictly below the signing threshold. No collusion is required.
- Every legitimate `respond_verify_foreign_tx` call publishes `(payload_hash, signature)` on-chain in plain sight. The attacker needs only to read a past transaction to obtain a valid `(H1, S1)` pair.
- The attacker can submit a new `verify_foreign_transaction` request for any transaction (real or fabricated) and immediately call `respond_verify_foreign_tx` with the recycled pair. No cryptographic work is required beyond reading the chain.
- `respond_verify_foreign_tx` has no leader restriction; any attested participant may call it.

---

### Recommendation

The contract must bind the signature check to the request, not to the caller-supplied hash. Two complementary fixes:

1. **Require the responding node to supply the extracted values** in the response, and have the contract recompute `expected_hash = ForeignTxSignPayload::V1 { request, values }.compute_msg_hash()`, then verify the signature against `expected_hash` rather than `response.payload_hash`.

2. **Short-term mitigation:** At minimum, verify that `response.payload_hash` encodes the correct `request` fields (domain, chain, tx-id) by including a deterministic prefix derived solely from the request in the hash construction, and checking that prefix on-chain.

---

### Proof of Concept

1. The MPC network legitimately processes request **R1** (for real transaction T1). The `respond_verify_foreign_tx` call is submitted on-chain, making **(H1, S1)** publicly visible.

2. Attacker (a single malicious MPC participant) calls `verify_foreign_transaction(R2)` for a fraudulent or non-existent transaction T2. A pending yield is created in `pending_verify_foreign_tx_requests`.

3. Attacker immediately calls:
   ```
   respond_verify_foreign_tx(
       request  = R2,
       response = { payload_hash: H1, signature: S1 }
   )
   ```

4. Contract executes (lines 726–734):
   ```rust
   let payload_hash: [u8; 32] = H1;   // attacker-supplied
   verify_ecdsa_signature(S1, H1, root_key)  // → Ok, because S1 was legitimately produced over H1
   ```
   `signature_is_valid = true`.

5. `pending_requests::resolve_yields_for` resolves R2's yield with `{ payload_hash: H1, signature: S1 }`.

6. The user who submitted R2 receives a `VerifyForeignTransactionResponse` asserting that T2 was verified, carrying H1 (T1's hash) and S1 (T1's signature). Any bridge service that does not independently recompute the expected hash for T2 will accept this as a valid attestation and execute the corresponding bridge action.     #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
