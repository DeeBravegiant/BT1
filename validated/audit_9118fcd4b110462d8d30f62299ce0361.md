### Title
Unrestricted BIP32 Path Signing via `execute_signing_instructions` Enables Arbitrary Message Signing with Pool Wallet Owner Keys — (`chia/wallet/wallet.py`)

### Summary

The `execute_signing_instructions` RPC endpoint accepts caller-supplied `PathHint` objects with no restriction on the BIP32 derivation path. Any caller with wallet RPC access can specify the pool wallet owner key path (`[12381, 8444, 5, N]`) and a `SigningTarget` with an attacker-chosen message, causing the wallet to produce a valid BLS signature over that message using the pool singleton owner secret key. This signature can be used to authorize an unauthorized pool singleton state transition (e.g., redirecting pool rewards to an attacker-controlled pool).

### Finding Description

**Entry point:** `POST /execute_signing_instructions` → `WalletRpcApi.execute_signing_instructions` → `WalletStateManager.execute_signing_instructions` → `Wallet.execute_signing_instructions`

In `chia/wallet/wallet.py`, `execute_signing_instructions` (lines 541–659):

**Step 1 — Path hint processing (lines 559–576):** The only guard on a `PathHint` is whether its `root_fingerprint` matches the wallet's own root pubkey fingerprint:

```python

    ...  # skip or raise
else:
```

There is **no restriction on what `path` values are accepted**. Any path derivable from the master key — including `[12381, 8444, 5, N]` (pool wallet owner key, as defined in `master_sk_to_singleton_owner_sk` at `chia/wallet/derive_keys.py:80-84`) — is derived and placed into `sk_lookup`.

**Step 2 — Signing (lines 604–618):** For each `SigningTarget`, if the fingerprint matches a key in `sk_lookup`, the wallet signs the **caller-supplied `target.message`** with no validation:

```python
elif pk_fingerprint in sk_lookup:

        SigningResponse(
            bytes(AugSchemeMPL.sign(sk_lookup[pk_fingerprint], target.message)),
            target.hook,
        )
    )
```

There is **no check that `target.message` corresponds to actual spend conditions**. The message is entirely attacker-controlled.

**RPC access:** The endpoint is registered at `chia/wallet/wallet_rpc_api.py:721` with no additional access control beyond the standard mutual TLS used for all wallet RPC endpoints. The private CA certificates are stored locally (`~/.chia/mainnet/config/ssl/`), accessible to any process running as the same OS user as the wallet.

### Impact Explanation

An attacker with wallet RPC access can:

1. Obtain the wallet's root pubkey fingerprint via `/get_logged_in_fingerprint` or `/get_public_keys`.
2. Enumerate pool wallet indices 0–99 (`MAX_POOL_WALLETS = 100`, `chia/wallet/derive_keys.py:16`) to find the active pool wallet owner key fingerprint.
3. Observe the blockchain to find the current pool singleton coin ID and construct a malicious spend bundle (e.g., changing the pool target to an attacker-controlled pool address).
4. Compute the exact AGG_SIG_ME message: `delegated_puzzle_hash + coin_id + AGG_SIG_ME_ADDITIONAL_DATA`.
5. POST to `/execute_signing_instructions` with `PathHint(root_fingerprint, [12381, 8444, 5, N])` and `SigningTarget(owner_pk_fingerprint, evil_msg, hook)`.
6. Receive a valid BLS signature over `evil_msg` from the pool wallet owner key.
7. Submit the malicious spend bundle with this signature, redirecting pool rewards to the attacker's pool.

This constitutes unauthorized use of the pool wallet owner key to authorize singleton state transitions — a direct match for the High impact category: "Bypass of wallet... authorization that enables unauthorized signing, key use... payout redirection, singleton mutation, or protected state transitions."

### Likelihood Explanation

- The wallet RPC is accessible to any local process running as the same OS user as the wallet (standard deployment).
- The root pubkey fingerprint is trivially obtainable via other RPC calls.
- Pool wallet derivation indices are enumerable (0–99).
- The pool singleton coin state is publicly visible on-chain.
- No user confirmation or additional authentication is required for `execute_signing_instructions`.
- The existing test `test_p2blsdohp_execute_signing_instructions` (`chia/_tests/wallet/test_signer_protocol.py:318`) already demonstrates signing arbitrary messages with arbitrary paths — confirming the path is fully exercised.

### Recommendation

1. **Restrict allowed paths in `PathHint`:** In `execute_signing_instructions`, validate that the supplied path corresponds to a known, wallet-registered derivation path (e.g., only paths present in the puzzle store, or only `[12381, 8444, 2, index]` wallet paths). Reject paths for pool owner keys (`[12381, 8444, 5, N]`), farmer keys, etc., unless the signing target message is derived from an actual spend condition.
2. **Bind signing to spend conditions:** Validate that `SigningTarget.message` matches a message derivable from the `AGG_SIG` conditions of a known spend bundle, rather than accepting arbitrary bytes.
3. **Separate the signing oracle from the signer protocol:** If `execute_signing_instructions` must remain a general-purpose signer, add explicit user confirmation for signing with sensitive key paths (pool owner, farmer, etc.).
