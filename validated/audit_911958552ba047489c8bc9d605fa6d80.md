### Title
Unvalidated `ProposalInit.builder` allows a malicious proposer to redirect all block transaction fees to an arbitrary address - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` never checks `ProposalInit.builder` against any locally-trusted reference. Because `builder` is forwarded verbatim as the `sequencer_address` (fee recipient) for every transaction executed in the block, a malicious proposer can set it to an arbitrary address and silently redirect all transaction fees. Validators accept the proposal without detecting the substitution, and the final `ProposalFin` commitment check passes because both sides derive the commitment from the same attacker-controlled `builder`.

### Finding Description

`ProposalInit.builder` is documented as "Address of the one who builds/sequences the block." [1](#0-0) 

In `convert_to_sn_api_block_info`, this field is mapped directly to `sequencer_address` in the `BlockInfo` that is passed to the batcher: [2](#0-1) 

During fee transfer, `execute_fee_transfer` sends the fee to `block_context.block_info.sequencer_address` — i.e., to `init.builder`: [3](#0-2) 

`is_proposal_init_valid` validates `height`, `l1_da_mode`, `l2_gas_price_fri`, `timestamp`, `starknet_version`, `version_constant_commitment`, and all four L1 gas prices. It does **not** validate `builder`: [4](#0-3) 

The `ProposalInitValidation` struct — the sole source of locally-trusted reference values — contains no `builder` field: [5](#0-4) 

`ProposalInitValidation` is constructed in `validate_proposal` (called from `validate_current_round_proposal`) with no `builder` entry: [6](#0-5) 

The `builder` value from the wire is therefore accepted unconditionally and forwarded to the batcher via `initiate_validation` → `convert_to_sn_api_block_info`: [7](#0-6) 

The final commitment check compares `batcher_block_commitment` (derived from the batcher's execution with the attacker-supplied `builder`) against `received_fin.proposal_commitment` (which the proposer computed using the same `builder`). Both sides agree, so the check passes: [8](#0-7) 

The `proposal_commitment_from` function that produces the commitment binds only `partial_block_hash` and `fee_proposal_fri` — not `builder` directly — but `partial_block_hash` is produced by the batcher after executing with the attacker-supplied `sequencer_address`, so it already encodes the attacker's address: [9](#0-8) 

### Impact Explanation

Every transaction fee in the block is transferred to `init.builder`. A malicious proposer who sets `builder = attacker_address` causes the batcher to execute all fee transfers to the attacker. The resulting `partial_block_hash` encodes the attacker's address; the proposer constructs a matching `ProposalFin`; the validator's commitment comparison passes; consensus finalizes the block. All user-paid fees for the entire block are permanently redirected to the attacker with no on-chain indication of wrongdoing.

**Impact category:** Critical — wrong fee/balance effect with direct economic impact.

### Likelihood Explanation

Requires a malicious or compromised proposer key. This is the direct analog of the external report's Registry/Factory admin: a single compromised consensus-participant key is sufficient. No user interaction is required; the attack is silent and affects every transaction in the block. The proposer role rotates across validators, so any single compromised node can execute the attack during its proposer turn.

### Recommendation

Add an `expected_builder` (or `expected_sequencer_address`) field to `ProposalInitValidation`, populated from the local node's own configuration (not from the incoming `ProposalInit`). In `is_proposal_init_valid`, reject any `init_proposed.builder` that does not match the expected value. Alternatively, enforce `builder == proposer` as a protocol invariant and validate it against the consensus-layer-supplied `proposer` identity.

### Proof of Concept

1. Malicious proposer constructs `ProposalInit { builder: attacker_address, height: H, l1_da_mode: X, l2_gas_price_fri: Y, ... }` with all other fields valid per the validator's checks.
2. Validator receives `ProposalInit`, calls `is_proposal_init_valid` → all checks pass (`builder` is never examined).
3. `initiate_validation` calls `batcher.validate_block` with `block_info.sequencer_address = attacker_address`.
4. Batcher executes all transactions; `execute_fee_transfer` sends every fee to `attacker_address`.
5. Batcher returns `partial_block_hash` that encodes `attacker_address` as sequencer.
6. Validator computes `batcher_block_commitment = proposal_commitment_from(partial_block_hash, fee_proposal_fri)`.
7. Proposer sends `ProposalFin { proposal_commitment: batcher_block_commitment, executed_transaction_count: N, ... }`.
8. Validator checks `built_block == received_fin.proposal_commitment` → equal, passes.
9. `valid_proposals.insert_proposal(args.init, ...)` stores the proposal with `builder = attacker_address`.
10. Consensus reaches decision; block is finalized; all fees for the block are permanently at `attacker_address`.
