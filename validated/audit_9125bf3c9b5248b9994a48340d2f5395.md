### Title
`respond_verify_foreign_tx()` Does Not Verify That `response.payload_hash` Was Derived From the Submitted `request` - (File: crates/contract/src/lib.rs)

### Summary

`MpcContract::respond_verify_foreign_tx()` accepts a `VerifyForeignTransactionResponse` whose `payload_hash` field is never checked against the `VerifyForeignTransactionRequest` it is supposed to resolve. A single malicious MPC participant (strictly below the signing threshold) can obtain a legitimate MPC collective signature for foreign transaction B, then submit it as the response to a different user's pending request for foreign transaction A. The contract accepts the call, resolves the yield, and delivers a cryptographically valid but semantically forged attestation to the waiting caller.

### Finding Description

In `respond_verify_foreign_tx()`, the contract performs two independent checks:

1. It looks up `request` in `pending_verify_foreign_tx_requests` to find the queued yield(s).
2. It verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's **root** public key. [1](#0-0) 

Critically, there is no step that binds `response.payload_hash` to `request`. The `payload_hash` is supposed to be `SHA-256(borsh(ForeignTxSignPayload::V1 { request: request.request, values: <extracted> }))`, meaning it must commit to the original `ForeignChainRpcRequest`. The contract never recomputes or cross-checks this commitment. [2](#0-1) 

Compare this with the regular `respond()` path, where the payload is taken **directly from the stored request**, not from the response: [3](#0-2) 

The `ForeignTxSignPayload` type commits to both the original `ForeignChainRpcRequest` and the extracted values: [4](#0-3) 

Because the contract never verifies that `response.payload_hash` encodes the correct `request`, the two checks (request lookup and signature verification) are completely decoupled.

### Impact Explanation

A malicious MPC participant can deliver a forged foreign-chain verification to any waiting caller. Concretely:

- User Alice submits `verify_foreign_transaction` for Bitcoin tx_id A (e.g., a bridge deposit of 1 BTC). Her yield is queued under `request_A`.
- The malicious node submits its own `verify_foreign_transaction` for Bitcoin tx_id B (a worthless or attacker-controlled transaction). The MPC network signs `payload_hash_B = SHA-256(borsh(ForeignTxSignPayload { request: tx_id_B, values: [...] }))`.
- The malicious node calls `respond_verify_foreign_tx(request_A, { payload_hash: payload_hash_B, signature: sig_B })`.
- The contract finds `request_A` in the pending map (step 1 passes), verifies `sig_B` over `payload_hash_B` under the root key (step 2 passes), and resolves Alice's yield with the forged response.
- Alice's application receives a `VerifyForeignTransactionResponse` with a valid root-key signature, but the `payload_hash` attests to tx_id B, not tx_id A.

Any bridge or application that trusts the returned `VerifyForeignTransactionResponse` as proof that tx_id A was verified will be deceived. This enables invalid bridge execution or double-spend conditions, matching the **High** impact category: *forged foreign-chain verification that causes invalid bridge execution or double-spend conditions*.

### Likelihood Explanation

The attack requires only a single malicious MPC participant who is an attested member of the network — strictly below the signing threshold. No threshold collusion is needed. The malicious node participates honestly in the MPC signing protocol for its own request (tx_id B), obtains the collective signature legitimately, and then misroutes it to a different pending request. The entry path is the public `respond_verify_foreign_tx` method, callable by any attested participant. [5](#0-4) 

### Recommendation

The contract must verify that `response.payload_hash` was derived from the submitted `request`. Since `ForeignTxSignPayload` commits to the `ForeignChainRpcRequest`, the response should carry the full `ForeignTxSignPayload` (including the extracted `values`) so the contract can recompute the hash and assert equality:

```rust
// In respond_verify_foreign_tx, after receiving response.payload and response.signature:
let expected_hash = response.payload.compute_msg_hash()
    .map_err(|_| RespondError::InvalidPayloadHash)?;

// Verify the payload's embedded request matches the pending request
if response.payload.request() != &request.request {
    return Err(RespondError::PayloadRequestMismatch.into());
}

// Verify the hash matches what was signed
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

This mirrors how `respond()` derives the payload directly from the stored request rather than trusting the response.

### Proof of Concept

The existing test `respond_verify_foreign_tx__should_succeed_when_response_is_valid_and_request_exists` demonstrates the normal flow: [6](#0-5) 

To demonstrate the vulnerability, modify the test to use a `payload` constructed from a **different** `ForeignChainRpcRequest` (e.g., `tx_id: [99u8; 32]` instead of `[7u8; 32]`), sign it with the same root key, and submit it as the response to the original request. The contract will accept it and resolve the yield with the mismatched `payload_hash`, confirming that no binding between `request` and `response.payload_hash` is enforced. [7](#0-6)

pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
    }
