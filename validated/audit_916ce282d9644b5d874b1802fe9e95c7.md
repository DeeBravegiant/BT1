Audit Report

## Title
Unguarded `l2bridge` Parameter in `sendETHToL1ViaBridge` Enables Block Stuffing — (`contracts/bridges/UnichainMessenger.sol`)

## Summary
`UnichainMessenger.sendETHToL1ViaBridge` is a public `external payable` function with only a `nonReentrant` guard and a `msg.value == value` check. Any caller can supply an arbitrary `l2bridge` address, including a malicious contract whose `bridgeETHTo` implementation consumes all forwarded gas. By repeatedly submitting such transactions, an attacker can saturate Unichain blocks and prevent the `BRIDGER_ROLE` operator's legitimate bridge transactions from landing. The identical pattern exists in `BaseMessenger`, `OptimismMessenger`, `ScrollMessenger`, and `ArbitrumMessenger`.

## Finding Description
`UnichainMessenger` at [1](#0-0)  exposes `sendETHToL1ViaBridge` with no role check and no validation of the `l2bridge` parameter. The `DEFAULT_GAS_LIMIT = 200_000` constant at [2](#0-1)  is passed as `_minGasLimit` to the callee — it is a protocol-level hint, not an EVM gas cap. The EVM forwards `gasleft() - overhead` to the external call, so a malicious `bridgeETHTo` implementation can consume all forwarded gas.

The intended callers (`RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge`) are correctly gated with `onlyRole(BRIDGER_ROLE)` at [3](#0-2)  and pass a governance-set `l2Bridge` stored in state at [4](#0-3) . However, this access control lives on the pool contract, not on the messenger. An attacker bypasses the pool entirely by calling the messenger directly with an arbitrary `l2bridge`.

The same unguarded pattern is present in `BaseMessenger` at [5](#0-4) , `OptimismMessenger` at [6](#0-5) , `ScrollMessenger` at [7](#0-6) , and `ArbitrumMessenger` at [8](#0-7) .

## Impact Explanation
An attacker can repeatedly call `sendETHToL1ViaBridge(gasBomb, anyAddress, 1)` with `msg.value = 1 wei`, each transaction consuming nearly the full Unichain block gas limit. By submitting enough such transactions per block, the attacker saturates blocks and delays or prevents the `BRIDGER_ROLE` operator's legitimate `bridgeAssetsViaNativeBridge` transaction from being included. This is a **Low — Block Stuffing** impact, which is explicitly listed in the allowed impact scope.

## Likelihood Explanation
The function is publicly callable with no preconditions beyond `msg.value == value` (minimum 1 wei). The attacker only needs to deploy a gas-bomb contract and call the messenger directly. No privileged access, no victim interaction, and no external protocol dependency is required. The cost is the gas fees paid per transaction, making sustained stuffing expensive but feasible for a motivated attacker targeting a specific bridge window.

## Recommendation
Add an access control guard to `sendETHToL1ViaBridge` in `UnichainMessenger`, `BaseMessenger`, `OptimismMessenger`, `ScrollMessenger`, and `ArbitrumMessenger` so only authorized callers (e.g., accounts with `BRIDGER_ROLE` on the corresponding pool, or an immutable allowlist of pool addresses set at deployment) can invoke it. Alternatively, validate `l2bridge` against an immutable or governance-set allowlist of legitimate bridge addresses to prevent arbitrary external calls.

## Proof of Concept
```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import { UnichainMessenger } from "contracts/bridges/UnichainMessenger.sol";

contract GasBomb {
    // Implements IUnichainMessenger.bridgeETHTo — burns all forwarded gas
    function bridgeETHTo(address, uint32, bytes memory) external payable {
        uint256 i;
        while (true) { unchecked { i++; } }
    }
    receive() external payable {}
}

contract PoC {
    function exploit(address messenger) external payable {
        GasBomb bomb = new GasBomb();
        // Call with 1 wei; the gas bomb consumes all remaining block gas
        UnichainMessenger(messenger).sendETHToL1ViaBridge{value: 1}(
            address(bomb), address(0xdead), 1
        );
    }
}
```

Deploy `GasBomb` and `PoC` on a local fork of Unichain, call `exploit` with `value = 1 wei`, and observe that the transaction consumes the full block gas limit. Repeat to demonstrate block saturation preventing the `BRIDGER_ROLE` bridge transaction from landing.
