Audit Report

## Title
Donation Attack via Direct Token Transfer Inflates TVL and Blocks Deposits — (`contracts/LRTDepositPool.sol`)

## Summary
`getAssetDistributionData` and `getETHDistributionData` compute protocol TVL using raw `IERC20.balanceOf` and `address.balance` reads on contracts that accept unrestricted token/ETH transfers. An unprivileged attacker can donate assets directly to `LRTDepositPool`, any `NodeDelegator`, or `LRTUnstakingVault` to inflate `getTotalAssetDeposits` past `depositLimitByAsset`, causing every subsequent `depositAsset`/`depositETH` call to revert with `MaximumDepositLimitReached` until an admin raises the limit. A secondary impact inflates `rsETHPrice`, causing future depositors to receive fewer rsETH tokens per unit deposited.

## Finding Description
**Root cause — raw balance reads in TVL accounting**

`getAssetDistributionData` sums three raw balance reads: [1](#0-0) 

`getETHDistributionData` does the same for native ETH: [2](#0-1) 

All three target contracts expose unrestricted receive paths: [3](#0-2) [4](#0-3) [5](#0-4) 

**Deposit-limit enforcement uses the inflatable total**

`getTotalAssetDeposits` aggregates all raw balance reads: [6](#0-5) 

`_checkIfDepositAmountExceedesCurrentLimit` compares this directly against `depositLimitByAsset`: [7](#0-6) 

`_beforeDeposit` reverts with `MaximumDepositLimitReached` if the check returns true: [8](#0-7) 

**Secondary impact — rsETH price inflation**

`updateRSETHPrice()` is a public, permissionless function: [9](#0-8) 

It calls `_getTotalEthInProtocol`, which calls `getTotalAssetDeposits` for every supported asset: [10](#0-9) 

A donation inflates `totalETHInProtocol`, raises `rsETHPrice`, and causes subsequent depositors to receive fewer rsETH tokens: [11](#0-10) 

There are no existing guards (e.g., internal accounting variables, deposit-only balance tracking, or donation rejection logic) that prevent raw balance inflation from affecting the TVL sum.

## Impact Explanation
**Primary — Medium: Temporary freezing of funds (deposit functionality)**
An attacker transfers `depositLimitByAsset(asset) − currentTotalAssetDeposits + 1` tokens directly to `LRTDepositPool`. Every subsequent `depositAsset` call for that asset reverts with `MaximumDepositLimitReached`. The freeze persists until an admin raises the deposit limit; no admin key compromise is required. The donated tokens remain permanently counted in `getTotalAssetDeposits` even after operators move them onward (they still count toward the limit).

**Secondary — Low: Contract fails to deliver promised returns**
After the donation, any caller invoking the public `updateRSETHPrice()` permanently raises the stored `rsETHPrice`. All future depositors receive fewer rsETH tokens than they would have without the donation, even though the ETH-denominated value of their deposit is unchanged. The donated tokens are absorbed into protocol TVL with no corresponding rsETH minted to the donor.

## Likelihood Explanation
The attack requires no special role, no front-running, and no oracle manipulation — only a direct ERC20 transfer or ETH send to a publicly accessible contract. Deposit limits are routinely set close to current TVL to manage concentration risk; when the gap is small, the attack cost is proportionally low. The attack is repeatable: each time an admin raises the limit, the attacker can donate again to re-exceed it.

## Recommendation
Replace raw `balanceOf`/`address.balance` reads with internal accounting variables that are incremented only through the official deposit entry points (`depositAsset`, `depositETH`, `transferAssetToNodeDelegator`, etc.) and decremented on withdrawals. This mirrors the `UniswapV2Pair` reserve-tracking pattern and eliminates the donation surface entirely. Alternatively, introduce a `sync`-style function restricted to operators that explicitly reconciles internal accounting with actual balances, preventing unprivileged inflation.

## Proof of Concept
```solidity
// Assume: depositLimitByAsset(stETH) = 10_000e18
//         getTotalAssetDeposits(stETH) = 9_999e18  (1 stETH gap)

// Step 1 — attacker donates 2 stETH directly to LRTDepositPool
IERC20(stETH).transfer(address(lrtDepositPool), 2e18);

// Step 2 — getTotalAssetDeposits(stETH) now returns 10_001e18 > 10_000e18

// Step 3 — any legitimate user deposit now reverts
lrtDepositPool.depositAsset(stETH, 1e18, 0, "");
// → reverts: MaximumDepositLimitReached

// Step 4 — attacker (or anyone) calls public updateRSETHPrice()
lrtOracle.updateRSETHPrice();
// → rsETHPrice is now inflated; future depositors receive fewer rsETH tokens

// Step 5 — same attack works for ETH via direct send to LRTDepositPool
// (address(lrtDepositPool).balance is read raw in getETHDistributionData)
```

**Foundry fork test plan**: Fork mainnet, snapshot current `getTotalAssetDeposits(stETH)` and `depositLimitByAsset(stETH)`, transfer the gap amount + 1 wei of stETH directly to `LRTDepositPool`, assert `depositAsset` reverts, then call `updateRSETHPrice()` and assert `rsETHPrice` increased and `getRsETHAmountToMint` returns a lower value for the same deposit amount.
