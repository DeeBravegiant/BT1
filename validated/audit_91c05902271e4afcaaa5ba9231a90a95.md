Audit Report

## Title
Donation to EigenLayer Strategy Inflates TVL, Causing Unearned Protocol Fee rsETH Minting — (`contracts/NodeDelegatorHelper.sol`, `contracts/LRTOracle.sol`)

## Summary

An unprivileged attacker can transfer underlying tokens directly to an EigenLayer `StrategyBase` contract, inflating `sharesToUnderlyingView`'s exchange rate. Because `NodeDelegatorHelper.getAssetBalance` and `NodeDelegator.getAssetUnstaking` both consume this rate without any sanity bound, the inflated value propagates through `getTotalAssetDeposits` → `_getTotalEthInProtocol` → `_updateRsETHPrice`, causing the protocol to treat the donation as genuine staking yield and mint excess protocol-fee rsETH to the treasury, diluting existing rsETH holders.

## Finding Description

**Root cause — `sharesToUnderlyingView` is manipulable by direct token transfer**

EigenLayer's `StrategyBase.sharesToUnderlyingView` computes:
```
shares * underlyingToken.balanceOf(address(strategy)) / totalShares
```
Sending tokens directly to the strategy address increases `balanceOf(strategy)` without minting new shares, inflating the per-share exchange rate. This is permissionless and irreversible.

**Step 1 — `getAssetBalance` reads the inflated rate without a sanity check**

`NodeDelegatorHelper.getAssetBalance` fetches the NDC's withdrawable shares from `DelegationManager.getWithdrawableShares`, then multiplies by the now-inflated `sharesToUnderlyingView` with no independent price oracle or deviation bound: [1](#0-0) 

**Step 2 — `getAssetUnstaking` also reads the inflated rate**

`NodeDelegator.getAssetUnstaking` similarly calls `strategy.sharesToUnderlyingView(sharesToUnstake)` for every queued withdrawal, amplifying the TVL inflation further: [2](#0-1) 

**Step 3 — Inflated values accumulate into `getTotalAssetDeposits`**

Both `assetStakedInEigenLayer` and `assetUnstakingFromEigenLayer` are inflated and summed: [3](#0-2) [4](#0-3) 

**Step 4 — Oracle treats the TVL inflation as yield and mints fee rsETH**

`_updateRsETHPrice` computes `totalETHInProtocol` via `_getTotalEthInProtocol`, compares it to `previousTVL = rsethSupply * rsETHPrice`, and treats any positive delta as `rewardAmount`: [5](#0-4) 

The fee is then minted as rsETH to the treasury: [6](#0-5) 

**Step 5 — `updateRSETHPrice` is public with no access control**

Any address can trigger the price update and fee minting: [7](#0-6) 

**Why existing guards fail**

- `pricePercentageLimit` guard: Only activates when `pricePercentageLimit > 0` AND the price increase exceeds the threshold. If the limit is unset (default `0`) or the donation is sized to stay within the threshold, the guard is bypassed entirely: [8](#0-7) 

- `maxFeeMintAmountPerDay`: When set to a non-zero value (required for any legitimate fee collection), this only caps the per-day impact; it does not prevent the attack. An attacker can repeat the attack across multiple days or size the donation to stay within the daily cap: [9](#0-8) 

## Impact Explanation

**High — Theft of unclaimed yield.** The treasury receives rsETH representing a protocol fee on a donation that is not genuine staking yield. Existing rsETH holders bear the dilution: instead of the full donation accruing to their rsETH value, `feeRate%` is permanently siphoned to the treasury. The donated tokens remain in the strategy and cannot be recovered by the attacker, but the fee portion is irreversibly transferred from rsETH holders to the treasury.

## Likelihood Explanation

**Low.** The attacker must sacrifice the donated tokens (permanently added to the strategy's TVL). The attack is not self-profitable; it is a griefing/dilution vector. Realistic motivations include a large short position on rsETH or a competitor willing to absorb the cost. The attack is repeatable across days (bounded only by `maxFeeMintAmountPerDay` per period) and requires no special permissions — any EOA can execute it.

Preconditions:
- `maxFeeMintAmountPerDay > 0` (required for any legitimate fee collection to be enabled)
- `protocolFeeInBPS > 0` (protocol must have a non-zero fee rate)
- Protocol not paused

## Recommendation

1. **Track deposited shares internally.** Record shares at deposit time and convert to underlying using a trusted external price oracle rather than the live `sharesToUnderlyingView` rate. This decouples TVL accounting from the manipulable strategy balance.
2. **Alternatively**, compare `sharesToUnderlyingView` against an external price oracle before computing fees; revert or skip fee minting if the deviation exceeds a configured threshold.
3. **Ensure `pricePercentageLimit` is always set to a non-zero value** in deployment configuration to provide a partial circuit-breaker against large single-block donations.

## Proof of Concept

```solidity
// Mainnet fork test — no public-mainnet state changes
function testDonationInflatesProtocolFee() public {
    // 1. Record baseline treasury balance
    uint256 treasuryBefore = rsETH.balanceOf(treasury);

    // 2. Attacker donates stETH directly to the EigenLayer stETH strategy
    //    (increases balanceOf(strategy) without changing totalShares)
    address strategy = lrtConfig.assetStrategy(stETH);
    deal(stETH, strategy, IERC20(stETH).balanceOf(strategy) + 100 ether);

    // 3. Anyone calls the public updateRSETHPrice — no role required
    lrtOracle.updateRSETHPrice();

    // 4. Treasury received excess rsETH beyond what genuine yield justifies
    uint256 treasuryAfter = rsETH.balanceOf(treasury);
    assertGt(treasuryAfter, treasuryBefore,
        "treasury minted excess rsETH from donation");
}
```

The test donates tokens directly to the strategy (inflating `sharesToUnderlyingView`), calls the public `updateRSETHPrice`, and asserts that the treasury received rsETH minted as a protocol fee on the artificial TVL increase — confirming the invariant violation.
