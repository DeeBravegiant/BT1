Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` distributes reward tokens over a fixed period set by `notifyRewardAmount`. If all stakers withdraw after a reward period starts, `totalKernelStaked` drops to zero. During any interval where `totalKernelStaked == 0`, the reward accumulation logic freezes and `updatedAt` is silently advanced, permanently discarding the reward tokens allocated to that interval. No on-chain recovery path exists.

## Finding Description
The root cause is a two-part interaction in the `updateReward` modifier and `rewardPerToken()`:

**`rewardPerToken()` (L408–414):** When `totalKernelStaked == 0`, it short-circuits and returns the frozen `rewardPerTokenStored` without advancing the accumulator. This means `rewardRate × elapsed_time` worth of tokens is silently skipped.

**`updateReward` modifier (L232–242):** It unconditionally sets `updatedAt = lastTimeRewardApplicable()` regardless of whether `totalKernelStaked` is zero. So even during a zero-staking gap, `updatedAt` advances to the current time. When the next staker arrives and triggers `updateReward`, `totalKernelStaked` is still 0 at modifier entry (the increment happens after the modifier in `stake()`), so `rewardPerToken()` again returns the frozen value and `updatedAt` is advanced past the gap. The entire gap's reward allocation — `rewardRate × gap_duration` — is irrecoverably lost.

**`notifyRewardAmount` (L569–570):** The guard `if (totalKernelStaked == 0) revert NoStakedTokens()` only prevents starting a period with zero stakers. It does not prevent stakers from exiting after the period starts.

**No recovery function:** The contract's only admin functions are `setRewardsDuration`, `notifyRewardAmount`, `setWithdrawalDelay`, and `setMaxNumberOfWithdrawalsPerUser` (L552–620). There is no `recoverERC20`, sweep, or any other mechanism to retrieve stranded reward tokens.

The contract's own NatSpec (L18–22) explicitly acknowledges this: *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract"* — but the stated mitigation is purely operational (off-chain), with no on-chain enforcement.

## Impact Explanation
Reward tokens deposited by the admin via `notifyRewardAmount` are permanently frozen in the contract for any time interval during which `totalKernelStaked == 0`. No user can claim them, and no admin function can recover them. This is **permanent freezing of unclaimed yield** (Medium severity per the allowed impact scope).

## Likelihood Explanation
Any staker can call `initiateWithdrawal` at any time, which is a normal, permissionless user action. If the last remaining staker exits during an active reward window — requiring no privileged access, no coordination, and no front-running — the remaining rewards are locked. This is plausible during market stress, protocol migration, or simply organic user behavior. The likelihood is medium.

## Recommendation
Add an admin-accessible token recovery function that can only sweep the difference between the contract's reward token balance and the total of all currently earned (but unclaimed) rewards:

```solidity
function recoverUnallocatedRewards(address recipient) external onlyRole(DEFAULT_ADMIN_ROLE) {
    require(block.timestamp > finishAt, "Reward period not finished");
    uint256 unallocated = rewardsToken.balanceOf(address(this)) - totalEarnedButUnclaimed();
    rewardsToken.safeTransfer(recipient, unallocated);
}
```

Alternatively, enforce the operational invariant on-chain by preventing `initiateWithdrawal` from reducing `totalKernelStaked` to zero during an active reward period, or by calling `notifyRewardAmount` with the leftover balance immediately after the period ends to roll unallocated rewards into the next period.

## Proof of Concept

1. Admin calls `setRewardsDuration(7 days)`.
2. Alice stakes 100 KERNEL → `totalKernelStaked = 100`.
3. Admin calls `notifyRewardAmount(1000e18)` → `rewardRate = 1000e18 / 7 days`, `finishAt = now + 7 days`.
4. After 1 day, Alice calls `initiateWithdrawal(100)` → `totalKernelStaked = 0`; `updateReward` runs, advancing `updatedAt` to `block.timestamp`.
5. After `withdrawalDelay`, Alice calls `claimWithdrawal` and receives her 100 KERNEL.
6. Alice calls `getReward()` and receives ~142e18 reward tokens (1 day worth).
7. Days 2–7 pass with `totalKernelStaked == 0`. Each time `updateReward` is triggered (e.g., by any call), `updatedAt` advances but `rewardPerTokenStored` stays frozen.
8. ~857e18 reward tokens remain in the contract. No function exists to recover them. Any future staker who joins before `finishAt` will earn rewards only from the moment they stake; the 6-day gap's allocation is irrecoverable.

**Foundry test sketch:**
```solidity
function test_rewardsLockedOnZeroStake() public {
    vm.prank(alice);
    pool.stake(100e18);
    vm.prank(admin);
    pool.notifyRewardAmount(1000e18);
    vm.warp(block.timestamp + 1 days);
    vm.prank(alice);
    pool.initiateWithdrawal(100e18);
    vm.warp(block.timestamp + withdrawalDelay);
    vm.prank(alice);
    pool.claimWithdrawal(1);
    vm.prank(alice);
    pool.getReward();
    vm.warp(block.timestamp + 6 days); // reward period ends
    // Assert: ~857e18 reward tokens remain in contract, no function to recover
    assertGt(rewardsToken.balanceOf(address(pool)), 800e18);
}
```
