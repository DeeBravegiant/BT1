Looking at the `ratifyTransition` function in `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`, I can identify a boundary condition bug directly analogous to the raffle report.

The external report's root cause: `supply > minTicketsThreshold` should be `supply >= minTicketsThreshold` — an off-by-one that allows two conflicting state transitions at the exact boundary value.

In Cardano Ledger, the same class of bug exists in the governance action expiry check.

---

### Title
Governance Action Expiry Off-by-One Allows Ratification One Epoch Past Intended Deadline - (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

### Summary

In `ratifyTransition`, the check that marks a governance action as expired uses strict less-than (`<`) instead of less-than-or-equal (`<=`). This is inconsistent with every other expiry check in the codebase, causing governance actions to survive one extra epoch boundary in a "limbo" state: they cannot receive new votes, but they are also not expired and can still be ratified. When a delaying action is enacted at the same epoch boundary as an expiring governance action, the expiring action escapes expiry and can be enacted one epoch after its intended deadline.

### Finding Description

**Root cause — `ratifyTransition`:**

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs, line 357
if gasExpiresAfter < reCurrentEpoch          -- BUG: should be <=
  then pure $ st' & rsExpiredL %~ Set.insert gasId
  else pure st'
``` [1](#0-0) 

The `gasExpiresAfter` field is named "expires after epoch N", meaning the action is valid through epoch `N` and should be expired at the first epoch boundary where `reCurrentEpoch >= N`. The RATIFY rule runs with `reCurrentEpoch = eNo - 1` (the epoch that just ended), so at the boundary entering epoch `gasExpiresAfter + 1`, `reCurrentEpoch == gasExpiresAfter`. The strict `<` check evaluates to `False`, so the action is **not** expired.

**Contrast with every other expiry check in the codebase:**

`checkVotesAreNotForExpiredActions` (GOV rule) uses `<=`:
```haskell
curEpoch <= gasExpiresAfter   -- voting allowed when curEpoch ≤ gasExpiresAfter
``` [2](#0-1) 

`updateNumDormantEpochs` (EPOCH rule) uses `<=`:
```haskell
OMap.filter ((currentEpoch <=) . gasExpiresAfter) $ ps ^. pPropsL
``` [3](#0-2) 

`committeeAcceptedRatio` uses `>` (equivalent to `not <=`):
```haskell
| currentEpoch > expiry = (yes, tot) -- member is expired
``` [4](#0-3) 

`dRepAcceptedRatio` uses `>`:
```haskell
| reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired
``` [5](#0-4) 

All other expiry checks treat `currentEpoch == expiryEpoch` as **expired**. Only `ratifyTransition` treats it as **not expired**.

**The "limbo" epoch:**

For a governance action with `gasExpiresAfter = E` and `reCurrentEpoch = eNo - 1`:

| Epoch boundary entered | `reCurrentEpoch` | `gasExpiresAfter < reCurrentEpoch` | Result |
|---|---|---|---|
| `E+1` | `E` | `E < E` → **False** | NOT expired (bug) |
| `E+2` | `E+1` | `E < E+1` → True | Expired |

During epoch `E+1`, `checkVotesAreNotForExpiredActions` correctly blocks new votes (`E+1 > E`). But the action is still in the proposals set and can be ratified at the `E+2` boundary using votes accumulated before epoch `E+1`.

**Critical interaction with `rsDelayed`:**

`ratifyTransition` only checks expiry in the `else` branch — i.e., only when the action is **not** ratified:

```haskell
if acceptedByEveryone env st gas
  then do
    -- enact
  else do
    st' <- trans @(RATIFY era) $ TRC (env, st, RatifySignal sigs)
    if gasExpiresAfter < reCurrentEpoch   -- expiry only checked here
      then pure $ st' & rsExpiredL %~ Set.insert gasId
      else pure st'
``` [6](#0-5) 

Delaying actions (`NoConfidence`, `UpdateCommittee`, `NewConstitution`, `HardForkInitiation`) set `rsDelayed = True`, which prevents any other action from being ratified in the same epoch boundary pass. These actions have the highest priority in `reorderActions` and are processed first:

```haskell
actionPriority NoConfidence {} = 0
actionPriority UpdateCommittee {} = 1
actionPriority NewConstitution {} = 2
actionPriority HardForkInitiation {} = 3
``` [7](#0-6) 

**Attack path:**

1. Attacker submits a `ParameterChange` or `TreasuryWithdrawals` governance action with `gasExpiresAfter = E`. It accumulates sufficient votes during epoch `E`.
2. At the epoch `E+1` boundary, a delaying action (e.g., `UpdateCommittee`) is also ratified. It is processed first, setting `rsDelayed = True`.
3. The attacker's action is **not** ratified (blocked by `rsDelayed = True`).
4. The attacker's action is **not** expired (bug: `E < E` is False).
5. During epoch `E+1`, no new votes can be cast on the action (correctly blocked by `checkVotesAreNotForExpiredActions`).
6. At the epoch `E+2` boundary, `rsDelayed` is reset to `False`. The attacker's action is ratified using the votes from epoch `E`.

The action is enacted one epoch after its intended expiry deadline.

**Secondary impact — incorrect dormant epoch accounting:**

`updateNumDormantEpochs` uses `currentEpoch <= gasExpiresAfter` with `currentEpoch = eNo` (the epoch being entered). At the `E+1` boundary, `E+1 <= E` is False, so the action is not counted as active and the dormant epoch counter is incremented — even though the action is still in the proposals set. This incorrectly extends DRep expiry by one epoch. [8](#0-7) 

### Impact Explanation

**Allowed impact matched:** *Critical. Unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork action is enacted.*

A `ParameterChange`, `TreasuryWithdrawals`, `UpdateCommittee`, `NewConstitution`, `HardForkInitiation`, or `NoConfidence` governance action that should have been expired can instead be enacted one epoch past its intended deadline. This constitutes an unauthorized governance state transition — the ledger accepts an enactment that the protocol rules were designed to prevent.

Secondary: *Medium. Attacker-controlled transactions modify treasury withdrawals or protocol parameters outside design parameters.*

### Likelihood Explanation

The trigger requires two conditions to coincide at the same epoch boundary:
1. A governance action reaches `gasExpiresAfter` with sufficient votes to be ratified but is not yet enacted.
2. A delaying action is also ratified at the same boundary, setting `rsDelayed = True`.

Delaying actions are common governance operations (committee elections, hard fork initiations, constitution updates). The combination is realistic in an active governance environment. An attacker who controls enough stake to ratify a `ParameterChange` or `TreasuryWithdrawals` action can time its submission to expire exactly when a known delaying action is expected to be enacted, exploiting the one-epoch window.

### Recommendation

Change the strict less-than to less-than-or-equal in `ratifyTransition`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs
- if gasExpiresAfter < reCurrentEpoch
+ if gasExpiresAfter <= reCurrentEpoch
    then pure $ st' & rsExpiredL %~ Set.insert gasId
    else pure st'
```

This aligns the expiry check with `checkVotesAreNotForExpiredActions`, `updateNumDormantEpochs`, `committeeAcceptedRatio`, and `dRepAcceptedRatio`, all of which treat `currentEpoch == expiryEpoch` as expired.

### Proof of Concept

Given:
- `ppGovActionLifetimeL = EpochInterval 1`
- Proposal submitted in epoch 0 → `gasExpiresAfter = 1`
- Sufficient DRep + SPO + committee votes cast in epoch 1
- A `NewConstitution` action also ratified at the epoch 2 boundary (sets `rsDelayed = True`)

**Current behavior:**
- Epoch 2 boundary: `reCurrentEpoch = 1`, `1 < 1` → False → action NOT expired, NOT ratified (blocked by `rsDelayed`)
- Epoch 3 boundary: `reCurrentEpoch = 2`, `1 < 2` → True → action expired (never enacted)

Wait — the action is expired at epoch 3, not enacted. The window for ratification is only at epoch 2 boundary. But `rsDelayed` blocks it there. So the action is expired at epoch 3 without being enacted.

**Revised PoC — the actual exploit window:**

The action can be ratified at the epoch 2 boundary if `rsDelayed = False` at that point. The `rsDelayed` flag is set only when a delaying action is enacted *before* the target action in the same boundary pass. If the target action has higher priority than the delaying action (impossible given the priority ordering), or if the delaying action was enacted in the *previous* epoch boundary (setting `rsDelayed = True` which persists into the current boundary), the target action is blocked.

More precisely: `rsDelayed` is reset to `False` at the start of each epoch boundary, then set to `True` if a delaying action is enacted during that boundary. So the scenario is:

- Epoch `E+1` boundary: delaying action enacted → `rsDelayed = True` → target action not ratified, not expired (bug)
- Epoch `E+2` boundary: `rsDelayed = False` (reset) → target action **ratified** using votes from epoch `E`

This is the valid exploit path. The target action is ratified at epoch `E+2` using votes that were cast before epoch `E+1`, one full epoch after the intended expiry.
