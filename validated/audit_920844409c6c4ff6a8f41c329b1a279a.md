Audit Report

## Title
Deposit Limit Checked at Execution Time With No Reservation Enables Block Stuffing to Revert Victim's `depositAsset` - (File: contracts/LRTDepositPool.sol)

## Summary
`depositAsset` delegates its limit check to `_beforeDeposit` → `_checkIfDepositAmountExceedesCurrentLimit`, which reads live on-chain state at execution time with no deadline or reservation mechanism. An attacker can observe a victim's pending deposit in the mempool, use block stuffing to delay it, and simultaneously consume the remaining deposit capacity, causing the victim's transaction to revert with `MaximumDepositLimitReached`.

## Finding Description
`depositAsset` calls `_beforeDeposit` at line 111, which calls `_checkIfDepositAmountExceedesCurrentLimit` at line 661. The check at line 681 evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)` against live state at execution time. There is no deadline parameter, no slippage guard on deposit-limit availability, and no reservation mechanism.

Attack path:
1. Deposit limit for stETH is `X`; current total deposits are `X - delta`.
2. Victim submits `depositAsset(stETH, delta, ...)` — valid at submission time.
3. Attacker observes the pending transaction in the mempool and begins block stuffing: submitting high-gas-price filler transactions to consume block space and delay the victim's transaction.
4. Concurrently, the attacker submits `depositAsset(stETH, delta, ...)` at higher gas price to consume the remaining `delta` capacity.
5. When the victim's transaction finally executes, `getTotalAssetDeposits(stETH) + delta > X` evaluates to `true` at line 681, and `_beforeDeposit` reverts with `MaximumDepositLimitReached` at line 662.
6. The `safeTransferFrom` at line 114 is never reached; no funds are transferred or frozen.

Existing guards are insufficient: `minRSETHAmountExpected` guards against price slippage but not deposit-limit availability; `nonReentrant` and `whenNotPaused` are irrelevant to this race condition.

## Impact Explanation
The victim's `depositAsset` transaction reverts. No funds are lost or frozen since the token transfer at line 114 is never executed. The concrete impact is denial of deposit access while the limit remains full, matching **Low. Block stuffing** from the allowed impact scope.

## Likelihood Explanation
Block stuffing on Ethereum mainnet requires paying for ~30M gas per block at prevailing prices, making it expensive. The attacker also needs sufficient stETH to fill the remaining limit. The attacker receives rsETH for their own deposits (obtainable without the attack) but pays substantial extra gas for stuffing, making the attack economically irrational in most scenarios. Likelihood is low but the attack is technically feasible and repeatable whenever deposit limits are near capacity.

## Recommendation
1. Add a `deadline` parameter to `depositAsset` and revert if `block.timestamp > deadline`, bounding how long a transaction can remain pending.
2. Alternatively, add a `minRemainingCapacity` parameter so users can specify the minimum deposit-limit headroom they require at execution time, reverting with a user-friendly error if capacity has been consumed since signing.
3. For competitive deposit limits, consider a commit-reveal or queue-based reservation scheme.

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
