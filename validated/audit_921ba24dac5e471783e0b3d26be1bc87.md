### Title
Fee-on-Transfer Token Accounting Divergence Creates Unbacked Cross-Chain Supply — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter verbatim in the cross-chain message after calling `safeTransferFrom`, without measuring the actual tokens received. For fee-on-transfer ERC20 tokens, the bridge locks fewer tokens than it reports, creating unbacked wrapped supply on the destination chain.

### Finding Description

In `initTransfer`, when the token is a plain ERC20 (not a bridge token and not a custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

Immediately after, the unmodified `amount` is passed to `initTransferExtension` and emitted in `InitTransfer`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← caller-supplied, not measured
    ...
);
``` [2](#0-1) 

In `OmniBridgeWormhole`, `initTransferExtension` encodes this same `amount` into the Wormhole VAA payload that is relayed to NEAR:

```solidity
Borsh.encodeUint128(amount),
``` [3](#0-2) 

No before/after balance snapshot is taken anywhere in `initTransfer`. [4](#0-3) 

### Impact Explanation

For a fee-on-transfer token with a `k%` fee:

- EVM bridge receives `amount × (1 − k)` tokens.
- NEAR side credits the user with `amount` wrapped tokens (full, unbacked amount).
- The bridge's EVM reserve is now undercollateralized by `amount × k` tokens per transfer.
- When any user redeems wrapped tokens back to EVM via `finTransfer`, the bridge calls `safeTransfer(recipient, amount)` but holds less than `amount`, causing either a revert (permanently freezing redemptions for other users) or draining tokens belonging to other depositors.

This matches **Critical** (irreversible fund lock / frozen redemption path) and **High** (balance-accounting divergence that breaks backing guarantees).

### Likelihood Explanation

The entry point is fully unprivileged — any address can call `initTransfer` with any ERC20 token address. No allowlist restricts which tokens can be deposited. Fee-on-transfer tokens (PAXG, STA, and future USDC/USDT variants) are real, deployed mainnet assets. A single deposit of such a token is sufficient to trigger the divergence.

### Recommendation

Measure the actual received amount using a before/after balance check:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived instead of amount for all downstream accounting
```

Alternatively, maintain an explicit token allowlist (admin-controlled) so only vetted, non-rebasing, non-fee-on-transfer tokens can be bridged.

### Proof of Concept

1. Deploy or use a mainnet fee-on-transfer ERC20 token `FEE` with a 10% transfer fee.
2. Approve the `OmniBridge` for 100 `FEE` tokens.
3. Call `initTransfer(FEE, 100, 0, 0, "alice.near", "")`.
4. Bridge receives 90 `FEE` tokens; Wormhole VAA encodes `amount = 100`.
5. NEAR relayer processes the VAA and mints 100 wrapped `FEE` to `alice.near`.
6. Alice calls `initTransfer` on NEAR to bridge 100 wrapped `FEE` back to EVM.
7. `finTransfer` on EVM attempts `safeTransfer(alice_evm, 100)` — bridge only holds 90 → reverts.
8. Alice's 100 wrapped tokens are now unredeemable; the 10-token shortfall accumulates with every deposit, eventually freezing all redemptions.
