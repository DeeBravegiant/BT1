Audit Report

## Title
Permanent Freezing of Unclaimed Yield When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
In `KernelDepositPool`, when all stakers exit during an active reward window, `totalKernelStaked` reaches zero and `rewardPerToken()` short-circuits, permanently discarding all reward tokens that should have accrued for the remainder of the period. No on-chain enforcement prevents full withdrawal during an active period, and no admin recovery path exists, leaving those reward tokens permanently locked in the contract.

## Finding Description
`rewardPerToken()` returns `rewardPerTokenStored` unchanged whenever `totalKernelStaked == 0`: [1](#0-0) 

The `updateReward` modifier advances `updatedAt` to `lastTimeRewardApplicable()` on every user-triggered action: [2](#0-1) 

When the last staker calls `initiateWithdrawal`, the modifier fires correctly — crediting that staker's earned rewards and setting `updatedAt = block.timestamp` — then `totalKernelStaked` is decremented to zero: [3](#0-2) 

From that point forward, every call to `rewardPerToken()` hits the `totalKernelStaked == 0` branch and returns the stale `rewardPerTokenStored`. `updatedAt` is never advanced again because no user action fires `updateReward`. The elapsed time (`lastTimeRewardApplicable() - updatedAt`) accumulates silently but is never applied. When a new staker eventually arrives, `rewardPerToken()` is called again — but the zero-stake gap is still silently discarded, not rolled forward.

The only existing guard is in `notifyRewardAmount`, which reverts if `totalKernelStaked == 0` at the time of a new reward deposit: [4](#0-3) 

This guard prevents starting a new period with no stakers, but does nothing to prevent stakers from exiting during an already-active period. There is no minimum-stake floor, no lock on withdrawals during an active reward window, and no admin sweep or rescue function anywhere in the contract. The NatSpec explicitly acknowledges this limitation and relies entirely on an off-chain operational commitment: [5](#0-4) 

## Impact Explanation
Reward tokens deposited via `notifyRewardAmount` that correspond to the zero-stake interval are permanently irrecoverable. They remain in the contract's `rewardsToken` balance but can never be claimed by any user and cannot be extracted by any admin function. This is a concrete, on-chain **permanent freezing of unclaimed yield**, matching the allowed Medium impact class.

## Likelihood Explanation
The scenario requires all stakers to exit during an active reward period — a realistic outcome during a sharp decline in KERNEL token value, where rational stakers simultaneously withdraw. The last staker to call `initiateWithdrawal` unilaterally causes the loss; the function is permissionless and has no guard against this outcome. The loss falls on the protocol (locked reward tokens), not on the exiting user. The scenario is repeatable across every reward period.

## Recommendation
Add an admin-callable `rescueUnallocatedRewards(address recipient)` function gated on `totalKernelStaked == 0 && block.timestamp >= finishAt`, which transfers the excess reward token balance back to the protocol. Alternatively, track cumulative "dead time" during which `totalKernelStaked == 0` and deduct it from the effective reward window so unallocated rewards roll forward into the next period rather than being silently discarded.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` with `duration = 7 days`; `rewardRate ≈ 1_653e12` tokens/second, `finishAt = T + 7 days`.
2. Alice is the sole staker: `balanceOf[Alice] = 100e18`, `totalKernelStaked = 100e18`.
3. At `T + 3.5 days`, Alice calls `initiateWithdrawal(100e18)`. The `updateReward` modifier fires: `rewardPerTokenStored` is advanced, `updatedAt = T + 3.5 days`, `totalKernelStaked = 0`.
4. For the remaining 3.5 days, every call to `rewardPerToken()` returns the stale `rewardPerTokenStored` (zero-stake branch). `updatedAt` is never advanced.
5. Alice calls `getReward()` and claims her ~500e18 tokens (for the first 3.5 days). The remaining ~500e18 tokens sit in the contract permanently — no user can claim them, no admin function can recover them.

**Foundry test plan:**
```solidity
function test_rewardsLockedOnZeroStake() public {
    // Setup: mint reward tokens, stake as Alice, start reward period
    vm.prank(admin);
    pool.notifyRewardAmount(1_000e18);

    // Fast-forward 3.5 days, Alice withdraws all
    vm.warp(block.timestamp + 3.5 days);
    vm.prank(alice);
    pool.initiateWithdrawal(100e18);

    // Fast-forward past finishAt
    vm.warp(block.timestamp + 4 days);

    // Alice claims her earned rewards (~500e18)
    vm.prank(alice);
    pool.getReward();

    // Assert ~500e18 reward tokens remain locked in contract with no claimant
    uint256 locked = rewardsToken.balanceOf(address(pool));
    assertGt(locked, 490e18); // ~500e18 permanently locked
    assertEq(pool.earned(alice), 0);
    // No admin function exists to recover `locked`
}
```
