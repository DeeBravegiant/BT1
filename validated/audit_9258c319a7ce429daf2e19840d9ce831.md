### Title
Accepted `EvmFinality::Latest` Bypasses Finality Enforcement, Enabling MPC Signature Over Reorgable Transactions — (`crates/foreign-chain-inspector/src/evm/inspector.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`)

### Summary

The MPC network accepts `EvmFinality::Latest` in `verify_foreign_transaction` requests for all EVM-family chains (including Abstract, BNB, Base, Arbitrum, Polygon, HyperEVM). The `verify_finality_level` check with `Latest` is trivially satisfied by any included transaction, because the "latest" block head is always at or past any block that contains a receipt. No guard anywhere in the production path rejects `Latest` finality before signing. The MPC network will therefore produce a valid threshold signature over a `ForeignTxSignPayload` for a zero-confirmation, reorgable transaction.

### Finding Description

**Step 1 — Attacker-controlled entry point.**

An unprivileged caller submits `verify_foreign_transaction` with `request = ForeignChainRpcRequest::Abstract(EvmRpcRequest { finality: EvmFinality::Latest, ... })`. The contract accepts this without restriction. [1](#0-0) 

**Step 2 — Conversion: `EvmFinality::Latest` → `EthereumFinality::Latest`.**

In `execute_foreign_chain_request`, the Abstract chain arm converts the DTO finality via `TryFrom`. The conversion explicitly maps `EvmFinality::Latest` → `EthereumFinality::Latest` with no error. [2](#0-1) [3](#0-2) 

**Step 3 — `verify_finality_level` with `Latest` always passes.**

`EthereumFinality::Latest` maps to `FinalityTag::Latest`, which causes `eth_getBlockByNumber("latest")` to be called. The returned head block number is always ≥ any receipt block number (the latest block is always at or past any included transaction's block). The guard `if head.number < receipt_block_number` therefore never fires. [4](#0-3) 

**Step 4 — `verify_block_is_canonical` does not protect against reorgs.**

This check only confirms that the RPC's view of the canonical chain at query time matches the receipt's block hash. It does not prevent signing for a block that has not yet been finalized or made safe. A transaction in the most recent block passes both checks. [5](#0-4) 

**Step 5 — MPC signs the payload.**

After `extract` returns `Ok`, `build_signature_request` hashes the `ForeignTxSignPayload` (which embeds the full request including `finality=Latest`) and the MPC threshold-signs it. [6](#0-5) [7](#0-6) 

**No guard exists anywhere in the production path** that rejects `EthereumFinality::Latest` before signing. The `EthereumFinality` enum explicitly includes `Latest` as a valid variant. [8](#0-7) 

### Impact Explanation

The MPC network produces a valid threshold signature attesting that a transaction occurred on the foreign chain, even when that transaction has zero confirmations and is trivially reorgable. A bridge contract that relies on the MPC signature as a finality attestation (without independently re-checking the finality level embedded in the signed payload) will process the bridge operation. If the underlying transaction is subsequently reorged, the attacker retains both the bridge output and the original funds — a classic double-spend.

Impact category: **High** — forged foreign-chain verification / verification bypass causing double-spend conditions.

### Likelihood Explanation

- The `EvmFinality::Latest` variant is a valid, serializable, accepted value in the public contract ABI (confirmed in the ABI snapshot).
- No privileged access is required; any caller can submit `verify_foreign_transaction`.
- The Abstract chain (and all other EVM-family chains) share the same `EvmInspector` code path.
- The `verify_finality_level` check with `Latest` is mathematically guaranteed to pass for any included transaction.
- The only external dependency is that the Abstract chain domain is registered as supported on-chain, which is a normal operational state.

### Recommendation

Reject `EthereumFinality::Latest` (and `EvmFinality::Latest`) in `execute_foreign_chain_request` before calling `inspector.extract`. For bridge-critical signing flows, only `Safe` or `Finalized` should be accepted. A concrete fix:

```rust
// In execute_foreign_chain_request, Abstract arm:
let finality: EthereumFinality = request.finality.clone().try_into()?;
if finality == EthereumFinality::Latest {
    bail!("EthereumFinality::Latest is not permitted for bridge-critical signing");
}
```

Alternatively, remove `Latest` from `EvmFinality` entirely, or enforce the restriction at the contract level by rejecting requests with `finality=Latest` in `verify_foreign_transaction`.

    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }

    async fn verify_block_is_canonical(
        &self,
        receipt_block_number: U64,
        receipt_block_hash: H256,
    ) -> Result<(), ForeignChainInspectionError> {
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Number(receipt_block_number),
            ReturnFullTransactionObjects::from(false),
        );
        let canonical: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        let hash_matches = canonical.hash == receipt_block_hash;
        let height_matches = canonical.number == receipt_block_number;
        if !hash_matches || !height_matches {
            return Err(ForeignChainInspectionError::NonCanonicalBlock {
                block_number: receipt_block_number.as_u64(),
                receipt_hash: receipt_block_hash.into(),
                canonical_hash: canonical.hash.into(),
            });
        }
        Ok(())
    }
