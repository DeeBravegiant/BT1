### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Enables Cross-Chain Replay / Double-Spend via Fan-Out Signature Delivery - (File: `crates/contract/src/lib.rs`, `crates/contract/src/dto_mapping.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint constructs a request key that omits the caller's account ID. Any number of distinct NEAR accounts can queue the same foreign-chain verification request, and a single `respond_verify_foreign_tx` call drains every queued yield with the identical MPC-signed attestation. Because the signed payload (`ForeignTxSignPayloadV1`) also contains no NEAR recipient binding, any party that receives the fanned-out response can present it to a bridge contract to claim funds from a foreign-chain deposit that belongs to a different user.

---

### Finding Description

**Root cause — caller-agnostic request key**

`verify_foreign_transaction` in `crates/contract/src/lib.rs` converts the user-supplied arguments via `args_into_verify_foreign_tx_request`:

```rust
// crates/contract/src/dto_mapping.rs  lines 840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
```

The caller's `predecessor_account_id` is logged but never incorporated into the request struct. [1](#0-0) 

Contrast this with the `sign` endpoint, which explicitly binds the caller into the request key:

```rust
// crates/contract/src/lib.rs  lines 379-384
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,   // ← caller identity baked in
    &request.path,
);
``` [2](#0-1) 

**Fan-out delivery of the same signature**

`respond_verify_foreign_tx` calls `pending_requests::resolve_yields_for`, which drains **all** queued yields for the shared request key with the same serialized response:

```rust
// crates/contract/src/lib.rs  lines 749-753
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [3](#0-2) 

The contract's own unit test explicitly documents and asserts this behaviour:

```
// Then: both queued yields are drained from the single map entry.
``` [4](#0-3) 

The sandbox integration test `verify_foreign_transaction__should_fan_out_response_to_duplicates_from_different_callers` confirms it end-to-end: Alice and Bob both receive the identical `VerifyForeignTransactionResponse` from one `respond_verify_foreign_tx` call. [5](#0-4) 

**Signed payload carries no NEAR recipient binding**

The payload the MPC network signs is:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs  lines 1499-1502
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,   // tx_id + extractors
    pub values: Vec<ExtractedValue>,       // e.g. BlockHash
}
``` [6](#0-5) 

Neither the NEAR account that submitted the request nor any recipient address appears in the signed material. The signature is produced with a **zero tweak** (root key), making it a generic, transferable attestation of the foreign transaction:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs  lines 39-47
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),   // root key, no per-caller derivation
    ...
})
``` [7](#0-6) 

The SDK verifier (`ForeignChainSignatureVerifier::verify_signature`) only checks that the payload hash and ECDSA signature are consistent; it performs no caller-identity check. [8](#0-7) 

---

### Impact Explanation

The primary use case for `verify_foreign_transaction` is the Omnibridge inbound flow: a user deposits funds on a foreign chain, calls `verify_foreign_transaction` to obtain an MPC-signed attestation, and presents that attestation to a NEAR bridge contract to release the corresponding NEAR-side funds.

Because the attestation is caller-agnostic and carries no NEAR recipient binding, an adversary who receives the fanned-out response can present the identical `VerifyForeignTransactionResponse` to the bridge contract and claim the funds that belong to the original depositor. Any bridge contract that verifies the signature and checks only that the foreign `tx_id` has not been consumed (without also verifying that the claimer matches the foreign-chain recipient) is directly exploitable.

This maps to the allowed impact: **cross-chain replay / forged foreign-chain verification that causes double-spend conditions** (High).

---

### Likelihood Explanation

The NEAR blockchain is fully public. An attacker can trivially monitor `pending_verify_foreign_tx_requests` for any incoming request, copy the exact `(domain_id, request, payload_version)` tuple, and submit an identical `verify_foreign_transaction` call before the MPC nodes deliver their response. MPC signing requires multiple network rounds, providing a reliable window. No privileged access, key material, or collusion is required — only the ability to read on-chain state and submit a transaction.

---

### Recommendation

1. **Bind the caller into the request key.** Include `env::predecessor_account_id()` in `VerifyForeignTransactionRequest` (mirroring how `SignatureRequest::new` incorporates `predecessor`). This ensures each caller receives a cryptographically distinct attestation.

2. **Include the NEAR recipient in the signed payload.** Add the submitting account (or an explicit `recipient` field) to `ForeignTxSignPayloadV1` so the attestation is non-transferable.

3. **Use a per-caller key derivation tweak.** Derive the signing key from the caller's identity (as `sign` does via `derive_key_secp256k1`) rather than using the zero-tweak root key, so signatures from different callers are cryptographically unlinkable.

---

### Proof of Concept

The contract's own test suite provides the proof:

```
// crates/contract/src/lib.rs  lines 3230-3297
// When: caller alice submits the request.
contract.verify_foreign_transaction(request_args.clone());

// And: caller bob submits the identical request.
contract.verify_foreign_transaction(request_args);

// Then: both yields are queued under the single (caller-agnostic) request key.
assert_eq!(queue.len(), Some(2));

// When: a single valid response is delivered.
contract.respond_verify_foreign_tx(request.clone(), response).expect("ok");

// Then: both queued yields are drained — Alice AND Bob received the same signature.
assert!(pending_verify_foreign_tx_requests.get(&request).is_none());
``` [9](#0-8) 

**Attack steps against a bridge contract:**

1. Alice deposits 1 BTC to the bridge's Bitcoin address (tx `X`).
2. Alice calls `verify_foreign_transaction({tx_id: X, ...})`, paying the required deposit.
3. Attacker observes the pending request on-chain and calls `verify_foreign_transaction({tx_id: X, ...})` with the same parameters.
4. MPC nodes sign once; `respond_verify_foreign_tx` fans the response out to both Alice and the attacker.
5. Alice presents her `VerifyForeignTransactionResponse` to the bridge contract and receives her NEAR tokens.
6. The attacker presents the **identical** `VerifyForeignTransactionResponse` to the bridge contract. If the bridge contract does not verify that the claimer matches the Bitcoin tx's recipient address, it releases a second payout for the same deposit — a direct double-spend.

#[tokio::test]
async fn verify_foreign_transaction__should_fan_out_response_to_duplicates_from_different_callers()
{
    // Given
    let rpc_request = bitcoin_request();
    let extracted_values = bitcoin_extracted_values();
    let chain = rpc_request.chain();
    let setup = SandboxTestSetup::builder()
        .with_foreign_tx_domain()
        .build()
        .await;
    let foreign_tx_key = setup.foreign_tx_key();
    register_foreign_chain_configuration(chain, &setup.contract, &setup.mpc_signer_accounts).await;

    let alice = setup.worker.dev_create_account().await.unwrap();
    let bob = setup.worker.dev_create_account().await.unwrap();
    let domain_id = dtos::DomainId(foreign_tx_key.domain_id().0);
    let request_args = dtos::VerifyForeignTransactionRequestArgs {
        domain_id,
        payload_version: ForeignTxPayloadVersion::V1,
        request: rpc_request.clone(),
    };
    let verify_request = VerifyForeignTransactionRequest {
        domain_id,
        payload_version: ForeignTxPayloadVersion::V1,
        request: rpc_request,
    };

    // When
    let status_alice = alice
        .call(
            setup.contract.id(),
            method_names::VERIFY_FOREIGN_TRANSACTION,
        )
        .args_json(json!({ "request": request_args }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact_async()
        .await
        .unwrap();
    let status_bob = bob
        .call(
            setup.contract.id(),
            method_names::VERIFY_FOREIGN_TRANSACTION,
        )
        .args_json(json!({ "request": request_args }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact_async()
        .await
        .unwrap();
    await_pending_foreign_tx_request_observed_on_contract(&setup.contract, &verify_request).await;

    let (payload, response) = sign_foreign_tx_response(
        &verify_request.request,
        extracted_values,
        foreign_tx_key.as_secp256k1(),
    );
    let respond_result = setup.mpc_signer_accounts[0]
        .call(setup.contract.id(), method_names::RESPOND_VERIFY_FOREIGN_TX)
        .args_json(json!({
            "request": verify_request,
            "response": response,
        }))
        .max_gas()
        .transact()
        .await
        .unwrap()
        .into_result();

    // Then
    assert!(
        respond_result.is_ok(),
        "single respond_verify_foreign_tx should drain both queued yields: {respond_result:?}",
    );
    let expected_payload_hash = payload.compute_msg_hash().unwrap();

    let exec_alice = status_alice.await.unwrap().into_result().unwrap();
    let returned_alice: VerifyForeignTransactionResponse = exec_alice.json().unwrap();
    assert_eq!(
        returned_alice.payload_hash, expected_payload_hash,
        "alice's tx should receive the fanned-out response",
    );

    let exec_bob = status_bob.await.unwrap().into_result().unwrap();
    let returned_bob: VerifyForeignTransactionResponse = exec_bob.json().unwrap();
    assert_eq!(
        returned_bob.payload_hash, expected_payload_hash,
        "bob's tx should receive the same fanned-out response and not be displaced by alice",
    );

impl ForeignChainSignatureVerifier {
    pub fn verify_signature(
        self,
        response: &VerifyForeignTransactionResponse,
        // TODO(#2232): don't use interface API types for public keys
        public_key: &PublicKey,
    ) -> Result<(), VerifyForeignChainError> {
        let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
            request: self.request,
            values: self.expected_extracted_values,
        });

        let expected_payload_hash = expected_payload
            .compute_msg_hash()
            .map_err(|_| VerifyForeignChainError::FailedToComputeMsgHash)?;

        let payload_is_correct = expected_payload_hash == response.payload_hash;

        if !payload_is_correct {
            return Err(VerifyForeignChainError::IncorrectPayloadSigned {
                got: response.payload_hash.clone(),
                expected: expected_payload_hash,
            });
        }
        let verification_result = match (public_key, &response.signature) {
            (
                PublicKey::Secp256k1(secp256k1_public_key),
                SignatureResponse::Secp256k1(k256_signature),
            ) => near_mpc_signature_verifier::verify_ecdsa_signature(
                k256_signature,
                &expected_payload_hash,
                secp256k1_public_key,
            ),
            (PublicKey::Ed25519(ed25519_public_key), SignatureResponse::Ed25519 { signature }) => {
                near_mpc_signature_verifier::verify_eddsa_signature(
                    signature,
                    expected_payload_hash.as_slice(),
                    ed25519_public_key,
                )
            }
            // TODO(#2234): improve types so these errors can't happen
            (PublicKey::Bls12381(_bls12381_g2_public_key), _) => {
                return Err(VerifyForeignChainError::UnexpectedSignatureScheme);
            }
            _ => return Err(VerifyForeignChainError::UnexpectedSignatureScheme),
        };

        verification_result.map_err(|_| VerifyForeignChainError::SignatureVerificationFailed)
    }
