### Title
Cross-Request Payload Replay in `respond_verify_foreign_tx`: Missing Binding Between `request` and `response.payload_hash` — (File: `crates/contract/src/lib.rs`)

### Summary

The `respond_verify_foreign_tx` contract method verifies that the submitted MPC signature is cryptographically valid over `response.payload_hash`, but never verifies that `response.payload_hash` was actually derived from the `request` argument supplied in the same call. A single malicious attested participant (Byzantine node below the signing threshold) can replay a legitimately-obtained MPC signature from a previously-processed foreign-tx request as the response for a completely different pending request, causing the contract to deliver a forged verification attestation to the caller.

### Finding Description

`respond_verify_foreign_tx` performs two independent checks:

1. The caller is an attested participant.
2. The signature in `response` is valid over `response.payload_hash` under the domain's root public key. [1](#0-0) 

What it does **not** check is whether `response.payload_hash` was derived from the `request` argument. The payload hash is supposed to encode `ForeignTxSignPayloadV1 { request: ForeignChainRpcRequest, values: Vec<ExtractedValue> }`, binding the signature to a specific foreign-chain transaction and its observed values. [2](#0-1) 

The `ForeignTxSignPayloadV1` structure used in tests and in the node confirms that the payload encodes the `ForeignChainRpcRequest` (including `tx_id`) and the extracted values, but the contract never recomputes or checks this binding at response time. [3](#0-2) 

**Attack path:**

1. Attacker is an attested participant (a single Byzantine node, below signing threshold).
2. Request **R2** (e.g., "verify Bitcoin tx Y, extract block hash B") is processed normally by the MPC network. The attacker observes or saves `(payload_hash_R2, signature_R2)` from the on-chain event or from their own participation in the signing session.
3. A victim submits request **R1** (e.g., "verify Bitcoin tx X, extract block hash A"). R1 enters `pending_verify_foreign_tx_requests`.
4. The attacker calls `respond_verify_foreign_tx(request = R1, response = { payload_hash: payload_hash_R2, signature: signature_R2 })`.
5. The contract checks: (a) caller is attested ✓, (b) `signature_R2` is valid over `payload_hash_R2` ✓. Both pass. The contract resolves R1's yield with R2's response.
6. The victim's contract receives `VerifyForeignTransactionResponse { payload_hash: payload_hash_R2, signature: signature_R2 }` — a valid MPC attestation that encodes tx Y's data, not tx X's.

The victim's contract cannot detect the substitution: `payload_hash` is an opaque 32-byte hash; the victim does not know the observed values or `observed_at` needed to recompute the expected hash. The signature verifies correctly. The victim's bridge logic acts on a forged attestation.

### Impact Explanation

A single Byzantine attested participant can cause the MPC network to deliver a forged foreign-chain verification attestation to any pending `verify_foreign_transaction` caller. A bridge contract relying on this attestation (e.g., the Omnibridge inbound flow) would release funds or execute state transitions based on a transaction it never actually verified. This enables double-spend or theft of bridged assets without requiring threshold collusion — only one malicious node and one previously-signed response are needed.

This matches: **High. Cross-chain replay, forged foreign-chain verification, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

Any single attested participant can call `respond_verify_foreign_tx` — the contract does not restrict this to the signing leader. [4](#0-3) 

A valid prior signature (`payload_hash_R2`, `signature_R2`) is observable on-chain from any previously-completed foreign-tx request. No key forgery, no threshold collusion, and no privileged access beyond being an attested participant is required. The attack is executable by any single Byzantine node in the participant set.

### Recommendation

The contract should verify that `response.payload_hash` was derived from the `request` argument. Since the contract does not know the observed `values`, it cannot recompute the full payload hash. Two complementary mitigations:

1. **Include the `ForeignChainRpcRequest` hash in the on-chain check.** Require that `response.payload_hash` commits to the same `ForeignChainRpcRequest` as `request.request`. This can be done by having nodes also submit the raw `values` alongside the response, allowing the contract to recompute and verify the full payload hash.

2. **Alternatively, bind the payload hash to the request at signing time.** Include the `VerifyForeignTransactionRequest` (or its hash) as an additional domain-separation prefix in `ForeignTxSignPayloadV1`, so a signature produced for R2 is cryptographically invalid as a response for R1 even if the contract cannot recompute the full hash.

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
