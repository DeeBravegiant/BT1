### Title
Bitcoin `verify_foreign_transaction` Signs Only Block Hash, Omitting Output/Input Binding — Enables Multi-Output Deposit Fraud - (File: `crates/near-mpc-contract-interface/src/types/foreign_chain.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`)

### Summary
The `BitcoinExtractor` enum exposes only a `BlockHash` variant. When MPC nodes process a `verify_foreign_transaction` request for Bitcoin, the signed payload commits to the transaction ID and its containing block hash, but never to any specific output index, output amount, or output address. A Bitcoin transaction with multiple outputs can be submitted to obtain a valid MPC signature that is indistinguishable from one where the full deposit amount went to the bridge address, enabling a depositor to claim credit for an output they did not actually send to the bridge.

### Finding Description

The `BitcoinExtractor` enum in the contract interface defines only one extractable datum:

```rust
pub enum BitcoinExtractor {
    BlockHash = 0,
}
``` [1](#0-0) 

The inspector-side implementation mirrors this exactly — the only field read from the `GetRawTransactionVerboseResponse` is `blockhash`:

```rust
impl BitcoinExtractor {
    fn extract_value(&self, rpc_response: &GetRawTransactionVerboseResponse) -> BitcoinExtractedValue {
        match self {
            BitcoinExtractor::BlockHash => {
                BitcoinExtractedValue::BlockHash(From::from(*rpc_response.blockhash))
            }
        }
    }
}
``` [2](#0-1) 

The canonical payload that MPC nodes sign is `ForeignTxSignPayloadV1`, which contains the original request (including `tx_id`) and the `Vec<ExtractedValue>` produced by the extractors:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [3](#0-2) 

For Bitcoin, `values` will always be `[BitcoinExtractedValue::BlockHash(...)]`. The `BitcoinRpcRequest` carries only `tx_id`, `confirmations`, and `extractors`:

```rust
pub struct BitcoinRpcRequest {
    pub tx_id: BitcoinTxId,
    pub confirmations: BlockConfirmations,
    pub extractors: Vec<BitcoinExtractor>,
}
``` [4](#0-3) 

There is no `Output { vout_index: u64 }` or `Input { vin_index: u64 }` extractor analogous to EVM's `Log { log_index: u64 }`. The Bitcoin RPC response (`GetRawTransactionVerboseResponse`) contains full `vout`/`vin` arrays, but the extractor never reads them. [5](#0-4) 

Contrast with EVM, where `EvmExtractor::Log { log_index }` binds the signature to a specific log entry (containing recipient address, token amount, etc.):

```rust
pub enum EvmExtractor {
    BlockHash = 0,
    Log { log_index: u64 } = 1,
}
``` [6](#0-5) 

Bitcoin has no equivalent. The threshold signature therefore commits only to `(tx_id, block_hash)` — it is silent on which output was the deposit, to which address, and for what amount.

### Impact Explanation

Any bridge or application that calls `verify_foreign_transaction` for Bitcoin and uses the returned MPC signature as proof of a deposit receives a signature that is equally valid for every output in that transaction. An attacker who crafts a Bitcoin transaction with:

- `vout[0]`: 0.001 BTC → bridge deposit address
- `vout[1]`: 10 BTC → attacker-controlled address

…obtains an MPC signature over `(tx_id, block_hash)`. The bridge application cannot distinguish from the signature alone which output was the deposit. If the application credits the user based on the MPC signature without independently re-verifying the specific output, the attacker claims 10 BTC worth of wrapped tokens while only depositing 0.001 BTC. This constitutes a direct theft of funds enabled by forged/incomplete foreign-chain verification — matching the **High** impact tier (cross-chain verification bypass causing invalid bridge execution or double-spend conditions).

### Likelihood Explanation

The `BitcoinExtractor::BlockHash` is the only option available to any caller of `verify_foreign_transaction` for Bitcoin. Every Bitcoin bridge application built on this SDK is structurally affected. Bitcoin transactions with multiple outputs are the norm (change outputs, batched payments). The attacker entry path requires only the ability to submit a Bitcoin transaction and call `verify_foreign_transaction` — both are unprivileged operations.

### Recommendation

Add an `Output { vout_index: u64 }` variant to `BitcoinExtractor` (and a corresponding `BitcoinExtractedValue::Output { value_satoshis: u64, script_pubkey: Vec<u8> }`) so that the signed payload commits to the specific output index, amount, and locking script. This mirrors the `EvmExtractor::Log { log_index }` pattern already used for EVM chains. Bridge applications should be required to include at least one `Output` extractor in every `BitcoinRpcRequest` so the MPC signature is bound to a concrete, unambiguous output.

### Proof of Concept

1. Attacker crafts a Bitcoin transaction: `vout[0]` = 0.001 BTC to bridge deposit address; `vout[1]` = 10 BTC to attacker's second address.
2. Attacker broadcasts the transaction and waits for the required `confirmations`.
3. Attacker calls `verify_foreign_transaction` on the MPC contract with `BitcoinRpcRequest { tx_id: <txid>, confirmations: 6, extractors: [BitcoinExtractor::BlockHash] }`.
4. MPC nodes execute `BitcoinInspector::extract`, confirm the transaction is canonical and sufficiently confirmed, and extract only `blockhash`.
5. Nodes sign `ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: Bitcoin(<txid>, 6, [BlockHash]), values: [BitcoinExtractedValue::BlockHash(<hash>)] })`.
6. Attacker presents the MPC signature to the bridge application claiming a 10 BTC deposit. The signature is cryptographically valid for this `tx_id`; nothing in the signed payload contradicts the claim.
7. Bridge application, trusting the MPC signature as proof of deposit, credits the attacker with 10 BTC worth of wrapped tokens — a ~9.999 BTC theft. [7](#0-6) [8](#0-7)

    async fn extract(
        &self,
        transaction: BitcoinTransactionHash,
        block_confirmations_threshold: BlockConfirmations,
        extractors: Vec<BitcoinExtractor>,
    ) -> Result<Vec<BitcoinExtractedValue>, ForeignChainInspectionError> {
        let request_parameters = GetRawTransactionArgs {
            transaction_hash: TransportBitcoinTransactionHash::from(*transaction),
            verbose: VERBOSE_RESPONSE,
        };

        // TODO(#1978): add retry mechanism if the error from the request is transient
        let rpc_response: GetRawTransactionVerboseResponse = self
            .client
            .request(GET_RAW_TRANSACTION_METHOD, &request_parameters)
            .await?;

        let transaction_block_confirmation = rpc_response.confirmations.into();
        let enough_block_confirmations =
            block_confirmations_threshold <= transaction_block_confirmation;

        if !enough_block_confirmations {
            return Err(ForeignChainInspectionError::NotEnoughBlockConfirmations {
                expected: block_confirmations_threshold,
                got: transaction_block_confirmation,
            });
        }

        self.verify_block_is_canonical(rpc_response.blockhash)
            .await?;

        let extracted_values = extractors
            .iter()
            .map(|extractor| extractor.extract_value(&rpc_response))
            .collect();

        Ok(extracted_values)
    }
