### Title
Missing On-Chain Output Validation in `respond_ckd` for `AppPublicKey` Variant Allows Single Malicious Participant to Forge Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in `crates/contract/src/lib.rs` performs a cryptographic pairing check (`ckd_output_check`) on the MPC response **only** when the request uses the `AppPublicKeyPV` variant. When the request uses the `AppPublicKey` (privately-verifiable) variant, the response body is accepted with **zero on-chain validation**. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with an arbitrary forged `CKDResponse`, and the contract will resolve every pending yield for that request with the attacker-controlled output.

---

### Finding Description

`respond_ckd` dispatches on the `app_public_key` variant of the incoming `CKDRequest`:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` verifies the BLS12-381 pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`, which cryptographically binds the response to the network master key and the user's app public key: [2](#0-1) 

For `AppPublicKey`, the arm is an empty no-op `{}`. The contract then unconditionally calls `resolve_yields_for`, delivering the unverified `response` to every waiting caller: [3](#0-2) 

The `AppPublicKey` variant is the standard (non-PV) CKD mode and is actively used in production — the e2e test `ckd_response__passes_cryptographic_verification` exercises it end-to-end: [4](#0-3) 

The CKD protocol for `AppPublicKey` works as follows: the user supplies `A = a·G1`; the MPC network computes `big_y = y·G1` and `big_c = H(pk ∥ app_id)·msk + A·y`; the user recovers the derived secret as `big_c − a·big_y = H(pk ∥ app_id)·msk`. Without on-chain verification, an attacker can set `big_y = G1_identity` and `big_c = s·G1` for any scalar `s` they choose. The user then decrypts to `s·G1`, a fully attacker-controlled value, breaking the confidentiality guarantee entirely.

The `respond_ckd` function does check that the caller is an attested participant: [5](#0-4) 

But attestation is a per-node property, not a threshold property. A **single** attested participant can call `respond_ckd` and win the race against the honest nodes, since the first successful call drains all pending yields for the request.

---

### Impact Explanation

**Critical.** A single malicious attested MPC participant (strictly below the signing threshold) can forge the CKD output for any pending `AppPublicKey` request. The forged `(big_c, big_y)` is delivered to every caller waiting on that yield. The user decrypts an attacker-controlled G1 point as their "derived secret", completely defeating the confidentiality guarantee of the CKD protocol. This constitutes unauthorized confidential key derivation output without the required threshold-participant authorization, and a bypass of the threshold-signature requirement for the CKD flow.

---

### Likelihood Explanation

**Medium-High.** Any single attested participant who turns malicious (or whose node is compromised) can exploit this immediately. The `AppPublicKey` variant is the default/legacy CKD mode used by existing integrations. The attacker only needs to submit a `respond_ckd` call before honest nodes do — a straightforward race on the NEAR mempool. No special tooling or collusion is required.

---

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. Because `AppPublicKey` provides only a G1 point (no G2 counterpart), the existing `ckd_output_check` (which requires `pk2` from `AppPublicKeyPV`) cannot be reused directly. The fix requires one of:

1. **Extend the check**: Derive a verifiable commitment from `AppPublicKey` alone (e.g., require nodes to also submit a proof-of-correct-encryption compatible with the G1-only key), or
2. **Deprecate `AppPublicKey` in `respond_ckd`**: Reject `AppPublicKey` responses at the contract level and require callers to migrate to `AppPublicKeyPV`, which already has a sound on-chain check.

At minimum, add a comment and a tracking issue making the missing check explicit, so it is not silently inherited by future response handlers.

---

### Proof of Concept

1. User submits `request_app_private_key` with `AppPublicKey(A)` where `A = a·G1`.
2. The contract stores the pending yield and emits an event.
3. Before any honest node responds, the malicious attested participant constructs a forged response: `big_y = G1_identity (0)`, `big_c = s·G1` for attacker-chosen scalar `s`.
4. Malicious participant calls `respond_ckd(request, CKDResponse { big_y: 0, big_c: s·G1 })`.
5. The contract executes lines 675–682: the `AppPublicKey(_) => {}` arm runs — no check.
6. `resolve_yields_for` is called, delivering `{ big_y: 0, big_c: s·G1 }` to the user.
7. User decrypts: `big_c − a·big_y = s·G1 − a·0 = s·G1`. The attacker knows `s`, so they know the user's "derived secret" in full. [6](#0-5) [2](#0-1)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

#[tokio::test]

async fn ckd_response__passes_cryptographic_verification() {
    // given
    let (cluster, running) =
        common::must_setup_cluster(common::CKD_VERIFICATION_PORT_SEED, |_| {}).await;

    let bls_domain = running
        .domains
        .domains
        .iter()
        .find(|d| {
            Curve::from(d.protocol) == Curve::Bls12381 && matches!(d.purpose, DomainPurpose::CKD)
        })
        .expect("no Bls12381 CKD domain found")
        .clone();

    let mpc_pk = common::must_get_bls_public_key(&running, bls_domain.id);
    let user = cluster.default_user_account().clone();

    let mut rng = rand::rngs::StdRng::seed_from_u64(1);
    let private_key = Scalar::random(&mut rng);
    let app_public_key = CKDAppPublicKey::AppPublicKey(Bls12381G1PublicKey::from(
        &(G1Projective::generator() * private_key),
    ));

    // when
    let outcome = cluster
        .send_ckd_request(bls_domain.id, app_public_key, &user)
        .await
        .expect("CKD request transaction failed");

    // then
    assert!(
        outcome.is_success(),
        "CKD request failed: {:?}",
        outcome.failure_message()
    );

    let response: serde_json::Value = outcome.json().expect("failed to deserialize CKD response");
    let big_y: Bls12381G1PublicKey =
        serde_json::from_value(response["big_y"].clone()).expect("failed to parse big_y");
    let big_c: Bls12381G1PublicKey =
        serde_json::from_value(response["big_c"].clone()).expect("failed to parse big_c");

    assert!(
        verify_ckd(&user, DERIVATION_PATH, &mpc_pk, private_key, &big_y, &big_c)
            .expect("verify_ckd failed"),

    );
}
