### Title
Deposits for Non-Winning Update Proposals Are Permanently Locked When `do_update` Clears All Entries Without Refunding - (`crates/contract/src/update.rs`)

### Summary

`propose_update` collects a storage-proportional NEAR deposit from each proposer, but `do_update` calls `self.entries.clear()` to wipe all competing proposals without iterating over them to refund those deposits. Because `UpdateEntry` does not record the proposer's account ID, the locked NEAR is irrecoverable.

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a deposit calculated from the byte size of the proposed update: [1](#0-0) 

The exact required amount is retained by the contract; only the excess is refunded. The deposit is meant to cover on-chain storage for the `UpdateEntry`.

When threshold votes are reached, `vote_update` calls `do_update`: [2](#0-1) 

`do_update` removes the winning entry, then calls `self.entries.clear()` and `self.vote_by_participant.clear()`, wiping every other pending proposal. No refund is issued for any of the cleared entries.

The `UpdateEntry` struct stores only the update payload and `bytes_used`; it does not record the proposer's account ID: [3](#0-2) 

Because the proposer's identity is not persisted in the entry, the contract has no information with which to issue a refund when `entries.clear()` runs. The NEAR tokens are permanently locked in the contract balance.

The test `test_proposed_updates_do_update_clears_all_state` explicitly verifies that all entries are wiped, confirming this is the intended behavior — but the deposit accounting consequence is absent: [4](#0-3) 

### Impact Explanation

Every participant who proposed a competing update loses their full deposit (storage-byte-cost × bytes_used) permanently. For a 1 MB contract binary, NEAR's storage cost (~10 NEAR/MB) means each non-winning proposer loses ~10 NEAR with no recovery path. The contract's NEAR balance grows by the sum of all non-winning deposits, breaking the accounting invariant that deposits paid for storage should be returned when that storage is freed. This matches the **Medium** allowed impact: *balance or contract execution-flow manipulation that breaks production safety/accounting invariants*.

### Likelihood Explanation

The scenario requires only that two or more participants call `propose_update` before the threshold is reached — a normal governance workflow explicitly supported by the contract (the README and tests both demonstrate multiple concurrent proposals). No adversarial action is needed; the loss occurs automatically on every multi-proposal upgrade cycle. Deposits for contract-binary proposals are non-trivial in size.

### Recommendation

Store the proposer's `AccountId` inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add this
}
```

Then in `do_update`, before calling `self.entries.clear()`, iterate over the remaining entries and schedule a `Promise::new(entry.proposer).transfer(required_deposit(entry.bytes_used))` for each one. The winning entry's deposit handling can follow the same pattern (refund on clear, or retain as intended).

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR deposit. Entry `id=0` is stored.
2. Participant B calls `propose_update` with a different 1 MB binary, attaching ~10 NEAR deposit. Entry `id=1` is stored.
3. Threshold participants vote for `id=1`. `vote_update` calls `do_update(&id=1, gas)`.
4. Inside `do_update`: entry `id=1` is removed and executed; `self.entries.clear()` removes entry `id=0`; `self.vote_by_participant.clear()` removes all votes.
5. Participant A's ~10 NEAR deposit is now permanently locked in the contract. There is no `remove_proposal` endpoint, no stored proposer address in the cleared entry, and no refund promise was ever scheduled. [5](#0-4) [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }
