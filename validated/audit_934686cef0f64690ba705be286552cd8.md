### Title
`accept_requests` Flag Blocks In-Flight `respond*` Calls, Stranding Already-Queued Pending Yields — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `accept_requests: bool` field in `MpcContract` is set to `false` by `verify_tee()` when TEE attestations expire and kicking out invalid participants would break the threshold relation. This flag is checked not only in the user-facing `sign()` / `request_app_private_key()` / `verify_foreign_transaction()` entry points, but also inside every `respond*` node-facing method. As a result, requests that were already accepted and are parked as pending yield-resume promises cannot be completed: MPC nodes cannot call `respond()`, `respond_ckd()`, or `respond_verify_foreign_tx()` to deliver their computed signatures. Every in-flight request times out after ~200 blocks, breaking the request-lifecycle invariant and leaving foreign-chain operations in an incomplete state.

---

### Finding Description

`MpcContract` stores a single boolean flag: [1](#0-0) 

`verify_tee()` sets it to `false` when the surviving valid-attestation set would violate the threshold relation: [2](#0-1) 

The same flag is then checked inside all three `respond*` methods before they are allowed to resolve pending yields:

```
respond()            → lib.rs:579-581
respond_ckd()        → lib.rs:662-664
respond_verify_foreign_tx() → lib.rs:711-713
``` [3](#0-2) [4](#0-3) [5](#0-4) 

When `accept_requests = false`, every `respond*` call returns `Err(TeeError::TeeValidationFailed)` and the pending yield is never resumed. The NEAR runtime fires the yield-callback after ~200 blocks of silence with `Err(PromiseError::Failed)`, which pops the oldest queued yield and calls `fail_on_timeout`: [6](#0-5) 

The pending-request maps (`pending_signature_requests`, `pending_ckd_requests`, `pending_verify_foreign_tx_requests`) hold the parked yields: [7](#0-6) 

`resolve_yields_for` — the only path that successfully delivers a response — is never reached: [8](#0-7) 

The flag is intentionally broad: the docstring for `verify_tee()` explicitly states it "stops the contract from accepting new signature requests **or responses**": [9](#0-8) 

This is the direct analog to `isMintEnabled` blocking `_mint()`: a single flag governs both the acceptance of new requests and the completion of already-accepted in-flight requests.

---

### Impact Explanation

**Medium — request-lifecycle and contract execution-flow invariant broken.**

Any request that was accepted while `accept_requests = true` and is still parked in a pending-yield queue when `verify_tee()` flips the flag to `false` will time out:

- **`sign()` requests**: The user's NEAR transaction fails after ~200 blocks. If the user had already committed a foreign-chain action (e.g., locked funds on Ethereum) expecting the MPC signature to unlock them, those funds are left in a stuck intermediate state until the user can retry.
- **`verify_foreign_transaction()` requests**: The foreign-chain transaction verification cannot be completed. The user's foreign-chain funds may be locked pending a signature that never arrives.
- **`request_app_private_key()` (CKD) requests**: Confidential key derivation flows are broken mid-flight.

The production safety invariant — "a request accepted by the contract will eventually be completed or explicitly failed with a meaningful error" — is violated. The failure mode is silent from the user's perspective: the NEAR transaction simply times out with no indication that the underlying cause is a TEE validation state change that occurred after the request was submitted.

---

### Likelihood Explanation

TEE attestations carry an expiry timestamp and expire naturally during normal operation. When enough attestations expire to break the threshold relation, **any single participant** (below threshold) can call `verify_tee()` — a normal, expected protocol operation — to trigger `accept_requests = false`. No threshold collusion, no privileged admin action, and no operator misconfiguration is required. The condition can arise organically during periods of high request volume (many yields in flight) coinciding with attestation expiry, making the impact window realistic. [10](#0-9) 

---

### Recommendation

Decouple the "accept new requests" gate from the "allow responses to in-flight requests" gate. Specifically:

1. Keep the `accept_requests` check in `check_request_preconditions()` to block new `sign()` / `request_app_private_key()` / `verify_foreign_transaction()` submissions.
2. **Remove** the `accept_requests` guard from `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()`. These methods already authenticate the caller as an attested participant (`assert_caller_is_attested_participant_and_protocol_active`) and verify the signature before resolving yields — blocking them when `accept_requests = false` provides no additional security benefit while stranding all in-flight requests.

Alternatively, introduce a separate `accept_responses: bool` flag that is only set to `false` under conditions that genuinely require halting response delivery (e.g., a critical key-compromise event), distinct from the TEE-attestation-expiry condition that sets `accept_requests = false`.

---

### Proof of Concept

1. User calls `sign(payload, path)` — the contract accepts the request, parks a yield in `pending_signature_requests`, and returns a pending promise to the user.
2. Between the `sign()` call and the MPC nodes' `respond()` call, one or more participants' TEE attestations expire.
3. A participant (any single one) calls `verify_tee()`. The surviving valid-attestation count would break the threshold relation, so `accept_requests` is set to `false` and `Ok(false)` is returned.
4. MPC nodes complete the threshold signing protocol off-chain and call `respond(request, signature)` on-chain. The call hits the guard at line 579–581 and returns `Err(TeeError::TeeValidationFailed)`. The pending yield is not resumed.
5. After ~200 blocks, the NEAR runtime fires `return_signature_and_clean_state_on_success` with `Err(PromiseError::Failed)`. The yield is popped and `fail_on_timeout` is called. The user's NEAR transaction fails.
6. If the user had already submitted a foreign-chain transaction (e.g., locked ETH in a bridge contract) expecting the MPC signature to authorize the release, those funds remain locked with no signature delivered. [11](#0-10) [12](#0-11)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
