### Title
`FanOut` Inspector Ignores On-Chain `ChainEntry.quorum`, Allowing Single-Provider Foreign-Chain Verification Bypass - (File: crates/foreign-chain-inspector/src/lib.rs)

### Summary
The `FanOut<Inspector>` implementation that fans out foreign-chain RPC queries to multiple providers does not enforce the `quorum` value stored in the on-chain `ChainEntry`. The governance-voted quorum requirement — the minimum number of independent RPC providers that must agree before a node accepts a verification result — is silently bypassed whenever other providers return transient errors. A single malicious or compromised RPC provider can therefore unilaterally determine a node's foreign-chain verification outcome.

### Finding Description
`ChainEntry.quorum` is voted in on-chain by participants via `vote_update_foreign_chain_providers` and validated at construction time in `TryFrom<dtos::ChainEntry>` to be non-zero and ≤ the number of providers. The design documentation and the `ForeignChainRpcWhitelist` architecture explicitly state that this value controls how many of a node's configured providers must return the same response before the node accepts a verification result.

The `FanOut<Inspector>` struct in `crates/foreign-chain-inspector/src/lib.rs` is the node-side implementation that fans out queries to multiple inspectors (one per provider). Its `extract` method:

1. Collects successes and non-transient errors from all inspectors concurrently.
2. Tolerates transient errors (network issues, finality not yet reached) — these are not counted as substantive verdicts.
3. If any successes exist and all agree → returns `Ok(first_values)` **immediately**.

The critical flaw: the number of agreeing successes is **never compared against the quorum value**. The `FanOut` struct has no `quorum` field and takes no quorum parameter. The `ChainEntry.quorum` value is never passed into `FanOut` anywhere in the construction path.
