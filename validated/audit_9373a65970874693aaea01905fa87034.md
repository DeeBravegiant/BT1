### Title
Missing Cryptographic Verification of CKD Response for `AppPublicKey` Variant Allows Byzantine Participant to Forge Confidential Key Derivation Output - (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_ckd` function in the MPC contract performs no cryptographic verification of the `CKDResponse` when the request uses the `AppPublicKey` (legacy, privately-verifiable) variant. A single Byzantine attested participant can submit an arbitrary `(big_y, big_c)` pair and the contract will accept it unconditionally, causing the user to derive a wrong or attacker-controlled key. The `AppPublicKeyPV` variant is protected by a BLS pairing check (`ckd_output_check`), but the `AppPublicKey` branch is a no-op.

---

### Finding Description

In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the request's `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk, app_id), mpc_pk)` via `ckd_output_check`: [2](#0-1) 

For `AppPublicKey`, the arm is empty — **any** `CKDResponse` is accepted and immediately forwarded to the user via `resolve_yields_for`: [3](#0-2) 

The `AppPublicKey` variant is the legacy, widely-used format (a single BLS12-381 G1 point). It is described as "privately verifiable" because only the user holding the secret scalar `a` can verify the response off-chain. However, the contract provides no on-chain guard, so a Byzantine participant can front-run the honest coordinator and inject a forged response before the correct one arrives.

The `respond_ckd` caller check (`assert_caller_is_attested_participant_and_protocol_active`) only verifies TEE attestation and participant membership — it does not verify that the submitted `(big_y, big_c)` is the correct ElGamal encryption of `msk · H(pk, app_id)`: [4](#0-3) 

The analog to the Android biometric report is exact: the contract trusts the `respond_ckd` callback result (the `CKDResponse`) without a cryptographic binding (the `CryptoObject` analog is `ckd_output_check`), and the `AppPublicKey` branch is the missing `CryptoObject` enforcement.

---

### Impact Explanation

The CKD protocol delivers `(Y, C)` where `C = msk·H(pk, app_id) + y·A` and `Y = y·G1`. The user decrypts: `sig = C − a·Y = msk·H(pk, app_id)`, then derives their app key as `s = HKDF(sig)`.

A Byzantine participant submitting forged `(big_y*, big_c*)` causes the user to compute `sig* = big_c* − a·big_y*`. If the attacker sets `big_y* = identity` (zero point), then `sig* = big_c*`, which is fully attacker-controlled and known to the attacker. The user's derived key `s* = HKDF(big_c*)` is then known to the attacker.

This constitutes **unauthorized confidential key derivation output** — the attacker controls the secret material the user derives, enabling them to impersonate the user's TEE application or decrypt data the user encrypted with the derived key. The first `respond_ckd` call wins (the contract resolves all queued yields immediately), so the Byzantine participant only needs to submit before the honest coordinator.

Impact: **Critical** — unauthorized access to confidential key derivation output; attacker can make the user's derived secret key predictable and known to the attacker.

---

### Likelihood Explanation

- Requires exactly **one** Byzantine attested participant (strictly below the signing threshold).
- The attacker must be a registered, TEE-attested participant — a realistic threat model for a Byzantine node operator.
- The attack is a simple front-run: submit `respond_ckd` with forged `(big_y, big_c)` before the honest coordinator does. No special timing or network access is required beyond normal participant operation.
- The `AppPublicKey` variant is the legacy default and is used by existing integrations.

Likelihood: **Medium** — requires a malicious participant with valid TEE attestation, but no threshold collusion.

---

### Recommendation

Apply the same `ckd_output_check` pairing verification to the `AppPublicKey` variant. Since `AppPublicKey` only provides a G1 point (`pk1`) without a G2 counterpart (`pk2`), the existing `ckd_output_check` (which requires `pk2`) cannot be applied directly. Two options:

1. **Require `AppPublicKeyPV` for all new requests** and deprecate `AppPublicKey`. The `AppPublicKeyPV` variant already has full on-chain verification.
2. **Add a BLS signature verification** for the `AppPublicKey` case: after the MPC nodes produce `sig = big_c − a·big_y`, verify `e(sig, G2) = e(H(pk, app_id), mpc_pk)` on-chain. This requires the user to submit their decrypted signature for verification, which changes the protocol flow.

The simplest safe fix is option 1: reject new `AppPublicKey` CKD requests at the contract level and require `AppPublicKeyPV`, which already enforces the pairing check in `respond_ckd`.

### Proof of Concept

1. User calls `request_app_private_key` with `AppPublicKey(A)` where `A = a·G1`.
2. Byzantine attested participant observes the pending request and calls:
   ```
   respond_ckd(
     request = <the pending CKDRequest>,
     response = CKDResponse { big_y: G1_IDENTITY_BYTES, big_c: ATTACKER_CHOSEN_POINT }
   )
   ```
3. Contract executes the `AppPublicKey(_) => {}` branch — no check performed.
4. `resolve_yields_for` resumes the user's yield with the forged response.
5. User receives `(big_y=0, big_c=K)` and computes `sig = K − a·0 = K`.
6. User derives `s = HKDF(K)` — a key fully known to the attacker who chose `K`.

The test at line 3424–3440 confirms the `AppPublicKey` path accepts arbitrary byte values `[1u8; 48]` and `[2u8; 48]` for `big_y` and `big_c` with no rejection: [5](#0-4)

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
