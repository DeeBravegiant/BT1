### Title
Deposits from Non-Winning Update Proposals Permanently Locked When `do_update` Clears All Entries - (File: `crates/contract/src/update.rs`)

### Summary
When `vote_update` reaches threshold and executes `do_update`, it bulk-clears **all** pending update proposals without refunding the storage deposits attached by the proposers of the non-winning proposals. Those deposits are permanently locked in the contract balance with no recovery path.

### Finding Description
`propose_update` in `crates/contract/src/lib.rs` requires each proposer to attach a deposit proportional to the size of their proposed update (contract binary or config), calculated by `ProposedUpdates::required_deposit`. The exact required amount is retained by the contract; only the excess is immediately refunded. [1](#0-0) 

The deposit is computed as: [2](#0-1) 

For a 1 MB contract binary this is roughly **10 NEAR** (NEAR storage byte cost ≈ 10^19 yoctoNEAR/byte × 10^6 bytes). The `UpdateEntry` also pre-allocates overhead for 128 participant-vote slots, adding further cost.

When `vote_update` reaches threshold it calls `do_update`: [3](#0-2) 

`do_update` calls `self.entries.clear()` (line 199), which removes every pending proposal — including all non-winning ones — and `self.vote_by_participant.clear()` (line 200). Neither call issues any refund to the proposers of the cleared entries. The NEAR tokens those proposers deposited remain in the contract's balance permanently; there is no subsequent path that returns them.

The `UpdateEntry` struct does not record the proposer's account ID or the deposit amount: [4](#0-3) 

So even if a refund were attempted later, the information needed to issue it is gone.

### Impact Explanation
Every participant whose proposal is cleared by a winning `do_update` permanently loses their deposit. For contract-binary proposals the deposit scales with the WASM size; a realistic 1–2 MB binary costs 10–20 NEAR per proposal. If N participants each propose a different binary and one wins, (N-1) × deposit is irrecoverably locked in the contract. This breaks the accounting invariant that "storage deposits are returned when the storage they cover is freed," and constitutes a direct, permanent loss of funds controlled by the MPC contract.

### Likelihood Explanation
Multiple concurrent update proposals are a realistic governance scenario: participants may disagree on which binary or config to adopt and each submit their own. The README explicitly acknowledges that multiple proposals can coexist simultaneously: [5](#0-4) 

The sandbox test `test_propose_update_contract_many` confirms the multi-proposal path is exercised in practice. [6](#0-5) 

No malicious intent is required; the loss occurs through ordinary governance use.

### Recommendation
Store the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry`. In `do_update`, before calling `self.entries.clear()`, iterate over all non-winning entries and schedule `Promise::new(entry.proposer).transfer(entry.deposit)` for each. This mirrors the refund pattern already used elsewhere in the contract (e.g., `propose_update` refunds excess deposit immediately, and `submit_participant_info` refunds on rejection).

### Proof of Concept
1. Participant A calls `propose_update` with a 1 MB contract binary, attaching ~10 NEAR. The contract stores the entry and retains the deposit.
2. Participant B calls `propose_update` with a different 1 MB binary, attaching ~10 NEAR.
3. Threshold participants call `vote_update` for B's proposal ID until the threshold is met.
4. `vote_update` calls `do_update(&B_id, gas)`:
   - `self.entries.remove(&B_id)` — removes B's entry (winning).
   - `self.entries.clear()` — removes A's entry (non-winning), freeing storage.
   - No refund is issued to A.
5. A's ~10 NEAR deposit is permanently locked in the contract balance. A has no method to recover it; `propose_update` only refunds excess above `required`, and there is no `cancel_proposal` or equivalent endpoint. [7](#0-6)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}

## Contract Updates

Participants can propose and vote on contract updates (code or configuration changes). When an update receives sufficient votes and is executed (via the `vote_update` endpoint which calls `do_update` internally), all pending update proposals and votes are cleared as they are no longer be valid after the contract migration. The update ID counter is preserved across migrations as part of the contract state to avoid race conditions where multiple participants might propose updates with colliding IDs immediately after an upgrade.

async fn test_propose_update_contract_many() {
    let SandboxTestSetup {
        contract,
        mpc_signer_accounts,
        ..
    } = SandboxTestSetup::builder()
        .with_protocols(ALL_PROTOCOLS)
        .build()
        .await;
    dbg!(contract.id());

    const PROPOSAL_COUNT: usize = 2;
    let mut proposals = Vec::with_capacity(PROPOSAL_COUNT);
    // Try to propose multiple updates to check if they are being proposed correctly
    // and that we can have many at once living in the contract state.
    for i in 0..PROPOSAL_COUNT {
        let execution = mpc_signer_accounts[i % mpc_signer_accounts.len()]
            .call(contract.id(), method_names::PROPOSE_UPDATE)
            .args_borsh(current_contract_proposal())
            .max_gas()
            .deposit(CURRENT_CONTRACT_DEPLOY_DEPOSIT)
            .transact()
            .await
            .unwrap();

        assert!(
            execution.is_success(),
            "failed to propose update [i={i}]; {execution:#?}"
        );
        let proposal_id = execution.json().expect("unable to convert into UpdateId");
        proposals.push(proposal_id);
    }

    // Vote for the last proposal
    vote_update_till_completion(&contract, &mpc_signer_accounts, proposals.last().unwrap()).await;

    // Ensure all proposals are removed after update
    for proposal in proposals {
        let voter = mpc_signer_accounts.first().unwrap();
        let execution = voter
            .call(contract.id(), method_names::VOTE_UPDATE)
            .args_json(serde_json::json!({
                "id": proposal,
            }))
            .gas(GAS_FOR_VOTE_UPDATE)
            .transact()
            .await
            .unwrap();
        dbg!(&execution);

        assert!(execution.is_failure());
    }

    // Let's check that we can call into the state and see all the proposals.
    let state: ProtocolContractState = get_state(&contract).await;
    dbg!(state);
}
