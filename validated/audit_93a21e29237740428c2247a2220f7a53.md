Audit Report

## Title
Nested Loops with External Calls in `updateRSETHPrice()` Cause Unbounded Gas Consumption — (File: contracts/LRTOracle.sol)

## Summary
`updateRSETHPrice()` is publicly callable and triggers a multiplicative chain of nested loops across supported assets, NodeDelegators, queued EigenLayer withdrawals, and per-withdrawal strategies, each innermost iteration making an external call to `strategy.sharesToUnderlyingView()`. At realistic operational scale (5 assets × 10 NDCs × 80 queued withdrawals × 2 strategies), the cumulative gas cost exceeds the Ethereum block gas limit, permanently preventing price updates and blocking all user deposits and withdrawal initiations that depend on `getTotalAssetDeposits()`.

## Finding Description
The full call chain reachable by any unprivileged caller:

1. `updateRSETHPrice()` (`LRTOracle.sol` L87–89) is `public whenNotPaused` and calls `_updateRsETHPrice()` → `_getTotalEthInProtocol()`.

2. `_getTotalEthInProtocol()` (`LRTOracle.sol` L336–348) loops over every supported asset and calls `ILRTDepositPool.getTotalAssetDeposits(asset)` for each.

3. `getTotalAssetDeposits()` (`LRTDepositPool.sol` L385–397) calls `getAssetDistributionData(asset)`, which (`LRTDepositPool.sol` L446–456) loops over every NDC in `nodeDelegatorQueue` and calls `INodeDelegator(ndc).getAssetUnstaking(asset)` for each.

4. `getAssetUnstaking()` (`NodeDelegator.sol` L405–427) first calls `_getDelegationManager().getQueuedWithdrawals(address(this))` (one external call loading all queued withdrawal structs), then runs a **nested loop**: for each queued withdrawal, for each strategy within it, it calls `strategy.sharesToUnderlyingView(sharesToUnstake)` — an external call to an EigenLayer strategy contract.

Total external calls to `sharesToUnderlyingView`: `N_assets × N_NDCs × N_withdrawals × N_strategies`. With N=5, M=10, K=80, S=2 this is 8,000 external calls, each costing ≥5,000 gas (cold CALL + SLOAD-heavy strategy execution) = **≥40M gas**, exceeding Ethereum's 30M block gas limit.

The same `getTotalAssetDeposits()` path is also triggered on every user deposit via `depositAsset()` / `depositETH()` → `_beforeDeposit()` → `_checkIfDepositAmountExceedesCurrentLimit()` (`LRTDepositPool.sol` L676–682), and on withdrawal initiation via `LRTWithdrawalManager`.

No existing guard prevents this: `maxNodeDelegatorLimit` (default 10, admin-raiseable) and `maxUncompletedWithdrawalCount` (up to 80) are independent admin parameters with no joint cap enforcing a safe product relative to the block gas limit.

## Impact Explanation
**Medium — Unbounded gas consumption / Temporary freezing of funds.**

Once the cumulative gas cost of `updateRSETHPrice()` or `depositAsset()`/`depositETH()` exceeds the block gas limit:
- `updateRSETHPrice()` becomes permanently uncallable: the stored `rsETHPrice` goes stale, fee minting is blocked, and TVL accounting breaks.
- `depositAsset()` and `depositETH()` revert on every call: users cannot deposit assets — temporary freezing of protocol inflows.
- `initiateWithdrawal()` in `LRTWithdrawalManager` similarly calls `getTotalAssetDeposits()` and is blocked.

Both "Unbounded gas consumption" and "Temporary freezing of funds" are explicitly listed as valid Medium impacts in scope.

## Likelihood Explanation
The protocol's `maxNodeDelegatorLimit` starts at 10 and is admin-raiseable. `maxUncompletedWithdrawalCount` can be set to 80. With 5 supported LST assets (a realistic count for a restaking protocol), 10 NDCs, and 80 queued withdrawals per NDC each containing 2 strategies, the iteration count reaches 8,000 external calls per `updateRSETHPrice()` invocation. This is a realistic operational state — not an adversarial setup — making likelihood **Medium**. No attacker action is required; the protocol reaches this state through normal growth and operator activity.

## Recommendation
1. **Decouple `getAssetUnstaking()` from the price update hot path**: Store per-NDC unstaking amounts in a mapping updated lazily (e.g., on `initiateUnstaking` / `completeUnstaking`) rather than recomputing via `getQueuedWithdrawals()` on every price update.
2. **Enforce a joint cap**: Add a protocol invariant that `supportedAssets × NDCs × maxUncompletedWithdrawalCount × maxStrategiesPerWithdrawal` stays below a safe gas budget, and enforce it in `updateMaxNodeDelegatorLimit` and `setMaxUncompletedWithdrawalCount`.
3. **Cache intermediate TVL results**: Maintain a running `totalUnstakingByAsset` accumulator updated on each unstaking state change, eliminating the need to iterate EigenLayer queued withdrawals on read.
4. **Paginate or batch**: Allow `updateRSETHPrice()` to accept a subset of assets/NDCs per call, accumulating results across multiple transactions.

## Proof of Concept
**Foundry fork test plan:**

```solidity
// Fork mainnet, deploy protocol with:
// - 5 supported assets
// - 10 NodeDelegators (maxNodeDelegatorLimit = 10)
// - Each NDC has 80 queued withdrawals (maxUncompletedWithdrawalCount = 80)
//   each with 2 strategies
// Call updateRSETHPrice() and measure gas:
function test_updateRSETHPrice_gasExhaustion() public {
    // setup: 5 assets, 10 NDCs, 80 queued withdrawals each
    // ...
    uint256 gasBefore = gasleft();
    lrtOracle.updateRSETHPrice();
    uint256 gasUsed = gasBefore - gasleft();
    assertGt(gasUsed, 30_000_000); // exceeds block gas limit
}
```

**Call chain (any unprivileged address):**
```
updateRSETHPrice()                         [LRTOracle.sol L87 — public]
  └─ _getTotalEthInProtocol()             [LRTOracle.sol L336 — loops N assets]
       └─ getTotalAssetDeposits(asset)    [LRTDepositPool.sol L385]
            └─ getAssetDistributionData() [LRTDepositPool.sol L447 — loops M NDCs]
                 └─ getAssetUnstaking()   [NodeDelegator.sol L409 — loops K×S]
                      └─ sharesToUnderlyingView()  [external call per strategy]
```

Gas ≈ O(N × M × K × S × ~5,000 gas/call). At N=5, M=10, K=80, S=2: **≥40M gas > 30M block limit**.
