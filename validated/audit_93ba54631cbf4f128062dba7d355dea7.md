### Title
Unauthenticated `ProposalInit.proposer` Enables First-Write-Wins Slot Squatting in Future-Proposal Cache, Causing Consensus DOS — (`crates/apollo_consensus/src/manager.rs`)

---

### Summary

`cache_future_proposal` stores the first proposal received for each `(height, round)` pair using `.or_insert()` semantics. The only guard before caching is a check that `init.proposer` equals the committee-derived expected proposer address. Because `ProposalInit.proposer` is an unauthenticated `ContractAddress` field with no cryptographic binding, any peer can forge it to match the expected proposer and pass the check. The first forged proposal permanently occupies the cache slot; the legitimate proposer's subsequent proposal is silently dropped. When the height/round arrives, the node validates the malicious (invalid or never-finalized) proposal, fails or times out, votes nil, and the round is lost. The developers explicitly acknowledge this in a `TODO` comment at the call site.

---

### Finding Description

**Root cause — `cache_future_proposal` (first-write-wins, no authentication):** [1](#0-0) 

```rust
fn cache_future_proposal(…) {
    self.future_proposals_cache
        .entry(init.height)
        .or_default()
        .entry(init.round)
        .or_insert((init, content_receiver));   // ← first writer wins; subsequent proposals dropped
}
```

**Guard that is bypassable — proposer check in `handle_proposal`:** [2](#0-1) 

The check compares `init.proposer` (a plain `ContractAddress` from the wire) against the committee-derived expected proposer. `ProposalInit` carries no signature: [3](#0-2) 

Any peer can set `init.proposer` to the expected address and pass the check.

**Developer acknowledgment of the attack:** [4](#0-3) 

```
// Note: new proposals with the same height/round will be ignored.
//
// TODO(matan): This only work for trusted peers. In the case of
// possibly malicious peers this is a
// possible DoS attack (malicious
// users can insert invalid/bad/malicious proposals before
// "good" nodes can propose).
//
// When moving to version 1.0 make sure this is addressed.
```

**Secondary surface — `queued_proposals.insert` (last-write-wins for current-height future rounds):** [5](#0-4) 

`BTreeMap::insert` overwrites any existing entry for the same round. A malicious peer who sends their forged proposal *after* the legitimate one (trivially achievable by observing the network) replaces the legitimate queued proposal with a fake one.

**Downstream validation path that consumes the poisoned cache entry:** [6](#0-5) 

`validate_proposal` calls `initiate_validation` (batcher) and then loops on `handle_proposal_part`. A fake proposal that never sends `ProposalFin` causes a `ValidationTimeout`; one with invalid transactions causes `InvalidProposal`. Either outcome causes the node to vote nil.

---

### Impact Explanation

A malicious validator (any committee member) can:

1. Observe the committee schedule to learn the expected proposer for `(H+1, round 0)`.
2. Send a `ProposalInit` with `proposer = expected_address`, `height = H+1`, `round = 0`, and a content stream that never delivers `ProposalFin`.
3. The `.or_insert()` call caches this fake proposal; the legitimate proposer's subsequent proposal is silently dropped.
4. When height `H+1` starts, the node validates the fake proposal, times out, and votes nil.
5. Consensus advances to round 1; the attacker repeats for round 1, and so on.

This permanently blocks block production for the targeted height. The attacker pays no bond and faces no burn penalty. The only recovery is a round timeout per attempt, which the attacker can sustain indefinitely.

---

### Likelihood Explanation

- **Trigger**: Any committee member; no privileged access required beyond being a validator.
- **Effort**: Trivial — forge one field (`proposer`) in a `ProposalInit` message and send it before the legitimate proposer.
- **Detection**: The node logs a warning but silently drops the legitimate proposal with no alert.
- **Repeatability**: Unlimited; the attacker can repeat for every round of every height.
- **Current mitigation**: The `TODO` comment states this "only works for trusted peers." Once the validator set expands beyond the current closed trusted set, the attack becomes immediately exploitable.

---

### Recommendation

1. **Authenticate `ProposalInit`**: Add a cryptographic signature over `ProposalInit` fields (height, round, proposer, timestamp, gas prices) verifiable against the proposer's registered public key. Reject any proposal whose signature does not verify before caching.
2. **Replace `.or_insert()` with rejection-on-duplicate**: After authentication, a second proposal for the same `(height, round)` from the same proposer is equivocation and should be reported, not silently dropped.
3. **Replace `queued_proposals.insert` with a guard**: Check whether an entry already exists for the round before inserting; if one exists and the new proposal differs, treat it as equivocation.
4. **Short-term**: Until signatures are added, rate-limit proposals per peer per `(height, round)` and log/report duplicates.

---

### Proof of Concept

```
Attacker (malicious validator V_evil):

1. Observe committee: expected proposer for (H+1, round=0) = V_good_addr.
2. Construct ProposalInit {
       height: H+1, round: 0,
       proposer: V_good_addr,   // forged — no signature required
       timestamp: <valid>,
       l2_gas_price_fri: <valid>,
       ...
   }
3. Open a proposal stream, send the forged ProposalInit as the first part.
   Never send ProposalFin (or send invalid transactions).
4. Transmit to all validators before V_good sends its real proposal.

Result on each honest validator node:
- handle_proposal() checks proposer == V_good_addr → passes.
- cache_future_proposal() stores the fake entry via .or_insert().
- V_good's real proposal arrives → .or_insert() is a no-op → dropped silently.
- At height H+1: validate_proposal() consumes the fake stream → ValidationTimeout.
- Node votes nil → round 0 fails.
- Attacker repeats for round 1, 2, … → height H+1 never finalizes.
