Audit Report

## Title
Slashing-Induced Price Drop Exceeding `pricePercentageLimit` Permanently Freezes Protocol Until Admin Intervention - (File: contracts/LRTOracle.sol)

## Summary
In `LRTOracle._updateRsETHPrice()`, when a slashing event causes `newRsETHPrice` to fall more than `pricePercentageLimit` below `highestRsethPrice`, the function pauses the entire protocol and returns without updating `rsETHPrice`. Because `highestRsethPrice` is monotonically non-decreasing and has no admin setter, and because `updateRSETHPriceAsManager()` invokes the same internal path without any bypass, every subsequent call re-triggers the same condition, keeping the protocol frozen indefinitely until the admin disables the limit check entirely.

## Finding Description
`_updateRsETHPrice()` implements a downside-protection guard at lines 270–282:

```solidity
if (newRsETHPrice < highestRsethPrice) {
    uint256 diff = highestRsethPrice - newRsETHPrice;
    bool isPriceDecreaseOffLimit =
        pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);

    if (isPriceDecreaseOffLimit) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;   // rsETHPrice is NOT updated
    }
``` [1](#0-0) 

`highestRsethPrice` is only ever written upward:

```solidity
if (newRsETHPrice > highestRsethPrice) {
    highestRsethPrice = newRsETHPrice;
}
``` [2](#0-1) 

There is no function in the contract that decreases or resets `highestRsethPrice`. The only admin setter is `setPricePercentageLimit`, which can only disable the guard entirely by setting it to zero. [3](#0-2) 

`updateRSETHPriceAsManager()` is documented as a bypass for the *upside* threshold case, but it calls the identical `_updateRsETHPrice()` path with no special flag:

```solidity
function updateRSETHPriceAsManager() external onlyLRTManager {
    _updateRsETHPrice();
}
``` [4](#0-3) 

The comment on the function confirms it was designed only for the upside case ("to be able to update the price in case of the price going above the threshold"), leaving the downside freeze path with no manager-level escape. [5](#0-4) 

After a slashing event that drops the price by more than `pricePercentageLimit` relative to `highestRsethPrice`:

1. Any public caller triggers `updateRSETHPrice()` (no access control, only `whenNotPaused`).
2. `isPriceDecreaseOffLimit` is true; deposit pool, withdrawal manager, and oracle are all paused; function returns without updating `rsETHPrice`.
3. `updateRSETHPrice()` is now blocked by `whenNotPaused`.
4. `updateRSETHPriceAsManager()` bypasses `whenNotPaused` but hits the same `return` — price is still never updated.
5. Because `highestRsethPrice` never decreases, the condition fires on every future call as long as the actual price remains below the threshold.
6. The only escape is admin calling `setPricePercentageLimit(0)`, then unpausing all three contracts, then triggering a price update — a multi-step privileged sequence that is not atomic.

## Impact Explanation
All user deposits and withdrawals are frozen for the entire duration between the slashing event and the completion of the admin recovery sequence. The freeze is not self-resolving: it persists indefinitely without privileged intervention. This constitutes **temporary (potentially indefinite) freezing of funds** for all rsETH holders and depositors, matching the allowed Medium impact scope.

## Likelihood Explanation
Slashing is a known, realistic event in any ETH-staking protocol. `pricePercentageLimit` is expected to be set to a small value (e.g., 1–5%) to guard against oracle manipulation. A single large slashing incident, or an accumulation of smaller ones between oracle updates, can produce a price drop exceeding this limit relative to `highestRsethPrice`. The entry point (`updateRSETHPrice()`) is public and permissionless — any user or keeper calling it after a slashing event triggers the freeze. No attacker capability is required; the condition is triggered by normal protocol operation.

## Recommendation
1. **Introduce a `highestRsethPrice` admin/manager setter** so the protocol can ratchet the peak price down after a confirmed slashing event without disabling the guard entirely.
2. Alternatively, measure the downside limit against `rsETHPrice` (the last confirmed price) rather than `highestRsethPrice` (the all-time peak), so the threshold is not permanently anchored to a historical high that may never be reached again after slashing.
3. Ensure `updateRSETHPriceAsManager()` can bypass the pause-and-return path (e.g., with a `forceUpdate` flag or a separate internal function) so a manager can push the corrected price through without first disabling the entire guard.

## Proof of Concept
1. Protocol operates normally; `highestRsethPrice = 1.10e18`, `rsETHPrice = 1.09e18`, `pricePercentageLimit = 2e16` (2%).
2. A large slashing event reduces underlying ETH backing; `_getTotalEthInProtocol()` now yields a price of `0.90e18`.
3. Anyone calls `updateRSETHPrice()`.
4. Inside `_updateRsETHPrice()`: `diff = 1.10e18 - 0.90e18 = 0.20e18`; `pricePercentageLimit.mulWad(highestRsethPrice) = 0.02 * 1.10e18 = 0.022e18`; `0.20e18 > 0.022e18` → `isPriceDecreaseOffLimit = true`.
5. Deposit pool, withdrawal manager, and oracle are all paused. Function returns. `rsETHPrice` remains `1.09e18` (stale).
6. Manager calls `updateRSETHPriceAsManager()`. Same `_updateRsETHPrice()` path executes; same condition fires; same early return. Price still `1.09e18`.
7. `highestRsethPrice` is still `1.10e18` (no setter). Every future call repeats step 6.
8. All user withdrawals and deposits are frozen until admin executes the multi-step recovery sequence: `setPricePercentageLimit(0)` → unpause all three contracts → trigger price update.

**Foundry test plan:** Deploy `LRTOracle` with `pricePercentageLimit = 2e16`; set `highestRsethPrice = 1.10e18` and `rsETHPrice = 1.09e18` via storage manipulation or normal operation; mock `_getTotalEthInProtocol()` to return a value yielding `newRsETHPrice = 0.90e18`; call `updateRSETHPrice()` as an unprivileged address; assert all three contracts are paused and `rsETHPrice` is unchanged; call `updateRSETHPriceAsManager()` as manager; assert `rsETHPrice` is still unchanged; assert `highestRsethPrice` is still `1.10e18`; confirm no non-admin path can unfreeze the protocol.
