### Title
Cross-Chain Replay via Missing Completed-Request Tracking in `verify_foreign_transaction` - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint in the MPC contract has no mechanism to record that a given foreign-chain transaction has already been verified and signed. After the pending-request queue for a `(domain_id, payload_version, tx_id)` tuple is drained by `respond_verify_foreign_tx`, the contract retains no memory of the completed request. Any unprivileged caller can immediately re-submit the identical `tx_id`, causing the MPC network to verify and produce a fresh threshold signature over the same foreign-chain transaction payload. This enables a bridge double-spend: the attacker collects multiple independent MPC signatures for the same foreign-chain event and redeems them sequentially on the destination chain.

---

### Finding Description

`verify_foreign_transaction` accepts a `VerifyForeignTransactionRequestArgs` containing `(domain_id, payload_version, ForeignChainRpcRequest{tx_id, …})`. [1](#0-0) 

The request is converted to a `VerifyForeignTransactionRequest` key and inserted into `pending_verify_foreign_tx_requests`. The key is **caller-agnostic** and contains no nonce, no NEAR block height, and no receipt ID — only the foreign-chain parameters. [2](#0-1) 

When `respond_verify_foreign_tx` is called by an attested MPC node, it validates the signature and calls `resolve_yields_for`, which drains the entire pending queue and **removes the map entry entirely**. [3](#0-2) 

After removal, the contract holds **no record** that this `tx_id` was ever processed. There is no `completed_verify_foreign_tx_requests` set, no nonce mapping, and no tombstone. The same `tx_id` can be submitted again in a new NEAR transaction, creating a fresh pending entry that the MPC nodes will process identically.

The `ForeignTxSignPayload` that the MPC nodes sign contains only the foreign-chain request and extracted values — no NEAR-specific nonce or receipt ID: [4](#0-3) 

Each replay produces a cryptographically independent but equally valid threshold signature over the same message `(tx_id, extracted_values)`. The node-side `VerifyForeignTransactionRequestStorage` deduplicates only within a single node's in-memory/DB lifetime keyed by the NEAR receipt ID, not by `tx_id` across separate NEAR transactions. [5](#0-4) 

---

### Impact Explanation

The `verify_foreign_transaction` feature is explicitly designed for bridge inbound flows (e.g., Omnibridge: foreign chain → NEAR). The MPC signature over `(tx_id, block_hash, …)` is the attestation that a bridge contract on the destination chain uses to release funds. Because the MPC contract issues a fresh, valid threshold signature on every replay, an attacker who controls the NEAR-side submission can accumulate N independent MPC signatures for the same foreign-chain deposit transaction and redeem them N times on the destination bridge contract — a direct double-spend of bridged funds.

Impact classification: **High — Cross-chain replay causing invalid bridge execution / double-spend conditions.**

---

### Likelihood Explanation

The attack requires only an unprivileged NEAR account and 1 yoctoNEAR per submission. No threshold collusion, no key material, and no privileged access are needed. The attacker simply re-submits the same `VerifyForeignTransactionRequestArgs` after each response is delivered. The MPC nodes have no on-chain signal to refuse the re-submission. Likelihood is **High** for any bridge that does not independently track consumed MPC attestations on the destination chain.

---

### Recommendation

Maintain a persistent set of completed `VerifyForeignTransactionRequest` keys (or a hash thereof) in contract storage. Before enqueuing a new yield in `verify_foreign_transaction`, check whether the key already appears in this completed set and panic if so. Alternatively, include a NEAR-specific nonce (e.g., the predecessor's account ID combined with a per-account counter, or the NEAR block height) inside `ForeignTxSignPayload` so that each signature is bound to a single NEAR transaction and cannot be reused on the destination chain even if re-signing occurs.

---

### Proof of Concept

1. Attacker calls `verify_foreign_transaction({ tx_id: X, domain_id: D, … })` on the MPC contract (NEAR receipt R1, deposit 1 yoctoNEAR).
2. MPC nodes observe R1, query the foreign chain, and call `respond_verify_foreign_tx`. The contract resolves the pending yield and removes the map entry. Attacker receives `sig1` over `hash(tx_id=X, block_hash=H, …)`.
3. Attacker immediately calls `verify_foreign_transaction({ tx_id: X, domain_id: D, … })` again (NEAR receipt R2).
4. The contract finds **no entry** in `pending_verify_foreign_tx_requests` for this key (it was removed in step 2) and enqueues a fresh yield. [6](#0-5) 

5. MPC nodes observe R2 (different receipt ID → different `VerifyForeignTxId` → passes `VerifyForeignTransactionRequestStorage.add()` deduplication), query the foreign chain again, and produce `sig2` — a distinct but equally valid threshold signature over the same message.
6. Attacker submits `sig1` to the destination bridge contract → funds released.
7. Attacker submits `sig2` to the destination bridge contract → funds released again (double-spend).

The existing sandbox test explicitly confirms that after `respond_verify_foreign_tx` the map entry is `None`, leaving no barrier to re-submission: [6](#0-5)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
