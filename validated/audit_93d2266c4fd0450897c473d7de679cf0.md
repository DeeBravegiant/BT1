### Title
NFT Royalty Bypass via Integer Division Truncation in Offer Settlement — (`chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
`compute_royalty_amount` uses sequential integer floor-division that truncates to zero when `(abs(offered_amount) // royalty_split) * percentage < MAX_ROYALTY_BASIS_POINTS` (10 000). When the computed royalty is zero, `make_nft1_offer` silently skips the royalty coin-spend and passes an empty `trade_prices_list` to the NFT transfer-program puzzle. The CLVM puzzle enforces only what the spender provides; with an empty list it creates no royalty conditions. Any unprivileged user can therefore settle a royalty-enabled NFT1 offer on-chain without paying the creator's royalty.

### Finding Description

`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py`:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

`MAX_ROYALTY_BASIS_POINTS = 10 000`. Because Python integer division truncates, the result is 0 whenever:

```
(abs(offered_amount) // royalty_split) * percentage < 10 000
```

The existing test suite explicitly documents this as accepted behaviour:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)   # 1 % royalty on 50 mojos → 0
``` [2](#0-1) 

When `compute_royalty_amount` returns 0, two downstream guards in `make_nft1_offer` both silently drop the royalty:

**Guard 1** — royalty coin-spend is skipped entirely:
```python
if sum(p.amount for _, p in payments) == 0:
    continue
``` [3](#0-2) 

**Guard 2** — `trade_prices_list` passed to the NFT puzzle is filtered to empty:
```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [4](#0-3) 

The NFT transfer program (`NFT_TRANSFER_PROGRAM_DEFAULT`) derives royalty conditions solely from the caller-supplied `trade_prices_list`. With an empty list the CLVM puzzle emits no royalty `CREATE_COIN` conditions, and the spend is accepted by consensus — as confirmed by the lifecycle test that passes a spend with no trade prices: [5](#0-4) 

### Impact Explanation

An unprivileged attacker can settle a royalty-enabled NFT1 offer on-chain and pay **zero royalty** to the NFT creator, regardless of the royalty percentage encoded in the NFT's ownership-layer puzzle. This is an unauthorized accounting change affecting NFT-linked coins during offer settlement — matching the Critical impact category.

Threshold amounts at which royalty becomes zero (single-NFT offer, `royalty_split = 1`):

| Royalty (bps) | Max offered amount for zero royalty |
|---|---|
| 1 (0.01 %) | < 10 000 mojos |
| 100 (1 %) | < 100 mojos |
| 1 000 (10 %) | < 10 mojos |
| 9 999 (99.99 %) | < 2 mojos (i.e. 1 mojo) |

### Likelihood Explanation

Any user can create and publish an offer file. The attacker only needs to choose an offered amount below the truncation threshold. No privileged access, leaked keys, or special network position is required. The offer is valid and accepted by the full node.

### Recommendation

Replace the two-step integer division with a single multiplication before division to preserve precision:

```python
# Before (truncates to zero for small amounts):
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS

# After (multiply first, then divide once):
amount = abs(offered_amount) * percentage // (royalty_split * MAX_ROYALTY_BASIS_POINTS)
```

Additionally, add a minimum-royalty guard: if `percentage > 0` and the computed royalty is 0, either reject the offer or enforce a minimum royalty of 1 mojo. The same fix should be applied to `NFTWallet.royalty_calculation` which contains the identical pattern: [6](#0-5) 

### Proof of Concept

1. Mint an NFT1 with `royalty_percentage = 100` (1 %) and any `royalty_address`.
2. Call `make_nft1_offer` offering **50 mojos** of XCH in exchange for the NFT.
3. Internally, `compute_royalty_amount(-50, 1, 100)` → `50 // 1 * 100 // 10000` → `5000 // 10000` → **0**.
4. Both guards fire: no royalty coin-spend is added; `trade_prices_list` passed to the NFT puzzle is `[]`.
5. The taker accepts the offer; the spend bundle is submitted and confirmed on-chain.
6. The NFT creator receives **0 mojos** in royalties despite the 1 % royalty encoded in the puzzle.

The test `test_small_amount_truncates_to_zero` already asserts this exact arithmetic result as expected behaviour, confirming the truncation is reachable and unguarded in production offer-creation code. [7](#0-6) [8](#0-7)

MAX_ROYALTY_BASIS_POINTS = 10000


def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
