### Title
Sub-Transaction Value Conservation Not Enforced in Dijkstra Era SUBUTXO Rule — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

The Dijkstra era introduces sub-transactions (nested transactions within a top-level transaction). The `SUBUTXO` rule that processes each sub-transaction deliberately omits the `validateValueNotConservedUTxO` check. The top-level `UTXO` rule's value conservation check only covers the top-level transaction body and uses the pre-sub-transaction UTxO snapshot (`originalUtxo`), leaving sub-transaction inputs and outputs entirely outside any conservation accounting. An unprivileged user can craft a sub-transaction whose outputs exceed its inputs, creating ADA from thin air.

---

### Finding Description

**Root cause — missing check in `dijkstraSubUtxoTransition`:**

The `dijkstraSubUtxoTransition` function in `SubUtxo.hs` performs validity-interval, forecast, output-size, input-set, bad-inputs, network-ID, and output-minimum checks, but never calls `validateValueNotConservedUTxO`. [1](#0-0) 

The mapping function `dijkstraUtxoToDijkstraSubUtxoPredFailure` makes this explicit — both `ValueNotConservedUTxO` and `FeeTooSmallUTxO` are marked `error "Impossible"` for the `SUBUTXO` rule, confirming the omission is structural: [2](#0-1) 

**Top-level check does not cover sub-transactions:**

The top-level `dijkstraUtxoTransition` does enforce value conservation, but only for the top-level `txBody` against `originalUtxo` (the UTxO snapshot taken *before* any sub-transactions ran): [3](#0-2) 

`validateValueNotConservedUTxO` computes `consumed` from `txBody ^. inputsTxBodyL` and `produced` from `txBody ^. outputsTxBodyL` — neither of which includes sub-transaction inputs or outputs: [4](#0-3) 

**Sub-transaction UTxO changes are applied without conservation check:**

`updateUTxOStateNoFees` (called from `dijkstraSubUtxoTransition`) mechanically removes inputs and adds outputs to the UTxO, updating only the deposit pot — it performs no balance check: [5](#0-4) 

**Top-level inputs cannot reference sub-transaction outputs:**

The top-level UTXO rule enforces that top-level spending inputs must exist in `originalUtxo`: [6](#0-5) 

This means the ADA created by an unbalanced sub-transaction is deposited into the UTxO at an attacker-controlled address and is immediately spendable in a subsequent transaction — the top-level transaction cannot spend it in the same batch, but any later transaction can.

**Exploit path:**

1. Attacker submits a Dijkstra-era top-level transaction containing a sub-transaction.
2. The sub-transaction spends a UTxO input worth `X` lovelace and creates an output worth `X + N` lovelace at an attacker-controlled address.
3. `SUBLEDGERS` → `SUBLEDGER` → `SUBUTXOW` → `SUBUTXO` applies the UTxO change without checking conservation.
4. The top-level value conservation check passes because it only examines the top-level `txBody` against `originalUtxo`.
5. `N` lovelace is permanently added to the UTxO.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An unprivileged transaction author can mint arbitrary amounts of ADA by embedding an unbalanced sub-transaction. This violates the fundamental preservation-of-value invariant of the Cardano ledger: [7](#0-6) 

The created ADA is indistinguishable from legitimate ADA and can be freely spent, staked, or used to dilute the total supply, causing direct financial harm to all ADA holders.

---

### Likelihood Explanation

Any user who can submit a valid Dijkstra-era transaction can trigger this. No privileged keys, governance majority, or special role is required. The sub-transaction body is entirely attacker-controlled. The Dijkstra era is the current production era (protocol version 12): [8](#0-7) 

---

### Recommendation

Add `validateValueNotConservedUTxO` (or an equivalent combined conservation check covering all sub-transactions) to `dijkstraSubUtxoTransition` in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`. Alternatively, implement a batch-level conservation check in the `SUBLEDGERS` rule that verifies the aggregate value change across all sub-transactions is zero (net inputs = net outputs + net deposit change). The `DijkstraSubUtxoPredFailure` type must be extended with a `SubValueNotConservedUTxO` constructor, and the `dijkstraUtxoToDijkstraSubUtxoPredFailure` mapping must be updated accordingly (removing the `error "Impossible"` for `ValueNotConservedUTxO`).

updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let UTxOState {utxosUtxo, utxosDeposited, utxosFees, utxosDonation} = utxos
      UTxO utxo = utxosUtxo
      !utxoAdd = txouts txBody -- These will be inserted into the UTxO
      {- utxoDel  = txins txb ◁ utxo -}
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      {- newUTxO = (txins txb ⋪ utxo) ∪ outs txb -}
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      deletedUTxO = UTxO utxoDel
      totalRefunds = certsTotalRefundsTxBody pp certState txBody
      totalDeposits = certsTotalDepositsTxBody pp certState txBody
      depositChange = totalDeposits <-> totalRefunds
  depositChangeEvent depositChange
  txUtxODiffEvent deletedUTxO utxoAdd
  pure $!
    UTxOState
      { utxosUtxo = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees = utxosFees
      , utxosGovState = govState
      , utxosInstantStake =
          deleteInstantStake deletedUTxO (addInstantStake utxoAdd (utxos ^. instantStakeL))
      , utxosDonation = utxosDonation
