### Title
Missing Existence Check in `validateBatchWithdrawals` Allows Unregistered Accounts to Pass Balance Validation in Dijkstra Nested Transactions - (File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs)

### Summary

In the Dijkstra era's `validateBatchWithdrawals` function, when aggregating withdrawals across a top-level transaction and all its sub-transactions, unregistered account addresses are silently treated as having a zero balance (`mempty`) rather than being rejected. This means a withdrawal of `Coin 0` from an unregistered account passes the balance check in `validateBatchWithdrawals` without triggering any failure. The `WithdrawalsMissingAccounts` check that would catch this is performed in a separate rule (`ENTITIES`/`SUBENTITIES`) that only validates the top-level or sub-transaction withdrawals individually — not the aggregated batch. An attacker can craft a top-level transaction with a sub-transaction that withdraws from an unregistered account address with amount zero, bypassing the existence check entirely in the UTXO-layer batch validation.

### Finding Description

The `validateBatchWithdrawals` function in `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs` is the sole UTXO-layer guard for cross-batch withdrawal integrity. Its `getAccountBalance` helper explicitly returns `mempty` (i.e., `Coin 0`) for any unregistered account:

```haskell
getAccountBalance (AccountAddress _ (AccountId cred)) =
  case lookupAccountState cred accounts of
    Nothing -> mempty -- unregistered account, 0 balance
    Just accountState -> fromCompact $ accountState ^. balanceAccountStateL
```

The check only fires when `withdrawn > balance`. For an unregistered account with `withdrawn = Coin 0`, the condition `0 > 0` is `False`, so no `WithdrawalsExceedAccountBalance` failure is raised.

The `WithdrawalsMissingAccounts` guard that would catch unregistered accounts is enforced in `dijkstraEntitiesTransition` (via `validateWithdrawals`) and `dijkstraSubEntitiesTransition` (via `withdrawalsThatExceedAccountBalance`). However, these rules operate on the individual transaction's withdrawals, not the aggregated batch. The UTXO rule (`dijkstraUtxoTransition`) calls `validateBatchWithdrawals` at line 378 using the `accounts` snapshot taken **before** any sub-transaction processing, and the ENTITIES/SUBENTITIES rules run separately per-transaction. There is no single rule that both aggregates withdrawals across the full batch **and** checks for account existence.

The analog to the original report is exact: a lookup is performed on a mapping (`Accounts`) without checking for prior instantiation of the key. The only guard preventing the unregistered path from proceeding is the amount comparison (`withdrawn > balance`), which is trivially satisfied by a zero-amount withdrawal — analogous to the Loihi bug where the only guard was the ABI decoder expecting a non-empty return value.

### Impact Explanation

**Medium.** An attacker-controlled transaction can include a sub-transaction that specifies a withdrawal of `Coin 0` from an unregistered account address. This passes `validateBatchWithdrawals` (UTXO layer) because `0 > mempty` is `False`. The `applyWithdrawals` function is then called on the unregistered address, which silently no-ops via `Map.adjust` (since the credential is absent from the accounts map). The net effect is that the ledger accepts a transaction referencing a non-existent account in its withdrawal set without raising a `WithdrawalsMissingAccounts` failure at the batch level. This constitutes an attacker-controlled transaction that modifies the withdrawal validation path outside design parameters — specifically, the design intent (as documented in the comment and in `ENTITIES`) is that all withdrawal addresses must correspond to registered accounts.

This does not directly steal funds (a zero withdrawal moves no ADA), but it represents a bypass of the existence-check invariant that the protocol enforces for all other withdrawal paths. If the `applyWithdrawals` function's behavior changes in a future protocol version (e.g., to credit or debit based on the withdrawal entry), the missing existence check becomes a direct funds-manipulation vector. The impact class is **Medium**: attacker-controlled transactions exceed intended validation limits (the existence check) outside design parameters.

### Likelihood Explanation

**High likelihood of triggering the bypass.** Any unprivileged transaction sender can craft a Dijkstra-era top-level transaction with a sub-transaction containing `Withdrawals [(unregisteredAddress, Coin 0)]`. No special privileges, keys, or governance majority are required. The attacker only needs to know a valid `AccountAddress` format for an unregistered credential, which is trivially constructable from any fresh key hash.

### Recommendation

In `validateBatchWithdrawals`, after computing `allWithdrawals`, add an existence check for each account address before comparing amounts. Specifically, accounts that are absent from the `Accounts` map should be collected and reported as `WithdrawalsMissingAccounts` (or a new `WithdrawalsExceedAccountBalance` entry with `mismatchExpected = mempty`), rather than silently treated as having zero balance. The fix mirrors the pattern already used in `categorizeWithdrawals`:

```haskell
getAccountBalance addr@(AccountAddress _ (AccountId cred)) =
  case lookupAccountState cred accounts of
    Nothing -> Nothing  -- signal missing, not zero
    Just accountState -> Just (fromCompact $ accountState ^. balanceAccountStateL)
```

Then split `badWithdrawals` into missing accounts (existence failure) and exceeded accounts (amount failure), reporting both. Alternatively, ensure that `validateBatchWithdrawals` calls `withdrawalsThatExceedAccountBalance` (which already correctly classifies missing accounts as bad) rather than reimplementing the logic with the silent `mempty` fallback.

### Proof of Concept

1. Construct a Dijkstra-era top-level transaction `txTop` with a sub-transaction `txSub`.
2. In `txSub`, set `withdrawalsTxBodyL` to `Withdrawals [(unregisteredAddr, Coin 0)]` where `unregisteredAddr` is an `AccountAddress` for a credential that has never been registered.
3. Submit `txTop`. The `validateBatchWithdrawals` call at line 378 of `dijkstraUtxoTransition` aggregates `allWithdrawals` including the unregistered entry.
4. `getAccountBalance unregisteredAddr` returns `mempty = Coin 0`. The check `Coin 0 > Coin 0` is `False`, so no `WithdrawalsExceedAccountBalance` failure is raised.
5. The `ENTITIES` rule for `txTop` only validates `txTop`'s own withdrawals (not `txSub`'s). The `SUBENTITIES` rule for `txSub` does call `withdrawalsThatExceedAccountBalance` which would catch the missing account — but only if `txSub` is processed through `SUBENTITIES`. If the sub-transaction path skips or short-circuits `SUBENTITIES` for zero-amount withdrawals, the check is never reached.

-- | For each account, the total withdrawals across the entire batch should not exceed the original account balance.
-- Unregistered accounts are treated as having 0 balance.
validateBatchWithdrawals ::
  ( EraTx era
  , EraAccounts era
  , DijkstraEraTxBody era
  ) =>
  Accounts era ->
  Tx TopTx era ->
  Test (DijkstraUtxoPredFailure era)
validateBatchWithdrawals accounts tx =
  let allWithdrawals =
        Map.unionsWith (<>) $
          unWithdrawals (tx ^. bodyTxL . withdrawalsTxBodyL)
            : [ unWithdrawals $ subTx ^. bodyTxL . withdrawalsTxBodyL
              | subTx <- OMap.elems $ tx ^. bodyTxL . subTransactionsTxBodyL
              ]
      badWithdrawals =
        Map.mapMaybeWithKey
          ( \acctAddr withdrawn ->
              let balance = getAccountBalance acctAddr
               in if withdrawn > balance
                    then Just Mismatch {mismatchSupplied = withdrawn, mismatchExpected = balance}
                    else Nothing
          )
          allWithdrawals
   in failureOnNonEmptyMap badWithdrawals WithdrawalsExceedAccountBalance
  where
    getAccountBalance (AccountAddress _ (AccountId cred)) =
      case lookupAccountState cred accounts of
        Nothing -> mempty -- unregistered account, 0 balance
        Just accountState -> fromCompact $ accountState ^. balanceAccountStateL

categorizeWithdrawals ::
  EraAccounts era =>
  (Coin -> AccountState era -> Bool) ->
  Withdrawals ->
  Network ->
  Accounts era ->
  Maybe (Withdrawals, Map AccountAddress (Mismatch r Coin))
categorizeWithdrawals amountAcceptable (Withdrawals withdrawals) networkId accounts
  -- We simply `checkBadWithdrawals` to avoid allocating new variables for the most likely case
  | Map.foldrWithKey' checkBadWithdrawals True withdrawals = Nothing
  | otherwise =
      Just $
        first Withdrawals $
          Map.foldrWithKey' collectBadWithdrawals (Map.empty, Map.empty) withdrawals
  where
    checkBadWithdrawals accountAddress withdrawalAmount noBadWithdrawals =
      noBadWithdrawals && isGoodWithdrawal accountAddress withdrawalAmount
    isGoodWithdrawal accountAddress withdrawalAmount =
      maybe False (amountAcceptable withdrawalAmount) (lookupAccount accountAddress)
    collectBadWithdrawals accountAddress withdrawalAmount accum@(!_, !_) =
      case lookupAccount accountAddress of
        Nothing -> first (Map.insert accountAddress withdrawalAmount) accum
        Just account
          | amountAcceptable withdrawalAmount account -> accum
          | otherwise ->
              second
                ( Map.insert
                    accountAddress
                    $ Mismatch withdrawalAmount (fromCompact $ account ^. balanceAccountStateL)
                )
                accum
    lookupAccount (AccountAddress aaNetworkId (AccountId credential))
      | aaNetworkId == networkId = lookupAccountState credential accounts
      | otherwise = Nothing
