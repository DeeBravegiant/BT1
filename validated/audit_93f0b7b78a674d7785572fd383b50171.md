### Title
Missing Infinity-Point Guard in `ecc_pub_key_t::verify_schnorr` Enables Signature Forgery for Any Message — (`src/cbmpc/crypto/base_ecc.cpp`)

---

### Summary

`ecc_pub_key_t::verify_schnorr` performs no validity check on the public key `Q` before using it in the Schnorr verification equation. If `Q` is the point at infinity, the scalar multiplication `e * Q` collapses to infinity for any `e`, making `R = s*G + e*Q = s*G`. An attacker who can present an infinity public key can then freely choose any `s`, compute `R = s*G`, derive `e = SHA256(∞, R, message) % q`, and produce a pair `(e, s)` that passes verification for any message. This requires no knowledge of any private key.

---

### Finding Description

The verification function at `src/cbmpc/crypto/base_ecc.cpp:373–391` is:

```cpp
error_t ecc_pub_key_t::verify_schnorr(mem_t message, mem_t signature) const {
  if (signature.size != curve.size() * 2) return coinbase::error(E_FORMAT);
  const auto& G = curve.generator();
  const mod_t& q = curve.order();
  ecc_point_t Q = *this;                                    // no validity check on Q

  bn_t e = bn_t::from_bin(signature.take(curve.size()));
  bn_t s = bn_t::from_bin(signature.skip(curve.size()));

  if (e <= 0 || e >= q) return coinbase::error(E_CRYPTO);
  if (s <= 0 || s >= q) return coinbase::error(E_CRYPTO);

  ecc_point_t R = s * G + e * Q;                           // if Q=∞, R = s*G always

  bn_t e_tag = bn_t(crypto::sha256_t::hash(Q, R, message)) % q;
  if (e_tag != e) return coinbase::error(E_CRYPTO);
  return SUCCESS;
}
``` [1](#0-0) 

There is no call to `curve.check(Q)`, `Q.is_infinity()`, or `Q.is_in_subgroup()` anywhere in this function. The only guards are the range checks on `e` and `s`.

**Why `allow_ecc_infinity_t` is irrelevant here.** The `allow_ecc_infinity_t` RAII guard only affects `ecurve_t::check()`:

```cpp
error_t ecurve_t::check(const ecc_point_t& point) const {
  ...
  if (!thread_local_store_allow_ecc_infinity) {
    if (point.is_infinity()) return crypto::error("EC-point is infinity");
  }
  return SUCCESS;
}
``` [2](#0-1) 

Since `verify_schnorr` never calls `curve.check()` on `Q`, the `allow_ecc_infinity_t` flag has no bearing on this attack. The forgery works whether or not `allow_ecc_infinity_t` is active.

**Why `is_in_subgroup` does not save it.** For secp256k1, `is_in_subgroup` delegates to `is_on_curve`, which calls `secp256k1_ge_is_valid_var`. That function explicitly returns 0 for infinity:

```c
static int secp256k1_ge_is_valid_var(const secp256k1_ge *a) {
    if (a->infinity) { return 0; }
    ...
}
``` [3](#0-2) 

So infinity is correctly identified as invalid — but `verify_schnorr` never invokes this check.

**Why the deserialization path does not save it.** The `convert_fixed_curve` deserializer does call `curve.check()` and would reject infinity:

```cpp
if (rv = curve.check(*this)) { converter.set_error(rv); return; }
``` [4](#0-3) 

However, `ecc_pub_key_t` can be constructed directly from any `ecc_point_t` via its constructor `ecc_pub_key_t(const ecc_point_t& P) : ecc_point_t(P) {}` with no validation, bypassing the deserializer entirely. [5](#0-4) 

---

### Impact Explanation

Any caller who constructs `ecc_pub_key_t(curve.infinity())` and calls `verify_schnorr` can forge a valid Schnorr signature for any message without knowing any private key. The forged signature is accepted as `SUCCESS`. This is a complete break of Schnorr signature security for the infinity-key case.

---

### Likelihood Explanation

The attack requires the caller to present an infinity public key. In normal protocol flows, public keys arrive via the serialization path which does call `curve.check()`. However, any internal code path that constructs an `ecc_pub_key_t` from a computed or received `ecc_point_t` without an explicit `curve.check()` call before invoking `verify_schnorr` is vulnerable. The missing guard is in the verifier itself, which is the correct place to enforce it.

---

### Recommendation

Add an explicit infinity (and subgroup) check at the top of `verify_schnorr`:

```cpp
error_t ecc_pub_key_t::verify_schnorr(mem_t message, mem_t signature) const {
  if (signature.size != curve.size() * 2) return coinbase::error(E_FORMAT);
  if (is_infinity()) return coinbase::error(E_CRYPTO);          // ADD THIS
  if (!is_in_subgroup()) return coinbase::error(E_CRYPTO);      // ADD THIS
  ...
}
```

This mirrors the guard already present in `ecurve_t::check()` and the deserialization path.

### Proof of Concept

```cpp
// No allow_ecc_infinity_t needed — verify_schnorr never calls curve.check()
ecurve_t curve = curve_secp256k1;
const mod_t& q = curve.order();
const auto& G  = curve.generator();

// Step 1: construct an infinity public key directly
ecc_pub_key_t Q_inf(curve.infinity());

// Step 2: pick any s in [1, q-1]
bn_t s = bn_t::rand(q);

// Step 3: compute R = s*G (a valid non-infinity point)
ecc_point_t R = s * G;

// Step 4: compute e = SHA256(Q_inf, R, message) % q
buf_t message = sha256_t::hash(mem_t("arbitrary message"));
bn_t e = bn_t(sha256_t::hash(Q_inf, R, message)) % q;
// (retry with different s if e == 0, which is negligible probability)

// Step 5: build signature (e || s) and verify
buf_t sig = e.to_bin(curve.size()) + s.to_bin(curve.size());
error_t rv = Q_inf.verify_schnorr(message, sig);
assert(rv == SUCCESS);  // passes — forged signature accepted
```

The forged `(e, s)` satisfies the verifier because:
- `e * Q_inf = e * ∞ = ∞`
- `R_computed = s*G + ∞ = s*G = R` ✓
- `e_tag = SHA256(Q_inf, R, message) % q = e` ✓
