Audit Report

## Title
Phantom `ethValueInWithdrawal` After Lido Slashing Permanently Inflates rsETH Price, Enabling Protocol Insolvency - (File: `contracts/LRTConverter.sol`)

## Summary

When stETH is transferred to `LRTConverter` via `transferAssetFromDepositPool`, `ethValueInWithdrawal` is incremented by the oracle-priced ETH value of the stETH. When `claimStEth` is later called, `_sendEthToDepositPool` decrements `ethValueInWithdrawal` only by the actual ETH received. If a Lido slashing event reduces the claimable amount below the tracked value, the difference remains as a permanent phantom balance in `ethValueInWithdrawal`. This phantom propagates through `getETHDistributionData` → `_getTotalEthInProtocol` → `rsETHPrice`, inflating the rsETH price above actual protocol value and allowing early withdrawers to drain assets from later withdrawers.

## Finding Description

**Root cause — `contracts/LRTConverter.sol`**

`transferAssetFromDepositPool` records the ETH value of stETH at oracle price:

```solidity
// L140
ethValueInWithdrawal += (_amount * lrtOracle.getAssetPrice(_asset)) / 1e18;
```

`claimStEth` sends the entire contract ETH balance to the deposit pool:

```solidity
// L180-183
function claimStEth(uint256 _requestId, uint256 _hint) external nonReentrant onlyLRTOperator {
    _claimStEth(_requestId, _hint);
    _sendEthToDepositPool(address(this).balance);
}
```

`_sendEthToDepositPool` decrements by actual ETH received:

```solidity
// L255-259
if (ethValueInWithdrawal > _amount) {
    ethValueInWithdrawal -= _amount;
} else {
    ethValueInWithdrawal = 0;
}
```

If Lido slashing reduces the claimable ETH from X to Y (Y < X), the residual `X − Y` remains permanently in `ethValueInWithdrawal`. There is no function to zero out or manually correct this phantom balance.

**Propagation — `contracts/LRTDepositPool.sol`**

`getETHDistributionData` reads `ethValueInWithdrawal` directly as `ethLyingInConverter`:

```solidity
// L498-499
address lrtConverter = lrtConfig.getContract(LRTConstants.LRT_CONVERTER);
ethLyingInConverter = ILRTConverter(lrtConverter).ethValueInWithdrawal();
```

`getTotalAssetDeposits` sums all components including `assetLyingInConverter`:

```solidity
// L395-396
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer + assetLyingInConverter
        + assetLyingUnstakingVault);
```

**Propagation — `contracts/LRTOracle.sol`**

`_getTotalEthInProtocol` calls `getTotalAssetDeposits` for every supported asset and accumulates the inflated total:

```solidity
// L341-343
uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
```

`_updateRsETHPrice` computes rsETH price from this inflated total:

```solidity
// L250
uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```

**Impact — `contracts/LRTWithdrawalManager.sol`**

`getExpectedAssetAmount` uses the stored (inflated) `rsETHPrice`:

```solidity
// L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

`initiateWithdrawal` locks this inflated amount into the withdrawal request:

```solidity
// L168-173
uint256 expectedAssetAmount = getExpectedAssetAmount(asset, rsETHUnstaked);
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`_processWithdrawalCompletion` pays out the locked inflated amount:

```solidity
// L734
_transferAsset(asset, user, request.expectedAssetAmount);
```

**Why existing guards are insufficient:**

1. **Downside protection in `_updateRsETHPrice` (L270-291)**: This pauses the protocol only if `pricePercentageLimit > 0` and the price drop exceeds the configured threshold. After `claimStEth`, the phantom is a small fraction of TVL (e.g., 20 ETH on a 540 ETH protocol), so the price drop may be well within the configured limit. If `pricePercentageLimit` is 0, no protection exists at all.

2. **`_calculatePayoutAmount` minimum (L833-834)**: This takes `min(expectedAssetAmount, currentReturn)` at `unlockQueue` time. Since the phantom persists after `claimStEth`, the rsETH price remains inflated when `unlockQueue` is called, so `currentReturn` equals `expectedAssetAmount` — the minimum provides no correction.

3. **No manual correction path**: There is no function to directly set `ethValueInWithdrawal` to zero or reduce it after a slashing event. `transferAssetToDepositPool` only decrements it when returning ERC20 tokens, not after ETH has already been claimed and sent.

## Impact Explanation

**Critical — Protocol insolvency.**

Concrete scenario: Protocol holds 600 stETH (≈600 ETH). Operator transfers 200 stETH to converter (`ethValueInWithdrawal` = 200). Lido slashing reduces all stETH by 10%. Remaining pool stETH = 360 stETH ≈ 360 ETH. Lido pays out 180 ETH on claim. After `claimStEth`: `ethValueInWithdrawal` = 20 (phantom). `_getTotalEthInProtocol` = 360 + 20 = 380 ETH; actual = 360 ETH. rsETH price is inflated by 20/supply. A user holding 1/3 of rsETH supply calls `initiateWithdrawal` and locks in ≈126.67 ETH (should be ≈120 ETH). After `completeWithdrawal`, remaining assets = 360 − 126.67 = 233.33 ETH for the remaining 2/3 of supply, but they are owed 240 ETH. The last withdrawer cannot be made whole. The phantom balance persists indefinitely, compounding with each subsequent slashing event.

## Likelihood Explanation

Low-to-medium. Lido slashing events are rare but have occurred historically and are explicitly acknowledged in Lido's documentation as a possible outcome of the withdrawal process. The `unstakeStEth`/`claimStEth` path is a live operational flow in this protocol. The vulnerability window spans the entire period between withdrawal request submission and claim finalization (days to weeks depending on Lido queue depth). No attacker action is required to cause the slashing; an attacker only needs to observe the event and call `initiateWithdrawal` before the phantom is corrected — which has no automatic correction path.

## Recommendation

In `_sendEthToDepositPool`, replace the partial decrement with a full reset when a claim is completed. The simplest fix is to set `ethValueInWithdrawal = 0` unconditionally after each `claimStEth` call (assuming one pending request at a time), or track per-request ETH values and zero out the specific request's contribution on claim. Additionally, `updateRSETHPrice()` should be called immediately after `claimStEth()` so the oracle reflects the post-slashing TVL before any withdrawals can be processed at the stale inflated price. A manual admin function to correct `ethValueInWithdrawal` should also be added as a backstop.

## Proof of Concept

Foundry fork test sequence against mainnet state:

1. Deploy/fork with 600 stETH in protocol (400 in deposit pool, 200 transferred to converter via `transferAssetFromDepositPool`). Verify `ethValueInWithdrawal` = 200e18.

2. Call `unstakeStEth(200e18)` — submits Lido withdrawal request for 200 stETH.

3. Simulate Lido slashing: use `vm.store` to reduce stETH total shares by 10%, causing all stETH balances to rebase down. Finalize the Lido withdrawal request at the post-slash share rate so it pays out 180 ETH.

4. Call `claimStEth(requestId, hint)`. Verify `ethValueInWithdrawal` = 20e18 (phantom residual).

5. Call `updateRSETHPrice()`. Record `rsETHPrice_inflated`. Compute expected correct price using actual TVL (360 ETH). Assert `rsETHPrice_inflated > rsETHPrice_correct`.

6. As a user holding 1/3 of rsETH supply, call `initiateWithdrawal(ETH, rsETHAmount)`. Record `expectedAssetAmount_inflated`. Assert it exceeds the user's proportional share of actual TVL (120 ETH).

7. Advance blocks past `withdrawalDelayBlocks`. Call `unlockQueue` then `completeWithdrawal`. Verify user receives the inflated amount.

8. Assert remaining protocol ETH < amount owed to remaining rsETH holders, proving insolvency.
