Audit Report

## Title
Missing Chainlink price validation (zero/negative price and staleness) in `ChainlinkPriceOracle` - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `price`, with no check for zero, negative, or stale values. A corrupted price propagates directly into `LRTOracle._updateRsETHPrice()`, which is callable by any unprivileged address via the public `updateRSETHPrice()`. Depending on the magnitude of the price deviation and the configured `pricePercentageLimit`, the result is either a temporary protocol-wide freeze of all deposits and withdrawals, or an incorrect rsETH price that dilutes existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` performs no validation on the Chainlink response:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L52-54
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

Three values are silently discarded: `roundId`, `updatedAt`, and `answeredInRound`. The sibling contract in the same repository, `ChainlinkOracleForRSETHPoolCollateral.getRate()`, performs all three checks, confirming the pattern is known:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L30-32
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [2](#0-1) 

The corrupted price flows through `LRTOracle.getAssetPrice()` → `_getTotalEthInProtocol()` → `_updateRsETHPrice()`, which is reachable by any unprivileged caller via the public `updateRSETHPrice()`: [3](#0-2) [4](#0-3) 

**Existing guards are insufficient:**
- The `pricePercentageLimit` check does not prevent the bad price from being consumed; it either pauses the protocol (causing a freeze) or allows the bad price through if the deviation is within the limit.
- `updatePriceOracleForValidated` checks the price at oracle registration time only, not at runtime. [5](#0-4) 

## Impact Explanation
**Primary – Medium: Temporary freezing of funds.**
If a Chainlink feed returns zero (circuit-breaker event) or a stale low price, `totalETHInProtocol` is understated, `newRsETHPrice` drops, and if the drop exceeds `pricePercentageLimit`, `lrtDepositPool`, `withdrawalManager`, and `LRTOracle` are all paused. All user deposits and withdrawals are frozen until an admin manually unpauses. [6](#0-5) 

**Secondary – Low: Contract fails to deliver promised returns.**
If the price drop is within `pricePercentageLimit` (or the limit is unset), `rsETHPrice` is written at the deflated value. Depositors minting rsETH at this price receive more rsETH than fair value, diluting all existing rsETH holders. [7](#0-6) 

**Negative price scenario** (`uint256` wrap to near `2^256`) would spike `newRsETHPrice` above `highestRsethPrice`, triggering `PriceAboveDailyThreshold` for unprivileged callers (revert, no state change). Exploitation of this path requires a manager role and is therefore out of scope. [8](#0-7) 

## Likelihood Explanation
Chainlink circuit-breaker events (feeds returning min/max answer or going stale) are documented and have occurred historically during extreme market volatility or L2 sequencer outages. The trigger function `updateRSETHPrice()` is `public` with no access control, so no special privilege is required. The attacker need only call it after a feed anomaly occurs. [3](#0-2) 

Likelihood: **Low** (requires a rare but real Chainlink feed anomaly).

## Recommendation
Replace the bare `latestRoundData` call in `ChainlinkPriceOracle.getAssetPrice()` with full validation, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (updatedAt == 0) revert IncompleteRound();
if (price <= 0) revert InvalidPrice();
if (block.timestamp - updatedAt > HEARTBEAT + BUFFER) revert StalePrice();
```

For L2 deployments, additionally consult the Chainlink sequencer uptime feed and enforce the grace period before consuming any price. [9](#0-8) 

## Proof of Concept
1. A Chainlink feed for a supported asset (e.g., stETH/ETH) goes stale or hits its circuit-breaker minimum, returning a price significantly below the true market price.
2. Any address calls `LRTOracle.updateRSETHPrice()` (public, no access control).
3. `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)`, which returns the corrupted low price without reverting.
4. `totalETHInProtocol` is understated; `newRsETHPrice` is computed below `highestRsethPrice`.
5. If the percentage drop exceeds `pricePercentageLimit`: `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()` are called — all user deposits and withdrawals are frozen until admin intervention.
6. If the drop is within `pricePercentageLimit` (or the limit is zero): `rsETHPrice` is written at the deflated value; subsequent depositors receive excess rsETH, diluting all existing holders.

```solidity

// Assert lrtDepositPool.paused() == true (if pricePercentageLimit is set)

``` [10](#0-9)
