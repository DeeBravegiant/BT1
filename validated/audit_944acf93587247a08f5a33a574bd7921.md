Audit Report

## Title
ETH Deposit Limit Check Ignores Deposit Amount, Allowing Limit Bypass - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses a strict `>` comparison without adding `amount` for ETH deposits, while the ERC-20 branch correctly adds `amount` before comparing. When `totalAssetDeposits == depositLimit`, the ETH check returns `false` ("not exceeded"), allowing any depositor to push the pool above the governance-approved cap.

## Finding Description
At [1](#0-0)  the ETH branch on line 679 evaluates `totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)` — the incoming `amount` is never added. The ERC-20 branch on line 681 correctly evaluates `totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)`.

When `totalAssetDeposits` equals the deposit limit exactly, `totalAssetDeposits > limit` is `false`, so the function returns `false` ("limit not exceeded"). The caller `_beforeDeposit` at [2](#0-1)  only reverts when the return value is `true`, so it proceeds to mint rsETH for the full deposit amount. The public entry point `depositETH` at [3](#0-2)  is permissionless and requires no special role.

## Impact Explanation
The ETH deposit cap is the protocol's primary risk-management gate for native ETH exposure. Bypassing it silently allows the pool to accumulate ETH beyond the governance-approved ceiling, breaking the cap invariant. No funds are stolen and no yield is lost, matching **Low — contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The bypass condition (`totalAssetDeposits == depositLimit`) is a transient state reached naturally as deposits accumulate. Any unprivileged depositor can observe this state on-chain and call `depositETH` with an arbitrary value. No special role, prior setup, or victim mistake is required. The condition recurs every time the pool is refilled to the limit.

## Recommendation
Apply the same prospective check used for ERC-20 assets:

```diff
 if (asset == LRTConstants.ETH_TOKEN) {
-    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
 }
```

## Proof of Concept
1. Governance sets `depositLimitByAsset(ETH_TOKEN) = 1000 ether`.
2. Legitimate deposits bring `totalAssetDeposits` to exactly `1000 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `1000 ether > 1000 ether` → `false`.
5. `_beforeDeposit` does not revert; `_mintRsETH` mints rsETH for 500 ETH.
6. `totalAssetDeposits` becomes `1500 ether` — 50% above the approved cap — with no revert or event.

**Foundry test sketch:**
```solidity
function test_ethDepositLimitBypass() public {
    uint256 limit = 1000 ether;
    lrtConfig.setDepositLimitByAsset(ETH_TOKEN, limit);
    // fill pool to exactly the limit via legitimate deposits
    depositPool.depositETH{value: limit}(0, "");
    // now totalAssetDeposits == limit; bypass should revert but doesn't
    vm.expectRevert(); // expect MaximumDepositLimitReached — this will FAIL, proving the bug
    depositPool.depositETH{value: 500 ether}(0, "");
}
```

        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
