Audit Report

## Title
Block Stuffing Suppresses Auto-Pause on Asset Rate Drop, Enabling Withdrawals at Stale Inflated rsETH Price - (File: `contracts/LRTOracle.sol`, `contracts/LRTWithdrawalManager.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is a permissionless `public` function and the sole trigger for the downside-protection auto-pause. An attacker can fill blocks with high-gas transactions to exclude it from inclusion. During the stuffed window, `rsETHPrice` in storage remains at its pre-drop value while `SwETHPriceOracle.getAssetPrice()` returns the live dropped rate. `instantWithdrawal` computes payout as `rsETHAmount * staleHighRsETHPrice / liveDroppedAssetPrice`, yielding an inflated asset amount that extracts value from remaining depositors.

## Finding Description

**Pause trigger is call-dependent, not event-driven.**

`updateRSETHPrice()` is `public whenNotPaused` and is the only unprivileged path that executes the downside check:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

The auto-pause fires only inside `_updateRsETHPrice()` at lines 270–281:

```solidity
if (newRsETHPrice < highestRsethPrice) {
    bool isPriceDecreaseOffLimit =
        pricePercentageLimit > 0 && diff > pricePercentageLimit.mulWad(highestRsethPrice);
    if (isPriceDecreaseOffLimit) {
        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!withdrawalManager.paused()) withdrawalManager.pause();
        _pause();
        return;
    }
}
```

If no call lands, `rsETHPrice` in storage is never updated and the pause never fires. The privileged `updateRSETHPriceAsManager()` (L94–96) is only callable by `onlyLRTManager`, not a public caller, so it does not mitigate the stuffing window.

**`SwETHPriceOracle.getAssetPrice()` is always live; `rsETHPrice` is always stale until updated.**

`SwETHPriceOracle.getAssetPrice()` calls `ISwETH(swETHAddress).getRate()` directly on every invocation (L34–40), returning the current on-chain rate with no caching. `rsETHPrice` is a storage variable updated only when `_updateRsETHPrice()` runs.

**Withdrawal math uses the stale stored price as numerator and the live dropped asset price as denominator.**

`getExpectedAssetAmount` at L590–594:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

When `rsETHPrice` is stale-high and `getAssetPrice(swETH)` is live-low, `underlyingToReceive` is inflated in both directions simultaneously.

**`instantWithdrawal` executes atomically with no operator gate.**

`instantWithdrawal` (L212–253) calls `getExpectedAssetAmount`, burns rsETH, and transfers assets in one transaction. The only supply-side guard is `CantInstantWithdrawMoreThanAvailable` (L231), which limits the payout to what is in the unstaking vault but does not prevent extraction at the inflated price up to that cap. There is no staleness check on `rsETHPrice`.

**Attack flow:**

1. swETH slashing event causes `ISwETH.getRate()` to drop by more than `pricePercentageLimit`.
2. Attacker stuffs every block to exclude any `updateRSETHPrice()` call.
3. `rsETHPrice` in `LRTOracle` remains at the pre-drop value; `LRTWithdrawalManager` remains unpaused.
4. Attacker calls `instantWithdrawal(swETH, rsETHAmount, ...)`.
   - `expectedAssetAmount = rsETHAmount * staleHighRsETHPrice / liveDroppedSwETHRate` → inflated swETH payout.
5. Attacker receives more swETH than the rsETH burned is actually worth, extracting value from remaining depositors.
6. Block stuffing ends; `updateRSETHPrice()` is called, triggers the pause — but the extraction has already occurred.

## Impact Explanation

The invariant "a price drop beyond `pricePercentageLimit` must pause the protocol" is violated for the duration of the stuffed window. During that window, `instantWithdrawal` allows redemption at a stale inflated rsETH price, extracting value from the protocol at the expense of remaining depositors. This matches the allowed impact **Low — Block stuffing**.

## Likelihood Explanation

Block stuffing on Ethereum mainnet is expensive but feasible when the profit opportunity (large TVL, large price drop) exceeds the cost of filling blocks for a few minutes. The attack is most attractive during a sharp LST slashing event when the price drop is large and the window before a keeper bot lands `updateRSETHPrice()` is exploitable. The requirement that `isInstantWithdrawalEnabled[asset]` be `true` narrows the realistic surface but does not eliminate it.

## Recommendation

1. **Make the price check pull-based at withdrawal time.** In `getExpectedAssetAmount`, compute the rsETH price on-the-fly from `_getTotalEthInProtocol() / rsethSupply` rather than reading the cached `rsETHPrice`. This eliminates the stale-price window entirely.
2. **Add a staleness guard.** Store `lastPriceUpdateTimestamp` and revert in `initiateWithdrawal` / `instantWithdrawal` if the price is older than a configurable threshold (e.g., 1 hour).
3. **Emit a price-update heartbeat requirement.** Require `updateRSETHPrice()` to have been called within the last N blocks before any withdrawal is processed, forcing the live check before funds move.

## Proof of Concept

```solidity
// Foundry fork test — run against a local fork with mocked swETH rate
// Preconditions: swETH is a supported asset, isInstantWithdrawalEnabled[swETH] = true,
//                attacker holds rsETH obtained via prior deposit at normal rate.

contract BlockStuffingPoC is Test {
    ILRTOracle            oracle = ILRTOracle(ORACLE_ADDR);
    ILRTWithdrawalManager wm     = ILRTWithdrawalManager(WM_ADDR);
    ISwETHMock            swETH  = ISwETHMock(SWETH_ADDR);

    function testBlockStuffingSuppressesPause() public {
        uint256 normalRate    = swETH.getRate(); // e.g. 1.05e18
        uint256 rsETHBalance  = _depositAndGetRsETH(1000 ether);

        // Simulate swETH rate drop > pricePercentageLimit (e.g. 6% drop)
        swETH.setRate(normalRate * 94 / 100);

        // Simulate block stuffing: simply do NOT call updateRSETHPrice().
        // rsETHPrice in storage is still the pre-drop value.

        // Protocol is NOT paused — invariant violated
        assertFalse(ILRTDepositPool(DEPOSIT_POOL_ADDR).paused());

        // Attacker calls instantWithdrawal — receives inflated swETH
        uint256 swETHBefore   = IERC20(SWETH_ADDR).balanceOf(address(this));
        wm.instantWithdrawal(SWETH_ADDR, rsETHBalance, "");
        uint256 swETHReceived = IERC20(SWETH_ADDR).balanceOf(address(this)) - swETHBefore;

        // Fair amount uses updated (lower) rsETHPrice
        uint256 fairRsETHPrice = oracle.rsETHPrice() * 94 / 100;
        uint256 fairAmount     = rsETHBalance * fairRsETHPrice / swETH.getRate();
        assertGt(swETHReceived, fairAmount, "attacker extracted more than fair share");
    }
}
```

The test confirms: (a) the protocol is not paused despite the price drop exceeding `pricePercentageLimit`, and (b) `instantWithdrawal` succeeds at the stale inflated price, yielding more swETH than the rsETH burned is actually worth.
