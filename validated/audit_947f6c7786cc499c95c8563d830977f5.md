Audit Report

## Title
Pool Admin Can Immediately Execute Oracle Swap With Zero `priceProviderTimelock`, Bypassing Intended Delay - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`MetricOmmPoolFactory.createPool` stores `params.priceProviderTimelock` directly without any minimum-value validation, allowing a pool to be created with a zero timelock. When the timelock is zero, `proposePoolPriceProvider` sets `pendingPriceProviderExecuteAfter[pool] = block.timestamp`, and `executePoolPriceProviderUpdate`'s guard `block.timestamp < block.timestamp` is always false, so the check never reverts. A malicious pool admin can therefore propose and execute an oracle replacement atomically in a single transaction, completely bypassing the timelock that is the sole protection LPs have against a rogue admin swapping in a manipulated price feed.

## Finding Description
`createPool` stores the caller-supplied `priceProviderTimelock` directly with no lower-bound validation:

```solidity
// MetricOmmPoolFactory.sol line 213
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

`_validatePoolParameters` (lines 548–563) checks fees, tokens, admin, and initial amounts, but never validates `priceProviderTimelock`. The propose step computes:

```solidity
// MetricOmmPoolFactory.sol line 487
uint256 executeAfter = block.timestamp + timelock;  // = block.timestamp when timelock == 0
```

The execute step enforces:

```solidity
// MetricOmmPoolFactory.sol line 499
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
```

When `timelock == 0`, `execAfter == block.timestamp`, so `block.timestamp < block.timestamp` is `false` and the guard never fires. Both calls can be batched in one transaction. The only other guard is the immutable-mode sentinel (`type(uint256).max`), which disables oracle rotation entirely. There is no enforced minimum for the mutable path.

## Impact Explanation
A pool admin who created a pool with `priceProviderTimelock = 0` can deploy a malicious `IPriceProvider` returning extreme bid/ask prices, then atomically call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` in a single transaction. The pool's active oracle is immediately replaced with the malicious feed, enabling the admin to execute a swap at the manipulated price and drain LP token balances. LP principal is directly at risk. This is a confirmed admin-boundary break: the pool admin bypasses the timelock that is the sole mechanism preventing rogue oracle substitution.

## Likelihood Explanation
`createPool` is permissionless. Any actor can deploy a pool with `priceProviderTimelock = 0`, set themselves as admin, and attract liquidity. The `PoolCreated` event emits `priceProviderTimelock` as a raw `uint256`; off-chain tooling that does not specifically flag a zero value will not warn users. The attack requires a malicious admin and LPs who deposit without verifying the timelock parameter. Likelihood is medium — it requires a malicious admin — but the admin-boundary break (bypassing a timelock) is exactly the class the contest marks as in-scope.

## Recommendation
Add a minimum timelock floor in `_validatePoolParameters` for the mutable path:

```solidity
if (params.priceProviderTimelock != type(uint256).max) {
    if (params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
        revert PriceProviderTimelockTooShort();
    }
}
```

`MIN_PRICE_PROVIDER_TIMELOCK` should be a protocol constant (e.g., 24 hours) that gives LPs sufficient time to observe a pending oracle change and exit before it takes effect.

## Proof of Concept
```solidity
// 1. Deploy pool with zero timelock
PoolParameters memory params = _defaultParams();
params.priceProviderTimelock = 0;          // mutable mode, zero delay
params.admin = attacker;
address pool = factory.createPool(params);

// 2. LPs add liquidity (pool looks legitimate)

// 3. Attacker atomically swaps oracle and executes
vm.startPrank(attacker);

MaliciousOracle mal = new MaliciousOracle(token0, token1);
mal.setBidAndAskPrice(1, type(uint128).max);   // near-zero bid

// propose — executeAfter = block.timestamp + 0 = block.timestamp
factory.proposePoolPriceProvider(pool, address(mal));

// execute — block.timestamp < block.timestamp is false, no revert
factory.executePoolPriceProviderUpdate(pool);

// 4. Swap at manipulated price, draining LP token1 balance
IMetricOmmPool(pool).swap(/* zeroForOne=true, large amountSpecified */);
vm.stopPrank();
```
