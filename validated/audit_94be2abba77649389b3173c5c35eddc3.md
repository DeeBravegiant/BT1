### Title
Recipient Contract Can Permanently Block `finTransfer` for Native ETH and ERC1155, Irreversibly Locking Sender Funds — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
In `OmniBridge.finTransfer`, when the destination asset is native ETH or an ERC1155 token, the delivery step invokes a callback on the recipient contract. A recipient contract that reverts on receipt causes `finTransfer` to always revert. Because there is no fallback path to return the locked/burned source-chain tokens to the sender, the funds are permanently unclaimable.

### Finding Description
`finTransfer` marks the destination nonce as consumed at line 287 and then attempts delivery. For native ETH it uses a low-level `.call{value}`:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
...
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();              // line 322
```

For ERC1155 it uses `safeTransferFrom`, which calls `onERC1155Received` on the recipient:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);                                                     // lines 324-330
```

Because the revert unwinds the nonce marking, the nonce is never permanently consumed and the call can be retried. However, if the recipient contract unconditionally reverts on receipt (no `receive()` / `fallback()`, or an `onERC1155Received` that always reverts), every retry also reverts. The corresponding tokens were already locked or burned on the source chain (NEAR) when `initTransfer` was called there. With no cancel/refund path on the source side, those tokens are permanently frozen.

The attacker's entry path is entirely public: deploy a contract whose `receive()` or `onERC1155Received` reverts by default, and have a victim send a cross-chain transfer to that address. The attacker can then demand a side-payment before enabling receipt — a direct grief/extortion analog to the reported NFT auction bug. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Matches **Critical — Irreversible fund lock / permanently unclaimable user value**. The sender's native ETH or ERC1155 tokens are locked in (or burned from) the source chain with no on-chain mechanism to reclaim them once the destination delivery is permanently blocked. The bridge contract itself retains the ETH indefinitely with no sweep or rescue path visible in the contract.

### Likelihood Explanation
Medium. The attacker only needs to:
1. Deploy a contract with a reverting `receive()` or `onERC1155Received`.
2. Convince (or wait for) a user to initiate a cross-chain transfer to that address.

Native ETH bridging is a primary use-case of the bridge, and ERC1155 support is explicitly implemented. No privileged role, leaked key, or external oracle compromise is required.

### Recommendation
Mirror the fix pattern from the referenced report: wrap the delivery in a try/catch (or use a low-level call that captures success without reverting), and if delivery fails, store the amount in a per-recipient claimable balance mapping rather than reverting the whole transaction. This decouples settlement finality from recipient cooperation:

```solidity
// Pseudocode
(bool ok, ) = payload.recipient.call{value: payload.amount}("");
if (!ok) {
    claimable[payload.recipient] += payload.amount;
    emit DeliveryFailed(payload.recipient, payload.amount);
}
```

Provide a public `claim()` function so the recipient can pull their funds later, and a separate `recoverFailed()` that lets the original sender reclaim after a timeout if the recipient never claims.

### Proof of Concept
1. Attacker deploys `Rejecter` on EVM:
   ```solidity
   contract Rejecter {
       bool public accepting;
       receive() external payable { require(accepting, "nope"); }
       function unlock() external { accepting = true; }
   }
   ```
2. Victim on NEAR calls `initTransfer` bridging 1 ETH to `Rejecter`'s address; NEAR locks the wETH.
3. Relayer calls `OmniBridge.finTransfer` with a valid MPC signature — reverts with `FailedToSendEther` every time.
4. Attacker contacts victim: "send me 0.5 ETH off-chain and I'll call `unlock()`."
5. Without the attacker's cooperation (or a contract upgrade), the victim's 1 ETH is permanently locked in the NEAR bridge with no cancel path. [4](#0-3)
