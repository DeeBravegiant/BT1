### Title
`vote_new_parameters` Reverts Entirely When Any Proposed Participant's TEE Attestation Expires Between Check and Execution, Blocking Resharing - (File: crates/contract/src/lib.rs)

### Summary

`vote_new_parameters` validates TEE attestations for **all** proposed participants atomically. If any single participant's attestation expires between the time a node checks validity off-chain and the time the transaction executes on-chain, the entire call returns an error and the vote is **not recorded**. This is a direct analog to the Notional M-9 bug: a batch check that reverts the whole operation when one item changes state between check and execution, causing the entire resharing process to stall.

### Finding Description

In `vote_new_parameters`, the contract calls `reverify_and_cleanup_participants` over the full proposed participant set. If the result is `TeeValidationResult::Partial` — meaning even one participant's attestation has expired or been invalidated — the function returns `Err(InvalidParameters::InvalidTeeRemoteAttestation {...})` and the vote is discarded entirely. [1](#0-0) 

The inner validation logic iterates all proposed participants and filters out those whose stored attestation fails `re_verify` at the current block timestamp: [2](#0-1) 

The race window is:

1. **T1** — A participant node queries the contract off-chain, confirms all proposed participants have valid attestations, and constructs a `vote_new_parameters` transaction.
2. **T2** — The transaction is submitted to the NEAR mempool.
3. **T3** — Between submission and execution, one proposed participant's TEE certificate expires (attestations carry a finite `expiry_timestamp_seconds`).
4. **T3 execution** — `reverify_and_cleanup_participants` returns `Partial`; the entire call fails; the vote is not recorded.

Because every participant must independently submit their own `vote_new_parameters` transaction, and each transaction re-runs the full attestation sweep at execution time, any single expiry event during the voting window causes **all** in-flight votes to fail simultaneously. The resharing cannot reach threshold until all proposed participants hold simultaneously-valid attestations at execution time.

### Impact Explanation

When resharing is blocked:

- If `verify_tee` has already set `accept_requests = false` (because an existing participant's attestation expired, triggering the kickout path), the contract stops accepting new `sign`, `request_app_private_key`, and `verify_foreign_transaction` calls. [3](#0-2) 

- All pending yield-resume promises for in-flight signature requests will time out, permanently failing those requests.
- The contract is stuck: it cannot accept new requests and cannot complete the resharing needed to restore the participant set.

This matches the **Medium** allowed impact: *participant-state or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.*

### Likelihood Explanation

TEE attestations carry real expiry timestamps enforced at execution time: [4](#0-3) 

NEAR transaction finality is typically 1–2 seconds, but under mempool congestion or during a resharing event (which itself involves many concurrent transactions), the gap between submission and execution can grow. A participant whose attestation expires within that window — whether naturally or by a malicious new participant timing their certificate expiry — causes every concurrent `vote_new_parameters` vote to fail. The sandbox test `vote_new_parameters_errors_if_new_participant_is_missing_valid_attestation` confirms the revert path is reachable: [5](#0-4) 

### Recommendation

Mirror the Notional fix: instead of reverting the entire vote when one proposed participant's attestation is invalid, **skip that participant and proceed** with recording the vote for the remaining valid participants. Concretely, change the `TeeValidationResult::Partial` branch in `vote_new_parameters` to continue with the reduced valid set rather than returning an error:

```diff
- TeeValidationResult::Partial { .. } => {
-     Err(InvalidParameters::InvalidTeeRemoteAttestation { ... }.into())
- }
+ TeeValidationResult::Partial { participants_with_valid_attestation } => {
+     // Proceed with the vote using only the valid subset.
+     // The caller's own validity is already enforced by assert_caller_is_attested_participant.
+     let reduced_proposal = proposal.with_participants(participants_with_valid_attestation);
+     if let Some(new_state) = self.protocol_state
+         .vote_new_parameters(prospective_epoch_id, &reduced_proposal)?
+     {
+         self.protocol_state = new_state;
+     }
+     Ok(())
+ }
```

Alternatively, validate only the **calling participant's** attestation eagerly (already done via `assert_caller_is_attested_participant_and_protocol_active`) and defer full-set validation to the threshold-crossing moment, so individual votes are not lost due to a third party's expiry.

### Proof of Concept

1. Deploy the contract in `Running` state with participants A, B, C (threshold 2).
2. Submit a `vote_new_parameters` proposal to add participant D, whose TEE attestation has `expiry_timestamp_seconds = now + 2`.
3. Participant A submits `vote_new_parameters` at block N.
4. Fast-forward 3 seconds (≈ 30 blocks in sandbox) so D's attestation expires.
5. Participant B submits `vote_new_parameters` at block N+30.
6. Both calls return `InvalidTeeRemoteAttestation`; the contract remains in `Running` with zero votes recorded.
7. If `verify_tee` was called before step 3 and set `accept_requests = false`, the contract now refuses all signature requests indefinitely until D renews their attestation and all participants re-vote simultaneously within a non-expiry window.

The existing sandbox test at `crates/contract/tests/sandbox/vote.rs:774` already demonstrates step 6 in isolation; extending it with a `fast_forward` between A's and B's votes reproduces the full race. [6](#0-5) [7](#0-6)

    pub fn vote_new_parameters(
        &mut self,
        prospective_epoch_id: EpochId,
        proposal: dtos::ProposedThresholdParameters,
    ) -> Result<(), Error> {
        Self::assert_caller_is_signer();
        let proposal: ProposedThresholdParameters = proposal.try_into_contract_type()?;
        log!(
            "vote_new_parameters: signer={}, proposal={:?}",
            env::signer_account_id(),
            proposal,
        );

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        let validation_result = self.tee_state.reverify_and_cleanup_participants(
            proposal.participants(),
            tee_upgrade_deadline_duration,
        );

        let proposed_participants = proposal.participants();
        match validation_result {
            TeeValidationResult::Full => {
                if let Some(new_state) = self
                    .protocol_state
                    .vote_new_parameters(prospective_epoch_id, &proposal)?
                {
                    self.protocol_state = new_state;
                }
                Ok(())
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let invalid_participants: Vec<_> = proposed_participants
                    .participants()
                    .iter()
                    .filter(|(account_id, _, _)| {
                        !participants_with_valid_attestation
                            .is_participant_given_account_id(account_id)
                    })
                    .collect();

                Err(InvalidParameters::InvalidTeeRemoteAttestation {
                    reason: format!(
                        "The following participants have invalid TEE status: {:?}",
                        invalid_participants
                    ),
                }
                .into())
            }
        }
    }

    pub fn reverify_and_cleanup_participants(
        &mut self,
        participants: &Participants,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeValidationResult {
        self.allowed_docker_image_hashes
            .cleanup_expired_hashes(tee_upgrade_deadline_duration);

        let participants_with_valid_attestation: Vec<_> = participants
            .participants()
            .iter()
            .filter(|(_, _, participant_info)| {
                // Use the stored NodeId (keyed by TLS public key) so the real
                // `account_public_key` participates in re-verification. If
                // there is no stored attestation for this TLS key, the
                // participant is invalid.
                let Some(node_id) = self.find_node_id_by_tls_key(&participant_info.tls_public_key)
                else {
                    return false;
                };

                let tee_status =
                    self.reverify_participants(&node_id, tee_upgrade_deadline_duration);

                matches!(tee_status, TeeQuoteStatus::Valid)
            })
            .cloned()
            .collect();

        if participants_with_valid_attestation.len() != participants.len() {
            let participants_with_valid_attestation =
                Participants::init(participants.next_id(), participants_with_valid_attestation);

            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            }
        } else {
            TeeValidationResult::Full
        }
    }

#[tokio::test]
async fn vote_new_parameters_errors_if_new_participant_is_missing_valid_attestation() {
    let SandboxTestSetup {
        worker,
        contract,
        mut mpc_signer_accounts,
        ..
    } = SandboxTestSetup::builder()
        .with_protocols(ALL_PROTOCOLS)
        .build()
        .await;

    let state = get_state(&contract).await;
    let ProtocolContractState::Running(ref running_state) = state else {
        panic!("expected running state");
    };
    let threshold = running_state.parameters.threshold;
    let epoch_id = running_state.keyset.epoch_id;
    let mut proposed_participants: mpc_contract::primitives::participants::Participants =
        (&running_state.parameters.participants).into_contract_type();

    let (new_account, new_account_id) = gen_account(&worker).await;
    let new_participant_info = gen_participant_info();

    // Add the new participant to the participant set, and propose this to the contract.
    proposed_participants
        .insert(new_account_id.clone(), new_participant_info)
        .unwrap();

    let threshold_parameters =
        ThresholdParameters::new(proposed_participants, Threshold::new(threshold.0 + 1)).unwrap();

    mpc_signer_accounts.push(new_account.clone());

    // Vote to transition to resharing state
    for account in &mpc_signer_accounts {
        let call_result = account
            .call(contract.id(), method_names::VOTE_NEW_PARAMETERS)
            .max_gas()
            .args_json(json!({
                "prospective_epoch_id": dtos::EpochId(epoch_id.0 + 1),
                "proposal": ProposedThresholdParameters::new(
                    threshold_parameters.clone(),
                    BTreeMap::new(),
                ),
            }))
            .transact()
            .await
            .unwrap()
            .into_result()
            .expect_err("calling `vote_new_parameters` must fail when one participant has invalid TEE status.");

        let error_message = call_result.to_string();
        let expected_prefix =
            mpc_contract::errors::InvalidParameters::InvalidTeeRemoteAttestation {
                reason: String::new(),
            }
            .to_string();
        assert!(
            error_message.contains(&expected_prefix),
            "Expected InvalidTeeRemoteAttestation error, got: {error_message}"
        );
    }

    let state: ProtocolContractState = get_state(&contract).await;

    assert_matches!(
        state,
        ProtocolContractState::Running(_),
        "Protocol state should not transition when new participant has invalid TEE status."
    );
