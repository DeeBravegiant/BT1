Audit Report

## Title
Chainlink `latestRoundData()` Returns Circuit-Breaker Floor Price With No Bounds Validation, Enabling Theft of rsETH Holder Funds — (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and discards every return value except `price`, with no check that `price > 0`, no staleness check, and no validation against the aggregator's `minAnswer`/`maxAnswer` circuit-breaker bounds. When a supported LST crashes below its Chainlink feed's `minAnswer`, the aggregator silently returns the floor price. Because `LRTDepositPool.getRsETHAmountToMint()` divides the live (inflated) Chainlink price by the stored `rsETHPrice`, an attacker can deposit the crashed asset and receive rsETH at a rate far above its true value, draining healthy assets from all other rsETH holders.

## Finding Description

**Root cause — `ChainlinkPriceOracle.getAssetPrice()` (lines 49–55):**

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

All five return values of `latestRoundData()` except `price` are discarded. There is:
- No `price > 0` guard (a negative `int256` silently wraps to a huge `uint256`).
- No staleness check (`updatedAt`, `answeredInRound < roundId`).
- No comparison against the aggregator's `minAnswer`/`maxAnswer` circuit-breaker bounds.

**Exploit path through the deposit flow:**

`LRTDepositPool.getRsETHAmountToMint()` (line 520) computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

- Numerator: **live** Chainlink price fetched at deposit time via `ChainlinkPriceOracle.getAssetPrice()`.
- Denominator: **stored** `rsETHPrice` (last value written by `updateRSETHPrice()`).

When Chainlink's circuit breaker activates for a crashed LST, `getAssetPrice()` returns `minAnswer` (e.g., 0.1 ETH) instead of the true market price (e.g., 0.001 ETH). The stored `rsETHPrice` has not yet been updated to reflect this inflation, so the ratio `inflatedAssetPrice / oldRsETHPrice` is far above the true exchange rate. The attacker receives rsETH priced at the inflated rate.

**Why existing guards are insufficient:**

`LRTOracle._updateRsETHPrice()` contains a `pricePercentageLimit` check (lines 252–266), but:
1. `pricePercentageLimit` is an uninitialized `uint256` that defaults to `0`.
2. The guard is explicitly skipped when it is zero: `pricePercentageLimit > 0 && priceDifference > ...`.
3. Even when set, it only fires during `updateRSETHPrice()` calls — it does **not** protect the deposit-time `getRsETHAmountToMint()` path, which reads the live Chainlink price directly.

The `updatePriceOracleForValidated` sanity check (`price > 1e16 && price < 1e19`) runs only at oracle registration time, not on every price read.

## Impact Explanation

**Critical — direct theft of user funds.**

An attacker deposits a crashed LST (acquired cheaply on the open market) and receives rsETH minted at the inflated Chainlink circuit-breaker price. They then redeem rsETH for ETH or healthy LSTs, extracting real value from all existing rsETH holders. This is direct, on-chain theft of funds at rest, matching the Critical impact class.

## Likelihood Explanation

Moderate. Chainlink circuit breakers are a documented, production-deployed feature. The LUNA/Venus incident is a confirmed real-world precedent. The protocol holds multiple LST assets, each with its own Chainlink feed and `minAnswer`. A slashing event, smart contract exploit, or liquidity crisis in any one LST could trigger the circuit breaker. No privileged access is required — any external depositor can execute the attack the moment the circuit breaker activates. The attack is repeatable until the protocol is manually paused.

## Recommendation

Add the following validations to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
interface AggregatorV2V3Interface is AggregatorV3Interface {
    function minAnswer() external view returns (int192);
    function maxAnswer() external view returns (int192);
}

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV2V3Interface priceFeed = AggregatorV2V3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (price <= 0) revert InvalidPrice();
    if (updatedAt == 0 || answeredInRound < roundId) revert StalePrice();

    int192 minAnswer = priceFeed.minAnswer();
    int192 maxAnswer = priceFeed.maxAnswer();
    if (price <= int256(minAnswer) || price >= int256(maxAnswer)) revert PriceOutOfBounds();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Additionally, ensure `pricePercentageLimit` is set to a non-zero value during initialization so the upside guard in `_updateRsETHPrice()` is always active.

## Proof of Concept

**Minimal call sequence (local fork test):**

1. Deploy or fork with `stETH` as a supported asset; its Chainlink feed has `minAnswer = 0.1e18` (0.1 ETH).
2. Simulate a black-swan event: mock `latestRoundData()` to return `minAnswer = 0.1e18` while the true market price is `0.001e18`.
3. Confirm `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `0.1e18` (100× true value).
4. Confirm `rsETHPrice` is still at its last stored value (e.g., `1.0e18`).
5. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
6. `getRsETHAmountToMint` computes `1000e18 * 0.1e18 / 1.0e18 = 100e18` rsETH minted.
7. True value of 1000 stETH at market = `1000 * 0.001 = 1 ETH`; attacker received rsETH worth `100 ETH`.
8. Attacker redeems rsETH for ETH, extracting ~99 ETH of value from other holders.
