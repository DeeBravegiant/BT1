### Title
Missing Request-to-Payload Binding in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is valid over `response.payload_hash` using the MPC root public key, but never verifies that `response.payload_hash` is the correct hash of the submitted `request`. A single attested participant can replay a legitimately-produced `(payload_hash, signature)` pair from one pending foreign-tx request to resolve a completely different pending request, delivering a forged attestation to the victim caller.

---

### Finding Description

In `respond_verify_foreign_tx` the contract performs two independent checks:

1. **Signature validity** — `response.signature` is valid over `response.payload_hash` under the root MPC public key.
2. **Request existence** — `request` is present in `pending_verify_foreign_tx_requests`. [1](#0-0) 

Critically, there is **no check** that `response.payload_hash` is the hash of `(request, extracted_values)` for the specific `request` being resolved. The `payload_hash` is taken verbatim from the caller-supplied `response` argument. [2](#0-1) 

Contrast this with the regular `respond` function, where the signature is verified against a **derived** public key computed from `request.tweak`, which cryptographically binds the signature to the exact request: [3](#0-2) 

In `respond_verify_foreign_tx` no such binding exists. The root key is used directly, and `payload_hash` is fully attacker-controlled.

The `ForeignTxSignPayload` that nodes are supposed to hash encodes the full request: [4](#0-3) 

But the contract never reconstructs or checks this hash server-side.

---

### Impact Explanation

**Attack scenario (cross-request replay / forged attestation):**

1. User A submits `verify_foreign_transaction` for Bitcoin tx `[0xAA; 32]` (e.g., a bridge deposit of 10 BTC). Request A is stored in `pending_verify_foreign_tx_requests`.
2. User B submits `verify_foreign_transaction` for Bitcoin tx `[0xBB; 32]`. The MPC network honestly processes request B and an honest node calls `respond_verify_foreign_tx(request_B, response_B)`, resolving request B on-chain. `response_B = { payload_hash_B, sig_B }` is now publicly visible.
3. The malicious attested participant immediately calls `respond_verify_foreign_tx(request_A, response_B)`.
4. The contract checks: is `sig_B` valid over `payload_hash_B` under the root key? **Yes.** Does `request_A` exist in pending requests? **Yes.** Both checks pass.
5. `resolve_yields_for` drains request A's yield queue, delivering `response_B` to User A's waiting NEAR contract.
6. User A's bridge contract receives `{ payload_hash_B, sig_B }` — an attestation for Bitcoin tx `[0xBB; 32]`, not `[0xAA; 32]`.

If User A's bridge contract verifies only that the MPC signature is valid (a reasonable assumption given the MPC contract is supposed to enforce the binding), it will release funds based on a false attestation. The same Bitcoin transaction `[0xBB; 32]` has now been used to satisfy two separate bridge requests — a **double-spend**.

Even if User A's contract detects the mismatch, request A is permanently consumed; User A must resubmit and pay again, while the attacker has griefed the request lifecycle.

**Impact class:** Forged foreign-chain verification enabling invalid bridge execution and double-spend conditions.

---

### Likelihood Explanation

- Requires only **one** attested participant — strictly below the signing threshold.
- The attacker needs no special capability beyond observing on-chain transactions (the valid `(payload_hash, signature)` pair is public once any honest node submits it).
- The attack window is the time between an honest `respond_verify_foreign_tx` for request B landing on-chain and the honest response for request A being submitted. With multiple concurrent pending requests this window is always open.
- No cryptographic material needs to be compromised; the attacker reuses a legitimately-produced signature.

---

### Recommendation

Reconstruct `payload_hash` inside the contract from the caller-supplied `request` and the `response.values`, and assert it equals `response.payload_hash` before accepting the response. Concretely, the contract should compute:

```rust
let expected_hash = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.values.clone(),
}).compute_msg_hash()?;
require!(expected_hash == response.payload_hash, "payload_hash mismatch");
```

This mirrors how `respond` binds the signature to the request via key derivation from `request.tweak`. [3](#0-2)

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
