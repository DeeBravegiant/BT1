### Title
Unguarded `assert` in `_validate_sub_slot_data` raises `AssertionError` on malformed peer-supplied `SubSlotData`, violating clean-failure invariant — (`chia/full_node/weight_proof.py`)

### Summary

`_validate_sub_slot_data` uses a bare `assert` at line 1109 to check `icc_slot_end_info`, rather than a conditional guard that returns `(False, [])`. A syncing peer can craft a `WeightProof` containing a `SubSlotData` where `icc_slot_end` is a non-None `VDFProof` with `normalized_to_identity=False` while `icc_slot_end_info` is `None`. This causes an `AssertionError` to propagate instead of a clean validation failure.

### Finding Description

In `_validate_sub_slot_data`: [1](#0-0) 

```python
if sub_slot_data.is_end_of_slot():
    if sub_slot_data.icc_slot_end is not None:          # line 1104 — attacker sets this
        input = ClassgroupElement.get_default_element()
        if not sub_slot_data.icc_slot_end.normalized_to_identity and prev_ssd.icc_ip_vdf_info is not None:
            assert prev_ssd.icc_ip_vdf_info
            input = prev_ssd.icc_ip_vdf_info.output
        assert sub_slot_data.icc_slot_end_info          # line 1109 — RAISES if None
        to_validate.append(...)
```

The outer guard at line 1104 only checks `icc_slot_end is not None`. It does **not** verify that `icc_slot_end_info` is also non-None before reaching line 1109. Since `SubSlotData` is imported from `chia_rs` with independently optional fields, a peer can serialize a `WeightProof` where `icc_slot_end` is present but `icc_slot_end_info` is absent — both are `Optional` in the streamable format. [2](#0-1) 

The function's declared return type is `tuple[bool, list[...]]`, but the `assert` bypasses this contract entirely.

### Impact Explanation

Weight proof validation runs in a `ProcessPoolExecutor`: [3](#0-2) 

An `AssertionError` in the subprocess causes the corresponding `Future` to raise when awaited by the main process. Depending on how the caller handles this exception, the result ranges from a failed sync attempt (the node cannot complete sync from any peer sending the crafted proof) to a crash of the main node process if the exception propagates uncaught. At minimum, this is a **High** impact: a malicious peer can permanently prevent a syncing node from completing sync by serving a crafted `WeightProof`, satisfying the "long-lived inability to process sync updates" criterion.

### Likelihood Explanation

The attack path is direct and unprivileged:
1. Attacker runs a Chia peer node.
2. Syncing victim connects and requests a weight proof via `request_proof_of_weight`.
3. Attacker responds with a crafted `WeightProof` containing a `SubEpochChallengeSegment` whose `SubSlotData` has `icc_slot_end` set (non-None, `normalized_to_identity=False`) and `icc_slot_end_info=None`.
4. `_validate_sub_slot_data` hits line 1109 and raises `AssertionError`.

No keys, signatures, or privileged access are required.

### Recommendation

Replace the bare `assert` at line 1109 with a proper guard:

```python
# Before (line 1109):
assert sub_slot_data.icc_slot_end_info

# After:
if sub_slot_data.icc_slot_end_info is None:
    log.error("icc_slot_end present but icc_slot_end_info is None")
    return False, []
```

Apply the same pattern to all other bare `assert` statements inside `_validate_sub_slot_data` that operate on peer-supplied fields (lines 1107, 1111, 1112, 1115, 1127, 1141, 1151, 1152, 1155).
