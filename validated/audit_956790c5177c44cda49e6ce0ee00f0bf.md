Audit Report

## Title
Stale `rsETHPrice` Cache in Deposit Minting Dilutes Existing Holders' Yield - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides a live Chainlink asset price by `lrtOracle.rsETHPrice()`, a stored state variable that is only refreshed when `updateRSETHPrice()` is called in a separate transaction. As staking rewards accrue and the cache grows stale, any depositor can mint more rsETH than their deposit warrants, diluting the yield of all existing rsETH holders once the price is eventually updated.

## Finding Description
`getRsETHAmountToMint` at `LRTDepositPool.sol:520` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` (LRTOracle.sol:156–158) is a live read through `IPriceFetcher`, while `lrtOracle.rsETHPrice()` (LRTOracle.sol:28) is a plain `uint256` storage variable written only inside `_updateRsETHPrice()`. The deposit path `depositETH` → `_beforeDeposit` (LRTDepositPool.sol:648–670) → `getRsETHAmountToMint` is entirely `view`; it cannot and does not call `updateRSETHPrice()` before computing the mint amount. There is no staleness check anywhere in this path.

`_updateRsETHPrice()` (LRTOracle.sol:214–265) computes the true price as `(totalETHInProtocol − protocolFee) / rsethSupply`. As EigenLayer staking rewards and LST appreciation accrue, `totalETHInProtocol` grows, so the true rsETH price rises above the stored `rsETHPrice`. Additionally, when the price increase exceeds `pricePercentageLimit`, the permissionless `updateRSETHPrice()` path reverts for non-managers (LRTOracle.sol:260–265), extending the staleness window precisely during high-reward periods when the discrepancy is largest.

## Impact Explanation
When `rsETHPrice` is stale and below the true current value, the denominator in `getRsETHAmountToMint` is artificially small, causing the depositor to receive excess rsETH. When `updateRSETHPrice()` is eventually called, the new price is computed as `totalETHInProtocol / rsethSupply`; because the supply was inflated by the excess mint, the price per rsETH is permanently lower than it would have been, directly reducing the redemption value of every existing holder's position. This constitutes **theft of unclaimed yield** from existing rsETH holders — a High-severity impact within the allowed scope.

## Likelihood Explanation
- `updateRSETHPrice()` is a separate call with no on-chain enforcement that it must precede deposits; the deposit path is `view` and structurally cannot enforce it.
- Staking rewards and LST appreciation accrue continuously, so the stale window opens naturally between keeper calls.
- The `pricePercentageLimit` guard (LRTOracle.sol:256–265) blocks permissionless updates during high-reward periods, widening the window when the discrepancy is greatest.
- Any unprivileged depositor can compare the on-chain `rsETHPrice` against a locally computed `_getTotalEthInProtocol() / rsethSupply` and time their deposit to maximize excess minting.
- No special privileges, flash loans, or external protocol compromise are required.

## Recommendation
Force a price refresh before computing the mint amount. Either:
1. Call `lrtOracle.updateRSETHPrice()` at the start of `depositETH` / `depositAsset` (requires making `_beforeDeposit` non-`view`), or
2. Expose a view-only price computation in `LRTOracle` that returns `_getTotalEthInProtocol() / rsethSupply` without writing state, and use that in `getRsETHAmountToMint` instead of the cached `rsETHPrice`.

## Proof of Concept
1. Staking rewards accrue over several hours; `_getTotalEthInProtocol()` grows so the true rsETH price is 1.001 ETH while `rsETHPrice` is still stored as 1.000 ETH.
2. `updateRSETHPrice()` has not been called (or is blocked by `pricePercentageLimit` for non-managers).
3. Attacker calls `depositETH(minRSETHAmountExpected, "")` with 100 ETH.
4. `getRsETHAmountToMint` computes: `(100e18 × 1e18) / 1.000e18 = 100 rsETH` instead of the correct `(100e18 × 1e18) / 1.001e18 ≈ 99.9 rsETH`.
5. Attacker receives ~0.1 rsETH excess extracted from existing holders' accrued yield.
6. When `updateRSETHPrice()` is eventually called, the inflated supply lowers the price for all holders.

**Foundry fork test plan**: Fork mainnet, snapshot `rsETHPrice` and `_getTotalEthInProtocol()`, advance time to let rewards accrue without calling `updateRSETHPrice()`, deposit as an unprivileged address, assert `rsethAmountToMint > (deposit × liveAssetPrice) / liveRsETHPrice`, then call `updateRSETHPrice()` and assert existing holders' per-rsETH ETH value decreased relative to the pre-deposit baseline.
