### Title
DelegateAction (Meta Transaction) Signed Payload Lacks Chain-ID Binding, Enabling Cross-Network Replay - (File: core/primitives/src/action/delegate.rs)

### Summary
`DelegateAction` and `DelegateActionV2` signed payloads do not include any chain-specific identifier (chain-id or network-id). A `SignedDelegateAction` created and signed for one NEAR network (e.g., testnet) can be replayed verbatim on any other NEAR network (e.g., mainnet) by an unprivileged attacker acting as a relayer, causing the inner actions to execute on behalf of the signer without their consent on the target network.

### Finding Description

`DelegateAction` (NEP-366 meta transactions) allows a user to sign a payload off-chain that a relayer then submits on-chain. The signed payload is hashed via `get_nep461_hash()`, which wraps the action in a `SignableMessage` containing only a `MessageDiscriminant` (a NEP number + on-chain/off-chain flag) and the action body itself. [1](#0-0) 

The `DelegateAction` struct fields are: `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, `public_key`. None of these fields encode a chain-id or network identifier. [2](#0-1) 

The `SignableMessage` wrapper adds only a NEP discriminant — no chain-id: [3](#0-2) 

The same is true for `DelegateActionV2` / `VersionedDelegateActionPayload`: [4](#0-3) 

At runtime, `apply_delegate_action` verifies the signature by calling `signed.verify()`, which recomputes `get_nep461_hash()` — again with no chain-id in scope: [5](#0-4) 

By contrast, regular `SignedTransaction` is protected because its hash covers a `block_hash` field that is chain-specific: [6](#0-5) 

`DelegateAction` has no equivalent binding.

### Impact Explanation

An attacker who obtains a `SignedDelegateAction` intended for NEAR testnet (or any NEAR-based network) can submit it to NEAR mainnet via a relayer transaction. If the signer's account exists on mainnet with the same public key and a nonce lower than the one in the delegate action, the runtime will accept and execute the inner actions — token transfers, key additions/deletions, contract calls, account deletions — on behalf of the signer without their authorization on that network.

The corrupted protocol values are: account balance (unauthorized transfers), access key state (unauthorized key additions/deletions), and contract state (unauthorized function calls). These are irreversible state changes.

### Likelihood Explanation

The preconditions are realistic:
- Many users and developers maintain the same account name and key pair on both NEAR mainnet and testnet.
- `max_block_height` is a plain block height number, not a block hash, so a value valid on testnet is likely also numerically valid on mainnet.
- The nonce only needs to be greater than the current access key nonce on the target network; accounts freshly created on mainnet start at nonce 0, making any testnet-signed action with nonce ≥ 1 immediately replayable.
- The attacker only needs to act as a relayer (an unprivileged role) to submit the outer transaction.

### Recommendation

Include the chain-id (the genesis `chain_id` string, e.g., `"mainnet"`, `"testnet"`) in the signed payload of `DelegateAction` and `DelegateActionV2`. This can be done by adding a `chain_id: String` field to both structs, or by incorporating it into the `MessageDiscriminant` / `SignableMessage` wrapper so that signatures are domain-separated per network. The `chain_id` is available from genesis config and is already accessible in the runtime context.

### Proof of Concept

1. On NEAR testnet, user Alice signs a `DelegateAction` transferring 10 NEAR to Bob, with `nonce=5`, `max_block_height=99999999`.
2. The `get_nep461_hash()` produces a hash over `[NEP-366 discriminant || borsh(DelegateAction)]` — no testnet identifier.
3. Attacker takes the `SignedDelegateAction` and wraps it in a mainnet transaction as the relayer.
4. Mainnet runtime calls `apply_delegate_action`, which calls `signed.verify()`.
5. `verify()` recomputes `get_nep461_hash()` over the same bytes — identical hash, valid signature.
6. If Alice's mainnet account has `nonce < 5` and block height < `99999999`, the transfer executes on mainnet, draining Alice's mainnet balance without her consent. [7](#0-6) [8](#0-7)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

impl<'a, T: BorshSerialize> SignableMessage<'a, T> {
    pub fn new(msg: &'a T, ty: SignableMessageType) -> Self {
        let discriminant = ty.into();
        Self { discriminant, msg }
    }

    pub fn sign(&self, signer: &Signer) -> Signature {
        let bytes = borsh::to_vec(&self).expect("Failed to deserialize");
        let hash = hash(&bytes);
        signer.sign(hash.as_bytes())
    }
