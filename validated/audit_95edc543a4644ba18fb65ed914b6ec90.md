Audit Report

## Title
Block Stuffing Extends Undercollateralized Bridge Window, Temporarily Freezing User Withdrawals - (File: contracts/agETH/AGETHTokenWrapper.sol)

## Summary
`AGETHTokenWrapper` implements a two-step bridge flow where `mint()` (L165–167) creates unbacked wrapper agETH before `depositBridgerAssets()` (L143–151) delivers the backing altAgETH. During this gap, `_withdraw()` (L111–119) has no balance guard and will revert at `safeTransfer` for any user whose requested amount exceeds the contract's current altAgETH balance. An unprivileged attacker can perform block stuffing to artificially extend this undercollateralized window, temporarily freezing all withdrawals.

## Finding Description
`mint()` at L165–167 is callable by `MINTER_ROLE` (the bridge) and mints wrapper tokens with no corresponding altAgETH deposit, intentionally breaking the invariant `altAgETH.balanceOf(address(this)) >= totalSupply()`. The invariant is restored only when `BRIDGER_ROLE` calls `depositBridgerAssets()` at L143–151. The `_withdraw()` function at L111–119 performs no balance sufficiency check: it calls `_burn(msg.sender, _amount)` at L114 and then unconditionally calls `ERC20Upgradeable(_asset).safeTransfer(_to, _amount)` at L116. If the contract's altAgETH balance is less than `_amount`, the `safeTransfer` reverts (and the burn is atomically rolled back). An attacker with no privileged role fills every block with high-gas transactions at a gas price sufficient to crowd out the `BRIDGER_ROLE`'s `depositBridgerAssets()` call, extending the undercollateralized window for as long as the stuffing is sustained. On L2 chains (the stated deployment target per the contract's NatSpec at L13), block gas limits are bounded and per-gas costs are low, making sustained block stuffing economically feasible.

## Impact Explanation
During the stuffed window, every `withdraw()` and `withdrawTo()` call reverts for any user whose requested amount exceeds the contract's altAgETH balance. Users holding wrapper agETH cannot redeem it for the underlying asset. Tokens are not lost (the burn reverts atomically), but funds are temporarily frozen. This matches **Medium. Temporary freezing of funds** and **Low. Block stuffing** from the allowed impact scope.

## Likelihood Explanation
The precondition — unbacked supply from `mint()` — is the normal, intended bridge operation and occurs on every cross-chain transfer before the bridger settles. The attack requires no privileged access, no oracle manipulation, and no governance capture; only ETH to pay for filler transactions. On L2 chains with low per-gas costs and bounded block gas limits, block stuffing is more economically viable than on Ethereum mainnet. The attack window is bounded by bridge settlement latency (minutes to hours), which is sufficient to cause meaningful disruption to users attempting withdrawals.

## Recommendation
1. **Track backed supply separately**: Introduce a `backedSupply` state variable incremented only in `_deposit()` and `depositBridgerAssets()`, and decremented in `_withdraw()`. Restrict `_withdraw()` to `backedSupply` rather than relying on the raw ERC20 balance, so unbacked minted supply is never redeemable until backing arrives.
2. **Add an explicit balance guard in `_withdraw()`**: Before burning, check `ERC20Upgradeable(_asset).balanceOf(address(this)) >= _amount` and revert with a descriptive error rather than allowing the opaque `safeTransfer` revert.
3. **Withdrawal queue**: If immediate satisfaction is not possible, queue the withdrawal rather than reverting, so block stuffing causes delay rather than hard reverts.

## Proof of Concept
```solidity
function testBlockStuffingFreezesWithdraw() public {
    address altAgETH = /* allowed token address */;
    uint256 N = 100e18;

    // Step 1: MINTER_ROLE mints unbacked agETH (normal bridge mint on L2)
    vm.prank(minterRole);
    wrapper.mint(user, N);

    // Invariant broken: totalSupply > altAgETH balance in wrapper
    assertGt(wrapper.totalSupply(), ERC20(altAgETH).balanceOf(address(wrapper)));

    // Step 2: Simulate block stuffing — advance blocks without depositBridgerAssets()
    vm.roll(block.number + 500);

    // Step 3: User tries to withdraw — reverts because safeTransfer fails
    vm.prank(user);
    vm.expectRevert();
    wrapper.withdraw(altAgETH, N);

    // Invariant still broken throughout the stuffed window
    assertLt(
        ERC20(altAgETH).balanceOf(address(wrapper)),
        wrapper.totalSupply()
    );
}
```
The test confirms that after `mint()` creates unbacked supply and `depositBridgerAssets()` is excluded (simulated by block advancement without calling it), `withdraw()` reverts for all users holding the unbacked tokens. The root cause is the absence of a balance guard in `_withdraw()` at L111–119, combined with the intentional mint-before-deposit bridge design at L165–167.
