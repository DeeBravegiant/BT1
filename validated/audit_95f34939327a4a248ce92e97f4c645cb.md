### Title
Missing Bounds Validation on `sub_epoch_n` Index into `summaries` List During Weight Proof Validation - (File: `chia/full_node/weight_proof.py`)

### Summary
`_validate_sub_epoch_segments` in `chia/full_node/weight_proof.py` indexes into the `summaries` list using `sub_epoch_n` values derived entirely from attacker-controlled `WeightProof` data, without first checking that `sub_epoch_n < len(summaries)`. A malicious peer can craft a `WeightProof` where segment `sub_epoch_n` values exceed the number of summaries, triggering an unhandled `IndexError` that crashes weight proof validation and prevents honest nodes from syncing.

### Finding Description

In `_validate_sub_epoch_segments`, the outer loop iterates over `segments_by_sub_epoch`, which is built from `weight_proof.sub_epoch_segments` — fully attacker-controlled:

```python
segments_by_sub_epoch = map_segments_by_sub_epoch(sub_epoch_segments.challenge_segments)
...
for sub_epoch_n, segments in segments_by_sub_epoch.items():
    ...
    prev_ses = summaries[sub_epoch_n - 1]          # line 983 — no bounds check
    rc_sub_slot_hash = rc_sub_slot.get_hash()
    if not summaries[sub_epoch_n].reward_chain_hash == rc_sub_slot_hash:  # line 985 — no bounds check
``` [1](#0-0) 

The `summaries` list is built from `weight_proof.sub_epoch_data` (also attacker-controlled, but independently sized). There is no guard of the form `if sub_epoch_n >= len(summaries): return None` before either index access. An attacker can set `sub_epoch_data` to contain N entries while embedding segments with `sub_epoch_n >= N`, causing a Python `IndexError` at line 983 or 985.

A second, related missing check exists in `__get_cc_sub_slot`, called from `__validate_pospace`:

```python
def __get_cc_sub_slot(sub_slots, idx, ses):
    sub_slot: SubSlotData | None = None
    for i in reversed(range(idx)):   # empty when idx == 0
        sub_slot = sub_slots[i]
        if sub_slot.cc_slot_end_info is not None:
            break
    assert sub_slot is not None          # AssertionError when idx == 0
    assert sub_slot.cc_slot_end_info is not None
``` [2](#0-1) 

When the challenge block is at `sub_slots[0]` and `sub_epoch_n > 0`, `__get_cc_sub_slot` is called with `idx=0`. `reversed(range(0))` is empty, `sub_slot` stays `None`, and the bare `assert` raises `AssertionError`. No length/bounds check guards this path.

The SEC-614 fix (CHANGELOG line 31) addressed `__get_rc_sub_slot`'s analogous `assert first_idx` crash, but `__get_cc_sub_slot` retains the same pattern. [3](#0-2) 

### Impact Explanation

Both `validate_weight_proof_single_proc` and `validate_weight_proof_inner` call `_validate_sub_epoch_segments` without wrapping it in a try/except:

```python
if _validate_sub_epoch_segments(...) is None:
    return False, uint32(0)
``` [4](#0-3) 

An `IndexError` or `AssertionError` is not `None`; it propagates as an unhandled exception, crashing the validation task. This prevents the honest node from completing weight proof validation and therefore from syncing to the chain — a **High** impact: long-lived inability for honest nodes to process sync updates under normal network assumptions.

### Likelihood Explanation

Any peer that can send a `RespondProofOfWeight` message can supply a crafted `WeightProof`. The `WeightProof` is a `Streamable` type deserialized from raw bytes; `sub_epoch_n` inside each `SubEpochChallengeSegment` is an unconstrained `uint32` field. No pre-validation step enforces `sub_epoch_n < len(sub_epoch_data)` before `_validate_sub_epoch_segments` is entered. The attacker needs no keys, no stake, and no privileged access.

### Recommendation

1. In `_validate_sub_epoch_segments`, add an explicit bounds check before indexing `summaries`:
   ```python
   if sub_epoch_n >= len(summaries):
       log.error(f"sub_epoch_n {sub_epoch_n} out of range for summaries (len={len(summaries)})")
       return None
   ```
2. In `__get_cc_sub_slot`, replace the bare `assert` statements with guarded returns (matching the pattern used in the SEC-614 fix for `__get_rc_sub_slot`):
   ```python
   if sub_slot is None or sub_slot.cc_slot_end_info is None:
       return None  # propagate as validation failure, not crash
   ```
   And update the call site in `__validate_pospace` to handle `None`.

### Proof of Concept

Craft a `WeightProof` where:
- `sub_epoch_data` contains exactly 2 `SubEpochData` entries (so `summaries` has length 2 after `_validate_sub_epoch_summaries`).
- `sub_epoch_segments` contains one `SubEpochChallengeSegment` with `sub_epoch_n = 5`.

When `_validate_sub_epoch_segments` iterates `segments_by_sub_epoch` and reaches `sub_epoch_n = 5`, it executes `summaries[5]` on a list of length 2, raising `IndexError`. This exception propagates uncaught through `validate_weight_proof_single_proc` or `validate_weight_proof_inner`, crashing weight proof validation and blocking the node's sync path. [5](#0-4)
