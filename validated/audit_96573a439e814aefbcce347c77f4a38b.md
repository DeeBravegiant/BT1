Audit Report

## Title
Stale Cross-Chain Rate Used for rsETH Minting Without Staleness Check ‚ÄĒ (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary

`CrossChainRateReceiver.getRate()` returns a cached `rate` with no comparison against `lastUpdated`, allowing the rate to be arbitrarily old. All L2 pool contracts (`RSETHPool`, `RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolV3WithNativeChainBridge`) use this rate to compute rsETH minting amounts. Because rsETH continuously accrues yield, a stale (lower) rate causes the division to yield excess rsETH, which the depositor can later redeem at the true higher rate, extracting value from existing holders' accumulated yield.

## Finding Description

`CrossChainRateReceiver` stores `rate` and `lastUpdated` but `getRate()` unconditionally returns `rate`:

```solidity
// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

The only update path is `lzReceive()`, which requires an off-chain actor to call `CrossChainRateProvider.updateRate()` and pay LayerZero fees. There is no on-chain heartbeat enforcement, no keeper obligation, and no revert path if the rate is arbitrarily old. [1](#0-0) 

Every L2 pool's `deposit()` path calls `viewSwapRsETHAmountAndFee()`, which calls `getRate()` and divides by it:

```solidity
// RSETHPoolV3.sol L299-308
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();   // stale, no check
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
}
``` [2](#0-1) 

The same pattern is present in `RSETHPool.viewSwapRsETHAmountAndFee()`: [3](#0-2) 

Because rsETH is yield-bearing, its true ETH redemption value grows monotonically. A stale (lower) `rate` causes the division `amountAfterFee * 1e18 / rsETHToETHrate` to produce a larger `rsETHAmount` than the depositor is entitled to at the current L1 rate. The `lastUpdated` field is publicly readable, making the staleness window trivially observable before executing a deposit. [4](#0-3) 

No existing guard in any pool contract checks `lastUpdated` before accepting a deposit. The `whenNotPaused` modifier and `dailyMintLimit` modifier do not address rate staleness. [5](#0-4) 

## Impact Explanation

**High ‚ÄĒ Theft of unclaimed yield.**

rsETH accrues staking and restaking rewards on L1, causing its ETH redemption value to increase over time. When the L2 oracle rate lags behind the true L1 rate, a depositor pays N ETH but receives rsETH priced at the old (lower) rate ‚ÄĒ more rsETH than N ETH currently buys at the true rate. When those tokens are later redeemed or traded at the current (higher) rate, the depositor extracts value that belongs to existing rsETH holders as accumulated yield. The magnitude scales with the rate gap and deposit size; a 0.57% rate lag on a 1,000 ETH deposit yields approximately 5.71 ETH of stolen yield. This matches the allowed impact: **High ‚ÄĒ Theft of unclaimed yield**.

## Likelihood Explanation

**Medium.** The rate update requires an off-chain actor to call `CrossChainRateProvider.updateRate()` and pay LayerZero fees. During periods of high gas prices, network congestion, or keeper downtime, the rate can remain stale for hours or days. rsETH accrues value continuously (~4‚Äď6% APY), so even a 24-hour staleness window creates a measurable exploitable gap. Any unprivileged depositor can read `lastUpdated` on-chain and time their deposit to coincide with maximum staleness. The attack requires no special privileges, no flash loans, and no victim interaction ‚ÄĒ only a public `deposit()` call.

## Recommendation

1. Add a `maxStaleness` parameter to `CrossChainRateReceiver` and revert in `getRate()` if `block.timestamp - lastUpdated > maxStaleness`.
2. Alternatively, have each L2 pool contract check `lastUpdated` before accepting deposits and revert or pause if the rate exceeds a configured staleness threshold.
3. Implement an automated keeper or Chainlink Automation job to push rate updates on a regular heartbeat, ensuring the staleness window is bounded.

## Proof of Concept

1. At `T=0`, the L2 oracle `rate` is `1.050e18`. The true L1 rate is also `1.050e18`. `lastUpdated = T`.
2. No `updateRate()` call is made for 48 hours. The true L1 rate grows to `1.056e18` (~0.57% increase). `CrossChainRateReceiver.rate` remains `1.050e18`.
3. Attacker reads `lastUpdated` on-chain, confirms staleness, then calls `RSETHPoolV3.deposit{value: 1000 ether}("")`.
4. `viewSwapRsETHAmountAndFee(1000e18)` computes `rsETHAmount = 1000e18 * 1e18 / 1.050e18 ‚Čą 952.38 rsETH`.
5. At the current L1 rate, 952.38 rsETH is worth `952.38 * 1.056e18 / 1e18 ‚Čą 1005.71 ETH`.
6. Attacker deposited 1000 ETH and holds rsETH worth ~1005.71 ETH ‚ÄĒ a ~5.71 ETH gain extracted from existing holders' accumulated yield.
