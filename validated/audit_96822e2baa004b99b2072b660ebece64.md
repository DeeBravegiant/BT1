### Title
Ciphertext Malleability via Non-Strict Deserialization in `parse_batch_ciphertext` - (`src/cbmpc/api/pve_batch_single_recipient.cpp`)

### Summary

`parse_batch_ciphertext` uses the non-strict `coinbase::convert<T>` template to deserialize the outer `pve_batch_ciphertext_blob_v1_t`. Unlike `coinbase::deser`, `convert<T>` does not check whether all input bytes were consumed. An attacker can append arbitrary trailing bytes to any valid ciphertext blob; the function accepts it, decryption succeeds, and the same plaintext scalar vector is returned. This produces ciphertext malleability: multiple distinct byte strings that all decrypt to the same value.

### Finding Description

`parse_batch_ciphertext` is the sole entry point for outer-blob parsing in every public decrypt/verify/get path:

```cpp

``` [1](#0-0) 

The non-strict `convert<T>` template returns as soon as the struct fields are consumed, without checking whether the input is fully consumed:

```cpp

The full reachable call chain from the public C API is:

`cbmpc_pve_batch_decrypt` → `coinbase::api::pve::decrypt_batch` → `parse_batch_ciphertext` → `coinbase::convert` (non-strict) → returns SUCCESS with trailing bytes silently ignored → `decrypt_batch` proceeds normally. [5](#0-4) [6](#0-5) 

The same non-strict call is also present in `verify_batch`, `get_batch_count`, `get_public_keys_compressed_batch`, and `get_Label_batch`, all of which call `parse_batch_ciphertext`. [7](#0-6) [8](#0-7) 

### Impact Explanation

Any caller that stores, hashes, or deduplicates ciphertext blobs by their raw bytes (e.g., an audit log, a replay-detection cache, a backup uniqueness check) will treat `ct` and `ct || <garbage>` as distinct entries even though both decrypt to the same scalar vector. An attacker who can submit ciphertexts to such a system can:

- Produce an unbounded number of "distinct" ciphertexts that all decrypt to the same key material, defeating deduplication.
- Cause an audit log to record a different byte string than the one the decryptor accepted, breaking log integrity.
- Bypass replay-detection that compares raw ciphertext bytes.

This falls under the Medium impact scope: a public API reachable parsing invariant break that causes unsafe state acceptance with security impact.

### Likelihood Explanation

The attacker only needs to be an unprivileged API caller who can supply a ciphertext blob. No threshold collusion, no key material, and no cryptographic capability is required. The trailing bytes can be any value; zero bytes suffice. The attack is deterministic and requires no brute force.

### Recommendation

Replace `coinbase::convert` with `coinbase::deser` in `parse_batch_ciphertext`:

```cpp
// src/cbmpc/api/pve_batch_single_recipient.cpp:35
// Before:
error_t rv = coinbase::convert(out_blob, ciphertext);
// After:
error_t rv = coinbase::deser(ciphertext, out_blob);
``` [9](#0-8) 

`deser` enforces `converter.get_offset() == converter.get_size()` and returns `E_BADARG` on any trailing bytes, making the encoding canonical. The same fix should be applied to the inner `coinbase::convert(pve_ct, blob.ct)` call at line 163 (and its equivalents in `verify_batch`, `get_public_keys_compressed_batch`, and `get_Label_batch`) to prevent malleability at the inner blob layer as well. TEST(ApiPveBatch, TrailingBytesAccepted_Malleability) {
  const toy_base_pke_t base_pke;
  const buf_t ek = buf_t("ek");
  const buf_t dk = buf_t("dk");
  const buf_t label = buf_t("label");

  std::array<uint8_t, 32> x_bytes{};
  x_bytes[31] = 0x42;
  std::vector<mem_t> xs;
  xs.emplace_back(x_bytes.data(), 32);

  // Step 1: produce a valid canonical ciphertext
  buf_t ct;
  ASSERT_EQ(coinbase::api::pve::encrypt_batch(
      base_pke, curve_id::secp256k1, ek, label, xs, ct), SUCCESS);

  // Step 2: append 16 trailing zero bytes
  buf_t ct_malleated = ct;
  uint8_t zeros[16] = {};
  ct_malleated = ct_malleated + mem_t(zeros, 16);

  // Step 3: decrypt the malleated ciphertext — SHOULD be rejected, IS accepted
  std::vector<buf_t> xs_out;
  error_t rv = coinbase::api::pve::decrypt_batch(
      base_pke, curve_id::secp256k1, dk, ek, ct_malleated, label, xs_out);

  // This assertion FAILS (rv == SUCCESS), demonstrating the vulnerability:
  EXPECT_NE(rv, SUCCESS);

  // The two ciphertexts are byte-distinct but produce identical plaintext:
  EXPECT_NE(ct, ct_malleated);
