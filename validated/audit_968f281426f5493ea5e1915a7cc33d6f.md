### Title
Proposer's deposit permanently lost when `do_update` clears all entries or proposer is removed from participant set — (`crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

When a participant calls `propose_update`, they must attach a required deposit (storage-staking cost proportional to the update payload size). This deposit is never refunded to the proposer. When any update reaches threshold and `do_update` executes, it silently clears **all** pending update entries — including those from other proposers — without returning their deposits. Additionally, a participant who is removed from the active set via resharing permanently loses their deposit because no recovery path exists and all relevant functions gate on current-participant status.

---

### Finding Description

`propose_update` collects a mandatory deposit from the caller: [1](#0-0) 

The required portion is kept by the contract; only the excess is refunded to the proposer. The `UpdateEntry` stored on-chain does **not** record the proposer's account ID: [2](#0-1) 

When any update reaches threshold, `do_update` is called, which removes the winning entry and then calls `entries.clear()` and `vote_by_participant.clear()` — wiping every other pending proposal without issuing any refund: [3](#0-2) 

Because the proposer's account ID is not stored in `UpdateEntry`, there is no information available to issue a refund even if the code attempted to do so.

The only other removal path — `remove_non_participant_votes` — only removes votes, not entries, and does not touch deposits: [4](#0-3) 

Furthermore, `remove_update_vote` (the only caller-initiated cleanup) is gated on `voter_or_panic`, which requires the caller to be a current participant: [5](#0-4) 

A participant who proposed an update and is subsequently removed from the active set via resharing cannot call any contract function to recover their deposit. The deposit is permanently absorbed into the contract's balance.

---

### Impact Explanation

Every participant who proposes an update that is not the one ultimately executed loses their full required deposit. For a realistic contract-code update (hundreds of KB of WASM), the deposit is computed as: [6](#0-5) 

At NEAR's storage byte cost (~10 yoctoNEAR/byte), a 500 KB WASM payload yields a deposit of ~5 NEAR per proposal. With multiple competing proposals (a realistic governance scenario), the total unrecoverable loss across all non-winning proposers can be significant. This breaks the production accounting invariant that a participant who pays a deposit to propose an update should be able to recover it if their proposal is not executed.

**Impact class: Medium** — balance/accounting invariant broken without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

This is triggered in any governance round where more than one update is proposed before threshold is reached — a normal operational pattern when participants disagree on which code version to deploy. It also triggers whenever a participant is removed via resharing after having proposed an update. Both scenarios are realistic in production.

---

### Recommendation

1. Store the proposer's `AccountId` inside `UpdateEntry` at proposal time.
2. In `do_update`, before calling `entries.clear()`, iterate over all remaining entries and transfer each entry's deposit back to its recorded proposer.
3. Alternatively, expose a `cancel_update(id: UpdateId)` function callable by the original proposer (identified by the stored account ID) that removes the entry and refunds the deposit — without requiring the caller to still be a current participant, analogous to the Morpho fix of removing the `onlyStaker` modifier.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB WASM payload, paying ~5 NEAR deposit. Entry `id=0` is stored; A's account ID is **not** recorded.
2. Participant B calls `propose_update` with a different WASM payload, paying ~5 NEAR deposit. Entry `id=1` is stored.
3. Participants vote for B's update (`id=1`). Threshold is reached; `vote_update` calls `do_update(&id=1, ...)`.
4. Inside `do_update`: `entries.remove(&id=1)` removes B's entry; `entries.clear()` removes A's entry `id=0`. No refund is issued to A.
5. A's ~5 NEAR deposit is permanently absorbed into the contract balance. A has no callable function to recover it.

Alternate path: after step 1, a resharing removes A from the participant set. A attempts to call `remove_update_vote` or any other cleanup function — all are gated on `voter_or_panic` / current-participant checks — and is rejected. A's deposit remains locked indefinitely. [7](#0-6) [8](#0-7)

    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
