### Title
Any Attested Participant Can Inject an Arbitrary CKD Response for the `AppPublicKey` Variant, Enabling Derived-Key Theft — (`File: crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC smart contract performs **no cryptographic verification** of the `CKDResponse` when the request uses the `AppPublicKey` (privately-verifiable / legacy) variant. Any single attested participant can race to call `respond_ckd` with an attacker-chosen `(big_y, big_c)` pair, causing the user to decrypt an attacker-controlled value as their derived secret key.

---

### Finding Description

`respond_ckd` branches on the request's `app_public_key` variant:

```rust
// crates/contract/src/lib.rs : 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)`. [2](#0-1) 

For `AppPublicKey`, the arm is an empty block — the response is accepted unconditionally. The only gate is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be **any** attested participant, not the designated leader. [3](#0-2) 

The node configuration explicitly acknowledges that `respond` calls are not account-authenticated:

```
// crates/node/src/config.rs : 412
/// It can be an arbitrary id because respond calls are not authenticated.
pub account_id: AccountId,
``` [4](#0-3) 

For signatures this is safe because the contract verifies the ECDSA/EdDSA signature cryptographically. For `AppPublicKey` CKD, there is no equivalent check.

The user decrypts the response as:

```rust
// crates/threshold-signatures/src/confidential_key_derivation.rs : 53-55
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
``` [5](#0-4) 

If an attacker supplies `big_y = 0` (the G1 identity point) and `big_c = s_attacker · G1`, then `a · big_y = 0` regardless of the user's private scalar `a`, and the user's derived key becomes `s_attacker · G1` — a value the attacker chose and knows.

---

### Impact Explanation

**Critical — Confidential key derivation output theft without required participant authorization.**

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Monitor the chain for a pending `AppPublicKey` CKD request.
2. Immediately call `respond_ckd` with `big_y = identity_G1` and `big_c = attacker_chosen_G1_point` before the legitimate leader completes the multi-party protocol.
3. The contract accepts the response (no verification branch for `AppPublicKey`).
4. The user decrypts `big_c - a · 0 = big_c`, receiving the attacker-chosen value as their derived secret.
5. The attacker knows the user's derived key.

The first valid `respond_ckd` call resolves the yield and removes the request from `pending_ckd_requests`; the legitimate leader's subsequent response is silently dropped. The user has no way to detect the substitution.

---

### Likelihood Explanation

- The `AppPublicKey` variant is the legacy/default path and is actively used in production (e2e tests, gas regression tests, and the `ckd-example-cli` all use it).
- The attacker only needs to be one attested participant — well below the threshold.
- The attack window is the latency between the request appearing on-chain and the legitimate leader completing the multi-party CKD protocol (at least one P2P round-trip). The attacker submits a single NEAR transaction with no cryptographic work.
- The `RespondConfig` design (arbitrary account, many access keys for throughput) makes it straightforward for an attacker to submit the fake response with low latency. [6](#0-5) 

---

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. The check must verify that `big_c` encodes a valid ElGamal ciphertext under the network's BLS public key, even without the G2 component of the app key. Concretely:

- Verify `big_y` and `big_c` are valid, non-identity G1 points.
- Require the caller to prove knowledge of the discrete log relationship between `big_y` and `big_c` relative to the network public key (e.g., via a Schnorr proof or by upgrading all callers to the `AppPublicKeyPV` path).
- Alternatively, deprecate `AppPublicKey` and require all new requests to use `AppPublicKeyPV`, which already has a sound on-chain check.

---

### Proof of Concept

1. User submits `request_app_private_key` with `AppPublicKey(A)` where `A = a · G1`.
2. Attacker (attested participant) observes the request on-chain.
3. Attacker calls:
   ```json
   respond_ckd(
     request = <user's CKDRequest>,
     response = { "big_y": "<G1 identity point>", "big_c": "<attacker_chosen_G1>" }
   )
   ```
4. Contract executes lines 675–676 (`AppPublicKey` arm is empty), skips all verification, and resolves the yield with the attacker's response.
5. User receives `{ big_y: 0, big_c: attacker_chosen }` and computes `S = attacker_chosen - a · 0 = attacker_chosen`.
6. The attacker, who chose `attacker_chosen`, now knows the user's derived secret key `S`. [7](#0-6) [8](#0-7)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
