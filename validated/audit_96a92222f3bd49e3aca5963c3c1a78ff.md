### Title
`directDeposits` Credited to Account Balances Without Being Included in Preservation-of-Value Check, Enabling ADA Creation from Nothing — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` transaction body field that allows a transaction to directly credit ADA to registered stake accounts. The `ENTITIES` and `SUBENTITIES` rules apply these credits via `applyDirectDeposits`, which adds the specified amounts to account balances. However, neither the `consumed` nor the `produced` side of the preservation-of-value equation (`validateValueNotConservedUTxO`) accounts for `directDeposits`. An unprivileged transaction sender can therefore include arbitrary `directDeposits` amounts, pass the value-conservation check, and have those amounts credited to accounts — creating ADA from nothing.

---

### Finding Description

**Step 1 — `directDeposits` is applied to account balances.**

In `dijkstraEntitiesTransition` (the `ENTITIES` rule for top-level transactions):

```haskell
let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
    accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
  injectFailure . DirectDepositsToMissingAccounts
pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits
``` [1](#0-0) 

`applyDirectDeposits` unconditionally adds each entry's coin amount to the matching account balance:

```haskell
applyDirectDeposits (DirectDeposits dd) =
  updateAccountBalances
    (\amount account -> addCompactCoin amount (account ^. balanceAccountStateL))
    dd
``` [2](#0-1) 

The same pattern is repeated in `dijkstraSubEntitiesTransition` for sub-transactions: [3](#0-2) 

**Step 2 — `directDeposits` is absent from both sides of the value-conservation equation.**

The `UTXO` rule enforces value conservation at line 381:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [4](#0-3) 

`validateValueNotConservedUTxO` delegates to `consumed` and `produced`: [5](#0-4) 

For Dijkstra, `consumed` resolves to `getConsumedDijkstraValue`, which sums UTxO inputs, withdrawals, refunds, and minted multi-assets — **no `directDeposits`**: [6](#0-5) 

`produced` resolves to `getProducedDijkstraValue` → `dijkstraProducedValue` → `conwayProducedValue` (outputs + fees + cert deposits + treasury donation + burned multi-assets) — **no `directDeposits`**: [7](#0-6) 

For sub-transactions, `dijkstraSubTxProducedValue` is equally missing `directDeposits`: [8](#0-7) 

**Step 3 — The `SUBUTXO` rule has no value-conservation check at all.**

`dijkstraSubUtxoTransition` validates inputs, outputs, and network IDs, but never calls `validateValueNotConservedUTxO`: [9](#0-8) 

This means sub-transactions can carry `directDeposits` with zero scrutiny on value conservation.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker submits a Dijkstra-era transaction with `directDeposits = {myStakeAddr: N ADA}` where `N` is arbitrarily large. The preservation-of-value check passes because `directDeposits` is not counted on either side. `applyDirectDeposits` then adds `N` ADA to the attacker's account balance. The ADA was never deducted from any UTxO input. Total ADA in circulation increases by `N`, violating the fixed-supply invariant and constituting direct creation of ADA through an invalid ledger state transition.

---

### Likelihood Explanation

Any unprivileged transaction sender can exploit this in the Dijkstra era. The only precondition is that the target account address is registered (which the attacker controls). No special keys, governance majority, or privileged access is required. The attack is deterministic and repeatable.

---

### Recommendation

Add the sum of all `directDeposits` coin values to the `produced` side of the value-conservation equation, analogously to how certificate deposits are handled. Concretely, `dijkstraProducedValue` and `dijkstraSubTxProducedValue` should include:

```haskell
<> inject (foldMap id . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
```

This ensures that the ADA credited to accounts via `directDeposits` must be funded by the transaction's UTxO inputs, preserving the total ADA supply.

---

### Proof of Concept

1. Register a stake credential `C` and obtain its `AccountAddress` `A`.
2. Construct a Dijkstra-era transaction `tx` with:
   - Inputs covering only fees (e.g., 0.2 ADA input, 0 ADA output, 0.2 ADA fee).
   - `directDeposits = {A: 1_000_000_000_000}` (one million ADA).
3. Submit `tx`. `validateValueNotConservedUTxO` checks `consumed == produced`:
   - `consumed = 0.2 ADA` (UTxO input)
   - `produced = 0 ADA (outputs) + 0.2 ADA (fee) = 0.2 ADA`
   - Check passes.
4. `ENTITIES` rule calls `applyDirectDeposits`, crediting 1,000,000 ADA to account `C`.
5. The attacker now has 1,000,000 ADA in their reward account, withdrawable via a standard withdrawal transaction. One million ADA has been created from nothing.

  ( DijkstraEraTxBody era
  , EraUTxO era
  , Value era ~ MaryValue
  , STxLevel l era ~ STxBothLevels l era
  ) =>
  PParams era ->
  (Credential Staking -> Maybe Coin) ->
  (Credential DRepRole -> Maybe Coin) ->
  UTxO era ->
  TxBody l era ->
  Value era
getConsumedDijkstraValue pp lookupStakingDeposit lookupDRepDeposit utxo txBody =
  withBothTxLevels
    txBody
    ( \topTxBody ->
        txBodyConsumedValue topTxBody <> subTransactionsConsumedValue topTxBody
    )
    txBodyConsumedValue
  where
    txBodyConsumedValue :: forall m. TxBody m era -> Value era
    txBodyConsumedValue = getConsumedMaryValue pp lookupStakingDeposit lookupDRepDeposit utxo
    subTransactionsConsumedValue topTxBody =
      foldMap'
        (getConsumedValue pp lookupStakingDeposit lookupDRepDeposit utxo . view bodyTxL)
        (topTxBody ^. subTransactionsTxBodyL)
