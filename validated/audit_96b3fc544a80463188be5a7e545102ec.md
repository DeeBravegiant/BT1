## Full Call Chain Trace

**Entry point:** `coin_state_update` in `wallet_node_api.py` [1](#0-0) 

→ `state_update_received` in `wallet_node.py` [2](#0-1) 

→ `add_states_from_peer` — **first guard check** [3](#0-2) 

For an **untrusted** peer with `fork_height is not None` (always the case for `CoinStateUpdate`), states go to the **race cache**, not directly applied: [4](#0-3) 

The race cache is later drained in `sync_from_untrusted_close_to_peak`, calling `add_states_from_peer` with `fork_height=None`, which routes through `validate_and_add` → `validate_received_state_from_peer`: [5](#0-4) 

## Coin-State Validation (Untrusted Path)

`validate_received_state_from_peer` performs real cryptographic validation:
1. Requests and validates the header block at the created/spent height
2. Validates an **additions Merkle proof** (coin exists in block)
3. Validates a **removals Merkle proof** (coin was spent in block)
4. Validates **block inclusion** via weight proof [6](#0-5) 

This means the coin state itself **cannot be forged** — the singleton must actually be spent on-chain at the claimed height.

    async def state_update_received(self, request: CoinStateUpdate, peer: WSChiaConnection) -> None:
        # This gets called every time there is a new coin or puzzle hash change in the DB
        # that is of interest to this wallet. It is not guaranteed to come for every height. This message is guaranteed
        # to come before the corresponding new_peak for each height. We handle this differently for trusted and
        # untrusted peers. For trusted, we always process the state, and we process reorgs as well.
        for coin in request.items:
            self.log.info(f"request coin: {coin.coin.name().hex()}{coin}")

        async with self.wallet_state_manager.lock:
            await self.add_states_from_peer(
                request.items,
                peer,
                request.fork_height,
                request.height,
            )
