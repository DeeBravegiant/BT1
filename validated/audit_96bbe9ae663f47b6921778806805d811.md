### Title
Expired TEE Attestation Not Checked in `is_caller_an_attested_participant`, Allowing Expired Participants to Submit Threshold Signatures and Governance Votes - (File: crates/contract/src/tee/tee_state.rs)

---

### Summary

`is_caller_an_attested_participant` — the sole gate used by `assert_caller_is_attested_participant_and_protocol_active` — verifies identity (account ID, account public key) but **never checks whether the stored attestation has expired**. A participant whose TEE attestation has lapsed can therefore continue calling `respond`, `vote_pk`, `vote_reshared`, `start_keygen_instance`, `start_reshare_instance`, and `vote_abort_key_event_instance` until `verify_tee()` is explicitly invoked by someone. This is the direct structural analog to the Switchboard finding: an expired oracle (here, an expired TEE attestation) is accepted without checking its validity timestamp.

---

### Finding Description

`assert_caller_is_attested_participant_and_protocol_active` is the authorization guard for every critical node-facing method: [1](#0-0) 

It delegates to `is_caller_an_attested_participant`: [2](#0-1) 

The function performs four checks:
1. Caller is in the active participants list.
2. A stored attestation exists for the caller's TLS key.
3. The stored `account_id` matches the caller.
4. The stored `account_public_key` matches the signer's Ed25519 key.

**There is no check against `env::block_timestamp()` or the attestation's `expiry_timestamp_seconds`.** The expiration check exists only in `reverify_participants`, which is called exclusively from `reverify_and_cleanup_participants` (triggered by `verify_tee()`) and `clean_invalid_attestations`: [3](#0-2) 

The `accept_requests` flag — checked inside `respond` — is only set to `false` when `verify_tee()` is called and finds fewer than threshold valid attestations: [4](#0-3) 

`verify_tee()` is not called automatically; it is a participant-initiated call scheduled every ~2 days per the design: [5](#0-4) 

Between the moment an attestation expires and the next `verify_tee()` call, `accept_requests` remains `true` and `is_caller_an_attested_participant` continues to return `Ok(())` for the expired participant.

---

### Impact Explanation

During the expiry window (up to 2 days, or indefinitely if `verify_tee()` is never called), a participant whose TEE attestation has expired can:

- Call `respond` / `respond_ckd` / `respond_verify_foreign_tx` and contribute their share to threshold signatures.
- Call `vote_pk` and `vote_reshared` to advance or complete key-generation and resharing ceremonies.
- Call `start_keygen_instance` / `start_reshare_instance` if they hold the leader position.

An expired attestation means the TEE certificate chain is no longer valid — the node may be running a non-approved image or operating outside a genuine TDX enclave. Allowing such a node to contribute to threshold signing breaks the core invariant that only nodes with live, verified TEE attestations participate in MPC operations. This maps to the **Medium** allowed impact: participant-state and contract execution-flow manipulation that breaks production safety/accounting invariants (the TEE security model) without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

Attestations have a 7-day validity window (`DEFAULT_EXPIRATION_DURATION_SECONDS`). Nodes are expected to resubmit every 7 days (or hourly per the newer design). A node that fails to resubmit — due to a crash, connectivity issue, or deliberate choice — will have an expired attestation. Because `verify_tee()` is called at most every 2 days and is not automatic, the window of exploitation is real and bounded only by operator diligence. Any participant (including a Byzantine one below the signing threshold) can exploit this by simply not renewing their attestation while continuing to call `respond` or governance vote methods.

---

### Recommendation

Add an expiration check inside `is_caller_an_attested_participant`. After confirming the attestation exists and the identity matches, call `reverify_participants` (or inline the equivalent `re_verify` call) and return an error if the attestation has expired:

```rust
// After identity checks pass:
let tee_status = self.reverify_participants(&attestation.node_id, tee_upgrade_deadline_duration);
if !matches!(tee_status, TeeQuoteStatus::Valid) {
    return Err(AttestationCheckError::AttestationExpiredOrInvalid);
}
```

This mirrors the check already performed in `reverify_and_cleanup_participants` and ensures that every call to `respond`, `vote_pk`, `vote_reshared`, etc. is gated on a live, non-expired attestation — consistent with the TEE security model.

---

### Proof of Concept

1. Contract is in `Running` state with 3 participants (threshold 2). All have valid attestations.
2. Participant P3's attestation expires (e.g., 7 days pass without resubmission). `verify_tee()` is **not** called.
3. A user calls `sign(...)`. The contract enqueues the request (`accept_requests == true`).
4. P3 calls `respond(request, signature_response)`:
   - `assert_caller_is_attested_participant_and_protocol_active()` is invoked.
   - `is_caller_an_attested_participant` finds P3 in the participants list, finds the stored attestation, confirms account ID and public key match — **returns `Ok(())`** without checking expiration.
   - `accept_requests` is `true` (no `verify_tee()` was called).
   - P3's response is accepted and contributes to the threshold signature.
5. P3 — whose TEE certificate is expired and whose enclave integrity is unverified — has successfully contributed to a threshold signature, violating the TEE security invariant. [2](#0-1) [6](#0-5) [7](#0-6)

    /// reverifies stored participant attestations.
    pub(crate) fn reverify_participants(
        &self,
        node_id: &NodeId,
        tee_upgrade_deadline_duration: Duration,
    ) -> TeeQuoteStatus {
        let allowed_mpc_docker_image_hashes =
            self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration);
        let allowed_launcher_compose_hashes = self.get_allowed_launcher_compose_hashes();
        let allowed_measurements = self.get_accepted_measurements();

        let participant_attestation = self.stored_attestations.get(&node_id.tls_public_key);
        let Some(participant_attestation) = participant_attestation else {
            return TeeQuoteStatus::Invalid("participant has no attestation".to_string());
        };

        // Verify the attestation quote
        let time_stamp_seconds = Self::current_time_seconds();
        match participant_attestation.verified_attestation.re_verify(
            time_stamp_seconds,
            &allowed_mpc_docker_image_hashes,
            &allowed_launcher_compose_hashes,
            &allowed_measurements,
        ) {
            Ok(()) => TeeQuoteStatus::Valid,
            Err(err) => TeeQuoteStatus::Invalid(err.to_string()),
        }
    }

    pub(crate) fn is_caller_an_attested_participant(
        &self,
        participants: &Participants,
    ) -> Result<(), AttestationCheckError> {
        let signer_account_pk = env::signer_account_pk();
        let signer_id = env::signer_account_id();

        let info = participants
            .info(&signer_id)
            .ok_or(AttestationCheckError::CallerNotParticipant)?;

        let attestation = self
            .stored_attestations
            .get(&info.tls_public_key)
            .ok_or(AttestationCheckError::AttestationNotFound)?;

        if attestation.node_id.account_id != signer_id {
            return Err(AttestationCheckError::AttestationOwnerMismatch);
        }

        // Stored account keys are Ed25519 by construction; a non-Ed25519
        // signer necessarily mismatches.
        let signer_ed25519 = Ed25519PublicKey::try_from(&signer_account_pk)
            .map_err(|_| AttestationCheckError::AttestationKeyMismatch)?;
        if attestation.node_id.account_public_key != signer_ed25519 {
            return Err(AttestationCheckError::AttestationKeyMismatch);
        }

        Ok(())
    }
