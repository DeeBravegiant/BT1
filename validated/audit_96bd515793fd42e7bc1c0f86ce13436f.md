Audit Report

## Title
Unset `minRsEthAmountToWithdraw` Allows Dust Spam to Bloat Queue, Delaying Legitimate Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary

`minRsEthAmountToWithdraw[asset]` defaults to `0` for every asset because `initialize` never assigns it. The guard in `initiateWithdrawal` only blocks a strictly-zero amount, so any value ≥ 1 wei passes. An attacker holding rsETH can spam the global withdrawal queue with thousands of dust requests, forcing the operator to iterate through every spam entry before `nextLockedNonce` reaches a legitimate user's entry, temporarily freezing their funds and consuming unbounded gas across multiple `unlockQueue` calls.

## Finding Description

**Root cause — uninitialized minimum**

`initialize` sets only `withdrawalDelayBlocks` and `lrtConfig`; `minRsEthAmountToWithdraw` is never assigned:

```solidity
// contracts/LRTWithdrawalManager.sol L90-98
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    __Pausable_init();
    __ReentrancyGuard_init();
    withdrawalDelayBlocks = 8 days / 12 seconds;
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
```

The mapping therefore returns `0` for every asset. The guard in `initiateWithdrawal`:

```solidity
// L162-164
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

When `minRsEthAmountToWithdraw[asset] == 0`, the second condition `rsETHUnstaked < 0` is always false for `uint256`, so any `rsETHUnstaked >= 1` is accepted.

**Queue bloat — sequential, non-skippable processing**

Every accepted request is appended to the global FIFO queue via `_addUserWithdrawalRequest`, incrementing `nextUnusedNonce[asset]`. `_unlockWithdrawalRequests` advances `nextLockedNonce` strictly in order; the operator cannot skip entries:

```solidity
// L790-815
while (nextLockedNonce_ < firstExcludedIndex) {
    bytes32 requestId = getRequestId(asset, nextLockedNonce_);
    WithdrawalRequest storage request = withdrawalRequests[requestId];
    if (block.number < request.withdrawalStartBlock + withdrawalDelayBlocks) break;
    uint256 payoutAmount = _calculatePayoutAmount(request, rsETHPrice, assetPrice);
    if (availableAssetAmount < payoutAmount) break;
    assetsCommitted[asset] -= request.expectedAssetAmount;
    request.expectedAssetAmount = payoutAmount;
    rsETHAmountToBurn += request.rsETHUnstaked;
    availableAssetAmount -= payoutAmount;
    assetAmountToUnlock += payoutAmount;
    unlockedWithdrawalsCount[asset]++;
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
```

A victim's `completeWithdrawal` reverts with `WithdrawalLocked` until `usersFirstWithdrawalRequestNonce < nextLockedNonce[asset]`. With K spam entries inserted before the victim's nonce, the operator must advance `nextLockedNonce` through all K entries. Each entry costs approximately two cold storage reads (`getRequestId` + `withdrawalRequests`), arithmetic in `_calculatePayoutAmount`, and two storage writes (`assetsCommitted`, `request.expectedAssetAmount`). At ~44,700 gas per entry, 10,000 spam entries require ~447M gas — roughly 15× the Ethereum block gas limit — forcing the operator into at least 15 separate `unlockQueue` calls before the victim's entry is reached.

`setMinRsEthAmountToWithdraw` exists but is admin-only and is never called during deployment, leaving the door open until an admin explicitly sets it post-deployment.

## Impact Explanation

**Temporary freezing of funds (Medium).** A victim who called `initiateWithdrawal` cannot call `completeWithdrawal` until `nextLockedNonce` advances past their nonce. With K spam entries ahead, the victim's withdrawal is frozen for however many `unlockQueue` batches are needed to drain K entries. The delay is attacker-controlled and unbounded.

**Unbounded gas consumption (Medium).** Each `unlockQueue` call traversing spam entries burns gas proportional to the batch size. With enough spam, no single call can process the entire backlog within the block gas limit, forcing the operator into O(K) calls. Both impacts are in the allowed scope.

## Likelihood Explanation

The attacker must hold rsETH and pay L1 gas per request. However, the underlying asset is returned when each dust entry is eventually processed, so the net cost is gas only. A contract loop can submit thousands of requests in a single block. `minRsEthAmountToWithdraw` is not set in `initialize` and there is no on-chain enforcement requiring the admin to set it before the contract is used, meaning the window of vulnerability exists from deployment until an admin explicitly calls `setMinRsEthAmountToWithdraw`. Likelihood is **Medium**.

## Recommendation

1. Set a sensible non-zero default inside `initialize`, e.g.:
   ```solidity
   minRsEthAmountToWithdraw[LRTConstants.ETH_TOKEN] = 0.001 ether;
   ```
2. In `setMinRsEthAmountToWithdraw`, enforce `minRsEthAmountToWithdraw_ > 0` to prevent accidental reset to zero.
3. Optionally, add a per-user cap on pending withdrawal requests per asset to bound queue growth from a single address.

## Proof of Concept

```solidity
// Attacker holds rsETH; minRsEthAmountToWithdraw[ETH] == 0 (default)
for (uint256 i = 0; i < 10_000; i++) {
    withdrawalManager.initiateWithdrawal(LRTConstants.ETH_TOKEN, 1, "");
}

// Victim submits a legitimate withdrawal — nonce is now 10_000
withdrawalManager.initiateWithdrawal(LRTConstants.ETH_TOKEN, 1 ether, "");

// Operator must call unlockQueue ~15 times (gas-bounded) to advance
// nextLockedNonce through all 10,000 spam entries before reaching nonce 10,000.
// Until then, victim's completeWithdrawal reverts with WithdrawalLocked.
// Attacker recovers ~10,000 wei of ETH; net cost is gas only.
```

A Foundry fork test can confirm this by: (1) deploying the contract without calling `setMinRsEthAmountToWithdraw`, (2) submitting 10,000 one-wei withdrawal requests from an attacker address, (3) submitting a 1 ETH withdrawal from a victim address, (4) fast-forwarding past `withdrawalDelayBlocks`, (5) asserting that `completeWithdrawal` reverts with `WithdrawalLocked` for the victim before the operator has processed all spam entries, and (6) asserting it succeeds only after all spam entries are drained.
