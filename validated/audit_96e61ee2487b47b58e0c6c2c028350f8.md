### Title
Blacklisted ERC20 Recipient Permanently Locks Bridged Funds — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol`'s `finTransfer` function unconditionally pushes tokens to `payload.recipient` with no fallback. When the destination token is a native ERC20 with a transfer blacklist (USDC, USDT), and the recipient address is blacklisted, every `finTransfer` call reverts. Because the recipient is cryptographically bound in the MPC-signed payload and there is no on-chain cancel or redirect mechanism, the user's funds are permanently locked on NEAR with no recovery path.

---

### Finding Description

`finTransfer` in `OmniBridge.sol` is the EVM-side finalization entry point for NEAR→EVM transfers. After verifying the MPC signature, it unconditionally transfers tokens to `payload.recipient`:

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
``` [1](#0-0) 

The `payload.recipient` is an EVM `address` field embedded in the Borsh-encoded, MPC-signed `TransferMessagePayload`:

```solidity
struct TransferMessagePayload {
    ...
    address recipient;
    ...
}
``` [2](#0-1) 

The signature covers the recipient field directly:

```solidity
Borsh.encodeAddress(payload.recipient),
``` [3](#0-2) 

This means the recipient is immutably bound to the signed payload. No alternative recipient can be substituted without a new MPC signature.

On the NEAR side, when a user initiates a NEAR→EVM transfer via `ft_on_transfer` → `init_transfer`, the tokens are locked in `pending_transfers` and burned/locked: [4](#0-3) 

There is no public `cancel_transfer` or refund function in the NEAR contract. The transfer remains in `pending_transfers` indefinitely: [5](#0-4) 

**Attack / grief path:**

1. User initiates a NEAR→EVM transfer of USDC (or any blacklist-capable ERC20). Tokens are locked on NEAR.
2. The user's EVM recipient address is blacklisted by the token issuer (before or after initiation — USDC can blacklist any address at any time).
3. A relayer calls `finTransfer` with the MPC-signed payload.
4. `safeTransfer(payload.recipient, payload.amount)` reverts because USDC rejects transfers to blacklisted addresses.
5. The entire transaction reverts. The nonce is not consumed.
6. Every subsequent `finTransfer` attempt for this transfer also reverts — the recipient cannot be changed.
7. The funds remain locked on NEAR in `pending_transfers` with no recovery path.

---

### Impact Explanation

This is a **Critical** irreversible fund lock. The user's tokens are locked on NEAR and can never be released to EVM. There is no cancel, redirect, or admin-rescue function in the NEAR contract. The only theoretical recovery would be a DAO-authorized contract upgrade — a privileged, non-standard intervention that is not guaranteed and may not be timely.

---

### Likelihood Explanation

USDC and USDT (both in-scope tokens) maintain active blacklists. USDC's blacklist is enforced at the token level and can be applied to any address at any time, including after a bridge transfer is initiated. A user whose address is blacklisted between initiation and finalization — or who was already blacklisted — triggers this permanently. This is a realistic, non-contrived scenario.

---

### Recommendation

Implement a pull-based withdrawal pattern instead of pushing tokens directly to the recipient in `finTransfer`. If the direct transfer fails, record the claimable balance on-chain and allow the recipient (or an alternative address they designate) to withdraw later. Alternatively, add a NEAR-side `cancel_transfer` function that allows a user to reclaim locked funds if finalization has not occurred within a timeout window, so that a permanently-failing EVM finalization does not result in permanent loss.

---

### Proof of Concept

1. User holds USDC on NEAR and calls `ft_transfer_call` to the NEAR bridge with `InitTransferMsg { recipient: OmniAddress::Eth(user_evm_addr), ... }`.
2. NEAR bridge locks USDC in `pending_transfers` and emits `InitTransferEvent`.
3. USDC issuer blacklists `user_evm_addr` on the EVM chain.
4. Relayer calls `OmniBridge.finTransfer(signatureData, payload)` where `payload.tokenAddress = USDC` and `payload.recipient = user_evm_addr`.
5. Execution reaches:
   ```solidity
   IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
   ``` [6](#0-5) 
6. USDC's `transfer` reverts with a blacklist error. `SafeERC20` propagates the revert.
7. The entire `finTransfer` transaction reverts. `completedTransfers[payload.destinationNonce]` is not persisted.
8. No alternative recipient can be used — the MPC signature binds `payload.recipient` immutably. [7](#0-6) 
9. NEAR-side `pending_transfers` retains the locked USDC with no cancel path. [8](#0-7)
