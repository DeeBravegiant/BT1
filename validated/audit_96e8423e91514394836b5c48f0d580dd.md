### Title
Silent Dropping of Multi-Action NEAR Receipts Causes Permanent Loss of Sign Requests - (File: `crates/node/src/indexer/handler.rs`, `crates/chain-gateway/src/event_subscriber/streamer/block_processor.rs`)

### Summary

Both the production MPC-node indexer and the newer chain-gateway block processor contain a `try_extract_function_call_args` helper that unconditionally returns `None` — silently discarding the entire receipt — whenever a NEAR receipt carries more than one action. Any `sign`, `request_app_private_key`, or `verify_foreign_transaction` call that arrives inside such a receipt is never indexed by the MPC nodes, so the on-chain pending request is never fulfilled and the caller's deposit is locked until the yield times out.

### Finding Description

In NEAR, a single transaction (and therefore a single receipt) may legally contain multiple actions. The `try_extract_function_call_args` function in both indexer paths enforces a hard `actions.len() != 1` guard:

```rust
// crates/node/src/indexer/handler.rs  (lines 281-300)
// crates/chain-gateway/src/event_subscriber/streamer/block_processor.rs (lines 191-210)
fn try_extract_function_call_args(receipt: &ReceiptView) -> Option<(&FunctionArgs, &String)> {
    let ReceiptEnumView::Action { ref actions, .. } = receipt.receipt else {
        return None;
    };
    if actions.len() != 1 {   // ← silently drops the entire receipt
        return None;
    }
    let ActionView::FunctionCall { ref method_name, ref args, .. } = actions[0] else {
        return None;
    };
    Some((args, method_name))
}
```

When this function returns `None`, the caller (`process_receipt` / `BlockEvents::process_receipt`) returns immediately without emitting any `MatchedEvent`. The receipt is never forwarded to the MPC node's signing pipeline.

A receipt with multiple actions can be produced in at least two realistic ways:

1. **User-submitted multi-action transaction** — A NEAR transaction may bundle a `Transfer` action (to satisfy the 1 yoctoNEAR deposit requirement) together with a `FunctionCall` action targeting `sign`. The contract executes the `FunctionCall` successfully, stores the yield-based pending request, but the indexer silently discards the receipt.

2. **Smart-contract batch receipt** — Any contract that calls `env::promise_batch_action_transfer` followed by `env::promise_batch_action_function_call` on the MPC contract produces a receipt with two actions. The MPC contract processes the `sign` call and registers the pending request; the indexer drops the receipt.

In both cases the on-chain state is consistent (the pending request exists), but the off-chain MPC nodes never learn about it. No node ever initiates the signing protocol, so the yield eventually times out and the request is cleaned up without producing a signature.

The same `actions.len() != 1` guard is duplicated verbatim in the newer `chain-gateway` crate, so the bug is present in both the current production path and the planned replacement.

### Impact Explanation

- **Request-lifecycle invariant broken**: every successfully enqueued `sign` / `request_app_private_key` / `verify_foreign_transaction` call should eventually receive a response from the MPC network. Receipts with multiple actions violate this invariant silently — no error is surfaced to the caller.
- **Deposit locked / signature never delivered**: the caller's 1 yoctoNEAR deposit is held by the contract until the NEAR yield timeout fires. More critically, any downstream operation that depends on the signature (e.g., signing a Bitcoin or Ethereum transaction) is permanently blocked for that request.
- **No retry signal**: because the indexer emits no warning and the contract shows the request as "pending", the caller has no indication that the request was dropped. They must independently discover the failure and resubmit.

This matches the **Medium** allowed impact: *"request-lifecycle manipulation that breaks production safety/accounting invariants"*.

### Likelihood Explanation

- Any smart contract that wraps MPC `sign` calls and uses NEAR's batch-action API (e.g., attaching a transfer alongside the function call) will silently lose every request it submits.
- The `parallel_sign_calls` devnet helper already demonstrates the pattern of programmatic multi-call batching; a production integration contract could easily produce multi-action receipts.
- No privileged access, threshold collusion, or TEE bypass is required — an ordinary unprivileged NEAR account or contract is sufficient.

### Recommendation

Replace the hard `actions.len() != 1` guard with iteration over all actions in the receipt, processing each `FunctionCall` action independently:

```rust
fn try_extract_function_call_args(receipt: &ReceiptView)
    -> impl Iterator<Item = (&FunctionArgs, &String)>
{
    let ReceiptEnumView::Action { ref actions, .. } = receipt.receipt else {
        return Either::Left(std::iter::empty());
    };
    Either::Right(actions.iter().filter_map(|action| {
        if let ActionView::FunctionCall { ref method_name, ref args, .. } = action {
            Some((args, method_name))
        } else {
            None
        }
    }))
}
```

`process_receipt` should then call this iterator and emit one `MatchedEvent` per matching `FunctionCall` action, mirroring the fix recommended in the ZetaChain report (remove the early-exit, process all events).

### Proof of Concept

1. Deploy a helper contract `BatchCaller` on NEAR testnet.
2. `BatchCaller::call_sign` creates a batch receipt targeting the MPC contract:
   ```rust
   let p = env::promise_batch_create(&mpc_contract);
   env::promise_batch_action_transfer(p, 1);          // action[0]
   env::promise_batch_action_function_call(           // action[1]
       p, "sign", sign_args, 1 /*deposit*/, gas);
   ```
3. The MPC contract executes the `sign` call, stores the pending request, and emits `SuccessReceiptId`.
4. The MPC-node indexer calls `try_extract_function_call_args` on the receipt, sees `actions.len() == 2`, returns `None`, and emits no `MatchedEvent`.
5. No MPC node ever starts the signing protocol. After the NEAR yield timeout the request is cleaned up with no signature produced.
6. Repeat with a single-action receipt for the same payload — the signature is produced normally — confirming the root cause. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
