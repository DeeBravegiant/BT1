### Title
Wrong contract-type variable in `action_delete_account` overstates storage usage for global-contract accounts — (`runtime/runtime/src/actions.rs`)

---

### Summary

The legacy code path in `action_delete_account` reads only the **local** contract hash to compute the bytes to subtract from the deletion-size check. For accounts whose contract is `AccountContract::Global` or `AccountContract::GlobalByAccount`, `local_contract_hash()` returns `None`, so zero bytes are subtracted. The global-contract identifier overhead (32 bytes for a hash-based identifier, or the account-ID length for an account-based identifier) is never removed, inflating the computed storage usage and causing the deletion guard to incorrectly reject valid `DeleteAccount` actions.

---

### Finding Description

In `action_delete_account`, the pre-fix branch computes `account_storage_usage` as:

```rust
let code_len = get_code_len_or_default(
    state_update,
    account_id.clone(),
    account_ref.local_contract_hash().unwrap_or_default(), // ← wrong for global contracts
)?;
account_storage_usage.saturating_sub(code_len)
``` [1](#0-0) 

For an account whose contract field is `AccountContract::Global(hash)` or `AccountContract::GlobalByAccount(id)`, `local_contract_hash()` returns `None`: [2](#0-1) 

`unwrap_or_default()` therefore yields `CryptoHash::default()`, and `get_code_len_or_default` returns `0` because no local code entry exists in the trie for that hash. The global-contract identifier storage (32 bytes for `Global`, or the referenced account-ID length for `GlobalByAccount`) is never subtracted. The resulting `account_storage_usage` is inflated by exactly that identifier size, and the guard:

```rust
if account_storage_usage > Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE { … }
``` [3](#0-2) 

incorrectly fires, returning `DeleteAccountWithLargeState` for an account whose actual non-contract storage is within the 10,000-byte limit.

The correct approach — used by the `FixDeleteAccountGlobalContractStorageUsage` branch — is `get_contract_storage_usage`, which dispatches on the actual contract variant: [4](#0-3) 

The protocol feature and its description confirm this is a known wrong-variable bug: [5](#0-4) 

---

### Impact Explanation

An account that has called `UseGlobalContract` and whose total `storage_usage` falls in the range `(MAX_ACCOUNT_DELETION_STORAGE_USAGE, MAX_ACCOUNT_DELETION_STORAGE_USAGE + identifier_size]` (e.g., 10,001–10,032 bytes for a 32-byte hash identifier) receives a `DeleteAccountWithLargeState` action error even though its non-contract storage is at or below the 10,000-byte limit. The account cannot be deleted, its balance cannot be recovered to the beneficiary, and the trie state is not updated. The corrupted protocol value is the **action execution outcome** stored in the outcome trie — a success that should be committed is replaced by a failure.

The test suite explicitly documents this before/after behaviour: [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged user who:
1. Calls `UseGlobalContract` on their account (setting `AccountContract::Global` or `AccountContract::GlobalByAccount`), and
2. Has accumulated storage usage in the narrow range `(10000, 10000 + identifier_size]` bytes,

will hit this bug when submitting a `DeleteAccount` action. The trigger is a standard signed transaction requiring no special privileges. The affected range is narrow (32 bytes for hash-based identifiers) but deterministically reachable.

---

### Recommendation

Replace the local-hash lookup with `get_contract_storage_usage`, which correctly handles all three contract variants (`None`, `Local`, `Global`, `GlobalByAccount`):

```rust
// Before (wrong for global contracts):
let code_len = get_code_len_or_default(
    state_update,
    account_id.clone(),
    account_ref.local_contract_hash().unwrap_or_default(),
)?;
let account_storage_usage = account_storage_usage.saturating_sub(code_len);

// After (correct):
let contract_storage = get_contract_storage_usage(state_update, account_id, account_ref)?;
let account_storage_usage = account_ref.storage_usage().saturating_sub(contract_storage);
```

This is exactly what the `FixDeleteAccountGlobalContractStorageUsage` protocol feature implements.

---

### Proof of Concept

1. Deploy a global contract (any code) via `DeployGlobalContractAction`.
2. Create account `alice` and call `UseGlobalContract` with a hash-based identifier, setting `AccountContract::Global(hash)` and `storage_usage = MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32` (10,032 bytes).
3. Submit a `DeleteAccount` action for `alice` on a node running a protocol version before `FixDeleteAccountGlobalContractStorageUsage`.
4. The runtime enters the legacy branch: `local_contract_hash()` → `None` → `code_len = 0`.
5. `account_storage_usage = 10032 − 0 = 10032 > 10000` → `DeleteAccountWithLargeState` error.
6. The account is not deleted; the beneficiary receives nothing; the outcome trie records a failure where a success is correct.

The actual non-contract storage is `10032 − 32 = 10000 = MAX_ACCOUNT_DELETION_STORAGE_USAGE`, which satisfies the strict `>` guard and should have succeeded.

fn get_contract_storage_usage(
    state_update: &TrieUpdate,
    account_id: &AccountId,
    account: &Account,
) -> Result<StorageUsage, StorageError> {
    Ok(match account.contract().as_ref() {
        AccountContract::None => 0,
        AccountContract::Local(code_hash) => {
            get_code_len_or_default(state_update, account_id.clone(), *code_hash)?
        }
        AccountContract::Global(_) | AccountContract::GlobalByAccount(_) => {
            account.contract().identifier_storage_usage()
        }
    })
}
