Audit Report

## Title
Rewards Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.updateReward` unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`, while `rewardPerToken()` silently skips accrual during that same zero-stake window. Any `rewardRate`-denominated tokens emitted during the gap are permanently unaccounted for and locked in the contract with no recovery path.

## Finding Description
Two code paths interact to create the loss:

**`rewardPerToken()` (L408-414):** When `totalKernelStaked == 0`, the function returns `rewardPerTokenStored` unchanged — no accrual occurs for the elapsed time. [1](#0-0) 

**`updateReward` modifier (L232-242):** `updatedAt = lastTimeRewardApplicable()` executes unconditionally on every call, regardless of whether `totalKernelStaked` is zero. This permanently discards the time window during which no accrual happened. [2](#0-1) 

**Exploit path:**
1. Admin calls `notifyRewardAmount` while Alice is staked — passes the `totalKernelStaked == 0` guard at L570.
2. Alice (sole staker) calls `initiateWithdrawal(all)` — `totalKernelStaked` drops to 0 immediately at L326; `updatedAt` advances to T=10.
3. 20 seconds pass. `rewardRate × 20` tokens are emitted but never credited to anyone.
4. Bob calls `stake(X)` — `updateReward` fires: `rewardPerToken()` returns the old stored value (no accrual), then `updatedAt` jumps to T=30.
5. Bob later claims — earns only rewards from T=30 onward. The T=10→T=30 rewards are permanently stuck. [3](#0-2) 

**Existing guards are insufficient:** The `notifyRewardAmount` guard at L570 only prevents *starting* a new reward period with zero stakers; it cannot prevent all stakers from exiting during an already-active period. The contract's own NatSpec (L17-22) acknowledges this exact issue but relies entirely on an off-chain operational assumption with no code-level enforcement. [4](#0-3) [5](#0-4) 

There is no `rescueTokens`, no `sweep`, and no admin function to recover the stuck reward tokens.

## Impact Explanation
Reward tokens transferred into the contract via `notifyRewardAmount` and emitted during any zero-stake gap are permanently frozen in the contract. The lost amount scales with `rewardRate × gap_duration`. With `MAX_WITHDRAWAL_DELAY = 30 days`, the gap window can be up to 30 days, making the frozen amount potentially equal to the entire reward allocation. This maps to **Medium — Permanent freezing of unclaimed yield**. [6](#0-5) 

## Likelihood Explanation
The scenario is reachable by any single unprivileged staker who is the sole depositor — realistic early in the protocol lifecycle or after a mass exit event. No attacker capability beyond normal `initiateWithdrawal` is required. The mandatory `withdrawalDelay` (up to 30 days) structurally guarantees a large gap window. The condition is repeatable across reward periods.

## Recommendation
Do not advance `updatedAt` when `totalKernelStaked == 0`. Modify the `updateReward` modifier to conditionally update `updatedAt` only when supply is nonzero:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This freezes the reward clock during zero-stake gaps so that when the next staker enters, the gap rewards are replayed and distributed correctly.

## Proof of Concept

```solidity
// Setup: Alice is sole staker, admin starts reward period
kernelDepositPool.setRewardsDuration(30 days);
kernelDepositPool.notifyRewardAmount(30_000e18); // rewardRate ≈ 1000e18/day

// Step 1: Alice initiates full withdrawal → totalKernelStaked = 0
vm.prank(alice);
kernelDepositPool.initiateWithdrawal(aliceStake);
// updatedAt is now T=0, totalKernelStaked = 0

// Step 2: 10 days pass — 10_000e18 tokens emitted, zero credited
vm.warp(block.timestamp + 10 days);

// Step 3: Bob stakes — updateReward fires, rewardPerToken() returns stored value,
//         updatedAt jumps to T+10days, gap rewards permanently lost
vm.prank(bob);
kernelDepositPool.stake(1e18);

// Step 4: Another 10 days pass
vm.warp(block.timestamp + 10 days);

// Step 5: Bob claims — earns only ~10_000e18 (T+10d to T+20d), not ~20_000e18
vm.prank(bob);
kernelDepositPool.getReward();

// Assert: ~10_000e18 permanently stuck in contract
assertGt(rewardsToken.balanceOf(address(kernelDepositPool)), 9_000e18);
```
