Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Mint Calculation Allows New Depositors to Steal Accrued Yield from Existing Holders — (`contracts/LRTDepositPool.sol`)

## Summary

`LRTDepositPool.getRsETHAmountToMint()` divides by `lrtOracle.rsETHPrice()`, a stored state variable that is only written inside `LRTOracle._updateRsETHPrice()`. Neither `depositETH` nor `depositAsset` triggers a price update before computing the mint amount. As staking yield accrues between oracle updates, the stored price falls below the true price, causing new depositors to receive more rsETH than they are entitled to and directly diluting existing holders' accrued yield.

## Finding Description

`depositETH` (L76–93) and `depositAsset` (L99–118) both call `_beforeDeposit`, which calls `getRsETHAmountToMint`. Neither function invokes `updateRSETHPrice()` or any equivalent price-sync before computing the mint amount.

`getRsETHAmountToMint` at L520 computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` fetches a live price from an external oracle, while `lrtOracle.rsETHPrice()` returns the last stored value. The stored value is only written at L313 of `LRTOracle.sol`:

```solidity
rsETHPrice = newRsETHPrice;
```

inside `_updateRsETHPrice()`, which is invoked only by the permissionless `updateRSETHPrice()` (L87–89) or the manager-only `updateRSETHPriceAsManager()` (L94–96).

Between two consecutive `updateRSETHPrice()` calls, EigenLayer pod shares and LST rebasing continuously increase `totalETHInProtocol` (read live at L341–343 via `getTotalAssetDeposits`), but `rsETHPrice` does not move. The denominator in the mint formula is therefore stale-low, and new depositors receive excess rsETH proportional to the accumulated yield gap.

The `pricePercentageLimit` guard (L256–265) does not prevent this: it only reverts a *price-update call* if the increase exceeds the daily threshold for non-managers. It does not block deposits, does not enforce a fresh price before minting, and in fact can worsen staleness by preventing permissionless updates when yield accumulation is large.

**Exploit path:**
1. Observe on-chain that `totalETHInProtocol` (computable from public view functions) exceeds `rsethSupply × rsETHPrice`, indicating a stale-low stored price.
2. Call `depositETH` or `depositAsset` with a large amount. The stale denominator mints excess rsETH.
3. Wait for a manager or keeper to call `updateRSETHPrice()`. The price rises to reflect the true TVL, but the supply is now inflated by the attacker's excess rsETH.
4. All existing holders' rsETH is worth less ETH per token than they were owed; the attacker captured the difference.

No privileged role is required for steps 1–2. The attacker can also deliberately refrain from calling `updateRSETHPrice()` to widen the staleness window before depositing.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders continuously earn staking yield reflected in a rising true TVL. When a new depositor mints at a stale-low price, the subsequent price update distributes the TVL across a larger-than-correct rsETH supply, permanently reducing the per-token value for all prior holders. The stolen amount scales with deposit size and staleness duration. This is a direct, concrete, on-chain transfer of accrued yield from existing holders to the attacker, matching the "Theft of unclaimed yield — High" impact class exactly.

## Likelihood Explanation

**Medium.** `updateRSETHPrice()` is permissionless and not called atomically with every deposit, so a non-zero staleness window always exists. An attacker needs no privileged access: they only need to observe the public on-chain state (TVL vs. stored price) and submit a deposit transaction. The attack is repeatable across every oracle update cycle and requires no victim interaction or external protocol compromise.

## Recommendation

Call `updateRSETHPrice()` (or an internal equivalent that refreshes `rsETHPrice` without the daily-limit side-effects) at the start of `_beforeDeposit`, before `getRsETHAmountToMint` is invoked. This ensures the denominator always reflects the current protocol TVL at the time of minting, eliminating the staleness window entirely. Alternatively, compute the mint amount directly from a freshly calculated price inside `getRsETHAmountToMint` rather than reading the stored state variable.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// 1. Fork mainnet at block N; record Alice's rsETH balance and rsETHPrice.
// 2. Roll forward ~1 day (vm.roll / vm.warp) so EigenLayer pod shares accrue yield.
//    Confirm lrtOracle.rsETHPrice() is unchanged (no updateRSETHPrice called).
// 3. Bob calls depositETH{value: 100 ether}().
//    Record rsethMinted = rsETH.balanceOf(Bob) - 0.
//    Compute expectedMint = 100e18 * getAssetPrice(ETH) / freshPrice
//    where freshPrice = _getTotalEthInProtocol() / rsETH.totalSupply() (pre-deposit).
//    Assert rsethMinted > expectedMint  ← excess rsETH minted.
// 4. Call updateRSETHPrice().
//    Record newPrice = lrtOracle.rsETHPrice().
//    Assert Alice's effective ETH value (aliceRsETH * newPrice) < pre-deposit TVL share.
//    Assert Bob's effective ETH value (bobRsETH * newPrice) > 100 ether deposited.
// 5. Difference = yield stolen from Alice by Bob.
```

This test requires no privileged keys, no governance actions, and no external oracle manipulation — only a time warp and a public deposit call.
