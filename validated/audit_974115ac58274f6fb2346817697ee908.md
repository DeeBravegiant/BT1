### Title
WalletContract `address_check_callback` Skips Nonce Increment on Registrar Failure, Enabling Replay of User-Signed Ethereum Transactions - (File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs)

### Summary
The `WalletContract` (the Ethereum-compatible wallet deployed for ETH-implicit accounts on NEAR) intentionally defers nonce increment to `address_check_callback` for `EOABaseTokenTransfer` transactions that require an address-registrar lookup. However, two early-return paths inside `address_check_callback` return without ever incrementing the nonce, leaving the user's signed Ethereum transaction permanently replayable by any unprivileged caller.

### Finding Description
In `inner_rlp_execute`, the nonce is explicitly **not** incremented for the `EOABaseTokenTransfer { address_check: Some(_) }` branch:

```rust
if let TransactionKind::EthEmulation(EthEmulationKind::EOABaseTokenTransfer {
    address_check: Some(_),
    ..
}) = &transaction_kind
{
} else {
    *nonce = nonce.saturating_add(1);
}
``` [1](#0-0) 

The code comment at line 175 explicitly acknowledges that the nonce **must** be incremented inside `address_check_callback` to prevent replay:

```rust
// We must increment the nonce at this point to prevent replay of the transaction.
// Recall that the nonce was not incremented in `inner_rlp_execute` in the case that
// the registrar contract was called (i.e. in the case we end up inside this callback).
self.nonce = self.nonce.saturating_add(1);
``` [2](#0-1) 

However, `address_check_callback` has two early-return paths that execute **before** the nonce increment at line 178:

**Path 1 – Registrar call fails (`PromiseResult::Failed`):**
```rust
PromiseResult::Failed => {
    return PromiseOrValue::Value(ExecuteResponse {
        success: false,
        success_value: None,
        error: Some("Call to Address Registrar contract failed".into()),
    });
}
``` [3](#0-2) 

**Path 2 – Registrar returns unparseable data:**
```rust
Err(_) => {
    return PromiseOrValue::Value(ExecuteResponse {
        success: false,
        success_value: None,
        error: Some("Unexpected response from account registrar".into()),
    });
}
``` [4](#0-3) 

In both cases, `has_in_flight_tx` is reset to `false` at line 140 (before the match), and `self.nonce` remains at its pre-transaction value. [5](#0-4) 

After either failure, the `WalletContract` state is:
- `has_in_flight_tx = false` → accepts new calls
- `nonce = N` (unchanged) → the same Ethereum transaction with nonce `N` passes the nonce check again

Any unprivileged caller can then invoke `rlp_execute` with the original RLP-encoded, user-signed Ethereum transaction bytes and the same `target`, and the contract will accept and execute it. [6](#0-5) 

The `nep_141_storage_balance_callback` path does **not** share this bug because the nonce is incremented in `inner_rlp_execute` before the ERC-20 callback is scheduled. [7](#0-6) 

### Impact Explanation
An attacker who observes a failed `rlp_execute` call (transaction data is public on-chain) for an `EOABaseTokenTransfer` where the registrar call failed can replay the identical user-signed Ethereum transaction at any future time. If the user's NEAR balance has increased since the original failure (e.g., they received a deposit), the replayed transaction succeeds and transfers the user's funds to the original target without the user's current consent. The corrupted protocol value is `WalletContract.nonce` (the on-chain replay-prevention counter stored in the ETH-implicit account's contract state), which fails to advance, and the corrupted outcome is an unauthorized NEAR token transfer from the user's ETH-implicit account.

### Likelihood Explanation
The registrar call can fail due to:
1. The relayer providing insufficient gas for the registrar cross-contract call.
2. A transient panic or unavailability of the registrar contract.

Both are realistic in production. Once a failure occurs, the signed Ethereum transaction bytes are permanently on-chain and replayable by any party. The attacker only needs to wait for the user's balance to be sufficient and then submit a NEAR transaction calling `rlp_execute` with the original bytes.

### Recommendation
Increment `self.nonce` in both early-return failure paths inside `address_check_callback`, mirroring the explicit increment at line 178. Specifically:

```rust
PromiseResult::Failed => {
    self.nonce = self.nonce.saturating_add(1); // prevent replay
    return PromiseOrValue::Value(ExecuteResponse { ... });
}
```

and similarly for the JSON deserialization failure branch. This matches the Ethereum protocol requirement (documented in `get_nonce`'s comment at lines 62–66) that the nonce increments on every submitted transaction regardless of outcome. [8](#0-7) 

### Proof of Concept
1. Alice holds 500 NEAR in her ETH-implicit account (`WalletContract.nonce = 5`).
2. Alice signs an Ethereum transaction (nonce=5) to transfer 500 NEAR to Bob's ETH-implicit address.
3. The relayer submits `rlp_execute`. The registrar cross-contract call runs out of gas → `PromiseResult::Failed`. `address_check_callback` returns early. `WalletContract.nonce` stays at 5.
4. Alice receives 600 NEAR from a third party. Her balance is now 600 NEAR.
5. An attacker (who observed step 3 on-chain) calls `rlp_execute` with the same RLP bytes and `target`. `has_in_flight_tx` is false, nonce is still 5, signature is valid → the transfer executes, draining 500 NEAR from Alice without her current consent.

    pub fn rlp_execute(
        &mut self,
        target: AccountId,
        tx_bytes_b64: String,
    ) -> PromiseOrValue<ExecuteResponse> {
        // To ensure user actions are executed in the desired order,
        // having multiple transactions in flight at the same time is
        // not allowed.
        if self.has_in_flight_tx {
            return PromiseOrValue::Value(ExecuteResponse {
                success: false,
                success_value: None,
                error: Some(
                    "Error: transaction already in progress, please try again later.".into(),
                ),
            });
        }
        let current_account_id = env::current_account_id();
        let predecessor_account_id = env::predecessor_account_id();
        let result = inner_rlp_execute(
            current_account_id.clone(),
            predecessor_account_id,
            target,
            tx_bytes_b64,
            &mut self.nonce,
        );

        match result {
            Ok(promise) => {
                self.has_in_flight_tx = true;
                PromiseOrValue::Promise(promise)
            }
            Err(Error::Relayer(_)) if env::signer_account_id() == current_account_id => {
                let promise = create_ban_relayer_promise(current_account_id);
                self.has_in_flight_tx = true;
                PromiseOrValue::Promise(promise)
            }
            Err(e) => PromiseOrValue::Value(e.into()),
        }
    }
