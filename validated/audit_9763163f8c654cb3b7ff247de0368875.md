### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Key Enables Queue Saturation DoS on Foreign Transaction Verification — (`crates/contract/src/pending_requests.rs`, `crates/contract/src/lib.rs`)

### Summary

The `verify_foreign_transaction` endpoint uses a request key (`VerifyForeignTransactionRequest`) that does **not** include the caller's account ID, unlike `sign` and `request_app_private_key` which both embed the predecessor account. Any unprivileged actor can observe a target foreign-chain transaction ID from public chain data, submit 128 identical `verify_foreign_transaction` calls to saturate the per-key fan-out queue, and permanently block legitimate users from getting that specific foreign transaction verified — for as long as the attacker continuously refills the queue after each timeout cycle. For bridge operations this translates to an indefinite freeze of the corresponding inbound bridge deposit.

### Finding Description

**Root cause — caller-agnostic request key**

`SignatureRequest` is constructed with the predecessor account ID baked in:

```rust
// crates/contract/src/lib.rs  (sign path)
let predecessor = env::predecessor_account_id();
SignatureRequest::new(domain_id, payload, &predecessor, &path)
``` [1](#0-0) 

`VerifyForeignTransactionRequest` carries no caller field at all:

```rust

    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [2](#0-1) 

The contract's own test acknowledges this explicitly:

``` [3](#0-2) 

**Queue cap enforcement — the choke point**

`push_pending_yield` enforces a hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` entries per request key. Once the cap is reached, every subsequent submission panics with `PendingRequestQueueFull`:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

pub(crate) fn push_pending_yield<K>(...) {
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(&RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string());
    }
    queue.push(YieldIndex { data_id });
}
``` [4](#0-3) 

**Attack flow**

1. Attacker observes a target Bitcoin/Ethereum `tx_id` from public chain data (or from a pending `verify_foreign_transaction` transaction in the NEAR mempool).
2. Attacker submits 128 `verify_foreign_transaction` calls with the identical `{request, domain_id, payload_version}` tuple, each attaching the minimum 1 yoctoNEAR deposit. Total cost: 128 × gas + 128 yoctoNEAR ≈ negligible.
3. The queue for that request key is now full.
4. The legitimate bridge relayer's `verify_foreign_transaction` call panics with `PendingRequestQueueFull` and is rejected.
5. The queue drains only via NEAR yield timeouts (one slot per timeout event). The attacker monitors the queue length and resubmits to refill it, sustaining the DoS indefinitely.

The `verify_foreign_transaction` function itself performs no per-caller rate limiting or origin check before calling `push_pending_yield`:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    self.check_request_preconditions(...);
    // ... chain support check ...
    let request = args_into_verify_foreign_tx_request(request);
    self.enqueue_yield_request(
        method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
        callback_args,
        callback_gas,
        move |this, id| this.add_verify_foreign_tx_request(request, id),
    );
}
``` [5](#0-4) 

### Impact Explanation

`verify_foreign_transaction` is the on-chain entry point for the Omnibridge inbound flow (foreign chain → NEAR). A successful call produces an MPC-signed attestation that a foreign deposit finalized; downstream NEAR contracts use this attestation to release bridged funds. If an attacker saturates the queue for a specific foreign `tx_id`, the corresponding bridge deposit cannot be attested and the bridged funds remain permanently locked on the foreign chain for as long as the attack is sustained. This is a **request-lifecycle manipulation that breaks a production safety/accounting invariant** (the guarantee that a valid foreign transaction can be verified and attested) without requiring any network-level DoS or operator misconfiguration. Classified **Medium** under the allowed impact scope.

### Likelihood Explanation

- The target `tx_id` is public on the foreign chain and visible in the NEAR mempool before the transaction lands.
- The attack costs 128 × ~7 TGas + 128 yoctoNEAR per queue fill cycle — economically trivial.
- No privileged access, TEE capability, or threshold collusion is required; any NEAR account suffices.
- The attacker can automate queue monitoring and refilling with a simple script.

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring the design of `SignatureRequest`:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
    pub caller: AccountId,   // add this
}
```

This makes each caller's queue slot independent, so an attacker can only saturate their own slots, not a victim's. Alternatively, enforce a per-`(caller, request_key)` submission limit at the contract level before `push_pending_yield` is called.

### Proof of Concept

```
// Attacker script (pseudo-code)
let target_tx_id = observe_from_mempool_or_foreign_chain();
let request_args = VerifyForeignTransactionRequestArgs {
    domain_id: FOREIGN_TX_DOMAIN_ID,
    payload_version: ForeignTxPayloadVersion::V1,
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: target_tx_id,
        confirmations: 2,
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
};

// Fill the queue to MAX_PENDING_REQUEST_FAN_OUT = 128
for _ in 0..128 {
    near_call("verify_foreign_transaction", request_args, deposit=1_yoctoNEAR);
}

// Legitimate relayer's call now panics: "Pending-request queue is full (limit: 128)"
// Attacker monitors queue length and refills after each timeout cycle
```

The unit test at `crates/contract/src/lib.rs:3300–3344` already demonstrates that filling the queue to 128 causes the next push to panic with `"Pending-request queue is full"`. The only missing piece is that the test uses a single caller; the production vulnerability arises because any caller can contribute to the same queue key for `verify_foreign_transaction`. /// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
}
