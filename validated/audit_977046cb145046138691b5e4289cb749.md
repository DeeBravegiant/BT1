Audit Report

## Title
Missing Access Control on `sendFunds()` Allows Anyone to Force MEV Reward Distribution - (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.sendFunds()` is declared `external` with no role or ownership guard, while every other state-mutating function in the same contract (`setDepositPool`) is gated behind `onlyRole(LRTConstants.MANAGER)`. Any unprivileged caller can invoke `sendFunds()` at any time, immediately forwarding the entire accumulated ETH balance of `FeeReceiver` to `LRTDepositPool`. This removes the protocol's ability to control the timing, batching, or conditioning of MEV/execution-layer reward distribution.

## Finding Description
`FeeReceiver` accumulates ETH via its `receive()` fallback. The intended flow is for the protocol to call `sendFunds()` on its own schedule to forward rewards to the deposit pool. However, the function carries no access modifier:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

The destination, `LRTDepositPool.receiveFromRewardReceiver()`, is also unguarded (`external payable { }`), so the full call chain completes with zero checks. By contrast, `setDepositPool` at L66 requires `onlyRole(LRTConstants.MANAGER)`, confirming that the omission on `sendFunds()` is inconsistent with the contract's own access-control pattern.

## Impact Explanation
The concrete allowed impact is **Low — Contract fails to deliver promised returns, but doesn't lose value.** The `FeeReceiver` contract is designed to give the protocol control over when MEV rewards enter the deposit pool (e.g., to coincide with oracle updates, paused states, or batching windows). Because `sendFunds()` is callable by anyone, that control is entirely absent. No ETH leaves the protocol system; it is merely moved from `FeeReceiver` to `LRTDepositPool` earlier than the protocol intends, preventing the protocol from delivering its promised behavior of controlled reward forwarding.

## Likelihood Explanation
The function is unconditionally callable by any EOA or contract with no preconditions beyond gas. `FeeReceiver` continuously accumulates ETH from validator MEV and execution-layer rewards, so a non-zero balance will routinely be present. No special privileges, flash loans, or external dependencies are required. Likelihood is **High**.

## Recommendation
Add the same role guard used by `setDepositPool`:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

## Proof of Concept
1. Deploy or fork with `FeeReceiver` holding a non-zero ETH balance (e.g., 1 ETH from simulated MEV).
2. From an arbitrary EOA (no roles), call `FeeReceiver.sendFunds()`.
3. Observe the call succeeds: `LRTDepositPool.receiveFromRewardReceiver()` receives the full balance, `MevRewardsAddedToTVL` is emitted, and `FeeReceiver.balance` drops to zero.
4. Repeat at any time — no role assignment or other precondition is ever required.

Foundry test sketch:
```solidity
function test_sendFunds_noAccessControl() public {
    vm.deal(address(feeReceiver), 1 ether);
    vm.prank(address(0xdead)); // unprivileged EOA
    feeReceiver.sendFunds();   // must not revert
    assertEq(address(feeReceiver).balance, 0);
    assertGt(address(depositPool).balance, 0);
}
```
