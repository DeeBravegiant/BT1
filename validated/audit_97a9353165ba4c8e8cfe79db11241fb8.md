Audit Report

## Title
Reward Tokens Permanently Frozen When All Stakers Exit During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.notifyRewardAmount` transfers reward tokens into the contract and sets a `rewardRate` over a fixed `duration`. If all stakers call `initiateWithdrawal` and subsequently `claimWithdrawal` during an active reward window, `totalKernelStaked` drops to zero, causing `rewardPerToken()` to freeze accumulation. The reward tokens corresponding to the zero-staked interval are permanently locked with no on-chain recovery path.

## Finding Description
`notifyRewardAmount` guards against starting a reward period with zero stakers:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();  // L570
```

However, this check only applies at the moment of calling `notifyRewardAmount`. It does not prevent stakers from exiting after the reward period has begun. Once `totalKernelStaked` reaches zero mid-period, `rewardPerToken()` short-circuits:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;  // L409-410 — accumulation halts
    }
    ...
}
```

The `rewardRate * elapsed` value for every second that `totalKernelStaked == 0` is silently discarded and never credited to any user. The contract's own NatSpec (L18-22) explicitly acknowledges this behavior and relies entirely on an off-chain operational assumption — that stakers will remain throughout the reward window — rather than any on-chain enforcement. The admin function set (`setRewardsDuration`, `notifyRewardAmount`, `setWithdrawalDelay`, `setMaxNumberOfWithdrawalsPerUser`) contains no `recoverTokens` or equivalent, leaving the stuck tokens permanently irrecoverable.

## Impact Explanation
Reward tokens transferred into the contract via `notifyRewardAmount` are permanently unrecoverable once `totalKernelStaked` hits zero mid-period. This constitutes **permanent freezing of unclaimed yield**, matching the Medium severity impact class in the allowed scope.

## Likelihood Explanation
Any staker can call `initiateWithdrawal(balance)` followed by `claimWithdrawal` after `withdrawalDelay` elapses. No privileged action is required beyond the normal admin setup of a reward period. If the last staker exits during an active window — whether voluntarily, in response to market conditions, or as part of a coordinated exit — `totalKernelStaked` reaches zero and the remaining rewards are frozen. The `withdrawalDelay` is capped at `MAX_WITHDRAWAL_DELAY = 30 days`, which is well within a typical reward `duration`.

## Recommendation
Add an admin-callable `recoverRewards` function that transfers excess `rewardsToken` balance (i.e., `rewardsToken.balanceOf(address(this))` minus the sum of all pending `rewards[user]` balances) to a configurable treasury address. Alternatively, enforce on-chain that `totalKernelStaked` cannot drop to zero during an active reward period by reverting `initiateWithdrawal` when `block.timestamp < finishAt && totalKernelStaked - _amount == 0`.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` — 1,000 reward tokens enter the contract; `rewardRate` is set, `finishAt = block.timestamp + duration`. Guard at L570 passes because stakers exist.
2. All stakers call `initiateWithdrawal(balance)`, reducing `totalKernelStaked` to 0. `initiateWithdrawal` has no check preventing this.
3. After `withdrawalDelay`, all stakers call `claimWithdrawal` — their KERNEL is returned.
4. `totalKernelStaked == 0`. `rewardPerToken()` at L409 now always returns the frozen `rewardPerTokenStored`.
5. Time passes; `finishAt` is reached. The undistributed reward tokens proportional to the zero-staked window remain in the contract.
6. No function exists to retrieve them. Calling `notifyRewardAmount` again would revert at L570 (`NoStakedTokens`) unless a new staker enters, but even then the previously stuck tokens are not redistributed — they are simply added to the new `rewardRate` calculation only if `block.timestamp < finishAt`, which it is not after the period ends.
