Audit Report

## Title
Missing Chainlink Staleness Check Enables Stale Price to Trigger Protocol Auto-Pause - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards the `updatedAt` timestamp entirely, applying no staleness threshold. A stale Chainlink feed for any supported LST asset causes `LRTOracle._updateRsETHPrice()` to compute an artificially low rsETH price. If the deviation exceeds `pricePercentageLimit`, the protocol's downside-protection logic automatically pauses both `LRTDepositPool` and `LRTWithdrawalManager`, temporarily freezing all user deposits and withdrawals until an admin manually unpauses.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` (L49–55) retrieves price data via `latestRoundData()` but only consumes the `price` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

The `updatedAt` (4th return value) and `answeredInRound` (5th return value) are both discarded. No `block.timestamp - updatedAt <= maxStaleness` guard exists, and no per-asset heartbeat mapping is present.

The stale price propagates as follows:
1. Any caller invokes `LRTOracle.updateRSETHPrice()` — a public function with no access control beyond `whenNotPaused`.
2. `_updateRsETHPrice()` calls `_getTotalEthInProtocol()` (L331–349), which iterates over all supported assets and calls `getAssetPrice(asset)` for each.
3. `LRTOracle.getAssetPrice()` (L156–158) delegates to `ChainlinkPriceOracle.getAssetPrice()`, which returns the stale (lower) price.
4. `totalETHInProtocol` is understated; `newRsETHPrice` is computed below `highestRsethPrice`.
5. At L270–282, if `highestRsethPrice - newRsETHPrice > pricePercentageLimit.mulWad(highestRsethPrice)`, the condition fires: `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()` are all called.

The pool-level wrapper `ChainlinkOracleForRSETHPoolCollateral.sol` (L26–37) implements partial checks (`answeredInRound < roundID`, `timestamp == 0`), demonstrating the protocol is aware of oracle data quality issues — yet the core `ChainlinkPriceOracle` used by `LRTOracle` for rsETH price computation has none of these guards.

## Impact Explanation
**Medium — Temporary freezing of funds.** A stale Chainlink feed for any one supported LST asset (e.g., ETHx/ETH with a 24-hour heartbeat) causes `_updateRsETHPrice()` to understate `totalETHInProtocol`. If the resulting price drop exceeds `pricePercentageLimit`, both `LRTDepositPool` and `LRTWithdrawalManager` are paused, preventing all deposits and withdrawals. Users with pending withdrawals or deposits are frozen with no recourse until an admin manually calls `unpause()`. This matches the allowed impact of "Temporary freezing of funds."

## Likelihood Explanation
**Low.** No attacker action is required — this is a passive, market-condition-triggered failure path. Chainlink feeds can go stale during network congestion or oracle node downtime. The impact is conditional on: (a) `pricePercentageLimit` being set to a non-zero value (the intended production configuration for downside protection), and (b) the stale price deviating enough to cross that threshold. The trigger function `updateRSETHPrice()` is public and callable by any address.

## Recommendation
Add a per-asset configurable staleness threshold to `ChainlinkPriceOracle`, with a setter restricted to `onlyLRTManager`:

```solidity
mapping(address asset => uint256 maxStaleness) public assetMaxStaleness;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale round");
    require(updatedAt != 0, "Incomplete round");
    uint256 staleness = assetMaxStaleness[asset];
    if (staleness > 0) require(block.timestamp - updatedAt <= staleness, "Stale price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`assetMaxStaleness` should be initialized to each asset's Chainlink heartbeat interval when the price feed is registered via `updatePriceFeedFor()`.

## Proof of Concept
1. Deploy a fork of mainnet with `pricePercentageLimit` set to a non-zero value (e.g., 1e16 = 1%).
2. Warp `block.timestamp` forward by 26 hours (past the ETHx/ETH 24-hour heartbeat).
3. The Chainlink ETHx/ETH feed's `latestRoundData()` now returns an `updatedAt` 26 hours in the past; the price reflects the last reported value (potentially lower than current).
4. Call `LRTOracle.updateRSETHPrice()` from any EOA.
5. Observe that `_getTotalEthInProtocol()` uses the stale ETHx price, understating `totalETHInProtocol`.
6. If `highestRsethPrice - newRsETHPrice > pricePercentageLimit * highestRsethPrice / 1e18`, assert that `lrtDepositPool.paused() == true` and `withdrawalManager.paused() == true`.
7. Confirm no admin action was taken and no privileged role was involved in triggering the pause.
