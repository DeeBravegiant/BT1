Audit Report

## Title
Stale Cached Rate Used in L2 Pool rsETH Minting Calculations Due to Missing Staleness Check in `CrossChainRateReceiver.getRate()` - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver.getRate()` returns the cached `rate` state variable unconditionally, with no validation against the stored `lastUpdated` timestamp. All L2 pool contracts (`RSETHPool`, `RSETHPoolV3`, and variants) call this function to price deposits and compute wrsETH mint amounts. If the cross-chain rate has not been refreshed recently, the stale lower rate silently causes over-issuance of wrsETH to new depositors, diluting the accrued yield of all existing holders.

## Finding Description
`CrossChainRateReceiver` stores two state variables: `rate` (the last received rsETH/ETH exchange rate) and `lastUpdated` (the timestamp of the last LayerZero message delivery). [1](#0-0) 

`lzReceive` correctly updates both fields when a LayerZero message arrives: [2](#0-1) 

However, `getRate()` returns `rate` with no check against `lastUpdated`: [3](#0-2) 

Every L2 pool contract calls this function to price deposits. In `RSETHPoolV3`: [4](#0-3) 

The stale rate is then used directly in `viewSwapRsETHAmountAndFee` to compute how many wrsETH tokens to mint: [5](#0-4) 

The same pattern exists in `RSETHPool.viewSwapRsETHAmountAndFee`: [6](#0-5) 

The rate update mechanism lives in `MultiChainRateProvider.updateRate()`, which is `external payable` with no access control — anyone can call it, but must pay LayerZero fees. There is no on-chain enforcement of update frequency, and no circuit-breaker in `getRate()` to reject a stale value. [7](#0-6) 

## Impact Explanation
rsETH price monotonically increases over time as staking rewards accrue. When the cached `rate` is lower than the true current L1 rate, the formula `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` produces a larger `rsETHAmount` than the depositor is entitled to. This over-issuance dilutes all existing rsETH/wrsETH holders by transferring their unclaimed staking yield to new depositors. This maps directly to **High — Theft of unclaimed yield**.

## Likelihood Explanation
`MultiChainRateProvider.updateRate()` is permissionless but requires the caller to pay LayerZero cross-chain messaging fees. A keeper outage, fee spike, or simply infrequent calls leaves `rate` stale for hours or days. The `lastUpdated` field is already stored on-chain, confirming the developers anticipated the need to track freshness, but the enforcement check was never implemented. Any unprivileged depositor can trigger the impact simply by calling `deposit()` during a stale-rate window — no special capability is required.

## Recommendation
Add a configurable `maxStaleness` parameter and enforce it inside `getRate()`:

```solidity
uint256 public maxStaleness; // e.g. 86400 (24 hours)

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

`maxStaleness` should be set by the owner and should reflect the expected keeper update cadence with a reasonable safety margin.

## Proof of Concept
1. At time `T`, `lzReceive` delivers `rate = 1.05e18` and sets `lastUpdated = T`.
2. Staking rewards accrue; the true L1 rsETH price rises to `1.07e18` by `T + 2 days`.
3. No one calls `MultiChainRateProvider.updateRate()` during this window (keeper outage or fee avoidance).
4. A user calls `RSETHPoolV3.deposit{value: 1 ether}()` at `T + 2 days`.
5. `getRate()` returns the stale `1.05e18` instead of `1.07e18`.
6. `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.9524 wrsETH`; correct would be `1e18 * 1e18 / 1.07e18 ≈ 0.9346 wrsETH`.
7. The user receives ~1.9% more wrsETH than entitled, at the expense of all existing holders.

**Foundry fork test plan:**
- Fork an L2 where `RSETHPoolV3` is deployed.
- Record the current `CrossChainRateReceiver.rate` and `lastUpdated`.
- `vm.warp(block.timestamp + 2 days)` to simulate staleness without a new `lzReceive` call.
- Call `deposit{value: 1 ether}()` and record `rsETHAmount` minted.
- Compare against the amount that would be minted if `rate` were updated to the current L1 value.
- Assert the difference represents yield transferred from existing holders to the new depositor.
