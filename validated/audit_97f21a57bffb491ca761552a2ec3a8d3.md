Audit Report

## Title
Polynomial Gas Complexity in `updateRSETHPrice()` via Nested Loops Over Assets × NDCs × Queued Withdrawals × Strategies — (`contracts/LRTOracle.sol`)

## Summary
`updateRSETHPrice()` is a public, permissionless function whose gas cost scales as O(assets × NDCs × queued-withdrawals × strategies) due to nested iteration across live EigenLayer state. The protocol's own code acknowledges the function breaks above ~120 uncompleted withdrawals, but the cap of 80 does not account for the multiplicative effect of multiple supported assets and NDCs. At realistic production scale, the function can exceed the block gas limit, permanently staling the rsETH price and breaking all deposit and withdrawal accounting.

## Finding Description

`updateRSETHPrice()` is `public` and calls `_updateRsETHPrice()` → `_getTotalEthInProtocol()`. [1](#0-0) 

`_getTotalEthInProtocol()` opens **Loop 1** over every supported asset, and for each calls `ILRTDepositPool.getTotalAssetDeposits(asset)`: [2](#0-1) 

`getTotalAssetDeposits` delegates to `getAssetDistributionData`, which opens **Loop 2** over every NDC and calls `getAssetUnstaking` on each: [3](#0-2) 

`NodeDelegator.getAssetUnstaking` makes an external call to `DelegationManager.getQueuedWithdrawals` and then opens **Loops 3 and 4** over every queued withdrawal and every strategy within it: [4](#0-3) 

The combined complexity is **O(N × M × W × S)** where N = supported assets, M = NDCs, W = queued withdrawals per NDC, S = strategies per withdrawal. Critically, `getQueuedWithdrawals` is called **N × M** times in a single transaction — once per (asset, NDC) pair — even though the withdrawal data for a given NDC is identical across all asset iterations.

The protocol's own comment in `setMaxUncompletedWithdrawalCount` acknowledges the function breaks above ~120 total withdrawals and caps the value at 80: [5](#0-4) 

However, this comment appears to assume a small, fixed number of assets and NDCs. With N=5 assets and M=10 NDCs, `getQueuedWithdrawals` is called 50 times per price update, and the total inner iterations reach 5 × 80 × 5 = 2,000 — each involving cold EigenLayer storage reads. Meanwhile, `updateMaxNodeDelegatorLimit` has no upper bound, allowing the admin to raise M arbitrarily: [6](#0-5) 

## Impact Explanation

**Medium — Unbounded gas consumption.** If `updateRSETHPrice()` consistently reverts due to out-of-gas, `rsETHPrice` becomes permanently stale. Every deposit (`getRsETHAmountToMint`) and every withdrawal (`getExpectedAssetAmount`) reads the stored `rsETHPrice` directly: [7](#0-6) 

A stale price causes all users minting or redeeming rsETH to receive incorrect amounts, constituting broken protocol accounting for all participants. The `updateRSETHPriceAsManager` escape hatch exists but requires a privileged caller and does not resolve the root cause.

## Likelihood Explanation

`updateRSETHPrice()` is permissionlessly callable by any address. No attacker action is required — the protocol reaching its own configured limits through normal operation (adding assets, adding NDCs, accumulating queued EigenLayer withdrawals) is sufficient to trigger the condition. The team's own comment in `setMaxUncompletedWithdrawalCount` confirms awareness that the function fails above ~120 withdrawals, and the current cap of 80 provides a margin that shrinks multiplicatively as more assets or NDCs are added. The `maxNodeDelegatorLimit` has no hard cap, so admin-driven protocol growth can push the function past the block gas limit without any malicious action.

## Recommendation

1. **Eliminate redundant `getQueuedWithdrawals` calls**: `getAssetUnstaking` is called N times per NDC (once per asset), but each call fetches the same withdrawal data from EigenLayer. Refactor to fetch `getQueuedWithdrawals(ndc)` once per NDC and compute all asset amounts in a single pass.
2. **Decouple unstaking accounting from live EigenLayer queries**: Maintain an on-chain `assetUnstaking[asset]` accumulator updated at `initiateUnstaking`/`completeUnstaking` time instead of re-deriving it from `getQueuedWithdrawals` on every price update.
3. **Hard-cap `maxNodeDelegatorLimit`**: Add an enforced upper bound in `updateMaxNodeDelegatorLimit` analogous to the cap on `maxUncompletedWithdrawalCount`.
4. **Consolidate ETH and LST distribution queries**: `getETHDistributionData` and `getAssetDistributionData` both loop over all NDCs; merge into a single pass per price update.

## Proof of Concept

Minimal call sequence to trigger the nested loops (no privileges required):

```
// Any EOA calls:
LRTOracle.updateRSETHPrice()
  → _updateRsETHPrice()
    → _getTotalEthInProtocol()
      → for each asset[i] in supportedAssets (N iterations):          // Loop 1
          LRTDepositPool.getTotalAssetDeposits(asset[i])
            → getAssetDistributionData(asset[i])
              → for each NDC[j] in nodeDelegatorQueue (M iterations): // Loop 2
                  NodeDelegator[j].getAssetUnstaking(asset[i])
                    → DelegationManager.getQueuedWithdrawals(NDC[j])  // external call
                    → for each withdrawal[k] (W iterations):          // Loop 3
                        for each strategy[l] (S iterations):          // Loop 4
                            strategy.sharesToUnderlyingView(shares)
```

**Foundry fork test plan**: Deploy against a mainnet fork with 5 supported assets, 10 NDCs, and 80 queued withdrawals distributed across NDCs (achievable within the current `maxUncompletedWithdrawalCount` cap). Call `updateRSETHPrice()` with a gas limit of 30,000,000 (block gas limit) and assert it reverts with out-of-gas. Vary N and M to demonstrate the multiplicative scaling — e.g., doubling M from 5 to 10 NDCs doubles the number of `getQueuedWithdrawals` calls from 25 to 50, proportionally increasing gas consumption.

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
