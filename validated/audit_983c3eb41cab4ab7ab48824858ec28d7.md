### Title
Single Byzantine Participant Can Permanently DoS `verify_foreign_transaction` via `register_foreign_chain_support` Empty-List Injection — (`File: crates/contract/src/lib.rs`)

---

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which computes the **strict intersection** of every active participant's registered chain set. A single Byzantine participant (strictly below the signing threshold) can call `register_foreign_chain_support` with an empty `SupportedForeignChains` set at any time, collapsing the intersection to the empty set and causing every subsequent `verify_foreign_transaction` call to panic with `ForeignChainNotSupported`. The attacker can repeat this call indefinitely to sustain the DoS.

---

### Finding Description

`verify_foreign_transaction` reads the supported chain set immediately before accepting a request:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(&InvalidParameters::ForeignChainNotSupported { ... }.to_string());
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the intersection by requiring every active participant to appear in the supporter set for each chain:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain { Some(foreign_chain) } else { None }
``` [2](#0-1) 

Any participant can overwrite their own entry in `node_foreign_chain_support.foreign_chain_support_by_node` at will via `register_foreign_chain_support`:

```rust
pub fn register_foreign_chain_support(
    &mut self,
    foreign_chain_support: dtos::SupportedForeignChains,
) -> Result<(), Error> {
    let account_id = self.voter_or_panic();
    self.node_foreign_chain_support
        .foreign_chain_support_by_node
        .insert(account_id, foreign_chain_support);
    Ok(())
}
``` [3](#0-2) 

Submitting an empty `SupportedForeignChains` set causes `is_superset` to return `false` for every chain, making `get_supported_foreign_chains()` return the empty set. The project's own design documentation explicitly acknowledges this root cause: *"A single node that registers an empty list (or hasn't registered yet) drops every chain — one operator can take the whole feature down."* [4](#0-3) 

The proposed fix (`get_available_foreign_chains()` with a threshold-based rule) exists as a view function and is tested, but `verify_foreign_transaction` still calls the vulnerable `get_supported_foreign_chains()` in production code. [5](#0-4) 

---

### Impact Explanation

Every `verify_foreign_transaction` call for any chain panics immediately at the chain-support check, before any yield is created. The user's 1 yoctonear deposit is refunded (the panic rolls back the call), but the bridge/omnibridge inbound flow is completely non-functional for as long as the attacker sustains the attack. No foreign-chain transaction can be verified or signed. This breaks the production safety invariant that a whitelisted, fully-covered chain must be serviceable.

**Impact class:** Medium — contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

The attacker needs only a single participant account (below the signing threshold). The call is cheap (one storage write), requires no deposit, and can be repeated every block. There is no on-chain rate limit or cooldown on `register_foreign_chain_support`. A participant who turns Byzantine — or whose key is compromised — can sustain the DoS indefinitely at negligible cost.

---

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which already exists and uses the threshold-based rule (chain is available iff ≥ `signing_threshold` active participants cover it). This is the fix already proposed in `docs/design/calculating-supported-foreign-chains.md` and tracked in issue #3434; it must be applied to the gate check in `verify_foreign_transaction` to close the vulnerability. [6](#0-5) 

---

### Proof of Concept

1. Network is `Running` with participants `[P1, P2, P3]`, threshold = 2. All three have registered Bitcoin support. `get_supported_foreign_chains()` returns `{Bitcoin}`.
2. Attacker controls `P1` (one Byzantine participant, below threshold). `P1` calls:
   ```
   register_foreign_chain_support(foreign_chain_support: [])
   ```
3. `node_foreign_chain_support.foreign_chain_support_by_node` now maps `P1 → {}`.
4. `get_supported_foreign_chains()` computes: for Bitcoin, supporters = `{P2, P3}`, active participants = `{P1, P2, P3}`. `{P2,P3}.is_superset({P1,P2,P3})` → `false`. Result: empty set.
5. Any user calling `verify_foreign_transaction` for Bitcoin now hits:
   ```
   ForeignChainNotSupported { requested: Bitcoin }
   ```
   and the call panics. The bridge is fully DoS'd.
6. `P1` re-submits the empty registration every block to prevent honest recovery. [1](#0-0) [3](#0-2)

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }
```
