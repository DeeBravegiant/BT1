Audit Report

## Title
JIT Liquidity Attack Captures LP Spread Fees from Existing Liquidity Providers — (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
The Metric OMM pool credits LP spread fees directly into per-bin token balances during swap execution in `SwapMath.sol`. Because `addLiquidity` and `removeLiquidity` carry no block-number guard, time-lock, or same-block restriction, an attacker can front-run a large pending swap by minting a dominant share position in the active bin, capture the majority of the LP fee when the swap settles, and immediately burn their position — stealing fee income that should have accrued to existing LPs. The only available mitigation (`DepositAllowlistExtension`) is opt-in and not deployed by default.

## Finding Description

**Fee flow into bin balances**

In every swap-in-bin function in `SwapMath.sol`, the LP fee (total fee minus the protocol portion) is credited directly into `binState.token*BalanceScaled`. For `buyToken0InBinSpecifiedOut`:

```solidity
// SwapMath.sol L409-427
uint256 feeAmountScaled = Math.ceilDiv(amountInScaled * currBinBuyFeeX64, ONE_X64);
amountInScaled += feeAmountScaled;
uint256 protocolFeeAmountScaled = (feeAmountScaled * spreadFeeE6) / 1e6;

binState.token0BalanceScaled -= amountOutScaled.toUint104();
binState.token1BalanceScaled =
  (uint256(binState.token1BalanceScaled) + amountInScaled - protocolFeeAmountScaled).toUint104();
// binLpFeeAmount = feeAmountScaled - protocolFeeAmountScaled stays in bin balance
```

The same pattern applies in `buyToken0InBinSpecifiedIn` (L639-641) and both `buyToken1In*` variants (L499-501, L779-781). The LP fee is permanently embedded in the bin balance immediately after each swap step.

**Removal is proportional to current shares**

`LiquidityLib.removeLiquidity` computes each LP's entitlement as a simple ratio of their shares to the total at the moment of removal:

```solidity
// LiquidityLib.sol L205-206
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

Whoever holds shares at the time of removal receives their proportional cut of the bin balance — including any LP fees that accrued during the swap.

**No time-lock or block-number guard on liquidity operations**

`addLiquidity` carries only a reentrancy guard and optional extension hooks; there is no `block.number` or `block.timestamp` check anywhere in `metric-core/contracts/**/*.sol` for liquidity operations (confirmed by grep). The `whenNotPaused` modifier is applied only to `swap`, not to `addLiquidity` or `removeLiquidity`:

```solidity
// MetricOmmPool.sol L188, L199-201, L224
function addLiquidity(...) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) { ... }
function removeLiquidity(...) external nonReentrant(PoolActions.REMOVE_LIQUIDITY) returns (...) { ... }
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (...) { ... }
```

**Exploit flow**

1. Attacker observes a large pending swap in the mempool.
2. Attacker calls `addLiquidity` with a large `sharesToAdd` value targeting the active bin. Because `binTotalSharesVal > 0`, the deposit amount is computed proportionally to the current bin balance (`amount*Scaled = ceil(binState.token*BalanceScaled * sharesToAdd / binTotalSharesVal)`), so the attacker pays fair value for their shares.
3. The victim's swap executes. The LP fee `F` is credited into `binState.token*BalanceScaled`.
4. Attacker calls `removeLiquidity` in the same block. They receive `(balance + F) * attackerShares / totalShares`, capturing `F * N/(N+M)` of the LP fee where N is attacker shares and M is existing LP shares.
5. Existing LPs receive only `F * M/(N+M)` instead of `F`.

**Existing guards are insufficient**

The `DepositAllowlistExtension` can gate deposits per pool, but it is a separate periphery contract (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`) that is entirely opt-in and not deployed by default. Pools without this extension configured are fully exposed.

## Impact Explanation

Existing LPs suffer a direct, quantifiable loss of LP fee income. For every large swap, a JIT attacker can capture an arbitrarily large fraction of the LP spread fee by transiently holding a dominant share position. The attacker deposits tokens at the oracle mid-price (no price risk within a single block), collects the fee, and withdraws — net cost is only gas. This constitutes a direct loss of owed LP assets, which is the primary economic incentive for liquidity provision and falls squarely within the allowed impact gate.

**Concrete numbers:**
- Existing LPs: 10,000 shares in bin 0
- Attacker mints: 990,000 shares → holds 99% of bin
- Victim swaps 100,000 token1 → token0; LP fee = 500 token1 (0.5% spread)
- Attacker withdraws: receives 495 token1 in LP fees
- Existing LPs receive: 5 token1 instead of 500 token1
- **Loss to existing LPs: 495 token1 per swap**

## Likelihood Explanation

- The attack is **permissionless**: any address can call `addLiquidity` and `removeLiquidity` on pools without the `DepositAllowlistExtension`.
- It is **profitable whenever the LP fee on a single swap exceeds gas cost**, which is routine for swaps of meaningful size.
- MEV bots routinely monitor mempools for large pending swaps and can execute the three-transaction bundle (add → swap executes → remove) atomically via a Flashbots bundle or equivalent private mempool relay.
- No special privileges, malicious setup, or non-standard tokens are required.
- The attack is repeatable on every large swap indefinitely.

## Recommendation

Implement a JIT liquidity guard at the core pool level. Options include:

1. **Block-number lock (lowest complexity):** Record `lastAddBlock[posKey]` on every `addLiquidity` and revert `removeLiquidity` if `block.number == lastAddBlock[posKey]`. This directly closes the same-block attack vector.
2. **Time-weighted fee accrual:** Instead of crediting LP fees directly into bin balances, accumulate them in a separate per-bin fee accumulator and distribute them based on time-weighted share ownership (similar to Uniswap v3's `feeGrowthInside`).
3. **Minimum hold period extension:** Expose a configurable `minHoldBlocks` parameter via the extension system (`beforeRemoveLiquidity`) so pool admins can enforce a minimum liquidity duration.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

contract JITAttackPoC {
    IMetricOmmPool pool;
    IERC20 token0;
    IERC20 token1;

    function attack(uint256 victimSwapAmount) external {
        // Step 1: Front-run — add large liquidity to the active bin
        // Attacker mints N shares; existing LPs hold M shares total.
        // Attacker's share fraction ≈ N / (N + M) → ~99% for large N.
        int256[] memory bins = new int256[](1);
        bins[0] = 0; // active bin
        uint256[] memory shares = new uint256[](1);
        shares[0] = 1_000_000e18; // dominate the bin
        pool.addLiquidity(
            address(this), 0,
            LiquidityDelta({binIdxs: bins, shares: shares}),
            "", ""
        );

        // Step 2: Victim's swap executes (same block, next tx).
        // LP fee = (bidAskSpread / 2) * victimSwapAmount * (1 - protocolFeeShare)
        // This fee is credited into binState.token1BalanceScaled.

        // Step 3: Back-run — remove all attacker shares immediately.
        pool.removeLiquidity(
            address(this), 0,
            LiquidityDelta({binIdxs: bins, shares: shares}),
            ""
        );
        // Attacker receives: original deposit + ~99% of LP fee from victim's swap.
        // Existing LPs receive: ~1% of LP fee they were entitled to.
    }

    function metricOmmModifyLiquidityCallback(
        uint256 amount0, uint256 amount1, bytes calldata
    ) external {
        if (amount0 > 0) token0.transfer(msg.sender, amount0);
        if (amount1 > 0) token1.transfer(msg.sender, amount1);
    }
}
```
