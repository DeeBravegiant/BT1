### Title
Single Byzantine Attested Participant Can Resolve Non-PV CKD Request With Arbitrary Output, Bypassing Threshold Authorization — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs no cryptographic output check for the `AppPublicKey` (non-PV) variant. A single Byzantine attested participant can call `respond_ckd` directly with any `(big_y, big_c)` pair and the contract will resolve the pending yield immediately, bypassing the threshold MPC protocol entirely. The requester's TEE app receives a key that is not bound to the MPC master secret.

---

### Finding Description

`respond_ckd` branches on the request variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(H(pk||app_id), msk_pk)`, cryptographically binding the response to the MPC master secret key. [2](#0-1) 

For `AppPublicKey`, the arm is a no-op. After the match, `resolve_yields_for` is called unconditionally with whatever bytes were supplied: [3](#0-2) 

`resolve_yields_for` requires exactly one call from one attested participant to drain the entire yield queue — there is no threshold counter, no quorum check, no aggregation: [4](#0-3) 

On the node side, the non-PV coordinator (`protocol.rs::do_ckd_coordinator`) aggregates shares from threshold participants but performs no `aggregated_output_check` before returning: [5](#0-4) 

The PV coordinator (`protocol_pv.rs::do_ckd_coordinator`) does run `aggregated_output_check` before returning, but this is a node-side guard only — it does not protect the contract entrypoint: [6](#0-5) 

A Byzantine node can skip the MPC protocol entirely and call `respond_ckd` directly on the contract. The only contract-side guards are:
- `assert_caller_is_attested_participant_and_protocol_active()` — requires the caller to be an attested participant, not threshold-many
- `protocol_state.is_running_or_resharing()` — liveness check only [7](#0-6) 

The existing unit test confirms this path is reachable: it passes `[1u8; 48]` and `[2u8; 48]` as `big_y`/`big_c` for a non-PV request and the contract accepts and resolves the yield: [8](#0-7) 

---

### Impact Explanation

The TEE app recovers the derived key as `s = big_c − app_sk · big_y`. If the attacker submits `big_y = G1_identity` and `big_c = P` (any chosen G1 point), then `s = P − app_sk · 0 = P`. The attacker chose `P`, so they know `s`. The TEE app's secret is now fully controlled and known by the attacker. Even without the identity trick, the attacker can inject a key that is not `msk · H(pk || app_id)`, breaking the determinism and confidentiality guarantees of the CKD system.

This matches: **Critical — confidential key derivation output without the required participant authorization.**

---

### Likelihood Explanation

Requires a single Byzantine attested participant — one compromised TEE node that passes attestation. The threshold system is explicitly designed to tolerate up to `t−1` Byzantine participants. The attack requires no collusion, no chain-level failure, and no special tooling beyond a direct contract call. The non-PV variant is the default in the CLI and is actively used in production. [9](#0-8) 

---

### Recommendation

Add an on-chain output check for the `AppPublicKey` branch. Since the non-PV variant only provides a G1 key (`app_pk`), the pairing check `e(big_c, g2) = e(big_y, g2) · e(H(pk||app_id), msk_pk)` cannot be applied directly (it requires the G2 component). Two options:

1. **Deprecate the non-PV variant** and require all callers to use `AppPublicKeyPV`, which supports the full on-chain pairing check.
2. **Require threshold-many `respond_ckd` calls** for non-PV requests, with the contract aggregating shares and verifying the result before resolving the yield — mirroring how threshold signing works.

Option 1 is simpler and eliminates the gap entirely.

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
