### Title
Permanently Locked `nativeFee` ETH in `initTransfer` / `initTransfer1155` with No Withdrawal Path — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

Every call to `initTransfer` or `initTransfer1155` that carries a non-zero `nativeFee` permanently locks that ETH inside `OmniBridge`. The contract contains no `withdraw`, `rescue`, or `sweep` function for native ETH, and the only ETH egress path (`finTransfer` for `tokenAddress == address(0)`) releases only the bridged `amount`, never the fee. A stub enum value `PayloadType.ClaimNativeFee` in `BridgeTypes.sol` confirms the design intent existed but the implementation was never completed.

---

### Finding Description

`initTransfer` computes `extensionValue` by subtracting both `amount` (for native-ETH transfers) and `nativeFee` from `msg.value`:

```solidity
// OmniBridge.sol lines 387–393
if (tokenAddress == address(0)) {
    extensionValue = msg.value - amount - nativeFee;
} else {
    extensionValue = msg.value - nativeFee;
    ...
}
``` [1](#0-0) 

Only `extensionValue` is forwarded downstream — in `OmniBridgeWormhole` it is passed as `{value: value}` to `_wormhole.publishMessage`:

```solidity
// OmniBridgeWormhole.sol line 143
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
``` [2](#0-1) 

The `nativeFee` slice of `msg.value` is never forwarded, never credited to any address, and never emitted in a way that triggers a payout on the EVM side. It simply accumulates in the contract's ETH balance.

`initTransfer1155` has the identical accounting gap:

```solidity
// OmniBridge.sol line 466
uint256 extensionValue = msg.value - nativeFee;
``` [3](#0-2) 

A search across the entire contract confirms there is no `withdraw`, `rescue`, or `sweep` function for native ETH. The only ETH egress is `finTransfer` when `payload.tokenAddress == address(0)`, which releases only `payload.amount` to the designated recipient — never any accumulated fee balance. [4](#0-3) 

The `receive()` fallback also silently accepts ETH with no recovery path:

```solidity
// OmniBridge.sol line 574
receive() external payable {}
``` [5](#0-4) 

The smoking-gun evidence that this is an incomplete implementation rather than intentional design is the `PayloadType.ClaimNativeFee` enum value in `BridgeTypes.sol`:

```solidity
// BridgeTypes.sol lines 67–71
enum PayloadType {
    TransferMessage,
    Metadata,
    ClaimNativeFee
}
``` [6](#0-5) 

No function in `OmniBridge.sol` or `OmniBridgeWormhole.sol` ever constructs or processes a `ClaimNativeFee` payload. The enum slot exists, the claim mechanism does not.

---

### Impact Explanation

Every `initTransfer` / `initTransfer1155` call with `nativeFee > 0` irreversibly locks that ETH. Over the lifetime of the bridge the locked balance grows monotonically. Because there is no admin rescue path either, the funds are unrecoverable even by the protocol owner. This satisfies **Critical — Irreversible fund lock, permanently unclaimable user or protocol value in bridge fee flows**.

---

### Likelihood Explanation

`nativeFee` is a first-class parameter of the public `initTransfer` interface and is included in the `InitTransfer` event, so relayers and users are expected to supply non-zero values. The NEAR-side `fin_transfer` logic reads `native_fee` from the event and uses it to compensate relayers in NEAR tokens, meaning the EVM-side ETH collection is the intended funding mechanism — but the disbursement half was never implemented. Every production transfer with a non-zero `nativeFee` triggers the lock.

---

### Recommendation

1. Implement the missing `ClaimNativeFee` flow: add a function (callable by an authorized relayer or the DAO) that constructs a `ClaimNativeFee` payload, has it signed by the MPC/Wormhole layer, and transfers the accumulated fee balance to the designated recipient.
2. As an immediate mitigation, add an admin-only ETH rescue function so that locked fees can be recovered while the full claim mechanism is built.
3. Alternatively, if `nativeFee` is meant to be paid entirely on the NEAR side, remove the `msg.value - nativeFee` deduction from `initTransfer` so users are not required to send ETH that will never be disbursed.

---

### Proof of Concept

1. Alice calls `initTransfer(USDC, 1000e6, 0, 5e15, "near:alice.near", "")` sending `msg.value = 5e15 + wormholeFee`.
2. `extensionValue = msg.value - 5e15 = wormholeFee`. Only `wormholeFee` is forwarded to Wormhole.
3. `5e15` wei (0.005 ETH) remains in `OmniBridge`'s balance.
4. No function exists to withdraw it. Repeat for every user who pays a `nativeFee`.
5. Confirm: call `address(omniBridge).balance` after N transfers — it equals `Σ nativeFee_i` plus any ETH sent to `receive()`, with no path to zero it out.
