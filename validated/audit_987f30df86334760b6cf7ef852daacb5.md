Audit Report

## Title
ETH Deposit Cap Bypass via Missing `amount` in Limit Check - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the incoming `amount` from the ETH branch of its comparison, checking only `totalAssetDeposits > limit` instead of `totalAssetDeposits + amount > limit`. This means the cap is only enforced after it has already been exceeded, not before. Any unprivileged depositor can call `depositETH()` and push total ETH deposits past the configured cap in a single transaction.

## Finding Description
At `contracts/LRTDepositPool.sol` lines 676â€“682, the function branches on asset type:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount excluded
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
```

For ETH, when `totalAssetDeposits == limit` (exactly at cap), the expression `totalAssetDeposits > limit` evaluates to `false`, so the function returns `false` and the deposit is not blocked â€” regardless of how large `amount` is. The LST branch correctly includes `amount`.

This function is called by `_beforeDeposit` (lines 648â€“670), which reverts on `true`. Since the ETH branch never returns `true` until the cap is already breached, the guard is ineffective at the boundary. `depositETH` (lines 76â€“93) is the public, payable, permissionless entry point that invokes this path. The view function `getAssetCurrentLimit` (lines 402â€“409) correctly computes remaining headroom as `limit - totalAssetDeposits`, so off-chain callers see a non-zero limit and expect enforcement that does not actually exist.

## Impact Explanation
The ETH deposit cap is a risk-management invariant bounding EigenLayer exposure. Because `amount` is excluded from the ETH branch, a single depositor can push `totalAssetDeposits[ETH]` from exactly `limit` to `limit + X` in one call, where `X` is unbounded. The protocol mints rsETH for the full over-limit amount, diluting the backing ratio for all existing holders and exposing the protocol to more EigenLayer risk than governance intended. No direct fund theft occurs in isolation.

**Impact: Low** â€” Contract fails to deliver the promised deposit-limit invariant.

## Likelihood Explanation
`depositETH` is permissionless and payable. No special role, flash loan, or multi-step setup is required. Any depositor who observes `totalAssetDeposits` at or near the ETH cap can send an arbitrarily large ETH value in a single transaction to exceed it. The condition is trivially observable on-chain and repeatable across blocks as long as `totalAssetDeposits` does not exceed the limit at the start of the transaction.

**Likelihood: High.**

## Recommendation
Include `amount` in the ETH branch, mirroring the LST branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This makes enforcement consistent across all asset types and matches the semantics of `getAssetCurrentLimit`.

## Proof of Concept

1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. Protocol accumulates `totalAssetDeposits[ETH] = 1000 ether` (exactly at cap).
3. For an LST at this point, `_checkIfDepositAmountExceedesCurrentLimit(lst, 1 ether)` returns `true` â†’ deposit reverts correctly.
4. Attacker calls `depositETH{value: 500 ether}(0, "")`.
5. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500 ether)` evaluates `1000 ether > 1000 ether` â†’ `false` â†’ deposit is not blocked.
6. `_mintRsETH` mints rsETH for 500 ETH; `totalAssetDeposits[ETH]` becomes 1500 ether â€” 50% over the configured cap.

**Foundry test plan:** Deploy `LRTDepositPool` on a local fork, set ETH cap to 1000 ether, fill deposits to exactly 1000 ether, then call `depositETH{value: 1 ether}()` and assert it reverts with `MaximumDepositLimitReached`. The test will fail (no revert), confirming the bug. A fuzz variant can sweep `amount` from 1 wei to 10 000 ether and assert the invariant `totalAssetDeposits[ETH] <= limit` after every call.
