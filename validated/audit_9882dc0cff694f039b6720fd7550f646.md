### Title
Missing `directDeposits` Amount in `produced` Value Calculation Enables ADA Creation — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field (CBOR key 25) in the transaction body that allows a transaction to credit ADA directly into registered account balances. However, the total amount of ADA specified in `directDeposits` is never added to the `produced` side of the preservation-of-value equation. Because the `consumed == produced` balance check passes without accounting for direct deposits, a transaction submitter can include direct deposit amounts in their UTxO outputs (to satisfy the balance check) while simultaneously having those same amounts credited to account balances — effectively creating ADA out of thin air.

---

### Finding Description

The Dijkstra era's `dijkstraProducedValue` function computes the produced value for a top-level transaction as:

```
outputs + fee + deposits (from certs) + treasury donation + (sub-tx produced values)
``` [1](#0-0) 

It delegates to `conwayProducedValue`, which adds `treasuryDonationTxBodyL` but has no knowledge of `directDeposits`: [2](#0-1) 

The `directDeposits` field is a `Map AccountAddress Coin` stored at key 25 of the Dijkstra transaction body: [3](#0-2) 

The Dijkstra UTXO transition rule enforces the balance check at line 381:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [4](#0-3) 

`validateValueNotConservedUTxO` calls `produced pp certState txBody`, which resolves to `getProducedDijkstraValue` → `dijkstraProducedValue` — which omits `directDeposits`: [5](#0-4) 

After the UTXO rule passes, the ENTITIES rule applies direct deposits to account balances via `applyDirectDeposits`: [6](#0-5) 

`applyDirectDeposits` unconditionally adds the specified coin amounts to account balances: [7](#0-6) 

The analogy to the external report is exact: just as the AE amount was missing from the smart-contract call confirmation (and thus not tracked), the `directDeposits` amount is missing from the `produced` accounting (and thus not deducted from the UTxO side), allowing the same ADA to appear in both UTxO outputs and account balances simultaneously.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker can craft a transaction where:

1. Inputs supply N ADA.
2. Outputs return `N − fee` ADA to the attacker's UTxO address (balance check passes because `directDeposits` is absent from `produced`).
3. `directDeposits` credits D ADA to the attacker's registered account address.

After the transaction:
- UTxO holds `N − fee` ADA (attacker's output).
- Account balance holds D ADA (from direct deposits).
- Fee pot holds `fee` ADA.
- **Total = N − fee + D + fee = N + D > N.**

D ADA is created from nothing. The attacker can repeat this for arbitrarily large D, bounded only by the need to have a registered account address as the deposit target.

---

### Likelihood Explanation

**Medium.** The Dijkstra era is experimental and not yet deployed on mainnet. However, the entry path requires only an unprivileged transaction sender with a registered account address — no privileged access, no governance majority, no key compromise. Any user who can submit a Dijkstra transaction can trigger this. Once the era is active, exploitation is straightforward and repeatable.

---

### Recommendation

Add the total `directDeposits` coin amount to the `produced` value in `dijkstraProducedValue`, mirroring how `treasuryDonationTxBodyL` is handled in `conwayProducedValue`:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> inject (foldMap id . unDirectDeposits $ txBody ^. directDepositsTxBodyL)
    <> foldMap'
         (getProducedValue pp isRegPoolId . view bodyTxL)
         (txBody ^. subTransactionsTxBodyL)
```

The same fix must be applied to `dijkstraSubTxProducedValue` if sub-transactions can also carry `directDeposits`: [8](#0-7) 

---

### Proof of Concept

**Setup:** Attacker registers account address `acct1` (e.g., via `RegTxCert`). Attacker holds a UTxO entry `u` worth 100 ADA.

**Malicious transaction:**
```
inputs:          { u }                          -- 100 ADA consumed
outputs:         { (attacker_addr, 99 ADA) }    -- 99 ADA to UTxO
fee:             1 ADA
directDeposits:  { acct1 → 10 ADA }            -- 10 ADA to account balance
```

**Balance check (UTXO rule):**
```
consumed = 100 ADA
produced = 99 (outputs) + 1 (fee) = 100 ADA   ← directDeposits absent
100 == 100  ✓  check passes
```

**ENTITIES rule:**
```
applyDirectDeposits { acct1 → 10 ADA }
→ acct1.balance += 10 ADA
```

**Ledger state after transaction:**
```
UTxO:            99 ADA  (attacker's output)
acct1 balance:   10 ADA  (from direct deposits)
fee pot:          1 ADA
─────────────────────────
Total:           110 ADA  (started with 100 ADA → +10 ADA created)
```

The attacker has gained 10 ADA at zero cost. The attack scales linearly: setting `directDeposits = { acct1 → X ADA }` creates X ADA per transaction.
