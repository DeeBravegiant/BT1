### Title
Byzantine Participant Bypasses Double-Vote Guard via Conflicting Public Key to Permanently Abort Key Generation Instances - (File: `crates/contract/src/state/key_event.rs`)

---

### Summary

In `KeyEventInstance::vote_success()`, the `PublicKeyDisagreement` check is evaluated **before** the `VoteAlreadySubmitted` guard. A single Byzantine participant (strictly below the signing threshold) can exploit this ordering to vote for key K1 (a legitimate success vote, recorded in `completed`), then immediately vote for key K2 (a conflicting key), bypassing the duplicate-vote guard and triggering instance destruction. This can be repeated across every new attempt, permanently preventing key generation from completing.

---

### Finding Description

In `KeyEventInstance::vote_success()`: [1](#0-0) 

```rust
fn vote_success(
    &mut self,
    candidate: AuthenticatedParticipantId,
    public_key: PublicKeyExtended,
) -> Result<VoteSuccessResult, Error> {
    if let Some(existing_public_key) = &self.public_key {
        if existing_public_key != &public_key {
            return Ok(VoteSuccessResult::PublicKeyDisagreement); // ← checked FIRST
        }
    } else {
        self.public_key = Some(public_key);
    }
    // return error if the candidate already submitted a vote.
    if self.completed.contains(&candidate) {
        return Err(VoteError::VoteAlreadySubmitted.into()); // ← checked SECOND
    }
    self.completed.insert(candidate.clone());
    Ok(VoteSuccessResult::Voted(self.completed.len()))
}
```

The `PublicKeyDisagreement` branch (lines 295–298) returns early **before** the `VoteAlreadySubmitted` check (lines 303–305). This means a participant already present in `completed` can submit a second `vote_pk` call with a different public key, bypass the duplicate-vote guard, and trigger instance destruction.

The caller in `KeyEvent::vote_success()` then sets `self.instance = None`: [2](#0-1) 

This destroys the active instance. The `next_attempt_id` was already incremented during `start()`: [3](#0-2) 

So the leader must call `start()` again with the new attempt ID, and the attacker can repeat the same two-vote sequence in every new instance.

The `vote_abort` function correctly prevents a participant who already voted success from aborting: [4](#0-3) 

But the double-vote-via-conflicting-key path entirely bypasses this protection, because `PublicKeyDisagreement` is checked before `completed.contains(&candidate)`.

This vulnerability applies specifically to `vote_pk` in the `Initializing` state, where the public key is participant-supplied: [5](#0-4) 

It does **not** apply to `vote_reshared` in the `Resharing` state, because there the public key is always derived from the previous keyset (not participant-supplied), so all participants necessarily vote for the same key: [6](#0-5) 

---

### Impact Explanation

A single Byzantine participant (strictly below the signing threshold) can permanently prevent the `Initializing` state from completing key generation for any domain. The attack loop is:

1. Call `vote_pk(key_event_id, K1)` → participant added to `completed`, vote recorded.
2. Call `vote_pk(key_event_id, K2)` (K2 ≠ K1) → `PublicKeyDisagreement` returned → `self.instance = None` → instance destroyed.
3. Leader calls `start()` for the next attempt.
4. Repeat indefinitely.

Since `vote_add_domains` transitions the contract into `Initializing` only when all participants agree, and `Initializing` can only exit to `Running` when key generation completes, a single attacker can permanently block the MPC network from adding new signature domains. This breaks the contract execution-flow invariant that key generation completes when all participants cooperate, without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

Any single registered participant can execute this attack with two sequential on-chain transactions. The attacker submits both votes in rapid succession (ideally in the same block), ensuring the second vote is processed before other participants reach the threshold. The attack is cheap (two function calls per instance), repeatable across every new attempt, and requires no coordination or special privileges beyond being a registered participant.

---

### Recommendation

Move the `VoteAlreadySubmitted` check **before** the `PublicKeyDisagreement` check in `KeyEventInstance::vote_success()`:

```rust
fn vote_success(
    &mut self,
    candidate: AuthenticatedParticipantId,
    public_key: PublicKeyExtended,
) -> Result<VoteSuccessResult, Error> {
    // Check duplicate vote FIRST, before any key comparison.
    if self.completed.contains(&candidate) {
        return Err(VoteError::VoteAlreadySubmitted.into());
    }
    if let Some(existing_public_key) = &self.public_key {
        if existing_public_key != &public_key {
            return Ok(VoteSuccessResult::PublicKeyDisagreement);
        }
    } else {
        self.public_key = Some(public_key);
    }
    self.completed.insert(candidate.clone());
    Ok(VoteSuccessResult::Voted(self.completed.len()))
}
```

This ensures a participant who has already cast a success vote cannot trigger `PublicKeyDisagreement` to destroy the instance, preserving the invariant that a participant's success vote is final within an attempt.

---

### Proof of Concept

```
Setup: 5 participants, threshold = 3. Instance N is active.

1. Attacker (participant A) calls vote_pk(instance_N_id, K1):
   - verify_vote() succeeds (A is a participant, instance is active)
   - vote_success(): public_key = None → set to K1; A not in completed → insert A
   - Returns Voted(1). A is now in completed.

2. Attacker (participant A) calls vote_pk(instance_N_id, K2) where K2 ≠ K1:
   - verify_vote() succeeds (A is still a participant, instance is still active)
   - vote_success(): existing_public_key = K1 ≠ K2 → return PublicKeyDisagreement
     (VoteAlreadySubmitted check is never reached, even though A is in completed)
   - KeyEvent::vote_success() receives PublicKeyDisagreement → self.instance = None

3. Instance N is destroyed. Leader must call start() for instance N+1.

4. Attacker repeats steps 1–3 for every new instance, indefinitely.

Result: Key generation for the domain never completes. The Initializing state
is permanently stuck. No new domains can be added to the MPC network.
```
