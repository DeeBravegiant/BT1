Audit Report

## Title
Stale Cross-Chain Oracle Rate Allows Depositors to Mint Excess rsETH and Extract Yield - (File: contracts/pools/RSETHPoolV3.sol)

## Summary

All L2 pool contracts (`RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`, `RSETHPoolNoWrapper`) compute the rsETH mint amount using a cross-chain oracle rate stored in `CrossChainRateReceiver`, which is only updated when a LayerZero message arrives from L1. Because rsETH is yield-bearing and its true rate monotonically increases, the L2 oracle is structurally stale between updates. Any depositor can exploit this gap to receive more rsETH than their deposit is worth at the true L1 rate, then bridge and redeem on L1 at the true price, extracting yield from existing rsETH holders.

## Finding Description

`CrossChainRateReceiver.getRate()` returns the stored `rate` field, which is only written in `lzReceive` when a LayerZero message arrives from the provider. There is no staleness check anywhere in the receiver:

```solidity
// CrossChainRateReceiver.sol L103-105
function getRate() external view returns (uint256) {
    return rate;
}
```

Every L2 pool's `getRate()` delegates to this receiver:

```solidity
// RSETHPoolV3.sol L234-237
function getRate() public view returns (uint256) {
    return IOracle(rsETHOracle).getRate();
}
```

The mint calculation for ETH deposits divides by this stale rate:

```solidity
// RSETHPoolV3.sol L299-307
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

For token deposits, a Chainlink oracle is also used, and its deviation band compounds the error:

```solidity
// RSETHPoolV3.sol L315-334
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

When `rsETHToETHrate` (stale, lower) is used as the denominator, the quotient is inflated. The minted wrsETH/rsETH is an OFT that can be bridged to L1 and redeemed via `LRTWithdrawalManager.instantWithdrawal` or `initiateWithdrawal`, both of which call `getExpectedAssetAmount` using the true L1 oracle price. The shortfall between what was deposited and what is redeemed is absorbed by all existing rsETH holders through dilution of the backing ratio.

No existing guard prevents this:
- `feeBps` is admin-controlled and can be 0; even at 30 bps it is easily exceeded by a 1% oracle lag.
- The daily mint limit caps per-day scale but does not prevent the attack.
- `ChainlinkOracleForRSETHPoolCollateral` checks for Chainlink-specific staleness (`answeredInRound < roundID`, `timestamp == 0`) but the rsETH cross-chain oracle has no heartbeat or staleness revert.

## Impact Explanation

**High — Theft of unclaimed yield.** The excess rsETH minted on L2 represents a claim on more ETH than was deposited. When redeemed on L1 at the true rsETH price, the deficit is borne by all existing rsETH holders through dilution of the backing ratio. This is a continuous, permissionless drain of accrued staking yield. The daily mint limit bounds per-day loss but does not eliminate it; across multiple L2 chains and pools the cumulative loss is material.

## Likelihood Explanation

The L2 oracle is structurally stale between every pair of cross-chain updates. rsETH accrues staking yield continuously, so the true rate always exceeds the last propagated rate. No special market conditions, privileged access, flash loans, or front-running are required. Any depositor can read the stale rate on-chain and deposit immediately. The Chainlink deviation threshold for collateral tokens (e.g. wstETH/ETH at 0.5%) is a publicly observable parameter that compounds the discrepancy on the token deposit path.

## Recommendation

1. **Enforce a minimum `feeBps`** that exceeds the maximum expected oracle lag plus the Chainlink deviation threshold for all supported tokens, making the attack unprofitable.
2. **Add a staleness revert** in the L2 pool's `getRate()` or in the oracle wrapper: revert if `block.timestamp - CrossChainRateReceiver.lastUpdated > heartbeatWindow`.
3. **Add a withdrawal fee** on the L1 side for rsETH redeemed within a short window after an L2 deposit, analogous to the Olympus bond discount fix.
4. **Use a TWAP or rate-limiting mechanism** on the L2 oracle to smooth sudden discrepancies and reduce the exploitable window.

## Proof of Concept

**Setup**: `RSETHPoolV3` on Arbitrum. True rsETH/ETH rate on L1 = 1.060 ETH/rsETH. Last propagated L2 oracle rate = 1.050 ETH/rsETH (stale by ~1%). Chainlink wstETH/ETH oracle at top of 0.5% deviation band: reports 1.005 ETH/wstETH. `feeBps = 0`.

**Step 1**: Attacker calls `deposit(wstETH, 1000e18, "")`.

`viewSwapRsETHAmountAndFee(1000e18, wstETH)` computes:
- `rsETHToETHrate = 1.050e18` (stale)
- `tokenToETHRate = 1.005e18` (Chainlink at top of band)
- `rsETHAmount = 1000e18 * 1.005e18 / 1.050e18 = 957.14e18` rsETH

True fair amount = `1000 * 1.000 / 1.060 = 943.40` rsETH. Excess = **13.74 rsETH**.

**Step 2**: Attacker unwraps wrsETH → rsETH, bridges to L1 via LayerZero OFT.

**Step 3**: Attacker calls `LRTWithdrawalManager.instantWithdrawal`. `getExpectedAssetAmount` uses the true L1 oracle price of 1.060 ETH/rsETH. 957.14 rsETH redeems for **1014.57 ETH**.

**Profit**: 1014.57 − 1000 = **14.57 ETH** extracted from existing rsETH holders per 1000 wstETH deposited.

**Foundry fork test plan**: Fork Arbitrum mainnet. Deploy or point to the live `RSETHPoolV3` and `CrossChainRateReceiver`. Warp time forward without triggering a LayerZero rate update. Assert `CrossChainRateReceiver.getRate()` < `ILRTOracle(l1Oracle).rsETHPrice()`. Call `deposit(wstETH, 1000e18, "")` and record `rsETHAmount`. Compute fair amount using L1 oracle. Assert `rsETHAmount > fairAmount`. Bridge rsETH to L1 fork and call `instantWithdrawal`. Assert ETH received > 1000e18.
