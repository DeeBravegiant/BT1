Audit Report

## Title
aWETH Balance in `LRTWithdrawalManager` Excluded from `getETHDistributionData()`, Understating `rsETHPrice` — (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool.getETHDistributionData()` enumerates all ETH-holding locations in the protocol but omits `LRTWithdrawalManager`. When the Aave integration is active, `LRTWithdrawalManager` converts received ETH into aWETH via `_depositToAave()`. That aWETH balance is never counted in the oracle's TVL, causing `rsETHPrice` to be understated for the entire duration the ETH sits in Aave, allowing new depositors to mint more rsETH than they are entitled to and diluting existing holders.

## Finding Description
`getETHDistributionData()` sums six ETH locations — deposit pool, NDCs, EigenLayer staked, EigenLayer unstaking, `LRTConverter`, and `LRTUnstakingVault` — but never queries `LRTWithdrawalManager`: [1](#0-0) 

The ETH flow that creates the gap is triggered by `unlockQueue()`. It calls `unstakingVault.redeem()` to pull ETH from `LRTUnstakingVault` into `LRTWithdrawalManager`, then immediately calls `_depositToAave()` when `isAaveIntegrationEnabled` is true: [2](#0-1) 

After this, `lrtUnstakingVault.balance` has decreased (ETH left the vault) and the aWETH now held by `LRTWithdrawalManager` is invisible to the accounting: [3](#0-2) [4](#0-3) 

`getTotalAssetDeposits()` sums only the six values returned by `getETHDistributionData()`: [5](#0-4) 

`_getTotalEthInProtocol()` in `LRTOracle` calls `getTotalAssetDeposits()` for each supported asset: [6](#0-5) 

And `_updateRsETHPrice()` uses that total to compute `rsETHPrice`: [7](#0-6) 

There are no existing guards that compensate for the missing aWETH balance. The `getAaveBalance()` function exists and is callable, but is never referenced in the deposit pool's distribution accounting.

## Impact Explanation
When ETH is deposited to Aave by `LRTWithdrawalManager`, `_getTotalEthInProtocol()` returns a value lower than the true protocol TVL. The resulting `rsETHPrice` is understated. Because `getRsETHAmountToMint = amount × assetPrice / rsETHPrice`, new depositors receive more rsETH per ETH than the true exchange rate entitles them to, diluting the value held by existing rsETH holders. The mis-accounting persists for as long as ETH remains in Aave (which can span the entire withdrawal queue processing period).

**Impact class**: Low — contract fails to deliver promised returns to existing rsETH holders; their share value is diluted by new mints priced against an understated TVL.

## Likelihood Explanation
The Aave integration is a deployed, configurable feature (`isAaveIntegrationEnabled`). The `unlockQueue()` function is callable by any address holding `ASSET_TRANSFER_ROLE` or `LRT_OPERATOR_ROLE`, which are operational roles used in normal protocol operation. No special attacker capability is required — the gap is created as a side effect of normal withdrawal processing. The gap is proportional to the total ETH committed to pending withdrawals that has been deposited to Aave, and can persist for days (the default `withdrawalDelayBlocks` is 8 days).

## Recommendation
Include the aWETH balance (and any idle ETH) of `LRTWithdrawalManager` in `getETHDistributionData()`. Add a return variable and a corresponding term:

```solidity
address lrtWithdrawalManager = lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER);
ethLyingInWithdrawalManager = ILRTWithdrawalManager(lrtWithdrawalManager).getAaveBalance()
    + lrtWithdrawalManager.balance;
```

Include `ethLyingInWithdrawalManager` in the return tuple of `getETHDistributionData()` and add it to the sum in `getTotalAssetDeposits()`.

## Proof of Concept
1. Deploy with Aave integration enabled (`isAaveIntegrationEnabled = true`).
2. Users initiate ETH withdrawal requests; `assetsCommitted[ETH]` grows to 500 ETH.
3. Operator calls `unlockQueue(ETH, ...)`. Internally: `unstakingVault.redeem(ETH, 500e18)` moves 500 ETH to `LRTWithdrawalManager`; then `depositToAaveExternal(500e18)` converts it to 500 aWETH.
4. `lrtUnstakingVault.balance` is now 500 ETH lower; `LRTWithdrawalManager` holds 500 aWETH.
5. Call `updateRSETHPrice()`. `getETHDistributionData()` returns a total 500 ETH less than the true TVL (aWETH is invisible). `rsETHPrice = (trueTotal - 500e18) / rsETHSupply` — understated.
6. A new depositor calls `depositETH(1 ether)`. `getRsETHAmountToMint = 1e18 × 1e18 / understatedPrice` — they receive more rsETH than the true exchange rate entitles them to.
7. Existing holders' rsETH is diluted; the protocol fee is computed on a deflated base.

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
