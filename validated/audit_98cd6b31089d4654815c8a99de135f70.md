Audit Report

## Title
ETH Deposit Limit Check Omits New Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: the ETH branch tests only whether the running total already exceeds the cap, while the ERC20 branch correctly adds the incoming `amount` before comparing. Any unprivileged depositor can push ETH holdings above the configured deposit limit in a single transaction.

## Finding Description
In `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682), the ETH branch at line 679 evaluates `totalAssetDeposits > depositLimit`, omitting `amount`. The ERC20 branch at line 681 correctly evaluates `totalAssetDeposits + amount > depositLimit`. Because `_beforeDeposit` (line 661) only reverts when this function returns `true`, any ETH deposit that does not push the *pre-deposit* total past the cap is accepted unconditionally, regardless of how large `msg.value` is. The public entry point `depositETH` (lines 76–93) passes `msg.value` as `depositAmount` to `_beforeDeposit`, making this reachable by any caller with no special role.

## Impact Explanation
The deposit limit is a protocol-level risk control governing maximum ETH exposure in EigenLayer strategies. Bypassing it causes the protocol to accumulate ETH beyond governance-approved caps. No deposited funds are lost or frozen. This matches the allowed impact: **Low — Contract fails to deliver promised returns (deposit cap enforcement), but doesn't lose value.**

## Likelihood Explanation
`depositETH` is a public, payable function gated only by `whenNotPaused` and `onlySupportedAsset`. No privileged role is required. The exploitable condition — current total at or below the cap — is the normal operating state of the protocol. Any depositor can trigger this repeatedly.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 path:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets ETH deposit limit to 100,000 ETH via `LRTConfig.updateAssetDepositLimit`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns 99,999 ETH.
3. Attacker calls `depositETH{value: 2000 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `99,999 > 100,000` → `false`; no revert.
5. 2,000 ETH is accepted; total becomes 101,999 ETH — 1,999 ETH above the cap.
6. **Foundry test plan:** Deploy `LRTDepositPool` with a mock config setting ETH limit to 100,000 ether; seed the pool with 99,999 ether of prior deposits; call `depositETH{value: 2000 ether}` from an unprivileged address; assert the call succeeds and `getTotalAssetDeposits(ETH_TOKEN)` exceeds the configured limit.
