Audit Report

## Title
Missing `msg.sender == account` Authorization in `MerkleDistributor.claim()` Allows Forced Claims with Involuntary Fee Deduction - (File: `contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary
`MerkleDistributor.claim()` accepts a caller-supplied `account` parameter but never verifies that `msg.sender == account`. Any external caller with a valid Merkle proof (publicly derivable) can trigger a claim on behalf of any account. When `feeInBPS > 0`, this permanently reduces the victim's claimable yield by up to 10%, with the fee irrecoverably sent to `protocolTreasury`.

## Finding Description
In `contracts/utils/MerkleDistributor/MerkleDistributor.sol`, `claim()` at L97–147 accepts `account` as a caller-supplied parameter and verifies the Merkle proof against `(index, account, cumulativeAmount)`. There is no check that `msg.sender == account`. [1](#0-0) 

After proof verification, the function computes a fee of up to 10% and transfers `claimableAmount - fee` to `account` and `fee` to `protocolTreasury`: [2](#0-1) 

The state update at L134–135 marks the claim as consumed, so the victim can never reclaim the fee portion: [3](#0-2) 

By contrast, `KernelMerkleDistributor._processClaim()` explicitly enforces the identity check at L311–313: [4](#0-3) 

`MerkleDistributor` has no equivalent guard. The `MAX_FEE_IN_BPS` constant confirms the fee can reach 10%: [5](#0-4) 

## Impact Explanation
**High — Theft of unclaimed yield.** When `feeInBPS > 0`, an attacker who forces a claim on behalf of a victim causes the victim to permanently lose `claimableAmount * feeInBPS / 10_000` tokens. These tokens are irrecoverably transferred to `protocolTreasury` without the victim's consent. The victim's entitled distribution is permanently reduced. This is a direct, concrete, irreversible loss of yield belonging to the victim. [2](#0-1) 

## Likelihood Explanation
- Merkle proofs and claim parameters are either emitted in events (`Claimed`) or derivable from off-chain distribution data, making them publicly observable.
- Any unprivileged external account can call `claim()` with a victim's valid parameters — no special role, key, or oracle access is required.
- `feeInBPS` is configurable up to 1000 BPS (10%) by the owner via `setFeeInBPS()`. When non-zero, every forced claim extracts yield from the victim.
- The attack is repeatable across every new Merkle root index published by the owner. [6](#0-5) 

## Recommendation
Add a caller authorization check at the top of `claim()`, before Merkle proof verification, mirroring the pattern already established in `KernelMerkleDistributor._processClaim()`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
``` [4](#0-3) 

## Proof of Concept
1. Owner sets `feeInBPS = 1000` (10%) and publishes a new Merkle root via `setMerkleRoot()`.
2. Victim (`0xVICTIM`) has a valid leaf `(index=1, account=0xVICTIM, cumulativeAmount=1000e18)` and is waiting for the fee to be reduced before claiming.
3. Attacker derives the Merkle proof from off-chain distribution data or on-chain events.
4. Attacker calls:
   ```solidity
   merkleDistributor.claim(1, 0xVICTIM, 1000e18, victimProof);
   ```
5. Contract verifies the proof (valid), computes `fee = 100e18`, transfers `900e18` to `0xVICTIM` and `100e18` to `protocolTreasury`.
6. `userClaims[0xVICTIM].lastClaimedIndex` is updated to `1`; the victim can never reclaim the 100e18 fee extracted without their consent.

**Foundry test plan:** Deploy `MerkleDistributor`, initialize with `feeInBPS = 1000`, construct a Merkle tree with a single leaf for `0xVICTIM`, call `setMerkleRoot()`, then call `claim()` from an attacker address with `0xVICTIM`'s valid proof. Assert that `0xVICTIM` received `900e18` and `protocolTreasury` received `100e18`, and that a subsequent call to `claim()` by `0xVICTIM` reverts with `AlreadyClaimed`. [7](#0-6)
