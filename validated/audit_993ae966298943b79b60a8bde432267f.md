### Title
Sub-transaction Reference Scripts Excluded from Fee Calculation and Block-Level Size Limit in Dijkstra Era — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs`, `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the minimum fee calculation and the block-level reference script size limit both use `txNonDistinctRefScriptsSize`, which only counts the **top-level transaction's** reference scripts. Sub-transactions (`SubTx`) embedded in a top-level transaction can freely use reference scripts without those scripts being charged in the fee or counted against the block-level cap. The per-transaction limit check (`validateAllRefScriptSize`) correctly uses `batchNonDistinctRefScriptsSize` — which includes sub-transactions — but the fee and block-level checks do not. This asymmetry allows an attacker to submit transactions that are significantly more expensive to validate than the fee and block-level limit imply.

---

### Finding Description

The Dijkstra era introduces sub-transactions (`SubTx`) embedded inside a top-level transaction. Sub-transactions do not pay fees directly; the top-level transaction's fee is intended to cover all validation costs. Three distinct resource-accounting paths exist:

**1. Fee calculation** — `getConwayMinFeeTxUtxo` is reused unchanged for Dijkstra:

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

`txNonDistinctRefScriptsSize` only inspects `inputsTxBodyL` and `referenceInputsTxBodyL` of the **top-level** transaction body. Sub-transactions' inputs are never examined. [1](#0-0) 

**2. Per-transaction limit check** — `validateAllRefScriptSize` in the Dijkstra LEDGER rule uses `batchNonDistinctRefScriptsSize`, which **does** include sub-transactions:

```haskell
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum (foldMap' (Sum . txNonDistinctRefScriptsSize utxo)
                       (tx ^. bodyTxL . subTransactionsTxBodyL))
``` [2](#0-1) [3](#0-2) 

**3. Block-level limit check** — The Dijkstra BBODY rule delegates directly to the Conway implementation:

```haskell
Conway.validateBodyRefScriptsSizeTooBig @era pp blockBody (ls ^. utxoL)
``` [4](#0-3) 

`validateBodyRefScriptsSizeTooBig` calls `totalRefScriptSizeInBlock`, which iterates over top-level transactions and calls `txNonDistinctRefScriptsSize` for each — again, only counting the top-level transaction's reference scripts:

```haskell
accum (!accUtxo, !accSum) tx =
  ...
  (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
``` [5](#0-4) 

The result is a structural asymmetry: the per-tx limit (path 2) accounts for sub-transaction reference scripts, but the fee (path 1) and block-level limit (path 3) do not.

A secondary compounding issue: `batchNonDistinctRefScriptsSize` passes the **original** pre-sub-transaction UTxO to every sub-transaction's size computation. If sub-transaction A creates a UTxO output carrying a reference script, and sub-transaction B references it, that script is invisible to the size calculation — so even the per-tx limit underestimates the true deserialization cost. [6](#0-5) 

---

### Impact Explanation

**Fee manipulation (Medium):** An attacker crafts a top-level transaction whose body references no scripts, but whose sub-transactions each reference large Plutus scripts. The `minFeeRefScriptCostPerByte` surcharge (15 lovelace/byte in current parameters) is never applied to sub-transaction reference scripts. With `maxRefScriptSizePerTx = 200 KiB`, the attacker avoids up to `200 × 1024 × 15 = 3,072,000 lovelace ≈ 3 ADA` in fees per transaction while imposing the full deserialization cost on validating nodes.

**Block-level limit bypass (Medium → approaching High):** The block-level cap `maxRefScriptSizePerBlock = 1 MiB` is designed to bound the total script deserialization work per block. Because sub-transaction reference scripts are invisible to `totalRefScriptSizeInBlock`, a block producer can fill a block with transactions each carrying sub-transactions that collectively use up to `maxRefScriptSizePerTx` bytes of reference scripts. With `maxRefScriptSizePerTx = 200 KiB` and a block that fits, say, 10 such transactions, the actual reference script deserialization burden is 2 MiB — double the intended cap — while the check reports 0 bytes. This matches the allowed impact: *"attacker-controlled transactions exceed intended validation limits."*

---

### Likelihood Explanation

Any unprivileged transaction sender can construct a Dijkstra-era top-level transaction with sub-transactions. No special privileges, governance majority, or key compromise is required. The attacker only needs to:
1. Produce UTxO outputs containing large reference scripts (standard transactions).
2. Embed sub-transactions that reference those outputs.

The attack is deterministic and reproducible.

---

### Recommendation

1. **Fee calculation**: Introduce a Dijkstra-specific `getMinFeeTxUtxo` that replaces `txNonDistinctRefScriptsSize utxo tx` with `batchNonDistinctRefScriptsSize utxo tx`, so sub-transaction reference scripts are included in the fee.

2. **Block-level limit**: Introduce a Dijkstra-specific `totalRefScriptSizeInBlock` (or override `validateBodyRefScriptsSizeTooBig`) that, for each top-level transaction, uses `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`.

3. **Per-tx limit with evolving UTxO**: Fix `batchNonDistinctRefScriptsSize` to thread the evolving UTxO through sub-transaction size computations (similar to how `totalRefScriptSizeInBlock` threads `accUtxo` across transactions in a block), so reference scripts created by earlier sub-transactions and consumed by later ones are counted.

### Proof of Concept

```
Setup:
  - Deploy a UTxO output O1 carrying a 100 KiB Plutus V3 reference script S.
  - maxRefScriptSizePerTx = 200 KiB, maxRefScriptSizePerBlock = 1 MiB.
  - minFeeRefScriptCostPerByte = 15 lovelace/byte.

Attack transaction T (top-level):
  - Top-level body: no reference inputs, no script inputs.
  - Sub-transaction ST1: referenceInputs = {O1}  → uses S (100 KiB)
  - Sub-transaction ST2: referenceInputs = {O1}  → uses S (100 KiB)
  - Total sub-tx ref script size = 200 KiB (within per-tx limit).

Fee charged:
  txNonDistinctRefScriptsSize(utxo, T) = 0 bytes (top-level has no ref inputs)
  → refScriptsFee = 0 lovelace
  Expected fee surcharge = 200 * 1024 * 15 = 3,072,000 lovelace → NOT charged.

Block with 10 such transactions:
  totalRefScriptSizeInBlock = 0 bytes (all top-level txs have 0 ref scripts)
  Actual deserialization burden = 10 × 200 KiB = 2 MiB > maxRefScriptSizePerBlock (1 MiB)
  → Block-level limit bypassed.

-- scripts will be counted as many times as they occur, since there is never a reason to
-- include an input with the same reference script.
--
-- Any input that appears in both regular inputs and reference inputs of a transaction is
-- only used once in this computation.
txNonDistinctRefScriptsSize :: (EraTx era, BabbageEraTxBody era) => UTxO era -> Tx l era -> Int
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where

totalRefScriptSizeInBlock ::
  (AlonzoEraTx era, BabbageEraTxBody era) => ProtVer -> StrictSeq (Tx TopTx era) -> UTxO era -> Int
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 =
      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      let updatedUtxo = accUtxo `Map.union` unUTxO toAdd
          toAdd
            | IsValid True <- tx ^. isValidTxL = txouts $ tx ^. bodyTxL
            | otherwise = collOuts $ tx ^. bodyTxL
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
