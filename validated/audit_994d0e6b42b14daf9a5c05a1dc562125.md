### Title
Validator Inserts Proposal into `valid_proposals` Before Verifying `ProposalFin` Commitment Match — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

In `validate_proposal`, the validator unconditionally inserts the received proposal (including its full transaction content and `FinishedProposalInfo`) into the shared `valid_proposals` map **before** checking whether the batcher-computed block commitment matches the network-received `ProposalFin.proposal_commitment`. A dishonest proposer can therefore cause the validator to permanently store a proposal under a commitment value that the validator itself computed — even when the proposer's claimed commitment is wrong — and that stored entry is later used verbatim by `decision_reached` and `repropose`.

### Finding Description

In `validate_proposal` the ordering of operations is:

1. Receive `ProposalFin` from the network.
2. Call `batcher.finish_proposal(...)` → get `batcher_block_commitment` (the locally-computed commitment).
3. **Insert** the proposal into `valid_proposals` under `batcher_block_commitment`.
4. **Then** compare `batcher_block_commitment` with `received_fin.proposal_commitment`.
5. If they differ, return `Err(ValidateProposalError::ProposalFinMismatch)`. [1](#0-0) 

The comment at line 238 explicitly acknowledges this ordering is intentional ("Update valid_proposals before sending fin to avoid a race condition with `repropose`"), but the consequence is that a mismatched proposal is **permanently stored** in `valid_proposals` before the mismatch is detected and the error is returned.

The `valid_proposals` map is keyed by `(height, round)` and the stored commitment is the **batcher-computed** value, not the proposer-supplied one. The `insert_proposal` function asserts that no prior entry exists for that `(height, round)`: [2](#0-1) 

Once inserted, the entry is never removed on `ProposalFinMismatch` — the error path simply returns without cleanup. The `valid_proposals` map is only pruned in `decision_reached` via `remove_proposals_below_or_at_height`.

### Impact Explanation

**Wrong state / receipt / block commitment used for an accepted block (High).**

The analog to the Router bug is: mid-execution (mid-validation), the validator's shared state (`valid_proposals`) is mutated with attacker-influenced data before the integrity check completes. Specifically:

- After `ProposalFinMismatch` is returned, the `valid_proposals` map for `(height, round)` contains the batcher-computed commitment paired with the full transaction content of the mismatched proposal.
- If consensus subsequently calls `repropose` for the same `(height, round)` — which is possible in multi-round BFT when `valid_round` is set — `update_for_reproposal` will find and use this stale/poisoned entry.
- `decision_reached` for the same `(height, round)` will also find this entry and commit the wrong block.

The corrupted value is the `ProposalContent` stored at `(height, round)` in `valid_proposals`, which includes the `ProposalInit` (with gas prices, L1 DA mode, timestamp), the full transaction list, and `FinishedProposalInfo` (partial block hash, commitments, L2 gas used). All of these feed directly into `finalize_decision` → `state_sync_client.add_new_block` → committed block state. [3](#0-2) 

### Likelihood Explanation

**Low-to-Medium.** The trigger requires:
1. A Byzantine proposer that sends a valid transaction stream (passes all per-transaction checks) but a mismatched `ProposalFin.proposal_commitment`.
2. The same `(height, round)` to be subsequently used in a `repropose` or `decision_reached` call — which requires the BFT state machine to have locked on that round's value before the mismatch was detected.

The proposer is authenticated (checked in `handle_proposal`), so this requires a Byzantine validator-set member, not an arbitrary peer. However, the `ProposalFinMismatch` error is treated as a non-fatal event (it increments a metric and returns an error, but does not panic or clear state), making the stale entry persistent. [4](#0-3) 

### Recommendation

Move `valid_proposals.insert_proposal(...)` to **after** the `ProposalFinMismatch` check:

```rust
// TODO(matan): Switch to signature validation.
if built_block != received_fin.proposal_commitment {
    CONSENSUS_PROPOSAL_FIN_MISMATCH.increment(1);
    return Err(ValidateProposalError::ProposalFinMismatch);
}

// Only insert after commitment is verified.
let mut valid_proposals = args.valid_proposals.lock().unwrap();
valid_proposals.insert_proposal(args.init, content, &args.proposal_id, finished_info);

Ok(built_block)
```

The comment about the race condition with `repropose` is only relevant when the proposal is **valid** — a mismatched proposal should never be reproposed. Moving the insert after the check eliminates the window where a stale entry can be used by `repropose` or `decision_reached`.

### Proof of Concept

```
1. Byzantine proposer P streams N valid transactions to validator V for height H, round R.
2. V calls batcher.finish_proposal() → batcher_commitment = C_batcher.
3. V inserts (H, R) → (C_batcher, txs, finished_info) into valid_proposals.
4. P sends ProposalFin { proposal_commitment: C_wrong ≠ C_batcher, ... }.
5. V detects mismatch, returns ProposalFinMismatch — but (H, R) entry remains in valid_proposals.
6. BFT state machine had already locked on (H, R) in a prior round (valid_round = R).
7. On round R+1, state machine emits SMRequest::Repropose(proposal_id, BuildParam { valid_round: Some(R), ... }).
8. repropose() calls valid_proposals.update_for_reproposal(&H, &C_batcher, &build_param).
9. The stale entry is found and re-streamed as a new proposal for round R+1.
10. Other validators accept this reproposal (it has valid transactions and a valid batcher commitment).
11. decision_reached(H, R+1, C_batcher) commits the block using the content from the poisoned entry.
```

The corrupted output is the committed block at height H: its transaction set, state diff, receipts, events, and block hash are all derived from the stale `valid_proposals` entry that was inserted before the `ProposalFinMismatch` check. [5](#0-4) [6](#0-5) [7](#0-6)

    pub(crate) fn insert_proposal(
        &mut self,
        init: ProposalInit,
        transactions: Vec<Vec<InternalConsensusTransaction>>,
        proposal_id: &ProposalId,
        finished_info: FinishedProposalInfo,
    ) {
        let proposal_commitment = proposal_commitment_from(
            finished_info.proposal_commitment.partial_block_hash,
            init.fee_proposal_fri,
        );

        let height = init.height;
        let round = init.round;
        let by_round = self.data.entry(height).or_default();
        let previous = by_round.insert(
            round,
            (proposal_commitment, (init, transactions, *proposal_id, finished_info)),
        );
        assert!(
            previous.is_none(),
            "Overwriting existing proposal for height {height} round {round}; at most one \
             proposal per (height, round) is allowed"
        );
    }

    fn update_for_reproposal(
        &mut self,
        height: &BlockNumber,
        proposal_commitment: &ProposalCommitment,
        build_param: &BuildParam,
    ) -> (ProposalInit, Vec<Vec<InternalConsensusTransaction>>, FinishedProposalInfo) {
        let lookup_round = build_param.valid_round.expect("Valid round must be set for reproposal");
        let (mut init, transactions, proposal_id, finished_info) =
            self.get_proposal(height, &lookup_round, proposal_commitment).clone();
        init.round = build_param.round;
        init.proposer = build_param.proposer;
        init.valid_round = build_param.valid_round;
        self.insert_proposal(
            init.clone(),
            transactions.clone(),
            &proposal_id,
            finished_info.clone(),
        );
        (init, transactions, finished_info)
    }
