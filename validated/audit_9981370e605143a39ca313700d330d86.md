Audit Report

## Title
Missing Chainlink Staleness Check Allows Stale Price Exploitation — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

## Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` omits a time-based staleness guard on the Chainlink `latestRoundData()` response. An attacker who performs block stuffing on Unichain to suppress Chainlink heartbeat updates can cause `getRate()` to return a stale, inflated price that passes all three existing checks. `RSETHPoolNoWrapper.deposit(token, amount, referralId)` consumes this price directly to compute `rsETHAmount`, allowing the attacker to drain rsETH from pool reserves in excess of the fair value of deposited collateral.

## Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 26–37) applies exactly three guards after calling `latestRoundData()`:

```solidity
if (answeredInRound < roundID) revert StalePrice();   // round completeness
if (timestamp == 0) revert IncompleteRound();          // round started
if (ethPrice <= 0) revert InvalidPrice();              // positive price
```

None of these checks whether the returned price is recent. A completed round with a non-zero timestamp and a positive price passes all three checks regardless of age. The standard guard `require(block.timestamp - updatedAt <= heartbeat)` is entirely absent. [1](#0-0) 

`RSETHPoolNoWrapper` is explicitly documented as the pool for Unichain: [2](#0-1) 

`deposit(token, amount, referralId)` calls `viewSwapRsETHAmountAndFee(amount, token)`, which fetches `tokenToETHRate` directly from the collateral oracle and computes `rsETHAmount`: [3](#0-2) 

The computed `rsETHAmount` is then transferred directly from the pool's rsETH reserves to the caller: [4](#0-3) 

**Exploit flow:**
1. Attacker stuffs blocks on Unichain for the Chainlink heartbeat window (~24 hours for wstETH/ETH), preventing oracle updates.
2. `getRate()` continues returning the last recorded price, which passes all three existing checks.
3. If the market price of the collateral has declined during the stuffing window, the stale price is inflated relative to fair value.
4. Attacker calls `deposit(wstETH, amount, referralId)` — `rsETHAmount` is computed using the stale inflated `tokenToETHRate`, yielding more rsETH than the deposited collateral is worth at current market price.
5. The pool's rsETH balance is depleted beyond what the deposited collateral backs.

## Impact Explanation

**Critical. Direct theft of any user funds.**

The attacker extracts rsETH from the pool's reserves proportional to the price deviation between the stale oracle price and the current fair market price. Every rsETH transferred in excess of fair value represents a direct, irreversible theft from the pool's rsETH reserves — funds held on behalf of the protocol and its users. The pool's invariant that every rsETH in reserves is fully collateralized is violated. The submission self-scoped this as "Low. Block stuffing," but the concrete outcome is theft of funds from pool reserves, which maps to "Critical. Direct theft of any user funds."

## Likelihood Explanation

Block stuffing on Unichain (~1-second block times, ~86,400 blocks per 24 hours) is expensive but feasible for a well-capitalized attacker when the profit from the stale-price arbitrage exceeds the cost of filling blocks for the heartbeat window. The wstETH/ETH feed's 24-hour heartbeat sets a high but not prohibitive bar. Additionally, the missing staleness check also exposes the protocol to natural oracle staleness (Chainlink node outages, network congestion) without any attacker involvement, requiring no block stuffing at all.

## Recommendation

Add a configurable heartbeat parameter to `ChainlinkOracleForRSETHPoolCollateral` and enforce it in `getRate()`:

```solidity
uint256 public immutable heartbeat;

constructor(address _oracle, uint256 _heartbeat) {
    oracle = _oracle;
    heartbeat = _heartbeat;
}

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    if (block.timestamp - timestamp > heartbeat) revert StalePrice(); // ADD THIS

    uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
    return normalizedPrice;
}
```

Set `heartbeat` to match the Chainlink feed's documented heartbeat (e.g., `86400` for the wstETH/ETH 24-hour feed).

## Proof of Concept

Foundry fork test on Unichain (or local fork simulating Unichain block times):

1. Deploy or fork `ChainlinkOracleForRSETHPoolCollateral` and `RSETHPoolNoWrapper` with wstETH as a supported token.
2. Record `IChainlinkOracle(ORACLE).getRate()` — call it `stalePriceBefore`.
3. `vm.warp(block.timestamp + 23 hours + 59 minutes)` to simulate the block stuffing window without a Chainlink update.
4. Call `IChainlinkOracle(ORACLE).getRate()` again — it returns the same value and does not revert, confirming the missing staleness check.
5. `deal(WSTETH, attacker, 1e18)` and approve the pool.
6. Call `IPool(POOL).deposit(WSTETH, 1e18, "poc")`.
7. Assert that `rsETHReceived > fairRsETHAmount` where `fairRsETHAmount` is computed using the current (lower) market price of wstETH/ETH, confirming excess rsETH was extracted from pool reserves.
