Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Blocks All Withdrawals Post-Deployment - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.sol`'s `initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal()` evaluates `0 >= 0` as `true` on the very first call and reverts with `WithdrawalLimitReached()`, blocking every staker from withdrawing until an admin separately calls `setMaxNumberOfWithdrawalsPerUser`. A secondary uninitialized variable, `withdrawalDelay`, also defaults to `0`, removing the intended time-lock once the admin does unblock withdrawals.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared at line 108 but never set inside `initialize()` (lines 259–271):

```solidity
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ...
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
    // maxNumberOfWithdrawalsPerUser never assigned → remains 0
}
```

`initiateWithdrawal()` at line 323 enforces:

```solidity
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

For a fresh staker, `userWithdrawalIds[msg.sender].length == 0` and `maxNumberOfWithdrawalsPerUser == 0`, so `0 >= 0` is always `true`. Every call reverts before any state is modified. The admin setter `setMaxNumberOfWithdrawalsPerUser` (line 614) explicitly rejects `0` as an input, confirming the protocol's intent that this value must be non-zero at all times — yet the initializer never enforces this invariant.

`withdrawalDelay` suffers the same omission: it defaults to `0`, so once the admin unblocks withdrawals, all initiated withdrawals are immediately claimable with no enforced delay.

## Impact Explanation
Every staker who calls `initiateWithdrawal()` after deployment is blocked until an admin intervenes. Their KERNEL tokens remain locked in the contract with no exit path during this window. This is a **Medium — Temporary freezing of funds**: the freeze is not permanent (admin can resolve it), but it affects 100% of stakers from the moment of deployment and requires no attacker action whatsoever.

## Likelihood Explanation
The condition is active from block 0 of the contract's life. Any user who stakes KERNEL tokens and attempts the core withdrawal flow triggers it immediately. No attacker sophistication, no special permissions, and no external conditions are required. The likelihood is effectively certain for any user who interacts with the contract before the admin configures the missing values.

## Recommendation
Initialize both `maxNumberOfWithdrawalsPerUser` and `withdrawalDelay` to validated non-zero values inside `initialize()`, adding them as constructor parameters with the same bounds checks used in their respective setters:

```solidity
function initialize(
    address _admin,
    address _kernelToken,
    address _rewardToken,
    uint256 _withdrawalDelay,
    uint256 _maxNumberOfWithdrawalsPerUser
) external initializer {
    // existing zero-address checks ...
    if (_withdrawalDelay == 0 || _withdrawalDelay > MAX_WITHDRAWAL_DELAY) revert InvalidWithdrawalDelay();
    if (_maxNumberOfWithdrawalsPerUser == 0 || _maxNumberOfWithdrawalsPerUser > MAX_WITHDRAWALS_PER_USER)
        revert InvalidMaxNumberOfWithdrawalsPerUser();

    withdrawalDelay = _withdrawalDelay;
    maxNumberOfWithdrawalsPerUser = _maxNumberOfWithdrawalsPerUser;
    // rest of existing init ...
}
```

## Proof of Concept
```solidity
function test_WithdrawalBlocked_UninitializedMax() public {
    // Contract freshly initialized; maxNumberOfWithdrawalsPerUser == 0
    vm.startPrank(user);
    kernelToken.approve(address(kernelDepositPool), 1 ether);
    kernelDepositPool.stake(1 ether);

    // userWithdrawalIds[user].length == 0, maxNumberOfWithdrawalsPerUser == 0
    // 0 >= 0 → true → revert WithdrawalLimitReached
    vm.expectRevert(KernelDepositPool.WithdrawalLimitReached.selector);
    kernelDepositPool.initiateWithdrawal(1 ether);
    // 1 ether is now locked; user has no exit path
    vm.stopPrank();
}
```

The revert is deterministic and reproducible on any local fork immediately after deployment without any admin configuration.
