Audit Report

## Title
Chainlink `minAnswer` Circuit Breaker Not Validated in `getAssetPrice()`, Enabling rsETH Over-Minting During LST Depeg — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and uses the returned price without checking it against the Chainlink aggregator's hardcoded `minAnswer` floor. If a supported LST depegs below `minAnswer`, the oracle silently returns the clamped floor price. Because `rsETHPrice` is a stored state variable updated only on explicit `updateRSETHPrice()` calls, an attacker can deposit the depegged asset in the window between the depeg and the next price update, receiving rsETH priced against the stale (pre-depeg) `rsETHPrice` denominator but the inflated `minAnswer` numerator, yielding excess rsETH redeemable for healthy collateral.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the price with no bounds validation:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (, int256 price,,,) = priceFeed.latestRoundData();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
``` [1](#0-0) 

Chainlink aggregators have a hardcoded `minAnswer`; when the true market price falls below it, `latestRoundData()` returns `minAnswer` rather than the actual price. No check is made against this bound.

This price flows into `LRTOracle.getAssetPrice()`:

```solidity
// contracts/LRTOracle.sol L156-158
function getAssetPrice(address asset) public view onlySupportedOracle(asset) returns (uint256) {
    return IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset);
}
``` [2](#0-1) 

Which is consumed by `_getTotalEthInProtocol()` and ultimately by `getRsETHAmountToMint()`:

```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [3](#0-2) 

`rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is explicitly called: [4](#0-3) 

The critical asymmetry: `lrtOracle.getAssetPrice(asset)` in the numerator is read **live** from the oracle (returning the inflated `minAnswer`), while `lrtOracle.rsETHPrice()` in the denominator is the **stale stored value** from before the depeg. An attacker who deposits the depegged asset before `updateRSETHPrice()` is called receives rsETH computed at the inflated numerator price against the pre-depeg denominator.

The downside protection in `_updateRsETHPrice()` (pausing when `newRsETHPrice < highestRsethPrice` by more than `pricePercentageLimit`) only activates when `updateRSETHPrice()` is called — it provides no protection during the attack window itself: [5](#0-4) 

## Impact Explanation

**Critical — Direct theft of user funds.** An attacker deposits a depegged LST at the inflated `minAnswer` price (before `rsETHPrice` is updated), receives excess rsETH, then redeems or sells that rsETH for healthy assets held in the pool. The loss is borne by all existing rsETH holders whose share of the pool is diluted. This matches the allowed Critical impact: "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation

**Medium.** LST depeg events are rare but have occurred historically (stETH in 2022). The protocol holds multiple LSTs (stETH, rETH, ETH-X, swETH, sfrxETH), each with its own Chainlink feed and `minAnswer`. The attack window is the block interval between the depeg and the next `updateRSETHPrice()` call. Although `updateRSETHPrice()` is publicly callable (no access control), the attacker can act in the same block as the depeg before any keeper or MEV bot updates the price. The SECURITY.md exclusion for "depegging of an external stablecoin" does not apply here — LSTs are not stablecoins. The "incorrect data supplied by third-party oracles" exclusion also does not apply because the `minAnswer` behavior is a documented, known Chainlink limitation that the protocol is responsible for validating; the root cause is the missing bounds check in the protocol's own code.

## Recommendation

In `ChainlinkPriceOracle.getAssetPrice()`, retrieve the aggregator's `minAnswer` and `maxAnswer` from the underlying `IAccessControlledOffchainAggregator` and revert if the returned price is at or outside those bounds:

```solidity
IAccessControlledOffchainAggregator aggregator =
    IAccessControlledOffchainAggregator(priceFeed.aggregator());
int192 minAnswer = aggregator.minAnswer();
int192 maxAnswer = aggregator.maxAnswer();
if (price <= minAnswer || price >= maxAnswer) revert PriceOutOfBounds();
```

Additionally, add staleness checks (`updatedAt` vs. `block.timestamp`) and validate that `answeredInRound >= roundId`.

## Proof of Concept

1. Assume stETH is a supported asset with a Chainlink feed whose `minAnswer` = `0.9e18` (90% of ETH). Current `rsETHPrice` = `1.0e18`.
2. A black-swan event causes stETH's true market price to drop to `0.5e18`.
3. Chainlink's aggregator clamps the return value to `minAnswer` = `0.9e18`.
4. `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `0.9e18`.
5. `rsETHPrice` is still `1.0e18` (not yet updated — `updateRSETHPrice()` has not been called).
6. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, ...)` in the same block as the depeg.
7. `getRsETHAmountToMint` computes: `1000e18 * 0.9e18 / 1.0e18 = 900 rsETH`.
8. True value of 1000 stETH = 500 ETH; attacker receives 900 rsETH worth ~900 ETH at the stale price.
9. Attacker redeems 900 rsETH for ~900 ETH worth of healthy assets, netting ~400 ETH profit at the expense of existing depositors.

**Foundry fork test plan:** Fork mainnet, mock the stETH/ETH Chainlink aggregator to return `minAnswer` while setting the true market price below it, call `depositAsset` before `updateRSETHPrice()`, and assert that the minted rsETH exceeds the fair-value equivalent of the deposited stETH.
