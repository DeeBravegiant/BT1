Audit Report

## Title
Stale L2 Exchange Rate Causes Structural wrsETH Over-Minting and RsETHTokenWrapper Undercollateralization - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
`RSETHPoolV3.deposit()` mints `wrsETH` to users using the stale rate stored in `CrossChainRateReceiver`, while `L1Vault.depositETHForL1VaultETH()` later mints rsETH on L1 using the live `lrtOracle.rsETHPrice()`. Because rsETH is a continuously appreciating yield-bearing token, the live L1 price is always higher than the stale L2 rate, causing L1 to produce fewer rsETH tokens than the wrsETH already minted on L2. The resulting shortfall is deposited into `RsETHTokenWrapper`, leaving it chronically undercollateralized and causing `_withdraw()` to revert for the last redeemers.

## Finding Description

**Root cause:** `RSETHPoolV3.viewSwapRsETHAmountAndFee()` computes the wrsETH amount as:

```solidity
uint256 rsETHToETHrate = getRate();   // CrossChainRateReceiver.rate — stale
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```
(`contracts/pools/RSETHPoolV3.sol`, lines 304–307)

`getRate()` delegates to `IOracle(rsETHOracle).getRate()`, which returns `CrossChainRateReceiver.rate` — the last value pushed from L1 via LayerZero. There is no staleness check; the value is updated only when a LayerZero message arrives:

```solidity
uint256 public rate;
uint256 public lastUpdated;
```
(`contracts/cross-chain/CrossChainRateReceiver.sol`, lines 13–16)

**Exploit flow:**

1. User calls `RSETHPoolV3.deposit()`. `wrsETH.mint(msg.sender, rsETHAmount)` is executed immediately at the stale rate. (`contracts/pools/RSETHPoolV3.sol`, line 262)

2. The BRIDGER later calls `moveAssetsForBridging()` to send the pooled ETH to L1. `L1Vault.depositETHForL1VaultETH()` mints rsETH using the **live** oracle price:

```solidity
uint256 rsETHAmountToMint = lrtDepositPool.getRsETHAmountToMint(ETH_IDENTIFIER, balanceOfETH);
lrtDepositPool.depositETH{ value: balanceOfETH }(rsETHAmountToMint, "");
```
(`contracts/L1Vault.sol`, lines 152–158)

`getRsETHAmountToMint` uses `lrtOracle.rsETHPrice()` — the live L1 price, which is strictly higher than the stale L2 rate. (`contracts/LRTDepositPool.sol`, line 520)

3. Because `rsETHPrice_L1 > rsETHRate_L2_stale`:
```
rsETH_minted_L1 = ETH / rsETHPrice_L1  <  wrsETH_minted_L2 = ETH / rsETHRate_L2_stale
```

4. The smaller rsETH amount is bridged back and deposited into `RsETHTokenWrapper` via `depositBridgerAssets()`. The wrapper's rsETH balance is now less than its wrsETH `totalSupply()`. (`contracts/L2/RsETHTokenWrapper.sol`, lines 162–169)

5. `maxAmountToDepositBridgerAsset()` confirms the deficit is tracked but not prevented:
```solidity
return wrsETHSupply - balanceOfAssetInWrapper;
```
(`contracts/L2/RsETHTokenWrapper.sol`, lines 107–109)

6. Any user calling `withdraw()` after the rsETH balance is exhausted hits:
```solidity
_burn(msg.sender, _amount);
ERC20Upgradeable(_asset).safeTransfer(_to, _amount);  // reverts
```
(`contracts/L2/RsETHTokenWrapper.sol`, lines 123–125)

**Why existing guards are insufficient:** The operator-only `swapAssetToPremintedRsETH()` function can inject rsETH into the wrapper in exchange for pool ETH, but this requires active operator intervention, is not automatic, and consumes pool ETH belonging to depositors. The `depositBridgerAssets()` guard only prevents depositing *more* than the deficit — it does not close the deficit itself. No on-chain mechanism forces the deficit to zero before withdrawals are processed.

## Impact Explanation

`RsETHTokenWrapper` becomes undercollateralized: total `wrsETH` supply exceeds the rsETH held in the wrapper. Users holding `wrsETH` who call `withdraw()` will have their transactions revert once the rsETH balance is exhausted. The deficit accumulates monotonically with every deposit cycle, constituting **protocol insolvency** and **permanent freezing of funds** for the last redeemers — both Critical-severity impacts under the allowed scope.

## Likelihood Explanation

No attacker action is required. rsETH appreciates continuously, so the stale L2 rate is virtually always below the live L1 rate during normal protocol operation. Every deposit cycle — triggered by any unprivileged user calling `deposit()` — contributes to the shortfall. The deficit grows proportionally with deposit volume and the duration between rate updates. The condition is structural and self-reinforcing.

## Recommendation

1. **Mint wrsETH only after rsETH arrives from L1:** Hold deposited ETH in escrow and mint `wrsETH` only once the corresponding rsETH has been bridged back and deposited into the wrapper.
2. **Apply a conservative discount to the stale rate:** Subtract an expected appreciation buffer so the L2 always under-mints relative to what L1 will produce.
3. **Enforce maximum rate staleness:** Revert deposits in `RSETHPoolV3.deposit()` if `block.timestamp - CrossChainRateReceiver.lastUpdated` exceeds a safe threshold (e.g., 1 hour), ensuring the minting rate is never materially stale.

## Proof of Concept

**Setup:**
- Live L1 rsETH price: `1.05e18` (current)
- Stale L2 rate: `1.04e18` (2 hours old, no staleness guard)
- User deposits `1 ETH`, fee = 0

**L2 mint (stale rate):**
```
wrsETH_minted = 1e18 * 1e18 / 1.04e18 = 961538461538461538 (~0.9615 wrsETH)
```

**L1 deposit (live rate):**
```
rsETH_minted = 1e18 * 1e18 / 1.05e18 = 952380952380952380 (~0.9524 rsETH)
```

**Deficit per cycle:** `~0.00915 rsETH`

After 100 such deposits the wrapper holds a shortfall of ~0.915 rsETH. The last ~0.915 rsETH worth of `wrsETH` holders cannot redeem; their `withdraw()` calls revert at `safeTransfer`.

**Foundry fork test plan:**
1. Fork mainnet; deploy `RSETHPoolV3` pointing to a `CrossChainRateReceiver` with `rate = 1.04e18`.
2. Call `deposit{value: 1 ether}()` 100 times; record `wrsETH.totalSupply()`.
3. Simulate the bridge cycle: call `L1Vault.depositETHForL1VaultETH()` (live oracle returns `1.05e18`); record rsETH minted.
4. Call `depositBridgerAssets(rsETH, rsETH_minted)` on `RsETHTokenWrapper`.
5. Assert `wrsETH.totalSupply() > rsETH.balanceOf(wrapper)` — deficit confirmed.
6. Have the last holder call `withdraw()`; assert the call reverts with an ERC20 insufficient-balance error.
