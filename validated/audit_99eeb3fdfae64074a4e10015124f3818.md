Audit Report

## Title
`rewardPerToken()` Increment Rounds to Zero with Low-Decimal Reward Tokens and Large Stake â€” (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` stores `rewardRate` in raw token-wei per second without any precision scaling. When `rewardRate * timeDelta * 1e18 < totalKernelStaked`, the per-token increment in `rewardPerToken()` truncates to zero in integer arithmetic. Reward tokens that should have accrued during those intervals are never added to `rewardPerTokenStored` and are permanently unrecoverable, constituting a permanent freezing of unclaimed yield.

## Finding Description

`notifyRewardAmount` sets `rewardRate` with no precision multiplier:

```solidity
rewardRate = receivedAmount / duration;   // L580
```

`rewardPerToken()` then computes the per-token increment as:

```solidity
return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;   // L412-413
```

Solidity integer division truncates. The increment is zero whenever:

```
rewardRate * timeDelta * 1e18 < totalKernelStaked
```

**Concrete scenario â€” USDC rewards, 1 M KERNEL staked:**

| Parameter | Value |
|---|---|
| `receivedAmount` | 1,000 USDC = 1e9 |
| `duration` | 30 days = 2,592,000 s |
| `rewardRate` | 1e9 / 2,592,000 â‰ˆ 385 |
| `totalKernelStaked` | 1,000,000 KERNEL = 1e24 |

Per-second increment: `385 * 1 * 1e18 / 1e24 = 0`.

The first non-zero increment requires `timeDelta â‰¥ 1e24 / (385 * 1e18) â‰ˆ 2,597 s`. Every `updateReward` call within that window silently discards the accrued rewards. The `updateReward` modifier fires on every user-facing call â€” `stake()`, `stakeFor()`, `initiateWithdrawal()`, and `getReward()` â€” so normal user activity continuously triggers the loss. The discarded rewards remain in the contract with no admin sweep or recovery path.

The `rewardsToken` is set at initialization with no decimal constraint, making USDC a valid and natural configuration.

## Impact Explanation

**Medium â€” Permanent freezing of unclaimed yield.** Reward tokens deposited by the admin via `notifyRewardAmount` are transferred into the contract but a portion can never be distributed to stakers. The fraction lost scales with interaction frequency; under realistic conditions (users interacting every minute over 30 days) approximately 99.96% of deposited USDC rewards are permanently frozen. There is no recovery mechanism.

## Likelihood Explanation

No privileged access or attacker action is required. The rounding occurs automatically as users interact normally with the contract. USDC is a natural reward token for a DeFi staking protocol. A protocol with meaningful TVL would routinely have millions of KERNEL staked (1e24+ wei), which is the threshold at which per-second increments round to zero for USDC rewards. The condition is deterministic and repeatable for the entire reward period.

## Recommendation

Scale `rewardRate` by `DECIMAL_PRECISION` at assignment time, matching the canonical Synthetix implementation:

```solidity
// In notifyRewardAmount:
rewardRate = (receivedAmount * DECIMAL_PRECISION) / duration;

// In rewardPerToken():
return rewardPerTokenStored
    + (rewardRate * (lastTimeRewardApplicable() - updatedAt))
    / totalKernelStaked;

// earned() is unchanged â€” DECIMAL_PRECISION cancels correctly
```

This eliminates the rounding-to-zero condition for any realistic combination of reward token decimals and stake size.

## Proof of Concept

1. Deploy with `rewardsToken = USDC` (6 decimals).
2. Stake 1,000,000 KERNEL â†’ `totalKernelStaked = 1e24`.
3. Admin calls `notifyRewardAmount(1_000_000_000)` (1,000 USDC). `rewardRate = 385`.
4. At `t+60s`, any user calls `getReward()`. `updateReward` fires: increment = `385 * 60 * 1e18 / 1e24 = 23,100 / 1e6 = 0`. `rewardPerTokenStored` unchanged.
5. Repeat step 4 every 60 seconds for 30 days. `rewardPerTokenStored` never advances past the first 2,597-second boundary per interval.
6. At period end, `getReward()` distributes near-zero rewards; ~999.6 USDC remains permanently locked in the contract.

**Foundry fuzz test plan:** Fuzz over `(rewardAmount, duration, totalStaked, callInterval)` with `rewardToken.decimals() < 18` and assert that `sum(distributed rewards) == rewardAmount` at period end. The invariant will fail whenever `rewardRate * callInterval * 1e18 < totalStaked`.
