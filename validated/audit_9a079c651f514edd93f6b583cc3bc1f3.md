Audit Report

## Title
`RSETHPriceFeed` Returns ETH/USD-Only Staleness Metadata for Composite rsETH/USD Price - (File: contracts/oracles/RSETHPriceFeed.sol)

## Summary
`RSETHPriceFeed` computes a composite rsETH/USD price by multiplying `RS_ETH_ORACLE.rsETHPrice()` (rsETH/ETH, 18-decimal) with the ETH/USD Chainlink answer (8-decimal), but the `updatedAt` and `answeredInRound` return values are sourced exclusively from the ETH/USD feed. Because `LRTOracle` stores no last-update timestamp for `rsETHPrice`, the contract has no mechanism to reflect the rsETH component's staleness in its metadata, breaking the core invariant of a Chainlink-compatible price feed.

## Finding Description
In both `latestRoundData()` and `getRoundData()`, all five return values are first populated from `ETH_TO_USD`, and only `answer` is subsequently overwritten: [1](#0-0) [2](#0-1) 

`updatedAt` and `answeredInRound` therefore always reflect the ETH/USD Chainlink feed's last update, not the composite price's true freshness.

The rsETH component (`rsETHPrice`) is a plain state variable in `LRTOracle` with no associated timestamp: [3](#0-2) 

It is updated via the public `updateRSETHPrice()` on an independent, slower cadence (typically daily): [4](#0-3) 

The update writes only the new price, with no timestamp recorded: [5](#0-4) 

During any window where `updateRSETHPrice()` has not been called but the ETH/USD Chainlink feed has ticked (its normal heartbeat of ≤1 hour), `latestRoundData()` returns a recent `updatedAt` while serving a stale rsETH/ETH rate. Any consumer applying a standard staleness guard (e.g., `require(block.timestamp - updatedAt < heartbeat)`) will pass the check and consume a price that is hours or days old in its rsETH component.

## Impact Explanation
The `RSETHPriceFeed` contract fails to deliver its promised function as a Chainlink-compatible price feed: `updatedAt` does not represent the age of the returned composite price. External protocols integrating this feed with standard staleness guards will silently consume stale rsETH/USD prices. No direct loss of LRT-rsETH depositor funds results from this contract alone. This matches the allowed **Low** impact: *Contract fails to deliver promised returns, but doesn't lose value.*

## Likelihood Explanation
No attacker action is required. The condition arises during normal operation whenever the ETH/USD Chainlink feed updates (every few minutes to 1 hour) while `rsETHPrice` has not been refreshed (daily cadence). Any caller of `latestRoundData()` during this window receives misleading metadata. The gap is a routine operating condition, not an edge case.

## Recommendation
Add a `lastPriceUpdateTime` state variable to `LRTOracle`, set it to `block.timestamp` inside `_updateRsETHPrice()` at the point where `rsETHPrice` is written. In `RSETHPriceFeed.latestRoundData()` and `getRoundData()`, return `min(ethToUsdUpdatedAt, rsETHOracleLastUpdatedAt)` as `updatedAt`, and derive `answeredInRound` to reflect the more stale of the two sources. This ensures consumers receive staleness metadata that accurately represents the composite price's true freshness.

## Proof of Concept
```solidity
// Fork test setup:
// 1. Deploy LRTOracle; call updateRSETHPrice() to set rsETHPrice (T=0).
// 2. Warp block.timestamp forward by 25 hours (rsETHPrice now 25h stale).
// 3. ETH/USD Chainlink feed has updated 30 seconds ago (normal heartbeat).
// 4. Call RSETHPriceFeed.latestRoundData():
(, int256 answer,, uint256 updatedAt,) = rsETHPriceFeed.latestRoundData();
// updatedAt == block.timestamp - 30  (from ETH/USD feed)
// answer    == stale_rsETH_per_ETH * current_ETH_USD / 1e18  (25h-old rsETH rate)

// Consumer staleness guard:
require(block.timestamp - updatedAt < 3600, "stale"); // PASSES (30s < 1h)
// Consumer proceeds with a 25-hour-old rsETH/USD composite price
```
A Foundry fork test against mainnet can confirm this by reading the actual `rsETHPrice` storage slot age versus the ETH/USD feed's `updatedAt` and asserting the discrepancy.
