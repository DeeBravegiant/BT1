Audit Report

## Title
Nested Loop Over EigenLayer Queued Withdrawals in `getAssetDistributionData` Causes Escalating Gas Cost on Every User Deposit — (`contracts/LRTDepositPool.sol`, `contracts/NodeDelegator.sol`)

## Summary
Every call to `depositETH()` or `depositAsset()` triggers `_checkIfDepositAmountExceedesCurrentLimit` → `getTotalAssetDeposits` → `getAssetDistributionData`, which loops over all NDCs and calls `getAssetUnstaking` on each. `getAssetUnstaking` fetches all EigenLayer queued withdrawals for that NDC and iterates over each withdrawal's strategies with external `sharesToUnderlyingView` calls. As the protocol scales, the cumulative gas cost of a single deposit grows multiplicatively across assets × NDCs × withdrawals × strategies, eventually causing deposits to revert out-of-gas.

## Finding Description
The confirmed call chain on every deposit:

```
depositETH() / depositAsset()
  └─ _beforeDeposit()                                    [LRTDepositPool.sol:661]
       └─ _checkIfDepositAmountExceedesCurrentLimit()    [LRTDepositPool.sol:677]
            └─ getTotalAssetDeposits(asset)              [LRTDepositPool.sol:385]
                 └─ getAssetDistributionData(asset)      [LRTDepositPool.sol:426]
                      └─ for each NDC                    [LRTDepositPool.sol:447]
                           └─ getAssetUnstaking(asset)   [NodeDelegator.sol:405]
                                └─ getQueuedWithdrawals  [NodeDelegator.sol:406]
                                     └─ for each withdrawal × strategy
                                          └─ strategy.underlyingToken()       [NodeDelegator.sol:417]
                                          └─ strategy.sharesToUnderlyingView() [NodeDelegator.sol:424]
```

`getAssetDistributionData` iterates over every NDC and calls `getAssetUnstaking` per NDC: [1](#0-0) 

`getAssetUnstaking` fetches the full queued withdrawal list from EigenLayer and iterates with a nested strategy loop, issuing external calls per strategy: [2](#0-1) 

Critically, `getQueuedWithdrawals(address(this))` is called **once per asset per NDC**, not once per NDC. For 5 assets and 10 NDCs, this is 50 separate `getQueuedWithdrawals` cross-contract calls, each returning the same NDC's full withdrawal list. The same path is also triggered by the public `updateRSETHPrice()` → `_getTotalEthInProtocol()`, which loops over all supported assets before entering the NDC loop: [3](#0-2) 

The `maxUncompletedWithdrawalCount` cap of 80 is a global counter across all NDCs in the single `LRTUnstakingVault`. The developers' own comment acknowledges the gas concern but the cap was set without accounting for the multiplicative effect of the per-asset re-fetching: [4](#0-3) 

The comment states "120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price" — confirming the developers are aware gas is a binding constraint, yet the per-asset redundant `getQueuedWithdrawals` calls were not accounted for. Additionally, forced undelegations by EigenLayer operators can push the total withdrawal count beyond the protocol's `uncompletedWithdrawalCount` counter, as acknowledged by the buffer comment ("ndc count * asset count = 15").

## Impact Explanation
**Medium — Unbounded gas consumption / Temporary freezing of funds.**

At maximum configured scale (5 assets, 10 NDCs, 80 total queued withdrawals, 2 strategies each), a single `depositETH()` call executes: 50 `getQueuedWithdrawals` cross-contract calls + up to 160 `strategy.underlyingToken()` calls + up to 160 `strategy.sharesToUnderlyingView()` calls, totalling ~370+ external calls. At ~5,000 gas per external call, this approaches ~1.85M gas for the unstaking loop alone, before accounting for storage reads and other computation. If `updateRSETHPrice()` becomes uncallable, `rsETHPrice` goes stale and `getRsETHAmountToMint` returns incorrect values for all subsequent deposits. [5](#0-4) 

## Likelihood Explanation
**Medium.** No attacker action is required. Ordinary protocol growth — adding NDCs, supporting more LSTs, and accumulating queued withdrawals during normal unstaking operations — drives the gas cost upward. The protocol is explicitly designed to scale (`maxNodeDelegatorLimit` is admin-settable upward), and the withdrawal queue fills naturally. The developers' own comment confirms awareness of the gas constraint, but the multiplicative per-asset re-fetching was not fully mitigated. [6](#0-5) 

## Recommendation
1. **Eliminate per-asset redundant `getQueuedWithdrawals` calls**: In `getAssetDistributionData`, fetch queued withdrawals once per NDC and compute all asset amounts in a single pass, rather than calling `getAssetUnstaking(asset)` separately for each asset.
2. **Decouple TVL accounting from the deposit hot path**: Cache `totalAssetDeposits` updated lazily (e.g., only via `updateRSETHPrice`) and use the cached value in `_checkIfDepositAmountExceedesCurrentLimit` instead of recomputing on every deposit.
3. **Track unstaking amounts in protocol storage**: Maintain a per-asset `assetUnstaking` mapping updated on `initiateUnstaking` / `completeUnstaking` to avoid re-querying EigenLayer on every view call.

## Proof of Concept
1. Fork mainnet/testnet with EigenLayer deployed.
2. Configure: 5 supported LST assets, 10 NDCs, initiate unstaking to accumulate 80 total queued EigenLayer withdrawals (8 per NDC, 2 strategies each).
3. Call `depositETH{value: 1 ether}(0, "")` from an unprivileged EOA and measure gas.
4. Execution path: `_checkIfDepositAmountExceedesCurrentLimit` calls `getTotalAssetDeposits` once per asset (×5) → `getAssetDistributionData` (×10 NDCs each) → `getAssetUnstaking` (×50 total `getQueuedWithdrawals` calls) → 80 withdrawal × 2 strategy iterations = 160 `sharesToUnderlyingView` + 160 `underlyingToken` external calls.
5. Repeat with `updateRSETHPrice()` to confirm the same path is triggered for the oracle update, which additionally multiplies by all supported assets.
6. Confirm gas consumption approaches or exceeds block gas limits as dimensions scale, causing reverts. [7](#0-6) [8](#0-7)

```text
    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
```
