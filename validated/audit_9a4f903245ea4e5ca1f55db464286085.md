Audit Report

## Title
Missing L2 Sequencer Uptime Check Allows Stale Oracle Prices to Reach Pool Swaps — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`, `smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary
All three price providers (`PriceProviderL2`, `ProtectedPriceProviderL2`, `AnchoredPriceProvider`) implement a staleness guard that is purely age-based and does not query a Chainlink L2 sequencer uptime feed. When the L2 sequencer goes offline and resumes, oracle `refTime` is frozen at the pre-outage value; if the outage duration is shorter than `MAX_TIME_DELTA`, `_isStale` returns `false` and the stale price flows into `MetricOmmPool.swap()`, causing bad-price execution and direct LP losses.

## Finding Description

All three providers share the same structural flaw. In `PriceProviderL2` and `ProtectedPriceProviderL2`, `_isStale` performs only an age check:

```solidity
// PriceProviderL2.sol L148-149 / ProtectedPriceProviderL2.sol L151-152
// refTime in the past or equal: check age
return (nowTs - refTime) > maxDelta;
```

`AnchoredPriceProvider._isStale` is identical in effect:

```solidity
// AnchoredPriceProvider.sol L229
return (nowTs - refTime) > maxDelta;
```

No contract queries a Chainlink L2 sequencer uptime feed at any point. The `FUTURE_TOLERANCE` immutable only handles clock skew (oracle timestamp slightly ahead of `block.timestamp`); it provides no protection against sequencer downtime.

The constructor permits `MAX_TIME_DELTA` up to 7 days:

```solidity
// PriceProviderL2.sol L92 / ProtectedPriceProviderL2.sol L96
if (_maxTimeDelta == 0 || _maxTimeDelta > 7 days) revert MaxTimeDeltaOutOfBounds();
```

Exploit path:
1. L2 sequencer goes offline. Oracle keeper transactions cannot land; `refTime` freezes at the pre-outage value.
2. Real market price moves significantly off-chain.
3. Sequencer resumes. `block.timestamp` has advanced, but `refTime` is still the pre-outage timestamp.
4. If `(block.timestamp − refTime) ≤ MAX_TIME_DELTA`, `_isStale` returns `false` — the stale price is accepted.
5. `MetricOmmPool._getBidAndAskPriceX64()` calls `IPriceProvider(activePriceProvider).getBidAndAskPrice()`, which returns the stale `(bid, ask)` pair.
6. The stale pair flows into `SwapMath.midAndSpreadFeeX64FromBidAsk` and then `_executeSwap`, settling real token transfers at the wrong price.

## Impact Explanation

When the sequencer resumes and the oracle price is stale but within `MAX_TIME_DELTA`, any unprivileged trader can call `pool.swap()` immediately. If the stale price is lower than the current market price, the trader buys token0 from the pool below market, draining LP value. If the stale price is higher, the trader sells token0 to the pool above market, again draining LP value. In both cases `binTotals.scaledToken0`/`scaledToken1` are updated at the wrong exchange rate, permanently mis-accounting LP claims. This is a direct, unbounded loss of LP principal matching the "bad-price execution" and "direct loss of user principal" allowed impact categories.

## Likelihood Explanation

The protocol explicitly targets L2 deployments (contracts are named `PriceProviderL2`, `ProtectedPriceProviderL2`). Base and Arbitrum have each experienced multi-minute sequencer outages. Any outage shorter than `MAX_TIME_DELTA` (which can be as short as seconds in production) creates the window. No special privilege is required — any address can call `swap()` immediately after the sequencer resumes. The condition is externally triggered but historically recurring.

## Recommendation

In `PriceProviderL2`, `ProtectedPriceProviderL2`, and `AnchoredPriceProvider` (when deployed on L2), query the Chainlink L2 sequencer uptime feed before accepting an oracle price. If the sequencer was recently down, revert with `FeedStalled()` for a configurable grace period after it recovers:

```solidity
(, int256 answer, , uint256 updatedAt, ) = sequencerUptimeFeed.latestRoundData();
if (answer != 0) return (0, type(uint128).max); // sequencer is down
if (block.timestamp - updatedAt < GRACE_PERIOD) return (0, type(uint128).max); // grace period
```

This guard should be placed at the top of `_getBidAndAskPrice()` in each provider, before the staleness check.

## Proof of Concept

```
Setup:
  Pool on Base, PriceProviderL2, MAX_TIME_DELTA = 120s
  ETH/USDC pool, oracle mid = 3000e8

T=0:   Oracle pushes refTime=T, mid=3000e8. Pool is live.
T=10:  Base sequencer goes offline. No txs land.
       Real ETH price falls to 2800 off-chain.
T=90:  Sequencer resumes. refTime is still T=0 (90s old < 120s MAX_TIME_DELTA).
       _isStale(T, T+90, 120, FUTURE_TOLERANCE) → false  ← stale price accepted

T=91:  Attacker calls pool.swap(zeroForOne=false, amountSpecified=+1000e6 USDC)
       _getBidAndAskPriceX64() returns bid/ask anchored at 3000e8 (stale).
       Pool sells ETH at ~3000 USDC/ETH.
       Real market price: 2800 USDC/ETH.
       Attacker receives ETH worth 2800 USDC for 3000 USDC paid → LP loss of ~200 USDC per ETH.
       binTotals.scaledToken0 decremented at wrong rate; LP claims permanently under-collateralized.

Foundry fork test: fork Base mainnet at a block after a historical sequencer outage,
deploy pool with PriceProviderL2 and MAX_TIME_DELTA > outage duration,
warp block.timestamp to post-resume, call swap(), assert LP token balance decreases
relative to fair-value accounting.
```
