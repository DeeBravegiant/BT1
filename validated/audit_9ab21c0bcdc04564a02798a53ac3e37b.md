### Title
DRep Expiry Check in `dRepAcceptedRatio` Ignores Accumulated Dormant Epochs — (`eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The `dRepAcceptedRatio` function in the Conway `RATIFY` rule checks whether a DRep is expired using only the stored `drepExpiry` field, without adding the accumulated `vsNumDormantEpochs` offset. Because the `RatifyEnv` is populated from a pulser snapshot taken at the start of the epoch — before any within-epoch `updateDormantDRepExpiries` bump — DReps whose stored expiry falls below `reCurrentEpoch` but whose *effective* expiry (`drepExpiry + numDormantEpochs`) is still valid are silently excluded from the vote tally. An unprivileged actor who submits a governance proposal at the right moment can exploit this to manipulate the ratification outcome of any concurrent governance action.

---

### Finding Description

**Background — dormant-epoch accounting**

Conway tracks epochs in which there are no active governance proposals in `VState.vsNumDormantEpochs`. [1](#0-0)  When a new proposal is submitted, `updateDormantDRepExpiries` adds `numDormantEpochs` to every DRep's stored `drepExpiry` and resets the counter to zero. [2](#0-1)  The *effective* expiry of a DRep at any moment is therefore `drepExpiry + numDormantEpochs`, as reflected in the test helper `isDRepExpired`: [3](#0-2) 

**The pulser snapshot problem**

At the end of each epoch the ledger calls `setFreshDRepPulsingState`, which initialises a `DRepPulser` with a snapshot of the current `vsDReps` map. [4](#0-3)  This snapshot captures the stored `drepExpiry` values *as they stand at that moment*. During the following epoch, any call to `updateDormantDRepExpiries` (triggered by a new proposal) updates the live `VState`, but the pulser snapshot is **not** refreshed.

**The incorrect expiry check**

When ratification runs, `finishDRepPulser` builds a `RatifyEnv` from the stale snapshot:

```haskell
reDRepState = dpDRepState   -- snapshot from start of epoch, pre-bump
reCurrentEpoch = dpCurrentEpoch
``` [5](#0-4) 

Inside `dRepAcceptedRatio`, the expiry check is:

```haskell
Just drepState
  | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired
``` [6](#0-5) 

`numDormantEpochs` is **never added** here. The `RatifyEnv` type carries no `reNumDormantEpochs` field, so the information is structurally unavailable to `dRepAcceptedRatio`. [7](#0-6) 

**Concrete discrepancy**

Let `drepActivity = 5`, `numDormantEpochs = 5` accumulated at the end of epoch 5, and DRep A's stored `drepExpiry = 5`.

| Step | Event | Stored `drepExpiry` | `numDormantEpochs` | Effective expiry |
|---|---|---|---|---|
| End epoch 5 | Pulser snapshot taken | 5 | 5 | 10 |
| Epoch 6 | Proposal submitted → `updateDormantDRepExpiries` | 10 (live VState) | 0 | 10 |
| End epoch 6 | Ratification via pulser | **5** (snapshot) | N/A | — |

Ratification check: `6 > 5` → **DRep A treated as expired**. Actual effective expiry: 10 ≥ 6 → **DRep A is active**. DRep A's vote (YES or NO) is silently dropped.

---

### Impact Explanation

An attacker who submits a governance proposal after dormant epochs have accumulated causes ratification to run against a stale DRep snapshot in which many DReps appear expired. Depending on how those DReps would have voted:

- **Malicious action passes**: DReps who would vote NO (or whose implicit NO from non-participation would count against the proposal) are excluded from the denominator, inflating the yes-ratio above the required threshold. This can enact an unauthorized `TreasuryWithdrawals`, `ParameterChange`, `HardForkInitiation`, `UpdateCommittee`, or `NewConstitution` action.
- **Legitimate action blocked**: DReps who voted YES are excluded, deflating the yes-ratio below the threshold.

Both outcomes constitute **unauthorized governance, treasury, protocol-parameter, committee, constitution, or hard-fork actions being enacted or blocked**, matching the Critical impact tier.

---

### Likelihood Explanation

- Dormant epochs accumulate naturally whenever no governance proposals are live for multiple consecutive epochs — a realistic condition in the early Conway era or during governance inactivity.
- The attacker's only required action is submitting a single governance proposal transaction at the right epoch, which is permissionless.
- The attacker does not need any privileged role, key, or majority stake. They only need to observe `vsNumDormantEpochs` (publicly readable from the ledger state) and time their proposal accordingly.
- The window of vulnerability is one full epoch (the epoch in which the proposal is submitted), giving the attacker ample time to act.

---

### Recommendation

Pass `numDormantEpochs` into the `RatifyEnv` (add a `reNumDormantEpochs :: EpochNo` field) and adjust the expiry check in `dRepAcceptedRatio`:

```haskell
| reCurrentEpoch > binOpEpochNo (+) reNumDormantEpochs (drepExpiry drepState)
    -> (yes, tot)  -- drep is expired
```

Alternatively, when initialising the `DRepPulser`, pre-compute the effective expiry for each DRep (`drepExpiry + numDormantEpochs`) and store that adjusted value in `dpDRepState`, so the snapshot already reflects the dormant-epoch offset at snapshot time.

The fix should mirror the logic already used in `isDRepExpired` and `computeDRepExpiry`. [8](#0-7) 

---

### Proof of Concept

**Setup** (`ppDRepActivity = 3`, `ppGovActionLifetime = 1`):

1. **Epoch 0**: DRep D registers. `drepExpiry = 3`. `numDormantEpochs = 0`.
2. **Epochs 1–3**: No proposals submitted. `numDormantEpochs` increments to 3 at each epoch boundary. [9](#0-8) 
3. **End of epoch 3**: Pulser for epoch 4 is initialised with snapshot `{drepExpiry = 3}`. `numDormantEpochs = 3`.
4. **Epoch 4**: Attacker submits a malicious `TreasuryWithdrawals` proposal. `updateDormantDRepExpiries` bumps DRep D's live `drepExpiry` to `3 + 3 = 6` and resets `numDormantEpochs = 0`. DRep D votes NO. [10](#0-9) 
5. **End of epoch 4**: `extractDRepPulsingState` finalises the pulser. `reDRepState` has `drepExpiry = 3` (snapshot). `reCurrentEpoch = 4`. [11](#0-10) 
6. **`dRepAcceptedRatio` check**: `4 > 3` → DRep D is treated as **expired**. DRep D's NO vote is discarded and their stake is removed from the denominator. [12](#0-11) 
7. **Result**: The treasury withdrawal passes ratification with an inflated yes-ratio, despite DRep D's active NO vote. DRep D's actual effective expiry is `3 + 3 = 6 ≥ 4`, so they were active and should have been counted.

-- | Update dormant expiry for all DReps that are active.
-- And also reset the `numDormantEpochs` counter.
updateDormantDRepExpiry ::
  -- | Current Epoch
  EpochNo ->
  VState era ->
  VState era
updateDormantDRepExpiry currentEpoch vState =
  if numDormantEpochs == EpochNo 0
    then vState
    else
      vState
        & vsNumDormantEpochsL .~ EpochNo 0
        & vsDRepsL %~ Map.map updateExpiry
  where
    numDormantEpochs = vState ^. vsNumDormantEpochsL
    updateExpiry =
      drepExpiryL
        %~ \currentExpiry ->
          let actualExpiry = binOpEpochNo (+) numDormantEpochs currentExpiry
           in if actualExpiry < currentEpoch
                then currentExpiry
                else actualExpiry
```

```haskell
dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr
```
