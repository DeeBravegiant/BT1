### Title
Cross-Chain Replay of `deploy_token` Signatures via Missing Chain ID in `MetadataPayload` Borsh Encoding — (`aptos/sources/bridge_types.move`, `starknet/src/omni_bridge.cairo`)

---

### Summary

The `MetadataPayload` Borsh encoding used for `deploy_token` does not include a destination chain ID, unlike `TransferMessagePayload` which explicitly binds to a chain. A valid NEAR MPC signature over a `MetadataPayload` produced for chain A can be replayed verbatim on chain B, deploying an unauthorized token and permanently blocking legitimate deployment on that chain.

---

### Finding Description

**`TransferMessagePayload` is chain-bound; `MetadataPayload` is not.**

`transfer_message_to_borsh` takes `chain_id` as a parameter and encodes it twice into the signed bytes: [1](#0-0) 

`metadata_to_borsh` takes no `chain_id` parameter and encodes only `token`, `name`, `symbol`, `decimals`: [2](#0-1) 

The Starknet test helper confirms the same layout — no chain ID in the deploy-token message hash: [3](#0-2) 

Compare with `fin_transfer` on Starknet, which explicitly passes `omni_bridge_chain_id` into the hash: [4](#0-3) 

And on EVM, `finTransfer` encodes `omniBridgeChainId` twice into the Borsh blob before hashing: [5](#0-4) 

The Starknet CLAUDE.md explicitly documents chain-ID binding as a security property for `fin_transfer` but makes no equivalent claim for `deploy_token`: [6](#0-5) 

On Aptos, `deploy_token` calls `metadata_to_borsh()` with no chain argument, then passes the result directly to `verify_signature`: [7](#0-6) 

---

### Impact Explanation

An attacker who observes a valid `deploy_token` transaction on chain A (e.g., EVM) can immediately replay the identical `(signature_rs, signature_v, token, name, symbol, decimals)` arguments against chain B (e.g., Aptos or Starknet). The signature verification passes because the signed bytes are byte-identical. Chain B records the token ID in its `near_to_*_token` mapping: [8](#0-7) 

When NEAR's relayer later attempts the legitimate deployment on chain B, it aborts with `E_TOKEN_ALREADY_DEPLOYED`. NEAR's bridge has no entry for chain B in its own token-address mapping, so it cannot construct a valid `fin_transfer` payload for that chain. Token bridging to chain B for that asset is permanently frozen without a contract upgrade or admin intervention. This satisfies the **irreversible fund lock / frozen redemption path** impact class.

---

### Likelihood Explanation

The attack is fully permissionless: the attacker only needs to watch the mempool or finalized blocks on chain A for a `deploy_token` call, then submit the same calldata to chain B. No privileged access, no leaked keys, no colluding parties. The window is the time between NEAR signing the payload and the legitimate relayer submitting it on chain B — typically seconds to minutes. Any token that NEAR authorizes for deployment on one chain is immediately at risk on every other chain.

---

### Recommendation

Include the destination chain ID in `MetadataPayload`'s Borsh encoding, mirroring the pattern already used by `TransferMessagePayload`. Concretely:

- In `aptos/sources/bridge_types.move`, change `metadata_to_borsh` to accept a `chain_id: u8` parameter and prepend it (or interleave it as the OmniAddress tag) before the token fields.
- Apply the same change to the equivalent encoder in `starknet/src/bridge_types.cairo`, `evm/src/omni-bridge/contracts/OmniBridge.sol` (`deployToken`), and `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs` (`serialize_for_near`).
- Update the NEAR-side payload construction to include the destination chain ID when requesting the MPC signature for `deploy_token`.

---

### Proof of Concept

1. NEAR MPC signs `MetadataPayload { token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6 }` for deployment on EVM (chain 1). The signed bytes contain no chain discriminator.
2. Attacker observes the EVM `deployToken` transaction and extracts `(signature_rs, signature_v, "usdc.near", "USD Coin", "USDC", 6)`.
3. Attacker calls `deploy_token(signature_rs, signature_v, "usdc.near", "USD Coin", "USDC", 6)` on the Aptos bridge.
4. `metadata_to_borsh()` produces identical bytes; `verify_signature` passes against `near_bridge_derived_address`.
5. `state.near_to_aptos_token.add("usdc.near", <attacker-triggered-address>)` executes.
6. NEAR's relayer later attempts the legitimate Aptos deployment; `assert!(!state.near_to_aptos_token.contains("usdc.near"), E_TOKEN_ALREADY_DEPLOYED)` aborts.
7. NEAR's bridge has no Aptos address for `usdc.near` in its own mapping; `sign_transfer` for Aptos destinations on this token fails permanently.
