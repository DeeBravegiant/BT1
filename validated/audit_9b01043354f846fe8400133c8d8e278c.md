### Title
Off-by-One in `valid_entries()` Keeps Expired MPC Image Hash Permanently Authorized - (`File: crates/contract/src/tee/proposal.rs`)

---

### Summary

`AllowedDockerImageHashes::valid_entries()` uses `rposition` to find the rightmost expired entry and then slices from that index inclusive (`get(cutoff_index..)`). Because the entry at `cutoff_index` is itself expired, the most recently expired MPC image hash is always included in the "valid" allowed set. This is a direct analog to the `is_permissioned()` bug: an expiration boundary is crossed but the expired state is still treated as authorized.

---

### Finding Description

`valid_entries()` in `crates/contract/src/tee/proposal.rs` is the single source of truth for which MPC Docker image hashes are currently allowed:

```rust
fn valid_entries(&self, tee_upgrade_deadline_duration: Duration) -> Vec<AllowedMpcDockerImage> {
    let current_time = Timestamp::now();
    let cutoff_index = self
        .allowed_tee_proposals
        .iter()
        .rposition(|allowed_docker_image| {
            let Some(grace_period_deadline) = allowed_docker_image
                .added
                .checked_add(tee_upgrade_deadline_duration)
            else {
                log!("Error: timestamp overflowed ...");
                return true;
            };
            grace_period_deadline < current_time   // true  ⟺  entry IS expired
        })
        .unwrap_or(0);

    self.allowed_tee_proposals
        .get(cutoff_index..)   // ← includes the expired entry at cutoff_index
        .unwrap_or(&[])
        .to_vec()
}
```

`allowed_tee_proposals` is sorted ascending by `added` (oldest first). `rposition` returns the index of the **last** entry whose grace period has already passed — i.e., the last **expired** entry. The slice `get(cutoff_index..)` then includes that expired entry.

**Concrete scenario (two hashes):**

| Hash | `added` | Grace-period deadline |
|------|---------|----------------------|
| A    | T₀      | T₀ + 7 d             |
| B    | T₁ > T₀ | T₁ + 7 d             |

At time `T₀ + 7d + 1 s` (only A is expired, B is still valid):

- `rposition` closure returns `true` for A, `false` for B → returns index `0`.
- `get(0..)` → `[A, B]` — **A is expired but included**.

The correct slice is `get(cutoff_index + 1..)` → `[B]`.

The existing unit test `test_clean_expired` does **not** catch this because it sets the clock past **both** entries' deadlines simultaneously, so `rposition` returns the index of the newest entry and the "disaster-recovery fallback" (keep at least one) masks the off-by-one. [1](#0-0) 

---

### Impact Explanation

Every caller that gates TEE authorization flows through `valid_entries`:

1. **`add_participant` / `submit_participant_info`** — calls `get_allowed_mpc_docker_image_hashes` → `valid_entries` to build the allowed-hash list passed to `verify_locally`. A node running the expired image hash A can re-submit a fresh attestation after T₀ + 7d and it will be accepted, because A is still in the returned list. [2](#0-1) 

2. **`reverify_participants` / `verify_tee`** — uses the same `get_allowed_mpc_docker_image_hashes` path. Because A is still present, `re_verify` succeeds for the node running A, so `verify_tee` never triggers a resharing to kick it out. [3](#0-2) 

3. **`assert_caller_is_attested_participant_and_protocol_active`** — gates `respond_sign`, `respond_verify_foreign_tx`, and other privileged callbacks. A node whose stored attestation was accepted under the expired hash A passes this check and can continue to submit signing responses. [4](#0-3) 

Net result: a participant running an MPC image whose grace period has expired is **never evicted** and continues to participate in threshold signing indefinitely. This breaks the core TEE invariant that all active signers run an approved image, and allows a node on a potentially vulnerable old image to contribute partial signatures — undermining the confidentiality and integrity guarantees of the MPC key-share protocol.

---

### Likelihood Explanation

The condition is triggered automatically whenever:
- Operators vote in a new MPC image hash (routine upgrade), and
- At least one participant does not upgrade before the grace period (7 days by default) elapses.

No special attacker capability is required. The participant simply continues running the old image and re-submits its attestation every 7 days. The contract accepts each re-submission because the expired hash is still in the allowed list. This is a realistic operational scenario during any rolling upgrade. [5](#0-4) 

---

### Recommendation

Change the slice to exclude the expired entry at `cutoff_index`:

```rust
// Before
self.allowed_tee_proposals
    .get(cutoff_index..)
    .unwrap_or(&[])
    .to_vec()

// After
let valid_start = if self
    .allowed_tee_proposals
    .get(cutoff_index)
    .map(|e| e.added.checked_add(tee_upgrade_deadline_duration)
              .map_or(true, |d| d < current_time))
    .unwrap_or(false)
{
    // cutoff_index itself is expired; start one past it (disaster-recovery
    // fallback: if that would be out of bounds, keep the last entry)
    (cutoff_index + 1).min(self.allowed_tee_proposals.len().saturating_sub(1))
} else {
    cutoff_index
};

self.allowed_tee_proposals
    .get(valid_start..)
    .unwrap_or(&[])
    .to_vec()
```

Alternatively, restructure `valid_entries` to filter entries whose own deadline has not yet passed, keeping at least the newest as the disaster-recovery fallback. Add a unit test that places the clock exactly one nanosecond past the first entry's deadline while the second entry is still valid, and asserts that only the second entry is returned. [6](#0-5)

### Proof of Concept

Using the existing test harness in `crates/contract/src/tee/proposal.rs`:

```rust
#[test]
fn test_expired_hash_excluded_when_newer_hash_is_valid() {
    let mut allowed = AllowedDockerImageHashes::default();
    let grace = Duration::from_secs(10 * 24 * 60 * 60); // 10 days
    let nanos = |s: u64| s * 1_000_000_000u64;

    // Insert hash A at t = 1 s
    testing_env!(VMContextBuilder::new().block_timestamp(nanos(1)).build());
    allowed.insert(dummy_code_hash(1), grace);

    // Insert hash B at t = 2 s
    testing_env!(VMContextBuilder::new().block_timestamp(nanos(2)).build());
    allowed.insert(dummy_code_hash(2), grace);

    // Advance to 1 ns past A's deadline (A expired, B still valid)
    let a_deadline_ns = nanos(1) + grace.as_nanos() as u64;
    testing_env!(VMContextBuilder::new().block_timestamp(a_deadline_ns + 1).build());

    let hashes = allowed.get_image_hashes(grace);

    // BUG: currently returns [hash_A, hash_B] — hash_A should be excluded
    assert_eq!(hashes.len(), 1, "expired hash A must not appear");
    assert_eq!(hashes[0], dummy_code_hash(2));
}
```

With the current code this assertion fails: `hashes.len()` is `2` and `hashes[0]` is the expired `dummy_code_hash(1)`. [7](#0-6)

impl AllowedDockerImageHashes {
    /// Checks if a Docker image hash is still valid (not expired).
    fn valid_entries(&self, tee_upgrade_deadline_duration: Duration) -> Vec<AllowedMpcDockerImage> {
        let current_time = Timestamp::now();
        // get the index of the most recently enforced docker image
        let cutoff_index = self
            .allowed_tee_proposals
            .iter()
            .rposition(|allowed_docker_image| {
                let Some(grace_period_deadline) = allowed_docker_image
                    .added
                    .checked_add(tee_upgrade_deadline_duration)
                else {
                    log!("Error: timestamp overflowed when calculating grace_period_deadline.");
                    return true;
                };
                // if the grace period for this docker hash is in the past, then older hashes are no longer accepted
                grace_period_deadline < current_time
            })
            .unwrap_or(0);

        self.allowed_tee_proposals
            .get(cutoff_index..)
            .unwrap_or(&[])

    }
