Audit Report

## Title
Uninitialized `withdrawalDelay` Allows Immediate Bypass of Withdrawal Lock - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns a value to `withdrawalDelay`, leaving it at the Solidity default of `0`. Any staker can call `initiateWithdrawal()` followed immediately by `claimWithdrawal()` in the same or next block, recovering their tokens with zero enforced wait. The contract fails to deliver its promised withdrawal delay guarantee.

## Finding Description
`withdrawalDelay` is declared as a plain `uint256` storage variable at L96, defaulting to `0`. [1](#0-0) 

`initialize()` sets `kernelToken`, `rewardsToken`, and the admin role, but never assigns `withdrawalDelay`. [2](#0-1) 

`initiateWithdrawal()` computes `unlockTime = block.timestamp + withdrawalDelay`. With `withdrawalDelay == 0`, `unlockTime` equals `block.timestamp` at the moment of initiation. [3](#0-2) 

`claimWithdrawal()` guards with `if (block.timestamp < withdrawal.unlockTime)`. Since `block.timestamp` is never strictly less than itself, this condition is immediately false — the revert never fires. [4](#0-3) 

The only other gate in `initiateWithdrawal()` is the per-user withdrawal cap check against `maxNumberOfWithdrawalsPerUser`, which also defaults to `0` and blocks all withdrawals until the admin calls `setMaxNumberOfWithdrawalsPerUser`. [5](#0-4) 

The setter `setWithdrawalDelay` explicitly rejects `_withdrawalDelay == 0`, confirming the protocol's intent that a non-zero delay must always be active. Yet `initialize()` never enforces this invariant. [6](#0-5) 

The realistic exploit window: admin calls `setMaxNumberOfWithdrawalsPerUser` to enable withdrawals but omits `setWithdrawalDelay`. In that window — which may persist indefinitely — any staker can initiate and immediately claim withdrawals with zero delay.

## Impact Explanation
The withdrawal delay is the sole mechanism preventing users from staking and immediately unstaking KERNEL tokens. With `withdrawalDelay == 0`, any staker bypasses it permissionlessly. No funds are permanently lost — users only recover their own staked tokens — placing this squarely in the **Low** impact tier: *contract fails to deliver promised returns, but doesn't lose value*.

## Likelihood Explanation
The vulnerability is active from the moment `maxNumberOfWithdrawalsPerUser` is set to a non-zero value until `setWithdrawalDelay` is explicitly called. Any staker can trigger it with no special preconditions, no privileged access, and no victim interaction. The window is realistic because `initialize()` silently leaves `withdrawalDelay` at `0` with no on-chain enforcement requiring it to be set before withdrawals are enabled.

## Recommendation
Initialize `withdrawalDelay` to a safe non-zero default inside `initialize()`, and add a guard in `initiateWithdrawal()` that reverts if `withdrawalDelay == 0`:

```solidity
// In initialize():
withdrawalDelay = 7 days; // or pass as a constructor/initializer parameter

// In initiateWithdrawal():
if (withdrawalDelay == 0) revert InvalidWithdrawalDelay();
```

This mirrors the existing pattern in `setWithdrawalDelay`, which already rejects zero values, and ensures the invariant is enforced from deployment.

## Proof of Concept
1. Admin deploys `KernelDepositPool` and calls `initialize(admin, kernelToken, rewardToken)`. `withdrawalDelay` is `0`.
2. Admin calls `setMaxNumberOfWithdrawalsPerUser(10)` to enable withdrawals, but does **not** call `setWithdrawalDelay`.
3. Attacker calls `stake(1000e18)`.
4. Attacker calls `initiateWithdrawal(1000e18)`. `unlockTime = block.timestamp + 0 = block.timestamp`.
5. In the same block (or next block), attacker calls `claimWithdrawal(withdrawalId)`. The check `block.timestamp < unlockTime` evaluates to `false` (equal), so `WithdrawalNotReady` is never reverted.
6. Attacker receives `1000e18` KERNEL tokens immediately with zero enforced delay.

**Foundry test sketch:**
```solidity
function test_immediateWithdrawal() public {
    vm.prank(admin);
    pool.setMaxNumberOfWithdrawalsPerUser(10);
    // withdrawalDelay intentionally NOT set

    deal(address(kernelToken), attacker, 1000e18);
    vm.startPrank(attacker);
    kernelToken.approve(address(pool), 1000e18);
    pool.stake(1000e18);
    pool.initiateWithdrawal(1000e18);
    pool.claimWithdrawal(1); // succeeds in same block — no delay enforced
    assertEq(kernelToken.balanceOf(attacker), 1000e18);
    vm.stopPrank();
}
```
