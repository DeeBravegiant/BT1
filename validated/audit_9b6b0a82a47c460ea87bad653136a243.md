### Title
Unprivileged Callers Can Exhaust the Presignature Pool via Cheap `sign()` Spam, Causing Legitimate Requests to Time Out — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `sign()` function requires only 1 yoctoNEAR deposit with no per-caller rate limit and no global cap on distinct pending requests. An attacker can flood the contract with unique sign requests at negligible cost, exhausting the finite presignature pool maintained by MPC nodes. Once the pool is depleted faster than it can be replenished, legitimate sign requests expire after `REQUEST_EXPIRATION_BLOCKS = 200` blocks, breaking the request-lifecycle invariant.

---

### Finding Description

**Root cause — contract layer (`crates/contract/src/lib.rs`)**

`MINIMUM_SIGN_REQUEST_DEPOSIT` is set to 1 yoctoNEAR:

```rust
const MINIMUM_SIGN_REQUEST_DEPOSIT: NearToken = NearToken::from_yoctonear(1);
```

The `sign()` function enforces no per-caller rate limit and no global cap on the number of distinct pending requests. Every unique `SignatureRequest` (different payload or derivation path) creates a new entry in the unbounded `LookupMap`:

```rust
pending_signature_requests: LookupMap<SignatureRequest, Vec<YieldIndex>>,
```

The only existing cap is `MAX_PENDING_REQUEST_FAN_OUT = 128`, which limits *duplicate* submissions of the *same* key — it does nothing to bound the number of *distinct* keys an attacker can insert.

**Root cause — node layer (`crates/node/src/providers/ecdsa/presign.rs`, `crates/node/src/requests/queue.rs`)**

Each ECDSA (Cait-Sith) signature consumes exactly one presignature from a finite owned pool (`desired_presignatures_to_buffer`, e.g. 64 in testnet/localnet configs). Presignature generation is bottlenecked by triple generation, which is described as "heavy OT-based cryptographic computation." The background generation loop continuously refills the pool, but at a rate far below what an attacker can consume via on-chain calls.

When the pool is empty, `take_owned()` blocks indefinitely waiting for a new presignature. Requests that cannot be served within `REQUEST_EXPIRATION_BLOCKS = 200` blocks are dropped by the node queue and the user's yield-resume promise fails.

The node's `PendingRequests` map is an unbounded `HashMap<RequestId, QueuedRequest>` — there is no admission control at the node level either.

---

### Impact Explanation

An attacker can cause all legitimate ECDSA sign requests to time out for the duration of the flood. The yield-resume promise returned to the legitimate caller fails after ~200 blocks (~200 seconds at 1 s/block). This breaks the production safety invariant that a correctly-formed, correctly-funded sign request will be served within the yield-resume window. The impact is **Medium**: request-lifecycle manipulation that breaks production accounting invariants without relying on network-level DoS.

---

### Likelihood Explanation

The attack cost is minimal: 1 yoctoNEAR + ~15 TGas per request. With a testnet presignature buffer of 64, an attacker needs only ~64 unique requests to exhaust the pool, then must sustain the flood for 200 blocks to force timeouts on legitimate requests. This is achievable by any NEAR account holder with a trivial balance. The attack is analogous to the Teleportation contract issue: a cheap, sustained flood of legitimate API calls exhausts a finite shared resource.

---

### Recommendation

1. **Increase the sign request fee** to a meaningful economic deterrent (e.g., proportional to the computational cost of a presignature).
2. **Add a per-caller pending-request cap** — reject `sign()` calls from an account that already has N unresolved requests in `pending_signature_requests`.
3. **Add a global pending-request cap** — reject new requests once the total number of distinct keys in `pending_signature_requests` exceeds a configurable threshold.

---

### Proof of Concept

1. Attacker creates a NEAR account with a small NEAR balance.
2. Attacker submits 64+ unique `sign()` calls (distinct 32-byte ECDSA payloads), each attaching 1 yoctoNEAR and ≥15 TGas.
3. MPC nodes index the requests and the leader begins consuming presignatures from its owned pool via `take_owned()`.
4. The presignature pool (size 64) is exhausted; `take_owned()` now blocks waiting for new presignatures.
5. Attacker continues submitting new unique requests at the rate presignatures are regenerated, keeping the pool perpetually empty.
6. A legitimate user submits a `sign()` call.
7. After `REQUEST_EXPIRATION_BLOCKS = 200` blocks, the node queue drops the legitimate request (`RequestStatus::Drop(DropReason::RequestTimedOut)`), and the user's yield-resume promise resolves with a failure.

/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
}
