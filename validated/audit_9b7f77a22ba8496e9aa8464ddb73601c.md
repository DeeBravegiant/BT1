### Title
Proposers' NEAR Deposits Permanently Locked When `do_update` Clears All Pending Proposals — (`File: crates/contract/src/update.rs`)

---

### Summary

`propose_update` is a `#[payable]` function that requires a storage-proportional NEAR deposit to hold contract code on-chain. When `vote_update` reaches threshold and `do_update` executes, it clears **all** pending proposals — including those from other proposers — without issuing any refund. Those deposited NEAR tokens are permanently locked in the contract, which has no `withdraw` or `rescue_near` function.

---

### Finding Description

`propose_update` collects a deposit from each proposer to cover on-chain storage of the uploaded contract binary or config: [1](#0-0) 

The required amount is computed from the byte size of the uploaded code: [2](#0-1) 

When threshold votes are reached, `do_update` is called. It removes the winning entry, then **unconditionally clears every other pending proposal and all votes**: [3](#0-2) 

Clearing the entries frees the on-chain storage, so the NEAR storage-staking cost is returned to the **contract's** liquid balance. However, the original depositors — the proposers of the non-winning proposals — receive nothing back. There is no refund loop over the cleared entries, no stored mapping of `UpdateId → proposer`, and no `rescue_near` / `withdraw` function anywhere in the contract.

The `submit_participant_info` function has a related but smaller instance of the same pattern: when an existing participant re-submits their attestation (`UpdatedExistingParticipant` branch), `attestation_storage_must_be_paid_by_caller` evaluates to `false`, so the entire `if` block — including the excess-refund path — is skipped, silently consuming any attached deposit: [4](#0-3) 

---

### Impact Explanation

Every participant who proposes a contract upgrade attaches a deposit that can reach tens of NEAR for a near-maximum-size binary (tests use `NearToken::from_near(40)` for a ~1.5 MB payload). [5](#0-4) 

When any one proposal wins, every other proposer's deposit is silently absorbed into the contract's balance with no recovery path. Over the lifetime of the network — which undergoes periodic upgrades — this constitutes a permanent, cumulative loss of participant funds controlled by the MPC contract, matching the **Medium** allowed impact: *balance/accounting invariant broken without requiring network-level DoS or operator misconfiguration*.

---

### Likelihood Explanation

The governance flow explicitly anticipates multiple competing proposals: each participant may independently call `propose_update`, and the README documents that all proposals are cleared on execution. In any upgrade cycle where more than one participant proposes a different binary or config, at least one proposer loses their deposit. This is a routine, foreseeable condition, not an edge case. [6](#0-5) 

---

### Recommendation

1. **Store the proposer's account ID alongside each `UpdateEntry`** so that `do_update` can iterate over cleared entries and issue `Promise::new(proposer).transfer(deposit)` for each one before clearing.
2. Alternatively, add a `rescue_near(amount, recipient)` governance-gated function (callable only by the contract itself or by threshold vote) analogous to the `rescueERC20` pattern from the referenced report.
3. For `submit_participant_info`: move the excess-refund logic outside the `if attestation_storage_must_be_paid_by_caller` guard, or add an explicit `require(env::attached_deposit() == 0)` assertion on the re-submission branch to prevent silent deposit consumption.

---

### Proof of Concept

**Scenario (propose_update)**

1. Participant A calls `propose_update` with a 1.5 MB binary, attaching 40 NEAR.
2. Participant B calls `propose_update` with a different binary, attaching 40 NEAR.
3. Threshold participants vote for A's proposal; `vote_update` calls `do_update`.
4. `do_update` removes A's entry, then calls `self.entries.clear()` — B's entry is deleted.
5. B's 40 NEAR is now in the contract's liquid balance. B has no mechanism to recover it. [7](#0-6) 

**Scenario (submit_participant_info)**

1. An existing participant re-submits their attestation after a TEE image upgrade, attaching 1 NEAR (as the node software does in tests).
2. `add_participant` returns `UpdatedExistingParticipant`; `is_new_attestation` is `false`; `caller_is_not_participant` is `false`.
3. `attestation_storage_must_be_paid_by_caller` is `false`; the refund block is skipped entirely.
4. The 1 NEAR deposit is absorbed into the contract with no refund. [8](#0-7)

```rust
        required_deposit(bytes_used(update))

    /// Propose an update given the new contract code and/or config.
    pub fn propose(&mut self, update: Update) -> UpdateId {
        let bytes_used = bytes_used(&update);

        let id = self.id.generate();
        self.entries.insert(id, UpdateEntry { update, bytes_used });

        id
    }
```

```markdown
## Contract Updates

Participants can propose and vote on contract updates (code or configuration changes). When an update receives sufficient votes and is executed (via the `vote_update` endpoint which calls `do_update` internally), all pending update proposals and votes are cleared as they are no longer be valid after the contract migration. The update ID counter is preserved across migrations as part of the contract state to avoid race conditions where multiple participants might propose updates with colliding IDs immediately after an upgrade.
```
