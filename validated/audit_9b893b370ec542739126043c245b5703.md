Audit Report

## Title
Bootstrap DoS: `highestRsethPrice == 0` Permanently Blocks Non-Manager `updateRSETHPrice()` When `pricePercentageLimit > 0` — (`contracts/LRTOracle.sol`)

## Summary
After deployment, `rsETHPrice` and `highestRsethPrice` both default to `0`. Once rsETH supply is non-zero, the `highestRsethPrice == 0` guard at line 225 is a no-op (it assigns `rsETHPrice = 0` to `highestRsethPrice`), leaving `highestRsethPrice` at `0`. Any computed `newRsETHPrice` will be positive, causing the threshold check to evaluate `priceDifference > pricePercentageLimit.mulWad(0)` → `newRsETHPrice > 0`, which is always true when `pricePercentageLimit > 0`. Every non-manager call to `updateRSETHPrice()` reverts with `PriceAboveDailyThreshold`, and because the revert rolls back the no-op assignment, the contract is permanently stuck in this state until a manager intervenes.

## Finding Description
The execution path in `_updateRsETHPrice()` (`contracts/LRTOracle.sol`):

1. **Lines 218–222**: When `rsethSupply == 0`, both prices are seeded to `1 ether` and the function returns. This guard is bypassed once any user deposits.
2. **Lines 224–226**: `if (highestRsethPrice == 0) { highestRsethPrice = rsETHPrice; }` — since `rsETHPrice` is also `0` at this point, this is a no-op. `highestRsethPrice` remains `0`.
3. **Line 250**: `newRsETHPrice = totalETHInProtocol.divWad(rsethSupply)` — a positive value (~1 ether) reflecting real TVL.
4. **Lines 252–257**: `newRsETHPrice > 0 == highestRsethPrice` → enters the branch. `priceDifference = newRsETHPrice`. `pricePercentageLimit.mulWad(0) = 0`. `isPriceIncreaseOffLimit = (pricePercentageLimit > 0) && (newRsETHPrice > 0)` → **TRUE**.
5. **Lines 260–265**: Non-manager callers hit `revert PriceAboveDailyThreshold()`. The revert undoes the line-225 assignment, restoring `highestRsethPrice = 0`. Every subsequent non-manager call repeats identically.

The only escape is `updateRSETHPriceAsManager()` (lines 94–96), which calls the same `_updateRsETHPrice()` but the manager role check at line 263 passes, allowing the price to be set and `highestRsethPrice` to advance to a positive value.

## Impact Explanation
**Medium. Temporary freezing of funds.**

While `rsETHPrice` remains `0`, downstream protocol components that read it are impaired. `getExpectedAssetAmount()` in `LRTWithdrawalManager.sol` (line 593) computes `amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)`, returning `0` when `rsETHPrice = 0`. The `unlockQueue` operator function (lines 268–320) reads `rsETHPrice` via `_createUnlockParams` (line 847) and passes it to `_validatePrices`, which will reject any call with `minimumRsEthPrice > 0`. The permissionless price-update guarantee is broken for the entire window between first deposit and first manager intervention. The freeze is temporary because `updateRSETHPriceAsManager()` provides a recovery path.

## Likelihood Explanation
No attacker action is required. The trigger is the normal deployment sequence: deploy → admin sets `pricePercentageLimit > 0` → any user deposits (minting rsETH, making `rsethSupply > 0`) → any public caller invokes `updateRSETHPrice()` before the manager has ever called it. This is a latent initialization bug that fires automatically on the first real-world call to the public function. It is repeatable on every call until the manager intervenes.

## Recommendation
In the `highestRsethPrice == 0` bootstrap branch, seed `highestRsethPrice` with the freshly computed `newRsETHPrice` rather than the stale `rsETHPrice = 0`. Since `newRsETHPrice` is computed after this branch, the simplest fix is to skip the threshold check entirely when `highestRsethPrice == 0`:

```solidity
if (highestRsethPrice == 0) {
    // No historical peak — skip threshold check, seed with current price
    // newRsETHPrice will be assigned to highestRsethPrice below
} else {
    // existing threshold check logic
}
```

Alternatively, compute `newRsETHPrice` before the bootstrap branch and assign `highestRsethPrice = newRsETHPrice` directly. A third option is to require `initialize`/`reinitialize` to set `highestRsethPrice` to a non-zero sentinel (e.g., `1 ether`) so the bootstrap state is never `0`.

## Proof of Concept
```
Preconditions:
  rsETHPrice = 0, highestRsethPrice = 0
  pricePercentageLimit = 1e16 (1%, set by admin)
  rsethSupply > 0 (one user has deposited)
  Caller: any non-manager EOA

Call: updateRSETHPrice()
  → rsethSupply > 0: skip early-return branch (L218–222)
  → highestRsethPrice == 0: highestRsethPrice = rsETHPrice = 0 (no-op) (L224–226)
  → newRsETHPrice = totalETHInProtocol / rsethSupply ≈ 1e18 (L250)
  → newRsETHPrice (1e18) > highestRsethPrice (0): enter branch (L252)
  → priceDifference = 1e18 (L254)
  → pricePercentageLimit.mulWad(0) = 0 (L257)
  → isPriceIncreaseOffLimit = true (L256–257)
  → caller lacks MANAGER role → revert PriceAboveDailyThreshold (L263–264)
  → all state changes rolled back: highestRsethPrice = 0 still

State after: rsETHPrice = 0, highestRsethPrice = 0 — identical to before.
Every subsequent non-manager call repeats identically.

Recovery: manager calls updateRSETHPriceAsManager() → same path but L263 passes
  → highestRsethPrice set to newRsETHPrice (~1e18) → non-manager calls now succeed.
```
