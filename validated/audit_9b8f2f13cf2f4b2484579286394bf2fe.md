### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay — (`File: crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies that `response.payload_hash` is signed by the root key, but never validates that `payload_hash` actually commits to the original `request` stored in the pending map. A single malicious attested participant can replay a legitimately-obtained root-key signature (over `hash_B`) against a different pending request (`request_A`), permanently consuming it with a forged verification response.

### Finding Description

In `respond_verify_foreign_tx` (lines 692–754 of `crates/contract/src/lib.rs`), the contract performs the following checks:

1. Caller is an attested participant.
2. `response.signature` is a valid ECDSA signature over `response.payload_hash` using the **root** public key (no tweak).
3. The pending yield for `request` is resolved and the entry is removed from `pending_verify_foreign_tx_requests`. [1](#0-0) 

The critical missing check: the contract never verifies that `response.payload_hash` is the correct hash for the supplied `request`. The correct hash would be `SHA-256(borsh(ForeignTxSignPayloadV1 { request, extracted_values }))`. The contract knows `request` (it is the lookup key in `pending_verify_foreign_tx_requests`) but does not verify that `payload_hash` commits to it. [2](#0-1) 

The MPC node signs foreign-tx payloads with a **zero tweak** (root key, no derivation): [3](#0-2) 

This means any threshold signature produced for a foreign-tx request is a valid root-key signature that can be replayed into `respond_verify_foreign_tx` for a **different** pending request.

**Attack path (single malicious attested participant, below threshold):**

1. Attacker submits `verify_foreign_transaction` for transaction B. Honest nodes verify B and cooperatively produce threshold signature `sig_B` over `hash_B = SHA-256(borsh(ForeignTxSignPayloadV1{request_B, values_B}))`. As a participant, the attacker observes `sig_B` during the signing protocol.
2. A victim submits `verify_foreign_transaction` for transaction A (`request_A`), creating a pending yield.
3. Before any honest node calls `respond_verify_foreign_tx(request_A, ...)`, the attacker calls `respond_verify_foreign_tx(request_A, { payload_hash: hash_B, signature: sig_B })`.
4. The contract checks: is `sig_B` a valid signature over `hash_B` under the root key? **Yes.** It does not check whether `hash_B` commits to `request_A`.
5. `resolve_yields_for` drains and removes the pending entry for `request_A`, delivering the forged response to the caller. [4](#0-3) 

6. When honest nodes subsequently call `respond_verify_foreign_tx(request_A, { hash_A, sig_A })`, they receive `RequestNotFound` — the pending entry is gone. [5](#0-4) 

The SDK-side verifier (`ForeignChainSignatureVerifier::verify_signature`) does check `expected_payload_hash == response.payload_hash`, so callers using the SDK would detect the mismatch. However, the contract itself does not enforce this, and callers that trust the contract's on-chain validation without re-verifying the hash are deceived. [6](#0-5) 

### Impact Explanation

**High.** This is a forged foreign-chain verification / cross-request replay attack. A single malicious attested participant (below signing threshold) can:

- Permanently consume any pending `verify_foreign_transaction` request by resolving it with a forged `payload_hash`. The legitimate response can never be delivered (`RequestNotFound`).
- Deliver to the caller a `VerifyForeignTransactionResponse` whose `payload_hash` does not correspond to the original request, constituting a forged foreign-chain verification attestation.
- Callers that do not independently re-verify the `payload_hash` (e.g., off-chain consumers or contracts that trust the on-chain validation) accept the forged result, enabling invalid bridge execution.

This breaks the core safety invariant of the `verify_foreign_transaction` flow: that the returned `payload_hash` and signature attest to the MPC network having verified the specific foreign transaction in the original request.

### Likelihood Explanation

A single malicious attested participant can execute this attack without any threshold collusion. The attacker only needs to:

1. Be an attested participant (required to call `respond_verify_foreign_tx`).
2. Observe a threshold signature produced during any legitimate foreign-tx signing round (the attacker participates in the protocol and sees the output).
3. Race to call `respond_verify_foreign_tx` for a victim's pending request before honest nodes do.

The race window is the block interval between when the MPC nodes compute the signature and when the first honest node submits the on-chain response. This is a realistic window in a live network.

### Recommendation

The contract must verify that `response.payload_hash` commits to the original `request`. Since `extracted_values` are not stored on-chain, the simplest fix is to require the responding node to include the full `ForeignTxSignPayload` in the response, and have the contract:

1. Recompute `expected_hash = SHA-256(borsh(payload))`.
2. Assert `expected_hash == response.payload_hash`.
3. Assert `payload.request == request` (the original pending request).
4. Then verify the signature over `expected_hash`.

Alternatively, store the `payload_hash` in the pending request map at submission time (if nodes commit to it before signing), or add a binding between `request` and `payload_hash` that the contract can verify without knowing `extracted_values`.
