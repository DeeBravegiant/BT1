### Title
Unbounded `fee_proposal_fri` During Startup Window Allows Proposer to Corrupt `fee_actual` and Future L2 Gas Prices - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary
`is_proposal_init_valid` explicitly skips the `fee_proposal_fri` bounds check whenever `fee_actual` is `None` â€” which is the case for every block during the first `fee_proposal_window_size` heights. A proposer who controls any block in that window can inject an arbitrary `fee_proposal_fri` (e.g. `u128::MAX`) that all validators accept without restriction. That value is then committed to `fee_proposals_window`, feeds into `compute_fee_actual`, and becomes the floor for `l2_gas_price` in every subsequent block, permanently corrupting the fee market.

### Finding Description

In `is_proposal_init_valid`, the bounds check is guarded by a pattern-match that silently does nothing when `fee_actual` is `None`:

```rust
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(...);
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound { ... }
}
```

`fee_actual` is `None` whenever `height < fee_proposal_window_size` (because `compute_fee_actual` returns `None` when the window is incomplete). During that entire startup window every validator computes `fee_actual = None` and therefore skips the check, so any `fee_proposal_fri` value â€” including `u128::MAX` â€” passes `is_proposal_init_valid` and the proposal is accepted.

After acceptance, `finalize_decision` unconditionally records the proposer-supplied value:

```rust
self.record_fee_proposal(height, init.fee_proposal_fri);
```

`record_fee_proposal` inserts it into `fee_proposals_window`. Once `window_size` blocks have been committed, `compute_fee_actual` returns the median of those stored values. If the attacker controlled a majority of the startup blocks, the median is `u128::MAX`. That value then flows into `calculate_next_l2_gas_price_for_fin`:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),  // fa = u128::MAX
    None => config_min,
};
calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
```

`effective_min = u128::MAX` becomes the floor passed to the EIP-1559 adjustment, pinning `l2_gas_price` at `u128::MAX` for every subsequent block.

### Impact Explanation

Once `l2_gas_price` is set to `u128::MAX`, every transaction whose `max_price_per_unit` for L2 gas is less than `u128::MAX` is rejected at execution time. This is a permanent, chain-wide denial of the fee market: no ordinary user transaction can be included, and the corrupted `fee_actual` persists in `fee_proposals_window` for `window_size` more blocks even after
