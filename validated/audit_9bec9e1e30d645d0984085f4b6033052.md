Audit Report

## Title
Contract Withdrawers Unable to Receive ETH Have Funds Permanently Frozen After rsETH Is Burned - (File: contracts/LRTWithdrawalManager.sol)

## Summary

When a smart contract without a `receive()` or `fallback()` function calls `initiateWithdrawal`, the rsETH is irreversibly burned during the operator-triggered `unlockQueue` phase. Every subsequent `completeWithdrawal` call reverts with `EthTransferFailed` because the low-level ETH push to the originating contract fails. The revert restores the withdrawal request state, but the rsETH burned in the prior transaction cannot be restored, and no alternative delivery path exists, permanently freezing both the burned rsETH and the corresponding ETH held in `LRTWithdrawalManager`.

## Finding Description

The withdrawal lifecycle spans two separate transactions:

**Phase 2 — `unlockQueue` (operator-triggered):**
At line 305, `IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned)` permanently destroys the rsETH held by the contract. At line 307, `unstakingVault.redeem(asset, assetAmountUnlocked)` pulls the corresponding ETH into `LRTWithdrawalManager`. These two operations are irreversible and occur in a transaction separate from the user's claim.

**Phase 3 — `completeWithdrawal` (user-triggered):**
`_processWithdrawalCompletion` (L699–738) pops the nonce at L705, deletes the request at L712, decrements `unlockedWithdrawalsCount` at L717, and finally calls `_transferAsset(asset, user, request.expectedAssetAmount)` at L734. Inside `_transferAsset` (L876–883):

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
```

If `user` is a contract without a `receive()` function, `sent == false` and the entire Phase 3 transaction reverts. The revert restores the nonce, the `withdrawalRequests` entry, and the `unlockedWithdrawalsCount` decrement — but the rsETH burned in Phase 2 is gone forever.

**No escape hatch exists:**
- `completeWithdrawalForUser` (L192–204) calls the same `_processWithdrawalCompletion(asset, user, referralId)` with the same hardcoded `user` address; its own NatSpec at L191 acknowledges "Not expected to be used for ETH."
- `sweepRemainingAssets` (L395–414) is blocked by the check `if (hasUnlockedWithdrawals(asset)) revert PendingWithdrawalsExist()` at L403, and `hasUnlockedWithdrawals` returns `true` as long as `unlockedWithdrawalsCount[asset] > 0` (L629–631), which remains nonzero while the stuck request exists.
- There is no `recipient` parameter, no position-transfer function, and no admin override to redirect ETH delivery.

## Impact Explanation

**Critical — Permanent freezing of funds.** The user's rsETH is irreversibly burned in Phase 2. The ETH allocated to them is held in `LRTWithdrawalManager` but can never be delivered: every Phase 3 call reverts, `sweepRemainingAssets` is blocked, and no redirect mechanism exists. Both the burned rsETH and the locked ETH are permanently lost to the user.

## Likelihood Explanation

**Medium.** Smart contracts holding rsETH and interacting with the withdrawal system are a realistic and expected use case: protocol-owned vaults, DAO treasuries, yield aggregators, and multisig-controlled strategies all commonly hold liquid restaking tokens. Many such contracts omit a `receive()` function by design (e.g., pure ERC-20 vaults, proxy contracts whose implementation does not accept ETH). No special privilege or front-running is required — the user simply needs to call `initiateWithdrawal` from such a contract.

## Recommendation

Add a `recipient` parameter to `initiateWithdrawal` (or to `WithdrawalRequest`) so the caller can designate a separate ETH-receiving address at initiation time:

```solidity
function initiateWithdrawal(
    address asset,
    uint256 rsETHUnstaked,
    address recipient,       // address that will receive the ETH/LST
    string calldata referralId
) external ...
```

Store `recipient` inside `WithdrawalRequest` and use it in `_processWithdrawalCompletion` instead of `user`. Alternatively, add a `transferWithdrawalRecipient(address asset, uint256 nonce, address newRecipient)` function that lets the current owner of a withdrawal request redirect its delivery address before `unlockQueue` is called.

## Proof of Concept

1. Deploy `VaultContract` (no `receive()`) holding rsETH. Call:
   ```solidity
   LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
   ```
   rsETH is transferred to the withdrawal manager; `withdrawalRequests[requestId]` is created keyed to `VaultContract`.

2. Operator calls `unlockQueue(ETH_TOKEN, ...)`.
   - L305: `IRSETH(...).burnFrom(address(this), rsETHBurned)` — rsETH **permanently burned**.
   - L307: `unstakingVault.redeem(ETH_TOKEN, assetAmountUnlocked)` — ETH pulled into `LRTWithdrawalManager`.

3. `VaultContract` calls `completeWithdrawal(ETH_TOKEN, "")`.
   - `_processWithdrawalCompletion` reaches L734: `_transferAsset(ETH_TOKEN, VaultContract, amount)`.
   - L878: `(bool sent,) = payable(VaultContract).call{ value: amount }("")` → `sent == false`.
   - L879: `revert EthTransferFailed()` — entire transaction reverts.

4. State after revert: `withdrawalRequests[requestId]` restored, `unlockedWithdrawalsCount[ETH_TOKEN] > 0`, `hasUnlockedWithdrawals` returns `true`. `sweepRemainingAssets` reverts with `PendingWithdrawalsExist`. The rsETH from step 2 is gone. The ETH in the withdrawal manager is permanently frozen.

**Foundry test plan:** Deploy a mock `VaultContract` with no `receive()`, fund it with rsETH, call `initiateWithdrawal`, simulate `unlockQueue` (burn rsETH, transfer ETH to manager), then assert that `completeWithdrawal` reverts with `EthTransferFailed` and that the ETH balance of `LRTWithdrawalManager` remains unchanged across repeated calls.
