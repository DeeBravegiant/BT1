### Title
Harvester Can Silently Redirect Farmer Reward to Arbitrary Address Without Notification — (`chia/farmer/farmer_api.py`)

### Summary

A connected harvester can redirect the farmer's 1/8-of-block-reward (`farmer_reward_puzzle_hash`) to any arbitrary address by setting `farmer_reward_address_override` in `RespondSignatures` without having declared it in the preceding `NewProofOfSpace` message. The farmer's only notification mechanism is gated on the `NewProofOfSpace` field, not on the `RespondSignatures` field where the override is actually applied, creating a silent payout-redirection path.

### Finding Description

CHIP-22 introduced `farmer_reward_address_override` as an opt-in fee mechanism: a third-party harvester declares in `NewProofOfSpace` that it intends to take the farmer reward as a fee, the farmer logs a warning, and the override is applied. However, the notification and the actual override are decoupled across two separate protocol messages.

**Step 1 — Notification gate (only on `NewProofOfSpace`):**

In `farmer_api.py`, the farmer only calls `notify_farmer_reward_taken_by_harvester_as_fee` when the `NewProofOfSpace` message carries a non-`None` override:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [1](#0-0) 

Even when triggered, `notify_farmer_reward_taken_by_harvester_as_fee` only emits log messages — it never blocks or rejects the proof: [2](#0-1) 

**Step 2 — Actual override (on `RespondSignatures`, unchecked):**

When the harvester's signature response arrives, `_process_respond_signatures` unconditionally replaces the farmer reward address with whatever the harvester supplies in `RespondSignatures.farmer_reward_address_override`, with no cross-check against the `NewProofOfSpace` field:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
``` [3](#0-2) 

The `farmer_reward_address` is then passed directly into `DeclareProofOfSpace` as `farmer_puzzle_hash`: [4](#0-3) 

The full node uses this value verbatim when constructing the unfinished block's `FoliageBlockData.farmer_reward_puzzle_hash`: [5](#0-4) 

**The gap:** A harvester can send `NewProofOfSpace` with `farmer_reward_address_override = None` (triggering no notification), then send `RespondSignatures` with `farmer_reward_address_override = <attacker_address>`. The farmer receives no log warning, no rejection, and the block is farmed with the attacker's address as the farmer reward destination.

The `RespondSignatures` protocol message explicitly carries this optional field: [6](#0-5) 

### Impact Explanation

Every block won by the farmer yields a base farmer reward (`calculate_base_farmer_reward(height)` — currently 0.25 XCH on mainnet). A malicious harvester silently redirects this reward to an address it controls. The farmer's wallet receives nothing for those blocks. This is an unauthorized payout redirection of XCH rewards, matching the **High** impact tier: "Bypass of … authorization that enables … payout redirection." [7](#0-6) 

### Likelihood Explanation

Any harvester connected to a farmer — including third-party harvesting services — can exploit this. No key compromise, admin access, or cryptographic break is required. The harvester only needs to hold valid plots that pass the proof-of-space quality check. The farmer has no in-band signal that the redirect occurred; the only observable effect is missing rewards in their wallet.

### Recommendation

In `_process_respond_signatures`, validate that `response.farmer_reward_address_override` matches the value declared in the corresponding `NewProofOfSpace` (stored alongside the proof in `self.farmer.proofs_of_space`). If the `RespondSignatures` carries an override that was not declared in `NewProofOfSpace`, reject the response or at minimum emit a warning and ignore the override. Additionally, the fee-quality threshold check in `notify_farmer_reward_taken_by_harvester_as_fee` should be enforced (not merely logged) before any override is accepted.

### Proof of Concept

1. Malicious harvester connects to a farmer.
2. Harvester finds a valid proof of space for a signage point.
3. Harvester sends `NewProofOfSpace` with `farmer_reward_address_override = None` — no notification is triggered on the farmer.
4. Farmer sends `RequestSignatures` to the harvester.
5. Harvester responds with `RespondSignatures` where `farmer_reward_address_override = <attacker_puzzle_hash>`.
6. `_process_respond_signatures` sets `farmer_reward_address = attacker_puzzle_hash` with no validation.
7. Farmer sends `DeclareProofOfSpace(farmer_puzzle_hash=attacker_puzzle_hash, ...)` to the full node.
8. Full node creates an unfinished block with `FoliageBlockData.farmer_reward_puzzle_hash = attacker_puzzle_hash`.
9. Block is finalized; the farmer reward coin is created at `attacker_puzzle_hash`.
10. The farmer's wallet receives 0 XCH for the block; the attacker receives the full 0.25 XCH farmer reward.

    def notify_farmer_reward_taken_by_harvester_as_fee(
        self, sp: farmer_protocol.NewSignagePoint, proof_of_space: harvester_protocol.NewProofOfSpace
    ) -> None:
        """
        Apply a fee quality convention (see CHIP-22: https://github.com/Chia-Network/chips/pull/88)
        given the proof and signage point. This will be tested against the fee threshold reported
        by the harvester (if any), and logged.
        """
        assert proof_of_space.farmer_reward_address_override is not None

        challenge_str = str(sp.challenge_hash)

        ph_prefix = self.config["network_overrides"]["config"][self.config["selected_network"]]["address_prefix"]
        farmer_reward_puzzle_hash = encode_puzzle_hash(proof_of_space.farmer_reward_address_override, ph_prefix)

        self.log.info(
            f"Farmer reward for challenge '{challenge_str}' "
            + f"taken by harvester for reward address '{farmer_reward_puzzle_hash}'"
        )

        fee_quality = calculate_harvester_fee_quality(proof_of_space.proof.proof, sp.challenge_hash)
        fee_quality_rate = float(fee_quality) / float(0xFFFFFFFF) * 100.0

        if proof_of_space.fee_info is not None:
            fee_threshold = proof_of_space.fee_info.applied_fee_threshold
            fee_threshold_rate = float(fee_threshold) / float(0xFFFFFFFF) * 100.0

            if fee_quality <= fee_threshold:
                self.log.info(
                    f"Fee threshold passed for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
            else:
                self.log.warning(
                    f"Invalid fee threshold for challenge '{challenge_str}': "
                    + f"{fee_quality_rate:.3f}%/{fee_threshold_rate:.3f}% ({fee_quality}/{fee_threshold})"
                )
                self.log.warning(
                    "Harvester illegitimately took a fee reward that "
                    + "did not belong to it or it incorrectly applied the fee convention."
                )
        else:
            self.log.warning(
                "Harvester illegitimately took reward by failing to provide its fee rate "
                + f"for challenge '{challenge_str}'. "
                + f"Fee quality was {fee_quality_rate:.3f}% ({fee_quality} or 0x{fee_quality:08x})"
            )
