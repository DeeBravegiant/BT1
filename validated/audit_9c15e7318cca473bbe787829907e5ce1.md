Audit Report

## Title
Aave WETH Pool High Utilization DoSes ETH Withdrawal Completions - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`_processWithdrawalCompletion` calls `_withdrawFromAave` without any pre-flight liquidity check or revert guard. When Aave's WETH pool utilization is near 100%, `aaveWETHGateway.withdrawETH` reverts, causing every `completeWithdrawal` call for ETH to revert. Users with already-unlocked withdrawal requests are temporarily unable to claim their ETH until Aave utilization drops or a privileged admin intervenes.

## Finding Description
In `_processWithdrawalCompletion` ( [1](#0-0) ), when the contract's idle ETH balance is insufficient, `_withdrawFromAave(amountNeeded)` is called at L724 with no try/catch and no pre-flight check against available Aave liquidity.

`_withdrawFromAave` unconditionally calls `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))` at L917 ( [2](#0-1) ). Aave's `withdrawETH` reverts when the aWETH contract holds less WETH than requested — the standard behavior when pool utilization is near 100%.

This is in direct contrast to the deposit direction: `unlockQueue` wraps its Aave deposit in a `try/catch` with an explicit comment acknowledging the pool-at-max-capacity failure mode ( [3](#0-2) ). No equivalent protection exists on the withdrawal path.

The contract even exposes `getAaveWithdrawableLiquidity()` ( [4](#0-3) ) confirming the team is aware that withdrawable liquidity can be constrained, yet this is never consulted before calling `_withdrawFromAave` in the completion path.

The only recovery mechanism is `emergencyWithdrawFromAave`, which requires PAUSER_ROLE — an unprivileged user cannot self-rescue.

## Impact Explanation
This is a **Medium — Temporary freezing of funds**. All ETH `completeWithdrawal` calls where the contract's idle balance is insufficient revert for the duration of the high-utilization condition. Users whose requests are already unlocked and past the withdrawal delay cannot claim their ETH. Funds are not permanently lost but are inaccessible without privileged admin action or natural market recovery.

## Likelihood Explanation
Aave WETH pool utilization naturally reaches high levels during market stress. Additionally, an attacker with sufficient capital can deliberately borrow WETH from Aave to push utilization to ~100%, then wait for users to attempt withdrawal completions — a griefing attack requiring no privileged access and no victim mistakes. The Aave integration is confirmed active (`isAaveIntegrationEnabled = true`). The condition is externally triggerable and repeatable.

## Recommendation
Mirror the deposit-side pattern: wrap the `_withdrawFromAave` call in `_processWithdrawalCompletion` with a pre-flight liquidity check using `getAaveWithdrawableLiquidity()`, or use a `try/catch`. If Aave liquidity is insufficient, revert with a descriptive error (e.g., `InsufficientAaveLiquidity`) so the user knows to retry later, rather than propagating an opaque Aave revert. Example:

```solidity
uint256 aaveLiquidity = getAaveWithdrawableLiquidity();
if (aaveLiquidity < amountNeeded) revert InsufficientAaveLiquidity();
_withdrawFromAave(amountNeeded);
```

Alternatively, cap the withdrawal to `min(amountNeeded, aaveLiquidity)` and revert with `InsufficientLiquidityForWithdrawal` if the resulting balance is still insufficient.

## Proof of Concept
1. `isAaveIntegrationEnabled = true`; the contract has deposited ETH into Aave via `_depositToAave`, so `address(this).balance` is low.
2. An attacker (or natural market) causes Aave WETH pool utilization to reach ~100% by borrowing most available WETH.
3. A user whose ETH withdrawal request is unlocked and past `withdrawalDelayBlocks` calls `completeWithdrawal(ETH_TOKEN, ...)`.
4. `_processWithdrawalCompletion` reaches L722: `address(this).balance < request.expectedAssetAmount` → true.
5. `_withdrawFromAave(amountNeeded)` is called at L724; internally `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))` is called at L917.
6. Aave reverts because `IERC20(WETH_ADDRESS).balanceOf(address(aaveAWETH)) < withdrawnAmount`.
7. The entire `completeWithdrawal` transaction reverts. The user cannot claim their ETH.
8. All ETH withdrawal completions remain blocked until Aave utilization drops or a PAUSER_ROLE admin calls `emergencyWithdrawFromAave`.

A Foundry fork test against Aave mainnet can reproduce this by: (a) deploying/configuring the contract with Aave integration enabled, (b) using `deal` to drain the contract's idle ETH, (c) using a flash loan or whale account to borrow WETH until utilization ≥ 99%, then (d) asserting that `completeWithdrawal` reverts.

```text
    function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
        if (amount == 0) return 0;

        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance == 0) revert InsufficientAaveBalance();

        // Only withdraw up to the principal amount (don't use accrued interest for user withdrawals)
        uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave ? aaveBalance : totalETHDepositedToAave;

        withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
        if (withdrawnAmount == 0) return 0;

        aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
        totalETHDepositedToAave -= withdrawnAmount;

        emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
    }
```
