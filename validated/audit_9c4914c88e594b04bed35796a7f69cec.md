Audit Report

## Title
Missing Staleness Check on Chainlink Price Feed Allows Stale Prices to Drive Deposits and Withdrawals - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice` calls `latestRoundData()` but silently discards the `updatedAt` timestamp, meaning a Chainlink feed that has stopped updating is treated as live. Every deposit and withdrawal in the protocol is priced through this oracle, so a stale feed directly controls how much rsETH is minted or how many assets are returned. A sibling contract in the same repository (`ChainlinkOracleForRSETHPoolCollateral`) correctly validates staleness, confirming the developers are aware of the pattern but did not apply it to the primary oracle.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice` at line 52 destructures `latestRoundData()` capturing only `price` and discarding `updatedAt`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No maximum-age check is present anywhere in the function or in the calling chain. The full deposit call chain is:

```
depositAsset()
  → _beforeDeposit()
    → getRsETHAmountToMint()
      → lrtOracle.getAssetPrice(asset)          // LRTOracle.sol:156-158
        → IPriceFetcher(assetPriceOracle[asset]).getAssetPrice(asset)
          → priceFeed.latestRoundData()          // ChainlinkPriceOracle.sol:52
            // updatedAt discarded — no staleness check
```

`LRTOracle.getAssetPrice` at line 156–158 is a pure delegation with no additional validation. `getRsETHAmountToMint` at line 520 computes `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()`, where `rsETHPrice` is a stored value updated separately by `updateRSETHPrice()`. The withdrawal path `getExpectedAssetAmount` at line 593 uses the same oracle call.

The `LRTOracle._updateRsETHPrice` function contains a `pricePercentageLimit` guard that can pause the protocol if the computed rsETH price moves beyond a threshold, but this is only triggered when `updateRSETHPrice()` is called and operates on the aggregate rsETH price — it does not protect individual asset price reads in the deposit/withdrawal flow in real time.

The `unlockQueue` function in `LRTWithdrawalManager` has operator-supplied min/max price bounds via `_validatePrices`, but this is restricted to `onlyAssetTransferOrOperatorRole` and does not protect the user-callable `initiateWithdrawal` path.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate` at lines 27–32 validates `answeredInRound < roundID` and `timestamp == 0`, demonstrating the developers know the staleness validation pattern but omitted it from the primary oracle.

## Impact Explanation

If a supported LST's Chainlink feed goes stale at a price higher than the true market price (e.g., after a slashing event or circuit-breaker pause), an attacker can:

1. Acquire the devalued LST at its true lower market price.
2. Call `depositAsset`, which prices the deposit using the stale inflated oracle price.
3. Receive more rsETH than the deposited LST is worth in ETH terms.
4. Redeem or sell the rsETH, extracting value from existing rsETH holders whose proportional share of TVL is diluted.

This constitutes direct theft of value from existing rsETH holders — matching the **Critical** allowed impact of "Direct theft of any user funds." The claim conservatively labels it High; the actual impact category is at minimum **High (Theft of unclaimed yield)** and plausibly Critical given that existing holders lose real principal value proportional to the price deviation and deposit size.

## Likelihood Explanation

Chainlink feeds can become stale during network congestion preventing oracle node transactions, circuit-breaker events where the deviation threshold is not met for extended periods, or oracle node failures. LST assets (stETH, ETHx) have historically experienced price deviations during market stress. The window between a real price move and a Chainlink update can be minutes to hours, which is sufficient for a monitoring attacker to act. The attacker requires no privileged access — only the ability to call `depositAsset` with a supported LST. **Likelihood: Medium.**

## Recommendation

Add a configurable staleness threshold and validate `updatedAt` in `ChainlinkPriceOracle.getAssetPrice`, consistent with the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
uint256 public constant MAX_STALENESS = 1 hours;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) = priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale round");
    require(updatedAt != 0, "Incomplete round");
    require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
    require(price > 0, "Invalid price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be configurable per asset to accommodate feeds with different heartbeat intervals.

## Proof of Concept

**Setup (Foundry fork test against mainnet):**

1. Fork mainnet at a block where the stETH/ETH Chainlink feed is live.
2. Deploy or reference the existing `ChainlinkPriceOracle` with the stETH feed configured.
3. Use `vm.mockCall` to make `latestRoundData()` return a stale `updatedAt` (e.g., `block.timestamp - 2 hours`) with a price of `1.0e18` (1 stETH = 1 ETH).
4. Set the true market price context: attacker acquires 1000 stETH at 0.85 ETH each (850 ETH cost) via a DEX.
5. Call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")` as the attacker.
6. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice` → returns stale `1.0e18`.
7. Attacker receives rsETH priced as if 1000 ETH was deposited, not 850 ETH.
8. Attacker redeems or sells rsETH, extracting ~150 ETH of value from existing holders.
