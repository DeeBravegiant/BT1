Audit Report

## Title
FIFO Queue Head-Blocking `break` in `_unlockWithdrawalRequests` Temporarily Freezes Subsequent Withdrawals - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
In `_unlockWithdrawalRequests`, the `break` at line 800 causes the entire unlock loop to halt when the head request's `payoutAmount` exceeds `availableAssetAmount`. Because the queue is strictly FIFO and always resumes from `nextLockedNonce[asset]`, any request behind an underfunded head remains permanently locked until the vault accumulates enough assets to satisfy the head. Users whose rsETH was already transferred in at `initiateWithdrawal` time cannot recover their tokens until the head is cleared.

## Finding Description
`_unlockWithdrawalRequests` iterates from `nextLockedNonce[asset]` up to `firstExcludedIndex`:

```solidity
// contracts/LRTWithdrawalManager.sol:790-815
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    if (availableAssetAmount < payoutAmount) break; // hard stop
    ...
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
```

`availableAssetAmount` is sourced from `unstakingVault.balanceOf(asset)` at call time (line 849). If the head request's `payoutAmount` exceeds this balance, the loop exits immediately and `nextLockedNonce[asset]` is not advanced. All subsequent requests remain locked.

`completeWithdrawal` enforces:
```solidity
// contracts/LRTWithdrawalManager.sol:707
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```
So any user whose nonce is at or behind the stuck head cannot complete their withdrawal.

`initiateWithdrawal` transfers rsETH into the contract immediately (line 166) and has no privileged role requirement — only `whenNotPaused`, `onlySupportedAsset`, and `onlySupportedStrategy`. The guard at line 170 (`expectedAssetAmount > getAvailableAssetAmount(asset)`) only checks total protocol assets minus already-committed amounts, not the vault's current liquid balance. A large request can therefore be queued even when the vault holds far less than the request's eventual `payoutAmount`.

**Exploit path:**
1. Attacker calls `initiateWithdrawal(asset, rsETH_large)` → assigned nonce N (head of locked queue).
2. Victim calls `initiateWithdrawal(asset, rsETH_small)` → assigned nonce N+1.
3. Operator calls `unlockQueue(asset, ...)` → `availableAssetAmount < payoutAmount_head` → `break` at line 800 → `nextLockedNonce[asset]` stays at N.
4. Victim's nonce N+1 ≥ `nextLockedNonce[asset]` → `revert WithdrawalLocked()` on every `completeWithdrawal` attempt.
5. Victim's rsETH is frozen until the vault accumulates enough to cover the head request.

## Impact Explanation
**Medium — Temporary freezing of funds.** Users behind the head request cannot complete withdrawals and cannot recover their rsETH (already transferred in at step 1 of `initiateWithdrawal`). The freeze persists across multiple `unlockQueue` calls for as long as the vault balance remains below the head request's `payoutAmount`, which can span multiple unstaking cycles.

## Likelihood Explanation
No special role is required; any rsETH holder can call `initiateWithdrawal`. The vault balance fluctuates with the unstaking cadence, making it realistic for a large head request to exceed the vault's liquid balance at unlock time. The attacker only needs to be first (or early) in the queue with a sufficiently large amount — no front-running, collusion, or oracle manipulation is needed. The condition is repeatable: each `unlockQueue` call re-hits the same `break` until the vault is sufficiently funded.

## Recommendation
Replace the `break` at line 800 with skip logic so that requests behind an underfunded head can still be processed:

```solidity
if (availableAssetAmount < payoutAmount) {
    unchecked { nextLockedNonce_++; }
    continue;
}
```

If strict FIFO ordering must be preserved for fairness, maintain a separate "skipped" set and allow the operator to re-attempt skipped requests once the vault is sufficiently funded, while still advancing `nextLockedNonce` past them so subsequent requests are not blocked.
