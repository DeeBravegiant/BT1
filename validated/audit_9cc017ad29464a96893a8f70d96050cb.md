### Title
Excess Deposit Permanently Stuck in Contract When Existing Participant Re-submits Attestation via `submit_participant_info` - (File: crates/contract/src/lib.rs)

---

### Summary

`submit_participant_info` is marked `#[payable]` and accepts any attached deposit, but its deposit-handling block is gated behind a condition that is `false` for the common case of an existing participant refreshing their attestation. Any NEAR tokens attached in that branch are silently absorbed by the contract with no refund path.

---

### Finding Description

`submit_participant_info` computes whether the caller must pay for storage:

```rust
let caller_is_not_participant = self.voter_account().is_err();
let is_new_attestation = matches!(
    attestation_insertion_result,
    ParticipantInsertion::NewlyInsertedParticipant
);

let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;

if attestation_storage_must_be_paid_by_caller {
    // ... check attached >= cost, then refund diff
    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
// ŌåÉ no else branch: deposit silently consumed when condition is false
``` [1](#0-0) 

When `attestation_storage_must_be_paid_by_caller` evaluates to `false` ŌĆö i.e., the caller is already a registered participant **and** the submission updates an existing attestation rather than inserting a new one ŌĆö the entire deposit-handling block is skipped. Because the function is `#[payable]`, the NEAR runtime accepts whatever deposit the caller attaches, but the contract never issues a refund. The deposit is credited to the contract's balance and is unrecoverable by the caller.

This is the direct analog of the reported Solidity pattern: an operation that accepts tokens/value but does not return the unused portion to the sender.

---

### Impact Explanation

Any NEAR tokens attached by an existing participant when re-submitting their attestation (e.g., during a TEE software upgrade cycle) are permanently locked in the MPC contract. There is no administrative withdrawal function and no subsequent code path that would return those funds. This breaks the accounting invariant that a `#[payable]` function must either consume or refund every yoctoNEAR it receives.

Severity: **Medium** ŌĆö matches the allowed scope criterion: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

Participants re-submit attestations during every TEE upgrade cycle. Operational scripts and tooling that call `submit_participant_info` commonly attach a small deposit as a safety margin (the same pattern used for `sign`, `request_app_private_key`, and `propose_update`, all of which do refund excess). A participant following the same deposit convention for this call will silently lose those tokens. The trigger requires no adversarial action ŌĆö it fires on any normal re-attestation call that includes a non-zero deposit.

---

### Recommendation

Add an unconditional refund for any attached deposit when `attestation_storage_must_be_paid_by_caller` is `false`:

```rust
} else {
    // Caller is an existing participant updating their attestation;
    // no storage cost applies, so refund the entire deposit.
    let attached = env::attached_deposit();
    if attached > NearToken::from_yoctonear(0) {
        Promise::new(account_id).transfer(attached).detach();
    }
}
```

Alternatively, restructure the function so that `require_deposit(NearToken::from_yoctonear(0), &account_id)` (or an equivalent helper) is always called, ensuring the refund path is never accidentally omitted for future code changes.

---

### Proof of Concept

1. Participant `alice.near` is already registered in the MPC network (`voter_account()` succeeds).
2. A TEE upgrade is announced; `alice.near` calls `submit_participant_info` with a new attestation and attaches `1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR) as a deposit margin.
3. `add_participant` returns `ParticipantInsertion::UpdatedExistingParticipant` (not `NewlyInsertedParticipant`), so `is_new_attestation = false`.
4. `caller_is_not_participant = false` because `alice.near` is already a voter.
5. `attestation_storage_must_be_paid_by_caller = false || false = false`.
6. The `if` block at line 826 is skipped entirely; no refund promise is created.
7. The call succeeds (`Ok(())`), the attestation is updated, and 1 NEAR is permanently credited to the contract's balance with no recovery mechanism. [2](#0-1)
