### Title
Unvalidated `builder` Field in `ProposalInit` Allows Malicious Proposer to Redirect All Block Sequencer Fees - (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

### Summary

`is_proposal_init_valid` validates many `ProposalInit` fields but never checks the `builder` field. That field is passed verbatim as `sequencer_address` into the block context, which is the sole recipient of every transaction fee in the block. A malicious proposer can set `builder` to an arbitrary address; all validators will accept the proposal and execute the block with that address as the fee recipient.

### Finding Description

`ProposalInit` carries a `builder` field (the block-builder / sequencer address): [1](#0-0) 

`is_proposal_init_valid` validates `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices (within margin), `starknet_version`, `version_constant_commitment`, and `fee_proposal_fri`. The `builder` field is absent from every check and is absent from `ProposalInitValidation` entirely: [2](#0-1) [3](#0-2) 

After `is_proposal_init_valid` returns `Ok`, `initiate_validation` calls `convert_to_sn_api_block_info(init)`, which maps `init.builder` directly to `sequencer_address`: [4](#0-3) 

That `sequencer_address` is the hard-coded recipient in every fee-transfer call executed by the blockifier: [5](#0-4) 

The proposer sets `builder` from its own static config: [6](#0-5) [7](#0-6) 

No validator-side reference value for `builder` is ever computed or compared. The TODO comment confirms the gap is known but unresolved: [6](#0-5) 

### Impact Explanation

Every transaction fee for the block is transferred to `block_context.block_info.sequencer_address`. Because `sequencer_address` is set from the unvalidated `init.builder`, a malicious proposer who sets `builder` to their own wallet address causes the batcher to credit all fees to that wallet. The `partial_block_hash` produced by the batcher already incorporates the spoofed `sequencer_address`, so the proposer's `ProposalFin.proposal_commitment` and the validator's locally-computed commitment agree — the `ProposalFinMismatch` guard does not fire: [8](#0-7) 

The result is a wrong fee-balance state committed to the chain: the legitimate sequencer receives nothing, and the attacker's address receives the full block reward. This matches the **Critical** impact category: "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."

Additionally, every contract that calls `get_sequencer_address` during that block receives the spoofed address, producing wrong execution results for any logic that branches on the sequencer identity.

### Likelihood Explanation

Any validator that wins a proposal slot can execute this with a one-line change to their `builder_address` config (or by patching `initiate_build`). No cryptographic material, no mempool access, and no special permissions beyond holding a validator slot are required. The attack is silent: no existing metric or log distinguishes a legitimate `builder` from a spoofed one.

### Recommendation

Add `builder` to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`. The expected value should be derived from the committee roster keyed on `init.proposer` (the TODO at line 806 already anticipates this). Until the committee lookup is available, the validator should at minimum reject any `builder` that differs from its own locally-configured `builder_address`, since in the current single-sequencer deployment all honest nodes share the same expected value.

### Proof of Concept

1. Malicious validator wins a proposal slot at height H.
2. In `initiate_build`, they set `init.builder = attacker_wallet` instead of the legitimate sequencer address.
3. The spoofed `ProposalInit` is streamed to all validators.
4. Each validator calls `is_proposal_init_valid` — `builder` is never read; the function returns `Ok`.
5. `initiate_validation` calls `convert_to_sn_api_block_info(init)`, producing `BlockInfo { sequencer_address: attacker_wallet, … }`.
6. The batcher executes all transactions; every `execute_fee_transfer` call sends fees to `attacker_wallet`.
7. The batcher's `partial_block_hash` already encodes `attacker_wallet` as sequencer, so `proposal_commitment_from(partial, fee_proposal)` matches the proposer's `ProposalFin.proposal_commitment`.
8. `built_block == received_fin.proposal_commitment` — no mismatch error.
9. Consensus decides on the block; the state diff records `attacker_wallet` balance increased by the full block fee, and the legitimate sequencer balance unchanged.
