### Title
Proposal Deposits for Non-Winning Updates Are Permanently Frozen When `do_update` Clears All Entries — (`File: crates/contract/src/update.rs`)

### Summary
`ProposedUpdates::do_update` clears every pending proposal entry when one proposal reaches threshold, but never refunds the storage-staking deposits paid by the proposers of the discarded entries. Those NEAR tokens remain in the contract's balance with no recovery path, permanently freezing participant funds.

### Finding Description
`propose_update` requires each proposer to attach a deposit proportional to the size of their update payload:

```rust
pub fn required_deposit(update: &Update) -> NearToken {
    required_deposit(bytes_used(update))
}
``` [1](#0-0) 

The deposit is accepted and only the excess above `required` is refunded to the proposer:

```rust
if let Some(diff) = attached.checked_sub(required)
    && diff > NearToken::from_yoctonear(0)
{
    Promise::new(proposer).transfer(diff).detach();
}
``` [2](#0-1) 

The `required` portion stays in the contract's balance as a storage-staking deposit for the entry. When threshold is reached and `do_update` executes, it removes the winning entry and then **clears all remaining entries** without issuing any refund:

```rust
pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
    let entry = self.entries.remove(id)?;

    // Clear all entries as they might be no longer valid
    self.entries.clear();
    self.vote_by_participant.clear();
    ...
}
``` [3](#0-2) 

Critically, `UpdateEntry` stores only `update` and `bytes_used` — it does **not** record the proposer's `AccountId` or the deposit amount:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
}
``` [4](#0-3) 

Because the proposer identity is never stored, there is no way to issue refunds at `do_update` time, and no other contract method exists to reclaim these deposits. The NEAR tokens freed from storage when entries are cleared flow back into the contract's own balance and are permanently inaccessible to the original depositors.

### Impact Explanation
Every time a contract upgrade or config-change vote completes, all competing proposals are silently discarded and their deposits are frozen. For a contract binary update the required deposit is substantial (the test constant `CURRENT_CONTRACT_DEPLOY_DEPOSIT` reflects a multi-NEAR amount for a full WASM binary). With N participants each proposing a different update, N−1 deposits are permanently lost. There is no administrative escape hatch: no `withdraw_deposit`, no `cancel_proposal`, and no migration path that would reconstruct the proposer-to-deposit mapping. This breaks the accounting invariant that storage-staking deposits are returned when the corresponding storage is freed, and results in permanent freezing of participant funds held by the MPC contract.

### Likelihood Explanation
The scenario is a normal operational pattern: participants routinely propose competing updates (the test `test_propose_update_contract_many` explicitly exercises multiple simultaneous proposals). Any upgrade cycle with more than one proposal triggers the loss. Because `UpdateEntry` was designed without a refund field from the start, the bug is structural and present in every execution of `do_update` that clears more than zero non-winning entries.

### Recommendation
Store the proposer `AccountId` and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, iterate over the entries being cleared and issue a `Promise::new(entry.proposer).transfer(entry.deposit)` for each one before calling `self.entries.clear()`. This restores the invariant that every depositor is made whole when their proposal is discarded.

### Proof of Concept
1. Participant A calls `propose_update` with a 500 KB contract binary, attaching the required deposit (~5 NEAR at current storage costs). The deposit is accepted; only excess is refunded.
2. Participant B calls `propose_update` with a different binary, also paying ~5 NEAR.
3. Participants vote for B's proposal until threshold is reached; `vote_update` calls `do_update(&B_id, gas)`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry is deleted with no refund issued.
5. A's ~5 NEAR is now in the contract's balance with no recovery path. Querying `proposed_updates()` shows an empty list; no contract method accepts a claim from A. [5](#0-4) [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

    #[handle_result]
    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }
