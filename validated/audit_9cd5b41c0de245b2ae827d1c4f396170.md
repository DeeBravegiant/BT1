### Title
Uninitialized Epoch State Causes Stale Presignatures to Survive Epoch Transition on First Run - (File: crates/node/src/assets/cleanup.rs)

### Summary
The `delete_stale_triples_and_presignatures` function in the MPC node's asset cleanup path treats the absence of persisted epoch data as a signal to unconditionally keep all existing assets. On the very first call after epoch data is absent from the database (e.g., after a partial DB reset or migration that clears the epoch marker but leaves asset rows intact), all presignatures and triples from a prior epoch survive into the new epoch without any validation. This is a direct analog of the reported "uninitialized variable" class: the first execution behaves differently from all subsequent ones, and the difference has a concrete impact on the request lifecycle.

### Finding Description
`delete_stale_triples_and_presignatures` is called every time the node enters the Running state. [1](#0-0) 

```rust
let asset_cleanup: AssetCleanup = match get_epoch_data(db)? {
    None => AssetCleanup::Keep,          // ← first run: no epoch in DB
    Some(previous_epoch_data) => cleanup_behavior(&previous_epoch_data, &current_epoch_data),
};
```

When `get_epoch_data` returns `None` (the `DBCol::EpochData` / `EPOCH_ID_KEY` entry is absent), the function unconditionally selects `AssetCleanup::Keep` and skips all deletion logic. [2](#0-1) 

On a truly fresh deployment this is harmless because the asset columns are also empty. However, if the epoch-data marker is absent while asset rows remain — a realistic outcome of a node migration, a partial DB reset, or a backup-restore that restores assets but not the metadata column — the function keeps every presignature and triple without comparing them against the current epoch ID or participant set.

The subsequent write of the current epoch data to the DB means this silent keep-all happens **only once** (the first call after the marker is absent). Every subsequent call enters the `cleanup_behavior` branch and performs the correct epoch comparison. [3](#0-2) 

The documentation explicitly records this asymmetry:

> | First run (no epoch in DB) | Keep all assets |
> | **Epoch ID changed** (resharing occurred) | **Delete all** triples and presignatures | [4](#0-3) 

### Impact Explanation
Presignatures incorporate key shares and are cryptographically invalidated by resharing. [5](#0-4) 

If stale presignatures from epoch E survive into epoch E+1 because the epoch-data marker was absent on startup:

1. The leader node dequeues a stale presignature via `take_owned()` and broadcasts its ID to followers.
2. Followers attempt `take_unowned(id)` for their share of the epoch-E presignature. If their shares were cleaned up (or never existed in the new epoch), the signing protocol fails immediately.
3. If followers still hold the epoch-E shares, the signing protocol runs but produces a signature under the old key material. The contract's `respond` function verifies the signature against the **current** public key and rejects it as `InvalidSignature`. [6](#0-5) 

In either case, every signing request that consumes a stale presignature fails. The node exhausts its presignature store on failed attempts and then blocks indefinitely in `take_owned()` waiting for new presignatures that the (now-dead) generation loop may not supply. This breaks the request lifecycle and freezes the node's ability to lead signature computations, matching the "Medium" impact class: request-lifecycle manipulation that breaks production safety/accounting invariants.

### Likelihood Explanation
The trigger condition — epoch-data marker absent while asset rows exist — is not the normal steady-state. It requires one of:
- A node migration or backup-restore that restores `DBCol::Presignature` / `DBCol::TripleV2` rows but not `DBCol::EpochData`.
- A partial DB reset that clears only the metadata column.
- A legacy node upgrade path where the old DB format did not write the `EpochData` JSON entry (the test `get_epoch_data__should_bail_on_legacy_8_byte_entry` confirms a legacy 8-byte format existed). [7](#0-6) 

The legacy format case is particularly relevant: a node upgrading from the old 8-byte epoch-ID format will find `get_epoch_data` returning an error (not `None`), so it panics rather than silently keeping assets. But any migration path that simply deletes the old key without writing the new JSON format would produce the `None` branch and trigger the silent keep-all.

Likelihood is **Low-Medium**: not reachable by an unprivileged external caller in normal operation, but reachable during node lifecycle events (migration, restore) that are part of documented production operations.

### Recommendation
Replace the unconditional `AssetCleanup::Keep` on first run with a conservative `AssetCleanup::DeleteAll` (or at minimum `AssetCleanup::KeepOnly` restricted to the current participant set). A fresh node with no epoch data has no legitimate reason to possess presignatures from a prior epoch; keeping them silently is the wrong default. If preserving assets on genuine first deployment is desired, distinguish "truly fresh DB" (no asset rows at all) from "epoch marker missing but assets present" and apply deletion only in the latter case.

### Proof of Concept

1. Node runs in epoch E, accumulates presignatures in `DBCol::Presignature`.
2. Resharing completes; epoch advances to E+1.
3. Operator performs a node migration that restores the RocksDB asset columns from a backup but does **not** restore `DBCol::EpochData` (e.g., the backup predates the epoch-data write, or the migration script only copies asset columns).
4. Node restarts and calls `run_mpc` → `delete_stale_triples_and_presignatures`.
5. `get_epoch_data` returns `None` → `AssetCleanup::Keep` is selected.
6. All epoch-E presignatures survive. The function writes epoch E+1 data to the DB and returns.
7. The node begins leading signing requests, dequeuing epoch-E presignatures.
8. Each signing attempt either fails at the follower `take_unowned` step or produces a signature rejected by the contract's `respond` verification.
9. The presignature store is exhausted on failed attempts; the node's signing leadership is frozen until new presignatures are generated — if the generation loop is still alive. [8](#0-7) [9](#0-8)

pub fn delete_stale_triples_and_presignatures(
    db: &Arc<SecretDB>,
    current_epoch_data: EpochData,
    my_participant_id: primitives::ParticipantId,
    ecdsa_domain_ds: Vec<DomainId>,
    triple_thresholds: Vec<ReconstructionThreshold>,
) -> anyhow::Result<()> {
    let asset_cleanup: AssetCleanup = match get_epoch_data(db)? {
        None => AssetCleanup::Keep,
        Some(previous_epoch_data) => cleanup_behavior(&previous_epoch_data, &current_epoch_data),
    };

    // All cleanup work + the EpochData marker bump are staged on a single
    // `SecretDBUpdate` so they commit as one atomic RocksDB batch. If the
    // process dies before the commit, none of the deletes or the marker are
    // persisted, and the next startup re-runs the full cleanup unchanged.
    let mut update_writer = db.update();
    match asset_cleanup {
        AssetCleanup::Keep => {}
        AssetCleanup::DeleteAll => {
            update_writer.delete_all(DBCol::Presignature)?;
            update_writer.delete_all(DBCol::TripleV2)?;
        }
        AssetCleanup::KeepOnly(persitent_participants) => {
            // Triples — `TripleV2` uses a `[t as u64 BE]` prefix per store.
            for t in &triple_thresholds {
                clean_db::<PairedTriple>(
                    db,
                    &mut update_writer,
                    DBCol::TripleV2,
                    &persitent_participants,
                    my_participant_id,
                    &t.inner().to_be_bytes(),
                )?;
            }
            // Presignatures (per-domain, unchanged).
            for domain_id in &ecdsa_domain_ds {
                clean_db::<PresignOutputWithParticipants>(
                    db,
                    &mut update_writer,
                    DBCol::Presignature,
                    &persitent_participants,
                    my_participant_id,
                    &domain_id.0.to_be_bytes(),
                )?;
            }
        }
    }

    tracing::info!("Updating epoch data: {:?}.", current_epoch_data);
    let current_epoch_data_ser = serde_json::to_vec(&current_epoch_data)?;
    update_writer.put(DBCol::EpochData, EPOCH_ID_KEY, &current_epoch_data_ser);
    update_writer.commit()?;
    tracing::info!("Updated epoch id entry");
    Ok(())
}
