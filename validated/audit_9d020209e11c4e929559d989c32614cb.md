### Title
Precommit Vote Signature Lacks Chain ID, Height, and Round Binding — (`crates/apollo_signature_manager/src/signature_manager.rs`)

### Summary

`build_precommit_vote_message_digest` signs only `PRECOMMIT_VOTE || block_hash_bytes`. It omits `chain_id`, `height`, and `round`. A validator's precommit signature for block hash `X` is therefore valid on any chain, at any height, and in any round where the same raw hash value appears. This is the direct Sequencer analog of the external M-17 bug: a signed structure lacks a context identifier, enabling cross-context signature reuse.

### Finding Description

In `crates/apollo_signature_manager/src/signature_manager.rs`, the message digest for a precommit vote is constructed as:

```rust
fn build_precommit_vote_message_digest(block_hash: BlockHash) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let mut message = Vec::with_capacity(PRECOMMIT_VOTE.len() + block_hash.len());
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);
    MessageDigest(blake2s_to_felt(&message))
}
``` [1](#0-0) 

The signed payload is `b"PRECOMMIT_VOTE" || block_hash_bytes`. The `Vote` wire struct carries `height`, `round`, `vote_type`, and `voter` as separate fields:

```rust
pub struct Vote {
    pub vote_type: VoteType,
    pub height: BlockNumber,
    pub round: Round,
    pub proposal_commitment: Option<ProposalCommitment>,
    pub voter: ContractAddress,
    pub signature: RawSignature,
}
``` [2](#0-1) 

None of `height`, `round`, `vote_type`, or `chain_id` are committed into the signature. The `SignatureManager` component exposes `sign_precommit_vote(block_hash)` and `verify_precommit_vote_signature(block_hash, sig, pubkey)` — both operate only on the raw hash value. [3](#0-2) [4](#0-3) 

Compounding this, the Starknet block hash computation in the OS Cairo program also omits `chain_id`:

```cairo
func calculate_block_hash{poseidon_ptr: PoseidonBuiltin*}(...) -> felt {
    hash_update_single(BLOCK_HASH_VERSION);
    hash_update_single(block_info.block_number);
    hash_update_single(state_root);
    hash_update_single(block_info.sequencer_address);
    hash_update_single(block_info.block_timestamp);
    ...
}
``` [5](#0-4) 

Neither the block hash nor the vote signature includes a chain identifier. The developers themselves noted the concern in a TODO:

> `// TODO(noam.s): Consider wrapping each field in fixed delimiters (e.g. parentheses or tags) to avoid delimiter ambiguity across implementations; see apollo_propeller/signature.rs and PR review.` [6](#0-5) 

By contrast, the Propeller signing module correctly binds `committee_id` and `nonce` into its payload:

```rust
fn build_signed_payload(message_id: &MessageRoot, committee_id: CommitteeId, nonce: u64) -> Vec<u8> {
    [SIGNING_PREFIX, &message_id.0, &committee_id.0, &nonce.to_be_bytes(), SIGNING_POSTFIX].concat()
}
``` [7](#0-6) 

The precommit vote signature scheme does not follow this pattern.

### Impact Explanation

**Cross-chain replay.** A validator that participates in both mainnet and a testnet (or any two chains sharing the same `chain_id`-free block hash) produces a precommit signature `S` for block hash `X`. Because neither the signature nor the block hash encodes `chain_id`, `S` is cryptographically valid on any other chain where hash `X` appears. An attacker who can observe signatures on one chain and inject them on another can forge a quorum of precommit votes, causing consensus to decide on a block that was never legitimately voted on by the local committee.

**Height/round replay.** The `Vote` wire message carries `height` and `round` as unsigned fields that are not covered by the signature. If the same `ProposalCommitment` value appears at a different height or round (possible in reproposal scenarios or if an attacker engineers a collision), the signature from round R is valid at round R′. This breaks the Tendermint safety invariant that a precommit is bound to a specific (height, round, value) triple.

This matches the allowed impact: **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

### Likelihood Explanation

The `SignatureManager` is a production component wired into the node. `sign_precommit_vote` and `verify_precommit_vote_signature` are the live signing/verification paths for consensus votes. Cross-chain replay requires a validator to operate on two chains simultaneously (common for stakers on mainnet + testnet). The block hash omits `chain_id`, so the hash space is shared across chains. The attack is unprivileged: any network observer can collect signatures from one chain and replay them on another.

### Recommendation

1. Include `chain_id`, `height`, and `round` in `build_precommit_vote_message_digest`:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    chain_id: &ChainId,
    height: BlockNumber,
    round: Round,
) -> MessageDigest {
    let mut message = Vec::new();
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash.to_bytes_be());
    message.extend_from_slice(chain_id.as_bytes());
    message.extend_from_slice(&height.0.to_be_bytes());
    message.extend_from_slice(&round.to_be_bytes());
    MessageDigest(blake2s_to_felt(&message))
}
```

2. Propagate `chain_id`, `height`, and `round` through `sign_precommit_vote` and `verify_precommit_vote_signature`.

3. Update `SignatureManagerRequest::SignPrecommitVote` to carry these fields.

4. Consider adding `chain_id` to the Starknet block hash computation in `calculate_block_hash` in `block_hash.cairo`.

### Proof of Concept

**Setup:** Validator V holds key pair `(sk, pk)`. V participates in both chain A (mainnet) and chain B (testnet). Both chains share the same `chain_id`-free block hash computation.

1. At height H, round R on chain A, V signs a precommit for block hash `X`:
   - `digest = blake2s(b"PRECOMMIT_VOTE" || X.to_bytes_be())`
   - `sig = ecdsa_sign(sk, digest)`
   - V broadcasts `Vote { vote_type: Precommit, height: H, round: R, proposal_commitment: X, voter: V, signature: sig }`

2. An attacker observes this vote on chain A's P2P network.

3. On chain B, the attacker constructs a forged vote:
   - `Vote { vote_type: Precommit, height: H', round: R', proposal_commitment: X, voter: V, signature: sig }`
   - The signature `sig` is unchanged. `verify_precommit_vote_signature(X, sig, pk)` returns `true` because the digest is identical — it does not include `height`, `round`, or `chain_id`.

4. The attacker injects enough such replayed signatures (one per committee member who also participates on chain A) to reach quorum on chain B, causing chain B's consensus to decide on block `X` without any honest validator on chain B having actually voted for it. [1](#0-0) [4](#0-3) [5](#0-4)
