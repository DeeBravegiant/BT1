### Title
Attacker-Triggered Irreversible CAT-to-CRCAT Wallet Conversion via Unsolicited CRCAT Coin Corrupts Victim Wallet State — (File: `chia/wallet/wallet_state_manager.py`)

---

### Summary

When a wallet receives a Credential-Restricted CAT (CRCAT) coin whose TAIL hash matches an existing plain `CATWallet`, `handle_cat` in `wallet_state_manager.py` automatically and permanently converts the plain CAT wallet into a CRCAT wallet, adopting `authorized_providers` and `proofs_checker` values taken directly from the attacker-controlled on-chain coin spend. Unlike the analogous `convert_to_revocable` path (which guards against conversion when the wallet already holds coins), `convert_to_cr` has no such guard. An unprivileged attacker who holds any amount of the same CAT can trigger this conversion by sending a single crafted CRCAT coin hinted with the victim's inner puzzle hash, permanently corrupting the victim's wallet state and rendering their existing plain CAT coins unspendable through the wallet.

---

### Finding Description

**Entry point — `handle_cat` in `wallet_state_manager.py`:**

When a new coin state arrives, `determine_coin_type` fetches the parent spend and calls `match_cat_puzzle` to detect CATs. [1](#0-0) 

Inside `handle_cat`, the wallet checks whether the received coin's puzzle hash matches the expected standard CAT puzzle hash. If it does not match, it checks for special CAT subtypes (RCAT, CRCAT): [2](#0-1) 

If the coin is identified as a CRCAT, the code searches for an existing plain `CATWallet` with the same TAIL hash and calls `convert_to_cr` unconditionally: [3](#0-2) 

**Missing guard in `convert_to_cr`:**

`convert_to_cr` permanently replaces the wallet type in the DB and in-memory, adopting `authorized_providers` and `proofs_checker` from the attacker's coin spend, with no check for whether the wallet already holds plain CAT coins: [4](#0-3) 

**Contrast with `convert_to_revocable`:**

The analogous RCAT conversion path explicitly guards against this scenario: [5](#0-4) 

`convert_to_cr` has no equivalent guard.

**How the wallet discovers the attacker's coin:**

The wallet subscribes to inner puzzle hashes as hints. When a coin is hinted with the victim's inner puzzle hash, the full node delivers its coin state to the wallet. The attacker only needs to hint their CRCAT coin with the victim's known inner puzzle hash: [6](#0-5) 

**After conversion — victim's plain CAT coins become unspendable through the wallet:**

The CRCAT wallet's spend path requires a VC from the `authorized_providers` stored in the wallet info. Since those providers are now attacker-controlled, and the victim's existing coins are plain CATs (not CRCATs), the wallet cannot construct valid spends for them: [7](#0-6) 

---

### Impact Explanation

After the conversion, the victim's `CATWallet` is permanently replaced in both the SQLite DB and the in-memory wallet map with a `CRCATWallet` whose `authorized_providers` are attacker-controlled. The victim's existing plain CAT coins — which are valid on-chain and protected by the victim's private key — cannot be spent through the wallet because the CRCAT spend path requires a VC from the attacker's providers. This constitutes **High** impact: corruption of wallet sync state and puzzle ownership with direct security impact (victim loses ability to transact their CAT holdings through the wallet).

This maps to the allowed impact: *"Corruption of coin records, lineage, puzzle ownership, offer/trade settlement state, mempool or hint indexes, wallet sync state, pool membership state, or Data Layer root/store state with direct security impact."*

---

### Likelihood Explanation

- The attacker only needs to hold any nonzero amount of the target CAT (same TAIL hash). For tokens in `DEFAULT_CATS` (Spacebucks, Marmot, Stably USD, etc.), these are publicly traded and trivially acquirable.
- The victim's inner puzzle hash is public on-chain (visible from any prior spend or hint).
- No privileged access, leaked keys, or cryptographic break is required.
- The attack requires a single on-chain transaction. [8](#0-7) 

---

### Recommendation

Add the same lineage-store guard to `convert_to_cr` that exists in `convert_to_revocable`:

```python
@classmethod
async def convert_to_cr(
    cls,
    cat_wallet: CATWallet,
    authorized_providers: list[bytes32],
    proofs_checker: ProofsChecker,
) -> bool:
    if not await cat_wallet.lineage_store.is_empty():
        cat_wallet.log.error(
            "Received a CR-CAT to a CAT wallet that already has plain CAT coins; refusing conversion"
        )
        return False
    # ... existing conversion logic ...
    return True
```

Additionally, the caller in `handle_cat` should check the return value and return `None` on failure, consistent with the RCAT path. [9](#0-8) 

---

### Proof of Concept

1. Victim holds 100 units of CAT with TAIL hash `X` in a plain `CATWallet`.
2. Attacker acquires 1 unit of the same CAT (TAIL hash `X`).
3. Attacker constructs a CRCAT spend: wraps their 1-unit CAT coin in a CR layer with `authorized_providers = [attacker_provider_id]` and `inner_puzzle = victim_standard_puzzle`. The coin is hinted with the victim's inner puzzle hash.
4. Attacker broadcasts the spend bundle. The CRCAT coin lands on-chain.
5. The full node delivers the coin state to the victim's wallet (hint subscription match).
6. `handle_cat` is called. `cat_puzzle.get_tree_hash() != coin_state.coin.puzzle_hash` (CR layer present). `CRCAT.get_next_from_coin_spend` succeeds. `wallet_type = CRCATWallet`.
7. The code finds the victim's existing `CATWallet` with TAIL hash `X` and calls `CRCATWallet.convert_to_cr(found_cat_wallet, [attacker_provider_id], ...)`.
8. The victim's wallet DB entry is overwritten: type = `CRCAT`, `authorized_providers = [attacker_provider_id]`.
9. Victim attempts to spend their 100 plain CAT coins. The CRCAT wallet's `_generate_unsigned_spendbundle` requires a VC from `attacker_provider_id`. Victim has no such VC. Spend fails. Coins are locked in the wallet. [10](#0-9) [11](#0-10)

    @classmethod
    async def convert_to_cr(
        cls,
        cat_wallet: CATWallet,
        authorized_providers: list[bytes32],
        proofs_checker: ProofsChecker,
    ) -> None:
        replace_self = cls()
        replace_self.standard_wallet = cat_wallet.standard_wallet
        replace_self.log = logging.getLogger(cat_wallet.get_name())
        replace_self.log.info(f"Converting CAT wallet {cat_wallet.id()} to CR-CAT wallet")
        replace_self.wallet_state_manager = cat_wallet.wallet_state_manager
        replace_self.info = cls.wallet_info_type(
            cat_wallet.cat_info.limitations_program_hash, None, authorized_providers, proofs_checker
        )
        await cat_wallet.wallet_state_manager.user_store.update_wallet(
            WalletInfo(
                cat_wallet.id(), cat_wallet.get_name(), uint8(WalletType.CRCAT.value), bytes(replace_self.info).hex()
            )
        )
        updated_wallet_info = await cat_wallet.wallet_state_manager.user_store.get_wallet_by_id(cat_wallet.id())
        assert updated_wallet_info is not None
        replace_self.wallet_info = updated_wallet_info

        cat_wallet.wallet_state_manager.wallets[cat_wallet.id()] = replace_self
