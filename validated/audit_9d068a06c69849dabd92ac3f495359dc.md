Audit Report

## Title
Self-Recipient Swap Permanently Drains LP Token Reserves Into Protocol-Fee Surplus — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
The `swap` function in `MetricOmmPool` performs no validation that `recipient != address(this)`. When a caller sets `recipient = address(pool)`, the output-token transfer is an ERC20 self-transfer (pool balance unchanged), but `_executeSwap` has already decremented `binTotals.scaledToken(0|1)`. The resulting accounting gap is permanently classified as spread-fee surplus and extracted on the next `collectFees` call, irrecoverably reducing LP claims on the output token.

## Finding Description
**Root cause — missing recipient guard:**

`swap` accepts an arbitrary `recipient` address with no check against `address(this)`:

```solidity
// MetricOmmPool.sol line 217-224
function swap(
    address recipient,   // ← never validated
    bool zeroForOne,
    int128 amountSpecified,
    ...
) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128) {
```

**Exploit call path (`zeroForOne`, exact-input):**

1. `_executeSwap` is called (line 247-248); inside, at line 739, `binTotals.scaledToken1` is decremented by the full output amount.
2. `transferToken1(recipient, uint256(-amount1Delta))` is called (line 254). `transferToken1` is `IERC20(TOKEN1).safeTransfer(to, amount)` (line 569-570). When `to == address(this)`, this is a standard ERC20 self-transfer: the pool's `balance1()` is unchanged.
3. The `IncorrectDelta` guard (line 261) checks only that the pool received the correct **input** token (token0). It says nothing about the output token destination.

**Surplus extraction in `collectFees`:**

```solidity
// MetricOmmPool.sol lines 387-388
uint256 surplus1Scaled =
    balance1() * TOKEN_1_SCALE_MULTIPLIER
    - uint256(binTotals.scaledToken1)      // ← artificially low after attack
    - notionalFee1AmountScaled;
```

Because `balance1()` is unchanged but `binTotals.scaledToken1` was decremented, `surplus1Scaled` is inflated by exactly the swap output amount. `collectFees` distributes this surplus to `adminFeeDestination_` and `FACTORY` (lines 419-426), permanently removing it from LP-accessible reserves.

**Existing guards are insufficient:**
- `nonReentrant` prevents reentrancy but not a straightforward self-recipient call.
- `whenNotPaused` is irrelevant; the pool is not paused by default.
- `IncorrectDelta` only validates input token receipt.
- No `require(recipient != address(this))` exists anywhere in the contract.

## Impact Explanation
Every LP's proportional claim on token1 is computed from `binTotals.scaledToken1`. After the attack, that value is understated by the output amount of the attack swap. When LPs call `removeLiquidity`, they receive proportionally less token1. The missing token1 (now in surplus) is not accessible to LPs via any path — it is a permanent, irrecoverable loss of LP principal transferred to the protocol/admin fee pool. The attack is repeatable: each self-recipient swap drains another slice of LP reserves. This constitutes a direct loss of user principal above Sherlock thresholds (High severity).

## Likelihood Explanation
- **Trigger**: Any address that can call `swap` (no role restriction) can set `recipient = address(pool)`. The pool is not paused by default.
- **Cost to attacker**: The attacker must pay the input token (token0) and receives nothing in return. This is a griefing/competitive attack; the attacker's loss equals the oracle-priced input for the given output.
- **Motivation**: A competitor, a malicious factory operator who later calls `collectFees`, or any actor willing to spend capital to harm LPs.
- **Detection**: Silent — no event distinguishes a self-recipient swap from a normal one; `Swap` is emitted with the pool address as `recipient`.
- **Repeatability**: Unlimited; each call drains an additional slice.

Likelihood: **Medium** (unprivileged, zero-code-change trigger; attacker must spend capital).

## Recommendation
Add a recipient validation at the top of `swap`, before any state mutation:

```solidity
require(recipient != address(this), InvalidRecipient());
```

Alternatively, validate inside each output-transfer branch:

```solidity
// zeroForOne branch
if (amount1Delta < 0) {
    require(recipient != address(this), InvalidRecipient());
    transferToken1(recipient, uint256(-amount1Delta));
}
// !zeroForOne branch
if (amount0Delta < 0) {
    require(recipient != address(this), InvalidRecipient());
    transferToken0(recipient, uint256(-amount0Delta));
}
```

## Proof of Concept

```solidity
contract DrainLPReserves is IMetricOmmSwapCallback {
    IMetricOmmPoolActions pool;
    IERC20 token0;

    constructor(address _pool, address _token0) {
        pool = IMetricOmmPoolActions(_pool);
        token0 = IERC20(_token0);
    }

    function attack(int128 amountIn) external {
        token0.approve(address(pool), type(uint256).max);
        // recipient = address(pool): output token1 self-transfers, balance unchanged
        pool.swap(address(pool), true, amountIn, 0, "", "");
        // Post-call:
        //   pool.balance1() unchanged
        //   pool.binTotals.scaledToken1 decreased by output amount
        //   surplus1Scaled inflated by output amount
        //   next collectFees() extracts LP token1 as protocol/admin fees
    }

    function metricOmmSwapCallback(int256 amount0Delta, int256, bytes calldata) external {
        if (amount0Delta > 0) {
            token0.transfer(msg.sender, uint256(amount0Delta));
        }
    }
}
```

**Foundry test plan:**
1. Deploy pool with token0/token1, add liquidity.
2. Record `binTotals.scaledToken1` and `balance1()` before attack.
3. Call `attack(amountIn)` with `recipient = address(pool)`.
4. Assert `balance1()` is unchanged and `binTotals.scaledToken1` decreased.
5. Call `collectFees`; assert admin/protocol received token1 equal to the output amount.
6. Call `removeLiquidity`; assert LPs receive less token1 than deposited (loss confirmed).
