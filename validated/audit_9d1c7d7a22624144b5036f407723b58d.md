Audit Report

## Title
Protocol Treasury Fee Permanently Bypassed in `sendFunds()` - (File: contracts/FeeReceiver.sol)

## Summary
`FeeReceiver.initialize()` mandates a non-zero `_protocolFeePercentInBPS` and stores both the fee rate and treasury address, establishing a required protocol invariant. However, `sendFunds()` — the sole ETH distribution function — forwards 100% of the contract's balance to the deposit pool without computing or transferring any treasury fee. The stored `_legacyProtocolFeePercentInBPS` and `_legacyProtocolTreasury` values are never read during distribution, causing the treasury to receive zero yield on every MEV/execution-layer reward cycle.

## Finding Description
`initialize()` enforces `_protocolFeePercentInBPS != 0` at line 34 and stores the fee rate and treasury address at lines 40–42:

```solidity
_legacyProtocolTreasury = _protocolTreasury;
depositPool = _depositPool;
_legacyProtocolFeePercentInBPS = _protocolFeePercentInBPS;
```

`sendFunds()` at lines 53–58 reads only `address(this).balance` and routes the entire amount to the deposit pool:

```solidity
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Neither `_legacyProtocolTreasury` nor `_legacyProtocolFeePercentInBPS` is referenced anywhere in `sendFunds()` or any other execution path in the contract. The function carries no `onlyRole` or equivalent modifier, making it callable by any address at any time.

## Impact Explanation
Every MEV/execution-layer reward that accumulates in `FeeReceiver` is fully redirected to `LRTDepositPool` via `receiveFromRewardReceiver`, inflating the rsETH exchange rate. The treasury — the designated fee recipient — receives none of its entitled share. This is a concrete, ongoing **theft of unclaimed yield** (High severity), matching the allowed impact scope exactly. The loss is permanent per call and repeatable across every reward cycle.

## Likelihood Explanation
`sendFunds()` is a public, permissionless `external` function requiring no capital, role, or special timing. Any EOA or contract can call it. Actors with an incentive to inflate the rsETH rate (e.g., large rsETH holders) or to grief the protocol treasury can trigger it immediately after any reward accumulation. The call is cheap and can be automated, making sustained exploitation trivial.

## Recommendation
1. Implement the fee split inside `sendFunds()`:
   ```solidity
   uint256 fee = balance * _legacyProtocolFeePercentInBPS / 10_000;
   (bool ok,) = _legacyProtocolTreasury.call{value: fee}("");
   require(ok);
   ILRTDepositPool(depositPool).receiveFromRewardReceiver{value: balance - fee}();
   ```
2. Add `onlyRole(LRTConstants.MANAGER)` to `sendFunds()` to prevent permissionless triggering and allow the protocol to control distribution timing.

## Proof of Concept
1. Deploy `FeeReceiver` with `_protocolFeePercentInBPS = 1000` (10%) and a non-zero `_legacyProtocolTreasury`.
2. Send 10 ETH to `FeeReceiver` (simulating MEV rewards via the `receive()` fallback).
3. Any external address calls `FeeReceiver.sendFunds()`.
4. Observe: `LRTDepositPool.receiveFromRewardReceiver` is called with `value = 10 ETH`; `_legacyProtocolTreasury` balance increases by 0 ETH.
5. Expected: treasury receives 1 ETH (10% of 10 ETH); deposit pool receives 9 ETH.
6. Actual: treasury receives 0 ETH; deposit pool receives 10 ETH.

Foundry test outline:
```solidity
function test_sendFunds_bypassesTreasuryFee() public {
    vm.deal(address(feeReceiver), 10 ether);
    uint256 treasuryBefore = treasury.balance;
    vm.prank(address(0xdead)); // unprivileged caller
    feeReceiver.sendFunds();
    assertEq(treasury.balance - treasuryBefore, 0);          // treasury gets nothing
    assertEq(address(depositPool).balance, 10 ether);        // pool gets everything
}
```
