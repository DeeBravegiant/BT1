### Title
Puzzle-hash-granular additions cache in `request_and_validate_additions` allows a malicious peer to inject fabricated coins without Merkle proof — (`chia/wallet/util/wallet_sync_utils.py`)

### Summary

`request_and_validate_additions` caches proof results keyed on `(header_hash, puzzle_hash)`. Once any coin with a given puzzle_hash is Merkle-proven in a block, every subsequent coin claiming the same `(block, puzzle_hash)` pair bypasses proof verification entirely and is unconditionally accepted as block-included. A peer-controlling attacker who can arrange for one real coin with the target puzzle_hash to appear in a block can then inject an arbitrary number of fabricated coins sharing that puzzle_hash into the wallet's coin store without providing any Merkle proof.

### Finding Description

In `chia/wallet/util/wallet_sync_utils.py`, `request_and_validate_additions`: [1](#0-0) 

returns `True` immediately on a cache hit, skipping the `peer.call_api` and `validate_additions` calls entirely. The cache is populated at: [2](#0-1) 

The cache key is `(header_hash, puzzle_hash)`: [3](#0-2) [4](#0-3) 

The Merkle proof that `validate_additions` verifies covers the puzzle_hash's inclusion AND the hash of **all coin IDs** with that puzzle_hash: [5](#0-4) 

This means the proof cryptographically binds the complete set of coins with that puzzle_hash. However, the cache stores only `(header_hash, puzzle_hash)` — not the proven coin set. Any subsequent `CoinState` for a different coin with the same `puzzle_hash` in the same block hits the cache and returns `True` without verifying that the new coin is actually in the proven set.

A secondary bug: `add_to_additions_in_block` is called unconditionally regardless of whether `result` is `True` or `False`: [6](#0-5) 

This means even a failed validation populates the cache (though in practice the peer is disconnected on `False`, limiting this path).

The caller in `validate_received_state_from_peer` passes the result directly to the coin acceptance pipeline: [7](#0-6) 

### Impact Explanation

An attacker controlling an untrusted peer can corrupt the wallet's coin store with fabricated coins. Specifically:

- **Wallet balance inflation**: The wallet displays XCH/CAT balances that include non-existent coins.
- **Wallet sync state corruption**: `add_coin_states` is called with fabricated `CoinState` objects that passed validation, corrupting the coin store.
- **False offer/trade settlement state**: For puzzle hashes shared across CAT or offer coins, the wallet may record fabricated coins as received, causing incorrect trade state.

The wallet cannot spend fabricated coins (the full node would reject the spend), so direct fund theft is not possible. However, the wallet's accounting state is durably corrupted for the session, and any logic that acts on coin store contents (balance checks, offer settlement display, singleton tracking) operates on false data.

### Likelihood Explanation

Prerequisites:
1. The attacker must control the peer (untrusted full node the wallet connects to).
2. At least one real coin with the target `puzzle_hash` must exist in the target block — the attacker must be able to arrange this (e.g., by sending a real transaction to a shared puzzle hash such as a CAT tail hash or offer puzzle hash, or waiting for one to appear naturally).

Both conditions are realistic for a malicious peer operator. Puzzle hashes shared across multiple coins (CAT outer puzzles, offer settlement puzzles) are common and make the prerequisite trivial to satisfy.

### Recommendation

Change the cache key from `(header_hash, puzzle_hash)` to `(header_hash, coin_id)` so each coin requires its own proof. Alternatively, store the proven coin ID set alongside the cache entry and verify that any subsequent coin's ID is a member of the proven set before returning `True` from the cache. Also move `add_to_additions_in_block` inside the `if result:` branch so a failed validation does not populate the cache.
