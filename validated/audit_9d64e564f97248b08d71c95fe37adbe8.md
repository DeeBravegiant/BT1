### Title
Missing CKD Response Verification for `AppPublicKey` Variant Allows Single Attested Participant to Forge Key Material - (File: crates/contract/src/lib.rs)

### Summary

`respond_ckd` in the MPC contract applies a cryptographic pairing check (`ckd_output_check`) only when the request uses the `AppPublicKeyPV` variant. When the request uses the legacy `AppPublicKey` variant, the response arm is an empty no-op. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with any fabricated `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept and deliver the forged key material to the user.

### Finding Description

`respond` and `respond_verify_foreign_tx` both perform mandatory cryptographic verification before resolving any pending yield. `respond` derives the expected public key and verifies the ECDSA/EdDSA signature; `respond_verify_foreign_tx` verifies the ECDSA signature against the root key. Both checks are unconditional.

`respond_ckd` applies the analogous check — `ckd_output_check`, a BLS12-381 pairing equation — only for the `AppPublicKeyPV` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKey` requests the arm is empty. After this match, `resolve_yields_for` is called unconditionally, delivering whatever `CKDResponse` the caller supplied to every queued yield. [2](#0-1) 

The `AppPublicKey` variant carries only a single BLS12-381 G1 point (`pk1`). The pairing check `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` requires a G2 component (`pk2`), which is absent in this variant. The `AppPublicKeyPV` variant was introduced precisely to enable on-chain verification by supplying both `pk1` (G1) and `pk2` (G2). The legacy variant was never made verifiable on-chain, and the empty arm is the result. [3](#0-2) 

The existing unit test at line 3403 confirms the behaviour: it passes `CKDResponse { big_y: [1u8; 48], big_c: [2u8; 48] }` — arbitrary bytes — and the contract accepts it without complaint. [4](#0-3) 

### Impact Explanation

The `CKDResponse` fields `(big_y, big_c)` are BLS12-381 G1 points. The user recovers the application secret as:

```
secret = big_c − a · big_y
```

where `a` is the user's ephemeral private scalar. An attacker who sets `big_y = G1_identity` and `big_c = arbitrary_G1_point` forces `secret = arbitrary_G1_point` regardless of `a`. The user then derives their application key from this attacker-chosen point via HKDF, yielding a fully attacker-controlled key. Any asset, credential, or secret protected by that derived key is compromised.

This is a **threshold bypass**: the threshold-signature model requires that threshold-many honest participants contribute to produce a valid CKD output. For `AppPublicKey` requests, a single attested participant can short-circuit the entire threshold computation and deliver arbitrary key material.

**Impact: Critical** — unauthorized confidential key derivation output without the required participant authorization.

### Likelihood Explanation

- The `AppPublicKey` variant is the legacy/default variant documented in the README and used by the example CLI (`ckd-example-cli`).
- Any single attested participant can call `respond_ckd` with a fabricated response for any pending `AppPublicKey` request. No collusion is required.
- The attacker only needs to be an attested participant (TEE-attested node), which is a realistic adversary model for a Byzantine participant strictly below the signing threshold. [5](#0-4) 

### Recommendation

1. **Deprecate and reject `AppPublicKey` requests** at the contract level. Require all new CKD requests to use `AppPublicKeyPV`, which enables on-chain verification. Existing pending `AppPublicKey` requests should be allowed to time out.
2. If backward compatibility must be preserved, document explicitly that `AppPublicKey` responses carry **no on-chain integrity guarantee** and rely entirely on all participants being honest — a weaker security model than the rest of the protocol.
3. Mirror the unconditional verification pattern used in `respond` and `respond_verify_foreign_tx`: the response check must not be gated on a variant that the caller controls.

### Proof of Concept

1. User submits `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(pk1)`.
2. The contract enqueues the request in `pending_ckd_requests`.
3. A single attested participant (below threshold) calls `respond_ckd` with the matching `CKDRequest` and a fabricated `CKDResponse { big_y: G1_identity, big_c: attacker_chosen_point }`.
4. The `AppPublicKey` arm in `respond_ckd` is a no-op; no pairing check is performed.
5. `resolve_yields_for` resumes the user's yield with the forged response.
6. The user decrypts: `secret = attacker_chosen_point − a · G1_identity = attacker_chosen_point`.
7. The user derives their application key from `attacker_chosen_point` — a value entirely under the attacker's control. [6](#0-5)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
