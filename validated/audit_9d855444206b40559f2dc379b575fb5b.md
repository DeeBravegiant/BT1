Audit Report

## Title
Zero wrsETH Minted on Dust Deposits Due to Missing Zero-Amount Guard - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The `deposit` functions in `RSETHPoolV3` and `RSETHPoolNoWrapper` compute the wrsETH/rsETH output via integer division that truncates to zero for small inputs, yet no guard reverts when the result is zero. A user who sends a non-zero dust amount of ETH receives zero wrsETH in return, with their ETH permanently absorbed into the pool.

## Finding Description
In `RSETHPoolV3.deposit(string)`, the only input guard is `if (amount == 0) revert InvalidAmount()` at line 256. The share calculation at line 307 performs `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`, which truncates to zero whenever `amountAfterFee * 1e18 < rsETHToETHrate`. Since `rsETHToETHrate` starts at `1e18` and grows monotonically, any deposit of fewer than `rsETHToETHrate / 1e18` wei (i.e., 1 wei once any yield has accrued) produces `rsETHAmount = 0`. The function then unconditionally calls `wrsETH.mint(msg.sender, 0)` at line 262 without reverting. The same pattern exists in `RSETHPoolNoWrapper` at lines 285 and 241 (`rsETH.safeTransfer(msg.sender, 0)`). The existing `amount == 0` guard does not protect against zero output.

## Impact Explanation
The contract accepts a non-zero ETH payment and emits a `SwapOccurred` event but delivers zero wrsETH/rsETH to the caller. The deposited ETH is permanently held in the pool balance, accruing to existing holders. This matches the allowed Low impact: **"Contract fails to deliver promised returns, but doesn't lose value"** — the protocol retains the ETH while the depositor receives nothing.

## Likelihood Explanation
The condition is reachable by any unprivileged caller via the public `deposit` functions. No special role, front-running, or external dependency is required. The condition activates as soon as `rsETHToETHrate > 1e18`, which is the normal post-launch operating state. Any scripting error, test transaction, or wallet rounding that sends 1 wei will silently lose it.

## Recommendation
Add a zero-output guard immediately after the `viewSwapRsETHAmountAndFee` call in every deposit path across all pool variants:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
require(rsETHAmount != 0, "zero wrsETH minted");
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);
```

Apply the same guard to the token deposit path (`viewSwapRsETHAmountAndFee(amount, token)`) and to `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`.

## Proof of Concept
Assume `rsETHToETHrate = 1.05e18` (5% yield accrued):

1. Alice calls `RSETHPoolV3.deposit{value: 1}("")` (1 wei ETH).
2. `fee = 1 * feeBps / 10_000 = 0` (rounds down).
3. `amountAfterFee = 1`.
4. `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (Solidity truncation).
5. `wrsETH.mint(Alice, 0)` — Alice receives 0 wrsETH; transaction succeeds.
6. Alice's 1 wei is permanently held in the pool.

**Foundry fuzz test plan:** Fuzz `amount` over `[1, rsETHToETHrate / 1e18 - 1]` with a mocked `getRate()` returning `1.05e18`; assert that `viewSwapRsETHAmountAndFee(amount)` returns `rsETHAmount == 0` and that calling `deposit{value: amount}` succeeds without minting any tokens to the caller.
