Audit Report

## Title
Unprivileged `updateRate()` Broadcasts Zero Rate to All L2 Chains, Temporarily Freezing L2 Pool Deposits — (`contracts/cross-chain/RSETHMultiChainRateProvider.sol`)

## Summary
`MultiChainRateProvider.updateRate()` has no access control beyond `nonReentrant`, and `RSETHMultiChainRateProvider.getLatestRate()` returns `LRTOracle.rsETHPrice` with no zero-value guard. During the deployment window before `updateRSETHPrice()` is first called, `rsETHPrice` is `0`. Any caller can invoke `updateRate()` to broadcast `0` to every registered `CrossChainRateReceiver`, which stores it unconditionally. Any `RSETHPoolV3` using that receiver as its oracle then reverts with division-by-zero on every `deposit()` call until the admin broadcasts a corrected rate.

## Finding Description

**`updateRate()` is permissionless:**
`MultiChainRateProvider.updateRate()` carries only a `nonReentrant` modifier with no role check, allowing any external caller to trigger a rate broadcast. [1](#0-0) 

**`rsETHPrice` defaults to zero:**
`LRTOracle.rsETHPrice` is a plain `uint256` storage slot. `initialize()` never sets it, so it remains `0` until `updateRSETHPrice()` / `_updateRsETHPrice()` is first called successfully. [2](#0-1) [3](#0-2) 

**`getLatestRate()` has no zero guard:**
`RSETHMultiChainRateProvider.getLatestRate()` returns `ILRTOracle(rsETHPriceOracle).rsETHPrice()` directly with no validation. [4](#0-3) 

**Zero is encoded and sent to every receiver:**
`updateRate()` encodes `latestRate` (which is `0`) and calls `ILayerZeroEndpoint.send` for every registered receiver without any value validation. [5](#0-4) 

**Receiver stores zero unconditionally:**
`CrossChainRateReceiver.lzReceive()` decodes the payload and writes `rate = _rate` with no zero-value check. [6](#0-5) 

**Division by zero in `RSETHPoolV3`:**
`viewSwapRsETHAmountAndFee` calls `getRate()` (which reads from the receiver's `rate`) and divides by it. When `rate == 0`, this is a division-by-zero revert. [7](#0-6) 

**`deposit()` is blocked via `limitDailyMint`:**
The `deposit()` function applies the `limitDailyMint` modifier, which calls `viewSwapRsETHAmountAndFee` before any state change, causing every deposit to revert. [8](#0-7) 

Note: `viewSwapAssetToPremintedRsETH` does include a zero-rate guard (`if (rsETHToETHrate == 0) revert UnsupportedOracle()`), confirming the developers were aware of the zero-rate risk in some paths but missed it in the deposit path. [9](#0-8) 

## Impact Explanation

All L2 `RSETHPoolV3` deposits revert with division-by-zero for as long as the receiver holds `rate == 0`. This is a **Medium — Temporary freezing of funds**: new deposits into L2 pools are completely blocked. Existing pool assets (ETH/tokens held by the pool) remain accessible to the `BRIDGER_ROLE` via `moveAssetsForBridging`, so the freeze is scoped to deposit functionality. The admin can recover by calling `updateRSETHPrice()` on L1 (setting a non-zero price) and then calling `updateRate()` again to broadcast the correct value.

## Likelihood Explanation

The precondition — `rsETHPrice == 0` — is satisfied only during the window between `LRTOracle` deployment/upgrade and the first successful `updateRSETHPrice()` call. This window is short in a well-operated deployment but is real and requires no privilege to exploit. The attacker only needs to pay the LayerZero messaging fee (a small ETH cost). Once `rsETHPrice` is set to a non-zero value by `_updateRsETHPrice()`, it can never return to `0` through normal operation, so the attack window is one-time per deployment.

## Recommendation

1. Add a zero-rate guard in `updateRate()`:
   ```solidity
   uint256 latestRate = getLatestRate();
   require(latestRate > 0, "Rate must be non-zero");
   ```
2. Add a zero-rate guard in `CrossChainRateReceiver.lzReceive()`:
   ```solidity
   require(_rate > 0, "Received zero rate");
   ```
3. Consider restricting `updateRate()` to an authorized keeper role to prevent griefing entirely.

## Proof of Concept

```solidity
// Precondition: LRTOracle deployed, updateRSETHPrice() never called => rsETHPrice == 0

// 1. Deploy RSETHMultiChainRateProvider pointing at LRTOracle
// 2. Register an RSETHRateReceiver on a test L2
// 3. Anyone calls (paying only the LZ messaging fee):
rateProvider.updateRate{value: fee}();
// => CrossChainRateReceiver.rate == 0 on L2

// 4. On L2, any user attempts:
pool.deposit{value: 1 ether}("ref");
// => limitDailyMint modifier calls viewSwapRsETHAmountAndFee(1 ether)
// => rsETHToETHrate = getRate() = 0
// => rsETHAmount = amountAfterFee * 1e18 / 0  => division-by-zero revert

// Recovery: admin calls updateRSETHPrice() on L1, then updateRate() with correct rate
```
