### Title
Attacker-Controlled `hidden_puzzle_hash` Injected into RCAT Wallet via Sequential Type Detection in `handle_cat` — (`chia/wallet/wallet_state_manager.py`)

### Summary
The sequential wallet-type detection in `WalletStateManager.handle_cat` checks `match_revocation_layer` before any CR-CAT logic. Because `match_revocation_layer` is a pure structural check with no authenticity validation, an attacker who owns any amount of a DEFAULT_CATS token can craft a spend whose inner layer is a `REVOCATION_LAYER` carrying an attacker-chosen `hidden_puzzle_hash`. This causes the victim's wallet to create an RCAT wallet with the attacker's revocation key, enabling the attacker to spend (steal or brick) any coins tracked in that wallet.

### Finding Description
In `handle_cat` (`wallet_state_manager.py`, lines 1258–1275), the code uncurries the parent spend's puzzle reveal and calls `match_revocation_layer` on the inner layer. If this returns non-`None`, `wallet_type` is set to `RCATWallet` and the entire CR-CAT detection branch is skipped. The `hidden_puzzle_hash` returned by `match_revocation_layer` — which is fully attacker-controlled — is then passed without validation to `RCATWallet.get_or_create_wallet_for_cat` at line 1342. There is no check that the `hidden_puzzle_hash` corresponds to any key the victim controls, nor any check that the child coin's puzzle hash is consistent with a legitimate RCAT structure for the victim.

### Impact Explanation
- The victim's wallet is silently converted to (or a new) RCAT wallet is created with the attacker's revocation key.
- The specific coin the attacker sent can be stolen via the hidden puzzle path before the victim spends it.
- All future receive addresses generated by the victim's RCAT wallet embed the attacker's `hidden_puzzle_hash`, so every subsequent coin sent to the victim is also under attacker control.
- For DEFAULT_CATS tokens (automatically tracked), no user interaction beyond receiving the coin is required.

### Likelihood Explanation
- Requires the attacker to hold a non-zero amount of the targeted DEFAULT_CATS token — a real but low barrier for a motivated attacker.
- The victim's wallet for that asset_id must be absent or empty; this is the common case for new users or first-time recipients of a given token.
- No cryptographic break, no leaked keys, no admin access required.
- The crafted spend is a valid on-chain transaction that passes consensus validation.

### Recommendation
1. After determining `wallet_type = RCATWallet`, verify that the child coin's puzzle hash matches `CAT(TAIL, REVOCATION_LAYER(revocation_layer_match[0], our_inner_puzzle_hash))` before proceeding. If it does not match, return `None`.
2. Alternatively, reorder the checks: detect CR-CAT (structural check on `CREDENTIAL_RESTRICTION`) before RCAT, since CR-CAT and RCAT inner layers are mutually exclusive mods.
3. Do not trust the `hidden_puzzle_hash` extracted from an attacker-supplied puzzle reveal without verifying it against a known-good value or a wallet-controlled key.
