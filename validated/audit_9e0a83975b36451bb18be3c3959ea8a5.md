### Title
`ValidateTransactionProvider` Does Not Enforce `max_l1_handler_txs_per_block_proposal` Limit, Allowing a Malicious Proposer to Commit Excess L1 Handler Transactions ŌĆö (`File: crates/apollo_batcher/src/transaction_provider.rs`)

---

### Summary

The `ProposeTransactionProvider` enforces a configurable cap (`max_l1_handler_txs_per_block_proposal`) on the number of L1 handler transactions included in a block proposal. The `ValidateTransactionProvider`, used by every validator node when re-executing a received proposal, performs no equivalent count check. A malicious proposer can therefore stream more L1 handler transactions than the cap allows; every honest validator will accept and commit the over-limit block.

---

### Finding Description

**Proposer side ŌĆö limit enforced:**

`ProposeTransactionProvider` carries two fields that together enforce the cap:

```rust
pub max_l1_handler_txs_per_block: usize,
n_l1handler_txs_so_far: usize,
```

Inside `get_txs`, the number of L1 handler transactions fetched is bounded:

```rust
let n_l1handler_txs_to_get =
    min(self.max_l1_handler_txs_per_block - self.n_l1handler_txs_so_far, n_txs);
```

Once `n_l1handler_txs_so_far == max_l1_handler_txs_per_block`, the provider switches to the `Mempool` phase and fetches no further L1 handler transactions. [1](#0-0) [2](#0-1) 

**Validator side ŌĆö limit absent:**

`ValidateTransactionProvider` holds no counter and no cap field. Its `get_txs` implementation iterates over every transaction received from the consensus stream and, for each `L1Handler` variant, calls `l1_events_provider_client.validate()` to check individual validity (not-already-included, not-consumed-on-L1, etc.). It never counts how many L1 handler transactions have been accepted in the current block.

```rust
for tx in &buffer {
    if let InternalConsensusTransaction::L1Handler(tx) = tx {
        let l1_validation_status = self
            .l1_events_provider_client
            .validate(tx.tx_hash, self.height)
            .await
            ...
        if let L1ValidationStatus::Invalid(validation_status) = l1_validation_status {
            return Err(...);
        }
        continue;   // ŌåÉ accepted; no running total checked
    }
}
Ok(buffer)
``` [3](#0-2) 

The cap is declared in `BatcherStaticConfig` and documented as "The maximum number of L1 handler transactions to include in a block proposal": [4](#0-3) [5](#0-4) 

**Structural parallel to the external bug:**

| External (Geode) | Sequencer |
|---|---|
| `proposeStake` checks `totalActiveValidators` but not `proposedValidators` | `ProposeTransactionProvider` enforces `max_l1_handler_txs_per_block`; `ValidateTransactionProvider` enforces nothing |
| Maintainer approval step has no monopoly check | Validator's `get_txs` has no L1 handler count check |
| Two-step flow: propose ŌåÆ activate | Two-role flow: proposer builds ŌåÆ validator re-executes |
| Threshold bypassed across two proposals | Cap bypassed because validator never checks it |

---

### Impact Explanation

A malicious proposer (any node that wins a consensus round) can stream an arbitrarily large number of valid L1 handler transactions ŌĆö up to the bouncer's `n_txs` ceiling (500ŌĆō600 in production config) ŌĆö while the honest-node cap is 3 (default) or 200 (deployment config). Every validator will accept and commit the block because `ValidateTransactionProvider` only checks per-transaction L1 validity, not the aggregate count.

Concrete consequences:
- **Block composition invariant broken**: the committed block contains more L1 handler transactions than the protocol intends to allow per block.
- **Regular user transaction starvation**: a full block of L1 handler transactions leaves zero capacity for mempool (RPC) transactions, denying service to all regular users for that block.
- **Asymmetric enforcement**: the proposer's own node enforces the cap; every other node does not. The cap is therefore not a consensus-level invariant ŌĆö it is a local proposer policy that can be trivially circumvented.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing"* ŌĆö the validator's admission path accepts a proposal that violates the declared admission policy.

---

### Likelihood Explanation

Any validator that wins a proposer slot can trigger this. No special key material or out-of-band access is required beyond being a consensus participant. The L1 handler transactions must individually pass `l1_events_provider_client.validate()`, so the attacker needs a pool of valid, unconsumed L1 handler transactions ŌĆö a realistic condition given normal L1ŌåÆL2 message traffic. Likelihood is **Medium**.

---

### Recommendation

Add an L1 handler transaction counter to `ValidateTransactionProvider`, initialised from the same `max_l1_handler_txs_per_block_proposal` config value used by `ProposeTransactionProvider`. In `get_txs`, increment the counter for each accepted `L1Handler` transaction and return a `TransactionProviderError` (or cause the block builder to mark the proposal invalid) when the counter exceeds the cap.

```rust
pub struct ValidateTransactionProvider {
    tx_receiver: ...,
    final_n_executed_txs_receiver: ...,
    l1_events_provider_client: SharedL1EventsProviderClient,
    height: BlockNumber,
    max_l1_handler_txs_per_block: usize,   // ŌåÉ add
    n_l1handler_txs_so_far: usize,         // ŌåÉ add
}
```

Inside `get_txs`, after the existing `validate()` call succeeds:

```rust
self.n_l1handler_txs_so_far += 1;
if self.n_l1handler_txs_so_far > self.max_l1_handler_txs_per_block {
    return Err(TransactionProviderError::TooManyL1HandlerTransactions {
        limit: self.max_l1_handler_txs_per_block,
    });
}
```

The `max_l1_handler_txs_per_block_proposal` value must be passed into `ValidateTransactionProvider::new()` from `Batcher::validate_block`, mirroring how `ProposeTransactionProvider` receives it. [6](#0-5) 

---

### Proof of Concept

1. Honest nodes configure `max_l1_handler_txs_per_block_proposal = 3`.
2. Attacker controls a validator node and wins a proposer slot.
3. Attacker modifies their `ProposeTransactionProvider` (or bypasses it entirely) to include 300 valid, unconsumed L1 handler transactions in the streamed proposal.
4. Each transaction passes `l1_events_provider_client.validate()` on every honest validator because it is individually valid.
5. `ValidateTransactionProvider::get_txs` returns all 300 transactions without error.
6. The block builder executes them; the bouncer accepts them (300 < `n_txs` cap of 500).
7. `finish_proposal` succeeds; consensus commits a block containing 300 L1 handler transactions and zero regular user transactions.
8. The `max_l1_handler_txs_per_block_proposal = 3` invariant is violated in the committed chain state. [7](#0-6) [8](#0-7)

#[derive(Clone, Debug, PartialEq)]
pub enum TxProviderPhase {
    L1,
    Mempool,
}

impl ProposeTransactionProvider {
    pub fn new(
        mempool_client: SharedMempoolClient,
        l1_events_provider_client: SharedL1EventsProviderClient,
        max_l1_handler_txs_per_block: usize,
        height: BlockNumber,
        start_phase: TxProviderPhase,
    ) -> Self {
        Self {
            mempool_client,
            l1_events_provider_client,
            max_l1_handler_txs_per_block,
            height,
            phase: start_phase,
            n_l1handler_txs_so_far: 0,
        }
    }

    async fn get_l1_handler_txs(
        &mut self,
        n_txs: usize,
    ) -> TransactionProviderResult<Vec<InternalConsensusTransaction>> {
        Ok(self
            .l1_events_provider_client
            .get_txs(n_txs, self.height)
            .await
            .inspect_err(|err| {
                warn!("L1 provider error while fetching L1 handler transactions: {:?}", err);
                BATCHER_L1_EVENTS_PROVIDER_ERRORS.increment(1);
            })
            .unwrap_or_default()
            .into_iter()
            .map(InternalConsensusTransaction::L1Handler)
            .collect())
    }

    async fn get_mempool_txs(
        &mut self,
        n_txs: usize,
    ) -> TransactionProviderResult<Vec<InternalConsensusTransaction>> {
        Ok(self
            .mempool_client
            .get_txs(n_txs)
            .await?
            .into_iter()
            .map(InternalConsensusTransaction::RpcTransaction)
            .collect())
    }

#[async_trait]
impl TransactionProvider for ProposeTransactionProvider {
    async fn get_txs(&mut self, n_txs: usize) -> TransactionProviderResult<NextTxs> {
        assert!(n_txs > 0, "The number of transactions requested must be greater than zero.");
        let mut txs = vec![];
        if self.phase == TxProviderPhase::L1 {
            let n_l1handler_txs_to_get =
                min(self.max_l1_handler_txs_per_block - self.n_l1handler_txs_so_far, n_txs);
            let mut l1handler_txs = self.get_l1_handler_txs(n_l1handler_txs_to_get).await?;
            self.n_l1handler_txs_so_far += l1handler_txs.len();

            // Determine whether we need to switch to mempool phase.
            let no_more_l1handler_in_provider = l1handler_txs.len() < n_l1handler_txs_to_get;
            let reached_max_l1handler_txs_in_block =
                self.n_l1handler_txs_so_far == self.max_l1_handler_txs_per_block;
            if no_more_l1handler_in_provider || reached_max_l1handler_txs_in_block {
                self.phase = TxProviderPhase::Mempool;
            }

            txs.append(&mut l1handler_txs);
            if txs.len() == n_txs {
                return Ok(txs);
            }
        }

        let mut mempool_txs = self.get_mempool_txs(n_txs - txs.len()).await?;
        txs.append(&mut mempool_txs);
        Ok(txs)
    }

    async fn get_final_n_executed_txs(&mut self) -> Option<usize> {
        None
    }

    // TODO(guyn): remove this after refactoring the batcher tests.
    #[cfg(test)]
    fn phase(&self) -> TxProviderPhase {
        self.phase.clone()
    }
}

#[async_trait]
impl TransactionProvider for ValidateTransactionProvider {
    async fn get_txs(&mut self, n_txs: usize) -> TransactionProviderResult<NextTxs> {
        assert!(n_txs > 0, "The number of transactions requested must be greater than zero.");

        if self.tx_receiver.is_empty() {
            // Return immediately to avoid blocking the caller.
            return Ok(vec![]);
        }

        let mut buffer = Vec::with_capacity(n_txs);
        self.tx_receiver.recv_many(&mut buffer, n_txs).await;

        for tx in &buffer {
            if let InternalConsensusTransaction::L1Handler(tx) = tx {
                let l1_validation_status = self
                    .l1_events_provider_client
                    .validate(tx.tx_hash, self.height)
                    .await
                    .inspect_err(|err| {
                        warn!(
                            "L1 provider error while validating L1 handler transaction: {:?}",
                            err
                        );
                        BATCHER_L1_EVENTS_PROVIDER_ERRORS.increment(1);
                    })
                    .unwrap_or(L1ValidationStatus::Invalid(
                        L1InvalidValidationStatus::L1EventsProviderError,
                    ));
                if let L1ValidationStatus::Invalid(validation_status) = l1_validation_status {
                    return Err(TransactionProviderError::L1HandlerTransactionValidationFailed {
                        tx_hash: tx.tx_hash,
                        validation_status,
                    });
                }
                continue;
            }
        }
        Ok(buffer)
    }
