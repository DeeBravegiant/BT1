### Title
Per-Block Reference Script Size Limit Bypassed via Sub-Transactions in Dijkstra Era - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs`)

---

### Summary

In the Dijkstra era, the per-transaction reference script size check correctly aggregates sizes across a top-level transaction and all its sub-transactions via `batchNonDistinctRefScriptsSize`. However, the per-block reference script size check reuses the Conway-era `totalRefScriptSizeInBlock`, which only calls `txNonDistinctRefScriptsSize` on each top-level transaction and is blind to sub-transaction reference inputs. A block producer can craft a block whose transactions place all large reference scripts inside sub-transactions, causing the block-level check to count near-zero bytes while the actual deserialization work far exceeds the `maxRefScriptSizePerBlock` limit.

---

### Finding Description

**Root cause — asymmetric size accounting between per-tx and per-block checks.**

The per-transaction check in `dijkstraLedgerTransition` calls `validateAllRefScriptSize`, which delegates to `batchNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs:271-277
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx          -- top-level tx inputs
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)  -- sub-tx inputs
      )
``` [1](#0-0) 

This correctly sums reference script sizes across the top-level transaction and every sub-transaction.

The per-block check in `dijkstraBbodyTransition` calls the inherited Conway function `Conway.validateBodyRefScriptsSizeTooBig`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Bbody.hs:363
Conway.validateBodyRefScriptsSizeTooBig @era pp blockBody (ls ^. utxoL)
``` [2](#0-1) 

That function calls `totalRefScriptSizeInBlock`, which iterates over the block's top-level transactions and for each calls only `txNonDistinctRefScriptsSize`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Bbody.hs:357-370
totalRefScriptSizeInBlock protVer txs (UTxO utxo)
  | pvMajor protVer <= natVersion @10 = ...
  | otherwise =
      snd $ F.foldl' accum (utxo, 0) txs
  where
    accum (!accUtxo, !accSum) tx =
      ...
       in (updatedUtxo, accSum + txNonDistinctRefScriptsSize (UTxO accUtxo) tx)
``` [3](#0-2) 

`txNonDistinctRefScriptsSize` only inspects `tx ^. bodyTxL . referenceInputsTxBodyL` and `tx ^. bodyTxL . inputsTxBodyL` — the top-level transaction's own inputs — and never descends into `subTransactionsTxBodyL`:

```haskell
-- eras/conway/impl/src/Cardano/Ledger/Conway/UTxO.hs:183-187
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
``` [4](#0-3) 

The result is a structural mismatch: the per-tx enforcement uses the Dijkstra-aware `batchNonDistinctRefScriptsSize`, but the per-block enforcement uses the Conway-era `txNonDistinctRefScriptsSize` that has no knowledge of sub-transactions.

---

### Impact Explanation

**Medium. Attacker-controlled transactions exceed intended validation limits.**

The per-block reference script size limit (`maxRefScriptSizePerBlock`, currently 1 MiB) was introduced specifically to cap the deserialization and validation work a node must perform per block, following the June 2024 DDoS incident described in ADR-009. [5](#0-4) 

By placing all large reference scripts inside sub-transactions, an attacker can submit a block where the block-level check counts near-zero bytes while the actual reference script deserialization work is a multiple of the intended limit. For example:

- `maxRefScriptSizePerTx` = 200 KiB; `maxRefScriptSizePerBlock` = 1 MiB.
- Craft 10 top-level transactions, each with zero top-level reference inputs but with sub-transactions whose combined reference script size is 200 KiB (passing the per-tx check).
- `totalRefScriptSizeInBlock` counts 0 bytes for each top-level tx → block-level check passes.
- Actual reference script deserialization work: 10 × 200 KiB = 2 MiB — twice the intended block cap.

This allows a block producer to force all validating nodes to perform unbounded reference script deserialization work per block, degrading throughput and potentially causing sustained resource exhaustion.

---

### Likelihood Explanation

**High.** Any block producer can exploit this without any privileged key or governance action. Constructing a Dijkstra-era top-level transaction with sub-transactions is a normal, unprivileged operation. The CDDL schema explicitly permits `sub_transactions` as field 23 of `transaction_body`. [6](#0-5) 

The per-tx check (`validateAllRefScriptSize`) is enforced in the LEDGER rule and will reject any single transaction whose combined (top + sub) reference script size exceeds 200 KiB, so the attacker must stay within that bound per transaction. However, there is no protocol-enforced limit on the number of transactions per block beyond the block body size limit, and the block body size limit is measured in serialized bytes — not in reference script deserialization cost.

---

### Recommendation

Replace the call to `Conway.validateBodyRefScriptsSizeTooBig` in `dijkstraBbodyTransition` with a Dijkstra-aware variant that uses `batchNonDistinctRefScriptsSize` (or an equivalent block-level aggregation) when computing the total reference script size across all top-level transactions and their sub-transactions. Specifically, `totalRefScriptSizeInBlock` should be extended or overridden for the Dijkstra era to iterate over sub-transactions of each top-level transaction and accumulate their reference script sizes alongside the top-level transaction's own reference script sizes.

---

### Proof of Concept

1. Produce N UTxO entries, each holding a large reference script (e.g., a 20 KiB Plutus script), so that each sub-transaction can reference up to K such entries.

2. Construct a top-level Dijkstra transaction with zero top-level reference inputs and M sub-transactions, each referencing K UTxO entries with reference scripts, such that M × K × 20 KiB ≤ 200 KiB (satisfying `validateAllRefScriptSize`).

3. Pack enough such top-level transactions into a single block so that the true aggregate reference script size exceeds `maxRefScriptSizePerBlock` (1 MiB).

4. Submit the block. `dijkstraBbodyTransition` calls `Conway.validateBodyRefScriptsSizeTooBig`, which calls `totalRefScriptSizeInBlock`. That function calls `txNonDistinctRefScriptsSize` on each top-level transaction, counting 0 bytes (since all reference inputs are in sub-transactions). The block-level check passes. All validating nodes must deserialize and process more than 1 MiB of reference scripts. [7](#0-6) [8](#0-7)

      getSum $ foldMap' (Monoid.Sum . txNonDistinctRefScriptsSize (UTxO utxo)) txs

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter.
