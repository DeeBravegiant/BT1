Audit Report

## Title
Missing Staleness and Round-Completeness Validation in `ChainlinkPriceOracle.getAssetPrice()` Enables Overvalued LST Deposits During Feed Outages - (`contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` fields, accepting any returned price as current regardless of its age. During a Chainlink feed outage or LST depeg event where the feed lags real market prices, any unprivileged depositor can call `LRTDepositPool.depositAsset()` to receive rsETH minted at the stale (inflated) price, receiving more rsETH than their deposit is worth and diluting all existing rsETH holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at line 52 destructures only the `price` (`answer`) field from `latestRoundData()`, discarding `updatedAt` and `answeredInRound`:

```solidity
// contracts/oracles/ChainlinkPriceOracle.sol L49-55
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check of the form `block.timestamp - updatedAt > heartbeat` or `answeredInRound < roundId` is present. The sister contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 26–36) correctly validates all three conditions (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`), demonstrating the pattern is known and intentionally applied elsewhere in the codebase.

The exploit path is:

1. Chainlink LST/ETH feed becomes stale (e.g., stETH depegs to 0.94 ETH; feed still reports 1e18).
2. Attacker calls `LRTDepositPool.depositAsset(stETH, amount, 0, "")`.
3. `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns the stale `1e18`.
4. `rsethAmountToMint = (amount × 1e18) / rsETHPrice` — attacker receives ~6.38% more rsETH than their deposit is worth.
5. The protocol now holds stETH worth 0.94 ETH per unit but has issued rsETH claims against 1 ETH per unit, reducing the backing ratio below 1:1 for all existing holders.

The `pricePercentageLimit` downside-protection mechanism in `_updateRsETHPrice()` only triggers when `updateRSETHPrice()` is called after the feed corrects. The dilution occurs at deposit time, before any price update. The `_getTotalEthInProtocol()` TVL calculation also routes through the same stale oracle, so the rsETH price does not self-correct until the Chainlink feed updates on-chain.

No existing guard in `_beforeDeposit()` or `getRsETHAmountToMint()` checks oracle freshness. The `minRSETHAmountExpected` slippage parameter protects the depositor, not existing holders.

## Impact Explanation

**Critical — Direct theft of user funds (dilution of existing rsETH holders) and protocol insolvency.**

Each deposit made against a stale inflated price issues rsETH backed by less ETH than the rsETH represents. When the feed corrects and `updateRSETHPrice()` is called, the rsETH price drops, transferring value from all pre-existing holders to the attacker. At scale (e.g., repeated deposits across the staleness window), the protocol's backing ratio falls below 1:1, constituting protocol insolvency. This maps directly to the allowed Critical impacts: direct theft of user funds and protocol insolvency.

## Likelihood Explanation

- The Chainlink stETH/ETH feed has a 24-hour heartbeat and 0.5% deviation threshold. Historical events (May 2022 stETH depeg, March 2023 USDC depeg) demonstrate multi-hour lags between real market prices and on-chain feed updates.
- Network congestion delaying Chainlink keeper transactions creates the same staleness window without any depeg.
- The attack requires no special permissions: `depositAsset()` is callable by any EOA or contract.
- The attacker only needs to observe that the on-chain Chainlink price diverges from the real market price (e.g., via a DEX spot price) and deposit during that window. The attack is repeatable for the entire duration of the staleness window.

## Recommendation

Add staleness and round-completeness checks to `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();
    if (answeredInRound < roundId) revert StalePrice();
    if (updatedAt == 0) revert IncompleteRound();
    if (price <= 0) revert InvalidPrice();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be set per-asset based on the feed's documented heartbeat (e.g., 25 hours for a 24-hour heartbeat feed). Consider storing it in a `mapping(address asset => uint256 maxStaleness)` configurable by the LRT manager.

## Proof of Concept

**Foundry fork test outline:**

```solidity
// Fork mainnet at a block where stETH/ETH Chainlink feed is fresh.
// Warp block.timestamp forward by 25 hours (past the 24h heartbeat) without
// triggering a Chainlink update (simulate by not calling the keeper).
// At this point priceFeed.latestRoundData() still returns the pre-warp price.

function testStaleOracleDeposit() public {
    // 1. Record attacker's stETH balance and current rsETH supply / price.
    uint256 rsETHPriceBefore = lrtOracle.rsETHPrice();
    uint256 totalSupplyBefore = rsETH.totalSupply();

    // 2. Warp 25 hours; Chainlink feed is now stale (no keeper update).
    vm.warp(block.timestamp + 25 hours);

    // 3. Simulate stETH market price drop to 0.94 ETH (feed still reports 1e18).
    // (In a real fork test, use a block where the feed has not updated but
    //  the market price has moved, or mock the aggregator to return an old answer.)

    // 4. Attacker deposits 100e18 stETH.
    vm.startPrank(attacker);
    stETH.approve(address(depositPool), 100e18);
    depositPool.depositAsset(address(stETH), 100e18, 0, "");
    vm.stopPrank();

    // 5. Assert attacker received rsETH valued at ~100 ETH (stale price),
    //    but only deposited 94 ETH worth of stETH.
    uint256 attackerRsETH = rsETH.balanceOf(attacker);
    uint256 attackerRsETHValueAtCurrentPrice = attackerRsETH * rsETHPriceBefore / 1e18;
    assertGt(attackerRsETHValueAtCurrentPrice, 94e18); // attacker over-minted

    // 6. Call updateRSETHPrice() after feed corrects to 0.94e18.
    //    rsETH price drops; existing holders are diluted.
    // (Mock feed to return 0.94e18, then call updateRSETHPrice().)
    lrtOracle.updateRSETHPrice();
    assertLt(lrtOracle.rsETHPrice(), rsETHPriceBefore); // price dropped for all holders
}
```
