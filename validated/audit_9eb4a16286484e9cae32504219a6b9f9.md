### Title
Stale `vote_add_domains` Votes Persist Across Epoch Changes, Enabling Unauthorized Domain Addition - (File: crates/contract/src/state/running.rs)

---

### Summary

`vote_add_domains` carries no epoch identifier or nonce. Votes cast in one epoch survive participant-set changes and can be activated in a later epoch by a new participant, allowing a single actor to force a domain addition that was previously blocked — without the knowledge of the participants whose stale votes are being exploited.

---

### Finding Description

`vote_new_parameters` protects against stale votes by requiring callers to supply a `prospective_epoch_id` that is validated against the contract's current prospective epoch: [1](#0-0) 

`vote_add_domains` has no such guard. Its only parameter is the list of domains to add: [2](#0-1) 

Votes are stored in `AddDomainsVotes`, a plain `BTreeMap<AuthenticatedParticipantId, Vec<DomainConfig>>` with no timestamp or epoch tag: [3](#0-2) 

When resharing completes and the contract re-enters `Running`, `RunningContractState::new` calls `get_remaining_votes`, which **retains** every vote whose caster is still a participant in the new epoch: [4](#0-3) [5](#0-4) 

The transition check in `vote_add_domains` then counts all retained votes against the **current** participant count: [6](#0-5) 

Because old votes are never invalidated by an epoch change, a vote cast in epoch N is indistinguishable from one cast in epoch N+1.

---

### Impact Explanation

An adversary can force the addition of a new cryptographic domain (e.g., a `ForeignTx` domain enabling bridge signing, or a `CKD` domain) that was previously blocked by unanimous-vote failure. This breaks the governance invariant that domain additions require unanimous consent from the **current** participant set in the **current** epoch. It is a participant-state and contract execution-flow manipulation that violates a production safety invariant.

**Impact: Medium** — participant-state manipulation that breaks production governance/accounting invariants without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

**Low-Medium.** The attack requires:
1. All-but-one current participants to have already voted for a specific domain proposal.
2. The holdout to successfully orchestrate their own replacement (requires threshold votes from others, which they obtain by presenting a legitimate reason such as key rotation).
3. The new participant (the holdout's new wallet) to vote for the same proposal.
4. The other participants to not actively revoke their stale votes before step 3.

Steps 2–4 are realistic in a production network where participants rotate keys periodically and do not monitor stale governance votes.

---

### Recommendation

1. **Add an epoch identifier to `vote_add_domains`** analogous to `prospective_epoch_id` in `vote_new_parameters`. Validate it against `self.keyset.epoch_id` at call time and reject votes from prior epochs.
2. **Clear `add_domains_votes` on every epoch transition** (resharing completion) rather than filtering and carrying them forward.
3. Optionally, add an `expires_at` block-timestamp field to domain-addition proposals so that votes automatically become invalid after a configurable window.

---

### Proof of Concept

```
Epoch N — participants {A, B, C}, governance threshold = 3 (unanimous)

1. A calls vote_add_domains([ForeignTx domain X])  → 1/3 votes
2. B calls vote_add_domains([ForeignTx domain X])  → 2/3 votes
3. C refuses to vote                               → proposal blocked

Epoch N+1 — C claims key rotation; A and B vote_new_parameters to replace C with D

4. Resharing completes; RunningContractState::new calls get_remaining_votes:
   - A's vote for domain X is retained  (A still a participant)
   - B's vote for domain X is retained  (B still a participant)
   - C's vote is absent                 (C no longer a participant)
   → add_domains_votes now holds 2 votes for domain X

5. D (C's new wallet) calls vote_add_domains([ForeignTx domain X])
   → n_votes = 3 == participants.len() (3)
   → contract transitions to Initializing state for domain X

A and B's votes were cast when C was the required third voter.
They never consented to D being the deciding vote.
Domain X is added without the unanimous approval of the current epoch's participant set.
```

pub struct AddDomainsVotes {
    pub(crate) proposal_by_account: BTreeMap<AuthenticatedParticipantId, Vec<DomainConfig>>,
}

impl AddDomainsVotes {
    /// Votes for the proposal, returning the total number of voters so far who
    /// have proposed the exact same domains to add.
    /// If the participant had voted already, this replaces the existing vote.
    pub fn vote(
        &mut self,
        proposal: Vec<DomainConfig>,
        participant: &AuthenticatedParticipantId,
    ) -> u64 {
        if self
            .proposal_by_account
            .insert(participant.clone(), proposal.clone())
            .is_some()
        {
            log!("removed old vote for signer");
        }
        let total = self
            .proposal_by_account
            .values()
            .filter(|&prop| prop == &proposal)
            .count() as u64;
        log!("total votes for proposal: {}", total);
        total
    }

    /// Filters out existing votes no longer in the participant set
    pub fn get_remaining_votes(&self, participants: &Participants) -> Self {
        let remaining_votes = self
            .proposal_by_account
            .iter()
            .filter(|&(participant_id, _vote)| {
                participants.is_participant_given_participant_id(&participant_id.get())
            })
            .map(|(participant_id, vote)| (participant_id.clone(), vote.clone()))
            .collect();
        AddDomainsVotes {
            proposal_by_account: remaining_votes,
        }
    }
