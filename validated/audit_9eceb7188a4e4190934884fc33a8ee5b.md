### Title
Presignature Irrevocably Consumed Before Foreign-Chain Verification Succeeds, Enabling Presignature Pool Exhaustion - (File: `crates/node/src/providers/verify_foreign_tx/sign.rs`)

---

### Summary

In `make_verify_foreign_tx_leader`, the leader node calls `presignature_store.take_owned().await` — permanently removing a presignature from its store — **before** calling `execute_foreign_chain_request(...)`. If the foreign-chain verification step fails for any reason (RPC timeout, finality not reached, chain not configured, sub-quorum disagreement), the function returns an error via `?` but the presignature is already gone with no rollback path. Because the generic coordinator queue retries failed requests, a single adversarial or consistently-failing `verify_foreign_transaction` request will consume one presignature per retry cycle, draining the pool faster than background generation can replenish it.

`take_owned()` is documented to atomically remove the asset from storage: "Each asset is consumed exactly once." There is no `put_back` or reservation mechanism. [2](#0-1) 

`execute_foreign_chain_request` can fail for multiple reasons: the chain is not configured on the node, the RPC quorum is not met, the transaction has not reached the required finality, or a timeout fires after `FOREIGN_CHAIN_INSPECTION_TIMEOUT` (5 seconds). [3](#0-2) 

The design document explicitly acknowledges that the signing set is inherited from a presignature whose participants were chosen for liveness, not chain coverage — meaning a presignature's participant set may not cover the requested foreign chain, causing `execute_foreign_chain_request` to fail on some or all nodes. [4](#0-3) 

Furthermore, the design document notes that the generic coordinator queue **retries every failed request**, meaning a single request that consistently fails will consume one presignature per retry cycle. [5](#0-4) 

---

### Impact Explanation

Presignatures are expensive multi-party cryptographic assets. Each requires a completed triple-generation protocol (batch of 64 triples via OT) followed by a presignature-generation protocol, both involving threshold-many nodes. When the presignature pool is exhausted, `take_owned()` blocks indefinitely:

> "Eventually the node will exhaust its owned presignatures. At that point `take_owned()` blocks indefinitely (waiting for a presignature that will never arrive), and the node stops being able to lead signature computations." [6](#0-5) 

A depleted leader node cannot process any signing request — including ordinary `sign()` requests from users who have attached deposits. Those requests remain pending on-chain until they time out, causing the attached deposits to be locked or lost. This breaks the request-lifecycle accounting invariant: the contract accepted a request and collected a deposit, but the network cannot fulfill it due to a resource-accounting failure that an adversarial foreign-chain input producer triggered.

**Impact class: Medium** — request-lifecycle and participant-state manipulation that breaks production safety/accounting invariants without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

The attacker-controlled entry path requires no privilege:

1. Any NEAR account calls `verify_foreign_transaction` on the MPC contract with a crafted `ForeignChainRpcRequest` (e.g., a Bitcoin transaction ID that has not reached the required confirmation depth, or a chain whose RPC quorum will not be met).
2. The contract accepts the request — it checks chain availability at the contract level but cannot check node-side presignature availability or RPC-quorum reachability.
3. The leader node picks up the request, calls `take_owned()` (presignature consumed), then calls `execute_foreign_chain_request()` which fails.
4. The coordinator queue retries the request; the leader consumes another presignature.
5. Repeat until the leader's presignature pool is exhausted.

The attack is amplified by the fact that presignature generation is significantly slower than triple generation (one at a time vs. batches of 64), and that the `FOREIGN_CHAIN_INSPECTION_TIMEOUT` of 5 seconds means each failed attempt is fast relative to presignature generation time. [7](#0-6) 

---

### Recommendation

Move `take_owned()` to **after** `execute_foreign_chain_request()` succeeds. The corrected order should be:

1. Fetch the foreign-tx request from the store.
2. Execute and verify the foreign chain request (fail fast if it errors).
3. Build the `SignatureRequest` from the verified payload.
4. Only then call `presignature_store.take_owned()`.
5. Open the network channel and proceed with signing.

This mirrors the principle from the external report: check that the resource is available and the operation can succeed **before** committing the resource. Additionally, the coordinator queue should be updated to treat sub-quorum foreign-chain failures as non-retryable (as already tracked in issue #3477), preventing repeated presignature consumption on permanently-failing requests. [5](#0-4) 

---

### Proof of Concept

**Scenario:**

1. Attacker submits a `verify_foreign_transaction` request for a Bitcoin transaction with `confirmations = 100` that currently has only 2 confirmations.
2. The MPC contract accepts the request (chain is whitelisted and available at the contract level).
3. Leader node's `make_verify_foreign_tx_leader` runs:
   - Line 63: `take_owned()` — presignature P1 is atomically removed from the store.
   - Line 147: `inspector.extract(tx_id, 100, ...)` — times out after 5 seconds because the block is not finalized to 100 confirmations.
   - Line 78: `?` propagates the error; P1 is permanently lost.
4. Coordinator retries the request. Leader consumes presignature P2. Same failure. P2 lost.
5. After `desired_presignatures_to_buffer` retries, the leader's presignature store is empty.
6. All subsequent `make_signature_leader` calls (for ordinary user `sign()` requests) block indefinitely on `take_owned()`.
7. User sign requests accumulate on-chain, deposits locked, network unable to fulfill them. [8](#0-7) [9](#0-8)

    pub(super) async fn make_verify_foreign_tx_leader(
        &self,
        id: SignatureId,
    ) -> anyhow::Result<((dtos::ForeignTxSignPayload, Signature), VerifyingKey)> {
        let foreign_tx_request = self.verify_foreign_tx_request_store.get(id).await?;

        let domain_data = self
            .ecdsa_signature_provider
            .domain_data(foreign_tx_request.domain_id)?;
        let (presignature_id, presignature) = domain_data.presignature_store.take_owned().await;
        let participants = presignature.participants.clone();
        let channel = self.ecdsa_signature_provider.new_channel_for_task(
            VerifyForeignTxTaskId::VerifyForeignTx {
                id,
                presignature_id,
            },
            participants,
        )?;

        let response_payload = self
            .execute_foreign_chain_request(
                &foreign_tx_request.request,
                foreign_tx_request.payload_version,
            )
            .await?;

        let sign_request = build_signature_request(&foreign_tx_request, &response_payload)?;

        let response = self
            .ecdsa_signature_provider
            .make_signature_leader_given_parameters(sign_request, presignature, channel)
            .await?;
        Ok(((response_payload, response.0), response.1))
    }
