Audit Report

## Title
ETH Deposit Limit Bypass Due to Missing Amount Check in `_checkIfDepositAmountExceedesCurrentLimit` — (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` omits the `amount` parameter from the ETH branch comparison, testing only the pre-deposit total against the configured limit. Any ETH deposit passes the check as long as the current total has not already exceeded the cap, regardless of how large the incoming deposit is. This breaks the protocol's deposit-limit invariant for ETH while correctly enforcing it for ERC20 assets.

## Finding Description
At `contracts/LRTDepositPool.sol` lines 676–682, the function branches on asset type:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
```

The ETH branch evaluates `totalAssetDeposits > limit`, which is `false` under normal operating conditions (i.e., when the pool has not already exceeded the cap). The incoming `amount` is never added, so a deposit of any size passes the check. `_beforeDeposit` (lines 648–670) calls this function and reverts with `MaximumDepositLimitReached` only when it returns `true` — which the ETH branch never does while the pool is below the cap. `getAssetCurrentLimit` (lines 402–409) correctly computes remaining capacity as `limit − totalAssetDeposits`, creating a direct contradiction: the view function advertises a finite remaining room while the guard allows unlimited ETH through.

## Impact Explanation
The deposit limit is the protocol's primary on-chain mechanism for capping ETH TVL. Bypassing it allows a single depositor to push ETH TVL arbitrarily above the configured cap in one transaction, minting rsETH against excess ETH that the protocol's risk model did not account for. This concretely maps to **Low — Contract fails to deliver promised returns**: the deposit-limit guarantee is advertised and enforced for ERC20 but silently not enforced for ETH.

## Likelihood Explanation
`depositETH` is a public, permissionless payable function requiring no special role or precondition. The bypass condition (`totalAssetDeposits ≤ depositLimit`) is the normal operating state of the protocol. Any external account holding ETH can trigger this at any time, repeatedly, without coordination.

## Recommendation
Include `amount` in the ETH branch to match the ERC20 branch:

```diff
 if (asset == LRTConstants.ETH_TOKEN) {
-    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
 }
```

## Proof of Concept
1. Configure `depositLimitByAsset(ETH) = 100 ETH`; current `totalAssetDeposits(ETH) = 50 ETH`.
2. `getAssetCurrentLimit(ETH)` returns `50 ETH`.
3. Call `depositETH{value: 10_000 ether}(minRsETH)` from any EOA.
4. `_beforeDeposit` calls `_checkIfDepositAmountExceedesCurrentLimit(ETH, 10_000e18)`.
5. ETH branch evaluates `50e18 > 100e18` → `false` → no revert.
6. `_mintRsETH` mints rsETH for the full 10,000 ETH.
7. `totalAssetDeposits(ETH)` becomes `10,050 ETH` — 100× the intended cap.

Foundry test plan: deploy `LRTDepositPool` on a local fork, set `depositLimitByAsset(ETH) = 100 ether`, seed 50 ETH of existing deposits, call `depositETH{value: 10_000 ether}`, assert the call succeeds and `getTotalAssetDeposits(ETH) > depositLimitByAsset(ETH)`.
