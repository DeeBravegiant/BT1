### Title
Stale `cancel_votes` Accumulate Across Domain Transitions, Enabling Below-Threshold Cancellation of Domain Keygen — (`crates/contract/src/state/initializing.rs`)

### Summary

`InitializingContractState::cancel_votes` is a `BTreeSet` that persists for the entire lifetime of the `Initializing` state. When `vote_pk` completes domain-N and advances `generating_key` to domain-(N+1), the `cancel_votes` set is **never cleared**. A stale cancel vote cast during domain-0 keygen silently carries over and counts toward the threshold for cancelling domain-1 keygen, allowing a single additional participant to trigger cancellation below the required threshold.

### Finding Description

`InitializingContractState` holds:

```rust
pub cancel_votes: BTreeSet<AuthenticatedParticipantId>,
``` [1](#0-0) 

When `vote_pk` collects the final success vote for domain-0, it replaces `generating_key` with a fresh `KeyEvent` for domain-1, but leaves `cancel_votes` untouched:

```rust
self.generating_key = KeyEvent::new(
    self.epoch_id,
    next_domain.clone(),
    self.generating_key.proposed_parameters().clone(),
);
``` [2](#0-1) 

`vote_cancel` then checks:

```rust
if self.cancel_votes.insert(participant) && self.cancel_votes.len() >= required_threshold {
``` [3](#0-2) 

The only guard against stale votes is the `next_domain_id` check:

```rust
if next_domain_id != self.domains.next_domain_id() {
    return Err(InvalidParameters::NextDomainIdMismatch.into());
}
``` [4](#0-3) 

`domains.next_domain_id()` is the ID that would be assigned to the *next newly added domain*. It is set once when `add_domains` is called and **does not change** as keygen advances from domain-0 to domain-1 to domain-N within the same `Initializing` session. `retain_domains` also does not update it:

```rust
pub fn retain_domains(&mut self, num_domains: usize) {
    self.domains.truncate(num_domains);
}
``` [5](#0-4) 

The test confirms `next_domain_id` is unchanged after `retain_domains`:

```rust
registry.retain_domains(3);
assert_eq!(registry.next_domain_id, 6);
``` [6](#0-5) 

So the `next_domain_id` guard provides **zero protection** against cross-domain vote accumulation within the same keygen session.

### Impact Explanation

With threshold=2 and two domains being generated:

1. Participant A calls `vote_cancel` during domain-0 keygen. `cancel_votes = {A}`, len=1 < 2, no transition.
2. All participants call `vote_pk` for domain-0. Domain-0 completes; `generating_key` is replaced with a new `KeyEvent` for domain-1. `cancel_votes` still contains `{A}`.
3. Participant B calls `vote_cancel` during domain-1 keygen. `cancel_votes.insert(B)` returns `true`; `cancel_votes.len() == 2 >= 2`. The contract transitions to `Running` with only domain-0's key. Domain-1 is permanently deleted via `retain_domains(1)`, and its `DomainId` is permanently consumed (`next_domain_id` is not rolled back).

Only **one** participant (B) needs to actively want to cancel domain-1. A's stale vote from domain-0 — cast when A may have had a different intent — is silently reused. This is a below-threshold cancellation of domain-1: the threshold for cancelling domain-1 is effectively reduced from 2 to 1 active participant.

The deleted domain's ID is permanently consumed. If domain-1 was the only domain for a given curve, signing for that curve is permanently frozen until a new `vote_add_domains` round assigns a fresh `DomainId`.

### Likelihood Explanation

The scenario is realistic. During a multi-domain keygen, it is normal for a participant to vote cancel if they believe the keygen is stalling (e.g., due to a timeout or network issue). If domain-0 subsequently completes, that participant's cancel vote becomes stale but is never cleared. Any single other participant can then exploit this to cancel domain-1 unilaterally.

### Recommendation

Reset `cancel_votes` to an empty `BTreeSet` whenever `generating_key` advances to a new domain inside `vote_pk`:

```rust
self.generating_key = KeyEvent::new(...);
self.cancel_votes = BTreeSet::new(); // add this line
``` [7](#0-6) 

Alternatively, bind cancel votes to a specific domain index (e.g., store `(domain_index, participant)` tuples) so that votes cast for domain-0 cannot count toward domain-1.

### Proof of Concept

```
Setup: threshold=2, 3 participants (A, B, C), 2 domains (domain-0, domain-1).

1. All participants vote_add_domains([domain-0, domain-1]).
   → State: Initializing { cancel_votes: {} }

2. Leader starts domain-0 keygen.

3. Participant A calls vote_cancel(next_domain_id=2).
   → cancel_votes = {A}, len=1 < 2, no transition.

4. All 3 participants call vote_pk for domain-0.
   → domain-0 key generated; generating_key advances to domain-1.
   → cancel_votes still = {A}  ← BUG: not cleared

5. Leader starts domain-1 keygen.

6. Participant B calls vote_cancel(next_domain_id=2).
   → cancel_votes.insert(B) = true; cancel_votes = {A, B}, len=2 >= 2.
   → Contract transitions to Running with only domain-0's key.
   → domain-1 is permanently deleted; DomainId(1) is consumed.

Assert: running.keyset.domains.len() == 1
Assert: running.domains.domains().len() == 1
Assert: running.domains.next_domain_id() == 2  (DomainId(1) permanently consumed)
```

Only participant B actively voted to cancel domain-1. A's stale vote from domain-0 was silently reused, reducing the effective cancellation threshold for domain-1 from 2 to 1.
