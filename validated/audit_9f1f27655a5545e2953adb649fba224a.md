### Title
Sub-Transaction Reference Script Size Excluded from Minimum Fee Calculation in Dijkstra Batches — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, the minimum fee for a batch transaction is computed using `getConwayMinFeeTxUtxo`, which calls `txNonDistinctRefScriptsSize` and counts only the reference scripts reachable from the **top-level** transaction's inputs. Sub-transactions embedded in the batch can carry their own reference inputs pointing to arbitrarily large Plutus scripts, but those scripts are **never included** in the fee calculation. The block-level and per-batch size-limit checks, however, use `batchNonDistinctRefScriptsSize`, which correctly sums reference script bytes across the top-level transaction **and every sub-transaction**. The result is a systematic underestimation of the minimum fee for any Dijkstra batch that uses reference scripts inside sub-transactions.

---

### Finding Description

`getConwayMinFeeTxUtxo` is the `getMinFeeTxUtxo` implementation inherited by `DijkstraEra`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
import Cardano.Ledger.Conway.UTxO (
  ...
  getConwayMinFeeTxUtxo,
  ...
  txNonDistinctRefScriptsSize,
)
```

`getConwayMinFeeTxUtxo` computes the fee as:

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
```

`txNonDistinctRefScriptsSize` only unions the **top-level** transaction's `referenceInputsTxBodyL` and `inputsTxBodyL`:

```haskell
txNonDistinctRefScriptsSize utxo tx = getSum $ foldMap (Sum . originalBytesSize . snd) refScripts
  where
    inputs = (tx ^. bodyTxL . referenceInputsTxBodyL) `Set.union` (tx ^. bodyTxL . inputsTxBodyL)
    refScripts = getReferenceScriptsNonDistinct utxo inputs
```

Sub-transactions are never consulted. By contrast, the Dijkstra-specific size-limit validator uses `batchNonDistinctRefScriptsSize`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
batchNonDistinctRefScriptsSize utxo tx =
  txNonDistinctRefScriptsSize utxo tx
    + getSum
      ( foldMap'
          (Sum . txNonDistinctRefScriptsSize utxo)
          (tx ^. bodyTxL . subTransactionsTxBodyL)
      )
```

And `validateAllRefScriptSize` in the Dijkstra LEDGER rule enforces the per-batch limit using this full-batch size:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Ledger.hs
validateAllRefScriptSize pp utxo tx =
  let totalRefScriptSize = batchNonDistinctRefScriptsSize utxo tx
      maxRefScriptSizePerTx = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerTxG
   in failureUnless (totalRefScriptSize <= maxRefScriptSizePerTx) $
        DijkstraTxRefScriptsSizeTooBig ...
```

The fee check in `dijkstraUtxoTransition` calls:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs
runTest $ Shelley.validateFeeTooSmallUTxO pp tx originalUtxo
```

which resolves to `getConwayMinFeeTxUtxo` — the top-level-only calculation. Sub-transaction reference script deserialization cost is therefore **never charged**.

---

### Impact Explanation

The ADR for reference-script fee changes documents that deserializing large Plutus scripts is expensive enough to enable a DDoS attack, which actually occurred on 25 June 2024. The Conway fix introduced `minFeeRefScriptCostPerByte` and the tiered `tierRefScriptFee` formula precisely to make large reference scripts expensive. In Dijkstra, an attacker can embed arbitrarily large reference scripts exclusively inside sub-transactions, paying only the base `a + b * txSize` fee for the top-level transaction. The node must still deserialize every sub-transaction reference script during validation, but the submitter pays nothing for that work. This allows attacker-controlled transactions to exceed intended validation costs and modify fees outside design parameters.

**Allowed impact matched**: Medium — attacker-controlled transactions exceed intended validation limits or modify fees outside design parameters.

---

### Likelihood Explanation

The Dijkstra era introduces sub-transactions as a first-class feature. Any user who constructs a `DijkstraTx` with sub-transactions that carry `referenceInputsTxBodyL` pointing to large Plutus scripts can trigger this path. No special privilege, governance majority, or key compromise is required. The attacker only needs to submit a valid on-chain transaction.

---

### Recommendation

Override `getMinFeeTxUtxo` in the `EraUTxO DijkstraEra` instance to use `batchNonDistinctRefScriptsSize` instead of `txNonDistinctRefScriptsSize`, so that the minimum fee for a Dijkstra batch includes the reference script deserialization cost for all sub-transactions:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs
getDijkstraMinFeeTxUtxo :: ... => PParams era -> Tx l era -> UTxO era -> Coin
getDijkstraMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ batchNonDistinctRefScriptsSize utxo tx
```

This mirrors the fix applied in Conway (ADR-009) for top-level reference scripts and closes the same attack surface for sub-transactions.

---

### Proof of Concept

1. Craft a `DijkstraTx` whose top-level body has no reference inputs (so `txNonDistinctRefScriptsSize` returns 0).
2. Embed N sub-transactions, each with `referenceInputsTxBodyL` pointing to a UTxO that holds a large Plutus script (e.g., 25 KiB each, staying under `ppMaxRefScriptSizePerTxG` for the batch total).
3. Set the transaction fee to the minimum computed by `getConwayMinFeeTxUtxo` — which ignores sub-transaction reference scripts entirely.
4. Submit the transaction. The fee check passes (top-level ref scripts = 0), the size-limit check passes (total batch size ≤ limit), but the node must deserialize N × 25 KiB of Plutus scripts at no extra cost to the submitter.
5. Repeat with many such transactions to saturate block validation capacity, replicating the June 2024 DDoS scenario for the Dijkstra era. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

It was identified a while ago that there is an overhead associated with deserializing scripts, so much so that it would be possible to create a very large script that was fairly expensive to deserialize, but very cheap to execute. This opened up an attack vector when such a Plutus script would be used as a reference script. This problem was exacerbated by the fact that there was no real limit on the total size of reference scripts that could be used in a transaction, thus being limited only by the size of the transaction itself. Therefore this opened up Cardano to a DDoS attack where an attacker could submit many such transactions that would cost very little, but would be expensive for a `cardano-node` to validate.

In order to prevent such an attack a `"minFeeRefScriptCostPerByte"` protocol parameter was introduced in the Conway era. The idea was fairly simple: we would calculate the total size of reference scripts used by a transaction and multiply it by the value specified by this protocol parameter. Result would be added to the transaction fee. However, in order for this approach to be a definite deterrent of such attacks this parameter would have to be set to a fairly high value. Community was not very keen on having this value set to a high enough value, since that could make reference scripts almost as expensive to use as regular scripts, thus significantly reducing their usability.
