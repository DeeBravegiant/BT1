Audit Report

## Title
`rsETHPrice` Defaults to `1 ether` When `rsethSupply == 0` Regardless of Protocol TVL, Enabling First-Depositor Theft of Orphaned Assets - (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle._updateRsETHPrice()` unconditionally sets `rsETHPrice = 1 ether` and returns early whenever `rsethSupply == 0`, without inspecting actual protocol TVL. Because `LRTDepositPool` accepts ETH via an unrestricted `receive()` fallback and an access-control-free `receiveFromRewardReceiver()`, ETH can accumulate in the pool while supply is zero. The first depositor then mints rsETH at a 1:1 rate and, after the next price update, holds tokens redeemable for the full protocol TVL, stealing the orphaned balance.

## Finding Description
In `LRTOracle._updateRsETHPrice()` (lines 218–222), when `rsethSupply == 0`, the function sets `rsETHPrice = 1 ether` and returns without ever calling `_getTotalEthInProtocol()`:

```solidity
if (rsethSupply == 0) {
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
``` [1](#0-0) 

`updateRSETHPrice()` is public and permissionless: [2](#0-1) 

ETH can enter the deposit pool with no supply minted via two unrestricted entry points: [3](#0-2) 

`getETHDistributionData()` counts `address(this).balance` directly, so any ETH received via these paths is immediately reflected in TVL: [4](#0-3) 

The minting formula in `getRsETHAmountToMint()` divides by the stored `rsETHPrice`: [5](#0-4) 

If `rsETHPrice = 1e18` but the pool holds 50 ETH, a depositor sending 1 ETH receives 1 rsETH. On the next `updateRSETHPrice()` call (with `rsethSupply = 1` and `totalETHInProtocol = 51 ETH`), the price correctly becomes 51e18. The attacker's 1 rsETH is now worth 51 ETH.

The withdrawal path via `initiateWithdrawal()` records `expectedAssetAmount` at the inflated price. The availability check at line 170 passes because `getAvailableAssetAmount(ETH)` returns the full deposit pool balance (51 ETH), which equals the requested 51 ETH: [6](#0-5) 

`_calculatePayoutAmount()` pays the minimum of `expectedAssetAmount` and `currentReturn`; both equal 51 ETH at unlock time, so the full amount is disbursed: [7](#0-6) 

## Impact Explanation
This is **direct theft of protocol assets** — specifically any ETH held in the deposit pool (rewards forwarded by the reward receiver, or ETH sent directly) at a moment when `rsethSupply == 0`. The attacker extracts ETH that belongs to the protocol/existing stakeholders, not merely unclaimed yield. This matches the Critical impact class: *Direct theft of any user funds, whether at-rest or in-motion, other than unclaimed yield.*

## Likelihood Explanation
The precondition (`rsethSupply == 0` with non-zero pool balance) is uncommon during steady-state operation but is realistic at two points: (1) protocol launch before the first deposit if any ETH is pre-seeded or rewards arrive early, and (2) any future period where all rsETH is redeemed and a reward drip arrives before a new depositor. Because `updateRSETHPrice()` is permissionless, an attacker can atomically observe the zero-supply state, call `updateRSETHPrice()` to lock in `rsETHPrice = 1e18`, and deposit in the same transaction. Likelihood is **Low**, but the attack is fully permissionless once the precondition exists and requires no privileged access.

## Recommendation
Replace the unconditional early return with a TVL-aware check:

```solidity
if (rsethSupply == 0) {
    uint256 totalETH = _getTotalEthInProtocol();
    if (totalETH > 0) revert OrphanedAssetsExist(totalETH);
    rsETHPrice = 1 ether;
    highestRsethPrice = 1 ether;
    return;
}
```

This forces governance to resolve any orphaned-asset state before new minting is allowed. Alternatively, restrict `receiveFromRewardReceiver()` to a designated reward receiver address so that unsolicited ETH cannot inflate TVL without minting rsETH.

## Proof of Concept
1. All rsETH has been redeemed → `rsethSupply == 0`. Reward receiver forwards 50 ETH to `LRTDepositPool` via `receiveFromRewardReceiver()`.
2. Attacker calls `LRTOracle.updateRSETHPrice()` → `rsETHPrice` is set to `1e18` (early return, TVL ignored).
3. Attacker calls `LRTDepositPool.depositETH{value: 1 ether}(0, "")` → `getRsETHAmountToMint` returns `(1e18 * 1e18) / 1e18 = 1e18`; attacker receives 1 rsETH.
4. Attacker calls `LRTOracle.updateRSETHPrice()` again → `rsethSupply = 1`, `totalETHInProtocol = 51 ETH`, `rsETHPrice = 51e18`.
5. Attacker calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 1e18, "")` → `expectedAssetAmount = 1e18 * 51e18 / 1e18 = 51 ETH`; availability check passes (51 ETH ≤ 51 ETH).
6. Operator performs normal `unlockQueue()` processing (routine operation, indistinguishable from legitimate withdrawal).
7. After `withdrawalDelayBlocks`, attacker calls `completeWithdrawal(ETH_TOKEN, "")` → receives 51 ETH, netting 50 ETH profit from the orphaned reward balance.
