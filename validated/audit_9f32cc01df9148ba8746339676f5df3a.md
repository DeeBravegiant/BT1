### Title
Proposer's Storage Deposit Permanently Lost When a Competing `propose_update` Wins — (File: `crates/contract/src/update.rs`)

---

### Summary

`propose_update` is `#[payable]` and requires the proposer to attach a NEAR deposit proportional to the update's storage footprint. When `do_update` executes for any winning proposal it unconditionally clears **all** pending entries — including those whose proposers paid deposits — without issuing any refund. The `bytes_used` field stored in every `UpdateEntry` is never used to return funds. Deposits paid for losing proposals are permanently absorbed into the contract's balance.

---

### Finding Description

`propose_update` collects a deposit calculated by `ProposedUpdates::required_deposit`: [1](#0-0) 

The exact deposit is computed from `bytes_used`, which for a contract-code update includes the full code length plus a fixed overhead for 128 hypothetical voter `AccountId` entries: [2](#0-1) 

Only the **excess** above the required amount is refunded to the proposer at proposal time: [3](#0-2) 

The required portion stays in the contract. When threshold votes accumulate for any proposal, `vote_update` calls `do_update`, which removes the winning entry and then **clears every other entry and every vote** with no refund loop: [4](#0-3) 

The `bytes_used` field stored in each `UpdateEntry` is never read back to issue a refund: [5](#0-4) 

There is no `cancel_proposal` or `withdraw_deposit` method anywhere in the contract. Once a competing proposal wins, the losing proposer's deposit is irrecoverable.

---

### Impact Explanation

A participant who proposes a contract upgrade must attach a deposit that scales with the code size. For a realistic NEAR contract (~500 KB), the required deposit is on the order of several NEAR tokens (NEAR charges ~10^19 yoctoNEAR per byte of storage). When a competing proposal reaches threshold and `do_update` fires, every other proposal's storage is freed but the corresponding deposits remain locked in the contract balance forever. This breaks the accounting invariant that storage deposits are returned when storage is freed, and constitutes a direct, permanent loss of funds for legitimate participants.

**Allowed impact match:** Medium — balance/accounting invariant broken without requiring network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

Any scenario where two or more participants independently propose different updates (e.g., one proposes a config change while another proposes a code upgrade, or two participants propose different code versions) triggers the loss for the non-winning proposer. This is a realistic operational scenario during governance activity, requiring no adversarial intent — normal threshold voting for a legitimate competing proposal is sufficient.

---

### Recommendation

1. **Refund on clear:** In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(proposer).transfer(entry.bytes_used * storage_byte_cost)` for each one. The `bytes_used` field already tracks the storage charged.
2. **Add a `cancel_proposal` method** that lets a proposer withdraw their own proposal and receive a refund before a competing update executes.
3. **Alternatively**, adopt a two-phase model: collect deposits only at vote-execution time (when the winning proposal is known), not at proposal time.

---

### Proof of Concept

1. Participant A calls `propose_update` with a 500 KB contract binary, attaching the required deposit (~5 NEAR). The deposit is kept; only excess is refunded.
2. Participant B calls `propose_update` with a different contract binary, attaching their own deposit.
3. Threshold participants call `vote_update` for B's proposal. The vote count crosses the threshold.
4. `do_update` is invoked for B's `UpdateId`. It removes B's entry, then calls `self.entries.clear()` and `self.vote_by_participant.clear()`.
5. A's entry — and the ~5 NEAR deposit attached to it — is erased with no refund transfer issued.
6. A's ~5 NEAR is permanently locked in the contract's balance with no recovery path. [6](#0-5)

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
