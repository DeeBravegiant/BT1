### Title
Missing CKD Response Validation for `AppPublicKey` Variant Allows Single Byzantine Participant to Inject Fake Confidential Key Derivation Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, when a pending CKD request was submitted with the `CKDAppPublicKey::AppPublicKey` variant, the contract performs **no on-chain verification** of the response before resolving the user's yield. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with a completely fabricated `CKDResponse`, and the contract will unconditionally deliver it to the user. This bypasses the threshold-participation guarantee that the CKD protocol is designed to enforce.

---

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` handles responses to confidential key derivation requests. For the `AppPublicKeyPV` variant, the contract calls `ckd_output_check` to mathematically verify the response against the BLS12-381 root public key and the user's app public key pair. For the `AppPublicKey` variant, the match arm is an explicit no-op:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

After the match, `resolve_yields_for` is called unconditionally. It removes the request from the pending map and resumes every queued yield with the supplied `response_bytes`:

```rust

``` [2](#0-1) 

Once `resolve_yields_for` drains the queue, the request is gone from contract state. No subsequent honest `respond_ckd` call can correct the outcome — the user's yield has already been resumed with the attacker-supplied bytes.

The contrast with `respond` (threshold ECDSA/EdDSA) is instructive: there, the contract calls `verify_ecdsa_signature` / `verify_eddsa_signature` against the derived public key. A valid signature can only be produced by threshold-many participants cooperating, so the mathematical check implicitly enforces the threshold requirement. For `AppPublicKey` CKD, no analogous check exists. [3](#0-2) 

The `AppPublicKey` variant is documented as "privately verifiable" (the user can check the result with their own private key), but this does not eliminate the on-chain security requirement: the contract must still ensure that the response was produced by threshold-many participants, not injected by a single Byzantine node.

---

### Impact Explanation

A single attested participant (one of the n nodes, strictly below the t+1 signing threshold) can:

1. Observe a pending `CKDRequest` keyed by `AppPublicKey` in `pending_ckd_requests`.
2. Construct an arbitrary `CKDResponse` (any encrypted blob).
3. Call `respond_ckd` with the fabricated response before the honest leader submits the real one.
4. The contract accepts the call (the caller is an attested participant, the protocol is running), skips the no-op match arm, and calls `resolve_yields_for`, which resumes the user's yield with the fake bytes and removes the request from state.

The user receives a fake confidential key derivation output. If the user's application uses this key (e.g., to decrypt a secret or derive a signing key for a foreign chain), the result is loss of funds or compromise of the derived secret. Even if the user detects the forgery, the original request has been permanently consumed; the user must resubmit and pay again, and the attacker can repeat the attack.

This is a direct bypass of the threshold-signature requirement for the CKD flow: one participant, not t+1, controls the output.

---

### Likelihood Explanation

Any single attested participant can execute this attack against any `AppPublicKey` CKD request at any time the contract is in `Running` or `Resharing` state. The attacker does not need to collude with other participants, does not need privileged access beyond being an attested node, and does not need to break any cryptographic primitive. The attack is a straightforward contract call. The `AppPublicKey` variant is described as the "legacy" path and is still accepted by the contract, so real users may submit requests using it.

---

### Recommendation

For `CKDAppPublicKey::AppPublicKey` requests, the contract cannot perform the same publicly-verifiable output check as `AppPublicKeyPV`. Two mitigations are possible:

1. **Deprecate and reject `AppPublicKey` at the contract level.** If `AppPublicKeyPV` provides the required public verifiability, refuse new `AppPublicKey` submissions and migrate existing users to `AppPublicKeyPV`. This eliminates the unverifiable path entirely.

2. **Restrict `respond_ckd` to the designated leader for `AppPublicKey` requests.** Record the leader's identity at request time and require that only the leader can submit the response for this variant. This does not provide a cryptographic guarantee but raises the bar from "any single participant" to "the leader must be Byzantine."

The cleanest fix, analogous to the M-06 mitigation (adding a revert on `address(0)`), is option 1: add an explicit rejection in `respond_ckd` for the `AppPublicKey` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {
        // AppPublicKey responses cannot be verified on-chain; reject to prevent
        // a single Byzantine participant from injecting a fake CKD output.
        env::panic_str("AppPublicKey CKD responses are not verifiable on-chain");
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. Alice calls `request_app_private_key` with `AppPublicKey(some_bls_g1_point)`, attaching 1 yoctoNEAR. The contract enqueues a yield and stores the `CKDRequest` in `pending_ckd_requests`.

2. Mallory is one of the n attested participants (e.g., participant index 3 in a 5-of-9 network). She observes the pending request on-chain.

3. Mallory constructs a `CKDResponse` containing arbitrary bytes (a fake encrypted key).

4. Mallory calls `respond_ckd(request, fake_response)`. The contract:
   - Confirms Mallory is an attested participant ✓
   - Confirms the protocol is running ✓
   - Enters the `AppPublicKey(_) => {}` arm — **no check** ✓
   - Calls `resolve_yields_for`, which resumes Alice's yield with `fake_response` and removes the request from `pending_ckd_requests`.

5. Alice's transaction resolves with Mallory's fabricated `CKDResponse`. The honest nodes' subsequent `respond_ckd` calls return `Err(RequestNotFound)` because the map entry is gone.

6. Alice's application decrypts the fake response with her private key, obtains garbage, and any funds or secrets protected by the derived key are at risk. [4](#0-3) [2](#0-1)

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
