Audit Report

## Title
Missing Chainlink Price Feed Staleness Check Allows Stale Prices to Corrupt rsETH Exchange Rate - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards `updatedAt` and `answeredInRound`, performing zero staleness validation. A stale low price for one LST deflates the stored `rsETHPrice`, enabling an attacker to deposit a non-stale asset at the deflated rate and mint excess rsETH, diluting existing holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` reads only the `answer` field:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No check of `block.timestamp - updatedAt > heartbeat` and no check that `answeredInRound >= roundId` is performed. This price flows through `LRTOracle.getAssetPrice()`: [2](#0-1) 

into `_getTotalEthInProtocol()`: [3](#0-2) 

and then into `_updateRsETHPrice()`, which stores the canonical `rsETHPrice`: [4](#0-3) 

`updateRSETHPrice()` is public and permissionless: [5](#0-4) 

The deposit calculation in `LRTDepositPool.getRsETHAmountToMint()` uses the stored `rsETHPrice` as the denominator and the live oracle price as the numerator:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [6](#0-5) 

**Exploit mechanics (multi-asset case):** Suppose Asset A (stale, low price) and Asset B (correct price) are both supported. When `updateRSETHPrice()` is called with Asset A's stale-low price, `rsETHPrice` is deflated below its true value. An attacker who then deposits Asset B receives `(amount × true_price_B) / deflated_rsETHPrice`, which is more rsETH than the deposited value warrants. The PoC in the submission incorrectly identifies the stale asset as the profitable deposit; the profitable deposit is the **non-stale** asset, because only the denominator (`rsETHPrice`) is deflated while the numerator (`getAssetPrice(B)`) remains correct.

The `pricePercentageLimit` guard provides partial mitigation: if the price drop exceeds the configured threshold, the protocol pauses. However, `pricePercentageLimit` defaults to `0` (no protection), and even when set, staleness within the threshold window is not caught. [7](#0-6) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral` performs partial checks (`answeredInRound < roundID`, `timestamp == 0`) but still omits the time-based heartbeat check; `ChainlinkPriceOracle` performs **no checks at all**. [8](#0-7) 

## Impact Explanation
When `pricePercentageLimit` is zero (default) or the staleness is within the configured threshold, an attacker can mint excess rsETH by depositing a non-stale asset against a deflated `rsETHPrice`. The excess rsETH represents a claim on more underlying ETH than was deposited, directly diluting the yield and principal of existing rsETH holders. This matches **High — Theft of unclaimed yield**. The SECURITY.md exclusion for "incorrect data supplied by third-party oracles" does not apply here because the root cause is a missing code-level validation, not oracle operator misbehavior per se; the note explicitly preserves oracle manipulation attacks. [9](#0-8) 

## Likelihood Explanation
Chainlink LST/ETH feeds have documented heartbeat intervals (1–24 hours). During low-volatility periods, feeds may not update for the full heartbeat window without any on-chain indication of staleness. `updateRSETHPrice()` is callable by any unprivileged address. No special role, flash loan, or governance action is required. The attack is repeatable whenever a feed is within its heartbeat window but has not updated recently. [5](#0-4) 

## Recommendation
Add configurable staleness validation in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

if (answeredInRound < roundId) revert StalePrice();
if (block.timestamp - updatedAt > stalenessThreshold) revert StalePrice();
if (price <= 0) revert InvalidPrice();
```

`stalenessThreshold` should be a per-feed configurable value set to slightly above the feed's documented heartbeat. Also add the time-based check to `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which currently only checks `answeredInRound < roundID` and `timestamp == 0`. [10](#0-9) 

## Proof of Concept
1. Two LST assets are supported: Asset A (Chainlink feed stale for >1 hour, last price 0.9 ETH, true price 1.0 ETH) and Asset B (feed current, price 1.0 ETH). Each has 100 tokens deposited; rsETH supply = 200; `pricePercentageLimit = 0`.
2. Attacker calls `LRTOracle.updateRSETHPrice()`. `_getTotalEthInProtocol()` computes `100×0.9 + 100×1.0 = 190 ETH`. `rsETHPrice` is stored as `190/200 = 0.95 ETH` (true value: 1.0 ETH).
3. Attacker calls `LRTDepositPool.depositAsset(AssetB, 1e18, 0, "")`. `getRsETHAmountToMint` computes `(1e18 × 1.0) / 0.95 ≈ 1.053e18` rsETH. True entitlement: `1.0e18` rsETH.
4. Attacker receives ~5.3% excess rsETH. When Asset A's oracle recovers and `rsETHPrice` corrects to 1.0 ETH, the attacker's excess rsETH redeems for more ETH than deposited, at the expense of existing holders.
5. **Foundry fork test plan:** fork mainnet, mock a Chainlink aggregator returning a stale `updatedAt` (e.g., `block.timestamp - 2 hours`) with a 5% lower price, call `updateRSETHPrice()`, then `depositAsset` with a non-stale asset, assert `rsethAmountToMint > (depositAmount × trueAssetPrice / trueRsETHPrice)`. [11](#0-10)
