### Title
Stale `stored_attestations` Entry After TLS Key Rotation Causes `lookup_node_id_by_signer_pk` to Return Wrong Node Identity, Silently Misdirecting Foreign-Chain Config Registration - (File: `crates/contract/src/tee/tee_state.rs`)

---

### Summary

`TeeState::add_participant` inserts a new `stored_attestations` entry keyed by the new TLS public key but never removes the old entry for the same account. After a TLS key rotation, two entries exist for the same account. `lookup_node_id_by_signer_pk` performs a linear scan and returns the **first** match by `account_public_key`; because `IterableMap` iterates in insertion order, it always returns the older (stale) entry. `register_foreign_chains_config` then registers the node's foreign-chain support under the stale TLS key, which is not present in the current `Participants` set, making the node's foreign-chain coverage permanently invisible to `recompute_available_foreign_chains` until the stale entry expires (up to 7 days).

---

### Finding Description

`stored_attestations` is an `IterableMap<Ed25519PublicKey, NodeAttestation>` keyed by TLS public key. [1](#0-0) 

`add_participant` guards against a *different* account overwriting an existing TLS-key entry, but performs no reverse lookup to evict any prior entry belonging to the *same* account under a different TLS key: [2](#0-1) 

When a node rotates its TLS key (K1 → K2) while keeping the same NEAR account public key P, the call sequence is:

1. `submit_participant_info(attestation, K1)` → `stored_attestations[K1] = {account: A, tls_pk: K1, account_pk: P}`
2. `submit_participant_info(attestation, K2)` → `stored_attestations[K2] = {account: A, tls_pk: K2, account_pk: P}`

The old entry under K1 is never removed. Both entries share the same `account_public_key = P`.

`lookup_node_id_by_signer_pk` performs a linear scan and returns the **first** match: [3](#0-2) 

Because `IterableMap` iterates in insertion order, K1 (inserted first) is always returned when the signer presents key P. `register_foreign_chains_config` uses this lookup to determine which TLS key to register the config under: [4](#0-3) 

The config is stored under K1 (stale). `recompute_available_foreign_chains` builds its active-TLS-key set from the current `Participants` struct, which contains K2: [5](#0-4) 

K1 is not in that set, so the config registered under K1 is never counted. The node appears to support no foreign chains.

---

### Impact Explanation

After a TLS key rotation, every subsequent call to `register_foreign_chains_config` by the affected node silently registers its foreign-chain support under the stale TLS key K1. `recompute_available_foreign_chains` never sees that config because it filters by the current participant TLS keys (K2). The node's foreign-chain coverage is permanently zeroed out from the contract's perspective until the stale attestation expires (up to 7 days, the `DEFAULT_EXPIRATION_DURATION_SECONDS` window). If multiple participants rotate TLS keys simultaneously, the available foreign-chain signing set can drop below the reconstruction threshold, blocking all cross-chain transaction signing. This is a participant-state and contract execution-flow invariant break: the contract's `available_foreign_chains` cache no longer reflects the actual capabilities of the participant set.

---

### Likelihood Explanation

TLS key rotation is a normal operational event (hardware refresh, CVM restart with new keys, migration). The `submit_participant_info` endpoint is open to any account. The stale entry persists for the full attestation validity window (up to 7 days) with no automatic per-account deduplication. The bug is deterministic: every call to `register_foreign_chains_config` after a rotation hits the stale entry because insertion order is stable. No adversarial intent is required; the corruption occurs during routine operations.

---

### Recommendation

In `add_participant`, before inserting the new entry, scan `stored_attestations` for any existing entry whose `node_id.account_id` matches the incoming `node_id.account_id` but whose key differs from the new `tls_pk`, and remove it:

```rust
// Before inserting the new entry:
let stale_key: Option<Ed25519PublicKey> = self
    .stored_attestations
    .iter()
    .find(|(k, v)| {
        v.node_id.account_id == node_id.account_id && **k != tls_pk
    })
    .map(|(k, _)| k.clone());

if let Some(old_key) = stale_key {
    self.stored_attestations.remove(&old_key);
}
```

This mirrors the `releaseTokenByName` recommendation from the reference report: dereserve the old mapping entry before inserting the new one.

### Proof of Concept

**State before rotation:**
- `stored_attestations[K1]` = `{account: "alice.near", tls_pk: K1, account_pk: P}`

**Node rotates TLS key, calls `submit_participant_info(attestation, K2)`:**
- `add_participant` checks: is K2 owned by another account? No → insert.
- `stored_attestations[K2]` = `{account: "alice.near", tls_pk: K2, account_pk: P}`
- `stored_attestations[K1]` still exists (not removed).

**`Participants` struct now contains K2 for `alice.near`.**

**Node calls `register_foreign_chains_config` signing with key P:**
- `lookup_node_id_by_signer_pk(P)` iterates: K1 comes first → returns `{account: "alice.near", tls_pk: K1}`.
- `node_id.account_id == signer_account_id` → passes.
- `is_existing_or_prospective_participant("alice.near")` → passes.
- `tls_key = K1` → `foreign_chains.register(K1, config)`.

**`recompute_available_foreign_chains` runs:**
- `active_tls_keys = {K2}` (from `Participants`).
- Config under K1 is not in `active_tls_keys` → not counted.
- `alice.near`'s foreign-chain support is invisible to the system. [6](#0-5) [7](#0-6) [5](#0-4)

    pub(crate) fn add_participant(
        &mut self,
        node_id: NodeId,
        attestation: Attestation,
        tee_upgrade_deadline_duration: Duration,
    ) -> Result<ParticipantInsertion, AttestationSubmissionError> {
        let expected_report_data: ReportData = ReportDataV1::new(
            *node_id.tls_public_key.as_bytes(),
            *node_id.account_public_key.as_bytes(),
        )
        .into();

        let accepted_measurements = self.get_accepted_measurements();
        // TODO(#3264): run DCAP in the verifier contract (Promise + callback) and
        // do the post-DCAP checks here, instead of verifying locally in-WASM.
        let AcceptedAttestation {
            attestation: verified_attestation,
            advisory_ids,
        } = attestation.verify_locally(
            expected_report_data.into(),
            Self::current_time_seconds(),
            &self.get_allowed_mpc_docker_image_hashes(tee_upgrade_deadline_duration),
            &self.get_allowed_launcher_compose_hashes(),
            &accepted_measurements,
        )?;

        log_informational_advisory_ids(&advisory_ids);

        let tls_pk = node_id.tls_public_key.clone();

        // Authorization: a TLS key registered to one account must not be
        // overwritten by a submission from a different account. Without this,
        // any caller could replace any participant's stored attestation, since
        // the entry is keyed only by `tls_public_key`.
        if let Some(existing) = self.stored_attestations.get(&tls_pk)
            && existing.node_id.account_id != node_id.account_id
        {
            return Err(AttestationSubmissionError::TlsKeyOwnedByOtherAccount);
        }

        let insertion = self.stored_attestations.insert(
            tls_pk,
            NodeAttestation {
                node_id,
                verified_attestation,
            },
        );

        Ok(match insertion {
            Some(_previous_attestation) => ParticipantInsertion::UpdatedExistingParticipant,
            None => ParticipantInsertion::NewlyInsertedParticipant,
        })

        self.stored_attestations
            .iter()

    pub fn register_foreign_chains_config(
        &mut self,
        foreign_chains_config: dtos::ForeignChainsConfig,
    ) -> Result<(), Error> {
        Self::assert_caller_is_signer();
        let signer_account_id = env::signer_account_id();
        let signer_account_pk = env::signer_account_pk();
        let signer_account_ed25519_pk = Ed25519PublicKey::try_from(&signer_account_pk)
            .unwrap_or_else(|_| env::panic_str("signer account key must be Ed25519"));
        let node_id = self
            .tee_state
            .lookup_node_id_by_signer_pk(&signer_account_ed25519_pk)
            .map_err(|_| InvalidState::NotParticipant {
                account_id: signer_account_id.clone(),
            })?;
        if node_id.account_id != signer_account_id {
            return Err(InvalidState::NotParticipant {
                account_id: signer_account_id,
            }
            .into());
        }
        let is_participant = self
            .protocol_state
            .is_existing_or_prospective_participant(&node_id.account_id)?;
        if !is_participant {
            return Err(InvalidState::NotParticipant {
                account_id: node_id.account_id.clone(),
            }
            .into());
        }
        let tls_key = node_id.tls_public_key.clone();

        self.foreign_chains
            .get_mut()
            .register(tls_key, foreign_chains_config);
        self.recompute_available_foreign_chains();

        Ok(())
    }
