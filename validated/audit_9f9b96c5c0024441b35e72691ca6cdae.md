Audit Report

## Title
Missing Staleness Check on Chainlink `latestRoundData()` Allows Stale Asset Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` fields, meaning a Chainlink feed that has stopped updating will continue to return its last known price indefinitely. This stale price flows directly into rsETH minting arithmetic in `LRTDepositPool.getRsETHAmountToMint()`, enabling depositors to receive more rsETH than the true value of their deposit warrants when a feed is stale-high, diluting existing rsETH holders and driving protocol insolvency.

## Finding Description

In `contracts/oracles/ChainlinkPriceOracle.sol` at L52, `getAssetPrice()` fetches the Chainlink price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values of `latestRoundData()` — `(roundId, answer, startedAt, updatedAt, answeredInRound)` — are discarded except `answer`. There is no check of the form `require(block.timestamp - updatedAt <= maxStaleness)`, no `require(answeredInRound >= roundId)`, and no `require(price > 0)`.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` at L30–32 does check `answeredInRound < roundID` and `timestamp == 0`, demonstrating the team is aware of partial staleness guards but failed to apply them to `ChainlinkPriceOracle`.

The stale price propagates through `LRTOracle.getAssetPrice()` (L156–158) into `LRTDepositPool.getRsETHAmountToMint()` at L520:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

The `rsETHPrice` denominator is a weighted average across all supported assets and is updated in a separate transaction via `updateRSETHPrice()`. If only one asset's feed is stale-high, the numerator is fully inflated by the stale price while the denominator is only partially inflated (proportional to that asset's share of TVL). The mismatch results in excess rsETH minted per unit of deposited asset.

## Impact Explanation

**Critical — Direct theft of funds from existing rsETH holders / Protocol insolvency.**

If the Chainlink feed for a supported LST (e.g., stETH) reports a stale-high price (e.g., 1.05e18 instead of the true 0.95e18 after a de-peg):

- `getAssetPrice(stETH)` returns 1.05e18.
- `getRsETHAmountToMint` mints `amount * 1.05e18 / rsETHPrice` — excess rsETH proportional to the price discrepancy.
- The attacker redeems the excess rsETH, extracting ETH worth more than the deposited assets at true market value.
- Every existing rsETH holder is diluted; the protocol's ETH backing per rsETH decreases, leading to insolvency.

This is a concrete, quantifiable loss to existing rsETH holders triggered by a public `depositAsset` call.

## Likelihood Explanation

Chainlink LST/ETH feeds (e.g., stETH/ETH) have 24-hour heartbeat windows and 0.5% deviation thresholds. During low-volatility periods, the feed can legally go many hours without an update. Any network congestion, oracle node failure, or L1 disruption that prevents a heartbeat update creates a window where stale prices are silently accepted. This is a realistic, non-adversarial scenario that has occurred historically on mainnet. The attacker does not need to cause the staleness — they only need to observe it and deposit before the feed updates. The attack is repeatable for every supported LST feed.

## Recommendation

Add a configurable `maxStaleness` parameter per feed and validate `updatedAt` and `answeredInRound` in `getAssetPrice()`:

```solidity
mapping(address asset => uint256 maxStaleness) public assetMaxStaleness;

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();
    require(answeredInRound >= roundId, "Stale round");
    require(updatedAt != 0, "Incomplete round");
    require(block.timestamp - updatedAt <= assetMaxStaleness[asset], "Stale price");
    require(price > 0, "Non-positive price");
    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Set `assetMaxStaleness` per feed to slightly exceed each feed's documented heartbeat interval. Mirror the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`.

## Proof of Concept

1. Chainlink stETH/ETH feed last updated at `T - 20 hours`; its price is 1.05e18 (stETH at a premium). The feed is within its 24-hour heartbeat so no on-chain circuit breaker fires.
2. stETH de-pegs off-chain; true market price falls to 0.95e18. The Chainlink feed has not yet updated.
3. `rsETHPrice` was last computed via `updateRSETHPrice()` at `T - 1 hour` using correct prices (0.95e18 for stETH at that time, or any price before the de-peg).
4. Attacker calls `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")`.
5. `getRsETHAmountToMint` computes `100e18 * 1.05e18 / rsETHPrice` — minting excess rsETH relative to the true 0.95e18 value.
6. Attacker redeems via `LRTWithdrawalManager`, extracting ETH worth more than the 100 stETH deposited at true market value.
7. Existing rsETH holders are diluted; the protocol's ETH backing per rsETH decreases — insolvency.

**Foundry fork test plan:** Fork mainnet, mock the stETH/ETH Chainlink feed to return a stale-high `answer` with an `updatedAt` 20 hours in the past, call `depositAsset` with stETH, assert that `rsethAmountToMint` exceeds the fair value computed at the true price, and assert that existing rsETH holders' redemption value decreases after the deposit.
