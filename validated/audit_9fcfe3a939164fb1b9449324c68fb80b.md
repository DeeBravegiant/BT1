Audit Report

## Title
Stale `rsETHPrice` vs. Fresh `getAssetPrice` Mismatch in Deposit Minting Allows Over-Minting of rsETH at Existing Holders' Expense — (`contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool.getRsETHAmountToMint` divides a live asset price by a stored, potentially stale `rsETHPrice` state variable. Because `rsETHPrice` is only written inside `_updateRsETHPrice()` and that function is never called atomically with a deposit, any period of rising LST prices creates a window where depositors receive more rsETH than their deposit is worth at the true current exchange rate, directly diluting the accrued yield of all existing rsETH holders.

## Finding Description
`getRsETHAmountToMint` at line 520 of `LRTDepositPool.sol` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`lrtOracle.getAssetPrice(asset)` is a live, view-only call that delegates to an external `IPriceFetcher` and returns the current market price at execution time (`LRTOracle.sol` line 156–158). `lrtOracle.rsETHPrice()` reads the public state variable `rsETHPrice` (`LRTOracle.sol` line 28), which is only written at line 313 inside `_updateRsETHPrice()`. That internal function is only reachable via `updateRSETHPrice()` (line 87) or `updateRSETHPriceAsManager()` (line 94); neither is called from `depositETH` (lines 76–93) or `depositAsset` (lines 99–118).

The true rsETH price at any moment is `totalETHInProtocol / rsethSupply`, where `totalETHInProtocol` is itself computed using live `getAssetPrice` calls for every supported asset (see `_getTotalEthInProtocol`, `LRTOracle.sol` lines 331–349). When any supported LST appreciates after the last oracle update:

- Stored `rsETHPrice` is **lower** than the true current price (computed with old, lower asset prices).
- Fresh `getAssetPrice(asset)` in the numerator is **higher** (current market).

The formula therefore mints more rsETH than the depositor is entitled to at the true current exchange rate. The excess rsETH represents the accrued yield of existing holders being diluted away.

Existing guards do not prevent this:
- `minRSETHAmountExpected` (the slippage parameter in `_beforeDeposit`, line 667) protects the *depositor* from receiving too little; it does not protect existing holders from dilution.
- The `pricePercentageLimit` check in `_updateRsETHPrice` (lines 252–266) only fires when the oracle is explicitly updated, not during deposits.
- `nonReentrant` and `whenNotPaused` modifiers are irrelevant to the price mismatch.

## Impact Explanation
**High — Theft of unclaimed yield.**

Every rsETH minted in excess of the true exchange rate permanently dilutes the proportional claim of all existing holders. The value extracted is the accrued appreciation of the underlying LSTs (stETH, ETHx, etc.) since the last oracle update — i.e., yield that existing holders had already earned but not yet claimed. This matches the allowed impact class "High. Theft of unclaimed yield." The dilution is irreversible once the excess rsETH is minted; a subsequent oracle update will correctly reprice rsETH but cannot un-dilute existing holders.

## Likelihood Explanation
**Medium-High.** No special privileges, flash loans, or governance access are required. `updateRSETHPrice()` is a separate public call that is not atomically bundled with deposits. LST prices (stETH, ETHx) accrue value continuously with each Ethereum epoch, so the stale-price window is always open to some degree between keeper runs. Any ordinary depositor calling `depositAsset()` or `depositETH()` benefits from the mismatch whenever asset prices have risen since the last oracle update. A sophisticated actor can monitor the oracle staleness and time large deposits to maximize the over-minting.

## Recommendation
Call `lrtOracle.updateRSETHPrice()` at the start of `depositETH()` and `depositAsset()` (before `_beforeDeposit` is invoked) so that both the asset price and the rsETH price are derived from the same on-chain state. Alternatively, replace the stored `rsETHPrice` read in `getRsETHAmountToMint` with a live, view-only computation: `_getTotalEthInProtocol() / rsethSupply`, ensuring both sides of the division use current prices. The latter approach avoids the gas cost and reentrancy surface of a state-mutating oracle update on every deposit.

## Proof of Concept
**Setup:**
- Protocol holds 100 stETH (each worth 1.00 ETH) and 100 ETHx (each worth 1.00 ETH).
- Total TVL = 200 ETH, rsETH supply = 200, stored `rsETHPrice` = 1.00 ETH.

**Price movement (no oracle update):**
- stETH appreciates to 1.05 ETH per token (normal LST yield accrual).
- True rsETH price = (100 × 1.05 + 100 × 1.00) / 200 = **1.025 ETH**.
- Stored `rsETHPrice` = **1.00 ETH** (stale).

**Attacker calls `depositAsset(stETH, 10e18, 0, "")` without calling `updateRSETHPrice()` first:**

```
rsethAmountToMint = (10e18 × 1.05e18) / 1.00e18 = 10.5 rsETH   ← actual minted
true entitlement  = (10e18 × 1.05e18) / 1.025e18 ≈ 10.244 rsETH
over-minted       ≈ 0.256 rsETH  (≈ 2.5% excess, ≈ 0.262 ETH of value)
```

**Foundry fork test plan:**
1. Fork mainnet at a block where `rsETHPrice` is known.
2. Warp forward by the oracle update interval without calling `updateRSETHPrice()`.
3. Record `lrtOracle.rsETHPrice()` (stale) and `lrtOracle.getAssetPrice(stETH)` (fresh).
4. Call `depositAsset` with a large stETH amount.
5. Assert that `rsethMinted > (depositAmount × freshAssetPrice) / trueCurrentRsETHPrice`, where `trueCurrentRsETHPrice` is computed as `_getTotalEthInProtocol() / rsethSupply` using current prices.

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
