### Title
Untrusted Peer Can Corrupt DataLayer Singleton Store via Crafted CoinSpend Solution — (`chia/data_layer/data_layer_wallet.py`, `chia/wallet/util/wallet_sync_utils.py`)

---

### Summary

When a tracked DataLayer singleton coin is spent, the wallet fetches the `CoinSpend` from a peer and passes it directly to `singleton_removed`. The peer-supplied **solution** is never validated against the blockchain. An attacker peer can return the correct puzzle (satisfying the only guard) paired with a crafted solution whose CLVM execution produces a `CREATE_COIN` condition with attacker-controlled bytes at hint indices `[1]` and `[2]`. These are read verbatim as `root` and `inner_puzzle_hash` and written into the DL singleton store, permanently corrupting the wallet's view of the DataLayer Merkle root.

---

### Finding Description

**Step 1 — Entry point: `_add_coin_states`**

When a `WalletType.DATA_LAYER` coin is observed as spent, the wallet unconditionally fetches the spend from the connected peer and dispatches it: [1](#0-0) 

**Step 2 — `fetch_coin_spend` only validates the puzzle hash, not the solution**

`fetch_coin_spend` checks that `puzzle.get_tree_hash() == coin.puzzle_hash`, which pins the puzzle to the on-chain value. The solution field is accepted verbatim with zero validation: [2](#0-1) 

**Step 3 — `singleton_removed` runs the puzzle+solution and blindly trusts the hint bytes**

`match_dl_singleton` only checks the puzzle's structural shape (it is a `SINGLETON_TOP_LAYER_MOD` wrapping an `NFT_STATE_LAYER_MOD` with the correct metadata updater hash). It says nothing about the solution: [3](#0-2) 

`run_with_cost` then executes the CLVM program with the attacker-supplied solution. No signature validation occurs here — `AGG_SIG_ME` conditions are simply emitted as output, not enforced. The first `CREATE_COIN` condition with an odd amount is found, and `condition[3][1]` / `condition[3][2]` are cast directly to `bytes32` and stored: [4](#0-3) 

**Step 4 — Why the CLVM execution can be made to succeed**

The DL singleton puzzle is `SINGLETON_TOP_LAYER_MOD.curry(singleton_struct, db_layer)`. Its lineage-proof check computes:

```
sha256(parent_name || sha256tree(singleton_puzzle(parent_inner_puzzle)) || amount) == coin.parent_coin_info
```

All inputs are public on-chain data the attacker already knows. The attacker supplies the correct `parent_name`, the correct `parent_inner_puzzle` (the actual inner puzzle, whose hash is curried into the fixed puzzle), and the correct `amount`. The lineage proof check passes.

The innermost puzzle is the standard wallet puzzle (`p2_delegated_puzzle_or_hidden_puzzle`). Its solution includes a *delegated puzzle* and *delegated solution*. The attacker sets the delegated puzzle to a trivial program that emits:

```clvm
(CREATE_COIN <any_puzzle_hash> <odd_amount> (<launcher_id> <attacker_root> <attacker_inner_puz_hash>))
```

`run_with_cost` returns this condition. No BLS signature is checked at this layer. The `AGG_SIG_ME` condition the standard wallet puzzle also emits is simply ignored by `singleton_removed` — it only iterates looking for `CREATE_COIN`.

### Impact Explanation

`dl_store.add_singleton_record` is called with `root=attacker_root` and `inner_puzzle_hash=attacker_inner_puz_hash`: [5](#0-4) 

This record becomes the wallet's authoritative view of the singleton's current state. Every subsequent operation that reads the latest singleton record — including `get_coins_to_offer`, `create_update_state_spend`, and offer settlement — will use the corrupted root and inner puzzle hash: [6](#0-5) 

Consequences include:
- The wallet constructs spends referencing a wrong Merkle root, making all future DL updates invalid.
- DL proof verification (`dl_verify_proof_internal`) will fail or succeed against the wrong root, breaking data integrity guarantees.
- The corrupted `lineage_proof` (also derived from the attacker-controlled values at line 854–857) will cause any subsequent singleton spend to be rejected by consensus.

### Likelihood Explanation

Any peer the wallet is connected to can trigger this. The wallet connects to full nodes over the standard peer protocol; an attacker only needs to operate a node that the victim wallet connects to (or perform a man-in-the-middle on an existing connection). The trigger condition — a tracked DL singleton being spent — is a normal network event. No keys, admin access, or cryptographic breaks are required.

### Recommendation

After fetching the `CoinSpend` from the peer, validate the solution before trusting its CLVM output. The minimum viable fix is to cross-check the child coin(s) produced by `compute_additions(parent_spend)` against the on-chain children fetched from the blockchain (already available as `children` in `_add_coin_states`). Specifically:

1. Compute `additions = compute_additions(singleton_spend)` locally.
2. Verify that the odd-amount child coin's `name()` matches a coin ID confirmed on-chain (e.g., from `wallet_node.fetch_children`).
3. Only then extract `root` and `inner_puzzle_hash` from the CLVM output.

This ensures the solution is consistent with the actual on-chain state, regardless of what the peer returns.

def match_dl_singleton(puzzle: Program | SerializedProgram) -> tuple[bool, Iterator[Program]]:
    """
    Given a puzzle test if it's a CAT and, if it is, return the curried arguments
    """
    mod, singleton_curried_args = uncurry(puzzle)
    if mod == SINGLETON_TOP_LAYER_MOD:
        mod, dl_curried_args = uncurry(singleton_curried_args.at("rf"))
        if mod == NFT_STATE_LAYER_MOD and dl_curried_args.at("rrf") == ACS_MU_PH:
            launcher_id = singleton_curried_args.at("frf")
            root = dl_curried_args.at("rff")
            innerpuz = dl_curried_args.at("rrrf")
            return True, iter((innerpuz, root, launcher_id))

    return False, iter(())
