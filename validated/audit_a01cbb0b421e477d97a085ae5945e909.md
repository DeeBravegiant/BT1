Audit Report

## Title
LP Full-Withdrawal DoS via Griefing `addLiquidity` on Victim's Position — (`metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no `msg.sender == owner` guard in the base pool, allowing any caller to credit shares to a third party's position. `LiquidityLib.removeLiquidity` reverts with `MinimalLiquidity` when the remaining shares after a burn fall strictly between 0 and `MINIMAL_MINTABLE_LIQUIDITY`. An attacker can front-run a victim's full-withdrawal transaction by adding a dust amount of shares to the victim's position, causing the victim's transaction to revert because the remaining share count falls in the forbidden dust range. The attack is repeatable at negligible cost, rendering the victim's withdraw flow permanently unusable for the targeted position.

## Finding Description
`MetricOmmPool.addLiquidity` (L182–196) passes `owner` directly to `LiquidityLib.addLiquidity` without any `msg.sender == owner` check. The only pre-call hook is `_beforeAddLiquidity` (ExtensionCalling.sol L88–99), which calls `_callExtensionsInOrder(BEFORE_ADD_LIQUIDITY_ORDER, ...)`. When `BEFORE_ADD_LIQUIDITY_ORDER == 0` (no allowlist extension configured — the default), this returns immediately with no access control. Inside `LiquidityLib.addLiquidity`, the position key is computed as `keccak256(abi.encode(owner, salt, bin))` and `positionBinShares[posKey]` is incremented by `sharesToAdd` with no check that `msg.sender == owner` (LiquidityLib.sol L72–121). The attacker must implement `IMetricOmmModifyLiquidityCallback` to satisfy the token payment callback (LiquidityLib.sol L147–148), which is a trivial barrier for a contract attacker.

`LiquidityLib.removeLiquidity` (L199–202) enforces:
```solidity
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```
This check fires on partial removals that leave a dust remainder, but it also fires when an attacker has inflated `userShares` by 1 before a full-exit transaction executes.

**Exploit path:**
1. Alice holds `N` shares in bin `B`, salt `S`, and submits `removeLiquidity(alice, S, [{B, N}])` (full exit, would leave 0 shares — valid).
2. Bob (attacker contract) front-runs with `addLiquidity(alice, S, [{B, 1}])`. Since `N + 1 ≥ MINIMAL_MINTABLE_LIQUIDITY`, the add succeeds. Bob pays ~`1/N` of the bin's token balance.
3. Alice's transaction executes: `userShares = N+1`, `sharesToRemove = N`, `newUserShares = 1`. The check `1 > 0 && 1 < MINIMAL_MINTABLE_LIQUIDITY` is true → `revert MinimalLiquidity(1, MINIMAL_MINTABLE_LIQUIDITY)`.
4. Bob repeats on every resubmission.

`removeLiquidity` correctly enforces `msg.sender == owner` (MetricOmmPool.sol L206), so only Alice can remove — but Alice cannot remove because the check always reverts as long as Bob keeps front-running.

## Impact Explanation
The victim LP's withdraw flow for the targeted position is rendered permanently unusable as long as the attacker continues front-running. The victim's funds are not stolen but are inaccessible via the normal `removeLiquidity` path. This matches the "broken core pool functionality causing unusable withdraw flow" criterion in the allowed impact gate. The victim has no on-chain recourse: any full-exit transaction specifying the original share count will revert after the attacker's front-run, and any resubmission with the updated count can be front-run again.

## Likelihood Explanation
No special privilege is required; any unprivileged address (as a contract implementing the callback) can call `addLiquidity` with an arbitrary `owner`. The victim's `(owner, salt, binIdx, shares)` tuple is fully visible in pending transaction calldata. The attacker's per-attempt cost is `1 / binTotalShares` of the bin's token balance — negligible for large bins. All pools without a `DEPOSIT_ALLOWLIST_PROVIDER` extension (the default configuration) are fully exposed. The attack is repeatable indefinitely.

## Recommendation
**Minimal fix in `LiquidityLib.removeLiquidity`:** Skip the `MinimalLiquidity` dust check when `sharesToRemove == userShares` (full exit), since leaving exactly 0 shares is always valid:
```solidity
uint256 newUserShares = userShares - sharesToRemove;
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```
Change to:
```solidity
uint256 newUserShares = userShares - sharesToRemove;
// Full exit (newUserShares == 0) is always valid; only enforce dust floor on partial removals.
if (newUserShares != 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```
**Complementary fix:** Require `msg.sender == owner` in `addLiquidity` unless an explicit operator approval exists, or document that pools without an allowlist extension are exposed to this griefing vector.

## Proof of Concept
```
Setup:
  MINIMAL_MINTABLE_LIQUIDITY = 1000
  Alice holds 10_000 shares in bin 4, salt = 42.

Step 1 – Alice submits:
  pool.removeLiquidity(alice, 42, [{binIdx: 4, shares: 10_000}], "")
  Expected: removes all shares, receives tokens.

Step 2 – Bob (attacker contract) front-runs (same block, higher gas):
  pool.addLiquidity(alice, 42, [{binIdx: 4, shares: 1}], callbackData, "")
  Bob's contract pays ~1/10_000 of bin 4's token balance via callback.
  Result: positionBinShares[key(alice,42,4)] = 10_001.

Step 3 – Alice's transaction executes:
  userShares      = 10_001
  sharesToRemove  = 10_000
  newUserShares   = 1
  Check: 1 > 0 && 1 < 1000 → true → revert MinimalLiquidity(1, 1000)

Alice's withdrawal fails. Bob repeats on every resubmission at ~1/N token cost per attempt.

Foundry test outline:
  1. Deploy pool with MINIMAL_MINTABLE_LIQUIDITY = 1000, no allowlist extension.
  2. Alice adds 10_000 shares to bin 4.
  3. Bob (attacker contract implementing IMetricOmmModifyLiquidityCallback) calls
     addLiquidity(alice, 42, [{4, 1}], ...) — succeeds.
  4. Alice calls removeLiquidity(alice, 42, [{4, 10_000}], "") — reverts MinimalLiquidity(1, 1000).
  5. Assert revert.
```
