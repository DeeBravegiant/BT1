Audit Report

## Title
`_withdrawFromAave` Does Not Cap Withdrawal to Actual Aave Pool Liquidity, Causing `completeWithdrawal` to Revert for ETH Withdrawers - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`_withdrawFromAave` computes `withdrawnAmount` using the aWETH accounting balance (`aaveAWETH.balanceOf(address(this))`) rather than the actual WETH liquidity available in the Aave pool. When Aave WETH pool utilization is high, the `aaveWETHGateway.withdrawETH` call reverts, propagating up through `_processWithdrawalCompletion` and causing every `completeWithdrawal(ETH_TOKEN, ...)` call to revert. All pending ETH withdrawers are temporarily frozen until Aave utilization recovers.

## Finding Description
In `_withdrawFromAave` (L905–921), `withdrawnAmount` is bounded by `min(amount, min(aaveBalance, totalETHDepositedToAave))`, where `aaveBalance = aaveAWETH.balanceOf(address(this))` is an accounting figure representing the protocol's claim on the pool, not the pool's actual WETH reserves. [1](#0-0) 

The contract already exposes the correct liquidity figure via `getAaveWithdrawableLiquidity()` (L684–689), which returns `IERC20(WETH_ADDRESS).balanceOf(address(aaveAWETH))`: [2](#0-1) 

However, `_withdrawFromAave` never consults this value. When pool liquidity `L < withdrawnAmount`, the call at L917 reverts unconditionally inside Aave.

This revert propagates through `_processWithdrawalCompletion` (L699–738), which is called directly by the user-facing `completeWithdrawal` (L183–185): [3](#0-2) 

Because the entire transaction reverts, the state mutations at L705 (`popFront()`), L712 (`delete withdrawalRequests[requestId]`), and L717 (`unlockedWithdrawalsCount[asset]--`) are also reverted, preserving the request but leaving the user unable to claim their ETH.

The existing post-withdrawal guard at L727–730 (`if (balanceAfter < request.expectedAssetAmount) revert InsufficientLiquidityForWithdrawal()`) is never reached because the external call at L917 reverts first, bypassing the graceful error path entirely. [4](#0-3) 

## Impact Explanation
**Medium — Temporary freezing of funds.** All users with unlocked ETH withdrawal requests are blocked from calling `completeWithdrawal(ETH_TOKEN, ...)` for the duration of the Aave pool liquidity shortage. The rsETH burned at `initiateWithdrawal` time is held in the contract, and the ETH owed cannot be claimed. No funds are permanently lost (requests survive the revert), but access to owed ETH is frozen until Aave utilization drops. This matches the allowed impact "Medium. Temporary freezing of funds."

## Likelihood Explanation
No attacker action is required. The Aave WETH market on Ethereum Mainnet regularly experiences utilization spikes above 95% during periods of market stress (ETH price volatility, liquidation cascades). Any user whose `expectedAssetAmount` exceeds the contract's idle ETH balance will trigger `_withdrawFromAave`, and if pool liquidity is insufficient, every such `completeWithdrawal` attempt reverts. The Aave integration is a live, enabled feature (`isAaveIntegrationEnabled`), making this a realistic and repeatable condition under normal market operations.

## Recommendation
Before calling `aaveWETHGateway.withdrawETH`, cap `withdrawnAmount` to the actual WETH liquidity available in the Aave pool:

```solidity
function _withdrawFromAave(uint256 amount) internal returns (uint256 withdrawnAmount) {
    if (amount == 0) return 0;

    uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
    if (aaveBalance == 0) revert InsufficientAaveBalance();

    uint256 withdrawablePrincipal = aaveBalance < totalETHDepositedToAave
        ? aaveBalance
        : totalETHDepositedToAave;

    // Cap to actual WETH liquidity in the Aave pool to avoid revert on high utilization
    uint256 poolLiquidity = IERC20(WETH_ADDRESS).balanceOf(address(aaveAWETH));
    if (withdrawablePrincipal > poolLiquidity) withdrawablePrincipal = poolLiquidity;

    withdrawnAmount = amount > withdrawablePrincipal ? withdrawablePrincipal : amount;
    if (withdrawnAmount == 0) return 0;

    aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this));
    totalETHDepositedToAave -= withdrawnAmount;
    emit ETHWithdrawnFromAave(withdrawnAmount, totalETHDepositedToAave);
}
```

This allows `_processWithdrawalCompletion` to withdraw whatever liquidity is available, and the existing `InsufficientLiquidityForWithdrawal` check at L727–730 will surface the shortfall gracefully without an unexpected external revert. [5](#0-4) 

## Proof of Concept
1. Operator enables Aave integration (`setAaveIntegrationEnabled(true)`) and calls `unlockQueue(ETH_TOKEN, ...)`, depositing ETH to Aave. `totalETHDepositedToAave = X`.
2. Aave WETH pool utilization spikes to ~100%; available WETH liquidity drops to `L < X`.
3. User calls `completeWithdrawal(ETH_TOKEN, referralId)`.
4. `_processWithdrawalCompletion` finds `address(this).balance < request.expectedAssetAmount`, computes `amountNeeded`, and calls `_withdrawFromAave(amountNeeded)`.
5. `_withdrawFromAave` computes `withdrawnAmount = min(amountNeeded, X)` — bounded by aWETH accounting balance, not pool liquidity.
6. `aaveWETHGateway.withdrawETH(aavePool, withdrawnAmount, address(this))` is called with `withdrawnAmount > L`.
7. Aave pool reverts due to insufficient liquidity; entire transaction reverts.
8. All ETH withdrawers are blocked until Aave utilization recovers.

**Foundry fork test plan:** Fork Ethereum mainnet at a block where Aave WETH utilization is high (>95%). Deploy/use the existing `LRTWithdrawalManager` with Aave enabled. Simulate `unlockQueue` to deposit ETH to Aave. Warp/roll to satisfy `withdrawalDelayBlocks`. Call `completeWithdrawal(ETH_TOKEN, "")` and assert the transaction reverts with the Aave pool's liquidity error rather than `InsufficientLiquidityForWithdrawal`.
