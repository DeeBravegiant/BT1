Audit Report

## Title
`protocolUnpausePool` Sets Pause Level to 1 Instead of 0, Leaving Pool Permanently Swap-Paused After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

## Summary
`MetricOmmPoolFactory.protocolUnpausePool` is intended to restore a protocol-paused pool (level 2) to active status (level 0), but it calls `setPause(1)` instead of `setPause(0)`. After the protocol owner calls `protocolUnpausePool`, the pool transitions to level 1 (admin-paused), not level 0 (active). Because `_checkNotPaused()` reverts on any non-zero `pauseLevel`, all swaps remain blocked. If the pool admin is absent or unresponsive, the pool is permanently stuck at level 1 with no protocol-owner path to restore swap functionality.

## Finding Description
`MetricOmmPool` defines three pause levels: 0 = active, 1 = paused by admin, 2 = paused by protocol (`MetricOmmPool.sol` L71–72). `_checkNotPaused()` reverts with `PoolPaused` whenever `pauseLevel != 0` (`MetricOmmPool.sol` L643–644). `swap` applies `whenNotPaused` (`MetricOmmPool.sol` L217–224).

`protocolUnpausePool` reads `cur`, asserts `cur == 2`, then calls `IMetricOmmPoolFactoryActions(pool).setPause(1)` (`MetricOmmPoolFactory.sol` L399–403). This sets `pauseLevel = 1`, which still satisfies `pauseLevel != 0`, so every subsequent `swap` call reverts with `PoolPaused`.

The only path to level 0 after this is the pool admin calling `unpausePool`, which checks `cur == 1` and calls `setPause(0)` (`MetricOmmPoolFactory.sol` L467–471). The protocol owner has no further lever to restore swaps unilaterally.

## Impact Explanation
Core swap functionality is broken after every protocol-pause/unpause cycle. Traders cannot execute swaps; fee accrual stops. If the pool was at level 0 before the protocol pause, the admin never issued a pause and may not know they must call `unpausePool`. If the admin is unresponsive, the pool is permanently stuck at level 1. This matches the allowed impact gate: broken core pool functionality causing unusable swap flows.

## Likelihood Explanation
The bug fires unconditionally every time the protocol owner calls `protocolUnpausePool`. No attacker, no special preconditions, no race condition required. Every protocol-pause/unpause cycle triggers it.

## Recommendation
Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool` (`MetricOmmPoolFactory.sol` L402):

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

Also update the revert argument from `1` to `0` to reflect the intended target level.

## Proof of Concept
1. Pool is active: `pauseLevel == 0`.
2. Protocol owner calls `protocolPausePool(pool)` → `setPause(2)` → `pauseLevel == 2`. Swaps revert.
3. Protocol owner calls `protocolUnpausePool(pool)`: `cur == 2` passes the guard; `setPause(1)` → `pauseLevel == 1`.
4. Any trader calls `swap(...)` → `whenNotPaused` → `_checkNotPaused()` → `pauseLevel == 1 != 0` → **`PoolPaused` revert**.
5. Pool admin must separately call `unpausePool(pool)` (checks `cur == 1`) → `setPause(0)` to restore swaps.
6. If admin is absent, pool remains permanently paused at level 1 despite the protocol owner having called "unpause."
