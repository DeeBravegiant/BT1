### Title
Unvalidated `ProposalInit.builder` (sequencer address) Allows Malicious Proposer to Redirect Transaction Fees and Corrupt Block Hash - (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` enforces strict checks on `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas price fields, `starknet_version`, `version_constant_commitment`, and `fee_proposal_fri`, but never validates `ProposalInit.builder`. The `builder` field is the sequencer address that flows directly into `BlockInfo.sequencer_address`, which is used for fee collection during execution and is hashed into `PartialBlockHashComponents.sequencer` → `partial_block_hash` → `ProposalCommitment`. A malicious-but-legitimately-selected proposer can set `builder` to any arbitrary address; every validator will accept the init, execute with the injected sequencer address, and vote for the resulting commitment.

### Finding Description

`is_proposal_init_valid` in `validate_proposal.rs` validates the following `ProposalInit` fields: [1](#0-0) 

It also validates `timestamp`, `starknet_version`, `version_constant_commitment`, all four L1 gas price fields, and `fee_proposal_fri`. The `builder` field is never mentioned.

`builder` is then passed verbatim to `convert_to_sn_api_block_info`, which maps it to `sequencer_address`: [2](#0-1) 

This `BlockInfo` is forwarded to the batcher via `initiate_validation`: [3](#0-2) 

The batcher executes all transactions with `sequencer_address` set to the proposer-supplied `builder`. Fees are credited to that address. The same `sequencer_address` is embedded in `PartialBlockHashComponents`: [4](#0-3) 

`PartialBlockHash::from_partial_block_hash_components` hashes these components, and `proposal_commitment_from` wraps the result: [5](#0-4) 

Because both proposer and validator derive `proposal_commitment` from the same (unvalidated) `builder`, the commitments match, consensus votes succeed, and the block is committed with the injected sequencer address.

The honest proposer sets `builder` from its own configured address: [6](#0-5) 

No corresponding check exists on the validator side.

### Impact Explanation

**Critical — Incorrect fee/balance effect with economic impact.**

Every transaction fee charged during block execution is credited to `sequencer_address`. By injecting an arbitrary `builder`, a malicious proposer redirects all transaction fees for that block to an address of their choosing. Additionally, `sequencer_address` is a committed field in the block hash (`calculate_block_hash` chains it at position 4): [7](#0-6) 

The on-chain block hash therefore reflects the injected address, permanently corrupting the canonical record of who sequenced the block.

### Likelihood Explanation

**Low.** Exploitation requires a validator that is legitimately selected as proposer in a given round (normal Tendermint rotation). No external attacker can trigger this; it requires a malicious insider validator. However, the validator set is not assumed to be fully trusted, and the entire purpose of the `is_proposal_init_valid` checks is to constrain what a proposer can assert. The `builder` field is the only economically significant `ProposalInit` field that escapes all validation.

### Recommendation

Add a check inside `is_proposal_init_valid` that `init_proposed.builder` equals the locally-known expected sequencer address (analogous to how `proposer` is checked against the committee in `single_height_consensus.rs`): [8](#0-7) 

The expected `builder` address should be stored in `ProposalInitValidation` (alongside `height`, `l1_da_mode`, etc.) and populated from the node's own configuration before validation begins, so a proposer cannot substitute an arbitrary sequencer address.

### Proof of Concept

1. A validator node is legitimately selected as proposer for height H, round R.
2. Instead of setting `builder: self.builder_address`, it sets `builder: attacker_controlled_address`.
3. It sends `ProposalPart::Init(init)` with this modified `builder` over the p2p network.
4. Every peer validator calls `validate_proposal` → `is_proposal_init_valid`. The function checks `height`, `l1_da_mode`, `l2_gas_price_fri`, all L1 gas prices, `starknet_version`, `version_constant_commitment`, and `fee_proposal_fri`. It does **not** check `builder`. Validation passes.
5. `initiate_validation` calls `convert_to_sn_api_block_info(init)`, producing `BlockInfo { sequencer_address: attacker_controlled_address, ... }`.
6. The batcher executes all transactions; every fee is credited to `attacker_controlled_address`.
7. `PartialBlockHashComponents { sequencer: attacker_controlled_address, ... }` is hashed into `partial_block_hash`, then into `proposal_commitment`.
8. Both proposer and validators compute the same `proposal_commitment` (since both used the same `builder`). The `ProposalFinMismatch` guard at line 244 does not fire. [9](#0-8) 

9. Consensus reaches a decision; the block is committed with `sequencer_address = attacker_controlled_address` and all fees redirected.
