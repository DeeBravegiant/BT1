Audit Report

## Title
Missing Zero-Amount Guard on Fee Transfer in `MerkleDistributor.claim()` Freezes Unclaimed Yield for Revert-on-Zero Tokens - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` without checking whether `fee > 0`. When `feeInBPS == 0` (a valid and explicitly permitted configuration), `fee` evaluates to zero, and tokens that revert on zero-value transfers will cause every `claim()` invocation to revert. Since `claim()` is the sole mechanism for users to receive their allocated tokens, this permanently freezes all unclaimed yield under that configuration.

## Finding Description
In `MerkleDistributor.claim()` at lines 138–144:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);   // ← no zero-check
```

When `feeInBPS == 0`, `fee = 0`, and the unconditional `safeTransfer(protocolTreasury, 0)` is executed. Tokens that revert on zero-value transfers (a documented class of non-standard ERC20s) will cause the entire call to revert.

`feeInBPS` is permitted to be zero: `initialize()` only rejects values **above** `MAX_FEE_IN_BPS` (1000), and `setFeeInBPS(0)` passes the same guard without restriction.

The sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` both correctly guard this transfer:

```solidity
if (fee > 0) {
    kernel.safeTransfer(protocolTreasury, fee);
}
```

`MerkleDistributor` is missing this guard entirely. The `token` address is also configurable via `setToken()` with no restriction on which ERC20 is used, making a revert-on-zero token a realistic deployment choice.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

All users' `claim()` calls revert when `feeInBPS == 0` and the distributed token reverts on zero-value transfers. Because `claim()` is the only path to receive allocated tokens, unclaimed yield is frozen for all claimants. If `feeInBPS` remains zero, the freeze is permanent. This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation
- `feeInBPS == 0` is a valid operational configuration (no fee taken), reachable at initialization or via `setFeeInBPS(0)`.
- The `token` is freely configurable via `setToken()` with no whitelist; a revert-on-zero ERC20 (e.g., LEND, BNB, and others documented in the weird-erc20 repository) is a realistic choice.
- No attacker action is required; the freeze is automatic for all claimants once both conditions hold.
- Any unprivileged user calling `claim()` with a valid merkle proof triggers the revert.

## Recommendation
Add a zero-amount guard before the fee transfer, consistent with the pattern in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy `MerkleDistributor` with a token that reverts on zero-value transfers and `feeInBPS = 0` (or call `setFeeInBPS(0)` post-deployment).
2. Call `setMerkleRoot(validRoot)` and fund the contract with the revert-on-zero token.
3. Any user calls `claim(index, account, cumulativeAmount, merkleProof)` with a valid proof.
4. `fee = (claimableAmount * 0) / 10_000 = 0`.
5. `IERC20(token).safeTransfer(protocolTreasury, 0)` reverts because the token disallows zero-value transfers.
6. The entire `claim()` call reverts; the user's merkle allocation is never transferred; unclaimed yield is frozen.

**Foundry test plan:**
- Deploy a mock ERC20 that reverts on `transfer(to, 0)`.
- Deploy `MerkleDistributor`, initialize with `feeInBPS = 0` and the mock token.
- Build a valid merkle tree with one leaf, set the root, fund the contract.
- Call `claim()` with the valid proof; assert the call reverts.
- Add `if (fee > 0)` guard; assert the call succeeds and the user receives `claimableAmount`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
