### Title
Byzantine Participant Inflates `maximum_height` via Unchecked `IndexerHeightMessage`, Causing Premature `RequestTimedOut` Drops — (`crates/node/src/network/indexer_heights.rs`, `crates/node/src/requests/queue.rs`)

---

### Summary

A single Byzantine registered participant, while bidirectionally connected, can send an `IndexerHeightMessage` with an arbitrarily large `height` value. Because `IndexerHeightTracker::set_height` accepts any value greater than the current stored value with no upper-bound validation, and because `eligible_leaders_and_maximum_height` derives `maximum_height` directly from alive participants' stored heights, the resulting `cutoff_block` becomes astronomically large. Every queued request at a realistic block height is then immediately dropped as `DropReason::RequestTimedOut`, permanently halting request processing until a node restart — which the attacker can immediately repeat.

---

### Finding Description

**Step 1 — Unchecked height storage.**

`IndexerHeightTracker::set_height` stores any value larger than the current one with no validation: [1](#0-0) 

**Step 2 — Message dispatch with no height bounds check.**

When a `PeerMessage::IndexerHeight` arrives, the transport-authenticated `from` field is used directly, and the raw `height` is forwarded to `set_height`: [2](#0-1) 

**Step 3 — Byzantine participant qualifies as "alive".**

`all_alive_participant_ids` includes a peer if `my_height <= peer_height + MAX_INDEXER_HEIGHT_DIFF` AND the peer is bidirectionally connected. If the Byzantine participant sends `height = H` where `H + MAX_INDEXER_HEIGHT_DIFF` does not overflow `u64` (e.g., `H = u64::MAX / 2`), then `my_height <= H + MAX_INDEXER_HEIGHT_DIFF` is trivially true for any realistic local height, so the Byzantine participant is included in `alive_participants`: [3](#0-2) 

**Step 4 — Inflated `maximum_height` propagates to `cutoff_block`.**

`eligible_leaders_and_maximum_height` takes the max height over all alive participants. With the Byzantine participant included, `maximum_height = H`: [4](#0-3) 

**Step 5 — `cutoff_block` becomes astronomically large.**

`get_requests_to_attempt` computes:

```
cutoff_block = (maximum_height.saturating_sub(REQUEST_EXPIRATION_BLOCKS) + 1)
```

With `maximum_height ≈ u64::MAX / 2` and `REQUEST_EXPIRATION_BLOCKS = 200`, `cutoff_block ≈ u64::MAX / 2`. Every request at a realistic block height (e.g., 1,000,000) satisfies `cutoff_block > block_height`: [5](#0-4) 

**Step 6 — All requests are dropped.**

`QueuedRequest::process` checks `is_older_than(cutoff_block)` before any leader or response logic: [6](#0-5) 

**Step 7 — Permanent until restart; trivially repeatable.**

`set_height` only ever increases the stored value. After a node restart (which resets heights to 0), the Byzantine participant simply re-sends the large height message, reinstating the attack. [7](#0-6) 

**Side effect — Legitimate participants lose eligible-leader status.**

The `STALE_PARTICIPANT_THRESHOLD` filter in `eligible_leaders_and_maximum_height` also eliminates all honest participants from `eligible_leaders` (since `realistic_height + 10 << H`), so even if a request somehow survived the cutoff check, no honest node would be selected as leader: [8](#0-7) 

---

### Impact Explanation

All pending signature, CKD, and `VerifyForeignTx` requests are silently dropped as `RequestTimedOut` even though they have not expired on-chain. The MPC network stops processing any user requests. This breaks the core request-lifecycle invariant that expiry is determined by on-chain block height, not by a peer-reported height. The impact matches **Medium**: request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.

---

### Likelihood Explanation

Any single registered MPC participant who is bidirectionally connected can execute this attack. No threshold collusion is required. The message is unauthenticated at the application layer (height value is not bounded or cross-checked). The attack is trivially repeatable after restarts, making it a persistent denial of service for request processing.

---

### Recommendation

1. **Bound accepted heights.** In `set_height`, reject any height that exceeds the local node's current indexer height by more than a configurable maximum (e.g., `MAX_INDEXER_HEIGHT_DIFF`). A peer claiming to be millions of blocks ahead of the local node is implausible and should be ignored or flagged.

2. **Cap `maximum_height` in `eligible_leaders_and_maximum_height`.** Derive `maximum_height` only from participants whose reported height is within a plausible range of the local node's own height, not from the raw maximum across all alive participants.

3. **Use median instead of max.** Computing `maximum_height` as the median (or a trimmed max) over alive participants' heights would make a single Byzantine participant unable to dominate the value.

### Proof of Concept

```rust
// Pseudocode integration test sketch
let participants = vec![p0, p1, p2]; // 3-of-3 network
let mut tracker = IndexerHeightTracker::new(&participants);

// Honest nodes at realistic height
tracker.set_height(p0, 1_000_000);
tracker.set_height(p1, 1_000_001);

// Byzantine participant sends astronomically large height
// Chosen so that p2_height + MAX_INDEXER_HEIGHT_DIFF does not overflow u64
let byzantine_height: u64 = u64::MAX / 2;
tracker.set_height(p2, byzantine_height); // stored permanently

// Simulate alive_participants including p2 (bidirectionally connected)
// maximum_height = u64::MAX / 2
// cutoff_block = (u64::MAX / 2 - 200 + 1) >> 1_000_000

// Queue a request at block 1_000_000
let mut pending = PendingRequests::new(...);
pending.notify_new_block(RequestsUpdate { block_height: 1_000_000, ... });

// Call get_requests_to_attempt — all requests dropped as RequestTimedOut
let attempts = pending.get_requests_to_attempt();
assert!(attempts.is_empty()); // no requests attempted
// Verify via metrics: MPC_CLUSTER_FAILED_SIGNATURES_COUNT["timeout"] incremented
```
