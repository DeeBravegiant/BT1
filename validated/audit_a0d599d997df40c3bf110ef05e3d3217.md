### Title
Missing Output Safeguard in `respond_ckd` for Legacy `AppPublicKey` Variant Allows Single Byzantine Participant to Corrupt Confidential Key Derivation Output - (File: crates/contract/src/lib.rs)

---

### Summary

The `respond_ckd` function in the MPC contract performs cryptographic output validation **only** for the `AppPublicKeyPV` (publicly verifiable) variant of CKD requests. For the legacy `AppPublicKey` variant, the match arm is an explicit no-op. A single attested-but-Byzantine participant can call `respond_ckd` with arbitrary `big_y` and `big_c` values for any pending `AppPublicKey` CKD request, and the contract will accept and resolve all queued yields with the garbage response. The user then derives a wrong confidential key, which can lead to permanent loss of funds or secrets protected by that key.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function contains the following match block:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For the `AppPublicKeyPV` variant, `ckd_output_check` verifies the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` on-chain, ensuring the response is the correct threshold output. [2](#0-1) 

For the `AppPublicKey` (legacy) variant, **no equivalent check exists**. The contract immediately proceeds to resolve all pending yields with whatever `CKDResponse` the caller supplied: [3](#0-2) 

The `respond` function for signatures avoids this problem by cryptographically verifying the signature against the derived public key before resolving yields: [4](#0-3) 

No analogous safeguard exists for `respond_ckd` when `AppPublicKey` is used.

The `AppPublicKey` variant is the documented legacy/privately-verifiable format and remains in active production use: [5](#0-4) 

The attacker entry point is `respond_ckd`, which requires only that the caller be an attested participant — a single node, not a threshold: [6](#0-5) 

---

### Impact Explanation

**Impact: Critical — confidential key derivation output produced without the required threshold-participant authorization.**

The CKD protocol is designed so that the correct output `(big_y, big_c)` can only be computed by a threshold of participants cooperating. The contract's role is to enforce that only a cryptographically valid threshold output is returned to the user. For `AppPublicKeyPV` requests this invariant is enforced on-chain. For `AppPublicKey` requests it is not enforced at all.

A single Byzantine attested participant can:
1. Observe a pending `AppPublicKey` CKD request on-chain.
2. Race to call `respond_ckd` with arbitrary `big_y` and `big_c` before the honest nodes submit the correct response.
3. The contract accepts the garbage response and resolves **all** queued yields for that request key with it (fan-out drain).

The user receives `big_c_garbage - a·big_y_garbage`, which is a random curve point unrelated to `msk·H(pk||app_id)`. Any asset, private key, or secret protected by the derived confidential key becomes permanently inaccessible — a direct, irreversible loss of funds or key material.

---

### Likelihood Explanation

**Likelihood: Low-Medium.**

- Requires one attested participant to act Byzantine. TEE attestation raises the bar, but does not eliminate the risk: a node operator whose TEE is compromised, whose software has a bug, or who deliberately acts maliciously satisfies this condition.
- The attack is a race condition: the Byzantine node must submit before the honest nodes complete their threshold computation and call `respond_ckd`. Honest nodes must perform multi-round MPC before responding; the attacker needs only to craft and submit a transaction, giving them a structural timing advantage.
- The `AppPublicKey` variant is still the default legacy path used by existing integrations (e.g., the `ckd-example-cli`), so pending requests of this type will appear regularly on-chain. [7](#0-6) 

---

### Recommendation

1. **Deprecate and gate the `AppPublicKey` variant.** Because the contract cannot verify the output without the user's private key, the only safe path is to require `AppPublicKeyPV` for all new requests and reject `AppPublicKey` in `respond_ckd` (or at submission time in `request_app_private_key`).

2. **If `AppPublicKey` must remain supported**, add a commitment scheme: at request time, store a hash commitment of the expected output (e.g., `H(msk·H(pk||app_id))` contributed by each participant during the MPC round), and verify the submitted `big_c` against it in `respond_ckd`.

3. **Align `respond_ckd` with `respond`**: just as `respond` cryptographically verifies the signature before resolving yields, `respond_ckd` must cryptographically verify the CKD output for every variant before resolving yields.

---

### Proof of Concept

```
1. Alice submits:
   request_app_private_key({
     derivation_path: "mykey",
     app_public_key: "bls12381g1:<alice_pk>",   // AppPublicKey variant
     domain_id: 2
   })
   → pending CKDRequest stored on-chain

2. Byzantine attested participant Eve observes the pending request.

3. Eve immediately calls:
   respond_ckd(
     request = <alice's CKDRequest>,
     response = CKDResponse {
       big_y: bls12381g1:<random_point>,
       big_c: bls12381g1:<random_point>,
     }
   )
   → Contract enters respond_ckd:
      - assert_caller_is_attested_participant_and_protocol_active() → passes (Eve is attested)
      - match AppPublicKey(_) => {}  ← no check, falls through
      - resolve_yields_for(...) → resolves Alice's yield with garbage response

4. Alice's transaction completes with:
   { big_y: <random>, big_c: <random> }

5. Alice computes: big_c - a·big_y = <random point> ≠ msk·H(pk||app_id)

6. Alice uses the wrong key to encrypt her Bitcoin private key → funds permanently lost.
```

/// Check that `e(big_c, g2) = e(big_y, app_pk2) . e(hash_point, public_key)`.
///
/// Point validation is fully delegated to the host, as in
/// [`app_public_key_check`].
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

**File:** crates/contract/README.md (L118-121)
```markdown
- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
```
