### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in Consensus L1Handler Conversion Bypasses Fee Adequacy Check and Corrupts Validator Storage - (`File: crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

`convert_consensus_l1_handler_to_internal_l1_handler` unconditionally injects `Fee(1)` as `paid_fee_on_l1` for every L1Handler transaction received through the consensus path. The `paid_fee_on_l1` field is absent from the `L1HandlerV0` protobuf wire format, so the real value paid on L1 is never transmitted to validators. The blockifier's fee adequacy check only rejects `Fee(0)`, meaning `Fee(1)` always passes. Validator nodes therefore store and serve the wrong `paid_fee_on_l1` value for every L1Handler transaction, and a malicious proposer can force validators to accept an L1Handler transaction that the proposer's own execution would reject.

### Finding Description

**Step 1 – Wire format drops `paid_fee_on_l1`.**

The consensus protobuf message for L1Handler transactions is `L1HandlerV0`, which carries only `nonce`, `address`, `entry_point_selector`, and `calldata`. There is no `paid_fee_on_l1` field. [1](#0-0) 

**Step 2 – Validator conversion hardcodes `Fee(1)`.**

When a validator receives a `ConsensusTransaction::L1Handler` from the proposer, `convert_consensus_tx_to_internal_consensus_tx` calls `convert_consensus_l1_handler_to_internal_l1_handler`, which calls `L1HandlerTransaction::create` with a hardcoded `Fee(1)` and an explicit TODO acknowledging the placeholder: [2](#0-1) 

**Step 3 – Blockifier fee check only rejects `Fee(0)`.**

The only fee adequacy check for L1Handler transactions is:

```rust
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(...InsufficientFee...));
}
```

Because `Fee(1) != Fee(0)`, the hardcoded value always passes this check, regardless of what was actually paid on L1. [3](#0-2) 

**Step 4 – Proposer has the real value; validator does not.**

The proposer obtains L1Handler transactions from the L1 events provider with the real `paid_fee_on_l1` (the ETH value sent to the Starknet core contract on L1). The validator receives only the `L1HandlerV0` protobuf and reconstructs the transaction with `Fee(1)`. The two sides therefore hold structurally different `L1HandlerTransaction` objects for the same transaction.

**Step 5 – Wrong value committed to validator storage and served by RPC.**

After consensus, every node commits the block. Validator nodes commit `paid_fee_on_l1 = Fee(1)` for every L1Handler transaction. The `CentralL1HandlerTransaction` serialization (used for the Aerospike/Cende blob) explicitly includes `paid_fee_on_l1`: [4](#0-3) 

RPC endpoints that expose `paid_fee_on_l1` (e.g., `starknet_getTransactionByHash`) on validator nodes will return `0x1` for every L1Handler transaction, regardless of the actual fee paid on L1.

**Step 6 – Malicious-proposer divergence path.**

If a malicious proposer crafts a proposal containing an L1Handler transaction whose real `paid_fee_on_l1` is `Fee(0)` (e.g., by replaying a message that was never properly funded on L1), the proposer's own blockifier execution returns `Err(InsufficientFee)` and the transaction is not included in the proposer's committed block. The validator, however, executes the same transaction with `Fee(1)` and succeeds, producing a different `ProposalCommitment`. This causes a `ProposalFinMismatch` and consensus stall rather than silent acceptance, but the root cause is the same missing-value forwarding defect.

### Impact Explanation

- **Wrong authoritative RPC view**: Every validator node permanently stores and serves `paid_fee_on_l1 = Fee(1)` for all L1Handler transactions. Any client querying a validator's RPC for L1Handler transaction details receives a fabricated fee value.
- **Fee adequacy check bypass**: The only guard against zero-fee L1Handler execution (`paid_fee == Fee(0)`) is defeated for any transaction arriving through the consensus path, because the hardcoded `Fee(1)` always satisfies it.
- **Proposer/validator execution divergence**: A malicious proposer can construct a scenario where the proposer's execution rejects an L1Handler transaction but the validator's execution accepts it, causing a `ProposalFinMismatch` and consensus disruption.

### Likelihood Explanation

The wrong `paid_fee_on_l1` value is injected on every L1Handler transaction processed by every validator node. The condition is unconditional and requires no special attacker input — it fires on every normal block containing an L1Handler transaction. The malicious-proposer divergence path requires a proposer willing to include a `Fee(0)` L1Handler transaction, which is an unusual but reachable condition.

### Recommendation

1. Transmit `paid_fee_on_l1` through the consensus wire format. Add a `paid_fee_on_l1` field to the `L1HandlerV0` protobuf message (or a new `L1HandlerV1` message) and populate it from the L1 events provider before the proposer streams the transaction.

2. Until the wire format is updated, at minimum validate that the received `paid_fee_on_l1` is non-zero before accepting the transaction in the validator path, and do not commit `Fee(1)` to storage.

3. Remove the hardcoded placeholder and resolve the `TODO(Gilad)` comment.

### Proof of Concept

```
1. Proposer receives L1Handler tx from L1 events provider:
   paid_fee_on_l1 = Fee(50_000_000_000)  // real ETH value from L1

2. Proposer serializes tx to L1HandlerV0 protobuf:
   { nonce, address, entry_point_selector, calldata }
   // paid_fee_on_l1 is silently dropped — no field in the proto

3. Validator deserializes L1HandlerV0 and calls:
   convert_consensus_l1_handler_to_internal_l1_handler(tx)
   → L1HandlerTransaction::create(tx, &chain_id, Fee(1))
   // paid_fee_on_l1 = Fee(1), not Fee(50_000_000_000)

4. Validator blockifier executes:
   if paid_fee == Fee(0) { return Err(...) }  // Fee(1) != Fee(0), passes

5. Validator commits block with paid_fee_on_l1 = Fee(1) in storage.

6. RPC query starknet_getTransactionByHash on validator node:
   → returns paid_fee_on_l1: "0x1"   // wrong; real value was 50 gwei
