### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Enables Attestation Frontrunning Without Requester Binding — (File: `crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` stores pending requests under a key that excludes the caller's identity, and MPC nodes sign the resulting attestation with the **root key** (zero tweak). Any unprivileged NEAR account can submit the identical request and receive the same signed attestation as the legitimate requester. Bridge protocols that rely on this attestation to authorize fund releases are exposed to a frontrunning attack in which an adversary claims the attestation before the legitimate depositor can use it.

### Finding Description

**`sign()` vs `verify_foreign_transaction()` — the asymmetry**

`sign()` derives a caller-specific tweak from `(predecessor_id, path)` and embeds it in `SignatureRequest`, so the resulting signature is cryptographically bound to the caller's derived key:

```rust
// crates/contract/src/lib.rs:379-384
let request = SignatureRequest::new(
    request.domain_id,
    request.payload,
    &predecessor,      // ← caller identity baked into tweak
    &request.path,
);
```

`verify_foreign_transaction()` does the opposite. The conversion function strips the caller entirely:

```rust
// crates/contract/src/dto_mapping.rs:840-848
pub fn args_into_verify_foreign_tx_request(
    args: dtos::VerifyForeignTransactionRequestArgs,
) -> dtos::VerifyForeignTransactionRequest {
    dtos::VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
        // predecessor / tweak: absent
    }
}
```

The resulting `VerifyForeignTransactionRequest` is the map key in `pending_verify_foreign_tx_requests`. Because the caller is absent, Alice's and Bob's submissions of the same `(chain, tx_id, extractors, domain_id)` tuple hash to the **same** map entry and are explicitly fan-out queued together — confirmed by the unit test comment:

> "both yields are queued under the single (caller-agnostic) request key" [1](#0-0) 

**Root key used for signing (zero tweak)**

On the node side, `build_signature_request` hard-codes the tweak to all-zeros:

```rust
// crates/node/src/providers/verify_foreign_tx/sign.rs:39-47
Ok(SignatureRequest {
    ...
    tweak: Tweak::new([0u8; 32]),   // ← root key, no derivation
    ...
})
``` [2](#0-1) 

`respond_verify_foreign_tx` verifies the signature against the **root public key** (no tweak applied), confirmed by the inline comment in the test:

> "simulate signature with the root key (no tweak for foreign tx)" [3](#0-2) [4](#0-3) 

**Fan-out delivers the identical response to every queued caller**

`resolve_yields_for` resumes every queued yield with the same `response_bytes`:

```rust

            env::promise_yield_resume(&data_id, response_bytes.clone());

        .count();

}

### Impact Explanation

The `VerifyForeignTransactionResponse` — a root-key ECDSA signature over `SHA-256(borsh(ForeignTxSignPayload))` — is the on-chain proof that a foreign-chain transaction occurred and was verified by the MPC network. Because the response is not bound to any requester, an adversary who obtains it has the same cryptographic proof as the legitimate depositor.

A bridge contract that gates fund release on this attestation (the primary stated use-case of `verify_foreign_transaction`) cannot distinguish Alice's legitimate claim from Bob's frontrun copy. Bob can drain Alice's deposit by:

1. Observing Alice's pending `verify_foreign_transaction` on-chain.
2. Submitting the identical request (same `tx_id`, chain, extractors, `domain_id`).
3. Receiving the same `VerifyForeignTransactionResponse` via the fan-out.
4. Presenting the attestation to the bridge contract before Alice does.

This maps to the allowed **High** impact: *"participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."*

### Likelihood Explanation

- The attack requires no special privilege — any NEAR account can call `verify_foreign_transaction`.
- Pending requests are publicly visible on-chain; an attacker needs only to watch the contract state.
- Bridge protocols are the primary consumer of this API; the scenario is not hypothetical.
- The fan-out is bounded at 128 entries, so the attacker's submission is guaranteed to succeed as long as the queue is not already full.

### Recommendation

Bind the attestation to the requester by including `predecessor_id` in the request key and in the signed payload, mirroring the pattern used by `sign()`:

1. Add a `tweak` (derived from `predecessor_id` and an optional path) to `VerifyForeignTransactionRequest` so that different callers produce different map keys.
2. Pass the tweak to `build_signature_request` instead of `Tweak::new([0u8; 32])`, so the MPC nodes sign with a caller-specific derived key.
3. Update `respond_verify_foreign_tx` to verify the signature against the derived public key (as `respond()` already does for `sign()`).

Alternatively, document explicitly that the attestation is caller-agnostic and that bridge protocols **must** independently verify the depositor's identity from the foreign-chain transaction data itself — and provide a `try/catch`-style pattern for consuming the attestation safely, analogous to the recommendation in the external report.

### Proof of Concept

```
1. Alice deposits 100 ETH on Ethereum; tx_id = 0xABCD...
2. Alice calls verify_foreign_transaction({chain: Ethereum, tx_id: 0xABCD, ...})
   → queued under key K = VerifyForeignTransactionRequest{domain_id, request, payload_version}
3. Bob observes K in pending_verify_foreign_tx_requests (public chain state).
4. Bob calls verify_foreign_transaction({chain: Ethereum, tx_id: 0xABCD, ...})
   → queued under the same key K (fan-out, no caller check).
5. MPC nodes call respond_verify_foreign_tx(K, response)
   where response.signature = ECDSA(root_key, SHA-256(borsh(ForeignTxSignPayload{tx_id=0xABCD, values=[...]})))
6. resolve_yields_for(K, response_bytes) resumes BOTH Alice's and Bob's yields
   with the identical VerifyForeignTransactionResponse.
7. Bob presents the attestation to the bridge contract first,
   claiming Alice's 100 ETH deposit. Alice's subsequent claim is rejected as a replay.
``` [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9)

    pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
        log!(
            "verify_foreign_transaction: predecessor={:?}, request={:?}",
            env::predecessor_account_id(),
            request
        );

        self.check_request_preconditions(
            request.domain_id,
            DomainPurpose::ForeignTx,
            Gas::from_tgas(self.config.sign_call_gas_attachment_requirement_tera_gas),
            MINIMUM_SIGN_REQUEST_DEPOSIT,
        );

        let requested_chain = request.request.chain();
        let supported_chains = self.get_supported_foreign_chains();
        if !supported_chains.contains(&requested_chain) {
            env::panic_str(
                &InvalidParameters::ForeignChainNotSupported {
                    requested: requested_chain,
                }
                .to_string(),
            );
        }

        let callback_gas = Gas::from_tgas(
            self.config
                .return_signature_and_clean_state_on_success_call_tera_gas,
        );

        let request = args_into_verify_foreign_tx_request(request);
        let callback_args = serde_json::to_vec(&(&request,)).unwrap();
        self.enqueue_yield_request(
            method_names::RETURN_VERIFY_FOREIGN_TX_AND_CLEAN_STATE_ON_SUCCESS,
            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),
        );
    }

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

fn build_signature_request(
    request: &VerifyForeignTxRequest,
    foreign_tx_payload: &dtos::ForeignTxSignPayload,
) -> anyhow::Result<SignatureRequest> {
    let payload_hash: [u8; ECDSA_PAYLOAD_SIZE_BYTES] =
        foreign_tx_payload.compute_msg_hash()?.into();
    let payload_bytes: BoundedVec<u8, ECDSA_PAYLOAD_SIZE_BYTES, ECDSA_PAYLOAD_SIZE_BYTES> =
        payload_hash.into();

    Ok(SignatureRequest {
        id: request.id,
        receipt_id: request.receipt_id,
        payload: Payload::Ecdsa(payload_bytes),
        tweak: Tweak::new([0u8; 32]),
        entropy: request.entropy,
        timestamp_nanosec: request.timestamp_nanosec,
        domain: request.domain_id,
    })
}

```rust
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
