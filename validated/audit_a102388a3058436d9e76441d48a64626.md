Audit Report

## Title
Unvalidated Arbitrary Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain Approved User Tokens via Callback — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address in all four `addLiquidity*` entry points without verifying it against any factory registry. A malicious pool contract can exploit the `metricOmmModifyLiquidityCallback` settlement path to pull arbitrary ERC-20 tokens from any victim who has approved the adder, up to the caller-provided max caps, by returning attacker-controlled token addresses from `getImmutables()`. The contract's own NatSpec explicitly acknowledges this missing guard.

## Finding Description

The constructor takes only a `weth` address — there is no `FACTORY` immutable and no `isPool` check anywhere in the contract. [1](#0-0) 

The contract's own documentation explicitly acknowledges the missing guard: [2](#0-1) 

The internal `_addLiquidity` flow stores the attacker-supplied pool as the expected callback caller in transient storage, then calls into it: [3](#0-2) 

Inside `metricOmmModifyLiquidityCallback`, all guards pass for a malicious pool: `kind == KIND_PAY` (attacker controls calldata), `msg.sender == expectedPool` (malicious pool is the stored expected pool), and `amount0Delta <= max0 && amount1Delta <= max1` (attacker sets amounts ≤ caps): [4](#0-3) 

The callback then fetches token addresses from `msg.sender` (the malicious pool itself), which can return any arbitrary addresses: [5](#0-4) 

`pay()` then executes `safeTransferFrom(payer, recipient, value)` — pulling attacker-chosen tokens from the victim's wallet directly to the malicious pool: [6](#0-5) 

By contrast, the sibling `MetricOmmSwapRouterBase` enforces factory validation on every pool address before any interaction: [7](#0-6) 

## Impact Explanation

Any user who has approved `MetricOmmPoolLiquidityAdder` to spend their tokens and is tricked into calling any `addLiquidity*` function with a malicious pool address loses up to `maxAmountToken0` of any ERC-20 token and `maxAmountToken1` of any other ERC-20 token. The malicious pool controls which tokens are pulled by returning arbitrary addresses from `getImmutables()`. This is a direct, unrecoverable loss of user principal — Critical/High severity under Sherlock thresholds.

## Likelihood Explanation

`MetricOmmPoolLiquidityAdder` is a shared periphery contract that users must pre-approve. A phishing front-end, compromised UI, or social-engineering attack substituting a malicious pool address for a legitimate one is a realistic trigger. The attack requires no privileged role, no special token behavior, no flash loan, and no technical sophistication from the victim — pool addresses are opaque 20-byte values most users do not verify on-chain.

## Recommendation

Add a factory reference to `MetricOmmPoolLiquidityAdder` (mirroring `MetricOmmSwapRouterBase`) and validate every caller-supplied pool address before storing it in the transient pay context:

```solidity
IMetricOmmPoolFactory internal immutable FACTORY;

constructor(address weth, address factory) PeripheryPayments(weth) {
    FACTORY = IMetricOmmPoolFactory(factory);
}

function _requireFactoryPool(address pool) internal view {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
}
```

Call `_requireFactoryPool(pool)` at the top of `_addLiquidity` and inside both `addLiquidityWeighted` overloads before the probe call.

## Proof of Concept

1. Victim approves `MetricOmmPoolLiquidityAdder` for USDC and WETH (`type(uint256).max`).
2. Attacker deploys `MaliciousPool(USDC, WETH, adder)` whose `getImmutables()` returns `token0=USDC, token1=WETH` and whose `addLiquidity` immediately calls back `metricOmmModifyLiquidityCallback(1_000e6, 1_000e18, abi.encode(uint8(1)))`.
3. Victim is tricked into calling:
   ```solidity
   adder.addLiquidityExactShares(
       address(maliciousPool), victim, 0, deltas,
       1_000e6,   // maxAmountToken0
       1_000e18,  // maxAmountToken1
       ""
   );
   ```
4. `_addLiquidity` calls `_setPayContext(maliciousPool, victim, 1_000e6, 1_000e18)`, then calls `maliciousPool.addLiquidity(...)`.
5. `MaliciousPool.addLiquidity` immediately re-enters `metricOmmModifyLiquidityCallback(1_000e6, 1_000e18, abi.encode(KIND_PAY))`.
6. All guards pass: `kind==KIND_PAY` ✓, `msg.sender==maliciousPool==expectedPool` ✓, amounts ≤ caps ✓.
7. `getImmutables()` returns `token0=USDC, token1=WETH`; `pay(USDC, victim, maliciousPool, 1_000e6)` and `pay(WETH, victim, maliciousPool, 1_000e18)` execute.
8. Victim loses 1,000 USDC + 1,000 WETH; malicious pool receives them.
