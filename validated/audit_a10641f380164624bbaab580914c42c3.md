### Title
Integer Division Truncation in `compute_royalty_amount` Silently Zeroes NFT Royalty Payments on Small Offers — (`File: chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
`compute_royalty_amount` uses sequential integer floor-division that can produce 0 for small-but-nonzero offered amounts. When it does, `make_nft1_offer` silently skips the royalty coin entirely, so the NFT creator receives no royalty for that trade. An unprivileged buyer can deliberately craft offers below the truncation threshold to avoid paying royalties on NFT transfers.

### Finding Description

`compute_royalty_amount` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
```

where `MAX_ROYALTY_BASIS_POINTS = 10000`. [1](#0-0) 

For any `offered_amount` where `abs(offered_amount) // royalty_split * percentage < 10000`, the result is 0. The project's own test suite explicitly documents and accepts this:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

When `compute_royalty_amount` returns 0, `make_nft1_offer` has two guards that both silently skip the royalty payment rather than rejecting the offer:

1. The `trade_prices_list` filter excludes any price entry whose royalty would be 0:

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [3](#0-2) 

2. The royalty coin spend is skipped entirely when the total royalty payment is 0:

```python
if sum(p.amount for _, p in payments) == 0:
    continue
``` [4](#0-3) 

Because the CLVM NFT transfer program enforces royalties only against the `trade_prices_list` passed in the solution, an empty list means no on-chain royalty enforcement occurs. The offer settles successfully with zero royalty paid.

The same truncation exists in the `royalty_calculation` static method used for display and RPC estimation:

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
``` [5](#0-4) 

### Impact Explanation

An unprivileged buyer can create an offer for an NFT with a fungible amount small enough that `abs(amount) * percentage < 10000`. The offer is accepted on-chain with no royalty coin created, and the NFT creator receives 0 royalty instead of the expected amount. This is an unauthorized accounting change affecting XCH/CAT/NFT offer settlement.

Truncation threshold (for a 1% royalty, `percentage=100`): any `offered_amount < 100 mojos`. For CATs (3 decimal places), `100 CAT mojos = 0.1 CAT`, which is economically meaningful in high-value CAT markets. For XCH (12 decimal places), the threshold is negligible in isolation but can be exploited by splitting a larger purchase into many sub-threshold offers.

### Likelihood Explanation

Likelihood is **low-to-moderate**. For XCH the threshold is so small that transaction fees exceed any royalty savings. For CATs with fewer decimals the threshold is more realistic. The behavior is explicitly tested and accepted by the developers, suggesting it is a known design trade-off rather than an undiscovered bug. Exploitation requires deliberate offer crafting below the threshold.

### Recommendation

1. **Reject offers where royalty truncates to zero**: in `make_nft1_offer`, if `compute_royalty_amount` returns 0 for a non-zero percentage, raise an error requiring a minimum offer amount rather than silently skipping the royalty.
2. **Use ceiling division** for royalty computation: `math.ceil(abs(offered_amount) * percentage / MAX_ROYALTY_BASIS_POINTS)` ensures at least 1 mojo is paid whenever `percentage > 0`.
3. **Enforce a minimum offer amount** at the wallet layer: reject offers where `abs(amount) * percentage < MAX_ROYALTY_BASIS_POINTS` when the NFT has a non-zero royalty.

from chia.wallet.nft_wallet.nft_wallet import compute_royalty_amount, MAX_ROYALTY_BASIS_POINTS

# which enforces no royalty on-chain.
print(f"Royalty for {offered_amount} mojos at {percentage} bps: {royalty} mojos")

MAX_ROYALTY_BASIS_POINTS = 10000


def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
