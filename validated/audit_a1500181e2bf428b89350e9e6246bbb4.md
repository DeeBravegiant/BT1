Audit Report

## Title
ETH Withdrawal Permanently Frozen When Recipient Contract Cannot Receive ETH - (`contracts/LRTWithdrawalManager.sol`)

## Summary
The withdrawal lifecycle splits rsETH burning (`unlockQueue`) from ETH delivery (`completeWithdrawal`) across two separate, independently-finalized transactions. If the withdrawing address is a smart contract that cannot receive ETH, `_transferAsset` reverts on every call to `completeWithdrawal` and `completeWithdrawalForUser`, while the rsETH burned in the prior `unlockQueue` transaction is irrecoverable. The corresponding ETH is permanently locked in `LRTWithdrawalManager` with no admin escape hatch.

## Finding Description

**Step 1 — rsETH burned in `unlockQueue` (separate, finalized transaction):** [1](#0-0) 

`burnFrom` is called on the contract's rsETH balance. Once this transaction is mined, the burn is irreversible regardless of what happens in future transactions.

**Step 2 — `completeWithdrawal` / `completeWithdrawalForUser` both route through `_processWithdrawalCompletion`:** [2](#0-1) [3](#0-2) 

**Step 3 — `_processWithdrawalCompletion` ends with a push-payment ETH transfer:** [4](#0-3) 

**Step 4 — `_transferAsset` uses a low-level call that reverts on failure:** [5](#0-4) 

If `payable(to).call{value: amount}("")` returns `false` (recipient has no `receive()`/`fallback()`, or one that reverts), the entire `_processWithdrawalCompletion` transaction reverts. All state mutations within that call — the `popFront()` of the nonce, the `delete withdrawalRequests[requestId]`, and the `unlockedWithdrawalsCount[asset]--` — are rolled back. The withdrawal request remains in the queue, permanently unlocked but uncollectable.

**Step 5 — `sweepRemainingAssets` is permanently blocked:** [6](#0-5) 

Because `unlockedWithdrawalsCount[asset]` is never decremented (the decrement at L717 always reverts with the transfer), `hasUnlockedWithdrawals(asset)` always returns `true`, and `sweepRemainingAssets` always reverts with `PendingWithdrawalsExist`. There is no admin function to redirect the stuck ETH to an alternate address or to forcibly clear the stuck request.

The protocol's own NatDoc comment on `completeWithdrawalForUser` acknowledges ETH is not the expected use case: [7](#0-6) 

Yet no alternative recovery path for ETH is provided.

## Impact Explanation

**Critical — Permanent freezing of funds.** The user's rsETH is burned in `unlockQueue` (an already-finalized on-chain transaction). If `completeWithdrawal` always reverts for that user, the ETH corresponding to the burned rsETH is locked in `LRTWithdrawalManager` indefinitely. The user loses both their rsETH and their ETH with no on-chain recovery mechanism. This matches the allowed impact: *"Critical. Permanent freezing of funds."*

## Likelihood Explanation

**Low-to-medium.** Smart contract wallets (Gnosis Safe multisigs, DAO treasuries, yield aggregator vaults) are standard DeFi participants and frequently interact with LST/LRT protocols. A contract without a `receive()` function, or whose `receive()` conditionally reverts (e.g., after an upgrade, when paused, or due to a gas-limited callback), triggers this path. The 8-day minimum delay between `initiateWithdrawal` and `completeWithdrawal`: [8](#0-7) 

means a contract's ETH-receiving capability can change between the two transactions, making the failure non-obvious at initiation time. No attacker action is required; the victim's own contract architecture is sufficient to trigger permanent loss.

## Recommendation

Replace the push-payment pattern with a pull-payment pattern for ETH withdrawals. Instead of calling `payable(to).call{value: amount}("")` inside `_processWithdrawalCompletion`, record the owed ETH in a `pendingETH[user]` mapping and provide a separate `claimETH()` function the user can call from any address they control. This decouples the accounting finalization (nonce pop, request deletion, count decrement) from the ETH delivery, eliminating the revert-and-stuck-funds scenario. Alternatively, add an admin function that allows redirecting a stuck ETH withdrawal to an alternate address provided by the original withdrawer.

## Proof of Concept

1. A DAO treasury contract (no `receive()`) calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred into `LRTWithdrawalManager` and a `WithdrawalRequest` is recorded.
2. After `withdrawalDelayBlocks` pass, the operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned via `burnFrom` at L305. `unlockedWithdrawalsCount[ETH_TOKEN]` is incremented. This transaction is finalized.
3. The DAO calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion`, `_transferAsset` executes `payable(daoAddress).call{value: amount}("")`. The DAO has no `receive()`, so `sent == false`. `EthTransferFailed` is reverted. All state changes in this call revert.
4. The operator calls `completeWithdrawalForUser(ETH_TOKEN, daoAddress, "")` — same code path, same revert.
5. The operator attempts `sweepRemainingAssets(ETH_TOKEN)` — reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.
6. The DAO's rsETH is permanently burned. The ETH is permanently locked in `LRTWithdrawalManager`.

**Foundry test sketch:**
```solidity
contract NoReceive { /* no receive() */ }

function test_ethFrozenForNoReceiveContract() public {
    NoReceive dao = new NoReceive();
    // Fund dao with rsETH, initiate withdrawal as dao
    vm.prank(address(dao));
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
    // Advance blocks past delay
    vm.roll(block.number + withdrawalDelayBlocks + 1);
    // Operator unlocks queue (burns rsETH)
    vm.prank(operator);
    withdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, ...);
    // completeWithdrawal reverts
    vm.prank(address(dao));
    vm.expectRevert(LRTWithdrawalManager.EthTransferFailed.selector);
    withdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    // sweepRemainingAssets also reverts
    vm.prank(manager);
    vm.expectRevert(LRTWithdrawalManager.PendingWithdrawalsExist.selector);
    withdrawalManager.sweepRemainingAssets(ETH_TOKEN);
    // ETH balance of withdrawalManager is unchanged; rsETH supply is reduced
}
```
