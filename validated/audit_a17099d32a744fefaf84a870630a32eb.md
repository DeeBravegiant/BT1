Audit Report

## Title
Permissionless `addLiquidity` Enables Frontrun Griefing of `removeLiquidity` via `MinimalLiquidity` Revert — (File: `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`addLiquidity` in `MetricOmmPool` accepts an arbitrary `owner` address with no consent check from that owner. An attacker can frontrun any victim's `removeLiquidity` call by injecting a single share into the victim's position, causing the post-removal dust-floor check to revert with `MinimalLiquidity`. This can be repeated indefinitely, permanently blocking the victim from exiting their position.

## Finding Description

**Root cause 1 — `addLiquidity` is permissionless with respect to `owner`.**

`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` parameter and contains no `msg.sender == owner` guard: [1](#0-0) 

Any caller (who implements `IMetricOmmModifyLiquidityCallback` and pays the tokens) can mint shares into any victim's position. The `_beforeAddLiquidity` extension hook receives `sender` and `owner` separately but the base pool enforces no ownership check. The `DepositAllowlistExtension`, when present, gates on `owner` — which the victim already satisfies as a legitimate LP — so it does not block the attacker either. [2](#0-1) 

**Root cause 2 — `removeLiquidity` reverts when remaining shares fall in `(0, MINIMAL_MINTABLE_LIQUIDITY)`.** [3](#0-2) 

**Root cause 3 — `addLiquidity`'s own dust check does not prevent the injection.**

The `addLiquidity` dust check only requires `newUserShares >= minimalMintableLiquidity` after the add. If the victim holds exactly `MINIMAL_MINTABLE_LIQUIDITY` shares and the attacker adds 1, `newUserShares = MINIMAL_MINTABLE_LIQUIDITY + 1`, which passes: [4](#0-3) 

**Attack flow:**

1. Victim holds `userShares = MINIMAL_MINTABLE_LIQUIDITY` (e.g., 1 000) in bin `B`, salt `S`.
2. Victim submits `removeLiquidity(owner=victim, salt=S, shares=[1000])`.
3. Attacker frontruns with `addLiquidity(owner=victim, salt=S, shares=[1])` — pays 1 share of tokens; victim's balance becomes 1 001.
4. Victim's tx executes: `newUserShares = 1001 − 1000 = 1`; `1 > 0 && 1 < 1000` → `MinimalLiquidity` revert.
5. Victim retries with `shares=[1001]`; attacker frontruns again with `shares=[1]`; `newUserShares = 1` → revert again.

The victim cannot submit a "remove all" amount that survives frontrunning because the attacker always adds exactly 1 share, leaving `remaining = 1 < MINIMAL_MINTABLE_LIQUIDITY`. The `removeLiquidity` path enforces `msg.sender == owner`, so the victim cannot delegate removal to a helper contract unless their `owner` address is itself a contract. [5](#0-4) 

## Impact Explanation

The victim's LP position becomes permanently inaccessible as long as the attacker continues frontrunning. The victim's token balances grow (attacker's injected tokens are credited to the victim's position) but can never be withdrawn to zero. Any partial removal that would leave `0 < remaining < MINIMAL_MINTABLE_LIQUIDITY` also reverts. This constitutes an **unusable `removeLiquidity` flow** — a broken core pool functionality causing effective loss of access to LP assets — matching the allowed impact gate.

## Likelihood Explanation

Medium. The attacker must be a contract implementing `IMetricOmmModifyLiquidityCallback` and must spend real tokens (1 share worth of collateral) per frontrun. The victim's salt is recoverable from the indexed `LiquidityAdded` event emitted at `LiquidityLib.sol` line 157. EOA owners have no atomic read-then-remove primitive and are fully exposed. Contract owners can mitigate by reading `positionBinShares` and removing the exact current balance atomically, but this requires the owner to be a contract. The attack is repeatable at low cost relative to the victim's locked principal. [6](#0-5) 

## Recommendation

**Option A (preferred):** In `removeLiquidity`, skip the `MinimalLiquidity` check when `newUserShares == 0` (full exit). A full exit to zero is always safe and should never be blocked by the dust floor:

```solidity
uint256 newUserShares = userShares - sharesToRemove;
// Only enforce dust floor on partial exits
if (newUserShares > 0 && newUserShares < ctx.minimalMintableLiquidity) {
    revert IMetricOmmPoolActions.MinimalLiquidity(newUserShares, ctx.minimalMintableLiquidity);
}
```

This is already the existing code structure — the fix is to confirm `newUserShares == 0` is always allowed, which it already is. The actual fix needed is to ensure the victim can always remove *all* shares atomically. A `removeAllLiquidity` entry point that reads `positionBinShares[posKey]` at execution time and removes that exact amount would eliminate the race condition entirely.

**Option B:** Require `msg.sender == owner` (or an approved operator) in `addLiquidity`, eliminating the griefing surface at the cost of breaking the operator/router pattern.

## Proof of Concept

```
Setup:
  MINIMAL_MINTABLE_LIQUIDITY = 1_000
  Victim (EOA) holds 1_000 shares in bin 0, salt = 0
  Attacker deploys GriefContract implementing IMetricOmmModifyLiquidityCallback

Step 1 — Victim submits:
  pool.removeLiquidity(victim, 0, {binIdxs:[0], shares:[1000]}, "")

Step 2 — Attacker frontruns (GriefContract pays ~1 share of token):
  pool.addLiquidity(victim, 0, {binIdxs:[0], shares:[1]}, "", "")
  // positionBinShares[key(victim,0,0)] = 1_001

Step 3 — Victim's tx executes:
  newUserShares = 1001 - 1000 = 1
  1 > 0 && 1 < 1000  →  revert MinimalLiquidity(1, 1000)

Step 4 — Victim retries with shares=[1001]:
  Attacker frontruns again with shares=[1]
  newUserShares = 1002 - 1001 = 1  →  revert

Result: Victim's position is permanently griefed; funds are inaccessible.

Foundry test sketch:
  1. Deploy pool with MINIMAL_MINTABLE_LIQUIDITY = 1_000
  2. victim.addLiquidity(victim, 0, {[0],[1000]}, ...)  // victim holds 1000 shares
  3. vm.prank(attacker); pool.addLiquidity(victim, 0, {[0],[1]}, ...)  // inject 1 share
  4. vm.prank(victim); vm.expectRevert(MinimalLiquidity.selector);
     pool.removeLiquidity(victim, 0, {[0],[1000]}, "")
```
