Audit Report

## Title
Unprivileged Caller Can Inject Shares Into Any LP Position via Unguarded `addLiquidity`, Enabling Persistent `minimalMintableLiquidity` DoS — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary

`addLiquidity` accepts an arbitrary `owner` address with no `msg.sender == owner` check, while `removeLiquidity` enforces this guard. Any caller implementing `IMetricOmmModifyLiquidityCallback` can credit shares to a victim's position key while paying the token cost themselves. Combined with the `minimalMintableLiquidity` floor enforced in `removeLiquidity`, an attacker can persistently block any smart-contract LP that submits a fixed share count from successfully withdrawing.

## Finding Description

`removeLiquidity` correctly restricts callers at [1](#0-0)  but `addLiquidity` has no equivalent guard: [2](#0-1) 

Inside `LiquidityLib.addLiquidity`, the position key is derived from the supplied `owner`, not `msg.sender`: [3](#0-2) 

The token payment callback is fired on `msg.sender` (attacker pays), but shares are credited to `owner` (victim receives): [4](#0-3) 

After injection, `removeLiquidity` enforces the dust floor: [5](#0-4) 

**Attack sequence:**
1. Victim (router/vault) holds exactly `X` shares in bin `b` and submits `removeLiquidity(victim, salt, [{bin: b, shares: X}])`.
2. Attacker front-runs with `addLiquidity(victim, salt, [{bin: b, shares: 1}])`, paying 1-wei-equivalent tokens via callback.
3. Victim now holds `X + 1` shares. Their call removes `X`, leaving `1`.
4. `1 < minimalMintableLiquidity` → `MinimalLiquidity` revert. Victim's transaction fails.
5. Attacker repeats on every retry at near-zero cost, permanently blocking the victim's fixed-amount withdrawal path.

The `DepositAllowlistExtension` does not mitigate this: it gates on `owner`, not `sender`, so an attacker can still inject shares into any allowlisted victim's position: [6](#0-5) 

## Impact Explanation

Any smart-contract LP — router, vault, or aggregator — that reads its share count before submission and passes a fixed value to `removeLiquidity` is permanently DoS-able at near-zero cost. The victim's liquidity is inaccessible via its normal code path, constituting an unusable withdraw/liquidity flow per the allowed impact gate. While the victim could theoretically adapt by re-querying its balance, the attacker can front-run every retry indefinitely, making the withdrawal path durably broken for inflexible integrators.

## Likelihood Explanation

- The attacker only needs to implement `IMetricOmmModifyLiquidityCallback` (trivial) and hold a dust amount of pool tokens.
- Front-running is straightforward on any EVM chain with a public mempool.
- Cost per attack round is near-zero (1 wei of scaled token value).
- Any protocol integrating Metric OMM liquidity management via a contract is exposed; EOA LPs are unaffected since they can re-query balance interactively.

## Recommendation

Add the same ownership guard to `addLiquidity` that already exists in `removeLiquidity`:

```solidity
function addLiquidity(address owner, ...) external nonReentrant(PoolActions.ADD_LIQUIDITY) ... {
+   if (msg.sender != owner) revert NotPositionOwner();
    ...
}
```

If "deposit on behalf of" is a desired feature, introduce an explicit allowance mapping (`approvedDepositor[owner][depositor]`) so the owner opts in, rather than allowing any caller to inject shares unconditionally.

## Proof of Concept

```solidity
contract Attacker is IMetricOmmModifyLiquidityCallback {
    IMetricOmmPool pool;
    IERC20 token0;
    IERC20 token1;

    function metricOmmModifyLiquidityCallback(
        uint256 amount0, uint256 amount1, bytes calldata
    ) external {
        if (amount0 > 0) token0.transfer(msg.sender, amount0);
        if (amount1 > 0) token1.transfer(msg.sender, amount1);
    }

    function grief(address victim, uint80 salt, int256 binIdx) external {
        int256[] memory bins = new int256[](1);
        uint256[] memory shares = new uint256[](1);
        bins[0] = binIdx;
        shares[0] = 1; // dust injection
        LiquidityDelta memory delta = LiquidityDelta({binIdxs: bins, shares: shares});
        pool.addLiquidity(victim, salt, delta, "", "");
        // victim now has originalShares + 1; their fixed-amount removeLiquidity
        // reverts with MinimalLiquidity(1, minimalMintableLiquidity)
    }
}
```

Foundry test plan: deploy pool, add `X` shares as victim, call `grief(victim, salt, binIdx)` as attacker, then assert `removeLiquidity(victim, salt, [{bin, X}])` reverts with `MinimalLiquidity`.
