Audit Report

## Title
Missing Decimal Normalization for Non-18-Decimal Tokens in `viewSwapRsETHAmountAndFee` Causes Depositors to Receive Near-Zero wrsETH - (File: `contracts/pools/RSETHPoolV3.sol`)

## Summary
`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18 decimals. Because `ChainlinkOracleForRSETHPoolCollateral.getRate()` returns the price of one full token unit in ETH scaled to 1e18, the formula is dimensionally inconsistent for tokens with fewer than 18 decimals (e.g., USDC at 6 decimals), producing a result that is `10^(18 − tokenDecimals)` times too small. Any user depositing such a token loses their entire deposit while receiving economically worthless wrsETH.

## Finding Description
`viewSwapRsETHAmountAndFee` at L324–L334 of `contracts/pools/RSETHPoolV3.sol`:

```solidity
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();                                       // 1e18-scaled
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();  // 1e18-scaled
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`ChainlinkOracleForRSETHPoolCollateral.getRate()` at L34 normalizes the Chainlink answer to represent the price of **one full human-readable token unit** in ETH, scaled to 1e18:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

For WETH (18 decimals), `amountAfterFee` in raw units equals the amount in wei (1e18-scaled), so the formula is dimensionally consistent. For USDC (6 decimals), `amountAfterFee` is in micro-USDC (1e6-scaled), but `tokenToETHRate` still represents the price of 1 full USDC in ETH (1e18-scaled). The product `amountAfterFee * tokenToETHRate` is therefore `10^12` times smaller than it should be, yielding an `rsETHAmount` that is `10^12` times too small.

The `deposit(address token, uint256 amount, string referralId)` function at L271–L293 unconditionally transfers the full token amount from the user (`safeTransferFrom` at L284) before calling `viewSwapRsETHAmountAndFee` and minting the (near-zero) wrsETH. There is no slippage check, minimum output check, or decimal guard anywhere in the call path. The `addSupportedToken` function at L541–L555 imposes no restriction on token decimals, and `ChainlinkOracleForRSETHPoolCollateral` is explicitly designed to handle Chainlink feeds with non-standard decimal counts, confirming the protocol's intent to support non-18-decimal assets.

The same arithmetic error is present in `RSETHPoolV3ExternalBridge.sol` L452, `RSETHPoolV3WithNativeChainBridge.sol` L370, `RSETHPool.sol` L346, `RSETHPoolNoWrapper.sol` L311, and `AGETHPoolV3.sol` L194.

The `viewSwapAssetToPremintedRsETH` function at L400 (`tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate`) has the symmetric error: it would return `10^12` times too many tokens for a given rsETH input when the output token has 6 decimals, enabling over-withdrawal of pool assets.

## Impact Explanation
**Critical — Direct theft of user funds.** A depositor calling `deposit(USDC, 1_000e6, "ref")` transfers 1,000 USDC (≈ $1,000) to the pool and receives approximately 317,000 raw wrsETH units (≈ 3.17 × 10⁻¹³ wrsETH, economically worthless). The USDC is irrecoverable by the depositor; it sits in the pool until `BRIDGER_ROLE` calls `moveAssetsForBridging`, at which point it is permanently removed. This matches the allowed impact: **Critical — Direct theft of any user funds**.

## Likelihood Explanation
**High.** The `addSupportedToken` function is a routine protocol operation gated by `TIMELOCK_ROLE`, not an attack vector. Once any non-18-decimal token is added (a legitimate governance action the protocol is designed to support), every subsequent deposit of that token by any ordinary user triggers the loss with no special preconditions. USDC is the most common non-18-decimal collateral on every chain where these pools are deployed. The `ChainlinkOracleForRSETHPoolCollateral` wrapper's explicit normalization of Chainlink feed decimals confirms the protocol intends to support such tokens. No victim mistake is required; the loss occurs on every deposit.

## Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate formula in `viewSwapRsETHAmountAndFee`:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply the inverse normalization in `viewSwapAssetToPremintedRsETH` (divide the result by `10 ** (18 - tokenDecimals)` to convert back to token raw units). Apply both fixes to all pool variants listed above. Consider adding a decimal check in `addSupportedToken` or enforcing the normalization at the oracle layer.

## Proof of Concept
1. Admin (via timelock) calls `addSupportedToken(USDC_ADDRESS, chainlinkUSDCOracle)` on `RSETHPoolV3`. This is a routine governance action; `ChainlinkOracleForRSETHPoolCollateral` is deployed with the Chainlink USDC/ETH feed.
2. User approves the pool for 1,000 USDC and calls `pool.deposit(USDC_ADDRESS, 1_000e6, "ref")`.
3. `safeTransferFrom` moves 1,000 USDC from the user to the pool (L284).
4. `viewSwapRsETHAmountAndFee(1_000e6, USDC)` computes:
   - `amountAfterFee ≈ 1_000e6` (assuming 0 fee for simplicity)
   - `tokenToETHRate ≈ 3.33e14` (price of 1 USDC in ETH at $3,000/ETH, 1e18-scaled)
   - `rsETHToETHrate ≈ 1.05e18`
   - `rsETHAmount = 1_000e6 * 3.33e14 / 1.05e18 ≈ 317_000` raw units
5. `wrsETH.mint(msg.sender, 317_000)` mints ≈ 3.17 × 10⁻¹³ wrsETH — economically worthless.
6. The user has lost 1,000 USDC (≈ $1,000) and received nothing of value.

**Foundry test plan:** Deploy `RSETHPoolV3` with a mock `wrsETH`, a mock `rsETHOracle` returning `1.05e18`, and a mock `ChainlinkOracleForRSETHPoolCollateral` wrapping a mock Chainlink feed returning `333_000_000` (8-decimal USDC/ETH feed at $3,000/ETH). Call `addSupportedToken(mockUSDC, oracle)`. Mint 1,000 mock USDC (6 decimals) to a test address, approve the pool, call `deposit`. Assert that `wrsETH.balanceOf(user) < 1e6` (near zero) and `mockUSDC.balanceOf(pool) == 1_000e6`.
