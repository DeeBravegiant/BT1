Audit Report

## Title
Missing Zero-Address Validation for `owner` in `addLiquidity` Permanently Locks LP Principal - (File: `metric-core/contracts/MetricOmmPool.sol`)

## Summary
`MetricOmmPool.addLiquidity` accepts a caller-supplied `owner` address with no zero-address guard, passing it directly to `LiquidityLib.addLiquidity` where shares are minted to `_positionBinKey(address(0), salt, binIdx)`. Because `removeLiquidity` enforces `msg.sender == owner`, and no account can have `msg.sender == address(0)`, any position minted to the zero address is permanently unredeemable, locking the deposited tokens forever.

## Finding Description
`MetricOmmPool.addLiquidity` (L182–196) passes `owner` directly to `LiquidityLib.addLiquidity` without any zero-address check:

```solidity
// metric-core/contracts/MetricOmmPool.sol L192-194
(amount0Added, amount1Added) = LiquidityLib.addLiquidity(
  _liquidityContext(), owner, salt, deltas, callbackData, binTotals, _binStates, _binTotalShares, _positionBinShares
);
```

Inside `LiquidityLib.addLiquidity` (L72), shares are credited to `_positionBinKey(address(0), salt, int8(binIdx))` and tokens are pulled from `msg.sender` via the modify-liquidity callback (L147–148). The pool's `binTotals` and `binTotalShares` are updated correctly, but the LP shares are stored under an unreachable key.

`MetricOmmPool.removeLiquidity` (L206) enforces:
```solidity
if (msg.sender != owner) revert NotPositionOwner();
```
Since `msg.sender` can never equal `address(0)`, the position is permanently locked. The periphery contract `MetricOmmPoolLiquidityAdder._validateOwner` (L247–249) guards against this, but the core pool itself does not, leaving any direct caller of the core pool unprotected.

## Impact Explanation
Direct, permanent loss of user principal. Tokens deposited into a zero-address-owned position are irrecoverable by any party. The pool's `binTotals` correctly account for the tokens (pool remains solvent), but the LP shares are permanently unredeemable — the underlying tokens can never be withdrawn. This constitutes a direct loss of user principal meeting the High severity threshold under Sherlock contest rules.

## Likelihood Explanation
Low-to-medium. Requires a caller to supply `owner = address(0)`, which is an integration mistake rather than a deliberate attack. However, the core pool is a public, permissionless interface callable by any contract or EOA. Third-party routers or integrators that interact with the core pool directly — bypassing `MetricOmmPoolLiquidityAdder` — have no on-chain protection. The inconsistency between the periphery guard and the absent core guard increases the probability of integration errors. No privileged role is required.

## Recommendation
Add a zero-address check at the top of `MetricOmmPool.addLiquidity`, before delegating to `LiquidityLib`:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external nonReentrant(PoolActions.ADD_LIQUIDITY) returns (uint256 amount0Added, uint256 amount1Added) {
    require(owner != address(0), InvalidOwner()); // <-- add this
    if (deltas.binIdxs.length == 0) return (0, 0);
    ...
}
```

This mirrors the guard already present in `MetricOmmPoolLiquidityAdder._validateOwner` and closes the gap between the periphery and core layers.

## Proof of Concept
1. Deploy a pool and a caller contract implementing `IMetricOmmModifyLiquidityCallback` holding token0/token1.
2. Call `pool.addLiquidity(address(0), 0, deltas, callbackData, "")` directly on the core pool.
3. The callback fires on `msg.sender`; tokens are transferred into the pool; shares are minted to `keccak256(abi.encode(address(0), uint80(0), int8(binIdx)))`.
4. Attempt `pool.removeLiquidity(address(0), 0, deltas, "")` — reverts with `NotPositionOwner` because `msg.sender != address(0)`.
5. No further call can ever redeem those shares. The deposited tokens are permanently locked in the pool.
