Audit Report

## Title
Stale Cross-Chain rsETH/ETH Rate Used for L2 Minting Without Freshness Check â€” (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` with no check against `lastUpdated`, meaning any depositor on L2 can mint rsETH/wrsETH using an arbitrarily stale rate. Because rsETH is a yield-bearing token whose L1 price monotonically increases, a stale (lower) rate in the denominator causes over-minting, diluting the ETH-per-rsETH ratio for all existing holders and constituting theft of their accrued yield.

## Finding Description
`CrossChainRateReceiver` stores the rate received via LayerZero and exposes it through `getRate()`:

```solidity
// contracts/cross-chain/CrossChainRateReceiver.sol L13-16
uint256 public rate;
uint256 public lastUpdated;

// L103-105
function getRate() external view returns (uint256) {
    return rate;   // no staleness check
}
```

`lastUpdated` is written in `lzReceive()` when a LayerZero message arrives, but is never read in `getRate()`. The rate is only refreshed when someone calls `updateRate()` on `MultiChainRateProvider` and pays the cross-chain gas fee â€” there is no on-chain enforcement of any update cadence.

Every L2 pool contract delegates to `IOracle(rsETHOracle).getRate()` (which resolves to `CrossChainRateReceiver.getRate()`) and uses the result as the denominator for minting:

- `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee()` L282â€“285: `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`
- `RSETHPoolV3.viewSwapRsETHAmountAndFee()` L303â€“307: identical pattern
- `RSETHPool`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`: all identical

A lower stale rate in the denominator produces a larger `rsETHAmount` for the same ETH input. The team demonstrably knows the staleness-check pattern: `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L30) reverts on `answeredInRound < roundID`, yet no equivalent guard exists on the cross-chain rate path.

Note: several pools have a `dailyMintLimit`, but `RSETHPoolNoWrapper` and `RSETHPool` do not. Even where the limit exists, it is denominated in rsETH and does not prevent the attack â€” it only caps the per-day magnitude.

## Impact Explanation
When the L2 oracle rate is stale and lower than the true L1 rsETH price, every depositor receives more rsETH than the deposited ETH warrants. The excess rsETH is unbacked; when redeemed on L1 at the true rate, it extracts ETH that was accrued as yield by existing holders. This is a direct, concrete instance of **theft of unclaimed yield** (High), which is an explicitly allowed impact class. The magnitude scales with deposit size and degree of staleness.

## Likelihood Explanation
`updateRate()` is permissionless but requires the caller to pay LayerZero cross-chain gas fees. No keeper is enforced on-chain. During periods of high gas prices, network congestion, or keeper downtime, the rate can remain stale for hours or days. rsETH yield accrues continuously (every block), so even a few hours of staleness creates a profitable window. The attack requires no special permissions â€” any L2 depositor can execute it by simply calling `deposit()`.

## Recommendation
Add a configurable maximum staleness threshold to `CrossChainRateReceiver.getRate()` and revert if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g., 24 hours

function getRate() external view returns (uint256) {
    if (block.timestamp - lastUpdated > maxStaleness) revert StaleRate();
    return rate;
}
```

Pair this with an automated keeper that calls `updateRate()` on a regular cadence (e.g., every 4â€“8 hours). The staleness bound should be set conservatively relative to the keeper interval.

## Proof of Concept
1. At time T, the true L1 rsETH price is 1.05 ETH/rsETH. `CrossChainRateReceiver.rate` was last updated 48 hours ago at 1.03 ETH/rsETH. No one has called `updateRate()` since.
2. Attacker calls `deposit{value: 1050 ETH}("")` on `RSETHPoolNoWrapper` (no daily mint limit).
3. `viewSwapRsETHAmountAndFee(1050 ETH)` calls `getRate()` â†’ returns stale `1.03e18`.
4. `rsETHAmount = 1050e18 * 1e18 / 1.03e18 â‰ˆ 1019.4 rsETH`. Correct amount at true rate: `1050 / 1.05 = 1000 rsETH`. Over-mint: ~19.4 rsETH.
5. Attacker bridges rsETH to L1 and redeems at the true rate of 1.05 ETH/rsETH, extracting ~20.4 ETH of value diluted from existing holders.

- Fork an L2 (e.g., Arbitrum) at a block where `lastUpdated` is >24h old.
- Call `RSETHPoolNoWrapper.deposit{value: X}("")`.
- Assert `rsETHAmount > X * 1e18 / trueL1Rate`, confirming over-minting.
- Bridge rsETH to L1 fork and redeem, confirming net ETH profit at existing holders' expense.
