### Title
Fee-on-Transfer Token Amount Discrepancy in `initTransfer` Creates Unbacked Supply on NEAR - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
The EVM `initTransfer` function records the caller-supplied `amount` parameter in the `InitTransfer` event without verifying the actual token amount received by the bridge contract. For fee-on-transfer ERC20 tokens, the bridge receives fewer tokens than `amount`, but the NEAR side credits the full `amount`, creating unbacked wrapped supply and a permanently unclaimable shortfall.

### Finding Description
In `OmniBridge.initTransfer`, the non-bridge ERC20 lock path executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // caller-supplied parameter
);
``` [1](#0-0) 

Immediately after, the event is emitted using the same caller-supplied `amount`, not the actual balance delta:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,         // parameter, NOT actual received amount
    ...
);
``` [2](#0-1) 

For a fee-on-transfer ERC20, `safeTransferFrom` succeeds (the ERC20 call itself does not revert), but the bridge receives `amount - fee_deducted`. The event still records the full `amount`.

The NEAR bridge's `fin_transfer_callback` then processes the proven EVM event and constructs a `TransferMessage` using `init_transfer.amount` directly from the event:

```rust
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [3](#0-2) 

This causes the NEAR side to mint or credit the full `amount` of bridge tokens to the recipient, while the EVM bridge only holds `amount - fee_deducted` as backing.

### Impact Explanation
An attacker deploys a fee-on-transfer ERC20 token on EVM (permissionless), calls `logMetadata` to register it (permissionless), then calls `initTransfer` with `amount = N`. The bridge receives `N - δ` tokens but the NEAR side mints `N` bridge tokens. The attacker can sell these `N` NEAR-side tokens to other users. When those users attempt to bridge back, the EVM bridge only holds `N - δ` locked tokens; the final `δ` tokens are permanently unclaimable — a fund lock. This breaks the 1:1 backing guarantee between EVM-locked tokens and NEAR-minted supply.

This matches: **High — balance-accounting divergence that breaks backing guarantees**, and potentially **Critical — irreversible fund lock / permanently unclaimable user value**.

### Likelihood Explanation
The attack requires no privileged access. Any unprivileged actor can deploy a fee-on-transfer ERC20 (standard Solidity), call the permissionless `logMetadata`, and then call `initTransfer`. The entry path is fully reachable and attacker-controlled. Fee-on-transfer tokens are a well-established token pattern.

### Recommendation
Use a balance-before / balance-after pattern to determine the actual amount received, and emit that value in the event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint128 actualReceived = uint128(IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore);
// use actualReceived in the event and downstream logic
```

This ensures the NEAR side is informed of the true locked amount, preserving the backing invariant regardless of token transfer mechanics.

### Proof of Concept
1. Attacker deploys `FeeToken` on EVM: a standard ERC20 that deducts 10% on every transfer.
2. Attacker calls `OmniBridge.logMetadata(address(FeeToken))` — permissionless, registers the token on NEAR.
3. Attacker approves the bridge for 1000 `FeeToken` and calls `initTransfer(FeeToken, 1000, 0, 0, nearRecipient, "")`.
4. `safeTransferFrom` succeeds; bridge receives 900 tokens (10% fee deducted). Event emits `amount = 1000`.
5. A relayer submits the proof to the NEAR bridge's `fin_transfer`. `fin_transfer_callback` reads `init_transfer.amount = 1000` from the proven event and mints 1000 bridge tokens to `nearRecipient`.
6. Attacker sells 1000 NEAR-side bridge tokens to other users.
7. Users attempt to bridge back 1000 tokens to EVM. The EVM bridge only holds 900 locked; the last 100 tokens cannot be redeemed — permanently locked. [4](#0-3) [5](#0-4)

        address tokenAddress,
        uint128 amount,
        uint128 fee,

        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;

            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {

                    amount

    pub fn fin_transfer_callback(
        &mut self,
        #[serializer(borsh)] storage_deposit_actions: &Vec<StorageDepositAction>,
        #[serializer(borsh)] predecessor_account_id: AccountId,
    ) -> PromiseOrValue<Nonce> {
        let Ok(ProverResult::InitTransfer(init_transfer)) = Self::decode_prover_result(0) else {
            env::panic_str(BridgeError::InvalidProofMessage.to_string().as_str())
        };
        require!(
            self.factories
                .get(&init_transfer.emitter_address.get_chain())
                == Some(init_transfer.emitter_address),
            BridgeError::UnknownFactory.as_ref()
        );

        let decimals = self
            .token_decimals
            .get(&init_transfer.token)
            .near_expect(BridgeError::TokenDecimalsNotFound);

        let destination_nonce =
            self.get_next_destination_nonce(init_transfer.recipient.get_chain());
        let transfer_message = TransferMessage {
            origin_nonce: init_transfer.origin_nonce,
            token: init_transfer.token,
            amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
            recipient: init_transfer.recipient,
            fee: Self::denormalize_fee(&init_transfer.fee, decimals),
            sender: init_transfer.sender,
            msg: init_transfer.msg,
            destination_nonce,
            origin_transfer_id: None,
        };

        if let OmniAddress::Near(recipient) = transfer_message.recipient.clone() {
            self.process_fin_transfer_to_near(
                recipient,
                &predecessor_account_id,
                transfer_message,
                storage_deposit_actions,
            )
            .into()
        } else {
            self.process_fin_transfer_to_other_chain(predecessor_account_id, transfer_message);
            PromiseOrValue::Value(destination_nonce)
        }
    }
