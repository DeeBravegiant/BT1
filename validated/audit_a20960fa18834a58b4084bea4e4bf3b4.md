Audit Report

## Title
`RSETH._transfer()` Missing `whenNotPaused` Allows Transfers and Temporary Fund Lock During Emergency Pause - (File: contracts/RSETH.sol)

## Summary
`RSETH` applies `whenNotPaused` to `mint()` and `burnFrom()` but omits it from the `_transfer()` override, meaning all ERC20 `transfer()` and `transferFrom()` calls proceed freely regardless of pause state. A concrete consequence is that any user can call `LRTWithdrawalManager.initiateWithdrawal()` while RSETH is paused — the `safeTransferFrom` succeeds (routing through the unguarded `_transfer()`), locking the user's rsETH inside the withdrawal manager, while the subsequent `unlockQueue()` → `burnFrom()` path reverts on the pause check, leaving the rsETH inaccessible until the pause is lifted.

## Finding Description
`RSETH` inherits `PausableUpgradeable` and gates supply-changing operations:

- `mint()` at `contracts/RSETH.sol:235` carries `whenNotPaused`.
- `burnFrom()` at `contracts/RSETH.sol:245` carries `whenNotPaused`.

However, the `_transfer()` override that backs every public ERC20 transfer has no such guard:

```solidity
// contracts/RSETH.sol:287-291
function _transfer(address from, address to, uint256 amount) internal override {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);   // no whenNotPaused
}
```

`LRTWithdrawalManager.initiateWithdrawal()` pulls rsETH from the caller via:

```solidity
// contracts/LRTWithdrawalManager.sol:166
IERC20(lrtConfig.rsETH()).safeTransferFrom(msg.sender, address(this), rsETHUnstaked);
```

Because `transferFrom` routes through `_transfer()`, this line succeeds even when RSETH is paused. The rsETH is now held by the withdrawal manager. The only path to release it is `unlockQueue()`, which calls:

```solidity
// contracts/LRTWithdrawalManager.sol:305
if (rsETHBurned != 0) IRSETH(lrtConfig.rsETH()).burnFrom(address(this), rsETHBurned);
```

`burnFrom()` carries `whenNotPaused` and reverts while RSETH is paused. There is no `cancelWithdrawal()` function in `LRTWithdrawalManager`, so the user has no mechanism to reclaim their rsETH until the pause is lifted.

The two contracts have independent pause mechanisms (`RSETH.pause()` and `LRTWithdrawalManager.pause()`), making the split-pause state a realistic operational scenario.

## Impact Explanation
**Medium — Temporary freezing of funds.**

A user who calls `initiateWithdrawal()` while RSETH is paused (and `LRTWithdrawalManager` is not) will have their rsETH transferred into the withdrawal manager with no ability to complete or cancel the withdrawal until RSETH is unpaused. The funds are not permanently lost but are inaccessible for the duration of the pause, which can be extended at admin discretion. This satisfies the "Temporary freezing of funds" impact class.

The broader bypass (any holder freely transferring rsETH during a pause) additionally satisfies "Low — Contract fails to deliver promised returns," as the emergency freeze mechanism does not halt token circulation.

## Likelihood Explanation
The split-pause state (RSETH paused, `LRTWithdrawalManager` not paused) is a realistic operational scenario — an admin responding to an oracle or minting exploit would naturally pause RSETH first. Any unprivileged rsETH holder can trigger the fund-lock path by calling the public `initiateWithdrawal()` function with no special access or preconditions beyond holding rsETH and having approved the withdrawal manager. The condition is repeatable for every user who attempts a withdrawal during the pause window.

## Recommendation
Add `whenNotPaused` to the `_transfer()` override so all token movements halt during an emergency:

```solidity
function _transfer(address from, address to, uint256 amount) internal override whenNotPaused {
    _enforceNotBlocked(from);
    _enforceNotBlocked(to);
    super._transfer(from, to, amount);
}
```

For the `recoverFrozenFunds` admin path, which already bypasses the block check by calling `super._transfer()` directly (`contracts/RSETH.sol:218`), no change is needed — it does not go through the overridden `_transfer()` and will continue to work even while paused.

## Proof of Concept

1. Admin detects an exploit and calls `RSETH.pause()`. `LRTWithdrawalManager` remains unpaused.
2. User holds rsETH and approves `LRTWithdrawalManager` to spend it.
3. User calls `LRTWithdrawalManager.initiateWithdrawal(asset, amount, "")`.
4. Line 166 of `LRTWithdrawalManager.sol` executes `safeTransferFrom` → `transferFrom` → `_transfer()`. Because `_transfer()` has no `whenNotPaused`, the call succeeds and rsETH moves to the withdrawal manager.
5. Operator calls `unlockQueue(...)`. At line 305, `burnFrom(address(this), rsETHBurned)` is called on RSETH. `burnFrom` carries `whenNotPaused` and reverts — the queue cannot be processed.
6. User calls `completeWithdrawal()` — reverts at line 707 (`WithdrawalLocked`) because `nextLockedNonce` was never advanced.
7. No `cancelWithdrawal()` exists. User's rsETH remains locked in the withdrawal manager until `RSETH.unpause()` is called by the LRT admin.

**Foundry test sketch:**
```solidity
