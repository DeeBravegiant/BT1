### Title
Malicious Third-Party Harvester Can Unconditionally Divert Farmer Block Rewards via Unenforced `farmer_reward_address_override` - (File: chia/farmer/farmer_api.py)

### Summary
CHIP-22 introduced `farmer_reward_address_override` to allow third-party harvesters to take a fee from the farmer's 1/8 block reward. The farmer is supposed to validate the fee convention (checking `fee_quality <= fee_threshold`) and reject illegitimate overrides. However, the farmer only **logs a warning** when the override is invalid — it never rejects the block or drops the override. A malicious harvester connected to a farmer can set `farmer_reward_address_override` to any arbitrary puzzle hash (including with `fee_info=None`) and the farmer will unconditionally use that address as the farmer reward destination, diverting 100% of the farmer's XCH block rewards to the attacker.

### Finding Description

The CHIP-22 fee convention allows third-party harvesters to redirect the farmer's block reward to themselves as a fee, governed by a probabilistic quality check. The farmer is expected to enforce this convention.

**Path 1 — via `NewProofOfSpace`:**

In `FarmerAPI.new_proof_of_space()`, when a harvester sends a `NewProofOfSpace` with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`: [1](#0-0) 

That function checks the fee quality but only emits log warnings on violation — it never returns an error or causes the block to be dropped: [2](#0-1) 

**Path 2 — via `RespondSignatures`:**

In `_process_respond_signatures()`, the harvester's `farmer_reward_address_override` from the `RespondSignatures` message is unconditionally accepted and replaces the farmer's configured `farmer_target`: [3](#0-2) 

The `RespondSignatures` message carries its own `farmer_reward_address_override` field, independent of the `NewProofOfSpace` message: [4](#0-3) 

There is **no enforcement** in either path that rejects the override when `fee_info is None` or when `fee_quality > fee_threshold`. The resulting `DeclareProofOfSpace` carries the attacker's address as `farmer_puzzle_hash`: [5](#0-4) 

The full node then uses `request.farmer_puzzle_hash` directly without any validation against the farmer's configured address: [6](#0-5) 

The farmer reward coin is created on-chain using whatever puzzle hash was declared: [7](#0-6) 

### Impact Explanation

Every time the farmer wins a block, the 1/8 farmer reward (XCH) is minted to the attacker's address instead of the farmer's configured `xch_target_address`. This is a permanent, on-chain diversion of XCH. The attacker receives real XCH; the farmer receives nothing for that block. This matches the **Critical** impact category: unauthorized reward diversion affecting XCH.

### Likelihood Explanation

Any third-party harvester (CHIP-22 explicitly supports external harvesters connecting to a farmer) can exploit this. The attacker needs only to:
1. Connect as a harvester to the target farmer (standard protocol connection)
2. Set `farmer_reward_address_override` to their own puzzle hash in `NewProofOfSpace` and/or `RespondSignatures`
3. Provide no `fee_info` or an invalid threshold — the farmer will log a warning and proceed anyway

No keys, admin access, or cryptographic break is required. The attack is silent (only a log warning is emitted) and persistent across all blocks the farmer wins while the malicious harvester is connected.

### Recommendation

The farmer must enforce the fee convention, not merely log violations. Specifically:

1. In `new_proof_of_space()`, if `farmer_reward_address_override is not None` and the fee convention check fails (either `fee_info is None` or `fee_quality > fee_threshold`), the farmer should **drop the proof** and return `None` rather than proceeding.
2. In `_process_respond_signatures()`, if `response.farmer_reward_address_override is not None`, the farmer should cross-check it against the override that was declared in the corresponding `NewProofOfSpace` (stored in `proofs_of_space`) and reject mismatches.
3. Consider requiring that `fee_info` always be present when `farmer_reward_address_override` is set, and that the threshold be validated before the block is submitted.

### Proof of Concept

A malicious harvester connects to a farmer and, upon receiving a valid signage point, sends:

```python
harvester_protocol.NewProofOfSpace(
    challenge_hash=...,
    sp_hash=...,
    plot_identifier=...,
    proof=<valid_proof>,
    signage_point_index=...,
    include_source_signature_data=False,
    farmer_reward_address_override=ATTACKER_PUZZLE_HASH,  # arbitrary address
    fee_info=None,  # no fee info provided — farmer only logs a warning
)
```

The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`, which logs:
> "Harvester illegitimately took reward by failing to provide its fee rate..."

...and then **continues** to request signatures and submit `DeclareProofOfSpace` with `farmer_puzzle_hash = ATTACKER_PUZZLE_HASH` to the full node. The block is farmed and the farmer reward coin is minted to `ATTACKER_PUZZLE_HASH`. The farmer's configured `xch_target_address` receives nothing.

The same attack works via `RespondSignatures.farmer_reward_address_override` even if `NewProofOfSpace.farmer_reward_address_override` was `None`, since `_process_respond_signatures()` checks `response.farmer_reward_address_override` independently with no enforcement: [8](#0-7)
