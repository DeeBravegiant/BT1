Audit Report

## Title
No Staleness Check on Chainlink `latestRoundData()` Allows Stale Asset Prices to Drive rsETH Minting - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and uses only the raw `price` return value, discarding `updatedAt`, `answeredInRound`, and performing no sign check. A stale Chainlink price silently propagates into both the rsETH minting calculation and the stored `rsETHPrice`, allowing depositors to mint excess rsETH at the expense of existing holders. The sibling contract `ChainlinkOracleForRSETHPoolCollateral` in the same repository performs all three missing checks, confirming the project is aware of the requirement.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

All five return values are available via the interface declared in the same file, but `roundId`, `startedAt`, `updatedAt`, and `answeredInRound` are all discarded. [2](#0-1) 

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` performs `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0` checks: [3](#0-2) 

The stale price feeds into two critical paths:

**Path 1 — rsETH price update:** `LRTOracle._updateRsETHPrice()` calls `_getTotalEthInProtocol()`, which calls `getAssetPrice(asset)` for every supported LST. A stale low price understates TVL, causing `rsETHPrice` to be stored too low. [4](#0-3) 

**Path 2 — Deposit minting:** `LRTDepositPool.getRsETHAmountToMint()` divides `lrtOracle.getAssetPrice(asset)` by `lrtOracle.rsETHPrice()`. When `rsETHPrice` is set too low (from Path 1), the denominator is understated and the depositor receives excess rsETH. [5](#0-4) 

The `pricePercentageLimit` guard in `_updateRsETHPrice()` provides partial downside protection (pausing the protocol if the price drop exceeds the configured threshold), but it does not protect against small deviations within the limit, and it does not guard the live `getAssetPrice()` call inside `getRsETHAmountToMint()`. [6](#0-5) 

## Impact Explanation

**High — Theft of unclaimed yield.** When a Chainlink feed goes stale with a price lower than the true market price, `rsETHPrice` is set below its correct value. Every depositor transacting during the stale window mints more rsETH than they are entitled to. When the feed corrects and `rsETHPrice` is updated upward, the over-minted supply dilutes the share value of all pre-existing rsETH holders, permanently reducing their accrued yield. This is a direct, quantifiable loss to existing holders triggered by any public `depositAsset` call during the stale window.

## Likelihood Explanation

Chainlink OCR feeds for LST/ETH pairs have documented heartbeat windows (typically 1 hour). Network congestion, gas price spikes, or OCR node failures can delay updates beyond the heartbeat. The condition is not attacker-controlled; any depositor who happens to call `depositAsset` during a stale window automatically exploits the mispricing without any special action or knowledge. The scenario is realistic and has occurred on mainnet during periods of high network activity.

## Recommendation

Add a configurable `maxStaleness` parameter to `ChainlinkPriceOracle` and validate all relevant fields from `latestRoundData()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
(uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();

require(price > 0,                                        "Invalid price");
require(updatedAt != 0,                                   "Incomplete round");
require(answeredInRound >= roundId,                       "Stale round");
require(block.timestamp - updatedAt <= maxStaleness,      "Price too stale");
```

`maxStaleness` should be set per feed based on its documented heartbeat (e.g., 3600 seconds for a 1-hour heartbeat feed, with a small buffer). It should be configurable by the LRT admin via a setter, consistent with the existing `updatePriceFeedFor` pattern. [7](#0-6) 

## Proof of Concept

1. Chainlink OCR for the stETH/ETH feed fails to push an update for 2 hours. The last reported price is `0.998e18` (stETH slightly below peg).
2. The true market price recovers to `1.000e18`, but the on-chain feed still returns `0.998e18`.
3. Anyone calls `LRTOracle.updateRSETHPrice()` → `_getTotalEthInProtocol()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns `0.998e18`.
4. TVL is understated by 0.2% → `rsETHPrice` is stored 0.2% too low (assuming the deviation is within `pricePercentageLimit`).
5. A depositor calls `LRTDepositPool.depositAsset(stETH, amount, ...)` → `getRsETHAmountToMint` computes `(amount * 0.998e18) / rsETHPrice_stale` where `rsETHPrice_stale` is ~0.2% below true value → depositor receives ~0.2% more rsETH than entitled.
6. When the feed updates and `rsETHPrice` corrects upward, all pre-existing rsETH holders have been diluted by the over-minted tokens.

**Foundry fork test plan:**
- Fork mainnet at a block where a Chainlink LST/ETH feed is near its heartbeat boundary.
- Mock `latestRoundData()` to return a price with `updatedAt = block.timestamp - 2 hours`.
- Call `updateRSETHPrice()` and record `rsETHPrice`.
- Call `depositAsset` and record rsETH minted.
- Advance time, allow the feed to "update" to the true price, call `updateRSETHPrice()` again.
- Assert that the rsETH minted in step 3 exceeds what would have been minted at the true price, and that existing holders' share value decreased.

    function _getTotalEthInProtocol() private view returns (uint256 totalETHInProtocol) {
        address lrtDepositPoolAddr = lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL);
        address[] memory supportedAssets = lrtConfig.getSupportedAssetList();
        uint256 supportedAssetCount = supportedAssets.length;

        for (uint16 assetIdx; assetIdx < supportedAssetCount;) {
            address asset = supportedAssets[assetIdx];
            // assetER is in 1e18 precision (1.0 = 1e18)
            uint256 assetER = getAssetPrice(asset);
            // totalAssetAmt is in 1e18 precision (standard token decimals)
            uint256 totalAssetAmt = ILRTDepositPool(lrtDepositPoolAddr).getTotalAssetDeposits(asset);

            totalETHInProtocol += totalAssetAmt.mulWad(assetER);

            unchecked {
                ++assetIdx;
            }
        }
    }
