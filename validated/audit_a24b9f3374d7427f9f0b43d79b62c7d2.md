Audit Report

## Title
Uninitialized `maxNumberOfWithdrawalsPerUser` Causes `initiateWithdrawal()` to Always Revert — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool.initialize()` never assigns `maxNumberOfWithdrawalsPerUser`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal()` checks `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, which evaluates to `0 >= 0 == true` on every first call, unconditionally reverting with `WithdrawalLimitReached`. All stakers are blocked from unstaking until an admin manually calls `setMaxNumberOfWithdrawalsPerUser()`.

## Finding Description
`initialize()` sets `kernelToken`, `rewardsToken`, and the admin role, but never assigns `maxNumberOfWithdrawalsPerUser`:

```solidity
// contracts/KERNEL/KernelDepositPool.sol L259-271
function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
    // ... no assignment to maxNumberOfWithdrawalsPerUser
    kernelToken = IERC20(_kernelToken);
    rewardsToken = IERC20(_rewardToken);
}
```

`initiateWithdrawal()` contains:

```solidity
// L323
if (userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser) revert WithdrawalLimitReached();
```

With `maxNumberOfWithdrawalsPerUser == 0`, a fresh user with zero open withdrawals hits `0 >= 0 → true` and the call reverts. The setter `setMaxNumberOfWithdrawalsPerUser()` explicitly rejects `0` as an input, so the state cannot self-correct; only an admin call with a non-zero value unblocks the function. `stake()` has no dependency on `maxNumberOfWithdrawalsPerUser`, so the contract accepts deposits while silently blocking all exits.

## Impact Explanation
Any user who has staked via `stake()` or `stakeFor()` cannot recover their KERNEL tokens until an admin intervenes. This is a **temporary freezing of funds** (Medium) affecting all stakers from the moment of deployment until `setMaxNumberOfWithdrawalsPerUser()` is called.

## Likelihood Explanation
The missing initialization is not enforced or documented in `initialize()`. Because `stake()` works normally, an admin can deploy the contract, observe successful staking activity, and never notice the broken withdrawal path. Any user who stakes immediately after deployment and attempts to withdraw will be blocked. No special attacker capability is required — the condition is triggered by the normal user flow.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`:

```solidity
maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER; // 100
```

This mirrors the pattern used for `withdrawalDelayBlocks` in `LRTWithdrawalManager.initialize()` and ensures the withdrawal path is functional immediately after deployment.

## Proof of Concept
1. Deploy `KernelDepositPool` (admin never calls `setMaxNumberOfWithdrawalsPerUser`).
2. Alice calls `stake(100e18)` — succeeds; `balanceOf[Alice] = 100e18`.
3. Alice calls `initiateWithdrawal(100e18)`.
4. Execution reaches L323:
   ```solidity
   if (userWithdrawalIds[Alice].length >= maxNumberOfWithdrawalsPerUser)
   //  0 >= 0  → true
       revert WithdrawalLimitReached();
   ```
5. Transaction reverts. Alice's 100 KERNEL tokens are locked with no recourse.
