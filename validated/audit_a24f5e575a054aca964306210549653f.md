### Title
Malicious Harvester Can Unconditionally Redirect Farmer Block Rewards via Unenforced `farmer_reward_address_override` in `RespondSignatures` - (File: `chia/farmer/farmer_api.py`)

### Summary

The `_process_respond_signatures` function in `FarmerAPI` unconditionally accepts `farmer_reward_address_override` from any connected harvester's `RespondSignatures` message with zero authorization check or fee-quality enforcement. A malicious harvester can set this field to any arbitrary puzzle hash, causing the farmer's entire block reward (1.75 XCH on mainnet) to be sent to the attacker's address instead of the farmer's configured address. The CHIP-22 fee-quality convention that is supposed to bound how much a harvester may take is advisory-only: violations are only logged, never blocked.

### Finding Description

**Root cause — two separate missing guards:**

**Path 1 — `RespondSignatures` (no check whatsoever):**

In `_process_respond_signatures` (`chia/farmer/farmer_api.py`, lines 916–919), the farmer blindly replaces its own configured reward address with whatever the harvester supplies:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override   # ← no guard
    include_source_signature_data = True
```

The resulting `farmer_reward_address` is placed directly into `DeclareProofOfSpace` (line 929) and broadcast to the full node, which uses it as the puzzle hash for the farmer reward coin. The `farmer_reward_address_override` field is **not** part of the BLS-signed data, so the existing signature checks (`AugSchemeMPL.verify` at lines 886, 893) do not cover it.

**Path 2 — `NewProofOfSpace` (advisory log only, never blocked):**

When the harvester sends `NewProofOfSpace` with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` (`chia/farmer/farmer.py`, lines 888–934). That function computes a `fee_quality` from the proof bytes and compares it to the harvester-supplied `applied_fee_threshold`. If the threshold is violated it emits a `log.warning` — but **returns normally and does not abort the override**. The harvester can supply `applied_fee_threshold = 0xFFFFFFFF` to always pass the check, or simply omit `fee_info` entirely (the `else` branch at line 929 also only logs a warning).

**End-to-end exploit flow:**

1. Attacker operates a harvester and connects it to the victim farmer (normal, unprivileged operation).
2. Farmer sends `NewSignagePointHarvester`; harvester finds a block-winning proof and sends `NewProofOfSpace` (override may be `None`).
3. Farmer sends `RequestSignatures` back to the harvester.
4. Harvester responds with `RespondSignatures` where `farmer_reward_address_override = attacker_puzzle_hash`.
5. `_process_respond_signatures` accepts the override with no check; `farmer_reward_address` is set to the attacker's puzzle hash.
6. Farmer broadcasts `DeclareProofOfSpace` with the attacker's address as the reward destination.
7. Full node creates the block; the 1.75 XCH farmer reward coin is locked to the attacker's puzzle hash.
8. Farmer receives nothing.

### Impact Explanation

**High — unauthorized payout redirection of XCH block rewards.**

Every block the farmer wins while the malicious harvester is connected can have its 1.75 XCH farmer reward silently redirected to any address the attacker chooses. The farmer has no on-node mechanism to detect or prevent this; the only signal is an optional log warning that is never emitted for the `RespondSignatures` path. This directly matches the allowed High impact: "Bypass of … authorization that enables … payout redirection … affecting XCH."

### Likelihood Explanation

Any farmer that uses a third-party harvester service (DrPlotter, GPU harvesters, remote harvesters) is exposed. The attacker only needs a valid TLS connection to the farmer's harvester port — the same connection used for legitimate plot-serving. No keys, no admin access, and no cryptographic break are required. The attack is silent and per-block.

### Recommendation

1. **Enforce the fee-quality check as a hard gate.** In `notify_farmer_reward_taken_by_harvester_as_fee`, return a boolean indicating whether the override is acceptable. In `new_proof_of_space`, abort (return `None`) if the check fails instead of proceeding.

2. **Apply the same gate in `_process_respond_signatures`.** Before accepting `response.farmer_reward_address_override`, verify that the corresponding `NewProofOfSpace` also carried a valid, passing `fee_info`, and that the override address matches. If not, ignore the override and use `self.farmer.farmer_target`.

3. **Include `farmer_reward_address_override` in the signed payload** so that the BLS signature covers the redirected address, making forgery cryptographically impossible.

4. **Allow farmers to configure a maximum fee rate** and reject any override that exceeds it.

### Proof of Concept

The `RespondSignatures` protocol message is defined with `farmer_reward_address_override: bytes32 | None`: [1](#0-0) 

The farmer's `_process_respond_signatures` unconditionally substitutes the attacker-supplied address with no guard: [2](#0-1) 

That address is then placed into `DeclareProofOfSpace` and sent to the full node: [3](#0-2) 

The `NewProofOfSpace` path calls `notify_farmer_reward_taken_by_harvester_as_fee`, which only logs and never blocks: [4](#0-3) 

The fee-quality check is advisory: a threshold violation produces a `log.warning` but execution continues and the override is still applied: [5](#0-4) 

A minimal exploit: connect a harvester, wait for a block-winning proof, return a `RespondSignatures` with `farmer_reward_address_override` set to the attacker's puzzle hash. The farmer will broadcast `DeclareProofOfSpace` with the attacker's address, and the full node will create the reward coin locked to that address. The existing test in `test_third_party_harvesters.py` already demonstrates that the farmer respects this field end-to-end: [6](#0-5)
