Audit Report

## Title
Unvalidated Chainlink `latestRoundData()` Return Values Enable Stale-Price rsETH Over-Minting â€” (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` and silently discards all validity metadata (`roundId`, `startedAt`, `updatedAt`, `answeredInRound`), using only the raw `answer` value. A stale inflated price causes `LRTDepositPool.getRsETHAmountToMint()` to issue excess rsETH to a depositor, diluting all existing rsETH holders. A zero price causes deposited assets to be accepted with zero rsETH minted, temporarily freezing the depositor's funds.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` at line 52 of `contracts/oracles/ChainlinkPriceOracle.sol` discards all five return values except `answer`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is performed for: staleness (`block.timestamp - updatedAt > threshold`), incomplete round (`updatedAt == 0`), invalid answer (`price <= 0`), or round consistency (`answeredInRound >= roundId`).

The sister contract `ChainlinkOracleForRSETHPoolCollateral.getRate()` (lines 27â€“32 of `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`) performs all three checks explicitly, demonstrating the protocol's awareness of the requirement and its intentional omission here.

The unvalidated price flows directly into `LRTDepositPool.getRsETHAmountToMint()` (line 520 of `contracts/LRTDepositPool.sol`):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Stale-price path:** `lrtOracle.rsETHPrice()` is computed as total protocol ETH value divided by total rsETH supply. If only one asset's feed is stale (e.g., stETH/ETH inflated 5%), and that asset represents less than 100% of TVL, the denominator (`rsETHPrice`) is inflated by a smaller proportion than the numerator (`getAssetPrice(stETH)`). The ratio is net-positive for the depositor, yielding excess rsETH at the expense of existing holders.

**Zero-price path:** If `price == 0`, `getAssetPrice()` returns `0`, making `rsethAmountToMint = 0`. The `_beforeDeposit` slippage guard (`rsethAmountToMint < minRSETHAmountExpected`) only reverts if the caller passes a non-zero minimum. A caller passing `minRSETHAmountExpected = 0` (a valid input) will have their assets transferred in at line 114 with zero rsETH minted, freezing their funds until the oracle recovers.

`RSETHPriceFeed.latestRoundData()` (lines 68â€“69 of `contracts/oracles/RSETHPriceFeed.sol`) also forwards the raw ETH/USD answer without any validity checks before multiplying it into the rsETH/USD price exposed to external consumers.

## Impact Explanation
**High â€” Theft of unclaimed yield:** When a Chainlink feed misses its heartbeat and reports a stale inflated price, any depositor calling `depositAsset()` receives more rsETH than the deposited asset is currently worth. This dilutes the proportional claim of all existing rsETH holders on the underlying assets, constituting theft of their accrued yield.

**Medium â€” Temporary freezing of funds:** If a feed returns `price == 0` and the depositor passes `minRSETHAmountExpected = 0`, assets are transferred into the pool with zero rsETH minted. The depositor's funds are locked until the oracle returns a valid price and a recovery mechanism is invoked.

## Likelihood Explanation
Chainlink heartbeat misses are documented real-world events (e.g., during the LUNA collapse and ETH flash crashes). No special permissions, governance access, or private keys are required. Any unprivileged user calling the public `depositAsset()` function during a heartbeat miss triggers the stale-price path. The zero-price path requires a feed returning zero, which is less common but possible for newly configured or deprecated feeds. Both paths are reachable by any external caller with no preconditions beyond normal protocol usage.

## Recommendation
Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral.getRate()` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
    priceFeed.latestRoundData();
require(answeredInRound >= roundId, "Stale price");
require(updatedAt != 0, "Incomplete round");
require(price > 0, "Invalid price");
require(block.timestamp - updatedAt <= MAX_STALENESS, "Price too old");
```

Apply equivalent checks to `RSETHPriceFeed.latestRoundData()` and `RSETHPriceFeed.getRoundData()` before forwarding the ETH/USD answer to external consumers.

## Proof of Concept
1. Deploy a mock Chainlink aggregator for stETH/ETH that returns a fixed stale price (e.g., 1.05e18) with `updatedAt` set to `block.timestamp - 3 hours`.
2. Configure `ChainlinkPriceOracle` to use this mock feed for stETH.
3. Call `LRTDepositPool.depositAsset(stETH, 1000e18, 0)` as an unprivileged user.
4. Observe that `getRsETHAmountToMint()` calls `ChainlinkPriceOracle.getAssetPrice(stETH)` and returns the stale inflated price without reverting.
5. Verify the caller receives rsETH in excess of what the current fair price would yield, while existing rsETH holders' redemption value decreases proportionally.

Foundry fork test plan: fork mainnet, warp `block.timestamp` forward past the Chainlink heartbeat threshold without triggering an oracle update, call `depositAsset()`, and assert that the minted rsETH exceeds the fair-value amount computed using the current market price.
