Audit Report

## Title
Zero wrsETH Minted for Dust ETH/Token Deposits Due to Integer Division Truncation in `viewSwapRsETHAmountAndFee` - (File: contracts/pools/RSETHPoolV2.sol, RSETHPoolV3.sol, RSETHPool.sol, RSETHPoolNoWrapper.sol, RSETHPoolV2ExternalBridge.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol)

## Summary

Every L2 RSETHPool variant computes the wrsETH output via integer division `amountAfterFee * 1e18 / rsETHToETHrate`. When the deposit amount is small enough that this expression truncates to zero, the pool accepts the depositor's ETH or token but mints (or transfers) zero wrsETH in return. The depositor permanently loses their funds with no recourse, as no minimum-output guard exists in any pool's `deposit` function.

## Finding Description

In every pool contract under `contracts/pools/`, `viewSwapRsETHAmountAndFee` computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Because `rsETHToETHrate` is always ≥ 1e18 and grows over time, any deposit where `amountAfterFee * 1e18 < rsETHToETHrate` yields `rsETHAmount = 0` due to Solidity's truncating integer division.

The `deposit` functions across all variants guard only against `amount == 0` but never check whether the computed `rsETHAmount` is non-zero before proceeding:

**RSETHPoolV2.sol** — minting path:
```solidity
if (amount == 0) revert InvalidAmount();
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // rsETHAmount can be 0
```

**RSETHPool.sol / RSETHPoolNoWrapper.sol** — transfer path:
```solidity
if (amount == 0) revert InvalidAmount();
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
feeEarnedInETH += fee;
IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount);  // rsETHAmount can be 0
```

In both cases the ETH (or token) is accepted by the pool before the output amount is computed, and a zero-amount ERC20 mint or transfer succeeds silently under standard OpenZeppelin implementations. The same truncation applies to the token-deposit overload via `amountAfterFee * tokenToETHRate / rsETHToETHrate`.

No existing guard in any pool variant catches this condition. The `limitDailyMint` modifier checks only that the cumulative minted amount does not exceed a daily cap; it does not enforce a non-zero per-transaction output.

## Impact Explanation

A depositor who sends a dust ETH or token amount (any amount where `amountAfterFee * 1e18 < rsETHToETHrate`) has their funds accepted by the pool but receives zero wrsETH. The ETH remains in the pool, accruing to existing wrsETH holders. The depositor has no mechanism to recover the lost funds. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value** from the allowed impact scope: the protocol retains the deposited value while the depositor receives nothing.

## Likelihood Explanation

The truncation threshold is currently small (for a rate of 1.05e18, only a 1-wei ETH deposit truncates to zero), but the condition is reachable by any unprivileged external caller with no preconditions beyond calling `deposit`. Dust deposits can arise from scripted integrations, wallet rounding errors, or direct contract calls. The pool contracts impose no minimum deposit amount, making this reachable by any external caller at any time. As the rsETH rate appreciates further, the threshold remains at 1 wei for ETH-denominated pools (since `1 * 1e18 / rate` truncates to 0 for any rate > 1e18), so the condition is permanently present.

## Recommendation

Add a zero-check on the computed `rsETHAmount` immediately after calling `viewSwapRsETHAmountAndFee`, and revert if it is zero, in every pool variant's `deposit` function:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount);
if (rsETHAmount == 0) revert InvalidAmount();
```

This mirrors the protection in `LRTDepositPool._beforeDeposit`, which reverts when `rsethAmountToMint < minRSETHAmountExpected`.

## Proof of Concept

Assume `rsETHToETHrate = 1.05e18` (rsETH has appreciated 5% since launch).

1. Any external caller invokes `RSETHPoolV2.deposit{value: 1}("")` (1 wei ETH).
2. `viewSwapRsETHAmountAndFee(1)` computes:
   - `fee = 1 * feeBps / 10_000 = 0` (for any `feeBps < 10_000`)
   - `amountAfterFee = 1`
   - `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (truncated)
3. `feeEarnedInETH += 0`.
4. `wrsETH.mint(msg.sender, 0)` — mints nothing; standard ERC20 mint of 0 succeeds.
5. The 1 wei ETH is held by the pool with no corresponding wrsETH issued.
6. The depositor has permanently lost 1 wei with no wrsETH received.

**Foundry fuzz test plan:**

```solidity
function testFuzz_depositZeroOutput(uint256 amount) public {
    uint256 rate = rsETHOracle.getRate(); // e.g. 1.05e18
    vm.assume(amount > 0 && amount * 1e18 < rate); // truncation condition
    uint256 balanceBefore = wrsETH.balanceOf(address(this));
    pool.deposit{value: amount}("");
    uint256 balanceAfter = wrsETH.balanceOf(address(this));
    assertEq(balanceAfter, balanceBefore); // received 0 wrsETH
    // ETH was consumed by the pool
    assertEq(address(pool).balance, amount);
}
```
