### Title
Removed Participant's TEE Governance Votes Persist Permanently After Resharing, Counting Toward Whitelist Thresholds — (File: `crates/contract/src/lib.rs`)

---

### Summary

When a participant is removed via resharing, `vote_reshared` spawns `clean_tee_status` as a **detached** (fire-and-forget) promise to purge stale TEE governance votes. Because `clean_tee_status` is `#[private]` (callable only by the contract itself), if this promise fails there is **no recovery path**. The stale votes from the removed participant remain permanently in `tee_state.votes`, `tee_state.launcher_votes`, and `tee_state.measurement_votes`, and are counted at face value by `vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, and `vote_remove_os_measurement`. This is the direct analog of the external report: a removed actor's accumulated state is left untouched and continues to exercise governance power.

---

### Finding Description

**Root cause — detached cleanup with no fallback**

In `vote_reshared`, after the protocol transitions to the new `Running` state, six cleanup promises are spawned with `.detach()`:

```rust
Promise::new(env::current_account_id())
    .function_call(
        method_names::CLEAN_TEE_STATUS.to_string(),
        vec![],
        NearToken::from_yoctonear(0),
        Gas::from_tgas(self.config.clean_tee_status_tera_gas),
    )
    .detach();
``` [1](#0-0) 

`clean_tee_status` is marked `#[private]`, so it can only be invoked as a self-call from the contract. If the detached receipt runs out of gas or panics, the failure is silent and **no external caller can re-trigger the cleanup**:

```rust
#[private]
#[handle_result]
pub fn clean_tee_status(&mut self) -> Result<(), Error> {
    ...
    self.tee_state.clean_non_participant_votes(participants);
    Ok(())
}
``` [2](#0-1) 

`clean_non_participant_votes` removes stale entries from all three vote maps:

```rust
pub fn clean_non_participant_votes(&mut self, participants: &Participants) {
    self.votes = self.votes.get_remaining_votes(participants);
    self.launcher_votes = self.launcher_votes.get_remaining_votes(participants);
    self.measurement_votes = self.measurement_votes.get_remaining_votes(participants);
}
``` [3](#0-2) 

**Stale votes count at governance call time — no inline filtering**

`vote_code_hash` reads the raw vote count and compares it directly to the threshold:

```rust
let votes = self.tee_state.vote(code_hash, &participant);
...
if votes >= self.threshold()?.value() {
    self.tee_state.whitelist_tee_proposal(code_hash, ...);
}
``` [4](#0-3) 

`vote_add_launcher_hash` does the same:

```rust
let votes = self.tee_state.vote_launcher(action, &participant);
...
if votes >= self.threshold()?.value() {
    let added = self.tee_state.add_launcher_image(launcher_hash, ...);
}
``` [5](#0-4) 

`vote_remove_launcher_hash` uses unanimity but also counts raw votes:

```rust
let votes = self.tee_state.vote_launcher(action, &participant);
let total_participants = threshold_parameters.participants().len() as u64;
if votes >= total_participants {
    let removed = self.tee_state.remove_launcher_image(&launcher_hash);
}
``` [6](#0-5) 

None of these functions filter non-participant votes at counting time. The codebase's own test documents the risk explicitly:

> *"Stale CodeHashesVotes entries from removed participants must not count toward quorum after resharing."* [7](#0-6) 

**Contrast with `vote_update`**

`vote_update` was retrofitted with inline participant filtering precisely because the cleanup promise can fail:

```rust
// Filter votes to only count current participants voting for this specific update.
// This ensures correctness even if the cleanup promise in MpcContract::vote_reshared() fails.
let valid_votes_count = running_state
    .parameters
    .participants()
    .participants()
    .iter()
    .filter(|(account_id, _, _)| {
        self.proposed_updates
            .vote_by_participant
            .get(account_id)
            .is_some_and(|voted_id| *voted_id == id)
    })
    .count();
``` [8](#0-7) 

No equivalent guard exists for any of the TEE governance vote functions.

### Impact Explanation

**Scenario A — malicious launcher hash whitelisted with fewer legitimate votes**

1. Participant P1 (later found to be malicious) calls `vote_add_launcher_hash(malicious_hash)` before resharing.
2. Resharing removes P1. The `clean_tee_status` detached promise fails (e.g., gas exhaustion).
3. P1's vote for `malicious_hash` remains in `tee_state.launcher_votes`.
4. Only `threshold − 1` remaining participants need to call `vote_add_launcher_hash(malicious_hash)` for the threshold to be reached.
5. `malicious_hash` is added to the allowed launcher set.
6. A compromised node running that launcher passes `verify_tee` attestation and joins the signing network.

This constitutes an **attestation authorization bypass**: a TEE image that should not be trusted is whitelisted because a removed participant's vote was silently counted.

**Scenario B — legitimate launcher hash removed with fewer legitimate votes**

For `vote_remove_launcher_hash` (unanimity), after resharing drops one participant, the new `total_participants` is `N−1`. The removed participant's stale vote counts as 1, so only `N−2` of the `N−1` remaining participants need to vote. Legitimate nodes running the targeted launcher are then kicked out of the network.

The impact maps to: **Medium — participant-state manipulation that breaks production safety/accounting invariants** (stale participant state persists and influences TEE governance outcomes). It could escalate to **High** (attestation authorization bypass enabling invalid bridge execution) if the whitelisted malicious launcher allows a compromised node to co-sign foreign-chain transactions.

---

### Likelihood Explanation

- The `clean_tee_status` gas budget is a configurable constant (`clean_tee_status_tera_gas`, default 5 TGas). For large participant sets or future storage growth, this budget can be insufficient, causing the promise to fail silently. [9](#0-8) 
- Because `clean_tee_status` is `#[private]`, no participant or external account can re-trigger it after a failure. The stale votes are permanent.
- A removed participant need only have cast a TEE governance vote before removal — a normal operational action — for the stale vote to persist.
- The codebase already acknowledges this failure mode for `vote_update` and added inline filtering as a mitigation; the same mitigation is absent for TEE governance votes.

---

### Recommendation

Apply the same inline-filtering pattern used in `vote_update` to all TEE governance vote functions (`vote_code_hash`, `vote_add_launcher_hash`, `vote_remove_launcher_hash`, `vote_add_os_measurement`, `vote_remove_os_measurement`). At counting time, filter the raw vote count to include only votes from accounts that are current participants:

```rust
// Example for vote_add_launcher_hash:
let raw_votes = self.tee_state.vote_launcher(action, &participant);
let valid_votes = self.tee_state
    .launcher_votes
    .count_for_participants(threshold_parameters.participants());
if valid_votes >= self.threshold()?.value() { ... }
```

Alternatively, make `clean_tee_status` callable by current participants (remove `#[private]`, add participant authentication) so that a failed detached promise can be retried — matching the design of `remove_non_participant_update_votes`. [10](#0-9)

### Proof of Concept

**State before resharing (N=5, T=3):**
- P1 calls `vote_add_launcher_hash(malicious_hash)` → 1 vote stored in `tee_state.launcher_votes`.
- P2 calls `vote_add_launcher_hash(malicious_hash)` → 2 votes stored.

**Resharing removes P1 and P2. New set: {P3, P4, P5}, T=3.**

**`clean_tee_status` detached promise fails (gas exhaustion).**

**State after resharing:**
- `tee_state.launcher_votes` still contains P1's and P2's votes for `malicious_hash`.
- `threshold_parameters.participants()` now returns {P3, P4, P5}.

**P3 calls `vote_add_launcher_hash(malicious_hash)`:**
```rust
let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
// ✓ P3 is a current participant — passes
let action = LauncherVoteAction::Add(malicious_hash);
let votes = self.tee_state.vote_launcher(action, &participant);
// votes = 3 (P1_stale + P2_stale + P3_fresh)
if votes >= self.threshold()?.value() {  // 3 >= 3 → TRUE
    self.tee_state.add_launcher_image(malicious_hash, ...);
    // malicious_hash is now whitelisted
}
``` [11](#0-10) 

`malicious_hash` is added to the allowed launcher set with only **1 legitimate vote** (P3), because P1's and P2's stale votes — from participants removed for malicious behavior — were never purged.

    /// Can only be called by participants or by the contract itself.
    #[handle_result]
    pub fn remove_non_participant_update_votes(&mut self) -> Result<(), Error> {
        log!(
            "remove_non_participant_update_votes: signer={}",
            env::signer_account_id()
        );

        let participants = match &self.protocol_state {
            ProtocolContractState::Running(state) => state.parameters.participants(),
            _ => {
                return Err(InvalidState::ProtocolStateNotRunning.into());
            }
        };

        // Authorize the caller: allow self-calls (the cleanup promise spawned after a
        // successful resharing, where the predecessor is the contract account) and
        // direct calls from a current participant. Reject everyone else so that
        // non-participants cannot drive this cleanup.
        let caller = env::predecessor_account_id();
        let is_self_call = caller == env::current_account_id();
        if !is_self_call && !participants.is_participant_given_account_id(&caller) {
            return Err(InvalidState::NotParticipant { account_id: caller }.into());
        }

        self.proposed_updates
            .remove_non_participant_votes(participants);
        Ok(())
