### Title
`EvmFinality::Latest` Accepted Without Reorg-Safety Check Enables Signing of Unfinalized Foreign-Chain Transactions - (File: `crates/foreign-chain-inspector/src/evm/inspector.rs`)

### Summary

The `EvmInspector::verify_finality_level` function accepts `EthereumFinality::Latest` as a valid finality level and performs only a trivial block-number comparison. The on-chain contract's `verify_foreign_transaction` entry point does not reject requests carrying `EvmFinality::Latest`. As a result, any unprivileged caller can obtain a valid MPC-signed attestation for an EVM transaction that exists only in the latest (unfinalized) block, which may be reorganized away. This is the direct analog of the Chainlink `latestRoundData` staleness issue: both involve consuming external chain state without verifying that the data has reached a stable, trustworthy finality level.

### Finding Description

`EvmInspector::verify_finality_level` maps the caller-supplied finality level directly to an Ethereum tag and fetches the corresponding head block:

```rust
let finality_tag = match finality {
    EthereumFinality::Finalized => FinalityTag::Finalized,
    EthereumFinality::Safe      => FinalityTag::Safe,
    EthereumFinality::Latest    => FinalityTag::Latest,   // ← no guard
};
let head: GetBlockByNumberResponse = self
    .client
    .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
    .await?;

if head.number < receipt_block_number {
    return Err(ForeignChainInspectionError::NotFinalized);
}
Ok(())
```

When `Latest` is supplied, `eth_getBlockByNumber("latest")` always returns a block number ≥ any receipt that already exists, so the check `head.number < receipt_block_number` is trivially false and the function returns `Ok(())` unconditionally. No additional reorg-safety window, confirmation count, or timestamp bound is applied.

The on-chain contract's `verify_foreign_transaction` entry point accepts the full `EvmFinality` enum (including `Latest`) without filtering:

```rust
pub fn verify_foreign_transaction(&mut self, request: VerifyForeignTransactionRequestArgs) {
    // checks domain purpose, chain whitelist, gas/deposit — but NOT finality level
    ...
    self.enqueue_yield_request(...);
}
```

The signed payload (`ForeignTxSignPayloadV1`) contains only `request` and `values` — no observation timestamp or block height is committed to the signature:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,   // includes EvmFinality::Latest
    pub values: Vec<ExtractedValue>,
}
```

### Impact Explanation

The primary stated use-case of `verify_foreign_transaction` is the Omnibridge inbound flow, where the MPC network attests that a foreign-chain transaction "finalized successfully." A signed attestation carrying `EvmFinality::Latest` is indistinguishable (from the NEAR contract's perspective) from one carrying `EvmFinality::Finalized` — both produce the same `VerifyForeignTransactionResponse { payload_hash, signature }` structure. A bridge contract that does not independently re-inspect the `finality` field embedded in the payload hash will accept a `Latest`-finality attestation as proof of finalization.

An attacker can:
1. Submit a bridge-deposit transaction on an EVM chain.
2. Immediately call `verify_foreign_transaction` with `EvmFinality::Latest` before the transaction is finalized.
3. Receive a valid MPC-signed attestation.
4. Redeem bridge funds on NEAR using the attestation.
5. Cause or wait for a chain reorganization that reverses the original deposit.
6. Retain both the redeemed NEAR-side funds and the reversed EVM-side deposit — a double-spend.

This matches the allowed High impact: "Cross-chain replay, forged foreign-chain verification, light-client-style verification bypass, or participant/attestation authorization bypass that causes invalid bridge execution or double-spend conditions."

### Likelihood Explanation

- The entry path is fully unprivileged: any account can call `verify_foreign_transaction` with `EvmFinality::Latest`.
- `EvmFinality::Latest` is a first-class enum variant exposed in the public ABI and SDK.
- Short-range reorganizations (1–2 blocks) occur naturally on Ethereum and are common on lower-security EVM chains (BNB, Polygon, etc.) that are explicitly supported.
- No threshold collusion, TEE bypass, or privileged access is required.

### Recommendation

1. **Reject `EvmFinality::Latest` at the contract level** in `verify_foreign_transaction`, or document it as explicitly unsafe and exclude it from bridge-facing domains.
2. **Add a minimum confirmation/depth guard** in `verify_finality_level` when `Latest` is used — analogous to Bitcoin's `BlockConfirmations` parameter — so the receipt block must be at least N blocks behind the current head.
3. **Commit the finality level and observation block height into the signed payload** so downstream verifiers can enforce their own minimum-finality policy without trusting the caller's request field alone.

### Proof of Concept

**Attacker-controlled entry path:**

```
verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    request: ForeignChainRpcRequest::Ethereum(EvmRpcRequest {
        tx_id: <attacker_deposit_tx>,
        extractors: vec![EvmExtractor::BlockHash],
        finality: EvmFinality::Latest,   // ← accepted, not filtered
    }),
    domain_id: <foreign_tx_domain>,
    payload_version: ForeignTxPayloadVersion::V1,

```

**Node-side execution path (no reorg guard):**

`VerifyForeignTxProvider::execute_foreign_chain_request`
→ `EvmInspector::extract`
→ `verify_finality_level(receipt_block_number, EthereumFinality::Latest)`
→ `eth_getBlockByNumber("latest")` → `head.number >= receipt_block_number` → `Ok(())`
→ `verify_block_is_canonical(...)` → passes (block exists at query time)
→ MPC signs `ForeignTxSignPayloadV1 { request, values }` and returns attestation

    /// head of the chain at `finality` is at or past `receipt_block_number`.
    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );

            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }
