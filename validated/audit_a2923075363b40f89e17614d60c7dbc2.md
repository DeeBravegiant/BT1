### Title
Dijkstra Era Sub-Transaction Double-Deposit-Refund: Missing Predicate Failure Allows Same Deposit to Be Claimed Multiple Times - (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxCert.hs`)

---

### Summary

In the Dijkstra era, `dijkstraTotalRefundsTxCerts` computes deposit refunds purely from the values embedded in `UnRegDepositTxCert` certificates, without consulting the actual ledger registration state. The protection against multiple sub-transactions claiming the same deposit refund is explicitly acknowledged as unimplemented (`xit` test with `error "TODO: predicate failure not yet implemented"`). An unprivileged transaction sender can include `UnRegDepositTxCert` for the same credential in two or more sub-transactions of a single top-level transaction, causing the deposit pot (`utxosDeposited`) to be decremented multiple times for a single deposit, creating ADA from nothing or corrupting the deposit accounting invariant.

---

### Finding Description

**Root cause — `dijkstraTotalRefundsTxCerts`** [1](#0-0) 

```haskell
-- Unlike previous eras, we no longer need to lookup refunds from the ledger state,
-- since all of the certificates specify the actual refund and ledger rules will
-- validate that they are accurate.
dijkstraTotalRefundsTxCerts ::
  ( Foldable f, ConwayEraTxCert era ) =>
  f (TxCert era) -> Coin
dijkstraTotalRefundsTxCerts = foldMap' $ \case
  UnRegDepositTxCert _ deposit -> deposit
  UnRegDRepTxCert  _ deposit -> deposit
  _ -> zero
```

The comment states "ledger rules will validate that they are accurate." However, the validation is not yet implemented. The function ignores both `lookupStakingDeposit` and `lookupDRepDeposit` entirely: [2](#0-1) 

```haskell
getTotalRefundsTxCerts _ _ _ = dijkstraTotalRefundsTxCerts
```

**Contrast with Conway**, which uses `lookupDeposit` to verify the credential is actually registered before counting the refund: [3](#0-2) 

**The deposit pot update path**

`updateUTxOStateNoFees` (used for every sub-transaction) calls `certsTotalRefundsTxBody`, which for Dijkstra resolves to `dijkstraTotalRefundsTxCerts`: [4](#0-3) 

```haskell
totalRefunds  = certsTotalRefundsTxBody pp certState txBody
totalDeposits = certsTotalDepositsTxBody pp certState txBody
depositChange = totalDeposits <-> totalRefunds
...
utxosDeposited = utxosDeposited <> depositChange
```

Each sub-transaction's refund is counted from the certificate value alone, not from the actual deposit pot entry.

**Sub-transactions are processed sequentially** [5](#0-4) 

```haskell
dijkstraSubLedgersTransition = do
  TRC (env, ledgerState, subTxs) <- judgmentContext
  foldM (\ls subTx -> trans @(EraRule "SUBLEDGER" era) $ TRC (env, ls, subTx))
        ledgerState subTxs
```

Each sub-transaction is applied to the state produced by the previous one. After subTx1 deregisters `stakingCred` and decrements `utxosDeposited` by `D`, subTx2 can present another `UnRegDepositTxCert stakingCred D`. Because `dijkstraTotalRefundsTxCerts` does not check registration state, it counts `D` again as a refund, decrementing `utxosDeposited` by `D` a second time.

**The missing predicate failure is explicitly acknowledged** [6](#0-5) 

```haskell
xit "Multiple subtransactions cannot get the same refund" $ do
  ...
  subTx1 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  subTx2 = ... & bodyTxL . certsTxBodyL .~ SSeq.singleton (UnRegDepositTxCert stakingCred keyDeposit)
  tx = mkBasicTx mkBasicTxBody & bodyTxL . subTransactionsTxBodyL .~ OMap.fromFoldable [subTx1, subTx2]
  submitFailingTx tx . NE.singleton $ error "TODO: predicate failure not yet implemented"
```

The test is disabled (`xit`) and the expected predicate failure is `error "TODO"`. This confirms the developers know the double-refund path exists and that the guard has not been implemented.

---

### Impact Explanation

If both sub-transactions succeed:

- `utxosDeposited` is decremented by `2 × D` while only `D` was ever deposited.
- The attacker's outputs across the two sub-transactions receive `2 × D` in value from the deposit pot.
- This constitutes direct creation of ADA through an invalid ledger state transition.
- Alternatively, if `utxosDeposited` underflows (it is backed by `Word64`), the deposit accounting invariant is permanently corrupted, requiring a hard fork to repair.

This matches: **Critical — Direct loss, creation, or destruction of ADA or native assets through an invalid ledger state transition**, and potentially **High — Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork** (if the deposit pot underflows).

---

### Likelihood Explanation

- No privileged access, governance majority, or special conditions are required.
- Any unprivileged transaction sender can register a stake credential, then submit a single top-level transaction containing two sub-transactions that both deregister the same credential.
- The attack is atomic and deterministic.
- **Likelihood: High.**

---

### Recommendation

1. **Implement the missing predicate failure** in the `SUBDELEG` rule: when processing `UnRegDepositTxCert` (or `DijkstraUnRegCert`) in a sub-transaction, explicitly check that the credential is still registered in the current `certState`. If it is not, emit a concrete predicate failure (not `error "TODO"`).

2. **Fix `dijkstraTotalRefundsTxCerts`** to use the `lookupStakingDeposit` / `lookupDRepDeposit` functions (as Conway does) rather than blindly summing certificate-embedded values. The comment "ledger rules will validate" must be backed by an actual rule check before this function can safely ignore the lookup.

3. **Enable and complete the disabled test** `"Multiple subtransactions cannot get the same refund"` in `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs`.

### Proof of Concept

```
1. Register stakingCred with deposit D (e.g., 2 ADA).
   utxosDeposited = D

2. Construct top-level Dijkstra transaction T with:
     subTx1 body: inputs=[UTxO1], certs=[UnRegDepositTxCert stakingCred D], outputs=[addr1: value1 + D]
     subTx2 body: inputs=[UTxO2], certs=[UnRegDepositTxCert stakingCred D], outputs=[addr2: value2 + D]

3. SUBLEDGERS processes subTx1:
   - SUBDELEG deregisters stakingCred (succeeds, credential was registered).
   - updateUTxOStateNoFees: dijkstraTotalRefundsTxCerts counts D from certificate.
   - depositChange = 0 - D = -D  →  utxosDeposited = 0
   - Value conservation: UTxO1 + D (refund) == outputs(subTx1) + 0 (deposits). ✓

4. SUBLEDGERS processes subTx2:
   - SUBDELEG attempts to deregister stakingCred again.
     * unregisterConwayAccount returns (Nothing, accounts) — credential already gone.
     * checkInvalidRefund: mAccountState = Nothing → no failure triggered.
     * checkStakeKeyHasZeroRewardBalance: mAccountState = Nothing → no failure triggered.
     * No predicate failure is emitted (TODO not implemented).
   - updateUTxOStateNoFees: dijkstraTotalRefundsTxCerts counts D again from certificate.
   - depositChange = 0 - D = -D  →  utxosDeposited = 0 - D  (underflow / negative)
   - Value conservation: UTxO2 + D (refund) == outputs(subTx2) + 0 (deposits). ✓

5. Result:
   - Attacker received D + D = 2D in refunds.
   - Only D was ever deposited.
   - Net gain: D ADA created from nothing.
   - utxosDeposited is corrupted (underflowed Word64 or negative).
```
