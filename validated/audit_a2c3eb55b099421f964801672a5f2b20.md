### Title
`respond_verify_foreign_tx` Does Not Bind `response.payload_hash` to the Submitted `request` — Cross-Request Replay Enables Forged Foreign-Chain Verification - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_verify_foreign_tx` verifies that `response.signature` is a valid ECDSA signature over `response.payload_hash`, but never checks that `response.payload_hash` is actually the hash of a payload derived from the submitted `request`. A malicious attested participant can replay a valid MPC signature produced for one foreign-chain transaction as the response to a completely different pending request, causing the contract to deliver a forged verification attestation to the original caller.

---

### Finding Description

The `respond_verify_foreign_tx` function in `crates/contract/src/lib.rs` performs the following checks:

1. Caller is an attested participant (`assert_caller_is_attested_participant_and_protocol_active`)
2. Protocol is running
3. `response.signature` is a valid ECDSA signature over `response.payload_hash` under the domain's root public key
4. `request` exists in `pending_verify_foreign_tx_requests` (via `resolve_yields_for`) [1](#0-0) 

What is **never checked** is whether `response.payload_hash` is the hash of `ForeignTxSignPayload { request: request.request, values: <observed_values> }`. The `payload_hash` is taken directly from the caller-supplied `response` struct and only verified for signature validity — not for binding to the pending request. [2](#0-1) 

The signed payload is defined as:

```
payload_hash = SHA-256(borsh(ForeignTxSignPayload { request, values }))
``` [3](#0-2) 

The `payload_hash` encodes both the original `ForeignChainRpcRequest` (tx_id, chain, extractors) **and** the extracted values observed from the foreign chain. The contract has no way to recompute this hash without the values, but it can and should at minimum verify that the hash is consistent with the `request` fields it already holds.

**Contrast with `respond()`** for regular signatures: there, the payload is taken directly from `request.payload` (which is part of the pending-request map key), so the contract inherently verifies the signature is over the correct payload — the payload cannot be substituted. [4](#0-3) 

In `respond_verify_foreign_tx`, the payload_hash is decoupled from the request key, creating the gap.

---

### Impact Explanation

**Impact: High** — Cross-chain replay / forged foreign-chain verification enabling invalid bridge execution.

An attested participant can call `respond_verify_foreign_tx` with:
- `request` = a legitimately pending request for tx_id **X** (passes the key lookup)
- `response.payload_hash` = the hash of `ForeignTxSignPayload { request_Y, values_Y }` for a **different** transaction Y
- `response.signature` = a valid MPC threshold signature over that hash (obtained from any prior legitimate signing operation, which is public on-chain)

The contract accepts this call, drains all yields queued under request X, and delivers the forged `VerifyForeignTransactionResponse` to every caller waiting on request X. [5](#0-4) 

The caller (e.g., an Omnibridge inbound contract) receives a response whose `payload_hash` attests to a **different** transaction than the one they submitted. If the caller trusts the contract's verification layer (the intended design), it will act on a forged attestation — enabling double-spend or unauthorized bridge execution.

The SDK's `ForeignChainSignatureVerifier::verify_signature` does perform the binding check on the client side: [6](#0-5) 

However, this is an off-chain SDK helper, not enforced by the contract. Callers that omit this check, or that trust the contract to have already validated the binding, are fully exposed.

---

### Likelihood Explanation

**Likelihood: Low** — requires a TEE-attested participant.

The attacker must be an active, TEE-attested participant in the MPC network to call `respond_verify_foreign_tx`. However, once that bar is cleared:

- No threshold collusion is required — a single attested participant can execute the replay
- The required valid signature is freely available: every prior `respond_verify_foreign_tx` call is public on-chain, and any of those signatures can be replayed
- The attacker does not need to compromise the MPC signing protocol itself — they only need to submit a previously observed `(payload_hash, signature)` pair against a new pending request [7](#0-6) 

---

### Recommendation

Embed the `ForeignChainRpcRequest` fields from the pending request into the payload hash verification. Since the contract cannot recompute the full `payload_hash` without the extracted `values`, the fix is to include the extracted values in the response and have the contract recompute and verify the hash:

```rust
// In respond_verify_foreign_tx, after receiving response:
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // add values to response DTO
});
let expected_hash = expected_payload.compute_msg_hash()?;
if expected_hash != response.payload_hash {
    return Err(RespondError::PayloadHashMismatch.into());
}
```

Alternatively, if adding `extracted_values` to the response DTO is undesirable due to size constraints, the `payload_hash` should be committed to the pending-request map at submission time (impossible since values are unknown at submission) — making the values-in-response approach the correct fix.

This mirrors the pattern already used in `respond()`, where the payload is part of the request key and cannot be substituted. [8](#0-7)

### Proof of Concept

**Setup:** Two pending `verify_foreign_transaction` requests exist:
- Request A: Bitcoin tx_id `[0xAA; 32]`, pending in `pending_verify_foreign_tx_requests`
- Request B: Bitcoin tx_id `[0xBB; 32]`, previously responded to with a valid MPC signature

**Steps:**

1. Attacker (attested participant) observes the on-chain `respond_verify_foreign_tx` call for request B, extracting `payload_hash_B` and `signature_B`.

2. Attacker calls `respond_verify_foreign_tx` with:
   ```json
   {
     "request": { "domain_id": ..., "payload_version": 1, "request": { "Bitcoin": { "tx_id": "0xAA...", ... } } },
     "response": { "payload_hash": "<payload_hash_B>", "signature": "<signature_B>" }
   }
   ```

3. Contract checks:
   - Caller is attested participant ✓
   - `signature_B` is valid over `payload_hash_B` under root public key ✓
   - Request A exists in `pending_verify_foreign_tx_requests` ✓

4. Contract calls `resolve_yields_for` with the forged response, delivering `payload_hash_B` (attesting to tx `[0xBB; 32]`) to the caller who submitted request A (tx `[0xAA; 32]`).

5. Bridge contract for request A receives a signed attestation for the wrong transaction and processes an unauthorized inbound transfer. [9](#0-8) [10](#0-9)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
