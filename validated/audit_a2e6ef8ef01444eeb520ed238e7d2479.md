Audit Report

## Title
Stale Chainlink Price Oracle Data Used Without Staleness Validation Causes Incorrect rsETH Price and Dilutes Existing Holders - (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt` and `answeredInRound`, performing no staleness, round-completeness, or positivity check on the returned price. A stale low price flows directly into `LRTOracle._getTotalEthInProtocol()`, which corrupts the stored `rsETHPrice`. Because `updateRSETHPrice()` is a public, permissionless function, any caller can commit the stale price and immediately deposit to receive excess rsETH, diluting all existing holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` fetches the price as:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

The five return values of `latestRoundData()` are `(roundId, answer, startedAt, updatedAt, answeredInRound)`. All values except `answer` are discarded. There is no check that:
- `price > 0` (price is positive),
- `answeredInRound >= roundId` (round was completed),
- `block.timestamp - updatedAt <= maxStaleness` (price is fresh).

`LRTOracle._getTotalEthInProtocol()` iterates every supported asset and calls `getAssetPrice(asset)` for each, accumulating the ETH-denominated TVL: [2](#0-1) 

The result feeds directly into `_updateRsETHPrice()`, which computes and stores `rsETHPrice`: [3](#0-2) [4](#0-3) 

`updateRSETHPrice()` is public and gated only by `whenNotPaused`: [5](#0-4) 

The stored `rsETHPrice` is then used by `LRTDepositPool.getRsETHAmountToMint()` to determine how many rsETH tokens to mint per deposit: [6](#0-5) 

**Existing guard insufficiency:** The downside protection in `_updateRsETHPrice()` only pauses the protocol if the price drop exceeds `pricePercentageLimit.mulWad(highestRsethPrice)`: [7](#0-6) 

This guard fails when `pricePercentageLimit == 0` (no limit configured), or when the stale price falls within the configured threshold (e.g., a stale price that is only 3% below the peak when the limit is 5%). In both cases the stale price is committed silently.

**Exploit path:**
1. A Chainlink LST/ETH feed (e.g., stETH/ETH) goes stale — its `updatedAt` is old but the last reported price is lower than the true current price.
2. Attacker calls the public `updateRSETHPrice()`. `_getTotalEthInProtocol()` uses the stale low price, understating `totalETHInProtocol`, and `rsETHPrice` is written to storage below the true rate.
3. Attacker immediately calls `depositETH()` or `depositAsset()`. Because `rsETHPrice` is artificially low, `getRsETHAmountToMint()` returns a larger rsETH amount than the depositor's ETH is actually worth.
4. Attacker receives excess rsETH, diluting all existing holders.

## Impact Explanation

**High — Theft of unclaimed yield.** When a stale low price is committed via the public `updateRSETHPrice()`, the stored `rsETHPrice` is understated. Every subsequent deposit mints excess rsETH, transferring accrued yield value from existing holders to the new depositor. The magnitude scales with deposit size and degree of price staleness. When the feed recovers and `rsETHPrice` is updated upward, the attacker's excess rsETH retains the full recovered value, permanently extracting yield from existing holders.

## Likelihood Explanation

**Medium.** Chainlink feeds can go stale during Ethereum network congestion, during low-volatility periods where the deviation threshold is not crossed, or if a feed is deprecated. The LST/ETH feeds used by this protocol (stETH/ETH, ETH-X/ETH) have historically experienced brief stale periods. No privileged access is required; any unprivileged external user can call `updateRSETHPrice()` followed by `depositETH()` or `depositAsset()`.

## Recommendation

Add staleness and validity checks inside `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price, , uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(price > 0, "Invalid price");
    require(answeredInRound >= roundId, "Stale round");
    require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`MAX_STALENESS` should be configured per-feed based on the feed's documented heartbeat interval (e.g., 3600 seconds for a 1-hour heartbeat feed).

## Proof of Concept

1. Protocol holds 10,000 stETH (true price 1.05 ETH each → true TVL = 10,500 ETH) and 10,000 rsETH in supply → true `rsETHPrice` = 1.05 ETH.
2. The stETH/ETH Chainlink feed goes stale at 1.00 ETH (a 4.76% drop, within a 5% `pricePercentageLimit`).
3. Attacker calls `updateRSETHPrice()`. `_getTotalEthInProtocol()` computes TVL = 10,000 ETH using the stale 1.00 price. `rsETHPrice` is stored as 1.00 ETH.
4. Attacker calls `depositETH{value: 1 ether}()`. `getRsETHAmountToMint` returns `1e18 * 1e18 / 1e18 = 1 rsETH`. True fair value would be `1e18 / 1.05e18 ≈ 0.952 rsETH`.
5. Attacker receives 1 rsETH instead of 0.952 rsETH — a ~5% excess — at the expense of all existing holders.
6. When the feed recovers and `updateRSETHPrice()` is called again, `rsETHPrice` rises back to ~1.05 ETH, but the attacker's excess rsETH retains that value.

**Foundry fork test plan:** Fork mainnet, mock the stETH/ETH Chainlink aggregator to return a stale `updatedAt` (e.g., `block.timestamp - 2 hours`) with a price of 1.00e18. Call `LRTOracle.updateRSETHPrice()` as an unprivileged address. Assert `rsETHPrice` is now 1.00e18. Call `LRTDepositPool.depositETH{value: 1 ether}()`. Assert the minted rsETH balance exceeds `1e18 / 1.05e18`, confirming excess minting.
