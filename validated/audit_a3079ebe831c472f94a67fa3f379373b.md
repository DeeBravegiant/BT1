Audit Report

## Title
Stale `rsETHPrice` Silently Propagated as Fresh by `RSETHPriceFeed` When Protocol Is Paused — (File: `contracts/oracles/RSETHPriceFeed.sol`)

## Summary
When `LRTOracle`'s downside-protection logic triggers a protocol pause on a large price drop, `rsETHPrice` is frozen at its pre-drop (inflated) value and cannot be refreshed because `updateRSETHPrice()` is gated by `whenNotPaused`. `RSETHPriceFeed.latestRoundData()` continues to serve the stale inflated rsETH/USD price while returning a fresh `updatedAt` sourced exclusively from the ETH/USD Chainlink feed, causing any integrated lending protocol to over-value rsETH collateral and enabling over-borrowing.

## Finding Description

**Root cause 1 — `RSETHPriceFeed` surfaces only the ETH/USD staleness timestamp:**

`latestRoundData()` at [1](#0-0)  fetches `updatedAt` entirely from `ETH_TO_USD.latestRoundData()`. The rsETH/ETH component — `RS_ETH_ORACLE.rsETHPrice()` — is a stored state variable in `LRTOracle` whose own update timestamp is never surfaced or considered.

**Root cause 2 — `updateRSETHPrice()` is blocked when the oracle is paused:** [2](#0-1)  gates the public price-update entry point behind `whenNotPaused`.

**Root cause 3 — Downside protection pauses and returns without writing the new price:** [3](#0-2)  calls `_pause()` and immediately `return`s. The assignment `rsETHPrice = newRsETHPrice` at [4](#0-3)  is never reached, so `rsETHPrice` remains at the pre-drop value.

**Why `updateRSETHPriceAsManager()` does not mitigate this:**

`updateRSETHPriceAsManager()` at [5](#0-4)  bypasses `whenNotPaused` but calls the same `_updateRsETHPrice()` internally. While the protocol is paused and the price is still below the threshold, `_updateRsETHPrice()` hits the same `isPriceDecreaseOffLimit` branch, calls `_pause()` (which is a no-op because `paused` is already `true` per [6](#0-5) ), and returns early — again without updating `rsETHPrice`. The price remains frozen until an admin unpauses **and** the price has recovered above the threshold.

**Exploit flow:**
1. A significant price drop causes `isPriceDecreaseOffLimit = true` in `_updateRsETHPrice()`.
2. `_pause()` is called; `rsETHPrice` stays at the pre-drop inflated value (e.g., `1.05 ETH`). True price is `0.95 ETH`.
3. All callers of `updateRSETHPrice()` revert (`whenNotPaused`). `updateRSETHPriceAsManager()` also cannot write the lower price while the threshold is still breached.
4. `RSETHPriceFeed.latestRoundData()` returns `answer = 1.05e18 * ETH_USD / 1e18` (inflated ~10%) with `updatedAt` = ETH/USD Chainlink timestamp (e.g., 2 minutes ago — fresh).
5. Lending protocol staleness check passes on `updatedAt`; rsETH is valued at the inflated price.
6. Attacker deposits rsETH, borrows the maximum against the inflated collateral value, and defaults.

## Impact Explanation
**Critical — Direct theft of user funds.** Lending protocol liquidity providers suffer direct, unrecoverable losses: the attacker borrows more than the true collateral value and walks away. For every 1000 rsETH deposited at a 10% inflated price, approximately 100 ETH worth of lender funds are stolen. `RSETHPriceFeed` is purpose-built as a Chainlink-compatible feed for external integrations, making consumption by lending protocols the intended and expected use case, not a hypothetical one.

## Likelihood Explanation
**Medium-High.** The trigger — a price drop exceeding `pricePercentageLimit` — is an explicitly designed circuit-breaker expected to fire during adverse market events (EigenLayer slashing, LST depeg). `pricePercentageLimit` is configurable; conservative settings (1–2%) can be triggered by routine volatility. The exploitation window lasts until an admin manually unpauses and the price recovers above the threshold, potentially hours. No privileged access is required by the attacker; any external user can call `latestRoundData()` and exploit the inflated price through an integrated lending protocol.

## Recommendation
1. **Surface the rsETH price update timestamp.** Store a `rsETHPriceUpdatedAt` timestamp in `LRTOracle` whenever `rsETHPrice` is written. In `RSETHPriceFeed.latestRoundData()`, return `min(ethUpdatedAt, rsETHPriceUpdatedAt)` as `updatedAt` so consumers see the true composite staleness.
2. **Write the lower price before pausing.** In `_updateRsETHPrice()`, assign `rsETHPrice = newRsETHPrice` before calling `_pause()` and `return`ing, so the stored value reflects the actual (lower) price even while the protocol is paused.
3. **Add a staleness guard in `RSETHPriceFeed`.** Revert if `rsETHPriceUpdatedAt` is older than a configurable threshold (e.g., 24 hours).

## Proof of Concept
**Minimal Foundry fork test outline:**

```solidity
// 1. Deploy/fork with rsETHPrice = 1.05e18, ETH/USD feed live.
// 2. Manipulate underlying asset prices so newRsETHPrice = 0.95e18
//    (> pricePercentageLimit drop from highestRsethPrice).
// 3. Call lrtOracle.updateRSETHPrice() — triggers pause, returns early.
//    Assert: lrtOracle.rsETHPrice() == 1.05e18 (unchanged).
//    Assert: lrtOracle.paused() == true.
// 4. Call lrtOracle.updateRSETHPrice() again — reverts (whenNotPaused).
// 5. Call rsETHPriceFeed.latestRoundData():
//    Assert: answer == 1.05e18 * ethUsdPrice / 1e18  (inflated).
//    Assert: updatedAt >= block.timestamp - 120  (appears fresh).
// 6. Simulate lending protocol borrow at inflated price:
//    borrowCapacity = collateral * inflatedPrice / borrowAssetPrice
//    Assert: borrowCapacity > trueCollateralValue (over-borrowing confirmed).
```

The sequence is fully reproducible on a local fork with no privileged attacker actions required — only a public call to `latestRoundData()` after the pause is triggered by normal price movement.
