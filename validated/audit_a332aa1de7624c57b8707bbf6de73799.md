Audit Report

## Title
Stale `rsETHPrice` Masked by Fresh ETH/USD `updatedAt` in `RSETHPriceFeed` — (`contracts/oracles/RSETHPriceFeed.sol`)

## Summary

`RSETHPriceFeed.latestRoundData()` and `getRoundData()` return `updatedAt` sourced entirely from the ETH/USD Chainlink feed, while `answer` is computed using `RS_ETH_ORACLE.rsETHPrice()` — a storage variable in `LRTOracle` updated independently via a keeper-driven `updateRSETHPrice()`. No timestamp is stored alongside `rsETHPrice`, and no staleness check exists for the rsETH component. Any downstream protocol relying on `updatedAt` for staleness detection will observe a fresh timestamp while silently consuming an arbitrarily old rsETH/ETH rate, enabling incorrect liquidation decisions that can temporarily freeze or misallocate user funds.

## Finding Description

In `RSETHPriceFeed.latestRoundData()`:

```solidity
// contracts/oracles/RSETHPriceFeed.sol L68-69
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

`updatedAt` is taken verbatim from the ETH/USD Chainlink feed. The `answer` is the product of two independent values:

- `ETH_TO_USD` price — updated by Chainlink on its own heartbeat (~1 h on mainnet).
- `RS_ETH_ORACLE.rsETHPrice()` — `LRTOracle.rsETHPrice`, a plain `uint256` storage variable:

```solidity
// contracts/LRTOracle.sol L28
uint256 public override rsETHPrice;
```

This variable is only updated when `_updateRsETHPrice()` is called, which is gated behind `whenNotPaused`:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

`_updateRsETHPrice()` stores only the new price at line 313 (`rsETHPrice = newRsETHPrice`) — no timestamp is recorded alongside it. The `IRSETHOracle` interface exposed to `RSETHPriceFeed` declares only `rsETHPrice()` with no timestamp accessor. A grep across the entire repository confirms no `rsETHPriceUpdatedAt` or equivalent variable exists anywhere.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` explicitly validates `answeredInRound < roundID`, `timestamp == 0`, and `ethPrice <= 0` before trusting the Chainlink answer. `RSETHPriceFeed` applies none of these guards to the rsETH component and adds no equivalent guard of its own.

**Exploit path:**
1. Keeper stops calling `updateRSETHPrice()` (outage, network congestion) — or `LRTOracle` is paused (blocking `updateRSETHPrice()` via `whenNotPaused`).
2. `rsETHPrice` freezes at its last stored value; ETH/USD Chainlink continues updating normally.
3. A downstream lending protocol calls `RSETHPriceFeed.latestRoundData()`.
4. It receives a fresh `updatedAt` (driven by the ETH/USD heartbeat) and passes its own staleness check.
5. It acts on `answer`, which embeds an hours- or days-old rsETH/ETH rate.
6. Depending on the direction of drift: valid liquidations are blocked (collateral over-valued) or borrowers are liquidated at an incorrect rate (collateral under-valued).

No attacker action is required; the divergence grows passively and silently.

## Impact Explanation

**Medium — Temporary freezing of funds.**

- **Stale-high rsETH price** (rsETH has depreciated but `rsETHPrice` was not updated): valid liquidations are blocked because collateral appears over-valued. Lenders cannot recover principal through liquidation until the price is corrected — a temporary freeze of lender funds.
- **Stale-low rsETH price** (rsETH has appreciated but `rsETHPrice` was not updated): borrowers are liquidated at an incorrect rate, losing collateral they should have retained — a temporary, partial loss of user funds.

Both scenarios map to **Medium — Temporary freezing of funds** in the allowed impact scope. The `RSETHPriceFeed` contract explicitly implements `AggregatorV3Interface` and is designed for downstream consumption; the downstream lending-market scenario is the contract's intended use case, not missing external context.

## Likelihood Explanation

`updateRSETHPrice()` is `public` but keeper-driven in practice. Keeper outages, network congestion, or a paused `LRTOracle` (which blocks `updateRSETHPrice()` via `whenNotPaused`) are realistic, non-collusive operational events. The ETH/USD Chainlink feed continues updating independently during any such outage, so the divergence between `updatedAt` and the actual rsETH price age grows silently and unboundedly. No on-chain mechanism alerts consumers or reverts the call. The condition is passively triggered — no privileged access or attacker coordination is required.

## Recommendation

1. **Store a `rsETHPriceUpdatedAt` timestamp** in `LRTOracle` alongside `rsETHPrice`, set to `block.timestamp` inside `_updateRsETHPrice()`.
2. **Expose it through `IRSETHOracle`** and read it in `RSETHPriceFeed`.
3. **Return the minimum of the two `updatedAt` values** so that `updatedAt` reflects the staleness of the composite price:

```solidity
uint256 rsEthUpdatedAt = RS_ETH_ORACLE.rsETHPriceUpdatedAt();
updatedAt = updatedAt < rsEthUpdatedAt ? updatedAt : rsEthUpdatedAt;
```

4. Optionally add a revert guard (mirroring `ChainlinkOracleForRSETHPoolCollateral`) if `rsEthUpdatedAt` is zero or older than a configurable heartbeat.

function test_staleRsEthPriceMaskedByFreshUpdatedAt() public {
    (, int256 initialAnswer,,uint256 initialUpdatedAt,) = rsEthPriceFeed.latestRoundData();

    assertEq(answer, initialAnswer);

// Fuzz variant: advance by arbitrary delta, assert answer never changes
// while updatedAt tracks block.timestamp — proving unbounded divergence.
function testFuzz_unboundedDivergence(uint32 delta) public {
    vm.assume(delta > 0);
    (, int256 initialAnswer,,,) = rsEthPriceFeed.latestRoundData();

    mockEthUsd.setUpdatedAt(block.timestamp);
    (, int256 answer,, uint256 updatedAt,) = rsEthPriceFeed.latestRoundData();
    assertEq(answer, initialAnswer);          // rsETH component frozen
    assertEq(updatedAt, block.timestamp);     // updatedAt appears fresh
}
