### Title
`combine_ac` accepts crafted ciphertext with mismatched `blob.batch_count` vs. inner Q-point count, silently returning wrong-count output as SUCCESS — (`src/cbmpc/api/pve_batch_ac.cpp`)

### Summary
`combine_ac` parses `blob.batch_count` from the outer `pve_ac_ciphertext_blob_v1_t` envelope but never uses it after parsing. The actual batch size is derived from `pve_ct.get_Q().size()` inside `aggregate_to_restore_row`. An attacker who controls the ciphertext blob can set `blob.batch_count=N` while serializing an inner `ec_pve_ac_t` with M ≠ N Q-points. `combine_ac` returns SUCCESS with M scalars in `out_xs`, while the caller expected N.

### Finding Description

`parse_ac_ciphertext` validates `blob.batch_count` for range only: [1](#0-0) 

After this, `blob.batch_count` is never referenced again inside `combine_ac`: [2](#0-1) 

`aggregate_to_restore_row` independently derives batch size from the inner Q vector: [3](#0-2) 

The output vector is sized from `xs_bn.size()` (M), not from `blob.batch_count` (N): [4](#0-3) 

The missing guard is: `if (pve_ct.get_Q().size() != blob.batch_count) return error(E_FORMAT, "batch count mismatch");` — it is absent from both `parse_ac_ciphertext` and `combine_ac`.

### Impact Explanation

An attacker who can supply the ciphertext blob (e.g., a malicious peer, a compromised storage layer, or any caller of the C/C++ API with a crafted buffer) can:

- **M < N**: Cause the caller to receive fewer scalars than expected. Any caller that trusts `blob.batch_count` (e.g., via a prior call to `get_ac_batch_count`) and indexes `out_xs[i]` for `i ≥ M` without re-checking `out_xs.size()` invokes undefined behavior — a potential memory safety issue on top of the logic error.
- **M > N**: Cause the caller to silently accept extra attacker-chosen scalars as recovered key material, substituting or extending the output with attacker-controlled values.

In both cases, `combine_ac` returns SUCCESS, giving the caller no signal that the output count is wrong. The API comment explicitly states that `combine_ac` is designed to not leak secret information on invalid ciphertexts, but this case produces *accepted wrong output* rather than a clean failure. [5](#0-4) 

### Likelihood Explanation

The attack requires the attacker to supply a crafted ciphertext blob. This is realistic in any deployment where the ciphertext is received from a network peer, stored in a database, or passed across a trust boundary. The C API comment explicitly notes that `combine_ac` skips verification and instructs callers to call `verify_ac` first — but this is advisory, not enforced. Callers who omit `verify_ac` (a common pattern in recovery/fallback flows) are fully exposed. The `verify_ac` path does check `Qs_compressed.size() == blob.batch_count` but only if the caller supplies the Q-points externally; it does not independently enforce `pve_ct.get_Q().size() == blob.batch_count` either. [6](#0-5) 

### Recommendation

Add a consistency check immediately after deserializing the inner `ec_pve_ac_t` in both `combine_ac` and `parse_ac_ciphertext` (or in a shared post-parse helper):

```cpp
if (pve_ct.get_Q().size() != static_cast<size_t>(blob.batch_count))
    return coinbase::error(E_FORMAT, "batch_count/Q-point count mismatch");
```

This should be applied in `combine_ac`, `verify_ac`, `partial_decrypt_ac_attempt`, and `get_public_keys_compressed_ac` — anywhere `blob.ct` is deserialized and `blob.batch_count` is trusted.

### Proof of Concept

```cpp
// 1. Encrypt M=3 scalars → get a valid ciphertext blob with batch_count=3, 3 Q-points.
// 2. Deserialize the outer pve_ac_ciphertext_blob_v1_t.
// 3. Patch blob.batch_count = 8 (N=8).
// 4. Re-serialize the blob.
// 5. Call combine_ac with a valid quorum on the patched ciphertext.
// 6. Assert: rv == SUCCESS && out_xs.size() == 3 (M), not 8 (N).
//    The API layer and protocol layer disagree: blob says 8, output has 3.
```

The test is fully deterministic: no randomness is needed beyond the initial honest encryption. The patched `batch_count` field passes `parse_ac_ciphertext`'s range checks, the inner `ec_pve_ac_t` decrypts correctly (it was honestly encrypted), and `aggregate_to_restore_row` returns SUCCESS with 3 scalars. The caller receives a wrong-count vector with no error indication.

  if (rv) return rv;
