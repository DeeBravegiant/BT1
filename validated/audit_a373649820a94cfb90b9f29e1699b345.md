Audit Report

## Title
Missing Staleness and Round Completeness Validation in `ChainlinkPriceOracle.getAssetPrice()` — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but discards all return values except `answer`, performing no staleness (`updatedAt`), round completeness (`answeredInRound >= roundId`), or non-negative price checks. A stale or incomplete Chainlink round silently propagates into deposit minting, withdrawal sizing, and rsETH price updates. The sibling contract `ChainlinkOracleForRSETHPoolCollateral` in the same repository applies all three checks, confirming the protocol is aware of the requirement.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` at line 52 of `contracts/oracles/ChainlinkPriceOracle.sol`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

All five return values (`roundId`, `answer`, `startedAt`, `updatedAt`, `answeredInRound`) are available but only `answer` is consumed. No validation is applied.

By contrast, `ChainlinkOracleForRSETHPoolCollateral.getRate()` at lines 30–32 of `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` applies all three guards:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

The unvalidated price flows through `LRTOracle.getAssetPrice()` (line 157, `contracts/LRTOracle.sol`) into two permissionless paths:

1. **Deposit minting** — `LRTDepositPool.getRsETHAmountToMint()` at line 520 of `contracts/LRTDepositPool.sol`:
   ```solidity
   rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
   ```
2. **Withdrawal sizing** — `LRTWithdrawalManager.getExpectedAssetAmount()` at line 593 of `contracts/LRTWithdrawalManager.sol`:
   ```solidity
   underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
   ```

The `pricePercentageLimit` guard in `_updateRsETHPrice()` (lines 252–281, `contracts/LRTOracle.sol`) does **not** protect these paths: deposits and withdrawals consume `lrtOracle.getAssetPrice(asset)` as a live call and `lrtOracle.rsETHPrice()` as a stored value, bypassing the price-change threshold check entirely.

## Impact Explanation

**Critical — Direct theft of user funds / Protocol insolvency.**

When a Chainlink feed stalls (no new round started), `latestRoundData()` continues returning the last completed round's answer. If the stale price is inflated relative to the true market price (e.g., an LST depegs after the oracle stalls), `getRsETHAmountToMint()` mints more rsETH than the deposited LST is worth. The attacker redeems the excess rsETH via `initiateWithdrawal`, extracting value from the pool at the expense of all existing rsETH holders. Repeated exploitation drains the protocol's backing, constituting direct theft of user funds and protocol insolvency — both Critical-class impacts under the allowed scope.

A zero or negative `answer` (incomplete round) causes `uint256(price)` to produce 0 or wrap to a huge value, resulting in either a DoS on deposits (0 rsETH minted) or an unbounded over-mint.

## Likelihood Explanation

Chainlink feeds can fail to advance rounds during network congestion, validator outages, or oracle node issues — all documented, real-world events. The protocol supports multiple LST assets (stETH, ETHx, rETH, sfrxETH, swETH), each with its own feed; any single feed stalling is sufficient. The entry points (`depositAsset`, `initiateWithdrawal`, `instantWithdrawal`) are fully permissionless. No privileged access is required. The exploit is repeatable for as long as the oracle remains stale.

## Recommendation

Apply the same validation pattern already present in `ChainlinkOracleForRSETHPoolCollateral` to `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(price > 0, "Invalid price");
    require(updatedAt != 0, "Incomplete round");
    require(answeredInRound >= roundId, "Stale price");
    require(block.timestamp - updatedAt <= HEARTBEAT, "Price too old");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`HEARTBEAT` should be set per-feed (e.g., 3600 s for 1-hour feeds, 86 400 s for 24-hour feeds).

## Proof of Concept

1. Fork mainnet at a block where the stETH/ETH Chainlink feed is live.
2. Using `vm.mockCall`, make `latestRoundData()` for the stETH feed return: `roundId = 100`, `answer = 1.05e18`, `updatedAt = block.timestamp - 2 days`, `answeredInRound = 99` (stale, incomplete round).
3. Call `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")` from an attacker address.
4. Assert that `getRsETHAmountToMint` returns a value based on `1.05e18` rather than the true price, minting ~10.5% excess rsETH.
5. Call `LRTWithdrawalManager.initiateWithdrawal(stETH, excessRsETH)` and process the withdrawal.
6. Assert the attacker receives more stETH than they deposited in ETH-equivalent terms, confirming extraction of value from existing holders.

A Foundry fork test with `vm.mockCall` on the Chainlink aggregator address is sufficient to reproduce this without requiring an actual oracle stall on mainnet.
