Audit Report

## Title
Permissionless `sendFunds()` + Stale Price Minting Enables Sandwich Attack to Steal MEV/EL Yield — (`contracts/FeeReceiver.sol` / `contracts/LRTOracle.sol` / `contracts/LRTDepositPool.sol`)

## Summary

`FeeReceiver.sendFunds()` has no access control, and `LRTOracle.updateRSETHPrice()` is callable by any unprivileged address. rsETH minting uses the stored, stale `rsETHPrice` rather than a live TVL calculation. An attacker can deposit at the pre-reward price, atomically trigger reward distribution and the price update, and immediately hold rsETH worth more than they paid — stealing a proportional share of MEV/EL yield that belongs to existing holders.

## Finding Description

**Root cause 1 — Permissionless reward flush:**
`FeeReceiver.sendFunds()` carries no role modifier; any EOA can call it to push accumulated ETH into `LRTDepositPool`. [1](#0-0) 

**Root cause 2 — Permissionless price update:**
`LRTOracle.updateRSETHPrice()` is `public` with only a `whenNotPaused` guard; no role is required. [2](#0-1) 

**Root cause 3 — Stale price used for minting:**
`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()`, which is the last *stored* value, not a freshly computed TVL ratio. [3](#0-2) 

**Price update mechanics:**
`_updateRsETHPrice()` computes `previousTVL = rsethSupply * rsETHPrice` (the stale stored price), then sets the new price from the *current* on-chain TVL. Any ETH added to the deposit pool between the last price update and this call is treated as new yield. [4](#0-3) 

**`pricePercentageLimit` bypass:**
The threshold check is gated on `pricePercentageLimit > 0`. If the admin has never set it, the check is entirely skipped, removing the only friction for a non-manager caller. [5](#0-4) 

**Exploit sequence (single block):**
1. Observe `FeeReceiver` balance (e.g., 10 ETH accumulated).
2. Call `LRTDepositPool.depositETH{value: 1000 ETH}()` — minted at stale price 1.0 → receive 1000 rsETH.
3. Call `FeeReceiver.sendFunds()` — 10 ETH enters the deposit pool; TVL becomes 2010 ETH.
4. Call `LRTOracle.updateRSETHPrice()` — new price = (2010 − fee) / 2000 ≈ 1.00450.
5. Attacker's 1000 rsETH is now worth ≈ 1004.5 ETH; they paid 1000 ETH, capturing ~4.5 ETH of the 9 ETH net yield.

## Impact Explanation

**High — Theft of unclaimed yield.** MEV/EL rewards accumulate in `FeeReceiver` and are intended to accrue to *existing* rsETH holders proportional to their time-weighted stake. The sandwich attack lets a late entrant dilute that accrual: in the PoC, existing holders receive only ~4.5 ETH of the 9 ETH net yield instead of the full amount. The attacker's rsETH is immediately redeemable at the higher price on secondary markets, making the yield theft concrete and repeatable every reward cycle.

## Likelihood Explanation

**Medium.** The attack requires no privileged access and no flash loan — only capital and two public function calls. `FeeReceiver` balance is trivially observable on-chain, so the attacker can time deposits precisely. The only friction is `pricePercentageLimit`, which is unset by default (defaults to 0, disabling the check entirely), and even when set, the attacker can target smaller reward accumulations that stay within the limit. The attack is repeatable every time rewards accumulate.

## Recommendation

1. **Restrict `FeeReceiver.sendFunds()`** to a privileged role (e.g., `MANAGER` or `OPERATOR`) so the timing of reward distribution cannot be controlled by an attacker. [6](#0-5) 

2. **Alternatively**, call `_updateRsETHPrice()` atomically inside `depositETH`/`depositAsset` before computing the mint amount, so every deposit uses the live TVL rather than the stale stored price. [7](#0-6) 

3. **Ensure `pricePercentageLimit` is always set** to a non-zero value during deployment/initialization to prevent the threshold bypass. [8](#0-7) 

## Proof of Concept

**Foundry fork test outline:**

```solidity
function test_sandwichMevRewards() public {
    // Setup: FeeReceiver holds 10 ETH; TVL = 1000 ETH; supply = 1000 rsETH; price = 1e18
    uint256 attackerDeposit = 1000 ether;

    // Step 1: Attacker deposits at stale price
    vm.prank(attacker);
    depositPool.depositETH{value: attackerDeposit}(0, "");
    uint256 rsethReceived = rsETH.balanceOf(attacker);
    assertEq(rsethReceived, 1000 ether); // minted at price 1.0

    // Step 2: Flush MEV rewards (permissionless)
    vm.prank(attacker);
    feeReceiver.sendFunds(); // 10 ETH → depositPool

    // Step 3: Update price (permissionless)
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();

    uint256 newPrice = lrtOracle.rsETHPrice();
    assertGt(newPrice, 1e18); // price increased

    // Step 4: Verify attacker captured yield
    uint256 attackerValueETH = rsethReceived * newPrice / 1e18;
    assertGt(attackerValueETH, attackerDeposit); // attacker profited from others' yield
}
```

The test is reproducible on a mainnet fork by pointing to the deployed `FeeReceiver`, `LRTDepositPool`, and `LRTOracle` addresses and seeding the `FeeReceiver` with ETH via `vm.deal`.
