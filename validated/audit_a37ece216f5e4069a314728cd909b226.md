Audit Report

## Title
Excess ETH Permanently Trapped in `updateRate()` Due to Missing Post-Loop Refund - (File: `contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary

`MultiChainRateProvider.updateRate()` is a public payable function that forwards exactly the on-chain `estimatedFee` to each LayerZero `send()` call in a loop. Any ETH sent by the caller beyond the sum of all per-chain fees accumulates in the contract with no recovery path. Neither the abstract base contract nor either concrete inheritor (`RSETHMultiChainRateProvider`, `AGETHMultiChainRateProvider`) implements a `receive()` fallback, ETH sweep, or admin withdrawal function.

## Finding Description

`updateRate()` at line 108 of `MultiChainRateProvider.sol` accepts arbitrary `msg.value` with no access control. Inside the loop (lines 119–134), it queries `estimateFees()` on-chain for each receiver and forwards exactly that amount to `ILayerZeroEndpoint.send()`:

```solidity
(uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
    .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(
    dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
);
```

The `_refundAddress` (`payable(msg.sender)`) passed to LayerZero's `send()` only covers excess within LayerZero's own internal accounting — it does not cover the gap between `msg.value` and `sum(estimatedFee)` that remains held by the `MultiChainRateProvider` contract itself. After the loop completes, no refund of this remainder is issued and no event is emitted for it.

The contract has no `receive()` function, no `withdraw()` function, and no ETH sweep mechanism anywhere in the abstract base or in either concrete inheritor (`RSETHMultiChainRateProvider` at line 9, `AGETHMultiChainRateProvider` at line 12). Any ETH balance accumulating in the contract is irrecoverable.

By contrast, `CrossChainRateProvider.updateRate()` (line 96) passes `msg.value` directly to a single LayerZero `send()`, so LayerZero's own refund mechanism handles any excess correctly.

Exploit path:
1. Caller queries `estimateTotalFee()` off-chain → receives `X` ETH.
2. Caller submits `updateRate{value: X + delta}()` to guard against gas price movement between estimation and execution (standard operational practice).
3. On-chain, the loop consumes exactly `sum(estimatedFee)` ≤ `X`.
4. `delta` (and any difference between off-chain and on-chain estimates) remains in the contract.
5. No function exists to recover it; the ETH is permanently frozen.

## Impact Explanation

This constitutes **permanent freezing of funds** (Critical). The caller's ETH is irrecoverably locked in the contract. There is no owner-only sweep, no `selfdestruct`, and no upgrade path visible in the contract that could rescue it. The loss is permanent and proportional to the over-payment, which grows with the number of configured rate receivers.

## Likelihood Explanation

`updateRate()` has no access control — any EOA or contract can call it. Callers are structurally incentivized to over-send: if `msg.value` is too low, the transaction reverts mid-loop (after some fees have already been consumed), so rational callers always pad their `msg.value`. The `estimateTotalFee()` view function provides an off-chain estimate, but on-chain gas prices can shift between the view call and execution, making exact-match sends unreliable. With multiple rate receivers configured, the cumulative over-payment per call is larger. This is a routine operational pattern, making accidental over-payment highly likely on every call.

## Recommendation

Track total fees consumed in the loop and refund the remainder to `msg.sender` after the loop:

```solidity
uint256 totalFeeUsed;
for (uint256 i; i < rateReceiversLength;) {
    // ... existing fee estimation and send logic ...
    totalFeeUsed += estimatedFee;
    unchecked { ++i; }
}
uint256 excess = msg.value - totalFeeUsed;
if (excess > 0) {
    (bool success,) = msg.sender.call{value: excess}("");
    require(success, "ETH refund failed");
}
```

Alternatively, add an `onlyOwner` ETH sweep function as a backstop.

## Proof of Concept

1. Deploy `RSETHMultiChainRateProvider` with two rate receivers configured.
2. Call `estimateTotalFee()` → returns `0.01 ETH`.
3. Submit `updateRate{value: 0.015 ETH}()`.
4. On-chain, `estimateFees()` returns `0.005 ETH` per receiver; loop sends `0.005 + 0.005 = 0.01 ETH` to LayerZero.
5. `address(rateProvider).balance` is now `0.005 ETH`.
6. No function in `MultiChainRateProvider`, `RSETHMultiChainRateProvider`, or `AGETHMultiChainRateProvider` can recover it.

Foundry fork test plan: fork Ethereum mainnet, deploy `RSETHMultiChainRateProvider` with a mock LayerZero endpoint that returns deterministic fee estimates and accepts `send()` calls, call `updateRate{value: fee + 1 ether}()`, assert `address(provider).balance == 1 ether` and that no function reduces it.
