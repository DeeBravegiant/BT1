### Title
Malicious Peer Can Corrupt DL Singleton Tracking via Crafted Launcher Solution — (`chia/data_layer/data_layer_wallet.py`)

---

### Summary

`_track_new_launcher_id()` unconditionally trusts the `full_puzhash` and `amount` extracted from a peer-supplied launcher solution. A malicious peer can return a syntactically valid but semantically wrong solution, causing the wallet to permanently store a `SingletonRecord` whose `coin_id` points to a non-existent coin, while the real singleton is never tracked.

---

### Finding Description

**`launch_solution_to_singleton_info` is a pure parser — no on-chain validation:** [1](#0-0) 

It reads `full_puzzle_hash` and `amount` directly from the solution bytes with no cross-check against the actual child coin created on-chain.

**`fetch_coin_spend` validates the launcher puzzle hash but not the solution:** [2](#0-1) 

The check at line 343 confirms the puzzle's tree hash matches `coin.puzzle_hash` (i.e., `SINGLETON_LAUNCHER_PUZZLE_HASH`). The solution bytes are returned verbatim from the peer with zero validation.

**`_track_new_launcher_id` builds the child coin entirely from the peer-supplied solution:** [3](#0-2) 

`new_singleton = Coin(launcher_id, full_puzhash, amount)` — both `full_puzhash` and `amount` come from the crafted solution. There is no step that fetches the actual on-chain child coin and compares its `puzzle_hash`/`amount`.

**The corrupted record is then persisted and interest registered:** [4](#0-3) 

`coin_id=new_singleton.name()` is stored in `dl_store`, the launcher entry is written, and the wallet subscribes to the wrong coin ID.

**Re-tracking is permanently blocked after the first write:** [5](#0-4) 

Once `add_launcher` succeeds (line 274), any subsequent call to `_track_new_launcher_id` for the same `launcher_id` returns immediately, making the corruption irreversible without manual DB surgery.

**The `match_dl_launcher` guard exists only in the sync path, not in the user-initiated path:** [6](#0-5) 

`match_dl_launcher` is called here before `track_new_launcher_id`, but this code path is only reached during automatic sync of *owned* singletons (gated by `puzzle_store.puzzle_hash_exists(inner_puzhash)`). The user-initiated `track_new_launcher_id(launcher_id, peer)` call — the primary path for tracking non-owned DL singletons — goes directly to `_track_new_launcher_id` with no equivalent guard. [7](#0-6)

### Impact Explanation

The wallet's `dl_store` ends up with:
- A `SingletonRecord` whose `coin_id` is the name of a non-existent coin
- A `lineage_proof` built from the wrong `inner_puzhash`, `root`, and `amount`
- An `add_interested_coin_ids` subscription to the phantom coin

The real singleton coin — which has a different `puzzle_hash`/`amount` — is never registered. The wallet cannot update the DL singleton's state, cannot construct valid spends (lineage proof is wrong), and cannot participate in DL offers for this launcher. This is permanent corruption of Data Layer root/store state with direct security impact.

---

### Likelihood Explanation

The attacker only needs to be a peer that the victim wallet connects to — any node in the P2P network qualifies. The trigger is the user calling `track_new_launcher_id` (e.g., to track a counterparty's DL singleton before a DL offer), which is a normal workflow. No keys, admin access, or cryptographic breaks are required.

---

### Recommendation

After extracting `full_puzhash` and `amount` from the solution, fetch the actual child coin state from the peer and verify:

```python

    [new_singleton.name()], peer=peer

if not child_states or child_states[0].coin != new_singleton:
    raise ValueError(
        f"Peer-supplied launcher solution does not match actual child coin for launcher {launcher_id}"

```

Alternatively, apply the same `match_dl_launcher` internal-consistency check that the sync path already performs before entering `_track_new_launcher_id`.

def launch_solution_to_singleton_info(launch_solution: Program) -> tuple[bytes32, uint64, bytes32, bytes32]:
    solution = launch_solution.as_python()
    try:
        full_puzzle_hash = bytes32(solution[0])
        amount = uint64(int.from_bytes(solution[1], "big"))
        root = bytes32(solution[2][0])
        inner_puzzle_hash = bytes32(solution[2][1])
    except (IndexError, TypeError):
        raise ValueError("Launcher is not a data layer launcher")

    return full_puzzle_hash, amount, root, inner_puzzle_hash

    )

    async def track_new_launcher_id(
        self,
        launcher_id: bytes32,
        peer: WSChiaConnection | None = None,
        spend: CoinSpend | None = None,
        height: uint32 | None = None,
    ) -> None:
        if peer is None:
            peer_list = self.wallet_state_manager.wallet_node.get_full_node_peers_in_order()
        else:
            peer_list = [peer]
        peer_length = len(peer_list)
        for i, peer_to_try in enumerate(peer_list):
            try:
                await self._track_new_launcher_id(
                    launcher_id,
                    peer_to_try,
                    spend,
                    height,
                )
            except LauncherCoinNotFoundError as e:
                if i == peer_length - 1:
                    raise e  # raise the error if we've tried all peers
                continue  # try some other peers, maybe someone has it
