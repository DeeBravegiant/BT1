Audit Report

## Title
ETH Deposit Limit Not Enforced — Incoming Amount Excluded from Cap Check in `_checkIfDepositAmountExceedesCurrentLimit()` - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit()` applies an asymmetric check: the ETH branch compares only `totalAssetDeposits` against the configured cap, omitting the incoming `amount`, while the ERC-20 branch correctly checks `totalAssetDeposits + amount`. As a result, any single ETH deposit that crosses the cap is never rejected — the limit is only enforced retroactively once a prior deposit has already exceeded it. This breaks the protocol's deposit-cap invariant for ETH.

## Finding Description
At `contracts/LRTDepositPool.sol` lines 676–682, the function reads:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
``` [1](#0-0) 

The ETH branch at line 679 evaluates `totalAssetDeposits > limit`, which is `false` whenever the current total is still below the cap. The incoming `amount` is never added, so the check cannot detect that the deposit would push the total over the cap.

`_beforeDeposit` (lines 661–663) calls this function and reverts only when it returns `true`:

```solidity
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
``` [2](#0-1) 

`depositETH()` (lines 76–93) is the public entry point, callable by any user with no role requirement: [3](#0-2) 

The root cause is the missing `+ amount` in the ETH branch. The ERC-20 path is correct; the ETH path is not. No other guard in the call chain compensates for this omission.

## Impact Explanation
The `depositLimitByAsset[ETH_TOKEN]` cap is a protocol-level risk control set by the manager. Because the cap is never checked against the incoming deposit amount for ETH, a single depositor can push total ETH holdings arbitrarily above the configured limit in one transaction, minting a proportionally oversized rsETH position. No funds are stolen and no freeze occurs, but the protocol silently fails to deliver its stated deposit-cap guarantee. This maps to the allowed impact: **Low — contract fails to deliver promised returns/invariants, but doesn't lose value**.

## Likelihood Explanation
Any unprivileged user calling `depositETH()` with `msg.value` large enough to push total ETH deposits past the configured limit will trigger this path. No special role, flash loan, oracle manipulation, or front-running is required. The condition is reachable whenever the ETH deposit limit has not already been exceeded by a prior deposit, which is the normal operating state of the protocol.

## Recommendation
Add `amount` to the ETH branch, making the check uniform across all asset types:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This correctly rejects any deposit — ETH or ERC-20 — that would push total holdings above the configured cap.

## Proof of Concept
1. Admin sets ETH deposit limit to 1 000 ETH via `LRTConfig.updateAssetDepositLimit(ETH_TOKEN, 1000e18)`.
2. Current `getTotalAssetDeposits(ETH_TOKEN)` = 999e18 (999 ETH).
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. Inside `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 500e18)`:
   - `totalAssetDeposits = 999e18`
   - ETH branch: `999e18 > 1000e18` → `false` → no revert.
5. Deposit succeeds; total ETH becomes 1 499 ETH — 499 ETH above the cap.
6. For comparison, the same call with an ERC-20 asset would evaluate `999e18 + 500e18 > 1000e18` → `true` → `MaximumDepositLimitReached` revert.

**Foundry test plan:**
```solidity
function test_ETHDepositExceedsLimit() public {
    vm.prank(manager);
    lrtConfig.updateAssetDepositLimit(LRTConstants.ETH_TOKEN, 1000 ether);

    // Seed pool to 999 ETH
    vm.deal(depositor, 1499 ether);
    vm.prank(depositor);
    lrtDepositPool.depositETH{value: 999 ether}(0, "");

    // This should revert but does not
    vm.prank(depositor);
    lrtDepositPool.depositETH{value: 500 ether}(0, ""); // crosses limit by 499 ETH

    assertGt(lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN), 1000 ether);
}
```
