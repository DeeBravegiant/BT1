### Title
Meta Transaction Deposit Refund Misdirection Allows Unprivileged User to Drain Relayer Funds - (File: runtime/runtime/src/actions.rs)

### Summary
In NEAR Protocol's meta transaction (NEP-366) system, when a relayer submits a `SignedDelegateAction` containing an inner action with an attached deposit, and that inner action fails on the receiver's shard, the deposit refund is sent to the `sender_id` (Alice/user) rather than the relayer who paid for it. An unprivileged user can exploit this by crafting a `SignedDelegateAction` that passes the relayer's off-chain validation but fails on-chain execution, causing the relayer to permanently lose the deposit amount to the user.

### Finding Description
The `apply_delegate_action` function creates a new inner receipt with `predecessor_id: sender_id.clone()` (Alice), making Alice the predecessor of the receipt forwarded to the receiver shard. [1](#0-0) 

When this inner receipt fails on the receiver's shard, `refund_unspent_gas_and_deposits` in `runtime/runtime/src/lib.rs` sends the deposit refund to `receipt.balance_refund_receiver()`, which resolves to the predecessor of the receipt — Alice — not the relayer (the signer who actually paid). [2](#0-1) 

The protocol documentation explicitly acknowledges this broken invariant:

> "But what if the transaction fails execution on Bob's shard? At this point, the predecessor is `Alice` and therefore she receives the token balance refunded, not the relayer. This is something relayer implementations must be aware of since there is a financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard." [3](#0-2) 

The relayer's off-chain validation checks the `SignedDelegateAction` content (signature, nonce, `max_block_height`) and may query current on-chain state via view calls. However, between the relayer's off-chain check and the on-chain execution (which can span multiple blocks), Alice can manipulate the receiver contract's state to cause the inner action to fail. The relayer has no way to atomically bind its off-chain check to the on-chain execution. [4](#0-3) 

The `validate_delegate_action_key` function only validates the access key, nonce, and permissions — it does not and cannot guarantee that the inner action will succeed on the receiver shard. [5](#0-4) 

### Impact Explanation
The corrupted protocol value is the **NEAR token balance**: the relayer's balance is reduced by the full deposit amount, and Alice's balance is increased by the same amount. This is a direct, irreversible financial transfer from the relayer to the attacker. The deposit amount is bounded only by the relayer's willingness to accept the `SignedDelegateAction`, and a malicious user can craft actions with arbitrarily large deposits (up to the relayer's balance). Gas costs are also lost by the relayer (gas refunds go to the signer/relayer, but the deposit does not). [6](#0-5) 

### Likelihood Explanation
Medium. Any unprivileged user can create a `SignedDelegateAction` with a large deposit attached to a function call on a contract they control. The user deploys a contract that behaves correctly during the relayer's off-chain view-call check, then updates the contract (or manipulates its storage) to revert before the relayer's transaction is included in a chunk. The window between the relayer's check and on-chain execution is at least one block (~1 second on mainnet), which is sufficient for the attacker to submit a state-changing transaction. The attack requires no privileged access — only the ability to deploy a contract and sign a `DelegateAction`. [7](#0-6) 

### Recommendation
1. **Protocol-level**: Modify the deposit refund logic for receipts originating from `DelegateAction` to send deposit refunds to the signer (relayer) rather than the predecessor (Alice). This would require tracking the origin of a receipt.
2. **Application-level**: Relayers should refuse to process `SignedDelegateAction`s with non-zero deposits unless they have strong trust in the user, or implement an allowlist/reputation system analogous to what the Atlas report recommends.
3. **Protocol-level alternative**: Introduce an optional `refund_to` field in `DelegateAction` that allows the user to explicitly designate the relayer as the deposit refund recipient, giving relayers a way to enforce this as a precondition.

### Proof of Concept
1. Alice deploys a contract `evil.alice` with a method `check_flag` that reads a storage key `flag` and panics if `flag == "fail"`, otherwise succeeds.
2. Alice sets `flag = "pass"` on `evil.alice`.
3. Alice creates a `DelegateAction` with `receiver_id = evil.alice`, `actions = [FunctionCall { method_name: "check_flag", deposit: 100 NEAR, gas: 10 TGas }]`, and signs it.
4. Alice sends the `SignedDelegateAction` to the relayer.
5. The relayer performs an off-chain view call to `evil.alice.check_flag` — it succeeds (flag is "pass"). The relayer validates the signature and nonce. Everything passes.
6. The relayer wraps the `SignedDelegateAction` in a transaction and submits it to the network.
7. **Before the transaction is included in a chunk**, Alice submits a separate transaction setting `flag = "fail"` on `evil.alice`. This transaction is included first (or in the same block on a different shard).
8. The relayer's transaction is processed: the outer delegate action succeeds on Alice's shard, creating an inner receipt to `evil.alice` with `predecessor_id = alice`, `deposit = 100 NEAR`.
9. On `evil.alice`'s shard, `check_flag` panics because `flag == "fail"`. The inner receipt fails.
10. A deposit refund receipt of 100 NEAR is created with `receiver_id = alice` (the predecessor).
11. Alice receives 100 NEAR. The relayer loses 100 NEAR plus all gas costs. [8](#0-7) [9](#0-8)

```rust
pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
    if result.result.is_err() {
        // Validation failed. Need to return Ok() because this is not a runtime error.
        // "result.result" will be return to the User as the action execution result.
        return Ok(());
    }
```

```markdown
## Relayer

Meta transactions only work with a relayer. This is an application layer
concept, implemented off-chain. Think of it as a server that accepts a
`SignedDelegateAction`, does some checks on them and eventually forwards it
inside a transaction to the blockchain network.

A relayer may choose to offer their service for free but that's not going to be
financially viable long-term. But they could easily have the user pay using
other means, outside of Near blockchain. And with some tricks, it can even be
paid using fungible tokens on Near.

In the example visualized above, the payment is done using \$FT. Together with
the transfer to John, Alice also adds an action to pay 0.1 \$FT to the relayer.
The relayer checks the content of the `SignedDelegateAction` and only processes
it if this payment is included as the first action. In this way, the relayer
will be paid in the same transaction as John.

Note that the payment to the relayer is still not guaranteed. It could be that
Alice does not have sufficient $FT and the transfer fails. To mitigate, the
relayer should check the $FT balance of Alice first.

Unfortunately, this still does not guarantee that the balance will be high
enough once the meta transaction executes. The relayer could waste NEAR gas
without compensation if Alice somehow reduces her \$FT balance in just the right
moment. Some level of trust between the relayer and its user is therefore
required.

The vision here is that there will be mostly application-specific relayers. A
general-purpose relayer is difficult to implement with just the MVP. See
limitations below.
```
