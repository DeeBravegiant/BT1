### Title
Unauthenticated P2P Peer Can Corrupt Node State Storage via Inflated `state_diff_length` Header — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts `SignedBlockHeader` messages from unauthenticated P2P peers without verifying the block hash, parent hash chain, or the cryptographic signature content. The `state_diff_length` field from that unverified header is then used as the authoritative loop bound in `StateDiffStreamBuilder::parse_data_for_block`. A malicious peer can inflate `state_diff_length` to N+K, serve N+K `StateDiffChunk` messages (N canonical + K fabricated), and cause the node to persist a corrupted `ThinStateDiff` — including fabricated storage writes, nonces, and class hashes — into its permanent state database.

---

### Finding Description

**Step 1 — Header accepted without block hash or signature verification.**

`HeaderStreamBuilder::parse_data_for_block` performs only two checks on an incoming `SignedBlockHeader`:
- The `block_number` field matches the expected sequence number.
- The `signatures` vector has exactly one element (length only, no cryptographic verification).

The explicit TODO at line 102 confirms that `parent_hash` chain verification is unimplemented:

```
// TODO(shahak): Check that parent_hash is the same as the previous block's hash
// and handle reverts.
```

No check is performed on `block_hash`, `state_diff_commitment`, or the signature bytes themselves. [1](#0-0) 

**Step 2 — `state_diff_length` from the unverified header becomes the loop bound.**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` directly from the stored header:

```rust
let target_state_diff_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_length
    .ok_or(...)?;
```

The loop then runs until `current_state_diff_len >= target_state_diff_len`, consuming exactly as many `StateDiffChunk` messages as the peer-controlled value demands. [2](#0-1) 

**Step 3 — No state diff commitment cross-check.**

After assembling the `ThinStateDiff`, the code calls only `validate_deprecated_declared_classes_non_conflicting`. It never computes `calculate_state_diff_hash` on the assembled diff and compares it against the `state_diff_commitment` stored in the header. The fabricated entries pass silently. [3](#0-2) 

**Step 4 — Corrupted `ThinStateDiff` is written directly to the state database.**

`append_state_diff` writes every entry in the `ThinStateDiff` into the live state tables — `contract_storage`, `nonces`, `deployed_contracts`, `compiled_class_hash` — with no further validation. These are the exact tables queried by `starknet_getStorageAt`, `starknet_getClassHashAt`, `starknet_getNonce`, etc. [4](#0-3) 

---

### Impact Explanation

After the attack, every RPC call that reads state for the affected block number returns attacker-controlled values. Specifically:
- `starknet_getStorageAt` returns fabricated storage slot values.
- `starknet_getClassHashAt` returns fabricated class hashes (enabling contract impersonation in client logic).
- `starknet_getNonce` returns fabricated nonces.
- `starknet_getStateUpdate` returns the fabricated `ThinStateDiff` verbatim.

This is a concrete instance of **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."** Any downstream system (wallets, bridges, indexers, fee estimators) that trusts this node's RPC will act on corrupted state.

---

### Likelihood Explanation

The attack requires only that a malicious node be accepted as a P2P peer, which is the normal operating condition for any node running `P2pSyncClient`. No privileged access, no validator key, no operator credential is needed. The attacker controls the entire `SignedBlockHeader` payload (including `state_diff_length` and `state_diff_commitment`) and the subsequent `StateDiffChunk` stream. The only constraint is that the number of chunks must exactly equal the inflated `state_diff_length`; the attacker controls both sides of that equation.

---

### Recommendation

1. **Verify the block hash**: After receiving a `SignedBlockHeader`, compute the block hash from its fields and compare it against the `block_hash` field. Reject the header if they differ. This is the unimplemented TODO at `header.rs:102`.
2. **Verify the signature**: Validate the consensus signature against the computed block hash using the known validator public key set.
3. **Cross-check the state diff commitment**: After assembling the `ThinStateDiff` in `parse_data_for_block`, call `calculate_state_diff_hash` and compare the result against the `state_diff_commitment` stored in the header. Reject the peer if they differ. [5](#0-4) [6](#0-5) 

---

### Proof of Concept

```
1. Attacker node connects as a P2P peer to the victim node.

2. Victim requests headers via HeaderQuery starting at block N.

3. Attacker sends SignedBlockHeader for block N with:
     block_number = N
     state_diff_length = 3          // real value is 1
     state_diff_commitment = <any>  // not verified
     signatures = [<any 1 element>] // length check passes, content not verified

4. Victim stores this header. target_state_diff_len = 3.

5. Victim requests state diff for block N via StateDiffQuery.

6. Attacker sends 3 StateDiffChunk messages:
     chunk 1: ContractDiff { contract_address: 0xREAL, storage_diffs: {0xREAL_KEY: 0xREAL_VAL} }
     chunk 2: ContractDiff { contract_address: 0xFAKE1, storage_diffs: {0xFAKE_KEY: 0xFAKE_VAL} }
     chunk 3: ContractDiff { contract_address: 0xFAKE2, nonce: 999 }
   followed by Fin.

7. current_state_diff_len (3) == target_state_diff_len (3): loop exits, no error.

8. validate_deprecated_declared_classes_non_conflicting passes (no deprecated classes).

9. append_state_diff writes all 3 entries to the database.

10. starknet_getStorageAt(0xFAKE1, 0xFAKE_KEY, block=N) → 0xFAKE_VAL  ✓ (corrupted)
    starknet_getNonce(0xFAKE2, block=N)                  → 999          ✓ (corrupted)
```

pub fn calculate_state_diff_hash(state_diff: &ThinStateDiff) -> StateDiffCommitment {
    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
    hash_chain = hash_chain.chain(&Felt::ONE) // placeholder.
        .chain(&Felt::ZERO); // placeholder.
    hash_chain = chain_storage_diffs(&state_diff.storage_diffs, hash_chain);
    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))
}
