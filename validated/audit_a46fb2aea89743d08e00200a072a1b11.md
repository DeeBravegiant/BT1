### Title
Meta-Transaction Signed Commitment Without Escrow Allows Sender to Steal Relayer's Attached Deposit - (File: runtime/runtime/src/actions.rs)

### Summary
A user (Alice) can sign a `DelegateAction` (meta-transaction) that includes a large attached deposit, deliver it to a relayer, and then ensure the inner action fails at execution time. Because the inner receipt's `predecessor_id` is set to Alice (the sender), not the relayer who actually paid the deposit, the deposit refund is sent to Alice rather than the relayer. The relayer loses both gas costs and the full attached deposit with no recourse.

### Finding Description
The `DelegateAction` (NEP-366 meta-transaction) system allows Alice to sign an action off-chain and hand it to a relayer, who wraps it in a transaction and pays all gas and attached deposits. The protocol does not escrow or lock any of Alice's resources when she signs the `DelegateAction`. This is the direct analog of the NFT private-sale spoofing bug: a signed commitment is created without locking the underlying asset.

In `apply_delegate_action` (`runtime/runtime/src/actions.rs`), when the `DelegateAction` is validated and the inner receipt is constructed, the inner receipt's `predecessor_id` is set to `sender_id` (Alice), not to the relayer:

```rust
let new_receipt = Receipt::V0(ReceiptV0 {
    predecessor_id: sender_id.clone(),   // ← Alice, not the relayer
    receiver_id: delegate_action.receiver_id().clone(),
    ...
});
``` [1](#0-0) 

When this inner receipt fails on the receiver's shard, the runtime computes the deposit refund and sends it to `receipt.balance_refund_receiver()`, which returns `predecessor_id`:

```rust
pub fn balance_refund_receiver(&self) -> &AccountId {
    self.refund_to().as_ref().unwrap_or_else(|| self.predecessor_id())
}
``` [2](#0-1) 

The deposit refund is then issued to Alice:

```rust
if deposit_refund > Balance::ZERO {
    result.new_receipts.push(Receipt::new_balance_refund(
        receipt.balance_refund_receiver(),
        deposit_refund,
    ));
}
``` [3](#0-2) 

**Concrete attack path:**

1. Alice signs a `DelegateAction` whose inner actions include a `FunctionCall` with a large attached deposit (e.g., 1000 NEAR) to a contract method that is designed to always revert, or to a `Transfer` targeting an account Alice will delete before execution.
2. Alice delivers the `SignedDelegateAction` to a relayer. The relayer validates the signature, nonce, and key — all pass because Alice has not yet altered any on-chain state.
3. The relayer wraps it in a transaction and submits it, paying the 1000 NEAR deposit plus gas from its own balance.
4. The outer `DelegateAction` receipt executes on Alice's shard: key exists, nonce is valid, inner receipt is created with `predecessor_id = Alice`.
5. The inner receipt executes on the receiver's shard and fails (function reverts, or account does not exist).
6. The runtime issues a deposit refund of 1000 NEAR to `predecessor_id` = Alice.
7. The relayer has lost 1000 NEAR (deposit) + gas, and Alice has gained 1000 NEAR.

The nearcore documentation explicitly acknowledges this as a known design limitation:

> "But what if the transaction fails execution on Bob's shard? At this point, the predecessor is Alice and therefore she receives the token balance refunded, not the relayer. This is something relayer implementations must be aware of since there is a financial incentive for Alice to submit meta transactions that have high balances attached but will fail on Bob's shard." [4](#0-3) 

The comment in the code itself also acknowledges this:

> "Note, Relayer prepaid all fees and all things required by actions: attached deposits and attached gas. If something goes wrong, deposit is refunded to the predecessor, this is sender_id/Sender in DelegateAction. … Therefore Relayer should verify DelegateAction before submitting it because it spends the attached deposit." [5](#0-4) 

The `DelegateAction` struct itself contains no mechanism to lock or escrow Alice's resources: [6](#0-5) 

### Impact Explanation
Any relayer that accepts `DelegateAction`s with attached deposits can be drained of NEAR tokens by a malicious sender. The corrupted protocol value is the relayer's NEAR balance: the relayer pays the deposit, the inner action fails, and the deposit is credited to Alice's account rather than refunded to the relayer. This is a direct, irreversible financial loss for the relayer, proportional to the deposit size Alice specifies. Since the relayer cannot atomically verify that the inner action will succeed at the time it checks the `SignedDelegateAction`, no amount of off-chain pre-validation fully closes the window.

### Likelihood Explanation
The attack requires no special privileges. Any NEAR account holder can create a `SignedDelegateAction` with a large attached deposit targeting a failing action, deliver it to any public relayer, and collect the refund. The attack is repeatable and scales linearly with the number of relayers and the deposit sizes they are willing to forward. The only friction is finding a relayer willing to forward large deposits, which is the normal use case for meta-transactions.

### Recommendation
The root fix is to set the inner receipt's `predecessor_id` to the relayer (the outer transaction signer) rather than to Alice, so that deposit refunds return to the party that paid them. Alternatively, the protocol could introduce an explicit `refund_to` field on the inner receipt that is set to the relayer's account ID when constructing the receipt inside `apply_delegate_action`. A partial mitigation is for relayers to refuse `DelegateAction`s that include attached deposits, or to simulate execution before submission — but neither is enforceable at the protocol level.

### Proof of Concept
1. Alice creates and signs:
   ```
   DelegateAction {
     sender_id: "alice.near",
     receiver_id: "some_contract.near",
     actions: [FunctionCall { method: "always_panic", deposit: 1_000_000_000_000_000_000_000_000 /* 1000 NEAR */ }],
     nonce: <valid>,
     max_block_height: <valid>,
     public_key: <alice_key>,
   }
   ```
2. Alice delivers `SignedDelegateAction` to a relayer. Relayer verifies: signature valid, key exists, nonce valid, balance sufficient on relayer side. Relayer submits the outer transaction paying 1000 NEAR deposit.
3. Outer receipt executes on Alice's shard: `validate_delegate_action_key` passes, inner receipt created with `predecessor_id = "alice.near"`.
4. Inner receipt executes on `some_contract.near`'s shard: `always_panic` reverts.
5. Runtime calls `refund_unspent_gas_and_deposits` → `receipt.balance_refund_receiver()` returns `"alice.near"` → `Receipt::new_balance_refund("alice.near", 1000 NEAR)` is emitted.
6. Alice's balance increases by 1000 NEAR; relayer's balance decreases by 1000 NEAR + gas.

    pub receiver_id: AccountId,

    /// With the meta transactions MVP defined in NEP-366, nested
