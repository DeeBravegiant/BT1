Audit Report

## Title
Block Stuffing Delays `InterimRSETHOracle` Rate Update, Exhausting `dailyMintLimit` Faster Than Intended — (`contracts/pools/oracle/InterimRSETHOracle.sol`, `contracts/pools/RSETHPoolV3.sol`)

## Summary
`InterimRSETHOracle` stores the rsETH/ETH rate as a plain storage variable with no on-chain staleness enforcement; it is updated only when a `MANAGER_ROLE` account calls `setRate()`. An attacker can perform block stuffing to prevent that transaction from landing, keeping the oracle at a stale lower rate. Because `RSETHPoolV3.viewSwapRsETHAmountAndFee` divides by the rate, a stale lower rate inflates the rsETH amount per deposit, exhausting `dailyMintLimit` faster than intended and denying service to legitimate depositors for the remainder of the day.

## Finding Description
`InterimRSETHOracle._setRate` writes the rate to storage with no timestamp and `getRate()` returns it unconditionally with no freshness check: [1](#0-0) 

`RSETHPoolV3.viewSwapRsETHAmountAndFee` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate`: [2](#0-1) 

The `limitDailyMint` modifier accumulates this `rsETHAmount` against `dailyMintLimit` and reverts with `DailyMintLimitExceeded` once the cap is reached: [3](#0-2) 

When the attacker stuffs blocks to exclude the manager's `setRate()` call, the rate remains at `stale_rate < new_rate`. Because the divisor is smaller, each deposit produces a larger `rsETHAmount`, consuming the daily cap faster. For example, at `stale_rate = 1e18` a 100 ETH deposit mints 100 rsETH; at `new_rate = 1.05e18` the same deposit mints only ~95.24 rsETH. The attacker needs to stuff blocks only long enough to prevent the single `setRate()` transaction; the damage (exhausted cap) persists until the next UTC day resets `dailyMintAmount`. [4](#0-3) 

## Impact Explanation
No user funds are lost, but the daily deposit window is effectively stolen: legitimate depositors receive `DailyMintLimitExceeded` reverts for the rest of the day. This concretely matches the allowed impact **Low. Block stuffing** and **Low. Contract fails to deliver promised returns, but doesn't lose value**. The `InterimRSETHOracle` is explicitly an interim, manually-operated oracle, making the timing of `setRate()` calls observable and predictable.

## Likelihood Explanation
The attacker requires no privileged access — only enough ETH to outbid the manager's gas price for the duration of the manager's retry window (seconds to minutes). Block stuffing is cheapest on low-throughput chains (L2s) where an interim oracle is most likely deployed during early phases. The rate update schedule is predictable (e.g., daily after staking rewards accrue), making the attack window easy to identify. The attack is repeatable every day.

## Recommendation
1. **Add staleness enforcement to `InterimRSETHOracle`**: record `lastUpdated = block.timestamp` in `_setRate` and revert in `getRate()` if `block.timestamp - lastUpdated > MAX_STALENESS`, halting deposits when the rate is stale rather than silently using an outdated value.
2. **Cap rsETH minted per deposit in `limitDailyMint`** using the higher of the current rate and a configured floor, so a stale low rate cannot inflate the rsETH amount beyond a safe bound.
3. **Use a time-weighted or commit-reveal rate update** so the effective rate cannot be manipulated by delaying a single transaction.

## Proof of Concept
```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

function testBlockStuffingExhaustsDailyLimit() public {
    // stale rate: 1 ETH = 1 rsETH
    oracle.setRate(1e18);
    pool.setDailyMintLimit(100e18);

    // Attacker stuffs blocks → manager's setRate(1.05e18) never lands
    // Deposit 100 ETH at stale rate → mints 100 rsETH → limit exhausted
    vm.deal(attacker, 100 ether);
    vm.prank(attacker);
    pool.deposit{value: 100 ether}("ref");

    // Legitimate user cannot deposit even 1 wei more
    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert(RSETHPoolV3.DailyMintLimitExceeded.selector);
    pool.deposit{value: 1 ether}("ref");

    // If rate had been updated to 1.05e18, 100 ETH → ~95.24 rsETH
    // → user's 1 ETH deposit would succeed (95.24 + 0.952 < 100)
}
```
The test demonstrates that at the stale rate the cap is exhausted by a single 100 ETH deposit, while at the updated rate the same deposit would leave ~4.76 rsETH of headroom, confirming the inflated consumption caused by the stale oracle.
