### Title
Unauthenticated Peer Advertising Fabricated `highest_block_height` Triggers Premature `StaleSyncHash` and DB Wipe — (`chain/client/src/sync/state/mod.rs`)

### Summary

An unauthenticated peer that connects and advertises an arbitrarily large `highest_block_height` (e.g., `u64::MAX`) with a fabricated `highest_block_hash` can force a syncing node to fire the stale-sync-hash check in `StateSync::run`, return `StateSyncResult::StaleSyncHash`, and ultimately trigger `ShutdownReason::EpochSyncDataReset` — wiping the node's DB and preventing it from ever completing state sync.

### Finding Description

**Step 1 — Peer height is accepted without an upper-bound check.**

`update_block_info` in `connected_peers.rs` is monotonic but unbounded: [1](#0-0) 

Any connected peer can advertise `height = u64::MAX` and the value is stored verbatim.

**Step 2 — The horizon filter in `highest_height_peers()` evicts all legitimate peers.** [2](#0-1) 

When `max_height = u64::MAX`, the filter `i.highest_block_height.saturating_add(horizon) >= u64::MAX` is false for every legitimate peer (e.g., height 1000 + horizon 5 = 1005 ≪ u64::MAX). The malicious peer becomes the **sole** entry in `highest_height_peers`.

**Step 3 — `syncing_info()` picks the malicious peer and derives `highest_height = u64::MAX`.** [3](#0-2) 

The only guard is `is_block_invalid(&p.highest_block_hash)`. A fabricated hash is not in the known-invalid set (it has never been seen), so the filter passes. The malicious peer is selected, and `highest_height = u64::MAX` is forwarded to `handle_sync_needed`.

**Step 4 — `StateSync::run` fires the stale check unconditionally.** [4](#0-3) 

`u64::MAX > block_header.height() + epoch_length + STALE_SYNC_HASH_THRESHOLD` is always true. `StaleSyncHash` is returned regardless of actual network state or download progress.

**Step 5 — `StaleSyncHash` maps directly to `EpochSyncDataReset` → DB wipe.** [5](#0-4) [6](#0-5) 

The node sends `ShutdownReason::EpochSyncDataReset`, wipes its DB, and restarts. If the malicious peer reconnects, the cycle repeats indefinitely.

### Impact Explanation

A node actively downloading state parts has its DB wiped and cannot complete state sync. The node never reaches canonical head. As long as the malicious peer remains connected, every restart triggers the same path. This is a persistent, targeted denial of state sync for any node in `SyncStatus::StateSync`.

### Likelihood Explanation

Any unauthenticated peer that can establish a T2 connection (the normal public peer connection) can perform this attack. No stake, validator key, or special privilege is required. The fabricated `highest_block_hash` only needs to be absent from the local invalid-block set, which is trivially satisfied by any random hash. The attack is deterministic once the malicious peer is the only entry in `highest_height_peers`.

### Recommendation

1. **Validate peer-advertised heights against locally known headers.** Before accepting a peer's `highest_block_height` into the stale-hash comparison, verify that the corresponding `highest_block_hash` is a known header (or at least plausibly within a few epochs of the local header head). Unknown hashes at implausible heights should be treated as invalid.

2. **Require a quorum for the stale-hash decision.** `StaleSyncHash` should only fire when a majority (or a configurable threshold) of connected peers report a height past the epoch boundary, not when a single randomly selected peer does.

3. **Cap `highest_height` at a sanity bound** (e.g., `local_header_head + some_large_constant`) before passing it into `StateSync::run`.

### Proof of Concept

```
1. Node A is in SyncStatus::StateSync with sync_hash at height H, epoch_length E.
2. Attacker connects as peer B, sends block_info { height: u64::MAX, hash: random_hash }.
3. highest_height_peers() returns [B] only (all legitimate peers filtered by horizon).
4. syncing_info() selects B; highest_height = u64::MAX.
5. StateSync::run: u64::MAX > H + E + 100 → true → StaleSyncHash.
6. handle_sync_needed → EpochSyncDataReset → ShutdownReason::EpochSyncDataReset.
7. Node wipes DB, restarts. Attacker reconnects → repeat.
```

The existing test `test_far_horizon_stale_sync_hash_detection` (under `test_features`) confirms the `StaleSyncHash` → `EpochSyncDataReset` path is live and reachable; the only difference is that test uses a legitimately advanced chain rather than a fabricated peer height. [7](#0-6)
