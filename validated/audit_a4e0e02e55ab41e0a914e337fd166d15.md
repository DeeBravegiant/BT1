Audit Report

## Title
Unbounded Nested Loop Gas Consumption in `updateRSETHPrice()` via `_getTotalEthInProtocol()` - (File: contracts/LRTOracle.sol, contracts/LRTDepositPool.sol, contracts/NodeDelegator.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is a public function that triggers a deeply nested external call chain across `supportedAssets`, `nodeDelegatorQueue`, and EigenLayer queued withdrawals. Because `supportedAssetList` has no maximum length cap and `maxNodeDelegatorLimit` has no upper bound, the total gas cost grows multiplicatively as the protocol scales, eventually causing `updateRSETHPrice()` to revert on every call and permanently stale the rsETH price.

## Finding Description
The call chain is confirmed in the code:

1. `LRTOracle.updateRSETHPrice()` (public, `whenNotPaused`) calls `_updateRsETHPrice()` → `_getTotalEthInProtocol()`, which iterates over every entry in `supportedAssets`: [1](#0-0) 

2. Each iteration calls `ILRTDepositPool.getTotalAssetDeposits(asset)` → `getAssetDistributionData(asset)`, which loops over the full `nodeDelegatorQueue`, making three external calls per NDC per asset (`balanceOf`, `getAssetBalance`, `getAssetUnstaking`): [2](#0-1) 

3. `getAssetUnstaking()` in `NodeDelegator` calls EigenLayer's `getQueuedWithdrawals(address(this))` and then double-loops over all returned withdrawals and their strategies: [3](#0-2) 

**Root cause — missing caps:**

- `_addNewSupportedAsset()` in `LRTConfig` pushes to `supportedAssetList` with no maximum length guard: [4](#0-3) 

- `updateMaxNodeDelegatorLimit()` in `LRTDepositPool` only enforces a floor (cannot shrink below current queue length) but has no ceiling: [5](#0-4) 

The total iteration count is `supportedAssets.length × nodeDelegatorQueue.length × queuedWithdrawals.length × strategies.length`. With `maxUncompletedWithdrawalCount` capped at 80 per NDC, the withdrawal dimension is bounded, but the outer two dimensions (assets and NDCs) are not. At realistic protocol scale (e.g., 10 assets × 10 NDCs × 80 withdrawals), the function will exceed the block gas limit.

## Impact Explanation
When `updateRSETHPrice()` reverts on every call, `rsETHPrice` becomes permanently stale. The price-deviation circuit breaker (which pauses deposits and withdrawals when price moves too far) cannot trigger. Deposits and withdrawals continue using a stale price, leading to incorrect rsETH minting/burning ratios. The protocol cannot update its NAV, which is a core invariant. This matches **Medium — Unbounded gas consumption**.

## Likelihood Explanation
The protocol is designed to support multiple LST assets and multiple NodeDelegators as it scales. Adding assets requires `TIME_LOCK_ROLE` and increasing `maxNodeDelegatorLimit` requires `onlyLRTAdmin`, but these are expected operational actions, not attacks. Once the arrays grow large enough through normal protocol operation, any caller of the public `updateRSETHPrice()` will encounter the revert. No adversarial action is required — the failure is a natural consequence of protocol growth. The condition is permanent and not self-healing.

## Recommendation
1. Add a maximum length cap (e.g., `MAX_SUPPORTED_ASSETS = 20`) in `LRTConfig._addNewSupportedAsset()`.
2. Add an explicit upper bound check (e.g., `<= 20`) in `LRTDepositPool.updateMaxNodeDelegatorLimit()`.
3. Refactor `_getTotalEthInProtocol()` to avoid repeated external calls per asset per NDC per withdrawal — consider caching `getAssetUnstaking()` results or splitting the computation across multiple transactions with an accumulator pattern.

## Proof of Concept
Minimal call sequence demonstrating the revert path:

1. Admin adds N supported assets via `LRTConfig.addNewSupportedAsset()` (no cap enforced).
2. Admin increases `maxNodeDelegatorLimit` to M and adds M NDCs via `LRTDepositPool.addNodeDelegatorContractToQueue()`.
3. Operator queues up to 80 withdrawals per NDC via `NodeDelegator.initiateUnstaking()`.
4. Any address calls `LRTOracle.updateRSETHPrice()`.
5. The function executes N × M calls to `getAssetUnstaking()`, each fetching up to 80 withdrawals from EigenLayer storage and iterating their strategies. At N=10, M=10, K=80, the cumulative gas (cold storage reads, external calls, strategy lookups) exceeds the Ethereum block gas limit (~30M gas), causing a revert.
6. `rsETHPrice` is never updated; all subsequent calls revert identically.

A Foundry fork test can demonstrate this by deploying the protocol on a mainnet fork, adding 10 assets and 10 NDCs, queuing 80 withdrawals per NDC, and asserting that `updateRSETHPrice()` reverts with an out-of-gas error.

        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
