Audit Report

## Title
Uninitialized `tokenFeeBps` in `addSupportedToken` Causes Zero Fee Collection on All ERC-20 Token Deposits - (File: contracts/pools/RSETHPool.sol)

## Summary

`RSETHPool.addSupportedToken` sets `supportedTokenOracle[token]` and `tokenBridge[token]` but never initializes `tokenFeeBps[token]`, leaving it at the Solidity default of `0`. Every subsequent call to `deposit(token, amount, referralId)` computes a fee of zero via `viewSwapRsETHAmountAndFee(amount, token)`, so `feeEarnedInToken[token]` never accumulates any value. All protocol fee revenue from the ERC-20 token deposit path is lost until an admin separately calls `setTokenFeeBps`.

## Finding Description

`RSETHPool` maintains two fee variables: `feeBps` for ETH deposits (initialized in `initialize()`) and `tokenFeeBps[token]` for ERC-20 deposits. When a token is added via `addSupportedToken` (L637–656), only `supportedTokenOracle[token]` and `tokenBridge[token]` are written; `tokenFeeBps[token]` is never set and remains `0`.

The token deposit path in `deposit(address token, uint256 amount, string referralId)` (L284–305) calls `viewSwapRsETHAmountAndFee(amount, token)` (L326–347), which reads:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token]; // always 0
fee = amount * feeBpsForToken / 10_000;      // always 0
```

Because `fee = 0`, `feeEarnedInToken[token] += 0` on every deposit (L300). The full `amount` is used as `amountAfterFee`, and the depositor receives the full rsETH equivalent with no fee deducted. The `setTokenFeeBps` function (L583–594) exists to correct this, but it is not called atomically with `addSupportedToken` and is not enforced as a required follow-up step.

The ETH deposit path is unaffected because `feeBps` is set during `initialize()`.

## Impact Explanation

**High — Theft of unclaimed yield.** The protocol is explicitly designed to collect swap fees on ERC-20 token deposits and accumulate them in `feeEarnedInToken[token]` for withdrawal by the `BRIDGER_ROLE`. Because `tokenFeeBps[token]` is always `0` after `addSupportedToken`, every token depositor receives the full rsETH equivalent of their deposit with no fee deducted. All protocol fee revenue from the token deposit path is permanently lost for the window between `addSupportedToken` and a subsequent `setTokenFeeBps` call. The existence of `setTokenFeeBps`, `feeEarnedInToken`, and `withdrawFees(receiver, token)` confirms the protocol's intent to collect these fees; the bug silently voids that intent.

## Likelihood Explanation

**High.** The zero-fee condition is triggered automatically for every supported ERC-20 token from the moment it is added. No attacker setup is required — any ordinary depositor calling `deposit(token, amount, referralId)` benefits from the zero-fee path. The only mitigation is an out-of-band admin call to `setTokenFeeBps`, which is not enforced or documented as a required post-`addSupportedToken` step, making it easy to overlook in practice.

## Recommendation

Initialize `tokenFeeBps[token]` inside `addSupportedToken`, either to the global `feeBps` value or to an explicit caller-supplied parameter:

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    // ... existing checks ...
    tokenFeeBps[token] = _feeBps;
    // ... existing assignments ...
}
```

Alternatively, fall back to `feeBps` when `tokenFeeBps[token]` is zero inside `viewSwapRsETHAmountAndFee`, mirroring the global fee setting.

## Proof of Concept

1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` is never written, remains `0` (L637–656).
2. User calls `deposit(wstETH, 10 ether, "")` (L284–305).
3. `viewSwapRsETHAmountAndFee(10 ether, wstETH)` executes (L335–336):
   - `feeBpsForToken = tokenFeeBps[wstETH]` → `0`
   - `fee = 10 ether * 0 / 10_000` → `0`
   - `amountAfterFee = 10 ether`
4. `feeEarnedInToken[wstETH] += 0` (L300) — protocol collects nothing.
5. User receives rsETH equivalent of the full `10 ether` with no fee deducted.

Foundry test plan:
```solidity
function test_zeroFeeOnTokenDeposit() public {
    // Setup: add token without calling setTokenFeeBps
    vm.prank(timelockRole);
    pool.addSupportedToken(address(wstETH), oracle, bridge);

    // Confirm tokenFeeBps is 0
    assertEq(pool.tokenFeeBps(address(wstETH)), 0);

    // Deposit
    uint256 amount = 10 ether;
    deal(address(wstETH), user, amount);
    vm.startPrank(user);
    wstETH.approve(address(pool), amount);
    pool.deposit(address(wstETH), amount, "");
    vm.stopPrank();

    // Fee earned is zero
    assertEq(pool.feeEarnedInToken(address(wstETH)), 0);
}
```
