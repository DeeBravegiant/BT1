Audit Report

## Title
Stale Chainlink Spot Price Accepted Without Staleness Validation Enables Favorable rsETH Minting Rate Exploitation - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` discards the `updatedAt` field returned by `latestRoundData()`, accepting stale prices without any heartbeat or maximum-age check. When a Chainlink LST/ETH feed goes stale at a price below the true market rate, an unprivileged caller can invoke the public `LRTOracle.updateRSETHPrice()` to store a deflated `rsETHPrice`, then deposit ETH (hardcoded to `1e18` via `OneETHPriceOracle`) to receive more rsETH than the deposited ETH is worth relative to the protocol's true NAV, extracting value from existing rsETH holders.

## Finding Description
`ChainlinkPriceOracle.getAssetPrice()` fetches the latest Chainlink round and silently discards `updatedAt`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
``` [1](#0-0) 

No staleness guard (e.g., `block.timestamp - updatedAt <= MAX_STALENESS`) is applied. This price flows into `LRTOracle._getTotalEthInProtocol()`, which multiplies each LST balance by its Chainlink spot price: [2](#0-1) 

`_getTotalEthInProtocol()` feeds `_updateRsETHPrice()`, which computes and stores the new `rsETHPrice`: [3](#0-2) 

`updateRSETHPrice()` is public and permissionless — any caller can trigger it: [4](#0-3) 

The stored `rsETHPrice` is then used as the denominator when computing rsETH to mint per deposited asset: [5](#0-4) 

ETH deposits use `OneETHPriceOracle`, which hardcodes the ETH price to exactly `1e18`, bypassing any Chainlink feed: [6](#0-5) 

The downside-protection guard in `_updateRsETHPrice()` only triggers if `pricePercentageLimit > 0` AND the deviation exceeds it: [7](#0-6) 

`pricePercentageLimit` is never set in `initialize()`, so it defaults to `0`, meaning the guard is permanently disabled until an admin explicitly configures it: [8](#0-7) 

**Exploit path:**
1. Protocol holds LSTs (e.g., stETH). Chainlink stETH/ETH feed goes stale at a price below the true market rate.
2. Attacker calls `LRTOracle.updateRSETHPrice()`. `_getTotalEthInProtocol()` uses the stale low price, understating total ETH. `rsETHPrice` is stored deflated.
3. If `pricePercentageLimit == 0` (default) or the deviation is within the limit, no pause occurs and the deflated price is accepted.
4. Attacker calls `LRTDepositPool.depositETH{value: N ether}("")`. ETH is priced at `1e18`; the deflated `rsETHPrice` denominator yields more rsETH than the deposited ETH is worth.
5. When the Chainlink feed updates and `rsETHPrice` recovers, the attacker redeems their excess rsETH for more ETH than deposited, extracting value from existing holders.

## Impact Explanation
The attacker receives rsETH tokens representing a larger share of the protocol's NAV than their deposit justifies. When `rsETHPrice` recovers to reflect true asset values, the attacker's excess rsETH redeems for ETH that was effectively owed to pre-existing rsETH holders as accumulated yield. This constitutes **theft of unclaimed yield** from existing rsETH holders, matching the High severity impact class. [5](#0-4) [3](#0-2) 

## Likelihood Explanation
Chainlink feeds have documented periods of staleness during Ethereum network congestion, gas price spikes, or oracle node downtime. The attacker does not need to cause the staleness — they only need to monitor `updatedAt` on-chain and act within the stale window. Both `updateRSETHPrice()` and `depositETH()` are fully permissionless and require no special role. The attack is repeatable across any stale window. Likelihood is **Medium**. [4](#0-3) [9](#0-8) 

## Recommendation
Add a staleness check in `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
(, int256 price,, uint256 updatedAt,) = priceFeed.latestRoundData();
require(block.timestamp - updatedAt <= MAX_STALENESS, "Stale price");
require(price > 0, "Invalid price");
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`MAX_STALENESS` should be configurable per feed (e.g., 3600 seconds for most ETH-denominated Chainlink feeds) and set by the LRT admin. Additionally, consider setting a non-zero `pricePercentageLimit` as a defense-in-depth measure so the downside-protection pause activates even if a stale price slips through. [10](#0-9) [11](#0-10) 

## Proof of Concept
**Setup:** Protocol holds 1000 stETH (true rate: 0.99 ETH/stETH), 0 ETH. rsETH supply = 990. True `rsETHPrice` ≈ 1.0 ETH/rsETH. `pricePercentageLimit == 0` (default).

**Steps:**
1. Chainlink stETH/ETH feed goes stale at 0.90 ETH/stETH.
2. Attacker calls `LRTOracle.updateRSETHPrice()`:
   - `_getTotalEthInProtocol()` = 1000 × 0.90 = 900 ETH (stale, understated)
   - `newRsETHPrice` = 900 / 990 ≈ 0.909 ETH/rsETH (deflated ~9.1%)
   - `pricePercentageLimit == 0` → `isPriceDecreaseOffLimit = false` → no pause, price stored.
3. Attacker calls `LRTDepositPool.depositETH{value: 10 ether}("")`:
   - `rsethAmountToMint` = (10e18 × 1e18) / 0.909e18 ≈ **11.0 rsETH**
4. Chainlink feed updates; anyone calls `updateRSETHPrice()`:
   - `_getTotalEthInProtocol()` = 1000 × 0.99 + 10 = 1000 ETH
   - rsETH supply = 990 + 11 = 1001
   - `newRsETHPrice` = 1000 / 1001 ≈ 0.999 ETH/rsETH
5. Attacker redeems 11.0 rsETH → receives ≈ 10.99 ETH.
6. **Net profit ≈ 0.99 ETH** on a 10 ETH deposit, extracted from existing holders.

**Foundry fork test plan:** Fork mainnet, deploy with `pricePercentageLimit = 0`, mock the Chainlink stETH/ETH aggregator to return a stale `updatedAt` (e.g., `block.timestamp - 2 hours`) with a depressed price, call `updateRSETHPrice()`, deposit ETH, advance time, call `updateRSETHPrice()` again with the true price, and assert the attacker's rsETH redemption value exceeds their deposit.
