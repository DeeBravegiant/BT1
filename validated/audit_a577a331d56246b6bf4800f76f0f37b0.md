Audit Report

## Title
Uninitialized `pricePercentageLimit` Permanently Disables Price-Increase Guard, Enabling First-Depositor Share Inflation Attack — (File: contracts/LRTOracle.sol)

## Summary
`LRTOracle.pricePercentageLimit` is never assigned in `initialize()` and therefore defaults to `0`. The short-circuit `pricePercentageLimit > 0` in `_updateRsETHPrice()` makes the price-increase guard permanently inactive, allowing any caller to inflate `rsETHPrice` by an arbitrary factor via a direct ETH donation to `LRTDepositPool`. A first-depositor attacker can exploit this to cause subsequent victim deposits to mint zero rsETH, stealing the deposited ETH.

## Finding Description

**Root cause 1 — uninitialized guard variable**

`initialize()` sets only `lrtConfig`; `pricePercentageLimit` is never written and remains `0`:

```solidity
// contracts/LRTOracle.sol L64-68
function initialize(address lrtConfigAddr) external initializer {
    UtilLib.checkNonZeroAddress(lrtConfigAddr);
    lrtConfig = ILRTConfig(lrtConfigAddr);
    emit UpdatedLRTConfig(lrtConfigAddr);
}
``` [1](#0-0) 

The guard at `_updateRsETHPrice()` is:

```solidity
bool isPriceIncreaseOffLimit =
    pricePercentageLimit > 0 && priceDifference > pricePercentageLimit.mulWad(highestRsethPrice);
``` [2](#0-1) 

Because `pricePercentageLimit == 0`, the left operand of `&&` is always `false`, so `isPriceIncreaseOffLimit` is permanently `false` regardless of how large the price jump is. [3](#0-2) 

**Root cause 2 — permissionless price update**

`updateRSETHPrice()` is `public` with no role restriction: [4](#0-3) 

**Root cause 3 — raw ETH balance counted in TVL**

`getETHDistributionData()` uses `address(this).balance` directly, so any ETH sent via the open `receive()` is immediately reflected in the oracle's TVL calculation: [5](#0-4) [6](#0-5) 

**Root cause 4 — mint amount uses stored (manipulable) price**

`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()`, the stored value that the attacker just inflated: [7](#0-6) 

**Exploit flow (concrete numbers)**

| Step | Action | State |
|------|--------|-------|
| 1 | Attacker deposits 1 wei ETH → receives 1 wei rsETH | supply = 1 wei, price = 1e18 |
| 2 | Attacker sends 1 ETH (1e18 wei) directly to `LRTDepositPool` via `receive()` | pool balance = 1e18 + 1 wei |
| 3 | Attacker calls `updateRSETHPrice()` | newPrice = (1e18+1)×1e18 / 1 ≈ **1e36** |
| 4 | Victim calls `depositETH{value: 0.5 ETH}(0, "")` | rsethToMint = (5e17 × 1e18) / 1e36 = **0** → victim gets 0 rsETH, 0.5 ETH stays in pool |
| 5 | Attacker redeems 1 wei rsETH (100% of supply) | receives 1e18 + 1 + 5e17 ≈ **1.5 ETH** |

Attacker net: spent ~1 ETH, recovered 1.5 ETH → **+0.5 ETH profit** (victim's entire deposit stolen).

The `_beforeDeposit` check `rsethAmountToMint < minRSETHAmountExpected` only protects victims who pass a non-zero `minRSETHAmountExpected`; victims passing `0` (a common default) receive no protection. [8](#0-7) 

## Impact Explanation

**Critical — Direct theft of user funds.**

Victim ETH deposited via `depositETH` is permanently transferred to the attacker through the withdrawal of the inflated rsETH position. The victim receives zero rsETH and has no recourse. This matches the "Direct theft of any user funds, whether at-rest or in-motion" critical impact class.

## Likelihood Explanation

The attack is executable by any unprivileged EOA at protocol launch before significant TVL accumulates. The attacker needs only: (a) to be the first depositor, (b) capital equal to the intended victim deposit size, and (c) victims who call `depositETH` with `minRSETHAmountExpected = 0`. Condition (c) is realistic — many frontends and integrations omit slippage parameters or default them to zero. The attack is repeatable across multiple victims in the same block before the admin sets `pricePercentageLimit`.

## Recommendation

1. **Initialize `pricePercentageLimit` in `initialize()`** to a safe non-zero value (e.g., `1e16` for 1%):
   ```solidity
   pricePercentageLimit = 1e16; // 1% max price increase per update
   ```
2. **Seed the protocol** with a non-trivial initial deposit (virtual shares / dead shares pattern) to make the donation cost economically prohibitive.
3. **Enforce a non-zero `minRSETHAmountExpected`** at the contract level, or document clearly that passing `0` disables slippage protection.
