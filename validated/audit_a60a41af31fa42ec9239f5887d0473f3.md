Audit Report

## Title
Permissionless `updateRate()` Enables Profitable Deposits Against Stale Cross-Chain Rate — (`contracts/cross-chain/CrossChainRateProvider.sol`, `contracts/cross-chain/MultiChainRateProvider.sol`)

## Summary
Both `CrossChainRateProvider.updateRate()` and `MultiChainRateProvider.updateRate()` are permissionless, allowing any caller to control the exact timing of L1→L2 rate propagation via LayerZero. `RSETHPoolV3` mints new `wrsETH` supply using the stale cached rate stored in `CrossChainRateReceiver`, so a depositor who acts while the L2 rate lags the L1 rate receives more `wrsETH` than their ETH is worth at the true rate. The surplus represents staking yield that belongs to existing holders but is instead captured by the attacker.

## Finding Description
`CrossChainRateProvider.updateRate()` and `MultiChainRateProvider.updateRate()` carry no `onlyOwner` or role check — they are `external payable nonReentrant` only. Either can be called by any EOA or contract at any time, provided the caller supplies the LayerZero messaging fee.

`RSETHRateProvider.getLatestRate()` and `RSETHMultiChainRateProvider.getLatestRate()` both read `ILRTOracle(rsETHPriceOracle).rsETHPrice()`, a public view on L1. The current L1 price is therefore observable by anyone without mempool access.

`CrossChainRateReceiver` stores the last received rate in the `rate` storage slot, updated only inside `lzReceive`. Between LayerZero deliveries, `getRate()` returns the stale value. There is no freshness check or maximum-age guard.

`RSETHPoolV3.getRate()` delegates to `IOracle(rsETHOracle).getRate()`, which resolves to `CrossChainRateReceiver.rate`. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` using this stale value. `deposit()` then calls `wrsETH.mint(msg.sender, rsETHAmount)`, creating new `wrsETH` supply. When the stale rate is lower than the true L1 rate, the division yields a larger `rsETHAmount` than the deposited ETH warrants, and the excess is freshly minted.

The `dailyMintLimit` modifier caps total daily minting but does not prevent the rate-discrepancy exploit within that cap. The `pricePercentageLimit` guard in `LRTOracle._updateRsETHPrice()` applies only to L1 oracle updates and has no effect on L2 rate staleness.

## Impact Explanation
**High — Theft of unclaimed yield.**

The rate increase from `1.04e18` to `1.05e18` represents staking rewards that have accrued to existing rsETH holders. By minting at the stale `1.04e18` rate, the attacker receives approximately `(1/1.04 − 1/1.05) × depositAmount` extra `wrsETH` per deposit. This excess is new supply backed by no additional ETH, diluting all existing `wrsETH` holders and transferring their unclaimed yield to the attacker. The profit scales linearly with deposit size and rate lag magnitude.

## Likelihood Explanation
**High.** Both the L1 oracle price (`LRTOracle.rsETHPrice()`) and the L2 cached rate (`CrossChainRateReceiver.rate`) are public on-chain state readable by any account. No mempool monitoring or privileged access is required to detect the discrepancy. Because `updateRate()` is permissionless, the attacker controls the exact moment the L2 rate is refreshed — they can deposit at the stale rate and call `updateRate()` in the same or immediately following block, making the attack deterministic and repeatable after every L1 reward accrual event.

## Recommendation
1. **Restrict `updateRate()`** in both `CrossChainRateProvider` and `MultiChainRateProvider` to a trusted role (e.g., `onlyOwner` or a dedicated `RATE_UPDATER_ROLE`), removing the attacker's ability to control rate-push timing.
2. **Add a minimum deposit lock-up or redemption delay** on `RSETHPoolV3` so that `wrsETH` minted in a given block cannot be immediately redeemed or sold against a freshly updated rate in the same block.
3. **Bound the acceptable rate delta** in `CrossChainRateReceiver.lzReceive()` — reject updates that increase the rate by more than a configurable threshold per period, consistent with the `pricePercentageLimit` guard already present in `LRTOracle._updateRsETHPrice()` on L1.

## Proof of Concept
1. Observe `LRTOracle.rsETHPrice()` on L1 → `1.05e18`.
2. Observe `CrossChainRateReceiver.rate` on L2 → `1.04e18` (stale).
3. Call `RSETHPoolV3.deposit{value: 100 ether}("")` on L2.
   - `viewSwapRsETHAmountAndFee(100e18)` computes `rsETHAmount = 100e18 * 1e18 / 1.04e18 ≈ 96.15 wrsETH`.
   - At the true rate: `100e18 * 1e18 / 1.05e18 ≈ 95.24 wrsETH`.
   - **Excess minted: ~0.91 wrsETH** (≈ 0.96 ETH of value at the true rate).
4. Call `RSETHRateProvider.updateRate{value: lzFee}()` on L1 — permissionless, no revert.
5. LayerZero delivers the message; `CrossChainRateReceiver.rate` updates to `1.05e18`.
6. Sell the 96.15 `wrsETH` on a DEX or Balancer pool that prices `wrsETH` at the new `1.05e18` rate, recovering ≈ 100.96 ETH — net profit ≈ 0.96 ETH on a 100 ETH deposit, minus gas and LayerZero fees.

**Foundry fork test plan:** Fork L2 (e.g., Arbitrum) at a block where `CrossChainRateReceiver.rate` lags the current L1 `rsETHPrice`. Call `RSETHPoolV3.deposit` with a large ETH value. Assert that `wrsETH.balanceOf(attacker) * CrossChainRateReceiver.rate > depositAmount * 1e18`, confirming the attacker received more value than deposited. Then simulate the `lzReceive` callback updating the rate and assert the attacker's `wrsETH` is now worth more ETH than was deposited.

        rate = latestRate;

        lastUpdated = block.timestamp;

        bytes memory _payload = abi.encode(latestRate);

        uint256 rateReceiversLength = rateReceivers.length;

        for (uint256 i; i < rateReceiversLength;) {

            bytes memory remoteAndLocalAddresses = abi.encodePacked(rateReceivers[i]._contract, address(this));

                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

                dstChainId, remoteAndLocalAddresses, _payload, payable(msg.sender), address(0x0), bytes("")
            );

            unchecked {
                ++i;
            }
        }

        emit RateUpdated(rate);
    }
