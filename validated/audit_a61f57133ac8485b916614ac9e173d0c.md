### Title
Missing Zero-Value Check on `target_puzzle_hash` in Pool Wallet Join Allows Permanent Reward Diversion to Burn Address - (File: `chia/wallet/wallet_rpc_api.py`, `chia/pools/pool_wallet.py`)

---

### Summary

The `pw_join_pool` RPC handler and the pool state validation chain (`_verify_pool_state`, `_verify_pooling_state`) accept `bytes32.zeros` as a valid `target_puzzle_hash` without any zero-value guard. This field controls the on-chain address to which pool block rewards are paid. If set to the zero puzzle hash, all pool rewards are permanently redirected to an unspendable burn address until the farmer notices and completes the multi-block exit process.

---

### Finding Description

`pw_join_pool` in `WalletRpcApi` constructs a new `PoolState` directly from the caller-supplied `request.target_puzzlehash` and passes it to `wallet.join_pool`: [1](#0-0) 

`join_pool` calls `PoolWallet._verify_initial_target_state`, which calls `_verify_pool_state`: [2](#0-1) 

The only check on `target_puzzle_hash` is `if state.target_puzzle_hash is None`. Because `bytes32.zeros` is a fully-constructed `bytes32` object (not `None`), this guard passes silently.

`_verify_pooling_state`, called for `FARMING_TO_POOL` and `LEAVING_POOL` states, checks only `relative_lock_height` bounds and that `pool_url` is non-empty — it never inspects `target_puzzle_hash`: [3](#0-2) 

`create_pool_state` likewise performs no zero check: [4](#0-3) 

The `target_puzzle_hash` field in `PoolState` is the final payout destination for pool block rewards: [5](#0-4) 

Once committed to the blockchain via a singleton state transition, this value is baked into the on-chain puzzle. The pool will send 1.75 XCH per farmed block to `bytes32.zeros` — an address with no known preimage and therefore permanently unspendable.

---

### Impact Explanation

**High — Pool payout redirection / permanent reward loss.**

When `target_puzzle_hash = bytes32.zeros` is committed on-chain in `FARMING_TO_POOL` state, every block reward the pool pays out goes to the zero address and is permanently burned. The farmer must wait `relative_lock_height` blocks (minimum 5, maximum 1000) to exit the pool, during which all farmed rewards are lost. This directly matches the allowed impact: *"payout redirection … singleton mutation"* and *"pool membership state … corruption with direct security impact."*

---

### Likelihood Explanation

The CLI `join_pool` flow reads `target_puzzle_hash` directly from the pool server's `/pool_info` JSON response without a zero check: [6](#0-5) 

A buggy or malicious pool server returning `"0000...0000"` as `target_puzzle_hash` would silently cause the wallet to commit the zero address on-chain. The existing test suite confirms this path is reachable: multiple tests explicitly pass `bytes32.zeros` as `pool_ph` to `pw_join_pool` and succeed without error: [7](#0-6) 

---

### Recommendation

Add a zero-value guard in `_verify_pool_state` (or `_verify_pooling_state`) and at the RPC entry point:

```python
# In _verify_pool_state (pool_wallet.py)
if state.target_puzzle_hash == bytes32.zeros:
    return "target_puzzle_hash must not be the zero address"
```

And in `pw_join_pool` (wallet_rpc_api.py):

```python
if request.target_puzzlehash == bytes32.zeros:
    raise ValueError("target_puzzlehash must not be the zero address")
```

---

### Proof of Concept

1. Start a wallet node and create a pool wallet in `SELF_POOLING` state.
2. Call `pw_join_pool` with `target_puzzlehash = bytes32.zeros` and any valid `pool_url` and `relative_lock_height >= 5`.
3. Observe the call succeeds — no error is raised by `_verify_pool_state` or `_verify_pooling_state`.
4. The singleton state transition is broadcast and confirmed on-chain with `target_puzzle_hash = 0x0000...0000`.
5. All subsequent pool block rewards are paid to the zero address and are permanently unrecoverable.

This is confirmed by the existing test `test_self_pooling_to_pooling` and `test_leave_pool`, both of which use `pool_ph = bytes32.zeros` as the `target_puzzlehash` and complete successfully: [8](#0-7)
