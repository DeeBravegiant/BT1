### Title
Unvalidated `ProposalInit.builder` Allows Malicious Proposer to Redirect All Block Fees and Corrupt the Committed Block Hash — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_proposal_init_valid` validates many fields of a received `ProposalInit` (height, L1/L2 gas prices, `l1_da_mode`, `starknet_version`, `version_constant_commitment`, `fee_proposal_fri`) but never checks `init_proposed.builder`. The `builder` field is consumed verbatim by `convert_to_sn_api_block_info` as `sequencer_address` in the `BlockInfo` passed to the batcher for execution. Because `sequencer_address` governs fee collection and is hashed into `PartialBlockHashComponents`, a malicious proposer can set `builder` to any address, redirect all block fees to themselves, and cause every validator to commit a block hash that encodes the wrong sequencer — with no detection at the `ProposalFin` comparison step.

---

### Finding Description

**Proposer side** (`crates/apollo_consensus_orchestrator/src/build_proposal.rs`):

```rust
let init = ProposalInit {
    ...
    builder: args.builder_address,   // from static config
    ...
};
```

A malicious proposer substitutes `builder` with an attacker-controlled address before broadcasting `ProposalPart::Init`.

**Validator side** — `is_proposal_init_valid` (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs` lines 252–419):

The function checks `height`, `l1_da_mode`, `l2_gas_price_fri`, all four L1 gas prices, `starknet_version`, `version_constant_commitment`, and `fee_proposal_fri`. It never reads or compares `init_proposed.builder`. [1](#0-0) 

**Conversion to block info** (`crates/apollo_consensus_orchestrator/src/utils.rs`):

```rust
Ok(starknet_api::block::BlockInfo {
    block_number: init.height,
    block_timestamp: BlockTimestamp(init.timestamp),
    sequencer_address: init.builder,   // attacker-supplied, never validated
    ...
})
``` [2](#0-1) 

This `BlockInfo` is forwarded to the batcher via `ValidateBlockInput` in `initiate_validation`. [3](#0-2) 

**Block hash computation** (`crates/starknet_api/src/block_hash/block_hash_calculator.rs`):

`PartialBlockHashComponents::new` stores `sequencer: SequencerContractAddress(block_info.sequencer_address)`, and `calculate_block_hash` chains it into the Poseidon hash. [4](#0-3) [5](#0-4) 

**Why `ProposalFin` comparison does not catch this:**

Both the proposer and every validator derive the `PartialBlockHash` from the same `init.builder` value. The comparison at line 244 (`built_block != received_fin.proposal_commitment`) therefore passes even when `builder` is attacker-controlled, because both sides computed the same wrong hash. [6](#0-5) 

---

### Impact Explanation

**Fee theft (Critical — economic impact):** In Starknet execution, `sequencer_address` is the address that receives transaction fees. By setting `builder` to an attacker-controlled address, the malicious proposer redirects every fee paid in that block to themselves. This is a direct, quantifiable economic loss for the legitimate sequencer operator.

**Wrong block hash committed (Critical — wrong state commitment):** The `sequencer_address` is a first-class input to `calculate_block_hash`. A block committed with an attacker-supplied `builder` produces a `BlockHash` that does not correspond to the legitimate sequencer. This wrong hash propagates to L1 anchoring and to the `retrospective_block_hash` used in subsequent blocks.

---

### Likelihood Explanation

In Tendermint-based consensus, the proposer role rotates among validators. Any validator that reaches its proposer turn can exploit this without any special privilege. The attack requires only crafting a `ProposalInit` with a modified `builder` field — a single-field substitution in a protobuf message. No cryptographic material needs to be forged. The `ProposalInitValidation` struct does not carry a `builder` field, so there is no locally-trusted reference value to compare against. [7](#0-6) 

---

### Recommendation

Add `builder` (and optionally `proposer`) to `ProposalInitValidation` and enforce the check inside `is_proposal_init_valid`:

```rust
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub builder: ContractAddress,   // add this
    ...
}
```

```rust
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The validator's `builder_address` is already available from `self.config.static_config.builder_address` and should be threaded into `ProposalInitValidation` at the call sites in `sequencer_consensus_context.rs`. [8](#0-7) 

---

### Proof of Concept

1. A malicious validator waits for its proposer turn at height H.
2. It constructs a `ProposalInit` identical to what a legitimate proposer would send, except `builder` is set to `attacker_address`.
3. It broadcasts `ProposalPart::Init(malicious_init)` followed by a valid transaction batch and `ProposalFin`.
4. Every honest validator calls `is_proposal_init_valid` — the check passes because `builder` is never read.
5. Each validator calls `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_address`.
6. The batcher executes the block; all fees are credited to `attacker_address`.
7. `PartialBlockHashComponents::new` stores `sequencer = attacker_address`; `calculate_block_hash` produces `H_wrong`.
8. Both the proposer's `ProposalFin.proposal_commitment` and the validator's locally-computed `batcher_block_commitment` equal `proposal_commitment_from(H_wrong, fee_proposal)` — the comparison at line 244 passes.
9. Consensus decides on the block; `H_wrong` is committed to storage and eventually anchored to L1.
