### Title
Caller-Agnostic `VerifyForeignTransactionRequest` Queue Key Allows Any Caller to Saturate the Fan-Out Cap and Permanently Block Bridge Verification — (File: `crates/contract/src/pending_requests.rs`)

---

### Summary

The `pending_verify_foreign_tx_requests` map uses a request key that does **not** include the caller's account ID. Any unprivileged caller can therefore fill the entire `MAX_PENDING_REQUEST_FAN_OUT = 128` slot queue for a specific foreign-chain transaction at a cost of 128 yoctoNEAR (≈ $0). Once the queue is full, every subsequent legitimate `verify_foreign_transaction` call for that transaction panics with `PendingRequestQueueFull`. The attacker can repeat the fill after each NEAR yield timeout, permanently blocking the bridge verification flow for any targeted transaction.

---

### Finding Description

`verify_foreign_transaction` stores pending yield indices in `pending_verify_foreign_tx_requests`, a `LookupMap<VerifyForeignTransactionRequest, Vec<YieldIndex>>`. The map key is `VerifyForeignTransactionRequest`:

```rust
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
``` [1](#0-0) 

The key contains **no caller identity** (`predecessor_account_id` is absent). This is by design — the contract comment explicitly calls it "caller-agnostic" — but it creates the attack surface. [2](#0-1) 

`push_pending_yield` enforces a hard cap of `MAX_PENDING_REQUEST_FAN_OUT = 128` per key and panics if exceeded:

```rust
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
    env::panic_str(
        &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }
            .to_string(),
    );
}
``` [3](#0-2) 

The minimum deposit for `verify_foreign_transaction` is `MINIMUM_SIGN_REQUEST_DEPOSIT = 1 yoctoNEAR`: [4](#0-3) 

The deposit is **not refunded on yield timeout** — only excess above the minimum is refunded. When a yield times out, `return_verify_foreign_tx_and_clean_state_on_success` pops the slot and calls `fail_on_timeout`; no deposit is returned to the caller: [5](#0-4) 

**Attack path:**

1. Attacker monitors the foreign chain (e.g., Bitcoin) for a bridge deposit transaction with a known `tx_id`.
2. Before or immediately after the legitimate user calls `verify_foreign_transaction`, the attacker submits 128 identical `verify_foreign_transaction` calls for the same `(request, domain_id, payload_version)` tuple, each with 1 yoctoNEAR deposit.
3. The queue for that request key is now full.
4. The legitimate user's call (or any retry) panics with `PendingRequestQueueFull`.
5. After the NEAR yield timeout drains the attacker's slots, the attacker immediately refills the queue. Total cost per round: 128 yoctoNEAR.

By contrast, `sign()` constructs its key as `SignatureRequest::new(domain_id, payload, &predecessor, &path)`, which **includes the caller's account ID**, so an attacker cannot fill another user's queue: [6](#0-5) 

---

### Impact Explanation

The `verify_foreign_transaction` flow is the on-chain entry point for the Omnibridge inbound bridge: a user sends assets on a foreign chain (Bitcoin, Ethereum, etc.) and then calls `verify_foreign_transaction` to have the MPC network attest the transaction and release corresponding assets on NEAR. If the attacker permanently blocks verification of a specific `tx_id`, the user's foreign-chain assets are already spent and irrecoverable, while the NEAR-side release never occurs. This constitutes **permanent freezing of funds in the verified foreign-chain flow**, matching the Critical impact tier.

---

### Likelihood Explanation

The attack requires only:
- Knowledge of the target `tx_id` (publicly visible on the foreign chain).
- 128 NEAR transactions with 1 yoctoNEAR each (negligible cost).
- Repeated refilling after each yield timeout (automatable).

No privileged access, threshold collusion, or TEE compromise is needed. Any unprivileged NEAR account can execute this.

---

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` map key, mirroring the approach used by `SignatureRequest`. Alternatively, enforce a per-caller sub-queue limit so a single account cannot consume more than a bounded fraction of the fan-out slots for any given request key.

---

### Proof of Concept

The existing unit test `verify_foreign_transaction__should_queue_duplicates_from_different_callers` already demonstrates that alice and bob share the same queue slot under one caller-agnostic key: [7](#0-6) 

Extending this: submit 128 calls from an attacker account for the same `(tx_id, domain_id, payload_version)`. The 129th call — from the legitimate user — will hit `push_pending_yield`'s cap check and panic. The attacker's 128 yields time out (no refund), and the attacker immediately resubmits 128 more, keeping the queue perpetually full at a cost of 128 yoctoNEAR per NEAR yield timeout period. [8](#0-7)

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
