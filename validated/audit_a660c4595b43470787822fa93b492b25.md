Audit Report

## Title
Stale `currentPeriodMintedAmount` Never Decremented on rsETH Burns Allows Premature Daily Mint Limit Exhaustion - (File: `contracts/RSETH.sol`)

## Summary
The `checkDailyMintLimit` modifier in `RSETH.sol` increments `currentPeriodMintedAmount` on every `mint()` call, but `burnFrom()` never decrements it. Because `LRTWithdrawalManager.instantWithdrawal()` is a user-callable function that burns rsETH without adjusting the period counter, an attacker can deposit ETH to inflate the counter, immediately burn via instant withdrawal to recover principal, and repeat until `maxMintAmountPerDay` is exhausted — blocking all new deposits for up to 24 hours.

## Finding Description
In `contracts/RSETH.sol` (lines 42–56), the `checkDailyMintLimit` modifier unconditionally adds the minted amount to `currentPeriodMintedAmount`:

```solidity
currentPeriodMintedAmount += amount;
```

The counter is only reset when `block.timestamp >= periodStartTime + 1 days`. The `burnFrom()` function (lines 245–248) performs no corresponding decrement:

```solidity
function burnFrom(address account, uint256 amount) external onlyRole(LRTConstants.BURNER_ROLE) whenNotPaused {
    _enforceNotBlocked(account);
    _burn(account, amount);
}
```

`burnFrom` is invoked by `LRTWithdrawalManager.instantWithdrawal()` (line 229), which is callable by any user when `isInstantWithdrawalEnabled[asset] == true`:

```solidity
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
```

Exploit flow:
1. Attacker calls `LRTDepositPool.depositETH{value: X}()` → `mint()` is called → `currentPeriodMintedAmount += X`.
2. Attacker calls `LRTWithdrawalManager.instantWithdrawal(ETH, X_rsETH, ...)` → `burnFrom()` is called → `currentPeriodMintedAmount` remains at `X`.
3. Attacker recovers `X * (1 - fee)` ETH. Net cost per cycle is only the instant withdrawal fee.
4. After enough cycles (or a single large deposit equal to `maxMintAmountPerDay`), `currentPeriodMintedAmount == maxMintAmountPerDay`.
5. Every subsequent `mint()` call reverts with `DailyMintLimitExceeded` for the remainder of the 24-hour period.

Existing guards are insufficient: the `onlyInstantWithdrawalAllowed` modifier only checks whether the feature is enabled, not whether the caller is cycling to inflate the counter. The `nonReentrant` guard prevents reentrancy but not sequential deposit-burn cycles.

## Impact Explanation
**Medium — Temporary freezing of funds.** All `LRTDepositPool.depositETH()` and `depositAsset()` calls invoke `IRSETH.mint()`, which will revert for the remainder of the period once `currentPeriodMintedAmount` reaches `maxMintAmountPerDay`. New depositors are completely blocked from minting rsETH for up to 24 hours. The protocol fails to deliver its core promised service for the duration of the period.

## Likelihood Explanation
**Low-to-Medium.** The attack requires `isInstantWithdrawalEnabled[asset]` to be `true` (a realistic operational state for the instant withdrawal feature) and sufficient assets in the unstaking vault (`getAssetsAvailableForInstantWithdrawal`). The attacker must front the capital equal to `maxMintAmountPerDay` but recovers the principal minus the fee on each cycle. With a low fee setting (e.g., 10–50 bps), the net cost to exhaust the daily limit is only a small fraction of `maxMintAmountPerDay`, making this economically viable as a griefing vector once instant withdrawal is live.

## Recommendation
Decrement `currentPeriodMintedAmount` in `burnFrom()` by the burned amount, capped to avoid underflow, and only within the current period:

```solidity
function burnFrom(address account, uint256 amount) external onlyRole(LRTConstants.BURNER_ROLE) whenNotPaused {
    _enforceNotBlocked(account);
    if (block.timestamp < periodStartTime + 1 days) {
        currentPeriodMintedAmount = amount > currentPeriodMintedAmount
            ? 0
            : currentPeriodMintedAmount - amount;
    }
    _burn(account, amount);
}
```

This ensures that burns within the same period correctly reduce the tracked minted amount, preventing the counter from being permanently inflated by deposit-burn cycles.

## Proof of Concept
Assume `maxMintAmountPerDay = 1000 ether`, `instantWithdrawalFee = 50 bps`, instant withdrawal enabled for ETH, and sufficient ETH in the unstaking vault.

1. Attacker calls `LRTDepositPool.depositETH{value: 1000 ether}()` → 1000 rsETH minted → `currentPeriodMintedAmount = 1000 ether`.
2. Attacker calls `LRTWithdrawalManager.instantWithdrawal(ETH, 1000e18, "")` → 1000 rsETH burned → `currentPeriodMintedAmount` remains `1000 ether`. Attacker receives ≈995 ETH back.
3. Any subsequent `RSETH.mint()` call within the same period reverts:
   ```
   DailyMintLimitExceeded(1000 ether + newAmount, 1000 ether)
   ```
4. All new depositors are blocked for up to 24 hours. Net attacker cost: ≈5 ETH in fees.

**Foundry test plan:** Deploy `RSETH`, `LRTDepositPool`, `LRTWithdrawalManager`, and `LRTUnstakingVault` with `maxMintAmountPerDay = 1000 ether` and `instantWithdrawalFee = 50`. Enable instant withdrawal for ETH. Seed the unstaking vault with 1000 ETH. Call `depositETH{value: 1000 ether}()` then `instantWithdrawal(ETH, 1000e18, "")`. Assert `currentPeriodMintedAmount == 1000 ether`. Then attempt another `depositETH{value: 1 ether}()` and assert it reverts with `DailyMintLimitExceeded`.
