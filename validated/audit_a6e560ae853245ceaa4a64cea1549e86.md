Audit Report

## Title
Stale `rsETHPrice` silently propagated through `RSETHPriceFeed.latestRoundData` with misleading `updatedAt` timestamp — (`contracts/oracles/RSETHPriceFeed.sol`)

## Summary
`RSETHPriceFeed.latestRoundData` computes a composite rsETH/USD price by multiplying the stored `rsETHPrice` from `LRTOracle` by a live ETH/USD Chainlink price, but returns `updatedAt` sourced solely from the ETH/USD feed. Because `LRTOracle` stores no timestamp for when `rsETHPrice` was last written, and `updateRSETHPrice()` is blocked during a protocol pause, the composite price can embed a stale rsETH/ETH rate while `updatedAt` appears fresh to any downstream consumer performing a standard staleness check.

## Finding Description
`RSETHPriceFeed.latestRoundData` (lines 63–70) fetches all five return values from `ETH_TO_USD.latestRoundData()` and then overwrites only `answer` with the composite price. `updatedAt` is never adjusted:

```solidity
(roundId, answer, startedAt, updatedAt, answeredInRound) = ETH_TO_USD.latestRoundData();
answer = int256(RS_ETH_ORACLE.rsETHPrice()) * answer / 1e18;
```

`RS_ETH_ORACLE.rsETHPrice()` reads the storage variable `rsETHPrice` in `LRTOracle` (line 28), which is written only inside `_updateRsETHPrice()` at line 313. No timestamp is stored alongside that write. The two public entry points to trigger a refresh are:

- `updateRSETHPrice()` — public but `whenNotPaused` (line 87).
- `updateRSETHPriceAsManager()` — manager-only (line 94).

The pause-on-price-drop path (lines 277–281) calls `_pause()` and returns immediately, leaving `rsETHPrice` at its pre-drop value and simultaneously blocking the public refresh path. During any pause of meaningful duration, `rsETHPrice` is frozen while `updatedAt` continues to reflect the freshness of the ETH/USD Chainlink feed, which updates independently. A downstream lending protocol calling `latestRoundData()` will pass its own staleness check (e.g., `require(block.timestamp - updatedAt < 3600)`) while consuming a composite price that embeds a potentially hours- or days-old rsETH/ETH rate.

## Impact Explanation
The `RSETHPriceFeed` contract fails to deliver the accurate staleness signal it promises through the standard Chainlink `AggregatorV3Interface`. Downstream integrators relying on `updatedAt` for freshness validation will silently accept a stale composite price. This matches **Low — contract fails to deliver promised returns, but doesn't lose value** within the LRT-rsETH scope: no funds are directly lost inside these contracts, but the oracle interface guarantee is broken.

## Likelihood Explanation
Under normal operation, `updateRSETHPrice()` is callable by anyone, so keeper infrastructure can maintain freshness. However, the pause-on-price-drop mechanism (lines 277–281) is an on-chain, automatic trigger that requires no attacker action. Any price deviation exceeding `pricePercentageLimit` will pause the protocol and simultaneously freeze `rsETHPrice` while blocking the public refresh path. This is a realistic, self-contained scenario. Even outside a pause, keeper failure silently accumulates staleness with no on-chain signal.

## Recommendation
Track the timestamp of the last `rsETHPrice` update and expose it through the oracle interface:

1. Add `uint256 public rsETHPriceUpdatedAt` to `LRTOracle` and set it to `block.timestamp` alongside `rsETHPrice = newRsETHPrice` at line 313.
2. Expose it via `IRSETHOracle` (add `rsETHPriceUpdatedAt()` to the interface).
3. In `RSETHPriceFeed.latestRoundData` (and `getRoundData`), override `updatedAt` with `min(updatedAt, RS_ETH_ORACLE.rsETHPriceUpdatedAt())` so downstream consumers see the true staleness of the composite price.
