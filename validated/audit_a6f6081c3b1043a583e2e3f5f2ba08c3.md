### Title
Sub-Transaction `guards` Field Populates `txInfoSignatories` Without Witness Enforcement, Enabling Plutus Script Authorization Bypass - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxBody.hs`)

---

### Summary

In the Dijkstra era, the `reqSignerHashesTxBodyG` getter for `DijkstraEra` is redefined to read from the `guardsTxBodyL` field instead of a dedicated `reqSignerHashes` field. For a sub-transaction (`DijkstraSubTxBodyRaw`), this means `txInfoSignatories` in the Plutus V4 script context is populated from the sub-transaction's own `dstbrGuards` field. However, the top-level witness-requirement check (`getConwayWitsVKeyNeeded`) only reads guards from the **top-level** transaction body (`dtbrGuards`), not from any sub-transaction's `dstbrGuards`. An attacker who authors a sub-transaction can therefore insert arbitrary key hashes into `dstbrGuards`, causing those hashes to appear in `txInfoSignatories` for any Plutus V4 script running in that sub-transaction context — without those keys ever signing the enclosing top-level transaction.

---

### Finding Description

**Root cause — `reqSignerHashesTxBodyG` reads sub-transaction guards:**

In `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/TxBody.hs`, the `AlonzoEraTxBody DijkstraEra` instance overrides the getter:

```haskell
reqSignerHashesTxBodyG = guardsTxBodyL . to (foldr' insertKeyHash mempty . OSet.toSet)
  where
    insertKeyHash (KeyHashObj kh) = Set.insert $ coerceKeyRole kh
    insertKeyHash (ScriptHashObj _) = id
``` [1](#0-0) 

`guardsTxBodyL` dispatches on the GADT constructor:

```haskell
guardsDijkstraTxBodyRawL = lens
  (\case
    DijkstraTxBodyRaw {dtbrGuards} -> dtbrGuards
    DijkstraSubTxBodyRaw {dstbrGuards} -> dstbrGuards)
  ...
``` [2](#0-1) 

So for a sub-transaction body, `reqSignerHashesTxBodyG` returns the key hashes from `dstbrGuards` — a field the sub-transaction author controls freely.

**`txInfoSignatories` is built from this getter:**

`transTxBodyReqSignerHashes` calls `txBody ^. reqSignerHashesTxBodyG`:

```haskell
transTxBodyReqSignerHashes :: AlonzoEraTxBody era => TxBody t era -> [PV1.PubKeyHash]
transTxBodyReqSignerHashes txBody = transKeyHash <$> Set.toList (txBody ^. reqSignerHashesTxBodyG)
``` [3](#0-2) 

In the Dijkstra PlutusV4 `mkAnyLevelTxInfo` (which handles both top-level and sub-transaction contexts), `txInfoSignatories` is set directly from this:

```haskell
, PV3.txInfoSignatories = Alonzo.transTxBodyReqSignerHashes txBody
``` [4](#0-3) 

**Witness enforcement only covers the top-level transaction:**

`getWitsVKeyNeeded` for `DijkstraEra` delegates to `getConwayWitsVKeyNeeded`:

```haskell
getWitsVKeyNeeded _ = getConwayWitsVKeyNeeded
``` [5](#0-4) 

`getConwayWitsVKeyNeeded` reads `reqSignerHashesTxBodyG` from the **top-level** transaction body passed to it:

```haskell
getConwayWitsVKeyNeeded utxo txBody =
  getShelleyWitsVKeyNeededNoGov utxo txBody
    `Set.union` Set.map asWitness (txBody ^. reqSignerHashesTxBodyG)
    `Set.union` voterWitnesses txBody
``` [6](#0-5) 

The UTXOW rule calls this on the top-level transaction body only. Sub-transaction `dstbrGuards` are never fed into the witness-requirement set. There is no separate UTXOW pass for sub-transactions.

**`DijkstraSubTxBodyRaw` has no `reqSignerHashes` field:**

The sub-transaction raw body has `dstbrGuards` but no dedicated `reqSignerHashes` field:

```haskell
DijkstraSubTxBodyRaw ::
  { dstbrSpendInputs :: !(Set TxIn)
  , ...
  , dstbrGuards :: !(OSet (Credential Guard))
  , ...
  , dstbrRequiredTopLevelGuards :: !(Map (Credential Guard) (StrictMaybe (Data era)))
  ...
  } -> DijkstraTxBodyRaw SubTx era
``` [7](#0-6) 

The `reqSignerHashesTxBodyL` lens is explicitly disabled for `DijkstraEra`:

```haskell
reqSignerHashesTxBodyL = notSupportedInThisEraL
``` [8](#0-7)

### Impact Explanation

**Critical — Direct loss of ADA or native assets through an invalid ledger state transition.**

Any Plutus V4 script that guards a UTxO and uses `txSignedBy txInfo someKeyHash` (or equivalent `txInfoSignatories` membership check) to authorize spending can be bypassed. An attacker who constructs a sub-transaction spending that UTxO simply sets `dstbrGuards` to include `KeyHashObj someKeyHash`. The ledger's UTXOW rule never checks that `someKeyHash` actually signed the enclosing top-level transaction, yet the script sees it in `txInfoSignatories` and passes. The UTxO is consumed and its ADA or native assets are transferred to attacker-controlled outputs, constituting a direct, irreversible fund loss.

---

### Likelihood Explanation

**High.** The Dijkstra era is the production target for sub-transactions. The attack requires only:
1. A UTxO locked by a Plutus V4 script that checks `txInfoSignatories` for authorization (a standard pattern).
2. The ability to author a sub-transaction — any unprivileged transaction sender can do this.
3. Setting `dstbrGuards` to the desired key hash — a freely settable field with no ledger-enforced constraint tying it to actual witnesses.

No privileged access, key leakage, or consensus majority is required. The attacker controls the sub-transaction body entirely.

---

### Recommendation

1. **Decouple `txInfoSignatories` from `dstbrGuards` for sub-transactions.** The `reqSignerHashesTxBodyG` getter for sub-transaction bodies should either return the empty set (if sub-transactions are not intended to declare independent required signers) or read from a dedicated, witness-enforced field.

2. **Enforce sub-transaction guard witnesses at the top level.** If `dstbrGuards` is intended to declare required signers for sub-transaction scripts, the UTXOW rule (or a dedicated sub-transaction validation pass) must include those key hashes in the top-level witness requirement set, analogous to how `reqSignerHashes` is included via `witsVKeyNeeded` in Alonzo/Conway.

3. **Alternatively, propagate top-level `txInfoSignatories` into sub-transaction script contexts.** If the design intent is that sub-transaction scripts should see the signers of the enclosing top-level transaction, `mkAnyLevelTxInfo` for sub-transactions should read `txInfoSignatories` from the top-level transaction body, not the sub-transaction body.

---

### Proof of Concept

**Setup:** Deploy a UTxO locked by a Plutus V4 script `S` that validates only when `txSignedBy txInfo aliceKeyHash` is true (i.e., Alice's key hash appears in `txInfoSignatories`).

**Attack:**
1. Attacker (Bob) constructs a top-level Dijkstra transaction `T` with no guards (`dtbrGuards = empty`).
2. Bob embeds a sub-transaction `SubTx` inside `T` via `dtbrSubTransactions`.
3. `SubTx` spends the UTxO locked by `S`, and sets `dstbrGuards = OSet.singleton (KeyHashObj aliceKeyHash)`.
4. Bob signs `T` with only his own key. Alice does not sign.
5. The UTXOW rule computes required witnesses from the top-level body: `dtbrGuards = empty`, so no guard-derived witness is required. Bob's key satisfies the input-spending witness requirement. The transaction passes phase-1 validation.
6. During phase-2 script evaluation for `SubTx`, `mkAnyLevelTxInfo` builds `txInfoSignatories` from `SubTx`'s `reqSignerHashesTxBodyG`, which reads `dstbrGuards = {aliceKeyHash}`. The script `S` sees `aliceKeyHash` in `txInfoSignatories`, evaluates `txSignedBy txInfo aliceKeyHash = True`, and passes.
7. The UTxO is consumed. Alice's funds are transferred to Bob's output. Alice never signed. [9](#0-8) [10](#0-9) [5](#0-4)
