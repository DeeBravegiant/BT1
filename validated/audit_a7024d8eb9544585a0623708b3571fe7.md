Audit Report

## Title
Publicly Callable `updateRSETHPrice()` Enables Atomic Sandwich to Extract Staking Yield from rsETH Holders - (File: `contracts/LRTOracle.sol`)

## Summary

`LRTOracle.updateRSETHPrice()` is callable by any address with no access restriction. Because `LRTDepositPool.depositETH` mints rsETH using the stale stored `rsETHPrice` and `LRTWithdrawalManager.instantWithdrawal` redeems rsETH at the same stored value, an attacker can atomically deposit at the stale lower price, trigger the price update themselves, and immediately withdraw at the higher price — extracting staking yield that belongs to existing rsETH holders.

## Finding Description

**Root cause:** `rsETHPrice` is a lazily-updated state variable. Any caller can advance it at will via the unrestricted public function:

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
``` [1](#0-0) 

Both the deposit and instant-withdrawal paths read this stored value directly rather than computing a fresh price:

```solidity
// LRTDepositPool.sol L520 — lower rsETHPrice → more rsETH minted
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
``` [2](#0-1) 

```solidity
// LRTWithdrawalManager.sol L593 — higher rsETHPrice → more ETH returned
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [3](#0-2) 

**Exploit path (single atomic transaction):**

1. Observe `rsETHPrice` is stale at `P1`; true NAV is `P2 > P1` (rewards have accrued).
2. Call `LRTDepositPool.depositETH{value: X}()` → receive `X * 1e18 / P1` rsETH (over-minted relative to true price).
3. Call `LRTOracle.updateRSETHPrice()` → `rsETHPrice` advances to `P2`.
4. Call `LRTWithdrawalManager.instantWithdrawal(ETH, rsETHAmount, "")` → receive `rsETHAmount * P2 / 1e18` ETH.
5. Net profit: `X * (P2/P1 − 1)` ETH, minus `instantWithdrawalFee`.

**Why existing guards are insufficient:**

The `pricePercentageLimit` check at `LRTOracle.sol:252-266` only reverts a non-manager caller when the price increase *exceeds* the configured threshold. [4](#0-3) 

Staking rewards accrue continuously at a small daily rate (e.g., ~0.01 %/day), which is always within any reasonable `pricePercentageLimit`. If `pricePercentageLimit == 0` (unset), there is no cap whatsoever. The guard therefore does not prevent the attack for normal reward accrual windows.

The `instantWithdrawal` function checks `isInstantWithdrawalEnabled[asset]` and `CantInstantWithdrawMoreThanAvailable`, but neither prevents the price-gap extraction — they only gate whether the path is open and whether vault liquidity is sufficient. [5](#0-4) [6](#0-5) 

## Impact Explanation

**High — Theft of unclaimed yield.**

Every unit of staking reward captured by the attacker is a unit not reflected in the NAV of honest rsETH holders. The attacker extracts yield that accrued to the existing holder base, proportional to the price gap `P2 − P1` and the attack size. This is a direct, repeatable transfer of value from existing holders to the attacker, matching the "High. Theft of unclaimed yield" impact class.

## Likelihood Explanation

- `updateRSETHPrice()` is unconditionally public; any EOA or contract can call it at any time.
- Staking rewards accrue continuously; the price is always stale between oracle updates, creating a persistent exploitable window.
- `isInstantWithdrawalEnabled` is a live, manager-configurable feature; when enabled for ETH, the full round-trip is executable in one transaction.
- `pricePercentageLimit` only blocks jumps above the threshold; normal daily reward accrual is always below any reasonable limit.
- The attack is repeatable every time rewards accrue, and can be executed as a mempool sandwich against any pending `updateRSETHPrice()` call.

## Recommendation

1. **Restrict `updateRSETHPrice()`** to an authorized role (manager/operator/keeper), removing the public entry point. The existing `updateRSETHPriceAsManager()` already provides the privileged path.
2. **Alternatively**, call `_updateRsETHPrice()` at the start of every `depositETH`, `depositAsset`, and `instantWithdrawal` execution so the price is always fresh before amounts are calculated, eliminating the stale-price window entirely.
3. **As a secondary defence**, ensure `instantWithdrawalFee` is always set above the maximum per-call price increase permitted by `pricePercentageLimit`.

## Proof of Concept

```
State:
  rsETHPrice (stored) = 1.05e18
  true NAV             = 1.06e18  (rewards accrued, not yet updated)
  isInstantWithdrawalEnabled[ETH] = true
  instantWithdrawalFee = 0 bps (or < 95 bps to remain profitable)

Attacker contract — single transaction:

  Step 1: depositETH{value: 1e18}()
    rsETHMinted = 1e18 * 1e18 / 1.05e18
               = 952_380_952_380_952_380 rsETH

  Step 2: lrtOracle.updateRSETHPrice()
    rsETHPrice → 1.06e18

  Step 3: withdrawalManager.instantWithdrawal(ETH, 952_380_952_380_952_380, "")
    assetAmountUnlocked = 952_380_952_380_952_380 * 1.06e18 / 1e18
                        = 1_009_523_809_523_809_523 wei (~1.0095 ETH)

Net gain: ~9_523_809_523_809_523 wei (~0.0095 ETH, ~0.95%) per 1 ETH deposited,
minus instantWithdrawalFee.

Foundry fork test outline:
  1. Fork mainnet at a block where rewards have accrued since last updateRSETHPrice().
  2. Deploy attacker contract that executes steps 1–3 atomically.
  3. Assert attacker ETH balance after > before (net of deposit).
  4. Assert existing rsETH holders' redeemable value decreased by the extracted amount.
```
