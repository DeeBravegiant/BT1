### Title
Unverified CKD Response Accepted for Legacy `AppPublicKey` Requests — (File: `crates/contract/src/lib.rs`)

---

### Summary

In `respond_ckd`, when a CKD request uses the legacy `AppPublicKey` variant, the contract resolves the pending yield and delivers the response to the user **without performing any cryptographic verification** of the CKD output. A single attested participant — strictly below the signing threshold — can submit a fabricated CKD response that the contract unconditionally accepts, bypassing the threshold requirement for confidential key derivation.

---

### Finding Description

The `respond_ckd` function contains a conditional branch that mirrors the structure of the external report's `deposit_sol` flag:

```rust
// crates/contract/src/lib.rs ~675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

When `app_public_key` is the `AppPublicKey` (privately verifiable / legacy) variant, the arm is **empty** — execution falls through directly to `resolve_yields_for`, which resolves the on-chain yield and delivers the response to the waiting user. No property of the CKD output is checked against the master public key or the user's ephemeral key.

By contrast, when `app_public_key` is `AppPublicKeyPV`, `ckd_output_check` is called and the response is rejected if it does not satisfy the publicly verifiable relation. The asymmetry is exact: the flag (`AppPublicKey` vs `AppPublicKeyPV`) controls whether verification runs, but in both cases the yield is resolved and state is updated.

The `respond` function for ordinary signatures does not have this gap — it always derives the expected public key and verifies the ECDSA signature before resolving:

```rust
// crates/contract/src/lib.rs ~597-608
let expected_public_key =
    derive_key_secp256k1(&affine, &request.tweak).map_err(RespondError::from)?;
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
).is_ok()
``` [2](#0-1) 

CKD with `AppPublicKey` has no equivalent guard.

---

### Impact Explanation

A single attested MPC participant (one node, strictly below the signing threshold) can call `respond_ckd` with an arbitrary `CKDResponse` for any pending `AppPublicKey` request. The contract will:

1. Pass the `assert_caller_is_attested_participant_and_protocol_active` check (one participant suffices).
2. Enter the empty `AppPublicKey` arm — no output verification.
3. Serialize and deliver the fabricated response to the user via `resolve_yields_for`.

The user receives key material that was **not produced by the threshold MPC protocol** and may have been chosen by the attacker. Depending on how the user employs the derived key (e.g., decrypting bridge-locked assets, signing on behalf of an application), this constitutes unauthorized confidential key derivation output without the required threshold participant authorization — matching the Critical impact class.

---

### Likelihood Explanation

The attacker must be a currently attested MPC participant. This is a meaningful barrier (TEE attestation required), but it is **strictly below the threshold** — no collusion with other participants is needed. A single compromised or malicious node operator can exploit this for every pending legacy CKD request. Legacy `AppPublicKey` requests are explicitly documented as still supported, so the attack surface is live.

---

### Recommendation

Apply the same output check to the `AppPublicKey` variant, or reject `AppPublicKey` requests at the `respond_ckd` level and require all new requests to use `AppPublicKeyPV`. At minimum, the `AppPublicKey` arm must not be a silent no-op when the response is about to be committed to on-chain state:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {
        // Either verify with an equivalent check, or reject:
        env::panic_str("Legacy AppPublicKey responses are not accepted; use AppPublicKeyPV");
    }
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key = AppPublicKey(user_pk)` and attaches 1 yoctoNEAR. The request is enqueued in `pending_ckd_requests`.
2. Malicious attested participant constructs an arbitrary `CKDResponse` (e.g., all-zero encrypted key material, or material encrypted to an attacker-controlled key).
3. Malicious participant calls `respond_ckd(request, fabricated_response)`.
4. Contract enters the `AppPublicKey` arm — **no check executes**.
5. `resolve_yields_for` resolves the yield with the fabricated bytes; the user's awaiting transaction receives the fake CKD output.
6. The user's application decrypts or uses the attacker-chosen key material, giving the attacker knowledge of or control over the derived key — without any threshold of honest participants having participated. [3](#0-2)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
