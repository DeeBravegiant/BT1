### Title
Unsigned `SpiceChunkStateWitness` Allows Any Peer to Inject Malicious Base State, Causing Validators to Endorse Wrong Execution Results - (File: `chain/chain/src/spice/chunk_validation.rs`, `core/primitives/src/spice/state_witness.rs`)

### Summary

In the SPICE execution path, `SpiceChunkStateWitness` carries no producer signature. The data-distribution layer verifies only internal Merkle consistency of the encoded parts, not the identity of the sender. Any network peer can therefore frontrun the legitimate chunk executor by sending a witness with a crafted `base_state` (wrong account trie values) and a self-consistent `post_state_root` / `execution_result_hash`. Chunk validators accept and re-execute against the malicious base state, compute the attacker-chosen post-state root, and emit signed endorsements for the wrong `ChunkExecutionResult`. If a quorum of validators is reached before the honest witness arrives, the wrong execution result is accepted as canonical, corrupting account balances and outgoing receipt roots.

### Finding Description

`SpiceChunkStateWitnessV1` is defined without any signature field:

```rust
pub struct SpiceChunkStateWitnessV1 {
    pub chunk_id: SpiceChunkId,
    pub main_state_transition: SpiceChunkStateTransition,  // contains base_state
    pub source_receipt_proofs: HashMap<ShardId, ReceiptProof>,
    pub applied_receipts_hash: CryptoHash,
    pub transactions: Vec<SignedTransaction>,
    pub execution_result_hash: ChunkExecutionResultHash,
    pub contract_accesses_hash: CryptoHash,
    pub proof_of_invalid_chunk: Option<Box<EncodedShardChunkBody>>,
}
``` [1](#0-0) 

Contrast this with `SpiceChunkContractAccesses`, which carries a `signature` field and is explicitly verified against the set of epoch chunk producers before being trusted: [2](#0-1) 

The witness arrives via the SPICE data-distribution layer. `SpiceDataDistributorActor` verifies only that each part's Merkle proof is consistent with the commitment root, and that the decoded data's hash matches `commitment.hash`. Neither the commitment root nor the commitment hash is signed by the chunk executor: [3](#0-2) 

When `SpiceChunkValidatorActor` receives the reconstructed witness it calls `process_chunk_state_witness`, which performs no sender-identity check: [4](#0-3) 

`spice_pre_validate_chunk_state_witness` validates receipts and transactions against on-chain data, but consumes `base_state` verbatim as the recorded storage for re-execution: [5](#0-4) 

`spice_validate_chunk_state_witness` then checks only that the computed post-state root equals the witness-supplied `post_state_root`, and that the computed `execution_result_hash` equals the witness-supplied value — both of which the attacker controls: [6](#0-5) 

Because all three values (`base_state`, `post_state_root`, `execution_result_hash`) are attacker-supplied and only checked for mutual consistency, a self-consistent but malicious triple passes every check.

### Impact Explanation

A successful attack causes chunk validators to emit `SpiceChunkEndorsement` messages signed with their own keys but attesting to an attacker-chosen `ChunkExecutionResult`. The `ChunkExecutionResult` contains `chunk_extra` (which embeds the wrong `state_root` and wrong validator rewards) and `outgoing_receipts_root`. If a quorum of validators endorses the malicious result before the honest witness is processed, the SPICE core writer accepts the wrong execution result as canonical. This corrupts:

- **Account balances** (state root derived from wrong base state)
- **Outgoing receipt roots** (receipts computed from wrong state)
- **Validator proposals** embedded in `chunk_extra`

The corrupted `execution_result_hash` is the exact protocol value that determines finality in SPICE.

### Likelihood Explanation

Any peer connected to the validator network can send `SpicePartialData` messages. The attacker only needs to deliver their malicious parts to a quorum of chunk validators before the legitimate chunk executor's parts arrive. A well-connected adversarial peer with low-latency links to validators can reliably win this race. No validator, block-producer, or node-operator privilege is required.

### Recommendation

1. **Add a producer signature to `SpiceChunkStateWitness`**: The chunk executor must sign the witness (or at minimum sign a commitment to it) with its validator key before distribution. Validators must verify this signature against the expected chunk executor for the given `SpiceChunkId` before accepting the witness.

2. **Sign the data-distribution commitment**: The `SpicePartialData` commitment (root + hash) should be signed by the chunk executor so that `SpiceDataDistributorActor` can reject commitments from non-authoritative senders, analogous to how `SpiceChunkContractAccesses` is signed and verified.

3. **Verify sender identity in `handle_incoming_partial_data`**: Before accepting a decoded `SpiceData::StateWitness`, confirm the sender is the expected chunk executor for the given `block_hash` and `shard_id`.

### Proof of Concept

1. Observe a new block `B` at height `H` for shard `S`.
2. Determine the chunk executor `E` for `(B, S)` from the epoch manager.
3. Obtain the correct transactions (from the chunk header's `tx_root`) and receipts (from the previous block's execution results, which are public).
4. Construct a modified `base_state` with a reduced balance for a target account (e.g., subtract 1 NEAR from account `alice.near`'s trie node).
5. Simulate chunk execution locally against the modified `base_state` to obtain the attacker-chosen `post_state_root'` and `execution_result_hash'`.
6. Build a `SpiceChunkStateWitness` with the modified `base_state`, `post_state_root'`, and `execution_result_hash'`; all receipt and transaction fields are correct.
7. Reed-Solomon encode the witness, compute a valid Merkle commitment, and send `SpicePartialData` messages to all chunk validators for `(B, S)` before `E` does.
8. Each validator reconstructs the witness, calls `spice_pre_validate_chunk_state_witness` (passes — receipts and transactions are correct), then `spice_validate_chunk_state_witness` (passes — `post_state_root'` matches re-execution against the modified `base_state`), and emits a `SpiceChunkEndorsement` for `execution_result_hash'`.
9. The SPICE core writer collects a quorum of endorsements for the wrong hash and records the wrong `ChunkExecutionResult` as canonical, permanently corrupting `alice.near`'s balance.

#[derive(Debug, Clone, PartialEq, Eq, BorshSerialize, BorshDeserialize, ProtocolSchema)]
pub struct SpiceChunkStateWitnessV1 {
    /// Witness contains information to derive execution results of chunk corresponding to
    /// chunk_id.
    pub chunk_id: SpiceChunkId,
    /// The base state and post-state-root of the main transition where we
    /// apply transactions and receipts. Corresponds to the state transition
    /// that takes us from the pre-state-root of the chunk of this shard to
    /// the post-state-root of that same chunk.
    pub main_state_transition: SpiceChunkStateTransition,
    /// For the main state transition, we apply transactions and receipts.
    /// Exactly which of them must be applied is a deterministic property
    /// based on the blockchain history this chunk is based on.
    ///
    /// The set of receipts is exactly
    ///   Filter(R, |receipt| receipt.target_shard = S), where
    ///     - R is the set of outgoing receipts included in the set of chunks of
    ///       previous block.
    ///     - S is the shard of this chunk.
    ///
    /// This field, `source_receipt_proofs`, is a (non-strict) superset of the
    /// receipts that must be applied, along with information that allows these
    /// receipts to be verifiable against the blockchain history.
    pub source_receipt_proofs: HashMap<ShardId, ReceiptProof>,
    /// An overall hash of the list of receipts that should be applied. This is
    /// redundant information but is useful for diagnosing why a witness might
    /// fail. This is the hash of the borsh encoding of the Vec<Receipt> in the
    /// order that they should be applied.
    pub applied_receipts_hash: CryptoHash,
    /// The transactions to apply. These must be in the correct order in which
    /// they are to be applied.
    pub transactions: Vec<SignedTransaction>,
    /// Hash of the execution results after executing the chunk. This is redundant
    /// since it can be calculated by applying the chunk, but it's still useful
    /// for debugging.
    pub execution_result_hash: ChunkExecutionResultHash,
    /// Hash of the contract code hashes accessed during chunk application.
    /// Validators use this to verify that the contract accesses message they
    /// received matches what the witness expects, preventing a malicious
    /// producer from injecting invalid contract accesses.
    pub contract_accesses_hash: CryptoHash,
    /// Proof that the chunk body is invalid (e.g. bad tx_root). Present when
    /// a malicious chunk producer sends an invalid chunk body. The body must
    /// be RS-reconstructed (all parts filled) so validators can independently
    /// verify the invalidity. When present, validators accept empty
    /// transactions instead of the chunk header's tx_root.
    pub proof_of_invalid_chunk: Option<Box<EncodedShardChunkBody>>,
}

impl SpiceChunkContractAccesses {
    pub fn new(
        chunk_id: SpiceChunkId,
        contracts: HashSet<CodeHash>,
        signer: &ValidatorSigner,
    ) -> Self {
        let inner = SpiceChunkContractAccessesInner::new(chunk_id, contracts);
        let signature = signer.sign_bytes(&borsh::to_vec(&inner).unwrap());
        Self { inner, signature }
    }

    pub fn chunk_id(&self) -> &SpiceChunkId {
        &self.inner.chunk_id
    }

    pub fn contracts(&self) -> &[CodeHash] {
        &self.inner.contracts
    }

    pub fn verify_signature(&self, public_key: &PublicKey) -> bool {
        self.signature.verify(&borsh::to_vec(&self.inner).unwrap(), public_key)
    }
