### Title
Malicious Peer Can Inject Fake `delayed_puzhash` via Unvalidated Launcher Spend Solution, Causing Pool Wallet to Subscribe to Wrong `p2_singleton_puzzle_hash` — (`chia/pools/pool_puzzles.py`, `chia/pools/pool_wallet.py`, `chia/wallet/util/wallet_sync_utils.py`)

---

### Summary

When a `PoolWallet` is first created during sync, the wallet fetches the launcher `CoinSpend` from its connected peer. `fetch_coin_spend` validates only the puzzle hash (must equal `SINGLETON_LAUNCHER_HASH`), but **never validates the solution**. A malicious peer can return the correct SINGLETON_LAUNCHER puzzle with a crafted solution whose `extra_data` contains attacker-controlled `delayed_puzhash` and `delayed_seconds`. `get_delayed_puz_info_from_launcher_spend` blindly reads these values, `launcher_id_to_p2_puzzle_hash` computes a wrong `p2_singleton_puzzle_hash`, and the wallet permanently subscribes to the wrong puzzle hash — causing all actual farming rewards to go untracked.

---

### Finding Description

**Step 1 — Peer fetch with no solution validation**

`fetch_coin_spend` in `chia/wallet/util/wallet_sync_utils.py` validates only two things:

```python
if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
    raise PeerRequestException(...)
if solution_response.response.coin_name != coin_id:
    raise PeerRequestException(...)
```

The solution is accepted verbatim from the peer. [1](#0-0) 

**Step 2 — Launcher spend detection and PoolWallet creation**

In `_add_coin_states`, when a child coin with `puzzle_hash == SINGLETON_LAUNCHER_HASH` is found spent, the wallet calls `fetch_coin_spend_for_coin_state(child, peer)` and passes the result directly to `PoolWallet.create`. [2](#0-1) 

**Step 3 — Crafted solution structure**

The SINGLETON_LAUNCHER solution format is `[puzzle_hash, amount, extra_data]`. The child coin's on-chain puzzle hash is determined solely by the first element. The attacker keeps `puzzle_hash` and `amount` correct (so `compute_additions` returns the real child coin), but substitutes a fake `extra_data` containing attacker-controlled `("t", delay_seconds)` and `("h", fake_delayed_puzhash)`.

**Step 4 — Unvalidated read of `delayed_puzhash`**

`get_delayed_puz_info_from_launcher_spend` reads `delayed_seconds` and `delayed_puzhash` directly from the solution's `extra_data` with no cross-check against the actual on-chain singleton inner puzzle:

```python
extra_data = Program.from_bytes(bytes(coinsol.solution)).rest().rest().first()
...
if key.atom == b"h":
    delayed_puzzle_hash = bytes32(value.atom)
``` [3](#0-2) 

**Step 5 — Wrong `p2_singleton_puzzle_hash` computed and stored**

`get_current_state` calls `get_delayed_puz_info_from_launcher_spend(all_spends[0])` and feeds the attacker-controlled values into `launcher_id_to_p2_puzzle_hash`:

```python
delayed_seconds, delayed_puzhash = get_delayed_puz_info_from_launcher_spend(all_spends[0])
p2_singleton_puzzle_hash = launcher_id_to_p2_puzzle_hash(launcher_id, delayed_seconds, delayed_puzhash)
``` [4](#0-3) 

**Step 6 — Subscription to wrong puzzle hash**

`PoolWallet.create` subscribes to the attacker-computed hash:

```python
p2_puzzle_hash: bytes32 = (await pool_wallet.get_current_state()).p2_singleton_puzzle_hash
await wallet_state_manager.add_interested_puzzle_hashes([p2_puzzle_hash], [pool_wallet.wallet_id])
``` [5](#0-4) 

The wrong hash is also persisted to the pool config file via `update_pool_config`. [6](#0-5) 

**No cross-validation exists.** `compute_additions(launcher_spend)` at line 2207 computes the child coin from the first solution element (which the attacker keeps correct), so the child coin check passes silently. [7](#0-6) 

---

### Impact Explanation

The wallet subscribes to an attacker-chosen `p2_singleton_puzzle_hash` instead of the real one. All farming rewards sent to the correct puzzle hash on-chain are invisible to the wallet — they accumulate untracked. The pool config file and the `pool_state_transitions` DB both store the wrong hash, so the corruption persists across restarts. The farmer also uses `p2_singleton_puzzle_hash` from the pool config to communicate with the pool, so pool-side reward accounting is also disrupted. This constitutes **corruption of pool membership state and wallet sync state with direct security impact** (farming rewards diverted/untracked).

---

### Likelihood Explanation

Any peer the wallet connects to can execute this attack. Chia wallets are designed to connect to untrusted full nodes; the wallet is supposed to validate peer-provided data. The attack requires no keys, no signatures, and no on-chain action — only returning a crafted `RespondPuzzleSolution` message for the launcher coin. The attack is triggered automatically on first sync of a new pool wallet.

---

### Recommendation

After fetching the launcher spend, cross-validate the `delayed_puzhash` and `delayed_seconds` from the solution against the actual on-chain singleton inner puzzle. Specifically:

1. Compute `compute_additions(launcher_spend)` to get the child (eve) coin.
2. Fetch the eve coin's puzzle reveal (its puzzle hash is known from the child coin on-chain).
3. Uncurry the singleton inner puzzle and call `get_seconds_and_delayed_puzhash_from_p2_singleton_puzzle` on the embedded `p2_singleton_puzzle_hash` to recover the canonical `delayed_seconds` and `delayed_puzhash`.
4. Assert these match what `get_delayed_puz_info_from_launcher_spend` returned from the solution.

Alternatively, derive `p2_singleton_puzzle_hash` by uncurrying the actual on-chain singleton inner puzzle rather than trusting the launcher solution's `extra_data`.

def get_delayed_puz_info_from_launcher_spend(coinsol: CoinSpend) -> tuple[uint64, bytes32]:
    extra_data = Program.from_bytes(bytes(coinsol.solution)).rest().rest().first()
    # Extra data is (pool_state delayed_puz_info)
    # Delayed puz info is (seconds delayed_puzzle_hash)
    seconds: uint64 | None = None
    delayed_puzzle_hash: bytes32 | None = None
    for key_value_pairs in extra_data.as_iter():
        key_value_pair = key_value_pairs.as_pair()
        if key_value_pair is None:
            continue
        key, value = key_value_pair
        if key.atom == b"t":
            seconds = uint64(value.as_int())
        if key.atom == b"h":
            assert value.atom is not None
            delayed_puzzle_hash = bytes32(value.atom)
    assert seconds is not None
    assert delayed_puzzle_hash is not None
    return seconds, delayed_puzzle_hash
```
