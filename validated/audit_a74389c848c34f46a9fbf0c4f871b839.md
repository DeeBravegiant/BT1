### Title
`MockAttestation::Valid` Bypasses TEE Verification in Production Contract Interface - (File: `crates/mpc-attestation/src/attestation.rs`, `crates/near-mpc-contract-interface/src/types/attestation.rs`)

---

### Summary

The production NEAR MPC contract exposes a `MockAttestation::Valid` variant through its public-facing `Attestation` enum. This variant unconditionally passes all TEE verification checks, analogous to the AuctionTimeControl stub functions that always return `true`. Any participant can submit `Attestation::Mock(MockAttestation::Valid)` to `submit_participant_info`, bypassing real Dstack/DCAP attestation and registering as a fully attested participant without running inside a genuine TEE.

---

### Finding Description

The `Attestation` enum in the public contract interface includes a `Mock(MockAttestation)` variant alongside the production `Dstack(DstackAttestation)` variant: [1](#0-0) 

`MockAttestation::Valid` is the default variant and its `verify_constraints` implementation unconditionally returns `Ok(())` with no checks whatsoever: [2](#0-1) 

This mock path is wired directly into the production `add_participant` flow via `attestation.verify_locally(...)`, which dispatches to `MockAttestation::verify` for the `Mock` arm: [3](#0-2) 

The DTO-to-internal mapping in `dto_mapping.rs` faithfully converts `dtos::MockAttestation::Valid` to the internal `MockAttestation::Valid` without any guard: [4](#0-3) 

The `TeeState::add_participant` function stores the resulting `VerifiedAttestation::Mock(MockAttestation::Valid)` into `stored_attestations` with no further check that the attestation type is `Dstack`: [5](#0-4) 

Furthermore, `TeeState::with_mocked_participant_attestations` seeds the initial participant set with `MockAttestation::Valid` entries, meaning the contract can operate entirely on mock attestations from genesis: [6](#0-5) 

The re-verification path (`re_verify`) also passes through the mock arm without enforcing real TEE properties: [7](#0-6) 

---

### Impact Explanation

A participant who submits `Attestation::Mock(MockAttestation::Valid)` is indistinguishable from a legitimately attested Dstack node in all downstream contract logic. Their `VerifiedAttestation` entry passes `reverify_participants`, keeps `accept_requests = true`, and allows them to call `respond`, `vote`, and other node-only methods. Because the node is not actually running inside a TEE, its key shares are not protected by hardware isolation — the attacker can exfiltrate partial key material from the signing protocol, which, combined with shares from other compromised or colluding nodes below the threshold, enables unauthorized key recovery or signature forgery.

**Impact class:** Critical — bypass of TEE-based participant authorization enabling unauthorized access to MPC key shares and signing capability.

---

### Likelihood Explanation

The `MockAttestation::Valid` variant is part of the stable, serialized, Borsh-encoded public interface (`near-mpc-contract-interface`). Any participant account can craft a JSON/Borsh payload containing `{"Mock": "Valid"}` and submit it to `submit_participant_info`. No special privilege, leaked key, or operator access is required beyond holding a participant slot, which is obtained through the normal resharing/initialization flow.

---

### Recommendation

1. **Remove `MockAttestation::Valid` from the production public interface.** Gate the entire `Mock` variant behind a compile-time feature flag (e.g., `#[cfg(feature = "test-utils")]`) so it is never present in WASM builds deployed to mainnet.
2. **Add an explicit runtime guard** in `add_participant` (or at the DTO boundary in `dto_mapping.rs`) that rejects any `Attestation::Mock(...)` submission when the contract is not in a designated test/dev mode.
3. **Audit `with_mocked_participant_attestations`** to ensure it is never reachable from a production initialization path.

---

### Proof of Concept

1. Obtain a participant slot via the normal resharing process.
2. Construct the following JSON call to `submit_participant_info`:
   ```json
   {
     "node_id": { "account_id": "attacker.near", "tls_public_key": "<key>", "account_public_key": "<key>" },
     "attestation": { "Mock": "Valid" }
   }
   ```
3. The contract deserializes `Attestation::Mock(MockAttestation::Valid)`, calls `verify_locally`, which dispatches to `MockAttestation::verify` → `verify_constraints` → `Ok(())` unconditionally.
4. `stored_attestations` now contains `VerifiedAttestation::Mock(MockAttestation::Valid)` for the attacker's TLS key.
5. `reverify_participants` returns `TeeQuoteStatus::Valid` for this entry on every subsequent check.
6. The attacker participates in threshold signing rounds outside a real TEE, with full access to their key share in plaintext memory. [8](#0-7) [9](#0-8)
