### Title
`respond_ckd` Unconditionally Resolves CKD Yields Without Verifying Legacy `AppPublicKey` Responses, Enabling Single-Participant Forgery — (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` always calls `resolve_yields_for` to deliver the CKD response to the user, but the on-chain correctness check (`ckd_output_check`) is **conditionally skipped** for the legacy `AppPublicKey` variant. This is structurally identical to the GMX finding: state is updated unconditionally even when the guard that should protect it was not consulted. A single malicious attested participant can call `respond_ckd` with a fabricated `CKDResponse` for any pending `AppPublicKey` request, bypassing the threshold requirement and delivering a forged confidential key derivation output to the user.

---

### Finding Description

In `respond_ckd` the contract branches on the `app_public_key` variant:

```rust
// crates/contract/src/lib.rs  ~675-688
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(          // ← ALWAYS EXECUTED
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the response is a well-formed BLS12-381 output consistent with the master public key and the app's ephemeral key pair. [2](#0-1) 

For `AppPublicKey` (the legacy "privately verifiable" variant), the branch body is empty — no check is performed — yet `resolve_yields_for` is still called, permanently consuming the pending yield and delivering whatever bytes the caller supplied. [3](#0-2) 

The only gate before this point is `assert_caller_is_attested_participant_and_protocol_active`, which requires the caller to be **one** attested participant — far below the signing threshold. [4](#0-3) 

This is the direct analog of the GMX pattern: `applyDeltaToPoolAmount` always called `applyDeltaToVirtualInventoryForSwaps` even when the virtual-impact path was skipped; here `resolve_yields_for` always delivers the response even when `ckd_output_check` was skipped.

---

### Impact Explanation

A single malicious attested participant can:

1. Observe any pending `AppPublicKey` CKD request (all request fields are stored in the public contract state under `pending_ckd_requests`).
2. Construct an arbitrary `CKDResponse` (`big_y`, `big_c` are free BLS12-381 G1 points with no constraint).
3. Call `respond_ckd(request, forged_response)` before honest nodes do.
4. The contract accepts the call, permanently resolves the yield, and returns the forged encrypted key to the user.

The user receives a confidential key derivation output that was never computed by the threshold MPC protocol. The pending request is consumed and cannot be retried with the same parameters. Any downstream use of the derived key (decryption, authentication, asset control) operates on attacker-controlled material.

This satisfies the Critical impact criterion: **confidential key derivation output without the required participant authorization** (threshold t-of-n is bypassed by a single participant).

---

### Likelihood Explanation

Every attested participant already has the access level required. Pending CKD request details (domain ID, app public key, predecessor, derivation path) are readable from on-chain state. The attacker only needs to submit `respond_ckd` before the honest quorum does — a race that is straightforward to win by monitoring the NEAR indexer and front-running. No collusion, no leaked keys, and no privileged operator access are required.

---

### Recommendation

Apply the same `ckd_output_check` guard unconditionally, regardless of the `app_public_key` variant:

```rust
// Verify the response for both variants before resolving yields.
if !ckd_output_check(&request.app_id, &response, &derived_app_pk, &public_key) {
    env::panic_str("CKD output check failed");
}
```

If the legacy `AppPublicKey` variant cannot support on-chain verification by construction, it should be deprecated and removed from the production contract, forcing all callers to use `AppPublicKeyPV`.

---

### Proof of Concept

1. Alice calls `request_app_private_key` with `AppPublicKey(bls12381g1:<alice_pk>)` and attaches 1 yoctoNEAR. The request is stored in `pending_ckd_requests`.
2. Mallory (a single attested participant) reads the pending request from chain state.
3. Mallory calls `respond_ckd(alice_request, CKDResponse { big_y: [0u8;48], big_c: [0u8;48] })`.
4. The contract executes the `AppPublicKey(_) => {}` branch (no check), then calls `resolve_yields_for`, which resolves Alice's yield with the forged response.
5. Alice's NEAR transaction returns `CKDResponse { big_y: [0;48], big_c: [0;48] }` — attacker-controlled garbage — as her "confidential" derived key.
6. The pending request is permanently consumed; Alice cannot resubmit.
