Audit Report

## Title
24-Hour Hard Expiry in `blockUserTransfers` Creates Exploitable Gap Window via Block Stuffing — (`contracts/RSETH.sol`)

## Summary
`blockUserTransfers` sets a hard 24-hour expiry on transfer blocks with no on-chain renewal mechanism. An attacker (the blocked address or a colluding party) can use block stuffing to delay the manager's refresh transaction past `blockedUntil`, at which point `_enforceNotBlocked` auto-clears the block and silently returns, allowing the blocked address to freely transfer rsETH. `recoverFrozenFunds` also becomes unavailable during this gap.

## Finding Description
`blockUserTransfers` sets `transfersBlockedUntil[account] = block.timestamp + 1 days` ( [1](#0-0) ) with no on-chain mechanism to extend the block without a new manager transaction landing on-chain.

`_enforceNotBlocked` only reverts while `block.timestamp < blockedUntil`; once `block.timestamp >= blockedUntil` it auto-deletes the mapping entry and returns without reverting: [2](#0-1) 

`_transfer` calls `_enforceNotBlocked` for both `from` and `to`, so once the block expires, transfers proceed normally: [3](#0-2) 

`recoverFrozenFunds` checks `blockedUntil == 0 || block.timestamp >= blockedUntil` and reverts with `NoActiveTransferBlock`, making fund recovery also unavailable during the gap: [4](#0-3) 

**Exploit path:**
1. Manager calls `blockUserTransfers([target])`, setting `transfersBlockedUntil[target] = T + 1 days`.
2. Near expiry, the blocked address (or a colluding party) fills every block with maximum-gas transactions, excluding the manager's refresh call from being included.
3. Once `block.timestamp >= T + 1 days`, `_enforceNotBlocked` auto-clears the block and returns.
4. The blocked address calls `transfer`, which now succeeds — the entire balance can be moved in a single transaction.
5. The manager's refresh lands after the funds have already moved.

## Impact Explanation
The impact is **Low — Block stuffing**, which is an explicitly listed allowed impact in the program scope. The blocked address can drain its entire rsETH balance during the gap window. Additionally, `recoverFrozenFunds` is unavailable during the same window, removing the admin's fallback recovery path. The gap persists for as long as the attacker can sustain block stuffing, but even a single block (≈12 s on mainnet) is sufficient to execute a full balance transfer.

## Likelihood Explanation
The attack is economically rational only when the value of rsETH held by the blocked address exceeds the cost of stuffing blocks for the duration of the manager's monitoring and submission latency. On Ethereum mainnet, stuffing even a few minutes of blocks costs tens of ETH, making this viable only for large holders. However, the gap also opens for free if the manager's transaction is delayed by ordinary network congestion or operational error — no block stuffing required in that case. The attack is repeatable each time a block nears expiry.

## Recommendation
1. **Rolling window on attempted transfer**: In `_enforceNotBlocked`, when `block.timestamp < blockedUntil`, reset `transfersBlockedUntil[account] = block.timestamp + 1 days` so every attempted transfer by the blocked address automatically extends the block, removing the hard expiry that can be weaponised.
2. **Grace period for `recoverFrozenFunds`**: Allow `recoverFrozenFunds` to operate for a short window (e.g., 1 hour) after `blockedUntil` has passed, giving the admin a fallback even if the refresh is delayed.
3. **Early-warning event**: Emit an event well before expiry (e.g., at `T − 1 hour`) so off-chain monitoring can trigger a refresh with sufficient lead time, reducing the stuffing window the attacker must sustain.

## Proof of Concept
The following Foundry test reproduces the gap window without any block-stuffing infrastructure; `vm.warp` simulates the effect of the manager's refresh being excluded from all blocks until expiry:

```solidity
function testGapWindowExploitable() public {
    // 1. Manager blocks target
    vm.prank(manager);
    address[] memory accounts = new address[](1);
    accounts[0] = target;
    rseth.blockUserTransfers(accounts);

    uint256 blockedUntil = rseth.transfersBlockedUntil(target);

    // 2. Warp to exactly the expiry timestamp (simulates block stuffing)
    vm.warp(blockedUntil);

    // 3. Transfer succeeds — block expired, manager refresh not yet landed
    vm.prank(target);
    rseth.transfer(other, rseth.balanceOf(target));

    // 4. Manager refresh lands too late
    vm.warp(blockedUntil + 1);
    vm.prank(manager);
    rseth.blockUserTransfers(accounts);

    assertEq(rseth.balanceOf(target), 0, "target drained during gap");
    assertGt(rseth.balanceOf(other),  0, "other received funds");
}
```

At `vm.warp(blockedUntil)`, `_enforceNotBlocked` evaluates `block.timestamp < blockedUntil` as false, auto-clears the mapping, and returns — the `transfer` call succeeds. [5](#0-4)
