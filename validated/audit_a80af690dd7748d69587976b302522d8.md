Audit Report

## Title
First-Depositor rsETH Price Inflation via Unrestricted ETH Donation to LRTDepositPool - (File: contracts/LRTDepositPool.sol)

## Summary
An attacker who is the first depositor can donate ETH directly to `LRTDepositPool` via its unrestricted `receive()` function, which inflates `address(this).balance` and therefore the protocol's reported TVL without minting any rsETH. A subsequent call to the permissionless `updateRSETHPrice()` stores an extreme `rsETHPrice`. Any victim who then deposits ETH with `minRSETHAmountExpected = 0` receives zero rsETH while their ETH is permanently absorbed into the pool, recoverable only by the attacker who holds 100% of rsETH supply.

## Finding Description

**Root cause 1 — unrestricted ETH ingress counted as TVL:**
`LRTDepositPool` accepts arbitrary ETH via `receive() external payable {}` at line 58. [1](#0-0) 

`getETHDistributionData()` reads `address(this).balance` directly, so any donated ETH is immediately counted as protocol TVL: [2](#0-1) 

**Root cause 2 — price guard disabled when `pricePercentageLimit == 0`:**
`_updateRsETHPrice()` computes `isPriceIncreaseOffLimit` as `pricePercentageLimit > 0 && ...`. When `pricePercentageLimit` is at its default value of `0`, the entire guard short-circuits to `false`, allowing an unbounded price increase: [3](#0-2) 

`updateRSETHPrice()` is public and permissionless: [4](#0-3) 

**Root cause 3 — zero-rsETH mint silently accepted:**
`_beforeDeposit()` only reverts if `rsethAmountToMint < minRSETHAmountExpected`. When the victim passes `minRSETHAmountExpected = 0`, a computed mint of `0` satisfies `0 < 0 == false` and the deposit proceeds: [5](#0-4) 

`RSETH.mint(to, 0)` also succeeds: `checkDailyMintLimit(0)` evaluates `currentPeriodMintedAmount + 0 > maxMintAmountPerDay` which is false as long as the daily limit has not been reached, and OpenZeppelin's `_mint` does not revert on zero amount: [6](#0-5) 

**Full exploit path:**
1. Attacker calls `depositETH{value: 1 wei}(0, "")`. With `rsethSupply == 0`, `rsETHPrice` is initialized to `1e18` on the next `updateRSETHPrice()` call. Attacker receives 1 wei rsETH (100% of supply).
2. Attacker sends `D` ETH directly to `LRTDepositPool` via `receive()`. No rsETH is minted; `address(this).balance` increases by `D`.
3. Attacker calls `updateRSETHPrice()`. `totalETHInProtocol ≈ D + 1 wei`, `rsethSupply = 1 wei`, so `newRsETHPrice ≈ D * 1e18`. For `D = 1000 ETH`, `rsETHPrice ≈ 1e39`.
4. Victim calls `depositETH{value: V}(0, "")`. `getRsETHAmountToMint` computes `(V * 1e18) / 1e39 = 0`. The check `0 < 0` is false; `_mintRsETH(0)` is called. Victim's `V` ETH enters the pool; victim receives 0 rsETH.
5. Attacker redeems their 1 wei rsETH (100% of supply), recovering `D + 1 wei + V` ETH. Net profit: `V` ETH.

## Impact Explanation

**Critical — Direct theft of any user funds.**

The victim's deposited ETH (`V`) is permanently transferred into the pool. The victim receives zero rsETH and has no claim on the protocol. The attacker, holding 100% of rsETH supply, redeems all pool assets including the victim's deposit. This is a concrete, non-hypothetical loss of principal matching the "Direct theft of any user funds" critical impact class.

## Likelihood Explanation

**Low.** Four conditions must hold simultaneously:
1. `pricePercentageLimit == 0` — this is the default state and must be explicitly configured by an admin; most dangerous at protocol launch.
2. The attacker must be the first depositor (or act before `pricePercentageLimit` is set).
3. The victim must pass `minRSETHAmountExpected = 0` — valid input, common in direct contract calls and integrations that omit slippage protection.
4. The attacker must lock `D` ETH for the EigenLayer withdrawal delay period to recover the donation.

The economic cost of locking `D ≈ V` ETH limits opportunistic exploitation, but a targeted attack on a large deposit at launch is feasible.

## Recommendation

1. **Replace `address(this).balance` with internal accounting.** Maintain an `ethDeposited` storage variable incremented only in `depositETH` and the authorized `receiveFrom*` functions. Use this variable instead of `address(this).balance` in `getETHDistributionData()`.
2. **Revert on zero rsETH mint.** In `_beforeDeposit`, add `if (rsethAmountToMint == 0) revert ZeroRsETHMinted();` unconditionally, regardless of `minRSETHAmountExpected`.
3. **Enforce `pricePercentageLimit` before first deposit.** Add a deployment-time check or initializer step that sets a non-zero `pricePercentageLimit` before any deposit is accepted.
4. **Mint minimum liquidity to a dead address at initialization** (analogous to Uniswap V2's `MINIMUM_LIQUIDITY`) so `rsethSupply` is never 1 wei and the price-per-share manipulation is bounded.

## Proof of Concept

```solidity
// Preconditions:
//   pricePercentageLimit == 0 (default)
//   maxMintAmountPerDay > 0 (protocol operational)
//   depositLimitByAsset[ETH] > 1000 ether

// Step 1: updateRSETHPrice with zero supply → rsETHPrice = 1e18
lrtOracle.updateRSETHPrice();

// Step 2: Attacker becomes sole depositor (1 wei rsETH = 100% supply)
lrtDepositPool.depositETH{value: 1 wei}(0, "attacker");
// rsethSupply = 1, rsETHPrice = 1e18

// Step 3: Attacker donates 1000 ETH directly — no rsETH minted
(bool ok,) = address(lrtDepositPool).call{value: 1000 ether}("");
require(ok);

// Step 4: Inflate stored rsETHPrice
lrtOracle.updateRSETHPrice();
// totalETHInProtocol ≈ 1000e18 + 1
// newRsETHPrice = (1000e18 + 1) * 1e18 / 1 ≈ 1e39

// Step 5: Victim deposits 1 ETH with no slippage guard
// rsethAmountToMint = 1e18 * 1e18 / 1e39 = 0
// check: 0 < 0 == false → no revert → victim loses 1 ETH
lrtDepositPool.depositETH{value: 1 ether}(0, "victim");

// Step 6: Attacker redeems 1 wei rsETH (100% supply)
// Recovers: 1000 ETH (donation) + 1 wei (initial) + 1 ETH (victim) ≈ 1001 ETH
// Net profit: 1 ETH (victim's deposit)
```
