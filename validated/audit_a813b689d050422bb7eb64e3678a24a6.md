### Title
Unauthenticated Sync Peer Can Inject Arbitrary Transaction Hashes via P2P Sync, Corrupting RPC `starknet_getTransactionByHash` - (`crates/apollo_p2p_sync/src/client/transaction.rs`, `crates/apollo_protobuf/src/converters/transaction.rs`)

### Summary

The `TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` implementation blindly trusts the `transaction_hash` field from the protobuf message. The p2p sync client explicitly acknowledges (via a TODO comment) that it does not validate transaction hashes from untrusted peers before writing them to storage. This allows any unauthenticated sync peer to inject arbitrary transaction hashes, corrupting the `transaction_hash_to_idx` index and causing `starknet_getTransactionByHash` to return authoritative-looking wrong data.

### Finding Description

**Step 1 — No hash verification in protobuf conversion.**

`TryFrom<protobuf::TransactionInBlock> for (Transaction, TransactionHash)` at lines 136–183 extracts `transaction_hash` directly from the wire message and returns it as-is, with no recomputation or comparison against the transaction body: [1](#0-0) 

**Step 2 — P2P sync client explicitly skips hash validation.**

`parse_data_for_block` in `TransactionStreamFactory` pushes the peer-supplied hash directly into `block_body.transaction_hashes` with an explicit TODO acknowledging the missing validation: [2](#0-1) 

The only check performed is that the count of received transactions matches `n_transactions` from the stored block header (line 66). No verification against the block header's `transaction_commitment` (Patricia root) is performed.

**Step 3 — Fake hash is written to storage as the canonical index key.**

`write_transactions` in `append_body` inserts the peer-supplied hash directly into `transaction_hash_to_idx_table` and `TransactionMetadata`: [3](#0-2) 

**Step 4 — RPC returns the fake hash as authoritative.**

`get_transaction_by_hash` looks up by the queried hash, retrieves the body at that index, and returns `TransactionWithHash { transaction: body, transaction_hash: queried_hash }`. Since the fake hash is the stored key, a query for `fake_hash` returns the legitimate body paired with the fake hash: [4](#0-3) 

The real transaction hash is simultaneously unreachable — `get_transaction_idx_by_hash(real_hash)` returns `None` because only `fake_hash` was inserted into the index.

### Impact Explanation

- `starknet_getTransactionByHash(fake_hash)` returns a `TransactionWithHash` whose body does not hash to `fake_hash` — a direct violation of the RPC invariant.
- `starknet_getTransactionByHash(real_hash)` returns `TRANSACTION_HASH_NOT_FOUND` for a transaction that is actually included in the block.
- `starknet_getBlockWithTxs` returns transactions paired with injected hashes, not their true hashes.
- The `transaction_commitment` stored in the block header is never verified against the stored transaction hashes, so the corruption is never detected locally.

This falls squarely within: **High. RPC execution returns an authoritative-looking wrong value** and **High. Transaction conversion or signature/hash logic binds the wrong hash or executable payload.**

### Likelihood Explanation

The attacker only needs to be a responding sync peer — any node that can answer the victim's `Protocol::Transaction` SQMR query. The p2p network is unauthenticated at the transaction-data level. The attack requires sending a well-formed protobuf `TransactionsResponse` with valid transaction bodies but substituted `transaction_hash` fields. The TODO comment at line 88 confirms this is a known, unmitigated gap in the current implementation.

### Recommendation

In `parse_data_for_block`, after receiving each `FullTransaction`, recompute the transaction hash from the body and `chain_id` using `get_transaction_hash` (from `starknet_api::transaction_hash`) and reject the peer (raise `BadPeerError`) if it does not match the peer-supplied hash. Alternatively, after collecting all transactions for a block, verify the computed `transaction_commitment` against the value stored in the block header before calling `write_to_storage`.

```rust
// In crates/apollo_protobuf/src/converters/transaction_test.rs
#[test]
fn arbitrary_hash_accepted_without_verification() {
    use starknet_types_core::felt::Felt;
    use crate::protobuf::{self, transaction_in_block};

    let fake_hash = Felt::from(0xdeadbeef_u64);

    let proto = protobuf::TransactionInBlock {
        transaction_hash: Some(fake_hash.into()),
        txn: Some(transaction_in_block::Txn::InvokeV1(
            transaction_in_block::InvokeV1 {
                sender: Some(Felt::ONE.into()),
                nonce: Some(Felt::ZERO.into()),
                max_fee: Some(Felt::ZERO.into()),
                signature: Some(protobuf::AccountSignature { parts: vec![] }),
                calldata: vec![],
            },
        )),
    };

    let (_, returned_hash) =
        <(starknet_api::transaction::Transaction,
          starknet_api::transaction::TransactionHash)>::try_from(proto).unwrap();

    // The returned hash equals the injected fake value, not hash(body, chain_id)
    assert_eq!(returned_hash.0, fake_hash);
}
```

This test passes today, confirming the injected hash flows through unchanged into the `FullTransaction` that is subsequently stored and served by the RPC.
