Audit Report

## Title
Stale `rsETHPrice` Read in Deposit Path Allows New Depositors to Capture Unclaimed Yield — (File: `contracts/LRTDepositPool.sol`, `contracts/LRTOracle.sol`)

## Summary
`LRTOracle.rsETHPrice` is a stored state variable updated only by an off-chain keeper calling `updateRSETHPrice()`. Neither `depositETH` nor `depositAsset` trigger a price update before computing the rsETH mint amount via `getRsETHAmountToMint`. During the window between yield accrual and the next keeper update, the stored price is lower than the true price, causing new depositors to receive excess rsETH that dilutes the unclaimed yield belonging to existing holders.

## Finding Description
`rsETHPrice` is declared as a mutable storage variable in `LRTOracle` and written only inside `_updateRsETHPrice()`, which is invoked exclusively by the keeper-called `updateRSETHPrice()` or the manager-only `updateRSETHPriceAsManager()`.

The deposit path is:
- `depositETH` / `depositAsset` → `_beforeDeposit` → `getRsETHAmountToMint`
- `getRsETHAmountToMint` computes `(amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice()` using the stored, potentially stale value.

No call to `updateRSETHPrice()` precedes this read in either deposit function. When staking rewards or EigenLayer yield have accrued since the last keeper update, `rsETHPrice` is lower than the true current price. The formula `amount / staleLowerPrice` yields more rsETH than `amount / truePrice`, and the excess is minted to the depositor.

An additional complication: `_updateRsETHPrice()` reverts with `PriceAboveDailyThreshold` for non-manager callers when the price increase exceeds `pricePercentageLimit`. This means naively prepending `updateRSETHPrice()` to deposit functions would cause deposits to revert during large yield accrual windows, making the fix non-trivial.

The `initiateWithdrawal` path reads `rsETHPrice` in `getExpectedAssetAmount` (`amount * rsETHPrice / assetPrice`). With a stale lower price, withdrawers receive *less* asset than they are owed — this harms withdrawers rather than benefiting them, so the primary theft vector is the deposit path.

## Impact Explanation
**High — Theft of unclaimed yield.** Yield accrued between keeper updates is not yet reflected in `rsETHPrice`. Every deposit made during this window mints excess rsETH. When the keeper eventually calls `updateRSETHPrice()`, the new price is computed over a larger rsETH supply, permanently reducing the per-token ETH value for all pre-existing holders. The accrued yield that should have increased the price for existing holders is instead partially captured by the new depositor. This matches the allowed impact "High. Theft of unclaimed yield."

## Likelihood Explanation
Any unprivileged user can trigger this by calling `depositETH` or `depositAsset` at any point between two keeper updates. No front-running, special role, or adversarial setup is required — the mis-accounting occurs for every deposit made while the price is stale. Keeper update intervals on the order of hours to a day create a persistent, repeatable window. The larger the accrued yield and the longer the keeper interval, the greater the per-deposit excess.

## Recommendation
- **Short term:** Before reading `rsETHPrice` in `getRsETHAmountToMint`, attempt to call `updateRSETHPrice()` and handle the `PriceAboveDailyThreshold` revert gracefully (e.g., catch and proceed only if the caller is a manager, or skip the update and use the stored price with an explicit staleness bound). Alternatively, add a `lastPriceUpdateTimestamp` and revert deposits if the price is older than a configurable maximum staleness threshold.
- **Long term:** Replace the stored `rsETHPrice` with an on-the-fly `view` computation (`totalETHInProtocol / rsethSupply`) so no stale state can exist. This eliminates the keeper-dependency for pricing reads entirely.

## Proof of Concept
1. Deploy with 1000 ETH TVL, 1000 rsETH supply → `rsETHPrice = 1.0e18`. Keeper last ran at `T=0`.
2. At `T+12h`, staking rewards add 10 ETH to TVL (now 1010 ETH). True price = `1.01e18`. Keeper has not called `updateRSETHPrice()`; stored `rsETHPrice` remains `1.0e18`.
3. Attacker calls `depositETH{value: 100 ether}(0, "")`. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.0e18 = 100e18` rsETH minted.
4. At the true price `1.01e18`, the attacker should receive `100e18 * 1e18 / 1.01e18 ≈ 99.01e18` rsETH. Excess minted: `≈ 0.99e18` rsETH.
5. Keeper calls `updateRSETHPrice()`. New state: TVL = 1110 ETH, supply = 1100 rsETH → new price = `1110/1100 ≈ 1.009e18` instead of the correct `1.01e18`. Existing 1000 rsETH holders' share of TVL is permanently reduced by the dilution.
