Audit Report

## Title
Missing Caller Authorization in `MerkleDistributor.claim()` Allows Anyone to Force Claims and Extract Fees from Victims - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` accepts a caller-supplied `account` parameter but never validates that `msg.sender == account`. Any unprivileged caller can trigger a token claim on behalf of any eligible user, forcing them to pay the current `feeInBPS` (up to 10%) even if the user was deliberately deferring their claim to wait for a lower fee. The sibling contract `KernelMerkleDistributor` already fixes this exact pattern with an explicit `Unauthorized` revert.

## Finding Description
`MerkleDistributor.claim()` performs merkle proof verification against the caller-supplied `account` and transfers tokens to it, but contains no check that `msg.sender == account`:

```solidity
// contracts/utils/MerkleDistributor/MerkleDistributor.sol L97-106
function claim(
    uint256 index,
    address account,       // caller-supplied, never validated against msg.sender
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
``` [1](#0-0) 

The fee deduction at lines 138–144 routes `claimableAmount * feeInBPS / 10_000` to `protocolTreasury`, irreversibly consuming the victim's yield: [2](#0-1) 

The sibling contract `KernelMerkleDistributor._processClaim()` explicitly guards against this at lines 311–313:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
``` [3](#0-2) 

`MerkleDistributor` has no equivalent guard. Merkle proof inputs (`index`, `account`, `cumulativeAmount`, `merkleProof`) are derived from public off-chain distribution data, so any attacker can reconstruct valid call arguments for any eligible account.

## Impact Explanation
When `feeInBPS > 0` (up to `MAX_FEE_IN_BPS = 1000`, i.e. 10%), a forced claim permanently deducts `claimableAmount * feeInBPS / 10_000` from the victim's claimable balance and routes it to `protocolTreasury`. A user deliberately deferring their claim — waiting for the owner to reduce or zero the fee — is forced to pay the current higher fee. The difference between the fee paid under the forced claim and the fee the user would have paid is an irreversible loss of unclaimed yield. This matches the allowed impact: **High — Theft of unclaimed yield**. [4](#0-3) 

## Likelihood Explanation
Merkle proof inputs are public. Any attacker can enumerate all eligible accounts from off-chain distribution data and call `claim()` for each one. No special role, private key, or privileged access is required. The only cost is gas. The attack is repeatable across all eligible accounts and across every new merkle root epoch. Likelihood is **High**.

## Recommendation
Add a caller-identity check at the top of `MerkleDistributor.claim()`, identical to the one already present in `KernelMerkleDistributor._processClaim()`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Insert this check before any state changes or transfers, and add the `Unauthorized` error to the `IMerkleDistributor` interface.

## Proof of Concept
1. `feeInBPS = 500` (5%). Alice is eligible to claim 1000 tokens and is waiting for the owner to reduce the fee to 0.
2. Bob reads Alice's merkle proof (`index`, `alice`, `1000e18`, `proof`) from the public off-chain distribution data.
3. Bob calls `MerkleDistributor.claim(index, alice, 1000e18, proof)`.
4. The function passes all checks: valid merkle proof, not yet claimed, contract not paused.
5. `fee = 1000e18 * 500 / 10_000 = 50e18` tokens are sent to `protocolTreasury`.
6. `950e18` tokens are sent to Alice.
7. `userClaims[alice].lastClaimedIndex` is updated; Alice can never reclaim the 50 tokens lost to the fee.
8. Bob spent only gas. Alice lost 50 tokens she would have avoided losing had she been able to claim after the fee was reduced.
