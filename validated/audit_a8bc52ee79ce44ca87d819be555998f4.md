### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in Consensus L1 Handler Conversion Produces Wrong Receipt Fee and Bypasses Fee Adequacy Check — (File: crates/apollo_transaction_converter/src/transaction_converter.rs)

---

### Summary

When a validator node processes an `L1HandlerTransaction` received through the consensus protocol, `convert_consensus_l1_handler_to_internal_l1_handler` unconditionally substitutes the actual fee paid on L1 with the hardcoded sentinel `Fee(1)`. This is the direct Sequencer-native analog of the external "address aliasing" bug: a field that is supposed to carry the caller-supplied value is silently replaced with a fixed placeholder, causing the receipt/blob to record wrong economic data and allowing the fee-adequacy check to be bypassed for every L1 handler transaction that travels through the consensus validator path.

---

### Finding Description

The `ConsensusTransaction::L1Handler` wire format carries only the raw `transaction::L1HandlerTransaction` struct, which does not include `paid_fee_on_l1`. When the validator reconstructs the executable transaction it calls:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs  lines 473-483
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

`Fee(1)` is hardcoded regardless of what was actually paid on L1. This value then flows into two downstream consumers:

**1. The blockifier fee-adequacy check** (`execute_raw`):

```rust
// crates/blockifier/src/transaction/l1_handler_transaction.rs  lines 103-113
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(...));
}
``` [2](#0-1) 

Because `Fee(1) ≠ Fee(0)`, the check always passes. The validator can never reject an L1 handler transaction for insufficient fee, regardless of what was actually paid on L1.

**2. The centralized blob / receipt** (`CentralL1HandlerTransaction`):

```rust
// crates/apollo_consensus_orchestrator/src/cende/central_objects.rs  lines 383-393
impl From<L1HandlerTransaction> for CentralL1HandlerTransaction {
    fn from(tx: L1HandlerTransaction) -> CentralL1HandlerTransaction {
        CentralL1HandlerTransaction {
            ...
            paid_fee_on_l1: tx.paid_fee_on_l1,   // always Fee(1) on validator path
            ...
        }
    }
}
``` [3](#0-2) 

Every L1 handler transaction committed through the consensus validator path will have `paid_fee_on_l1 = Fee(1)` in the blob sent to the centralized pipeline, regardless of the actual fee (which could be `Fee(1_000_000)` or more).

The proposer side correctly drops `paid_fee_on_l1` when converting to the consensus wire format:

```rust
// crates/apollo_transaction_converter/src/transaction_converter.rs  lines 178-180
InternalConsensusTransaction::L1Handler(tx) => {
    Ok(ConsensusTransaction::L1Handler(tx.tx))   // paid_fee_on_l1 is dropped here
}
``` [4](#0-3) 

There is no mechanism for the validator to recover the real `paid_fee_on_l1` from the wire message; the field is structurally absent from `ConsensusTransaction::L1Handler`.

---

### Impact Explanation

| Corrupted value | Location | Effect |
|---|---|---|
| `paid_fee_on_l1 = Fee(1)` instead of real L1 fee | `executable_transaction::L1HandlerTransaction` on validator | Fee-adequacy check always passes; validator cannot enforce L1 fee requirement |
| `paid_fee_on_l1 = Fee(1)` in blob | `CentralL1HandlerTransaction` | Centralized pipeline records wrong economic data for every L1 handler transaction |

This matches the allowed impact: **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact"** and **"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."**

---

### Likelihood Explanation

This is not an edge case. It is the normal code path for every `L1HandlerTransaction` that a validator node processes through consensus. Any block containing at least one L1 handler transaction will produce a blob with wrong `paid_fee_on_l1` values. No special attacker capability is required; the bug is triggered by ordinary L1→L2 messaging.

---

### Recommendation

1. **Include `paid_fee_on_l1` in the consensus wire format.** Add the field to `ConsensusTransaction::L1Handler` (and the corresponding protobuf message) so the proposer can transmit the real value and the validator can use it.
2. **Until the wire format is extended**, document that the validator path cannot enforce fee adequacy for L1 handler transactions and that blob `paid_fee_on_l1` values are unreliable.
3. **Remove the `TODO` placeholder** once the wire format carries the real value, and add a test that asserts the validator path preserves `paid_fee_on_l1` end-to-end.

---

### Proof of Concept

1. An L1 user calls `sendMessageToL2` on the Starknet core contract, paying `paid_fee_on_l1 = 1_000_000` wei.
2. The proposer reads the resulting `L1HandlerTransaction` from the base layer with `paid_fee_on_l1 = Fee(1_000_000)`.
3. The proposer converts it to `ConsensusTransaction::L1Handler(tx.tx)` — `paid_fee_on_l1` is dropped.
4. The validator receives the wire message and calls `convert_consensus_l1_handler_to_internal_l1_handler`, which hardcodes `Fee(1)`.
5. `execute_raw` checks `paid_fee == Fee(0)` → `Fee(1) ≠ Fee(0)` → passes unconditionally.
6. `CentralL1HandlerTransaction` is serialized with `paid_fee_on_l1 = Fee(1)` and written to the blob.
7. The centralized pipeline records `paid_fee_on_l1 = 1` instead of `1_000_000` for this transaction.

The structural parallel to the external bug is exact: just as the Mailbox contract silently aliased the refund-recipient address to a value the caller did not intend, `convert_consensus_l1_handler_to_internal_l1_handler` silently replaces the caller-supplied `paid_fee_on_l1` with a fixed sentinel, producing a wrong authoritative value in the receipt/blob and disabling the fee guard on the validator path.
