### Title
Token-2022 MetadataPointer Mint-Binding Missing — Attacker Can Register Arbitrary Mint with Stolen Token Identity on NEAR - (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`parse_metadata_account` verifies that the supplied metadata account's address matches the mint's `MetadataPointer` extension value and that the account is owned by the Metaplex program, but it never checks that the Metaplex metadata account's `mint` field equals `self.mint.key()`. Because the `MetadataPointer` address is freely chosen by the Token-2022 mint creator, an unprivileged attacker can create a fresh mint whose pointer targets an existing, legitimate token's Metaplex metadata account, causing `log_metadata` to emit a `LogMetadataPayload` that carries the attacker's mint pubkey paired with a high-value token's name and symbol.

---

### Finding Description

**Vulnerable path — Token-2022 "third-party metadata" branch:**

In `process()`, when the mint uses Token-2022 and its `MetadataPointer` extension points to an address that is neither the mint itself nor `Pubkey::default()`, the code delegates to `parse_metadata_account` with the pointer's raw address: [1](#0-0) 

Inside `parse_metadata_account`, only two checks are performed:

1. The passed-in account key matches the pointer address: [2](#0-1) 

2. The account is owned by the Metaplex program: [3](#0-2) 

Neither check binds the Metaplex metadata account to `self.mint`. The deserialized `MplMetadata` struct contains a `mint` field that identifies which mint the metadata belongs to, but it is never compared against `self.mint.key()`.

**Contrast with the safe paths:**

- *Classic SPL tokens* (lines 117–127): the metadata address is derived via `find_program_address` seeded with `self.mint.key()`, so it is cryptographically bound to the mint. [4](#0-3) 

- *Embedded Token-2022 metadata* (lines 99–103): the metadata is read directly from the mint's own account data, so it is also self-bound. [5](#0-4) 

Only the third-party pointer branch is unbound.

**Resulting payload:**

The `LogMetadataPayload` is constructed with `token: self.mint.key()` (the attacker's mint) but `name`/`symbol` sourced from the victim token's Metaplex account: [6](#0-5) 

This payload is serialized and posted via Wormhole: [7](#0-6) 

---

### Impact Explanation

The NEAR side receives a `LogMetadataMessage` containing the attacker's mint address paired with a legitimate token's `name` and `symbol`: [8](#0-7) 

NEAR then MPC-signs a `MetadataPayload` over this data and emits it for downstream `deploy_token` calls on EVM/Starknet/Aptos. The attacker's mint is permanently registered across the bridge ecosystem under a stolen identity. This breaks the invariant that `name`/`symbol` in a `LogMetadataPayload` must belong to the mint being registered, constituting **asset-identity divergence** — the exact impact class listed as High in the scope.

---

### Likelihood Explanation

- No privileged access required. `log_metadata` is a fully permissionless instruction.
- Creating a Token-2022 mint with an arbitrary `MetadataPointer` is a standard, documented operation.
- All target Metaplex metadata account addresses for mainnet tokens (USDC, WETH, etc.) are public and deterministic.
- The attacker needs only: one Token-2022 mint creation transaction + one `log_metadata` call.

---

### Recommendation

After deserializing the Metaplex metadata account, add a binding check:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
    // MISSING CHECK — add this:
    require_keys_eq!(
        mpl.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress,
    );
    Ok((mpl.name.clone(), mpl.symbol.clone()))
}
```

This mirrors the implicit binding already present in the classic SPL path (PDA seeded from `self.mint.key()`) and the embedded-metadata path (data read from the mint account itself).

### Proof of Concept

```rust

let (usdc_metadata_pda, _) = Pubkey::find_program_address(
    &[b"metadata", MetaplexID.as_ref(), usdc_mint.as_ref()],
    &MetaplexID,
);

// 2. Attacker creates a Token-2022 mint M with MetadataPointer → usdc_metadata_pda
//    (standard spl-token-2022 mint init with MetadataPointerExtension)
let attacker_mint = create_token2022_mint_with_metadata_pointer(usdc_metadata_pda);

// 3. Attacker calls log_metadata(mint=attacker_mint, metadata=usdc_metadata_pda)
//    parse_metadata_account checks:
//      metadata.key() == usdc_metadata_pda  ✓ (attacker passed it)
//      metadata.owner == MetaplexID          ✓ (USDC's real metadata)
//      metadata.mint  == attacker_mint       ✗ (NOT CHECKED — is usdc_mint)
//    Returns ("USD Coin", "USDC")

// 4. LogMetadataPayload emitted:
//    { token: attacker_mint, name: "USD Coin", symbol: "USDC", decimals: attacker_decimals }

// 5. NEAR registers attacker_mint as "USD Coin / USDC" across the bridge.
```
