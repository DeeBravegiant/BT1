Audit Report

## Title
Token Decimal Mismatch in `viewSwapRsETHAmountAndFee` Causes Incorrect wrsETH Minting for Non-18-Decimal Collateral Tokens - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The `viewSwapRsETHAmountAndFee` function in all L2 pool variants computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalising `amountAfterFee` to 18 decimals. Because both rate values are 1e18-scaled but `amountAfterFee` retains the token's native decimal scale, any token with fewer than 18 decimals (e.g., USDC at 6) produces a result that is `10^(18 - tokenDecimals)` times too small. A depositor of 1 000 USDC receives effectively zero wrsETH while their tokens are locked in the pool.

## Finding Description
The ETH deposit path correctly multiplies by `1e18` to normalise:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
``` [1](#0-0) 

The token deposit path omits this normalisation:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

`ChainlinkOracleForRSETHPoolCollateral.getRate()` always returns a 1e18-scaled price regardless of the collateral token's own decimals:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [3](#0-2) 

So `tokenToETHRate / rsETHToETHrate` is dimensionless (≈1 for a 1:1 ETH-priced token), and the result inherits the raw decimal scale of `amountAfterFee`. For a 6-decimal token the output is `1e12` times smaller than the economically correct value.

`addSupportedToken` enforces no decimal constraint — only that the oracle returns a non-zero rate: [4](#0-3) 

The identical flaw is present in:
- `RSETHPoolNoWrapper.sol` line 311 [5](#0-4) 
- `RSETHPoolV3ExternalBridge.sol` line 452 [6](#0-5) 
- `RSETHPoolV3WithNativeChainBridge.sol` line 370 [7](#0-6) 

The reverse-swap path in `viewSwapAssetToPremintedRsETH` has the symmetric error (`tokenAmount = rsETHAmount * rsETHToETHrate / tokenToETHRate`), returning `1e12` times more tokens than expected for a 6-decimal token, but that function is consumed by an operator-only swap path. [8](#0-7) 

## Impact Explanation
**Critical — Direct theft of user funds / Permanent freezing of funds.**
A user depositing 1 000 USDC (`1_000e6`) receives `≈285_714` wei of wrsETH (≈`2.86e-13` wrsETH) instead of `≈2.86e17` wei (≈0.286 wrsETH). The 1 000 USDC is transferred into the pool via `safeTransferFrom` before `mint` is called, so the tokens are locked with no recourse. This is direct, permanent loss of deposited principal — a Critical impact under the allowed scope. [9](#0-8) 

## Likelihood Explanation
The precondition is that a `TIMELOCK_ROLE` holder adds a non-18-decimal token. This is a legitimate, expected administrative action — the code provides no documentation or on-chain guard indicating only 18-decimal tokens are supported, and `ChainlinkOracleForRSETHPoolCollateral` is already designed to handle arbitrary Chainlink feeds. USDC (6 decimals) and USDT (6 decimals) are the dominant liquid assets on Arbitrum, Base, and Unichain — the exact chains these pool variants target. Once such a token is added, every depositor is affected without any further privileged action. Likelihood is **Medium**.

## Recommendation
Normalise `amountAfterFee` to 18 decimals before applying the rate formula in `viewSwapRsETHAmountAndFee`:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 amountNormalized = amountAfterFee * 1e18 / (10 ** tokenDecimals);
rsETHAmount = amountNormalized * tokenToETHRate / rsETHToETHrate;
```

Apply the symmetric inverse in `viewSwapAssetToPremintedRsETH`:

```solidity
uint256 tokenAmountNormalized = rsETHAmount * rsETHToETHrate / tokenToETHRate;
tokenAmount = tokenAmountNormalized * (10 ** tokenDecimals) / 1e18;
```

Apply the same fix to all four affected pool variants. Optionally, enforce `IERC20Metadata(token).decimals() == 18` in `addSupportedToken` as a defence-in-depth guard.

## Proof of Concept
1. Admin calls `RSETHPoolV3.addSupportedToken(USDC_ADDRESS, chainlinkUSDCOracle)` on an L2 where USDC has 6 decimals.
2. User approves 1 000 USDC and calls `deposit(USDC_ADDRESS, 1_000e6, "ref")`.
3. `viewSwapRsETHAmountAndFee(1_000e6, USDC_ADDRESS)` executes:
   - `amountAfterFee ≈ 1_000e6` (6-decimal scale)
   - `tokenToETHRate = 3e14` (Chainlink USDC/ETH normalised to 1e18 by `ChainlinkOracleForRSETHPoolCollateral`)
   - `rsETHToETHrate ≈ 1.05e18`
   - `rsETHAmount = 1_000e6 * 3e14 / 1.05e18 ≈ 285_714`
4. `wrsETH.mint(msg.sender, 285_714)` — user receives `285_714` wei of wrsETH (`≈2.86e-13` wrsETH).
5. Expected correct output: `1_000e18 * 3e14 / 1.05e18 ≈ 2.86e17` wei (`≈0.286` wrsETH).
6. User's 1 000 USDC is permanently locked in the pool.

**Foundry fork test plan:** Fork an L2 (e.g., Arbitrum), deploy `ChainlinkOracleForRSETHPoolCollateral` with the live Chainlink USDC/ETH feed, call `addSupportedToken(USDC, oracle)`, then call `deposit(USDC, 1_000e6, "")` from a funded address and assert `wrsETH.balanceOf(user) < 1e6` (effectively zero) while `USDC.balanceOf(pool)` increased by `1_000e6`.
