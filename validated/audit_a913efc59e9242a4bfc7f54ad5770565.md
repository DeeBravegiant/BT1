Audit Report

## Title
Unvalidated Chainlink `int256` Price Cast to `uint256` Enables Temporary Fund Freeze - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` retrieves `int256 price` from Chainlink's `latestRoundData()` and casts it directly to `uint256` without a `require(price > 0)` guard. A zero price causes division-by-zero reverts in `LRTWithdrawalManager.getExpectedAssetAmount()` and an artificially deflated TVL in `LRTOracle._getTotalEthInProtocol()`, which can trigger the automatic protocol-wide pause via the publicly callable `updateRSETHPrice()`. A negative price wraps to near `type(uint256).max` and causes overflow reverts under Solidity 0.8.x checked arithmetic across all oracle-dependent paths.

## Finding Description
In `ChainlinkPriceOracle.getAssetPrice()` (lines 52â€“54), the raw `int256` return value of `latestRoundData()` is cast directly to `uint256` with no positivity check:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

**Case 1 â€” `price == 0`:**
- `LRTWithdrawalManager.getExpectedAssetAmount()` (line 593) computes `amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset)`. With `getAssetPrice` returning 0, this is a division-by-zero revert. Every call to `initiateWithdrawal()` and `instantWithdrawal()` for the affected asset reverts, freezing all withdrawal initiations.
- `LRTOracle._getTotalEthInProtocol()` (lines 339â€“343) accumulates `totalAssetAmt.mulWad(assetER)` per asset. With `assetER = 0`, the affected asset's entire TVL is excluded, causing `newRsETHPrice` to drop sharply. If `pricePercentageLimit > 0` and the drop exceeds the threshold, `_updateRsETHPrice()` (lines 277â€“281) calls `lrtDepositPool.pause()`, `withdrawalManager.pause()`, and `_pause()`, freezing the entire protocol. `updateRSETHPrice()` is publicly callable with no access restriction.
- `LRTDepositPool.getRsETHAmountToMint()` (line 520) computes `(amount * 0) / rsETHPrice = 0`. A depositor who passes `minRSETHAmountExpected = 0` receives zero rsETH for their deposited LST.

**Case 2 â€” `price < 0`:**
Casting a negative `int256` to `uint256` in Solidity is a bitwise reinterpretation (no revert), yielding a value near `type(uint256).max`. The subsequent multiplication `uint256(price) * 1e18` overflows under Solidity 0.8.x checked arithmetic, causing a revert in `getAssetPrice` itself. This propagates to every caller: `initiateWithdrawal`, `instantWithdrawal`, `updateRSETHPrice`, `getRsETHAmountToMint`, `unlockQueue`, and `getSwapETHToAssetReturnAmount`.

**Existing guards are insufficient:**
- `updatePriceOracleForValidated()` (lines 102â€“108) validates the price is between `1e16` and `1e19` at oracle setup time, but this is a one-time check that does not prevent the feed from later returning 0 or negative during normal operation.
- `updatePriceOracleFor()` (line 113) can be called directly by admin without any price validation at all.
- There is no runtime check on the returned price value anywhere in the call chain.

## Impact Explanation
**Medium â€” Temporary freezing of funds.**

A zero price from any supported asset's Chainlink feed causes withdrawal initiations for that asset to revert (division-by-zero in `getExpectedAssetAmount`), and enables any unprivileged caller to trigger the automatic protocol-wide pause via `updateRSETHPrice()`, locking both deposits and withdrawals until an admin manually unpauses. A negative price causes overflow reverts across all oracle-dependent paths, achieving the same freeze. The freeze is temporary (admin can unpause), matching the "Medium. Temporary freezing of funds." impact class.

## Likelihood Explanation
Chainlink price feeds for major LSTs (stETH/ETH, ETHx/ETH) rarely return zero or negative values on mainnet under normal conditions. However, the condition is reachable in documented edge cases: a deprecated aggregator round, a Chainlink circuit-breaker activation, or a misconfigured feed set via `updatePriceOracleFor` (which has no price sanity check). No attacker action is required; the impact materializes passively once the feed returns an invalid value, and any unprivileged user can accelerate the protocol pause by calling the public `updateRSETHPrice()`. Likelihood is Low-to-Medium.

## Recommendation
Add a positivity check in `ChainlinkPriceOracle.getAssetPrice()` immediately after reading from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
require(price > 0, "ChainlinkPriceOracle: invalid price");
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

Additionally, consider also validating `updatedAt` for staleness and checking `answeredInRound >= roundId` to fully harden the Chainlink integration.

## Proof of Concept

1. A supported LST's Chainlink feed (e.g., stETH/ETH) returns `price = 0` from `latestRoundData()` (circuit-breaker or deprecated round).
2. `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `0` â€” no revert, no check.
3. Any user calls `LRTWithdrawalManager.initiateWithdrawal(stETH, amount, ...)`:
   - `getExpectedAssetAmount(stETH, amount)` â†’ `amount * rsETHPrice / 0` â†’ **revert** (division-by-zero). All stETH withdrawal initiations are frozen.
