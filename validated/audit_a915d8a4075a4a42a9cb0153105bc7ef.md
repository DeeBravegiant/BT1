### Title
Consensus Vote Signature Never Verified — Any Network Peer Can Forge Validator Votes to Manufacture a Quorum - (File: `crates/apollo_consensus/src/single_height_consensus.rs`)

---

### Summary

`handle_vote` in `SingleHeightConsensus` accepts incoming prevotes and precommits from the P2P network without verifying the cryptographic signature on the `Vote`. The only identity check is that `vote.voter` (a `ContractAddress`) appears in the committee member list. Because committee member addresses are public on-chain, any network peer can craft a `Vote` with `voter = <any_validator_address>` and `signature = RawSignature::default()` and have it accepted as a legitimate vote. This allows an unprivileged peer to manufacture a 2/3+ quorum for any block commitment that a target node has already validated, forcing a premature or incorrect consensus decision and breaking the Byzantine fault-tolerance safety guarantee.

### Impact Explanation

An unprivileged network peer who can send P2P broadcast messages can:

1. Read the current committee member addresses from the public staking contract.
2. Observe the current `height`, `round`, and any `proposal_commitment` that has been streamed over the network (proposals are broadcast in plaintext).
3. Craft `Vote` messages with `voter = <any_committee_member_address>`, `signature = RawSignature::default()`, and the observed `proposal_commitment`.
4. Broadcast enough such forged votes to exceed the 2/3+ quorum threshold.
5. The receiving node accepts every forged vote (address is in committee, height matches, no duplicate), accumulates the forged weight, and fires `DecisionReached`.

The `decision_reached` path then calls `batcher.decision_reached()`, which commits the block's state diff to storage, updates nonces in the mempool, and notifies state sync — producing a committed block that was decided without the real validators' cryptographic consent.

**Safety violation:** In a multi-node network, the attacker can selectively send forged votes to a subset of nodes, causing those nodes to decide on a commitment while others do not, breaking the one-decision-per-height safety invariant of Tendermint.

**Liveness manipulation:** The attacker can force any node to decide on a commitment the moment that node has validated a proposal, bypassing the requirement for honest 2/3+ agreement.

This maps to: **Critical — Wrong state committed through forged consensus quorum**, and **High — Consensus admission accepts unauthorized votes**.

---

### Likelihood Explanation

- **Trigger is unprivileged:** any peer that can connect to the P2P gossip network can send `Vote` messages.
- **No secret knowledge required:** committee addresses are public; the `proposal_commitment` is broadcast in the `ProposalFin` message.
- **No existing guard:** the TODO comment at line 242 of `single_height_consensus.rs` explicitly confirms the check is absent. The `manager.rs` `handle_vote` path also performs no signature check before forwarding to `shc.handle_vote`. [8](#0-7) 

---

### Recommendation

1. **Verify the signature in `handle_vote`** before accepting any vote into the state machine. Look up the validator's public key from the committee (the `Staker` struct should carry or be mappable to a public key), then call `verify_precommit_vote_signature` (for precommits) and the equivalent for prevotes.
2. **Sign outgoing votes** in `make_self_vote` using the node's `SignatureManager` before broadcasting, so that honest nodes produce verifiable votes.
3. **Map `ContractAddress` → `PublicKey`** in the committee data structure so that `handle_vote` can retrieve the expected public key for `vote.voter` without an additional async lookup.
4. **Reject votes with empty/default signatures** as a cheap pre-filter before the more expensive ECDSA check.
