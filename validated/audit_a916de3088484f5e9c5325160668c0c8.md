Audit Report

## Title
Signature Replay in `allowPushers` Lets Creator Nullify Pusher's Revocation — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

## Summary
`CompressedOracleV1.allowPushers` verifies a pusher's EIP-191 consent signature but never marks it as consumed. After a pusher calls `revokePusher()`, the creator can replay the original signature within the same deadline window to silently re-establish delegation, making the pusher's revocation permanently ineffective until the deadline expires. The contract's own NatSpec explicitly acknowledges this risk but provides no mitigation.

## Finding Description
`allowPushers` (L192–212) computes a hash over `(chainid, address(this), deadline, pusher, msg.sender)` and verifies the pusher's signature, then writes `namespaceRemapping[pusher] = msg.sender` with no used-signature guard:

```solidity
// L204-209
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
```

`revokePusher` (L238–243) zeroes the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
```

Because no `_usedDelegations` mapping or nonce exists, the creator retains the original `(deadline, pusher, creator)` tuple and can call `allowPushers` again with the identical signature. The hash recomputes identically, `ECDSA.recover` succeeds, and `namespaceRemapping[pusher]` is overwritten back to the creator — undoing the revocation without any new consent from the pusher. The NatSpec at L186–191 explicitly acknowledges the deadline is the only replay guard and warns that "an undated signature could re-establish a delegation AFTER the pusher revoked it," yet no code-level fix is present.

## Impact Explanation
After the replay, the pusher's subsequent `fallback()` calls read `namespaceRemapping[msg.sender]` (L315–316), which is now non-zero again, routing the pusher's data into the creator's namespace. This corrupts the oracle slots consumed by `IPriceProvider.getBidAndAskPrice` during pool swaps. Any pool anchored to a feed in that namespace will execute swaps against prices the pusher did not intend to publish for the creator — satisfying the bad-price execution impact gate. This constitutes a direct integrity failure of the oracle feed backing live pool swaps.

## Likelihood Explanation
- The creator already holds the signature from the initial `allowPushers` call — no additional off-chain action is required.
- Long-lived deadlines (days/weeks) are normal operational practice, leaving a wide replay window.
- The replay requires only a single on-chain transaction (`allowPushers` with the original arguments).
- The pusher has no on-chain mechanism to detect or prevent the replay before the deadline expires.

## Recommendation
Track consumed signatures with a `mapping(bytes32 => bool) private _usedDelegations` and revert if the hash has already been seen:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(!_usedDelegations[hash], "signature already used");
require(pusher == ECDSA.recover(hash, signatures[i]));
_usedDelegations[hash] = true;
namespaceRemapping[pusher] = msg.sender;
```

Alternatively, include a per-pusher nonce in the signed payload so each consent is cryptographically unique and cannot be replayed even within the same deadline window.

## Proof of Concept
1. Pusher signs: `keccak256(abi.encode(chainid, oracle, deadline=T+7days, pusher, creator))`.
2. Creator calls `allowPushers(T+7days, [pusher], [sig])` → `namespaceRemapping[pusher] = creator`.
3. Pusher calls `revokePusher()` → `namespaceRemapping[pusher] = address(0)`.
4. Creator calls `allowPushers(T+7days, [pusher], [sig])` again with the **same** `sig`:
   - `_ensureDeadline(T+7days)` passes (still in the future).
   - Hash recomputes identically; `ECDSA.recover` returns `pusher`.
   - `namespaceRemapping[pusher] = creator` is written again.
5. Pusher's revocation is silently undone. Any subsequent `fallback()` push by the pusher lands in the creator's namespace, corrupting the oracle feed consumed by any pool using that feed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
```
