Audit Report

## Title
`instantWithdrawal` Ignores `assetsCommitted`, Allowing Vault Drain That Freezes Queued Withdrawals — (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`instantWithdrawal` checks vault availability via `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal`, which subtracts only the static, manually-set `queuedWithdrawalsBuffer` from the vault balance. It never consults `assetsCommitted[asset]`, the live accounting variable that tracks assets already reserved for pending queued withdrawal requests. Because `queuedWithdrawalsBuffer` defaults to `0` and is never automatically synchronized with `assetsCommitted`, an unprivileged caller can drain the entire unstaking vault even when its full balance is already committed to queued withdrawals, leaving those users' rsETH locked in `LRTWithdrawalManager` with no self-service exit.

## Finding Description

**Root cause — two independent availability checks over the same pool of assets.**

`LRTDepositPool.getTotalAssetDeposits` includes `assetLyingUnstakingVault` (the raw vault balance) in the protocol-wide total:

```solidity
// LRTDepositPool.sol:385-396
return (assetLyingInDepositPool + assetLyingInNDCs + effectiveAssetWithEigenLayer
        + assetLyingInConverter + assetLyingUnstakingVault);
```

`getAvailableAssetAmount` (used by `initiateWithdrawal`) correctly reserves against this total:

```solidity
// LRTWithdrawalManager.sol:599-603
availableAssetAmount = totalAssets > assetsCommitted[asset]
    ? totalAssets - assetsCommitted[asset] : 0;
```

When a user calls `initiateWithdrawal`, `assetsCommitted[asset]` grows and their rsETH is transferred into `LRTWithdrawalManager`. The vault balance is the intended source of funds for fulfilling those requests.

`instantWithdrawal`, however, uses a completely separate check:

```solidity
// LRTWithdrawalManager.sol:231-233
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
```

`getAssetsAvailableForInstantWithdrawal` only subtracts `queuedWithdrawalsBuffer[asset]`, which defaults to `0` and is never automatically updated:

```solidity
// LRTUnstakingVault.sol:235-237
uint256 vaultBalance = balanceOf(asset);
uint256 reservedBuffer = queuedWithdrawalsBuffer[asset];
availableAmount = reservedBuffer >= vaultBalance ? 0 : vaultBalance - reservedBuffer;
```

`instantWithdrawal` also does **not** decrement `assetsCommitted[asset]`, so after draining the vault, `assetsCommitted` still reflects the full committed amount while the vault is empty.

**Exploit path:**

1. Operator calls `setInstantWithdrawalEnabled(ETH, true)`. `queuedWithdrawalsBuffer[ETH]` remains `0`.
2. Unstaking vault holds 100 ETH. `getTotalAssetDeposits(ETH)` = 100 ETH.
3. User A calls `initiateWithdrawal(ETH, ...)` → `assetsCommitted[ETH] = 80 ETH`. User A's rsETH is locked in `LRTWithdrawalManager`.
4. Attacker calls `instantWithdrawal(ETH, ...)` for 100 ETH worth of rsETH.
   - Check: `getAssetsAvailableForInstantWithdrawal(ETH)` = `100 - 0 = 100 ETH` → **passes**.
   - Vault drained to 0 ETH. Attacker receives 100 ETH (minus fee).
5. Operator calls `unlockQueue(ETH, ...)`:
   - `totalAvailableAssets = unstakingVault.balanceOf(ETH) = 0`.
   - Line 297: `if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero()` → **reverts**.
   - User A's queued withdrawal cannot be unlocked. Their rsETH remains locked indefinitely.

## Impact Explanation
Queued withdrawal users have already transferred their rsETH into `LRTWithdrawalManager` at `initiateWithdrawal` time. Once the vault is drained, `unlockQueue` reverts unconditionally (line 297), and `completeWithdrawal` also fails because the request is never unlocked. The users' rsETH is frozen inside the contract until the operator manually replenishes the unstaking vault by unstaking from EigenLayer — a process that takes days. This is **temporary freezing of funds** (Medium).

## Likelihood Explanation
- Enabling instant withdrawal is a normal, expected operational step; the protocol explicitly provides `setInstantWithdrawalEnabled`.
- `queuedWithdrawalsBuffer` defaults to `0`. The protocol provides no enforcement, event, or reminder to set the buffer when enabling instant withdrawal. An operator enabling the feature without setting the buffer is a realistic omission.
- Once both conditions hold, any unprivileged rsETH holder can trigger the drain in a single transaction with no capital at risk (they receive fair-value assets in return for their rsETH).
- The attack is repeatable: every time the vault is replenished and instant withdrawal remains enabled without a buffer, the drain can be repeated.

## Recommendation
In `instantWithdrawal`, subtract `assetsCommitted[asset]` from the vault balance before comparing:

```solidity
uint256 vaultBalance = unstakingVault.balanceOf(asset);
uint256 committed    = assetsCommitted[asset];
uint256 trueAvailable = vaultBalance > committed ? vaultBalance - committed : 0;
if (assetAmountUnlocked > trueAvailable) revert CantInstantWithdrawMoreThanAvailable();
```

Alternatively, enforce that `setQueuedWithdrawalsBuffer` is called (and set to at least `assetsCommitted[asset]`) atomically when `setInstantWithdrawalEnabled` is called with `enabled = true`, and update the buffer automatically whenever `assetsCommitted` changes.

## Proof of Concept

Foundry fork/unit test outline:

```solidity
function test_instantWithdrawalDrainsCommittedAssets() public {
    // Setup: fund unstaking vault with 100 ETH

    withdrawalManager.setInstantWithdrawalEnabled(ETH, true);

    vm.prank(userA);

    assertGt(rsETH.balanceOf(address(withdrawalManager)), 0);
```
