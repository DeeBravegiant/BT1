### Title
Dual Foreign-Chain Support Registries With Divergent Read/Write Paths Permanently Block `verify_foreign_transaction` — (File: `crates/contract/src/lib.rs`)

---

### Summary

The MPC contract maintains **two independent, non-synchronized storage structures** for tracking which foreign chains each node supports. The gate check inside `verify_foreign_transaction` reads exclusively from the legacy `node_foreign_chain_support` map, while the current (non-deprecated) registration method `register_foreign_chains_config` writes exclusively to the new `foreign_chains` (`ForeignChainsMetadata`) structure. Any node that registers via the new path is invisible to the gate check, causing all `verify_foreign_transaction` calls for those chains to be permanently rejected.

---

### Finding Description

The contract stores foreign-chain support in two completely separate fields:

**Legacy structure** — `node_foreign_chain_support: SupportedForeignChainsByNode`, keyed by `AccountId`, written by `register_foreign_chain_support()` and the deprecated `register_foreign_chain_config()`. [1](#0-0) 

**New structure** — `foreign_chains: Lazy<ForeignChainsMetadata>`, keyed by TLS `Ed25519PublicKey`, written by `register_foreign_chains_config()`. [2](#0-1) 

The new registration method `register_foreign_chains_config` writes only to `foreign_chains`: [3](#0-2) 

The legacy method `register_foreign_chain_support` writes only to `node_foreign_chain_support`: [4](#0-3) 

The gate check inside `verify_foreign_transaction` calls `get_supported_foreign_chains()`, which reads **only** from `node_foreign_chain_support` using a strict all-participant intersection rule: [5](#0-4) [6](#0-5) 

The design document explicitly states that `verify_foreign_transaction` should be gated on `get_available_foreign_chains()` (the new threshold-based view derived from `foreign_chains`), and that `get_supported_foreign_chains()` is to be deprecated: [7](#0-6) 

The two structures are never synchronized. A node that calls `register_foreign_chains_config()` updates `foreign_chains.foreign_chains_configs` and triggers `recompute_available_foreign_chains()`, but leaves `node_foreign_chain_support` untouched. Because `get_supported_foreign_chains()` requires **all** active participants to appear in `node_foreign_chain_support`, any single node that uses only the new path causes the intersection to be empty, and every `verify_foreign_transaction` call is rejected.

---

### Impact Explanation

This is a **Medium** impact finding under the allowed scope: *"request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

When one or more nodes register via `register_foreign_chains_config()` without also calling `register_foreign_chain_support()`, `get_supported_foreign_chains()` returns an empty set (or a set missing those chains). Every subsequent `verify_foreign_transaction` call for those chains panics with `ForeignChainNotSupported`, permanently blocking the foreign-chain verification feature. Any bridge or omnibridge inbound flow that depends on `verify_foreign_transaction` is completely frozen — user funds locked in a bridge contract waiting for an MPC attestation will never receive one.

---

### Likelihood Explanation

The design documentation and the non-deprecated API surface actively direct nodes toward `register_foreign_chains_config()`. The deprecated `register_foreign_chain_config()` carries a `#[deprecated]` attribute and a note pointing to the new path. A node operator following current documentation will call the new method, silently breaking the gate check. The inconsistency requires no adversarial action — it is triggered by normal, correct node operation. [8](#0-7) 

---

### Recommendation

Replace the `get_supported_foreign_chains()` call inside `verify_foreign_transaction` with `get_available_foreign_chains()`, which reads from the new `foreign_chains` structure and applies the correct threshold-based rule. This is already the stated design intent. Additionally, either remove `node_foreign_chain_support` and its associated registration/view methods entirely, or add an explicit synchronization step so that `register_foreign_chains_config()` also updates `node_foreign_chain_support`, eliminating the split-state hazard.

---

### Proof of Concept

1. Deploy the contract with 3 participants (threshold = 2).
2. All 3 nodes call `register_foreign_chains_config({ Bitcoin })` — the current, non-deprecated method. This populates `foreign_chains.foreign_chains_configs` for all three nodes and sets `available_foreign_chains = { Bitcoin }`.
3. No node calls `register_foreign_chain_support` or `register_foreign_chain_config`. `node_foreign_chain_support` remains empty.
4. Any user calls `verify_foreign_transaction({ chain: Bitcoin, ... })` with a 1 yoctoNEAR deposit.
5. The contract executes `get_supported_foreign_chains()`, which iterates `node_foreign_chain_support` (empty) and returns `{}`.
6. The check `!supported_chains.contains(&Bitcoin)` is `true`; the contract panics with `ForeignChainNotSupported`.
7. Meanwhile, `get_available_foreign_chains()` correctly returns `{ Bitcoin }` — the network is fully capable of servicing the request, but the gate check permanently blocks it. [9](#0-8) [10](#0-9)

    pub fn register_foreign_chains_config(
        &mut self,
        foreign_chains_config: dtos::ForeignChainsConfig,
    ) -> Result<(), Error> {
        Self::assert_caller_is_signer();
        let signer_account_id = env::signer_account_id();
        let signer_account_pk = env::signer_account_pk();
        let signer_account_ed25519_pk = Ed25519PublicKey::try_from(&signer_account_pk)
            .unwrap_or_else(|_| env::panic_str("signer account key must be Ed25519"));
        let node_id = self
            .tee_state
            .lookup_node_id_by_signer_pk(&signer_account_ed25519_pk)
            .map_err(|_| InvalidState::NotParticipant {
                account_id: signer_account_id.clone(),
            })?;
        if node_id.account_id != signer_account_id {
            return Err(InvalidState::NotParticipant {
                account_id: signer_account_id,
            }
            .into());
        }
        let is_participant = self
            .protocol_state
            .is_existing_or_prospective_participant(&node_id.account_id)?;
        if !is_participant {
            return Err(InvalidState::NotParticipant {
                account_id: node_id.account_id.clone(),
            }
            .into());
        }
        let tls_key = node_id.tls_public_key.clone();

        self.foreign_chains
            .get_mut()
            .register(tls_key, foreign_chains_config);
        self.recompute_available_foreign_chains();

        Ok(())
    }

    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }

**File:** docs/design/calculating-supported-foreign-chains.md (L9-37)
```markdown
Today, `get_supported_foreign_chains()` returns the **strict intersection** of every
active participant's registered chains, and `verify_foreign_transaction` rejects any
request whose target chain is not in it. A single node that registers an empty list
(or hasn't registered yet) drops **every** chain — one operator can take the whole
feature down. That is what this proposal fixes.

It builds on the per-chain RPC whitelist (`ForeignChainRpcWhitelist`), which holds,
per chain, the network-trusted providers and the **RPC quorum** (`ChainEntry.quorum`
— how many of a node's providers must agree for it to accept a result).

## Proposal: two sets of chains

> **Terms** (whitelisted, available, RPC quorum, signing threshold, *covers*) are defined in
> [Foreign Chain Transaction Verification Design — Terminology](../foreign-chain-transactions.md#terminology).

The network distinguishes the **whitelisted** set (vote-driven policy, `get_whitelisted_foreign_chains()`)
from the **available** set (servable right now, `get_available_foreign_chains()`):

- **Whitelisted** is derived purely from the on-chain RPC whitelist — **no per-node input can add or
  remove a chain**, so no single operator can change it.
- **Available** is computed dynamically from the per-node config reports: `C` is available iff
  ≥ `signing_threshold` active participants cover `C`. `available ⊆ whitelisted` always.

`verify_foreign_transaction(C)` is **rejected unless `C` is available**: the contract fails fast
instead of accepting a request that can't reach the signing threshold and letting it time out. The
rejection is temporary — `C` becomes serviceable again as soon as enough nodes report coverage.

The legacy `get_supported_foreign_chains()` (the intersection rule) is **to be deprecated** in favour
of the two views above.
```
