Audit Report

## Title
Repeated `getQueuedWithdrawals` Calls Across All (Asset × NDC) Pairs in `updateRSETHPrice()` Can Exceed Block Gas Limit and Stale rsETH Price - (File: contracts/NodeDelegator.sol)

## Summary
`LRTOracle.updateRSETHPrice()` is a permissionless public function whose call chain invokes `NodeDelegator.getAssetUnstaking()` — and therefore EigenLayer's `getQueuedWithdrawals()` — once per supported-asset per NDC, not once per NDC. With 5 assets and 10 NDCs this produces 50 separate external calls each loading the full withdrawal queue. The protocol's own comment acknowledges a 120-withdrawal gas ceiling; operator-forced undelegations can push the actual EigenLayer queue above that ceiling without the protocol's internal counter reflecting it, causing `updateRSETHPrice()` to revert out-of-gas and permanently stale the rsETH exchange rate.

## Finding Description
**Call chain:**

```
updateRSETHPrice()                        // LRTOracle.sol:87 — public, whenNotPaused only
  └─ _updateRsETHPrice()
       └─ _getTotalEthInProtocol()         // LRTOracle.sol:331 — loops supportedAssets
            └─ getTotalAssetDeposits(asset) // LRTDepositPool.sol:385
                 └─ getAssetDistributionData(asset) // LRTDepositPool.sol:426
                      └─ [for each NDC] getAssetUnstaking(asset) // LRTDepositPool.sol:451
                           └─ getQueuedWithdrawals(address(this)) // NodeDelegator.sol:407 — external EigenLayer call
                                └─ for withdrawal in withdrawals:
                                     for strategy in withdrawal.strategies: // NodeDelegator.sol:409-426
```

`getAssetDistributionData` is called once per supported asset. For non-ETH assets it loops over all NDCs and calls `getAssetUnstaking(asset)` per NDC (`LRTDepositPool.sol:447-456`). For ETH it delegates to `getETHDistributionData()` which also calls `getAssetUnstaking(LRTConstants.ETH_TOKEN)` per NDC (`LRTDepositPool.sol:488-489`). Each `getAssetUnstaking` call unconditionally invokes `_getDelegationManager().getQueuedWithdrawals(address(this))` (`NodeDelegator.sol:406-407`), loading the entire withdrawal queue from EigenLayer storage regardless of which asset is being queried.

**Why existing guards are insufficient:**

`LRTUnstakingVault.setMaxUncompletedWithdrawalCount` enforces a hard cap of 80 (`LRTUnstakingVault.sol:153`). The protocol's comment acknowledges "120 is the max number of uncompleted withdrawals that allows us to still perform update rsETH price" and reserves a 15-unit buffer for forced undelegations (`LRTUnstakingVault.sol:151-152`). However, `uncompletedWithdrawalCount` is incremented only inside `initiateUnstaking` and `undelegate` (`NodeDelegator.sol:329`, `NodeDelegator.sol:279`). When an EigenLayer operator force-undelegates an NDC, EigenLayer creates new queued withdrawals directly in its `DelegationManager` storage without calling either of those functions, so the protocol's counter does not reflect the true queue length. `getQueuedWithdrawals` returns the actual EigenLayer-side queue, which can exceed 120 after forced undelegations, causing OOG on the next `updateRSETHPrice()` call.

Additionally, `updateRSETHPriceAsManager()` (`LRTOracle.sol:94-96`) calls the same `_updateRsETHPrice()` internal function and would also OOG, leaving no privileged escape hatch.

## Impact Explanation
If `updateRSETHPrice()` reverts OOG, `rsETHPrice` in `LRTOracle` becomes stale. This price is read by `LRTDepositPool.getRsETHAmountToMint()` (`LRTDepositPool.sol:520`) for every deposit and by `LRTWithdrawalManager.unlockQueue()` for every withdrawal payout. A stale price causes incorrect rsETH minting ratios and incorrect asset payouts, constituting a protocol-wide accounting failure. This maps to **Medium — Unbounded gas consumption** (and temporary freezing of the price-update mechanism until withdrawals are manually completed to reduce queue depth).

## Likelihood Explanation
The protocol explicitly acknowledges the gas sensitivity in its own source comments. Forced undelegations are a standard EigenLayer operator action, not an attack or external compromise — the protocol's own comment anticipates them. As the protocol scales toward its configured maximums (10 NDCs, 5+ assets, 80 protocol-initiated withdrawals), a single operator force-undelegation event across multiple NDCs can push the total EigenLayer queue above the 120-withdrawal threshold. This is a realistic operational scenario.

## Recommendation
1. **Cache `getQueuedWithdrawals` per NDC**: Fetch the withdrawal queue once per NDC and reuse it across all asset iterations, reducing external calls from `assets × NDCs` to `NDCs`.
2. **Incremental accounting**: Maintain a per-NDC, per-asset unstaking balance updated on `initiateUnstaking` / `completeUnstaking` / forced-undelegation sync, eliminating the need to query EigenLayer on every price update.
3. **Sync counter on forced undelegation**: Ensure `setUncompletedWithdrawalCount()` is called promptly after any operator force-undelegation and that the 80-cap enforcement accounts for the actual EigenLayer queue depth, not just the protocol-tracked counter.

## Proof of Concept
**Minimal call sequence demonstrating OOG path:**

1. Deploy with 5 supported assets, 10 NDCs, `maxUncompletedWithdrawalCount = 80`.
2. Fill each NDC to 8 protocol-initiated queued withdrawals (80 total, at the cap).
3. EigenLayer operator force-undelegates all 10 NDCs, adding ~10 additional withdrawals each (100 new entries), pushing the EigenLayer-side queue to ~18 per NDC (180 total).
4. Call `updateRSETHPrice()` from any EOA.
5. Observe: `getQueuedWithdrawals` is called 50 times (5 assets × 10 NDCs), each returning 18 withdrawals. Total inner iterations: 5 × 10 × 18 × 2 = 1,800 plus 50 cold external SLOAD-heavy calls. Transaction reverts OOG.
6. `rsETHPrice` is not updated; all subsequent deposits and withdrawal unlocks use the stale price.

    function getAssetUnstaking(address asset) external view returns (uint256 amount) {
        (IDelegationManager.Withdrawal[] memory queuedWithdrawals, uint256[][] memory withdrawalShares) =
            _getDelegationManager().getQueuedWithdrawals(address(this));

        for (uint256 withdrawalIndex = 0; withdrawalIndex < queuedWithdrawals.length; withdrawalIndex++) {
            IDelegationManager.Withdrawal memory withdrawal = queuedWithdrawals[withdrawalIndex];

            for (uint256 strategyIndex = 0; strategyIndex < withdrawal.strategies.length; strategyIndex++) {
                IStrategy strategy = withdrawal.strategies[strategyIndex];

                address strategyAsset = address(strategy) == address(lrtConfig.beaconChainETHStrategy())
                    ? LRTConstants.ETH_TOKEN
                    : address(strategy.underlyingToken());

                if (strategyAsset != asset) continue;

                uint256 sharesToUnstake = withdrawalShares[withdrawalIndex][strategyIndex];
                amount += strategyAsset == LRTConstants.ETH_TOKEN
                    ? sharesToUnstake
                    : strategy.sharesToUnderlyingView(sharesToUnstake);
            }
        }
    }
