Audit Report

## Title
Missing Sequencer Uptime Feed Check in `ProtectedPriceProviderL2` Allows Swaps at Stale Prices During L2 Sequencer Downtime — (`smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary

`ProtectedPriceProviderL2` relies solely on a `refTime`-based staleness check (`_isStale`) as its L2 safety gate. It contains no sequencer uptime feed query and no post-restart grace period. When the L2 sequencer goes down, the oracle's `refTime` freezes at the pre-downtime value; `_isStale` continues returning `false` for up to `MAX_TIME_DELTA` seconds (up to 7 days), allowing swaps to execute at the frozen stale price. This directly violates the protocol's stated invariant that "swaps revert on stale price... or (L2) sequencer down."

## Finding Description

`ProtectedPriceProviderL2._computeBidAsk` applies a single staleness gate before returning bid/ask prices to the pool:

```solidity
// ProtectedPriceProviderL2.sol lines 206-209
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
```

`_isStale` only checks whether the oracle's `refTime` is older than `MAX_TIME_DELTA` seconds:

```solidity
// ProtectedPriceProviderL2.sol lines 138-153
function _isStale(uint256 refTime, uint256 nowTs, uint256 maxDelta, uint256 futureTol)
    internal pure returns (bool)
{
    if (refTime == 0) return true;
    if (refTime > nowTs) { return (refTime - nowTs) > futureTol; }
    return (nowTs - refTime) > maxDelta;
}
```

There is no `sequencerUptimeFeed` state variable, no `GRACE_PERIOD` constant, and no call to a sequencer uptime feed anywhere in the contract. The constructor accepts no `_sequencerUptimeFeed` parameter. The registry ABI for `PriceProviderL2` (the intended deployed interface) explicitly includes `_sequencerUptimeFeed` as a constructor parameter and exposes `GRACE_PERIOD()` and `sequencerUptimeFeed()` view functions, confirming the protocol designed this check but it is absent from the `ProtectedPriceProviderL2` source.

**Exploit path:**
1. `ProtectedPriceProviderL2` is deployed on Base or HyperEVM (both L2s per the README) with `MAX_TIME_DELTA = 3600`.
2. Oracle publishes price `P = 2000 USDC/ETH` at `T = 0`; `refTime = T`.
3. L2 sequencer goes down at `T = 1`. Oracle stops updating; `refTime` stays at `T`.
4. Real market price drops to `P' = 1000 USDC/ETH` during downtime.
5. At `T = 1800`, attacker submits a swap via the L1 rollup inbox (bypassing the sequencer).
6. `_isStale(T, T+1800, 3600, ...)` → `1800 ≤ 3600` → `false` (not stale).
7. `getBidAndAskPrice()` returns bid/ask derived from the frozen `P = 2000`.
8. Attacker buys ETH at the stale 2000 USDC price; LP loses ~1000 USDC per ETH traded.
9. On sequencer restart, oracle data is still within `MAX_TIME_DELTA`; attacker front-runs the first oracle update and repeats.

## Impact Explanation

This is a bad-price execution impact: a stale, frozen bid/ask quote reaches the pool swap, allowing a trader to extract value from LPs at pre-downtime prices. The pool is a pure oracle market maker — its solvency guarantee rests entirely on the oracle price being current. The loss per swap is bounded only by pool liquidity and the price gap between the frozen and real market price, which can be arbitrarily large during extended downtime. This directly violates the protocol's stated invariant ("No trade on bad oracle... or (L2) sequencer down") and constitutes a direct loss of LP principal.

## Likelihood Explanation

Requires L2 sequencer downtime, which is an external condition but a documented real-world event on Base and other L2s. No privileged action is required — any public trader can submit transactions through the L1 rollup inbox during sequencer downtime. `MAX_TIME_DELTA` can be set up to 7 days, creating a very wide exploitation window. The attack is repeatable on every sequencer downtime event.

## Recommendation

Add a sequencer uptime feed check to `ProtectedPriceProviderL2`, consistent with the Chainlink-recommended pattern and the `GRACE_PERIOD`/`sequencerUptimeFeed` interface already present in the registry ABI:

```solidity
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600;

function _isSequencerUp() internal view returns (bool) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer != 0) return false;
    if (block.timestamp - startedAt < GRACE_PERIOD) return false;
    return true;
}
```

Call `_isSequencerUp()` at the top of `_computeBidAsk` and return `(0, type(uint128).max)` if it returns `false`. Pass `_sequencerUptimeFeed` through the constructor.

The test demonstrates that `getBidAndAskPrice()` returns a valid (non-stale-sentinel) bid/ask during simulated sequencer downtime, as long as `block.timestamp - refTime < MAX_TIME_DELTA`, allowing a swap to execute at the frozen pre-downtime price.
