### Title
Single Participant Can Unilaterally Freeze All Signature Requests or Force Resharing via `verify_tee()` — (File: `crates/contract/src/lib.rs`)

---

### Summary

`verify_tee()` is gated by a single-participant check (`voter_or_panic()`), not a threshold vote. Any one participant can call it at any time and either set `accept_requests = false` — blocking every `sign()`, `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()` call — or unilaterally trigger a resharing via `transition_to_resharing_no_checks()`, without any threshold consensus.

---

### Finding Description

`verify_tee()` is the on-chain TEE health check that the design doc says nodes should call every 7 days. Its access control is:

```rust
pub fn verify_tee(&mut self) -> Result<bool, Error> {
    self.voter_or_panic();   // ← only requires: caller is any single participant
    ...
}
``` [1](#0-0) 

`voter_or_panic()` calls `voter_account()`, which calls `authenticate_update_vote()` — requiring only that the signer is a current participant, with no threshold requirement. [2](#0-1) 

Once inside, the function has two dangerous unilateral effects:

**Effect 1 — Freeze all requests (`accept_requests = false`):**

When `reverify_and_cleanup_participants()` returns `TeeValidationResult::Partial` and the surviving set would break the threshold relation, the contract sets:

```rust
self.accept_requests = false;
return Ok(false);
``` [3](#0-2) 

This flag is checked in `check_request_preconditions()`, which is called by every user-facing method (`sign`, `request_app_private_key`, `verify_foreign_transaction`) and every node response method (`respond`, `respond_ckd`, `respond_verify_foreign_tx`):

```rust
if !self.accept_requests {
    env::panic_str(&TeeError::TeeValidationFailed.to_string())
}
``` [4](#0-3) 

```rust
if !self.accept_requests {
    return Err(TeeError::TeeValidationFailed.into());
}
``` [5](#0-4) 

**Effect 2 — Unilateral resharing trigger:**

When the surviving set is large enough to maintain threshold, the function calls `transition_to_resharing_no_checks()` directly — bypassing the normal `vote_new_parameters` threshold-voting flow entirely:

```rust
let res = running_state.transition_to_resharing_no_checks(&proposed_parameters);
if let Some(resharing) = res {
    self.protocol_state = ProtocolContractState::Resharing(resharing);
}
``` [6](#0-5) 

Normal resharing requires threshold votes via `vote_new_parameters`. Here, a single participant bypasses that entirely.

---

### Impact Explanation

**Impact: Medium.**

A single participant (strictly below threshold) can:

1. **Freeze the entire request lifecycle**: All `sign()` calls from users fail with `TeeValidationFailed`. All `respond*()` calls from nodes fail. Pending yield-resume requests time out. The MPC network stops producing signatures.
2. **Force an unwanted resharing**: Disrupts the running state, forces all nodes to participate in a new key-resharing ceremony, and blocks signing during the resharing period.

Both effects match the allowed impact: *"request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants."*

---

### Likelihood Explanation

**Likelihood: Medium.**

- Attestations expire every 7 days by design. During any expiry window, a malicious participant can call `verify_tee()` to exploit the degraded state.
- The call requires no special setup beyond being a current participant — no leaked keys, no TEE attacks, no collusion.
- The attacker does not need to be above threshold; a single participant suffices.
- The 7-day attestation cycle means this window recurs regularly in production.

---

### Recommendation

1. **Require threshold consensus before `verify_tee()` can mutate `accept_requests` or trigger resharing.** Accumulate `verify_tee` votes and only act when a threshold of participants agree on the degraded state.
2. **Separate the read path from the write path**: allow any participant to query TEE health, but require threshold votes before any state-mutating consequence (freeze or resharing).
3. **Emit an on-chain event** when `accept_requests` is set to `false` so the network can react and recover quickly.

---

### Proof of Concept

1. Network is `Running` with N=5 participants, threshold T=3.
2. Participant P4's attestation expires (normal 7-day cycle).
3. Malicious participant P1 calls `verify_tee()`.
4. `reverify_and_cleanup_participants()` returns `TeeValidationResult::Partial` with 4 valid participants.
5. **Path A** (if kicking P4 would break threshold): `accept_requests = false`. All `sign()` calls from users now panic with `TeeValidationFailed`. All `respond()` calls from nodes fail. The network stops serving signatures — triggered by one participant, no threshold required.
6. **Path B** (if 4 remaining participants satisfy threshold): `transition_to_resharing_no_checks()` is called. The contract transitions to `Resharing` state, disrupting the running protocol and blocking signing during the resharing ceremony — again triggered by one participant, no threshold vote. [7](#0-6)
