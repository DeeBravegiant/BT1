### Title
ValidationState Not Rolled Back on `block_to_block_record` Failure, Corrupting Batch Difficulty for Subsequent Blocks — (`chia/consensus/multiprocess_validation.py`)

### Summary
In `pre_validate_block`, `vs.difficulty` and `vs.ssi` are unconditionally overwritten from attacker-controlled `finished_sub_slots` data before `block_to_block_record` is called. If `block_to_block_record` raises `ValueError` (invalid sub-epoch summary hash), the function returns `Err.INVALID_SUB_EPOCH_SUMMARY` without restoring the original `vs` values. Because `vs` is the shared in-out state passed across all blocks in a batch, every subsequent block in the batch is validated against the forged difficulty/ssi.

### Finding Description
**Root cause — lines 219–223:**
```python
if len(block.finished_sub_slots) > 0:
    if block.finished_sub_slots[0].challenge_chain.new_difficulty is not None:
        vs.difficulty = block.finished_sub_slots[0].challenge_chain.new_difficulty
    if block.finished_sub_slots[0].challenge_chain.new_sub_slot_iters is not None:
        vs.ssi = block.finished_sub_slots[0].challenge_chain.new_sub_slot_iters
```
There is no guard that the block is actually at a sub-epoch boundary, and no snapshot of the old values is taken.

**Failure path — lines 249–260:**
```python
try:
    block_rec = block_to_block_record(
        constants, blockchain, required_iters, block,
        sub_slot_iters=vs.ssi, prev_ses_block=vs.prev_ses_block,
    )
except ValueError:
    return return_error(Err.INVALID_SUB_EPOCH_SUMMARY)
```
`return_error` returns immediately; `vs` is never restored.

**`block_to_block_record` raises `ValueError` when:**
```python
if ses.get_hash() != found_ses_hash:
    raise ValueError(Err.INVALID_SUB_EPOCH_SUMMARY)
``` [3](#0-2) 

An attacker can set `new_difficulty` to any value and set `subepoch_summary_hash` to a garbage hash. `validate_pospace_and_get_required_iters` (line 231) uses the already-mutated `vs.difficulty`, so the attacker crafts a PoSpace valid for the forged difficulty. The SES hash check then fails, returning the error — but `vs` is left with the forged difficulty.

### Impact Explanation
Every block after the poisoned block in the same batch is pre-validated against the wrong difficulty. If the forged difficulty is higher than the real one, valid blocks fail `validate_pospace_and_get_required_iters` and are rejected, halting the node's ability to advance the chain. This is a **chain halt** reachable by an unprivileged remote peer sending a crafted `FullBlock`.

### Likelihood Explanation
The attacker only needs to:
1. Know the current chain tip (public information).
2. Craft a `FullBlock` with `finished_sub_slots[0].challenge_chain.new_difficulty = D_forged` and a garbage `subepoch_summary_hash`.
3. Find a PoSpace valid for the actual challenge at difficulty `D_forged` (trivial if `D_forged` is set low enough to corrupt `vs`, then a second block in the batch with `D_forged` high causes the halt).

No keys, admin access, or cryptographic breaks are required.

### Recommendation
Save the original `vs` values before mutation and restore them on any early-return error path:

```python
old_difficulty = vs.difficulty
old_ssi = vs.ssi
if len(block.finished_sub_slots) > 0:
    if block.finished_sub_slots[0].challenge_chain.new_difficulty is not None:
        vs.difficulty = block.finished_sub_slots[0].challenge_chain.new_difficulty
    if block.finished_sub_slots[0].challenge_chain.new_sub_slot_iters is not None:
        vs.ssi = block.finished_sub_slots[0].challenge_chain.new_sub_slot_iters

try:
    block_rec = block_to_block_record(...)
except ValueError:
    vs.difficulty = old_difficulty   # rollback
    vs.ssi = old_ssi
    return return_error(Err.INVALID_SUB_EPOCH_SUMMARY)
```

Alternatively, move the `vs` mutation to after `block_to_block_record` succeeds, since the new difficulty/ssi values are only valid if the block record is successfully constructed.

### Proof of Concept
1. Build a batch of two blocks: block N and block N+1 (both connected to the real chain tip).
2. Block N: set `finished_sub_slots[0].challenge_chain.new_difficulty = D_high` (e.g., `uint64(2**60)`), set `subepoch_summary_hash` to 32 zero bytes (invalid). Craft a PoSpace valid for the actual challenge at `D_high` (or use any difficulty that passes `validate_pospace_and_get_required_iters` with the forged `vs.difficulty`).
3. Block N+1: a legitimately valid block at the real difficulty `D_actual`.
4. Call `pre_validate_block(block_N, vs)` → returns `Err.INVALID_SUB_EPOCH_SUMMARY`; assert `vs.difficulty == D_high`.
5. Call `pre_validate_block(block_N+1, vs)` → `validate_pospace_and_get_required_iters` uses `D_high`; valid block N+1 fails with `Err.INVALID_POSPACE`.
6. Expected (correct) behavior: step 4 should restore `vs.difficulty = D_actual`; step 5 should succeed.
