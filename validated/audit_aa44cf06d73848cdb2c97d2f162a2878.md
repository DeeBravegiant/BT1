### Title
Permissionless Absorb Spend Resets LEAVING_POOL Singleton Lock Height, Indefinitely Delaying Pool Exit — (File: `chia/pools/pool_puzzles.py`, `chia/pools/pool_wallet.py`)

### Summary
The pool singleton's absorb ("claim rewards") spend path requires **no owner signature**. When a farmer is in `LEAVING_POOL` state and waiting for `relative_lock_height` blocks before completing the exit, any unprivileged attacker who observes pending farming-reward coins at the farmer's `p2_singleton_puzzle_hash` can construct and submit a valid absorb spend bundle. Each accepted absorb spend creates a new singleton coin at the current block height, resetting the on-chain `ASSERT_HEIGHT_RELATIVE` lock enforced by the waiting-room CLVM puzzle. The farmer is forced to wait another full `relative_lock_height` window, and the attack can be repeated for every new reward coin that accumulates, permanently trapping an actively-farming pool NFT in `LEAVING_POOL`.

---

### Finding Description

**Vulnerability class:** Auth bypass / state-lock reset (direct analog of H-08).

**Step 1 – No signature required for absorb spends.**

`create_absorb_spend` in `pool_puzzles.py` builds a spend bundle that is submitted with an empty (`G2Element()`) aggregate signature: [1](#0-0) 

The comment in `pool_wallet.py` confirms this is intentional: [2](#0-1) 

**Step 2 – Absorb spend is valid while in LEAVING_POOL (waiting-room) state.**

`pool_state_to_inner_puzzle` returns the waiting-room inner puzzle for `LEAVING_POOL`, and `create_absorb_spend` handles it with `inner_sol = Program.to([0, reward_amount, height])`: [3](#0-2) 

The lifecycle test explicitly exercises this path ("ABSORB WHILE IN WAITING ROOM"): [4](#0-3) 

**Step 3 – Absorb spend updates the singleton tip height in the pool store.**

`apply_state_transition` is called for every singleton spend (including absorb spends) and records it with its block height: [5](#0-4) 

**Step 4 – `new_peak` computes the exit eligibility window from `tip_height`.**

After an absorb spend at block height H, `get_tip()` returns H. The wallet then computes:

```
leave_height = H + relative_lock_height
```

and only submits the second travel transaction when `peak_height > leave_height + 2`: [6](#0-5) 

**Step 5 – The on-chain CLVM puzzle enforces the same reset.**

The waiting-room inner puzzle is curried with `relative_lock_height` and emits `ASSERT_HEIGHT_RELATIVE relative_lock_height` relative to the singleton coin being spent. Because the absorb spend creates a **new** singleton coin at height H, any subsequent exit spend must satisfy `current_height ≥ H + relative_lock_height`. This is the on-chain enforcement that the lifecycle test demonstrates fails with `ASSERT_HEIGHT_RELATIVE_FAILED` when attempted too early: [7](#0-6) 

**Step 6 – All inputs needed to construct the absorb spend are public.**

The attacker needs: the current singleton coin spend (on-chain), the launcher coin (on-chain), the genesis challenge (constant), `delay_time`/`delay_ph` from the launcher spend (on-chain), and the height of a pending farming-reward coin at the `p2_singleton_puzzle_hash` (on-chain). No private key is required. [8](#0-7) 

---

### Impact Explanation

A farmer in `LEAVING_POOL` state who continues to farm (the normal case) accumulates new `p2_singleton` reward coins with every block they win. An attacker who monitors the chain can submit a fee-0 absorb spend for each new reward coin, resetting the `relative_lock_height` window each time. The farmer is indefinitely prevented from completing the pool-exit state transition. The maximum `relative_lock_height` accepted by the CLI is 1000 blocks (~5.5 hours), so each reset costs the farmer at least that delay. This constitutes a **long-lived inability for farmers to process pool actions** (pool exit / singleton state transition).

No funds are directly stolen, but the farmer's singleton is held hostage in `LEAVING_POOL`, preventing them from switching to self-pooling or another pool, and forcing them to choose between stopping farming (losing rewards) or remaining trapped.

---

### Likelihood Explanation

- The attacker is fully unprivileged; only mempool access is required.
- All data needed to construct the absorb spend is public on-chain.
- The attack is cheap: fee-0 absorb spends are accepted in a non-full mempool.
- The precondition (pending farming-reward coins at the `p2_singleton_puzzle_hash`) is the **normal operating state** for any actively-farming pool NFT.
- The attack can be automated to fire on every new reward coin observed.

---

### Recommendation

1. **Guard the absorb spend path in LEAVING_POOL state.** The waiting-room inner puzzle (`POOL_WAITINGROOM_INNERPUZ`) should require the owner's signature (`AGG_SIG_ME` on the owner pubkey) for the absorb spend type when the singleton is in the waiting-room state, or alternatively disallow absorb spends entirely while in `LEAVING_POOL`.

2. **Decouple the lock-height anchor from the singleton tip.** Instead of measuring `relative_lock_height` from the most recent singleton spend (which absorb spends update), anchor it to the height of the **travel spend** that entered `LEAVING_POOL`. This height is immutable once the state transition is recorded and cannot be reset by absorb spends.

3. **Short-term mitigation:** Farmers leaving a pool should stop farming (remove plots) for the duration of the `relative_lock_height` window to eliminate the supply of reward coins the attacker can exploit.

def create_absorb_spend(
    last_coin_spend: CoinSpend,
    current_state: PoolState,
    launcher_coin: Coin,
    height: uint32,
    genesis_challenge: bytes32,
    delay_time: uint64,
    delay_ph: bytes32,
) -> list[CoinSpend]:
    inner_puzzle: Program = pool_state_to_inner_puzzle(
        current_state, launcher_coin.name(), genesis_challenge, delay_time, delay_ph
    )
    reward_amount: uint64 = calculate_pool_reward(height)
    if is_pool_member_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol: Program = Program.to([reward_amount, height])
    elif is_pool_waitingroom_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, destination_puzhash, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol = Program.to([0, reward_amount, height])
    else:
        raise ValueError
    # full sol = (parent_info, my_amount, inner_solution)
    coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(last_coin_spend)
    assert coin is not None

    if coin.parent_coin_info == launcher_coin.name():
        parent_info: Program = Program.to([launcher_coin.parent_coin_info, launcher_coin.amount])
    else:
        p = Program.from_bytes(bytes(last_coin_spend.puzzle_reveal))
        last_coin_spend_inner_puzzle: Program | None = get_inner_puzzle_from_puzzle(p)
        assert last_coin_spend_inner_puzzle is not None
        parent_info = Program.to(

                last_coin_spend_inner_puzzle.get_tree_hash(),
                last_coin_spend.coin.amount,
            ]
        )
    full_solution: SerializedProgram = SerializedProgram.to([parent_info, last_coin_spend.coin.amount, inner_sol])

    assert coin.puzzle_hash == full_puzzle.get_tree_hash()

    reward_parent: bytes32 = pool_parent_id(height, genesis_challenge)
    p2_singleton_puzzle = create_p2_singleton_puzzle(
        SINGLETON_MOD_HASH, launcher_coin.name(), delay_time, delay_ph
    ).to_serialized()
    reward_coin: Coin = Coin(reward_parent, p2_singleton_puzzle.get_tree_hash(), reward_amount)
    p2_singleton_solution = SerializedProgram.to([inner_puzzle.get_tree_hash(), reward_coin.name()])

    assert full_puzzle.get_tree_hash() == coin.puzzle_hash
    assert get_inner_puzzle_from_puzzle(Program.from_bytes(bytes(full_puzzle))) is not None

    coin_spends = [
        CoinSpend(coin, full_puzzle, full_solution),
        CoinSpend(reward_coin, p2_singleton_puzzle, p2_singleton_solution),
    ]
    return coin_spends
