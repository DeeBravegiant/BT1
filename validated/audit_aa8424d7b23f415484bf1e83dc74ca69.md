### Title
Missing On-Chain Response Verification for `AppPublicKey` CKD Variant Allows Single Attested Participant to Deliver Arbitrary Confidential Key Derivation Output — (`crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` contract method applies cryptographic verification to `AppPublicKeyPV` CKD responses via `ckd_output_check`, but performs **no verification whatsoever** for the `AppPublicKey` variant. Any single attested participant can call `respond_ckd` with an arbitrary `CKDResponse` for a pending `AppPublicKey` request and the contract will accept and deliver it to the user. This is directly analogous to the zkSync `extcodehash` divergence: just as `getCodeHash` checked nonce and code but silently skipped the balance condition, `respond_ckd` checks the response for one variant but silently skips it for the other — breaking the threshold-authorization invariant for an entire class of requests.

---

### Finding Description

`respond_ckd` in `crates/contract/src/lib.rs` handles two variants of the `CKDAppPublicKey` enum:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO CHECK
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check` and panics if the response is inconsistent with the app's public key. For `AppPublicKey`, the arm is a no-op — the response is accepted unconditionally and immediately forwarded to `resolve_yields_for`, which resolves the pending yield and delivers the value to the waiting caller. [2](#0-1) 

Compare this with `respond`, which **always** verifies the signature against the derived public key before resolving: [3](#0-2) 

And `respond_verify_foreign_tx`, which **always** verifies the ECDSA signature against the root key: [4](#0-3) 

There is no leader-only guard in `respond_ckd` — any attested participant may call it: [5](#0-4) 

The attestation check (`assert_caller_is_attested_participant_and_protocol_active`) confirms the caller is a current participant with a valid TEE attestation, but it does not restrict which participant may respond, nor does it verify the content of the response for `AppPublicKey` requests. [6](#0-5) 

---

### Impact Explanation

A single Byzantine attested participant — strictly below the signing threshold — can:

1. Observe a pending `AppPublicKey` CKD request in the contract's `pending_ckd_requests` map.
2. Construct an arbitrary `CKDResponse` (any `big_y`, `big_c` values).
3. Call `respond_ckd` before the legitimate leader does.
4. The contract accepts the response without any cryptographic check and resolves the yield, delivering the fabricated output to the requesting user.

The user receives a fake confidential key derivation output that was never computed by the threshold MPC protocol. Because `resolve_yields_for` drains the entire queue for that request key on the first successful call, the legitimate leader's subsequent call will find no pending yield to resolve — the damage is irreversible for that request. [2](#0-1) 

This directly satisfies the allowed Critical impact: **confidential key derivation output without the required participant authorization** and **bypass of threshold-signature requirements**.

---

### Likelihood Explanation

- The attacker must be an attested participant (TEE attestation required), which is a meaningful barrier.
- However, the threshold model is specifically designed to tolerate up to `n − threshold` Byzantine participants. A single malicious node inside the participant set is the canonical threat model for MPC systems.
- The `AppPublicKey` variant is a production code path (not deprecated, not test-only), reachable by any user calling `request_app_private_key` with `CKDAppPublicKey::AppPublicKey`.
- No race-condition mitigation exists: the first `respond_ckd` call wins and permanently resolves the yield.

---

### Recommendation

1. **Require threshold agreement for `AppPublicKey` responses**: introduce a vote-accumulation map (keyed by `(request, response_hash)`) so that `respond_ckd` only resolves the yield once `threshold` distinct attested participants have submitted the same response.
2. **Alternatively, deprecate `AppPublicKey`**: require all CKD callers to use `AppPublicKeyPV`, which already has on-chain verification via `ckd_output_check`, making the unverifiable code path unreachable.
3. **Consistency fix**: align `respond_ckd` with `respond` and `respond_verify_foreign_tx` by always performing a cryptographic check before resolving any yield.

---

### Proof of Concept

1. User calls `request_app_private_key` with `CKDAppPublicKey::AppPublicKey(some_pk)`, creating a pending `CKDRequest` in `pending_ckd_requests`.
2. Malicious attested participant constructs `CKDResponse { big_y: [0u8; 48].into(), big_c: [0u8; 48].into() }` (or any chosen value).
3. Malicious participant calls `respond_ckd(ckd_request, fake_response)`.
4. Contract executes: `assert_caller_is_attested_participant_and_protocol_active()` passes (caller is a valid participant), the `AppPublicKey` match arm is a no-op, `resolve_yields_for` resolves the yield with the fake response.
5. User's `request_app_private_key` promise resolves with the fabricated `CKDResponse` — the threshold MPC computation was never completed, yet the user receives a "result."
6. The legitimate leader's subsequent `respond_ckd` call finds no pending yield and silently fails, leaving no trace of the attack. [7](#0-6)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
