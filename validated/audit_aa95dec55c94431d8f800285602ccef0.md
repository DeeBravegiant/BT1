### Title
Unconstrained `farmer_reward_address_override` in `RespondSignatures` Enables Malicious Harvester to Redirect Farmer Block Rewards — (`chia/farmer/farmer_api.py`)

---

### Summary

A malicious third-party harvester can redirect the farmer's XCH block reward (1/8 of every block reward) to an arbitrary attacker-controlled address by setting `farmer_reward_address_override` in a `RespondSignatures` message. The farmer accepts this field unconditionally — no fee-quality check, no cryptographic proof, no cross-validation against the earlier `NewProofOfSpace` message — and uses it verbatim as the `farmer_reward_address` in `DeclareProofOfSpace`.

---

### Finding Description

The `RespondSignatures` protocol message contains an optional `farmer_reward_address_override: bytes32 | None` field: [1](#0-0) 

In `FarmerAPI._process_respond_signatures`, the farmer unconditionally replaces its configured `farmer_target` with whatever address the harvester supplies: [2](#0-1) 

The only fee-quality gate in the codebase applies to the **earlier** `NewProofOfSpace.farmer_reward_address_override` field (checked in `new_proof_of_space`): [3](#0-2) 

That check calls `notify_farmer_reward_taken_by_harvester_as_fee`, which only **logs a warning** when the fee quality threshold is not met — it never rejects the override: [4](#0-3) 

Critically, `RespondSignatures.farmer_reward_address_override` is a **separate field** from `NewProofOfSpace.farmer_reward_address_override`. The farmer never checks whether the two match, and applies **no fee-quality check at all** to the `RespondSignatures` version. A malicious harvester can therefore:

1. Send `NewProofOfSpace` with `farmer_reward_address_override=None` — no fee-quality check is triggered.
2. Receive `RequestSignatures` from the farmer.
3. Reply with `RespondSignatures` containing `farmer_reward_address_override=<attacker_address>`.
4. The farmer builds `DeclareProofOfSpace` with `farmer_reward_address=<attacker_address>`: [5](#0-4) 

The full node then farms the block and pays the 1/8 farmer reward to the attacker's address.

---

### Impact Explanation

Every block the farmer wins while connected to the malicious harvester results in the farmer's entire 1/8 block reward (e.g., 0.25 XCH at current schedule) being paid to the attacker. This is an unauthorized, on-chain payout redirection of XCH with no recourse once the block is confirmed. Impact is **High**: unauthorized payout redirection of XCH block rewards via singleton/farmer reward mechanism. [6](#0-5) 

---

### Likelihood Explanation

Third-party harvesters are an explicitly supported and documented feature (CHIP-22). A farmer who connects to a third-party harvester service (e.g., a compressed-plot provider) is exposed to this attack with zero additional prerequisites. The malicious harvester needs only to be connected — no leaked keys, no chain compromise, no cryptographic break required. The attack is silent: the farmer sees no error, only a log warning that is never emitted (because `NewProofOfSpace.farmer_reward_address_override` is `None`).

---

### Recommendation

1. **Cross-validate the two fields**: In `_process_respond_signatures`, if `response.farmer_reward_address_override` is set, verify it matches the `farmer_reward_address_override` that was present in the original `NewProofOfSpace` for this `sp_hash`/`plot_identifier`. Store the override from `NewProofOfSpace` alongside the proof-of-space entry in `self.farmer.proofs_of_space`.

2. **Enforce the fee-quality check, not just log it**: In `notify_farmer_reward_taken_by_harvester_as_fee`, return a boolean indicating whether the threshold was met, and reject the override (return `None` from `_process_respond_signatures`) when the fee quality fails.

3. **Require the override to be committed in the signed data**: The `farmer_reward_address_override` should be included in the messages the harvester signs (e.g., as part of `FoliageBlockData`), so the farmer can verify the harvester actually committed to that address cryptographically.

```python
# and injects an arbitrary farmer_reward_address_override

async def malicious_request_signatures(
    self, request: harvester_protocol.RequestSignatures
) -> Message | None:
    # Sign normally
    result_msg = await HarvesterAPI.request_signatures(self, request)
    assert result_msg is not None

    # Inject attacker-controlled reward address — no fee quality check applies here
    attacker_address = bytes32(b"\xde\xad" * 16)
    response = dataclasses.replace(
        RespondSignatures.from_bytes(result_msg.data),
        farmer_reward_address_override=attacker_address,
        # NewProofOfSpace.farmer_reward_address_override was None,
        # so notify_farmer_reward_taken_by_harvester_as_fee was never called.
    )
    return make_msg(ProtocolMessageTypes.respond_signatures, response)
```
