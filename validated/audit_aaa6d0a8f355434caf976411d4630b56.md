### Title
Cross-Protocol Key Blob Confusion: ECDSA-MP and Schnorr-MP Key Blobs Are Mutually Accepted Without a Protocol-Type Discriminator — (File: `src/cbmpc/api/schnorr_mp.cpp`, `src/cbmpc/api/ecdsa_mp.cpp`)

---

### Summary

`ecdsa_mp` and `schnorr_mp` define structurally identical key blob formats with the same version numbers and overlapping curve support. Neither format carries a protocol-type tag. `schnorr_mp::deserialize_key_blob` silently accepts an ECDSA-MP secp256k1 key blob, and `ecdsa_mp::deserialize_key_blob` silently accepts a Schnorr-MP secp256k1 key blob. A party can be induced to sign an arbitrary message under their ECDSA key share via the Schnorr-MP protocol (or vice versa), producing a valid signature under the wrong protocol with no error.

---

### Finding Description

Both `src/cbmpc/api/ecdsa_mp.cpp` and `src/cbmpc/api/schnorr_mp.cpp` define a local `key_blob_v1_t` struct with **identical fields and identical serialization order**:

```cpp
// ecdsa_mp.cpp lines 27-41 — and schnorr_mp.cpp lines 24-38 — byte-for-byte identical layout
struct key_blob_v1_t {
  uint32_t version;
  uint32_t curve;
  std::string party_name;
  buf_t Q_compressed;
  std::map<std::string, buf_t> Qis_compressed;
  coinbase::crypto::bn_t x_share;
  void convert(coinbase::converter_t& c) {
    c.convert(version, curve, party_name, Q_compressed, Qis_compressed, x_share);
  }
};
```

Both modules assign the same version constants: [1](#0-0) [2](#0-1) 

`ecdsa_mp::deserialize_key_blob` rejects only `ed25519`, accepting any other curve including `secp256k1`: [3](#0-2) 

`schnorr_mp::deserialize_key_blob` accepts only `secp256k1`, which is exactly what an ECDSA-MP secp256k1 blob carries: [4](#0-3) 

Neither deserializer checks a protocol-type field — because no such field exists in the blob format. The only discriminating checks are `version` (same values in both modules) and `curve` (both accept secp256k1). An ECDSA-MP secp256k1 key blob therefore passes every validation gate in `schnorr_mp::deserialize_key_blob`, and a Schnorr-MP secp256k1 key blob passes every gate in `ecdsa_mp::deserialize_key_blob`.

The same confusion applies to access-structure blobs (`ac_key_blob_version_v1 = 2`), which are also identical in layout and version number across both modules: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

When `schnorr_mp::sign_additive` (or `sign_ac`) is called with an ECDSA-MP key blob:

1. `schnorr_mp::deserialize_key_blob` parses the blob without error.
2. The Schnorr-MP protocol uses the ECDSA key share `x_share` as the Schnorr additive share.
3. A valid BIP340 Schnorr signature is produced under the ECDSA public key `Q`.

The honest party has signed an arbitrary message under their ECDSA key via the Schnorr protocol, with no error, no warning, and no awareness of the protocol mismatch. The accepted bad output is a valid Schnorr signature over attacker-chosen message content, produced using key material that was never intended for Schnorr use.

The reverse path is equally reachable: a Schnorr-MP secp256k1 key blob passes `ecdsa_mp::deserialize_key_blob` and produces a valid ECDSA signature under the Schnorr public key. [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

The attack requires a party to call `schnorr_mp::sign_additive` (or `sign_ac`) with an ECDSA-MP key blob. This is realistic when:

- The application layer stores key blobs as opaque byte buffers without out-of-band protocol-type metadata (the blob itself carries no such tag, so the application cannot distinguish them by inspection).
- A malicious peer controls session setup and directs the honest party's application to invoke the wrong signing API for a given session.
- A generic dispatch layer routes signing requests to the wrong protocol module.

The library's own README explicitly states that access-control policies and session routing are the integrating application's responsibility, which means the library itself is the last line of defense against protocol-blob confusion — and it provides none.

---

### Recommendation

Add a `uint32_t proto_type` field to `key_blob_v1_t` with distinct, non-overlapping values for each protocol (e.g., `PROTO_ECDSA_MP = 0x01`, `PROTO_SCHNORR_MP = 0x02`), and validate it as the first check in each `deserialize_key_blob` function. Alternatively, assign non-overlapping version number ranges per protocol so that a Schnorr-MP blob's version is structurally rejected by the ECDSA-MP deserializer and vice versa.

---

### Proof of Concept

1. Party A runs `ecdsa_mp::dkg_additive(job, curve_id::secp256k1, ecdsa_key_blob, sid)` — produces a blob with `version=1, curve=secp256k1`.
2. Party A is induced (e.g., by a malicious peer controlling session routing) to call `schnorr_mp::sign_additive(job, ecdsa_key_blob, msg_32bytes, sig_receiver, sig)`.
3. Inside `schnorr_mp::deserialize_key_blob`:
   - `blob.version == 1` ✓ (matches `key_blob_version_v1`)
   - `curve == secp256k1` ✓
   - `x_share` in `[0, q)` ✓
   - `x_share * G == Qi_self` ✓ (the ECDSA share is a valid secp256k1 scalar)
   - No protocol-type check exists — deserialization succeeds.
4. `schnorr_mp::sign` runs the BIP340 Schnorr-MP protocol using the ECDSA key share as the Schnorr additive share.
5. A valid 64-byte BIP340 Schnorr signature is returned under the ECDSA public key `Q`, verifiable with `crypto::bip340::verify(Q, msg, sig)`.

Party A has signed `msg` under their ECDSA key via the Schnorr protocol without consent or awareness of the protocol substitution. [9](#0-8) [10](#0-9)

static error_t deserialize_key_blob(const coinbase::api::job_mp_t& job, mem_t in, coinbase::mpc::ecdsampc::key_t& key) {
  error_t rv = UNINITIALIZED_ERROR;

  if (job.self < 0 || static_cast<size_t>(job.self) >= job.party_names.size())
    return coinbase::error(E_BADARG, "invalid job.self");
  const std::string self_name(job.party_names[static_cast<size_t>(job.self)]);

  key_blob_v1_t blob;
  if (rv = coinbase::convert(blob, in)) return rv;
  if (blob.version != key_blob_version_v1) return coinbase::error(E_FORMAT, "unsupported key blob version");
  if (blob.party_name.empty()) return coinbase::error(E_FORMAT, "invalid key blob");
  if (blob.party_name != self_name) return coinbase::error(E_BADARG, "job.self mismatch key blob");
  if (job.party_names.size() != blob.Qis_compressed.size()) return coinbase::error(E_BADARG, "invalid key blob");

  // Ensure the party name set matches the job (order can differ).
  for (const auto& name_view : job.party_names) {
    const std::string name(name_view);
    if (blob.Qis_compressed.find(name) == blob.Qis_compressed.end())
      return coinbase::error(E_BADARG, "job.party_names mismatch key blob");
  }

  const auto cid = static_cast<curve_id>(blob.curve);
  if (cid == curve_id::ed25519) return coinbase::error(E_FORMAT, "invalid key blob curve");
  const auto curve = to_internal_curve(cid);
  if (!curve.valid()) return coinbase::error(E_FORMAT, "invalid key blob curve");

  const coinbase::crypto::mod_t& q = curve.order();
  if (!q.is_in_range(blob.x_share)) return coinbase::error(E_FORMAT, "invalid key blob");

  coinbase::crypto::ecc_point_t Q;
  if (rv = Q.from_bin(curve, blob.Q_compressed)) return coinbase::error(rv, "invalid key blob");

  coinbase::crypto::ss::party_map_t<coinbase::crypto::ecc_point_t> Qis;
  for (const auto& name_view : job.party_names) {
    const std::string name(name_view);
    const auto it = blob.Qis_compressed.find(name);
    if (it == blob.Qis_compressed.end()) return coinbase::error(E_BADARG, "job.party_names mismatch key blob");

    coinbase::crypto::ecc_point_t Qi;
    if (rv = Qi.from_bin(curve, it->second)) return coinbase::error(rv, "invalid key blob");
    Qis[name] = std::move(Qi);
  }

  coinbase::crypto::ecc_point_t Q_sum = curve.infinity();
  for (const auto& kv : Qis) Q_sum += kv.second;
  if (Q != Q_sum) return coinbase::error(E_FORMAT, "invalid key blob");

  const auto& G = curve.generator();
  const auto it_self = Qis.find(blob.party_name);
  if (it_self == Qis.end()) return coinbase::error(E_FORMAT, "invalid key blob");
  if (blob.x_share * G != it_self->second) return coinbase::error(E_FORMAT, "invalid key blob");

  key.party_name = blob.party_name;
  key.curve = curve;
  key.x_share = blob.x_share;
  key.Qis = std::move(Qis);
  key.Q = std::move(Q);
  return SUCCESS;
}
