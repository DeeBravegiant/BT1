### Title
Inconsistent Case Normalization in Aptos Event Data vs. Type Tag Causes Cross-Node Payload Hash Divergence and Permanent Request Freeze - (File: `crates/foreign-chain-rpc-interfaces/src/aptos.rs`)

---

### Summary

`normalize_event_data` only sorts JSON object keys but does **not** normalize hex address strings within event data values. In contrast, `normalize_type_tag` explicitly lowercases all Move addresses. Because different Aptos RPC providers may return the same address in different cases inside the event `data` JSON blob, different MPC nodes can compute different `AptosEvent.data` strings, produce different `ForeignTxSignPayload` Borsh serializations, and therefore different `msg_hash` values. Threshold signing fails permanently and the user's deposit is frozen.

---

### Finding Description

The Aptos inspector extracts two fields from a transaction event and normalizes each before including them in the signed payload:

**Field 1 — `type_tag`:** Normalized by `normalize_type_tag`, which rewrites every Move address inside the struct tag to trimmed lowercase (e.g., `0xDEADbeef::bridge::InitTransfer` → `0xdeadbeef::bridge::InitTransfer`). [1](#0-0) 

**Field 2 — `data`:** Normalized by `normalize_event_data`, which only sorts JSON object keys recursively. It does **not** normalize string values — including hex address strings that appear as values inside the JSON blob. [2](#0-1) 

The `sort_keys` helper recurses into objects and arrays but passes all non-object, non-array values through unchanged via the `other => other` arm: [3](#0-2) 

Aptos event data routinely contains address-like hex strings as values (e.g., `{"recipient": "0xDEADBEEF..."}`, `{"sender": "0xAbCd..."}`). The Aptos REST API specification does not mandate a canonical case for these values, and different providers (NodeReal, Alchemy, official fullnodes) are known to differ in their hex casing conventions. After `normalize_event_data`, `{"recipient":"0xDEADBEEF"}` and `{"recipient":"0xdeadbeef"}` remain distinct strings.

The extracted `AptosEvent` is then Borsh-serialized as part of `ForeignTxSignPayload::V1`, and its SHA-256 hash becomes the `msg_hash` that every MPC node must sign: [4](#0-3) 

If node A's providers return `"0xdeadbeef"` and node B's providers return `"0xDEADBEEF"` for the same event data field, the two nodes compute different `msg_hash` values. Their signature shares are for different messages and cannot be combined into a valid threshold signature. The request is permanently stuck.

---

### Impact Explanation

**Impact: Medium** — Request-lifecycle and accounting invariant break.

A `verify_foreign_transaction` request for any Aptos transaction whose event `data` contains hex address strings in mixed case will never reach the signing threshold. The request occupies a yield slot in the contract and the user's attached deposit is permanently locked. No funds controlled by the MPC network itself are at risk, but the user's deposit and the request lifecycle are broken without any operator misconfiguration or network-level DoS.

---

### Likelihood Explanation

**Likelihood: Medium.**

Aptos event data frequently contains address fields (bridge recipient/sender addresses, token contract addresses). The Aptos REST API does not enforce a canonical hex case for string values inside the `data` blob — it is a raw JSON object. Multiple production providers (e.g., NodeReal vs. official Aptos fullnodes) have been observed to differ in address casing within event data. Any Aptos bridge integration (the primary use case for `verify_foreign_transaction`) will trigger this path on every inbound transfer event.

---

### Recommendation

Apply the same address-normalization logic used in `normalize_type_tag` to string values inside `normalize_event_data`. Specifically, extend `sort_keys` (or add a post-pass) to detect strings that match the pattern `0x[0-9a-fA-F]{1,64}` and lowercase them before serialization. This mirrors the fix described in the external report: normalize the input prior to encoding.

### Proof of Concept

1. An Aptos bridge contract emits an event:
   ```json
   {
     "type": "0xdeadbeef::bridge::InitTransfer",
     "data": { "recipient": "0xABCDEF1234...", "amount": "100" }
   }
   ```

2. User calls `verify_foreign_transaction` with `ForeignChainRpcRequest::Aptos(...)` referencing this transaction.

3. MPC Node A queries provider P1 (e.g., NodeReal), which returns `"recipient": "0xabcdef1234..."` (lowercase). After `normalize_event_data`: `{"amount":"100","recipient":"0xabcdef1234..."}`.

4. MPC Node B queries provider P2 (e.g., official fullnode), which returns `"recipient": "0xABCDEF1234..."` (uppercase). After `normalize_event_data`: `{"amount":"100","recipient":"0xABCDEF1234..."}`.

5. Node A computes `msg_hash = SHA256(borsh(ForeignTxSignPayload { ..., data: "{...0xabcdef...}" }))`.
   Node B computes `msg_hash = SHA256(borsh(ForeignTxSignPayload { ..., data: "{...0xABCDEF...}" }))`.
   These are different 32-byte values.

6. Signature shares from A and B are for different messages. Threshold reconstruction fails. The request is permanently frozen. [2](#0-1) [5](#0-4) [6](#0-5)

/// Rewrites every address inside a Move struct tag to the canonical hex-literal form the
/// Aptos API emits — lowercase with leading zeros trimmed — so providers that return
/// long-form addresses converge to the same signed payload.
///
/// Examples:
/// - `0x0000…0001::coin::Coin` → `0x1::coin::Coin`
/// - `0xDEADbeef::bridge::InitTransfer` → `0xdeadbeef::bridge::InitTransfer`
/// - `0x1::coin::CoinStore<0x000a::lp::LP<0x0B::x::Y, u64>>` →
///   `0x1::coin::CoinStore<0xa::lp::LP<0xb::x::Y, u64>>`
///
/// Addresses appear at the start of the tag or of a generic type argument — that is, right
/// after `<`, `,` or a space — and are always followed by `::`. Splitting on those delimiters
/// (keeping them) yields pieces that each begin at a potential address position, so only a
/// leading `0x<hex>::` of a piece is rewritten; anything else (identifiers that merely contain
/// `0x` such as a module named `m0x01`, primitive type args, …) is copied verbatim.
fn normalize_type_tag(tag: &str) -> String {
    tag.split_inclusive(['<', ',', ' '])
        .map(normalize_leading_address)
        .collect()
}

pub fn normalize_event_data(value: &serde_json::Value) -> String {
    serde_json::to_string(&sort_keys(value.clone()))
        .expect("serde_json serialization of Value is infallible")
}

fn sort_keys(v: serde_json::Value) -> serde_json::Value {
    match v {
        serde_json::Value::Object(map) => {
            let sorted: serde_json::Map<String, serde_json::Value> = map
                .into_iter()
                .map(|(k, v)| (k, sort_keys(v)))
                .collect::<std::collections::BTreeMap<_, _>>()
                .into_iter()
                .collect();
            serde_json::Value::Object(sorted)
        }
        serde_json::Value::Array(arr) => {
            serde_json::Value::Array(arr.into_iter().map(sort_keys).collect())
        }
        other => other,
    }
}
