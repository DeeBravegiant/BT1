### Title
Missing CKD Output Validation for `AppPublicKey` Variant Allows Single Attested Participant to Deliver Fabricated Key Derivation Response - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs cryptographic output validation only for the `AppPublicKeyPV` variant of CKD requests, but performs **no validation whatsoever** for the legacy `AppPublicKey` variant. A single attested MPC participant — strictly below the signing threshold — can call `respond_ckd` with an entirely fabricated `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept and deliver it to the requesting user without any cryptographic check.

---

### Finding Description

In `crates/contract/src/lib.rs`, the `respond_ckd` function branches on the app public key variant: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← zero validation
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` enforces the pairing equation:

> `e(big_c, g2) = e(big_y, pk2) · e(hash_point, public_key)` [2](#0-1) 

This cryptographically binds the response to the MPC network's public key and the user's app public key. If the check fails, the call panics and the response is never delivered.

For `AppPublicKey`, the arm is an empty block. The response is passed directly to `resolve_yields_for`, which resumes the user's yield promise with whatever bytes were supplied: [3](#0-2) 

The `AppPublicKey` variant is the "privately verifiable (legacy)" mode: the user supplies only `pk1 = a·G1`. Without the paired `pk2 = a·G2`, the pairing equation cannot be evaluated on-chain. However, this architectural gap means the contract has **no mechanism** to reject a fabricated response for this variant. The correct output satisfies `big_c − a·big_y = H(pk, app_id)·msk`, but the user cannot verify this without knowing the MPC network's master secret `msk`. They receive the response and must trust it.

The `AppPublicKey` variant is still actively supported and documented as a valid request path: [4](#0-3) 

---

### Impact Explanation

A single attested MPC participant (below the signing threshold) can:

1. Observe any pending `AppPublicKey` CKD request in `pending_ckd_requests`.
2. Call `respond_ckd` with an arbitrary fabricated `CKDResponse` — any `big_y` and `big_c` values.
3. The contract passes all access checks (`assert_caller_is_attested_participant_and_protocol_active`) and the empty `AppPublicKey` arm, then calls `resolve_yields_for`, delivering the fabricated bytes to the user.
4. The user receives incorrect key material. Since they cannot verify correctness without `msk`, they may use this material to derive a foreign-chain address, deposit funds there, and permanently lose access — the correct derived key is never recoverable from the fabricated output.

This is **confidential key derivation output delivered without the required threshold-participant authorization**: the threshold of participants never computed or authorized this output; a single Byzantine node forged it.

**Impact class**: Critical — unauthorized confidential key derivation output without required participant authorization.

---

### Likelihood Explanation

The attacker must be an attested MPC participant. This is a meaningful barrier (TEE attestation required), but it is strictly below the signing threshold. In a network of `n` participants with governance threshold `t` (≥ 60% of `n`), any one of the `n` attested participants can exploit this independently. The `AppPublicKey` variant is the legacy default and remains in production use. No collusion, no key leakage, and no privileged operator access is required — only a valid TEE attestation.

---

### Recommendation

Because `pk2` is absent in the `AppPublicKey` variant, the pairing check cannot be replicated exactly. Two mitigations are viable:

1. **Deprecate and gate `AppPublicKey`**: Reject new `request_app_private_key` calls that use `AppPublicKey` and require callers to migrate to `AppPublicKeyPV`, which supports full on-chain output verification.
2. **Require `AppPublicKeyPV` for `respond_ckd`**: Add a guard at the top of `respond_ckd` that panics if `request.app_public_key` is the `AppPublicKey` variant, forcing all CKD responses through the verified path.

Either approach closes the gap by ensuring every `respond_ckd` call is subject to the same cryptographic output check that `AppPublicKeyPV` already enforces.

---

### Proof of Concept

```
1. User calls `request_app_private_key` with:
       app_public_key = AppPublicKey(pk1)   // legacy variant
   Contract stores the pending CKD request.

2. Malicious attested participant calls `respond_ckd` with:
       request  = <the pending CKDRequest>
       response = CKDResponse { big_y: [0xff; 48], big_c: [0xde; 48] }
                  // completely fabricated values

3. Contract executes:
       match &request.app_public_key {
           AppPublicKey(_) => {}   // no check — falls through
           ...
       }
       resolve_yields_for(...)    // delivers fabricated bytes to user

4. User's yield promise resumes with the fabricated (big_y, big_c).
   User computes big_c − a·big_y and obtains garbage key material.
   Any funds sent to the foreign-chain address derived from this
   material are permanently inaccessible.
```

```rust
pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
```
