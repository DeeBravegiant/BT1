Audit Report

## Title
Permanent ETH Freeze via Reverting `receive()` in Push-ETH Withdrawal Pattern тАФ (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
Any smart contract with a reverting `receive()` can call `initiateWithdrawal` to queue an ETH withdrawal. Once the operator calls `unlockQueue` тАФ which irreversibly burns the rsETH and redeems ETH from the unstaking vault into `LRTWithdrawalManager` тАФ every subsequent attempt to complete the withdrawal reverts with `EthTransferFailed`, leaving the ETH permanently locked in the contract. No on-chain function can recover it without a contract upgrade.

## Finding Description
**Root cause тАФ hard-revert push-ETH pattern:**

`_transferAsset` at lines 876тАУ879 pushes ETH to an arbitrary caller-controlled address and reverts the entire call on failure:

```solidity
(bool sent,) = payable(to).call{ value: amount }("");
if (!sent) revert EthTransferFailed();
``` [1](#0-0) 

**State mutation ordering in `_processWithdrawalCompletion`:**

The function performs all state mutations тАФ `popFront()` (line 705), `delete withdrawalRequests[requestId]` (line 712), and `unlockedWithdrawalsCount[asset]--` (line 717) тАФ before calling `_transferAsset` at line 734. [2](#0-1) [3](#0-2) 

When `_transferAsset` reverts, the EVM rolls back every state change in the transaction, including the `popFront()`, the `delete`, and the decrement. The request is restored to the unlocked state and the ETH remains in the contract. Because the user's `receive()` always reverts, every future call to `completeWithdrawal` or `completeWithdrawalForUser` will also revert тАФ the request is permanently stuck.

**Two-phase design makes the window irreversible:**

1. `initiateWithdrawal` (line 150) тАФ open to any address, including smart contracts; rsETH is pulled into the manager. [4](#0-3) 
2. `unlockQueue` (line 268) тАФ burns the rsETH held by the manager (`burnFrom` at line 305) and redeems ETH from the unstaking vault (`redeem` at line 307). After this point the rsETH is gone. [5](#0-4) 
3. `completeWithdrawal` / `completeWithdrawalForUser` (lines 183, 192) тАФ both call `_processWithdrawalCompletion(asset, user, ...)`, which always pushes ETH to the original `user` address. There is no way to redirect the ETH to a different address. [6](#0-5) 

**No admin rescue path:**

`sweepRemainingAssets` is gated on `hasUnlockedWithdrawals(asset) == false` (line 403). Because `unlockedWithdrawalsCount[asset]` can never be decremented for the stuck request (the decrement always reverts with the transfer), `hasUnlockedWithdrawals` permanently returns `true`, and `sweepRemainingAssets` permanently reverts with `PendingWithdrawalsExist`. [7](#0-6) [8](#0-7) 

No other cancel, force-complete, or emergency-ETH-recovery function exists in the contract.

**Misleading NatDoc compounds the risk:**

The comment on `completeWithdrawalForUser` (line 191) reads "potential gas grief scenarios are non-impactful for ETH," conflating gas-griefing (execution continues) with a full revert (execution aborts, state rolled back). This misdirects reviewers away from the actual severity. [9](#0-8) 

## Impact Explanation
After `unlockQueue` executes, the rsETH is burned and the corresponding ETH is held in `LRTWithdrawalManager`. If the recipient contract's `receive()` reverts, that ETH can never leave the contract through any existing on-chain path. This is **permanent freezing of user funds**, matching the Critical impact class.

## Likelihood Explanation
The attacker needs only to: (1) acquire rsETH (tradeable on DEXes), (2) deploy a contract whose `receive()` reverts, (3) call `initiateWithdrawal` from that contract, and (4) wait for the operator's routine `unlockQueue` call. No privileged access is required. The attack is repeatable and can be executed by any external actor. Legitimate smart-contract wallets (multisigs, vaults) that happen to lack a payable `receive()` are also at risk without any malicious intent.

## Recommendation
Replace the push-ETH pattern with a **pull-payment** pattern: store the owed ETH amount per user in a mapping after `unlockQueue` and expose a separate `claimETH()` function that lets users pull their own funds. Alternatively, wrap the `_transferAsset` call in a `try/catch` (or check-and-store pattern) so that a single user's revert stores the failed amount in a claimable mapping rather than reverting the entire completion transaction.

2. Deploy `MaliciousReceiver`, fund it with rsETH.
3. Call `initiateWithdrawal(ETH_TOKEN, amount, "")` from `MaliciousReceiver`.
4. Prank the operator role and call `unlockQueue(ETH_TOKEN, ...)`.
5. Assert rsETH balance of manager is 0 (burned) and `address(withdrawalManager).balance > 0`.
6. Call `completeWithdrawal(ETH_TOKEN, "")` from `MaliciousReceiver` тАФ assert it reverts with `EthTransferFailed`.
7. Prank operator and call `completeWithdrawalForUser(ETH_TOKEN, address(maliciousReceiver), "")` тАФ assert it also reverts.
