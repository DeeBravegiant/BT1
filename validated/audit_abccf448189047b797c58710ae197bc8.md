Looking at the loop in `_add_coin_states` and all the validation layers involved.

### Title
Unbounded Pool Wallet Singleton Loop via Malicious Peer in `_add_coin_states` — (`chia/wallet/wallet_state_manager.py`)

### Summary

A malicious connected peer can cause the wallet's `_add_coin_states` method to loop indefinitely inside a held DB writer transaction by controlling responses to `get_coin_state` and `fetch_coin_spend`. The loop has no iteration bound, no timeout, and no cycle detection. It holds `db_wrapper.writer()` for its entire duration, permanently blocking all subsequent wallet sync updates.

---

### Finding Description

The loop at lines 2089–2123 of `wallet_state_manager.py` is entered whenever a `POOLING_WALLET` coin record is found with `spent_height is not None` and `coin.amount == 1`: [1](#0-0) 

The loop's only exit conditions are:
1. `apply_state_transition` returns `False`
2. `get_most_recent_singleton_coin_from_coin_spend(cs)` returns `None`
3. `curr_coin_state.spent_height is None` (loop guard)

**`apply_state_transition` validation is minimal.** The only check is whether the coin being spent matches the current tip: [2](#0-1) 

There is no signature validation, no puzzle hash validation against the actual blockchain, and no pool state validation. After a successful call, the tip advances to the output of the peer-supplied spend.

**`compute_additions` does not validate signatures.** It runs the CLVM puzzle and extracts `CREATE_COIN` conditions. `AGG_SIG_ME` and all other signature conditions are skipped (only counted for cost): [3](#0-2) 

This means a peer can supply a solution that creates a new singleton coin without a valid signature.

**`fetch_coin_spend` validates puzzle hash only against the peer-supplied `CoinState`.** In the first iteration, `curr_coin_state.coin` is the trusted wallet coin record. In all subsequent iterations, `curr_coin_state` is the `CoinState` returned by `get_coin_state`, which is entirely peer-controlled: [4](#0-3) 

**`pool_store.add_spend` only checks lineage continuity**, not puzzle validity or signatures: [5](#0-4) 

This check is satisfiable by the peer because the output coin's `parent_coin_info` is the name of the coin just spent, which equals `prev.coin.name()` by construction.

**The entire loop runs inside a held DB writer lock:** [6](#0-5) 

---

### Impact Explanation

The wallet node hangs indefinitely inside `async with self.db_wrapper.writer()`, blocking all subsequent DB writes. No other coin state processing, sync update, or wallet operation that requires a DB write can proceed. This is a permanent denial of service against the wallet's sync subsystem, matching the **High** impact category: *permanent or long-lived inability for honest wallets to process sync updates*.

---

### Likelihood Explanation

- The attacker only needs to be a connected peer — achievable via standard peer discovery with no special privileges.
- The original singleton coin's puzzle is revealed on-chain when the coin is spent; any peer that has synced the chain knows it for any active pool wallet.
- No cryptographic material (private keys, etc.) is required.
- The attack is deterministic and reproducible.

---

### Recommendation

Add a hard iteration cap to the loop (e.g., refuse to process more than a few hundred transitions in a single sync pass). Additionally, validate that each successive `CoinState` returned by `get_coin_state` has a `created_height` consistent with the blockchain (i.e., cross-check against the coin store or a trusted source rather than trusting the peer unconditionally). Consider moving the loop outside the DB writer transaction and committing incrementally with a guard against unbounded growth.

---

### Proof of Concept

**Setup:** Wallet has an active `POOLING_WALLET` with singleton coin S whose puzzle P is known (revealed on-chain).

**Attack loop (peer-side):**

1. Peer responds to the initial `get_coin_state([S])` with `spent_height = H` (non-None).
2. `_add_coin_states` enters the `while` loop. `fetch_coin_spend_for_coin_state` is called; peer returns `(P, crafted_absorb_solution)` where the solution's `CREATE_COIN` condition creates coin S' with puzzle hash `H'` (hash of a trivial puzzle `Q`).
3. `compute_additions` runs CLVM, extracts `CREATE_COIN(H', 1)` → coin S'. No signature check occurs.
4. `apply_state_transition` checks `S.name() == tip_coin.name()` ✓, calls `pool_store.add_spend`, returns `True`.
5. `get_coin_state([S'.name()])` is called; peer returns `CoinState(coin=S', spent_height=H+1)`.
6. Next iteration: `fetch_coin_spend` is called for S'; peer returns `(Q, solution_creating_S'')`. Puzzle hash of S' is `H'` = `Q.get_tree_hash()` ✓.
7. `apply_state_transition` checks `S'.name() == tip_coin.name()` — peer set both, so ✓. Returns `True`.
8. Peer returns `CoinState(coin=S'', spent_height=H+2)`.
9. **Repeat indefinitely.**

The wallet process hangs inside `db_wrapper.writer()`. All sync updates are blocked for the lifetime of the process.

async def fetch_coin_spend(height: uint32, coin: Coin, peer: WSChiaConnection) -> CoinSpend:
    solution_response = await peer.call_api(
        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)
    )
    if solution_response is None or not isinstance(solution_response, RespondPuzzleSolution):
        raise PeerRequestException(f"Was not able to obtain solution {solution_response}")
    coin_id = coin.name()
    if solution_response.response.puzzle.get_tree_hash() != coin.puzzle_hash:
        raise PeerRequestException(f"Peer returned wrong puzzle hash for coin {coin_id}")
    if solution_response.response.coin_name != coin_id:
        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(
        coin,
        solution_response.response.puzzle,
        solution_response.response.solution,
    )
