### Title
Permissionless Direct Deposits Enable Withdrawal DoS via Strict Equality Balance Check — (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`)

---

### Summary

In the Dijkstra era, any unprivileged transaction sender can submit a `directDeposits` field targeting any registered staking account, incrementing its reward balance by an arbitrary amount. The `validateWithdrawals` function in legacy mode enforces a **strict equality** check (`==`) between the withdrawal amount declared in a transaction and the current account balance. An attacker who observes a victim's pending withdrawal transaction can front-run it with a tiny direct deposit, causing the victim's withdrawal to fail with `IncompleteWithdrawals`. This is a direct structural analog to the GMX `createDeposit` DoS: a permissionless write to a shared accounting store breaks a strict equality invariant relied upon by another user's pending operation.

---

### Finding Description

**Strict equality check — `withdrawalsThatDoNotDrainAccounts`**

`libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs` lines 196–210 defines:


This function is called in the Dijkstra era's `validateWithdrawals` when `legacyMode = True`:

```haskell
-- eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs, lines 225–234
validateWithdrawals legacyMode network withdrawals accounts = do
  missingWithdrawals <-
    if legacyMode
      then do
        let (missingWithdrawals, incompleteWithdrawals) =
              case withdrawalsThatDoNotDrainAccounts withdrawals network accounts of
                Nothing -> (Map.empty, Map.empty)
                Just (missing, incomplete) -> (unWithdrawals missing, incomplete)
        failOnNonEmptyMap incompleteWithdrawals IncompleteWithdrawals
        pure missingWithdrawals
```

**Permissionless direct deposits**

`applyDirectDeposits` in `libs/cardano-ledger-core/src/Cardano/Ledger/State/Account.hs` lines 290–298 adds arbitrary coin to any registered account's balance. The only validation in `dijkstraEntitiesTransition` (lines 211–216) is that target accounts are registered — there is no authorization check:

```haskell

  injectFailure . DirectDepositsToMissingAccounts
```

**Attack sequence**

1. Victim holds reward balance `X` in their staking account.
2. Victim constructs a legacy-mode withdrawal transaction `T_victim` with `withdrawalsTxBodyL = [(victimAddr, X)]`.
3. Attacker observes `T_victim` in the mempool and submits `T_attack` containing `directDepositsTxBodyL = [(victimAddr, ε)]` for any `ε > 0`.
4. `T_attack` is processed first (or in the same block before `T_victim`); victim's balance becomes `X + ε`.
5. `T_victim` is processed; `validateWithdrawals` evaluates `X == X + ε` → `False` → `IncompleteWithdrawals` failure.
6. Attacker can repeat step 3 indefinitely to maintain the DoS.

The test in `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertsSpec.hs` lines 93–106 confirms that in legacy mode a withdrawal amount that does not exactly equal the balance produces `IncompleteWithdrawals`.

---

### Impact Explanation

An unprivileged attacker can permanently prevent a victim from executing a withdrawal transaction in legacy mode by continuously prepending tiny direct deposits. The victim's funds are not stolen, but their reward withdrawals are blocked for as long as the attacker continues the attack. Each re-submission by the victim can be countered by another direct deposit. This constitutes attacker-controlled modification of reward withdrawals outside design parameters (the direct deposit mechanism is not intended to be used as a griefing tool against other users' withdrawals).

**Allowed impact matched**: *Medium — Attacker-controlled transactions modify rewards/withdrawals outside design parameters.*

---

### Likelihood Explanation

- The Dijkstra era is present in the production repository and is the next scheduled era.
- Legacy mode applies to any transaction that spends a Plutus script output (confirmed by the test at `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertsSpec.hs` lines 93–106), which is the common case for DeFi-style withdrawal transactions.
- The attack requires no special privilege: any ADA holder can submit a direct deposit transaction.
- The attacker's cost is only the transaction fee per direct deposit; the victim must pay a fee for each failed re-submission.
- The attack is repeatable and can be automated.

---

### Recommendation

1. **Remove the strict equality requirement in legacy mode.** Replace `withdrawalsThatDoNotDrainAccounts` (which uses `==`) with `withdrawalsThatExceedAccountBalance` (which uses `<=`) uniformly across all modes, as is already done in non-legacy mode. This mirrors the GMX recommendation to "remove the strict equality for the `executionFee`."

2. **Alternatively, restrict direct deposits.** Require that the transaction sender is the account owner (or an authorized party) when specifying `directDepositsTxBodyL` entries, so that third parties cannot unilaterally modify another user's balance.

### Proof of Concept

```

  victimAddr balance = 1_000_000 lovelace (reward1)

Step 1 — Victim constructs T_victim (legacy mode, Plutus input):
  withdrawalsTxBodyL = [(victimAddr, 1_000_000)]

Step 2 — Attacker submits T_attack before T_victim:
  directDepositsTxBodyL = [(victimAddr, 1)]   -- adds 1 lovelace

Step 3 — T_attack processed:
  victimAddr balance = 1_000_001 lovelace

Step 4 — T_victim processed:
  validateWithdrawals legacyMode:
    withdrawalsThatDoNotDrainAccounts:
      1_000_000 == 1_000_001  →  False
  → IncompleteWithdrawals [(victimAddr, Mismatch 1_000_000 1_000_001)]
  → T_victim REJECTED

Step 5 — Attacker repeats Step 2 for each re-submission by victim.
```
