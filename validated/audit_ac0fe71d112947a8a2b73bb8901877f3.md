### Title
Single Byzantine Participant Can Deliver Unverifiable Garbage CKD Response, Permanently Corrupting Derived-Key Output for `AppPublicKey` Requests - (File: `crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` in the MPC contract enforces on-chain cryptographic verification of the CKD response **only** for the `AppPublicKeyPV` variant. For the `AppPublicKey` (legacy, single-G1-point) variant, the contract accepts any `(big_c, big_y)` pair from any single attested participant without verification. Because `resolve_yields_for` permanently removes the request from the pending map on the first accepted call, a single Byzantine participant below the signing threshold can front-run honest nodes, deliver garbage, and permanently corrupt the caller's derived-key output with no recovery path.

---

### Finding Description

`respond` (for threshold signatures) always verifies the submitted signature on-chain before resolving yields:

```rust
// crates/contract/src/lib.rs:586-644
let signature_is_valid = match (&response, public_key) {
    (Secp256k1(...), ...) => verify_ecdsa_signature(...).is_ok(),
    (Ed25519 {...}, ...) => verify_eddsa_signature(...).is_ok(),
    ...
};
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
```

`respond_ckd` applies the same pairing-based check **only** for `AppPublicKeyPV`:

```rust
// crates/contract/src/lib.rs:675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // ← no verification at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

Immediately after this match, regardless of variant, the contract calls:

```rust
// crates/contract/src/lib.rs:684-688
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

`resolve_yields_for` removes the request from the map and resumes all queued yields with the supplied bytes in a single pass. Once called, the request is gone; there is no second chance.

The `ckd_output_check` function that **is** used for `AppPublicKeyPV` verifies the pairing equation `e(big_c, G2) = e(big_y, app_pk2) · e(H(pk, app_id), pk)`, which cryptographically binds the response to the MPC master key and the caller's `app_id`. No equivalent binding exists for `AppPublicKey`.

---

### Impact Explanation

The CKD protocol derives a deterministic secret `s = HKDF(msk · H(pk, app_id))` that is independent of the ephemeral `app_public_key`; the ephemeral key only encrypts `s` in transit. A garbage `(big_c, big_y)` pair decrypts to a random, wrong scalar. The caller:

1. Receives the garbage blob via yield-resume.
2. Attempts to decrypt: `sig = big_c + (−a) · big_y` — produces a random point, not `msk · H(pk, app_id)`.
3. Cannot derive the correct `s`.
4. Cannot retry with the **same** `app_public_key` (request is resolved and removed).
5. Can retry with a **new** ephemeral key, but the attacker can front-run every retry indefinitely.

If the caller (a TEE app or developer contract) had already committed funds to an address derived from the expected `s` — the standard CKD use-case — those funds are permanently inaccessible. This satisfies the "permanent freezing of funds" criterion. At minimum, the request-lifecycle invariant (honest response delivered to caller) is broken by a single sub-threshold participant.

---

### Likelihood Explanation

- Only **one** attested participant is required; the attack is strictly below the signing threshold.
- The attacker monitors the NEAR chain for `request_app_private_key` events (public mempool / indexer), then submits `respond_ckd` with arbitrary `big_c`/`big_y` before honest nodes respond.
- The `AppPublicKey` variant is still fully supported and is the default in the example CLI (`crates/ckd-example-cli/src/ckd.rs:33`), so real callers use it.
- No special tooling is needed beyond a standard NEAR transaction.

---

### Recommendation

Apply the same pairing-based output check to the `AppPublicKey` variant. Because `AppPublicKey` provides only `pk1` (a G1 point), the caller cannot supply `pk2` for the full `ckd_output_check` equation. Two options:

1. **Require `AppPublicKeyPV` for all new requests** and deprecate `AppPublicKey` at the contract level (reject it in `request_app_private_key`). The example CLI already supports `--publicly-verifiable`.
2. **Add a weaker binding check for `AppPublicKey`**: verify that `e(big_c − big_y · pk1, G2) = e(H(pk, app_id), pk)` using only the G1 point. This requires the caller to supply `pk1` in a form that allows the pairing, which is already present.

Either way, the fix must be applied before `resolve_yields_for` is called.

---

### Proof of Concept

1. Alice (a TEE app) calls `request_app_private_key` with `AppPublicKey(pk1)` and `derivation_path = "vault"`. A yield is queued under the `CKDRequest` key.

2. Mallory, a single Byzantine attested participant, observes the pending request on-chain via `get_pending_ckd_request`.

3. Mallory calls `respond_ckd(request, CKDResponse { big_c: G1::generator(), big_y: G1::generator() })` — arbitrary garbage points.

4. The contract reaches line 676: `dtos::CKDAppPublicKey::AppPublicKey(_) => {}` — no check is performed.

5. `resolve_yields_for` is called at line 684. The request is removed from `pending_ckd_requests` and all queued yields are resumed with the garbage bytes.

6. Alice's yield-resume callback fires with `Ok(CKDResponse { big_c: G1::generator(), big_y: G1::generator() })`.

7. Alice computes `sig = big_c + (−a) · big_y = G1 − a·G1 = (1−a)·G1`, which is not `msk · H(pk, app_id)`. The derived key `s` is wrong.

8. Alice retries with a fresh ephemeral key. Mallory front-runs again. Alice can never obtain the correct `s`.

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}
