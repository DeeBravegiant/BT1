**The assert at line 1073 is unconditional:** [1](#0-0)


This runs regardless of whether `cc_signage_point` was set. The only conditional block above it (lines 1062–1071) guards the signage point path, but the infusion point assert is always reached.

**The call site in `_validate_segment`:** [2](#0-1) 

`_get_challenge_block_vdfs` is called when `sampled and sub_slot_data.is_challenge()` is True. `is_challenge()` (defined in `chia_rs`) returns True when `proof_of_space` is set — confirmed by the unconditional `assert sub_slot_data.proof_of_space is not None` in `__validate_pospace` at line 1424. [3](#0-2) 

**The propagation path:**

`_get_challenge_block_vdfs` → `_validate_segment` → `_validate_sub_epoch_segments` — none of these have try/except around the assert. [4](#0-3) 

`_validate_sub_epoch_segments` runs inside a `ProcessPoolExecutor` worker spawned by `validate_weight_proof`. [5](#0-4)

**Crafting the trigger is trivial:** `SubSlotData` fields are all optional. A peer can serialize a `SubSlotData` with `proof_of_space` populated (satisfying `is_challenge()`) and `cc_infusion_point=None`. No signature or key is required — the weight proof is accepted from any peer during sync.

### Title
Unconditional `assert sub_slot_data.cc_infusion_point` in `_get_challenge_block_vdfs` triggered by crafted weight proof — (`chia/full_node/weight_proof.py`)

### Summary
An unprivileged remote peer can send a `WeightProof` containing a `SubEpochChallengeSegment` whose challenge-block `SubSlotData` has `proof_of_space` set but `cc_infusion_point=None`. This causes an unconditional `AssertionError` at line 1073 of `weight_proof.py`, which propagates unhandled through `_validate_segment` and `_validate_sub_epoch_segments`, crashing the `ProcessPoolExecutor` worker and preventing the syncing node from completing weight proof validation.

### Finding Description
In `_get_challenge_block_vdfs` (line 1054), after an optional block guarding the signage-point path (lines 1062–1071), the function unconditionally executes:

```python
assert sub_slot_data.cc_infusion_point   # line 1073
assert sub_slot_data.cc_ip_vdf_info      # line 1074
```

There is no prior guard that checks whether `cc_infusion_point` is `None` before reaching these asserts. A `SubSlotData` with `proof_of_space != None` (making `is_challenge()` return True) and `cc_infusion_point=None` is a valid serialized object that any peer can send. When the sampled segment contains such a slot, `_validate_segment` calls `_get_challenge_block_vdfs`, which immediately raises `AssertionError`. No exception handler exists in `_validate_segment` or `_validate_sub_epoch_segments`.

### Impact Explanation
The `AssertionError` propagates out of the `ProcessPoolExecutor` worker, is re-raised at `await task` in `validate_weight_proof`, and surfaces to the full node's sync handler. The syncing node cannot complete weight proof validation for any weight proof containing the crafted segment. If the full node does not catch this exception (no try/except is visible around the `validate_weight_proof` call in `full_node.py`), the node process itself may crash. At minimum, the syncing node is permanently unable to sync via weight proof from any peer delivering the crafted proof, qualifying as a High-severity sync disruption. If the exception reaches the main event loop uncaught, it is Critical (node crash/chain halt for the affected node).

### Likelihood Explanation
Any peer on the network can send a `WeightProof` in response to a sync request. Crafting the malicious `SubSlotData` requires only setting `proof_of_space` to a valid-looking value and leaving `cc_infusion_point=None` — no keys, no mining, no privileged access required. The attack is trivially reproducible.

### Recommendation
Replace the bare asserts with explicit validation that returns a failure result rather than raising:

```python
if sub_slot_data.cc_infusion_point is None or sub_slot_data.cc_ip_vdf_info is None:
    log.error("challenge block SubSlotData missing cc_infusion_point or cc_ip_vdf_info")
    return []  # or raise a handled validation error
```

Additionally, add a top-level try/except in `validate_weight_proof` (and `validate_weight_proof_inner`) to catch unexpected exceptions from worker processes and return `(False, uint32(0), [])` rather than propagating them.
