### Title
NFT Royalty Bypass via Integer Division Truncation in `compute_royalty_amount` - (File: `chia/wallet/nft_wallet/nft_wallet.py`)

### Summary
`compute_royalty_amount` uses integer floor division that silently truncates small royalty amounts to zero. When this occurs, the wallet omits the royalty payment entirely from the offer spend bundle and passes an empty trade prices list to the on-chain NFT transfer program, causing the NFT to settle without paying any royalty to the creator.

### Finding Description
`compute_royalty_amount` computes the royalty owed using two sequential integer floor divisions:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

`MAX_ROYALTY_BASIS_POINTS` is 10000. For any `offered_amount` where `abs(offered_amount) * percentage < 10000`, the result is 0. The project's own test suite explicitly documents this behavior:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

When `compute_royalty_amount` returns 0 for all NFTs in the offer, two downstream effects occur:

**Effect 1 — Royalty coin is skipped entirely.** In `make_nft1_offer`, if the sum of all royalty payment amounts is 0, the wallet skips creating the royalty coin spend:

```python
if sum(p.amount for _, p in payments) == 0:
    continue
``` [3](#0-2) 

**Effect 2 — Trade prices list passed to the on-chain NFT transfer program is emptied.** The wallet filters out any trade price whose computed royalty is zero before embedding it in the NFT spend solution:

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [4](#0-3) 

The on-chain `NFT_OWNERSHIP_TRANSFER_PROGRAM_ONE_WAY_CLAIM_WITH_ROYALTIES` puzzle enforces royalties based solely on the trade prices list embedded in the solution. With an empty list, the puzzle enforces no royalty, and the NFT ownership transfer is accepted by consensus without any payment to the creator.

Additionally, the `royalty_calculation` static method used by the RPC endpoint `nft_calculate_royalties` contains the same truncation:

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
``` [5](#0-4) 

### Impact Explanation
An unprivileged NFT buyer can craft an offer where the fungible payment amount is small enough that `compute_royalty_amount` returns 0. For a 1% royalty (100 basis points), any offered amount below 10,000 mojos triggers this. The offer settles on-chain with no royalty coin created and no on-chain enforcement of the royalty obligation. NFT creators permanently lose royalty payments they are entitled to. This is an unauthorized accounting change affecting NFT offer settlement — a Critical-class impact.

### Likelihood Explanation
Any user who understands the royalty calculation can deliberately choose a small fungible amount (or split an offer across many NFTs to reduce the per-NFT amount below the truncation threshold via `royalty_split`). No special privileges, leaked keys, or cryptographic breaks are required. The truncation threshold scales with the royalty percentage, making it reachable for any NFT with a non-trivial royalty.

### Recommendation
Replace the two-step integer floor division with a single multiplication before division to minimize truncation loss:

```python
amount = abs(offered_amount) * percentage // (royalty_split * MAX_ROYALTY_BASIS_POINTS)
```

Additionally, after computing the royalty, check whether the result is zero while the percentage is non-zero and either reject the offer or round up to 1 mojo to ensure the creator always receives at least a minimal payment when a royalty is configured.

### Proof of Concept
With `royalty_percentage = 100` (1%) and `offered_amount = 50` mojos:

```python
# Current behavior — documented by the existing test
result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
# abs(-50) // 1 * 100 // 10000 = 50 * 100 // 10000 = 5000 // 10000 = 0
assert result == uint64(0)  # royalty silently zeroed
``` [2](#0-1) 

A buyer offers 50 mojos for an NFT with a 1% royalty. `compute_royalty_amount` returns 0. The wallet skips the royalty coin spend and passes an empty trade prices list to the NFT transfer program. The spend bundle is accepted by consensus, the NFT changes hands, and the creator receives 0 mojos instead of the expected royalty.

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
