Audit Report

## Title
`notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` Unconditionally Cleared to Zero Without Corresponding Transfer, Permanently Losing Accumulated Notional Fees - (File: `metric-core/contracts/MetricOmmPool.sol`)

## Summary

In `MetricOmmPool.collectFees`, the notional fee accumulators `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` are unconditionally reset to zero at lines 429–430 regardless of whether the floor-rounded external transfer amount is zero. For low-decimal tokens such as USDC (6 decimals), where `TOKEN_0_SCALE_MULTIPLIER = 10^12`, any accumulated scaled amount below `10^12` rounds to zero external units, no transfer occurs, yet the accumulator is wiped. The orphaned scaled units remain in the pool's ERC-20 balance but are no longer tracked by any accounting variable, and in pools configured with zero spread fees they can never be recovered.

## Finding Description

`collectFees` (lines 411–430) converts scaled fee amounts to external token units using `Math.Rounding.Floor` via `deltasScaledToExternal`:

```solidity
// MetricOmmPool.sol L626
deltaAmount0 = scaledDeltaAmount0 / TOKEN_0_SCALE_MULTIPLIER;
```

Transfers are conditional on the result being `> 0` (lines 416–427), but the accumulator reset is unconditional:

```solidity
// MetricOmmPool.sol L429-430
notionalFeeToken0Scaled = 0;
notionalFeeToken1Scaled = 0;
```

`TOKEN_0_SCALE_MULTIPLIER` is `10 ** (internalDecimals - token0Decimals)` (factory lines 658–664). For a USDC/18-decimal pool, this equals `10^12`. Any `notionalFeeToken0Scaled < 10^12` produces zero external USDC, no transfer fires, but the accumulator is still zeroed.

After the reset, the orphaned tokens remain in `balance0()` but are absent from both `binTotals.scaledToken0` and `notionalFeeToken0Scaled`. On the next `collectFees` call the surplus formula (lines 385–388) picks them up:

```solidity
surplus0Scaled = balance0() * TOKEN_0_SCALE_MULTIPLIER
               - uint256(binTotals.scaledToken0)
               - notionalFee0AmountScaled;  // now 0
```

However, spread fee distribution (lines 391–395) is gated on `spreadSumE6 != 0`:

```solidity
uint256 spreadFee0ToProtocolScaled = spreadSumE6 == 0 ? 0 : (surplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
```

In a notional-only pool (`protocolSpreadFeeE6 = 0, adminSpreadFeeE6 = 0`), `spreadSumE6 == 0`, so the orphaned surplus is never distributed. The tokens are permanently stuck.

The trigger is fully unprivileged: `collectPoolFees` (factory line 379) carries no access control beyond `nonReentrant`, allowing any address to call it repeatedly after small swaps to drain the accumulator in dust increments.

## Impact Explanation

Direct, permanent loss of protocol and admin fee revenue. Accumulated notional fees paid by traders are destroyed — neither the protocol nor the admin receives them. In a notional-only fee pool (spread fees = 0), the orphaned tokens can never be recovered. The maximum loss per `collectFees` invocation is `TOKEN_0_SCALE_MULTIPLIER − 1` scaled units per token per recipient leg (up to ~1 USDC per call for a USDC/X pool). An adversary can call `collectPoolFees` repeatedly after each small swap to cause unbounded cumulative loss. This constitutes a direct loss of protocol fees above Sherlock thresholds.

## Likelihood Explanation

`collectPoolFees` is callable by any address with no restriction. Pools pairing 18-decimal tokens with low-decimal tokens (USDC, USDT) are the primary deployment target, and `TOKEN_0_SCALE_MULTIPLIER = 10^12` for such pairs. Small swaps generating sub-unit notional fees are routine in any active pool. No special setup, privileged access, or non-standard ERC20 behavior is required. The attack is repeatable indefinitely.

## Recommendation

Only subtract from `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` the scaled equivalent of the amount actually transferred, not unconditionally zero the entire accumulator:

```diff
-     notionalFeeToken0Scaled = 0;
-     notionalFeeToken1Scaled = 0;
+     uint256 paid0Scaled = (totalFee0ToAdmin + totalFee0ToProtocol) * TOKEN_0_SCALE_MULTIPLIER;
+     uint256 paid1Scaled = (totalFee1ToAdmin + totalFee1ToProtocol) * TOKEN_1_SCALE_MULTIPLIER;
+     notionalFeeToken0Scaled = uint128(notionalFee0AmountScaled > paid0Scaled
+         ? notionalFee0AmountScaled - paid0Scaled : 0);
+     notionalFeeToken1Scaled = uint128(notionalFee1AmountScaled > paid1Scaled
+         ? notionalFee1AmountScaled - paid1Scaled : 0);
```

This preserves sub-unit dust in the accumulator so it can be collected once it crosses the external-unit threshold.

## Proof of Concept

1. Deploy a pool with USDC (6 decimals) as token0 and an 18-decimal token as token1. Set `protocolNotionalFeeE8 = 1_000_000` (1%), `protocolSpreadFeeE6 = 0`, `adminSpreadFeeE6 = 0`. `TOKEN_0_SCALE_MULTIPLIER = 10^12`.
2. Execute a swap that produces `notionalFeeToken0Scaled = 5 * 10^11` (< `10^12`), requiring a token0 output of ~0.5 USDC — a realistic small swap.
3. Any address calls `factory.collectPoolFees(pool)` (no access control on line 379).
4. Inside `collectFees`: `totalFee0ToProtocol = (5 * 10^11) / 10^12 = 0` (floor). No transfer occurs.
5. `notionalFeeToken0Scaled` is set to 0 (line 429). The 5 * 10^11 scaled units remain in the pool's ERC-20 balance but are now untracked.
6. On the next `collectFees` call, `surplus0Scaled` includes the orphaned units, but since `spreadSumE6 = 0`, `spreadFee0ToProtocolScaled = 0` and `spreadFee0ToAdminScaled = 0`. Tokens are never distributed.
