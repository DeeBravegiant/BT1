Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` uses a Synthetix-style staking rewards model where `rewardPerToken()` short-circuits to return the stored value when `totalKernelStaked == 0`, but the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()`. Any rewards that accrued during a zero-staker interval are permanently unclaimable, and no admin recovery function exists to retrieve them.

## Finding Description

`rewardPerToken()` freezes accumulation when `totalKernelStaked == 0`:

```solidity
// L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;   // frozen
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

However, the `updateReward` modifier at L232-242 unconditionally advances `updatedAt`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();   // frozen when totalKernelStaked == 0
    updatedAt = lastTimeRewardApplicable();    // always advances
    ...
}
```

When the next interaction occurs after a zero-staker gap, `updatedAt` is set to the current time, permanently consuming the gap interval without distributing `rewardRate * gap_duration` tokens to anyone. `initiateWithdrawal` at L320-326 has no floor on `totalKernelStaked`, so any combination of stakers can drive it to zero. There is no `recover`, `sweep`, or emergency withdrawal function in the contract — the locked reward tokens have no exit path.

The contract's own NatSpec at L18-22 explicitly acknowledges this design gap and relies entirely on an off-chain operational assumption ("ensuring there are always some tokens staked"), with zero on-chain enforcement.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** Rewards equal to `rewardRate * (T2 - T1)` for any zero-staker interval `[T1, T2]` are permanently locked in the contract. No staker can ever claim them, and no admin function can recover them. This is a concrete, irreversible loss of yield, not a hypothetical one.

## Likelihood Explanation

The trigger is fully permissionless. Any staker can call `initiateWithdrawal` at any time; no privileged access is required. The scenario is reachable organically (mass exit during market stress) or deliberately by a coordinated group. The developer comment at L18-22 confirms the team is aware of the risk, which further validates that the scenario is considered realistic. The only mitigation is operational, not on-chain.

## Recommendation

In the `updateReward` modifier, only advance `updatedAt` when stakers are present:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```

This preserves the unallocated time window so rewards are distributed once stakers return. As a secondary safeguard, add an admin recovery function to sweep reward tokens that were never distributed, callable only after `finishAt` has passed and the unallocated amount can be computed.

## Proof of Concept

```
Setup:
  rewardRate = R, set by notifyRewardAmount at T=0 (totalKernelStaked > 0, passes guard)
  finishAt = T + duration

T=10:  All stakers call initiateWithdrawal → totalKernelStaked = 0

T=20:  New staker calls stake(1e18) → updateReward fires:
         rewardPerTokenStored = rewardPerToken()
           → totalKernelStaked == 0, returns old rewardPerTokenStored (no accumulation)
         updatedAt = lastTimeRewardApplicable() = T=20
           → gap [T=10, T=20] is consumed

T=20+: New staker's earned() uses updatedAt=T=20, skipping R*(20-10) entirely.

Locked rewards = R * 10 seconds = permanently unclaimable.
No recovery function exists in the contract (grep for recover/sweep/rescue returns no matches).
```
