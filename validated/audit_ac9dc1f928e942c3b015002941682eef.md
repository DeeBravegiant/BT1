Audit Report

## Title
Missing Slippage Protection in L2 Pool Deposit Functions - (`contracts/pools/RSETHPoolV3.sol`, `contracts/pools/RSETHPool.sol`, `contracts/pools/RSETHPoolNoWrapper.sol`, `contracts/pools/RSETHPoolV3ExternalBridge.sol`)

## Summary
All four L2 pool contracts expose public `deposit()` functions that compute the output rsETH/wrsETH amount from a live oracle rate at execution time, with no caller-supplied minimum output parameter. A user who previews the expected output off-chain via `viewSwapRsETHAmountAndFee` and then submits a deposit may receive materially fewer tokens than expected if the oracle rate is updated before their transaction is mined, with no on-chain protection and no recourse. The L1 counterpart (`LRTDepositPool._beforeDeposit`) already implements this protection, confirming the protocol's own design intent.

## Finding Description
Every L2 deposit entry point reads the live oracle rate at execution time and mints/transfers the computed amount with no floor check:

In `RSETHPoolV3.sol` (L246–265 and L271–293), `deposit(string)` and `deposit(address,uint256,string)` call `viewSwapRsETHAmountAndFee`, which internally calls `getRate()` (L304), and immediately mint `rsETHAmount` to the caller with no minimum check. The identical pattern is present in `RSETHPool.sol` (L265–278, L284–305), `RSETHPoolNoWrapper.sol` (L231–244, L250–271), and `RSETHPoolV3ExternalBridge.sol` (L366–384, L390–412).

The L1 path in `LRTDepositPool._beforeDeposit` (L648–669) explicitly accepts `minRSETHAmountExpected` and reverts with `MinimumAmountToReceiveNotMet()` if `rsethAmountToMint < minRSETHAmountExpected`. No equivalent guard exists in any L2 pool.

Exploit flow:
1. User calls `viewSwapRsETHAmountAndFee(1 ether)` off-chain; oracle reports `rsETHToETHrate = 1.05e18`; expected output ≈ 0.952 wrsETH (after fee).
2. User submits `deposit{value: 1 ether}("ref")`.
3. Before the transaction is mined, the protocol oracle is updated to `rsETHToETHrate = 1.10e18`.
4. `deposit()` executes: `rsETHAmount = 1e18 * 1e18 / 1.10e18 ≈ 0.909 wrsETH`.
5. User receives ≈ 0.909 wrsETH — a ~4.5% shortfall — with no revert.

No privileged role, no adversarial actor, and no external protocol failure is required. Normal periodic oracle updates by the protocol are sufficient.

## Impact Explanation
The user's ETH/token input is fully consumed, but the contract delivers fewer rsETH/wrsETH than the user was shown at simulation time. The user does not lose ETH value (the rsETH received is worth the deposited ETH at the updated rate), but the contract fails to deliver the quantity of tokens the user expected and had no mechanism to enforce. This maps precisely to **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
The rsETH oracle rate is updated periodically by the protocol as the underlying staking yield accrues. On L2 networks with short block times (e.g., Arbitrum, Optimism, Base), an oracle update can land in the same block or the immediately preceding block relative to a user's deposit. No attacker is needed; any user depositing on any of the four affected contracts is exposed on every deposit. The discrepancy scales with the magnitude of the rate update and the deposit size.

## Recommendation
Add a `minRsETHAmountExpected` parameter to every `deposit()` overload in all four L2 pool contracts, mirroring the L1 pattern in `LRTDepositPool._beforeDeposit`. After computing `rsETHAmount`, revert if it falls below the caller-supplied minimum:

```solidity
function deposit(string memory referralId, uint256 minRsETHAmountExpected) external payable ... {
    (uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(msg.value);
    if (rsETHAmount < minRsETHAmountExpected) revert MinimumAmountToReceiveNotMet();
    feeEarnedInETH += fee;
    wrsETH.mint(msg.sender, rsETHAmount);
    emit SwapOccurred(msg.sender, rsETHAmount, fee, referralId);
}
```

Apply the same pattern to the ERC-20 `deposit(address,uint256,string,uint256)` overload in all four contracts.

## Proof of Concept
Foundry fork test plan:

```solidity
function test_slippage_no_protection() public {
    // Fork L2 at block N where rsETHToETHrate = 1.05e18
    uint256 depositAmount = 1 ether;
    (uint256 expectedRsETH,) = pool.viewSwapRsETHAmountAndFee(depositAmount);
    // expectedRsETH ≈ 0.952e18

    // Simulate oracle update to 1.10e18 (warp or mock oracle)
    vm.mockCall(oracle, abi.encodeWithSelector(IOracle.getRate.selector), abi.encode(1.10e18));

    uint256 balanceBefore = wrsETH.balanceOf(user);
    vm.prank(user);
    pool.deposit{value: depositAmount}("ref");
    uint256 received = wrsETH.balanceOf(user) - balanceBefore;
    // received ≈ 0.909e18, less than expectedRsETH, no revert occurred
    assertLt(received, expectedRsETH);
}
```

Contrast: calling `LRTDepositPool.depositETH{value: 1 ether}(0.952e18, "ref")` after the same oracle update would revert with `MinimumAmountToReceiveNotMet()`, demonstrating the asymmetry between L1 and L2 deposit paths.
