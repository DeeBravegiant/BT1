Audit Report

## Title
Missing Slippage Protection on `removeLiquidity` Exposes Active-Bin LPs to Sandwich Attacks - (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary

`MetricOmmPool.removeLiquidity` accepts no minimum-output parameters and no deadline. Token amounts returned to the LP are computed as a straight proportional share of the live bin balances at execution time. An attacker can sandwich the removal transaction to shift the active bin's token composition before the LP's burn executes, causing the LP to receive a materially worse token mix than expected with no on-chain guard to revert.

## Finding Description

The `removeLiquidity` function signature carries no slippage parameters:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata extensionData
) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
  returns (uint256 amount0Removed, uint256 amount1Removed)
``` [1](#0-0) 

Inside `LiquidityLib.removeLiquidity`, amounts are computed as a proportional share of the **current** bin balances with no floor check:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [2](#0-1) 

These values are transferred directly to `owner` with no minimum check: [3](#0-2) 

The `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` hooks are optional extension callbacks that carry no slippage enforcement — they merely forward the call to configured extension contracts and cannot substitute for a minimum-output guard: [4](#0-3) 

A grep of all `metric-periphery/**/*.sol` files returns zero matches for `removeLiquidity`, confirming there is no periphery wrapper analogous to `MetricOmmPoolLiquidityAdder` for withdrawals.

In the active bin (`curBinIdx`), both `token0BalanceScaled` and `token1BalanceScaled` are non-zero and their ratio changes continuously as swaps move the cursor position. The swap math confirms that each swap within the active bin directly mutates both balances: [5](#0-4) 

By contrast, slippage protection is present for swaps (`amountOutMinimum`): [6](#0-5) 

And for liquidity additions (`maxAmountToken0`/`maxAmountToken1`): [7](#0-6) 

No equivalent protection exists for removals.

## Impact Explanation

An LP removing shares from the active bin receives whatever token mix exists at execution time. A sandwich attacker front-runs with a large swap that shifts the cursor within the active bin (e.g., draining token1 and filling with token0), causing the LP's proportional withdrawal to be dominated by the cheaper-to-acquire token. The attacker back-runs to restore the price, capturing the round-trip profit minus fees. The LP receives less total market value than the pre-sandwich composition would have yielded, with no on-chain mechanism to revert. This constitutes a direct loss of LP principal above Sherlock thresholds for large positions in pools with low spread fees.

## Likelihood Explanation

- `removeLiquidity` is a public, permissionless function callable by any position owner (`msg.sender != owner` is the only guard).
- Active-bin positions are the normal case for LPs who deposit across a range that includes the current price.
- The attack requires only a standard two-swap sandwich in the same block, which is routine on any chain with a public mempool.
- No special setup, governance action, or malicious token is needed.
- The attack is repeatable on every removal transaction.

## Recommendation

Add minimum-output parameters to `removeLiquidity` at the pool level, or provide a periphery wrapper (analogous to `MetricOmmPoolLiquidityAdder`) that enforces them:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Out,
    uint256 minAmount1Out,
    uint256 deadline,
    bytes calldata extensionData
) external returns (uint256 amount0Removed, uint256 amount1Removed);
```

After `LiquidityLib.removeLiquidity` returns, revert if `amount0Removed < minAmount0Out || amount1Removed < minAmount1Out || block.timestamp > deadline`. This mirrors the protection already present for swaps and liquidity additions.

## Proof of Concept

```
Setup:
  - Pool with token0 (WETH) / token1 (USDC), active bin = 0
  - Bin 0 holds 50% token0 / 50% token1 (both balances non-zero)
  - LP holds 10,000 shares in bin 0
  - LP submits removeLiquidity(owner, salt, [{binIdx:0, shares:10000}], "")
    expecting ~500 WETH + ~500 USDC

Attack (same block, before LP tx):
  1. Attacker calls pool.swap(zeroForOne=false, largeAmount)
     → cursor moves within active bin 0
     → token0BalanceScaled increases, token1BalanceScaled decreases
     → bin 0 now holds ~90% token0 / ~10% token1

LP tx executes:
  2. amount0Scaled = token0BalanceScaled * 10000 / totalShares → ~900 WETH
     amount1Scaled = token1BalanceScaled * 10000 / totalShares → ~100 USDC
     LP receives 900 WETH + 100 USDC instead of 500 WETH + 500 USDC
     At market price (WETH < USDC), LP receives less total value.
     No minAmount1Out guard → transaction does not revert.

Back-run:
  3. Attacker swaps back, restoring price and capturing the spread.

Foundry test outline:
  - Deploy pool, add liquidity to bin 0 with both tokens
  - Record LP's expected amounts at current composition
  - Simulate attacker swap (SwapMath confirms balance mutation)
  - Call removeLiquidity, assert amount1Removed << expected
  - Assert no revert occurs (no guard present)
``` [8](#0-7) [1](#0-0)
