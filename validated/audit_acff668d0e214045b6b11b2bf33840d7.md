Audit Report

## Title
Zero Fee Charged on Token Deposits Due to Uninitialized `tokenFeeBps` — (File: contracts/pools/RSETHPool.sol)

## Summary

`RSETHPool.sol` maintains two separate fee variables: `feeBps` for ETH deposits and `tokenFeeBps[token]` for token deposits. Because `addSupportedToken` never initializes `tokenFeeBps[token]`, every newly added token defaults to a fee of 0. Any user depositing a supported token pays zero protocol fees until an admin separately calls `setTokenFeeBps`, causing the protocol to lose all fee revenue from token deposits during this window.

## Finding Description

The contract declares a per-token fee mapping at line 88:

```solidity
mapping(address token => uint256 feeBps) public tokenFeeBps;
```

The ETH deposit path (`viewSwapRsETHAmountAndFee(uint256)`, line 312) correctly uses the initialized `feeBps`:

```solidity
fee = amount * feeBps / 10_000;
```

The token deposit path (`viewSwapRsETHAmountAndFee(uint256, address)`, lines 335–336) uses `tokenFeeBps[token]`:

```solidity
uint256 feeBpsForToken = tokenFeeBps[token];
fee = amount * feeBpsForToken / 10_000;
```

`addSupportedToken` (lines 637–656) only stores the oracle and bridge — it never assigns `tokenFeeBps[token]`:

```solidity
supportedTokenList.push(token);
supportedTokenOracle[token] = oracle;
tokenBridge[token] = bridge;
// tokenFeeBps[token] is never set
```

Because Solidity mappings default to zero, `tokenFeeBps[token] == 0` for every newly added token, making `fee = amount * 0 / 10_000 = 0`. The only remediation path is a separate admin call to `setTokenFeeBps` (lines 583–594), which is not enforced or coupled to `addSupportedToken`.

The public `deposit(address token, uint256 amount, string referralId)` function (lines 284–305) is callable by any user immediately after `addSupportedToken` is called, with no guard preventing zero-fee deposits.

## Impact Explanation

**High — Theft of unclaimed yield.** Protocol fee revenue from token deposits is the yield being lost. Every token deposit made before `setTokenFeeBps` is called results in `feeEarnedInToken[token]` remaining at zero, meaning the protocol collects nothing. The value that should accrue as protocol fees instead remains entirely with the depositor. This is a concrete, quantifiable loss of protocol revenue triggered by an unprivileged external caller.

## Likelihood Explanation

Likelihood is **High**. `addSupportedToken` is the sole entry point for enabling token deposits, and it structurally guarantees `tokenFeeBps[token] == 0` at the moment of activation. There is no on-chain enforcement requiring `setTokenFeeBps` to be called before deposits are accepted. Any user monitoring the chain for `AddSupportedToken` events can immediately call `deposit(token, amount, referralId)` to benefit from zero fees. The window persists until the admin acts, and the admin call is not atomic with token addition (it requires a separate `DEFAULT_ADMIN_ROLE` transaction, distinct from the `TIMELOCK_ROLE` that calls `addSupportedToken`).

## Recommendation

Initialize `tokenFeeBps[token]` inside `addSupportedToken`, either to the global `feeBps` as a default or to a caller-supplied value:

```solidity
function addSupportedToken(
    address token,
    address oracle,
    address bridge,
    uint256 _feeBps
) external onlyRole(TIMELOCK_ROLE) {
    // ... existing checks ...
    if (_feeBps > 10_000) revert InvalidFeeAmount();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    tokenBridge[token] = bridge;
    tokenFeeBps[token] = _feeBps;
    emit AddSupportedToken(token, oracle, bridge);
}
```

Alternatively, fall back to `feeBps` when `tokenFeeBps[token]` is zero in `viewSwapRsETHAmountAndFee`.

## Proof of Concept

1. Admin calls `addSupportedToken(wstETH, oracle, bridge)` — `tokenFeeBps[wstETH]` remains `0` (confirmed: lines 637–656 make no assignment to `tokenFeeBps`).
2. User immediately calls `deposit(wstETH, 10 ether, "ref")` (lines 284–305).
3. `viewSwapRsETHAmountAndFee(10 ether, wstETH)` executes (lines 335–336):
   - `feeBpsForToken = tokenFeeBps[wstETH]` → `0`
   - `fee = 10 ether * 0 / 10_000` → `0`
   - `amountAfterFee = 10 ether`
4. User receives rsETH equivalent to the full `10 ether` of wstETH; `feeEarnedInToken[wstETH]` remains `0`.
5. An ETH depositor of equivalent value pays `feeBps` (e.g., 10 bps = 0.1%), losing `0.01 ether` to fees (line 312), demonstrating the asymmetry.

**Foundry fork test plan:**
- Fork Arbitrum mainnet.
- Call `addSupportedToken` from the `TIMELOCK_ROLE` address.
- Assert `tokenFeeBps[token] == 0`.
- Call `deposit(token, 1e18, "")` from an unprivileged address.
- Assert `feeEarnedInToken[token] == 0` and user received full rsETH equivalent.
