### Title
`psVRFKeyHashes` Secondary Index Incorrectly Cleared on Pool Re-registration, Enabling VRF Key Uniqueness Bypass — (`eras/shelley/impl/src/Cardano/Ledger/Shelley/Rules/PoolReap.hs`)

---

### Summary

`PState` maintains two coupled data structures: the primary pool map `psStakePools` (each entry holds a `spsVrf` field) and a secondary reference-count index `psVRFKeyHashes`. The `POOLREAP` rule removes "dangling" old VRF keys using `Map.withoutKeys`, which deletes the key entirely regardless of its reference count. When multiple pools share a VRF key — permitted before protocol version 11 — this incorrectly zeroes out the index entry while other pools still hold that VRF key. A subsequent pool registration certificate can then pass the `VRFKeyHashAlreadyRegistered` guard for a VRF key that is still actively in use, bypassing the uniqueness invariant introduced at v11.

---

### Finding Description

`PState` holds two structures that must stay in sync:

```
psStakePools          :: Map (KeyHash StakePool) StakePoolState
psVRFKeyHashes        :: Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
``` [1](#0-0) 

`psVRFKeyHashes` is a reference-counted secondary index. It is the sole data structure consulted by the `VRFKeyHashAlreadyRegistered` guard in `poolTransition`: [2](#0-1) 

At the v11 hard fork, `populateVRFKeyHashes` correctly seeds the map by counting every VRF occurrence across both `psStakePools` and `psFutureStakePoolParams`, so a VRF shared by three pre-v11 pools gets count `3`: [3](#0-2) 

In `poolReapTransition`, when a pool re-registers with a new VRF key, the old VRF key is classified as a "dangling" hash and removed from the index. The removal uses `Map.withoutKeys`, which **deletes the key entirely** regardless of its count: [4](#0-3) [5](#0-4) 

By contrast, the code path for **retiring** pools uses `removeVRFKeyHashOccurrences`, which correctly decrements the count and only removes the key when it reaches zero: [6](#0-5) 

The asymmetry is the root cause: re-registration uses unconditional deletion; retirement uses reference-counted decrement.

---

### Impact Explanation

**Medium — attacker-controlled pool registration certificate exceeds the intended VRF-uniqueness validation limit.**

After the inconsistency is introduced, `Map.notMember sppVrf psVRFKeyHashes` returns `True` for a VRF key that is still actively used by another pool. The `POOL` rule therefore accepts a `RegPool` certificate that should be rejected, registering a new pool with a duplicate VRF key. This violates the protocol invariant introduced at protocol version 11 and allows a single VRF key to be associated with multiple registered pools in the ledger state.

---

### Likelihood Explanation

The precondition is the existence of two or more pools registered with the same VRF key before protocol version 11 (when duplicate VRF keys were permitted). The v11 hard fork test suite explicitly exercises this scenario: [7](#0-6) 

Any pool operator who controls one of those pre-v11 duplicate-VRF pools can trigger the inconsistency by submitting a single re-registration certificate after v11, then registering a fresh pool with the freed VRF key. No privileged access, governance majority, or key compromise is required.

---

### Recommendation

Replace the unconditional `Map.withoutKeys danglingVRFKeyHashes` with the same reference-counted removal already used for retiring pools:

```haskell
-- current (incorrect for shared VRFs):
. (`Map.withoutKeys` danglingVRFKeyHashes)

-- corrected:
. removeVRFKeyHashOccurrences (Set.toList danglingVRFKeyHashes)
```

This ensures that when Pool A re-registers with a new VRF, the count for the old VRF is decremented by one rather than zeroed, preserving the invariant that `psVRFKeyHashes` accurately reflects the number of active pools using each VRF key.

---

### Proof of Concept

1. **Before v11**: Register Pool A (`kh_A`) and Pool B (`kh_B`) both with `vrf_X`. `psVRFKeyHashes` is empty (feature not yet active).
2. **v11 hard fork**: `populateVRFKeyHashes` runs → `psVRFKeyHashes[vrf_X] = 2`.
3. **Post-v11, same epoch**: Pool A re-registers with fresh `vrf_Y`.
   - `poolTransition` inserts `psVRFKeyHashes[vrf_Y] = 1`; `psFutureStakePoolParams[kh_A] = vrf_Y`.
4. **Epoch boundary (POOLREAP)**:
   - `danglingVRFKeyHashes = {vrf_X}` (Pool A's current VRF ≠ future VRF).
   - `Map.withoutKeys {vrf_X}` removes `vrf_X` entirely from `psVRFKeyHashes`.
   - Pool B still has `psStakePools[kh_B].spsVrf = vrf_X` — the index is now inconsistent.
5. **Attacker submits `RegPool` for new Pool C with `vrf_X`**:
   - Guard: `Map.notMember vrf_X psVRFKeyHashes` → `True` (key was deleted in step 4).
   - `VRFKeyHashAlreadyRegistered` is **not** raised; Pool C is accepted.
6. **Result**: Pool B and Pool C both hold `vrf_X` in the post-v11 ledger state, violating the uniqueness invariant. [8](#0-7) [4](#0-3) [5](#0-4)

populateVRFKeyHashes :: PState era -> PState era
populateVRFKeyHashes pState =
  pState
    & psVRFKeyHashesL
      %~ accumulateVRFKeyHashes (pState ^. psStakePoolsL) (^. spsVrfL)
        . accumulateVRFKeyHashes (pState ^. psFutureStakePoolParamsL) (^. sppVrfL)
  where
    accumulateVRFKeyHashes ::
      Map (KeyHash StakePool) a ->
      (a -> VRFVerKeyHash StakePoolVRF) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64) ->
      Map (VRFVerKeyHash StakePoolVRF) (NonZero Word64)
    accumulateVRFKeyHashes spMap getVrf acc =
      Map.foldr' (addVRFKeyHashOccurrence . getVrf) acc spMap
    addVRFKeyHashOccurrence vrfKeyHash =
      Map.insertWith combine vrfKeyHash (knownNonZeroBounded @1)
      where
        -- Saturates at maxBound: if (+1) would overflow to 0, keep existing value
        combine _ oldVal = fromMaybe oldVal $ mapNonZero (+ 1) oldVal
