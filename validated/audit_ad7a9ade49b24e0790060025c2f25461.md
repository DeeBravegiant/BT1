Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero Mid-Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
In `KernelDepositPool`, the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked == 0`, while `rewardPerTokenStored` remains frozen. Any time elapsed with zero stakers is permanently consumed without distributing rewards, leaving those tokens irrecoverably locked in the contract. The contract's own NatSpec acknowledges this and relies solely on off-chain operational guarantees with no on-chain enforcement.

## Finding Description
The `updateReward` modifier (L232-242) always executes two assignments:
```solidity
rewardPerTokenStored = rewardPerToken();   // frozen when totalKernelStaked == 0
updatedAt = lastTimeRewardApplicable();    // always advances
```
`rewardPerToken()` (L408-414) short-circuits and returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`, but `updatedAt` is still set to the current applicable timestamp. The time gap is permanently consumed.

The exploit path:
1. Alice is the sole staker during an active reward period.
2. Alice calls `initiateWithdrawal` — `updateReward(Alice)` runs first, correctly snapshotting rewards up to that moment, then `totalKernelStaked` is decremented to 0 (L325-326).
3. Time passes. No staker exists. `updatedAt` is frozen at the withdrawal timestamp.
4. Bob stakes. `updateReward(Bob)` runs: `totalKernelStaked > 0` now, but `rewardPerToken()` computes `rewardPerTokenStored + rewardRate * (now - updatedAt) * DECIMAL_PRECISION / totalKernelStaked`. Wait — actually at this point `updatedAt` was last set when Alice withdrew, so the gap IS included in the calculation for Bob's entry. Let me re-examine.

Re-examining precisely: when Bob stakes, `updateReward(Bob)` runs. At that moment `totalKernelStaked` is still 0 (Bob's stake hasn't been added yet — `stake()` calls `updateReward` as a modifier, then adds to `totalKernelStaked` in the function body). So `rewardPerToken()` returns `rewardPerTokenStored` unchanged, and `updatedAt` advances to `now`. The entire empty window's rewards are discarded. Bob's `userRewardPerTokenPaid` is set to the unchanged `rewardPerTokenStored`. The rewards for the zero-supply interval (`rewardRate × gap_seconds`) are never attributed to anyone and remain locked in the contract with no sweep or recovery function.

The only guard, `if (totalKernelStaked == 0) revert NoStakedTokens()` in `notifyRewardAmount` (L569-570), only prevents starting a new period with zero stakers. It does not prevent all stakers from exiting mid-period, and there is no on-chain mechanism that does.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens deposited by the admin via `notifyRewardAmount` and intended for stakers are permanently locked in the contract for any interval where `totalKernelStaked == 0`. No recovery mechanism exists. The magnitude is `rewardRate × zero_supply_duration`, which can be substantial. This matches the allowed impact: *"Medium. Permanent freezing of unclaimed yield."*

## Likelihood Explanation
Any staker can call `initiateWithdrawal` at any time — no privileged access required. A coordinated exit (e.g., due to a better yield opportunity, market downturn, or panic) can reduce `totalKernelStaked` to zero. The contract's own NatSpec (L18-22) explicitly acknowledges this scenario is possible and unmitigated on-chain, confirming the developers are aware it can happen. No external oracle, governance, or admin action is needed to trigger it.

## Recommendation
Freeze `updatedAt` when `totalKernelStaked == 0` so the empty time window is not consumed. In `rewardPerToken()`, return early without updating the stored timestamp reference, and adjust the modifier to only advance `updatedAt` when `totalKernelStaked > 0`:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    ...
}
```
This preserves full user freedom to withdraw while ensuring the reward clock only ticks when there are stakers to receive rewards.

## Proof of Concept
```
Setup:
  duration = 10,000s, rewardAmount = 100e18, rewardRate = 1e16/s
  Alice stakes 1e18 KERNEL. Admin calls notifyRewardAmount(100e18).
  → finishAt = T+10000, updatedAt = T

At T+5000:
  Alice calls initiateWithdrawal(1e18)
  → updateReward(Alice): totalKernelStaked=1e18 > 0
    rewardPerTokenStored += 1e16 * 5000 * 1e18 / 1e18 = 5e19
    updatedAt = T+5000
  → totalKernelStaked = 0

At T+7500 (empty window, 2500s elapsed):
  Bob calls stake(1e18)
  → updateReward(Bob) runs as modifier BEFORE totalKernelStaked is incremented
    totalKernelStaked == 0 → rewardPerToken() returns 5e19 (unchanged)
    updatedAt = T+7500  ← 2500s of rewards permanently discarded
    Bob's userRewardPerTokenPaid = 5e19

At T+10000:
  Bob calls getReward():
  → earned(Bob) = 1e18 * (rewardPerToken() - 5e19) / 1e18
    rewardPerToken() = 5e19 + 1e16 * 2500 * 1e18 / 1e18 = 7.5e19
    earned(Bob) = 7.5e19 - 5e19 = 2.5e19 (last 2500s only)

  Alice calls getReward():
  → rewards[Alice] = 5e19 (first 5000s, correctly snapshotted)

Result:
  Alice: 50e18 tokens ✓
  Bob:   25e18 tokens ✓
  Locked: 25e18 tokens (the T+5000 to T+7500 window) — permanently unclaimable
```
A Foundry test can reproduce this by warping timestamps and verifying the contract's reward token balance exceeds the sum of all claimable rewards after the period ends.
