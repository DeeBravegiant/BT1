### Title

---

### Summary

The `SUBUTXOW` transition rule for Dijkstra-era sub-transactions omits the `babbageMissingScripts` (or equivalent `validateMissingScripts`) check that is present in every other era's witness rule. Because sub-transactions carry their own `mint` field and the needed minting-policy script hashes are correctly computed but never verified to be present, an unprivileged transaction author can mint or burn native tokens under any native-script minting policy inside a sub-transaction without supplying or satisfying the policy script.

---

### Finding Description

**Vulnerability class:** Missing access control / witness validation bypass for minting/burning in sub-transactions.

**Affected era:** Dijkstra (the current development era introducing nested/sub-transactions).

**Root cause â€” three cooperating facts:**

1. **Sub-transactions have a `mint` field.** `DijkstraSubTxBodyRaw` contains `dstbrMint :: !MultiAsset`, exposed via `mintDijkstraTxBodyRawL` which handles both `DijkstraTxBodyRaw` and `DijkstraSubTxBodyRaw` uniformly. [1](#0-0) 

2. **`getDijkstraScriptsNeeded` correctly includes minting-policy script hashes for sub-transactions.** It delegates to `getConwayScriptsNeeded`, which calls `getMintingScriptsNeeded`, which reads `txBody ^. mintedTxBodyF` â€” a getter that works for both `TopTx` and `SubTx` bodies. [2](#0-1) 

3. **`dijkstraSubUtxowTransition` never checks that all needed scripts are provided.** The Babbage/Conway/Alonzo top-level `UTXOW` rules all contain a call to `babbageMissingScripts` (or `validateMissingScripts`) that enforces `scriptHashesNeeded âŠ† dom(scriptsProvided) âˆª dom(refScripts)`. The sub-transaction rule contains no such check: [3](#0-2) 

The only script-related checks present are:
- `validateFailedBabbageScripts` â€” validates native scripts that **are** provided; silently passes when none are provided.
- `hasExactSetOfRedeemers` â€” covers Plutus scripts only (native scripts have no redeemers).

For comparison, the Babbage top-level `UTXOW` rule explicitly enforces the missing-scripts check: [4](#0-3) 

The `dijkstraSubUtxowTransition` also contains a telling comment in the error-mapping function: `MissingScriptWitnessesUTXOW` and `ExtraneousScriptWitnessesUTXOW` are mapped to `error "Impossible: ..."`, confirming the authors know these predicates are not enforced for sub-transactions. [5](#0-4)

### Impact Explanation

**Critical â€” Direct creation or destruction of native assets through an invalid ledger state transition.**

An attacker who can author a Dijkstra-era top-level transaction (any unprivileged user) can embed a sub-transaction whose `mint` field contains:

- **Positive quantities** of tokens under any native-script minting policy â†’ tokens are created without the policy being satisfied (e.g., without the required key signatures, or outside the allowed time window).
- **Negative quantities** of tokens under a policy that prohibits burning â†’ tokens are destroyed in violation of the policy.

The value-conservation check in `dijkstraSubUtxoTransition` is satisfied because the `mint` field is included in the consumed-value calculation (`burnedMultiAssets` / `getConsumedMaryValue`), so the ledger accepts the transaction as balanced. [6](#0-5) 

The result is an invalid ledger state: native-asset supply is altered without policy authorization, which is the exact invariant the minting-policy mechanism is designed to enforce.

---

### Likelihood Explanation

**High.** The attacker entry path requires only the ability to submit a valid Dijkstra-era top-level transaction â€” no privileged role, no key compromise, no governance majority. The Dijkstra era is the current development era; once deployed, any user can exploit this. The `mint` field is optional in sub-transactions (CDDL field `? 9 : mint`), so a crafted sub-transaction with a non-empty mint field and no minting-policy witness is syntactically valid and will pass all current checks. [7](#0-6) 

---

### Recommendation

Add the missing-scripts check to `dijkstraSubUtxowTransition`, analogous to the Babbage rule:

```haskell
-- In dijkstraSubUtxowTransition, after computing scriptHashesNeeded:
let sReceived = Map.keysSet $ tx ^. witsTxL . scriptTxWitsL
    sRefs     = Map.keysSet $ getReferenceScripts originalUtxo inputs
runTest $ babbageMissingScripts pp scriptHashesNeeded sRefs sReceived
```

This ensures that every minting-policy script hash required by a sub-transaction's `mint` field is either provided in the witness set or resolvable via a reference script, and that native scripts among them are validated against the sub-transaction body.

### Proof of Concept

1. Deploy a native-script minting policy `P` requiring a specific key signature (e.g., `RequireSignature keyHash`). The policy ID is `hash(P)`.
2. Construct a Dijkstra top-level transaction containing a sub-transaction `subTx` with:
   - `dstbMint = MultiAsset { PolicyID (hash P) â†’ { "token" â†’ 1_000_000 } }` (positive: minting)
   - An output sending those 1,000,000 tokens to the attacker's address.
   - **No** minting-policy script in the witness set of either the top-level tx or `subTx`.
3. Submit the transaction.
4. `dijkstraSubUtxowTransition` runs:
   - `scriptsNeeded` contains `(MintingPurpose â€¦, hash P)` â€” correctly computed.
   - `validateFailedBabbageScripts`: `scriptsProvided` does not contain `hash P`, so the filter produces an empty set â†’ **passes**.
   - `hasExactSetOfRedeemers`: `P` is a native script, no redeemer needed â†’ **passes**.
   - No `babbageMissingScripts` call â†’ **no failure**.
5. `dijkstraSubUtxoTransition` value-conservation check: `consumed = inputs + mint(+1M tokens)`, `produced = outputs(+1M tokens)` â†’ **balanced, passes**.
6. The transaction is accepted; 1,000,000 tokens under policy `P` are created without the required key signature. [3](#0-2) [2](#0-1)
