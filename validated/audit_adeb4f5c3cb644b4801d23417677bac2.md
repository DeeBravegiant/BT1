### Title
Solana `deploy_token` Bypasses Bridge Pause — Missing Pause Check Allows Wrapped Token Deployment When Bridge Is Fully Paused (`solana/programs/bridge_token_factory/src/lib.rs`)

---

### Summary

The Solana `bridge_token_factory` program enforces pause flags on every user-facing transfer instruction but omits any pause check on `deploy_token`. Furthermore, the `ALL_PAUSED` bitmask is defined as only `INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED` — there is no `DEPLOY_TOKEN_PAUSED` constant at all. An unprivileged caller who holds a valid MPC-signed `DeployTokenPayload` (obtained before the pause) can deploy new wrapped tokens on Solana even while the bridge is in a fully-paused emergency state, directly analogous to the external report's "spreads can be minted with a deactivated oracle."

---

### Finding Description

**Bug class: state machine** — a pause/deactivation state is enforced on some code paths but silently absent on another.

In `solana/programs/bridge_token_factory/src/lib.rs`, every user-facing instruction that moves value checks the pause bitmask before proceeding:

- `init_transfer` (line 125–128): `require!(config.paused & INIT_TRANSFER_PAUSED == 0, Paused)`
- `init_transfer_sol` (line 140–143): same check
- `finalize_transfer` (line 82–85): `require!(config.paused & FINALIZE_TRANSFER_PAUSED == 0, Paused)`
- `finalize_transfer_sol` (line 101–104): same check

`deploy_token` (lines 66–76) has **no pause check whatsoever**:

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
```

The constants file confirms there is no `DEPLOY_TOKEN_PAUSED` bit and `ALL_PAUSED` covers only the two transfer flags:

```rust
pub const INIT_TRANSFER_PAUSED: u8 = 1 << 0;
pub const FINALIZE_TRANSFER_PAUSED: u8 = 1 << 1;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED;
```

This contrasts with the EVM implementation, which defines `PAUSED_DEPLOY_TOKEN = 1 << 2` and guards `deployToken` with `whenNotPaused(PAUSED_DEPLOY_TOKEN)`, and with the NEAR implementation, which decorates `deploy_token` with `#[pause(except(roles(Role::DAO)))]`. The Solana chain is the only one where the pause mechanism does not cover token deployment.

The `deploy_token` instruction is permissionless — any caller may submit it provided they supply a valid MPC-signed payload. MPC signatures for `DeployTokenPayload` are not time-limited or nonce-protected in the Solana path; a signature obtained before the pause remains valid indefinitely.

---

### Impact Explanation

When the bridge operator triggers `pause` (setting `ALL_PAUSED`) or `unpause` with a partial flag in response to a security incident, the intent is to halt all bridge activity on Solana. However, any party holding a previously issued MPC-signed `DeployTokenPayload` can still call `deploy_token` and register a new wrapped-token mint on Solana. This:

1. Creates new wrapped bridge assets on Solana during an emergency stop — unauthorized creation of wrapped bridge assets through deployment.
2. Undermines the operator's ability to contain an incident: if the vulnerability being patched relates to token deployment or token-mapping integrity, the pause provides no protection on Solana.
3. Produces a token-mapping entry (`near_to_solana_token`, mint account) that the NEAR side may not expect to exist, potentially causing asset-identity or balance-accounting divergence once the bridge is unpaused.

Impact category: **Critical** — unauthorized creation of wrapped bridge assets through deployment failure of the pause gate.

---

### Likelihood Explanation

- MPC-signed `DeployTokenPayload` messages are produced whenever a `LogMetadata` event is processed on NEAR. These signatures are broadcast publicly (on-chain events) and are not invalidated by a pause.
- Any observer of NEAR events can collect valid signatures before a pause is triggered and replay them on Solana after the pause.
- No special privilege is required; the attacker only needs to submit a transaction to the Solana program.
- The gap exists on every deployment of the current code.

Likelihood: **High**.

---

### Recommendation

1. Add a `DEPLOY_TOKEN_PAUSED: u8 = 1 << 2` constant to `constants.rs` and update `ALL_PAUSED` to include it.
2. Add a pause guard to `deploy_token` in `lib.rs`, mirroring the pattern used for the transfer instructions:

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    require!(
        ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
        error::ErrorCode::Paused
    );
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
```

This aligns Solana with the EVM (`PAUSED_DEPLOY_TOKEN` + `whenNotPaused`) and NEAR (`#[pause]` on `deploy_token`) implementations.

---

### Proof of Concept

1. NEAR emits a `LogMetadata` event for token `X`; the MPC signs a `DeployTokenPayload` for it. The signature is visible on-chain.
2. A security incident is discovered; the operator calls `pause` on the Solana program, setting `config.paused = ALL_PAUSED = 0x03`.
3. An attacker submits `deploy_token` with the previously collected signed payload. The instruction executes:
   - No pause check is present, so execution continues.
   - `data.verify_signature` passes (signature is valid).
   - `initialize_token_metadata` creates the wrapped mint for token `X` on Solana.
4. Token `X` is now deployed on Solana during the emergency pause, contrary to operator intent. The attacker has created a wrapped bridge asset that the operator cannot prevent.
