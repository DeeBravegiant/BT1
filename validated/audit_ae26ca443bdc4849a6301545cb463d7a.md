### Title
Permanent Loss of Funds via Beneficiary Account Deletion During In-Flight `DeleteAccount` Receipt - (File: `runtime/runtime/src/actions.rs`)

### Summary

When a `DeleteAccountAction` receipt is in-flight across shards, an attacker who controls the `beneficiary_id` account can delete it before the receipt executes. The `action_delete_account` function unconditionally creates a `new_balance_refund` receipt to `beneficiary_id` without verifying the beneficiary exists at execution time. When this refund receipt subsequently fails, the protocol's documented refund-failure policy permanently burns the funds. There is no recovery path.

### Finding Description

In `runtime/runtime/src/actions.rs`, `action_delete_account` deletes the account from state and emits a balance-transfer receipt to `beneficiary_id`:

```rust
// We use current amount as a pay out to beneficiary.
let account_balance = account_ref.amount();
if account_balance > Balance::ZERO {
    result
        .new_receipts
        .push(Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance));
}
let remove_result = remove_account(state_update, account_id)?;
...
*account = None;
``` [1](#0-0) 

`Receipt::new_balance_refund` sets `predecessor_id = "system"`, which the protocol uses to classify the receipt as a **refund receipt**: [2](#0-1) 

The NEAR protocol specification is explicit about what happens when a refund receipt fails:

> **"If the execution of a refund fails, the refund amount is burnt."** [3](#0-2) 

A transfer to a non-existent named account fails with `AccountDoesNotExist`. For refund receipts, `implicit_account_creation_eligible = false`, so even implicit accounts are not auto-created: [4](#0-3) 

There is **no validation** that `beneficiary_id` exists either at transaction submission time or at receipt execution time. The `DeleteAccountAction` validation only checks that `beneficiary_id` is a syntactically valid account ID, that the action is last in the list, and that the account is not staking: [5](#0-4) 

The account is irrevocably deleted from state before the refund receipt is dispatched. If the refund receipt fails, there is no path to recover the funds — the source account no longer exists, and the balance is permanently burnt.

### Impact Explanation

An attacker who controls the `beneficiary_id` account can cause the victim's entire account balance to be permanently burnt. The corrupted protocol value is the victim's `balance` field — it is deducted from the deleted account and then destroyed rather than transferred. This is a material, irreversible loss of funds with no on-chain recovery mechanism.

### Likelihood Explanation

The attack requires the attacker to:
1. Control the `beneficiary_id` account (a named account they own).
2. Observe the victim's `DeleteAccount` transaction in the mempool or chain.
3. Submit their own `DeleteAccount` for `beneficiary_id` before the victim's cross-shard receipt executes (a window of 1–2 blocks).

This is a realistic front-running attack. Additionally, the loss can occur non-maliciously: if the beneficiary independently deletes their account in the same block window, the victim's funds are still permanently burnt. The attack is also applicable to smart contracts that programmatically call `promise_batch_action_delete_account` with an attacker-controlled `beneficiary_id`.

### Recommendation

1. **Validate beneficiary existence at execution time**: In `action_delete_account`, check that `beneficiary_id` exists in the trie before proceeding. If it does not exist, abort the deletion with an error (preserving the account and its balance).
2. **Fallback recipient**: If the beneficiary does not exist when the `new_balance_refund` receipt executes, redirect the funds to the receipt's `predecessor_id` (the original sender) rather than burning them.
3. **Change refund failure policy for beneficiary payouts**: Distinguish between protocol-generated gas/deposit refunds (where burning on failure is acceptable) and `DeleteAccount` beneficiary payouts (where burning is catastrophic), and handle the latter with a non-destructive fallback.

### Proof of Concept

1. Alice (on shard 0) submits `DeleteAccount { beneficiary_id: "bob.near" }` where `bob.near` is on shard 1.
2. The transaction is included in block B. Alice's account balance is deducted and the receipt is dispatched cross-shard.
3. Bob (attacker, who controls `bob.near`) observes Alice's transaction in the mempool and submits `DeleteAccount { beneficiary_id: "anyone.near" }` for his own account.
4. Bob's transaction is included in block B or B+1, deleting `bob.near` from state.
5. In block B+1, Alice's `DeleteAccount` receipt executes on shard 1 via `action_delete_account`: Alice's account is removed from state and a `new_balance_refund` receipt is created targeting `bob.near` with Alice's full balance.
6. In block B+2, the `new_balance_refund` receipt executes: the `Transfer` action to `bob.near` fails with `AccountDoesNotExist` because `bob.near` no longer exists and `implicit_account_creation_eligible = false` for refund receipts.
7. Per `docs/RuntimeSpec/Refunds.md:12`, the refund amount is **permanently burnt**. Alice's funds are irrecoverably lost.

/// Invalid account ID.

- If this action is not the last action in the action list of a receipt, the following error will be returned

```rust
/// The delete action must be a final action in transaction
DeleteActionMustBeFinal

- If the account still has locked balance due to staking, the following error will be returned

```rust
/// Account is staking and can not be deleted
DeleteAccountStaking { account_id: AccountId }
