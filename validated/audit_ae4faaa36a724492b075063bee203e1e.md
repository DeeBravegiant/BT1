Audit Report

## Title
Unprivileged Caller Can Force Claim on Behalf of Any Account, Extracting Protocol Fee from Victim's Unclaimed Yield — (File: `contracts/utils/MerkleDistributor/MerkleDistributor.sol`)

## Summary
`MerkleDistributor.claim()` contains no `msg.sender == account` guard, allowing any caller to trigger a claim for an arbitrary beneficiary using publicly available merkle proof data. Because the fee deduction is unconditional, the attacker can force the victim's claim at the current `feeInBPS` (up to 10%), permanently reducing the victim's yield without consent and with no recourse.

## Finding Description
The `claim` function at L97–106 accepts `account` as a caller-supplied parameter and performs no check that `msg.sender == account`:

```solidity
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
```

The only guards present are `whenNotPaused`, a non-zero merkle root check, an index range check, an `isClaimed` replay check, and merkle proof validity. None of these prevent a third party from supplying a valid proof for a victim's leaf and triggering the claim on their behalf.

After proof verification, the fee is computed and deducted unconditionally at L138–144:

```solidity
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;
IERC20(token).safeTransfer(account, amountToSend);
IERC20(token).safeTransfer(protocolTreasury, fee);
```

`feeInBPS` is mutable via `setFeeInBPS` (L198–206), so a victim may deliberately defer their claim waiting for the owner to lower the fee. An attacker can front-run or independently call `claim` at the current maximum fee (`MAX_FEE_IN_BPS = 1000`, i.e. 10%), permanently locking in the higher deduction before any reduction takes effect. The state update at L134–135 marks the claim as consumed, so the victim cannot re-claim.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: **Theft of unclaimed yield**. The victim permanently loses up to 10% of their claimable yield — transferred irrecoverably to `protocolTreasury` — without ever consenting to or initiating the claim. The loss is concrete, irreversible, and directly caused by the missing caller restriction in this repository.

## Likelihood Explanation
Likelihood is **High**:
- Merkle tree data (index, account, cumulativeAmount, proof) is published off-chain so any user can construct their own proof; any observer can reconstruct the same data for any other leaf.
- No special role, key, or privileged access is required.
- The attack is a single permissionless transaction callable by any EOA or contract.
- It is repeatable across every unclaimed leaf in the tree whenever `feeInBPS > 0`.

## Recommendation
Add a caller restriction to `claim` so only the beneficiary can trigger their own claim:

```solidity
function claim(
    uint256 index,
    address account,
    uint256 cumulativeAmount,
    bytes32[] calldata merkleProof
) external override whenNotPaused {
    require(msg.sender == account, "MerkleDistributor: caller is not the account");
    // ...
}
```

This preserves the beneficiary's right to time their claim relative to the current `feeInBPS`.

## Proof of Concept
```solidity
// Preconditions:
// - feeInBPS = 1000 (10%)
// - victim has a valid unclaimed leaf: (victimIndex, victimAddress, victimCumulativeAmount)
// - merkle proof is obtained from the published off-chain tree

// Attacker (any EOA) executes one transaction:
merkleDistributor.claim(
    victimIndex,
    victimAddress,
    victimCumulativeAmount,
    victimMerkleProof
);

// Post-conditions:
// victim receives: victimCumulativeAmount * 9000 / 10_000  (90%)
// protocolTreasury receives: victimCumulativeAmount * 1000 / 10_000  (10%)
// victim's claim is marked consumed; they cannot re-claim
assertEq(
    token.balanceOf(victimAddress),
    victimCumulativeAmount * 9000 / 10_000
);
// Victim lost 10% of yield without consent; attacker spent only gas.
```

A Foundry fork test can confirm this by: (1) deploying `MerkleDistributor` with `feeInBPS = 1000`, (2) setting a merkle root containing a victim leaf, (3) calling `claim` from a separate attacker address with the victim's valid proof, and (4) asserting the victim's balance equals 90% of `cumulativeAmount` and `isClaimed` returns `true`.
