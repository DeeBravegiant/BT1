### Title
DelegateAction `max_block_height` Expiry Off-by-One Allows Execution One Block Beyond Intended Expiry - (File: runtime/runtime/src/actions.rs)

### Summary
The `DelegateAction` expiry check in nearcore's runtime uses a strict `>` comparison instead of `>=`, allowing a relayer to execute a meta-transaction at exactly `block_height == max_block_height`, one block beyond the user's intended expiry boundary.

### Finding Description
The `DelegateAction` struct carries a `max_block_height` field documented as "The maximal height of the block in the blockchain **below which** the given DelegateAction is valid." The protocol specification and error documentation both state that the action should be rejected when `current_block_height >= max_block_height`:

> "If the current block is **equal or greater than** `max_block_height`" → `DelegateActionExpired` [1](#0-0) [2](#0-1) 

However, the runtime test suite reveals the actual comparison is strict `>` (not `>=`). The test `test_delegate_action_max_height` only triggers `DelegateActionExpired` when `block_height = max_block_height + 1`, while `test_delegate_action_validate_sender_account` confirms that at `block_height == max_block_height` the action is **not** expired — it proceeds past the expiry check and fails on a different condition: [3](#0-2) [4](#0-3) 

This is the same off-by-one class as the ERC721 report: a boundary check uses `>` where `>=` is required, allowing one extra unit of access beyond the intended limit.

### Impact Explanation
A relayer can execute a user-signed `DelegateAction` at exactly `block_height == max_block_height`, one block after the user intended the action to expire. The corrupted protocol value is the **validity decision** for the DelegateAction receipt: the runtime accepts and executes an action that should be rejected as `DelegateActionExpired`. Any state changes produced by the inner actions (token transfers, key additions, contract calls, etc.) are committed to the state trie at a block height the user explicitly excluded.

**Impact: Low** — the window is exactly one block, but the user's signed intent is violated.

### Likelihood Explanation
Any unprivileged relayer can exploit this. The relayer controls when to submit the outer transaction containing the `DelegateAction`. By timing submission so the receipt is processed at exactly `block_height == max_block_height`, the relayer executes the action one block beyond the user's intended deadline. No special privileges are required; this is reachable via normal public RPC transaction submission.

**Likelihood: High** — the relayer has full control over submission timing and can trivially target the exact boundary block.

### Recommendation
Change the expiry comparison in `apply_delegate_action` (in `runtime/runtime/src/actions.rs`) from strict greater-than to greater-than-or-equal:

```diff
- if apply_state.block_height > delegate_action.max_block_height() {
+ if apply_state.block_height >= delegate_action.max_block_height() {
      result.result = Err(ActionErrorKind::DelegateActionExpired.into());
      return Ok(());
  }
```

This aligns the code with the documented invariant and the field's semantic ("below which the action is valid").

### Proof of Concept
1. User signs a `DelegateAction` with `max_block_height = N` (e.g., N = 1000), intending the action to be invalid at block 1000 and beyond.
2. Relayer holds the signed action and waits until the chain reaches block height N.
3. Relayer submits the outer transaction at block N. The receipt is processed at `block_height == max_block_height`.
4. The runtime's expiry check (`block_height > max_block_height` → `1000 > 1000` → `false`) does **not** fire.
5. The inner actions execute and their effects (e.g., a token transfer) are committed to state, despite the user's explicit `max_block_height = N` boundary. [5](#0-4) [6](#0-5)

pub struct DelegateAction {
    /// Signer of the delegated actions
    pub sender_id: AccountId,
    /// Receiver of the delegated actions.
    pub receiver_id: AccountId,
    /// List of actions to be executed.
    ///
    /// With the meta transactions MVP defined in NEP-366, nested
    /// DelegateActions are not allowed. A separate type is used to enforce it.
    pub actions: Vec<NonDelegateAction>,
    /// Nonce to ensure that the same delegate action is not sent twice by a
    /// relayer and should match for given account's `public_key`.
    /// After this action is processed it will increment.
    pub nonce: Nonce,
    /// The maximal height of the block in the blockchain below which the given DelegateAction is valid.
    pub max_block_height: BlockHeight,
    /// Public key used to sign this delegated action.
    pub public_key: PublicKey,
}
