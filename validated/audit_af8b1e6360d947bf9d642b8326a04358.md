Audit Report

## Title
ETH Withdrawal Permanently Frozen When Withdrawing User Address Cannot Receive ETH - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager._transferAsset` uses a push-payment pattern that hard-reverts with `EthTransferFailed` when the recipient cannot accept ETH. Because the entire transaction reverts, the withdrawal request remains permanently in the unlocked queue with no cancellation path, no alternative recipient mechanism, and no way to sweep the stuck ETH — permanently freezing both the user's ETH and their already-burned rsETH.

## Finding Description
The root cause is the push-payment pattern in `_transferAsset` (L876-883): a low-level `.call{value:}` is made to the user address, and if it returns `false`, the function reverts with `EthTransferFailed`. This revert propagates up through `_processWithdrawalCompletion` (L699-738), rolling back all state mutations — `popFront`, `delete withdrawalRequests[requestId]`, and `unlockedWithdrawalsCount[asset]--` — leaving the request permanently stuck in the unlocked queue.

The critical precondition is established in `unlockQueue` (L305-307): rsETH is burned via `burnFrom` and ETH is redeemed from the unstaking vault into `LRTWithdrawalManager` before any individual withdrawal is completed. Once this happens, the ETH sits in the contract with no recovery path:

- `completeWithdrawal` (L183-185) always reverts for a non-ETH-accepting address.
- `completeWithdrawalForUser` (L192-204) suffers the same revert via the identical `_processWithdrawalCompletion` call path.
- `sweepRemainingAssets` (L402-403) is permanently blocked because `hasUnlockedWithdrawals(asset)` remains `true` while the stuck request is counted.
- A grep search confirms no `cancelWithdrawal`, `rescueETH`, `redirectPayout`, or `claimable[]` mechanism exists anywhere in the contract.

The developer comment on `completeWithdrawalForUser` — "Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH" — acknowledges ETH-transfer edge cases but incorrectly dismisses the permanent-freeze scenario.

## Impact Explanation
**Critical — Permanent freezing of funds.** Once `unlockQueue` executes, the user's rsETH is irreversibly burned and the corresponding ETH is held in `LRTWithdrawalManager`. Any smart-contract withdrawer whose address cannot receive ETH loses both assets permanently with no admin recovery path available.

## Likelihood Explanation
**Medium.** Any smart-contract account holding rsETH is at risk: multisigs with non-standard `receive()` logic, DeFi protocols, proxy contracts whose implementation does not forward ETH, or contracts that deliberately revert in their fallback. The protocol explicitly supports contract callers — `completeWithdrawalForUser` exists for this purpose — yet the operator path is equally broken. No special attacker capability is required; a normal `initiateWithdrawal` call from a non-ETH-accepting contract is sufficient to trigger the freeze after `unlockQueue` runs.

## Recommendation
Replace the push-payment pattern with a pull-payment (withdrawal pattern):

1. In `_processWithdrawalCompletion`, instead of calling `_transferAsset(asset, user, amount)`, credit a `claimable[user][asset] += amount` mapping and emit an event.
2. Add a separate `claimAsset(address asset)` function that transfers `claimable[msg.sender][asset]` to `msg.sender` and zeroes the balance.

This decouples state finalization from ETH delivery, ensuring the withdrawal request is always removed from the queue regardless of whether the recipient can accept ETH at that moment.

Alternatively, add a `cancelWithdrawal` function allowing users to reclaim rsETH before unlocking, and an admin/operator function to redirect a stuck ETH payout to a user-specified rescue address after unlocking.

## Proof of Concept
1. Deploy `MaliciousReceiver` with `receive() external payable { revert(); }` holding rsETH.
2. Call `MaliciousReceiver.doWithdraw(amount)` → rsETH transferred to `LRTWithdrawalManager`, request queued.
3. Operator calls `unlockQueue(ETH_TOKEN, ...)` → rsETH burned (L305), ETH redeemed into `LRTWithdrawalManager` (L307).
4. Call `wm.completeWithdrawal(ETH_TOKEN, "")` from `MaliciousReceiver` → `_transferAsset` calls `payable(MaliciousReceiver).call{value:}("")` → returns `false` → reverts with `EthTransferFailed`; all state rolled back.
5. Operator calls `wm.completeWithdrawalForUser(ETH_TOKEN, address(MaliciousReceiver), "")` → same revert.
6. `wm.sweepRemainingAssets(ETH_TOKEN)` → reverts with `PendingWithdrawalsExist` because `unlockedWithdrawalsCount[ETH_TOKEN] > 0`.
7. ETH is permanently locked; rsETH is permanently burned.
