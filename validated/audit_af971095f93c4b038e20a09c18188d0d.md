### Title
Hardcoded `paid_fee_on_l1 = Fee(1)` in Consensus L1 Handler Conversion Produces Wrong Executable Payload - (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

`TransactionConverter::convert_consensus_l1_handler_to_internal_l1_handler` permanently hardcodes `paid_fee_on_l1 = Fee(1)` for every L1 handler transaction received through the consensus path. This is the direct Sequencer analog of the uninitialized `feeRecipient` bug: a critical economic field is never populated from its authoritative source and instead carries a permanent placeholder, causing every validator node to bind the wrong executable payload to every L1 handler transaction it processes.

### Finding Description

When the proposer streams an L1 handler transaction to the network, it converts `InternalConsensusTransaction::L1Handler(tx)` (which carries the real `paid_fee_on_l1` from the L1 monitoring component) to `ConsensusTransaction::L1Handler(tx.tx)`, which is `transaction::L1HandlerTransaction` — a type that has no `paid_fee_on_l1` field. The real fee is silently dropped. [1](#0-0) 

On the validator side, `convert_consensus_tx_to_internal_consensus_tx` calls `convert_consensus_l1_handler_to_internal_l1_handler`, which reconstructs the executable transaction with a hardcoded `Fee(1)`: [2](#0-1) 

The `TODO(Gilad)` comment confirms this is a known placeholder, not a deliberate design choice. The `paid_fee_on_l1` field on `executable_transaction::L1HandlerTransaction` is the authoritative value used downstream: [3](#0-2) 

### Impact Explanation

**1. Wrong executable payload bound to every L1 handler on every validator node.**

Every validator that receives a proposal containing L1 handler transactions will execute them with `paid_fee_on_l1 = Fee(1)` regardless of the actual fee paid on L1. This is the wrong executable payload for the transaction.

**2. Fee check bypass for zero-fee L1 handlers.**

The blockifier's fee check in `execute_raw` only rejects `paid_fee == Fee(0)`: [4](#0-3) 

With `Fee(1)` hardcoded, this check always passes on the validator side. If the real `paid_fee_on_l1` were `Fee(0)` (which the proposer would reject), the validator would accept and commit the execution — a proposer/validator divergence.

**3. Wrong `paid_fee_on_l1` in the centralized blob.**

`CentralL1HandlerTransaction` serializes `paid_fee_on_l1` directly from the executable transaction into the blob sent to the Aerospike/centralized pipeline: [5](#0-4) 

Every validator node's blob will carry `paid_fee_on_l1 = 1` for all L1 handler transactions, producing systematically wrong receipt data in the centralized pipeline.

### Likelihood Explanation

This is triggered by every L1 handler transaction that passes through the consensus path on any validator node. L1 handler transactions are a normal part of Starknet operation (L1→L2 messages). No special attacker action is required — the bug fires unconditionally for all such transactions on all non-proposer nodes.

### Recommendation

The `paid_fee_on_l1` must be carried through the consensus wire format. Options:
1. Add `paid_fee_on_l1` to `ConsensusTransaction::L1Handler` (or a wrapper) so the proposer can include it and the validator can read it.
2. Have the validator look up the real `paid_fee_on_l1` from its own L1 monitoring component using the transaction nonce/hash as a key.

Remove the `Fee(1)` placeholder and the `TODO(Gilad)` comment only after one of these approaches is implemented.

### Proof of Concept

1. Proposer receives an L1 handler transaction from the L1 monitor with `paid_fee_on_l1 = Fee(5_000_000_000)`.
2. Proposer calls `convert_internal_consensus_tx_to_consensus_tx` → `ConsensusTransaction::L1Handler(tx.tx)` — `paid_fee_on_l1` is dropped. [6](#0-5) 
3. Validator receives `ConsensusTransaction::L1Handler(tx)` and calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_consensus_l1_handler_to_internal_l1_handler` → `Fee(1)` is hardcoded. [7](#0-6) 
4. Validator executes the L1 handler with `paid_fee_on_l1 = Fee(1)`. The fee check passes (`Fee(1) != Fee(0)`).
5. Validator's blob is written with `paid_fee_on_l1 = 1` instead of `5_000_000_000`.
6. Centralized pipeline receives wrong `paid_fee_on_l1` for every L1 handler transaction processed by every validator node.
