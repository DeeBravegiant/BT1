Audit Report

## Title
Stale `rsETHPrice` Read in `depositETH` Allows Depositors to Dilute Existing Holders' Accrued Yield — (`contracts/LRTDepositPool.sol`)

## Summary

`depositETH` and `depositAsset` mint rsETH using the stored `rsETHPrice` state variable in `LRTOracle`, which is only refreshed when `updateRSETHPrice()` is explicitly called. Because staking yield continuously increases `totalETHInProtocol` between price updates while `rsETHPrice` remains fixed at the last snapshot, any depositor who deposits during this stale window receives more rsETH than the current TVL justifies. The surplus rsETH value is extracted directly from the unclaimed yield that existing holders had already earned.

## Finding Description

The mint calculation in `getRsETHAmountToMint` divides by the stored `rsETHPrice` state variable: [1](#0-0) 

`rsETHPrice` is a plain storage slot in `LRTOracle`: [2](#0-1) 

It is written only at the end of `_updateRsETHPrice()`: [3](#0-2) 

`updateRSETHPrice()` is public and permissionless: [4](#0-3) 

The deposit path (`depositETH` → `_beforeDeposit` → `getRsETHAmountToMint`) performs no freshness check and does not call `updateRSETHPrice()` before computing the mint amount: [5](#0-4) [6](#0-5) 

**Exploit flow:**

1. Yield accrues (e.g., staking rewards increase ETH balance in NDCs), so `totalETHInProtocol` grows while `rsETHPrice` remains at the prior snapshot.
2. Attacker calls `depositETH` before `updateRSETHPrice()` is called. The stale (lower) `rsETHPrice` denominator causes more rsETH to be minted than the fair backing ratio justifies.
3. After the deposit, `updateRSETHPrice()` is called (by anyone). The new price is computed using the post-deposit supply and TVL: [7](#0-6) 

   Because the attacker's inflated rsETH balance is now included in `rsethSupply`, the price update bakes in the dilution — the attacker's rsETH is worth more than they deposited, and the difference comes from existing holders' accrued yield.

**Concrete example (from the report):**
- Pre-deposit: 99 ETH TVL, 99 rsETH supply, stale `rsETHPrice = 1.000`, fair price ≈ 1.0101 (1 ETH yield accrued).
- Attacker deposits 99 ETH at stale price → minted 99 rsETH (fair: ≈98.01 rsETH).
- Post-update: `newRsETHPrice = 199/198 ≈ 1.00505`. Attacker's 99 rsETH ≈ 99.5 ETH — ~0.5 ETH profit extracted from existing holders' yield.

## Impact Explanation

**High — Theft of unclaimed yield.** Existing rsETH holders lose a portion of their accrued staking yield proportional to the attacker's deposit size relative to total TVL. The attacker's principal is not at risk; the attack extracts yield that existing holders had already earned but not yet realized. The attack does not threaten principal or cause insolvency, so Critical is not warranted. The impact maps exactly to the allowed scope: "Theft of unclaimed yield."

## Likelihood Explanation

- No special role, governance access, or oracle manipulation is required.
- `updateRSETHPrice()` is public; the attacker simply deposits in any block before it is called.
- Yield accrues continuously; the stale window is always non-zero between updates.
- The attack is repeatable every price-update cycle.
- Front-running the `updateRSETHPrice()` transaction is sufficient but not even necessary — depositing in any block during the stale window achieves the same result.

## Recommendation

1. **Atomic price refresh on deposit**: Call `_updateRsETHPrice()` (made `internal`) at the start of `_beforeDeposit`, so the mint rate always reflects the current TVL before any rsETH is minted.
2. **Alternatively**, compute the mint rate on-the-fly as `totalETHInProtocol / rsethSupply` directly in `getRsETHAmountToMint`, eliminating the stale-read window entirely without requiring a state-writing call.
3. **At minimum**, enforce a maximum staleness interval on-chain (e.g., store `lastPriceUpdateTimestamp` and revert deposits if `rsETHPrice` is older than N blocks/seconds).

## Proof of Concept

```solidity
// Foundry fork test sketch
function test_staleRsETHPriceDilutesYield() public {
    // 1. Simulate yield accrual without updating rsETHPrice
    //    (e.g., send ETH rewards to a NodeDelegator so totalETHInProtocol increases)
    vm.deal(address(nodeDelegator), address(nodeDelegator).balance + 1 ether);

    uint256 rsETHPriceBefore = oracle.rsETHPrice(); // stale, lower than fair

    // 2. Attacker deposits at stale price
    uint256 depositAmt = 99 ether;
    vm.deal(attacker, depositAmt);
    vm.prank(attacker);
    pool.depositETH{value: depositAmt}(0, "");

    uint256 attackerRsETH = rsETH.balanceOf(attacker);

    // 3. Price update now includes attacker's deposit in supply
    oracle.updateRSETHPrice();
    uint256 rsETHPriceAfter = oracle.rsETHPrice();

    // 4. Attacker's rsETH value exceeds their deposit — yield stolen
    uint256 attackerValueAfter = attackerRsETH * rsETHPriceAfter / 1e18;
    assertGt(attackerValueAfter, depositAmt, "Attacker profited from stale price");
}
```

The invariant `attackerValueAfter <= depositAmt` breaks whenever `rsETHPrice` is stale and yield has accrued, directly confirming the finding. The same attack applies to `depositAsset` via the identical `_beforeDeposit` → `getRsETHAmountToMint` path.
