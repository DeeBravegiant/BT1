### Title
No Presignature-Use Ledger Survives Disk Rollback, Enabling ECDSA Nonce Reuse and Full Key Recovery - (File: `crates/node/src/assets.rs`)

---

### Summary

The MPC node's only protection against presignature reuse is an atomic RocksDB delete inside `take_owned()`. There is no separate, append-only "used presignature" ledger that persists beyond the current disk state. A host-level attacker who replaces the node's encrypted disk with an older snapshot restores already-consumed presignatures to their pre-consumption state. When those presignatures are subsequently used in new signing sessions, the ECDSA nonce is effectively reused, enabling full private key recovery via standard nonce-reuse attacks. The codebase explicitly acknowledges this as an open, unmitigated risk (T6).

---

### Finding Description

**Root cause — no durable "used" ledger:**

`take_owned()` in `crates/node/src/assets.rs` atomically removes a presignature from RocksDB at the moment of consumption:

```rust
pub async fn take_owned(&self) -> (UniqueId, T) {
    let (id, asset) = self.owned_queue.take_owned().await;
    let mut update = self.db.update();
    update.delete(self.col, &self.make_key(id));
    update.commit().expect("Unrecoverable error writing to database");
    (id, asset)
}
``` [1](#0-0) 

This delete is the *only* record that the presignature was consumed. There is no separate append-only log of used presignature IDs. If the disk is rolled back to a snapshot taken before the delete, the presignature reappears in RocksDB as if it were fresh.

**Startup cleanup does not detect rollback:**

`delete_stale_triples_and_presignatures()` in `crates/node/src/assets/cleanup.rs` only acts on epoch-ID changes and participant TLS-key changes: [2](#0-1) 

It has no mechanism to detect that a presignature was already used in a prior run. After a rollback within the same epoch with the same participant set, the cleanup function takes the `AssetCleanup::Keep` path and leaves all restored presignatures intact.

**Cryptographic consequence — full key recovery:**

The Robust ECDSA security documentation states explicitly:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [3](#0-2) 

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." [4](#0-3) 

The OT-based ECDSA orchestration documentation similarly requires destruction of outputs after use:

> "It's critical that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase." [5](#0-4) 

**Acknowledged open risk:**

The threat model explicitly lists this as T6 with no mitigation:

> "T6 | Rollback / replay attack | Replace disk with old snapshot | Host OS -> Encrypted Disk | **Open risk** — triples/presigs could be reused | Future mitigation needed" [6](#0-5) 

The TEE design document confirms:

> "An entire MPC node disk may be replaced with a previous snapshot, leading to reuse of cryptographic assets (e.g., triples, presignatures). Currently, these assets are persisted on disk — mitigation will be required in future iterations." [7](#0-6) 

---

### Impact Explanation

A host-level attacker who can replace the node's disk image with an older snapshot causes the node to reuse presignatures that were already consumed in prior signing sessions. Two signing sessions that use the same presignature with different message hashes produce ECDSA signatures with multiplicatively related nonces. From these two signatures, the full threshold private key share can be algebraically recovered. With enough shares (≥ threshold), the complete MPC private key is recovered, enabling the attacker to forge arbitrary signatures for any foreign-chain transaction — constituting direct theft of all funds controlled by the MPC network.

**Severity: Critical** — matches "Bypass of threshold-signature requirements or unauthorized access to MPC key shares … that materially enables forgery or secret recovery."

---

### Likelihood Explanation

The TEE threat model explicitly places the host OS in the attacker's control: "A privileged attacker with host-level (root) access can interrupt execution, inspect memory (but can't decrypt the CVM memory), and control scheduling." The operator is explicitly "Not trusted to securely run the MPC node." Replacing a virtual disk image or network-attached storage volume with an older snapshot is a routine host-OS-level operation requiring no physical hardware access and no cryptographic break. The Gramine sealing key is derived from CVM measurements, so the same CVM binary can decrypt the restored snapshot transparently. The attacker then simply waits for the node to process new signature requests using the restored presignatures.

---

### Recommendation

1. Maintain a separate, monotonically growing append-only ledger of consumed presignature IDs (e.g., in a dedicated RocksDB column that is never rolled back). On startup, cross-reference all presignatures in the store against this ledger and delete any that appear as already used.
2. Alternatively, bind presignature consumption to an on-chain counter or block height that cannot be rolled back, so the node can detect that its local state is behind the canonical chain state.
3. As a defense-in-depth measure, implement a monotonic counter sealed inside the TEE (using a TEE-monotonic counter service if available) that increments on every presignature consumption and is checked on startup.

---

### Proof of Concept

1. Node A (leader) generates presignature `P` with ID `X` and stores it in RocksDB. Borrower nodes store their shares of `X`.
2. Host-level attacker takes a snapshot of Node A's encrypted disk at this point (snapshot `S`).
3. Node A processes a signature request: `take_owned()` is called, `X` is deleted from RocksDB, and the signing protocol runs with message hash `h1`, producing signature `σ1`.
4. Attacker replaces Node A's disk with snapshot `S`. Node A restarts; `delete_stale_triples_and_presignatures()` finds the same epoch ID and same participants — takes `AssetCleanup::Keep` — presignature `X` is back in RocksDB.
5. Attacker triggers a second signature request. Node A calls `take_owned()` again, retrieves `X`, and runs the signing protocol with message hash `h2 ≠ h1`, producing signature `σ2`.
6. From `(σ1, h1)` and `(σ2, h2)` — both derived from the same presignature nonce — the attacker applies standard ECDSA nonce-reuse algebra to recover the private key share held by Node A. Repeating across threshold-many nodes recovers the full MPC key.

pub fn delete_stale_triples_and_presignatures(
    db: &Arc<SecretDB>,
    current_epoch_data: EpochData,
    my_participant_id: primitives::ParticipantId,
    ecdsa_domain_ds: Vec<DomainId>,
    triple_thresholds: Vec<ReconstructionThreshold>,
) -> anyhow::Result<()> {
    let asset_cleanup: AssetCleanup = match get_epoch_data(db)? {
        None => AssetCleanup::Keep,
        Some(previous_epoch_data) => cleanup_behavior(&previous_epoch_data, &current_epoch_data),
    };

    // All cleanup work + the EpochData marker bump are staged on a single
    // `SecretDBUpdate` so they commit as one atomic RocksDB batch. If the
    // process dies before the commit, none of the deletes or the marker are
    // persisted, and the next startup re-runs the full cleanup unchanged.
    let mut update_writer = db.update();
    match asset_cleanup {
        AssetCleanup::Keep => {}
        AssetCleanup::DeleteAll => {
            update_writer.delete_all(DBCol::Presignature)?;
            update_writer.delete_all(DBCol::TripleV2)?;
        }
        AssetCleanup::KeepOnly(persitent_participants) => {
            // Triples — `TripleV2` uses a `[t as u64 BE]` prefix per store.
            for t in &triple_thresholds {
                clean_db::<PairedTriple>(
                    db,
                    &mut update_writer,
                    DBCol::TripleV2,
                    &persitent_participants,
                    my_participant_id,
                    &t.inner().to_be_bytes(),
                )?;
            }
            // Presignatures (per-domain, unchanged).
            for domain_id in &ecdsa_domain_ds {
                clean_db::<PresignOutputWithParticipants>(
                    db,
                    &mut update_writer,
                    DBCol::Presignature,
                    &persitent_participants,
                    my_participant_id,
                    &domain_id.0.to_be_bytes(),
                )?;
            }
        }
    }

    tracing::info!("Updating epoch data: {:?}.", current_epoch_data);
    let current_epoch_data_ser = serde_json::to_vec(&current_epoch_data)?;
    update_writer.put(DBCol::EpochData, EPOCH_ID_KEY, &current_epoch_data_ser);
    update_writer.commit()?;
    tracing::info!("Updated epoch id entry");
    Ok(())
```
