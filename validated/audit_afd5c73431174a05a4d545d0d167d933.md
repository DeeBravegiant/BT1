Audit Report

## Title
Pool Admin Can Permanently Lock Protocol Fees via Reverting `adminFeeDestination` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

## Summary
`setPoolAdminFeeDestination` accepts any non-zero address without validating that it can receive ERC-20 tokens. Because `collectFees` transfers to `adminFeeDestination_` before `FACTORY`, a reverting destination permanently blocks all fee collection paths — including the permissionless `collectPoolFees` and the owner-only `setPoolProtocolFee` — locking all accrued protocol and admin fees inside the pool with no recovery mechanism.

## Finding Description
`setPoolAdminFeeDestination` stores any non-zero address immediately with no timelock and no capability check: [1](#0-0) 

`collectFees` (restricted to `onlyFactory`) transfers to `adminFeeDestination_` first via `safeTransfer`, which propagates any revert from the destination: [2](#0-1) [3](#0-2) 

The notional-fee accumulators are cleared only after all transfers succeed, so a revert leaves them intact but permanently unclaimable. Every factory entry-point that collects fees passes `poolAdminFeeDestination[pool]` unchanged with no alternative overload: [4](#0-3) [5](#0-4) [6](#0-5) 

The factory owner has no bypass: `collectFees` is `onlyFactory`, and the factory always forwards the stored `poolAdminFeeDestination[pool]`. [7](#0-6) 

## Impact Explanation
All accrued spread fees (pool balance surplus) and notional fees (`notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`) are permanently locked inside the pool. The factory owner loses the ability to adjust protocol fee rates on the affected pool because `setPoolProtocolFee` calls `collectFees` as its first step and will always revert. This is a direct, irrecoverable loss of protocol fee revenue and a broken core governance function — matching the allowed impact gate for "broken core pool functionality causing loss of funds" and "direct loss of protocol fees."

## Likelihood Explanation
The pool admin is a distinct, semi-trusted role. `setPoolAdminFeeDestination` has no timelock and no cap on the destination address — only a zero-address check. A malicious or compromised pool admin can execute this in a single transaction at any time after deployment, with no preconditions. The only on-chain signal is the `PoolAdminFeeDestinationUpdated` event. This constitutes a fund-impacting action that exceeds the pool admin's intended scope (caps and timelocks), satisfying the contest's semi-trusted bypass criterion.

## Recommendation
Separate the admin and protocol fee transfers so a failure in the admin leg cannot block the protocol leg. Options:
1. Use `try/catch` around the admin `safeTransfer` calls and emit an event on failure, leaving admin fees claimable via a pull pattern.
2. Split `collectFees` into two independent functions — one for the protocol share and one for the admin share — so the protocol can always collect its portion.
3. At minimum, add a timelock to `setPoolAdminFeeDestination` so the factory owner can intervene before a malicious destination takes effect.

## Proof of Concept
1. Pool admin deploys a `RevertOnReceive` contract whose fallback/receive reverts unconditionally.
2. Pool admin calls `factory.setPoolAdminFeeDestination(pool, address(revertOnReceive))`.
3. Swaps accumulate spread surplus and `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`.
4. Anyone calls `factory.collectPoolFees(pool)` → `pool.collectFees(...)` → `transferToken0(revertOnReceive, ...)` → **revert**. Protocol receives nothing.
5. Factory owner calls `factory.setPoolProtocolFee(pool, newFee, 0)` → same revert path → **cannot update fee rates**.
6. Fees remain locked in the pool indefinitely with no recovery mechanism available to the protocol.
