### Title
Unauthenticated Wallet Rollback to Arbitrary Depth via Pre-Validation Commit in `wallet_short_sync_backtrack` — (`chia/wallet/wallet_node.py`)

### Summary

An unprivileged remote peer can cause the wallet to commit a `perform_atomic_rollback` to an arbitrary depth (including genesis) for chains below `WEIGHT_PROOF_RECENT_BLOCKS`, before any per-block VDF/PoSpace validation occurs. The rollback is durably written to the DB; if the attacker's blocks subsequently fail `validate_finished_header_block`, the wallet is left in a permanently rolled-back state until it re-syncs from an honest peer.

### Finding Description

**Entry point — `new_peak_from_untrusted` (line 1279):**

When `new_peak_hb.height < constants.WEIGHT_PROOF_RECENT_BLOCKS`, the function short-circuits directly to `sync_from_untrusted_close_to_peak` without ever calling `long_sync_from_untrusted` (which would invoke `fetch_and_update_weight_proof`). [1](#0-0) 

**No cap on backtrack depth — `wallet_short_sync_backtrack` (lines 1408–1412):**

The `LONG_SYNC_THRESHOLD` guard that would abort the backtrack and return `None` is conditioned on `header_block.height >= constants.WEIGHT_PROOF_RECENT_BLOCKS`. For short chains this condition is never true, so the loop walks all the way to genesis. [2](#0-1) 

**Hash-chain continuity check is the only gate during backtrack (line 1425):**

The loop only verifies that `prev_head.header_hash == top.prev_header_hash`. An attacker can trivially satisfy this by constructing a chain of `HeaderBlock` objects whose `header_hash` values (computed from their content) form a consistent linked list, regardless of whether the VDFs inside are valid. [3](#0-2) 

**Rollback committed before validation (lines 1444–1454):**

`perform_atomic_rollback(fork_height)` is called and durably committed inside its own `db_wrapper.writer()` transaction. Only *after* that does the code iterate over the collected blocks and call `WalletBlockchain.add_block`, which is where `validate_finished_header_block` (VDF/PoSpace checks) actually runs. [4](#0-3) 

**`WalletBlockchain.add_block` validates VDFs — but too late:**

The wallet blockchain does call `validate_finished_header_block` for every block, which would reject blocks with invalid VDFs. However, by the time this runs, the rollback is already committed. [5](#0-4) 

**`perform_atomic_rollback` is a durable, non-reversible DB write:** [6](#0-5) 

**Exception propagation closes the peer but does not undo the rollback:**

The `ValueError` raised at line 1454 propagates through `sync_from_untrusted_close_to_peak` and `new_peak_from_untrusted` (neither catches it on this path) and is caught only at `_process_new_subscriptions` line 714, which closes the peer. The DB rollback is not undone. [7](#0-6) 

### Impact Explanation

After a successful attack:
- `_finished_sync_up_to` is set to `fork_height` (potentially 0) in the DB.
- All coin records above `fork_height` are deleted (`reorg_rollback`).
- Confirmed transactions appear unconfirmed; wallet balances are incorrect.
- The wallet must re-sync from `fork_height` on the next honest peer connection.
- An attacker who repeatedly connects can keep the wallet in a perpetually rolled-back state, preventing it from maintaining a stable synced view of confirmed coins.

This maps to **High: Corruption of wallet sync state with direct security impact** and **High: Long-lived inability for the wallet to process sync updates under normal network assumptions**.

### Likelihood Explanation

- Requires only a TCP connection to the wallet's full-node port — no keys, no mining, no admin access.
- Constructing a hash-linked chain of `HeaderBlock` objects with invalid VDFs is trivial (SHA256 hashing only).
- The attacker must announce a weight higher than the current peak, which is a field they control in the `NewPeakWallet` message and the served `HeaderBlock`.
- The timestamp check (`is_timestamp_in_sync`) is also answered by the attacker and is trivially satisfied.
- The only precondition is that the canonical chain is currently below `WEIGHT_PROOF_RECENT_BLOCKS` (e.g., early mainnet, testnets, or a wallet that has never synced past that height).

### Recommendation

Move `perform_atomic_rollback` to occur **after** all blocks in the backtrack list have been successfully validated by `WalletBlockchain.add_block`. Alternatively, wrap both the rollback and the `add_block` loop in a single atomic DB transaction so that a validation failure causes the rollback to be reverted. A secondary fix is to validate each fetched `HeaderBlock` during the backtrack loop itself (before appending to `blocks`), so that invalid blocks are rejected before any state mutation occurs.

        self.log.info(f"perform_atomic_rollback to {fork_height}")
