Audit Report

## Title
Unguarded `addNewSupportedAsset()` Freezes `updateRSETHPrice()`, Enabling Deposits at Stale Rate — (`contracts/LRTOracle.sol`, `contracts/LRTConfig.sol`)

## Summary
`LRTConfig.addNewSupportedAsset()` adds an asset to `supportedAssetList` with no requirement that a price oracle exists in `LRTOracle`. Until `updatePriceOracleFor()` is called, every invocation of `updateRSETHPrice()` reverts because `_getTotalEthInProtocol()` calls `getAssetPrice()` on the oracle-less asset, which hard-reverts via the `onlySupportedOracle` modifier. During this window, `rsETHPrice` is frozen at its last committed value, and any depositor of existing assets (stETH, ETH, etc.) receives more rsETH than they are entitled to, diluting accrued yield from all existing rsETH holders.

## Finding Description
**Root cause:** `_addNewSupportedAsset()` sets `isSupportedAsset[asset] = true` and pushes to `supportedAssetList` with no corresponding oracle registration check.

**Revert path:** `updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` iterates every entry in `lrtConfig.getSupportedAssetList()` and calls `getAssetPrice(asset)`. The `onlySupportedOracle` modifier on `getAssetPrice()` reverts with `AssetOracleNotSupported` when `assetPriceOracle[asset] == address(0)`. This makes `updateRSETHPrice()` and `updateRSETHPriceAsManager()` both uncallable for the entire window.

**Exploit path:** While `rsETHPrice` is stale-low, a depositor calls `depositAsset(stETH, amount, 0, "")`. `_beforeDeposit()` calls `getRsETHAmountToMint(stETH, amount)`, which computes `(amount * getAssetPrice(stETH)) / lrtOracle.rsETHPrice()`. The numerator uses a live oracle price for stETH; the denominator uses the frozen, below-true-value `rsETHPrice`. The depositor receives excess rsETH. When the oracle is eventually registered and `rsETHPrice` is updated to the true rate, the dilution is already permanent.

**Existing checks insufficient:** `updatePriceOracleFor()` requires `onlyLRTAdmin` and is a separate transaction from the timelock-gated `addNewSupportedAsset()`. There is no atomic coupling between the two, and no guard in `depositAsset()` / `depositETH()` that forces a price refresh before minting.

## Impact Explanation
**High — Theft of unclaimed yield.** Staking rewards accrue continuously. The frozen `rsETHPrice` is lower than the true rate. Every deposit of an existing asset during the window mints excess rsETH, permanently diluting the yield that existing rsETH holders have accrued but not yet claimed. The magnitude scales with: (a) the duration of the window, (b) the rate of staking reward accrual, and (c) the volume of deposits during the window.

## Likelihood Explanation
`addNewSupportedAsset()` is gated by `TIME_LOCK_ROLE` (a timelock contract), while `updatePriceOracleFor()` is gated by `onlyLRTAdmin`. These are structurally separate transactions — the timelock execution of the former creates a **guaranteed non-zero window** before the admin can call the latter. No admin error or key compromise is required; the window is an inherent consequence of normal governance operations. Any depositor can exploit it permissionlessly by depositing any existing supported asset during the window.

## Recommendation
1. **Atomic registration:** Extend `addNewSupportedAsset()` (or a privileged wrapper) to atomically call `LRTOracle.updatePriceOracleFor()` in the same transaction, ensuring no asset ever enters `supportedAssetList` without a live oracle.
2. **Graceful skip:** In `_getTotalEthInProtocol()`, skip assets whose `assetPriceOracle` is `address(0)` rather than reverting, so a missing oracle degrades gracefully instead of freezing all price updates.
3. **Deposit guard:** In `_beforeDeposit()`, assert that `updateRSETHPrice()` does not revert (or call it directly) before computing `getRsETHAmountToMint()`, so deposits cannot proceed against a stale price.

## Proof of Concept
1. Protocol is live; `rsETHPrice` was last updated at `T=0` (e.g., `1.02e18`).
2. At `T=1`, timelock executes `LRTConfig.addNewSupportedAsset(newLST, limit)`. `newLST` is now in `supportedAssetList`; `assetPriceOracle[newLST] == address(0)`.
3. At `T=2`, staking rewards push the true rsETH/ETH rate to `1.025e18`. Any call to `updateRSETHPrice()` reverts: `_getTotalEthInProtocol()` → `getAssetPrice(newLST)` → `onlySupportedOracle` → `revert AssetOracleNotSupported`.
4. Attacker calls `depositAsset(stETH, 100e18, 0, "")`. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.02e18 ≈ 98.04 rsETH` instead of the correct `≈ 97.56 rsETH`. Attacker receives ~0.48 rsETH extra per 100 stETH.
5. At `T=3`, admin calls `updatePriceOracleFor(newLST, oracle)`. `updateRSETHPrice()` now succeeds and `rsETHPrice` jumps to `1.025e18`, but the dilution is already permanent.
