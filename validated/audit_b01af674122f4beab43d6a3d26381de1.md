### Title
Production Use of `aws-lc-rs::unstable` ML-DSA-65 API for Transaction Signature Verification - (File: core/crypto/src/signature.rs)

### Summary
nearcore uses the explicitly-unstable `aws-lc-rs::unstable::signature::ML_DSA_65` API in production to verify ML-DSA-65 (post-quantum) transaction signatures. This is the direct analog to the ABIEncoderV2 finding: an experimental/unstable cryptographic primitive is used in a stabilized production protocol feature (protocol version 85), where any latent bug in the unstable API could corrupt the signature verification result that gates transaction admission, balance transfers, and access-key operations.

### Finding Description

`Cargo.toml` pins `aws-lc-rs` at an exact version with the `unstable` feature enabled: [1](#0-0) 

`core/crypto/src/signature.rs` imports directly from the unstable module: [2](#0-1) 

The production `Signature::verify` path for ML-DSA-65 calls the unstable `UnparsedPublicKey::verify`: [3](#0-2) 

The `PostQuantumSignatures` feature is stabilized at protocol version 85, meaning ML-DSA-65 transactions are accepted on mainnet: [4](#0-3) 

The project's own documentation acknowledges the risk explicitly as Caveat 3: [5](#0-4) 

The `unstable` gate exists precisely because AWS-LC may evolve or fix these APIs, meaning the current implementation may contain bugs that are not yet publicly known.

### Impact Explanation

If the `aws-lc-rs::unstable` ML-DSA-65 `verify` implementation contains a bug:

1. **Signature bypass**: A crafted 3309-byte ML-DSA-65 signature blob could pass `unparsed.verify(data, &signature.0[..]).is_ok()` for a message it does not actually sign, allowing an attacker to execute transactions (transfers, `AddKey`, `DeleteKey`) from any account that holds an ML-DSA-65 access key — corrupting the account balance and nonce stored in the trie state root.

2. **Consensus divergence**: If different nodes run different builds of `aws-lc-rs` (e.g., after a silent patch to the pinned version's C layer), the boolean verification result for the same `(pubkey, message, signature)` triple could differ across nodes, causing a state-root mismatch and a consensus split.

The corrupted protocol values are: account balance, nonce, access-key set, and the chunk/block state root that commits them.

### Likelihood Explanation

- The `unstable` designation is an explicit API-stability disclaimer from AWS-LC; the PQDSA APIs are newer than the stable ed25519/ECDSA paths and have had less production exposure.
- The version is pinned (`=1.16.2`), which prevents silent API drift but does not protect against bugs already present in that version.
- An unprivileged user can reach the vulnerable path by simply submitting any ML-DSA-65-signed transaction after protocol version 85 activates — no special privilege is required.
- The `BorshDeserialize` for `Signature::MLDSA65` accepts any 3309-byte blob without structural validation before passing it to `verify`: [6](#0-5) 

### Recommendation

1. **Add a structural pre-check** on the ML-DSA-65 signature bytes before calling `unparsed.verify` (e.g., validate the hint-count field at `sig[3308] <= 55` per the FIPS 204 spec), so malformed blobs are rejected before reaching the unstable library.
2. **Track the `unstable` feature gate** as a protocol-level risk item; each `aws-lc-rs` version bump should include a regression test against the known-answer test vector already present in the codebase: [7](#0-6) 

3. **Consider a wrapper** that cross-checks the `aws-lc-rs` result against a second independent ML-DSA-65 implementation for a sample of signatures during the initial mainnet rollout period, analogous to the dYdX recommendation to upgrade to a fixed compiler version.

### Proof of Concept

**Entry path** (unprivileged user):

1. User holds an account with an ML-DSA-65 access key (added via `AddKey` after protocol version 85).
2. User constructs a `SignedTransaction` with `signature = Signature::MLDSA65(MlDsa65Signature(crafted_3309_bytes))` where `crafted_3309_bytes` is chosen to trigger a hypothetical bug in `aws_lc_rs::unstable::signature::ML_DSA_65`'s verify path.
3. The transaction is submitted via RPC. The mempool and chunk-production path call `Signature::verify` at: [3](#0-2) 

4. If the unstable API returns `Ok(())` for the crafted bytes, the transaction is admitted, included in a chunk, and executed — transferring balance or modifying keys without a valid signature.
5. The corrupted state root is committed to the block, propagating the invalid state to all nodes that accept the block.
