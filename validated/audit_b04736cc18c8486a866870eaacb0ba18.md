Audit Report

## Title
Unbounded `merkleProof` Length in `_processClaim` Enables Block Stuffing — (`contracts/KERNEL/KernelMerkleDistributor.sol`)

## Summary
`KernelMerkleDistributor._processClaim` accepts a `bytes32[] calldata merkleProof` with no upper-bound check and passes it directly to `MerkleProofUpgradeable.verify`, which iterates over every element via an unbounded loop. An attacker can submit transactions with arbitrarily large proof arrays that consume gas proportional to proof length, revert without updating state, and be resubmitted indefinitely with the same address and index to fill blocks and delay legitimate KERNEL claims.

## Finding Description
`_processClaim` accepts `merkleProof` with no length cap: [1](#0-0) 

Five guards precede proof verification, all of which an attacker with their own funded EOA and any valid `index` (1 … `currentIndex`, readable on-chain) can satisfy: [2](#0-1) 

After all guards pass, execution reaches `MerkleProofUpgradeable.verify`, which calls `processProof` — an unconditional loop over every proof element: [3](#0-2) 

Because the transaction reverts with `InvalidMerkleProof` after the loop completes, **no state is written**: [4](#0-3) 

The attacker can therefore reuse the same address and index indefinitely, submitting a stream of high-gas-cost reverting transactions.

## Impact Explanation
An attacker submits `claim` transactions carrying proof arrays sized to consume nearly the full block gas limit (~30 M gas on Ethereum mainnet). Each transaction reverts after burning gas proportional to proof length. By filling consecutive blocks, the attacker delays or prevents legitimate claimants from having their transactions included. This matches the allowed impact: **Low — Block stuffing**.

## Likelihood Explanation
The attack requires only a funded EOA and knowledge of any valid `index` value (readable from `currentIndex` on-chain). No admin key, governance capture, or external protocol compromise is needed. The attacker pays for the gas consumed, but block stuffing is inherently a paid attack; the cost is bounded only by the attacker's budget. The attack is repeatable indefinitely because reverting transactions never update `isClaimed`.

## Recommendation
Add a maximum proof length check at the top of `_processClaim`:

```solidity
uint256 constant MAX_PROOF_LENGTH = 32; // ceil(log2(max expected leaves))

function _processClaim(..., bytes32[] calldata merkleProof) internal returns (uint256) {
    if (merkleProof.length > MAX_PROOF_LENGTH) revert InvalidMerkleProof();
    // ... existing logic
}
```

A cap of 32–64 is safe for any realistic distribution (2^32 leaves is astronomically large) while making per-transaction gas cost bounded and predictable.

## Proof of Concept
Deploy `KernelMerkleDistributor` with a non-zero merkle root and `currentIndex >= 1`. Call `claim(1, address(this), 1 ether, proof)` with increasing `proof` lengths (1, 100, 1_000, 10_000 zero-valued `bytes32` elements). Measure `gasleft()` before and after each call. Gas usage scales linearly with proof length — a proof of ~200,000 zero-valued `bytes32` elements consumes approximately 30 M gas (calldata: 200k × 32 × 4 = 25.6 M + compute: 200k × 30 = 6 M), sufficient to fill one Ethereum block per transaction. Each call reverts with `InvalidMerkleProof` and writes no state, allowing indefinite resubmission with the same `index` and `account`.
