Audit Report

## Title
First Depositor rsETH Share Inflation via Unrestricted ETH Donation and Public Price Update - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool` is vulnerable to a classic first-depositor share inflation attack. An attacker deposits 1 wei of ETH to mint 1 rsETH unit, donates a large ETH amount via the unrestricted `receive()` function, then calls the public `updateRSETHPrice()` to commit the inflated price. Any subsequent depositor who passes `minRSETHAmountExpected = 0` receives 0 rsETH due to integer truncation, and the attacker's single rsETH unit absorbs 100% of the pool's TVL including the victim's deposit.

## Finding Description

**Root cause 1 — No zero-share guard in `_beforeDeposit`.**

`_beforeDeposit` only checks `rsethAmountToMint < minRSETHAmountExpected`. When `minRSETHAmountExpected == 0` and `rsethAmountToMint == 0`, the condition `0 < 0` is false and no revert occurs. `_mintRsETH(0)` is silently called. [1](#0-0) 

**Root cause 2 — Integer division in `getRsETHAmountToMint`.**

If `rsETHPrice` has been inflated beyond `amount * assetPrice`, the division truncates to 0. [2](#0-1) 

**Root cause 3 — `updateRSETHPrice()` is public with no access control.**

Any unprivileged caller can commit a price update at any time. [3](#0-2) 

**Root cause 4 — `totalETHInProtocol` includes raw ETH balance of `LRTDepositPool`.**

`getETHDistributionData()` sets `ethLyingInDepositPool = address(this).balance`, so any ETH sent directly to the contract is immediately reflected in the price calculation. [4](#0-3) 

**Root cause 5 — `receive()` is unrestricted.**

Anyone can donate ETH to inflate `totalETHInProtocol` without minting any rsETH. [5](#0-4) 

**Root cause 6 — `pricePercentageLimit` defaults to 0, disabling the only price-spike guard.**

`initialize()` never sets `pricePercentageLimit`, so it remains 0. The guard `pricePercentageLimit > 0 && ...` short-circuits to `false` permanently, allowing an arbitrarily large single-step price jump. [6](#0-5) [7](#0-6) 

**Existing checks reviewed and shown insufficient:**

- `minAmountToDeposit` defaults to 0, so a 1 wei deposit passes the `depositAmount < minAmountToDeposit` check.
- The deposit limit check for ETH only reverts if `totalAssetDeposits > depositLimit`, which does not block a small seed deposit.
- `_updateRsETHPrice` returns early only when `rsethSupply == 0`; after the attacker's seed deposit, supply is 1 and the early return does not trigger. [8](#0-7) [9](#0-8) 

## Impact Explanation

**Critical — Direct theft of user funds.**

A victim depositing Y ETH with `minRSETHAmountExpected = 0` receives 0 rsETH. Their Y ETH is permanently absorbed into the pool. The attacker, holding the sole rsETH unit, redeems it for the entire pool balance (`1 wei + X ETH donation + Y ETH victim deposit`), netting a profit of approximately Y ETH minus gas costs. This is direct, concrete theft of depositor funds, matching the Critical impact class.

## Likelihood Explanation

**Medium.** All four required conditions are realistic at protocol launch:
1. Being the first depositor (or acting when rsETH supply is negligibly small) — plausible at launch or after a large burn event.
2. Donating sufficient ETH to inflate the price past the victim's deposit — capital is recovered via rsETH redemption; the only cost is gas and the 1 wei seed.
3. Victim calling `depositETH` with `minRSETHAmountExpected = 0` — the default for many contract integrations, aggregators, and naive front-end calls.
4. `pricePercentageLimit == 0` — the uninitialized default confirmed in `initialize()`.

No privileged access is required. The attack is repeatable as long as the attacker can act before any other depositor.

## Recommendation

1. **Add a zero-share guard in `_beforeDeposit`:**
   ```solidity
   require(rsethAmountToMint != 0, "zero rsETH minted");
   ``` [1](#0-0) 

2. **Seed the supply at initialization** (Uniswap V2 pattern): mint a small amount of rsETH to `address(0)` on the first deposit when `totalSupply() == 0`, making share price manipulation economically impractical.

3. **Set a non-zero `pricePercentageLimit` in `initialize()`** so that a single `updateRSETHPrice()` call cannot commit an arbitrarily large price jump. [10](#0-9) 

## Proof of Concept

```
Initial state: rsETHPrice = 1e18, rsethSupply = 0, minAmountToDeposit = 0, pricePercentageLimit = 0

1. Attacker calls depositETH{value: 1}(0, "")
   → rsethAmountToMint = (1 * 1e18) / 1e18 = 1
   → rsethSupply = 1, pool ETH balance = 1 wei

2. Attacker sends 10_000 ether directly to LRTDepositPool (receive())
   → pool ETH balance = 1 + 10_000e18 wei
   → rsethSupply still = 1

3. Attacker calls LRTOracle.updateRSETHPrice()
   → totalETHInProtocol ≈ 10_000e18
   → newRsETHPrice = 10_000e18 * 1e18 / 1 = 10_000e36
   → pricePercentageLimit == 0 → isPriceIncreaseOffLimit = false → no revert
   → rsETHPrice = 10_000e36

4. Victim calls depositETH{value: 9_999 ether}(0, "")
   → rsethAmountToMint = (9_999e18 * 1e18) / 10_000e36
                       = 9_999e36 / 10_000e36 = 0 (integer truncation)
   → minRSETHAmountExpected = 0 → 0 < 0 is false → no revert
   → _mintRsETH(0) called; victim receives 0 rsETH
   → 9_999 ETH enters pool

5. Attacker redeems 1 rsETH (100% of supply)
   → Recovers ≈ 1 + 10_000e18 + 9_999e18 ≈ 20_000 ETH
   → Net profit ≈ 9_999 ETH (victim's full deposit)
```
