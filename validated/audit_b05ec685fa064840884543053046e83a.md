### Title
Missing Output-Creation Validation Allows Permanent Freezing of ADA/Native Assets Sent to Plutus V1/V2 Script Addresses Without Datum Hashes - (File: `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxow.hs`)

---

### Summary

The Cardano Ledger's UTXOW rule enforces datum-hash presence only at **spending time**, not at **output creation time**. Any unprivileged transaction sender can create a UTxO output to a Plutus V1/V2 script address without a datum hash; the ledger accepts the producing transaction unconditionally, but every subsequent attempt to spend that output is permanently rejected with `UnspendableUTxONoDatumHash`. The funds are irrecoverably frozen with no on-chain recovery path short of a hard fork.

---

### Finding Description

In the Alonzo era and all subsequent eras (Babbage, Conway, Dijkstra), the UTXOW rule's `missingRequiredDatums` function checks for `UnspendableUTxONoDatumHash` only when **consuming** inputs from script addresses:

```haskell
-- eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxow.hs:219-234
missingRequiredDatums scriptsProvided utxo tx = do
  let txBody = tx ^. bodyTxL
      (inputHashes, txInsNoDataHash) = getInputDataHashesTxBody utxo txBody scriptsProvided
      ...
  sequenceA_
    [ failureOnNonEmptySet txInsNoDataHash UnspendableUTxONoDatumHash
    , ...
    ]
```

There is no symmetric check in the UTXO rule that rejects **creating** an output to a Plutus V1/V2 script address without a datum hash. The formal specification explicitly acknowledges this asymmetry:

> "a phase-2 script-locked output that does not include a datum hash is unspendable."
> — `eras/alonzo/formal-spec/utxo.tex:289-290`

The spec's stated justification is:

> "Since it is in general impossible for the ledger implementation to know at the time of transaction creation whether or not an output belongs to a phase-2 script, we simply allow any output to contain a hash of a datum."
> — `eras/alonzo/formal-spec/transactions.tex:38-40`

However, the payment credential of a Shelley address is either a `KeyHashObj` or a `ScriptHashObj`. When it is a `ScriptHashObj`, the ledger knows the output is locked by a script. The only information unavailable at creation time is whether that script is phase-1 (native/timelock) or phase-2 (Plutus). This gap is the root cause: the ledger performs no output-side guard to prevent the creation of permanently unspendable UTxOs.

The test suite confirms the two-step failure path:

```haskell
-- eras/alonzo/impl/testlib/Test/Cardano/Ledger/Alonzo/Imp/UtxowSpec/Invalid.hs:129-153
it "UnspendableUTxONoDatumHash" $ do
  txIn <- impAnn "Produce script at a txout with a missing datahash" $ do
    ...
    txInAt 0
      <$> withPostFixup
        (fixupResetAddrWits <$> resetTxOutDataHash)
        (submitTx tx)          -- ← SUCCEEDS: output created without datum hash
  let tx = mkBasicTx (mkBasicTxBody & inputsTxBodyL .~ [txIn])
  if lang >= PlutusV3
    then submitPhase2Invalid_ tx
    else submitFailingTx tx [injectFailure $ UnspendableUTxONoDatumHash $ NES.singleton txIn]
    -- ↑ FAILS: funds permanently frozen for PlutusV1/V2
```

The same pattern is confirmed in the Conway era test:

```haskell
-- eras/conway/impl/testlib/Test/Cardano/Ledger/Conway/Imp/UtxosSpec.hs:64-78
describe "Spending script without a Datum" $ do
  forM_ ([minBound .. eraMaxLanguage @era] :: [Language]) $ \lang -> do
    it (show lang) $ do
      ...
      txIn <- sendCoinTo addr amount   -- ← SUCCEEDS: funds sent to script address without datum
      let tx = mkBasicTx (mkBasicTxBody & inputsTxBodyL .~ [txIn])
      if lang >= PlutusV3
        then submitTx_ tx
        else submitFailingTx tx
              [ injectFailure $ Alonzo.UnspendableUTxONoDatumHash $ NES.singleton txIn ]
```

The `UnspendableUTxONoDatumHash` predicate failure is defined in:
- `eras/alonzo/impl/src/Cardano/Ledger/Alonzo/Rules/Utxow.hs:107-108` (Alonzo)
- `eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Utxow.hs:95-97` (Conway)
- `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxow.hs:119-121` (Dijkstra)

All three eras carry this failure forward, confirming the issue persists across the entire post-Alonzo ledger lineage for Plutus V1/V2 scripts.

---

### Impact Explanation

**High — Permanent freezing of funds where recovery requires a hard fork.**

Any ADA or native assets sent to a Plutus V1/V2 script address without a datum hash enter the UTxO set and can never be spent. The `UnspendableUTxONoDatumHash` failure is a phase-1 (pre-script) ledger rejection, meaning no transaction can ever consume the output regardless of what witnesses, redeemers, or datums are supplied later. The only recovery mechanism would be a protocol rule change via a hard fork. This matches the allowed impact: *"High. Permanent freezing of funds, deposits, rewards, or withdrawals where recovery requires a hard fork."*

---

### Likelihood Explanation

**Medium.** The producing transaction is accepted silently with no warning. Any user, wallet, or DApp that constructs a transaction sending value to a Plutus V1/V2 script address and omits the datum hash (e.g., due to a wallet bug, a DApp frontend error, or a misconfigured transaction builder) will permanently lose those funds. This failure mode has occurred on Cardano mainnet. The ledger provides no output-side guard to prevent it.

---

### Recommendation

Add a validation predicate in the UTXO rule (output creation path) that rejects outputs whose payment credential is a `ScriptHashObj` and whose datum field is absent, when the script is known to be phase-2 (Plutus V1/V2). For cases where the script is not available in the current transaction, consider requiring a datum hash (or inline datum) for **all** outputs to `ScriptHashObj` addresses, since phase-1 (native/timelock) scripts do not use datum hashes and such a requirement would be harmless for them while preventing permanent fund loss for phase-2 scripts.

### Proof of Concept

**Step 1 — Create a Plutus V1/V2 script address and send ADA without a datum hash:**

```
Transaction T1:
  outputs:
    - address: Addr Testnet (ScriptHashObj <plutusV1ScriptHash>) StakeRefNull
      value:   10_000_000 lovelace
      datum:   (none)        ← no datum hash, no inline datum
```

The ledger accepts T1. The UTxO entry `(T1#0, 10 ADA, scriptAddr, noDatum)` is added to the UTxO set.

**Step 2 — Attempt to spend the output:**

```
Transaction T2:
  inputs:  [T1#0]
  witnesses:
    - script: <plutusV1Script>
    - redeemer: <any>
    - datum: <any>
```

The ledger rejects T2 at phase-1 (UTXOW rule) with:

```
UnspendableUTxONoDatumHash { T1#0 }
```

No combination of witnesses, datums, or redeemers can make T2 valid. The 10 ADA in `T1#0` is permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)

```haskell
missingRequiredDatums scriptsProvided utxo tx = do
  let txBody = tx ^. bodyTxL
      (inputHashes, txInsNoDataHash) = getInputDataHashesTxBody utxo txBody scriptsProvided
      txHashes = Map.keysSet (tx ^. witsTxL . datsTxWitsL . unTxDatsL)
      unmatchedDatumHashes = Set.difference inputHashes txHashes
      allowedSupplementalDataHashes = getSupplementalDataHashes utxo txBody
      supplimentalDatumHashes = Set.difference txHashes inputHashes
      (okSupplimentalDHs, notOkSupplimentalDHs) =
        Set.partition (`Set.member` allowedSupplementalDataHashes) supplimentalDatumHashes
  sequenceA_
    [ failureOnNonEmptySet txInsNoDataHash UnspendableUTxONoDatumHash
    , failureOnNonEmptySet unmatchedDatumHashes (\unmatched -> MissingRequiredDatums unmatched txHashes)
    , failureOnNonEmptySet
        notOkSupplimentalDHs
        (\notOk -> NotAllowedSupplementalDatums notOk okSupplimentalDHs)
    ]
```
