### Title
Unvalidated `builder` Field in `ProposalInit` Allows Proposer to Redirect All Block Fees and Corrupt Block Hash — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_proposal_init_valid` validates many fields of a received `ProposalInit` (height, L1/L2 gas prices, timestamp, `starknet_version`, `version_constant_commitment`, `fee_proposal_fri`) but never validates the `builder` field. The `builder` field is passed verbatim into `convert_to_sn_api_block_info` as `sequencer_address`, which is the recipient of all transaction fees and a component of the block hash. A malicious proposer can set `builder` to any arbitrary address, redirecting all block fees to an attacker-controlled account and producing a wrong block hash and wrong `get_sequencer_address` syscall result — all of which validators accept without objection.

---

### Finding Description

`ProposalInit` carries a `builder` field: [1](#0-0) 

`is_proposal_init_valid` checks `height`, `l1_da_mode`, `l2_gas_price_fri`, `timestamp`, `starknet_version`, `version_constant_commitment`, all four L1 gas price fields, and `fee_proposal_fri`. The `ProposalInitValidation` struct that carries the locally-trusted reference values has no `builder` field at all: [2](#0-1) 

The combined check at lines 312–321 validates `height`, `l1_da_mode`, and `l2_gas_price_fri` — `builder` is absent: [3](#0-2) 

After `is_proposal_init_valid` returns `Ok`, `initiate_validation` calls `convert_to_sn_api_block_info(init)`, which maps `init.builder` directly to `sequencer_address`: [4](#0-3) 

This `BlockInfo` is forwarded to the batcher as `ValidateBlockInput.block_info`: [5](#0-4) 

The batcher executes every transaction in the block with this `sequencer_address`. The fee transfer recipient is `block_context.block_info.sequencer_address`: [6](#0-5) 

The same `sequencer_address` is hashed into `PartialBlockHashComponents` and thus into the final block hash: [7](#0-6) 

The `get_sequencer_address` syscall also returns this value to executing contracts: [8](#0-7) 

---

### Impact Explanation

**Wrong fee accounting with direct economic impact (Critical):** Every transaction fee in the block is transferred to the attacker-supplied `builder` address. The legitimate sequencer receives nothing; the attacker's address accumulates all fees. This is a direct, quantifiable economic loss per block the attacker proposes.

**Wrong block hash (Critical):** `sequencer_address` is a committed field in `PartialBlockHashComponents`. A block committed with an attacker-controlled `builder` produces a permanently wrong block hash that all validators accept, corrupting the canonical chain state.

**Wrong `get_sequencer_address` syscall result (High):** Contracts that call `get_sequencer_address` during execution receive the attacker-controlled address, which can affect contract logic that gates behavior on the sequencer identity.

---

### Likelihood Explanation

Any node that reaches its turn as proposer in the BFT rotation can trigger this. No special privilege beyond normal proposer rotation is required. The attack is silent — validators log no warning and return `Ok` from `validate_proposal`. The attacker simply sets `builder` to their own wallet address in `initiate_build`: [9](#0-8) 

---

### Recommendation

Add `builder` to `ProposalInitValidation` and check it in `is_proposal_init_valid`, analogous to how `l2_gas_price_fri` is checked:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid:
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The validator's own `builder_address` (its sequencer identity) should be the reference value, populated when constructing `ProposalInitValidation` in `validate_proposal` inside `sequencer_consensus_context.rs`: [10](#0-9) 

---

### Proof of Concept

1. A malicious node waits for its proposer turn.
2. In `initiate_build`, it sets `builder: attacker_wallet_address` instead of `args.builder_address`.
3. It broadcasts the `ProposalInit` with this modified `builder`.
4. Each validator calls `is_proposal_init_valid` — `builder` is not checked, so validation passes.
5. `initiate_validation` calls `convert_to_sn_api_block_info(init)`, producing a `BlockInfo` with `sequencer_address = attacker_wallet_address`.
6. The batcher executes all transactions; every fee transfer sends tokens to `attacker_wallet_address`.
7. The batcher computes `partial_block_hash` with `attacker_wallet_address` as the sequencer; the proposer sends a matching `ProposalFin`.
8. The validator's `built_block == received_fin.proposal_commitment` check passes.
9. Consensus commits the block. All fees are in the attacker's wallet; the block hash encodes the wrong sequencer. [11](#0-10)
