Audit Report

## Title
Missing Time-Based Staleness Check in `getRate()` Allows Stale Oracle Price Exploitation — (`contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

## Summary

`ChainlinkOracleForRSETHPoolCollateral.getRate()` validates round completeness and non-zero timestamp but performs no `block.timestamp - updatedAt > heartbeat` check. A Chainlink feed can remain in a valid, non-stale round for up to its full heartbeat interval (e.g. 24 h on Arbitrum) while the market price has moved significantly. An unprivileged attacker who deposits a supported token during a staleness window where the oracle price is stale-high receives proportionally more wrsETH from the pool than the deposited token is worth, draining the pool's wrsETH inventory.

## Finding Description

`ChainlinkOracleForRSETHPoolCollateral.getRate()` (L26–37) applies exactly three guards after calling `latestRoundData()`:

```solidity
if (answeredInRound < roundID) revert StalePrice();   // round completeness only
if (timestamp == 0)            revert IncompleteRound();
if (ethPrice <= 0)             revert InvalidPrice();
```

There is no guard of the form `if (block.timestamp - timestamp > heartbeat) revert StalePrice()`. Chainlink feeds update only when price deviation exceeds a threshold (e.g. 0.5%) **or** a heartbeat elapses (e.g. 24 h). Between those events `answeredInRound == roundID` and `timestamp != 0`, so all three checks pass even if the price is many hours old.

The token-deposit path in `RSETHPool.deposit(address,uint256,string)` (L284–305) is public and permissionless. It calls `viewSwapRsETHAmountAndFee` (L326–347), which computes:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

If `tokenToETHRate` is stale-high, `rsETHAmount` is proportionally inflated. The pool then executes `IERC20(address(wrsETH)).safeTransfer(msg.sender, rsETHAmount)` (L302), transferring excess wrsETH to the attacker. The pool's wrsETH inventory is permanently reduced by the difference between the stale and true value.

## Impact Explanation

The pool holds a wrsETH inventory that it exchanges for deposited tokens. A stale-high oracle price causes the pool to over-pay in wrsETH for every deposit during the staleness window. The attacker retains the excess wrsETH; the pool's inventory is permanently reduced. This constitutes **theft of unclaimed yield** (High): the pool's wrsETH balance represents accrued protocol value, and the attacker extracts value that was not legitimately earned by the deposit.

## Likelihood Explanation

Chainlink LST/ETH feeds on Arbitrum (e.g. wstETH/ETH) have 24-hour heartbeats and 0.5% deviation thresholds. During low-volatility periods the price can be many hours old while all three existing guards pass. No privileged access is required — `deposit(address,uint256,string)` is a public, permissionless function with only `nonReentrant` and `whenNotPaused` guards. A sophisticated attacker monitoring on-chain oracle state can identify the staleness window and execute the deposit atomically. The attack is repeatable for the full duration of any staleness window.

## Recommendation

Add a configurable heartbeat staleness check in `ChainlinkOracleForRSETHPoolCollateral.getRate()`:

```solidity
uint256 public immutable heartbeat; // e.g. 86_400 + 300 for a 24-hour feed

constructor(address _oracle, uint256 _heartbeat) {
    oracle = _oracle;
    heartbeat = _heartbeat;
}

function getRate() public view returns (uint256) {
    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (block.timestamp - timestamp > heartbeat) revert StalePrice();
    if (ethPrice <= 0) revert InvalidPrice();

    uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
    return normalizedPrice;
}
```

Set `heartbeat` to slightly above the feed's documented update interval (e.g. `86_400 + 300` seconds for a 24-hour feed).
