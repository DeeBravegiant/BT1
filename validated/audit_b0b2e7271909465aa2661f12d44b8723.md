### Title
Unvalidated `payload_hash` in `respond_verify_foreign_tx` Enables Cross-Request Signature Replay by a Single Malicious Attested Participant — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` function verifies the MPC signature against the **caller-supplied** `response.payload_hash` without checking that this hash corresponds to the actual pending request content. A single malicious attested participant can replay a valid MPC signature obtained from any prior signing session to resolve any pending foreign-transaction verification request with a forged response, enabling forged foreign-chain verification.

---

### Finding Description

In `respond_verify_foreign_tx`, the contract performs signature verification as follows:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;

// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The `payload_hash` is taken directly from the caller-supplied `response` object, not recomputed from the pending request. The contract only checks that the signature is valid over *some* hash — it does not verify that the hash corresponds to the actual foreign transaction request content (i.e., the specific `ForeignChainRpcRequest` and extracted values).

Contrast this with the regular `respond` function, which correctly derives the payload hash from the stored request itself:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");

near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,
    &expected_public_key,
)
.is_ok()
``` [2](#0-1) 

In `respond`, the hash is derived from `request.payload` — the stored, user-submitted payload — so a responding node cannot substitute an arbitrary hash. In `respond_verify_foreign_tx`, no such binding exists: the responding node freely chooses `response.payload_hash`.

The `ForeignTxSignPayload` that the hash is supposed to commit to encodes both the `ForeignChainRpcRequest` and the extracted values (`values`): [3](#0-2) 

The contract stores only the `VerifyForeignTransactionRequest` (not the extracted values), so it cannot independently recompute the expected hash — but it also makes no attempt to bind `response.payload_hash` to the pending request in any other way.

---

### Impact Explanation

A single malicious attested participant (strictly below the signing threshold) can:

1. Participate in a legitimate signing session for request **R1** (e.g., verifying Bitcoin tx T1 with extracted values V1), obtaining a valid MPC threshold signature **S1** over hash **H1 = hash(R1, V1)**.
2. Observe a new pending request **R2** (for a different foreign transaction T2) in the contract state.
3. Call `respond_verify_foreign_tx(R2, { payload_hash: H1, signature: S1 })`.
4. The contract verifies S1 is a valid signature over H1 under the root public key — which it is — and resolves all pending yields for R2 with the forged response.
5. Every user who submitted R2 receives `{ payload_hash: H1, signature: S1 }` — a cryptographically valid MPC signature, but one that attests to the verification of T1 (with V1), not T2.

If the receiving bridge contract does not independently verify that `response.payload_hash` equals the hash of the expected `(R2, expected_values)` pair, it will accept the forged attestation. This enables **forged foreign-chain verification**: the MPC network's signature is presented as proof that a foreign transaction (T2) was verified and produced specific extracted values, when in reality no such verification occurred. In a bridge context this directly enables double-spend or invalid inbound settlement. [4](#0-3) 

---

### Likelihood Explanation

The attack requires only that the adversary:

1. **Is an attested participant** (below threshold) — a normal operational role.
2. **Has participated in any prior signing session** — routine for any active node — and retained the resulting signature.
3. **Front-runs the legitimate nodes** when a new `verify_foreign_transaction` request appears — feasible by monitoring the NEAR indexer and submitting the forged `respond_verify_foreign_tx` call before honest nodes respond.

No threshold collusion, no key leakage, and no privileged access beyond being an attested participant is required. The attack is repeatable for every new pending request.

---

### Recommendation

Require the responding node to supply the full `ForeignTxSignPayload` (including the extracted `values`) alongside the response. The contract should then:

1. Verify that the payload's embedded `request` field matches the pending `VerifyForeignTransactionRequest`.
2. Recompute `expected_hash = payload.compute_msg_hash()` on-chain.
3. Assert `response.payload_hash == expected_hash`.
4. Verify the signature over `expected_hash` (not over the caller-supplied hash).

This mirrors the pattern already used in the SDK's `ForeignChainSignatureVerifier::verify_signature`: [5](#0-4) 

---

### Proof of Concept

```
1. Node M (malicious, attested, below threshold) participates in a legitimate
   signing session for request R1 = { chain: Bitcoin, tx_id: T1, extractors: [BlockHash] }.
   The MPC network produces signature S1 over H1 = hash(ForeignTxSignPayloadV1 { R1, V1 }).

2. User Alice submits verify_foreign_transaction(R2) where R2 = { chain: Bitcoin, tx_id: T2, ... }.
   The contract stores R2 as a pending request and issues a yield.

3. Node M immediately calls:
     respond_verify_foreign_tx(
       request  = R2,                          // valid pending key — lookup succeeds
       response = { payload_hash: H1,          // hash of a DIFFERENT request
                    signature:    S1 }         // valid MPC sig over H1
     )

4. Contract checks:
     verify_ecdsa_signature(S1, H1, root_pk)  // ✓ valid — S1 was produced by the MPC network
   → signature_is_valid = true

5. Contract calls resolve_yields_for(&R2, serialize(response))
   → Alice's yield is resolved; she receives { payload_hash: H1, signature: S1 }.

6. Alice's bridge contract receives a cryptographically valid MPC signature,
   but the signed payload attests to T1's block-hash, not T2's.
   If Alice's contract does not re-derive the expected hash from (R2, expected_values)
   and compare it to payload_hash, it accepts the forged attestation and
   credits an inbound transfer for T2 that was never actually verified.
```

    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
