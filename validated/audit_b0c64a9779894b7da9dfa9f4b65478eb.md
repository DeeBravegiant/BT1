### Title
Parent `OmniBridge.initialize()` Uses `initializer` Instead of `onlyInitializing`, Permanently Blocking `OmniBridgeWormhole` Initialization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge` is a base contract intended to be inherited by `OmniBridgeWormhole`. Its `initialize()` function uses the `initializer` modifier. `OmniBridgeWormhole.initializeWormhole()` also uses `initializer` and calls `OmniBridge.initialize()` internally. Because both use `initializer`, the nested call always reverts, making `OmniBridgeWormhole` permanently un-initializable.

### Finding Description

`OmniBridge.initialize()` is decorated with `initializer`: [1](#0-0) 

`OmniBridgeWormhole` inherits `OmniBridge` and its entry-point `initializeWormhole()` also uses `initializer`, then calls the parent's `initialize()`: [2](#0-1) 

OpenZeppelin's `initializer` modifier sets `_initializing = true` and `_initialized = 1` on the first (top-level) call. When `initializeWormhole()` enters, it sets those flags. The nested call to `OmniBridge.initialize()` then evaluates the `initializer` guard with `isTopLevelCall = false` (because `_initializing` is already `true`). The guard's two branches both fail:

- `(isTopLevelCall && _initialized < 1)` → `false`
- `(!isContract(address(this)) && _initialized == 1)` → `false` (it is a contract)

The transaction reverts unconditionally. There is no workaround: every call to `initializeWormhole()` will revert at the parent's `initializer` check.

### Impact Explanation

`OmniBridgeWormhole` can never be initialized. The Wormhole-backed bridge path — `deployToken`, `finTransfer`, `initTransfer`, and all Wormhole message publishing — is permanently inaccessible. Any funds sent to an un-initialized proxy would be permanently locked, and the Wormhole settlement path is a frozen redemption path.

### Likelihood Explanation

This triggers on the very first (and only possible) deployment call to `initializeWormhole()`. It is 100% reproducible with no special conditions. Any deployer attempting to stand up `OmniBridgeWormhole` will hit this revert immediately.

### Recommendation

Replace the `initializer` modifier on `OmniBridge.initialize()` with `onlyInitializing`. This allows the parent's initialization logic to run when called from within a child's `initializer`-guarded function, which is exactly the pattern OpenZeppelin's `onlyInitializing` is designed for.

```diff
// OmniBridge.sol
    function initialize(
        address tokenImplementationAddress_,
        address nearBridgeDerivedAddress_,
        uint8 omniBridgeChainId_
-   ) public initializer {
+   ) public onlyInitializing {
``` [3](#0-2) 

### Proof of Concept

1. Deploy an `ERC1967Proxy` pointing to `OmniBridgeWormhole` implementation.
2. Call `initializeWormhole(tokenImpl, nearAddr, chainId, wormholeAddr, consistencyLevel)`.
3. Execution enters `initializeWormhole`'s `initializer` guard → sets `_initializing = true`, `_initialized = 1`.
4. `initialize(tokenImpl, nearAddr, chainId)` is called → its `initializer` guard evaluates: `isTopLevelCall = false`, `_initialized = 1` (not `< 1`), `isContract = true` → **revert**.
5. `OmniBridgeWormhole` is permanently un-initializable. [2](#0-1) [3](#0-2)
