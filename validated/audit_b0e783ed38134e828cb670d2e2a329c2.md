Audit Report

## Title
rsETH Price Inflation via Direct ETH/ERC20 Donation to LRTDepositPool Enables Theft of Depositor Funds - (File: contracts/LRTOracle.sol)

## Summary
`LRTDepositPool` accepts unrestricted ETH via an open `receive()` function and uses raw `address(this).balance` / `IERC20.balanceOf()` as the source of truth for TVL. Because `updateRSETHPrice()` is publicly callable and `pricePercentageLimit` is never set in `initialize()` (defaults to `0`, disabling the only on-chain guard), an attacker can inflate `rsETHPrice` to an arbitrary value and cause subsequent depositors who pass `minRSETHAmountExpected = 0` to receive zero rsETH for their full ETH deposit.

## Finding Description

**Root cause — raw balance accounting:**

`getETHDistributionData()` returns `address(this).balance` as `ethLyingInDepositPool`: [1](#0-0) 

`getAssetDistributionData()` returns `IERC20(asset).balanceOf(address(this))` for LSTs: [2](#0-1) 

Both values are consumed by `_getTotalEthInProtocol()` → `_updateRsETHPrice()` to compute `rsETHPrice`: [3](#0-2) [4](#0-3) 

**Open receive function:**

Anyone can send ETH directly to `LRTDepositPool` without minting rsETH: [5](#0-4) 

**Public price update with no effective guard:**

`updateRSETHPrice()` is callable by any address: [6](#0-5) 

The only guard is `pricePercentageLimit`, but it is never initialized in `initialize()` (which only sets `lrtConfig`): [7](#0-6) 

Because `pricePercentageLimit` defaults to `0`, the condition `pricePercentageLimit > 0 && ...` is always `false`, making `isPriceIncreaseOffLimit` permanently `false`: [8](#0-7) 

**Minting formula uses the manipulated price:**

`getRsETHAmountToMint` divides by `lrtOracle.rsETHPrice()`, which is the inflated stored value: [9](#0-8) 

**Exploit path:**

1. Attacker deposits 1 wei ETH → `rsethSupply = 1 wei`, `rsETHPrice = 1e18`.
2. Attacker sends `X` ETH directly to `LRTDepositPool` via `receive()`.
3. Attacker calls `updateRSETHPrice()` → `totalETHInProtocol ≈ X`, `rsETHPrice ≈ X * 1e18`.
4. Victim calls `depositETH{value: Y}(0, "")` with `minRSETHAmountExpected = 0`.
   - `rsethAmountToMint = Y * 1e18 / (X * 1e18) = Y/X`, which rounds to `0` when `Y < X`.
   - The slippage check `0 < 0` is `false`, so the call succeeds and victim receives 0 rsETH.
5. Attacker's 1 wei rsETH now represents 100% of `(X + Y)` ETH in the protocol.
6. Attacker redeems 1 wei rsETH and recovers `X + Y − ε`, netting `Y` ETH profit.

**Why the slippage check does not fully mitigate this:**

`_beforeDeposit` reverts only if `rsethAmountToMint < minRSETHAmountExpected`: [10](#0-9) 

When `minRSETHAmountExpected = 0` (the zero-default commonly passed by users and scripts), `0 < 0` evaluates to `false` and the deposit proceeds, transferring the victim's ETH into the protocol while minting them nothing. The attacker can also calibrate the inflation to give the victim exactly `minRSETHAmountExpected` wei of rsETH (a dust amount) while still capturing nearly all of the victim's deposit value.

## Impact Explanation

**Critical — Direct theft of user funds.**

A victim who deposits ETH with `minRSETHAmountExpected = 0` (or any value the attacker can satisfy with dust rsETH) loses their entire deposit. The ETH is absorbed into the protocol and backs the attacker's single wei of rsETH. This is a concrete, on-chain loss of deposited principal, matching the "Direct theft of any user funds" critical impact class.

## Likelihood Explanation

**Medium.**

- The attack is most effective at protocol launch or after a full withdrawal cycle when `rsethSupply` is near zero.
- `updateRSETHPrice()` is permissionless and callable by any EOA or contract.
- `pricePercentageLimit` is `0` by default and requires an explicit admin call to `setPricePercentageLimit()` to activate — a misconfiguration that is easy to miss at deployment.
- The attacker must commit capital (`X` ETH) but recovers it plus the victim's deposit.
- Front-running a known large deposit on Ethereum is straightforward using mempool monitoring.
- The attack is repeatable as long as `rsethSupply` remains low relative to the donated amount.

## Recommendation

1. **Replace raw balance accounting with tracked deposit storage**: Do not use `address(this).balance` or `IERC20.balanceOf(address(this))` as TVL. Maintain explicit `depositedAmount` storage variables incremented only through the protocol's deposit functions and decremented through withdrawals.
2. **Set `pricePercentageLimit` in `initialize()`**: Ensure a non-zero limit (e.g., `1e16` for 1%) is enforced from deployment rather than left as an optional admin action.
3. **Seed the protocol with a non-trivial initial deposit**: Ensure `rsethSupply` is large enough at launch that the cost of the attack exceeds any possible gain (standard ERC-4626 dead-shares mitigation).
4. **Restrict `updateRSETHPrice()` to authorized callers** or add a time-lock/cooldown to prevent rapid repeated manipulation.

## Proof of Concept

```solidity
// Foundry fork test outline
function test_inflationAttack() public {
    // Precondition: fresh protocol, rsethSupply == 0
    // rsETHPrice is initialized to 1e18 when supply is 0

    // Step 1: Attacker deposits 1 wei ETH
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 1}(0, "");
    // rsethSupply == 1 wei

    // Step 2: Attacker donates 1000 ETH directly (no rsETH minted)
    vm.prank(attacker);
    (bool ok,) = address(lrtDepositPool).call{value: 1000 ether}("");
    assertTrue(ok);

    // Step 3: Attacker calls updateRSETHPrice()
    // totalETHInProtocol ≈ 1000 ether + 1 wei
    // rsethSupply = 1 wei
    // newRsETHPrice ≈ 1000e18 * 1e18 / 1 = 1000e36
    // pricePercentageLimit == 0 → isPriceIncreaseOffLimit == false → no revert
    lrtOracle.updateRSETHPrice();
    assertGt(lrtOracle.rsETHPrice(), 1e18);

    // Step 4: Victim deposits 500 ETH with minRSETHAmountExpected = 0
    vm.prank(victim);
    lrtDepositPool.depositETH{value: 500 ether}(0, "");
    // rsethAmountToMint = 500e18 * 1e18 / rsETHPrice ≈ 0
    // victim receives 0 rsETH — full loss of 500 ETH

    assertEq(rsETH.balanceOf(victim), 0);

    // Step 5: Attacker's 1 wei rsETH represents 100% of ~1500 ETH TVL
    // Attacker redeems via withdrawal mechanism and recovers ~1500 ETH
}
```
