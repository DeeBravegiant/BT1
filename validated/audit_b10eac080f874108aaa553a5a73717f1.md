### Title
Key-blob deserialization silently accepts trailing bytes, enabling message malleability across all public signing and key-management APIs - (File: include-internal/cbmpc/internal/core/convert.h)

### Summary
The `coinbase::convert<T>(T& dst, mem_t src)` template used in every public-API key-blob deserialization path does not verify that the entire input buffer was consumed. Trailing bytes are silently ignored. The sibling `coinbase::deser()` function explicitly rejects trailing bytes with the comment *"Strict deserialization: reject trailing bytes — This prevents message malleability where two different byte sequences could deserialize to the same value."* Because `convert` lacks this check, any caller-supplied key blob (ECDSA-2PC, ECDSA-MP, Schnorr-2PC, Schnorr-MP, EdDSA-2PC, EdDSA-MP, TDH2, HD-keyset) with appended bytes is accepted as valid and produces the same parsed cryptographic state as the canonical blob.

### Finding Description

**Root cause — `convert` vs `deser`:**

`deser()` at `include-internal/cbmpc/internal/core/convert.h:254-266` enforces:

```cpp
if (converter.get_offset() != converter.get_size()) {
    return coinbase::error(E_BADARG);   // reject trailing bytes
}
```

`convert<T>(T& dst, mem_t src)` at lines 277-282 of the same file does not:

```cpp
template <typename T>
error_t convert(T& dst, mem_t src) {
  if (src.size < 0 || (src.size && !src.data)) return coinbase::error(E_BADARG);
  converter_t converter(src);
  converter.convert(dst);
  return converter.get_rv();   // ← no trailing-byte check
}
```

**Every public-API key-blob deserialization call uses `coinbase::convert`, not `coinbase::deser`:**

A caller can append arbitrary bytes to any valid key blob. The API accepts the blob, parses the same key material, and proceeds normally. Two distinct byte strings therefore map to the same cryptographic state — the invariant the library's own `deser` was designed to prevent.

**Analog to the UniswapV4 bug class:** In UniswapV4, `getLiquidityForAmounts` consumes only `min(liquidity0, liquidity1)` of the two provided token amounts; the excess of the non-limiting token is silently left in the contract. Here, `convert` consumes only the bytes needed to parse the declared fields; the excess trailing bytes are silently left unconsumed and unvalidated. In both cases the caller provides more input than is consumed, and the surplus is silently discarded rather than rejected.

### Impact Explanation

1. **Message malleability at the key-blob boundary.** `blob_A || garbage` and `blob_A` are both accepted and produce identical parsed keys. This breaks the invariant that `deser` enforces and that the codebase comment explicitly calls out as a security property.

2. **AEAD associated-data confusion.** `SECURE_USAGE.md` recommends protecting stored blobs with application-managed AEAD and binding the raw blob bytes as associated data. An attacker who can append bytes to a stored blob (e.g., in a database without per-record AEAD) causes the AEAD tag to mismatch on the next load, denying the party access to its key share — a targeted denial of signing capability.

3. **`attach_private_scalar` normalization bypass.** The `attach_private_scalar` paths in `ecdsa2pc.cpp:242`, `eddsa2pc.cpp:182`, and `schnorr_mp.cpp:503` parse the "public" blob with `coinbase::convert`. A blob with trailing bytes is accepted as a valid public blob, and the resulting re-attached full blob is silently "normalized" (trailing bytes stripped). The application cannot detect that the input blob was malformed, breaking any integrity audit trail.

4. **Cross-version confusion.** If a future blob version adds fields at the end, an old parser using `convert` will silently accept the new blob, treating the new fields as trailing garbage. This is the exact scenario `deser`'s trailing-byte check was designed to prevent.

### Likelihood Explanation

Every public API function that accepts a `mem_t key_blob` argument is reachable by any caller. The key blobs are opaque byte strings passed back into the API by the integrating application. If the application stores blobs in a location where an attacker can append bytes (e.g., a database row, a file, a network buffer without AEAD), the attacker can create malformed blobs that are accepted. The library explicitly delegates blob protection to the application, but the library should still enforce its own parsing invariants at the API boundary.

### Recommendation

Replace all `coinbase::convert(blob, in)` calls in the public-API key-blob deserialization paths with `coinbase::deser(in, blob)`, which already enforces the trailing-byte check. The `deser` function is already present and used correctly in internal protocol message parsing (e.g., `src/cbmpc/protocol/pve_base.cpp:26`). Applying it consistently at the public API boundary closes the malleability gap without any change to the serialization format.

### Proof of Concept

```cpp
// 1. Generate a valid Schnorr-2PC key blob via DKG (normal API usage)
buf_t key_blob_1, key_blob_2;
coinbase::api::schnorr_2p::dkg(job1, curve_id::secp256k1, key_blob_1);
coinbase::api::schnorr_2p::dkg(job2, curve_id::secp256k1, key_blob_2);

// 2. Append garbage bytes — simulates attacker-controlled storage modification
byte_t garbage[4] = {0xDE, 0xAD, 0xBE, 0xEF};
buf_t malformed = key_blob_1 + mem_t(garbage, sizeof(garbage));

// 3. All public API entry points accept the malformed blob
buf_t pub_key;
error_t rv = coinbase::api::schnorr_2p::get_public_key_compressed(malformed, pub_key);
// rv == SUCCESS  (expected: E_BADARG — trailing bytes should be rejected)
// pub_key is identical to the result from the canonical key_blob_1

// 4. sign() also accepts it — the private scalar is used normally
buf_t sig;
rv = coinbase::api::schnorr_2p::sign(job1, malformed, msg32, sig);
// rv == SUCCESS — a valid BIP340 signature is produced from a malformed blob
```

template <typename... ARGS>
error_t deser(mem_t bin, ARGS&... args) {
  converter_t converter(bin);
  converter.convert(args...);
  error_t rv = converter.get_rv();
  if (rv != SUCCESS) return rv;

  // Strict deserialization: reject trailing bytes
  if (converter.get_offset() != converter.get_size()) {
    return coinbase::error(E_BADARG);
  }

  return SUCCESS;
}

  out.x = blob.x;

TEST(CoreConvert, RejectsTrailingBytes) {
  // Strict deserialization should fail when there are unconsumed trailing bytes.
  // This prevents message malleability where two different byte sequences
  // could deserialize to the same value.
  int original_value = 42;
  buf_t serialized = coinbase::ser(original_value);

  byte_t garbage[4] = {0xDE, 0xAD, 0xBE, 0xEF};
  buf_t with_trailing = serialized + mem_t(garbage, 4);

  int result = 0;
  error_t rv = deser(with_trailing, result);

  EXPECT_NE(rv, SUCCESS);

  EXPECT_OK(deser(serialized, result));
  EXPECT_EQ(result, original_value);
}
