Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Window - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary

`KernelDepositPool` uses a Synthetix-style staking rewards mechanism where `rewardPerToken()` short-circuits when `totalKernelStaked == 0`, but the `updateReward` modifier unconditionally advances `updatedAt`. Any time elapsed with zero stakers consumes the reward budget without distributing it, and no admin or user function can recover those tokens. The contract's own NatSpec explicitly acknowledges this behavior while relying solely on an off-chain operational promise rather than any code-level guard.

## Finding Description

`rewardPerToken()` returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`: [1](#0-0) 

The `updateReward` modifier always advances `updatedAt` to `lastTimeRewardApplicable()`, regardless of whether anyone is staked: [2](#0-1) 

`initiateWithdrawal` immediately decrements `totalKernelStaked`, so the last staker's exit drives it to zero in a single transaction: [3](#0-2) 

When the next staker arrives, `updatedAt` already equals `lastTimeRewardApplicable()`, so `lastTimeRewardApplicable() - updatedAt` evaluates to zero (or near-zero), silently skipping the entire zero-staking interval. The `rewardRate * elapsed_zero_period` worth of tokens remain in the contract with no accounting entry and no recovery path. `notifyRewardAmount` does not roll over stranded rewards from a prior period: [4](#0-3) 

The `notifyRewardAmount` guard (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new period with zero stakers; it does not prevent stakers from exiting after a period has already begun: [5](#0-4) 

The NatSpec at lines 17–22 confirms the behavior and explicitly states the mitigation is purely operational: [6](#0-5) 

## Impact Explanation

Reward tokens allocated for the zero-staking interval are permanently frozen inside `KernelDepositPool`. No user can ever claim them; no admin function recovers or redistributes them. This is a concrete instance of **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation

Any unprivileged staker can call `initiateWithdrawal` at any time. If they are the last staker — whether by organic exit, market conditions, or deliberate coordination — `totalKernelStaked` hits zero and the loss is immediately triggered. No special privileges, flash loans, or external dependencies are required. The scenario is repeatable across every reward period.

## Recommendation

In the `updateReward` modifier, only advance `updatedAt` when there are active stakers, so the unspent reward interval is preserved for the next staker:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

Alternatively, track the "lost" interval explicitly and roll it into the next `notifyRewardAmount` call.

## Proof of Concept

1. Admin calls `notifyRewardAmount` with `rewardRate = R`, `finishAt = T + D` (requires `totalKernelStaked > 0` at call time — satisfied).
2. Alice is the sole staker. At time `T + 1`, she calls `initiateWithdrawal` for her full balance. `totalKernelStaked` drops to 0. `updateReward` fires: `updatedAt` advances to `T + 1`.
3. No one stakes for the interval `[T+1, T+D]`. Every subsequent `updateReward` call advances `updatedAt` while `rewardPerTokenStored` stays flat.
4. At `T + D`, the reward window closes. `R * (D - 1)` worth of reward tokens remain in the contract. No user has a claim to them.
5. Admin calls `notifyRewardAmount` for the next period. The stranded tokens are not rolled over — `rewardRate` is computed only from the newly transferred amount.
6. Foundry test plan: deploy contract, stake 1e18 tokens as Alice, call `notifyRewardAmount`, warp 1 second, call `initiateWithdrawal` for full balance, warp to `finishAt`, assert `rewardsToken.balanceOf(address(pool)) >= R * (D - 1)` and that no user's `earned()` accounts for those tokens.
