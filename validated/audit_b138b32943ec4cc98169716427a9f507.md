### Title
Missing On-Chain Validation of CKD Response for Legacy `AppPublicKey` Variant Allows Single Byzantine Participant to Forge Derived Key Output - (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs cryptographic on-chain verification of the CKD response **only** for the `AppPublicKeyPV` variant. For the legacy `AppPublicKey` variant, the response is accepted with **zero validation**. A single Byzantine attested participant can race a legitimate CKD request and call `respond_ckd` with an arbitrary, attacker-chosen `(big_y, big_c)` pair, delivering a derived key the attacker controls to the requesting user.

---

### Finding Description

In `respond_ckd`, the contract branches on the request's `app_public_key` type:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check whatsoever
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

For `AppPublicKeyPV`, the contract calls `ckd_output_check`, which enforces the pairing equation `e(big_c, g2) = e(big_y, app_pk2) · e(hash_point, public_key)` on-chain. [2](#0-1) 

For the legacy `AppPublicKey` variant, the arm is an empty block — the contract performs **no cryptographic check** on `big_y` or `big_c` before resolving the pending yield and delivering the response to the caller. [3](#0-2) 

This is structurally identical to the Lending analog: just as `testGracePeriod()` accepted a 250% interest rate because no range check existed, `respond_ckd` accepts any CKD output for `AppPublicKey` requests because no pairing check exists.

The `AppPublicKey` variant is still live in production per the contract README and ABI snapshot: [4](#0-3) 

The `respond_ckd` endpoint is callable by any single attested participant: [5](#0-4) 

---

### Impact Explanation

**Critical — confidential key derivation output without required participant authorization.**

The CKD decryption formula for a user holding app secret scalar `a` (where `app_pk1 = a·G1`) is:

```
derived_key = big_c − big_y · a
```

An attacker who submits `big_y = G1·0` (the identity/point-at-infinity) and `big_c = G1·z` for any attacker-chosen scalar `z` causes:

```
derived_key = G1·z − (G1·0)·a = G1·z
```

The user receives `G1·z` as their derived key. The attacker knows `z`, so they know the user's derived private key and can drain any funds held at the corresponding derived address on any foreign chain. The user cannot detect the forgery on-chain because `AppPublicKey` is "privately verifiable" — there is no public pairing check available to the contract or to observers.

This bypasses the threshold requirement: the off-chain MPC protocol requires `t` participants to cooperate to produce a valid CKD output, but a **single** Byzantine attested participant can deliver a forged output on-chain with no threshold cooperation.

---

### Likelihood Explanation

**Medium.** The attacker must be an attested participant (pass `assert_caller_is_attested_participant_and_protocol_active`). In the TEE model this means holding a valid TEE attestation. A software vulnerability in the TEE runtime (not a physical hardware attack) or a compromised node binary would satisfy this. The attacker must also race the legitimate coordinator's `respond_ckd` call for a pending `AppPublicKey` request — a realistic window given that MPC signing rounds take non-trivial time. The `respond` endpoint for signatures provides defense-in-depth via on-chain ECDSA/EdDSA verification; the absence of an equivalent check for `AppPublicKey` CKD is an inconsistency that materially weakens the security posture.

---

### Recommendation

1. **Deprecate `AppPublicKey` in favor of `AppPublicKeyPV`**, which supports on-chain pairing verification. Reject new `AppPublicKey` CKD requests at the contract level.
2. If `AppPublicKey` must remain supported, add a note in the contract that its security relies entirely on TEE integrity with no on-chain fallback, and document this explicitly as a known weaker security tier.
3. Add a unit test that attempts to call `respond_ckd` with a forged `(big_y, big_c)` for an `AppPublicKey` request and asserts it is rejected — analogous to the `respond_ckd_pv__should_reject_invalid_response` test that already exists for `AppPublicKeyPV`. [6](#0-5) 

---

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key: AppPublicKey(pk1)` and attaches 1 yoctoNEAR deposit. Request enters `pending_ckd_requests`.
2. Byzantine attested participant (one node, below threshold) calls `respond_ckd` with:
   - `request` = the pending `CKDRequest`
   - `response.big_y` = compressed encoding of `G1·0` (identity point)
   - `response.big_c` = compressed encoding of `G1·z` for attacker-chosen scalar `z`
3. Contract executes the `AppPublicKey` branch (empty), skips all validation, and calls `resolve_yields_for`, resolving the pending yield with the forged response.
4. User's callback receives `CKDResponse { big_y: G1·0, big_c: G1·z }`.
5. User decrypts: `derived_key = G1·z − G1·0·a = G1·z`. Attacker knows `z` and therefore the user's derived private key.
6. Attacker sweeps funds from the user's derived address on any foreign chain.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn ckd_output_check(
    app_id: &dtos::CkdAppId,
    output: &CKDResponse,
    app_public_key: &dtos::CKDAppPublicKeyPV,
    public_key: &dtos::Bls12381G2PublicKey,
) -> bool {
    let big_c = env::bls12381_p1_decompress(&output.big_c);
    let big_y = env::bls12381_p1_decompress(&output.big_y);
    let pk2 = env::bls12381_p2_decompress(&app_public_key.pk2);
    let pk = env::bls12381_p2_decompress(public_key);
    let hash_point = hash_app_id_with_pk(public_key.as_slice(), app_id.as_ref());

    let pairing_input = [
        big_c.as_slice(),
        MINUS_G2_GENERATOR_UNCOMPRESSED.as_slice(),
        big_y.as_slice(),
        pk2.as_slice(),
        hash_point.as_slice(),
        pk.as_slice(),
    ]
    .concat();
    env::bls12381_pairing_check(&pairing_input)
}

  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
- `domain_id` (integer): identifies the master key to use for deriving the ckd, and must correspond to bls12381.
