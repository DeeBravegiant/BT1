Audit Report

## Title
Shared `dailyMintAmount` Counter Across Independent ETH and LST Deposit Paths Causes Temporary Deposit Freeze - (File: contracts/pools/RSETHPoolV3.sol, contracts/pools/RSETHPoolV3ExternalBridge.sol, contracts/pools/RSETHPoolV3WithNativeChainBridge.sol)

## Summary
In `RSETHPoolV3`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge`, a single `dailyMintAmount` counter is shared across both the native ETH deposit path and all LST token deposit paths. When ETH deposits exhaust the daily limit, all LST token deposits revert with `DailyMintLimitExceeded` for up to 24 hours, and vice versa, even though the two asset classes are entirely independent. Note: `RSETHPoolV2ExternalBridge` is not affected because it only exposes a single ETH deposit path with no LST token deposit function.

## Finding Description
In `RSETHPoolV3.sol`, the `limitDailyMint` modifier at lines 96–125 maintains a single `uint256 public dailyMintAmount` state variable. Both deposit functions apply this modifier:

- `deposit(string referralId)` (ETH path, line 251): `limitDailyMint(msg.value, ETH_IDENTIFIER)`
- `deposit(address token, uint256 amount, string referralId)` (LST path, line 280): `limitDailyMint(amount, token)`

Both paths write to the same `dailyMintAmount` (line 123: `dailyMintAmount += rsETHAmount`). The check at line 119 (`if (dailyMintAmount + rsETHAmount > dailyMintLimit)`) does not distinguish between asset types. The identical pattern exists in `RSETHPoolV3ExternalBridge.sol` (lines 130–159, deposit functions at lines 366–412) and `RSETHPoolV3WithNativeChainBridge.sol` (lines 108–137, deposit functions at lines 282–329).

The daily counter resets only when `currentDay > lastMintDay` (line 113), meaning a block caused by one asset type persists for the remainder of the 24-hour window. No privileged action is required to trigger this; any depositor making a sufficiently large ETH deposit can exhaust the shared limit and block all LST depositors until the next day.

## Impact Explanation
**Medium — Temporary freezing of funds.** When the shared `dailyMintAmount` counter is exhausted by one deposit path, all other deposit paths are blocked for up to 24 hours. Users holding LST tokens (e.g., wstETH) who wish to deposit are denied service even though their asset is entirely independent of ETH deposits. Depositors cannot enter the protocol during this window, cannot obtain wrsETH, and are denied yield accrual that would have started from their intended deposit time. This constitutes a temporary freezing of user funds from the protocol's perspective.

## Likelihood Explanation
**Medium.** The daily mint limit is a finite cap. Any single large ETH depositor — or organic aggregate ETH deposit volume — can exhaust the limit and block LST token depositors for the remainder of the 24-hour window. No special privilege is required; the trigger is a normal, public `deposit()` call. The pools are deployed on multiple L2 chains, increasing the probability of the limit being hit organically.

## Recommendation
Replace the single shared `dailyMintAmount` / `dailyMintLimit` / `lastMintDay` triple with per-asset mappings in all three affected contracts:

```solidity
mapping(address token => uint256) public dailyMintAmount;
mapping(address token => uint256) public dailyMintLimit;
mapping(address token => uint256) public lastMintDay;
```

Update `limitDailyMint` to key all reads and writes on the `token` parameter. This ensures ETH deposits filling the ETH daily limit do not affect LST token deposit paths, and vice versa.

## Proof of Concept
1. Deploy `RSETHPoolV3` with `dailyMintLimit = L` (e.g., 1000 wrsETH) and both ETH and wstETH deposit paths enabled.
2. Alice calls `deposit(referralId)` with enough ETH to mint exactly `L` wrsETH. `dailyMintAmount` is now `L`.
3. Bob calls `deposit(wstETH, amount, referralId)`. The `limitDailyMint` modifier computes `rsETHAmount` for Bob's wstETH, then checks `dailyMintAmount + rsETHAmount > dailyMintLimit` → `L + X > L` → reverts with `DailyMintLimitExceeded`.
4. Bob's deposit is blocked until `currentDay > lastMintDay` resets `dailyMintAmount` to zero (up to 24 hours later).
5. Foundry test: fork the target L2, set `dailyMintLimit` to a small value, call `deposit{value: ...}` to exhaust it, then call `deposit(wstETH, ...)` and assert it reverts with `DailyMintLimitExceeded`.
