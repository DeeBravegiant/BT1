### Title
Sub-Transaction Treasury Donation Included in `produced` Accounting But Not Credited to Treasury Pot — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/UTxO.hs`)

---

### Summary

In the Dijkstra era, `dijkstraSubTxProducedValue` includes the sub-transaction's `treasuryDonation` field in the "produced" value used for the preservation-of-value check. However, the state-update path for sub-transactions (`updateUTxOStateNoFees`) does not increment `utxosDonation`. The preservation-of-value check therefore forces UTxO inputs to cover the declared donation amount, but that amount is never credited to the treasury pot — it is silently destroyed. This is the direct Cardano analog of the fee-on-transfer accounting mismatch: the accounting layer assumes a value flow that the execution layer never performs.

---

### Finding Description

**Step 1 — Sub-transaction "produced" value includes treasury donation**

`dijkstraSubTxProducedValue` is the function used to compute the produced value for a `SubTx`-level transaction body:

```haskell
dijkstraSubTxProducedValue pp isRegPoolId txBody =
  sumAllValue (txBody ^. outputsTxBodyL)
    <> inject (getTotalDepositsTxBody pp isRegPoolId txBody
               <> txBody ^. treasuryDonationTxBodyL)   -- ← donation included
    <> burnedMultiAssets txBody
``` [1](#0-0) 

This value is aggregated into the top-level batch produced value via `dijkstraProducedValue`:

```haskell
dijkstraProducedValue pp isRegPoolId txBody =
  conwayProducedValue pp isRegPoolId txBody
    <> foldMap'
         (getProducedValue pp isRegPoolId . view bodyTxL)
         (txBody ^. subTransactionsTxBodyL)
``` [2](#0-1) 

**Step 2 — The preservation-of-value check enforces this accounting**

The Dijkstra UTXO transition rule runs the standard conservation check against the aggregated produced value:

```haskell
{- consumed pp utxo₀ txb = produced pp certState txb -}
runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
``` [3](#0-2) 

Because `produced` now includes the sub-tx treasury donation `D`, the consumed side (UTxO inputs + withdrawals + refunds) must also cover `D`. The check therefore forces the sub-tx's UTxO inputs to be larger by exactly `D`.

**Step 3 — The state-update path for sub-transactions never credits the treasury**

`updateUTxOStateNoFees` is the function explicitly designated for sub-transactions ("fees are not applicable"):

```haskell
-- | Like 'updateUTxOState', but does not collect fees. This is used for sub-transactions
-- where fees are not applicable.
updateUTxOStateNoFees ...
  pure $!
    UTxOState
      { utxosUtxo      = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees      = utxosFees          -- unchanged
      , utxosGovState  = govState
      , utxosInstantStake = ...
      , utxosDonation  = utxosDonation      -- ← unchanged, donation never credited
      }
``` [4](#0-3) 

`utxosDonation` is left unchanged. The `D` lovelace consumed from UTxO inputs to satisfy the conservation check is never deposited into the treasury pot.

**Step 4 — Sub-transaction bodies expose the treasury donation field**

`DijkstraSubTxBodyRaw` carries `dstbrTreasuryDonation :: !Coin`, and the CDDL schema encodes it as field `22` (`positive_coin`) in `sub_transaction_body`: [5](#0-4) [6](#0-5) 

Any unprivileged transaction author can set this field to an arbitrary positive value.

---

### Impact Explanation

**Critical — Direct destruction of ADA through an invalid ledger state transition.**

For each sub-transaction carrying a treasury donation `D`:
- The preservation-of-value check forces the sub-tx's UTxO inputs to be larger by `D`.
- The state update removes those inputs from the UTxO.
- `utxosDonation` is never incremented.
- `D` lovelace is permanently removed from circulation without entering any pot (treasury, fees, deposits, or outputs).

This violates the full ADA preservation invariant and constitutes direct, irreversible destruction of ADA on every such transaction.

---

### Likelihood Explanation

The `dstbrTreasuryDonation` field is a standard, serializable field in the sub-transaction body, reachable by any transaction author in the Dijkstra era. No privileged role, governance majority, or key compromise is required. The attacker only needs to submit a valid top-level Dijkstra transaction containing a sub-transaction with a non-zero `treasuryDonation`.

---

### Recommendation

Two consistent fixes exist:

1. **Remove treasury donation from sub-transaction produced value**: If sub-transactions are not intended to donate to the treasury, remove `txBody ^. treasuryDonationTxBodyL` from `dijkstraSubTxProducedValue` and reject sub-transactions that set this field to a non-zero value.

2. **Credit the treasury in the sub-transaction state update**: If sub-transactions are intended to donate to the treasury, update `updateUTxOStateNoFees` (or the Dijkstra-specific sub-transaction UTXOS rule) to increment `utxosDonation` by `txBody ^. treasuryDonationTxBodyL`, mirroring the top-level `updateUTxOState` behaviour.

---

### Proof of Concept

```
1. Construct a DijkstraEra top-level transaction T_top with:
     - One or more UTxO inputs sufficient to cover outputs + fee + sub-tx donation
     - One sub-transaction T_sub with:
         dstbrTreasuryDonation = 1_000_000  (1 ADA)
         dstbrSpendInputs      = { utxo_input_covering_1_ADA }
         dstbrOutputs          = []         (no outputs; all value "donated")

2. The preservation-of-value check computes:
     produced(T_top) + produced(T_sub)
       = ... + (0 outputs + 0 deposits + 1_000_000 donation + 0 burned)
       = ... + 1_000_000

   consumed(T_sub) = balance(utxo_input_covering_1_ADA) = 1_000_000
   → check passes.

3. State update for T_sub via updateUTxOStateNoFees:
     utxosUtxo      ← removes utxo_input_covering_1_ADA
     utxosDonation  ← unchanged (still 0)

4. Result: 1 ADA has been removed from the UTxO and credited to no pot.
   Total ADA in the ledger has decreased by 1 ADA.
   Repeating this with larger donations destroys proportionally more ADA.
```

-- | Like 'updateUTxOState', but does not collect fees. This is used for sub-transactions
-- where fees are not applicable.
updateUTxOStateNoFees ::
  ( EraTxBody era
  , EraStake era
  , EraCertState era
  , Monad m
  ) =>
  PParams era ->
  UTxOState era ->
  TxBody l era ->
  CertState era ->
  GovState era ->
  (Coin -> m ()) ->
  (UTxO era -> UTxO era -> m ()) ->
  m (UTxOState era)
updateUTxOStateNoFees pp utxos txBody certState govState depositChangeEvent txUtxODiffEvent = do
  let UTxOState {utxosUtxo, utxosDeposited, utxosFees, utxosDonation} = utxos
      UTxO utxo = utxosUtxo
      !utxoAdd = txouts txBody -- These will be inserted into the UTxO
      {- utxoDel  = txins txb ◁ utxo -}
      !(utxoWithout, utxoDel) = extractKeys utxo (txBody ^. inputsTxBodyL)
      {- newUTxO = (txins txb ⋪ utxo) ∪ outs txb -}
      newUTxO = utxoWithout `Map.union` unUTxO utxoAdd
      deletedUTxO = UTxO utxoDel
      totalRefunds = certsTotalRefundsTxBody pp certState txBody
      totalDeposits = certsTotalDepositsTxBody pp certState txBody
      depositChange = totalDeposits <-> totalRefunds
  depositChangeEvent depositChange
  txUtxODiffEvent deletedUTxO utxoAdd
  pure $!
    UTxOState
      { utxosUtxo = UTxO newUTxO
      , utxosDeposited = utxosDeposited <> depositChange
      , utxosFees = utxosFees
      , utxosGovState = govState
      , utxosInstantStake =
          deleteInstantStake deletedUTxO (addInstantStake utxoAdd (utxos ^. instantStakeL))
      , utxosDonation = utxosDonation
      }
