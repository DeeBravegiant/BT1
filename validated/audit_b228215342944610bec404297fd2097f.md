Audit Report

## Title
ETH Deposit Limit Check Ignores Incoming Deposit Amount, Allowing Unlimited Over-Deposit Beyond Protocol Cap - (File: contracts/LRTDepositPool.sol)

## Summary

`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric guard: for ERC20 assets it correctly tests `totalAssetDeposits + amount > limit`, but for ETH it silently discards `amount` and only tests whether the pre-deposit total already exceeds the cap. Any unprivileged caller can therefore push ETH holdings arbitrarily above the configured limit in a single `depositETH` call, causing unbounded rsETH over-minting and exposing the protocol to insolvency under any EigenLayer slashing event at the inflated scale.

## Finding Description

`_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682) contains the asymmetric branch:

```solidity
// contracts/LRTDepositPool.sol L676-L682
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount never used
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct for ERC20
}
``` [1](#0-0) 

For ETH, the `amount` parameter is never referenced. The guard only fires when the cap is **already** exceeded before the deposit. Whenever `totalAssetDeposits <= depositLimit` — the normal operating state — the function returns `false` regardless of how large `amount` is.

The call path is fully public and unprivileged:

```
depositETH{value: X}(minRSETHAmountExpected, referralId)   [payable, nonReentrant, whenNotPaused]
  └─ _beforeDeposit(ETH_TOKEN, msg.value, ...)
       └─ _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)
            └─ returns false when totalAssetDeposits <= depositLimit  ← bug
  └─ _mintRsETH(rsethAmountToMint)   ← rsETH minted for full msg.value
``` [2](#0-1) [3](#0-2) 

The `_beforeDeposit` function at line 661 calls `_checkIfDepositAmountExceedesCurrentLimit` and reverts with `MaximumDepositLimitReached` only if it returns `true` — which it never does for ETH while the cap has not yet been breached.

## Impact Explanation

**Critical — Protocol insolvency.**

The deposit limit exists to cap the protocol's EigenLayer exposure. When bypassed, `_mintRsETH` mints rsETH proportional to the full `msg.value` at the current oracle rate. [4](#0-3) 

`getTotalAssetDeposits` will then exceed `depositLimitByAsset(ETH)` by the full over-deposit amount, and `getAssetCurrentLimit` returns 0 only after the fact — it cannot undo already-minted rsETH. The rsETH supply is inflated beyond what the protocol's risk parameters were designed to support. Under any EigenLayer slashing event at the over-exposed scale, the protocol cannot make all rsETH holders whole, constituting protocol insolvency.

## Likelihood Explanation

**High.** The entry point `depositETH` is public, payable, and requires no special role. The precondition `totalAssetDeposits <= depositLimit` is the normal operating state of the protocol for most of its lifetime and is trivially readable on-chain via `getTotalAssetDeposits`. No front-running, governance capture, or oracle manipulation is required — a single large ETH deposit suffices.

## Recommendation

Apply the same prospective check used for ERC20 tokens:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This unifies the ETH and ERC20 paths and ensures a deposit that would breach the limit is rejected before any state change or minting occurs.

## Proof of Concept

```
depositLimit              = 1_000 ETH
totalAssetDeposits before = 999 ETH

Attacker calls: depositETH{value: 5_000 ETH}(0, "")

_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 5_000 ETH):
  totalAssetDeposits = 999 ETH
  branch ETH → return (999 > 1_000) = false   ← NOT blocked

_mintRsETH mints rsETH for 5_000 ETH at current oracle rate.

totalAssetDeposits after = 5_999 ETH  (≈ 6× the intended cap)
```

**Foundry test plan:**
1. Deploy `LRTDepositPool` with `depositLimitByAsset(ETH) = 1_000 ether`.
2. Seed the pool so `getTotalAssetDeposits(ETH) = 999 ether`.
3. Call `depositETH{value: 5_000 ether}(0, "")` from an unprivileged address.
4. Assert the call succeeds (no revert).
5. Assert `getTotalAssetDeposits(ETH) == 5_999 ether` — 5× over the cap.
6. Assert rsETH balance of the attacker equals the minted amount for 5_000 ETH.
