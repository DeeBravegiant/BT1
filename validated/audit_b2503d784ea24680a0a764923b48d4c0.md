Audit Report

## Title
Unguarded `sendETHToL1ViaBridge` in `LineaMessenger` Allows Any Caller to Exhaust Bridge Rate Limit, Temporarily Freezing Pool ETH Bridging — (`contracts/bridges/LineaMessenger.sol`)

## Summary

`LineaMessenger.sendETHToL1ViaBridge` is a fully public `external payable` function with no access control. Any caller can supply ETH to consume the Linea bridge's per-period rate limit (`limitInWei`). Once exhausted, the `BRIDGER_ROLE`'s `bridgeAssetsViaNativeBridge` call reverts because the inner `ILineaMessageService.sendMessage` call exceeds the cap, temporarily preventing the pool's accumulated ETH from being bridged to L1 until the bridge period resets.

## Finding Description

`LineaMessenger` inherits from `Recoverable`, which only gates token/ETH recovery helpers behind `DEFAULT_ADMIN_ROLE`. The bridging function itself carries no modifier:

```solidity
// contracts/bridges/LineaMessenger.sol L31
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value) external payable nonReentrant {
``` [1](#0-0) 

Any EOA or contract can call this with arbitrary ETH and an attacker-controlled `target` on L1. The Linea bridge enforces a rolling per-period cap, exposed via:

```solidity
function limitInWei() external view returns (uint256);
function currentPeriodAmountInWei() external view returns (uint256);
``` [2](#0-1) 

When `BRIDGER_ROLE` calls `bridgeAssetsViaNativeBridge`, the pool forwards its full `ethBalanceMinusFees` through the same messenger:

```solidity
IL2Messenger(messenger).sendETHToL1ViaBridge{ value: ethBalanceMinusFees }(
    l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees
);
``` [3](#0-2) [4](#0-3) 

If an attacker has already consumed the remaining capacity via a direct call to `sendETHToL1ViaBridge`, the `sendMessage` call inside the messenger reverts, bubbling up through `bridgeAssetsViaNativeBridge`. The attacker's ETH is not lost — it is bridged to an L1 address they control. The only cost is gas and temporary capital lockup.

`Recoverable` provides no relevant protection here; it only guards `recoverTokens` and `recoverETH` behind `DEFAULT_ADMIN_ROLE`. [5](#0-4) 

## Impact Explanation

**Medium — Temporary Freezing of Funds.** The pool's accumulated ETH cannot be bridged to L1 for the remainder of the current bridge period. Withdrawals and rebalancing that depend on L1 liquidity are delayed. The funds are not permanently lost, but the protocol's bridging operation is blocked for the duration of the rate-limit window. This matches the allowed impact class "Temporary freezing of funds."

## Likelihood Explanation

- No privileged role is required; the function is fully public and callable by any EOA or contract.
- The attacker's ETH is not lost — it is bridged to their own L1 address. The only cost is gas and temporary capital lockup.
- The attack does not require mempool front-running. The attacker can exhaust capacity at any time before `BRIDGER_ROLE` acts.
- The attack is repeatable each bridge period.
- Linea's centralized sequencer prevents indefinite block-stuffing, so the freeze is bounded to the current rate-limit period, not permanent. This limits severity to Medium rather than High.

## Recommendation

Add an access-control guard to `sendETHToL1ViaBridge` in `LineaMessenger`, restricting callers to the pool contract(s) that legitimately use it (e.g., an immutable `allowedCaller` address set at construction, or a `CALLER_ROLE`). The same open signature exists in `ArbitrumMessenger`, `BaseMessenger`, `OptimismMessenger`, `ScrollMessenger`, and `UnichainMessenger` and should be reviewed for the same pattern.



## Proof of Concept

```solidity
// Fork test on Linea mainnet (or local fork with Linea bridge mock)
function test_exhaustBridgeCapacity() external {
    ILineaMessageService bridge = ILineaMessageService(LINEA_BRIDGE);
    LineaMessenger messenger = LineaMessenger(MESSENGER_ADDRESS);

    uint256 remaining = bridge.limitInWei() - bridge.currentPeriodAmountInWei();

    // Attacker exhausts remaining capacity; ETH goes to attacker's L1 address
    vm.deal(attacker, remaining);
    vm.prank(attacker);
    messenger.sendETHToL1ViaBridge{value: remaining}(
        LINEA_BRIDGE, attackerL1, remaining
    );

    // BRIDGER_ROLE's call now reverts because rate limit is exhausted
    vm.prank(bridger);
    vm.expectRevert(); // Linea bridge: rate limit exceeded
    pool.bridgeAssetsViaNativeBridge();
}
```
