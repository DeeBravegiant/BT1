Audit Report

## Title
Missing Decimal Normalization in Token Deposit Swap Calculation Causes Massive rsETH Underpayment - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The `viewSwapRsETHAmountAndFee(uint256 amount, address token)` overload in `RSETHPoolV3`, `RSETHPool`, `RSETHPoolNoWrapper`, `RSETHPoolV3ExternalBridge`, and `RSETHPoolV3WithNativeChainBridge` computes rsETH output by multiplying the raw token amount (in native units) by a Chainlink-normalized 18-decimal rate, without first scaling the amount to 18-decimal precision. For any supported token with fewer than 18 decimals, the user receives orders of magnitude fewer rsETH tokens than entitled, and the deposited collateral is permanently stranded in the pool.

## Finding Description
The ETH deposit path correctly scales the amount to 18-decimal precision before dividing by the rsETH/ETH rate:

```solidity
// RSETHPoolV3.sol L307 — correct
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

The token deposit path omits this scaling:

```solidity
// RSETHPoolV3.sol L334 — missing normalization
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink answer to 18 decimals:

```solidity
// ChainlinkOracleForRSETHPoolCollateral.sol L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

This means `tokenToETHRate` represents the price of **one whole token** (e.g., 1 WBTC = 20e18). But `amountAfterFee` is in native token units (e.g., 1e8 satoshis for 1 WBTC). The formula treats `amountAfterFee` as if it were already 18-decimal, which it is not for sub-18-decimal tokens.

**Exploit flow:**
1. Admin calls `addSupportedToken(wbtc, chainlinkOracle)` — a legitimate use of the protocol's intended multi-token functionality.
2. User calls `deposit(wbtc, 1e8, "ref")` — depositing 1 WBTC.
3. `viewSwapRsETHAmountAndFee(1e8, wbtc)` computes: `rsETHAmount = 1e8 * 20e18 / 1.05e18 ≈ 1.9e9`.
4. User receives `1.9e9` wrsETH (≈ 0.000000002 rsETH in 18-decimal terms).
5. User's 1 WBTC (~20 ETH) is permanently locked; the user holds no meaningful rsETH to redeem it.

No existing guard checks token decimals or validates the output magnitude. The `limitDailyMint` modifier also consumes the same underflowed `rsETHAmount`, so the daily cap is never meaningfully reached for sub-18-decimal deposits.

The identical bug exists in all five pool variants:
- `contracts/pools/RSETHPoolV3.sol` L334
- `contracts/pools/RSETHPool.sol` L346
- `contracts/pools/RSETHPoolNoWrapper.sol` L311
- `contracts/pools/RSETHPoolV3ExternalBridge.sol` L452
- `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol` L370

## Impact Explanation
**Critical — Permanent freezing of funds / Direct theft of user funds.**

For 1 WBTC (8 decimals, price = 20 ETH, rsETH rate = 1.05 ETH):
- Actual output: `1e8 * 20e18 / 1.05e18 ≈ 1.9e9` (essentially zero rsETH)
- Expected output: `1e18 * 20e18 / 1.05e18 ≈ 19.05e18` rsETH

The user receives 10^10× fewer rsETH tokens than owed. Since the user holds no meaningful rsETH, they cannot redeem the deposited WBTC through any user-accessible path. The collateral is permanently stranded in the pool — a total loss of the deposited asset, matching the Critical "permanent freezing of funds" and "direct theft of user funds" impact categories.

## Likelihood Explanation
The contracts explicitly expose `deposit(address token, uint256 amount, string referralId)` and maintain a `supportedTokenOracle` mapping for arbitrary ERC20 tokens. `ChainlinkOracleForRSETHPoolCollateral` is purpose-built to wrap any Chainlink feed. Adding a sub-18-decimal token (WBTC, USDC, USDT) is a legitimate and intended administrative action — not an attack. Once any such token is added, every depositor of that token is affected without any further privileged action. The bug is triggered by a normal, unprivileged `deposit` call.

## Recommendation
Normalize `amountAfterFee` to 18-decimal precision before applying the rate:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 amountAfterFeeNormalized = amountAfterFee * 1e18 / (10 ** uint256(tokenDecimals));
rsETHAmount = amountAfterFeeNormalized * tokenToETHRate / rsETHToETHrate;
```

Apply this fix consistently across all five affected pool contracts.

## Proof of Concept
**Foundry unit test plan (no fork required):**

```solidity
function test_wbtcDepositUnderpayment() public {
    // Deploy mock WBTC (8 decimals), mock Chainlink oracle returning 20e8 (20 ETH, 8-decimal feed)
    // Deploy ChainlinkOracleForRSETHPoolCollateral wrapping the mock feed
    // Deploy RSETHPoolV3, add WBTC as supported token
    // Set rsETH/ETH rate to 1.05e18

    // User deposits 1 WBTC (1e8 units)
    uint256 amount = 1e8;
    (uint256 rsETHAmount,) = pool.viewSwapRsETHAmountAndFee(amount, wbtc);

    // Actual: ~1.9e9 (essentially zero)
    // Expected: ~19.05e18
    assertApproxEqRel(rsETHAmount, 19.05e18, 0.01e18); // FAILS — proves the bug
}
```
