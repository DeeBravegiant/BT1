### Title
Single Byzantine Participant Can Forge CKD Response for `AppPublicKey` Requests, Bypassing Threshold Requirement - (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs cryptographic output validation only for `AppPublicKeyPV`-variant CKD requests. For the `AppPublicKey` variant, the response is accepted with **no cryptographic verification**, allowing a single attested-but-Byzantine participant (strictly below the signing threshold) to submit an arbitrary forged `CKDResponse` that the contract immediately delivers to the waiting user.

---

### Finding Description

`respond_ckd` is the node-facing callback through which MPC participants deliver the result of a Confidential Key Derivation (CKD) operation. After confirming the caller is an attested participant, the function branches on the request's `app_public_key` variant:

```rust
// crates/contract/src/lib.rs  lines 675-682
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO validation
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

For `AppPublicKeyPV`, `ckd_output_check` cryptographically verifies that the submitted response is consistent with the master public key and the user's app public key. For `AppPublicKey`, the branch is an empty no-op. The function then unconditionally resolves all queued yield-resume promises with the (potentially forged) response bytes:

```rust
// crates/contract/src/lib.rs  lines 684-688
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
```

Because `resolve_yields_for` drains the entire fan-out queue for that request key in one call, a single forged `respond_ckd` call permanently resolves the pending request and delivers the attacker-chosen bytes to every waiting caller.

The analog to the Arrakis `setModule` vulnerability is direct: just as a semi-trusted executor could supply malicious `payloads_` during a module transition and drain funds before any post-condition check, a single semi-trusted (attested) MPC participant can supply a malicious `CKDResponse` during the CKD response phase and deliver forged key material before any threshold agreement is reached.

---

### Impact Explanation

A user who calls `request_app_private_key` with the `AppPublicKey` variant receives a derived key that is entirely attacker-controlled. The attacker knows the corresponding private key. Any data the user subsequently encrypts to that derived key is immediately readable by the attacker; any signature the user believes is tied to a threshold-protected secret is instead tied to a key the attacker holds outright. This constitutes **unauthorized confidential key derivation output without the required participant authorization** and a **bypass of the threshold-signature requirement** for CKD operations.

---

### Likelihood Explanation

Any single attested MPC participant (one node out of N, well below the threshold T) can execute this attack unilaterally. The attacker does not need to collude with other participants, compromise TEE hardware, or perform any off-chain action beyond submitting a crafted `respond_ckd` transaction. The `AppPublicKey` variant is a documented, user-facing API path (`request_app_private_key` accepts it), so real users will exercise it in production.

---

### Recommendation

Apply the same cryptographic output check to `AppPublicKey` requests that is already applied to `AppPublicKeyPV` requests. If a suitable on-chain verifiable check cannot be constructed for the `AppPublicKey` variant (because there is no user-supplied commitment to verify against), the contract should require that only the threshold-agreed response is accepted — for example by requiring a quorum of matching `respond_ckd` calls before resolving the yield, mirroring how `respond` for signatures works in practice at the node layer.

At minimum, the empty match arm must not silently pass:

```rust
dtos::CKDAppPublicKey::AppPublicKey(_) => {
    // Must add equivalent cryptographic verification here,
    // or reject this variant at the contract level until one exists.
}
```

---

### Proof of Concept

1. User calls `request_app_private_key` with `app_public_key = AppPublicKey(pk_user)` and attaches the required deposit. The contract enqueues a yield-resume promise and stores the `CKDRequest` in `pending_ckd_requests`.

2. A single Byzantine attested participant (account `evil.near`) constructs an arbitrary `CKDResponse` containing a key pair it controls: `(forged_app_private_key, forged_app_public_key)`.

3. `evil.near` calls `respond_ckd(request, forged_response)`. The contract:
   - Confirms `evil.near` is an attested participant ✓
   - Enters the `AppPublicKey(_) => {}` branch — **no validation performed**
   - Calls `resolve_yields_for`, which resumes the user's yield with `forged_response`

4. The user's `request_app_private_key` call returns `forged_app_public_key`. The user believes this is their threshold-protected derived key. The attacker holds the corresponding `forged_app_private_key` and can decrypt all data the user encrypts to it.

5. No other participant needs to be involved; the threshold is completely bypassed for this request type. [1](#0-0) [2](#0-1) [3](#0-2)

    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
