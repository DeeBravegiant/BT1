Audit Report

## Title
`LRTDepositPool::getAssetCurrentLimit()` Ignores RSETH Daily Mint Limit, Causing Deposit Revert — (`contracts/LRTDepositPool.sol`)

## Summary
`getAssetCurrentLimit()` computes remaining deposit capacity solely from the per-asset deposit cap in `lrtConfig`, without consulting the separate daily rsETH mint limit enforced by `RSETH::checkDailyMintLimit`. When `maxMintAmountPerDay` is configured and `currentPeriodMintedAmount` is near that cap, the function returns a non-zero value that cannot actually be deposited, causing any deposit attempt for that amount to revert inside `RSETH::mint()` with `DailyMintLimitExceeded`.

## Finding Description
`getAssetCurrentLimit()` at lines 402–409 of `contracts/LRTDepositPool.sol` returns `lrtConfig.depositLimitByAsset(asset) - getTotalAssetDeposits(asset)`, consulting only the per-asset cap:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) {
        return 0;
    }
    return lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;
}
``` [1](#0-0) 

`depositAsset()` and `depositETH()` call `_beforeDeposit()`, which validates only the per-asset cap via `_checkIfDepositAmountExceedesCurrentLimit()`: [2](#0-1) 

After passing that check, `_mintRsETH()` calls `IRSETH(rsethToken).mint()`: [3](#0-2) 

`RSETH::mint()` applies the `checkDailyMintLimit` modifier, which reverts if `currentPeriodMintedAmount + amount > maxMintAmountPerDay`: [4](#0-3) 

`RSETH` exposes `remainingDailyMintLimit()` as a view function that correctly accounts for period resets: [5](#0-4) 

Neither `getAssetCurrentLimit()` nor `_beforeDeposit()` consults this value. The two limits are independently configurable and can diverge, making the advertised deposit capacity incorrect whenever the daily mint cap is the binding constraint.

## Impact Explanation
The contract fails to deliver the deposit it advertises as available. A user or integrator who queries `getAssetCurrentLimit()` and submits a deposit for the returned amount receives a revert at `RSETH::mint()`, wasting gas. No funds are lost or permanently frozen; the condition resolves when the 24-hour period resets. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
Requires `maxMintAmountPerDay` to be set to a non-zero value (callable by `onlyLRTManager` via `setMaxMintAmountPerDay`) and `currentPeriodMintedAmount` to be near that cap while the per-asset deposit limit still has remaining capacity. Both limits are independently configurable and the scenario is realistic during high-volume deposit periods. Any unprivileged depositor can trigger the revert by calling `depositAsset()` or `depositETH()` with the value returned by `getAssetCurrentLimit()`.

## Recommendation
`getAssetCurrentLimit()` should also factor in the remaining rsETH daily mint capacity, converting it to asset units via the oracle and returning the minimum of the two limits:

```solidity
function getAssetCurrentLimit(address asset) public view override returns (uint256) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)) return 0;
    uint256 assetLimit = lrtConfig.depositLimitByAsset(asset) - totalAssetDeposits;

    uint256 rsethRemaining = IRSETH(lrtConfig.rsETH()).remainingDailyMintLimit();
    if (rsethRemaining == 0 && IRSETH(lrtConfig.rsETH()).maxMintAmountPerDay() > 0) return 0;
    if (IRSETH(lrtConfig.rsETH()).maxMintAmountPerDay() == 0) return assetLimit;

    // Convert rsETH remaining to asset units: assetAmount = rsethAmount * rsETHPrice / assetPrice
    address lrtOracleAddress = lrtConfig.getContract(LRTConstants.LRT_ORACLE);
    ILRTOracle lrtOracle = ILRTOracle(lrtOracleAddress);
    uint256 assetEquivalent = rsethRemaining * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);

    return assetLimit < assetEquivalent ? assetLimit : assetEquivalent;
}
```

`_beforeDeposit()` should similarly check the daily mint limit before proceeding, to prevent the revert from propagating to `RSETH::mint()`.

## Proof of Concept
1. `maxMintAmountPerDay` in `RSETH` is set to `1000e18`; `currentPeriodMintedAmount` is `990e18` (10 rsETH remaining in the current period).
2. The per-asset deposit limit for stETH has `500e18` stETH of remaining capacity.
3. User calls `getAssetCurrentLimit(stETH)` → returns `500e18` (daily mint limit not consulted).
4. User calls `depositAsset(stETH, 500e18, 0, "")`.
5. `_beforeDeposit` passes: `totalAssetDeposits + 500e18 ≤ depositLimitByAsset(stETH)`.
6. `_mintRsETH(rsethAmountToMint)` calls `RSETH.mint(msg.sender, rsethAmountToMint)`.
7. `checkDailyMintLimit`: `990e18 + rsethAmountToMint > 1000e18` → reverts with `DailyMintLimitExceeded`.
8. Deposit fails despite `getAssetCurrentLimit()` indicating it was safe.

Foundry test plan: deploy `RSETH` and `LRTDepositPool` with mocks; call `setMaxMintAmountPerDay(1000e18)`; simulate prior mints totalling `990e18` to set `currentPeriodMintedAmount`; assert `getAssetCurrentLimit(stETH) > 0`; call `depositAsset` and assert revert with `DailyMintLimitExceeded`.

        uint256 effectiveDailyMintAmount = (block.timestamp >= periodStartTime + 1 days) ? 0 : currentPeriodMintedAmount;
