### Title
Missing Shard-ID Consistency Check in `set_state_header` Allows Malicious Peer to Install Wrong Shard State Under Shard 0's DB Key - (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever verifying that the embedded `ShardChunk.shard_id()` matches the caller-supplied `shard_id`. The `verify_path` Merkle check used to validate the chunk's inclusion in the block is position-agnostic: it only confirms the chunk hash is *somewhere* in the `chunk_headers_root` tree, not that it is at the index corresponding to the requested shard. A malicious peer can therefore supply shard 1's chunk body as the header for shard 0, pass all validations, and cause `set_state_finalize` to write a `ChunkExtra` with shard 1's state root under shard 0's `(shard_uid, block_hash)` DB key.

---

### Finding Description

**Root cause — missing shard-id equality check in `set_state_header`:**

`set_state_header` performs five validation steps before persisting the header:

1. `validate_chunk_proofs` — verifies the chunk's internal hash, tx-root, and receipts-root. It does **not** inspect `chunk.shard_id()`. [1](#0-0) 

2. `verify_path` against `sync_prev_block_header.chunk_headers_root()` — confirms `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is in the Merkle tree. It does **not** check the leaf's position (shard index). [2](#0-1) 

3. `verify_path` for `prev_chunk` — same position-agnostic check. [3](#0-2) 

4. Receipt-proof validation — uses the caller-supplied `shard_id` (0) to compute `ReceiptList(shard_id, receipts)`, so the attacker simply provides valid shard-0 receipt proofs (publicly derivable from chain data). [4](#0-3) 

5. `validate_state_root_node` — checks the node is consistent with `chunk.prev_state_root()`, which is shard 1's root. Passes trivially. [5](#0-4) 

After all checks pass, the header is stored under `StateHeaderKey(shard_id=0, sync_hash)`: [6](#0-5) 

**Why `verify_path` does not prevent cross-shard confusion:**

`verify_path` reduces to `compute_root_from_path(path, item_hash) == root`. For a 2-shard block with leaf hashes H0 (shard 0) and H1 (shard 1) and root R = combine_hash(H0, H1):

- Shard-1 proof: `[{direction: Left, hash: H0}]` → `combine_hash(H0, H1) = R` ✓

The same root is produced regardless of which leaf's proof is supplied. There is no `verify_path_with_index` call (which *does* check position) here. [7](#0-6) [8](#0-7) 

**Propagation into `set_state_finalize`:**

`Chain::set_state_finalize` retrieves the stored header and calls `chain_update.set_state_finalize(shard_id=0, sync_hash, shard_state_header)`: [9](#0-8) 

Inside `ChainUpdate::set_state_finalize`, `shard_uid` is derived from the caller-supplied `shard_id=0`, not from `chunk.shard_id()`: [10](#0-9) 

`apply_chunk` is then called with `shard_uid=0` but `chunk_header.prev_state_root()` from shard 1's chunk. Because the state parts downloaded earlier were validated against shard 1's `prev_state_root` and applied via `apply_state_part(shard_id=0, state_root=shard_1_root, ...)`, shard 1's trie data is present under shard 0's storage. The apply succeeds and the resulting `ChunkExtra` (containing shard 1's post-application state root) is written under shard 0's `(shard_uid, block_hash)` key: [11](#0-10) 

---

### Impact Explanation

The syncing node stores a `ChunkExtra` for shard 0 whose `state_root` is derived from shard 1's state and shard 1's transactions/receipts. Every subsequent block application, flat-storage update, and RPC query for shard 0 on this node operates on the wrong trie. The node diverges from all honest nodes and cannot produce or validate blocks correctly for shard 0. This is a concrete, named DB-entry delta: `DBCol::ChunkExtra` at key `(shard_uid(0), block_hash)` holds a state root that differs from the canonical value on honest nodes.

---

### Likelihood Explanation

The attacker must be a peer that the syncing node selects to serve the state sync header request. Any peer that joins the network and reaches sufficient height can be selected. The `receive_peer_message` routing in `StateSyncDownloadSourcePeerSharedState` matches responses by `(shard_id, sync_hash, PartIdOrHeader::Header)`, so the attacker only needs to respond to a header request with the correct `shard_id=0` and `sync_hash` in the envelope while embedding shard 1's chunk body. No validator or block-producer privileges are required. [12](#0-11) 

---

### Recommendation

Add an explicit equality check in `set_state_header` immediately after extracting the chunk:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the position-agnostic `verify_path` call with `verify_path_with_index` (already used for chunk parts) so that the Merkle proof is bound to the correct shard index in `chunk_headers_root`.

### Proof of Concept

1. Run a two-shard testnet. Let node B enter state sync for epoch E.
2. Intercept node B's `StateRequestHeader(shard_id=0, sync_hash)` request.
3. Respond with a `VersionedStateResponse` whose `shard_id=0` envelope contains a `ShardStateSyncResponseHeaderV2` built from shard 1's `ShardChunk`, shard 1's `chunk_proof` (valid Merkle proof at position 1), valid shard-0 receipt proofs, and shard 1's `state_root_node`.
4. Call `set_state_header(shard_id=0, sync_hash, crafted_header)` on node B — all five validation steps pass.
5. Download and apply shard 1's state parts via `apply_state_part(shard_id=0, state_root=shard_1_root, ...)`.
6. Call `chain.set_state_finalize(shard_id=0, sync_hash)`.
7. Assert: `node_B.get_chunk_extra(sync_prev_hash, shard_uid(0)).state_root() != node_A.get_chunk_extra(sync_prev_hash, shard_uid(0)).state_root()`.

The assertion holds because node B's shard-0 `ChunkExtra` contains shard 1's post-application state root. [13](#0-12) [14](#0-13)

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
