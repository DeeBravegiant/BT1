Audit Report

## Title
`_updateRsETHPrice` Downside Protection Returns Early Without Updating `rsETHPrice`, Leaving Stale Inflated Price After Unpause - (File: contracts/LRTOracle.sol)

## Summary
In `LRTOracle._updateRsETHPrice()`, when the downside-protection circuit-breaker fires, the function pauses all three contracts and returns immediately at line 282 — before the `rsETHPrice = newRsETHPrice` assignment at line 313 is ever reached. The stale, pre-drop price persists in storage. After the admin unpauses the protocol, any call to `LRTWithdrawalManager.instantWithdrawal()` that races ahead of `updateRSETHPrice()` uses the inflated stale price, allowing an attacker to burn rsETH and receive more underlying assets than the current TVL supports.

## Finding Description
`_updateRsETHPrice()` computes `newRsETHPrice` at line 250 and then enters the downside-protection block at lines 270–282:

```solidity
if (isPriceDecreaseOffLimit) {
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    return;   // exits before rsETHPrice is written
}
```

The only write to `rsETHPrice` is at line 313, which is unreachable when the early return fires. `updateRSETHPrice()` carries a `whenNotPaused` guard (line 87), so the price cannot be corrected through the public path while the oracle is paused. `updateRSETHPriceAsManager()` (line 94) bypasses the pause but is restricted to `onlyLRTManager` and is not called automatically on unpause — the contract imposes no ordering constraint.

Once the admin unpauses all three contracts, `instantWithdrawal()` (line 212) is immediately callable by any user. It computes the payout via `getExpectedAssetAmount()` (line 228), which reads `lrtOracle.rsETHPrice()` directly (line 593) with no staleness check. Because `rsETHPrice` still holds the pre-drop value, `assetAmountUnlocked` is inflated. The only guard is a check against `unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)` (line 231), which limits the per-call drain to available vault liquidity but does not prevent the price mismatch. The attacker burns rsETH at the stale high price and receives excess assets; the shortfall is borne by remaining rsETH holders.

## Impact Explanation
**Critical — direct theft of user funds.** An unprivileged caller can burn rsETH and receive underlying assets priced at the stale (pre-drop) `rsETHPrice`. The excess assets are drawn from the protocol's TVL, directly diluting all remaining rsETH holders. The magnitude per call is `(stalePriceRatio − 1) × rsETHBurned × assetPrice`; the attack is repeatable until `updateRSETHPrice()` is called and the price is corrected. This matches the allowed Critical impact: *Direct theft of any user funds*.

## Likelihood Explanation
Three conditions must align: (1) TVL drops by more than `pricePercentageLimit` in a single oracle update — realistic during a large slashing event or oracle manipulation; (2) the admin subsequently unpauses the protocol — a normal operational response after investigating the cause; (3) an attacker front-runs the `updateRSETHPrice()` transaction that would correct the price — a straightforward mempool front-run requiring no special privileges. Instant withdrawal must be enabled for the target asset, which is a normal operational configuration. Overall likelihood is Low–Medium, but the impact is Critical, making the risk significant.

## Recommendation
Write `rsETHPrice = newRsETHPrice` **before** calling `_pause()` and returning in the downside-protection block, so the stored price always reflects the actual current state even when the circuit-breaker fires:

```solidity
if (isPriceDecreaseOffLimit) {
    rsETHPrice = newRsETHPrice;          // add this line
    if (!lrtDepositPool.paused()) lrtDepositPool.pause();
    if (!withdrawalManager.paused()) withdrawalManager.pause();
    _pause();
    emit RsETHPriceUpdate(rsETHPrice, previousPrice);
    return;
}
```

This ensures that after unpause, any call to `instantWithdrawal()` or `unlockQueue()` uses the correct (post-drop) price, eliminating the exploitation window entirely.

## Proof of Concept
1. `pricePercentageLimit` is set to `1e16` (1%). `rsETHPrice` is `1.00e18`.
2. A slashing event reduces TVL. Anyone calls `LRTOracle.updateRSETHPrice()`.
3. Inside `_updateRsETHPrice()`, `newRsETHPrice` is computed as `0.97e18`. `isPriceDecreaseOffLimit` is `true` (3% drop > 1% limit).
4. `lrtDepositPool`, `withdrawalManager`, and the oracle are paused. The function returns. `rsETHPrice` remains `1.00e18`.
5. Admin investigates and calls `unpause()` on all three contracts.
6. Attacker immediately calls `LRTWithdrawalManager.instantWithdrawal(ETH, 100e18, "")`.
7. `getExpectedAssetAmount` returns `100e18 * 1.00e18 / 1e18 = 100 ETH` instead of the correct `97 ETH`.
8. Attacker receives 100 ETH, burning only 100 rsETH worth 97 ETH at current prices — a 3 ETH gain at the expense of remaining holders.
9. Attacker repeats until `updateRSETHPrice()` is called and the price is corrected.
