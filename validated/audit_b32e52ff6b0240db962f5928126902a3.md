### Title
Missing CKD Output Validation for Legacy `AppPublicKey` Format Allows Single Byzantine Participant to Deliver Forged Key Derivation Output - (File: crates/contract/src/lib.rs)

### Summary
The `respond_ckd` function accepts any CKD response without cryptographic verification when the request uses the legacy `AppPublicKey` format. A single malicious attested participant — strictly below the signing threshold — can submit a forged CKD response, causing the user to receive an attacker-controlled confidential key derivation output.

### Finding Description
In `respond_ckd`, the contract validates the CKD response only when the request uses the `AppPublicKeyPV` format. For the legacy `AppPublicKey` format, the validation branch is entirely empty — a direct structural analog to the external report's missing minimum token amounts:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}   // NO VALIDATION — any response accepted
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [1](#0-0) 

Just as the GMX emergency functions leave `minTokenAAmt`/`minTokenBAmt` at their default zero values (allowing 100% slippage), `respond_ckd` leaves the CKD output validation absent for the legacy format, allowing any response to be accepted without verification.

The `respond_ckd` function is callable by any single attested participant — no threshold agreement is required on-chain: [2](#0-1) 

The `AppPublicKeyPV` format explicitly enables on-chain verifiable output via `ckd_output_check`, while the legacy `AppPublicKey` path has no equivalent guard. The contract README acknowledges this asymmetry — `AppPublicKeyPV` "allows anyone to verify the encrypted result on-chain without the app's secret key" — but the legacy path remains live and unguarded. [3](#0-2) 

A single Byzantine participant can:
1. Wait for a user to submit `request_app_private_key` using the legacy `AppPublicKey` format.
2. Construct a `CKDResponse` that encrypts an attacker-chosen key `K` under the user's `app_public_key`.
3. Race to call `respond_ckd(request, forged_response)` before legitimate MPC nodes.
4. The contract executes the empty `AppPublicKey` branch — no validation — and accepts the forged response.
5. `resolve_yields_for` removes the request from the pending map and resolves the user's yield with the forged bytes. [4](#0-3) 

Because `resolve_yields_for` removes the entry on first call, subsequent legitimate responses receive `RequestNotFound` — the forged response is final.

### Impact Explanation
The user receives an attacker-controlled CKD output. Since the attacker chose `K` and encrypted it with the user's public key, the attacker knows the user's derived key. Any funds or secrets protected by that key are compromised. This constitutes **unauthorized confidential key derivation output without the required participant authorization** — matching the Critical allowed impact scope.

### Likelihood Explanation
Any single attested MPC node (below signing threshold) can exploit this. The attack is a race condition: submit the forged response before legitimate nodes. Since MPC nodes monitor the same NEAR indexer feed, a malicious node has equivalent timing to honest nodes. The legacy `AppPublicKey` format remains supported and callable by any user without restriction.

### Recommendation
1. **Preferred**: Deprecate and remove the legacy `AppPublicKey` format, requiring all callers to use `AppPublicKeyPV`, which has on-chain verifiable output via `ckd_output_check`.
2. **Alternative**: If the legacy format must be retained, require threshold-many participants to submit the same response before accepting it — analogous to how `vote_pk` requires threshold votes before a public key is accepted. [5](#0-4) 

### Proof of Concept
1. User calls `request_app_private_key` with `app_public_key = AppPublicKey(user_pk)`.
2. Malicious attested participant constructs `forged_response = BLS_encrypt(K, user_pk)` for attacker-chosen key `K`.
3. Malicious participant calls `respond_ckd(request, forged_response)` — races legitimate nodes.
4. Contract executes the empty `AppPublicKey(_) => {}` branch: no validation, response accepted.
5. `resolve_yields_for` resolves the user's yield with `forged_response`; subsequent legitimate responses get `RequestNotFound`.
6. User decrypts `forged_response` and uses `K` as their derived application key.
7. Attacker, knowing `K`, can steal any funds or secrets protected by `K` on any foreign chain.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
