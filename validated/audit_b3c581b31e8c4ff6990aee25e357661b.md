### Title
Unvalidated `difficulty` in `NewUnfinishedBlockTimelord` Allows Timelord Weight Poisoning — (`chia/timelord/timelord.py`)

### Summary

The `difficulty` field of `NewUnfinishedBlockTimelord` is consumed verbatim at line 694 of `timelord.py` to compute the timelord's self-constructed peak weight. No check is made that this value matches the consensus-validated difficulty. A peer who can send `NewUnfinishedBlockTimelord` with `difficulty = 2^64 - 1` will cause the timelord to store an astronomically inflated `last_weight`, after which every subsequent legitimate `NewPeakTimelord` from the full node is silently dropped, permanently desynchronizing the timelord.

---

### Finding Description

**Entrypoint — `new_unfinished_block_timelord`**

`TimelordAPI.new_unfinished_block_timelord` receives the message and schedules the block for infusion. [1](#0-0) 

Critically, the iteration calculation uses the timelord's *own* known difficulty (`self.timelord.last_state.get_difficulty()`), **not** `new_unfinished_block.difficulty`. The `difficulty` field carried in the message is never compared against consensus state at this stage. [2](#0-1) 

`_can_infuse_unfinished_block` performs the same pattern — it uses `self.last_state.get_difficulty()` for all iteration math and never inspects `block.difficulty`. [3](#0-2) 

**Weight inflation — `_check_for_new_ip` line 694**

Once the timelord's VDF clients finish the infusion-point proof, `_check_for_new_ip` constructs a `RewardChainBlock` using the attacker-supplied `block.difficulty` directly:

```python
new_reward_chain_block = RewardChainBlock(
    uint128(self.last_state.get_weight() + block.difficulty),   # ← attacker-controlled
    ...
)
``` [4](#0-3) 

This inflated block is then wrapped in a `NewPeakTimelord` and stored in `self.new_peak`: [5](#0-4) 

**State commit — `_handle_new_peak`**

`_handle_new_peak` calls `self.last_state.set_state(self.new_peak)` with no further validation: [6](#0-5) 

`set_state` writes the inflated weight directly into `self.last_weight`: [7](#0-6) 

**Rejection of all future legitimate peaks — `new_peak_timelord`**

Every subsequent `NewPeakTimelord` from the honest full node is compared against `last_state.get_weight()`:

```python
if self.timelord.last_state.get_weight() < new_peak.reward_chain_block.weight:
    ...  # accept
# else: silently skip
``` [8](#0-7) 

With `last_weight ≈ MAX_UINT128`, no honest peak can ever exceed it, so all are silently dropped.

---

### Impact Explanation

The timelord stops tracking the canonical chain. It continues computing VDFs but for a phantom peak, producing signage points and infusion VDFs that are useless to the network. Farmers connected to this timelord cannot finish blocks. If the victim timelord is the only or fastest timelord on the network, block production stalls. This matches the High impact criterion: **permanent or long-lived inability for timelords to process valid blocks under normal network assumptions**.

---

### Likelihood Explanation

The timelord's `on_connect` performs no authentication: [9](#0-8) 

Any peer that can establish a full-node connection to the timelord can send `NewUnfinishedBlockTimelord`. The attacker needs:
1. A network path to the timelord's Chia server port.
2. A `RewardChainBlockUnfinished` with any valid PoS (even a tiny plot) and a `rc_prev` value drawn from the public chain tip — both trivially obtainable.
3. `difficulty` set to `uint64(2**64 - 1)`.

No leaked keys, no admin access, no broken cryptography required.

---

### Recommendation

In `_check_for_new_ip`, replace `block.difficulty` with the timelord's own consensus-validated difficulty:

```python
# Before (vulnerable):
uint128(self.last_state.get_weight() + block.difficulty)

# After (safe):
uint128(self.last_state.get_weight() + self.last_state.get_difficulty())
```

The same fix should be applied to the `block.difficulty` and `block.sub_slot_iters` fields used to construct `NewPeakTimelord` at lines 751 and 753 — replace them with `self.last_state.get_difficulty()` and `self.last_state.get_sub_slot_iters()` respectively. [5](#0-4) 

---

### Proof of Concept

```python
# Attacker connects to timelord as a full node peer.
# Attacker knows the current rc_prev from the public chain.

crafted_block = NewUnfinishedBlockTimelord(
    reward_chain_block=<valid RewardChainBlockUnfinished with attacker's PoS>,
    difficulty=uint64(2**64 - 1),          # MAX_UINT64
    sub_slot_iters=<current sub_slot_iters>,
    foliage=<any valid foliage>,
    sub_epoch_summary=None,
    rc_prev=<current chain tip reward challenge>,  # public
    header_mmr_root=None,
)
await timelord_api.new_unfinished_block_timelord(crafted_block)

# Wait for timelord VDF clients to finish the infusion point proof.
# After _check_for_new_ip fires:
assert timelord.last_state.get_weight() >= 2**64 - 1

# Now send a legitimate peak:
await timelord_api.new_peak_timelord(honest_peak)
# Result: silently skipped — timelord is permanently desynchronized.
```
