### Title
`validatePerasCert` Stub Unconditionally Returns `True`, Permanently Bypassing Peras Certificate Validation in the BBODY Rule - (File: `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/BlockBody/Internal.hs`)

---

### Summary

The Dijkstra era's `validatePerasCert` function is a hardcoded stub that always returns `True` regardless of its inputs. The `dijkstraBbodyTransition` rule calls this function to validate any `PerasCert` present in a block body, then uses the `(?!)` predicate operator to reject the block if validation fails. Because the stub never fails, any block producer can embed an arbitrary or cryptographically invalid `PerasCert` in a Dijkstra block body and every honest node will unconditionally accept it. The `PerasCertValidationFailed` predicate failure is structurally unreachable.

---

### Finding Description

**Root cause — the stub:**

```haskell
-- | Mocked-up Peras certificate validation routine
--
-- NOTE: this function will be replaced with the real implementation from
-- 'cardano-base' once it's ready.
validatePerasCert :: Nonce -> PerasKey -> PerasCert -> Bool
validatePerasCert _ _ _ = True
``` [1](#0-0) 

All three arguments — the epoch nonce, the Peras public key, and the certificate bytes — are discarded with wildcard patterns. The function is a constant `True`.

**Call site in the BBODY transition rule:**

```haskell
case blockBody ^. perasCertBlockBodyL of
  SNothing -> pure ()
  SJust cert ->
    let nonce = block ^. prevNonceBlockHeaderL
     in validatePerasCert nonce PerasKey cert ?! injectFailure (PerasCertValidationFailed cert nonce)
``` [2](#0-1) 

The `(?!)` operator raises `PerasCertValidationFailed` only when `validatePerasCert` returns `False`. Because it never does, the predicate failure constructor defined in `DijkstraBbodyPredFailure` is dead code:

```haskell
| PerasCertValidationFailed PerasCert Nonce
``` [3](#0-2) 

**The `PerasKey` placeholder is also empty:**

```haskell
data PerasKey = PerasKey
  deriving (Eq, Show, Generic, NoThunks)
``` [4](#0-3) 

There is no actual key material. The constructor `PerasKey` used at the call site carries no cryptographic content, so even if the stub were replaced with a real verifier, the key passed to it would still be empty.

**Analog to the ERC20 vulnerability class:**

The ERC20 report describes a function (`approve`) that is supposed to return a meaningful `bool` but some implementations return nothing — the caller assumes success when it should not. Here, `validatePerasCert` is supposed to return a meaningful `Bool` (cryptographically valid / invalid) but always returns `True` — the BBODY rule assumes every certificate is valid when it should not. The structural pattern is identical: a validation call whose return value is checked by the caller, but whose implementation makes the check vacuous.

### Impact Explanation

**Allowed impact matched:** *Critical — Honest nodes accept an invalid block or transaction causing permanent ledger divergence requiring a hard fork.*

Any block producer can craft a Dijkstra block body containing an arbitrary byte sequence as a `PerasCert`. Every honest node running the current code will accept that block unconditionally, because `validatePerasCert` never rejects anything. When the real `validatePerasCert` implementation is eventually shipped from `cardano-base` (as the comment promises), nodes running the new code will reject blocks whose `PerasCert` fails cryptographic verification, while nodes still running the stub will accept them. This creates a deterministic, permanent chain split between the two populations of nodes — exactly the ledger-divergence scenario that requires a hard fork to resolve.

Even before that transition, the bypass allows a block producer to assert false Peras finality certificates on-chain. Peras is a finality layer; forged certificates can misrepresent which blocks are finalized, undermining the safety guarantees the protocol is designed to provide.

---

### Likelihood Explanation

The entry path requires only a block producer role — no privileged key, no governance majority, no Sybil attack. Any slot leader in the Dijkstra era can include a `PerasCert` field in their block body. The CBOR encoding and decoding of `PerasCert` is fully functional:

```haskell
newtype PerasCert = PerasCert ByteArray
  deriving (Eq, Show, Generic)
  deriving newtype (EncCBOR, DecCBOR)
``` [5](#0-4) 

A block producer needs only to set `dbbrPerasCert` to `SJust (PerasCert <arbitrary bytes>)` when constructing a block. The BBODY rule will accept it. Likelihood is **high** for any block producer operating in the Dijkstra era.

---

### Recommendation

1. **Do not ship a stub that returns `True`** as a security check. If the real `validatePerasCert` from `cardano-base` is not yet available, the safer interim design is to reject any block that contains a `PerasCert` outright (i.e., treat `SJust cert -> fail "PerasCert validation not yet supported"`) rather than silently accepting all certificates.

2. **Replace `PerasKey` with a real key type** before enabling Peras certificate validation. The current `data PerasKey = PerasKey` carries no cryptographic material; passing it to any real verifier would still produce incorrect results.

3. **Audit the transition from stub to real implementation** for the ledger-divergence window: nodes must be upgraded atomically (via a hard-fork combinator boundary) so that the stub and the real verifier are never simultaneously active on the same network.

### Proof of Concept

1. Construct a Dijkstra block body with `dbbrPerasCert = SJust (PerasCert <random 32 bytes>)`.
2. Serialize it with the standard `EncCBOR (DijkstraBlockBodyRaw era)` instance.
3. Submit the block to any Dijkstra-era node.
4. Observe: `dijkstraBbodyTransition` reaches the `SJust cert` branch, calls `validatePerasCert nonce PerasKey cert`, receives `True`, and the `(?!)` check passes without emitting `PerasCertValidationFailed`.
5. The block is accepted by all honest nodes despite containing a cryptographically meaningless certificate.

The unreachability of `PerasCertValidationFailed` can be confirmed statically: `validatePerasCert _ _ _ = True` is the only definition, so the `False` branch of `(?!)` is dead code. [6](#0-5) [7](#0-6)
