Audit Report

## Title
`AnchoredPriceProvider` applies a single `MAX_REF_STALENESS` to both legs of a synthetic ratio feed, enabling stale-price execution — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary
`AnchoredPriceProvider` in synthetic ratio mode (`quoteFeedId != bytes32(0)`) computes `price(baseFeedId) / price(quoteFeedId)` but validates both legs against the same immutable `MAX_REF_STALENESS` via `_readLeg()`. Because different oracle feeds have different update heartbeats, a single threshold cannot correctly guard both legs simultaneously, producing either a stale fast-feed price reaching a pool swap (bad-price execution, LP loss) or near-constant downtime from the slow feed failing the tight threshold. The `AnchoredProviderFactory` compounds this by validating `maxRefStaleness` only against the base feed's class envelope, leaving the quote feed's appropriate staleness window entirely unchecked.

## Finding Description
`AnchoredPriceProvider` declares one staleness immutable:

```solidity
// L77
uint256 public immutable MAX_REF_STALENESS;
```

`_readLeg(bytes32 feedId)` applies it unconditionally to whichever feed it is called with:

```solidity
// L283
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
```

`_getBidAndAskPrice()` calls `_readLeg` for both the base and quote legs:

```solidity
// L259, L264
(uint256 mid,  uint256 spreadBps,  , bool ok)  = _readLeg(baseFeedId);
(uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
```

Both calls use the same `MAX_REF_STALENESS`. No per-leg threshold exists. The factory's `createAnchoredProvider` validates `maxRefStaleness` only against the envelope of `baseFeedId`:

```solidity
// L171-L180
bytes32 classId = feedClass[baseFeedId];
if (classId == bytes32(0)) classId = DEFAULT_CLASS;
Envelope storage env = envelopes[classId];
if (
    ...
    || maxRefStaleness < env.stalenessMin || maxRefStaleness > env.stalenessMax
    ...
) revert ParamsOutOfEnvelope();
```

The quote feed's appropriate staleness window is never validated against any envelope. The factory NatSpec explicitly acknowledges this: "The envelope is keyed on `baseFeedId`; the ref feed only contributes its uncertainty and is validated for existence at provider construction." No other guard in `_readLeg()` compensates — the `spreadBps >= ORACLE_BPS` check only catches the Chainlink RWA off-hours marker, not general staleness of a faster feed within a looser window.

## Impact Explanation
Two mutually exclusive failure modes arise from the shared threshold:

1. **Bad-price execution (LP principal loss):** If `MAX_REF_STALENESS` is set to accommodate the slower feed (e.g. 24 hours), the faster feed can be up to 24 hours stale before the check triggers. A stale fast-feed price passes `_readLeg`, reaches `_computeBidAsk`, and produces a synthetic ratio bid/ask that does not reflect current market conditions. An unprivileged swapper can exploit the mispriced ratio to extract value from LPs — a direct loss of LP principal. This matches the Metric OMM Allowed Impact Gate: "bad-price execution: stale bid/ask quote reaches a pool swap."

2. **Broken core pool functionality:** If `MAX_REF_STALENESS` is set to accommodate the faster feed (e.g. 1 minute), the slower feed will fail the staleness check on almost every swap, causing `FeedStalled` reverts and rendering the pool's swap functionality unusable. This matches: "broken core pool functionality causing loss of funds or unusable swap flows."

## Likelihood Explanation
- Synthetic ratio mode is an explicitly documented and factory-supported deployment path (`quoteFeedId` parameter in `createAnchoredProvider`).
- Different oracle feeds routinely have different heartbeats; this is not an edge case.
- `createAnchoredProvider` is permissionless — any caller can deploy a synthetic ratio provider.
- No privileged action is required to trigger the bad-price path — any unprivileged swapper can execute a swap when the faster feed is stale but within the looser window.
- The factory provides no mechanism to set per-leg staleness, so every synthetic ratio provider deployed through it is structurally affected.

## Recommendation
Add a separate staleness immutable for the quote feed leg:

```solidity
uint256 public immutable MAX_BASE_REF_STALENESS;
uint256 public immutable MAX_QUOTE_REF_STALENESS;
```

Pass both through the constructor and `createAnchoredProvider`. Accept the applicable threshold as a parameter in `_readLeg`:

```solidity
function _readLeg(bytes32 feedId, uint256 maxStaleness)
    internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
{
    ...
    if (_isStale(refTime, block.timestamp, maxStaleness)) return (..., false);
    ...
}
```

Call with the correct threshold per leg in `_getBidAndAskPrice`:

```solidity
(uint256 mid,  uint256 spreadBps,  , bool ok)  = _readLeg(baseFeedId, MAX_BASE_REF_STALENESS);
(uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote,     MAX_QUOTE_REF_STALENESS);
```

Extend `AnchoredProviderFactory` envelope validation to also validate the quote feed's staleness against its own class envelope.

## Proof of Concept
**Setup:** Deploy `AnchoredPriceProvider` in synthetic ratio mode via `createAnchoredProvider` with:
- `baseFeedId` = ETH/USD (updates every ~1 second via Data Streams)
- `quoteFeedId` = BTC/USD (updates every ~24 hours)
- `MAX_REF_STALENESS` = 25 hours (required to avoid downtime from the slow BTC feed; passes the base-feed envelope check)

**Attack:**
1. Wait until the ETH/USD feed is 2 hours stale (no new report pushed).
2. The BTC/USD feed was updated 1 hour ago — both pass the 25-hour staleness check in `_readLeg`.
3. Call `swap` on the pool. `_getBidAndAskPrice()` reads the 2-hour-old ETH/USD price and the fresh BTC/USD price, computing a synthetic BTC/ETH ratio using a stale ETH denominator.
4. If ETH moved 5% in those 2 hours, the synthetic ratio is off by ~5%, allowing the attacker to buy BTC/ETH at a price ~5% below fair value, extracting that value from LPs.

**Foundry test plan:** Fork mainnet, deploy the provider in synthetic ratio mode with the above parameters, mock the oracle to return a 2-hour-old ETH/USD price and a fresh BTC/USD price, call `getBidAndAskPrice()`, and assert the returned bid/ask reflects the stale ratio rather than the current market price.
