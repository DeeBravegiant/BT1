Audit Report

## Title
Stale `rsETHPrice` Allows Deposit at Stale Price and Withdrawal at Updated Price, Extracting Accumulated Yield — (`contracts/LRTOracle.sol`)

## Summary

`LRTOracle.rsETHPrice` is a cached state variable updated only when `updateRSETHPrice()` is explicitly called. Because `updateRSETHPrice()` is `public` with no access control, an attacker can deposit at the stale (lower) price to receive excess rsETH, immediately trigger the price update, and then withdraw at the freshly updated (higher) price — extracting the accumulated staking yield that belongs to existing depositors. The `instantWithdrawal` path collapses this into a single block with no holding-period risk.

## Finding Description

**Root cause:** `rsETHPrice` is a stored value in `LRTOracle` that is not recalculated on every deposit or withdrawal. [1](#0-0) 

It is only updated when `updateRSETHPrice()` is called, which is declared `public` with no role restriction: [2](#0-1) 

**Deposit path** (`LRTDepositPool.getRsETHAmountToMint`): a lower (stale) `rsETHPrice` in the denominator yields a larger rsETH mint amount — the depositor receives more shares than fair value warrants. [3](#0-2) 

**Withdrawal path** (`LRTWithdrawalManager.getExpectedAssetAmount`): a higher (freshly updated) `rsETHPrice` in the numerator yields a larger asset payout. [4](#0-3) 

**Exploit flow:**

1. While `rsETHPrice` is stale (e.g., 1.00 ETH, true rate 1.05 ETH), attacker calls `depositETH{value: 100 ETH}`. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.00e18 = 100 rsETH`. Fair value would be ~95.24 rsETH; attacker receives ~4.76 rsETH in excess.
2. Attacker calls `LRTOracle.updateRSETHPrice()` (public, no access control). `rsETHPrice` is now 1.05 ETH.
3. Attacker calls `initiateWithdrawal(ETH, 100 rsETH)`. `getExpectedAssetAmount` computes `100e18 * 1.05e18 / 1e18 = 105 ETH`, which is stored as `expectedAssetAmount`.
4. After `withdrawalDelayBlocks`, `completeWithdrawal` pays out 105 ETH. Net profit: 5 ETH extracted from existing depositors' accumulated yield.

**Why existing checks are insufficient:**

*`pricePercentageLimit` check:* The `pricePercentageLimit` variable is initialized to `0` (no explicit initialization in `initialize()`), meaning the guard is entirely bypassed by default: [5](#0-4) 

Even when set to a non-zero value, it only caps a single update increment — not cumulative drift — so any yield that accrued within the limit window remains extractable.

*`_calculatePayoutAmount` min() protection:* During `unlockQueue`, the payout is `min(expectedAssetAmount, currentReturn)`: [6](#0-5) 

This only prevents the attacker from benefiting from *further* price increases after `initiateWithdrawal`. It does not prevent the attacker from locking in the already-updated higher price at initiation time, which is the core of the exploit.

*`instantWithdrawal` path:* This path has no min() protection at all and no withdrawal delay, collapsing the entire attack into a single block: [7](#0-6) 

## Impact Explanation

**High — Theft of unclaimed yield.** The attacker mints rsETH at a price that does not yet reflect accumulated staking rewards, then redeems at the price that does. The difference is yield earned by existing long-term depositors that is directly diverted to the attacker. The profit is quantifiable and bounded by the gap between the stale stored price and the true price at the time of the attack.

## Likelihood Explanation

- `updateRSETHPrice()` is `public` with no access control; the attacker controls exactly when the price update fires.
- `pricePercentageLimit` defaults to `0`, meaning the threshold check is disabled unless explicitly configured.
- All inputs to the price calculation (`getTotalAssetDeposits`, underlying LST oracle prices, rsETH total supply) are on-chain and publicly readable, so the attacker can compute exact profit before committing.
- No privileged role, governance capture, or external protocol compromise is required.
- When `instantWithdrawal` is enabled for an asset, the attack requires no holding-period risk and executes atomically.

**Likelihood: Medium** (requires stale price gap to exist, which is a normal operational condition between keeper calls).

## Recommendation

1. **Call `_updateRsETHPrice()` atomically inside `depositETH`/`depositAsset` and `initiateWithdrawal`/`instantWithdrawal`** before computing the mint or redemption amount. This ensures every user interaction uses the current TVL-derived price.
2. Alternatively, implement a **commit-reveal / epoch-based** scheme: collect deposit and withdrawal requests during the current epoch and settle them all at the price computed at the start of the next epoch, so no single actor can observe the future price before committing funds.
3. Ensure `pricePercentageLimit` is always set to a non-zero value as a defence-in-depth measure, and enforce it consistently across all callers.

## Proof of Concept

**Precondition:** Staking rewards have accrued since the last `updateRSETHPrice()` call. True rsETH/ETH rate is 1.05 ETH per rsETH, but `rsETHPrice` is stored as 1.00 ETH. `pricePercentageLimit == 0` (default) or accumulated drift is within the limit.

**Step 1 — Deposit at stale price:**
```solidity
LRTDepositPool.depositETH{value: 100 ether}(0, "");
// rsethMinted = (100e18 * 1e18) / 1.00e18 = 100 rsETH
// Fair value: 100 / 1.05 ≈ 95.24 rsETH → attacker receives ~4.76 rsETH excess
```

**Step 2 — Trigger price update:**
```solidity
LRTOracle.updateRSETHPrice();
// rsETHPrice is now 1.05e18
```

**Step 3a — Initiate withdrawal (standard path):**
```solidity
LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, 100 ether, "");
// expectedAssetAmount = 100e18 * 1.05e18 / 1e18 = 105 ETH (locked)
```

**Step 3b — Instant withdrawal (if enabled, single-block):**
```solidity
LRTWithdrawalManager.instantWithdrawal(ETH_TOKEN, 100 ether, "");
// assetAmountUnlocked = 100e18 * 1.05e18 / 1e18 = 105 ETH (minus fee)
```

**Step 4 — Complete withdrawal (standard path only):**
After `withdrawalDelayBlocks`, call `completeWithdrawal`. Attacker receives 105 ETH having deposited 100 ETH — a risk-free 5 ETH profit extracted from existing depositors' accumulated yield.

**Foundry test plan:** Deploy `LRTOracle`, `LRTDepositPool`, and `LRTWithdrawalManager` in a fork test. Simulate reward accrual by directly increasing the ETH balance of a `NodeDelegator` (increasing `getTotalAssetDeposits`). Verify `rsETHPrice` is stale, execute the above call sequence, and assert the attacker's final ETH balance exceeds the initial deposit by the expected yield amount.
