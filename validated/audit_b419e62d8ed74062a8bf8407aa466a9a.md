Audit Report

## Title
Precision Loss in `rewardPerToken()` Permanently Destroys Unclaimed Yield via Unconditional `updatedAt` Advancement — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
Integer division in `rewardPerToken()` truncates the per-token increment to zero whenever `rewardRate * elapsed * DECIMAL_PRECISION < totalKernelStaked`. The `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when the computed increment is zero, permanently consuming the elapsed seconds that produced no increment. Reward tokens deposited via `notifyRewardAmount` are irreversibly locked in the contract with no recovery path.

## Finding Description
`rewardPerToken()` (L408–414) computes:

```solidity
rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
    / totalKernelStaked;
```

When `rewardRate * elapsed * 1e18 < totalKernelStaked`, Solidity integer division truncates the result to `0`, returning `rewardPerTokenStored` unchanged.

The `updateReward` modifier (L232–242) then:
1. Writes the unchanged value back to `rewardPerTokenStored`
2. **Unconditionally sets** `updatedAt = lastTimeRewardApplicable()` (L234)

Step 2 is the root cause. The elapsed window that produced a zero increment is permanently consumed — `updatedAt` advances past it — so those seconds can never contribute to future accumulation. Any subsequent call to a `updateReward`-gated function (`stake`, `stakeFor`, `initiateWithdrawal`, `getReward`) repeats this cycle.

The `rewardRate == 0` guard at L586 does not prevent `rewardRate = 1`. With `rewardRate = 1` (achievable by distributing `duration` wei of a low-decimal token over `duration` seconds) and `totalKernelStaked = 2e18`, each 1-second window truncates: `(1 * 1 * 1e18) / 2e18 = 0`. The same truncation occurs at higher `rewardRate` values when `totalKernelStaked` is proportionally larger (e.g., `rewardRate = 1000`, `totalKernelStaked = 2e21`).

Existing guards are insufficient: the `rewardRate == 0` revert (L586) only blocks a zero rate, not a rate of 1 or any value where the per-second increment truncates. The `totalKernelStaked == 0` guard in `rewardPerToken()` (L409) is unrelated. No remainder accumulator or conditional `updatedAt` advancement exists.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

Reward tokens transferred into the contract via `notifyRewardAmount` (L574) are permanently locked when the truncation condition holds across the entire reward window. There is no `rescueTokens`, `recoverERC20`, or equivalent function to retrieve stranded reward tokens. Users accrue zero rewards for every elapsed window where truncation occurs, and the corresponding token balance remains frozen in the contract indefinitely. The staked KERNEL principal is unaffected; only the yield is destroyed.

## Likelihood Explanation
**Low-to-medium.** The condition requires either a small `rewardRate` (close to 1, reachable with low-decimal reward tokens or small reward amounts over long durations) or a large `totalKernelStaked` relative to `rewardRate`. Both are reachable without privileged access: any user can increase `totalKernelStaked` by staking, and `rewardRate` is set by the admin via `notifyRewardAmount`. The condition worsens as TVL grows. Additionally, any unprivileged user can deliberately worsen precision loss by calling public `updateReward`-gated functions (e.g., `getReward()`) at high frequency, advancing `updatedAt` every second and ensuring each window truncates independently.

## Recommendation
Scale `rewardRate` by `DECIMAL_PRECISION` at the point it is set in `notifyRewardAmount`, deferring the division to `rewardPerToken()`:

```solidity
// In notifyRewardAmount:
rewardRate = (receivedAmount * DECIMAL_PRECISION) / duration;

// In rewardPerToken():
return rewardPerTokenStored + (rewardRate * elapsed) / totalKernelStaked;
```

This shifts precision loss to a much smaller magnitude. Alternatively, track a `rewardPerTokenRemainder` accumulator and only advance `updatedAt` when the accumulated numerator meets or exceeds `totalKernelStaked`, preserving sub-threshold elapsed time for future windows.

## Proof of Concept
Deploy `KernelDepositPool`, stake `2e18` KERNEL as Alice, and call `notifyRewardAmount(2_592_000)` with `duration = 30 days` to set `rewardRate = 1`. Then:

```solidity
for (uint i = 0; i < 100; i++) {
    vm.warp(block.timestamp + 1);
    vm.prank(alice);
    pool.getReward(); // triggers updateReward, advances updatedAt by 1s each iteration
}
assertEq(pool.earned(alice), 0); // 100 seconds of yield permanently destroyed
```

Each 1-second window computes `(1 * 1 * 1e18) / 2e18 = 0`; `updatedAt` advances past it. After 100 iterations, `earned(alice) == 0` despite 100 wei of reward having been allocated by `notifyRewardAmount`. The reward tokens remain locked in the contract.
