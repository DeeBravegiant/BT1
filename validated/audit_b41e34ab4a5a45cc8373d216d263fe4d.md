### Title
Caller-Agnostic `verify_foreign_transaction` Enables Front-Running and Cross-Chain Replay of MPC-Signed Attestations - (File: `crates/contract/src/lib.rs`)

### Summary
The `verify_foreign_transaction` function does not bind the caller's identity to the request or the resulting MPC threshold signature. Any unprivileged party can submit the same foreign-chain verification request and receive the identical MPC-signed attestation. This enables a front-running attacker to obtain the MPC signature before the legitimate bridge contract and replay it to claim bridged funds, causing double-spend conditions in the Omnibridge inbound flow.

### Finding Description
The `sign()` function explicitly binds the caller's `predecessor_account_id` into the `SignatureRequest` via `SignatureRequest::new(request.domain_id, request.payload, &predecessor, &request.path)`, making the resulting signature cryptographically specific to the caller's account and derivation path. [1](#0-0) 

The `verify_foreign_transaction` function takes a different path. It calls `args_into_verify_foreign_tx_request`, which converts `VerifyForeignTransactionRequestArgs` into `VerifyForeignTransactionRequest` by copying only `domain_id`, `request`, and `payload_version` — the caller's `predecessor_account_id` is silently dropped. [2](#0-1) 

The resulting `VerifyForeignTransactionRequest` (the map key and the value signed by MPC nodes) contains no caller identity whatsoever. [3](#0-2) 

The contract's own test confirms this is structural: Alice and Bob submitting identical parameters are queued under the **same** caller-agnostic key, and a single `respond_verify_foreign_tx` drains both yields, returning the same `VerifyForeignTransactionResponse` to both. [4](#0-3) 

The MPC nodes sign `(request, observed_values)` with no caller binding, so the resulting signature is universally valid for anyone who possesses it. [5](#0-4) 

### Impact Explanation
The primary production use case of `verify_foreign_transaction` is the Omnibridge inbound flow: a bridge contract submits the function to obtain an MPC-signed attestation that a foreign-chain transaction (e.g., a Bitcoin deposit) finalized, then uses that attestation to authorize minting tokens on NEAR.

Because the MPC signature is not bound to any caller, an attacker who observes a pending `verify_foreign_transaction` call in the NEAR mempool can:

1. Submit the identical request parameters (all public on-chain).
2. Receive the same `VerifyForeignTransactionResponse` (same `payload_hash` + same MPC signature).
3. Call the bridge contract with this response before the legitimate bridge contract's own transaction settles.
4. Claim the bridged funds (e.g., mint tokens) for themselves.

This is a direct double-spend / invalid bridge execution: the foreign-chain deposit is real, the MPC attestation is valid, but the funds are minted to the attacker rather than the intended recipient. The legitimate bridge contract's transaction then fails because the deposit has already been processed.

**Impact class**: High — cross-chain replay / forged foreign-chain verification causing double-spend conditions.

### Likelihood Explanation
- NEAR transactions are publicly visible before finalization; the attacker can observe any pending `verify_foreign_transaction` call.
- The attacker needs no special privileges, no collusion with MPC participants, and no key material — only the ability to submit a NEAR transaction with 1 yoctonear deposit.
- The request parameters (chain, tx_id, extractors, domain_id) are fully public.
- The attack is profitable whenever the bridged value exceeds the gas cost of the attacker's transaction.
- The fan-out design (multiple callers queued under one key) means the attacker does not even need to race the MPC computation — they can submit after the MPC nodes have already computed the signature and simply wait for the response to be delivered to their own yield slot.

### Recommendation
Bind `predecessor_account_id` into the `VerifyForeignTransactionRequest` (and therefore into the signed payload), mirroring the approach used by `sign()`. Concretely:

1. Add a `caller: AccountId` field to `VerifyForeignTransactionRequest`.
2. In `verify_foreign_transaction`, populate it with `env::predecessor_account_id()` before storing the request.
3. Include `caller` in the canonical encoding that MPC nodes sign (`ForeignTxSignPayload`).

This makes each attestation cryptographically specific to the requesting account, eliminating the replay surface. Bridge contracts can then verify that the attestation was issued for their own account ID.
