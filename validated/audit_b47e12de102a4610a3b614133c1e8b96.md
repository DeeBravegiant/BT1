### Title
CHIP-22 Harvester Fee Threshold Is Never Enforced — Harvester Can Redirect 100% of Farmer Block Rewards - (File: chia/farmer/farmer_api.py, chia/farmer/farmer.py)

### Summary
The CHIP-22 third-party harvester fee convention allows a harvester to redirect the farmer's block reward (0.25 XCH) to its own address via `farmer_reward_address_override`. The fee threshold check in `notify_farmer_reward_taken_by_harvester_as_fee()` is **only logged, never enforced**. A malicious harvester can set `fee_info=None` (or any threshold value) and still have its address override unconditionally accepted, diverting 100% of the farmer's block rewards.

### Finding Description

The CHIP-22 convention defines a fee quality mechanism: a harvester is only supposed to redirect the farmer reward when `fee_quality <= applied_fee_threshold`. The farmer computes `fee_quality = calculate_harvester_fee_quality(proof, challenge)` and compares it against the harvester-supplied `applied_fee_threshold`.

In `FarmerAPI.new_proof_of_space()`, when a `farmer_reward_address_override` is present, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`:

```python
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
``` [1](#0-0) 

Inside `notify_farmer_reward_taken_by_harvester_as_fee()`, when `fee_info is None` or when `fee_quality > fee_threshold`, the function **only emits a log warning** and returns normally — it does not raise, does not return a sentinel, and does not block the override:

```python
if proof_of_space.fee_info is not None:
    ...
    if fee_quality <= fee_threshold:
        self.log.info(...)
    else:
        self.log.warning("Harvester illegitimately took a fee reward ...")
else:
    self.log.warning("Harvester illegitimately took reward by failing to provide its fee rate ...")
``` [2](#0-1) 

After this non-enforcing check, `_process_respond_signatures()` unconditionally accepts the override from `RespondSignatures`:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
``` [3](#0-2) 

This `farmer_reward_address` is then placed directly into `DeclareProofOfSpace`, which is sent to the full node and determines where the 0.25 XCH farmer block reward is paid. [4](#0-3) 

The `fee_info` field in `NewProofOfSpace` is harvester-controlled and optional (`fee_info: ProofOfSpaceFeeInfo | None`): [5](#0-4) 

### Impact Explanation

A malicious third-party harvester can redirect 100% of the farmer's block rewards (0.25 XCH per block) to its own address by simply setting `farmer_reward_address_override` to its own puzzle hash and `fee_info=None`. The farmer will log a warning but proceed to sign and submit the block with the harvester's address as the reward destination. This is unauthorized payout redirection of XCH, matching the **High** impact category: "Bypass of authorization that enables payout redirection."

### Likelihood Explanation

Any third-party harvester (the exact use-case CHIP-22 targets) can exploit this permissionlessly. No keys, admin access, or cryptographic break is required — the harvester simply sets the fields in its protocol messages. The farmer has no way to distinguish a legitimate fee override from a malicious full-reward theft.

### Recommendation

The farmer must enforce the fee threshold check, not merely log it. `notify_farmer_reward_taken_by_harvester_as_fee()` should return a boolean indicating whether the override is legitimate, and `new_proof_of_space` / `_process_respond_signatures` should reject the override (and drop the block submission or revert to `self.farmer.farmer_target`) when the check fails. Additionally, `fee_info=None` should be treated as a failed check when `farmer_reward_address_override` is set.

### Proof of Concept

1. A third-party harvester connects to a farmer.
2. The harvester finds a valid proof of space and sends `NewProofOfSpace` with:
   - `farmer_reward_address_override = <harvester_own_puzzle_hash>`
   - `fee_info = None`
3. The farmer calls `notify_farmer_reward_taken_by_harvester_as_fee()`, which logs `"Harvester illegitimately took reward by failing to provide its fee rate"` but returns normally.
4. The farmer sends `RequestSignatures` to the harvester.
5. The harvester responds with `RespondSignatures` also containing `farmer_reward_address_override = <harvester_own_puzzle_hash>`.
6. `_process_respond_signatures()` sets `farmer_reward_address = response.farmer_reward_address_override` unconditionally.
7. The farmer submits `DeclareProofOfSpace` with the harvester's address as `farmer_reward_puzzle_hash`.
8. The full node includes the block; the 0.25 XCH farmer reward is paid to the harvester, not the farmer.
