### Title
Tiered Reference-Script Fee Curve Resets Per Transaction, Allowing Fee Reduction by Splitting — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Tx.hs`)

### Summary

The Conway/Dijkstra era `tierRefScriptFee` function applies an exponentially growing (convex) pricing curve to the total reference-script size **within a single transaction**. Because the curve resets to the base tier for every new transaction, an unprivileged sender can split a large reference-script workload across multiple transactions and pay materially less in total fees than a single transaction carrying the same aggregate script size would require. This directly undermines the DDoS-deterrent purpose of the tiered pricing mechanism.

### Finding Description

The ADR `docs/adr/2024-08-14_009-refscripts-fee-change.md` documents that the tiered pricing was introduced specifically to make large reference-script usage expensive while keeping the first 25 KiB cheap. The implementation in `getConwayMinFeeTx` calls `tierRefScriptFee` with the **per-transaction** reference-script size:

```haskell
getConwayMinFeeTx pp tx refScriptsSize =
  alonzoMinFeeTx pp tx <+> refScriptsFee
  where
    refScriptsFee =
      tierRefScriptFee
        (unboundRational $ pp ^. ppRefScriptCostMultiplierG)
        (fromIntegral @Word32 @Int . unNonZero $ pp ^. ppRefScriptCostStrideG)
        refScriptCostPerByte
        refScriptsSize
``` [1](#0-0) 

`tierRefScriptFee` accumulates cost in tiers of `sizeIncrement` bytes, multiplying the price by `multiplier` (1.2 in production) for each successive tier:

```haskell
go !acc !curTierPrice !n
  | n < sizeIncrement =
      Coin $ floor (acc + toRational n * curTierPrice)
  | otherwise =
      go (acc + sizeIncrementRational * curTierPrice) (multiplier * curTierPrice) (n - sizeIncrement)
``` [2](#0-1) 

Because `refScriptsSize` is computed fresh for each transaction via `txNonDistinctRefScriptsSize`, the tier counter always starts at zero for every new transaction:

```haskell
getConwayMinFeeTxUtxo pparams tx utxo =
  getMinFeeTx pparams tx $ txNonDistinctRefScriptsSize utxo tx
``` [3](#0-2) 

**Concrete fee comparison** (production parameters: stride = 25,600 bytes, multiplier = 1.2, base = 15 lovelace/byte):

| Scenario | Ref-script bytes | Ref-script fee |
|---|---|---|
| 1 tx × 204,800 bytes (8 tiers) | 204,800 | ≈ 6,335,648 lovelace |
| 8 txs × 25,600 bytes (1 tier each) | 204,800 | 8 × 384,000 = 3,072,000 lovelace |

Splitting saves **≈ 51 %** of the reference-script fee component for the maximum per-transaction size. The base fee (`a + b × txSize`) is negligible because reference inputs are just 32-byte `TxIn` pointers; the scripts themselves live in UTxO outputs and do not inflate the transaction body.

The block-level size check in `validateBodyRefScriptsSizeTooBig` / `totalRefScriptSizeInBlock` enforces a hard cap on total bytes per block but does **not** aggregate fees across transactions:

```haskell
validateBodyRefScriptsSizeTooBig pp blockBody utxo =
  let totalSize = totalRefScriptSizeInBlock protVer txs utxo
      maxSize   = fromIntegral @Word32 @Int $ pp ^. ppMaxRefScriptSizePerBlockG
   in totalSize <= maxSize ?! ...
``` [4](#0-3) 

The block limit (1 MiB in production) is therefore still reachable by an attacker who fills a block with many small transactions, each paying only the base tier price. [5](#0-4) 

### Impact Explanation

An unprivileged transaction sender can pay materially less in reference-script fees than the protocol intends for the same total computational burden on validating nodes. The tiered pricing was introduced explicitly to make large-scale reference-script usage economically prohibitive; splitting defeats that deterrent. The attacker can approach the per-block size limit (1 MiB) while paying only the base-tier price for every 25 KiB chunk, reducing the economic cost of a sustained DDoS-style workload by up to ~51 % compared to the intended pricing. This fits the **Medium** impact category: attacker-controlled transactions modify fees outside design parameters.

### Likelihood Explanation

The attack requires no special privilege, no key compromise, and no governance majority. Any wallet or script author can construct multiple transactions each referencing a disjoint set of UTxO outputs that carry large reference scripts. The technique is straightforward and the savings are significant at the upper end of the per-transaction size limit.

### Recommendation

Aggregate the reference-script fee across all transactions in a block (or across a sliding window) rather than resetting the tier counter per transaction. Alternatively, apply the tiered pricing to the **cumulative** reference-script bytes seen from a given sender within a block, analogous to how `totalRefScriptSizeInBlock` already tracks cumulative size for the hard cap. A simpler short-term mitigation is to lower the `refScriptCostStride` parameter so that the base tier is smaller and the multiplier kicks in sooner, reducing the benefit of splitting.

### Proof of Concept

Using production parameters (stride = 25,600, multiplier = 1.2, base = 15):

```
-- Single transaction: 8 × 25,600 = 204,800 bytes
tierRefScriptFee 1.2 25600 15 204800
  = 384000 + 460800 + 552960 + 663552 + 796262 + 955515 + 1146618 + 1375941
  = 6,335,648 lovelace

-- Eight transactions of 25,600 bytes each
8 × tierRefScriptFee 1.2 25600 15 25600
  = 8 × 384,000
  = 3,072,000 lovelace

-- Savings: 3,263,648 lovelace (~51%)
```

The test in `eras/conway/impl/test/Main.hs` confirms the tier values:

```haskell
map (tierRefScriptFee 1.5 step 15) [0, step .. 204800]
  `shouldBe` map Coin [0, 384000, 960000, 1824000, 3120000, 5064000, 7980000, 12354000, 18915000]
``` [6](#0-5) 

The same splitting technique applies in the Dijkstra era, where `refScriptCostStride` and `refScriptCostMultiplier` are now proper governance-updatable protocol parameters, making the attack surface persistent across future eras. [7](#0-6)

**File:** docs/adr/2024-08-14_009-refscripts-fee-change.md (L43-64)
```markdown
Once we have the total size of reference scripts used in a transaction we can proceed to computing the amount of Lovelace that will be added to the fee of a transaction. Instead of using the same linear cost for the whole size we split this total size into `25KiB` chunks and each subsequent chunk will get a linear pricing cost that is higher than the previous one by a multiplier of `1.2`. In other words pricing for the first `25KiB` will be as with the initial approach, just the value of `minFeeRefScriptCostPerByte`. The following `25KiB` will have the price of `minFeeRefScriptCostPerByte * multiplier` and  so on. These are the two new hardcoded values in the fee computation:

* Size increment: `25KiB` (or 25,600 bytes)
* Multiplier: `1.2`
* minFeeRefScriptCostPerByte: `15` (supplied in Conway genesis)

This tiered pricing for reference scripts is defined by this recursive function:

```haskell
tierRefScriptFee :: Integer -> Integer
tierRefScriptFee = go 0 minFeeRefScriptCostPerByte
  where
    go acc curTierPrice n
      | n < sizeIncrement =
          floor (acc + (n % 1) * curTierPrice)
      | otherwise =
          let acc' = acc + curTierPrice * (sizeIncrement % 1)
           in go acc' (multiplier * curTierPrice) (n - sizeIncrement)
    sizeIncrement = 25600
    multiplier = 1.2
    minFeeRefScriptCostPerByte = 15
```
