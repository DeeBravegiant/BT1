### Title
Missing Cryptographic Signature Verification in P2P Header Sync Allows Malicious Peer to Inject Arbitrary Block Metadata - (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` validates only that `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (== 1). It never calls `verify_block_signature`. A malicious unauthenticated P2P peer can send a `SignedBlockHeader` carrying any `BlockSignature{r, s}` values — including `Felt::ONE, Felt::ONE` — and the node will store the header and its signature unconditionally.

---

### Finding Description

`parse_data_for_block` performs exactly two checks before returning `Ok(Some(signed_block_header))`:

1. Block number ordering [1](#0-0) 
2. Signature vector length equals `ALLOWED_SIGNATURES_LENGTH` (1) [2](#0-1) 

No cryptographic check is performed. The comment on line 46-47 even acknowledges that "The verification that the size of the vector is 1 is done in the data verification" — implying cryptographic verification was intended elsewhere but was never implemented. [3](#0-2) 

The returned header flows directly into `write_to_storage`, which calls `append_header` and `append_block_signature` with no intervening verification: [4](#0-3) 

The function `verify_block_signature` — which computes `poseidon_hash(block_hash, state_diff_commitment)` and checks it against the sequencer's public key — exists in `starknet_api`: [5](#0-4) 

A global search confirms `verify_block_signature` has **zero call sites** inside `crates/apollo_p2p_sync`. It appears only in `block_test.rs`. [6](#0-5) 

---

### Impact Explanation

A malicious peer that wins a sync query response can supply a `SignedBlockHeader` with:
- Arbitrary `block_hash`
- Arbitrary `state_diff_commitment`
- Arbitrary `l1_gas_price`, `l1_data_gas_price`, `l2_gas_price`
- Arbitrary `timestamp`, `sequencer`, `starknet_version`
- A structurally valid but cryptographically bogus `BlockSignature`

All of this is committed to storage. Any RPC call that reads block headers (gas price queries, fee estimation, `starknet_getBlockWithTxHashes`, etc.) will return the attacker-controlled values as authoritative. This matches the **High** impact category: *RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

The parent-hash chain check is also explicitly absent (noted as a TODO on line 102): [7](#0-6) 

---

### Likelihood Explanation

Any node on the P2P network can respond to a header query. No authentication is required to participate as a peer. The attacker only needs to win the race to respond to a `HeaderQuery` before a legitimate peer does, or to be the only peer the victim connects to (eclipse attack). The structural requirements are trivial: send one `SignedBlockHeader` with `signatures.len() == 1` and the correct `block_number`.

---

### Recommendation

Call `verify_block_signature` inside `parse_data_for_block` after the length check, using the node's configured `sequencer_pub_key`. Reject (return `ParseDataError::BadPeer`) if verification fails or returns `false`. The sequencer public key should be available from node configuration and passed into the stream builder. The `state_diff_commitment` required for verification is present in `signed_block_header.block_header.state_diff_commitment`.

pub fn verify_block_signature(
    sequencer_pub_key: &SequencerPublicKey,
    signature: &BlockSignature,
    state_diff_commitment: &GlobalRoot,
    block_hash: &BlockHash,
) -> Result<bool, BlockVerificationError> {
    let message_hash = Poseidon::hash_array(&[block_hash.0, state_diff_commitment.0]);
    verify_message_hash_signature(&message_hash, &signature.0, &sequencer_pub_key.0).map_err(
        |err| BlockVerificationError::BlockSignatureVerificationFailed {
            block_hash: *block_hash,
            error: err,
        },
    )
