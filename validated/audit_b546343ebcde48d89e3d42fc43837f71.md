Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Limit Bypass - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses `totalAssetDeposits > limit` for ETH but `totalAssetDeposits + amount > limit` for ERC-20 assets. When total ETH deposits equal the configured cap, the ETH guard evaluates to `false` and does not revert, allowing the deposit to proceed and push total ETH holdings above the protocol-configured ceiling.

## Finding Description
The root cause is the missing `+ amount` in the ETH branch of `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

`_beforeDeposit` (lines 648–670) calls this function and reverts with `MaximumDepositLimitReached` only when it returns `true`. When `totalAssetDeposits == limit`, the ETH expression `limit > limit` is `false`, so no revert occurs. `depositETH` proceeds to mint rsETH, and after the call `getTotalAssetDeposits(ETH_TOKEN)` equals `limit + msg.value`, violating the cap invariant. Any unprivileged depositor can trigger this with a single public call to `depositETH`.

## Impact Explanation
The ETH deposit limit is the protocol's primary risk-management guardrail for native ETH. Bypassing it allows additional ETH to enter beyond the intended ceiling, minting rsETH against assets the protocol did not plan to hold. No funds are directly stolen, but the contract fails to enforce its promised deposit cap. This matches the allowed Low impact: **"Contract fails to deliver promised returns, but doesn't lose value."**

## Likelihood Explanation
The triggering condition — total ETH deposits exactly at the cap — is a normal operational state for a popular restaking protocol. Any depositor monitoring `getTotalAssetDeposits(ETH_TOKEN)` on-chain can exploit this deterministically with a single transaction. No privileged role, front-running, or special timing is required.

## Recommendation
Apply the same `+ amount` inclusion used for ERC-20 assets:

```diff
 if (asset == LRTConstants.ETH_TOKEN) {
-    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
 }
```

## Proof of Concept
```
Precondition:
  depositLimitByAsset[ETH_TOKEN] = 100_000 ether
  getTotalAssetDeposits(ETH_TOKEN) = 100_000 ether

1. Attacker calls depositETH{value: 1 ether}(0, "")
2. _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 1 ether):
     totalAssetDeposits = 100_000 ether
     check: 100_000 ether > 100_000 ether → false → no revert
3. rsETH minted for 1 ETH
4. getTotalAssetDeposits(ETH_TOKEN) = 100_001 ether (limit exceeded)

Foundry test plan:
- Fork or deploy LRTDepositPool with depositLimitByAsset[ETH_TOKEN] = 100_000 ether
- Seed the pool so getTotalAssetDeposits(ETH_TOKEN) == 100_000 ether
- Call depositETH{value: 1 ether} from an unprivileged address
- Assert the call does not revert and getTotalAssetDeposits(ETH_TOKEN) > depositLimitByAsset[ETH_TOKEN]
```
