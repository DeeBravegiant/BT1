### Title
Proposal Deposits Permanently Locked When `do_update` Clears All Pending Proposals — (`crates/contract/src/update.rs`)

---

### Summary

When `do_update` executes a winning governance proposal, it unconditionally clears **all** pending proposals via `self.entries.clear()`. Because `UpdateEntry` does not record the proposer's account ID or the deposit amount, the storage-staking deposits paid by every non-winning proposer are permanently locked in the contract with no recovery path.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a deposit proportional to the size of the proposed update (contract binary or config blob) to cover NEAR storage staking costs. [1](#0-0) 

The exact required deposit is retained; only the excess is refunded to the proposer. The proposer's identity is **not** stored anywhere in `UpdateEntry`: [2](#0-1) 

`UpdateEntry` holds only `update` and `bytes_used` — no `proposer: AccountId`, no `deposit: NearToken`. When `do_update` is triggered by a threshold vote, it removes the winning entry and then calls `self.entries.clear()` and `self.vote_by_participant.clear()`: [3](#0-2) 

The `clear()` call frees the on-chain storage occupied by every non-winning proposal, but the NEAR tokens that were deposited to stake that storage are never returned to their respective proposers. There is no refund loop, no stored proposer map, and no separate administrative recovery function anywhere in the contract.

---

### Impact Explanation

Every participant who proposed an update that was not selected permanently loses their storage-staking deposit. For a near-maximum-size contract binary (~1.5 MB), `required_deposit` is approximately 15 NEAR at current storage byte costs (`env::storage_byte_cost()` × bytes). With multiple concurrent proposals — a normal governance scenario — the total locked amount scales linearly with the number of non-winning proposals. The funds remain in the contract balance indefinitely with no mechanism to recover them.

This breaks the production safety/accounting invariant that a deposit paid to cover temporary storage must be returned when that storage is freed.

**Impact: Medium** — balance/accounting invariant violation that permanently freezes participant-paid deposits inside the chain-signature contract.

---

### Likelihood Explanation

This triggers in every normal governance cycle where more than one participant proposes a different update before threshold is reached. The `test_propose_update_contract_many` sandbox test already demonstrates multiple concurrent proposals existing simultaneously. [4](#0-3) 

Any participant (no special privilege beyond being in the voter set) can trigger the loss by proposing a competing update and gathering threshold votes for it, causing all other proposers to lose their deposits.

---

### Recommendation

Store the proposer's `AccountId` and the exact deposit amount inside `UpdateEntry`:

```rust
pub(crate) struct UpdateEntry {
    pub(super) update: Update,
    pub(super) bytes_used: u128,
    pub(super) proposer: AccountId,   // add
    pub(super) deposit: NearToken,    // add
}
```

In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue a `Promise::new(entry.proposer).transfer(entry.deposit)` for each non-winning proposal. This mirrors the pattern already used in `propose_update` for excess-deposit refunds. [5](#0-4) 

---

### Proof of Concept

1. Participant A calls `propose_update` with a 1 MB contract binary, paying ~10 NEAR deposit.
2. Participant B calls `propose_update` with a different 1 MB contract binary, paying ~10 NEAR deposit.
3. Participants vote for B's proposal until threshold is reached; `vote_update` calls `do_update`.
4. `do_update` removes B's entry, then calls `self.entries.clear()` — A's entry (and its ~10 NEAR deposit) is erased with no refund issued.
5. A's 10 NEAR is permanently locked in the contract. No function exists to recover it. [6](#0-5)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

async fn test_propose_update_contract_many() {
    let SandboxTestSetup {
        contract,
        mpc_signer_accounts,
        ..
    } = SandboxTestSetup::builder()
        .with_protocols(ALL_PROTOCOLS)
        .build()
        .await;
    dbg!(contract.id());

    const PROPOSAL_COUNT: usize = 2;
    let mut proposals = Vec::with_capacity(PROPOSAL_COUNT);
    // Try to propose multiple updates to check if they are being proposed correctly
    // and that we can have many at once living in the contract state.
    for i in 0..PROPOSAL_COUNT {
        let execution = mpc_signer_accounts[i % mpc_signer_accounts.len()]
            .call(contract.id(), method_names::PROPOSE_UPDATE)
            .args_borsh(current_contract_proposal())
            .max_gas()
            .deposit(CURRENT_CONTRACT_DEPLOY_DEPOSIT)
            .transact()
            .await
            .unwrap();

        assert!(
            execution.is_success(),
            "failed to propose update [i={i}]; {execution:#?}"
        );
        let proposal_id = execution.json().expect("unable to convert into UpdateId");
        proposals.push(proposal_id);
    }

    // Vote for the last proposal
    vote_update_till_completion(&contract, &mpc_signer_accounts, proposals.last().unwrap()).await;

    // Ensure all proposals are removed after update
    for proposal in proposals {
        let voter = mpc_signer_accounts.first().unwrap();
        let execution = voter
            .call(contract.id(), method_names::VOTE_UPDATE)
            .args_json(serde_json::json!({
                "id": proposal,
            }))
            .gas(GAS_FOR_VOTE_UPDATE)
            .transact()
            .await
            .unwrap();
        dbg!(&execution);

        assert!(execution.is_failure());
    }

    // Let's check that we can call into the state and see all the proposals.
    let state: ProtocolContractState = get_state(&contract).await;
    dbg!(state);
}
