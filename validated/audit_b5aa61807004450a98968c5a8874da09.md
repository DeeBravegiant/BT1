### Title
Caller-Supplied `payload_hash` in `respond_verify_foreign_tx` Enables Forged Foreign-Chain Verification Results - (File: `crates/contract/src/lib.rs`)

### Summary

`respond_verify_foreign_tx` verifies the MPC signature against a `payload_hash` taken directly from the caller-supplied `response` struct rather than independently computing the expected hash from the on-chain request. A single Byzantine attested participant can reuse any legitimately produced MPC signature (from any prior signing round) to resolve a pending foreign-tx verification request with an attacker-chosen `payload_hash`, causing the user's NEAR contract to receive a valid MPC signature over fabricated verification data.

### Finding Description

`respond_verify_foreign_tx` in `crates/contract/src/lib.rs` accepts a `VerifyForeignTransactionResponse` that contains two caller-supplied fields: `payload_hash` and `signature`. The contract's only integrity check is that `signature` is a valid ECDSA signature over `payload_hash` under the root MPC public key:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;   // ← caller-supplied
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,                                        // ← not derived from request
    &secp_pk,
)
``` [1](#0-0) 

The contract never independently computes the expected hash `SHA-256(borsh(ForeignTxSignPayload{request, extracted_values}))` from the on-chain `request` parameter and compares it to `response.payload_hash`. The `ForeignTxSignPayload::compute_msg_hash()` function exists and is used correctly by honest nodes, but the contract does not call it during response validation. [2](#0-1) 

Contrast this with `respond()` for regular signatures, where the payload hash is derived from the on-chain stored request — not from the response:

```rust
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    payload_hash,          // ← derived from on-chain request, not caller-supplied
    &expected_public_key,
)
``` [3](#0-2) 

The `VerifyForeignTransactionResponse` struct exposes `payload_hash` as a free field with no on-chain binding to the request: [4](#0-3) 

### Impact Explanation

A single Byzantine attested participant can:

1. Observe (or participate in) any legitimate MPC signing round for request R1, obtaining `sig_over_H1` where `H1 = SHA-256(borsh(ForeignTxSignPayload{R1, extracted_values_1}))`.
2. Call `respond_verify_foreign_tx(request=R2, response={payload_hash=H1, signature=sig_over_H1})` targeting a different pending request R2.
3. The contract checks `verify_ecdsa_signature(sig_over_H1, H1, root_pk)` → **true**, and resolves R2's pending yield with `{payload_hash=H1, sig_over_H1}`.
4. The user of R2 receives a valid MPC signature, but `payload_hash` encodes the extracted values of R1 (a different transaction), not R2.

For the Omnibridge inbound flow (the primary use case described in `docs/foreign-chain-transactions.md`), this allows an attacker to make a NEAR bridge contract believe that a foreign-chain transaction carried different extracted values (e.g., a different block hash, a different deposited amount, a different recipient) than what was actually observed. This enables forged bridge execution and potential double-spend conditions.

Additionally, once `resolve_yields_for` drains the pending queue for R2, honest nodes can no longer deliver the correct response — the request lifecycle is permanently corrupted for that request. [5](#0-4) 

### Likelihood Explanation

- Requires only a **single** Byzantine attested participant — strictly below the signing threshold.
- The attacker does not need to forge a threshold signature; they only need to reuse a legitimately produced one in the wrong context. NEAR transactions are public, so the attacker can observe `sig_over_H1` from any honest `respond_verify_foreign_tx` call for R1 and immediately replay it against R2.
- The attacker's account must be in the participant set and have a valid TEE attestation, but no collusion with other participants is required.
- The attack is a single on-chain transaction with no off-chain coordination.

### Recommendation

In `respond_verify_foreign_tx`, independently recompute the expected payload hash from the `request` parameter and the extracted values embedded in the response, then assert it equals `response.payload_hash` before accepting the response. Alternatively, restructure the response so that the contract receives the raw `ExtractedValue` list (not just the hash), computes `SHA-256(borsh(ForeignTxSignPayload{request, values}))` on-chain, and verifies the signature against that computed hash — mirroring how `respond()` derives `payload_hash` from the stored on-chain request rather than trusting the caller.

### Proof of Concept

```
// Setup: two pending foreign-tx requests R1 and R2 exist on-chain.
// Honest MPC nodes process R1 and produce sig_over_H1.
// Byzantine attested participant observes sig_over_H1 from the on-chain
// respond_verify_foreign_tx call for R1 (NEAR txns are public).

// Attacker calls:
respond_verify_foreign_tx(
    request = R2,                          // a different pending request
    response = VerifyForeignTransactionResponse {
        payload_hash: H1,                  // hash of R1's verification result
        signature: sig_over_H1,            // valid MPC sig over H1
    }
)

// Contract executes:
// 1. assert_caller_is_attested_participant_and_protocol_active() → passes (attacker is attested)
// 2. payload_hash = response.payload_hash = H1  (caller-supplied, not derived from R2)
// 3. verify_ecdsa_signature(sig_over_H1, H1, root_pk) → true (signature is genuinely valid)
// 4. resolve_yields_for(pending_verify_foreign_tx_requests, R2, serialize({H1, sig_over_H1}))
//    → R2's pending yield is resolved with forged data; honest response can never arrive.

// User of R2 receives VerifyForeignTransactionResponse { payload_hash: H1, sig_over_H1 }
// where H1 encodes R1's extracted values, not R2's.
// If the user's bridge contract does not independently reconstruct the expected hash
// from R2's request, it accepts the forged verification result.

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

impl ForeignTxSignPayload {
    pub fn compute_msg_hash(&self) -> std::io::Result<Hash256> {
        let mut hasher = sha2::Sha256::new();
        borsh::BorshSerialize::serialize(self, &mut hasher)?;
        Ok(Hash256(hasher.finalize().into()))
