### Title
Single Byzantine Participant Can Permanently Disable All Foreign-Chain Verification — (`File: crates/contract/src/lib.rs`)

### Summary

`verify_foreign_transaction` gates on `get_supported_foreign_chains()`, which uses a **strict all-participant intersection rule**. A single Byzantine participant (below signing threshold) can register an empty foreign-chain configuration, causing `get_supported_foreign_chains()` to return an empty set and making every `verify_foreign_transaction` call revert with `ForeignChainNotSupported`. This is the NEAR MPC analog of the reported pattern: a missing/incorrect configuration causes a function to explicitly fail for an entire network/chain class.

### Finding Description

`verify_foreign_transaction` in `crates/contract/src/lib.rs` checks whether the requested chain is in the supported set before queuing the request:

```rust
let requested_chain = request.request.chain();
let supported_chains = self.get_supported_foreign_chains();
if !supported_chains.contains(&requested_chain) {
    env::panic_str(
        &InvalidParameters::ForeignChainNotSupported {
            requested: requested_chain,
        }
        .to_string(),
    );
}
``` [1](#0-0) 

`get_supported_foreign_chains()` computes the **strict intersection** across all active participants:

```rust
let all_active_nodes_supports_chain =
    nodes_supporting_chain.is_superset(&active_participant_account_ids);
if all_active_nodes_supports_chain {
    Some(foreign_chain)
} else {
    None
}
``` [2](#0-1) 

Any active participant can call `register_foreign_chain_config` (authenticated only as a voter/participant) with an empty `ForeignChainConfiguration`. Once registered, that participant's empty set causes `nodes_supporting_chain.is_superset(active_participant_account_ids)` to be `false` for every chain, collapsing the supported set to empty. The design documentation explicitly acknowledges this: *"A single node that registers an empty list (or hasn't registered yet) drops every chain — one operator can take the whole feature down."* [3](#0-2) 

The newer threshold-based `get_available_foreign_chains()` exists and is the intended replacement, but `verify_foreign_transaction` still calls the broken intersection-based method. [4](#0-3) 

### Impact Explanation

Every `verify_foreign_transaction` call — for Bitcoin, Ethereum, Starknet, Solana, and all other supported chains — reverts with `ForeignChainNotSupported`. The entire bridge inbound flow (foreign chain → NEAR) is halted. Pending requests time out and fail. No funds are directly stolen, but the bridge's core safety invariant — that verified foreign-chain events can be attested — is broken for the duration of the attack. This maps to the **Medium** allowed impact: *"contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

### Likelihood Explanation

High. Any single active participant (below signing threshold) can trigger this at any time by submitting one on-chain transaction. No coordination, no key material, and no threshold collusion is required. The call is authenticated only as a participant voter, which is the exact attacker class the prompt identifies as in-scope ("Byzantine participant strictly below the signing threshold").

### Recommendation

Replace the `get_supported_foreign_chains()` gate in `verify_foreign_transaction` with `get_available_foreign_chains()`, which requires only ≥ signing-threshold participants to cover the chain rather than unanimous coverage. This is the fix already designed and tracked under issue [#3434](https://github.com/near/mpc/issues/3434) and described in `docs/design/calculating-supported-foreign-chains.md`. [5](#0-4) 

### Proof of Concept

1. Network is running with N active participants, all having registered Bitcoin as a supported chain. `verify_foreign_transaction` for Bitcoin succeeds.
2. Byzantine participant P (one of the N, below threshold) calls `register_foreign_chain_config` with an empty `ForeignChainConfiguration {}`.
3. `get_supported_foreign_chains()` now iterates `foreign_chain_to_node_mapping`: Bitcoin's supporter set no longer contains P, so `is_superset(active_participant_account_ids)` is `false`. The method returns an empty set.
4. Any subsequent `verify_foreign_transaction` call (for any chain) hits the `!supported_chains.contains(&requested_chain)` branch and panics with `ForeignChainNotSupported`.
5. All bridge inbound requests time out and fail until P re-registers a non-empty configuration. [6](#0-5)

```rust
    pub fn get_supported_foreign_chains(&self) -> dtos::SupportedForeignChains {
        let active_participant_account_ids = self
            .protocol_state
            .active_participants()
            .participants()
            .iter()
            .map(|(account_id, _, _)| account_id.clone())
            .collect::<BTreeSet<_>>();

        let mut foreign_chain_to_node_mapping: BTreeMap<
            &dtos::ForeignChain,
            BTreeSet<dtos::AccountId>,
        > = BTreeMap::new();

        for (account_id, chains) in self
            .node_foreign_chain_support
            .foreign_chain_support_by_node
            .iter()
        {
            for chain in chains.iter() {
                foreign_chain_to_node_mapping
                    .entry(chain)
                    .or_default()
                    .insert(account_id.clone());
            }
        }

        foreign_chain_to_node_mapping
            .into_iter()
            .filter_map(|(foreign_chain, nodes_supporting_chain)| {
                let all_active_nodes_supports_chain =
                    nodes_supporting_chain.is_superset(&active_participant_account_ids);

                if all_active_nodes_supports_chain {
                    Some(foreign_chain)
                } else {
                    None
                }
            })
            .cloned()
            .collect::<BTreeSet<dtos::ForeignChain>>()
            .into()
    }

  ≥ `signing_threshold` active participants cover `C`. `available ⊆ whitelisted` always.
