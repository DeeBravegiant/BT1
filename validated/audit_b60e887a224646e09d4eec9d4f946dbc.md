### Title
Mempool Eviction Not Rolled Back on Rejected Transaction Permanently Removes Valid Transactions — (`crates/apollo_mempool/src/mempool.rs`)

### Summary

`try_make_space` permanently evicts gap-account transactions from the mempool to make room for an incoming transaction. If the freed space is still insufficient and the incoming transaction is ultimately rejected with `MempoolFull`, the evictions are **never rolled back**. An attacker can exploit this to permanently remove valid transactions from the mempool at zero cost by submitting a large transaction that triggers eviction but is too large to be accepted.

### Finding Description

When the mempool is full and an incoming transaction is not creating a nonce gap, `handle_capacity_overflow` calls `try_make_space` to evict gap-account transactions: [1](#0-0) 

`try_make_space` iterates over evictable accounts (those in `accounts_with_gap`), permanently removing their transactions from `tx_pool` and `tx_queue` one by one: [2](#0-1) 

If `try_make_space` exhausts all evictable accounts before freeing enough space, it returns `false`. `handle_capacity_overflow` then returns `Err(MempoolFull)` and the incoming transaction is rejected. **The evictions already performed are not reversed.** This is explicitly documented as intentional in the test suite: [3](#0-2) 

The comment at line 1936–1937 reads:
> "Transactions tx1 and tx2 are evicted regardless of whether large_tx is accepted or rejected. We do not revert the eviction attempt even if adding large_tx ultimately fails."

The `stage` call in `get_txs` increments the staged nonce for each transaction returned to the batcher: [4](#0-3) 

The `MempoolState::stage` function only increments; there is no corresponding decrement path triggered by a rejected `add_tx`: [5](#0-4) 

### Impact Explanation

An attacker can craft a transaction that:
1. Passes gateway stateful validation (valid signature, nonce == account_nonce, so `creating_gap = false`)
2. Has a large payload (e.g., large `signature` field) so that `required_space > total_freed_space`
3. Triggers `try_make_space`, which permanently evicts victim gap-account transactions
4. Is then rejected with `MempoolFull` because the freed space is insufficient

Since rejected transactions are never included in a block, **no fee is charged to the attacker**. The attacker can repeat this indefinitely, permanently evicting valid transactions from the mempool before sequencing. This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

The `validate_tx` path used by the gateway also calls `validate_fee_escalation` and `validate_incoming_tx` but does **not** call `handle_capacity_overflow`, so the gateway pre-check does not prevent this: [6](#0-5) 

### Likelihood Explanation

- The attacker needs a valid account to pass gateway `__validate__` (low barrier).
- The attacker needs to submit a transaction whose `total_bytes()` exceeds the sum of all gap-account transaction bytes in the pool. With a large signature array, this is straightforward.
- No fee is paid for rejected transactions, so the attack is free to repeat.
- Evictable accounts are selected randomly, so the attacker cannot target a specific victim precisely, but can drain all gap-account transactions from the pool.

### Recommendation

Roll back evictions if the incoming transaction is ultimately rejected. Before calling `try_make_space`, save the transactions to be evicted. If `handle_capacity_overflow` returns `Err(MempoolFull)`, re-insert the evicted transactions into `tx_pool` and `tx_queue`. Alternatively, perform a dry-run capacity check before committing any evictions.

### Proof of Concept

1. Victim submits transactions with a nonce gap: address `0x1`, nonces 1 and 2, `account_nonce = 0`. These enter `accounts_with_gap`.
2. Attacker submits a transaction from address `0x0`, nonce 0 (`account_nonce = 0`), with a large signature (e.g., 20 felt elements). This transaction is not creating a gap (`closing_gap = true`, `creating_gap = false`).
3. Mempool is at capacity. `add_tx_validations` → `handle_capacity_overflow` → `try_make_space(required_space)`.
4. `try_make_space` evicts victim's nonce-1 and nonce-2 transactions, freeing their bytes. But `large_tx.total_bytes() > freed_bytes`, so `try_make_space` returns `false`.
5. `handle_capacity_overflow` returns `Err(MempoolFull)`. Attacker's transaction is rejected. No fee is charged.
6. Victim's transactions are permanently gone from the pool and queue.
7. Attacker repeats from step 2 whenever victim resubmits.

```rust
    pub fn try_make_space(&mut self, required_space: u64) -> bool {
        let mut total_space_freed = 0;
        let mut evicted_txs = Vec::new();

        while total_space_freed < required_space {
            let Some(address) = self.get_evictable_account() else {
                break;
            };

            let txs: Vec<_> = self.tx_pool.account_txs_sorted_by_nonce(address).copied().collect();
            for tx_ref in txs.iter().rev() {
                let tx = self
                    .tx_pool
                    .remove(tx_ref.tx_hash)
                    .expect("Transaction must exist in the pool.");
                total_space_freed += tx.total_bytes();
                evicted_txs.push(*tx_ref);
                metric_count_evicted_txs(1);
                self.decrement_stuck_txs_if_gap_account(address, 1);
                if total_space_freed >= required_space {
                    break;
                }
            }

            // Clean up if account is now empty.
            if !self.tx_pool.contains_account(address) {
                self.accounts_with_gap.swap_remove(&address);
            }
        }

        // Keep the queue consistent with the pool: the evicted txs were removed from the pool, so
        // drop their (now-orphaned) queue references too. In fee mode gap accounts are never
        // queued, so this is a no-op there; in Echonet/FIFO mode they are, and skipping this leaves
        // a dangling reference that panics the next `get_txs`.
        self.tx_queue.remove_txs(&evicted_txs);

        total_space_freed >= required_space
    }
```

```rust
    fn handle_capacity_overflow(
        &mut self,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
        freed_bytes: u64,
    ) -> Result<(), MempoolError> {
        let address = tx.contract_address();

        let account_has_gap = self.accounts_with_gap.contains(&address);
        let account_has_txs = self.tx_pool.contains_account(address);
        let closing_gap = tx.nonce() == account_nonce;
        let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;

        // Only the net growth must be evicted: an accompanying replacement removal frees
        // `freed_bytes` (0 when there is no replacement).
        let required_space = tx.total_bytes().saturating_sub(freed_bytes);
        if !creating_gap && self.try_make_space(required_space) {
            return Ok(());
        }

        Err(MempoolError::MempoolFull)
    }
```

```rust
#[case::insufficient_eviction_space(20, false)]
#[case::sufficient_eviction_space(8, true)]
fn rejects_or_accepts_tx_based_on_freed_space(
    #[case] signature_size: usize,
    #[case] expect_success: bool,

    let tx1 = add_tx_input!(tx_hash: 1, address: "0x1", tx_nonce: 1, account_nonce: 0);

    let large_signature = vec![felt!("0x0"); signature_size];
    let large_tx = AddTransactionArgs {

            tx_hash: tx_hash!(3),
            signature: TransactionSignature(large_signature.into())
        )),
        account_state: AccountState { address: contract_address!("0x0"), nonce: nonce!(0) },
    };

    let capacity = tx1.tx.total_bytes() + tx2.tx.total_bytes();
    let mut mempool = Mempool::new(
        MempoolConfig {
            static_config: MempoolStaticConfig {
                capacity_in_bytes: capacity,
                ..Default::default()
            },
            ..Default::default()
        },

    );
```
