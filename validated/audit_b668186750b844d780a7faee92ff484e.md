Audit Report

## Title
TOCTOU Race Condition in `bridgeKernelToBSC` Silently Advances `lastBridgedDepositId` Past Unbridged Deposits â€” (`contracts/KERNEL/KernelVaultETH.sol`)

## Summary
`bridgeKernelToBSC` unconditionally sets `lastBridgedDepositId = counter - 1` using the on-chain `counter` at execution time, not the counter value the operator observed when computing `amount`. A user deposit that lands between the operator's off-chain read and the transaction's on-chain execution is permanently marked as bridged while its tokens remain in the vault, with no on-chain recovery path.

## Finding Description
The root cause is at [1](#0-0)  â€” `lastBridgedDepositId = counter - 1` uses the live on-chain `counter`, not a caller-supplied bound. The only guard is the balance check at [2](#0-1) , which verifies the vault holds enough tokens to cover `amount` but does not verify that `amount` equals the sum of all deposits from `lastBridgedDepositId+1` through `counter-1`.

The exploit path:
1. Operator reads `counter = N`, computes `amount = sum(userDeposits[0..N-1])`, submits `bridgeKernelToBSC(amount, ...)`.
2. Before the operator's tx mines, a user calls `depositKernel`, which executes `_depositKernel` at [3](#0-2) , recording `userDeposits[N]` and incrementing `counter` to `N+1`.
3. Operator's tx executes: `lastBridgedDepositId = (N+1) - 1 = N`. Only `amount` (covering deposits `0..N-1`) is bridged. Deposit `N`'s tokens remain in the vault.
4. The watermark now claims deposit `N` was bridged. The off-chain system reads `lastBridgedDepositId = N` and `counter = N+1`, computes zero pending deposits, and never triggers another bridge for deposit `N`.

There is no setter for `lastBridgedDepositId` and no rescue function in the contract, so on-chain recovery is impossible without a contract upgrade.

## Impact Explanation
Deposit `N`'s KERNEL tokens are permanently stranded in `KernelVaultETH` on Ethereum mainnet. They are never bridged to BSC and never restaked in Kernel Protocol. The user's yield-generating position is never established. This constitutes **permanent freezing of unclaimed yield** (Medium), as the tokens are locked behind a watermark that the automated bridging system will never revisit.

## Likelihood Explanation
No attacker coordination is required. Any normal user deposit that is mined in the window between the operator's off-chain counter read and the operator's transaction execution triggers the bug. Ethereum block times are ~12 seconds; the operator's read-to-submission window routinely spans multiple blocks. The condition is repeatable on every bridging cycle and requires only a coincident user deposit â€” a routine, unprivileged action.

## Recommendation
Add an `expectedLastDepositId` parameter and assert it matches `counter - 1` at execution time:

```solidity
function bridgeKernelToBSC(
    uint256 amount,
    uint256 minAmount,
    uint256 nativeFee,
    address refundAddress,
    uint256 expectedLastDepositId
) external payable nonReentrant onlyRole(OPERATOR_ROLE) {
    require(expectedLastDepositId == counter - 1, "counter mismatch");
    lastBridgedDepositId = expectedLastDepositId;
    // ... rest unchanged
}
```

This causes the transaction to revert (rather than silently mismark) if a new deposit lands before execution, allowing the operator to retry with the correct `amount`.

## Proof of Concept

```solidity
// 1. Alice deposits 100 KERNEL â†’ depositId=0, counter=1
vault.depositKernel(100e18);  // Alice

// 2. Operator reads counter=1, computes amount=100e18, submits tx

// 3. Bob deposits 50 KERNEL â†’ depositId=1, counter=2  (mines first)
vault.depositKernel(50e18);   // Bob

// 4. Operator's tx executes:
//    lastBridgedDepositId = counter-1 = 1
//    bridges only 100e18 (Alice's amount)
//    Bob's 50e18 stays in vault
vault.bridgeKernelToBSC{value: fee}(100e18, 99e18, fee, refund);

// 5. Invariant broken:
assert(lastBridgedDepositId == 1);          // watermark says Bob was bridged
assert(kernel.balanceOf(vault) >= 50e18);   // but Bob's tokens are still here

// 6. Next operator cycle: lastBridgedDepositId=1, counter=2 â†’ 0 pending â†’ Bob never bridged
```

A Foundry fuzz test parameterizing the interleaving of `depositKernel` calls between the operator's read and `bridgeKernelToBSC` execution would reliably reproduce this invariant violation.
