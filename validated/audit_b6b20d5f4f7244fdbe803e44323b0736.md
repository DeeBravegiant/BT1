Audit Report

## Title
Uninitialized `minRsEthAmountToWithdraw` Allows Dust-Request Queue Flooding, Causing Unbounded Gas Consumption and Temporary Fund Freezing - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.initiateWithdrawal` enforces no effective minimum withdrawal size when `minRsEthAmountToWithdraw[asset]` has not been explicitly set by an admin, as the mapping defaults to `0`. Any rsETH holder can flood the FIFO withdrawal queue with 1-wei requests. Because `_unlockWithdrawalRequests` iterates every queued entry in order, a flooded queue causes the operator's unlock transaction to consume unbounded gas and forces legitimate users whose requests trail the dust entries to wait until every preceding entry is processed.

## Finding Description
`minRsEthAmountToWithdraw` is declared as a plain mapping with no initialization in `initialize()`:

```solidity
// LRTWithdrawalManager.sol:35
mapping(address asset => uint256) public minRsEthAmountToWithdraw;
```

The guard in `initiateWithdrawal` is:

```solidity
// LRTWithdrawalManager.sol:162-164
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```

When `minRsEthAmountToWithdraw[asset] == 0`, the second condition `rsETHUnstaked < 0` is vacuously false for `uint256`, so any amount ≥ 1 wei passes. The secondary check at line 170 (`expectedAssetAmount > getAvailableAssetAmount`) also passes for dust amounts because integer division rounds `expectedAssetAmount` to 0, which is never greater than available assets.

Each accepted request is appended to the global FIFO queue:

```solidity
// LRTWithdrawalManager.sol:756-757
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
```

The operator's unlock path iterates every entry between `nextLockedNonce` and `firstExcludedIndex`:

```solidity
// LRTWithdrawalManager.sol:790-815
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    assetsCommitted[asset] -= request.expectedAssetAmount;
    request.expectedAssetAmount = payoutAmount;
    rsETHAmountToBurn += request.rsETHUnstaked;
    ...
    unlockedWithdrawalsCount[asset]++;
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
```

For dust entries with `expectedAssetAmount == 0`, neither the `availableAssetAmount < payoutAmount` break nor the delay break (after the 8-day delay passes) will stop the loop early. The loop must iterate every dust entry before reaching legitimate requests. If the operator passes `firstExcludedIndex = nextUnusedNonce[asset]`, the transaction runs out of gas. Even with a bounded `firstExcludedIndex`, the operator must issue many transactions to drain the dust entries, delaying legitimate users' withdrawals.

## Impact Explanation
**Medium — Unbounded gas consumption / Temporary freezing of funds.**

The `_unlockWithdrawalRequests` loop's gas cost scales linearly with the number of queued entries. A sufficiently large flood causes the operator's unlock transaction to revert out-of-gas. Even with chunked processing, legitimate users whose requests are queued behind dust entries cannot complete their withdrawals until all preceding entries are drained, constituting a temporary freeze of their funds. Both impacts are in the allowed scope.

## Likelihood Explanation
Any rsETH holder can call `initiateWithdrawal` with 1-wei amounts. The only cost to the attacker is gas per transaction and the temporary lock-up of negligible rsETH (recovered after the withdrawal delay). No special role or coordination is required. The attack is repeatable and externally reachable.

## Recommendation
1. Initialize `minRsEthAmountToWithdraw` to a sensible non-zero default (e.g., `1e15` wei) in `initialize()` for each supported asset, or require it to be set before an asset is usable for withdrawals.
2. Strengthen the guard to ensure the minimum is always non-zero:
   ```solidity
   if (rsETHUnstaked == 0 || minRsEthAmountToWithdraw[asset] == 0
       || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
       revert InvalidAmountToWithdraw();
   }
   ```
3. Consider adding a per-user cap on pending withdrawal requests to further limit queue flooding.

## Proof of Concept
1. Deploy `LRTWithdrawalManager`; support asset `ETH`. `minRsEthAmountToWithdraw[ETH]` is `0` (never set by admin).
2. Attacker acquires N wei of rsETH (N can be as small as 10,000 wei).
3. Attacker calls `initiateWithdrawal(ETH, 1, "")` N times. Each call passes both checks (1 ≠ 0; 1 < 0 is false). Each call appends a new entry to the queue and increments `nextUnusedNonce[ETH]`.
4. Legitimate user calls `initiateWithdrawal(ETH, 1 ether, "")`. Their request is at nonce N.
5. After the 8-day withdrawal delay, operator calls `unlockQueue` with `firstExcludedIndex = nextUnusedNonce[ETH]` (N+1). The `_unlockWithdrawalRequests` while loop iterates all N+1 entries, each performing multiple storage reads/writes, consuming gas proportional to N. For large N, the transaction reverts out-of-gas.
6. Even with chunked `firstExcludedIndex`, the operator must drain all N dust entries before the legitimate user's request at nonce N can be unlocked, temporarily freezing their funds.
