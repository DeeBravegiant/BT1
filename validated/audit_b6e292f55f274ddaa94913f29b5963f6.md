## Vulnerability Analysis

### Tracing the Attack Path

**Step 1: Entry point — `_add_coin_states` in `wallet_state_manager.py`**

When a DL singleton coin state arrives as spent, the code fetches the coin spend from the peer and calls `singleton_removed`: [1](#0-0) 

**Step 2: `fetch_coin_spend` — what is and is NOT validated**

`fetch_coin_spend` validates the puzzle against the coin's known `puzzle_hash`, and validates the `coin_name`. It does **not** validate the solution: [2](#0-1) 

The puzzle is pinned to the real on-chain DL singleton puzzle (attacker cannot substitute a different puzzle). However, the **solution is entirely attacker-controlled**.

**Step 3: `singleton_removed` — the vulnerable extraction**

The function runs the real puzzle with the attacker-supplied solution, then blindly trusts the hint field: [3](#0-2) 

`root = bytes32(condition[3][1])` and `inner_puzzle_hash = bytes32(condition[3][2])` come from the hint field of the CREATE_COIN condition. Hints are optional metadata — the DL singleton puzzle does not enforce that they match the values used to compute `condition[1]` (the child puzzle hash).

**Step 4: Unchecked write to the DB**

The extracted `root` and `inner_puzzle_hash` are written directly to `singleton_records` with no cross-validation: [4](#0-3) 

**The missing guard:** There is no check that:
```python
create_host_fullpuz(inner_puzzle_hash, root, launcher_id).get_tree_hash_precalc(inner_puzzle_hash) == full_puzzle_hash
```

This cross-check exists in `match_dl_launcher` for the genesis spend: [5](#0-4) 

But it is absent in `singleton_removed` for all subsequent spends.

**Step 5: `run_with_cost` does not validate BLS signatures**

CLVM execution produces conditions (including AGG_SIG_ME) but does not verify them. The attacker can craft a solution with an invalid signature that still runs to completion and produces a CREATE_COIN with attacker-chosen hints. Signature validation only happens at the consensus layer (full node), not in the wallet's sync path.

### Verdict

### Title
Attacker-Controlled Solution Allows Corruption of DL Singleton Root/InnerPuzzleHash in Wallet DB — (`chia/data_layer/data_layer_wallet.py`)

### Summary
An unprivileged peer in untrusted mode can return a crafted solution for a spent DL singleton coin. Because `fetch_coin_spend` validates only the puzzle (not the solution), and `singleton_removed` extracts `root` and `inner_puzzle_hash` from the hint field of the CREATE_COIN condition without cross-checking them against the child coin's actual puzzle hash, the attacker can write arbitrary values into the `singleton_records` table.

### Finding Description
In `DataLayerWallet.singleton_removed` (`chia/data_layer/data_layer_wallet.py`, lines 828–829), `root` and `inner_puzzle_hash` are read from `condition[3][1]` and `condition[3][2]` — the hint field of the first odd CREATE_COIN condition produced by running the puzzle with the peer-supplied solution. No check is performed to verify that `create_host_fullpuz(inner_puzzle_hash, root, launcher_id)` hashes to `full_puzzle_hash` (condition[1]). Since `run_with_cost` does not validate BLS signatures, the attacker can supply any solution that the CLVM accepts syntactically, including one with an invalid signature, causing the puzzle to emit a CREATE_COIN with attacker-chosen hints. The resulting `SingletonRecord` with corrupted `root` and `inner_puzzle_hash` is written via `INSERT OR REPLACE`, overwriting any prior correct record. [6](#0-5) 

### Impact Explanation
The wallet's local `singleton_records` table stores the DL root hash that the wallet uses to construct future spends (`create_update_state_spend` reads `singleton_record.root` directly). A corrupted root causes the wallet to build spends against a non-existent puzzle hash, permanently breaking the ability to update or spend the DL singleton. It also causes `get_coins_to_offer` to compute a wrong coin identity, breaking DL offer settlement. The `lineage_proof` stored at line 854–857 is also computed from the fake values, propagating the corruption to child spends. [7](#0-6) 

### Likelihood Explanation
Requires the wallet to be in untrusted mode (no trusted peers configured) and the attacker to be connected as a peer. The attacker must know the launcher ID of a tracked DL singleton (observable from the blockchain). No key material is needed. The crafted solution only needs to pass CLVM execution (not signature verification), which is straightforward.

### Recommendation
After extracting `full_puzzle_hash`, `root`, and `inner_puzzle_hash` from the conditions, add the cross-check that already exists in `match_dl_launcher`:

```python
expected_hash = create_host_fullpuz(
    inner_puzzle_hash, root, singleton_record.launcher_id
).get_tree_hash_precalc(inner_puzzle_hash)
if full_puzzle_hash != expected_hash:
    self.log.warning("Hint mismatch in singleton spend; ignoring")
    return
```

This ensures the stored root and inner_puzzle_hash are cryptographically committed to by the actual child coin's puzzle hash, regardless of what the peer claims in the hint field.

### Proof of Concept
1. Set up a wallet in untrusted mode tracking a DL singleton with launcher ID `L`.
2. On-chain, legitimately spend the singleton to a child with root `R_real` and inner puzzle hash `IPH_real`.
3. Intercept the `request_puzzle_solution` call for the spent singleton coin.
4. Return a `RespondPuzzleSolution` with the correct puzzle (matching `coin.puzzle_hash`) but a crafted solution whose inner solution causes the NFT state layer to emit a CREATE_COIN with hints `[_, R_fake, IPH_fake]` where `R_fake ≠ R_real`.
5. Assert that `dl_store.get_latest_singleton(L).root == R_fake` — the corrupted value is stored.
6. Attempt `create_update_state_spend` — it will construct a spend using `R_fake`, producing a puzzle hash that does not match any on-chain coin, causing the spend to be rejected by the mempool.

        FullNodeAPI.request_puzzle_solution, RequestPuzzleSolution(coin.name(), height)

        raise PeerRequestException(f"Peer returned wrong coin name in puzzle solution for coin {coin_id}")

    return make_spend(

        solution_response.response.solution,
    )
