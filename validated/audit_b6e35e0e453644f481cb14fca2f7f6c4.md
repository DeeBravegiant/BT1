### Title
Missing `app_pk` Validation in Non-PV CKD Variant Allows Masking Bypass — (`crates/threshold-signatures/src/confidential_key_derivation/protocol.rs`)

---

### Summary

The non-PV CKD variant (`protocol.rs`) accepts `app_pk` as a raw `ElementG1` with no validation. An attacker can submit `app_pk = G1::identity()` through the on-chain `request_app_private_key` entrypoint. Because the identity element annihilates the masking term, each participant's `big_c` collapses to the unmasked share `big_s`, and the coordinator aggregates and returns `msk · H(pk ∥ app_id)` in plaintext — the derived key — without the requester ever possessing `app_sk`.

---

### Finding Description

**Entrypoint — contract (`lib.rs`)**

`request_app_private_key` dispatches on the `CKDAppPublicKey` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check
    dtos::CKDAppPublicKey::AppPublicKeyPV(pk) => {
        if !app_public_key_check(pk) {
            env::panic_str("app public key check failed")
        }
    }
}
``` [1](#0-0) 

The `AppPublicKey` arm is a no-op. Any G1 point — including the identity — passes through.

**MPC node dispatch (`sign.rs`)**

The node converts the stored point and calls the non-PV protocol directly:

```rust
dtos::CKDAppPublicKey::AppPublicKey(pk) => {
    let protocol = ckd(
        ...
        ElementG1::try_from(&pk)?,   // identity converts fine
        OsRng,
    )?;
``` [2](#0-1) 

`ElementG1::try_from` does not reject the identity point; the BLS12-381 identity has a valid compressed encoding and decompresses without error.

**Per-participant computation (`protocol.rs`)**

```rust
let big_c = big_s + app_pk * y.0;   // app_pk = identity → big_c = big_s
``` [3](#0-2) 

When `app_pk = G1::identity()`, `app_pk * y = identity`, so `big_c = big_s = x_i · H(pk ∥ app_id)`. After Lagrange normalization each participant sends `λ_i · x_i · H(pk ∥ app_id)` to the coordinator. [4](#0-3) 

**Coordinator aggregation**

```rust
norm_big_y += participant_output.big_y();
norm_big_c += participant_output.big_c();
``` [5](#0-4) 

The sum `Σ λ_i · x_i · H = msk · H(pk ∥ app_id)` is returned as `CKDOutput.big_c` — the derived key in plaintext.

**Contrast with the PV variant**

`protocol_pv.rs` validates `app_pk` before running the protocol:

```rust
if !app_public_key_check(&app_pk) {
    return Err(InitializationError::BadParameters(...));
}
``` [6](#0-5) 

And after aggregation it runs `aggregated_output_check` (a pairing equation). Neither guard exists in the non-PV path.

**Note on `check_valid_point_g1`**

Even if a point-validity check were added, it would not help: the function only tests `is_on_curve() & is_torsion_free()`, both of which are trivially true for the identity element. [7](#0-6) 

---

### Impact Explanation

The attacker receives `msk · H(pk ∥ app_id)` — the BLS derived key — without holding `app_sk`. This is the secret material the CKD protocol is designed to keep confidential. Possession of the derived key allows the attacker to use it as a BLS signing key for that `app_id`, impersonating the legitimate key holder for any downstream protocol that relies on it.

**Correction to the question's claim:** The attack does *not* recover individual signing shares `x_i`. The coordinator sums the Lagrange-weighted contributions before returning the result; the attacker sees only the final aggregate `msk · H`. Recovering `x_i` from that would require solving the discrete-log problem.

---

### Likelihood Explanation

The attack requires only a standard on-chain call to `request_app_private_key` with a well-formed `AppPublicKey` payload whose G1 bytes encode the identity point. No threshold collusion, TEE access, or privileged position is needed. The identity point has a valid compressed BLS12-381 encoding and passes every deserialization step.

---

### Recommendation

1. **Reject the identity point for `AppPublicKey`** in `request_app_private_key` (mirror the `AppPublicKeyPV` branch).
2. **Add a non-identity check inside `ckd()` in `protocol.rs`** as a defense-in-depth guard before the protocol runs.
3. **Document intent explicitly**: the existing comment states identity points are accepted in `AppPublicKeyPV` for intentionally-public derived keys; if the same intent applies to `AppPublicKey`, document it and make the behavior explicit rather than accidental.

---

```
