Audit Report

## Title
`instantWithdrawal()` Static Buffer Allows Draining Vault of Queued Withdrawal Commitments, Temporarily Freezing Queued Users' Funds - (File: contracts/LRTWithdrawalManager.sol)

## Summary

`LRTWithdrawalManager.instantWithdrawal()` gates vault access using `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal()`, which subtracts only the static, operator-set `queuedWithdrawalsBuffer` from the vault balance. This buffer defaults to zero and is never automatically updated when `assetsCommitted` grows via `initiateWithdrawal()`. Any user can therefore drain the vault of assets already committed to pending queued withdrawals, causing `unlockQueue()` to revert and temporarily freezing queued withdrawal users' rsETH inside `LRTWithdrawalManager`.

## Finding Description

**Two withdrawal paths share the same `LRTUnstakingVault`:**

**Path 1 – Queued withdrawal (`initiateWithdrawal` → `unlockQueue`):**
- `initiateWithdrawal` transfers the user's rsETH into `LRTWithdrawalManager` and increments `assetsCommitted[asset]` by the expected payout amount.
- `unlockQueue` calls `_createUnlockParams`, which sets `totalAvailableAssets = unstakingVault.balanceOf(asset)` — the raw vault balance — and then calls `unstakingVault.redeem()` to pull funds.

**Path 2 – Instant withdrawal (`instantWithdrawal`):**
- Checks `unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)`, which computes `vaultBalance - queuedWithdrawalsBuffer[asset]`.
- `queuedWithdrawalsBuffer` is a static mapping that defaults to `0` and is only updated by an explicit operator call to `setQueuedWithdrawalsBuffer`.

**Root cause:** `assetsCommitted[asset]` grows every time a user calls `initiateWithdrawal`, but `queuedWithdrawalsBuffer[asset]` remains `0` (or stale). `getAssetsAvailableForInstantWithdrawal` therefore returns the full vault balance, making the entire vault available for instant withdrawal even when all of it is already committed to queued users.

**Exploit flow:**
1. Vault holds 100 ETH; `queuedWithdrawalsBuffer[ETH]` = 0 (default).
2. Users call `initiateWithdrawal` for a total of 100 ETH → `assetsCommitted[ETH]` = 100 ETH; their rsETH is locked in `LRTWithdrawalManager` at line 166.
3. An attacker calls `instantWithdrawal` for 100 ETH. The check at line 231 passes: `getAssetsAvailableForInstantWithdrawal` = 100 − 0 = 100 ETH. The vault is drained to 0.
4. Operator calls `unlockQueue`. `_createUnlockParams` reads `unstakingVault.balanceOf(asset)` = 0, so `totalAvailableAssets` = 0. `unlockQueue` immediately reverts at line 297: `if (params.totalAvailableAssets == 0) revert AmountMustBeGreaterThanZero()`.
5. Queued withdrawal users' rsETH remains locked in `LRTWithdrawalManager` with no path to completion until operators manually refill the vault.

**Why existing checks fail:**
- `getAssetsAvailableForInstantWithdrawal` only subtracts `queuedWithdrawalsBuffer`, not `assetsCommitted`. These are entirely separate accounting variables with no automatic synchronization.
- `setQueuedWithdrawalsBuffer` is a manual operator action; it is never called automatically when `initiateWithdrawal` increments `assetsCommitted`.
- There is no check in `instantWithdrawal` against `assetsCommitted[asset]`.

## Impact Explanation

**Medium – Temporary freezing of funds.**

Queued withdrawal users have already transferred their rsETH into `LRTWithdrawalManager` and cannot retrieve it until `unlockQueue` succeeds. With the vault drained, `unlockQueue` reverts unconditionally. Funds are not permanently lost — the protocol still holds equivalent assets in EigenLayer/NDCs — but users are frozen until operators manually move assets back into the vault. This constitutes a temporary freeze of user funds, matching the allowed impact class.

## Likelihood Explanation

**High.** `queuedWithdrawalsBuffer` defaults to `0` for all assets (Solidity mapping default). Any user holding rsETH can call `instantWithdrawal` at any time when instant withdrawal is enabled for an asset. No special role, privilege, or coordination is required. The condition is reachable whenever the vault holds funds that are simultaneously committed to queued withdrawals, which is the normal operating state of the protocol.

## Recommendation

Replace the static `queuedWithdrawalsBuffer` check in `getAssetsAvailableForInstantWithdrawal` with a dynamic check that reads the actual `assetsCommitted` from `LRTWithdrawalManager`. Specifically, `LRTUnstakingVault.getAssetsAvailableForInstantWithdrawal` should subtract `ILRTWithdrawalManager(withdrawalManager).assetsCommitted(asset)` from the vault balance instead of (or in addition to) the manually maintained buffer. Alternatively, `instantWithdrawal` in `LRTWithdrawalManager` can directly compare `assetAmountUnlocked` against `unstakingVault.balanceOf(asset) - assetsCommitted[asset]` before proceeding.
