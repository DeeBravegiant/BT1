All four code-level claims check out against the production source. The vulnerability is confirmed.

Audit Report

## Title
Unpermissioned `addLiquidity` Allows Anyone to Inject Shares Into Any Position, Permanently Disrupting the Owner's `removeLiquidity` Exit — (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/libraries/LiquidityLib.sol`)

## Summary
`MetricOmmPool.addLiquidity` accepts an arbitrary `owner` address with no `msg.sender == owner` guard, while `removeLiquidity` enforces that check. Any caller implementing `IMetricOmmModifyLiquidityCallback` can credit LP shares to a victim's position while paying the required tokens themselves. Because `LiquidityLib.removeLiquidity` reverts when the remaining share count is non-zero but below `minimalMintableLiquidity`, a single dust-share injection before a victim's "remove-all" transaction causes that transaction to revert with `MinimalLiquidity`, locking the victim's principal until they detect and correct the injected share count.

## Finding Description
`addLiquidity` in `MetricOmmPool` takes an arbitrary `owner` parameter with no caller identity check: [1](#0-0) 

`removeLiquidity` has the symmetric guard at line 206: [2](#0-1) 

Inside `LiquidityLib.addLiquidity`, the payment callback is dispatched to `msg.sender` (the attacker's contract), not to `owner`. The attacker pays the tokens; the shares are credited to the victim's position key: [3](#0-2) 

`LiquidityLib.removeLiquidity` enforces the `minimalMintableLiquidity` floor on the *remaining* share count after removal: [4](#0-3) 

**Attack flow:**
1. Victim holds `X` shares (`X ≥ minimalMintableLiquidity`) and broadcasts `removeLiquidity(owner, salt, X shares)`.
2. Attacker front-runs with `pool.addLiquidity(victim, salt, 1 share, ...)`, paying the required tokens in the callback.
3. Victim's position now holds `X + 1` shares.
4. Victim's `removeLiquidity(X shares)` leaves `1` share. Since `1 < minimalMintableLiquidity`, the call reverts with `MinimalLiquidity(1, minimalMintableLiquidity)`.
5. Victim's principal remains locked until they detect the injected share and resubmit with the corrected total.

The optional `DepositAllowlistExtension` does not mitigate this: it checks whether `owner` (the victim) is an allowed depositor, not whether `sender` (the attacker) is authorized to deposit on behalf of `owner`. Since the victim already holds a position, they are necessarily allowed, and the attacker's injection passes the extension check. [5](#0-4) 

## Impact Explanation
The victim's "remove all" exit — the most common LP withdrawal pattern — can be persistently blocked by an unprivileged third party at the cost of a dust token deposit per block. The victim's principal is not stolen but is effectively frozen until they detect the injected shares and resubmit with the exact current total. This constitutes broken core pool functionality causing an unusable withdraw/liquidity flow, satisfying the allowed impact gate.

## Likelihood Explanation
`addLiquidity` is a public, permissionless function with no factory or role gate. The attacker only needs to deploy a trivial callback contract and hold a dust amount of the pool's tokens. The attack is repeatable: every corrected victim transaction can be front-run again. The victim has no on-chain mechanism to prevent share injection into their position.

## Recommendation
Add a caller-identity check to `addLiquidity` symmetric with the one already present in `removeLiquidity`:

```solidity
// In MetricOmmPool.addLiquidity, before calling LiquidityLib:
if (msg.sender != owner) revert NotPositionOwner();
```

If the protocol intends to allow approved routers (e.g., `MetricOmmPoolLiquidityAdder`) to add liquidity on behalf of users, introduce an explicit allowlist or a per-owner approval mapping rather than leaving the function fully open. The `DepositAllowlistExtension` checks `owner`, not `sender`, and does not close this gap.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IMetricOmmPoolActions} from "metric-core/interfaces/IMetricOmmPool/IMetricOmmPoolActions.sol";
import {IMetricOmmModifyLiquidityCallback} from "metric-core/interfaces/callbacks/IMetricOmmModifyLiquidityCallback.sol";
import {LiquidityDelta} from "metric-core/types/PoolOperation.sol";
import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";

contract DustInjector is IMetricOmmModifyLiquidityCallback {
    address immutable pool;
    address immutable token0;
    address immutable token1;

    constructor(address _pool, address _token0, address _token1) {
        pool = _pool; token0 = _token0; token1 = _token1;
        IERC20(_token0).approve(_pool, type(uint256).max);
        IERC20(_token1).approve(_pool, type(uint256).max);
    }

    function attack(address victim, uint80 salt, int256 binIdx) external {
        int256[] memory bins = new int256[](1);
        bins[0] = binIdx;
        uint256[] memory shares = new uint256[](1);
        shares[0] = 1;
        LiquidityDelta memory delta = LiquidityDelta({binIdxs: bins, shares: shares});
        // No msg.sender == owner check — succeeds for any victim address.
        IMetricOmmPoolActions(pool).addLiquidity(victim, salt, delta, "", "");
    }

    function metricOmmModifyLiquidityCallback(
        uint256 amount0Delta, uint256 amount1Delta, bytes calldata
    ) external override {
        if (amount0Delta > 0) IERC20(token0).transfer(msg.sender, amount0Delta);
        if (amount1Delta > 0) IERC20(token1).transfer(msg.sender, amount1Delta);
    }
}
```

**Expected result:** After `attack()` is mined before the victim's `removeLiquidity(X shares)`, the victim's call reverts with `MinimalLiquidity(1, minimalMintableLiquidity)`. The victim must discover the injected share and resubmit with `X + 1` shares to exit cleanly. The attacker can repeat this indefinitely.
