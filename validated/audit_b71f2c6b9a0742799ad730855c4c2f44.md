Audit Report

## Title
Aave WETH Pool 100% Utilization Causes Temporary Freezing of ETH Withdrawal Funds — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`_withdrawFromAave` validates only the aToken accounting balance (`aaveAWETH.balanceOf(address(this))`), which remains positive regardless of borrow utilization, but does not check the underlying WETH liquidity available in the Aave pool. When external borrowers exhaust the Aave v3 WETH pool, `aaveWETHGateway.withdrawETH` reverts at the Aave level, propagating uncaught through `_processWithdrawalCompletion` and blocking all unlocked ETH withdrawal completions until liquidity is restored.

## Finding Description
In `_processWithdrawalCompletion`, state mutations (`popFront`, `delete withdrawalRequests[requestId]`, `unlockedWithdrawalsCount--`) occur before the Aave call at lines 705–717. When `_withdrawFromAave` is called at line 724, it checks only `aaveAWETH.balanceOf(address(this))` at line 908 — the aToken balance — which accrues interest and stays positive even at 100% pool utilization. It then calls `aaveWETHGateway.withdrawETH` at line 917 without verifying that the Aave pool holds sufficient underlying WETH. If the pool is fully borrowed out, this call reverts at the Aave level. The revert propagates uncaught, rolling back all state changes and leaving the user's request intact but uncompleable.

The protocol already exposes `getAaveWithdrawableLiquidity()` (lines 684–689), which returns `IERC20(WETH_ADDRESS).balanceOf(address(aaveAWETH))` — the exact value needed — but this is never consulted in the withdrawal path. The deposit path (`depositToAaveExternal`, lines 507–516) uses a `try/catch` pattern for resilience against Aave-level failures, but no equivalent guard exists on the withdrawal path. The `InsufficientLiquidityForWithdrawal` revert at line 729 is unreachable in this scenario because it is placed after the already-reverted Aave call.

## Impact Explanation
All users with unlocked ETH withdrawal requests are unable to call `completeWithdrawal` for as long as the Aave WETH pool remains at 100% utilization. Funds are not permanently lost — the revert rolls back all state — but users cannot access their ETH without operator intervention (`emergencyWithdrawFromAave`, which requires `PAUSER_ROLE`) or waiting for Aave liquidity to be replenished by external repayments. This matches **Medium — Temporary freezing of funds**.

## Likelihood Explanation
No attacker action is required. Normal market borrowing activity on Aave v3 mainnet is sufficient. Aave v3 WETH regularly reaches high utilization (>90%) during market stress events, and 100% utilization (zero withdrawable liquidity) is a documented, historically observed Aave state. The condition is observable on-chain via `getAaveWithdrawableLiquidity()`. Any user with an unlocked ETH withdrawal request can trigger the revert simply by calling `completeWithdrawal` during such a period.

## Recommendation
Before calling `aaveWETHGateway.withdrawETH` in `_withdrawFromAave`, check the underlying WETH liquidity using the already-available `getAaveWithdrawableLiquidity()` logic:

```solidity
uint256 availableLiquidity = IERC20(WETH_ADDRESS).balanceOf(address(aaveAWETH));
if (availableLiquidity < withdrawnAmount) revert InsufficientLiquidityForWithdrawal();
```

Alternatively, wrap the `aaveWETHGateway.withdrawETH` call in a `try/catch` (mirroring the pattern already used in `depositToAaveExternal`) and surface a clear, protocol-defined error rather than propagating an opaque Aave revert.

## Proof of Concept
1. Fork mainnet at a recent block with `LRTWithdrawalManager` deployed and `isAaveIntegrationEnabled = true` with ETH deposited to Aave.
2. Use `deal` (Foundry) to set the WETH balance of the aWETH contract to 0, simulating 100% borrow utilization.
3. Assert `getAaveWithdrawableLiquidity() == 0` while `aaveAWETH.balanceOf(withdrawalManager) > 0`.
4. Ensure a user has an unlocked ETH withdrawal request where `address(withdrawalManager).balance < request.expectedAssetAmount`.
5. Call `completeWithdrawal(ETH_TOKEN, "")` from that user.
6. Assert the transaction reverts with an Aave-level error (not `InsufficientLiquidityForWithdrawal`).
7. Assert the user's withdrawal request remains intact in the queue (all state rolled back).
