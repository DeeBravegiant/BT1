### Title
Unconstrained `fee_proposal_fri` During Initialization Window Enables Permanent L2 Gas Price Manipulation — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The SNIP-35 fee-market mechanism enforces that each proposer's `fee_proposal_fri` must lie within a multiplicative margin of `fee_actual` (the median of the last `window_size` proposals). However, this bound is **explicitly skipped** whenever `fee_actual` is `None` — which occurs for the first `window_size` blocks of the chain (genesis) and for the first `window_size` blocks after the V0_14_3 protocol upgrade (when the window contains pre-V0_14_3 `None` entries). During this unguarded window, any proposer may inject an arbitrarily extreme `fee_proposal_fri` value. Validators accept it without objection. Once the window fills with enough extreme values, `fee_actual` becomes extreme, permanently driving the L2 gas price toward `u128::MAX` (all transactions fail) or toward `1` (chain is spammable at zero cost).

---

### Finding Description

**Root cause — `is_proposal_init_valid` skips bounds when `fee_actual` is `None`:**

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(ValidateProposalError::InvalidProposalInit(...));
    }
}
``` [1](#0-0) 

`fee_actual` is `None` whenever `compute_fee_actual` cannot produce a median — specifically when `height < window_size` or when any entry in the window is `None` (pre-V0_14_3 blocks):

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
// ...
Some(None) | None => {
    warn!("Cannot compute fee_actual for height {height}: fee_proposals_window has no recorded fee_proposal for height {source_height}");
    return None;
}
``` [2](#0-1) 

The validator computes `fee_actual` from its own `fee_proposals_window` in the same way:

```rust
fee_actual: compute_fee_actual(
    &self.fee_proposals_window,
    init.height,
    VersionedConstants::latest_constants().fee_proposal_window_size,
),
``` [3](#0-2) 

So both proposer and validator independently skip the bounds check during the same initialization window. Any `fee_proposal_fri` value — including `u128::MAX` or `1` — is accepted by all validators.

**Propagation path — `fee_proposal_fri` → `fee_actual` → L2 gas price floor:**

After a block is decided, `fee_proposal_fri` is recorded in the window:

```rust
self.update_l2_gas_price(height, l2_gas_used);
self.record_fee_proposal(height, init.fee_proposal_fri);
``` [4](#0-3) 

`fee_actual` (the median of the window) is then used as a hard floor for the minimum gas price in the EIP-1559 adjustment:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
``` [5](#0-4) 

If `effective_min = u128::MAX`, `calculate_next_base_gas_price` enters the "price below minimum" branch and ratchets the L2 gas price upward by `price / 333` per block until it reaches `u128::MAX`:

```rust
if price < min_gas_price {
    let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
    let adjusted = price.0 + max_increase;
    let adjusted_price = adjusted.min(min_gas_price.0);
    return GasPrice(adjusted_price);
}
``` [6](#0-5) 

The resulting `l2_gas_price` is placed in the next `ProposalInit.l2_gas_price_fri`, which validators check for exact equality:

```rust
&& init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri
``` [7](#0-6) 

Because both proposer and validator derive `l2_gas_price` from the same (now-corrupted) `fee_actual`, the inflated price is accepted by all validators, locking in the distortion permanently.

**`fee_proposal_fri` is also bound into the `ProposalCommitment`:**

```rust
ProposalCommitment(Poseidon::hash_array(&[partial.0, Felt::from(fee_proposal.0)]))
``` [8](#0-7) 

Validators vote on this commitment, so they are explicitly signing over the extreme `fee_proposal_fri` value without any bounds check during the initialization window.

---

### Impact Explanation

Once the `fee_proposals_window` is filled with extreme values (e.g., `u128::MAX`), `fee_actual` becomes `u128::MAX`. The EIP-1559 adjustment then ratchets `l2_gas_price` upward by ~0.3% per block until it reaches `u128::MAX`. At that point, every transaction's fee computation overflows or exceeds any user's balance, causing all transactions to revert or be rejected at the gateway. The chain becomes permanently unusable for all users until a governance intervention resets the gas price. This matches the **Critical** impact category: "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."

Conversely, setting `fee_proposal_fri = 1` drives `fee_actual` to `1 fri`, collapsing the L2 gas price floor to near zero and enabling unlimited spam at negligible cost.

---

### Likelihood Explanation

The attack window is bounded to:
1. **Genesis**: the first `window_size` blocks of the chain, when `height < window_size`.
2. **V0_14_3 transition**: the first `window_size` blocks after the upgrade, when the window contains pre-V0_14_3 `None` entries.

To set the median to an extreme value, the attacker must control **more than half** of the proposals in the window. In a BFT system with round-robin proposer selection, this requires controlling >50% of the validator set during the window. At genesis with a small initial validator set, or if a single operator controls the sequencer (as is currently the case for Starknet), this is a realistic scenario. The V0_14_3 transition is a particularly concrete trigger because it is a known, scheduled event.

---

### Recommendation

1. **Enforce an absolute cap on `fee_proposal_fri`**: Reject any `fee_proposal_fri` above a protocol-defined maximum (e.g., `max_l2_gas_price`) regardless of whether `fee_actual` is available. This prevents extreme values from entering the window even during initialization.

2. **Seed the window at genesis/upgrade with a safe default**: Before the first V0_14_3 block is proposed, pre-populate the `fee_proposals_window` with `window_size` copies of the configured `min_gas_price` (or the last known `l2_gas_price`). This ensures `fee_actual` is always `Some` from the first V0_14_3 block onward, so the bounds check is never skipped.

3. **Clamp `fee_proposal_fri` to `[min_gas_price, max_gas_price]` unconditionally** in `is_proposal_init_valid`, analogous to how Uniswap V2 burns a minimum liquidity amount at genesis to prevent the first-depositor attack.

### Proof of Concept

**Setup**: Chain starts at genesis (height 0). `window_size = 10`. Single malicious validator controls all proposals for heights 0–9.

**Step 1**: For each of heights 0–9, the malicious proposer emits `ProposalInit { fee_proposal_fri: Some(GasPrice(u128::MAX)), ... }`.

**Step 2**: Each validator calls `compute_fee_actual` for height 0–9. Since `height < window_size`, it returns `None`. The bounds check in `is_proposal_init_valid` is skipped (the `if let (Some(fee_actual), Some(fee_proposal))` arm does not match). All proposals are accepted.

**Step 3**: After height 9 is decided, `fee_proposals_window` contains `{0: Some(u128::MAX), 1: Some(u128::MAX), ..., 9: Some(u128::MAX)}`.

**Step 4**: At height 10, `compute_fee_actual` returns `Some(GasPrice(u128::MAX))` (median of 10 identical values). `calculate_next_l2_gas_price_for_fin` sets `effective_min = u128::MAX`. `calculate_next_base_gas_price` enters the "price below minimum" branch and returns `GasPrice(u128::MAX)`.

**Step 5**: `self.l2_gas_price = u128::MAX`. The next `ProposalInit.l2_gas_price_fri = u128::MAX`. All validators accept this (exact equality check passes). Every subsequent block has `l2_gas_price = u128::MAX`. All user transactions fail fee payment. The chain is permanently halted for users.
