### Title
Griefer Can Front-Run `SignedDelegateAction` to Block Meta-Transaction Execution - (File: `runtime/runtime/src/actions.rs`)

### Summary

An unprivileged attacker can observe a `SignedDelegateAction` (meta-transaction) embedded in a relayer's outer transaction in the public mempool, extract it, and re-wrap it in their own outer transaction. If the attacker's transaction is included first, the delegate action's nonce is consumed on-chain, causing the original relayer's transaction to fail with `DelegateActionInvalidNonce`. This is the direct protocol-layer analog of the EIP-2612 `delegateBySig` front-running pattern.

### Finding Description

NEAR's meta-transaction mechanism (NEP-366) allows a user (Alice) to sign a `DelegateAction` off-chain and hand it to a relayer. The relayer wraps the `SignedDelegateAction` inside a normal `SignedTransaction` and broadcasts it to the network. [1](#0-0) [2](#0-1) 

The outer `SignedTransaction` is gossiped to all peers and sits in the public mempool. Because the `SignedDelegateAction` is fully serialized inside the transaction body, any observer can deserialize it, re-wrap it in a fresh outer transaction signed by themselves, and submit it to the network. [3](#0-2) 

When `apply_delegate_action` is called, it invokes `validate_delegate_action_key`, which checks the nonce and — if valid — **immediately advances it** in the trie: [4](#0-3) [5](#0-4) 

If the attacker's outer transaction is included in a chunk first, the access-key nonce is advanced to `delegate_nonce`. When the original relayer's transaction is subsequently executed, `delegate_nonce.nonce() <= current_nonce` is true and the receipt fails with `DelegateActionInvalidNonce`. The relayer's gas is burned; the user's intended inner actions are never executed.

### Impact Explanation

The concrete corrupted protocol value is the **action receipt outcome**: it is recorded as a failure (`DelegateActionInvalidNonce`) instead of the success that would have spawned the inner-action receipt. The user's intended on-chain operation (token transfer, contract call, key addition, etc.) is suppressed. A persistent attacker who monitors the mempool and re-submits the same `SignedDelegateAction` for every new nonce the user signs can indefinitely block the user from executing any meta-transaction, wasting relayer gas each time.

### Likelihood Explanation

The attack requires only an unprivileged NEAR account and the ability to submit transactions via the public RPC. The `SignedDelegateAction` is fully visible in the mempool because it is borsh-serialized inside the outer transaction body. No special infrastructure is needed beyond a mempool-watching script and a funded account to pay the outer transaction's gas.

### Recommendation

Bind the `SignedDelegateAction` to a specific relayer by including the relayer's `account_id` (or the outer transaction's `signer_id`) inside the signed payload. Because the relayer field would be covered by Alice's signature, an attacker who re-wraps the payload in their own outer transaction would fail the signature check (`DelegateActionInvalidSignature`) rather than the nonce check. Alternatively, wrap the nonce-consuming call in a try-catch equivalent at the receipt level so that a nonce collision does not burn the relayer's gas and the relayer can retry.

### Proof of Concept

1. Alice signs a `DelegateAction` (nonce = N, `max_block_height` = H) and sends it off-chain to a relayer.
2. The relayer constructs `outer_tx_relayer = SignedTransaction { actions: [Action::Delegate(signed_delegate_action)], signer: relayer, ... }` and broadcasts it.
3. Attacker observes `outer_tx_relayer` in the mempool, deserializes the `SignedDelegateAction`, and constructs `outer_tx_attacker = SignedTransaction { actions: [Action::Delegate(same_signed_delegate_action)], signer: attacker, receiver: alice, ... }`.
4. Attacker submits `outer_tx_attacker` with sufficient gas. Both transactions are valid at the mempool-admission stage because the nonce check there uses the **current** certified state (nonce = N-1).
5. `outer_tx_attacker` is included in chunk C. `apply_delegate_action` → `validate_delegate_action_key` passes (nonce N > N-1), advances the access-key nonce to N, and emits the inner-action receipt. Alice's intended action executes — but with the attacker as the outer signer/gas payer, not the relayer.
6. `outer_tx_relayer` is included in chunk C+k. `validate_delegate_action_key` now finds `current_nonce = N`, so `delegate_nonce.nonce() (= N) <= current_nonce (= N)` → `DelegateActionInvalidNonce`. The relayer's gas is burned; no inner receipt is emitted. [4](#0-3)

pub struct SignedDelegateAction {
    pub delegate_action: DelegateAction,
    pub signature: Signature,
}

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }
}

pub(crate) fn apply_delegate_action(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    action_receipt: &VersionedActionReceipt,
    sender_id: &AccountId,
    signed_delegate_action: VersionedSignedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    let delegate_action = signed_delegate_action.delegate_action();
    if apply_state.block_height > delegate_action.max_block_height() {
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    if delegate_action.sender_id().as_str() != sender_id.as_str() {
        result.result = Err(ActionErrorKind::DelegateActionSenderDoesNotMatchTxReceiver {
            sender_id: delegate_action.sender_id().clone(),
            receiver_id: sender_id.clone(),
        }
        .into());
        return Ok(());
    }

    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
    if result.result.is_err() {
        // Validation failed. Need to return Ok() because this is not a runtime error.
        // "result.result" will be return to the User as the action execution result.
        return Ok(());
    }
