Audit Report

## Title
FIFO Queue Head `break` Permanently Blocks Smaller Subsequent Withdrawal Requests Until Head Is Satisfied - (`contracts/LRTWithdrawalManager.sol`)

## Summary
`_unlockWithdrawalRequests` iterates the per-asset withdrawal queue in strict FIFO order and unconditionally `break`s when the head request's `payoutAmount` exceeds the vault's current liquid balance. Because `nextLockedNonce` is never advanced past the blocking head, every request behind it ŌĆö regardless of individual size ŌĆö remains in the locked state and reverts with `WithdrawalLocked` when users attempt `completeWithdrawal`. The freeze persists until the unstaking vault accumulates enough assets to cover the head request.

## Finding Description
Two separate balance sources govern the two lifecycle checkpoints:

- **`initiateWithdrawal` (line 170):** guards against `getAvailableAssetAmount(asset)`, which calls `lrtDepositPool.getTotalAssetDeposits(asset)` ŌĆö the aggregate of all protocol assets including those still deployed in EigenLayer strategies. A large withdrawal can pass this check even when the unstaking vault holds far less.
- **`_createUnlockParams` (line 849):** sets `totalAvailableAssets = unstakingVault.balanceOf(asset)` ŌĆö only the liquid balance in the vault, typically a small fraction of total deposits.

Inside `_unlockWithdrawalRequests` (lines 790ŌĆō814), the loop processes requests in strict nonce order. At line 800:

```solidity
if (availableAssetAmount < payoutAmount) break; // Exit if not enough assets to cover this request
```

This is an unconditional `break`, not a `continue`. When the head request cannot be covered, `nextLockedNonce_` is not incremented and the loop exits. `nextLockedNonce[asset]` is written back at line 815 with the unchanged value. All subsequent requests ŌĆö even those with `payoutAmount` well within the vault balance ŌĆö remain locked.

At `completeWithdrawal` ŌåÆ `_processWithdrawalCompletion` (line 707):

```solidity
if (usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]) revert WithdrawalLocked();
```

Any user whose nonce is Ōēź `nextLockedNonce[asset]` is blocked. Because the head request prevents `nextLockedNonce` from advancing, all victims behind it are frozen.

**Exploit path (no privileges required):**
1. Attacker calls `initiateWithdrawal(asset, largeRsETH, ...)` ŌĆö passes because total deposits are large. Request assigned nonce `N`.
2. Victims call `initiateWithdrawal` with small amounts; nonces `N+1`, `N+2`, ŌĆ”
3. Operator calls `unlockQueue` when `unstakingVault.balanceOf(asset)` is just below the head's `payoutAmount` (e.g., vault has 99 ETH, head needs 100 ETH, victims need 1 ETH each).
4. Loop hits nonce `N`: `availableAssetAmount < payoutAmount` ŌåÆ `break`.
5. `nextLockedNonce[asset]` stays at `N`. Victims' nonces `N+1`, `N+2`, ŌĆ” all satisfy `nonce >= nextLockedNonce` ŌåÆ `WithdrawalLocked`.

## Impact Explanation
Victims' funds are frozen for the duration of the EigenLayer unbonding period plus operator cadence ŌĆö potentially days to weeks ŌĆö until the vault accumulates enough to cover the head request. This is a concrete, time-bounded loss of fund access. Impact: **Medium ŌĆö Temporary freezing of funds**, which is explicitly within the allowed scope.

## Likelihood Explanation
No special role is required; any rsETH holder can call `initiateWithdrawal`. The attacker only needs to queue before victims. The condition is naturally reachable: the unstaking vault is replenished incrementally as EigenLayer withdrawals complete, so windows where vault balance is below a large head request are routine, not exceptional. The attacker's own funds are also locked, making this a costly but feasible griefing vector for a well-capitalised adversary.

## Recommendation
Replace the `break` at line 800 with a `continue` so the loop skips requests it cannot currently satisfy and processes smaller ones behind them. If strict FIFO ordering is a protocol requirement, introduce a secondary check that skips (but does not drop) a request when `payoutAmount > availableAssetAmount`, advancing `nextLockedNonce` only for requests that are actually unlocked. Alternatively, cap `expectedAssetAmount` at `initiateWithdrawal` time against `unstakingVault.balanceOf(asset)` rather than `getTotalAssetDeposits`, ensuring no request can be queued for more than the vault can realistically service in the near term.
