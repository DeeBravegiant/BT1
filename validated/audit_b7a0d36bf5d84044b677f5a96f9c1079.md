### Title
`respond_verify_foreign_tx` Accepts Caller-Supplied `payload_hash` Without Binding It to the Pending Request — (`File: crates/contract/src/lib.rs`)

---

### Summary

In `respond_verify_foreign_tx`, the contract verifies that the MPC signature is valid for `response.payload_hash`, but never checks that `response.payload_hash` was actually computed from the `request.request` data stored in the pending queue. A single Byzantine attested participant (below signing threshold) can reuse a legitimately-produced MPC signature — obtained for a request with weaker finality/confirmation parameters — to resolve a pending request that demanded stronger finality, causing the contract to deliver a forged foreign-chain verification result to the caller.

---

### Finding Description

In the regular `respond` path, the payload hash used for signature verification is taken directly from the on-chain `request.payload` field:

```rust
// crates/contract/src/lib.rs:600
let payload_hash = request.payload.as_ecdsa().expect("Payload is not ECDSA");
near_mpc_signature_verifier::verify_ecdsa_signature(signature_response, payload_hash, &expected_public_key)
```

The hash is immutable and bound to the on-chain request. No substitution is possible. [1](#0-0) 

In `respond_verify_foreign_tx`, the hash used for verification is instead taken from the **response itself** — a value supplied by the calling MPC node:

```rust
// crates/contract/src/lib.rs:726-733
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [2](#0-1) 

The contract performs no check that `response.payload_hash` equals `SHA-256(borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 { request: request.request, values: <anything> })))`. The `request` argument (which carries the user's original parameters including `confirmations`, `finality`, `tx_id`, etc.) is used only as a lookup key into `pending_verify_foreign_tx_requests` — it is never used to recompute or bound the expected hash. [3](#0-2) 

The signed payload structure is:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,  // includes confirmations, finality, tx_id
    pub values: Vec<ExtractedValue>,
}
``` [4](#0-3) 

The hash is `SHA-256(borsh(ForeignTxSignPayload))`. [5](#0-4) 

On the node side, `build_signature_request` computes the hash from the actual observed payload and uses a zero tweak (root key):

```rust
tweak: Tweak::new([0u8; 32]),
``` [6](#0-5) 

This means any valid MPC signature over **any** `ForeignTxSignPayload` hash — regardless of which request it was produced for — will pass the contract's verification check, as long as the attacker can match a pending request key.

---

### Impact Explanation

A Byzantine attested participant (a single node, strictly below the signing threshold) can execute the following cross-request replay:

1. Submit `verify_foreign_transaction` with `BitcoinRpcRequest { tx_id: X, confirmations: 1, ... }`. The MPC network legitimately signs `hash_1 = SHA-256(borsh(ForeignTxSignPayload::V1({ request: Bitcoin(X, confirmations=1), values: [block_Y] })))`.
2. Save `(hash_1, sig_1)`.
3. A victim submits `verify_foreign_transaction` with `BitcoinRpcRequest { tx_id: X, confirmations: 6, ... }` — a request requiring finality.
4. The Byzantine node calls `respond_verify_foreign_tx(request={X, confirmations=6}, response={payload_hash=hash_1, signature=sig_1})`.
5. The contract checks `verify_ecdsa_signature(sig_1, hash_1, root_pk)` → **passes**, because the MPC network did sign `hash_1`.
6. The contract resolves the victim's pending request (keyed on `{X, confirmations=6}`) and delivers `response={payload_hash=hash_1, sig_1}` to the victim's callback.

The victim's bridge contract receives a `VerifyForeignTransactionResponse` that the MPC contract has accepted as valid. The `payload_hash` inside it corresponds to a 1-confirmation payload, not a 6-confirmation one. Any bridge contract that trusts the MPC contract's acceptance — the intended design — will process a transaction that is not yet final, enabling a **double-spend**.

Impact class: **High** — forged foreign-chain verification / light-client-style verification bypass causing invalid bridge execution or double-spend conditions.

---

### Likelihood Explanation

- The attacker must be a single attested MPC participant (below signing threshold). This is a realistic Byzantine adversary explicitly in scope.
- No threshold collusion is required. The attacker only needs a legitimately-produced signature from a prior request with weaker parameters — something any participant can obtain by submitting their own request.
- The attack is fully on-chain and requires no special access beyond being an attested node.
- The `confirmations` and `finality` parameters are user-controlled inputs that directly affect what the MPC network signs, making it straightforward to craft a weak-parameter request to harvest a reusable signature.

---

### Recommendation

In `respond_verify_foreign_tx`, do not use `response.payload_hash` as the verification target. Instead, include the extracted values in the response and recompute the expected hash on-chain:

```rust
// Proposed fix sketch
let expected_payload = ForeignTxSignPayload::V1(ForeignTxSignPayloadV1 {
    request: request.request.clone(),
    values: response.extracted_values.clone(), // add to response DTO
});
let expected_hash = expected_payload.compute_msg_hash()?;
assert_eq!(expected_hash, response.payload_hash, "payload_hash mismatch");
// then verify signature against expected_hash
```

If including extracted values in the response is too large for NEAR's promise data limits, an alternative is to include a commitment to `request.request` inside the signed payload that the contract can independently verify (e.g., `SHA-256(borsh(request.request))` as a prefix field), so the contract can at minimum confirm the hash was computed over the correct request parameters.

### Proof of Concept

```
// Step 1: Byzantine node (attested participant) submits a weak request
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: TARGET_TX_ID,
        confirmations: BlockConfirmations(1),  // weak: 1 confirmation
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
    domain_id: DOMAIN_ID,
    payload_version: ForeignTxPayloadVersion::V1,
});

// Step 2: MPC network signs hash_1 for the 1-confirmation request
// Byzantine node saves (hash_1, sig_1)

// Step 3: Victim submits a strong request (6 confirmations)
contract.verify_foreign_transaction(VerifyForeignTransactionRequestArgs {
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: TARGET_TX_ID,
        confirmations: BlockConfirmations(6),  // strong: 6 confirmations
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
    domain_id: DOMAIN_ID,
    payload_version: ForeignTxPayloadVersion::V1,
});

// Step 4: Byzantine node submits the weak signature as a response to the strong request
contract.respond_verify_foreign_tx(
    VerifyForeignTransactionRequest {
        request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
            tx_id: TARGET_TX_ID,
            confirmations: BlockConfirmations(6),  // matches victim's pending key
            extractors: vec![BitcoinExtractor::BlockHash],
        }),
        domain_id: DOMAIN_ID,
        payload_version: ForeignTxPayloadVersion::V1,
    },
    VerifyForeignTransactionResponse {
        payload_hash: hash_1,  // hash from 1-confirmation request — NOT from 6-confirmation
        signature: sig_1,      // valid MPC signature over hash_1
    },
);
// Contract accepts: sig_1 is valid for hash_1 ✓
// Victim's callback fires with payload_hash=hash_1 (1-confirmation semantics)
// Bridge contract processes the transaction as if it had 6 confirmations → double-spend enabled
```
