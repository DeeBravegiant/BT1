### Title
Stale DRep Reverse Delegation During Bootstrap Phase Allows Silent Erasure of Active Vote Delegation Upon DRep Unregistration - (File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Deleg.hs)

---

### Summary

During the Conway bootstrap phase (protocol version 9), when a stake credential re-delegates its vote from DRep A to DRep B, the old reverse delegation entry in DRep A's `drepDelegs` set is intentionally not removed. When DRep A subsequently unregisters, the `ConwayUnRegDRep` handler in `GovCert.hs` uses the stale `drepDelegs` to clear the credential's `dRepDelegationAccountStateL` to `Nothing`, silently erasing the credential's active delegation to DRep B. The credential's stake is then excluded from DRep B's governance vote distribution, corrupting ratification calculations.

---

### Finding Description

**Root cause â€” `processDelegationInternal` (Deleg.hs, lines 347â€“377):**

When `preserveIncorrectDelegation` is `True` (bootstrap phase, PV9, or new accounts), re-delegating from DRep A to DRep B only *adds* the credential to DRep B's `drepDelegs` but does **not** remove it from DRep A's `drepDelegs`. The account state is correctly updated to point to DRep B. [1](#0-0) 

The `preserveIncorrectDelegation` flag is set to `True` when `pvMajor pv < natVersion @10`, i.e., during the bootstrap phase: [2](#0-1) 

The bootstrap phase is defined as protocol version 9 exactly: [3](#0-2) 

**Propagation â€” `ConwayUnRegDRep` in `GovCert.hs` (lines 244â€“254):**

When DRep A unregisters, the handler iterates over `drepDelegs dRepState` (DRep A's stored delegator set) and calls `clearDRepDelegations`, which sets `dRepDelegationAccountStateL .~ Nothing` for every credential in that set: [4](#0-3) 

Because credential C is still in DRep A's stale `drepDelegs`, C's delegation is cleared to `Nothing` even though C is actively delegated to DRep B.

**Effect on governance distribution â€” `computeDRepDistr` (DRepPulser.hs, lines 228â€“241):**

`computeDRepDistr` reads `dRepDelegationAccountStateL` from each account. After the erasure, C's field is `Nothing`, so `addToDRepDistr` returns `distr` unchanged â€” C's stake is not added to DRep B's distribution: [5](#0-4) 

**Effect on ratification â€” `dRepAcceptedRatio` (Ratify.hs, lines 258â€“281):**

The ratification ratio is computed from `reDRepDistr`, which is the output of `computeDRepDistr`. DRep B's entry in `reDRepDistr` is now missing C's stake, so the yes/total ratio is understated: [6](#0-5) 

**Confirmed by existing test (DelegSpec.hs, lines 313â€“323):**

The test explicitly asserts that in the bootstrap phase, after DRep A unregisters, the credential's delegation is `Nothing` even though it should still point to DRep B: [7](#0-6)

### Impact Explanation

This is a **Medium** impact finding: an attacker-controlled certificate (`ConwayUnRegDRep`) modifies governance vote distribution outside design parameters.

- A malicious actor registers as DRep A (paying the DRep deposit â€” no privileged access required).
- Delegators who previously delegated to DRep A and have since re-delegated to DRep B during the bootstrap phase retain a stale entry in DRep A's `drepDelegs`.
- When DRep A unregisters, all such credentials have their active delegation to DRep B silently erased.
- DRep B's effective voting power is reduced in `reDRepDistr`, potentially causing governance proposals that DRep B votes on to fail to reach the required ratification threshold â€” or allowing proposals that should fail to pass.

This maps to: *"Medium. Attacker-controlled transactions, blocks, certificates, votes, proposals, scripts, witnesses, or serialized inputs exceed intended validation limits or modify fees, deposits, refunds, rewards, treasury donations, or withdrawals outside design parameters."*

---

### Likelihood Explanation

- The bootstrap phase (PV9) is a bounded window; the attack is only possible while `hardforkConwayBootstrapPhase` returns `True`.
- Any registered DRep can trigger this by unregistering â€” no privileged role, leaked key, or supermajority is required.
- The attack requires only standard on-chain certificate operations (`ConwayDelegCert` for re-delegation, `ConwayUnRegDRep` for unregistration), both of which are fully attacker-controlled.
- The victim need not take any unusual action; normal re-delegation during the bootstrap phase is sufficient to create the vulnerable state.

---

### Recommendation

The fix is already implemented for post-bootstrap (PV10+): `processDelegationInternal` with `preserveIncorrectDelegation = False` correctly calls `unDelegReDelegDRep`, which removes the old reverse delegation before adding the new one. [8](#0-7) 

For the bootstrap phase, either:
1. Remove the `preserveIncorrectDelegation` guard so that `unDelegReDelegDRep` is always called on re-delegation (accepting the minor state divergence from the original PV9 behavior), or
2. Make `clearDRepDelegations` in `GovCert.hs` check the credential's *current* `dRepDelegationAccountStateL` before clearing it, skipping credentials whose current delegation no longer points to the unregistering DRep.

---

### Proof of Concept

**Attacker-controlled entry path (all standard on-chain operations):**

1. **Bootstrap phase active** (`pvMajor == 9`).
2. Credential C submits `ConwayDelegCert C (DelegVote (DRepCredential drepA))` â†’ C is delegated to DRep A; DRep A's `drepDelegs` = `{C}`.
3. Credential C submits `ConwayDelegCert C (DelegVote (DRepCredential drepB))` â†’ C's account state updated to DRep B; DRep B's `drepDelegs` = `{C}`; DRep A's `drepDelegs` = `{C}` (stale, not removed â€” bug #4772 preserved).
4. DRep A submits `ConwayUnRegDRep drepA refund` â†’ `clearDRepDelegations {C} accountsMap` sets C's `dRepDelegationAccountStateL = Nothing`.
5. At epoch boundary, `computeDRepDistr` finds C's delegation is `Nothing` â†’ C's stake is not added to DRep B's distribution.
6. `dRepAcceptedRatio` computes DRep B's yes-ratio without C's stake â†’ governance proposals voted on by DRep B may fail to ratify. [9](#0-8) [10](#0-9)

dRepAcceptedRatio RatifyEnv {reDRepDistr, reDRepState, reCurrentEpoch} gasDRepVotes govAction =
  toInteger yesStake %? toInteger totalExcludingAbstainStake
  where
    accumStake (!yes, !tot) drep (CompactCoin stake) =
      case drep of
        DRepCredential cred ->
          case Map.lookup cred reDRepState of
            Nothing -> (yes, tot) -- drep is not registered, so we don't consider it
            Just drepState
              | reCurrentEpoch > drepExpiry drepState -> (yes, tot) -- drep is expired, so we don't consider it
              | otherwise ->
                  case Map.lookup cred gasDRepVotes of
                    -- drep hasn't voted for this action, so we don't count
                    -- the vote but we consider it in the denominator:
                    Nothing -> (yes, tot + stake)
                    Just VoteYes -> (yes + stake, tot + stake)
                    Just Abstain -> (yes, tot)
                    Just VoteNo -> (yes, tot + stake)
        DRepAlwaysNoConfidence ->
          case govAction of
            NoConfidence _ -> (yes + stake, tot + stake)
            _ -> (yes, tot + stake)
        DRepAlwaysAbstain -> (yes, tot)
    (yesStake, totalExcludingAbstainStake) = Map.foldlWithKey' accumStake (0, 0) reDRepDistr

unDelegReDelegDRep ::
  ConwayEraAccounts era =>
  Credential Staking ->
  -- | Account that is losing its current delegation and/or acquiring a new one
  AccountState era ->
  -- | Potential new delegation. In case when stake credential unregisters this must be `Nothing`.
  Maybe DRep ->
  VState era ->
  VState era
unDelegReDelegDRep stakeCred accountState mNewDRep =
  fromMaybe (vsDRepsL %~ addNewDelegation) $ do
    dRep@(DRepCredential dRepCred) <- accountState ^. dRepDelegationAccountStateL
    pure $
      -- There is no need to update set of delegations if delegation is unchanged
      if Just dRep == mNewDRep
        then id
        else
          vsDRepsL %~ addNewDelegation . Map.adjust (drepDelegsL %~ Set.delete stakeCred) dRepCred
  where
    addNewDelegation =
      case mNewDRep of
        Just (DRepCredential dRepCred) ->
          Map.adjust (drepDelegsL %~ Set.insert stakeCred) dRepCred
        _ -> id
