Audit Report

## Title
Unbounded `assetsCommitted` Accumulation Enables Temporary DOS of Withdrawal Initiation - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`initiateWithdrawal` in `LRTWithdrawalManager` increments the shared `assetsCommitted[asset]` counter with no per-user ceiling. An attacker holding sufficient rsETH can saturate this counter up to the total protocol TVL for a given asset, causing `getAvailableAssetAmount` to return zero and reverting every subsequent `initiateWithdrawal` call from legitimate users. Because the attacker recovers their capital after each operator `unlockQueue` cycle, the DOS can be sustained indefinitely at the cost of gas and opportunity cost only.

## Finding Description
`initiateWithdrawal` gates new requests on `getAvailableAssetAmount`:

```solidity
// L170-173
if (expectedAssetAmount > getAvailableAssetAmount(asset)) revert ExceedAmountToWithdraw();
assetsCommitted[asset] += expectedAssetAmount;
```

`getAvailableAssetAmount` computes:

```solidity
// L599-603
uint256 totalAssets = lrtDepositPool.getTotalAssetDeposits(asset);
availableAssetAmount = totalAssets > assetsCommitted[asset]
    ? totalAssets - assetsCommitted[asset] : 0;
```

`assetsCommitted[asset]` is a single shared counter with no per-user cap. `_addUserWithdrawalRequest` (L744–757) imposes no limit on the number of open requests per user. An attacker calls `initiateWithdrawal` in a loop — each call committing `minRsEthAmountToWithdraw[asset]` worth of the underlying asset — until `assetsCommitted[asset] >= totalAssets`. At that point `getAvailableAssetAmount` returns 0 and every honest user's call reverts with `ExceedAmountToWithdraw`.

`assetsCommitted` is decremented only inside `_unlockWithdrawalRequests` (L802), which is called by the operator's `unlockQueue`. After the operator unlocks, the attacker calls `completeWithdrawal` → `_processWithdrawalCompletion` (L699–738) to recover the underlying asset, then re-deposits to obtain rsETH and immediately re-saturates the counter. No existing guard (`nonReentrant`, `whenNotPaused`, `onlySupportedAsset`, `onlySupportedStrategy`) prevents this cycle.

## Impact Explanation
While the attack is sustained, `initiateWithdrawal` — the sole entry point for the standard withdrawal lifecycle — is completely blocked for all users. rsETH holders cannot begin the process of redeeming their tokens for the underlying LST or ETH. This constitutes **temporary freezing of funds** (Medium), matching the allowed impact scope. The operator can call `unlockQueue` to partially relieve pressure, but the attacker can immediately re-saturate the counter, making the DOS persistent across cycles until the attacker voluntarily stops.

## Likelihood Explanation
The attacker must hold rsETH whose redemption value equals the total protocol TVL for the targeted asset. For a mature, high-TVL deployment this is a very high capital bar. However, the attacker recovers their capital after each cycle (net cost: gas + opportunity cost of the ~8-day withdrawal delay). The attack is most feasible when protocol TVL is low (early deployment) or when targeting a lower-TVL supported LST. No privileged access is required; the entry path is fully open to any rsETH holder.

## Recommendation
Introduce a per-user cap on open withdrawal requests in `_addUserWithdrawalRequest`, analogous to `maxNumberOfWithdrawalsPerUser` already present in `KernelDepositPool` (L107–108, L323). For example:

```solidity
if (userAssociatedNonces[asset][msg.sender].length() >= maxWithdrawalsPerUser) {
    revert WithdrawalLimitReached();
}
```

Alternatively, track each user's committed amount in a separate mapping and reject requests that would push any single user's committed share above a configurable fraction of `totalAssets`. Either approach bounds the share of `assetsCommitted` any single address can consume.

## Proof of Concept
1. Attacker acquires rsETH (e.g., via `LRTDepositPool.depositETH`) sufficient to cover `getAvailableAssetAmount(ETH_TOKEN)`.
2. Attacker calls `LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, minRsEthAmountToWithdraw[ETH_TOKEN], "")` in a loop until `assetsCommitted[ETH_TOKEN] >= totalAssets`.
3. Any honest user calling `initiateWithdrawal` now receives `ExceedAmountToWithdraw`.
4. Operator calls `unlockQueue` → `assetsCommitted` decreases for unlocked requests (L802).
5. Attacker calls `completeWithdrawal` to recover ETH (L183–185, L699–738), re-deposits to obtain rsETH, and repeats from step 2.
6. The withdrawal initiation path remains blocked for honest users across cycles.

**Foundry test sketch:**
```solidity
function testDOSInitiateWithdrawal() public {
    // Fund attacker with rsETH equal to total ETH TVL
    uint256 tvl = depositPool.getTotalAssetDeposits(ETH_TOKEN);
    deal(rsETH, attacker, tvl * 1e18 / rsETHPrice);

    vm.startPrank(attacker);
    uint256 chunk = withdrawalManager.minRsEthAmountToWithdraw(ETH_TOKEN);
    while (withdrawalManager.getAvailableAssetAmount(ETH_TOKEN) >= chunk) {
        withdrawalManager.initiateWithdrawal(ETH_TOKEN, chunk, "");
    }
    vm.stopPrank();

    // Honest user attempt
    vm.prank(honestUser);
    vm.expectRevert(ILRTWithdrawalManager.ExceedAmountToWithdraw.selector);
    withdrawalManager.initiateWithdrawal(ETH_TOKEN, chunk, "");
}
```
