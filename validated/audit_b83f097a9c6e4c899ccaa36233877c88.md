### Title
Propose-Update Storage Deposits Permanently Trapped on `do_update` Execution - (File: `crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

When a participant calls `propose_update`, they must attach a NEAR deposit covering the storage cost of the update payload. When any update reaches threshold and `do_update` is executed, it clears **all** pending update entries — including proposals from other participants — without refunding their deposits. Those NEAR tokens are permanently trapped in the contract balance with no recovery path.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires an attached deposit proportional to the update's storage footprint: [1](#0-0) 

Only the excess above the required amount is refunded to the proposer: [2](#0-1) 

The exact required deposit is consumed and retained by the contract. The `UpdateEntry` struct that is stored records `bytes_used` but **not** the proposer's account ID: [3](#0-2) 

When `vote_update` reaches threshold, it calls `do_update`, which removes the executed entry and then **clears all remaining entries and all votes**: [4](#0-3) 

No refund is issued to the proposers of the cleared entries. Because `UpdateEntry` does not record the proposer's account, there is no information available to issue a refund even if the code attempted to do so. The deposits are permanently locked in the contract balance.

---

### Impact Explanation

Every participant who proposed a competing update loses their storage deposit when any other update is executed. For a contract-code update, the deposit is computed as `env::storage_byte_cost() * bytes_used`, where `bytes_used` includes the full contract bytecode plus a fixed overhead for 128 participant votes: [5](#0-4) 

At NEAR's storage cost (~10 NEAR/MB), a 500 KB contract binary yields a deposit of ~5 NEAR per proposal. With multiple competing proposals, the total trapped amount scales linearly. There is no administrative function, no withdrawal path, and no migration hook that recovers these funds. This permanently breaks the balance accounting invariant of the contract.

**Impact class:** Medium — balance/accounting invariant violation that permanently freezes participant funds without any recovery path.

---

### Likelihood Explanation

The governance flow naturally produces competing proposals: participants may disagree on which contract version to deploy and submit independent proposals. The moment any one proposal reaches threshold, all others are silently cleared and their deposits are lost. This is a normal, expected governance scenario, not an edge case. Any participant who proposes an update that is not ultimately executed loses their deposit with certainty.

---

### Recommendation

**Short term:** Store the proposer's `AccountId` inside `UpdateEntry` alongside `bytes_used`. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(refund_amount)` for each non-executed proposal.

**Long term:** Add an invariant test asserting that the contract's NEAR balance does not increase after a full governance cycle (propose → vote → execute), ensuring no deposits are silently retained.

---

### Proof of Concept

1. Participant Alice calls `propose_update` with a 500 KB contract binary, attaching ~5 NEAR deposit. The deposit is consumed; only excess is refunded.
2. Participant Bob calls `propose_update` with a different 500 KB binary, also attaching ~5 NEAR.
3. A threshold of participants votes for Bob's update via `vote_update`.
4. `vote_update` calls `do_update`, which executes Bob's update, then calls `self.entries.clear()` and `self.vote_by_participant.clear()`.
5. Alice's entry is deleted. No refund is issued because `UpdateEntry` contains no proposer field and no refund logic exists in `do_update`.
6. Alice's ~5 NEAR is permanently locked in the MPC contract balance with no recovery mechanism. [6](#0-5) [7](#0-6)

    pub fn propose_update(
        &mut self,
        #[serializer(borsh)] args: ProposeUpdateArgs,
    ) -> Result<UpdateId, Error> {
        // Only voters can propose updates:
        let proposer = self.voter_or_panic();
        let update: Update = args.try_into()?;

        let attached = env::attached_deposit();
        let required = ProposedUpdates::required_deposit(&update);
        if attached < required {
            return Err(InvalidParameters::InsufficientDeposit {
                attached: attached.as_yoctonear(),
                required: required.as_yoctonear(),
            }
            .into());
        }

        let id = self.proposed_updates.propose(update);

        log!(
            "propose_update: signer={}, id={:?}",
            env::signer_account_id(),
            id,
        );

        // Refund the difference if the proposer attached more than required.
        if let Some(diff) = attached.checked_sub(required)
            && diff > NearToken::from_yoctonear(0)
        {
            Promise::new(proposer).transfer(diff).detach();
        }

        Ok(id)
    }

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}
