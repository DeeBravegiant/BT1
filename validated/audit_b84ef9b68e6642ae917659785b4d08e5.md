### Title
`estimate_minimal_gas_vector` Omits `allocation_cost` from Pre-Validation Gas Lower Bound, Allowing Admission of Transactions That Will Revert — (File: `crates/blockifier/src/fee/gas_usage.rs`)

---

### Summary

`estimate_minimal_gas_vector` computes only DA gas cost plus VM resource cost, but completely omits the `allocation_cost × n_allocated_keys` component that is charged at execution time via `StateResources::to_gas_vector`. Because `check_fee_bounds` uses this underestimated lower bound to gate admission, a transaction whose resource bounds cover DA costs but not allocation costs passes pre-validation at the gateway/mempool layer, is admitted, and then reverts during block execution with "Insufficient max resource."

---

### Finding Description

`estimate_minimal_gas_vector` in `crates/blockifier/src/fee/gas_usage.rs` is the sole source of the minimum-gas lower bound used by `check_fee_bounds` during pre-validation:

```rust
// gas_usage.rs:157-214
pub fn estimate_minimal_gas_vector(...) -> GasVector {
    let state_changes_by_account_tx = match &tx.tx {
        Transaction::DeployAccount(_) => StateChangesCount {
            n_storage_updates: 1,
            n_class_hash_updates: 1,
            n_compiled_class_hash_updates: 0,
            n_modified_contracts: 1,
        },
        ...
    };
    let da_gas_cost = get_da_gas_cost(&state_changes_by_account_tx, block_info.use_kzg_da);
    let vm_resources_cost = get_extended_vm_resources_cost(...);
    da_gas_cost.checked_add(vm_resources_cost)   // ← allocation_cost never added
}
```

The actual per-execution gas computation in `StateResources::to_gas_vector` (`crates/blockifier/src/fee/resources.rs:228-249`) charges **both** DA cost and allocation cost:

```rust
pub fn to_gas_vector(&self, use_kzg_da: bool, allocation_cost: &AllocationCost) -> GasVector {
    let n_allocated_keys: u64 = self.state_changes_for_fee.n_allocated_keys...;
    let allocation_gas_vector = allocation_cost.get_cost(use_kzg_da);
    let total_allocation_cost = allocation_gas_vector.checked_scalar_mul(n_allocated_keys)...;
    let da_gas_cost = self.da_gas_vector(use_kzg_da);
    total_allocation_cost.checked_add(da_gas_cost)   // ← both components
}
```

`allocation_cost` is non-zero from Starknet version 0.13.5 onward. In the current production constants (`blockifier_versioned_constants_0_14_4.json`):

- **Blob (KZG) mode**: `l1_data_gas: 32` + `l2_gas: 402_000` per allocated key
- **Calldata mode**: `l1_gas: 551` + `l2_gas: 402_000` per allocated key

`n_allocated_keys` is the count of storage slots written from zero to non-zero (`AllocatedKeys`, tracked in `StateChanges::allocated_keys`). For a `DeployAccount` transaction, the new account's class-hash mapping and balance slot are both fresh allocations, so `n_allocated_keys ≥ 2`.

`check_fee_bounds` (`account_transaction.rs:374-476`) compares `minimal_gas_amount_vector` (from `estimate_minimal_gas_vector`) against the user-supplied `max_amount` per resource. Because the allocation component is absent from the estimate, a user can submit a `DeployAccount` (or any transaction that writes new storage slots) with `max_amount` set just above the DA+VM lower bound but below the true minimum that includes allocation cost. The transaction passes `check_fee_bounds`, passes `verify_can_pay_committed_bounds` (balance check uses the same committed bounds), is admitted to the mempool, and then reverts during execution.

The `StateChanges::count_for_fee_charge` function (`cached_state.rs:899-932`) populates `n_allocated_keys` from `self.allocated_keys.len()`, which is only known after execution — but for `DeployAccount` the minimum number of new allocations is statically known (at least the class-hash slot and the sender balance slot), so a conservative lower bound could be included in `estimate_minimal_gas_vector`.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Any unprivileged user can craft a `DeployAccount` (or `Invoke` that writes to new storage slots) with resource bounds that are deliberately set between the underestimated lower bound and the true minimum. The gateway's `check_fee_bounds` guard passes, the transaction enters the mempool, is sequenced, and reverts. The sequencer collects the fee up to the user's bounds, but block space is consumed by a predictably-reverting transaction. An attacker can repeat this at scale to fill blocks with reverted `DeployAccount` transactions, degrading throughput for legitimate users.

---

### Likelihood Explanation

The gap is reachable by any user submitting a V3 `DeployAccount` or `Invoke` transaction with `AllResources` bounds. The `allocation_cost` values in production constants (551 L1 gas or 32 L1 data gas, plus 402,000 L2 gas per key) are large enough to be the deciding factor when a user sets tight bounds. No privileged access or special network position is required; the ordinary RPC `add_transaction` endpoint is the trigger path.

---

### Recommendation

Add the known-minimum allocation cost to `estimate_minimal_gas_vector`. For `DeployAccount`, at least 2 allocated keys are guaranteed (class-hash mapping + sender balance slot). For `Invoke` and `Declare`, at least 1 key (sender balance slot, if not previously allocated) should be assumed. Concretely:

```rust
let n_min_allocated_keys: u64 = match &tx.tx {
    Transaction::DeployAccount(_) => 2,
    Transaction::Invoke(_) | Transaction::Declare(_) => 1,
};
let allocation_cost = versioned_constants.allocation_cost.get_cost(block_info.use_kzg_da);
let allocation_gas = allocation_cost.checked_scalar_mul(n_min_allocated_keys)
    .expect("allocation cost overflow");
da_gas_cost
    .checked_add(vm_resources_cost).expect("overflow")
    .checked_add(allocation_gas).expect("overflow")
```

### Proof of Concept

1. Read current production `allocation_cost` from `blockifier_versioned_constants_0_14_4.json`: `gas_cost.l1_gas = 551`, `gas_cost.l2_gas = 402_000`.
2. Call `estimate_minimal_gas_vector` for a `DeployAccount` V3 transaction in calldata mode. The result contains **zero** L1 gas from allocation and **zero** L2 gas from allocation.
3. Submit a `DeployAccount` V3 (`AllResources`) with:
   - `l1_gas.max_amount` = `estimate.l1_gas` (covers DA+VM, misses `551 × 2 = 1_102` allocation L1 gas)
   - `l2_gas.max_amount` = `estimate.l2_gas` (misses `402_000 × 2 = 804_000` allocation L2 gas)
4. `check_fee_bounds` compares `minimal_gas_amount_vector` (from `estimate_minimal_gas_vector`) against the user bounds — passes, because the estimate omits allocation.
5. `verify_can_pay_committed_bounds` passes (balance covers the underestimated committed fee).
6. Transaction is admitted to the mempool and sequenced.
7. During execution, `StateResources::to_gas_vector` computes the true gas including `allocation_cost × n_allocated_keys`; the result exceeds `max_amount`; the transaction reverts with "Insufficient max L1Gas / L2Gas."

The test at `crates/blockifier/src/transaction/transactions_test.rs:1646` explicitly sets `block_context.versioned_constants.allocation_cost = AllocationCost::ZERO` to avoid this exact interference, confirming the production gap. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

            n_class_hash_updates: 1,
            n_compiled_class_hash_updates: 0,
            n_modified_contracts: 1,
        },
    };

    /// Returns the gas cost of the transaction's state changes.
    // The allocation_cost L2 gas component prices the extra Patricia Merkle tree proving
    // overhead for writing to a new (zero → nonzero) state cell. This is computed from
    // the final state diff, not per-syscall. Implications:
    // - A 0→x→0 write avoids the charge (correct: no net state change, no proving cost).
    // - System-side allocations (fee transfers, etc.) are also charged.
    // - This cost flows into receipt.gas.l2_gas (user fees + EIP-1559) and the bouncer's
    //   receipt_l2_gas dimension (block closure), but NOT into bouncer sierra_gas.
    pub fn to_gas_vector(&self, use_kzg_da: bool, allocation_cost: &AllocationCost) -> GasVector {
        let n_allocated_keys: u64 = self
            .state_changes_for_fee
            .n_allocated_keys
            .try_into()
            .expect("n_allocated_keys overflowed");
        let allocation_gas_vector = allocation_cost.get_cost(use_kzg_da);
        let total_allocation_cost =
            allocation_gas_vector.checked_scalar_mul(n_allocated_keys).unwrap_or_else(|| {
                panic!(
                    "State resources to gas vector overflowed: tried to multiply \
                     {allocation_gas_vector:?} by {n_allocated_keys:?}",
                )
            });
        let da_gas_cost = self.da_gas_vector(use_kzg_da);
        total_allocation_cost.checked_add(da_gas_cost).unwrap_or_else(|| {
            panic!(
                "State resources to gas vector overflowed: tried to add {total_allocation_cost:?} \
                 to {da_gas_cost:?}",
            )
        })
    }
