### Title
Unenforced CHIP-22 Harvester Fee Threshold Allows Unconditional Farmer Reward Diversion - (`chia/farmer/farmer_api.py`, `chia/farmer/farmer.py`)

### Summary

The CHIP-22 third-party harvester fee convention allows a connected harvester to redirect the farmer's 1/8 block reward (base farmer reward) to an arbitrary address via `farmer_reward_address_override`. The fee quality threshold check in `notify_farmer_reward_taken_by_harvester_as_fee` is **purely advisory (logging only)** and does not block the override from being applied. A malicious third-party harvester can unconditionally divert 100% of the farmer's block reward to itself on every won block, regardless of the agreed fee rate.

### Finding Description

CHIP-22 introduces a convention where a third-party harvester may take a portion of the farmer's reward by setting `farmer_reward_address_override` in its `NewProofOfSpace` and `RespondSignatures` messages. The intended enforcement mechanism is a "fee quality" check: the harvester self-reports an `applied_fee_threshold` in `ProofOfSpaceFeeInfo`, and the farmer is supposed to verify that `fee_quality <= applied_fee_threshold` before accepting the override.

In `farmer_api.py`, when a `NewProofOfSpace` arrives with `farmer_reward_address_override` set, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee`:

```python
# farmer_api.py line 128-129
if new_proof_of_space.farmer_reward_address_override is not None:
    self.farmer.notify_farmer_reward_taken_by_harvester_as_fee(sp, new_proof_of_space)
```

Inside `notify_farmer_reward_taken_by_harvester_as_fee` (`farmer.py` lines 911–934), the check is:

```python
if fee_quality <= fee_threshold:
    self.log.info(...)   # passes: only logs
else:
    self.log.warning(...)  # fails: only logs a warning
```

No rejection, no return, no flag set. The function returns `None` in all cases and the farmer continues processing.

Then in `_process_respond_signatures` (`farmer_api.py` lines 916–919), the override is applied unconditionally:

```python
farmer_reward_address = self.farmer.farmer_target
if response.farmer_reward_address_override is not None:
    farmer_reward_address = response.farmer_reward_address_override
    include_source_signature_data = True
```

This `farmer_reward_address` is passed directly into `DeclareProofOfSpace` (line 929), which is used to construct the block with the harvester's address as the farmer reward recipient. The full node validates the cryptographic signature but does not enforce who the farmer reward address must be.

Additionally, if the harvester omits `fee_info` entirely (`fee_info = None`), the farmer still applies the override — the else branch at lines 929–934 only logs a warning.

### Impact Explanation

A malicious third-party harvester can:
1. Set `farmer_reward_address_override` to its own puzzle hash in every `NewProofOfSpace` and `RespondSignatures` message.
2. Set `applied_fee_threshold = 0xFFFFFFFF` (or omit `fee_info`) to suppress or pass the advisory check.
3. Receive 100% of the farmer's base block reward (1/8 of total block reward, e.g., 0.25 XCH per block at current halving) on every block the farmer wins.

The farmer receives only a log warning and no XCH. This is a direct, on-chain diversion of XCH from the farmer to the harvester, confirmed in the block's `FoliageBlockData.farmer_reward_puzzle_hash`.

### Likelihood Explanation

Any third-party harvester service (the explicit target of CHIP-22) can exploit this. The farmer must have connected to the harvester, but this is the normal operating mode for farmers using remote or pooled harvesting infrastructure. The harvester needs no special privileges beyond the existing authenticated connection. The attack is silent — the farmer sees only a log warning, not an error or rejection.

### Recommendation

The farmer must enforce the fee threshold, not merely log it. In `notify_farmer_reward_taken_by_harvester_as_fee`, return a boolean indicating whether the override is legitimate, and in `new_proof_of_space` / `_process_respond_signatures`, reject the override (and drop the proof) if the fee quality does not satisfy the threshold. Specifically:

- If `fee_info` is `None` and `farmer_reward_address_override` is set, reject the proof.
- If `fee_quality > applied_fee_threshold`, reject the proof and do not apply the override.

### Proof of Concept

1. Malicious harvester connects to farmer.
2. For every `NewProofOfSpace` sent to the farmer, harvester sets:
   - `farmer_reward_address_override = <harvester_puzzle_hash>`
   - `fee_info = ProofOfSpaceFeeInfo(applied_fee_threshold=0xFFFFFFFF)`
3. Farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` → logs "Fee threshold passed" (since `fee_quality <= 0xFFFFFFFF` is always true) → continues.
4. Farmer sends `RequestSignatures` to harvester.
5. Harvester responds with `RespondSignatures` with `farmer_reward_address_override = <harvester_puzzle_hash>`.
6. Farmer executes lines 917–918 of `farmer_api.py`: `farmer_reward_address = response.farmer_reward_address_override`.
7. `DeclareProofOfSpace` is sent to the full node with the harvester's address as `farmer_reward_puzzle_hash`.

def calculate_harvester_fee_quality(proof: bytes, challenge: bytes32) -> uint32:
    """
    This calculates the 'fee quality' given a convention between farmers and third party harvesters.
    See CHIP-22: https://github.com/Chia-Network/chips/pull/88
    """
    return uint32(int.from_bytes(std_hash(proof + challenge)[32 - 4 :], byteorder="big", signed=False))
