Audit Report

## Title
Unscoped `address(this).balance` in `pay()` allows leftover ETH from prior caller to be consumed by next WETH-swap caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` as the available native ETH when settling a WETH payment, without restricting it to the ETH contributed by the current caller. If a prior user sent excess ETH and did not call `refundETH()`, that surplus persists in the contract and is silently consumed to pay the next user's WETH swap, causing a direct, quantifiable loss of the first user's principal.

## Finding Description
In `pay()`, the WETH branch reads the contract's total ETH balance:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // total balance, not caller-scoped
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

ETH enters the contract exclusively via `msg.value` on payable entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `addLiquidityExactShares`, `addLiquidityWeighted`). The `receive()` guard blocks direct ETH sends from non-WETH addresses: [2](#0-1) 

However, when a user sends more ETH than the swap consumes and does not call `refundETH()` in the same multicall, the surplus ETH persists. The next caller whose swap triggers `pay()` with `token == WETH` will have that surplus applied toward their payment — paying nothing from their own WETH or ETH balance.

`refundETH()` sends all contract ETH to `msg.sender`, not the original depositor, so a third party can also drain stranded ETH directly: [3](#0-2) 

The same `pay()` function is shared by both `MetricOmmSimpleRouter` (via `_justPayCallback`) and `MetricOmmPoolLiquidityAdder` (via `metricOmmModifyLiquidityCallback`), so both entry points are affected: [4](#0-3) [5](#0-4) 

The exploit path:
1. UserA calls `exactInputSingle{value: 2 ether}` with `amountIn = 0.5 ether` (WETH as tokenIn), without appending `refundETH()`.
2. `pay()` deposits 0.5 ETH as WETH and transfers it to the pool. Router retains 1.5 ETH.
3. UserB calls `exactInputSingle{value: 0}` with `amountIn = 0.5 ether` (WETH as tokenIn).
4. `pay()` reads `nativeBalance = 1.5 ether >= 0.5 ether`, deposits 0.5 ETH from UserA's leftover, and transfers WETH to the pool — UserB pays nothing.

## Impact Explanation
Direct loss of user principal: UserA loses all excess ETH sent beyond the swap amount. The loss is quantifiable (e.g., 1.5 ETH in the example above), exceeds the Critical threshold (>20% of sent value, >$100), and requires no privileged access. The attacker (UserB) pays zero ETH and zero WETH yet receives the full swap output.

## Likelihood Explanation
Any user who calls a WETH-input swap function directly (not via multicall) with excess ETH cannot call `refundETH()` in the same transaction, making the loss unavoidable. Users who use multicall but omit `refundETH()` are equally affected. The test suite confirms the intended pattern requires an explicit `refundETH()` call: [6](#0-5) 
The stranded ETH can be claimed by the next organic WETH swap in the same block, or front-run by a MEV bot watching the mempool.

## Recommendation
Track per-call ETH contribution in transient storage (EIP-1153, already used elsewhere in the protocol via `TransientCallbackPool`) and cap `nativeBalance` in `pay()` to only the ETH attributed to the current call. Alternatively, enforce that `msg.value` exactly equals the ETH portion of the swap at the entry-point level, reverting if excess ETH is supplied without a corresponding `refundETH()` in the same multicall. [7](#0-6) 

## Proof of Concept
```solidity
function test_ethTheft_via_leftoverBalance() public {
    uint128 swapAmount = 1_000;
    uint256 excessEth = 2 ether;

    uint256 userAEthBefore = userA.balance;

    // Step 1: userA swaps with excess ETH, forgets refundETH()
    vm.prank(userA);
    bytes[] memory calls = new bytes[](1); // no refundETH call
    calls[0] = abi.encodeWithSelector(
        router.exactInputSingle.selector,
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: swapAmount, amountOutMinimum: 0,
            recipient: userA, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
        })
    );
    router.multicall{value: excessEth}(calls);
    assertGt(address(router).balance, 0, "leftover ETH in router");

    // Step 2: userB swaps WETH with msg.value = 0; pay() uses userA's leftover
    vm.prank(userB);
    router.exactInputSingle{value: 0}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: swapAmount, amountOutMinimum: 0,
            recipient: userB, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
        })
    );

    // userB paid 0 ETH and 0 WETH, yet received token1
    // userA lost (excessEth - swapAmount) ETH beyond their intended swap cost
    assertLt(userA.balance, userAEthBefore - swapAmount, "userA lost excess ETH");
}
```
