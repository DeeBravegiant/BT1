### Title
Storage Deposits Paid to `propose_update` Are Permanently Locked When Any Update Executes - (File: `crates/contract/src/update.rs`, `crates/contract/src/lib.rs`)

---

### Summary

When a participant calls `propose_update`, they must attach a NEAR deposit proportional to the size of the proposed update (including the full WASM binary for contract upgrades). When any update reaches threshold and `do_update` is called, **all** pending proposals are cleared from storage — but the deposits paid by proposers of the non-executed proposals are never refunded. There is no emergency withdrawal function, no `cancel_proposal` endpoint, and no refund path in `do_update`. The NEAR tokens accumulate permanently in the contract balance with no recovery mechanism.

---

### Finding Description

`propose_update` in `crates/contract/src/lib.rs` requires a deposit calculated by `ProposedUpdates::required_deposit`, which is proportional to the serialized size of the update payload (WASM binary or config) plus overhead for up to 128 participant vote slots: [1](#0-0) 

The exact required deposit is retained by the contract; only the excess is refunded immediately: [2](#0-1) 

When `vote_update` reaches threshold, it calls `do_update`, which removes the winning entry and then **clears all remaining entries and votes** without issuing any refunds: [3](#0-2) 

The `bytes_used` calculation shows that a contract-code proposal for a typical WASM binary (hundreds of KB) requires a non-trivial deposit: [4](#0-3) 

There is no `cancel_proposal`, `withdraw_proposal`, or emergency-withdrawal function anywhere in the contract. The contract has no owner-controlled drain path. The only way proposals are ever removed is via `do_update`, which does not refund.

---

### Impact Explanation

Every participant who proposed an update that was not the one ultimately executed permanently loses their storage deposit. In a realistic governance scenario where multiple participants each propose a different update (e.g., different WASM versions), only one proposal can win; all others are silently cleared with no refund. The NEAR tokens paid by the losing proposers are locked in the contract balance indefinitely with no recovery path. This breaks the production accounting invariant that storage deposits are returned when the storage they cover is freed.

This maps to: **Medium — Balance/accounting invariant broken without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

This is triggered in any governance round where more than one participant proposes an update. The sandbox tests already demonstrate exactly this scenario — multiple participants each call `propose_update` before voting: [5](#0-4) 

The test at line 246 (`test_propose_update_contract_many`) explicitly creates two proposals and executes one, confirming the multi-proposal path is a supported and expected usage pattern: [6](#0-5) 

Every production upgrade cycle where participants disagree on which WASM to deploy will trigger this loss.

---

### Recommendation

1. Track the proposer's `AccountId` alongside each `UpdateEntry` in `ProposedUpdates`.
2. In `do_update`, before calling `self.entries.clear()`, iterate over all remaining entries and issue `Promise::new(entry.proposer).transfer(entry.deposit)` for each non-winning proposal.
3. Alternatively, add a `cancel_proposal(id: UpdateId)` endpoint callable only by the original proposer, which removes the entry and refunds the deposit.
4. As a safety net, add an owner-controlled emergency-withdrawal function (analogous to the recommendation in M-02) that can drain accumulated balance in case of unforeseen locking scenarios.

---

### Proof of Concept

1. Participant A calls `propose_update` with WASM-A, attaching 1 NEAR deposit. Contract stores `UpdateEntry { update: WASM-A, bytes_used: N }` and retains exactly `required_deposit(N)`.
2. Participant B calls `propose_update` with WASM-B, attaching 1 NEAR deposit. Same storage path.
3. Threshold participants vote for proposal A via `vote_update`. `do_update` is triggered:
   - `self.entries.remove(&id_A)` — removes WASM-A entry.
   - `self.entries.clear()` — removes WASM-B entry. **No refund issued.**
   - `self.vote_by_participant.clear()` — clears votes.
4. Participant B's 1 NEAR deposit is now permanently locked in the contract balance.
5. No function exists to recover it: there is no `cancel_proposal`, no `withdraw`, and no owner drain path anywhere in `crates/contract/src/lib.rs` or `crates/contract/src/update.rs`. [7](#0-6)

    pub fn do_update(&mut self, id: &UpdateId, gas: Gas) -> Option<Promise> {
        let entry = self.entries.remove(id)?;

        // Clear all entries as they might be no longer valid
        self.entries.clear();
        self.vote_by_participant.clear();

        let mut promise = Promise::new(env::current_account_id());
        match entry.update {
            Update::Contract(code) => {
                // deploy contract then do a `migrate` call to migrate state.
                promise = promise.deploy_contract(code).function_call(
                    method_names::MIGRATE,
                    Vec::new(),
                    NearToken::from_near(0),
                    gas,
                );
            }
            Update::Config(config) => {
                // If we vote for a new config, we should use
                // the value `contract_upgrade_deposit_tera_gas` from the config
                // as the new gas value
                let new_config_gas_value = Gas::from_tgas(config.contract_upgrade_deposit_tera_gas);
                promise = promise.function_call(
                    method_names::UPDATE_CONFIG,
                    serde_json::to_vec(&(&config,)).unwrap(),
                    NearToken::from_near(0),
                    new_config_gas_value,
                );
            }
        }
        Some(promise)
    }

fn bytes_used(update: &Update) -> u128 {
    let mut bytes_used = std::mem::size_of::<UpdateEntry>() as u128;

    // Assume a high max of 128 participant votes per update entry.
    bytes_used += 128 * std::mem::size_of::<AccountId>() as u128;

    match update {
        Update::Contract(code) => {
            bytes_used += code.len() as u128;
        }
        Update::Config(config) => {
            let bytes = serde_json::to_vec(&config).unwrap();
            bytes_used += bytes.len() as u128;
        }
    }

    bytes_used
}

fn required_deposit(bytes_used: u128) -> NearToken {
    env::storage_byte_cost().saturating_mul(bytes_used)
}
