### Title
Negative `RequireMOf` Threshold Unconditionally Satisfies Native Script Validation, Enabling Unauthorized Spending and Minting - (File: `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs`, `eras/shelley/impl/src/Cardano/Ledger/Shelley/Scripts.hs`)

---

### Summary

The `TimelockMOf` and `MultiSigMOf` constructors store their threshold as a signed `Int`. When a negative value is deserialized, `isValidMOf` immediately returns `True` without evaluating any sub-scripts, bypassing all signature and timelock requirements. This is an overly permissive validation bound — the direct analog of the external report's `maxDataDelaySeconds` being too large — and allows an attacker-controlled script to unconditionally satisfy native script validation.

---

### Finding Description

The `TimelockRaw` data type in `eras/allegra/impl/src/Cardano/Ledger/Allegra/Scripts.hs` stores the M-of-N threshold as `!Int`: [1](#0-0) 

The code itself documents the consequence:

> `-- Note that the Int may be negative in which case (TimelockMOf (-2) [..]) is always True`

The evaluator `isValidMOf` inside `evalTimelock` short-circuits to `True` the moment `n <= 0`: [2](#0-1) 

So `RequireMOf (-1) [RequireSignature k1, RequireSignature k2]` evaluates to `True` with zero signatures provided. The same flaw exists in the Shelley `MultiSigMOf`: [3](#0-2) [4](#0-3) 

The CBOR decoder for `TimelockMOf` (tag 3) decodes the threshold field as a plain `Int` with no non-negativity guard: [5](#0-4) 

CBOR major type 1 encodes negative integers natively, so a crafted byte string with a negative threshold deserializes cleanly into a valid `TimelockMOf` or `MultiSigMOf` node.

The formal specification explicitly uses `N` (natural numbers) for the threshold: [6](#0-5) 

And the spec's own Plutus reference implementation notes the missing check: [7](#0-6) 

The implementation diverges from the spec by accepting negative values without rejection.

The `validateNativeScript` / `validateTimelock` path that calls `evalTimelock` is invoked during UTXOW witnessing for every native script in the transaction: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

**Applicable impact**: *Critical — Direct loss or creation of ADA or native assets through an invalid ledger state transition.*

Two concrete scenarios:

1. **Fund theft**: A UTxO is locked at the address `hash(RequireMOf (-1) [RequireSignature victimKey])`. Any party who knows the script preimage can spend the UTxO without providing `victimKey`'s signature. The ledger accepts the transaction because `isValidMOf (-1) [...]` returns `True` unconditionally.

2. **Unauthorized minting**: A minting policy is `hash(RequireMOf (-1) [RequireSignature policyOwner])`. Any party who knows the script can mint or burn tokens under that policy without the owner's authorization, creating or destroying native assets outside design parameters.

Both transitions are invalid per the formal specification (which requires `N ∈ ℕ`) but are accepted by every node running the current implementation.

---

### Likelihood Explanation

**Medium.** The attack requires a UTxO or minting policy to be locked by a script containing a negative threshold. This arises when:

- A script author makes a typo (e.g., `-1` instead of `1`) in a programmatically generated script.
- An attacker socially engineers a victim into deploying a script the attacker crafted (presenting it as a "secure multisig").
- A Babbage+ reference script stored on-chain is publicly visible; an attacker can scan for scripts with negative MOf thresholds and exploit them without any prior interaction with the victim.

The CBOR encoding is valid and accepted by all nodes, so no special network position is required.

---

### Recommendation

1. **Reject negative thresholds at deserialization**: In the `DecCBOR` instance for `TimelockRaw` (tag 3) and `MultiSigRaw` (tag 3), decode the threshold as `Natural` or add an explicit guard that fails with a `DecoderErrorCustom` if the decoded `Int` is negative — mirroring the existing `decodeNatural` guard already present in the binary library: [10](#0-9) 

2. **Change the field type** from `!Int` to `!Natural` (or `!Word`) in `TimelockMOf` and `MultiSigMOf` to make the invariant structural.

3. **Add a guard in `isValidMOf`** as a defense-in-depth measure: treat any `n < 0` as `n = 0` (always true) only if the behavior is intentionally preserved for backward compatibility, but document it as a protocol-level invariant violation.

---

### Proof of Concept

Craft a CBOR-encoded `TimelockRaw` with tag 3 and threshold `-1`:

```
# CBOR: array(2) [ 3, -1, array(1) [ array(2) [0, <keyhash>] ] ]
# Major type 1 encodes -1 as 0x20

82          -- array(2)
  03        -- tag = 3 (TimelockMOf)
  20        -- Int = -1  (CBOR negative integer)
  81        -- array(1)
    82      -- array(2)
      00    -- tag = 0 (TimelockSignature)
      58 1c <28-byte keyhash>
```

When deserialized, this produces `TimelockMOf (-1) [TimelockSignature k]`. Submitting a transaction that:

1. Provides this script in the witness set (matching the UTxO's script hash)
2. Provides **no** `addrTxWits` key signatures

will pass `evalTimelock` because `isValidMOf (-1) [...]` returns `True` at line 487 before evaluating any sub-scripts. [11](#0-10)

evalTimelock vhks (ValidityInterval txStart txExp) = go
  where
    -- The important part of this validator is that it will stop as soon as it reaches the
    -- required number of valid scripts
    isValidMOf n SSeq.Empty = n <= 0
    isValidMOf n (ts SSeq.:<| tss) =
      n <= 0 || if go ts then isValidMOf (n - 1) tss else isValidMOf n tss
    go = \case
      RequireTimeStart lockStart -> lockStart `lteNegInfty` txStart
      RequireTimeExpire lockExp -> txExp `ltePosInfty` lockExp
      RequireSignature hash -> hash `Set.member` vhks
      RequireAllOf xs -> all go xs
      RequireAnyOf xs -> any go xs
      RequireMOf m xs -> isValidMOf m xs
      _ -> error "Impossible: All NativeScripts should have been accounted for"
