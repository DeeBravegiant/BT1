### Title
`UpdateCommittee` Governance Action Allows Enacting Already-Expired Committee Member Terms — (`File: eras/conway/impl/src/Cardano/Ledger/Conway/Rules/Ratify.hs`)

---

### Summary

The Conway `GOV` rule enforces that new committee members in an `UpdateCommittee` proposal must have expiration epochs strictly in the future. However, the `RATIFY` rule's `validCommitteeTerm` guard only checks an upper-bound (max term length) and **omits the lower-bound check** (expiration still in the future). The `ENACT` rule then blindly installs the stored expiration epochs. If a proposal lingers in the queue long enough for the proposed members' terms to expire before ratification, those members are added to the committee in an already-expired state, silently hollowing out the committee.

---

### Finding Description

**Stage 1 — GOV rule (proposal submission):** `conwayGovTransition` rejects any `UpdateCommittee` proposal whose new members have expiration epochs `<= currentEpoch`:

```haskell
let invalidMembers = Map.filter (<= currentEpoch) membersToAdd
 in failOnNonEmptyMap invalidMembers (injectFailure . ExpirationEpochTooSmall)
``` [1](#0-0) 

**Stage 2 — RATIFY rule (ratification):** `validCommitteeTerm` is the only committee-related guard applied at ratification time. It checks only the **upper** bound — that the expiration epoch does not exceed `currentEpoch + maxTermLength` — and says nothing about whether the expiration epoch is still in the future:

```haskell
validCommitteeTerm govAction pp currentEpoch =
  case govAction of
    UpdateCommittee _ _ newMembers _ -> withinMaxTermLength newMembers
    _ -> True
  where
    committeeMaxTermLength = pp ^. ppCommitteeMaxTermLengthL
    withinMaxTermLength = all (<= addEpochInterval currentEpoch committeeMaxTermLength)
``` [2](#0-1) 

The condition `expirationEpoch > currentEpoch` (the lower-bound check present in `GOV`) is **absent** from `validCommitteeTerm`. An expiration epoch of `currentEpoch - 5` trivially satisfies `<= currentEpoch + maxTermLength`, so the ratification guard passes silently.

**Stage 3 — ENACT rule (enactment):** `enactmentTransition` calls `updatedCommittee` which unconditionally inserts the stored expiration epochs into the committee map with no validity re-check:

```haskell
UpdateCommittee _ membersToRemove membersToAdd newThreshold -> do
  st
    & ensCommitteeL %~ SJust . updatedCommittee membersToRemove membersToAdd newThreshold
    & ensPrevCommitteeL .~ SJust (GovPurposeId govActionId)
``` [3](#0-2) 

`updatedCommittee` performs a plain `Map.union` of the new members into the committee, with no expiry check: [4](#0-3)

### Impact Explanation

`committeeAcceptedRatio` treats any member whose `currentEpoch > expiry` as abstaining (excluded from both numerator and denominator):

```haskell
accumVotes (!yes, !tot) member expiry
  | currentEpoch > expiry = (yes, tot) -- member is expired, vote "abstain" (don't count it)
``` [5](#0-4) 

`committeeAccepted` returns `False` (committee vote = No) when `votingCommitteeThreshold` returns `SNothing`, which happens when the committee is too small:

```haskell
SNothing -> False -- this happens if we have no committee, or if the committee is too small
``` [6](#0-5) 

A property test in the test suite confirms that only `NoConfidence` and `UpdateCommittee` actions can pass without a functioning committee; `ParameterChange`, `TreasuryWithdrawals`, `HardForkInitiation`, and `NewConstitution` all require committee approval: [7](#0-6) 

If all enacted committee members are already expired, the committee is effectively empty. Protocol parameter changes, treasury withdrawals, hard-fork initiations, and constitution updates are all blocked from ratification. This matches the **Medium** allowed impact: attacker-controlled proposals modify governance and treasury withdrawal behaviour outside design parameters.

---

### Likelihood Explanation

The attack is reachable by an unprivileged transaction sender. The attacker:

1. Submits an `UpdateCommittee` proposal with members whose expiration epochs are only a few epochs in the future (the `GOV` check passes because `expirationEpoch > currentEpoch` at submission time).
2. Relies on the proposal sitting in the queue for longer than the proposed term length — plausible when DRep participation is low, when the governance action lifetime (`ppGovActionLifetimeL`) exceeds the proposed term length, or when the attacker deliberately delays by withholding votes.
3. Once ratified, the members are enacted with already-expired terms.

The `UpdateCommittee` action does not require committee approval when no functioning committee exists (confirmed by the test above), so this can be triggered even when the committee is already degraded.

---

### Recommendation

Add a lower-bound check inside `validCommitteeTerm` (or directly in `ratifyTransition`) mirroring the `ExpirationEpochTooSmall` guard in `GOV`:

```haskell
validCommitteeTerm govAction pp currentEpoch =
  case govAction of
    UpdateCommittee _ _ newMembers _ ->
      withinMaxTermLength newMembers && allTermsInFuture newMembers
    _ -> True
  where
    committeeMaxTermLength = pp ^. ppCommitteeMaxTermLengthL
    withinMaxTermLength = all (<= addEpochInterval currentEpoch committeeMaxTermLength)
    allTermsInFuture    = all (> currentEpoch)   -- <-- missing lower-bound check
``` [2](#0-1) 

---

### Proof of Concept

1. At epoch **E = 10**, submit an `UpdateCommittee` proposal adding member `M` with `expirationEpoch = 12`. The `GOV` check passes: `12 > 10`.
2. Set `ppGovActionLifetimeL = 5` epochs and `ppCommitteeMaxTermLengthL = 20` epochs.
3. DRep and SPO votes trickle in slowly; the proposal is not ratified until epoch **E = 13**.
4. At ratification, `validCommitteeTerm` checks `12 <= 13 + 20 = 33` — **passes**.
5. `ENACT` installs `M` with `expirationEpoch = 12` into the committee.
6. At epoch 13, `committeeAcceptedRatio` sees `13 > 12` for member `M` → treats `M` as abstaining → committee is below `ppCommitteeMinSizeL` → `committeeAccepted` returns `False`.
7. All subsequent `ParameterChange`, `TreasuryWithdrawals`, `HardForkInitiation`, and `NewConstitution` proposals fail ratification until a new `UpdateCommittee` or `NoConfidence` action is enacted.
