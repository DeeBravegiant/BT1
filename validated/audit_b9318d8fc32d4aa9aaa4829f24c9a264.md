Audit Report

## Title
`block.number` Dependency in `PriceVelocityGuardExtension.beforeSwap` Causes Swap DoS on L2s — (File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol)

## Summary
`PriceVelocityGuardExtension` uses `block.number` to compute `blockDiff`, which scales the allowed price-change budget in `beforeSwap`. On Arbitrum and similar L2s, `block.number` returns the L1 block number, meaning dozens of consecutive L2 blocks share the same value. This collapses `blockDiff` to `0` for the majority of swaps, permanently capping `allowedSq` at `maxChange²` with no relaxation, causing `PriceVelocityExceeded` reverts on any swap where the oracle mid moves more than `maxChangePerBlockE18` within a single L1 epoch.

## Finding Description
In `PriceVelocityGuardExtension.beforeSwap`, the velocity check is:

```solidity
uint256 blockDiff = block.number - prevBlock;          // line 63
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);  // line 70
if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);   // lines 72-74
``` [1](#0-0) 

`prevBlock` is set to `block.number` on every swap (line 58) and on every admin `setLastMidPrice` call (line 32): [2](#0-1) [3](#0-2) 

On Arbitrum, `block.number` returns the L1 block number (≈ one new value every 12 s), while L2 blocks are produced every ≈ 0.25 s — roughly 48 L2 blocks per L1 block. After the first swap in a new L1 block sets `lastUpdateBlock = N`, every subsequent swap in the same L1 epoch reads `block.number = N` again, so `blockDiff = 0` and `allowedSq = maxChange²` for all of them. The intended `sqrt(1 + blockDiff)` relaxation never fires for ~47 of every 48 L2 blocks.

Existing guards are insufficient: there is no chain-ID check, no deployment guard, and no NatSpec warning restricting use to chains where `block.number` increments per produced block.

## Impact Explanation
Any pool on Arbitrum (or another L2 where `block.number` reflects L1 cadence) that attaches this extension will have its `beforeSwap` hook revert with `PriceVelocityExceeded` whenever the oracle mid moves by more than `maxChangePerBlockE18` within a single L1 epoch. Because the oracle price can legitimately move several percent during a 12-second L1 block, and because `blockDiff = 0` for ~47 of every 48 L2 blocks, the guard fires on virtually every swap during any period of normal price movement. This renders the swap path of the affected pool unusable — a broken core pool flow per the contest impact gate ("Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows"). [4](#0-3) 

## Likelihood Explanation
The extension is explicitly designed for deployment on any chain the Metric OMM pool supports, with no chain restriction. A pool admin who calibrates `maxChangePerBlockE18` against L2 block times (the natural unit for a "per-block" parameter) will trigger the bug on every active trading session. Even an admin who calibrates against L1 block times will still see `blockDiff = 0` for the majority of swaps, making the guard far more restrictive than documented. The impact falls on unprivileged traders calling `swap()`, whose transactions revert in `beforeSwap` with no recourse.

## Recommendation
Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension` and rename the parameter to `maxChangePerSecondE18`. The velocity formula becomes:

```solidity
uint256 timeDiff = block.timestamp - prevTimestamp;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + timeDiff);
```

This is chain-agnostic and matches the `block.timestamp`-based approach already used by `OracleValueStopLossExtension` in the same codebase: [5](#0-4) 

Alternatively, add a deployment-time check or explicit NatSpec warning that the extension must not be deployed on any chain where `block.number` does not increment with every produced block.

## Proof of Concept
1. Deploy a Metric OMM pool on Arbitrum with `PriceVelocityGuardExtension` attached.
2. Pool admin calls `setMaxChangePerBlock(pool, 1e15)` — 0.1% per block, a reasonable L2-block-time calibration.
3. Pool admin calls `setLastMidPrice(pool, P0)` — stamps `lastUpdateBlock = block.number` (current L1 block N).
4. Within the same L1 block (same `block.number = N`), the oracle mid moves 0.5% to `P1`.
5. Any user calls `swap(...)`:
   - `blockDiff = N − N = 0`
   - `allowedSq = (1e15)² × 1 = 1e30`
   - `changeE18 = 5e15`, `actualSq = 25e30`
   - `25e30 > 1e30` → revert `PriceVelocityExceeded`
6. All swaps revert for the remaining ~47 L2 blocks of that L1 epoch, even though the price movement is entirely legitimate. The pool is effectively frozen for trading during any period of normal volatility.
