### Title
`get_events` RPC Continuation Token Bypasses `from_block` Filter, Returning Events from Unauthorized Block Range — (File: `crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

The `starknet_getEvents` RPC handler accepts a user-supplied `continuation_token` and uses it directly as the iteration start point without validating that the token's embedded block number falls within the caller-specified `[from_block, to_block]` range. An unprivileged caller can craft a token pointing to block 0 (or any earlier block) and receive events from blocks entirely outside the requested window, producing an authoritative-looking wrong RPC result.

---

### Finding Description

In `get_events`, when a `continuation_token` is present the handler extracts `start_event_index` directly from the token with no bounds check against `from_block_number`:

```rust
// lines 753-759
let start_event_index = match &filter.continuation_token {
    Some(token) => token.parse()?.0,   // ← no check: is token.block >= from_block_number?
    None => EventIndex(
        TransactionIndex(from_block_number, TransactionOffsetInBlock(0)),
        EventIndexInTransactionOutput(0),
    ),
};
``` [1](#0-0) 

The `from_block_number > to_block_number` early-return guard runs before this assignment and cannot protect against a token whose embedded block number is below `from_block_number`: [2](#0-1) 

The subsequent iterator call then scans from the token's block all the way to `to_block_number`, returning every matching event in that wider range: [3](#0-2) 

**Why the token is trivially forgeable.** `ContinuationToken` is a JSON-serialized `EventIndex`. The `parse()` implementation is:

```rust
fn parse(&self) -> Result<ContinuationTokenAsStruct, ErrorObjectOwned> {
    let ct = serde_json::from_str(&self.0)
        .map_err(|_| ErrorObjectOwned::from(INVALID_CONTINUATION_TOKEN))?;
    Ok(ContinuationTokenAsStruct(ct))
}
``` [4](#0-3) 

`EventIndex`, `TransactionIndex`, and `BlockNumber` are all public tuple structs that derive serde with no custom serialization: [5](#0-4) [6](#0-5) 

Serde serializes tuple structs as JSON arrays and newtype structs as their inner value, so the wire format is simply `[[block_number, tx_offset], event_offset]`. The token `"[[0, 0], 0]"` is a valid, parseable continuation token pointing to block 0.

---

### Impact Explanation

A caller who sends:

```json
{
  "from_block": {"block_number": 300},
  "to_block":   {"block_number": 400},
  "continuation_token": "[[0, 0], 0]",
  "chunk_size": 100
}
```

receives events from blocks 0–400 instead of 300–400. The response is structurally identical to a legitimate reply; each event carries its real block number and transaction hash, so the result is **authoritative-looking**. Applications that rely on `starknet_getEvents` to audit a specific block window — L1 bridge monitors, indexers, proof-of-inclusion checks — will silently process events from outside the requested range, potentially re-processing already-handled L1 messages or missing the correct window entirely.

Impact category: **High — RPC returns an authoritative-looking wrong value** (events from blocks outside the requested range).

---

### Likelihood Explanation

The `ContinuationToken` format is trivially reversible from any legitimate response. No authentication or privilege is required to call `starknet_getEvents`. The bypass requires only a single crafted request with a hand-written token string.

---

### Recommendation

After parsing the continuation token, assert that its embedded block number is ≥ `from_block_number`:

```rust
let start_event_index = match &filter.continuation_token {
    Some(token) => {
        let idx = token.parse()?.0;
        if idx.0.0 < from_block_number {
            return Err(ErrorObjectOwned::from(INVALID_CONTINUATION_TOKEN));
        }
        idx
    }
    None => EventIndex(
        TransactionIndex(from_block_number, TransactionOffsetInBlock(0)),
        EventIndexInTransactionOutput(0),
    ),
};
```

### Proof of Concept

1. Node has blocks 0–500 with events in each block.
2. Send `starknet_getEvents` with `from_block={"block_number":300}`, `to_block={"block_number":400}`, `chunk_size=5`, no token → receive 5 events from block 300+, legitimate continuation token `"[[300, 0], 1]"`.
3. Replace the token with `"[[0, 0], 0]"` and resend the identical filter.
4. **Observed**: response contains events from block 0 onwards, not from block 300. The `block_number` fields in the returned events confirm they originate from blocks < 300, violating the `from_block` constraint.
