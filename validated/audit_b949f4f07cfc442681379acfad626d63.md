### Title
Caller-Agnostic `verify_foreign_transaction` Request Key Enables Queue Saturation, Blocking Legitimate Bridge Requests - (File: crates/contract/src/lib.rs)

### Summary
The `verify_foreign_transaction` function uses a caller-agnostic request key (no predecessor account ID), allowing any unprivileged caller to fill the fan-out queue for a specific foreign transaction to its hard cap of 128. Once full, every subsequent legitimate request for that same foreign transaction panics and fails, permanently blocking bridge-inbound flows for that transaction until the attacker stops refilling the queue.

### Finding Description
`SignatureRequest` binds the request key to the caller by hashing `predecessor_id` into the `tweak` field:

```rust
// crates/near-mpc-crypto-types/src/sign.rs
pub fn new(domain: DomainId, payload: Payload, predecessor_id: &AccountId, path: &str) -> Self {
    let tweak = crate::kdf::derive_tweak(predecessor_id, path);
    SignatureRequest { domain_id: domain, tweak, payload }
}
```

`VerifyForeignTransactionRequest` contains no such binding — the struct is purely `(request, domain_id, payload_version)` with no caller field:

```rust
// crates/near-mpc-contract-interface/src/types/foreign_chain.rs
pub struct VerifyForeignTransactionRequest {
    pub request: ForeignChainRpcRequest,
    pub domain_id: DomainId,
    pub payload_version: ForeignTxPayloadVersion,
}
```

The contract itself confirms this design in a test comment: *"a different account would today be blocked from receiving a response by alice's submission"* — acknowledging that any caller submitting the same `(tx_id, domain_id, payload_version)` tuple lands in the same fan-out queue.

The fan-out queue enforces a hard cap:

```rust
// crates/contract/src/pending_requests.rs
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

pub(crate) fn push_pending_yield<K>(...) {
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull { limit: MAX_PENDING_REQUEST_FAN_OUT }.to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
}
```

When the queue is full, `env::panic_str` is called, reverting the entire transaction. There is no per-caller rate limit, no per-caller slot reservation, and no way for a legitimate user to bypass a saturated queue.

### Impact Explanation
An attacker who knows a target foreign `tx_id` (observable from any bridge front-end or mempool) can submit 128 identical `verify_foreign_transaction` calls for that transaction. Every subsequent call from the legitimate bridge user fails with `PendingRequestQueueFull`. When MPC nodes eventually respond and drain the queue, the attacker immediately re-saturates it with another 128 calls. This creates a sustained blockade: the legitimate bridge transfer never completes. For Omnibridge inbound flows with time-bounded settlement windows, this causes permanent loss of the bridged funds for the victim. This is a request-lifecycle manipulation that breaks the production safety invariant that a valid `verify_foreign_transaction` submission will eventually be serviced.

### Likelihood Explanation
The minimum deposit per call is 1 yoctoNEAR (as used throughout the test suite). Saturating the queue costs 128 yoctoNEAR per cycle — effectively free. The target `tx_id` is public on the foreign chain and visible in any bridge UI. No privileged access, TEE bypass, or threshold collusion is required. The attacker only needs a NEAR account and knowledge of the target transaction.

### Recommendation
Bind the `VerifyForeignTransactionRequest` key to the caller's account ID, mirroring the `sign` path. The simplest approach is to add a `predecessor_id` field to `VerifyForeignTransactionRequest` and populate it from `env::predecessor_account_id()` inside `verify_foreign_transaction`, so each caller gets an independent queue slot and cannot saturate another caller's queue.

### Proof of Concept

```
1. Alice initiates a bridge transfer; she will call verify_foreign_transaction
   for Bitcoin tx_id = [0xAB; 32].

2. Attacker observes the target tx_id from the bridge UI or foreign-chain mempool.

3. Attacker submits 128 verify_foreign_transaction calls for the same
   (tx_id=[0xAB;32], domain_id, payload_version) from any NEAR account.
   Cost: 128 × 1 yoctoNEAR ≈ 0.

4. Alice submits her verify_foreign_transaction call.
   → push_pending_yield panics: "Pending-request queue is full (limit: 128)"
   → Alice's transaction is reverted; she receives no yield promise.

5. MPC nodes process the request and call respond_verify_foreign_tx.
   → resolve_yields_for drains all 128 attacker yields.

6. Attacker immediately re-submits 128 calls (next NEAR block).

7. Steps 4–6 repeat indefinitely. Alice's bridge transfer never completes.
   If the bridge has a settlement deadline, her funds are permanently lost.
```

pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
