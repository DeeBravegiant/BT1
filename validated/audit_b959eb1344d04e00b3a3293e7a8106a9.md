Audit Report

## Title
`purgeStateGuardRole` Does Not Clear `pendingStateGuard`, Allowing Stale Nominee to Seize Oracle Guard Role — (File: smart-contracts-poc/contracts/oracles/providers/OracleBase.sol)

## Summary

`purgeStateGuardRole` deletes `stateGuard[feedId]` but leaves `pendingStateGuard[feedId]` intact. A pending nominee set before abdication can call the public `acceptStateGuardRole` and install themselves as the new `stateGuard`, gaining unauthorized control over `setPriceGuard` for that feed. This allows the attacker to disable the price band, permitting any oracle price to reach pool swap execution.

## Finding Description

In `smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`, `purgeStateGuardRole` (L120–124) only deletes `stateGuard[feedId]`:

```solidity
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];          // pendingStateGuard[feedId] NOT cleared
    emit StateGuardDeleted(feedId);
}
```

`acceptStateGuardRole` (L111–118) is a public function that checks only `pendingStateGuard`, with no dependency on `stateGuard`:

```solidity
function acceptStateGuardRole(bytes32 feedId) external {
    require(pendingStateGuard[feedId] == msg.sender, InvalidGuard(msg.sender));
    delete pendingStateGuard[feedId];
    stateGuard[feedId] = msg.sender;
    emit StateGuardUpdated(feedId, msg.sender);
}
```

After `purgeStateGuardRole`, `stateGuard[feedId]` is `address(0)` and `checkRole` falls back to `ADMIN_ROLE`. However, the pending nominee can immediately call `acceptStateGuardRole` and install themselves as the new `stateGuard`, overriding ADMIN's restored authority. A separate `purgePendingStateGuardRole` (L105–109) exists and confirms the protocol intended pending state to be clearable, but `purgeStateGuardRole` never invokes it.

Exploit path:
1. Guard calls `setStateGuardRole(feedId, attacker)` → `pendingStateGuard[feedId] = attacker`
2. Guard calls `purgeStateGuardRole(feedId)` → `stateGuard[feedId] = address(0)`, `pendingStateGuard[feedId]` remains `attacker`
3. Attacker calls `acceptStateGuardRole(feedId)` → `stateGuard[feedId] = attacker` (unauthorized seizure)
4. Attacker calls `setPriceGuard(feedId, 1, type(uint128).max)` → price band disabled

## Impact Explanation

`stateGuard` is the sole authority for `setPriceGuard` once set (ADMIN is locked out by `checkRole`). The `priceGuard` is enforced at the price provider layer in `PriceProvider.sol` (L208–210), `PriceProviderL2.sol` (L225–228), `ProtectedPriceProvider.sol` (L202–205), and `AnchoredPriceProvider.sol` (L290–292). Setting `guardMin = 1` and `guardMax = type(uint128).max` effectively disables the price band, allowing any oracle price — including stale or manipulated values — to pass the guard check and reach pool swap execution. This satisfies the allowed impact gate: **bad-price execution** (unbounded bid/ask quote reaches a pool swap) and **oracle role check bypassed by an unprivileged path**.

## Likelihood Explanation

The trigger requires the current guard to: (1) nominate a pending successor via `setStateGuardRole`, then (2) abdicate via `purgeStateGuardRole`. This is a realistic operational sequence — a guard proposes a successor, then decides to fully step down instead of completing the transfer. The attacker's final step (`acceptStateGuardRole`) is a public, permissionless call requiring no role. The attacker can also front-run any ADMIN `purgePendingStateGuardRole` call in the same block as the abdication.

## Recommendation

Clear `pendingStateGuard` inside `purgeStateGuardRole`:

```solidity
function purgeStateGuardRole(bytes32 feedId) external checkRole(feedId) {
    delete stateGuard[feedId];
+   delete pendingStateGuard[feedId];   // prevent stale nominee from claiming
    emit StateGuardDeleted(feedId);
}
```

## Proof of Concept

```
1. ADMIN calls setStateGuardRole(feedId, guard1)
2. guard1 calls acceptStateGuardRole(feedId)
   → stateGuard[feedId] = guard1

3. guard1 calls setStateGuardRole(feedId, attacker)
   → pendingStateGuard[feedId] = attacker

4. guard1 calls purgeStateGuardRole(feedId)
   → stateGuard[feedId] = address(0)
   → pendingStateGuard[feedId] = attacker  ← NOT cleared

5. attacker calls acceptStateGuardRole(feedId)
   → pendingStateGuard[feedId] == attacker ✓
   → stateGuard[feedId] = attacker         ← unauthorized seizure

6. attacker calls setPriceGuard(feedId, 1, type(uint128).max)
   → price guard disabled; any oracle price passes provider guard check
   → pool swaps execute at unbounded prices
```
