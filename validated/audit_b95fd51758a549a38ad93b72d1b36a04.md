Audit Report

## Title
Excess ETH Permanently Locked in `updateRate()` Due to Missing Refund and No Recovery Path - (File: contracts/cross-chain/MultiChainRateProvider.sol)

## Summary

`MultiChainRateProvider.updateRate()` is an unrestricted `external payable` function that consumes exactly the on-chain `estimatedFee` per LayerZero receiver but enforces no equality between `msg.value` and total fees consumed. The contract inherits only `Ownable` and `ReentrancyGuard`, contains no `receive()` fallback, no `withdraw()`, and no sweep function. Any ETH sent above the sum of per-receiver fees is permanently irrecoverable.

## Finding Description

`updateRate()` at lines 108–137 of `contracts/cross-chain/MultiChainRateProvider.sol` is callable by any external account with no access control:

```solidity
function updateRate() external payable nonReentrant {
    ...
    for (uint256 i; i < rateReceiversLength;) {
        (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
            .estimateFees(dstChainId, address(this), _payload, false, bytes(""));
        ILayerZeroEndpoint(layerZeroEndpoint).send{ value: estimatedFee }(...);
        ...
    }
}
```

The actual fee consumed per receiver is determined at execution time by the on-chain `estimateFees` call. Callers must pre-commit ETH before knowing the exact total, making over-sending structurally necessary. There is:

1. No pre-loop accumulation and `require(msg.value == totalFee)` guard.
2. No post-loop refund of `msg.value - totalConsumed` to `msg.sender`.
3. No ETH recovery path anywhere in the contract — `MultiChainRateProvider` inherits only `Ownable` and `ReentrancyGuard` (line 13), neither of which provides ETH withdrawal.

The helper `estimateTotalFee()` (lines 154–173) exists as a view function but is not called by `updateRate()` and cannot atomically enforce equality. `RSETHMultiChainRateProvider` (line 9 of `RSETHMultiChainRateProvider.sol`) inherits this behavior directly without adding any guard or override.

By contrast, `L1Vault.bridgeRsETHToL2()` (lines 236–238) enforces `if (msg.value != nativeFee) revert IncorrectNativeFee()` — the same pattern is absent here.

## Impact Explanation

Any ETH sent in excess of the sum of per-receiver `estimatedFee` values is permanently locked in `MultiChainRateProvider` / `RSETHMultiChainRateProvider` with no recovery path for any party. This constitutes **Critical — Permanent freezing of funds**.

## Likelihood Explanation

`updateRate()` carries no access control — any EOA, keeper bot, or integrating contract can call it. Callers must estimate the correct total fee off-chain before the transaction, but LayerZero fees fluctuate with gas prices. Over-sending is a standard defensive pattern. The number of receivers can change between an off-chain estimate and the actual call (owner can add/remove receivers), making exact-fee computation non-atomic. Keeper bots and integrators commonly do not implement exact-fee logic. The condition is easily and repeatedly triggerable.

## Recommendation

Add a pre-loop fee accumulation and equality check, mirroring the pattern already used in `L1Vault.bridgeRsETHToL2()`:

```solidity
uint256 totalFee = 0;
for (uint256 i; i < rateReceiversLength;) {
    uint16 dstChainId = uint16(rateReceivers[i]._chainId);
    (uint256 estimatedFee,) = ILayerZeroEndpoint(layerZeroEndpoint)
        .estimateFees(dstChainId, address(this), _payload, false, bytes(""));
    totalFee += estimatedFee;
    unchecked { ++i; }
}
require(msg.value == totalFee, "Incorrect ETH amount sent");
// then execute sends
```

Alternatively, refund `address(msg.sender).call{value: msg.value - totalConsumed}("")` after the send loop.

## Proof of Concept

1. Owner configures two `rateReceivers` on `RSETHMultiChainRateProvider`.
2. Caller queries `estimateTotalFee()` off-chain; it returns `0.01 ETH`.
3. Caller calls `updateRate{ value: 0.02 ETH }()` (over-estimating defensively, or using a stale estimate).
4. The loop calls `estimateFees` on-chain per receiver and forwards exactly `estimatedFee` each time, consuming `0.01 ETH` total.
5. The remaining `0.01 ETH` remains in the contract.
6. No `withdraw()`, no `receive()`, no sweep function exists — the `0.01 ETH` is permanently locked.

Foundry fork test: deploy `RSETHMultiChainRateProvider` with a mock `ILayerZeroEndpoint` that returns a fixed `estimatedFee`, add two receivers, call `updateRate{value: 2 * estimatedFee + 1 wei}()`, assert `address(provider).balance == 1 wei` and that no function can recover it.         for (uint256 i; i < rateReceiversLength;) {

                .estimateFees(dstChainId, address(this), _payload, false, bytes(""));

            );

            }

    }
