Audit Report

## Title
Unconditional Zero-Value Fee Transfer in `claim()` Permanently Freezes Unclaimed Yield for Revert-on-Zero ERC20 Tokens - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim()` unconditionally calls `IERC20(token).safeTransfer(protocolTreasury, fee)` at line 144 without checking whether `fee > 0`. When `feeInBPS` is set to `0` (a valid configuration) or when `claimableAmount * feeInBPS < 10_000` (integer truncation), `fee` evaluates to zero. Any ERC20 token that reverts on zero-value transfers will cause every `claim()` invocation to revert, permanently freezing all unclaimed yield inside the contract. The sibling contracts `KernelMerkleDistributor` and `KernelTop100MerkleDistributor` already apply the correct `if (fee > 0)` guard, confirming the team is aware of this risk class.

## Finding Description
In `MerkleDistributor.sol`, the `claim()` function computes the fee and transfers it unconditionally:

```solidity
// L138-144
uint256 fee = (claimableAmount * feeInBPS) / 10_000;
uint256 amountToSend = claimableAmount - fee;

IERC20(token).safeTransfer(account, amountToSend);

// Send the fee to the protocol treasury
IERC20(token).safeTransfer(protocolTreasury, fee);  // no zero-check
``` [1](#0-0) 

`fee` evaluates to `0` in two realistic scenarios:
1. `feeInBPS == 0`: `setFeeInBPS` only enforces `_feeInBPS > MAX_FEE_IN_BPS`, so `0` passes validation. When set, every claim produces `fee = 0`.
2. `claimableAmount * feeInBPS < 10_000`: integer division truncates to zero for small claim amounts even with a non-zero `feeInBPS`. [2](#0-1) 

The `token` address is owner-configurable via `setToken` with no restriction on token type. Tokens that revert on zero-value transfers (e.g., LEND, BNB) will cause `safeTransfer(protocolTreasury, 0)` to revert. Because the entire transaction reverts atomically, the state update at lines 134–135 is also rolled back, meaning the user's claim is never marked as consumed — but the call will revert on every subsequent attempt as well, permanently blocking all claims. [3](#0-2) 

The correct guard is already present in both sibling contracts: [4](#0-3) [5](#0-4) 

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

When the distributed token reverts on zero-value transfers and `fee` computes to `0`, every call to `claim()` reverts. All token allocations committed to the merkle tree remain locked inside the contract indefinitely. Users cannot receive their entitled yield. If `feeInBPS` is `0`, this affects 100% of claims for the lifetime of that configuration.

## Likelihood Explanation
- `feeInBPS = 0` is explicitly permitted by the `setFeeInBPS` validation logic — no lower-bound guard exists.
- The token is generic and owner-configurable; deploying with or switching to a zero-revert token is a realistic operational scenario for a generic distributor.
- No attacker action is required — any ordinary user calling `claim()` under these conditions triggers the freeze.
- The fix is already applied in two sibling contracts in the same repository, confirming the team recognizes this risk class.

## Recommendation
Add a zero-check before the fee transfer, consistent with the pattern used in `KernelMerkleDistributor` and `KernelTop100MerkleDistributor`:

```solidity
if (fee > 0) {
    IERC20(token).safeTransfer(protocolTreasury, fee);
}
```

## Proof of Concept
1. Deploy `MerkleDistributor` with a token that reverts on zero-value transfers (e.g., LEND).
2. Call `setFeeInBPS(0)` — passes the `> MAX_FEE_IN_BPS` guard.
3. Set a valid merkle root and fund the contract with the token.
4. Have any eligible user call `claim()` with a valid proof.
5. Execution reaches line 144: `fee = (claimableAmount * 0) / 10_000 = 0`.
6. `IERC20(token).safeTransfer(protocolTreasury, 0)` reverts because the token rejects zero-value transfers.
7. The entire transaction reverts; the user's claim state is not updated. Every subsequent attempt by any user produces the same revert. All yield is permanently frozen.

**Foundry test sketch:**
```solidity
function test_claimRevertsOnZeroFeeWithZeroRevertToken() public {
    // Deploy MerkleDistributor with a mock ERC20 that reverts on transfer(addr, 0)
    distributor.setFeeInBPS(0);
    // Set merkle root, fund contract
    vm.expectRevert();
    distributor.claim(index, account, cumulativeAmount, proof);
}
```
