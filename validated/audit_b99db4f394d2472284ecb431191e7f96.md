Audit Report

## Title
ETH Deposit Limit Check Ignores Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` in `LRTDepositPool` omits the incoming `amount` from the ETH branch of the limit comparison, checking only whether the pre-deposit total already exceeds the cap. Any unprivileged caller can bypass the ETH deposit cap in a single transaction by sending an arbitrarily large ETH value while the pre-deposit total remains at or below the limit.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, the function `_checkIfDepositAmountExceedesCurrentLimit` contains an asymmetric check:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));  // amount ignored
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));  // correct
}
```

The ETH branch never adds `amount` to `totalAssetDeposits`, so the check only returns `true` if the cap was already exceeded before the deposit. The function is called unconditionally at line 661 inside `_beforeDeposit`, which is invoked by the public, permissionless `depositETH` at line 87. Because `_beforeDeposit` reverts only when the function returns `true`, any ETH deposit is accepted as long as the pre-deposit total is ≤ the configured limit, regardless of the deposit size.

## Impact Explanation
The deposit limit is a risk-management invariant that caps protocol ETH exposure. Because the check ignores `amount` for ETH, a single depositor can push ETH TVL arbitrarily above the configured cap in one transaction. This breaks the protocol's stated deposit cap guarantee, causing over-allocation into EigenLayer ETH strategies and inflated rsETH supply relative to the risk budget. Deposited funds are not directly lost, but the contract fails to deliver its promised deposit cap control. **Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
`depositETH` is external and requires no role or special condition. Any depositor who observes that `getTotalAssetDeposits(ETH_TOKEN) ≤ depositLimitByAsset(ETH_TOKEN)` can immediately exploit this by sending a large ETH value in a single call. No front-running, privileged access, or multi-step setup is required. **Likelihood: High.**

## Recommendation
Include `amount` in the ETH branch to mirror the ERC20 logic. The two branches can be unified:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
Preconditions:
- `depositLimitByAsset(ETH_TOKEN) = 1000 ether`
- `getTotalAssetDeposits(ETH_TOKEN) = 900 ether`

Steps:
1. Attacker calls `depositETH{value: 5000 ether}(minRSETH, "")`.
2. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 5000 ether)` evaluates `900 ether > 1000 ether` → `false`.
3. `_beforeDeposit` does not revert; deposit proceeds.
4. `getTotalAssetDeposits(ETH_TOKEN)` becomes `5900 ether` — 5.9× the intended cap.
5. Attacker receives rsETH minted at the current rate; the protocol holds ETH far beyond its risk budget.

Foundry test plan: deploy `LRTDepositPool` with a mock `lrtConfig` returning `depositLimitByAsset = 1000 ether` and `getTotalAssetDeposits` returning `900 ether`; call `depositETH{value: 5000 ether}`; assert the call succeeds and `getTotalAssetDeposits` returns `5900 ether`, confirming the cap is bypassed.
