Audit Report

## Title
Stale Cross-Chain Oracle Rate Allows Over-Minting of wrsETH on L2 Pools - (File: contracts/cross-chain/CrossChainRateReceiver.sol)

## Summary
`CrossChainRateReceiver.getRate()` returns the stored `rate` unconditionally with no staleness check, despite `lastUpdated` being recorded on every `lzReceive` call. Because rsETH accrues value continuously on L1, any delay in LayerZero message delivery leaves the L2 oracle quoting a lower-than-current rate. Every L2 pool variant uses this rate as the denominator when computing wrsETH to mint, so a stale (lower) rate yields a larger mint amount, allowing an attacker to extract unclaimed yield from existing holders.

## Finding Description
`CrossChainRateReceiver.lzReceive` records both `rate` and `lastUpdated` on receipt of a LayerZero message from L1: [1](#0-0) 

`getRate()` returns the stored value with no staleness guard: [2](#0-1) 

`RSETHRateReceiver` inherits this implementation directly and is wired into every L2 pool as `rsETHOracle`: [3](#0-2) 

Every L2 pool's deposit path calls `getRate()` via `viewSwapRsETHAmountAndFee` and divides: [4](#0-3) 

The same pattern is present in `RSETHPoolV3ExternalBridge`: [5](#0-4) 

On L1, `LRTOracle._updateRsETHPrice` computes a new price that is monotonically non-decreasing as EigenLayer rewards accrue: [6](#0-5) 

When the L2 oracle has not been refreshed, `rsETHToETHrate` is lower than the true L1 value. A lower denominator yields a larger `rsETHAmount`, so the depositor receives more wrsETH than their ETH is worth at the current exchange rate. The `dailyMintLimit` guard limits scale but does not prevent the attack; the `paused` modifier is irrelevant to rate staleness.

## Impact Explanation
Every wrsETH token is a proportional claim on the L1 rsETH pool. Over-minting wrsETH at a stale rate dilutes the yield claims of all existing wrsETH/rsETH holders. The attacker bridges or redeems the surplus wrsETH on L1 at the true current rate, extracting ETH value that rightfully belongs to prior depositors as accrued yield. This is a concrete, quantifiable **theft of unclaimed yield — High severity**.

## Likelihood Explanation
The L2 oracle is updated only when a LayerZero message is relayed from L1. Network congestion, relayer downtime, or infrequent pushes all create exploitable windows. Because rsETH price increases continuously with EigenLayer rewards, even a few hours of staleness produces a measurable discrepancy. `lastUpdated` is a public storage variable, so an attacker can monitor it on-chain and deposit precisely when the gap is largest. No special privileges are required; any address can call `deposit()`.

## Recommendation
Add a configurable maximum staleness threshold and revert in `getRate()` if the rate is too old:

```solidity
uint256 public maxStaleness; // e.g. 1 hours

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= maxStaleness, "Rate is stale");
    return rate;
}
```

Alternatively, each L2 pool's deposit path should check `CrossChainRateReceiver.lastUpdated` directly before accepting a deposit. The `maxStaleness` value should be set conservatively relative to the expected LayerZero relay frequency.

## Proof of Concept
1. L1 `LRTOracle.updateRSETHPrice()` is called; `rsETHPrice` rises from `1.05e18` to `1.10e18`.
2. The LayerZero relayer has not yet delivered the updated rate to `RSETHRateReceiver` on L2; `rate` still reads `1.05e18`.
3. Attacker calls `RSETHPoolV3.deposit{value: 1 ether}("")`.
4. `viewSwapRsETHAmountAndFee(1 ether)` computes `rsETHAmount = 1e18 * 1e18 / 1.05e18 ≈ 0.9524 wrsETH`.
5. Fair amount at current rate: `1e18 * 1e18 / 1.10e18 ≈ 0.9091 wrsETH`.
6. Attacker receives ≈ 0.0433 wrsETH surplus — worth ≈ 0.0476 ETH at the true rate — at the expense of existing holders.
7. Attacker bridges the wrsETH to L1 and redeems for full current rsETH value, realising the profit.

**Foundry fork test plan**: Fork an L2 where `RSETHRateReceiver` is deployed. Warp `block.timestamp` forward by several hours without triggering `lzReceive`. Call `RSETHPoolV3.deposit` and assert that the minted `wrsETH` amount exceeds `deposit_amount * 1e18 / true_L1_rate`, confirming over-minting proportional to the staleness gap.
