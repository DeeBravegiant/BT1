Tracing the full call chain to verify the guard logic:

**Call chain:** `cbmpc_pve_encrypt_with_kem` → `c_base_kem_adapter_t::encrypt` → `pve_base_pke_runtime_kem().encrypt` → `kem_aead_ciphertext_t::seal` → `kem_policy_runtime_kem_t::encapsulate` → `kem_encap_shim`

**The guard in `kem_encap_shim`:**

```cpp
if (kem_ss.size < 0 || (kem_ss.size > 0 && !kem_ss.data)) {
    ...
    return E_FORMAT;
}
out_kem_ss = coinbase::buf_t(kem_ss.data, kem_ss.size);
```

When a callback returns `{NULL, 0}`: `size < 0` is false, `size > 0 && !data` is false → the guard passes, `out_kem_ss` is set to an empty `buf_t`, and `CBMPC_SUCCESS` is returned.

**No downstream check either:** `kem_policy_runtime_kem_t::encapsulate` returns the callback result directly with no size check, and `kem_aead_ciphertext_t::seal` immediately uses `kem_ss` in HKDF with no non-empty assertion.

```cpp

buf_t aes_key = crypto::hkdf_expand_sha256(prk, mem_t("CBMPC|KEM-AEAD|..."), 32);
crypto::aes_gcm_t::encrypt(aes_key, ...);  // succeeds, returns ciphertext
```

With empty `kem_ss`, HKDF-Extract produces a deterministic PRK (HMAC-SHA256 with all-zero key over empty IKM), so the AES-GCM key is fully determined and known to the attacker. The encryption succeeds and returns a well-formed ciphertext.

### Title
Missing non-empty shared-secret guard in `kem_encap_shim` allows zero-entropy PVE ciphertext production - (`src/cbmpc/c_api/pve_base_pke.cpp`)

### Summary
`kem_encap_shim` accepts a KEM callback output of `{NULL, 0}` for `kem_ss` (empty shared secret) as valid, propagating an empty `buf_t` into `kem_aead_ciphertext_t::seal`, which silently derives a deterministic, attacker-known AES-GCM key and produces a structurally valid PVE ciphertext encrypting the scalar `x` under zero-entropy key material.

### Finding Description
The ABI validation guard in `kem_encap_shim` only rejects negative sizes or a positive size paired with a null pointer: [1](#0-0) 

A callback returning `{NULL, 0}` for `kem_ss` satisfies neither rejection condition. The shim then copies the empty buffer into `out_kem_ss` and returns `CBMPC_SUCCESS`: [2](#0-1) 

`kem_policy_runtime_kem_t::encapsulate` performs no post-callback size check and returns the result directly: [3](#0-2) 

`kem_aead_ciphertext_t::seal` then uses the empty `kem_ss` directly in HKDF without any non-empty assertion: [4](#0-3) 

HKDF-Extract over empty IKM with empty salt is deterministic and produces a fixed PRK, so the derived AES-256-GCM key is fully known to anyone who knows the shared secret is empty (i.e., the attacker who controls the callback). The `seal` call succeeds and returns a well-formed ciphertext blob.

### Impact Explanation
The attacker (malicious KEM callback provider) causes `cbmpc_pve_encrypt_with_kem` to return `CBMPC_SUCCESS` with a structurally valid PVE ciphertext whose KEM component encapsulates an empty shared secret. The AES-GCM key is deterministic and known to the attacker, so the scalar `x` (the PVE plaintext) is exposed. Additionally, an honest decryptor using a correct decap callback will derive a different, non-empty shared secret and fail to decrypt, creating an encap/decap disagreement. This falls squarely in the **High** impact category: a public API reachable validation bypass in PVE creates accepted invalid cryptographic output.

### Likelihood Explanation
The path is directly reachable from the public C API `cbmpc_pve_encrypt_with_kem` by any caller who supplies a custom KEM struct. The attacker only needs to return `{NULL, 0}` from their `encap` callback. No threshold collusion, no memory compromise, and no special privilege is required.

### Recommendation
Add an explicit non-empty check for `kem_ss` in `kem_encap_shim` immediately after the existing ABI check:

```cpp
// existing check
if (kem_ss.size < 0 || (kem_ss.size > 0 && !kem_ss.data)) { ... return E_FORMAT; }
// add:
if (kem_ss.size == 0) {
    if (kem_ct.data) cbmpc_cmem_free(kem_ct);
    return E_FORMAT;  // or a dedicated E_CRYPTO / E_BADARG
}
```

The same guard should be added symmetrically in `kem_decap_shim` for `kem_ss`. Optionally, `kem_policy_runtime_kem_t::encapsulate` could add a post-callback assertion as a defence-in-depth layer.
