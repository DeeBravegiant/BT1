Audit Report

## Title
ETH Withdrawal Permanently Frozen for Contract Recipients That Revert on ETH Receive - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager` uses a push-payment model to deliver ETH to withdrawing users. When a user's address is a contract that reverts on ETH receipt, `completeWithdrawal` permanently fails for that user. Because rsETH is burned in a prior, already-finalized transaction (`unlockQueue`), the user's rsETH is permanently destroyed while their corresponding ETH is frozen inside `LRTWithdrawalManager` with no on-chain recovery path.

## Finding Description
The root cause is a two-transaction design where an irreversible state change (rsETH burn) occurs in Transaction 1, while the ETH delivery that depends on recipient cooperation occurs in Transaction 2.

**Transaction 1 – `unlockQueue`** finalizes the burn at [1](#0-0)  — rsETH is permanently burned via `burnFrom` and ETH is redeemed from `LRTUnstakingVault` into `LRTWithdrawalManager`. This transaction is finalized on-chain and cannot be undone.

**Transaction 2 – `completeWithdrawal`** calls `_processWithdrawalCompletion`, which at [2](#0-1)  calls `_transferAsset`. The ETH delivery is implemented as a push via low-level call at [3](#0-2)  — if the recipient contract has no `receive()` or a reverting fallback, `sent = false` and `revert EthTransferFailed()` is triggered.

When Transaction 2 reverts, all in-transaction state changes are undone: the `popFront()` at [4](#0-3) , the `delete withdrawalRequests[requestId]` at [5](#0-4) , and the `unlockedWithdrawalsCount[asset]--` at [6](#0-5)  are all reverted. The withdrawal request is restored to the queue. However, the rsETH burn from Transaction 1 is not undone — it is permanently gone.

Every subsequent retry of `completeWithdrawal` or `completeWithdrawalForUser` will identically revert. There is no admin function to redirect ETH to an alternate address or to cancel the withdrawal and restore rsETH.

The only potential recovery path, `sweepRemainingAssets`, is permanently blocked because `unlockedWithdrawalsCount[asset]` remains > 0 (restored by the revert), causing the check at [7](#0-6)  to always revert with `PendingWithdrawalsExist`.

## Impact Explanation
This constitutes **Critical — Permanent freezing of funds**. The affected user loses their rsETH (burned, irreversible) and their corresponding ETH is locked inside `LRTWithdrawalManager` indefinitely. No on-chain mechanism exists to recover or redirect the frozen ETH. The `sweepRemainingAssets` escape hatch is also blocked, meaning the ETH cannot even be recovered to treasury.

## Likelihood Explanation
Smart-contract wallets (Gnosis Safe multisigs), protocol-owned treasuries, and DeFi integrations routinely hold rsETH and initiate withdrawals. Many such contracts do not implement a `receive()` function. The protocol's own NatDoc on `completeWithdrawalForUser` at [8](#0-7)  explicitly states "Not expected to be used for ETH; potential gas grief scenarios are non-impactful for ETH," confirming developer awareness of the fragility of ETH push-delivery to arbitrary addresses. Any smart-contract wallet holder who initiates an ETH withdrawal triggers this path without any attacker involvement.

## Recommendation
Replace the push-payment model for ETH with a pull-payment (claimable) model:
- In `_processWithdrawalCompletion`, instead of calling `_transferAsset` immediately, record the owed amount in a `mapping(address => uint256) public claimableETH`.
- Add a separate `claimETH()` function that allows the user (or any address they designate) to pull their ETH at any time.
- This eliminates the dependency on the recipient's ability to receive ETH and decouples the accounting step from the delivery step, preventing the permanent freeze scenario.

## Proof of Concept
1. A smart-contract wallet with no `receive()` holds rsETH and calls `initiateWithdrawal(ETH_TOKEN, rsETHAmount, "")`. rsETH is transferred to `LRTWithdrawalManager`.
2. Operator calls `unlockQueue(ETH_TOKEN, ...)`. rsETH is burned via `burnFrom` (L305) and ETH is redeemed from `LRTUnstakingVault` (L307). Transaction finalized.
3. The smart-contract wallet calls `completeWithdrawal(ETH_TOKEN, "")`. Inside `_processWithdrawalCompletion`, `_transferAsset` executes `payable(wallet).call{value: amount}("")`. The wallet has no `receive()`, so `sent = false`. `revert EthTransferFailed()` triggers; entire transaction reverts.
4. All in-tx state is restored. rsETH burn (prior tx) is not undone. ETH remains in `LRTWithdrawalManager`.
5. Every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` repeats step 3 and reverts.
6. `sweepRemainingAssets` is permanently blocked: `unlockedWithdrawalsCount[ETH_TOKEN] > 0` causes `hasUnlockedWithdrawals` to return true, triggering `PendingWithdrawalsExist` revert at L403.
7. User's ETH is permanently frozen with no recovery path.
