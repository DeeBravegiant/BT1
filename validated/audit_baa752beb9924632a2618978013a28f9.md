Audit Report

## Title
`viewSwapRsETHAmountAndFee` Returns Non-Zero Quotes When Deposits Would Revert Due to Paused State, Daily Mint Limit, or Disabled ETH Deposits - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The public view function `viewSwapRsETHAmountAndFee` in `RSETHPoolV3.sol` (and sibling pool contracts) is intended to preview the wrsETH amount a user would receive for a given ETH deposit. However, it omits the same guards enforced by `deposit`: `whenNotPaused`, `limitDailyMint` (which checks `startTimestamp` and `dailyMintLimit`), and the `isEthDepositEnabled` flag. As a result, the function returns a positive, non-zero quote even when the corresponding `deposit` call would revert, misleading users and off-chain integrators into submitting failing transactions.

## Finding Description
In `RSETHPoolV3.sol`, `deposit(string)` at lines 246–253 is decorated with `whenNotPaused`, `limitDailyMint(msg.value, ETH_IDENTIFIER)`, and an inline `if (!isEthDepositEnabled) revert EthDepositDisabled()` check. The `limitDailyMint` modifier at lines 96–125 additionally enforces `block.timestamp >= startTimestamp` and `dailyMintAmount + rsETHAmount <= dailyMintLimit`.

By contrast, `viewSwapRsETHAmountAndFee(uint256)` at lines 299–308 only computes `fee` and `rsETHAmount` from the oracle rate — it performs none of these checks. The same pattern is present in `RSETHPoolV2.sol` lines 225–234, and the claim's references to `RSETHPoolV3ExternalBridge.sol` and `RSETHPoolV3WithNativeChainBridge.sol` follow the identical structure.

Exploit flow:
1. The `PAUSER_ROLE` calls `pause()`, setting `paused = true` (or the daily cap is exhausted, or `isEthDepositEnabled` is `false`).
2. Any external caller invokes `viewSwapRsETHAmountAndFee(1 ether)` — it returns `(rsETHAmount > 0, fee > 0)` with no revert.
3. The caller submits `deposit{value: 1 ether}("")` — the `whenNotPaused` modifier reverts with `ContractPaused` (or `DailyMintLimitExceeded` / `EthDepositDisabled`).
4. The caller loses gas; the view function's promise is broken.

Automated integrators (aggregators, routing contracts) that treat a non-zero return from `viewSwapRsETHAmountAndFee` as a signal to proceed will repeatedly submit failing transactions under any of these three blocking conditions.

## Impact Explanation
**Low. Contract fails to deliver promised returns, but doesn't lose value.**

The user loses only gas; the deposit reverts and principal is not transferred. The broken invariant is that the protocol's own preview function advertises a deliverable amount that the protocol cannot fulfill at that moment. No funds are stolen or permanently frozen.

## Likelihood Explanation
**Medium.** All three blocking conditions are operationally realistic and independently sufficient to trigger the discrepancy: the `paused` flag is an active emergency mechanism controlled by `PAUSER_ROLE`; `dailyMintLimit` is an enforced cap that will be reached under normal high-volume conditions; and `isEthDepositEnabled` is a configurable admin flag. Any of these can be active while a user or integrator queries the view function. No attacker capability is required — any unprivileged caller can observe the inconsistency.

## Recommendation
Mirror the deposit guards inside `viewSwapRsETHAmountAndFee`:

```solidity
function viewSwapRsETHAmountAndFee(uint256 amount) public view returns (uint256 rsETHAmount, uint256 fee) {
    if (paused) revert ContractPaused();
    if (!isEthDepositEnabled) revert EthDepositDisabled();
    if (block.timestamp < startTimestamp) revert MintBeforeStartTimestamp();

    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;
    uint256 rsETHToETHrate = getRate();
    rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;

    uint256 currentDay = getCurrentDay();
    uint256 effectiveDailyMintAmount = (currentDay > lastMintDay) ? 0 : dailyMintAmount;
    if (effectiveDailyMintAmount + rsETHAmount > dailyMintLimit) revert DailyMintLimitExceeded();
}
```

Note: since `limitDailyMint` internally calls `viewSwapRsETHAmountAndFee` to compute `rsETHAmount`, adding the daily-limit revert inside the view function would cause `limitDailyMint` to revert before the modifier's own check — this is acceptable and consistent. Apply the same pattern to the token-variant overload and to `RSETHPoolV2.sol`, `RSETHPoolV3ExternalBridge.sol`, and `RSETHPoolV3WithNativeChainBridge.sol`.

## Proof of Concept
1. Deploy `RSETHPoolV3` with a valid oracle and `isEthDepositEnabled = true`.
2. Call `reinitialize(dailyMintLimit, startTimestamp)` to activate the daily cap.
3. Grant `PAUSER_ROLE` to an address and call `pause()`.
4. Call `viewSwapRsETHAmountAndFee(1 ether)` — observe it returns `(rsETHAmount > 0, fee > 0)`.
5. Call `deposit{value: 1 ether}("")` — observe it reverts with `ContractPaused`.
6. Repeat steps 3–5 with `unpause()` followed by filling the daily cap (deposit until `dailyMintAmount == dailyMintLimit`), then call `viewSwapRsETHAmountAndFee` again — still returns non-zero while `deposit` would revert with `DailyMintLimitExceeded`.
7. Repeat with `setIsEthDepositEnabled(false)` — `viewSwapRsETHAmountAndFee` still returns non-zero while `deposit` reverts with `EthDepositDisabled`.
