Audit Report

## Title
ETH Deposit Limit Check Excludes Deposit Amount, Allowing Bypass of `depositLimitByAsset` - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric check: for ETH it evaluates `totalAssetDeposits > limit` (omitting the incoming `amount`), while for LST assets it correctly evaluates `totalAssetDeposits + amount > limit`. Any unprivileged caller can invoke `depositETH` with an arbitrarily large value and push total ETH deposits well above the configured cap in a single transaction, because the guard only reverts when the cap has already been exceeded before the deposit arrives.

## Finding Description
The function at lines 676–682 of `contracts/LRTDepositPool.sol` branches on asset type:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));   // amount excluded
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
}
``` [1](#0-0) 

`depositETH` (lines 76–93) is `public payable`, guarded only by `nonReentrant` and `whenNotPaused`, and routes through `_beforeDeposit` which calls this function with `msg.value` as `amount`. [2](#0-1) 

The `amount` parameter is passed correctly into `_checkIfDepositAmountExceedesCurrentLimit` but is silently dropped in the ETH branch. The check therefore only blocks deposits when `totalAssetDeposits` already exceeds the limit before the new deposit, leaving the cap unenforceable for any deposit that arrives while the running total is still below the limit.

## Impact Explanation
`depositLimitByAsset` is the protocol's primary risk-management ceiling for each accepted asset. Bypassing it for ETH allows the pool to accumulate unbounded ETH exposure beyond the admin-configured cap. No depositor funds are lost or frozen, but the contract fails to deliver the deposit-cap guarantee it is designed to enforce. This maps exactly to the allowed Low impact: **"Contract fails to deliver promised returns, but doesn't lose value."**

## Likelihood Explanation
`depositETH` has no access control beyond `whenNotPaused`. Any externally-owned account can call it with an arbitrary `msg.value` in a single transaction. No special role, flash loan, front-running, or external dependency is required. The condition is exploitable whenever `getTotalAssetDeposits(ETH_TOKEN) ≤ depositLimitByAsset(ETH_TOKEN)`, which is the normal operating state of the protocol. **Likelihood: High.**

## Recommendation
Include the deposit amount in the ETH branch to match the LST branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset[ETH_TOKEN] = 1000 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `999 ether`.
3. Attacker calls `depositETH{value: 500 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit` evaluates `999e18 > 1000e18` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full 500 ETH; total ETH deposits become `1499 ether`.
6. The same call with an LST would evaluate `999e18 + 500e18 > 1000e18` → `true` → `MaximumDepositLimitReached` revert.

A Foundry unit test can confirm this by: (a) deploying the contract with a mock `lrtConfig` returning a `depositLimitByAsset` of `1000 ether` and a `getTotalAssetDeposits` stub returning `999 ether`; (b) calling `depositETH{value: 500 ether}`; (c) asserting the call succeeds and total deposits exceed the limit, then repeating with an LST asset to assert revert.
