### Title
V3 Shard Tracking Bypass: `check_if_descendant_of_tracked_shard_impl` Omits Self-Membership Check for Ancestor-less Shards — (`chain/epoch-manager/src/shard_tracker.rs`)

---

### Summary

`check_if_descendant_of_tracked_shard_impl` takes a fast path for `ShardLayoutV3` that checks only the ancestor set, never the shard itself. For any shard in a V3 layout that has no ancestors (i.e., was never produced by a split), `ancestor_uids` returns `Some(vec![])`, the ancestor `HashSet` is empty, `!ancestors.is_disjoint(&tracked_shards)` evaluates to `false`, and the function returns `Ok(false)` — even when the shard is explicitly listed in `TrackedShardsConfig::Shards`. The non-V3 fallback path correctly checks `tracked_shards.contains(&shard_uid)` before walking ancestry; the V3 branch does not.

---

### Finding Description

**Root cause — `validate_and_derive_shard_ancestor_map` inserts every shard, including those with no ancestors:** [1](#0-0) 

The type comment says "IDs of shards which have no ancestors are **not** present in the mapping," but the implementation contradicts this: [2](#0-1) 

Every shard ID is unconditionally inserted. For a shard with no ancestors the loop body never executes, so `ancestors = vec![]` is inserted. `shards_ancestor_map.get(&shard_id)` therefore returns `Some(&vec![])`.

**`ShardLayoutV3::ancestor_uids` propagates the empty vec as a non-None result:** [3](#0-2) 

`Some(&vec![])` → `.map(...)` → `Some(vec![])` → `.unwrap_or_default()` → `vec![]`. The outer `ShardLayout::ancestor_uids` wraps this in `Some(vec![])`. [4](#0-3) 

**`check_if_descendant_of_tracked_shard_impl` takes the V3 branch and returns `false` without checking self-membership:** [5](#0-4) 

`ancestors` is an empty `HashSet`. `!ancestors.is_disjoint(&tracked_shards)` is `!true = false`. The function returns `Ok(false)`.

The non-V3 path that correctly handles this case is never reached: [6](#0-5) 

---

### Impact Explanation

The broken return value propagates directly to `rpc_tracks_shard_at_epoch`: [7](#0-6) 

A non-validator RPC node configured with `TrackedShardsConfig::Shards` containing a V3-layout shard that has no ancestors will have `rpc_tracks_shard_at_epoch` return `false` for that shard. The RPC coordinator then misroutes any `call_function` (or other query) targeting an account in that shard away from the local node, causing spurious "shard not tracked" errors or incorrect forwarding for every such request.

---

### Likelihood Explanation

The condition is reachable whenever:
1. The network has activated `ShardLayoutV3` (dynamic resharding).
2. At least one shard in the V3 layout was never produced by a split (e.g., shards carried over from a V2 layout when only one of several shards was split).
3. A node is configured with `TrackedShardsConfig::Shards` listing one of those unsplit shards.

This is the normal migration path described in the codebase: `ShardLayoutV3::derive_with_layout_history` is called when transitioning from V2, and the resulting V3 layout will contain both newly-split children and unchanged original shards. Any node tracking an unchanged original shard by UID hits the bug immediately.

An unprivileged RPC client triggers the misrouting simply by issuing a `call_function` query for an account in the affected shard.

---

### Recommendation

In the V3 branch of `check_if_descendant_of_tracked_shard_impl`, check whether the shard itself is in `tracked_shards` before (or alongside) checking ancestors:

```rust
if let Some(ancestors) = shard_layout.ancestor_uids(shard_id) {
    let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
    if tracked_shards.contains(&shard_uid) {
        return Ok(true);
    }
    let ancestors = HashSet::from_iter(ancestors);
    return Ok(!ancestors.is_disjoint(&tracked_shards));
}
```

Alternatively, fix `validate_and_derive_shard_ancestor_map` to match its own comment (skip inserting shards with empty ancestor lists), and update `ShardLayoutV3::ancestor_uids` to return `None` (not `vec![]`) for shards absent from the map — but this requires also fixing the `ShardLayout::ancestor_uids` wrapper to distinguish "no ancestors" from "not a V3 layout."

---

### Proof of Concept

Construct a V3 layout with at least one unsplit shard (e.g., start from a two-shard V2 layout, split only one):

```rust
// V2: shards [0, 1]
// V3: split shard 0 → [2, 3]; shard 1 has no ancestors
let v2 = ShardLayout::v2(vec!["boundary".parse().unwrap()], vec![ShardId::new(0), ShardId::new(1)], None);
let v3 = ShardLayout::derive_shard_layout(&v2, "mid".parse().unwrap());
// shard 1 is still present in v3 with no ancestors

let shard_1_uid = ShardUId::from_shard_id_and_layout(ShardId::new(1), &v3);
// ancestor_uids returns Some(vec![]) for shard 1
assert_eq!(v3.ancestor_uids(ShardId::new(1)), Some(vec![]));

// Build a tracker explicitly tracking shard 1
let tracker = ShardTracker::new(
    TrackedShardsConfig::Shards(vec![shard_1_uid]),
    epoch_manager.clone(),
    MutableConfigValue::new(None, "validator_signer"),
);

// BUG: returns false even though shard 1 is explicitly tracked
let result = tracker.rpc_tracks_shard_at_epoch(ShardId::new(1), &v3_epoch_id).unwrap();
assert!(result, "shard 1 should be tracked but rpc_tracks_shard_at_epoch returns false");
```
