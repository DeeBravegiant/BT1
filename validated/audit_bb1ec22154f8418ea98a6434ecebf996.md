Audit Report

## Title
ETH Deposit Limit Check Omits Incoming Amount, Allowing Cap Bypass - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` uses an asymmetric comparison for ETH deposits: the ETH branch checks only `totalAssetDeposits > depositLimit` while the ERC20 branch correctly checks `totalAssetDeposits + amount > depositLimit`. Any unprivileged caller can invoke `depositETH` with an arbitrarily large value and push total ETH deposits above the configured cap, causing the protocol to mint rsETH against ETH that exceeds the intended ceiling.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682, `_checkIfDepositAmountExceedesCurrentLimit` branches on asset type:

```solidity
// L678-681
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // missing + amount
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // correct
```

The ETH branch returns `true` (reject) only when the **pre-deposit** aggregate already exceeds the limit. It never factors in the incoming `amount`. The function is the sole guard called from `_beforeDeposit` (L661), which is itself the only check in `depositETH` (L87). No other cap enforcement exists in the deposit path. Because the check evaluates the pre-deposit state, any deposit where `totalAssetDeposits ≤ depositLimit` passes regardless of `amount`, allowing the post-deposit total to exceed the limit by the full deposit size.

## Impact Explanation
The ETH deposit limit (`depositLimitByAsset`) is the protocol's primary risk-management cap on ETH exposure. Bypassing it causes the protocol to mint rsETH against ETH holdings that exceed the configured ceiling, violating the invariant `totalETHDeposits ≤ depositLimit`. No direct fund loss occurs, but the protocol fails to deliver its promised deposit cap. This matches the allowed impact: **Low — Contract fails to deliver promised returns, but doesn't lose value**.

## Likelihood Explanation
`depositETH` is public and payable, callable by any user with ETH. No special role, oracle manipulation, or front-running is required. The vulnerable condition (`totalAssetDeposits ≤ depositLimit`) is the normal operating state of the protocol. Any depositor sending nonzero ETH while the cap has not already been exceeded will silently bypass the limit. The exploit is repeatable across multiple transactions.

## Recommendation
Add `amount` to the ETH branch to match the ERC20 branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

## Proof of Concept
1. Admin sets `depositLimitByAsset(ETH_TOKEN) = 100 ether`.
2. `getTotalAssetDeposits(ETH_TOKEN)` returns `99 ether` (normal operating state).
3. Attacker calls `depositETH{value: 50 ether}(0, "")`.
4. `_checkIfDepositAmountExceedesCurrentLimit(ETH, 50 ether)` evaluates `99 ether > 100 ether` → `false` → no revert.
5. `_mintRsETH` mints rsETH for the full 50 ETH.
6. Post-deposit: `getTotalAssetDeposits(ETH_TOKEN)` = 149 ETH, 49 ETH above the configured cap.

**Foundry test plan:**
```solidity
function test_ethDepositLimitBypass() public {
    vm.prank(admin);
    lrtConfig.setDepositLimitByAsset(LRTConstants.ETH_TOKEN, 100 ether);
    // seed 99 ether via prior deposits
    depositor.depositETH{value: 99 ether}(0, "");
    // now bypass: deposit 50 ether when limit is 100 ether
    attacker.depositETH{value: 50 ether}(0, "");
    uint256 total = lrtDepositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN);
    assertGt(total, 100 ether); // proves cap was exceeded
}
