Audit Report

## Title
Rewards Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary
In `KernelDepositPool.sol`, the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()` even when `totalKernelStaked` is zero. If all stakers call `initiateWithdrawal` during an active reward window, the rewards accrued during the zero-staked interval are permanently skipped and locked in the contract. The contract's own NatSpec acknowledges this behavior and relies solely on an off-chain operational mitigation that is not enforced by any on-chain guard.

## Finding Description
The `updateReward` modifier at L232–242 always executes:

```solidity
rewardPerTokenStored = rewardPerToken();
updatedAt = lastTimeRewardApplicable();
```

When `totalKernelStaked == 0`, `rewardPerToken()` (L408–414) returns the frozen `rewardPerTokenStored` without accumulating any new rewards. However, `updatedAt` is still advanced to the current applicable time. This means the time interval during which no tokens were staked is silently consumed without distributing any rewards.

Exact exploit sequence:
1. Alice is the sole staker. She calls `initiateWithdrawal(totalStake)`. The `updateReward` modifier fires first (while `totalKernelStaked > 0`), correctly checkpointing `rewardPerTokenStored` and setting `updatedAt = T1`. Then `totalKernelStaked -= _amount` drops to 0.
2. Time advances from T1 to T2. `rewardRate` is still active (`finishAt > T2`), but `rewardPerToken()` returns the frozen value because `totalKernelStaked == 0`. No rewards accumulate.
3. Bob calls `stake`. `updateReward` fires: `rewardPerToken()` still returns the frozen `rewardPerTokenStored` (since `totalKernelStaked` is still 0 at modifier entry). `updatedAt` is advanced to T2. Then `totalKernelStaked += _amount`.
4. The `rewardRate × (T2 − T1)` tokens are permanently skipped. Bob's `userRewardPerTokenPaid` is set to the frozen `rewardPerTokenStored`, so he only earns from T2 onward.

The `notifyRewardAmount` guard at L570 (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new reward period with zero stakers; it does not prevent stakers from withdrawing after a period has started. There is no recovery function, no rollover mechanism, and no way to reclaim the locked reward tokens.

The contract's own NatSpec at L18–22 explicitly acknowledges this:
> *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract."*

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens (`rewardsToken`) are permanently locked in `KernelDepositPool` with no recovery path. The locked amount equals `rewardRate × (duration of zero-staked interval)`. For a 30-day reward window, this can represent a substantial fraction of the total reward allocation. This matches the allowed impact class "Medium. Permanent freezing of unclaimed yield."

## Likelihood Explanation
No privileged access is required. Any combination of stakers calling the public `initiateWithdrawal` function is sufficient to drain `totalKernelStaked` to zero. The `withdrawalDelay` is admin-configurable and can be set to as little as 1 second (the only lower bound is `> 0` per L599). A coordinated or organic exit during a reward period is a realistic scenario, particularly if stakers lose confidence or if the delay is short. The scenario is repeatable across reward periods.

## Recommendation
Advance `updatedAt` only when `totalKernelStaked > 0`, so the zero-staked interval is not silently consumed:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    if (totalKernelStaked > 0) {
        updatedAt = lastTimeRewardApplicable();
    }
    if (_account != address(0)) {
        rewards[_account] = earned(_account);
        userRewardPerTokenPaid[_account] = rewardPerTokenStored;
    }
    _;
}
```

This ensures that when the next staker arrives, `rewardPerToken()` correctly accounts for the full elapsed time (including the zero-staked gap) once `totalKernelStaked > 0` again, rather than silently skipping it.

## Proof of Concept

```
1. Admin calls notifyRewardAmount(1_000e18) with totalKernelStaked = 100e18.
   → rewardRate = 1_000e18 / duration; finishAt = block.timestamp + duration.

2. Alice (sole staker, balance = 100e18) calls initiateWithdrawal(100e18).
   → updateReward fires: rewardPerToken() computed correctly (totalKernelStaked > 0),
     updatedAt = T1, totalKernelStaked → 0.

3. Warp forward 15 days (half the reward period).
   → rewardRate × 15 days worth of rewards sit in the contract, unaccumulated.

4. Bob calls stake(1e18).
   → updateReward fires: rewardPerToken() returns frozen rewardPerTokenStored
     (totalKernelStaked still 0 at modifier entry), updatedAt advanced to T2.
     totalKernelStaked → 1e18.
   → Bob's userRewardPerTokenPaid = rewardPerTokenStored.

5. Bob earns only from T2 onward. Alice earned only up to T1.
   → rewardRate × 15 days tokens are permanently locked with no recovery mechanism.
```
