### Title
Unprotected `init` Function Allows Front-Running to Hijack MPC Participant Set - (File: `crates/contract/src/lib.rs`)

### Summary
The `MpcContract::init` function lacks the `#[private]` attribute, meaning any external NEAR account can call it before the legitimate deployer does. An attacker who monitors the mempool and front-runs the initialization can install themselves as the sole participant with a threshold of 1, gaining full unilateral control over the MPC network's signing capability.

### Finding Description
In `crates/contract/src/lib.rs`, there are three initialization methods:

- `init` (line 1926): decorated with `#[init]` only — **no `#[private]`**
- `init_running` (line 1979): decorated with both `#[init]` and `#[private]`
- `migrate` (line 2063): decorated with both `#[private]` and `#[init(ignore_state)]`

In NEAR's SDK, `#[init]` only prevents re-initialization (panics if state already exists). It does **not** restrict the caller. The `#[private]` attribute is what enforces `predecessor_account_id == current_account_id`, restricting calls to the contract itself. Without `#[private]`, `init` is callable by any external account.

The deployment workflow documented in the codebase explicitly separates deployment from initialization into two separate transactions:

```bash
# Step 1: deploy without init
near contract deploy "$MPC_CONTRACT_ACCOUNT" use-file "$MPC_CONTRACT_PATH" without-init-call ...

# Step 2: init (separate transaction — attackable window)
near contract call-function as-transaction mpc-contract.test.near init file-args ...
```

This creates a window between deployment and initialization during which any account can call `init` with attacker-controlled parameters.

Additionally, `init` calls `TeeState::with_mocked_participant_attestations` for the supplied participants, meaning attacker-controlled participants are immediately granted valid (mocked) attestations, bypassing the TEE attestation requirement entirely.

```rust
// crates/contract/src/lib.rs:1924-1945
#[handle_result]
#[init]  // ← no #[private] here
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
    ...
    let tee_state = TeeState::with_mocked_participant_attestations(initial_participants);
    ...
}
```

Compare with `init_running`, which is correctly protected:

```rust
// crates/contract/src/lib.rs:1976-1979
#[private]
#[init]
#[handle_result]
pub fn init_running(...) -> Result<Self, Error> {
```

### Impact Explanation
An attacker who successfully front-runs `init` can:
1. Set themselves as the sole participant with threshold 1.
2. Receive mocked TEE attestations automatically via `with_mocked_participant_attestations`.
3. Vote to add signing domains (`vote_add_domains`).
4. Trigger key generation and become the sole keyholder.
5. Issue threshold signatures unilaterally for any payload.

The legitimate deployer's subsequent `init` call will fail because `#[init]` prevents re-initialization once state exists. The network is permanently hijacked from the first block.

This constitutes **unauthorized threshold signature issuance** and **unauthorized access to MPC key shares/signing capability** — a Critical impact under the allowed scope.

### Likelihood Explanation
The deployment scripts (`deploy-tee-cluster.sh`, `launch-localnet.sh`, testnet setup guides) all perform deployment and initialization as separate transactions. NEAR transactions are publicly visible in the mempool. Any attacker watching for a new deployment of the MPC contract WASM can immediately submit a competing `init` call. The attack requires no special privileges, no collusion, and no cryptographic capability — only the ability to submit a NEAR transaction.

### Recommendation
Add `#[private]` to `init`, consistent with how `init_running` and `migrate` are protected:

```rust
#[private]
#[handle_result]
#[init]
pub fn init(
    parameters: dtos::ThresholdParameters,
    init_config: Option<dtos::InitConfig>,
) -> Result<Self, Error> {
```

This restricts `init` so it can only be called by the contract account itself (i.e., as part of a deploy-with-init atomic call), eliminating the front-running window.

### Proof of Concept

1. Attacker monitors NEAR for a deployment of the MPC contract WASM.
2. Immediately after the deploy transaction lands (before the operator's `init` call), attacker submits:
   ```json
   {
     "parameters": {
       "threshold": 1,
       "participants": {
         "next_id": 1,
         "participants": [["attacker.near", 0, {"tls_public_key": "ed25519:<attacker_key>", "url": "http://attacker.com"}]]
       }
     }
   }
   ```
   to `MpcContract::init`.
3. Contract initializes with attacker as sole participant; `TeeState::with_mocked_participant_attestations` grants attacker a valid mocked attestation immediately.
4. Operator's `init` call fails: NEAR `#[init]` panics because state already exists.
5. Attacker calls `vote_add_domains` (threshold=1, only one vote needed) to add a signing domain.
6. Attacker's node completes DKG alone and holds the full key share.
7. Attacker calls `sign` and `respond` to issue arbitrary threshold signatures — no legitimate participant is involved.

        let mut tee_state = Self::default();

        for (account_id, _, participant_info) in participants.participants() {
            let tls_public_key = participant_info.tls_public_key.clone();
            // TODO(#1087): replace account_public_key with a real account public
            // key passed in by the caller. `Participants` does not currently
            // carry the operator's account public key, so a mocked entry
            // cannot record the real one and we use the TLS key as a unique
            // per-participant placeholder. The mock keeps the
            // participant from being kicked out of an empty `TeeState` until
            // a real `submit_participant_info` call replaces it (keyed by
            // TLS), but any caller-facing check that compares
            // `signer_account_pk` against the stored key will fail until
            // then. #1087 tracks threading real attestations through
            // initialization so this sentinel can go away.
            let node_id = NodeId {
                account_id: account_id.clone(),
                tls_public_key: tls_public_key.clone(),
                // Use tls_public_key as account_public_key instead of hardcoded
                // Ed25519PublicKey::from([0u8; 32]) so that same account public
                // key isn't associated with different tls keys.
                // This is not a fix for above issue: #1087, which should be
                // addressed outside this PR.
                account_public_key: tls_public_key.clone(),

                tls_public_key,
                NodeAttestation {
                    node_id,

                        attestation::MockAttestation::Valid,
                    ),
                },
            );
        }
