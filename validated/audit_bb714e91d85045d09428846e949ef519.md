### Title
Singleton Records Permanently Deleted Instead of Restored During Reorg Rollback — (`chia/wallet/wallet_singleton_store.py`)

### Summary

`WalletSingletonStore.rollback()` executes a `DELETE` on singleton records whose `removed_height > height`, when the correct behavior is to `UPDATE` those records back to `removed_height = 0`. After a chain reorg to height H, any singleton that was spent at a height > H is permanently erased from the wallet's store rather than being restored to its pre-spend (active) state.

### Finding Description

The `rollback` method at line 223–234 runs:

```sql
DELETE FROM singletons WHERE removed_height>? AND wallet_id=?
``` [1](#0-0) 

The `removed_height` column is set to a non-zero block height when a singleton is spent (via `delete_singleton_by_coin_id` or `delete_singleton_by_singleton_id`). A value of `0` means the singleton is currently active. [2](#0-1) 

When a reorg rolls back to height H, any singleton that was spent at height > H had its `removed_height` set to that spend height. The rollback should **restore** those records to `removed_height = 0` (active), because the spend that removed them is being undone. Instead, the current code **deletes** them entirely.

The docstring even acknowledges the mismatch — it refers to an `entry_height` column that does not exist in the schema: [3](#0-2) 

The table schema has no `entry_height` or `created_height` column: [4](#0-3) 

So the rollback is operating on the wrong semantic: it treats `removed_height` as if it were a creation timestamp, but it is actually a deletion timestamp.

### Impact Explanation

After a reorg to height H, every singleton (NFT, DID, pool wallet coin, VC, Data Layer singleton) that was spent in the reorged blocks (heights > H) is permanently removed from the wallet's singleton store. The wallet has no remaining record of those singletons and cannot spend, transfer, or otherwise manage them. This constitutes **corruption of wallet sync state and singleton lineage tracking with direct security impact** — the user loses wallet-level control over assets they legitimately own on-chain.

This maps to: **High — Corruption of coin records, lineage, puzzle ownership, wallet sync state, or pool membership state with direct security impact.**

### Likelihood Explanation

Small reorgs (1–3 blocks) occur naturally on the Chia network without any attacker involvement. Any natural reorg that includes a singleton spend in the reorged blocks will silently trigger this data loss. A targeted attacker with moderate hash power could deliberately induce a reorg to cause this. The code path is exercised on every reorg event for any wallet that holds singletons.

### Recommendation

Replace the `DELETE` with an `UPDATE` that resets `removed_height` to `0` for records whose spend was in the reorged range:

```sql
UPDATE singletons SET removed_height=0 WHERE removed_height>? AND wallet_id=?
```

Additionally, a separate `DELETE` is needed for singleton records that were **created** (not just spent) in the reorged blocks. Since there is no `created_height` column, one should be added to the schema to support correct rollback semantics for both cases.

### Proof of Concept

    async def delete_singleton_by_coin_id(self, coin_id: bytes32, height: uint32) -> bool:
        """Tries to mark a given singleton as deleted at specific height

        This is due to how re-org works
        Returns `True` if singleton was found and marked deleted or `False` if not."""
        async with self.db_wrapper.writer_maybe_transaction() as conn:
            cursor = await conn.execute(
                "UPDATE singletons SET removed_height=? WHERE coin_id=?", (int(height), coin_id.hex())
            )
            if cursor.rowcount > 0:
                log.info("Deleted singleton with coin id: %s", coin_id.hex())
                return True
            log.warning("Couldn't find singleton with coin id to delete: %s", coin_id.hex())
            return False

        """
