Audit Report

## Title
L2 Price Providers Accept Stale Prices After Sequencer Restart Without Sequencer Uptime Check — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are deployed on L2 (Arbitrum) and rely solely on a timestamp-delta staleness check (`_isStale`) to validate oracle prices. Neither contract checks whether the Arbitrum sequencer is live. When the sequencer restarts after an outage shorter than `MAX_TIME_DELTA`, the frozen pre-outage `refTime` passes the staleness check, and the pool executes swaps at a stale price — causing direct LP loss.

## Finding Description

Both contracts implement `_isStale` identically: [1](#0-0) 

This function only checks whether `(nowTs - refTime) > MAX_TIME_DELTA`. It does not check whether the L2 sequencer was down during that interval.

`_getBidAndAskPrice()` in `PriceProviderL2` reads the oracle price and applies only this staleness check before returning bid/ask to the pool: [2](#0-1) 

`ProtectedPriceProviderL2` delegates to `_computeBidAsk`, which applies the same check: [3](#0-2) 

The oracle backends (`ChainlinkOracle`, `PythOracle`) are push-based: prices are stored only when `updateReport()` is called. When the Arbitrum sequencer is down, no transactions land on L2, so no fresh price can be pushed. The stored `refTime` is frozen at the last pre-outage push. When the sequencer restarts:

- `refTime` = last push before outage (e.g., `T`)
- `block.timestamp` = `T + outage_duration`
- If `outage_duration < MAX_TIME_DELTA`, `_isStale` returns `false`
- The pool receives the pre-outage price as if it were current

`FUTURE_TOLERANCE` only addresses L2 sequencer clock skew (oracle `refTime` slightly ahead of `block.timestamp`), not sequencer downtime: [4](#0-3) 

`MAX_TIME_DELTA` is bounded between 0 and 7 days: [5](#0-4) 

Even a conservative 30-minute `MAX_TIME_DELTA` creates a 30-minute exploitation window after every sequencer restart. There is no sequencer uptime feed check anywhere in either contract.

## Impact Explanation

This is a bad-price execution impact: a stale bid/ask quote reaches a pool swap. An attacker who executes a swap immediately after sequencer restart (before any fresh price is pushed) receives tokens at the pre-outage price. If the real market price moved during the outage (e.g., 5–10% during volatile periods that often coincide with sequencer stress), the attacker receives excess tokens and LPs are underpaid — a direct loss of LP principal satisfying the allowed impact gate.

## Likelihood Explanation

Arbitrum sequencer outages have occurred historically. No special permissions are required: any address can call `swap()` on the pool. The exploitation window opens automatically on every sequencer restart and closes only when a fresh price is pushed by off-chain bots, which may themselves be delayed after a restart. A prepared attacker monitoring the sequencer can exploit this reliably and repeatably.

## Recommendation

Add a Chainlink L2 sequencer uptime check in both `PriceProviderL2` and `ProtectedPriceProviderL2`. Store the sequencer uptime feed address as an immutable set at construction. In `_getBidAndAskPrice()` (or `_computeBidAsk()`), before using the price, verify:

```solidity
(, int256 answer, uint256 startedAt, ,) = sequencerUptimeFeed.latestRoundData();
require(answer == 0, SequencerDown());
require(block.timestamp - startedAt > GRACE_PERIOD, SequencerJustRestarted());
```

This is the standard Chainlink recommendation for L2 deployments and mirrors fixes applied in comparable protocols.

## Proof of Concept

1. Deploy with `MAX_TIME_DELTA = 30 minutes`. Last price push at `T = block.timestamp - 5 minutes`.
2. Arbitrum sequencer goes down at `T + 1 minute`; real ETH price drops 8% during outage.
3. Sequencer restarts at `T + 20 minutes`; stored `refTime = T + 1 minute` (last push before outage).
4. Attacker calls `MetricOmmPool.swap(...)` before any fresh price is pushed.
5. `PriceProviderL2._getBidAndAskPrice()` reads `refTime = T + 1 minute`; `nowTs - refTime = 19 minutes < 30 minutes` → `_isStale` returns `false`.
6. Pool executes swap at pre-outage (8% inflated) ask price; attacker receives excess tokens; LPs are underpaid.

Foundry fork test: fork Arbitrum mainnet at a block just after a historical sequencer restart, set `MAX_TIME_DELTA` to a value exceeding the outage duration, call `swap()` with a prepared attacker address, and assert that the returned bid/ask matches the pre-outage price rather than reverting.

```text
    function _isStale(
        uint256 refTime,
        uint256 nowTs,
        uint256 maxDelta,
        uint256 futureTol
    ) internal pure returns (bool) {
        if (refTime == 0) return true;

        if (refTime > nowTs) {
            // refTime in the future: tolerate only within futureTol
            return (refTime - nowTs) > futureTol;
        }

        // refTime in the past or equal: check age
        return (nowTs - refTime) > maxDelta;
    }
```
