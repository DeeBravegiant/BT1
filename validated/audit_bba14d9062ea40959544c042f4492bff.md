Audit Report

## Title
Missing Slippage Protection in `removeLiquidity` with No Periphery Wrapper Path — (File: metric-core/contracts/MetricOmmPool.sol, metric-core/contracts/libraries/LiquidityLib.sol)

## Summary
`MetricOmmPool.removeLiquidity` accepts no `minAmount0Out`/`minAmount1Out` parameters, and the `msg.sender == owner` constraint prevents any periphery wrapper from adding them. A swap executed before the LP's removal settles mutates the bin's `token0BalanceScaled`/`token1BalanceScaled`, changing the token composition the LP receives. The LP has no on-chain mechanism to revert if the returned amounts fall below their acceptable threshold.

## Finding Description
`MetricOmmPool.removeLiquidity` enforces `msg.sender != owner` revert at line 206, meaning the LP must call the pool directly — no intermediary contract can wrap the call to add slippage checks without triggering `NotPositionOwner`. [1](#0-0) 

`LiquidityLib.removeLiquidity` reads live bin state at execution time and transfers immediately: [2](#0-1) [3](#0-2) 

Swaps directly mutate `token0BalanceScaled`/`token1BalanceScaled` in storage: [4](#0-3) 

The periphery provides `MetricOmmPoolLiquidityAdder` with `maxAmountToken0`/`maxAmountToken1` caps enforced in the callback, and `addLiquidityWeighted` adds cursor-bound guards. No equivalent `MetricOmmPoolLiquidityRemover` exists: [5](#0-4) 

The extension system's `afterRemoveLiquidity` hook receives `amount0Removed`/`amount1Removed` and could theoretically revert, but this is a pool-level configuration set at deployment by the pool admin — it is not a per-LP, per-transaction slippage guard that an individual LP can set: [6](#0-5) 

The `msg.sender == owner` constraint is the critical blocker: unlike `addLiquidity` (where `msg.sender` can be a periphery contract paying on behalf of `owner`), `removeLiquidity` requires the LP to be the direct caller, closing off any wrapper-based protection path. [7](#0-6) [8](#0-7) 

## Impact Explanation
An LP holding shares in the current bin expects a proportional mix of token0 and token1 based on `curPosInBin`. A swap that traverses the current bin before the LP's removal settles shifts `token0BalanceScaled` down and `token1BalanceScaled` up (or vice versa). The LP receives a composition they did not consent to — potentially receiving all of one token when they expected a mix — with no on-chain revert path. If the LP's intended use requires a specific token (e.g., to repay a loan), they must execute an additional swap at oracle spread cost, constituting a direct, measurable reduction in the value of LP principal recovered. This meets the Medium threshold for direct loss of user principal.

## Likelihood Explanation
Any swap caller can front-run a pending `removeLiquidity` transaction visible in the mempool. No special privilege is required. On chains with public mempools (Ethereum mainnet, most L2s without private RPCs), this is straightforward to execute. The `msg.sender == owner` constraint means the LP cannot protect themselves via any existing or future periphery wrapper without a core interface change. Likelihood is **medium** — it requires mempool visibility and a willing attacker, but the mechanism is standard MEV.

## Recommendation
1. Add `minAmount0Out` and `minAmount1Out` parameters to `MetricOmmPool.removeLiquidity` and enforce them after `LiquidityLib.removeLiquidity` returns:

```solidity
function removeLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 minAmount0Out,
    uint256 minAmount1Out,
    bytes calldata extensionData
) external ... {
    ...
    (amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(...);
    if (amount0Removed < minAmount0Out || amount1Removed < minAmount1Out)
        revert SlippageExceeded();
    ...
}
```

2. Alternatively, relax the `msg.sender == owner` constraint to allow an approved operator pattern (analogous to `addLiquidity`), enabling a periphery `MetricOmmPoolLiquidityRemover` to wrap `removeLiquidity` with minimum-output checks and a deadline.

## Proof of Concept
**Setup:** Pool with bins [-1, 0, 1]. Bin 0 is the current bin with `curPosInBin = type(uint104).max / 2` (50% through), holding both token0 and token1. LP holds 10,000 shares in bin 0.

**Attack sequence:**
1. LP broadcasts `removeLiquidity(owner, salt, [{binIdx: 0, shares: 10000}], "")`.
2. Attacker front-runs with `swap(zeroForOne=false, amountSpecified=large, ...)` — buying all available token0 from bin 0, draining `token0BalanceScaled` to near zero while filling `token1BalanceScaled`.
3. LP's `removeLiquidity` executes. `LiquidityLib` reads the now-modified bin state:
   - `amount0Scaled = binState.token0BalanceScaled * 10000 / binTotalShares ≈ 0`
   - `amount1Scaled = binState.token1BalanceScaled * 10000 / binTotalShares ≈ full token1 share`
4. LP receives ~0 token0 and all token1. If the LP needed token0, they must now swap token1 → token0 at oracle spread cost.
5. The LP had no parameter to revert this outcome. With a `minAmount0Out` guard, step 3 would have reverted, protecting the LP.

**Foundry test plan:** Deploy pool with 3 bins, add liquidity to bin 0 at mid-cursor, record initial `token0BalanceScaled`/`token1BalanceScaled`, execute a large `zeroForOne=false` swap to drain token0, then call `removeLiquidity` and assert `amount0Removed ≈ 0` while `amount1Removed ≈ full share` — demonstrating the composition shift with no revert path.
