### Title
Single Attested Participant Can Deliver Fabricated CKD Response for `AppPublicKey` Requests — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract performs **no cryptographic validation** of the `CKDResponse` when the request uses the `AppPublicKey` variant of `CKDAppPublicKey`. Any single attested participant — a role analogous to the "delegated contractor" in M-17 — can call `respond_ckd` with a completely fabricated key-derivation response and the contract will accept it, drain the pending yield queue, and deliver the forged output to the waiting user. This bypasses the threshold-signature requirement that is the entire security foundation of the MPC network.

---

### Finding Description

`respond_ckd` is the node-facing endpoint that delivers a Confidential Key Derivation (CKD) result back to a user who called `request_app_private_key`. The function is gated only by `assert_caller_is_attested_participant_and_protocol_active()` — a single-participant check — and then branches on the request's `app_public_key` variant: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO validation
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
``` [2](#0-1) 

For the `AppPublicKey` arm the branch body is empty — the response is accepted unconditionally. Immediately after, `resolve_yields_for` drains the entire pending-yield queue for that request key and resumes every waiting caller with the attacker-supplied bytes: [3](#0-2) 

Compare this with `respond` (ECDSA/EdDSA), which always verifies the signature cryptographically before accepting it: [4](#0-3) 

The `AppPublicKey` variant is a legitimate, documented production path. The contract README explicitly notes it and the `request_app_private_key` endpoint accepts both variants: [5](#0-4) 

The `resolve_yields_for` helper removes the request from the map on first call, so once the attacker's fabricated response is submitted, the honest MPC network's subsequent `respond_ckd` call returns `RequestNotFound` and is silently discarded: [6](#0-5) 

---

### Impact Explanation

**Critical — Unauthorized confidential key derivation output without the required participant authorization.**

A malicious single participant can:

1. Monitor the NEAR chain for any pending `CKDRequest` whose `app_public_key` field uses the `AppPublicKey` variant (visible in contract state via `get_pending_ckd_request`).
2. Construct an arbitrary `CKDResponse` (`big_y`, `big_c`) — for example, one whose corresponding private key the attacker already knows.
3. Call `respond_ckd` with that fabricated response before the honest MPC network finishes its threshold computation.
4. The contract accepts the response, drains the yield queue, and delivers the forged key material to the user.

The user receives a derived key that the attacker controls or that is cryptographically invalid, completely defeating the purpose of the threshold MPC network. This matches the allowed Critical impact: *"Unauthorized … confidential key derivation output without the required participant authorization."*

---

### Likelihood Explanation

**Low.** Becoming an attested participant requires a threshold vote from the existing participant set, so the attacker must first be admitted to the network — a significant barrier analogous to the "delegated contractor" trust assumption in M-17. Once admitted, however, the attack requires only a single on-chain transaction and no further collusion. The M-17 judge's reasoning applies directly: the trust model assumes participants are honest, but the code provides no enforcement mechanism for CKD responses under `AppPublicKey`, making a rogue insider's attack trivially executable.

---

### Recommendation

Apply the same cryptographic output-verification discipline to `AppPublicKey` that already exists for `AppPublicKeyPV`. Concretely:

1. **Extend `ckd_output_check`** (or introduce a parallel check) that validates the `CKDResponse` against the BLS12-381 domain public key even when no app-side public key is provided for verification — e.g., verify that `big_y` is a valid point on the curve and that `big_c` is consistent with the MPC network's root public key.
2. **Alternatively**, require that all `AppPublicKey` requests be upgraded to `AppPublicKeyPV` so the existing `ckd_output_check` path is always exercised.
3. As a defense-in-depth measure, consider requiring a threshold-quorum of `respond_ckd` calls (similar to how signing requires threshold participation off-chain) before the contract resolves the yield, rather than accepting the first response from any single participant.

---

### Proof of Concept

1. Alice calls `request_app_private_key` with `app_public_key: AppPublicKey(alice_bls_pk)` and attaches 1 yoctoNEAR deposit. The contract enqueues a yield and stores the `CKDRequest` in `pending_ckd_requests`.

2. Mallory, a malicious but legitimately attested participant, reads the pending request from chain state.

3. Mallory constructs `CKDResponse { big_y: mallory_controlled_key, big_c: garbage }` and calls:
   ```
   respond_ckd(request=<alice's CKDRequest>, response=<fabricated>)
   ```

4. The contract executes the `AppPublicKey` branch — empty body, no check — then calls `resolve_yields_for`, which resumes Alice's yield with Mallory's bytes and removes the request from the map.

5. Alice's transaction resolves with Mallory's fabricated `CKDResponse`. The honest MPC network's subsequent `respond_ckd` returns `RequestNotFound`.

6. Alice believes she holds a valid derived key; in reality Mallory controls it. [1](#0-0) [7](#0-6)

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

/// Resume every yield queued for `request` with `response_bytes`, draining the
/// fan-out map in one pass. Returns `Err(RequestNotFound)` if the map held no entry.
///
/// Resuming a yield that has already timed out is a no-op at the SDK level.
pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
