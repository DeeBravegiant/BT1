Audit Report

## Title
Fee Calculation Truncates to Zero for Sub-Threshold Deposits Due to Integer Division - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
The `viewSwapRsETHAmountAndFee` function in `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, and `RSETHPoolV3WithNativeChainBridge.sol` computes protocol fees as `fee = amount * feeBps / 10_000`. When `amount * feeBps < 10_000`, Solidity integer division truncates `fee` to zero. The deposit proceeds in full with no fee deducted, and `feeEarnedInETH` / `feeEarnedInToken` is incremented by zero. The protocol fails to collect the fee revenue it is designed to earn on these deposits.

## Finding Description
All three pool variants share the identical fee computation pattern, confirmed at:
- `RSETHPoolV3.sol` L300: `fee = amount * feeBps / 10_000;` (ETH path)
- `RSETHPoolV3.sol` L324: `fee = amount * feeBps / 10_000;` (token path)
- `RSETHPoolV3ExternalBridge.sol` L419, L442: same pattern
- `RSETHPoolV3WithNativeChainBridge.sol` L336, L360: same pattern

The deposit functions at `RSETHPoolV3.sol` L260 and L288 unconditionally execute `feeEarnedInETH += fee` and `feeEarnedInToken[token] += fee` after the view function returns. The only existing guard is `if (amount == 0) revert InvalidAmount()` at L256 and L282, which does not prevent amounts in the range `1 <= amount < 10_000 / feeBps`. There is no check for `fee == 0` when `feeBps > 0`. The user receives rsETH computed against the full `amount` (since `amountAfterFee = amount - 0 = amount`), and the protocol's fee accounting is not updated.

## Impact Explanation
The protocol fails to collect fee revenue on deposits below the truncation threshold. This matches the allowed Low impact: **"Contract fails to deliver promised returns, but doesn't lose value."** The protocol's fee mechanism is its designed economic behaviour; silently producing zero fee while minting full rsETH means the contract does not deliver its promised accounting. No user principal is lost.

## Likelihood Explanation
Any unprivileged depositor can trigger this by calling the public `deposit()` or `deposit(address, uint256, string)` functions with an amount below `10_000 / feeBps`. No special role, front-running, or external dependency is required. The threshold is small in absolute terms (e.g., < 2000 wei ETH for `feeBps = 5`), but the path is fully permissionless and repeatable across unlimited transactions.

## Recommendation
Add a guard in `viewSwapRsETHAmountAndFee` (or in each deposit function) that reverts when a non-zero fee rate produces a zero fee:

```solidity
if (feeBps > 0 && fee == 0) revert InvalidAmount();
```

Alternatively, enforce a protocol-level minimum deposit amount such that `amount * feeBps >= 10_000` is always satisfied for any configured `feeBps`.

## Proof of Concept
1. Deploy `RSETHPoolV3` with `feeBps = 5`.
2. Call `deposit{value: 1999}("")` from any EOA.
3. `viewSwapRsETHAmountAndFee(1999)` computes: `fee = 1999 * 5 / 10_000 = 9995 / 10_000 = 0`.
4. `amountAfterFee = 1999 - 0 = 1999`; `rsETHAmount = 1999 * 1e18 / rsETHToETHrate` (full amount).
5. `feeEarnedInETH += 0` — no-op; `wrsETH.mint(msg.sender, rsETHAmount)` — full rsETH minted.
6. Repeat across N transactions: protocol `feeEarnedInETH` remains 0 while rsETH is minted against the full deposited value each time.

Foundry fuzz test plan: fuzz `amount` in range `[1, 10_000/feeBps - 1]` and assert `fee == 0` while `rsETHAmount > 0`, confirming the protocol earns nothing on every such call.
