Audit Report

## Title
Cross-Asset Withdrawal Allows Extraction of Value from rsETH Holders After LST Price Drop - (`contracts/LRTWithdrawalManager.sol`)

## Summary
`initiateWithdrawal` and `instantWithdrawal` accept any supported asset as the withdrawal target with no link to the deposited asset. Because rsETH price is a basket-weighted average of all supported LSTs, a drop in one LST reduces rsETH price by only a fraction of the individual asset's drop. An attacker who deposited the depreciating LST and withdraws a different, stable asset (e.g., ETH) recovers more ETH than their LST is currently worth, with the difference borne by all other rsETH holders. The `_calculatePayoutAmount` min-cap reduces but does not eliminate the loss.

## Finding Description

**Deposit path** — `LRTDepositPool.depositAsset()` mints rsETH proportional to the deposited asset's current price:
```
rsethAmountToMint = (amount × assetPrice) / rsETHPrice
``` [1](#0-0) 

**Withdrawal path** — `initiateWithdrawal` accepts any supported asset as the withdrawal target, completely independent of what was deposited. The `expectedAssetAmount` is locked at initiation time:
```
expectedAssetAmount = rsETHUnstaked × rsETHPrice / assetPrice
``` [2](#0-1) [3](#0-2) 

**Payout cap** — `_calculatePayoutAmount` applies `min(expectedAssetAmount, currentReturn)` at unlock time:
```solidity
uint256 currentReturn = (request.rsETHUnstaked * rsETHPrice) / assetPrice;
return (request.expectedAssetAmount < currentReturn) ? request.expectedAssetAmount : currentReturn;
``` [4](#0-3) 

This cap only prevents the attacker from receiving *more* than the originally locked amount. It does **not** prevent the cross-asset arbitrage: when withdrawing ETH (assetPrice = 1.0), `currentReturn = rsETHUnstaked × rsETHPrice_after`. Since rsETH price drops by only `w × d` (basket weight × individual drop), the attacker recovers more ETH than their stETH is currently worth.

**rsETH price is a basket average** computed in `_getTotalEthInProtocol()`:
```solidity
totalETHInProtocol += totalAssetAmt.mulWad(assetER);
``` [5](#0-4) 

If stETH is weight `w` of the basket and drops by `d`, rsETH price drops by only `w × d`. The attacker recovers `deposit_value × (1 − w×d)` in ETH, while their stETH is worth `deposit_value × (1 − d)`. The difference — `deposit_value × d × (1 − w)` — is extracted from other rsETH holders.

**The attack does not require predicting the drop in advance.** An attacker who already holds rsETH (from a prior stETH deposit) can simply react to a price drop by choosing to withdraw ETH instead of stETH. The 8-day `withdrawalDelayBlocks` window is the only friction.

**`pricePercentageLimit` guard** only pauses the protocol on drops exceeding the configured threshold; smaller drops (or drops within the limit) remain fully exploitable. [6](#0-5) 

**`instantWithdrawal`** has no delay and uses the same `getExpectedAssetAmount()` formula, making the attack executable immediately when enabled for the target asset. [7](#0-6) 

## Impact Explanation

**Critical — Direct theft of any user funds.** The PoC demonstrates a concrete, quantifiable ETH loss for existing rsETH holders. The attacker extracts `deposit_value × drop_rate × (1 − LST_basket_weight)` in ETH from other protocol participants. This is not a theoretical or yield-only loss — it is a direct reduction in the ETH backing of all outstanding rsETH. The attack is repeatable and scales linearly with attacker capital.

## Likelihood Explanation

**Medium-High.** The attack is reactive, not predictive: any rsETH holder who originally deposited a now-depreciating LST can exploit this immediately after a price drop by choosing to withdraw a different asset. No privileged access, oracle manipulation, or advance knowledge is required. The only constraints are the 8-day withdrawal delay (for the standard path) and the `pricePercentageLimit` threshold (which only blocks large drops). The `instantWithdrawal` path removes even the delay constraint when enabled.

## Recommendation

1. **Restrict withdrawal asset to deposited asset** — track which asset each user deposited (or the weighted composition of their rsETH) and only allow withdrawal of that same asset.
2. **Per-asset rsETH accounting** — instead of a single fungible rsETH, track asset-specific rsETH shares so that stETH depositors can only withdraw stETH-backed value.
3. **Deposit freeze on oracle price anomalies** — pause new deposits when a supported LST's oracle price drops significantly within a short window, before `updateRSETHPrice()` is called.
4. **Minimum deposit-to-withdrawal period** — enforce a lock-up period longer than the maximum predictable slashing window to reduce the reactive attack surface.

## Proof of Concept

**Setup**: Protocol holds 100 ETH + 100 stETH (at 1.05 ETH/stETH). TVL = 205 ETH. rsETH supply = 205. rsETH price = 1.0 ETH.

**Step 1 — Attacker deposits 100 stETH** (worth 105 ETH):
- `rsETH minted = (100 × 1.05) / 1.0 = 105 rsETH`
- New TVL = 310 ETH, rsETH supply = 310, rsETH price = 1.0 ETH

**Step 2 — stETH drops 10%** → stETH price = 0.945 ETH:
- New TVL = 100 + 200 × 0.945 = 289 ETH
- `updateRSETHPrice()` → rsETH price = 289 / 310 ≈ 0.9323 ETH

**Step 3 — Attacker calls `initiateWithdrawal(ETH, 105)`**:
- `expectedAssetAmount = 105 × 0.9323 / 1.0 ≈ 97.89 ETH` (locked in)

**Step 4 — After 8 days, operator calls `unlockQueue(ETH, ...)`**:
- `currentReturn = 105 × 0.9323 / 1.0 ≈ 97.89 ETH`
- `payout = min(97.89, 97.89) = 97.89 ETH`

**Result**:
- Attacker receives **97.89 ETH**
- Attacker's 100 stETH is worth **94.5 ETH** (100 × 0.945)
- **Attacker profit: ~3.39 ETH** transferred from other rsETH holders
- Original 205 rsETH holders hold **191.11 ETH** instead of **194.5 ETH** — a **3.39 ETH direct loss**

    function instantWithdrawal(
        address asset,
        uint256 rsETHUnstaked,
        string calldata referralId
    )
        external
        nonReentrant
        whenNotPaused
        onlySupportedAsset(asset)
        onlySupportedStrategy(asset)
        onlyInstantWithdrawalAllowed(asset)
    {
        if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
            revert InvalidAmountToWithdraw();
        }
        if (IERC20(lrtConfig.rsETH()).balanceOf(msg.sender) < rsETHUnstaked) revert NotEnoughRsETH();
        uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
        IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
        ILRTUnstakingVault unstakingVault = ILRTUnstakingVault(lrtConfig.getContract(LRTConstants.LRT_UNSTAKING_VAULT));
        if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
            revert CantInstantWithdrawMoreThanAvailable();
        }

        unstakingVault.redeem(asset, assetAmountUnlocked);

        uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
        uint256 userAmount = assetAmountUnlocked - fee;

        address feeRecipient = instantWithdrawalFeeRecipient;
        if (feeRecipient == address(0)) {
            // Backwards-compatible default: send fees to the protocol treasury
            feeRecipient = lrtConfig.getContract(LRTConstants.PROTOCOL_TREASURY);
        }
        if (fee > 0) {
            _transferAsset(asset, feeRecipient, fee);
            emit InstantWithdrawalFeeCollected(msg.sender, asset, fee);
        }

        _transferAsset(asset, msg.sender, userAmount);
        emit ReferralIdEmitted(referralId);
        emit AssetWithdrawalFinalized(msg.sender, asset, rsETHUnstaked, userAmount);
    }
```
