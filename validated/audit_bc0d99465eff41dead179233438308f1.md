Audit Report

## Title
Permanent Freezing of Unclaimed Yield When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: `contracts/KERNEL/KernelDepositPool.sol`)

## Summary

`KernelDepositPool` uses a Synthetix-style reward accumulator that halts when `totalKernelStaked == 0`. Any staker can call `initiateWithdrawal` for their full balance at any time, including during an active reward window. If the last staker exits mid-period, all reward tokens that would have accrued from that point until `finishAt` are permanently trapped in the contract with no on-chain recovery path.

## Finding Description

The `rewardPerToken()` function returns `rewardPerTokenStored` unchanged whenever `totalKernelStaked == 0`:

```solidity
// L408-414
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

The `updateReward` modifier (L232-242) correctly snapshots the withdrawing user's earned rewards before their balance is reduced, so the exiting staker receives their fair share up to the withdrawal timestamp. However, the reward tokens that would have been emitted at `rewardRate` per second from the withdrawal timestamp until `finishAt` are never allocated to any account.

`initiateWithdrawal` (L320-338) imposes no restriction on reducing `totalKernelStaked` to zero during an active reward period (`block.timestamp < finishAt`). The `notifyRewardAmount` guard (L569-570) only prevents *starting* a new period with zero stakers; it does not prevent the supply from reaching zero after a period has begun.

The contract ends at L621 with no `recoverUnallocatedRewards`, `sweep`, or equivalent admin function. The NatSpec at L17-22 explicitly acknowledges this scenario and relies entirely on off-chain operational discipline ("ensuring there are always some tokens staked … for the entire duration of the reward period"), which is not enforced at the code level.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.**

Reward tokens transferred into the contract via `notifyRewardAmount` become permanently inaccessible if `totalKernelStaked` reaches zero at any point during the active reward window. The quantity frozen is `rewardRate × (finishAt − withdrawalTimestamp)` tokens. These tokens remain in the contract balance forever with no mechanism to redistribute or recover them. This maps exactly to the allowed impact class: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation

Any staker can unilaterally call `initiateWithdrawal` for their full balance at any time. No admin compromise, front-running, or external dependency is required. If a single dominant staker (or the last remaining staker) exits during an active reward period, the condition is met. The scenario is acknowledged in the contract's own NatSpec, confirming the protocol authors consider it reachable. The only mitigation is off-chain discipline, which is not enforceable on-chain.

## Recommendation

**Short term:** Add an admin `recoverUnallocatedRewards()` function that transfers out reward tokens in excess of the amount owed to current stakers (i.e., `rewardsToken.balanceOf(address(this)) - totalOwed`), callable only when `totalKernelStaked == 0`.

**Long term:** Either (a) enforce at the code level that `totalKernelStaked` cannot reach zero while `block.timestamp < finishAt` by reverting in `initiateWithdrawal` when the withdrawal would zero out the supply during an active period, or (b) automatically roll unallocated rewards into the next period via `notifyRewardAmount` when `totalKernelStaked` returns to a non-zero value.

## Proof of Concept

1. Admin calls `notifyRewardAmount(1_000e18)` with `totalKernelStaked > 0`. Sets `rewardRate` and `finishAt = block.timestamp + duration`.
2. The sole remaining staker calls `initiateWithdrawal(totalStake)`. The `updateReward(msg.sender)` modifier snapshots their earned rewards correctly up to this point. `totalKernelStaked` drops to `0`.
3. Time advances; `block.timestamp < finishAt`. `rewardPerToken()` returns `rewardPerTokenStored` unchanged because `totalKernelStaked == 0`. Rewards continue to be emitted at `rewardRate` per second but are allocated to no account.
4. `finishAt` passes. The unallocated reward quantity — `rewardRate × (finishAt − withdrawalTimestamp)` tokens — remains in the contract balance.
5. No function in the contract (confirmed by reading L1-621) can recover these tokens. They are permanently frozen.

**Foundry test sketch:**
```solidity
function test_rewardsFrozenOnZeroStake() public {
    // Setup: stake, start reward period
    vm.prank(admin);
    pool.notifyRewardAmount(1_000e18);

    // Last staker exits mid-period
    vm.prank(staker);
    pool.initiateWithdrawal(stakerBalance);
    assertEq(pool.totalKernelStaked(), 0);

    // Advance to end of reward period
    vm.warp(pool.finishAt());

    // Confirm reward tokens are stuck: no claimant, no recovery function
    uint256 stuck = rewardsToken.balanceOf(address(pool));
    assertGt(stuck, 0); // tokens remain, permanently frozen
}
```
