Audit Report

## Title
`initiateWithdrawal()` Always Reverts Due to Uninitialized `maxNumberOfWithdrawalsPerUser` — (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`maxNumberOfWithdrawalsPerUser` is declared as a `uint256` state variable but is never assigned in `initialize()`, leaving it at the Solidity default of `0`. The guard in `initiateWithdrawal()` checks `userWithdrawalIds[msg.sender].length >= maxNumberOfWithdrawalsPerUser`, which reduces to `length >= 0` — unconditionally `true` for every `uint256`. Every call to `initiateWithdrawal()` reverts with `WithdrawalLimitReached`, permanently blocking all withdrawals until an admin manually calls `setMaxNumberOfWithdrawalsPerUser()`.

## Finding Description
`maxNumberOfWithdrawalsPerUser` is declared at [1](#0-0)  with no default value, so it holds `0` after deployment.

`initialize()` sets `kernelToken`, `rewardsToken`, and roles, but never assigns `maxNumberOfWithdrawalsPerUser`: [2](#0-1) 

Inside `initiateWithdrawal()`, the guard at line 323 is: [3](#0-2) 

With `maxNumberOfWithdrawalsPerUser == 0`, the condition `userWithdrawalIds[msg.sender].length >= 0` is always `true` for any `uint256`, so every invocation reverts immediately with `WithdrawalLimitReached`.

The admin setter enforces a non-zero value and is the only way to fix the state post-deployment: [4](#0-3) 

There is no in-code enforcement that `setMaxNumberOfWithdrawalsPerUser()` must be called before staking opens. `stake()` and `stakeFor()` are fully functional immediately after `initialize()`, so users can deposit funds into a state where no withdrawal path exists.

## Impact Explanation
Every user who calls `stake()` or `stakeFor()` after deployment has their KERNEL tokens locked in the contract with no exit path. This is a **temporary freezing of funds (Medium)** — the funds are not permanently lost (an admin can unblock by calling the setter), but users are unable to withdraw their principal for an indefinite period until admin intervention occurs. The `MAX_WITHDRAWALS_PER_USER` constant of `100` confirms the intended non-zero default was simply omitted. [5](#0-4) 

## Likelihood Explanation
Medium. The contract is upgradeable and `initialize()` is the sole setup entry point. Any deployment or re-initialization that omits the post-init `setMaxNumberOfWithdrawalsPerUser()` call leaves the pool in the broken state. No attacker action is required — the broken state is the default. Any user who stakes immediately after deployment (which is permitted) is affected.

## Recommendation
Initialize `maxNumberOfWithdrawalsPerUser` to a safe non-zero default inside `initialize()`:

```diff
 function initialize(address _admin, address _kernelToken, address _rewardToken) external initializer {
     ...
     kernelToken = IERC20(_kernelToken);
     rewardsToken = IERC20(_rewardToken);
+    maxNumberOfWithdrawalsPerUser = MAX_WITHDRAWALS_PER_USER;
 }
```

## Proof of Concept
1. Deploy `KernelDepositPool` and call `initialize(admin, kernelToken, rewardToken)` — do **not** call `setMaxNumberOfWithdrawalsPerUser()`.
2. Confirm `maxNumberOfWithdrawalsPerUser == 0`.
3. User approves and calls `stake(100e18)`. Succeeds; `balanceOf[user] == 100e18`.
4. User calls `initiateWithdrawal(100e18)`.
5. Execution reaches line 323: `0 >= 0` → `true` → reverts with `WithdrawalLimitReached`.
6. User's 100 KERNEL are frozen with no exit path.
7. Admin calls `setMaxNumberOfWithdrawalsPerUser(10)`. Step 4 now succeeds — confirming the root cause is the uninitialized variable, not a logic error in the guard itself.

        external
        onlyRole(DEFAULT_ADMIN_ROLE)
