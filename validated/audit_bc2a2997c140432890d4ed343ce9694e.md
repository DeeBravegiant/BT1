### Title
Unconstrained `fee_proposal_fri` During Startup Window Poisons `fee_actual` Floor, Forcing Inflated L2 Gas Prices — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

During the first `fee_proposal_window_size` (= 10) blocks after Starknet V0_14_3 activation, `is_proposal_init_valid` explicitly skips the `fee_proposal_fri` bounds check because `fee_actual` is `None`. A malicious proposer controlling a majority of these startup blocks can set `fee_proposal_fri` to an arbitrarily large value. These values are recorded in `fee_proposals_window` and, once the window fills, become the `fee_actual` median that is used as a hard floor for the EIP-1559 L2 gas price. The result is that all subsequent blocks have their `l2_gas_price_fri` forced toward an extreme value, making transactions unaffordably expensive.

### Finding Description

**Step 1 — The intentional bypass.**

In `is_proposal_init_valid`, the bounds check on `fee_proposal_fri` is gated on `fee_actual` being `Some`:

```rust
// Validate fee_proposal is within the configured margin of fee_actual.
// During initiation (fee_actual is None, <window_size blocks), bounds are not enforced.
if let (Some(fee_actual), Some(fee_proposal)) =
    (proposal_init_validation.fee_actual, init_proposed.fee_proposal_fri)
{
    let (lower_bound, upper_bound) = fee_proposal_bounds(
        fee_actual,
        VersionedConstants::latest_constants().fee_proposal_margin_ppt,
    );
    if fee_proposal.0 < lower_bound || fee_proposal.0 > upper_bound {
        return Err(...);
    }
}

`fee_actual` is `None` whenever `compute_fee_actual` cannot fill the full `window_size` window:

```rust
let Some(start) = height.0.checked_sub(window_size) else {
    warn!("Cannot compute fee_actual for height {height}: height is below window_size ({window_size})");
    return None;
};
``` [2](#0-1) 

The window size is 10 blocks:

```json

``` [3](#0-2) 

**Step 2 — Unconstrained recording into the window.**

Every decided block records the proposer-stated `fee_proposal_fri` into `fee_proposals_window`:

```rust
fn record_fee_proposal(&mut self, height: BlockNumber, fee_proposal_fri: Option<GasPrice>) {
    self.fee_proposals_window.insert(height, fee_proposal_fri);
}
``` [4](#0-3) 

The value stored is taken directly from `init.fee_proposal_fri` — the proposer-supplied wire value — with no independent cap applied during the startup window.

**Step 3 — `fee_actual` becomes the EIP-1559 floor.**

After the window fills, `fee_actual` (the median of the recorded values) is used as a hard floor for the next L2 gas price:

```rust
let effective_min = match fee_actual {
    Some(fa) => GasPrice(max(config_min.0, fa.0)),
    None => config_min,
};
calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
``` [5](#0-4) 

If `fee_actual` is `u128::MAX`, `effective_min` is `u128::MAX`. `calculate_next_base_gas_price` then ratchets `l2_gas_price` upward by `price / 333` per block until it reaches `u128::MAX`:

```rust
if price < min_gas_price {
    let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
    let adjusted = price.0 + max_increase;
    let adjusted_price = adjusted.min(min_gas_price.0);
    return GasPrice(adjusted_price);
}
``` [6](#0-5) 

**Step 4 — The `l2_gas_price_fri` in `ProposalInit` is derived from this poisoned value.**

`initiate_build` sets `l2_gas_price_fri: args.l2_gas_price`, which is the `l2_gas_price` field updated by `update_l2_gas_price` after each decided block:

```rust
l2_gas_price_fri: args.l2_gas_price,
``` [7](#0-6) 

```rust

    self.l2_gas_price = self.calculate_next_l2_gas_price(height, l2_gas_used);
``` [8](#0-7) 

Validators enforce that `init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri`, so once the proposer's `l2_gas_price` is poisoned, all validators agree on the inflated price and the block is accepted.

**Step 5 — Honest proposer behavior during startup.**

An honest proposer freezes `fee_proposal` at `l2_gas_price` when `fee_actual` is `None`:

```rust
let Some(fee_actual) = fee_actual else {
    warn!("fee_actual unavailable, freezing fee_proposal at l2_gas_price");
    return self.l2_gas_price;
};
``` [9](#0-8) 

A malicious proposer ignores this convention and sets `fee_proposal_fri` to `u128::MAX` in the `ProposalInit` it broadcasts. Validators accept it because the bounds check is skipped.

### Impact Explanation

After the startup window, `fee_actual` equals the median of the 10 recorded `fee_proposal_fri` values. If a malicious proposer controls ≥ 6 of the 10 startup blocks and sets `fee_proposal_fri = u128::MAX` for each, the median is `u128::MAX`. The `effective_min` in `calculate_next_l2_gas_price_for_fin` becomes `u128::MAX`, and the EIP-1559 ratchet drives `l2_gas_price_fri` toward `u128::MAX` over subsequent blocks (~230 blocks to double). All transactions in those blocks pay inflated fees; eventually no transaction can afford the gas price, effectively halting the chain economically.

This matches the allowed impact: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

### Likelihood Explanation

The attack window is the first 10 blocks after V0_14_3 activation (a known, scheduled network upgrade). In a small validator set (e.g., 10 validators with equal weight), a single malicious validator controls ~1 block per 10 on average; colluding validators controlling ≥ 6 of 10 startup blocks is feasible in a permissioned or early-stage network. The attack requires no special on-chain privilege beyond being a scheduled proposer — exactly the analog of the external bug's "voter" role.

### Recommendation

1. **Bound `fee_proposal_fri` during startup.** Even when `fee_actual` is `None`, enforce that `fee_proposal_fri` is within a reasonable multiple of `l2_gas_price` (e.g., `[l2_gas_price / K, l2_gas_price * K]` for some constant `K`). This mirrors the external bug's fix of adding a hard gas limit.

2. **Cap `fee_proposal_fri` at `max_block_size * max_price` or a similar protocol-level ceiling** so that even a malicious proposer cannot inject `u128::MAX` into the window.

3. **Consider initializing the window from the last known `l2_gas_price`** rather than accepting arbitrary proposer values, so the median cannot deviate far from the bootstrapped price.

### Proof of Concept

```
Network upgrades to V0_14_3 at block N.
fee_proposals_window is empty; fee_actual = None for blocks N..N+9.

Malicious validator M is proposer for blocks N, N+1, N+2, N+3, N+4, N+5 (6 of 10).
M broadcasts ProposalInit { fee_proposal_fri: Some(GasPrice(u128::MAX)), ... } for each.

Validators call is_proposal_init_valid:
  fee_actual = None  →  bounds check is skipped  →  proposal accepted.

fee_proposals_window after block N+9:
  { N: MAX, N+1: MAX, N+2: MAX, N+3: MAX, N+4: MAX, N+5: MAX,
    N+6: 8gwei, N+7: 8gwei, N+8: 8gwei, N+9: 8gwei }

compute_fee_actual at block N+10:
  sorted = [8g, 8g, 8g, 8g, MAX, MAX, MAX, MAX, MAX, MAX]  (window_size=10, even)
  median = avg(window[4], window[5]) = avg(MAX, MAX) = MAX

calculate_next_l2_gas_price_for_fin at block N+10:
  effective_min = max(config_min, MAX) = MAX
  l2_gas_price < MAX  →  ratchet: new_price = l2_gas_price + l2_gas_price/333

Every subsequent block ratchets l2_gas_price upward.
After ~230 blocks, l2_gas_price doubles; after ~1600 blocks, it reaches u128::MAX.
All transactions fail: max_price_per_unit < l2_gas_price.
```
