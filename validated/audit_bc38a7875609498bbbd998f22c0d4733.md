Audit Report

## Title
Stale `rsETHPrice` Used in Deposit Minting Calculation Without Prior Refresh - (File: contracts/LRTDepositPool.sol)

## Summary

`LRTDepositPool.depositETH()` and `depositAsset()` compute the rsETH mint amount by dividing deposited asset value by `lrtOracle.rsETHPrice()`, a stored state variable that is only written when `updateRSETHPrice()` is explicitly called. Neither deposit path refreshes this value before use. When restaking yield has accrued since the last price update, the stored price is lower than the true rate, causing depositors to receive excess rsETH at the expense of existing holders' accrued yield.

## Finding Description

`getRsETHAmountToMint` computes:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

`rsETHPrice` is a plain storage variable in `LRTOracle` ( [1](#0-0) ) written only inside `_updateRsETHPrice()` ( [2](#0-1) ), which is invoked by the public `updateRSETHPrice()` ( [3](#0-2) ) or the manager-only `updateRSETHPriceAsManager()`.

The deposit entry points `depositETH()` ( [4](#0-3) ) and `depositAsset()` ( [5](#0-4) ) both delegate to `_beforeDeposit()`, which is declared `private view` ( [6](#0-5) ) — structurally preventing it from calling any state-mutating price refresh. `_beforeDeposit` calls `getRsETHAmountToMint` directly without any staleness guard. ( [7](#0-6) )

`_updateRsETHPrice` computes the true price as `totalETHInProtocol / rsethSupply` ( [8](#0-7) ). As restaking yield accrues, `totalETHInProtocol` rises continuously while `rsETHPrice` only catches up on explicit keeper calls. In the window between updates, the stored price is lower than the true rate, so the division `assetValue / rsETHPrice` yields a larger rsETH output than the depositor's proportional share warrants.

An additional aggravating factor: if the accumulated price increase exceeds `pricePercentageLimit`, a non-manager call to `updateRSETHPrice()` reverts with `PriceAboveDailyThreshold` ( [9](#0-8) ), extending the stale window until a manager intervenes and making the exploitable gap larger.

## Impact Explanation

Every rsETH token represents a proportional claim on the protocol's total ETH. When a depositor mints rsETH at a stale (understated) price, they receive a larger fraction than their deposit justifies. The excess rsETH is sourced from the yield that existing holders had already earned: the same total ETH is now split among more rsETH tokens, reducing the ETH value of each pre-existing token. This is a direct, quantifiable transfer of accrued yield from existing holders to the depositor.

**Impact class: High — Theft of unclaimed yield.**

## Likelihood Explanation

`updateRSETHPrice()` is driven by an off-chain keeper on a periodic schedule, not atomically with deposits. Any unprivileged address can call `depositETH()` or `depositAsset()`. An attacker only needs to observe that `rsETHPrice` has not been refreshed recently (trivially detectable on-chain by comparing `rsETHPrice` against a freshly computed `totalETHInProtocol / rsethSupply`) and submit a deposit before the next keeper update. The window is predictable, repeatable, and requires no special privilege or victim cooperation.

## Recommendation

Call `updateRSETHPrice()` (or inline `_updateRsETHPrice()`) at the start of `depositETH()` and `depositAsset()` before `_beforeDeposit` reads `rsETHPrice`, ensuring every deposit uses the current exchange rate. Because `_beforeDeposit` is `private view`, the price refresh must occur in the non-view deposit functions themselves, prior to the `_beforeDeposit` call. Alternatively, record a `lastPriceUpdateTimestamp` in `_updateRsETHPrice` and revert in `getRsETHAmountToMint` if the price is older than an acceptable staleness threshold.

## Proof of Concept

1. Protocol state: 1 000 ETH TVL, 1 000 rsETH supply → true price = 1.01 ETH/rsETH after yield accrual; stored `rsETHPrice` = 1.00 ETH/rsETH (not yet updated by keeper).
2. Attacker calls `depositETH{value: 10 ether}(0, "")`.
3. `getRsETHAmountToMint` computes: `10e18 * 1e18 / 1.00e18 = 10 rsETH` (stale denominator).
4. Fair amount at true price: `10e18 * 1e18 / 1.01e18 ≈ 9.9009 rsETH`.
5. Attacker receives ~0.0991 rsETH excess.
6. Post-deposit state: 1 010 ETH TVL, 1 010 rsETH supply → new price ≈ 1.00 ETH/rsETH. Existing 1 000 rsETH holders' claim dropped from 1 010 ETH to ~990 ETH worth — the ~0.1 ETH difference is the stolen yield.
7. Attacker repeats on every accrual cycle before the keeper fires.

**Foundry fork test plan:**
- Fork mainnet (or a testnet deployment).
- Warp time forward to simulate yield accrual without calling `updateRSETHPrice()`.
- Record `rsETHPrice` and compute `_getTotalEthInProtocol() / rsethSupply` to confirm divergence.
- Call `depositETH` from an unprivileged address and assert `rsethAmountToMint > fairAmount`.
- Call `updateRSETHPrice()` and assert existing holders' per-token ETH value decreased relative to pre-deposit.
