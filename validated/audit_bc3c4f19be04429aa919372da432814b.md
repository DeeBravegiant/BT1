### Title
`DirectDeposits` Not Included in Value Conservation Check Allows ADA Creation from Nothing — (`File: eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Entities.hs`, `eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/Utxo.hs`)

---

### Summary

The Dijkstra era introduces a `directDeposits` field in the transaction body that allows a transaction to directly credit ADA into registered stake account balances. However, the `directDeposits` amount is never included in the `produced` side of the ledger's value conservation check (`validateValueNotConservedUTxO`). As a result, any transaction submitter can include arbitrarily large `directDeposits` in a Dijkstra transaction and have those amounts credited to account balances without funding them from UTxO inputs, creating ADA from nothing.

---

### Finding Description

The Dijkstra era adds `DirectDeposits` as a new transaction body field: [1](#0-0) 

These deposits are applied in the `ENTITIES` rule via `applyDirectDeposits`, which unconditionally adds coin to registered account balances: [2](#0-1) 

The only validation performed on `directDeposits` before application is a check that target accounts are registered: [3](#0-2) 

Meanwhile, the Dijkstra UTXO rule enforces value conservation using the standard Shelley function: [4](#0-3) 

The `produced` function used by `validateValueNotConservedUTxO` sums UTxO outputs, fees, and certificate deposits. It does **not** include `directDeposits`. No Dijkstra-era override of `produced` or `consumed` was found that accounts for `directDeposits`. A grep search for any connection between `directDeposits` and `produced`/`consumed` returned no matches.

The `addCompactCoin` call inside `applyDirectDeposits` is explicitly documented as having no safety checks: [5](#0-4) 

The preservation-of-value property requires that every ADA credited to accounts must be debited from somewhere else in the same transition: [6](#0-5) 

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

An attacker submits a Dijkstra transaction with:
- UTxO inputs worth X ADA
- UTxO outputs worth X − fee ADA
- Fee of `fee` ADA
- `directDeposits` crediting Y ADA to one or more registered accounts

The value conservation check passes (`X = (X − fee) + fee`), but Y ADA is added to account balances without being funded by any input. This violates the global preservation-of-value invariant and constitutes direct, unbounded ADA creation. The credited ADA can subsequently be withdrawn via normal reward withdrawal transactions, permanently inflating the ADA supply and causing ledger divergence between nodes that enforce the invariant and those that do not.

---

### Likelihood Explanation

Any unprivileged transaction submitter on the Dijkstra era can exploit this. The attacker needs only:
1. A valid UTxO input to fund the fee.
2. One or more registered stake accounts as deposit targets (trivially obtained by registering their own).
3. Knowledge of the `directDeposits` transaction body field.

No special role, key, or governance threshold is required. The attack is repeatable and costless beyond the transaction fee.

---

### Recommendation

Include the sum of all `directDeposits` amounts in the `produced` calculation for the Dijkstra era. Specifically, override the `produced` type-class method for `DijkstraEra` to add `fold (unDirectDeposits (txBody ^. directDepositsTxBodyL))` to the produced value, mirroring how certificate deposits are already handled. This ensures that `directDeposits` must be funded by the transaction's UTxO inputs, restoring the preservation-of-value invariant.

Additionally, apply the same correction to sub-transaction bodies (`DijkstraSubTxBodyRaw`), which also carry a `dstbrDirectDeposits` field processed by the `SUBENTITIES` rule: [7](#0-6) 

---

### Proof of Concept

1. Register a stake credential `cred` and obtain its `AccountAddress`.
2. Construct a Dijkstra top-level transaction:
   - `inputs`: one UTxO worth 2 ADA
   - `outputs`: one UTxO returning 1 ADA to self
   - `fee`: 1 ADA
   - `directDeposits`: `{ AccountAddress(cred) → 1_000_000_000_000 ADA }`
3. Submit the transaction. `validateValueNotConservedUTxO` checks `consumed = 2 ADA` vs `produced = 1 + 1 = 2 ADA` — passes.
4. `dijkstraEntitiesTransition` calls `directDepositsMissingAccounts` — `cred` is registered, passes.
5. `applyDirectDeposits` credits 1,000,000,000,000 ADA to `cred`'s account balance.
6. Submit a withdrawal transaction draining the account balance into a UTxO.

Net result: 1,000,000,000,000 ADA created from 2 ADA of inputs, with no governance action or privileged access required. [8](#0-7) [9](#0-8)

dijkstraEntitiesTransition = do
  TRC (EntitiesEnv legacyMode certsEnv, certState, certificates) <- judgmentContext
  let Conway.CertsEnv tx pp curEpoch _committee _committeeProposals = certsEnv
      withdrawals = tx ^. bodyTxL . withdrawalsTxBodyL
      accounts = certState ^. certDStateL . accountsL

  runTest $ Conway.validateWithdrawalsDelegated accounts tx

  network <- liftSTS $ asks networkId

  validateWithdrawals legacyMode network withdrawals accounts

  let certStateBeforeCerts =
        certState
          & Conway.updateDormantDRepExpiries tx curEpoch
          & Conway.updateVotingDRepExpiries tx curEpoch (pp ^. ppDRepActivityL)
          & certDStateL . accountsL %~ applyWithdrawals withdrawals
  certStateAfterCerts <-
    trans @(EraRule "CERTS" era) $ TRC (certsEnv, certStateBeforeCerts, certificates)

  let directDeposits = tx ^. bodyTxL . directDepositsTxBodyL
      accountsAfterCerts = certStateAfterCerts ^. certDStateL . accountsL
  failOnJust (directDepositsMissingAccounts directDeposits accountsAfterCerts) $
    injectFailure . DirectDepositsToMissingAccounts

  pure $ certStateAfterCerts & certDStateL . accountsL %~ applyDirectDeposits directDeposits

dijkstraUtxoTransition = do
  TRC (DijkstraUtxoEnv slot pp certState originalUtxo, utxos, stAnnTx) <-
    judgmentContext
  let tx = stAnnTx ^. txStAnnTxG
  -- this is the original Accounts, before any transactions were applied
  let accounts = certState ^. certDStateL . accountsL

  let txBody = tx ^. bodyTxL

  {- inInterval (SlotOf Γ) (ValidIntervalOf txTop) -}
  runTest $ Allegra.validateOutsideValidityIntervalUTxO slot txBody

  sysSt <- liftSTS $ asks systemStart
  ei <- liftSTS $ asks epochInfo

  runTest $ Alonzo.validateOutsideForecast ei slot sysSt tx

  {- SpendInputs ≠ ∅ -}
  runTestOnSignal $ Shelley.validateInputSetEmptyUTxO txBody

  let allInputs = txBody ^. allInputsTxBodyF
      inputs = txBody ^. inputsTxBodyL

  {- SpendInputsOf txTop ∪ RefInputsOf txTop ∪ CollInputsOf txTop ⊆ dom(utxo₀) -}
  runTest $ Shelley.validateBadInputsUTxO originalUtxo allInputs

  {- SpendInputsOf txTop ⊆ dom(utxo_s) — prevents double-spend with subtxs -}
  runTest $ Shelley.validateBadInputsUTxO (utxosUtxo utxos) inputs

  validate $ validateBatchCollateral pp tx originalUtxo

  runTest $ Shelley.validateValueNotConservedUTxO pp originalUtxo certState txBody
