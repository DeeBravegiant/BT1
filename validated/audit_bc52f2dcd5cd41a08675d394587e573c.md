### Title
Relayer Fee Transferred Unconditionally Before ERC-20 `ft_transfer` Outcome Is Known — (`File: runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

### Summary
In the Wallet Contract's `inner_rlp_execute`, the NEAR-denominated relayer fee for an emulated ERC-20 transfer is dispatched as an independent promise **before** the `ft_transfer` promise chain executes. If the subsequent `ft_transfer` (or its prerequisite `storage_deposit`) fails, the relayer still receives the fee, the user's nonce is already incremented, and the user loses NEAR tokens with no token transfer having occurred.

### Finding Description
In `inner_rlp_execute`, when the parsed transaction is an `EthEmulation(EthEmulationKind::ERC20Transfer { fee, .. })` with a non-zero fee and an external relayer, the code unconditionally creates a promise to transfer the fee to the relayer: [1](#0-0) 

This promise is created via `env::promise_batch_create` + `env::promise_batch_action_transfer` and is **not** chained to the `ft_transfer` promise. It executes independently of whether the subsequent NEP-141 call succeeds.

Immediately after, the nonce is already incremented (line 364) for all non-registrar paths: [2](#0-1) 

Then the ERC-20 transfer path sets up a separate promise chain (`storage_balance_of` → `nep_141_storage_balance_callback` → `ft_transfer` → `rlp_execute_callback`): [3](#0-2) 

The `rlp_execute_callback` correctly detects failure and refunds the `caller_deposit` (NEAR attached by the external caller to `rlp_execute`): [4](#0-3) 

However, the relayer fee — taken from the wallet contract's own NEAR balance — is never refunded on failure. The fee promise and the `ft_transfer` promise chain are independent; there is no mechanism to cancel or condition the fee on the outcome of the transfer.

In the `None` branch of `nep_141_storage_balance_callback`, `storage_deposit` and `ft_transfer` are batched together: [5](#0-4) 

If `storage_deposit` fails (e.g., the token contract rejects it), `ft_transfer` is also skipped, but the relayer fee has already been sent.

### Impact Explanation
**Impact: High.** The user's wallet contract NEAR balance is reduced by the fee amount without the intended ERC-20 token transfer completing. The user's nonce is also incremented, preventing replay of the same signed transaction. The corrupted protocol values are:
- The wallet contract's NEAR **balance** (debited by the fee)
- The relayer's NEAR **balance** (credited by the fee)
- The wallet contract's **nonce** (incremented, preventing retry)

The user suffers a direct, irreversible loss of NEAR tokens with no corresponding benefit.

### Likelihood Explanation
**Likelihood: Low-to-Medium.** The trigger requires:
1. A user signing an ERC-20 transfer with a non-zero relayer fee (`gas_price > 0` in the Ethereum transaction)
2. The `ft_transfer` failing at runtime (e.g., insufficient token balance, token contract panicking, or `storage_deposit` being rejected by a non-standard token contract)

A malicious or careless relayer could submit a transaction they know will fail (e.g., the user has insufficient token balance) to collect the fee without the transfer completing. The user cannot prevent this once they have signed the transaction with a non-zero fee.

### Recommendation
The relayer fee transfer should be conditioned on the success of the `ft_transfer`. One approach is to move the fee payment into `rlp_execute_callback` and only dispatch it when `env::promise_result(0)` is `PromiseResult::Successful`. Alternatively, the fee promise should be chained after the `ft_transfer` promise so that it only executes if the transfer succeeds (using NEAR's promise chaining with a conditional callback).

### Proof of Concept
1. User signs an ERC-20 transfer Ethereum transaction with `gas_price > 0` (non-zero relayer fee) and a token amount exceeding their balance.
2. Relayer calls `rlp_execute` on the wallet contract.
3. `inner_rlp_execute` dispatches the fee transfer to the relayer unconditionally (lines 382–383).
4. The `storage_balance_of` → `nep_141_storage_balance_callback` → `ft_transfer` chain executes; `ft_transfer` fails due to insufficient balance.
5. `rlp_execute_callback` receives `PromiseResult::Failed`, returns `success: false`, and refunds only the `caller_deposit` (not the fee).
6. Result: relayer received the NEAR fee, user's nonce was incremented, no token transfer occurred, and the fee is permanently lost from the user's wallet contract balance.
