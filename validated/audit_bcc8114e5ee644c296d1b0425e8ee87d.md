Audit Report

## Title
Unguarded `msg.value` on Non-WETH Swap and Liquidity Functions Allows Accidental ETH to Be Stolen via `refundETH()` — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
All four swap entry-points in `MetricOmmSimpleRouter` and all four liquidity entry-points in `MetricOmmPoolLiquidityAdder` are declared `payable` but contain no `msg.value == 0` guard for non-WETH token paths. ETH sent alongside such calls is silently retained in the contract and immediately claimable by any caller via the permissionless `refundETH()` helper, which forwards the entire `address(this).balance` to `msg.sender`. The interface explicitly documents these as ERC-20-only paths, confirming native ETH is not an intended input.

## Finding Description
**Root cause — `payable` functions with no ETH guard:**

All four swap functions in `MetricOmmSimpleRouter` are declared `payable` with no check on `msg.value`: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

All four liquidity functions in `MetricOmmPoolLiquidityAdder` are similarly `payable` with no ETH guard: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

**Why the `receive()` guard does not protect against this:**

`PeripheryPayments` does guard against direct ETH transfers from non-WETH addresses: [9](#0-8) 

However, this `receive()` guard is only invoked for plain ETH transfers (zero calldata). When ETH is sent alongside a `payable` function call (e.g., `exactInputSingle{value: 1 ether}(...)`), the EVM routes execution to the function itself — `receive()` is never called — and the ETH is silently accepted.

**Why the ETH is not consumed:**

When `token != WETH`, `pay()` takes the `else` branch and calls `safeTransferFrom`, never touching `address(this).balance`: [10](#0-9) 

The sent ETH remains stranded in the contract.

**Permissionless `refundETH()` drains the full balance:**

`refundETH()` has no access control and sends the entire contract balance to `msg.sender`: [11](#0-10) 

**Secondary misappropriation path:**

If ETH is already sitting in the contract when a different user performs a WETH swap, `pay()` will consume that ETH first (lines 74–77) to satisfy the second user's WETH obligation, forwarding the victim's ETH to the pool on their behalf: [12](#0-11) 

**Interface confirms ERC-20-only scope:** [13](#0-12) 

## Impact Explanation
A user who accidentally sends ETH alongside a non-WETH swap or liquidity call permanently loses that ETH. The ETH is either stolen by any MEV bot or observer calling `refundETH()` in the same or a subsequent block, or misappropriated by a subsequent WETH swap that causes `pay()` to wrap and forward the stuck ETH to the pool on behalf of a different user. In both cases the victim has no on-chain recourse. This constitutes a direct loss of user principal, meeting the Critical/High threshold under the allowed impact gate.

## Likelihood Explanation
All swap and liquidity functions are `payable`, so wallets, aggregators, and multicall bundles can forward ETH without any EVM-level rejection. Users frequently confuse native ETH with WETH, especially when building multicall bundles that mix `unwrapWETH9`/`refundETH` with swap calls. MEV bots routinely monitor for unprotected `refundETH`-style helpers; the window between the victim's transaction and a theft call can be a single block. No special privilege or setup is required to trigger or exploit the issue.

## Recommendation
Add a `msg.value == 0` guard at the top of every swap and liquidity entry-point that does not intentionally accept native ETH:

```solidity
// In exactInputSingle, exactInput, exactOutputSingle, exactOutput,
// addLiquidityExactShares, addLiquidityWeighted:
require(msg.value == 0, "Router: unexpected ETH");
```

Alternatively, remove the `payable` modifier from all ERC-20-only entry-points and reserve `payable` only for functions that explicitly wrap ETH (e.g., a future `exactInputSingleWithETH`). The `multicall` wrapper should also be reviewed, as it is `payable` and delegates to these functions.

## Proof of Concept

```solidity
// Preconditions:
//   router   = deployed MetricOmmSimpleRouter
//   pool     = valid MetricOmm pool for USDC/DAI (neither is WETH)
//   victim   = EOA that mistakenly sends 1 ETH with a USDC→DAI swap
//   attacker = any EOA

// Step 1 – victim calls exactInputSingle with 1 ETH attached
router.exactInputSingle{value: 1 ether}(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool:             address(pool),
        tokenIn:          USDC,          // not WETH → pay() takes else branch
        tokenOut:         DAI,
        zeroForOne:       true,
        amountIn:         1_000e6,
        amountOutMinimum: 990e18,
        recipient:        victim,
        deadline:         block.timestamp + 60,
        priceLimitX64:    0,
        extensionData:    ""
    })
);
// Swap succeeds. pay() calls safeTransferFrom(USDC, victim, pool, 1000e6).
// 1 ETH is silently retained in the router (receive() was never called).

// Step 2 – attacker (or MEV bot) calls refundETH in the same or next block
router.refundETH();
// attacker receives 1 ETH; victim has no recourse
```
