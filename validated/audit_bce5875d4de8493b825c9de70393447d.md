### Title
Double Deposit Refund via Multiple Sub-Transactions Claiming the Same Staking Credential Deregistration — (`eras/dijkstra/impl/src/Cardano/Ledger/Dijkstra/Rules/SubUtxo.hs`)

---

### Summary

In the Dijkstra era, the `SUBUTXO` rule computes deposit refunds using a `certState` snapshot from the rule environment. When a top-level transaction contains multiple sub-transactions that each include an `UnRegDepositTxCert` for the **same** staking credential, each sub-transaction independently computes a refund against the same unmodified `certState`. This allows the deposit pot (`utxosDeposited`) to be decremented multiple times for a single deposit, creating ADA from nothing and breaking the preservation-of-value invariant.

---

Inside `updateUTxOStateNoFees`, the refund is computed as:

totalDeposits = certsTotalDepositsTxBody pp certState txBody

...

`certsTotalRefundsTxBody` resolves the refund amount by looking up the staking credential's deposit in the `certState`:

```haskell
conwayCertsTotalRefundsTxBody pp ConwayCertState {conwayCertDState, conwayCertVState} =
  getTotalRefundsTxBody
    pp
    (lookupDepositDState conwayCertDState)
    (lookupDepositVState conwayCertVState)
``` [3](#0-2) 

If the `certState` snapshot passed to each sub-transaction's `SUBUTXO` invocation is the **original** pre-batch certState (i.e., not updated after each sub-transaction's `SUBCERTS` rule runs), then two sub-transactions each containing `UnRegDepositTxCert stakingCred deposit` will both find `stakingCred` registered, both compute `deposit` as a refund, and both decrement `utxosDeposited` by `deposit`. The deposit pot is drained twice for a single deposit.

The developers have explicitly acknowledged this gap. The test file `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs` contains a disabled test (`xit`) with the comment `"TODO: predicate failure not yet implemented"`:

```haskell

The `xit` marker means the test is currently **skipped because the rejection logic does not yet exist**. The transaction is expected to fail but currently succeeds.

The `Obligations` type tracks the invariant that `utxosDeposited` must equal the sum of all individual deposits:

```haskell
data Obligations = Obligations
  { oblStake :: !Coin
  , oblPool  :: !Coin
  , oblDRep  :: !Coin

The `potEqualsObligation` invariant check confirms this must hold:

```haskell
potEqualsObligation certState utxoSt = obligations == pot
  where
    obligations = totalObligation certState (utxoSt ^. utxosGovStateL)
    pot = utxoSt ^. utxosDepositedL
``` [6](#0-5) 

A double-refund violates this invariant: `utxosDeposited` falls below `totalObligation`, meaning the deposit pot no longer covers all outstanding obligations.

---

### Impact Explanation

**Critical — Direct creation of ADA through an invalid ledger state transition.**

When two sub-transactions each claim the same deposit refund:
- `utxosDeposited` is decremented by `deposit` twice, but only one deposit was ever paid.
- The "extra" `deposit` Lovelace appears in the UTxO outputs of the sub-transactions.
- ADA is effectively minted from the deposit pot without a corresponding deposit having been made.
- The preservation-of-value invariant (`Circulation + Deposits + Fees + Rewards + Reserves + Treasury = constant`) is broken.

---

### Likelihood Explanation

**Medium-High.** The Dijkstra era is the newest era (protocol version 12) and sub-transactions are a new feature. The attack requires:
1. Registering a staking credential (standard operation, costs `keyDeposit`).
2. Crafting a top-level transaction with two sub-transactions, each containing `UnRegDepositTxCert` for the same credential.
3. Submitting the transaction.

No privileged access, governance majority, or leaked keys are required. Any unprivileged transaction author can execute this. The only prerequisite is having a registered staking credential, which is a routine operation.

---

### Recommendation

In the `SUBLEDGER` (or `SUBLEDGERS`) rule, the `certState` passed to each sub-transaction's `SUBUTXO` invocation must be the **updated** certState after the preceding sub-transaction's `SUBCERTS` rule has run — not the original snapshot. This ensures that once a credential is deregistered by sub-tx 1, sub-tx 2's `certsTotalRefundsTxBody` call returns zero for that credential.

Alternatively, implement a cross-sub-transaction uniqueness check on deregistration certificates before processing begins, rejecting any top-level transaction where the same credential appears in `UnRegDepositTxCert` across multiple sub-transactions.

The disabled test at `eras/dijkstra/impl/testlib/Test/Cardano/Ledger/Dijkstra/Imp/CertSpec.hs:53` should be re-enabled once the predicate failure is implemented.
