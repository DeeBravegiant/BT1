### Title
Byzantine Representative Sends Equivocating `c_all`/`h_all` to Non-Quorum Parties, Causing Inconsistent Key Share Acceptance — (`src/cbmpc/protocol/ec_dkg.cpp`)

### Summary

In `dkg_or_refresh_ac`, the `h_all` guard for non-quorum parties only verifies local consistency (the representative's `h_all` matches the hash of commitments *that party itself received*). A Byzantine representative can send different `c_all` commitment bytes to different non-quorum parties in the first `plain_broadcast`, then send per-recipient `h_all` values that each individually pass the local check, while also sending different `ac_pub_all`/`pi_r_all`/`rho_all`/`xij` openings. Each non-quorum party passes all verification steps but ends up with a different key share and a different public key `Q`.

### Finding Description

**Round 1 — `plain_broadcast(c_all)` (line 358):**

`c_all` is declared `uniform_msg<buf_t>`. The honest `pack` always serializes `msg`. However, `send_to_parties` sends `in[i]` to each party `i` via point-to-point `send_impl(i, in[i])`. A Byzantine representative controlling its own transport can supply `c_R_A` to quorum parties and `c_R_B` to non-quorum party P_x. [1](#0-0) [2](#0-1) 

**`h_all` computation diverges per party:**

Each party computes `h_all._i = sha256(all_received_c_s, quorum_pids, sid)` from the `c_all` values it received. Quorum parties compute `h_all_A` (using `c_R_A`); P_x computes an expected hash using `c_R_B`. [3](#0-2) 

**Round 2 — `plain_broadcast(h_all, ac_pub_all, pi_r_all, rho_all, xij)` (line 369):**

The Byzantine representative sends `h_all_A, ac_pub_all_A, pi_A, rho_A, xij_A` to quorum parties and `h_all_B, ac_pub_all_B, pi_B, rho_B, xij_B` to P_x. Both sets are internally consistent (each commitment opens correctly under its own `rho`/`pi`). [4](#0-3) 

**Quorum-party guard (line 377) — passes:**

Quorum parties check `h_all._j != h_all.received(representative_quorum_pid_index)`. All quorum parties received `h_all_A` from the representative, so this check passes. [5](#0-4) 

**Non-quorum-party guard (lines 400-403) — passes, but only locally:**

P_x checks `h_all.received(representative_quorum_pid_index) != sha256(cs, quorum_pids, sid)`. P_x received `h_all_B` from the representative and built `cs` using `c_R_B`, so `h_all_B == sha256(cs_B, ...)` and the check passes. There is **no cross-check** between non-quorum parties to detect that P_x received a different `h_all` than P_y. [6](#0-5) 

**Commitment opening and ZK verification — passes:**

P_x verifies `com_R_tag.set(rho_B, c_R_B).open(Rs_B, pi_B)` and `pi_B.verify(Rs_B, sid, R)`. Both pass because the representative prepared a valid, self-consistent commitment B. [7](#0-6) 

**Share verification — passes:**

`ac.verify_share_against_ancestors_pub_data(Qj, xij_B, ac_pub_all_B, ...)` passes because `xij_B` is consistent with `ac_pub_all_B`. [8](#0-7) 

**Final self-consistency check — passes:**

`x_i * G != Qis[job.get_name(i)]` passes because P_x's accumulated `x_i` and `Qis` are both derived from the same (equivocated) `ac_pub_all_B` and `xij_B`. [9](#0-8) 

**Result:** P_x completes `dkg_or_refresh_ac` successfully with a key share `x_Px` for a public key `Q_B` that differs from the quorum's `Q_A`. P_y (another non-quorum party) may receive yet another equivocated set and end up with `Q_C`. All parties return `SUCCESS`.

### Impact Explanation

Non-quorum parties end up holding key shares for mutually inconsistent public keys. Any subsequent signing attempt involving a non-quorum party will fail or produce an invalid signature, because the quorum's view of the public key differs from the non-quorum party's view. The Byzantine representative can selectively target specific non-quorum parties, causing silent, undetected divergence in the distributed key state. This constitutes a transport-message invariant break causing honest-party divergence and invalid cryptographic output with security impact (Medium per scope).

### Likelihood Explanation

The attack requires a single Byzantine quorum party that happens to be the `representative_quorum_pid_index` (the last quorum member in the party-index loop). Since `representative_quorum_pid_index` is deterministically the highest-indexed quorum party, any quorum party that knows the party ordering can predict whether it will be the representative. The attack requires only the ability to send different bytes to different recipients over the point-to-point transport, which is the standard Byzantine capability.

### Recommendation

Replace the non-quorum `h_all` check with a mechanism that provides global consistency, not just local consistency. The standard approach is to use `committed_group_broadcast` (already present in the codebase) for the `h_all` round, which hashes all parties' `h_all` values together and requires every party to agree on that aggregate hash before proceeding. Alternatively, non-quorum parties should cross-check their received `h_all` against a value that all parties (including other non-quorum parties) can independently verify — for example, by having all parties broadcast their locally computed `h_all` and requiring unanimous agreement. [10](#0-9)

template <typename... ARGS>
inline error_t committed_group_broadcast(job_mp_t& job, ARGS&... args) {
  error_t rv = UNINITIALIZED_ERROR;

  const crypto::mpc_pid_t& pid = job.get_pid();
  coinbase::crypto::commitment_t com(pid);
  // effectively, it does com.gen(args.msg...)
  std::apply([&com](auto&&... args) { com.gen(args...); },
             map_args_to_tuple([](auto& arg) { return arg.msg; }, args...));

  auto com_rand = job.uniform_msg<buf256_t>(com.rand);
  auto com_msg = job.uniform_msg<buf_t>(com.msg);

  if (rv = job.plain_broadcast(com_msg)) return rv;

  auto v = job.uniform_msg<buf256_t>(crypto::sha256_t::hash(com_msg.all_received()));

  if (rv = job.plain_broadcast(v, com_rand, args...)) return rv;

  for (int i = 0; i < job.get_n_parties(); i++) {
    if (v.msg != v.received(i)) return rv = job.mpc_abort(E_CRYPTO, "received hash mismatch");

    coinbase::crypto::commitment_t com(job.get_pid(i));
    com.set(com_rand.received(i), com_msg.received(i));
    rv = std::apply([&com](auto&&... args) { return com.open(args...); },
                    map_args_to_tuple([i](auto& arg) { return arg.received(i); }, args...));
    if (rv) return rv;
  }

  return SUCCESS;
}
