Audit Report

## Title
`minAmountToDeposit` Defaults to Zero Allowing Dust Deposits That Yield Zero rsETH Due to Integer Truncation - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool` never initializes `minAmountToDeposit` in `initialize()`, leaving it at zero by default. When a depositor supplies a dust amount of an LST with `minRSETHAmountExpected = 0`, integer division in `getRsETHAmountToMint()` truncates the result to zero, the zero-rsETH output passes all guards, and the user's tokens are permanently transferred to the protocol while they receive nothing.

## Finding Description
**Root cause 1 — zero default, no lower bound.**
`minAmountToDeposit` is declared at line 30 but never assigned in `initialize()` (lines 45–52), so it starts at `0`. The admin setter at lines 282–285 accepts any value including zero with no floor check.

**Root cause 2 — dust passes the guard.**
`_beforeDeposit` at lines 657–659 only rejects `depositAmount == 0` or amounts strictly below `minAmountToDeposit`. When `minAmountToDeposit == 0`, every non-zero deposit passes unconditionally.

**Root cause 3 — integer truncation to zero.**
Line 520 computes:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
For a 1-wei deposit with `getAssetPrice ≈ 1e18` and `rsETHPrice = 1.05e18`: `(1 × 1e18) / 1.05e18 = 0`.

**Root cause 4 — no zero-rsETH guard.**
Lines 667–669 check `rsethAmountToMint < minRSETHAmountExpected`. When the caller passes `minRSETHAmountExpected = 0`, a zero result satisfies `0 < 0 == false` and passes silently.

**Root cause 5 — `mint(address, 0)` succeeds.**
`RSETH.mint()` (lines 229–240) uses the `checkDailyMintLimit` modifier (lines 42–56). With `amount = 0`, the check `currentPeriodMintedAmount + 0 > maxMintAmountPerDay` only fails if the daily limit is already exhausted; otherwise it passes. OpenZeppelin's `_mint(to, 0)` does not revert. The full call sequence at lines 113–115 therefore executes `safeTransferFrom` (user loses tokens) then `_mintRsETH(0)` (user receives nothing).

## Impact Explanation
A depositor who supplies a dust LST amount with `minRSETHAmountExpected = 0` has their tokens permanently absorbed by the protocol while receiving zero rsETH. The absorbed assets inflate TVL, which raises `rsETHPrice` on the next oracle update, silently redistributing value to all existing rsETH holders at the depositor's expense. This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**: the protocol does not lose value; the depositor does.

## Likelihood Explanation
`minAmountToDeposit` is zero from deployment with no admin action required to trigger the condition. Any caller who passes `minRSETHAmountExpected = 0` (a common default in integrations, scripts, and aggregators) and deposits a sufficiently small amount triggers the issue. The condition is always present in the default deployment state. Likelihood is **Low** because the per-transaction loss is dust-sized, but the vulnerability is permanently active unless the admin explicitly calls `setMinAmountToDeposit` with a safe value.

## Recommendation
1. **Initialize `minAmountToDeposit` to a safe non-zero value** inside `initialize()` (e.g., `0.001 ether`).
2. **Add a lower bound in `setMinAmountToDeposit()`** — reject values below a protocol-defined minimum, analogous to `KernelVaultETH.setMinDeposit()` which reverts on zero.
3. **Add an explicit zero-rsETH guard in `_beforeDeposit()`**:
   ```solidity
   if (rsethAmountToMint == 0) revert InvalidAmountToDeposit();
   ```

## Proof of Concept
```
State:
  minAmountToDeposit = 0  (default, never set in initialize())
  rsETHPrice         = 1.05e18  (5% appreciation post-launch)
  stETH assetPrice   = 1e18

1. User calls:
   depositAsset(stETH, 1 wei, minRSETHAmountExpected=0, "")

2. _beforeDeposit (L657-659):
   depositAmount (1) == 0?              → false
   depositAmount (1) < minAmountToDeposit (0)? → false  ✓ passes

3. getRsETHAmountToMint(stETH, 1) (L520):
   = (1 * 1e18) / 1.05e18 = 0  (integer truncation)

4. rsethAmountToMint (0) < minRSETHAmountExpected (0)? → false  ✓ passes

5. safeTransferFrom(user → depositPool, 1 wei stETH)  → user loses 1 wei
6. _mintRsETH(0) → RSETH.mint(user, 0) → succeeds, user receives 0 rsETH

Result: 1 wei stETH permanently absorbed; rsETH supply unchanged;
        next rsETHPrice update benefits all existing holders.
```

Foundry fuzz test plan: fuzz `depositAmount` in range `[1, rsETHPrice / assetPrice - 1]` with `minRSETHAmountExpected = 0` and assert `rsethAmountToMint == 0` while `balanceOf(user)` decreases by `depositAmount`.
