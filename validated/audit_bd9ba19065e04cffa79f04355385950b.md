Audit Report

## Title
ETH Deposit Limit Check Omits Deposit Amount, Allowing Limit Bypass - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric boundary check: the ETH branch evaluates only `totalAssetDeposits > limit`, ignoring the incoming `amount`, while the ERC20 branch correctly evaluates `totalAssetDeposits + amount > limit`. Any depositor can therefore push ETH deposits arbitrarily above the configured cap in a single transaction.

## Finding Description
In `LRTDepositPool._checkIfDepositAmountExceedesCurrentLimit` (lines 676–682), the ETH branch omits `amount` from the comparison:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount ignored
}
return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // amount included
``` [1](#0-0) 

The guard returns `false` (i.e., "not exceeded") whenever the pre-deposit running total has not yet crossed the limit, regardless of how large `msg.value` is. The call chain is fully public:

- `depositETH` (line 76) passes `msg.value` as `depositAmount` to `_beforeDeposit`. [2](#0-1) 
- `_beforeDeposit` (line 661) calls `_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)` and reverts only if it returns `true`. [3](#0-2) 

Because the ETH branch never adds `amount` to `totalAssetDeposits`, the check will never revert for a single deposit as long as the pre-deposit total is at or below the limit, no matter how large the deposit is.

## Impact Explanation
The `depositLimitByAsset` cap is the protocol's primary mechanism for bounding ETH exposure. Because the check is broken for ETH, any depositor can bypass the cap in a single transaction, minting rsETH against an unbounded ETH balance. This violates the protocol's stated invariant and pushes total ETH deposits far beyond the intended ceiling. Impact: **Low — contract fails to deliver promised returns (deposit cap not enforced), but deposited funds are not directly stolen or frozen.**

## Likelihood Explanation
The entry point is the public, permissionless `depositETH` function. No special role, front-running, or external dependency is required. Any depositor who observes that `totalAssetDeposits ≤ depositLimit` can immediately exploit this in a single transaction. Likelihood: **High.**

## Recommendation
Include the incoming deposit amount in the ETH branch, mirroring the ERC20 branch:

```solidity
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
``` [4](#0-3) 

## Proof of Concept
1. Assume `depositLimitByAsset[ETH_TOKEN] = 100 ether` and `totalAssetDeposits(ETH) = 50 ether`.
2. Attacker calls `depositETH{value: 200 ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 200 ether)` evaluates `50 ether > 100 ether` → `false` (not exceeded). [5](#0-4) 
4. `_beforeDeposit` does not revert; the deposit proceeds and `totalAssetDeposits(ETH)` becomes `250 ether` — 2.5× the configured cap. [3](#0-2) 
5. The equivalent ERC20 call would evaluate `50 ether + 200 ether > 100 ether` → `true` (reverts), confirming the asymmetry. [6](#0-5) 

**Foundry fuzz test plan:**
```solidity
function testFuzz_ETHDepositBypassesLimit(uint96 depositAmount) public {
    vm.assume(depositAmount > depositLimit);
    // totalAssetDeposits starts at 0, which is <= depositLimit
    vm.deal(attacker, depositAmount);
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: depositAmount}(0, "");
    assertGt(lrtDepositPool.getTotalAssetDeposits(ETH_TOKEN), depositLimit);
}
```
