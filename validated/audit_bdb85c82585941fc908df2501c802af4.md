Audit Report

## Title
Reward Tokens Permanently Frozen When `totalKernelStaked` Drops to Zero During Active Reward Period - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
`KernelDepositPool` uses a Synthetix-style staking rewards model where `rewardPerToken()` short-circuits to `rewardPerTokenStored` when `totalKernelStaked == 0`, but the `updateReward` modifier unconditionally advances `updatedAt` to `lastTimeRewardApplicable()`. If all stakers withdraw during an active reward window, the entire zero-staked interval is silently skipped when the next staker arrives, permanently locking the corresponding reward tokens in the contract with no recovery path.

## Finding Description
`rewardPerToken()` (L408–414) returns `rewardPerTokenStored` unchanged when `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    return rewardPerTokenStored + (rewardRate * (lastTimeRewardApplicable() - updatedAt) * DECIMAL_PRECISION)
        / totalKernelStaked;
}
```

The `updateReward` modifier (L232–242) always executes `updatedAt = lastTimeRewardApplicable()` regardless of whether `totalKernelStaked` is zero:

```solidity
modifier updateReward(address _account) {
    rewardPerTokenStored = rewardPerToken();
    updatedAt = lastTimeRewardApplicable();   // advances even when totalKernelStaked == 0
    ...
}
```

`initiateWithdrawal()` (L320–338) decrements `totalKernelStaked` with no guard preventing it from reaching zero during an active reward period. The `updateReward` modifier runs before the function body, so `rewardPerToken()` is computed while `totalKernelStaked` is still non-zero (correctly crediting the withdrawing user), but `updatedAt` is then advanced to the current timestamp. After the function body executes, `totalKernelStaked` becomes zero.

When the next staker calls `stake()`, `updateReward` fires again with `totalKernelStaked` still zero (the increment happens after the modifier), so `rewardPerToken()` returns `rewardPerTokenStored` unchanged and `updatedAt` is advanced again to the new current time. The entire zero-staked interval `[T_withdraw, T_newStake]` is permanently skipped: `rewardRate * (T_newStake - T_withdraw)` worth of reward tokens remain in the contract with no function to recover them.

`notifyRewardAmount()` (L569–570) guards against starting a reward period with zero stakers, but provides no protection against `totalKernelStaked` dropping to zero after a period has begun. The NatSpec at L18–22 explicitly acknowledges this limitation but relies entirely on an off-chain operational assumption with no on-chain enforcement.

## Impact Explanation
Reward tokens (`rewardsToken` ERC-20) that should have been distributed to stakers are permanently locked in `KernelDepositPool`. There is no admin rescue function, no `recoverERC20`, and no mechanism to restart the reward period to cover the lost interval. This constitutes **Medium — Permanent freezing of unclaimed yield**.

## Likelihood Explanation
The condition requires `totalKernelStaked` to reach zero during an active reward window. `initiateWithdrawal()` is permissionless for any staker; no admin action is required. A single dominant staker withdrawing their full balance is sufficient. Early-stage deployments with few stakers are particularly susceptible. The NatSpec itself acknowledges the risk, confirming the developers consider it a realistic scenario.

## Recommendation
Add an on-chain guard in `initiateWithdrawal()` that prevents `totalKernelStaked` from reaching zero while a reward period is active (`block.timestamp < finishAt`):

```solidity
if (totalKernelStaked - _amount == 0 && block.timestamp < finishAt) revert ActiveRewardPeriod();
```

Alternatively, implement an admin `recoverStuckRewards()` function that sweeps reward tokens that were never distributed, computed as `rewardRate * zero_staked_duration`, analogous to the `stuckEmissionsRecovery` pattern.

## Proof of Concept
1. Admin calls `notifyRewardAmount(1_000e18)` while Alice has 100 KERNEL staked → `rewardRate = 1_000e18 / duration`, `finishAt = now + duration`, `updatedAt = now` (T0).
2. Alice calls `initiateWithdrawal(100)` at T1 → `updateReward` fires with `totalKernelStaked = 100`: `rewardPerTokenStored` correctly accumulates rewards for `[T0, T1]`, `updatedAt = T1`. Function body then sets `totalKernelStaked = 0`.
3. 50% of the reward period elapses (`T2 = T0 + duration/2`) with `totalKernelStaked == 0`. Every call to `rewardPerToken()` returns `rewardPerTokenStored` unchanged.
4. Bob calls `stake(1)` at T2 → `updateReward` fires with `totalKernelStaked` still 0 (increment is post-modifier): `rewardPerToken()` returns `rewardPerTokenStored` unchanged, `updatedAt = T2`. Function body sets `totalKernelStaked = 1`.
5. `rewardRate * (T2 - T1)` reward tokens (≈50% of the remaining period's rewards) are permanently stuck. No function exists to recover them.

Foundry invariant test: assert that `rewardsToken.balanceOf(address(pool)) == 0` after all reward periods end and all stakers have claimed — this invariant will be violated after a zero-staked interval.
