### Title
Single Malicious Participant Can Replay Stale Foreign-Tx Attestations via `respond_verify_foreign_tx` - (File: crates/contract/src/lib.rs)

### Summary

`respond_verify_foreign_tx` verifies the ECDSA signature against a **caller-supplied** `payload_hash` with no binding to the current pending request and no replay protection. A single malicious attested participant (below threshold) can reuse a previously-seen valid `(payload_hash, signature)` pair to resolve a new pending `verify_foreign_transaction` request, bypassing the threshold requirement and delivering a stale attestation to the caller.

### Finding Description

`respond_verify_foreign_tx` (lines 691–754 of `crates/contract/src/lib.rs`) accepts a `VerifyForeignTransactionResponse` containing two caller-supplied fields: `payload_hash` and `signature`. The contract's only cryptographic check is:

```rust
let payload_hash: [u8; 32] = response.payload_hash.0;
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
)
.is_ok()
``` [1](#0-0) 

The contract verifies that `signature` is a valid ECDSA signature over `payload_hash` using the root public key. It does **not** verify that `payload_hash` is the correct hash of `(request, extracted_values)` for the current pending request. There is no nonce, receipt ID, timestamp, or any other replay-prevention mechanism.

Contrast this with the regular `respond` path, where the payload hash is taken directly from the submitted `request.payload` (not from the response), making replay impossible: [2](#0-1) 

The `VerifyForeignTransactionRequest` key used to look up pending yields is caller-agnostic — it contains only `request`, `domain_id`, and `payload_version`, with no caller account ID or nonce: [3](#0-2) 

This is confirmed by `args_into_verify_foreign_tx_request`, which strips all caller context when constructing the map key: [4](#0-3) 

Once a valid `(payload_hash, signature)` pair is produced by the threshold of participants and broadcast on-chain, any single attested participant can reuse it to resolve any future pending entry under the same `VerifyForeignTransactionRequest` key, because `resolve_yields_for` only checks that the entry exists and then removes it: [5](#0-4) 

The `ForeignTxSignPayloadV1` that produces `payload_hash` contains only `(request, extracted_values)` — no receipt ID, block height, or timestamp — so the same hash is reproduced identically for the same foreign transaction regardless of when it was computed: [6](#0-5) 

### Impact Explanation

A single malicious attested participant (below threshold) can resolve a pending `verify_foreign_transaction` request with a stale, replayed attestation, bypassing the threshold requirement for the new request. If the foreign chain has undergone a reorg between the original and replayed attestation, the user's NEAR contract receives a cryptographically valid response (correct signature, valid `payload_hash`) attesting to a transaction that no longer exists on the canonical foreign chain. For bridge contracts (the primary use case of `verify_foreign_transaction`, per the design doc), this enables a double-spend: the user can claim bridged funds a second time using the replayed attestation for a reversed deposit transaction.

This matches the allowed impact: **High — cross-chain replay / forged foreign-chain verification causing invalid bridge execution or double-spend conditions.**

### Likelihood Explanation

Requires: (1) one malicious attested participant, (2) the same foreign tx submitted twice (realistic after a timeout or user retry), and (3) a foreign chain reorg between submissions. Foreign chain reorgs are realistic for chains with low confirmation requirements (e.g., Bitcoin with `confirmations=1`, EVM chains before finality). The attacker needs no special capability beyond being an attested participant and having observed a prior on-chain response.

### Recommendation

Bind `payload_hash` to the specific pending request instance by including a request-specific nonce (e.g., the NEAR receipt ID of the `verify_foreign_transaction` call) in the signed payload. The contract already has access to receipt IDs via the yield mechanism. Alternatively, require the responder to submit the raw `extracted_values` alongside the response, and have the contract recompute and compare `payload_hash` on-chain, ensuring the hash corresponds to the current request.

### Proof of Concept

1. User calls `verify_foreign_transaction(tx_T)` → pending entry created under key `K = {tx_T, domain_id, V1}`.
2. Threshold of MPC nodes verify tx_T (valid in block B1), compute `payload_hash_T = hash(tx_T, block_B1_hash)`, produce `sig_T = ECDSA_root(payload_hash_T)`, call `respond_verify_foreign_tx(K, {payload_hash_T, sig_T})`.
3. `resolve_yields_for` removes entry for K; user receives `{payload_hash_T, sig_T}` and mints bridged funds.
4. Foreign chain reorgs; tx_T is reversed; block B1 is orphaned.
5. User calls `verify_foreign_transaction(tx_T)` again (retry) → new pending entry created under same key K.
6. Single malicious participant calls `respond_verify_foreign_tx(K, {payload_hash_T, sig_T})` (replaying step 2's response).
7. Contract checks: `sig_T` valid over `payload_hash_T` using root key ✓; entry for K exists ✓ → `resolve_yields_for` succeeds.
8. User receives `{payload_hash_T, sig_T}` — a valid attestation for a reversed transaction — and mints bridged funds a second time. [7](#0-6)

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }

pub(crate) fn resolve_yields_for<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: &K,
    response_bytes: Vec<u8>,
) -> Result<(), Error>
where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let resumed = requests
        .remove(request)
        .unwrap_or_default()
        .into_iter()
        .map(|YieldIndex { data_id }| {
            env::promise_yield_resume(&data_id, response_bytes.clone());
        })
        .count();

    if resumed > 0 {
        Ok(())
    } else {
        Err(InvalidParameters::RequestNotFound.into())
    }
}
