Audit Report

## Title
ETH Deposit Cap Check Missing `amount` in `_checkIfDepositAmountExceedesCurrentLimit()` - (File: `contracts/LRTDepositPool.sol`)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit()` uses an asymmetric comparison: for LST assets it correctly evaluates `totalAssetDeposits + amount > depositLimit`, but for ETH it omits `amount` and only evaluates `totalAssetDeposits > depositLimit`. Any ETH deposit that would push total deposits above the governance-set cap is silently accepted, bypassing the protocol's deposit ceiling.

## Finding Description
In `contracts/LRTDepositPool.sol` at lines 676–682:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // ← amount omitted
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset)); // ← correct
}
```

The ETH branch evaluates `totalAssetDeposits > limit` rather than `totalAssetDeposits + amount > limit`. This function is the sole cap guard, called from `_beforeDeposit()` (lines 648–670), which is invoked by the public, permissionless `depositETH()` (lines 76–93).

**Exploit path:**
1. `totalAssetDeposits(ETH) == depositLimit` (at or below cap).
2. Attacker calls `depositETH{value: X ether}(0, "")`.
3. `_checkIfDepositAmountExceedesCurrentLimit(ETH, X)` evaluates `depositLimit > depositLimit` → `false` → no revert.
4. `_mintRsETH` mints rsETH; total ETH becomes `depositLimit + X`, exceeding the cap.
5. Repeatable as long as `totalAssetDeposits <= depositLimit` at call time.

The existing `nonReentrant`, `whenNotPaused`, and `onlySupportedAsset` guards do not address this logic flaw.

## Impact Explanation
The ETH deposit cap is a governance-set risk-management parameter. Bypassing it allows more ETH than intended to flow into EigenLayer strategies, causing the protocol to fail to enforce its own promised deposit ceiling. No funds are directly stolen or frozen. **Impact: Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
`depositETH()` is public and requires no special role or privilege. The condition is met whenever `totalAssetDeposits` is at or near the limit, which is a normal operational state as the protocol approaches capacity. Any depositor can trigger this repeatedly. **Likelihood: High.**

## Recommendation

```diff
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
-       return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
+       return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This makes the ETH branch consistent with the LST branch and correctly accounts for the incoming deposit amount before comparing against the cap.

## Proof of Concept

**Foundry unit test plan:**
1. Deploy `LRTDepositPool` with a mock `lrtConfig` setting `depositLimitByAsset[ETH] = 1000 ether`.
2. Seed the pool so `getTotalAssetDeposits(ETH)` returns exactly `1000 ether`.
3. Call `depositETH{value: 100 ether}(0, "")` from an unprivileged address.
4. Assert the call does **not** revert (demonstrating the bypass).
5. Assert `getTotalAssetDeposits(ETH)` returns `1100 ether` (10% above cap).
6. For comparison, repeat with an LST asset at the same conditions and assert it **does** revert with `MaximumDepositLimitReached`.
