Audit Report

## Title
`stakedButUnverifiedNativeETH` Never Adjusted for Pre-Verification Beacon-Chain Slashing, Permanently Overstating TVL and Inflating rsETH Price — (`contracts/NodeDelegator.sol`)

## Summary

`stake32Eth` increments `stakedButUnverifiedNativeETH` by 32 ETH, but the only decrement path is inside `verifyWithdrawalCredentials`. If a validator is slashed on the beacon chain before credentials are verified, the actual EigenPod balance falls below 32 ETH while `stakedButUnverifiedNativeETH` remains unchanged. This overstates TVL, inflates the rsETH price via the permissionless `updateRSETHPrice()`, and allows any rsETH holder to redeem at the inflated rate — extracting more ETH than their proportional share, with the deficit borne by remaining holders when the true balance is eventually reflected.

## Finding Description

**Root cause — no decrement path for pre-verification slashing:**

`stake32Eth` unconditionally adds 32 ETH: [1](#0-0) 

The only decrement is in `verifyWithdrawalCredentials`, which subtracts a flat `validatorFields.length * 32 ether` regardless of actual awarded shares: [2](#0-1) 

A `grep` of the entire codebase confirms there are only two write sites for `stakedButUnverifiedNativeETH` — the increment and this single decrement — with no path that reduces it when a beacon-chain slash occurs before credentials are verified.

**TVL propagation:**

`getEffectivePodShares()` sums `stakedButUnverifiedNativeETH` with EigenLayer's `withdrawableShare` (which is 0 until credentials are verified): [3](#0-2) 

This feeds into `getETHDistributionData()`: [4](#0-3) 

Which is consumed by `_getTotalEthInProtocol()` → `_updateRsETHPrice()`: [5](#0-4) 

**Permissionless price update:**

`updateRSETHPrice()` has no role check — any caller can trigger it: [6](#0-5) 

**`pricePercentageLimit` guard is insufficient:**

The guard only reverts for non-managers when `newRsETHPrice > highestRsethPrice` and the percentage increase exceeds the configured limit: [7](#0-6) 

A typical initial slashing penalty (~1–2 ETH on a 32 ETH validator) against a protocol with significant TVL (e.g., 1000 ETH) produces a price inflation of ~0.1–0.2%, well within any reasonable `pricePercentageLimit` threshold.

**Redemption at inflated price:**

`instantWithdrawal` calls `getExpectedAssetAmount`, which reads the current (inflated) `rsETHPrice` at execution time: [8](#0-7) [9](#0-8) 

## Impact Explanation

**High — Theft of unclaimed yield.**

When a validator is slashed (e.g., 2 ETH penalty on a 32 ETH validator), `stakedButUnverifiedNativeETH` still counts 32 ETH. The rsETH price is inflated by `slashingLoss / rsETHSupply`. An attacker who holds rsETH redeems at this inflated rate, receiving more ETH than their proportional share. The deficit materialises for remaining holders when `verifyWithdrawalCredentials` is eventually called and EigenLayer awards shares only for the actual (slashed) effective balance — at that point the TVL drops and the rsETH price corrects downward. The attacker has extracted yield/principal that belongs to other depositors.

## Likelihood Explanation

- Beacon-chain slashings are rare but real production events.
- The window between `stake32Eth` and `verifyWithdrawalCredentials` is typically days to weeks, giving ample time for exploitation.
- `updateRSETHPrice()` is permissionless; the attacker triggers it themselves with no role required.
- `instantWithdrawal` is a supported production path requiring no operator involvement.
- For a protocol with meaningful TVL, the slashing-induced price inflation is a small percentage unlikely to exceed `pricePercentageLimit`, so the guard does not block the attack.
- No admin compromise, governance capture, or external oracle manipulation is required.

## Recommendation

When `verifyWithdrawalCredentials` is called, EigenLayer awards shares equal to the validator's **effective balance** (which may be less than 32 ETH after slashing). The fix should:

1. Read `podOwnerDepositShares` from `IEigenPodManager` before and after calling `eigenPod.verifyWithdrawalCredentials`.
2. Replace the flat `stakedButUnverifiedNativeETH -= validatorFields.length * 32 ether` with a decrement of exactly 32 ETH per validator, and separately track the delta of awarded shares so `getEffectivePodShares()` reflects the actual verified balance immediately.

Alternatively, add an operator-callable function to write down `stakedButUnverifiedNativeETH` by a proven slashing amount (with a beacon proof), so the TVL can be corrected before `verifyWithdrawalCredentials` is called.
