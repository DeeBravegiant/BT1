### Title
Missing On-Chain Response Verification in `respond_ckd` for `AppPublicKey` Variant Allows Any Single Attested Participant to Forge Confidential Key Derivation Outputs — (`crates/contract/src/lib.rs`)

---

### Summary

`respond_ckd` performs an on-chain cryptographic check for the `AppPublicKeyPV` variant of a CKD request, but the `AppPublicKey` (legacy) branch is a complete no-op. Any single attested participant can call `respond_ckd` with an arbitrary forged `CKDResponse` for any pending `AppPublicKey` request, and the contract will accept it and deliver it to every waiting caller — no threshold agreement required.

---

### Finding Description

In `respond_ckd`, after verifying the caller is an attested participant, the contract branches on the request's app public key type: [1](#0-0) 

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no verification
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
```

The `AppPublicKeyPV` arm calls `ckd_output_check`, which performs an on-chain BLS12-381 pairing check to confirm the encrypted output is consistent with the master public key and the app's ephemeral key pair. The `AppPublicKey` arm does nothing. The unverified `response` is then immediately passed to `resolve_yields_for`, which drains **all** pending yield slots queued under that request key: [2](#0-1) 

Compare this with `respond` for ECDSA/EdDSA signatures, which always derives the expected public key and verifies the signature before resolving any yield: [3](#0-2) 

The asymmetry is structural: for every other response path the contract can independently verify correctness on-chain; for `AppPublicKey` CKD it cannot (the verification requires the app's BLS secret key, which only the user holds). The result is that the contract's only defence for this path is the TEE attestation of the responder — a single-participant trust assumption that contradicts the threshold model.

The `AppPublicKey` variant is still documented and accepted as a production input: [4](#0-3) 

---

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Observe any pending `request_app_private_key` call that uses the `AppPublicKey` variant (the request is public on-chain).
2. Construct an arbitrary `CKDResponse` — for example, encrypting an attacker-chosen BLS scalar under the user's app public key.
3. Call `respond_ckd` with the forged response. The contract passes all access-control checks and resolves every queued yield for that request with the attacker-supplied ciphertext.
4. The user decrypts the response and derives a key that the attacker already knows.
5. Any asset, secret, or capability protected by that derived key is fully compromised.

This matches the allowed critical impact: **"Unauthorized … confidential key derivation output without the required participant authorization."**

---

### Likelihood Explanation

- The `AppPublicKey` variant is the legacy production format; existing integrations use it.
- Exploitation requires only one attested participant — no threshold collusion, no TEE break, no network-level attack.
- The attacker's forged call is indistinguishable from a legitimate response at the contract level.
- The window of opportunity is the entire lifetime of a pending CKD yield (up to ~200 blocks), which is ample time for an online participant to race an honest response.

---

### Recommendation

Two complementary mitigations:

1. **Deprecate `AppPublicKey` in favour of `AppPublicKeyPV`**, which supports the on-chain pairing check. Reject `AppPublicKey` requests at the `request_app_private_key` entry point after a migration window.
2. **Until deprecation is complete**, require threshold agreement before `respond_ckd` resolves any yield for the `AppPublicKey` variant — analogous to how the signing flow requires threshold-many nodes to independently compute and submit the same signature before it is accepted.

---

### Proof of Concept

```
1. User calls request_app_private_key(app_public_key = AppPublicKey(pk_user), ...)
   → contract stores pending CKD yield Y under request key K.

2. Attacker (single attested participant) constructs:
     forged_response = BLS_encrypt(attacker_chosen_scalar, pk_user)

3. Attacker calls respond_ckd(request = K, response = forged_response).
   → assert_caller_is_signer()                          ✓ (attacker signs directly)
   → assert_caller_is_attested_participant_and_protocol_active()  ✓ (attacker is attested)
   → AppPublicKey branch: {}                            ✓ (no check)
   → resolve_yields_for(K, forged_response)             ✓ (yield Y resolved)

4. User receives forged_response, decrypts it, and uses the attacker-known scalar
   as their app-private key.  Attacker can now sign on behalf of the user.
```

- `app_public_key`: the ephemeral public key for the CKD request. Two formats are supported:
  - **Privately verifiable** (legacy): a single G1 point, e.g. `"bls12381g1:<base58>"` or `{"AppPublicKey": "bls12381g1:<base58>"}`.
  - **Publicly verifiable**: a pair of points `(pk1, pk2) = (a·G1, a·G2)`, passed as `{"AppPublicKeyPV": {"pk1": "bls12381g1:<base58>", "pk2": "bls12381g2:<base58>"}}`. This allows anyone to verify the encrypted result on-chain without the app's secret key.
