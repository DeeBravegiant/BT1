### Title
FIFO Mempool Sibling Transaction Permanently Stranded After Rejection — (`crates/apollo_mempool/src/fifo_transaction_queue.rs`, `crates/apollo_mempool/src/mempool.rs`)

---

### Summary

In `Mempool::commit_block` (Echonet/FIFO mode), `rewind_txs` unconditionally calls `staged_txs.clear()` before `remove_rejected_txs` runs. When the lowest-nonce staged transaction for an address is rejected, `collect_txs_to_rewind` excludes that address from rewind, so its sibling transactions are neither rewound to the queue nor removed from the pool. `remove_rejected_txs` then calls `tx_queue.remove_by_address`, but `staged_txs` is already empty and the sibling was never in `queue` (it was moved to `staged_txs` during `get_txs`). The sibling transaction is permanently stranded in `tx_pool` with no queue entry and no recovery path.

---

### Finding Description

**Step-by-step trace:**

1. Address A submits T0 (nonce=0) and T1 (nonce=1). In FIFO mode, both are inserted directly into `queue` via `add_tx_inner`. [1](#0-0) 

2. `get_txs` pops both from `queue` and pushes them into `staged_txs`. [2](#0-1) 

3. `commit_block` is called with `address_to_nonce = {}` (neither T0 nor T1 committed) and `rejected_tx_hashes = {hash(T0)}`. [3](#0-2) 

4. `rewind_txs` is called first. Inside `collect_txs_to_rewind`, address A has no committed nonces, so the else-branch fires: the first (lowest-nonce) staged tx for A is T0, which is in `rejected_tx_hashes`, so the predicate `!rejected_tx_hashes.contains(&first_tx.tx_hash)` evaluates to `false`. Address A is **excluded from rewind**. [4](#0-3) 

5. Because no addresses are rewound, `txs_to_rewind` is empty. `rewind_txs` then unconditionally calls `self.staged_txs.clear()`, destroying the only reference to T1 in the queue layer. [5](#0-4) 

6. `remove_rejected_txs` runs next. For hash(T0): it removes T0 from `tx_pool`, then calls `tx_queue.remove_by_address(A)`. [6](#0-5) 

7. Inside `remove_by_address`, both `queue.retain` and `staged_txs.retain` are called. T1 is not in `queue` (it was moved to `staged_txs` during `get_txs`) and `staged_txs` is already empty (cleared in step 5). Nothing is removed. T1 remains in `tx_pool` with no queue entry. [7](#0-6) 

**After `commit_block`:** T0 is removed from `tx_pool`. T1 is in `tx_pool` but absent from `queue` and `staged_txs`. In FIFO mode, `get_txs` only serves transactions via `pop_ready_chunk` from `queue`. There is no mechanism to re-enqueue a pooled transaction after `commit_block` in FIFO mode. T1 is permanently excluded from all future proposals.

---

### Impact Explanation

A valid transaction (T1) is permanently stranded in `tx_pool` and silently excluded from all future block proposals. From the user's perspective, the transaction was submitted, accepted by the mempool, and returned by `get_txs`, but it is never sequenced and never explicitly rejected — it simply disappears from the proposal pipeline. This matches the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The trigger requires two conditions:
1. An address has two or more transactions staged simultaneously (both returned by `get_txs` in the same proposal round).
2. The lowest-nonce staged transaction for that address is rejected during block building (e.g., execution failure, insufficient balance at execution time, or any blockifier-level rejection).

In Echonet mode, transactions are replayed from a source chain. Rejections during block building are a normal operational event. Any user who submits two sequential transactions where the first fails execution will trigger this silently. No privileged access is required.

---

### Recommendation

The root cause is that `staged_txs.clear()` in `rewind_txs` destroys the queue-layer record of sibling transactions before `remove_rejected_txs` has a chance to clean them up. Two complementary fixes:

1. **In `rewind_txs`**: Before clearing `staged_txs`, collect the hashes of all staged transactions that are *not* being rewound and *not* in `rejected_tx_hashes`. These are the orphaned siblings. Return them (or pass them) so the caller can remove them from `tx_pool` as well.

2. **In `remove_rejected_txs` / `commit_block`**: After `remove_by_address` is called for a rejected tx's address, also remove from `tx_pool` any remaining staged siblings for that address that were not committed and not rewound.

Alternatively, `rewind_txs` could defer `staged_txs.clear()` until after `remove_rejected_txs` has completed, so that `remove_by_address` can still find and account for the siblings in `staged_txs`.

---

### Proof of Concept

```
// Echonet/FIFO mode mempool integration test
1. add_tx(A, nonce=0, hash=H0)   // T0 admitted, inserted to queue
2. add_tx(A, nonce=1, hash=H1)   // T1 admitted, inserted to queue
3. get_txs(2)                    // Both T0, T1 moved from queue → staged_txs
4. commit_block(
       address_to_nonce = {},    // neither committed
       rejected_tx_hashes = {H0} // T0 rejected
   )
5. assert tx_pool.contains(H1)   // T1 still in pool  ← TRUE (bug)
6. get_txs(10)                   // returns []         ← T1 never proposed again
7. assert queue_snapshot() does not contain H1  ← TRUE (stranded)
```

    fn remove_by_address(&mut self, address: ContractAddress) -> bool {
        let len_before = self.queue.len() + self.staged_txs.len();
        self.queue.retain(|tx| tx.tx_reference.address != address);
        self.staged_txs.retain(|tx| tx.tx_reference.address != address);
        let len_after = self.queue.len() + self.staged_txs.len();
        len_before != len_after
    }
