Audit Report

## Title
Zero-initialized `rsETHPrice` Causes DoS on Public `updateRSETHPrice()` When `pricePercentageLimit > 0` — (`contracts/LRTOracle.sol`)

## Summary
When `rsETHPrice` and `highestRsethPrice` are both `0` (initial state post-deployment) and `rsethSupply > 0`, the guard at line 224 seeds `highestRsethPrice = 0`. Any subsequent non-zero `newRsETHPrice` then trivially exceeds the threshold check (`priceDifference > pricePercentageLimit.mulWad(0)` reduces to `newRsETHPrice > 0`), causing every non-manager call to `updateRSETHPrice()` to revert with `PriceAboveDailyThreshold`. The protocol's `rsETHPrice` remains `0` until a manager bootstraps it via `updateRSETHPriceAsManager()`, during which time any protocol component relying on `rsETHPrice` for share math operates on a zero price.

## Finding Description

**Root cause — lines 224–226:** [1](#0-0) 

When `highestRsethPrice == 0`, the guard unconditionally assigns `highestRsethPrice = rsETHPrice`. If `rsETHPrice` is also `0` (never bootstrapped), `highestRsethPrice` remains `0`.

**Threshold check — lines 252–257:** [2](#0-1) 

`pricePercentageLimit.mulWad(0) == 0`, so `priceDifference > 0` is always true for any positive `newRsETHPrice`. `isPriceIncreaseOffLimit` is always `true`.

**Revert for non-managers — lines 263–265:** [3](#0-2) 

Non-manager callers always revert. The public entry point `updateRSETHPrice()` is inaccessible.

**Execution trace for the vulnerable state:**

| Variable | Value |
|---|---|
| `rsETHPrice` | `0` (never set) |
| `highestRsethPrice` | `0` (never set) |
| `rsethSupply` | `> 0` |
| `pricePercentageLimit` | `> 0` |

1. `rsethSupply > 0` → skip early-return at line 218. [4](#0-3) 
2. `highestRsethPrice == 0` → `highestRsethPrice = rsETHPrice = 0`. [1](#0-0) 
3. `previousTVL = rsethSupply.mulWad(0) = 0`. [5](#0-4) 
4. `totalETHInProtocol > 0` → fee computed, `newRsETHPrice > 0`. [6](#0-5) 
5. `priceDifference = newRsETHPrice`; `pricePercentageLimit.mulWad(0) = 0`; `isPriceIncreaseOffLimit = true`. [7](#0-6) 
6. Non-manager → revert `PriceAboveDailyThreshold()`. [8](#0-7) 

## Impact Explanation

**Medium. Temporary freezing of funds.**

Until the manager calls `updateRSETHPriceAsManager()` to bootstrap the price, `rsETHPrice` remains `0`. Any protocol component reading `rsETHPrice` for deposit/withdrawal share math operates on a zero price, causing incorrect accounting. Public keepers and bots cannot fulfill their role of keeping the price fresh. The freeze is temporary and manager-recoverable — one call to `updateRSETHPriceAsManager()` sets `rsETHPrice > 0` and `highestRsethPrice > 0`, after which non-manager calls succeed normally. [9](#0-8) 

## Likelihood Explanation

Low-to-Medium. The vulnerable state arises from a natural initialization-ordering issue requiring no attacker action:
1. Protocol is deployed; `rsETHPrice` and `highestRsethPrice` default to `0`.
2. Users deposit, making `rsethSupply > 0`.
3. Admin calls `setPricePercentageLimit` with a non-zero value. [10](#0-9) 
4. Manager has not yet called `updateRSETHPriceAsManager()` to bootstrap the price.

The window closes as soon as the manager bootstraps the price. No privileged access is required to trigger the bug; privileged access is only required to recover from it.

## Recommendation

In the `highestRsethPrice == 0` guard, do not copy `rsETHPrice` when it is also `0`. After computing `newRsETHPrice`, add a fallback to seed `highestRsethPrice` from the first real price:

```solidity
if (highestRsethPrice == 0) {
    if (rsETHPrice > 0) {
        highestRsethPrice = rsETHPrice;
    }
    // else: defer to newRsETHPrice below
}

// ... compute newRsETHPrice ...

if (highestRsethPrice == 0) {
    highestRsethPrice = newRsETHPrice; // bootstrap from first real price
}
```

This ensures the threshold comparison is always against a meaningful non-zero baseline.

## Proof of Concept

```solidity
// Foundry test (local fork or mock)
// Preconditions: rsETHPrice = 0, highestRsethPrice = 0,
//                pricePercentageLimit = 1e16, rsethSupply > 0

// Step 1: Non-manager call reverts
vm.prank(nonManager);
vm.expectRevert(ILRTOracle.PriceAboveDailyThreshold.selector);
lrtOracle.updateRSETHPrice();

// Step 2: Manager bootstraps the price
vm.prank(manager);
lrtOracle.updateRSETHPriceAsManager(); // succeeds, sets rsETHPrice > 0

// Step 3: Non-manager can now call successfully
vm.prank(nonManager);
lrtOracle.updateRSETHPrice(); // succeeds
```
