Audit Report

## Title
Scaled-to-native rounding deficit in `_executeSwap` causes permanent `binTotals.scaledToken1` over-accounting, leading to LP insolvency — (`metric-core/contracts/MetricOmmPool.sol`)

## Summary
In `_executeSwap`, `binTotals.scaledToken1` is decremented by the full raw `amountOutScaled` before the scaled-to-native conversion. `deltasScaledToExternal` applies `SignedMath.ceilDiv` to the negative delta, which for a negative numerator and positive denominator rounds toward zero (i.e., truncates the magnitude), so the pool transfers fewer native tokens than the scaled deduction implies. The remainder (`amountOutScaled % TOKEN_1_SCALE_MULTIPLIER`) is permanently lost from the pool's real balance but remains charged against `binTotals.scaledToken1`, accumulating across every exact-input swap and eventually making `binTotals.scaledToken1 > balance1() * TOKEN_1_SCALE_MULTIPLIER`, rendering the pool insolvent for LPs and breaking `collectFees`.

## Finding Description

**Root cause — misaligned deduction and transfer:**

For an exact-input `zeroForOne = true` swap, `_executeSwap` sets:

```solidity
amount1DeltaScaled = -int256(amountOutScaled);   // e.g. -1_500_000
``` [1](#0-0) 

Then immediately decrements `binTotals.scaledToken1` by the full raw value:

```solidity
binTotals.scaledToken1 = uint128(uint256(binTotals.scaledToken1) - uint256(-amount1DeltaScaled));
// decremented by 1_500_000
``` [2](#0-1) 

Afterward, `deltasScaledToExternal` converts to native units:

```solidity
deltaAmount1 = SignedMath.ceilDiv(scaledDeltaAmount1, TOKEN_1_SCALE_MULTIPLIER);
// ceilDiv(-1_500_000, 1_000_000) = -1
``` [3](#0-2) 

`SignedMath.ceilDiv` rounds toward positive infinity. For a negative numerator and positive denominator the exact result is negative, so ceiling means truncating the magnitude toward zero:

```solidity
int256 quotient = a / b;          // -1_500_000 / 1_000_000 = -1
int256 remainder = a % b;         // -500_000
if (remainder != 0 && (a ^ b) >= 0) { quotient += 1; }
// (a ^ b) is negative (different signs) → condition FALSE → no rounding up
// result = -1
``` [4](#0-3) 

The test suite confirms: `ceilDiv(-7, 3) = -2`, `ceilDiv(-1, 2) = 0`. [5](#0-4) 

The pool then transfers only 1 native token out:

```solidity
if (amount1Delta < 0) {
    transferToken1(recipient, uint256(-amount1Delta));  // transfers 1, not 1.5
}
``` [6](#0-5) 

**Per-swap deficit (example: `TOKEN_1_SCALE_MULTIPLIER = 10^6`):**

| Quantity | Value |
|---|---|
| `amountOutScaled` deducted from `binTotals.scaledToken1` | 1,500,000 |
| Native tokens transferred out | 1 |
| `balance1() * TOKEN_1_SCALE_MULTIPLIER` decrease | 1,000,000 |
| **Deficit per swap** | **500,000 scaled units** |

After N such swaps, `binTotals.scaledToken1` exceeds `balance1() * TOKEN_1_SCALE_MULTIPLIER` by up to `N * (TOKEN_1_SCALE_MULTIPLIER - 1)` scaled units.

**`collectFees` revert path:**

The surplus computation at lines 387–388 is a checked subtraction (outside the `unchecked` block that begins at line 390):

```solidity
uint256 surplus1Scaled =
  balance1() * TOKEN_1_SCALE_MULTIPLIER - uint256(binTotals.scaledToken1) - notionalFee1AmountScaled;
``` [7](#0-6) 

Once the deficit exceeds any initial dust surplus, this underflows and reverts, permanently breaking fee collection.

**LP insolvency path:**

`removeLiquidity` computes each LP's entitlement from `binState.token1BalanceScaled` (which is consistent with `binTotals.scaledToken1`), then converts to native with `Math.Rounding.Floor` and calls `safeTransfer`: [8](#0-7) 

Because the sum of all `token1BalanceScaled` across bins equals `binTotals.scaledToken1`, which exceeds `balance1() * TOKEN_1_SCALE_MULTIPLIER`, the last LP(s) to withdraw will find the pool has insufficient token1, causing their `safeTransfer` to revert.

## Impact Explanation
**Pool insolvency / direct LP principal loss.** Every exact-input swap where `amountOutScaled % TOKEN_1_SCALE_MULTIPLIER != 0` permanently widens the gap between `binTotals.scaledToken1` and `balance1() * TOKEN_1_SCALE_MULTIPLIER`. For a USDC pool (`TOKEN_1_SCALE_MULTIPLIER = 10^6`), the maximum deficit per swap is just under 1 USDC. Over thousands of swaps the deficit grows to material amounts, and the last LPs to withdraw cannot recover their full principal. `collectFees` also becomes permanently broken once the deficit exceeds the initial dust surplus. This meets the pool insolvency and direct LP principal loss impact criteria.

## Likelihood Explanation
Any pool with `TOKEN_1_SCALE_MULTIPLIER > 1` (e.g., USDC/USDT as token1) is affected by every ordinary exact-input swap. No special attacker action is required — normal trading activity is sufficient. The condition `amountOutScaled % TOKEN_1_SCALE_MULTIPLIER != 0` is the common case for arbitrary swap sizes, as `amountOutScaled` is determined by continuous price math with no alignment guarantee.

## Recommendation
Align the `binTotals.scaledToken1` deduction with the actual scaled amount that leaves the pool. After computing `amount1DeltaExternal` (the rounded native amount), derive the true scaled deduction as `uint256(-amount1DeltaExternal) * TOKEN_1_SCALE_MULTIPLIER` and use that value — not the raw `amountOutScaled` — when updating `binTotals.scaledToken1`. The rounding remainder should remain in the pool's accounting (and in the bin's `token1BalanceScaled`) as unclaimable dust, preserving the invariant `balance1() * TOKEN_1_SCALE_MULTIPLIER >= binTotals.scaledToken1`.

## Proof of Concept

```solidity
// TOKEN_1_SCALE_MULTIPLIER = 1_000_000 (USDC-like)
// Suppose one exact-input swap produces amountOutScaled = 1_500_000

// _executeSwap, line 739:
binTotals.scaledToken1 -= 1_500_000;   // full raw deduction

// deltasScaledToExternal (line 613):
// ceilDiv(-1_500_000, 1_000_000) = -1  (rounds toward zero for negative/positive)
amount1Delta = -1;

// swap(), line 254:
transferToken1(recipient, 1);           // only 1 USDC leaves

// Invariant check:
// balance1() * 1_000_000 decreased by 1_000_000
// binTotals.scaledToken1 decreased by 1_500_000
// => binTotals.scaledToken1 now exceeds balance1() * TOKEN_1_SCALE_MULTIPLIER by 500_000

// After enough swaps:
// collectFees reverts at line 387-388 (checked underflow)
// Last LP's removeLiquidity safeTransfer reverts (insufficient balance)
```

Fuzz assertion to add after every swap:
```solidity
assert(balance1() * TOKEN_1_SCALE_MULTIPLIER >= uint256(binTotals.scaledToken1));
```
