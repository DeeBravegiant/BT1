Audit Report

## Title
Reward tokens permanently frozen when `totalKernelStaked` drops to zero mid-period — (`contracts/KERNEL/KernelDepositPool.sol`)

## Summary
`KernelDepositPool` guards `notifyRewardAmount` against zero stakers at period start but provides no equivalent protection once a reward period is active. When all stakers call `initiateWithdrawal` mid-period, `totalKernelStaked` immediately drops to zero, causing `rewardPerToken()` to freeze `rewardPerTokenStored` while `updatedAt` advances, permanently stranding all reward tokens emitted during the zero-staker interval with no recovery path.

## Finding Description
The `updateReward` modifier executes on every state-changing call:

```solidity
rewardPerTokenStored = rewardPerToken();
updatedAt = lastTimeRewardApplicable();
``` [1](#0-0) 

`rewardPerToken()` short-circuits when `totalKernelStaked == 0`, returning the stored value unchanged:

```solidity
if (totalKernelStaked == 0) {
    return rewardPerTokenStored;
}
``` [2](#0-1) 

`initiateWithdrawal` immediately decrements `totalKernelStaked` before the withdrawal delay elapses:

```solidity
balanceOf[msg.sender] -= _amount;
totalKernelStaked -= _amount;
``` [3](#0-2) 

If all stakers call `initiateWithdrawal` during an active reward window, `totalKernelStaked` reaches zero. From that point, every `updateReward` invocation freezes `rewardPerTokenStored` while advancing `updatedAt`. The reward tokens that should have been distributed during the zero-staker interval — `rewardRate × (zero-staker duration)` — are never credited to any account and accumulate as an untracked surplus.

The only existing guard is at `notifyRewardAmount` start:

```solidity
if (totalKernelStaked == 0) revert NoStakedTokens();
``` [4](#0-3) 

This prevents starting a period with zero stakers but provides no protection mid-period. When a new reward period is eventually started, the rollover calculation:

```solidity
uint256 remaining = (finishAt - block.timestamp) * rewardRate;
rewardRate = (receivedAmount + remaining) / duration;
``` [5](#0-4) 

only accounts for future undistributed rewards based on `rewardRate` and time remaining — it does not recover the tokens already silently discarded during the zero-staker gap. There is no admin rescue or sweep function in the contract.

Notably, the contract's own NatSpec acknowledges this behavior and states the mitigation is operational ("ensuring there are always some tokens staked... for the entire duration of the reward period"): [6](#0-5) 

However, this operational assumption is not enforced on-chain. Any staker can call `initiateWithdrawal` permissionlessly at any time, making the invariant unenforceable by the contract itself.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Reward tokens emitted during any interval where `totalKernelStaked == 0` are permanently locked in the contract with no recovery path. The magnitude of loss is `rewardRate × (duration of zero-staker period)`. This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation
**Low.** Triggering the loss requires all stakers to exit simultaneously during an active reward window. This is realistic in adversarial conditions such as a coordinated exit, a depeg event for the staking token, or a migration to a new contract version. The `initiateWithdrawal` path is fully permissionless and requires no admin action. A single large staker holding the majority of `totalKernelStaked` can trigger this unilaterally.

## Recommendation
Two complementary mitigations:

1. **Track unallocated rewards.** In the `updateReward` modifier, when `totalKernelStaked == 0`, accumulate the reward tokens that would have been emitted into a dedicated `unallocatedRewards` variable. Roll this surplus into the next `notifyRewardAmount` call.

2. **Add an admin rescue function.** Provide a `recoverUnallocatedRewards` function restricted to `DEFAULT_ADMIN_ROLE` that transfers any surplus reward tokens (`rewardsToken.balanceOf(address(this)) - totalTrackedRewards`) to a treasury address.

## Proof of Concept

```solidity
function test_KernelDepositPool_rewardsLost_noStakers() public {
    address admin = address(this);
    address user1 = address(0xa11ce);

    MockERC20 kernelToken = new MockERC20("KERNEL", "K", 18);
    MockERC20 rewardToken  = new MockERC20("Reward", "R", 18);

    KernelDepositPool pool = new KernelDepositPool();
    pool.initialize(admin, address(kernelToken), address(rewardToken));
    pool.setRewardsDuration(7 days);
    pool.setWithdrawalDelay(1);

    // User1 stakes
    kernelToken.mint(user1, 100 ether);
    vm.startPrank(user1);
    kernelToken.approve(address(pool), 100 ether);
    pool.stake(100 ether);
    vm.stopPrank();

    // Admin notifies rewards (stakers present → succeeds)
    rewardToken.mint(admin, 1 ether);
    rewardToken.approve(address(pool), 1 ether);
    pool.notifyRewardAmount(1 ether);

    // User1 immediately initiates withdrawal → totalKernelStaked = 0
    vm.prank(user1);
    pool.initiateWithdrawal(100 ether);

    // Reward period elapses with zero stakers
    skip(7 days);

    // User1 claims withdrawal and tries to collect rewards
    vm.prank(user1);
    pool.claimWithdrawal(1);

    vm.prank(user1);
    pool.getReward();

    // Nearly all rewards are stuck — only dust earned in the instant before withdrawal
    assertGt(rewardToken.balanceOf(address(pool)), 0.99 ether);
}
```
