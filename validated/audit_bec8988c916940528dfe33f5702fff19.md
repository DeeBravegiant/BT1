### Title
Unvalidated `ProposalInit.builder` Field Allows Proposer to Inject Arbitrary `sequencer_address` into Block Hash and Syscall Results — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`ProposalInit.builder` is a proposer-supplied field that is accepted from the network and passed verbatim as `sequencer_address` into the block's `BlockInfo`. It is never checked in `is_proposal_init_valid` or anywhere else in the validation path. Because `sequencer_address` is hashed into `PartialBlockHashComponents` → `calculate_block_hash` and is returned to every contract that calls the `get_block_info` syscall, a malicious proposer can commit a block with an attacker-controlled `sequencer_address` that all honest validators will accept and sign.

### Finding Description

`ProposalInit` carries a `builder` field defined as "Address of the one who builds/sequences the block." [1](#0-0) 

`is_proposal_init_valid` validates `height`, `l1_da_mode`, `l2_gas_price_fri`, `starknet_version`, `version_constant_commitment`, `timestamp`, all L1 gas prices, and `fee_proposal_fri`. The `builder` field is absent from `ProposalInitValidation` and is never checked. [2](#0-1) 

`convert_to_sn_api_block_info` maps `init.builder` directly to `sequencer_address` with no guard: [3](#0-2) 

This `BlockInfo` is passed to the batcher in `initiate_validation`: [4](#0-3) 

`PartialBlockHashComponents::new` then binds `sequencer_address` into the partial block hash: [5](#0-4) 

`calculate_block_hash` chains the sequencer address into the final Poseidon hash: [6](#0-5) 

The same `sequencer_address` is returned to every contract that calls the `get_block_info` syscall during execution: [7](#0-6) 

The `ProposalFinMismatch` guard at line 244 of `validate_proposal.rs` does **not** protect against this: both the proposer and the validator derive the commitment from the same `builder` value, so the hashes agree and the check passes regardless of what address was injected. [8](#0-7) 

### Impact Explanation

A malicious proposer sets `ProposalInit.builder` to an attacker-controlled address. Every honest validator executes the block with `sequencer_address = attacker_address`, producing a `PartialBlockHash` that embeds the wrong sequencer. The `ProposalFinMismatch` check passes because both sides used the same injected value. The committed block hash is permanently wrong, and every contract that calls `get_block_info` during that block sees the attacker-controlled address as the sequencer — affecting any contract that uses `get_sequencer_address()` for access control, fee routing, or staking logic.

This matches: **Critical — Wrong state/syscall result from blockifier/syscall/execution logic for accepted input.**

### Likelihood Explanation

Requires the attacker to be the designated proposer for a round. In a BFT consensus system tolerating up to `f` Byzantine validators, a malicious committee member can be the proposer for any round assigned to them. No additional privileges beyond being a scheduled proposer are needed; the attack is a single-field substitution in the `ProposalInit` message.

### Recommendation

Add `builder` (expected sequencer address) to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`:

```rust
// In ProposalInitValidation:
pub builder: ContractAddress,

// In is_proposal_init_valid:
if init_proposed.builder != proposal_init_validation.builder {
    return Err(ValidateProposalError::InvalidProposalInit(
        init_proposed.clone(),
        proposal_init_validation.clone(),
        format!(
            "builder mismatch: expected={:?}, proposed={:?}",
            proposal_init_validation.builder, init_proposed.builder
        ),
    ));
}
```

The expected `builder` value should be sourced from the node's own local configuration (the same `builder_address` used in `ProposalBuildArguments`), not from the network. [9](#0-8) 

### Proof of Concept

1. Malicious proposer constructs `ProposalInit { builder: attacker_address, … }` for their assigned round.
2. Honest validator receives it; `is_proposal_init_valid` checks every field except `builder` — passes.
3. `initiate_validation` calls `convert_to_sn_api_block_info(init)` → `sequencer_address = attacker_address`.
4. Batcher executes the block; all `get_block_info` syscalls return `sequencer_address = attacker_address`.
5. `PartialBlockHashComponents` embeds `attacker_address`; `calculate_block_hash` produces a hash over it.
6. Proposer's `ProposalFin.proposal_commitment` was computed with the same `attacker_address`, so `built_block == received_fin.proposal_commitment` — `ProposalFinMismatch` does not fire.
7. Consensus reaches decision; block is committed with permanently wrong `sequencer_address` in the block hash and in all execution receipts for that block.
