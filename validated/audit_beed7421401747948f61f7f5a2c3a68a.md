### Title
Stale Pre-Resharing Measurement Vote Counted in Post-Resharing Threshold Check — (`crates/contract/src/tee/measurements.rs`, `crates/contract/src/lib.rs`)

---

### Summary

`MeasurementVotes::count_votes` counts every entry in `vote_by_account` without filtering by the current participant set. The cleanup of stale votes (`clean_tee_status`) is dispatched as a **detached promise** after resharing completes, creating a window of at least one NEAR block where a removed participant's vote persists and is counted. An attacker controlling T-1 participants in the new set (strictly below the signing threshold) can exploit this window to reach the threshold for `vote_add_os_measurement` using the stale vote as the T-th vote.

---

### Finding Description

**`MeasurementVotes::count_votes` has no participant-set filter.** [1](#0-0) 

It counts every `vote_by_account` entry matching the action, regardless of whether the voter is still a participant.

**`vote_add_os_measurement` authenticates the caller against the current set, then calls `count_votes` which includes stale entries.** [2](#0-1) 

The `AuthenticatedParticipantId::new(threshold_parameters.participants())?` call on line 1512 correctly rejects the removed participant from casting a new vote, but it does nothing to remove their already-stored vote from `vote_by_account`.

**The cleanup is deferred via a detached promise.** [3](#0-2) 

`clean_tee_status` is spawned with `.detach()`, meaning it executes as a separate receipt in a subsequent block. Until that receipt is processed, `measurement_votes` still contains the removed participant's entry.

**`clean_non_participant_votes` correctly removes stale votes — but only when it runs.** [4](#0-3) 

The developers are aware of the stale-vote concern (the test at lines 1496–1545 documents it for `CodeHashesVotes`), but the fix is asynchronous and leaves a window.

**Attack sequence (N=4, T=3):**

1. Attacker controls Byzantine participant B (P0) and T-1=2 participants in the new set (P1, P2). P1 and P2 are below the signing threshold T=3.
2. B votes for malicious `ContractExpectedMeasurements` M → `vote_by_account[ParticipantId(0)] = Add(M)`, count = 1.
3. Resharing completes, removing B. New set: {P1, P2, P3}. `clean_tee_status` receipt is queued but not yet processed.
4. In the same or next block, P1 calls `vote_add_os_measurement(M)` → count = 2 (B's stale + P1's fresh).
5. P2 calls `vote_add_os_measurement(M)` → count = 3 ≥ T=3 → `add_measurement(M)` executes → M is added to `AllowedMeasurements`.

`AuthenticatedParticipantId` wraps a monotonically-assigned `ParticipantId(u32)`: [5](#0-4) 

Removed participants' IDs are never reused, so B's stale entry `ParticipantId(0)` is distinct from any new participant's key and is counted alongside fresh votes by `count_votes`.

---

### Impact Explanation

Once M is in `AllowedMeasurements`, a node running attacker-controlled OS code whose TDX measurements match M will pass `submit_participant_info` attestation verification. That node can then be admitted as a participant and receive MPC key shares, enabling unauthorized access to signing capability and secret material. This matches the Critical impact scope: bypass of threshold-signature requirements and unauthorized access to MPC key shares. [6](#0-5) 

---

### Likelihood Explanation

The attacker must control T-1 participants in the new set (below the signing threshold). For small networks (N=4, T=3) this means controlling 2 participants; for larger networks the bar rises. The timing window is at least one NEAR block (~1 second), during which T-1 transactions must be submitted — feasible since NEAR blocks accommodate many transactions. The TEE attestation requirement for each participant is an operational barrier but not a code-level guard against this attack.

---

### Recommendation

Replace the deferred cleanup with an **eager, synchronous filter** at vote-counting time. In `vote_add_os_measurement`, pass the current participant set into `count_votes` and filter `vote_by_account` to only count entries whose `ParticipantId` is still in the active set:

```rust
fn count_votes_for_participants(&self, action: &MeasurementVoteAction, participants: &Participants) -> u64 {
    u64::try_from(
        self.vote_by_account
            .iter()
            .filter(|(pid, a)| *a == action && participants.is_participant_given_participant_id(&pid.get()))
            .count(),
    ).expect("participant count should not overflow u64")
}
```

This eliminates the race window entirely. The deferred `clean_tee_status` can remain for storage hygiene but must not be the sole correctness guard.

---

### Proof of Concept

```rust
#[test]
fn stale_measurement_vote_counted_in_window() {
    // N=4, T=3. Byzantine participant is P0.
    let mut all_participants = Participants::new();
    let mut account_ids = Vec::new();
    for i in 0..4 { let (a, info) = gen_participant(i); account_ids.push(a.clone()); all_participants.insert(a, info).unwrap(); }

    let mut tee_state = TeeState::default();
    let malicious_m = make_measurement(0xDE);

    // Step 1: Byzantine P0 votes for malicious measurement before resharing.
    {
        let mut ctx = VMContextBuilder::new(); ctx.signer_account_id(account_ids[0].clone()); testing_env!(ctx.build());
        let auth = AuthenticatedParticipantId::new(&all_participants).unwrap();
        tee_state.measurement_votes.vote(MeasurementVoteAction::Add(malicious_m.clone()), &auth);
    }

    // Step 2: Resharing removes P0. New set: {P1, P2, P3}, T=3.
    // clean_tee_status NOT called yet (simulating the detached-promise window).
    let new_participants = all_participants.subset(1..4);

    // Step 3: P1 votes (attacker-controlled, below-threshold participant).
    let votes = {
        let mut ctx = VMContextBuilder::new(); ctx.signer_account_id(account_ids[1].clone()); testing_env!(ctx.build());
        let auth = AuthenticatedParticipantId::new(&new_participants).unwrap();
        tee_state.measurement_votes.vote(MeasurementVoteAction::Add(malicious_m.clone()), &auth)
    };
    assert_eq!(votes, 2); // P0 stale + P1 fresh

    // Step 4: P2 votes — count reaches T=3.
    let votes = {
        let mut ctx = VMContextBuilder::new(); ctx.signer_account_id(account_ids[2].clone()); testing_env!(ctx.build());
        let auth = AuthenticatedParticipantId::new(&new_participants).unwrap();
        tee_state.measurement_votes.vote(MeasurementVoteAction::Add(malicious_m.clone()), &auth)
    };
    assert_eq!(votes, 3); // threshold met with only T-1=2 legitimate votes

    // Malicious measurement would be added to AllowedMeasurements.
    if votes >= 3 { tee_state.add_measurement(malicious_m.clone()); }
    assert!(tee_state.get_allowed_measurements().contains(&malicious_m),
        "malicious measurement added with only T-1 legitimate votes");
}

    pub fn vote_add_os_measurement(
        &mut self,
        measurement: ContractExpectedMeasurements,
    ) -> Result<(), Error> {
        log!(
            "vote_add_os_measurement: signer={}, measurement={:?}",
            env::signer_account_id(),
            measurement,
        );
        self.voter_or_panic();

        let threshold_parameters = self.protocol_state.threshold_parameters_or_panic();

        let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
        let action = MeasurementVoteAction::Add(measurement.clone());
        let votes = self.tee_state.vote_measurement(action, &participant);

        if votes >= self.threshold()?.value() {
            let added = self.tee_state.add_measurement(measurement);
            log!("OS measurement add result: {}", added);
        }

        Ok(())
    }

pub struct AuthenticatedParticipantId(ParticipantId);
impl AuthenticatedParticipantId {
    pub fn get(&self) -> ParticipantId {
        self.0
    }

    pub fn new(participants: &Participants) -> Result<Self, Error> {
        let signer = env::signer_account_id();
        participants
            .participants()
            .iter()
            .find(|(a_id, _, _)| *a_id == signer)
            .map(|(_, p_id, _)| AuthenticatedParticipantId(*p_id))
            .ok_or_else(|| InvalidState::NotParticipant { account_id: signer }.into())
    }
