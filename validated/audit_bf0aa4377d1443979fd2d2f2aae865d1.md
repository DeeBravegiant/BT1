Audit Report

## Title
Stale `rsETHPrice` in `instantWithdrawal` Bypasses Downside-Protection Pause, Enabling Over-Redemption After Slashing — (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTOracle.rsETHPrice` is a stored state variable updated only when `updateRSETHPrice()` is explicitly called. `LRTWithdrawalManager.instantWithdrawal()` reads this value via `getExpectedAssetAmount()` without first invoking `updateRSETHPrice()`. After a slashing event reduces the true rsETH/ETH rate, an attacker can call `instantWithdrawal()` before the price is refreshed, burning rsETH at the stale (inflated) rate and redeeming more assets from the unstaking vault than the current rate entitles them to. Critically, the downside-protection pause logic inside `_updateRsETHPrice()` — which would halt the protocol on a large price drop — is never triggered because `updateRSETHPrice()` is never called.

## Finding Description

`rsETHPrice` is a plain storage variable in `LRTOracle`, written only inside `_updateRsETHPrice()` and exposed via `updateRSETHPrice()` (public, callable by anyone):

```solidity
// LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
// LRTOracle.sol L313
rsETHPrice = newRsETHPrice;
```

`getExpectedAssetAmount()` reads this stored value directly without any freshness guarantee:

```solidity
// LRTWithdrawalManager.sol L593
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

`instantWithdrawal()` calls `getExpectedAssetAmount()` before burning rsETH, with no prior call to `updateRSETHPrice()`:

```solidity
// LRTWithdrawalManager.sol L228-235
uint256 assetAmountUnlocked = getExpectedAssetAmount(asset, rsETHUnstaked);
IRSETH(lrtConfig.rsETH()).burnFrom(address(msg.sender), rsETHUnstaked);
...
if (assetAmountUnlocked > unstakingVault.getAssetsAvailableForInstantWithdrawal(asset)) {
    revert CantInstantWithdrawMoreThanAvailable();
}
unstakingVault.redeem(asset, assetAmountUnlocked);
```

The downside-protection logic in `_updateRsETHPrice()` (L270-281) computes `newRsETHPrice`, compares it to `highestRsethPrice`, and pauses `lrtDepositPool`, `withdrawalManager`, and the oracle if the drop exceeds `pricePercentageLimit`. Because `updateRSETHPrice()` is never called inside `instantWithdrawal()`, this entire guard is skipped. An attacker who observes a slashing event on-chain can race to call `instantWithdrawal()` before any operator or keeper refreshes the price, redeeming assets at the pre-slash rate.

The `getAssetsAvailableForInstantWithdrawal` cap limits the size of a single transaction but does not close the rate gap; the attacker can repeat calls across multiple transactions within the same stale-price window.

## Impact Explanation

**Direct theft of user funds — Critical.**

After a slashing event that reduces the true rsETH/ETH rate (e.g., from 1.05 ETH to 1.00 ETH), an attacker burns rsETH at the stale 1.05 rate and redeems 5% more assets per rsETH than the current rate justifies. The excess assets are drawn from the unstaking vault, which holds funds earmarked for users who have already queued withdrawals and burned their rsETH. Those users' payouts are reduced by the amount the attacker over-redeems. This constitutes direct theft of at-rest user funds, not merely unclaimed yield.

## Likelihood Explanation

Slashing events in EigenLayer are infrequent but are an explicitly anticipated protocol risk (evidenced by the downside-protection pause mechanism). The price-update window can span hours between operator calls. Any unprivileged external user can trigger `instantWithdrawal()` without calling `updateRSETHPrice()` first. An attacker monitoring EigenLayer slashing events on-chain can act within the same block or within minutes, well before a keeper refreshes the price. No special role, leaked key, or governance action is required. Likelihood is **Medium** (requires a slashing event), but the combination of realistic precondition and zero-privilege exploit path makes this a valid Critical finding.

## Recommendation

Call `updateRSETHPrice()` (or an internal equivalent that does not revert when paused) at the top of both `instantWithdrawal()` and `initiateWithdrawal()` before reading `lrtOracle.rsETHPrice()`:

```solidity
// At the start of instantWithdrawal and initiateWithdrawal:
ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice();
```

This ensures the downside-protection pause fires before any withdrawal arithmetic executes, and that the rate used for `assetAmountUnlocked` always reflects the latest on-chain state.

## Proof of Concept

1. EigenLayer slashing event reduces total ETH in protocol; true rsETH/ETH rate drops from 1.05 ETH to 1.00 ETH.
2. `rsETHPrice` stored in `LRTOracle` is still 1.05 ETH — no one has called `updateRSETHPrice()`.
3. Attacker calls `instantWithdrawal(ETH, 100e18, "")`.
4. `getExpectedAssetAmount` computes `100e18 * 1.05e18 / 1e18 = 105 ETH`.
5. Attacker's 100 rsETH is burned; `unstakingVault.redeem(ETH, 105e18)` is called.
6. Attacker receives 105 ETH (minus fee) for rsETH worth only 100 ETH at the current rate.
7. The 5 ETH excess is drawn from the unstaking vault, reducing the assets available to users with queued withdrawals.
8. The downside-protection pause in `_updateRsETHPrice()` is never triggered; the protocol remains open for further exploitation until a keeper manually calls `updateRSETHPrice()`.
9. Attacker repeats until `getAssetsAvailableForInstantWithdrawal` is exhausted or the price is refreshed.

**Foundry fork test outline:**
- Fork mainnet/testnet at a block where `rsETHPrice` is set.
- Directly manipulate `rsETHPrice` storage slot to simulate a post-slash stale value (or mock `LRTOracle` to return a higher price than `_updateRsETHPrice()` would compute).
- Call `instantWithdrawal` as an unprivileged address holding rsETH.
- Assert `assetAmountUnlocked > amount * trueCurrentPrice / assetPrice`, confirming over-redemption.
- Assert `withdrawalManager.paused() == false`, confirming the downside-protection pause was not triggered.
