Audit Report

## Title
L2 Price Providers Accept Stale Pre-Outage Oracle Prices After Sequencer Restart - (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary
`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly designed for L2 deployment (evidenced by their `FUTURE_TOLERANCE` immutable for sequencer clock skew) but neither checks a Chainlink sequencer uptime feed before accepting oracle prices. When the L2 sequencer goes offline, oracle prices freeze on-chain. If the outage duration is shorter than `MAX_TIME_DELTA` (up to 7 days), the staleness check passes on restart and an attacker can execute swaps against the stale pre-outage price before any oracle keeper pushes a fresh update, causing direct LP principal loss.

## Finding Description
Both contracts implement `_isStale` which compares `refTime` against `block.timestamp` using `MAX_TIME_DELTA`:

In `PriceProviderL2._getBidAndAskPrice()`, the only guard before accepting the oracle price is: [1](#0-0) 

In `ProtectedPriceProviderL2._computeBidAsk()`, the identical pattern: [2](#0-1) 

`MAX_TIME_DELTA` is bounded only between `> 0` and `<= 7 days`: [3](#0-2) 

Neither constructor accepts a `_sequencerUptimeFeed` parameter: [4](#0-3) 

By contrast, the registry confirms that other contracts in the same system (a Chainlink-hybrid L2 provider and `ChainlinkVerifierL2`) accept `_sequencerUptimeFeed` and expose `GRACE_PERIOD`: [5](#0-4) 

The exploit path: when the L2 sequencer goes offline, no transactions can be processed, so oracle keepers cannot push price updates. The oracle's stored `refTime` remains at the pre-outage timestamp. When the sequencer restarts, `_isStale(preOutageRefTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)` returns `false` if the outage duration is less than `MAX_TIME_DELTA`. An attacker immediately calls `swap()` before any keeper pushes a fresh price, executing against the stale pre-outage bid/ask while the true market price has moved.

## Impact Explanation
This is bad-price execution leading to direct LP principal loss. The pool quotes bid/ask derived from a stale pre-outage oracle price. The attacker buys the underpriced token (or sells the overpriced one) at the stale rate and immediately arbitrages on an external DEX. LPs receive the overpriced token and give up the underpriced one; the pool's token balances no longer cover fair-value LP claims at current market prices. This satisfies the "bad-price execution" and "pool insolvency" impact gates. Severity is High given the direct, quantifiable LP loss proportional to price movement during the outage.

## Likelihood Explanation
Both contracts are explicitly L2-targeted (they carry `FUTURE_TOLERANCE` for sequencer clock skew). L2 sequencer outages are documented historical events (Arbitrum had a ~7-hour outage in 2022). The attack requires no special privilege — any EOA can call `swap()` immediately after sequencer restart. The vulnerability window is bounded only by how quickly oracle keepers push a fresh price after restart, which can be minutes. The attack is repeatable on every sequencer restart event.

## Recommendation
Add a Chainlink sequencer uptime check with a grace period to both `PriceProviderL2._getBidAndAskPrice()` and `ProtectedPriceProviderL2._computeBidAsk()`, mirroring the pattern already present in the Chainlink-hybrid L2 provider registered in the registry:

```solidity
// Add immutable to constructor:
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

// Add to _getBidAndAskPrice() / _computeBidAsk() before the staleness check:
if (address(sequencerUptimeFeed) != address(0)) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer != 0 || block.timestamp - startedAt < GRACE_PERIOD) {
        return (0, type(uint128).max); // fail closed
    }
}
```

## Proof of Concept
```
Setup:
  - Deploy PriceProviderL2 on Arbitrum fork with MAX_TIME_DELTA = 1 days
  - Oracle last updated at T=0 with price = 100 (refTime = T=0)

Attack:
  T=0:    Sequencer goes offline. Oracle price frozen at 100.
  T=4h:   True market price moves to 110 (10% up). No oracle update possible.
  T=4h:   Sequencer restarts.
  T=4h+ε: Attacker calls pool.swap(zeroForOne=true) before any oracle update.
           PriceProviderL2._isStale(T=0, T=4h, 1 day, FUTURE_TOLERANCE) → false (4h < 1 day).
           Pool quotes bid/ask around 100 (stale).
           Attacker buys token1 at price 100 when market is 110.
           Attacker sells on external DEX at 110 → 10% profit.
           LPs receive token0 worth 100 but gave up token1 worth 110.

Foundry fork test: fork Arbitrum at a block just after a sequencer restart,
mock the oracle to return a pre-outage refTime within MAX_TIME_DELTA,
call swap() as an unprivileged EOA, and assert the pool accepted the stale price.
```
