### Title
Out-of-Order Concurrent Coin State Application Corrupts Pool Wallet Singleton State - (`File: chia/wallet/wallet_node.py`)

### Summary
In `add_states_from_peer`, when processing coin state updates from an **untrusted peer** without a fork height (normal steady-state sync), up to 30 concurrent `validate_and_add` tasks are spawned. Because validation involves async round-trips back to the peer (header block fetches, Merkle proof requests), an adversarial peer can delay responses for early batches while answering later batches promptly, causing higher-height coin state batches to be committed to the DB before lower-height ones. This violates the strict height-ordering invariant that `_add_coin_states` and `PoolWallet.apply_state_transition` depend on, permanently corrupting the pool wallet's singleton state history.

### Finding Description

In `chia/wallet/wallet_node.py`, `add_states_from_peer` sorts all incoming coin states and splits them into batches of 10 (for untrusted peers). When `fork_height is None` (the normal non-reorg path), it fires up to 30 concurrent asyncio tasks: [1](#0-0) 

Each `validate_and_add` task:
1. Acquires `validation_semaphore` (limit 10)
2. Calls `_collect_valid_states`, which makes multiple async peer requests (header blocks, addition/removal Merkle proofs) via `validate_received_state_from_peer`
3. Acquires `db_wrapper.writer()` and calls `add_coin_states` [2](#0-1) 

The DB writer lock serializes writes but does **not** enforce ordering. Whichever task finishes validation first writes first. An adversarial peer controls the timing of responses to the wallet's validation requests, so it can make batch [11–20] complete before batch [1–10].

`_add_coin_states` checks that its own input is sorted: [3](#0-2) 

But this check is per-call. Each batch is individually sorted, so the check passes even when batch [11–20] is committed before batch [1–10].

For `PoolWallet`, `apply_state_transition` enforces a strict chain: the coin being spent must match the current tip: [4](#0-3) 

If batch [11–20] (containing singleton coin at height 15, child of height-10 coin) is committed first, `apply_state_transition` silently returns `False` because the wallet's tip is still the launcher coin. When batch [1–10] is later committed, the height-10 transition succeeds, but the height-15 transition is **never retried** — it is permanently lost. [5](#0-4) 

### Impact Explanation

The pool wallet's singleton state history becomes permanently incomplete. If the missed transition was a move to `FARMING_TO_POOL` (with a specific pool URL and payout address), the wallet incorrectly believes it is still in `LEAVING_POOL`. Consequences include:

- **Wrong payout address**: `update_pool_config` is called with stale state, writing the wrong `target_puzzle_hash` to the pool config file, redirecting farming rewards to the wrong destination.
- **Duplicate/invalid transactions**: `PoolWallet.new_peak` will keep attempting to submit the second travel transaction (to complete the leave) even though the singleton has already transitioned, wasting fees and potentially causing double-spend errors.
- **Missed reward claims**: The wallet's view of `tip_singleton_coin_id` is stale, so `claim_pool_rewards` constructs absorb spends against the wrong singleton version.

This matches the allowed High impact: *Corruption of pool membership state or singleton mutation with direct security impact*.

### Likelihood Explanation

Any node on the network can connect to a wallet as an untrusted peer. The wallet actively subscribes to puzzle hashes and coin IDs, so the peer legitimately receives and sends `CoinStateUpdate` messages. The peer controls the latency of its own responses to the wallet's header-block and Merkle-proof requests during `validate_received_state_from_peer`. By selectively delaying responses for early batches, the peer deterministically causes later batches to commit first. No keys, admin access, or cryptographic breaks are required.

### Recommendation

Replace the concurrent fan-out with sequential processing for untrusted peers, or introduce an ordering barrier that prevents batch N+1 from being committed until batch N has been committed. Concretely:

- After `asyncio.gather(*all_tasks)`, re-sort and re-apply any states that were written out of order, or
- Process untrusted batches sequentially (as trusted peers already do), accepting the performance trade-off, or
- After gathering all tasks, verify the DB-committed height sequence is monotone and roll back + retry any out-of-order writes.

### Proof of Concept

1. Attacker runs a full node and connects to a victim wallet as an untrusted peer.
2. Victim wallet has a `PoolWallet` singleton currently in `LEAVING_POOL` state (tip at height H, coin C₀).
3. Two blocks are farmed: block H+1 transitions singleton to `FARMING_TO_POOL` (coin C₁), block H+2 is an absorb (coin C₂).
4. Attacker sends a `CoinStateUpdate` containing both `CoinState(C₁, spent_height=H+2, created_height=H+1)` and `CoinState(C₀, spent_height=H+1, created_height=H)`.
5. After `sort_coin_states`, batch 1 = `[C₀ state]`, batch 2 = `[C₁ state]`.
6. Attacker delays the header-block response for batch 1's validation but answers batch 2's immediately.
7. Batch 2's `validate_and_add` task acquires the DB writer first and calls `apply_state_transition(C₁_spend, H+2)`. The wallet tip is C₀, but `new_state.coin` is C₁ (child of C₀), so `spent_coin_name != new_state.coin.name()` → returns `False`. Transition silently dropped.
8. Batch 1's task eventually completes, commits `apply_state_transition(C₀_spend, H+1)` successfully. Wallet tip advances to C₁.
9. The `FARMING_TO_POOL` transition (C₁ spend) is never re-applied. Wallet remains in `LEAVING_POOL` with stale pool config pointing to the old pool URL and payout address. [6](#0-5) [7](#0-6) [8](#0-7)

    async def apply_state_transition(
        self, new_state: CoinSpend, block_height: uint32, action_scope: WalletActionScope
    ) -> bool:
        """
        Updates the Pool state (including DB) with new singleton spends.
        The DB must be committed after calling this method. All validation should be done here. Returns True iff
        the spend is a valid transition spend for the singleton, False otherwise.
        """
        tip: tuple[uint32, CoinSpend] = await self.get_tip()
        tip_spend = tip[1]

        tip_coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(tip_spend)
        assert tip_coin is not None
        spent_coin_name: bytes32 = tip_coin.name()

        if spent_coin_name != new_state.coin.name():
            history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
            if new_state.coin.name() in [sp.coin.name() for _, sp in history]:
                self.log.info(f"Already have state transition: {new_state.coin.name().hex()}")
            else:
                self.log.warning(
                    f"Failed to apply state transition. tip: {tip_coin} new_state: {new_state} height {block_height}"
                )
            return False

        await self.wallet_state_manager.pool_store.add_spend(self.wallet_id, new_state, block_height)
        tip_spend = (await self.get_tip())[1]
        self.log.info(f"New PoolWallet singleton tip_coin: {tip_spend} farmed at height {block_height}")

        # If we have reached the target state, resets it to None. Loops back to get current state
        for _, added_spend in reversed(
            await self.wallet_state_manager.pool_store.get_spends_for_wallet(self.wallet_id)
        ):
            latest_state: PoolState | None = solution_to_pool_state(added_spend)
            if latest_state is not None:
                if self.target_state == latest_state:
                    self.target_state = None
                    self.next_transaction_fee = uint64(0)
                    self.next_tx_config = DEFAULT_TX_CONFIG
                break

        await self.update_pool_config(action_scope)
        return True
