Audit Report

## Title
`getExpectedAssetAmount` Returns Gross Amount Excluding `instantWithdrawalFee`, Overstating Actual User Proceeds - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`getExpectedAssetAmount` is documented as returning "Amount of underlying to receive" and is exposed on the `ILRTWithdrawalManager` interface, but it returns the gross oracle-derived amount without deducting `instantWithdrawalFee`. When `instantWithdrawalFee > 0`, `instantWithdrawal` transfers only `assetAmountUnlocked - fee` to the caller, meaning the view function systematically overstates actual proceeds by up to 10%. Any integrator or on-chain contract that uses `getExpectedAssetAmount` to preview instant-withdrawal output will receive a value that does not match what is transferred.

## Finding Description
`getExpectedAssetAmount` (L580–594) computes:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
```

No fee is deducted. `instantWithdrawal` (L228) calls this same function to obtain `assetAmountUnlocked`, then at L237–238 silently computes:

```solidity
uint256 fee = (assetAmountUnlocked * instantWithdrawalFee) / 10_000;
uint256 userAmount = assetAmountUnlocked - fee;
```

and transfers only `userAmount` to `msg.sender` at L250. `setInstantWithdrawalFee` (L372–374) permits fees up to 1000 bps (10%). The result is a persistent gap between the value returned by the public view function and the value actually transferred. Any on-chain contract that calls `getExpectedAssetAmount` to determine a minimum acceptable output and then calls `instantWithdrawal` will receive `fee`-percent less than the previewed amount, causing the downstream check to fail.

## Impact Explanation
This matches **Low — Contract fails to deliver promised returns, but doesn't lose value**. The function is part of the public `ILRTWithdrawalManager` interface, is documented as the canonical preview for withdrawal proceeds, and returns a value that is structurally higher than what `instantWithdrawal` transfers whenever `instantWithdrawalFee > 0`. No funds are permanently lost (the fee is redirected to a fee recipient), but the contract does not deliver the amount it promises to callers of the view function.

## Likelihood Explanation
`instantWithdrawalFee` is a configurable parameter actively managed by `onlyLRTManager`. Any non-zero fee setting — the normal operational state for a fee-bearing instant-withdrawal feature — triggers the discrepancy. The function is public and is the natural entry point for integrators to preview proceeds before executing `instantWithdrawal`. No attacker capability is required; the discrepancy is present for every ordinary user or integrating contract whenever the fee is non-zero.

## Recommendation
Either modify `getExpectedAssetAmount` to deduct `instantWithdrawalFee` when computing the expected user proceeds, or introduce a separate `getExpectedInstantWithdrawalAmount` function that returns the net amount after fee. The standard (queued) withdrawal path, which has no fee, can continue to use the gross amount. The NatSpec on `getExpectedAssetAmount` should also be updated to clarify which path it applies to.

## Proof of Concept
1. `instantWithdrawalFee` is set to 500 bps (5%) by the LRT manager via `setInstantWithdrawalFee(500)`.
2. An integrator (or on-chain contract) calls `getExpectedAssetAmount(stETH, 1e18)` → receives value `X`.
3. The integrator calls `instantWithdrawal(stETH, 1e18, "")` expecting to receive `X`.
4. Inside `instantWithdrawal`: `fee = X * 500 / 10_000 = X * 0.05`; `userAmount = X * 0.95` is transferred (L237–238, L250).
5. Any downstream logic asserting `received >= X` (e.g., a flash-loan repayment or slippage guard) fails because actual proceeds are `X * 0.95`.

Foundry test outline:
```solidity
// fork test
uint256 preview = withdrawalManager.getExpectedAssetAmount(stETH, 1e18);
uint256 balBefore = IERC20(stETH).balanceOf(address(this));
withdrawalManager.instantWithdrawal(stETH, 1e18, "");
uint256 received = IERC20(stETH).balanceOf(address(this)) - balBefore;
assertLt(received, preview); // passes when instantWithdrawalFee > 0
