### Title
Malicious Third-Party Harvester Can Unconditionally Redirect Farmer Block Rewards to Arbitrary Address - (File: `chia/farmer/farmer_api.py`)

### Summary
The CHIP-22 fee convention allows third-party harvesters to redirect the farmer's XCH block reward by setting `farmer_reward_address_override` in the `RespondSignatures` protocol message. The farmer applies this override unconditionally â€” no fee-threshold check is performed on the `RespondSignatures` path, and even on the `NewProofOfSpace` path the threshold violation only produces a log warning without blocking the override. Any connected harvester can redirect 100% of the farmer's block reward to an attacker-controlled address.

### Finding Description

**Bug class:** Reward diversion via incorrect/unvalidated recipient address â€” direct analog to the external report's "funds sent to wrong address with no recovery path."

**Root cause â€” `RespondSignatures` path (no threshold check at all):**

In `FarmerAPI._process_respond_signatures` (`chia/farmer/farmer_api.py` lines 916â€“919), the harvester-supplied `farmer_reward_address_override` from `RespondSignatures` is applied unconditionally:

```python

    farmer_reward_address = response.farmer_reward_address_override   # no check

This `farmer_reward_address` is then placed directly into `DeclareProofOfSpace` (line 929) and broadcast to the network, causing the block reward to be paid to the harvester-chosen address. [2](#0-1) 

**Root cause â€” `NewProofOfSpace` path (warning only, no enforcement):**

When `NewProofOfSpace.farmer_reward_address_override` is non-None, the farmer calls `notify_farmer_reward_taken_by_harvester_as_fee` (line 128â€“129): [3](#0-2) 

That function checks the fee quality against the threshold and emits a log warning if violated â€” but it **never returns an error, never rejects the proof, and never prevents the override from being applied**: [4](#0-3) 

**Protocol message definitions:**

`RespondSignatures.farmer_reward_address_override` is a plain optional field with no protocol-level constraint: [5](#0-4) 

`NewProofOfSpace.farmer_reward_address_override` is similarly unconstrained: [6](#0-5) 

### Impact Explanation

A malicious harvester redirects the farmer's XCH block reward (currently ~0.25 XCH per block) to any address it controls. The loss is permanent â€” once the block is accepted on-chain, the reward coin is unrecoverable. This matches the Critical impact category: *unauthorized reward diversion affecting XCH*.

### Likelihood Explanation

Any harvester connected to the farmer over the harvester protocol can exploit this. Third-party harvesters (GPU harvesters, DrPlotter, etc.) are a common production setup. No keys, no admin access, and no prior agreement are required â€” the attacker only needs an active harvester connection. The cleanest attack path requires zero warnings:

1. Send `NewProofOfSpace` with `farmer_reward_address_override = None` â†’ no warning logged.
2. Farmer sends `RequestSignatures` back to the harvester.
3. Harvester replies with `RespondSignatures` containing `farmer_reward_address_override = attacker_puzzle_hash`.
4. Farmer unconditionally sets `farmer_reward_address = attacker_puzzle_hash` and broadcasts `DeclareProofOfSpace`.
5. Block reward is paid to the attacker.

### Recommendation

In `_process_respond_signatures`, before applying `response.farmer_reward_address_override`, the farmer must verify that the fee quality for this proof meets the configured threshold. If the threshold is not met, the override must be **rejected** (not merely logged) and the farmer's own `farmer_target` must be used. Additionally, the `NewProofOfSpace` path should enforce the same check and abort processing (not just log) when the threshold is violated.
