Audit Report

## Title
Users Can Front-Run rsETH Price Decreases via `instantWithdrawal` to Avoid Losses, Shifting Burden to Honest Stakers - (File: contracts/LRTWithdrawalManager.sol)

## Summary
`LRTWithdrawalManager.instantWithdrawal()` redeems rsETH for underlying assets using the stored `rsETHPrice` from `LRTOracle`, which is a state variable updated only when `updateRSETHPrice()` is explicitly called. Because both `updateRSETHPrice()` and EigenLayer's `verifyStaleBalance()` are permissionless, an attacker can exit at a stale (pre-slash) price, then trigger the price drop themselves, extracting more assets than their rsETH is entitled to and shifting the shortfall onto remaining stakers.

## Finding Description

`instantWithdrawal` computes the payout via `getExpectedAssetAmount`:

```solidity
underlyingToReceive = amount * lrtOracle.rsETHPrice() / lrtOracle.getAssetPrice(asset);
``` [1](#0-0) 

`lrtOracle.rsETHPrice()` is a stored state variable in `LRTOracle`: [2](#0-1) 

It is only updated when `updateRSETHPrice()` is called, which is a public, permissionless function: [3](#0-2) 

The price is computed from `getEffectivePodShares()`, which reads EigenLayer's withdrawable shares — a value that is only updated on-chain when a checkpoint is completed (e.g., via EigenLayer's permissionless `verifyStaleBalance` for slashed validators): [4](#0-3) 

The `_updateRsETHPrice()` function does include a downside protection: if the price drop exceeds `pricePercentageLimit`, it pauses the protocol and returns without updating `rsETHPrice`: [5](#0-4) 

**However, this protection does not prevent the attack.** The attacker exits via `instantWithdrawal` *before* calling `updateRSETHPrice()`. The pause only triggers *when* `updateRSETHPrice()` is called — after the attacker has already redeemed at the stale price. For small price drops (within the limit), the price updates normally with no pause, and the attacker still extracts excess value.

The exploit sequence:
1. A beacon chain slashing event occurs. EigenLayer pod shares have not yet been updated (no checkpoint completed).
2. `rsETHPrice` in `LRTOracle` is stale — still reflects the pre-slash value.
3. Attacker calls `instantWithdrawal(asset, rsETHAmount, "")`, receiving assets computed at the inflated stale price.
4. Attacker calls EigenLayer's permissionless `verifyStaleBalance()` on the affected EigenPod, completing the checkpoint and reducing EigenLayer shares.
5. Attacker calls `LRTOracle.updateRSETHPrice()` — price drops. If the drop exceeds `pricePercentageLimit`, the protocol pauses, but the attacker has already exited.
6. Remaining rsETH holders now hold rsETH backed by fewer assets than the protocol's accounting reflects.

`instantWithdrawal` is accessible to any rsETH holder when `isInstantWithdrawalEnabled[asset]` is `true`: [6](#0-5) 

The `isInstantWithdrawalEnabled` flag is set by the manager as a normal operational configuration: [7](#0-6) 

## Impact Explanation

The attacker redeems rsETH at a price higher than the post-slash true value, extracting more underlying assets than they are entitled to. The shortfall is absorbed by remaining rsETH holders, whose rsETH is now backed by fewer assets than the protocol's accounting reflects. This constitutes **theft of unclaimed yield** — the attacker captures value that should have been distributed (or lost) proportionally across all holders. Impact: **High**.

## Likelihood Explanation

- Beacon chain slashing events are publicly observable via beacon chain APIs.
- EigenLayer's `verifyStaleBalance` is permissionless for slashed validators, meaning the attacker can trigger the balance update themselves after exiting.
- `LRTOracle.updateRSETHPrice()` is permissionless, so the attacker controls the timing of the price update.
- `instantWithdrawal` requires only holding rsETH and the feature being enabled for an asset — no privileged access.
- Generating beacon chain proofs for `verifyStaleBalance` requires off-chain infrastructure but is technically feasible and has been done by MEV searchers in practice.

Likelihood: **Medium**.

## Recommendation

1. **Force a price update at the start of `instantWithdrawal`**: Call `updateRSETHPrice()` (or an internal equivalent) before computing `assetAmountUnlocked`, ensuring the payout uses the current price.
2. **Alternatively, introduce a time-lock**: Require instant withdrawals to use a price that was set at least N blocks ago, preventing same-block exploitation.
3. **Restrict `updateRSETHPrice()` to authorized callers** (e.g., a keeper/manager role) to prevent attackers from controlling the timing of price updates after exiting.

## Proof of Concept

```
Setup:
  - isInstantWithdrawalEnabled[ETH] = true
  - LRTOracle.rsETHPrice = 1.05e18 (stale, pre-slash)
  - Kelp validator is slashed on beacon chain; EigenLayer pod shares not yet updated

Block N:
  Attacker holds 100e18 rsETH.
  Attacker calls LRTWithdrawalManager.instantWithdrawal(ETH, 100e18, "")
    → getExpectedAssetAmount: 100e18 * 1.05e18 / 1e18 = 105 ETH
    → Attacker receives 105 ETH (minus fee)

Block N (same tx or next):
  Attacker calls EigenPod.verifyStaleBalance(...) with beacon chain proof
    → EigenLayer records the slash; pod shares decrease
  Attacker calls LRTOracle.updateRSETHPrice()
    → _getTotalEthInProtocol() now reflects reduced shares
    → rsETHPrice drops to 1.03e18
    → If drop > pricePercentageLimit: protocol pauses (attacker already exited)
    → If drop ≤ pricePercentageLimit: price updates normally

Result:
  Attacker received 105 ETH for 100 rsETH.
  True value at new price: 100 * 1.03 = 103 ETH.
  Attacker extracted 2 ETH excess.
  Remaining rsETH holders absorb the 2 ETH shortfall.
```

Foundry fork test plan: fork mainnet, deploy with `isInstantWithdrawalEnabled[ETH] = true`, mock `lrtOracle.rsETHPrice()` to return a pre-slash value, mock `getEffectivePodShares()` to return a post-slash value after a state change, call `instantWithdrawal`, then call `updateRSETHPrice()`, and assert that the attacker received more ETH than `rsETHAmount * newPrice / 1e18`.
