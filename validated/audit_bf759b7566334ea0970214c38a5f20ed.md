### Title
`accept_requests` Flag Blocks `respond*` for Already-Queued In-Flight Requests When TEE Validation Fails — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `accept_requests: bool` field in `MpcContract` is set to `false` by `verify_tee()` when fewer than `threshold` participants hold valid TEE attestations. This flag is checked not only in user-facing submission functions (`sign`, `request_app_private_key`, `verify_foreign_transaction`) but also in all three node-facing `respond*` functions. As a result, sign/CKD/foreign-tx requests that were already accepted and queued (yield-resume promises already created and stored) before `accept_requests` became `false` cannot be responded to by MPC nodes. These in-flight requests are frozen until the NEAR yield-resume timeout (~200 blocks) fires and fails them.

---

### Finding Description

`MpcContract` stores the `accept_requests` flag: [1](#0-0) 

`verify_tee()`, callable by any single participant, sets this flag to `false` when the surviving valid-attestation count falls below the threshold: [2](#0-1) 

All three node-facing respond functions check this flag and reject the call if it is `false`:

`respond()`: [3](#0-2) 

`respond_ckd()`: [4](#0-3) 

`respond_verify_foreign_tx()`: [5](#0-4) 

These functions are the only path through which MPC nodes can fulfill already-queued requests stored in: [6](#0-5) 

When `accept_requests` becomes `false` after a request has been queued (yield-resume promise created, entry stored in the pending map), MPC nodes are unable to call `respond*` to deliver the computed signature. The request sits in the pending map until the NEAR yield-resume timeout (~200 blocks, `REQUEST_EXPIRATION_BLOCKS`) fires: [7](#0-6) 

At that point the yield-callback (`return_signature_and_clean_state_on_success`) pops the entry and fires `fail_on_timeout`, returning a timeout error to the caller. The user's 1 yoctonear deposit is refunded, but the signature is never produced.

The `accept_requests` flag is intended to gate **new** request submissions (checked in `check_request_preconditions`): [8](#0-7) 

Applying the same gate to `respond*` breaks the lifecycle invariant: a request that the contract already accepted (deposit taken, yield created, entry stored) should remain respondable regardless of subsequent TEE-validation state changes.

---

### Impact Explanation

Any `sign`, `request_app_private_key`, or `verify_foreign_transaction` request that is in-flight (queued but not yet responded to) at the moment `accept_requests` becomes `false` is frozen. MPC nodes compute the correct signature but cannot deliver it on-chain. The request fails after ~200 blocks with a timeout error. While the 1 yoctonear deposit is refunded, the signature is never produced. For time-sensitive cross-chain operations (e.g., a Bitcoin or Ethereum transaction that must be broadcast within a narrow window), this constitutes a meaningful denial of the service the MPC network is contracted to provide. This breaks the request-lifecycle safety invariant: accepted requests must be respondable.

This maps to: **Medium — request-lifecycle manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration.**

---

### Likelihood Explanation

TEE attestations expire on a fixed schedule (every 7 days per the design). When they expire, any participant can call `verify_tee()` — a single-participant call, below the signing threshold: [9](#0-8) 

A Byzantine participant (strictly below threshold) can observe the mempool for pending `sign` transactions, wait until attestations are near expiry, and then call `verify_tee()` immediately after a batch of sign requests lands on-chain but before MPC nodes respond. This maximizes the number of in-flight requests that are frozen. No collusion above threshold is required; no privileged access is needed beyond being a participant.

---

### Recommendation

Remove the `accept_requests` check from `respond`, `respond_ckd`, and `respond_verify_foreign_tx`. The flag should only gate new request submissions in `check_request_preconditions`. Requests that were already accepted and queued before the flag was set must remain respondable; the TEE-validation failure does not retroactively invalidate a signature that MPC nodes have already computed correctly. If the intent is to prevent nodes with expired attestations from responding, the existing `assert_caller_is_attested_participant_and_protocol_active()` check (which verifies the caller's own attestation) already enforces that invariant without blocking all responses globally.

---

### Proof of Concept

1. User calls `sign()` with a valid payload. The contract accepts it, creates a yield-resume promise, and stores a `YieldIndex` in `pending_signature_requests`.
2. One or more participants' TEE attestations expire (normal 7-day cycle).
3. A single Byzantine participant calls `verify_tee()`. Because fewer than `threshold` participants have valid attestations, `accept_requests` is set to `false`.
4. MPC nodes finish computing the threshold signature and call `respond(request, response)`.
5. `respond()` evaluates `if !self.accept_requests` at line 579 and returns `Err(TeeError::TeeValidationFailed)`. The transaction fails.
6. MPC nodes retry (up to `MAX_ATTEMPTS_PER_REQUEST_AS_LEADER = 10` times) but every attempt is rejected by the same check.
7. After ~200 blocks, the NEAR runtime fires the yield-callback with `Err(PromiseError::Failed)`. `return_signature_and_clean_state_on_success` pops the entry and calls `fail_on_timeout`. The user receives a timeout error; the signature is never produced. [10](#0-9) [11](#0-10)
