Audit Report

## Title
Integer Division Truncation in `viewSwapRsETHAmountAndFee` Allows Zero-Output Deposits - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
All pool variants compute `rsETHAmount` via integer division without a subsequent zero-check. When a depositor sends a dust amount (1–2 wei) such that `amountAfterFee * rate < rsETHToETHrate`, Solidity truncates the result to zero. The deposit proceeds: the user's tokens are transferred into the pool and zero wrsETH is minted or transferred back, permanently locking the deposited dust with no user-facing recovery path.

## Finding Description
In `RSETHPoolV3.sol`, the ETH path computes:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

and the token path computes:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

Both truncate to zero when the numerator is smaller than the denominator. The `deposit` functions perform no zero-check on `rsETHAmount` before minting:

```solidity
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);   // mints 0 — no revert
```

The `limitDailyMint` modifier also calls `viewSwapRsETHAmountAndFee` and adds `rsETHAmount` (zero) to `dailyMintAmount`, so it does not block the transaction. The same pattern is confirmed in `RSETHPoolV2.sol` (L225-233, L207-219), `RSETHPool.sol` (L311-347, L265-305), `RSETHPoolV3ExternalBridge.sol` (L418-453, L366-412), and `RSETHPoolV2ExternalBridge.sol` (L307-316, L289-301). In `RSETHPool.sol`, the deposit uses `safeTransfer(msg.sender, 0)` instead of `mint`, which also silently succeeds. There is no user-facing withdrawal function in any pool variant; all asset-recovery paths (`moveAssetsForBridging`, `withdrawFees`, `bridgeAssets`) are restricted to privileged roles.

## Impact Explanation
**Low. Contract fails to deliver promised returns.**

A depositor who sends a dust amount (e.g., 1 wei of ETH when `rsETHToETHrate > 1e18`, or 1 wei of a supported LST) receives zero wrsETH while their tokens are permanently transferred into the pool. The loss per transaction is bounded by at most a few wei for 18-decimal tokens, making the monetary impact negligible. However, the contract does not deliver its promised exchange and the deposited dust is unrecoverable by the user.

## Likelihood Explanation
Low. The condition requires a deposit amount small enough that `amountAfterFee * tokenToETHRate < rsETHToETHrate`. For standard 18-decimal LSTs with rates near `1e18`, this means deposits of 1–2 wei. Normal users are extremely unlikely to deposit such amounts accidentally. A malicious actor could repeatedly trigger this to inflate pool token balances without minting wrsETH, but the cost and impact are negligible. No special privileges are required; any external caller can trigger this via the public `deposit` functions.

## Recommendation
Add a zero-check on `rsETHAmount` in each `deposit` function and revert if the computed output is zero:

```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
```

Alternatively, enforce a `minRsETHAmountExpected` parameter (as `LRTDepositPool.depositAsset` already does) so callers can specify a slippage floor.

## Proof of Concept
Assume `rsETHToETHrate = 1.05e18`, `tokenToETHRate = 1e18`, `feeBps = 0`.

1. User calls `RSETHPoolV3.deposit(stETH, 1, "")` with `amount = 1 wei`.
2. `fee = 0`; `amountAfterFee = 1`.
3. `rsETHAmount = 1 * 1e18 / 1.05e18 = 0` (integer truncation).
4. `limitDailyMint`: `dailyMintAmount + 0 <= dailyMintLimit` → passes.
5. `IERC20(stETH).safeTransferFrom(user, pool, 1)` — 1 wei stETH leaves user.
6. `wrsETH.mint(user, 0)` — user receives nothing; transaction succeeds.
7. User has permanently lost 1 wei stETH with no recourse.

Foundry fuzz test plan: fuzz `amount` in `[1, rsETHToETHrate/1e18]` for the ETH path and assert `rsETHAmount == 0` while `balanceOf(pool, token)` increases, confirming the invariant violation.
