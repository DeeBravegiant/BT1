Let me look at the `plain_broadcast` and `uniform_msg` implementations to understand the network model assumptions.

### Title
Incomplete `h_all` Consistency Check for Quorum Parties Allows Malicious Representative to Cause Inconsistent AC-DKG/Refresh Key State — (`src/cbmpc/protocol/ec_dkg.cpp`)

---

### Summary

In `key_share_mp_t::dkg_or_refresh_ac`, the `h_all` hash is intended to ensure all parties agree on the same set of commitment values (`c_all`) before they are opened. However, the check for quorum parties only verifies that *other* quorum parties' `h_all` values match the representative's `h_all`. It never verifies that the **local party's own** computed `h_all` matches the representative's. Because `representative_quorum_pid_index` is always the highest-index quorum party, the representative's self-check is trivially vacuous. A malicious representative can send different `c_all` values to different quorum parties, craft per-party `h_all` responses to pass the check, and cause honest parties to accept inconsistent public key material.

---

### Finding Description

**`representative_quorum_pid_index` selection** — the loop unconditionally overwrites the variable on every quorum party, leaving it as the highest-index quorum party: [1](#0-0) 

**`h_all` check for quorum parties** — for each other quorum party `j`, the code checks `h_all._j == h_all.received(representative_quorum_pid_index)`. When `j == representative_quorum_pid_index`, this is `h_all.received(rep) != h_all.received(rep)` — trivially false. Critically, the local party's own `h_all._i` (computed from the `c_all` values it actually received) is **never compared** against the representative's `h_all`: [2](#0-1) 

**Non-quorum parties do have the correct check** — they compare their locally-computed hash against the representative's: [3](#0-2) 

**`plain_broadcast` is not Byzantine-fault-tolerant** — it is a simple send-to-all with no consistency enforcement. A malicious party can send different values to different recipients: [4](#0-3) 

`uniform_msg_t` is intent-only; `pack` always serializes `msg` (the local value) and `unpack` stores whatever is received per-party with no cross-party equality enforcement: [5](#0-4) 

---

### Proof of Concept

Setup: `n=3`, `quorum = {0, 1, 2}`, `representative_quorum_pid_index = 2` (malicious).

**Round 1** — `plain_broadcast(c_all)`:
- Party 2 sends `c_2_a` to party 0 and `c_2_b` to party 1 (different commitments).
- Party 0 computes: `h_all_0 = SHA256({c_0, c_1, c_2_a}, quorum_pids, sid)`
- Party 1 computes: `h_all_1 = SHA256({c_0, c_1, c_2_b}, quorum_pids, sid)`
- `h_all_0 ≠ h_all_1` since `c_2_a ≠ c_2_b`.

**Round 2** — `plain_broadcast(h_all, ac_pub_all, pi_r_all, rho_all, xij)`:
- Party 2 sends `h_all = h_all_1` to party 0, and `h_all = h_all_0` to party 1.

**Party 0's check** (`i=0`):
- `j=1`: `h_all.received(1)` (= `h_all_1`) == `h_all.received(2)` (= `h_all_1`) ✓
- `j=2`: `h_all.received(2)` == `h_all.received(2)` ✓ (trivially)
- Own `h_all_0` is **never compared** against `h_all.received(2)` → gap undetected.

**Party 1's check** (`i=1`):
- `j=0`: `h_all.received(0)` (= `h_all_0`) == `h_all.received(2)` (= `h_all_0`) ✓
- `j=2`: trivially ✓
- Own `h_all_1` is **never compared** → gap undetected.

Both honest parties pass the `h_all` check. Party 2 then opens `c_2_a` to party 0 (with `Rs_a`, `rho_a`, valid ZK proof) and `c_2_b` to party 1 (with `Rs_b`, `rho_b`, valid ZK proof). Both commitment openings and ZK verifications succeed. Party 2 sends shares `xij_a` (consistent with `Rs_a`) to party 0 and `xij_b` (consistent with `Rs_b`) to party 1; both pass `verify_share_against_ancestors_pub_data`.

**Result:**
- Party 0 accepts `Q_a = Q_0 + Q_1 + Q_2_a`
- Party 1 accepts `Q_b = Q_0 + Q_1 + Q_2_b`
- `Q_a ≠ Q_b` — inconsistent key state accepted by both honest parties. [6](#0-5) 

---

### Impact Explanation

Honest quorum parties complete `dkg_ac` or `refresh_ac` with inconsistent public keys and mismatched key shares. The DKG/refresh session is accepted as valid by all honest parties despite producing cryptographically invalid output. Any subsequent threshold signing or key-use operation will fail or produce incorrect results. In the refresh case, the existing key is corrupted. This is a **High** impact: public API reachable validation bypass in DKG/refresh creates accepted invalid cryptographic output.

---

### Likelihood Explanation

The attacker must be the highest-index quorum party (the representative). In a threshold scheme with `t`-of-`n`, a single malicious party below threshold can occupy the highest quorum index. The attack requires only the ability to send different messages to different parties over the unauthenticated point-to-point channels that `plain_broadcast` uses — a standard Byzantine adversary capability. No cryptographic hardness assumption is broken.

---

### Recommendation

For quorum parties, add a self-check comparing the locally computed `h_all._i` against the representative's received value before proceeding:

```cpp
// After computing h_all._i and before the verification loop:
if (quorum_party_set.has(i) && i != representative_quorum_pid_index) {
    if (h_all._i != h_all.received(representative_quorum_pid_index))
        return coinbase::error(E_CRYPTO, "h_all self-check mismatch");
}
```

Alternatively, replace the representative-anchored comparison with a pairwise check against the local party's own `h_all._i` for all parties (quorum and non-quorum alike), which is the pattern used correctly in `key_share_mp_t::dkg` and `key_share_mp_t::refresh`: [7](#0-6) [8](#0-7)
