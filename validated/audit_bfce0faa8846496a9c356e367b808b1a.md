Audit Report

## Title
Reward Tokens Permanently Locked When `totalKernelStaked` Drops to Zero During Active Reward Window - (File: contracts/KERNEL/KernelDepositPool.sol)

## Summary
When all stakers exit during an active reward distribution window, `rewardPerToken()` stops accumulating because it short-circuits on `totalKernelStaked == 0`. The reward tokens corresponding to the remaining window time (`rewardRate * remainingSeconds`) are permanently locked in the contract, as no admin or user function exists to recover them. The contract's own NatSpec explicitly acknowledges this behavior.

## Finding Description
`rewardPerToken()` at L408–414 returns `rewardPerTokenStored` unchanged whenever `totalKernelStaked == 0`:

```solidity
function rewardPerToken() public view returns (uint256) {
    if (totalKernelStaked == 0) {
        return rewardPerTokenStored;
    }
    ...
}
```

The `updateReward` modifier (L232–241) snapshots `updatedAt = lastTimeRewardApplicable()` on every user action. When the last staker calls `initiateWithdrawal`, `totalKernelStaked` is decremented to zero and `updatedAt` is set to the current timestamp. From that point forward, `rewardPerToken()` always returns the same stored value, so `rewardRate * (finishAt - updatedAt)` worth of reward tokens accumulate in the contract balance with no distribution path.

The `notifyRewardAmount` guard at L570 (`if (totalKernelStaked == 0) revert NoStakedTokens()`) only prevents starting a new reward period with zero stakers — it does not prevent stakers from exiting mid-window. Any staker can call `initiateWithdrawal` (a public, permissionless function) at any time.

A full audit of the contract confirms there is no `withdrawTokens`, `recoverTokens`, or equivalent admin rescue function for `rewardsToken` anywhere in `KernelDepositPool.sol`. The contract's own NatSpec at L18–22 explicitly acknowledges the lock: *"If `totalKernelStaked` ever hits zero during a reward distribution window, any remaining rewards for that period will stay locked in the contract."* The stated operational mitigation (ensuring tokens are always staked) is not enforced by code.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.** Any reward tokens corresponding to the interval `[lastStakerWithdrawal, finishAt]` are permanently unrecoverable. The quantity is `rewardRate * remainingSeconds`. No future call to `notifyRewardAmount` can unlock them; it only starts a fresh window on top of the stranded balance (and would itself revert if `totalKernelStaked` is still zero). This matches the allowed impact: *Medium. Permanent freezing of unclaimed yield.*

## Likelihood Explanation
**Medium.** The trigger is a normal, permissionless user action — calling `initiateWithdrawal` — requiring no admin collusion, key compromise, or privileged access. In a low-TVL pool or at the end of a campaign, all stakers exiting before `finishAt` is a realistic scenario. The condition is met by ordinary user behavior.

## Recommendation
1. Add an admin-callable `recoverRewardTokens(uint256 amount)` function (mirroring `KernelTop100MerkleDistributor.withdrawTokens`) restricted to calls after `block.timestamp >= finishAt`, allowing recovery of undistributed rewards.
2. Alternatively, track undistributed rewards explicitly and allow the admin to roll them into the next reward window via `notifyRewardAmount`.

## Proof of Concept
1. Admin calls `setRewardsDuration(7 days)`, then `notifyRewardAmount(700_000e18)` with at least one staker present → `rewardRate = 100_000e18 / day`, `finishAt = now + 7 days`.
2. Alice stakes 1000 KERNEL; `totalKernelStaked = 1000`.
3. After 1 day, Alice calls `initiateWithdrawal(1000)` → `updateReward` fires: `updatedAt = now`, `totalKernelStaked = 0`.
4. Alice waits `withdrawalDelay` and calls `claimWithdrawal` — she receives her 1000 KERNEL back correctly.
5. For the remaining 6 days, every call to `rewardPerToken()` returns `rewardPerTokenStored` unchanged (L409–411), because `totalKernelStaked == 0`.
6. At `finishAt`, `600_000e18` reward tokens remain in the contract. No function in `KernelDepositPool.sol` can retrieve them. They are permanently locked.

**Foundry test sketch:**
```solidity
function test_rewardTokensLockedWhenNoStakers() public {
    // Setup: admin sets duration, notifies reward with Alice staked
    vm.prank(admin); pool.setRewardsDuration(7 days);
    vm.prank(alice); pool.stake(1000e18);
    vm.prank(admin); pool.notifyRewardAmount(700_000e18);

    // Alice exits after 1 day
    vm.warp(block.timestamp + 1 days);
    vm.prank(alice); pool.initiateWithdrawal(1000e18);
    assertEq(pool.totalKernelStaked(), 0);

    // Fast-forward to end of reward window
    vm.warp(pool.finishAt());

    // rewardPerToken has not advanced since Alice's withdrawal
    uint256 rpt = pool.rewardPerToken();
    // rpt == rewardPerTokenStored (unchanged for 6 days)

    // 600_000e18 reward tokens are locked; no recovery function exists
    uint256 locked = rewardsToken.balanceOf(address(pool)) - pool.rewards(alice);
    assertGt(locked, 0); // permanently locked
}
```
