### Title
Pool Singleton Waiting-Room Lock Height Pre-Satisfaction Enables Instant Exit to Steal Block Rewards - (File: chia/pools/pool_wallet.py, chia/pools/pool_puzzles.py)

### Summary
A farmer who joins a pool can immediately signal leaving (entering `LEAVING_POOL` state), wait through the `relative_lock_height` cooldown, and then remain in `LEAVING_POOL` state indefinitely. Because the on-chain `pool_waiting_room_innerpuz` puzzle only enforces a **minimum** coin age (`ASSERT_HEIGHT_RELATIVE`) with no maximum, the farmer can exit to `SELF_POOLING` at any chosen moment — including the exact block in which they farm a winning proof — stealing the block reward before the pool can claim it.

### Finding Description

Chia's pool protocol uses a two-step exit to protect pools from farmers who might farm a block and immediately leave:

1. **Step 1 (signal):** Farmer transitions `FARMING_TO_POOL → LEAVING_POOL`. The singleton's inner puzzle becomes `pool_waiting_room_innerpuz`, which curries in `relative_lock_height`.
2. **Step 2 (complete):** After `relative_lock_height` blocks have elapsed since the LEAVING_POOL coin was created, the farmer can transition `LEAVING_POOL → SELF_POOLING`.

The `pool_waiting_room_innerpuz` enforces the delay via `ASSERT_HEIGHT_RELATIVE` — a condition that checks the **age of the LEAVING_POOL coin**, not the current block height relative to any join event. Once the coin is old enough, the exit spend is valid at **any future block** with no expiry.

A malicious farmer can exploit this by:

1. Joining a pool (entering `FARMING_TO_POOL`).
2. Immediately calling `self_pool()` / `pw_self_pool` to enter `LEAVING_POOL`.
3. Waiting `relative_lock_height` blocks while continuing to farm for the pool (the LEAVING_POOL coin ages).
4. Holding the LEAVING_POOL state indefinitely — the wallet's automatic `new_peak()` submission can be suppressed by a custom client.
5. When the farmer's plots win a challenge, submitting the Step 2 exit transaction (`LEAVING_POOL → SELF_POOLING`) in the **same block** as the farming transaction.
6. In the next block, claiming the p2_singleton reward coin using the now-SELF_POOLING singleton, before the pool has any opportunity to absorb it.

The pool can only absorb p2_singleton coins while the singleton is in `FARMING_TO_POOL` or `LEAVING_POOL` state. Once the singleton is in `SELF_POOLING` state, only the farmer can claim those coins. Because the farmer controls which transactions appear in their own farmed block, they can ensure the exit transaction precedes any pool absorb attempt.

### Impact Explanation

The farmer diverts a full block reward (coinbase + fee reward, currently ~2 XCH) that was pledged to the pool. The pool's front-running protection — the `relative_lock_height` — is entirely bypassed because the cooldown was pre-satisfied at join time. This is an unauthorized diversion of XCH from the pool's expected reward stream, matching the Critical impact category: *reward diversion affecting XCH via singleton-controlled assets*.

### Likelihood Explanation

Any farmer who understands the pool protocol can execute this. No privileged access, leaked keys, or cryptographic break is required — only the ability to submit transactions and control a custom wallet client. The minimum `relative_lock_height` is 5 blocks; a farmer need only wait 5 blocks after joining before they hold a perpetual instant-exit capability. The pool has no on-chain mechanism to detect or prevent this.

### Recommendation

The `pool_waiting_room_innerpuz` (and the v2 `PlotNFTPuzzle` heightlock mechanism) should enforce that the LEAVING_POOL coin cannot be spent to exit **after** a maximum age, or alternatively, the protocol should record the block height at which the farmer **joined** the pool and measure the lock height from that point rather than from when the LEAVING_POOL coin was created. A simpler short-term mitigation: pools should track singleton state on-chain and refuse to accept proofs from farmers whose singleton is in `LEAVING_POOL` state with a satisfied lock height.

### Proof of Concept

```
Block N:   Farmer joins pool → FARMING_TO_POOL singleton created
Block N+1: Farmer calls pw_self_pool → LEAVING_POOL singleton created (coin age = 0)
           [relative_lock_height = 5]
Block N+6: LEAVING_POOL coin age = 5 → ASSERT_HEIGHT_RELATIVE satisfied
           Farmer suppresses wallet's automatic new_peak() exit submission
           Farmer continues farming for the pool (pool absorbs rewards normally)

Block M (M >> N+6):
  Farmer's plot wins challenge.
  Farmer constructs block M containing:
    TX-A: LEAVING_POOL → SELF_POOLING exit spend (valid because coin age >> 5)
    [Block reward p2_singleton coin created at block M]

Block M+1:
  Farmer submits: SELF_POOLING singleton + p2_singleton absorb spend
  → Full block reward claimed by farmer
  Pool receives nothing; its absorb attempt references a LEAVING_POOL
    singleton that no longer exists.
```

The wallet-side lock height check in `self_pool()` only prevents the wallet from submitting the exit while the lock is unsatisfied; it does not prevent a custom client from holding the exit transaction and submitting it at will once the lock is satisfied. class PoolSingletonState(IntEnum):
    """
    From the user's point of view, a pool group can be in these states:
    `SELF_POOLING`: The singleton exists on the blockchain, and we are farming
        block rewards to a wallet address controlled by the user

    `LEAVING_POOL`: The singleton exists, and we have entered the "escaping" state, which
        means we are waiting for a number of blocks = `relative_lock_height` to pass, so we can leave.

    `FARMING_TO_POOL`: The singleton exists, and it is assigned to a pool.

    `CLAIMING_SELF_POOLED_REWARDS`: We have submitted a transaction to sweep our
        self-pooled funds.
    """

    SELF_POOLING = 1
    LEAVING_POOL = 2
    FARMING_TO_POOL = 3
