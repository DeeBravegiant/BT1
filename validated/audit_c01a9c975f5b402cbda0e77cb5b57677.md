Audit Report

## Title
ETH Permanently Frozen When Withdrawal Recipient Contract Rejects ETH — (`contracts/LRTWithdrawalManager.sol`)

## Summary
A smart contract that calls `initiateWithdrawal` will have its rsETH irreversibly burned during `unlockQueue`, yet every subsequent call to `completeWithdrawal` or `completeWithdrawalForUser` permanently reverts because `_transferAsset` unconditionally pushes ETH to the original `user` address with no alternative recipient. The ETH is permanently frozen inside `LRTWithdrawalManager` with no admin recovery mechanism, and `sweepRemainingAssets` is permanently blocked by the non-zero `unlockedWithdrawalsCount`.

## Finding Description

**Step 1 — No EOA restriction on `initiateWithdrawal`**

`initiateWithdrawal` (L150–178) accepts calls from any address. rsETH is transferred from `msg.sender` to the withdrawal manager via `safeTransferFrom` at L166. No `tx.origin == msg.sender` or similar EOA check exists. [1](#0-0) 

**Step 2 — rsETH burned irreversibly in `unlockQueue`**

When an operator calls `unlockQueue`, rsETH held by the contract is burned at L305 via `burnFrom(address(this), rsETHBurned)`. This is a permanent, irreversible state change. [2](#0-1) 

**Step 3 — `_transferAsset` hard-reverts on ETH send failure**

`_transferAsset` (L876–883) performs a bare low-level call to `payable(to)` and reverts with `EthTransferFailed` if the call returns `false`. If `to` is a contract whose `receive()` reverts or is absent, this always fails with no fallback. [3](#0-2) 

**Step 4 — Both completion paths route ETH to the original `user` with no override**

`completeWithdrawal` passes `msg.sender` as `user` (L183–184). `completeWithdrawalForUser` passes the original requester address (L192–203). Both call `_processWithdrawalCompletion(asset, user, ...)`, which calls `_transferAsset(asset, user, ...)` at L734. There is no parameter to redirect ETH to a different recipient. [4](#0-3) [5](#0-4) [6](#0-5) 

The NatSpec on `completeWithdrawalForUser` at L191 even acknowledges this is not expected to be used for ETH, but dismisses the risk as "non-impactful" — which is incorrect. [7](#0-6) 

**Step 5 — Atomic revert preserves the stuck state**

Inside `_processWithdrawalCompletion` (L699–738), state changes at L705 (`popFront`), L712 (`delete withdrawalRequests`), and L717 (`unlockedWithdrawalsCount[asset]--`) all occur before the `_transferAsset` call at L734. When `_transferAsset` reverts, the entire transaction is rolled back atomically — `unlockedWithdrawalsCount` is never decremented, and the request remains in the queue. [8](#0-7) 

**Step 6 — `sweepRemainingAssets` permanently blocked**

`sweepRemainingAssets` (L395–414) is gated by `hasUnlockedWithdrawals(asset)` at L403. Since every `completeWithdrawal` call reverts atomically, `unlockedWithdrawalsCount` never reaches zero, permanently blocking the sweep. No admin function exists to forcibly cancel a stuck unlocked withdrawal request. [9](#0-8) 

## Impact Explanation

**Critical. Permanent freezing of funds.**

- rsETH is burned in `unlockQueue` — irreversible.
- ETH is moved from the unstaking vault into `LRTWithdrawalManager` — it sits there permanently.
- Every call to `completeWithdrawal` / `completeWithdrawalForUser` reverts with `EthTransferFailed`.
- `sweepRemainingAssets` is permanently blocked by `PendingWithdrawalsExist`.
- No admin escape hatch exists to cancel the stuck withdrawal or reclaim the ETH.

The user loses both their rsETH (burned) and their ETH (frozen), with zero recovery path.

## Likelihood Explanation

**Medium.** Smart contract wallets (multisigs, DAO treasuries, smart accounts) are common DeFi participants. A contract that does not implement `receive()` or has a reverting fallback — whether by design, upgrade, or bug — can trigger this. The protocol does not restrict `initiateWithdrawal` to EOAs. The scenario requires no privileged access or external compromise, only a contract-initiated withdrawal of ETH.

## Recommendation

1. **Add a `recipient` parameter** to `completeWithdrawal` (or a separate `claimTo(address recipient)` function) so the user can redirect ETH to an address capable of receiving it.
2. **Alternatively**, wrap the ETH transfer in a pull-payment pattern: credit the user's claimable balance and let them pull it separately, avoiding push-to-contract failures.
3. **Add an admin escape hatch** — a manager-callable function to cancel a stuck unlocked withdrawal, return the ETH to the treasury, and decrement `unlockedWithdrawalsCount`, so `sweepRemainingAssets` can eventually clear the contract.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract RevertingReceiver {
    receive() external payable { revert("no ETH"); }

    function doInitiate(address withdrawalManager, address rsETH, uint256 rsETHAmount) external {
        IERC20(rsETH).approve(withdrawalManager, rsETHAmount);
        ILRTWithdrawalManager(withdrawalManager)
            .initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
    }
}

// Foundry fork test sequence:
// 1. Deploy RevertingReceiver, fund it with rsETH
// 2. Call doInitiate() → rsETH transferred to WithdrawalManager (L166)
// 3. Operator calls unlockQueue(ETH_TOKEN, ...) → rsETH burned (L305), ETH moved in (L307)
// 4. vm.roll(block.number + withdrawalDelayBlocks + 1)
// 5. RevertingReceiver calls completeWithdrawal(ETH_TOKEN, "") → REVERTS (EthTransferFailed, L879)
// 6. Operator calls completeWithdrawalForUser(ETH_TOKEN, address(revertingReceiver), "") → REVERTS
// 7. Assert: address(withdrawalManager).balance > 0 (ETH stuck)
// 8. Assert: rsETH.totalSupply() decreased (rsETH burned, unrecoverable)
// 9. Assert: sweepRemainingAssets(ETH_TOKEN) reverts with PendingWithdrawalsExist (L403)
```
