### Title
Cross-Request Payload Hash Replay in `respond_verify_foreign_tx` Allows Request-Lifecycle Manipulation — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `respond_verify_foreign_tx` contract function verifies that the submitted MPC signature is valid over the provided `payload_hash`, but does **not** verify that `payload_hash` encodes the original `request` parameter. A single malicious attested participant can replay a legitimately produced MPC signature from one `verify_foreign_transaction` request to resolve a different pending request, consuming its yield and delivering an incorrect (stale/mismatched) response to the caller.

---

### Finding Description

`respond_verify_foreign_tx` (lib.rs:691–754) accepts two arguments:
- `request: VerifyForeignTransactionRequest` — the original on-chain request
- `response: VerifyForeignTransactionResponse` — contains `payload_hash` and `signature`

The contract performs these checks:

```rust
// Check the signature is correct against the root public key
near_mpc_signature_verifier::verify_ecdsa_signature(
    signature_response,
    &payload_hash,
    &secp_pk,
).is_ok()
``` [1](#0-0) 

It then resolves the pending yield:

```rust
pending_requests::resolve_yields_for(
    &mut self.pending_verify_foreign_tx_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [2](#0-1) 

**The missing check**: the contract never verifies that `response.payload_hash` is the hash of a `ForeignTxSignPayload` that encodes the original `request`. The signed payload type is:

```rust
pub struct ForeignTxSignPayloadV1 {
    pub request: ForeignChainRpcRequest,
    pub values: Vec<ExtractedValue>,
}
``` [3](#0-2) 

where `msg_hash = SHA-256(borsh(ForeignTxSignPayload))`. [4](#0-3) 

Because the contract only checks `signature_valid_over(payload_hash)` and `pending_request_exists(request)` — but never checks `payload_hash == hash_of(request, ...)` — the two checks are **decoupled**. A valid signature produced for request B can be submitted against request A's pending yield.

---

### Impact Explanation

**Attack path** (single Byzantine participant below signing threshold):

1. Request A and Request B are both pending in `pending_verify_foreign_tx_requests`.
2. The MPC network legitimately processes request B, producing `(payload_hash_B, signature_B)`. The malicious participant, as an attested node, participates in this signing and observes the result.
3. The malicious participant calls `respond_verify_foreign_tx(request=A, response={payload_hash_B, signature_B})`.
4. The contract verifies `signature_B` is valid over `payload_hash_B` — it is. It then resolves request A's yield with `{payload_hash_B, signature_B}`.

**Consequences**:
- Request A's pending yield is permanently consumed (removed from `pending_verify_foreign_tx_requests`). The request cannot be retried.
- Request A's caller receives a response whose `payload_hash` encodes request B's data (a different `tx_id`, different `extracted_values`), not request A's.
- For a bridge contract (the primary use case — Omnibridge inbound flow), this breaks the accounting invariant that each `verify_foreign_transaction` call is resolved with a response corresponding to its own foreign-chain observation. A bridge that does not independently re-verify the payload hash would accept the incorrect attestation.
- The caller's deposit is effectively lost; they must resubmit.

This maps to the **Medium** allowed impact: *"Balance, request-lifecycle, participant-state, or contract execution-flow manipulation that breaks production safety/accounting invariants without relying on network-level DoS or operator misconfiguration."*

---

### Likelihood Explanation

- **Single attested participant** can call `respond_verify_foreign_tx` — no threshold collusion required.
- The attacker only needs two concurrent pending requests (realistic in production under any load).
- The attacker legitimately participates in signing request B (as a normal MPC participant), then reuses the resulting signature against request A. No cryptographic forgery is needed.
- The design doc explicitly notes that the generic queue retries failed requests, increasing the window of concurrent pending requests. [5](#0-4) 

---

### Recommendation

The contract must bind `response.payload_hash` to the original `request`. Since the full `ForeignTxSignPayload` cannot be included in the response (NEAR promise data limits), two practical mitigations exist:

1. **Include a `request_hash` commitment in the response**: Add `request_hash: Hash256` to `VerifyForeignTransactionResponse`, where `request_hash = SHA-256(borsh(request))`. The contract independently computes `SHA-256(borsh(request))` and asserts it equals `response.request_hash`. Nodes compute and include this field when calling `respond_verify_foreign_tx`.

2. **Verify the payload hash prefix**: Since `borsh(ForeignTxSignPayload::V1(ForeignTxSignPayloadV1{request, values}))` always begins with the borsh encoding of `request`, the contract could verify that `payload_hash` is consistent with a payload whose `request` field matches the submitted `request`. This requires the contract to compute `borsh(request)` and check it is a prefix of the pre-image — feasible since `request` is already available.

### Proof of Concept

```
State: pending_verify_foreign_tx_requests = {A: [yield_A], B: [yield_B]}

Step 1: MPC network signs request B normally.
        → payload_hash_B = SHA-256(borsh(ForeignTxSignPayload{request: B, values: B_vals}))
        → signature_B = MPC_sign(payload_hash_B)

Step 2: Malicious attested participant calls:
        respond_verify_foreign_tx(
            request = A,                                          // request A's key
            response = { payload_hash: payload_hash_B,           // B's hash
                         signature:   signature_B }              // B's signature
        )

Step 3: Contract checks:
        ✓ verify_ecdsa_signature(signature_B, payload_hash_B, pk)  → valid
        ✓ pending_verify_foreign_tx_requests.get(A)                → exists
        → resolve_yields_for(A, {payload_hash_B, signature_B})

Result: yield_A resolved with B's payload hash.
        Caller A receives {payload_hash_B, signature_B} — incorrect attestation.
        Request A is permanently consumed from storage.
```

    #[handle_result]
    pub fn respond_verify_foreign_tx(
        &mut self,
        request: VerifyForeignTransactionRequest,
        response: VerifyForeignTransactionResponse,
    ) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();

        log!(
            "respond_verify_foreign_tx: signer={}, request={:?}",
            &signer,
            &request
        );

        self.assert_caller_is_attested_participant_and_protocol_active();

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        let domain = request.domain_id;
        let public_key = self.public_key_extended(domain.0.into())?;

        let signature_is_valid = match (&response.signature, public_key) {
            (
                dtos::SignatureResponse::Secp256k1(signature_response),
                PublicKeyExtended::Secp256k1 { near_public_key },
            ) => {
                let secp_pk = dtos::Secp256k1PublicKey::try_from(&near_public_key)
                    .expect("Secp256k1 variant always has a secp256k1 key");

                let payload_hash: [u8; 32] = response.payload_hash.0;

                // Check the signature is correct against the root public key
                near_mpc_signature_verifier::verify_ecdsa_signature(
                    signature_response,
                    &payload_hash,
                    &secp_pk,
                )
                .is_ok()
            }
            (signature_response, public_key_requested) => {
                return Err(RespondError::SignatureSchemeMismatch {
                    mpc_scheme: Box::new(signature_response.clone()),
                    user_scheme: Box::new(public_key_requested),
                }
                .into());
            }
        };

        if !signature_is_valid {
            return Err(RespondError::InvalidSignature.into());
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_verify_foreign_tx_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
