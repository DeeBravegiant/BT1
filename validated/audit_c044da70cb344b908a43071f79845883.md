Audit Report

## Title
`setAaveIntegrationEnabled(false)` Reverts When Aave Is Paused, Permanently Blocking ETH Withdrawal Completion - (File: `contracts/LRTWithdrawalManager.sol`)

## Summary
`setAaveIntegrationEnabled(false)` performs two external calls to `aaveWETHGateway.withdrawETH(...)` before assigning `isAaveIntegrationEnabled = false`. When Aave's WETH reserve is paused, both calls revert and the flag is never cleared. Because `_processWithdrawalCompletion` unconditionally calls `_withdrawFromAave` whenever `isAaveIntegrationEnabled == true` and the contract's ETH balance is insufficient, all pending ETH withdrawal completions also revert for the entire duration of the Aave pause.

## Finding Description
In `setAaveIntegrationEnabled` (L486–503), when `enabled = false` and `aaveBalance > 0`, the function calls `_collectInterestToTreasury()` (L490) and `_withdrawFromAave(aaveBalance)` (L495) before reaching the state assignment `isAaveIntegrationEnabled = enabled` at L503. Both helpers ultimately call `aaveWETHGateway.withdrawETH(aavePool, ...)` — `_collectInterestToTreasury` at L954 and `_withdrawFromAave` at L917. Aave v3 reverts `withdrawETH` when the reserve is paused, so the entire transaction reverts and `isAaveIntegrationEnabled` remains `true`.

With the flag stuck at `true`, every call to `completeWithdrawal` or `completeWithdrawalForUser` for ETH reaches `_processWithdrawalCompletion` (L719–731), which calls `_withdrawFromAave(amountNeeded)` at L724 whenever `address(this).balance < request.expectedAssetAmount`. That call also reverts, freezing all ETH withdrawal completions.

`emergencyWithdrawFromAave` (L551–563) has the identical defect: it calls `_collectInterestToTreasury()` at L558 and `_withdrawFromAave(amount)` at L560 with no try/catch, so it too reverts when Aave is paused and provides no escape hatch.

No existing guard in the contract detects or bypasses an Aave pause state; there is no `try/catch` around either external call, and no alternative code path that sets the flag without first withdrawing.

## Impact Explanation
**Medium — Temporary freezing of funds.** All users with unlocked ETH withdrawal requests are unable to call `completeWithdrawal` for the full duration of the Aave pause. Funds are not lost — they remain in Aave — but they are completely inaccessible to users until Aave unpauses. The freeze is bounded by the Aave pause duration and requires no attacker; it is a direct consequence of the checks-effects-interactions violation in the LRT code.

## Likelihood Explanation
Aave v3 has a documented guardian/emergency-admin role that can pause individual asset reserves. The WETH reserve is one of the largest Aave v3 markets and has been paused during past security incidents. No attacker action is required; a routine Aave guardian pause is sufficient. The precondition (ETH deposited to Aave, `isAaveIntegrationEnabled == true`) is the normal operating state of the protocol.

## Recommendation
Apply checks-effects-interactions: set `isAaveIntegrationEnabled = false` **before** attempting any withdrawal, and wrap the withdrawal in a `try/catch` so a revert does not block disabling the integration:

```solidity
function setAaveIntegrationEnabled(bool enabled) external nonReentrant onlyLRTManager {
    if (enabled == isAaveIntegrationEnabled) revert AaveIntegrationAlreadyInDesiredState(enabled);

    isAaveIntegrationEnabled = enabled;   // state change first
    emit AaveIntegrationEnabled(enabled);

    if (!enabled) {
        uint256 aaveBalance = aaveAWETH.balanceOf(address(this));
        if (aaveBalance > 0) {
            try this.withdrawFromAaveExternal(aaveBalance) {} catch {}
        }
        _revokeApprovalToAaveWETHGateway();
    }
}
```

Apply the same pattern to `emergencyWithdrawFromAave`. Expose a separate `withdrawFromAave()` function callable once Aave unpauses to retrieve any funds left behind.

## Proof of Concept
1. Protocol has ETH in Aave (`isAaveIntegrationEnabled == true`, `totalETHDepositedToAave > 0`, `aaveAWETH.balanceOf(address(this)) > 0`).
2. Aave guardian pauses the WETH reserve.
3. Manager calls `setAaveIntegrationEnabled(false)`.
4. Execution enters the `if (!enabled)` branch at L486; `aaveBalance > 0` so `_collectInterestToTreasury()` is called at L490.
5. `_collectInterestToTreasury` calls `aaveWETHGateway.withdrawETH(...)` at L954 → Aave reverts with `RESERVE_PAUSED`.
6. Entire transaction reverts; `isAaveIntegrationEnabled` remains `true`.
7. User calls `completeWithdrawal(ETH_TOKEN, ...)` with an unlocked request.
8. `_processWithdrawalCompletion` at L720 evaluates `isAaveIntegrationEnabled && asset == ETH_TOKEN` → `true`; `address(this).balance < request.expectedAssetAmount` → `true`; calls `_withdrawFromAave(amountNeeded)` at L724.
9. `_withdrawFromAave` calls `aaveWETHGateway.withdrawETH(...)` at L917 → Aave reverts again.
10. All ETH withdrawal completions revert until Aave unpauses.
