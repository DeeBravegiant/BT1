### Title
FanOut Inspector Silently Discards `NotFinalized`/`NotEnoughBlockConfirmations` as Transient, Allowing Single-Provider Finality Bypass — (File: `crates/foreign-chain-inspector/src/lib.rs`)

---

### Summary

The `FanOut` inspector's `is_transient()` classification of `NotFinalized` and `NotEnoughBlockConfirmations` errors causes the multi-provider quorum logic to silently accept a single RPC provider's success response even when all other configured providers reject the transaction as unfinalized. This collapses the intended per-node RPC quorum to an effective quorum of 1 for finality decisions, allowing the MPC network to produce a threshold signature over an unfinalized foreign-chain transaction during a natural reorg window.

---

### Finding Description

In `crates/foreign-chain-inspector/src/lib.rs`, `ForeignChainInspectionError::is_transient()` classifies both `NotFinalized` and `NotEnoughBlockConfirmations` as transient: [1](#0-0) 

The `FanOut` inspector, which fans a verification request out to all locally-configured RPC providers for a chain, separates results into three buckets: successes, non-transient errors, and transient errors. Transient errors are silently discarded whenever at least one success exists: [2](#0-1) 

The critical path is:

```
if let Some(first_values) = successes.first() {
    // all_successes_agree check only compares successes to each other
    return Ok(first);   // transient errors (NotFinalized) are never consulted
}
```

This means: if provider A returns `Ok(block_hash)` and providers B and C return `NotFinalized` (transient), the `FanOut` returns `Ok(block_hash)` — a single provider's affirmative verdict wins unconditionally.

The design document explicitly requires a per-node RPC quorum:

> "Every node partaking in a foreign signature verification request queries all its locally configured RPC providers for the relevant chain, independently of other nodes. A quorum of those RPC providers must agree on the verification; if fewer than the quorum agree, the node errors out." [3](#0-2) 

The current implementation violates this requirement: `NotFinalized` and `NotEnoughBlockConfirmations` are substantive verdicts about chain state (not mere availability failures), yet they are treated identically to `ClientError` (network timeout, connection refused). A provider that says "this transaction has not reached the required finality level" is making a deterministic claim about the foreign chain, not reporting a transient transport failure.

The node-side signing path in `crates/node/src/providers/verify_foreign_tx/sign.rs` propagates the `FanOut` result directly into the MPC signing pipeline: [4](#0-3) 

The on-chain `respond_verify_foreign_tx` only verifies the ECDSA signature against the root public key; it does not independently validate that the extracted values correspond to a finalized transaction: [5](#0-4) 

---

### Impact Explanation

The MPC network's `verify_foreign_transaction` feature is the trust anchor for the Omnibridge inbound flow: a NEAR bridge contract mints tokens upon receiving a valid `VerifyForeignTransactionResponse` signed by the MPC key. If the MPC network signs a payload for a transaction that is subsequently reorged out of the canonical chain, the bridge contract has no mechanism to detect this — it trusts the MPC signature. The attacker retains both the minted NEAR-side tokens and the reorged foreign-chain funds, constituting a direct double-spend.

This matches the allowed impact: **High — forged foreign-chain verification that causes invalid bridge execution or double-spend conditions.**

---

### Likelihood Explanation

Natural short-lived reorgs occur on all supported chains (Bitcoin, EVM L1/L2). During a reorg, a subset of RPC providers temporarily serve the orphaned fork. If even one whitelisted provider per node (across ≥ signing-threshold nodes) is on the fork while the remaining providers return `NotFinalized`, the `FanOut` on each such node accepts the forked provider's response. The attacker does not need to compromise any provider — they only need to submit the `verify_foreign_transaction` request during the reorg window. For chains where `EvmFinality::Latest` or `EvmFinality::Safe` is used, reorgs are a routine occurrence. For Bitcoin with low confirmation counts (the user-supplied `confirmations` field has no contract-enforced minimum), the window is wider. [6](#0-5) 

---

### Recommendation

1. **Reclassify `NotFinalized` and `NotEnoughBlockConfirmations` as non-transient.** These are deterministic verdicts about chain state. A provider that says "not finalized" is not unavailable — it is disagreeing with a provider that says "finalized." The `FanOut` must treat this as `InspectorResponseMismatch`, not as a tolerable absence.

2. **Enforce the RPC quorum explicitly.** Count the number of providers that returned `Ok` with agreeing values. If that count is below the configured `rpc_quorum` for the chain, return `InspectorResponseMismatch` regardless of whether the remaining providers returned transient or non-transient errors.

3. **Separate availability-transient from finality-transient.** `ClientError` and `RpcRequestFailed` (pure transport failures) are legitimately transient. `NotFinalized` and `NotEnoughBlockConfirmations` are chain-state verdicts and must participate in the quorum comparison.

---

### Proof of Concept

**Setup**: Bitcoin chain, node configured with 3 RPC providers (A, B, C), signing threshold = 2 nodes, `rpc_quorum` = 2 providers.

1. Attacker broadcasts a Bitcoin transaction `TX` and immediately submits `verify_foreign_transaction` with `confirmations: 1`.
2. `TX` is included in an orphan block visible to provider A but not yet confirmed on the canonical chain (providers B and C see 0 confirmations).
3. On each of the 2 signing-threshold nodes:
   - Provider A → `Ok(block_hash_of_orphan_block)` (1 confirmation on the fork)
   - Provider B → `NotEnoughBlockConfirmations { expected: 1, got: 0 }` (transient → discarded)
   - Provider C → `NotEnoughBlockConfirmations { expected: 1, got: 0 }` (transient → discarded)
   - `FanOut` returns `Ok(block_hash_of_orphan_block)` — quorum of 1 accepted. [7](#0-6) 

4. Both nodes build `ForeignTxSignPayload::V1 { request, values: [BlockHash(orphan_block_hash)] }` and produce ECDSA signature shares.
5. The MPC network assembles the threshold signature and calls `respond_verify_foreign_tx` on the NEAR contract.
6. The NEAR contract verifies the signature is valid against the root key and resolves the yield, returning `VerifyForeignTransactionResponse` to the bridge contract.
7. The bridge contract mints tokens for the attacker on NEAR.
8. The orphan block is reorged; `TX` is evicted from the canonical chain. The attacker's foreign-chain funds are returned.
9. Attacker holds both the minted NEAR tokens and the original foreign-chain funds — double-spend complete.
