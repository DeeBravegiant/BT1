Audit Report

## Title
ETH Deposit Limit Check Ignores Incoming Deposit Amount, Allowing Unlimited Over-Deposit Beyond Protocol Cap - (File: contracts/LRTDepositPool.sol)

## Summary
`_checkIfDepositAmountExceedesCurrentLimit` applies an asymmetric guard: for ERC20 assets it correctly tests `totalAssetDeposits + amount > limit`, but for ETH it silently discards `amount` and only tests `totalAssetDeposits > limit`. Any unprivileged caller can therefore send an arbitrarily large ETH deposit in a single transaction whenever the current total is at or below the cap, causing unbounded rsETH over-minting and pushing the protocol's EigenLayer exposure far beyond its configured risk limit.

## Finding Description
The root cause is in `_checkIfDepositAmountExceedesCurrentLimit` (lines 676–682):

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    if (asset == LRTConstants.ETH_TOKEN) {
        return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset)); // amount never used
    }
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

The ETH branch at line 679 evaluates only the pre-deposit total against the cap. The `amount` parameter (i.e., `msg.value`) is never incorporated. The guard therefore returns `false` — meaning "not exceeded" — for any deposit made while `totalAssetDeposits <= depositLimit`, regardless of how large `msg.value` is.

The full call path is:

```
depositETH{value: X}(minRSETHAmountExpected, referralId)   // public, payable, nonReentrant, whenNotPaused
  └─ _beforeDeposit(ETH_TOKEN, msg.value, ...)              // line 87
       └─ _checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, msg.value)
            └─ returns false when totalAssetDeposits <= depositLimit  ← bug (line 679)
  └─ _mintRsETH(rsethAmountToMint)                          // mints rsETH for full msg.value (line 90)
```

No existing check compensates for this. `_beforeDeposit` only reverts on `MaximumDepositLimitReached` when the function returns `true`, which never happens for ETH until the cap is already breached by a prior deposit.

## Impact Explanation
**Critical — Protocol insolvency.**

The deposit limit exists to cap the protocol's EigenLayer exposure. When bypassed:
1. `_mintRsETH` mints rsETH proportional to the full `msg.value` at the current oracle rate, inflating supply beyond what the protocol's risk parameters support.
2. `getTotalAssetDeposits` will reflect an ETH balance that can be multiples of `depositLimitByAsset(ETH)`, meaning the protocol's EigenLayer exposure is unboundedly over-extended.
3. The already-minted rsETH cannot be recalled without governance intervention; existing rsETH holders are diluted and face insolvency risk under any EigenLayer slashing event at the over-exposed scale.

This matches the **Critical: Protocol insolvency** impact class.

## Likelihood Explanation
**High.** `depositETH` is public, payable, and requires no special role or privileged access. The precondition — `totalAssetDeposits <= depositLimit` — is the normal operating state of the protocol for the vast majority of its lifetime and is trivially readable on-chain via `getTotalAssetDeposits`. No front-running, oracle manipulation, or governance capture is required. A single transaction suffices.

## Recommendation
Apply the same prospective check used for ERC20 tokens, unifying the branch:

```solidity
function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
    uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
    return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
}
```

This ensures the incoming deposit amount is always included in the comparison, consistently enforcing the cap for both ETH and ERC20 assets.

## Proof of Concept
```
depositLimit              = 1_000 ETH
totalAssetDeposits before = 999 ETH

Attacker calls: depositETH{value: 5_000 ETH}(0, "")

_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, 5_000 ETH):
  totalAssetDeposits = 999 ETH
  ETH branch: return (999 > 1_000) = false   ← NOT blocked

_mintRsETH mints rsETH for 5_000 ETH at current oracle rate.

totalAssetDeposits after = 5_999 ETH  (≈ 6× the intended cap)
```

**Foundry test plan:**
1. Deploy `LRTDepositPool` with `depositLimitByAsset(ETH) = 1_000 ether`.
2. Seed the pool with 999 ETH (e.g., via a prior legitimate deposit).
3. Call `depositETH{value: 5_000 ether}(0, "")` from an unprivileged address.
4. Assert the call succeeds (no revert).
5. Assert `getTotalAssetDeposits(ETH_TOKEN) == 5_999 ether` (far above the 1_000 ETH cap).
