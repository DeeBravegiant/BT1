Audit Report

## Title
Block Stuffing Delays `bridgeAssetsViaNativeBridge` Causing Temporary Multi-Cycle ETH Freeze — (`contracts/pools/RSETHPoolNoWrapper.sol`)

## Summary
`RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge()` is the sole path for moving accumulated user ETH from Unichain to L1, callable only by `BRIDGER_ROLE`. Because the contract enforces no on-chain deadline, accumulation cap, or permissionless fallback, an attacker can fill consecutive Unichain blocks with high-gas-price transactions to prevent the operator's transaction from being included, causing ETH from multiple deposit cycles to accumulate in the pool. Funds are not permanently lost but are temporarily frozen until the attacker stops and the operator successfully submits the call.

## Finding Description
`bridgeAssetsViaNativeBridge()` (lines 431–444) bridges the entire `ethBalanceMinusFees` in a single call via `IL2Messenger(messenger).sendETHToL1ViaBridge{value: ethBalanceMinusFees}(l2Bridge, l1VaultETHForL2Chain, ethBalanceMinusFees)`. The call chain terminates at `IUnichainMessenger(l2bridge).bridgeETHTo{value: value}(target, DEFAULT_GAS_LIMIT, bytes(""))` in `UnichainMessenger.sendETHToL1ViaBridge` (line 26). The contract contains no deadline by which bridging must occur, no maximum ETH accumulation cap, no fallback or permissionless bridging path, and no enforcement of bridging frequency. Unichain is an OP-stack L2 with a substantially lower block gas limit than Ethereum mainnet, making block stuffing materially cheaper. An attacker submits high-gas-price filler transactions that consume the entire block gas budget during the operator's scheduled call window, preventing inclusion of `bridgeAssetsViaNativeBridge`. Each stuffed cycle causes one additional cycle of user deposits to remain in the pool. Existing guards (`nonReentrant`, `onlyRole(BRIDGER_ROLE)`) address reentrancy and access control but do nothing to ensure timely inclusion of the operator's transaction.

## Impact Explanation
**Low — Block stuffing / Medium — Temporary freezing of funds.** User ETH deposited into `RSETHPoolNoWrapper` is not permanently lost; once stuffing stops, the operator can call `bridgeAssetsViaNativeBridge()` and the full accumulated balance is bridged in one call. However, during the stuffing period all deposited ETH is frozen on Unichain and unavailable on L1. The frozen amount grows linearly with the number of stuffed cycles (K × average daily deposit volume), violating the operational invariant that at most one cycle's worth of deposits is ever pending. This matches the explicitly allowed impact "Low. Block stuffing" and "Medium. Temporary freezing of funds."

## Likelihood Explanation
Unichain's low block gas limit makes block stuffing significantly cheaper than on Ethereum mainnet. The attacker only needs to sustain stuffing for the duration of the operator's scheduled call window (e.g., a few minutes around a daily trigger). No special permissions or insider access are required — any funded account can submit high-gas-price filler transactions. The cost scales with block gas limit and window duration, not with the amount of ETH being frozen, making this economically viable for a motivated attacker targeting a high-TVL pool.

## Recommendation
1. **Retry with escalating gas price**: Operate the bridger with a retry loop that resubmits `bridgeAssetsViaNativeBridge` with a higher priority fee if the transaction is not included within N blocks, outbidding stuffing transactions.
2. **Add a maximum accumulation guard**: Emit an on-chain alert or revert new deposits if `ethBalanceMinusFees` exceeds a configurable threshold, signalling that bridging is overdue.
3. **Consider a permissionless bridging path**: Allow any caller to trigger bridging after a configurable delay has elapsed since the last bridge call, removing the single-operator dependency.
4. **Off-chain monitoring**: Detect when a submitted `bridgeAssetsViaNativeBridge` transaction is not mined within an expected window and escalate gas price automatically.

## Proof of Concept
Deploy `RSETHPoolNoWrapper` on a local Unichain fork. Run the following invariant test:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract BlockStuffingPoC {
    RSETHPoolNoWrapper pool;
    address bridger;

    uint256 constant DAILY_DEPOSIT = 10 ether;
    uint256 constant K = 3;

    function testBlockStuffingAccumulation() external {
        for (uint256 i = 0; i < K; i++) {
            vm.deal(address(this), DAILY_DEPOSIT);
            pool.swapETHForRsETH{value: DAILY_DEPOSIT}(...);
            // Attacker stuffs blocks: operator tx cannot be included.
            // Simulate by NOT calling bridgeAssetsViaNativeBridge here.
            vm.roll(block.number + BLOCKS_PER_DAY);
        }

        uint256 frozen = pool.getETHBalanceMinusFees();
        // frozen ≈ K * DAILY_DEPOSIT — one-cycle invariant violated
        assertApproxEqAbs(frozen, K * DAILY_DEPOSIT, FEE_TOLERANCE);

        vm.prank(bridger);
        pool.bridgeAssetsViaNativeBridge();

        // All K cycles bridged in one call — freeze was real but temporary
        assertEq(pool.getETHBalanceMinusFees(), 0);
    }
}
```

The test confirms `frozen` grows as `K × DAILY_DEPOSIT`, violating the one-cycle bound, and that a single post-stuffing call to `bridgeAssetsViaNativeBridge()` successfully bridges the entire accumulated balance.
