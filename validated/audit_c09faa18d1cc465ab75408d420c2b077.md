The code confirms this is a real, reachable vulnerability. Here is the full analysis:

---

### Title
Unauthenticated P2P Peer Can Inject Mismatched Transaction Type/Hash Pairs Into Node Storage, Corrupting `starknet_getTransactionByHash` RPC Responses — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `FullTransaction` messages from unauthenticated p2p peers and stores the peer-supplied `transaction` and `transaction_hash` fields independently, with no check that the hash is actually the hash of the accompanying transaction. A malicious peer can send `Transaction::Invoke(...)` paired with a `TransactionHash` that is the hash of a `Transaction::Declare` or `Transaction::DeployAccount`, causing the syncing node to persist a type-hash mismatch that is then served verbatim by the RPC.

---

### Finding Description

In `parse_data_for_block`, the three fields of each `FullTransaction` are pushed directly into `block_body` with no cross-validation:

```rust
block_body.transactions.push(transaction);          // line 86
block_body.transaction_outputs.push(transaction_output); // line 87
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);   // line 89
``` [1](#0-0) 

The developer-acknowledged TODO at line 88 confirms this validation is intentionally deferred and not yet implemented. [2](#0-1) 

The resulting `BlockBody` is written to storage via `append_body` without any hash-type consistency check at the storage layer either:

```rust
storage_writer.begin_rw_txn()?.append_body(self.1, self.0)?.commit()?;
``` [3](#0-2) 

`write_transactions` in the storage layer simply zips `transactions`, `transaction_outputs`, and `transaction_hashes` together and stores them as-is, with no invariant check: [4](#0-3) 

The block header already stored for the same block number contains a `transaction_commitment` field (a Merkle/Poseidon commitment over the transaction hashes), but `parse_data_for_block` never reads or verifies it against the received hashes. [5](#0-4) 

The RPC handler `get_transaction_by_hash` then looks up the stored `Transaction` by the stored `TransactionHash` index and returns them together without re-deriving or re-checking the hash:

```rust
Ok(TransactionWithHash { transaction: transaction.try_into()?, transaction_hash })
``` [6](#0-5) 

---

### Impact Explanation

**High** — RPC returns an authoritative-looking wrong value.

A node syncing via p2p will serve `starknet_getTransactionByHash(declare_hash)` with a response whose `type` field is `INVOKE` (or vice versa), violating the invariant that the transaction type encoded in the stored `Transaction` matches the type implied by the stored `TransactionHash`. Clients relying on the returned type to dispatch execution logic (e.g., wallets, explorers, re-execution tools) will receive a structurally incorrect response. The corruption is persistent (written to LMDB) and survives node restarts.

The question's "Critical" framing is overstated: this does not affect consensus, block production, execution, or the actual on-chain state. The impact is bounded to the corrupted node's RPC authority.

---

### Likelihood Explanation

Any unauthenticated peer on the p2p network can trigger this. The p2p sync client connects to peers without authentication and processes their responses directly. The only guard is the transaction count check (`n_transactions` from the stored header), which the attacker satisfies by sending the correct number of `FullTransaction` messages. No cryptographic barrier prevents the type-hash mismatch.

---

### Recommendation

In `parse_data_for_block`, after receiving each `FullTransaction`, recompute the transaction hash from the `transaction` body and `chain_id`, then assert it equals the peer-supplied `transaction_hash`. Reject the peer (return `ParseDataError::BadPeer`) on mismatch. Alternatively, verify the full set of received hashes against the `transaction_commitment` already stored in the block header before writing the body to storage.

---

### Proof of Concept

1. Operate a malicious p2p peer that responds to `TransactionQuery` for block N.
2. For each transaction slot, send a `FullTransaction` where `transaction = Transaction::Invoke(valid_invoke_fields)` but `transaction_hash = hash_of_a_declare_tx_in_that_block`.
3. The syncing node's `parse_data_for_block` accepts all messages (count matches `n_transactions`), pushes the mismatched pairs, and calls `append_body`.
4. Query the node's RPC: `starknet_getTransactionByHash(declare_hash)` returns `{ "type": "INVOKE", ... }` — wrong type for the given hash.

A property-based test asserting `get_transaction_hash(stored_tx, chain_id) == stored_tx_hash` for all entries written by `parse_data_for_block` would fail under this attack.
