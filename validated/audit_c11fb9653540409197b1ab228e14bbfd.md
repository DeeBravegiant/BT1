Audit Report

## Title
Unguarded `sendFunds()` Allows Anyone to Front-Run MEV Reward Distribution and Steal Yield - (File: contracts/FeeReceiver.sol)

## Summary

`FeeReceiver.sendFunds()` carries no access control modifier, allowing any external caller to flush accumulated MEV/execution-layer ETH rewards into `LRTDepositPool` at an arbitrary time. Because the deposit pool's raw ETH balance is directly included in the TVL used to compute the rsETH price, an attacker can sandwich this flush between a deposit and a withdrawal initiation to capture a disproportionate share of rewards that rightfully belong to existing rsETH holders.

## Finding Description

`FeeReceiver.sendFunds()` is declared `external` with no role check:

```solidity
// contracts/FeeReceiver.sol L53-58
function sendFunds() external {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

The receiving function in `LRTDepositPool` is also unguarded:

```solidity
// contracts/LRTDepositPool.sol L61
function receiveFromRewardReceiver() external payable { }
```

The ETH received is immediately reflected in `getETHDistributionData()`:

```solidity
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

`_getTotalEthInProtocol()` in `LRTOracle` calls `getTotalAssetDeposits()` → `getETHDistributionData()`, so the newly arrived reward ETH is immediately included in the TVL used by `_updateRsETHPrice()`. `updateRSETHPrice()` is itself permissionless:

```solidity
// contracts/LRTOracle.sol L87-89
function updateRSETHPrice() public whenNotPaused {
    _updateRsETHPrice();
}
```

**Exploit path:**

1. Observe `FeeReceiver` holds `R` ETH in accumulated rewards (publicly visible on-chain).
2. Call `LRTDepositPool.depositETH{value: D}(0, "")` at the current price `P = TVL/S`. Receive `D/P` rsETH. New supply = `S + D/P`.
3. Call `FeeReceiver.sendFunds()`. `R` ETH moves to `LRTDepositPool`; new TVL = `TVL + D + R`.
4. Call `LRTOracle.updateRSETHPrice()`. New price = `(TVL + D + R) / (S + D/P)`. Since `P = TVL/S`, this simplifies to `P * (TVL + D + R) / (TVL + D)` — strictly greater than `P`.
5. Call `LRTWithdrawalManager.initiateWithdrawal(...)` at the new higher price, locking in the inflated ETH entitlement.
6. After `withdrawalDelayBlocks` (~8 days), claim. Net gain = `R * D / (TVL + D)` ETH, taken directly from existing holders' unclaimed yield.

**Why existing guards are insufficient:**

The `pricePercentageLimit` check (lines 263–265 of `LRTOracle.sol`) only blocks `updateRSETHPrice()` when the price increase exceeds the configured threshold. When `pricePercentageLimit` is zero (disabled) or when accumulated rewards are small relative to TVL, the check does not trigger and the full attack succeeds. Even when the limit is active, the attacker can wait for reward accumulations that stay just below the threshold.

## Impact Explanation

**High — Theft of unclaimed yield.**

Existing rsETH holders have an accrued but not-yet-distributed claim on the MEV rewards sitting in `FeeReceiver`. The attack dilutes that claim: if existing holders would have received `R` ETH in rewards, after the attack they receive only `R * TVL / (TVL + D)`, with the remainder `R * D / (TVL + D)` captured by the attacker. This is a direct, quantifiable transfer of yield from legitimate holders to the attacker, matching the "Theft of unclaimed yield" impact class exactly.

## Likelihood Explanation

The attack is fully permissionless — no special role, no flash loan, no oracle manipulation required. The only friction is the `withdrawalDelayBlocks` (~8 days), which delays profit realization but does not prevent it, since the withdrawal price is locked at initiation. The `FeeReceiver` balance is publicly visible on-chain, making it straightforward to monitor and time. Any address with enough ETH to deposit can execute this repeatedly whenever rewards accumulate.

## Recommendation

Add an access control modifier to `sendFunds()` so only a trusted role can trigger the reward flush:

```solidity
function sendFunds() external onlyRole(LRTConstants.MANAGER) {
    uint256 balance = address(this).balance;
    ILRTDepositPool(depositPool).receiveFromRewardReceiver{ value: balance }();
    emit MevRewardsAddedToTVL(balance);
}
```

Optionally, also restrict `receiveFromRewardReceiver()` in `LRTDepositPool` to only accept calls from the registered `FeeReceiver` address, providing defense-in-depth.
