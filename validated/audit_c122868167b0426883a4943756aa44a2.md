Audit Report

## Title
Stale Cached `rsETHPrice` Allows Depositors to Receive Excess rsETH, Stealing Yield from Existing Holders — (File: `contracts/LRTOracle.sol`, `contracts/LRTDepositPool.sol`)

## Summary
`LRTDepositPool.getRsETHAmountToMint()` divides by the cached `rsETHPrice` state variable rather than a live-computed rate. Because `rsETHPrice` is only updated when `_updateRsETHPrice()` is invoked, any gap between updates allows a depositor to receive more rsETH than the actual TVL justifies. When the price is subsequently updated, the attacker's inflated rsETH balance is redeemed at the higher rate, permanently diluting the yield that existing holders had accrued.

## Finding Description
`rsETHPrice` is a persistent state variable updated only inside `_updateRsETHPrice()`: [1](#0-0) [2](#0-1) 

`updateRSETHPrice()` is `public` with no access control beyond `whenNotPaused`, so any caller can invoke it: [3](#0-2) 

`getRsETHAmountToMint()` reads the stale cached value directly instead of computing the live rate: [4](#0-3) 

The live rate is only computed inside `_getTotalEthInProtocol()`, which is `private` and only called from `_updateRsETHPrice()`: [5](#0-4) 

**Exploit path:**

1. Rewards accrue between price updates: `rsETHPrice` (cached) = 1.00 ETH, actual TVL/supply = 1.05 ETH.
2. Attacker calls `depositETH(100 ETH)`. `getRsETHAmountToMint` computes `100e18 * 1e18 / 1.00e18 = 100 rsETH`. Correct amount would be `100 / 1.05 ≈ 95.24 rsETH`.
3. Attacker calls `updateRSETHPrice()`. New TVL = 205 ETH, new supply = 200 rsETH, new price = `205/200 = 1.025 ETH` (diluted from 1.05).
4. Attacker calls `initiateWithdrawal(ETH, 100 rsETH)`. `getExpectedAssetAmount` = `100 * 1.025 / 1.0 = 102.5 ETH`, stored as `expectedAssetAmount`. [6](#0-5) 
5. After the withdrawal delay, `unlockQueue()` is called by an operator. `_calculatePayoutAmount` returns `min(request.expectedAssetAmount, currentReturn)`. With price unchanged at 1.025, `currentReturn = 102.5 ETH = expectedAssetAmount`, so payout = 102.5 ETH. [7](#0-6) 
6. Attacker completes withdrawal, receives 102.5 ETH. Net profit: +2.5 ETH. Alice's 100 rsETH is now worth 102.5 ETH instead of 105 ETH — 2.5 ETH of accrued yield stolen.

**Why existing guards are insufficient:**

- The `pricePercentageLimit` check only blocks non-manager callers when the price increase from `highestRsethPrice` exceeds the configured threshold. If `pricePercentageLimit == 0` (no cap), or if the reward accrual is within the limit, the attacker can call `updateRSETHPrice()` directly. Alternatively, the attacker can simply wait for the off-chain bot to update the price — no special call is needed. [8](#0-7) 
- The `min()` in `_calculatePayoutAmount` protects against price increases *after* initiation but does not undo the inflation already baked into `expectedAssetAmount` at initiation time.
- The `ExceedAmountToWithdraw` check in `initiateWithdrawal` passes because the attacker's 100 ETH deposit increases total available assets by 100 ETH, which covers the 102.5 ETH withdrawal request. [9](#0-8) 

## Impact Explanation
**High — Theft of unclaimed yield.** Existing rsETH holders continuously accrue yield as underlying assets appreciate. This yield is reflected in the rising actual exchange rate but is not yet captured in the stale `rsETHPrice`. A depositor who enters at the stale price receives a larger share of the rsETH supply than the TVL justifies. When `updateRSETHPrice()` is called, the new price is computed over a larger supply, permanently diluting the yield existing holders had accrued. The attacker exits with more assets than deposited; the loss is borne proportionally by all existing rsETH holders.

## Likelihood Explanation
**Medium.** The exploitable window exists during every normal price-update interval (routine bot scheduling, gas spikes, or network congestion). No special privileges are required — any EOA can call `depositETH()` and `updateRSETHPrice()`. The `pricePercentageLimit` check is a partial mitigation but is bypassable when `pricePercentageLimit == 0`, when reward accrual is within the configured limit, or when the attacker simply waits for the bot to update the price. The attack is repeatable on every update cycle.

## Recommendation
Compute the rsETH mint amount using the live exchange rate rather than the stale cached `rsETHPrice`. `getRsETHAmountToMint()` should derive the current rate on-the-fly by calling `_getTotalEthInProtocol()` (or an equivalent public view) and dividing by `rsethSupply`, rather than reading `rsETHPrice` from storage. Alternatively, atomically call `_updateRsETHPrice()` (or an internal equivalent) at the start of every deposit transaction so the cached price is always fresh before minting is computed.

## Proof of Concept

```
// Foundry fork test outline
function test_stalePrice_yieldTheft() public {
    // 1. Setup: rsETHPrice = 1.00e18, actual TVL/supply = 1.05e18
    //    Alice holds 100e18 rsETH; simulate reward accrual without price update.

    // 2. Attacker deposits 100 ETH
    vm.prank(attacker);
    lrtDepositPool.depositETH{value: 100 ether}(0, "");
    uint256 attackerRsETH = rsETH.balanceOf(attacker);
    assertGt(attackerRsETH, 95.24 ether); // inflated: should be ~95.24, gets 100

    // 3. Attacker updates price
    vm.prank(attacker);
    lrtOracle.updateRSETHPrice();
    uint256 newPrice = lrtOracle.rsETHPrice();
    assertLt(newPrice, 1.05e18); // diluted: 1.025e18

    // 4. Attacker initiates withdrawal
    vm.prank(attacker);
    rsETH.approve(address(lrtWithdrawalManager), attackerRsETH);
    vm.prank(attacker);
    lrtWithdrawalManager.initiateWithdrawal(ETH_TOKEN, attackerRsETH, "");

    // 5. Advance blocks past withdrawal delay, operator unlocks queue
    vm.roll(block.number + withdrawalDelayBlocks + 1);
    vm.prank(operator);
    lrtWithdrawalManager.unlockQueue(ETH_TOKEN, type(uint256).max, 0, type(uint256).max, 0, type(uint256).max);

    // 6. Attacker completes withdrawal
    uint256 balBefore = attacker.balance;
    vm.prank(attacker);
    lrtWithdrawalManager.completeWithdrawal(ETH_TOKEN, "");
    uint256 profit = attacker.balance - balBefore - 100 ether;
    assertGt(profit, 0); // attacker received > 100 ETH back

    // 7. Verify Alice's yield was diluted
    uint256 aliceRedeemable = lrtWithdrawalManager.getExpectedAssetAmount(ETH_TOKEN, 100 ether);
    assertLt(aliceRedeemable, 105 ether); // Alice lost yield
}
```
