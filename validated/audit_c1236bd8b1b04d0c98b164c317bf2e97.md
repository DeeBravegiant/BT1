### Title
Integer Division Truncation in NFT Royalty Calculation Silently Bypasses Royalty Payments in Offer Settlement - (File: chia/wallet/nft_wallet/nft_wallet.py)

### Summary
`compute_royalty_amount` uses sequential integer floor-division that truncates to zero for small offered amounts. When the computed royalty is zero, `make_nft1_offer` silently skips the royalty coin creation and passes an empty `trade_prices_list` to the NFT ownership-layer puzzle, so the on-chain transfer program enforces no royalty at all. An unprivileged NFT seller can deliberately trigger this by listing the NFT for a dust amount and settling the real payment off-chain, permanently depriving the royalty recipient of their entitled share.

### Finding Description

`compute_royalty_amount` in `chia/wallet/nft_wallet/nft_wallet.py` computes the royalty as:

```python
amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
``` [1](#0-0) 

Two sequential integer floor-divisions are applied. For any `offered_amount` where `abs(offered_amount) // royalty_split * percentage < 10000`, the result is 0. The project's own test suite explicitly documents and accepts this:

```python
def test_small_amount_truncates_to_zero() -> None:
    result = compute_royalty_amount(offered_amount=-50, royalty_split=1, percentage=100)
    assert result == uint64(0)
``` [2](#0-1) 

When `compute_royalty_amount` returns 0, `make_nft1_offer` takes two compounding actions that together eliminate on-chain royalty enforcement:

**1. The royalty coin is never created.** The payment list sum is 0, so the entire royalty spend path is skipped:

```python
if sum(p.amount for _, p in payments) == 0:
    continue
``` [3](#0-2) 

**2. The `trade_prices_list` passed to the NFT puzzle is filtered to empty.** Entries where `price[0] * royalty_percentage // 10000 == 0` are excluded:

```python
trade_prices_list=[
    list(price)
    for price in trade_prices
    if price[0] * offered_royalty_percentages[asset] // MAX_ROYALTY_BASIS_POINTS != 0
],
``` [4](#0-3) 

This empty list is injected into the NFT singleton spend via condition opcode `-10` (the ownership-layer transfer condition):

```python
UnknownCondition(
    opcode=Program.to(-10),
    args=[
        Program.to(new_owner),
        Program.to(trade_prices_list),
        Program.to(new_did_inner_hash),
    ],
),
``` [5](#0-4) 

The NFT1 royalty transfer program puzzle uses the `trade_prices_list` to derive the required royalty outputs. With an empty list, the puzzle imposes no royalty obligation, and the spend is accepted by consensus without any royalty coin being created.

The same truncation exists in the display-path `royalty_calculation` static method:

```python
"amount": abs(amount) // len(royalty_assets_dict) * percentage // MAX_ROYALTY_BASIS_POINTS,
``` [6](#0-5) 

### Impact Explanation

The royalty recipient (the original NFT creator, encoded permanently in the NFT puzzle) loses their entitled royalty on every sale where the seller deliberately sets a dust offer price. Because the NFT transfer program puzzle receives an empty `trade_prices_list`, the on-chain enforcement layer is completely bypassed — this is not a wallet display error but a consensus-accepted spend with zero royalty output. The royalty recipient's XCH or CAT coins that should have been created are simply never created, constituting an unauthorized accounting change affecting XCH, CATs, and NFT-linked singleton assets in offer settlement.

### Likelihood Explanation

Any current NFT owner (unprivileged) can exploit this. The seller creates a legitimate offer for 1–9999 mojos (depending on royalty basis points), the colluding buyer accepts it on-chain, and the real consideration is transferred off-chain. No keys, admin access, or cryptographic break is required. The threshold below which royalty truncates to zero is:

```
offered_amount < MAX_ROYALTY_BASIS_POINTS / percentage
```

For a 10% royalty (1000 bps): threshold = 10 mojos. For 1% (100 bps): threshold = 100 mojos. For 0.1% (10 bps): threshold = 1000 mojos. All are trivially small amounts that any wallet can produce.

### Recommendation

1. **Reject offers where royalty truncates to zero.** In `compute_royalty_amount`, raise a `ValueError` (or return a sentinel) when `amount == 0` and `percentage > 0`, preventing offer creation rather than silently omitting the royalty.
2. **Do not filter `trade_prices_list` entries that produce zero royalty.** Instead, include the full price entry so the NFT puzzle can enforce the royalty obligation even when the computed amount is sub-mojo — or reject the offer outright.
3. **Add a minimum fungible amount guard** in `make_nft1_offer` that ensures `offered_amount >= MAX_ROYALTY_BASIS_POINTS // percentage` for any royalty-bearing NFT in the offer.

### Proof of Concept

1. Alice mints an NFT with 10% royalty (1000 basis points) to her royalty address.
2. Bob acquires the NFT legitimately.
3. Bob creates an NFT1 offer: sell NFT, request **9 mojos** XCH.
4. Wallet computes: `9 // 1 * 1000 // 10000 = 9000 // 10000 = 0` → royalty = 0.
5. `make_nft1_offer` skips royalty coin creation; `trade_prices_list = []` is passed to the NFT puzzle.
6. Carol (colluding buyer) accepts the offer on-chain; pays 9 mojos; receives the NFT.
7. Carol separately sends Bob 1 XCH off-chain.
8. Alice's royalty address receives **0 mojos** instead of the 0.1 XCH she is entitled to.
9. The spend is accepted by consensus because the NFT transfer program puzzle, given an empty `trade_prices_list`, imposes no royalty output requirement. [1](#0-0) [7](#0-6) [3](#0-2)

def compute_royalty_amount(offered_amount: int, royalty_split: int, percentage: int) -> uint64:
    """Compute royalty using integer arithmetic, validating against overflow and excessive percentage."""
    if percentage > MAX_ROYALTY_BASIS_POINTS:
        raise ValueError(f"NFT royalty percentage {percentage} exceeds 100% ({MAX_ROYALTY_BASIS_POINTS} basis points)")
    amount = abs(offered_amount) // royalty_split * percentage // MAX_ROYALTY_BASIS_POINTS
    royalty = uint64(amount)
    if royalty >= abs(offered_amount):
        raise ValueError("Royalty amount meets or exceeds the offered amount")
    return royalty
```
