Audit Report

## Title
Arithmetic Underflow in `bridgeKernelToBSC` Temporarily Blocks Bridging When `counter == 0` — (`contracts/KERNEL/KernelVaultETH.sol`)

## Summary
`bridgeKernelToBSC` unconditionally executes `lastBridgedDepositId = counter - 1` at line 262 before performing the bridge. When `counter == 0` — the default state from deployment until the first `depositKernel` call — Solidity 0.8.27 checked arithmetic causes an underflow panic, making the function uncallable. Any KERNEL balance in the vault sourced from a direct ERC20 transfer cannot be bridged during this window. The freeze is temporary: the operator can self-remediate by calling `depositKernel(minDeposit)` to set `counter = 1`, after which `counter - 1 = 0` is valid and bridging succeeds.

## Finding Description
`counter` is a `uint256` state variable initialized to `0` by default; `initialize()` does not set it. [1](#0-0) 

`_depositKernel` is the sole code path that increments `counter` via `++counter`. [2](#0-1) 

`bridgeKernelToBSC` performs no guard on `counter` before computing `lastBridgedDepositId = counter - 1`. [3](#0-2) 

Under Solidity 0.8.27 checked arithmetic, `0 - 1` panics unconditionally. Because `KERNEL` is a plain ERC20 with no transfer hooks, any token holder can call `kernel.transfer(address(vault), amount)` to place tokens in the vault without going through `depositKernel`, leaving `counter` at `0`. [4](#0-3) 

There is no rescue or sweep function in `KernelVaultETH`; `bridgeKernelToBSC` is the only egress path for KERNEL tokens. [5](#0-4) 

**Exploit path:**
1. Contract is deployed; `counter == 0`.
2. Any token holder calls `kernel.transfer(address(vault), amount)` — a public ERC20 action requiring no privilege.
3. Operator calls `bridgeKernelToBSC` — panics at `counter - 1` (underflow).
4. Tokens in the vault are unbridge-able until the operator calls `depositKernel(minDeposit)`, incrementing `counter` to `1`.
5. After step 4, `bridgeKernelToBSC` succeeds and bridges the full vault balance.

## Impact Explanation
**Medium — Temporary freezing of funds.** KERNEL tokens present in the vault (including any directly-transferred balance) cannot be bridged while `counter == 0`. The freeze is not permanent: the operator can self-remediate by making one `depositKernel` call. However, the condition is externally triggerable by any unprivileged token holder and silently blocks the bridge egress path during the deployment window.

## Likelihood Explanation
Low-to-medium. The vulnerable window exists from deployment until the first `depositKernel` call. A direct ERC20 transfer to the vault is trivially executable by any KERNEL holder with no special access. The operator can self-remediate, but the bug is silently present and could delay or block bridging operations.

## Recommendation
Add a guard in `bridgeKernelToBSC` before computing `lastBridgedDepositId`:

```solidity
if (counter == 0) revert NoDepositsYet();
lastBridgedDepositId = counter - 1;
```

Alternatively, initialize `lastBridgedDepositId` to `type(uint256).max` as a sentinel, or restructure bookkeeping so `lastBridgedDepositId` is only updated when `counter > 0`.

## Proof of Concept
```solidity
// 1. Deploy KernelVaultETH (counter == 0)
// 2. Any token holder directly transfers KERNEL to vault
kernel.transfer(address(vault), 1e18);
// counter is still 0

// 3. Operator attempts to bridge — reverts with arithmetic underflow panic
vault.bridgeKernelToBSC{value: fee}(1e18, 0.99e18, fee, refundAddress);
// ↑ panics at: lastBridgedDepositId = counter - 1  (0 - 1 underflows)

// 4. Operator self-remediates
kernel.approve(address(vault), minDeposit);
vault.depositKernel(minDeposit); // counter becomes 1

// 5. Bridge now succeeds
vault.bridgeKernelToBSC{value: fee}(vault.kernel().balanceOf(address(vault)), 0.99e18, fee, refundAddress);
```

A Foundry fork test can confirm the panic by deploying the contract, calling `kernel.transfer` to the vault address, then calling `bridgeKernelToBSC` and asserting it reverts with `Panic(0x11)`.

    }
