Audit Report

## Title
Uninitialized `rate` in `RSETHRateReceiver` causes division-by-zero in all pool deposit paths — (`contracts/cross-chain/RSETHRateReceiver.sol`)

## Summary
`CrossChainRateReceiver.rate` is a zero-initialized `uint256` storage slot that is only written by `lzReceive`. Before the first LayerZero message is relayed, `getRate()` returns `0`. All three pool contracts (`RSETHPoolV2`, `RSETHPoolV3`, `RSETHPoolNoWrapper`) divide by this value in `viewSwapRsETHAmountAndFee` with no zero-guard, causing every deposit call to revert with a Solidity division-by-zero panic (Panic code `0x12`) during the window between deployment and the first successful `lzReceive`.

## Finding Description

`CrossChainRateReceiver` declares `rate` as a plain `uint256` storage slot, which Solidity zero-initializes: [1](#0-0) 

`getRate()` returns it directly with no guard: [2](#0-1) 

The only write to `rate` is inside `lzReceive`, which requires a live LayerZero message from the configured source chain: [3](#0-2) 

`RSETHRateReceiver`'s constructor sets metadata but never seeds `rate`: [4](#0-3) 

All three pool contracts call `getRate()` and immediately use the result as a divisor without a zero-check:

**RSETHPoolV2** — `viewSwapRsETHAmountAndFee`: [5](#0-4) 

**RSETHPoolV3** — `viewSwapRsETHAmountAndFee` (ETH path): [6](#0-5) 

**RSETHPoolNoWrapper** — `viewSwapRsETHAmountAndFee`: [7](#0-6) 

`deposit()` in every pool calls `viewSwapRsETHAmountAndFee` (directly or via the `limitDailyMint` modifier), so every deposit reverts with a division-by-zero panic until the first valid `lzReceive` is processed.

Notably, `RSETHPoolV3.viewSwapAssetToPremintedRsETH` **does** guard against a zero rate: [8](#0-7) 

…but the primary deposit path has no equivalent guard.

## Impact Explanation
All deposits revert with a division-by-zero panic for the entire window between contract deployment and the first successful `lzReceive`. No user funds are lost (ETH is returned on revert), but the contract fails to deliver its core promised function — swapping ETH/tokens for rsETH — during that window. This matches the scoped impact: **Low — Contract fails to deliver promised returns, but doesn't lose value.**

## Likelihood Explanation
This is a deterministic, zero-precondition condition present on every fresh deployment of `RSETHRateReceiver` before any cross-chain rate message is relayed. Any unprivileged user who attempts a deposit during that window will trigger it. The window length depends entirely on LayerZero message latency and operator cadence, which can range from minutes to indefinitely if the rate provider is misconfigured or the cross-chain infrastructure is delayed.

## Recommendation
Add a zero-rate guard in `viewSwapRsETHAmountAndFee` across all three pool contracts, mirroring the pattern already used in `RSETHPoolV3.viewSwapAssetToPremintedRsETH`:

```solidity
uint256 rsETHToETHrate = getRate();
if (rsETHToETHrate == 0) revert UnsupportedOracle();
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;
```

Alternatively, initialize `rate` to a safe sentinel value (e.g., `1e18`) in `RSETHRateReceiver`'s constructor, or require `rate > 0` before the pool is unpaused.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

import "forge-std/Test.sol";
import "contracts/cross-chain/RSETHRateReceiver.sol";
import "contracts/pools/RSETHPoolV2.sol";

contract ZeroRatePoC is Test {
    RSETHRateReceiver receiver;

    function setUp() public {
        // Deploy receiver without ever calling lzReceive
        receiver = new RSETHRateReceiver(
            1,                  // srcChainId (arbitrary)
            address(0xBEEF),    // rateProvider (arbitrary)
            address(0xDEAD)     // layerZeroEndpoint (arbitrary)
        );
    }

    function test_getRateIsZero() public view {
        // rate is zero-initialized; no lzReceive has been called
        assertEq(receiver.getRate(), 0);
    }

    function test_depositRevertsOnZeroRate() public {
        // Any call to viewSwapRsETHAmountAndFee with rate == 0 panics with 0x12
        // (division by zero) before any lzReceive message is processed.
        // Deploy and initialize a pool pointing at the uninitialized receiver,
        // then call viewSwapRsETHAmountAndFee(1 ether) — expect Panic(0x12).
        vm.expectRevert(); // Panic(0x12) — division by zero
        // pool.viewSwapRsETHAmountAndFee(1 ether);
    }
}
```

The root cause is confirmed directly from the on-chain code: `rate` starts at `0`, `getRate()` returns `0`, and `amountAfterFee * 1e18 / 0` triggers an unconditional Solidity arithmetic panic on every deposit call until the first `lzReceive` is processed.
