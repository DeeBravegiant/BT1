### Title
`verify_foreign_transaction()` Discards Caller Identity, Making Requests Caller-Agnostic and Enabling Cross-User Response Sharing — (File: `crates/contract/src/lib.rs`)

---

### Summary

The `verify_foreign_transaction()` function in `MpcContract` silently discards the `predecessor` (caller identity) returned by `check_request_preconditions()`. Unlike `sign()` and `request_app_private_key()`, which both bind their requests to the caller's identity via a `tweak` derived from `(predecessor_id, path)`, `verify_foreign_transaction()` produces a caller-agnostic `VerifyForeignTransactionRequest` with no caller binding. Any unprivileged account — including a malicious intermediary contract — can submit the same request and receive the same MPC-signed response, breaking the request-lifecycle accounting invariant that each caller's request should be independent and attributed to them.

---

### Finding Description

`check_request_preconditions()` is the shared precondition helper for all three user-facing request methods. It returns `(DomainConfig, AccountId)` where the `AccountId` is `env::predecessor_account_id()` — the caller's identity. [1](#0-0) 

`sign()` and `request_app_private_key()` both capture and use this predecessor to bind the request to the caller:

```rust
// sign()
let (domain_config, predecessor) = self.check_request_preconditions(...);
let request = SignatureRequest::new(request.domain_id, request.payload, &predecessor, &request.path);
``` [2](#0-1) 

```rust
// request_app_private_key()
let (_, predecessor) = self.check_request_preconditions(...);
let request = CKDRequest::new(request.app_public_key, domain_id, &predecessor, &request.derivation_path);
``` [3](#0-2) 

`verify_foreign_transaction()`, however, **discards the return value entirely**:

```rust
// verify_foreign_transaction()
self.check_request_preconditions(   // ← return value discarded
    request.domain_id,
    DomainPurpose::ForeignTx,
    ...
);
// predecessor is never captured or used
let request = args_into_verify_foreign_tx_request(request);
``` [4](#0-3) 

`args_into_verify_foreign_tx_request` simply copies the three caller-agnostic fields — no `predecessor_id`, no `tweak`:

```rust
pub fn args_into_verify_foreign_tx_request(args: ...) -> VerifyForeignTransactionRequest {
    VerifyForeignTransactionRequest {
        domain_id: args.domain_id,
        request: args.request,
        payload_version: args.payload_version,
    }
}
``` [5](#0-4) 

The `VerifyForeignTransactionRequest` struct itself has no `tweak` or `predecessor_id` field: [6](#0-5) 

This is in direct contradiction with the design document, which specifies that the request **should** carry a `tweak` derived from `(predecessor_id, derivation_path)` using a foreign-tx-specific prefix, and that `VerifyForeignTransactionRequestArgs` should include a `derivation_path` field: [7](#0-6) 

Furthermore, `respond_verify_foreign_tx()` verifies the MPC signature against the **root public key** (not a derived key), confirming the caller binding is entirely absent end-to-end: [8](#0-7) 

A unit test explicitly documents the resulting caller-agnostic behavior — two different callers (Alice and Bob) submitting the same request are both queued under the **same** map key and both receive the same response from a single MPC reply: [9](#0-8) 

---

### Impact Explanation

**Medium.** The request-lifecycle accounting invariant is broken: `verify_foreign_transaction` requests are not bound to the submitting caller's identity, unlike every other user-facing request type. Concretely:

1. **Cross-user response sharing**: Any account that submits an identical `VerifyForeignTransactionRequestArgs` is queued under the same map key and receives the same MPC-signed response. A single MPC reply drains all queued yields simultaneously, meaning a caller who did not initiate the verification work receives the same attestation as the one who did.

2. **Malicious intermediary contract**: A contract acting as a proxy can call `verify_foreign_transaction()` on behalf of a user. Because the predecessor is discarded and the request key is caller-agnostic, the proxy contract receives the MPC-signed response in its own promise callback. The actual user whose foreign-chain transaction was verified gets no response — their yield is consumed by the proxy's queue entry. This is the direct analog of the SiloGateway bug: the intermediary (gateway/proxy) receives the position/response that should have been attributed to the user.

3. **Design-intent violation**: The design document specifies that `verify_foreign_transaction` should derive a user-specific key via `(predecessor_id, derivation_path)` with a distinct prefix, ensuring the resulting signature is cryptographically bound to the requesting account. The current implementation produces a root-key signature with no such binding, meaning the attestation cannot be used to prove *who* requested the verification.

---

### Likelihood Explanation

**High.** The entry path requires no privilege: any NEAR account can call `verify_foreign_transaction()` with a 1 yoctoNEAR deposit. The caller-agnostic behavior is confirmed by an existing unit test. Any contract acting as a bridge, aggregator, or proxy that forwards `verify_foreign_transaction` calls will silently receive responses intended for its users, with no on-chain mechanism to detect or prevent this.

---

### Recommendation

1. **Capture the predecessor** returned by `check_request_preconditions()` in `verify_foreign_transaction()`, matching the pattern used by `sign()` and `request_app_private_key()`.

2. **Add `predecessor_id` and `derivation_path` to `VerifyForeignTransactionRequestArgs`** and derive a `tweak` from them using the foreign-tx-specific prefix described in the design document.

3. **Include the `tweak` in `VerifyForeignTransactionRequest`** so the map key is caller-specific, preventing cross-user response sharing.

4. **Verify the MPC response against the derived key** (not the root key) in `respond_verify_foreign_tx()`, consistent with how `respond()` handles `SignatureRequest`.

---

### Proof of Concept

```
1. Alice calls verify_foreign_transaction({bitcoin_tx_id: X, domain_id: 0, ...})
   with 1 yoctoNEAR deposit.
   → VerifyForeignTransactionRequest{request: Bitcoin(X), domain_id: 0, ...} is stored
     under key K in pending_verify_foreign_tx_requests.

2. Mallory (a malicious proxy contract) calls verify_foreign_transaction with the
   identical arguments before the MPC nodes respond.
   → The same key K is used; Mallory's yield index is appended to K's queue.

3. MPC nodes respond via respond_verify_foreign_tx(K, signature_over_root_key).
   → Both Alice's and Mallory's yields are resolved with the same signature.

4. Mallory's promise callback receives the MPC-signed attestation that Bitcoin tx X
   was verified. Mallory can present this attestation to any downstream protocol as
   proof of foreign-chain verification, without Alice's consent and without the
   attestation being bound to any specific NEAR account.

5. If Mallory submitted first and Alice second, Alice's yield is also resolved —
   but if Mallory is the sole submitter acting on Alice's behalf (e.g., Alice called
   Mallory's contract expecting it to forward the call and return the result),
   Mallory intercepts the response and Alice receives nothing.
```

The root cause — discarding `predecessor` and omitting `tweak` from `VerifyForeignTransactionRequest` — is confirmed at: [10](#0-9) [5](#0-4) [6](#0-5)

    fn check_request_preconditions(
        &self,
        domain_id: DomainId,
        expected_purpose: DomainPurpose,
        minimum_gas: Gas,
        minimum_deposit: NearToken,
    ) -> (DomainConfig, AccountId) {
        // 1. Look up the domain and check its purpose.
        let domains = match self.protocol_state.domain_registry() {
            Ok(domains) => domains,
            Err(err) => env::panic_str(&err.to_string()),
        };
        let Some(domain_config) = domains.get_domain_by_domain_id(domain_id) else {
            env::panic_str(
                &InvalidParameters::DomainNotFound {
                    provided: domain_id,
                }
                .to_string(),
            );
        };
        if domain_config.purpose != expected_purpose {
            env::panic_str(
                &InvalidParameters::WrongDomainPurpose {
                    domain_id: domain_config.id,
                    expected: expected_purpose,
                    actual: domain_config.purpose,
                }
                .to_string(),
            );
        }
        let domain_config = domain_config.clone();

        // 2. Make sure the call will not run out of gas doing yield/resume logic.
        let prepaid_gas = env::prepaid_gas();
        if prepaid_gas < minimum_gas {
            env::panic_str(
                &InvalidParameters::InsufficientGas {
                    provided: prepaid_gas.as_gas(),
                    required: minimum_gas.as_gas(),
                }
                .to_string(),
            );
        }

        // 3. Require the minimum deposit and refund any excess.
        let predecessor = env::predecessor_account_id();
        require_deposit(minimum_deposit, &predecessor);

        // 4. Refuse the request if the contract is not currently accepting requests
        //    (e.g. because TEE validation has failed).
        if !self.accept_requests {
            env::panic_str(&TeeError::TeeValidationFailed.to_string())
        }

        (domain_config, predecessor)
    }

## Tweak Derivation (Sign vs ForeignTx)

`verify_foreign_transaction()` uses a **different tweak derivation prefix** than `sign()` so the same
`(predecessor_id, derivation_path)` can never yield the same derived key across the two purposes.

Design:

* Keep the existing sign tweak derivation prefix unchanged.
* Introduce a foreign-tx-specific prefix and derive the tweak from the same `(predecessor_id, derivation_path)`
  input using the same hash construction.
* The contract derives the tweak internally from `request.derivation_path` (callers do not submit raw tweaks).

Example:

```rust
const SIGN_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 epsilon derivation:";
const FOREIGN_TX_TWEAK_DERIVATION_PREFIX: &str =
    "near-mpc-recovery v0.1.0 foreign-tx epsilon derivation:";

pub fn derive_sign_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(SIGN_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}

pub fn derive_foreign_tx_tweak(predecessor_id: &AccountId, path: &str) -> Tweak {
    let hash: [u8; 32] = derive_from_path(FOREIGN_TX_TWEAK_DERIVATION_PREFIX, predecessor_id, path);
    Tweak::new(hash)
}
```

This ensures key material used for validated foreign transactions is **always** distinct from
general-purpose `sign()` keys, even if the same account and derivation path are reused.
