Audit Report

## Title
Stale Cross-Chain Oracle Rate Allows Over-Minting of wrsETH, Causing Protocol Insolvency — (`contracts/pools/RSETHPoolV2.sol`)

## Summary

`CrossChainRateReceiver.getRate()` unconditionally returns the stored `rate` without checking `lastUpdated`, meaning any oracle lag during which the L1 rsETH/ETH rate rises causes `RSETHPoolV2.deposit()` to mint more wrsETH than the deposited ETH will back on L1. The resulting wrsETH supply shortfall is permanent and constitutes protocol insolvency.

## Finding Description

`CrossChainRateReceiver.getRate()` returns `rate` with no staleness validation: [1](#0-0) 

`lastUpdated` is recorded on every `lzReceive` call: [2](#0-1) 

but is never read by `getRate()` or any caller in the mint path.

`RSETHPoolV2.viewSwapRsETHAmountAndFee()` computes the wrsETH amount directly from this unchecked rate: [3](#0-2) 

`deposit()` calls `viewSwapRsETHAmountAndFee` and mints without any additional freshness guard: [4](#0-3) 

The formula `rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate` means a stale (lower) rate produces a higher `rsETHAmount`. The ETH is later bridged to L1 via `bridgeAssets()`: [5](#0-4) 

On L1, the same ETH mints rsETH at the current (higher) rate, producing fewer rsETH tokens than wrsETH already outstanding. The `limitDailyMint` modifier does not prevent this because it computes the cap using the same stale rate, so the limit itself is inflated: [6](#0-5) 

## Impact Explanation

**Critical — Protocol insolvency.** Every deposit made while the oracle is stale mints more wrsETH than the corresponding rsETH that will be created on L1 from the bridged ETH. The shortfall is permanent: once ETH is bridged and L1 mints at the current rate, the backing deficit cannot be recovered. wrsETH holders who redeem last receive less rsETH than their wrsETH represents, or redemptions fail entirely once backing is exhausted.

## Likelihood Explanation

No privileged action, oracle compromise, or attacker coordination is required. The condition is triggered by: (1) the L1 rsETH/ETH rate increasing — a routine, expected event as staking rewards accrue; (2) the LayerZero cross-chain message being delayed — normal operational latency with no on-chain mechanism to force an update; (3) any user calling `deposit()` during that window. The `lastUpdated` field is publicly readable on-chain, making the staleness window precisely detectable. The vulnerability is repeatable across every oracle update cycle.

## Recommendation

Add a maximum staleness threshold in `CrossChainRateReceiver.getRate()` (or in `RSETHPoolV2.getRate()`):

```solidity
uint256 public constant MAX_STALENESS = 24 hours;

function getRate() external view returns (uint256) {
    require(block.timestamp - lastUpdated <= MAX_STALENESS, "Rate stale");
    return rate;
}
```

`MAX_STALENESS` should be set conservatively relative to the expected LayerZero update frequency. This causes `deposit()` to revert when the oracle is stale rather than silently minting unbacked wrsETH.

## Proof of Concept

```solidity
// Fork test (L2 fork, e.g. Arbitrum)
// 1. Deploy / point to live RSETHPoolV2 with CrossChainRateReceiver as rsETHOracle
// 2. Record current oracle rate (e.g. 1.05e18); lastUpdated = block.timestamp
// 3. vm.warp(block.timestamp + 2 days) — no lzReceive triggered
// 4. Simulate L1 rate rising to 1.10e18 (query L1 LRTOracle.rsETHPrice())
// 5. Depositor calls deposit{value: 10 ether}(""):
//      rsETHMinted = 10e18 * 1e18 / 1.05e18 ≈ 9.523e18 wrsETH
// 6. L1 will mint for same 10 ETH at current rate:
//      rsETHOnL1 = 10e18 * 1e18 / 1.10e18 ≈ 9.090e18 rsETH
// 7. Shortfall = 9.523e18 - 9.090e18 = 0.433e18 unbacked wrsETH per 10 ETH deposit
// 8. Assert wrsETH.totalSupply() > total rsETH backing → insolvency confirmed
```
