Audit Report

## Title
Unbounded Per-User Withdrawal Request Queue Enables Temporary Freezing of Subsequent Users' Withdrawals — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`initiateWithdrawal` imposes no cap on the number of pending requests a single user may create. Because `_unlockWithdrawalRequests` advances `nextLockedNonce[asset]` strictly in FIFO order with no skip mechanism, an attacker who front-loads the queue with many dust-sized requests forces every subsequent user's withdrawal to wait until all preceding entries are processed. The operator has no way to reorder or skip entries.

## Finding Description
`initiateWithdrawal` (L150–178) accepts any `rsETHUnstaked >= 1` when `minRsEthAmountToWithdraw[asset] == 0` (the default, since `initialize` never sets it) and has no per-user or global request-count guard. Each call transfers rsETH from the caller, records `assetsCommitted[asset] += expectedAssetAmount`, and calls `_addUserWithdrawalRequest`, which appends to the global nonce counter:

```solidity
// L756–757
userAssociatedNonces[asset][msg.sender].pushBack(nextUnusedNonce_);
nextUnusedNonce[asset] = nextUnusedNonce_ + 1;
```

The operator's `unlockQueue` → `_unlockWithdrawalRequests` (L786–815) iterates from `nextLockedNonce[asset]` to `firstExcludedIndex` in strict sequential order:

```solidity
while (nextLockedNonce_ < firstExcludedIndex) {
    ...
    unchecked { nextLockedNonce_++; }
}
nextLockedNonce[asset] = nextLockedNonce_;
```

There is no `continue`, skip, or cancel path. The operator must advance `nextLockedNonce[asset]` through every entry in sequence. An attacker who submits N dust requests before a legitimate user's request forces the operator to process all N entries before the legitimate request becomes unlockable.

The `getAvailableAssetAmount` check (L170, L599–603) prevents committing more than total deposited assets, but does not prevent many small requests that each commit a tiny amount. The attacker recovers all committed rsETH (converted to the underlying asset) when their requests are eventually processed, so the net cost is gas only.

## Impact Explanation
**Medium — Temporary freezing of funds.** Legitimate users who submit withdrawal requests after the attacker's dust requests cannot have their requests unlocked until the operator iterates through every preceding dust entry. With a sufficient number of dust requests (achievable with a moderate rsETH balance and gas budget), the operator must call `unlockQueue` many times before reaching the legitimate user's request, delaying their withdrawal for an attacker-controlled duration. This matches the allowed impact "Medium. Temporary freezing of funds."

## Likelihood Explanation
**Medium.** The attack is permissionless and requires only rsETH holdings (a real but recoverable cost) and gas. `minRsEthAmountToWithdraw` defaults to 0, making 1 wei the effective minimum. Any rsETH holder can execute this without special roles or external dependencies. The attacker recovers their principal when requests are processed, making the net cost gas-only. The attack is repeatable.

## Recommendation
1. **Enforce a non-zero `minRsEthAmountToWithdraw`** at initialization and prevent it from being set to zero, raising the cost per request.
2. **Cap pending requests per user per asset**, e.g.:
   ```solidity
   require(userAssociatedNonces[asset][msg.sender].length() < MAX_REQUESTS_PER_USER, "Too many pending requests");
   ```
3. **Add an operator skip/cancel mechanism** so the operator can advance `nextLockedNonce[asset]` past stale or dust entries without processing them, preventing permanent queue blockage.

## Proof of Concept
1. Deploy `LRTWithdrawalManager`; `minRsEthAmountToWithdraw[ETH]` is 0 (default from `initialize`, L90–98).
2. Attacker acquires rsETH (e.g., 10,000 wei).
3. Attacker calls `initiateWithdrawal(ETH, 1, "")` 10,000 times across multiple transactions; each call commits ~1 wei of `assetsCommitted[ETH]` and advances `nextUnusedNonce[ETH]` by 1 (L756–757).
4. Legitimate user calls `initiateWithdrawal(ETH, 1 ether, "")` — succeeds, placed at nonce 10,000.
5. Operator calls `unlockQueue(ETH, 1000, ...)` — only nonces 0–999 are processed (all attacker dust).
6. Operator must call `unlockQueue` 10 more times before reaching the legitimate user's request at nonce 10,000, delaying their withdrawal by an attacker-controlled number of operator cycles.
7. Foundry fuzz test: fuzz `N` (number of attacker requests) and assert that `nextLockedNonce[ETH]` after a single `unlockQueue(ETH, N/10, ...)` call is less than the legitimate user's nonce `N`, confirming the delay.
