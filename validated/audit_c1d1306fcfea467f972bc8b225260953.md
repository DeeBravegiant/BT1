Audit Report

## Title
First-Depositor rsETH Price Inflation via Unrestricted ETH Donation to LRTDepositPool - (File: contracts/LRTDepositPool.sol)

## Summary
An attacker who is the first depositor can donate ETH directly to `LRTDepositPool` via its unrestricted `receive()` function, inflating `address(this).balance` without minting rsETH. When `updateRSETHPrice()` is called, the stored `rsETHPrice` is set to an extreme value. Subsequent depositors who pass `minRSETHAmountExpected = 0` receive zero rsETH while their ETH is permanently absorbed into the pool, constituting direct theft of user funds.

## Finding Description

**Root cause chain:**

1. `LRTDepositPool` has an unrestricted `receive()` at line 58 that accepts arbitrary ETH with no accounting: [1](#0-0) 

2. `getETHDistributionData()` reads `address(this).balance` directly, so any donated ETH is immediately counted as protocol TVL: [2](#0-1) 

3. `_updateRsETHPrice()` computes the new price as `(totalETHInProtocol - fee).divWad(rsethSupply)`. With 1 wei rsETH supply and a large donated balance, this produces an astronomically large `rsETHPrice`: [3](#0-2) 

4. The price-increase guard is entirely bypassed when `pricePercentageLimit == 0` (the default, since the variable is never set in `initialize()`): [4](#0-3) [5](#0-4) 

5. `getRsETHAmountToMint()` divides by the inflated `rsETHPrice`, producing 0 for any victim deposit smaller than the donation: [6](#0-5) 

6. `_beforeDeposit()` only reverts if `rsethAmountToMint < minRSETHAmountExpected`. When the victim passes `minRSETHAmountExpected = 0`, the condition `0 < 0` is false and the zero-rsETH deposit is silently accepted: [7](#0-6) 

**Exploit path:**
1. Call `updateRSETHPrice()` with zero supply → sets `rsETHPrice = 1 ether`.
2. Attacker calls `depositETH{value: 1 wei}(0, "")` → receives 1 wei rsETH (100% of supply).
3. Attacker sends `D` ETH directly to `LRTDepositPool` via `receive()`. No rsETH minted.
4. Attacker calls `updateRSETHPrice()`. New price ≈ `D * 1e18 / 1 = D * 1e18` (using `divWad`). Guard is skipped because `pricePercentageLimit == 0`.
5. Victim calls `depositETH{value: V}(0, "")` where `V < D`. Computed `rsethAmountToMint = V * 1e18 / (D * 1e18) = V/D < 1` → truncates to 0. Victim's ETH is accepted; 0 rsETH minted.
6. Attacker redeems their 1 wei rsETH (100% of supply), recovering `D + 1 wei + V` ETH. Net profit: `V` ETH.

## Impact Explanation

**Critical — Direct theft of user funds.** The victim's deposited ETH (`V`) is permanently transferred to the attacker upon redemption. The victim holds no rsETH and has no claim on the pool. This matches the "Direct theft of any user funds" critical impact class exactly.

## Likelihood Explanation

**Low.** Required conditions: (a) `pricePercentageLimit` is unconfigured (default state at deployment); (b) the victim passes `minRSETHAmountExpected = 0`, which is a realistic scenario for direct contract callers or integrations that omit slippage protection; (c) the attacker must be the first depositor and lock `D` ETH for the EigenLayer withdrawal delay. The attack is most dangerous at protocol launch before `pricePercentageLimit` is configured. The economic cost of locking capital limits opportunistic use, but targeted theft of a large deposit remains feasible.

## Recommendation

1. **Remove or restrict `receive()`**: Replace raw `address(this).balance` in `getETHDistributionData()` with an internal accounting variable incremented only on legitimate deposits (`depositETH`, `receiveFromNodeDelegator`, etc.).
2. **Revert on zero rsETH mint**: In `_beforeDeposit()`, add `if (rsethAmountToMint == 0) revert ZeroRsETHMinted();` to prevent silent zero-mint deposits regardless of `minRSETHAmountExpected`.
3. **Enforce `pricePercentageLimit` before first deposit**: Add a deployment checklist requirement or an `initialize`-time setter so the guard is active from day one.
4. **Minimum liquidity lock**: Mint a small amount of rsETH to a dead address at initialization (analogous to Uniswap V2's minimum liquidity) to prevent the total supply from ever being 1 wei.

## Proof of Concept

```solidity
// Preconditions: pricePercentageLimit == 0 (default), minAmountToDeposit == 0

// Setup: initialize rsETHPrice
lrtOracle.updateRSETHPrice();
// rsETHPrice = 1e18 (supply is 0)

// Step 1: Attacker becomes sole rsETH holder
lrtDepositPool.depositETH{value: 1}(0, "attacker");
// rsETH supply = 1 wei, rsETHPrice = 1e18

// Step 2: Attacker donates 1000 ETH directly — no rsETH minted
(bool ok,) = address(lrtDepositPool).call{value: 1000 ether}("");
require(ok);

// Step 3: Inflate stored price (guard bypassed: pricePercentageLimit == 0)
lrtOracle.updateRSETHPrice();
// totalETHInProtocol ≈ 1000e18, rsethSupply = 1
// newRsETHPrice = 1000e18 * 1e18 / 1 = 1000e36

// Step 4: Victim deposits 1 ETH with no slippage protection
// rsethAmountToMint = 1e18 * 1e18 / 1000e36 = 0 → victim gets 0 rsETH
lrtDepositPool.depositETH{value: 1 ether}(0, "victim");
// Victim's 1 ETH accepted, 0 rsETH minted — funds permanently lost

// Step 5: Attacker redeems 1 wei rsETH (100% of supply)
// Recovers: 1000 ETH (donation) + 1 wei (initial) + 1 ETH (victim) ≈ 1001 ETH
// Net profit: 1 ETH (victim's deposit)
```

A Foundry fork test can confirm this by asserting `rsETH.balanceOf(victim) == 0` after step 4 and `address(attacker).balance` increases by `~1 ETH` after step 5.
