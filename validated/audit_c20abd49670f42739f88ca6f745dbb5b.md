Audit Report

## Title
`depositAsset()` / `depositETH()` Mint rsETH Using Stale Cached `rsETHPrice` Without Oracle Sync — (File: contracts/LRTDepositPool.sol)

## Summary

`LRTOracle` stores `rsETHPrice` as a state variable updated only by explicit calls to `updateRSETHPrice()` or `updateRSETHPriceAsManager()`. Neither `LRTDepositPool.depositAsset()` nor `depositETH()` triggers a price update before computing the rsETH mint amount. When protocol TVL has grown (e.g., stETH rebase) but the stored price has not yet been refreshed, depositors receive more rsETH than their deposit is worth at the true current exchange rate, diluting existing holders and transferring their accrued yield to the depositor.

## Finding Description

`LRTOracle.rsETHPrice` is a plain storage variable:

```solidity
uint256 public override rsETHPrice;  // LRTOracle.sol L28
```

It is only updated inside `_updateRsETHPrice()`, which is called by `updateRSETHPrice()` (public, permissionless when unpaused) or `updateRSETHPriceAsManager()` (manager-only). The true current price is computed on-demand by summing all protocol TVL across EigenLayer strategies and dividing by rsETH supply.

The deposit flow never calls `updateRSETHPrice()`:

1. `depositAsset()` / `depositETH()` → `_beforeDeposit()` → `getRsETHAmountToMint()`
2. `getRsETHAmountToMint()` reads `lrtOracle.rsETHPrice()` — the cached value — directly:

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
// LRTDepositPool.sol L520
```

When stETH rebases or EigenLayer rewards accrue between two `updateRSETHPrice()` calls, the true rsETH/ETH rate is higher than the stored `rsETHPrice`. Because the denominator in the mint formula is stale-low, the depositor receives more rsETH than their deposit is worth at the true current rate.

When `updateRSETHPrice()` is eventually called, `_updateRsETHPrice()` computes `previousTVL = rsethSupply * rsETHPrice`. Because rsethSupply is now inflated (extra rsETH was minted at the stale price), the reward delta is understated and the new price settles lower than it would have been had the correct amount been minted. Existing holders' rsETH is permanently worth less ETH — their accrued yield has been partially transferred to the depositor.

Additionally, `_updateRsETHPrice()` contains a downside-protection auto-pause (if price drops beyond `pricePercentageLimit`) that is never triggered during a deposit, allowing continued minting at a wrong price even when a slashing event should have halted the protocol.

Existing guards do not prevent this:
- `minRSETHAmountExpected` is a depositor-side slippage floor; it protects the depositor from receiving *less* than expected, not from receiving *more*.
- `whenNotPaused` only checks whether the contract is already paused; it does not force a price sync.
- `nonReentrant` is irrelevant to the stale-price issue.

## Impact Explanation

**High — Theft of unclaimed yield.**

Accrued rewards (stETH rebase, EigenLayer yield) belong to existing rsETH holders. A depositor who calls `depositAsset()` or `depositETH()` in the window between reward accrual and the next `updateRSETHPrice()` call receives rsETH computed against the stale lower price, obtaining more rsETH than their deposit is worth at the true current rate. After the price is updated, the inflated supply causes the new price to settle below what it would have been, permanently reducing the ETH value of every existing holder's rsETH. The stolen amount scales with both the size of the deposit and the magnitude of the accrued-but-unrecorded rewards.

## Likelihood Explanation

**Medium.** `updateRSETHPrice()` is not called atomically within the deposit flow; the protocol relies on off-chain keepers. stETH rebases every ~24 hours; EigenLayer rewards accumulate continuously. Any window between a reward accrual event and the next keeper call is exploitable. A sophisticated depositor can monitor on-chain TVL growth (via `LRTDepositPool.getTotalAssetDeposits()` and `LRTOracle.getAssetPrice()`) versus the stored `rsETHPrice` and time their deposit to capture the maximum spread. No special privileges are required — only a standard `depositAsset()` or `depositETH()` call.

## Recommendation

At the start of `depositAsset()` and `depositETH()`, before `_beforeDeposit()` is called, invoke `ILRTOracle(lrtConfig.getContract(LRTConstants.LRT_ORACLE)).updateRSETHPrice()`. This ensures the price used for minting reflects the current true protocol state, including accrued rewards, fee minting, and any required safety pauses. Because `updateRSETHPrice()` reverts for non-managers when the price increase exceeds `pricePercentageLimit`, the deposit functions should either call `updateRSETHPrice()` in a try/catch (proceeding only if the price is already fresh) or require that the oracle is not stale beyond a configurable staleness threshold before allowing deposits.

## Proof of Concept

**State at T₀** (after last `updateRSETHPrice()` call):
- Protocol TVL = 1050 ETH, rsETH supply = 1000, `rsETHPrice = 1.05e18`

**Between T₀ and T₁** — stETH rebases:
- Protocol TVL grows to 1060 ETH. True rsETH price = `1060/1000 = 1.06e18`. Stored `rsETHPrice` is still `1.05e18`.

**Attacker calls `depositAsset(stETH, 100e18, 0, "")`:**

`getRsETHAmountToMint` computes (LRTDepositPool.sol L520, assuming stETH price = 1e18):
```
rsethAmountToMint = (100e18 × 1e18) / 1.05e18 ≈ 95.238 rsETH
```
At the true price of `1.06e18`, the correct amount would be:
```
100e18 / 1.06e18 ≈ 94.340 rsETH
```
Attacker receives **~0.898 rsETH extra**.

**Keeper calls `updateRSETHPrice()`:**
- New TVL = 1160 ETH, new supply = 1095.238 rsETH
- `previousTVL = 1095.238 × 1.05 ≈ 1149.999 ETH`
- `newRsETHPrice = 1160 / 1095.238 ≈ 1.0591e18`

If the correct amount had been minted (94.340 rsETH):
- New supply = 1094.340, `newRsETHPrice = 1160 / 1094.340 ≈ 1.0600e18`

Existing 1000 rsETH holders:
- With stale-price deposit: `1000 × 1.0591 = 1059.1 ETH`
- With correct-price deposit: `1000 × 1.0600 = 1060.0 ETH`

**~0.9 ETH of accrued yield is permanently transferred from existing holders to the attacker.**

Foundry fork test plan: fork mainnet, warp past a stETH rebase event without calling `updateRSETHPrice()`, call `depositAsset(stETH, largeAmount, 0, "")`, then call `updateRSETHPrice()`, and assert that `rsETHPrice` after the update is lower than `(TVL_before_deposit + deposit_value) / supply_before_deposit`, confirming dilution of existing holders.     function _updateRsETHPrice() internal {
```text
        address rsETHTokenAddress = lrtConfig.rsETH();
        uint256 rsethSupply = IRSETH(rsETHTokenAddress).totalSupply();

        if (rsethSupply == 0) {
            rsETHPrice = 1 ether;
            highestRsethPrice = 1 ether;
            return;
        }

        if (highestRsethPrice == 0) {
            highestRsethPrice = rsETHPrice;
        }

        uint256 previousPrice = rsETHPrice;

        // get total ETH in the protocol (normalized to 1e18)
        uint256 totalETHInProtocol = _getTotalEthInProtocol();

        // calculate previousTVL using rsethSupply multiplied by rsETHPrice
        uint256 previousTVL = rsethSupply.mulWad(rsETHPrice);

        IPausable lrtDepositPool = IPausable(lrtConfig.getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IPausable withdrawalManager = IPausable(lrtConfig.getContract(LRTConstants.LRT_WITHDRAW_MANAGER));

        // determine if the protocol is active (not paused)
        bool protocolPaused = lrtDepositPool.paused() || withdrawalManager.paused() || paused;

        // only take fee if TVL increased and protocol is not paused
        uint256 protocolFeeInETH = 0;
        if (!protocolPaused && totalETHInProtocol > previousTVL) {
            uint256 rewardAmount = totalETHInProtocol - previousTVL;
            protocolFeeInETH = (rewardAmount * lrtConfig.protocolFeeInBPS()) / 10_000;
        }

        // compute new rsETH price based on total ETH minus fee
        uint256 newRsETHPrice = (totalETHInProtocol - protocolFeeInETH).divWad(rsethSupply);
```
