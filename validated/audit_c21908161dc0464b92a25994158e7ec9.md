Audit Report

## Title
Missing L2 Sequencer Uptime Check in Chainlink Oracle Enables Stale-Price Deposit Exploitation - (File: contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol)

## Summary
`ChainlinkOracleForRSETHPoolCollateral.getRate()` calls Chainlink's `latestRoundData()` with no check against the L2 Sequencer Uptime Feed. After an L2 sequencer outage, the feed continues returning the last pre-downtime price. An unprivileged attacker can immediately deposit collateral at the stale inflated price and receive more `wrsETH` than the deposited collateral is worth, extracting value from the protocol's backing pool.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` performs only three checks after calling `latestRoundData()`:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
``` [1](#0-0) 

None of these detect L2 sequencer downtime. When the sequencer restarts, the last round was answered before downtime (`answeredInRound == roundID`), `timestamp != 0`, and `ethPrice > 0` — all three guards pass while the price is arbitrarily stale.

This oracle is consumed by `RSETHPoolV3.deposit(address token, ...)` via `viewSwapRsETHAmountAndFee`:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

And identically in `RSETHPoolV3WithNativeChainBridge.viewSwapRsETHAmountAndFee`: [3](#0-2) 

If `tokenToETHRate` is stale-high (collateral crashed during downtime but oracle still shows pre-crash price), the attacker receives more `wrsETH` than the deposited collateral is worth. The `deposit()` function in both pool contracts is callable by any unprivileged user: [4](#0-3) [5](#0-4) 

`RSETHPoolV3WithNativeChainBridge` is explicitly an L2 contract, evidenced by its use of `IL2TokenBridge` and `l1VaultETHForL2Chain`: [6](#0-5) 

## Impact Explanation
**Critical — Direct theft of user/protocol funds.** The attacker deposits collateral tokens at a stale inflated price and receives `wrsETH` minted at the old (higher) collateral valuation. Since `wrsETH` is redeemable at the true rsETH/ETH rate, the attacker extracts more ETH value than deposited. The loss is borne by all existing rsETH holders through dilution of the backing pool. The daily mint limit caps per-window loss but does not prevent the attack from being profitable.

## Likelihood Explanation
L2 sequencer outages are documented recurring events on Arbitrum, Optimism, and other OP-stack chains. The attack requires no privileged access — only a standard EOA and knowledge of the sequencer restart. The attack window opens the moment the sequencer resumes and closes when Chainlink's L2 feed updates (typically minutes to hours). It is straightforward to automate by monitoring the Chainlink L2 Sequencer Uptime Feed and submitting a deposit in the first available block after restart.

## Recommendation
Add a Chainlink L2 Sequencer Uptime Feed check inside `ChainlinkOracleForRSETHPoolCollateral.getRate()`, reverting if the sequencer is down or has been back online for less than a grace period (e.g., 1 hour):

```solidity
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 1 hours;

function getRate() public view returns (uint256) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer != 0) revert SequencerDown();
    if (block.timestamp - startedAt < GRACE_PERIOD) revert GracePeriodNotOver();

    (uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
        AggregatorV3Interface(oracle).latestRoundData();
    if (answeredInRound < roundID) revert StalePrice();
    if (timestamp == 0) revert IncompleteRound();
    if (ethPrice <= 0) revert InvalidPrice();
    if (block.timestamp - timestamp > MAX_STALENESS) revert StalePrice();
    ...
}
```

Also add a maximum staleness check (`block.timestamp - timestamp > MAX_STALENESS`) to catch feeds stale for reasons other than sequencer downtime.

## Proof of Concept
1. Protocol deploys `RSETHPoolV3WithNativeChainBridge` on Arbitrum with `wstETH` as a supported token, oracle set to `ChainlinkOracleForRSETHPoolCollateral` wrapping the `wstETH/ETH` Chainlink feed.
2. `wstETH/ETH` Chainlink feed on Arbitrum last updated at `P_old = 1.15 ETH` before sequencer downtime.
3. During sequencer downtime, `wstETH` market price drops to `P_new = 1.05 ETH`.
4. Sequencer restarts. Chainlink L2 feed still returns `P_old = 1.15 ETH`. `answeredInRound == roundID`, `timestamp != 0`, `ethPrice > 0` — all checks in `getRate()` pass.
5. Attacker calls `deposit(wstETH, 1000e18, "")`:
   - `getRate()` returns `1.15e18` (stale)
   - `rsETHAmount = 1000e18 * 1.15e18 / rsETHPrice` — attacker receives rsETH backed by 1150 ETH
   - Attacker deposited 1000 wstETH worth only 1050 ETH at true market price
6. Attacker redeems/bridges `wrsETH` for ~1150 ETH equivalent, netting ~100 ETH profit at the expense of the protocol's backing pool.

**Foundry fork test plan:** Fork Arbitrum mainnet at a block just after a historical sequencer restart. Mock the `wstETH/ETH` Chainlink feed to return a pre-downtime price higher than the current market price. Call `deposit(wstETH, amount, "")` as an unprivileged EOA. Assert that `wrsETH` minted exceeds the true ETH value of the deposited `wstETH` at the current market price, confirming the over-minting.
