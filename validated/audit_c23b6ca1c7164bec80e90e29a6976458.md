### Title
Unvalidated `builder` Field in `ProposalInit` Allows Proposer to Redirect All Block Transaction Fees to Arbitrary Address — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`, `crates/apollo_consensus_orchestrator/src/utils.rs`)

---

### Summary

The `builder` field of `ProposalInit` is accepted verbatim from the network without any validation in `is_proposal_init_valid`. This field is mapped directly to `sequencer_address` in `BlockInfo`, which is the sole fee recipient for every transaction in the block. Any consensus participant whose turn it is to propose can set `builder` to an arbitrary address — including their own — causing all transaction fees for that block to be transferred to the attacker's address instead of the legitimate sequencer. Validators re-execute the block with the same malicious `builder`, produce the same `ProposalCommitment`, and finalize the block, permanently committing the wrong fee-recipient state.

---

### Finding Description

**Step 1 — `builder` becomes `sequencer_address`.**

`convert_to_sn_api_block_info` in `utils.rs` maps `init.builder` directly to `sequencer_address` in `starknet_api::block::BlockInfo` with no transformation or guard:

```rust
Ok(starknet_api::block::BlockInfo {
    ...
    sequencer_address: init.builder,   // ← proposer-supplied, unvalidated
    ...
})
``` [1](#0-0) 

**Step 2 — `sequencer_address` is the fee recipient.**

`execute_fee_transfer` in `account_transaction.rs` hard-codes the fee transfer recipient as `block_context.block_info.sequencer_address`:

```rust
calldata: calldata![
    *block_context.block_info.sequencer_address.0.key(), // Recipient.
    lsb_amount,
    msb_amount
],
``` [2](#0-1) 

The same path is taken in the Cairo OS `charge_fee` function:

```cairo
local calldata: TransferCallData = TransferCallData(
    recipient=block_context.block_info_for_execute.sequencer_address,
    ...
);
``` [3](#0-2) 

**Step 3 — `is_proposal_init_valid` never checks `builder`.**

The full set of fields validated in `is_proposal_init_valid` is: `timestamp`, `starknet_version`, `version_constant_commitment`, `height`, `l1_da_mode`, `l2_gas_price_fri`, L1 gas prices (within margin), and `fee_proposal_fri` (presence/bounds). The `builder` field is absent from every check:

```rust
if !(init_proposed.height == proposal_init_validation.height
    && init_proposed.l1_da_mode == proposal_init_validation.l1_da_mode
    && init_proposed.l2_gas_price_fri == proposal_init_validation.l2_gas_price_fri)
{ ... }
``` [4](#0-3) 

`ProposalInitValidation` — the struct that carries the validator's reference values — has no `builder` field at all:

```rust
pub(crate) struct ProposalInitValidation {
    pub height: BlockNumber,
    pub block_timestamp_window_seconds: u64,
    pub previous_proposal_init: Option<PreviousProposalInitInfo>,
    pub l1_da_mode: L1DataAvailabilityMode,
    pub l2_gas_price_fri: GasPrice,
    pub starknet_version: StarknetVersion,
    pub fee_actual: Option<GasPrice>,
}
``` [5](#0-4) 

**Step 4 — Validators re-execute with the malicious `builder`, producing the same commitment.**

`initiate_validation` calls `convert_to_sn_api_block_info(init)` with the proposer-supplied `init` (including the malicious `builder`) and passes the resulting `BlockInfo` to the batcher:

```rust
block_info: convert_to_sn_api_block_info(init)?,
``` [6](#0-5) 

Because every validator executes transactions with the same malicious `sequencer_address`, they all produce the same `partial_block_hash` and the same `ProposalCommitment`. The `ProposalFinMismatch` guard at line 244 passes, and the block is finalized.

**Step 5 — The proposer sets `builder` from local config; no consensus-layer check exists.**

During `build_proposal`, `builder` is set from `self.config.static_config.builder_address`:

```rust
builder_address: self.config.static_config.builder_address,
``` [7](#0-6) 

This is a local configuration value. A malicious proposer simply sets this config to any address they control.

---

### Impact Explanation

Every transaction fee in the block is an ERC-20 `transfer` from the user's account to `sequencer_address`. With a malicious `builder`, all fees are transferred to the attacker's address. The resulting state diff (fee-token storage writes) reflects the wrong recipient, is committed to the Patricia tree, and is finalized on-chain. The legitimate sequencer receives zero fees for the block. This is a direct, permanent economic loss matching the "Incorrect fee … balance … with economic impact" critical impact category.

---

### Likelihood Explanation

In Tendermint-style consensus, proposer rotation means every validator gets a turn as proposer. Any validator node operator can exploit this by modifying `builder_address` in their local config before their proposer turn. No special privilege beyond being a consensus participant is required. The attack is silent — it produces a valid `ProposalCommitment` that all honest validators accept.

---

### Recommendation

Add `builder` to `ProposalInitValidation` and enforce it in `is_proposal_init_valid`. The validator's expected `builder` address should be derived from the same source the proposer uses (e.g., the committee/validator-set record for the current height), and the check should be a strict equality:

```rust
if init_proposed.builder != proposal_init_validation.expected_builder {
    return Err(ValidateProposalError::InvalidProposalInit(...));
}
```

Until a committee-derived `builder` is available, the validator can at minimum reject any `builder` that differs from the locally configured `builder_address`, mirroring the existing `version_constant_commitment` sentinel pattern.

---

### Proof of Concept

1. Attacker controls validator node `V` with address `ATTACKER_ADDR`.
2. Attacker sets `builder_address = ATTACKER_ADDR` in `V`'s static config.
3. When `V` is selected as proposer for height `H`, `initiate_build` constructs `ProposalInit { builder: ATTACKER_ADDR, ... }` and streams it to peers.
4. Each honest validator calls `is_proposal_init_valid` — `builder` is not checked, validation passes.
5. Each validator calls `convert_to_sn_api_block_info(init)` → `BlockInfo { sequencer_address: ATTACKER_ADDR, ... }`.
6. The batcher executes all transactions; `execute_fee_transfer` transfers each fee to `ATTACKER_ADDR`.
7. All validators compute the same `partial_block_hash` (same state diff, same fee recipient), the same `ProposalCommitment`, and finalize the block.
8. The committed state has all fee-token balances credited to `ATTACKER_ADDR`; the legitimate sequencer receives nothing.
