Audit Report

## Title
Zero-Initialized `rate` in `CrossChainRateReceiver` Causes Division-by-Zero DoS on All Pool Deposits Before First LayerZero Message - (`contracts/cross-chain/CrossChainRateReceiver.sol`)

## Summary
`CrossChainRateReceiver` declares `uint256 public rate` with no initial value, leaving it at `0` after deployment. Every pool variant's `deposit()` path calls `viewSwapRsETHAmountAndFee()`, which fetches this zero rate via `getRate()` and immediately divides by it, triggering a Solidity 0.8.x checked-arithmetic `Panic(0x12)` revert. All deposits are blocked until the first LayerZero rate message is delivered.

## Finding Description
`CrossChainRateReceiver` declares `uint256 public rate` at line 13 with no initializer, so it is `0` at construction. `RSETHRateReceiver`'s constructor (lines 10–15) sets `rateInfo`, `srcChainId`, `rateProvider`, and `layerZeroEndpoint` but never assigns `rate`. `getRate()` (lines 103–105) returns `rate` unconditionally with no zero-guard.

In `RSETHPoolV3`, the `limitDailyMint` modifier (lines 96–125) is applied to `deposit()`. Before any `dailyMintLimit` check, it calls `viewSwapRsETHAmountAndFee(amount)` (line 105). That function (lines 299–308) executes:

```solidity
uint256 rsETHToETHrate = getRate();          // returns 0
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;  // Panic(0x12)
```

The division by zero fires inside the modifier before the function body is ever reached. The identical pattern exists in `RSETHPoolV2.viewSwapRsETHAmountAndFee` (lines 225–234) and `RSETHPool.viewSwapRsETHAmountAndFee` (lines 311–320), and all remaining pool variants (`RSETHPoolNoWrapper`, `RSETHPoolV2ExternalBridge`, `RSETHPoolV2NBA`, `RSETHPoolV3ExternalBridge`, `RSETHPoolV3WithNativeChainBridge`). No existing check prevents the panic: the only path to a non-zero `rate` is a successful `lzReceive` call from the LayerZero endpoint, which cannot happen until the cross-chain relayer delivers the first message.

## Impact Explanation
**Medium. Temporary freezing of funds.**

Every `deposit()` call reverts with a division-by-zero panic for the entire window between pool configuration and the first successful `lzReceive`. No user ETH is permanently lost (the revert returns the ETH), but the pool is completely non-functional for deposits during this window. The freeze ends as soon as the LZ oracle operator delivers the first rate message, making the impact temporary rather than permanent.

## Likelihood Explanation
The race window is an inherent consequence of the deployment sequence: the pool can be configured and made publicly accessible before the LZ relayer delivers its first message. No attacker action is required — any ordinary user who calls `deposit()` during this window triggers the revert naturally. If the LZ relayer is delayed or congested, the window extends. The condition is deterministic and reproducible on any fresh deployment.

## Recommendation
Add a non-zero guard in `CrossChainRateReceiver.getRate()`:

```solidity
function getRate() external view returns (uint256) {
    require(rate != 0, "Rate not initialized");
    return rate;
}
```

Alternatively, set a conservative initial rate in the `RSETHRateReceiver` constructor, or add a `require(rsETHToETHrate != 0)` check inside each pool's `viewSwapRsETHAmountAndFee` before the division. Note that `viewSwapAssetToPremintedRsETH` in `RSETHPoolV3` already performs this check (`if (rsETHToETHrate == 0) revert UnsupportedOracle()` at line 393), confirming the pattern is known and simply missing from the deposit path.

## Proof of Concept

```solidity
// Foundry test — no fork required
function test_zeroRateDoS() public {
    // Deploy receiver without any lzReceive call — rate stays 0
    RSETHRateReceiver receiver = new RSETHRateReceiver(
        1, address(0xBEEF), address(0xDEAD)
    );
    assertEq(receiver.getRate(), 0);

    // Deploy and initialize pool pointing at the zero-rate oracle
    RSETHPoolV3 pool = new RSETHPoolV3();
    pool.initialize(admin, bridger, address(wrsETH), 0, address(receiver), true);
    // reinitialize sets dailyMintLimit > 0 and startTimestamp in the past
    pool.reinitialize(1000 ether, block.timestamp - 1);

    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert();  // Panic(0x12): division by zero
    pool.deposit{value: 1 ether}("ref");
}
```

Deploy `RSETHRateReceiver`, configure any pool variant to use it as `rsETHOracle`, call `deposit()` — the call reverts with a division-by-zero panic every time until `lzReceive` delivers a non-zero rate.
