### Title
`deploy_token` MetadataPayload Signature Is Not Chain-ID-Bound — Cross-Chain Replay Deploys Unauthorized Bridged Tokens — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `aptos/sources/bridge_types.move`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The NEAR MPC signature consumed by every chain's `deploy_token` entry point covers only `(type_byte | token | name | symbol | decimals)`. No destination chain ID is bound into the signed hash. Because all deployed bridge contracts share the same `nearBridgeDerivedAddress` (derived from the single NEAR MPC key), a signature produced for chain A is cryptographically valid on chains B, C, and D. An unprivileged attacker who observes a `deploy_token` transaction on any one chain can replay the extracted signature on every other chain, deploying the bridged token there without NEAR's per-chain authorization.

---

### Finding Description

**The asymmetry between `fin_transfer` and `deploy_token` encoding:**

`fin_transfer` / `FinalizeTransferPayload` correctly binds the destination chain by interleaving `omniBridgeChainId` (as the OmniAddress discriminant) before both `tokenAddress` and `recipient` in the Borsh encoding: [1](#0-0) 

The same pattern is present in Starknet: [2](#0-1) 

And in Solana's `FinalizeTransferPayload::serialize_for_near`, which writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` before both the mint and the recipient: [3](#0-2) 

**`MetadataPayload` / `deploy_token` omits the chain ID entirely on every chain:**

EVM — the hash covers only type byte + token + name + symbol + decimals: [4](#0-3) 

Starknet `MetadataPayload::to_borsh` — identical layout, no chain ID: [5](#0-4) 

Aptos `metadata_to_borsh` — same: [6](#0-5) 

Solana `DeployTokenPayload::serialize_for_near` — same: [7](#0-6) 

Because the Borsh byte strings are identical across all four chains for the same `(token, name, symbol, decimals)` tuple, and because `ECDSA.recover(hash, sig) == nearBridgeDerivedAddress` is the sole authorization gate on every chain, a signature that passes on one chain passes on all others.

---

### Impact Explanation

**Unauthorized cross-chain token deployment.** An attacker replays a valid `deploy_token` signature from chain A onto chains B, C, D. Each chain's bridge contract accepts the signature, deploys the bridged token, and writes the NEAR-token-ID → local-address mapping. The NEAR bridge has not authorized these deployments for those chains.

**Permanent blocking of legitimate deployment.** Each chain guards against re-deployment with a one-time check (`"ERR_TOKEN_EXIST"` / `isBridgeToken` / `near_to_aptos_token.contains`): [8](#0-7) [9](#0-8) [10](#0-9) 

Once the attacker's replay has set the mapping, the bridge's own relayer can never call `deploy_token` for that token on that chain again. The NEAR bridge has no on-chain record of the token address on the attacked chain, so it cannot route `fin_transfer` payloads to it. Bridging that token to the attacked chain is permanently broken unless an admin manually intervenes — a frozen redemption path for all users of that token on that chain.

This falls squarely within:
- **High — Acceptance of insufficiently-bound signatures that bypass execution gates** (the chain-ID gate is entirely absent from the metadata path).
- **Critical — Irreversible fund lock / frozen redemption path** (once the mapping is poisoned, legitimate `deploy_token` is permanently blocked and the NEAR side cannot route transfers).

---

### Likelihood Explanation

**High.** The attack requires no privileged access. The attacker only needs to:
1. Watch any public chain for a `deploy_token` transaction (or the emitted `DeployToken` event).
2. Extract the `signatureData` / `signature_rs` / `signature` field — it is a plain call argument, fully visible on-chain.
3. Replay the same call on every other chain before the bridge's own relayer does so.

The window is the time between the first `deploy_token` on chain A and the relayer's subsequent calls on chains B–D. In practice this window is minutes to hours, easily exploitable by a monitoring bot.

---

### Recommendation

Bind the destination chain ID into the `MetadataPayload` Borsh encoding, mirroring the pattern already used by `TransferMessagePayload`. Concretely, prepend (or append) the chain-specific constant before hashing:

```solidity
// EVM — OmniBridge.sol
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent change in `starknet/src/bridge_types.cairo` (`MetadataPayload::to_borsh`), `aptos/sources/bridge_types.move` (`metadata_to_borsh`), and `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs` (`DeployTokenPayload::serialize_for_near`). The NEAR MPC signing side must include the same chain ID when constructing the payload to sign.

### Proof of Concept

1. NEAR MPC signs a `MetadataPayload` for `token = "usdc.near"`, `name = "USD Coin"`, `symbol = "USDC"`, `decimals = 6`, targeting Ethereum. The relayer submits `deployToken(sig, payload)` on Ethereum — transaction is public.

2. Attacker extracts `sig` from the Ethereum calldata.

3. Attacker calls `deploy_token(sig, payload)` on Starknet, Aptos, and Solana with the identical payload and signature. Each chain computes:
   - EVM: `keccak256([0x01] ++ borsh("usdc.near") ++ borsh("USD Coin") ++ borsh("USDC") ++ [6])`
   - Starknet: `keccak256([0x01] ++ borsh("usdc.near") ++ borsh("USD Coin") ++ borsh("USDC") ++ [6])`
   - Aptos: `keccak256([0x01] ++ borsh("usdc.near") ++ borsh("USD Coin") ++ borsh("USDC") ++ [6])`
   - Solana: `keccak256([Metadata_tag] ++ borsh("usdc.near") ++ borsh("USD Coin") ++ borsh("USDC") ++ [6])`

   All four hashes are identical (same byte string). `ECDSA.recover(hash, sig)` returns `nearBridgeDerivedAddress` on every chain. All four `deploy_token` calls succeed.

4. Each chain now has `usdc.near` mapped to an attacker-triggered token address. The NEAR bridge has no record of these addresses. The bridge's own relayer subsequently tries to call `deploy_token` on Starknet/Aptos/Solana and is permanently rejected with `ERR_TOKEN_ALREADY_DEPLOYED`. Bridging USDC to those chains is frozen.
