### Title
Excess Attached Deposit Not Refunded in `submit_participant_info` When Caller Is an Existing Participant - (File: crates/contract/src/lib.rs)

### Summary

`MpcContract::submit_participant_info` is marked `#[payable]` and therefore explicitly accepts attached NEAR deposits. However, when an existing participant re-submits their attestation (i.e., `is_new_attestation == false` and `caller_is_not_participant == false`), the entire deposit-handling block is skipped. Any NEAR tokens attached to the call in this code path are silently absorbed by the contract with no refund, permanently trapping the caller's funds.

### Finding Description

The function `submit_participant_info` computes a boolean gate:

```rust
let attestation_storage_must_be_paid_by_caller =
    is_new_attestation || caller_is_not_participant;
``` [1](#0-0) 

Only when this flag is `true` does the function inspect `env::attached_deposit()`, enforce a minimum, and refund any surplus:

```rust
if attestation_storage_must_be_paid_by_caller {
    let storage_used = env::storage_usage().saturating_sub(initial_storage);
    let cost = env::storage_byte_cost().saturating_mul(storage_used as u128);
    let attached = env::attached_deposit();

    if attached < cost {
        return Err(InvalidParameters::InsufficientDeposit { ... }.into());
    }

    if let Some(diff) = attached.checked_sub(cost)
        && diff > NearToken::from_yoctonear(0)
    {
        Promise::new(account_id).transfer(diff).detach();
    }
}
``` [2](#0-1) 

When the flag is `false` — which happens whenever an **existing participant** re-submits their attestation (e.g., to rotate their TLS key or refresh an expiring quote) — the block is entirely bypassed. Because the function is `#[payable]`, the NEAR runtime does not reject an attached deposit; it is simply credited to the contract balance and never returned. [3](#0-2) 

Contrast this with the three user-facing request methods (`sign`, `request_app_private_key`, `verify_foreign_transaction`), which all call the shared `require_deposit` helper that unconditionally refunds any surplus:

```rust
fn require_deposit(minimum_deposit: NearToken, predecessor: &AccountId) {
    ...
    Some(diff) => {
        if diff > NearToken::from_yoctonear(0) {
            log!("refund excess deposit {diff} to {predecessor}");
            Promise::new(predecessor.clone()).transfer(diff).detach();
        }
    }
}
``` [4](#0-3) 

No equivalent guard exists for the `false` branch of `submit_participant_info`.

### Impact Explanation

Any NEAR tokens attached by an existing participant when re-submitting their attestation are permanently locked in the contract balance. While the contract is upgradeable via the `propose_update` / `vote_update` governance path, recovery requires a threshold of participants to agree on and execute an upgrade — there is no direct, per-caller recovery mechanism. This breaks the production accounting invariant that `#[payable]` functions must not silently consume caller funds beyond what is explicitly required.

**Impact: Medium** — balance accounting invariant violation; caller funds are permanently trapped without any direct recovery path for the affected account.

### Likelihood Explanation

Existing participants must periodically re-submit attestations (e.g., when their TEE quote expires or their TLS key rotates). Node operators running automated scripts or using tooling that always attaches a small deposit "to be safe" will silently lose those funds on every re-submission. The scenario is realistic and requires no special privileges beyond being an existing participant.

### Recommendation

1. Move the deposit refund logic **outside** the `attestation_storage_must_be_paid_by_caller` gate so that any excess is always returned, regardless of whether storage payment is required.
2. Alternatively, call the existing `require_deposit` helper (with `minimum_deposit = NearToken::from_yoctonear(0)`) unconditionally at the top of `submit_participant_info` so that any non-zero attached deposit is immediately refunded when no storage cost is owed.

### Proof of Concept

1. Participant `alice.near` is already registered in `tee_state` (existing participant, existing attestation).
2. `alice.near` calls `submit_participant_info` with a refreshed attestation and attaches `1 NEAR` as a deposit (e.g., because her tooling always attaches a deposit).
3. Inside the function:
   - `caller_is_not_participant` = `false` (alice is a participant)
   - `is_new_attestation` = `false` (her entry already exists; this is an update)
   - `attestation_storage_must_be_paid_by_caller` = `false || false` = `false`
4. The entire `if attestation_storage_must_be_paid_by_caller { ... }` block is skipped.
5. `env::attached_deposit()` is never read; no `Promise::transfer` is issued.
6. The `1 NEAR` is credited to the contract's balance and is unrecoverable by `alice.near` without a governance-approved contract upgrade. [5](#0-4)
