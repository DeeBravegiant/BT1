### Title
Caller-Agnostic `verify_foreign_transaction` Fan-Out Queue Allows Attacker to Permanently Block Specific Foreign-Tx Verification Requests - (File: crates/contract/src/lib.rs, crates/contract/src/pending_requests.rs)

### Summary

The `verify_foreign_transaction` endpoint uses a caller-agnostic request key, meaning any number of distinct NEAR accounts can fill the 128-slot fan-out queue for a specific foreign-chain transaction at a cost of 1 yoctoNEAR per slot. Once the queue is saturated, every subsequent legitimate call for that same foreign tx panics with `PendingRequestQueueFull`, permanently blocking that request from being queued until the attacker stops refilling it.

### Finding Description

The `MpcContract::verify_foreign_transaction` method converts caller arguments into a `VerifyForeignTransactionRequest` that contains only `(domain_id, payload_version, ForeignChainRpcRequest)` — no caller identity is included in the map key. [1](#0-0) 

This is in contrast to `sign()`, where `SignatureRequest::new` folds the caller's account ID into a `tweak` via `derive_tweak(predecessor_id, path)`, making each caller's key unique: [2](#0-1) [3](#0-2) 

The fan-out queue is capped at `MAX_PENDING_REQUEST_FAN_OUT = 128`. When the cap is reached, `push_pending_yield` calls `env::panic_str`, rejecting the caller's transaction outright: [4](#0-3) 

The minimum deposit required to call `verify_foreign_transaction` is `MINIMUM_SIGN_REQUEST_DEPOSIT = 1 yoctoNEAR`: [5](#0-4) [6](#0-5) 

The codebase itself acknowledges the caller-agnostic nature of the key in a test comment: [7](#0-6) 

### Impact Explanation

An attacker submits 128 `verify_foreign_transaction` calls for a specific foreign-chain transaction (e.g., a Bitcoin `tx_id` + extractor set) at a total cost of 128 yoctoNEAR plus gas. The fan-out queue for that request key is now saturated. Any subsequent legitimate user who calls `verify_foreign_transaction` with the same parameters receives a `PendingRequestQueueFull` panic and their transaction is rejected.

The MPC nodes will still process the request (since the attacker's 128 entries are in the queue) and call `respond_verify_foreign_tx`, which drains all 128 entries at once. However, the attacker can immediately refill the queue after each drain cycle, maintaining a persistent block on that specific foreign tx at essentially zero cost. This breaks the request-lifecycle invariant: a legitimate user is permanently unable to queue a valid foreign-tx verification request for a targeted transaction as long as the attacker continues the attack.

If the foreign-tx verification is used to authorize a time-sensitive bridge operation (e.g., a cross-chain swap with a deadline), the victim loses their funds or misses the window.

### Likelihood Explanation

- Cost to attacker: 128 yoctoNEAR + NEAR gas per cycle (negligible).
- No privileged access required: any NEAR account can call `verify_foreign_transaction`.
- The attacker only needs to know the target `(domain_id, tx_id, chain, extractors)` tuple, which is observable on-chain once the victim submits their first request.
- The attack can be automated to refill the queue immediately after each MPC response drains it.

### Recommendation

Include the caller's `predecessor_account_id` in the `VerifyForeignTransactionRequest` key, mirroring the `sign()` design where `derive_tweak(predecessor_id, path)` makes each caller's key unique. Alternatively, enforce a per-account rate limit or per-account slot reservation within the fan-out queue so a single attacker cannot monopolize all 128 slots for a given foreign tx.

### Proof of Concept

1. Alice submits `verify_foreign_transaction` for Bitcoin `tx_id = [0xAA; 32]` with 1 yoctoNEAR deposit. Her yield is queued at slot 0.
2. Attacker submits the identical request 127 more times from 127 different accounts (or the same account in sequence), filling slots 1–127. Total cost: 127 yoctoNEAR.
3. Alice's legitimate retry (or Bob's first attempt for the same tx) panics: `Pending-request queue is full (limit: 128)`.
4. MPC nodes observe the 128 on-chain events, process the foreign-tx verification once, and call `respond_verify_foreign_tx`, draining all 128 slots.
5. Attacker immediately resubmits 128 calls, re-saturating the queue before Alice or Bob can retry.
6. Alice/Bob are permanently blocked from receiving a foreign-tx verification response for that transaction. [8](#0-7) [1](#0-0) [9](#0-8)

```rust
/// Maximum number of concurrent yield-resume promises that can be queued for a single
/// request key (i.e. the number of duplicate submissions whose responses fan out from
/// one MPC reply).
///
/// The ceiling is needed because `respond*` drains the entire queue in one call: every
/// queued yield triggers a host-side `promise_yield_resume`, paid for out of the
/// responder's 300 TGas budget. Without a cap, an attacker could enqueue enough
/// duplicates to make `respond*` run out of gas and strand every queued caller.
///
/// 128 is validated empirically by the sandbox test
/// `test_contract_request_duplicate_requests_fan_out`, which fills the queue to this
/// cap across all four signature schemes and confirms `respond*` drains it inside its
/// 300 TGas budget.
pub const MAX_PENDING_REQUEST_FAN_OUT: u8 = 128;

/// Append a yield index to the pending-request fan-out queue for `request`.
///
/// Panics with `RequestError::PendingRequestQueueFull` if the resulting queue would
/// exceed `MAX_PENDING_REQUEST_FAN_OUT`.
pub(crate) fn push_pending_yield<K>(
    requests: &mut LookupMap<K, Vec<YieldIndex>>,
    request: K,
    data_id: CryptoHash,
) where
    K: BorshSerialize + BorshDeserialize + Clone + Ord,
{
    let queue = requests.entry(request).or_default();
    if queue.len() >= usize::from(MAX_PENDING_REQUEST_FAN_OUT) {
        env::panic_str(
            &RequestError::PendingRequestQueueFull {
                limit: MAX_PENDING_REQUEST_FAN_OUT,
            }
            .to_string(),
        );
    }
    queue.push(YieldIndex { data_id });
```
