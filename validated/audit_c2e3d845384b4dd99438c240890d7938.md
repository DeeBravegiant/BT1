### Title
Stale Votes from Removed Participants Counted Without Participant-Set Filtering in `LauncherHashVotes::count_votes`, Enabling Launcher-Hash Whitelisting Below True Threshold - (File: crates/contract/src/tee/proposal.rs)

### Summary

`LauncherHashVotes::count_votes` tallies every entry in `vote_by_account` without filtering against the current participant set. After a resharing removes participants, their votes persist in the map and are still counted toward the governance threshold in `vote_add_launcher_hash`. An adversary who controls one or more participants that are subsequently removed can pre-cast a vote for a malicious launcher image hash; those stale votes combine with votes from fewer current participants than the threshold actually requires to whitelist the hash.

### Finding Description

`vote_add_launcher_hash` in `crates/contract/src/lib.rs` authenticates the caller against the current participant set, then delegates vote counting to `tee_state.vote_launcher`, which calls `LauncherHashVotes::vote` and then `count_votes`:

```rust
// lib.rs ~1450
let participant = AuthenticatedParticipantId::new(threshold_parameters.participants())?;
let action = LauncherVoteAction::Add(launcher_hash);
let votes = self.tee_state.vote_launcher(action, &participant);

if votes >= self.threshold()?.value() {
    let added = self.tee_state.add_launcher_image(launcher_hash, ...);
}
```

`count_votes` iterates the entire `vote_by_account` map with no participant-membership filter:

```rust
// proposal.rs ~112
fn count_votes(&self, action: &LauncherVoteAction) -> u64 {
    u64::try_from(
        self.vote_by_account
            .values()
            .filter(|a| *a == action)
            .count(),
    )
    .expect("participant count should not overflow u64")
}
```

Contrast this with `ForeignChainRpcWhitelist`, which explicitly filters stale votes at count time via a `count_for` predicate that checks `participants.is_participant_given_account_id`, and with `TeeVerifierVotes::retain`, which purges non-participant entries. `LauncherHashVotes` provides only `get_remaining_votes` (a read-only helper that returns a filtered copy) but does not apply that filter inside `count_votes` itself. If the post-resharing cleanup promise (`clean_tee_status`) fails or is delayed, stale votes accumulate and are counted. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

A whitelisted launcher image hash is used to derive `LauncherDockerComposeHash` values that TEE nodes accept as valid boot images. Whitelisting a malicious launcher hash allows an attacker to boot nodes under a compromised image that can exfiltrate key shares or produce unauthorized signatures. Because the threshold check uses a vote count inflated by stale entries, the effective governance threshold for this critical security decision is lower than the protocol intends, constituting a bypass of threshold-signature requirements and unauthorized access to MPC key-share material. [4](#0-3) 

### Likelihood Explanation

The precondition is that one or more participants cast a vote for a target launcher hash and are subsequently removed from the participant set (e.g., via `verify_tee` kicking out nodes with expired TEE attestations) before the cleanup promise executes or while it is pending. NEAR promise failures are rare but possible under gas exhaustion or contract panics. The scenario is realistic in any epoch where participants rotate out while governance votes are in flight, which is an expected operational condition. [5](#0-4) 

### Recommendation

Apply the same participant-membership filter inside `count_votes` that `ForeignChainRpcWhitelist` applies at count time. Pass the current `Participants` set into `count_votes` and filter `vote_by_account` entries against it before summing:

```rust
fn count_votes(&self, action: &LauncherVoteAction, participants: &Participants) -> u64 {
    u64::try_from(
        self.vote_by_account
            .iter()
            .filter(|(pid, a)| {
                *a == action
                    && participants.is_participant_given_participant_id(&pid.get())
            })
            .count(),
    )
    .expect("participant count should not overflow u64")
}
```

Propagate `participants` through `LauncherHashVotes::vote` and `TeeState::vote_launcher`. Apply the same fix to the analogous code-hash vote counter in `TeeState::vote` if it shares the same pattern.

### Proof of Concept

1. Network has 5 participants `{P1, P2, P3, P4, P5}`, governance threshold = 4.
2. `P1` calls `vote_add_launcher_hash(H_evil)`. `vote_by_account = {P1 → Add(H_evil)}`, count = 1 < 4.
3. `P2` calls `vote_add_launcher_hash(H_evil)`. count = 2 < 4.
4. Legitimate resharing removes `P1` and `P2` (e.g., TEE attestations expired). New participant set = `{P3, P4, P5}`, threshold = 3. Post-resharing cleanup promise fails or is not yet executed; `vote_by_account` still contains `{P1 → Add(H_evil), P2 → Add(H_evil)}`.
5. `P3` calls `vote_add_launcher_hash(H_evil)`. `count_votes` iterates all three entries (P1, P2, P3) and returns 3 ≥ threshold 3. `H_evil` is whitelisted.
6. Only 1 current participant (`P3`) voted for `H_evil`, but the threshold of 3 was satisfied using 2 stale votes from removed participants. [6](#0-5) [7](#0-6)
