### Title
Gateway Admission Validates L2 Gas Price Against Stale Previous-Block Price Instead of Next-Block Price — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` checks a transaction's `max_l2_gas_price` against the **previous block's** L2 gas price, but actual execution will use the **next block's** L2 gas price (computed via EIP-1559 from the previous block's price and gas usage). This is the sequencer-native analog of the BellumNursery "wrong snapshot" bug: the admission decision is anchored to a stale price snapshot rather than the price that will govern execution.

### Finding Description

In `validate_resource_bounds`, the gateway reads `previous_block_l2_gas_price` from `get_block_info()` and passes it to `validate_tx_l2_gas_price_within_threshold`. The code itself carries an explicit developer acknowledgment of the error:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold computed inside `validate_tx_l2_gas_price_within_threshold` is:

```rust
let threshold = (gas_price_threshold_multiplier * previous_block_l2_gas_price.get().0).to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(StarknetError { ... "GAS_PRICE_TOO_LOW" ... });
}
``` [2](#0-1) 

The **next** block's L2 gas price is computed via EIP-1559 (`calculate_next_base_gas_price`) and is already stored in the block header as `next_l2_gas_price`: [3](#0-2) 

The gateway's `run_validate_entry_point` also runs the `__validate__` entry point using the previous block's gas prices (with only `block_number` incremented), so the on-chain validate call does not catch the mismatch either: [4](#0-3) 

This creates two failure modes:

**Mode 1 — Admission of transactions that will fail execution** (when `next_price > previous_price`, i.e., high gas usage):

A transaction with `max_l2_gas_price M` satisfying:

```
threshold(previous_price) ≤ M < next_block_price
```

passes the gateway check (admitted) but will be rejected or reverted by the blockifier during execution because `M < next_block_price`.

**Mode 2 — Rejection of valid transactions** (when `next_price < previous_price`, i.e., low gas usage):

A transaction with `max_l2_gas_price M` satisfying:

```
next_block_price ≤ M < threshold(previous_price)
```

is rejected by the gateway even though it would succeed during execution.

### Impact Explanation

This is a **High** impact finding matching: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

- **Mode 1**: The gateway admits transactions into the mempool that will fail during blockifier execution. These transactions consume mempool capacity, are included in blocks, and cause users to pay fees for failed transactions. An unprivileged user can trigger this by submitting a transaction with `max_l2_gas_price` just above the stale threshold but below the actual next-block price — a gap that widens with high network utilization.
- **Mode 2**: The gateway rejects transactions that would execute successfully, causing valid user transactions to be silently dropped at the admission boundary.

The `run_validate_entry_point` path does not compensate: it also uses the previous block's gas prices for the `__validate__` call, so no secondary guard catches the mismatch.

### Likelihood Explanation

- The `validate_resource_bounds` flag must be enabled (`self.config.validate_resource_bounds = true`). The code path is production-present and the TODO comment confirms it is intended to be active.
- EIP-1559 bounds the per-block price change, so the exploitable gap is proportional to the price delta per block. Under sustained high utilization the gap accumulates across multiple blocks.
- No privilege is required; any user submitting a v3 (`AllResources`) transaction can trigger either mode.
- The developer TODO comment is a direct admission that the wrong price is being used.

### Recommendation

Replace the `previous_block_l2_gas_price` read with the `next_l2_gas_price` field already stored in the block header:

```rust
// Before (wrong snapshot):
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;

// After (correct snapshot):
let next_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_next_l2_gas_price()   // read next_l2_gas_price from block header
    .await?;
```

`StorageBlockHeader.next_l2_gas_price` is already populated by the sequencer and is the exact value the blockifier will use as the block's L2 gas price for the next block.

### Proof of Concept

```
Previous block:
  l2_gas_price        = 100 gwei  (stored in block header)
  next_l2_gas_price   = 101 gwei  (EIP-1559 increase due to high usage)
  min_gas_price_percentage = 100%

Gateway threshold = 100% × 100 gwei = 100 gwei

User submits tx with max_l2_gas_price = 100 gwei:
  Gateway check:  100 >= 100  → ADMITTED
  Blockifier:     100 < 101   → EXECUTION FAILS (fee too low)

User submits tx with max_l2_gas_price = 99 gwei (next block price = 98 gwei):
  Gateway check:  99 < 100    → REJECTED
  Blockifier:     99 >= 98    → would have SUCCEEDED
```

The wrong snapshot (previous-block price used at admission time instead of next-block price used at execution time) is the direct sequencer analog of BellumNursery's wrong-time share-ratio snapshot: the reference value is read from the wrong point in the lifecycle, causing the admission decision to diverge from the execution outcome.     // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
