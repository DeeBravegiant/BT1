Audit Report

## Title
Block Stuffing Can Delay Emergency Deposit-Limit or Fee Reduction by Manager — (`contracts/LRTConfig.sol`)

## Summary
`updateAssetDepositLimit` and `setProtocolFeeBps` in `LRTConfig.sol` are single-SSTORE manager transactions with no timelock or private-relay guarantee. A well-capitalised attacker can fill consecutive blocks to delay these transactions, keeping the old deposit limit or fee rate in effect for additional blocks. During the delay window, `_beforeDeposit` continues to enforce the stale config value, allowing deposits that the manager intended to halt and fee accrual at a rate the manager intended to reduce.

## Finding Description
`updateAssetDepositLimit` writes a single storage slot with no timelock: [1](#0-0) 

`setProtocolFeeBps` similarly writes one storage slot with no timelock: [2](#0-1) 

The deposit limit is enforced at deposit time via `_beforeDeposit`, which calls `_checkIfDepositAmountExceedesCurrentLimit`: [3](#0-2) 

That check reads `lrtConfig.depositLimitByAsset(asset)` live from storage: [4](#0-3) 

Until the manager's transaction is mined, the old value persists and `_beforeDeposit` continues to accept deposits up to the old limit. An attacker monitoring the mempool for an emergency `updateAssetDepositLimit(asset, 0)` can submit high-gas filler transactions (e.g., tight SSTORE loops) with a higher effective gas price, consuming the entire block gas limit and pushing the manager's transaction out for K blocks. The `pauseAll()` path is equally subject to this attack, as it is also a plain mempool transaction: [5](#0-4) 

## Impact Explanation
**Low — Block stuffing / Contract fails to deliver promised returns.**

During the delay window the protocol continues to accept deposits into an asset the manager intended to halt, and continues to collect fees at a rate the manager intended to reduce. No funds are directly stolen, but the protocol fails to deliver the emergency halt it promised. This matches the allowed impact class: *"Low. Block stuffing."* and *"Low. Contract fails to deliver promised returns, but doesn't lose value."*

## Likelihood Explanation
Block stuffing on Ethereum mainnet requires filling approximately 30 M gas per block, which is expensive but economically rational for an attacker who benefits from continued deposits into a compromised or paused asset (e.g., an LST that has depegged). The attack requires no privileged access — any EOA can submit filler transactions. The manager has no on-chain fallback that bypasses the mempool for these specific config writes.

## Recommendation
1. **Use a private relay (Flashbots `eth_sendPrivateTransaction`) for all emergency manager transactions** so filler transactions cannot displace them.
2. **Establish a runbook policy**: in an emergency, the `PAUSER_ROLE` should call `pauseAll()` first; while equally subject to stuffing, it is a simpler, lower-gas target and halts all deposits immediately.
3. **Optionally add an immediate-effect flag for limit reductions**: if the new deposit limit is strictly lower than the current one, enforce it via a flag that takes effect at the next block boundary rather than requiring a separate mined transaction.

## Proof of Concept
Foundry local fork test (no mainnet testing):

```solidity
// 1. Fork mainnet state locally.
// 2. Manager submits updateAssetDepositLimit(stETH, 0) with base-fee priority.
// 3. Attacker submits N filler transactions each consuming ~29.9 M gas per block
//    (tight SSTORE loop) with higher effective gas price.
// 4. Assert manager tx is not included for K blocks.
// 5. Assert depositAsset(stETH, amount, 0, "") still succeeds during those K blocks
//    because depositLimitByAsset[stETH] still equals the old value.
// 6. After K blocks, manager tx lands; assert depositAsset now reverts with
//    MaximumDepositLimitReached.

contract BlockStuffingFiller {
    uint256[500] private _slots;
    function fill() external {
        for (uint256 i; i < 500; ++i) {
            _slots[i] = block.number; // 500 SSTOREs ≈ 10 M gas
        }
    }
}
```

The test measures `lrtConfig.depositLimitByAsset(stETH)` before and after the stuffed blocks to confirm the old limit persists, and calls `depositPool.depositAsset(stETH, amount, 0, "")` within the stuffed window to confirm deposits are accepted at the old limit. [6](#0-5)

        IPausable lrtDepositPool = IPausable(getContract(LRTConstants.LRT_DEPOSIT_POOL));
        IPausable lrtWithdrawalManager = IPausable(getContract(LRTConstants.LRT_WITHDRAW_MANAGER));
        IPausable lrtOracle = IPausable(getContract(LRTConstants.LRT_ORACLE));
        IPausable rsETHContract = IPausable(rsETH);

        if (!lrtDepositPool.paused()) lrtDepositPool.pause();
        if (!lrtWithdrawalManager.paused()) lrtWithdrawalManager.pause();
        if (!lrtOracle.paused()) lrtOracle.pause();
        if (!rsETHContract.paused()) rsETHContract.pause();

        address[] memory nodeDelegatorQueue = ILRTDepositPool(address(lrtDepositPool)).getNodeDelegatorQueue();
        uint256 nodeDelegatorCount = nodeDelegatorQueue.length;

        for (uint256 i = 0; i < nodeDelegatorCount;) {
            IPausable nodeDelegator = IPausable(nodeDelegatorQueue[i]);
            if (!nodeDelegator.paused()) nodeDelegator.pause();
            unchecked {
                ++i;
            }
        }

        emit PausedAll(msg.sender);
    }

    function _checkIfDepositAmountExceedesCurrentLimit(address asset, uint256 amount) internal view returns (bool) {
        uint256 totalAssetDeposits = getTotalAssetDeposits(asset);
        if (asset == LRTConstants.ETH_TOKEN) {
            return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
        }
        return (totalAssetDeposits + amount > lrtConfig.depositLimitByAsset(asset));
    }
