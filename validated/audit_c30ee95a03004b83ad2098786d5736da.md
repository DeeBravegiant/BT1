### Title
Participant Joins Active Set Without Valid TEE Attestation After Resharing Completes — (File: `crates/contract/src/lib.rs`)

### Summary

The TEE attestation check performed inside `vote_new_parameters` is a point-in-time snapshot. Once the contract transitions to the Resharing state, no re-validation of all proposed participants' attestations is performed when `vote_reshared` crosses the threshold and the contract transitions back to Running. A candidate node whose attestation expires during the resharing window is silently promoted into the active participant set without a valid TEE attestation, directly analogous to the TSS deposit-withdrawal frontrun: the eligibility check passes at election time, but the security invariant is violated by the time the node is actually active.

---

### Finding Description

`vote_new_parameters` calls `self.tee_state.reverify_and_cleanup_participants(proposal.participants(), ...)` and rejects the vote if any proposed participant lacks a valid attestation (`TeeValidationResult::Partial` → `InvalidTeeRemoteAttestation` error). This is the only gate that enforces the "all proposed participants must be TEE-attested" invariant before the contract transitions to `ProtocolContractState::Resharing`. [1](#0-0) 

Once the contract is in Resharing state, `vote_reshared` calls `assert_caller_is_attested_participant_and_protocol_active()`, which checks only the **caller's own** attestation status — not the attestation status of every participant in the proposed set. [2](#0-1) 

When the threshold of `vote_reshared` votes is reached, `ResharingContractState::vote_reshared` constructs the new `RunningContractState` directly from `self.resharing_key.proposed_parameters().clone()` — the parameters that were frozen at the time `vote_new_parameters` transitioned to Resharing. There is no attestation re-check at this transition point. [3](#0-2) 

`assert_caller_is_attested_participant_and_protocol_active` only validates the signer, not the full proposed set: [4](#0-3) 

**Attack scenario:**

1. Candidate node A submits a valid TEE attestation with a short remaining validity window.
2. Existing participants call `vote_new_parameters` including A in the proposal. `reverify_and_cleanup_participants` passes for all proposed participants → contract enters Resharing state.
3. A's attestation expires during the resharing period (which can span multiple domains and multiple timed-out attempts).
4. Participants B, C, … call `vote_reshared`; each call checks only the caller's own attestation. The threshold is reached without A needing to call `vote_reshared` at all.
5. `ResharingContractState::vote_reshared` returns `Some(RunningContractState::new(..., proposed_parameters.clone(), ...))` — A is now an active participant in the Running state with no valid TEE attestation. [5](#0-4) 

The post-resharing cleanup promises (`CLEAN_TEE_STATUS`, `CLEAN_INVALID_ATTESTATIONS`) prune stale stored attestation entries but do **not** remove A from the active participant set; only a subsequent `verify_tee()` call can do that, and it requires a separate transaction. [6](#0-5) 

---

### Impact Explanation

A node is promoted into the active MPC participant set without a valid TEE attestation. The TEE attestation is the contract's mechanism for ensuring that key-share material is held inside a trusted hardware enclave. A participant active without a valid attestation:

- Holds a live key share that was distributed during resharing, potentially outside a TEE boundary.
- Can participate in threshold signing rounds (subject to whether `TeeQuoteStatus::None` is accepted by `is_caller_an_attested_participant`; if it is, the node signs immediately; if not, it can re-submit a weaker attestation to regain call access).
- Breaks the production safety invariant that every active participant must satisfy the TEE security policy, which is the contract's primary accountability and trust mechanism.

This maps to the **Medium** allowed impact: *participant-state manipulation that breaks production safety/accounting invariants*.

---

### Likelihood Explanation

- TEE attestations have finite validity periods (certificate expiry, image-hash grace periods). A resharing involving multiple domains and/or timed-out attempts can easily span the remaining validity window of a freshly submitted attestation.
- The attacker controls when they submit their attestation and can choose an expiry that falls inside the expected resharing window.
- No privileged access is required; the attacker is a candidate node that legitimately passes the `vote_new_parameters` TEE check.
- The window is not closed by any existing on-chain mechanism until `verify_tee()` is explicitly called by another participant.

---

### Recommendation

Re-validate the TEE attestation status of **all** proposed participants at the moment `vote_reshared` causes the final transition to Running state, before constructing the new `RunningContractState`. Concretely, inside `MpcContract::vote_reshared`, after `self.protocol_state.vote_reshared(key_event_id)?` returns `Some(new_state)`, call `self.tee_state.reverify_and_cleanup_participants(new_state.parameters.participants(), ...)` and either reject the transition or automatically exclude participants whose attestations have since expired (analogous to the on-chain solution proposed in the original report: add a check to `setTssGroupMember()` that validates each elected node still has a sufficient deposit).

---

    pub fn vote_new_parameters(
        &mut self,
        prospective_epoch_id: EpochId,
        proposal: dtos::ProposedThresholdParameters,
    ) -> Result<(), Error> {
        Self::assert_caller_is_signer();
        let proposal: ProposedThresholdParameters = proposal.try_into_contract_type()?;
        log!(
            "vote_new_parameters: signer={}, proposal={:?}",
            env::signer_account_id(),
            proposal,
        );

        let tee_upgrade_deadline_duration =
            Duration::from_secs(self.config.tee_upgrade_deadline_duration_seconds);

        let validation_result = self.tee_state.reverify_and_cleanup_participants(
            proposal.participants(),
            tee_upgrade_deadline_duration,
        );

        let proposed_participants = proposal.participants();
        match validation_result {
            TeeValidationResult::Full => {
                if let Some(new_state) = self
                    .protocol_state
                    .vote_new_parameters(prospective_epoch_id, &proposal)?
                {
                    self.protocol_state = new_state;
                }
                Ok(())
            }
            TeeValidationResult::Partial {
                participants_with_valid_attestation,
            } => {
                let invalid_participants: Vec<_> = proposed_participants
                    .participants()
                    .iter()
                    .filter(|(account_id, _, _)| {
                        !participants_with_valid_attestation
                            .is_participant_given_account_id(account_id)
                    })
                    .collect();

                Err(InvalidParameters::InvalidTeeRemoteAttestation {
                    reason: format!(
                        "The following participants have invalid TEE status: {:?}",
                        invalid_participants
                    ),
                }
                .into())
            }
        }
    }
