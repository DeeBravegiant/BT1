Audit Report

## Title
Missing Token Decimal Normalization in `viewSwapRsETHAmountAndFee` Causes Severe rsETH Underpayment for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
`viewSwapRsETHAmountAndFee(uint256 amount, address token)` in `RSETHPoolV3` (and five sibling contracts) multiplies a raw token amount directly by a Chainlink-normalized 18-decimal rate without first scaling the token amount to 18 decimals. For any non-18-decimal token added as supported collateral, depositors receive `10^(18 - tokenDecimals)` times fewer rsETH/wrsETH than owed while their deposited tokens are permanently retained by the pool.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink answer to 18 decimals, returning the price of **1 whole token** in ETH:

```solidity
// contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol L34
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

`viewSwapRsETHAmountAndFee` then uses this rate directly against the raw token amount:

```solidity
// contracts/pools/RSETHPoolV3.sol L324-334
fee = amount * feeBps / 10_000;
uint256 amountAfterFee = amount - fee;
uint256 rsETHToETHrate = getRate();                                    // 18-decimal
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate(); // 18-decimal
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

`tokenToETHRate` represents the ETH value of 1 whole token (i.e., `1e18` raw units). When `amountAfterFee` is in native raw units (e.g., `1e6` for 1 USDT), the formula is dimensionally incorrect: it treats `1e6` raw units as if they were `1e6` units of 18-decimal precision, producing a result `10^12` times too small.

`addSupportedToken` in `RSETHPoolV3` imposes **no restriction on token decimals**:

```solidity
// contracts/pools/RSETHPoolV3.sol L541-554
function addSupportedToken(address token, address oracle) external onlyRole(TIMELOCK_ROLE) {
    UtilLib.checkNonZeroAddress(token);
    UtilLib.checkNonZeroAddress(oracle);
    if (supportedTokenOracle[token] != address(0)) revert AlreadySupportedToken();
    if (IOracle(oracle).getRate() == 0) revert UnsupportedOracle();
    supportedTokenList.push(token);
    supportedTokenOracle[token] = oracle;
    emit AddSupportedToken(token);
}
```

Once a non-18-decimal token is added (a legitimate admin action), every subsequent call to the public `deposit(address token, uint256 amount, string referralId)` function by any user triggers the accounting error. The same broken formula is present identically in:
- `contracts/pools/RSETHPool.sol` L346
- `contracts/pools/RSETHPoolNoWrapper.sol` L311
- `contracts/pools/RSETHPoolV3ExternalBridge.sol` L452
- `contracts/pools/RSETHPoolV3WithNativeChainBridge.sol` L370
- `contracts/agETH/AGETHPoolV3.sol` L194

## Impact Explanation
**Critical — Direct theft of user funds.**

For 1 USDT (6 decimals, ETH price $3000, rsETH ≈ 1.05 ETH):
- `amountAfterFee = 1e6`
- `tokenToETHRate ≈ 3.33e14` (1 USDT in ETH, 18-dec normalized)
- `rsETHToETHrate ≈ 1.05e18`
- **Actual minted:** `1e6 * 3.33e14 / 1.05e18 ≈ 317 wei` (~$0)
- **Correct amount:** `≈ 3.17e14 wei` (~$0.001 worth of rsETH)

The user's USDT is transferred into the pool via `safeTransferFrom` before `viewSwapRsETHAmountAndFee` is called. There is no slippage guard or minimum-output check in the token deposit path. The deposited tokens accumulate in the pool with no recovery mechanism for the depositor.

## Likelihood Explanation
Medium. The precondition is a TIMELOCK_ROLE admin adding a non-18-decimal token, which is a legitimate and expected protocol operation given the contract's design to accept arbitrary ERC-20 collateral. No decimal restriction exists anywhere in the codebase. Once such a token is listed, the bug is triggered by any unprivileged user calling the public `deposit(address, uint256, string)` function — no special role, no victim mistake, no external dependency failure required.

## Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate calculation in all affected contracts:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

function viewSwapRsETHAmountAndFee(uint256 amount, address token)
    public view onlySupportedToken(token)
    returns (uint256 rsETHAmount, uint256 fee)
{
    fee = amount * feeBps / 10_000;
    uint256 amountAfterFee = amount - fee;

    uint8 tokenDecimals = IERC20Metadata(token).decimals();
    uint256 normalizedAmount = amountAfterFee * 1e18 / 10 ** uint256(tokenDecimals);

    uint256 rsETHToETHrate = getRate();
    uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();

    rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
}
```

Apply the same fix to all six affected pool contracts.

## Proof of Concept
1. Deploy `RSETHPoolV3` with a valid `wrsETH` and `rsETHOracle`.
2. Admin calls `addSupportedToken(USDT_ADDRESS, chainlinkUSDTOracle)` — no decimal check blocks this.
3. Any user calls `deposit(USDT_ADDRESS, 1_000_000, "ref")` (1 USDT = `1e6` raw units).
4. Pool executes `IERC20(USDT).safeTransferFrom(msg.sender, address(this), 1_000_000)` — 1 USDT leaves the user.
5. `viewSwapRsETHAmountAndFee(1_000_000, USDT)` computes:
   - `fee = 0` (0 bps)
   - `tokenToETHRate ≈ 3.33e14`
   - `rsETHToETHrate ≈ 1.05e18`
   - `rsETHAmount = 1e6 * 3.33e14 / 1.05e18 = 317`
6. `wrsETH.mint(msg.sender, 317)` — user receives 317 wei of wrsETH (worth ~$0).
7. The 1 USDT (~$1) remains in the pool; no depositor recovery path exists.

**Foundry fork test plan:** Fork a chain where USDT is deployed. Deploy `RSETHPoolV3`, add USDT via `addSupportedToken`. Call `deposit(USDT, 1e6, "")`. Assert `wrsETH.balanceOf(depositor) < 1000` and `IERC20(USDT).balanceOf(pool) == 1e6`. Compare against the correct formula output to confirm the `10^12` discrepancy.
