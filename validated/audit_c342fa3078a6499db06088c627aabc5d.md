Audit Report

## Title
Missing Zero-Rate Validation in `viewSwapRsETHAmountAndFee` Allows Silent Loss of Deposited Tokens - (File: contracts/pools/RSETHPoolV3.sol)

## Summary
`RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)` fetches the token-to-ETH rate from the oracle and uses it directly in the rsETH calculation without a zero-check. If the oracle returns 0, the function silently returns `rsETHAmount = 0`. The `deposit` function transfers the user's tokens into the pool first, then mints 0 rsETH — the user's assets are locked in the pool with no user-accessible on-chain recovery path. The identical pattern exists in `RSETHPool` and `RSETHPoolNoWrapper`.

## Finding Description
In `RSETHPoolV3.viewSwapRsETHAmountAndFee(uint256 amount, address token)`, the oracle rate is used directly without validation:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [1](#0-0) 

When `tokenToETHRate == 0`, the multiplication yields 0 and `rsETHAmount` returns 0 with no revert.

The sibling function `viewSwapAssetToPremintedRsETH` in the same contract performs the validation that is missing here:

```solidity
uint256 tokenToETHRate = token == ETH_IDENTIFIER ? 1e18 : IOracle(supportedTokenOracle[token]).getRate();
if (tokenToETHRate == 0) revert UnsupportedOracle();
``` [2](#0-1) 

The `deposit(address token, uint256 amount, string memory referralId)` function transfers tokens **before** computing `rsETHAmount`, and has no guard on the returned value:

```solidity
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
feeEarnedInToken[token] += fee;
wrsETH.mint(msg.sender, rsETHAmount);
``` [3](#0-2) 

`wrsETH.mint(msg.sender, 0)` is a valid ERC-20 call that succeeds silently, completing the transaction with the user receiving nothing.

The `limitDailyMint` modifier also calls `viewSwapRsETHAmountAndFee` and checks `dailyMintAmount + rsETHAmount > dailyMintLimit`. When `rsETHAmount == 0`, this check trivially passes (0 ≤ limit), providing no protection. [4](#0-3) 

The one-time zero-rate check in `addSupportedToken` does not protect against the oracle returning 0 after registration: [5](#0-4) 

The same vulnerable pattern exists in `RSETHPool.viewSwapRsETHAmountAndFee`: [6](#0-5) 

And in `RSETHPoolNoWrapper.viewSwapRsETHAmountAndFee`: [7](#0-6) 

In `RSETHPool` and `RSETHPoolNoWrapper`, the outcome is slightly different: `safeTransfer(msg.sender, 0)` is called instead of `mint`, but the result is identical — 0 rsETH delivered to the user while their tokens remain locked. [8](#0-7) [9](#0-8) 

There is no user-accessible withdrawal function in any of the three pool contracts. All asset-recovery functions (`moveAssetsForBridging`, `withdrawFees`, `bridgeAssets`, `bridgeTokens`) are restricted to `BRIDGER_ROLE` or `TIMELOCK_ROLE`.

## Impact Explanation
When `tokenToETHRate` is 0 at deposit time: the user's ERC-20 tokens are transferred into the pool, `rsETHAmount = 0` is computed silently, and `mint(user, 0)` or `safeTransfer(user, 0)` completes without revert. The user receives 0 rsETH while their tokens are locked in the pool with no user-accessible on-chain recovery path. This constitutes **Medium — Temporary freezing of funds** (admin can potentially recover via contract upgrade or out-of-band intervention, but no user-facing recovery function exists on-chain).

## Likelihood Explanation
The oracle returning 0 is a recognized failure mode — the protocol itself guards against it at registration time (`addSupportedToken`) and in `viewSwapAssetToPremintedRsETH`, demonstrating explicit awareness of the risk. Price feed contracts can return 0 on staleness, circuit-breaker activation, or an upgrade bug. No privileged access is required to trigger the loss: any unprivileged depositor calling `deposit(token, amount, referralId)` during a window where the oracle returns 0 is affected. The vulnerability is present across all three pool contracts deployed on multiple L2 chains, widening the affected surface.

## Recommendation
Add a zero-rate guard in `viewSwapRsETHAmountAndFee(uint256 amount, address token)` in all three contracts, consistent with `viewSwapAssetToPremintedRsETH`:
```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
if (tokenToETHRate == 0) revert UnsupportedOracle();
```
Additionally, reorder `deposit` to compute `rsETHAmount` before the token transfer and add a zero-amount guard:
```solidity
(uint256 rsETHAmount, uint256 fee) = viewSwapRsETHAmountAndFee(amount, token);
if (rsETHAmount == 0) revert InvalidAmount();
IERC20(token).safeTransferFrom(msg.sender, address(this), amount);
```
Apply the same fix to `RSETHPool` and `RSETHPoolNoWrapper`.

## Proof of Concept
1. Token `T` is added via `addSupportedToken(T, oracle, bridge)` — oracle returns a valid non-zero rate at this point.
2. The oracle for `T` later returns 0 (staleness, circuit-breaker, or upgrade bug).
3. User calls `RSETHPoolV3.deposit(T, 1e18, "ref")`.
4. `limitDailyMint` modifier calls `viewSwapRsETHAmountAndFee(1e18, T)` → `rsETHAmount = 0`; daily limit check passes (`0 ≤ limit`).
5. `deposit` executes: `safeTransferFrom` moves 1e18 T tokens from user to pool.
6. `viewSwapRsETHAmountAndFee` returns `rsETHAmount = 0`.
7. `wrsETH.mint(user, 0)` succeeds silently.
8. User holds 0 rsETH; 1e18 T tokens are locked in the pool.

**Foundry test plan:**
```solidity
function test_zeroOracleRateSilentLoss() public {
    // Deploy mock oracle that returns 0
    MockOracle zeroOracle = new MockOracle(0);
    // Add token with valid oracle, then swap oracle to zero-returning one
    pool.addSupportedToken(address(token), address(validOracle), bridge);
    pool.setSupportedTokenOracle(address(token), address(zeroOracle)); // requires TIMELOCK_ROLE
    // Alternatively: mock the oracle to return 0 after registration
    
    uint256 balanceBefore = token.balanceOf(user);
    vm.prank(user);
    pool.deposit(address(token), 1e18, "ref");
    
    assertEq(wrsETH.balanceOf(user), 0);           // user got nothing
    assertEq(token.balanceOf(user), balanceBefore - 1e18); // tokens gone
    assertEq(token.balanceOf(address(pool)), 1e18); // tokens locked in pool
}
```
