Audit Report

## Title
Missing Staleness Check on Chainlink Price Feed Allows rsETH Minting at Stale Asset Prices - (File: contracts/oracles/ChainlinkPriceOracle.sol)

## Summary
`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards the `updatedAt` timestamp and `answeredInRound` return values, accepting any price regardless of age. A stale price propagates directly into `LRTDepositPool.getRsETHAmountToMint()`, allowing a depositor to receive more rsETH than the deposited asset is currently worth, diluting the accrued yield of all existing rsETH holders.

## Finding Description
`contracts/oracles/ChainlinkPriceOracle.sol` L52 reads only the raw price from `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

`updatedAt` and `answeredInRound` are never inspected. No maximum-age window (e.g., `block.timestamp - updatedAt > MAX_DELAY`) is enforced, and no `answeredInRound < roundId` guard is present.

The stale price propagates through two paths:

**Path 1 — rsETH minting:**
`depositAsset()` / `depositETH()` → `_beforeDeposit()` → `getRsETHAmountToMint()` → `lrtOracle.getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice()`. The mint amount is computed as:
```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```
A stale-high asset price inflates `rsethAmountToMint` without any on-chain check.

**Path 2 — rsETH price update:**
`updateRSETHPrice()` → `_updateRsETHPrice()` → `_getTotalEthInProtocol()` → `getAssetPrice(asset)` → `ChainlinkPriceOracle.getAssetPrice()`. The stored `rsETHPrice` is recalculated using the stale asset price, corrupting all subsequent minting until the next update.

By contrast, `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` — used for pool collateral — does implement partial guards (`answeredInRound < roundID`, `timestamp == 0`, `ethPrice <= 0`), confirming the protocol is aware of the need for such checks but omitted them from the core L1 LST oracle.

The `pricePercentageLimit` circuit-breaker in `_updateRsETHPrice()` only fires when `updateRSETHPrice()` is called; it does not protect the deposit path, which reads the stale price live from Chainlink and the previously stored `rsETHPrice`.

## Impact Explanation
When a Chainlink LST feed (stETH, ethX, sfrxETH, rETH) goes stale at a price higher than the current market rate, any depositor calling `depositAsset()` receives more rsETH than the deposited asset is worth. The excess minted supply dilutes the share of all existing rsETH holders, constituting theft of their accrued yield. This matches the allowed impact: **High — Theft of unclaimed yield**.

## Likelihood Explanation
Chainlink feeds have documented heartbeat intervals (1 hour for ETH/USD, up to 24 hours for some LST feeds). During network congestion a feed can miss its heartbeat; deprecated feeds freeze permanently at their last price. No special attacker capability is required — any depositor transacting during a staleness window triggers the vulnerability. The condition is externally observable (on-chain `updatedAt` is public), making targeted exploitation straightforward.

## Recommendation
Add a staleness guard in `ChainlinkPriceOracle.getAssetPrice()`, mirroring the pattern already used in `ChainlinkOracleForRSETHPoolCollateral`:

```solidity
uint256 public constant MAX_STALENESS = 1 hours; // tune per feed

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);
    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    if (answeredInRound < roundId) revert StalePrice();
    if (block.timestamp - updatedAt > MAX_STALENESS) revert StalePrice();
    if (price <= 0) revert InvalidPrice();

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

Per-asset staleness thresholds should be configurable via a mapping, since different LST feeds have different heartbeat intervals.

## Proof of Concept
1. Fork mainnet. Set `block.timestamp` to `T`.
2. Mock the stETH/ETH Chainlink feed so `latestRoundData()` returns `updatedAt = T - 2 hours` (heartbeat missed) and `price = 1.05e18` (stale high; real rate has dropped to `1.00e18` due to a slashing event).
3. Call `LRTDepositPool.depositAsset(stETH, 100e18, 0, "")` as an unprivileged attacker.
4. `getRsETHAmountToMint` calls `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` → returns stale `1.05e18`.
5. `rsethAmountToMint = (100e18 * 1.05e18) / rsETHPrice` — attacker receives ~5% more rsETH than the deposited stETH is currently worth.
6. Assert that the attacker's rsETH balance exceeds `100e18 * 1.00e18 / rsETHPrice` (the fair amount), confirming dilution of existing holders' yield.
