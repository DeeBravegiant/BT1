Audit Report

## Title
Decimal Precision Mismatch in Token-Overload `viewSwapRsETHAmountAndFee` Causes Near-Total Loss of Depositor Principal — (`contracts/pools/RSETHPool.sol`, `RSETHPoolNoWrapper.sol`, `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`)

## Summary

The token-overload of `viewSwapRsETHAmountAndFee(uint256 amount, address token)` computes `rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalising `amountAfterFee` from the token's native decimal precision to 18 decimals. For any supported token with fewer than 18 decimals (e.g., USDC at 6 decimals), the result is `10^(18 − d)` times smaller than the correct value. A depositor of 1 USDC receives ~475 wei rsETH instead of ~4.74 × 10¹⁴ wei rsETH, while their 0.997 USDC principal is permanently locked in the pool with no user-accessible recovery path.

## Finding Description

**Root cause — dimensional inconsistency in the rate formula.**

The ETH-only overload correctly scales its input:

```solidity
rsETHAmount = amountAfterFee * 1e18 / rsETHToETHrate;   // RSETHPool.sol L319
```

Here `amountAfterFee` is in wei (1e18 units per ETH), `rsETHToETHrate` is a 1e18-precision value, and the `* 1e18` restores the correct magnitude.

The token overload omits that normalisation:

```solidity
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;   // RSETHPool.sol L346
```

`tokenToETHRate` is returned by `ChainlinkOracleForRSETHPoolCollateral.getRate()`, which normalises the Chainlink answer to 1e18 precision — i.e., it expresses the price of **one whole token** in ETH, scaled to 1e18:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
```

For USDC (6 decimals, ~0.0005 ETH/USDC), `tokenToETHRate ≈ 5e14`. But `amountAfterFee` for a 1 USDC deposit is `997000` (in 1e6 units, not 1e18). The formula therefore computes:

```
997000 * 5e14 / 1.05e18 ≈ 475 wei rsETH
```

The correct value is:

```
(997000 / 1e6) * 5e14 / (1.05e18 / 1e18) ≈ 4.747e14 wei rsETH
```

The error factor is exactly `10^(18 − 6) = 10^12`.

**Exploit path (no privileged attacker required):**

1. `TIMELOCK_ROLE` calls `addSupportedToken(usdc, chainlinkOracle, bridge)` — a legitimate, expected protocol action; no decimal check is performed.
2. Any unprivileged user calls `deposit(usdc, 1_000_000, "ref")` (public function, `RSETHPool.sol` L284–305).
3. `viewSwapRsETHAmountAndFee(1_000_000, usdc)` returns `rsETHAmount ≈ 475`, `fee = 300`.
4. The pool transfers 0.997 USDC from the user and sends 475 wei rsETH back.
5. The user's 0.997 USDC is held by the pool; there is no user-callable refund or withdrawal function.

**Existing guards are insufficient.** `addSupportedToken` only checks `IOracle(oracle).getRate() != 0`; it performs no decimal validation. The `onlySupportedToken` modifier only checks that the oracle mapping is non-zero. Neither guard detects or prevents the precision mismatch.

The identical pattern is present in all five pool variants:
- `RSETHPool.sol` L346
- `RSETHPoolNoWrapper.sol` L311
- `RSETHPoolV3.sol` L334
- `RSETHPoolV3ExternalBridge.sol` L452
- `RSETHPoolV3WithNativeChainBridge.sol` L370

## Impact Explanation

**Critical — Direct theft of user funds.**

A depositor of any non-18-decimal token loses essentially their entire principal. The tokens are transferred into the pool (`safeTransferFrom` at L296/L403), the rsETH minted is negligible (~0 in practical terms), and no user-accessible mechanism exists to recover the deposited tokens. This is direct, permanent loss of user principal at-rest, matching the "Direct theft of any user funds, whether at-rest or in-motion" critical impact class.

## Likelihood Explanation

**Medium.** The sole precondition is that a non-18-decimal token (USDC, USDT, WBTC) is added via `addSupportedToken` by `TIMELOCK_ROLE`. This is a normal, foreseeable protocol operation — the multi-token deposit architecture exists precisely to support such tokens. No malicious actor, key compromise, or oracle manipulation is required. Once any such token is listed, every depositor of that token is affected automatically and repeatably.

## Recommendation

Normalise `amountAfterFee` to 18 decimals before the rate division in all five pool variants:

```solidity
import { IERC20Metadata } from "@openzeppelin/contracts/token/ERC20/extensions/IERC20Metadata.sol";

uint8 decimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * (10 ** (18 - decimals));
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Optionally, enforce `decimals <= 18` inside `addSupportedToken` to prevent tokens with more than 18 decimals from causing overflow.

## Proof of Concept

Deploy against a local fork with the following call sequence:

```solidity
// 1. Deploy MockOracle(5e14) for USDC/ETH rate, MockOracle(1.05e18) for rsETH/ETH rate
// 2. addSupportedToken(usdc, usdcOracle, bridge)  [TIMELOCK_ROLE]
// 3. Call viewSwapRsETHAmountAndFee(1_000_000, usdc)
//    Expected: rsETHAmount >= 4e14  (~0.000476 rsETH in wei)
//    Actual:   rsETHAmount ≈ 475    (1e12x too small)
// 4. Call deposit(usdc, 1_000_000, "ref") as an unprivileged user
//    Observe: user receives 475 wei rsETH; 997000 USDC locked in pool permanently

// Foundry assertion:
(uint256 rsETHAmount,) = pool.viewSwapRsETHAmountAndFee(1_000_000, usdc);
assertGe(rsETHAmount, 4e14, "precision loss: rsETHAmount 1e12x too small");
// This assertion FAILS on unmodified code, returning rsETHAmount ≈ 475
```

The arithmetic is deterministic and requires no fork state beyond deploying the pool with a 6-decimal token oracle.
