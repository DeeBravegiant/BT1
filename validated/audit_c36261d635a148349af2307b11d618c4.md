Audit Report

## Title
Untracked ETH via `receive()` Causes TVL Undercount and Temporary Fund Freeze — (File: contracts/LRTConverter.sol)

## Summary
`LRTConverter` accepts ETH via `receive() external payable {}` without updating `ethValueInWithdrawal`. Because `LRTDepositPool.getETHDistributionData()` reads only `ethValueInWithdrawal` (not `address(lrtConverter).balance`) to account for ETH held in the converter, any ETH arriving through the fallback is invisible to TVL and rsETH price calculations. Additionally, the only exit paths for ETH in the converter are `claimStEth` and `claimSwEth`, both of which require a live Lido/Swell withdrawal request; absent one, the ETH is temporarily frozen. A secondary defect causes `_sendEthToDepositPool` to zero out `ethValueInWithdrawal` prematurely when extra ETH is present, corrupting accounting for remaining pending withdrawal requests.

## Finding Description
**Root cause — untracked ETH in `receive()`:**

`LRTConverter` declares a bare fallback at [1](#0-0)  that accepts ETH from any sender. `ethValueInWithdrawal` is only mutated in `transferAssetFromDepositPool` (line 140) and `transferAssetToDepositPool` (line 163); ETH arriving via `receive()` never increments it. [2](#0-1) 

**TVL read path:**

`getETHDistributionData()` reads `ILRTConverter(lrtConverter).ethValueInWithdrawal()` to represent the converter's contribution to TVL. It never reads `address(lrtConverter).balance`. [3](#0-2) 

**No exit path without an active withdrawal request:**

The only internal function that moves ETH out of the converter is `_sendEthToDepositPool`, called exclusively from `claimStEth` and `claimSwEth`. [4](#0-3)  Both require a valid, claimable Lido/Swell withdrawal NFT. There is no `sweep()`, `recoverETH()`, or equivalent. If no such request exists when ETH arrives via `receive()`, the ETH has no exit path until the operator manually initiates a new stETH unstake cycle (multi-day unbonding period).

**Secondary accounting defect — premature zeroing of `ethValueInWithdrawal`:**

`claimStEth` passes `address(this).balance` (entire balance, including any extra ETH) to `_sendEthToDepositPool`. Inside that function: [5](#0-4) 

If `address(this).balance > ethValueInWithdrawal` (because extra ETH is present), `ethValueInWithdrawal` is set to `0` even though other Lido withdrawal requests are still pending. Those remaining requests become invisible to TVL until their own `claimStEth` calls are made.

**Exploit flow:**
1. Protocol has 100 stETH in a pending Lido withdrawal; `ethValueInWithdrawal = 100e18`.
2. Attacker calls `address(lrtConverter).transfer(10 ether)` (or any ETH send). `ethValueInWithdrawal` stays `100e18`; `address(lrtConverter).balance = 10 ETH`. TVL is understated by 10 ETH; rsETH price drops.
3. Attacker deposits ETH into `LRTDepositPool.depositETH()` at the artificially lower rsETH price, receiving more rsETH than fair value.
4. Operator calls `claimStEth(requestId, hint)`. `address(this).balance = 110 ETH`. `_sendEthToDepositPool(110e18)` is called. Since `100e18 < 110e18`, `ethValueInWithdrawal` is set to `0` prematurely.
5. Any remaining pending Lido requests (e.g., another 50 ETH) are now invisible to TVL. rsETH price is further understated.
6. Attacker withdraws rsETH at the restored (or higher) price, profiting from the dilution of existing holders.

## Impact Explanation
**Temporary freezing of funds (Medium):** ETH arriving via `receive()` when no Lido/Swell withdrawal request is outstanding has no exit path. The operator must transfer stETH into the converter, initiate a new Lido withdrawal, and wait through the unbonding period (typically 1–5 days) before the ETH can be recovered. This constitutes a temporary freeze of funds matching the allowed Medium impact class.

**TVL undercount / share mis-accounting (Low):** While extra ETH is untracked, `rsETHPrice()` is lower than true value. Depositors during this window receive more rsETH than they are entitled to, diluting existing holders. The secondary defect (premature zeroing of `ethValueInWithdrawal`) compounds this by also hiding legitimately pending withdrawal ETH from TVL.

## Likelihood Explanation
ETH can reach `LRTConverter.receive()` via: (a) accidental direct transfer by any user or contract, (b) `selfdestruct` proceeds from another contract targeting this address, or (c) protocol-level ETH distributions (e.g., EigenLayer/Lido airdrops). The protocol already handles large ETH flows through this contract, making incidental ETH arrival plausible. A deliberate attacker can trigger the TVL undercount at will by sending any nonzero ETH amount. The temporary freeze scenario (no active withdrawal request) is less common but possible during quiet periods or after all requests have been claimed.

## Recommendation
1. In `_sendEthToDepositPool`, track the amount actually claimed (passed from `_claimStEth`/`_claimSwEth`) separately from any pre-existing balance, and decrement `ethValueInWithdrawal` only by the claimed amount — not by `address(this).balance`.
2. Add a permissioned `sweepETH(address recipient)` function (callable by admin/manager) that forwards `address(this).balance - ethValueInWithdrawal` (i.e., only the untracked surplus) to the deposit pool or treasury.
3. Alternatively, in `getETHDistributionData()`, read `address(lrtConverter).balance` instead of (or in addition to) `ethValueInWithdrawal` to ensure all ETH in the converter is always reflected in TVL.

## Proof of Concept
**Minimal call sequence (local fork):**

```solidity
// Setup: protocol has 100 stETH pending in Lido withdrawal
// ethValueInWithdrawal == 100e18, address(lrtConverter).balance == 0

// Step 1: Attacker sends 10 ETH directly to LRTConverter
(bool ok,) = address(lrtConverter).call{value: 10 ether}("");
// ethValueInWithdrawal still == 100e18
// address(lrtConverter).balance == 10 ETH
// getETHDistributionData().ethLyingInConverter == 100e18 (10 ETH invisible)

// Step 2: Attacker deposits at artificially low rsETH price
lrtDepositPool.depositETH{value: 50 ether}(0, "");
// Attacker receives more rsETH than fair value

// Step 3: Operator claims the Lido request (100 ETH arrives)
// address(this).balance = 110 ETH
lrtConverter.claimStEth(requestId, hint);
// _sendEthToDepositPool(110e18) called
// ethValueInWithdrawal = 0 (premature, if other requests pending)

// Step 4: Attacker withdraws rsETH at restored price, profiting from dilution
```

**Foundry invariant test plan:** Assert that `ILRTConverter(lrtConverter).ethValueInWithdrawal() <= address(lrtConverter).balance` always holds, and that after any `claimStEth`/`claimSwEth` call, `ethValueInWithdrawal` equals the pre-call value minus only the ETH received from Lido/Swell (not the full balance). Fuzz with arbitrary ETH sent to `receive()` between claim operations.
