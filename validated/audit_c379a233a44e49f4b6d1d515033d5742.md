Audit Report

## Title
Missing `msg.sender == account` Guard in `MerkleDistributor.claim` Enables Force-Claiming to Token-Incompatible Contracts, Permanently Freezing Unclaimed Yield - (File: contracts/utils/MerkleDistributor/MerkleDistributor.sol)

## Summary
`MerkleDistributor.claim` accepts an arbitrary `account` parameter and transfers tokens to it without verifying `msg.sender == account`. Any caller with a valid merkle proof for a target address can trigger that address's claim. If the target is a smart contract unable to handle ERC20 tokens, the distributed yield is permanently locked. The sibling contract `KernelMerkleDistributor` correctly enforces the caller restriction; `MerkleDistributor` does not.

## Finding Description
In `MerkleDistributor.claim` (lines 97–147), after merkle proof verification, the contract unconditionally transfers tokens to the supplied `account`:

```solidity
IERC20(token).safeTransfer(account, amountToSend);   // line 141
IERC20(token).safeTransfer(protocolTreasury, fee);   // line 144
```

There is no check that `msg.sender == account`. The node is constructed as:

```solidity
bytes32 node = keccak256(abi.encodePacked(index, account, cumulativeAmount)); // line 120
```

Merkle proofs for all accounts are published off-chain (standard practice for merkle distributors), so any attacker can obtain a valid proof for any address in the tree.

By contrast, `KernelMerkleDistributor._processClaim` (lines 311–313) enforces:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

`MerkleDistributor` omits this guard entirely.

**Exploit path:**
1. A smart contract address (e.g., a Uniswap pool, lending protocol, or any DeFi contract holding rsETH) is included in the merkle tree as a yield recipient.
2. The attacker retrieves the published merkle proof for that contract address.
3. The attacker calls `MerkleDistributor.claim(index, victimContract, cumulativeAmount, merkleProof)`.
4. `safeTransfer` sends the tokens to `victimContract`. Since ERC20 `safeTransfer` does not require the recipient to implement any callback, the transfer succeeds even if `victimContract` has no mechanism to move the tokens.
5. The yield is permanently locked in `victimContract`; `userClaims[victimContract]` is updated, so the claim cannot be re-triggered.

The existing `AlreadyClaimed` check (line 115) and `whenNotPaused` modifier do not mitigate this — they only prevent double-claiming and paused-state execution, respectively.

## Impact Explanation
**Medium — Permanent freezing of unclaimed yield.**

The distributed tokens are yield/rewards (rsETH or equivalent) allocated via the merkle distributor. Once force-claimed to a contract that cannot move ERC20 tokens, the yield is irrecoverably locked. The claim state is marked as consumed, so the rightful recipient can never re-claim. This matches the allowed impact: *"Medium. Permanent freezing of unclaimed yield."*

## Likelihood Explanation
Any unprivileged external caller can execute this attack. Merkle proofs are public by design. The only precondition is that at least one smart contract address appears in the merkle tree — a realistic condition given that DeFi protocols, liquidity pools, and vaults routinely hold rsETH and can accumulate yield entitlements. The attack is repeatable for every such address in every merkle root update, requires no special privileges, and costs only gas.

## Recommendation
Add a caller restriction in `MerkleDistributor.claim`, mirroring `KernelMerkleDistributor._processClaim`:

```solidity
if (account != msg.sender) {
    revert Unauthorized();
}
```

Insert this check before the merkle proof verification (after the `index` and `isClaimed` checks) at approximately line 114 of `contracts/utils/MerkleDistributor/MerkleDistributor.sol`.
