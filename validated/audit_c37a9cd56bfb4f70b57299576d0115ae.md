### Title
Proposer-Controlled Timestamp Used as Reference Anchor for L1 Gas Price Validation Allows Selective Historical Price Injection — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

In `is_proposal_init_valid`, the validator computes its own reference L1 gas prices by querying the price provider with the **proposer-supplied** `init_proposed.timestamp`, rather than the validator's own current wall-clock time. Because the timestamp window has no lower bound relative to `now` (only a lower bound of the previous block's timestamp), a proposer can freely choose any timestamp in the range `[last_block_timestamp, now + block_timestamp_window_seconds]` and thereby select which historical L1 gas price the validator uses as its reference. The validator then accepts the proposer's claimed prices as long as they are within margin of that self-selected reference — defeating the purpose of the price validation entirely.

---

### Finding Description

**Root cause — `is_proposal_init_valid` in `validate_proposal.rs`:**

The timestamp bounds check is:

```rust
// lower bound: only monotonicity relative to previous block
if init_proposed.timestamp < last_block_timestamp { reject }
// upper bound: not more than 1 second in the future
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds { reject }
``` [1](#0-0) 

There is **no lower bound relative to `now`**. The lower bound is only the previous block's timestamp (`last_block_timestamp`), which can be arbitrarily far in the past.

Immediately after this check, the validator computes its reference L1 gas prices using the proposer-supplied timestamp:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,   // ← proposer-controlled input
    proposal_init_validation.previous_proposal_init.as_ref(),
    gas_price_params,
).await;
``` [2](#0-1) 

The validator then checks that the proposer's claimed prices are within margin of this reference: [3](#0-2) 

The code's own comment on `within_margin` explicitly warns against anchoring the band to a proposer-supplied value:

> *"The band is anchored to `reference` (the node's own L1 oracle read), not to the proposer-supplied `proposed`: anchoring to `proposed` would let a malicious proposer scale the band width with its own input and widen it in its favor."* [4](#0-3) 

Yet the reference itself is computed using the proposer's timestamp, so the proposer can still manipulate the reference by choosing a timestamp that corresponds to a favorable historical price period.

**How `get_l1_prices_in_fri_and_wei` uses the timestamp:**

The function passes the timestamp directly to `get_price_info(BlockTimestamp(timestamp))` and `get_rate(timestamp)`: [5](#0-4) 

Inside `L1GasPriceProvider::get_price_info`, the timestamp selects which historical L1 blocks are averaged. A lower timestamp selects older (potentially cheaper) blocks: [6](#0-5) 

The only staleness guard (`max_time_gap_seconds`) rejects timestamps that are **too far in the future** relative to the latest sample — it does not prevent choosing timestamps in the past: [7](#0-6) 

---

### Impact Explanation

A malicious proposer can set `init_proposed.timestamp` to any value in `[last_block_timestamp, now + 1]`. If the previous block was produced minutes or hours ago (e.g., during network slowdowns, high consensus round counts, or after a node restart), this window is wide. The proposer scans the L1 gas price history, identifies a period with lower prices, sets the block timestamp to that period, and sets the `l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, and `l1_data_gas_price_wei` fields in `ProposalInit` to match. The validator computes the same reference prices for that timestamp and the `within_margin` check passes.

The result is a committed block with artificially low L1 gas prices. All transactions in that block are charged lower fees than the current market rate. This is a direct economic impact matching: **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."**

The corrupted values are `l1_gas_price_fri`, `l1_data_gas_price_fri`, `l1_gas_price_wei`, and `l1_data_gas_price_wei` in the committed `BlockInfo`, which flow into `convert_to_sn_api_block_info` and are used for all fee calculations in that block: [8](#0-7) 

---

### Likelihood Explanation

The attack requires the proposer role, which is held by a validator node. In a BFT consensus system, any staked validator can become a proposer. The attack window (`[last_block_timestamp, now + 1]`) is narrow during normal fast block production but widens significantly during:
- Network slowdowns or partitions
- High consensus round counts (each round adds latency)
- Node restarts or sync delays

The L1 gas price history is retained in a ring buffer, so historical prices are available for lookup as long as they are within the buffer's storage limit.

---

### Recommendation

The validator should compute its reference L1 gas prices using its own current wall-clock time (`now`), not the proposer-supplied `init_proposed.timestamp`. The proposer's timestamp should only be used for block metadata (block header timestamp), not as the key for price oracle lookups during validation.

```rust
// Use validator's own clock for the reference price lookup
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    now,   // validator's own current time, not init_proposed.timestamp
    proposal_init_validation.previous_proposal_init.as_ref(),
    gas_price_params,
).await;
```

Additionally, a lower-bound freshness check should be added to the timestamp validation, e.g.:

```rust
if now > init_proposed.timestamp + max_allowed_past_seconds {
    return Err(...); // timestamp is too far in the past
}
```

This mirrors the existing upper-bound check (`block_timestamp_window_seconds`) and closes the window for historical price selection.

### Proof of Concept

**Setup:** Previous block produced at `T_prev = now - 600` (10 minutes ago, e.g., due to a slow consensus round). L1 gas prices at `T_prev + 60` were 50% lower than current prices.

**Attack steps:**

1. Proposer observes `last_block_timestamp = T_prev` and `now = T_prev + 600`.
2. Proposer queries its own L1 gas price provider for `T_low = T_prev + 60` and finds lower prices `P_low`.
3. Proposer constructs `ProposalInit` with:
   - `timestamp = T_low` (valid: `T_low >= T_prev` and `T_low <= now + 1`)
   - `l1_gas_price_fri = P_low.fri`, `l1_data_gas_price_fri = P_low.data_fri`, etc.
4. Proposer broadcasts this `ProposalInit` to validators.
5. Each validator calls `is_proposal_init_valid`:
   - Timestamp check passes: `T_low >= T_prev` ✓ and `T_low <= now + 1` ✓
   - Validator calls `get_l1_prices_in_fri_and_wei(provider, T_low, ...)` → returns `P_low`
   - `within_margin(P_low, P_low, margin)` → passes ✓
6. Block is committed with `l1_gas_price = P_low` (50% below current market rate).
7. All transactions in the block are charged fees based on `P_low`, not the current market price. [1](#0-0) [2](#0-1) [5](#0-4)
