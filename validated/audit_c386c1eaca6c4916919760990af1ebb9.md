Audit Report

## Title
ETH Donation via `receive()` Inflates `address(this).balance`, Blocking All New ETH Deposits via Deposit Limit DOS - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool` exposes an unrestricted `receive()` function that allows any address to donate ETH directly to the contract. Because `getETHDistributionData()` reads `address(this).balance` as the canonical ETH balance, donated ETH inflates `getTotalAssetDeposits(ETH_TOKEN)`. Once the inflated total exceeds `depositLimitByAsset(ETH_TOKEN)`, every subsequent `depositETH()` call reverts with `MaximumDepositLimitReached`, blocking all new ETH deposits until an admin raises the cap.

## Finding Description
`LRTDepositPool` accepts ETH from any sender via its open receiver:

```solidity
// contracts/LRTDepositPool.sol L58
receive() external payable { }
```

`getETHDistributionData()` uses the raw contract balance as the pool's ETH accounting:

```solidity
// contracts/LRTDepositPool.sol L480
ethLyingInDepositPool = address(this).balance;
```

This value flows into `getTotalAssetDeposits(ETH_TOKEN)` via `getAssetDistributionData` → `getETHDistributionData`. The deposit limit check for ETH in `_checkIfDepositAmountExceedesCurrentLimit` is:

```solidity
// contracts/LRTDepositPool.sol L678-679
if (asset == LRTConstants.ETH_TOKEN) {
    return (totalAssetDeposits > lrtConfig.depositLimitByAsset(asset));
}
```

Unlike the ERC-20 branch at L681 (`totalAssetDeposits + amount > limit`), the ETH branch does not add the incoming deposit amount. This means the check triggers solely on whether the current total already exceeds the limit — regardless of the size of the incoming deposit. Once an attacker donates enough ETH to push `totalAssetDeposits` above the limit, `_beforeDeposit` reverts unconditionally:

```solidity
// contracts/LRTDepositPool.sol L661-663
if (_checkIfDepositAmountExceedesCurrentLimit(asset, depositAmount)) {
    revert MaximumDepositLimitReached();
}
```

Donated ETH cannot be purged from accounting: transferring it to a NodeDelegator keeps it in `ethLyingInNDCs` (L485); transferring it to `LRTUnstakingVault` keeps it in `ethLyingInUnstakingVault` (L496). Both are summed in `getTotalAssetDeposits`. The only remediation is an admin raising `depositLimitByAsset(ETH_TOKEN)` in `LRTConfig`.

## Impact Explanation
**Low — Contract fails to deliver promised returns, but does not lose value.**

All new ETH deposits via `depositETH()` are blocked for as long as the donated ETH keeps `getTotalAssetDeposits(ETH_TOKEN)` above the configured limit. Existing depositor funds are not frozen and withdrawals are unaffected. No ETH is stolen. The attacker permanently sacrifices the donated ETH into the protocol.

## Likelihood Explanation
**Medium.** The attack requires no special role, no front-running, and no external dependency. Any EOA or contract can call `LRTDepositPool.receive()` with a small ETH amount. The attack is practical whenever `getTotalAssetDeposits(ETH_TOKEN)` is within a small margin of `depositLimitByAsset(ETH_TOKEN)` — a common operational state as the protocol approaches its cap. The attacker's only cost is the donated ETH, which is permanently absorbed by the protocol.

## Recommendation
Replace the raw `address(this).balance` read in `getETHDistributionData()` with an internal accounting variable (e.g., `totalETHDeposited`) that is incremented only on legitimate `depositETH()` calls and decremented on authorized transfers out (`transferETHToNodeDelegator`, `transferETHToLRTUnstakingVault`, `swapAssetForETHWithinDepositPool`, `stakeEthForStETH`). Additionally, align the ETH limit check with the ERC-20 branch by checking `totalAssetDeposits + depositAmount > limit` rather than `totalAssetDeposits > limit`.

## Proof of Concept
1. Read `getTotalAssetDeposits(LRTConstants.ETH_TOKEN)` → returns `D`.
2. Read `lrtConfig.depositLimitByAsset(ETH_TOKEN)` → returns `L`, where `D < L`.
3. Attacker sends `L - D + 1 wei` directly to `LRTDepositPool` (no function call; `receive()` accepts it).
4. `address(this).balance` increases by `L - D + 1 wei`.
5. `getETHDistributionData()` now returns `ethLyingInDepositPool = L + 1`.
6. `getTotalAssetDeposits(ETH_TOKEN)` returns `L + 1`.
7. `_checkIfDepositAmountExceedesCurrentLimit(ETH_TOKEN, any_amount)` returns `true` (`L + 1 > L`).
8. Every call to `depositETH(...)` reverts with `MaximumDepositLimitReached`.
9. Transferring the donated ETH to any NodeDelegator or `LRTUnstakingVault` does not remove it from `getTotalAssetDeposits`; the block persists until admin raises the deposit cap.

**Foundry test sketch:**
```solidity
function test_donationBlocksDeposits() public {
    uint256 limit = lrtConfig.depositLimitByAsset(LRTConstants.ETH_TOKEN);
    uint256 current = depositPool.getTotalAssetDeposits(LRTConstants.ETH_TOKEN);
    uint256 donationAmount = limit - current + 1;

    vm.deal(attacker, donationAmount);
    vm.prank(attacker);
    (bool ok,) = address(depositPool).call{value: donationAmount}("");
    assertTrue(ok);

    vm.deal(user, 1 ether);
    vm.prank(user);
    vm.expectRevert(ILRTDepositPool.MaximumDepositLimitReached.selector);
    depositPool.depositETH{value: 1 ether}(0, "");
}
```
