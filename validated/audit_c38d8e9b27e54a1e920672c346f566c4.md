### Title
Keyshare Encryption Key (`MPC_SECRET_STORE_KEY`) Exposed via Environment Variable, Undermining TEE Keyshare-at-Rest Protection — (File: `deployment/start.sh`)

---

### Summary

The production deployment script `deployment/start.sh` provisions the AES-128 keyshare encryption key (`MPC_SECRET_STORE_KEY`) and the node's P2P/signing private keys (`MPC_P2P_PRIVATE_KEY`, `MPC_ACCOUNT_SK`) as plain OS environment variables and explicitly `export`s them before launching the MPC node process. The project's own threat model explicitly classifies the host OS as **UNTRUSTED** and lists "Read/modify env vars" as a host capability. This means the host OS — or any crash-report/telemetry service running on it — can read `MPC_SECRET_STORE_KEY` and, combined with access to the encrypted keyshare files on the same host, decrypt the node's threshold key share, directly undermining the TEE's keyshare-at-rest protection.

---

### Finding Description

In `deployment/start.sh`, three categories of sensitive key material are accepted and re-exported as environment variables:

**1. `MPC_SECRET_STORE_KEY` — keyshare AES encryption key** [1](#0-0) 

This key is the AES-128 secret used by `SecretsConfig` to encrypt every keyshare stored on disk: [2](#0-1) [3](#0-2) 

**2. `MPC_P2P_PRIVATE_KEY` and `MPC_ACCOUNT_SK` — node identity keys** [4](#0-3) [5](#0-4) 

These are the Ed25519 private keys for P2P mutual-TLS authentication and NEAR transaction signing, stored in `PersistentSecrets`: [6](#0-5) 

**3. The threat model explicitly acknowledges the host OS can read env vars** [7](#0-6) 

The key protection summary states keyshares are protected by "TDX CVM memory encryption" at runtime and "Encrypted disk (Gramine sealed)" at rest: [8](#0-7) 

However, the Gramine-sealed disk encryption key is derived from TDX measurements — it is never passed as an env var. `MPC_SECRET_STORE_KEY` is passed as an env var, which the host OS can read before it enters the CVM, breaking the at-rest protection chain.

**4. `SecretsStartConfig` has a custom `Debug` redaction, but `PersistentSecrets` does not**

`SecretsStartConfig` (which holds `secret_store_key_hex`) correctly implements a custom `Debug` that redacts both fields: [9](#0-8) 

But `PersistentSecrets` (which holds the actual `SigningKey` objects) uses `#[derive(Debug)]` with no redaction: [6](#0-5) 

If this struct is ever included in a panic message, error chain, or structured log event, the key material is exposed to the logging pipeline.

---

### Impact Explanation

- **`MPC_SECRET_STORE_KEY` leakage**: The host OS (or a crash-report/telemetry agent running on it) can read this env var. Combined with access to the encrypted keyshare files on the same host (also accessible to the host OS), the attacker can decrypt the node's threshold key share. A single share does not reconstruct the master key, but if the same infrastructure provider hosts multiple MPC nodes (a realistic cloud deployment), threshold-many shares can be collected, enabling full master key reconstruction and unauthorized threshold signature issuance.

- **`MPC_P2P_PRIVATE_KEY` leakage**: An attacker who obtains this key can impersonate the node in the P2P mTLS mesh, potentially injecting or replaying protocol messages in triple/presignature/signing rounds.

- **`MPC_ACCOUNT_SK` leakage**: Allows calling any MPC contract method (the key is granted access to all methods with no function-name restriction) as the compromised node.

The primary impact maps to: **Critical — unauthorized access to MPC key shares and secret material that materially enables forgery or secret recovery** (threshold-many nodes on shared infrastructure) and **High — participant/attestation authorization bypass** (P2P key impersonation).

---

### Likelihood Explanation

- The host OS is explicitly modeled as untrusted and capable of reading env vars.
- Cloud-managed container environments (GCP Cloud Run, Kubernetes) routinely expose env vars to the platform operator, container inspection APIs, and any crash-reporting sidecar.
- The comment at line 204 of `deployment/start.sh` confirms this path is used in production upgrades ("for 2.2.0 -> 3.0.0 upgrade"), not just localnet.
- The `export` statements at lines 152, 163, and 211 make the keys available to every child process spawned after that point, widening the exposure surface. [10](#0-9) 

---

### Recommendation

**Short term**: Remove `MPC_SECRET_STORE_KEY`, `MPC_P2P_PRIVATE_KEY`, and `MPC_ACCOUNT_SK` from the environment variable surface. Instead, provision `MPC_SECRET_STORE_KEY` using the same Gramine-sealed key derivation mechanism already used for the disk encryption key (derived from TDX RTMR measurements inside the CVM), so the host OS never sees it. For the P2P and account keys, the TEE design doc already describes generating them inside the CVM and never exporting them: [11](#0-10) 

**Long term**: Adopt the same pattern used for the Gramine disk encryption key for all secret material: derive or seal secrets inside the CVM using TDX measurements, eliminating any env-var provisioning path for key material. For signing operations, consider a dedicated in-CVM signing service so key material never transits the host boundary at all.

---

### Proof of Concept

1. Deploy an MPC node using `deployment/start.sh` with `MPC_SECRET_STORE_KEY` set in the environment.
2. From the host OS (or a container inspection API), read the process environment:
   ```bash
   cat /proc/$(pgrep mpc-node)/environ | tr '\0' '\n' | grep MPC_SECRET_STORE_KEY
   ```
3. Locate the encrypted keyshare file in `$MPC_HOME_DIR` (e.g., `keyshare.enc`).
4. Use the recovered `MPC_SECRET_STORE_KEY` hex value to AES-decrypt the keyshare file, recovering the node's threshold key share.
5. Repeat for threshold-many nodes on the same infrastructure to reconstruct the master signing key.

The env var is also visible in crash dumps, `docker inspect`, Kubernetes secret-less pod specs, and any telemetry agent that collects process environment — matching the exact exploit scenario described in the external report. [12](#0-11)
