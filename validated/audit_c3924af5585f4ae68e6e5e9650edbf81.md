Audit Report

## Title
Missing Staleness Validation on Chainlink `latestRoundData()` Allows Stale LST Prices to Over-Mint rsETH — (File: `contracts/oracles/ChainlinkPriceOracle.sol`)

## Summary

`ChainlinkPriceOracle.getAssetPrice()` calls `latestRoundData()` but silently discards `updatedAt`, `answeredInRound`, and `roundId`, retaining only the raw `price`. A stale Chainlink feed for any supported LST will return an outdated (potentially inflated) price that drives `getRsETHAmountToMint`, causing an attacker to receive more rsETH than the deposited collateral is worth and diluting all existing rsETH holders.

## Finding Description

`ChainlinkPriceOracle.getAssetPrice()` (L49–55) binds only the second return value of `latestRoundData()`:

```solidity
(, int256 price,,,) = priceFeed.latestRoundData();
return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
```

No check is performed on:
- `answeredInRound >= roundId` — round-completeness
- `updatedAt > 0` — non-zero timestamp (incomplete round guard)
- `block.timestamp - updatedAt <= maxStaleness` — time-based freshness

This price flows directly into `LRTOracle.getAssetPrice()` (L156–158), which is consumed by `LRTDepositPool.getRsETHAmountToMint()` (L519–520):

```solidity
rsethAmountToMint = (amount * lrtOracle.getAssetPrice(asset)) / lrtOracle.rsETHPrice();
```

This calculation is invoked by the public, permissionless `depositAsset()` (L99–118) and `depositETH()` (L76–93). No existing check in `_beforeDeposit` (L648–670) validates oracle freshness — only deposit amount bounds and a depositor-side `minRSETHAmountExpected` slippage guard are present, neither of which protects existing holders.

The same codebase's `ChainlinkOracleForRSETHPoolCollateral.getRate()` (L27–32) demonstrates the team is aware of the requirement:

```solidity
if (answeredInRound < roundID) revert StalePrice();
if (timestamp == 0) revert IncompleteRound();
if (ethPrice <= 0) revert InvalidPrice();
```

These guards are entirely absent from the mainnet `ChainlinkPriceOracle`.

## Impact Explanation

When a Chainlink LST/ETH feed is stale and the last recorded price is higher than the current market price, `getRsETHAmountToMint` returns a value larger than the deposited collateral justifies. The attacker receives rsETH backed by insufficient collateral. Because rsETH is a share token whose price is derived from total collateral value divided by total supply, every existing rsETH holder's share is diluted — their holdings represent less ETH value than before the attack. This constitutes **direct theft of user funds (Critical)**: value is transferred from existing rsETH holders to the attacker without their consent and without any privileged access.

## Likelihood Explanation

Chainlink feeds have documented heartbeat intervals (e.g., 1 hour for stETH/ETH on mainnet). Staleness occurs during Ethereum network congestion, Chainlink node outages, or extreme market volatility — precisely the conditions under which LST prices move most sharply. An attacker only needs to monitor the on-chain `updatedAt` field of the relevant Chainlink aggregator. No privileged role, governance action, or external protocol compromise is required. `depositAsset` is callable by any EOA or contract. The attack is repeatable for as long as the feed remains stale.

## Recommendation

Add staleness validation inside `ChainlinkPriceOracle.getAssetPrice()`:

```solidity
uint256 public maxStaleness; // set per-asset or globally by LRTManager

function getAssetPrice(address asset) external view onlySupportedAsset(asset) returns (uint256) {
    AggregatorV3Interface priceFeed = AggregatorV3Interface(assetPriceFeed[asset]);

    (uint80 roundId, int256 price,, uint256 updatedAt, uint80 answeredInRound) =
        priceFeed.latestRoundData();

    require(answeredInRound >= roundId, "Stale round");
    require(updatedAt > 0, "Incomplete round");
    require(block.timestamp - updatedAt <= maxStaleness, "Stale price");
    require(price > 0, "Invalid price");

    return uint256(price) * 1e18 / 10 ** uint256(priceFeed.decimals());
}
```

`maxStaleness` should be configurable by the LRT manager role and set per-asset to match each feed's documented heartbeat (e.g., 3600 seconds for stETH/ETH).

## Proof of Concept

1. The Chainlink stETH/ETH feed on mainnet has a 1-hour heartbeat. Suppose the feed has not updated for 90 minutes (stale) and the last recorded price is `1.00 ETH` per stETH.
2. During this window, stETH depegs to `0.90 ETH` on the open market.
3. Attacker calls `LRTDepositPool.depositAsset(stETH, 1000e18, 0, "")`.
4. `_beforeDeposit` → `getRsETHAmountToMint` → `lrtOracle.getAssetPrice(stETH)` → `ChainlinkPriceOracle.getAssetPrice(stETH)` returns `1.00e18` (stale).
5. `rsethAmountToMint = (1000e18 * 1.00e18) / rsETHPrice` — ~11% more rsETH than the deposited stETH is currently worth.
6. Attacker redeems or sells the excess rsETH, extracting value from existing holders.
7. No admin key, no governance action, no oracle operator compromise required — only monitoring the `updatedAt` field of the Chainlink aggregator.
