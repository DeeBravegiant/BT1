Audit Report

## Title
First Depositor Price Inflation via Donation Freezes Subsequent Depositors' Funds - (File: contracts/LRTOracle.sol / contracts/LRTDepositPool.sol)

## Summary
`LRTOracle` stores `rsETHPrice` as a cached value updatable by anyone via the public `updateRSETHPrice()`. Because TVL accounting uses raw `address(this).balance` / `IERC20.balanceOf`, unsolicited donations inflate the computed price. With `pricePercentageLimit` defaulting to `0` (disabling the price-increase guard) and `_beforeDeposit` lacking a `rsethAmountToMint > 0` check, a first depositor can donate assets to commit an arbitrarily inflated price, causing all subsequent depositors who pass `minRSETHAmountExpected = 0` to receive 0 rsETH while permanently losing their deposited assets.

## Finding Description

**Root cause 1 — Public, unrestricted price update.**
`updateRSETHPrice()` carries only a `whenNotPaused` modifier; any EOA can call it at any time. [1](#0-0) 

**Root cause 2 — TVL uses raw balances, so donations inflate it.**
For LST assets, `assetLyingInDepositPool` is `IERC20(asset).balanceOf(address(this))`. [2](#0-1) 
For ETH, `ethLyingInDepositPool` is `address(this).balance`, and the contract accepts arbitrary ETH via an open `receive()`. [3](#0-2) [4](#0-3) 

**Root cause 3 — Price-increase guard is disabled by default.**
`pricePercentageLimit` is a `uint256` storage variable never set in `initialize()`, so it defaults to `0`. The guard condition short-circuits on `pricePercentageLimit > 0`, meaning any arbitrarily large price increase is accepted. [5](#0-4) 

**Root cause 4 — `_beforeDeposit` does not enforce `rsethAmountToMint > 0`.**
The only slippage check is `rsethAmountToMint < minRSETHAmountExpected`. When a caller passes `minRSETHAmountExpected = 0` and the inflated price causes `getRsETHAmountToMint` to return `0`, the condition `0 < 0` is false and execution continues to `_mintRsETH(0)`. [6](#0-5) [7](#0-6) 

**Root cause 5 — `protocolFeeInBPS` also defaults to `0`.**
`LRTConfig.initialize()` never sets `protocolFeeInBPS`, so it is `0` by default. This means `protocolFeeInETH = 0` inside `_updateRsETHPrice`, and the `_checkAndUpdateDailyFeeMintLimit(0)` call does not revert, allowing the inflated price to be committed without obstruction. [8](#0-7) [9](#0-8) 

**Exploit flow:**
1. Attacker calls `updateRSETHPrice()` with zero supply → `rsETHPrice = 1 ether`, `highestRsethPrice = 1 ether`.
2. Attacker calls `depositETH{value: 1 wei}(0, "")` → mints 1 wei rsETH. `rsETHPrice` is still `1 ether`.
3. Attacker sends 10 000 ETH directly to `LRTDepositPool` (accepted by `receive()`).
4. Attacker calls `updateRSETHPrice()`: `totalETHInProtocol ≈ 10_000e18 + 1`, `rsethSupply = 1`, `newRsETHPrice ≈ 10_001e18`. `pricePercentageLimit == 0` → guard skipped → `rsETHPrice` committed as `≈ 10_001e18`.
5. Victim calls `depositETH{value: 10_000e18}(0, "")`: `getRsETHAmountToMint` → `(10_000e18 * 1e18) / 10_001e18 = 0`. `_beforeDeposit`: `0 < 0` is false → no revert. `_mintRsETH(0)` → victim receives 0 rsETH; 10 000 ETH is permanently lost.
6. Attacker calls `updateRSETHPrice()` again; their 1 wei rsETH now backs ~20 001 ETH of TVL and can be redeemed via withdrawal for the full pool.

## Impact Explanation
This is **Critical — Direct theft of user funds and permanent freezing of deposited funds**. A victim who deposits and receives 0 rsETH holds no token to redeem and has no refund path. The attacker recovers the full pool via their 1 wei rsETH position. The impact matches the allowed scope: "Direct theft of any user funds" and "Permanent freezing of funds." [10](#0-9) 

## Likelihood Explanation
All preconditions are default storage values requiring no admin action: `pricePercentageLimit == 0`, `protocolFeeInBPS == 0`. `updateRSETHPrice()` is callable by any EOA. The `receive()` fallback accepts arbitrary ETH. Any user who calls `depositETH(0, "")` or `depositAsset(asset, amount, 0, "")` — a natural pattern when no slippage protection is desired — is vulnerable. The attack is executable in a single block sequence and is repeatable. [11](#0-10) [12](#0-11) 

## Recommendation
1. **Enforce `rsethAmountToMint > 0` unconditionally** in `_beforeDeposit`:
   ```solidity
   if (rsethAmountToMint == 0) revert ZeroRsETHMinted();
   ```
2. **Set a non-zero default for `pricePercentageLimit`** in `LRTOracle.initialize()` (e.g., `1e16` for 1%), or require it to be set before the first deposit is accepted.
3. **Pre-mint a meaningful amount of rsETH** to the protocol treasury at initialization to make the first-depositor attack economically infeasible.
4. **Consider snapshot-based or deposit-tracked TVL** rather than raw `balanceOf` / `address(this).balance` to prevent donation-based inflation.

## Proof of Concept

// Step 2: attacker makes minimal deposit

lrtDepositPool.depositETH{value: 1}(0, ""); // mints 1 wei rsETH

// Step 5: victim deposits with no slippage guard

lrtDepositPool.depositETH{value: 10_000 ether}(0, "");

assertEq(rsETH.balanceOf(victim), 0); // victim lost 10_000 ETH
