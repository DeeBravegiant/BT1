Audit Report

## Title
First Depositor ETH Donation Inflates rsETH Price, Enabling Fund Theft from Depositors with Zero Slippage Guard - (File: contracts/LRTDepositPool.sol)

## Summary
`LRTDepositPool` accepts unrestricted ETH donations via `receive()` and related payable functions, all of which are counted directly as protocol TVL through `address(this).balance` in `getETHDistributionData()`. Because `pricePercentageLimit` defaults to `0` and is never set in `initialize()`, the price guard in `_updateRsETHPrice()` is permanently disabled at deployment. A first depositor can donate ETH to inflate `rsETHPrice` to an arbitrary value, causing a subsequent depositor who passes `minRSETHAmountExpected = 0` to receive zero rsETH while their ETH is retained by the contract — a direct theft.

## Finding Description

**Root cause 1 — Unrestricted ETH donation.**
`LRTDepositPool` exposes four unrestricted payable entry points with no access control: [1](#0-0) 

**Root cause 2 — Donated ETH counted as protocol TVL.**
`getETHDistributionData()` returns `address(this).balance` directly as `ethLyingInDepositPool`: [2](#0-1) 

This feeds into `getTotalAssetDeposits` → `_getTotalEthInProtocol()` → `_updateRsETHPrice()`, so any donated ETH inflates the computed `totalETHInProtocol`. [3](#0-2) 

**Root cause 3 — `pricePercentageLimit` is never initialized; price guard is a no-op.**
`LRTOracle.initialize()` never sets `pricePercentageLimit`, so it defaults to `0`: [4](#0-3) 

The upside guard in `_updateRsETHPrice()` short-circuits on `pricePercentageLimit > 0`, making it permanently inactive until an admin explicitly calls `setPricePercentageLimit`: [5](#0-4) 

**Root cause 4 — Zero rsETH mint is not rejected.**
`getRsETHAmountToMint` uses integer division; if `rsETHPrice` is inflated beyond `amount * assetPrice`, the result truncates to `0`: [6](#0-5) 

`_beforeDeposit` only reverts if `rsethAmountToMint < minRSETHAmountExpected`. When both are `0`, the check `0 < 0` is `false`, so the call proceeds and `_mintRsETH(0)` is executed — minting nothing while the victim's ETH (already transferred via `msg.value`) remains in the contract: [7](#0-6) 

**Why existing checks fail.**
The `pricePercentageLimit` guard is the only mechanism that could prevent unbounded price inflation, but it is disabled by default. The `minRSETHAmountExpected` slippage check is the only mechanism that could protect the depositor, but it is caller-supplied and commonly passed as `0` by integrators and aggregators. There is no protocol-enforced minimum rsETH output.

## Impact Explanation

**Critical — Direct theft of user funds.**

The victim's ETH enters `address(this).balance` at the moment `depositETH` is called (via `msg.value`), before any minting logic runs. If `rsethAmountToMint == 0` and `minRSETHAmountExpected == 0`, the transaction succeeds, the victim receives zero rsETH, and their ETH is permanently held by the contract. The attacker holds the only rsETH in existence (1 wei), which after a second `updateRSETHPrice()` call represents the entire protocol TVL (attacker's initial deposit + donation + victim's deposit), redeemable via the withdrawal path. This constitutes direct theft of user funds, matching the Critical impact class.

## Likelihood Explanation

Two conditions must hold simultaneously:

1. `pricePercentageLimit == 0` — this is the **default state** at every fresh deployment. The admin must call `setPricePercentageLimit` explicitly after deployment. During the window before that call (including at protocol launch), the guard is absent.
2. Victim passes `minRSETHAmountExpected = 0` — this is a common real-world pattern: simple front-ends, aggregators, and early integrators frequently omit slippage protection or pass `0` as a default.

Both conditions are realistic and likely to co-occur at protocol launch, making the overall likelihood **Medium**, yielding a **Critical** severity finding.

## Recommendation

1. **Enforce non-zero rsETH output in `_beforeDeposit`**: add `if (rsethAmountToMint == 0) revert ZeroRsETHMinted();` unconditionally, regardless of `minRSETHAmountExpected`.
2. **Initialize `pricePercentageLimit` in `LRTOracle.initialize()`** to a safe non-zero value (e.g., `1e16` for 1%) so the price guard is active from block 0.
3. **Restrict or account for ETH donations separately**: either add access control to `receive()` and the other payable functions, or track "legitimate" ETH inflows in a separate accounting variable so arbitrary donations are not counted as protocol TVL.
4. **Consider the dead-shares pattern**: mint a small amount of rsETH to a dead address on the first deposit to prevent the supply from ever being 1 wei, eliminating the price manipulation surface.

## Proof of Concept

```
Initial state: rsethSupply = 0, pricePercentageLimit = 0, depositLimit set to > 2e18

1. Attacker calls depositETH{value: 1 wei}(0, "")
   → rsethSupply == 0 → rsETHPrice = 1e18 (hardcoded)
   → rsethAmountToMint = (1 * 1e18) / 1e18 = 1
   → Attacker receives 1 wei rsETH

2. Attacker sends 1e18 ETH directly to LRTDepositPool via receive()
   → address(this).balance = 1 + 1e18

3. Attacker calls updateRSETHPrice()
   → totalETHInProtocol = 1 + 1e18
   → newRsETHPrice = (1 + 1e18) / 1 = 1e18 + 1
   → pricePercentageLimit == 0 → guard skipped → rsETHPrice = 1e18 + 1

4. Victim calls depositETH{value: 1e18}(0, "")
   → rsethAmountToMint = (1e18 * 1e18) / (1e18 + 1) = 0  (integer truncation)
   → minRSETHAmountExpected = 0 → 0 < 0 is false → no revert
   → _mintRsETH(0) → victim receives 0 rsETH
   → victim's 1e18 ETH is now in the contract

5. Attacker calls updateRSETHPrice()
   → totalETHInProtocol ≈ 2e18 + 1
   → newRsETHPrice = (2e18 + 1) / 1 ≈ 2e18 + 1
   → pricePercentageLimit == 0 → guard skipped → rsETHPrice updated

6. Attacker initiates withdrawal with 1 wei rsETH
   → redeems ≈ 2e18 + 1 ETH
   → Net profit: ≈ 1e18 ETH (victim's entire deposit)
   → Victim's 1e18 ETH is stolen
```

A Foundry fork test can reproduce this by: deploying the contracts, calling `depositETH{value: 1}` as attacker, sending `1e18` ETH to the pool, calling `updateRSETHPrice()`, then calling `depositETH{value: 1e18}(0, "")` as victim, and asserting `rsETH.balanceOf(victim) == 0`.
