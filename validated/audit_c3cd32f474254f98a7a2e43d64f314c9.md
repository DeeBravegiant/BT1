Audit Report

## Title
Unbounded Withdrawal Queue Enables Temporary Freezing of Funds via Dust Withdrawal Requests — (`contracts/LRTWithdrawalManager.sol`)

## Summary
`minRsEthAmountToWithdraw[asset]` is never initialized in `initialize()`, defaulting to `0`. This allows any rsETH holder to enqueue arbitrarily many dust withdrawal requests (e.g., 1 wei rsETH each). Because `_unlockWithdrawalRequests` iterates the FIFO queue with no per-call cap, and legitimate requests cannot be unlocked until all preceding entries are processed, a sufficiently large dust queue temporarily freezes legitimate users' rsETH held by the contract.

## Finding Description
**Root cause — uninitialized minimum:**
`minRsEthAmountToWithdraw` is declared as a storage mapping at line 35 but is never set in `initialize()` (lines 90–98). Its default value is `0`. The guard in `initiateWithdrawal` at line 162 is:
```solidity
if (rsETHUnstaked == 0 || rsETHUnstaked < minRsEthAmountToWithdraw[asset]) {
    revert InvalidAmountToWithdraw();
}
```
When the mapping value is `0`, the second condition `rsETHUnstaked < 0` is always false for a `uint256`, so any positive amount — including 1 wei — passes.

**Secondary check does not prevent the attack:**
Line 170 checks `expectedAssetAmount > getAvailableAssetAmount(asset)`. For 1 wei rsETH with rsETHPrice ≈ assetPrice, `expectedAssetAmount` ≈ 1 wei, and `assetsCommitted` grows by only 1 wei per dust request. A pool holding any meaningful balance (e.g., thousands of ETH) can absorb tens of thousands of such requests before this check triggers.

**Unbounded loop — `_unlockWithdrawalRequests`:**
Lines 790–814 iterate from `nextLockedNonce` to `firstExcludedIndex` with no per-call cap. Each iteration performs a `keccak256`, two storage reads, arithmetic, and multiple storage writes (~5,000–10,000 gas). With a 30M gas block limit, ~3,000–6,000 entries per call is the practical ceiling.

**FIFO enforcement:**
`nextLockedNonce[asset]` advances strictly in order. `completeWithdrawal` at line 707 reverts with `WithdrawalLocked` if `usersFirstWithdrawalRequestNonce >= nextLockedNonce[asset]`. Legitimate requests queued after dust entries cannot be unlocked until all preceding dust entries are processed.

**Exploit path:**
1. `minRsEthAmountToWithdraw[ETH_TOKEN]` is `0` (never set).
2. Attacker holds a trivial amount of rsETH (e.g., 50,000 wei total).
3. Attacker calls `initiateWithdrawal(ETH_TOKEN, 1, "")` 50,000 times; rsETH is transferred to the contract at each call (line 166).
4. Legitimate users call `initiateWithdrawal` with real amounts; their requests are queued after the dust entries.
5. Operator calls `unlockQueue`. The while loop must iterate through all 50,000 dust entries before reaching legitimate requests, exhausting block gas.
6. Operator can call with small `firstExcludedIndex` to process in batches, but must execute ~50,000 separate transactions to drain the dust queue before any legitimate user can call `completeWithdrawal`.
7. Legitimate users' rsETH (already transferred to the contract at step 4 of their own flow) is inaccessible until the queue is drained.

## Impact Explanation
Legitimate users' rsETH is transferred to the contract at `initiateWithdrawal` time (line 166) and cannot be recovered via `completeWithdrawal` until `nextLockedNonce` advances past their request. An attacker can make this advancement arbitrarily expensive by flooding the queue with dust entries. This constitutes **temporary freezing of funds** (Medium), which is an explicitly allowed impact in scope.

## Likelihood Explanation
- No special role is required; any rsETH holder can call `initiateWithdrawal`.
- `minRsEthAmountToWithdraw` is `0` by default and requires explicit admin action to set; assets that have never had it configured are permanently vulnerable.
- The attacker's rsETH is not lost — it is returned when each dust request is eventually processed — so the net cost is only gas, making the attack economically viable on low-fee networks or during low-gas periods.
- The attack requires no flash loans, oracle manipulation, or privileged access.

## Recommendation
1. **Enforce a non-zero minimum at initialization:** Set a protocol-wide floor in `initialize()`:
```solidity
minRsEthAmountToWithdraw[LRTConstants.ETH_TOKEN] = 1e15;
```
2. **Reject zero in `setMinRsEthAmountToWithdraw`:**
```solidity
function setMinRsEthAmountToWithdraw(address asset, uint256 minRsEthAmountToWithdraw_) external onlyLRTAdmin {
    if (minRsEthAmountToWithdraw_ == 0) revert InvalidMinAmount();
    minRsEthAmountToWithdraw[asset] = minRsEthAmountToWithdraw_;
    emit MinAmountToWithdrawUpdated(asset, minRsEthAmountToWithdraw_);
}
```
3. **Cap per-call iterations in `_unlockWithdrawalRequests`:** Add a `maxIterations` parameter or hard-coded cap so a single `unlockQueue` call processes at most N entries regardless of queue depth.

## Proof of Concept
**Foundry test plan:**
1. Deploy `LRTWithdrawalManager` with a mock `LRTConfig`; confirm `minRsEthAmountToWithdraw[ETH_TOKEN] == 0`.
2. Mint attacker 50,000 wei of rsETH; call `initiateWithdrawal(ETH_TOKEN, 1, "")` in a loop 50,000 times. Assert each call succeeds and `nextUnusedNonce[ETH_TOKEN]` reaches 50,000.
3. Have a legitimate user call `initiateWithdrawal(ETH_TOKEN, 1e18, "")`. Their nonce is 50,000.
4. Advance blocks past `withdrawalDelayBlocks`.
5. Call `unlockQueue(ETH_TOKEN, 60000, ...)` and measure gas. Assert it reverts with out-of-gas or consumes >30M gas.
6. Call `completeWithdrawal(ETH_TOKEN, "")` as the legitimate user. Assert it reverts with `WithdrawalLocked`, confirming the temporary freeze.
7. Apply fix (set `minRsEthAmountToWithdraw[ETH_TOKEN] = 1e15`) and confirm step 2 reverts with `InvalidAmountToWithdraw` for 1 wei inputs.
