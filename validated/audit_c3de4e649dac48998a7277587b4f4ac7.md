### Title
Caller-Controlled Finality Threshold Allows Weak-Finality Foreign-Chain Verification, Enabling Double-Spend via Reorg - (`crates/contract/src/lib.rs`, `crates/node/src/providers/verify_foreign_tx/sign.rs`, `crates/foreign-chain-inspector/src/bitcoin/inspector.rs`, `crates/foreign-chain-inspector/src/evm/inspector.rs`)

---

### Summary

The `verify_foreign_transaction` flow accepts the finality/confirmation threshold entirely from the unprivileged caller. Neither the on-chain contract nor the MPC nodes enforce any minimum finality floor. An attacker can request MPC attestation of a foreign-chain transaction with `confirmations: 1` (Bitcoin) or `finality: Latest` (EVM), obtain a valid threshold signature, claim bridge funds on NEAR, and then profit from a chain reorganization that erases the underlying deposit — a classic double-spend via forged foreign-chain verification.

---

### Finding Description

The `verify_foreign_transaction` contract method accepts a `VerifyForeignTransactionRequestArgs` struct whose chain-specific sub-request carries the finality requirement chosen by the caller:

- **Bitcoin**: `BitcoinRpcRequest { confirmations: BlockConfirmations, ... }` — the caller sets how many confirmations are required.
- **EVM chains** (Abstract, BNB, Base, Arbitrum, HyperEVM, Polygon): `EvmRpcRequest { finality: EvmFinality, ... }` — the caller can choose `Latest`, `Safe`, or `Finalized`.
- **Starknet**: `StarknetRpcRequest { finality: StarknetFinality, ... }` — the caller can choose `AcceptedOnL2` (weaker) or `AcceptedOnL1`.

The contract's `check_request_preconditions` validates only domain existence, gas, deposit, and the `accept_requests` flag. It performs **no validation of the finality parameters**: [1](#0-0) 

The `verify_foreign_transaction` method itself only additionally checks that the requested chain is in the supported set: [2](#0-1) 

On the node side, `execute_foreign_chain_request` passes the caller-supplied threshold directly to the inspector without any floor check: [3](#0-2) 

The Bitcoin inspector enforces only that the actual confirmation count meets the **caller-supplied** threshold: [4](#0-3) 

The EVM inspector similarly uses the caller-supplied `finality` tag to select which chain head to compare against: [5](#0-4) 

The ABI schema confirms `BlockConfirmations` accepts `minimum: 0`, meaning a caller can request verification with zero confirmations: [6](#0-5) 

The signed payload (`ForeignTxSignPayloadV1`) includes the full `request` struct — meaning the weak finality parameter is embedded in the signed output. A downstream bridge contract that trusts the MPC signature without independently re-checking the finality field will accept the attestation as authoritative. [7](#0-6) 

---

### Impact Explanation

**Impact: High** — forged foreign-chain verification / light-client-style verification bypass causing invalid bridge execution or double-spend conditions.

An attacker who controls a bridge deposit flow can:

1. Deposit funds on the foreign chain (e.g., Bitcoin or an EVM chain).
2. Submit `verify_foreign_transaction` with `confirmations: 1` (Bitcoin) or `finality: Latest` (EVM) immediately after the transaction appears in a block.
3. The MPC network verifies the transaction against the weak threshold and issues a threshold signature over `(request, extracted_values)`.
4. The attacker presents the MPC signature to the NEAR bridge contract to claim the bridged funds.
5. The attacker then double-spends or waits for a natural reorg to erase the original deposit transaction.
6. The bridge has paid out on NEAR but the underlying foreign-chain deposit no longer exists.

For EVM chains with `finality: Latest`, any transaction in the latest block qualifies — these blocks are trivially reorganized on chains like Polygon (which has frequent short reorgs). For Bitcoin, a 1-confirmation reorg is rare but has occurred historically and is economically feasible for a well-capitalized attacker.

---

### Likelihood Explanation

**Likelihood: Medium.**

- EVM chains configured in this system (Polygon, BNB, Base, Arbitrum, Abstract, HyperEVM) all support `finality: Latest`. Polygon in particular has a documented history of short reorgs. An attacker targeting a high-value bridge deposit needs only to submit the request immediately after inclusion and race the reorg.
- Bitcoin 1-confirmation reorgs are rare (~once per year historically) but the economic incentive scales with bridge TVL. A miner with even a small hash-rate share can attempt a selfish-mining reorg against a 1-confirmation threshold.
- The attack requires no privileged access, no threshold collusion, and no TEE compromise — only the ability to call `verify_foreign_transaction` with a crafted request, which is an open public API.

---

### Recommendation

Enforce a **minimum finality floor** at the contract level, not just at the caller's discretion:

1. **On-chain**: Add per-chain minimum finality configuration to the `ChainEntry` stored in `ForeignChainRpcWhitelist`. Participants vote on the minimum acceptable `BlockConfirmations` (Bitcoin) and minimum `EvmFinality` level (EVM) when they vote in a chain's RPC providers. The contract should reject any `verify_foreign_transaction` request whose finality parameter falls below the voted minimum.

2. **Node-side**: As a defense-in-depth measure, nodes should also refuse to execute a foreign chain request whose finality parameter is below the on-chain minimum, even if the contract somehow accepted it.

3. **Downstream contracts**: Document clearly that the `confirmations` / `finality` field in the signed payload must be independently validated by bridge contracts before acting on the MPC attestation.

### Proof of Concept

```rust
// Attacker submits verify_foreign_transaction with confirmations: 1 (Bitcoin)
// or finality: Latest (EVM) immediately after their deposit tx is included.

// Step 1: Attacker deposits on Bitcoin (tx_id = attacker_tx)
// Step 2: After 1 confirmation, attacker calls:
let request = VerifyForeignTransactionRequestArgs {
    request: ForeignChainRpcRequest::Bitcoin(BitcoinRpcRequest {
        tx_id: attacker_tx,
        confirmations: BlockConfirmations(1), // minimum possible; no floor enforced
        extractors: vec![BitcoinExtractor::BlockHash],
    }),
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
};
// MPC nodes verify: 1 >= 1 → passes. Threshold signature issued.

// Step 3: Attacker presents MPC signature to NEAR bridge → receives bridged funds.
// Step 4: Attacker mines a competing Bitcoin block that omits attacker_tx (reorg).
// Step 5: Deposit is gone; bridge has already paid out → double-spend complete.

// For EVM (e.g., Polygon):
let request = VerifyForeignTransactionRequestArgs {
    request: ForeignChainRpcRequest::Polygon(EvmRpcRequest {
        tx_id: attacker_tx,
        finality: EvmFinality::Latest, // weakest finality; no floor enforced
        extractors: vec![EvmExtractor::BlockHash],
    }),
    domain_id: foreign_tx_domain_id,
    payload_version: ForeignTxPayloadVersion::V1,
};
// MPC nodes verify: latest_block >= receipt_block → trivially passes.
// Attacker waits for natural Polygon reorg (common) → same double-spend.
```

    /// Common preconditions enforced on every user-facing request method (`sign`,
    /// `request_app_private_key`, `verify_foreign_transaction`):
    ///
    /// 1. The target domain exists and its purpose matches `expected_purpose`.
    /// 2. The caller attached enough prepaid gas to perform the yield/resume flow.
    /// 3. The caller attached at least `minimum_deposit` (excess is refunded).
    /// 4. The contract is currently accepting user requests.
    ///
    /// Returns the validated domain config and the caller's account id.
    fn check_request_preconditions(
        &self,
        domain_id: DomainId,
        expected_purpose: DomainPurpose,
        minimum_gas: Gas,
        minimum_deposit: NearToken,
    ) -> (DomainConfig, AccountId) {
        // 1. Look up the domain and check its purpose.
        let domains = match self.protocol_state.domain_registry() {
            Ok(domains) => domains,
            Err(err) => env::panic_str(&err.to_string()),
        };
        let Some(domain_config) = domains.get_domain_by_domain_id(domain_id) else {
            env::panic_str(
                &InvalidParameters::DomainNotFound {
                    provided: domain_id,
                }
                .to_string(),
            );
        };
        if domain_config.purpose != expected_purpose {
            env::panic_str(
                &InvalidParameters::WrongDomainPurpose {
                    domain_id: domain_config.id,
                    expected: expected_purpose,
                    actual: domain_config.purpose,
                }
                .to_string(),
            );
        }
        let domain_config = domain_config.clone();

        // 2. Make sure the call will not run out of gas doing yield/resume logic.
        let prepaid_gas = env::prepaid_gas();
        if prepaid_gas < minimum_gas {
            env::panic_str(
                &InvalidParameters::InsufficientGas {
                    provided: prepaid_gas.as_gas(),
                    required: minimum_gas.as_gas(),
                }
                .to_string(),
            );
        }

        // 3. Require the minimum deposit and refund any excess.
        let predecessor = env::predecessor_account_id();
        require_deposit(minimum_deposit, &predecessor);

        // 4. Refuse the request if the contract is not currently accepting requests
        //    (e.g. because TEE validation has failed).
        if !self.accept_requests {
            env::panic_str(&TeeError::TeeValidationFailed.to_string())
        }

        (domain_config, predecessor)
    }

            callback_args,
            callback_gas,
            move |this, id| this.add_verify_foreign_tx_request(request, id),

    /// Checks that the receipt's block has reached the requested finality level — i.e. that the
    /// head of the chain at `finality` is at or past `receipt_block_number`.
    async fn verify_finality_level(
        &self,
        receipt_block_number: U64,
        finality: EthereumFinality,
    ) -> Result<(), ForeignChainInspectionError> {
        let finality_tag = match finality {
            EthereumFinality::Finalized => FinalityTag::Finalized,
            EthereumFinality::Safe => FinalityTag::Safe,
            EthereumFinality::Latest => FinalityTag::Latest,
        };
        let args = GetBlockByNumberArgs::new(
            BlockNumberOrTag::Tag(finality_tag),
            ReturnFullTransactionObjects::from(false),
        );
        let head: GetBlockByNumberResponse = self
            .client
            .request(GET_BLOCK_BY_NUMBER_METHOD, &args)
            .await?;

        if head.number < receipt_block_number {
            return Err(ForeignChainInspectionError::NotFinalized);
        }
        Ok(())
    }
