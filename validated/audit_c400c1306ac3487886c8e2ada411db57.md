### Title
Single Attested Participant Can Resolve CKD Requests With Arbitrary Key Material, Bypassing Threshold Requirement — (`File: crates/contract/src/lib.rs`)

---

### Summary

The `respond_ckd` function in the MPC contract resolves Confidential Key Derivation (CKD) requests without any cryptographic validation of the response when the request uses the `CKDAppPublicKey::AppPublicKey` variant. A single attested participant — strictly below the signing threshold — can call `respond_ckd` with attacker-controlled `big_y` / `big_c` values and the contract will immediately deliver that malicious response to the waiting user, bypassing the threshold-signature requirement entirely.

---

### Finding Description

`respond_ckd` performs the following checks before resolving a pending CKD yield:

1. `assert_caller_is_signer()` — signer must equal predecessor.
2. `is_running_or_resharing()` — protocol state guard.
3. `accept_requests` — TEE validation flag.
4. `assert_caller_is_attested_participant_and_protocol_active()` — caller must be an attested participant.

After those guards, the function branches on the `app_public_key` variant:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← NO validation
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}
pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

For the `AppPublicKey` arm, the branch body is empty — no `ckd_output_check` is called, no signature or proof is verified. The raw `CKDResponse { big_y, big_c }` supplied by the caller is serialised and immediately passed to `resolve_yields_for`, which calls `promise_yield_resume` for every queued yield under that request key, delivering the attacker-supplied bytes directly to the user's callback.

By contrast, `respond` (ECDSA) and `respond_verify_foreign_tx` both perform a full cryptographic signature check before resolving:

```rust
if !signature_is_valid {
    return Err(RespondError::InvalidSignature.into());
}
``` [2](#0-1) 

No equivalent guard exists for the `AppPublicKey` CKD path.

The `AppPublicKey` variant is a production-facing type exposed through `request_app_private_key` and the public ABI: [1](#0-0)

### Impact Explanation

**Critical — Unauthorized threshold signature / key derivation output without required participant authorization.**

A single malicious attested participant can:

1. Monitor the chain for any pending `CKDRequest` whose `app_public_key` field is the `AppPublicKey` variant.
2. Immediately call `respond_ckd(request, CKDResponse { big_y: attacker_point, big_c: attacker_point })`.
3. Because no proof is checked, the contract accepts the call and resolves all queued yields with the attacker-supplied BLS12-381 points.
4. The user's application receives a derived key that the attacker chose — a key the attacker may already know the discrete log of, or one that maps to an address the attacker controls on the target chain.

This is a direct threshold bypass: the design requires t-of-n participants to collaboratively compute the CKD output, but the `AppPublicKey` path allows 1-of-n to unilaterally substitute an arbitrary result. Funds or secrets protected by the derived key are at risk of theft.

---

### Likelihood Explanation

**Medium.** The attacker must already be an attested participant (requires governance vote and valid TEE attestation, or mock attestation in non-TEE deployments). However, once inside the participant set, exploitation requires only:

- Watching the NEAR chain for `request_app_private_key` calls that use `AppPublicKey`.
- Submitting `respond_ckd` before honest nodes do (a simple race on a public chain).

No collusion with other participants is needed. No key material needs to be leaked. The attacker wins the race by submitting the transaction immediately upon seeing the request, which is straightforward given NEAR's public mempool visibility.

---

### Recommendation

**Short term:** Add a cryptographic validity check for the `AppPublicKey` variant analogous to `ckd_output_check`. At minimum, require that the submitted `big_y` is a valid BLS12-381 G1 point consistent with the domain's public key and the request's derivation path. If no such check can be made without the full MPC output, require threshold-many identical responses before resolving the yield (mirroring how ECDSA `respond` works in practice via the node coordinator).

**Long term:** Unify the response-validation path so that every `respond_*` entry point performs a cryptographic proof of correctness before calling `resolve_yields_for`. The current asymmetry — where `respond` and `respond_verify_foreign_tx` validate but `respond_ckd` (AppPublicKey branch) does not — is structurally analogous to the RocketStorage `sstore` bypass: a privileged-but-not-fully-trusted caller can write arbitrary values into a protected output slot.

### Proof of Concept

```
1. Attacker (attested participant P_evil) observes a pending CKDRequest R on-chain
   where R.app_public_key == AppPublicKey(some_bls_pk).

2. P_evil constructs a CKDResponse with attacker-controlled points:
   response = CKDResponse {
       big_y: G  // generator point — attacker knows discrete log = 1
       big_c: G
   }

3. P_evil calls:
   mpc_contract.respond_ckd(R, response)

4. Contract execution:
   - assert_caller_is_signer()                          ✓ (P_evil calls directly)
   - is_running_or_resharing()                          ✓
   - accept_requests                                    ✓
   - assert_caller_is_attested_participant_...          ✓ (P_evil is attested)
   - match app_public_key { AppPublicKey(_) => {} }    ← no check, falls through
   - resolve_yields_for(pending_ckd_requests, R, response_bytes)
     → promise_yield_resume called for every queued yield under R

5. User's callback receives attacker-supplied (big_y, big_c).
   User derives a key the attacker controls → funds at risk.
```

The honest nodes' subsequent `respond_ckd` calls for the same request will fail because the yield is already resolved, leaving no on-chain trace of the substitution beyond the single attacker transaction.

    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
