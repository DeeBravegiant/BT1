### Title
Single Attested Participant Can Forge CKD Response When `AppPublicKey` Variant Is Used - (`File: crates/contract/src/lib.rs`)

### Summary

`respond_ckd` contains a conditional verification check that becomes a complete no-op for one variant of the `CKDAppPublicKey` input type. When the request carries `CKDAppPublicKey::AppPublicKey(_)`, the cryptographic output check is silently skipped, allowing any single attested participant to deliver an arbitrary forged CKD response and resolve the pending yield — bypassing the threshold requirement entirely.

### Finding Description

In `respond_ckd` (`crates/contract/src/lib.rs`), after confirming the caller is an attested participant, the contract conditionally verifies the CKD output:

```rust
match &request.app_public_key {
    dtos::CKDAppPublicKey::AppPublicKey(_) => {}          // ← no check at all
    dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
        if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
            env::panic_str("CKD output check failed");
        }
    }
}

pending_requests::resolve_yields_for(
    &mut self.pending_ckd_requests,
    &request,
    serde_json::to_vec(&response).unwrap(),
)
``` [1](#0-0) 

The `AppPublicKey(_)` arm is an empty block — execution falls through directly to `resolve_yields_for`, which resolves the user's yield with whatever `CKDResponse` the caller supplied, without any cryptographic verification.

This is structurally identical to the external report's root cause: a conditional check that becomes trivially true (or in this case, trivially absent) for a specific class of inputs. In the external report, `walletAddress.length != 20` caused `receiver = msg.sender`, making `require(... || msg.sender == receiver)` always pass. Here, `AppPublicKey(_)` causes the entire output-validity check to be skipped.

The `AppPublicKeyPV` path correctly calls `ckd_output_check` to verify the response against the app public key and the root BLS12-381 public key. The `AppPublicKey` path has no equivalent guard.

### Impact Explanation

A single Byzantine attested participant (strictly below the signing threshold) can:

1. Monitor the NEAR blockchain for `request_app_private_key` calls that use the `AppPublicKey` variant (all transactions are public).
2. Call `respond_ckd` with the matching `CKDRequest` and an arbitrary forged `CKDResponse` before the honest threshold response arrives.
3. `resolve_yields_for` resolves the user's yield with the forged output.

The user receives a CKD-derived key that was not produced by the honest MPC threshold computation. Depending on how the derived key is used (e.g., as a wallet key for a NEAR application), this enables the attacker to know or control the derived secret, leading to theft of funds or unauthorized signing capability. This is a direct bypass of the threshold-signature requirement for CKD: one node out of N achieves what should require ≥ threshold nodes.

This matches the allowed impact: **Bypass of threshold-signature requirements or unauthorized access to MPC key shares, signing capability, or secret material that materially enables forgery or secret recovery.**

### Likelihood Explanation

- The attacker must be an attested participant — a legitimate node operator. This is a Byzantine participant strictly below the signing threshold, which is explicitly in scope.
- The `AppPublicKey` variant is a valid, reachable code path (it is a named enum variant in the public API).
- All `request_app_private_key` calls are visible on-chain, so the attacker can trivially identify targets.
- The attack requires only a single `respond_ckd` transaction, with no front-running complexity beyond standard NEAR transaction ordering.

### Recommendation

Apply the same cryptographic output verification for `AppPublicKey` as is done for `AppPublicKeyPV`. If `AppPublicKey` is intentionally designed to skip on-chain verification (e.g., because the app verifies off-chain), the contract must at minimum require that the response carries a valid threshold signature over the CKD output, verifiable against the root BLS12-381 public key, before resolving the yield. The empty match arm should never be the path that leads directly to `resolve_yields_for`.

```rust
    #[handle_result]
    pub fn respond_ckd(&mut self, request: CKDRequest, response: CKDResponse) -> Result<(), Error> {
        let signer = Self::assert_caller_is_signer();
        log!("respond_ckd: signer={}, request={:?}", &signer, &request);

        if !self.protocol_state.is_running_or_resharing() {
            return Err(InvalidState::ProtocolStateNotRunning.into());
        }

        if !self.accept_requests {
            return Err(TeeError::TeeValidationFailed.into());
        }

        self.assert_caller_is_attested_participant_and_protocol_active();

        let PublicKeyExtended::Bls12381 {
            public_key: dtos::PublicKey::Bls12381(public_key),
        } = self.public_key_extended(request.domain_id)?
        else {
            env::panic_str("Domain is not compatible with CKD (expected Bls12381 curve)");
        };

        match &request.app_public_key {
            dtos::CKDAppPublicKey::AppPublicKey(_) => {}
            dtos::CKDAppPublicKey::AppPublicKeyPV(app_pk) => {
                if !ckd_output_check(&request.app_id, &response, app_pk, &public_key) {
                    env::panic_str("CKD output check failed");
                }
            }
        }

        pending_requests::resolve_yields_for(
            &mut self.pending_ckd_requests,
            &request,
            serde_json::to_vec(&response).unwrap(),
        )
    }
