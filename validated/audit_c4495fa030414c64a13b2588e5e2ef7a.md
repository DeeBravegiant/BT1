### Title
Missing Removal Function for Whitelisted MPC Docker Image Hashes Prevents Emergency Revocation of Compromised Images - (File: `crates/contract/src/tee/proposal.rs`)

---

### Summary

`AllowedDockerImageHashes` exposes no explicit removal path for whitelisted MPC Docker image hashes. Unlike `AllowedLauncherImages` — which supports both `Add` and `Remove` via the `LauncherVoteAction` enum — the `CodeHashesVotes` voting structure stores only a bare `NodeImageHash` per participant, with no `Remove` action variant. Once a compromised or vulnerable Docker image hash is whitelisted, it cannot be immediately revoked by participant vote; the only removal mechanism is passive time-based expiry via `cleanup_expired_hashes()`.

---

### Finding Description

**`AllowedDockerImageHashes` has no `remove()` method:** [1](#0-0) 

The struct exposes only `insert()`, `cleanup_expired_hashes()`, and read-only accessors. There is no `remove()` counterpart.

**`CodeHashesVotes` stores only an additive vote — no action enum:** [2](#0-1) 

The map value is `NodeImageHash` directly, meaning a participant can only vote *for* a hash to be added. There is no `Remove` variant.

**Contrast with `LauncherHashVotes`, which correctly models both directions:** [3](#0-2) [4](#0-3) 

`LauncherVoteAction` has `Add(LauncherImageHash)` and `Remove(LauncherImageHash)`, and `AllowedLauncherImages` has a corresponding `remove()` method: [5](#0-4) 

**The only removal path for MPC image hashes is passive expiry:** [6](#0-5) 

Expiry is time-gated by `tee_upgrade_deadline_duration` (configured in `config.tee_upgrade_deadline_duration_seconds`). Until that window closes, any whitelisted hash — including a newly discovered compromised one — remains accepted.

**Attestation verification uses the whitelist directly:** [7](#0-6) 

A node whose Docker image hash is in `allowed_docker_image_hashes` will pass `verify_locally()` and have its attestation stored in `stored_attestations`, making it an accepted TEE participant eligible to call `respond()`, `respond_ckd()`, and `respond_verify_foreign_tx()`.

---

### Impact Explanation

If a whitelisted MPC Docker image hash is later found to be compromised (e.g., via a supply-chain attack, a post-deployment CVE, or a leaked build secret), participants have **no on-chain mechanism to immediately revoke it**. Nodes running the compromised image continue to:

1. Pass `add_participant()` / `submit_participant_info()` attestation checks.
2. Be accepted as valid signers in `respond()` / `respond_ckd()`.
3. Participate in threshold signing rounds, potentially issuing unauthorized signatures or leaking key-share material.

This maps to the allowed impact: **unauthorized transaction execution or threshold signature issuance without the required participant authorization**, and **unauthorized access to MPC key shares or signing capability that materially enables forgery or secret recovery**.

---

### Likelihood Explanation

A legitimate image hash is voted in by honest participants. A vulnerability is subsequently discovered in that image (e.g., a memory-disclosure bug, a side-channel, or a compromised build pipeline). Operators cannot issue an emergency removal vote — they must wait for the full grace period to expire. During that window, adversarial nodes running the vulnerable image remain protocol-eligible. This scenario requires no collusion and no privileged access beyond what a normal external attacker exploiting a published CVE would have.

---

### Recommendation

1. Add a `Remove` variant to the `CodeHashesVotes` voting mechanism, mirroring `LauncherVoteAction`:

```rust
pub enum CodeHashVoteAction {
    Add(NodeImageHash),
    Remove(NodeImageHash),
}
```

2. Implement a `remove()` method on `AllowedDockerImageHashes` (analogous to `AllowedLauncherImages::remove()`), enforcing the same "at least one entry must remain" invariant already present in `AllowedMeasurements::remove()` and `AllowedLauncherImages::remove()`.

3. Wire a threshold-vote-triggered removal path through `TeeState`, consistent with how `AllowedLauncherImages` and `AllowedMeasurements` handle removal today.

---

### Proof of Concept

1. Participants vote in `NodeImageHash(H)` via `vote_code_hash`. Threshold is reached; `AllowedDockerImageHashes::insert(H, ...)` is called.
2. A critical vulnerability is disclosed in the image identified by `H`.
3. Participants attempt to revoke `H`. No `Remove` action exists in `CodeHashesVotes`; no `remove()` exists on `AllowedDockerImageHashes`.
4. An adversary operates a node running the vulnerable image. `submit_participant_info()` succeeds because `verify_locally()` accepts `H`.
5. The adversary's node is stored in `stored_attestations` and passes `assert_caller_is_attested_participant_and_protocol_active()`.
6. The adversary participates in signing rounds via `respond()`, issuing signatures or extracting key-share material, until `H` naturally expires after `tee_upgrade_deadline_duration` — with no emergency revocation available to honest participants. [8](#0-7) [9](#0-8)

    pub fn insert(&mut self, code_hash: NodeImageHash, tee_upgrade_deadline_duration: Duration) {
        self.cleanup_expired_hashes(tee_upgrade_deadline_duration);

        // Remove the old entry if it exists
        if let Some(pos) = self
            .allowed_tee_proposals
            .iter()
            .position(|entry| entry.image_hash == code_hash)
        {
            self.allowed_tee_proposals.remove(pos);
        }

        let new_entry = AllowedMpcDockerImage {
            image_hash: code_hash,
            added: Timestamp::now(),
        };

        // Find the correct position to maintain sorted order by `added`
        let insert_index = self
            .allowed_tee_proposals
            .iter()
            // strictly less, `<`, such that new entries take higher precedence
            // if two entries have the exact same time stamp.
            .rposition(|entry| new_entry.added < entry.added)
            .unwrap_or(self.allowed_tee_proposals.len());

        self.allowed_tee_proposals.insert(insert_index, new_entry);
    }

    pub fn remove(&mut self, launcher_hash: &LauncherImageHash) -> bool {
        let would_remain = self
            .entries
            .iter()
            .filter(|e| &e.launcher_hash != launcher_hash)
            .count();
        if would_remain == 0 {
            return false;
        }
        let len_before = self.entries.len();
        self.entries.retain(|e| &e.launcher_hash != launcher_hash);
        self.entries.len() < len_before
    }
