Audit Report

## Title
Cumulative Aave Scaled-Balance Rounding Permanently Breaks `collectInterestToTreasury` — (`contracts/LRTWithdrawalManager.sol`)

## Summary

`_depositToAave` records the full ETH `amount` sent to Aave rather than the actual aToken balance received. Because Aave v3 uses floor-division when minting scaled aTokens, each deposit can produce a 1-wei deficit in `totalETHDepositedToAave` vs. `aaveAWETH.balanceOf(address(this))`. This deficit is never corrected by withdrawals. After 3 such deposits, the deficit exceeds the hard-coded 2-wei tolerance in `_checkAaveHealth`, causing `collectInterestToTreasury` to permanently revert with `AaveHealthCheckFailed`.

## Finding Description

**Root cause — `_depositToAave` records sent amount, not received aToken balance:** [1](#0-0) 

Aave v3 mints `floor(amount × RAY / liquidityIndex)` scaled tokens, so `aaveAWETH.balanceOf(address(this))` can be `amount − 1` after the deposit. `totalETHDepositedToAave` is incremented by `amount`, not by the actual balance delta, creating a 1-wei deficit per deposit cycle.

**`_withdrawFromAave` preserves the deficit unchanged:** [2](#0-1) 

Both `totalETHDepositedToAave` and `aaveBalance` decrease by the same `withdrawnAmount`, so `totalETHDepositedToAave − aaveBalance` is invariant across any withdrawal. The deficit only grows.

**`_checkAaveHealth` uses a fixed 2-wei tolerance:** [3](#0-2) 

After 3 deposit cycles each incurring a 1-wei rounding loss, `deficit = 3 > 2` and this function permanently returns `false`.

**`collectInterestToTreasury` gates on this check and permanently reverts:** [4](#0-3) 

There is no function that resets or corrects `totalETHDepositedToAave`. Once the deficit exceeds 2 wei, the normal yield-collection path is permanently broken. Emergency recovery paths (`emergencyWithdrawFromAave`, `setAaveIntegrationEnabled(false)`) call `_collectInterestToTreasury` directly and bypass `_checkAaveHealth`, but both carry side effects (full principal withdrawal or disabling the integration entirely).

**Deposit triggers in normal operation:** [5](#0-4) [6](#0-5) 

Both `unlockQueue` and `depositIdleETHToAave` route through `_depositToAave`. No malicious action is required — routine protocol operation accumulates the deficit.

## Impact Explanation

**Medium — Permanent freezing of unclaimed yield.** The operator-callable `collectInterestToTreasury` permanently reverts after ≥3 deposit cycles with 1-wei rounding. Accrued Aave yield cannot be collected through the intended operational path. Recovery requires privileged emergency intervention with material side effects. This matches the allowed impact class "Permanent freezing of unclaimed yield."

## Likelihood Explanation

The rounding error occurs whenever `amount × RAY mod liquidityIndex ≠ 0`, which holds for the vast majority of deposit amounts given Aave v3 mainnet WETH liquidityIndex (~`1.0001×10²⁷`). With `unlockQueue` called routinely to process user withdrawal requests, 3 rounding-loss deposits occur under ordinary protocol operation with no attacker involvement. The condition is self-triggering and irreversible once reached.

## Recommendation

In `_depositToAave`, record the actual aToken balance delta rather than the ETH amount sent:

```solidity
uint256 balanceBefore = aaveAWETH.balanceOf(address(this));
aaveWETHGateway.depositETH{ value: amount }(aavePool, address(this), 0);
totalETHDepositedToAave += aaveAWETH.balanceOf(address(this)) - balanceBefore;
```

Alternatively, remove the `_checkAaveHealth` gate from `collectInterestToTreasury` entirely and rely solely on `_collectInterestToTreasury`'s own guard (`if (aaveBalance <= principal) return 0`), which already handles the rounding case gracefully without reverting.

## Proof of Concept

```solidity
// Foundry fork test (Aave v3 mainnet fork)
function test_healthCheckFailsAfterRoundingAccumulation() public {
    // Setup: configure and enable Aave integration
    uint256 depositAmount = 1e15; // 0.001 ETH — reliably triggers 1-wei rounding

    for (uint256 i = 0; i < 3; i++) {
        vm.deal(address(withdrawalManager), depositAmount);
        // Operator calls unlockQueue → depositToAaveExternal → _depositToAave
        // Each call: totalETHDepositedToAave += depositAmount
        //            aaveAWETH.balanceOf() += depositAmount - 1  (floor rounding)
        // deficit grows by 1 each iteration
    }

    // After 3 cycles: totalETHDepositedToAave - aaveAWETH.balanceOf() == 3 > 2
    // _checkAaveHealth() returns false permanently
    vm.prank(operator);
    vm.expectRevert(ILRTWithdrawalManager.AaveHealthCheckFailed.selector);
    withdrawalManager.collectInterestToTreasury();
}
```

The key invariant violated: `_withdrawFromAave` never reduces the deficit, so 3 rounding-loss deposits are sufficient and permanent. The `_checkAaveHealth` tolerance of 2 wei is insufficient for any protocol with more than 2 deposit cycles.
