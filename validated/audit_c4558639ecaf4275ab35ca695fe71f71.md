### Title
Pool Absorb Spends Reset Wallet Lock-Height Check, Permanently Trapping Farmer in LEAVING_POOL State - (File: chia/pools/pool_wallet.py)

### Summary

`PoolWallet.self_pool()`, `join_pool()`, and `new_peak()` compute the elapsed lock height using `history[-1][0]` — the height of the **most recent singleton spend**, including permissionless absorb spends. Because `apply_state_transition()` records every singleton spend (including absorb spends that carry no pool-state change) into the pool store, a pool can keep submitting absorb spends to continuously advance `history[-1][0]`, making the wallet's lock-expiry check never pass. The farmer is permanently unable to leave the pool through any normal RPC path, while the pool continues to receive farming rewards.

### Finding Description

**Root cause — wrong height anchor in lock checks**

`self_pool()` and `join_pool()` both contain:

```python
history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
last_height: uint32 = history[-1][0]
if (
    await self.wallet_state_manager.blockchain.get_finished_sync_up_to()
    <= last_height + current_state.current.relative_lock_height
):
    raise ValueError(...)
``` [1](#0-0) [2](#0-1) 

`new_peak()` uses the same anchor via `get_tip()`:

```python
tip_height, tip_spend = await self.get_tip()
...
leave_height = tip_height + pool_wallet_info.current.relative_lock_height
if peak_height > leave_height + 2:
    # submit exit transaction
``` [3](#0-2) 

`get_tip()` returns `get_spends_for_wallet()[-1]` — the last row in the pool store, regardless of whether it is a state-changing spend or an absorb spend. [4](#0-3) 

**Absorb spends are recorded in the pool store**

`apply_state_transition()` calls `pool_store.add_spend()` for every singleton spend it processes, including absorb spends: [5](#0-4) 

**Absorb spends return `None` from `solution_to_pool_state()`**

For the waiting-room absorb path (`inner_sol = Program.to([0, reward_amount, height])`), `solution_to_pool_state()` returns `None` because `inner_solution.first().as_int() == 0`: [6](#0-5) 

This means `get_current_state()` correctly skips absorb spends when computing `last_singleton_spend_height` (it iterates backwards until `solution_to_pool_state` is non-`None`): [7](#0-6) 

But `self_pool()`, `join_pool()`, and `new_peak()` bypass this correct value and read `history[-1][0]` / `get_tip()` directly, picking up the absorb height instead.

**Absorb spends are permissionless**

`create_absorb_spend()` requires no signature from the pool or farmer. It spends the singleton coin and a coinbase reward coin at the `p2_singleton` puzzle hash. Anyone who knows the singleton's current state can construct and submit one: [8](#0-7) 

The pool farms blocks whose rewards land at the `p2_singleton` address. The pool can submit absorb spends at will, each one advancing `history[-1][0]` to the current block height.

**Absorb while in LEAVING_POOL is explicitly valid**

The pool-lifecycle test confirms absorb spends succeed while the singleton is in the waiting room: [9](#0-8) 

### Impact Explanation

The CLVM-level `ASSERT_HEIGHT_RELATIVE` condition in the waiting-room puzzle is anchored to the **coin's creation height** — the block at which the singleton entered `LEAVING_POOL`. This is enforced by the blockchain and is unaffected by absorb spends. The blockchain would therefore accept the farmer's exit transaction.

However, the wallet's pre-submission checks in `self_pool()`, `join_pool()`, and `new_peak()` use `history[-1][0]`, which the pool can keep advancing by submitting absorb spends. All three code paths raise `ValueError` or skip submission, so:

- The farmer cannot call `self_pool()` via RPC to leave.
- The farmer cannot call `join_pool()` via RPC to switch pools.
- `new_peak()` never automatically submits the exit transaction.

The pool continues to receive the farmer's block rewards indefinitely. This constitutes unauthorized payout redirection and a long-lived inability for the farmer to execute a valid pool state transition through any normal wallet path.

### Likelihood Explanation

Any pool that farms blocks regularly (which is the normal operating condition) generates `p2_singleton` reward coins continuously. Submitting an absorb spend costs only a transaction fee and requires no special privilege. A malicious pool needs only to absorb one reward coin per `relative_lock_height` blocks (minimum 5, maximum 1000) to keep the timer reset. For a pool farming even a modest share of blocks, this is trivially achievable.

### Recommendation

Replace `history[-1][0]` and `get_tip()[0]` in the lock-height checks with `current_state.singleton_block_height`, which is already computed correctly by `get_current_state()` as the height of the last **state-changing** spend (skipping absorb spends):

```python
# In self_pool() and join_pool(), replace:
history: list[tuple[uint32, CoinSpend]] = await self.get_spend_history()
last_height: uint32 = history[-1][0]

# With:
last_height: uint32 = current_state.singleton_block_height
```

```python
# In new_peak(), replace:
tip_height, tip_spend = await self.get_tip()
leave_height = tip_height + pool_wallet_info.current.relative_lock_height

# With:
leave_height = pool_wallet_info.singleton_block_height + pool_wallet_info.current.relative_lock_height
```

`PoolWalletInfo.singleton_block_height` is already set to `last_singleton_spend_height` from `get_current_state()`, which correctly skips absorb spends. [10](#0-9) 

### Proof of Concept

1. Farmer joins pool with `relative_lock_height = 5`. Pool is assigned at block H.
2. Farmer calls `self_pool()` (or `join_pool()`), triggering a travel spend to `LEAVING_POOL` at block H. `history[-1][0] = H`.
3. Pool waits until block H+4 (one block before lock expires). A `p2_singleton` reward coin exists from normal farming.
4. Pool submits `create_absorb_spend(...)` for the waiting-room singleton. This is accepted by the blockchain (no signature required, valid reward coin). `apply_state_transition()` records it; `history[-1][0]` is now H+4.
5. Farmer's wallet calls `self_pool()`: `last_height = H+4`, check is `current_height <= (H+4)+5 = H+9`. If current height is H+5, the check fails and raises `ValueError`.
6. Pool repeats step 4 every ~5 blocks. `history[-1][0]` is always within the lock window.
7. `new_peak()` computes `leave_height = tip_height + 5` where `tip_height` is always the latest absorb height. `peak_height > leave_height + 2` is never true.
8. The farmer is permanently trapped. The blockchain would accept the exit transaction (CLVM `ASSERT_HEIGHT_RELATIVE` passes based on coin creation height), but the wallet never submits it.

def create_absorb_spend(
    last_coin_spend: CoinSpend,
    current_state: PoolState,
    launcher_coin: Coin,
    height: uint32,
    genesis_challenge: bytes32,
    delay_time: uint64,
    delay_ph: bytes32,
) -> list[CoinSpend]:
    inner_puzzle: Program = pool_state_to_inner_puzzle(
        current_state, launcher_coin.name(), genesis_challenge, delay_time, delay_ph
    )
    reward_amount: uint64 = calculate_pool_reward(height)
    if is_pool_member_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol: Program = Program.to([reward_amount, height])
    elif is_pool_waitingroom_inner_puzzle(inner_puzzle):
        # inner sol is (spend_type, destination_puzhash, pool_reward_amount, pool_reward_height, extra_data)
        inner_sol = Program.to([0, reward_amount, height])
    else:
        raise ValueError
    # full sol = (parent_info, my_amount, inner_solution)
    coin: Coin | None = get_most_recent_singleton_coin_from_coin_spend(last_coin_spend)
    assert coin is not None

    if coin.parent_coin_info == launcher_coin.name():
        parent_info: Program = Program.to([launcher_coin.parent_coin_info, launcher_coin.amount])
    else:
        p = Program.from_bytes(bytes(last_coin_spend.puzzle_reveal))
        last_coin_spend_inner_puzzle: Program | None = get_inner_puzzle_from_puzzle(p)
        assert last_coin_spend_inner_puzzle is not None
        parent_info = Program.to(
            [
                last_coin_spend.coin.parent_coin_info,
                last_coin_spend_inner_puzzle.get_tree_hash(),
                last_coin_spend.coin.amount,
            ]
        )
    full_solution: SerializedProgram = SerializedProgram.to([parent_info, last_coin_spend.coin.amount, inner_sol])
    full_puzzle: SerializedProgram = create_full_puzzle(inner_puzzle, launcher_coin.name()).to_serialized()
    assert coin.puzzle_hash == full_puzzle.get_tree_hash()

    reward_parent: bytes32 = pool_parent_id(height, genesis_challenge)
    p2_singleton_puzzle = create_p2_singleton_puzzle(
        SINGLETON_MOD_HASH, launcher_coin.name(), delay_time, delay_ph
    ).to_serialized()
    reward_coin: Coin = Coin(reward_parent, p2_singleton_puzzle.get_tree_hash(), reward_amount)
    p2_singleton_solution = SerializedProgram.to([inner_puzzle.get_tree_hash(), reward_coin.name()])
    assert p2_singleton_puzzle.get_tree_hash() == reward_coin.puzzle_hash
    assert full_puzzle.get_tree_hash() == coin.puzzle_hash
    assert get_inner_puzzle_from_puzzle(Program.from_bytes(bytes(full_puzzle))) is not None

    coin_spends = [
        CoinSpend(coin, full_puzzle, full_solution),
        CoinSpend(reward_coin, p2_singleton_puzzle, p2_singleton_solution),
    ]
    return coin_spends
