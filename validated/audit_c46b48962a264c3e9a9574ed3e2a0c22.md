Audit Report

## Title
Contract Withdrawers Without `receive()` Have ETH Permanently Frozen After rsETH Is Burned - (File: contracts/LRTWithdrawalManager.sol)

## Summary

When a smart contract without a `receive()` or `fallback()` function calls `initiateWithdrawal`, the rsETH is irreversibly burned by the operator in a separate `unlockQueue` transaction. Every subsequent `completeWithdrawal` call then reverts at `_transferAsset` because the low-level ETH push fails, leaving the allocated ETH permanently locked in `LRTWithdrawalManager` with no redirect or recovery path.

## Finding Description

The withdrawal lifecycle spans three separate transactions:

**Phase 1 — `initiateWithdrawal` (user-triggered):**
rsETH is pulled from `msg.sender` into the withdrawal manager and a `WithdrawalRequest` is stored keyed to `msg.sender`. [1](#0-0) 

**Phase 2 — `unlockQueue` (operator-triggered):**
rsETH held by the contract is permanently burned and the corresponding ETH is pulled from `LRTUnstakingVault` into the withdrawal manager. This is an irreversible state change in a separate transaction. [2](#0-1) 

**Phase 3 — `completeWithdrawal` (user-triggered):**
`_processWithdrawalCompletion` is called with `user = msg.sender`. It pops the nonce queue, deletes the request, decrements `unlockedWithdrawalsCount`, and then calls `_transferAsset(asset, user, request.expectedAssetAmount)`. [3](#0-2) 

`_transferAsset` for ETH uses a low-level push: [4](#0-3) 

If `user` is a contract without `receive()`, `sent == false` and the function reverts with `EthTransferFailed`. The revert rolls back all state changes in Phase 3 — the nonce queue is restored, the request is un-deleted, and `unlockedWithdrawalsCount` is restored — but the rsETH burned in Phase 2 (a committed, separate transaction) is not restored.

**No escape hatch exists:**

- `completeWithdrawalForUser` (operator-accessible) also calls `_processWithdrawalCompletion(asset, user, ...)` with the same hardcoded `user` address. Its own NatSpec acknowledges ETH delivery issues: *"Not expected to be used for ETH"*. [5](#0-4) 

- `sweepRemainingAssets` is gated by `hasUnlockedWithdrawals(asset)`, which returns `true` as long as `unlockedWithdrawalsCount[asset] > 0`. Since Phase 3 always reverts, this counter never reaches zero, permanently blocking the sweep. [6](#0-5) [7](#0-6) 

- There is no `recipient` parameter, no position-transfer function, and no admin override that sends to an alternative address.

## Impact Explanation

**Critical — Permanent freezing of funds.**

The user's rsETH is irreversibly destroyed in Phase 2. The ETH allocated to them sits in `LRTWithdrawalManager` but can never be delivered: every Phase 3 attempt reverts, no redirect path exists, and the admin sweep is permanently blocked by the stuck `unlockedWithdrawalsCount`. Both the burned rsETH and the locked ETH are unrecoverable.

## Likelihood Explanation

**Medium.** Smart contracts that hold rsETH and interact with the withdrawal system are a realistic and expected use case: protocol-owned vaults, DAO treasuries, yield aggregators, and multisig-controlled strategies. Many such contracts omit `receive()` by design (e.g., pure ERC-20 vaults, proxy contracts whose implementation does not accept ETH). No special privilege or front-running is required — the triggering condition is simply calling `initiateWithdrawal` from such a contract.

## Recommendation

Add a `recipient` parameter to `initiateWithdrawal` (or to `completeWithdrawal`) and store it in `WithdrawalRequest`. Use `recipient` instead of `user` in `_processWithdrawalCompletion` when calling `_transferAsset`. This allows the caller to designate an ETH-capable receiving address at initiation time. Alternatively, add a `transferWithdrawalRecipient(address asset, uint256 nonce, address newRecipient)` function that lets the current owner of a withdrawal request redirect its delivery address before Phase 3.

## Proof of Concept

1. Deploy `VaultContract` (no `receive()`) holding rsETH. Call:
   ```solidity
   LRTWithdrawalManager.initiateWithdrawal(ETH_TOKEN, rsETHAmount, "");
   ```
   rsETH is transferred to the withdrawal manager; `withdrawalRequests[requestId]` is created for `VaultContract`.

2. Operator calls `unlockQueue(ETH_TOKEN, ...)`.
   Line 305 executes: `IRSETH(...).burnFrom(address(this), rsETHBurned)` — rsETH is **permanently burned**. ETH is pulled from `LRTUnstakingVault` into `LRTWithdrawalManager`.

3. `VaultContract` calls `completeWithdrawal(ETH_TOKEN, "")`.
   `_processWithdrawalCompletion` reaches line 734: `_transferAsset(ETH_TOKEN, VaultContract, amount)`.
   Inside `_transferAsset`: `payable(VaultContract).call{value: amount}("")` returns `false`; `EthTransferFailed` is thrown; entire transaction reverts.

4. State after revert: `withdrawalRequests[requestId]` is restored, `unlockedWithdrawalsCount[ETH] > 0`, `sweepRemainingAssets` is blocked. rsETH burned in step 2 is gone. ETH in the withdrawal manager is permanently frozen. Steps 3–4 repeat indefinitely with the same result.
