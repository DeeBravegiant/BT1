### Title
`check_storage_stake` in RPC/Chunk-Production Transaction Validation Ignores Pending Balance Constraint - (File: runtime/runtime/src/verifier.rs)

### Summary

In `verify_and_charge_tx_ephemeral` and `verify_and_charge_gas_key_tx_ephemeral`, the balance passed to `check_storage_stake` does not subtract `pending.paid_from_balance`. The balance check for transaction cost correctly uses `available_balance = account.amount() - pending.paid_from_balance`, but the subsequent storage-stake check uses `new_amount = account.amount() - total_cost` — omitting the pending deduction. This causes the RPC and chunk-production path to accept transactions that will fail at execution time with `LackBalanceForState`, producing inaccurate admission decisions analogous to the `sUSX` `maxWithdraw`/`maxRedeem` functions not accounting for `mintCap`.

### Finding Description

In `verify_and_charge_tx_ephemeral` (`runtime/runtime/src/verifier.rs`):

```
let available_balance = account.amount().saturating_sub(pending.paid_from_balance);
if available_balance < total_cost {
    return TxVerdict::Failed(InvalidTxError::NotEnoughBalance { ... });
}
// BUG: new_amount ignores pending.paid_from_balance
let new_amount = account.amount().checked_sub(total_cost).unwrap();
...
match check_storage_stake(account, new_amount, config) { ... }
```

The balance check on line 310 correctly subtracts `pending.paid_from_balance` to compute `available_balance`. However, `new_amount` on line 320 is computed from `account.amount()` (the full certified-state balance), not from `available_balance`. The `check_storage_stake` call on line 332 therefore evaluates storage staking against a balance that is too high by exactly `pending.paid_from_balance`.

The same pattern exists in `verify_and_charge_gas_key_tx_ephemeral` at lines 490 and 503–505: `available_balance` is computed correctly, but `new_account_amount` passed to `check_storage_stake` is `account.amount() - deposit_cost` rather than `available_balance - deposit_cost`. [1](#0-0) [2](#0-1) 

### Impact Explanation

The corrupted value is the RPC admission decision (`ProcessTxResponse::ValidTx` vs `ProcessTxResponse::InvalidTx`) and the chunk-production inclusion decision. A transaction is admitted/included when it should be rejected, because the storage-stake check uses a balance that is inflated by the amount already committed to pending (uncertified) transactions.

At execution time, `check_storage_stake` in `runtime/runtime/src/lib.rs` (line 878) is called again with the actual post-execution balance. If the pending transactions have already drained the account's balance below the storage-staking threshold, this execution-time check fails with `LackBalanceForState`, causing the transaction to fail — contradicting the earlier RPC/chunk-production acceptance. [3](#0-2) 

The corrupted protocol value is the `InvalidTxError` result returned by `can_verify_and_charge_tx` to the RPC handler, which drives the `ProcessTxResponse` sent back to the user. [4](#0-3) 

### Likelihood Explanation

This is reachable by any unprivileged user on the Spice protocol path where `spice_pending_transaction_queue_enabled` is true. The attacker-controlled entry path is:

1. Submit transaction T1 with a large deposit that nearly exhausts the account balance. T1 is included in a block but not yet certified — `paid_from_balance` in the pending transaction queue now reflects T1's cost.
2. Submit transaction T2 (e.g., a small transfer). The RPC handler calls `get_pending_constraints` to obtain `paid_from_balance`, then calls `can_verify_and_charge_tx` → `verify_and_charge_tx_ephemeral`. The balance check correctly rejects if `available_balance < total_cost`, but the storage-stake check uses `account.amount() - total_cost` (ignoring the pending deduction), so it passes even though the post-T1 balance would be below the storage-staking threshold.
3. T2 is accepted by the RPC and included in a chunk.
4. T1 executes first, draining the balance. T2 then executes and fails with `LackBalanceForState`.

The existing test `test_ptq_account_balance_access_key_gas_key_combined` demonstrates the pending-balance enforcement for the deposit check, but no test covers the storage-stake path with pending constraints. [5](#0-4) 

### Recommendation

In `verify_and_charge_tx_ephemeral`, replace:

```rust
let new_amount = account.amount().checked_sub(total_cost).unwrap();
```

with:

```rust
let new_amount = available_balance.checked_sub(total_cost).unwrap();
```

Apply the same fix in `verify_and_charge_gas_key_tx_ephemeral`: replace `account.amount().checked_sub(deposit_cost)` with `available_balance.checked_sub(deposit_cost)` before passing the result to `check_storage_stake`. [6](#0-5) 

### Proof of Concept

1. Create account `alice` with balance exactly equal to storage-staking requirement + 2 × transfer cost (e.g., `S + 2C`).
2. Submit T1: transfer of `S + C` to `bob` (deposit = `S + C`). T1 is included but not certified. `paid_from_balance` = `S + C + gas`.
3. Submit T2: transfer of `1 yoctoNEAR` to `bob`. RPC calls `verify_and_charge_tx_ephemeral`:
   - `available_balance = (S + 2C) - (S + C + gas) ≈ C - gas` → passes balance check (≥ `total_cost` of T2).
   - `new_amount = (S + 2C) - total_cost_T2 ≈ S + 2C - C = S + C` → `check_storage_stake` passes (balance ≥ S).
   - RPC returns `ValidTx`.
4. T1 executes: alice's balance drops to `≈ C - gas`.
5. T2 executes: `check_storage_stake` in `lib.rs` sees balance `≈ C - gas - total_cost_T2 < S` → `LackBalanceForState` error. T2 fails despite being accepted by the RPC. [1](#0-0) [7](#0-6) [8](#0-7)

/// Account balance enforcement across access key and gas key transactions.
///
/// Submit an access key tx with a large deposit that nearly exhausts the
/// account balance (1000 NEAR). Then submit a gas key tx whose deposit
/// exceeds the remaining balance. The gas key tx's deposit is paid from the
/// account balance, so the pending transaction queue's paid_from_balance
/// constraint causes the RPC to reject it with NotEnoughBalanceForDeposit.
#[test]
#[cfg_attr(not(feature = "protocol_feature_spice"), ignore)]
fn test_ptq_account_balance_access_key_gas_key_combined() {
    init_test_logger();

    let account = create_account_id("combo_account");
    let receiver = create_account_id("receiver");
    let fund_amount = Balance::from_millinear(100);
    let setup = setup_gas_key_spice_env(&account, &receiver, 1, fund_amount);
    let mut env = setup.env;

    // Submit an access key tx with a large deposit that nearly exhausts the
    // 1000 NEAR account balance.
    let block_hash = env.validator().head().last_block_hash;
    let access_key_tx = SignedTransaction::send_money(
        setup.next_access_key_nonce,
        account.clone(),
        receiver.clone(),
        &create_user_test_signer(&account),
        Balance::from_near(999),
        block_hash,
    );
    env.validator().submit_tx(access_key_tx.clone());

    // Wait for inclusion so the pending transaction queue tracks paid_from_balance.
    let access_key_hash = access_key_tx.get_hash();
    env.validator_runner().run_until_included(&[access_key_hash]);

    // Submit a gas key tx whose deposit exceeds the remaining balance.
    // Gas is paid from the gas key, but deposit is paid from the account.
    let block_hash = env.validator().head().last_block_hash;
    let gas_key_tx = SignedTransaction::from_actions_v1(
        TransactionNonce::from_nonce_and_index(setup.gas_key_nonces[0] + 1, 0),
        account,
        receiver,
        &setup.gas_key_signer,
        vec![Action::Transfer(TransferAction { deposit: Balance::from_near(2) })],
        block_hash,
    );
    let result = env.validator_runner().execute_tx(gas_key_tx, Duration::seconds(5));
    assert!(matches!(result, Err(InvalidTxError::NotEnoughBalanceForDeposit { .. })),);
}

    /// Extract constraints for a given transaction without Skip/Admit logic.
    /// Used by the RPC handler for balance/nonce verification against certified state.
    pub fn get_pending_constraints(&self, tx: &SignedTransaction) -> PendingConstraints {
        let snapshot = self.query_pending_state(
            tx.transaction.signer_id(),
            tx.transaction.public_key(),
            tx.transaction.nonce().nonce_index(),
        );
        PendingConstraints {
            paid_from_balance: snapshot.paid_from_balance,
            paid_from_gas_key: snapshot.pending_gas_key_cost,
            max_nonce: snapshot.max_nonce,
        }
    }
