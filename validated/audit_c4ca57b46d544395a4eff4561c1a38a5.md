Audit Report

## Title
Decimal Precision Oversight in Token-to-rsETH Swap Calculation Causes Massive Under-Minting for Non-18-Decimal Tokens - (File: contracts/pools/RSETHPoolV3.sol, RSETHPoolV3ExternalBridge.sol, RSETHPoolV3WithNativeChainBridge.sol, RSETHPoolNoWrapper.sol)

## Summary
The `viewSwapRsETHAmountAndFee` function in all four pool variants computes rsETH output as `amountAfterFee * tokenToETHRate / rsETHToETHrate` without normalizing `amountAfterFee` to 18 decimals. The oracle always returns a price scaled to 1e18 (representing the ETH value of one whole token), so the formula is only dimensionally correct for 18-decimal tokens. If a token with fewer decimals (e.g., USDC at 6) is added via `addSupportedToken`, every depositor receives up to `10^(18 - tokenDecimals)` times fewer rsETH than owed while their full token balance is transferred and bridged to L1.

## Finding Description
`ChainlinkOracleForRSETHPoolCollateral.getRate()` normalizes the Chainlink price to a fixed 1e18 scale:

```solidity
uint256 normalizedPrice = uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
``` [1](#0-0) 

This returns the ETH value of **one whole token** × 1e18, regardless of the token's own decimal count.

The swap formula in `RSETHPoolV3.viewSwapRsETHAmountAndFee` is:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
``` [2](#0-1) 

For this to be dimensionally correct, `amountAfterFee` must be in 1e18 units. For a 6-decimal token, `amountAfterFee` is in 1e6 units, producing a result 1e12 times too small. The identical pattern exists in all four variants: [3](#0-2) [4](#0-3) 

`addSupportedToken` performs no decimal check before registering a token: [5](#0-4) 

No decimal enforcement exists anywhere in the pool contracts: [6](#0-5) 

**Exploit path:**
1. Governance calls `addSupportedToken(USDC, usdcOracle, usdcBridge)` — a routine protocol expansion.
2. Alice calls `deposit(USDC, 1_000_000e6, "ref")`.
3. `IERC20(USDC).safeTransferFrom` moves 1,000,000 USDC (full value) to the pool.
4. `viewSwapRsETHAmountAndFee` computes `rsETHAmount = 1e12 * 3e14 / 1e18 = 3e8` instead of the correct `3e20`.
5. `wrsETH.mint(msg.sender, 3e8)` — Alice receives ~0.0000003 rsETH instead of ~300 rsETH.
6. The 1,000,000 USDC is subsequently bridged to L1 and deposited into `LRTDepositPool`, capturing full ETH value for the protocol. Alice has no recourse.

No existing guard prevents this: `onlySupportedToken` only checks that the token is registered; `limitDailyMint` operates on the raw `amount` in token units; `nonReentrant` and `whenNotPaused` are irrelevant to the accounting error.

## Impact Explanation
**Critical — Direct theft of user funds in motion.** The depositor's full token balance is transferred to the pool and bridged to L1 (capturing its full ETH value), while the minted rsETH/wrsETH is `10^(18 - tokenDecimals)` times too small. For USDC (6 decimals), the shortfall is exactly 1e12×. The loss is immediate, automatic, and irreversible upon deposit. This matches the allowed Critical impact: "Direct theft of any user funds, whether at-rest or in-motion."

## Likelihood Explanation
**Medium.** The bug is latent under the current token set (wstETH, WETH — both 18 decimals). It activates the moment governance adds any sub-18-decimal token via `addSupportedToken`. Adding widely-used stablecoins (USDC, USDT) is a natural and documented protocol expansion path. No attacker action is required beyond depositing the newly-supported token; every depositor of that token is automatically and fully harmed. The precondition is a legitimate governance action, not a governance compromise.

## Recommendation
Normalize `amountAfterFee` to 18 decimals before applying the rate formula in all four pool variants:

```solidity
uint8 tokenDecimals = IERC20Metadata(token).decimals();
uint256 normalizedAmount = amountAfterFee * 10 ** (18 - tokenDecimals);
rsETHAmount = normalizedAmount * tokenToETHRate / rsETHToETHrate;
```

Apply consistently to `RSETHPoolV3.sol`, `RSETHPoolV3ExternalBridge.sol`, `RSETHPoolV3WithNativeChainBridge.sol`, and `RSETHPoolNoWrapper.sol`. Alternatively, enforce at `addSupportedToken` time that `IERC20Metadata(token).decimals() == 18`.

## Proof of Concept
**Foundry fork test outline:**

```solidity
function test_usdc_deposit_undermints() public {
    // 1. Deploy/fork with RSETHPoolV3
    // 2. Prank TIMELOCK_ROLE: addSupportedToken(USDC, usdcOracle, usdcBridge)
    // 3. Deal Alice 1_000_000e6 USDC; Alice approves pool
    // 4. uint256 balBefore = wrsETH.balanceOf(alice);
    // 5. vm.prank(alice); pool.deposit(USDC, 1_000_000e6, "");
    // 6. uint256 minted = wrsETH.balanceOf(alice) - balBefore;
    // 7. uint256 expected = 1_000_000e18 * usdcToETHRate / rsETHToETHrate;
    // 8. assertApproxEqRel(minted, expected, 1e15); // FAILS: minted is 1e12x too small
    // 9. assertEq(IERC20(USDC).balanceOf(address(pool)), 1_000_000e6); // full transfer confirmed
}
```

The test will demonstrate that `minted` ≈ `3e8` while `expected` ≈ `3e20`, confirming the 1e12× shortfall and that the full USDC balance was transferred to the pool.
