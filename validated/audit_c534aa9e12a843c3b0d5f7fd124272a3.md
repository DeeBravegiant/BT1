Audit Report

## Title
`ChainlinkOracleForRSETHPoolCollateral::getRate` Relies on Deprecated `answeredInRound` Staleness Guard, Allowing Stale Prices to Pass Unchecked - (File: `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol`)

## Summary

`ChainlinkOracleForRSETHPoolCollateral::getRate` performs its only staleness check using the deprecated Chainlink `answeredInRound` field, which on all modern OCR feeds is always equal to `roundId`, making the `StalePrice` revert permanently unreachable. No timestamp-based fallback exists. Stale collateral token prices flow directly into `RSETHPoolV3::viewSwapRsETHAmountAndFee`, causing incorrect rsETH minting amounts on every deposit made during a period of feed staleness.

## Finding Description

In `contracts/pools/oracle/ChainlinkOracleForRSETHPoolCollateral.sol` at lines 27–30:

```solidity
(uint80 roundID, int256 ethPrice,, uint256 timestamp, uint80 answeredInRound) =
    AggregatorV3Interface(oracle).latestRoundData();

if (answeredInRound < roundID) revert StalePrice();   // always false on OCR feeds
if (timestamp == 0) revert IncompleteRound();          // only catches incomplete rounds
if (ethPrice <= 0) revert InvalidPrice();
```

On all current Chainlink OCR feeds, `answeredInRound` is unconditionally equal to `roundId`. The condition `answeredInRound < roundID` is therefore never true, and `StalePrice` is dead code. The `timestamp == 0` guard only catches rounds that never completed — it does not bound how old the price is. There is no `block.timestamp - timestamp > threshold` check anywhere in the function.

This oracle is registered as the price source for supported collateral tokens via `supportedTokenOracle[token]` in `RSETHPoolV3` (line 41). When a user calls `deposit(token, amount, referralId)` (line 271), the pool calls `viewSwapRsETHAmountAndFee(amount, token)` (line 286), which fetches the collateral rate at line 331:

```solidity
uint256 tokenToETHRate = IOracle(supportedTokenOracle[token]).getRate();
rsETHAmount = amountAfterFee * tokenToETHRate / rsETHToETHrate;
```

A stale `tokenToETHRate` passes all three checks and directly determines how many rsETH tokens are minted.

## Impact Explanation

**Direct theft of user funds (Critical):** If the Chainlink feed for a supported collateral token goes stale and the last reported price is below the current market price, an attacker depositing during that window receives more rsETH than the deposited collateral is worth. The excess rsETH represents value extracted from the pool at the expense of existing rsETH holders. This is direct theft of funds at-rest in the pool.

**Contract fails to deliver promised returns (Low):** If the stale price is above market, depositors receive less rsETH than owed.

## Likelihood Explanation

Chainlink feeds update on a heartbeat schedule (e.g., 1 hour or 24 hours depending on the feed) or when price deviation exceeds a threshold. During low-volatility periods, network congestion, or node issues, feeds can remain stale for the full heartbeat window without triggering the deviation update. No privileged access or oracle compromise is required — the attacker only needs to observe that the feed has not updated recently and submit a deposit. The condition is externally observable on-chain and repeatable across any deposit window where the feed is stale.

## Recommendation

Remove `answeredInRound` from the destructuring entirely and replace the staleness guard with a configurable `updatedAt` timestamp comparison:

```solidity
uint256 public constant STALENESS_THRESHOLD = 3600; // configure per feed heartbeat

function getRate() public view returns (uint256) {
    (, int256 ethPrice,, uint256 updatedAt,) =
        AggregatorV3Interface(oracle).latestRoundData();

    if (updatedAt == 0) revert IncompleteRound();
    if (block.timestamp - updatedAt > STALENESS_THRESHOLD) revert StalePrice();
    if (ethPrice <= 0) revert InvalidPrice();

    return uint256(ethPrice) * 1e18 / 10 ** uint256(AggregatorV3Interface(oracle).decimals());
}
```

The threshold should be set per-oracle to match the feed's documented heartbeat interval.

## Proof of Concept

1. Deploy a mock `AggregatorV3Interface` that returns `roundId == answeredInRound == 5`, `ethPrice = 1800e8` (stale, 10% below current market of 2000e8), and `updatedAt = block.timestamp - 7200` (2 hours ago).
2. Register this mock as the oracle for a supported collateral token in `RSETHPoolV3` via `addSupportedToken`.
3. Call `RSETHPoolV3::deposit(token, largeAmount, "")` as an unprivileged attacker.
4. Observe: `getRate()` returns `1800e18 / 1e18 = 1800` (stale price). `answeredInRound == roundID` so `StalePrice` is never reverted. `updatedAt != 0` so `IncompleteRound` is not reverted.
5. `rsETHAmount = amountAfterFee * 1800e18 / rsETHToETHrate` — inflated relative to the true 2000e18 rate, minting excess rsETH to the attacker.
